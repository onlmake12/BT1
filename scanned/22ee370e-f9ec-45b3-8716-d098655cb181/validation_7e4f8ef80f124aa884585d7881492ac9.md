Audit Report

## Title
Missing `compact_target` Epoch Validation in Compact Block Relay Header Verifier Enables Zero-Work Memory Exhaustion DoS — (`sync/src/relayer/compact_block_process.rs`, `verification/src/header_verifier.rs`)

## Summary
The `HeaderVerifier` used in the compact block relay path verifies PoW against the `compact_target` declared in the header itself, but never checks that this target matches the epoch's consensus-required difficulty. With `compact_target=0x20ffffff`, the PoW target becomes `0xffffff << 232` — so large that virtually any nonce satisfies it. Combined with the explicit rate-limit exemption for `CompactBlock` messages and an unbounded `pending_compact_blocks` map, an attacker can flood any reachable CKB node with zero mining work, causing unbounded memory growth and eventual node crash.

## Finding Description
**Root cause:** `contextual_check` in `compact_block_process.rs` (line 324) constructs a `HeaderVerifier` and calls `verify()`. This invokes `EpochVerifier::verify()` (lines 133–148 of `header_verifier.rs`), which only checks `is_well_formed()` and `is_successor_of()` on the epoch fraction — it holds only `parent: EpochNumberWithFraction` and has no access to the epoch's `compact_target`. The check that enforces `self.epoch.compact_target() == header.compact_target()` exists only in the contextual `EpochVerifier` inside `contextual_block_verifier.rs` (lines 500–507), which is reached only during full block acceptance, not during relay header verification.

**Exploit flow:**
1. Attacker connects as a P2P peer and fetches the current tip header (number N, hash H, epoch E).
2. Attacker crafts a `RawHeader` with `parent_hash=H`, `number=N+1`, `compact_target=0x20ffffff`, a valid epoch successor of E, a valid timestamp, and `nonce=0`.
3. `compact_to_target(0x20ffffff)` returns `(0xffffff << 232, false)`. The PoW guard `block_target.is_zero() || overflow` = `false || false` passes. The hash comparison `output > block_target` fails (i.e., PoW passes) for ~`(1 - 1/2^24)` ≈ 99.9999% of nonces.
4. `non_contextual_check` passes (valid uncle/proposal counts, height in range). `contextual_check` passes (parent known, `HeaderVerifier` passes including PoW). `CompactBlockVerifier::verify` passes.
5. The compact block includes short_ids not present in the target's tx pool, so `reconstruct_block` returns `ReconstructionResult::Missing`.
6. `missing_or_collided_post_process` inserts the compact block into `pending_compact_blocks` (line 354–361).
7. The `shrink_to_fit!` / epoch-based `retain` cleanup (lines 112–117) only executes on the `ReconstructionResult::Block` path — it never runs for the attacker's blocks.
8. Repeating with incrementing nonce/timestamp generates unique block hashes, each occupying a new map entry. The map grows without bound.

**Why existing guards fail:**
- Rate limiting: explicitly disabled for all `CompactBlock` messages (lines 112–114 of `relayer/mod.rs`) on the assumption that PoW is the cost barrier. With a self-declared minimum-difficulty target, this assumption is broken.
- `pending_compact_blocks` deduplication: only prevents the same `(block_hash, peer)` pair from being inserted twice; unique hashes bypass it entirely.
- Epoch-based eviction: only triggered when a legitimate block is accepted, which the attacker prevents by never providing resolvable short_ids.

## Impact Explanation
Unbounded memory growth in `pending_compact_blocks` causes the node process to exhaust available memory and crash. The `tokio::sync::Mutex` guarding the map is held during PoW verification of each incoming block, so a high-volume flood also causes async task starvation. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**. The attack is repeatable and requires no mining hardware, keys, or privileged access.

## Likelihood Explanation
Any party capable of establishing a P2P connection to a CKB node can execute this attack. The current tip is publicly available via normal sync. No hashpower is required — nonce=0 passes PoW with ~99.9999% probability. The rate-limit exemption removes the only practical throttle. The attack is fully automatable and can be sustained indefinitely.

## Recommendation
1. **Add `compact_target` validation to the relay-path header verifier:** In `contextual_check`, after `HeaderVerifier::verify` passes, derive the expected `compact_target` from the parent's epoch extension and reject headers where `header.compact_target() != expected_compact_target`. Alternatively, extend `HeaderVerifier` to accept and enforce the epoch's `compact_target`.
2. **Remove or scope the `CompactBlock` rate-limit exemption:** The exemption comment states "CompactBlock will be verified by POW" — this is only safe if PoW is enforced against the consensus difficulty. Until fix (1) is in place, apply the standard rate limiter to `CompactBlock` messages.
3. **Bound `pending_compact_blocks`:** Add a maximum capacity (e.g., 512 entries) with LRU or timestamp-based eviction, so the map cannot grow without bound regardless of attacker behavior.

## Proof of Concept
```
1. Connect to a CKB mainnet/testnet node as a P2P peer.
2. Fetch tip header T (number N, hash H, epoch E).
3. For i = 0, 1, 2, ...:
   a. Construct RawHeader:
        parent_hash    = H
        number         = N+1
        compact_target = 0x20ffffff
        epoch          = valid successor of E
        timestamp      = median_time(T) + 1 + i   // ensures unique hash
        nonce          = 0
   b. Wrap in CompactBlock with one prefilled cellbase tx and
      one short_id that does not exist in the target's tx pool.
   c. Send via RelayMessage::CompactBlock over P2P.
4. Assert: each message passes non_contextual_check and contextual_check
   (including HeaderVerifier/PoW), enters missing_or_collided_post_process,
   and is inserted into pending_compact_blocks with a unique key.
5. Monitor node RSS / heap: observe unbounded growth proportional to i.
6. Node OOM-crashes or becomes unresponsive as the map and associated
   tokio::sync::Mutex contention grow without bound.
```

pub fn compact_to_target(compact: u32) -> (U256, bool) {
    let exponent = compact >> 24;
    let mut mantissa = U256::from(compact & 0x00ff_ffff);

    let mut ret;
    if exponent <= 3 {
        mantissa >>= 8 * (3 - exponent);
        ret = mantissa.clone();
    } else {
        ret = mantissa.clone();
        ret <<= 8 * (exponent - 3);
    }

    let overflow = !mantissa.is_zero() && (exponent > 32);
    (ret, overflow)

pub struct EpochVerifier<'a> {
    parent: EpochNumberWithFraction,
    header: &'a HeaderView,
}

impl<'a> EpochVerifier<'a> {
    pub fn new(parent: EpochNumberWithFraction, header: &'a HeaderView) -> Self {
        EpochVerifier { parent, header }
    }

    pub fn verify(&self) -> Result<(), Error> {
        if !self.header.epoch().is_well_formed() {
            return Err(EpochError::Malformed {
                value: self.header.epoch(),
            }
            .into());
        }
        if !self.parent.is_genesis() && !self.header.epoch().is_successor_of(self.parent) {
            return Err(EpochError::NonContinuous {
                current: self.header.epoch(),
                parent: self.parent,
            }
            .into());
        }
        Ok(())
    }
