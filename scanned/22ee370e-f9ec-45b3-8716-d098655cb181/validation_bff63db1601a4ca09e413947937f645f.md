Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Spoofed `from` PeerId in `ConnectionRequestProcess::execute()` — (`network/src/protocols/hole_punching/mod.rs`, `network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

The `HolePunching` protocol's `forward_rate_limiter` is keyed by attacker-controlled `(content.from, content.to, item_id)` tuples parsed directly from message bytes, with no validation that `content.from` matches the actual sending peer. The outer `rate_limiter` permits 30 `ConnectionRequest` messages per second per session, each carrying a distinct spoofed `from` PeerId, creating 30 new `HashMapStateStore` entries per second. Because `retain_recent()` is never called during the connection lifetime — only on `disconnected()` — a long-lived or multi-session attacker causes unbounded memory growth, leading to OOM and node crash.

## Finding Description

**Outer rate limiter (Layer 1):** The `received()` handler gates on `(session_id, msg.item_id())` at 30 req/sec. [1](#0-0) 
`ConnectionRequest` has a fixed `item_id`, so exactly 30 messages per second per session pass this check.

**Inner rate limiter (Layer 2):** Inside `ConnectionRequestProcess::execute()`, the `forward_rate_limiter` is checked using the attacker-supplied `content.from` and `content.to` fields: [2](#0-1) 

**No sender validation:** `content.from` is parsed directly from message bytes with no check that it matches the actual sending session's peer ID: [3](#0-2) 

Each unique `from` value creates a new `HashMapStateStore<(PeerId, PeerId, u32)>` entry. Since the outer limiter allows 30 distinct messages per second, an attacker can create 30 new entries per second by varying `from` in each message.

**No periodic cleanup:** `retain_recent()` is called only in `disconnected()`: [4](#0-3) 

The `notify()` callback fires every `CHECK_INTERVAL = 5 minutes` but only prunes `pending_delivered` and `inflight_requests` — it never calls `retain_recent()` on either rate limiter: [5](#0-4) 

**Shared state:** `forward_rate_limiter` is a single instance on the `HolePunching` struct, shared across all sessions: [6](#0-5) 

Multiple concurrent attacker connections each contribute 30 entries/sec to the same HashMap. Governor's `retain_recent()` would remove entries whose 1-second quota has replenished, but since it is never invoked during the connection lifetime, all entries from time T remain resident until disconnect.

## Impact Explanation

This is a **High** severity vulnerability matching the allowed impact: **"Vulnerabilities which could easily crash a CKB node."**

Each `(PeerId, PeerId, u32)` entry in `HashMapStateStore` occupies approximately 150–200 bytes (two 39-byte multihash PeerIds + u32 + HashMap overhead + governor `AtomicU64` state). With 1 session at 30 entries/sec:

| Duration | Entries | Memory |
|---|---|---|
| 5 min | ~9,000 | ~1.8 MB |
| 1 hour | ~108,000 | ~21 MB |
| 8 hours | ~864,000 | ~173 MB |

With 50 concurrent attacker sessions: ~90 MB in 5 minutes, ~1 GB in 1 hour, ~8.6 GB in 8 hours — sufficient to exhaust typical node memory and cause OOM crash.

## Likelihood Explanation

- Only a standard P2P connection is required — no special privileges, no PoW, no key material.
- Generating 30 syntactically valid `PeerId` values per second is trivial; `PeerId::from_bytes` accepts any valid multihash bytes.
- The outer rate limiter is not a defense — it is precisely what guarantees 30 new `forward_rate_limiter` entries per second per session.
- CKB nodes accept inbound connections from the public internet, making this reachable from any unprivileged peer.
- The attack is repeatable and scales linearly with the number of concurrent attacker sessions.

## Recommendation

1. **Validate `from` against the actual sender**: In `ConnectionRequestProcess::execute()`, reject messages where `content.from` does not match the peer ID of the sending session. This eliminates the ability to create arbitrary new keys.
2. **Call `retain_recent()` periodically**: Inside the `notify()` callback (every `CHECK_INTERVAL`), call `retain_recent()` on both `rate_limiter` and `forward_rate_limiter`, not only on disconnect.
3. **Cap the `forward_rate_limiter` store size**: Add a maximum entry count and evict oldest entries when the cap is reached, as a defense-in-depth measure.

## Proof of Concept

```rust
// Setup: connect one attacker session
let mut protocol = HolePunching::new(network_state);
let session_id = PeerIndex::new(1);

// Attack: send 30 ConnectionRequest/sec for 300 seconds, each with unique `from`
for second in 0..300 {
    for i in 0..30 {
        let from = generate_unique_peer_id(second * 30 + i); // distinct each time
        let to   = generate_peer_id(0);
        let msg  = build_connection_request(from, to, max_hops=1);
        protocol.received(ctx_for(session_id), msg).await;
    }
    advance_clock_by(1_second);
    // NOTE: notify() fires every 5 min but does NOT call retain_recent()
}

// Assert: forward_rate_limiter has grown to ~9000 entries
// (retain_recent() was never called — no disconnect occurred)
assert!(
    forward_rate_limiter_len(&protocol) <= 30,  // FAILS: actual ~9000
    "invariant violated: forward_rate_limiter grew unboundedly"
);
```

The outer limiter passes all 30 messages/sec (quota = 30). Each unique `from` creates a new `HashMapStateStore` entry. After 300 seconds with no disconnect, `retain_recent()` has never been called, so all ~9,000 entries remain resident. With 50 concurrent attacker sessions, the store reaches ~450,000 entries (~90 MB) in 5 minutes and continues growing until OOM.
