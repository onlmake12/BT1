Audit Report

## Title
tx-pool `total_tx_size`/`total_tx_cycles` Inflated When Evictions Occur During `add_entry` - (File: tx-pool/src/component/pool_map.rs)

## Summary

In `add_entry`, pre-eviction totals are captured in local variables before `check_and_record_ancestors` is called. If that function evicts transactions via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, the correct decrements to `self.total_tx_size`/`self.total_tx_cycles` are immediately overwritten by the stale local snapshots at the end of `add_entry`. The pool's running totals are permanently inflated by the size and cycles of every evicted transaction, causing `limit_size` to evict valid pending transactions that should remain in the pool.

## Finding Description

`updated_stat_for_add_tx` (lines 711–728) reads `self.total_tx_size` and returns `self.total_tx_size + entry.size` as a local variable — it does **not** mutate `self`.

```
// line 210-211: snapshot taken BEFORE any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
```

`check_and_record_ancestors` (lines 588–640) may enter the eviction branch (lines 603–625) when `ancestors_count > max_ancestors_count` but `cell_ref_parents` can bring it under the limit. Each evicted entry is removed via `remove_entry_and_descendants` → `update_stat_for_remove_tx` (lines 733–758), which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in place.

```
// line 213: evictions happen here; self.total_tx_size is correctly decremented
evicts = self.check_and_record_ancestors(&mut entry)?;
```

At the end of `add_entry`, lines 218–219 unconditionally overwrite the now-correct `self.total_tx_size`/`self.total_tx_cycles` with the stale pre-eviction snapshots:

```
// lines 218-219: OVERWRITES post-eviction self.total_tx_size with pre-eviction value
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Every evicted entry's `size` and `cycles` are permanently lost from the accounting. Repeated submissions compound the inflation monotonically.

## Impact Explanation

`limit_size` (pool.rs lines 298–328) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts the lowest-fee-rate pending/gap/proposed transactions. With an inflated `total_tx_size`, this loop evicts valid transactions that are within the actual size budget. An attacker can use this to selectively drain the mempool of competing transactions at low cost, and repeated triggering compounds the inflation without bound until the pool is effectively emptied of legitimate transactions. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The attacker pays only the fees for their own crafted transactions; the cost to evict arbitrary victim transactions scales sub-linearly with the inflation accumulated.

## Likelihood Explanation

The eviction branch in `check_and_record_ancestors` is reachable by any unprivileged P2P peer or RPC caller. No special privileges are required. The default `max_ancestors_count` is 25, reachable with a crafted chain. The attacker needs only to: (1) build a transaction chain up to `max_ancestors_count - 1` depth where the last ancestor shares a cell dep with existing pool transactions, and (2) submit a new transaction spending that ancestor and referencing the same cell dep. This is a standard submission path. The attack is repeatable and each iteration compounds the inflation.

## Recommendation

Move the assignment of `self.total_tx_size` and `self.total_tx_cycles` to **before** the call to `check_and_record_ancestors`, so that subsequent eviction decrements are applied to the already-updated totals:

```rust
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;

evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Remove the duplicate assignments at lines 218-219
```

Alternatively, call `recompute_total_stat()` unconditionally at the end of `add_entry` (or conditionally when `!evicts.is_empty()`), mirroring the recovery path already present in `update_stat_for_remove_tx`.

## Proof of Concept

1. Submit transactions `D_1 … D_N` that all reference the same cell dep `C`. Record their combined size as `S_dep`.
2. Build a chain `A_1 → A_2 → … → A_{max-1}` where `A_{max-1}` also references cell dep `C` (making it a cell-ref parent of any future tx referencing `C`).
3. Submit transaction `B` spending an output of `A_{max-1}` and referencing `C`. Its ancestor count equals `max_ancestors_count`, triggering the eviction branch in `check_and_record_ancestors`.
4. `remove_entry_and_descendants(A_{max-1})` is called; `update_stat_for_remove_tx` decrements `self.total_tx_size` by `A_{max-1}.size`.
5. `add_entry` then overwrites `self.total_tx_size` with the pre-eviction snapshot `(old_total + B.size)`, losing the decrement.
6. Query `get_tx_pool_info` RPC and observe `total_tx_size` is inflated by `A_{max-1}.size`. Repeat steps 2–5 to compound the inflation until `limit_size` begins evicting unrelated valid transactions.
