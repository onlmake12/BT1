Audit Report

## Title
Missing Per-Peer Rate Limit in BlockFilter Protocol Enables DB Read Amplification and Bandwidth Exhaustion — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_hashes_process.rs`)

## Summary

The `BlockFilter` protocol handler contains no per-peer rate limiter. Any peer with a valid P2P connection can flood the node with `GetBlockFilterHashes`, `GetBlockFilterCheckPoints`, or `GetBlockFilters` messages. Each request triggers up to 4000 sequential RocksDB reads and produces a response of up to ~64 KB, with no throttle, no ban, and no backpressure at the application layer. This is a concrete asymmetric amplification path absent from every other comparable protocol handler in the codebase.

## Finding Description

`BlockFilter` struct holds only a `shared` field — no `rate_limiter`: [1](#0-0) 

`try_process` dispatches directly to all three handler variants with zero rate checks before any DB access: [2](#0-1) 

`GetBlockFilterHashesProcess::execute` iterates up to `BATCH_SIZE = 2000` blocks, calling `get_block_hash` and `get_block_filter_hash` for each — two RocksDB reads per iteration, up to 4000 per request: [3](#0-2) [4](#0-3) 

`GetBlockFilterCheckPointsProcess::execute` shares the same `BATCH_SIZE = 2000` and the same two-read-per-iteration pattern: [5](#0-4) [6](#0-5) 

`GetBlockFiltersProcess::execute` uses `BATCH_SIZE = 1000` and has a 1.8 MB per-response size cap, but still no rate limit: [7](#0-6) [8](#0-7) 

A grep for `rate_limiter` scoped to `sync/src/filter/` returns zero matches — the guard is entirely absent.

By contrast, `Relayer` carries an explicit `RateLimiter<(PeerIndex, u32)>` initialized at 30 req/sec: [9](#0-8) 

And checks it before every handler dispatch: [10](#0-9) 

`HolePunching` applies the same pattern at the `received` entry point: [11](#0-10) [12](#0-11) 

The broken assumption is that the Filter protocol, like all other request-response protocols in this node, should enforce per-peer rate limits before performing expensive DB work. It does not.

## Impact Explanation

Each `GetBlockFilterHashes(start_number=0)` request causes up to 4000 RocksDB reads and produces a ~64 KB response. With no application-level rate limit, a single peer can sustain this at whatever rate the network and async runtime allow. A handful of attacker connections can saturate the node's RocksDB read throughput and outbound bandwidth, causing severe performance degradation or node unresponsiveness. This matches the allowed CKB bounty impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, given the extreme asymmetry between attacker cost (8-byte message) and victim cost (4000 DB reads + 64 KB response).

## Likelihood Explanation

The attack requires only a valid P2P connection and a node that has built at least one block of filter data (the `latest >= start_number` guard at line 39 of `get_block_filter_hashes_process.rs` must hold). No proof-of-work, no keys, no special privileges are needed. The attacker's inbound cost is negligible (8 bytes per request); the node's outbound cost is large and bounded only by chain length. The attack is repeatable indefinitely with no ban or `TooManyRequests` response ever issued.

## Recommendation

Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer::rate_limiter`) and check it at the top of `try_process` before dispatching to any of the three filter handlers. The existing 30 req/sec quota used by `Relayer` and `HolePunching` is a reasonable starting point. Call `rate_limiter.retain_recent()` in the `disconnected` handler to prevent unbounded memory growth, as done in `Relayer`.

## Proof of Concept

1. Connect a single peer to a node that has built at least 1 block of filter data.
2. In a tight loop, send `GetBlockFilterHashes { start_number: 0 }` repeatedly over the Filter protocol.
3. Observe: the node performs up to 4000 RocksDB reads per message and sends ~64 KB responses, with no `TooManyRequests` status returned and no ban issued.
4. Assert: node DB read metrics spike proportionally to the request rate; no application-level throttle engages regardless of request frequency.
5. Scale to multiple attacker connections to demonstrate bandwidth and I/O saturation.
