Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Unique-`from` `ConnectionRequest` Flood — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
An unprivileged remote peer can exhaust heap memory on a victim CKB node by flooding it with `ConnectionRequest` messages carrying unique `from` PeerIds. All rate-limiting and deduplication guards are bypassed by this technique, causing unbounded insertions into `pending_delivered`, which has no size cap and is only evicted every 5 minutes.

## Finding Description
`HolePunching::pending_delivered` is declared as `HashMap<PeerId, PendingDeliveredInfo>` with no capacity bound. [1](#0-0) 

The insertion path proceeds as follows:

**Step 1 — Outer rate limiter (`received()`):** Keyed by `(session_id, msg.item_id())`, allowing 30 `ConnectionRequest` messages/sec per session. This does not limit unique-`from` flooding. [2](#0-1) 

**Step 2 — `forward_rate_limiter` in `execute()`:** Keyed by `(content.from, content.to, msg_item_id)`. With a unique `from` PeerId per message, every message gets its own fresh bucket — the limiter is fully bypassed. [3](#0-2) 

**Step 3 — Deduplication in `respond_delivered()`:** Only suppresses a reply if the **same** `from_peer_id` was seen within `HOLE_PUNCHING_INTERVAL`. With unique `from` PeerIds, this check never fires. [4](#0-3) 

**Step 4 — Unbounded insert:** After a successful `send_message_to`, every unique `from_peer_id` is inserted unconditionally with no size guard. [5](#0-4) 

**Step 5 — Eviction only in `notify()`:** `pending_delivered.retain(...)` runs every `CHECK_INTERVAL = 300 s`, retaining entries younger than `TIMEOUT = 300 s`. Between two `notify()` calls, no eviction occurs. [6](#0-5) 

The only additional guard is the `remote_listens.is_empty()` check, which is trivially bypassed by including one valid TCP/IPv4 address. [7](#0-6) 

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

At 30 msg/sec per session over a 300-second window: 9,000 entries inserted before the first eviction. Each entry stores up to `ADDRS_COUNT_LIMIT = 24` `Multiaddr` objects (~100 bytes each), yielding ~21 MB per session per 5-minute window. [8](#0-7) 

Multiple concurrent sessions multiply this linearly. Sustained over time, this causes heap exhaustion and OOM crash of the victim node.

## Likelihood Explanation
The attack requires only a single standard P2P connection. The victim's `PeerId` is publicly discoverable via the P2P network. No special privileges, keys, or hashpower are needed. The attacker constructs valid `ConnectionRequest` messages with random `from` PeerIds and one valid TCP/IPv4 listen address — both trivially achievable with standard tooling. The attack is repeatable and fully automated.

## Recommendation
1. **Cap `pending_delivered`**: Enforce a maximum size (e.g., 1,000 entries). On overflow, reject new insertions or evict the oldest entry.
2. **Rate-limit insertions per session**: Track how many `pending_delivered` entries originated from each session and cap that count.
3. **Shorten or decouple the eviction interval**: Run `pending_delivered` eviction more frequently than `CHECK_INTERVAL`, or evict eagerly on insertion when the map exceeds a threshold.
4. **Validate `from` PeerId against the sending session**: Reject `ConnectionRequest` messages where `from` does not match the session's authenticated peer identity, eliminating spoofed `from` PeerIds entirely.

## Proof of Concept
```
1. Connect to victim node V (obtain its PeerId P from the P2P discovery layer).
2. From a single session, send 30 ConnectionRequest messages per second:
     from = PeerId::random()          // unique each time
     to   = P                         // victim's local_peer_id
     listen_addrs = [/ip4/1.2.3.4/tcp/1234/p2p/<from>]  // valid TCP/IPv4 addr
     max_hops = 6
3. After 300 seconds (before the next notify() fires):
     assert pending_delivered.len() == 9000
     assert no eviction has occurred
4. Observe victim heap growth of ~21 MB and degraded HashMap performance.
5. Scale with additional concurrent sessions to accelerate OOM.
```
