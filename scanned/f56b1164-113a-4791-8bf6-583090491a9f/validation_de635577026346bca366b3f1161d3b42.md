### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any Pending Transaction from the Pool — (`File: rpc/src/module/pool.rs`)

### Summary
The `remove_transaction` RPC method removes any transaction and all its dependents from the tx-pool by hash, with no authentication or ownership check. Any RPC caller — explicitly listed as an in-scope attacker profile — can call it to evict a victim's pending transaction before it is mined, achieving targeted transaction censorship. The RPC server has no authentication middleware whatsoever, and the `Pool` module is enabled by default.

### Finding Description

**Root cause — no authentication on a destructive state-mutating RPC**

`remove_transaction` is registered as a plain public RPC method: [1](#0-0) 

Its implementation unconditionally forwards the caller-supplied hash to the tx-pool controller: [2](#0-1) 

The controller dispatches `RemoveLocalTx`, which removes the entry and all descendants: [3](#0-2) 

The RPC server (`rpc/src/server.rs`) starts an Axum HTTP listener with a `CorsLayer::permissive()` layer and **no authentication middleware**: [4](#0-3) 

There is no token, session, or credential check anywhere in the request path. The `Pool` module — which includes `remove_transaction` — is enabled by default in the production config: [5](#0-4) 

The only protection is the default bind address of `127.0.0.1:8114`: [6](#0-5) 

However, the RPC config type has no `auth_token` or equivalent field: [7](#0-6) 

**Exploit flow**

1. Victim submits transaction `T` via `send_transaction`; `T` enters the pending pool.
2. Attacker (any process that can reach the RPC port — local or remote if operator exposed it) observes `T`'s hash via `get_raw_tx_pool`.
3. Attacker calls `remove_transaction(T.hash)` — no credential required.
4. `T` and all its descendants are evicted from the pool.
5. `T` is never mined; the victim's operation is permanently blocked until they resubmit (and the attacker can repeat indefinitely).

The same pattern applies to `clear_tx_pool` (evicts every pending transaction) and `clear_tx_verify_queue` (drains the verification queue), both equally unauthenticated: [8](#0-7) 

### Impact Explanation

**High.** An attacker with RPC access can:
- Silently evict any specific pending transaction (targeted censorship), preventing it from ever being committed to a block.
- Repeatedly evict resubmissions, making the victim's transaction permanently unmineable as long as the attacker monitors the pool.
- Call `clear_tx_pool` to wipe all pending transactions simultaneously, disrupting every user of the node.

This directly maps to the original report's impact: "the user won't be able to recover access to his wallet." Here, the user's transaction is permanently blocked from inclusion.

### Likelihood Explanation

**Medium-High.** The RPC is localhost-only by default, but:
- Many operators expose the RPC port for wallets, dApps, or monitoring (the documentation warns against it but does not enforce restriction).
- Any co-located process (another service, a compromised dependency, a web app with SSRF) can reach `127.0.0.1:8114` without any credential.
- The attack requires only a single HTTP POST with the victim's tx hash, which is publicly observable from `get_raw_tx_pool`.
- The `RPC caller` attacker profile is explicitly listed as in-scope.

### Recommendation

Add an optional authentication token to the RPC config (e.g., a `rpc_secret_token` field). The server should reject any request that does not supply the correct `Authorization: Bearer <token>` header. Destructive state-mutating methods (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`, `clear_banned_addresses`) should require this token even when the RPC is localhost-bound, because localhost is not a trust boundary for multi-tenant or containerized deployments.

Alternatively, split the RPC into a read-only public surface and a privileged admin surface bound to a separate port or Unix socket, with the admin surface requiring authentication.

### Proof of Concept

```bash
# Step 1: victim submits a transaction
TX_HASH=$(curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<TX_JSON>,"passthrough"]}' \
  | jq -r '.result')

# Step 2: attacker (no credentials needed) removes it immediately
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d "{\"id\":2,\"jsonrpc\":\"2.0\",\"method\":\"remove_transaction\",\"params\":[\"$TX_HASH\"]}"
# Returns: {"result": true}

# Step 3: victim's transaction is gone from the pool and will never be mined
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d "{\"id\":3,\"jsonrpc\":\"2.0\",\"method\":\"get_transaction\",\"params\":[\"$TX_HASH\"]}"
# Returns status: "rejected" or unknown
```

The attacker can automate this in a loop to permanently censor any transaction the victim attempts to submit, with zero credentials and a single HTTP call per eviction.
