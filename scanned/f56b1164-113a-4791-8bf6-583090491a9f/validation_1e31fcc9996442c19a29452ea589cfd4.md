### Title
Incorrect Strict Boundary Check in Block Assembler Silently Drops Valid Proposals — (File: `tx-pool/src/block_assembler/mod.rs`)

---

### Summary

The `update_proposals` (and `update_uncles`) function in the block assembler uses a strict `<` comparison against `max_block_bytes`, while the consensus `BlockBytesVerifier` accepts blocks where `block_bytes <= block_bytes_limit`. This off-by-one mismatch causes the block assembler to silently skip a valid proposals update whenever the computed total block size exactly equals the consensus byte limit, producing a suboptimal block template for the miner.

---

### Finding Description

In `tx-pool/src/block_assembler/mod.rs`, `update_proposals` computes the new total block size after incorporating fresh proposals and then guards the update with:

```rust
// tx-pool/src/block_assembler/mod.rs  line 391
if new_total_size < max_block_bytes {
    // ... apply proposals to template
}
``` [1](#0-0) 

The same strict `<` guard appears in `update_uncles`:

```rust
// tx-pool/src/block_assembler/mod.rs  line 350
if new_total_size < max_block_bytes {
    // ... apply uncles to template
}
``` [2](#0-1) 

However, the authoritative consensus rule in `BlockBytesVerifier::verify` accepts any block whose serialized size is **less than or equal to** the limit:

```rust
// verification/src/block_verifier.rs  line 257
if block_bytes <= self.block_bytes_limit {
    Ok(())
} else {
    Err(BlockErrorKind::ExceededMaximumBlockBytes.into())
}
``` [3](#0-2) 

When `new_total_size == max_block_bytes`, the block assembler's guard evaluates to `false` and the proposals (or uncles) update is silently skipped — even though the resulting block would pass consensus validation. The miner's block template is left with the previous (smaller) set of proposals.

---

### Impact Explanation

**Impact: Low**

A miner calling `get_block_template` receives a block template that omits a valid batch of proposals when the computed total size lands exactly on the byte limit. The miner loses the associated proposer reward for those proposals. The block itself is never invalid; only the template is suboptimal. No funds are at risk and no consensus split occurs.

---

### Likelihood Explanation

**Likelihood: Low**

The condition `new_total_size == max_block_bytes` requires the sum of the serialized cellbase, uncles, existing transactions, and the new proposals to land on an exact byte boundary. `ProposalShortId` is 10 bytes, so the alignment is possible but not common under normal mempool activity. The probability increases when the mempool is near-full and proposals are being added or removed frequently.

---

### Recommendation

Change the strict `<` to `<=` in both `update_proposals` and `update_uncles` to match the consensus verifier's accepted boundary:

```rust
// update_proposals  (line 391)
- if new_total_size < max_block_bytes {
+ if new_total_size <= max_block_bytes {

// update_uncles  (line 350)
- if new_total_size < max_block_bytes {
+ if new_total_size <= max_block_bytes {
``` [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. The consensus byte limit is `max_block_bytes` (e.g., 597,000 bytes on mainnet).
2. A miner's node has a mempool with enough pending transactions that the current block template's `total` size is, say, `max_block_bytes - 10` bytes.
3. One new `ProposalShortId` (10 bytes) arrives in the mempool.
4. `update_proposals` is triggered. It computes:
   - `new_proposals_size` = (old count + 1) × 10
   - `new_total_size` = `max_block_bytes - 10 + 10` = `max_block_bytes`
5. The guard `new_total_size < max_block_bytes` evaluates to `false`.
6. The proposals update is silently skipped; the new proposal is not included in the template.
7. `BlockBytesVerifier` would have accepted a block of exactly `max_block_bytes` bytes, so the skip is incorrect.
8. The miner mines the block without the proposal and forfeits the proposer reward for that transaction. [6](#0-5) [7](#0-6)
