### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth for Persistent Connections — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

`HolePunching::forward_rate_limiter` uses `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`. The `governor` crate only reclaims expired entries when `retain_recent()` is explicitly called. In this implementation, `retain_recent()` is called **only** in `disconnected()`. The periodic `notify()` callback (fired every 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter. An unprivileged remote peer that maintains a persistent connection and sends messages with unique attacker-controlled `(from, to)` PeerIds causes the `HashMapStateStore` to grow monotonically with no reclamation, bounded only by the outer `rate_limiter`'s 30 req/sec cap.

---

### Finding Description

`forward_rate_limiter` is declared as:

```rust
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>
``` [1](#0-0) 

The `retain_recent()` cleanup is wired only to the disconnect event:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    ...
}
``` [2](#0-1) 

The `notify()` callback fires every `CHECK_INTERVAL` (5 minutes) and cleans `pending_delivered` and `inflight_requests`, but **does not** call `retain_recent()` on either rate limiter: [3](#0-2) 

The `forward_rate_limiter` key is `(content.from, content.to, msg_item_id)`, where `from` and `to` are PeerIds parsed directly from the incoming message payload — fully attacker-controlled: [4](#0-3) 

The same pattern applies to `ConnectionRequest` and `ConnectionRequestDelivered` processors. [5](#0-4) 

---

### Impact Explanation

An attacker connects once and sends HolePunching messages (any of the three types) with a fresh unique `(from, to)` PeerId pair per message. Each unique pair inserts a new entry into the `HashMapStateStore`. The outer `rate_limiter` (keyed by `(PeerIndex, msg_type)`) caps throughput at 30 messages/sec per session, so the growth rate is bounded at **~30 new entries/sec per persistent session**.

Each entry holds two `PeerId` values (~39 bytes each) plus governor's internal state, totalling roughly 150–200 bytes per entry:

| Session duration | Entries | Approx. memory |
|---|---|---|
| 10 minutes | 18,000 | ~3.6 MB |
| 1 hour | 108,000 | ~21 MB |
| 24 hours | 2,592,000 | ~500 MB |

With multiple concurrent attacker sessions the impact scales linearly. There is no cap, no eviction, and no periodic cleanup path that would bound this growth.

---

### Likelihood Explanation

The HolePunching protocol is an opt-in feature enabled via `support_protocols` config. [6](#0-5) 

When enabled, any unprivileged peer that can establish a P2P connection can exploit this. No authentication, PoW, or privileged role is required. The attacker only needs to maintain a TCP session and send valid-but-unique HolePunching messages at the permitted rate. The `from`/`to` PeerIds in the message body are not validated against the actual session identity, so the attacker can freely generate arbitrary unique pairs.

---

### Recommendation

Add `retain_recent()` calls inside the `notify()` callback so expired entries are reclaimed periodically regardless of session lifetime:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();           // add
    self.forward_rate_limiter.retain_recent();   // add
    // ... existing cleanup ...
}
``` [3](#0-2) 

---

### Proof of Concept

1. Connect to a CKB node with HolePunching enabled.
2. In a loop, send `ConnectionSync` messages where `from` and `to` are freshly generated random PeerIds (valid multihash bytes), at 30 messages/sec (within the outer rate limit).
3. Never disconnect.
4. After N seconds, assert that the internal map of `forward_rate_limiter` contains N×30 entries with zero reclamation.
5. After 10 minutes: ~18,000 entries; after 1 hour: ~108,000 entries — all stale but never freed.
