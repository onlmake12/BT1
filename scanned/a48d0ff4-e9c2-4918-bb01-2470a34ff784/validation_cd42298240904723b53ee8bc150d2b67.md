### Title
Unsafe `u128 as u64` Cast in DAO Withdrawal Calculation Silently Truncates Capacity — (File: `util/dao/src/lib.rs`)

---

### Summary

The `calculate_maximum_withdraw` function in `util/dao/src/lib.rs` performs an unsafe `u128 as u64` truncating cast on the computed withdrawal capacity. If the intermediate `u128` result exceeds `u64::MAX`, the value silently wraps to a much smaller number, causing DAO withdrawal transactions to be rejected and effectively locking depositors' interest. This is directly analogous to the H-05 pattern: a wider integer is computed correctly in a wider type, then silently narrowed without a bounds check.

---

### Finding Description

In `DaoCalculator::calculate_maximum_withdraw`, the interest-bearing portion of a DAO withdrawal is computed using `u128` arithmetic to avoid intermediate overflow:

```rust
// util/dao/src/lib.rs  lines 152–156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← unsafe truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits if `withdraw_counted_capacity > u64::MAX`. The resulting `withdraw_capacity` is then returned as the authoritative maximum a depositor may withdraw.

This is a direct inconsistency with every other analogous `u128 → u64` narrowing in the same file, all of which use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

where `counted_capacity = output_capacity − occupied_capacity`, `deposit_ar` is the accumulate-rate at deposit time (genesis value: `10^16`), and `withdrawing_ar` is the accumulate-rate at withdrawal time. [5](#0-4) 

---

### Impact Explanation

`calculate_maximum_withdraw` is called in two critical paths:

1. **`transaction_fee`** — used by the tx-pool to validate DAO withdrawal transactions:

```rust
// util/dao/src/lib.rs  lines 30–36
pub fn transaction_fee(&self, rtx: &ResolvedTransaction) -> Result<Capacity, DaoError> {
    let maximum_withdraw = self.transaction_maximum_withdraw(rtx)?;
    rtx.transaction
        .outputs_capacity()
        .and_then(|y| maximum_withdraw.safe_sub(y))
        .map_err(Into::into)
}
``` [6](#0-5) 

If `withdraw_counted_capacity` overflows and truncates to a small value, `maximum_withdraw` becomes far smaller than the depositor's actual entitlement. `safe_sub` then returns an underflow error, causing the withdrawal transaction to be **permanently rejected** by every node. The depositor's principal and accrued interest remain locked in the DAO cell with no recovery path.

2. **`withdrawed_interests` → `dao_field_with_current_epoch` → `DaoHeaderVerifier::verify`** — the same truncated value propagates into the block-level DAO field calculation, producing an incorrect `s` (secondary issuance) field that all nodes agree on (no fork), but which permanently corrupts the DAO accounting state. [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

The overflow threshold for a cell with `counted_capacity = C` shannons is:

```
withdrawing_ar / deposit_ar  >  u64::MAX / C
```

With `deposit_ar = 10^16` and the total CKB supply ≈ `3.36 × 10^18` shannons, a single cell holding the entire supply would overflow when `withdrawing_ar / deposit_ar > 5.47`. The AR grows at roughly `g2 / parent_c ≈ 10^{-8}` per block; reaching a 5.47× ratio takes on the order of `~10^8` blocks (≈ 100+ years at 8 s/block). For smaller, more realistic cell sizes the threshold is proportionally higher. Likelihood is therefore **Low** — not triggerable by an attacker today, but a latent time-bomb inconsistent with the safe patterns used everywhere else in the same file.

---

### Recommendation

Replace the unsafe cast with the same checked pattern used in the three adjacent calculations:

```rust
let withdraw_capacity = Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?;
```

This makes overflow an explicit, recoverable error rather than a silent data corruption.

---

### Proof of Concept

1. A depositor locks a very large DAO cell (e.g., close to the total CKB supply) at epoch 0 when `deposit_ar = 10^16`.
2. After sufficient time, `withdrawing_ar` grows so that `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`.
3. The depositor submits a withdrawal transaction claiming the correct amount.
4. Every node calls `calculate_maximum_withdraw`; `withdraw_counted_capacity as u64` silently truncates to a small value.
5. `transaction_fee` computes `maximum_withdraw.safe_sub(outputs_capacity)` → underflow error.
6. The withdrawal transaction is rejected by every node in the network.
7. The depositor's funds are permanently inaccessible.
