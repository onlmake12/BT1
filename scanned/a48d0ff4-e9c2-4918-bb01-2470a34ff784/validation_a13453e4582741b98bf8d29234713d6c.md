### Title
Duplicate Header Hashes Accumulate in `unknown_header_list` Without Deduplication — (`File: sync/src/types/mod.rs`)

---

### Summary

`insert_unknown_header_hash` appends peer-supplied header hashes to a `Vec<Byte32>` with no duplicate check. A sync peer that causes the block fetcher to re-insert the same unknown header hash across multiple scheduling cycles will silently inflate the per-peer `unknown_header_list`, producing incorrect sync-state accounting and unbounded memory growth.

---

### Finding Description

`Peers::insert_unknown_header_hash` unconditionally calls `Vec::push` on `state.unknown_header_list`:

```rust
pub fn insert_unknown_header_hash(&self, peer: PeerIndex, hash: Byte32) {
    self.state
        .entry(peer)
        .and_modify(|state| state.unknown_header_list.push(hash));
}
``` [1](#0-0) 

There is no `contains` check, no `HashSet` guard, and no capacity cap on `unknown_header_list` before the push. The field is a plain `Vec<Byte32>`: [2](#0-1) 

`take_unknown_last` simply pops from the back of this Vec, so every duplicate entry is consumed as a separate work item:

```rust
pub fn take_unknown_last(&self, peer: PeerIndex) -> Option<Byte32> {
    self.state
        .get_mut(&peer)
        .and_then(|mut state| state.unknown_header_list.pop())
}
```

The function is called from `sync/src/synchronizer/block_fetcher.rs` (three call sites confirmed by grep). The block fetcher runs on a timer and, for each scheduling cycle, may re-insert the same hash if the peer's `best_known_header` still points to an unresolved header. Because there is no deduplication guard analogous to the `isNodeDelegator` mapping recommended in the reference report, every scheduling cycle appends another copy.

The gate function `get_best_known_less_than_tip_and_unknown_empty` excludes a peer from further block-fetch scheduling only while `unknown_header_list` is non-empty:

```rust
if !state.unknown_header_list.is_empty() {
    return None;
}
``` [3](#0-2) 

With duplicates present, the list drains more slowly than it should (one pop per cycle), keeping the peer excluded from normal block-fetch scheduling for longer than the actual number of distinct unknown headers warrants. Meanwhile `clear_unknown_list` is the only bulk-reset path: [4](#0-3) 

---

### Impact Explanation

1. **Incorrect sync-state accounting.** The `unknown_header_list` length is used as a proxy for "how many headers this peer still needs to resolve." Duplicates inflate that count, causing the node to issue redundant `GetHeaders` round-trips for the same hash and to incorrectly exclude the peer from block-fetch scheduling for extra cycles.

2. **Unbounded per-peer memory growth.** Each `Byte32` is 32 bytes. If the block fetcher re-inserts the same hash on every scheduling tick before `clear_unknown_list` fires, the Vec grows without bound. A single malicious peer that keeps its `best_known_header` pointing at an unknown tip can drive this growth continuously, eventually exhausting heap memory and crashing the node (OOM / SIGKILL).

---

### Likelihood Explanation

The block fetcher is invoked on a recurring timer for every connected sync peer. Any outbound or whitelisted peer that advertises a `best_known_header` the local node has not yet resolved will trigger repeated calls to `insert_unknown_header_hash` with the same hash on every scheduling cycle until the header is fetched and resolved. This is a normal operating condition during initial block download, making accidental triggering likely and deliberate triggering trivial for any peer that controls what header it advertises.

---

### Recommendation

Replace `Vec<Byte32>` with a `HashSet<Byte32>` (or maintain a parallel `HashSet` as a membership guard) for `unknown_header_list`, mirroring the `isNodeDelegator` mapping fix described in the reference report. Before inserting, check membership:

```rust
pub fn insert_unknown_header_hash(&self, peer: PeerIndex, hash: Byte32) {
    self.state
        .entry(peer)
        .and_modify(|state| {
            if !state.unknown_header_set.contains(&hash) {
                state.unknown_header_set.insert(hash.clone());
                state.unknown_header_list.push(hash);
            }
        });
}
```

Alternatively, enforce a maximum list length per peer to bound memory growth independently of deduplication.

---

### Proof of Concept

1. Connect a peer to a CKB node that is in IBD.
2. Have the peer advertise a `best_known_header` whose hash the local node does not recognise.
3. The block fetcher fires on its scheduling interval and calls `insert_unknown_header_hash(peer, unknown_hash)`.
4. Because the hash is never resolved (the peer does not respond to `GetHeaders` for it), the block fetcher fires again on the next tick and calls `insert_unknown_header_hash(peer, unknown_hash)` a second time — no duplicate check prevents this.
5. After N scheduling ticks, `unknown_header_list` contains N copies of the same hash, consuming `N × 32` bytes of heap.
6. Calling `take_unknown_last` N times returns the same hash N times, causing N redundant `GetHeaders` messages to be sent to the peer — double-counting the work exactly as the `assetLyingInNDCs` calculation double-counted assets in the reference report. [1](#0-0) [2](#0-1) [5](#0-4)
