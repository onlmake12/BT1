Audit Report

## Title
Off-by-One Guard Allows Maximum-Work `GetLastStateProof` Requests, Enabling Sustained DoS ‚Äî (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The size guard in `GetLastStateProofProcess::execute` uses a strict `>` comparison against `GET_LAST_STATE_PROOF_LIMIT` (1000), so a request with exactly 1000 difficulties and `last_n_blocks=0` evaluates to `1000 > 1000 = false` and passes unchecked. Each such request forces up to 1000 binary-search DB reads, 1000 MMR `get_root()` calls, and one `gen_proof(1000 positions)` call. Because the request succeeds with `StatusCode::OK`, the peer is never banned, enabling any unauthenticated peer to repeat this indefinitely.

## Finding Description

**Off-by-one guard:** The check at `get_last_state_proof.rs` lines 201‚Äì204 reads:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
``` [1](#0-0) 

`GET_LAST_STATE_PROOF_LIMIT` is 1000: [2](#0-1) 

With `difficulties.len() = 1000` and `last_n_blocks = 0`, the expression is `1000 + 0 = 1000`, which is **not** `> 1000`. The guard does not fire and `MalformedProtocolMessage` (the only ban-triggering path here) is never returned.

**Work per passing request:**

1. `get_block_numbers_via_difficulties` iterates all 1000 difficulties; each calls `get_first_block_total_difficulty_is_not_less_than`, which performs a binary search doing O(log N) DB reads per difficulty: [3](#0-2) 

2. `complete_headers` calls `snapshot.chain_root_mmr(*number - 1).get_root()` for each of the up to 1000 block numbers: [4](#0-3) 

3. `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1000 positions: [5](#0-4) 

**No ban for successful requests:** `should_ban()` only returns `Some(ban_time)` for 4xx status codes: [6](#0-5) 

A request that passes the guard and completes normally returns `StatusCode::OK` (200), so `should_ban()` returns `None` and the peer is never penalized regardless of how many such requests it sends: [7](#0-6) 

## Impact Explanation

For a chain of N blocks, a single request forces 1000 √ó O(log N) DB reads, 1000 MMR `get_root()` computations, and one O(N_positions √ó log N) `gen_proof` call. An attacker sending concurrent such requests can saturate disk I/O and CPU, starving legitimate peers of responses. This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **High: Vulnerabilities which could easily crash a CKB node** under sustained load.

## Likelihood Explanation

The attack requires only a valid P2P connection. The message structure is part of the documented light client protocol. The chain's total difficulty range is publicly observable from any connected peer. No proof-of-work, key, or privileged role is required. The attacker can repeat indefinitely with no ban or back-pressure applied.

## Recommendation

1. Fix the guard to use `>=` instead of `>`:
   ```rust
   if self.message.difficulties().len() + (last_n_blocks as usize) * 2
       >= constant::GET_LAST_STATE_PROOF_LIMIT
   ```
2. Add per-peer request rate limiting for `GetLastStateProof` messages in the `received` handler.
3. Consider banning peers that repeatedly trigger `InvalidRequest` or `InvaildDifficultyBoundary` status codes, not only `MalformedProtocolMessage`.

## Proof of Concept

1. Spin up a CKB node with a chain of 1,000,000 blocks and enable the light client protocol server.
2. Connect as a light-client peer.
3. Send 10 concurrent `GetLastStateProof` messages, each with:
   - `difficulties` = 1000 distinct, monotonically increasing values within the chain's total difficulty range
   - `last_n_blocks = 0`
   - `difficulty_boundary` set above the highest difficulty value
   - `last_hash` = current tip hash
4. Observe: each request passes the guard (`1000 > 1000` is false), triggers ~1000 binary-search DB reads + 1000 MMR root computations + 1 gen_proof call, and returns `OK` with no ban issued.
5. Repeat indefinitely ‚Äî no ban is issued, no rate limit applies. Measure degradation in response latency for other peers.
