Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the new pool-size totals are snapshotted before `check_and_record_ancestors` runs. When that function evicts transactions via `remove_entry_and_descendants`, it correctly decrements `self.total_tx_size`/`self.total_tx_cycles` through `update_stat_for_remove_tx`. However, lines 218–219 then unconditionally overwrite those fields with the stale pre-eviction snapshot, permanently inflating the counters by the cumulative size/cycles of every evicted transaction. The pool never self-corrects because `recompute_total_stat` is only invoked on underflow.

## Finding Description
**Root cause:** `updated_stat_for_add_tx` (lines 210–211) reads `self.total_tx_size` and returns `self.total_tx_size + entry.size` as a local. `check_and_record_ancestors` (line 213) may then call `remove_entry_and_descendants` for each cell-dep ancestor that pushes the count over `max_ancestors_count` (lines 615–625). Each such removal calls `update_stat_for_remove_tx`, which writes the decremented value back to `self.total_tx_size` (lines 738–740). Finally, lines 218–219 blindly assign the stale local snapshot to `self.total_tx_size`, discarding all eviction-driven decrements.

**Eviction path reachability:** The eviction branch at line 603 fires when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, i.e., the excess ancestors are reachable only via cell-dep references. This is a documented, intentional feature path, not a corner case.

**Why existing guards fail:** `update_stat_for_remove_tx` correctly handles the decrement, but its result is immediately clobbered. `recompute_total_stat` is only called on arithmetic underflow, so inflation is never detected or corrected.

## Impact Explanation
`total_tx_size` is the authoritative pool-fullness counter checked on every submission via `updated_stat_for_add_tx`, which returns `Reject::Full` when the counter would overflow the configured limit. An attacker who repeatedly triggers the eviction path inflates `total_tx_size` by the size of each evicted transaction per round. Once the inflated counter exceeds `max_tx_pool_size`, every subsequent transaction submission is rejected with `Reject::Full` regardless of actual pool occupancy. The node's mempool becomes permanently non-functional until restarted. This matches: **High — Vulnerabilities which could easily crash a CKB node** (the mempool subsystem is rendered inoperable). The counter is also exposed directly via RPC `tx_pool_info`, misleading operators and downstream tooling.

## Likelihood Explanation
An unprivileged attacker with a modest UTXO set can trigger this via standard RPC or P2P relay. The required setup — a chain of up to `max_ancestors_count − 1` pending transactions where one output is used as a cell dep by a later transaction — is achievable with normal transaction construction. No privileged access, majority hashpower, or Sybil capability is required. Each inflation round costs only the transaction fees for the submitted chain. The attack is repeatable with fresh UTXOs, making it a low-cost, high-impact DoS.

## Recommendation
Move the stat computation to **after** `check_and_record_ancestors` returns, so eviction decrements are already reflected in `self.total_tx_size`/`self.total_tx_cycles` before the addition is applied:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER evictions have already updated self.total_tx_*
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Note that moving the `updated_stat_for_add_tx` call also means the `Reject::Full` check now reflects the post-eviction pool size, which is the correct and more permissive behavior.

## Proof of Concept
**Configuration:** `max_ancestors_count = 25`, `max_tx_pool_size = 1_000_000` bytes, empty pool.

1. Submit `tx_root` producing output `O` (~200 bytes). Pool: 1 entry, `total_tx_size = 200`.
2. Submit chain `tx_1 → tx_2 → … → tx_24` each spending the previous output (~200 bytes each). Pool: 25 entries, `total_tx_size ≈ 5000`.
3. Submit `tx_attack` (~200 bytes) that:
   - Spends `tx_24`'s output (input-ancestor chain: `tx_root … tx_24`, count = 25)
   - Declares `tx_root`'s output `O` as a **cell dep** (making `tx_root` a `cell_ref_parent`)
   - Total ancestor count = 26 > 25; `cell_ref_parents = {tx_root}`; eviction fires.

**Inside `add_entry`:**
```
// Line 210-211: snapshot BEFORE eviction
local total_tx_size = 5000 + 200 = 5200

// Line 213 → remove_entry_and_descendants(tx_root) → update_stat_for_remove_tx(200, ...)
self.total_tx_size = 5000 − 200 = 4800   ← correct

// Lines 218-219: stale overwrite
self.total_tx_size = 5200               ← inflated by 200 (tx_root's ghost size)
```

4. Actual pool: 25 entries (~5000 bytes). Reported `total_tx_size`: **5200**.
5. Repeat step 3 with fresh UTXOs. Each iteration adds ~200 bytes of phantom size.
6. After ~`max_tx_pool_size / 200 = 5000` iterations, `total_tx_size` exceeds `max_tx_pool_size`; all subsequent submissions return `Reject::Full`. Pool is DoS'd.
