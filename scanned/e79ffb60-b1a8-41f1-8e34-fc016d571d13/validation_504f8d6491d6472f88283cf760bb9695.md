Audit Report

## Title
Integer overflow in `GET_LAST_STATE_PROOF_LIMIT` guard bypasses work bound, enabling O(chain_length) DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The guard at line 201 of `GetLastStateProofProcess::execute` uses plain Rust multiplication `(last_n_blocks as usize) * 2`, which wraps in release mode. An attacker supplying `last_n_blocks = 0x8000000000000001u64` causes the product to wrap to `2`, bypassing the `> 1000` limit check. Downstream, the astronomically large `last_n_blocks` value causes the server to collect every block number in the chain into a `Vec` and perform O(chain_length) DB reads and MMR root computations per request.

## Finding Description
**Root cause — line 201:** [1](#0-0) 

`last_n_blocks` is a `u64` read directly from the peer message. On a 64-bit target, `last_n_blocks as usize` is lossless. The subsequent `* 2` is plain Rust multiplication. In release builds Rust wraps on overflow (no panic, no `checked_mul`/`saturating_mul`).

With `last_n_blocks = 0x8000000000000001u64`:
- `0x8000000000000001usize * 2 = 0x0000000000000002usize` (wraps)
- Guard: `0 + 2 > 1000` → **false** → execution continues

**Downstream impact — lines 291–296:** [2](#0-1) 

Because `last_n_blocks ≈ 9.2×10¹⁸`, the condition `last_block_number - start_block_number <= last_n_blocks` is always true for any real chain. The server therefore collects every block number from `start_block_number` to `last_block_number` into `last_n_numbers`, then passes it to `complete_headers`: [3](#0-2) 

`complete_headers` performs a DB ancestor lookup, block fetch, uncle hash computation, and MMR root computation for **every entry**: [4](#0-3) 

The constant being bypassed: [5](#0-4) 

**Existing guards are insufficient:**
- The `is_main_chain` check (line 210) only requires a valid tip hash, which is public.
- The `start_block_number > last_block_number` check (line 231) is satisfied by setting `start_block_number = 0`.
- The difficulties validation block (lines 252–288) is skipped entirely when the attacker sends an empty difficulties list.
- The `reorg_last_n_numbers` path (line 237) returns `Vec::new()` when `start_block_number == 0`, so it adds no additional work.

## Impact Explanation
A single unauthenticated P2P peer can send one crafted `GetLastStateProof` message and force the light-client-serving node to allocate a `Vec<BlockNumber>` proportional to the full chain length, then perform O(chain_length) DB reads and MMR computations. On a chain with millions of blocks this causes severe CPU and memory pressure. Repeated requests cause sustained degradation or OOM crash of the node. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node** and **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
Any peer on the P2P network can send this message with no authentication, PoW, or stake required. The crafted value is a single field in the flatbuffers message. The overflow is deterministic and reproducible on any 64-bit release build. The attack is repeatable at will.

## Recommendation
Replace the plain multiplication with overflow-safe arithmetic:

```rust
if self.message.difficulties().len()
    .saturating_add((last_n_blocks as usize).saturating_mul(2))
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, reject any `last_n_blocks > GET_LAST_STATE_PROOF_LIMIT as u64` before the cast, which is simpler and avoids the type-width dependency entirely.

## Proof of Concept
```rust
// Minimal unit test demonstrating guard bypass in release mode
#[test]
fn test_overflow_bypass() {
    let last_n_blocks: u64 = 0x8000000000000001u64;
    // Replicates the guard expression exactly as written in execute()
    let guard_value = (last_n_blocks as usize).wrapping_mul(2); // = 2
    assert_eq!(guard_value, 2);
    // Guard fails to reject: 0 + 2 > 1000 is false
    assert!(0usize + guard_value <= 1000);
    // Downstream: last_block_number - start_block_number <= last_n_blocks
    // is always true for any real chain, so all block numbers are collected.
}
```

Manual steps:
1. Connect a peer to a CKB light-client node built in release mode.
2. Send a `GetLastStateProof` message with: `last_n_blocks = 0x8000000000000001`, `start_number = 0`, `last_hash` = current tip hash (public), empty `difficulties` list.
3. Observe the node allocating a `Vec` of all block numbers and performing O(chain_length) DB reads and MMR computations.
