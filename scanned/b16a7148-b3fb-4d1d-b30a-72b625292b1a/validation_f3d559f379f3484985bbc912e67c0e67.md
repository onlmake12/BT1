### Title
Minimum Fee Rate Check Uses Serialized Size Instead of Actual Weight, Allowing Low-Fee-Rate High-Cycle Transactions to Bypass Pool Admission — (`tx-pool/src/util.rs`)

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` computes the minimum required fee using the transaction's serialized byte size as a proxy for its weight. However, CKB's actual weight formula is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For transactions with high cycle consumption but small serialized size, the actual weight can be orders of magnitude larger than the size. This means the size-based check can be satisfied with a fee that corresponds to a true fee rate far below `min_fee_rate`, allowing an unprivileged transaction submitter to bypass the minimum fee rate gate.

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee rate as follows:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The code comment explicitly acknowledges the inaccuracy. The actual weight used everywhere else in the system is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

The `check_tx_fee` call occurs before `verify_rtx` (which determines actual cycles through script execution). After `verify_rtx` returns the real cycle count, the `TxEntry` is constructed with actual cycles, but no second fee-rate check against the true weight is performed. [4](#0-3) 

**Concrete example:**
- `tx_size` = 200 bytes, `cycles` = 5,000,000
- Actual weight = `max(200, 5_000_000 × 0.000_170_571_4)` = `max(200, 852)` = **852**
- Size-based min fee = `min_fee_rate × 200 / 1000`
- Actual fee rate = `fee × 1000 / 852` ≈ `min_fee_rate × 200 / 852` ≈ **23% of min_fee_rate**

The transaction passes the admission gate while its true fee rate is only ~23% of the configured minimum.

### Impact Explanation

An unprivileged transaction submitter can craft transactions with small serialized size but high cycle consumption. Such transactions pass the size-based fee gate with a fee far below what the actual weight demands. These transactions enter the tx-pool with a true fee rate well below `min_fee_rate`, which:

1. **Wastes node CPU**: Script execution for high-cycle transactions is expensive. Nodes execute these scripts during `verify_rtx` before the entry is admitted.
2. **Pollutes the tx-pool**: Low-fee-rate entries occupy pool space, displacing legitimate transactions and degrading pool quality.
3. **Undermines fee market protection**: The `min_fee_rate` guard is the primary anti-spam mechanism for the tx-pool; bypassing it removes the economic disincentive for pool flooding.

### Likelihood Explanation

Any RPC caller or P2P relay peer can submit transactions. Crafting a transaction with small size but high cycles is straightforward: use a script that performs many iterations of computation while keeping the transaction's witness and output data small. No privileged access, key material, or majority hashpower is required.

### Recommendation

After `verify_rtx` returns the actual cycle count, re-check the fee rate using the true weight:

```rust
let actual_weight = get_transaction_weight(tx_size, actual_cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors how `TxEntry::fee_rate()` and `AncestorsScoreSortKey` already compute fee rate using `get_transaction_weight`. [5](#0-4) 

### Proof of Concept

1. Configure a CKB node with `min_fee_rate = 1000` shannons/KW.
2. Craft a transaction:
   - Serialized size ≈ 200 bytes
   - Lock script that loops ~5,000,000 cycles
   - Fee = `1000 × 200 / 1000` = 200 shannons (just enough to pass the size check)
3. Submit via `send_transaction` RPC.
4. The `check_tx_fee` check passes: `fee (200) >= min_fee_rate.fee(200) = 200`.
5. `verify_rtx` executes the script, consuming 5,000,000 cycles.
6. Actual weight = `max(200, 5_000_000 × 0.000_170_571_4)` = 852.
7. Actual fee rate = `200 × 1000 / 852` ≈ 234 shannons/KW — only 23% of the 1000 shannons/KW minimum.
8. The transaction is admitted to the pool despite its true fee rate being far below `min_fee_rate`. [4](#0-3) [6](#0-5)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
