### Title
`forward_rate_limiter` Bypass via Attacker-Controlled `from` Field Enables Unconditional Relay of `ConnectionSync` Messages — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

An unprivileged remote peer can send crafted `ConnectionSync` messages with an empty `route` and an arbitrary `to` PeerId to cause the victim node to unconditionally forward those messages to any directly connected peer. The `forward_rate_limiter` — the intended guard against relay abuse — is trivially bypassed by varying the attacker-controlled `from` field. The only real cap is the outer per-session rate limiter at 30 messages/second.

---

### Finding Description

**Execution path:**

In `ConnectionSyncProcess::execute()`, the message is parsed into a `SyncContent` struct whose `from`, `to`, and `route` fields are taken directly from the wire message with no verification against the actual sender's session identity. [1](#0-0) 

The `forward_rate_limiter` check is keyed on `(content.from, content.to, self.msg_item_id)`: [2](#0-1) 

Since `content.from` is fully attacker-controlled wire data, the attacker generates a fresh rate-limiter bucket for every message simply by using a different `from` PeerId each time. The limiter is completely bypassed.

When `route` is empty and `self_peer_id != content.to`, `forward_sync(&content.to)` is called unconditionally: [3](#0-2) 

`forward_sync` looks up `to` in the peer registry and sends the message with no legitimacy check — no verification against `inflight_requests`, no verification against `pending_delivered`, no check that the sender is the actual `from` peer: [4](#0-3) 

**The only real cap** is the outer `rate_limiter`, keyed on `(session_id, msg.item_id())` at 30 messages/second per session per message type: [5](#0-4) 

The `forward_rate_limiter` quota is 1/second per `(from, to, item_id)` tuple, but since `from` is attacker-controlled, this provides zero protection: [6](#0-5) 

---

### Impact Explanation

The victim node acts as an unconditional relay: for every `ConnectionSync{from=X_i, to=T, route=[]}` the attacker sends (with a fresh random `X_i` each time), the victim forwards it to T. The attacker is capped at 30 forwarded messages/second per session by the outer limiter. With multiple attacker sessions to V, this multiplies linearly. The victim consumes outbound bandwidth and the target peer T receives unsolicited messages, contributing to network congestion at low cost to the attacker.

---

### Likelihood Explanation

The preconditions are minimal: the attacker needs one P2P connection to V and knowledge of T's PeerId (discoverable via the peer discovery protocol). No special privileges, no PoW, no leaked keys. The bypass is mechanical — just vary the `from` bytes in each message.

---

### Recommendation

1. **Verify `from` against the actual sender**: in `ConnectionSyncProcess`, check that `content.from` matches the peer ID of the session that sent the message. This is the root cause — the `forward_rate_limiter` key must include the actual sender identity, not the attacker-supplied `from` field.
2. **Gate forwarding on a legitimate flow**: only forward a `ConnectionSync` if the `(from, to)` pair corresponds to an active entry in `inflight_requests` or `pending_delivered`, mirroring the check already present in `ConnectionRequestDeliveredProcess::execute()`. [7](#0-6) 
3. **Re-key `forward_rate_limiter`** on `(actual_sender_session_id, to, item_id)` rather than `(from, to, item_id)`.

---

### Proof of Concept

```
1. Attacker A connects to victim V (single session).
2. A sends 30 messages per second, each:
     ConnectionSync { from = PeerId::random(), to = T_peer_id, route = [] }
3. For each message:
   - Outer rate_limiter: (session_id, 2) → passes (30/sec budget)
   - forward_rate_limiter: (random_from_i, T, 2) → passes (fresh bucket each time)
   - route.last() == None, self_peer_id != T → forward_sync(&T) called
   - T is in peer_registry → message sent to T
4. T receives 30 unsolicited ConnectionSync messages/second from V.
5. With N attacker sessions to V: T receives 30*N messages/second.
```
