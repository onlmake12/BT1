### Title
Attacker Can Abuse Node's Full Cycle Budget via Unauthenticated `estimate_cycles` RPC — (File: `rpc/src/module/chain.rs`)

### Summary
`CyclesEstimator::run()` executes attacker-supplied scripts up to `consensus.max_block_cycles` with no fee payment, no rate limiting, and no authentication. Any RPC caller can repeatedly saturate the node's script-execution workers at zero cost. Additionally, the `CellProvider` implementation deliberately treats spent (dead) cells as live, allowing an attacker to reference cells they no longer own to trigger script execution.

### Finding Description

`estimate_cycles` is registered as a standard `Chain` module RPC method: [1](#0-0) 

Its implementation delegates to `CyclesEstimator::run()`, which sets the execution limit to the full block cycle budget and calls `ScriptVerifier::verify`: [2](#0-1) 

The `CellProvider` for this estimator explicitly treats every cell — including already-spent ones — as live: [3](#0-2) 

The RPC config has no per-method rate limiting or authentication: [4](#0-3) 

The `Chain` module (which includes `estimate_cycles`) is a standard, default-enabled module. The RPC documentation explicitly states: *"This method will not check the transaction validity, but only run the lock script and type script."* [5](#0-4) 

The same `CyclesEstimator::run()` path is also reachable via the deprecated `dry_run_transaction` in the `Experiment` module: [6](#0-5) 

### Impact Explanation

An attacker with no privileged access can:

1. Deploy a script that consumes exactly `max_block_cycles` cycles (e.g., a tight RISC-V loop).
2. Craft a transaction referencing that script as a lock script on any cell — including already-spent cells, since the `CellProvider` treats them as live.
3. Call `estimate_cycles` in a tight loop from multiple connections.

Each call blocks a script-execution worker thread for the full duration of `max_block_cycles` execution. The node's RPC thread pool and async runtime are saturated, degrading or halting:
- Block template generation (`get_block_template`)
- Transaction relay and pool admission
- Chain sync responses

The "spent cells treated as live" aspect amplifies the attack: the attacker does not need to own any live cell with a high-cycle script. They can reference any historical cell on chain, including system script cells, to trigger expensive execution paths they do not control.

### Likelihood Explanation

- The `Chain` RPC module is enabled by default on all public CKB nodes.
- No fee, no signature, no on-chain asset is required.
- The attack is repeatable indefinitely at negligible cost (only network bandwidth).
- `max_block_cycles` on mainnet is `3,500,000,000` cycles — a single call can occupy a worker for seconds. [7](#0-6) 

### Recommendation

1. **Impose a per-call cycle cap** for `estimate_cycles` that is significantly lower than `max_block_cycles` (e.g., `max_tx_verify_cycles`).
2. **Add rate limiting** at the RPC server layer for compute-heavy endpoints (`estimate_cycles`, `dry_run_transaction`, `test_tx_pool_accept`).
3. **Reject spent cells** in `CyclesEstimator`'s `CellProvider` — do not treat dead cells as live. This prevents referencing scripts the caller does not own.
4. Consider requiring a small proof-of-work or fee commitment before executing scripts via the estimation endpoint.

### Proof of Concept

```
# 1. Compile a RISC-V ELF that loops for max_block_cycles cycles and deploy it on-chain.
# 2. Spend the cell containing that script (making it a dead cell).
# 3. Craft a transaction whose input references the now-spent cell.
# 4. In a loop, call estimate_cycles with that transaction:

while true; do
  curl -s -X POST http://<node>:8114 \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"estimate_cycles","params":[<crafted_tx_json>],"id":1}' &
done
```

Each concurrent call occupies a script-execution worker for the full `max_block_cycles` duration. The node's RPC and tx-pool become unresponsive to legitimate requests. Because the `CellProvider` treats the spent cell as live, the attacker does not need to hold any live asset to trigger execution. [8](#0-7) [9](#0-8)

impl<'a> CellProvider for CyclesEstimator<'a> {
    fn cell(&self, out_point: &packed::OutPoint, eager_load: bool) -> CellStatus {
        let snapshot = self.shared.snapshot();
        snapshot
            .get_cell(out_point)
            .map(|mut cell_meta| {
                if eager_load
                    && let Some((data, data_hash)) = snapshot.get_cell_data(out_point) {
                        cell_meta.mem_cell_data = Some(data);
                        cell_meta.mem_cell_data_hash = Some(data_hash);
                    }
                CellStatus::live_cell(cell_meta)
            })  // treat as live cell, regardless of live or dead
            .unwrap_or(CellStatus::Unknown)
    }

    pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),
                    Err(err) => Err(RPCError::custom_with_error(
                        RPCError::TransactionFailedToVerify,
                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
                RPCError::TransactionFailedToResolve,
                err,
            )),
        }
    }
