### Title
Unbounded `pending_get_block_proposals` Growth via `GetBlockProposal` Message Flooding — (`sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

---

### Summary

An unprivileged relay peer can cause unbounded growth of `SyncState::pending_get_block_proposals` by repeatedly sending `GetBlockProposal` messages containing unique `ProposalShortId`s that are absent from the victim's tx pool. Because `insert_get_block_proposals` has no size cap and the map is only cleared on a periodic timer drain, an attacker can accumulate an arbitrarily large number of entries between drain ticks, leading to memory exhaustion and/or severe performance degradation when the drain fires.

---

### Finding Description

**`pending_get_block_proposals` is an unbounded `DashMap`.**

`SyncState` declares the map with no capacity limit:

```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
``` [1](#0-0) 

It is initialized with `DashMap::new()` — no capacity bound: [2](#0-1) 

**`insert_get_block_proposals` performs no size check.**

Every short ID not found in the tx pool is inserted unconditionally:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [3](#0-2) 

**`GetBlockProposalProcess::execute` feeds all pool-absent IDs directly into the map.**

After the tx pool lookup, every short ID not returned by `fetch_txs` is passed to `insert_get_block_proposals` with no further guard:

```rust
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

self.relayer
    .shared()
    .state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [4](#0-3) 

**The only per-message guard is a count ceiling, not a map-size ceiling.**

Each message is rejected only if it exceeds `max_block_proposals_limit * max_uncles_num` IDs (≈ 1500 × 2 = 3000 on mainnet). There is no limit on how many such messages a peer may send, and no limit on the total size of the map: [5](#0-4) 

**The drain is timer-based and non-bounding.**

`drain_get_block_proposals` clones the entire map and clears it — it fires on a periodic timer via `prune_tx_proposal_request`, not on every insert:

```rust
pub fn drain_get_block_proposals(&self) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
``` [6](#0-5) 

Between two consecutive timer ticks, an attacker sending M messages each containing up to 3000 distinct short IDs absent from the pool accumulates M × 3000 entries in the map with no eviction.

---

### Impact Explanation

1. **Memory exhaustion**: Each `DashMap` entry holds a 10-byte `ProposalShortId` key plus a `HashSet<PeerIndex>`. At high message rates (multiple peers, pipelined messages), the map can grow to millions of entries before the next drain tick, consuming gigabytes of heap.

2. **Performance degradation at drain time**: `drain_get_block_proposals` performs a full `DashMap::clone()` of the accumulated map. A very large map makes this clone expensive (CPU + allocation spike). Immediately after, `prune_tx_proposal_request` sends all accumulated IDs to the tx pool in a single `fetch_txs` call, potentially stalling the tx pool service: [7](#0-6) 

---

### Likelihood Explanation

- The attacker needs only a single P2P connection (no PoW, no stake, no privileged role).
- Short IDs are 10 bytes; fabricating millions of IDs absent from the pool is trivial.
- The async `fetch_txs` call provides some natural throttling (tx pool is a single-threaded service), but multiple concurrent peers each sending at the per-message limit multiply the insertion rate linearly.
- No per-peer rate limit on `GetBlockProposal` messages is present in the handler.

---

### Recommendation

1. **Cap `pending_get_block_proposals` at a fixed maximum size** (e.g., `max_block_proposals_limit * max_uncles_num * max_connected_peers`) and drop or ignore inserts that would exceed it.
2. **Add per-peer rate limiting** on `GetBlockProposal` messages (e.g., token bucket per peer).
3. **Replace the full-clone drain** with a `std::mem::take`-style swap to avoid the O(n) clone cost at drain time.

---

### Proof of Concept

```
1. Connect to victim as a relay peer (SupportProtocols::RelayV3).
2. In a tight loop, send GetBlockProposal messages each containing
   max_block_proposals_limit * max_uncles_num (~3000) distinct
   ProposalShortIds that are NOT in the victim's tx pool.
3. Between two prune_tx_proposal_request timer ticks, send M such messages.
4. Observe pending_get_block_proposals.len() == M * 3000 (unbounded growth).
5. At drain time, observe CPU spike from DashMap::clone() and large
   fetch_txs batch sent to the tx pool service.
```
