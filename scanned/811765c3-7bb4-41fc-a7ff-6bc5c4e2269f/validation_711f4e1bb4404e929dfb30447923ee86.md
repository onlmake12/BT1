### Title
Missing `BLOCK_INVALID` Early-Return in `HeaderAcceptor::accept` Allows Known-Invalid Headers to Be Re-Inserted as Valid â€” (`File: sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderAcceptor::accept()` in `sync/src/synchronizer/headers_process.rs` contains a developer-acknowledged `FIXME` comment noting that a header whose status is `BLOCK_INVALID` is never rejected early. The function checks only for `BLOCK_INVALID`'s absence from the `HEADER_VALID` bitmask, which is always true because the two flags occupy disjoint bits. As a result, any peer can send a `SendHeaders` message containing a header whose block body was previously rejected, bypass the early-exit guard, pass all header-level checks (PoW, timestamp, version), and cause `insert_valid_header` to re-insert the header as `HEADER_VALID` â€” overwriting the node's knowledge that the block is invalid.

### Finding Description

**Root cause â€” `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept`, lines 301â€“322:**

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer best-known and return
    return result;
}
```

`BlockStatus` is a bitflag type defined in `shared/src/block_status.rs`:

```
HEADER_VALID  = 1
BLOCK_INVALID = 1 << 12   (= 4096)
```

`BLOCK_INVALID` does **not** contain the `HEADER_VALID` bit, so `status.contains(BlockStatus::HEADER_VALID)` is `false` when `status == BLOCK_INVALID`. The guard never fires for invalid blocks.

**Code path after the missing guard:**

1. `prev_block_check` â€” passes if the *parent* is not `BLOCK_INVALID` (the header itself is not checked here).
2. `non_contextual_check` â€” runs `HeaderVerifier`, which validates PoW, timestamp, and other header-only fields. A block can be `BLOCK_INVALID` because its *body* (transactions, uncles) failed verification while its header is perfectly valid.
3. `version_check` â€” trivially passes for any version-0 header.
4. `sync_shared.insert_valid_header(self.peer, self.header)` â€” **called unconditionally**, re-inserting the header as `HEADER_VALID` and updating the peer's best-known header.

**Concrete attack scenario:**

1. Block B arrives via the relay protocol. Its header is valid (correct PoW, timestamp, etc.) but its body fails verification (e.g., invalid transaction). The chain service marks B as `BLOCK_INVALID` in the in-memory `block_status_map`.
2. A malicious peer sends a `SendHeaders` P2P message containing B's header.
3. `HeadersProcess::execute` â†’ `accept_first` / loop â†’ `HeaderAcceptor::accept` is called for B's header.
4. The `BLOCK_INVALID` guard is missing. All three subsequent checks pass (header is cryptographically valid).
5. `insert_valid_header` is called: B's header is written back into the `header_map` as `HEADER_VALID`; the peer's best-known header is updated to B.
6. The node's sync state now believes B's header is valid and that this peer is on a chain that includes B.

### Impact Explanation

- **Sync state corruption:** The node's `shared_best_header` and per-peer best-known-header can be advanced to a chain the node itself has already determined to be invalid. The node may then issue `GetBlocks` requests for blocks it already knows are invalid, wasting bandwidth and CPU on re-download and re-verification.
- **Persistent disruption during a session:** Because `block_status_map` is in-memory, the corruption persists until the node restarts. A peer can repeatedly re-inject the same invalid header, keeping the node's sync pointer anchored to a dead chain.
- **No privilege required:** Any connected P2P peer can send `SendHeaders` messages. The attacker only needs a block whose header passes `HeaderVerifier` but whose body was previously rejected â€” a condition that arises naturally from ordinary block relay (e.g., a block with an invalid transaction that the attacker themselves submitted).

### Likelihood Explanation

The `FIXME` comment is an explicit developer acknowledgment that the guard is missing. The `BLOCK_INVALID` flag is set routinely whenever a block body fails verification. Any peer that observed such a block (or crafted one) can immediately exploit this by replaying the header over the sync protocol. No special privilege, key material, or majority hash power is required.

### Recommendation

Add an explicit early-return for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept`, before the `HEADER_VALID` check:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

// Reject headers whose block is already known-invalid.
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return path
}
```

This mirrors the guard already present in `compact_block_process.rs` (`contextual_check`, line 259: `} else if status.contains(BlockStatus::BLOCK_INVALID) { return StatusCode::BlockIsInvalid... }`), making the two code paths consistent.

### Proof of Concept

1. Run a CKB node in dev mode.
2. Submit a transaction with an invalid script to produce a block whose body fails verification. The chain service sets `BLOCK_INVALID` for that block hash.
3. From a second peer, send a `SendHeaders` P2P message (`packed::SendHeaders`) containing only that block's header (which has valid PoW for dev mode).
4. Observe that `HeaderAcceptor::accept` does **not** return early, passes all three sub-checks, and calls `insert_valid_header`.
