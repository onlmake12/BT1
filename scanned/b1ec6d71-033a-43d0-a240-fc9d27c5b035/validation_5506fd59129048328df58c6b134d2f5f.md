Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Correct Accounting in `add_entry`, Inflating `total_tx_size`/`total_tx_cycles` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures a pre-eviction snapshot of `total_tx_size`/`total_tx_cycles` into local variables. When `check_and_record_ancestors` subsequently evicts entries via `remove_entry_and_descendants` → `update_stat_for_remove_tx` (which correctly decrements `self.total_tx_size`/`self.total_tx_cycles` in-place), those correct post-eviction values are immediately overwritten by the stale local snapshot. The result is permanent, cumulative inflation of pool accounting after every eviction-triggering insertion, eventually causing the pool to reject all new transactions with `Reject::Full`.

## Finding Description

The exact sequence in `add_entry` (lines 200–221):

```rust
// Step 1: snapshot BEFORE eviction — local vars only, self not modified
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// total_tx_size  = self.total_tx_size + entry.size   (stale)
// total_tx_cycles = self.total_tx_cycles + entry.cycles (stale)

// Step 2: may evict N entries; each calls update_stat_for_remove_tx,
//         which writes self.total_tx_size -= evict_size (correct, in-place)
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3: OVERWRITES the correctly-decremented self.total_tx_size
//         with the stale pre-eviction snapshot
self.total_tx_size  = total_tx_size;   // loses all eviction decrements
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` is a pure read — it returns `self.total_tx_size + tx_size` as a local value without touching `self`: [2](#0-1) 

`update_stat_for_remove_tx` writes directly to `self.total_tx_size` and `self.total_tx_cycles`: [3](#0-2) 

The eviction path in `check_and_record_ancestors` (line 603) is entered when `ancestors_count > max_ancestors_count` but the excess is covered by `cell_ref_parents`. It calls `remove_entry_and_descendants` in a loop, each call triggering `update_stat_for_remove_tx`: [4](#0-3) 

Concrete arithmetic: if before insertion `self.total_tx_size = S` and eviction removes entries totalling `evict_size`, the correct final value is `S - evict_size + entry.size`. Instead, `S + entry.size` is written back — inflated by `evict_size` per eviction event.

## Impact Explanation

`total_tx_size` is the primary guard for the `max_tx_pool_size` limit. Inflation causes the pool to report itself as fuller than it actually is, causing legitimate transactions to be rejected with `Reject::Full` even though real capacity exists. Repeated triggering compounds the error without bound. This permanently disables the tx pool of the targeted node until restart, mapping to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — an attacker can disable the mempool of any reachable node with a small, repeatable transaction sequence.

## Likelihood Explanation

The eviction branch is reachable by any unprivileged tx-pool submitter. The attacker constructs a chain of transactions where some act as cell-dep parents (`cell_ref_parents`), pushing `ancestors_count` above `max_ancestors_count` (default 125) while keeping non-cell-ref ancestors within the limit. Each such submission permanently inflates the counters. No special privileges, leaked keys, or victim mistakes are required — only the ability to submit transactions to the node's RPC.

## Recommendation

Move `updated_stat_for_add_tx` to **after** `check_and_record_ancestors` completes, so it reads the post-eviction `self.total_tx_size`/`self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Now compute and apply totals on the post-eviction state
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, eliminate the local-variable pattern entirely and apply the increment directly to `self.total_tx_size`/`self.total_tx_cycles` after all mutations are complete.

## Proof of Concept

1. Submit a chain of 125 transactions `T1 → T2 → … → T125` where each `Ti` references the output of `T(i-1)` as a **cell dep** (not an input), making them all `cell_ref_parents`.
2. Submit `T_new` referencing `T1`'s output as a cell dep. Its `ancestors_count` = 126 > 125.
3. `check_and_record_ancestors` enters the eviction branch (line 603): `126 - 125 = 1 ≤ 125`, so it evicts `T125` (lowest fee), calling `update_stat_for_remove_tx(T125.size, T125.cycles)` — `self.total_tx_size` is correctly decremented.
4. `add_entry` then writes back the stale `total_tx_size` (computed before eviction), re-inflating it by `T125.size`.
5. Repeat steps 1–4. Each iteration inflates `total_tx_size` by one evicted entry's size.
6. After enough iterations, `total_tx_size >= max_tx_pool_size` even though the pool holds far fewer bytes; all subsequent `send_transaction` RPC calls return `Reject::Full`.

A unit test can assert the invariant `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum()` before and after a sequence of insertions that trigger the eviction path, and observe it fails after step 4.
