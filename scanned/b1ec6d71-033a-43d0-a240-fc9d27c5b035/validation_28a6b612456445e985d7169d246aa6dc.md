Audit Report

## Title
Predictable Time-Based Ping Nonce Enables RTT Spoofing and Eviction Protection Bypass - (File: `network/src/protocols/ping.rs`)

## Summary
The `nonce` function in `ping.rs` derives the ping challenge from elapsed seconds since node start, making it fully deterministic and predictable by any connected peer. Because the same nonce is broadcast to all peers simultaneously, a malicious inbound peer can pre-compute future nonces and send pong responses before the ping even traverses the network, recording a sub-actual-RTT value. The eviction logic in `peer_registry.rs` explicitly relies on ping RTT being hard to manipulate, and this assumption is broken.

## Finding Description
The nonce is computed as: [1](#0-0) 

This yields `elapsed_seconds_since_start` — a fully deterministic value. In `ping_peers`, a single `send_nonce` is computed once and assigned to every peer: [2](#0-1) 

The same message is then broadcast to all sessions simultaneously: [3](#0-2) 

Pong validation accepts a response if and only if `processing == true` and the nonce matches: [4](#0-3) 

An attacker who receives one ping with nonce `N` immediately learns the node's `start_time` offset. With a default `ping_interval_secs = 120`, the next nonce is exactly `N + 120`. The attacker pre-builds the pong message and, at the moment the next ping round fires (setting `processing = true` server-side), transmits the pre-built pong immediately — before the ping message has even traversed the network to the attacker. The measured RTT becomes approximately one-way latency rather than round-trip latency, and for a co-located or low-latency attacker this is sub-second, stored as 0 by `as_secs()`.

Existing guards are insufficient: the only checks are `processing == true` (which is set server-side at ping dispatch, before the ping arrives at the peer) and nonce equality (which the attacker already knows). There is no timestamp bound, no per-peer randomness, and no replay protection.

## Impact Explanation
The eviction logic explicitly protects the 8 inbound peers with the lowest ping RTT under the stated assumption that RTT is "hard to simulate or manipulate": [5](#0-4) 

By faking a near-zero RTT, an attacker permanently occupies slots in the eviction-protected set. When the inbound connection limit is reached, only unprotected peers are candidates for eviction. An attacker running multiple inbound connections with spoofed RTTs can displace all legitimate peers from the protected set, enabling a sustained eclipse-style attack against the node. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — an attacker with modest resources can eclipse targeted nodes at scale, degrading block propagation and sync quality across the network.

## Likelihood Explanation
The attack requires only an unprivileged inbound TCP connection — no authentication, no special capability. The attacker observes one ping to calibrate the nonce formula, then runs a simple timer loop. The 1200-second pong acceptance window makes timing trivial. The `ping_peers` RPC: [6](#0-5) 
can be used by a local operator to force an immediate ping round, but the attack works without it. The attack is repeatable every 120 seconds indefinitely.

## Recommendation
Replace the global time-based nonce with a per-peer cryptographically random `u32` generated independently for each session at ping time. `PingStatus` already has a `nonce: u32` field: [7](#0-6) 

Change `ping_peers` to generate `ps.nonce = rand::thread_rng().gen::<u32>()` per peer and build a per-peer ping message using that nonce, removing the shared `send_nonce`. This eliminates predictability and ensures each peer faces a unique, unguessable challenge per round, restoring the security property the eviction logic depends on.

## Proof of Concept
1. Connect to a CKB node as an inbound peer.
2. Receive the first `Ping` message; decode its little-endian `u32` nonce field. Call this `N`. Record local time `T`.
3. Compute `next_nonce = N + 120` (default `ping_interval_secs`).
4. Pre-build a `Pong` message with `nonce = next_nonce`.
5. At time `T + 120s + 10ms` (just after the server fires the next ping round, setting `processing = true`), transmit the pre-built `Pong` on the same session — before the `Ping` message has completed its round trip.
6. The server's `received` handler finds `processing == true` and `nonce == next_nonce` — both pass; `pong_received` is called and `peer.ping_rtt` is set to ~10ms + one-way latency.
7. Repeat every 120 seconds. The peer's RTT is permanently recorded as sub-second (stored as 0 by `as_secs()`), placing it in the eviction-protected set regardless of actual network distance.
