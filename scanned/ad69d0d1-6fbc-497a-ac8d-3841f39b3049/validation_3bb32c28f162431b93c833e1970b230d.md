Audit Report

## Title
Unbounded `OrphanBlockPool` Growth via Parentless Block Flooding — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::insert` enforces no capacity limit; `HashMap::with_capacity` is a Rust pre-allocation hint only and does not cap insertions. An attacker who can produce PoW-valid blocks with distinct unknown parent hashes and epoch numbers ≥ `tip_epoch` can grow all three internal data structures (`blocks`, `parents`, `leaders`) without bound, exhausting node memory and causing an OOM kill. The epoch-based eviction path cannot remove these blocks because the expiry condition `epoch_number + EXPIRED_EPOCH < tip_epoch` is never satisfied when the attacker sets `epoch_number ≥ tip_epoch`.

## Finding Description

**No capacity cap in `InnerPool::insert`.**
`InnerPool::with_capacity` calls `HashMap::with_capacity(capacity)`, which is a Rust pre-allocation hint only and imposes no insertion limit: [1](#0-0) 

`InnerPool::insert` performs zero size checks before inserting into all three data structures: [2](#0-1) 

The production pool is initialized with `ORPHAN_BLOCK_SIZE = BLOCK_DOWNLOAD_WINDOW as usize = 8192` — again, only a hint: [3](#0-2) 

**The only eviction path is epoch-based expiry**, triggered by a 60-second timer: [4](#0-3) 

The expiry condition is `epoch_number + EXPIRED_EPOCH < tip_epoch` where `EXPIRED_EPOCH = 6`. An attacker sets the epoch field of crafted blocks to `>= tip_epoch`, making this condition permanently false: [5](#0-4) 

**Attack path (P2P → orphan pool):**
A remote peer sends a block via the sync protocol → `asynchronous_process_block` → `non_contextual_verify` (PoW checked here) → `insert_block` (DB write) → `OrphanBroker::process_lonely_block`. Because the parent hash is unknown and not invalid, the block falls into the unconditional `insert` branch: [6](#0-5) 

Each block with a unique parent hash adds one entry to `blocks`, one to `parents`, and one to `leaders` — all unbounded. The `process_block_rx` channel is bounded at 24 slots, providing backpressure but not preventing the attack — it only throttles the rate. [7](#0-6) 

## Impact Explanation

Continuous insertion of N orphan blocks with distinct parent hashes and current-epoch numbers causes linear memory growth in `InnerPool::blocks`, `::parents`, and `::leaders` with no eviction. Each block is also written to the DB (`insert_block`), adding disk pressure. The process is eventually OOM-killed. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

**Critical constraint — PoW is required.** `BlockVerifier` (called inside `non_contextual_verify`) includes Proof-of-Work verification via `PowVerifier` / `EaglesongBlake2bPowEngine`, which is non-contextual in CKB: [8](#0-7) 

This makes the attack:
- **Trivially cheap** on testnet/devnet (low difficulty) — fully exploitable by any participant.
- **Expensive but feasible** on mainnet for a well-resourced attacker — does not require majority hashpower, only a sustained stream of valid blocks. The epoch-expiry bypass (setting `epoch ≥ tip_epoch`) is unconditional and requires no special capability beyond valid PoW.

## Recommendation

1. **Enforce a hard capacity cap in `InnerPool::insert`**: store `capacity` as a field on `InnerPool` and check `self.parents.len() >= self.capacity` before inserting; when the cap is reached, drop the incoming block or evict a random leader and its descendants.
2. **Add per-peer orphan block accounting** to limit how many orphans a single peer can contribute before being banned.
3. **Tighten epoch range checks** in non-contextual verification to reject blocks whose epoch number is implausibly far from the current tip (requires passing tip epoch into the verifier).

## Proof of Concept

```rust
// Minimal integration test plan:
// 1. Spin up a devnet node (PoW difficulty near zero).
// 2. Connect a crafted peer that sends blocks in a loop:
let pool = OrphanBlockPool::with_capacity(BLOCK_DOWNLOAD_WINDOW as usize);
let tip_epoch = shared.snapshot().tip_header().epoch().number();
for i in 0..100_000u64 {
    let block = BlockBuilder::default()
        .parent_hash(random_byte32())   // unique unknown parent each iteration
        .epoch(EpochNumberWithFraction::new(tip_epoch, i % 1000, 1000))
        .number(tip_epoch * 1000 + i)
        // valid PoW (trivial on devnet), valid cellbase
        .build();
    pool.insert(block.into());
}
assert_eq!(pool.len(), 100_000); // no eviction occurred
// 3. Monitor RSS: grows linearly, no bound enforced.
// 4. Trigger clean_expired_orphans(): zero blocks removed (epoch bypass confirmed).
```

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
    }

    pub(crate) fn process_lonely_block(&self, lonely_block: LonelyBlockHash) {
        let block_hash = lonely_block.block_number_and_hash.hash();
        let block_number = lonely_block.block_number_and_hash.number();
        let parent_hash = lonely_block.parent_hash();
        let parent_is_pending_verify = self.is_pending_verify.contains(&parent_hash);
        let parent_status = self.shared.get_block_status(&parent_hash);
        if parent_is_pending_verify || parent_status.contains(BlockStatus::BLOCK_STORED) {
            debug!(
                "parent {} has stored: {:?} or is_pending_verify: {}, processing descendant directly {}-{}",
                parent_hash, parent_status, parent_is_pending_verify, block_number, block_hash,
            );
            self.process_descendant(lonely_block);
        } else if parent_status.eq(&BlockStatus::BLOCK_INVALID) {
            self.process_invalid_block(lonely_block);
        } else {
            self.orphan_blocks_broker.insert(lonely_block);
        }

        self.search_orphan_leaders();

        if let Some(metrics) = ckb_metrics::handle() {
            metrics
                .ckb_chain_orphan_count
                .set(self.orphan_blocks_broker.len() as i64)
