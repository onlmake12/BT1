Audit Report

## Title
Rate-Limit Bypass: `TooManyRequests = 110` Outside `should_ban()` Guard Allows Persistent Relay Flooding — (`sync/src/relayer/mod.rs`, `sync/src/status.rs`)

## Summary
`StatusCode::TooManyRequests` is assigned value `110`, placing it in the 1xx informational range. `Status::should_ban()` gates exclusively on codes in `400..500`. Consequently, a peer that continuously exceeds the relay rate limiter is never banned or disconnected — it receives only a `debug!` log entry and can retry indefinitely, permanently occupying a peer slot while sustaining ongoing processing overhead on the victim node.

## Finding Description
In `sync/src/relayer/mod.rs`, `try_process()` checks the rate limiter keyed by `(PeerIndex, message_type)` at 30 req/s. On failure it returns `StatusCode::TooManyRequests.with_context(message.item_name())`: [1](#0-0) 

`process()` then calls `status.should_ban()` on the returned status: [2](#0-1) 

`should_ban()` returns `None` for any code outside `400..500`: [3](#0-2) 

`TooManyRequests` is defined as `110`: [4](#0-3) 

Since `110 ∉ [400, 500)`, `should_ban()` returns `None`, `nc.ban_peer()` is never called, and the peer falls through to the `debug!` branch: [5](#0-4) 

By contrast, `HolePunching` handles the rate-limit case by returning immediately at the check site, never propagating a `TooManyRequests` status to any ban logic: [6](#0-5) 

The relayer does not do this — it propagates the status through `process()` where the ban guard silently ignores it.

## Impact Explanation
An attacker with multiple TCP connections can fill all available peer slots with rate-limit-violating peers. Each rejected message still incurs network receive, molecule deserialization through `item_id()`, and a hashmap lookup in the rate limiter. With peer slots permanently consumed by flooding peers, legitimate peers are crowded out. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — the attacker needs only standard bandwidth and a CKB client, with no hashpower or privilege required.

## Likelihood Explanation
Any connected peer can trigger this. No special privilege, key, or hashpower is required. The attacker needs only a single TCP connection and the ability to send relay protocol messages faster than 30/s — trivially achievable with a standard CKB client or raw socket. The attack is repeatable and persistent across the lifetime of the connection.

## Recommendation
The cleanest fix is to mirror the `HolePunching` pattern: handle the rate-limit rejection at the check site in `try_process()` by returning early (e.g., `return Status::ok()` or a dedicated ignored status) rather than propagating `TooManyRequests` upward. Alternatively, either renumber `TooManyRequests` into the 4xx range (e.g., `= 429`) or add an explicit arm in `should_ban()` for `StatusCode::TooManyRequests` returning a ban duration. The `HolePunching` early-return approach is preferred as it avoids the numeric-range coupling entirely. [1](#0-0) [7](#0-6) 

## Proof of Concept
1. Connect to a CKB node as a peer.
2. Send `GetRelayTransactions` (or any non-`CompactBlock` relay message) at >30 req/s for 60 seconds.
3. Observe: the node logs `debug … TooManyRequests` for each excess message but never calls `ban_peer`. The peer remains connected throughout.
4. Repeat with multiple connections to exhaust peer slots, blocking legitimate peers from connecting.
5. Compare with a patched version where `TooManyRequests` triggers `should_ban()` or causes an early return: the peer is disconnected after the first rate-limit violation.

The invariant — that a peer persistently violating the rate limit must be disconnected — is broken because `StatusCode::TooManyRequests = 110` is numerically outside the `400..500` guard in `should_ban()`. [8](#0-7)
