Audit Report

## Title
Peer Store Exhaustion via Single-/16-Group Eviction Failure Enabling Node Isolation — (File: `network/src/peer_store/peer_store_impl.rs`)

## Summary
`AddrManager::add()` uses the full multiaddr including the `/p2p/<PeerId>` component as its deduplication key, allowing an attacker to insert thousands of distinct entries from the same IP:port by varying only the peer ID. When all 16,384 entries (`ADDR_COUNT_LIMIT`) belong to a single `/16` network group, the `check_purge()` eviction routine computes `take(len / 2)` = `take(0)`, producing an empty candidate set and returning `Err(PeerStoreError::EvictionFailed)`. Every subsequent `add_addr()` call then fails silently, permanently preventing legitimate peer addresses from entering the store and isolating the node.

## Finding Description

**Root cause 1 — Full multiaddr (including peer ID) used as deduplication key**

`base_addr()` strips only `Ws`, `Wss`, `Memory`, and `Tls` protocol components but retains the `P2P` (peer ID) component: [1](#0-0) 

`AddrInfo::new()` stores `base_addr(&addr)` as the canonical address: [2](#0-1) 

`AddrManager::add()` uses this full multiaddr (peer ID included) as the `HashMap` key: [3](#0-2) 

Therefore `/ip4/1.2.3.4/tcp/8115/p2p/PeerA` and `/ip4/1.2.3.4/tcp/8115/p2p/PeerB` are stored as two independent entries. An attacker needs only one IP:port to generate up to 16,384 distinct multiaddrs by varying the peer ID.

**Root cause 2 — Integer-division truncation in `check_purge()` step 2**

When all entries are fresh (`attempts_count = 0`, `last_connected_at_ms = 0`), `is_connectable()` returns `true` for all of them, so step 1 finds nothing to remove: [4](#0-3) 

Step 2 groups entries by `/16` network segment and then calls: [5](#0-4) 

If all 16,384 entries share the same `/16` group, `len = 1` and `take(1 / 2)` = `take(0)`. The iterator is empty, `candidate_peers` is empty, and the function returns: [6](#0-5) 

**Root cause 3 — `EvictionFailed` error silently discarded at both call sites**

`DiscoveryAddressManager::add_new_addrs()` only emits a `debug!` log: [7](#0-6) 

`IdentifyProtocol` only emits an `error!` log: [8](#0-7) 

Neither call site disconnects the offending peer, rate-limits further address submissions, or takes any protective action.

**Attack delivery**

`verify_nodes_message()` permits up to `MAX_ADDR_TO_SEND` items per `Nodes` message (when `announce=false`) with up to `MAX_ADDRS` addresses per item: [9](#0-8) 

`is_valid_addr` rejects loopback/private IPs but accepts any globally routable address. An attacker can use addresses spread across `225.0.0.0/16` (or any public /16) with distinct ports and freshly generated peer IDs, filling 16,384 slots in approximately 6 messages.

## Impact Explanation

Once the peer store is exhausted and eviction is permanently broken, `fetch_addrs_to_feeler` and `fetch_addrs_to_attempt` return only attacker-controlled addresses. Legitimate peer addresses advertised by honest nodes are silently dropped. When the node's existing connections drop (restart, network interruption), it cannot reconnect to honest peers and becomes isolated — unable to sync headers or blocks, relay transactions, or confirm new transactions. This matches the **High** severity class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"* — a single attacker peer can isolate targeted nodes at negligible cost, and if applied at scale, degrades the broader network's ability to propagate transactions and blocks.

## Likelihood Explanation

- **Entry path**: Any single inbound or outbound peer speaking the Discovery protocol can trigger this. No privileged role is required.
- **Cost**: ~6 crafted `Nodes` messages (≈16,384 addresses total). All addresses can use one real IP with varying ports and freshly generated peer IDs, or spread across a /16 subnet.
- **Persistence**: The condition persists until node restart. Since the peer store is persisted to disk and reloaded on startup, the attacker may need to re-flood after a restart, but the re-flood cost is the same ~6 messages.
- **No Sybil requirement**: A single IP in a single `/16` block is sufficient to trigger `take(0)`.

## Recommendation

1. **Fix the integer-division bug in `check_purge()`** — replace `take(len / 2)` with ceiling division so a single-group store still evicts entries:
   ```rust
   .take((len + 1) / 2)
   ``` [10](#0-9) 

2. **Add IP:port deduplication in `AddrManager::add()`** — before inserting a new entry, check whether any existing entry shares the same `(IP, port)` pair. If so, update the existing entry's peer ID rather than inserting a duplicate.

3. **Add a per-IP cap** — limit the number of entries per IP address in `AddrManager` (e.g., max 4 per IP), consistent with the eviction threshold already used in `check_purge()`.

4. **Penalize or disconnect peers that trigger `EvictionFailed`** — rather than silently logging, call `misbehave()` on the session that delivered the flooding addresses.

## Proof of Concept

```
1. Establish a connection to a target CKB node (inbound or outbound).

2. Send repeated Discovery `Nodes` messages (announce=false), each containing
   up to MAX_ADDR_TO_SEND items × MAX_ADDRS addresses, where every address has
   the form:
       /ip4/225.0.0.<i>/tcp/<port>/p2p/<freshly_generated_peer_id>
   varying port and peer ID for each entry, keeping all IPs in 225.0.0.0/16.

3. After ~6 messages (≥16384 addresses delivered), the peer store reaches
   ADDR_COUNT_LIMIT.

4. check_purge() is called:
   - Step 1: all entries have attempts_count=0, last_connected_at_ms=0,
     last_tried_at_ms=0 → is_connectable() returns true for all → no eviction.
   - Step 2: len=1 (single /16 group) → take(1/2)=take(0) → candidate_peers
     empty → returns Err(PeerStoreError::EvictionFailed).

5. All subsequent add_addr() calls return EvictionFailed and are silently
   dropped (debug!/error! log only).

6. The target node can no longer learn about legitimate peers. When its
   existing connections drop, it cannot reconnect and becomes isolated.

Unit test plan:
- Create a PeerStore, insert 16384 AddrInfo entries all with IPs in
  225.0.0.0/16, attempts_count=0, last_connected_at_ms=0.
- Call add_addr() with a new legitimate address.
- Assert the return value is Err(PeerStoreError::EvictionFailed).
```
