Audit Report

## Title
Unbounded DoS via `GetBlocksProof` with Mismatched `last_hash`/`block_hashes` Triggering Unpenalized `InternalError` — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`, `util/light-client-protocol-server/src/lib.rs`)

## Summary
An unprivileged remote peer can send a `GetBlocksProof` message with `last_hash` pointing to a low-numbered main-chain block (e.g., block 1) and `block_hashes` containing up to 1000 hashes of higher-numbered main-chain blocks. All validation checks pass, the server performs ~3000 database reads, then `reply_proof` calls `mmr.gen_proof` with positions out-of-range for the MMR anchored at `last_block.number() - 1`, returning `InternalError` (500). Because `should_ban()` only triggers for codes in `400..500`, the peer is never penalized and can repeat the attack indefinitely with no rate limiting in place.

## Finding Description
In `GetBlocksProofProcess::execute` (`get_blocks_proof.rs` lines 38–40), the size check is strictly `> GET_BLOCKS_PROOF_LIMIT` (1000), so exactly 1000 hashes passes. [1](#0-0) [2](#0-1) 

The `last_hash` is validated only via `is_main_chain` (lines 45–50), and each entry in `block_hashes` is partitioned by `is_main_chain` (lines 72–74). There is no check that any found block's `header.number()` is ≤ `last_block.number()`. [3](#0-2) 

For each found block, three DB reads are performed: `get_block_header`, `get_block_uncles`, `get_block_extension` (lines 81–95) — 3000 reads for 1000 blocks. [4](#0-3) 

In `reply_proof` (`lib.rs` line 199), the MMR is constructed as `chain_root_mmr(last_block.number() - 1)`. With `last_block` at height 1, this yields an MMR of size 1 (genesis only). Calling `mmr.gen_proof` with positions for blocks 2–1001 returns `Err`, and the function returns `StatusCode::InternalError` (500). [5](#0-4) 

Back in `received`, `status.should_ban()` checks `(400..500).contains(&code)`. Code 500 is excluded, so no ban is issued. [6](#0-5) 

There is no rate-limiting layer anywhere in `LightClientProtocol::received`. [7](#0-6) 

## Impact Explanation
Each malicious request forces ~3000 synchronous database reads and an MMR proof attempt before failing. With no ban and no rate limit, a single persistent connection can saturate the node's I/O and CPU, crashing or severely degrading the CKB node running the light-client server. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node" (10001–15000 points).

## Likelihood Explanation
The attack requires only a valid P2P connection to a node running the light-client protocol server. No keys, no PoW, no privileged access. The attacker needs only two main-chain block hashes at different heights, which are publicly available on-chain. The attack is trivially scriptable and repeatable without any penalty.

## Recommendation
1. **Add a number-range check in `execute`**: after resolving each found block's header, reject with `StatusCode::MalformedProtocolMessage` (4xx) any block whose `header.number() > last_block.number()`. This converts the attacker-controlled error path from 5xx to 4xx, triggering the existing ban logic.
2. **Add per-peer rate limiting** to `LightClientProtocol::received`, analogous to rate limiters present in other protocol handlers in the codebase.

## Proof of Concept
```
// Precondition: chain has ≥ 1001 blocks on main chain.
// block_1_hash  = hash of block at height 1
// block_N_hash  = hash of block at height N, for N in 2..=1001

let msg = GetBlocksProof {
    last_hash:    block_1_hash,
    block_hashes: [block_2_hash, block_3_hash, ..., block_1001_hash],  // exactly 1000 entries
};

// Server path:
// 1. block_hashes.len() == 1000 == GET_BLOCKS_PROOF_LIMIT  → passes (check is strictly >)
// 2. is_main_chain(block_1_hash)                           → true
// 3. all 1000 block_hashes pass is_main_chain              → all in `found`
// 4. 3000 DB reads (get_block_header + get_block_uncles + get_block_extension × 1000)
// 5. reply_proof: mmr = chain_root_mmr(0), size=1
// 6. gen_proof([pos(2)..pos(1001)])                        → Err → InternalError(500)
// 7. should_ban(): 500 ∉ 400..500                         → None  ← no ban
// Repeat indefinitely with no penalty.
```

                item_name, peer, ban_time, status
