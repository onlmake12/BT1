Audit Report

## Title
Unbounded Cell Scan in `get_cells_capacity` with No Default Timeout Causes CPU Exhaustion — (`util/indexer/src/service.rs`, `rpc/src/module/indexer.rs`)

## Summary

The `get_cells_capacity` RPC endpoint accepts only a `search_key` with no count limit and no default timeout, causing the indexer to iterate over every matching live cell in the database. While a `TimeoutIterator` guard exists in the implementation, `timeout_limit` defaults to `None`, rendering it inactive. An attacker or misconfigured caller can trigger an unbounded CPU-intensive scan. The memory exhaustion claim in the report is inaccurate — the `.sum()` accumulator processes cells one at a time without buffering — but the CPU exhaustion via prolonged iteration and per-cell `memmem::find` scans is real.

## Finding Description

`get_cells_capacity` in the RPC module accepts only a `search_key` with no `limit` parameter: [1](#0-0) 

The service implementation wraps the DB iterator in a `TimeoutIterator`: [2](#0-1) 

However, `timeout_limit` defaults to `None` in `IndexerConfig`: [3](#0-2) 

When `timeout_limit` is `None`, the `TimeoutIterator` imposes no deadline, and the loop at lines 726–836 runs until every key matching the prefix is exhausted, applying per-cell filters including `memmem::find` partial scans: [4](#0-3) 

The timeout check at line 838 only returns an error if the iterator was actually timed out — which never happens with a `None` limit: [5](#0-4) 

By contrast, the rich indexer's `get_cells` enforces a hard `request_limit`: [6](#0-5) 

The basic indexer also has `request_limit: None` by default: [3](#0-2) 

The RPC batch limit is also disabled by default, allowing many concurrent calls in one HTTP request: [7](#0-6) 

The batch enforcement only applies when `JSONRPC_BATCH_LIMIT` is initialized, which requires `rpc_batch_limit` to be set in config: [8](#0-7) 

**Correction to the submitted report:** The memory exhaustion claim is inaccurate. The implementation uses `.sum()` which processes cells one at a time without accumulating them in memory. The actual resource concern is CPU time, not memory.

## Impact Explanation

The RPC is bound to `127.0.0.1` by default: [9](#0-8) 

This limits the attack surface to local processes. The impact is CPU exhaustion of the RPC worker thread for the duration of the scan, matching **Note (0–500 points): Any local RPC API crash/slowdown**. If the operator has exposed the RPC publicly (explicitly warned against in the config), the impact extends to remote DoS of the RPC service, but this requires operator misconfiguration and is outside the default threat model.

## Likelihood Explanation

The attacker must deploy a large number of cells on-chain (e.g., 100,000 cells × ~61 CKB minimum capacity = ~6.1 million CKB), which is a substantial and non-trivial upfront cost. Once deployed, the cells are permanent and the scan condition persists. The attack is only amplified by the disabled batch limit. For the default localhost-only configuration, the attacker must be a local process on the same host.

## Recommendation

1. Set a non-`None` default for `timeout_limit` in `IndexerConfig` (e.g., 10 seconds) so the `TimeoutIterator` guard is active out of the box.
2. Add a `max_cells_capacity_scan_limit` count-based cap analogous to `request_limit` in the rich indexer, and enforce it in the `get_cells_capacity` loop.
3. Enable `rpc_batch_limit` by default (e.g., 2000) to bound batch amplification.

## Proof of Concept

1. Configure a CKB node with the built-in indexer enabled (`modules = ["Indexer"]`) and default settings (no `timeout_limit`, no `rpc_batch_limit`).
2. Deploy N cells (e.g., N = 50,000) sharing the same `lock.code_hash` and a common `args` prefix.
3. Send a JSON-RPC batch of `get_cells_capacity` calls with the matching `script_search_mode: "prefix"` to `127.0.0.1:8114`.
4. Observe that each call triggers a full prefix scan with no early exit, saturating the RPC thread for the duration of the scan.
