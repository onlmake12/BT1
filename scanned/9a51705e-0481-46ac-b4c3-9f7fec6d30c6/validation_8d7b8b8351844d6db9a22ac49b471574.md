### Title
Ban Evasion via Early Return on Known `alert_id` in `AlertRelayer::received` — (`util/network-alert/src/alert_relayer.rs`)

### Summary
`AlertRelayer::received` checks `has_received(alert_id)` **before** verifying signatures. When the check returns `true`, the function returns immediately without calling `ban_peer`. An attacker who knows any currently-active `alert_id` (trivially observable via the unauthenticated `get_blockchain_info` RPC) can send structurally-valid but signature-garbage `Alert` P2P messages indefinitely without ever being banned.

### Finding Description

The ordering of checks in `AlertRelayer::received` is:

1. **Parse** the molecule-encoded alert — ban on failure. [1](#0-0) 
2. **`has_received(alert_id)`** — if `true`, `return` with no ban. [2](#0-1) 
3. **Signature verification** — ban on failure. [3](#0-2) 

Step 2 short-circuits step 3. `has_received` returns `true` for any ID present in `received_alerts` or `cancel_filter`: [4](#0-3) 

Active alert IDs are exposed publicly through `get_blockchain_info` (no authentication required). The attacker only needs to craft a molecule-valid `packed::Alert` (public schema, UTF-8 fields, any garbage in the signatures field) with a matching ID. The parse check at step 1 passes; step 2 fires and returns; step 3 is never reached; `ban_peer` is never called.

### Impact Explanation
The attacker maintains a persistent P2P connection and sends alert messages at line rate. Each message costs the node: molecule parse, a `Mutex` lock on the shared `notifier`, and a `HashMap` lookup. No banning ever occurs, so the connection is never terminated. This is a concrete, repeatable ban-evasion path that removes the only per-peer rate-control mechanism for the Alert protocol.

### Likelihood Explanation
Preconditions are low-cost: (1) any public CKB node exposes `get_blockchain_info` without authentication, leaking active alert IDs; (2) the molecule schema for `packed::Alert` is public; (3) establishing a P2P connection to a CKB node is normal network behavior. The exploit requires no privileged access, no key material, and no majority hashpower.

### Recommendation
Move the `has_received` check to **after** signature verification, or add a `ban_peer` call on the early-return path for replayed/known IDs with invalid signatures. The corrected order should be: parse → verify signatures (ban on failure) → `has_received` (silent return if already known).

### Proof of Concept
```
1. Call get_blockchain_info → observe alerts[0].id = 42
2. Build a molecule-valid packed::Alert with id=42, valid UTF-8 message,
   and a signatures list containing random 65-byte blobs.
3. Connect to target node on the Alert P2P protocol.
4. Send the crafted message repeatedly.
5. Observe: node never calls ban_peer; connection persists indefinitely.
6. Contrast: send the same message with id=999 (unknown) → ban_peer fires
   after signature verification fails.
```

The invariant "peers sending malformed or replayed alerts must be penalized regardless of alert_id" is concretely violated by the ordering at lines 147–149 vs. 151–162. [5](#0-4)
