### Title
Relay Path Enforces `max_block_cycles` But Not `max_tx_verify_cycles`, Allowing Remote Peers to Bypass Per-Transaction Cycle Limit — (`sync/src/relayer/transactions_process.rs`)

### Summary

The CKB relay protocol enforces `max_block_cycles` (consensus-level) as the upper bound for a remotely declared transaction's cycles, but does not enforce `max_tx_verify_cycles` (tx-pool config-level), which is the intended per-transaction cycle limit. This is a direct analog to the FantiumMinterV1/FantiumNFTV1 inconsistency: two components that govern the same value apply different, unsynchronized upper bounds.

### Finding Description

`TxPoolConfig` defines `max_tx_verify_cycles` as the per-transaction cycle cap: [1](#0-0) 

Its documented purpose is: *"tx pool rejects txs that cycles greater than max_tx_verify_cycles"*. The default is `70_000_000` cycles. [2](#0-1) 

However, in `TransactionsProcess::execute`, the relay handler only validates declared cycles against the consensus `max_block_cycles` (`3_500_000_000` on mainnet): [3](#0-2) 

Any declared cycle value that passes this relay check is then forwarded directly to `submit_remote_tx`. Inside `_process_tx`, the `max_cycles` used for script execution is set to `declared_cycles` for remote transactions — not to `max_tx_verify_cycles`: [4](#0-3) 

The `max_tx_verify_cycles` value is stored in the verify queue only as a `large_cycle_threshold` for classification purposes: [5](#0-4) 

There is no hard rejection of remote transactions whose `declared_cycles` exceeds `max_tx_verify_cycles` in the relay ingestion path. The two limits are never compared against each other.

### Impact Explanation

An unprivileged remote peer can relay a transaction with `declared_cycles` set to just below `max_block_cycles` (e.g., `3_499_999_999`). The relay check passes. The node then executes the transaction's scripts with a cycle budget of `3_499_999_999`, which is ~50× the intended `max_tx_verify_cycles = 70_000_000`. This allows a single malicious peer to force the receiving node to spend up to `max_block_cycles` of CPU time verifying one transaction, defeating the DoS-reduction purpose of `max_tx_verify_cycles` (introduced explicitly to "reduce DDOS vulnerability"). [6](#0-5) 

### Likelihood Explanation

Any unprivileged P2P peer can send a `RelayTransactions` message with an arbitrarily large `cycles` field (up to `max_block_cycles`). No special privilege, key, or majority hashpower is required. The attacker only needs a single TCP connection to a CKB node with the Relay protocol enabled.

### Recommendation

In `TransactionsProcess::execute`, add a check that rejects (and bans) peers whose declared cycles exceed `max_tx_verify_cycles` from the tx-pool configuration, mirroring the existing `max_block_cycles` check:

```rust
// existing check
if txs.iter().any(|(_, declared_cycles)| declared_cycles > &max_block_cycles) { ... }

// missing check — add:
let max_tx_verify_cycles = self.relayer.shared().tx_pool_config().max_tx_verify_cycles;
if txs.iter().any(|(_, declared_cycles)| *declared_cycles > max_tx_verify_cycles) {
    self.nc.ban_peer(..., "relay declared cycles greater than max_tx_verify_cycles");
    return Status::ok();
}
```

This synchronizes the two limits, consistent with the intent of `max_tx_verify_cycles`.

### Proof of Concept

1. Connect to a CKB node as a P2P peer supporting `RelayV3`.
2. Construct a valid transaction whose script actually consumes `N` cycles where `N ≤ max_tx_verify_cycles` (e.g., `N = 537`, the `ALWAYS_SUCCESS` script).
3. Send a `RelayTransactions` message with `declared_cycles = max_block_cycles - 1` (e.g., `3_499_999_999`).
4. The relay check at `transactions_process.rs:64-74` passes because `3_499_999_999 ≤ max_block_cycles`.
5. `_process_tx` sets `max_cycles = 3_499_999_999` and runs script verification with that budget.
6. The node expends up to `3_499_999_999` cycles of CPU time on a single transaction, far exceeding the operator-configured `max_tx_verify_cycles = 70_000_000`.
7. Repeat with many transactions to sustain the DoS. [3](#0-2) [7](#0-6)
