### Title
Integer Overflow in `GetLastStateProofProcess::execute` Guard Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Unbounded CPU/IO Exhaustion — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged remote light-client peer can send a crafted `GetLastStateProof` P2P message with `last_n_blocks` set to a value just above `usize::MAX / 2` (e.g., `9223372036854775809` on 64-bit). In Rust release mode, the multiplication `(last_n_blocks as usize) * 2` wraps to a small value, causing the guard check to pass. The server then iterates over the entire chain history — one `get_ancestor` + `chain_root_mmr().get_root()` DB read per block — violating the invariant that `GET_LAST_STATE_PROOF_LIMIT = 1000` bounds all server-side work per request.

---

### Finding Description

**Step 1 — The overflowing guard.**

`last_n_blocks` is decoded as `u64` from the wire message: [1](#0-0) 

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

On a 64-bit host `usize == u64`, so the cast is lossless. The multiplication `(last_n_blocks as usize) * 2` is **unchecked** arithmetic. In Rust release mode, overflow wraps (two's complement). With `last_n_blocks = 9223372036854775809` (`usize::MAX/2 + 2`):

```
9223372036854775809 * 2 mod 2^64 = 2
```

`difficulties.len() + 2 > 1000` is `false` for any small `difficulties` vector, so the guard returns nothing and execution continues.

The limit constant itself: [2](#0-1) 

**Step 2 — Triggering the unbounded reorg path.**

After the guard, the server computes `reorg_last_n_numbers`: [3](#0-2) 

```rust
let reorg_last_n_numbers = if start_block_number == 0
    || snapshot.get_ancestor(&last_block_hash, start_block_number)
        .map(|header| header.hash() == start_block_hash)
        .unwrap_or(false)
{
    Vec::new()
} else {
    let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
    (min_block_number..start_block_number).collect()
};
```

The attacker triggers the `else` branch by supplying a `start_hash` that does **not** match the actual block at `start_number` on the main chain (a trivially crafted mismatch). With `last_n_blocks = 9223372036854775809`:

```
min(start_block_number, 9223372036854775809) = start_block_number
min_block_number = start_block_number - start_block_number = 0
reorg_last_n_numbers = (0..start_block_number)   // millions of entries
```

**Step 3 — The `last_n_numbers` path is also unbounded.** [4](#0-3) 

Because `last_n_blocks` is enormous, `last_block_number - start_block_number <= last_n_blocks` is always true, so `last_n_numbers = (start_block_number..last_block_number)`.

**Step 4 — Combined block list spans the entire chain.** [5](#0-4) 

```rust
let block_numbers = reorg_last_n_numbers   // (0..start_block_number)
    .into_iter()
    .chain(sampled_numbers)
    .chain(last_n_numbers)                 // (start_block_number..last_block_number)
    .collect::<Vec<_>>();
```

`block_numbers` = `(0..last_block_number)` — the entire chain.

**Step 5 — Per-block DB cost.**

`complete_headers` is called on every entry: [6](#0-5) 

Each iteration performs `snapshot.get_ancestor(...)` and `snapshot.chain_root_mmr(number - 1).get_root()` — two expensive DB reads per block.

---

### Impact Explanation

A single malicious `GetLastStateProof` message causes the full node to perform O(chain_height) `get_ancestor` + MMR root DB reads synchronously in the light-client protocol handler. On a mainnet node with millions of blocks this saturates I/O and CPU, degrading block validation, peer sync, and network participation. The attack is repeatable at will by any unauthenticated peer.

---

### Likelihood Explanation

The `GetLastStateProof` message is accepted from any peer without authentication or rate-limiting beyond the broken guard. The overflow value is a single fixed constant. No PoW, no key, no privileged role is required. Any peer that can open a light-client protocol connection can trigger this.

---

### Recommendation

Replace the unchecked arithmetic with a saturating or checked multiply before the comparison:

```rust
// Replace:
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// With:
let last_n_blocks_usize = usize::try_from(last_n_blocks).unwrap_or(usize::MAX);
if self.message.difficulties().len()
    + last_n_blocks_usize.saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

Additionally, add an explicit cap on `reorg_last_n_numbers` length after it is computed, asserting it cannot exceed `GET_LAST_STATE_PROOF_LIMIT`, as defense-in-depth.

### Proof of Concept

```rust
// Craft a GetLastStateProof message:
// - last_hash   = valid tip hash on main chain (obtained via GetLastState)
// - start_hash  = all-zeros (does NOT match any real block at start_number)
// - start_number = chain_height - 1  (large, e.g. 1_000_000)
// - last_n_blocks = 9_223_372_036_854_775_809  // usize::MAX/2 + 2
// - difficulty_boundary = U256::MAX
// - difficulties = []  (empty)

let content = packed::GetLastStateProof::new_builder()
    .last_hash(tip_hash)
    .start_hash(Byte32::zero())          // mismatch → triggers reorg path
    .start_number((chain_height - 1).pack())
    .last_n_blocks(9_223_372_036_854_775_809u64.pack())
    .difficulty_boundary(U256::MAX.pack())
    .build();
```

Guard evaluation: `0 + (9223372036854775809usize * 2) = 2` (wraps) → `2 > 1000` is `false` → guard passes.

`reorg_last_n_numbers` = `(0..chain_height-1)` → server performs `(chain_height - 1)` × 2 DB reads before returning.
