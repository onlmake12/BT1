Audit Report

## Title
Unbounded Outbound Batch Amplification via `GetRelayTransactions` — (`sync/src/relayer/get_transactions_process.rs`)

## Summary
A single `GetRelayTransactions` P2P message containing up to 32,767 tx hashes causes the responding node to emit one outbound `RelayTransactions` message per ~1 MiB batch with no cap on the total number of batches or total outbound bytes per request. The inbound rate limiter (30 req/s per peer) throttles only incoming messages, so a single admitted request can trigger an outbound burst bounded only by the tx pool size (~180 MiB default), yielding a sustained amplification attack from any unprivileged peer.

## Finding Description
`MAX_RELAY_TXS_NUM_PER_BATCH = 32767` and `MAX_RELAY_TXS_BYTES_PER_BATCH = 1 MiB` are defined at `sync/src/relayer/mod.rs` lines 60–61. The `execute()` method in `get_transactions_process.rs` validates only the inbound hash count (lines 33–40), then fetches all matching transactions from the pool (lines 51–84). The batching loop (lines 86–102) iterates over every fetched transaction and calls `send_relay_transactions()` each time the running byte total would exceed 1 MiB, with no guard on the number of such sends:

```rust
if relay_bytes + tx.total_size() > MAX_RELAY_TXS_BYTES_PER_BATCH {
    self.send_relay_transactions(std::mem::take(&mut relay_txs)).await;
    relay_bytes = tx.total_size();
} else {
    relay_bytes += tx.total_size();
}
relay_txs.push(tx);
```

If the pool contains N transactions whose cumulative size is S bytes, the loop emits ⌈S / 1 MiB⌉ outbound messages for a single inbound request. The rate limiter in `try_process` (lines 116–123 of `mod.rs`) uses `governor` keyed on `(PeerIndex, message_type)` at 30 req/s, but it is checked once on receipt of the inbound message and has no effect on the number of outbound messages that request generates.

## Impact Explanation
Outbound bandwidth exhaustion on the targeted node. With the default tx pool cap (~180 MB), a single admitted request can force ~180 MiB of outbound traffic. At the rate-limiter maximum of 30 `GetRelayTransactions` messages per second from one peer, sustained throughput reaches ~5.4 GiB/s of outbound data from a single peer connection, sufficient to saturate a typical node's uplink and degrade relay performance for all other peers. This matches the **Low (501–2000)** bounty impact: "Any other important performance improvements for CKB."

## Likelihood Explanation
No privileged access is required. Any P2P peer can send `GetRelayTransactions`. The precondition — a populated tx pool — is realistic under normal network load; an attacker can also submit valid transactions to fill the pool themselves. Tx hashes can be learned passively from `RelayTransactionHashes` broadcasts. The attack is repeatable at 30 req/s per peer and can be parallelized across multiple peer connections.

## Recommendation
Add a hard cap on the total number of batches (or total outbound bytes) emitted per single `GetRelayTransactions` request. For example, introduce a `MAX_RELAY_BATCHES_PER_REQUEST` constant (e.g., 4–8) and break out of the batching loop once that many `send_relay_transactions` calls have been made. Alternatively, track cumulative bytes sent within the loop and stop once a per-request outbound byte limit is reached. Consider also applying per-peer outbound rate limiting at the send layer independent of inbound message rate limiting.

## Proof of Concept
1. Connect to a target CKB node as an unprivileged P2P peer.
2. Ensure the node's tx pool is populated with N transactions (submit large valid transactions or observe hashes from `RelayTransactionHashes` broadcasts).
3. Send one `GetRelayTransactions` message containing all N tx hashes (N ≤ 32,767).
4. Observe: the node emits ⌈total_pool_bytes / 1 MiB⌉ separate `RelayTransactions` messages back to the requesting peer.
5. Measure total outbound bytes: up to ~180 MiB from a single ~1 MiB inbound message (~180× amplification).
6. Repeat at 30 req/s (rate-limiter maximum per peer) to sustain amplified outbound load; open additional peer connections to multiply throughput linearly.

The invariant violated: a single peer request must not cause outbound bandwidth consumption unbounded relative to the inbound message size. The batching loop at lines 86–102 of `sync/src/relayer/get_transactions_process.rs` contains no guard on the number of batches sent.
