Audit Report

## Title
Single-Child Epoch Sampling in `need_clean` Causes Non-Expired Orphan Blocks to Be Incorrectly Purged — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` samples only the first child returned by a non-deterministic `HashMap` iterator to decide whether to purge an entire orphan subtree. An unprivileged P2P peer can insert two orphan blocks sharing the same parent hash but carrying different epoch numbers — both passing non-contextual verification — causing the entire subtree (including non-expired blocks) to be permanently deleted from both the in-memory pool and the RocksDB store when the expired child is sampled first.

## Finding Description

`need_clean` at `chain/src/utils/orphan_block_pool.rs` lines 113–122 calls `map.iter().next()` on a `HashMap<packed::Byte32, LonelyBlockHash>`, whose iteration order is non-deterministic:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
``` [1](#0-0) 

If the sampled child is expired, `clean_expired_blocks` calls `remove_blocks_by_parent`, which recursively removes **all** children of that leader from `self.blocks`, `self.parents`, and `self.leaders`: [2](#0-1) 

The caller `clean_expired_orphans` then permanently deletes every returned block from the database and removes its `block_status` and `header_view`: [3](#0-2) 

**Why two blocks with the same parent hash but different epoch numbers can both enter the orphan pool:**

`non_contextual_verify` in `chain_service.rs` calls only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [4](#0-3) 

`BlockVerifier::verify` checks only cellbase structure, block bytes, proposals limit, duplicate transactions, and merkle root — **no epoch check, no PoW check**: [5](#0-4) 

The epoch is only validated contextually (requiring the parent block) in `HeaderVerifier` and `ContextualBlockVerifier::EpochVerifier`, neither of which is invoked during the orphan insertion path. An attacker can therefore craft two structurally valid blocks with the same parent hash but arbitrary epoch numbers — e.g., epoch 3 (expired) and epoch 10 (not expired) — at zero PoW cost, and both will be accepted into the orphan pool under the same leader. [6](#0-5) 

`clean_expired_orphans` fires every 60 seconds from the `ChainService` event loop: [7](#0-6) 

## Impact Explanation

When the expired child is sampled first by `map.iter().next()`, `remove_blocks_by_parent` purges the entire subtree — including non-expired blocks — and `clean_expired_orphans` permanently deletes their data from the RocksDB store and erases their `block_status`. When the parent block eventually arrives, the node cannot process the non-expired orphans (they are gone from both pool and DB) and must re-request them from peers. This constitutes an incorrect implementation of the CKB state storage mechanism: blocks that should be retained are permanently deleted from the store based on a non-deterministic sampling decision.

**Allowed impact: Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism.**

The inverse case (non-expired child sampled first) causes expired blocks to never be cleaned, leading to unbounded orphan pool growth.

## Likelihood Explanation

- Any unprivileged P2P peer can send blocks to the node.
- Crafting two structurally valid blocks with the same parent hash but different epoch numbers requires no PoW (the non-contextual verifier does not check PoW or epoch continuity).
- The attacker can observe which block hashes the target node is requesting via the sync protocol to identify a suitable parent hash.
- `clean_expired_orphans` fires every 60 seconds, providing a reliable trigger window.
- The attack is repeatable and cheap.

## Recommendation

`need_clean` must inspect **all** children of a leader. The subtree should only be purged if every direct child is expired:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            !map.is_empty()
                && map.values().all(|lonely_block| {
                    lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
                })
        })
        .unwrap_or_default()
}
```

## Proof of Concept

```
Setup:
  tip_epoch = 10, EXPIRED_EPOCH = 6

Attacker sends to target node:
  B1: parent_hash = P, epoch = 3  (expired: 3 + 6 = 9 < 10)
  B2: parent_hash = P, epoch = 10 (not expired: 10 + 6 = 16 >= 10)

Both pass non_contextual_verify (BlockVerifier checks no epoch field).
Both are inserted into orphan pool: blocks[P] = {B1_hash: B1, B2_hash: B2}
Leader P is added to self.leaders.

60 seconds later, clean_expired_orphans fires:
  tip_epoch_number = 10
  clean_expired_blocks(10) iterates leaders, calls need_clean(P, 10)
  map.iter().next() → returns B1 (non-deterministic)
  B1.epoch_number() + EXPIRED_EPOCH = 9 < 10 → need_clean returns true
  remove_blocks_by_parent(P) removes BOTH B1 and B2 from pool
  delete_block called on both; remove_block_status called on both

Result:
  B2 (non-expired, potentially legitimate) is permanently deleted from DB.
  When P later arrives, the node cannot process B2 and must re-request it.
  Sync is disrupted.

Unit test plan:
  Construct an OrphanBlockPool, insert two LonelyBlockHash entries under the
  same parent hash with epoch numbers 3 and 10. Call clean_expired_blocks(10)
  in a loop 1000 times (with fresh pool each time) and assert that in runs
  where need_clean returns true, both blocks are removed — demonstrating that
  the non-expired block is incorrectly purged.
```

    pub fn remove_blocks_by_parent(&mut self, parent_hash: &ParentHash) -> Vec<LonelyBlockHash> {
        // try remove leaders first
        if !self.leaders.remove(parent_hash) {
            return Vec::new();
        }

        let mut queue: VecDeque<packed::Byte32> = VecDeque::new();
        queue.push_back(parent_hash.to_owned());

        let mut removed: Vec<LonelyBlockHash> = Vec::new();
        while let Some(parent_hash) = queue.pop_front() {
            if let Some(orphaned) = self.blocks.remove(&parent_hash) {
                let (hashes, blocks): (Vec<_>, Vec<_>) = orphaned.into_iter().unzip();
                for hash in hashes.iter() {
                    self.parents.remove(hash);
                }
                queue.extend(hashes);
                removed.extend(blocks);
            }
        }

        debug!("orphan pool pop chain len: {}", removed.len());
        debug_assert_ne!(
            removed.len(),
            0,
            "orphan pool removed list must not be zero"
        );

        shrink_to_fit!(self.blocks, SHRINK_THRESHOLD);
        shrink_to_fit!(self.parents, SHRINK_THRESHOLD);
        shrink_to_fit!(self.leaders, SHRINK_THRESHOLD);
        removed
    }
