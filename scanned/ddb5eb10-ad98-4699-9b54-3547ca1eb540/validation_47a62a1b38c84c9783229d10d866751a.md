Audit Report

## Title
Stale `descendants_*` Accounting in `PoolMap::remove_entry_and_descendants` Allows Ancestor Eviction Priority Inflation — (`tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::remove_entry_and_descendants` pre-clears all link records for every entry in the removed subtree before calling `remove_entry` on each. Because `update_ancestors_index_key` relies on those link records to identify still-live ancestors, the pre-clearing silently prevents any ancestor from receiving a `sub_descendant_weight` call. Ancestors of the removed root transaction permanently retain inflated `descendants_size`, `descendants_cycles`, `descendants_fee`, and `descendants_count` values, causing their `EvictKey.fee_rate` to be overstated. An unprivileged submitter can exploit this via repeated RBF replacements to make a low-fee transaction immune to pool eviction, causing legitimate high-fee transactions to be rejected when the pool is full.

## Finding Description

**Root cause — ordering inversion in `remove_entry_and_descendants`:**

`remove_entry_and_descendants` (L252–265) first iterates over all removed IDs and calls `remove_entry_links` on each, then calls `remove_entry` on each:

```rust
// tx-pool/src/component/pool_map.rs L256-264
for id in &removed_ids {
    self.remove_entry_links(id);   // links cleared for ALL removed entries
}
removed_ids
    .iter()
    .filter_map(|id| self.remove_entry(id))
    .collect()
```

`remove_entry_links` (L418–430) removes the entry from its parents' children sets, removes the entry from its children's parents sets, and removes the entry's own record from `self.links`.

Inside `remove_entry` (L235–250), `update_ancestors_index_key` is called at L242:

```rust
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
```

`update_ancestors_index_key` (L432–445) calls `self.links.calc_ancestors(&child.proposal_short_id())`. Because `remove_entry_links` already removed the entry's record from `self.links`, `calc_ancestors` returns an empty `HashSet`. The `for anc_id in &ancestors` loop body never executes. No still-live ancestor receives `sub_descendant_weight`.

By contrast, when `remove_entry` is called directly (not via `remove_entry_and_descendants`), `remove_entry_links` is called at L245 — *after* `update_ancestors_index_key` at L242 — so ancestors are correctly updated before links are torn down.

The comment at L256 acknowledges the intent: "update links state for remove, so that we won't update_descendants_index_key in remove_entry." The intent was to skip updating descendants of removed entries (correct — they are also being removed). The unaddressed side-effect is that `update_ancestors_index_key` for the root entry's still-live ancestors is also silently skipped.

**Exploit path via RBF:**

`process_rbf` (process.rs L203–206) calls `remove_entry_and_descendants` for each conflicting transaction, then inserts the replacement. Each cycle:

1. `remove_entry_and_descendants(tx_B)` is called — tx_A's `descendants_*` is **not decremented** (links already cleared).
2. tx_B' is inserted — `update_ancestors_index_key(tx_B', Add)` correctly **increments** tx_A's `descendants_*`.

After N replacements, `tx_A.descendants_fee ≈ tx_A.fee + N × tx_B_fee + tx_B'_fee` instead of the correct `tx_A.fee + tx_B'_fee`.

**Effect on eviction:**

`EvictKey` (entry.rs L234–247) sets `fee_rate = descendants_feerate.max(feerate)`. Inflated `descendants_fee` raises `descendants_feerate`, raising `EvictKey.fee_rate`. `next_evict_entry` (pool_map.rs L380–385) selects by lowest `EvictKey` via `iter_by_evict_key()`. An inflated `EvictKey.fee_rate` on tx_A means `limit_size` (pool.rs L292–329) will never select tx_A for eviction, even when the pool is full and tx_A's true fee rate is below the eviction threshold.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A low-fee transaction can be kept alive indefinitely in a full pool by repeatedly RBF-replacing a high-fee child. Each replacement cycle compounds the inflation of the ancestor's `descendants_*` fields. When `limit_size` runs, it evicts transactions with the lowest `EvictKey.fee_rate` first; the artificially inflated ancestor is never selected. Legitimate high-fee transactions submitted by honest users are rejected with `Reject::Full` while the attacker's low-fee transaction survives. With a full 180 MB pool, this constitutes sustained, low-cost disruption of the mempool's fee-based ordering and admission policy.

## Likelihood Explanation

RBF is enabled by default in the mainnet configuration (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`). Any unprivileged tx-pool submitter can trigger this path by submitting a parent transaction and then repeatedly submitting and RBF-replacing a child transaction. The attacker pays the RBF fee premium per replacement, but the cost is bounded and predictable (minimum increment per replacement is the RBF rate differential applied to the child's size). The inflation benefit accumulates without bound across replacements, making the attack economically favorable over time.

## Recommendation

Before clearing any link records in `remove_entry_and_descendants`, update the ancestors of the root entry being removed. Specifically, call `update_ancestors_index_key(&root_entry, EntryOp::Remove)` while the root entry's links are still intact, so that every still-live ancestor receives a correct `sub_descendant_weight` call. The descendants of the root (which are also being removed) do not need ancestor updates since they are being expelled entirely. Alternatively, restructure `remove_entry_and_descendants` to call `remove_entry` on the root entry first (which correctly updates its still-live ancestors via the intact links), then remove links and entries for the remaining descendants.

## Proof of Concept

1. Submit `tx_A` (low fee rate, just above `min_fee_rate = 1_000`) to the pool.
2. Submit `tx_B` (child of `tx_A`, high fee rate) to the pool. `tx_A.descendants_fee` is incremented by `tx_B.fee`; `tx_A.EvictKey.fee_rate` reflects `tx_B`'s high rate.
3. Submit `tx_B'` (same inputs as `tx_B`, fee satisfying RBF rules) to trigger `process_rbf`:
   - `remove_entry_and_descendants(tx_B)` is called.
   - `remove_entry_links(tx_B)` clears `tx_B`'s link record.
   - `remove_entry(tx_B)` calls `update_ancestors_index_key(tx_B, Remove)`, which finds no ancestors (links gone); `tx_A.descendants_*` is **not decremented**.
   - `tx_B'` is inserted; `update_ancestors_index_key(tx_B', Add)` correctly increments `tx_A.descendants_*`.
   - `tx_A.descendants_fee` now equals `tx_A.fee + tx_B.fee + tx_B'.fee` — doubly inflated.
4. Repeat step 3 N times. After N replacements, `tx_A.descendants_fee ≈ tx_A.fee + N × tx_B_fee + tx_B'_fee`.
5. Fill the pool to capacity with other transactions. Observe that `limit_size` never selects `tx_A` for eviction despite its true fee rate being near `min_fee_rate`, because its `EvictKey.fee_rate` is inflated to reflect the accumulated phantom descendant fees.
6. A unit test can assert this invariant: after `remove_entry_and_descendants` on a child, the parent's `descendants_fee` must equal its pre-child-insertion value.
