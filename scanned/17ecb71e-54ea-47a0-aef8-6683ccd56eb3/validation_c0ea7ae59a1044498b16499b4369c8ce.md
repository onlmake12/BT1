### Title
`CapacityVerifier::valid_dao_withdraw_transaction()` Skips Overflow Check for DAO Deposit Cells, Enabling Non-DAO Capacity Inflation — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` for **any** transaction whose inputs include a DAO-typed cell — including **deposit-phase cells** (`cell_data == 0x0000000000000000`, `deposited_block_number == 0`). This causes the `OutputsSumOverflow` guard to be skipped for the entire transaction, including non-DAO outputs. Because the DAO type script only validates DAO cells and not non-DAO cells, a transaction sender can inflate non-DAO output capacity arbitrarily in a deposit→prepare (phase 1) transaction, creating CKB from nothing.

---

### Finding Description

The Nervos DAO has three phases:

| Phase | Cell data | Meaning |
|---|---|---|
| Deposit | `0x0000000000000000` (`deposited_block_number == 0`) | Funds locked in DAO |
| Prepare (phase 1) | `block_number_of_deposit` (`deposited_block_number > 0`) | Preparing to withdraw |
| Withdraw (phase 2) | Spend the prepare cell | Receive principal + interest |

`DaoCalculator::transaction_maximum_withdraw()` correctly distinguishes these phases at line 66:

```rust
if deposited_block_number > 0 {
    // phase 2: calculate interest
} else {
    Ok(output.capacity().into())  // phase 1: face value only
}
```

However, `CapacityVerifier::valid_dao_withdraw_transaction()` does **not** make this distinction:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
```

It returns `true` for **any** DAO-typed input, including deposit cells (`deposited_block_number == 0`). This causes `CapacityVerifier::verify()` to skip the `OutputsSumOverflow` check for the entire transaction:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // This block is skipped for ALL DAO-input transactions, including deposit→prepare
    if inputs_sum < outputs_sum {
        return Err(TransactionError::OutputsSumOverflow { ... }.into());
    }
}
```

The DAO type script (a C script running in CKB-VM) validates only the DAO cells it is attached to. It does not validate non-DAO outputs. Therefore, when the `OutputsSumOverflow` check is skipped for a deposit→prepare transaction, non-DAO outputs are completely unchecked for capacity conservation.

---

### Impact Explanation

An attacker can craft a deposit→prepare transaction that:

1. Spends a DAO deposit cell (e.g., 100 CKB, `cell_data = 0x0000000000000000`)
2. Spends a regular cell (e.g., 1 CKB)
3. Produces a DAO prepare cell (100 CKB, `cell_data = block_number`) — validated by the DAO script ✓
4. Produces a regular cell with arbitrarily inflated capacity (e.g., 1,000,000 CKB) — **not validated by the DAO script, and `OutputsSumOverflow` check is skipped** ✗

The DAO script validates Output 1 (100 CKB == Input 1 100 CKB). No verifier checks Output 2 (1,000,000 CKB) against Input 2 (1 CKB). The attacker creates 999,999 CKB from nothing. This is a consensus-level violation: any node accepting such a transaction inflates the total CKB supply, breaking the economic model and the security of all CKB holders.

---

### Likelihood Explanation

The attack requires only:
- Owning a DAO deposit cell (trivially created by any user)
- Submitting a crafted deposit→prepare transaction via the standard `send_transaction` RPC

No privileged access, no majority hashpower, no social engineering. Any unprivileged transaction sender can trigger this. The attack is deterministic and repeatable.

---

### Recommendation

`valid_dao_withdraw_transaction()` must restrict its `true` return to **phase 2** (prepare→withdraw) transactions only — i.e., inputs where `deposited_block_number > 0`. This mirrors the existing logic in `DaoCalculator::transaction_maximum_withdraw()`. `CapacityVerifier` needs access to a `CellDataProvider` (as `DaoScriptSizeVerifier` already has) to load cell data and check the deposit block number:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| {
            cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash)
                && self.data_loader.load_cell_data(cell_meta)
                    .map(|data| data.len() == 8 && LittleEndian::read_u64(&data) > 0)
                    .unwrap_or(false)
        })
}
```

This ensures the `OutputsSumOverflow` bypass is only applied when the DAO script is actually permitted to produce outputs exceeding inputs (phase 2 interest accrual), not during the capacity-neutral deposit→prepare transition.

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
