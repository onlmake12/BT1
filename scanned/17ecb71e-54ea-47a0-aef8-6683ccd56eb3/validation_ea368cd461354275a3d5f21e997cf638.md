### Title
`ScriptHashTypeVerifier` Omits Type Script Hash-Type Validation, Allowing Unactivated `ScriptHashType` Values to Bypass Non-Contextual Checks — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that every output's **lock** script uses only a currently-enabled `ScriptHashType`, but performs **no equivalent check on the output's type script**. Any unprivileged transaction sender can craft a transaction whose output carries a type script with a future/unactivated hash-type value (e.g., `Data3 = 6`). That transaction passes the entire `NonContextualTransactionVerifier` pipeline and is admitted to the tx-pool, where it consumes resources before being rejected later at script-execution time.

---

### Finding Description

`ScriptHashTypeVerifier` is embedded in `NonContextualTransactionVerifier` and is documented as:

> "Check whether output lock hash type within enabled range" [1](#0-0) 

Its `verify()` implementation iterates over every output and checks only `output.lock().hash_type()`: [2](#0-1) 

The type script field (`output.type_()`) is never read. The constant being checked against is: [3](#0-2) 

`ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}` — Data, Type, Data1, Data2. Future variants such as `Data3 = 6`, `Data4 = 8`, … (generated by the `seq!` macro) are absent from this set: [4](#0-3) 

The existing unit tests confirm the gap: only the lock-script path is exercised (`test_not_enabled_hash_type_output_lock`); there is no corresponding test for the type-script path: [5](#0-4) 

By contrast, the P2P-layer `check_data` helper **does** validate both lock and type scripts: [6](#0-5) 

This creates an inconsistency: a compact-block message carrying such a transaction is rejected at the P2P layer, but the same transaction submitted via `send_transaction` RPC passes `NonContextualTransactionVerifier` and enters the tx-pool.

---

### Impact Explanation

1. An attacker submits a transaction whose output has a valid lock script (`hash_type = 0`) and a type script with `hash_type = 6` (Data3, not yet activated).
2. `ScriptHashTypeVerifier::verify()` passes — it never inspects the type script.
3. The transaction is admitted to the tx-pool, consuming pool capacity and triggering full contextual verification work.
4. Script execution calls `select_version()`, which returns `Err(ScriptError::InvalidVmVersion(3))` for the unactivated hash type, and the transaction is finally rejected.

The practical impact is **tx-pool resource exhaustion**: an attacker can repeatedly inject structurally valid but semantically invalid transactions that each consume a full contextual-verification cycle before eviction. Additionally, the missing check undermines the hardfork-protection invariant that `ScriptHashTypeVerifier` is designed to enforce — a future hardfork activating `Data3` would silently make previously-injected transactions valid without the node ever having enforced the pre-activation barrier on type scripts.

---

### Likelihood Explanation

The attack requires no privileges, no stake, and no special network position. Any RPC caller with access to `send_transaction` can craft such a transaction. The `ScriptHashType` enum values are public and well-documented. Exploitation is trivially scriptable.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's hash type for every output, mirroring the existing lock-script logic:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_ht = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?;
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&(Into::<u8>::into(lock_ht))) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: lock_ht.into() }.into());
        }
        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?;
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&(Into::<u8>::into(type_ht))) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: type_ht.into() }.into());
            }
        }
    }
    Ok(())
}
```

Also add a corresponding unit test (`test_not_enabled_hash_type_output_type`) analogous to the existing lock-script test.

### Proof of Concept

Construct a transaction with one output:

```
lock  = { code_hash: <any>, hash_type: 0 (Data), args: 0x }   // passes ScriptHashTypeVerifier
type_ = { code_hash: <any>, hash_type: 6 (Data3), args: 0x }  // NOT checked — bypasses ScriptHashTypeVerifier
```

Submit via `send_transaction` RPC with `outputs_validator: "passthrough"`. The transaction passes `NonContextualTransactionVerifier::verify()` (including `ScriptHashTypeVerifier`) and enters the tx-pool. It is only rejected later when `select_version()` is called during script execution and returns `Err(ScriptError::InvalidVmVersion(3))`.

The same transaction submitted as a compact-block prefilled transaction is rejected immediately at the P2P layer by `check_data`, demonstrating the inconsistency between the two validation paths. [7](#0-6) [6](#0-5)

// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}
