Audit Report

## Title
Blocking crossbeam `send` on bounded(24) channel inside async `SendBlock` handler stalls tokio worker threads — (`sync/src/synchronizer/mod.rs`, `chain/src/chain_controller.rs`)

## Summary

`Synchronizer::try_process` is an `async fn` running on a tokio worker thread. For `SendBlock` messages, it calls `BlockProcess::execute()` without `tokio::task::block_in_place`, unlike every other message type. The call chain terminates at `Request::call_without_response`, which performs a blocking `crossbeam_channel::Sender::send()` on a `bounded(24)` channel. When that channel is full, the tokio worker thread blocks indefinitely, starving the runtime and preventing all P2P message processing.

## Finding Description

All code references are confirmed in the repository:

**Channel is bounded(24):**
`chain/src/init.rs:93` — `let (process_block_tx, process_block_rx) = channel::bounded(24);`

**`call_without_response` performs a blocking send:**
`util/channel/src/lib.rs:44-50` — `sender.send(...)` is the crossbeam blocking variant. The `let _ =` discards the `Result` but does not prevent the thread from blocking when the channel is full.

**`asynchronous_process_lonely_block` is the direct call site:**
`chain/src/chain_controller.rs:61-63` — calls `Request::call_without_response(&self.process_block_sender, lonely_block)`.

**`SendBlock` is the only handler missing `block_in_place`:**
`sync/src/synchronizer/mod.rs:396-418` — `GetHeaders`, `SendHeaders`, and `GetBlocks` are all wrapped in `tokio::task::block_in_place`. The `SendBlock` arm at line 412 calls `BlockProcess::new(...).execute()` directly with no such wrapper.

**`try_process` is `async fn`:**
`sync/src/synchronizer/mod.rs:381` — confirmed.

Full blocking call chain:
```
Synchronizer::try_process (async, tokio worker thread)
  └─ BlockProcess::execute()                          [no block_in_place]
       └─ Synchronizer::asynchronous_process_remote_block()
            └─ SyncShared::accept_remote_block()
                 └─ ChainController::asynchronous_process_remote_block()
                      └─ ChainController::asynchronous_process_lonely_block()
                           └─ Request::call_without_response()
                                └─ crossbeam Sender::send()  ← BLOCKS if channel full
```

`ChainService` drains the channel sequentially, one block per iteration, with a DB write per block (`chain/src/chain_service.rs:43-69`). Under any I/O pressure, the drain rate is far below the ingest rate of a flooding attacker.

## Impact Explanation

When all tokio worker threads are blocked at `sender.send()`, the entire P2P message processing loop stalls: no `GetHeaders`, `SendHeaders`, `GetBlocks`, or any other message can be processed. The node is effectively partitioned from the network for the duration of the attack. This matches **High: Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

An attacker needs only to:
1. Complete a `SendHeaders` exchange to give 24+ block hashes `HEADER_VALID` status in the node's `block_status_map` — a normal, unauthenticated P2P operation.
2. Send 24+ `SendBlock` messages for those blocks faster than `ChainService` drains them.

`ChainService` processes blocks one at a time with synchronous DB writes. The channel saturates quickly under any I/O load. The number of worker threads needed to stall the runtime equals the tokio worker count (typically CPU core count), achievable with 24 + N concurrent `SendBlock` messages from one or a few peers. The attack is repeatable and requires no special privilege.

## Recommendation

1. **Immediate**: Wrap the `SendBlock` arm in `tokio::task::block_in_place` at `sync/src/synchronizer/mod.rs:412-418`, matching the pattern used for all other message types.
2. **Better**: Replace `sender.send()` in `Request::call_without_response` (`util/channel/src/lib.rs:46`) with `sender.try_send()`, returning an error or dropping the block when the channel is full, making the function truly non-blocking.
3. **Structural**: Apply per-peer rate limiting on `SendBlock` ingress so the channel cannot be saturated by a single attacker.

## Proof of Concept

1. Connect a mock peer and perform a `SendHeaders` exchange for 30 blocks, establishing `HEADER_VALID` status for their hashes in `block_status_map`.
2. Replace `ChainService`'s consumer with a mock that never reads from `process_block_rx` (simulating slow DB).
3. Spawn 30 concurrent tasks, each sending one `SendBlock` message via `BlockProcess::execute()` on a tokio worker thread.
4. Assert that the 25th call does not return within a 1-second timeout — it will not, because `sender.send()` blocks with the channel at capacity 24.
5. Confirm that a concurrent `GetHeaders` message also fails to be processed during the stall, demonstrating full runtime starvation.
