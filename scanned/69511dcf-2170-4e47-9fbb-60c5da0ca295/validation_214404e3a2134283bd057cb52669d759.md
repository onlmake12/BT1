Audit Report

## Title
`script_exists_in_output` Re-reads `row_lock` Instead of `row_type` on PostgreSQL, Causing Incorrect Type-Script Deletion During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary

In `script_exists_in_output`, line 252 evaluates `row_lock.try_get::<bool, _>(0)` a second time instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, `try_get::<bool, _>` always succeeds, so the function only reaches line 252 when `row_lock` already returned `false`; the final `match` therefore always resolves to `Ok(false)`, unconditionally reporting that no type-script exists. During any rollback, every type-script from the rolled-back block is deleted from the `script` table — even when it is still referenced by surviving outputs in earlier blocks — permanently corrupting the indexer's script-to-output mapping.

## Finding Description

`script_exists_in_output` (lines 204–257 of `util/rich-indexer/src/indexer/remove.rs`) performs two `SELECT EXISTS` queries: one for `lock_script_id` (`row_lock`) and one for `type_script_id` (`row_type`). After the second query, the final `match` at line 252 mistakenly reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`:

```rust
// line 252 — copy-paste bug: row_lock should be row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),          // always Ok(false) on PG — row_type is never consulted
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // correct SQLite path
}
```

On PostgreSQL, `try_get::<bool, _>` never fails, so the `Err` branch (which correctly reads `row_type`) is dead code. The function only reaches line 252 when the lock-script check at lines 222–235 already returned `false`; therefore `Ok(r)` at line 253 is always `Ok(false)`, regardless of `row_type`'s value.

`rollback_block` (lines 27–39) first removes all `output` rows for the tip block, then calls `script_exists_in_output` for every script referenced by those outputs to decide whether to delete the script row. Because the type-script check always returns `false` on PostgreSQL, every type-script from the rolled-back block is added to `script_id_list_to_remove` and deleted — even when the same type-script ID is still referenced by outputs in earlier, surviving blocks.

All existing tests use `connect_sqlite(MEMORY_DB)`, so the SQLite `Err` path is exercised and passes; the PostgreSQL path is never tested.

## Impact Explanation

This is a concrete data-corruption bug in CKB's rich-indexer state storage mechanism. After any rollback on a PostgreSQL-backed rich-indexer, the `script` table loses type-script rows that are still referenced by live outputs. All subsequent RPC calls (`get_cells`, `get_transactions`) filtered by those type-scripts silently return empty results, hiding live cells from callers. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**, as the indexer is CKB's primary cell/script state storage and query layer, and the corruption is permanent until the indexer is fully re-synced.

## Likelihood Explanation

The trigger is any chain reorganization on a PostgreSQL-backed rich-indexer deployment. Reorgs are a normal part of CKB network operation and require no attacker action — a natural 1-block reorg suffices. PostgreSQL is a supported and documented backend. No special privileges, hashpower, or victim mistakes are required.

## Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Add a PostgreSQL integration test for `rollback_block` that verifies a type-script shared across two blocks is retained in the `script` table after rolling back only one of those blocks.

## Proof of Concept

1. Start a CKB rich-indexer backed by PostgreSQL.
2. Index **block A** containing two outputs sharing the same `type_script T` (different lock-scripts).
3. Index **block B** (a fork tip) containing one output also referencing `type_script T`.
4. Trigger rollback of block B (simulating a reorg back to block A).
5. Query the `script` table: `type_script T`'s row is absent despite block A's outputs still referencing it.
6. Issue `get_cells` filtered by `type_script T`: returns 0 results despite live cells existing in block A.
