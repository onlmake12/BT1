### Title
Unauthenticated Block Template Injection via Miner Notify HTTP Server — (`miner/src/client.rs`)

### Summary

The CKB miner's "notify mode" HTTP server accepts `BlockTemplate` payloads from any TCP client without authentication. Any attacker who can reach the miner's notification port can inject a crafted `BlockTemplate`, redirecting all mining hashpower to an attacker-controlled block — enabling mining reward theft or permanent denial of service.

### Finding Description

When the miner is configured with a `listen` address in `ckb-miner.toml`, `Client::spawn_background` starts an HTTP server via `listen_block_template_notify`: [1](#0-0) 

The server dispatches every incoming HTTP request to the `handle` free function: [2](#0-1) 

`handle` reads the raw request body, attempts to deserialize it as a `BlockTemplate`, and — if deserialization succeeds — immediately calls `client.update_block_template(template)`. There is **no authentication, no IP allowlist check, and no HMAC/token verification** of any kind. The `parse_authorization` helper defined in the same file is used only for outbound RPC calls to the CKB node; it is never invoked on the inbound notification path.

`update_block_template` forwards the injected work to all mining workers: [3](#0-2) 

The update fires whenever the incoming `work_id` differs from the currently stored one, which an attacker trivially controls by incrementing the field in each injected payload.

The `listen` field in `MinerClientConfig` is an arbitrary `SocketAddr`: [4](#0-3) 

The default template comments it out (`# listen = "127.0.0.1:8888"`), but the documented use-case explicitly instructs operators to enable it and pair it with the node's `block_assembler.notify` list: [5](#0-4) [6](#0-5) 

Operators who follow the documented notify-mode setup expose the port. If bound to `0.0.0.0` or any non-loopback address (common in pool/cloud mining setups), the port is reachable from the network.

### Impact Explanation

**Mining reward theft.** An attacker injects a `BlockTemplate` whose `cellbase` output carries the attacker's lock script. The miner solves PoW on that template and submits the block; the block reward is paid to the attacker's address, not the legitimate miner's.

**Permanent DoS.** An attacker continuously injects templates with an invalid `parent_hash` or stale block number. The miner wastes all hashpower on blocks that will be rejected by the CKB node on `submit_block`, and the legitimate node-pushed templates are immediately overwritten by the next attacker injection.

Both outcomes match the HIGH-impact class of the reference report: an unprivileged attacker can change a core operational state variable (`current_work_id` + the active `Work` sent to workers) with no preconditions beyond network reachability of the port.

### Likelihood Explanation

Notify mode is the **recommended** low-latency mining configuration. Pool operators and cloud miners routinely bind services to non-loopback addresses. Any process or peer that can reach the TCP port — including co-located processes, LAN peers, or remote attackers if the port is exposed — can exploit this with a single HTTP POST.

### Recommendation

In `handle`, verify the request originates from the configured CKB node before calling `update_block_template`. Concretely:

1. Compare the TCP peer address against the host portion of `config.rpc_url` and reject connections from any other source.
2. Alternatively, require an HMAC or shared-secret token in a request header, generated by the CKB node and verified by the miner.
3. At minimum, enforce that `listen` can only be bound to loopback (`127.0.0.1`) and document this restriction prominently.

    fn update_block_template(&self, block_template: BlockTemplate) {
        let work_id = block_template.work_id.into();
        let updated = |id| {
            if id != work_id || id == 0 {
                Some(work_id)
            } else {
                None
            }
        };
        if self
            .current_work_id
            .fetch_update(Ordering::SeqCst, Ordering::SeqCst, updated)
            .is_ok()
        {
            let work: Work = block_template.into();
            if let Err(e) = self.new_work_tx.send(Works::New(work)) {
                error!("notify_new_block error: {:?}", e);
            }
        }
    }

async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
