Audit Report

## Title
Unbounded `last_n_numbers` Vec Allocation via Attacker-Controlled `difficulty_boundary` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

In `GetLastStateProofProcess::execute`, the guard at lines 201–205 bounds only the raw `last_n_blocks` field from the message, not the actual size of the `last_n_numbers` Vec computed later. By sending `difficulty_boundary = U256::one()` and `last_n_blocks = 1`, an attacker causes `get_first_block_total_difficulty_is_not_less_than` to return block 0 (genesis) as the boundary, and the subsequent one-sided clamp at line 313 never fires, resulting in `(0..N).collect()` — an N-element allocation proportional to the full chain height. The `complete_headers` loop then performs O(N log N) work, causing OOM or CPU exhaustion on the serving node.

## Finding Description

**Guard bypass** — The check at lines 201–205:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // 1000
```

With `difficulties = []` and `last_n_blocks = 1`, evaluates to `0 + 2 = 2 ≤ 1000` and passes. [1](#0-0) 

**Else-branch entry** — The condition at line 291 is `last_block_number - start_block_number <= last_n_blocks`. With `start_block_number = 0`, `last_n_blocks = 1`, and `last_block_number = N` (mainnet tip), `N ≤ 1` is false, so execution enters the `else` branch. [2](#0-1) 

**Early return in `get_first_block_total_difficulty_is_not_less_than`** — With `start_block_number = 0` and `min_total_difficulty = U256::one()`, the function immediately checks whether genesis total difficulty ≥ 1. On any real chain this is true, so it returns `Some((0, genesis_td))`, setting `difficulty_boundary_block_number = 0`. [3](#0-2) 

**One-sided clamp does not fire** — The only adjustment at lines 313–316 only expands the window when it is too small (`N - 0 < 1` is false for any N > 0). There is no upper-bound clamp. [4](#0-3) 

**Unbounded allocation** — `(0..N).collect::<Vec<_>>()` produces an N-element Vec with no cap. [5](#0-4) 

**Expensive loop** — `complete_headers` iterates over every element, calling `get_ancestor` (O(log N) skip-list traversal), `get_block`, `calc_uncles_hash`, and `chain_root_mmr(...).get_root()` per block — O(N log N) total work, all in a single async task. [6](#0-5) 

The constant `GET_LAST_STATE_PROOF_LIMIT = 1000` is confirmed in `constant.rs`. [7](#0-6) 

## Impact Explanation

A single malicious peer can cause the light-client server to allocate hundreds of megabytes and perform O(N log N) CPU work per request, with N equal to the full chain height. On mainnet this is sufficient to OOM-crash the node or render it unresponsive. This matches the allowed CKB bounty impact: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation

The light-client P2P protocol is unauthenticated and open to any peer. The exploit message is trivially constructable: the attacker only needs the current tip hash (publicly broadcast via `SendLastState`) and sets `difficulty_boundary = 0x1`, `last_n_blocks = 1`, `difficulties = []`, `start_number = 0`. No PoW, stake, or special privilege is required. The attack is repeatable at will from a single peer.

## Recommendation

After resolving `difficulty_boundary_block_number`, add an upper-bound clamp so that `last_n_numbers` cannot exceed `last_n_blocks` elements:

```rust
// Existing: expand window when too few blocks after boundary
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
// ADD: shrink window when too many blocks after boundary
if last_block_number - difficulty_boundary_block_number > last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

Or replace both with a single `clamp`. Additionally, restructure the guard at lines 201–205 to bound the effective `last_n_numbers` size (i.e., the resolved window width), not just the raw `last_n_blocks` field from the message.

## Proof of Concept

Send the following `GetLastStateProof` message to a synced light-client server node:

```
GetLastStateProof {
    last_hash:           <current tip hash from SendLastState>,
    start_hash:          <genesis block hash>,
    start_number:        0,
    last_n_blocks:       1,
    difficulty_boundary: 0x0000...0001,  // U256::one()
    difficulties:        [],
}
```

Execution trace:
1. Guard: `0 + 1*2 = 2 ≤ 1000` → **passes**
2. `N - 0 > 1` → enters `else` branch
3. `get_first_block_total_difficulty_is_not_less_than(0, N, 1)` → genesis_td ≥ 1 → returns `Some((0, genesis_td))`
4. `difficulty_boundary_block_number = 0`
5. `N - 0 < 1` → **false**, no adjustment
6. `(0..N).collect::<Vec<_>>()` → **N-element Vec allocated**
7. `complete_headers` loops N times → **OOM / CPU exhaustion**
