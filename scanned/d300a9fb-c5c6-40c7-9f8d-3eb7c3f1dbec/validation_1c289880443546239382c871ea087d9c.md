Audit Report

## Title
Integer Overflow in Guard Check Bypasses Per-Message Work Limit, Enabling O(chain_length) DB Read DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
`GetLastStateProofProcess::execute()` performs an unchecked multiplication `(last_n_blocks as usize) * 2` in its sole rate-limiting guard. Sending `last_n_blocks = 2^63` causes this multiplication to wrap to `0` in Rust release mode, bypassing the `GET_LAST_STATE_PROOF_LIMIT = 1000` check. The same value then causes the server to collect every block number from `start_block_number` to the chain tip and perform multiple DB reads per block via `complete_headers`, resulting in O(chain_length) work per message.

## Finding Description

**Guard overflow (lines 199–205):**

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

On a 64-bit host, `usize` is 64 bits. With `last_n_blocks = 2^63`:
- `last_n_blocks as usize` = `2^63` (lossless cast)
- `2^63 * 2` = `2^64` → wraps to **`0`** in Rust release mode (no overflow check by default)
- Guard becomes `difficulties.len() + 0 > 1000`, which is false for any `difficulties.len() ≤ 1000`

The `Uint64` wire type accepts any 64-bit value; `2^63` is a valid encoding.

**Post-bypass amplification (lines 291–297):**

```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks   // always true: real chain << 2^63
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    (sampled_numbers, last_n_numbers)
``` [2](#0-1) 

Because `last_n_blocks = 2^63` dwarfs any realistic chain length, the condition is always true, and `last_n_numbers` becomes a `Vec` of every block number from `start_block_number` to the tip.

**DB amplification in `complete_headers` (lines 132–146):**

For each entry in `last_n_numbers`, `complete_headers` calls:
- `snapshot.get_ancestor(last_hash, *number)` — ancestor traversal
- `snapshot.get_block(&ancestor_header.hash())` — block DB read
- `snapshot.chain_root_mmr(*number - 1).get_root()` — MMR root computation [3](#0-2) 

This is O(chain_length) DB reads per message. The limit constant itself is: [4](#0-3) 

**Concrete attack path:**
1. Attacker obtains any valid main-chain tip hash (publicly observable).
2. Crafts `GetLastStateProof` with `last_n_blocks = 2^63`, `start_number = 0`, `start_hash = genesis_hash`, empty `difficulties`.
3. Guard check: `0 + 0 > 1000` → false → passes.
4. `start_block_number == 0` → `reorg_last_n_numbers` is empty (line 238–243).
5. `last_block_number - 0 <= 2^63` → true → `last_n_numbers = (0..last_block_number).collect()`.
6. `complete_headers` iterates every block number → O(chain_length) DB reads. [5](#0-4) 

## Impact Explanation

A single ~100-byte P2P message forces the full node to allocate a `Vec<BlockNumber>` of length equal to the entire chain and execute multiple DB reads per entry. On mainnet (millions of blocks), this saturates disk I/O and CPU, and the `Vec` allocation alone can trigger OOM. Multiple concurrent peers can send this message with no per-peer rate limit visible in this code path. This matches **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation

- No privilege required — any peer speaking the `LightClient` protocol can send this message.
- `2^63` is trivially constructable as a `Uint64` wire value.
- Rust release builds wrap on integer overflow by default; this is not a debug-only behavior.
- The attack is stateless and repeatable; the attacker only needs a valid main-chain hash (publicly observable from any block explorer or light node).

## Recommendation

Replace the unchecked arithmetic with overflow-safe operations:

```rust
// Option A: saturating arithmetic
let n_blocks_contribution = (last_n_blocks as usize).saturating_mul(2);
if self.message.difficulties().len().saturating_add(n_blocks_contribution)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: early rejection of oversized last_n_blocks
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, add a standalone cap on `difficulties.len()` independent of `last_n_blocks`, and consider a per-peer request rate limit for this message type.

## Proof of Concept

```rust
// Craft a GetLastStateProof message:
let content = packed::GetLastStateProof::new_builder()
    .last_hash(valid_tip_hash)          // any valid main-chain hash
    .start_hash(genesis_hash)
    .start_number(0u64.pack())
    .last_n_blocks((u64::MAX / 2 + 1).pack())  // = 2^63; (as usize)*2 wraps to 0
    .difficulty_boundary(some_u256.pack())
    // difficulties: empty — guard check: 0 + 0 > 1000 → false → passes
    .build();
// Send via LightClient P2P protocol.
// Server allocates Vec of length == chain_tip_number and performs O(chain_length) DB reads.
// Observable: server CPU/IO spikes; OOM or timeout on large chains.
```
