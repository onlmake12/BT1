Audit Report

## Title
Off-by-One in `MAX_VMS_COUNT` Guard Allows 17th VM to Boot — (`script/src/scheduler.rs`)

## Summary

The spawn guard in `process_message_box` uses strict `>` instead of `>=` when comparing against `MAX_VMS_COUNT = 16`. When exactly 16 VMs are live, the expression `16 > 16` evaluates to `false`, causing `boot_vm` to be called and a 17th VM to be inserted unconditionally. The stated invariant "total live VMs ≤ 16" is violated by exactly one.

## Finding Description

`MAX_VMS_COUNT` is defined as `16` at [1](#0-0)  In `process_message_box`, the guard for `Message::Spawn` is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
``` [2](#0-1) 

When the combined count equals exactly 16, `16 > 16` is `false`, so the guard does not fire and execution falls through to `boot_vm`. `boot_vm` unconditionally inserts the new VM into both `instantiated` and `states`: [3](#0-2) 

Only on the *next* spawn attempt does `17 > 16` evaluate to `true`, returning `MAX_VMS_SPAWNED`. The correct operator is `>=`.

**Exploit path:**
1. Root VM (1 live VM) issues 15 `Message::Spawn` syscalls → total = 16 live VMs.
2. One more `Message::Spawn` is issued. Guard: `16 > 16` = `false` → `boot_vm` called → 17th VM created.
3. Subsequent spawn: `17 > 16` = `true` → correctly blocked.

The scheduler holds exclusive mutable access to `self` throughout `process_message_box`; there is no concurrent modification. This is a plain off-by-one, not a TOCTOU race.

## Impact Explanation

This is an incorrect implementation of CKB-VM scheduler behavior, matching the allowed impact: **High — Incorrect implementation or behavior of CKB-VM or system scripts.** The documented and enforced invariant that at most 16 VMs may exist simultaneously is violated. While the excess is bounded to exactly 1 VM and all nodes share the same off-by-one (preventing consensus deviation), the CKB-VM specification is concretely broken: a script can exceed the stated VM count limit, which constitutes incorrect VM behavior.

## Likelihood Explanation

Any `ScriptVersion::V2` script submitted in a transaction can trigger this. No privileged access, leaked keys, or majority hashpower is required. The path is fully deterministic and locally reproducible without any external dependencies.

## Recommendation

Change the guard operator from `>` to `>=`:

```rust
// Before (off-by-one):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
``` [4](#0-3) 

## Proof of Concept

A Rust invariant test in the scheduler test suite:

```rust
// Spawn 16 children from root (root = 1, so 15 spawns reach 16 total)
for _ in 0..15 {
    assert_eq!(spawn_child(), SUCCESS);
}
// 16th spawn: guard fires at 16 > 16 = false → 17th VM boots
assert_eq!(spawn_child(), SUCCESS); // off-by-one: should return MAX_VMS_SPAWNED
// Verify invariant: total VMs must be <= MAX_VMS_COUNT
assert!(scheduler.suspended.len() + scheduler.instantiated.len() <= MAX_VMS_COUNT as usize);
// This assertion FAILS: actual count is 17
```

A proptest fuzzer asserting `spawned_count <= MAX_VMS_COUNT` would catch this immediately on the 16th successful spawn from the root VM.
