Audit Report

## Title
`InnerPool::need_clean` samples only one HashMap child, enabling permanent orphan-pool cleanup suppression via epoch-spoofed shield blocks — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` uses `map.iter().next()` to inspect exactly one non-deterministically chosen child block when deciding whether to evict a parent group. Because `non_contextual_verify` does not check PoW or epoch numbers, any P2P peer can insert a "shield" block with a recent epoch alongside an expired-epoch sibling under the same fake parent hash. Whenever the shield block is iterated first, `need_clean` returns `false` and neither block is ever evicted, causing unbounded orphan-pool memory growth and eventual node OOM crash.

## Finding Description

`InnerPool::need_clean` at `chain/src/utils/orphan_block_pool.rs` lines 113–122 calls `map.iter().next()` on the inner `HashMap<Byte32, LonelyBlockHash>`, sampling exactly one child:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← only first entry
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
``` [1](#0-0) 

`clean_expired_blocks` calls `need_clean` per leader and only removes the group if it returns `true`: [2](#0-1) 

The precondition — inserting blocks with arbitrary epoch values — is trivially reachable. `asynchronous_process_block` in `chain_service.rs` calls `non_contextual_verify` before orphan insertion: [3](#0-2) 

`non_contextual_verify` invokes only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [4](#0-3) 

`BlockVerifier::verify` runs `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — no PoW, no epoch check: [5](#0-4) 

PoW and epoch validation live in `HeaderVerifier`, which requires the parent block (contextual) and is never called on this path: [6](#0-5) 

The `epoch_number` stored in `LonelyBlockHash` is taken directly from the block's header field with no validation: [7](#0-6) 

**Exploit flow:**
1. Attacker sends two `SendBlock` P2P messages sharing the same fake `parent_hash`.
2. Block A: epoch = 1 (expired when tip_epoch = 20, EXPIRED_EPOCH = 6).
3. Block B: epoch = 19 (recent, within EXPIRED_EPOCH of tip).
4. Both pass `non_contextual_verify` (no PoW/epoch gate) and are inserted into the orphan pool.
5. Every 60 seconds, `clean_expired_blocks` calls `need_clean` for the shared parent hash.
6. `map.iter().next()` returns either A or B non-deterministically. If B is returned, `need_clean` returns `false` and neither block is evicted.
7. Repeating with distinct fake parent hashes causes monotonically growing pool size.

## Impact Explanation

The orphan block pool has no hard size cap. Unlike the tx-orphan pool which enforces `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`, `OrphanBlockPool` is unbounded: [8](#0-7) 

The sole reclamation path is `clean_expired_blocks` fired every 60 seconds: [9](#0-8) 

With the shield technique, ~50% of inserted expired groups survive each cleanup pass. A sustained stream of (expired, shield) pairs under distinct fake parent hashes causes monotonically growing RSS, degrading node performance and eventually causing OOM. This matches the **High** impact: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation

- No PoW is required; any P2P peer can send arbitrary `SendBlock` messages.
- No privileged role is needed.
- The attack cost is minimal: two small crafted blocks per fake parent hash.
- The 60-second cleanup interval means a modest send rate suffices to outpace cleanup.
- HashMap iteration non-determinism is in the attacker's favor: on average half of all inserted groups escape cleanup permanently per pass.

## Recommendation

Replace the single-sample check with a check over all children using the minimum epoch:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().any(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

Additionally, add PoW verification (via `HeaderVerifier`'s `PowVerifier`) to `non_contextual_verify` so that epoch-spoofed blocks are rejected at the network boundary before reaching the orphan pool.

## Proof of Concept

```rust
// Unit test — insert two children under the same fake parent,
// one expired, one recent; assert the expired one is always cleaned.
let pool = OrphanBlockPool::with_capacity(10);
let fake_parent = random_byte32();

// expired: epoch 1, tip will be 20, EXPIRED_EPOCH = 6 → 1 + 6 < 20 → expired
let expired = make_lonely_block_hash(fake_parent.clone(), epoch = 1);
// shield: epoch 19 → 19 + 6 >= 20 → not expired
let shield  = make_lonely_block_hash(fake_parent.clone(), epoch = 19);

pool.insert(expired);
pool.insert(shield);

let removed = pool.clean_expired_blocks(20);

// BUG: if shield is iterated first by HashMap, removed.len() == 0
// The expired block is never cleaned.
// Run repeatedly to observe ~50% failure rate.
assert_eq!(removed.len(), 1);  // fails ~50% of the time
```

The existing test `test_remove_expired_blocks` at `chain/src/tests/orphan_block_pool.rs` lines 233–262 only inserts blocks with a single uniform epoch per parent, so it does not exercise the mixed-epoch case and does not catch this bug. [10](#0-9)

    fn verify(&self, header: &Self::Target) -> Result<(), Error> {
        // POW check first
        PowVerifier::new(header, self.consensus.pow_engine().as_ref()).verify()?;
        let parent_fields = self

            self.data_loader,
            header,
