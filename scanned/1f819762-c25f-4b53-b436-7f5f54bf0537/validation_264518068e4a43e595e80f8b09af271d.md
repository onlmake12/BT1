Audit Report

## Title
Unbounded Cell Iteration in `get_cells_capacity` Enables RPC-Triggered Indexer Exhaustion â€” (`File: util/indexer/src/service.rs`)

## Summary
`IndexerHandle::get_cells_capacity` iterates every matching live cell in RocksDB without any count cap, relying solely on a 10-second wall-clock `TimeoutIterator` as its only bound. Unlike `get_cells` and `get_transactions`, which enforce both a caller-supplied `limit` and the `request_limit` field, `get_cells_capacity` consults neither. An unprivileged caller can issue concurrent requests with a broad prefix (e.g., the secp256k1 `code_hash` with empty args) to sustain I/O and CPU exhaustion on the indexer service, making it unavailable to legitimate callers for the duration of the attack.

## Finding Description
`get_cells` enforces `limit > self.request_limit` at lines 216â€“221 and terminates the iterator chain with `.take(limit)` at line 371. `get_transactions` applies the same guard at lines 392â€“397 and uses `.take(limit)` at line 674.

`get_cells_capacity` (lines 686â€“854) takes no `limit` parameter at all. It constructs a `TimeoutIterator` at line 720 and chains `.take_while(|(key, _value)| key.starts_with(&prefix)).filter_map(...)` at lines 728â€“835 with no `.take(n)` count cap. The `request_limit` field stored on `IndexerHandle` (line 170) is never read inside this function. For each matching key the function performs an additional RocksDB point-lookup (`Key::OutPoint`) at lines 748â€“752, deserializes the full cell value, and applies all filter checks â€” making each iteration CPU- and I/O-intensive.

The `TimeoutIterator::next` (lines 55â€“61) only checks `self.start_time.elapsed() > self.timeout`; it does not bound the number of RocksDB reads or the CPU work done per request. With a default `timeout_limit` of 10 seconds (line 99), each request can hold the indexer busy for the full window.

The rich-indexer variant (`AsyncRichIndexerHandle::get_cells_capacity`, lines 26â€“27 of `get_cells_capacity.rs`) issues `SUM(capacity)` with no `LIMIT` clause, forcing the database engine to scan all matching rows.

## Impact Explanation
The indexer RPC service becomes unavailable to legitimate callers during the attack window. A small number of concurrent `get_cells_capacity` requests with a broad prefix saturates the indexer's I/O and CPU for up to 10 seconds per request. This matches **Note (0â€“500 points): Any local RPC API crash**, as the indexer RPC is rendered non-functional for the duration. The node's consensus and P2P layers are unaffected.

## Likelihood Explanation
The indexer RPC is exposed by default on any CKB node with the indexer module enabled. No authentication is required. The secp256k1 lock `code_hash` is publicly documented and matches every standard wallet cell on mainnet. The attack requires a single JSON-RPC call with a well-known parameter and is trivially repeatable from any network-connected client.

## Recommendation
1. Add a `limit: Option<Uint32>` parameter to `get_cells_capacity` and enforce `self.request_limit` as the maximum, consistent with `get_cells` and `get_transactions`.
2. Alternatively, add a hard internal `.take(self.request_limit)` cap on the iterator chain inside `get_cells_capacity` regardless of caller input, returning an error if the cap is reached.
3. For the rich-indexer variant, add a `LIMIT` clause to the SQL aggregate query.
4. Retain `TimeoutIterator` as a secondary defense only.

## Proof of Concept
Send the following to a CKB node with the indexer enabled:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [{
    "script": {
      "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",

      "args": "0x"

    "script_type": "lock",
    "script_search_mode": "prefix"

Each iteration performs a RocksDB point-lookup at lines 748â€“752. Issuing this request from multiple concurrent connections sustains exhaustion for the full 10-second timeout window per request.
