### Title
Missing Low-S Enforcement in `Signature::is_valid()` Allows Alert Signature Malleability via P2P - (File: `util/crypto/src/secp/signature.rs`)

### Summary

`Signature::is_valid()` only checks that the `s` component is in the range `[1, N)`, but does not enforce that `s` is in the lower half of the secp256k1 curve order (`s <= N/2`). Because `verify_m_of_n` uses `sig.recover()` — which accepts both low-s and high-s forms — an unprivileged P2P peer that intercepts a legitimate network alert can malleate its signatures (replacing each `s` with `N - s` and flipping `v`) and re-broadcast the modified alert. The malleated alert passes all verification checks and is stored and relayed by receiving nodes, causing network-wide inconsistency in which form of the alert is propagated.

### Finding Description

`Signature::is_valid()` in `util/crypto/src/secp/signature.rs` enforces:

```rust
self.v() <= 1 && h_r < N && h_r >= ONE && h_s < N && h_s >= ONE
``` [1](#0-0) 

The upper bound is `h_s < N` (full curve order), not `h_s <= N/2` (lower half). High-s signatures therefore pass `is_valid()`.

`Verifier::verify_signatures()` in `util/network-alert/src/verifier.rs` uses `sig.is_valid()` as the sole pre-filter before calling `verify_m_of_n`:

```rust
if sig.is_valid() {
    Some(sig)
} else { ... }
``` [2](#0-1) 

`verify_m_of_n` in `util/multisig/src/secp256k1.rs` recovers the public key via `sig.recover(message)` without any low-s check: [3](#0-2) 

Because ECDSA malleation `(r, s, v) → (r, N-s, 1-v)` preserves the recovered public key, a malleated high-s signature passes both `is_valid()` and `verify_m_of_n()`.

The P2P entry point is `AlertRelayer::received()` in `util/network-alert/src/alert_relayer.rs`: [4](#0-3) 

Any connected peer can send an alert. After the `has_received(alert_id)` early-exit check, the alert is passed to `verify_signatures()`. A malleated alert with high-s signatures passes this check, is stored in `received_alerts` keyed by `alert_id`, and is relayed to all other connected peers.

The `Notifier::add()` deduplicates by `alert_id`: [5](#0-4) 

Once a node stores the malleated version, the legitimate version (same `alert_id`) is silently dropped. The CHANGELOG explicitly records the removal of the low-s check: `#938: Remove low S check from util-crypto`. [6](#0-5) 

### Impact Explanation

An unprivileged P2P peer that receives a legitimate alert (with valid low-s signatures from Nervos Foundation key holders) can:

1. Compute the malleated form: `s' = N - s`, `v' = 1 - v` for each signature.
2. Broadcast the malleated alert to other nodes before the legitimate version arrives.
3. Receiving nodes accept and store the malleated version (it passes all checks).
4. When the legitimate version arrives, `has_received(alert_id)` returns `true` and it is silently discarded.
5. The malleated version is further propagated to newly connecting peers via `connected()`.

The result is network-wide inconsistency: nodes are split between the original and malleated forms of the same alert. While the alert message content is unchanged (so users still see the correct warning), the alert system's integrity is undermined — the network relays a form of the alert that was not produced by the authorized signers.

### Likelihood Explanation

Any peer connected to the CKB P2P network can execute this attack. No privileged access, key material, or majority hashpower is required. The attacker only needs to receive the alert before other nodes and re-broadcast the malleated form. This is straightforward for a well-connected peer.

### Recommendation

Enforce the low-s constraint in `Signature::is_valid()`:

```rust
const HALF_N: H256 = h256!("0x7fffffff_ffffffff_ffffffff_ffffffff_5d576e73_57a4501d_dfe92f46_681b20a0");

// Change:
self.v() <= 1 && h_r < N && h_r >= ONE && h_s < N && h_s >= ONE
// To:
self.v() <= 1 && h_r < N && h_r >= ONE && h_s <= HALF_N && h_s >= ONE
```

This mirrors the check used in `BondedECDSAKeep` in the referenced report and ensures only canonical (non-malleable) signatures are accepted by the alert verifier.

### Proof of Concept

Given a legitimate alert with signature `(r, s, v)` where `s` is in the lower half:

1. Compute `s' = N - s` and `v' = 1 - v`.
2. Construct a new alert with the same raw fields but replace each signature byte array with `(r || s' || v')`.
3. Submit via RPC `send_alert` or inject directly into the P2P network.
4. The node calls `Signature::is_valid()` → passes (since `s' < N`).
5. `verify_m_of_n` calls `sig.recover(message)` → recovers the same authorized public key.
6. The malleated alert is stored and relayed. The legitimate alert, when it arrives, is rejected by `has_received(alert_id)`. [7](#0-6) [8](#0-7) [9](#0-8)

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }
