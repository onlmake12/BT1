### Title
Unauthenticated `CompactBlock.proposals` Field Consumed Before Integrity Check Enables Fake Proposal Injection — (`sync/src/relayer/compact_block_process.rs`)

---

### Summary

The `proposals` field of the `CompactBlock` P2P message is used to trigger `GetBlockProposal` network requests before any integrity check against the block header's `proposals_hash` commitment. A malicious unprivileged peer can forge this field on an otherwise-valid compact block, causing victim nodes to send spurious proposal requests, pollute their `inflight_proposals` state, and delay block propagation.

---

### Finding Description

**The data flow**

When a `CompactBlock` is received, `CompactBlockProcess::execute` in `sync/src/relayer/compact_block_process.rs` performs the following sequence:

1. Verifies the block header (PoW, timestamp, etc.) — this covers only `RawHeader` fields.
2. Calls `CompactBlockVerifier::verify` — checks only `prefilled_transactions` ordering and `short_ids` uniqueness.
3. **Immediately consumes `compact_block.proposals()` to fire `request_proposal_txs`** — before the block is reconstructed or any hash commitment is checked.
4. Reconstructs the block and eventually submits it for full verification, where `MerkleRootVerifier` checks `proposals_hash`. [1](#0-0) 

**Why the field is unauthenticated at the relay layer**

The block header hash covers only the `RawHeader` struct: [2](#0-1) 

The `proposals_hash` field inside `RawHeader` commits to the block's proposals, but this commitment is only verified during full block processing in `MerkleRootVerifier`: [3](#0-2) 

At the P2P relay layer, `non_contextual_check` only enforces a count limit on `compact_block.proposals()`: [4](#0-3) 

And `CompactBlockVerifier::verify` checks only `PrefilledVerifier` and `ShortIdsVerifier` — neither touches `proposals`: [5](#0-4) 

**The `CompactBlock` schema confirms `proposals` is a separate, unchecked field:** [6](#0-5) 

**The only hash computed from a `CompactBlock` is the header hash:** [7](#0-6) 

**The tampered proposals are also used verbatim when reconstructing the block body:** [8](#0-7) 

**`request_proposal_txs` inserts the fake IDs into `inflight_proposals` and sends `GetBlockProposal` to the peer:** [9](#0-8) 

**`BlockProposalProcess` then submits any returned transactions directly to the tx pool:** [10](#0-9) 

---

### Impact Explanation

A malicious peer that intercepts a valid compact block (or constructs one with valid PoW) and replaces the `proposals` field with arbitrary `ProposalShortId` values causes the victim to:

1. **Send spurious `GetBlockProposal` requests** for proposal IDs that do not belong to the block, wasting outbound bandwidth.
2. **Pollute `inflight_proposals`** with fake IDs, which can interfere with legitimate proposal tracking for the duration of the TTL window.
3. **Accept arbitrary transactions into the tx pool** if the attacker responds to the `GetBlockProposal` with valid (but unrelated) transactions — these would pass tx-pool validation independently.
4. **Delay block propagation**: the block is eventually rejected when `MerkleRootVerifier` detects the `proposals_hash` mismatch, forcing the victim to re-fetch the block from a different peer.

The attacker peer is banned after the block is rejected, making this a one-shot attack per connection, but Sybil connections (multiple cheap peer connections) multiply the effect.

---

### Likelihood Explanation

Any unprivileged P2P peer can execute this attack. No mining capability is required if the attacker acts as a relay node that intercepts and modifies compact blocks from legitimate miners before forwarding them. The modification is trivial: replace the `proposals` field bytes in the molecule-encoded message. The attack is reachable on mainnet from any peer that can establish a connection.

---

### Recommendation

Before consuming `compact_block.proposals()` to fire `request_proposal_txs`, verify that the hash of the proposals matches the `proposals_hash` committed in the block header. Add this check to `non_contextual_check` or `CompactBlockVerifier::verify`:

```rust
let expected_proposals_hash = compact_block.header().raw().proposals_hash();
let actual_proposals_hash = compact_block.proposals().calc_proposals_hash();
if expected_proposals_hash != actual_proposals_hash {
    return StatusCode::ProtocolMessageIsMalformed.with_context(
        "CompactBlock proposals_hash mismatch"
    );
}
```

This mirrors the integrity check already enforced for uncle proposals in `UnclesVerifier`: [11](#0-10) 

---

### Proof of Concept

1. Attacker establishes a P2P connection to a victim CKB node.
2. Attacker receives a valid `CompactBlock` relayed from a legitimate miner (valid header, valid PoW).
3. Attacker replaces `compact_block.proposals` with a list of arbitrary `ProposalShortId` values (e.g., 10-byte IDs for transactions the attacker controls or that do not exist).
4. Attacker forwards the tampered `CompactBlock` to the victim.
5. Victim's `compact_block_process.rs` passes header verification (header hash is unaffected by `proposals`), then at line 81 extracts the fake proposals and calls `request_proposal_txs`.
6. Victim sends `GetBlockProposal { block_hash, proposals: [fake_ids...] }` back to the attacker.
7. Attacker responds with `BlockProposal { transactions: [...] }` containing arbitrary valid transactions; these are submitted to the victim's tx pool via `notify_txs_async`.
8. Victim attempts to reconstruct the block using the tampered proposals; `MerkleRootVerifier` detects `proposals_hash` mismatch and rejects the block.
9. Victim bans the attacker peer — but the attacker has already caused spurious requests, state pollution, and tx-pool injection.
