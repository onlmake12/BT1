### Title
Unbounded Per-Peer Request Rate on Filter Protocol Enables Bandwidth and RocksDB Read Exhaustion — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

### Summary

The `BlockFilter` protocol handler processes `GetBlockFilters` messages with no per-peer rate limiting. A single unprivileged remote peer can flood the node with repeated requests, each triggering up to 1000 RocksDB reads and a ~1.8 MB response, with no throttle, ban, or backpressure applied.

### Finding Description

The `Relayer` protocol handler explicitly constructs a `governor::RateLimiter` keyed by `(PeerIndex, message_id)` and rejects requests exceeding 30/second per peer per message type with `StatusCode::TooManyRequests`. [1](#0-0) [2](#0-1) 

The `BlockFilter` handler has no equivalent. Its `received` → `process` → `try_process` chain dispatches directly to `GetBlockFiltersProcess::execute()` with zero rate-limit checks. [3](#0-2) [4](#0-3) 

`GetBlockFiltersProcess::execute()` iterates up to `BATCH_SIZE = 1000` blocks, performing two RocksDB reads per block (`get_block_hash` + `get_block_filter`), and sends a response capped at 1.8 MB. The 1.8 MB cap (PR #4972) prevents oversized frames but does **not** prevent repeated requests. [5](#0-4) [6](#0-5) 

The `TooManyRequests` status code exists in the sync module but is only wired into the Relayer, not the Filter handler. [7](#0-6) 

### Impact Explanation

Per request: up to ~2000 RocksDB reads (1000 × `get_block_hash` + 1000 × `get_block_filter`) and up to ~1.8 MB outbound. A single attacker peer sending requests in a tight loop can saturate the node's RocksDB read IOPS and outbound bandwidth at negligible cost. This affects any node with `block-filter` enabled.

### Likelihood Explanation

The Filter protocol is opt-in (`block-filter` feature), but nodes that enable it (e.g., to serve light clients) are fully exposed. The attack requires only a valid P2P connection and a loop sending `GetBlockFilters(start_number=0)`. No special privileges, no PoW, no keys needed.

### Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32)>` to `BlockFilter` (mirroring the `Relayer` pattern) and check it in `try_process` before dispatching `GetBlockFiltersProcess`, `GetBlockFilterHashesProcess`, and `GetBlockFilterCheckPointsProcess`. Return `StatusCode::TooManyRequests` on excess. A limit of 10–30 requests/second per peer per message type is consistent with the Relayer's existing cap.

### Proof of Concept

1. Connect a single peer to a node with `block-filter` enabled and filters built to tip.
2. In a tight loop, send `GetBlockFilters { start_number: 0 }` over the Filter protocol (protocol id `121`).
3. Observe: no ban is applied, no rate-limit response is returned, each iteration triggers ~2000 RocksDB reads and returns ~1.8 MB. Outbound bandwidth and DB read IOPS grow linearly with send rate, unbounded per peer. [8](#0-7)

    pub async fn execute(self) -> Status {
        let active_chain = self.filter.shared.active_chain();
        let start_number: BlockNumber = self.message.to_entity().start_number().into();
        let latest: BlockNumber = active_chain.get_latest_built_filter_block_number();

        if latest >= start_number {
            let mut block_hashes = Vec::new();
            let mut filters = Vec::new();
            let mut current_content_size = 0;
            current_content_size += 8; // Size of start_number
            current_content_size += 4 * 2; // Size of the header field `full-size` of `block_hash` and `block_filter`
            let mut block_number = start_number;
            for _ in 0..BATCH_SIZE {
                if let Some(block_hash) = active_chain.get_block_hash(block_number) {
                    if let Some(block_filter) = active_chain.get_block_filter(&block_hash) {
                        if current_content_size
                            + block_hash.as_slice().len()
                            + 4
                            + block_filter.as_slice().len()
                            + 4
                            >= (1.8 * 1024.0 * 1024.0) as usize
                        {
                            // Break if the encoded size of `block_hash` + `block_filter` + `start_number` + molecule header increase reaches 1.8MB, to avoid frame size too large
                            break;
                        }
                        current_content_size +=
                            block_hash.as_slice().len() + block_filter.as_slice().len() + 4;
                        block_hashes.push(block_hash);
                        filters.push(block_filter);
                    } else {
                        break;
                    }
                } else {
                    break;
                }
                let Some(next_block_number) = block_number.checked_add(1) else {
                    break;
                };
                block_number = next_block_number;
            }
            let content = packed::BlockFilters::new_builder()
                .start_number(start_number)
                .block_hashes(block_hashes)
                .filters(filters)
                .build();
            let message = packed::BlockFilterMessage::new_builder()
                .set(content)
                .build();
            async_send_message_to(&self.nc, self.peer, &message).await
        } else {
            Status::ignored()
        }
    }
