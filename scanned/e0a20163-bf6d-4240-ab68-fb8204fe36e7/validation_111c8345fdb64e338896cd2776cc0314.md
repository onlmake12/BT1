### Title
Unbounded Spot-Fee-Rate Drives Massive Vector Allocation in `WeightUnitsFlow::do_estimate`, Enabling RPC-Triggered Node Crash - (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

The `WeightUnitsFlow` fee-estimator algorithm derives `max_bucket_index` directly from the **highest fee rate currently in the mempool** — a spot reading that any tx-pool submitter can inflate at will. Because the two bucket vectors are allocated with size `max_bucket_index + 1` and there is no cap on that index, an attacker who places a single high-fee transaction in the pool and then calls the public `estimate_fee_rate` RPC can force the node to attempt a multi-gigabyte (or larger) heap allocation, crashing the process.

---

### Finding Description

`WeightUnitsFlow::do_estimate` is called every time `estimate_fee_rate` is invoked via the JSON-RPC `Experiment` module. The function:

1. Reads `max_fee_rate` from the first (highest) element of `sorted_current_txs`, which is the live mempool snapshot — a pure spot value.
2. Passes it to `max_bucket_index_by_fee_rate`, which for any fee rate above 2 000 000 shannons/KW returns `(fee_rate + 11_500_000) / 100_000`.
3. Allocates **two** `Vec<u64>` of length `max_bucket_index + 1` with no upper-bound guard. [1](#0-0) [2](#0-1) 

`FeeRate::calculate` is:

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
``` [3](#0-2) 

`saturating_mul` caps the intermediate product at `u64::MAX`, so the maximum reachable fee rate is `u64::MAX / min_weight`. With a minimum transaction weight of ~100 bytes and a fee of 1 000 000 CKB (10¹⁴ shannons):

```
fee_rate = 10¹⁴ × 1000 / 100 = 10¹⁵ shannons/KW
max_bucket_index ≈ 10¹⁵ / 100_000 = 10¹⁰
allocation per vec = 10¹⁰ × 8 bytes = 80 GB  (×2 = 160 GB)
```

The `check_tx_fee` admission gate enforces only a **minimum** fee rate; there is no maximum: [4](#0-3) 

The `estimate_fee_rate` RPC is publicly reachable with no authentication: [5](#0-4) 

The call chain is:

```
RPC estimate_fee_rate
  → TxPoolService::estimate_fee_rate (process.rs:945)
    → FeeEstimator::estimate_fee_rate (mod.rs:92)
      → WeightUnitsFlow::estimate_fee_rate (weight_units_flow.rs:164)
        → do_estimate → vec![0u64; max_bucket_index + 1]  ← OOM
``` [6](#0-5) 

The `max_bucket_index_by_fee_rate` function has no ceiling: [7](#0-6) 

---

### Impact Explanation

An attacker who controls a wallet with a large CKB balance can submit a single valid transaction whose fee rate is extremely high (e.g., 1 000 000 CKB fee, ~100-byte body). Once that transaction sits in the mempool, a single call to `estimate_fee_rate` causes the node to attempt allocating ~160 GB of heap memory. On any real server this triggers an OOM kill or a Rust allocation panic, crashing the node process. The crash is repeatable: the attacker re-submits the transaction after each restart and calls the RPC again.

Secondary effect: even at lower fee rates (e.g., 10 000 CKB fee → ~1.6 GB allocation), the call stalls the async runtime long enough to degrade all other RPC and P2P services.

---

### Likelihood Explanation

- The `estimate_fee_rate` RPC is unauthenticated and publicly documented.
- The tx-pool admission check has no upper fee-rate bound.
- The attacker needs only one valid UTXO with sufficient capacity; they do not need to mine a block or control any privileged role.
- The attack is repeatable at low marginal cost after the initial CKB outlay (the transaction is never mined while the node is crashing).
- The `WeightUnitsFlow` estimator is a supported, documented algorithm variant.

---

### Recommendation

1. **Cap `max_bucket_index`** to a safe constant (e.g., 200, matching the highest bucket the algorithm was designed for) inside `do_estimate` before any allocation:

   ```rust
   let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
       .min(MAX_BUCKET_INDEX);
   ```

2. **Clamp fee rate** at pool admission to a protocol-defined maximum (e.g., `10 * min_fee_rate` or an absolute cap), rejecting transactions whose fee rate exceeds it.

3. **Add an overflow-safe path** in `max_bucket_index_by_fee_rate` using `checked_add` and returning a capped value on overflow.

---

```rust

//   input capacity  = 1_000_000 CKB = 10^14 shannons

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }

    pub(crate) async fn estimate_fee_rate(
        &self,
        estimate_mode: EstimateMode,
        enable_fallback: bool,
    ) -> Result<FeeRate, AnyError> {
        let all_entry_info = self.tx_pool.read().await.get_all_entry_info();
        match self
            .fee_estimator
            .estimate_fee_rate(estimate_mode, all_entry_info)
        {
            Ok(fee_rate) => Ok(fee_rate),
            Err(err) => {
                if enable_fallback {
                    let target_blocks =
                        FeeEstimator::target_blocks_for_estimate_mode(estimate_mode);
                    self.tx_pool
                        .read()
                        .await
                        .estimate_fee_rate(target_blocks)
                        .map_err(Into::into)
                } else {
                    Err(err.into())
                }
            }
        }
    }
