### Title
Sentry Integration Forwards Unsanitized Log Breadcrumbs and Operational Data to Third-Party Service Without Scrubbing - (`util/app-config/src/sentry_config.rs`)

### Summary
CKB uses Sentry for error reporting on testnet/preview/staging networks. The `before_send` hook in `sentry_config.rs` only groups/deduplicates events by error type but performs **no data scrubbing**. Combined with the logger automatically enabling `emit_sentry_breadcrumbs` whenever Sentry is active, every log record — including those containing transaction data, peer addresses, ban reasons, and other operational details — is forwarded to `sentry.nervos.org` without sanitization.

### Finding Description

**1. Sentry is enabled by default on testnet/preview/staging.**

The default DSN `https://dda4f353e15f4b62800d273a2afe70c2@sentry.nervos.org/4` is baked into the template configs for testnet, preview, and staging: [1](#0-0) [2](#0-1) 

**2. `emit_sentry_breadcrumbs` is automatically enabled when Sentry is active.**

In `SetupGuard::from_setup`, if `is_sentry_enabled` is true and the logger config does not explicitly override `emit_sentry_breadcrumbs`, it is set to `true`: [3](#0-2) 

**3. Every log record is forwarded to Sentry as a breadcrumb.**

Inside `Logger::log`, when `emit_sentry_breadcrumbs` is true, `breadcrumb_from_record(record)` is called for every matching log record and added to the Sentry scope: [4](#0-3) 

**4. The `before_send` hook performs no data scrubbing.**

The hook only groups events by error type (DB errors, SQLite failures) or drops known-noisy events. It does not redact, mask, or strip any field from the event payload, breadcrumbs, or extra data: [5](#0-4) 

**5. `org_contact` is explicitly sent to Sentry as extra data.**

A node operator who sets `org_contact` in `ckb.toml` has that value sent to Sentry as an extra field, potentially without realizing it is transmitted to a third-party service: [6](#0-5) 

**6. `capture_message` calls include peer-derived data.**

In `tx-pool/src/process.rs`, `ban_malformed` explicitly calls `capture_message` with the ban reason string (which is derived from the transaction rejection reason and peer index) and sends it to Sentry: [7](#0-6) 

Similarly in `network/src/network.rs`, protocol panic details are sent via `capture_message`: [8](#0-7) 

**7. Panic messages (including payload) are logged and forwarded as breadcrumbs.**

The panic hook logs the full panic message and backtrace via `log::error!`. Because breadcrumbs are enabled, this log record is forwarded to Sentry. If a panic occurs during transaction or script processing, the panic payload (which may include raw data from the triggering input) is included: [9](#0-8) 

### Impact Explanation

On testnet/preview/staging nodes (which have Sentry enabled by default), the following data is transmitted to `sentry.nervos.org` without any scrubbing:

- All log messages at or above the configured log level, as Sentry breadcrumbs. These include transaction processing errors, peer ban reasons, script verification failures, and other operational details that may contain transaction hashes, script arguments, or peer IP addresses.
- Panic payloads, which may include raw data from the input that triggered the panic (e.g., a malformed transaction's fields appearing in an error string).
- The `org_contact` value from `ckb.toml`, which is a user-provided string sent as Sentry extra data.
- Explicit `capture_message` calls that include peer-derived ban reasons.

This constitutes unintended disclosure of operational and potentially user-identifying data to a third-party service, analogous to the ShakeBugs mnemonic leak.

### Likelihood Explanation

Any unprivileged peer, transaction submitter, or block relayer can trigger log messages by sending malformed or invalid data. On testnet/preview/staging (the most common non-mainnet deployment targets), Sentry is enabled by default and breadcrumbs are automatically activated. No special access is required; the data flow is triggered by normal (or adversarial) network activity.

### Recommendation

- **Short term:** Add a data-scrubbing step in the `before_send` hook (`sentry_config.rs`) that strips or redacts breadcrumb messages and event extra fields before transmission. At minimum, breadcrumbs should be disabled or filtered to exclude log records that may contain transaction or peer data.
- **Long term:** Audit all `capture_message`, `set_extra`, and `add_breadcrumb` call sites to ensure no sensitive operational data (transaction content, peer-derived strings, user-configured contact info) is included in Sentry payloads. Consider whether Sentry breadcrumb forwarding should be opt-in rather than automatically enabled whenever Sentry is active.

### Proof of Concept

1. Run a CKB node on testnet (Sentry DSN is set by default in the generated `ckb.toml`).
2. Submit a malformed transaction via the RPC (`send_transaction`). The tx-pool will reject it, log an error, and call `ban_malformed`, which calls `capture_message` with the rejection reason.
3. Observe in the Sentry dashboard at `sentry.nervos.org` that the event appears, including the ban reason string and any breadcrumbs accumulated from prior log messages (which include transaction processing details, peer addresses, etc.).
4. Alternatively, trigger a panic in a protocol handler from a peer message — the panic payload is logged via `setup_panic_logger` and forwarded to Sentry as a breadcrumb, with no scrubbing applied by `before_send`.

```rust
fn setup_panic_logger() {
    let panic_logger = |info: &panic::PanicHookInfo| {
        let backtrace = Backtrace::new();
        let thread = thread::current();
        let name = thread.name().unwrap_or("unnamed");
        let location = info.location().unwrap(); // The current implementation always returns Some
        let msg = match info.payload().downcast_ref::<&'static str>() {
            Some(s) => *s,
            None => match info.payload().downcast_ref::<String>() {
                Some(s) => s,
                None => "Box<Any>",
            },
        };
        log::error!(
            target: "panic", "thread '{}' panicked at '{}': {}:{}{:?}",
            name,
            msg,
            location.file(),
            location.line(),
            backtrace,
        );
    };
    panic::set_hook(Box::new(panic_logger));
```

```rust
fn before_send(mut event: Event<'static>) -> Option<Event<'static>> {
    if let Some(name) = std::thread::current().name() {
        event.extra.insert("thread.name".to_string(), name.into());
    }

    let ex = match event
        .exception
        .values
        .first()
        .and_then(|ex| ex.value.as_ref())
    {
        Some(ex) => ex,
        None => return Some(event),
    };

    // Group events via fingerprint, or ignore

    if ex.starts_with("DBError failed to open the database") {
        event.level = Level::Warning;
        event.fingerprint = Cow::Borrowed(DB_OPEN_FINGERPRINT);
    } else if ex.contains("SqliteFailure") {
        event.level = Level::Warning;
        event.fingerprint = Cow::Borrowed(SQLITE_FINGERPRINT);
    } else if ex.starts_with("DBError the database version")
        || ex.contains("kind: AddrInUse")
        || ex.contains("kind: AddrNotAvailable")
        || ex.contains("IO error: No space left")
    {
        // ignore
        return None;
    }

    Some(event)
}
```
