### Title
Expired Transaction Descendants Remain in Pool with Stale Unresolvable Inputs — (File: `tx-pool/src/pool.rs`)

### Summary
The `remove_expired` function in `tx-pool/src/pool.rs` removes expired transactions from the pool's entry map but does not remove their child (descendant) transactions. Descendants remain in the pool with inputs that reference the expired parent's outputs — outputs that no longer exist anywhere — creating a data inconsistency between the pool's entry tracking and the actual validity of those transactions. This is a direct analog to the ERC1155 bug: one tracking system (the expired parent's entry) is cleaned up, while a second tracking system (the descendants' entries and their `edges.inputs` records) is left stale.

---

### Finding Description

`TxPool` maintains two tightly coupled tracking structures inside `PoolMap`:

1. **`entries: MultiIndexPoolEntryMap`** — the canonical set of pool transactions, indexed by `ProposalShortId`, score, status, and eviction key.
2. **`edges: Edges`** — maps each consumed `OutPoint` (input) to the `ProposalShortId` of the transaction consuming it, and tracks cell-dep and header-dep relationships.
3. **`links: TxLinksMap`** — tracks parent/child relationships between transactions. [1](#0-0) 

When a transaction expires, `remove_expired` is called: [2](#0-1) 

For each expired entry it calls `pool_map.remove_entry(&entry.proposal_short_id())`: [3](#0-2) 

`remove_entry` correctly cleans up the expired transaction's own edges and links. Specifically, `remove_entry_links` removes the expired tx as a parent from its children's link records, and `update_descendants_index_key` adjusts descendants' score keys: [4](#0-3) [5](#0-4) 

**However, the descendants themselves are never removed from `entries` or `edges`.** Their `edges.inputs` entries — which map the expired parent's output `OutPoint`s to the descendants' `ProposalShortId`s — remain intact. The descendants stay in the pool as if they are valid, pending or proposed transactions, even though their inputs reference outputs that do not exist on-chain or in the pool.

Compare this with every other removal path, which uses `remove_entry_and_descendants`: [6](#0-5) 

`remove_committed_tx` also calls `resolve_conflict` to evict descendants: [7](#0-6) 

`remove_by_detached_proposal` similarly calls `remove_entry_and_descendants`: [8](#0-7) 

`remove_expired` is the only removal path that calls bare `remove_entry`, leaving descendants orphaned in the pool.

---

### Impact Explanation

**Stale proposed transactions cause repeated miner block-assembly failures.**

If a descendant transaction (`tx_B`) was proposed (moved to `Status::Proposed`) before its parent (`tx_A`) expired, it remains in the proposed pool after `tx_A` is removed. The block assembler (`package_txs` → `TxSelector`) selects proposed transactions by score without re-validating inputs: [9](#0-8) 

`tx_B` will be included in every block template. When the miner mines a block containing `tx_B`, the block fails chain validation because `tx_B`'s inputs (the expired parent's outputs) do not exist. The miner discards the block, requests a new template, and the cycle repeats — because `tx_B` is never removed from the pool. This continues until `tx_B` itself expires (up to `expiry_hours` after the parent expired).

**Secondary impact — false conflict detection blocking legitimate transactions.**

The stale `edges.inputs` entries (`expired_parent_output → tx_B`) remain. `find_conflict_tx` consults `edges.inputs` to detect double-spends: [10](#0-9) 

Any new transaction attempting to spend the same outputs as `tx_B` will be rejected as a conflict, even though `tx_B` is invalid and its parent is gone.

**Pool capacity waste.** Stale descendants occupy pool space (counted in `total_tx_size`, `total_tx_cycles`, and status counters) until they individually expire, reducing available capacity for legitimate transactions.

---

### Likelihood Explanation

The attack is reachable by any unprivileged transaction sender:

1. Submit `tx_A` (parent) and `tx_B` (child spending `tx_A`'s outputs) to the pool via the standard RPC or P2P relay path.
2. Ensure `tx_B` is submitted slightly later than `tx_A` so it does not expire at the same time.
3. Wait for `tx_A` to expire (`expiry_hours`, default 12 hours).
4. `tx_B` remains in the pool in whatever status it held, with unresolvable inputs.

No privileged access, no majority hashpower, and no social engineering is required. The only cost to the attacker is submitting two transactions and waiting.

---

### Recommendation

Replace the bare `remove_entry` call in `remove_expired` with `remove_entry_and_descendants`, consistent with all other removal paths:

```rust
// tx-pool/src/pool.rs — remove_expired
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
    // Use remove_entry_and_descendants to also evict children
    // whose inputs now reference non-existent outputs.
    let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    for e in evicted {
        let reject = Reject::Expiry(e.timestamp);
        callbacks.call_reject(self, &e, reject);
    }
}
```

This mirrors the behavior of `remove_by_detached_proposal` and `limit_size`, ensuring both tracking systems (`entries` and `edges`/`links`) remain consistent when a transaction is expired.

---

### Proof of Concept

1. Submit `tx_A` (spends live cell `C1`, creates output `O1`) to the tx-pool.
2. Submit `tx_B` (spends `O1`) to the tx-pool one minute later.
3. Both are proposed (appear in a block's proposal zone).
4. After `expiry_hours` (default 12 h), `remove_expired` fires.
5. `tx_A` is removed; `tx_B` remains in `Status::Proposed` with `edges.inputs[O1] = tx_B`.
6. The block assembler includes `tx_B` in every subsequent block template.
7. Every mined block containing `tx_B` fails chain validation (`O1` does not exist on-chain).
8. The miner loops, wasting PoW work, until `tx_B` expires ~12 h later.
9. During this window, any attempt to submit `tx_C` spending `O1` is rejected by `find_conflict_tx` as a false double-spend.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_descendants_index_key(&mut self, parent: &TxEntry, op: EntryOp) {
        let descendants: HashSet<ProposalShortId> =
            self.links.calc_descendants(&parent.proposal_short_id());
        for desc_id in &descendants {
            // update child score
            self.entries.modify_by_id(desc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_ancestor_weight(parent),
                    EntryOp::Add => e.inner.add_ancestor_weight(parent),
                };
                e.score = e.inner.as_score_key();
            });
        }
    }

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }

    pub(crate) fn remove_by_detached_proposal<'a>(
        &mut self,
        ids: impl Iterator<Item = &'a ProposalShortId>,
    ) {
        for id in ids {
            if let Some(e) = self.pool_map.get_by_id(id) {
                let status = e.status;
                if status == Status::Pending {
                    continue;
                }
                let mut entries = self.pool_map.remove_entry_and_descendants(id);
                entries.sort_unstable_by_key(|entry| entry.ancestors_count);
                for mut entry in entries {
                    let tx_hash = entry.transaction().hash();
                    entry.reset_statistic_state();
                    let ret = self.add_pending(entry);
                    debug!(
                        "remove_by_detached_proposal from {:?} {} add_pending {:?}",
                        status, tx_hash, ret
                    );
                }
            }
        }
    }
