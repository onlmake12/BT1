### Title
Missing Lower-Bound Check on Peer-Declared Cycles Enables Transaction Censorship via Forced Verification Failure — (`sync/src/relayer/transactions_process.rs`)

---

### Summary

When a remote peer relays a transaction, it declares the cycle count alongside the transaction data. The node enforces an **upper** bound (banning peers that declare cycles exceeding `max_block_cycles`), but applies **no lower bound**. A malicious peer can declare `declared_cycles = 0` or any arbitrarily small value for a valid transaction. The node then uses this attacker-controlled value as the `max_cycles` limit for script verification, causing the verifier to fail with `ExceededMaximumCycles` before the `DeclaredWrongCycles` check is ever reached. The peer is not banned, and the transaction is rejected — a direct analog to the external report's "Censorship of Fixed Side Participation" scenario.

---

### Finding Description

**Entry path — `TransactionsProcess::execute()`:**

The relayer receives a `RelayTransactions` message from a peer. It checks only the upper bound:

```rust
// sync/src/relayer/transactions_process.rs:63-74
let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
if txs
    .iter()
    .any(|(_, declared_cycles)| declared_cycles > &max_block_cycles)
{
    self.nc.ban_peer(self.peer, DEFAULT_BAN_TIME, ...);
    return Status::ok();
}
``` [1](#0-0) 

There is **no lower-bound check**. `declared_cycles = 0` passes this gate without triggering a ban.

**Propagation — `_process_tx()`:**

The declared value is used directly as `max_cycles` for the script verifier:

```rust
// tx-pool/src/process.rs:720
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
let verified_ret = verify_rtx(..., max_cycles, ...).await;
``` [2](#0-1) 

If `declared_cycles = 0` (or any value below the transaction's actual cycle cost), `verify_rtx` returns `Err(Reject::Verification(ExceededMaximumCycles))` immediately. The `DeclaredWrongCycles` check that follows is **never reached**:

```rust
// tx-pool/src/process.rs:736-748
if let Some(declared) = declared_cycles
    && declared != verified.cycles
{
    return Some((Err(Reject::DeclaredWrongCycles(declared, verified.cycles)), snapshot));
}
``` [3](#0-2) 

**Consequence in `after_process()`:**

A `Reject::Verification(...)` error does not trigger `ban_malformed`. The peer is not penalized:

```rust
// tx-pool/src/process.rs:502-525
Err(reject) => {
    if is_missing_input(reject) {
        self.add_orphan(tx, peer, declared_cycle).await;
    } else {
        if reject.is_malformed_tx() {
            self.ban_malformed(peer, ...).await;
        }
        if reject.is_allowed_relay() {
            self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash: ... });
        }
        ...
    }
}
``` [4](#0-3) 

`Reject::Verification(ExceededMaximumCycles)` is not `is_malformed_tx`, so the peer is not banned. Whether `is_allowed_relay()` returns true for `Verification` errors determines whether the tx hash is cleared from the known-tx filter; if it is not cleared, the node will not re-request the transaction from other peers until the filter expires.

**Contrast with the existing `DeclaredWrongCycles` protection:**

The existing protection only fires when the verifier **completes successfully** but the declared count mismatches the actual count. It is entirely bypassed when declared cycles are so low that the verifier aborts early. [3](#0-2) 

---

### Impact Explanation

A malicious peer that has been asked by the node to supply a specific transaction (via `GetRelayTransactions`) can respond with `declared_cycles = 0`. The node:

1. Runs the script verifier with `max_cycles = 0`, which immediately fails.
2. Rejects the transaction with `Verification(ExceededMaximumCycles)`.
3. Does **not** ban the peer.
4. May leave the transaction hash in the known-tx filter, preventing re-request from honest peers until the filter expires.

The attacker can repeat this for every request the node makes, effectively censoring a target transaction from the node's tx-pool for the duration of the filter window. This is the CKB analog of the external report's "Censorship of Fixed Side Participation" scenario: an attacker-controlled data source (the peer's declared cycles) causes a legitimate operation (transaction admission) to fail, without the attacker being penalized.

**Impact: Medium** — Temporary per-node tx-pool censorship of targeted transactions. Does not affect consensus or chain validity.

---

### Likelihood Explanation

**Likelihood: Low-Medium.**

- The attacker must be a connected peer that the victim node has specifically requested the transaction from (the `requesting_peer()` filter in `TransactionsProcess`).
- The attack is repeatable without ban risk, so a persistent attacker can sustain censorship across filter expiry windows.
- The attack is cheap: no PoW, no stake, just a malformed relay response. [5](#0-4) 

---

### Recommendation

1. **Add a lower-bound check on `declared_cycles`** in `TransactionsProcess::execute()`. Any declared value of `0` (or below a configurable minimum, e.g., `1`) should be treated as malformed and result in a peer ban, symmetric with the existing upper-bound ban:

   ```rust
   if txs.iter().any(|(_, declared_cycles)| *declared_cycles == 0) {
       self.nc.ban_peer(self.peer, DEFAULT_BAN_TIME,
           String::from("relay declared cycles is zero"));
       return Status::ok();
   }
   ```

2. **Treat `Verification(ExceededMaximumCycles)` from a remote transaction as equivalent to `DeclaredWrongCycles`** in `after_process`, triggering a peer ban and clearing the tx hash from the known-tx filter so honest peers can supply it.

---

### Proof of Concept

1. Connect a malicious peer to a CKB node.
2. Announce a valid transaction hash via `RelayTransactionHashes`.
3. When the node sends `GetRelayTransactions`, respond with the transaction but set `cycles = 0` in the `RelayTransaction` message.
4. Observe: the node rejects the transaction with `ExceededMaximumCycles`; the peer is not banned; the transaction does not enter the tx-pool.
5. Repeat step 3 on each subsequent request to sustain censorship.

The existing test `DeclaredWrongCyclesAndRelayAgain` demonstrates the correct handling for the `DeclaredWrongCycles` case (declared slightly above actual), but does not cover the case where declared cycles are below actual, confirming the gap. [6](#0-5) [7](#0-6) [8](#0-7)
