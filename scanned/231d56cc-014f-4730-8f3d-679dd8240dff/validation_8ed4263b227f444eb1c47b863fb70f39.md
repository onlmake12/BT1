### Title
Snappy Varint Pre-Allocation Amplification in `LengthDelimitedCodecWithCompress::decode` — (`network/src/compress.rs`)

---

### Summary

In `LengthDelimitedCodecWithCompress::decode`, the decoder unconditionally allocates a zeroed buffer sized to the attacker-supplied snappy varint *before* attempting decompression. An unprivileged remote peer can craft a wire frame of ~10 bytes whose snappy varint claims 8,388,607 bytes of decompressed output, triggering an ~8 MB heap allocation per frame. Because the allocation precedes decompression validation, the amplification ratio from wire bytes to heap bytes is up to ~800,000×.

The snappy raw format begins with a LEB128 varint encoding the uncompressed length. `decompress_len` reads only this varint — it does not validate the rest of the stream. An attacker can craft a payload where:

- Bytes 0–3: LEB128 varint = `8,388,607` (`0xFF 0xFF 0xFF 0x03`)
- Bytes 4+: arbitrary garbage (or a minimal valid snappy literal)

`decompress_len` returns `8,388,607`. The guard at line 235 passes (`8,388,607 ≤ 8,388,608`). Line 242 allocates `BytesMut::zeroed(8_388_607)` — an ~8 MB zeroed buffer. The subsequent `decompress` call fails because the payload does not actually decompress to 8 MB, returning `Err(InvalidData)`. The connection is closed and the buffer is freed — but the allocation already occurred.

### Impact Explanation

Each malicious frame causes the connection to be closed (the `Err` propagates up through the tentacle codec layer). The attacker must reconnect for each attack cycle. However:

1. The amplification ratio is extreme (~8,000× for small-frame protocols).
2. With 117 concurrent inbound connections, the attacker can sustain ~936 MB of simultaneous transient heap pressure per attack wave.
3. Rust's allocator (jemalloc by default in CKB) does not immediately return freed memory to the OS; repeated allocation/free cycles cause heap fragmentation and RSS growth disproportionate to live allocations.
4. Under sustained reconnection flooding, the node's RSS grows continuously, eventually triggering OOM on memory-constrained deployments.

**Scoped impact:** node crash / severe memory exhaustion enabling low-cost network congestion. Matches the stated scope.

---

### Likelihood Explanation

- No authentication, PoW, or privileged role required — any peer can connect.
- The attack is locally reproducible: craft a 10-byte snappy payload with varint = `8,388,607`, wrap it in a length-delimited frame with compress flag set, feed it to `decode` in a loop.
- The attacker needs only a single IP (or a small number) to open 117 inbound connections.
- The attack is cheap: ~10 bytes of wire traffic per 8 MB allocation cycle.

---

### Recommendation

Move the allocation to *after* decompression validation, or use a streaming/bounded decompressor that does not require pre-allocating the full declared output size. At minimum, add a secondary guard that rejects frames where the declared decompressed size is disproportionate to the compressed payload size (e.g., reject if `decompressed_bytes_len > compressed_payload_len * MAX_COMPRESSION_RATIO`).

The `Message::decompress` path (line 81) has the same issue (`vec![0; decompressed_bytes_len]`) and should be fixed in parallel. [4](#0-3)
