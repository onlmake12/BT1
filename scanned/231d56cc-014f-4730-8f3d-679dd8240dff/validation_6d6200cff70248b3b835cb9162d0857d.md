### Title
Missing Re-verification of `since` Conditions After Chain Reorganization Allows Invalid Transactions in Block Templates — (`tx-pool/src/process.rs`)

### Summary
After a chain reorganization, the CKB tx-pool updates its snapshot to the new tip but does **not** re-verify `since` time-lock conditions for transactions already resident in the pool. If the reorg produces a new canonical tip with a lower median time (for timestamp-based `since`) or lower block number (for block-number-based `since`), transactions whose `since` conditions were satisfied at the old tip may be invalid at the new tip. These transactions remain in the pool undetected and are packaged into block templates by the block assembler. When the miner submits the resulting block, the block verifier rejects it, causing the miner to lose the block reward — a direct financial loss.

---

### Finding Description

**Vulnerability class**: tx-pool admission vs. on-chain commit inconsistency for time-based `since` locks after chain reorganization.

**Entry path**: Any unprivileged transaction sender can craft a transaction with an absolute-timestamp or absolute-block-number `since` field. The tx is admitted to the pool when the condition is satisfied at the current tip. A subsequent reorg (triggered by a competing miner or natural network partition) can invalidate the condition without the pool noticing.

**Root cause — `_update_tx_pool_for_reorg`** [1](#0-0) 

When a reorg occurs, `_update_tx_pool_for_reorg` is called. It:
1. Updates the pool snapshot to the new tip (`tx_pool.snapshot = Arc::clone(&snapshot)`).
2. Removes committed and detached-proposal transactions.
3. Moves transactions between pending/gap/proposed states.
4. Calls `remove_expired` (wall-clock expiry only).

**It never re-runs `TimeRelativeTransactionVerifier` / `SinceVerifier` for transactions that remain in the pool.** Only transactions that are re-added via `readd_detached_tx` go through re-verification. [2](#0-1) 

**How `since` is checked at admission**

At submission, `TxVerifyPhase::Submitted` is used: [3](#0-2) 

For absolute-timestamp `since`, `SinceVerifier::verify_absolute_lock` computes the median time from the **current tip's hash**: [4](#0-3) 

After a reorg that replaces high-timestamp blocks with lower-timestamp blocks, the new tip's median time can be lower than the old tip's. The admitted transaction's `since` condition is no longer satisfied, but the pool does not know this.

**How `since` is checked at block commit**

`SinceVerifier` is part of `ContextualTransactionVerifier`, which is run by the block verifier using `TxVerifyPhase::Committed` — the **actual** commit block number and its parent's median time: [5](#0-4) 

This check will fail for the stale transaction, causing the block to be rejected.

**Block assembler does not re-verify `since`**

`BlockAssembler::update_full` calls `package_txs` to select proposed transactions, then `calc_dao`. Neither step re-runs `SinceVerifier`: [6](#0-5) 

The `failed_txs` path only handles cell-resolve failures, not `since` violations. The stale transaction is included in `checked_txs` and placed in the block template.

**Re-verification only happens for new submissions, not existing pool entries**

`submit_entry` does re-verify `since` if the snapshot changes between pre-check and submission: [7](#0-6) 

But this guard only protects the submission path. Transactions already resident in the pool after a reorg are never re-checked.

**`time_relative_verify` exists but is not called on reorg** [8](#0-7) 

This function is available and correct, but `_update_tx_pool_for_reorg` never invokes it for existing pool entries.

---

### Impact Explanation

A miner running in mine mode assembles a block template from the proposed pool. If the pool contains a transaction whose absolute-timestamp `since` condition was satisfied at the pre-reorg tip but is no longer satisfied at the post-reorg tip, the block template will include that transaction. The miner expends real PoW work to solve the puzzle, submits the block, and the block verifier rejects it (via `SinceVerifier` inside `ContextualTransactionVerifier`). The miner loses the block reward — a direct, quantifiable financial loss equivalent to the block subsidy plus transaction fees.

**Impact: Medium** — financial loss proportional to the block reward.

---

### Likelihood Explanation

A reorg that reduces the median time requires the new canonical chain to contain blocks with lower timestamps than the replaced blocks. This is possible when:
- A competing miner produces a fork with blocks whose timestamps are close to the minimum allowed (just above the parent median time).
- Network partitions cause two branches to diverge in timestamp progression.

The `since` condition must be an absolute timestamp that was just barely satisfied at the old tip. This is a realistic scenario for time-sensitive contracts (e.g., time-locked cells that unlock at a specific epoch boundary).

**Likelihood: Medium** — reorgs are uncommon on mainnet but occur; the specific timestamp regression condition narrows the window but does not eliminate it.

---

### Recommendation

In `_update_tx_pool_for_reorg`, after updating the snapshot, iterate over all transactions in the proposed and pending pools and re-run `time_relative_verify` against the new tip. Transactions that fail should be evicted and their rejection callbacks invoked. This mirrors the existing guard in `submit_entry` (lines 118–134) but applied pool-wide on reorg.

Alternatively, the block assembler's `calc_dao` / `package_txs` path should run `TimeRelativeTransactionVerifier` for each candidate transaction before including it in the block template, so that any stale `since` condition is caught before PoW work is performed.

---

### Proof of Concept

1. Submit a transaction `T` with `since = absolute_timestamp(S)` where `S` is the current tip's median time (just satisfied).
2. `T` is admitted to the pending pool; `SinceVerifier` passes using `TxVerifyPhase::Submitted` against the current tip.
3. `T` is proposed and moves to the proposed pool.
4. A chain reorganization occurs. The new canonical tip replaces several high-timestamp blocks with lower-timestamp blocks, reducing the median time below `S`.
5. `_update_tx_pool_for_reorg` runs: the snapshot is updated, but `T` remains in the proposed pool without re-verification.
6. The block assembler calls `package_txs` → `calc_dao`; `T` passes the resolve check and is included in `checked_txs`.
7. The miner receives the block template containing `T`, solves the PoW puzzle, and submits the block.
8. The block verifier runs `ContextualTransactionVerifier` → `SinceVerifier` with `TxVerifyPhase::Committed` against the new tip's median time. The check `new_tip_median_time < S` fails → `TransactionError::Immature`.
9. The block is rejected; the miner loses the block reward.

fn _update_tx_pool_for_reorg(
    tx_pool: &mut TxPool,
    attached: &LinkedHashSet<TransactionView>,
    detached_headers: &HashSet<Byte32>,
    detached_proposal_id: HashSet<ProposalShortId>,
    snapshot: Arc<Snapshot>,
    callbacks: &Callbacks,
    mine_mode: bool,
) {
    tx_pool.snapshot = Arc::clone(&snapshot);

    // NOTE: `remove_by_detached_proposal` will try to re-put the given expired/detached proposals into
    // pending-pool if they can be found within txpool. As for a transaction
    // which is both expired and committed at the one time(commit at its end of commit-window),
    // we should treat it as a committed and not re-put into pending-pool. So we should ensure
    // that involves `remove_committed_txs` before `remove_expired`.
    tx_pool.remove_committed_txs(attached.iter(), callbacks, detached_headers);
    tx_pool.remove_by_detached_proposal(detached_proposal_id.iter());

    // mine mode:
    // pending ---> gap ----> proposed
    // try move gap to proposed
    if mine_mode {
        let mut proposals = Vec::new();
        let mut gaps = Vec::new();

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) {
            let short_id = entry.inner.proposal_short_id();
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push((short_id, entry.inner.clone()));
            }
        }

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) {
            let short_id = entry.inner.proposal_short_id();
            let elem = (short_id.clone(), entry.inner.clone());
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push(elem);
            } else if snapshot.proposals().contains_gap(&short_id) {
                gaps.push(elem);
            }
        }

        for (id, entry) in proposals {
            debug!("begin to proposed: {:x}", id);
            if let Err(e) = tx_pool.proposed_rtx(&id) {
                debug!(
                    "Failed to add proposed tx {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e);
            } else {
                callbacks.call_proposed(&entry)
            }
        }

        for (id, entry) in gaps {
            debug!("begin to gap: {:x}", id);
            if let Err(e) = tx_pool.gap_rtx(&id) {
                debug!(
                    "Failed to add tx to gap {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e.clone());
            }
        }
    }

    // Remove expired transaction from pending
    tx_pool.remove_expired(callbacks);

    // Remove transactions from the pool until its size <= size_limit.
    let _ = tx_pool.limit_size(callbacks, None);

    fn verify_absolute_lock(&self, index: usize, since: Since) -> Result<(), Error> {
        if since.is_absolute() {
            match since.extract_metric() {
                Some(SinceMetric::BlockNumber(block_number)) => {
                    let proposal_window = self.consensus.tx_proposal_window();
                    if self.tx_env.block_number(proposal_window) < block_number {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::EpochNumberWithFraction(epoch_number_with_fraction)) => {
                    if !epoch_number_with_fraction.is_well_formed_increment() {
                        return Err((TransactionError::InvalidSince { index }).into());
                    }
                    let a = self.tx_env.epoch().to_rational();
                    let b = epoch_number_with_fraction.normalize().to_rational();
                    if a < b {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::Timestamp(timestamp)) => {
                    let parent_hash = self.tx_env.parent_hash();
                    let tip_timestamp = self.block_median_time(&parent_hash);
                    if tip_timestamp < timestamp {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                None => {
                    return Err((TransactionError::InvalidSince { index }).into());
                }
            }
        }
        Ok(())
    }

pub(crate) fn time_relative_verify(
    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: TxVerifyEnv,
) -> Result<(), Reject> {
    let consensus = snapshot.cloned_consensus();
    TimeRelativeTransactionVerifier::new(
        rtx,
        consensus,
        snapshot.as_data_loader(),
        Arc::new(tx_env),
    )
    .verify()
    .map_err(Reject::Verification)
}
