### Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`)

---

### Summary

`UnknownTxHashPriority::push_peer` appends a `PeerIndex` to an internal `Vec` with no cap and no deduplication. The global overflow guard in `add_ask_for_txs` counts only the number of **distinct tx hash keys** in `unknown_tx_hashes`, not the total number of peer entries across all priorities. A single unprivileged peer can therefore grow the `peers` Vec for any one entry without bound by repeatedly announcing the same tx hash via `RelayTransactionHashes`, bypassing every existing guard.

---

### Finding Description

**`push_peer` — no cap, no dedup** [1](#0-0) 

Every call unconditionally appends. There is no length check and no check for duplicate `PeerIndex` values.

**`add_ask_for_txs` — guard counts keys, not peer entries** [2](#0-1) 

The condition `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50 000) tests the number of **distinct hash keys** in the `KeyedPriorityQueue`, not the cumulative length of all `peers` Vecs. With a single tx hash entry the map length is 1, so this branch is never entered and the per-peer counter check at line 1524 is never reached.

**`TransactionHashesProcess::execute` — no dedup before forwarding** [3](#0-2) 

The only filter applied is `tx_filter.contains`, which only removes hashes already marked as **known**. An unknown hash passes through every time, regardless of how many times the same peer has announced it before.

**`take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` does not protect per-entry Vec length** [4](#0-3) 

`take(32767)` limits the number of hashes processed per call, not the number of times the same hash can be pushed. If the attacker includes hash H 32767 times in one `RelayTransactionHashes` message, `push_peer` is called 32767 times for H in a single invocation, growing that entry's `peers` Vec by 32767 in one round-trip.

**`requesting_peer` filter in `TransactionsProcess::execute` is irrelevant to the growth path** [5](#0-4) 

This filter only governs whether a received transaction body is accepted. It has no effect on `add_ask_for_txs`, which is driven by `RelayTransactionHashes`, not `RelayTransactions`.

---

### Impact Explanation

`unknown_tx_hashes` is a process-wide `Mutex<KeyedPriorityQueue<Byte32, UnknownTxHashPriority>>`. [6](#0-5) 

Each `UnknownTxHashPriority` holds a `Vec<PeerIndex>`. [7](#0-6) 

A single attacker peer sending repeated `RelayTransactionHashes` messages each containing 32767 copies of one unknown hash H causes the `peers` Vec for H to grow by 32767 per message. At 8 bytes per `PeerIndex` (usize on 64-bit), 1 000 messages → ~262 MB for a single entry. The mutex is held during the entire `add_ask_for_txs` loop, so growth also causes lock contention on every other operation that touches `unknown_tx_hashes` (including `pop_ask_for_txs` and `mark_as_known_txs`).

---

### Likelihood Explanation

The attack requires only a standard P2P connection. `RelayTransactionHashes` is an ordinary relay message accepted from any connected peer. No PoW, no key material, no privileged role is needed. The attacker needs only to know (or guess) one tx hash that is currently in `unknown_tx_hashes` and not yet in `tx_filter` — or simply announce a fresh fabricated hash that will never be resolved, keeping the entry alive indefinitely.

---

### Recommendation

1. **Cap `peers` Vec length in `push_peer`**: reject the push if `self.peers.len()` already exceeds a small constant (e.g., the number of connected peers or a fixed limit like 256).
2. **Deduplicate in `push_peer`**: skip the push if `peer_index` is already present.
3. **Move the per-peer counter check before insertion**, not after, and make it count peer occurrences across all entries before allowing a new push.
4. **Deduplicate hashes within a single `RelayTransactionHashes` message** before passing them to `add_ask_for_txs`.

---

### Proof of Concept

```
1. Attacker peer A connects to victim node.
2. A sends RelayTransactionHashes{ tx_hashes: [H; 32767] }
   → add_ask_for_txs called; H is unknown, so Occupied branch fires 32767 times,
     pushing A into peers Vec 32767 times.
   → unknown_tx_hashes.len() == 1 → global guard never triggers.
3. Repeat step 2 indefinitely.
4. After K messages: peers.len() for H == 32767 * K, consuming ~262 MB per 1000 messages.
5. Every pop_ask_for_txs / mark_as_known_txs call must acquire the same mutex,
   causing increasing lock contention as the Vec grows.
```

Assertion: after calling `add_ask_for_txs(peer_A, vec![H; 32767])` twice with the same H already present as an Occupied entry, `unknown_tx_hashes.get_priority(&H).unwrap().peers.len() == 32767 * 2 + 1` (initial insert + two batches), confirming no cap is enforced.
