### Title
CompactBlock Relay Rate Limiter Bypass via Trivial PoW — Missing `compact_target` Epoch Validation in `HeaderVerifier` — (`sync/src/relayer/mod.rs`, `verification/src/header_verifier.rs`)

---

### Summary

The `Relayer::try_process` function explicitly exempts `CompactBlock` messages from the 30-req/sec rate limiter, relying on PoW as the cost barrier. However, `HeaderVerifier::verify` — the verifier used in the relay path — only checks that the nonce satisfies the `compact_target` **embedded in the header itself**. It never validates that the embedded `compact_target` matches the expected epoch target. An attacker can therefore craft CompactBlocks with the genesis `compact_target = 0x1e015555` (trivially cheap PoW), mine valid nonces at negligible cost, and flood any testnet node with an unbounded stream of messages that each pass all relay-path checks and trigger full compact-block reconstruction logic.

---

### Finding Description

**Claim 1 — Rate limiter is explicitly bypassed for CompactBlock:**

In `Relayer::try_process`, the rate limiter keyed by `(PeerIndex, item_id)` is skipped for every `CompactBlock` message:

```rust
// CompactBlock will be verified by POW, it's OK to skip rate limit checking.
let should_check_rate =
    !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));
``` [1](#0-0) 

The developer comment makes the design intent explicit: PoW is the intended rate-control mechanism.

**Claim 2 — `HeaderVerifier` does NOT check that `compact_target` matches the expected epoch target:**

`HeaderVerifier::verify` in the relay path calls `PowVerifier`, `NumberVerifier`, `EpochVerifier`, and `TimestampVerifier`: [2](#0-1) 

The `EpochVerifier` used here takes only `parent: EpochNumberWithFraction` — the epoch number/index/length — and checks only that the epoch is well-formed and is a valid successor of the parent's epoch. It has no access to the expected `compact_target` and performs no such check: [3](#0-2) 

The `PowVerifier` only checks that the nonce satisfies the `compact_target` **stored in the header itself**: [4](#0-3) [5](#0-4) 

**Claim 3 — The compact_target epoch check exists only in the contextual block verifier, not the relay path:**

The check that `header.compact_target()` must equal `epoch_ext.compact_target()` lives in `contextual_block_verifier.rs`: [6](#0-5) 

This verifier is only invoked during full block acceptance (after reconstruction), not during the relay-path `contextual_check`.

**Claim 4 — Testnet genesis compact_target is trivially easy:**

`resource/specs/testnet.toml` sets `compact_target = 0x1e015555`: [7](#0-6) 

`0x1e015555` decodes to target = `0x015555 × 2^(8×27)` — an astronomically large target. A single CPU core can mine valid nonces at thousands per second.

**Claim 5 — The full relay processing path is reached for each crafted block:**

`CompactBlockProcess::execute` runs `non_contextual_check` (uncles/proposals count, stale height), then `contextual_check` (block status, parent existence, pending dedup, `HeaderVerifier`), then `CompactBlockVerifier::verify`, then `reconstruct_block` (tx-pool lookups): [8](#0-7) 

Each crafted block has a unique hash (different nonce), so the block-status and pending-dedup guards do not fire. The parent is set to the current tip (known to the attacker), so the parent check passes. The timestamp is set to `now`, so the timestamp check passes. The epoch field is set to a valid successor of the parent's epoch, so the epoch-continuity check passes. The PoW check passes because the nonce satisfies the trivial target embedded in the header.

---

### Impact Explanation

Every receiving testnet node executes two sequential hash computations (eaglesong → blake2b_256) per message plus tx-pool lookups and compact-block reconstruction logic, with no per-peer message cap. A single attacker connection can saturate the relay processing thread, delaying or blocking propagation of legitimate blocks and causing effective network congestion on testnet.

---

### Likelihood Explanation

The attack requires only a P2P connection to a testnet node, knowledge of the current tip hash (publicly available), and a CPU loop mining against `compact_target = 0x1e015555`. No privileged access, no hashpower majority, no key material. The rate-limiter bypass is intentional and documented in source. The missing compact_target check in `HeaderVerifier` is a concrete code gap, not a theoretical one.

---

### Recommendation

1. **Add compact_target validation to the relay-path `HeaderVerifier`**: The `EpochVerifier` in `header_verifier.rs` should receive the expected `compact_target` (derivable from the parent's epoch extension) and reject headers whose embedded `compact_target` does not match.
2. **Apply a per-peer rate limit to CompactBlock messages**: The assumption that PoW cost is sufficient is only valid when the compact_target is at real network difficulty. A separate, lower rate cap (e.g., 2–5/sec) for CompactBlock per peer would bound damage even if PoW is cheap.
3. **Alternatively, verify compact_target before PoW**: Reject the message immediately if the header's `compact_target` does not match the expected epoch target, before running the hash computation.
