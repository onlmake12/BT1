Audit Report

## Title
`check_purge` `take(len/2)` Integer Truncation Causes O(N) Amortized `add_addr` Cost Under 2-Group Adversarial Fill — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

When the peer store is at capacity (`ADDR_COUNT_LIMIT` = 16384) and all stored addresses belong to exactly 2 network groups and are connectable, `check_purge`'s second-pass `take(len / 2)` evaluates to `take(1)`, evicting only 2 peers per invocation. Because each invocation performs two full O(N) scans over all 16384 entries and only reduces the store by 2, the purge re-triggers every 2 insertions, degrading the amortized cost of `add_addr` from O(1) to O(N).

## Finding Description

`add_addr` unconditionally calls `check_purge` before inserting:

```rust
// peer_store_impl.rs L71-79
pub fn add_addr(&mut self, addr: Multiaddr, flags: Flags) -> Result<()> {
    if self.ban_list.is_addr_banned(&addr) { return Ok(()); }
    self.check_purge()?;
    ...
}
```

`check_purge` (L327) returns early if `count() < ADDR_COUNT_LIMIT`. Otherwise it runs two passes:

**Pass 1 (L341–355):** Iterates all entries via `addrs_iter()` collecting non-connectable peers. Addresses added via `add_addr` have `last_connected_at_ms=0` and `attempts_count=0`; `is_connectable` returns `true` for these because the non-connectable conditions require `attempts_count >= ADDR_MAX_RETRIES (3)` or `>= ADDR_MAX_FAILURES (10)`. So pass 1 collects nothing and the `if candidate_peers.is_empty()` branch (L357) is entered.

**Pass 2 (L358–393):** Iterates all entries again, groups by network segment, then:

```rust
let len = peers_by_network_group.len();   // = 2 with adversarial fill
peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));
peers.into_iter().take(len / 2)           // take(1) — integer truncation
    .flat_map(|addrs| {
        if addrs.len() > 4 { Some(choose_multiple(..., 2)) } else { None }
    })
```

With 2 groups of 8192 peers each: `take(1)` visits only the largest group; since 8192 > 4, exactly 2 peers are evicted. The store drops from 16384 → 16382. The new address is then inserted → 16383. The next `add_addr` call finds 16383 < 16384 and skips purge; the address is inserted → 16384. The call after that triggers purge again. The cycle repeats every 2 insertions.

Each purge performs two full O(16384) iterations (pass 1 + pass 2 grouping), making the amortized cost per `add_addr` O(N) rather than O(1).

## Impact Explanation

This is a sustained CPU overhead attack on a single CKB node. An attacker keeping the peer store in the 2-group steady state forces ~16384 × 2 hash-map iterations per 2 peer advertisements processed. While this does not crash the node or cause consensus deviation, it constitutes a meaningful and repeatable performance degradation exploitable at negligible cost, fitting **Low (501–2000 points): Any other important performance improvements for CKB**.

## Likelihood Explanation

The P2P discovery protocol calls `add_addr` for each advertised peer address with no authentication or PoW. A single attacker-controlled peer can advertise arbitrary addresses. Two /16 IPv4 subnets (e.g., `10.0.0.0/16` and `172.16.0.0/16`) provide 65536 × 2 = 131072 unique addresses — enough to sustain the attack indefinitely. No privilege escalation or victim mistake is required; the attacker only needs one inbound or outbound connection to the victim node.

## Recommendation

Replace `take(len / 2)` with logic that computes the minimum number of evictions needed to bring the store below `ADDR_COUNT_LIMIT` and iterates over enough groups to guarantee that. A minimal fix:

```rust
let needed = self.addr_manager.count() - ADDR_COUNT_LIMIT + 1;
let mut collected = 0;
let candidate_peers: Vec<_> = peers
    .into_iter()
    .take_while(|_| collected < needed)
    .flat_map(|addrs| {
        if addrs.len() > 4 {
            let chosen = addrs.iter().choose_multiple(&mut rand::thread_rng(), 2)
                .into_iter().map(|a| a.addr.clone()).collect::<Vec<_>>();
            collected += chosen.len();
            Some(chosen)
        } else { None }
    })
    .flatten()
    .collect();
```

Alternatively, remove the `take` entirely and evict from all groups until the store is below the limit.

## Proof of Concept

```rust
#[test]
fn test_check_purge_two_group_degraded_cost() {
    let mut peer_store = PeerStore::default();
    // Fill with 8192 addrs in group 10.0.x.x and 8192 in 172.16.x.x
    for i in 0u32..8192 {
        let addr: Multiaddr = format!("/ip4/10.0.{}/{}/tcp/1234",
            i / 256, i % 256).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
    for i in 0u32..8192 {
        let addr: Multiaddr = format!("/ip4/172.16.{}/{}/tcp/1234",
            i / 256, i % 256).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
    assert_eq!(peer_store.addr_manager().count(), 16384);

    // Each pair of subsequent add_addr calls triggers one O(N) check_purge.
    // Instrument or count via a wrapper; assert purge fires ~K/2 times for K calls.
    for i in 0u32..100 {
        let addr: Multiaddr = format!("/ip4/10.1.{}/{}/tcp/9999",
            i / 256, i % 256).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
    // Store oscillates between 16382 and 16384, confirming only 2 evictions per purge.
    let count = peer_store.addr_manager().count();
    assert!(count == 16382 || count == 16383 || count == 16384);
}
```
