Audit Report

## Title
Missing Per-Peer Rate Limiter in `LightClientProtocol::try_process` Enables Unbounded DB-Read and MMR-Computation Flood — (`util/light-client-protocol-server/src/lib.rs`)

## Summary
`LightClientProtocol` holds only `shared: Shared` with no rate-limiter field, and its `try_process` dispatcher performs no per-peer frequency check before executing expensive handlers. Each of `GetBlocksProof`, `GetTransactionsProof`, and `GetLastStateProof` accepts up to 1000 items per message and triggers proportional DB reads plus MMR proof generation. An unprivileged remote peer can flood the node with back-to-back max-size requests, saturating the async executor and degrading full-node performance for all other peers.

## Finding Description
`LightClientProtocol` is defined with only a `shared: Shared` field — no `rate_limiter` exists. [1](#0-0) 

The `received` handler calls `try_process` immediately after deserialization with no throttle check: [2](#0-1) 

`try_process` dispatches all four message types with no rate-limit gate: [3](#0-2) 

By contrast, `Relayer` holds a `governor::RateLimiter<(PeerIndex, u32)>` and rejects excess messages before any work: [4](#0-3) [5](#0-4) 

`HolePunching` applies the same pattern: [6](#0-5) [7](#0-6) 

The per-message item ceilings are all 1000: [8](#0-7) 

`GetBlocksProof` performs up to 4 DB reads per item (`is_main_chain`, `get_block_header`, `get_block_uncles`, `get_block_extension`) followed by `mmr.gen_proof`: [9](#0-8) 

`GetLastStateProof`'s `complete_headers` loop calls `mmr.get_root()` for every sampled block number, up to the 1000-item limit: [10](#0-9) [11](#0-10) 

The item-count ceiling blocks oversized single messages but imposes no frequency ceiling per peer, leaving the server open to rapid successive max-size requests.

## Impact Explanation
A single peer issuing back-to-back `GetBlocksProof` messages each carrying 1000 hashes triggers ~4000 synchronous DB reads plus one MMR proof generation per message with no mechanism to shed load. Sustained at line rate this saturates the shared snapshot read path and the async executor, increasing block-processing and sync latency for all other peers and potentially rendering the node unresponsive. This maps to **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points), as executor saturation can cause the node to stop processing blocks and peer messages entirely.

## Likelihood Explanation
Any peer that negotiates the light-client sub-stream can exploit this without any privilege. The attack requires only a TCP connection and knowledge of the publicly documented protocol message format. It is repeatable and requires no victim interaction or special network position.

## Recommendation
Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, mirroring the pattern in `Relayer::new` (30 req/s quota, keyed by `(PeerIndex, message_item_id)`). Check it at the top of `try_process` and return `StatusCode::TooManyRequests` on excess, before any snapshot or MMR work is performed. A quota of 10–30 requests per second per peer per message type is consistent with existing limits. [12](#0-11) 

## Proof of Concept
1. Connect a peer to a CKB full node with the light-client protocol enabled.
2. In a tight loop, send `GetBlocksProof` messages each containing 1000 distinct valid block hashes and a valid `last_hash`.
3. Observe that every message is processed without rejection, each triggering ~4000 DB reads and one MMR proof generation.
4. Measure full-node block-processing latency and peer-message throughput under this load versus baseline; expect measurable degradation progressing toward node unresponsiveness as request rate increases.
5. Confirm no `TooManyRequests` status is ever returned by instrumenting `try_process`.

    async fn try_process(
        &mut self,
        nc: &Arc<dyn CKBProtocolContext + Sync>,
        peer_index: PeerIndex,
        message: packed::LightClientMessageUnionReader<'_>,
    ) -> Status {
        match message {
            packed::LightClientMessageUnionReader::GetLastState(reader) => {

                    .await
