Audit Report

## Title
`RecentReject::put` Never Persists Incremented `total_keys_num`, Disabling `shrink()` and Allowing Unbounded Disk Growth — (File: `tx-pool/src/component/recent_reject.rs`)

## Summary

In `RecentReject::put`, the result of `self.total_keys_num.checked_add(1)` is bound to a local variable `total_keys_num` but is never written back to `self.total_keys_num`. The field remains frozen at its initial RocksDB estimate (0 on a fresh DB), so the `count_limit` guard is never satisfied and `shrink()` is never called. An unprivileged attacker who continuously submits rejected transactions causes the `DBWithTTL` reject store to grow without bound, exhausting disk space and crashing the node.

## Finding Description

In `RecentReject::put` ( [1](#0-0) ):

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

`total_keys_num` here is a **local binding** — the result of `checked_add(1)` on `self.total_keys_num`. `self.total_keys_num` is never assigned the incremented value. On a fresh DB, `self.total_keys_num` is initialized to 0 via `estimate_num_keys_cf` at construction time. [2](#0-1) 

Every subsequent call to `put()` computes `0 + 1 = 1`, checks `1 > count_limit` (default 10,000,000), which is always `false`, and returns without calling `shrink()` or updating `self.total_keys_num`. The counter stays frozen at 0 indefinitely.

`shrink()` does correctly update `self.total_keys_num` when called, but it is never reached: [3](#0-2) 

The existing unit test does not catch this bug — it asserts `recent_reject.total_keys_num < 100` after 160 puts with `count_limit = 100`, which passes trivially because `total_keys_num` stays at 0, not because `shrink()` correctly enforced the limit. [4](#0-3) 

## Impact Explanation

The `DBWithTTL` reject store grows without bound. Each stored entry is a JSON-serialized `PoolTransactionReject` keyed by a 32-byte tx hash. Although `DBWithTTL` applies a 7-day TTL, RocksDB TTL expiration only occurs during compaction — it is not immediate. An attacker submitting unique rejected transactions faster than compaction expires old entries will exhaust disk space, crashing the node.

This matches the **High** impact category: *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation

The attack path is fully unprivileged:

1. `Reject::should_recorded()` returns `true` for every reject variant except `Duplicated`. [5](#0-4) 
2. `Reject::LowFeeRate` satisfies `should_recorded()`. An attacker submits zero-fee transactions via P2P relay or the `send_transaction` RPC, which are rejected with `LowFeeRate`.
3. Each rejection flows through `after_process` → `put_recent_reject` → `RecentReject::put`. [6](#0-5) 
4. Each unique transaction hash produces a new DB entry. No rate limiting on rejected transactions is present in the code path.

The attack is repeatable, requires no special privileges, and scales with attacker bandwidth.

## Recommendation

In `RecentReject::put`, assign the incremented value back to `self.total_keys_num` before the limit check:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;   // ← missing assignment
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

Also update the unit test to assert that `total_keys_num` actually tracks the number of inserted keys and that `shrink()` is triggered when `count_limit` is exceeded (i.e., assert `total_keys_num` is non-zero and approximately equal to the number of puts, not just `< 100`).

## Proof of Concept

```rust
#[test]
fn test_counter_frozen_bug() {
    let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
    // count_limit = 10, so shrink() should trigger after 10 puts
    let mut rr = RecentReject::build(tmp_dir.path(), 2, 10, -1).unwrap();
    assert_eq!(rr.total_keys_num, 0); // fresh DB

    for i in 0u64..50 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        rr.put(&key, Reject::Malformed(i.to_string(), Default::default())).unwrap();
    }

    // Bug: total_keys_num is still 0, not ~50
    // shrink() was never called despite count_limit=10 being exceeded
    assert_eq!(rr.total_keys_num, 0); // demonstrates the frozen counter
}
```

Running this test against the current code will pass (demonstrating the bug). After applying the fix (`self.total_keys_num = total_keys_num;`), `total_keys_num` will be non-zero and `shrink()` will have been called, causing the test to fail at the final assertion — confirming the fix works.
