### Title
Peer Store Permanently Blocked by Crafted Discovery Flood: `check_purge` Returns `EvictionFailed` When All Groups Have Exactly 4 Peers — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

`check_purge` contains two compounding logic flaws: a strict `> 4` guard (instead of `>= 4`) and a `take(len/2)` that only examines half the network groups. An attacker who fills the peer store with exactly 16,384 addresses distributed across 4,096 distinct /16 subnets (4 per subnet), all connectable, causes `check_purge` to permanently return `PeerStoreError::EvictionFailed` on every subsequent `add_addr` call, permanently blocking peer discovery.

---

### Finding Description

**Constants:**
- `ADDR_COUNT_LIMIT = 16384` [1](#0-0) 
- `ADDR_MAX_RETRIES = 3` [2](#0-1) 

**`add_addr` always inserts with `last_connected_at_ms=0` and `attempts_count=0`:** [3](#0-2) 

**`is_connectable` returns `true` for any peer with `last_connected_at_ms=0` and `attempts_count < 3`:** [4](#0-3) 

Because `add_addr` always creates entries with `attempts_count=0`, every adversarially-inserted address is connectable.

**`check_purge` Step 1** — filters non-connectable peers. With all entries connectable, `candidate_peers` is empty and no removals occur. [5](#0-4) 

**`check_purge` Step 2** — groups by network segment, sorts by group size descending, takes `len/2` groups, then only evicts from groups with `addrs.len() > 4`: [6](#0-5) 

With 4096 groups of exactly 4 peers each:
- `len = 4096`, `take(len/2) = take(2048)` — only 2048 groups examined
- For every group: `addrs.len() > 4` → `4 > 4` → **false** → `None` returned
- `candidate_peers` remains empty
- Line 399–400: `return Err(PeerStoreError::EvictionFailed.into())` fires unconditionally

Even if `take(len/2)` were removed, the `> 4` guard alone would still block eviction for all groups of size ≤ 4.

---

### Impact Explanation

Once the peer store is saturated in this configuration, every call to `add_addr` (the entry point for all P2P-discovered peer addresses) returns `Err(EvictionFailed)`. The node can no longer learn new peer addresses from the network. Existing connections are unaffected, but the node's ability to discover new peers is permanently disabled until restart (and the persisted `addr_manager.db` would reload the same poisoned state). This causes peer discovery starvation: honest nodes cannot learn new peers, degrading connectivity and potentially isolating nodes from the honest chain tip.

---

### Likelihood Explanation

The attack is reachable via the standard CKB discovery protocol. A single adversarial peer (or a small set of colluding peers) can advertise up to 16,384 addresses across 4,096 /16 subnets. No special privilege, PoW, or key material is required. The attacker only needs to be accepted as a P2P peer and send `GetNodes`/`Nodes` discovery messages. The fill condition (4096 × 4 = 16384) is exact and deterministic. The state persists across restarts via `addr_manager.db`.

---

### Recommendation

1. **Fix the off-by-one**: Change `addrs.len() > 4` to `addrs.len() >= 4` (or `> 3`) so groups of exactly 4 are eligible for eviction.
2. **Remove or reconsider `take(len/2)`**: There is no documented reason to only examine half the groups during eviction pressure. Consider examining all groups.
3. **Add a fallback**: If `candidate_peers` is still empty after both steps, fall back to evicting the oldest/least-recently-seen entries regardless of group size, to guarantee the invariant that the store can always accept new addresses.

---

### Proof of Concept

```rust
#[test]
fn test_eviction_failed_with_4_per_group() {
    use crate::peer_store::{PeerStore, ADDR_COUNT_LIMIT};
    use crate::{Flags, PeerId};
    use p2p::multiaddr::Multiaddr;

    let mut peer_store = PeerStore::default();
    // Fill with 4096 groups * 4 peers each = 16384 = ADDR_COUNT_LIMIT
    // Each group is a distinct /16 subnet: 1.0.x.x, 1.1.x.x, ..., 16.0.x.x, etc.
    let mut count = 0;
    'outer: for a in 0u8..=255 {
        for b in 0u8..=255 {
            for port in 1u16..=4 {
                if count >= ADDR_COUNT_LIMIT { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/{}.{}.0.1/tcp/{}/p2p/{}",
                    a, b, port, PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(peer_store.addr_manager().count(), ADDR_COUNT_LIMIT);

    // Now any new add_addr must fail with EvictionFailed
    let new_addr: Multiaddr = format!(
        "/ip4/200.200.200.1/tcp/9999/p2p/{}",
        PeerId::random().to_base58()
    ).parse().unwrap();
    let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
    assert!(result.is_err(), "Expected EvictionFailed, got Ok");
}
```

The test fills the store with 4096 groups × 4 connectable peers each (all `last_connected_at_ms=0`, `attempts_count=0`). The final `add_addr` call triggers `check_purge`, which finds no non-connectable peers (step 1), then finds no groups with `len > 4` (step 2), and returns `Err(EvictionFailed)`.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;

        // consider addr is not connectable if failed too many times

        true
