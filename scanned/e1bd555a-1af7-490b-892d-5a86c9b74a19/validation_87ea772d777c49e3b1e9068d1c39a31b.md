### Title
Division Before Multiplication Causes Precision Loss in Epoch Hash Rate Estimation - (File: spec/src/consensus.rs)

### Summary
In `spec/src/consensus.rs`, the `next_epoch_ext` function converts `epoch_duration_in_milliseconds` to seconds via integer division **before** using the result as a denominator in the hash rate calculation. This is a classic division-before-multiplication ordering error that silently truncates up to 999 milliseconds of epoch duration, producing a systematically biased `last_epoch_hash_rate` and, consequently, a biased difficulty target for every subsequent epoch.

### Finding Description
Inside `next_epoch_ext`, the adjusted hash rate is computed as follows:

```rust
let last_epoch_duration = U256::from(cmp::max(
    epoch_duration_in_milliseconds / MILLISECONDS_IN_A_SECOND,  // ← division first
    1,
));

let last_epoch_hash_rate = last_difficulty
    * (epoch.length() + epoch_uncles_count)
    / &last_epoch_duration;
``` [1](#0-0) 

The integer division `epoch_duration_in_milliseconds / MILLISECONDS_IN_A_SECOND` discards the sub-second remainder (0–999 ms) **before** it is used as the denominator. The effective computation is:

```
hash_rate = difficulty × (length + uncles) / floor(duration_ms / 1000)
```

whereas the mathematically correct, precision-preserving form is:

```
hash_rate = difficulty × (length + uncles) × 1000 / duration_ms
```

Because `floor(duration_ms / 1000)` is always ≤ `duration_ms / 1000`, the denominator is artificially small, making `last_epoch_hash_rate` **overestimated** by up to `999 / (duration_ms − 999)` relative error. This biased value is then fed into `bounding_hash_rate` and used to derive the next epoch's compact difficulty target. [2](#0-1) [3](#0-2) 

### Impact Explanation
Every epoch boundary recalculates the proof-of-work difficulty target using the biased hash rate. An overestimated hash rate produces a higher-than-correct difficulty, causing blocks to arrive slightly slower than the 4-hour target. The error is bounded (≤ 999 ms per epoch, ~0.007 % relative for a normal 14 400 s epoch) but is **systematic and cumulative**: it never cancels out because the truncation always rounds down. Over many epochs the chain's actual block interval drifts above the protocol target, reducing throughput and miner revenue in a way that cannot be corrected without a code fix.

### Likelihood Explanation
The code path executes unconditionally at every epoch tail block — approximately once every 4 hours on mainnet. No special conditions or attacker action are required; the precision loss is inherent to the current integer arithmetic. Any block relayer, sync peer, or miner submitting a valid tail block triggers the biased calculation.

### Recommendation
Defer the millisecond-to-second conversion until after all multiplications are complete:

```rust
// Before (lossy):
let last_epoch_duration = U256::from(cmp::max(
    epoch_duration_in_milliseconds / MILLISECONDS_IN_A_SECOND,
    1,
));
let last_epoch_hash_rate = last_difficulty
    * (epoch.length() + epoch_uncles_count)
    / &last_epoch_duration;

// After (precision-preserving):
let epoch_duration_ms = U256::from(cmp::max(epoch_duration_in_milliseconds, 1));
let last_epoch_hash_rate = last_difficulty
    * (epoch.length() + epoch_uncles_count)
    * MILLISECONDS_IN_A_SECOND
    / &epoch_duration_ms;
```

This keeps all multiplications before the single final division, eliminating the truncation error entirely.

### Proof of Concept
Suppose an epoch lasts exactly `14_400_999` ms (4 hours + 999 ms):

**Current code:**
```
last_epoch_duration = 14_400_999 / 1_000 = 14_400   (999 ms discarded)
hash_rate = D × N / 14_400
```

**Correct computation:**
```
hash_rate = D × N × 1_000 / 14_400_999
          ≈ D × N / 14_400.999
```

The current code uses `14_400` instead of `14_400.999` as the effective divisor, overestimating the hash rate by a factor of `14_400_999 / 14_400_000 ≈ 1.0000694`. This inflated hash rate propagates directly into the difficulty formula at lines 903–934, producing a difficulty target that is ~0.007 % too high every epoch — a small but permanent, systematic upward bias on block time.
