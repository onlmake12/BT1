### Title
`consecutive_failed` Never Resets and Lacks `failed_txs` Guard for Pool-Iterator Path, Enabling Early Loop Termination — (`tx-pool/src/component/tx_selector.rs`)

### Summary

In `txs_to_commit`, the `consecutive_failed` counter is incremented whenever a proposed transaction has a non-proposed ancestor, but it is **never reset** when a transaction is successfully committed, and transactions that fail via the pool iterator path (`using_modified = false`) are **not added to `failed_txs`**. An unprivileged submitter can craft ≥ 4001 proposed transactions each with a Pending/Gap ancestor and higher fee rates than legitimate fully-proposed transactions, driving `consecutive_failed` past `MAX_CONSECUTIVE_FAILURES` and causing the block template loop to terminate before the valid transactions are ever evaluated.

---

### Finding Description

**The counter never resets on success.**

`consecutive_failed` is initialized once at line 104 and incremented at lines 153 and 184, but there is no `consecutive_failed = 0` anywhere in the success path: [1](#0-0) [2](#0-1) 

**The pool-iterator path does not record failures in `failed_txs`.**

When `using_modified = false` (transaction came from `sorted_proposed_iter`), the failure branch at lines 180–183 is skipped, so the transaction is never added to `failed_txs`: [3](#0-2) 

`skip_proposed_entry` only skips entries already in `fetched_txs`, `modified_entries`, or `failed_txs`: [4](#0-3) 

*(Note: the question's "re-evaluation" claim is mechanically wrong — `iter.next()` at lines 131/136 does advance the iterator past each entry, so the same entry is not re-evaluated. The actual problem is that 4001 **distinct** bad transactions each increment the counter once, which is sufficient.)*

**The sorted iterator orders by fee rate.**

`sorted_proposed_iter` returns entries in descending `AncestorsScoreSortKey` order: [5](#0-4) [6](#0-5) 

An attacker who sets high fees on their bad transactions ensures those 4001 entries appear before any legitimate low-fee fully-proposed transactions in the iterator.

**`MAX_CONSECUTIVE_FAILURES` is 4000.** [7](#0-6) 

After 4001 increments the guard at line 185 fires and the loop breaks, discarding all remaining iterator entries.

---

### Impact Explanation

A miner's `get_block_template` call invokes `txs_to_commit`. If the attacker's 4001 bad transactions sort above the legitimate ones, the returned `(entries, size, cycles)` tuple contains zero (or fewer) valid committed transactions even though fully-proposed, fee-paying transactions exist in the pool. The miner produces an empty (or impoverished) block, losing all fee revenue for that block. Repeated across multiple block intervals this constitutes a sustained fee-denial attack against miners and a liveness degradation for users whose transactions are indefinitely excluded from templates.

---

### Likelihood Explanation

The attacker needs to:
1. Submit 4001 parent–child transaction pairs where the parent stays Pending/Gap and the child reaches Proposed status.
2. Set fees on the children high enough to sort above the victim transactions.

Step 1 is normal protocol behavior: any submitted transaction can be proposed by any miner within the proposal window. The attacker does not need privileged access, hashpower, or miner cooperation beyond the ordinary mempool relay path (`_submit_entry` → `add_proposed`): [8](#0-7) 

The cost is 4001 on-chain transaction fees, which is bounded and predictable.

---

### Recommendation

1. **Reset `consecutive_failed` to 0 on every successful commit**, mirroring Bitcoin Core's original intent for the heuristic.
2. **Add the failing short-id to `failed_txs` regardless of `using_modified`** so that `skip_proposed_entry` prevents the same logical failure from being re-counted if the entry somehow re-enters consideration (e.g., via `modified_entries`).
3. Consider replacing the absolute counter with a ratio-based or sliding-window heuristic that is harder to saturate with crafted inputs.

---

### Proof of Concept

```
1. Build a PoolMap.
2. Insert 4001 entries with Status::Proposed, each linked (via TxLinksMap) to a
   parent with Status::Gap. Set fee = HIGH on all 4001.
3. Insert 100 entries with Status::Proposed, no ancestors. Set fee = LOW.
4. Call TxSelector::new(&pool).txs_to_commit(usize::MAX, Cycle::MAX).
5. Assert result.0.len() == 0  // the 100 valid txs were never reached.
```

The loop processes the 4001 high-fee bad entries first; each hits the `!has_proposed(ancestor)` branch at line 178, increments `consecutive_failed`, and `continue`s. After the 4001st entry `consecutive_failed` (4001) exceeds `MAX_CONSECUTIVE_FAILURES` (4000) and the loop breaks before the 100 valid entries are ever dequeued from the iterator. [9](#0-8)

fn _submit_entry(
    tx_pool: &mut TxPool,
    status: TxStatus,
    entry: TxEntry,
    callbacks: &Callbacks,
) -> Result<HashSet<TxEntry>, Reject> {
    let tx_hash = entry.transaction().hash();
    debug!("submit_entry {:?} {}", status, tx_hash);
    let (succ, evicts) = match status {
        TxStatus::Fresh => tx_pool.add_pending(entry.clone())?,
        TxStatus::Gap => tx_pool.add_gap(entry.clone())?,
        TxStatus::Proposed => tx_pool.add_proposed(entry.clone())?,
    };
