### Title
Peer Store Permanently Blocked by Single-/16-Group Flood via Discovery Nodes — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
`check_purge`'s second-pass eviction uses `len / 2` (integer division) to determine how many network groups to process. When all 16384 entries belong to the same /16 group (`len = 1`), `1 / 2 = 0` groups are taken, no eviction occurs, and `EvictionFailed` is returned. Since `add_addr` is called unconditionally for every address received via the Discovery protocol, an attacker who floods the store with 16384 addresses from the same /16 permanently blocks all future `add_addr` calls.

### Finding Description

`check_purge` has two eviction passes:

**Pass 1** — evict non-connectable entries. A freshly-added `AddrInfo` (via `add_addr`) always has `last_connected_at_ms=0`, `attempts_count=0`, `last_tried_at_ms=0`. `is_connectable` returns `true` for all such entries, so pass 1 finds zero candidates. [6](#0-5) 

**Pass 2** — group by network segment, take the top `len/2` groups, evict 2 from each group with `> 4` entries. The network group for IPv4 is `Group::IP4([octet0, octet1])` — a /16, not a /24. [1](#0-0) 

When all entries share the same /16, `peers_by_network_group.len() == 1`, so `len / 2 == 0`, `.take(0)` produces an empty iterator, `candidate_peers` is empty, and the function returns `Err(PeerStoreError::EvictionFailed)`. [7](#0-6) 

The error propagates to `add_addr` and is silently swallowed by the Discovery handler: [8](#0-7) 

### Impact Explanation

Once the store is saturated with single-/16 entries, every subsequent `add_addr` call (from both Discovery and Identify protocols) returns `EvictionFailed`. The node can no longer learn about new peers. Existing peers in the store are unaffected, but the node's ability to recover connectivity after peer churn is permanently impaired until the node is restarted and the persisted peer store is cleared.

### Likelihood Explanation

A single malicious peer connected via P2P can send repeated Discovery Nodes messages. Each message can contain up to `MAX_ADDR_TO_SEND` entries. Since each entry only needs a unique peer ID (same /16 IP, different peer IDs are all distinct multiaddrs), the attacker can generate 16384 unique entries from a single /16 block (65536 available IPs). The `is_valid_addr` filter requires `is_reachable` IPs, but any public /16 range satisfies this. The attack requires sustained message sending but no privileged access.

### Recommendation

Replace `len / 2` with `(len + 1) / 2` or `len.max(1)` to ensure at least one group is processed when `len >= 1`. Additionally, add a per-/16 admission cap in `add_addr` to prevent a single network group from monopolizing the store.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
