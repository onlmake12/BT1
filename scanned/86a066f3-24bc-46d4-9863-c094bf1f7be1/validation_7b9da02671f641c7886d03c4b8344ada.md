### Title
Unbounded Per-Peer Request Rate on Filter Protocol Enables I/O Amplification DoS — (`sync/src/filter/get_block_filter_hashes_process.rs`)

---

### Summary

The `BlockFilter` protocol handler has no rate limiter. Any connected peer can repeatedly send `GetBlockFilterHashes(start_number=0)`, each triggering up to **4 000 RocksDB reads** and a **~64 KB response**, with no throttle. N peers doing this simultaneously produce N × 4 000 reads and N × 64 KB of outbound traffic per request cycle, saturating disk I/O and network bandwidth and degrading sync/relay responsiveness.

---

### Finding Description

`GetBlockFilterHashesProcess::execute` iterates up to `BATCH_SIZE = 2000` times. Each iteration issues two sequential store lookups:

```
get_block_hash(block_number)          // RocksDB read #1
get_block_filter_hash(&block_hash)    // RocksDB read #2
``` [1](#0-0) [2](#0-1) 

That is up to **4 000 RocksDB point-reads** per message, producing a response of up to 2 000 × 32 = **64 KB**.

The `BlockFilter` handler struct carries no rate-limiter field and the `try_process` dispatch path performs no rate check before calling `execute`: [3](#0-2) [4](#0-3) 

By contrast, the `Relayer` handler explicitly carries a `RateLimiter<(PeerIndex, u32)>` keyed by `(peer, message_type)` and rejects requests that exceed 30 req/s per peer per message: [5](#0-4) [6](#0-5) 

The Filter protocol is included in the default protocol set, so every standard full node exposes it: [7](#0-6) 

---

### Impact Explanation

With N peers (up to `max_peers`, configurable, typically 125) each sending `GetBlockFilterHashes(start_number=0)` in a tight loop:

- **Disk I/O**: N × 4 000 RocksDB reads per cycle. At 125 peers this is 500 000 reads per round, easily saturating a spinning disk and significantly loading SSDs.
- **Outbound bandwidth**: N × 64 KB per round = 8 MB per round at 125 peers, at whatever rate the async runtime can drain the queue.
- **Collateral damage**: The sync and relay protocols share the same async executor and RocksDB instance. Saturating either resource delays block/header propagation and transaction relay, degrading the node's participation in consensus.

The cost to the attacker is negligible: open N TCP connections, send a 9-byte `GetBlockFilterHashes` struct repeatedly. No PoW, no stake, no authentication required.

---

### Likelihood Explanation

The attack path is entirely unprivileged and reachable from the public P2P interface. The Filter protocol is on by default. The attacker needs only standard TCP connectivity to the node's P2P port. The asymmetry with the Relayer (which has an explicit rate limiter) shows the omission is unintentional rather than a deliberate design choice.

---

### Recommendation

Apply the same `governor`-based per-peer rate limiter pattern already used in `Relayer` to the `BlockFilter` handler:

1. Add a `RateLimiter<PeerIndex>` field to `BlockFilter`.
2. In `try_process`, check the limiter before dispatching `GetBlockFilterHashes` (and the other two request variants).
3. Return `StatusCode::TooManyRequests` (and optionally ban the peer after repeated violations) on limit breach.

Additionally, consider a global (cross-peer) concurrency cap on in-flight Filter requests to bound aggregate RocksDB read pressure regardless of peer count.

---

### Proof of Concept

```rust
// Pseudocode – spawn N mock peers, each in a tight loop
for _ in 0..N_PEERS {
    tokio::spawn(async move {
        let mut stream = connect_to_node_filter_protocol().await;
        let msg = GetBlockFilterHashes { start_number: 0 }.encode();
        loop {
            stream.send(msg.clone()).await;
            // do not wait for response; just flood
        }
    });
}
// Observe: node's RocksDB read latency spikes, sync/relay message
// processing latency increases proportionally to N_PEERS.
```

Each spawned peer triggers up to 4 000 RocksDB reads and receives a ~64 KB `BlockFilterHashes` response with zero authentication or proof-of-work cost, confirming the multiplicative I/O amplification proportional to peer count.

        message: packed::RelayMessageUnionReader<'_>,

        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =

        if should_check_rate
            && self
                .rate_limiter
