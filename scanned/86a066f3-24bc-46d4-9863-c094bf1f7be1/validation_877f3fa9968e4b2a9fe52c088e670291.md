### Title
Zero-Cost Orphan Pool Slot Reservation Enables Griefing of Legitimate Orphan Transactions - (`tx-pool/src/component/orphan.rs`)

### Summary

The `OrphanPool::add_orphan_tx` function accepts transactions into the orphan pool with no fee check and no per-peer admission limit. A single unprivileged P2P peer can fill all 100 orphan pool slots with fabricated zero-cost transactions, causing legitimate orphan transactions from honest peers to be evicted at essentially zero attacker cost.

### Finding Description

When a relayed transaction fails resolution due to missing inputs, it is routed to the orphan pool via `add_orphan` in `process.rs`. The fee check (`check_tx_fee`) is only performed inside `pre_check`, which requires successfully resolving the transaction's inputs. Because orphan transactions by definition have unresolvable inputs, the fee check is never reached for them. [1](#0-0) 

The `add_orphan` call lands in `OrphanPool::add_orphan_tx`, which performs no fee validation whatsoever: [2](#0-1) 

The only DoS protection is a global cap of 100 entries and an expiry time: [3](#0-2) 

When the pool is full, `limit_size` evicts entries using `keys().next()` (HashMap insertion-order dependent, not truly random): [4](#0-3) 

An attacker can craft transactions whose inputs reference non-existent parent hashes. These pass `NonContextualTransactionVerifier` (which does not check input existence or fee rate), fail resolution with `is_missing_input`, and are admitted to the orphan pool at zero on-chain cost. [5](#0-4) 

There is no per-peer limit anywhere in `OrphanPool`. A single peer can occupy all 100 slots. [6](#0-5) 

### Impact Explanation

The orphan pool is a shared resource used during normal transaction relay: when a child transaction arrives before its parent (common in high-throughput relay), the child is parked in the orphan pool until the parent arrives. If the pool is saturated with attacker-controlled entries, legitimate child transactions are evicted. The evicted transactions are reported back to the relayer as rejected, causing the relaying peer to mark them as unknown and not re-relay them. This disrupts the two-step relay flow for honest users and can delay or permanently drop in-flight transactions. [7](#0-6) 

### Likelihood Explanation

Any connected P2P peer can execute this attack. The attacker needs only to send 100 relay messages containing transactions with fabricated input `OutPoint` hashes. No CKB balance, no valid UTXO ownership, and no fee payment is required. The attack can be sustained indefinitely by continuously sending new fake orphan transactions as old ones expire, at the cost of only network bandwidth. [8](#0-7) 

### Recommendation

1. **Per-peer orphan limit**: Track how many orphan slots each `PeerIndex` occupies and cap it (e.g., `DEFAULT_MAX_ORPHAN_TRANSACTIONS / max_peers`). This is the standard Bitcoin Core mitigation.
2. **Declared fee-rate pre-filter**: For remote transactions, require the declared cycle count to imply a minimum fee rate relative to transaction size before admitting to the orphan pool.
3. **Prioritized eviction**: When evicting, prefer entries from peers that have contributed the most orphans rather than using HashMap-order eviction.

### Proof of Concept

```
1. Connect to a CKB node as a P2P peer using the Relay protocol.
2. Craft 100 transactions, each with:
   - inputs referencing random 32-byte hashes (non-existent parents)
   - outputs with any valid capacity
   - witnesses that pass NonContextualTransactionVerifier
   - zero fee (outputs capacity == declared inputs capacity, unverifiable without resolution)
3. Send each via RelayTransaction with any declared cycle count.
4. Each transaction fails pre_check with is_missing_input → added to OrphanPool.
5. After 100 messages, OrphanPool.len() == DEFAULT_MAX_ORPHAN_TRANSACTIONS.
6. Now relay a legitimate child transaction (whose real parent is also being relayed).
7. Observe: the legitimate orphan is inserted then immediately evicted by limit_size,
   or evicts one attacker entry but the attacker immediately refills the slot.
8. The legitimate child transaction is lost from the orphan pool and its relay fails.
``` [9](#0-8) [10](#0-9)

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
