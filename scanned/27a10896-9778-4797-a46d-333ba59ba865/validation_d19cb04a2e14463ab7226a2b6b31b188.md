Audit Report

## Title
Relay Path Enforces `max_block_cycles` But Not `max_tx_verify_cycles`, Allowing Remote Peers to Force Expensive Script Verification — (`sync/src/relayer/transactions_process.rs`)

## Summary

The CKB relay handler validates remotely declared transaction cycles only against the consensus-level `max_block_cycles` (~3.5B), not against the tx-pool config-level `max_tx_verify_cycles` (70M default). A remote peer can relay a transaction whose scripts genuinely consume up to `max_block_cycles` cycles, forcing the receiving node to run script verification with a cycle budget up to ~50× the operator-configured per-transaction limit, defeating the DoS-reduction purpose of `max_tx_verify_cycles`.

## Finding Description

**Relay check (only `max_block_cycles` enforced):**

In `TransactionsProcess::execute`, the sole cycle-limit check is:

```rust
let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
if txs.iter().any(|(_, declared_cycles)| declared_cycles > &max_block_cycles) {
    self.nc.ban_peer(...);
    return Status::ok();
}
``` [1](#0-0) 

There is no corresponding check against `tx_pool_config.max_tx_verify_cycles`. The transaction is then forwarded to `submit_remote_tx` with the full `declared_cycles` value. [2](#0-1) 

**Script execution uses `declared_cycles` as the cycle budget:**

In `_process_tx`, for remote transactions `max_cycles` is set directly to `declared_cycles`:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
``` [3](#0-2) 

This `max_cycles` value is passed as the VM cycle limit to `verify_rtx`, which passes it to `verify_with_pause` / `verify` — the actual CKB-VM execution cap. [4](#0-3) 

**`max_tx_verify_cycles` is only used for scheduling, not rejection:**

`max_tx_verify_cycles` appears in the relay/tx-pool path only as `large_cycle_threshold` in `VerifyQueue`, used to classify transactions as "large cycle" for worker scheduling — not to reject them. [5](#0-4) [6](#0-5) 

In `process_orphan_tx`, the check `if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles` routes to the verify queue vs. direct processing — it is not a rejection path. [7](#0-6) 

**`declared_cycles` must match actual cycles — PoC correction:**

After script execution, `_process_tx` enforces that `declared_cycles == verified.cycles`, rejecting with `DeclaredWrongCycles` if they differ: [8](#0-7) 

This means the PoC as written (ALWAYS_SUCCESS script with `declared_cycles = max_block_cycles - 1`) would be rejected at this check. The correct attack requires a transaction whose scripts **genuinely consume** between `max_tx_verify_cycles` (70M) and `max_block_cycles` (3.5B) cycles, with `declared_cycles` set to match. Such transactions are constructible (e.g., a loop script consuming 500M cycles).

**`max_tx_verify_cycles` definition:** [9](#0-8) 

## Impact Explanation

An attacker who crafts a transaction consuming between 70M and 3.5B actual cycles can relay it to any CKB node. The relay check passes (cycles < `max_block_cycles`), and the node runs full script verification with a cycle budget up to 3.5B — up to ~50× the intended `max_tx_verify_cycles` limit. Repeating this with many transactions (each requiring a new script construction but no special privilege) can sustain CPU exhaustion on the target node. This matches the **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The `max_tx_verify_cycles` feature was introduced explicitly to "reduce DDOS vulnerability" (CHANGELOG), and the relay path entirely bypasses it.

## Likelihood Explanation

Any unprivileged P2P peer with a single TCP connection can send `RelayTransactions` messages. No key, hashpower, or majority is required. The attacker must construct transactions whose scripts genuinely consume more than 70M cycles, which is achievable with a simple loop script. The attack is repeatable and requires no victim interaction.

## Recommendation

In `TransactionsProcess::execute` (`sync/src/relayer/transactions_process.rs`), add a check against `max_tx_verify_cycles` immediately after the existing `max_block_cycles` check:

```rust
let max_tx_verify_cycles = self.relayer.shared().tx_pool_config().max_tx_verify_cycles;
if txs.iter().any(|(_, declared_cycles)| *declared_cycles > max_tx_verify_cycles) {
    self.nc.ban_peer(
        self.peer,
        DEFAULT_BAN_TIME,
        String::from("relay declared cycles greater than max_tx_verify_cycles"),
    );
    return Status::ok();
}
```

This ensures the relay ingestion path enforces the same per-transaction cycle cap as the tx-pool configuration intends.

## Proof of Concept

1. Construct a CKB script (e.g., a loop) that genuinely consumes `N` cycles where `70_000_001 ≤ N < max_block_cycles` (e.g., `N = 100_000_000`).
2. Build a valid transaction using this script as a lock or type script.
3. Connect to a target CKB node as a P2P peer supporting `RelayV3`.
4. Send a `RelayTransactions` message with `declared_cycles = N` (matching actual consumption).
5. The check at `transactions_process.rs:63-74` passes because `N < max_block_cycles`.
6. `_process_tx` sets `max_cycles = N` and runs script verification with a 100M-cycle budget — exceeding the operator-configured 70M limit.
7. The `declared_cycles == verified.cycles` check at `process.rs:736-748` passes because declared matches actual.
8. Repeat with many such transactions to sustain CPU load on the node, bypassing the DoS protection that `max_tx_verify_cycles` was designed to provide.

    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: Arc<TxVerifyEnv>,
    cache_entry: &Option<CacheEntry>,
    max_tx_verify_cycles: Cycle,
    command_rx: Option<&mut watch::Receiver<ChunkCommand>>,

    let consensus = snapshot.cloned_consensus();
    let data_loader = snapshot.as_data_loader();

    if let Some(completed) = cache_entry {
        TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
            .verify()
            .map(|_| *completed)
            .map_err(Reject::Verification)
    } else if let Some(command_rx) = command_rx {
        ContextualTransactionVerifier::new(
            Arc::clone(&rtx),
            consensus,
            data_loader,
            Arc::clone(&tx_env),

        .verify_with_pause(max_tx_verify_cycles, command_rx)
        .await
        .and_then(|result| {
            DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
                .verify()?;
            Ok(result)
        })
        .map_err(Reject::Verification)
    } else {
        block_in_place(|| {
            ContextualTransactionVerifier::new(Arc::clone(&rtx), consensus, data_loader, tx_env)
                .verify(max_tx_verify_cycles, false)
                .and_then(|result| {
                    DaoScriptSizeVerifier::new(
                        rtx,
                        snapshot.cloned_consensus(),
                        snapshot.as_data_loader(),
                    )
                    .verify()?;
                    Ok(result)
                })
                .map_err(Reject::Verification)
        })
    }
