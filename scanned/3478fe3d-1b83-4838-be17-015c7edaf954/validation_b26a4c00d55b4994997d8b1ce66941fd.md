Audit Report

## Title
`need_clean` samples a single arbitrary HashMap entry to decide subtree expiry, allowing suppression of orphan pool cleanup via trivially-mined far-future-epoch blocks — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` calls `map.iter().next()` on an unordered `HashMap` to sample exactly one direct child of a leader and uses that child's epoch to decide whether the entire subtree should be evicted. Because `non_contextual_verify` performs no PoW check and no epoch check, an unprivileged attacker can insert blocks with arbitrarily large epoch numbers into the orphan pool with trivial CPU cost. By inserting enough far-future-epoch siblings alongside expired orphans under the same leader, the attacker can statistically suppress cleanup, causing unbounded growth of in-memory structures and on-disk storage, ultimately crashing the node.

## Finding Description

**Root cause 1 — `need_clean` samples one arbitrary entry:**

`InnerPool::need_clean` in `chain/src/utils/orphan_block_pool.rs` (lines 113–122) calls `map.iter().next()` on a `HashMap<packed::Byte32, LonelyBlockHash>`. HashMap iteration order is non-deterministic. If a leader has `N` children and `M` carry a far-future epoch, the probability that cleanup is suppressed per cycle is `M/N`. The attacker controls `M`. [1](#0-0) 

**Root cause 2 — no PoW or epoch check before orphan pool insertion:**

`non_contextual_verify` in `chain/src/chain_service.rs` (lines 72–89) only calls `BlockVerifier` and `NonContextualBlockTxsVerifier`. [2](#0-1) 

`BlockVerifier::verify` in `verification/src/block_verifier.rs` (lines 39–47) runs only `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — no `PowVerifier`, no `EpochVerifier`. [3](#0-2) 

`PowVerifier` and `EpochVerifier` exist only inside `HeaderVerifier::verify` in `verification/src/header_verifier.rs` (lines 32–50), which is contextual and requires the parent block to be present. [4](#0-3) 

Notably, `PowVerifier` itself (lines 151–167) only takes the header and a `PowEngine` reference — it does not require the parent — so it could be run non-contextually but is not. [5](#0-4) 

**Orphan pool insertion path:**

After passing `non_contextual_verify`, the block is written to DB (`insert_block`, line 133) and forwarded to `orphan_broker.process_lonely_block` (line 143). If the parent is not stored and not invalid, the block is inserted into the orphan pool with no further validation. [6](#0-5) [7](#0-6) 

**Attack flow:**

1. Attacker picks or creates a leader hash `L` (any unknown parent hash).
2. Attacker inserts `K` expired-epoch blocks (epoch ≪ `tip_epoch − 6`) as children of `L`.
3. Attacker inserts `K` far-future-epoch blocks (`epoch = tip_epoch + 10000`, `compact_target = 0x207fffff`) as children of `L`. Each requires only trivial PoW (milliseconds of CPU).
4. Every 60 s, `clean_expired_orphan_timer` fires → `clean_expired_orphans` → `clean_expired_blocks` → `need_clean(L, tip_epoch)`.
5. With probability ≈ 50% per cycle, a far-future block is sampled → `need_clean` returns `false` → subtree not cleaned.
6. Attacker replenishes far-future siblings to maintain the ratio. Over time the orphan pool grows without bound. [8](#0-7) [9](#0-8) 

## Impact Explanation

`InnerPool::blocks` and `InnerPool::parents` (both `HashMap`s held under a `RwLock`) grow without bound. Each orphan block is also written to the DB before orphan pool insertion (`db_txn.insert_block`), so disk space is consumed in parallel. Sustained attack causes node OOM and/or disk exhaustion, crashing the node or making it unresponsive. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**. [10](#0-9) [11](#0-10) 

## Likelihood Explanation

The P2P block relay path is open to any peer with no authentication or visible rate-limiting in this code path. Mining a block with `compact_target = 0x207fffff` (minimum difficulty) takes milliseconds. The 60-second cleanup timer gives the attacker ample time to replenish far-future siblings between cycles. The attack is repeatable, cheap, and requires no special privilege.

## Recommendation

1. **Fix `need_clean`**: check that **all** direct children are expired, not just one arbitrary sample:
   ```rust
   fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
       self.blocks
           .get(parent_hash)
           .map(|map| {
               !map.is_empty()
                   && map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
           })
           .unwrap_or_default()
   }
   ```

2. **Add `PowVerifier` to `non_contextual_verify`**: `PowVerifier` requires only the header and `PowEngine`, not the parent block. Running it before orphan pool insertion closes the trivial-mining vector for all orphan-pool abuse.

3. **Cap orphan pool size**: enforce a hard limit on `InnerPool::parents.len()` and reject insertions beyond it (with peer penalization).

## Proof of Concept

Construct a leader hash `L` (any unknown parent hash). Insert 100 blocks with `epoch=1` (expired at `tip_epoch=20`) as children of `L`. Insert 100 blocks with `epoch=10020` (far-future) as children of `L`. Call `clean_expired_blocks(20)` 10,000 times. Assert that in a statistically significant fraction of calls `need_clean` returns `false` and the 100 expired blocks remain in the pool. Repeat with randomized insertion order to confirm the non-determinism is driven purely by HashMap iteration order. Separately, verify that a block with `compact_target = 0x207fffff` and `epoch = tip_epoch + 10000` passes `non_contextual_verify` without error. [12](#0-11)

    /// cleanup expired blocks(epoch + EXPIRED_EPOCH < tip_epoch)
    pub fn clean_expired_blocks(&mut self, tip_epoch: EpochNumber) -> Vec<LonelyBlockHash> {
        let mut result = vec![];

        for hash in self.leaders.clone().iter() {
            if self.need_clean(hash, tip_epoch) {
                // remove items in orphan pool and return hash to callee(clean header map)
                let descendants = self.remove_blocks_by_parent(hash);
                result.extend(descendants);
            }
        }
        result
    }

    /// get 1st block belongs to that parent and check if it's expired block
    fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
        self.blocks
            .get(parent_hash)
            .and_then(|map| {
                map.iter().next().map(|(_, lonely_block)| {
                    lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
                })
            })
            .unwrap_or_default()
    }
