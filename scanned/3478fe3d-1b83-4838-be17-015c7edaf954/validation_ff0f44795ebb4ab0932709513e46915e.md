Audit Report

## Title
Fee Rate Admission Check Uses Transaction Size Instead of Actual Weight, Allowing High-Cycle Transactions to Bypass Minimum Fee Rate — (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only serialized byte size (`tx_size`) as the weight denominator, while the actual weight used for pool scoring and eviction is `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. Because fee admission runs before script verification (before cycles are known), and no second check is performed after cycles are determined, a transaction with high cycles and small size can enter the pool — and be relayed across the network — with an actual fee rate far below `min_fee_rate`. This enables a cheap, repeatable CPU-and-pool-pollution griefing attack against any CKB node and the broader network.

## Finding Description
In `tx-pool/src/util.rs` at line 45, `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code comment at lines 42–44 explicitly acknowledges this is a proxy: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."*

The actual weight used for pool scoring, eviction, and fee rate reporting is defined in `util/types/src/core/tx_pool.rs` at lines 298–303:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

The processing pipeline in `_process_tx` (`tx-pool/src/process.rs`, lines 715–751) is:
1. `pre_check` → `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` — fee check using **size only** (line 289)
2. `verify_rtx(...)` — script execution, actual cycles determined here (line 724)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with actual cycles (line 751)
4. `submit_entry(...)` — added to pool (line 753)

There is **no second fee rate check** after step 2 using actual cycles. The `declared_cycles` value is checked only for correctness (lines 736–748), not used to re-validate the fee rate against actual weight.

`TxEntry::fee_rate()` in `tx-pool/src/component/entry.rs` (lines 114–118) correctly uses `get_transaction_weight(self.size, self.cycles)`, confirming the pool internally tracks the true fee rate — but this is never compared against `min_fee_rate` at admission time.

**Concrete example:**
- `tx_size = 1,000` bytes, `cycles = 70,000,000`, `min_fee_rate = 1,000 shannons/KW`
- Fee check threshold: `1,000 × 1,000 / 1,000 = 1,000 shannons`
- Actual weight: `max(1000, 70,000,000 × 0.000_170_571_4) = 11,940`
- Actual minimum fee needed: `11,940 shannons`
- A transaction paying 1,001 shannons passes `check_tx_fee` but has an actual fee rate of ≈83 shannons/KW — roughly 12× below the minimum.

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

An attacker can repeatedly submit transactions that:
1. Pass the size-based fee check with a minimal fee
2. Force every receiving node to execute full script verification (up to `max_tx_verify_cycles = 70,000,000` cycles per transaction)
3. Enter the tx pool and be relayed to peers, who apply the same flawed check and also accept them
4. Persist in pools across the network, consuming memory and polluting pool ordering

Because the same `check_tx_fee` logic runs on every node, the attack propagates network-wide. The attacker pays a fee proportional to `tx_size` but causes CPU work proportional to `cycles` on every node that processes the transaction.

## Likelihood Explanation
- Entry path is fully open: any `send_transaction` RPC caller or relaying peer can submit such a transaction
- No special privileges, keys, or majority hashpower required
- The attack is cheap: a 1,000-byte transaction with 70M cycles costs only ~1,000 shannons to pass the fee check
- The attack is repeatable: each submission forces a full script verification pass on every receiving node
- The code comment explicitly acknowledges the imprecision, confirming this is a known gap

## Recommendation
After `verify_rtx` determines the actual cycles in `_process_tx`, perform a second fee rate check using the true weight:

```rust
// After verify_rtx returns verified.cycles (after line 734 in process.rs):
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, actual_min_fee.as_u64(), fee.as_u64())), snapshot));
}
```

Alternatively, for remote transactions where `declared_cycles` is available before verification, use `get_transaction_weight(tx_size, declared_cycles)` in `check_tx_fee` as a tighter pre-check, eliminating the CPU cost before verification even begins.

## Proof of Concept
1. Construct a CKB transaction with `tx_size ≈ 1,000` bytes and a script that consumes close to `max_tx_verify_cycles` (70,000,000) cycles.
2. Set the fee to just above `min_fee_rate.fee(tx_size)` (e.g., 1,001 shannons for `min_fee_rate = 1,000 shannons/KW`).
3. Submit via `send_transaction` RPC.
4. Observe: the transaction passes `check_tx_fee`, triggers full script verification, and enters the pool.
5. Query the pool; confirm the entry's `fee_rate()` (which uses actual weight) is far below `min_fee_rate`.
6. Repeat to demonstrate repeatability and CPU exhaustion.
7. Confirm the transaction is relayed to peers, which also accept it under the same flawed check.
