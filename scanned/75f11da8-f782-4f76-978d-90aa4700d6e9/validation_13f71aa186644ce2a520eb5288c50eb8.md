### Title
Unbounded GetBlocks Amplification via Missing Rate Limiter in Synchronizer::try_process — (`sync/src/synchronizer/get_blocks_process.rs`)

### Summary
`Synchronizer::try_process` dispatches `GetBlocks` to `GetBlocksProcess::execute` with no rate limiting. An unprivileged remote peer can send `GetBlocks` messages at wire speed, each triggering up to 32 full block responses, creating a large bandwidth amplification attack.

### Finding Description
`Relayer::try_process` gates every non-`CompactBlock` message through `self.rate_limiter.check_key(&(peer, message.item_id()))` at 30 req/s per `(peer, message_type)`. The `Synchronizer` struct has no `rate_limiter` field and no equivalent check. `GetBlocksProcess::execute` accepts up to 32 block hashes per message and, for each hash that resolves to a `BLOCK_VALID` block, spawns an async task to send the full serialized block to the requesting peer. Valid block hashes are public information trivially obtained from any block explorer or by syncing headers first.

### Impact Explanation
Each `GetBlocks` message is ~1 KB (32 × 32-byte hashes). Each response is a full block, which can be up to the consensus block size limit (several hundred KB to MB). At 32 blocks per request, the amplification ratio is on the order of hundreds to thousands. An attacker with a single connection can saturate the victim's outbound bandwidth, degrading its ability to relay blocks and transactions to legitimate peers.

### Likelihood Explanation
The attack requires only a standard P2P connection — no special privileges, no PoW, no keys. Valid block hashes are freely available. The attacker controls message rate only by their own inbound bandwidth, which is far cheaper than the victim's outbound cost.

### Recommendation
Add a per-peer rate limiter to `Synchronizer::try_process` for `GetBlocks` (and `GetHeaders`), mirroring the `governor::RateLimiter` pattern already used in `Relayer`. A limit of 10–30 req/s per peer per message type would eliminate the amplification while not impacting legitimate sync behavior.

### Proof of Concept
1. Connect to a victim CKB node as a normal peer.
2. Collect 32 valid block hashes from the chain (trivially available).
3. In a tight loop, send `SyncMessage::GetBlocks` containing those 32 hashes.
4. Observe: victim sends 32 full blocks per request with no throttle; attacker inbound cost ~1 KB/req, victim outbound cost ~32× block_size/req.
5. Differential: repeat with `Relayer::GetBlockTransactions` — rate-limited to 30 req/s; `Synchronizer::GetBlocks` — unlimited. Assert outbound bandwidth ratio exceeds 100× attacker inbound bandwidth.
