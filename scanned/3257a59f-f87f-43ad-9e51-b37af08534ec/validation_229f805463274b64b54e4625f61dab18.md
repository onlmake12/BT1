### Title
OnlySmallCycleTx Worker O(N) Scan + `re_notify()` Busy-Loop When Queue Is Saturated With Large-Cycle Transactions — (`tx-pool/src/component/verify_queue.rs`, `tx-pool/src/verify_mgr.rs`)

---

### Summary

When `max_tx_verify_workers > 1`, worker index 0 is assigned `WorkerRole::OnlySmallCycleTx`. If an unprivileged remote peer floods the verify queue with transactions whose declared cycle count exceeds `large_cycle_threshold`, the `OnlySmallCycleTx` worker enters a persistent wakeup loop: each iteration performs two full O(N) linear scans of the queue, finds nothing, calls `re_notify()`, and immediately wakes itself up again. The `MultiIndexMap` already maintains a `hashed_non_unique` index on `is_large_cycle` that would allow O(1) lookup, but the code ignores it.

---

### Finding Description

**Attacker-controlled entry point:**

A remote peer submits transactions via P2P relay with a declared cycle count above `large_cycle_threshold` (default `max_tx_verify_cycles = 70_000_000`). The `is_large_cycle` flag is set purely from the peer-supplied value: [1](#0-0) 

No PoW or consensus check gates this flag — any peer can set it to `true` for every submitted transaction.

**O(N) scan in `peek(only_small_cycle=true)`:**

When the queue contains only large-cycle entries, `peek()` performs two full sequential scans: [2](#0-1) 

- Scan 1 (line 183): `iter_by_added_time().find(|e| e.is_proposal_tx)` — exhausts all N entries.
- Scan 2 (line 188): `iter_by_added_time().find(|e| !e.is_large_cycle)` — exhausts all N entries again.
- Both return `None`. Total cost: 2N comparisons.

The `VerifyEntry` struct already declares `is_large_cycle` with `#[multi_index(hashed_non_unique)]`, meaning the `MultiIndexMap` maintains a hash bucket for it. An O(1) `get_by_is_large_cycle(false)` call exists but is not used. [3](#0-2) 

**`re_notify()` wakeup loop in `process_inner()`:**

After `pop_front(true)` returns `None` (because `peek()` found nothing), `process_inner()` checks `is_empty()` and, finding the queue non-empty, calls `re_notify()`: [4](#0-3) 

`re_notify()` calls `tokio::sync::Notify::notify_one()`: [5](#0-4) 

Tokio's `Notify` stores one permit. If the `SubmitTimeFirst` workers are busy (holding the write lock processing large-cycle txs), the `OnlySmallCycleTx` worker is the only waiter. The permit is consumed immediately on the next `queue_ready.notified()` in the `run()` loop: [6](#0-5) 

This creates the cycle: `notified()` fires → `process_inner()` → 2×O(N) scan → `re_notify()` → `notified()` fires → repeat.

**Worker role assignment:**

Worker 0 is unconditionally `OnlySmallCycleTx` whenever `worker_num > 1`, which is the default (3/4 of CPU cores): [7](#0-6) [8](#0-7) 

---

### Impact Explanation

The `OnlySmallCycleTx` worker consumes a Tokio task thread performing 2N comparisons per async iteration with no useful work done. Each iteration also acquires the `RwLock<VerifyQueue>` write lock, contending with `SubmitTimeFirst` workers. This:

1. Wastes CPU proportional to queue depth on every wakeup cycle.
2. Increases write-lock contention, slowing down `SubmitTimeFirst` workers that are doing real work.
3. Degrades overall tx verification throughput on the node.

The queue is bounded at 256 MB (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`), so N is bounded. The impact is performance degradation, not complete halt. `SubmitTimeFirst` workers continue draining the queue, but more slowly due to lock contention. [9](#0-8) 

---

### Likelihood Explanation

- The attacker needs only to submit valid transactions (meeting `min_fee_rate`) with a declared cycle count above `large_cycle_threshold`. No key, PoW, or privileged access is required.
- The default `max_tx_verify_workers` is `max(num_cpus * 3/4, 1)`, so on any multi-core node `worker_num > 1` and the `OnlySmallCycleTx` role is always active.
- The attacker must sustain the queue fill rate faster than `SubmitTimeFirst` workers drain it. Since large-cycle txs take longer to verify, this is achievable with moderate tx submission rate.
- The condition is locally testable: fill queue with N large-cycle txs, observe CPU time in `peek()` vs actual verification time.

---

### Recommendation

Replace the O(N) linear scan in `peek()` with the existing `hashed_non_unique` index on `is_large_cycle`. The `MultiIndexMap`-generated API provides `get_by_is_large_cycle(false)` (or equivalent) for O(1) existence check. Additionally, add a fast-path in `process_inner()` for the `OnlySmallCycleTx` role: if the queue contains zero small-cycle entries (checkable in O(1)), skip the write-lock acquisition and `re_notify()` entirely, and instead wait on a separate `Notify` that is only signaled when a small-cycle tx is added.

---

### Proof of Concept

```
1. Configure node with max_tx_verify_workers >= 2 (default on any multi-core machine).
2. Set large_cycle_threshold = max_tx_verify_cycles (default 70_000_000).
3. Submit N transactions via P2P with declared_cycles = large_cycle_threshold + 1.
   Each tx must meet min_fee_rate but need not actually execute that many cycles.
4. Observe: OnlySmallCycleTx worker (worker 0) wakes up, calls peek(true),
   performs 2N comparisons, returns None, calls re_notify(), immediately wakes up again.
5. Measure: CPU time in peek() grows linearly with N.
   With N = 10,000 txs, each wakeup costs ~20,000 comparisons.
   With SubmitTimeFirst workers busy on large-cycle verification (seconds per tx),
   the OnlySmallCycleTx worker can spin thousands of times per second.
6. Assert: scan cost is O(N) not O(1); total wasted CPU = O(N * wakeup_rate).
```

    pub fn peek(&self, only_small_cycle: bool) -> Option<ProposalShortId> {
        let mut iter = self.inner.iter_by_added_time();

        if let Some(proposal_entry) = iter.find(|e| e.is_proposal_tx) {
            return Some(proposal_entry.inner.tx.proposal_short_id());
        }

        let entry = if only_small_cycle {
            self.inner.iter_by_added_time().find(|e| !e.is_large_cycle)
        } else {
            self.inner.iter_by_added_time().next()
        };

        entry.map(|e| e.inner.tx.proposal_short_id())
    }
