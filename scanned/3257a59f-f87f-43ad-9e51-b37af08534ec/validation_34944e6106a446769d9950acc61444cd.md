### Title
Out-of-Bounds Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

`TxConfirmStat::remove_unconfirmed_tx` guards the `block_unconfirmed_txs` array access with a bounds check (`tx_age >= len`), but applies **no equivalent guard** before indexing `confirm_blocks_to_failed_txs[tx_age - 1]`. When a tracked transaction is rejected after more than `MAX_CONFIRM_BLOCKS` (1000) blocks in the pool, `tx_age - 1 >= 1000` exceeds the array length, causing an index-out-of-bounds **panic** that crashes the node process.

---

### Finding Description

`confirm_blocks_to_failed_txs` is allocated with exactly `MAX_CONFIRM_BLOCKS = 1000` rows (valid indices 0–999): [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, the guard at line 208 only protects the `block_unconfirmed_txs` write. The subsequent `count_failure` branch at line 215 uses `tx_age - 1` as an unchecked index into `confirm_blocks_to_failed_txs`: [3](#0-2) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds → **panic**.

The call chain from a tx rejection is fully wired in production:

1. Any rejection fires the registered reject callback in `shared_builder.rs`: [4](#0-3) 

2. That callback calls `fee_estimator.reject_tx(&tx_hash)`, which dispatches to `Algorithm::reject_tx`: [5](#0-4) 

3. `reject_tx` calls `drop_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx`: [6](#0-5) 

---

### Impact Explanation

A Rust index-out-of-bounds panic aborts the process. Any CKB node configured with the `ConfirmationFraction` fee estimator will crash when any tracked transaction older than 1000 blocks is evicted from the pool. This is a remotely triggerable, unprivileged **node crash (DoS)**.

---

### Likelihood Explanation

- Default tx expiry is 72 hours (time-based). At ~8 s/block, 1001 blocks ≈ 2.2 hours — well within the expiry window.
- A low-fee tx submitted by any user naturally ages past 1000 blocks on a congested network.
- Eviction triggers include: pool-full eviction (`limit_size`), expiry, RBF conflict, or chain reorg invalidation — all of which call the reject callback unconditionally.
- No special privilege, key, or hashpower is required. A single valid (but low-fee) transaction submission is sufficient.

---

### Recommendation

Add a bounds check before the `count_failure` write, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure {
    if let Some(row) = self.confirm_blocks_to_failed_txs.get_mut(tx_age - 1) {
        row[bucket_index] += 1f64;
    }
    // silently drop samples for txs older than MAX_CONFIRM_BLOCKS
}
```

This matches the semantic intent: transactions older than the tracking window are already classified as `old_unconfirmed_txs` and their failure data is not meaningful for the estimator.

### Proof of Concept

```rust
#[test]
fn test_remove_unconfirmed_tx_no_panic_for_old_txs() {
    let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
    let max_confirm_blocks = 1000;
    let decay = 0.993f64;
    let mut stat = TxConfirmStat::new(buckets, max_confirm_blocks, decay);

    // Simulate a tx tracked at height 0, bucket 0
    stat.add_unconfirmed_tx(0, FeeRate::from_u64(1500));

    // Move all 1000 windows so the tx becomes old_unconfirmed
    for h in 1..=1000u64 {
        stat.move_track_window(h);
    }
    // Manually bump old_unconfirmed_txs to avoid underflow on the -= 1
    stat.bucket_stats[0].old_unconfirmed_txs += 1;

    // tx_age = 999: index 998 — OK
    stat.remove_unconfirmed_tx(0, 999, 0, true);

    stat.bucket_stats[0].old_unconfirmed_txs += 1;
    // tx_age = 1000: index 999 — OK (boundary)
    stat.remove_unconfirmed_tx(0, 1000, 0, true);

    stat.bucket_stats[0].old_unconfirmed_txs += 1;
    // tx_age = 1001: index 1000 — OUT OF BOUNDS → panic on unpatched code
    stat.remove_unconfirmed_tx(0, 1001, 0, true);

    stat.bucket_stats[0].old_unconfirmed_txs += 1;
    // tx_age = 1002: index 1001 — also panics
    stat.remove_unconfirmed_tx(0, 1002, 0, true);
}
```

Running this test against the unpatched code panics at `tx_age = 1001` with `index out of bounds: the len is 1000 but the index is 1000`.
