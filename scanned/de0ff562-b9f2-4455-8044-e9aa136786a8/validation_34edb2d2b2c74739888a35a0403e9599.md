### Title
Unbounded MMR Proof Generation Cost via `GetBlocksProof` with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

### Summary

An unprivileged remote peer can send repeated `GetBlocksProof` messages each containing up to 1000 valid main-chain block hashes, causing the server to call `mmr.gen_proof(positions)` with 1000 positions per request against an MMR of size proportional to the chain height, with no per-peer rate limiting in place.

### Finding Description

`GET_BLOCKS_PROOF_LIMIT` is set to 1000: [1](#0-0) 

`GetBlocksProofProcess::execute()` enforces only a count check and a duplicate check, then passes all found positions directly to `reply_proof`: [2](#0-1) [3](#0-2) [4](#0-3) 

`reply_proof` unconditionally calls `mmr.gen_proof(items_positions)` with all positions: [5](#0-4) 

There is no rate limiting, request throttling, or per-peer concurrency control anywhere in the light-client-protocol-server:



The `received` handler dispatches directly to `try_process` with no guard: [6](#0-5) 

### Impact Explanation

Each request with 1000 valid main-chain block hashes triggers O(1000 × log H) MMR node reads from the database, where H is the chain height. On a chain of height 500,000, that is approximately 19,000 DB reads per request. A single peer can send these requests in a tight loop with no server-side throttle, causing sustained elevated DB I/O and CPU load. The impact is bounded per request but unbounded in aggregate frequency — **Low resource exhaustion DoS**.

### Likelihood Explanation

The attack requires only knowledge of valid main-chain block hashes, which is public information available to any synced peer. No privilege, key, or PoW is required. The path is fully reachable via the P2P light client protocol.

### Recommendation

1. Add per-peer rate limiting on `GetBlocksProof` (and similarly `GetTransactionsProof` / `GetLastStateProof`) requests.
2. Consider reducing `GET_BLOCKS_PROOF_LIMIT` or adding a cost-based cap (e.g., positions × log(chain_height) ≤ threshold).
3. Enforce a concurrency limit so that at most one expensive proof generation runs per peer at a time.

### Proof of Concept

1. Connect a peer to a full node with the light client protocol enabled.
2. Collect 1000 block hashes from the main chain (publicly available via any synced node).
3. Send a `GetBlocksProof` message with all 1000 hashes and a valid `last_hash`.
4. Repeat in a tight loop.
5. Observe elevated DB read counts and CPU usage on the server, with no ban or throttle applied.
