Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Post-Eviction `total_tx_size`/`total_tx_cycles` in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures new pool-size totals into local variables before `check_and_record_ancestors` runs. When that call evicts transactions via `remove_entry_and_descendants → remove_entry → update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size`/`self.total_tx_cycles`. However, the stale pre-eviction locals are then unconditionally written back at lines 218–219, erasing those decrements. The result is a permanently inflated size counter that causes `limit_size` to evict valid transactions and causes future submissions to be rejected with `Reject::Full` even when the pool has real capacity.

## Finding Description
The exact sequence in `add_entry` (L210–219):

```
L210-211: total_tx_size = self.total_tx_size + entry.size   // snapshot taken; self.total_tx_size = S
L213:     check_and_record_ancestors(&mut entry)
            └─ remove_entry_and_descendants (L618)
                 └─ remove_entry (L263)
                      └─ update_stat_for_remove_tx (L247)
                           └─ self.total_tx_size -= evicted_size  // now S - evicted_size
L218:     self.total_tx_size = total_tx_size   // overwrites with S + new_size  ← BUG
```

`updated_stat_for_add_tx` (L711–729) is a pure read of `self.total_tx_size` plus `entry.size`; it does not mutate state. `remove_entry_and_descendants` (L252–264) iterates all descendants and calls `remove_entry` (L235–249) for each, which calls `update_stat_for_remove_tx` (L733–757) to decrement `self.total_tx_size`. After eviction, `self.total_tx_size = S − evicted_size`. The unconditional assignment at L218 then sets it to `S + new_size`, inflating the counter by exactly `evicted_size` instead of the correct `S + new_size − evicted_size`. The eviction path is only reached when `ancestors_count > self.max_ancestors_count` and `cell_ref_parents` is non-empty (L598–628), so the bug is silent in the common case and only fires on the specific eviction branch.

## Impact Explanation
`limit_size` (pool.rs L298) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting lowest-fee-rate transactions. An inflated counter causes it to evict valid, fee-paying transactions that should remain. Subsequent calls to `updated_stat_for_add_tx` start from the inflated baseline, causing valid transactions to be rejected with `Reject::Full`. An attacker who repeatedly triggers the eviction path accumulates inflation across submissions, progressively degrading mempool capacity until legitimate transactions are systematically rejected. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (10001–15000 points). The attacker needs no privileges, only the ability to submit transactions via `send_transaction` or the P2P relay.

## Likelihood Explanation
The trigger condition requires a new transaction whose ancestor count (including `cell_ref_parents`) exceeds `max_ancestors_count` (default 25), with at least one `cell_ref_parent` that can be evicted to bring the count within limit. An attacker who observes the mempool via `get_raw_tx_pool` can construct exactly this scenario: build a 24-deep chain, submit a cell-dep consumer `R`, then submit `T_new` spending the same cell. This is deliberate but straightforward, requires no special privilege, and can be repeated to accumulate inflation. The condition also arises naturally in organic mempool usage, meaning the bug can trigger without a deliberate attacker.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so the snapshot reflects the post-eviction state of `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, replace the local-variable pattern with in-place increments (`self.total_tx_size += entry.size`) applied only after all mutations are complete, and handle overflow separately.

## Proof of Concept
1. Fill the pool with a chain `T1 → T2 → … → T24` (each spending the previous output).
2. Submit transaction `R` that takes `T1`'s output as a **cell dep** (`R` becomes a `cell_ref_parent`).
3. Submit `T_new` spending `T1`'s output. Ancestor count = 24 + `R` = 25, exceeding `max_ancestors_count`.
4. `check_and_record_ancestors` evicts `R` (size `S_R`) via `remove_entry_and_descendants`, decrementing `self.total_tx_size` by `S_R`.
5. L218 overwrites `self.total_tx_size` with the pre-eviction snapshot, inflating it by `S_R`.
6. Call `get_tip_tx_pool_info` and compare `total_tx_size` against the sum of actual entry sizes — the counter exceeds the real sum by `S_R`.
7. Observe that `limit_size` subsequently evicts a valid transaction that fits within `max_tx_pool_size`, confirming the spurious eviction.
8. Repeat steps 1–5 multiple times to accumulate inflation and eventually cause `Reject::Full` for transactions that fit within the real pool capacity.
