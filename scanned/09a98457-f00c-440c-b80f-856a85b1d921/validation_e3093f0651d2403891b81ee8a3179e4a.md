Audit Report

## Title
Peer-Controlled `declared_cycles` as VM Cap Enables Permanent Transaction Relay Suppression - (File: `tx-pool/src/process.rs`)

## Summary
A remote peer supplying `declared_cycles` below the actual script cost causes CKB-VM to fail with `ExceededMaximumCycles`, which is classified as a malformed-transaction Script error. Because `is_allowed_relay()` returns `false` for this error class, `TxVerificationResult::Reject` is never sent to the relayer, so the tx hash is never removed from the known-tx filter. Since the hash was inserted into the filter before verification, honest peers are permanently blocked from relaying the same transaction to the victim node for the duration of the filter TTL (4 hours), and the attacker can re-inject before expiry.

## Finding Description

**Step 1 — Hash inserted before verification** (`sync/src/relayer/transactions_process.rs`):

After the only pre-filter (banning `declared_cycles > max_block_cycles`), the tx hash is unconditionally marked as known:

```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
``` [1](#0-0) 

A value of `declared_cycles = 1` passes this check trivially.

**Step 2 — Peer-supplied value used verbatim as VM cap** (`tx-pool/src/process.rs`):

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
``` [2](#0-1) 

`verify_rtx` is then called with this attacker-controlled cap. When `declared_cycles = 1` and the actual script cost is, e.g., 537 cycles (standard secp256k1), the VM halts with `ExceededMaximumCycles`.

**Step 3 — Crucially, `DeclaredWrongCycles` is NOT produced** (`tx-pool/src/process.rs`):

The `DeclaredWrongCycles` path at L736–748 is only reached when `verify_rtx` returns `Ok`. When the VM aborts, `try_or_return_with_snapshot!` at L734 short-circuits with `Reject::Verification(Script error)` — the cycles-mismatch branch is never evaluated. [3](#0-2) 

**Step 4 — Script error is classified as malformed** (`util/types/src/core/tx_pool.rs`):

```rust
ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
``` [4](#0-3) 

`ExceededMaximumCycles` does not contain `ARGV_TOO_LONG_TEXT`, so `is_malformed_from_verification` returns `true`, making `is_malformed_tx()` return `true` for `Reject::Verification(Script error)`. [5](#0-4) 

**Step 5 — `is_allowed_relay()` returns `false`** (`util/types/src/core/tx_pool.rs`):

```rust
pub fn is_allowed_relay(&self) -> bool {
    matches!(self, Reject::DeclaredWrongCycles(..))
        || (!matches!(self, Reject::LowFeeRate(..)) && !self.is_malformed_tx())
}
``` [6](#0-5) 

Since the reject is malformed and not `DeclaredWrongCycles`, `is_allowed_relay()` is `false`.

**Step 6 — `TxVerificationResult::Reject` is never sent** (`tx-pool/src/process.rs`):

```rust
if reject.is_malformed_tx() {
    self.ban_malformed(peer, format!("reject {reject}")).await;
}
if reject.is_allowed_relay() {
    self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash: tx_hash.clone() });
}
``` [7](#0-6) 

The peer is banned, but `TxVerificationResult::Reject` is never enqueued.

**Step 7 — Without `TxVerificationResult::Reject`, the hash is never removed** (`sync/src/relayer/mod.rs`):

```rust
TxVerificationResult::Reject { tx_hash } => {
    self.shared.state().remove_from_known_txs(&tx_hash);
}
``` [8](#0-7) 

This branch is the only place the hash is removed from the filter. It is never reached.

**Contrast with over-declaration**: When `declared_cycles > actual`, verification succeeds, the `DeclaredWrongCycles` branch fires, `is_allowed_relay()` returns `true`, `TxVerificationResult::Reject` is sent, and the hash IS removed from the filter — honest peers can re-relay. [9](#0-8) 

**Filter TTL**: Entries expire after `FILTER_TTL = 4 * 60 * 60` seconds (4 hours). [10](#0-9) 

The attacker can re-inject before expiry using rotated IPs after the 3-day ban expires, or immediately from a different IP.

## Impact Explanation

An unprivileged remote peer can permanently suppress any specific valid transaction from a victim node's tx-pool for the 4-hour filter TTL, renewable indefinitely. If the victim is a mining node, the targeted transaction is blocked from confirmation. If applied at scale across multiple nodes, this constitutes targeted transaction censorship affecting CKB economy. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **Critical — Vulnerabilities which could easily damage CKB economy** when applied to high-value or time-sensitive transactions across multiple nodes.

## Likelihood Explanation

The attack requires only a standard P2P connection and knowledge of the target transaction hash (observable from any node's mempool broadcast). No key material, hashpower, or privileged access is needed. The attacker sends one `RelayTransactionHashes` message, waits for `GetRelayTransactions`, then sends `RelayTransactions` with `cycles = 1`. The 3-day IP ban is bypassable via IP rotation or Tor. The attack is repeatable every 4 hours per victim node per transaction.

## Recommendation

1. **Do not use `declared_cycles` as the VM `max_cycles` cap.** Always verify with `consensus.max_block_cycles()` (or `max_tx_verify_cycles`) as the hard cap. Use `declared_cycles` only as a hint for fee/priority calculation after successful verification.
2. **Treat `ExceededMaximumCycles` caused by a peer-supplied cap as `DeclaredWrongCycles`**: detect when `verified_ret` fails with a cycles-exceeded error and the declared value is below the consensus cap, then return `Reject::DeclaredWrongCycles` so `is_allowed_relay()` returns `true` and the hash is cleared.
3. **Alternatively, mark the tx hash as known only after successful verification**, or unconditionally remove it from the filter on any non-`Duplicated` failure path.

## Proof of Concept

1. Observe a valid pending transaction `T` with actual script cost of 537 cycles (standard secp256k1 lock) on the network.
2. Connect to victim node as a peer via `RelayV3`.
3. Send `RelayTransactionHashes` containing `T`'s hash; wait for `GetRelayTransactions` response.
4. Send `RelayTransactions` with `T` and `cycles = 1`.
5. **Victim node**: marks `T`'s hash as known (L76), runs VM with `max_cycles = 1`, gets `ExceededMaximumCycles` → `Reject::Verification(Script)`, bans attacker peer, but does **not** send `TxVerificationResult::Reject` → hash remains in filter.
6. Honest peer B now tries to relay `T` with correct `cycles = 537`. The victim's filter check `!tx_filter.contains(&T.hash())` returns `false` → `T` is silently dropped at L49–55.
7. `T` never enters the victim's tx-pool.
8. Repeat from step 2 using a new IP before the 4-hour TTL expires to maintain suppression indefinitely.

The existing integration test `DeclaredWrongCyclesChunk` confirms the over-declaration path is handled correctly, but the under-declaration path (cycles below actual) produces a different error code that bypasses the re-relay mechanism entirely. [11](#0-10)

fn is_malformed_from_verification(error: &Error) -> bool {
    match error.kind() {
        ErrorKind::Transaction => error
            .downcast_ref::<TransactionError>()
            .expect("error kind checked")
            .is_malformed_tx(),
        ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
        ErrorKind::Internal => {
            error
                .downcast_ref::<InternalError>()
                .expect("error kind checked")
                .kind()
                == InternalErrorKind::CapacityOverflow
        }
        _ => false,
    }
}
