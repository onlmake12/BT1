### Title
Unbounded Inline Script Verification in `process_orphan_tx` Enables CPU-Exhaustion DoS — (File: `tx-pool/src/process.rs`)

---

### Summary

`process_orphan_tx` in the CKB tx-pool service performs inline CKB-VM script verification for every orphan transaction that depends on a newly accepted parent, without any per-invocation cap on total work. An unprivileged relay peer can pre-load the orphan pool (hard-capped at 100 entries) with transactions all spending distinct outputs of the same crafted parent, then submit the parent to trigger up to 100 consecutive synchronous script verifications in a single call, causing CPU exhaustion and repeated tx-pool write-lock contention that degrades or stalls transaction processing for all other peers.

---

### Finding Description

**Root cause — `process_orphan_tx` (lines 591–671, `tx-pool/src/process.rs`)**

When a transaction is accepted into the pool, `after_process` unconditionally calls `process_orphan_tx` (lines 500 and 536). The function performs a BFS over the orphan pool: for every orphan whose `declared_cycle ≤ max_tx_verify_cycles`, it calls `_process_tx` with `command_rx = None` (line 626), which runs full CKB-VM script verification synchronously to completion with no pause or interrupt mechanism. [1](#0-0) 

The orphan pool is capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries: [2](#0-1) 

However, `limit_size` only enforces the total count; it places **no limit on how many orphans may reference the same parent output**: [3](#0-2) 

`find_by_previous` returns every orphan that references any output of the given transaction, so all 100 pool slots can be filled with orphans spending distinct outputs of one parent: [4](#0-3) 

**Attack path**

1. Attacker crafts a parent transaction with ≥ 100 outputs (well within the block-size limit; each minimal `CellOutput` is ~50 bytes).
2. Attacker relays 100 orphan transactions via the P2P relay protocol, each spending a different output of the parent. Because the parent is unknown, each orphan is accepted into the orphan pool.
3. Attacker submits the parent. `after_process` calls `process_orphan_tx`, which finds all 100 orphans and calls `_process_tx` for each one inline.
4. Each `_process_tx` call runs `verify_rtx` with `command_rx = None` (no chunking/pause), consuming up to `max_tx_verify_cycles` CKB-VM cycles per orphan: [5](#0-4) [6](#0-5) 

**Amplification via `remove_orphan_txs_by_attach`**

During a chain reorg, `update_tx_pool_for_reorg` calls `remove_orphan_txs_by_attach`, which invokes `process_orphan_tx` for **every transaction in every attached block**. If the orphan pool is kept full by the attacker across multiple blocks, the total inline verification work scales as `attached_tx_count × 100 × max_tx_verify_cycles`: [7](#0-6) 

**Lock contention**

Each of the 100 inline `_process_tx` calls acquires and releases the tx-pool write lock (via `pre_check` and `submit_entry`). Concurrent tasks that need this lock — including legitimate transaction submissions — are blocked during each acquisition, stalling normal pool operation. [8](#0-7) 

---

### Impact Explanation

The tx-pool service task processes each incoming message in a spawned sub-task. While `process_orphan_tx` does not block the main dispatch loop, it:

- Consumes up to `100 × max_tx_verify_cycles` CKB-VM cycles of CPU in a single call, potentially taking seconds on mainnet cycle budgets.
- Causes repeated write-lock contention on the shared `TxPool` structure, serialising all concurrent transaction submissions for the duration.
- Can be triggered repeatedly by any relay peer at the cost of crafting cheap transactions, with no rate limit specific to orphan-triggered verification.

The net effect is a sustained degradation or temporary stall of transaction processing — a denial-of-service against the tx-pool service reachable by any unprivileged P2P peer.

---

### Likelihood Explanation

The attack requires no special privileges. Any peer connected via the relay protocol (`SupportProtocols::RelayV3`) can submit orphan transactions. Crafting a parent with 100 outputs and 100 child orphans is straightforward and cheap on-chain. The orphan pool accepts transactions from any peer up to the 100-entry cap, and there is no per-peer or per-parent-output limit on orphan submissions. [9](#0-8) 

---

### Recommendation

1. **Cap inline orphan processing per call.** Introduce a maximum number of orphans processed inline per `process_orphan_tx` invocation (e.g., 10). Remaining orphans can be re-queued or deferred to the async verify queue.

2. **Limit orphans per parent output.** In `add_orphan_tx`, cap the number of orphan entries that may reference the same `OutPoint` (e.g., 5), preventing a single parent from triggering a full-pool sweep.

3. **Route all orphan verification through the verify queue.** Remove the inline `_process_tx` path in `process_orphan_tx` entirely; always enqueue orphans to the async `VerifyQueue` regardless of declared cycle count. This decouples orphan resolution from the calling task and subjects it to the queue's existing size and rate controls. [10](#0-9) 

---

### Proof of Concept

```
1. Connect to a CKB node via the RelayV3 P2P protocol.

2. Construct parent_tx with 100 outputs, each a minimal always-success cell.

3. For i in 0..100:
     child_i = Transaction {
         inputs:  [OutPoint(parent_tx.hash(), i)],   // spends output i
         outputs: [one output],
         witnesses: [always-success witness],
         declared_cycles: ALWAYS_SUCCESS_SCRIPT_CYCLE  // ≤ max_tx_verify_cycles
     }
     relay child_i to the node  →  accepted into orphan pool (parent unknown)

4. After all 100 children are in the orphan pool, relay parent_tx.
   → parent_tx is accepted into the pending pool.
   → after_process calls process_orphan_tx(parent_tx).
   → process_orphan_tx finds all 100 children via find_by_previous.
   → For each child: _process_tx is called with command_rx = None,
     running full CKB-VM verification synchronously.
   → 100 consecutive script verifications execute inline, each acquiring
     and releasing the tx-pool write lock.

5. During step 4, submit legitimate transactions from a second peer and
   observe significantly increased latency or timeouts on tx-pool RPC calls,
   confirming lock contention and CPU exhaustion.

        while let Some(previous) = orphan_queue.pop_front() {
            let orphans = self.find_orphan_by_previous(&previous).await;
            for orphan in orphans.into_iter() {
                if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles {
                    debug!(
                        "process_orphan {} added to verify queue; find previous from {}",
                        orphan.tx.hash(),
                        tx.hash(),
                    );
                    let orphan_id = orphan.tx.proposal_short_id();
                    match self
                        .enqueue_verify_queue(
                            orphan.tx.clone(),
                            false,
                            Some((orphan.cycle, orphan.peer)),
                        )
                        .await
                    {
                        Ok(_) => {
                            self.remove_orphan_tx(&orphan_id).await;

                        Err(reject) => {
                            warn!(

                                tx.hash(),

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
