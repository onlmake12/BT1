### Title
Unauthenticated `from` PeerId in `ConnectionRequest` Allows `pending_delivered` Overwrite with Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The `respond_delivered` function in `ConnectionRequestProcess` inserts attacker-supplied `listen_addrs` into `pending_delivered` keyed by the message's `from` PeerId field. Because the `from` field is never cryptographically verified against the actual sender's session, any connected peer can inject or overwrite entries for arbitrary PeerIds. The `HOLE_PUNCHING_INTERVAL` guard only blocks re-insertion within 2 minutes; after that window, the entry is unconditionally overwritten. When a `ConnectionSync` subsequently arrives for that PeerId, the victim node attempts NAT traversal to the attacker-controlled addresses.

---

### Finding Description

**Entry point:** Any connected (unprivileged) peer sends a `ConnectionRequest` P2P message with `to` set to the victim's own PeerId and `from` set to any target PeerId (e.g., a legitimate peer A).

**Guard examined — `respond_delivered`, lines 161–167:**

```rust
if let Some((_, t)) = self.protocol.pending_delivered.get(&from_peer_id) {
    let now = unix_time_as_millis();
    if now - t < HOLE_PUNCHING_INTERVAL {
        return StatusCode::Ignore
            .with_context("a same message is already replied in a moment ago");
    }
}
``` [1](#0-0) 

This guard only rejects re-insertion when the existing entry is **less than** `HOLE_PUNCHING_INTERVAL` (2 minutes) old. After 2 minutes, the guard passes and the unconditional overwrite at lines 234–237 executes:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
``` [2](#0-1) 

There is **no check** that the sender's actual session PeerId matches `content.from`. The `from` field is parsed purely from message bytes: [3](#0-2) 

**Consumption of the poisoned entry — `ConnectionSyncProcess::execute`, lines 111–115:**

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
``` [4](#0-3) 

These addresses are passed directly to `try_nat_traversal`, which opens repeated outbound TCP connections to them for up to 30 seconds: [5](#0-4) 

---

### Impact Explanation

1. **Misdirected NAT traversal:** The victim node makes repeated outbound TCP connection attempts to attacker-controlled IP:port pairs instead of the legitimate peer's addresses.
2. **Unauthorized p2p session establishment:** If the TCP connection succeeds, `raw_session` is called, establishing a full p2p session with the attacker's node — bypassing normal connection admission logic.
3. **Hole-punch denial:** The legitimate hole-punch attempt for peer A fails because the `pending_delivered` entry has been replaced. [6](#0-5) 

---

### Likelihood Explanation

- The attacker only needs to be a **connected peer** — no special privilege, no PoW, no key material.
- The victim's PeerId is public (exchanged during peer identification).
- The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1 req/sec; waiting 2 minutes between the two `ConnectionRequest` messages trivially satisfies both the rate limiter and the `HOLE_PUNCHING_INTERVAL` guard. [7](#0-6) [8](#0-7) 

---

### Recommendation

1. **Bind `pending_delivered` to the session, not just the PeerId.** Key the map on `(session_id, from_peer_id)` or verify that the direct sender's session PeerId matches `content.from` before inserting.
2. **Do not allow overwrite of an in-flight entry.** If an entry already exists and has not yet been consumed by a `ConnectionSync`, reject the new `ConnectionRequest` regardless of age.
3. **Authenticate `from` against the session.** The `from` field should be verified to match the PeerId of the actual TCP session that delivered the message, preventing spoofing entirely.
