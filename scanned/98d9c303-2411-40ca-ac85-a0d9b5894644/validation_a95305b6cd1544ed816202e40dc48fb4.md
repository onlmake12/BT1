### Title
Incomplete Script Hash Type Validation in `ScriptHashTypeVerifier` — Lock Scripts Checked, Type Scripts Skipped - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` field of each output's **lock script** against `ENABLED_SCRIPT_HASH_TYPE`, but performs no equivalent check on each output's **type script**. Any transaction sender can submit a transaction whose outputs carry a type script with a hash type that is not permitted by consensus — including completely undefined byte values — and the non-contextual verifier will accept it.

---

### Finding Description

The structural analog to the `transferFrom()` blacklist bypass is exact:

| ERC-20 context | CKB context |
|---|---|
| `notBlacklisted(msg.sender)` checked | `output.lock().hash_type()` checked |
| `notBlacklisted(to)` checked | (no second lock-script field) |
| `notBlacklisted(from)` **missing** | `output.type_().hash_type()` **missing** |

`ScriptHashTypeVerifier::verify()` is defined in `verification/src/transaction_verifier.rs`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The loop body calls `output.lock().hash_type()` exclusively. There is no corresponding call to `output.type_().to_opt()` to retrieve and validate the type script's `hash_type` field. The `NonContextualTransactionVerifier` that wraps this verifier therefore passes transactions whose outputs contain type scripts with any arbitrary `hash_type` byte. [2](#0-1) 

The `ScriptHashTypeVerifier` is composed into `NonContextualTransactionVerifier`, which is the first gate applied to every transaction before contextual or script-execution checks. [3](#0-2) 

---

### Impact Explanation

Two concrete consequences follow from the missing type-script check:

**1. Permanently unspendable cells (invalid raw byte)**
A transaction output whose type script carries a `hash_type` byte that does not map to any `ScriptHashType` enum variant (e.g., `0x05`) passes `ScriptHashTypeVerifier` because the verifier never inspects it. The cell is committed to the chain. Any later attempt to spend it reaches `select_version()` inside `TxInfo`:

```rust
ScriptHashType::try_from(script.hash_type())
    .map_err(|err| ScriptError::InvalidScriptHashType(err.to_string()))?;
``` [4](#0-3) 

This returns an error unconditionally, making the cell permanently unspendable. Any CKB locked in that cell is burned.

**2. Pre-hardfork creation of cells gated by `ENABLED_SCRIPT_HASH_TYPE`**
`ENABLED_SCRIPT_HASH_TYPE` is a consensus constant that gates hash types such as `Data2` (VM version 2) behind the CKB2023 hardfork. The lock-script check enforces this gate for lock scripts. Because the type-script check is absent, an attacker can create outputs with `Data2` type scripts before the hardfork epoch. Those cells are unspendable until the hardfork activates, creating a class of outputs that violate the intended pre-hardfork invariant.

---

### Likelihood Explanation

The entry path requires no privilege: any RPC caller or P2P transaction submitter can craft a `Transaction` with an output whose type script carries an arbitrary `hash_type` byte. The `NonContextualTransactionVerifier` is the first check applied and it will pass the transaction. The attack is deterministic and requires no brute force, no key material, and no special network position.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's type script, mirroring the existing lock-script logic:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) =
        TryInto::<ScriptHashType>::try_into(type_script.hash_type())
    {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        })
        .into());
    }
}
```

This mirrors the existing lock-script branch and closes the gap.

---

### Proof of Concept

1. Construct a `Transaction` with one output whose lock script is valid and whose type script has `hash_type = 0x05` (undefined byte, not a valid `ScriptHashType`).
2. Submit via RPC (`send_transaction`) or P2P relay.
3. `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`.
4. The verifier loops over outputs, inspects only `output.lock().hash_type()` (valid), and returns `Ok(())`.
5. The transaction passes all non-contextual checks and is admitted to the tx-pool.
6. The transaction is included in a block; the output cell is committed on-chain.
7. Any subsequent transaction attempting to spend this cell reaches `TxInfo::select_version()`, which calls `ScriptHashType::try_from(0x05)`, receives an error, and returns `ScriptError::InvalidScriptHashType`.
8. The cell is permanently unspendable; all CKB capacity in it is burned. [5](#0-4) [6](#0-5)
