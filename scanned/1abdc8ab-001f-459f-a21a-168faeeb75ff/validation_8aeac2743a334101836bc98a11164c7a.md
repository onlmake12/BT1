The vulnerability claim is structurally accurate. Here is the precise analysis.

### Title
Unbounded Memory Aggregation in `get_tx_with_cells` Before LIMIT Is Applied — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

### Summary

When `group_by_transaction=true`, `get_transactions` builds a three-level SQL query where `GROUP_CONCAT`/`ARRAY_AGG` aggregates **all** matching rows across the entire indexed chain before the outer `LIMIT` is applied. An unprivileged RPC caller can trigger full-table aggregation with `limit=1`.

### Finding Description

The query is constructed in two stages inside `get_tx_with_cells`:

**Stage 1** — `build_tx_with_cell_union_sub_query` produces an unbounded `UNION ALL` subquery with no `LIMIT`: [1](#0-0) 

**Stage 2** — that subquery is wrapped in a middle subquery (also no `LIMIT`), then the outer query applies `GROUP BY` with `GROUP_CONCAT`/`ARRAY_AGG` and only then applies `LIMIT`:

```sql
SELECT tx_id, block_number, tx_index, tx_hash,
  GROUP_CONCAT(io_type || ',' || io_index, '","') AS io_pairs
FROM (
  SELECT ... FROM (UNION ALL subquery) AS res_union
  JOIN ckb_transaction ... JOIN block ...
) AS res
WHERE tx_id > ?
GROUP BY tx_id, block_number, tx_index, tx_hash
ORDER BY tx_id ASC
LIMIT 1          -- applied AFTER full GROUP BY aggregation
``` [2](#0-1) 

SQL semantics require the database engine to materialize and aggregate **every matching row** before it can emit the first `LIMIT`-bounded result. The `LIMIT` cannot be pushed past a `GROUP BY` with an aggregate function.

**Contrast with the non-grouped path** (`get_tx_with_cell`): there the `LIMIT` is applied directly to the flat `UNION ALL` subquery, so the DB stops after `limit` rows: [3](#0-2) 

**The `request_limit` guard is ineffective here**: it only rejects calls where the caller's `limit` parameter exceeds `self.request_limit`. It does not bound the number of rows the DB processes internally. [4](#0-3) 

**No default timeout**: `timeout_limit` defaults to `None` in the config, so there is no server-side query timeout to bound execution time or memory. [5](#0-4) 

### Impact Explanation

On a mainnet-scale node where the secp256k1 lock script (used by virtually all CKB addresses) is indexed, a single RPC call:

```json
{
  "method": "get_transactions",
  "params": [{
    "script": { "code_hash": "<secp256k1_code_hash>", "hash_type": "type", "args": "0x" },
    "script_type": "lock",
    "script_search_mode": "prefix",
    "group_by_transaction": true
  }, "asc", "0x1", null]
}
```

forces the SQLite/Postgres engine to aggregate millions of `(tx_id, io_type, io_index)` rows into `GROUP_CONCAT`/`ARRAY_AGG` in memory before returning 1 result. Peak RSS grows proportionally to total matching rows, not to `limit`. This can exhaust node memory and cause OOM or extreme latency, constituting a remote DoS against any node with the rich indexer enabled.

### Likelihood Explanation

- The rich indexer is an opt-in module but is documented and used in production deployments.
- The attack requires no authentication, no key, no PoW — just a valid JSON-RPC call.
- Empty `args` with `prefix` mode is the default search mode and is explicitly documented as supported.
- No rate limiting, no per-query memory cap, and no default timeout exist in the code path.

### Recommendation

Apply a `LIMIT` to the inner subquery before the `GROUP BY` layer. One approach: add a `LIMIT (limit * max_cells_per_tx_heuristic)` to the UNION ALL subquery, or restructure the query to first select the top-`limit` distinct `tx_id` values (with `LIMIT` applied before aggregation), then join back to collect `io_pairs` only for those `tx_id`s. Additionally, enforce a server-side query timeout via `timeout_limit`.

### Proof of Concept

1. Run a CKB node with rich indexer enabled, sync a large chain.
2. Record baseline RSS.
3. Call `get_transactions` with `group_by_transaction=true`, `limit=1`, `script.args="0x"` (prefix match on secp256k1 code hash).
4. Measure peak RSS during query execution.
5. Assert: peak RSS grows proportionally to total indexed rows matching the script, not to `limit=1`.
