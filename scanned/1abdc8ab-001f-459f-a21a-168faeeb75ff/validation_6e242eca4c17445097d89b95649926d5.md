### Title
Single Peer Can Monopolize the Shared VerifyQueue Budget, Blocking All Remote Transaction Submissions — (File: `tx-pool/src/component/verify_queue.rs`)

---

### Summary

The `VerifyQueue` enforces a single global byte-budget (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256 MB`) with no per-peer sub-limit. A single malicious connected peer can fill the entire queue with large transactions, causing every subsequent `submit_remote_tx` call from any peer to return `Reject::Full`. This is a direct structural analog to the GaslessPaymaster bug: a shared resource pool (ETH balance / verify-queue bytes) is exhaustible by one participant because there is no per-participant cap.

---

### Finding Description

`VerifyQueue::add_tx()` tracks only a single global counter `total_tx_size` and rejects new entries only when that counter reaches `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB). There is no per-peer byte budget, no per-peer transaction count, and no eviction of the largest contributor when the queue is full. [1](#0-0) 

The fullness check in `is_full()` is purely global: [2](#0-1) 

When `is_full` returns `true`, `add_tx` immediately returns `Reject::Full` for every caller, regardless of which peer submitted the blocking transactions: [3](#0-2) 

The `Entry` struct does record the originating `PeerIndex` in the `remote` field, and `remove_txs_by_peer` exists for cleanup on disconnect, but neither is used to enforce a per-peer admission budget: [4](#0-3) [5](#0-4) 

The relay path that feeds the queue is `TransactionsProcess::execute()` → `tx_pool.submit_remote_tx()` → `enqueue_verify_queue()` → `verify_queue.add_tx()`. The only guard before the queue is a per-peer message-level rate limiter (30 relay messages/second) in the `Relayer`: [6](#0-5) 

This rate limiter is per *message*, not per *byte*. Each `RelayTransactions` message may carry multiple transactions. The per-transaction size ceiling is `TRANSACTION_SIZE_LIMIT = 512 KB`: [7](#0-6) 

A peer sending 30 messages/second, each containing several 512 KB transactions, can saturate the 256 MB queue in seconds. Once saturated, `submit_remote_tx` returns `Reject::Full` for every peer, including honest ones.

---

### Impact Explanation

When the verify queue is full, every call to `submit_remote_tx` (and `notify_tx`) returns `Reject::Full`. The tx-pool service propagates this rejection back to the relayer, which marks the transaction as unknown and stops requesting it. Honest users' transactions are silently dropped at the queue boundary; they are never verified, never enter the pending pool, and are never mined. The node continues to operate (it does not crash), but its ability to accept new transactions from the network is completely suspended for as long as the attacker keeps the queue saturated. This matches the GaslessPaymaster impact: a shared resource is depleted, and honest participants are locked out.

---

### Likelihood Explanation

The attack requires only a single connected peer with no special privileges. The attacker needs valid-structure transactions (they must pass `pre_check` and the declared-cycles bound check in `TransactionsProcess`), but they do not need to be mined or even have valid scripts — verification happens *inside* the queue, not before admission. The rate limiter (30 messages/second) slows but does not prevent the attack; 256 MB / 512 KB = 512 transactions, achievable in well under a minute even at a conservative rate. The attacker can reconnect after a disconnect to re-fill the queue, since `remove_txs_by_peer` only cleans up on disconnect.

---

### Recommendation

Track per-peer byte usage inside `VerifyQueue`. Enforce a per-peer cap (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / expected_max_peers`) so that no single peer can consume more than its fair share of the queue. When a peer's quota is exhausted, reject only that peer's new submissions rather than blocking all peers. This mirrors the `MAX_SPONSORED_ETH` fix applied to GaslessPaymaster: a per-participant ceiling on the shared resource.

---

### Proof of Concept

1. Connect to a CKB mainnet/testnet node as a relay peer (SupportProtocols::RelayV3).
2. Construct ~512 syntactically valid transactions, each close to `TRANSACTION_SIZE_LIMIT` (512 KB), referencing non-existent inputs (so they fail verification inside the queue, but are admitted to it first).
3. Send them via `RelayTransactions` messages at ≤30 messages/second to stay under the rate limiter.
4. Observe via `get_tip_tx_pool_info` that `verify_queue_size` climbs to its maximum.
5. Attempt to submit a legitimate transaction from any other peer or via RPC `send_transaction`; it is rejected with `Reject::Full("verify_queue total_tx_size exceeded …")`. [8](#0-7) [9](#0-8)

    /// If the queue did not have this tx present, true is returned.
    /// If the queue did have this tx present, false is returned.
    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }

    pub fn execute(self) -> Status {
        let shared_state = self.relayer.shared().state();
        let txs: Vec<(TransactionView, Cycle)> = {
            // ignore the tx if it's already known or it has never been requested before
            let mut tx_filter = shared_state.tx_filter();
            tx_filter.remove_expired();
            let unknown_tx_hashes = shared_state.unknown_tx_hashes();

            self.message
                .transactions()
                .iter()
                .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
                .filter(|(tx, _)| {
                    !tx_filter.contains(&tx.hash())
                        && unknown_tx_hashes
                            .get_priority(&tx.hash())
                            .map(|priority| priority.requesting_peer() == Some(self.peer))
                            .unwrap_or_default()
                })
                .collect()
        };

        if txs.is_empty() {
            return Status::ok();
        }

        let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
        if txs
            .iter()
            .any(|(_, declared_cycles)| declared_cycles > &max_block_cycles)
        {
            self.nc.ban_peer(
                self.peer,
                DEFAULT_BAN_TIME,
                String::from("relay declared cycles greater than max_block_cycles"),
            );
            return Status::ok();
        }

        shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));

        let tx_pool = self.relayer.shared.shared().tx_pool_controller().clone();
        let peer = self.peer;
        self.relayer
            .shared
            .shared()
            .async_handle()
            .spawn(async move {
                for (tx, declared_cycles) in txs {
                    if let Err(e) = tx_pool
                        .submit_remote_tx(tx.clone(), declared_cycles, peer)
                        .await
                    {
                        error!("submit_tx error {}", e);
                    }
                }
            });

        Status::ok()
    }
