### Title
Miner Permanently Stalls After Reorg Revisiting a Previously-Submitted `parent_hash` Due to Uncleared `legacy_work` — (`miner/src/miner.rs`)

---

### Summary

`Miner::notify_new_work` unconditionally skips dispatching work whenever the incoming template's `parent_hash` is present in `legacy_work`. `legacy_work` is only ever cleared on a failed submission (`Works::FailSubmit`). There is no reorg-detection path that removes entries from `legacy_work`. A natural reorg that returns the canonical tip to a block whose hash was previously used as a `parent_hash` causes the miner to silently skip every subsequent template for that tip, leaving workers mining stale (already-submitted) work indefinitely.

---

### Finding Description

**Entry point — `submit_nonce`:**

After a nonce is found, `submit_nonce` unconditionally inserts the block's `parent_hash` into `legacy_work` before the RPC call returns: [1](#0-0) 

The entry is only removed if the RPC call itself fails: [2](#0-1) 

A successful submission leaves `parent_hash = P` permanently in the cache (until 32 later entries evict it via LRU).

**Guard in `notify_new_work`:** [3](#0-2) 

If `legacy_work.contains(&parent_hash)` is true the entire dispatch block is skipped — no `WorkerMessage::NewWork` is sent.

**Reorg scenario (concrete call sequence):**

1. Miner finds nonce for block B (height N, parent P). `submit_nonce` runs:
   - `legacy_work.put(P, ())` — line 170
   - RPC `submit_block` succeeds — no `FailSubmit`, so P stays in cache
   - `blocking_fetch_block_template()` — line 186 — fetches next template
   - `notify_workers(Start)` — line 187
2. A competing chain of equal or greater work is announced; the node reorgs, orphaning B and returning the canonical tip to P.
3. The node issues a new block template with `parent_hash = P` (and a fresh `work_id`).
4. `update_block_template` passes the `work_id` dedup check (new id ≠ old id) and sends `Works::New(work)`: [4](#0-3) 

5. `notify_new_work` is called; `legacy_work.contains(&P)` = **true** → entire dispatch block is skipped.
6. Workers receive `Start` (from step 1 line 187) but their `pow_work` is still the old, already-submitted template. They mine on a block the network has already seen and will reject.

**No recovery path exists in the miner loop.** The polling timer keeps calling `fetch_block_template`, but every returned template (parent = P) is silently dropped by the same guard. The LRU cache holds 32 entries: [5](#0-4) 

Because the miner is stalled it cannot submit new blocks, so no new entries are added and P is never evicted.

---

### Impact Explanation

The miner stalls: workers spin on stale, already-submitted work and never receive a valid new template. The miner fails to extend the canonical chain for as long as the canonical tip remains at P. Recovery requires either an external miner finding the next block (changing the tip's parent_hash) or a manual miner restart. This is a **miner liveness / availability** failure. It does not cause network-wide consensus deviation — other nodes and miners are unaffected — but the impacted miner loses all mining revenue and ceases to contribute hashpower to the canonical chain for the duration of the stall.

---

### Likelihood Explanation

Reorgs returning to a previously-mined parent are a normal, non-adversarial occurrence (network partition, near-simultaneous block discovery). No attacker capability is required; the bug triggers under ordinary network conditions whenever the miner's own block is orphaned and the chain rolls back to its parent. The higher the miner's hashrate, the more frequently it submits blocks and the more often it is exposed to this scenario.

---

### Recommendation

Remove the `parent_hash` from `legacy_work` whenever a new template arrives whose `parent_hash` is already in the cache **and** the template's `work_id` differs from the one that produced the cached entry (i.e., the node has issued genuinely new work for that parent). Alternatively, clear `legacy_work` entirely whenever the node signals a reorg (detectable via a tip-change in the block template's `parent_hash` relative to the previously accepted tip), or store `(parent_hash, work_id)` pairs instead of bare `parent_hash` values so that a reorg-issued template with the same parent but a new `work_id` is never suppressed.

---

### Proof of Concept

**State test (no network required):**

```rust
// 1. Construct a Work whose block has parent_hash = P.
// 2. Call submit_nonce with a valid nonce → legacy_work.put(P, ()).
// 3. Simulate a reorg: construct a new Work with the same parent_hash = P
//    but a different work_id (as the node would issue after a reorg).
// 4. Call notify_new_work with the new Work.
// 5. Assert: WorkerMessage::NewWork was sent to workers.
//    (Currently fails — no message is sent because legacy_work.contains(P) = true.)
```

The invariant under test: `notify_new_work` must dispatch work whenever the node issues a template with a `work_id` not previously seen, regardless of whether `parent_hash` appears in `legacy_work`. [3](#0-2) [6](#0-5)

    fn notify_new_work(&mut self, work: Work) {
        let parent_hash = work.block.header().into_view().parent_hash();
        if !self.legacy_work.contains(&parent_hash) {
            let pow_hash = work.block.header().calc_pow_hash();
            let (target, _) = compact_to_target(work.block.header().raw().compact_target().into());
            self.notify_workers(WorkerMessage::NewWork {
                pow_hash,
                work,
                target,
            });
        }
    }
