### Title
Peer Reputation Scoring System Permanently Disabled in Production — (`network/src/behaviour.rs`)

### Summary
The CKB network layer declares a full peer reputation/scoring infrastructure — including `PeerStore::report()`, `NetworkState::report_session()`, and the `CKBProtocolContext::report_peer()` trait method — but the sole mechanism for triggering it, the `Behaviour` enum, has **all production variants removed**. Every variant is gated behind `#[cfg(test)]`, and `Behaviour::score()` unconditionally returns `0` in non-test builds. The graduated scoring/banning system is therefore completely non-functional in production, giving a false sense of security analogous to the `PausableUpgradeable` issue.

### Finding Description

`network/src/behaviour.rs` defines the `Behaviour` enum that protocol handlers are supposed to use to report peer misbehavior:

```rust
// Currently this feature is disabled, maybe someday we will add it back or totally remove it.
pub enum Behaviour {
    #[cfg(test)]
    TestGood,
    #[cfg(test)]
    TestBad,
}

impl Behaviour {
    pub fn score(self) -> Score {
        #[cfg(test)]
        match self { ... }
        #[cfg(not(test))]
        0   // always zero in production
    }
}
``` [1](#0-0) 

The full scoring pipeline is wired up and functional:

1. `PeerStore::report()` applies `behaviour.score()` to a peer's score and bans them if it drops below `ban_score` (default: 40): [2](#0-1) 

2. `NetworkState::report_session()` calls `peer_store.report()` and disconnects the peer if banned: [3](#0-2) 

3. `CKBProtocolContext::report_peer()` is declared in the trait and implemented: [4](#0-3) 

However, a `grep` across all production protocol handlers (`sync/`, `util/light-client-protocol-server/`, `util/network-alert/`) shows **zero production call sites** for `report_peer`. The only call sites are in test helpers (`sync/src/relayer/tests/helper.rs`, `sync/src/tests/mod.rs`, `sync/src/tests/synchronizer/functions.rs`). Since no production `Behaviour` variant exists, no protocol handler can ever call `report_peer` with a meaningful value.

The `PeerScoreConfig` with its `default_score: 100`, `ban_score: 40`, and `ban_timeout_ms: 86400000` is entirely inert in production: [5](#0-4) 

### Impact Explanation

The graduated peer reputation system — designed to penalize peers for borderline misbehavior (e.g., slow responses, partially invalid data, protocol violations below the hard-ban threshold) and eventually ban them — is completely non-functional in production. Peers exhibiting such behavior can operate indefinitely without any score-based penalty. The `PeerScoreConfig` thresholds and `ban_timeout_ms` are dead configuration. Any protocol behavior that was intended to be handled via `report_peer` (graduated scoring) is silently ignored, leaving only the hard `ban_peer` path for clear-cut violations.

### Likelihood Explanation

This is a permanent, unconditional code-level disablement — not a configuration issue or race condition. Every production CKB node is affected. Any peer that would have been gradually penalized by the scoring system is immune to it. The comment "Currently this feature is disabled, maybe someday we will add it back or totally remove it" confirms this is an intentional but unresolved state, not a temporary oversight.

### Recommendation

Either:
1. **Remove** the dead infrastructure (`PeerStore::report()`, `NetworkState::report_session()`, `CKBProtocolContext::report_peer()`, `PeerScoreConfig`) to eliminate the false sense of security, or
2. **Re-enable** the system by adding production `Behaviour` variants with meaningful scores and calling `report_peer` from protocol handlers for appropriate misbehavior cases (e.g., slow block delivery, repeated invalid headers, etc.).

The current state — where the infrastructure exists and is documented as a security mechanism but is permanently disabled — is the direct analog of the `PausableUpgradeable` issue.

### Proof of Concept

1. In any production CKB build, `Behaviour` has zero variants (the enum is empty outside `#[cfg(test)]`).
2. `Behaviour::score()` returns `0` unconditionally in production (`#[cfg(not(test))] 0`).
3. `grep -r "report_peer" --include="*.rs"` in production protocol directories (`sync/src/relayer/mod.rs`, `sync/src/synchronizer/mod.rs`, `util/network-alert/src/alert_relayer.rs`) returns zero matches — the function is never called.
4. Therefore `PeerStore::report()` is never invoked in production, `peer_addr.score` is never decremented, and no peer is ever banned via the scoring path.
5. The `PeerScoreConfig` stored in every `PeerStore` instance is permanently inert. [6](#0-5) [7](#0-6)

    /// Report peer behaviours
    pub fn report(&mut self, addr: &Multiaddr, behaviour: Behaviour) -> ReportResult {
        if let Some(peer_addr) = self.addr_manager.get_mut(addr) {
            let score = peer_addr.score.saturating_add(behaviour.score());
            peer_addr.score = score;
            if score < self.score_config.ban_score {
                self.ban_addr(
                    addr,
                    self.score_config.ban_timeout_ms,
                    format!("report behaviour {behaviour:?}"),
                );
                return ReportResult::Banned;
            }
        }
        ReportResult::Ok
    }

    pub(crate) fn report_session(
        &self,
        p2p_control: &ServiceControl,
        session_id: SessionId,
        behaviour: Behaviour,
    ) {
        if let Some(addr) = self.with_peer_registry(|reg| {
            reg.get_peer(session_id)
                .filter(|peer| !peer.is_whitelist)
                .map(|peer| peer.connected_addr.clone())
        }) {
            trace!("Report {:?} because {:?}", addr, behaviour);
            let report_result = self.peer_store.lock().report(&addr, behaviour);
            if report_result.is_banned()
                && let Err(err) = disconnect_with_message(p2p_control, session_id, "banned")
            {
                debug!("Disconnect failed {:?}, error: {:?}", session_id, err);
            }
        } else {
            debug!(
                "Report {} failure: not found in peer registry or it is on the whitelist",
                session_id
            );
        }
    }
