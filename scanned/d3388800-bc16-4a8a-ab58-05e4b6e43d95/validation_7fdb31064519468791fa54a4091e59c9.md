### Title
Unauthenticated Route-Bypass Message Injection via Missing Route Membership Check in `ConnectionRequestDeliveredProcess::execute` — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

A relay node processing a `ConnectionRequestDelivered` message with an empty `route` field and an attacker-controlled `from` field will unconditionally forward the message to any connected peer matching `content.from`, without verifying that the relay was ever a legitimate member of the original hole-punching route. This violates the routing invariant and enables attacker-directed message injection to arbitrary connected peers via a single relay hop.

---

### Finding Description

The `execute()` method in `ConnectionRequestDeliveredProcess` dispatches on `content.route.last()`: [1](#0-0) 

When `route` is empty, `last()` returns `None`. The code then checks whether the local node's peer ID equals `content.from`. If not — which is always true when the attacker sets `from` to a victim peer ID — it calls `self.forward_delivered(&content.from)`: [2](#0-1) 

`forward_delivered` performs a `peer_registry` read-lock lookup for the attacker-supplied peer ID and, if that peer is connected, sends the full (attacker-crafted) message to it: [3](#0-2) 

There is no check that the relay node was ever inserted into the `route` of the original `ConnectionRequest`, no check that the message originated from a legitimate `to` peer, and no check that the `from` peer ever initiated a hole-punching session through this relay. The relay blindly trusts the attacker-supplied `from` field as the forwarding target.

The `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)`: [4](#0-3) 

All three fields are attacker-controlled, so the rate limiter is trivially bypassed by varying `content.to` across requests.

---

### Impact Explanation

**Relay-side (confirmed, unconditional):** Any connected peer can cause the relay to forward an attacker-crafted `ConnectionRequestDelivered` to any other peer connected to the relay, with fully attacker-controlled `from`, `to`, `listen_addrs`, and `sync_route` fields. The routing invariant — that forwarded messages must only traverse pre-established legitimate routes — is broken.

**Victim-side (conditional):** When the victim receives the injected message, it enters the "target peer" branch (since `from` = victim's own peer ID) and checks `inflight_requests.remove(&content.to)`: [5](#0-4) 

- If no matching inflight request exists: victim returns `Ignore` — no further action.
- If a matching inflight request exists (e.g., the victim recently sent a `ConnectionRequest` for `content.to`): the victim calls `try_nat_traversal(ttl, content.listen_addrs)` with fully attacker-controlled addresses: [6](#0-5) 

This causes the victim to make repeated outbound TCP connection attempts (up to 30 seconds, ~150 attempts) to attacker-controlled IP:port endpoints — a server-side request forgery (SSRF) / port-scan primitive: [7](#0-6) 

---

### Likelihood Explanation

The relay-side injection requires only that the attacker be connected to the relay (any unprivileged P2P peer) and know the peer ID of a victim also connected to the relay. Peer IDs are publicly observable on the P2P network. The SSRF escalation requires additionally that the victim has an active `inflight_requests` entry, which is populated whenever the victim's `notify` timer fires and it sends `ConnectionRequest` messages — a routine background operation: [8](#0-7) 

---

### Recommendation

In the `None` (empty route) branch of `execute()`, before calling `forward_delivered(&content.from)`, verify that the local node was a legitimate relay for this `(from, to)` pair — for example, by maintaining a set of `(from, to)` pairs for which the node has previously forwarded a `ConnectionRequest`, and rejecting `ConnectionRequestDelivered` messages that do not match any such entry. Alternatively, require that the sender of the `ConnectionRequestDelivered` message is the peer recorded as the next hop in the original route.

---

### Proof of Concept

1. Attacker peer A connects to relay node R.
2. Victim peer V is also connected to R (peer ID of V is observable from the P2P network).
3. A sends a `ConnectionRequestDelivered` to R with:
   - `route = []` (empty)
   - `from = V.peer_id` (victim's peer ID)
   - `to = <any valid PeerId>`
   - `listen_addrs = [attacker-controlled IP:port]`
   - `sync_route = []`
4. R evaluates: `route.last() == None`, `local_peer_id != V.peer_id` → calls `forward_delivered(V.peer_id)`.
5. R looks up V in `peer_registry`, finds it connected, and sends the crafted message to V.
6. V receives the message: `route.last() == None`, `local_peer_id == V.peer_id` → enters "target peer" branch.
7. If V has an active `inflight_requests` entry for `content.to`, V calls `try_nat_traversal` with attacker-controlled addresses, making repeated TCP probes to the attacker's chosen endpoint.

            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);

            base_retry_interval + jitter

            base_retry_interval.saturating_sub(jitter)
        };
