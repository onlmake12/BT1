### Title
Wrong Accounting of Ancestor `descendants_*` Fields in `PoolMap::remove_entry_and_descendants` — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all transaction links are stripped from the link map **before** `remove_entry` is called for each entry. Because `update_ancestors_index_key` resolves ancestors through those same links, it finds an empty ancestor set for every removed entry and never calls `sub_descendant_weight` on any remaining ancestor. The `descendants_size`, `descendants_cycles`, `descendants_fee`, and `descendants_count` fields of ancestor transactions that stay in the pool are permanently inflated — the exact wrong-accounting pattern from the reference report.

---

### Finding Description

`remove_entry_and_descendants` first collects the target and all its descendants, then strips every link in one pass, and only then calls `remove_entry` for each collected id:

```rust
// tx-pool/src/component/pool_map.rs  L252-L264
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← ALL links gone before any remove_entry call
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

Inside `remove_entry`, the first thing done is:

```rust
// tx-pool/src/component/pool_map.rs  L242-L243
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
```

`update_ancestors_index_key` resolves ancestors through the link map:

```rust
// tx-pool/src/component/pool_map.rs  L432-L445
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());   // ← empty: links already gone
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),  // never reached
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because `remove_entry_links` already deleted the link entry for every removed id, `calc_ancestors` returns an empty set for each call. No ancestor that **remains** in the pool ever has `sub_descendant_weight` called on it. Its `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` are never decremented — they stay at the values that included the now-removed chain.

The comment in `remove_entry_and_descendants` ("so that we won't `update_descendants_index_key` in `remove_entry`") shows the intent was only to suppress descendant-side updates (correct, since descendants are also being removed). The ancestor-side update (`update_ancestors_index_key`) was collaterally disabled as well — an unintentional accounting omission directly analogous to the reference bug.

---

### Impact Explanation

The `descendants_*` fields feed directly into `EvictKey`:

```rust
// tx-pool/src/component/entry.rs  L234-L247
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),
            timestamp: entry.timestamp,
            descendants_count: entry.descendants_count,   // ← inflated
        }
    }
}
```

`EvictKey` is the sole ordering used by `next_evict_entry`, which drives `limit_size` (pool-full eviction). An ancestor whose `descendants_*` are inflated appears to carry more high-fee descendants than it actually does, so its `fee_rate` in the eviction key is overstated and its `descendants_count` is overstated. Both factors make it rank lower in the eviction order (i.e., it is evicted later than it should be). Legitimate high-fee transactions with no descendants may be evicted in its place.

---

### Likelihood Explanation

`remove_entry_and_descendants` is called from multiple externally reachable paths:

- **`resolve_conflict`** — triggered whenever a committed block contains a transaction whose inputs overlap with pool entries. Any tx-pool submitter can craft a chain (parent tx0 → child tx1) and then have tx1's input double-spent in a block (or via RBF), causing `remove_entry_and_descendants(tx1)` while tx0 remains in the pool with inflated stats.
- **`check_and_record_ancestors`** — triggered when a new submission exceeds `max_ancestors_count` and cell-ref parents are evicted. An unprivileged submitter can engineer this by building a long chain and then submitting a transaction that references a cell consumed by a mid-chain entry.
- **`resolve_conflict_header_dep`** — triggered on fork/reorg when a header dep becomes invalid.

The RBF path is the most accessible: a tx-pool submitter submits tx0 (low fee) → tx1 (high fee), then replaces tx1 with a higher-fee conflicting transaction. tx1 is removed via `remove_entry_and_descendants`; tx0's `descendants_*` remain inflated for the lifetime of tx0 in the pool.

---

### Recommendation

Before stripping links in `remove_entry_and_descendants`, collect the set of **external ancestors** (ancestors of the root that are not themselves in `removed_ids`) and call `sub_descendant_weight` on each of them for every entry being removed. Alternatively, call `update_ancestors_index_key` for each entry **before** `remove_entry_links` is called for that entry, so the link map is still intact at the time of the ancestor walk.

### Proof of Concept

```
Pool state: tx0 (low fee, no parents) → tx1 (high fee) → tx2 (high fee)

After add_entry for all three:
  tx0.descendants_count = 3
  tx0.descendants_fee   = fee0 + fee1 + fee2
  tx0.descendants_size  = size0 + size1 + size2

Step 1: Attacker submits tx1_conflict (spends same input as tx1, higher fee → RBF accepted).
        remove_entry_and_descendants(tx1) is called.

Step 2: remove_entry_links called for tx1 and tx2 — all links gone.

Step 3: remove_entry(tx1):
        update_ancestors_index_key(tx1, Remove)
          → calc_ancestors(tx1) = {} (links gone)
          → tx0.sub_descendant_weight(tx1) NEVER CALLED

Step 4: remove_entry(tx2):
        update_ancestors_index_key(tx2, Remove)
          → calc_ancestors(tx2) = {} (links gone)
          → tx0.sub_descendant_weight(tx2) NEVER CALLED

Result:
  tx0.descendants_count = 3   (should be 1)
  tx0.descendants_fee   = fee0 + fee1 + fee2  (should be fee0)
  tx0.descendants_size  = size0 + size1 + size2 (should be size0)

tx0's EvictKey now reflects a falsely high descendants_feerate and descendants_count.
When the pool is full, tx0 is ranked as if it still has two high-fee children,
and legitimate transactions are evicted in its place.
```
