### Title
Wrong Row Variable in `script_exists_in_output` Returns `false` for Type-Script-Only Scripts on PostgreSQL During Rollback — (`File: util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`script_exists_in_output` in the Rich Indexer's rollback path uses `row_lock` instead of `row_type` in its final `match` block. On PostgreSQL this causes the function to always return `false` for scripts that are only referenced as a `type_script_id` (not a `lock_script_id`), making `rollback_block` incorrectly delete those scripts from the `script` table even though live outputs still reference them.

---

### Finding Description

`script_exists_in_output` is a guard called during `rollback_block` to decide whether a script row can be safely deleted. It runs two SQL `EXISTS` queries — one for `lock_script_id` and one for `type_script_id` — and returns `true` if either query finds a remaining reference.

The first query result is stored in `row_lock`. If the lock check returns `false`, execution falls through to the second query, whose result is stored in `row_type`. The final `match` block is supposed to read `row_type`, but it reads `row_lock` again:

```rust
// line 252 — BUG: row_lock used instead of row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),                          // returns lock result again
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // correct only on SQLite
}
```

On **PostgreSQL**, `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL `EXISTS` returns a `BOOLEAN`). Because execution only reaches line 252 when the lock check already returned `false`, `r` is always `false` here. The type-script check result in `row_type` is never read. The function returns `Ok(false)` regardless of whether the script is still referenced as a type script.

On **SQLite**, `try_get::<bool, _>` fails (SQLite `EXISTS` returns `BIGINT`), so the `Err` branch correctly reads `row_type`. [1](#0-0) 

The caller `rollback_block` uses the return value to build a deletion list:

```rust
if let Some(type_script_id) = type_script_id
    && !script_exists_in_output(type_script_id, tx).await?
{
    script_id_list_to_remove.push(type_script_id);
}
``` [2](#0-1) 

Because `script_exists_in_output` always returns `false` for type-script-only scripts on PostgreSQL, every such script is pushed into `script_id_list_to_remove` and deleted from the `script` table, even when other outputs in the database still reference it via `type_script_id`. [3](#0-2) 

---

### Impact Explanation

After a chain reorganization on a PostgreSQL-backed Rich Indexer deployment:

1. The `script` table loses rows for every script that is used **only** as a type script (not a lock script) in any output that survives the rollback.
2. The `output` table retains `type_script_id` foreign-key values that now point to deleted `script` rows — a dangling reference.
3. All subsequent Rich Indexer RPC queries that join `output` with `script` on `type_script_id` — including `get_cells`, `get_transactions`, and `get_cells_capacity` when filtered by type script — silently return incomplete or empty results for those scripts.
4. Clients (wallets, dApps, block explorers) relying on the Rich Indexer for type-script-based cell lookups receive wrong data, which can lead to incorrect balance displays, missed UTXOs, or failed transaction construction. [4](#0-3) 

---

### Likelihood Explanation

Chain reorganizations are a normal, routine event on any live CKB network. Any block relayer or miner who produces a competing chain of sufficient work triggers a reorg, which calls `indexer.rollback()` → `AsyncRichIndexer::rollback()` → `rollback_block()` → `script_exists_in_output()`. [5](#0-4) 

No special privilege is required. The bug fires on every rollback that involves an output whose script is used only as a type script. Type scripts are extremely common (DAO, UDT/SUDT, NFT, etc.), so virtually every real-world reorg on a PostgreSQL deployment will trigger corruption. The existing test suite uses SQLite exclusively and therefore does not catch this. [6](#0-5) 

---

### Recommendation

Change line 252 to read `row_type` instead of `row_lock`:

```rust
// Before (buggy):
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// After (fixed):
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Additionally, add a rollback test that runs against PostgreSQL and includes outputs whose scripts appear **only** as type scripts (not lock scripts), asserting that those scripts remain in the `script` table after rollback when other outputs still reference them. [7](#0-6) 

---

### Proof of Concept

1. Start a CKB node with the Rich Indexer configured to use PostgreSQL.
2. Index two blocks:
   - **Block A**: contains a transaction with one output whose `lock_script` is `S_lock` and `type_script` is `S_type`. Both scripts are inserted into the `script` table.
   - **Block B** (on a fork): contains a transaction with a different output that also uses `S_type` as its type script but uses a **different** lock script. After indexing Block B, `S_type` is referenced by two outputs.
3. Trigger a chain reorganization that rolls back Block B. `rollback_block` is called.
4. Inside `script_exists_in_output(S_type_id, tx)`:
   - Lock check: `SELECT EXISTS (... WHERE lock_script_id = S_type_id)` → `false` (S_type is not a lock script).
   - Type check: `SELECT EXISTS (... WHERE type_script_id = S_type_id)` → `true` (Block A's output still references it).
   - **Bug**: the final `match` reads `row_lock` again → returns `Ok(false)`.
5. `S_type_id` is added to `script_id_list_to_remove` and deleted from the `script` table.
6. Query `get_cells` filtered by `S_type` → returns 0 results, even though Block A's output is still live.

```sql
-- After the buggy rollback, on PostgreSQL:
SELECT COUNT(*) FROM script WHERE id = <S_type_id>;
-- Returns: 0  (incorrectly deleted)

SELECT COUNT(*) FROM output WHERE type_script_id = <S_type_id>;
-- Returns: 1  (dangling reference still present)
``` [8](#0-7)

    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
