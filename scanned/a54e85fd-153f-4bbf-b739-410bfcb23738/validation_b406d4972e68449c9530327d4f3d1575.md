### Title
Concurrent `HeaderMapKernel::get` Destructive-Read Race Causes Node Panic — (`shared/src/types/header_map/kernel_lru.rs`)

---

### Summary

`HeaderMapKernel::get` implements a "promote-on-read" pattern: on a memory miss it calls `backend.remove(hash)` to move the entry from sled back into memory. This remove is destructive and unguarded. Two threads racing on the same sled-resident key will both miss in memory, both call `backend.remove`, and the second caller receives `None`. The caller chain (`get_header_index_view` → `insert_valid_header`) unconditionally panics on `None` with `.expect("parent should be verified")`, crashing the node.

---

### Finding Description

`HeaderMapKernel::get` in `kernel_lru.rs`:

```rust
pub(crate) fn get(&self, hash: &Byte32) -> Option<HeaderIndexView> {
    if let Some(view) = self.memory.get_refresh(hash) {   // acquires+releases write lock
        return Some(view);
    }
    if self.backend.is_empty() {
        return None;
    }
    if let Some(view) = self.backend.remove(hash) {       // NO lock held here
        self.memory.insert(view.clone());
        Some(view)
    } else {
        None   // ← Thread B lands here
    }
}
```

`memory.get_refresh` acquires a `RwLock` write guard and releases it before returning. [1](#0-0) 

After that lock is released there is no synchronization primitive protecting the check-then-remove sequence against `backend`. Two threads can both observe a memory miss, both pass the `backend.is_empty()` guard, and both call `backend.remove(hash)`. Sled's `remove` is atomic per-key: the first caller receives `Some(view)`, the second receives `None`. [2](#0-1) [3](#0-2) 

Thread B returns `None` from `get()`. The caller `get_header_index_view` with `store_first=false` tries `header_map().get(hash)` first, then falls back to the chain store. Because header H has not yet been committed to the chain store (it is only in the header map), both lookups return `None`. [4](#0-3) 

`insert_valid_header` then panics:

```rust
let parent_header_index = self
    .get_header_index_view(&parent_hash, store_first)
    .expect("parent should be verified");   // ← PANIC
``` [5](#0-4) 

`insert_valid_header` is called from two concurrent P2P message-processing paths with no external serialization:
- `HeaderAcceptor::accept` in `headers_process.rs` (one call per peer per `SendHeaders` message)
- `CompactBlockProcess::execute` in `compact_block_process.rs` [6](#0-5) [7](#0-6) 

---

### Impact Explanation

An unprivileged remote peer can trigger this by sending two concurrent header chains that share a common parent header H. If H has been evicted from the in-memory LRU to the sled backend (which happens automatically every 5 seconds via `limit_memory`), and two sync threads race on `get(H)`, one thread panics. Because this is a `panic!` (not a recoverable error), the entire node process crashes. [8](#0-7) 

---

### Likelihood Explanation

During IBD, the node connects to multiple peers simultaneously and processes their `SendHeaders` messages concurrently. The memory limit is frequently hit (default threshold is 100,000 headers × `size_of::<HeaderIndexView>()`), causing regular evictions to sled. Any two peers that share a common ancestor header in the sled tier and whose child headers arrive within the same scheduling quantum can trigger the race. This is a normal operating condition, not an edge case. [9](#0-8) [10](#0-9) 

---

### Recommendation

Replace the destructive `backend.remove` in `get` with a non-destructive `backend.get`, so that concurrent reads do not race on ownership of the entry. The promotion to memory can remain (both threads will simply insert the same value, which is idempotent), but the backend entry should only be removed once it is confirmed to be in memory, under a coordinating lock or via a compare-and-delete. Alternatively, wrap the entire memory-miss → backend-lookup → memory-insert sequence in a per-key mutex or a single coarse lock.

---

### Proof of Concept

```rust
// Pseudocode concurrent test
let map = Arc::new(HeaderMapKernel::new(...));
// Insert H and evict it to sled backend
map.insert(H.clone());
map.limit_memory(); // forces eviction to sled

let map_a = Arc::clone(&map);
let map_b = Arc::clone(&map);

let t_a = thread::spawn(move || map_a.get(&H.hash()));
let t_b = thread::spawn(move || map_b.get(&H.hash()));

let r_a = t_a.join().unwrap();
let r_b = t_b.join().unwrap();

// One of r_a / r_b will be None — assert both are Some to demonstrate the bug
assert!(r_a.is_some(), "Thread A lost the race");
assert!(r_b.is_some(), "Thread B lost the race");  // ← this assertion fails
```

The failing assertion maps directly to the production panic at `.expect("parent should be verified")` in `insert_valid_header`. [11](#0-10)

            })
        }
