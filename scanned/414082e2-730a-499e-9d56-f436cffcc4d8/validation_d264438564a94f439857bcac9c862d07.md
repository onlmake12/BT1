### Title
Hardcoded `TRANSACTION_SIZE_LIMIT` Does Not Derive from Consensus `max_block_bytes`, Breaking Integrations on Protocol Upgrades — (`util/types/src/core/tx_pool.rs`)

---

### Summary

`TRANSACTION_SIZE_LIMIT` is a compile-time constant hardcoded to `512 * 1_000` bytes in the tx-pool. It is used as the sole gate for rejecting oversized transactions at pool admission. It is **not** derived from the consensus `max_block_bytes` parameter. If the consensus block-size limit changes (e.g., via a hard fork), the tx-pool limit silently diverges, either permanently rejecting valid transactions or permanently accepting transactions that can never be mined.

---

### Finding Description

In `util/types/src/core/tx_pool.rs`, the tx-pool admission limit is defined as a standalone constant:

```rust
pub const TRANSACTION_SIZE_LIMIT: u64 = 512 * 1_000;
``` [1](#0-0) 

The comment explicitly acknowledges the relationship to the block limit:

> "The ckb consensus does not limit the size of a single transaction, but if the size of the transaction is close to the limit of the block, it may cause the transaction to fail to be packed"

The consensus block-size limit is defined separately as:

```rust
pub const MAX_BLOCK_BYTES: u64 = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT; // 597 * 1000 = 597,000
``` [2](#0-1) 

`TRANSACTION_SIZE_LIMIT` (512,000) is **not** computed from `MAX_BLOCK_BYTES` (597,000). They are independent constants. The tx-pool admission check in `non_contextual_verify` uses only the hardcoded constant:

```rust
let tx_size = tx.data().serialized_size_in_block() as u64;
if tx_size > TRANSACTION_SIZE_LIMIT {
    return Err(Reject::ExceededTransactionSizeLimit(tx_size, TRANSACTION_SIZE_LIMIT));
}
``` [3](#0-2) 

This hardcoded value is also what is reported to RPC callers via `tx_pool_info` as `tx_size_limit`:

```rust
tx_size_limit: TRANSACTION_SIZE_LIMIT,
``` [4](#0-3) 

---

### Impact Explanation

Two failure modes arise when `MAX_BLOCK_BYTES` changes relative to the hardcoded `TRANSACTION_SIZE_LIMIT`:

**Case 1 — `MAX_BLOCK_BYTES` increases (e.g., hard fork raises block size to 1 MB):**  
Transactions between 512 KB and the new block limit are valid by consensus but are permanently rejected by every node's tx-pool. Integrations (wallets, dApps, smart-contract scripts) that construct large transactions relying on the published `tx_size_limit` from `tx_pool_info` will silently fail. The node reports `tx_size_limit = 0x7d000` (512 KB) via RPC, so callers have no way to know the real consensus limit.

**Case 2 — `MAX_BLOCK_BYTES` decreases (e.g., hard fork reduces block size):**  
Transactions up to 512 KB are accepted into the tx-pool but can never be committed to a block. An unprivileged transaction sender can fill the pool with permanently-unmineble transactions, wasting pool capacity and degrading node performance for all peers.

---

### Likelihood Explanation

CKB's consensus parameters are defined in the chain spec and can be changed via hard forks. The `max_block_bytes` value is already derived from `TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT`, both of which are named constants that have been adjusted in the past. Any future protocol upgrade that touches block size will silently break the tx-pool admission logic because `TRANSACTION_SIZE_LIMIT` is a separate, unlinked constant. Any RPC caller or transaction-submitting peer is the unprivileged entry point.

---

### Recommendation

Derive `TRANSACTION_SIZE_LIMIT` from the consensus `max_block_bytes` at runtime rather than hardcoding it. The `non_contextual_verify` function already receives a `Consensus` reference; the limit should be computed as a fraction of `consensus.max_block_bytes()` (e.g., `consensus.max_block_bytes() * 85 / 100`) so it automatically tracks protocol upgrades. The `TxPoolInfo::tx_size_limit` field reported via RPC should reflect the same dynamically-computed value.

---

### Proof of Concept

1. Current state: `TRANSACTION_SIZE_LIMIT = 512_000`, `MAX_BLOCK_BYTES = 597_000`.
2. Suppose a hard fork increases `TWO_IN_TWO_OUT_COUNT` from 1,000 to 2,000, making `MAX_BLOCK_BYTES = 1,194,000`.
3. `TRANSACTION_SIZE_LIMIT` remains `512_000` — no code change required, no recompilation needed for the constant to be wrong.
4. A transaction sender submits a 600 KB transaction via `send_transaction` RPC. It is valid by consensus (fits in a block) but is rejected by every node with `PoolRejectedTransactionBySizeLimit: Transaction size 600000 exceeded maximum limit 512000`.
5. Conversely, if `MAX_BLOCK_BYTES` shrinks to 400,000, a sender submits a 450 KB transaction. It is accepted into the pool (`450_000 < 512_000`) but can never be mined (`450_000 > 400_000`), permanently occupying pool space. [1](#0-0) [5](#0-4) [6](#0-5)

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
