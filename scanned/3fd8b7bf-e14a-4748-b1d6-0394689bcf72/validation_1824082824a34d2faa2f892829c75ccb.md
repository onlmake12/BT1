Audit Report

## Title
`check_purge` Eviction Deadlock via Crafted /16-Diverse Discovery Addresses — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
The `check_purge` function in `PeerStore` contains two sequential eviction paths, both of which can be simultaneously rendered inert by an attacker who fills all 16384 peer store slots with addresses from distinct /16 subnets (≤1 per group). When this layout is in place, Path 1 finds no non-connectable addresses and Path 2 finds no group exceeding 4 members, causing `check_purge` to return `Err(PeerStoreError::EvictionFailed)`. Every subsequent `add_addr` call from the discovery or identify protocols silently fails, permanently disabling peer discovery for the victim node until the attacker's addresses naturally expire.

## Finding Description

**Root cause — `check_purge` (lines 327–404):**

`check_purge` is called at the top of `add_addr` whenever `addr_manager.count() >= ADDR_COUNT_LIMIT` (16384). It runs two sequential strategies:

**Path 1 (lines 341–355):** Collects addresses where `!addr.is_connectable(now_ms)`. `add_addr` creates every address via `AddrInfo::new(addr, 0, score, flags.bits())`, setting `last_connected_at_ms = 0` and `attempts_count = 0`. With those values, `is_connectable` returns `true` (the `attempts_count >= ADDR_MAX_RETRIES` branch requires `attempts_count >= 3`). All attacker-inserted addresses are connectable, so `candidate_peers` is empty and nothing is evicted.

**Path 2 (lines 358–401):** Groups addresses by `/16` subnet (`Group::IP4([bits[0], bits[1]])`), sorts groups by descending size, then:
- `take(len / 2)` — considers only the top half of groups.
- `if addrs.len() > 4` — only groups with **more than 4** members yield eviction candidates.

If the attacker fills all 16384 slots with one address per distinct /16 subnet:
- `len = 16384`, `take(8192)` iterates 8192 groups.
- Every group has exactly 1 address → `1 > 4` is `false` → `None` for every group.
- `candidate_peers` is empty → `return Err(PeerStoreError::EvictionFailed)`.

**Delivery path:** `DiscoveryAddressManager::add_new_addrs` (lines 347–363) calls `peer_store.add_addr` in a loop and only logs failures at `debug!` level, silently discarding every `Err(EvictionFailed)`. The discovery protocol accepts up to `MAX_ADDR_TO_SEND = 1000` nodes per message; filling 16384 slots requires approximately 6 messages from a single connected peer.

## Impact Explanation

Once the peer store is saturated under the described layout, the victim node cannot add any newly discovered peer addresses. Peer discovery is effectively disabled for the duration of the attack. If applied broadly across many nodes, this disrupts the CKB network's ability to propagate peer information and maintain healthy connectivity, matching the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*. The cost to the attacker is minimal — a single P2P connection and ~6 discovery messages.

## Likelihood Explanation

- Requires only one inbound or outbound P2P connection to the victim; no privileged access, no PoW, no key material.
- The discovery protocol imposes no per-session or per-/16 rate limit on advertised subnets.
- The attack is self-sustaining: the attacker re-advertises fresh addresses as old ones accumulate failed connection attempts and become non-connectable (after `ADDR_MAX_RETRIES = 3` failures with `last_connected_at_ms == 0`).
- Locally reproducible with a unit test (see PoC below).

## Recommendation

1. **Remove or widen `take(len / 2)`** in Path 2 so all groups are considered, not just the top half.
2. **Lower or remove the `> 4` threshold**: always evict at least one address from the largest group regardless of its size, ensuring at least one slot is freed.
3. **Add a per-session or per-/16 rate limit** in `add_new_addrs` to bound how many distinct subnets a single peer can contribute.
4. **Promote `EvictionFailed` to `warn!`** so operators can detect saturation in production logs.

## Proof of Concept

```rust
let mut peer_store = PeerStore::default();
// Fill with 16384 addresses, one per distinct /16 subnet
for i in 0u32..16384 {
    let a = (i >> 8) as u8;
    let b = (i & 0xff) as u8;
    let addr: Multiaddr = format!(
        "/ip4/{}.{}.0.1/tcp/8114/p2p/{}",
        a, b,
        PeerId::random().to_base58()
    ).parse().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Any new legitimate address now fails
let new_addr: Multiaddr = format!(
    "/ip4/200.200.200.1/tcp/8114/p2p/{}",
    PeerId::random().to_base58()
).parse().unwrap();
assert!(peer_store.add_addr(new_addr, Flags::COMPATIBILITY).is_err());
// Returns Err(PeerStore(EvictionFailed))
```

The test exercises the exact code path: `add_addr` → `check_purge` → Path 1 empty (all connectable) → Path 2 empty (`1 > 4` false for every group) → `Err(EvictionFailed)`.
