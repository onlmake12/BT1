Audit Report

## Title
Unauthenticated Inbound Peer Can Poison Peer Store via Identify Listen Addresses — (`network/src/protocols/identify/mod.rs`)

## Summary
`process_listens` in the Identify protocol unconditionally calls `add_remote_listen_addrs` for both inbound and outbound sessions, with no ownership verification for inbound peers. An attacker cycling ~1639 cheap inbound connections, each injecting 10 globally-routable but unowned addresses, can saturate the peer store (`ADDR_COUNT_LIMIT = 16384`). Once full, the eviction mechanism (`check_purge`) cannot make progress against attacker-crafted addresses from distinct `/16` subnets, permanently blocking legitimate address insertion and hijacking the feeler mechanism.

## Finding Description

**Root cause — `process_listens` has no session-type guard.**

`received_identify` explicitly guards `add_outbound_addr` behind `is_outbound()` with a comment acknowledging that inbound addresses are unverified: [1](#0-0) 

However, `process_listens` — called unconditionally for all sessions — has no such guard: [2](#0-1) 

`add_remote_listen_addrs` then calls `peer_store.add_addr()` for every address in the message: [3](#0-2) 

`add_addr` inserts each address with `last_connected_at_ms = 0` (never verified): [4](#0-3) 

**Eviction bypass.** When the store reaches `ADDR_COUNT_LIMIT = 16384`: [5](#0-4) 

`check_purge` Step 1 evicts non-connectable addresses. Attacker addresses have `attempts_count = 0` and `last_connected_at_ms = 0`. Examining `is_connectable`: [6](#0-5) 

- `tried_in_last_minute` → false (`last_tried_at_ms = 0`)
- `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES(3)` → false (`0 >= 3` is false)
- `attempts_count >= ADDR_MAX_FAILURES(10)` → false (`0 >= 10` is false)

Result: `is_connectable` returns `true` for all attacker addresses → Step 1 evicts nothing.

`check_purge` Step 2 groups by network segment and only evicts from groups with `> 4` peers: [7](#0-6) 

If the attacker uses one address per distinct `/16` subnet, every group has exactly 1 entry → Step 2 evicts nothing → `Err(PeerStoreError::EvictionFailed)` is returned, and no new address can ever be inserted.

**Feeler hijack.** `fetch_addrs_to_feeler` selects addresses where `!peer_addr.connected(|t| t > addr_expired_ms)`. For attacker addresses, `last_connected_at_ms = 0`, so `0 > addr_expired_ms` is false → `!false = true` → all attacker addresses are returned by feeler: [8](#0-7) 

**Threshold.** `MAX_ADDRS = 10` is the disconnect threshold: [9](#0-8) 

Sending exactly 10 addresses per connection avoids disconnection while maximizing injection. `ceil(16384 / 10) = 1639` connections suffice.

## Impact Explanation

Once the peer store is saturated with attacker addresses from distinct `/16` subnets:
- All subsequent `add_addr` calls fail silently with `EvictionFailed`
- `fetch_addrs_to_feeler` returns exclusively attacker-controlled dead-end addresses
- The victim node cannot discover or validate new honest peers
- The effect is persistent until node restart and manual peer store purge

Applied at scale against many nodes simultaneously, this degrades the CKB network's peer discovery fabric with very low cost, fitting the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

- **No privilege required**: any peer that can open a TCP connection to the victim's P2P port qualifies
- **Low cost**: ~1639 TCP connections, each sending a ~200-byte identify message; achievable from a single machine
- **No PoW, no stake, no key**: only network access required
- **Cycling**: the attacker does not need 1639 simultaneous connections; connect → send identify → disconnect → repeat suffices within the default `max_inbound ≈ 117` limit
- **Persistent**: once the store is full with diverse-subnet attacker addresses, the eviction mechanism cannot recover without a restart

## Recommendation

1. **Reject inbound listen-address injection**: apply the same `is_outbound()` guard used for `add_outbound_addr` to `process_listens` / `add_remote_listen_addrs` for inbound sessions. The existing comment at L416-419 already documents the rationale.
2. **Source-IP diversity cap**: limit the number of addresses insertable per source `/16` network group regardless of session type.
3. **Feeler confirmation before persistence**: only persist an address after a successful feeler connection verifies reachability.
4. **Eviction hardening**: ensure `check_purge` can always make progress by unconditionally evicting the oldest-inserted or lowest-scored address when both strategies fail.

## Proof of Concept

```rust
// Pseudocode for the attack
for i in 0..1640 {
    let conn = open_inbound_tcp_connection(victim_p2p_addr);
    let listen_addrs: Vec<Multiaddr> = (0..10)
        .map(|j| format!("/ip4/{}.{}.{}.{}/tcp/8115/p2p/{}",
            i/256, i%256, j, 1, random_peer_id()).parse().unwrap())
        .collect();
    // Each address is from a distinct globally-routable /16 subnet
    send_identify_message(conn, listen_addrs);
    close(conn);
}
// After loop: peer_store.addr_manager.count() == 16384
// check_purge returns EvictionFailed for all subsequent add_addr calls
// fetch_addrs_to_feeler returns only attacker addresses
```

A unit test constructing 1640 mock inbound sessions × 10 addresses each (from distinct `/16` subnets), then asserting `addr_manager.count() == ADDR_COUNT_LIMIT` and that no legitimate address survives `fetch_addrs_to_feeler`, would pass against the current production code. The `is_connectable` logic in `types.rs` L89-105 and the `check_purge` grouping logic in `peer_store_impl.rs` L357-401 confirm both eviction steps fail for this address pattern.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
