Audit Report

## Title
Stale-Snapshot Overwrite in `add_entry` Inflates `total_tx_size`/`total_tx_cycles`, Enabling Tx-Pool DoS — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` snapshots projected pool totals into local variables before calling `check_and_record_ancestors`, which may evict existing entries and correctly decrement `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`. After evictions complete, `add_entry` unconditionally overwrites those fields with the pre-eviction snapshot, re-inflating the totals by the aggregate size and cycles of every evicted transaction. An unprivileged RPC caller can repeatedly trigger this path to permanently inflate pool accounting until `limit_size` evicts all legitimate transactions and the pool rejects every new submission with `Reject::Full`.

## Finding Description

**Root cause — `add_entry`, lines 200–221:**

```rust
// Line 210-211: snapshot taken BEFORE evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: may evict entries; each eviction calls update_stat_for_remove_tx,
// which correctly decrements self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218-219: OVERWRITES correctly-decremented fields with stale snapshot
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 710–729) is a pure read that returns `(self.total_tx_size + tx_size, self.total_tx_cycles + cycles)` without modifying `self`. It captures the pool state at the moment of the call, before any evictions.

**Eviction path — `check_and_record_ancestors`, lines 603–625:**

When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, the function evicts `cell_ref_parents` one by one via `remove_entry_and_descendants` (line 618). That call chain is: `remove_entry_and_descendants` → `remove_entry` (line 263) → `update_stat_for_remove_tx` (line 247), which correctly subtracts each evicted entry's `size` and `cycles` from `self.total_tx_size`/`self.total_tx_cycles` (lines 738–740).

**Accounting error table:**

| Step | `self.total_tx_size` |
|---|---|
| After `updated_stat_for_add_tx` snapshot | `S + N` (stored in local var) |
| After evictions inside `check_and_record_ancestors` | `S − E` (correct) |
| After overwrite at line 218 | `S + N` (stale, **E is lost**) |

Correct post-call value: `S − E + N`. Actual value: `S + N`. Inflation per call: `E` (total size of evicted entries).

**Existing guards are insufficient:** `update_stat_for_remove_tx` (lines 733–758) has an underflow guard that falls back to `recompute_total_stat`, but this guard only fires on subtraction underflow. The overwrite at lines 218–219 is an unconditional assignment that bypasses all guards entirely.

## Impact Explanation

`limit_size` in `pool.rs` (line 298) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` (default 180 MB), evicting lowest-fee-rate transactions until the condition is false. With `total_tx_size` artificially inflated:

- Legitimate pending transactions are evicted from the pool even when actual memory usage is well within the 180 MB limit.
- New transactions submitted by honest users are rejected with `Reject::Full`.
- Repeated exploitation accumulates inflation until `total_tx_size >= max_tx_pool_size`, at which point `limit_size` evicts all pending transactions and the pool permanently rejects every subsequent submission.
- The only recovery is a node restart.

This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**. A single attacker with RPC access can render the mempool of a targeted node non-functional, preventing transaction propagation and mining from that node.

## Likelihood Explanation

The trigger requires constructing a transaction graph where a new transaction has more than `max_ancestors_count` (default 25) ancestors and at least one ancestor is a `cell_ref_parent`. Referencing an unconfirmed transaction's output as a cell dep is a valid, protocol-permitted CKB pattern. No privileged access, no key material, and no majority hashpower is required. The attack is executable by any user with RPC access (`send_transaction`), including remote callers if the RPC port is exposed. The required chain depth of 25 is easily achievable. Each attack round requires submitting approximately 26 transactions; the inflation per round equals the size of the evicted dep-provider transaction(s). The attack is repeatable with fresh transactions each round.

## Recommendation

Move the stat update **after** all mutations (evictions, insertion) have completed, so it reflects the true post-eviction pool state. Replace the pre-eviction snapshot pattern with a direct increment applied after `check_and_record_ancestors` returns:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Validate capacity (overflow check) without storing the snapshot
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Recompute AFTER evictions so evicted sizes are not re-added
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size  = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, remove the local snapshot entirely and call a direct `self.total_tx_size += entry.size` / `self.total_tx_cycles += entry.cycles` (with overflow checks) after all mutations, mirroring the paired decrement already used in `update_stat_for_remove_tx`.

## Proof of Concept

**Setup (default config: `max_ancestors_count = 25`, `max_tx_pool_size = 180_000_000`):**

1. Submit a "dep-provider" transaction `D` to the pool. Record `size(D)`.
2. Submit a chain of 24 ancestor transactions `A1 → A2 → … → A24` (each spending the previous output).
3. Submit a new transaction `T` that:
   - Spends `A24`'s output (giving it 25 ancestors: `A1…A24` + itself = 25, but `T` itself makes `ancestors_count = 25`; to exceed the limit, ensure `A1` also spends an output that makes `D` a `cell_ref_parent` of `T`, giving `ancestors_count = 26`).
   - References `D`'s output as a cell dep, making `D` a member of `cell_ref_parents`.

**Execution trace:**

1. `add_entry(T)` is called.
2. `updated_stat_for_add_tx` snapshots `total_tx_size = S + size(T)` into a local variable.
3. `check_and_record_ancestors`: `ancestors_count = 26 > 25`; `cell_ref_parents = {D}`; `26 − 1 = 25 ≤ 25` → eviction branch fires.
4. `remove_entry_and_descendants(D)` → `update_stat_for_remove_tx(size(D), cycles(D))` → `self.total_tx_size = S − size(D)`.
5. Back in `add_entry`: `self.total_tx_size = total_tx_size` → `self.total_tx_size = S + size(T)`.
6. **Correct value:** `S − size(D) + size(T)`. **Actual value:** `S + size(T)`. **Inflation: `size(D)`.**

Repeat steps 1–3 with fresh `D` and `T` transactions each round. Each round inflates `total_tx_size` by `size(D)`. After enough rounds, `total_tx_size ≥ 180_000_000`, `limit_size` evicts all pending transactions, and the pool rejects every subsequent submission with `Reject::Full`.

A unit test can be written directly against `PoolMap::add_entry` by:
- Constructing a mock pool with `max_ancestors_count = 3` (to reduce setup).
- Inserting dep-provider and ancestor transactions.
- Calling `add_entry` with a transaction that triggers the eviction branch.
- Asserting `pool_map.total_tx_size == expected_correct_value` after the call — this assertion will fail, confirming the bug.
