Audit Report

## Title
Missing Duplicate-Index Deduplication and Response Byte-Size Limit in `GetBlockTransactionsProcess::execute` Enables Multi-Peer Memory Exhaustion — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32,767 transaction indexes per `GetBlockTransactions` message but performs no deduplication and enforces no byte-size ceiling on the serialized response. An attacker with many peer connections can send requests containing 32,767 identical valid indexes, causing the node to allocate a flat serialized buffer of `32767 × tx_size` bytes per request inside `.build()` before any send attempt. Concurrent requests across 128 peers at 30 req/sec can hold enormous in-memory allocations simultaneously, exhausting node RAM and crashing the process.

## Finding Description
**Entrypoint:** Any unprivileged peer on the `RelayV3` P2P protocol can send a `GetBlockTransactions` message, dispatched via `Relayer::try_process` → `GetBlockTransactionsProcess::execute`.

**Rate limiter:** Keyed by `(PeerIndex, message.item_id())` at 30 req/sec per peer. With `MAX_RELAY_PEERS = 128`, the aggregate ceiling is 3,840 req/sec reaching `execute()`.

**Root cause — no deduplication:** The `filter_map` over `indexes()` collects one cloned `TransactionView` per index entry, including duplicates:
```rust
let transactions = self.message.indexes().iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
```
32,767 identical indexes → 32,767 entries in `transactions`.

**Root cause — no byte-size limit:** All collected entries are immediately serialized into a single flat molecule byte buffer via `packed::BlockTransactions::new_builder()...build()`. This allocation of `32767 × tx_size` bytes occurs unconditionally before `async_send_message_to` is called. Even if the subsequent send fails due to the 4 MB `RelayV3` frame limit, the allocation has already been made and is held until the future completes.

**Contrast with sibling handlers:** Both `GetTransactionsProcess::execute` and `GetBlockProposalProcess::execute` (a) reject duplicate entries via `HashSet` with `StatusCode::RequestDuplicate`, and (b) enforce `MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MB) on the response, chunking it if needed. `GetBlockTransactionsProcess` has neither protection.

## Impact Explanation
**Concrete impact: crash of a CKB node (High, 10001–15000 points).**

- Incoming message size: 32,767 × 4 bytes ≈ 128 KB — well within the 4 MB frame limit, so the message is accepted without issue.
- Outgoing allocation per request: `32767 × tx_size`. For a 10 KB transaction this is ~320 MB; for a 50 KB transaction ~1.6 GB.
- At 3,840 req/sec with even modest per-request allocation lifetimes, concurrent in-flight allocations easily exhaust available RAM, triggering an OOM kill or Rust allocator panic and crashing the node.
- Block data (block hash, transaction index) is fully public and trivially discoverable from the chain.

## Likelihood Explanation
The attack requires only 128 simultaneous TCP connections (within the default `max_peers` range of 125–500) and knowledge of any stored block containing a non-trivial transaction. No privileged access, no proof-of-work, and no key material is needed. The attack is fully scriptable, locally testable, and repeatable indefinitely.

## Recommendation
Apply both fixes already present in sibling handlers:

1. **Deduplicate indexes before processing:** Collect `indexes()` into a `HashSet<u32>`; if `set.len() != indexes.len()`, return `StatusCode::RequestDuplicate`.
2. **Enforce `MAX_RELAY_TXS_BYTES_PER_BATCH` on the response:** Accumulate `tx.data().total_size()` while building the transaction list; truncate or split the response when the byte limit is reached, mirroring the chunking pattern in `GetBlockProposalProcess::execute` and `GetTransactionsProcess::execute`.

## Proof of Concept
```
1. Identify any stored block hash H containing a transaction T at index 0
   (any non-trivial block on mainnet/testnet suffices).
2. Open 128 TCP connections to the target node on the RelayV3 protocol.
3. From each connection, send 30 GetBlockTransactions messages/sec:
       block_hash = H
       indexes    = [0, 0, 0, ..., 0]   # 32767 copies of index 0
4. Each execute() call enters packed::BlockTransactions::new_builder()...build()
   and allocates ~32767 × sizeof(T) bytes before attempting async_send_message_to.
5. Monitor target node RSS; expect rapid memory growth and OOM crash within seconds
   for any block containing a transaction of a few KB or larger.
```
