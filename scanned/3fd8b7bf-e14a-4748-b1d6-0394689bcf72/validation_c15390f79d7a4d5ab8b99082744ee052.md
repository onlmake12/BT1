Audit Report

## Title
Missing Zero-Check on `deposit_ar` Before Division in DAO Withdrawal Calculation — (`File: util/dao/src/lib.rs`)

## Summary
`calculate_maximum_withdraw` in `util/dao/src/lib.rs` extracts `deposit_ar` from the deposit block header's DAO field and immediately uses it as a divisor without checking for zero. In Rust, integer division by zero causes an unconditional panic, crashing the node process. The public RPC endpoint `calculate_dao_maximum_withdraw` calls this function directly, making the crash reachable via RPC if a block with `ar = 0` is present in the node's store.

## Finding Description
At lines 152–154 of `util/dao/src/lib.rs`, `deposit_ar` (a raw `u64` decoded from the 32-byte DAO field via `extract_dao_data`) is used as a divisor with no zero-guard:

```rust
let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
// ...
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);   // panics if deposit_ar == 0
```

The same unchecked pattern appears in `secondary_block_reward` (lines 202–203, dividing by `target_parent_c`) and `dao_field_with_current_epoch` (lines 242–243 and 256–257, dividing by `parent_c`).

The `DaoHeaderVerifier` (in `verification/contextual/src/contextual_block_verifier.rs`, lines 670–672) enforces that each block's DAO field matches the computed value — but this check is gated behind `Switch::DISABLE_DAOHEADER`:

```rust
if !self.switch.disable_daoheader() {
    DaoHeaderVerifier::new(&self.context, resolved, &parent, &block.header()).verify()?;
}
```

When `DISABLE_DAOHEADER` is set, blocks with arbitrary DAO fields (including `ar = 0`) are accepted into the store. The genesis function `genesis_dao_data_with_satoshi_gift` hardcodes `ar = DEFAULT_GENESIS_ACCUMULATE_RATE = 10_000_000_000_000_000` and only guards `c != 0`, not `ar != 0`. No equivalent guard exists for `deposit_ar` at the call site.

The RPC entry point `calculate_dao_maximum_withdraw` in `rpc/src/module/experiment.rs` (lines 259–267) calls `calculate_maximum_withdraw` directly with caller-supplied `out_point` and header-hash arguments, with no additional validation of the DAO field contents.

## Impact Explanation
If a block with `ar = 0` is present in the node's store and an RPC caller invokes `calculate_dao_maximum_withdraw` referencing a transaction from that block, the division at line 154 triggers a Rust panic (`attempt to divide by zero`), crashing the node process. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
On a fully-validated mainnet/testnet chain, `ar` starts at `10^16` and only increases monotonically, so `deposit_ar = 0` cannot occur under normal operation. Triggering the panic requires two non-standard preconditions: (1) the node must be running with `Switch::DISABLE_DAOHEADER` enabled, allowing blocks with arbitrary DAO fields to be accepted; and (2) a block with a crafted DAO field encoding `ar = 0` must be present in the store. `DISABLE_DAOHEADER` is a real, functional flag defined in `verification/traits/src/lib.rs` and checked in `contextual_block_verifier.rs`, but it is not used in any production code path in this codebase. Its use is limited to testing or custom tooling scenarios. Once those preconditions are met, any unprivileged RPC caller can trigger the crash with a single request.

## Recommendation
Add an explicit zero-check for `deposit_ar` before the division in `calculate_maximum_withdraw`, mirroring the existing `ZeroC` guard in `genesis_dao_data_with_satoshi_gift`:

```rust
if deposit_ar == 0 {
    return Err(DaoError::InvalidHeader);
}
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
```

Apply equivalent defensive checks to `target_parent_c` in `secondary_block_reward` (line 203) and `parent_c` in `dao_field_with_current_epoch` (lines 243 and 257).

## Proof of Concept
1. Start a CKB dev node with `Switch::DISABLE_DAOHEADER` enabled (e.g., by passing the flag to `blocking_process_block_with_switch`).
2. Craft and submit a block whose DAO field encodes `ar = 0` (using `pack_dao_data(0, c, s, u)` with any non-zero `c`). The node accepts it because `DaoHeaderVerifier` is skipped.
3. Identify a transaction output in that block.
4. Call the RPC:
   ```json
   {
     "method": "calculate_dao_maximum_withdraw",
     "params": [
       { "tx_hash": "<tx_in_block_with_ar_0>", "index": "0x0" },
       "<any_later_block_hash>"
     ]
   }
   ```
5. `calculate_maximum_withdraw` extracts `deposit_ar = 0` from the deposit header's DAO field and executes `/ u128::from(0)`, triggering a Rust panic and crashing the node process.
