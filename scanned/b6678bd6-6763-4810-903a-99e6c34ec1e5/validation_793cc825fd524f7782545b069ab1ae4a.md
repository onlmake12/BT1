### Title
Weak Cleaning Condition in `need_clean` Checks Only Arbitrary First Child's Epoch, Enabling Orphan Pool Memory Exhaustion — (File: `chain/src/utils/orphan_block_pool.rs`)

---

### Summary
The `need_clean` function in `InnerPool` decides whether to evict an entire orphan subtree by inspecting only the **first** child block returned by a non-deterministic `HashMap` iterator. Because a parent can have multiple children at different epoch numbers, an unprivileged peer can craft two orphan blocks sharing the same parent — one with a recent epoch, one with an expired epoch — so that the recent-epoch child is iterated first, permanently blocking eviction of the expired child. With no enforced size cap on `OrphanBlockPool`, repeated submissions cause unbounded memory growth.

---

### Finding Description

`InnerPool::need_clean` at line 113 of `chain/src/utils/orphan_block_pool.rs`:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← only first child
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

`self.blocks` maps each parent hash to a `HashMap<packed::Byte32, LonelyBlockHash>` of its children. `map.iter().next()` returns an **arbitrary** child because `HashMap` iteration order is unspecified in Rust. The cleaning decision for the entire subtree — including all siblings — is therefore based on whichever child happens to be iterated first.

`clean_expired_blocks` (line 99) iterates over all leaders and calls `need_clean` for each:

```rust
pub fn clean_expired_blocks(&mut self, tip_epoch: EpochNumber) -> Vec<LonelyBlockHash> {
    let mut result = vec![];
    for hash in self.leaders.clone().iter() {
        if self.need_clean(hash, tip_epoch) {
            let descendants = self.remove_blocks_by_parent(hash);
            result.extend(descendants);
        }
    }
    result
}
```

`OrphanBlockPool::insert` (line 141) has no size guard:

```rust
pub fn insert(&self, lonely_block: LonelyBlockHash) {
    self.inner.write().insert(lonely_block);
}
```

There is also a secondary off-by-one: the expiry condition uses strict `<` (`epoch_number + EXPIRED_EPOCH < tip_epoch`) rather than `<=`, so blocks at exactly `tip_epoch − EXPIRED_EPOCH` are retained for one extra epoch beyond the documented intent.

---

### Impact Explanation

**Memory exhaustion (resource accounting).** An attacker who can send orphan blocks can grow the orphan pool without bound:

1. Send block **A** (parent = P, epoch = `tip_epoch − 1`, i.e., recent) — not yet expired.
2. Send block **B** (parent = P, epoch = `tip_epoch − 10`, i.e., expired) — should be cleaned.

Both are inserted under the same parent key P. When `clean_expired_blocks` runs, `need_clean(P, tip_epoch)` samples one child. If A is sampled, `(tip_epoch − 1) + 6 < tip_epoch` is false → neither A nor B is evicted. The attacker repeats this for many distinct parents. Because there is no pool size limit, the orphan pool grows until the node runs out of memory and crashes or becomes unresponsive.

A secondary consequence is **premature eviction**: if B is sampled first, the valid recent block A is also removed, forcing the node to re-request it and stalling sync.

---

### Likelihood Explanation

Any peer connected to the CKB P2P network can relay block messages. Orphan blocks (blocks whose parent is unknown) are accepted and stored without full validation. The attacker only needs to craft block headers with arbitrary epoch fields — no PoW solution is required for the orphan pool insertion path, and epoch numbers in headers are not validated at insertion time. The attack is cheap, repeatable, and requires no privileged access.

---

### Recommendation

1. **Fix the weak condition**: replace the single-child sample with a check over **all** children, e.g., use the minimum epoch number among all children so that a subtree is only retained if at least one child is still within the expiry window.

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

2. **Fix the off-by-one**: change `<` to `<=` to match the documented expiry semantics.

3. **Enforce a pool size cap**: add a maximum entry count to `OrphanBlockPool::insert` and evict the oldest entries when the cap is reached, independent of the epoch-based cleanup path.

4. **Extract helpers**: split `clean_expired_blocks` into clearly named helper functions (one for expiry check, one for removal) to reduce tangled logic, mirroring the refactoring recommended in the original report.
