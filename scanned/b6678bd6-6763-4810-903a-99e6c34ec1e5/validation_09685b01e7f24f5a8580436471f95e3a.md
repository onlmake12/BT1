### Title
Unbounded CPU Exhaustion via `test_tx_pool_accept` RPC — Full `verify_rtx` Execution with No Rate Limiting, No Fee, and No Deduplication — (`tx-pool/src/process.rs`)

### Summary

`test_tx_pool_accept` triggers a full CKB-VM script execution (up to `max_block_cycles` = 70,000,000 cycles) per call, runs it in a freshly spawned tokio task with no interruption channel, and imposes zero cost on the caller because the transaction is never inserted into the pool. There is no rate limiting, no per-caller quota, no deduplication guard (the same tx can be resubmitted immediately after each call), and no pool-full check. An attacker with any live cell on-chain can flood this endpoint from multiple concurrent HTTP connections and saturate the node's tokio thread pool, starving legitimate `send_transaction` processing.

### Finding Description

**Entrypoint:** `rpc/src/module/pool.rs` → `PoolRpcImpl::test_tx_pool_accept`

The RPC handler sends a `Message::TestAcceptTx` to the tx-pool service channel: [1](#0-0) 

The service loop immediately spawns a new tokio task for every received message: [2](#0-1) 

The `TestAcceptTx` message handler calls `service.test_accept_tx(tx)`: [3](#0-2) 

`test_accept_tx` checks for duplicates only against the `verify_queue` and `orphan_pool` — but since `_test_accept_tx` never inserts the tx into either, the same transaction can be resubmitted immediately after each call: [4](#0-3) 

`_test_accept_tx` then runs a full `verify_rtx` with `max_cycles = consensus.max_block_cycles()` (70,000,000) and `command_rx = None` — meaning the VM execution cannot be interrupted or chunked: [5](#0-4) 

Contrast this with the `VerifyMgr` workers, which pass a live `command_rx` to allow chunked/interruptible execution: [6](#0-5) 

**No mitigating guards exist on this path:**
- No rate limiting or per-IP quota anywhere in the RPC stack
- No pool-full check (unlike `resumeble_process_tx` which checks `verify_queue` size)
- No fee payment (tx is never inserted)
- The verify cache (`txs_verify_cache`) is keyed by `witness_hash`; the attacker trivially bypasses it by varying witness bytes across calls

The channel capacity is 512 messages: [7](#0-6) 

But because each message is immediately dequeued and spawned as a concurrent tokio task, the channel drains instantly and does not self-limit the number of concurrent `verify_rtx` executions.

### Impact Explanation

Each concurrent `test_tx_pool_accept` call with a near-max-cycle script occupies a tokio worker thread for the duration of the VM execution. With N concurrent connections each submitting a 70M-cycle transaction, N tokio threads are pinned. The `send_transaction` path (`Message::SubmitLocalTx`) competes for the same tokio thread pool. Legitimate transactions experience unbounded latency entering the pool. On a node with the default worker count (3/4 of CPU cores), a small number of concurrent attacker connections can fully saturate available compute. [8](#0-7) 

### Likelihood Explanation

- The Pool RPC module is enabled by default and many node operators expose it for dApp use.
- The attacker only needs knowledge of any live cell on-chain (publicly visible) to craft a valid transaction that passes `pre_check`.
- The attack requires zero on-chain cost: the transaction is never broadcast or inserted.
- The same transaction (with varied witnesses) can be reused indefinitely.
- No special privileges, keys, or network position are required.

### Recommendation

1. **Add a cycle-based cost gate**: Before calling `_test_accept_tx`, check a per-IP or global rate limit based on declared/estimated cycles.
2. **Enforce a concurrency cap**: Limit the number of simultaneously executing `TestAcceptTx` tasks (e.g., via a semaphore).
3. **Reuse the verify queue**: Route `test_accept_tx` through the existing `VerifyQueue` with a size cap, so it competes fairly with real submissions and is subject to the `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB) limit.
4. **Require a minimum fee rate**: Even for test calls, enforce `min_fee_rate` as a spam deterrent (the fee check already exists in `pre_check` but does not prevent zero-cost repeated calls since the tx is never inserted).

            "params": [tx, "passthrough"], "id": 1
        }, timeout=120)

# Launch N concurrent threads
for i in range(32):

    pub(crate) async fn test_accept_tx(&self, tx: TransactionView) -> Result<Completed, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, None).await?;

        if self.verify_queue_contains(&tx).await {
            return Err(Reject::Duplicated(tx.hash()));
        }

        if self.orphan_contains(&tx).await {
            debug!("reject tx {} already in orphan pool", tx.hash());
            return Err(Reject::Duplicated(tx.hash()));
        }
        self._test_accept_tx(tx.clone()).await
    }
