### Title
Unbounded NAT Traversal Task Spawning via Non-Consuming `pending_delivered.get()` in `ConnectionSyncProcess::execute` тАФ (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary

`ConnectionSyncProcess::execute` reads `pending_delivered` with `.get()` instead of `.remove()`. Because the entry is never consumed, every `ConnectionSync` message that passes the rate limiter for a given `(from, to)` pair spawns a new `select_ok` task with up to `ADDRS_COUNT_LIMIT=24` TCP connection futures. An attacker with a single P2P connection can pre-seed `pending_delivered` via a `ConnectionRequest`, then drain file descriptors by sending one `ConnectionSync` per second indefinitely.

### Finding Description

**Step 1 тАФ Seeding `pending_delivered`**

`ConnectionRequestProcess::respond_delivered` inserts into `pending_delivered` when the victim node is the `to` target of a `ConnectionRequest`: [1](#0-0) 

The attacker sends a `ConnectionRequest` with `from=attacker_peer_id`, `to=victim_peer_id`, and up to 24 unreachable TCP addresses. The victim inserts `(attacker_peer_id, ([24 addrs], now))`.

A re-insertion guard exists (`HOLE_PUNCHING_INTERVAL = 2 min`), but the entry only needs to be created once: [2](#0-1) 

**Step 2 тАФ Non-consuming read in `ConnectionSyncProcess::execute`**

When the victim receives a `ConnectionSync` with `to=victim_peer_id`, it looks up `pending_delivered` with `.get()`, not `.remove()`: [3](#0-2) 

The entry is never removed. Every subsequent `ConnectionSync` for the same `from` finds the same entry and spawns a new `runtime::spawn` task: [4](#0-3) 

**Step 3 тАФ Rate limiter does not prevent accumulation**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1 req/sec: [5](#0-4) 

This limits the attacker to 1 task/sec per `(from, to)` pair. However:
- The attacker can use N distinct `from` PeerIDs (each seeded with one `ConnectionRequest`).
- The outer per-session rate limiter allows 30 messages/sec: [6](#0-5) 

So with 30 distinct `from` PeerIDs, the attacker spawns 30 tasks/sec, each holding up to 24 TCP FDs.

**Step 4 тАФ `pending_delivered` is only cleaned up every 5 minutes** [7](#0-6) 

Entries live for up to `TIMEOUT = 5 min`. Over that window, 30 tasks/sec ├Ч 300 sec = **9,000 tasks**, each with up to 24 pending TCP connections = **216,000 concurrent FD-holding connection attempts**.

### Impact Explanation

Each spawned task holds up to 24 TCP sockets in `SYN_SENT` state until the OS connect timeout (~75 s on Linux). A typical CKB node process has a default FD limit of 1024тАУ65536. Sustained task spawning exhausts FDs, causing the node to fail to accept new P2P connections, fail RPC socket creation, and potentially panic on internal socket operations тАФ a full local DoS.

### Likelihood Explanation

- Attacker only needs one live P2P connection to the victim.
- Victim's peer ID is public (advertised in the P2P network).
- No PoW, no key material, no privileged role required.
- The attack is fully automatable and sustainable for the lifetime of the P2P session.

### Recommendation

Replace `.get()` with `.remove()` in `ConnectionSyncProcess::execute`:

```rust
// connection_sync.rs, line 113-114
let listens_info = self
    .protocol
    .pending_delivered
    .remove(&content.from)   // was: .get(&content.from)
    .map(|info| info.0);
```

This enforces the invariant that each `pending_delivered` entry triggers at most one NAT traversal attempt, matching the behavior of `inflight_requests.remove()` used in `ConnectionRequestDeliveredProcess`: [8](#0-7) 

### Proof of Concept

1. Connect to victim node via P2P.
2. Send `ConnectionRequest` with `from=A`, `to=victim`, `listen_addrs=[24 ├Ч 1.2.3.4:9999]` (unreachable).
3. Victim inserts `pending_delivered[A] = ([24 addrs], now)`.
4. Send `ConnectionSync` with `from=A`, `to=victim` once per second.
5. Each message passes the 1/sec rate limiter and spawns a new `select_ok` task with 24 TCP futures.
6. After 60 seconds: 60 tasks ├Ч 24 sockets = 1,440 FDs consumed.
7. Scale with 30 distinct `from` PeerIDs to reach 43,200 FDs/min.
8. Observe node failure to accept new connections / RPC socket errors.
