### Title
Stale `header_map` Entry After Block Verification Failure — (`chain/src/verify.rs`)

---

### Summary

In `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks`, the success path removes a block from **both** `block_status_map` and `header_map`, but the error path only updates `block_status_map` (marking the block `BLOCK_INVALID`) and never calls `remove_header_view`. This leaves a permanent stale entry in `header_map` for every block that passes header validation but fails full contextual verification. An unprivileged block relayer can exploit this to exhaust the bounded `header_map`, displacing valid headers and disrupting sync.

---

### Finding Description

`consume_unverified_blocks` in `chain/src/verify.rs` handles two outcomes asymmetrically:

**Success path** (lines 141–151): both parallel structures are cleaned up:
```rust
self.shared.remove_block_status(&block_hash);   // cleans block_status_map
self.shared.remove_header_view(&block_hash);    // cleans header_map
```

**Error path** (lines 153–190): only `block_status_map` is updated; `header_map` is never touched:
```rust
self.shared.insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
// remove_header_view(&block_hash) is MISSING
``` [1](#0-0) 

The `header_map` entry was inserted earlier by `insert_valid_header` (called during header sync): [2](#0-1) 

The correct pattern — removing both structures together — is already established in `OrphanBroker::clean_expired_orphans`:
```rust
self.shared.remove_header_view(&expired_orphan.hash());
self.shared.remove_block_status(&expired_orphan.hash());
``` [3](#0-2) 

A secondary variant exists in the internal-DB-error sub-path (lines 178–181): `remove_block_status` is called (removing the `BLOCK_INVALID` marker), but `remove_header_view` is still not called. After this, `get_block_status` falls through to the `header_map` check and returns `HEADER_VALID` for a block that actually failed verification:

```rust
pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
    match self.block_status_map().get(block_hash) {
        Some(status_ref) => *status_ref.value(),
        None => {
            if self.header_map().contains_key(block_hash) {
                BlockStatus::HEADER_VALID   // ← stale entry causes wrong status
            } ...
        }
    }
}
``` [4](#0-3) 

---

### Impact Explanation

1. **Memory exhaustion / header_map saturation**: `header_map` is bounded by `sync_config.header_map.memory_limit`. Stale entries for invalid blocks accumulate without bound, consuming this budget and evicting valid headers needed for sync. Loss of valid headers forces redundant `GetHeaders` round-trips and can stall IBD or normal sync.

2. **Incorrect block status (DB-error sub-path)**: After an internal DB error, `block_status_map` is cleared but `header_map` retains the entry. `get_block_status` then returns `HEADER_VALID` for a block that failed verification, potentially causing the sync state machine to re-request and re-process the same block in a loop.

3. **Incorrect skip-pointer construction**: `get_header_index_view` reads `header_map` directly without consulting `block_status_map`. A stale invalid-block entry can be returned as a valid ancestor when building skip pointers for future headers, corrupting chain-navigation data structures. [5](#0-4) 

---

### Likelihood Explanation

A remote, unprivileged block relayer is the attacker. The entry path is:

1. Peer sends a compact block or full block over the relay/sync protocol.
2. The block passes non-contextual (header-level) verification — PoW, structure, timestamp — and is accepted into `header_map` via `insert_valid_header`.
3. The block is queued as an `UnverifiedBlock` and processed by `consume_unverified_blocks`.
4. Full contextual verification fails (e.g., invalid transaction, capacity overflow, script failure).
5. The error path fires, leaving the stale `header_map` entry.

Crafting a block with valid PoW but invalid contextual content is straightforward on testnet or with low-difficulty chains. On mainnet the PoW cost is higher, but the attacker only needs to produce enough invalid blocks to saturate the bounded `header_map`. Because the stale entries are never evicted, the attack is cumulative across restarts (the `header_map` is re-populated from the DB on startup for stored headers, but in-memory stale entries from the current session accumulate until the node is restarted).

---

### Recommendation

In the error branch of `consume_unverified_blocks`, add the missing `remove_header_view` call immediately after updating `block_status_map`, mirroring the success path and the existing pattern in `clean_expired_orphans`:

```rust
Err(err) => {
    // ... existing error handling ...
    if !is_internal_db_error(err) {
        self.shared.insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
    } else {
        self.shared.remove_block_status(&block_hash);
    }
    // ADD THIS:
    self.shared.remove_header_view(&block_hash);
}
``` [6](#0-5) 

---

### Proof of Concept

1. Connect to a CKB node as a peer via the relay/sync protocol.
2. Craft a block `B` whose header satisfies all non-contextual checks (valid PoW, correct parent hash, valid timestamp) but whose body fails contextual verification (e.g., a transaction that double-spends a cell, or a cellbase with wrong capacity).
3. Send `B` as a `SendBlock` or compact block message. The node calls `insert_valid_header`, inserting `B`'s hash into `header_map`.
4. `consume_unverified_blocks` runs, contextual verification fails, `BLOCK_INVALID` is written to `block_status_map`, but `header_map` retains the entry.
5. Repeat with distinct block hashes (varying nonce/timestamp). Each iteration consumes one slot in the bounded `header_map`.
6. Once `header_map` is saturated, valid headers from honest peers are evicted, causing the victim node's sync to stall or regress.
