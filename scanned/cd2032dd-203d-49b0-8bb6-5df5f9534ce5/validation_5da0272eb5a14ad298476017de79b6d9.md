### Title
Unbounded `runtime::spawn` of 30-second NAT traversal futures via crafted `ConnectionSync` messages causes Tokio runtime exhaustion — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
An unprivileged remote peer can exhaust the Tokio runtime by: (1) poisoning `pending_delivered` with many entries (each holding up to 24 listen addresses) via spoofed `ConnectionRequest` messages, then (2) sending `ConnectionSync` messages at the rate-limiter ceiling (30/second), each triggering an unbounded `runtime::spawn` of a `select_ok` task over 24 concurrent 30-second `try_nat_traversal` futures. After 30 seconds of sustained attack, ~900 concurrent tasks holding ~21,600 live futures accumulate, exhausting file descriptors, Tokio task memory, and scheduler capacity, rendering the RPC API unresponsive.

### Finding Description

**Phase 1 — Poison `pending_delivered`**

The attacker sends `ConnectionRequest` messages with `to = local_peer_id`, `from = fake_peer_id_N` (N distinct values), and `listen_addrs` containing 24 crafted TCP/IPv4 addresses. The `content.from` field is taken directly from the message without verifying it matches the actual session peer ID. [9](#0-8) 

`respond_delivered` filters addresses to TCP/IPv4 (which the attacker satisfies trivially), sends a `ConnectionRequestDelivered` back to the attacker's session, then inserts `(fake_peer_id_N → ([24 addrs], now))` into `pending_delivered`. [10](#0-9) 

The 2-minute `HOLE_PUNCHING_INTERVAL` guard only blocks re-insertion for the **same** `from_peer_id`. With N distinct fake peer IDs, the attacker inserts N entries with no cross-key throttle. The top-level rate limiter allows 30 `ConnectionRequest` per second, so 30 entries are inserted per second. `pending_delivered` has no size limit and entries live for 5 minutes. [11](#0-10) 

**Phase 2 — Trigger unbounded spawns via `ConnectionSync`**

The attacker sends `ConnectionSync` with `from = fake_peer_id_N`, `to = local_peer_id`, `route = []`. The local node is the `to` target, so `execute` reaches the NAT traversal branch: [12](#0-11) 

For each message, `pending_delivered.get(&content.from)` returns `Some(24 addrs)`, 24 `try_nat_traversal` futures are created, and `runtime::spawn` is called unconditionally with no concurrency cap: [6](#0-5) 

The forward rate limiter key is `(from, to, msg_item_id)`. Using 30 distinct `from` values bypasses it entirely. The only effective cap is the top-level limiter at 30/second per session. Each spawned task runs `select_ok` over 24 futures, each looping for up to 30 seconds: [13](#0-12) 

**Accumulation math:**
- Spawn rate: 30 tasks/second (top-level limiter ceiling)
- Task lifetime: 30 seconds (hard-coded `timeout_duration`)
- Steady-state concurrent tasks: 30 × 30 = **900 tasks**
- Concurrent futures: 900 × 24 = **21,600 futures**
- Each future opens a TCP socket every ~400ms → up to ~54,000 socket open/close operations per second

### Impact Explanation
The Tokio runtime is shared between the P2P layer and the RPC server. Accumulation of 21,600 live futures each repeatedly creating TCP sockets exhausts the process file descriptor limit, saturates the Tokio task queue, and starves the RPC handler tasks. The RPC API becomes unresponsive or crashes. This matches the stated scope: **local RPC API crash via Tokio runtime exhaustion**.

### Likelihood Explanation
The attack requires only a single P2P connection (no special privileges, no PoW, no key material). The `from` field in `ConnectionRequest` is not authenticated against the session. The attack is fully scriptable: connect, send 30 `ConnectionRequest`/second for a few seconds to build up `pending_delivered`, then send 30 `ConnectionSync`/second indefinitely. No Sybil attack or majority hashpower is needed.

### Recommendation
1. **Cap `pending_delivered` size**: enforce a hard maximum (e.g., 256 entries) and evict oldest entries on overflow.
2. **Bound concurrent NAT traversal tasks per `pending_delivered` entry**: track an in-flight counter per `from` peer ID; skip `runtime::spawn` if a task is already running for that entry.
3. **Validate `content.from` against the actual session peer ID** in `ConnectionRequest` processing to prevent spoofed-`from` poisoning.
4. **Reduce `try_nat_traversal` timeout** or add a global cap on total concurrent NAT traversal tasks.

### Proof of Concept
```
1. Connect to target node (single TCP connection).
2. For i in 0..300:
     Send ConnectionRequest {
       from: random_peer_id_i,
       to: local_peer_id,
       listen_addrs: [1.2.3.4:1000, 1.2.3.4:1001, ..., 1.2.3.4:1023]  // 24 addrs
     }

3. Loop forever (30/second):
     For each fake_peer_id_i in pending_delivered:
       Send ConnectionSync {
         from: fake_peer_id_i,
         to: local_peer_id,
         route: []
       }
   // Each message spawns select_ok(24 × try_nat_traversal(30s))
4. After 30 seconds: ~900 concurrent tasks, ~21,600 live futures.
5. Assert: RPC endpoint stops responding.
```
