### Title
`total_tx_size` and `total_tx_cycles` Inflated When Entries Are Evicted During `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

### Summary

In `PoolMap::add_entry()`, the updated `total_tx_size` and `total_tx_cycles` values are pre-computed before potential in-place evictions occur inside `check_and_record_ancestors()`. When entries are evicted, `update_stat_for_remove_tx()` decrements `self.total_tx_size` and `self.total_tx_cycles` in place, but those decrements are then silently overwritten by the stale pre-computed values assigned at the end of `add_entry()`. The evicted entries' sizes and cycles are never subtracted from the final totals, causing both counters to be permanently inflated.

### Finding Description

In `add_entry()`: [1](#0-0) 

The sequence is:

1. **Line 210–211**: `updated_stat_for_add_tx()` reads the current `self.total_tx_size` and `self.total_tx_cycles` and returns `(old_total_size + entry.size, old_total_cycles + entry.cycles)` as local variables `total_tx_size` / `total_tx_cycles`. [2](#0-1) 

2. **Line 213**: `check_and_record_ancestors()` may call `remove_entry_and_descendants()` → `remove_entry()` → `update_stat_for_remove_tx()`, which **mutates `self.total_tx_size` and `self.total_tx_cycles` in place** to subtract the evicted entries. [3](#0-2) 

3. **Lines 218–219**: `self.total_tx_size = total_tx_size` and `self.total_tx_cycles = total_tx_cycles` **overwrite** the already-decremented live fields with the stale pre-eviction snapshot, erasing the subtractions performed in step 2. [4](#0-3) 

The eviction path in `check_and_record_ancestors()` is triggered when a new transaction's ancestor count exceeds `max_ancestors_count` but can be reduced by evicting "cell_ref_parents" — existing pool entries that reference the same cell as a dep that the new transaction consumes as an input: [5](#0-4) 

### Impact Explanation

`total_tx_size` is the primary pool-size enforcement variable: [6](#0-5) 

When inflated, `limit_size()` evicts legitimate pending/proposed transactions that would otherwise fit, constituting a tx-pool denial-of-service. Additionally, `updated_stat_for_add_tx()` uses the inflated `total_tx_size` to gate new admissions: [7](#0-6) 

This causes premature `Reject::Full` errors for new transactions even when the real pool occupancy is well below `max_tx_pool_size`. Both `total_tx_size` and `total_tx_cycles` are also surfaced directly to callers via the `tx_pool_info` RPC: [8](#0-7) 

Returning incorrect values misleads monitoring tools, fee-rate estimators, and any external protocol that relies on these figures.

### Likelihood Explanation

The eviction path is reachable by any unprivileged RPC/relay transaction submitter. The attacker:

1. Submits a set of transactions `D1…Dk` that each reference cell `C` as a `cell_dep`. These are recorded in `edges.deps[C]` and become `cell_ref_parents` for any future transaction that consumes `C`.
2. Submits a new transaction `T` that (a) consumes `C` as an input and (b) has a long ancestor chain in the pool exceeding `max_ancestors_count` (default 125). `T` is a valid transaction from the node's perspective.
3. `check_and_record_ancestors()` detects `ancestors_count > max_ancestors_count` but `ancestors_count − |cell_ref_parents| ≤ max_ancestors_count`, so it evicts `D1…Dk`.
4. The evictions decrement `self.total_tx_size` / `self.total_tx_cycles`, but the stale snapshot is written back, inflating both counters by the sum of evicted sizes/cycles.

This can be repeated across multiple submissions to progressively inflate the counters without bound.

### Recommendation

Compute the new totals **after** `check_and_record_ancestors()` returns, using the live (already-eviction-adjusted) `self.total_tx_size` and `self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER evictions have already adjusted self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, recompute from scratch via `recompute_total_stat()` after all mutations are complete.

### Proof of Concept

```
Initial pool state:
  total_tx_size  = 1_000_000
  total_tx_cycles = 500_000
  Pool contains D1 (size=200_000, cycles=100_000) referencing cell C as dep.

Attacker submits T (size=50_000, cycles=25_000):
  - T consumes cell C as input  →  D1 is a cell_ref_parent
  - T has 126 ancestors in pool  →  ancestors_count (127) > max_ancestors_count (125)
  - 127 − 1 (D1) = 126 > 125, so D1 is evicted

Step-by-step in add_entry():
  1. updated_stat_for_add_tx(50_000, 25_000):
       total_tx_size_snapshot  = 1_000_000 + 50_000  = 1_050_000
       total_tx_cycles_snapshot = 500_000  + 25_000  = 525_000

  2. check_and_record_ancestors() evicts D1:
       self.total_tx_size  = 1_000_000 − 200_000 = 800_000
       self.total_tx_cycles = 500_000  − 100_000 = 400_000

  3. self.total_tx_size  = 1_050_000  ← snapshot overwrites 800_000
     self.total_tx_cycles = 525_000   ← snapshot overwrites 400_000

Correct values:  total_tx_size=850_000, total_tx_cycles=425_000
Actual values:   total_tx_size=1_050_000, total_tx_cycles=525_000
Inflation:       +200_000 bytes, +100_000 cycles (D1's contribution never subtracted)
```

Repeating this attack inflates `total_tx_size` until it exceeds `max_tx_pool_size` (default 180 MB), at which point `limit_size()` begins evicting honest transactions and `updated_stat_for_add_tx()` rejects all new submissions with `Reject::Full`.

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
