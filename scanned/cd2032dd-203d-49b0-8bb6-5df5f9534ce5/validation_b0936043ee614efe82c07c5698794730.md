### Title
Wrong Variable Reference in `script_exists_in_output` Causes Script Deletion During Block Rollback on PostgreSQL - (`File: util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

The function `script_exists_in_output` in the Rich Indexer's rollback path contains a copy-paste bug: after fetching the `row_type` query result, the final match arm reads from `row_lock` instead of `row_type`. On PostgreSQL, this causes the function to always return `false` for the type-script existence check, so any script referenced only as a `type_script_id` is incorrectly deleted from the `script` table during block rollback, corrupting the Rich Indexer's relational database.

---

### Finding Description

`script_exists_in_output` is called during `rollback_block` to decide whether a script record is still referenced by any surviving output before deleting it. The function performs two SQL `EXISTS` queries: one for `lock_script_id` and one for `type_script_id`. [1](#0-0) 

The first half is correct: if `row_lock` is `true`, the function returns `Ok(true)` early. After that block, the code fetches `row_type` (the type-script query), but the final match statement on line 252 reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`:

```rust
// BUG on line 252: row_lock should be row_type
match row_lock.try_get::<bool, _>(0) {   // <-- wrong variable
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

**On PostgreSQL** (`try_get::<bool, _>` succeeds): at this point in the code, `row_lock` is already known to be `Ok(false)` — if it had been `Ok(true)`, the function would have returned early. Therefore the final match always evaluates to `Ok(false)`, completely ignoring `row_type`. Any script that is referenced only as a `type_script_id` (not as a `lock_script_id`) is reported as non-existent and is deleted.

**On SQLite** (`try_get::<bool, _>` fails with `Err`): the `Err` branch correctly reads `row_type.get::<i64, _>(0) == 1`, so SQLite is unaffected. [2](#0-1) 

The deletion of the now-orphaned script IDs is performed unconditionally: [3](#0-2) 

---

### Impact Explanation

During any block rollback (chain reorganization), the Rich Indexer on PostgreSQL will delete `script` rows that are still referenced by surviving outputs via `type_script_id`. This corrupts the Rich Indexer's relational database:

1. Subsequent queries against the Rich Indexer RPC (e.g., `get_cells`, `get_transactions`) that filter by type script will return incorrect or empty results for affected scripts.
2. If foreign-key constraints are enforced, subsequent inserts or queries may produce database errors, halting the indexer.
3. The only recovery is a full re-index from genesis.

The `script` table schema enforces `UNIQUE(code_hash, hash_type, args)`, and `output` rows reference `type_script_id` as a foreign key, so the deletion of a still-referenced script row is a direct integrity violation. [4](#0-3) 

---

### Likelihood Explanation

Block rollbacks occur naturally during any chain reorganization, which is a routine event on a live network. No attacker action is required to trigger the bug — any reorg of depth ≥ 1 that involves outputs with type scripts will exercise the faulty path. The bug is deterministic and reproducible on any PostgreSQL-backed Rich Indexer deployment that experiences a reorg.

The Rich Indexer is an optional but production-supported component exposed via the CKB RPC layer. Operators running it with PostgreSQL (the recommended backend for production) are affected.

---

### Recommendation

On line 252, replace `row_lock` with `row_type`:

```rust
// BEFORE (buggy):
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// AFTER (correct):
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

---

### Proof of Concept

1. Run a CKB node with the Rich Indexer configured to use PostgreSQL.
2. Submit a transaction whose output has a `type_script` (e.g., a Simple UDT cell). Let it be included in block N.
3. Trigger a chain reorganization that rolls back block N (e.g., by submitting a competing block at the same height with higher difficulty, or by using the test RPC).
4. Observe that `rollback_block` is called, which calls `script_exists_in_output(type_script_id, tx)`.
5. Because `row_lock` (lock-script query) returns `false` and the final match reads `row_lock` instead of `row_type`, the function returns `false` even though the type script is still referenced by other surviving outputs.
6. The script row is deleted from the `script` table.
7. Subsequent `get_cells` RPC calls filtering by that type script return empty results, confirming database corruption. [5](#0-4)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
