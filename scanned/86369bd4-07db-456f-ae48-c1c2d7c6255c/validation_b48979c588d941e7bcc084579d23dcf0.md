Audit Report

## Title
`check_tx_fee` Minimum Fee Gate Uses Serialized Size Only, Ignoring Cycle-Based Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the actual block-packing weight used everywhere else in the pool (eviction, scoring, miner selection) is `get_transaction_weight(size, cycles) = max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction with small serialized size but near-maximum cycles passes the admission gate with a fee calibrated only to its size, while its true weight-adjusted fee rate is orders of magnitude lower. Such transactions are admitted to the pool, never selected by miners, and persist until expiry, enabling sustained pool resource exhaustion.

## Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` (L45) computes:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code itself acknowledges this is a deliberate approximation (L42–44):
> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly"

However, once admitted, every weight-sensitive operation uses `get_transaction_weight(size, cycles)` defined in `util/types/src/core/tx_pool.rs` (L298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. For a 250-byte transaction consuming 69,000,000 cycles:
- `get_transaction_weight(250, 69_000_000) = max(250, 11769) = 11769`
- Admission min_fee: `1000 * 250 / 1000 = 250 shannons`
- True weight min_fee: `1000 * 11769 / 1000 = 11769 shannons`

The attacker pays 251 shannons and passes admission. The actual fee rate inside the pool (`TxEntry::fee_rate()`, `entry.rs` L115–118) and the eviction key (`EvictKey` via `From<&TxEntry>`, `entry.rs` L234–247) both use `get_transaction_weight`, so the transaction's effective fee rate is `251 * 1000 / 11769 ≈ 21 shannons/KW` — far below `min_fee_rate = 1000`. Miners computing block templates rank by this weight-adjusted fee rate and never select it.

The pool size limit (`limit_size`, `pool.rs` L292–329) triggers only when `total_tx_size > max_tx_pool_size` (180 MB), where `total_tx_size` is the sum of serialized sizes. Eviction selects by lowest `EvictKey` (weight-adjusted fee rate), so these underpriced-by-weight transactions are evicted first — but only after the pool is already full. The attacker can fill the 180 MB pool with ~720,000 such 250-byte transactions at a cost of ~180,720 shannons total, while the true weight-based cost should be ~8,473,680 shannons — a ~47× discount.

The exploit path:
1. `pre_check` → `check_tx_fee` called with `tx_size` only (no cycles known yet, cycles are determined after script execution in `verify_rtx`)
2. `verify_rtx` executes scripts, determines actual cycles
3. `submit_entry` → `_submit_entry` → `limit_size` — by this point the tx is already in the pool

The structural reason the fix cannot be trivially applied at `check_tx_fee` call time is that cycles are not yet known at `pre_check` time. However, a post-verification fee check using actual cycles is feasible and is the correct fix.

## Impact Explanation

This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. An attacker can fill the 180 MB tx-pool with transactions that are never mined, at a fee cost ~47× below what the protocol intends for the block-weight consumed. Legitimate high-fee transactions may be delayed or evicted. The attack is sustained (12-hour expiry per transaction) and repeatable.

## Likelihood Explanation

**Medium-High.** No special privilege is required — any RPC caller or P2P relay peer can submit transactions. The attacker must craft a lock script that executes a tight computational loop consuming near `max_tx_verify_cycles = 70,000,000` cycles with minimal serialized size. This is straightforward with CKB-VM (a tight RISC-V loop in a lock script). The fee cost is low (250 shannons per transaction). The 12-hour expiry and weight-based eviction provide partial mitigation but do not prevent sustained pressure since the attacker can continuously resubmit.

## Recommendation

After `verify_rtx` completes and the actual cycle count is known, perform a second fee check using the true weight:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors how `TxEntry::fee_rate()` and `AncestorsScoreSortKey` already compute effective fee rates, closing the gap between admission policy and actual resource cost. The existing size-only check in `check_tx_fee` can remain as a cheap pre-verification filter.

## Proof of Concept

1. Deploy a CKB lock script containing a tight RISC-V loop that consumes ~69,000,000 cycles (just under `max_tx_verify_cycles`).
2. Construct a minimal transaction (~250 bytes serialized) spending a cell locked by this script.
3. With default `min_fee_rate = 1000` shannons/KW:
   - Size-based `min_fee = 1000 * 250 / 1000 = 250` shannons
   - Submit with fee = 251 shannons via `send_transaction` RPC
4. `check_tx_fee` passes (251 > 250). `verify_rtx` executes the script, consuming ~69M cycles. `TxEntry` is created with `cycles ≈ 69_000_000`.
5. Inside the pool, `fee_rate() = FeeRate::calculate(251, get_transaction_weight(250, 69_000_000)) = 251*1000/11769 ≈ 21 shannons/KW` — far below `min_fee_rate`.
6. Miners never select this transaction. It persists for 12 hours.
7. Repeat ~720,000 times to fill the 180 MB pool (`180_000_000 / 250 = 720,000` transactions), total cost ≈ 180,720 shannons.
8. Observe that legitimate transactions with proper fee rates are evicted or delayed.
