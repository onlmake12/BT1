Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites Correctly-Decremented `total_tx_size` in `PoolMap::add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts cell-dep parent transactions via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, those evictions correctly decrement `self.total_tx_size`. However, the stale pre-eviction snapshot is then unconditionally written back to `self.total_tx_size`, permanently inflating the counter by the aggregate size of all evicted transactions. An unprivileged attacker can exploit this repeatedly to make `limit_size` believe the pool is over capacity, causing valid transactions to be evicted and new submissions to be rejected.

## Finding Description
The exact sequence in `add_entry` (lines 210–220):

```rust
// Line 210-211: snapshot taken BEFORE evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: evictions happen here, correctly decrementing self.total_tx_size
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218-219: stale snapshot written back, overwriting the decremented value
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) computes `self.total_tx_size.checked_add(tx_size)` — a pure read of the current counter — and returns it as a local. Inside `check_and_record_ancestors` (lines 603–625), when `ancestors_count > max_ancestors_count` but reducible by evicting cell-dep parents, `remove_entry_and_descendants` is called in a loop. Each call reaches `update_stat_for_remove_tx` (lines 733–758), which correctly does `self.total_tx_size -= evicted.size`. After the loop returns, lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, erasing every decrement.

The self-healing path (`recompute_total_stat`) is only triggered on underflow (lines 742–756), never on overflow, so the inflation is never corrected.

**Net effect per invocation:**

| Counter | Correct value | Actual value written |
|---|---|---|
| `total_tx_size` | `original − Σ(evicted.size) + entry.size` | `original + entry.size` |
| `total_tx_cycles` | `original − Σ(evicted.cycles) + entry.cycles` | `original + entry.cycles` |

## Impact Explanation
`limit_size` (pool.rs lines 298–327) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting transactions and issuing `Reject::Full` until the counter drops below the cap. An inflated `total_tx_size` causes this loop to fire even when actual pool occupancy is far below the limit, evicting legitimate pending transactions and rejecting new valid submissions. The inflation accumulates across repeated attack invocations. This constitutes a **High** severity impact: a bad design reachable with few costs that causes CKB network congestion by degrading the mempool's ability to accept and relay valid transactions.

## Likelihood Explanation
The trigger condition — a new transaction whose ancestor count via cell-dep references exceeds `max_ancestors_count` but can be reduced by evicting cell-dep parents — is reachable by any unprivileged caller of the `send_transaction` RPC or any P2P relay peer. No key material, privileged access, or majority hash power is required. The attacker pre-populates the pool with a cell-dep chain, then submits a crafted transaction to trigger the eviction branch. The attack is repeatable and the inflation compounds with each iteration.

## Recommendation
Move the stat computation to after `check_and_record_ancestors` completes, so evictions are already reflected in `self.total_tx_size` before the snapshot is taken:

```diff
- let (total_tx_size, total_tx_cycles) =
-     self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
  trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
  evicts = self.check_and_record_ancestors(&mut entry)?;
  self.record_entry_edges(&entry)?;
  self.insert_entry(&entry, status);
  self.record_entry_descendants(&entry);
  self.track_entry_statics(None, Some(status));
+ let (total_tx_size, total_tx_cycles) =
+     self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
  self.total_tx_size = total_tx_size;
  self.total_tx_cycles = total_tx_cycles;
```

This ensures the snapshot reflects the post-eviction state of `self.total_tx_size`. Alternatively, remove the snapshot pattern entirely and call `update_stat_for_remove_tx` / increment inline so there is no stale local variable to overwrite.

## Proof of Concept
1. Configure a node with `max_ancestors_count = N` and a modest `max_tx_pool_size`.
2. Submit transactions `T1 … TN` where each `Ti+1` references `Ti` as a cell dep, forming a chain of depth `N`. Record `total_tx_size_before`.
3. Submit `Tnew` referencing `T1` as a cell dep. Its ancestor count is `N+1 > max_ancestors_count`, but `ancestors_count - cell_ref_parents.len() = N <= max_ancestors_count`, so the eviction branch fires. `T1` (and its descendants) are evicted; `self.total_tx_size` is decremented by their aggregate size. Then line 218 writes back the pre-eviction snapshot, re-inflating `total_tx_size` by that same aggregate.
4. Query pool stats and observe `total_tx_size > total_tx_size_before + Tnew.size` — the inflation equals the evicted transactions' sizes.
5. Repeat steps 2–4. Each iteration inflates `total_tx_size` further.
6. After enough iterations, `total_tx_size > max_tx_pool_size` despite the pool being nearly empty. `limit_size` begins evicting all remaining transactions and rejecting new submissions with `Reject::Full`.

A unit test can be written against `PoolMap::add_entry` directly: assert that after a triggered eviction, `pool_map.total_tx_size` equals the sum of sizes of entries actually present in `pool_map.entries`, and verify the current code fails this invariant.
