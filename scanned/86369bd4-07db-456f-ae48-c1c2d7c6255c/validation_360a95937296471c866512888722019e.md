Audit Report

## Title
Peer Store Permanently Blocked by `Group::None` Flooding via Onion3 Discovery Addresses — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
The `check_purge` second-pass eviction logic uses `take(len / 2)` where `len` is the number of distinct network groups. When all 16384 stored addresses map to `Group::None` (as Onion3 addresses do), integer division yields `take(0)`, evicting nothing. Combined with the first pass being a no-op for fresh entries, `check_purge` returns `Err(PeerStoreError::EvictionFailed)` permanently, blocking all future `add_addr` calls and preventing the node from ever adding new peer addresses.

## Finding Description

**Step 1 — Constants confirmed:**

`ADDR_COUNT_LIMIT = 16384` and `ADDR_MAX_RETRIES = 3` are confirmed in the source. [1](#0-0) 

**Step 2 — Onion3 → `Group::None`:**

`Group::from(&Multiaddr)` calls `multiaddr_to_socketaddr`. Onion3 addresses have no socket address representation, so they return `None` and fall through to `Group::None`. [2](#0-1) 

**Step 3 — First pass is a no-op for fresh entries:**

`AddrInfo::new` initializes `attempts_count = 0` and `last_connected_at_ms = 0`. In `is_connectable`, the only path returning `false` for `last_connected_at_ms == 0` requires `attempts_count >= ADDR_MAX_RETRIES` (i.e., `0 >= 3`), which is false. All 16384 freshly-added Onion3 entries are considered connectable, so the first pass evicts nothing. [3](#0-2) [4](#0-3) 

**Step 4 — The broken `take(len / 2)` in the second pass:**

With all 16384 entries in a single `Group::None` bucket, `peers_by_network_group.len() == 1`. The expression `take(len / 2)` = `take(1 / 2)` = `take(0)` due to integer division, iterating over zero groups and producing an empty `candidate_peers`. [5](#0-4) 

**Step 5 — Permanent `EvictionFailed`:**

With `candidate_peers` empty, `check_purge` returns `Err(PeerStoreError::EvictionFailed)`. Every subsequent `add_addr` call hits the same path since the store remains at 16384 entries and the same logic repeats indefinitely. [6](#0-5) 

**Step 6 — Reachability via discovery:**

`add_addr` is the entry point for discovery-received addresses. It calls `check_purge` before inserting, with no rate limiting or per-source quota visible. [7](#0-6) 

## Impact Explanation

Once the peer store is saturated with `Group::None` entries, no legitimate IPv4/IPv6 peer address can ever be added. The node permanently loses the ability to discover new peers, reconnect after disconnects, or bootstrap. This matches **Medium (2001–10000): Suboptimal implementation of CKB state storage mechanism** — the peer store is the node's peer address storage mechanism, and this flaw allows a trivial attacker to render it permanently inoperable for new entries.

## Likelihood Explanation

An attacker needs only a single P2P connection to a victim node. By sending `GetNodes` responses containing 16384 unique Onion3 `Multiaddr` values (the 35-byte host field provides a vast unique address space), the attacker saturates the store. The attack is cheap, deterministic, requires no special privileges beyond one P2P connection, and is not rate-limited at the `add_addr` level.

## Recommendation

Fix the integer division edge case in `check_purge`:

1. Replace `take(len / 2)` with `take(len.saturating_add(1) / 2)` (ceiling division) so a single group is still considered for eviction.
2. Alternatively, use `take(std::cmp::max(1, len / 2))`.
3. Additionally, consider assigning each `Group::None` address its own unique group key (e.g., a hash of the full multiaddr) to prevent all non-IP addresses from collapsing into one eviction-resistant bucket. [8](#0-7) 

## Proof of Concept

```rust
#[test]
fn test_group_none_eviction_failure() {
    use crate::peer_store::{PeerStore, ADDR_COUNT_LIMIT};
    use crate::Flags;
    use p2p::multiaddr::Multiaddr;

    let mut store = PeerStore::default();

    // Fill store with 16384 unique Onion3 addresses (all map to Group::None)
    for i in 0u64..ADDR_COUNT_LIMIT as u64 {
        let mut host = [0u8; 35];
        host[..8].copy_from_slice(&i.to_le_bytes());
        let onion_addr: Multiaddr = format!(
            "/onion3/{}:1234",
            base32::encode(base32::Alphabet::RFC4648 { padding: false }, &host)
        ).parse().unwrap();
        let _ = store.add_addr(onion_addr, Flags::COMPATIBILITY);
    }

    // Attempt to add a legitimate IPv4 peer — returns Err(EvictionFailed)
    let ipv4_addr: Multiaddr = "/ip4/1.2.3.4/tcp/8115".parse().unwrap();
    let result = store.add_addr(ipv4_addr, Flags::COMPATIBILITY);
    assert!(result.is_ok(), "Legitimate peer should be addable after eviction");
}
```

The test will fail at the assertion, confirming the permanent `EvictionFailed` state.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
