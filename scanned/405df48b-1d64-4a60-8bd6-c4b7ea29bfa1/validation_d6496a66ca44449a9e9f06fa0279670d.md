### Title
Transaction Relay Acceptance Restricted to Single Requesting Peer, Enabling Permanent Transaction Suppression — (`sync/src/relayer/transactions_process.rs`)

### Summary

In CKB's relay protocol, when a node receives a `RelayTransactions` message, it only accepts transactions from the exact peer that was previously sent a `GetRelayTransactions` request. Any other peer delivering the same transaction is silently dropped. When the single requesting peer fails to respond and no other peer has announced the same hash, the transaction entry is permanently evicted from the `unknown_tx_hashes` queue with no retry path, causing the node to permanently miss that transaction.

### Finding Description

**Root cause — `TransactionsProcess::execute`:**

In `sync/src/relayer/transactions_process.rs`, the filter at lines 49–55 rejects any transaction whose sender is not the currently-designated `requesting_peer`:

```rust
.filter(|(tx, _)| {
    !tx_filter.contains(&tx.hash())
        && unknown_tx_hashes
            .get_priority(&tx.hash())
            .map(|priority| priority.requesting_peer() == Some(self.peer))
            .unwrap_or_default()
})
```

`requesting_peer()` is defined in `sync/src/types/mod.rs` at lines 1295–1301:

```rust
pub fn requesting_peer(&self) -> Option<PeerIndex> {
    if self.requested {
        self.peers.first().cloned()
    } else {
        None
    }
}
```

It returns `Some(peers[0])` only after a `GetRelayTransactions` has been sent. Any other peer delivering the same transaction is unconditionally rejected.

**Permanent eviction path — `next_request_peer` + `pop_ask_for_txs`:**

When the requesting peer does not respond, `pop_ask_for_txs` (lines 1466–1479 of `sync/src/types/mod.rs`) calls `next_request_peer()`:

```rust
pub fn next_request_peer(&mut self) -> Option<PeerIndex> {
    if self.requested {
        if self.peers.len() > 1 {
            // rotate to next peer
            self.peers.swap_remove(0);
            self.peers.first().cloned()
        } else {
            None   // ← single-peer case: no retry
        }
    } else { ... }
}
```

When only one peer announced the hash (the common case), `next_request_peer()` returns `None`. In `pop_ask_for_txs`, the entry is **not pushed back** into `unknown_tx_hashes`:

```rust
if let Some(peer_index) = priority.next_request_peer() {
    result.entry(peer_index)...;
    unknown_tx_hashes.push(tx_hash, priority);  // only pushed back if Some
}
// If None: entry is silently dropped from the queue forever
```

The entry is also not added to `tx_filter`, so the node is left in a state where the transaction is neither known nor queued for re-request.

**Exploit flow:**

1. Malicious peer M connects to victim node V.
2. M sends `RelayTransactionHashes` containing hash H for a valid pending transaction.
3. V records M as the sole peer for H in `unknown_tx_hashes` (`requested = false`, `peers = [M]`).
4. V sends `GetRelayTransactions{H}` to M. M is now `requesting_peer`.
5. Honest peer P also knows H and sends `RelayTransactions{H}` to V.
6. V rejects P's delivery: `requesting_peer() == Some(M) != Some(P)` → filtered out.
7. M disconnects or never responds.
8. After `RETRY_ASK_TX_TIMEOUT_INCREASE` (30 s), `pop_ask_for_txs` fires, calls `next_request_peer()` → `None` (only one peer). Entry is permanently dropped.
9. Transaction H is now permanently absent from V's mempool unless another peer re-announces it.

### Impact Explanation

A malicious peer can selectively suppress specific transactions from a victim node's mempool by announcing their hashes and then refusing to deliver them. If no other connected peer re-announces the same hash, the transaction is permanently unavailable to that node. For a mining node, this means targeted transactions are excluded from block templates, enabling fee-sniping or censorship of specific transactions. For a relay node, it degrades transaction propagation across the network. The impact is transaction-level liveness failure triggered by a single unprivileged peer.

### Likelihood Explanation

Any peer that can connect to a CKB node (no privilege required) can execute this attack. The attacker only needs to: (1) know a valid pending transaction hash, (2) announce it, and (3) not respond to the fetch request. The 30-second retry window is short enough that the attack completes before the victim can recover. The attack is cheap, repeatable, and requires no cryptographic capability.

### Recommendation

Remove the `requesting_peer() == Some(self.peer)` restriction in `TransactionsProcess::execute`. Accept a transaction from any peer that delivers it, provided it passes the existing validity checks (not already known, cycles within limit). The `requesting_peer` check provides no security guarantee — a malicious peer can still send invalid transactions regardless — and its only effect is to block honest peers from delivering transactions the node needs. Alternatively, when `next_request_peer()` returns `None`, re-insert the entry into `unknown_tx_hashes` with a fresh timeout so that a future peer announcement can trigger a new request.

### Proof of Concept

1. Connect a custom peer M to a CKB node V.
2. Observe a valid pending transaction hash H from the network.
3. Send `RelayTransactionHashes{H}` from M to V.
4. V responds with `GetRelayTransactions{H}` to M.
5. From a second honest peer P, send `RelayTransactions{H}` to V.
6. Observe V silently drops P's delivery (no pool entry created).
7. M disconnects without responding.
8. After 30 seconds, confirm H is absent from V's `unknown_tx_hashes` and absent from V's tx pool.
9. Confirm V never requests H from any other peer.
