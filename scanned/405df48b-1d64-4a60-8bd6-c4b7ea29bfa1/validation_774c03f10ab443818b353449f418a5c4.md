### Title
Inconsistent `script_len_range` Upper-Bound Check Between `get_cells` and `get_cells_capacity` — (`util/indexer/src/service.rs`)

---

### Summary

The `get_cells` and `get_cells_capacity` RPC handlers in the CKB indexer apply the `script_len_range` filter with different comparison operators for the upper bound. `get_cells` correctly uses `>= r1` (exclusive upper bound, matching the documented `[inclusive, exclusive]` semantics), while `get_cells_capacity` uses `> r1` (inclusive upper bound). This means cells with `script_len == r1` are excluded from `get_cells` results but included in `get_cells_capacity` totals, producing inconsistent and incorrect capacity values for any caller relying on both endpoints with the same filter.

---

### Finding Description

The `script_len_range` filter is documented as `[inclusive, exclusive]` — i.e., a cell passes the filter if `r0 <= script_len < r1`.

In `get_cells` (the cell-listing path), the check is:

```rust
// util/indexer/src/service.rs  lines 299–317
if let Some([r0, r1]) = filter_options.script_len_range {
    match filter_script_type {
        IndexerScriptType::Lock => {
            let script_len = extract_raw_data(&output.lock()).len();
            if script_len < r0 || script_len >= r1 {   // ✓ exclusive upper bound
                return None;
            }
        }
        IndexerScriptType::Type => {
            ...
            if script_len < r0 || script_len >= r1 {   // ✓ exclusive upper bound
                return None;
            }
        }
    }
}
```

In `get_cells_capacity` (the capacity-summing path), the same filter is applied with a different operator:

```rust
// util/indexer/src/service.rs  lines 776–794
if let Some([r0, r1]) = filter_options.script_len_range {
    match filter_script_type {
        IndexerScriptType::Lock => {
            let script_len = extract_raw_data(&output.lock()).len();
            if script_len < r0 || script_len > r1 {    // ✗ inclusive upper bound
                return None;
            }
        }
        IndexerScriptType::Type => {
            ...
            if script_len < r0 || script_len > r1 {    // ✗ inclusive upper bound
                return None;
            }
        }
    }
}
```

The `get_cells` path uses `>= r1` (correct), while `get_cells_capacity` uses `> r1` (incorrect). Any cell whose script length equals exactly `r1` is excluded by `get_cells` but included by `get_cells_capacity`.

The `FilterOptions` struct and its construction from `IndexerSearchKey` are shared between both paths, so the discrepancy is purely in the comparison operator used at query time. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

An RPC caller using `get_cells_capacity` with a `script_len_range` filter receives an inflated capacity total: it includes cells whose script length equals `r1`, which `get_cells` with the same parameters would exclude. This breaks the invariant that `get_cells_capacity` returns the sum of capacities of cells returned by `get_cells`. Applications that use `get_cells_capacity` to determine available balance or collateral before iterating cells via `get_cells` will observe a discrepancy, potentially leading to incorrect accounting decisions (e.g., believing more capacity is available than `get_cells` can actually enumerate).

The documentation explicitly states the range is `[inclusive, exclusive]`: [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged RPC caller can trigger this by calling `get_cells_capacity` with a `script_len_range` filter whose upper bound `r1` matches the script length of at least one live cell. This is a common use case — the RPC documentation even provides an example of filtering empty type scripts with `script_len_range: [0, 1)`. The condition is trivially reachable with no special privileges. [5](#0-4) 

---

### Recommendation

Change the upper-bound comparison in `get_cells_capacity`'s `script_len_range` filter from `> r1` to `>= r1` to match the documented exclusive semantics and the behavior of `get_cells`:

```rust
// util/indexer/src/service.rs  ~line 780
if script_len < r0 || script_len >= r1 {   // was: script_len > r1
    return None;
}
// and the Type branch at ~line 790
if script_len < r0 || script_len >= r1 {   // was: script_len > r1
    return None;
}
``` [6](#0-5) 

---

### Proof of Concept

1. Index a chain containing a cell whose type script has a serialized length of exactly `N` bytes.
2. Call `get_cells` with `script_len_range: [0, N]` (i.e., `r1 = N`). The cell is **excluded** (`script_len >= r1` triggers).
3. Call `get_cells_capacity` with the same `script_len_range: [0, N]`. The cell is **included** (`script_len > r1` does not trigger because `N > N` is false).
4. Observe that `get_cells_capacity` returns a non-zero capacity while `get_cells` returns an empty list — a direct contradiction. [1](#0-0) [2](#0-1)
