### Title
`BlockReader::extension()` Panics on Malformed Extra Field — (`File: util/gen-types/src/extension/shortcut.rs`)

---

### Summary

`BlockReader::extension()` unconditionally calls `.unwrap()` when parsing the first extra field of a block as a molecule `Bytes` structure. If a peer sends a block whose extra field bytes exist but are not a valid molecule `Bytes` encoding, the call panics. This is a direct analog to the TON report: a sum-type field (absent vs. present-and-valid vs. present-but-malformed) is accessed with an assumption that only one variant is possible.

---

### Finding Description

`BlockReader::extension()` is defined as:

```rust
/// # Panics
///
/// Panics if the first extra field exists but not a valid [`BytesReader`].
pub fn extension(&self) -> Option<packed::BytesReader<'_>> {
    self.extra_field(0)
        .map(|data| packed::BytesReader::from_slice(data).unwrap())
}
``` [1](#0-0) 

The function models three states but only handles two:

| State | Handled? |
|---|---|
| No extra field → `None` | Yes |
| Extra field present, valid `Bytes` encoding → `Some(reader)` | Yes |
| Extra field present, **invalid** `Bytes` encoding → **panic** | **No** |

Blocks received from peers are parsed with `from_compatible_slice`, which validates outer molecule structure (field count, offsets) but does **not** validate the semantic content of extra fields. A molecule `Bytes` value requires a 4-byte little-endian length prefix followed by exactly that many bytes. Any deviation causes `BytesReader::from_slice` to return `Err`, and `.unwrap()` panics.

The `ShouldBeOk` trait used elsewhere in the codebase is an explicit alias for `.unwrap()` with a panic message, confirming that panics are the intended failure mode when assumptions are violated: [2](#0-1) 

`extension()` is called in the contextual block verifier, the store transaction layer, the relayer's `reconstruct_block`, and the compact block builder — all of which can be reached via externally-supplied block data: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A panic in the block-processing or relay thread terminates that thread. The chain service and relayer are long-running services; if their processing goroutine/thread panics without a `catch_unwind` guard, the node stops processing inbound blocks entirely. This is equivalent to the TON finding where the observer goroutine crashes and no further inbound deposits are processed.

---

### Likelihood Explanation

Any unprivileged P2P peer can send a `CompactBlock` or full `Block` message. The relay handler parses the message with `from_compatible_slice` (lenient) and only checks `count_extra_fields() > 1` before calling `reconstruct_block`, which calls `compact_block.extension()`. A single crafted message with a 1-byte extra field (e.g., `[0xFF]`, which is not a valid molecule `Bytes` header) is sufficient to trigger the panic. No authentication, stake, or mining power is required. [5](#0-4) 

---

### Recommendation

Replace the `.unwrap()` in `extension()` with a graceful error path:

```rust
pub fn extension(&self) -> Option<packed::BytesReader<'_>> {
    self.extra_field(0).and_then(|data| {
        packed::BytesReader::from_slice(data).ok()
    })
}
```

This returns `None` for a malformed extra field instead of panicking, and callers that previously relied on `Some(...)` being a valid reader will simply treat the block as having no extension — which is the safe fallback.

---

### Proof of Concept

1. Craft a molecule-encoded `Block` (or `CompactBlock`) with exactly one extra field whose bytes are `[0x01, 0x00, 0x00, 0x00]` — a molecule `Bytes` header claiming length 1 but providing 0 content bytes. `from_compatible_slice` accepts this because the outer `Block` structure is valid.
2. Connect to a CKB node as a peer and send the crafted block via the relay or sync protocol.
3. The node calls `extension()` during contextual verification or compact-block reconstruction.
4. `packed::BytesReader::from_slice(&[0x01, 0x00, 0x00, 0x00])` returns `Err` (declared length 1 but slice is only 4 bytes total, mismatching the molecule `Bytes` layout).
5. `.unwrap()` panics, crashing the processing thread. [6](#0-5)
