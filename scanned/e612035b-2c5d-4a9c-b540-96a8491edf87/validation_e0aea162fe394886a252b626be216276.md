### Title
`NetworkController::ban` Passes Absolute Timestamp as Relative Duration to `ban_network`, Making All RPC-Initiated Bans Effectively Permanent - (`network/src/network.rs`)

---

### Summary

The `set_ban` RPC computes an absolute `ban_until` timestamp and passes it to `NetworkController::ban`. That function forwards the value unchanged to `PeerStore::ban_network`, which treats its second argument as a **relative** `timeout_ms` and adds the current time to it. The result is `ban_until = now_ms + absolute_timestamp ≈ 2 × now_ms + intended_duration`, placing the expiry approximately 54 years in the future. Every ban inserted through the RPC is therefore effectively permanent, regardless of the duration the caller requested.

---

### Finding Description

**RPC layer** (`rpc/src/module/net.rs`, `set_ban`):

```rust
let ban_until = if absolute.unwrap_or(false) {
    ban_time.unwrap_or_default().into()          // absolute ms timestamp
} else {
    unix_time_as_millis()
        + ban_time.unwrap_or_else(|| DEFAULT_BAN_DURATION.into()).value()
                                                  // also absolute ms timestamp
};
self.network_controller
    .ban(ip_network, ban_until, reason.unwrap_or_default());
```

`ban_until` is always an **absolute** Unix millisecond timestamp before it is handed to `NetworkController::ban`.

**Controller layer** (`network/src/network.rs`, `NetworkController::ban`):

```rust
pub fn ban(&self, address: IpNetwork, ban_until: u64, ban_reason: String) {
    self.disconnect_peers_in_ip_range(address, &ban_reason);
    self.network_state
        .peer_store
        .lock()
        .ban_network(address, ban_until, ban_reason)   // ← passes absolute ts
}
```

The absolute timestamp is forwarded verbatim to `ban_network`.

**Store layer** (`network/src/peer_store/peer_store_impl.rs`, `ban_network`):

```rust
pub(crate) fn ban_network(&mut self, network: IpNetwork, timeout_ms: u64, ban_reason: String) {
    let now_ms = ckb_systemtime::unix_time_as_millis();
    let ban_addr = BannedAddr {
        address: network,
        ban_until: now_ms + timeout_ms,   // ← treats argument as relative duration
        created_at: now_ms,
        ban_reason,
    };
    self.mut_ban_list().ban(ban_addr);
}
```

`ban_network` names its parameter `timeout_ms` and adds `now_ms` to it. When the value it receives is already an absolute timestamp (~1.7 × 10¹² ms today), the stored `ban_until` becomes ~3.4 × 10¹² ms, i.e., roughly year 2078.

By contrast, the **internal** ban path (`ban_session` → `ban_addr` → `ban_network`) correctly passes a relative `duration.as_millis()`, so that path is unaffected.

---

### Impact Explanation

Every peer banned through the `set_ban` RPC (`command = "insert"`) receives a `ban_until` value approximately 54 years beyond the intended expiry. The ban is functionally permanent until the operator explicitly calls `set_ban` with `command = "delete"` or `clear_banned_addresses`. A node operator who intends to impose a short-term ban (e.g., 24 hours) to quarantine a misbehaving peer will instead permanently exclude that peer, degrading the node's ability to reconnect to legitimate peers after the intended quarantine period. A malicious local process with RPC access can exploit this to permanently sever the node's connections to chosen peers.

---

### Likelihood Explanation

The bug is triggered by every invocation of `set_ban` with `command = "insert"`, which is a standard operator action for peer management. No special conditions are required. The RPC is enabled by default and accessible to any process on localhost.

---

### Recommendation

`NetworkController::ban` should store the absolute timestamp directly rather than delegating to `ban_network`, or `ban_network` should be refactored to accept an absolute timestamp. The simplest fix is to construct the `BannedAddr` directly in `NetworkController::ban` (or add a separate `ban_network_until` variant), bypassing the `now_ms + timeout_ms` addition:

```rust
// In peer_store_impl.rs – add an absolute-timestamp variant:
pub(crate) fn ban_network_until(&mut self, network: IpNetwork, ban_until: u64, ban_reason: String) {
    let now_ms = ckb_systemtime::unix_time_as_millis();
    let ban_addr = BannedAddr { address: network, ban_until, created_at: now_ms, ban_reason };
    self.mut_ban_list().ban(ban_addr);
}
```

And call it from `NetworkController::ban` instead of `ban_network`.

---

### Proof of Concept

1. Start a CKB node with the default configuration.
2. Call the RPC:
   ```json
   {"method":"set_ban","params":["192.168.0.2","insert","0x5265C00",false,"test"]}
   ```
   (`0x5265C00` = 86 400 000 ms = 24 hours)
3. Call `get_banned_addresses` and inspect the returned `ban_until`.
4. Observe that `ban_until ≈ 2 × unix_time_as_millis() + 86_400_000`, placing expiry in approximately year 2078 rather than 24 hours from now.
