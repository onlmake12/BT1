### Title
Missing Index-Parity Check in `verify_transaction_and_witness_proof` Allows Witness-Transaction Decoupling — (`rpc/src/module/chain.rs`)

### Summary

`verify_transaction_and_witness_proof` verifies the `transactions_proof` and `witnesses_proof` as two fully independent Merkle proofs, then checks only that both computed roots combine to equal `block.transactions_root()`. There is no check that the two proofs cover the **same leaf positions**. An unprivileged RPC caller can supply a proof where `transactions_proof.indices = [i]` and `witnesses_proof.indices = [j]` (i ≠ j), both with correct lemmas drawn from the real block, and the final equality check will pass — returning `tx_hash[i]` as "verified" while the witness proof actually covers `witness_hash[j]`.

---

### Finding Description

The block's `transactions_root` is defined as:

```
transactions_root = merkle_root([
    merkle_root(all_tx_hashes),       // raw_transactions_root
    merkle_root(all_witness_hashes),  // witnesses_root
])
``` [1](#0-0) 

The verifier reconstructs both sub-roots independently: [2](#0-1) 

Step by step:

1. `CBMT::retrieve_leaves(block.tx_witness_hashes(), &witnesses_merkle_proof)` — reads the actual witness hashes from the block at the positions named in `witnesses_proof.indices`.
2. `witnesses_merkle_proof.root(...)` — recomputes the full witness-tree root using those leaves + the supplied lemmas.
3. `CBMT::retrieve_leaves(block.tx_hashes(), &transactions_merkle_proof)` — reads the actual tx hashes from the block at the positions named in `transactions_proof.indices`.
4. `transactions_merkle_proof.root(...)` — recomputes the full tx-tree root.
5. Final check: `block.transactions_root() == merkle_root([raw_transactions_root, witnesses_proof_root])`.

Because both sub-roots are recomputed from the **real block data** using the supplied indices and lemmas, any valid single-leaf proof for *any* position in each tree will reconstruct the correct full-tree root. The check at step 5 passes as long as each proof is individually valid — it does not require the two proofs to name the same position.

There is no guard of the form `transactions_proof.indices == witnesses_proof.indices` anywhere in the function. [3](#0-2) 

---

### Impact Explanation

An attacker constructs a `TransactionAndWitnessProof` for a block containing transactions `tx[0]…tx[N-1]`:

- `transactions_proof`: valid CBMT proof for `tx_hash[i]` at position `i` (correct lemmas from the real tx tree).
- `witnesses_proof`: valid CBMT proof for `witness_hash[j]` at position `j ≠ i` (correct lemmas from the real witness tree).

Both proofs individually reconstruct the correct sub-roots, so the final `merkle_root([raw_transactions_root, witnesses_proof_root]) == block.transactions_root()` check passes. The RPC returns `[tx_hash[i]]` as "verified."

A downstream consumer (light client, bridge, cross-chain relay, audit tool) that calls `verify_transaction_and_witness_proof` and trusts the result as proof that `tx_hash[i]` is bound to its witness receives a false guarantee. The proof actually binds `tx_hash[i]` to `witness_hash[j]`. This enables **witness substitution fraud**: an attacker can present a "valid" proof that associates an arbitrary transaction with an arbitrary witness from the same block.

---

### Likelihood Explanation

- Requires only an unprivileged JSON-RPC call — no keys, no PoW, no privileged access.
- Precondition (a block with ≥ 2 transactions) is met by every non-genesis block on mainnet.
- The crafted proof uses only real, publicly available block data (tx hashes, witness hashes, and their Merkle siblings), all obtainable via standard RPC calls.
- The attack is deterministic and requires no brute force or hash collision.

---

### Recommendation

Add an explicit check that the two proof index sets are identical before proceeding with verification, in `verify_transaction_and_witness_proof` in `rpc/src/module/chain.rs`:

```rust
if tx_proof.transactions_proof.indices != tx_proof.witnesses_proof.indices {
    return Err(RPCError::invalid_params(
        "transactions_proof and witnesses_proof must cover the same indices",
    ));
}
```

This check must be applied to the raw index values (after converting from `Uint32`) before constructing the two `MerkleProof` objects. [4](#0-3) 

---

### Proof of Concept

Given a block with transactions `[tx0, tx1]` and witnesses `[w0, w1]`:

1. Call `get_transaction_and_witness_proof(["<tx0_hash>"])` → receive a proof with `transactions_proof.indices=[leaf_pos_0]`, `witnesses_proof.indices=[leaf_pos_0]`.
2. Call `get_transaction_and_witness_proof(["<tx1_hash>"])` → receive a proof with `transactions_proof.indices=[leaf_pos_1]`, `witnesses_proof.indices=[leaf_pos_1]`.
3. Craft a new `TransactionAndWitnessProof`:
   - `block_hash`: same block
   - `transactions_proof`: from step 1 (proves `tx_hash[0]` at position 0)
   - `witnesses_proof`: from step 2 (proves `witness_hash[1]` at position 1)
4. Call `verify_transaction_and_witness_proof` with the crafted proof.
5. The RPC returns `["<tx0_hash>"]` with status OK — a "valid" proof that decouples `tx_hash[0]` from `witness_hash[0]` and instead binds it to `witness_hash[1]`.

No hash collision, no privileged access, no brute force required.
