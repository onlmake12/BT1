### Title
Unencrypted P2P Network Secret Key and Onion Private Key Stored in Plaintext Flat Files Without Platform Encryption — (`File: util/app-config/src/configs/network.rs`, `util/onion/src/onion_service.rs`)

---

### Summary

The CKB node stores its secp256k1 P2P network identity key (`secret_key`) and its Tor onion service private key (`onion_private_key`) as unencrypted raw bytes in flat files on disk. No platform-level encryption (e.g., Windows DPAPI, macOS Keychain, Linux kernel keyring) is used. On non-Unix platforms, the onion private key file is created with **no file permission restriction at all**, and the network secret key file receives only a weak `set_readonly` attribute (not ACL-based). A local attacker with filesystem access can trivially read both keys.

---

### Finding Description

**1. Network secret key — `write_secret_to_file` in `util/app-config/src/configs/network.rs`**

The function `write_secret_to_file` writes the raw 32-byte secp256k1 private key to a flat file: [1](#0-0) 

On Unix, the file is set to `0o400` (owner read-only). On non-Unix (Windows), only `set_readonly(true)` is called — this sets the Windows "read-only file attribute" flag, which is **not** an ACL-based restriction and does not prevent other local users from reading the file. No encryption is applied on any platform.

**2. Onion private key — `create_tor_secret_key` in `util/onion/src/onion_service.rs`**

The function `create_tor_secret_key` writes the Tor v3 private key (base64-encoded) to a flat file: [2](#0-1) 

The `#[cfg(unix)]` block sets `0o600` on Unix. However, there is **no `#[cfg(not(unix))]` fallback** — on Windows, the file is created with default OS permissions (world-readable by all local users). No encryption is applied on any platform.

**3. Key loading — `read_secret_key` in `util/app-config/src/configs/network.rs`**

The key is loaded back as raw bytes with only a warning (not enforcement) if permissions are wrong: [3](#0-2) 

The permission check is advisory only — the node continues to operate even if the file is world-readable.

**4. Key path construction — `Config::secret_key_path` and `Config::onion_private_key_path`** [4](#0-3) 

Both keys are stored under the network data directory (`data/network/secret_key` and `data/network/onion_private_key`) as plain files.

---

### Impact Explanation

- **P2P network secret key theft**: The secp256k1 key is the node's P2P identity used in the secio handshake. An attacker who reads this key can impersonate the node to other peers, enabling eclipse attacks — the attacker can present themselves as the legitimate node to other peers, intercept or manipulate peer discovery, and disrupt the node's view of the network. [5](#0-4) 

- **Onion private key theft**: The Tor v3 private key controls the node's `.onion` address. An attacker who reads this key can impersonate the node's hidden service, intercepting connections from peers who connect to the node via Tor.

- On non-Unix systems, the onion key file has **no permission restriction whatsoever**, making it readable by any local user account without any privilege escalation.

---

### Likelihood Explanation

Any local user account on the same machine as the CKB node can read the key files. This is the exact scenario described in the original report: a user walks away from an unlocked workstation, and another local user reads the key files. On Windows, the onion key file is created with default permissions (world-readable), requiring zero effort. On Unix, the network key is `0o400` (owner-only), but the onion key is `0o600` — still no encryption at rest. No platform secure storage is used on any platform.

---

### Recommendation

- **Short term**: On non-Unix platforms, apply explicit ACL-based restrictions to both key files at creation time (e.g., using Windows `SetFileSecurity` or the `windows-acl` crate). Add a `#[cfg(not(unix))]` block to `create_tor_secret_key` analogous to the one in `write_secret_to_file`.
- **Long term**: Encrypt both keys at rest using platform-provided secure storage: Windows DPAPI (`CryptProtectData`), macOS Keychain, or Linux kernel keyring/secret service. This prevents key theft even when filesystem permissions are bypassed (e.g., via backup access, shared storage, or privilege escalation).

---

### Proof of Concept

On a Windows machine running CKB with onion service enabled:

1. Start `ckb run` with `[network.onion] listen_on_onion = true`.
2. The file `data/network/onion_private_key` is created by `create_tor_secret_key` with no permission restriction (no `#[cfg(not(unix))]` fallback exists).
3. Any local user account can run: `type data\network\onion_private_key` and obtain the base64-encoded Tor v3 private key.
4. The attacker decodes the base64 and uses the key to stand up a Tor hidden service at the same `.onion` address, impersonating the victim node.

For the network secret key on Windows:
1. `data/network/secret_key` is created with `set_readonly(true)` only.
2. `set_readonly` sets the Windows read-only file attribute but does **not** restrict read access via ACLs — other local users can still open and read the file.
3. The attacker reads the 32-byte raw secp256k1 key and uses it to impersonate the node's P2P identity in secio handshakes. [6](#0-5) [7](#0-6)

pub fn write_secret_to_file(secret: &[u8], path: PathBuf) -> Result<(), Error> {
    fs::OpenOptions::new()
        .create(true)
        .write(true)
        .truncate(true)
        .open(path)
        .and_then(|mut file| {
            file.write_all(secret)?;
            #[cfg(unix)]
            {
                use std::os::unix::fs::PermissionsExt;
                file.set_permissions(fs::Permissions::from_mode(0o400))
            }
            #[cfg(not(unix))]
            {
                let mut permissions = file.metadata()?.permissions();
                permissions.set_readonly(true);
                file.set_permissions(permissions)
            }
        })
}

pub fn read_secret_key(path: PathBuf) -> Result<Option<secio::SecioKeyPair>, Error> {
    let mut file = match fs::File::open(path.clone()) {
        Ok(file) => file,
        Err(_) => return Ok(None),
    };
    let warn = |m: bool, d: &str| {
        if m {
            ckb_logger::warn!(
                "Your network secret file's permission is not {}, path: {:?}. \
                Please fix it as soon as possible",
                d,
                path
            )
        }
    };
    #[cfg(unix)]
    {
        use std::os::unix::fs::PermissionsExt;
        warn(
            file.metadata()?.permissions().mode() & 0o177 != 0,
            "less than 0o600",
        );
    }
    #[cfg(not(unix))]
    {
        warn(!file.metadata()?.permissions().readonly(), "readonly");
    }
    let mut buf = Vec::new();
    file.read_to_end(&mut buf).and_then(|_read_size| {
        secio::SecioKeyPair::secp256k1_raw_key(&buf)
            .map(Some)
            .map_err(|_| Error::new(ErrorKind::InvalidData, "invalid secret key data"))
    })
}

fn create_tor_secret_key(onion_private_key_path: String) -> Result<TorSecretKeyV3, Error> {
    let key = torut::onion::TorSecretKeyV3::generate();
    info!(
        "Generated new onion service v3 key for address: {}",
        key.public().get_onion_address()
    );

    #[cfg_attr(not(unix), allow(unused_mut))]
    let mut file_options = OpenOptions::new();
    #[cfg_attr(not(unix), allow(unused_mut))]
    let mut options = file_options.create(true).truncate(true).write(true);

    #[cfg(unix)]
    {
        use std::os::unix::fs::OpenOptionsExt;
        options = options.mode(0o600);
    }

    let mut file = options.open(&onion_private_key_path).map_err(|err| {
        InternalErrorKind::Other.other(format!(
            "Failed to open onion private key for writing: {:?}",
            err
        ))
    })?;
    file.write_all(
        &base64::engine::general_purpose::STANDARD
            .encode(key.as_bytes())
            .into_bytes(),
    )
    .map_err(|err| {
        InternalErrorKind::Other.other(format!("Failed to write onion private key: {:?}", err))
    })?;

    Ok(key)
}
