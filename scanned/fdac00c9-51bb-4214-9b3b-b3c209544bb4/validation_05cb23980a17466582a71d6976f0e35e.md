### Title
Unbounded O(N) Linear Scan in `get_pool_tx_detail_info` RPC Enables CPU/Lock Exhaustion — (`tx-pool/src/pool.rs`)

---

### Summary

The `get_pool_tx_detail_info` JSON-RPC endpoint calls `get_tx_detail`, which performs an O(N) linear scan over **all** pending transactions to compute `rank_in_pending`, plus two unbounded graph traversals (`calc_descendants`, `calc_ancestors`). Any unprivileged RPC caller can trigger this repeatedly while the mempool is large, causing sustained CPU exhaustion and blocking the tx-pool service's event loop for other operations (transaction submission, block assembly).

---

### Finding Description

`get_tx_detail` in `tx-pool/src/pool.rs` (lines 682–711) executes three expensive operations on every call:

**1. Full pending-ID collection + linear scan for rank:**

```rust
let ids = self.get_ids();          // collects ALL pending+gap tx hashes into a Vec
let rank_in_pending = ...
    ids.pending
        .iter()
        .enumerate()
        .find(|(_, hash)| &tx_hash == *hash)   // O(N) scan
        ...
```

`get_ids()` iterates the entire score-sorted index and collects every pending/gap transaction hash into a `Vec`. The subsequent `.find()` then scans that entire Vec linearly. With a mempool bounded only by `max_tx_pool_size` (default 180 MB) and minimum transaction size ~100 bytes, this can be ~1.8 million entries. [1](#0-0) 

**2. Unbounded descendant graph traversal:**

```rust
descendants_count: self.pool_map.calc_descendants(id).len(),
```

`calc_descendants` calls `calc_relative_ids` in `TxLinksMap`, which is an iterative BFS with **no depth or count limit**:

```rust
while let Some(id) = stage.iter().next().cloned() {
    if let Some(tx_links) = self.inner.get(&id) {
        for direct_id in tx_links.get_direct_ids(relation) {
            if !relation_ids.contains(direct_id) {
                stage.insert(direct_id.clone());
            }
        }
    }
    stage.remove(&id);
    relation_ids.insert(id);
}
```

While `max_ancestors_count` (1 000) caps the ancestor chain depth during insertion, **no corresponding limit exists for descendants**. A root transaction with many outputs can have a descendant fan-out spanning the entire pool. [2](#0-1) [3](#0-2) 

**3. Unbounded ancestor graph traversal:**

```rust
ancestors_count: self.pool_map.calc_ancestors(id).len(),
```

Same BFS, traversing up to `max_ancestors_count` ancestors (up to 999 nodes). [4](#0-3) 

The RPC is exposed publicly with no per-method rate limiting: [5](#0-4) 

---

### Impact Explanation

Each `get_pool_tx_detail_info` call:
- Allocates and scans a Vec of all pending tx hashes (O(N) CPU + memory)
- Performs two full BFS graph traversals (O(N) each)
- Executes while holding the tx-pool service's processing slot, blocking concurrent transaction submissions and block-assembly requests

An attacker who repeatedly calls this RPC while the mempool is large can cause sustained CPU exhaustion and degrade or stall the node's tx-pool service, affecting transaction relay, block template generation, and fee estimation.

---

### Likelihood Explanation

- The RPC is reachable by any node with RPC access (default: localhost, but commonly exposed to local tooling or proxied).
- The attacker does not need to fill the mempool themselves — they only need to call the RPC when the mempool is naturally large (e.g., during high network activity).
- No authentication, rate limiting, or cycle budget is applied to this RPC method.
- The attack is repeatable at negligible cost to the caller.

---

### Recommendation

1. **Remove or cap the `rank_in_pending` linear scan.** The rank can be computed in O(log N) using the existing score-sorted index, or simply omitted from this diagnostic endpoint.
2. **Cap `calc_descendants` and `calc_ancestors` calls** in `get_tx_detail` with an explicit upper bound (e.g., return early once a configurable limit is reached).
3. **Apply per-method rate limiting** to `get_pool_tx_detail_info` in the RPC layer, consistent with its stated "trouble shooting only" purpose.

---

### Proof of Concept

1. Fill the mempool with many small transactions (each paying minimum fee rate) until `pending_count` is large.
2. Repeatedly call:
   ```json
   {"jsonrpc":"2.0","method":"get_pool_tx_detail_info","params":["<any_pending_tx_hash>"],"id":1}
   ```
3. Observe that each call takes time proportional to the number of pending transactions, and that concurrent `send_transaction` RPC calls experience increased latency or queuing delays.

The root cause is in `get_tx_detail` at `tx-pool/src/pool.rs` lines 684–704, specifically the `get_ids()` + linear `.find()` pattern and the uncapped `calc_descendants`/`calc_ancestors` calls. [6](#0-5) [7](#0-6) [8](#0-7)

    pub(crate) fn get_tx_detail(&self, id: &ProposalShortId) -> Option<PoolTxDetailInfo> {
        if let Some(entry) = self.pool_map.get_by_id(id) {
            let ids = self.get_ids();
            let rank_in_pending = if entry.status == Status::Proposed {
                0
            } else {
                let tx_hash = entry.inner.transaction().hash();
                ids.pending
                    .iter()
                    .enumerate()
                    .find(|(_, hash)| &tx_hash == *hash)
                    .map(|r| r.0)
                    .unwrap_or_default()
                    + 1
            };
            let res = PoolTxDetailInfo {
                timestamp: entry.inner.timestamp,
                entry_status: entry.status.to_string(),
                pending_count: self.pool_map.pending_size(),
                rank_in_pending,
                proposed_count: ids.proposed.len(),

                ancestors_count: self.pool_map.calc_ancestors(id).len(),
                score_sortkey: entry.inner.as_score_key().into(),
            };
            Some(res)
        } else {
            None
        }
    }

    fn calc_relative_ids(
        &self,
        short_id: &ProposalShortId,
        relation: Relation,

        let direct = self
            .inner
            .get(short_id)

            .cloned()
            .unwrap_or_default();

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }
