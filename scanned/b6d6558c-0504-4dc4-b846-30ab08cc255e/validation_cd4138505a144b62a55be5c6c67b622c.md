### Title
Network Alert Signed Message Lacks Chain Identifier, Enabling Cross-Network Replay - (File: `util/network-alert/src/verifier.rs`)

### Summary
The `RawAlert` hash used for signature verification does not include any network or chain identifier. Because the same four hard-coded alert signing keys are compiled into every CKB binary (mainnet and testnet alike), a legitimately signed alert from one network can be replayed verbatim on another network by any unprivileged RPC caller or P2P peer, causing all nodes on the target network to accept and broadcast the foreign alert.

### Finding Description
`Verifier::verify_signatures()` derives the message to verify from `alert.calc_alert_hash()`:

```rust
// util/network-alert/src/verifier.rs
let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
```

`calc_alert_hash()` hashes only the serialized `RawAlert` molecule struct:

```rust
// util/gen-types/src/extension/calc_hash.rs
impl<'r> packed::RawAlertReader<'r> {
    pub fn calc_alert_hash(&self) -> packed::Byte32 {
        self.calc_hash()   // blake2b over RawAlert bytes only
    }
}
```

`RawAlert` contains: `id`, `cancel`, `min_version`, `max_version`, `priority`, `notice_until`, `message`. **No genesis hash, no network magic, no chain identifier of any kind is included.**

The trusted signing keys are hard-coded in a single embedded TOML file:

```toml
# util/app-config/src/configs/alert_signature.toml
signatures_threshold = 2
public_keys = [
  "0x03933a9b116c5017561742c37ae69acb0dca3329a52c479c85df5bb387c8ac8715",
  "0x038da23240e5a4234601902cf3db3cdfc1b3fdb2db2a54ba6204f6ca1d6ef6129a",
  "0x02adc94b64a9809019139fe70bd26aa0d787772a1ad645a4bcb1456fb3e1105f09",
  "0x0369eca725513fc94685cd0b8ccebc7be874afea38d23ccb090566aa1c50d696b1",
]
```

`NetworkAlertConfig::default()` loads this file at compile time via `include_bytes!`:

```rust
// util/app-config/src/configs/network_alert.rs
impl Default for Config {
    fn default() -> Self {
        let alert_config = include_bytes!("./alert_signature.toml");
        toml::from_slice(&alert_config[..]).expect("alert system config")
    }
}
```

Because the same binary is deployed on both mainnet and testnet, the **identical four public keys** are the trusted alert signers on every CKB network. A signed alert is therefore valid on every network simultaneously.

Once accepted, `AlertRelayer::received()` re-broadcasts the alert to all connected peers without any additional network-scoped check:

```rust
// util/network-alert/src/alert_relayer.rs
if let Err(err) = self.verifier.verify_signatures(&alert) { ... ban peer ... }
// broadcast message
nc.quick_filter_broadcast(TargetSession::Multi(...), data);
self.notifier.lock().add(&alert);
```

### Impact Explanation
An attacker who captures a legitimately signed testnet alert (e.g., "CKB testnet v0.x has a critical bug, upgrade immediately") can submit it to any mainnet node via the `send_alert` RPC. The mainnet node passes signature verification (same keys, same hash, no network binding), stores the alert, and floods it to every connected mainnet peer. Every mainnet node then displays the misleading message to operators and users. This can trigger mass panic, unnecessary emergency upgrades, or coordinated social-engineering follow-up attacks. The reverse direction (mainnet alert replayed on testnet) is equally possible.

### Likelihood Explanation
The Nervos foundation has already issued real testnet alerts (e.g., alert `20230001` is hardcoded in the test suite). Any observer connected to the testnet P2P network can capture such an alert in plaintext. Submitting it to a mainnet node requires only RPC access, which is available to any local user or any operator who exposes the Alert RPC module. No key material, no privileged network position, and no consensus power is required of the attacker.

### Recommendation
Include the genesis block hash (or a dedicated network identifier) in the data that is hashed and signed for each alert. For example, extend `RawAlert` with a `genesis_hash` field, or prepend the genesis hash to the data fed into `calc_alert_hash()`. Alternatively, maintain separate signing key sets per network so that a testnet-signed alert cannot satisfy mainnet's trusted-key set.

### Proof of Concept
1. Connect a node to CKB testnet and capture a live alert message from the P2P stream (or extract the bytes of alert `20230001` from the test vectors in `util/network-alert/src/tests/generate_alert_signature.rs`).
2. Confirm the alert passes `Verifier::verify_signatures()` against `NetworkAlertConfig::default()` — it does, because the same keys are embedded.
3. Call `send_alert` on a CKB **mainnet** node with the captured alert bytes.
4. Observe that the mainnet node accepts the alert (signature check passes), stores it in the notifier, and broadcasts it to all connected mainnet peers via `AlertRelayer`.
5. All reachable mainnet nodes now display the testnet alert message.
