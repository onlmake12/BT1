### Title
`ScriptHashTypeVerifier` Fails to Validate Type Script Hash Type Against Permitted Consensus Values - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that script hash types in transaction outputs are within the consensus-permitted range (`ENABLED_SCRIPT_HASH_TYPE`), but it only checks the **lock** script of each output. The **type** script hash type is never validated. Any transaction sender can craft an output whose type script uses a disallowed `ScriptHashType` value and bypass this consensus-level gate entirely.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, the `ScriptHashTypeVerifier` is documented as:

> "Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules."

Its `verify()` method iterates over all outputs and checks only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs, lines 796–815
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

The type script — accessed via `output.type_().to_opt()` — is never inspected. There is no corresponding check for `output.type_().hash_type()` against `ENABLED_SCRIPT_HASH_TYPE`.

This is structurally identical to the HiFi bug: a permission flag (`isRedeemHTokenAllowed` / `ENABLED_SCRIPT_HASH_TYPE`) exists and is consulted for one code path (lock script) but silently skipped for another (type script), leaving the gate half-open.

`ScriptHashTypeVerifier` is composed into `NonContextualTransactionVerifier`:

```rust
// verification/src/transaction_verifier.rs, lines 71–103
pub struct NonContextualTransactionVerifier<'a> {
    ...
    pub(crate) script_hash_type: ScriptHashTypeVerifier<'a>,
}

pub fn verify(&self) -> Result<(), Error> {
    ...
    self.script_hash_type.verify()?;
    Ok(())
}
```

`NonContextualTransactionVerifier` is invoked in the tx-pool admission path via `non_contextual_verify` in `tx-pool/src/util.rs`, which is called by `TxPoolService::non_contextual_verify()` before any transaction is accepted — both from RPC (`process_tx`) and from P2P relay (`submit_remote_tx`).

---

### Impact Explanation

An unprivileged transaction sender (RPC caller or P2P relay peer) can submit a transaction whose output contains a type script with a hash type value outside `ENABLED_SCRIPT_HASH_TYPE`. This bypasses the consensus-level restriction that `ScriptHashTypeVerifier` is meant to enforce. Such a transaction passes non-contextual verification, enters the tx-pool, and can be included in a block — violating the consensus rule that gates which script hash types are permitted at a given epoch/hardfork boundary.

---

### Likelihood Explanation

The entry path is fully unprivileged: any node reachable via RPC or P2P can submit such a transaction. Constructing a transaction with a type script using an arbitrary `hash_type` byte requires no special access. The missing check is in the non-contextual (stateless) verifier, so it is exercised on every submitted transaction.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the hash type of the type script when present:

```rust
for output in self.transaction.outputs() {
    // existing lock script check
    check_hash_type(output.lock().hash_type())?;
    // missing type script check
    if let Some(type_script) = output.type_().to_opt() {
        check_hash_type(type_script.hash_type())?;
    }
}
```

where `check_hash_type` encapsulates the existing `ENABLED_SCRIPT_HASH_TYPE` membership test and error return.

---

### Proof of Concept

1. Craft a `TransactionView` with one output whose type script has `hash_type` set to a byte value not in `ENABLED_SCRIPT_HASH_TYPE` (e.g., a reserved future value).
2. Submit via RPC `send_transaction` or relay via P2P.
3. `NonContextualTransactionVerifier::verify()` calls `self.script_hash_type.verify()`.
4. `ScriptHashTypeVerifier::verify()` iterates outputs, checks only `output.lock().hash_type()` — the lock script uses a valid hash type, so no error is returned.
5. The type script's disallowed hash type is never inspected; the transaction passes non-contextual verification and proceeds into the tx-pool. [1](#0-0) [2](#0-1) [3](#0-2)
