### Title
Copy-paste bug in `script_exists_in_output` causes incorrect script deletion on PostgreSQL during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

`script_exists_in_output` in `remove.rs` contains a copy-paste error at line 252: the final `match` branch reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, this causes the function to always return `false` for the type-script check, meaning any script that is referenced only as a `type_script_id` (not as a `lock_script_id`) in surviving outputs will be incorrectly deleted during rollback.

### Finding Description

`script_exists_in_output` performs two `SELECT EXISTS` queries — one for `lock_script_id` and one for `type_script_id` — and returns `true` if either finds a row: [1](#0-0) 

The first half is correct. After the `row_type` query is executed, the final `match` at line 252 mistakenly re-reads `row_lock` instead of `row_type`: [2](#0-1) 

**PostgreSQL path:** `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL returns `BOOLEAN`). If the script was not found as a lock script, `row_lock` holds `false`. The final `match` re-evaluates `row_lock` — still `false` — and returns `Ok(false)`, ignoring `row_type` entirely. A script referenced only as a `type_script_id` in surviving outputs is therefore always reported as unreferenced.

**SQLite path:** `row_lock.try_get::<bool, _>(0)` returns `Err` (SQLite returns `BIGINT`), so the `Err` branch executes `Ok(row_type.get::<i64, _>(0) == 1)`, which correctly reads `row_type`. The bug does **not** manifest on SQLite.

`rollback_block` calls `script_exists_in_output` after deleting the rolled-back block's outputs, and deletes any script the function reports as unreferenced: [3](#0-2) 

`bulk_insert_script_table` uses `ON CONFLICT DO NOTHING` so a script shared across two blocks is stored only once under a single `id`: [4](#0-3) 

### Impact Explanation

On PostgreSQL, during any rollback where a rolled-back block shares a `type_script` with a surviving block's outputs, `script_exists_in_output` returns `false` for that script. The script row is deleted. The surviving outputs retain their `type_script_id` foreign key pointing to the now-deleted row. All subsequent RPC queries that join `output` to `script` on `type_script_id` return no results for those cells, permanently corrupting the indexer's live-cell view for that script until a full re-index.

### Likelihood Explanation

Chain reorgs are a normal part of CKB operation and require no attacker. Any 1-block reorg where two consecutive blocks contain outputs sharing a type script (e.g., any UDT/xUDT token script, which appears in many consecutive blocks) on a PostgreSQL-backed node triggers the corruption. No special privileges, hashpower majority, or key material are required.

### Recommendation

Change line 252 in `remove.rs` from:

```rust
match row_lock.try_get::<bool, _>(0) {
```

to:

```rust
match row_type.try_get::<bool, _>(0) {
```

Alternatively, unify the two queries into a single `SELECT EXISTS (... WHERE lock_script_id = $1 OR type_script_id = $1)` to eliminate the duplication entirely.

### Proof of Concept

1. Run a CKB rich-indexer node backed by PostgreSQL.
2. Append block 1 containing a transaction whose output has `type_script = S`.
3. Append block 2 containing a transaction whose output also has `type_script = S`. `bulk_insert_script_table` silently skips the duplicate via `ON CONFLICT DO NOTHING`; both outputs share the same `script.id`.
4. Call `rollback()`. `rollback_block` deletes block 2's outputs, then calls `script_exists_in_output(S.id)`. The lock-script query returns `false` (S is not a lock script). The type-script query returns `true` (block 1's output still references S), but the final `match` re-reads `row_lock` (still `false`) and returns `false`. Script S is deleted.
5. Query the indexer for live cells with `type_script = S`. The result is empty, even though block 1's output is unspent and should be returned.

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
