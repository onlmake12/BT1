### Title
`DaoScriptSizeVerifier` Bypassed via Index Mismatch in DAO Phase-1 Transactions — (`File: verification/src/transaction_verifier.rs`)

### Summary

`DaoScriptSizeVerifier` iterates over `(resolved_inputs[i], outputs[i])` pairs using `zip`. It assumes DAO deposit inputs and their corresponding DAO withdrawal outputs occupy the **same index**. This assumption is never enforced. A transaction sender can place the DAO deposit input at index `i` and the DAO withdrawal output at index `j ≠ i`, causing the lock-script-size check to be silently skipped. Because the DAO type script itself cannot verify lock script sizes (the verifier's own comment calls it "a temporary solution till Nervos DAO script can be properly upgraded"), the bypass is undetected on-chain, allowing the attacker to use a smaller lock script in the withdrawal cell and extract more interest than entitled.

---

### Finding Description

`DaoScriptSizeVerifier::verify()` zips `resolved_inputs` with `transaction.outputs()` positionally:

```rust
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;                    // ← silently skipped
    }
    if input_data.into_iter().any(|b| b != 0) { continue; }
    ...
    if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
        return Err(...DaoLockSizeMismatch...);
    }
}
``` [1](#0-0) 

The check fires only when **both** `input[i]` and `output[i]` carry the DAO type script. If an attacker crafts a phase-1 transaction as:

| Index | Input | Output |
|-------|-------|--------|
| 0 | regular cell | DAO withdrawal cell (small lock) |
| 1 | DAO deposit cell (data = 0x00…) | regular change cell |

then at `i=0` the input is non-DAO → `continue`; at `i=1` the output is non-DAO → `continue`. The size check never executes, even though a DAO deposit cell and a DAO withdrawal cell are present in the same transaction.

The `DaoScriptSizeVerifier` comment explicitly acknowledges the DAO type script cannot enforce this constraint: [2](#0-1) 

The interest calculation in `DaoCalculator::calculate_maximum_withdraw` uses the **withdrawal cell's** `occupied_capacity`:

```rust
let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
let output_capacity: Capacity = output.capacity().into();
let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
``` [3](#0-2) 

A smaller lock script in the withdrawal cell → smaller `occupied_capacity` → larger `counted_capacity` → larger `withdraw_counted_capacity` → more CKB extracted than deposited.

---

### Impact Explanation

An unprivileged transaction sender can:
1. Deposit CKB into the DAO with a large lock script.
2. Submit a phase-1 withdrawal transaction with the DAO deposit input at index 1 and a DAO withdrawal output (carrying a **smaller** lock script) at index 0.
3. `DaoScriptSizeVerifier` skips the size check; the DAO type script does not enforce it.
4. In phase 2, `calculate_maximum_withdraw` computes interest on a larger `counted_capacity` than the deposit entitled, minting excess CKB.

This is a **capacity inflation / incorrect interest calculation** bug reachable by any tx-pool submitter or block-template caller.

---

### Likelihood Explanation

- No privileged access required; any wallet or script author can construct the transaction.
- The CKB protocol imposes no ordering constraint on inputs/outputs relative to DAO cells.
- The bypass requires only reordering inputs and outputs, which is trivially achievable.
- The `DaoScriptSizeVerifier` is applied in both the tx-pool admission path (`tx-pool/src/util.rs`) and the contextual block verifier (`verification/contextual/src/contextual_block_verifier.rs`), so the bypass affects both layers. [4](#0-3) [5](#0-4) 

---

### Recommendation

Replace the positional `zip` with an explicit pairing that matches each DAO deposit input to its corresponding DAO withdrawal output regardless of index. Concretely, collect all DAO deposit inputs and all DAO withdrawal outputs separately, then enforce a bijection between them (e.g., by matching on the cell's out-point or by requiring the transaction to declare the pairing explicitly). Alternatively, enforce that DAO deposit inputs and their corresponding withdrawal outputs must occupy the same index, and reject any transaction that violates this ordering.

---

### Proof of Concept

Construct a phase-1 DAO transaction:

```
inputs:
  [0] regular cell (lock: secp256k1, capacity: 100 CKB)
  [1] DAO deposit cell (lock: secp256k1 large, data: 0x0000000000000000, capacity: 1000 CKB)

outputs:
  [0] DAO withdrawal cell (lock: secp256k1 SMALL — fewer bytes, capacity: 1000 CKB)
  [1] regular change cell (capacity: ~100 CKB minus fee)
```

`DaoScriptSizeVerifier` iterates:
- `i=0`: `input[0]` is non-DAO → `continue`
- `i=1`: `output[1]` is non-DAO → `continue`

Lock-script size mismatch between `input[1]` (large) and `output[0]` (small) is **never checked**.

In phase 2, `calculate_maximum_withdraw` is called with `output[0]` (the withdrawal cell). Its smaller lock script reduces `occupied_capacity`, inflating `counted_capacity` and thus the interest payout beyond what the original deposit entitled. [6](#0-5) [7](#0-6)

    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
