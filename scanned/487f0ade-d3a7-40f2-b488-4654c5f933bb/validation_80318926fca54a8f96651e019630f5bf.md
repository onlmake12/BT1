### Title
Inconsistent `script_len_range` Upper-Bound Operator in `get_cells_capacity` Returns Incorrect Capacity — (`File: util/indexer/src/service.rs`)

---

### Summary

In `util/indexer/src/service.rs`, the `get_cells_capacity` function applies the `script_len_range` filter using `> r1` (inclusive upper bound), while the sibling `get_cells` function correctly uses `>= r1` (exclusive upper bound). This is a copy-paste typographical error — the wrong comparison operator — directly analogous to the wrong string value in `INDEXER_ROLE`. The result is that `get_cells_capacity` includes cells whose script length equals the range's upper bound, while `get_cells` excludes them, making the two APIs inconsistent for any caller supplying a `script_len_range` filter.

---

### Finding Description

The `IndexerRange` type documents its semantics as `[inclusive, exclusive]`: [1](#0-0) 

The RPC documentation repeats this contract: [2](#0-1) 

`get_cells` correctly implements the exclusive upper bound for `script_len_range`: [3](#0-2) 

`get_cells_capacity` uses `> r1` (inclusive) instead of `>= r1` (exclusive): [4](#0-3) 

The divergence is on lines 780 and 790 (`script_len > r1`) versus the correct `script_len >= r1` used at lines 303 and 313. Every other range filter in `get_cells_capacity` — `output_data_len_range`, `output_capacity_range`, `block_range` — correctly uses `>=` for the exclusive upper bound: [5](#0-4) 

---

### Impact Explanation

Any RPC caller that queries `get_cells_capacity` with a `script_len_range` filter receives a capacity total that includes cells whose script length equals exactly `r1`. The same caller querying `get_cells` with the identical filter receives a result set that excludes those cells. The two APIs are supposed to be consistent: `get_cells_capacity` is the aggregate of the capacity of cells returned by `get_cells`. The inconsistency means:

- Applications (wallets, dApps, tooling) that use `get_cells_capacity` to determine available capacity before constructing a transaction will receive an **inflated** capacity figure.
- If the application then calls `get_cells` to collect the actual cells, it will find fewer cells than expected, potentially causing transaction construction to fail or to use incorrect inputs.
- The discrepancy is silent — no error is returned; the wrong number is returned as if it were correct.

---

### Likelihood Explanation

The entry path is a standard unprivileged JSON-RPC call to `get_cells_capacity` with a `script_len_range` filter. No authentication, no special role, no privileged key is required. Any RPC client — wallet, block explorer, dApp backend — that uses this filter hits the bug deterministically. The filter is a documented, supported parameter. [6](#0-5) 

---

### Recommendation

Change the two `> r1` comparisons in `get_cells_capacity`'s `script_len_range` block to `>= r1`, matching the exclusive-upper-bound contract and the implementation in `get_cells`:

```rust
// Line 780 — change:
if script_len < r0 || script_len > r1 {
// to:
if script_len < r0 || script_len >= r1 {

// Line 790 — change:
if script_len < r0 || script_len > r1 {
// to:
if script_len < r0 || script_len >= r1 {
``` [7](#0-6) 

---

### Proof of Concept

1. Index a set of cells where some cells have a type script whose serialized length is exactly `N` bytes.
2. Call `get_cells` with `filter.script_len_range = [0, N]` (exclusive upper bound means length `N` is excluded).
3. Call `get_cells_capacity` with the same `filter.script_len_range = [0, N]`.
4. Observe that `get_cells_capacity` returns a capacity that includes the cells with script length `N`, while `get_cells` does not return those cells.
5. The capacity reported by `get_cells_capacity` is larger than the sum of capacities of cells returned by `get_cells` — a direct contradiction of the documented `[inclusive, exclusive]` semantics. [8](#0-7)
