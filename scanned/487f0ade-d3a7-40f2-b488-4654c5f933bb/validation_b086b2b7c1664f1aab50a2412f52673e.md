### Title
Incomplete ScriptHashType Validation in `ScriptHashTypeVerifier` — Type Script `hash_type` Not Checked Against Enabled Consensus Set - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that output script `hash_type` values are within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`), but it only inspects each output's **lock script**. The **type script** `hash_type` is never validated. This is a direct structural analog to the reported bug: the wrong entity is checked (lock script) while the entity that should also be checked (type script) is silently skipped.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, the `ScriptHashTypeVerifier` is documented as:

> "Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules."

Its `verify()` implementation iterates over all outputs and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The call `output.lock().hash_type()` retrieves only the lock script. The type script — accessed via `output.type_().to_opt()` — is never consulted. A transaction output carrying a type script with a disallowed `hash_type` (e.g., `ScriptHashType::Data3`, confirmed non-enabled by the existing test `test_not_enabled_hash_type_output_lock`) passes this verifier without error. [2](#0-1) 

The `NonContextualTransactionVerifier` composes this verifier and is invoked for every transaction entering the node, both via the RPC path and during block validation: [3](#0-2) 

`NonContextualBlockTxsVerifier` also calls it for every block transaction: [4](#0-3) 

The only other hash-type check in the pipeline is `check_data()`, which only validates that the raw byte is a recognized enum discriminant — it does **not** enforce the `ENABLED_SCRIPT_HASH_TYPE` consensus restriction: [5](#0-4) 

---

### Impact Explanation

`ENABLED_SCRIPT_HASH_TYPE` is a consensus-level gate that controls which `hash_type` values are active in the current protocol version. Its purpose is to prevent transactions that use not-yet-activated script semantics from entering the chain, avoiding consensus splits between nodes running different software versions.

Because the type script `hash_type` is never checked against this set, an attacker can submit a transaction whose output carries a type script with a disallowed `hash_type` (e.g., `Data3`). Such a transaction:

1. Passes `ScriptHashTypeVerifier` (lock script is valid; type script is unchecked).
2. Passes `check_data()` (the byte value is a valid enum discriminant).
3. Proceeds to script execution, where behavior diverges across node versions: nodes whose VM supports the new hash type may accept the transaction; nodes that do not will reject it.

This creates a **consensus split** — a class of transactions that some honest nodes accept and others reject — reachable by any unprivileged transaction sender.

---

### Likelihood Explanation

The entry path requires only a standard RPC call to `send_transaction` with a crafted transaction. No privileged access, key material, or majority hashpower is needed. The `ScriptHashType` enum already defines `Data3` as a named variant (confirmed by the test at line 108 of `verification/src/tests/transaction_verifier.rs`), so constructing the malformed transaction requires no reverse engineering. [6](#0-5) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Check lock script hash_type
        let lock_hash_type = output.lock().hash_type();
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(lock_hash_type) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType { hash_type: lock_hash_type }.into());
        }

        // Check type script hash_type (currently missing)
        if let Some(type_script) = output.type_().to_opt() {
            let type_hash_type = type_script.hash_type();
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_hash_type) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType { hash_type: type_hash_type }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Construct a transaction with one output whose **lock script** uses `ScriptHashType::Type` (enabled) and whose **type script** uses `ScriptHashType::Data3` (not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC.
3. `ScriptHashTypeVerifier::verify()` iterates outputs, calls `output.lock().hash_type()` → `Type` → passes. The type script is never inspected.
4. The transaction clears `NonContextualTransactionVerifier` and enters the tx pool / block validation pipeline with a consensus-disallowed type script hash type, producing divergent behavior across node versions.
