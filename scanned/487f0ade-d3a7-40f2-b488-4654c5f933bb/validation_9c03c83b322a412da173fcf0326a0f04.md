### Title
`script_exists_in_output` re-reads `row_lock` instead of `row_type` on PostgreSQL, causing premature deletion of still-referenced type scripts during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

In `script_exists_in_output`, the second `match` block at line 252 erroneously uses `row_lock.try_get` instead of `row_type.try_get`. On PostgreSQL this silently discards the `type_script_id` existence check, causing `rollback_block` to delete `script` table rows that are still referenced by surviving outputs' `type_script_id` columns.

### Finding Description

`rollback_block` calls `script_exists_in_output(script_id, tx)` for every lock and type script of the rolled-back outputs to decide whether to delete the script row. The function runs two SQL `EXISTS` queries:

1. `SELECT EXISTS (SELECT 1 FROM output WHERE lock_script_id = $1)` → `row_lock`
2. `SELECT EXISTS (SELECT 1 FROM output WHERE type_script_id = $1)` → `row_type`

The intended logic is: return `true` if either query returns true. The first `match` (line 223) correctly short-circuits on `row_lock`. The second `match` (line 252) is supposed to evaluate `row_type`, but reads `row_lock` again: [1](#0-0) 

On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds, so `row_type` is never consulted. The function therefore returns the lock-check result for both checks. If a `script_id` is used only as a `type_script_id` in surviving outputs (not as any `lock_script_id`), the function returns `false`, and the script is pushed into `script_id_list_to_remove` and deleted. [2](#0-1) 

### Impact Explanation

After the erroneous deletion, the `script` table no longer contains the row for the shared type script. Any subsequent indexer query that joins `output` with `script` on `type_script_id` — including `get_cells` and `get_cells_capacity` — will silently drop or fail to return cells whose type script was deleted. This corrupts the indexer's live-cell view for all users of the PostgreSQL-backed rich-indexer.

### Likelihood Explanation

Reorgs are a normal part of CKB network operation and are triggered via standard P2P block relay (an explicitly listed entry point). The precondition — two blocks sharing a type script where the second is rolled back — is trivially satisfied by any token/UDT deployment (e.g., sUDT, xUDT), where the same type script appears across many outputs in many blocks. No special attacker capability beyond sending a valid competing chain tip is required; even a natural 1-block reorg suffices. The bug is silent: no error is returned, the deletion succeeds, and the corruption is only discovered when queries return wrong results.

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// Correct fix
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

### Proof of Concept

State-transition test (PostgreSQL):
1. Index Block 1 containing output O1 with `type_script_id = S`.
2. Index Block 2 containing output O2 with `type_script_id = S` (same script row, same `id`).
3. Call `rollback_block` to remove Block 2.
4. Assert: `SELECT COUNT(*) FROM script WHERE id = S` returns `1` — the script must still exist because O1 in Block 1 still references it.
5. On the buggy code (PostgreSQL), the assertion fails: the row is gone, and subsequent `get_cells` for O1 returns no type-script information. [3](#0-2)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
