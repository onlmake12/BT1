### Title
Stale Descendant Statistics in `remove_entry_and_descendants` Due to Premature Link Clearing - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all parent/child links for the removed entries are cleared **before** `remove_entry` is called on each entry. Because `update_ancestors_index_key` resolves ancestors through those same links, it finds an empty ancestor set and never decrements the `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` fields of ancestor entries that remain in the pool. This is a direct analog to the ERC721Votes delegation-to-address(0) bug: in both cases, a "removal" operation fails to update a related counter because the lookup used to find the target of the update returns a null/empty result, leaving the counter permanently inflated.

---

### Finding Description

`remove_entry_and_descendants` is the central removal path used by conflict resolution, RBF, and size-limit eviction. Its implementation is:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← ALL links cleared here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links are gone
        .collect()
}
``` [1](#0-0) 

Inside `remove_entry`, the first thing that runs is:

```rust
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
``` [2](#0-1) 

`update_ancestors_index_key` resolves the ancestor set by calling `self.links.calc_ancestors(...)`. Because all links were already cleared in the pre-pass, this returns an empty set, and the inner loop that calls `sub_descendant_weight` on each ancestor never executes:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id()); // ← always empty after pre-clear
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child), // ← never reached
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [3](#0-2) 

`sub_descendant_weight` is the function that would have decremented the ancestor's `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee`:

```rust
pub fn sub_descendant_weight(&mut self, entry: &TxEntry) {
    self.descendants_count = self.descendants_count.saturating_sub(1);
    self.descendants_size  = self.descendants_size.saturating_sub(entry.size);
    self.descendants_cycles = self.descendants_cycles.saturating_sub(entry.cycles);
    self.descendants_fee   = Capacity::shannons(
        self.descendants_fee.as_u64().saturating_sub(entry.fee.as_u64()),
    );
}
``` [4](#0-3) 

The comment in `remove_entry_and_descendants` acknowledges the pre-clearing is intentional to suppress `update_descendants_index_key` (since the descendants are being removed anyway), but it silently also suppresses `update_ancestors_index_key`, which is the bug.

---

### Impact Explanation

After `remove_entry_and_descendants` returns, every ancestor of the removed cluster that **remains in the pool** retains permanently inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`. These fields directly feed into `EvictKey`:

```rust
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),
            ...
            descendants_count: entry.descendants_count,
        }
    }
}
``` [5](#0-4) 

A transaction with inflated `descendants_fee` will appear to have a higher effective fee rate than it actually does, making it **harder to evict** than it should be. The pool's eviction ordering (`next_evict_entry`) is therefore corrupted for any ancestor of a removed cluster. [6](#0-5) 

The `total_tx_size` and `total_tx_cycles` pool-level counters are correctly decremented (they use a separate path), so the pool size limit still functions. However, the per-entry descendant statistics used for eviction priority are permanently wrong until the ancestor itself is eventually removed.

---

### Likelihood Explanation

This is triggered by any code path that calls `remove_entry_and_descendants` when the removed cluster has ancestors that remain in the pool. The three main callers are:

1. **`resolve_conflict`** — triggered whenever a newly submitted transaction spends an input already consumed by a pool transaction. Any unprivileged RPC caller submitting a conflicting transaction triggers this.
2. **`resolve_conflict_header_dep`** — triggered on block arrival when a pool transaction's header dep is invalidated.
3. **`limit_size`** — triggered when the pool exceeds `max_tx_pool_size`. [7](#0-6) 

The attacker-controlled entry path is straightforward: submit `tx_parent` (low fee) then `tx_child` (high fee, spending `tx_parent`'s output), then submit `tx_conflict` that spends the same input as `tx_child`. `tx_child` is removed via `resolve_conflict` → `remove_entry_and_descendants`, but `tx_parent`'s `descendants_fee` is never decremented. `tx_parent` now appears to have a high-fee descendant that no longer exists, making it resistant to eviction.

---

### Recommendation

Before clearing links in `remove_entry_and_descendants`, compute the set of ancestors of the root entry that are **not** in `removed_ids`. After all entries are removed, iterate over those surviving ancestors and call `sub_descendant_weight` for each removed entry that was a descendant of that ancestor. Alternatively, restructure `remove_entry` so that `update_ancestors_index_key` is called **before** `remove_entry_links`, using a snapshot of the ancestor set taken prior to link removal.

---

### Proof of Concept

1. Submit `tx1` (low fee, e.g. 1 shannon) to the pending pool.
2. Submit `tx2` (high fee, e.g. 10,000 shannons) spending `tx1`'s output. Now `tx1.descendants_fee = 10,001`, `tx1.descendants_count = 2`.
3. Submit `tx3` that spends the same input as `tx2` (conflict). `resolve_conflict` calls `remove_entry_and_descendants(&tx2_id)`.
4. After step 3: `tx2` is gone, but `tx1.descendants_fee` is still `10,001` and `tx1.descendants_count` is still `2`.
5. Query `get_pool_tx_detail_info(tx1_hash)` via RPC — `descendants_count` reports `2` despite `tx1` having no descendants.
6. Fill the pool to trigger `limit_size`. `tx1` is not evicted because its `EvictKey.fee_rate` is computed from the ghost `descendants_fee`, making it appear more valuable than its actual 1-shannon fee warrants. [1](#0-0) [3](#0-2) [4](#0-3)
