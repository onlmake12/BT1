### Title
Unbounded `pending_get_block_proposals` Map Inflation via Unauthenticated `GetBlockProposal` Relay Messages — (`sync/src/types/mod.rs`)

---

### Summary

Any connected peer can repeatedly send `GetBlockProposal` relay messages containing up to 3,000 unique, non-existent proposal short IDs per message. Each message unconditionally inserts all non-found IDs into the global `pending_get_block_proposals` `DashMap` with no size cap. Over time this map grows without bound, causing memory exhaustion and O(n) CPU spikes every time the periodic `prune_tx_proposal_request` timer fires and clones the entire map.

---

### Finding Description

`SyncState` holds a shared, process-wide map:

```
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>
``` [1](#0-0) 

It is initialised with no capacity limit: [2](#0-1) 

The only write path is `insert_get_block_proposals`, which performs a plain `entry().or_default().insert()` loop with **no size guard**: [3](#0-2) 

This function is called from `GetBlockProposalProcess::execute` for every proposal ID that is not currently in the tx-pool: [4](#0-3) 

The only validation applied to an incoming `GetBlockProposal` message is a per-message proposals-count ceiling (`max_block_proposals_limit × max_uncles_num = 1 500 × 2 = 3 000`) and a duplicate-within-message check: [5](#0-4) 

There is **no rate limit per peer**, **no cumulative cap** on `pending_get_block_proposals`, and **no PoW or stake requirement** to send this message. An attacker therefore sends an unbounded stream of `GetBlockProposal` messages, each carrying 3,000 fresh random `ProposalShortId` values (10 bytes each) that are absent from the tx-pool, inflating the map by 3,000 entries per message.

The map is consumed by `drain_get_block_proposals`, which **clones the entire DashMap** before clearing it: [6](#0-5) 

This clone is O(n) in both time and memory. The cloned map is then passed to `prune_tx_proposal_request`, which iterates every entry and issues a `fetch_txs` call to the tx-pool controller with all accumulated IDs: [7](#0-6) 

---

### Impact Explanation

**Memory exhaustion**: Each `ProposalShortId` entry occupies ~10 bytes of key plus `HashSet<PeerIndex>` overhead. At 3,000 entries per message and a modest 100 messages/second, the map reaches ~1.8 million entries (~tens of MB) within minutes. Sustained attack drives the node OOM.

**CPU / lock contention**: Every timer tick, `drain_get_block_proposals` acquires internal DashMap shard locks to clone millions of entries, then `prune_tx_proposal_request` iterates all of them and submits a single `fetch_txs` request with millions of IDs to the tx-pool actor. This stalls the relay timer loop and the tx-pool service, degrading block propagation and transaction relay for all peers.

**Impact: 4/5** — sustained attack can crash the node or make it unable to relay blocks/transactions, equivalent to a remote DoS against any non-rate-limited CKB full node.

---

### Likelihood Explanation

**Likelihood: 3/5** — the attacker needs only a single TCP connection to a CKB node (no key, no stake, no PoW). `GetBlockProposal` is a standard relay protocol message. The only friction is the 3,000-proposals-per-message cap, which is easily saturated in a tight loop. A single malicious peer is sufficient; Sybil amplification is not required.

---

### Recommendation

1. **Hard cap on `pending_get_block_proposals`**: Reject `insert_get_block_proposals` (or drop the oldest entries) once the map exceeds a configurable maximum (e.g., `MAX_BLOCK_PROPOSALS_LIMIT × some_small_constant`).
2. **Per-peer quota**: Track how many pending proposal IDs each peer has contributed and reject further insertions once a per-peer ceiling is reached, analogous to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` used for `unknown_tx_hashes`.
3. **Rate-limit `GetBlockProposal` messages per peer** at the protocol handler level (e.g., ban or throttle peers that exceed N messages per second).
4. **Avoid full clone in `drain_get_block_proposals`**: Use `std::mem::take` or swap with an empty map to avoid the O(n) clone under lock.

---

### Proof of Concept

```
attacker_peer connects to victim CKB node via Tentacle P2P

loop forever:
    proposals = [random_10_byte_id() for _ in range(3000)]   # all absent from tx-pool
    send RelayMessage::GetBlockProposal {
        block_hash: any_valid_hash,
        proposals:  proposals,
    }
```

Each iteration:
- Passes the `message_len <= 3000` check in `GetBlockProposalProcess::execute` [8](#0-7) 
- `fetch_txs` returns empty (IDs are random / absent)
- All 3,000 IDs are inserted into `pending_get_block_proposals` with no guard [3](#0-2) 

After N iterations the map holds `3000 × N` entries. The next `prune_tx_proposal_request` tick clones and iterates all of them, causing O(N) memory allocation and CPU work while holding DashMap shard locks, stalling relay processing for all peers. [9](#0-8)

    async fn prune_tx_proposal_request(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());
                }
            }
        }

        let mut relay_bytes = 0;
        let mut relay_proposals = Vec::new();
        for (peer_index, txs) in peer_txs {
            for tx in txs {
                let data = tx.data();
                let tx_size = data.total_size();
                if relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH {
                    send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals))
                        .await;
                    relay_bytes = tx_size;
                } else {
                    relay_bytes += tx_size;
                }
                relay_proposals.push(data);
            }
            if !relay_proposals.is_empty() {
                send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals)).await;
                relay_bytes = 0;
            }
        }
