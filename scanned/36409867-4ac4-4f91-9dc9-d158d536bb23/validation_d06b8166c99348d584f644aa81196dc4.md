### Title
`block_status_map` Grows Unboundedly with `BLOCK_INVALID` Entries from Unauthenticated Peer Headers — (`sync/src/synchronizer/headers_process.rs`)

---

### Summary

The in-memory `block_status_map` (`DashMap<Byte32, BlockStatus>`) accumulates `BLOCK_INVALID` entries indefinitely. In `headers_process.rs`, a header that fails `prev_block_check` (unknown parent) is immediately stamped `BLOCK_INVALID` **before** proof-of-work is verified. Those entries are never evicted. Any connected peer can flood the node with unique headers carrying random parent hashes — no PoW required — causing unbounded memory growth and eventual node crash.

---

### Finding Description

**Root cause — insertion without removal:**

`HeadersProcess::accept()` processes each header in a `SendHeaders` message in this order:

1. Early-return only if status already contains `HEADER_VALID`.
2. `prev_block_check` — verifies the parent is known.
3. `non_contextual_check` — verifies PoW and other non-contextual rules.
4. `version_check`. [1](#0-0) 

If `prev_block_check` fails (step 2), the code immediately calls `insert_block_status(..., BLOCK_INVALID)` and returns — **before** PoW is ever checked:

```rust
if self.prev_block_check(&mut result).is_err() {
    shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
    return result;
}
``` [2](#0-1) 

`insert_block_status` unconditionally inserts into the shared `DashMap`:

```rust
pub fn insert_block_status(&self, block_hash: Byte32, status: BlockStatus) {
    self.block_status_map.insert(block_hash, status);
}
``` [3](#0-2) 

**`BLOCK_INVALID` entries are never removed.** `remove_block_status` is called only on two paths:

- Successful contextual verification (`Ok` branch in `consume_unverified_blocks`).
- Internal DB error. [4](#0-3) 

Neither path is ever reached for a header that was rejected at `prev_block_check`. The `shrink_to_fit!` macro inside `remove_block_status` is therefore also never triggered for these entries:

```rust
pub fn remove_block_status(&self, block_hash: &Byte32) {
    self.block_status_map.remove(block_hash);
    shrink_to_fit!(self.block_status_map, SHRINK_THRESHOLD);
}
``` [5](#0-4) 

The same unbounded-insertion pattern exists in `compact_block_process.rs` (header verifier failure path) and `orphan_broker.rs` (parent-invalid propagation), all writing `BLOCK_INVALID` with no corresponding removal: [6](#0-5) [7](#0-6) 

**The early-return guard does not protect against unique hashes.** The only early-return at the top of `accept()` is for `HEADER_VALID`; `BLOCK_INVALID` is not checked, so each unique header hash causes a fresh map insertion: [8](#0-7) 

**`block_status_map` is consulted on every block/header event.** Every compact-block relay, every header sync step, and every orphan-block resolution calls `get_block_status`, which reads from this map: [9](#0-8) 

As the map grows into millions of entries, DashMap's internal shard operations (rehash, memory allocation, cache pressure) degrade all callers proportionally — analogous to the Union Finance `withdrawSeq` iteration cost growing with each adapter cycle.

---

### Impact Explanation

An attacker who maintains a single peer connection can stream `SendHeaders` messages containing batches of unique headers with random 32-byte parent hashes. Each header costs the attacker nothing (no PoW), but costs the victim node one permanent `DashMap` entry. Over time:

- **Memory exhaustion**: each entry is ~64 bytes (two `Byte32` values plus map overhead); 10 million entries ≈ ~640 MB of map data alone.
- **Performance degradation**: DashMap rehashing and increased cache-miss rates slow every `get_block_status` call, which is on the hot path for sync, relay, and block assembly.
- **Node crash / OOM kill**: the process is terminated by the OS, causing a full node outage.

---

### Likelihood Explanation

- Requires only a standard peer connection — no privileged access, no PoW, no key material.
- The `SendHeaders` message is a normal part of the sync protocol; no special message type is needed.
- No per-peer rate limit on the number of headers that can be sent is visible in the production sync code.
- The attack is persistent across reconnections because the map is never cleaned.

---

### Recommendation

1. **Add a `BLOCK_INVALID` guard at the top of `accept()`** — if the hash is already `BLOCK_INVALID`, return immediately without re-inserting. This prevents re-processing but does not fix the growth.

2. **Bound `block_status_map` size** — introduce a maximum capacity (e.g., 1 million entries) with LRU or FIFO eviction for `BLOCK_INVALID` entries, similar to how `OrphanBlockPool` uses `EXPIRED_EPOCH` expiry.

3. **Do not mark headers `BLOCK_INVALID` when only the parent is unknown** — an unknown parent means the header is an orphan candidate, not provably invalid. Reserve `BLOCK_INVALID` for headers that fail PoW or other non-contextual checks. This is the most semantically correct fix and directly mirrors the Union Finance recommendation of aligning the invariant check with the correct reference collection.

4. **Periodic cleanup** — add a background task (analogous to `clean_expired_orphans`) that evicts `BLOCK_INVALID` entries older than a configurable TTL.

---

### Proof of Concept

```
Attacker peer loop:
  for i in 0..N:
    header = craft_header(
        parent_hash = random_bytes_32(),   // unknown parent, no PoW needed
        nonce       = i,                   // unique hash per iteration
    )
    send SendHeaders([header]) to victim node

// Each iteration:
//   headers_process::accept() -> prev_block_check fails (parent unknown)
//   -> insert_block_status(header.hash(), BLOCK_INVALID)
//   -> entry never removed
//   -> block_status_map grows by 1 entry

// After N iterations: block_status_map has N permanent BLOCK_INVALID entries.
// N = ~10_000_000 exhausts ~640 MB of map memory on a typical node.
``` [2](#0-1) [3](#0-2)

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

    pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
        match self.block_status_map().get(block_hash) {
            Some(status_ref) => *status_ref.value(),
            None => {
                if self.header_map().contains_key(block_hash) {
                    BlockStatus::HEADER_VALID
                } else {
                    let verified = self
                        .snapshot()
                        .get_block_ext(block_hash)
                        .map(|block_ext| block_ext.verified);
                    match verified {
                        None => BlockStatus::UNKNOWN,
                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,
                    }
                }
            }
        }
    }
