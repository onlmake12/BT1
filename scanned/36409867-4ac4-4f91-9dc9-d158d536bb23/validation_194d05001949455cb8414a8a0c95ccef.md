### Title
Rich-Indexer Input Loop Silently Truncates on Filtered-Out Cell, Leaving Indexed Cells Permanently Marked Live — (`util/rich-indexer/src/indexer/mod.rs`)

---

### Summary

When `cell_filter` is configured, `insert_transaction` calls `spend_cell` for each input. If `spend_cell` returns `Ok(false)` (the output row is absent from the DB because it was filtered), the loop unconditionally `break`s at line 232, silently skipping all remaining inputs. Any subsequent inputs whose output rows **are** in the DB never have `is_spent` set to `1`, so `get_cells` permanently returns those cells as live even after they are spent on-chain.

---

### Finding Description

**Root cause — `insert_transaction`, `util/rich-indexer/src/indexer/mod.rs` lines 228–248:**

```rust
if tx_index != 0 {
    for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
        let out_point = input.previous_output();
        if !spend_cell(&out_point, tx).await? {
            break;          // ← exits the entire loop
        }
        // build input rows for this input ...
    }
}
``` [1](#0-0) 

**`spend_cell` returns `Ok(false)` when 0 rows are updated**, which happens whenever the referenced output was never inserted into the `output` table — i.e., it was excluded by `cell_filter`, `block_filter`, or `init_tip_hash`. [2](#0-1) 

**`get_cells` filters live cells with `output.is_spent = 0`:** [3](#0-2) 

If `is_spent` is never flipped to `1` for a cell that was actually spent, `get_cells` returns it as live indefinitely.

---

### Impact Explanation

An attacker who controls a transaction can deliberately place a filtered-out cell as `input[0]` and an indexed cell (e.g., a UDT cell) as `input[1]`. After the block is appended:

- `spend_cell(input[0])` → `Ok(false)` → `break`
- `input[1]`'s output row retains `is_spent = 0`
- `get_cells` returns that UDT cell as live

The attacker can present this "live" cell to an exchange, DEX, or any RPC consumer that relies on `get_cells` for liveness checks, while having already spent it on-chain. This enables double-spend confusion at the indexer-RPC layer.

---

### Likelihood Explanation

- `cell_filter` is a documented, production-supported feature (see `README.md` and `ckb.toml` examples).
- The attacker needs only to own one filtered-out cell (e.g., a plain CKB cell) and one indexed cell (e.g., a UDT cell), both of which are ordinary on-chain assets.
- Input ordering in a CKB transaction is attacker-controlled; placing the filtered cell first is trivial.
- No privileged access, hashpower, or special network position is required.

---

### Recommendation

Replace `break` with `continue` at line 232 so that inputs whose outputs are absent from the DB are skipped individually without aborting the rest of the loop:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // was: break
}
``` [4](#0-3) 

---

### Proof of Concept

1. Configure `cell_filter` to match only cells with a specific type script (e.g., UDT cells).
2. Index a block containing `tx_A` with `output[0]` = a UDT cell (passes filter, inserted into `output` table with `is_spent=0`).
3. Index a block containing `tx_B` with:
   - `input[0]` spending a plain CKB cell (filtered out, absent from `output` table)
   - `input[1]` spending `tx_A.output[0]` (present in `output` table)
4. After `append(block_B)`, query `get_cells` for the UDT cell.
5. **Expected:** `is_spent=1`, cell absent from results. **Actual:** `is_spent=0`, cell returned as live.

pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
