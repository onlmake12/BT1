Audit Report

## Title
Tokio Mutex Guard Held Across `tx_pool.fetch_txs().await` in `BlockTransactionsProcess::execute` — (`sync/src/relayer/block_transactions_process.rs`)

## Summary

`BlockTransactionsProcess::execute` acquires the `pending_compact_blocks` Tokio async Mutex and holds its guard across the `.await` point of `Relayer::reconstruct_block`, which internally awaits `tx_pool_controller.fetch_txs()`. Because `compact_block` is a mutable reference borrowed through the `OccupiedEntry` into the guard, the borrow checker forces the guard to remain live for the entire `if let` block, including the async call. Any connected peer that sends a `BlockTransactions` message for a pending compact block serializes all other tasks needing `pending_compact_blocks` for the full duration of the tx-pool round-trip, stalling the compact-block relay pipeline.

## Finding Description

`pending_compact_blocks` is declared as `tokio::sync::Mutex<PendingCompactBlockMap>` in `SyncState`: [1](#0-0) 

The accessor returns a `tokio::sync::MutexGuard`: [2](#0-1) 

In `BlockTransactionsProcess::execute`, the guard is created as a temporary in the scrutinee of an `if let`. Rust's temporary-lifetime-extension rules keep it alive for the entire `if let` block: [3](#0-2) 

`compact_block` on line 71 is a `&mut packed::CompactBlock` borrowed through `pending.get_mut()` into the guard. It is passed directly to `reconstruct_block`, forcing the guard to remain live across the `.await`: [4](#0-3) 

Inside `reconstruct_block`, `tx_pool.fetch_txs(short_ids_set).await` is an inter-task channel call to the tx-pool service. Under any non-trivial tx-pool load the future may not resolve immediately, leaving the Tokio Mutex locked for the entire round-trip duration: [5](#0-4) 

While the guard is held, every other async task calling `pending_compact_blocks().await` blocks:

- `contextual_check` inside `CompactBlockProcess::execute` — new compact blocks from any peer stall here: [6](#0-5) 

- The post-reconstruction cleanup path in `CompactBlockProcess::execute`: [7](#0-6) 

- `missing_or_collided_post_process`: [8](#0-7) 

The rate limiter in `try_process` explicitly skips `CompactBlock` messages but does apply to `BlockTransactions`; however, the limit is 30 req/s per peer, so a single well-timed message is sufficient to open the contention window: [9](#0-8) 

## Impact Explanation

A single `BlockTransactions` message from any connected peer serializes the entire compact-block relay pipeline for the duration of the tx-pool round-trip. During this window, no new compact block can be checked for pending status, no successfully reconstructed block can be cleaned up, and no missing-transaction request can be recorded. Repeated triggering (up to 30 times/s per peer) keeps the lock contended continuously under tx-pool load, causing sustained block propagation delay across the node. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

Preconditions are minimal and normal: at least one compact block must be in the pending map (standard during block relay) and the tx pool must be under any non-trivial load. No privilege is required — any connected peer can send a `BlockTransactions` message. The rate limiter permits 30 such messages per second per peer, making repeated triggering trivial. The contention window scales with tx-pool load, which is attacker-influenceable by flooding the mempool with transactions.

## Recommendation

Clone the required data out of the map before releasing the lock, then re-acquire after reconstruction:

```rust
// 1. Acquire lock, clone needed data, release immediately.
let (compact_block_clone, expected_tx_indexes, expected_uncle_indexes) = {
    let guard = shared.state().pending_compact_blocks().await;
    let entry = guard.get(&block_hash)?;
    let (cb, peers_map, _) = entry;
    let (tx_idx, uncle_idx) = peers_map.get(&self.peer)?;
    (cb.clone(), tx_idx.clone(), uncle_idx.clone())
    // guard dropped here
};

// 2. Reconstruct without holding the lock.
let ret = self.relayer.reconstruct_block(
    &active_chain, &compact_block_clone, received_transactions,
    &expected_uncle_indexes, &received_uncles,
).await;

// 3. Re-acquire to update or remove the entry.
let mut guard = shared.state().pending_compact_blocks().await;
```

This eliminates the guard-across-await anti-pattern entirely and allows all other tasks to proceed during reconstruction.
