### Title
Unauthenticated Notify Endpoint + `work_id=0` Permanent Bypass in `update_block_template` Allows Continuous Worker Interruption — (`miner/src/client.rs`)

---

### Summary

The miner's notify-mode HTTP listener accepts `BlockTemplate` POSTs with no authentication. A logic flaw in `update_block_template` causes the condition `id == 0` to permanently evaluate to `true` whenever `current_work_id` is `0`, because updating with `work_id=0` leaves the atomic at `0`. An attacker who can reach the listen address can flood it with `work_id=0` templates, triggering an unbounded stream of `Works::New` messages that continuously interrupt all mining workers and prevent nonce discovery.

---

### Finding Description

**Root cause — `update_block_template` (`miner/src/client.rs:293–312`)**

```rust
fn update_block_template(&self, block_template: BlockTemplate) {
    let work_id = block_template.work_id.into();   // new id from incoming template
    let updated = |id| {                            // id = current stored value
        if id != work_id || id == 0 {              // ← id==0 is the permanent bypass
            Some(work_id)
        } else {
            None
        }
    };
    if self.current_work_id
           .fetch_update(Ordering::SeqCst, Ordering::SeqCst, updated)
           .is_ok()
    {
        self.new_work_tx.send(Works::New(block_template.into())).ok();
    }
}
```

`current_work_id` is initialised to `0`: [1](#0-0) 

The `id == 0` branch is intended as a "fresh-start" guard, but it checks the **current stored** value, not the incoming one. When a `work_id=0` template is accepted, `fetch_update` writes `0` back — leaving `current_work_id` at `0` forever. Every subsequent `work_id=0` POST re-enters the same branch and fires again.

State trace:

| Step | `current_work_id` (before) | incoming `work_id` | `id != work_id` | `id == 0` | fires? | `current_work_id` (after) |
|------|----------------------------|--------------------|-----------------|-----------|--------|---------------------------|
| 1    | 0                          | 0                  | false           | **true**  | ✓      | 0                         |
| 2    | 0                          | 0                  | false           | **true**  | ✓      | 0                         |
| N    | 0                          | 0                  | false           | **true**  | ✓      | 0                         |

The u64::MAX detour described in the question is unnecessary; the bypass is reachable from the initial state with `work_id=0` alone.

Any TCP peer that can reach the `SocketAddr` bound by `TcpListener::bind` can POST arbitrary `BlockTemplate` JSON with zero credentials. [2](#0-1) 

**Downstream worker interruption (`miner/src/miner.rs:127–138`)**

Each accepted `Works::New` message calls `notify_new_work`, which sends `WorkerMessage::NewWork` to every worker thread, causing them to abandon their current nonce search and restart on the injected (fake) template. [3](#0-2) 

The only guard in `notify_new_work` is `legacy_work.contains(&parent_hash)`, which only holds hashes of **submitted** blocks. An attacker-controlled template with any unseen `parent_hash` bypasses it unconditionally. [4](#0-3) 

---

### Impact Explanation

Every `work_id=0` POST causes all worker threads to drop their current nonce search and restart on the attacker's fake template. At flood rate the workers never complete a single nonce search, halting block production entirely for the targeted miner. The attacker pays only the cost of sending HTTP POST requests.

---

### Likelihood Explanation

Notify mode is an explicitly documented, production-supported configuration. [5](#0-4) 

The listen address is a plain `SocketAddr`; operators commonly bind it to a non-loopback interface so the CKB node (potentially on a different host) can reach it. Any attacker on the same network segment — or on the internet if the port is not firewalled — can exploit this with a trivial HTTP client.

---

### Recommendation

1. **Fix the bypass**: change the update condition so that `work_id=0` is not permanently sticky. The correct invariant is "update only when the incoming id differs from the current id":

   ```rust
   let updated = |id| {
       if id != work_id { Some(work_id) } else { None }
   };
   ```

   If `work_id=0` must be treated as a special "always-refresh" sentinel, the check should test the **incoming** value, not the stored one: `if id != work_id || work_id == 0`.

2. **Authenticate the notify endpoint**: require a shared secret (e.g., a configurable bearer token or HMAC) on every inbound POST, or bind the listener exclusively to `127.0.0.1` and document that it must not be exposed to untrusted networks.
