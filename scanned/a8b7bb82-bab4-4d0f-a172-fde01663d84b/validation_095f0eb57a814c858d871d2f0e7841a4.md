### Title
Silent u128→u64 Truncation in DAO Withdrawal Capacity Calculation — (`File: util/dao/src/lib.rs`)

### Summary
In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `withdraw_counted_capacity` (a `u128`) is cast to `u64` with a bare `as u64`, which silently truncates the upper 64 bits on overflow. Every other analogous u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The inconsistency means a sufficiently large deposit combined with a sufficiently grown accumulate-rate ratio can produce a silently wrong (drastically underestimated) withdrawal capacity that passes all subsequent checks.

### Finding Description
In `calculate_maximum_withdraw` (`util/dao/src/lib.rs`, lines 152–156):

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The product `counted_capacity * withdrawing_ar` is computed in u128 (correct), but the final division result is cast to u64 with `as u64` instead of a checked conversion. If `withdraw_counted_capacity > u64::MAX`, the high bits are silently discarded, producing a value that is `withdraw_counted_capacity % 2^64` — potentially near zero — which then passes the `safe_add` check without error.

Compare with the identical pattern used correctly just 90 lines later in `dao_field_with_current_epoch`:

```rust
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

And again at line 258:

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The `calculate_maximum_withdraw` function is the only site that uses the unchecked `as u64` cast.

### Impact Explanation
`calculate_maximum_withdraw` is called in two production paths:

1. **Block validation / DAO field computation**: `dao_field_with_current_epoch` → `withdrawed_interests` → `transaction_maximum_withdraw` → `calculate_maximum_withdraw`. A silently truncated result causes `withdrawed_interests` to be computed as nearly zero, so `current_s` (the NervosDAO secondary-issuance accumulator stored in every block header) is inflated by the full interest amount that should have been subtracted. This corrupts the on-chain DAO state for all subsequent blocks. [4](#0-3) 

2. **RPC `calculate_dao_maximum_withdraw`**: Returns a silently wrong (far too small) capacity to the caller, misleading wallets and users about their entitled withdrawal amount. [5](#0-4) 

### Likelihood Explanation
The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX  (~1.84 × 10¹⁹)
```

`ar` starts at `10_000_000_000_000_000` (10¹⁶) and grows with each block's secondary issuance. A depositor who locks a very large cell early (low `deposit_ar`) and withdraws after `ar` has grown substantially (high `withdrawing_ar`) can reach the overflow threshold. With the total CKB supply at ~3.36 × 10¹⁸ shannons and `ar` growing at roughly 4% per year (secondary issuance / total capacity), a ratio of ~6× would be needed for a near-maximum deposit — achievable over multi-decade time horizons given the compounding nature of `ar`. The likelihood is low in the near term but non-zero over the full protocol lifetime, and the impact (silent fund loss + consensus state corruption) is critical.

### Recommendation
Replace the unchecked cast with the same checked conversion used elsewhere in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with lines 245 and 258):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [6](#0-5) 

### Proof of Concept
Construct a scenario where `withdraw_counted_capacity` overflows u64:

- `counted_capacity` = 18,000,000,000,000,000,000 shannons (≈ 1.8 × 10¹⁹, near u64::MAX)
- `deposit_ar` = 10,000,000,000,000,000 (initial genesis value)
- `withdrawing_ar` = 11,000,000,000,000,000 (10% growth, achievable after ~2.5 years)

```
withdraw_counted_capacity = 18,000,000,000,000,000,000 × 11,000,000,000,000,000
                            / 10,000,000,000,000,000
                          = 19,800,000,000,000,000,000
```

`19,800,000,000,000,000,000 > u64::MAX (18,446,744,073,709,551,615)`

With `as u64`: `19,800,000,000,000,000,000 % 2^64 = 1,353,255,926,290,448,384`

The depositor receives ~135 million CKB instead of ~1,980 million CKB — a loss of ~1,845 million CKB — with no error raised. The block's DAO field `S` is simultaneously corrupted by the same magnitude, affecting all future NervosDAO interest calculations. [7](#0-6)

        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
