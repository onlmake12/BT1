Audit Report

## Title
Stale `EvictKey` on Surviving Ancestors After `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-clears all link entries for the entire removed subtree before calling `remove_entry` on each node. Because `remove_entry` calls `update_ancestors_index_key`, which resolves ancestors via the now-cleared links, it finds an empty ancestor set and performs no updates. Surviving ancestors of the removed root permanently retain inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`, and a correspondingly stale `evict_key`. An attacker can exploit this to keep a low-fee transaction in the pool indefinitely, enabling sustained mempool manipulation at low cost.

## Finding Description

**Root cause:** In `remove_entry_and_descendants` (lines 252–265), all link entries for the subtree are cleared via `remove_entry_links` before `remove_entry` is called on each node. [1](#0-0) 

`remove_entry` (line 242) calls `update_ancestors_index_key` to subtract descendant weight from surviving ancestors. [2](#0-1) 

`update_ancestors_index_key` (lines 432–434) resolves ancestors by calling `self.links.calc_ancestors(&child.proposal_short_id())`. Because `remove_entry_links` already removed the node from `self.links.inner` (line 429), `calc_relative_ids` finds no entry and returns an empty `HashSet`. [3](#0-2) [4](#0-3) 

The loop body in `update_ancestors_index_key` never executes. `sub_descendant_weight` is never called on any surviving ancestor, and `evict_key` is never refreshed. [5](#0-4) 

`EvictKey` is computed from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`, all of which remain inflated. [6](#0-5) 

**Exploit path:**
1. Submit `ancestor_tx` with a low fee rate.
2. Submit `tx_root` + a descendant chain (all high fee rate, spending `ancestor_tx`'s output). `ancestor_tx.evict_key` is now boosted by the high-fee descendants.
3. Submit `conflict_tx` spending the same input as `tx_root`. This triggers `resolve_conflict` (lines 305–332), which calls `remove_entry_and_descendants(&tx_root_id)`. [7](#0-6) 
4. Due to the bug, `ancestor_tx.evict_key` retains the inflated `fee_rate` and `descendants_count` for the rest of its lifetime in the pool.
5. `next_evict_entry` iterates `iter_by_evict_key()` in ascending order; `ancestor_tx` is never selected for eviction despite its true low fee rate. [8](#0-7) 

`EvictKey` ordering places lowest `fee_rate` first; the stale inflated `fee_rate` pushes `ancestor_tx` toward the back of the eviction queue. [9](#0-8) 

## Impact Explanation

A low-fee transaction can be shielded from eviction indefinitely by exploiting the stale `evict_key`. An attacker can repeat this pattern to fill the mempool with low-fee transactions that are never evicted, blocking legitimate higher-fee transactions from entering the pool. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (10001–15000 points). The attacker does not need the high-fee chain to be confirmed; the fees for `tx_root` and its descendants are never paid on-chain since they are removed from the pool by the conflict.

## Likelihood Explanation

The attack requires only standard P2P transaction relay — no privileged access, no PoW, no key material. The attacker submits two sets of ordinary valid transactions (the descendant chain and the conflict). The pool size limit bounds the descendant chain length but does not prevent the attack. The exploit is repeatable: each round costs only the fees for `ancestor_tx` and `conflict_tx`, while the high-fee chain is never confirmed. Multiple attackers or a single attacker with multiple UTXOs can saturate the pool.

## Recommendation

In `remove_entry_and_descendants`, collect the ancestor set of the root node **before** any link removal, then apply `sub_descendant_weight` and refresh `evict_key` on each surviving ancestor for every node in the removed subtree. Concretely: before the `for id in &removed_ids { self.remove_entry_links(id); }` loop, call `self.links.calc_ancestors(id)` to capture surviving ancestors, then after removal iterate over that set and update their descendant weight fields and `evict_key`. Alternatively, restructure `remove_entry_and_descendants` to call `update_ancestors_index_key` for the root while links are still intact, before pre-clearing links.

## Proof of Concept

Invariant test: after `remove_entry_and_descendants(tx_root)`, for every remaining entry `e` that was an ancestor of `tx_root`:

```rust
assert_eq!(
    e.descendants_count,
    1 + pool_map.calc_descendants(&e.proposal_short_id()).len()
);
```

This assertion fails because `e.descendants_count` still includes the removed subtree nodes, while `calc_descendants` (operating on the now-correct links) returns the true current set. A unit test constructing a three-node chain (`ancestor_tx → tx_root → child_tx`), calling `remove_entry_and_descendants(&tx_root_id)`, and then asserting the above invariant on `ancestor_tx` will reproduce the bug deterministically.     pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
