Audit Report

## Title
Out-of-Bounds Index Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` correctly guards the `block_unconfirmed_txs` access for `tx_age >= len`, but performs an **unguarded** index into `confirm_blocks_to_failed_txs[tx_age - 1]` immediately after. When a transaction has been unconfirmed for more than `MAX_CONFIRM_BLOCKS` blocks, `tx_age - 1` exceeds the vec's length, causing a Rust index-out-of-bounds panic that crashes the tx-pool service thread and the node.

## Finding Description

`confirm_blocks_to_failed_txs` is initialized with length `MAX_CONFIRM_BLOCKS` (valid indices `0..=MAX_CONFIRM_BLOCKS-1`): [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 correctly handles the `block_unconfirmed_txs` access for `tx_age >= len`: [2](#0-1) 

But the subsequent `count_failure` branch has **no corresponding bounds check**: [3](#0-2) 

When `tx_age = MAX_CONFIRM_BLOCKS + 1`, `tx_age - 1 = MAX_CONFIRM_BLOCKS` is an out-of-bounds index on a `Vec` of length `MAX_CONFIRM_BLOCKS` → **panic**.

`drop_tx_inner` passes `self.best_height` as `tip_height` and `tx_record.height` as `entry_height`: [4](#0-3) 

`drop_tx` always passes `count_failure = true`: [5](#0-4) 

`reject_tx` calls `drop_tx`: [6](#0-5) 

The tx-pool eviction callback wires directly to `fee_estimator.reject_tx`: [7](#0-6) 

Both `remove_expired` and `limit_size` (called on every block update) invoke `callbacks.call_reject`: [8](#0-7) 

## Impact Explanation

A panic in the fee estimator write-lock propagates to the tx-pool service thread, crashing the node. This matches the **High (10001–15000 points)** impact: *Vulnerabilities which could easily crash a CKB node*. The crash is deterministic and repeatable — any eviction of a sufficiently old tracked tx triggers it unconditionally.

## Likelihood Explanation

The default expiry is 12 hours: [9](#0-8) 

At CKB's ~8-second block time, `MAX_CONFIRM_BLOCKS + 1 = 1001` blocks ≈ 2.2 hours — well within the 12-hour expiry window. Any unprivileged user can submit a valid low-fee-rate transaction via P2P or RPC and simply wait. No special privileges, no victim interaction, and no external dependencies are required. The attack is repeatable: after a node restart, the same tx (or a new one) can re-trigger the panic.

## Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` access, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
``` [3](#0-2) 

## Proof of Concept

```rust
#[test]
fn test_no_panic_on_old_drop() {
    let mut stat = TxConfirmStat::default();
    // MAX_CONFIRM_BLOCKS = 1000, so tx_age = 1001 triggers the bug
    let entry_height: u64 = 0;
    let tip_height: u64 = 1001;
    let bucket_index: usize = 0;
    // This must not panic:
    stat.remove_unconfirmed_tx(entry_height, tip_height, bucket_index, true);
}
```

With the unpatched code, `tx_age = 1001`, `confirm_blocks_to_failed_txs[1000]` on a `Vec` of length 1000 → index-out-of-bounds panic. The fix causes the branch to be skipped, and the test passes.
