### Title
No-Fee Orphan Pool Flooding via Missing-Input Admission Without Fee Gate — (`tx-pool/src/component/orphan.rs`, `tx-pool/src/process.rs`)

---

### Summary

The orphan pool admission path in `after_process` stores any transaction whose inputs are unresolvable (missing parent) without performing any fee-rate check. Because `limit_size` evicts entries randomly when the pool exceeds `DEFAULT_MAX_ORPHAN_TRANSACTIONS` (100), an unprivileged remote peer can permanently occupy the entire orphan pool with zero-fee or structurally-invalid-fee transactions at the cost of only network bandwidth, causing legitimate high-fee orphans to be randomly evicted.

---

### Finding Description

**Step 1 — Admission path has no fee gate.**

When a remote peer relays a transaction, `process_tx` is called: [1](#0-0) 

`_process_tx` calls `pre_check` → `resolve_tx`. If any input references an unknown UTXO, resolution fails with `Reject::Resolve(OutPointError::Unknown)`. Control returns to `after_process`: [2](#0-1) 

The only gate before `add_orphan` is `is_missing_input`: [3](#0-2) 

`check_tx_fee` — which enforces `min_fee_rate` — is only reached inside `_process_tx` after successful input resolution: [4](#0-3) 

It is **never called** for orphan-bound transactions. `non_contextual_verify` (the only check that runs before orphan admission) does not inspect fee or capacity: [5](#0-4) 

**Step 2 — `add_orphan_tx` stores unconditionally, then calls `limit_size`.** [6](#0-5) 

**Step 3 — `limit_size` evicts randomly, not by fee.** [7](#0-6) 

`entries.keys().next()` on a `HashMap` is effectively random. There is no fee-ordered eviction policy.

**Step 4 — Orphan entries persist until `ORPHAN_TX_EXPIRE_TIME`.** [8](#0-7) [9](#0-8) 

`ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL`. Attacker-controlled orphans whose parents are never sent will sit in the pool for the full expiry window. The attacker can continuously send fresh unique orphans to replace any that expire or are evicted.

**Step 5 — No per-peer orphan limit exists.**

There is no per-peer quota in `add_orphan_tx` or `add_orphan`. A single peer can contribute all 100 slots. [10](#0-9) 

---

### Impact Explanation

- The orphan pool (100 slots) can be permanently saturated by one attacker peer at zero CKB cost.
- Every legitimate high-fee orphan arriving at a saturated pool faces a random eviction (1-in-101 chance per arrival, worsening as the attacker replenishes evicted slots).
- Orphan transactions that are evicted must be re-fetched and re-relayed, increasing relay latency and bandwidth waste across the network.
- The attacker-controlled orphans will never successfully transition to the pending pool (their parents are never submitted, or if submitted, `check_tx_fee` will reject them), so they provide no legitimate value while occupying all slots.

---

### Likelihood Explanation

- The attack requires only the ability to send P2P relay messages — no CKB tokens, no privileged access, no PoW.
- Constructing 100 structurally valid transactions with fabricated input `OutPoint`s is trivial.
- The attack is self-sustaining: the attacker sends replacements as slots open, maintaining saturation indefinitely.
- A single malicious peer is sufficient; no Sybil attack is needed.

---

### Recommendation

1. **Add a fee-rate pre-screen at orphan admission.** Before calling `add_orphan`, perform a lightweight capacity sanity check on the declared outputs vs. a minimum threshold, or require a minimum `declared_cycle`-based fee estimate.
2. **Replace random eviction with fee-rate-ordered eviction.** `limit_size` should evict the entry with the lowest `fee / size` ratio (computable from declared cycles and tx size), not a random entry.
3. **Add a per-peer orphan quota.** Limit the number of orphan slots any single `PeerIndex` can occupy (e.g., 10 out of 100), preventing a single peer from saturating the pool.
4. **Reduce `DEFAULT_MAX_ORPHAN_TRANSACTIONS` or add a declared-fee floor.** Require that `declared_cycle` implies a minimum fee rate before admission.

---

### Proof of Concept

```
1. Attacker constructs 100 transactions T_1..T_100:
   - Each has one input referencing a random, non-existent OutPoint
   - Each has one output with capacity equal to or greater than the input's claimed capacity
     (zero or negative fee — irrelevant since fee is never checked at admission)
   - Each passes NonContextualTransactionVerifier (valid version, size, non-empty, no dup deps)

2. Attacker relays T_1..T_100 to a target node via SendTransaction P2P messages.

3. For each T_i:
   - process_tx → non_contextual_verify (passes)
   - _process_tx → pre_check → resolve_tx → Err(OutPointError::Unknown)
   - after_process → is_missing_input → add_orphan(T_i, peer, declared_cycle)
   - add_orphan_tx inserts T_i; limit_size does nothing (pool not yet full)

4. After T_100: orphan pool is at capacity (100/100).

5. A legitimate node relays a high-fee orphan L:
   - add_orphan_tx inserts L (pool becomes 101)
   - limit_size evicts one entry at random — L has a ~1% chance of immediate eviction

6. Attacker monitors evictions (via TxVerificationResult::Reject signals or by probing)
   and sends T_101, T_102, ... to replace evicted slots, maintaining saturation.

7. Assert: orphan pool remains full of attacker-controlled zero-fee orphans;
   legitimate orphans are continuously at risk of random eviction.
```

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
