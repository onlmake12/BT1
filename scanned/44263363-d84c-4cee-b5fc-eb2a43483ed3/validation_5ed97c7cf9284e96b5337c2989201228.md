### Title
Unbounded DB Read Cost Per Incoming `GetHeaders` Message with No Per-Peer Rate Limit — (`sync/src/synchronizer/get_headers_process.rs`)

### Summary
An unprivileged remote peer can craft a `GetHeaders` message with exactly `MAX_LOCATOR_SIZE` (101) locator entries — 100 unknown hashes followed by the genesis hash — that forces the receiving node to perform up to ~4,101 DB reads per message. Because there is no per-peer incoming rate limit on `GetHeaders`, a single peer can sustain this at wire speed, causing sustained I/O and CPU exhaustion.

### Finding Description

**Step 1 — Size check passes.** [1](#0-0) 
The guard is `locator_size > MAX_LOCATOR_SIZE`, so exactly 101 entries is accepted. [2](#0-1) 

**Step 2 — IBD guard is bypassed.**
The precondition is that the node is out of IBD, so execution falls through to `locate_latest_common_block`. [3](#0-2) 

**Step 3 — `locate_latest_common_block` iterates all 101 entries.**
The iterator is lazy (`.find()`), but since the first 100 hashes are unknown, `get_block_number` is called for all 101 entries before the genesis hash matches at index 100. The condition `latest_common == Some(0)` then causes an immediate return of `Some(0)` — the peer is **not** rejected. [4](#0-3) 

**Step 4 — `get_locator_response` performs up to 4,000 DB reads.**
Starting from block 0 (genesis), it iterates up to `MAX_HEADERS_LEN = 2,000` block numbers, calling `get_block_hash` (snapshot read) and `get_block_header` (store read) for each. [5](#0-4) [6](#0-5) 

**Step 5 — No rate limit on incoming `GetHeaders`.**
The `pending_get_headers` LRU cache with `GET_HEADERS_TIMEOUT = 15s` is used exclusively for the node's **own outgoing** `send_getheaders_to_peer` calls — it does not throttle incoming messages from peers. [7](#0-6) 
There is no `RateLimiter` or equivalent guard in the sync protocol handler for incoming `GetHeaders`. [8](#0-7) 

**Step 6 — Peer is never banned.**
`GetHeadersMissCommonAncestors` (the only ban path in this handler) is only triggered when `locate_latest_common_block` returns `None`. With a valid genesis-terminated locator, it returns `Some(0)`, so `Status::ok()` is returned and the peer is never penalized. [9](#0-8) [10](#0-9) 

### Impact Explanation
Each crafted `GetHeaders` message causes ~4,101 DB reads (101 `get_block_number` + up to 2,000 `get_block_hash` + up to 2,000 `get_block_header`). With no incoming rate limit and no ban triggered, a single peer can sustain this at TCP wire speed, causing sustained RocksDB I/O saturation and degraded node responsiveness. This falls within the **Low (501–2000)** scoped impact band: it is a resource exhaustion DoS, not a consensus or fund-safety issue.

### Likelihood Explanation
The attack requires only a standard P2P connection — no privileges, no PoW, no keys. The crafted message is trivially constructable (100 random 32-byte hashes + genesis hash). The node's connection limit is the only practical throttle, but a single connection is sufficient to sustain the load.

### Recommendation
1. Add a per-peer incoming `GetHeaders` rate limit (e.g., max N messages per second per peer) in `GetHeadersProcess::execute` or the sync protocol dispatcher.
2. Consider banning or disconnecting peers that repeatedly send all-unknown locators (heuristic: if `locate_latest_common_block` returns `Some(0)` more than K times from the same peer within a window, apply `SYNC_USELESS_BAN_TIME`).
3. Alternatively, short-circuit `get_locator_response` when `block_number == 0` and the chain height is large, or cap the response to a smaller window.

### Proof of Concept
```
1. Connect to a non-IBD CKB node (chain height > 2000).
2. Craft GetHeaders { block_locator_hashes: [rand_hash * 100, genesis_hash], hash_stop: 0x00..00 }.
3. Send in a tight loop (no sleep).
4. Observe: node never bans the peer; each message triggers ~4101 DB reads;
   node I/O and CPU spike proportionally to send rate.
5. Instrument: assert total DB reads == N * 4101 for N messages sent.
```
