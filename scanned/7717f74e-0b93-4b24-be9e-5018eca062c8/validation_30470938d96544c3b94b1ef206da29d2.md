### Title
Unbounded Per-Peer MMR Proof Computation in LightClient `GetBlocksProof` Handler — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

---

### Summary

When the LightClient protocol is enabled, any unprivileged remote peer can repeatedly send `GetBlocksProof` messages with up to 1000 block hashes. Each message triggers thousands of RocksDB reads and a full MMR proof generation over up to 1000 leaf positions, with **no per-peer rate limit, no request queue depth limit, and no concurrency cap**. N concurrent peers each sending max-size requests force N parallel MMR proof computations, saturating I/O and the async runtime.

---

### Finding Description

**Entry point:** `LightClientProtocol::received` dispatches directly to `GetBlocksProofProcess::execute` with no rate-limiting gate. [1](#0-0) 

The only guard in `execute` is a size check against `GET_BLOCKS_PROOF_LIMIT = 1000`: [2](#0-1) [3](#0-2) 

After that check, the handler iterates all 1000 hashes, performing three separate DB reads per hash (`get_block_header`, `get_block_uncles`, `get_block_extension`): [4](#0-3) 

Then `reply_proof` is called, which calls `mmr.gen_proof(positions)` over all 1000 leaf positions. Each `get_elem` call in the MMR store is a live RocksDB read: [5](#0-4) [6](#0-5) 

The `LightClientProtocol` struct holds only `shared: Shared` — no per-peer state, no request counter, no semaphore: [7](#0-6) 

The `connected` handler does nothing but log: [8](#0-7) 

A search across the entire codebase for `rate_limit`, `per_peer`, `peer_limit`, `request_limit`, `flood`, `throttle` returns zero matches in the light-client-protocol-server. The network layer has `max_inbound`/`max_outbound` connection limits, but no per-protocol, per-peer message rate limiting.

The launcher registers `LightClientProtocol` as a plain `CKBProtocol` with no additional middleware: [9](#0-8) 

---

### Impact Explanation

Each max-size `GetBlocksProof` request causes:
- ~3000 synchronous RocksDB point reads (header + uncles + extension per hash)
- One `chain_root_mmr` construction and `gen_proof` call over 1000 positions, requiring O(1000 × log N) additional MMR node reads from RocksDB

With N concurrent peers each sending requests in a tight loop, the node experiences:
- RocksDB read I/O saturation (shared across chain processing, tx-pool, and sync)
- Async task queue depth growth, delaying block relay and sync responses
- Potential starvation of the chain's critical path (block import, compact block relay)

This matches the stated scope: **network congestion with few costs** — the attacker needs only a TCP connection and a valid `last_hash` (the tip hash is publicly observable).

---

### Likelihood Explanation

- LightClient is an opt-in protocol (`SupportProtocol::LightClient` in config), but when enabled it is reachable by any peer without authentication.
- The attack requires no PoW, no stake, no key material — only a P2P connection.
- The tip hash needed for `last_hash` is trivially obtained from any public node or block explorer.
- The cost to the attacker is negligible (sending a ~32 KB message per request); the cost to the server is thousands of DB reads and an MMR proof computation.

---

### Recommendation

1. **Per-peer request rate limiting:** Track the last request timestamp per peer in `LightClientProtocol` and enforce a minimum interval (e.g., 1 request per second per peer) for expensive message types (`GetBlocksProof`, `GetTransactionsProof`, `GetLastStateProof`).
2. **Global concurrency cap:** Use a `tokio::sync::Semaphore` to bound the number of simultaneously executing proof-generation tasks.
3. **Reduce the limit or add cost scaling:** Lower `GET_BLOCKS_PROOF_LIMIT` or charge a sliding cost (e.g., ban peers that send max-size requests more than K times per minute).
4. **Peer scoring:** Penalize peers that repeatedly send large requests without consuming the responses.

---

### Proof of Concept

```
1. Enable LightClient protocol on a CKB full node (set SupportProtocol::LightClient in config).
2. Obtain the current tip hash via any public RPC or peer.
3. From 50 concurrent TCP connections, each send in a tight loop:
     GetBlocksProof {
       last_hash: <tip_hash>,
       block_hashes: [h1, h2, ..., h1000]  // 1000 valid main-chain hashes
     }
4. Observe on the server:
   - RocksDB read IOPS spike to saturation
   - Async task queue depth grows unboundedly
   - Block relay latency increases; sync stalls
   - No peer is banned (all messages are well-formed and within the 1000-hash limit)
```

The `BAD_MESSAGE_BAN_TIME` of 5 minutes only triggers on malformed messages — a well-formed max-size request is never penalized. [10](#0-9)

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
