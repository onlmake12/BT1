### Title
Unbounded `last_n_numbers` in `GetLastStateProofProcess::execute` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The per-message work guard in `GetLastStateProofProcess::execute` only checks `difficulties.len() + last_n_blocks * 2` against `GET_LAST_STATE_PROOF_LIMIT`. However, the else-branch at line 291 can produce a `last_n_numbers` vector of size `last_block_number − difficulty_boundary_block_number`, which is entirely unchecked. An unprivileged peer can craft a single `GetLastStateProof` message that forces the server to perform O(L) MMR `get_root()` calls and O(L) `get_ancestor()` calls, where L is the full chain height.

---

### Finding Description

**Guard (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `difficulties = []` and `last_n_blocks = 0`, the expression evaluates to `0 > 1000` → false. The guard is trivially bypassed. [1](#0-0) 

**Branch selection (lines 291–292):**

```rust
if last_block_number - start_block_number <= last_n_blocks
```

With `start_block_number = 0`, `last_block_number = L`, `last_n_blocks = 0`: evaluates to `L <= 0`, which is false for any non-genesis chain. The else-branch executes. [2](#0-1) 

**Else-branch `last_n_numbers` (lines 299–319):**

`difficulty_boundary_block_number` is found via binary search for the first block whose total difficulty ≥ `difficulty_boundary`. If the attacker sets `difficulty_boundary = 1` (any value ≤ the genesis block's total difficulty), `difficulty_boundary_block_number` resolves to block 0 or 1.

The expansion guard at line 313 only *widens* the window when there are too few blocks; it never *caps* it:

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

With `last_n_blocks = 0`, `L − 0 < 0` is always false (unsigned), so no adjustment occurs.

Result: `last_n_numbers = (0..L)` — a vector of L elements. [3](#0-2) 

**`complete_headers` cost (lines 132–163):**

For every element in `block_numbers` (all L of them), the server performs:
- `snapshot.get_ancestor(last_hash, number)` — chain traversal
- `snapshot.get_block(hash)` — DB read
- `snapshot.chain_root_mmr(number - 1).get_root()` — MMR root computation

This is O(L) heavy DB/MMR operations per single P2P message. [4](#0-3) 

---

### Impact Explanation

A single malformed `GetLastStateProof` message causes the server to perform work proportional to the full chain height L. On a mainnet node with millions of blocks, this translates to millions of MMR and DB operations per message. An attacker can flood the server with such messages (each trivially constructed, no PoW required) to exhaust CPU and I/O, causing node resource exhaustion.

---

### Likelihood Explanation

The attack requires only a P2P connection — no authentication, no stake, no PoW. The crafted message is valid at the protocol-parsing level and passes all structural checks. The attacker needs only to know the tip block hash (publicly available via `GetLastState`). The exploit is deterministic and repeatable.

---

### Recommendation

After computing `last_n_numbers` (and `sampled_numbers`, `reorg_last_n_numbers`), enforce a hard cap on the total `block_numbers` length before calling `complete_headers`:

```rust
let block_numbers = reorg_last_n_numbers
    .into_iter()
    .chain(sampled_numbers)
    .chain(last_n_numbers)
    .collect::<Vec<_>>();

if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, cap `last_n_numbers` at construction time to `max(last_n_blocks, difficulties.len())` so the guard at lines 201–205 remains the single source of truth. [5](#0-4) 

---

### Proof of Concept

```
1. Connect to a CKB full node with the light-client protocol enabled.
2. Send GetLastState to obtain the current tip hash (tip_hash) at height L.
3. Construct GetLastStateProof:
     start_hash    = zero_hash
     start_number  = 0
     last_hash     = tip_hash          // valid main-chain block
     last_n_blocks = 0                 // bypasses guard: 0 + 0*2 = 0 <= 1000
     difficulty_boundary = 1           // resolves to block 0 or 1
     difficulties  = []                // empty
4. Send the message.
5. Server enters the else-branch (L - 0 <= 0 is false),
   resolves difficulty_boundary_block_number ≈ 0,
   builds last_n_numbers = (0..L) with L entries,
   calls complete_headers() with L entries,
   performing L × (get_ancestor + get_block + mmr.get_root) operations.
6. Repeat at high frequency; observe CPU/IO saturation.
   Assert: number of DB reads >> GET_LAST_STATE_PROOF_LIMIT (1000).
```
