Audit Report

## Title
Asymmetric Descendant-Weight Accounting in `remove_entry_and_descendants` Allows Inflation of Ancestor's Effective Fee Rate - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::remove_entry_and_descendants` erases all link records for the removed subtree before invoking `remove_entry` on each entry. Because `update_ancestors_index_key` resolves ancestors via `self.links.calc_ancestors`, and those links are already gone, surviving ancestors of the removed root never have `sub_descendant_weight` called on them. Since `add_entry` correctly increments those fields, the accounting is permanently asymmetric. An unprivileged attacker can exploit this to inflate a low-fee parent transaction's apparent descendant fee rate, making it eviction-resistant and allowing it to occupy pool space indefinitely.

## Finding Description

**Add path (correct):**

`add_entry` calls `record_entry_descendants`, which ends with:

```rust
self.update_ancestors_index_key(entry, EntryOp::Add);
``` [1](#0-0) 

`update_ancestors_index_key` resolves ancestors via `self.links.calc_ancestors` and calls `add_descendant_weight` on each surviving ancestor. [2](#0-1) 

**Remove path (broken):**

`remove_entry_and_descendants` first strips all link records for every entry in the subtree, then calls `remove_entry` for each:

```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
for id in &removed_ids {
    self.remove_entry_links(id);
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
``` [3](#0-2) 

`remove_entry_links` calls `self.links.remove(id)`, which removes the entry's own record from the links map, and also removes it from its parents' children lists: [4](#0-3) 

Inside `remove_entry`, `update_ancestors_index_key` is called with `EntryOp::Remove`: [5](#0-4) 

But `update_ancestors_index_key` resolves ancestors via `self.links.calc_ancestors(&child.proposal_short_id())`. Because the link entry for `child` was already erased by `remove_entry_links`, `calc_ancestors` returns an **empty set**, so `sub_descendant_weight` is never called on any surviving ancestor. [2](#0-1) 

The comment in `remove_entry_and_descendants` acknowledges the intent ("so that we won't update_descendants_index_key in remove_entry"), but it inadvertently also suppresses the necessary `update_ancestors_index_key` call for surviving ancestors.

**Trigger path:**

`resolve_conflict` is called every time a submitted transaction spends an input already claimed by a pool transaction. It calls `remove_entry_and_descendants` on the conflicting pool entry, leaving its surviving ancestors with stale (inflated) descendant weights: [6](#0-5) 

**Stale fields feed `EvictKey`:**

The inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` directly feed the `EvictKey` computation: [7](#0-6) 

`add_descendant_weight` and `sub_descendant_weight` are the symmetric pair that should balance: [8](#0-7) 

## Impact Explanation

The stale `descendants_fee` raises `descendants_feerate`, which raises `EvictKey.fee_rate`. Since eviction picks the entry with the **lowest** `EvictKey`, an inflated entry is pushed to the back of the eviction queue and becomes eviction-resistant. A low-fee parent transaction can be kept alive in the pool indefinitely, consuming pool capacity and distorting fee estimation. By repeating the cycle across many parent transactions, an attacker can fill the pool with low-fee transactions that cannot be evicted, preventing legitimate high-fee transactions from entering the pool.

This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attack requires only the ability to call `send_transaction` RPC — available to any unprivileged user. The attacker needs only two UTXOs: one to fund `tx_parent` (output becomes `UTXO_B`) and one attacker-controlled (`UTXO_C`). The cycle can be executed with minimal cost because the conflict transactions are never confirmed on-chain, so the UTXOs are never actually consumed. The cycle is:

1. Submit `tx_child_N` spending `UTXO_B` + `UTXO_C` → `tx_parent.descendants_fee` incremented.
2. Submit `tx_conflict_N` spending `UTXO_C` → `tx_child_N` removed via `resolve_conflict` → `tx_parent.descendants_fee` **not** decremented.
3. Submit `tx_child_{N+1}` spending `UTXO_B` + `UTXO_C` → conflicts with `tx_conflict_N`, removes it, then repeat.

This loop is self-sustaining with just two UTXOs and inflates `tx_parent.descendants_fee` by `tx_child.fee` per iteration.

## Recommendation

Before erasing link records in `remove_entry_and_descendants`, collect the surviving ancestors of the root entry and decrement their descendant weights while links are still intact. Concretely:

1. Before the link-removal loop, call `update_ancestors_index_key(root_entry, EntryOp::Remove)` for the root (and each removed entry whose ancestors are not also in the removed set) while links are still intact.
2. Only then proceed to strip links and call `remove_entry`.

Alternatively, restructure `remove_entry` to accept a pre-computed ancestor set so that the link teardown order does not matter.

## Proof of Concept

**Setup:** `tx_parent` in pool with `fee = 1 shannon`, `size = 100`. Attacker controls `UTXO_C`.

**Initial state:** `tx_parent.descendants_fee = 1`, `tx_parent.descendants_count = 1`.

**Cycle (repeat N times):**

1. Submit `tx_child_N` spending `tx_parent`'s output (`UTXO_B`) + `UTXO_C`, `fee = 1000 shannons`.
   - `add_entry` → `update_ancestors_index_key(tx_child_N, Add)` → `tx_parent.descendants_fee += 1000`.

2. Submit `tx_conflict_N` spending `UTXO_C` (same input as `tx_child_N`).
   - `resolve_conflict` → `remove_entry_and_descendants(tx_child_N)` → links erased first → `update_ancestors_index_key(tx_child_N, Remove)` finds **no ancestors** → `tx_parent.descendants_fee` is **not decremented**.
   - `tx_conflict_N` is added to pool (does not spend `UTXO_B`, so no conflict with `tx_parent`).

3. Submit `tx_child_{N+1}` spending `UTXO_B` + `UTXO_C`.
   - Conflicts with `tx_conflict_N` → removes it → `tx_child_{N+1}` added → `tx_parent.descendants_fee += 1000` again.

**After N cycles:**

```
tx_parent.descendants_fee  ≈ 1 + N × 1000  (should be ≤ 1001)
tx_parent.EvictKey.fee_rate ≈ N × 1000 / 100  (should be ~10 shannon/byte)
```

`tx_parent` is now effectively immune to eviction despite having a real fee rate of `1/100 = 0.01 shannon/byte`. A unit test can verify this by asserting `tx_parent.descendants_fee == tx_parent.fee` after the remove cycle completes.

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    /// Update ancestor state for add an entry
    pub fn add_descendant_weight(&mut self, entry: &TxEntry) {
        self.descendants_count = self.descendants_count.saturating_add(1);
        self.descendants_size = self.descendants_size.saturating_add(entry.size);
        self.descendants_cycles = self.descendants_cycles.saturating_add(entry.cycles);
