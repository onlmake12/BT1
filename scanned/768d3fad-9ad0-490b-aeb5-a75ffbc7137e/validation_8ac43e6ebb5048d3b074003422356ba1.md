Audit Report

## Title
Size-Only Minimum Fee Gate Allows Cycle-Heavy Transactions to Bypass `min_fee_rate` Enforcement — (`tx-pool/src/util.rs`, `tx-pool/src/pool.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the serialized byte size of the transaction, while the pool's internal weight — used for scoring, eviction, and fee-rate reporting — is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For a cycle-heavy transaction, the gate check can underestimate the true weight by up to ~12× at default settings, allowing the transaction to enter the pool with an effective fee rate far below `min_fee_rate`. The same size-only approximation appears in `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, where cycles are already known for pool entries, making that instance unambiguously incorrect.

## Finding Description

**Root cause — `check_tx_fee` (`tx-pool/src/util.rs:42-52`):**

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code comment itself acknowledges the approximation is theoretically incorrect. `check_tx_fee` is called inside `pre_check` before `verify_rtx` runs, so actual cycles are not yet available. After verification, the entry is constructed as `TxEntry::new(rtx, verified.cycles, fee, tx_size)` and admitted to the pool. Once inside, `fee_rate()` in `tx-pool/src/component/entry.rs:115-118` uses the correct weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
```

This creates a systematic discrepancy: the admission gate uses `size`, but all post-admission logic (scoring, eviction via `limit_size`, fee-rate reporting) uses `get_transaction_weight(size, cycles)`.

**Root cause — `calculate_min_replace_fee` (`tx-pool/src/pool.rs:101-103`):**

```rust
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
    let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
```

Here, cycles are already known for every pool entry, yet only `size` is used. An RBF replacement transaction can satisfy the `min_rbf_rate` check while paying far less than the weight-based minimum.

**Exploit flow:**
1. Attacker constructs a transaction with small serialized size (~1,000 bytes) but near-maximum cycles (~70,000,000, the default `max_tx_verify_cycles`).
2. Actual weight = `max(1000, 70_000_000 × 0.000_170_571_4)` = `max(1000, 11,940)` = **11,940**.
3. Gate check requires `1,000 × 1,000 / 1,000` = **1,000 shannons**.
4. Weight-based minimum would require `11,940 × 1,000 / 1,000` = **11,940 shannons**.
5. Transaction paying 1,000 shannons passes `check_tx_fee` and enters the pool with an effective fee rate of ~84 shannons/KW — roughly 8.4% of the configured minimum.
6. Repeat to fill the mempool with cycle-heavy, below-minimum-fee-rate transactions.

## Impact Explanation

This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points).**

An attacker can flood the mempool with transactions that consume the block cycle budget (up to 70M cycles each) while paying only ~8.4% of the fee that `min_fee_rate` is intended to enforce. Although the eviction logic (`limit_size`) correctly ranks these transactions low by weight-based fee rate and will eventually evict them when the pool is full, the attacker can continuously resubmit them, keeping the pool saturated with cycle-heavy spam at a fraction of the intended cost. This degrades the fee market and can cause sustained network congestion. The RBF bypass additionally allows replacement transactions to violate the `min_rbf_rate` invariant.

## Likelihood Explanation

Any unprivileged caller of the `send_transaction` RPC or any P2P relay peer can trigger this. Crafting a cycle-heavy transaction requires writing a CKB-VM lock or type script that executes a tight loop — a straightforward task for any script author. The default `max_tx_verify_cycles = TWO_IN_TWO_OUT_CYCLES * 20 = 70,000,000` bounds the amplification factor to approximately 12× at default settings. No special privileges, leaked keys, or victim mistakes are required. The attack is repeatable and low-cost.

## Recommendation

**For `check_tx_fee`:** Defer the weight-based minimum fee check to after `verify_rtx` returns the actual cycle count, then use `get_transaction_weight(tx_size, verified.cycles)` to compute `min_fee`. The size-only pre-check can be retained as a fast-path rejection only for transactions whose fee is below `min_fee_rate.fee(tx_size)` (i.e., it cannot pass even the lower bound), but the authoritative check must use the true weight post-verification.

**For `calculate_min_replace_fee`:** Replace `self.config.min_rbf_rate.fee(size as u64)` with `self.config.min_rbf_rate.fee(get_transaction_weight(size, entry_cycles))`, since cycles are already known for all pool entries at the point this function is called.

## Proof of Concept

1. Write a CKB-VM lock script that executes a tight RISC-V loop consuming ~70,000,000 cycles. Keep the script binary small so the total serialized transaction size is ~1,000 bytes.
2. Construct a transaction using this lock script. Set the fee to exactly `min_fee_rate * tx_size / 1000` shannons (e.g., 1,000 shannons at default `min_fee_rate = 1,000`).
3. Submit via `send_transaction` RPC.
4. Observe: `check_tx_fee` passes because `fee (1,000) >= min_fee_rate.fee(tx_size=1000) (1,000)`.
5. Observe: after admission, `entry.fee_rate()` reports ~84 shannons/KW, far below the 1,000 shannons/KW minimum, confirming the discrepancy.
6. Repeat in a loop to fill the 180 MB pool with cycle-heavy, below-minimum-fee-rate transactions, causing sustained mempool congestion.
7. For the RBF variant: submit a conflicting replacement transaction with fee just above `min_rbf_rate.fee(size)` but below `min_rbf_rate.fee(get_transaction_weight(size, cycles))` and observe it is accepted by `check_rbf`.
