### Title
`forward_rate_limiter` Bypass via Distinct `(from, to)` Pairs Enables `isqrt(N)` Fan-Out Amplification per Inbound `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

An unprivileged remote peer can send up to 30 `ConnectionRequest` messages per second (the outer rate-limiter cap), each carrying a distinct attacker-chosen `(from, to)` pair targeting a peer unknown to the node. Because `content.from` is never validated against the actual session peer, each message gets its own `forward_rate_limiter` bucket and passes the 1-req/sec inner check. Every such message that reaches `forward_message` triggers a `filter_broadcast` to `isqrt(peer_count)` peers, producing a sustained `30 × isqrt(N)` outbound message rate from a single inbound connection.

---

### Finding Description

**Outer rate limiter** — keyed by `(session_id, msg.item_id())`: [1](#0-0) 

`msg.item_id()` is the fixed union discriminant (always `0` for `ConnectionRequest`), so this bucket is shared across all `ConnectionRequest` messages from the same session. Quota: **30 req/sec**.

**Inner `forward_rate_limiter`** — keyed by `(content.from, content.to, msg_item_id)`: [2](#0-1) 

`content.from` and `content.to` are parsed directly from the attacker-controlled message payload. There is no check that `content.from` matches the actual connecting session peer. Each unique `(from, to)` pair creates a fresh bucket with its own 1-req/sec allowance, so an attacker rotating pairs exhausts none of them.

**Fan-out in `forward_message`** when `to` is not in the peer registry: [3](#0-2) 

Each message that reaches this branch causes `filter_broadcast` to `isqrt(peer_count)` peers.

**No `from`-session binding** — the `execute()` path never asserts `content.from == self.peer`'s peer ID: [4](#0-3) 

---

### Impact Explanation

| Peers (N) | isqrt(N) | Outbound msgs/sec (30 req/s cap) |
|-----------|----------|----------------------------------|
| 100       | 10       | 300                              |
| 500       | 22       | 660                              |
| 10 000    | 100      | 3 000                            |

Each outbound message requires serialization, a `filter_broadcast` closure evaluation over all peers, and a socket write. The amplification is proportional to `sqrt(N)` and is sustained as long as the attacker maintains the connection and rotates `(from, to)` pairs. Multiple attacker connections multiply the effect linearly.

---

### Likelihood Explanation

The attack requires only a standard P2P connection to the target node — no authentication, no PoW, no privileged role. Generating distinct random `(from, to)` byte pairs is trivial. The outer rate limiter (30/sec) is the only real cap; the `forward_rate_limiter` is fully defeated by pair rotation.

---

### Recommendation

Bind `content.from` to the actual session peer ID: reject any `ConnectionRequest` where `content.from` does not match the peer ID of the sending session. This collapses all messages from one session into a single `forward_rate_limiter` bucket, making the inner limiter effective and reducing the per-session fan-out to at most `isqrt(N)` messages per second regardless of how many distinct `(from, to)` pairs the attacker supplies.

Alternatively, key the `forward_rate_limiter` on `session_id` alone (or `(session_id, to)`) rather than on the attacker-supplied `from` field.

---

### Proof of Concept

1. Connect to a target node that has N=100 peers.
2. Send 30 `ConnectionRequest` messages in one second, each with a freshly generated random `from` peer ID and a random `to` peer ID that is not in the node's peer registry.
3. Observe via network capture or metrics that the node emits ≈300 outbound `HolePunching` messages (10 per inbound message).
4. Confirm the `forward_rate_limiter` never fires (each `(from, to)` pair is new) while the outer limiter allows all 30 inbound messages through.

    pub(crate) async fn execute(mut self) -> Status {
        let content = match RequestContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }
