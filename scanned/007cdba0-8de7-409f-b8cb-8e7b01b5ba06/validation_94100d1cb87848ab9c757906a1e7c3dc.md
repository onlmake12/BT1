### Title
IPv4-Mapped IPv6 Address Representation Bypass in Peer Ban Enforcement - (File: network/src/peer_store/ban_list.rs)

### Summary
The CKB peer store's ban enforcement uses exact `IpNetwork` key matching that does not normalize IPv4-mapped IPv6 addresses (`::ffff:x.x.x.x`) to their IPv4 equivalents. A peer banned under its IPv4 address can immediately reconnect using the IPv4-mapped IPv6 representation of the same address, bypassing the ban entirely. This is the direct CKB analog to the external report's URL byte-equality uniqueness failure: logically identical endpoints have different byte representations that defeat the deduplication/exclusion check.

### Finding Description

The `BanList` stores banned addresses as `HashMap<IpNetwork, BannedAddr>` keyed by the exact `IpNetwork` value produced by `ip_to_network()`. [1](#0-0) 

When a peer is banned via `ban_addr()`, the connected address is converted through `multiaddr_to_socketaddr` → `.ip()` → `ip_to_network()`: [2](#0-1) 

`ip_to_network()` preserves the IP family verbatim — an IPv4 address produces `IpNetwork::V4(x.x.x.x/32)`, an IPv6 address produces `IpNetwork::V6(...)/128`: [3](#0-2) 

The ban check in `is_ip_banned_until()` first does an exact HashMap lookup, then falls back to `ip_network.contains(ip)`: [4](#0-3) 

The `ipnetwork` crate's `IpNetwork::V4(...).contains(IpAddr::V6(...))` always returns `false` — a V4 network cannot contain a V6 address. Therefore, if a peer was banned as `IpAddr::V4(1.2.3.4)` (stored as `IpNetwork::V4(1.2.3.4/32)`), a reconnection arriving as `/ip6/::ffff:1.2.3.4/tcp/.../p2p/...` produces `IpAddr::V6(::ffff:1.2.3.4)` → `IpNetwork::V6(::ffff:1.2.3.4/128)`, which matches neither the exact key nor the `contains` check.

The ban check is invoked in `PeerRegistry::accept_peer()` for every inbound connection: [5](#0-4) 

The `base_addr()` normalization used by `AddrManager` only strips transport-layer protocols (`Ws`, `Wss`, `Memory`, `Tls`) and does not normalize IPv4-mapped IPv6 to IPv4: [6](#0-5) 

This means `/ip4/1.2.3.4/tcp/8115/p2p/X` and `/ip6/::ffff:1.2.3.4/tcp/8115/p2p/X` are stored as two distinct entries in `AddrManager`'s `addr_to_id` HashMap: [7](#0-6) 

The per-IP deduplication in `fetch_random` also fails for the same reason — `IpAddr::V4(1.2.3.4)` and `IpAddr::V6(::ffff:1.2.3.4)` are distinct `HashSet` keys: [8](#0-7) 

The RPC documentation itself shows this dual-representation is real and observed in production: a peer appears simultaneously as `/ip6/::ffff:18.185.102.19/tcp/8115/...` and `/ip4/18.185.102.19/tcp/8115/...`: [9](#0-8) 

### Impact Explanation

1. **Ban bypass**: A peer banned for misbehavior (e.g., sending invalid blocks/headers, triggering score-based banning via `report()`) can immediately reconnect using the IPv4-mapped IPv6 form of its address. The ban is completely ineffective on dual-stack nodes, which are the norm on modern Linux servers.

2. **Peer store slot inflation**: The same physical peer can occupy two entries in `AddrManager`, consuming peer store capacity (capped at `ADDR_COUNT_LIMIT = 16384`) and causing duplicate outbound connection attempts to the same endpoint. [10](#0-9) 

### Likelihood Explanation

Modern Linux servers listen on `::` (IPv6 wildcard) by default, which accepts both native IPv6 and IPv4-mapped IPv6 connections. A banned peer on any such node can trivially reconnect. The attacker needs only to reconnect — no special capability is required. The CKB RPC documentation itself demonstrates that the dual-representation is observed in real deployments.

### Recommendation

In `ip_to_network()` and `is_ip_banned_until()`, normalize IPv4-mapped IPv6 addresses to their IPv4 equivalents before constructing the `IpNetwork` key or performing the lookup:

```rust
pub fn ip_to_network(ip: IpAddr) -> IpNetwork {
    let ip = match ip {
        IpAddr::V6(v6) => v6.to_ipv4_mapped()
            .map(IpAddr::V4)
            .unwrap_or(IpAddr::V6(v6)),
        v4 => v4,
    };
    match ip {
        IpAddr::V4(ipv4) => IpNetwork::V4(ipv4.into()),
        IpAddr::V6(ipv6) => IpNetwork::V6(ipv6.into()),
    }
}
```

Apply the same normalization in `base_addr()` so that `AddrManager` treats `/ip4/x.x.x.x/...` and `/ip6/::ffff:x.x.x.x/...` as the same key.

### Proof of Concept

```

# ban_addr stores IpNetwork::V4(1.2.3.4/32)

# Step 2: Same peer reconnects using IPv4-mapped IPv6 multiaddr:
#   /ip6/::ffff:1.2.3.4/tcp/8115/p2p/<same_or_new_peer_id>

# Step 3: accept_peer() calls peer_store.is_addr_banned(&remote_addr)
#   multiaddr_to_socketaddr → IpAddr::V6(::ffff:1.2.3.4)
#   ip_to_network → IpNetwork::V6(::ffff:1.2.3.4/128)
#   HashMap lookup: no match (stored key is V4)
#   contains() loop: IpNetwork::V4(1.2.3.4/32).contains(IpAddr::V6(...)) → false
#   Result: is_addr_banned returns false → peer accepted despite active ban
