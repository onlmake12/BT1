### Title
Concurrent `HeaderMapKernel::get` Calls on Sled-Resident Entry Cause Entry Loss and Node Panic — (`shared/src/types/header_map/kernel_lru.rs`)

---

### Summary

`HeaderMapKernel::get` contains a TOCTOU race: it checks memory under a write lock, releases that lock, then calls `SledBackend::remove` without any synchronization. Two concurrent callers for the same sled-resident hash can both miss the memory check and both call `remove`; only one receives the value, the other receives `None`. The losing thread propagates `None` through `get_header_index_view` into `insert_valid_header`, which panics unconditionally with `"parent should be verified"`, crashing the node.

---

### Finding Description

**`HeaderMapKernel::get`** (kernel_lru.rs:109–142):

```rust
pub(crate) fn get(&self, hash: &Byte32) -> Option<HeaderIndexView> {
    if let Some(view) = self.memory.get_refresh(hash) {   // (1) write-lock acquired + released
        return Some(view);
    }
    if self.backend.is_empty() { return None; }
    if let Some(view) = self.backend.remove(hash) {       // (2) NO lock held here
        self.memory.insert(view.clone());
        Some(view)
    } else {
        None                                               // (3) losing thread returns None
    }
}
``` [1](#0-0) 

**`MemoryMap::get_refresh`** acquires and immediately releases a write lock:

```rust
pub(crate) fn get_refresh(&self, key: &Byte32) -> Option<HeaderIndexView> {
    let mut guard = self.0.write();
    guard.get_refresh(key).cloned().map(|inner| (key.clone(), inner).into())
}
``` [2](#0-1) 

The lock is dropped before `backend.remove` is called. There is no mechanism preventing a second thread from also observing the key absent from memory and proceeding to `backend.remove`.

**`SledBackend::remove`** is not idempotent under concurrency:

```rust
fn remove(&self, key: &Byte32) -> Option<HeaderIndexView> {
    let old_value = self.db.remove(key.as_slice()).expect("...");
    old_value.map(|slice| {
        self.count.fetch_sub(1, Ordering::SeqCst);
        HeaderIndexView::from_slice_should_be_ok(key.as_slice(), &slice)
    })
}
``` [3](#0-2) 

Sled's `remove` is atomic at the DB level, so the first caller gets `Some`, the second gets `None`. The second caller returns `None` from `get()`.

**`insert_valid_header`** panics unconditionally on `None`:

```rust
let parent_header_index = self
    .get_header_index_view(&parent_hash, store_first)
    .expect("parent should be verified");
``` [4](#0-3) 

`get_header_index_view` calls `self.shared.header_map().get(hash)` which calls `HeaderMapKernel::get`. If the parent header H is not yet in the chain store (normal during IBD) and the race causes `None` to be returned, the `.expect` panics. [5](#0-4) 

---

### Impact Explanation

A panic in `insert_valid_header` terminates the node process. This is reachable from two production P2P paths:

- **`headers_process.rs:356`** — `sync_shared.insert_valid_header(self.peer, self.header)` called from the `Headers` sync message handler.
- **`compact_block_process.rs:78`** — `shared.insert_valid_header(self.peer, &header)` called from the compact block relay handler. [6](#0-5) [7](#0-6) 

Both handlers run in concurrent async tasks with no outer lock protecting `insert_valid_header`. `HeaderMap` itself holds no per-operation mutex; `HeaderMap::get` simply delegates to `self.inner.get(hash)` with no additional synchronization. [8](#0-7) 

---

### Likelihood Explanation

**Preconditions that are routinely met during IBD:**

1. The memory map exceeds `memory_limit`, causing `limit_memory()` to evict entries to sled — this is the designed behavior during sync of a long chain.
2. Two peers (attacker-controlled or even honest peers) send a child header C whose parent H is sled-resident. Both trigger concurrent `insert_valid_header(C)` calls.
3. The race window between `memory.get_refresh` returning `None` and `backend.remove` completing is small but non-zero and reproducible under load.

The `HEADER_VALID` early-exit check in `HeaderAcceptor::accept` is not atomic with the subsequent `insert_valid_header` call, so two threads can both pass the check and both call `insert_valid_header` for the same child header. [9](#0-8) 

---

### Recommendation

Hold the memory write lock across the entire check-and-promote sequence, or introduce a per-key mutex/`DashMap`-style entry API so that the "not in memory → remove from backend → insert into memory" sequence is atomic. A simpler mitigation is to re-check memory after a failed `backend.remove` (since the winning thread will have re-inserted the entry) before returning `None`.

### Proof of Concept

```
Thread A                                    Thread B
─────────────────────────────────────────────────────────────────────
insert_valid_header(child_of_H)             insert_valid_header(child_of_H)
  get_header_index_view(&H.hash)              get_header_index_view(&H.hash)
    header_map().get(&H.hash)                   header_map().get(&H.hash)
      memory.get_refresh(&H.hash) → None          memory.get_refresh(&H.hash) → None
      backend.remove(&H.hash) → Some(v)           backend.remove(&H.hash) → None ← H lost
      memory.insert(v)                            return None
      return Some(v)                            get_header_index_view → None
                                              .expect("parent should be verified") → PANIC
```

Two honest or attacker-controlled peers simultaneously relaying the same child header during IBD (when H has been evicted to sled) is sufficient to trigger this. No PoW, no privileged access, and no majority hashpower is required.
