### Title
Capacity Conservation Check Bypassed for Mixed DAO/Non-DAO Transactions Allowing Unauthorized CKB Minting - (File: `verification/src/transaction_verifier.rs`)

### Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` whenever **any** input carries the DAO type script, causing the entire `OutputsSumOverflow` guard to be skipped. The comment delegates enforcement to "the type script of DAO cells," but the DAO type script only runs on cells that carry the DAO type script — it has no visibility into, and performs no check on, non-DAO inputs and outputs in the same transaction. A transaction sender can therefore include one DAO input alongside inflated non-DAO outputs to mint CKB out of thin air.

### Finding Description

`CapacityVerifier::verify()` enforces the fundamental CKB supply invariant: `sum(outputs.capacity) ≤ sum(inputs.capacity)`. It has exactly two exceptions:

1. Cellbase transactions — verified by `RewardVerifier`.
2. Any transaction where `valid_dao_withdraw_transaction()` returns `true`.

`valid_dao_withdraw_transaction()` uses `.any(...)` — it returns `true` the moment a single input carries the DAO type script, regardless of how many non-DAO inputs and outputs are also present:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
```

When this returns `true`, the block:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    let inputs_sum  = self.resolved_transaction.inputs_capacity()?;
    let outputs_sum = self.resolved_transaction.outputs_capacity()?;
    if inputs_sum < outputs_sum { return Err(...); }
}
```

is skipped entirely. The DAO type script (a compiled C binary in CKB-VM) only executes on cells that carry the DAO type script. It verifies the DAO-specific withdrawal amount but has no mandate to check non-DAO cells in the same transaction. No other verifier in the pipeline re-checks total capacity conservation for this class of transaction.

This applies to **both** DAO phases:
- **Phase 1** (deposit → withdrawal marker): DAO type script checks that the output DAO cell has the same capacity as the input DAO cell. Non-DAO cells are unchecked.
- **Phase 2** (withdrawal marker → free cell): DAO type script checks that the output capacity equals the maximum DAO withdrawal amount. Non-DAO cells are unchecked.

### Impact Explanation

An unprivileged transaction sender can mint arbitrary CKB by constructing a transaction such as:

| Role | Cell | Capacity |
|---|---|---|
| Input 0 | DAO withdrawal-marker cell (phase 2) | 100 CKB |
| Input 1 | Normal cell | 10 CKB |
| Output 0 | Free cell (DAO withdrawal) | 105 CKB (correct DAO amount) |
| Output 1 | Normal cell | 60 CKB (**50 CKB minted**) |

`CapacityVerifier` skips the check because input 0 has the DAO type script. The DAO type script approves output 0 (correct withdrawal amount). Output 1 is never checked against input 1. The 50 CKB excess is minted from nothing, violating CKB's fixed-supply invariant and inflating the total circulating supply.

### Likelihood Explanation

Any holder of a DAO deposit cell can trigger this. No special privilege, key, or majority hashpower is required. The attacker only needs to:
1. Deposit CKB into the DAO (standard operation).
2. Execute phase 1 withdrawal (standard operation).
3. Craft a phase 2 withdrawal transaction with additional non-DAO inputs and inflated non-DAO outputs.

The transaction is valid from the tx-pool's perspective (passes `CapacityVerifier`) and from the DAO type script's perspective (DAO withdrawal amount is correct). It would be committed to a block and accepted by all nodes running this code.

### Recommendation

Replace the blanket skip with a precise check: verify that `sum(outputs.capacity) ≤ sum(inputs.capacity) + dao_interest`, where `dao_interest` is the incremental capacity granted by the DAO withdrawal (i.e., `transaction_maximum_withdraw(rtx) - inputs_capacity`). Alternatively, decompose the check so that non-DAO inputs and outputs are always subject to the standard conservation rule, and only the DAO-specific delta is exempted.

fn cell_uses_dao_type_script(cell_output: &CellOutput, dao_type_hash: &Byte32) -> bool {
    cell_output
        .type_()
        .to_opt()
        .map(|t| {
            Into::<u8>::into(t.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
                && &t.code_hash() == dao_type_hash
        })
        .unwrap_or(false)
}
