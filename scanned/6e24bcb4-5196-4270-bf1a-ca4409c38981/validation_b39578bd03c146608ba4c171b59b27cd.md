### Title
Unauthenticated Inbound Peer Can Poison `observed_addrs` via Crafted `IdentifyMessage`, Propagating Attacker-Controlled Addresses to All Subsequent Peers — (`network/src/protocols/identify/mod.rs`)

### Summary

An unprivileged inbound peer can send an `IdentifyMessage` containing an arbitrary `observed_addr`. The node stores it unconditionally in `NetworkState::observed_addrs`, then includes it in the `listen_addrs` field of every subsequent identify handshake. Receiving peers add those addresses to their peer stores, propagating the poisoned address network-wide.

### Finding Description

**Step 1 — No validation in `process_observed`.**

`process_observed` accepts any `Multiaddr` from the remote peer and forwards it directly to `add_observed_addr` with no content checks: [1](#0-0) 

The return value of `add_observed_addr` is discarded; the function unconditionally returns `MisbehaveResult::Continue`, so the peer is never disconnected for sending a bogus address.

**Step 2 — `IdentifyCallback::add_observed_addr` stores the address without validation.** [2](#0-1) 

The only transformation is appending the local peer-id if absent. No check is made that the address matches the session's actual remote address, that it is reachable, or that it is plausible. It is stored in `network_state.observed_addrs` keyed by `session_id`.

**Step 3 — `local_listen_addrs` includes observed addresses when public addresses are scarce.** [3](#0-2) 

If the node has fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses — the common case for a node with one public IP — the shortfall is filled from `observed_addrs`.

**Step 4 — The poisoned address is broadcast to every new peer.**

In `connected`, `local_listen_addrs()` is called and the result is filtered only by `is_reachable` (global IP check) before being sent: [4](#0-3) 

An attacker who supplies an `observed_addr` with a globally routable IP (e.g., `1.2.3.4`) passes this filter trivially.

**Step 5 — Receiving peers add the address to their peer store.**

`add_remote_listen_addrs` unconditionally calls `peer_store.add_addr` for every address in the received `listen_addrs`: [5](#0-4) 

### Impact Explanation

Any peer that connects after the attacker can receive the poisoned address in the identify handshake and store it. If the address points to an attacker-controlled node, subsequent connection attempts by honest peers are directed there (eclipse / man-in-the-middle setup). If the address is unreachable, peer discovery degrades. Because the address propagates transitively through the peer store and the discovery protocol, the poisoning is not limited to direct peers of the victim node.

### Likelihood Explanation

- Requires only a standard inbound TCP connection — no credentials, no stake, no hashpower.
- The attacker can hold the connection open while the poisoned address propagates to many peers.
- Most production nodes have fewer than 10 public addresses, so the `observed_addrs` fallback is always active.
- The `is_reachable` filter is trivially bypassed by using any public IP.

### Recommendation

1. **Validate `observed_addr` against the session's actual remote address.** The node already knows the peer's socket address (`session.address`); the `observed_addr` should be rejected if its IP does not match.
2. **Apply `is_reachable` filtering inside `add_observed_addr`**, not only at send time.
3. **Do not include observed addresses in `listen_addrs` sent to other peers**, or at minimum mark them as unverified and exclude them from peer-store propagation.
4. **Return `MisbehaveResult::Disconnect`** (or at least penalize) when an implausible observed address is received.

### Proof of Concept

```
1. Attacker connects to victim node (inbound session, session_id = S1).
2. Attacker sends IdentifyMessage {
       listen_addrs: [],
       observed_addr: /ip4/1.2.3.4/tcp/8115,   // attacker-controlled IP
       identify: <valid network identify bytes>
   }
3. Victim calls process_observed → add_observed_addr → network_state.observed_addrs[S1] = /ip4/1.2.3.4/tcp/8115/p2p/<victim_peer_id>
4. Honest peer P2 connects to victim.
5. Victim calls local_listen_addrs():
       public_addrs returns < 10 entries
       → observed_addrs fills the gap with /ip4/1.2.3.4/tcp/8115/p2p/<victim_peer_id>
6. Victim sends IdentifyMessage to P2 with listen_addrs containing the poisoned address.
7. P2 calls add_remote_listen_addrs → peer_store.add_addr(/ip4/1.2.3.4/tcp/8115/...)
8. P2 later dials 1.2.3.4:8115, connecting to the attacker.
```

The poisoned address appears in the identify message sent to P2, confirming the address-book poisoning.

        let mut listens = self.listen_addrs();

        if listens.len() < MAX_RETURN_LISTEN_ADDRS {
            let observe_addrs = self
                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
            listens.extend(observe_addrs);
            listens
        } else {
            listens
        }
    }
