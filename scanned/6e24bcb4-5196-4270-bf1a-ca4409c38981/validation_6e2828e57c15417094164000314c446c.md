## Code Path Trace

**Step 1 — Attacker sends crafted `observed_addr`**

Any connected peer can send an `IdentifyMessage`. In `received()`, the message is decoded and `process_observed` is called unconditionally (after duplicate/identify checks): [1](#0-0) 

**Step 2 — `process_observed` passes the addr directly to `add_observed_addr`** [2](#0-1) 

There is no IP validation, no check that the address matches the session's actual remote address, and no check that the address is plausible for the victim.

**Step 3 — `IdentifyCallback::add_observed_addr` appends the victim's own peer ID** [3](#0-2) 

If the attacker sends `/ip4/1.2.3.4/tcp/9999` (no `/p2p/` component), `extract_peer_id(&addr).is_none()` is `true`, and the victim's own peer ID is appended, producing `/ip4/1.2.3.4/tcp/9999/p2p/<victim_peer_id>`. This is then stored via `network_state.add_observed_addr(session_id, addr)`.

**Step 4 — `local_listen_addrs` includes `observed_addrs`** [4](#0-3) 

When the victim has fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses, it fills the remainder from `observed_addrs`.

**Step 5 — The crafted address is sent to newly connected peers**

In `connected()`, `local_listen_addrs()` is called and the result is sent to the new peer: [5](#0-4) 

There is a `global_ip_only` filter applied here: [6](#0-5) 

This filter calls `is_reachable(socket_addr.ip())`. For the attacker's crafted address `/ip4/1.2.3.4/tcp/9999/p2p/<victim_peer_id>`, if `multiaddr_to_socketaddr` extracts `1.2.3.4:9999` and `1.2.3.4` is a globally routable IP, `is_reachable` returns `true` and the address **passes the filter**.

## Assessment

The `global_ip_only` filter is the only guard between the attacker-controlled `observed_addr` and propagation. It only rejects private/loopback IPs. An attacker using any public IP (e.g., `1.2.3.4`) trivially bypasses it. There is no check that the `observed_addr` matches the session's actual remote address, no rate limiting visible in this layer, and no cryptographic binding.

The invariant that a node's advertised listen addresses must be its own legitimate addresses is broken: any inbound peer can inject an arbitrary public IP+port (with the victim's peer ID appended) into the victim's advertised addresses.

### Title
Unauthenticated Remote Peer Can Inject Arbitrary Addresses Into Victim's Advertised Listen Addresses via Identify `observed_addr` — (`network/src/protocols/identify/mod.rs`)

### Summary
An unprivileged inbound peer can send a crafted `IdentifyMessage` with an `observed_addr` containing an arbitrary public IP/port and no `/p2p/` component. The victim appends its own peer ID to the address and stores it in `observed_addrs`, which is then included in the victim's own `listen_addrs` sent to all subsequently connected peers.

### Finding Description
In `IdentifyCallback::add_observed_addr` [3](#0-2) , the `observed_addr` from a remote peer is accepted without any validation that it corresponds to the session's actual remote address. If the address lacks a `/p2p/` component, the victim's own peer ID is appended and the result is stored in `observed_addrs`. In `local_listen_addrs` [4](#0-3) , these observed addresses are included in the victim's advertised listen addresses when fewer than `MAX_RETURN_LISTEN_ADDRS` public addresses are available. The only guard is a `global_ip_only` / `is_reachable` check [6](#0-5) , which passes any globally routable IP — trivially satisfied by the attacker.

### Impact Explanation
Any peer receiving the victim's identify message will store the crafted address as a valid way to reach the victim. This pollutes peer discovery network-wide: nodes attempting to connect to the victim may try the attacker-controlled address, fail, and deprioritize the victim. The attacker can also redirect connection attempts to infrastructure they control (since the victim's peer ID is in the address, the connection will fail authentication, but the attempt itself is made). Impact is degraded peer discovery and potential targeted isolation of the victim node.

### Likelihood Explanation
Exploitation requires only a single inbound TCP connection — no authentication, no PoW, no privileged role. The attacker sends one `IdentifyMessage` with a crafted `observed_addr`. The effect propagates to every peer the victim subsequently connects to.

### Recommendation
In `add_observed_addr`, validate that the IP in the `observed_addr` matches the session's actual remote address (i.e., `info.session.address`). Reject or ignore `observed_addr` values whose transport endpoint does not match the session's observed remote endpoint. Additionally, consider rate-limiting how many distinct `observed_addrs` are stored per session.

### Proof of Concept
1. Connect to victim as an inbound peer.
2. Complete the identify handshake (send a valid `IdentifyMessage` with `identify` matching the network name and a valid flag).
3. In the same or a subsequent `IdentifyMessage`, set `observed_addr = /ip4/1.2.3.4/tcp/9999` (a public IP, no `/p2p/` component).
4. Connect a third node to the victim.
5. Observe the `IdentifyMessage` the victim sends to the third node; assert that `/ip4/1.2.3.4/tcp/9999/p2p/<victim_peer_id>` appears in the victim's `listen_addrs`.
