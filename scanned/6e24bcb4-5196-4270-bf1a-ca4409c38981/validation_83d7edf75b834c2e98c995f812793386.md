### Title
Panic in `ConsumeUnverifiedBlocks` Leaves `unverified_tip` Unreset, Permanently Stalling Block Synchronization — (File: `chain/src/verify.rs`)

---

### Summary

In `chain/src/verify.rs`, `ConsumeUnverifiedBlocks::start()` wraps `consume_unverified_blocks` in a `catch_unwind` to handle panics. The panic recovery path only removes the block from `is_pending_verify` but omits the critical step of resetting `unverified_tip` back to the actual verified chain tip. The normal error path inside `consume_unverified_blocks` does perform this reset. This state inconsistency leaves the sync state machine with an artificially elevated `unverified_tip`, which causes `block_fetcher` to permanently stop downloading blocks, stalling synchronization until node restart.

---

### Finding Description

`ConsumeUnverifiedBlocks::start()` processes blocks from the unverified queue:

```rust
// chain/src/verify.rs lines 86-96
if let Err(payload) = catch_unwind(AssertUnwindSafe(|| {
    self.processor.consume_unverified_blocks(unverified_task);
})) {
    error!(
        "consume unverified block {}-{} panicked: {}",
        block_number,
        block_hash,
        panic_payload_to_string(payload.as_ref())
    );
    self.processor.is_pending_verify.remove(&block_hash);
    // ← unverified_tip NOT reset here
    // ← block_status NOT updated here
    // ← header_map NOT cleaned here
}

When a panic occurs during verification, `unverified_tip` remains at the panicked block's number. The `block_fetcher` uses `unverified_tip` as a hard gate:

```rust
// sync/src/synchronizer/block_fetcher.rs lines 122-129
if self.sync_shared.shared().get_unverified_tip().number() >= unverified_tip_limit {
    trace!(
        "unverified_tip - tip > BLOCK_DOWNLOAD_WINDOW * 9, skip fetch..."
    );
    return None;
}
``` [4](#0-3) 

Where `unverified_tip_limit = tip_number + BLOCK_DOWNLOAD_WINDOW * 9`. If `unverified_tip` is stuck far above the actual verified tip, this condition is permanently satisfied and no further blocks are fetched.

---

### Impact Explanation

After a panic in `consume_unverified_blocks`, the node's `unverified_tip` is frozen at the panicked block's number. The `block_fetcher` interprets this as "there are already too many unverified blocks queued" and returns `None` on every call to `fetch()`, halting all block downloads. The node's chain tip stops advancing. This is a permanent sync stall that persists until the node is restarted, since `unverified_tip` is in-memory state with no self-healing mechanism.

The `block_status_map` also retains the block's `BLOCK_RECEIVED` status. Because `new_block_received` checks for exactly `HEADER_VALID` status before accepting a re-delivered block, the panicked block cannot be re-processed even if re-sent by a peer. [5](#0-4) 

---

### Likelihood Explanation

The developers explicitly anticipated panics by wrapping `consume_unverified_blocks` in `catch_unwind`. This is not a theoretical construct — it is production code designed to survive panics. Panics in Rust can arise from `expect()`/`unwrap()` on `None`, integer overflow in debug builds, out-of-bounds slice access, or any `panic!()` call in the verification pipeline. An unprivileged P2P peer can submit arbitrary blocks to any reachable CKB node. If any block-content-dependent code path in the verification pipeline panics (e.g., in script execution, capacity accounting, or block structure parsing), the attacker achieves a persistent sync stall with a single crafted block.

---

### Recommendation

In the panic handler inside `ConsumeUnverifiedBlocks::start()`, perform the same `unverified_tip` reset that the normal error path performs:

```rust
if let Err(payload) = catch_unwind(AssertUnwindSafe(|| {
    self.processor.consume_unverified_blocks(unverified_task);
})) {
    error!(...);
    self.processor.is_pending_verify.remove(&block_hash);

    // Add: reset unverified_tip to actual verified tip
    if let Some(tip) = self.processor.shared.store().get_tip_header() {
        if let Some(tip_ext) = self.processor.shared.store().get_block_ext(&tip.hash()) {
            self.processor.shared.set_unverified_tip(
                ckb_shared::HeaderIndex::new(tip.number(), tip.hash(), tip_ext.total_difficulty)
            );
        }
    }
    // Add: mark block invalid so it is not re-accepted
    self.processor.shared.insert_block_status(block_hash, BlockStatus::BLOCK_INVALID);
}
``` [6](#0-5)

### Proof of Concept

1. Attacker connects as a P2P sync peer to a CKB node.
2. Attacker crafts a block whose content triggers a panic inside `consume_unverified_blocks` (e.g., via a code path that calls `unwrap()` or `expect()` on attacker-influenced data during contextual verification).
3. The block passes non-contextual checks in `asynchronous_process_block`, is stored, and dispatched to the `PreloadUnverified` thread. `unverified_tip` is advanced to the block's number.
4. `ConsumeUnverifiedBlocks` dequeues the block and calls `consume_unverified_blocks`, which panics.
5. `catch_unwind` catches the panic. Only `is_pending_verify.remove(&block_hash)` is executed. `unverified_tip` is not reset.
6. `block_fetcher` evaluates `unverified_tip.number() >= tip_number + BLOCK_DOWNLOAD_WINDOW * 9`. With `unverified_tip` stuck at the attacker's block number and the verified tip far behind, this condition holds permanently.
7. The node stops downloading any further blocks. Chain synchronization halts until the node is restarted. [1](#0-0) [7](#0-6)
