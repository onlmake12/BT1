### Title
Unbounded NAT Traversal Task Spawning via Repeated ConnectionSync Messages — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

An unprivileged connected peer can cause the victim node to spawn an unbounded number of concurrent async tasks (each holding a `TcpSocket` and running for up to 30 seconds) by sending repeated `ConnectionSync` P2P messages. The `forward_rate_limiter` intended to throttle this path is bypassable by varying the attacker-controlled `content.from` field. The only effective cap is the per-session `rate_limiter` at 30 req/sec, which still allows ~21,600 concurrent tasks from a single session over the 30-second task lifetime.

---

### Finding Description

**Two rate limiters exist in the `HolePunching` protocol:**

1. **`rate_limiter`** — keyed by `(session_id, msg.item_id())`, 30 req/sec per session per message type. Checked in `received()` before dispatch.
2. **`forward_rate_limiter`** — keyed by `(content.from, content.to, msg_item_id)`, 1 req/sec per (from, to, item_id) tuple. Checked inside `execute()`. [1](#0-0) 

**The `forward_rate_limiter` is bypassable.** Its key includes `content.from`, which is attacker-controlled message content. By using a different `from` peer ID in each `ConnectionSync` message, the attacker creates a fresh rate-limiter bucket for each message, bypassing the 1/sec cap entirely. [2](#0-1) 

**The `pending_delivered` precondition is satisfiable.** The map is populated when the victim processes a `ConnectionRequest` with `to == victim_peer_id`. The attacker sends `ConnectionRequest` messages with different fake `from` peer IDs and fake TCP listen addresses. Each inserts a new entry. The map has no size limit and entries expire only after `TIMEOUT = 5 minutes`. [3](#0-2) 

**The spawn path.** Once a `ConnectionSync` message passes both rate limiters and finds a `pending_delivered` entry for `content.from`, `execute()` unconditionally calls `runtime::spawn(select_ok(tasks))` with up to `ADDRS_COUNT_LIMIT = 24` concurrent `try_nat_traversal` futures, with no cap on total outstanding spawns. [4](#0-3) 

**Each `try_nat_traversal` task** runs a retry loop for up to 30 seconds, creating a new `TcpSocket` each iteration (~200ms per attempt). [5](#0-4) 

---

### Impact Explanation

- **Steady-state concurrent tasks**: `rate_limiter` cap of 30 `ConnectionSync`/sec × 30-second task lifetime × 24 tasks/message = **21,600 concurrent async tasks** from a single session.
- **Open file descriptors**: Each task holds one `TcpSocket` at a time (~200ms per attempt), yielding ~10,800 simultaneously open sockets — sufficient to exhaust the default Linux per-process fd limit (often 1,024–65,536).
- **Async runtime pressure**: 21,600 live tasks each with active timers (`delay_for`) saturates the Tokio timer wheel and task scheduler, degrading or halting all other node operations (block sync, transaction relay, RPC).

---

### Likelihood Explanation

Any single connected peer can execute this attack. No special privilege, leaked key, or majority hashpower is required. The attacker only needs:
1. One P2P session to the victim node.
2. The ability to craft `ConnectionRequest` and `ConnectionSync` messages with arbitrary `from` peer IDs — both are standard P2P messages with no cryptographic authentication on the `from` field.

The `pending_delivered` entries can be pre-populated in under a minute (30/sec × 60 sec = 1,800 entries), and the attack sustains itself as long as the session remains open.

---

### Recommendation

1. **Cap total outstanding NAT traversal tasks**: Maintain an atomic counter or semaphore (e.g., `Arc<Semaphore>`) and reject the spawn if the limit is exceeded.
2. **Key the `forward_rate_limiter` on the sending session**, not on attacker-controlled `content.from`/`content.to`, or add a per-session cap on how many task-spawning `ConnectionSync` messages are processed.
3. **Bound the `pending_delivered` map size** to prevent unbounded pre-population.
4. **Authenticate the `from` field**: Require that `content.from` matches the peer ID of the sending session, so the attacker cannot use arbitrary fake peer IDs to bypass per-(from,to) rate limits.

### Proof of Concept

```
1. Connect to victim node via P2P (one session).
2. Send 1,800 ConnectionRequest messages (over ~60 seconds) each with a unique
   fake `from` peer ID (e.g., random 32-byte keys) and 24 fake TCP addresses
   (e.g., 192.0.2.1:1–192.0.2.24:1). Each passes rate_limiter (30/sec) and
   forward_rate_limiter (different from each time). Each inserts one entry into
   pending_delivered.
3. Send 30 ConnectionSync messages/sec, each with a different `from` matching
   one of the pending_delivered entries, empty route, and to = victim_peer_id.
4. Observe: after 30 seconds, `lsof -p <ckb_pid> | wc -l` shows ~10,000+ open
   sockets; `tokio` task count metric shows ~21,600 live tasks; node becomes
   unresponsive to block sync and RPC.
```
