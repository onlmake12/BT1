### Title
Unbounded `last_n_numbers` via `difficulty_boundary=0` causes server-side DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
Any unprivileged P2P peer can send a `GetLastStateProof` message with `difficulty_boundary=U256::zero()` and an empty `difficulties` array. This bypasses the intended size guard and causes the server to compute and serialize headers for every block from `start_block_number` to the chain tip, performing O(chain-length) disk I/O and memory allocation per request.

### Finding Description

The size guard at line 201 checks:
```
difficulties.len() + last_n_blocks * 2 > GET_LAST_STATE_PROOF_LIMIT
``` [1](#0-0) 

This uses `last_n_blocks` from the attacker-controlled message, not the actual number of blocks that will be returned. In the `else` branch (when `last_block_number - start_block_number > last_n_blocks`), `last_n_numbers` is computed as:
```rust
let last_n_numbers = (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
``` [2](#0-1) 

With `difficulty_boundary=0`, `get_first_block_total_difficulty_is_not_less_than` immediately returns `Some((start_block_number, ...))` because any U256 total difficulty satisfies `>= 0`: [3](#0-2) 

This sets `difficulty_boundary_block_number = start_block_number`, making `last_n_numbers` span the entire range `[start_block_number, last_block_number)` — potentially millions of blocks on mainnet.

The empty-array guard at lines 259–266 also vacuously passes because `.unwrap_or(false)` returns `false` for `None`: [4](#0-3) 

`complete_headers` is then called for every block in `last_n_numbers`, performing `get_ancestor`, `get_block`, and `chain_root_mmr` per block: [5](#0-4) 

### Impact Explanation
A single malicious peer can trigger O(chain-length) disk reads, memory allocation, and MMR root computations on the server per request. On mainnet with millions of blocks, this exhausts server memory and CPU, causing a denial of service. The `GET_LAST_STATE_PROOF_LIMIT = 1000` constant provides no protection because it is checked against the wrong variable. [6](#0-5) 

### Likelihood Explanation
The attack requires no privileges, no keys, and no hashpower. Any peer that can send a `GetLastStateProof` P2P message can trigger it. The crafted message is trivially small (empty difficulties, `last_n_blocks=1`, `difficulty_boundary=0`).

### Recommendation
After computing `last_n_numbers` and `sampled_numbers`, add a guard:
```rust
if sampled_numbers.len() + last_n_numbers.len() + reorg_last_n_numbers.len()
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many blocks");
}
```
Additionally, reject `difficulty_boundary=0` explicitly, since it is semantically meaningless as a FlyClient threshold.

### Proof of Concept
Send a `GetLastStateProof` with:
- `last_hash` = current tip hash (on main chain)
- `start_hash` = genesis hash, `start_number = 0`
- `difficulty_boundary = U256::zero()`
- `difficulties = []`
- `last_n_blocks = 1`

Observe that `execute()` calls `complete_headers` for every block from 0 to tip, performing unbounded I/O proportional to chain length, while the size check passes with value `2`.
