### Title
Unauthenticated Block Template Injection via Miner Notify HTTP Endpoint — (`miner/src/client.rs`)

### Summary

When the CKB miner is configured in "notify mode," it binds an HTTP server on a configurable `SocketAddr` to receive block template push notifications from the CKB node. The `handle()` function that processes these notifications performs no authentication or source-address check: any caller who can reach the port can POST a crafted `BlockTemplate` JSON body and immediately redirect the miner's workers to mine attacker-controlled work. This is a direct analog to the unrestricted `fallback()` handler in the Nexus report — a dispatch function that routes inputs to a privileged handler with no caller authorization.

### Impact Explanation

An attacker who can reach the miner's notification port can:

1. **Continuously inject fake `BlockTemplate` payloads**, keeping the miner's workers permanently redirected to attacker-controlled work. Because `update_block_template` only skips the update when `work_id` is unchanged and non-zero, the attacker can rotate `work_id` values to guarantee every injection is accepted.
2. **Cause the miner to waste all hashpower** on templates that will never produce a valid block on the canonical chain (e.g., templates with a wrong `parent_hash`, wrong `transactions_root`, or pointing to a stale tip).
3. **Deny block rewards** to the miner operator for the duration of the attack — a direct economic impact proportional to the miner's hashrate.
4. **Suppress legitimate block production** from that miner, which at scale (targeting multiple miners) could degrade network liveness.

The miner does eventually re-poll the CKB node for a fresh template (via `blocking_fetch_block_template` after a submit), but an attacker injecting at a rate faster than the poll interval can maintain persistent disruption.

### Likelihood Explanation

- **Precondition**: Miner must be configured with `listen` set to a reachable address. This is an opt-in feature, but it is the recommended low-latency mode for production miners and is documented in the startup log message.
- **Attacker capability**: No credentials, no privileged access, no cryptographic material required. A single HTTP POST with a valid-looking JSON body suffices.
- **Reachability**: If the operator binds to `0.0.0.0:PORT` (common for miners on separate machines from the CKB node), the port is reachable from the network. Even `127.0.0.1` bindings are reachable from any process on the same host (e.g., via SSRF from another service).
- **No existing mitigations**: The `handle()` function contains zero authorization logic. The `parse_authorization` helper exists only for the outbound RPC client (sending requests *to* the CKB node), not for the inbound notification server. [6](#0-5) 

### Recommendation

1. **Allowlist the source IP**: In `listen_block_template_notify`, capture the remote address from `listener.accept()` and reject connections not originating from the configured CKB node's address.
2. **Add a shared secret / HMAC**: Introduce an optional `notify_secret` field in `ClientConfig`. The CKB node includes it as an HTTP header; `handle()` rejects requests missing or failing the check.
3. **Bind to loopback by default**: Change the default example and documentation to recommend `127.0.0.1:PORT` rather than `0.0.0.0:PORT`, and emit a warning when a non-loopback address is configured without an allowlist.
