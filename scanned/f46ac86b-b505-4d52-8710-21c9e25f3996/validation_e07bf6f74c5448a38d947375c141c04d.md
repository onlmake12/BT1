### Title
Unbounded Per-Request DB and MMR Work in `GetLastStateProofProcess::execute` Enables Sustained DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged remote peer can send a single well-formed `GetLastStateProof` message with exactly 1000 difficulties and `last_n_blocks=0`. The guard uses a strict `>` comparison, so `1000 + 0 = 1000` is not `> 1000` and the check passes. The server then performs up to 1000 binary-search DB reads, 1000 MMR root computations, and one `gen_proof(1000 positions)` call — all in a single request — with no rate limiting and no peer ban.

---

### Finding Description

**Guard is off-by-one (strict `>` instead of `>=`):** [1](#0-0) 

`GET_LAST_STATE_PROOF_LIMIT` is 1000: [2](#0-1) 

With `difficulties.len() = 1000` and `last_n_blocks = 0`, the expression evaluates to `1000 + 0 * 2 = 1000`, which is **not** `> 1000`. The guard does not fire, and `MalformedProtocolMessage` (which would trigger a ban) is never returned.

**Work done per passing request:**

1. `get_block_numbers_via_difficulties` iterates all 1000 difficulties, each invoking `get_first_block_total_difficulty_is_not_less_than`, which is a binary search doing O(log N) DB reads per difficulty: [3](#0-2) 

2. `complete_headers` is called with up to 1000 block numbers; for each it calls `chain_root_mmr(*number - 1).get_root()`: [4](#0-3) 

3. `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1000 positions: [5](#0-4) 

**No ban for valid-but-expensive requests:**

The ban logic only fires for 4xx status codes: [6](#0-5) 

A request that passes the guard returns `StatusCode::OK` (200), so `should_ban()` returns `None` and the peer is never penalized regardless of how many such requests it sends.

---

### Impact Explanation

For a chain of N blocks, a single request forces:
- **1000 × O(log N) DB reads** (binary search per difficulty)
- **1000 MMR `get_root()` computations**
- **1 `gen_proof(1000 positions)` call** (O(log N × 1000) work)

An attacker sending 10 concurrent such requests against a node with 1,000,000 blocks triggers ~200,000 DB reads and 10,000 MMR root computations simultaneously, with no back-pressure. This can saturate disk I/O and CPU, starving legitimate peers of responses — a sustained, amplified DoS from a single unauthenticated connection.

---

### Likelihood Explanation

The attack requires only a valid P2P connection and knowledge of the chain's total difficulty range (publicly observable). No PoW, no key, no privileged role. The message structure is well-documented in the light client protocol. Any peer can craft and send this message repeatedly.

---

### Recommendation

1. **Fix the guard to use `>=`** instead of `>`:
   ```rust
   if self.message.difficulties().len() + (last_n_blocks as usize) * 2
       >= constant::GET_LAST_STATE_PROOF_LIMIT
   ```
2. **Add per-peer request rate limiting** for `GetLastStateProof` messages.
3. **Consider banning peers** that send requests resulting in `InvalidRequest` or `InvaildDifficultyBoundary` status codes repeatedly, not only `MalformedProtocolMessage`.

---

### Proof of Concept

1. Spin up a CKB node with a chain of 1,000,000 blocks.
2. Connect as a light-client peer.
3. Send 10 concurrent `GetLastStateProof` messages, each with:
   - `difficulties = [D1, D2, ..., D1000]` — 1000 distinct, monotonically increasing values within the chain's total difficulty range
   - `last_n_blocks = 0`
   - `difficulty_boundary` set above `D1000`
   - `last_hash` = current tip hash
4. Measure: total DB reads ≈ 10 × 1000 × 20 = 200,000; wall-clock time for other peers to receive responses degrades significantly.
5. Repeat indefinitely — no ban is issued.
