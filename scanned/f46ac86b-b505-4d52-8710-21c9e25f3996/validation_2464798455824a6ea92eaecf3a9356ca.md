### Title
Incorrect `total_tx_size`/`total_tx_cycles` Accounting in `add_entry` When Evictions Occur — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the updated pool-size statistics are computed **before** any evictions take place, but are written back to `self` **after** evictions have already modified those same fields in-place. This causes `total_tx_size` and `total_tx_cycles` to be permanently overcounted whenever `check_and_record_ancestors` evicts transactions, directly analogous to the double-spending of deposited gas in the reference report.

---

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```rust
// Step 1 – snapshot new totals into LOCAL variables (before any eviction)
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Step 2 – may call remove_entry_and_descendants → remove_entry →
//           update_stat_for_remove_tx, which MODIFIES self.total_tx_size /
//           self.total_tx_cycles IN PLACE
evicts = self.check_and_record_ancestors(&mut entry)?;

// ... insert, record edges, etc. ...

// Step 3 – OVERWRITES the in-place modifications from Step 2
self.total_tx_size  = total_tx_size;   // ← stale value, ignores evictions
self.total_tx_cycles = total_tx_cycles; // ← stale value, ignores evictions
```

`updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` into a local variable. If `check_and_record_ancestors` subsequently evicts one or more entries (via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`), those subtractions are applied to `self.total_tx_size` in-place. The final assignment on lines 218–219 then blindly overwrites `self.total_tx_size` with the pre-eviction snapshot, erasing the subtraction entirely.

**Concrete accounting trace** (suppose pool starts at `total_tx_size = 100`):

| Step | Action | `self.total_tx_size` | local `total_tx_size` |
|------|--------|----------------------|-----------------------|
| 1 | `updated_stat_for_add_tx(size=10)` | 100 (unchanged) | 110 |
| 2 | evict entry of size 20 via `update_stat_for_remove_tx` | 80 | 110 |
| 3 | `self.total_tx_size = total_tx_size` | **110** (wrong) | — |

Correct value after inserting one entry and evicting one: `80 + 10 = 90`. Actual value: `110`. Overcounted by `20` (the evicted entry's size).

The eviction path inside `check_and_record_ancestors` is triggered when a new transaction's ancestor count exceeds `max_ancestors_count` but can be reduced by evicting "cell-ref parents" — transactions already in the pool that reference the same cell dep as the incoming transaction. [1](#0-0) 

The `update_stat_for_remove_tx` that is silently overwritten: [2](#0-1) 

The `check_and_record_ancestors` eviction path that triggers the in-place modification: [3](#0-2)

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict transactions from the pool:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee transactions
}
``` [4](#0-3) 

When `total_tx_size` is overcounted by `N` bytes (the aggregate size of evicted entries), `limit_size` will expel an additional `N` bytes worth of legitimate, fee-paying transactions from the pool that should not have been removed. Each such eviction is reported as `Reject::Full`, causing those transactions to be dropped and requiring resubmission. Repeated triggering inflates the apparent pool occupancy monotonically, progressively starving the pool of capacity.

`total_tx_cycles` is similarly overcounted, affecting fee-rate estimation and the `tx_pool_info` RPC. [5](#0-4) 

---

### Likelihood Explanation

The vulnerable path is reachable by any unprivileged transaction sender:

1. Attacker submits a chain of transactions that share a common cell dep with existing pool transactions (creating "cell-ref parents").
2. Attacker submits a new transaction that (a) also references that cell dep and (b) has enough in-pool ancestors to exceed `max_ancestors_count` once the cell-ref parents are counted.
3. `check_and_record_ancestors` evicts the cell-ref parents to bring the ancestor count within the limit.
4. The overwrite on lines 218–219 inflates `total_tx_size` by the evicted entries' sizes.
5. `limit_size` then expels legitimate third-party transactions to compensate for the phantom inflation.

No privileged access, key material, or majority hash power is required. The attacker pays only the fees for their own submitted transactions.

---

### Recommendation

Compute the new totals **after** all evictions have completed, not before:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, Default::default()));
    }
    // Validate that adding this entry won't overflow, but do NOT commit yet.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    let evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Compute final totals AFTER evictions have already adjusted self.total_tx_*.
    self.total_tx_size  = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

Alternatively, defer the stat update entirely to after `check_and_record_ancestors` returns, using the already-correct in-place values left by `update_stat_for_remove_tx`.

### Proof of Concept

```
Initial pool state:
  total_tx_size  = 1000
  total_tx_cycles = 10000
  entries: [tx_A (size=200, uses cell_dep_X), tx_B (size=200, uses cell_dep_X),
            ... 24 more ancestors of tx_new ...]

Attacker submits tx_new:
  - spends outputs of 25 in-pool ancestors  → ancestors_count = 26 > max_ancestors_count (25)
  - uses cell_dep_X                         → cell_ref_parents = {tx_A, tx_B}
  - ancestors_count - cell_ref_parents.len() = 24 ≤ 25  → eviction path taken

Inside add_entry:
  Step 1: updated_stat_for_add_tx(size=50)
          local total_tx_size  = 1000 + 50 = 1050
          local total_tx_cycles = 10000 + 500 = 10500

  Step 2: check_and_record_ancestors evicts tx_A (size=200):
          self.total_tx_size  = 1000 - 200 = 800
          self.total_tx_cycles = 10000 - 2000 = 8000

  Step 3: self.total_tx_size  = 1050   ← WRONG (should be 850)
          self.total_tx_cycles = 10500  ← WRONG (should be 8500)

Overcounting: +200 bytes, +2000 cycles

limit_size (max_tx_pool_size = 900) now sees 1050 > 900 and evicts
200 bytes of legitimate third-party transactions unnecessarily.
```

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    /// Calculate size and cycles statistics for adding a tx.
    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

        current_entry_id: Option<&ProposalShortId>,

        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))

                let removed = self.pool_map.remove_entry_and_descendants(&id);
