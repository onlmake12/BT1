### Title
`get_block_filter` RPC Returns `null` for Both Non-Existent Block Hash and Valid Block With No Filter Built Yet — Ambiguous Null Response Indistinguishable by Callers - (File: `rpc/src/module/chain.rs`)

### Summary

The `get_block_filter` RPC handler in `rpc/src/module/chain.rs` returns `Ok(None)` (JSON `null`) for two semantically distinct cases: (1) a block hash that is not on the canonical chain (invalid/non-existent), and (2) a block hash that IS on the canonical chain but whose filter data has not yet been built. Callers — including light clients using the block filter protocol — cannot distinguish between these two cases from the response alone.

### Finding Description

The implementation of `get_block_filter` at `rpc/src/module/chain.rs:1732–1747`:

```rust
fn get_block_filter(&self, block_hash: H256) -> Result<Option<BlockFilter>> {
    let store = self.shared.store();
    let block_hash = block_hash.into();
    if !store.is_main_chain(&block_hash) {
        return Ok(None);   // Case 1: block hash not on canonical chain
    }
    Ok(store.get_block_filter(&block_hash).map(|data| {
        let hash = store
            .get_block_filter_hash(&block_hash)
            .expect("stored filter hash");
        BlockFilter { data: data.into(), hash: hash.into() }
    }))
    // Case 2: block IS on canonical chain but filter not yet built → also Ok(None)
}

The store-level `get_block_filter` returns `Option<packed::Bytes>` — `None` when the filter row is absent from `COLUMN_BLOCK_FILTER`: [2](#0-1) 

Block filter data is built asynchronously by the `block-filter` background task. A block that has been committed to the canonical chain may not yet have its filter row written: [3](#0-2) 

The RPC documentation only says "Returns the block filter data" and shows `null` as the example response without distinguishing the two null cases: [4](#0-3) 

By contrast, other RPC methods in the same file that encounter a similar "index says it exists but data is missing" situation correctly return a `ChainIndexIsInconsistent` error rather than a silent null: [5](#0-4) 

### Impact Explanation

Any RPC caller — including light clients implementing the CKB block filter protocol — that queries `get_block_filter` for a recently committed canonical block may receive `null` and incorrectly conclude the block does not exist on the canonical chain, when in fact the filter simply has not been built yet. This causes the caller to be unable to determine whether:

- The supplied block hash is invalid/off-chain (should be an error or explicit "not found"), or
- The block is valid and on-chain but the filter is pending (a transient state that should be retried).

Light clients relying on block filters for transaction detection may silently skip valid canonical blocks, leading to missed transactions. The ambiguity is directly analogous to the EIP-721 `tokenURI()` issue: a function that should signal an error for invalid inputs instead returns the same empty/null value as a legitimate "no data yet" response.

**Impact: Low** — No consensus or fund-safety impact; affects correctness of light client filter queries.

### Likelihood Explanation

**Likelihood: High** — Any unprivileged RPC caller can trigger this by querying `get_block_filter` for a block hash that was recently committed to the canonical chain before the background filter-builder task has processed it. This is a normal operational window on any running CKB node with block filter enabled. The existing integration test explicitly confirms the ambiguity is observable: [6](#0-5) 

### Recommendation

The `get_block_filter` implementation should distinguish between the two null cases:

1. **Block hash not on canonical chain** → return an explicit error (e.g., `RPCError::invalid_params("block not found in canonical chain")`) rather than `Ok(None)`, consistent with how `get_header_by_number` handles index inconsistencies.
2. **Block on canonical chain but filter not yet built** → return `Ok(None)` (acceptable transient state, callers should retry).

This mirrors the pattern already used in `get_header_by_number` and `get_block_by_number`, which return `ChainIndexIsInconsistent` errors when the index says a block exists but the data cannot be retrieved.

### Proof of Concept

```
# 1. Mine a block so it is on the canonical chain
curl -X POST http://localhost:8114 -d '{"jsonrpc":"2.0","method":"generate_block","params":[],"id":1}'
# → returns block_hash e.g. "0xabc..."

# 2. Immediately query get_block_filter before the background task builds the filter
curl -X POST http://localhost:8114 -d '{"jsonrpc":"2.0","method":"get_block_filter","params":["0xabc..."],"id":2}'
# → {"result": null}   ← indistinguishable from a completely invalid hash

# 3. Query with a completely fabricated hash
curl -X POST http://localhost:8114 -d '{"jsonrpc":"2.0","method":"get_block_filter","params":["0x0000...0000"],"id":3}'
# → {"result": null}   ← same response; caller cannot tell the difference
```

        debug!(

            header.hash()
        );
        let db = self.shared.store();
        if db.get_block_filter_hash(&header.hash()).is_some() {
            debug!(
                "Filter data for block {:#x} already exists. Skip building.",
                header.hash()

            return;
        }
        let parent_block_filter_hash = if header.is_genesis() {
            Byte32::zero()
        } else {
            db.get_block_filter_hash(&header.parent_hash())

        let transactions_size: usize = transactions.iter().map(|tx| tx.data().total_size()).sum();
        let provider = WrappedChainDB::new(db);
        let (filter_data, missing_out_points) = build_filter_data(provider, &transactions);
        for out_point in missing_out_points {

                out_point
            );
        }
        let db_transaction = db.begin_transaction();
        db_transaction
            .insert_block_filter(

                &(filter_data.clone().into()),
                &parent_block_filter_hash,
            )

        db_transaction.commit().expect("commit should be ok");
        debug!(
            "Inserted filter data for block: {}, hash: {:#x}, filter data size: {}, transactions size: {}",
            header.number(),
            header.hash(),
            filter_data.len(),
            transactions_size
        );
    }
