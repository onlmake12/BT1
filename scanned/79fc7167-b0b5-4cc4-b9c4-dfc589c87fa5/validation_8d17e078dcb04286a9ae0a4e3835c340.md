### Title
Silent `u128`-to-`u64` Truncation in `calculate_maximum_withdraw` Breaks DAO Capacity Accounting Invariant - (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` uses a bare `as u64` cast to narrow a `u128` intermediate result, silently discarding the high bits on overflow. Every other analogous `u128`→`u64` narrowing in the same codebase uses `u64::try_from(…).map_err(|_| DaoError::Overflow)?`. The inconsistency means that when the intermediate product exceeds `u64::MAX`, the function can return a silently wrong (too-small) capacity instead of an error, corrupting the on-chain DAO accounting field and the RPC withdrawal estimate.

### Finding Description
In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity as:

```rust
// util/dao/src/lib.rs  L152-L156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
```

The `as u64` cast silently keeps only the low 64 bits of `withdraw_counted_capacity`. If the true mathematical result exceeds `u64::MAX`, the returned capacity is wrong.

Every other `u128`→`u64` narrowing in the same file uses the checked form:

```rust
// L244-L245  (miner_issuance)
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)

// L258  (ar_increase)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;

// L204  (secondary_block_reward)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
```

The truncation produces two distinct failure modes:

1. **Silent wrong result** – when `withdraw_counted_capacity` is just above a multiple of `2^64` (e.g. `2^64 + k` for small `k`), the cast yields `k`. The subsequent `safe_add(occupied_capacity)` does not overflow, so the function returns `Ok(Capacity::shannons(k + occupied_capacity))` — a value orders of magnitude smaller than the correct answer — with no error.

2. **Wrong-reason error** – when the truncated value is still near `u64::MAX`, `safe_add(occupied_capacity)` overflows and returns an error. The error is correct in outcome but masks the real root cause (the truncation), making the code misleading and fragile.

`calculate_maximum_withdraw` is called from two production paths:

- **Block verification** (`dao_field_with_current_epoch` → `withdrawed_interests` → `transaction_maximum_withdraw` → `calculate_maximum_withdraw`): a wrong return value causes `withdrawed_interests` to be too small, inflating `current_s` (the NervosDAO secondary-issuance accumulator stored in the block header's DAO field). This corrupts the on-chain DAO accounting invariant.
- **RPC** (`calculate_dao_maximum_withdraw`): returns a wrong withdrawal estimate to the caller.

### Impact Explanation
If the silent-wrong-result path is triggered during block processing, the DAO field written into the block header contains an inflated `S` value. All subsequent DAO interest calculations that read `parent_s` will compound the error. Users relying on the RPC estimate to construct withdrawal transactions will build transactions with the wrong output capacity, which will be rejected by the on-chain DAO type script. The accounting invariant `S = accumulated_secondary_issuance − withdrawn_interests` is violated without any node raising an error.

### Likelihood Explanation
The overflow condition requires `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. Because `counted_capacity ≤ total_CKB_supply ≈ 3.36 × 10^18` shannons and the accumulate rate ratio `withdrawing_ar / deposit_ar` grows slowly from the genesis value of `10^16`, the ratio must exceed ~5.5× before the overflow is reachable with a maximally-sized cell. This makes the condition practically unreachable on mainnet in the near term. However, the bug is a latent correctness defect: the code is inconsistent with every other analogous conversion in the same file, there is no test that exercises the silent-truncation path (the existing `check_withdraw_calculation_overflows` test only exercises the `safe_add` overflow path), and the condition becomes reachable if the accumulate rate ever grows sufficiently or if the total supply ceiling is approached.

### Recommendation
Replace the bare cast with the same checked conversion used everywhere else in the file:

```rust
// util/dao/src/lib.rs  L155-L156  — replace:
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// with:
Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?
```

Add a unit test that sets `withdraw_counted_capacity` to a value just above `u64::MAX` and asserts that `calculate_maximum_withdraw` returns `Err(DaoError::Overflow)`.
