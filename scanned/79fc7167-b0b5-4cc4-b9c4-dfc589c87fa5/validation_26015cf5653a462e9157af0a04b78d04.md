### Title
Unbounded RocksDB Read Amplification in `GetLastStateProof` Binary Search — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

The `GET_LAST_STATE_PROOF_LIMIT` guard in `GetLastStateProofProcess::execute` only bounds the *count* of difficulty entries in the request (`difficulties.len() + last_n_blocks * 2 ≤ 1000`), but does not bound the total number of RocksDB reads triggered. Each difficulty entry causes `get_first_block_total_difficulty_is_not_less_than` to execute a binary search over the block range, performing O(log H) calls to `get_block_total_difficulty`, each of which issues two RocksDB reads (`get_block_hash` + `get_block_ext`). No per-message or per-peer rate limiting exists on the light client protocol handler. A sustained flood of crafted `GetLastStateProof` messages can therefore saturate RocksDB read I/O.

---

### Finding Description

**Entrypoint:** Any unprivileged peer connected via the P2P `LightClient` protocol can send `GetLastStateProof` messages. No PoW, no authentication, no rate limit.

**Guard that is supposed to bound work:** [1](#0-0) 

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // = 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

This only limits N (the number of difficulty entries). It does not limit the number of storage reads those N entries trigger.

**Where the amplification occurs — the binary search loop:** [2](#0-1) 

For each difficulty, `get_first_block_total_difficulty_is_not_less_than` performs a binary search over `[start_block_number, end_block_number)`. Each iteration calls `get_block_total_difficulty`: [3](#0-2) 

```rust
fn get_block_total_difficulty(&self, number: BlockNumber) -> Option<U256> {
    self.snapshot
        .get_block_hash(number)                          // RocksDB read #1
        .and_then(|h| self.snapshot.get_block_ext(&h))  // RocksDB read #2
        .map(|ext| ext.total_difficulty)
}
```

That is **2 RocksDB reads per binary-search step**, and O(log H) steps per difficulty entry.

**The `start_block_number` advancement does not eliminate worst-case amplification:** [4](#0-3) 

After finding block `num` for difficulty `d_i`, `start_block_number` is set to `num - 1`. An attacker who crafts difficulties that each resolve to a block near the *beginning* of the chain (e.g., blocks 1, 2, 3, …, N) keeps the search range at approximately `[0, H)` for every iteration, yielding O(N × log H) total reads.

**Additional binary search for `difficulty_boundary`** (always executed, unconditionally): [5](#0-4) 

This adds another O(log H) reads per request regardless of the difficulties list.

**No rate limiting on the light client handler:** [6](#0-5) 

The `received` handler dispatches directly to `try_process` with no per-peer message rate limiter. Compare this to the `HolePunching` protocol, which has an explicit `governor::RateLimiter` keyed by peer and message type. The light client protocol has no equivalent. [7](#0-6) 

---

### Impact Explanation

**Reads per request (worst case):**
- N = 1000 difficulties (maximum allowed with `last_n_blocks = 0`)
- H = 10^6 (approximate mainnet chain height)
- log₂(H) ≈ 20
- Binary search reads: 1000 × 20 × 2 = **40,000 RocksDB reads**
- Plus `difficulty_boundary` search: +40 reads
- Plus `complete_headers` (`get_ancestor` + `get_block` per sampled block): another O(N × log H) reads

**Flood scenario:** 100 concurrent connections × 1000 difficulties × 20 log steps × 2 reads = **4,000,000 RocksDB reads per second**, sustained indefinitely. This saturates disk I/O and degrades block sync, transaction relay, and all other peers sharing the same node.

The `GET_LAST_STATE_PROOF_LIMIT = 1000` constant creates a false sense of safety: it bounds output size but not I/O cost.

---

### Likelihood Explanation

- No PoW, no stake, no authentication required — any peer can connect.
- The default `max_peers = 125` allows up to 125 simultaneous inbound connections.
- The attack is trivially scriptable: open connections, send crafted `GetLastStateProof` in a tight loop.
- The attacker needs only to know the chain's total difficulty range (publicly available from any synced node or block explorer) to craft valid, maximally-amplifying difficulty lists.

---

### Recommendation

1. **Bound total binary-search steps, not just difficulty count.** Replace the current guard with one that accounts for the O(log H) factor, e.g., limit `difficulties.len() ≤ GET_LAST_STATE_PROOF_LIMIT / log2(chain_height_estimate)`.
2. **Add per-peer rate limiting** on `GetLastStateProof` messages, analogous to the `governor::RateLimiter` used in the `HolePunching` protocol.
3. **Cap the binary search range** by enforcing a maximum `end_block_number - start_block_number` in `get_first_block_total_difficulty_is_not_less_than`.

---

### Proof of Concept

```
1. Connect to a CKB full node running the LightClient protocol (default port 8115).
2. Obtain the total difficulty of block #1 and block #N (N = chain tip) from any public source.
3. Generate 1000 monotonically increasing U256 difficulty values d_1 < d_2 < ... < d_1000,
   each chosen to fall within [total_diff(block 1), total_diff(block N)],
   spaced so that each d_i resolves to a block near the start of the chain.
4. Craft a GetLastStateProof message:
     last_hash       = current tip hash
     start_hash      = genesis hash
     start_number    = 0
     last_n_blocks   = 0
     difficulty_boundary = total_diff(tip) + 1
     difficulties    = [d_1, ..., d_1000]
5. Send 100 such messages concurrently from 100 connections in a tight loop.
6. Monitor RocksDB read counters (via /proc/<pid>/io or RocksDB statistics).
7. Assert: observed reads/sec >> 1000 (the nominal GET_LAST_STATE_PROOF_LIMIT),
   confirming O(N × log H) amplification per request.
```

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
