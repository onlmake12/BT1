Audit Report

## Title
RPC Thread-Pool Exhaustion via Unbounded `block_in_place` in `get_overview(refresh=4)` — (`rpc/src/module/terminal.rs`)

## Summary
When the Terminal RPC module is enabled, any unauthenticated caller can send concurrent `get_overview` requests with `refresh=4`. Each request synchronously blocks a tokio worker thread inside `block_in_place(|| response.recv())` while waiting for the tx-pool service to process queued messages. If all worker threads are blocked, the tx-pool's async event loop cannot run on any thread, responses never arrive, and the deadlock is self-reinforcing — rendering the entire RPC server unresponsive for up to 30 seconds per wave.

## Finding Description
The `send_message!` macro at `tx-pool/src/service.rs:182` calls `block_in_place(|| response.recv())`, which blocks the calling tokio worker thread in place until the tx-pool event loop dequeues and responds to the message. [1](#0-0) 

`TxPoolController::get_block_template` at line 225 and `TxPoolController::get_tx_pool_info` at line 306 both expand this macro, each issuing a separate blocking call. [2](#0-1) [3](#0-2) 

`TerminalRpcImpl::get_tx_pool_info` (called when `refresh & 0x4 != 0`) makes **two** sequential `block_in_place` calls per invocation: one for `tx_pool.get_tx_pool_info()` (line 582) and one for `self.shared.get_block_template(None, None, None)` (line 590–600), plus a third for `get_total_recent_reject_num` (line 602). [4](#0-3) 

`get_overview` is a synchronous `fn` (not `async fn`) that calls `get_tx_pool_info` unconditionally when the `TX_POOL_INFO` bit is set, with no concurrency cap. [5](#0-4) 

The axum server applies only a global 30-second `TimeoutLayer` — no per-method concurrency limiter, no semaphore, no `spawn_blocking` isolation. [6](#0-5) 

**Deadlock amplification**: The tx-pool event loop is itself an async task on the same shared tokio runtime. Once all worker threads are occupied in `block_in_place`, the event loop cannot be scheduled, so queued `BlockTemplate` and `GetTxPoolInfo` messages are never processed, so the `recv()` calls never return — extending the blockage until the 30-second timeout fires. An attacker can immediately re-flood after each timeout wave.

## Impact Explanation
All tokio worker threads become occupied. Because the tx-pool service runs on the same runtime, a secondary starvation effect extends each blocking window. During the window, **all** RPC methods — `get_tip_block_number`, `send_transaction`, `get_block_template` for miners — time out or queue indefinitely. This maps to **High: Vulnerabilities which could easily crash a CKB node** (RPC layer rendered fully unresponsive, node appears offline to clients and miners).

## Likelihood Explanation
- The Terminal module must be operator-enabled, which narrows exposure. However, it is a documented production monitoring feature, not a debug/test-only flag. [7](#0-6) 
- Once enabled, the endpoint is unauthenticated — no auth middleware is present in the server stack.
- The attacker requires only HTTP access to the RPC port and the ability to issue concurrent requests; no keys, no PoW, no peer relationship.
- Filling the verify queue (to slow `BlockTemplate` responses) is achievable via `send_transaction` with low-fee transactions, but even without it, flooding with enough concurrent requests exhausts the fixed-size thread pool.

## Recommendation
1. Replace `block_in_place(|| response.recv())` in the `send_message!` macro with a proper async oneshot channel (`tokio::sync::oneshot`) and `.await`, so the tokio thread is yielded rather than blocked.
2. Add a `tokio::sync::Semaphore` on `get_overview` to cap simultaneous in-flight requests.
3. Apply a shorter per-call timeout specifically on the `get_block_template` sub-call inside `get_tx_pool_info`.
4. Consider running the Terminal RPC handler on a dedicated `spawn_blocking` thread pool to isolate thread exhaustion from the main RPC runtime.

## Proof of Concept
```python
import asyncio, aiohttp, json

TARGET = "http://127.0.0.1:8114"

async def flood(session, i):
    payload = {"jsonrpc":"2.0","method":"get_overview","params":[4],"id":i}
    async with session.post(TARGET, json=payload) as r:
        return await r.text()

async def check_liveness(session):
    payload = {"jsonrpc":"2.0","method":"get_tip_block_number","params":[],"id":9999}
    try:
        async with session.post(TARGET, json=payload,
                                timeout=aiohttp.ClientTimeout(total=3)) as r:
            return r.status == 200
    except Exception:
        return False

async def main():
    async with aiohttp.ClientSession() as session:
        tasks = [flood(session, i) for i in range(64)]
        asyncio.gather(*tasks)   # fire and forget
        await asyncio.sleep(1)
        alive = await check_liveness(session)
        assert not alive, "RPC should be unresponsive"
        print("CONFIRMED: RPC unresponsive during flood")

asyncio.run(main())
```

macro_rules! send_message {
    ($self:ident, $msg_type:ident, $args:expr) => {{
        let (responder, response) = oneshot::channel();
        let request = Request::call($args, responder);
        $self
            .sender

                let (_m, e) = handle_try_send_error(e);
                e
            })?;
        block_in_place(|| response.recv())
            .map_err(handle_recv_error)

    }};
}

        let refresh = refresh

        let network = self.get_network_info(refresh)?;

            sys,
            cells,
            mining,
            pool,
            network,
