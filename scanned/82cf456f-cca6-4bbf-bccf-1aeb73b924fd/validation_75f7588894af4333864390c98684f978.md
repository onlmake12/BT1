### Title
`check_tx_fee` Uses Serialized Size Only, Ignoring Cycles Multiplier — Tx-Pool Admission Guard Is Too Loose for High-Cycle Transactions - (File: `tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission gate in `check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the actual block-resource cost (weight) is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction with a small serialized size but near-maximum cycles passes the fee gate while paying up to ~60× less than the true resource cost. This is directly analogous to the Deriverse slippage guard: a fixed, size-only cap is applied uniformly regardless of the cycles "multiplier," just as a fixed ±12.5% cap was applied regardless of leverage.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee rate using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The actual transaction weight used everywhere else in the codebase is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` (≈ `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`). [3](#0-2) 

The `check_tx_fee` call happens inside `pre_check`, **before** `verify_rtx` runs and determines the actual cycles:

```rust
let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
// ...
let verified_ret = verify_rtx(..., max_cycles, ...).await;
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
``` [4](#0-3) 

So the fee gate is checked against size alone, but the `TxEntry` is later created with the real cycles. The entry's actual fee rate (used for sorting and eviction) correctly uses `get_transaction_weight(size, cycles)`:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

This creates a split: the **admission gate** uses size-only weight, while **all post-admission logic** (sorting, eviction, fee estimation) uses the full weight including cycles.

---

### Impact Explanation

**Concrete numbers** with default config (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):

| Metric | Value |
|---|---|
| Attacker tx serialized size | 200 bytes |
| Attacker tx cycles | 70,000,000 |
| Actual weight | `max(200, 70_000_000 × 0.000_170_571_4)` = **11,940** |
| Fee required by `check_tx_fee` | `1000 × 200 / 1000` = **200 shannons** |
| Fee that should be required | `1000 × 11,940 / 1000` = **11,940 shannons** |
| Discount factor | **~60×** |

An unprivileged attacker can:

1. **Flood the tx-pool** with high-cycle, low-size transactions at ~60× below the intended fee floor, consuming pool memory slots.
2. **Amplify network relay cost**: relayed transactions also pass through `check_tx_fee` (via `process_tx` → `pre_check`). Nodes relay these transactions to peers, consuming bandwidth and peer CPU for script verification, all at a fraction of the intended relay cost.
3. **Distort fee estimation**: the `weight_units_flow` and `confirmation_fraction` estimators both use `get_transaction_weight` correctly, but the admission gate does not, creating an inconsistency where the estimator recommends higher fees than what is actually enforced at the gate. [6](#0-5) 

---

### Likelihood Explanation

**High.** The attacker entry path requires no privilege:

- **RPC path**: `send_transaction` RPC → `submit_local_tx` → `process_tx` → `pre_check` → `check_tx_fee`. Any RPC caller can submit a transaction.
- **P2P relay path**: `submit_remote_tx` → `process_tx` → `pre_check` → `check_tx_fee`. Any peer can relay a transaction. [7](#0-6) 

Crafting the exploit is straightforward: deploy a lock script that loops for many cycles (e.g., a tight RISC-V loop), then create a transaction spending a cell locked by that script with minimal inputs/outputs to keep serialized size small. The script can be deployed once and reused across many attack transactions.

The code comment itself acknowledges the gap: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check."* [8](#0-7) 

---

### Recommendation

Replace the size-only fee check in `check_tx_fee` with a weight-aware check. Since cycles are not yet known at `pre_check` time (script verification hasn't run), two approaches are viable:

1. **Use a declared-cycles lower bound**: For remote transactions, the caller already declares cycles (`declared_cycles`). Use `get_transaction_weight(tx_size, declared_cycles)` for the fee gate when a declared value is available.

2. **Re-check fee after verification**: After `verify_rtx` returns the actual cycles, perform a second fee check using `get_transaction_weight(tx_size, verified.cycles)` before calling `submit_entry`. This is the most accurate approach and closes the gap entirely.

```rust
// After verify_rtx:
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

### Proof of Concept

**Setup:**
- Deploy a lock script that executes a tight RISC-V loop consuming ~69,000,000 cycles (just under `max_tx_verify_cycles = 70,000,000`).
- Create a cell locked by this script with capacity 100 CKB.

**Attack transaction:**
- 1 input (the cell above), 1 output (same capacity minus fee)
- Serialized size ≈ 200 bytes
- Fee paid: `1000 × 200 / 1000 = 200 shannons` (passes `check_tx_fee`)
- Actual weight: `max(200, 69_000_000 × 0.000_170_571_4) ≈ 11,769`
- Actual fee rate: `200 × 1000 / 11,769 ≈ 17 shannons/KW` (far below `min_fee_rate = 1000`)

**Result:** The transaction is admitted to the pool and relayed to all peers. Each peer spends ~69M cycles of CPU verifying it. The attacker pays 200 shannons instead of the ~11,769 shannons that the `min_fee_rate` policy intends to require, achieving a ~59× cost reduction per attack transaction. Repeating this with many UTXOs floods the network at a fraction of the intended cost. [9](#0-8) [10](#0-9)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

/// The precision is set so that the difference between MAX_BLOCK_CYCLES * DEFAULT_BYTES_PER_CYCLES
/// and MAX_BLOCK_BYTES is less than 1.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"

pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
