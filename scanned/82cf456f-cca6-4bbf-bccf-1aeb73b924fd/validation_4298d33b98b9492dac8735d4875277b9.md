### Title
Fee Rate Admission Check Uses Transaction Size Instead of Actual Weight, Allowing High-Cycle Transactions to Bypass Minimum Fee Rate — (`tx-pool/src/util.rs`)

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate by computing `min_fee = min_fee_rate.fee(tx_size)`, using only the serialized byte size as the weight denominator. However, the actual weight used throughout block assembly and pool scoring is `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. When a transaction's cycle count dominates, the actual weight is far larger than `tx_size`, meaning the real fee rate is far below the minimum. Because the fee check runs before script verification (before cycles are known), and no second check is performed after cycles are determined, a transaction with high cycles and small size can enter the pool with an actual fee rate orders of magnitude below `min_fee_rate`.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

The code itself acknowledges the imprecision with the comment: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."*

The actual weight used everywhere else — pool scoring, block assembly, eviction — is defined in `util/types/src/core/tx_pool.rs`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`.

The processing pipeline in `_process_tx` (`tx-pool/src/process.rs`) is:

1. `pre_check` → `check_tx_fee(tx_pool, snapshot, rtx, tx_size)` — fee check using **size only**
2. `verify_rtx(...)` — script execution, actual cycles determined here
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with actual cycles
4. `submit_entry(...)` — added to pool

There is **no second fee rate check** after step 2 that uses the actual cycles. The `declared_cycles` value available in `_process_tx` is never forwarded to `check_tx_fee`.

**Concrete example:**
- `tx_size = 1,000` bytes
- `cycles = 70,000,000` (the `max_tx_verify_cycles` default)
- `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`
- `actual_weight = max(1000, 70_000_000 × 0.000_170_571_4) = max(1000, 11,940) = 11,940`
- `min_fee_rate = 1,000 shannons/KW`
- Fee check threshold: `1,000 × 1,000 / 1,000 = 1,000 shannons`
- Actual minimum fee needed: `1,000 × 11,940 / 1,000 = 11,940 shannons`

A transaction paying 1,001 shannons passes `check_tx_fee` but has an actual fee rate of `≈ 83 shannons/KW` — roughly 12× below the minimum.

---

### Impact Explanation

An unprivileged attacker (RPC caller or transaction relay peer) can submit transactions that:

1. Pass the size-based fee check with a small fee
2. Force the node to execute expensive script verification (up to `max_tx_verify_cycles = 70,000,000` cycles)
3. Enter the tx pool with an actual fee rate far below `min_fee_rate`
4. Persist in the pool, consuming memory and polluting pool ordering, but never being mined

This is a CPU and memory griefing attack. The attacker pays a fee proportional to `tx_size` but causes the node to perform work proportional to `cycles`. The pool eviction mechanism (which uses actual weight) will eventually remove these transactions when the pool is full, but the verification CPU cost is already paid by the node.

---

### Likelihood Explanation

- Entry path is fully open: any `send_transaction` RPC caller or relaying peer can submit such a transaction
- No special privileges, keys, or majority hashpower required
- The attack is cheap: a 1,000-byte transaction with 70M cycles costs only ~1,000 shannons to pass the fee check
- The attack is repeatable: each submission forces a full script verification pass
- The code comment explicitly acknowledges the imprecision, confirming this is a known gap rather than an intentional design

---

### Recommendation

After `verify_rtx` determines the actual cycles, perform a second fee rate check using the true weight:

```rust
// After verify_rtx returns verified.cycles:
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Alternatively, for remote transactions where `declared_cycles` is available before verification, use `get_transaction_weight(tx_size, declared_cycles)` in `check_tx_fee` as a tighter pre-check.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
