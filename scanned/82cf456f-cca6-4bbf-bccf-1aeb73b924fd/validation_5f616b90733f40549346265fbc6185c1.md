### Title
Hole-Punching `forward_message` Gossip Amplification via Distinct `to` PeerId Bypass of `forward_rate_limiter` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

An unprivileged remote peer can send up to 30 `ConnectionRequest` messages per second, each with a distinct unknown `to` PeerId, causing the victim node to call `filter_broadcast(sqrt(K) peers)` for every message. The `forward_rate_limiter` — the only per-forwarding guard — is keyed on `(from, to, msg_item_id)` and is trivially bypassed by rotating `to`. The outer session rate limiter caps inbound at 30/sec but does not prevent the sqrt(K) amplification per message. No ban is triggered for the attacker.

---

### Finding Description

**Two rate limiters exist:**

**1. Outer `rate_limiter`** — keyed by `(session_id, msg.item_id())`: [1](#0-0) 

For `ConnectionRequest`, `item_id()` is always `0` (the union discriminant). Quota: **30 requests/second** per `(session_id, 0)`. This is the only absolute cap on inbound rate from a single peer. [2](#0-1) 

**2. Inner `forward_rate_limiter`** — keyed by `(content.from, content.to, self.msg_item_id)`: [3](#0-2) 

Quota: **1 request/second** per `(from, to, 0)` triple. This is intended to prevent repeated forwarding of the same request. However, if the attacker uses a **distinct `to` PeerId per message**, each message produces a unique key `(from, to_1, 0)`, `(from, to_2, 0)`, etc. — the limiter never fires. [4](#0-3) 

**The broadcast path when `to` is unknown:**

When `to` is not found in the peer registry, `forward_message` calls `filter_broadcast` targeting `sqrt(K)` peers: [5](#0-4) 

**No ban is triggered for the attacker.** `TooManyRequests = 110` is a 1xx code; `should_ban()` only fires for 4xx (400–499): [6](#0-5) 

---

### Impact Explanation

With K connected peers and the attacker sending 30 messages/second (the outer cap):

- **Per-node amplification**: 30 inbound → 30 × floor(sqrt(K)) outbound per second
- K=100: 300 outbound/sec; K=1000: ~930 outbound/sec; K=10000: 3000 outbound/sec

**Cascading amplification**: Each of the sqrt(K) peers that receives the forwarded message also sees an unknown `to`, passes its own `forward_rate_limiter` (new key), and re-broadcasts to sqrt(K') more peers. With `MAX_HOPS = 6`, the theoretical fan-out is sqrt(K)^6 across the network: [7](#0-6) 

The attacker is never banned (1xx status code), so the attack is sustainable indefinitely from a single connection.

---

### Likelihood Explanation

- Requires only a single valid P2P connection — no privilege, no PoW, no key material
- The bypass is trivial: generate random 32-byte `to` PeerIds per message
- The outer 30/sec cap is the only real throttle; it does not prevent amplification, only bounds the absolute inbound rate
- The `forward_rate_limiter` design intent (1/sec per `(from, to)`) is completely defeated by key rotation

---

### Recommendation

1. **Rate-limit by sender session, not by `(from, to)` tuple**, for the forwarding path. A per-session cap of N forwards/second regardless of `to` would close the bypass.
2. **Add a global outbound broadcast budget** for the hole-punching protocol to cap total `filter_broadcast` calls per second.
3. **Treat `TooManyRequests` as a bannable offense** after repeated violations, or at minimum disconnect the session.
4. Consider **requiring `to` to be a known peer** before forwarding, or limiting forwarding to nodes that have previously announced the `to` peer via discovery.

---

### Proof of Concept

```
1. Connect to victim node via P2P (single session).
2. In a loop at 30 msg/sec:
   - Generate a fresh random 32-byte PeerId as `to` (not in victim's peer registry)
   - Set `from` = attacker's own PeerId (valid)
   - Set `max_hops` = 6, `route` = [], `listen_addrs` = [one valid TCP addr]
   - Send ConnectionRequest message
3. Observe: victim calls filter_broadcast(sqrt(K) peers) for each message.
4. Differential test: compare outbound message count with unknown `to` vs known `to`.
   - Known `to`: 1 outbound (send_message_to)
   - Unknown `to`: floor(sqrt(K)) outbound (filter_broadcast)
   - Ratio = floor(sqrt(K)), confirming amplification.
5. With K=100: 30 inbound → 300 outbound/sec sustained, no ban, indefinitely.
```
