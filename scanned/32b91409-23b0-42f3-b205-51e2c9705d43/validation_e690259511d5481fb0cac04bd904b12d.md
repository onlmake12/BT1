### Title
Unconditional `.expect()` on `txs_sizes` in `FeeRateCollector::statistics` Panics on Legacy `BlockExt` Records — (`rpc/src/util/fee_rate.rs`)

---

### Summary

`FeeRateCollector::statistics`, called by both `get_fee_rate_statistics` and `get_fee_rate_statics` RPCs, unconditionally calls `.expect()` on `BlockExt.txs_sizes`. Any block stored in the legacy 5-field `BlockExt` format (introduced before v0.106) deserializes with `txs_sizes = None`. When the RPC's iteration window covers such a block, the `.expect()` panics, crashing the RPC handler thread.

---

### Finding Description

**The panic site** — `rpc/src/util/fee_rate.rs:93`:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
``` [1](#0-0) 

There is no `None` guard before this line. If `txs_sizes` is `None`, the thread panics unconditionally regardless of the `target` value supplied by the caller.

**How `txs_sizes` becomes `None`** — Two storage formats exist:

- Legacy `BlockExt` (5-field, pre-v0.106): deserialized by `packed::BlockExtReader::into()`, which hardcodes `txs_sizes: None`.
- `BlockExtV1` (7-field, v0.106+): deserialized by `packed::BlockExtV1Reader::into()`, which reads the stored `Uint64VecOpt`. [2](#0-1) 

The store's `get_block_ext` dispatches on `count_extra_fields()`:

```rust
match reader.count_extra_fields() {
    0 => reader.into(),   // legacy path → txs_sizes: None
    2 => packed::BlockExtV1Reader::from_slice_should_be_ok(slice.as_ref()).into(),
    ...
}
``` [3](#0-2) 

The schema confirms the two formats: [4](#0-3) 

**No migration rewrites legacy records.** Inspecting all migration modules, none converts `BlockExt` (5-field) to `BlockExtV1` (7-field): [5](#0-4) 

**The iteration window** — `collect` iterates from `max(1, tip_number + 1 − target)` to `tip_number`: [6](#0-5) 

For a node whose tip is below 101 (e.g., a node that was first synced before v0.106 and has not yet passed block 101), the window includes early blocks stored in legacy format. With `target` capped at 101, any such node is vulnerable.

**RPC entry point** — `get_fee_rate_statistics` and `get_fee_rate_statics` are in the `Chain` module, enabled by default, with no authentication: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A Rust `.expect()` failure triggers a panic that unwinds the calling thread. In the jsonrpc-core thread pool, this kills the worker thread handling the request. Depending on whether the server uses `catch_unwind`, repeated calls can exhaust the thread pool or crash the process entirely, rendering the node's RPC interface non-functional (local DoS). Scope: **Note — Any local command line crash (0–500 pts)**.

---

### Likelihood Explanation

- The Chain RPC module is enabled by default; no credentials are required.
- The RPC binds to `127.0.0.1:8114` by default, so the attacker needs local (unprivileged) access to the machine.
- Any node whose chain tip was below 101 at the time it was running pre-v0.106 software retains legacy `BlockExt` records in its RocksDB that are never migrated.
- The trigger is a single unauthenticated HTTP POST with any `target` value.

---

### Recommendation

Replace the unconditional `.expect()` with a graceful `None` skip:

```rust
// Instead of:
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");

// Use:
let Some(txs_sizes) = txs_sizes else { return fee_rates; };
```

This makes `statistics` silently skip legacy blocks (which carry no fee-rate data anyway) rather than panicking.

---

### Proof of Concept

```bash
# 1. Start a CKB node that has legacy BlockExt records
#    (tip < 101 on a pre-v0.106 database, or a freshly initialized node
#     whose genesis/early blocks were stored in 5-field BlockExt format)

# 2. Call the RPC with any target value
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"get_fee_rate_statistics","params":[]}'

# Result: the RPC worker thread panics at:
#   rpc/src/util/fee_rate.rs:93
#   thread '<unnamed>' panicked at 'expect txs_size's length >= 1'
# The node's RPC becomes unresponsive or the process exits.
```
