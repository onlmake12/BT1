### Title
Incorrect `CKB2021_START_EPOCH` Constant Set to `0` Instead of Historical Activation Epoch, Causing Incorrect Hardfork Rule Application — (`util/constant/src/hardfork/mainnet.rs`, `util/constant/src/hardfork/testnet.rs`)

---

### Summary

`CKB2021_START_EPOCH` is hardcoded to `0` in both the mainnet and testnet hardfork constant files, while the comments in those same files explicitly document the correct historical activation epochs as `5414` (mainnet) and `3113` (testnet). These constants are consumed by `spec/src/hardfork.rs` to configure the activation epoch for multiple CKB2021 consensus rules (RFC0029, RFC0030, RFC0031, RFC0036, RFC0038). Setting the activation epoch to `0` causes the node to treat these rules as active from genesis, rather than from their true on-chain activation point.

---

### Finding Description

In `util/constant/src/hardfork/mainnet.rs`:

```rust
/// First epoch number for CKB v2021, at about 2022/05/10 1:00 UTC
// pub const CKB2021_START_EPOCH: u64 = 5414;
pub const CKB2021_START_EPOCH: u64 = 0;
``` [1](#0-0) 

In `util/constant/src/hardfork/testnet.rs`:

```rust
/// First epoch number for CKB v2021, at about 2021/10/24 3:15 UTC.
// pub const CKB2021_START_EPOCH: u64 = 3113;
pub const CKB2021_START_EPOCH: u64 = 0;
``` [2](#0-1) 

The commented-out lines are the historically correct values. The active constants are `0`.

These constants flow directly into `spec/src/hardfork.rs` via `complete_mainnet()` and `complete_testnet()`:

```rust
ckb2021 = self.update_2021(
    ckb2021,
    mainnet::CKB2021_START_EPOCH,                          // = 0 (wrong)
    mainnet::RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH,  // = 5414 (correct)
)?;
``` [3](#0-2) 

The `update_2021` helper assigns `CKB2021_START_EPOCH` to RFC0029, RFC0030, RFC0031, RFC0036, and RFC0038:

```rust
let builder = builder
    .rfc_0028(rfc_0028_0032_0033_0034_start)
    .rfc_0029(ckb2021)
    .rfc_0030(ckb2021)
    .rfc_0031(ckb2021)
    .rfc_0032(rfc_0028_0032_0033_0034_start)
    .rfc_0036(ckb2021)
    .rfc_0038(ckb2021);
``` [4](#0-3) 

`CKB2021::new_mirana()` in `util/types/src/core/hardfork/ckb2021.rs` also directly reads `hardfork::mainnet::CKB2021_START_EPOCH` for the same five RFCs: [5](#0-4) 

The affected rules include:
- **RFC0030** (`check_length_in_epoch_since`): a **stricter** validation rule — applying it from epoch 0 causes the node to reject historical blocks (epoch 0–5414 on mainnet) whose transactions were valid under the pre-fork rules.
- **RFC0031** (`reuse_uncles_hash_as_extra_hash`): changes block structure validation.
- **RFC0038** (`disallow_over_max_dep_expansion_limit`): a **stricter** rule — same rejection risk for historical blocks.
- RFC0029, RFC0036: relaxations (lower severity, but still incorrect).

Note: `RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH` (VM version selection, the most security-critical feature) is still correctly set to `5414`/`3113`.

---

### Impact Explanation

A node syncing from genesis applies RFC0030 and RFC0038 validation from epoch 0 instead of epoch 5414 (mainnet). Any historical block between epoch 0 and 5414 containing a transaction that was valid under pre-fork rules but invalid under RFC0030/RFC0038 will be **rejected** by this node. This causes a consensus split: the affected node diverges from the canonical chain and cannot sync mainnet history correctly. An unprivileged peer or sync initiator can trigger this simply by causing the node to sync from genesis (e.g., fresh node startup, database wipe, or IBD).

---

### Likelihood Explanation

Any operator running a fresh CKB node (initial block download from genesis) is affected. This is a standard operational scenario — new nodes, archive nodes, and nodes recovering from database corruption all sync from genesis. The incorrect constant is compiled into the binary and cannot be overridden at runtime without a code change.

---

### Recommendation

Restore the correct historical activation epochs:

```rust
// util/constant/src/hardfork/mainnet.rs
pub const CKB2021_START_EPOCH: u64 = 5414;

// util/constant/src/hardfork/testnet.rs
pub const CKB2021_START_EPOCH: u64 = 3113;
```

These match the values documented in the comments and in `RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH`, which was not incorrectly zeroed.

---

### Proof of Concept

1. Build CKB from this repository.
2. Start a fresh node configured for mainnet (no existing database).
3. The node begins IBD (Initial Block Download) from genesis.
4. The node's hardfork switch has RFC0030 active from epoch 0 (due to `CKB2021_START_EPOCH = 0`).
5. When the node processes a historical block in epoch 1–5413 containing a transaction with an epoch-based `since` field that was valid under pre-fork rules but fails RFC0030's stricter length check, the node rejects the block.
6. The node is permanently unable to sync the canonical mainnet chain from genesis, diverging at the first such block.
