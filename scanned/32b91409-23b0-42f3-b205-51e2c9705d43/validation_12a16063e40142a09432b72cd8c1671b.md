### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `from` PeerId ‚Äî (`network/src/protocols/hole_punching/mod.rs` + `connection_request.rs`)

---

### Summary

The `HolePunching` protocol uses a two-layer rate limiter. The outer `rate_limiter` is keyed by `(session_id, item_id)` and caps a single session at 30 `ConnectionRequest` messages per second. The inner `forward_rate_limiter` is keyed by `(from: PeerId, to: PeerId, item_id: u32)`. Because the `from` field is taken directly from message content without being validated against the session's actual peer identity, an attacker can send 30 messages/sec each with a distinct fabricated `from` PeerId, creating 30 new `HashMapStateStore` buckets per second. `retain_recent()` is only called in `disconnected()`, never in the periodic `notify()` handler, so the store grows without bound for the lifetime of the connection.

The `notify()` handler (fired every `CHECK_INTERVAL` = 5 minutes) cleans up `pending_delivered` and `inflight_requests` but does **not** call `retain_recent()` on either rate limiter: [8](#0-7)

### Impact Explanation

Each `HashMapStateStore` entry holds two `PeerId` values (~39 bytes each) plus a `u32` and governor state (~8 bytes), totalling ~90 bytes. At 30 entries/sec:

| Duration | Entries | Approx. memory |
|---|---|---|
| 5 min (one `CHECK_INTERVAL`) | 9,000 | ~810 KB |
| 1 hour | 108,000 | ~9.7 MB |
| 24 hours | 2,592,000 | ~233 MB |

With multiple attacker sessions the rate multiplies linearly. The store is never pruned until the attacker voluntarily disconnects, making this an unbounded memory leak that can exhaust node memory and cause an OOM crash.

**Scope note:** The direct impact is **memory exhaustion / DoS** (node crash), not consensus deviation. A crashed node falls off the network and misses blocks, which is an availability failure rather than a consensus rule violation.

---

### Likelihood Explanation

- The attack requires only a single persistent P2P connection ‚Äî no special privileges, no PoW, no keys.
- The `from` field is fully attacker-controlled and is never cross-checked against the session's authenticated peer ID.
- The outer rate limiter is the only throughput cap, and it is set to exactly 30/sec ‚Äî the same rate that drives the growth.
- The attack is sustainable indefinitely; the only natural cleanup is a disconnect that the attacker controls.

---

### Recommendation

1. **Call `retain_recent()` in `notify()`** on both `rate_limiter` and `forward_rate_limiter` every `CHECK_INTERVAL`, mirroring what `disconnected()` already does.
2. **Validate `from` against session identity**: reject any `ConnectionRequest` where `content.from` does not match the peer ID of the sending session.
3. **Cap `HashMapStateStore` size**: add a hard upper bound on the number of live keys in `forward_rate_limiter` and drop/ban the session if it is exceeded.

---

### Proof of Concept

```
invariant: after T seconds of sustained attack from one session,
  forward_rate_limiter.len() == 30 * T   (grows without bound)

expected invariant (if correctly bounded):
  forward_rate_limiter.len() <= outer_rate_limit * CHECK_INTERVAL_secs
                              == 30 * 300 == 9000  (and pruned every 5 min)

test sketch:
  1. Connect one session.
  2. Send 30 ConnectionRequest/sec for 600 s, each with a fresh random `from` PeerId.
  3. Assert forward_rate_limiter.len() == 18,000 (not pruned).
  4. Trigger notify() ‚Äî assert len is unchanged (no retain_recent call).
  5. Trigger disconnected() ‚Äî assert len drops to 0.
```
