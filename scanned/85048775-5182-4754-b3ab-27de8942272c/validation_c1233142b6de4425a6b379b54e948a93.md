Audit Report

## Title
Deflated `declared_cycles` in `RelayTransaction` Leaves Tx Hash Stuck in Known Filter, Preventing Re-relay — (`sync/src/relayer/transactions_process.rs`, `tx-pool/src/process.rs`)

## Summary
The `RelayTransaction` P2P message carries a peer-supplied `cycles` field that is not covered by the transaction hash. When a malicious peer sends a valid transaction with `cycles` set below the actual execution cost, `verify_rtx` aborts with `ExceededMaximumCycles` before the `DeclaredWrongCycles` comparison is reached. Unlike `DeclaredWrongCycles`, this rejection path does not trigger `TxVerificationResult::Reject`, so `remove_from_known_txs` is never called. The tx hash remains in `tx_filter` for the full 4-hour TTL, silently dropping all subsequent legitimate relays of the same transaction from any peer.

## Finding Description
In `TransactionsProcess::execute`, `mark_as_known_txs` is called at line 76 before `submit_remote_tx` at line 87. Inside `_process_tx`, `max_cycles = declared_cycles` (line 720) is passed to `verify_rtx`. If `declared_cycles < actual_cycles`, `verify_rtx` returns `Err(ExceededMaximumCycles)`, and `try_or_return_with_snapshot!` at line 734 exits before the `DeclaredWrongCycles` comparison at lines 736–749 is ever reached.

In `after_process`, `ExceededMaximumCycles` is a `Script`-kind error, so `is_malformed_tx()` returns `true` and `ban_malformed` is called — the peer **is** banned. However, `is_allowed_relay()` returns `false` for malformed rejections (it only returns `true` for `DeclaredWrongCycles` or non-malformed errors), so `TxVerificationResult::Reject` is not sent to the relayer and `remove_from_known_txs` is never invoked. For `DeclaredWrongCycles`, `is_allowed_relay()` explicitly returns `true`, causing `remove_from_known_txs` to be called and allowing re-relay. This asymmetry means deflated cycles leave the tx hash permanently stuck in the filter until TTL expiry.

## Impact Explanation
A valid, fee-paying transaction can be prevented from entering the mempool of a target node for up to 4 hours (`FILTER_TTL = 4 * 60 * 60`). During this window, all legitimate relays of the same transaction from any peer are silently discarded by the `tx_filter` check. This constitutes temporary, targeted transaction censorship. The attacker is banned per IP (3 days) but can use different network identities. This matches the allowed impact class of a bad design that can cause targeted transaction suppression, at low-to-medium severity.

## Likelihood Explanation
Any unprivileged peer can execute this attack by announcing the target tx hash via `RelayTransactionHashes` before other peers, waiting for the node to issue `GetRelayTransactions` (establishing them as the designated source), then responding with `cycles = 1`. No special privileges, keys, or hashpower are required. The attack is repeatable across different IPs and is reachable via the standard `RelayV3` P2P protocol.

## Recommendation
When `verify_rtx` returns `ExceededMaximumCycles` and a remote `declared_cycles` was provided, treat it identically to `DeclaredWrongCycles`: call `send_result_to_relayer(TxVerificationResult::Reject { tx_hash })` so that `remove_from_known_txs` is invoked, allowing legitimate peers to re-relay the transaction. Alternatively, always execute with `max_block_cycles` as the cap and compare the result against `declared_cycles` afterward, eliminating the asymmetry entirely.

## Proof of Concept
1. Attacker peer connects to target node via `RelayV3`.
2. Attacker sends `RelayTransactionHashes` with the hash of a valid high-cycle transaction before any other peer.
3. Node responds with `GetRelayTransactions` for that hash, designating the attacker as the source.
4. Attacker sends `RelayTransactions` with `cycles = 1` and the correct transaction body.
5. In `transactions_process.rs` line 76, `mark_as_known_txs` marks the hash as known.
6. In `_process_tx`, `max_cycles = 1`; `verify_rtx` aborts with `ExceededMaximumCycles`; `DeclaredWrongCycles` branch is never reached; `is_allowed_relay()` returns `false`; `remove_from_known_txs` is never called.
7. Attacker peer is banned (3 days), but the tx hash remains in `tx_filter` for 4 hours.
8. Any subsequent `RelayTransactions` from legitimate peers carrying the same tx hash is silently dropped by the `tx_filter.contains` check at line 50 of `transactions_process.rs`.

fn is_malformed_from_verification(error: &Error) -> bool {
    match error.kind() {
        ErrorKind::Transaction => error
            .downcast_ref::<TransactionError>()
            .expect("error kind checked")
            .is_malformed_tx(),
        ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
        ErrorKind::Internal => {
            error
                .downcast_ref::<InternalError>()
                .expect("error kind checked")
                .kind()
                == InternalErrorKind::CapacityOverflow
        }
        _ => false,
    }
}

impl Reject {
    /// Returns true if the reject reason is malformed tx.
    pub fn is_malformed_tx(&self) -> bool {
        match self {
            Reject::Malformed(_, _) => true,
            Reject::DeclaredWrongCycles(..) => true,
            Reject::Verification(err) => is_malformed_from_verification(err),
            Reject::Resolve(OutPointError::OverMaxDepExpansionLimit) => true,
            _ => false,
        }
    }
