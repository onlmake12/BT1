Audit Report

## Title
Unauthenticated Block Template Injection via Miner Notify HTTP Server — (File: miner/src/client.rs)

## Summary
When the CKB miner is configured in notify mode, `listen_block_template_notify` binds a plain HTTP server that accepts `BlockTemplate` payloads from any TCP client. The `handle` free function deserializes the body and calls `update_block_template` with no authentication, IP filtering, or token verification. An attacker who can reach the port can redirect all mining hashpower to an attacker-controlled cellbase, stealing block rewards or causing continuous PoW waste.

## Finding Description
`Client::spawn_background` (lines 204–232) starts the inbound HTTP server when `config.listen` is set. Every accepted connection is dispatched to the `handle` free function (lines 358–369):

```rust
async fn handle(client: Client, req: Request<hyper::body::Incoming>)
    -> Result<Response<Empty<Bytes>>, Error>
{
    let body = BodyExt::collect(req).await?.aggregate();
    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);   // no auth check
    }
    Ok(Response::new(Empty::new()))
}
```

`parse_authorization` (lines 380–394) is invoked only when building outbound RPC requests (line 77); it is never consulted on the inbound path. `update_block_template` (lines 293–311) fires whenever the incoming `work_id` differs from the stored atomic value — a field the attacker controls freely — and immediately sends `Works::New(work)` to all mining workers. The `listen` field is an unconstrained `Option<SocketAddr>` (miner.rs config line 29); no loopback restriction is enforced anywhere in the code.

## Impact Explanation
**Mining reward theft** matches the allowed CKB bounty impact "Vulnerabilities which could easily damage CKB economy" (Critical, 15001–25000 points). An attacker injects a `BlockTemplate` whose `cellbase` output carries the attacker's lock script. The miner solves valid PoW on that template and submits it; the block reward is paid to the attacker's address. At pool scale, this constitutes systematic theft of CKB block rewards, directly damaging the CKB mining economy. A secondary impact is continuous PoW DoS: injecting templates with invalid `parent_hash` or stale block numbers causes all hashpower to be wasted on blocks the node will reject on `submit_block`.

## Likelihood Explanation
Notify mode is the recommended low-latency mining configuration. The default example (`127.0.0.1:8888`) uses loopback, but pool operators and cloud mining setups routinely bind services to `0.0.0.0` or a specific non-loopback interface to receive notifications from a remote CKB node. Any process or peer that can reach the TCP port — co-located processes, LAN peers, or remote attackers if the port is exposed — can exploit this with a single HTTP POST. No credentials, no prior knowledge of internal state, and no special tooling are required.

## Recommendation
In the `handle` function, verify the request source before calling `update_block_template`:

1. Extract the peer address from the accepted TCP connection in `listen_block_template_notify` and pass it into `handle`; compare it against the host portion of `config.rpc_url` and reject any connection from a different source.
2. Alternatively, require a shared-secret token in a request header (e.g., `X-CKB-Notify-Token`), configured in `ckb-miner.toml` and set by the CKB node's notifier, verified in `handle` before deserialization.
3. At minimum, enforce at startup that `config.listen` is a loopback address and abort with a clear error if a non-loopback address is supplied.
