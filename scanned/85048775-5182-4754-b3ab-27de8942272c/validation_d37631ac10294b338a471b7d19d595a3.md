Audit Report

## Title
Single Peer Can Monopolize the Shared VerifyQueue Budget, Blocking All Remote Transaction Submissions — (File: `tx-pool/src/component/verify_queue.rs`)

## Summary
`VerifyQueue` enforces a single global byte-budget (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256 MB`) with no per-peer sub-limit. A single malicious connected peer can fill the entire queue by announcing transaction hashes, waiting for the node to request them, then delivering large transactions. Once the queue is saturated, every subsequent `submit_remote_tx` call from any peer returns `Reject::Full`, blocking all honest remote transaction submissions until the attacker disconnects or the queue drains naturally.

## Finding Description

**Root cause — global-only budget in `VerifyQueue`:**

`VerifyQueue` tracks a single `total_tx_size` counter. `is_full()` compares only against the global constant:

```rust
// tx-pool/src/component/verify_queue.rs L103-106
pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}
```

`add_tx()` returns `Reject::Full` for every caller once this global limit is reached, regardless of which peer contributed the blocking bytes. No per-peer byte counter exists anywhere in `VerifyQueue`.

**Reachable exploit path:**

The relay path is: `RelayTransactionHashes` → `TransactionHashesProcess` → `add_ask_for_txs` → node sends `GetRelayTransactions` → `RelayTransactions` → `TransactionsProcess::execute()` → `submit_remote_tx()` → `resumeble_process_tx()` → `non_contextual_verify()` → `enqueue_verify_queue()` → `verify_queue.add_tx()`.

The filter in `TransactionsProcess::execute()` (lines 49–55) requires the node to have previously requested the tx:

```rust
// sync/src/relayer/transactions_process.rs L49-55
.filter(|(tx, _)| {
    !tx_filter.contains(&tx.hash())
        && unknown_tx_hashes
            .get_priority(&tx.hash())
            .map(|priority| priority.requesting_peer() == Some(self.peer))
            .unwrap_or_default()
})
```

This is satisfied by the standard two-step relay protocol: the attacker first sends `RelayTransactionHashes` with target tx hashes, the node's `ask_for_txs()` timer (100 ms interval) sends `GetRelayTransactions` back, and the attacker then delivers the large transactions via `RelayTransactions`. This is normal relay protocol behavior — no special privileges required.

**Why existing guards are insufficient:**

- The rate limiter (30 messages/second per `(PeerIndex, message_type)`) is per-message, not per-byte. Each `RelayTransactions` message may carry multiple transactions.
- `TRANSACTION_SIZE_LIMIT = 512 KB` is enforced by `non_contextual_verify` before enqueue, but transactions with non-existent inputs pass this check — full script verification happens only inside the queue worker, not before admission.
- `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767` far exceeds the 512 transactions needed to fill the 256 MB queue (256 MB / 512 KB = 512).
- `remove_txs_by_peer` exists but is only called on disconnect; it is not used for admission control.

## Impact Explanation

When the verify queue is full, every call to `submit_remote_tx` (and `notify_tx`) returns `Reject::Full`. The tx-pool service propagates this rejection back to the relayer, which marks the transaction as unknown and stops requesting it. Honest peers' transactions are silently dropped at the queue boundary — never verified, never entering the pending pool, never mined. The node continues to operate but its ability to accept new transactions from the network is completely suspended for as long as the attacker keeps the queue saturated. This matches the **High** impact class: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs." An attacker targeting multiple nodes simultaneously can degrade transaction propagation across the network.

## Likelihood Explanation

The attack requires only a single connected peer with no special privileges. The attacker needs syntactically valid transactions (passing `non_contextual_verify`: correct structure, size ≤ 512 KB, not cellbase) but does not need valid scripts or real inputs — script verification happens inside the queue. The two-step protocol (announce hashes → receive request → deliver txs) is standard relay behavior. At 30 `RelayTransactions` messages/second, each carrying several transactions, the 512-transaction threshold is reachable in seconds. The attacker can reconnect after a disconnect to re-fill the queue, since `remove_txs_by_peer` only cleans up on disconnect and the queue drains only as the worker processes entries.

## Recommendation

Track per-peer byte usage inside `VerifyQueue`. Add a `HashMap<PeerIndex, usize>` field for peer byte totals. In `add_tx()`, enforce a per-peer cap (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / expected_max_peers`) so that no single peer can consume more than its fair share. When a peer's quota is exhausted, return `Reject::Full` only for that peer's submissions rather than blocking all peers. Update `remove_tx()` and `remove_txs_by_peer()` to decrement the per-peer counter accordingly.

## Proof of Concept

1. Connect to a CKB node as a relay peer (SupportProtocols::RelayV3).
2. Construct 512 syntactically valid transactions, each close to `TRANSACTION_SIZE_LIMIT` (512 KB), referencing non-existent inputs (so they fail script verification inside the queue but are admitted to it first).
3. Send `RelayTransactionHashes` messages announcing the hashes of these transactions. The node adds them to `unknown_tx_hashes` (limit `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767` >> 512).
4. Wait for the node's `ask_for_txs()` timer (fires every 100 ms) to send `GetRelayTransactions` back to the attacker.
5. Respond with `RelayTransactions` messages carrying the large transactions. Each passes `non_contextual_verify` and the `requesting_peer` filter, then enters `enqueue_verify_queue`.
6. Observe via `get_tip_tx_pool_info` that `verify_queue_size` climbs to its maximum.
7. Attempt to submit a legitimate transaction from any other peer or via RPC `send_transaction`; it is rejected with `Reject::Full("verify_queue total_tx_size exceeded …")`.
8. Repeat from step 3 after disconnect to maintain saturation.
