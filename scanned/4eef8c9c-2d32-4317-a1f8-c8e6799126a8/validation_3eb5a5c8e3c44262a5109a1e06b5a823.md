Audit Report

## Title
Intra-Message Duplicate Hash Bypass Inflates `UnknownTxHashPriority::peers` Vec Without Bound Check — (`sync/src/relayer/transaction_hashes_process.rs`, `sync/src/types/mod.rs`)

## Summary
The `execute()` handler for `RelayTransactionHashes` filters hashes against `tx_filter` in a read-only pass, so all K copies of a repeated hash survive into `tx_hashes`. Inside `add_ask_for_txs`, the first copy creates a queue entry and every subsequent copy unconditionally calls `push_peer`, inflating the `peers` Vec to K entries. The post-insertion overflow guard is keyed on `unknown_tx_hashes.len()` (unique hash count = 1), so it never fires and the attacker is never banned.

## Finding Description

**Root cause — read-only filter in `execute()`** [1](#0-0) 

The filter only reads `tx_filter`; it never inserts hashes into it. All K copies of hash H pass the `.filter()` predicate and land in `tx_hashes`. The only upstream guard is the count check: [2](#0-1) 

This allows up to `MAX_RELAY_TXS_NUM_PER_BATCH = 32,767` entries — all identical. [3](#0-2) 

**State mutation — `add_ask_for_txs()`** [4](#0-3) 

Copy 1 hits `Vacant` (creates entry, `peers = [P]`); copies 2–32,767 hit `Occupied` and call `push_peer` unconditionally: [5](#0-4) 

Result: `peers = [P, P, P, …]` with 32,767 identical entries for a single hash.

**Why the overflow guard is bypassed** [6](#0-5) 

With all-same-hash input, `unknown_tx_hashes.len()` = 1 after the loop. Both conditions are false, `peer_unknown_counter` is never computed, and `Status::ok()` is returned. The attacker is not banned.

**Downstream amplification — `pop_ask_for_txs` / `next_request_peer`** [7](#0-6) 

`next_request_peer` drains the `peers` Vec one entry per call via `swap_remove(0)`. With 32,767 identical entries, the queue entry stays live for 32,767 `pop_ask_for_txs` cycles, each emitting a `GetRelayTransactions` message to the attacker's peer for a hash the attacker will never fulfill.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker using distinct hashes across messages (32,767 copies of H1 in msg 1, H2 in msg 2, …) causes:
- **Memory**: Each message adds one queue entry with a 32,767-element `peers` Vec. At 30 msg/s (rate-limit ceiling), memory in `unknown_tx_hashes` grows at ~3.84 MB/s. The overflow guard fires only at 50,000 unique hashes (~28 minutes), by which point `peers` Vecs alone consume ~6.5 GB.
- **Outbound network amplification**: Each `pop_ask_for_txs` tick emits `GetRelayTransactions` for every ready entry, cycling through 32,767 peer entries per hash — a 32,767× amplification of outbound traffic per unique hash.
- **CPU**: `pop_ask_for_txs` iterates all 32,767 entries per hash per tick.

The combination of memory exhaustion and outbound request storm can crash or severely degrade a targeted CKB node, and the amplified outbound traffic contributes to network congestion.

## Likelihood Explanation

The attack requires only a standard P2P connection — no keys, no PoW, no privileged role. The message is protocol-valid (count ≤ 32,767). Constructing a `RelayTransactionHashes` flatbuffer with one hash repeated 32,767 times is trivially scriptable. The rate limiter (30 msg/s per peer/message-type) does not prevent the attack because the full damage is packed into one message. The attack is repeatable and deterministic.

## Recommendation

1. **Deduplicate in `execute()`**: Insert each hash into a local `HashSet` during the filter pass so duplicates are dropped before reaching `add_ask_for_txs`:
   ```rust
   let mut seen = std::collections::HashSet::new();
   .filter(|tx_hash| !tx_filter.contains(tx_hash) && seen.insert(tx_hash.clone()))
   ```
2. **Guard `push_peer`**: Check for duplicate `PeerIndex` before appending:
   ```rust
   pub fn push_peer(&mut self, peer_index: PeerIndex) {
       if !self.peers.contains(&peer_index) {
           self.peers.push(peer_index);
       }
   }
   ```
Both fixes together eliminate the inflation at both the entry point and the data structure level.

## Proof of Concept

Unit test (before fix — assertion fails; after fix — passes):
```rust
// Construct tx_hashes = vec![H; 1000] (1000 copies of the same hash)
let hash = Byte32::zero();
let tx_hashes = vec![hash.clone(); 1000];
state.add_ask_for_txs(peer_index, tx_hashes);
let unknown = state.unknown_tx_hashes.lock();
let priority = unknown.get(&hash).unwrap();
assert_eq!(priority.peers.len(), 1); // fails without fix: len == 1000
```

Manual steps: connect to a CKB node, send a single `RelayTransactionHashes` message with 32,767 copies of an unknown hash, observe `unknown_tx_hashes` entry with a 32,767-element `peers` Vec and subsequent `GetRelayTransactions` storm in outbound traffic.
