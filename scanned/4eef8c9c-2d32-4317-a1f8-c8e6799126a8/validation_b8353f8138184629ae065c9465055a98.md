Audit Report

## Title
`DaoScriptSizeVerifier` Lock-Script Size Check Bypassed via Input/Output Index Mismatch in DAO Phase-1 Transactions — (`File: verification/src/transaction_verifier.rs`)

## Summary

`DaoScriptSizeVerifier::verify()` pairs `resolved_inputs[i]` with `transaction.outputs()[i]` using positional `zip`. A transaction sender can place a DAO deposit input at index `j` and the corresponding DAO withdrawal output at index `k ≠ j`, causing the lock-script size check to be silently skipped at every iteration. Because the DAO type script on-chain explicitly cannot enforce this constraint (acknowledged in the verifier's own doc comment), the bypass is undetected, allowing the withdrawal cell to carry a smaller lock script and thus extract more interest than the deposit entitled.

## Finding Description

`DaoScriptSizeVerifier::verify()` iterates over positionally-zipped `(resolved_inputs[i], outputs[i])` pairs: [1](#0-0) 

The guard at line 855 requires **both** `input[i]` and `output[i]` to carry the DAO type script. If an attacker crafts a phase-1 transaction with:

| Index | Input | Output |
|-------|-------|--------|
| 0 | regular cell | DAO withdrawal cell (small lock) |
| 1 | DAO deposit cell (data = `0x00…`) | regular change cell |

then at `i=0` the input is non-DAO → `continue`; at `i=1` the output is non-DAO → `continue`. The size check at line 885 never executes: [2](#0-1) 

The verifier's own doc comment confirms the DAO type script cannot enforce this constraint independently: [3](#0-2) 

In phase 2, `transaction_maximum_withdraw` iterates over `resolved_inputs` and calls `calculate_maximum_withdraw` with the **withdrawal cell's** `cell_output` as `output`: [4](#0-3) 

`calculate_maximum_withdraw` computes interest based on the withdrawal cell's `occupied_capacity`: [5](#0-4) 

A smaller lock script in the withdrawal cell → smaller `occupied_capacity` → larger `counted_capacity` → larger `withdraw_counted_capacity` → more CKB extracted than the deposit entitled.

The existing unit test `test_dao_disables_different_lock_script_size` only covers the case where the DAO deposit input and DAO withdrawal output share the same index (index 1), confirming the check works in the aligned case but leaving the misaligned case untested: [6](#0-5) 

`DaoScriptSizeVerifier` is applied in both the tx-pool admission path and the contextual block verifier, so the bypass affects both layers: [7](#0-6) [8](#0-7) 

## Impact Explanation

An unprivileged attacker can mint excess CKB by exploiting the interest calculation: the withdrawal cell's smaller lock script reduces `occupied_capacity`, inflating `counted_capacity` and thus the interest payout. The excess per transaction is bounded by the lock-script size difference in shannons (e.g., a 10,000-byte difference yields 10,000 shannons of extra counted capacity, multiplied by the AR ratio). While the per-transaction gain is small, the attack is permissionless, repeatable, and results in CKB being minted beyond what was deposited — a concrete violation of the DAO's economic invariant. This matches **"Vulnerabilities which could easily damage CKB economy" (Critical, 15001–25000 points)**, as the protocol's capacity-conservation guarantee is broken without any privileged access.

## Likelihood Explanation

No privileged access is required. Any wallet or script author can construct the transaction by simply reordering inputs and outputs. The CKB protocol imposes no ordering constraint on inputs/outputs relative to DAO cells. The bypass requires only index misalignment, which is trivially achievable. The attack is repeatable across any number of DAO deposit/withdrawal cycles.

## Recommendation

Replace the positional `zip` in `DaoScriptSizeVerifier::verify()` with an explicit pairing that matches each DAO deposit input to its corresponding DAO withdrawal output regardless of index. Concretely:

1. Collect all DAO deposit inputs (those with DAO type script and all-zero data) into a set.
2. Collect all DAO withdrawal outputs (those with DAO type script) into a set.
3. Enforce a bijection between them — for example, by requiring the transaction to declare the pairing explicitly via witness data, or by enforcing that DAO deposit inputs and their corresponding withdrawal outputs must occupy the same index and rejecting any transaction that violates this ordering.

## Proof of Concept

Construct a phase-1 DAO transaction:

```
inputs:
  [0] regular cell (lock: secp256k1, capacity: 100 CKB)
  [1] DAO deposit cell (lock: secp256k1-large-args, data: 0x0000000000000000, capacity: 1000 CKB)

outputs:
  [0] DAO withdrawal cell (lock: secp256k1-small-args, capacity: 1000 CKB)
  [1] regular change cell (capacity: ~100 CKB minus fee)
```

`DaoScriptSizeVerifier::verify()` iterates:
- `i=0`: `input[0]` is non-DAO → `continue`
- `i=1`: `output[1]` is non-DAO → `continue`

The lock-script size mismatch between `input[1]` (large) and `output[0]` (small) is never checked. In phase 2, `calculate_maximum_withdraw` is called with `output[0]` (the withdrawal cell). Its smaller lock script reduces `occupied_capacity`, inflating `counted_capacity` and thus the interest payout beyond what the original deposit entitled.

A minimal unit test can be written mirroring `test_dao_disables_different_lock_script_size` but swapping the DAO withdrawal output to index 0 and the regular cell to index 1 — the verifier will return `Ok(())` instead of `Err(DaoLockSizeMismatch { index: 0 })`, confirming the bypass.
