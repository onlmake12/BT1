### Can this be triggered by an unprivileged attacker?

The `has_proposed` check at line 178 fails when a transaction T is in `Status::Proposed` but one of its ancestors A is in `Status::Gap`. [5](#0-4) 

This is a reachable pool state: A is proposed in block N, not committed in block N+1 (moves to `Gap`), while child C is proposed in block N+1. When block N+2 is assembled, C is in `Proposed` but A is in `Gap` — C passes `sorted_proposed_iter()` but fails the ancestor check. An unprivileged attacker can manufacture many such (A, C) pairs by submitting transaction chains to the P2P network; miners automatically include proposals from the tx-pool.

With `MAX_CONSECUTIVE_FAILURES = 4000`, the attacker needs 4001 such failing transactions interspersed with valid ones to cause premature loop exit. [6](#0-5)

### Title
`consecutive_failed` never resets on successful selection, causing premature block-template truncation — (`tx-pool/src/component/tx_selector.rs`)

### Summary
`txs_to_commit` increments `consecutive_failed` on two failure paths but never resets it to 0 when a transaction is successfully packaged. The counter therefore measures **total** failures, not **consecutive** failures, violating the stated invariant and the semantic implied by both the variable name and the constant name `MAX_CONSECUTIVE_FAILURES`.

### Finding Description
In `TxSelector::txs_to_commit`, `consecutive_failed` is incremented at lines 153 and 184 but has no reset statement anywhere in the success path (lines 191–220). The intended semantics — "give up only after 4000 failures in a row with no successes" — are not implemented. The actual semantics are "give up after 4000 total failures regardless of intervening successes." [7](#0-6) 

An attacker exploits this by populating the proposed pool with transactions whose ancestors are in `Status::Gap`. Each such transaction passes `sorted_proposed_iter()` (it is proposed) but fails the `has_proposed` ancestor check at line 178, incrementing `consecutive_failed`. Valid transactions interspersed in the sorted order do not reset the counter. Once the counter exceeds 4000, the loop breaks even though valid, fee-paying transactions remain.

### Impact Explanation
Block templates are truncated prematurely. Miners produce blocks with fewer transactions than the block limits allow, reducing fee revenue. High-fee transactions that would have been selected are left out, degrading throughput and potentially causing mempool congestion.

### Likelihood Explanation
An unprivileged attacker submits transaction chains via the standard P2P interface. Miners automatically include proposals from the tx-pool. The attacker times submissions so that parent transactions move to `Gap` before children are committed. Producing 4001 such pairs requires submitting ~8002 transactions — a non-trivial but realistic cost on mainnet, and trivially achievable on a local or test node.

### Recommendation
Add `consecutive_failed = 0;` at the start of the success path, immediately before or after `self.update_modified_entries(&ancestors)` at line 220:

```rust
self.update_modified_entries(&ancestors);
consecutive_failed = 0;  // reset: a successful selection breaks any failure streak
```

### Proof of Concept
1. Fill the proposed pool with 4001 transactions each having one ancestor in `Status::Gap`.
2. Intersperse 1 valid proposed transaction (all ancestors proposed) at position 4002 in sorted order.
3. Call `package_txs` / `txs_to_commit`.
4. Assert: the valid transaction at position 4002 is absent from the returned `entries` vector, even though it fits within size and cycle limits.
5. Apply the one-line fix; assert the valid transaction is now included.
