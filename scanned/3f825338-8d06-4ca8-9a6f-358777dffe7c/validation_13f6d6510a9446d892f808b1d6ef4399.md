### Title
Orphan Pool Random Eviction Allows Any P2P Peer to Continuously Displace Legitimate Orphan Transactions - (File: `tx-pool/src/component/orphan.rs`)

### Summary
The `OrphanPool` enforces a global cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries with no per-peer sub-limit. When the pool is full and a new orphan arrives, the pool inserts it first (reaching 101) and then evicts **one entry chosen by arbitrary HashMap iteration order** — effectively random. Any unprivileged P2P peer can pre-fill all 100 slots with crafted orphan transactions, giving legitimate orphan transactions a ≈99% probability of being immediately evicted on arrival. The attacker can continuously re-announce evicted entries to maintain the pool at capacity, creating a sustained DOS against orphan transaction propagation.

### Finding Description

`OrphanPool::add_orphan_tx` inserts the new transaction unconditionally and then calls `limit_size`:

```rust
// tx-pool/src/component/orphan.rs
pub(crate) const DEFAULT_MAX_ORPHAN_TRANSACTIONS: usize = 100;

pub fn add_orphan_tx(&mut self, tx: TransactionView, peer: PeerIndex, declared_cycle: Cycle) -> Vec<Byte32> {
    if self.entries.contains_key(&tx.proposal_short_id()) {
        return vec![];
    }
    self.entries.insert(tx.proposal_short_id(), Entry::new(tx.clone(), peer, declared_cycle));
    // ...
    self.limit_size()   // evicts after insertion
}

There is **no per-peer quota** anywhere in `OrphanPool`. A single peer can occupy all 100 slots.

A transaction is routed into the orphan pool inside `after_process` whenever verification returns `is_missing_input`:

```rust
if is_missing_input(reject) {
    self.send_result_to_relayer(TxVerificationResult::UnknownParents { peer, parents: tx.unique_parents() });
    self.add_orphan(tx, peer, declared_cycle).await;
}
``` [3](#0-2) 

Each evicted entry is forwarded to the relayer as `TxVerificationResult::Reject`, which removes it from the "known" filter so the node is willing to re-request it — but the attacker can immediately re-announce the same hashes to re-fill the slot. [4](#0-3) 

Orphan entries carry an expiry of `100 × MAX_BLOCK_INTERVAL` seconds, so attacker entries remain valid for many minutes before natural expiry can clear them. [5](#0-4) 

### Impact Explanation

With 100 attacker-controlled orphan entries already in the pool, any legitimate orphan transaction that arrives is inserted (pool = 101) and then one of the 101 entries is evicted at random. The probability that the legitimate transaction is the one evicted is 100/101 ≈ 99%. The attacker re-submits any of their own entries that happen to be evicted, maintaining the pool at capacity. The result is that legitimate orphan transactions are continuously displaced, preventing them from being resolved when their parent transactions arrive. This delays or permanently prevents propagation of dependent transaction chains through the node.

**Impact: Medium** — orphan transaction propagation is disrupted; the node cannot relay dependent transaction chains correctly.

### Likelihood Explanation

The attack requires only a standard P2P connection. The attacker announces 100 transaction hashes, waits for the victim to request them via the relay protocol, then delivers transactions whose inputs reference non-existent cells. No special privilege, key material, or majority hashpower is needed. The 100-entry limit is small enough that a single peer can saturate it in one round-trip. The attack is cheap to sustain because the attacker's orphan transactions cost nothing beyond network bandwidth.

**Likelihood: Medium** — requires an active P2P peer but no other privilege; the small pool size makes saturation trivial.

### Recommendation

1. **Add a per-peer cap** inside `OrphanPool` (e.g., no more than `DEFAULT_MAX_ORPHAN_TRANSACTIONS / connected_peers` entries per peer index). This prevents a single peer from monopolising the pool.
2. **Replace random eviction with fee-rate or declared-cycle priority eviction** so that low-value attacker entries are evicted before legitimate high-value ones.
3. **Increase `DEFAULT_MAX_ORPHAN_TRANSACTIONS`** or make it proportional to the number of connected peers.

### Proof of Concept

1. Attacker connects to a CKB node as a P2P peer.
2. Attacker announces 100 distinct transaction hashes via `RelayTransactionHashes`.
3. Victim node requests the transactions via `GetRelayTransactions`.
4. Attacker responds with 100 transactions whose inputs reference cells that do not exist on-chain or in the tx-pool. Each fails `is_missing_input` verification and is inserted into `OrphanPool` via `add_orphan_tx`.
5. Pool is now at capacity (100 entries, all attacker-controlled).
6. A legitimate user submits a transaction whose parent is not yet confirmed. It is relayed to the victim node, passes non-contextual checks, fails `is_missing_input`, and `add_orphan_tx` is called. The pool reaches 101; `limit_size` evicts one entry at random. With probability 100/101 the legitimate entry is evicted and sent as `TxVerificationResult::Reject` to the relayer.
7. Attacker monitors the relayer's `TxVerificationResult::Reject` stream (or simply re-announces all 100 hashes periodically) to re-fill any slot that was evicted, maintaining the pool at 100 attacker entries indefinitely.
8. The legitimate orphan transaction is never resolved; its dependent chain cannot propagate through this node.

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
