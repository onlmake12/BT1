### Title
`ScriptHashTypeVerifier::verify()` Only Checks Output Lock Scripts, Not Type Scripts, Allowing Cells with Non-Permitted Type Script Hash Types to Be Created — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that output scripts use only `ScriptHashType` values within `ENABLED_SCRIPT_HASH_TYPE`, but it only iterates over each output's **lock script** hash type. The **type script** hash type is never checked. This is a direct structural analog to the external report: one side of the output cell (lock) is validated while the other side (type) is silently skipped. Any unprivileged transaction sender can create on-chain cells whose type scripts carry a non-permitted or invalid hash type, making those cells permanently unspendable and destroying the CKB capacity locked in them.

---

### Finding Description

`ScriptHashTypeVerifier` is a non-contextual verifier that runs as part of `NonContextualTransactionVerifier` before any script execution. Its stated purpose is to prevent outputs from using `ScriptHashType` values that are not yet activated by the current consensus rules (e.g., a future hardfork hash type).

The implementation iterates only over `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  lines 796–814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

`output.type_()` — the type script — is never inspected. A cell output has two scripts: a lock script and an optional type script. Both are executed when the cell is consumed. The verifier checks one and ignores the other. [1](#0-0) 

The `NonContextualTransactionVerifier` that calls this verifier is the gating check applied to every transaction before it enters the tx-pool or is included in a block: [2](#0-1) 

`ENABLED_SCRIPT_HASH_TYPE` is the consensus-controlled set of permitted hash type byte values: [3](#0-2) 

The `CellbaseVerifier` separately enforces that cellbase outputs carry **no** type script at all, but that check is unrelated to hash-type validation for regular transactions: [4](#0-3) 

---

### Impact Explanation

A cell output has two scripts: lock and type. Both are executed during consumption. By only validating the lock script's hash type, the verifier allows a transaction to create a cell whose type script carries:

1. **A completely undefined byte value** (e.g., `0x03`, `0x05`) — when the cell is later consumed, `TryInto::<ScriptHashType>::try_into()` fails, script resolution aborts, and the consuming transaction is rejected. The cell is **permanently unspendable**; the CKB capacity locked in it is destroyed.

2. **A future hardfork hash type** (e.g., `Data2 = 0x04` before the hardfork activating it) — the cell is created before the hardfork, bypassing the activation gate for type scripts. After the hardfork the cell becomes spendable, but the pre-hardfork window allows state to exist that the consensus rules were explicitly designed to prevent.

In both cases the cell is accepted onto the canonical chain by all nodes (since all nodes run the same incomplete verifier), so there is no consensus split — the damage is one-sided: the capacity in the cell is lost or the hardfork gating is bypassed.

---

### Likelihood Explanation

The entry path requires no privilege. Any RPC caller or P2P transaction relayer can submit a transaction via `send_transaction` (RPC) or the P2P relay protocol. The transaction passes `NonContextualTransactionVerifier` (which calls `ScriptHashTypeVerifier`) and enters the tx-pool. No miner collusion, no key leak, and no majority hashpower is required. The attacker only needs to spend the minimum CKB capacity to create the output cell.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the hash type of each output's type script, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }

        // NEW: type script check (symmetric with lock script check)
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }).into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Construct a `TransactionView` with one output cell:
   - Lock script: valid hash type (e.g., `Data = 0x00`).
   - Type script: non-permitted hash type byte (e.g., `0x05`, which is not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC or P2P relay.
3. `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`, which only checks the lock script hash type (`0x00` — valid) and returns `Ok(())` without inspecting the type script.
4. The transaction is accepted into the tx-pool and eventually included in a block.
5. Any subsequent transaction attempting to consume this cell will fail during contextual script verification because the type script hash type `0x05` cannot be resolved to a known `ScriptHashType` variant.
6. The CKB capacity in the cell is permanently inaccessible. [5](#0-4)
