Audit Report

## Title
Unconditional Out-of-Bounds Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` guards the `block_unconfirmed_txs` access with `if tx_age >= self.block_unconfirmed_txs.len()` but applies no equivalent guard to the `confirm_blocks_to_failed_txs[tx_age - 1]` write. When a tracked transaction is rejected from the pool after 1001+ blocks, `tx_age - 1 >= 1000` indexes past the end of the 1000-element `Vec`, causing an unconditional Rust index-out-of-bounds panic and crashing the node process. An unprivileged remote peer can trigger this by submitting a valid transaction and waiting approximately 2.2–2.8 hours.

## Finding Description

All three parallel `Vec`s in `TxConfirmStat` are initialized with `max_confirm_blocks = MAX_CONFIRM_BLOCKS = 1000` elements: [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 correctly handles the `block_unconfirmed_txs` access for `tx_age >= 1000`: [2](#0-1) 

However, the `count_failure` branch at line 214–216 is **outside** that guard and has no bounds check of its own: [3](#0-2) 

When `tx_age = 1001`, `tx_age - 1 = 1000`, which is past the last valid index (999) of `confirm_blocks_to_failed_txs`. Rust panics unconditionally with an index-out-of-bounds error.

The call chain that reaches this with `count_failure = true`:

- `reject_tx` calls `drop_tx`: [4](#0-3) 

- `drop_tx` calls `drop_tx_inner(tx_hash, true)`: [5](#0-4) 

- `drop_tx_inner` calls `remove_unconfirmed_tx` with `self.best_height` as `tip_height` and `count_failure = true`: [6](#0-5) 

A transaction is tracked at `self.current_tip` at the time of `accept_tx`. If `best_height` later advances to `entry_height + 1001` or more before the tx is rejected, the panic is triggered.

## Impact Explanation

This is a **High** severity vulnerability matching the allowed impact: **"Vulnerabilities which could easily crash a CKB node."** The panic propagates up through the fee-estimator service and crashes the node process. It is remotely triggerable by any unprivileged peer with no special privileges, no PoW, and no majority hashpower required.

## Likelihood Explanation

CKB produces blocks roughly every 8–10 seconds; 1001 blocks ≈ 2.2–2.8 hours. The tx-pool expiry is time-based (default ~12 hours), so a transaction easily survives 1001 blocks within the expiry window. When expiry fires, `remove_expired` in `tx-pool/src/pool.rs` calls `callbacks.call_reject`, which chains to `reject_tx` → `drop_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx` → panic. Alternatively, an attacker can submit a conflicting (double-spend) transaction after 1001 blocks to trigger the same rejection path immediately. The attack is repeatable and requires only the ability to submit a valid transaction to the mempool.

## Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` write, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This silently drops the failure sample for transactions older than `MAX_CONFIRM_BLOCKS`, consistent with how `block_unconfirmed_txs` already handles the same case.

## Proof of Concept

```rust
// Reproduces with max_confirm_blocks = 10
let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
let mut stat = TxConfirmStat::new(buckets, 10, 0.993);

// Track a tx at height 0
let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000)).unwrap();

// 11 blocks later, reject it with count_failure=true
// tx_age = 11, confirm_blocks_to_failed_txs[10] → index 10 on a 10-element Vec → PANIC
stat.remove_unconfirmed_tx(0, 11, bucket_index, true);
```

In production: submit a valid tx at height H, wait for `best_height` to reach H+1001 (≈2.2 hours at 8s/block), then either wait for expiry or submit a conflicting tx. The original tx is rejected via `reject_tx` → `drop_tx_inner(count_failure=true)` → `remove_unconfirmed_tx` → `confirm_blocks_to_failed_txs[1000]` → **index out of bounds → node process crash**.     fn remove_unconfirmed_tx(
```rust
        &mut self,
        entry_height: u64,
        tip_height: u64,
        bucket_index: usize,
        count_failure: bool,
    ) {
        let tx_age = tip_height.saturating_sub(entry_height) as usize;
        if tx_age < 1 {
            return;
        }
        if tx_age >= self.block_unconfirmed_txs.len() {
            self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;
        } else {
            let block_index = (entry_height % self.block_unconfirmed_txs.len() as u64) as usize;
            self.block_unconfirmed_txs[block_index][bucket_index] -= 1;
        }
        if count_failure {
            self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
        }
    }
```
