Audit Report

## Title
Missing Peer Ban for Unsolicited Block Filter Response Messages Enables Unbounded Handler Flooding - (File: `sync/src/filter/mod.rs`)

## Summary
The `BlockFilter` protocol handler returns `Status::ignored()` (code `101`) for unsolicited `BlockFilters`, `BlockFilterHashes`, and `BlockFilterCheckPoints` messages. Because `should_ban()` only triggers for status codes in the `400–499` range, this status never results in a ban or disconnect. The handler carries no rate limiter, so an unprivileged peer can flood the Filter protocol handler indefinitely at zero cost beyond bandwidth, degrading service for legitimate light-client consumers of the Filter protocol.

## Finding Description
In `sync/src/filter/mod.rs`, `try_process` matches all six `BlockFilterMessageUnion` variants. The three response-type variants fall into a catch-all arm that logs a warning and returns `Status::ignored()`:

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    // remote peer should not send block filter to us without asking
    // TODO: ban remote peer
    warn_target!(...);
    Status::ignored()
}
``` [1](#0-0) 

`Status::ignored()` maps to `StatusCode::Ignored = 101`. [2](#0-1) 

`should_ban()` only returns `Some(ban_time)` for codes in `400..500`; code `101` falls outside that range and always returns `None`: [3](#0-2) 

The outer `process()` function calls `nc.ban_peer()` only when `status.should_ban()` returns `Some`, which never happens here: [4](#0-3) 

The `BlockFilter` struct has no rate limiter field: [5](#0-4) 

By contrast, `Relayer` has a `RateLimiter<(PeerIndex, u32)>` capping 30 req/sec per peer per message type: [6](#0-5) 

The 1.8 MB outbound size cap in `get_block_filters_process.rs` applies only to messages the node *sends*, not to inbound messages from an attacker: [7](#0-6) 

The `received()` entry point bans for parse failures but passes well-formed messages directly to `process()` with no further gating: [8](#0-7) 

## Impact Explanation
An attacker with only a TCP connection can send a continuous stream of well-formed, maximum-size `BlockFilters` messages to any CKB full node with block filter building enabled. Each message passes the parse check, triggers a `warn_target!` log write and a metrics update, and returns without any ban or disconnect. The Filter protocol handler's async task is continuously occupied processing attacker messages, delaying or starving legitimate `GetBlockFilters` responses to real light clients. Targeting multiple nodes simultaneously degrades the Filter protocol service network-wide at negligible attacker cost. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points).

## Likelihood Explanation
The `SupportProtocols::Filter` sub-stream is publicly reachable on any CKB full node with block filter building enabled. No authentication, stake, or privilege is required. The attacker needs only a TCP connection and the ability to encode a valid molecule `BlockFilterMessage`. The `// TODO: ban remote peer` comment confirms the developers are aware of the gap. The attack is repeatable indefinitely and requires no victim mistakes.

## Recommendation
Replace `Status::ignored()` in the unsolicited-response arm with a status code in the `4xx` ban range, consistent with how all other CKB protocol handlers treat protocol violations:

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    StatusCode::ProtocolMessageIsMalformed
        .with_context("peer sent unsolicited block filter response")
}
```

Additionally, add a per-peer rate limiter to `BlockFilter` mirroring the `Relayer`'s `RateLimiter<(PeerIndex, u32)>` so that even messages that arrive before a ban is issued are rate-capped.

## Proof of Concept
1. Connect to a CKB full node on the Filter protocol sub-stream (`SupportProtocols::Filter`).
2. Construct a valid molecule-encoded `BlockFilterMessage` wrapping a `BlockFilters` table with arbitrary `start_number`, `block_hashes`, and `filters` vectors filled to the Tentacle frame size limit.
3. Send this message in a tight loop from a single TCP connection.
4. Observe: the node logs a `warn_target!` warning for every message, updates metrics, and never disconnects or bans the sender. The Filter handler's async task is continuously occupied, delaying legitimate `GetBlockFilters` responses.
5. Scale to multiple connections or multiple target nodes to amplify the effect across the network's Filter protocol service.
