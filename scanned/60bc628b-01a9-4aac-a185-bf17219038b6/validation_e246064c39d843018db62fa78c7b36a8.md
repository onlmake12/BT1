### Title
`n_sync_started` Counter Permanently Inflated via TOCTOU Race Between Atomic Increment and `peer_state.start_sync()` — (`sync/src/types/mod.rs`, `sync/src/synchronizer/mod.rs`)

---

### Summary

The `n_sync_started` atomic counter in `Peers` tracks how many peers are actively performing header sync. In `start_sync_headers`, the counter is incremented atomically **before** `peer_state.start_sync()` is called on the peer's state entry. If the peer disconnects between these two operations, the `disconnected()` handler observes `sync_started() == false` and does not decrement the counter. The counter is permanently inflated with no rollback path, directly analogous to the original report's stale `signerCount` that was never corrected when the actual valid count diverged from the cached value.

---

### Finding Description

In `sync/src/synchronizer/mod.rs`, `start_sync_headers` performs a two-step operation that is not atomic with respect to peer disconnection:

**Step 1 — atomic increment of `n_sync_started`:** [1](#0-0) 

**Step 2 — conditional call to `peer_state.start_sync()` (sets `HeadersSyncState::Started`):** [2](#0-1) 

The `get_mut` on `peers().state` returns `None` if the peer was already removed by a concurrent disconnect. In that case `start_sync` is never called, so `peer_state.sync_started()` remains `false`.

The `disconnected()` handler only decrements `n_sync_started` when `peer_state.sync_started()` is `true`: [3](#0-2) 

Because `sync_started()` checks for `HeadersSyncState::Started` exactly: [4](#0-3) 

If the peer disconnects between Step 1 and Step 2, `disconnected()` has already run (removing the peer from `state`), `sync_started()` returns `false`, and `n_sync_started` is never decremented. There is no rollback of the increment in `start_sync_headers`.

The `n_sync_started` counter is defined as: [5](#0-4) 

The IBD guard in `start_sync_headers` uses this counter to enforce single-peer sync during Initial Block Download: [6](#0-5) 

Once `n_sync_started` is inflated to 1 (or higher) with no actual syncing peer, the `if ibd && x != 0 { None }` branch permanently prevents any new sync from starting in IBD mode.

The developers themselves acknowledge the fragility of this counter in a test comment: [7](#0-6) 

---

### Impact Explanation

In IBD (Initial Block Download) mode, `n_sync_started > 0` causes `fetch_update` to return `Err`, breaking out of the peer loop and preventing any header sync from starting. A node that enters this state cannot advance its chain tip and is permanently stuck in IBD until restarted. This is a denial-of-sync condition: the node cannot download blocks, cannot validate the chain, and cannot participate in the network as a full node.

In non-IBD mode the impact is lower — the counter is used to limit concurrency but the strict `x != 0` guard only applies during IBD — however the counter still drifts from reality, causing incorrect scheduling decisions.

---

### Likelihood Explanation

The race window is small but real: `n_sync_started` is incremented atomically, then a separate lock acquisition (`peers().state.get_mut(&peer)`) is performed. Peer disconnection events are processed by the network layer concurrently with the sync loop. A peer that connects, passes the sync handshake (appearing in `peers().state`), and then disconnects at the moment between the `fetch_update` and the `get_mut` triggers the bug. This can happen naturally under network churn or be deliberately triggered by a remote peer that connects and immediately drops the connection. No special privileges are required — any inbound or outbound peer can cause this.

---

### Recommendation

The increment of `n_sync_started` and the call to `peer_state.start_sync()` must be made atomic with respect to peer removal. One approach: acquire the peer state lock first, verify the peer still exists, call `start_sync()`, and only then increment `n_sync_started`. If `get_mut` returns `None`, roll back the increment before proceeding:

```rust
// Increment first, then verify peer still exists
if self.shared().state().n_sync_started()
    .fetch_update(..., |x| if ibd && x != 0 { None } else { Some(x + 1) })
    .is_err()
{
    break;
}
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
} else {
    // Peer already gone — roll back the increment
    self.shared().state().n_sync_started().fetch_sub(1, Ordering::AcqRel);
}
```

Alternatively, restructure so the peer state lock is held across both operations, or use a dynamic count derived from iterating `peers().state` (analogous to the upstream recommendation to use a dynamic `getSignerCount` instead of a cached state variable).

---

### Proof of Concept

1. Node A enters IBD mode (`is_initial_block_download() == true`).
2. Remote peer P connects and completes the sync protocol handshake; P appears in `peers().state` with `can_start_sync() == true`.
3. `start_sync_headers` runs:
   - `n_sync_started.fetch_update(...)` succeeds → `n_sync_started = 1`.
   - Concurrently, P drops the TCP connection; `disconnected(P)` runs, removes P from `state`, observes `sync_started() == false`, does **not** decrement `n_sync_started`.
   - `peers().state.get_mut(&P)` returns `None`; `start_sync` is never called.
4. `n_sync_started` is now `1` with zero actual syncing peers.
5. Every subsequent call to `start_sync_headers` hits `if ibd && x != 0 { None }` → `fetch_update` returns `Err` → `break` → no peer ever starts syncing.
6. Node A is permanently stuck in IBD until restarted.
