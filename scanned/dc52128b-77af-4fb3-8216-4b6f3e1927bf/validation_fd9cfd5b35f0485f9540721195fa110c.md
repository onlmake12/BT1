Audit Report

## Title
`check_tx_fee` Minimum Fee Rate Admission Uses Only Serialized Size, Allowing Cycle-Heavy Transactions to Bypass the Threshold — (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the true transaction weight — used everywhere else in the pool — is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns the actual cycle count, no second fee-rate gate is applied. A cycle-heavy, byte-small transaction can therefore enter the pool while paying an actual fee rate up to ~119× below the configured `min_fee_rate`.

## Finding Description

**Root cause — size-only admission gate:**

`check_tx_fee` in `tx-pool/src/util.rs` (lines 42–45) explicitly uses `tx_size` as the weight denominator, with a comment acknowledging the limitation:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The true weight formula is defined in `util/types/src/core/tx_pool.rs`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

**No post-verification re-check:**

In `_process_tx` (`tx-pool/src/process.rs`), the flow is:
1. `pre_check` → calls `check_tx_fee` with only `tx_size` (line 289, 294).
2. `verify_rtx` → returns `Completed` with actual `verified.cycles` (line 734).
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is created (line 751) — **no fee-rate re-check with actual weight occurs**. [3](#0-2) 

`TxEntry::fee_rate()` does correctly compute the weight-aware fee rate using `get_transaction_weight(self.size, self.cycles)`, but this is only used for eviction ordering and block assembly — never as an admission gate. [4](#0-3) 

**Concrete arithmetic:**

With `max_tx_verify_cycles = 70,000,000` and `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`: [5](#0-4) 

- Max weight from cycles: `70,000,000 × 0.000_170_571_4 ≈ 11,940`
- A 100-byte transaction's actual weight is ~119× its serialized size
- With `min_fee_rate = 1,000 shannons/KW`: size-based gate requires 100 shannons; weight-based requirement is 11,940 shannons
- Actual fee rate of such a transaction: `100 × 1,000 / 11,940 ≈ 8 shannons/KW` — 125× below threshold [6](#0-5) 

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker submitting cycle-heavy, byte-small transactions:
1. Passes `check_tx_fee` by paying only the size-based minimum (e.g., 100–200 shannons).
2. Forces every receiving node to execute up to 70M cycles of script verification per transaction — significant CPU load.
3. Because `check_tx_fee` is also the admission gate on relay peers, these transactions propagate across the network, multiplying the CPU cost.
4. Fills the pool with entries whose actual fee rate is far below `min_fee_rate`, degrading pool quality for miners.
5. The 180 MB pool size cap bounds memory damage but does not prevent sustained CPU exhaustion or pool quality degradation.

## Likelihood Explanation

The attack requires only a valid RPC connection or relay peer path — no privileged access. The attacker needs valid UTXOs and a deployed script cell that consumes near-maximum cycles. The fee cost is ~125× lower than the configured threshold, making sustained attack economically viable. The `max_tx_verify_cycles` limit (70M) is the only natural bound on the per-transaction discrepancy.

## Recommendation

After `verify_rtx` returns the actual cycle count in `_process_tx`, add a weight-aware fee rate check before creating the `TxEntry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

This check should be inserted at `tx-pool/src/process.rs` between lines 734 and 751, after `verified` is obtained and before `TxEntry::new` is called. [7](#0-6) 

## Proof of Concept

1. Deploy a type script cell that loops to consume ~70M cycles.
2. Craft a transaction: serialized size ≈ 200 bytes, one input, one output, the type script as cell dep. Fee = `1,000 × 200 / 1,000 = 200 shannons`.
3. Submit via `send_transaction` RPC (no `declared_cycles` required).
4. `check_tx_fee` computes `min_fee = fee(200) = 200 shannons`. Fee ≥ min_fee → **accepted**.
5. `verify_rtx` executes ~70M cycles. Actual weight ≈ 11,940. Actual fee rate ≈ `200 × 1,000 / 11,940 ≈ 16 shannons/KW` — 62× below the 1,000 shannons/KW threshold.
6. Transaction enters the pool. Repeat with fresh UTXOs to fill the pool with sub-threshold entries and impose sustained CPU load on all relay nodes. [8](#0-7) [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
