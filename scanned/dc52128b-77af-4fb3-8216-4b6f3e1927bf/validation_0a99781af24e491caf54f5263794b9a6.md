Audit Report

## Title
Fee-Rate Unit Mismatch in `calculate_min_replace_fee` and `check_tx_fee` Allows RBF Policy Bypass and Pool Admission Bypass — (`tx-pool/src/pool.rs`, `tx-pool/src/util.rs`)

## Summary

`FeeRate::fee()` expects a `weight` argument defined as `max(tx_size_bytes, cycles × DEFAULT_BYTES_PER_CYCLES)`, but both `calculate_min_replace_fee` and `check_tx_fee` pass raw `tx_size` (bytes) instead. For a high-cycles, small-size transaction, this underestimates the required fee by up to ~60×. An unprivileged `send_transaction` caller can exploit this to replace pool transactions while paying far less than the `min_rbf_rate` policy requires, and to admit transactions whose effective fee rate is far below `min_fee_rate`.

## Finding Description

**`FeeRate` unit definition** — `FeeRate` is shannons per kilo-weight, and `FeeRate::fee(weight)` computes `rate × weight / 1000`: [1](#0-0) [2](#0-1) 

**`get_transaction_weight`** — the correct weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [3](#0-2) 

**Root cause 1 — `calculate_min_replace_fee`:** The function receives only `size: usize` and passes it directly to `min_rbf_rate.fee()`, ignoring cycles entirely: [4](#0-3) 

At the call site in `submit_entry` → `check_rbf`, the `entry` is a fully-constructed `TxEntry` with both `entry.size` and `entry.cycles` available, but only `entry.size` is forwarded: [5](#0-4) 

**Root cause 2 — `check_tx_fee`:** The code explicitly acknowledges the mismatch in a comment but calls it a "cheap check" and uses `tx_size` as the weight argument: [6](#0-5) 

This is called in `pre_check` **before** script execution, so cycles are not yet known. After `verify_rtx` returns the actual cycles, a `TxEntry` is constructed and `submit_entry` is called — but there is no second fee-rate check using the correct weight: [7](#0-6) 

The `TxEntry::fee_rate()` method correctly uses `get_transaction_weight`, but it is only used for sorting/eviction, not for admission or RBF gating: [8](#0-7) 

**Quantified discrepancy:** At `max_tx_verify_cycles = 70,000,000` and `DEFAULT_BYTES_PER_CYCLES ≈ 0.000170571`, the cycle-derived weight is `≈ 11,940`. A minimal transaction of ~200 bytes yields a discrepancy factor of `11,940 / 200 ≈ 60×`. The shipped default config enables RBF (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`): [9](#0-8) 

## Impact Explanation

**RBF bypass (primary):** An attacker crafts a replacing transaction with small serialized size but a lock script consuming near-maximum cycles. `calculate_min_replace_fee` computes the required fee bump using size-only weight, so the attacker pays ~1/60th of the intended extra fee. The replacing transaction is accepted into the pool with an effective fee rate far below `min_rbf_rate`, displacing the victim transaction. The replacing transaction will not be prioritized by miners, indefinitely delaying confirmation of the original transaction. This constitutes a low-cost griefing attack on specific transactions.

**Pool admission bypass (secondary):** Any `send_transaction` caller can submit high-cycles, small-size transactions with a fee just above `min_fee_rate.fee(size)`. Each such transaction is admitted to the pool with an effective fee rate potentially 60× below `min_fee_rate`, wastes a pool slot, and consumes significant CPU during script verification (up to 70M cycles per tx). Repeated submissions can cause CKB network congestion with minimal cost.

This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and additionally enables targeted transaction griefing via the RBF path.

## Likelihood Explanation

- RBF is enabled by default in the shipped `ckb.toml` (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`).
- The attack requires only an unprivileged `send_transaction` RPC call — no special role, key, or majority hashpower.
- Crafting a transaction with a RISC-V loop consuming near-maximum cycles is straightforward; CKB-VM supports arbitrary RISC-V programs.
- `max_tx_verify_cycles = 70,000,000` is the only upper bound, and it is large enough to produce a ~60× weight discrepancy for a minimal-size transaction.
- The attack is repeatable: after each replacement, the attacker can re-submit to keep the victim's transaction out of the pool indefinitely.

## Recommendation

**For `calculate_min_replace_fee`:** Add a `cycles: u64` parameter alongside `size`, compute the correct weight, and pass it to `fee()`:

```rust
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize, cycles: u64) -> Option<Capacity> {
    let weight = get_transaction_weight(size, cycles);
    let extra_rbf_fee = self.config.min_rbf_rate.fee(weight);
    ...
}
```

Update the call site at `pool.rs:665` to pass `entry.cycles` alongside `entry.size`.

**For `check_tx_fee`:** The cycles are not available at call time. The fix is to perform the fee-rate check **after** script execution when cycles are known, or to pass a conservative upper-bound cycles estimate. At minimum, remove the misleading "cheap check" framing — the current code is the **only** fee-rate gate before pool admission.

## Proof of Concept

1. **Setup:** Node with default config (`min_fee_rate = 1000`, `min_rbf_rate = 1500`).

2. **Submit victim tx1:** Serialized size ≈ 200 bytes, cycles ≈ 1,000, fee = 200 shannons. Effective fee rate = `200 × 1000 / 200 = 1000 shannons/KW` — passes.

3. **Craft attacker tx2:** Same inputs as tx1 (conflict), lock script is a RISC-V loop consuming 70,000,000 cycles, serialized size ≈ 200 bytes.
   - `check_tx_fee`: `min_fee = 1000 × 200 / 1000 = 200 shannons`. Pay 501 shannons → passes.
   - `calculate_min_replace_fee`: `extra_rbf_fee = 1500 × 200 / 1000 = 300 shannons`. `min_replace_fee = 200 + 300 = 500 shannons`. Pay 501 shannons → passes RBF check.

4. **Actual weight of tx2:** `max(200, 70,000,000 × 0.000170571) = max(200, 11,940) = 11,940`.
   - Effective fee rate of tx2 = `501 × 1000 / 11,940 ≈ 41 shannons/KW`.
   - This is 24× below `min_fee_rate` and 36× below `min_rbf_rate`.

5. **Result:** tx1 is evicted from the pool. tx2 occupies the pool with an effective fee rate of ~41 shannons/KW. Miners sort by fee rate and will not include tx2, indefinitely delaying the original transaction. The correct extra RBF fee should have been `1500 × 11,940 / 1000 = 17,910 shannons`; the attacker paid 300 shannons — a 60× shortfall.
