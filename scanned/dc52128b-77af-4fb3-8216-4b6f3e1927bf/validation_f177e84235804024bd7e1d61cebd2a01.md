Audit Report

## Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `from` PeerId — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(from: PeerId, to: PeerId, item_id: u32)` where `from` is parsed directly from the message payload without being validated against the sending session's authenticated peer identity. An attacker can send 30 `ConnectionRequest` messages per second (the outer rate limit) each with a distinct fabricated `from` PeerId, creating 30 new `HashMapStateStore` entries per second. Because `retain_recent()` is never called in the periodic `notify()` handler — only in `disconnected()` — the store grows without bound for the entire lifetime of the connection, enabling memory exhaustion and node crash.

## Finding Description

**Root cause — `from` is attacker-controlled:**
In `connection_request.rs`, `from` is parsed from the raw message bytes with no cross-check against the session's actual peer ID: [1](#0-0) 

This attacker-supplied value is then used directly as the key into `forward_rate_limiter`: [2](#0-1) 

**Outer rate limiter allows 30 messages/sec per session:** [3](#0-2) 

Each of those 30 messages can carry a distinct `from` PeerId, creating 30 new `HashMapStateStore` buckets per second.

**`retain_recent()` is absent from `notify()`:**
The `notify()` handler (fired every `CHECK_INTERVAL` = 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter: [4](#0-3) 

`retain_recent()` is only called on disconnect: [5](#0-4) 

Since the attacker controls the connection lifetime, cleanup never occurs during the attack.

**`forward_rate_limiter` type — unbounded `HashMapStateStore`:** [6](#0-5) [7](#0-6) 

## Impact Explanation

Each `HashMapStateStore` entry holds two `PeerId` values (~39 bytes each) plus a `u32` and governor state. At 30 entries/sec sustained over hours, memory consumption grows linearly without any periodic pruning. This directly causes memory exhaustion and OOM crash of the targeted CKB node.

**Matched impact:** *"Vulnerabilities which could easily crash a CKB node"* — **High (10001–15000 points)**.

## Likelihood Explanation

- Requires only a single persistent P2P connection — no special privileges, no PoW, no keys.
- The `from` field is fully attacker-controlled and is never cross-checked against the session's authenticated peer ID.
- The outer rate limiter (30/sec) is the only throughput cap, and it is set to exactly the rate that drives unbounded growth.
- The attacker controls when to disconnect, so `retain_recent()` is never triggered during the attack.
- The attack is repeatable and sustainable indefinitely.

## Recommendation

1. **Call `retain_recent()` in `notify()`** on both `rate_limiter` and `forward_rate_limiter` every `CHECK_INTERVAL`, mirroring what `disconnected()` already does.
2. **Validate `from` against session identity**: reject any `ConnectionRequest` where `content.from` does not match the authenticated peer ID of the sending session.
3. **Cap `HashMapStateStore` size**: add a hard upper bound on the number of live keys in `forward_rate_limiter` and drop/ban the session if it is exceeded.

## Proof of Concept

```
1. Connect one session to the target node.
2. Send 30 ConnectionRequest/sec for 600 s (10 minutes),
   each with a fresh random `from` PeerId and a fixed `to` PeerId.
3. Assert forward_rate_limiter.len() == 18,000 (not pruned).
4. Trigger notify() — assert len is unchanged (no retain_recent call).
5. Trigger disconnected() — assert len drops to 0.

invariant violated:
  forward_rate_limiter.len() grows as 30 * T (unbounded)

expected invariant (with fix):
  forward_rate_limiter.len() <= 30 * CHECK_INTERVAL_secs == 9,000
  and pruned every 5 min via notify()
```
