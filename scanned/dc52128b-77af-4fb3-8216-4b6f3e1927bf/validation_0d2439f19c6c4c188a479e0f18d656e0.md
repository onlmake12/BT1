## Title
Incorrect `timeout_count` Accumulation in `DownloadScheduler::decrease` Causes Overly Aggressive Peer Penalization â€” (File: `sync/src/types/mod.rs`)

## Summary
In `sync/src/types/mod.rs`, the `DownloadScheduler::decrease` function at line 522 writes `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` is initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32), the expression always produces a value well above 2, making the guard `if self.timeout_count > 2` unconditionally true on every invocation. The intended two-strike tolerance buffer is completely bypassed: `task_count` is decremented on every slow block response, and after 32 such responses the peer's `task_count` reaches 0 and `prune()` forcibly disconnects it.

## Finding Description
`INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` is confirmed at `util/constant/src/sync.rs` line 14. The buggy line is at `sync/src/types/mod.rs` line 522:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // always >= 33
    if self.timeout_count > 2 {                               // always true
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

With the correct fix (`self.timeout_count.saturating_add(num)`), a peer would require ~96 `decrease(1)` calls before disconnection (every 3 calls yields one decrement). The bug reduces this to exactly 32 calls â€” a 3Ă— acceleration of peer eviction.

## Impact Explanation
This maps to **High â€” bad design which could cause CKB network congestion with few costs**. During IBD, a node with many peers (> `protect_num`) where all exhibit moderate latency (NormalToUpper or UpperToMax range, which is the common real-world range under load) will have every peer's `task_count` decremented on every slow block delivery. Peers are evicted 3Ă— faster than intended. If peer replacement cannot keep pace with eviction, the node loses all unprotected sync peers and stalls block download. An adversary controlling multiple peers can deliberately respond at moderate latency to accelerate this eviction without any privileged access.

## Likelihood Explanation
The `NormalToUpper` and `UpperToMax` quantiles cover the majority of real-world cross-continental latency during IBD. The `adjustment` flag is enabled by default during sync. The only guard is `should_punish` (peer count > `protect_num`), which is satisfied whenever a node has more than a small number of connected peers â€” the normal operating state. No special privileges are required; any peer can respond at moderate latency. The condition is reproducible and repeatable.

## Recommendation
Fix line 522 to accumulate `timeout_count` rather than overwrite it with `task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was: self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```
