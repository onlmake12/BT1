Audit Report

## Title
Unauthenticated Resource Exhaustion via Spoofed `from` in Hole-Punching Protocol Spawns Unbounded 30-Second NAT Traversal Tasks — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
A directly connected peer can cause a CKB node to spawn an unbounded number of concurrent 30-second async tasks, each making repeated TCP `connect()` calls to attacker-controlled addresses (including loopback and RFC-1918 ranges), by sending pairs of crafted `ConnectionRequest` and `ConnectionSync` messages with spoofed `from` peer IDs. The per-`from` rate limits are trivially bypassed by rotating fake peer IDs. Sustained attack exhausts file descriptors, async task memory, and CPU, crashing or severely degrading the node.

## Finding Description

**Root cause — no sender identity validation on `from`:**

In `ConnectionRequestProcess::execute`, the `content.from` field is parsed from the message payload but is never checked against the actual authenticated sender's peer ID. Any connected peer can set `from` to an arbitrary `PeerId`. [1](#0-0) 

**Step 1 — Poison `pending_delivered` with attacker-controlled addresses:**

When `content.to == self_peer_id`, `respond_delivered` is called. The supplied `listen_addrs` are filtered only to remove non-TCP transports and addresses without an IP component. Loopback (`127.0.0.1`), link-local, and RFC-1918 private addresses pass through without rejection. [2](#0-1) 

The surviving addresses are stored in `pending_delivered` keyed by the attacker-supplied `from_peer_id` with no verification that it matches the actual sender: [3](#0-2) 

The per-`from` cooldown guard (2-minute `HOLE_PUNCHING_INTERVAL`) is trivially bypassed by using a fresh random `FAKE_ID` for each pair of messages: [4](#0-3) 

**Step 2 — Trigger unbounded task spawning via `ConnectionSync`:**

When the attacker sends `ConnectionSync` with `from=FAKE_ID`, `to=TARGET`, `route=[]`, the node finds `pending_delivered[FAKE_ID]` and spawns a new async task calling `try_nat_traversal` for each stored address. There is no check that the `ConnectionSync` sender is the peer whose ID is in `pending_delivered`: [5](#0-4) 

**`try_nat_traversal` resource cost per task:**

Each spawned task runs for up to 30 seconds, creating a new `TcpSocket` and calling `socket.connect()` every ~175–225 ms (~150 attempts per task): [6](#0-5) 

**Rate limiter analysis — insufficient:**

The session-level `rate_limiter` is keyed by `(session_id, msg.item_id())` and allows **30 messages per second** per message type per session: [7](#0-6) 

The `forward_rate_limiter` is keyed by `(from_peer_id, to_peer_id, msg_item_id)` — with a fresh `FAKE_ID` per pair, each pair gets its own bucket and the limiter is fully bypassed: [8](#0-7) 

At 30 pairs/second, after 30 seconds there are **900 concurrent tasks**, each consuming a goroutine stack, file descriptors, and making ~5 TCP socket operations per second — totalling ~4,500 socket operations/second from a single attacker connection.

## Impact Explanation

This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

900+ concurrent async tasks each holding open sockets and looping for 30 seconds will exhaust the process's file descriptor limit, heap memory for task stacks, and CPU time for the async runtime scheduler. The node becomes unresponsive or crashes. The attack is sustained: as old tasks expire after 30 seconds, the attacker continues sending new pairs to keep the task count at saturation. Additionally, the TCP connections reach arbitrary internal addresses (loopback, RFC-1918), enabling internal port scanning as a secondary impact.

## Likelihood Explanation

The attack requires only a standard authenticated P2P session — no special privileges, no leaked keys, no PoW. Both message types are structurally valid and pass all existing deserialization checks. The two-message sequence is trivially scriptable. The attacker needs only one persistent connection to sustain the attack indefinitely. Multiple attacker connections multiply the effect linearly.

## Recommendation

1. **Validate sender identity in `ConnectionRequest`**: In `ConnectionRequestProcess::execute`, retrieve the actual sender's peer ID from the session context (via `network_state.peer_registry`) and reject any message where `content.from` does not match the authenticated sender peer ID.
2. **Filter non-routable addresses in `respond_delivered`**: Before inserting into `pending_delivered`, reject `listen_addrs` whose IP component is loopback, link-local, or falls within RFC-1918 private ranges.
3. **Correlate `ConnectionSync` sender**: In `ConnectionSyncProcess::execute`, before looking up `pending_delivered[content.from]`, verify that the actual sender's peer ID equals `content.from`. Reject mismatches.
4. **Cap `pending_delivered` size**: Enforce a maximum number of entries in `pending_delivered` to bound memory usage regardless of other fixes.

## Proof of Concept

```
Prerequisites: one standard P2P connection to the target CKB node.

Loop (i = 0, 1, 2, ... up to 30/sec):
  FAKE_ID_i = random PeerId bytes

  1. Send ConnectionRequest:
       from         = FAKE_ID_i
       to           = TARGET_NODE_PEER_ID
       listen_addrs = [/ip4/127.0.0.1/tcp/<PORT_i>/p2p/<FAKE_ID_i>]
       route        = []
       max_hops     = 3

     → Target stores pending_delivered[FAKE_ID_i] = ([/ip4/127.0.0.1/tcp/<PORT_i>/...], now)

  2. Send ConnectionSync:
       from  = FAKE_ID_i
       to    = TARGET_NODE_PEER_ID
       route = []

     → Target finds pending_delivered[FAKE_ID_i],
       spawns runtime::spawn(try_nat_traversal(bind_addr, /ip4/127.0.0.1/tcp/<PORT_i>/...))
       which loops for 30s making TCP connect() to 127.0.0.1:<PORT_i>.

After 30 seconds: 900 concurrent tasks active.
Observable: fd exhaustion, rising RSS, node RPC unresponsive.
Vary PORT_i and IP to scan internal network as secondary effect.
```
