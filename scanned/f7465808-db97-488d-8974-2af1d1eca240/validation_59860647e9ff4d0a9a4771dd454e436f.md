Audit Report

## Title
Unverified Message-Body Peer IDs Allow Any Connected Peer to Exhaust `forward_rate_limiter` Tokens for Arbitrary Peer Pairs, Causing Hole-Punching DoS and Unbounded Memory Growth — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` values taken directly from the message payload, with no verification that the actual TCP session sender matches the claimed `from` peer ID. Any connected peer can send `ConnectionRequest` or `ConnectionRequestDelivered` messages with arbitrary `from`/`to` peer IDs, consuming the forwarding rate-limit token for any peer pair and silently blocking legitimate hole-punching relay for those peers. A secondary consequence is unbounded growth of the `forward_rate_limiter` `HashMapStateStore` while the attacker remains connected, since `retain_recent()` is only called on disconnect, creating a memory exhaustion path toward node crash.

## Finding Description

**Root cause — unverified payload fields used as rate-limit key:**

`HolePunching` maintains two rate limiters:

- `rate_limiter: RateLimiter<(PeerIndex, u32)>` — keyed by the transport-layer `session_id`. Correctly bounds per-sender throughput.
- `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` — keyed by `(content.from, content.to, msg_item_id)` parsed from the wire message. No cross-check against the actual session's peer ID.

The outer check at `mod.rs` L95–107 uses `session_id` from the transport layer and correctly limits to 30 req/sec per session:

```rust
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() { return; }
```

Inside `ConnectionRequestProcess::execute()` at `connection_request.rs` L132–143, the `forward_rate_limiter` is checked using unverified payload values:

```rust
self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
```

The same pattern appears in `ConnectionRequestDeliveredProcess::execute()` at `connection_request_delivered.rs` L134–145. The quota is 1 req/sec per `(from, to, msg_type)` key (`mod.rs` L256).

**Exploit flow:**

1. Attacker (peer E) connects to relay node R.
2. Attacker sends `ConnectionRequest { from=peer_A, to=peer_B, ... }` — R's `forward_rate_limiter` consumes the 1 req/sec token for `(peer_A, peer_B, ConnectionRequest_id)`.
3. Legitimate peer A sends its real `ConnectionRequest { from=peer_A, to=peer_B }` — `forward_rate_limiter.check_key(...)` returns `Err`, message is silently dropped. Peer B never receives the hole-punching request.
4. Attacker repeats step 2 once per second to maintain the DoS. With 30 req/sec budget, attacker can simultaneously block 30 distinct `(from, to)` pairs per session.

**Secondary — unbounded `HashMapStateStore` growth:**

`retain_recent()` is only called in `disconnected()` (`mod.rs` L67–68). While the attacker remains connected and continuously introduces new unique `(from, to)` pairs, the `HashMapStateStore` backing `forward_rate_limiter` grows without bound. At 30 new unique entries/sec per session, with multiple attacker sessions, memory consumption grows continuously until the process is killed or the attacker disconnects.

**Secondary — `inflight_requests` poisoning:**

At `connection_request_delivered.rs` L150–160, when `content.from == local_peer_id` (the node's own public peer ID), the code unconditionally calls `self.protocol.inflight_requests.remove(&content.to)`. An attacker who knows the relay node's peer ID (publicly advertised via discovery) can craft a `ConnectionRequestDelivered` with `from=relay_peer_id, to=victim_peer_id`, destroying the legitimate in-flight request record and causing the real response to be silently ignored as "not in flight."

## Impact Explanation

**Primary — targeted DoS on hole-punching relay (network connectivity degradation):** Nodes behind NAT that rely on this relay node for NAT traversal will silently fail to establish connections. This degrades network connectivity for NAT-ed participants.

**Secondary — memory exhaustion leading to node crash (High: crash a CKB node):** The `forward_rate_limiter` hashmap grows unboundedly while an attacker session is active. An attacker maintaining one or more sessions and continuously varying `(from, to)` pairs can exhaust the node's heap memory, crashing the process. This matches the allowed impact: **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

Any peer that can establish a P2P connection to a CKB node can trigger this — no special privileges, keys, or hashpower are required. Peer IDs are publicly advertised via the discovery protocol. The attacker needs only to maintain 1 spoofed message per second per targeted pair, well within the outer 30 req/sec budget. The memory exhaustion path requires a sustained connection but no additional capability. The hole-punching protocol is active on nodes with `reuse_port_on_linux` or NAT traversal enabled.

## Recommendation

1. **Validate `content.from` against the actual session peer ID** before using it as a rate-limit key. Resolve the authenticated peer ID from `session_id` via the peer registry and reject messages where `content.from` does not match.
2. **Alternatively**, key `forward_rate_limiter` on `(actual_session_peer_id, to, msg_item_id)` rather than the unverified `content.from`, consistent with how the outer `rate_limiter` uses `session_id`.
3. **Call `forward_rate_limiter.retain_recent()` periodically** (e.g., in the `notify` timer at `mod.rs` L169) rather than only on disconnect, to bound memory growth.
4. **Guard the `inflight_requests.remove` path** by verifying the sending session's peer ID matches `content.from` before treating the message as a local delivery.

## Proof of Concept

```
Setup:
  - Relay node R with public peer ID R_id
  - Victim peer A attempting hole-punching to peer B through R
  - Attacker peer E connected to R

Step 1 (rate-limit token exhaustion):
  E sends: ConnectionRequest { from=peer_A, to=peer_B, listen_addrs=[...], ... }
  → R's forward_rate_limiter consumes the 1 req/sec token for (peer_A, peer_B, ConnectionRequest_id)

Step 2 (legitimate request blocked):
  A sends: ConnectionRequest { from=peer_A, to=peer_B, ... }
  → forward_rate_limiter.check_key((peer_A, peer_B, ...)) returns Err(TooManyRequests)
  → R drops the message silently; B never receives the hole-punching request

Step 3 (sustained DoS):
  E repeats Step 1 once per second; A's requests are permanently blocked.
  With 30 req/sec budget, E can simultaneously block 30 distinct (from, to) pairs.

Step 4 (memory exhaustion):
  E continuously sends messages with fresh unique (from_i, to_i) pairs at 30/sec.
  forward_rate_limiter HashMapStateStore grows ~30 entries/sec indefinitely.
  retain_recent() is never called while E stays connected.
  Over hours/days, heap memory is exhausted and the node crashes.

Step 5 (inflight_requests poisoning):
  E sends: ConnectionRequestDelivered { from=R_id, to=peer_A, route=[], ... }
  → R sees content.from == self_peer_id, calls inflight_requests.remove(&peer_A)
  → Legitimate in-flight request for peer_A is destroyed; real response is ignored.
```
