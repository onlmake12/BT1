Audit Report

## Title
Insert-Before-Check Ordering in `add_ask_for_txs` Allows Global Queue Exhaustion and Transaction Relay Disruption - (`sync/src/types/mod.rs`)

## Summary

`SyncState::add_ask_for_txs` inserts up to 32,767 hashes from a peer into the global `unknown_tx_hashes` queue **before** checking whether the global or per-peer limits are exceeded. An attacker with two P2P connections can fill the 50,000-entry global queue with fake hashes, causing all subsequent `RelayTransactionHashes` messages from legitimate peers to return `Status::ignored()` — silently dropping their hashes without requesting the underlying transactions. The "indefinite persistence" claim in the report is overstated (entries do drain via `pop_ask_for_txs` after ~30 seconds), but the attack can be sustained with a rotating pool of source IPs.

## Finding Description

In `add_ask_for_txs` (`sync/src/types/mod.rs`, lines 1486–1528), hashes are unconditionally inserted into `unknown_tx_hashes` first:

```rust
for tx_hash in tx_hashes.into_iter().take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER) {
    match unknown_tx_hashes.entry(tx_hash) {
        Occupied(entry) => { ... }
        Vacant(entry)   => { entry.set_priority(...) }
    }
}
// Check AFTER insertion
if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE { ... }
``` [1](#0-0) 

The per-peer ban (`TooManyUnknownTransactions`, status 416) fires only when the current peer's contribution reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767`. Legitimate peers whose per-peer count is below that threshold receive `Status::ignored()` — no ban, no error, silent drop. [2](#0-1) 

Constants confirmed: [3](#0-2) 

**Correction to the report's "indefinite persistence" claim:** Entries are NOT permanent. `pop_ask_for_txs` naturally evicts entries when `next_request_peer()` returns `None` — which happens for a single-peer entry after it has already been requested once (i.e., after `RETRY_ASK_TX_TIMEOUT_INCREASE = 30s`):

```rust
// next_request_peer() returns None when requested=true and peers.len()==1
// → entry is NOT pushed back → silently dropped
if let Some(peer_index) = priority.next_request_peer() {
    ...
    unknown_tx_hashes.push(tx_hash, priority); // only re-queued if Some
}
``` [4](#0-3) [5](#0-4) 

So fake hashes drain within ~30 seconds of the first `pop_ask_for_txs` pass. However, the attacker can sustain the DoS by cycling through a pool of source IPs, refilling the queue every ~30 seconds. The ban is per-peer-identity (`BAD_MESSAGE_BAN_TIME = 5 minutes`), so a new IP can immediately reconnect and re-fill. [6](#0-5) 

The entry path is confirmed in `TransactionHashesProcess::execute`, which enforces only a per-message size check before calling `add_ask_for_txs`: [7](#0-6) 

## Impact Explanation

When `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) and a legitimate peer's per-peer count is below 32,767, `add_ask_for_txs` returns `Status::ignored()`. The victim node never issues `GetRelayTransactions` for those hashes, so it stops learning about new unconfirmed transactions from that peer for the duration of the attack window. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* Miners on the victim node miss mempool transactions; users whose transactions propagate only through the victim's peers see delayed or missed confirmations.

## Likelihood Explanation

- Any unauthenticated P2P peer can send `RelayTransactionHashes`. No stake, PoW, or privilege required.
- Two connections suffice to fill the queue (32,767 + 17,233 unique fake hashes ≈ 2 MB total).
- The attack window per cycle is ~30 seconds (until `pop_ask_for_txs` drains the fake entries). Sustaining the attack requires one new source IP every ~30 seconds — feasible with a modest botnet or IP rotation service.
- The victim node emits only a `warn!` log; no alarm, no disconnect of legitimate peers.

## Recommendation

1. **Check limits before inserting**: Move the global and per-peer limit checks to the top of `add_ask_for_txs`, before any insertion. Reject or truncate the input if limits would be exceeded.
2. **Evict entries from banned peers**: When a peer is banned, remove all `unknown_tx_hashes` entries exclusively associated with that peer (`priority.peers == [banned_peer]`). This eliminates the ~30-second window of queue pollution.
3. **Rate-limit `RelayTransactionHashes` per peer**: Apply the existing `rate_limiter` (already present in `Relayer`) to this message type to reduce the fill rate.

## Proof of Concept

1. Attacker connects as peer A, sends one `RelayTransactionHashes` with 32,767 unique non-existent hashes.
   - `add_ask_for_txs` inserts all 32,767 → `peer_unknown_counter = 32,767 >= MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` → peer A banned (5 min).
   - 32,767 fake hashes remain in queue.
2. Attacker connects as peer B (new IP), sends 17,233 more unique fake hashes.
   - Queue reaches 50,000. Peer B's per-peer count = 17,233 < 32,767 → `Status::ignored()` (no ban yet).
3. Legitimate peer C sends real tx hashes → `unknown_tx_hashes.len() >= 50,000` → `Status::ignored()` → victim never fetches those transactions.
4. After ~30 seconds, `pop_ask_for_txs` drains the fake entries (single-peer entries with `requested=true` and `peers.len()==1` are dropped).
5. Attacker connects as peer C (new IP) and repeats step 1–2 to refill the queue, sustaining the DoS.
   - Existing integration test `TooManyUnknownTransactions` confirms step 1 (ban on per-peer overflow) but does not cover the cross-peer queue exhaustion scenario in steps 2–5. [8](#0-7)

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(

                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
