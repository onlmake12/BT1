Audit Report

## Title
Missing Duplicate-Hash Deduplication in `GetTransactionsProofProcess::execute` Enables DB Read Amplification — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` accepts up to 1000 tx hashes with no duplicate check, allowing an unprivileged remote peer to send a single `GetTransactionsProof` message containing 1000 copies of the same valid on-chain tx hash. This causes the server to perform 2000 redundant DB reads per message with no rate limiting and no peer ban, since the request completes with `Status::ok()`. The sibling handler `GetBlocksProofProcess::execute` explicitly guards against this with a `HashSet` check, confirming the omission is unintentional.

## Finding Description

**Root cause — missing dedup guard:**

`GetBlocksProofProcess::execute` explicitly rejects duplicate hashes at lines 62–70:

```rust
let mut uniq = HashSet::new();
if !block_hashes
    .iter()
    .chain([last_block_hash].iter())
    .all(|hash| uniq.insert(hash))
{
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate block hash exists");
}
```

`GetTransactionsProofProcess::execute` has no equivalent check. After the size guard at lines 37–39 (which allows exactly 1000 hashes), it immediately iterates the raw list with no deduplication.

**Phase 1 — 1000 `get_transaction_info` DB calls:**

The `partition` closure at lines 54–64 calls `snapshot.get_transaction_info(tx_hash)` for every element in the iterator, including all 1000 duplicates.

**Phase 2 — 1000 `get_transaction_with_info` DB calls:**

The `for tx_hash in found` loop at lines 67–75 calls `snapshot.get_transaction_with_info(&tx_hash)` for every element in `found`, again including all 1000 duplicates.

**Phase 3 — CBMT proof over duplicate indices:**

Because `txs_in_blocks` is a `HashMap` keyed by `block_hash`, all 1000 duplicate entries collapse into a single Vec containing 1000 `(tx, same_index)` tuples. `CBMT::build_merkle_proof` is then called at lines 86–97 with a 1000-element slice of identical indices.

**No rate limiter on the light client path:**

The `received` handler in `lib.rs` at lines 55–92 calls `try_process` directly with no throttling. A grep for `RateLimiter` in the entire `util/light-client-protocol-server/` directory returns no matches.

**Peer is never banned:**

`Status::should_ban()` at lines 95–101 of `status.rs` only triggers on 4xx status codes. Since the duplicate request completes successfully and returns `Status::ok()`, the peer is never disconnected or banned.

## Impact Explanation

Each malicious message causes 1000 × `get_transaction_info` DB reads and 1000 × `get_transaction_with_info` DB reads — a 1000× amplification over a legitimate single-hash request. A single peer can send these messages in a tight loop with no back-pressure, no rate limiting, and no ban. This maps to the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points). The attack requires only a valid on-chain tx hash (publicly observable) and a light client peer connection — no privilege, key, or hashpower.

## Likelihood Explanation

The attack is trivially executable by any unprivileged peer. A valid tx hash is obtainable from any block explorer or full node. The attacker constructs a `GetTransactionsProof` message with `tx_hashes = [H; 1000]` and sends it in a loop. The server processes each message fully, returns `Status::ok()`, and never bans the peer. The asymmetry with `GetBlocksProofProcess` (which has the guard) confirms this is an unintentional omission, not a deliberate design choice.

## Recommendation

Add a duplicate-hash check immediately after the size guard, mirroring `GetBlocksProofProcess`:

```rust
let tx_hashes: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();
let mut uniq = HashSet::new();
if !tx_hashes.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This returns a 4xx status, which `should_ban()` converts into a peer ban, eliminating both the redundant DB work and the attack vector.

## Proof of Concept

1. Obtain any valid on-chain tx hash `H` (e.g., from block 1, publicly visible).
2. Build a `GetTransactionsProof` message: `last_hash = <tip>`, `tx_hashes = [H; 1000]`.
3. Connect as a light client peer and send the message in a tight loop.
4. Observe: server performs 2000 DB reads per message, peer is never banned, no rate limiting applies.
5. Compare DB read count against a request with `tx_hashes = [H]` (1 unique hash → 2 DB reads).
6. Contrast with sending `GetBlocksProof` with 2 duplicate block hashes — that returns `MalformedProtocolMessage` and bans the peer immediately.
