Audit Report

## Title
Minimum Fee Rate Admission Check Uses Raw Byte Size Instead of Transaction Weight, Allowing Cycle-Heavy Transactions to Bypass Fee Rate Enforcement - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate by passing raw serialized byte size to `FeeRate::fee()`, which is defined as shannons per kilo-**weight**. For cycle-heavy transactions, the correct weight (`max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`) can be orders of magnitude larger than the byte size alone, making the admission gate far too lenient. The code itself contains a comment acknowledging this is a known approximation, but no subsequent weight-based fee rate gate exists in the admission path after cycles become known.

## Finding Description
`FeeRate` is explicitly documented as "shannons per kilo-weight":
```rust
/// shannons per kilo-weight
pub struct FeeRate(pub u64);
```
`FeeRate::fee` computes `self.0.saturating_mul(weight) / KW` where `KW = 1000`.

The sole admission-time fee rate gate in `check_tx_fee` passes raw `tx_size` (bytes) as the weight argument:
```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```
The correct weight is computed by `get_transaction_weight`:
```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```
`TxEntry::fee_rate()` already uses this correct weight-based calculation for sorting and eviction, but the admission check does not. For a transaction with 200 serialized bytes and 70,000,000 cycles, the correct weight is `max(200, 70_000_000 * 0.000_170_571_4) ≈ 11,940`, but the check uses only `200`. With `min_fee_rate = 1000 shannons/KW`, the check requires only 200 shannons instead of the correct 11,940 shannons. A transaction paying 201 shannons passes the gate with an actual fee rate of `≈ 16.8 shannons/KW` — roughly 60× below the configured minimum. No subsequent weight-based fee rate check exists in the admission path; `verify_rtx` / `ContextualTransactionVerifier` returns the fee amount and cycles but there is no post-verification fee rate gate.

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker can flood the tx-pool and relay network with computationally expensive transactions (near-maximum VM cycles) while paying fees 60× below the configured minimum. All peers apply the same flawed check, so the spam propagates network-wide. This degrades node performance by forcing full CKB-VM execution of high-cycle transactions that paid inadequate fees, and reduces miner revenue by allowing below-minimum-fee-rate transactions to be mined.

## Likelihood Explanation
The attack requires no special privileges — any RPC caller or relay peer can submit transactions. The maximum cycle limit (70,000,000) is publicly documented. A script author can write a loop script consuming near-maximum cycles with minimal transaction bytes. The discrepancy factor between bytes and weight reaches up to `11,940 / 200 ≈ 60×` for maximally cycle-heavy transactions. The attack is repeatable and systematic.

## Recommendation
Replace the bytes-based minimum fee computation in `check_tx_fee` with a weight-based computation. Since cycles are not available before script verification, the admission pipeline should be restructured so the fee rate check occurs after `verify_rtx` returns actual cycles:
```rust
let weight = get_transaction_weight(tx_size, actual_cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```
This matches how `TxEntry::fee_rate()` already correctly computes the fee rate for sorting and eviction.

## Proof of Concept
1. Craft a CKB transaction with a lock script that loops for ~70,000,000 cycles. Serialized transaction size: ~200 bytes.
2. Submit via RPC `send_transaction` with fee = 201 shannons.
3. `check_tx_fee` computes `min_fee = 1000 * 200 / 1000 = 200 shannons`. Since `201 >= 200`, the transaction is admitted.
4. Actual weight = `max(200, 70_000_000 * 0.000_170_571_4) = 11,940`. Actual fee rate = `201 * 1000 / 11940 ≈ 16.8 shannons/KW`, which is ~60× below the 1,000 shannons/KW minimum.
5. The transaction propagates to all peers (which apply the same flawed check) and may be mined. Repeat to flood the mempool and relay network with high-cycle, low-fee transactions.
