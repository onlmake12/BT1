Audit Report

## Title
Missing Duplicate Index Check in `GetBlockTransactionsProcess` Enables Bandwidth Amplification — (`File: sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute()` validates the *count* of `indexes` and `uncle_indexes` against configured limits but never checks for duplicate values within those lists. An unprivileged peer can send a single `GetBlockTransactions` message with up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) identical indexes, causing the node to serialize and transmit the same transaction 32,767 times in one `BlockTransactions` response. This is a direct, low-cost bandwidth amplification primitive against any reachable CKB relay node.

## Finding Description
In `GetBlockTransactionsProcess::execute()`, the only input guards are count-ceiling checks: [1](#0-0) 

After those checks pass, the handler iterates the raw `indexes` list with no deduplication: [2](#0-1) 

And the same for `uncle_indexes`: [3](#0-2) 

All three analogous relay handlers perform a uniqueness check that this handler omits:

- `GetTransactionsProcess` builds a `HashSet` of `ProposalShortId`s and rejects if `message_len != tx_hashes_set.len()`: [4](#0-3) 

- `GetBlockProposalProcess` converts to a `HashSet` and rejects if `proposals.len() != message_len`: [5](#0-4) 

- `GetBlocksProcess` uses a `dedup` `HashSet` and returns `StatusCode::RequestDuplicate` on collision: [6](#0-5) 

`GetBlockTransactionsProcess` is the sole handler in this class that omits the guard.

## Impact Explanation
`MAX_RELAY_TXS_NUM_PER_BATCH` is defined as **32,767**: [7](#0-6) 

An attacker sends one small fixed-size message with 32,767 copies of index `0` (the cellbase, always present). The node clones, serializes, and transmits 32,767 copies of that transaction in a single response. For a 1 KB transaction this is a ~32 MB response from a negligible request — a ~32,000× amplification ratio. Repeated rapidly from multiple connections, this exhausts the victim node's outbound bandwidth and CPU, degrading or severing its relay participation. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, because the per-attacker cost is a single small message while the per-node cost is orders-of-magnitude larger, and the attack is trivially parallelizable across many nodes simultaneously.

## Likelihood Explanation
The `GetBlockTransactions` message is part of the public `RelayV3` protocol. Any peer that completes the standard handshake — no authentication, no stake, no special role — can send it. The only prerequisite is a valid `block_hash` (trivially obtained from any block explorer or `get_tip_block_hash` RPC) and index `0` (the cellbase always exists). No hash collision, no key material, and no privileged access is required. The attack is repeatable in a tight loop.

## Recommendation
Add a duplicate-index check immediately after the existing count checks in `GetBlockTransactionsProcess::execute()`, consistent with every other relay handler:

```rust
use std::collections::HashSet;

let indexes_set: HashSet<u32> = self.message.indexes().iter()
    .map(Into::<u32>::into).collect();
if indexes_set.len() != self.message.indexes().len() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("Duplicate indexes in GetBlockTransactions");
}

let uncle_indexes_set: HashSet<u32> = self.message.uncle_indexes().iter()
    .map(Into::<u32>::into).collect();
if uncle_indexes_set.len() != self.message.uncle_indexes().len() {
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("Duplicate uncle_indexes in GetBlockTransactions");
}
```

This mirrors the pattern in `get_transactions_process.rs`, `get_block_proposal_process.rs`, and `get_blocks_process.rs`.

## Proof of Concept
1. Connect to any CKB mainnet/testnet node as a relay peer (complete the `RelayV3` handshake).
2. Obtain any committed block hash `H` via `get_tip_block_hash` RPC or a block explorer.
3. Craft a `GetBlockTransactions` message: `block_hash = H`, `indexes = [0, 0, 0, …, 0]` repeated 32,767 times, `uncle_indexes = []`.
4. Send the message to the node.
5. Observe the `BlockTransactions` response: it will contain 32,767 identical serialized copies of the cellbase transaction, with total size `32767 × sizeof(cellbase_tx)` bytes.
6. Repeat in a tight loop from multiple connections to exhaust the node's outbound bandwidth.

The root cause is confirmed at `sync/src/relayer/get_block_transactions_process.rs` lines 61–78, where the raw `indexes` iterator is consumed without any deduplication, in contrast to all three analogous handlers which reject duplicate requests before processing. [8](#0-7)
