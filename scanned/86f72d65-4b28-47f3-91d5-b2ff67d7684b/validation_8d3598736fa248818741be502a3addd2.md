Audit Report

## Title
Missing `from != to` Peer Identity Validation in Hole Punching `ConnectionRequest` Allows Amplified Network Traffic and State Pollution - (File: `network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

The `ConnectionRequestProcess::execute()` function performs multiple input validations but never checks that `content.from != content.to`. Any connected peer can craft a `ConnectionRequest` with identical `from` and `to` peer IDs, causing the receiving node to either (a) insert a self-referential entry into `pending_delivered` and emit a `ConnectionRequestDelivered` with `from == to`, or (b) broadcast the malformed message to `sqrt(peers)` relay nodes per hop across up to `MAX_HOPS = 6` hops, producing meaningful amplified network traffic bounded only by the per-node rate limiter.

## Finding Description

**Root cause — confirmed absent check:**

In `connection_request.rs` lines 110–153, `execute()` validates listen address count, `max_hops`, route length, route self-inclusion, and rate limits, but contains no guard against `content.from == content.to`. [1](#0-0) 

The `from` and `to` fields are free-form attacker-controlled bytes parsed from the molecule message with no authentication binding them to the actual session.

**Case 1 — target node (`self_peer_id == content.to == content.from`):**

`respond_delivered` is called with `from_peer_id = content.from = X`. The deduplication guard at line 161 only suppresses a repeat within `HOLE_PUNCHING_INTERVAL` (2 minutes); the first message per `X` passes through. The function inserts `(X, (remote_listens, now))` into `pending_delivered` keyed by `X` — a self-referential entry where the key equals the local peer ID. [2](#0-1) 

A subsequent `ConnectionSync` with `from == to == X` causes `connection_sync.rs` line 114 to look up `pending_delivered.get(&content.from)`, find the self-referential entry, and spawn async NAT traversal tasks attempting TCP connections to the attacker-supplied listen addresses for up to 30 seconds per task. [3](#0-2) 

**Case 2 — relay node (`self_peer_id != content.to`):**

`forward_message` is called. Because `X` is a spoofed ID absent from the peer registry, the broadcast path is always taken, forwarding to `sqrt(total_peers)` connected nodes. [4](#0-3) 

Each relay node repeats this logic. The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1 req/sec per key per node. [5](#0-4) 

With `from == to == X` and a fixed `msg_item_id` (the message type discriminant, constant for all `ConnectionRequest` messages), each relay node forwards at most 1 message per second per distinct `X`. The session-level rate limiter allows 30 messages/sec per `(session_id, msg_item_id)`. [6](#0-5) 

**Effective amplification:** With 30 distinct spoofed `X` values per second, each relay node processes up to 30 messages/sec. Across a network of N nodes, total traffic reaches 30×N messages/sec from a single attacker session. Route deduplication (`content.route.contains(self_peer_id)`) prevents cycles within a single chain but does not prevent the same node from receiving the message via independent paths with different routes. [7](#0-6) 

**`pending_delivered` growth:** Entries persist for `TIMEOUT = 5 minutes`. At 30 distinct `X` values/sec, up to 9,000 entries accumulate, each storing up to 24 `Multiaddr` values. [8](#0-7) 

**Note on the "routing loop" sub-claim:** The `ConnectionRequestDelivered` response is sent only to `self.peer` (the attacker's session), not re-injected into the node itself. An automatic loop does not occur; the loop requires the attacker to re-send the delivered message, which is an additional attacker action, not a protocol-internal loop.

## Impact Explanation

The concrete impact is amplified P2P overlay traffic and wasted NAT traversal resources reachable by any connected peer with a single session. A single attacker sending 30 crafted messages/sec causes each relay node in a network of N nodes to process up to 30 extra messages/sec, producing 30×N total messages/sec across the overlay. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — the attacker cost is one session and 30 messages/sec; the network cost scales linearly with node count. Additionally, `pending_delivered` accumulates self-referential entries and NAT traversal tasks are spawned against attacker-controlled addresses, consuming sockets and CPU on the target node.

## Likelihood Explanation

Any peer that has established a session with a CKB node supporting the `HolePunching` protocol can send this message. No privilege, key, or special role is required. The `from` and `to` bytes are never authenticated against the session. The only barrier is the session-level rate limiter (30 req/sec), which the attacker saturates by design using 30 distinct spoofed peer ID pairs. The exploit is trivially constructable from the public molecule schema. [9](#0-8) 

## Recommendation

Add an explicit identity check immediately after parsing `content` in `ConnectionRequestProcess::execute()`, and mirror it in `ConnectionRequestDeliveredProcess::execute()` and `ConnectionSyncProcess::execute()`:

```rust
if content.from == content.to {
    return StatusCode::InvalidFromPeerId
        .with_context("from and to peer ids must be different");
}
```

This should be inserted after line 113 (after `RequestContent::try_from`) and before any rate limiter or routing logic, so the check fires before any state mutation or forwarding occurs.

## Proof of Concept

1. Establish a session with a CKB node that has `HolePunching` protocol enabled.
2. Choose any valid-length byte sequence as a peer ID (e.g., a 39-byte multihash `X`). Set both `from` and `to` to `X`.
3. Construct a `ConnectionRequest` molecule message: `from = to = X`, `max_hops = 6`, empty `route`, 1–24 valid TCP/IPv4 multiaddresses in `listen_addrs` (with peer ID `X` appended).
4. Send the message over the hole punching protocol stream.
5. **If the target node's peer ID equals `X`**: observe `respond_delivered` is called, `pending_delivered[X]` is set with the supplied listen addresses, and a `ConnectionRequestDelivered` with `from == to == X` is returned. Then send a `ConnectionSync` with `from == to == X`; observe NAT traversal tasks spawned to the attacker-supplied addresses.
6. **If the target node's peer ID differs from `X`**: observe the message is forwarded to `sqrt(peers)` other nodes. Repeat with 30 distinct `X` values per second to sustain amplification across the overlay.
7. Monitor relay node message rates to confirm 30×N total messages/sec across the network from a single attacker session.

    pub(crate) async fn execute(mut self) -> Status {
        let content = match RequestContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }
