### Title
DAO Withdrawal Header-Dep Index Width Mismatch Between Rust `DaoCalculator` and On-Chain `dao.c` Script — (`File: util/dao/src/lib.rs`)

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the header-deps index from the witness `input_type` field as a full `u64`, while the on-chain `dao.c` script reads only the lowest byte (`u8`). When a DAO withdrawal transaction encodes an index value greater than 255, the two implementations resolve different entries in `header_deps`, producing a consensus split: the on-chain script accepts the transaction while the Rust node rejects it. This is documented by the test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs`, which explicitly records the divergence.

### Finding Description

**Root cause — `util/dao/src/lib.rs`, lines 91 and 96:**

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))   // line 91
// ...
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)                             // line 96
```

The witness `input_type` field is 8 bytes. The Rust code reads all 8 bytes as a `u64` and uses that value directly as the `header_deps` array index. The on-chain `dao.c` script reads only the lowest byte (effectively treating the index as `u8`). [1](#0-0) 

**Documented discrepancy — `util/dao/src/tests.rs`, lines 489–536:**

The test constructs a transaction with 258 `header_deps`, places the deposit block at position 1 and the withdraw block at position 257, then encodes witness index `257` (little-endian u64). The comment explicitly states:

> "Position 1: correct deposit block (what C VM resolves via lowest byte). Position 257: withdraw block (wrong — Rust resolves this with full u64)."

The test asserts `result.is_err()`, confirming the Rust node rejects a transaction the on-chain script would accept. [2](#0-1) 

**Propagation into block verification:**

`transaction_maximum_withdraw` is also called by `withdrawed_interests`, which feeds into `dao_field_with_current_epoch` / `dao_field`. The `DaoHeaderVerifier` in `verification/contextual/src/contextual_block_verifier.rs` calls `dao_field` to verify the DAO field in every block header. A block containing a crafted DAO withdrawal would produce a different `withdrawed_interests` value in the Rust node than the on-chain script computes, causing the Rust node to reject the block as `InvalidDAO`. [3](#0-2) [4](#0-3) 

`transaction_fee` (which calls `transaction_maximum_withdraw`) is also used in `check_tx_fee` in the tx-pool admission path: [5](#0-4) 

### Impact Explanation

An attacker (or any DAO user) who crafts a DAO withdrawal transaction with a witness index > 255 and arranges `header_deps` so that `header_deps[index & 0xFF]` is the deposit block while `header_deps[index]` is a different block will produce a transaction that:

1. **Is accepted by the on-chain `dao.c` script** (CKB-VM uses lowest byte).
2. **Is rejected by the Rust node's tx-pool** (`check_tx_fee` → `transaction_fee` → `transaction_maximum_withdraw` resolves the wrong header dep, fails the block-number cross-check at line 105, returns `DaoError::InvalidOutPoint`).
3. **Causes the Rust node to reject any block containing this transaction** as `InvalidDAO` because `dao_field` / `withdrawed_interests` computes a different interest value than the on-chain script.

This is a **consensus split**: non-Rust nodes (or nodes using the on-chain script as the authority) accept the block; Rust nodes reject it. A miner can produce a valid block that a majority of Rust-based nodes refuse to follow, or a DAO user's valid withdrawal is permanently blocked at the Rust node layer.

### Likelihood Explanation

- Requires constructing a DAO withdrawal with ≥ 258 `header_deps` entries (index 257 is the minimum to exceed u8). Each `header_dep` is 32 bytes; 258 entries = ~8 KB, well within the transaction size limit.
- No privileged access is required. Any DAO depositor can craft this transaction.
- The attacker controls the witness bytes and the `header_deps` list entirely.
- The discrepancy is already documented in the test suite, confirming it is a known, reproducible divergence.

### Recommendation

The Rust `DaoCalculator` must read the header-deps index using only the lowest byte to match `dao.c` behavior:

```rust
// Current (wrong):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))

// Corrected (match dao.c u8 read):
Ok(u64::from(header_deps_index_data.unwrap()[0]))
```

Alternatively, add an explicit bounds check that rejects any index > 255 with `DaoError::InvalidDaoFormat` before the lookup, so both the Rust node and the on-chain script agree on the rejection. Either fix must be applied consistently in `transaction_maximum_withdraw` (used by both `transaction_fee` and `withdrawed_interests`). [6](#0-5) 

### Proof of Concept

1. Deposit CKB into the Nervos DAO normally (phase 1).
2. Construct the phase-2 withdrawal transaction:
   - Set `header_deps` to a list of 258 entries: `[dummy×256, deposit_block_hash, dummy, withdraw_block_hash]` — deposit block at index 1 (= `257 & 0xFF`), withdraw block at index 257.
   - Set the witness `input_type` to `257u64` in little-endian (8 bytes: `0x01 0x01 0x00 0x00 0x00 0x00 0x00 0x00`).
   - Cell data contains the deposit block number (e.g., 100).
3. Submit to the Rust node: `check_tx_fee` calls `transaction_fee` → `transaction_maximum_withdraw` reads index 257 → resolves `header_deps[257]` = withdraw block (number 200) → `deposit_header.number()` (200) ≠ `deposited_block_number` (100) → `DaoError::InvalidOutPoint` → tx rejected.
4. Submit the same transaction to a node running the on-chain script directly: `dao.c` reads lowest byte = 1 → resolves `header_deps[1]` = deposit block (number 100) → 100 == 100 → accepted.
5. A miner who includes this transaction produces a block that Rust nodes reject as `InvalidDAO`, confirming the consensus split.

        &self,
        mut rtxs: impl Iterator<Item = &'a ResolvedTransaction> + Clone,
    ) -> Result<Capacity, DaoError> {
        let maximum_withdraws = rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
