### Title
`get_cells_capacity` Uses Inclusive Upper Bound for `script_len_range` While `get_cells` Uses Exclusive Upper Bound, Causing Inflated Capacity Results - (`util/indexer/src/service.rs`)

### Summary

The Indexer RPC's `get_cells_capacity` function applies `script_len > r1` (inclusive upper bound) when filtering by `script_len_range`, while the sibling `get_cells` function correctly applies `script_len >= r1` (exclusive upper bound). The documented API contract for `script_len_range` is `[inclusive, exclusive]`. As a result, any RPC caller supplying a `script_len_range` filter to `get_cells_capacity` receives an inflated total capacity that includes cells whose script length equals the exclusive upper bound, cells that `get_cells` would correctly exclude.

### Finding Description

The `script_len_range` filter is documented as `[inclusive, exclusive]` across all indexer RPC methods.

In `get_cells` (`util/indexer/src/service.rs`, lines 299â€“318), the filter is applied correctly with `>= r1`:

```rust
if script_len < r0 || script_len >= r1 {   // exclusive upper bound â€” correct
    return None;
}
```

In `get_cells_capacity` (`util/indexer/src/service.rs`, lines 776â€“794), the same filter is applied with `> r1`:

```rust
if script_len < r0 || script_len > r1 {    // inclusive upper bound â€” wrong
    return None;
}
```

This means cells where `script_len == r1` pass the filter in `get_cells_capacity` but are rejected in `get_cells`. The two functions are supposed to be consistent: `get_cells_capacity` should return the sum of capacities of exactly the cells that `get_cells` would enumerate.

An unprivileged RPC caller triggers this by calling `get_cells_capacity` with any `filter.script_len_range` whose upper bound coincides with a real script length present in the indexed UTXO set.

### Impact Explanation

Any application or wallet that uses `get_cells_capacity` to determine available CKB capacity for a given script-length range will receive an overstated value. The overcounting equals the total capacity of all live cells whose script length is exactly `r1`. For a caller constructing transactions based on this capacity estimate, the discrepancy can cause transaction assembly failures or incorrect fee/balance calculations. The inconsistency between `get_cells` and `get_cells_capacity` also breaks the invariant that the two endpoints agree on which cells are in scope, which can be exploited to confuse applications that cross-check both endpoints.

### Likelihood Explanation

The entry path is a standard, unauthenticated JSON-RPC call to `get_cells_capacity` with a `filter.script_len_range` parameter. No privilege is required. Any RPC caller â€” including a wallet, dApp, or automated script â€” that uses this filter is affected. The bug is deterministic and reproducible with any `script_len_range` whose upper bound matches an existing script length.

### Recommendation

Change the comparison in `get_cells_capacity`'s `script_len_range` filter from `> r1` to `>= r1` to match the documented exclusive upper bound and the behavior of `get_cells`:

```diff
- if script_len < r0 || script_len > r1 {
+ if script_len < r0 || script_len >= r1 {
```

Apply this fix to both the `Lock` and `Type` branches inside the `get_cells_capacity` closure.

**Concrete scenario:**

1. The indexed UTXO set contains a live cell whose type script serialized length is exactly 65 bytes.
2. Caller A calls `get_cells` with `script_len_range: ["0x0", "0x41"]` (i.e., `[0, 65)`). The cell with `script_len == 65` is **excluded** (correct per spec).
3. Caller A calls `get_cells_capacity` with the same filter. The cell with `script_len == 65` is **included** because `65 > 65` is false, so the filter passes.
4. The capacity returned by `get_cells_capacity` is larger than the sum of capacities of cells returned by `get_cells`, violating the invariant that the two endpoints agree on the in-scope cell set.
