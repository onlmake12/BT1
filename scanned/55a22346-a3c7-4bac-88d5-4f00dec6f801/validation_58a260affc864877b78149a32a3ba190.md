### Title
Unauthenticated Miner Notify HTTP Endpoint Allows Any Network Peer to Inject Arbitrary Block Templates and Redirect Mining Rewards - (`miner/src/client.rs`)

### Summary

When the CKB miner is configured in "notify mode" (`listen` option set), it binds a TCP HTTP server that accepts block template updates from any connecting client with no authentication whatsoever. The `handle()` function at `miner/src/client.rs:358-368` accepts any well-formed `BlockTemplate` JSON payload and immediately passes it to `update_block_template()`, which dispatches the injected work to all mining workers. An attacker who can reach the miner's listen port can inject a crafted `BlockTemplate` containing an attacker-controlled coinbase lock script, causing the miner to expend its full hashpower mining blocks whose rewards are paid to the attacker.

### Finding Description

The miner supports two operating modes: poll mode (default) and notify mode. In notify mode, the miner binds a TCP listener at the configured `listen` address and waits for the CKB node's block assembler to POST new block templates via HTTP.

The HTTP handler is:

```rust
// miner/src/client.rs:358-368
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
```

There is no check on the source IP, no `Authorization` header validation, no shared secret, and no token. Any HTTP client that can reach the port and POST a valid `BlockTemplate` JSON object will have its template accepted.

`update_block_template()` then converts the injected `BlockTemplate` into a `Work` struct and sends it to all mining workers via the `new_work_tx` channel:

```rust
// miner/src/client.rs:293-311
fn update_block_template(&self, block_template: BlockTemplate) {
    let work_id = block_template.work_id.into();
    let updated = |id| {
        if id != work_id || id == 0 { Some(work_id) } else { None }
    };
    if self.current_work_id
        .fetch_update(Ordering::SeqCst, Ordering::SeqCst, updated)
        .is_ok()
    {
        let work: Work = block_template.into();
        if let Err(e) = self.new_work_tx.send(Works::New(work)) { ... }
    }
}
```

The `Work` struct wraps the full `Block` derived from the template, including the `cellbase` transaction. The `cellbase` contains the lock script that controls who receives the block reward. The `BlockTemplate.cellbase` field is fully attacker-controlled in the injected payload.

When a worker finds a valid nonce, `submit_nonce()` in `miner/src/miner.rs:140-188` assembles the final block from the injected work and submits it to the CKB node via `submit_block`. The CKB node will accept a block with any valid cellbase lock script — it does not enforce that the coinbase address matches the node's configured `block_assembler`.

The `listen` field is typed as `Option<SocketAddr>` in `util/app-config/src/configs/miner.rs:29`, meaning it can be bound to any interface including `0.0.0.0`. The default config template comments it out with an example of `127.0.0.1:8888`, but operators running mining pools or distributed setups routinely bind to non-loopback addresses.

The legitimate notify flow from the CKB node's block assembler (`tx-pool/src/block_assembler/mod.rs:683-711`) POSTs to the configured `notify` URLs with no authentication either — the design assumes the channel is trusted, but the miner's listener enforces no such trust.

### Impact Explanation

An attacker who can reach the miner's notify port can:

1. **Redirect all mining rewards**: By injecting a `BlockTemplate` with a cellbase lock script pointing to the attacker's address, every block the miner finds pays the block reward (primary + secondary issuance) to the attacker. At current CKB issuance rates this is a direct, continuous theft of CKB tokens proportional to the miner's hashrate.
2. **Waste miner hashpower**: Even without stealing rewards, the attacker can inject templates that cause the miner to mine stale or invalid blocks, denying the legitimate operator all revenue.
3. **Persistent control**: The attacker can continuously re-inject templates (e.g., every few seconds) to override any legitimate template the CKB node sends, maintaining persistent control for as long as they have network access to the port.

### Likelihood Explanation

- The `listen` option is opt-in and disabled by default, which limits exposure to operators who explicitly enable notify mode.
- However, notify mode is the recommended configuration for mining pools and high-performance setups (the log message at `miner/src/client.rs:208-219` actively encourages its use).
- Any operator who binds to a non-loopback address (e.g., `0.0.0.0:8888` in a multi-machine pool) is fully exposed to any peer on that network segment.
- Even a `127.0.0.1` binding is exploitable via SSRF vulnerabilities in co-located services, shared-host environments, or container misconfigurations.
- The attack requires only the ability to send an HTTP POST — no cryptographic material, no privileged access, and no prior knowledge beyond the port number.

### Recommendation

Add authentication to the `handle()` function. The miner already implements `parse_authorization()` for outgoing RPC requests (`miner/src/client.rs:380-394`). The same Basic Auth mechanism should be applied to incoming notify requests:

1. Add a `notify_auth` secret field to `ClientConfig` in `util/app-config/src/configs/miner.rs`.
2. In `handle()`, extract the `Authorization` header from the incoming request and compare it against the configured secret using a constant-time comparison before calling `update_block_template()`.
3. Reject (return HTTP 401) any request that lacks or fails the authorization check.
4. Document that the `listen` address should be bound to `127.0.0.1` unless the operator has secured the network path independently.

### Proof of Concept

**Precondition**: Miner is configured with `listen = "0.0.0.0:8888"` (or any reachable address).

**Steps**:

1. Craft a `BlockTemplate` JSON with an attacker-controlled `cellbase.data.outputs[0].lock` (the coinbase lock script pointing to the attacker's address). The `work_id` must differ from the current one (e.g., use `"0x1"` or any value not currently active).

2. POST the payload to the miner's notify endpoint:
   ```
   curl -X POST http://<miner-ip>:8888/ \
     -H "Content-Type: application/json" \
     -d '{"version":"0x0","compact_target":"0x1e083126","current_time":"0x...","number":"0x...","epoch":"0x...","parent_hash":"0x...","cycles_limit":"0x...","bytes_limit":"0x...","uncles_count_limit":"0x2","uncles":[],"transactions":[],"proposals":[],"cellbase":{"cycles":null,"data":{"cell_deps":[],"header_deps":[],"inputs":[{"previous_output":{"index":"0xffffffff","tx_hash":"0x000..."},"since":"0x..."}],"outputs":[{"capacity":"0x...","lock":{"args":"<ATTACKER_LOCK_ARG>","code_hash":"<SECP256K1_CODE_HASH>","hash_type":"type"},"type":null}],"outputs_data":["0x"],"version":"0x0","witnesses":["0x..."]}},"work_id":"0x1","dao":"0x..."}'
   ```

3. The `handle()` function accepts the payload without any authentication check and calls `update_block_template()`.

4. The miner's workers immediately switch to mining the injected block. When a valid nonce is found, `submit_nonce()` submits the block to the CKB node with the attacker's coinbase address.

5. The CKB node accepts the block (it validates PoW and consensus rules, not coinbase ownership relative to the node's config), and the block reward is credited to the attacker's address.

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}

        let parent_hash = block.parent_hash();

            debug!(

                block.number(),
                pow_hash,
                block.header()
            );
            self.notify_workers(WorkerMessage::Start);
            return;
        } else {
            debug!(
                "block {} pow_hash: {:#x}, header: {}",
                block.number(),
                pow_hash,
                block.header()
            );
        }

        self.legacy_work.put(parent_hash, ());
        if self.stderr_is_tty {
            debug!("Found! #{} {:#x}", block.number(), block_hash);

            info!("Found! #{} {:#x}", block.number(), block_hash);
