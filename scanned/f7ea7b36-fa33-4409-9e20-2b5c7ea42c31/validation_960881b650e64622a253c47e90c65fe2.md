The code confirms the claim. Here is the full trace:

1. `remove_entry_and_descendants` calls `remove_entry_links` for **every** entry in the removed set (including the root) before any `remove_entry` call.
2. `remove_entry_links` calls `self.links.remove(id)`, which deletes the entry from `TxLinksMap::inner`.
3. When `remove_entry` is subsequently called, it invokes `update_ancestors_index_key`, which calls `self.links.calc_ancestors(short_id)`.
4. `calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)` — returning `None` since the entry was already removed — and returns an empty `HashSet`.
5. The loop in `update_ancestors_index_key` never executes; `sub_descendant_weight` is never called on any surviving ancestor.

The developer comment at L256 acknowledges only the intentional suppression of `update_descendants_index_key` (correct, since those entries are being removed), but the same mechanism silently suppresses the necessary ancestor update.

Audit Report

## Title
`remove_entry_and_descendants` Fails to Decrement Surviving Ancestors' `descendants_*` Fields, Permanently Inflating Their Eviction Weight — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::remove_entry_and_descendants`, all link records for the removed subtree are torn down via `remove_entry_links` before `remove_entry` is called on each entry. Because `update_ancestors_index_key` resolves ancestors through the live link map (`TxLinksMap`), it finds nothing and silently skips decrementing `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` on any ancestor that survives outside the removed subtree. Those fields remain permanently inflated for the lifetime of the surviving ancestor, corrupting its `EvictKey` and causing incorrect eviction ordering.

## Finding Description
`remove_entry_and_descendants` (L252–265) first collects all descendant IDs, then iterates over the full set calling `remove_entry_links` on each, including the root:

```rust
for id in &removed_ids {
    self.remove_entry_links(id);   // strips id from TxLinksMap
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` (L418–430) removes the entry from its parents' children sets, from its children's parents sets, and then calls `self.links.remove(id)`, which deletes the entry's record from `TxLinksMap::inner` entirely.

When `remove_entry` (L235–250) is subsequently called, it invokes `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` (L242). That function (L432–445) calls `self.links.calc_ancestors(&child.proposal_short_id())`, which resolves to `calc_relative_ids` → `self.inner.get(short_id)`. Since the entry's record was already deleted, `get` returns `None`, the direct set is empty, and `calc_ancestors` returns an empty `HashSet`. The `for anc_id in &ancestors` loop body never executes; `sub_descendant_weight` is never called on any surviving ancestor.

The developer comment at L256 reads:
```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
```
This acknowledges only the intentional suppression of descendant updates (correct, since those entries are being removed), but the same mechanism also silently suppresses the necessary ancestor update.

The `EvictKey` (entry.rs L234–247) is computed directly from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`. An ancestor with inflated values appears more valuable than it actually is and is systematically deprioritized for eviction.

## Impact Explanation
Every surviving ancestor of the removed subtree retains stale, inflated `descendants_*` values permanently. These feed directly into `EvictKey::fee_rate` (computed as `max(descendants_feerate, own_feerate)`), which governs which transactions are selected for eviction in `limit_size` (pool.rs L290–328). A low-fee parent whose child was removed via RBF or size-limit eviction will have an inflated `EvictKey`, making it resistant to eviction. Legitimate higher-fee transactions submitted later will be evicted in its place. An attacker can repeat this pattern at low cost to keep a low-fee parent alive indefinitely, displacing legitimate transactions and causing effective transaction censorship and pool quality degradation. This matches the allowed impact: **High — bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The trigger path is fully reachable by any unprivileged RPC caller via `send_transaction`. No special privileges, keys, or majority hash power are required. The attacker submits a low-fee parent (tx_A) and a high-fee child (tx_B), then replaces tx_B via RBF. Each replacement cycle inflates tx_A's `descendants_*` fields by the replaced child's fee/size/cycles. The cost per inflation cycle is bounded only by the minimum RBF fee bump, which is small. The scenario is reproducible with two ordinary transactions and is repeatable indefinitely.

## Recommendation
Before the `remove_entry_links` loop, identify the surviving ancestors of the root entry and update their descendant accounting while the link map is still intact. Concretely:

1. Before the loop at L257, call `update_ancestors_index_key(root_entry, EntryOp::Remove)` while the root's link record still exists.
2. Then proceed with the existing link-removal loop and `remove_entry` calls. The subsequent `update_ancestors_index_key` call inside `remove_entry` for the root will find an empty ancestor set (links already gone) and correctly do nothing; descendant entries being removed will also correctly skip their ancestor updates.

Alternatively, collect the set of surviving ancestors of the root before any link removal, and explicitly call `sub_descendant_weight` on each of them after the subtree is removed.

## Proof of Concept
```
Setup: tx_A (parent, fee=1 shannon) → tx_B (child, fee=10000 shannons)

1. submit tx_A  → pool accepts; tx_A.descendants_fee = 1 (self only)
2. submit tx_B  → pool accepts; tx_A.descendants_fee = 10001 (self + tx_B)
3. submit tx_B' replacing tx_B via RBF
   → remove_entry_and_descendants(tx_B) is called
   → remove_entry_links(tx_B): removes tx_B from TxLinksMap::inner
   → remove_entry(tx_B): calc_ancestors(tx_B) == {} (links gone)
   → sub_descendant_weight never called on tx_A
   → tx_A.descendants_fee remains 10001  ← BUG: should be 1
4. Pool approaches max_tx_pool_size; eviction runs
   → tx_A.EvictKey.fee_rate computed from inflated descendants_fee=10001
   → tx_A survives eviction despite having no real high-fee descendants
   → a legitimate high-fee transaction is rejected instead
5. Repeat steps 3–4 with tx_B'', tx_B''', etc. to inflate tx_A further each cycle
```

A unit test can be written against `PoolMap` directly: add tx_A and tx_B, call `remove_entry_and_descendants(&tx_B_id)`, then assert `pool.get(&tx_A_id).unwrap().descendants_fee == tx_A.fee` — this assertion will fail, demonstrating the bug.
