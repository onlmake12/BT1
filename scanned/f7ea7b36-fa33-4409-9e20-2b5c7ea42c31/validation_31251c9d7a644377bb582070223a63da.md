Audit Report

## Title
Unbounded Linear Walk in `last_common_ancestor` Enables Repeated CPU Overhead via PoW-Backed Fork Headers — (File: `sync/src/types/mod.rs`)

## Summary
`ActiveChain::last_common_ancestor` contains a `while` loop that walks back one block at a time with no iteration cap. An attacker who mines a valid fork of depth D and repeatedly connects/disconnects forces the victim to execute O(D) `get_ancestor` calls per connection event. No bound, timeout, or iteration guard exists anywhere in the call chain from the timer callback to the loop body.

## Finding Description
The loop at lines 1846–1853 of `sync/src/types/mod.rs` decrements both chain tips by exactly 1 per iteration until they converge:

```rust
while m_left != m_right {
    m_left = self.get_ancestor(&m_left.hash(), m_left.number() - 1)?.number_and_hash();
    m_right = self.get_ancestor(&m_right.hash(), m_right.number() - 1)?.number_and_hash();
}
```

`ActiveChain::get_ancestor` (lines 1721–1772 of `sync/src/types/mod.rs`) uses a skip-list internally, but since each call requests `number - 1` (exactly one step back), the skip-list provides no benefit and each call is effectively O(1). The loop therefore runs O(fork_depth) iterations total.

`update_last_common_header` in `sync/src/synchronizer/block_fetcher.rs` (lines 61–104) calls `last_common_ancestor` unconditionally on every invocation. After the first call, `set_last_common_header` stores the fork point, and subsequent timer ticks start from that stored value — meaning the loop runs 0 iterations in steady state. However, the O(D) walk recurs on every new peer connection: when the attacker disconnects and reconnects, the stored `last_common_header` for that peer index is cleared, and the full D-iteration walk is triggered again on the next timer tick.

`find_blocks_to_fetch` in `sync/src/synchronizer/mod.rs` (lines 735–847) dispatches `BlockFetcher::fetch` for every qualifying peer on every firing of `NOT_IBD_BLOCK_FETCH_TOKEN` and `IBD_BLOCK_FETCH_TOKEN`. The `HeadersProcess` accepts up to `MAX_HEADERS_LEN = 2000` headers per message and automatically requests more if the batch is full (line 183–186 of `sync/src/synchronizer/headers_process.rs`), allowing an attacker to build an arbitrarily deep fork across multiple message batches. No cap on total headers stored per peer exists.

The claim that the O(D) walk fires "on every subsequent timer tick" is inaccurate for the steady-state case — it fires once per connection event, not once per timer tick. The attacker must reconnect to re-trigger it.

## Impact Explanation
Each reconnection by an attacker with a pre-mined fork of depth D causes O(D) in-memory `HeaderMap` lookups in the fetch worker thread. With multiple attacker-controlled peers each reconnecting rapidly, the aggregate CPU load scales as O(K × D) per connection wave. While individual iterations are fast (in-memory skip-list lookups), a sufficiently deep fork (D in the tens of thousands) combined with rapid reconnection cycles and multiple peers can produce sustained elevated CPU usage. This maps to **Low (501–2000 points): Any other important performance improvements for CKB**, as the practical impact is CPU overhead and performance degradation rather than a node crash, given the PoW cost required to build the fork.

## Likelihood Explanation
The attacker must mine D valid PoW-verified headers — a real, non-trivial cost. However, this cost is paid once; thereafter, the attacker can reconnect repeatedly using the same pre-mined headers with only network overhead. There is no reconnection rate limit or ban triggered by valid header submission. The attack is self-sustaining after the initial PoW investment and requires no further mining. A moderately resourced attacker can sustain this against a target node indefinitely.

## Recommendation
Add an explicit iteration cap to `last_common_ancestor`. If the common ancestor is not found within a configurable maximum number of steps (e.g., `BLOCK_DOWNLOAD_WINDOW = 8192` or a fixed constant), return `None` and treat the peer as unusable for block fetching. Additionally, consider replacing the O(N) linear walk with a binary-search approach using the existing skip-list `get_ancestor` path, which already supports O(log N) ancestor lookup and would reduce the per-connection cost from O(D) to O(log²D).

## Proof of Concept
1. Attacker pre-mines a private fork of depth D (e.g., D = 10,000) starting from a recent main-chain block at height F. All headers pass PoW verification.
2. Attacker connects to the victim as a sync peer and sends the D fork headers across multiple `SendHeaders` batches (each up to 2,000 headers). `HeadersProcess::execute` accepts each batch and updates `best_known_header` to the fork tip at height F+D.
3. Victim's `NOT_IBD_BLOCK_FETCH_TOKEN` timer fires. `find_blocks_to_fetch` dispatches `BlockFetcher::fetch` for the attacker peer. `update_last_common_header` calls `last_common_ancestor`, which runs D iterations.
4. Attacker disconnects and immediately reconnects, re-sending the same D headers. The stored `last_common_header` for the new peer index is absent, so the full D-iteration walk recurs on the next timer tick.

    fn get_ancestor_internal(
        &self,
        base: &Byte32,
        number: BlockNumber,
        with_unverified: bool,
    ) -> Option<HeaderIndexView> {
        let tip_number = {
            if with_unverified {
                self.unverified_tip_number()
            } else {
                self.tip_number()
            }
        };

        let block_is_on_chain_fn = |hash: &Byte32| {
            if with_unverified {
                self.is_unverified_chain(hash)
            } else {
                self.is_main_chain(hash)
            }
        };

        let get_header_view_fn = |hash: &Byte32, store_first: bool| {
            self.sync_shared.get_header_index_view(hash, store_first)
        };

        let fast_scanner_fn = |number: BlockNumber, current: BlockNumberAndHash| {
            // shortcut to return an ancestor block
            if current.number <= tip_number && block_is_on_chain_fn(&current.hash) {
                self.get_block_hash(number)
                    .and_then(|hash| self.sync_shared.get_header_index_view(&hash, true))
            } else {
                None
            }
        };

        self.sync_shared
            .get_header_index_view(base, false)?
            .get_ancestor(tip_number, number, get_header_view_fn, fast_scanner_fn)
    }
