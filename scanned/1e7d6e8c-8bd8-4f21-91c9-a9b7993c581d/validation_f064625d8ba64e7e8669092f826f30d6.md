Audit Report

## Title
Missing Per-Peer Rate Limit in Filter Protocol Enables DB Read Amplification and Bandwidth Exhaustion — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_hashes_process.rs`)

## Summary
The `BlockFilter` protocol handler contains no per-peer rate limiter, while both `Relayer` and `HolePunching` enforce 30 req/sec per `(PeerIndex, message_type)`. Any unprivileged peer can flood a node with `GetBlockFilterHashes` messages, each triggering up to 2000 sequential RocksDB reads and a ~64 KB response, with no throttle, no ban, and no backpressure applied.

## Finding Description
`BlockFilter` struct holds only `shared: Arc<SyncShared>` with no `rate_limiter` field. [1](#0-0) 

`try_process` dispatches directly to all three filter handlers with zero rate checks before any handler is invoked. [2](#0-1) 

`GetBlockFilterHashesProcess::execute` iterates up to `BATCH_SIZE = 2000` blocks, calling `get_block_hash` + `get_block_filter_hash` per iteration (2 DB reads each), then serializes and sends the full result. [3](#0-2) [4](#0-3) 

By contrast, `Relayer` carries an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` initialized at 30 req/sec and checks it before every handler dispatch, returning `StatusCode::TooManyRequests` on excess. [5](#0-4) [6](#0-5) 

The only existing guard in `BlockFilter` is a ban for malformed messages; there is no equivalent guard for well-formed but high-frequency requests. [7](#0-6) 

`GetBlockFilterCheckPointsProcess` and `GetBlockFiltersProcess` share the same unguarded dispatch path and are equally exploitable. [8](#0-7) 

## Impact Explanation
Each `GetBlockFilterHashes(start_number=0)` from a single peer causes up to 4000 RocksDB reads and a ~64 KB outbound response. With no rate limit, a small number of attacker connections can saturate the node's I/O and outbound bandwidth, making the node unresponsive to legitimate peers. This maps to **High: Vulnerabilities which could easily crash a CKB node** and **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points).

## Likelihood Explanation
The attack requires only a valid P2P connection to a node that has built at least one block of filter data (the `latest >= start_number` precondition at line 39 of `get_block_filter_hashes_process.rs`). No PoW, no keys, no special privileges are needed. The request message is ~8 bytes; the response is ~64 KB — a classic amplification scenario. The attack is trivially repeatable in a tight loop.

## Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer::rate_limiter`) initialized at 30 req/sec. At the top of `try_process`, check `rate_limiter.check_key(&(peer, message.item_id()))` and return `StatusCode::TooManyRequests` on failure, before dispatching to any of the three filter handlers. This mirrors the existing pattern in `Relayer::try_process`. [9](#0-8) 

## Proof of Concept
1. Connect a single peer to a node that has built at least 1 block of filter data.
2. In a tight loop, send `GetBlockFilterHashes { start_number: 0 }` 1000 times over the Filter protocol.
3. Observe: the node performs ~4,000,000 RocksDB reads and sends ~64 MB of responses.
4. Assert: no `TooManyRequests` status is returned, no ban is issued, and the node's DB read metrics spike proportionally to the request rate.
5. Repeat with `GetBlockFilterCheckPoints` and `GetBlockFilters` to confirm all three handlers are equally unguarded.
