Audit Report

## Title
Unbounded HTTP body allocation in miner notify handler causes OOM crash — (`miner/src/client.rs`)

## Summary
The miner's notify-mode HTTP server collects the full incoming request body into memory with no size cap. Any process that can reach the configured listen address can POST an arbitrarily large body, exhausting the miner process's heap and triggering an OOM kill. The vulnerability is confirmed by the code and the absence of any body-size guard in the miner codebase.

## Finding Description
When `listen = Some(addr)` is configured, `listen_block_template_notify` (line 234) binds a TCP listener and dispatches every accepted connection to the `handle` async function via `service_fn`. Inside `handle` (lines 358–369), the entire request body is collected unconditionally:

```rust
let body = BodyExt::collect(req).await?.aggregate();
```

The `auto::Builder` setup (line 236) applies no middleware, and a grep for `body_limit`, `max_body`, `content_length`, `size_limit`, `Content-Length`, and `Limited` across the entire `miner/` tree returns zero matches. There is no guard anywhere in the call path. An attacker sends a chunked HTTP/1.1 POST with an arbitrarily large body; `BodyExt::collect` streams all chunks into a growing heap allocation until the OS OOM-killer terminates the process.

## Impact Explanation
The affected process is the `ckb-miner` command-line tool, which is a separate process from the CKB node. Crashing it does not crash the CKB network, a CKB node, or cause consensus deviation. The concrete impact is termination of the miner process. This matches the allowed CKB bounty impact: **"Any local command line crash"** — Note level (0–500 points).

## Likelihood Explanation
The notify listen address is a documented production feature. When bound to `127.0.0.1`, any unprivileged local process can open a TCP connection. No credentials or elevated privileges are required. The attack is trivially repeatable: the miner restarts (if supervised) and can be crashed again immediately.

## Recommendation
Wrap the incoming body with `http_body_util::Limited` before collecting:

```rust
use http_body_util::Limited;
const MAX_BODY: u64 = 4 * 1024 * 1024; // 4 MiB

async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let limited = Limited::new(req, MAX_BODY);
    let body = BodyExt::collect(limited).await?.aggregate();
    // ...
}
```

Alternatively, reject requests whose `Content-Length` header exceeds the limit before reading the body.

## Proof of Concept
```bash
# Start ckb-miner with listen = "127.0.0.1:18114" in miner.toml

s = socket.create_connection(('127.0.0.1', 18114))

    'Transfer-Encoding: chunked\r\nContent-Type: application/json\r\n\r\n'

s.sendall(header)
chunk = ('1000000\r\n' + 'A' * 0x1000000 + '\r\n').encode()
for _ in range(256):
    s.sendall(chunk)
s.sendall(b'0\r\n\r\n')
"
# Observe: miner process RSS grows unboundedly and is OOM-killed.
```
