Audit Report

## Title
`required_flags_filter` COMPATIBILITY Bypass Allows Peer Store Pollution by Unprivileged Peers — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`required_flags_filter` contains a permanent backward-compatibility shortcut that accepts any peer advertising only `Flags::COMPATIBILITY` (0b1) as a valid full-node peer when the default `required == RELAY | DISCOVERY | SYNC`. An unprivileged attacker on the same CKB network can exploit this by sending a crafted `IdentifyMessage` with `flag = 1`, causing the victim to store the attacker's address (and all advertised listen addresses) in the peer store tagged as full-node candidates, and to open all non-Feeler protocols on the outbound session. At scale, this degrades the victim's ability to find legitimate peers and wastes outbound connection slots.

## Finding Description

**Root cause — `required_flags_filter`** (`network/src/peer_store/peer_store_impl.rs`, lines 407–413):

```rust
pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    if required == Flags::RELAY | Flags::DISCOVERY | Flags::SYNC {
        t.contains(required) || t.contains(Flags::COMPATIBILITY)  // bypass
    } else {
        t.contains(required)
    }
}
```

`Flags::COMPATIBILITY = 0b1`, `Flags::RELAY | DISCOVERY | SYNC = 0b1110`. A peer with only `COMPATIBILITY` does not contain `RELAY | DISCOVERY | SYNC`, but the `||` branch makes the function return `true` anyway.

**Attacker-controlled entry — `verify`** (`network/src/protocols/identify/mod.rs`, lines 541–561):

`verify` only checks (a) network name matches and (b) `flag != 0`. `flag = 1` passes both checks and is returned as `Flags::from_bits_truncate(1)` = `Flags::COMPATIBILITY`. No check for SYNC, RELAY, or DISCOVERY is performed.

**Unconditional peer store write — `received_identify`** (`network/src/protocols/identify/mod.rs`, line 422):

```rust
peer_store.add_outbound_addr(context.session.address.clone(), flags);
```

This write happens **before** the `required_flags_filter` check at line 434. The attacker's address is stored regardless of whether the filter would pass.

**Protocol opening** (lines 434–443): Because `required_flags_filter(RELAY|DISCOVERY|SYNC, COMPATIBILITY)` returns `true`, `open_protocols` is called with `TargetProtocol::Filter(|id| id != Feeler)`, opening SYNC, RELAY, DISCOVERY, and Ping on the outbound session with a peer that does not actually support them.

**Listen address propagation — `add_remote_listen_addrs`** (lines 488–494): The attacker's advertised listen addresses are added to the peer store with the same `COMPATIBILITY` flags via `peer_store.add_addr`. These entries have `last_connected_at_ms == 0`, making them immediately eligible for `fetch_nat_addrs`, which also uses `required_flags_filter`.

**Peer store re-read**: `fetch_addrs_to_attempt` (line 208), `fetch_nat_addrs` (line 252), and `fetch_random_addrs` (line 277) all call `required_flags_filter` on stored entries. COMPATIBILITY-flagged entries pass the filter in all three, so the victim continuously dials attacker-controlled addresses as full-node candidates.

**Eviction blindness — `check_purge`** (lines 327–404): Eviction is based on `is_connectable` status and network-group density (>4 peers per /16 subnet), not flag quality. An attacker with diverse IPs across many subnets can fill `ADDR_COUNT_LIMIT` with COMPATIBILITY-only entries without triggering density-based eviction, displacing legitimate SYNC|RELAY|DISCOVERY peers.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A victim node's peer store becomes populated with attacker-controlled COMPATIBILITY-only addresses that are indistinguishable from legitimate full-node addresses in all three fetch paths. The victim wastes outbound connection slots attempting SYNC/RELAY/DISCOVERY with peers that do not support those protocols. At scale (attacker controlling many IPs across diverse /16 subnets), the peer store fills with attacker entries, legitimate peers are evicted, and the victim node becomes unable to maintain functional outbound connections to the real CKB network — effectively causing network-level degradation with minimal per-attacker cost.

## Likelihood Explanation

- Requires only a standard TCP connection to the victim's P2P port — no privileged access, no PoW, no key material.
- The only attacker-controlled inputs are: network name (public, e.g., `"ckb"`) and `flag = 1` in a molecule-encoded `IdentifyMessage`.
- Scalable: a botnet or a node with many IP addresses can flood the peer store. The eviction threshold of >4 peers per /16 subnet means the attacker needs IPs spread across multiple /16 subnets, which is achievable with cloud infrastructure.
- Repeatable: after eviction or disconnection, the attacker can reconnect and re-pollute.

## Recommendation

Remove the `COMPATIBILITY` shortcut from `required_flags_filter`. The correct implementation is:

```rust
pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    t.contains(required)
}
```

If backward compatibility with pre-flag nodes is still required, gate the shortcut on a configurable epoch/block height that has already passed, rather than a permanent code path. Additionally, consider moving `add_outbound_addr` to after the `required_flags_filter` check so that addresses failing the flag check are never written to the peer store.

## Proof of Concept

```rust
// Unit test demonstrating the bypass
use network::peer_store::peer_store_impl::required_flags_filter;

let required = Flags::RELAY | Flags::DISCOVERY | Flags::SYNC; // 0b1110

assert!(!attacker_flags.contains(required));

// But required_flags_filter returns true — bypass confirmed
assert!(required_flags_filter(required, attacker_flags));
```
