### Title
Unauthenticated Notify Endpoint + `id == 0` Deduplication Bypass Allows Continuous Work Injection — (`miner/src/client.rs`)

---

### Summary

The `update_block_template` function contains a logical flaw in its deduplication guard: the condition `id != work_id || id == 0` permanently evaluates to `true` whenever `current_work_id` holds the value `0`. Because any incoming template with `work_id=0` sets `current_work_id` to `0`, every subsequent injection with `work_id=0` also passes the guard. Combined with a completely unauthenticated HTTP notify endpoint, an attacker with network access to the miner's listen port can continuously inject fake `BlockTemplate` payloads with `work_id=0`, redirecting all workers to mine on attacker-controlled (invalid) work indefinitely.

---

### Finding Description

**The deduplication guard — `client.rs` lines 295–301:**

```rust
let updated = |id| {
    if id != work_id || id == 0 {
        Some(work_id)
    } else {
        None
    }
};
```

Here `id` is the **current stored** value of `current_work_id` (an `AtomicU64`), and `work_id` is the incoming template's `work_id`.

The intent is: only accept a new template if its `work_id` differs from the one already being worked on. The `|| id == 0` clause was presumably added to force acceptance of the very first template (when the stored value is the initial `0`). However, it creates a permanent bypass:

| Call | `current_work_id` (id) | incoming `work_id` | `id != work_id` | `id == 0` | Result |
|------|------------------------|-------------------|-----------------|-----------|--------|
| 1st  | 0 (initial)            | 0                 | false           | **true**  | update → stored becomes 0 |
| 2nd  | 0 (just set)           | 0                 | false           | **true**  | update again |
| 3rd  | 0                      | 0                 | false           | **true**  | update again |
| …    | …                      | …                 | …               | …         | … forever |

Because setting `current_work_id` to `0` leaves it at `0`, the `id == 0` branch fires on every subsequent call with `work_id=0`. The deduplication invariant is permanently broken for this value. [1](#0-0) 

**The unauthenticated notify endpoint — `client.rs` lines 358–369:**

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();
    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }
    Ok(Response::new(Empty::new()))
}
```

There is **no authentication, no IP allowlist, no HMAC, no token** — any TCP connection that can reach the bound address and POST valid JSON is accepted and forwarded directly to `update_block_template`. [2](#0-1) 

**Worker effect — `eaglesong_simple.rs` lines 44–51:**

On each accepted `Works::New`, `notify_new_work` sends `WorkerMessage::NewWork` to every worker thread. Workers immediately replace their active `pow_work` with the attacker's payload and begin hashing it:

```rust
WorkerMessage::NewWork { pow_hash, work, target } => {
    self.pow_work = Some((pow_hash, work));
    self.target = target;
}
``` [3](#0-2) 

Workers do not stop — they keep running but now hash the attacker-supplied `pow_hash` against the attacker-supplied `target`. Any nonce found is submitted to the node, which rejects it (invalid block), wasting the entire mining cycle. [4](#0-3) 

---

### Impact Explanation

- All worker threads are continuously redirected to mine on attacker-controlled, invalid block templates.
- Every nonce found is submitted and rejected by the node; `nonces_found` in `miner.rs` never increments for valid blocks.
- The operator's node never produces a valid block while the attack is ongoing, effectively halting block production for that miner.
- The attack requires only the ability to send HTTP POST requests to the miner's notify port — no credentials, no keys, no privileged access. [5](#0-4) 

---

### Likelihood Explanation

- Notify mode is opt-in (`config.listen`), but operators who enable it (as documented in the log message at line 216) are fully exposed.
- The listen address defaults to a `SocketAddr` — if bound to `0.0.0.0` or any externally reachable interface, the attack is trivially executable from the internet.
- No exploit tooling is needed: a simple `curl` loop suffices. [6](#0-5) 

---

### Recommendation

1. **Fix the deduplication logic.** The correct guard is: accept the template only if its `work_id` differs from the currently stored one. The `|| id == 0` clause should be removed or replaced with a check on the **incoming** `work_id` (e.g., treat `work_id=0` as always-stale rather than always-fresh):

   ```rust
   // Before (buggy):
   if id != work_id || id == 0 { Some(work_id) } else { None }

   // After (correct):
   if id != work_id { Some(work_id) } else { None }
   ``` [7](#0-6) 

2. **Add authentication to the notify endpoint.** At minimum, require a shared secret token in the request header, or restrict accepted connections to `127.0.0.1` by default. [2](#0-1)

### Proof of Concept

```bash
# Craft a minimal valid BlockTemplate JSON with work_id=0
TEMPLATE='{"version":"0x0","compact_target":"0x1a08a97e","current_time":"0x...","number":"0x1","epoch":"0x...","parent_hash":"0x...","cycles_limit":"0x...","bytes_limit":"0x...","uncles_count_limit":"0x2","uncles":[],"transactions":[],"proposals":[],"cellbase":{"hash":"0x...","cycles":null,"min_fee_rate":"0x0","time_added_to_pool":null,"transaction":{"cell_deps":[],"header_deps":[],"inputs":[{"previous_output":{"index":"0xffffffff","tx_hash":"0x0000..."},"since":"0x0"}],"outputs":[],"outputs_data":[],"version":"0x0","witnesses":["0x..."]}},"work_id":"0x0","dao":"0x..."}'

# Flood the notify endpoint
while true; do
  curl -s -X POST http://<miner-notify-addr> \
       -H 'Content-Type: application/json' \
       -d "$TEMPLATE"
done
```

Each POST bypasses the `id == 0` deduplication guard, sends `Works::New` to the miner loop, and redirects all worker threads to hash the attacker's fake template. The operator's `nonces_found` counter remains at zero for valid blocks for the duration of the attack. [1](#0-0)
