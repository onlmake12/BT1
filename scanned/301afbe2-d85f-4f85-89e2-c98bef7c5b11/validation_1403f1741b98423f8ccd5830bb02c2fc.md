### Title
Size-Only Linear Approximation in Minimum Fee Check Allows Cycle-Heavy Transactions to Bypass `min_fee_rate` Enforcement — (`tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` enforces the minimum fee rate by computing the required fee using only the transaction's serialized byte size. However, the actual transaction weight used everywhere else in the pool is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For a cycle-heavy transaction whose cycles dominate, the minimum fee check uses a value that can be an order of magnitude smaller than the true weight, allowing the transaction to enter the pool with an effective fee rate far below the configured `min_fee_rate`. The same linear approximation error also appears in the RBF minimum-replace-fee calculation in `tx-pool/src/pool.rs`.

---

### Finding Description

CKB's transaction weight is defined as:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` is derived from `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`. [2](#0-1) 

The minimum fee check in `check_tx_fee`, however, uses only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [3](#0-2) 

The code comment itself acknowledges the approximation is theoretically incorrect. The same pattern appears in `calculate_min_replace_fee` for RBF:

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [4](#0-3) 

Everywhere else in the pool — scoring, eviction, fee rate reporting — the correct `get_transaction_weight` is used: [5](#0-4) 

This creates a systematic discrepancy: the gate check uses `size`, but the pool's internal accounting uses `max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`.

---

### Impact Explanation

Consider a transaction with:
- Serialized size: **1,000 bytes**
- Cycles: **70,000,000** (the default `max_tx_verify_cycles`)

Actual weight = `max(1000, 70_000_000 × 0.000_170_571_4)` = `max(1000, 11,940)` = **11,940 bytes**

With the default `min_fee_rate = 1,000 shannons/KW`:

| Check | Formula | Required fee |
|---|---|---|
| Gate check (`check_tx_fee`) | `1000 × 1000 / 1000` | **1,000 shannons** |
| Actual weight-based minimum | `11940 × 1000 / 1000` | **11,940 shannons** |

A transaction paying exactly 1,000 shannons passes the gate but has an effective fee rate of `1000 × 1000 / 11940 ≈ 84 shannons/KW` — roughly **8.4% of the configured minimum**. The same bypass applies to the RBF `min_rbf_rate` check.

Consequences:
1. Cycle-heavy transactions enter the pool with effective fee rates far below `min_fee_rate`, undermining the fee market.
2. The pool's fee estimator and eviction logic operate on the correct weight, so these transactions are correctly ranked low — but they have already been admitted.
3. An attacker can flood the mempool with cycle-heavy, low-fee-rate transactions that consume block cycle budget without paying the corresponding fee.
4. RBF replacement transactions can be submitted with insufficient fee bumps, violating the `min_rbf_rate` invariant.

---

### Likelihood Explanation

Any unprivileged RPC caller (`send_transaction`) or P2P relay peer can submit transactions. Crafting a cycle-heavy transaction requires writing a CKB-VM lock or type script that executes a tight loop — a straightforward task for any script author. The `max_tx_verify_cycles` cap (70M by default) bounds the maximum amplification factor to roughly 12× at default settings. This is a realistic, low-effort attack requiring no special privileges. [6](#0-5) 

---

### Recommendation

Replace the size-only approximation in `check_tx_fee` with the actual weight calculation. Since cycles are not yet known at the pre-verification stage, the check should be deferred to after cycle counting, or the declared cycles (which are validated against actual cycles) should be used:

```rust
// After cycles are known (post-verification):
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

Similarly, `calculate_min_replace_fee` should use `get_transaction_weight(entry.size, entry.cycles)` instead of `entry.size` alone, since cycles are already known for pool entries. [7](#0-6) 

---

### Proof of Concept

1. Construct a CKB transaction whose lock script is a tight RISC-V loop consuming ~70,000,000 cycles but whose serialized size is ~1,000 bytes.
2. Set the transaction fee to exactly `min_fee_rate * tx_size / 1000` shannons (e.g., 1,000 shannons at default settings).
3. Submit via `send_transaction` RPC.
4. Observe: `check_tx_fee` passes (fee ≥ `min_fee_rate.fee(tx_size)`).
5. Observe: the transaction's actual effective fee rate in the pool is `fee * 1000 / get_transaction_weight(size, cycles)` ≈ 84 shannons/KW, far below the 1,000 shannons/KW minimum.
6. Repeat to fill the pool with cycle-heavy, below-minimum-fee-rate transactions.
