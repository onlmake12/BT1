### Title
Redundant Double-Iteration in `VerifyQueue::peek` Wastes CPU on Every Transaction Dequeue — (`File: tx-pool/src/component/verify_queue.rs`)

### Summary
The `VerifyQueue::peek` function, called on every transaction dequeue in the tx-pool admission path, unconditionally performs two full linear scans of the queue instead of one. Any unprivileged peer who submits transactions via P2P relay or RPC can fill the verify queue, causing each dequeue to do 2× the necessary work.

### Finding Description

`VerifyQueue::peek` is structured as follows: [1](#0-0) 

```rust
pub fn peek(&self, only_small_cycle: bool) -> Option<ProposalShortId> {
    let mut iter = self.inner.iter_by_added_time();

    if let Some(proposal_entry) = iter.find(|e| e.is_proposal_tx) {
        return Some(proposal_entry.inner.tx.proposal_short_id());
    }

    // ← iter is dropped here; a brand-new iterator is created below
    let entry = if only_small_cycle {
        self.inner.iter_by_added_time().find(|e| !e.is_large_cycle)
    } else {
        self.inner.iter_by_added_time().next()
    };

    entry.map(|e| e.inner.tx.proposal_short_id())
}
```

When no proposal transaction is present (the common case), the first `iter.find(…)` exhausts the entire ordered index to confirm the absence of a proposal tx. Then a **second, independent** `iter_by_added_time()` iterator is created from the beginning to locate the first small-cycle (or any) entry. The two goals — "is there a proposal tx?" and "what is the first eligible non-proposal tx?" — could be resolved in a single pass.

`peek` is called by `pop_front` on every dequeue: [2](#0-1) 

The verify queue accepts up to 256 MB of serialized transaction data: [3](#0-2) 

The `VerifyEntry` struct uses a `MultiIndexMap` with an `ordered_non_unique` index on `added_time`, meaning `iter_by_added_time()` is a B-tree traversal: [4](#0-3) 

### Impact Explanation

Every transaction dequeue from the verify queue performs O(2n) work instead of O(n). Under a full queue (up to 256 MB of transactions, each ~200–500 bytes → tens of thousands of entries), the CPU cost per dequeue is doubled. Because the tx-pool worker continuously calls `pop_front` to drain the queue and verify transactions, this sustained overhead slows the rate at which transactions are admitted to the pool. Slower admission means delayed block template generation and reduced throughput for all users.

### Likelihood Explanation

Any unprivileged peer can submit transactions via the P2P relay protocol or the `send_transaction` RPC endpoint, filling the verify queue to its limit. The double-iteration occurs on every single dequeue regardless of queue contents — it is not a corner case. The condition is trivially reachable in normal operation and requires no special privileges.

### Recommendation

Combine both searches into a single pass over `iter_by_added_time()`. Track the first eligible non-proposal entry while scanning for a proposal entry:

```rust
pub fn peek(&self, only_small_cycle: bool) -> Option<ProposalShortId> {
    let mut first_eligible: Option<ProposalShortId> = None;
    for entry in self.inner.iter_by_added_time() {
        if entry.is_proposal_tx {
            return Some(entry.inner.tx.proposal_short_id());
        }
        if first_eligible.is_none() {
            if !only_small_cycle || !entry.is_large_cycle {
                first_eligible = Some(entry.inner.tx.proposal_short_id());
            }
        }
    }
    first_eligible
}
```

This achieves the same priority semantics (proposal txs first, then small-cycle txs if `only_small_cycle`) in a single O(n) pass.

### Proof of Concept

1. Fill the verify queue with `N` non-proposal, large-cycle transactions (each submitted via `send_transaction` RPC or P2P relay).
2. Call `pop_front(false)` (the normal dequeue path used by the tx-pool worker).
3. Observe that `peek` first scans all `N` entries looking for a proposal tx (finding none), then scans again from entry 0 to return the first entry — 2N comparisons instead of N.

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
