Audit Report

## Title
Tx-Pool Min-Fee-Rate Admission Check Uses Only Serialized Size, Allowing Cycle-Heavy Transactions to Bypass the Minimum Fee Rate Guard - (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size, not the actual block weight (`max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`). Because cycles are unknown at admission time and no second check is performed after verification, a transaction with small serialized size but near-maximum cycle consumption is admitted to the pool with an effective fee rate up to ~40× below the configured `min_fee_rate`. This allows an unprivileged attacker to flood the network with computationally expensive transactions at a fraction of the intended economic cost.

## Finding Description

`check_tx_fee` (`tx-pool/src/util.rs`, lines 42–52) computes the minimum required fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(reject);
}
```

This function is called inside `pre_check` (`tx-pool/src/process.rs`, line 289) before script execution, so `cycles` are not yet known. After `verify_rtx` returns `verified.cycles` (line 734), the code immediately constructs `TxEntry::new(rtx, verified.cycles, fee, tx_size)` at line 751 and calls `submit_entry` at line 753 — with no second fee-rate check against the actual weight.

The actual weight used everywhere else in the system is defined in `util/types/src/core/tx_pool.rs` (lines 298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

`TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs`, lines 115–118) uses this correct weight for pool sorting and eviction, but it is never consulted at admission time. The `EvictKey` and `AncestorsScoreSortKey` also use the correct weight-based fee rate for internal ordering, but only after the transaction is already in the pool.

The code comment itself acknowledges the discrepancy: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check."* No compensating check exists after cycles are known.

## Impact Explanation

An attacker submitting via the standard `send_transaction` RPC can craft a transaction with ~300 bytes serialized size and ~70,000,000 cycles (the `max_tx_verify_cycles` default). The admission check computes `min_fee = 1000 * 300 / 1000 = 300 shannons`; the transaction passes. After verification, the actual weight is `max(300, 70_000_000 × 0.000_170_571_4) ≈ 11,940`. The effective fee rate is `300 × 1000 / 11,940 ≈ 25 shannons/KW` — approximately 40× below the configured `min_fee_rate` of 1,000 shannons/KW.

Each such transaction forces every receiving node to execute 70,000,000 VM cycles of verification. Because the economic barrier is reduced ~40×, an attacker can sustain a much larger volume of high-cycle transactions than the `min_fee_rate` was designed to permit, causing CPU saturation across the network. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

- **Entry path**: The standard public `send_transaction` RPC — no special privileges required.
- **Ease**: Constructing a small-serialized transaction whose lock script runs a tight computation loop consuming ~70M cycles is straightforward.
- **Cost**: The attacker pays ~25 shannons/KW effective fee rate instead of the intended 1,000 shannons/KW — a ~40× reduction in the economic barrier to pool spam.
- **Repeatability**: The attacker can continuously submit new transactions as long as they hold UTXOs; the pool's `limit_size` eviction (which correctly uses weight-based fee rate) only helps after the pool is already full and the verification CPU cost has already been paid by all nodes.
- **Confirmed by code**: The discrepancy is explicitly acknowledged in the source comment, confirming this is a known approximation with no compensating guard.

## Recommendation

After `verify_rtx` completes and `verified.cycles` is known, perform a second fee-rate check using the actual transaction weight before calling `submit_entry`:

```rust
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let actual_fee_rate = entry.fee_rate(); // uses get_transaction_weight(size, cycles)
if actual_fee_rate < tx_pool.config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        tx_pool.config.min_fee_rate.fee(get_transaction_weight(tx_size, verified.cycles)).as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

This mirrors the pattern already used in `TxEntry::fee_rate()` and `get_transaction_weight`, ensuring the admission gate is consistent with the weight model used for block packing and pool eviction.

## Proof of Concept

1. Construct a CKB transaction with a lock script that runs a tight computation loop consuming ~70,000,000 cycles, minimal witnesses and outputs so serialized size is ~300 bytes, and fee = 300 shannons (`min_fee_rate * tx_size / 1000 = 1000 * 300 / 1000`).
2. Submit via `send_transaction` RPC.
3. `check_tx_fee` computes `min_fee = 300 shannons`. Fee (300) ≥ min_fee (300) → admitted.
4. `verify_rtx` executes ~70,000,000 VM cycles.
5. No second fee-rate check is performed. `TxEntry` is created with `cycles ≈ 70_000_000`.
6. Actual weight = `max(300, 70_000_000 × 0.000_170_571_4) ≈ 11,940`. Actual fee rate ≈ 25 shannons/KW — 40× below `min_fee_rate`.
7. Repeat with many UTXOs. Each submission forces all receiving nodes to spend 70M cycles of verification at 1/40th the intended economic cost, causing network-wide CPU congestion.
