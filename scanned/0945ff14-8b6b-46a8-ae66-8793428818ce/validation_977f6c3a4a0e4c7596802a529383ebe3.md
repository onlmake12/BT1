Audit Report

## Title
Unbounded RocksDB Iteration in `get_cells_capacity` Enables RPC Thread Pool Exhaustion — (File: `util/indexer/src/service.rs`)

## Summary
`IndexerHandle::get_cells_capacity` iterates over all matching RocksDB cells with no count-based limit, bounded only by a 10-second `TimeoutIterator`. Unlike `get_cells` and `get_transactions`, which enforce a `request_limit` guard and a `.take(limit)` early-stop, `get_cells_capacity` accepts unbounded broad-prefix queries from any unprivileged RPC caller. Concurrent requests each hold a worker thread for the full timeout duration, exhausting the thread pool and rendering the node's RPC unresponsive.

## Finding Description
`IndexerHandle::get_cells_capacity` (defined at `util/indexer/src/service.rs:686`) builds a RocksDB prefix iterator and folds over every matching cell with no count guard:

```rust
// service.rs:720 — sole protection is a timeout wrapper
let mut iter = TimeoutIterator::new(snapshot.iterator(mode).skip(skip), self.timeout_limit);

let capacity: u64 = iter
    .by_ref()
    .take_while(|(key, _value)| key.starts_with(&prefix))
    .filter_map(|(key, value)| { ... })
    .sum();  // line 836 — no .take(N) anywhere in this chain
```

By contrast, `get_cells` enforces both a `request_limit` rejection (lines 212–221) and a `.take(limit)` early-stop (line 370 area). `get_transactions` has the identical `request_limit` guard (lines 388–397). Neither protection exists in `get_cells_capacity`.

The `TimeoutIterator` checks elapsed time only at the top of each `next()` call (lines 55–61). A single expensive RocksDB read inside the loop body can still exceed the timeout boundary before the check fires again — the guard is cooperative, not preemptive.

`request_limit` defaults to `usize::MAX` when unconfigured (line 98), and `timeout_limit` defaults to 10 seconds (line 99). The RPC trait exposes `get_cells_capacity` with no `limit` parameter (lines 879–883), and the implementation passes the call directly to `IndexerHandle::get_cells_capacity` with no additional guard (lines 929–936).

## Impact Explanation
Each concurrent `get_cells_capacity` request with a broad script prefix (e.g., empty `args`) scans every matching live cell in the indexer store, holding a worker thread for up to 10 seconds. On a mainnet node with millions of live cells, 20–50 concurrent requests saturate the thread pool, causing all subsequent RPC calls — including `send_transaction` and `get_block_template` — to queue indefinitely. This maps to **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points), as the node becomes operationally non-functional for its operator and any dependent services.

## Likelihood Explanation
`get_cells_capacity` is part of the standard, publicly documented CKB indexer API. No authentication, signature, or privileged key is required — any process that can reach the RPC port can issue the attack. The default configuration (`request_limit` absent, `timeout_limit = 10s`) maximises exposure. The attack requires only repeated JSON-RPC HTTP POST requests, reproducible with any HTTP client. Nodes exposing the RPC to a local network or (misconfigured) to the public internet are directly reachable.

## Recommendation
1. Add the same `request_limit` guard present in `get_cells` and `get_transactions` to `get_cells_capacity`, rejecting requests that would scan more than the configured limit.
2. Alternatively, add an internal `.take(self.request_limit)` to the iterator chain in `get_cells_capacity` so the scan stops after at most `request_limit` cells, returning a partial result or an explicit error.
3. Lower the default `timeout_limit` and document that `request_limit` must be set for any publicly or semi-publicly reachable node.

## Proof of Concept
Send N concurrent HTTP POST requests to the CKB RPC port:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [{
    "script": {
      "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
      "hash_type": "type",
      "args": "0x"
    },
    "script_type": "lock"
  }]
}
```

Empty `args` prefix-matches every cell whose lock script uses this `code_hash`, covering the majority of mainnet live cells. Each request iterates the full matching set (bounded only by the 10-second timeout). Sending 20–50 concurrent requests saturates the thread pool; subsequent `send_transaction` and `get_block_template` calls queue indefinitely, confirming node RPC unresponsiveness.
