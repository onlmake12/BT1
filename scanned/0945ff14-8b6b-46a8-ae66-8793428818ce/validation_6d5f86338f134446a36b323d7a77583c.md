Audit Report

## Title
Unbounded Memory Allocation via Indexer RPC `get_cells`/`get_transactions` When `request_limit` Is Unset — (`util/indexer/src/service.rs`)

## Summary
When `IndexerConfig.request_limit` is `None` (the default), `IndexerService::new` stores `usize::MAX` as the internal limit. The guard in `get_cells` and `get_transactions` compares the caller-supplied `u32` limit (cast to `usize`) against `usize::MAX`, which always passes on 64-bit platforms. The only real bound is a 10-second `TimeoutIterator`, within which millions of RocksDB records can be deserialized and heap-allocated into a single `Vec`, enabling memory exhaustion or OOM-killer invocation on a populated node.

## Finding Description

**Step 1 — Default configuration leaves `request_limit` as `None`:** [1](#0-0) [2](#0-1) 

**Step 2 — `IndexerService::new` converts `None` to `usize::MAX`:** [3](#0-2) 

**Step 3 — The limit guard is trivially bypassed:**

```rust
let limit = limit.value() as usize;   // u32::MAX → 4_294_967_295usize
if limit == 0 { return Err(...) }
if limit > self.request_limit { ... } // 4_294_967_295 > usize::MAX → false, passes
``` [4](#0-3) 

The same pattern is present in `get_transactions`: [5](#0-4) 

**Step 4 — The iterator is bounded only by a 10-second wall-clock timeout, not by memory:** [6](#0-5) [7](#0-6) 

**Step 5 — All matching records within the timeout window are materialized into a `Vec`:** [8](#0-7) 

`RichIndexerService` has the identical default: [9](#0-8) 

## Impact Explanation

An unprivileged caller with access to the RPC port can send repeated `get_cells` or `get_transactions` requests with `limit = 0xFFFFFFFF`. Each request causes the node to open a RocksDB snapshot iterator, scan and deserialize as many records as possible within 10 seconds, and allocate all results into a single `Vec` before returning. On a fast NVMe-backed node with a large indexer store, millions of `IndexerCell` records (each containing output data, scripts, capacity, etc.) can be read and heap-allocated within the 10-second window. Multiple concurrent such requests compound the allocation. The result is sustained high RSS growth, potential OOM-killer invocation, or severe performance degradation making the node unresponsive to legitimate traffic.

This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

- The vulnerable default (`request_limit = None` → `usize::MAX`) requires no special operator misconfiguration — it is the out-of-the-box state.
- The RPC endpoint is a standard, documented attack surface; many operators expose it publicly or it is reachable by any local process.
- The exploit requires only a single JSON-RPC call with a large `limit` field.
- The `TimeoutIterator` mitigates the absolute worst case (full store scan) but does not prevent large allocations within the 10-second window on a populated store.
- Concurrent requests amplify the effect multiplicatively.

## Recommendation

1. Change the default for `request_limit` from `None`/`usize::MAX` to a safe bounded value (e.g., `500` or `1000`), matching what the RPC documentation implies as a reasonable page size.
2. Alternatively, enforce a hard cap independent of configuration:
   ```rust
   const MAX_INDEXER_LIMIT: usize = 1000;
   let effective_limit = limit.min(self.request_limit).min(MAX_INDEXER_LIMIT);
   ```
3. Apply the same fix to `RichIndexerService` and `AsyncRichIndexerHandle`.
