Audit Report

## Title
`migrate_transaction_info` Traverses `COLUMN_UNCLES` Instead of `COLUMN_TRANSACTION_INFO`, Leaving Transaction Index Uncorrected After `ChangeMoleculeTableToStruct` Migration — (`File: util/migrate/src/migrations/table_to_struct.rs`)

## Summary

In `util/migrate/src/migrations/table_to_struct.rs`, the `migrate_transaction_info` function inside `ChangeMoleculeTableToStruct` contains a copy-paste error: it calls `db.traverse(COLUMN_UNCLES, ...)` on line 93 instead of `db.traverse(COLUMN_TRANSACTION_INFO, ...)`. As a result, the migration reads uncle entries a second time, writes uncle-derived garbage into `COLUMN_TRANSACTION_INFO` under uncle-block-hash keys, and never touches the actual transaction-hash-keyed entries in `COLUMN_TRANSACTION_INFO`. After the migration completes with `Ok(db)`, the transaction index is left in the old Molecule table format and is additionally polluted with malformed uncle data.

## Finding Description

The four sub-functions of `ChangeMoleculeTableToStruct` are each responsible for reformatting one column. Three are correct:

- `migrate_header` (line 39–40): traverses `COLUMN_BLOCK_HEADER`, writes to `COLUMN_BLOCK_HEADER`
- `migrate_uncles` (line 66): traverses `COLUMN_UNCLES`, writes to `COLUMN_UNCLES`
- `migrate_epoch_ext` (line 120): traverses `COLUMN_EPOCH`, writes to `COLUMN_EPOCH`

`migrate_transaction_info` (lines 77–102) defines a closure `transaction_info_migration` that writes to `COLUMN_TRANSACTION_INFO` (line 85), but the `db.traverse` call on line 93 passes `COLUMN_UNCLES` as the source column. The constant `COLUMN_TRANSACTION_INFO` imported at line 4 is never passed to any `traverse` call anywhere in the file.

Execution flow:
1. Node operator upgrades binary; stored DB version is older than `20200703124523`.
2. `ChangeMoleculeTableToStruct::migrate` is called automatically.
3. `migrate_uncles` correctly reformats `COLUMN_UNCLES`.
4. `migrate_transaction_info` re-reads `COLUMN_UNCLES` (already in new format, so `value.len() == 240 != 52`), strips 16 bytes from each uncle entry, and writes the result into `COLUMN_TRANSACTION_INFO` keyed by uncle block hashes.
5. The real `COLUMN_TRANSACTION_INFO` entries (keyed by tx hash, mapping each tx to its block) are never read and remain in old Molecule table format.
6. `migrate` returns `Ok(db)` with no error indication.

After migration, `COLUMN_TRANSACTION_INFO` contains: (a) all original tx-hash-keyed entries still in old table format, and (b) uncle-hash-keyed entries containing truncated uncle header data. Any call to `get_transaction_info` for a real transaction hash returns data in the old format, which `packed::TransactionInfoReader::from_slice_should_be_ok` will misparse, or returns `None` if the hash happens not to collide.

## Impact Explanation

`COLUMN_TRANSACTION_INFO` is the sole index mapping transaction hashes to block locations (`block_hash`, `block_number`, `tx_index`). Corrupting it breaks the CKB state storage mechanism for transaction lookup. `get_transaction_info`, `get_transaction`, `get_transaction_with_info`, and `transaction_exists` all read exclusively from this column. After migration, historical transaction lookups return wrong or absent data. This matches the allowed impact: **Medium (2001–10000 points) — Suboptimal/incorrect implementation of CKB state storage mechanism**, specifically the transaction index component of the chain store.

## Likelihood Explanation

The migration is triggered automatically on any node whose stored DB version predates `20200703124523` (v0.35.0, released 2020). The trigger conditions are: (1) a pre-v0.35.0 database exists on disk, and (2) a post-v0.35.0 binary is run against it. This applies to any operator who restores an archival backup from that era or upgrades a long-dormant node. No attacker action is required; the corruption is self-inflicted by the upgrade process. Likelihood is low in practice (most nodes migrated years ago) but the bug remains present in the production codebase and will silently corrupt any node that encounters this upgrade path.

## Recommendation

Change line 93 in `util/migrate/src/migrations/table_to_struct.rs`:

```diff
- db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
+ db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```

Additionally, add a regression test that pre-populates `COLUMN_UNCLES` and `COLUMN_TRANSACTION_INFO` with old-format (table-encoded) entries, runs `ChangeMoleculeTableToStruct::migrate`, and asserts: (a) every `COLUMN_TRANSACTION_INFO` entry is exactly `TRANSACTION_INFO_SIZE` (52) bytes, and (b) no uncle-block-hash-keyed entry appears in `COLUMN_TRANSACTION_INFO`.
