### Title

### Summary

A peer relaying a `Data2` hash-type transaction to a node where RFC-0049 is not yet active will be incorrectly banned for 3 days. The transaction passes `NonContextualTransactionVerifier` (because `Data2` is in `ENABLED_SCRIPT_HASH_TYPE`), then fails contextual verification with `ScriptError::InvalidVmVersion(2)`. This Script-kind error is unconditionally classified as "malformed" by `is_malformed_from_verification`, triggering `ban_malformed` in `after_process`. The invariant that peer banning is reserved for structurally malformed transactions is violated.

### Finding Description

**Step 1 â€” Non-contextual check passes for `Data2`.**

`ENABLED_SCRIPT_HASH_TYPE` explicitly includes `4u8` (`Data2`): [1](#0-0) 

`ScriptHashTypeVerifier::verify()` checks only against this set, with no reference to RFC-0049 activation epoch: [2](#0-1) 

So `NonContextualTransactionVerifier` passes for any `Data2` transaction, regardless of hardfork state.

**Step 2 â€” Contextual verification fails with `InvalidVmVersion(2)` before RFC-0049.**

`TxInfo::select_version()` checks `is_vm_version_2_and_syscalls_3_enabled()` (which reads the epoch-gated RFC-0049 flag). Before activation it returns `Err(ScriptError::InvalidVmVersion(2))`: [3](#0-2) 

`is_vm_version_2_and_syscalls_3_enabled` uses `epoch_number_without_proposal_window()` against the hardfork switch: [4](#0-3) 

**Step 3 â€” `InvalidVmVersion(2)` is classified as malformed.**

`ScriptError::InvalidVmVersion(2)` is converted to `ErrorKind::Script` (not `ErrorKind::Internal`) via the `From<TransactionScriptError> for Error` impl: [5](#0-4) 

`is_malformed_from_verification` treats **all** `ErrorKind::Script` errors as malformed, with the sole exception of errors whose formatted string contains `ARGV_TOO_LONG_TEXT`: [6](#0-5) 

`ScriptError::InvalidVmVersion(2)` formats as `"Invalid VM Version: 2"`, which does not contain `ARGV_TOO_LONG_TEXT`. Therefore `is_malformed_tx()` returns `true`. [7](#0-6) 

This is confirmed by the existing test: any `ErrorKind::Script` error (except ARGV_TOO_LONG) asserts `is_malformed_tx() == true`: [8](#0-7) 

**Step 4 â€” Peer is banned in `after_process`.**

When contextual verification returns `Reject::Verification(InvalidVmVersion(2))` for a remote-sourced transaction, `after_process` calls `ban_malformed`, issuing a 3-day ban: [9](#0-8) [10](#0-9) 

**Correction to the question's framing:** The banning does NOT happen in `non_contextual_verify`. That function passes for `Data2` (since it's in `ENABLED_SCRIPT_HASH_TYPE`). The banning happens in `after_process` after contextual verification fails. The question's description of the banning location is wrong, but the core vulnerability is real.

### Impact Explanation

A legitimate relay peer that forwards a `Data2` transaction (e.g., one it received from a user testing pre-hardfork compatibility) to a node where RFC-0049 is not yet active will be banned for 72 hours. This can cause network fragmentation during the pre-hardfork window: nodes will ban peers that relay `Data2` transactions, even though those transactions are only invalid due to epoch timing, not structural malformation.

### Likelihood Explanation

This is exploitable by any unprivileged P2P peer during the pre-RFC-0049 epoch window. An attacker can deliberately relay `Data2` transactions to cause victim nodes to ban legitimate relay peers, or to get themselves banned (denial of self-service). The more impactful scenario is a relay node being banned by multiple peers simultaneously, degrading network connectivity.

### Recommendation

`is_malformed_from_verification` must not treat `ScriptError::InvalidVmVersion` as malformed. This error is epoch-dependent (hardfork timing), not structural. Either:
1. Add `InvalidVmVersion` to the ARGV_TOO_LONG-style exclusion list in `is_malformed_from_verification`, or
2. Introduce a finer-grained `ScriptError` classification that distinguishes timing-dependent invalidity from structural malformation.

    match error.kind() {
        ErrorKind::Transaction => error
            .downcast_ref::<TransactionError>()
            .expect("error kind checked")
            .is_malformed_tx(),
