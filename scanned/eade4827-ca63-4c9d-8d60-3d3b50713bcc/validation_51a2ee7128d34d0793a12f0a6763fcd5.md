### Title
Relay Batch Size Constant Reused as Per-Peer Unknown TX Hash DoS Limit — (File: util/constant/src/sync.rs)

### Summary

`MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` is defined as an alias of `MAX_RELAY_TXS_NUM_PER_BATCH` in `util/constant/src/sync.rs`. These two constants serve fundamentally different purposes: one is a P2P protocol message size limit, the other is a per-peer DoS protection and banning threshold. Changing `MAX_RELAY_TXS_NUM_PER_BATCH` for protocol reasons silently changes the per-peer unknown-tx-hash tracking limit, potentially weakening DoS protection or causing over-aggressive peer banning.

### Finding Description

The value 32767 was selected to bound relay batch messages, not to calibrate the per-peer unknown-tx-hash memory budget or the banning threshold. These are distinct concerns with different optimal values.

### Impact Explanation

**If `MAX_RELAY_TXS_NUM_PER_BATCH` is increased** (e.g., to allow larger relay batches for throughput), `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` silently increases by the same factor. A malicious peer can then advertise proportionally more unknown tx hashes before triggering the `TooManyUnknownTransactions` ban, expanding the per-peer memory footprint and weakening the DoS guard in `add_ask_for_txs`.

**If `MAX_RELAY_TXS_NUM_PER_BATCH` is decreased** (e.g., to reduce message size), `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` decreases equally, causing honest peers that legitimately announce many new transactions to be banned prematurely, disrupting relay connectivity.

In both cases the global cap `self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` also shifts, compounding the effect across all connected peers.

### Likelihood Explanation

The coupling is invisible at the definition site: `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` is defined as a direct alias of `MAX_RELAY_TXS_NUM_PER_BATCH` with no comment explaining the relationship. Any future protocol change that adjusts the relay batch size will silently alter the DoS protection threshold. The test in `test/src/specs/relay/too_many_unknown_transactions.rs` line 34 (`assert!(MAX_RELAY_TXS_NUM_PER_BATCH >= tx_hashes.len())`) encodes this coupling as an assumption, making it easy to miss during review.

### Recommendation

Define `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` as an independent constant with a value chosen specifically for the per-peer unknown-tx-hash tracking budget and banning threshold, separate from the relay batch message size limit. Add explicit documentation to both constants explaining their distinct purposes so future changes to one do not inadvertently affect the other.

```rust
/// The maximum number of tx hashes inside a single `RelayTransactionHashes` message
pub const MAX_RELAY_TXS_NUM_PER_BATCH: usize = 32767;

/// Per-peer soft limit on tracked unknown tx hashes before banning (DoS protection)
/// NOTE: intentionally independent of MAX_RELAY_TXS_NUM_PER_BATCH
pub const MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER: usize = 32767; // tune separately
```
