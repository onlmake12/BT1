### Title
DAO Withdrawal `header_dep_index` Interpretation Discrepancy Between Rust `DaoCalculator` and On-Chain C VM Corrupts DAO Accounting Field - (File: `util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator` in `util/dao/src/lib.rs` reads the `header_dep_index` from the DAO withdrawal witness as a full 8-byte `u64`, while the on-chain C VM (`dao.c`) reads only the lowest byte (`uint8_t`). An unprivileged transaction sender can craft a DAO phase-2 withdrawal transaction that exploits this discrepancy — the C VM resolves the correct canonical deposit header (via the lowest byte), while the Rust calculator resolves a different block at the same height (e.g., an uncle block) via the full `u64` index. This causes the Rust node to compute a wrong `withdrawed_interests` value, permanently corrupting the DAO field in the block header.

---

### Finding Description

**Root cause — production code:**

In `transaction_maximum_withdraw`, the witness `input_type` field is decoded as a full little-endian `u64` and used directly as the index into `header_deps`: [1](#0-0) 

```rust
// dao contract stores header deps index as u64 in the input_type field of WitnessArgs
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.get(header_dep_index as usize)
```

The on-chain C VM (`dao.c`) reads the same field as a single byte (`uint8_t`). For any index value `> 255`, the two implementations resolve **different entries** in `header_deps`.

**The only guard in Rust is a block-number equality check:** [2](#0-1) 

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
```

This check catches the mismatch only when the block resolved by the full `u64` index has a **different block number** than the deposited block number stored in cell data. It does **not** catch the case where a second block at the **same height** (e.g., an uncle block) is placed at the full-`u64` position.

**The discrepancy is explicitly documented in the test suite:** [3](#0-2) 

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

The test verifies Rust rejects the case where the full-`u64` block has a **different** block number. It does not test the uncle-block case where the block numbers match.
