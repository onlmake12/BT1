### Title
Unbounded `vec![0u64; max_bucket_index + 1]` allocation in `do_estimate` via attacker-controlled fee rate — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

`max_bucket_index_by_fee_rate` has no upper bound on its return value. When a transaction with an abnormally high fee rate is in the mempool, `do_estimate` allocates two `Vec<u64>` of size `max_bucket_index + 1`, which grows linearly and without limit with the fee rate. An unprivileged attacker who owns enough CKB to burn as fee can force arbitrarily large heap allocations on every `estimate_fee_rate` RPC call, causing OOM crashes or severe memory pressure.

---

### Finding Description

**Root cause — `max_bucket_index_by_fee_rate` is unbounded:**

```rust
fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
    let t = FEE_RATE_UNIT; // 1000
    let index = match fee_rate.as_u64() {
        // ...
        x => (x + t * 11_500) / (100 * t),  // no ceiling
    };
    index as usize
}
``` [1](#0-0) 

For any `fee_rate > 2_000_000`, the index grows as `fee_rate / 100_000`. There is no cap.

**Allocation site — two unbounded `Vec` allocations in `do_estimate`:**

```rust
let mut buckets = vec![0u64; max_bucket_index + 1];  // line 219
// ...
let mut buckets = vec![0u64; max_bucket_index + 1];  // line 248
``` [2](#0-1) [3](#0-2) 

**`FeeRate::calculate` can produce values in the dangerous range:**

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
// KW = 1000
``` [4](#0-3) 

The existing test confirms: `FeeRate::calculate(100 CKB, weight=100)` = `10^11`. For a realistic minimum-weight tx (~255 weight units from secp256k1 cycles):

| Fee burned | fee_rate (shannons/kW) | max_bucket_index | Two-vec allocation |
|---|---|---|---|
| 100 CKB | ~3.9 × 10^10 | ~390,000 | ~6 MB |
| 1,000 CKB | ~3.9 × 10^11 | ~3,900,000 | ~62 MB |
| 10,000 CKB | ~3.9 × 10^12 | ~39,000,000 | ~624 MB |
| 100,000 CKB | ~3.9 × 10^13 | ~390,000,000 | ~6.2 GB | [5](#0-4) 

**No cap on fee rate in tx pool admission:**

`check_tx_fee` only enforces a minimum fee rate floor; there is no maximum fee rate check anywhere in the tx pool admission path. [6](#0-5) 

**Call chain:**

1. Attacker calls `send_transaction` RPC with a high-fee, minimal-weight tx
2. Tx passes validation and enters the pool
3. `register_pending` callback fires → `fee_estimator_clone.accept_tx(tx_hash, entry_info)` → `weight_units_flow::Algorithm::accept_tx()` stores the `TxStatus` with the inflated fee rate [7](#0-6) 

4. Any caller of `estimate_fee_rate` RPC → `do_estimate` → reads the high-fee tx as `max_fee_rate` from the current mempool snapshot → `vec![0u64; max_bucket_index + 1]` × 2 [8](#0-7) 

---

### Impact Explanation

Every `estimate_fee_rate` RPC call while the high-fee tx remains in the mempool triggers two heap allocations proportional to the attacker's fee rate. At 10,000 CKB burned, each call allocates ~624 MB. At 100,000 CKB, each call attempts ~6.2 GB. Repeated calls (by the attacker or any other user) cause OOM crashes or severe memory pressure degrading the entire node. This matches the "Critical" scope: node crash / severe degradation.

---

### Likelihood Explanation

**Precondition**: The `WeightUnitsFlow` estimator must be explicitly configured — it is not the default. The default is `FeeEstimator::Dummy`. [9](#0-8) [10](#0-9) 

Nodes that have opted into `WeightUnitsFlow` (a supported, documented production option) are fully exposed. The attacker must own and burn CKB, which is a real economic cost, but the cost is not prohibitive for a motivated attacker targeting a specific node.

---

### Recommendation

1. **Cap `max_bucket_index_by_fee_rate`**: add a `const MAX_BUCKET_INDEX: usize` (e.g., 1000) and clamp the return value.
2. **Cap `max_fee_rate` before computing `max_bucket_index`**: clamp `max_fee_rate` to a safe maximum (e.g., `FeeRate::from_u64(2_000_000)`) before passing it to `max_bucket_index_by_fee_rate`.
3. **Add a maximum fee rate check in tx pool admission** to reject transactions with absurdly high fee rates.

---

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
