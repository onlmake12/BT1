### Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Allows Low-Fee Transactions to Evade Pool Eviction — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` removes all link records for a subtree of transactions **before** calling `remove_entry` on each. Because `update_ancestors_index_key` relies on those links to find which remaining pool entries need their `descendants_fee/size/cycles/count` decremented, the ancestors of the removed subtree that **stay in the pool** are never updated. Their `descendants_*` fields remain permanently inflated, causing their `EvictKey` to reflect a higher fee rate than they actually have, making them resistant to eviction.

---

### Finding Description

`remove_entry_and_descendants` is the primary removal path used during conflict resolution, header-dep invalidation, and ancestor-count eviction. Its implementation is:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← ALL links torn down first
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← then remove each entry
        .collect()
}
``` [1](#0-0) 

Inside `remove_entry`, the call to `update_ancestors_index_key` is supposed to walk up the link graph and subtract the removed entry's weight from every surviving ancestor's `descendants_*` fields:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());  // ← empty: links already gone
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [2](#0-1) 

Because `remove_entry_links` was already called for every entry in the subtree, `calc_ancestors` returns an empty set. The loop body never executes. Every ancestor of the removed subtree that remains in the pool retains the stale, inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` values that were accumulated when the now-removed descendants were added.

The `EvictKey` for those ancestors is computed directly from these stale fields:

```rust
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),   // ← inflated
            ...
        }
    }
}
``` [3](#0-2) 

The eviction loop in `limit_size` uses `next_evict_entry`, which iterates by `evict_key` to find the lowest-priority transaction to drop:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    if let Some(id) = next_evict_entry() {
        let removed = self.pool_map.remove_entry_and_descendants(&id);
        ...

``` [4](#0-3) 

A transaction with an inflated `EvictKey` appears to have a higher fee rate than it actually does, so it is sorted later in the eviction order and survives pool-full conditions that should have removed it.

---

### Impact Explanation

An unprivileged attacker can keep a low-fee transaction alive in the tx-pool indefinitely, even when the pool is full and the node is actively evicting low-fee transactions. This degrades the quality of the mempool for honest users and miners: block templates assembled from the pool will include transactions with lower effective fee rates than the node's `min_fee_rate` policy intends to enforce, and legitimate higher-fee transactions may be rejected as the pool fills with stale low-fee entries.

The `descendants_*` drift is permanent for the lifetime of the affected ancestor entry — there is no periodic recomputation (unlike `total_tx_size`/`total_tx_cycles`, which have a `recompute_total_stat` fallback). [5](#0-4) 

---

### Likelihood Explanation

The trigger path is fully reachable by any node peer or RPC caller with no privileges:

1. Submit transaction **A** (low fee) via `send_transaction`.
2. Submit transaction **B** (high fee, spending an output of A) via `send_transaction`. A's `descendants_*` are now inflated to include B's fee/size/cycles.
3. Submit transaction **C** (spending the same input as B, i.e., a double-spend of B's input) via `send_transaction`. This triggers `resolve_conflict` → `remove_entry_and_descendants(B)`.
4. A's `descendants_*` are never decremented. A's `EvictKey` permanently reflects B's high fee rate.
5. Repeat steps 2–3 to keep refreshing A's apparent fee rate as the pool fills.

This requires only valid, well-formed transactions and standard RPC access. No privileged role, key, or majority hash power is needed.

---

### Recommendation

In `remove_entry_and_descendants`, update the surviving ancestors' `descendants_*` fields **before** tearing down the links. Specifically, for the root entry `id` (whose ancestors remain in the pool), walk up the link graph and call `sub_descendant_weight` on each ancestor for every entry in the removed subtree, then remove the links. Alternatively, restructure `remove_entry` so that `update_ancestors_index_key` is called while links are still intact, and suppress only the `update_descendants_index_key` call (which is the stated intent of the early link removal).

---

### Proof of Concept

```
1. Pool is at capacity. Eviction threshold active.
2. Attacker submits tx_A: fee=1 shannon, size=100 bytes → EvictKey.fee_rate = 10 sat/kB
3. Attacker submits tx_B: fee=10000 shannons, size=100 bytes, input=tx_A.output[0]
   → tx_A.descendants_fee += 10000, tx_A.EvictKey.fee_rate = max(10, 100000) = 100000 sat/kB
4. Attacker submits tx_C: same input as tx_B (conflict)
   → resolve_conflict() → remove_entry_and_descendants(tx_B)
   → tx_A.descendants_fee still = 10000 (never decremented)
   → tx_A.EvictKey.fee_rate still = 100000 sat/kB
5. Pool eviction runs: tx_A sorts near the top of the evict-key order (high apparent fee rate)
   and is never selected for eviction, despite its true fee rate of 10 sat/kB.
6. Attacker repeats steps 3–4 to keep tx_A alive indefinitely.
```
