### Title
No Rate Limiting on `estimate_cycles` RPC Allows Unbounded CKB-VM Execution — (File: `rpc/src/module/chain.rs`)

---

### Summary

The `estimate_cycles` RPC method (and its deprecated alias `dry_run_transaction`) accepts an arbitrary attacker-supplied transaction and runs the full CKB-VM script verifier up to `consensus.max_block_cycles` cycles per call, with no rate limiting, no concurrency cap, and no per-caller throttle at the RPC server layer. An unprivileged RPC caller can flood the node with concurrent requests, each consuming up to the maximum block cycle budget of CPU time, starving the node of resources needed for block processing, P2P relay, and tx-pool operation.

---

### Finding Description

`estimate_cycles` is implemented in `ChainRpcImpl` as a thin wrapper that immediately delegates to `CyclesEstimator::run`:

```rust
fn estimate_cycles(&self, tx: Transaction) -> Result<EstimateCycles> {
    let tx: packed::Transaction = tx.into();
    CyclesEstimator::new(&self.shared).run(tx)
}
``` [1](#0-0) 

Inside `CyclesEstimator::run`, the method resolves the transaction's cell dependencies from the live chain snapshot and then invokes `ScriptVerifier::verify(max_cycles)`, where `max_cycles` is set to `consensus.max_block_cycles` — the maximum cycle budget for an entire block:

```rust
let max_cycles = consensus.max_block_cycles;
...
ScriptVerifier::new(Arc::new(resolved), snapshot.as_data_loader(), consensus, Arc::new(tx_env))
    .verify(max_cycles)
``` [2](#0-1) 

The RPC server (`rpc/src/server.rs`) applies only a 30-second HTTP timeout and an **opt-in** batch size limit (`JSONRPC_BATCH_LIMIT`, disabled by default per the config comment "By default, there is no limitation on the size of batch request size"). There is no per-IP rate limit, no per-method concurrency cap, and no global request throttle anywhere in the RPC stack — confirmed by the complete absence of any rate-limiting logic in `rpc/src/**`: [3](#0-2) 

The deprecated `dry_run_transaction` in the Experiment module is identical — it calls the same `CyclesEstimator::run` with no additional guard:

```rust
fn dry_run_transaction(&self, tx: Transaction) -> Result<EstimateCycles> {
    let tx: packed::Transaction = tx.into();
    CyclesEstimator::new(&self.shared).run(tx)
}
``` [4](#0-3)

### Impact Explanation

Each `estimate_cycles` call runs the CKB-VM synchronously on the Tokio thread pool for up to `max_block_cycles` cycles (a consensus-level ceiling, typically hundreds of millions of cycles). An attacker sending N concurrent requests forces N simultaneous VM executions. Because the CKB node's block processing, P2P relay, and tx-pool verification all share the same async runtime and CPU resources, saturating the thread pool with VM work degrades or halts:

- Block template assembly (`get_block_template`)
- Compact block relay and reconstruction
- Transaction pool admission and verification
- Peer-to-peer sync progress

This constitutes a resource-exhaustion denial-of-service against the node's core consensus-critical functions.

---

### Likelihood Explanation

The RPC endpoint is reachable by any caller who can reach the node's RPC port. While the default bind address is `127.0.0.1:8114`, many operators expose the RPC publicly (e.g., for dApp backends, block explorers, or wallet services). No authentication, no API key, and no proof-of-work is required. The attack requires only the ability to send HTTP POST requests and knowledge of any live cell on-chain to construct a valid resolvable transaction with a maximally-looping script. The 30-second timeout per request means an attacker can sustain continuous CPU pressure with a modest number of concurrent connections.

---

### Recommendation

1. **Add a per-method or global concurrency limit** for CPU-intensive RPC calls (`estimate_cycles`, `dry_run_transaction`) using a semaphore or token-bucket rate limiter at the RPC handler layer, analogous to the `governor`-based `RateLimiter` already used in the relay and hole-punching protocols.
2. **Cap `max_cycles` for RPC estimation** to a value lower than `consensus.max_block_cycles` (e.g., `max_tx_verify_cycles` from the tx-pool config, which is already operator-configurable).
3. **Enable `rpc_batch_limit` by default** and document the CPU-exhaustion risk of `estimate_cycles` in the RPC reference.

### Proof of Concept

```python
import requests, threading, json

# Craft a transaction referencing a live cell with a maximally-looping lock script.
# The script body loops until cycle limit is hit (standard CKB-VM infinite loop pattern).
payload = {
    "jsonrpc": "2.0", "method": "estimate_cycles", "id": 1,
    "params": [{ "cell_deps": [...], "inputs": [...], "outputs": [...],
                 "outputs_data": ["0x"], "version": "0x0", "witnesses": [] }]
}

def flood():
    while True:
        requests.post("http://<node>:8114/", json=payload, timeout=35)

# Launch N concurrent threads — each holds a VM execution for up to max_block_cycles
threads = [threading.Thread(target=flood) for _ in range(64)]
for t in threads: t.start()
# Node's block processing and P2P relay degrade under sustained CPU saturation
```

The attacker needs only one valid live cell reference on-chain (trivially obtained from any block explorer) and a script that loops to the cycle limit. No privileged access, no key material, and no majority hashpower is required.
