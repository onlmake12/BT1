### Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from` PeerIDs in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary
An unprivileged remote peer can cause the `pending_delivered` HashMap (and the `forward_rate_limiter`'s internal `HashMapStateStore`) in `HolePunching` to grow to O(N) entries between `notify()` cleanup ticks by sending `ConnectionRequest` messages addressed to the victim node (`to = victim_peer_id`) with a distinct, attacker-generated `from_peer_id` in each message.

### Finding Description
`HolePunching::pending_delivered` is a `HashMap<PeerId, (Vec<Multiaddr>, u64)>` that is only pruned inside `notify()`, which fires every `CHECK_INTERVAL = 5 minutes`. [8](#0-7) 

Entries are inserted in `respond_delivered()` when the victim node is the `to` target of a `ConnectionRequest`: [2](#0-1) 

The only guard against repeated insertion for the same `from_peer_id` is a 2-minute dedup check: [3](#0-2) 

This check is keyed on `from_peer_id`. An attacker who generates a fresh `from_peer_id` per message bypasses it entirely. The `forward_rate_limiter` is also keyed by `(from, to, msg_item_id)`, so each new `from` value is a new bucket — it never throttles novel `from` values: [5](#0-4) 

The only effective rate limit is the outer `rate_limiter` at 30 messages/sec per session: [4](#0-3) 

This means a single session can insert **30 × 300 = 9,000 entries** into `pending_delivered` before the next `notify()` cleanup. With `max_connections` peers (typically 125+), the total is ~1.1 million entries.

Additionally, the `forward_rate_limiter`'s internal `HashMapStateStore` accumulates one entry per unique `(from, to, msg_item_id)` tuple and is only cleaned via `retain_recent()` on `disconnected()` — never in `notify()`: [7](#0-6) 

A persistent attacker connection that never disconnects causes this state to grow for the entire session lifetime.

### Impact Explanation
Each `pending_delivered` entry holds a `PeerId` (~39 bytes) plus a `Vec<Multiaddr>` of up to `ADDRS_COUNT_LIMIT = 24` addresses (each ~50–100 bytes), totaling ~2–3 KB per entry. [9](#0-8) 

At 9,000 entries per session × 125 sessions × ~2.5 KB = **~2.8 GB** of heap growth before the first cleanup tick. This causes OOM on typical validator nodes, constituting a remote DoS.

### Likelihood Explanation
- The attacker only needs a standard P2P connection to a node with `HolePunching` enabled.
- Generating arbitrary `PeerId` values (Ed25519 keypairs) is trivial and requires no PoW or privileged access.
- The attack is sustainable: the attacker maintains the connection and streams messages at 30/sec.
- No consensus participation, no leaked keys, no Sybil majority required.

### Recommendation
1. **Cap map size**: Enforce a hard upper bound (e.g., 1,024 entries) on `pending_delivered` and `inflight_requests`, rejecting new inserts when the cap is reached.
2. **Periodic rate-limiter cleanup**: Call `self.forward_rate_limiter.retain_recent()` inside `notify()`, not only in `disconnected()`.
3. **Per-session insert quota**: Track how many `pending_delivered` entries originated from each session and evict or reject when a per-session limit is exceeded.

### Proof of Concept
```
1. Attacker establishes a P2P connection to victim node V (peer_id = V).
2. Attacker generates N distinct Ed25519 keypairs → N distinct from_peer_ids F_1..F_N.
3. For each F_i, attacker sends:
     ConnectionRequest { from: F_i, to: V, listen_addrs: [<valid TCP addr>], route: [], max_hops: 6 }
   at 30 msg/sec (within rate_limiter quota).
4. For each F_i:
   - forward_rate_limiter: new key (F_i, V, item_id) → allowed
   - pending_delivered.get(F_i): absent → no dedup skip
   - respond_delivered() sends reply, then inserts (F_i, ([addr], now)) into pending_delivered
5. After 300 seconds: pending_delivered.len() == 9,000 (per session).
6. notify() has not fired yet (CHECK_INTERVAL = 5 min).
7. Repeat across all connected sessions → OOM.

Invariant test: assert pending_delivered.len() <= BOUND after inserting at max rate for 5 min.
Currently fails: no such BOUND exists.
```
