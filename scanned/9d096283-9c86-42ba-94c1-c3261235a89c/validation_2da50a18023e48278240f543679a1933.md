Audit Report

## Title
TOCTOU Race in `start_sync_headers` Permanently Inflates `n_sync_started`, Blocking IBD Sync — (File: sync/src/synchronizer/mod.rs)

## Summary

`start_sync_headers` increments `n_sync_started` atomically but writes the corresponding peer state in a separate, non-atomic step. If a peer disconnects between these two operations, `n_sync_started` is permanently inflated by 1 with no decrement path. In IBD mode, this causes every subsequent sync-peer selection attempt to abort immediately, permanently stalling initial block download until the node is restarted — and the attack is trivially repeatable across restarts.

## Finding Description

The three non-atomic steps in `start_sync_headers` are confirmed in the code:

**Step 1** (L656–662): Eligible peers are collected into a `Vec`, releasing all DashMap read references. Peer X is captured as eligible. [1](#0-0) 

**Step 2** (L672–682): `n_sync_started` is incremented atomically via `fetch_update` — no peer-state lock is held at this point. [2](#0-1) 

**Step 3** (L683–687): `get_mut(&peer)` acquires a fresh DashMap entry lock. If the peer disconnected between Steps 2 and 3, this returns `None` and `start_sync` is never called. Critically, there is **no `fetch_sub` rollback** in the `None` branch. [3](#0-2) 

**Why `disconnected` does not compensate:** `Peers::disconnected` only decrements `n_sync_started` when `peer_state.sync_started()` returns `true`. Since `start_sync` (which calls `chain_sync.start()`) was never reached, `sync_started()` returns `false`, and the decrement is skipped. [4](#0-3) [5](#0-4) 

**IBD lock-out:** The IBD guard `if ibd && x != 0 { None }` causes every subsequent `fetch_update` call to return `Err` when `n_sync_started != 0`, triggering `break` and selecting no peer. [6](#0-5) 

**Test acknowledgment of the race:** [7](#0-6) 

The `Peers` struct holding the two unsynchronized pieces of state: [8](#0-7) 

## Impact Explanation

This matches **High: Vulnerabilities which could easily crash a CKB node**. A node in IBD with `n_sync_started` stuck at 1 can never select a sync peer again. It cannot download headers, cannot advance its chain, and is permanently non-functional for its primary purpose. The node process continues running but is effectively dead for block synchronization. Because the attack is repeatable with zero cost (connect + immediately disconnect), restarting the node does not provide lasting recovery — the attacker can re-trigger the condition on the next IBD attempt.

## Likelihood Explanation

Any unprivileged P2P peer can trigger this: connect to the victim node, wait for `SEND_GET_HEADERS_TOKEN` to fire (periodic), then disconnect. The race window between `fetch_update` and `get_mut` is narrow but realistic — network disconnect callbacks fire asynchronously from the network layer concurrently with the sync timer. No special knowledge of node internals is required. The attack requires no authentication, no funds, and is freely repeatable. The test suite's own comment confirms the developers are aware concurrent execution of this path is possible.

## Recommendation

The increment of `n_sync_started` and the call to `peer_state.start_sync()` must be performed atomically under the same DashMap entry lock. Acquire the DashMap write entry for the peer first (via `state.get_mut(&peer)`), verify the peer is still eligible, then increment `n_sync_started` and call `start_sync` while holding the entry — mirroring the pattern in `Peers::disconnected` which holds the entry while checking `sync_started()`. Alternatively, add an explicit rollback (`fetch_sub`) immediately after the `None` branch in Step 3 to compensate for the orphaned increment.

## Proof of Concept

1. Start node A in IBD (`is_initial_block_download()` = true).
2. Malicious peer B connects; `relay_connected` sets state to `SyncProtocolConnected`.
3. `SEND_GET_HEADERS_TOKEN` fires → `start_sync_headers` collects peer B as eligible (Step 1).
4. `fetch_update` succeeds → `n_sync_started` = 1 (Step 2).
5. Peer B disconnects → `Peers::disconnected` removes B, checks `sync_started()` = false, does **not** decrement `n_sync_started`.
6. `get_mut(&B)` returns `None` → `start_sync` never called, no rollback (Step 3).
7. `n_sync_started` = 1 permanently.
8. All subsequent `SEND_GET_HEADERS_TOKEN` fires: `fetch_update` returns `Err` (IBD + `n_sync_started != 0`) → `break` → no peer selected.
9. Node A is permanently stuck in IBD. Restart and repeat from step 2 to re-trigger.
