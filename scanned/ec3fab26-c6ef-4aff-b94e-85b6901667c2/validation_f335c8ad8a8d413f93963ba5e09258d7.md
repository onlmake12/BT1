Audit Report

## Title
Attacker Can Block RBF Transaction Replacement by Front-Running with Descendant Transaction Tree — (`tx-pool/src/pool.rs`)

## Summary

The `check_rbf` function in `tx-pool/src/pool.rs` enforces `MAX_REPLACEMENT_CANDIDATES = 100`: if the transactions being replaced have more than 100 total descendants in the pool, the replacement is rejected. An attacker who receives even one output from the victim's in-pool transaction can front-run the victim's RBF attempt by submitting a tree of ≥100 cheap descendant transactions (structured to stay within `max_ancestors_count = 25`), causing the replacement to fail indefinitely with `RBFRejected("Tx conflict with too many txs")`. This is a direct instance of mutable pool state (descendant count) being manipulated by an unprivileged actor to block a legitimate operation (fee-bump via RBF).

## Finding Description

**Confirmed constants and configuration:**

`MAX_REPLACEMENT_CANDIDATES = 100` is set at: [1](#0-0) 

RBF is enabled by default in the production config (`min_rbf_rate = 1500 > min_fee_rate = 1000`): [2](#0-1) 

The `enable_rbf()` guard confirms this: [3](#0-2) 

**The vulnerable check (Rule #5):**

`check_rbf` iterates over conflicting pool entries and accumulates their descendant counts: [4](#0-3) 

`calc_descendants` reads live, mutable pool state — any transaction that has been accepted into the pool and descends from the conflict is counted. There is no check on *who* submitted those descendants or *when* they were added relative to the original transaction.

**Why the `max_ancestors_count = 25` limit does not prevent the attack:**

The default config sets `max_ancestors_count = 25`: [5](#0-4) 

This limits the depth of a *linear* descendant chain to 25, not the total number of descendants. An attacker can instead build a **binary tree** of descendants: `tx_b1` (spending Bob's output from `tx_alice`) produces 2 outputs; each is spent by 2 child transactions, each producing 2 outputs, and so on. At tree depth 7, the total descendant count reaches 2^7 − 1 = 127 > 100, while each leaf transaction has only 7 ancestors — well within the limit of 25. The `max_ancestors_count` check is applied per-transaction at submission time and does not cap the total number of descendants of any given pool entry.

**Attack flow (corrected PoC):**

1. Alice submits `tx_alice`: input = cell\_A (confirmed, owned by Alice); output0 = cell\_B (Alice's change); output1 = cell\_C (Bob's payment).
2. Bob submits `tx_b1` spending cell\_C, with 2 outputs (cell\_D0, cell\_D1). Ancestors: {`tx_alice`} = 1.
3. Bob submits `tx_b2a` spending cell\_D0 (2 outputs), `tx_b2b` spending cell\_D1 (2 outputs). Ancestors: {`tx_alice`, `tx_b1`} = 2.
4. Bob continues branching for 7 levels, producing 127 total descendants of `tx_alice`. Each leaf has ≤ 7 ancestors.
5. Alice submits `tx_alice2` (same input cell\_A, higher fee).
6. `check_rbf` computes `replace_count = 127 + 1 = 128 > 100`.
7. Alice's replacement is rejected: `RBFRejected("Tx conflict with too many txs, conflict txs count: 128, expect <= 100")`.

The `check_rbf` call is made inside the write lock in `submit_entry`, so the pool state read by `calc_descendants` is the live, attacker-inflated state: [6](#0-5) 

**Eviction does not protect Alice.** The `limit_size` eviction removes the lowest-fee-rate entries first: [7](#0-6) 

Bob's minimum-fee descendants are the most likely to be evicted, but Bob can resubmit the entire tree before Alice's replacement attempt, sustaining the attack indefinitely.

## Impact Explanation

This is a **bad design that allows CKB network congestion with few costs** (High, 10001–15000 points). An attacker can target any user whose transaction produces an output the attacker controls (a normal payment recipient), permanently blocking that user's ability to fee-bump via RBF. At scale — targeting many users simultaneously — the mempool fills with attacker-submitted low-fee descendant trees, degrading the RBF mechanism for the entire network. The cost per attack cycle is approximately 127 × `min_fee_rate` (≈ 127,000 shannons/KB × tx\_size), negligible on mainnet. The victim's transaction remains stuck in the pool, unconfirmable during congestion, with no recourse.

## Likelihood Explanation

- The attacker only needs to be a **payment recipient** of the victim's transaction — a completely normal, unprivileged role requiring no special keys or admin access.
- RBF is **enabled by default** on mainnet (`min_rbf_rate = 1500 > min_fee_rate = 1000`).
- The attack requires constructing a binary tree of ~127 transactions, each paying only `min_fee_rate`. This is straightforward to automate.
- The attack is **repeatable**: whenever Bob's descendants are evicted by `limit_size`, he resubmits the tree before Alice can get her replacement accepted.
- No victim mistake is required; Alice's only error is having paid Bob.

## Recommendation

- **Primary fix**: When counting replacement candidates in Rule #5, exclude descendants whose *only* connection to the conflict set is through outputs the original sender did not control (i.e., outputs sent to third parties). The replacement candidate limit should apply only to descendants that the original sender could have avoided.
- **Alternative**: Introduce a signed "priority RBF" path that allows the owner of the conflicted transaction's inputs to bypass the descendant count limit when replacing their own transaction, provided the replacement fee satisfies all other rules.
- **Do not simply reduce `MAX_REPLACEMENT_CANDIDATES`**: lowering the threshold makes the attack cheaper, not harder.
- **Mitigation**: Rate-limit descendant submissions per UTXO lineage from a single peer to raise the cost of the tree-building phase.

## Proof of Concept

```
# Prerequisites: CKB node with default config (min_rbf_rate=1500 > min_fee_rate=1000)

# Step 1: Alice submits tx_alice
send_transaction(tx_alice):
  input:   cell_A  (confirmed, owned by Alice)
  output0: cell_B  (Alice's change)
  output1: cell_C  (Bob's payment, 2 sub-outputs for branching)

# Step 2: Bob builds a binary tree of descendants (7 levels, 127 total txs)
# Level 1:
send_transaction(tx_b1): input=cell_C, outputs=[cell_D0, cell_D1], fee=min_fee_rate
# Level 2:
send_transaction(tx_b2a): input=cell_D0, outputs=[cell_E0, cell_E1], fee=min_fee_rate
send_transaction(tx_b2b): input=cell_D1, outputs=[cell_E2, cell_E3], fee=min_fee_rate
# ... continue for 7 levels (each leaf has 7 ancestors <= max_ancestors_count=25)
# Total: 2^7 - 1 = 127 descendants of tx_alice in pool

# Step 3: Alice attempts RBF
send_transaction(tx_alice2):
  input:   cell_A  (same as tx_alice, higher fee)
  output0: cell_B' (less change = higher fee)

# Result:
# check_rbf: replace_count = 127 + 1 = 128 > MAX_REPLACEMENT_CANDIDATES(100)
# Error -1111 (PoolRejectedRBF):
#   "RBFRejected: Tx conflict with too many txs,
#    conflict txs count: 128, expect <= 100"

# Step 4: Bob resubmits tree after any eviction -> attack sustained indefinitely
```

The root cause is the unconditional descendant count check at: [8](#0-7) 

which reads mutable pool state (`calc_descendants`) that any unprivileged transaction recipient can inflate via a binary tree of minimum-fee descendants, without any attribution or rate-limiting tied to the original transaction's sender.
