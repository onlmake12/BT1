Audit Report

## Title
Integer Overflow in `GetLastStateProof` Guard Bypasses 1000-Sample Limit — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The guard at line 201 of `get_last_state_proof.rs` computes `(last_n_blocks as usize) * 2` without overflow protection. In Rust release mode (where integer overflow wraps silently), an attacker can supply `last_n_blocks = 2^63` to cause the multiplication to wrap to `0`, making the guard evaluate to `false` and allowing the handler to proceed. The server then iterates over every block in the chain, performing O(chain_length) storage reads and MMR root computations per request.

## Finding Description

**Guard code** — the unchecked expression: [1](#0-0) 

`last_n_blocks` is decoded directly from the P2P message as a `u64` with no prior bounds check. The constant being guarded is `GET_LAST_STATE_PROOF_LIMIT = 1000`: [2](#0-1) 

**Overflow mechanics (64-bit release mode):**

On a 64-bit target, `usize` is 64 bits. With `last_n_blocks = 2^63`:
- `last_n_blocks as usize` = `9223372036854775808`
- `* 2` overflows 64-bit usize → wraps to `0`
- `0 + 0 > 1000` → `false` → guard is bypassed

In Rust release mode, integer overflow wraps silently (no panic). Debug mode would panic, but production deployments use release builds.

**Post-bypass execution path:**

With `start_block_number = 0` (attacker-controlled) and `last_n_blocks = 2^63`, the reorg branch is skipped: [3](#0-2) 

Then at line 291, `last_block_number - 0 <= 2^63` is always true for any real chain, routing into the "not enough blocks" branch: [4](#0-3) 

This sets `last_n_numbers = (0..last_block_number)` — every block in the chain. These are then passed to `complete_headers`: [5](#0-4) 

Inside `complete_headers`, for **each** block number, the server calls `snapshot.get_ancestor`, `snapshot.get_block`, and `snapshot.chain_root_mmr(*number - 1).get_root()`: [6](#0-5) 

Each `chain_root_mmr` call constructs an MMR over the entire chain up to that block: [7](#0-6) 

This makes the per-request cost O(chain_length²) in the worst case (MMR root computation per block, each reading from the store).

## Impact Explanation

A single malformed P2P request causes the server to perform O(chain_length) storage reads and MMR root computations, far exceeding the intended 1000-block cap. On a chain with millions of blocks, repeated requests from one or more peers can saturate I/O and CPU, degrading or halting service for legitimate peers. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"* and *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation

The attack requires only:
1. A valid tip block hash (public on-chain data, no privilege needed).
2. The ability to connect as a light-client peer (no authentication required).
3. Encoding `last_n_blocks = 2^63` in a molecule-serialized message (trivial).

No key material, authentication, or privileged access is needed. The attack is repeatable and can be automated. Any external actor who can reach the node's P2P port can trigger it.

## Recommendation

Replace the unchecked multiplication with a saturating variant, or add an explicit pre-check on `last_n_blocks` before the combined expression:

```rust
// Option A: saturating_mul — safe, no panic, no wrap
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: explicit pre-check
if last_n_blocks > (constant::GET_LAST_STATE_PROOF_LIMIT as u64) {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Either fix prevents the wrap-around and correctly rejects any `last_n_blocks` that would exceed the limit.

## Proof of Concept

```rust
// Simulates the guard expression as written, in release mode on a 64-bit target:
let last_n_blocks: u64 = (usize::MAX / 2) as u64 + 1; // = 2^63
let difficulties_len: usize = 0;
let limit: usize = 1000;

// The guard expression as written:
let check = difficulties_len + (last_n_blocks as usize) * 2; // wraps to 0 in release mode
assert!(check <= limit, "guard bypassed: {} <= {}", check, limit);
// assert passes — handler proceeds past the guard with last_n_blocks = 2^63
// post-bypass: last_n_numbers = (0..chain_tip), complete_headers iterates all blocks
```

To reproduce end-to-end: connect to a CKB node as a light-client peer, send a `GetLastStateProof` message with `last_n_blocks = 9223372036854775808` (`2^63`), `start_number = 0`, and a valid `last_hash` for the current tip. Observe the node performing storage reads and MMR computations for every block in the chain before responding.
