### Title
Unbounded Memory Allocation via `get_cells` with Default `request_limit=usize::MAX` and `with_data=None` — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

---

### Summary

The rich-indexer RPC `get_cells` defaults `request_limit` to `usize::MAX` when unconfigured, and defaults `with_data` to `true`. An unprivileged caller can send a single `get_cells` request with a large `limit` targeting a script that matches many cells with large `output_data` blobs. The `fetch_all` call materializes the entire result set into a `Vec<AnyRow>` in memory simultaneously, with no streaming or size cap, causing an OOM crash.

---

### Finding Description

**Default `request_limit = usize::MAX`:**

In `util/rich-indexer/src/service.rs`, `RichIndexerService::new` sets:

```rust
request_limit: config.request_limit.unwrap_or(usize::MAX),
``` [1](#0-0) 

The config comment in `resource/ckb.toml` explicitly acknowledges this is dangerous:

> "By default, there is no limitation on the size of indexer request. However, because serde json serialization consumes too much memory (10x), it may cause the physical machine to become unresponsive. We recommend a consumption limit of 2g, which is 400 as the limit." [2](#0-1) 

The recommended safe value (`request_limit = 400`) is **commented out** — the default is `usize::MAX`.

**Guard check is ineffective at default:**

In `get_cells`, the only guard is:

```rust
if limit as usize > self.request_limit {
    return Err(...);
}
``` [3](#0-2) 

Since `limit` is `Uint32` (max ~4.3 billion) and `request_limit` defaults to `usize::MAX`, this check never triggers.

**`with_data` defaults to `true`:**

When `with_data` is `None`, the query selects the full `output.data` blob:

```rust
match search_key.with_data {
    Some(true) | None => {
        query_builder.field("output.data as output_data");
    }
``` [4](#0-3) 

**`fetch_all` loads everything into memory at once:**

```rust
let cells = self
    .store
    .fetch_all(query)
    .await
    ...
    .collect::<Vec<_>>();
``` [5](#0-4) 

`fetch_all` is implemented as `query.fetch_all(pool)` — sqlx's bulk-load API that materializes all rows into a `Vec<AnyRow>` before returning: [6](#0-5) 

There is no streaming path, no per-response byte cap, and no timeout guard (unlike the RocksDB-backed indexer which uses `TimeoutIterator`).

---

### Impact Explanation

An attacker sends one RPC call:
```json
{"method": "get_cells", "params": [{"script": <common_script>, "script_type": "lock"}, "asc", "0xFFFFFFFF", null]}
```

With `with_data` omitted (defaults `true`) and `limit = u32::MAX`, the SQL query runs `SELECT ... output.data ... LIMIT 4294967295`. If the DB contains N cells each with D bytes of `output_data`, `fetch_all` allocates N×D bytes in a single `Vec<AnyRow>`. The config comment notes serde serialization adds another 10× overhead. With 1,000 cells × 1 MB each, this is ~10 GB of process memory — sufficient to OOM-kill the node process.

---

### Likelihood Explanation

- The rich-indexer RPC is publicly accessible to any caller who can reach the RPC port.
- `request_limit` is **not set by default** — operators must opt-in to the safe value. The safe value is only mentioned in a comment.
- Mainnet/testnet cells routinely carry large `output_data` (e.g., on-chain scripts, NFT metadata, spore cells).
- No authentication, PoW, or consensus check gates this path.

---

### Recommendation

1. **Change the default** of `request_limit` from `usize::MAX` to a safe value (e.g., 400, as the comment recommends) in `RichIndexerService::new`.
2. **Add a per-response byte budget**: track cumulative `output_data` size during row iteration and abort if it exceeds a configurable threshold.
3. **Use streaming** (`fetch` with `try_next`) instead of `fetch_all` so memory is bounded to one row at a time, with early termination on budget exhaustion.
4. **Add a timeout guard** analogous to `TimeoutIterator` used in the RocksDB indexer path. [1](#0-0) 

---

### Proof of Concept

1. Start a CKB node with rich-indexer enabled and no `request_limit` configured (the default).
2. Index 1,000 transactions each with a cell under the same lock script and 1 MB `output_data`.
3. Send:
   ```bash
   curl -X POST http://localhost:8114 -H 'Content-Type: application/json' \
     -d '{"jsonrpc":"2.0","method":"get_cells","params":[{"script":{"code_hash":"<hash>","hash_type":"type","args":"0x"},"script_type":"lock"},"asc","0x3E8",null],"id":1}'
   ```
   (`0x3E8` = 1000, `with_data` omitted → defaults `true`)
4. Observe node RSS grow by ~1 GB (plus 10× serde overhead) and OOM-kill.
