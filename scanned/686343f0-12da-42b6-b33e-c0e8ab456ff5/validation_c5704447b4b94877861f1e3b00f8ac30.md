### Title
`ScriptHashTypeVerifier` Validates Only `output.lock()` Hash Type, Silently Skipping `output.type_()` — (`verification/src/transaction_verifier.rs`)

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only against each output's **lock** script. It never inspects the **type** script hash type. This mirrors the original report's pattern exactly: one function (`getSwapData`) accepts a broader input set than the downstream validator (`verifyQuote`). Here, the non-contextual gate accepts outputs whose type scripts carry a future/unpermitted hash type (e.g., `Data3`), while the contextual script execution engine (`select_version`) rejects them — but only after the transaction has already been admitted to the tx-pool.

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as the set `{0, 1, 2, 4}` (Data, Type, Data1, Data2): [1](#0-0) 

`ScriptHashType` is generated via a `seq!` macro that produces variants `Data3` through `Data127` (values 6, 8, 10, …, 254), none of which are in the enabled set: [2](#0-1) 

`ScriptHashTypeVerifier::verify()` iterates over every output and checks only `output.lock().hash_type()`. The `output.type_()` field is never touched: [3](#0-2) 

`ScriptHashTypeVerifier` is a sub-verifier of `NonContextualTransactionVerifier`, which is the first gate called by the tx-pool's `non_contextual_verify`: [4](#0-3) [5](#0-4) 

The downstream contextual path — `ContextualTransactionVerifier` → `ScriptVerifier` → `select_version` — **does** check the type script hash type and returns `InvalidScriptHashType` for any value outside the four enabled variants: [6](#0-5) 

The gap: a transaction output with `type_` carrying hash type `Data3` (byte value `6`) passes `ScriptHashTypeVerifier` (lock is valid), passes `non_contextual_verify`, enters the tx-pool, and is only rejected during the more expensive contextual verification step.

### Impact Explanation

Any tx-pool submitter (RPC caller via `send_transaction`) can craft a transaction whose outputs have a valid lock script but a type script with a future/unpermitted hash type. Such a transaction:

1. Passes `NonContextualTransactionVerifier` — the only non-contextual gate.
2. Is admitted to the tx-pool, consuming a slot and triggering full contextual verification.
3. Fails contextual verification at `select_version` with `InvalidScriptHashType`.
4. Is evicted, but the CPU and memory cost of contextual verification has already been paid.

Because the non-contextual check is the intended cheap filter before the expensive contextual check, bypassing it for type scripts undermines the two-stage admission design. An attacker with any live UTXO can flood the tx-pool with such transactions at low cost, forcing repeated contextual verification cycles.

### Likelihood Explanation

The attacker needs only a single live cell (any UTXO) to construct an unlimited number of distinct transactions (varying outputs, amounts, etc.) that each carry an invalid type script hash type. The RPC endpoint `send_transaction` is publicly reachable. The non-contextual check is the intended DoS barrier; its gap here is directly reachable by any unprivileged RPC caller.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the hash type of each output's optional type script, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Obtain any live UTXO on-chain.
2. Construct a transaction spending that UTXO with one output whose lock is a valid `Type`-hash-type script and whose `type_` is set to any script with `hash_type = 6` (`Data3`).
3. Submit via `send_transaction` RPC.
4. Observe: `non_contextual_verify` returns `Ok` (lock hash type is valid; type hash type is never checked).
5. Observe: contextual verification (`verify_rtx`) fails with `InvalidScriptHashType` from `select_version`.
6. The transaction was admitted to the tx-pool and triggered full contextual verification before being evicted — the non-contextual filter failed to catch it.

The test at `verification/src/tests/transaction_verifier.rs` line 101 (`test_not_enabled_hash_type_output_lock`) confirms the lock-only scope of the current verifier and the absence of any analogous test for type scripts: [7](#0-6)

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
