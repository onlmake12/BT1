### Title
DAO Withdrawal Fee Calculator Uses Full u64 Witness Index While C VM Uses Only Lowest Byte — Divergent Header Resolution Causes Valid Transactions to Be Rejected by Tx Pool - (File: util/dao/src/lib.rs)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` function resolves the deposit block header by reading the witness `input_type` field as a full `u64` index into `header_deps`. The on-chain C VM (`dao.c`) resolves the same field using only the **lowest byte** of that 8-byte value. For any witness index ≥ 256, the two implementations look at different positions in `header_deps`, selecting different blocks for the DAO interest calculation. This is the direct CKB analog of the "incorrect swap path" class: the wrong routing key is used to resolve a critical parameter, causing the Rust node's fee/capacity accounting to diverge from what the on-chain script actually enforces.

---

The C VM (`dao.c`) interprets the same 8-byte little-endian witness field using only its **lowest byte** as the `header_deps` array index. This is explicitly documented in the test suite:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
``` [2](#0-1) 

The test `check_dao_withdraw_header_dep_index_exceeds_u8` constructs a transaction with witness index `257` (lowest byte = `1`), places the deposit block at `header_deps[1]` and the withdraw block at `header_deps[257]`. The C VM resolves `header_deps[1]` (deposit block, correct), while Rust resolves `header_deps[257]` (withdraw block, wrong). The Rust block-number cross-check at line 105 catches this specific case and returns `DaoError::InvalidOutPoint`:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [3](#0-2) 

The block-number check is a partial mitigation, but it does not close the divergence for the **reverse direction**: a transaction that is valid on-chain (C VM resolves the correct deposit block via the lowest byte) is rejected by the Rust tx-pool fee calculator (which resolves a different block via the full u64 and fails the block-number check).

**Concrete attack path (DoS direction):**

1. A transaction submitter constructs a DAO phase-2 withdrawal with 258+ `header_deps`.
2. The deposit block is placed at `header_deps[1]` (index 1 in the C VM's view, lowest byte of `257`).
3. A different canonical block (with a different block number) is placed at `header_deps[257]`.
4. The witness `input_type` is set to `257` (little-endian u64).
5. The C VM resolves `header_deps[1]` → deposit block → block-number check passes → script accepts.
6. The Rust `DaoCalculator` resolves `header_deps[257]` → different block → block-number check fails → `DaoError::InvalidOutPoint` → tx-pool rejects the transaction.

The transaction is valid on-chain but permanently rejected by every standard CKB node's tx-pool.

**Concrete attack path (tx-pool pollution direction):**

1. Attacker places the deposit block at `header_deps[257]` (Rust resolves correctly, block-number check passes).
2. A different canonical block is placed at `header_deps[1]` (C VM resolves this, block-number check fails in the C VM).
3. Rust fee check passes; the transaction enters the tx-pool.
4. Script execution runs the C VM, which rejects the transaction.
5. The transaction is evicted, but it consumed tx-pool admission resources and cycles. [4](#0-3) 

---

### Impact Explanation

- **DoS against DAO withdrawers**: Any DAO phase-2 withdrawal transaction whose witness index ≥ 256 and whose lowest byte differs from the full u64 value will be rejected by the Rust tx-pool even though it is valid on-chain. The user cannot complete their withdrawal through the standard tx-pool submission path.
- **Tx-pool resource waste**: The reverse direction allows an attacker to craft transactions that pass the Rust fee check but fail C VM script execution, consuming tx-pool admission cycles and memory before eviction.
- **Incorrect capacity accounting**: The `transaction_fee` function, which is called during tx-pool admission (`check_tx_fee` in `tx-pool/src/util.rs`), computes the maximum withdrawal using the wrong block's `ar` accumulation ratio, producing an incorrect fee figure. [5](#0-4) 

---

### Likelihood Explanation

A witness index ≥ 256 requires a transaction with ≥ 256 `header_deps`. While unusual in normal usage, this is not protocol-prohibited. An attacker who wants to grief a specific DAO depositor can construct such a transaction on their behalf (or the depositor may independently construct one if they have many header deps). The discrepancy is already documented in the test suite, confirming the developers are aware of the behavioral split between Rust and C VM, but no protocol-level guard prevents the divergence from being triggered.

---

### Recommendation

1. **Align the Rust index interpretation with the C VM**: If `dao.c` uses only the lowest byte of the 8-byte witness index, the Rust `DaoCalculator` should apply the same mask (`header_dep_index & 0xFF`) before indexing into `header_deps`. Alternatively, if the C VM is the one that should be fixed, the Rust code should remain as-is and the C VM updated.
2. **Add an explicit upper-bound check**: Reject any witness index ≥ 256 (or whatever the agreed maximum is) at the Rust layer so the two implementations are guaranteed to agree on the valid index range.
3. **Add a consensus-level limit on `header_deps` count** to prevent the construction of transactions with 256+ header deps, eliminating the divergence surface entirely.

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` directly demonstrates the divergence:

- Witness index = `257` (lowest byte = `1`)
- `header_deps[1]` = deposit block (block number 100) — what C VM resolves
- `header_deps[257]` = withdraw block (block number 200) — what Rust resolves
- Rust block-number check: `200 ≠ 100` → `DaoError::InvalidOutPoint`
- C VM would resolve `header_deps[1]` → deposit block → accept [6](#0-5) 

To demonstrate the DoS direction, swap the positions: place the deposit block at `header_deps[1]` and any other canonical block at `header_deps[257]`. The C VM accepts (resolves deposit block via lowest byte); Rust rejects (resolves wrong block, block-number mismatch). The transaction is valid on-chain but permanently blocked from the tx-pool.

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
