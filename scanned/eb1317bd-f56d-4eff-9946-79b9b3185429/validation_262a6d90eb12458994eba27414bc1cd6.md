### Title
Fee Rate Estimation Uses Full `max_block_bytes` Without Subtracting Basic Block Overhead — (`File: tx-pool/src/pool.rs`)

---

### Summary

The fallback `estimate_fee_rate` function in `tx-pool/src/pool.rs` simulates block-filling using the raw consensus `max_block_bytes` limit, without subtracting the bytes consumed by the block's fixed structure (cellbase, proposals, uncles, extension). The actual block assembler correctly subtracts this overhead before selecting transactions. The mismatch causes the fee estimator to overestimate available block space, returning a fee rate that is systematically too low to achieve the requested confirmation target.

---

### Finding Description

In `tx-pool/src/pool.rs`, the fallback `estimate_fee_rate` function passes the raw consensus `max_block_bytes` directly to `pool_map.estimate_fee_rate`: [1](#0-0) 

```rust
let fee_rate = self.pool_map.estimate_fee_rate(
    (target_to_be_committed - self.snapshot.consensus().tx_proposal_window().closest()) as usize,
    self.snapshot.consensus().max_block_bytes() as usize,   // ← raw limit, no overhead subtracted
    self.snapshot.consensus().max_block_cycles(),
    self.config.min_fee_rate,
);
```

The inner `pool_map.estimate_fee_rate` then simulates filling blocks by accumulating transaction sizes until `current_block_bytes >= max_block_bytes`: [2](#0-1) 

In contrast, the actual block assembler (`update_full` and `update_transactions`) always computes a `basic_size` for the fixed block structure and subtracts it before selecting transactions: [3](#0-2) 

```rust
let basic_size = Self::basic_block_size(
    current_template.cellbase.data(),
    uncles,
    proposals.iter(),
    current_template.extension.clone(),
);
let txs_size_limit = max_block_bytes.checked_sub(basic_size)...;
let (txs, ..) = tx_pool_reader.package_txs(max_block_cycles, txs_size_limit);
```

The `basic_block_size` accounts for the serialized cellbase transaction, uncle block headers, proposal short IDs (up to `MAX_BLOCK_PROPOSALS_LIMIT = 1500` × 10 bytes = 15,000 bytes), and any extension field. [4](#0-3) 

`MAX_BLOCK_BYTES` is `597 × 1000 = 597,000` bytes. The basic block overhead (cellbase + proposals alone) can reach ~15,000–16,000 bytes, representing roughly 2.5–3% of the total block budget. The fee estimator treats all 597,000 bytes as available for transactions, while the assembler only makes ~581,000–582,000 bytes available.

---

### Impact Explanation

The fee estimator simulates more transactions fitting per block than the assembler actually selects. This causes the simulated block counter to decrement faster than reality, so the returned fee rate is lower than what is actually required to be included within `target_to_be_committed` blocks. Any RPC caller (wallet, dApp, exchange) that uses the `estimate_fee_rate` RPC to set transaction fees receives a systematically underestimated fee rate. Transactions submitted at this fee rate may not be included in the expected number of blocks, causing unexpected delays.

---

### Likelihood Explanation

The `estimate_fee_rate` fallback is the primary fee estimation path when the node lacks sufficient historical data (e.g., after a fresh start or during low-traffic periods). It is reachable by any unprivileged RPC caller. The discrepancy is systematic and proportional to the basic block overhead, which is always non-zero. The error is small in percentage terms (~2–3%) but is a consistent underestimate that compounds across all fee estimates produced by this path.

---

### Recommendation

In `tx-pool/src/pool.rs`, subtract a representative basic block overhead from `max_block_bytes` before passing it to `pool_map.estimate_fee_rate`. The simplest correct approach is to use the same `basic_block_size` calculation used in the block assembler, or at minimum subtract a conservative fixed overhead (e.g., the serialized size of a minimal cellbase plus the maximum proposals overhead: `MAX_BLOCK_PROPOSALS_LIMIT * ProposalShortId::serialized_size()`).

---

### Proof of Concept

1. Start a CKB node and submit enough transactions to fill several blocks.
2. Call `estimate_fee_rate` via RPC with `target_to_be_committed = 3`.
3. The returned fee rate `R` is computed assuming all `597,000` bytes per block are available for transactions.
4. Submit a transaction at fee rate `R`.
5. Observe that the block assembler, which uses `txs_size_limit = 597,000 - basic_size` (where `basic_size ≥ ~15,000` bytes), selects fewer transactions per block than the estimator assumed.
6. The transaction is not included within 3 blocks as expected, because the estimator underestimated congestion by overestimating available block space.

    pub(crate) fn estimate_fee_rate(
        &self,
        mut target_blocks: usize,
        max_block_bytes: usize,
        max_block_cycles: Cycle,
        min_fee_rate: FeeRate,
    ) -> FeeRate {
        debug_assert!(target_blocks > 0);
        let iter = self.entries.iter_by_score().rev();
        let mut current_block_bytes = 0;
        let mut current_block_cycles = 0;
        for entry in iter {
            current_block_bytes += entry.inner.size;
            current_block_cycles += entry.inner.cycles;
            if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
                target_blocks -= 1;
                if target_blocks == 0 {
                    return entry.inner.fee_rate();
                }
                current_block_bytes = entry.inner.size;
                current_block_cycles = entry.inner.cycles;
            }
        }

        min_fee_rate
    }
