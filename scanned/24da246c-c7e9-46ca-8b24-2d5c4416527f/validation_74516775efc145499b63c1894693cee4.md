Audit Report

## Title
Stale `descendants_count` and `evict_key` on Surviving Parent After `remove_entry_and_descendants` Corrupts Eviction Order — (`File: tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` strips all link records via `remove_entry_links` before calling `remove_entry` on each member of the removed subtree. Because `remove_entry` relies on the live link graph to locate ancestors via `calc_ancestors`, and those links are already gone, the surviving parent of the removed subtree never has `sub_descendant_weight` called on it and never has its `evict_key` recomputed. The parent's `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` remain permanently inflated, causing incorrect eviction ordering for transactions sharing the same fee_rate.

## Finding Description

`remove_entry_and_descendants` first calls `remove_entry_links` for every entry in the removed set:

```rust
// pool_map.rs L252-265
for id in &removed_ids {
    self.remove_entry_links(id);  // strips id from self.links.inner
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` calls `self.links.remove(id)` which deletes the entry from `self.links.inner`. When `remove_entry` is subsequently called, it invokes `update_ancestors_index_key(&entry.inner, EntryOp::Remove)`, which calls `self.links.calc_ancestors(&child.proposal_short_id())`. `calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)` — returning `None` since the entry was already removed — and falls through to `unwrap_or_default()`, yielding an empty `HashSet`. The loop over ancestors never executes, so `sub_descendant_weight` and `evict_key` recomputation are never reached for the surviving parent.

The `EvictKey` ordering (primary: `fee_rate` ascending, secondary: `descendants_count` ascending) means an inflated `descendants_count` on the surviving parent produces a **larger** evict_key than correct. `next_evict_entry` iterates `iter_by_evict_key()` in ascending order, so the parent is evicted **later** than it should be. When the pool is full and `limit_size` fires, a transaction with the same fee_rate but a correctly-computed lower `descendants_count` is evicted first instead of the parent whose count is stale.

The comment at line 256 — `"update links state for remove, so that we won't update_descendants_index_key in remove_entry"` — confirms the intent was only to skip `update_descendants_index_key` (updating descendants' ancestor scores, which is unnecessary since all descendants are being removed). The side effect of also nullifying `update_ancestors_index_key` for the surviving parent is unintended.

## Impact Explanation

This is a correctness bug in tx-pool eviction ordering. An unprivileged user can keep a low-fee-rate transaction alive in the pool beyond its intended eviction point, causing a transaction with the same fee_rate submitted by another user to be evicted and receive `Reject::Full` instead. The stale state persists until the parent is itself removed or the node restarts. This matches **Low (501–2000 points): Any other important performance improvements for CKB**, as it degrades the correctness of the pool's eviction policy without crashing the node or causing consensus deviation. The claim's assertion that a "higher-fee-rate transaction is evicted instead" is factually incorrect: fee_rate is the primary EvictKey comparator, so a higher-fee-rate transaction has a larger evict_key and is evicted later, not first. The actual displaced transaction has the same fee_rate.

## Likelihood Explanation

RBF is enabled by default (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`). Any unprivileged peer can submit transactions via `send_transaction` RPC. The two-step setup (submit parent + child, then RBF-replace child) requires no special privilege. The bug also triggers via `resolve_conflict` and `limit_size` itself, not only RBF. The effect is most pronounced when the pool is under load and many transactions share the same fee_rate, which is a realistic mainnet condition.

## Recommendation

Before stripping links in `remove_entry_and_descendants`, collect the direct parents of the root entry that are **not** in `removed_ids`. After all removals are complete, explicitly call `sub_descendant_weight` and recompute `evict_key` for each surviving parent. Alternatively, restructure so that `update_ancestors_index_key` is called for the root entry (using the live graph) before any `remove_entry_links` call is made.

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =

            // update parent score

                    EntryOp::Add => e.inner.add_descendant_weight(child),
