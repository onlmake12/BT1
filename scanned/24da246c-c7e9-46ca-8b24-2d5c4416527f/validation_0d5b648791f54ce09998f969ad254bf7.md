Audit Report

## Title
`PoolMap::add_entry()` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Eviction Snapshot — (`File: tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry()`, `updated_stat_for_add_tx` snapshots the current pool totals plus the new entry's contribution into local variables before `check_and_record_ancestors` runs. When that function evicts cell-ref parent transactions via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in-place. However, lines 218–219 then unconditionally overwrite those fields with the stale pre-eviction locals, permanently re-inflating the counters by the size/cycles of every evicted transaction. An unprivileged attacker can repeatedly trigger this path to make the pool's accounting diverge arbitrarily from reality, causing over-eviction of valid transactions and rejection of legitimate submissions.

## Finding Description

**Root cause — stale snapshot written after in-place mutation:**

- **Lines 210–211** (`add_entry`): `updated_stat_for_add_tx` returns `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` into locals. At this moment the evicted transactions are still in the pool, so the snapshot includes their contribution.
- **Line 213** (`add_entry`): `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (lines 603–625 of `pool_map.rs`), it calls `remove_entry_and_descendants` in a loop. Each call reaches `update_stat_for_remove_tx` (lines 733–758), which **directly writes** decremented values into `self.total_tx_size` and `self.total_tx_cycles`.
- **Lines 218–219** (`add_entry`): `self.total_tx_size = total_tx_size; self.total_tx_cycles = total_tx_cycles;` — the stale locals (which still include the evicted entries' sizes/cycles) overwrite the correctly-decremented fields.

The net effect after one eviction cycle: `self.total_tx_size` is inflated by `Σ size(evicted_tx)` and `self.total_tx_cycles` by `Σ cycles(evicted_tx)`. This inflation is permanent and cumulative across repeated triggers.

**Why existing guards are insufficient:**

`update_stat_for_remove_tx` has an underflow guard that falls back to `recompute_total_stat()` (lines 742–756), but this guard only fires on underflow. The overwrite at lines 218–219 happens unconditionally after the correct decrement, so the recompute path is never reached for this bug.

## Impact Explanation

The inflated `total_tx_size` is the authoritative counter used in two critical paths:

1. **`limit_size()` over-eviction** (pool.rs line 298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` — an inflated counter causes the loop to evict valid, fee-paying transactions even when the real pool is within its configured limit.
2. **Future admission rejection**: Every subsequent `add_entry` call reads the inflated `self.total_tx_size` as its baseline in `updated_stat_for_add_tx`, causing legitimate transactions to be rejected with `Reject::Full` even when the pool has real capacity.

An attacker can repeat the trigger to inflate the counter without bound, effectively making the node's mempool refuse all new transactions and continuously evict existing ones. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The attacker needs only standard `send_transaction` RPC access and a small number of crafted transactions per trigger cycle.

## Likelihood Explanation

The eviction branch in `check_and_record_ancestors` requires:
1. A submitted transaction whose `ancestors_count > max_ancestors_count`, AND
2. `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (i.e., evicting cell-ref parents brings it within limits).

Both conditions are fully attacker-controlled via standard transaction submission. The default `max_ancestors_count` on mainnet is 25, so the attacker needs to submit ~25 chained transactions plus one cell-dep transaction plus one trigger transaction — a total of ~27 transactions per inflation cycle. This is cheap, repeatable, and requires no privileged access or key material.

## Recommendation

Do not snapshot totals before `check_and_record_ancestors`. Instead, after all evictions have completed, add only the new entry's contribution directly to the post-eviction `self.total_tx_size`/`self.total_tx_cycles`:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Overflow pre-check only; do NOT capture totals into locals.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Add new entry's contribution on top of post-eviction totals.
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

`updated_stat_for_add_tx` should be changed to a pure overflow check (or its return value discarded), since the actual write now happens after eviction.

## Proof of Concept

1. Configure a node with `max_ancestors_count = 5` and a small `max_tx_pool_size`.
2. Submit 5 chained transactions A→B→C→D→E (fills ancestor limit).
3. Submit transaction F that uses transaction C as a **cell dep** (C becomes a cell-ref parent of F).
4. Submit transaction G that spends E's output and also depends on F. G's ancestor count = 6 > 5, but cell-ref parent C can be evicted to bring it to 5 — triggering the eviction branch.
5. Observe via `tx_pool_info` RPC: `total_tx_size` is now inflated by the size of C and its descendants (D, E, F — all evicted as descendants of C), even though those transactions are no longer in the pool.
6. Repeat steps 2–4 to inflate the counter further. Eventually all new `send_transaction` calls return `Reject::Full` and `limit_size` continuously evicts valid transactions.

A unit test can assert `pool_map.total_tx_size == pool_map.recompute_total_stat().0` after each `add_entry` call to detect the invariant violation directly.
