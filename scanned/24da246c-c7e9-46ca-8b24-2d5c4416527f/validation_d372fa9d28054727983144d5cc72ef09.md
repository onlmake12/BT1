Audit Report

## Title
`check_purge` Eviction Logic Defeated by Distinct /16 Subnet Flooding — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
The `check_purge` function in `PeerStore` has two sequential eviction paths, both of which produce empty candidate sets when an attacker fills all 16384 peer store slots with addresses from distinct /16 subnets (≤4 per group). Path 1 skips freshly-added addresses because they are all `is_connectable`, and Path 2 skips every group because `addrs.len() > 4` is never satisfied with 1 address per group. The result is `Err(PeerStoreError::EvictionFailed)`, silently dropped by the discovery protocol, permanently blocking new legitimate peer addresses from being added until the attacker stops re-advertising.

## Finding Description
`check_purge` is called on every `add_addr` invocation once `addr_manager.count() >= ADDR_COUNT_LIMIT (16384)`.

**Path 1** collects addresses where `!addr.is_connectable(now_ms)`. Addresses inserted via `add_addr` are created with `last_connected_at_ms = 0`, `attempts_count = 0`, and `last_tried_at_ms = 0`. In `is_connectable`:
- `tried_in_last_minute`: `0 >= now_ms - 60000` → false
- `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES(3)`: `0 >= 3` → false
- timeout+failures check: `attempts_count >= ADDR_MAX_FAILURES(10)`: `0 >= 10` → false
- Returns `true` for every fresh address

So `candidate_peers` is empty and Path 1 removes nothing.

**Path 2** groups addresses by `Group`, sorts groups by descending size, takes `len / 2` groups, then only evicts from groups where `addrs.len() > 4`. With 16384 addresses each in a distinct /16 subnet (`Group::IP4([bits[0], bits[1]])`):
- `len = 16384`, `take(8192)` processes 8192 groups
- Each group has exactly 1 address → `1 > 4` is false → `None` for every group
- `candidate_peers` is empty → `return Err(PeerStoreError::EvictionFailed)`

The error propagates up through `add_addr` and is silently swallowed at `debug` level in `DiscoveryAddressManager::add_new_addrs`.

The discovery protocol enforces `MAX_ADDR_TO_SEND = 1000` nodes per message and `MAX_ADDRS = 3` addresses per node, yielding up to 3000 addresses per message. Filling 16384 slots requires approximately 6 messages, deliverable from a single connected peer across multiple rounds.

## Impact Explanation
Once the peer store is saturated, every subsequent `add_addr` call returns `Err(EvictionFailed)` and is silently dropped. The victim node cannot register any newly discovered peer addresses from the discovery or identify protocols. As existing connections age out and are not replaced, the node progressively loses network connectivity. Applied simultaneously to multiple nodes (each requiring only one inbound/outbound connection), this can cause widespread peer discovery failure and network partitioning with minimal attacker resources. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
- Requires only a single P2P connection (inbound or outbound) to the victim.
- No privileged access, no PoW, no key material required.
- The discovery protocol imposes no per-session or per-/16 rate limit on distinct subnets advertised.
- The attacker must continuously re-advertise fresh addresses to maintain the filled state (as failed connection attempts eventually make addresses non-connectable after `ADDR_MAX_RETRIES = 3` failures), but this is trivially automated.
- The attack is local-testable without any network infrastructure.

## Recommendation
1. **Remove or relax the `take(len / 2)` truncation** in Path 2 so all groups are considered for eviction, not just the top half.
2. **Lower or remove the `> 4` threshold**: always evict at least one address from the largest group regardless of its size, ensuring `candidate_peers` is never empty when the store is full.
3. **Add a per-session or per-/16 subnet cap** in `add_new_addrs` to bound how many distinct subnets a single peer can contribute to the store in one session.
4. **Elevate `EvictionFailed` to `warn!`** (not `debug!`) so operators can detect saturation events.

## Proof of Concept
```rust
let mut peer_store = PeerStore::default();
// Fill with 16384 addresses, one per distinct /16 subnet
for i in 0u32..16384 {
    let a = (i >> 8) as u8;
    let b = (i & 0xff) as u8;
    let addr: Multiaddr = format!(
        "/ip4/{}.{}.0.1/tcp/8114/p2p/{}",
        a, b, PeerId::random().to_base58()
    ).parse().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Attempt to add a new legitimate address — returns Err(EvictionFailed)
let new_addr: Multiaddr = format!(
    "/ip4/200.200.200.1/tcp/8114/p2p/{}",
    PeerId::random().to_base58()
).parse().unwrap();
assert!(peer_store.add_addr(new_addr, Flags::COMPATIBILITY).is_err());
```

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }

                        err, addr
                    );
                }
            });
        }
