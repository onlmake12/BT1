### Title
Missing Competing-Proposer Check in Pre-Loop Parent-Block Scan Causes Proposal Reward Overpayment — (`util/reward-calculator/src/lib.rs`)

---

### Summary

`proposal_reward` in `util/reward-calculator/src/lib.rs` scans the parent block for committed transactions **before** the while loop that populates the `proposed` HashSet. The pre-loop scan unconditionally awards the 40% proposal reward to the target block for every matching committed transaction, with no check for earlier competing proposers. If all target proposals are committed in the parent block, `target_proposals` is drained to empty before the while loop is entered, causing the loop to exit immediately on the `!target_proposals.is_empty()` guard. The `proposed` set is never populated, so any competing block that proposed the same transaction at an earlier height within the valid proposal window is never detected. The target block receives the full proposal reward it does not deserve.

---

### Finding Description

The function `proposal_reward` (`util/reward-calculator/src/lib.rs`, lines 173–272) implements the "earliest proposer gets 40% of tx fee" rule. Its structure is:

**Phase 1 — pre-loop scan of the parent block (lines 220–236):**

```rust
let committed_idx = committed_idx_proc(&index.hash());   // index == parent
let has_committed = target_proposals
    .intersection(&committed_idx.iter().cloned().collect::<HashSet<_>>())
    .next()
    .is_some();
if has_committed {
    for (id, tx_fee) in committed_idx.into_iter().zip(txs_fees_proc(&index.hash()).iter()) {
        // "target block is the earliest block with effective proposals for the parent block"
        if target_proposals.remove(&id) {
            reward = reward.safe_add(tx_fee.safe_mul_ratio(proposer_ratio)?)?;
        }
    }
}
```

No check against `proposed` is performed here. The comment at line 231 encodes the incorrect assumption that the target block is always the earliest proposer for transactions committed in the parent block.

**Phase 2 — while loop (lines 238–270):**

```rust
while index.number() > competing_commit_start && !target_proposals.is_empty() {
    index = store.get_block_header(&index.data().raw().parent_hash())...;
    let competing_proposal_start =
        cmp::max(index.number().saturating_sub(proposal_window.farthest()), 1);
    let previous_ids = store.get_block_hash(competing_proposal_start)
        .map(|hash| self.get_proposal_ids_by_hash(&hash))...;
    proposed.extend(previous_ids);   // ← only populated here
    ...
    if target_proposals.remove(&id) && !proposed.contains(&id) {  // ← guard
        reward = ...;
    }
}
```

The `proposed` set is only populated inside the while loop. For the parent block, the equivalent `competing_proposal_start` would be `parent.number() − farthest` — the earliest valid proposer for a transaction committed at `parent`. This check is entirely absent from Phase 1.

**The early-exit path:** If every proposal in `target_proposals` is committed in the parent block, Phase 1 drains `target_proposals` to `{}`. The while loop condition `!target_proposals.is_empty()` is immediately false; the loop body never executes; `proposed` stays empty; no competing proposer is ever detected.

---

### Impact Explanation

A miner who mines both block T (the target) and block T+2 (the parent, i.e., the closest-window commitment block) can:

1. Observe that a competing block at height T−(farthest−closest) = T−8 (mainnet: farthest=10, closest=2) already proposed transaction P.
2. Include P in block T's proposal set (redundant but valid — `TwoPhaseCommitVerifier` only checks that committed transactions were proposed somewhere in the window, not who proposed first).
3. Commit P in block T+2.

`proposal_reward(parent=T+2, target=T)` then:
- Phase 1: finds P committed in T+2, removes it from `target_proposals`, adds 40% fee to reward.
- While loop: `target_proposals.is_empty()` → exits immediately.
- Block T−8's earlier proposal is never loaded into `proposed`.
- Block T receives the full 40% proposal reward; block T−8 receives nothing.

The miner of T−8 loses their deserved reward. The miner of T receives an overpayment equal to `tx_fee × 40%` for every such transaction. For high-fee transactions this is a direct, repeatable economic gain.

---

### Likelihood Explanation

- **No special privilege required.** Any miner can include already-proposed transactions in their own block's proposal set; `TwoPhaseCommitVerifier` does not reject this.
- **Mining requirement is modest.** The attacker needs to mine block T and block T+2 (one intervening block mined by anyone). With mainnet `closest=2`, these two blocks are only 2 apart. A miner with even a small fraction of hash power will occasionally mine consecutive or near-consecutive blocks.
- **Deterministic trigger.** The attacker fully controls whether P is committed in the parent block (by including it in their own T+2 block). There is no probabilistic element beyond mining the two blocks.
- **Repeatable.** The same technique applies to every high-fee pending transaction that has already been proposed by an earlier block.

---

### Recommendation

Before Phase 1 awards reward for transactions committed in the parent block, compute and check the same `competing_proposal_start` that the while loop would use for the parent block, and populate `proposed` accordingly:

```rust
// Before the pre-loop scan:
let parent_competing_proposal_start =
    cmp::max(parent.number().saturating_sub(proposal_window.farthest()), 1);
if let Some(hash) = store.get_block_hash(parent_competing_proposal_start) {
    proposed.extend(self.get_proposal_ids_by_hash(&hash));
}

// Then in the pre-loop scan, mirror the while-loop guard:
if target_proposals.remove(&id) && !proposed.contains(&id) {
    reward = reward.safe_add(tx_fee.safe_mul_ratio(proposer_ratio)?)?;
}
```

This makes Phase 1 and Phase 2 symmetric with respect to competing-proposer detection.

### Proof of Concept

**Chain setup** (ProposalWindow(2, 10), mainnet parameters):

| Height | Block | Action |
|--------|-------|--------|
| T−8 | Competing block | Proposes P (earliest valid proposer for commitment at T+2) |
| T | Target block (attacker) | Also proposes P |
| T+1 | Any block | Empty |
| T+2 | Parent block (attacker) | Commits P |
| T+11 | Current block | Finalizes reward for T |

**Expected result:** `proposal_reward` for target T should return 0 for P (competing block T−8 was earlier).

**Actual result:** Phase 1 finds P committed in T+2, removes it from `target_proposals` (now empty), adds `fee × 40%` to reward. While loop exits immediately. `proposed` is never populated. Block T−8's proposal is never seen. Target T receives the full 40% proposal reward.

**Assertion that fails:**
```rust
let reward = reward_calculator.proposal_reward(&block_t_plus_2.header(), &block_t.header()).unwrap();
assert_eq!(reward, Capacity::zero()); // FAILS: reward == fee * 40%
```

The `proposed` set is declared empty at line 199 and is only ever extended inside the while loop body (line 252), never before Phase 1. The while loop guard at line 238 (`!target_proposals.is_empty()`) causes the loop — and thus all competing-proposer detection — to be skipped entirely when Phase 1 drains `target_proposals`.
