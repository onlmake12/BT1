### Title
Inconsistent `total_tx_size`/`total_tx_cycles` Accounting When Cell-Dep Evictions Occur During `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new pool-size statistics are pre-computed **before** any evictions take place. When `check_and_record_ancestors` evicts cell-dep-referencing transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, those decrements are applied to `self.total_tx_size` / `self.total_tx_cycles`. Immediately afterward, the pre-computed (pre-eviction) values are unconditionally written back, silently discarding the eviction-adjusted decrements. The counters are permanently inflated by the aggregate size and cycles of every evicted transaction.

---

### Finding Description

In `add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221):

```
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // (1) snapshot T + new_size
// ...
evicts = self.check_and_record_ancestors(&mut entry)?;          // (2) may call remove_entry → update_stat_for_remove_tx
// ...
self.total_tx_size = total_tx_size;    // (3) OVERWRITES eviction-adjusted value
self.total_tx_cycles = total_tx_cycles;
```

Step (1) captures `total_tx_size = T + new_size` into a local variable.

Step (2) — inside `check_and_record_ancestors` — when the incoming transaction's ancestor count exceeds `max_ancestors_count` but can be reduced by evicting cell-dep-referencing parents, calls `remove_entry_and_descendants` for each evicted candidate. Each call reaches `remove_entry` (line 247), which calls `update_stat_for_remove_tx`, **decrementing** `self.total_tx_size` and `self.total_tx_cycles` in place.

Step (3) then **overwrites** those decremented values with the stale pre-eviction snapshot, erasing all the decrements applied in step (2). [1](#0-0) 

The eviction path that triggers this is in `check_and_record_ancestors` (lines 603–625): when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, the lowest-fee cell-dep-referencing parents are removed. [2](#0-1) 

`remove_entry` (line 247) is the site where `update_stat_for_remove_tx` is called, decrementing the counters that are then overwritten. [3](#0-2) 

---

### Impact Explanation

**Pool-size enforcement is broken.** `limit_size` evicts transactions whenever `total_tx_size > max_tx_pool_size`: [4](#0-3) 

With an inflated `total_tx_size`, this loop fires even when the pool has real capacity, evicting legitimate pending transactions with `Reject::Full`.

**New-transaction admission is broken.** `updated_stat_for_add_tx` rejects any incoming transaction whose addition would overflow the counter: [5](#0-4) 

An inflated baseline causes honest transactions to be rejected with `Reject::Full` even when the pool is not actually full.

**RPC reporting is incorrect.** `tx_pool_info` reads `total_tx_size` and `total_tx_cycles` directly: [6](#0-5) 

Callers (monitoring tools, wallets, fee estimators) receive inflated values, leading to incorrect fee-rate decisions.

---

### Likelihood Explanation

The trigger requires an unprivileged transaction sender to craft a scenario where:

1. A transaction **T1** is already in the pool.
2. Transactions **T2, T3, …** reference T1's outputs as **cell deps** (not inputs), making them `cell_ref_parents` of any future transaction that also descends from T1 via inputs.
3. A new transaction **T_new** is submitted whose input-chain ancestor count, combined with the cell-dep ancestors, exceeds `max_ancestors_count` (default 25), but whose pure input-chain ancestor count alone does not.

This is a realistic construction: cell-dep references to in-pool transactions are a normal CKB pattern (e.g., referencing a type-script cell that is itself being updated). No privileged access, no majority hashpower, and no external dependency is required — only the ability to submit transactions to the mempool via the standard RPC or P2P relay.

Each successful trigger inflates `total_tx_size` by the aggregate serialized size of the evicted transactions. Repeated submissions compound the inflation monotonically until the pool begins rejecting or evicting legitimate transactions.

---

### Recommendation

Compute the final statistics **after** all evictions have completed, rather than pre-computing them before. Replace the pre-computation pattern with a post-eviction adjustment:

```rust
// After check_and_record_ancestors returns `evicts`:
let evicted_size: usize  = evicts.iter().map(|e| e.size).sum();
let evicted_cycles: Cycle = evicts.iter().map(|e| e.cycles).sum();

// Now compute the correct final totals
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx_net(entry.size, entry.cycles, evicted_size, evicted_cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, move the `updated_stat_for_add_tx` call to after `check_and_record_ancestors` returns, so it reads the already-decremented `self.total_tx_size` / `self.total_tx_cycles` as its baseline.

### Proof of Concept

**Setup:**
- `max_ancestors_count = 25` (default).
- Submit 24 transactions T1 → T2 → … → T24 as a linear input chain (T_i spends T_{i-1}'s output). All are in the pool.
- Submit transaction T_dep that takes T1's output as a **cell dep** (not input). T_dep is now a `cell_ref_parent` of any future transaction that also descends from T1.

**Trigger:**
- Submit T_new that spends T24's output (giving it 25 input-chain ancestors: T1…T24 + itself) **and** lists T_dep's output as a cell dep.
- `ancestors_count = 25 + 1 = 26 > max_ancestors_count = 25`.
- `cell_ref_parents = {T_dep}`, so `26 - 1 = 25 <= 25` → eviction path is taken.
- `remove_entry_and_descendants(T_dep)` is called → `update_stat_for_remove_tx(T_dep.size, T_dep.cycles)` decrements `self.total_tx_size`.
- Back in `add_entry`, line 218 overwrites `self.total_tx_size` with the pre-eviction snapshot `T + T_new.size`, which does **not** subtract `T_dep.size`.

**Observable effect:**
- `tx_pool_info` reports `total_tx_size` inflated by `T_dep.size` (and any of its descendants).
- Repeating this pattern N times inflates `total_tx_size` by `N × T_dep.size`.
- Once `total_tx_size` exceeds `max_tx_pool_size`, `limit_size` begins evicting legitimate transactions with `Reject::Full`, and new honest transactions are rejected even though the pool has real capacity.
