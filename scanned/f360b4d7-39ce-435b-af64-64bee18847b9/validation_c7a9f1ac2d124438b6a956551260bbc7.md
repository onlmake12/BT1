### Title
Diverging `total_tx_cycles` Running Total Causes Incorrect Tx-Pool Admission Control and RPC Reporting - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap` maintains a running total `total_tx_cycles` that is incremented when a transaction is added and decremented when it is removed. The code itself acknowledges that the cycles value used for removal may differ from the value used for addition, causing the running total to permanently diverge from the true sum of individual entry cycles. This is the direct CKB analog to the Gearbox `totalBorrowed` divergence: a pool-level aggregate counter grows at a different rate than the sum of its constituent parts, leading to incorrect downstream calculations.

---

### Finding Description

`PoolMap` tracks two pool-wide running totals: [1](#0-0) 

These are incremented via `updated_stat_for_add_tx` and decremented via `update_stat_for_remove_tx`. The decrement function carries an explicit acknowledgment of the divergence: [2](#0-1) 

The comment **"cycles overflow is possible, currently obtaining cycles is not accurate"** directly states that the cycles value used at removal time (`entry.inner.cycles`) may not match the value that was used to increment `total_tx_cycles` at insertion time. This mismatch arises because a transaction's cycles can be updated between insertion and removal (e.g., after re-verification following a chain reorg or snapshot refresh), while `total_tx_cycles` was already incremented with the original value. Each such mismatch permanently biases the running total upward or downward.

The increment path uses `checked_add` and returns `Reject::Full` on overflow: [3](#0-2) 

If `total_tx_cycles` is inflated above its true value (because additions used higher cycle estimates than the values later subtracted), it will approach `u64::MAX` over time. Once it does, every subsequent `checked_add` fails and the pool rejects all new transactions with `Reject::Full`, regardless of actual pool state.

The incorrect value is also directly exposed to external callers via the `tx_pool_info` RPC: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

1. **Pool admission DoS**: If `total_tx_cycles` is inflated (additions used higher cycle counts than removals subtract), it drifts toward `u64::MAX`. Once near the ceiling, `updated_stat_for_add_tx`'s `checked_add` returns `Reject::Full` for every new submission, effectively halting the mempool from accepting any transaction — a complete tx-pool DoS without requiring any privileged access.

2. **Incorrect RPC reporting**: `tx_pool_info` returns a `total_tx_cycles` that does not reflect the actual sum of cycles across pool entries. Miners and fee-estimation tools that rely on this value to gauge pool saturation receive incorrect data, potentially leading to suboptimal block assembly or fee misjudgment.

---

### Likelihood Explanation

Any scenario where a transaction's cycles value is updated in its `TxEntry` after the entry was inserted (and `total_tx_cycles` was incremented with the original value) produces a permanent bias. Chain reorgs trigger snapshot refreshes and re-admission of transactions; if re-verification produces a different cycle count, the mismatch accumulates. An unprivileged transaction submitter can trigger this by submitting transactions whose script execution cycles vary across verification contexts (e.g., transactions whose cycle cost depends on chain state that changes during a reorg). The attacker needs no special privileges — only the ability to submit transactions to the RPC or P2P relay.

---

### Recommendation

1. Store the exact cycles value used to increment `total_tx_cycles` alongside the entry (or re-use `entry.inner.cycles` consistently and ensure it is never mutated after insertion without a corresponding correction to `total_tx_cycles`).
2. When a transaction's cycles are updated post-insertion, apply a delta correction to `total_tx_cycles` atomically: `total_tx_cycles = total_tx_cycles - old_cycles + new_cycles`.
3. Consider replacing the `checked_add` overflow guard with a comparison against a configured maximum cycle budget, rather than relying on `u64::MAX` as the sentinel, so that pool fullness is bounded by policy rather than arithmetic limits.

---

### Proof of Concept

1. Submit transaction `T` to the tx-pool. At insertion, `T.cycles = 1000`; `total_tx_cycles += 1000`.
2. A chain reorg occurs. `T` is re-admitted after re-verification with `T.cycles = 800` (the entry is updated in place).
3. `T` is committed in a block and removed: `update_stat_for_remove_tx(T.size, 800)` → `total_tx_cycles -= 800`.
4. Net effect: `total_tx_cycles` is permanently inflated by `200` per such cycle.
5. Repeat with many transactions. `total_tx_cycles` drifts toward `u64::MAX`.
6. Once `total_tx_cycles` is near `u64::MAX`, `updated_stat_for_add_tx` returns `Reject::Full` for every new submission, halting the pool. [6](#0-5) [7](#0-6)

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
