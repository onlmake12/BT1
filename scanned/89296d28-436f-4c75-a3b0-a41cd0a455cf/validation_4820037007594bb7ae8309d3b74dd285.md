Audit Report

## Title
Missing `BLOCK_INVALID` Early-Return in `HeaderAcceptor::accept()` Allows Known-Invalid Headers to Shadow DB Status via `header_map` — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`ValidationState::Valid` is the Rust `#[default]` variant, meaning a freshly created `ValidationResult` is indistinguishable from an explicitly accepted one. A FIXME comment at line 301–302 of `headers_process.rs` explicitly acknowledges a missing early-return guard for `BLOCK_INVALID` status. After a node restart (which clears both in-memory maps), a peer can send the header of any previously-rejected block; the header passes all three structural checks and is inserted into `header_map`, which has higher lookup priority than the DB, permanently shadowing the persisted `BLOCK_INVALID` status for the session and causing the node to re-request and re-process the full block.

## Finding Description

**Root cause — `ValidationState::Valid` as default:**
`ValidationState` uses `Valid` as its `#[default]` variant (lines 361–367 of `headers_process.rs`). `ValidationResult::default()` therefore starts as `Valid` with no validation performed. There is no `Uninitialized` sentinel to distinguish "not yet evaluated" from "explicitly accepted."

**The unguarded FIXME path:**
`HeaderAcceptor::accept()` (lines 295–358) creates `result = ValidationResult::default()` (i.e., `Valid`), reads the block status, and the FIXME comment at lines 301–302 explicitly acknowledges the missing guard:
```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
```
The only early-return guard that follows checks `status.contains(BlockStatus::HEADER_VALID)` — not `BLOCK_INVALID`. A block stored in the DB with `verified = Some(false)` returns `BLOCK_INVALID` from `get_block_status`, which does not satisfy `HEADER_VALID`, so execution falls through to `prev_block_check`, `non_contextual_check`, and `version_check`. A structurally valid header for a previously-rejected block (e.g., one that failed script execution) passes all three. `insert_valid_header` is then called and the function returns `ValidationResult { state: Valid }`.

**Status lookup priority — `header_map` shadows the DB:**
`get_block_status` (lines 425–444 of `shared/src/shared.rs`) resolves in this order:
1. `block_status_map` (in-memory, cleared on restart)
2. `header_map` (in-memory, cleared on restart)
3. DB (`block_ext.verified`)

After a node restart both in-memory maps are empty. A block stored in the DB with `verified = Some(false)` returns `BLOCK_INVALID`. Once `insert_valid_header` (line 1129 of `sync/src/types/mod.rs`) inserts the header into `header_map`, subsequent calls to `get_block_status` return `HEADER_VALID` — the DB-backed `BLOCK_INVALID` is permanently shadowed for the session. `insert_valid_header` writes only to `header_map` and never touches `block_status_map`, so the DB entry is never cleared but is also never consulted again.

**Downstream consequences:**
- `contextual_check` in `compact_block_process.rs` (lines 259–260) rejects a compact block only if `status.contains(BlockStatus::BLOCK_INVALID)`. After re-insertion, status is `HEADER_VALID`, so this guard is bypassed and the compact block is processed.
- `asynchronous_process_remote_block` in `synchronizer/mod.rs` (lines 477–478) accepts the full block when `status.contains(BlockStatus::HEADER_VALID)`, calling `accept_remote_block` and re-queuing the block for chain processing.
- `insert_valid_header` also calls `may_set_best_known_header` and `may_set_shared_best_header`, updating the node's sync state to reflect a chain that includes a known-invalid block.

## Impact Explanation
This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**. The `header_map` in-memory store is consulted before the DB in `get_block_status`, but `insert_valid_header` does not check whether the block is already known-invalid in the DB before inserting. This allows the in-memory state to silently override a persisted `BLOCK_INVALID` status, causing the node to waste CPU, memory, and bandwidth re-processing known-invalid blocks, and temporarily corrupting `shared_best_header` and per-peer best-known-header state until the block fails full validation again.

## Likelihood Explanation
Medium. The precondition — a block stored in the DB with `verified = Some(false)` — arises naturally during async block processing when a block fails full validation after being stored. A node restart is a routine operational event. Any peer that observed the original block can replay its header. The FIXME comment in production code at lines 301–302 explicitly confirms the developers are aware the path is reachable but have not resolved it. No special privileges are required; any unprivileged P2P peer can send a `SendHeaders` message.

## Recommendation
1. **Fix the FIXME** in `HeaderAcceptor::accept()` by adding an early return when `status.contains(BlockStatus::BLOCK_INVALID)`:
```rust
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
```
2. **Add an `Uninitialized` sentinel** to `ValidationState` and make it the `#[default]`, so a freshly created `ValidationResult` is never silently treated as a success.
3. **Guard `insert_valid_header`** to check `get_block_status` before inserting into `header_map`, or have it also write `BLOCK_INVALID` to `block_status_map` if the DB status is `BLOCK_INVALID`.

## Proof of Concept
1. Run a CKB node and let it process a block that fails full validation (e.g., a block with an invalid script). The block is stored in the DB with `block_ext.verified = Some(false)`.
2. Restart the node. `block_status_map` and `header_map` are now empty.
3. Connect as an unprivileged peer and send a `SendHeaders` P2P message containing the header of the rejected block.
4. `HeadersProcess::execute()` calls `HeaderAcceptor::accept()`. `get_block_status` returns `BLOCK_INVALID` (from DB). The FIXME path is taken: no early return. `prev_block_check`, `non_contextual_check`, and `version_check` all pass for a structurally valid header. `insert_valid_header` is called.
5. Call `get_block_status` for the block hash. Observe it now returns `HEADER_VALID` instead of `BLOCK_INVALID`.
6. Send a compact block for the same block. `contextual_check` no longer rejects it (the `BLOCK_INVALID` guard is bypassed). The node re-processes the block, wasting resources and temporarily corrupting its sync state. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
