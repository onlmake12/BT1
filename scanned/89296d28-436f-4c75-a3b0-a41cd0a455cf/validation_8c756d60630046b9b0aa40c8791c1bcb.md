Audit Report

## Title
Off-by-one in `ConnectionRequestProcess::execute` allows attacker to cause legitimate relay nodes to be banned by their peers — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
Both route-length guards in `execute()` use strict `>`, so a crafted `ConnectionRequest` with `route.len() = MAX_HOPS` and `max_hops = MAX_HOPS` passes all checks on Node A. `forward_request` then appends Node A's ID to the route, producing `route.len() = 7`. Node B receives this structurally invalid message, returns `InvalidRoute` (status 413), and bans Node A for 24 hours. An attacker with a single P2P connection to Node A can systematically sever Node A's connections to all peers it would forward to.

## Finding Description
`MAX_HOPS` is defined as `6`: [1](#0-0) 

Both guards in `execute()` use strict `>`: [2](#0-1) 

A crafted message with `route.len() = 6` and `max_hops = 6` passes both checks (`6 > 6 = false`). The node then calls `forward_message` → `forward_request`, which unconditionally pushes `current_id` onto the route and decrements `max_hops`: [3](#0-2) 

The forwarded message now has `route.len() = 7`. Node B receives it, evaluates `7 > 6 = true`, and returns `StatusCode::InvalidRoute` (413). The `should_ban()` method maps any 4xx code to a 24-hour ban: [4](#0-3) 

Node B then bans the session of Node A: [5](#0-4) 

The invariant `route.len() + max_hops <= MAX_HOPS` is never enforced. A legitimately-initialized message always satisfies it (starts at `route=[], max_hops=6`), but an attacker can violate it freely.

## Impact Explanation
An attacker with a single P2P connection can cause any relay node (Node A) to be banned by any of its forwarding peers (Node B) for 24 hours (`BAD_MESSAGE_BAN_TIME`). By repeating this across different `(from, to)` pairs — which the attacker controls in the message fields — the attacker can exhaust Node A's peer connections, effectively isolating it from the network. Scaled across multiple relay nodes, this degrades the CKB P2P topology. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

## Likelihood Explanation
The attacker only needs a single P2P connection to a relay node. No proof-of-work, no keys, no privileged role. The crafted message is trivially constructable by setting `route` to 6 arbitrary peer IDs and `max_hops = 6`. The outer rate limiter (`rate_limiter`) allows 30 requests/second per session, but only one crafted message is needed per target (Node A, Node B) pair. The `forward_rate_limiter` is keyed by `(content.from, content.to, item_id)`, all of which the attacker controls by varying the `from`/`to` fields, making the attack repeatable at low cost.

## Recommendation
Change the route-length guard to account for the pending append before forwarding:

```rust
// Before forwarding, route will grow by 1; reject if it would exceed MAX_HOPS
if content.route.len() >= MAX_HOPS as usize {
    return StatusCode::InvalidRoute.with_context("route too long to forward");
}
```

This ensures that after `forward_request` appends `current_id`, the resulting `route.len()` is at most `MAX_HOPS`, which passes the next hop's check. [6](#0-5) 

## Proof of Concept

```rust
// Craft a ConnectionRequest with route.len() = MAX_HOPS and max_hops = MAX_HOPS
let route: Vec<PeerId> = (0..MAX_HOPS).map(|_| PeerId::random()).collect();
let msg = packed::ConnectionRequest::new_builder()
    .from(attacker_id.as_bytes())
    .to(target_id.as_bytes())
    .max_hops(MAX_HOPS)
    .route(/* 6 peer IDs */)
    .listen_addrs(/* valid addrs */)
    .build();

// Send to Node A. Node A passes both checks (6 > 6 = false).
// forward_request produces route.len()=7, max_hops=5.
// Node B receives it: route.len()=7 > MAX_HOPS=6 → InvalidRoute (413) → bans Node A for 24h.
let forwarded = forward_request(msg.as_reader(), &relay_a_id);
assert!(forwarded.route().len() > MAX_HOPS as usize); // 7 > 6 — next hop will ban
```

A unit test can be added to `network/src/protocols/hole_punching/component/mod.rs` alongside the existing `test_route` test, verifying that `forward_request` on a message with `route.len() = MAX_HOPS` produces a route that fails the guard at the next hop.

```rust
pub(crate) fn forward_request(
    request: packed::ConnectionRequestReader<'_>,
    current_id: &PeerId,
) -> packed::ConnectionRequest {
    let max_hops: u8 = request.max_hops().into();
    let message = request.to_entity();
    let new_route = message
        .route()
        .as_builder()
        .push(current_id.as_bytes())
        .build();
    message
        .as_builder()
        .max_hops(max_hops.saturating_sub(1))
        .route(new_route)
        .build()
}
```
