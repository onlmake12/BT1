Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Stale Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` snapshots prospective new totals via `updated_stat_for_add_tx` before calling `check_and_record_ancestors`, which may evict entries and correctly decrement `self.total_tx_size`/`self.total_tx_cycles`. The stale pre-eviction snapshot is then unconditionally written back, permanently inflating both accumulators by the size/cycles of every evicted entry. `limit_size` uses the inflated `total_tx_size` to drive eviction of legitimate pool entries, enabling a repeatable tx-pool denial-of-service by an unprivileged caller.

## Finding Description

In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

```rust
// Line 210-211: snapshot taken BEFORE evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: may call remove_entry_and_descendants → remove_entry
//           → update_stat_for_remove_tx → self.total_tx_size -= evicted.size
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218-219: stale snapshot overwrites the decremented values
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) computes `self.total_tx_size + entry.size` and returns it as a local. [1](#0-0) 

`check_and_record_ancestors` (lines 588–640), when `ancestors_count > max_ancestors_count` and the excess is attributable to `cell_ref_parents`, calls `remove_entry_and_descendants` in a loop. [2](#0-1) 

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` and `self.total_tx_cycles`. [3](#0-2) 

`update_stat_for_remove_tx` writes the decremented value directly into `self.total_tx_size`. [4](#0-3) 

Lines 218–219 then unconditionally overwrite `self.total_tx_size` with the pre-eviction snapshot, discarding those decrements. [5](#0-4) 

Net inflation per trigger cycle: `size(evicted entries)`. The inflation is permanent — `recompute_total_stat` is only invoked on underflow, never on this overwrite path.

`limit_size` drives eviction purely from the inflated accumulator: [6](#0-5) 

Once the phantom total exceeds `max_tx_pool_size`, `limit_size` evicts real pending/proposed transactions even though the actual pool byte count is within limits.

## Impact Explanation

Repeated triggering inflates `total_tx_size` without bound. Once the phantom total crosses `max_tx_pool_size`, `limit_size` continuously evicts legitimate transactions, effectively emptying the pool and preventing re-admission of honest users' transactions. This constitutes a **tx-pool denial-of-service** against a targeted CKB node, matching the allowed High impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

**Severity: High (10001–15000 points)**

## Likelihood Explanation

The trigger requires:
1. A chain of ≥ `max_ancestors_count − 1` (default: 24) transactions in the pool.
2. At least one in-pool transaction using a chain output as a **cell dep** (`cell_ref_parent`).
3. A new transaction consuming that same output as an **input**, pushing `ancestors_count` over the limit.

All three conditions are constructible with normal CKB transactions via the unprivileged `send_transaction` RPC. The attacker needs CKB to fund ~26 transactions per cycle; the attack is repeatable with fresh outputs. No privileged access, no victim mistakes, and no external dependencies are required.

**Likelihood: Medium**

## Recommendation

Move the stat computation to after `check_and_record_ancestors` completes, so it bases the addition on the post-eviction `self.total_tx_size`:

```rust
// In add_entry:
evicts = self.check_and_record_ancestors(&mut entry)?;  // evictions happen first
// Now self.total_tx_size reflects evictions; compute the addition on the current value:
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, add a debug-build invariant assertion after every `add_entry` that `total_tx_size == entries.iter().map(|e| e.inner.size).sum()`.

## Proof of Concept

**Configuration:** `max_ancestors_count = 25`, `max_tx_pool_size = 20 MB`

**Steps:**
1. Submit `tx_0` creating `cell_0`.
2. Submit `tx_1 … tx_24`, each spending the previous output (chain of 24 descendants of `tx_0`).
3. Submit `tx_dep` using `tx_24`'s output as a **cell dep** — it becomes a `cell_ref_parent` for any future tx spending that output.
4. Submit `tx_25` spending `tx_24`'s output as an **input**:
   - `get_tx_ancenstors` returns 25 ancestors + `tx_dep` as `cell_ref_parent` → `ancestors_count = 26 > 25`.
   - `check_and_record_ancestors` evicts `tx_dep` via `remove_entry_and_descendants` → `self.total_tx_size -= size(tx_dep)`.
   - `add_entry` writes back `total_tx_size = old_total + size(tx_25)` (stale snapshot, ignoring the subtraction).
   - **Net inflation: `+size(tx_dep)`**.
5. Repeat steps 3–4 with fresh outputs. Each iteration inflates `total_tx_size` by `size(tx_dep)`.
6. Once the phantom `total_tx_size` exceeds `max_tx_pool_size`, `limit_size` evicts all real transactions from the pool, even though the actual pool byte count is well within limits.

            ))
        })?;

    }
