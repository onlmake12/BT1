Audit Report

## Title
Missing Count Sanity Check on `RelayTransactions` Allows Unbounded Pre-Filter Deserialization — (`sync/src/relayer/transactions_process.rs`)

## Summary

`TransactionsProcess::execute()` iterates over all transactions in a `RelayTransactions` P2P message and fully deserializes each one via `.to_entity().into_view()` before any filtering occurs, with no upper-bound check on the transaction count. Every other relay message handler in the same dispatch loop enforces `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) before doing any work. Any connected peer can exploit this to force O(n) deserialization work — including hash computation — on the receiving node at the rate-limiter ceiling of 30 messages per second.

## Finding Description

In `sync/src/relayer/mod.rs` the dispatcher routes `RelayTransactions` to `TransactionsProcess` after only a molecule structural check (`check_data()`): [1](#0-0) 

`TransactionsProcess::execute()` then immediately chains `.map()` (full deserialization) before `.filter()` (hash-based filtering), with no count guard anywhere in the path: [2](#0-1) 

Because `.map()` is evaluated lazily but `.collect()` forces the entire iterator, every transaction in the message is fully deserialized — including `into_view()` which computes the transaction hash — before the filter discards those not in `unknown_tx_hashes`. An attacker sending transactions that were never requested by the node will have all of them deserialized and then dropped, maximizing wasted work.

All sibling handlers enforce `MAX_RELAY_TXS_NUM_PER_BATCH` before processing: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

`TransactionsProcess` is the sole handler in this dispatch loop with no such guard. The relevant constants: [7](#0-6) 

The rate limiter is keyed per `(PeerIndex, message_type)` at 30 requests per second: [8](#0-7) 

## Impact Explanation

A connected peer can send `RelayTransactions` messages containing the maximum number of transactions fitting within the P2P frame size. The node performs O(n) molecule validation (`check_data()`) followed by O(n) full deserialization (including hash computation via `into_view()`) for every message, with no count guard to short-circuit. At 30 messages/second per peer, and with multiple peers, this scales linearly into sustained CPU and memory pressure. Under sufficient peer count this can degrade or crash a CKB node's ability to process legitimate blocks and transactions.

**Impact: High** — matches "Vulnerabilities which could easily crash a CKB node" and "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation

Any peer that completes the standard P2P handshake can send `RelayTransactions` messages. No privileged key, majority hashpower, or social engineering is required. The attack is trivially scriptable: construct a molecule-valid `RelayTransactions` message with the maximum number of minimal-size transactions and send it at the rate-limiter ceiling. The rate limiter provides only partial mitigation — it caps per-peer throughput but does not eliminate the attack surface, and multiple peers multiply the load linearly.

**Likelihood: Medium** — requires only a connected peer; rate limiter provides partial but insufficient mitigation.

## Recommendation

Add a count check at the top of `TransactionsProcess::execute()`, consistent with all sibling handlers:

```rust
pub fn execute(self) -> Status {
    let tx_count = self.message.transactions().len();
    if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
        return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
            "Transactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})"
        ));
    }
    // ... existing logic
}
```

Import `MAX_RELAY_TXS_NUM_PER_BATCH` from `crate::relayer` as the sibling handlers do.

## Proof of Concept

1. Connect to a CKB node as a peer (complete the P2P handshake).
2. Construct a molecule-valid `RelayTransactions` message containing `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) minimal `RelayTransaction` entries. Each entry requires only a minimal transaction body and a cycles field (~88 bytes each, ~2.9 MB total).
3. Send this message at 30 messages/second (the rate-limiter ceiling for this `(peer, message_type)` key).
4. The target node executes `check_data()` (O(n) molecule validation) then `TransactionsProcess::execute()` (O(n) full deserialization via `to_entity().into_view()`) for every message. Since the crafted transactions were never requested by the node, `unknown_tx_hashes.get_priority()` returns `None` for all of them, `unwrap_or_default()` returns `false`, and all are discarded — but only after full deserialization.
5. Observe elevated CPU usage on the target node proportional to transactions-per-message × messages-per-second. Scale with additional peers to amplify.
