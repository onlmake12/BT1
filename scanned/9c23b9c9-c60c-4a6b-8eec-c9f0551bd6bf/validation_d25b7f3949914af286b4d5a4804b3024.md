### Title
`max_tx_verify_cycles` Upper Bound Not Enforced for Local RPC Transaction Submissions - (File: `tx-pool/src/process.rs`)

---

### Summary

The `TxPoolConfig.max_tx_verify_cycles` parameter is documented as the per-transaction cycle limit to prevent DoS, but it is never applied as a hard rejection bound in `_process_tx`. For locally submitted transactions (via `send_transaction` RPC), the verification limit silently falls back to `consensus.max_block_cycles()`, which is 50× larger than the default configured limit. An unprivileged RPC caller can exploit this to force the node to spend far more CPU than the operator intended.

---

### Finding Description

In `tx-pool/src/process.rs`, the core transaction processing function `_process_tx` selects the cycle limit for script verification as follows:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```

`declared_cycles` is `Some(cycles)` only for remote peer-relayed transactions. For all locally submitted transactions (RPC path `send_transaction` → `process_tx` → `_process_tx` with `remote = None`), `declared_cycles` is `None`, so `max_cycles` becomes `consensus.max_block_cycles()`. [1](#0-0) 

The `TxPoolConfig` struct documents `max_tx_verify_cycles` explicitly as a rejection guard:

```rust
/// tx pool rejects txs that cycles greater than max_tx_verify_cycles
pub max_tx_verify_cycles: Cycle,
``` [2](#0-1) 

The default value is `TWO_IN_TWO_OUT_CYCLES * 20 = 70,000,000` cycles: [3](#0-2) 

The consensus `max_block_cycles` is `TWO_IN_TWO_OUT_CYCLES * 1_000 = 3,500,000,000` cycles — **50× larger**: [4](#0-3) 

In practice, `max_tx_verify_cycles` is only used as a scheduling threshold (`large_cycle_threshold`) inside the verify queue to classify transactions for ordering, not as a hard admission bound: [5](#0-4) 

The CHANGELOG confirms `max_tx_verify_cycles` was introduced specifically to reduce DoS vulnerability by limiting per-transaction cycles. That intent is not realized for the local RPC path.

---

### Impact Explanation

A node operator sets `max_tx_verify_cycles = 70_000_000` (default) expecting the tx-pool to reject any single transaction that would consume more than 70M cycles of script verification CPU. Instead, any transaction submitted via the `send_transaction` RPC is verified against the full `max_block_cycles = 3,500,000,000` limit. An attacker with access to the RPC endpoint can submit transactions whose scripts consume close to 3.5B cycles, causing the node to spend ~50× more CPU per transaction than the operator configured. This degrades node responsiveness, delays block assembly, and can starve the verify worker threads.

---

### Likelihood Explanation

CKB nodes commonly expose the RPC endpoint publicly (the default `ckb.toml` binds RPC to `0.0.0.0`). Any unprivileged network peer that can reach the RPC port can call `send_transaction`. No key, privilege, or social engineering is required. The attacker only needs to craft a transaction with a script that loops near the cycle limit — a straightforward operation given CKB-VM's deterministic cycle accounting. [6](#0-5) 

---

### Recommendation

In `_process_tx`, when `declared_cycles` is `None` (local submission), use `min(consensus.max_block_cycles(), tx_pool_config.max_tx_verify_cycles)` as the verification limit, or reject the transaction outright if its actual verified cycles exceed `max_tx_verify_cycles`. This restores the documented and originally intended behavior of the configuration parameter.

---

### Proof of Concept

1. Start a CKB node with default config (`max_tx_verify_cycles = 70_000_000`, `max_block_cycles = 3_500_000_000`).
2. Deploy a lock script that executes a tight loop consuming ~3,000,000,000 cycles before returning success.
3. Submit a transaction spending a cell locked by that script via `send_transaction` RPC.
4. Observe: the node's verify worker thread runs the script for ~3B cycles (≈ seconds of CPU), far exceeding the 70M-cycle limit the operator configured.
5. Repeat with multiple concurrent submissions to exhaust the verify worker pool and stall block assembly.

The root cause is confirmed at: [1](#0-0) 

with the missing guard being the absence of a comparison against `self.tx_pool_config.max_tx_verify_cycles` before or instead of falling back to `self.consensus.max_block_cycles()`.
