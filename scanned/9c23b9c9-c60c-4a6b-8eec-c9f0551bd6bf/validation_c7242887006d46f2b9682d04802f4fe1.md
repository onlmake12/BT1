### Title
Silent Zero Miner-Fee Reward When Blocks Are Imported With `--skip-all-verify` Due to Empty `txs_fees` in `BlockExt` - (File: `util/reward-calculator/src/lib.rs`)

---

### Summary

When blocks are imported via `ckb import --skip-all-verify`, `BlockExt.txs_fees` is stored as an empty `vec![]` because `insert_ok_ext` is called with `cache_entries = None`. Later, when `RewardCalculator::proposal_reward` and `RewardCalculator::txs_fees` compute rewards for those blocks, Rust's `zip` iterator silently truncates to zero iterations when `txs_fees` is empty — producing a zero reward with no error or warning. Miners who proposed or committed transactions in those imported blocks lose their deserved fee rewards.

---

### Finding Description

**Step 1 — Missing prerequisite: `txs_fees` is never stored for `--skip-all-verify` blocks.**

In `chain/src/verify.rs`, every block starts with a `BlockExt` whose `txs_fees` is an empty vector: [1](#0-0) 

When `switch.disable_all()` is true, the code skips the full verification path and calls `insert_ok_ext` with `cache_entries = None` and `txs_sizes = None`: [2](#0-1) 

Inside `insert_ok_ext`, `ext.txs_fees` is only populated when `cache_entries` is `Some`. When it is `None`, the field stays as the initial empty `vec![]` and is persisted to the database: [3](#0-2) 

**Step 2 — Entry path: `ckb import --skip-all-verify` sets `Switch::DISABLE_ALL`.**

The `ckb import` subcommand explicitly sets `switch = Switch::DISABLE_ALL` when the `--skip-all-verify` flag is passed: [4](#0-3) 

`Switch::DISABLE_ALL` is defined to include every disable bit, including `DISABLE_SCRIPT`: [5](#0-4) 

**Step 3 — Silent failure: `zip` truncates silently when `txs_fees` is empty.**

`RewardCalculator::proposal_reward` reads `txs_fees` from the stored `BlockExt` via `txs_fees_proc`: [6](#0-5) 

It then pairs `committed_idx` (non-empty, one entry per non-cellbase tx) with `txs_fees_proc(...)` using `zip`: [7](#0-6) [8](#0-7) 

Rust's `zip` silently truncates to the shorter iterator. When `txs_fees` is `vec![]`, the zip produces **zero iterations** — `reward` stays at `Capacity::zero()` and no error is returned. The same silent truncation applies to `txs_fees()`: [9](#0-8) 

**Step 4 — Downstream enforcement locks in the wrong reward.**

`RewardVerifier::verify` enforces that the cellbase output capacity equals `block_reward.total` computed by `RewardCalculator`: [10](#0-9) 

Because `block_reward.total` is computed with zero `txs_fees` and zero `proposal_reward`, the finalization-block miner is forced to claim a lower cellbase amount or be rejected. The original miner's fee rewards are permanently lost.

---

### Impact Explanation

Any miner who proposed or committed transactions in a block that was imported with `ckb import --skip-all-verify` loses:
- **60% of tx fees** (committed reward, via `txs_fees`)
- **40% of tx fees** (proposal reward, via `proposal_reward`)

These are silently zeroed with no error, no log warning, and no way for the affected miner to detect or recover the loss. The `RewardVerifier` then enforces the wrong (lower) reward on the finalization block, making the loss permanent and consensus-enforced.

---

### Likelihood Explanation

`ckb import --skip-all-verify` is a documented, supported production CLI feature intended for fast-syncing from a trusted source. Any operator who uses it on a chain with fee-bearing transactions will silently corrupt the `txs_fees` of every imported block. The finalization delay (`finalization_delay_length`, typically 11 blocks on mainnet) means the damage surfaces only when those blocks become reward targets — well after the import completes, making it hard to diagnose.

---

### Recommendation

1. **In `insert_ok_ext`**: When `cache_entries` is `None` (i.e., `switch.disable_all()` path), recompute `txs_fees` by iterating the block's transactions and computing fees from the store, or at minimum store a sentinel that downstream code can detect.

2. **In `RewardCalculator::proposal_reward` and `txs_fees`**: Assert or return an error when `txs_fees` is empty but the block has non-cellbase transactions. This converts the silent truncation into a detectable failure.

3. **In `ckb import`**: Document that `--skip-all-verify` corrupts `txs_fees` and therefore invalidates miner fee rewards for imported blocks, or refuse to use `DISABLE_ALL` for blocks that contain non-cellbase transactions.

---

### Proof of Concept

```
# 1. Import a chain snapshot that contains fee-bearing transactions
ckb import --skip-all-verify --source chain.bin

# 2. The imported blocks now have BlockExt.txs_fees = vec![]
#    (verified by querying get_block_economic_state RPC — txs_fee will be 0)

# 3. When block N + finalization_delay_length is processed normally,
#    RewardCalculator reads txs_fees = [] for block N,
#    zip produces 0 iterations, proposal_reward = 0, txs_fees = 0

# 4. RewardVerifier enforces block_reward.total (which excludes all fees)
#    as the only valid cellbase amount — miner of block N loses all fee rewards
```

The analog to the Tigris `_handleOpenFees` bug is exact: a prerequisite state (`txs_fees` population, analogous to `approve`) is skipped in one code path (`disable_all` branch, analogous to `_handleOpenFees`), the downstream consumer (`zip`/`try_fold`, analogous to `transferFrom` in `try/catch`) silently produces zero instead of an error, and legitimate reward recipients (miners, analogous to NFT holders) lose deserved rewards.

```rust
    fn insert_ok_ext(
        &self,
        txn: &StoreTransaction,
        hash: &Byte32,
        mut ext: BlockExt,
        cache_entries: Option<&[Completed]>,
        txs_sizes: Option<Vec<u64>>,
    ) -> Result<(), Error> {
        ext.verified = Some(true);
        if let Some(entries) = cache_entries {
            let (txs_fees, cycles) = entries
                .iter()
                .map(|entry| (entry.fee, entry.cycles))
                .unzip();
            ext.txs_fees = txs_fees;
            ext.cycles = Some(cycles);
        }
        ext.txs_sizes = txs_sizes;
        txn.insert_block_ext(hash, &ext)
    }
```
