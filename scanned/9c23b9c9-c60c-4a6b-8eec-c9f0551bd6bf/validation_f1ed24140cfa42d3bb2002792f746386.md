### Title
Unbounded response amplification via duplicate indexes in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

### Summary
`GetBlockTransactionsProcess::execute` accepts up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32 767) transaction indexes from an untrusted peer but performs no deduplication and no byte-cap check on the assembled response. Sending 32 767 copies of index `0` causes the node to clone and serialize the same transaction (e.g., the cellbase) 32 767 times into a single `BlockTransactions` message, with no upper bound on the resulting byte size.

### Finding Description

The only guard present is a count check:

```rust
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
    return StatusCode::ProtocolMessageIsMalformed…
}
``` [1](#0-0) 

After passing that check, the handler maps every index directly to a transaction with no deduplication:

```rust
let transactions = self.message.indexes().iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
``` [2](#0-1) 

All 32 767 clones are then serialized into one message and sent:

```rust
let content = packed::BlockTransactions::new_builder()
    .block_hash(block_hash)
    .transactions(transactions.into_iter().map(|tx| tx.data()).collect::<Vec<_>>())
    …
    .build();
``` [3](#0-2) 

The sibling handlers `GetTransactionsProcess` and `GetBlockProposalProcess` both deduplicate with a `HashSet` and enforce `MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB): [4](#0-3) [5](#0-4) [6](#0-5) 

`GetBlockTransactionsProcess` imports only `MAX_RELAY_TXS_NUM_PER_BATCH` and never imports or uses `MAX_RELAY_TXS_BYTES_PER_BATCH`: [7](#0-6) 

### Impact Explanation

A CKB mainnet cellbase transaction with a full witness can easily be several kilobytes. At 32 767 copies × 4 KB = ~128 MB per response message. The rate limiter allows 30 messages/second per peer: [8](#0-7) 

With `MAX_RELAY_PEERS = 128` peers each sending 30 requests/second: [9](#0-8) 

- **Outbound bandwidth**: up to 128 × 30 × 128 MB ≈ hundreds of GB/s — far beyond any node's capacity.
- **Memory**: each in-flight response allocates the full cloned vector before serialization; concurrent requests cause OOM.
- **Block propagation**: the relay thread is saturated, starving legitimate compact-block reconstruction requests.

### Likelihood Explanation

- Requires only a valid P2P connection — no key, no PoW, no stake.
- The attacker needs only one stored block hash (trivially obtained from any public block explorer or by observing the network).
- The exploit is deterministic and repeatable; a single malicious peer can sustain it indefinitely within the rate limit.
- The rate limiter does not bound response *size*, only request *count*.

### Recommendation

1. **Deduplicate indexes** before lookup, mirroring `GetTransactionsProcess`:
   ```rust
   let indexes: HashSet<u32> = self.message.indexes().iter()
       .map(Into::<u32>::into).collect();
   if indexes.len() != self.message.indexes().len() {
       return StatusCode::RequestDuplicate.with_context("Duplicate indexes");
   }
   ```
2. **Enforce a byte cap** on the assembled response, aborting or splitting once the total serialized size exceeds `MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB), consistent with `GetTransactionsProcess` and `GetBlockProposalProcess`. [6](#0-5) 

### Proof of Concept

```
1. Connect to a CKB node over P2P (RelayV3 protocol).
2. Obtain any stored block hash H (e.g., from the genesis block).
3. Craft a GetBlockTransactions message:
     block_hash = H
     indexes    = [0] * 32767   # 32767 copies of index 0
     uncle_indexes = []
4. Send the message.
5. Observe: the node allocates and serializes 32767 copies of the cellbase
   transaction and sends a single BlockTransactions response.
6. Assert: response_bytes >> MAX_RELAY_TXS_BYTES_PER_BATCH (1 MiB).
7. Repeat at 30 req/s to sustain bandwidth/memory exhaustion.
```
