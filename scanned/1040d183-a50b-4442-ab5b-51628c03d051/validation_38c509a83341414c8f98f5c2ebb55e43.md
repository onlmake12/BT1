### Title
`ScriptHashTypeVerifier` Only Checks Lock Script `hash_type`, Skips Type Script — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` field of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the **type** script's `hash_type`. A transaction output carries two optional scripts (`lock` and `type_`); only one is checked. This is a direct structural analog to the reported finding: a verification routine checks only a subset of the components it is responsible for.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `ScriptHashTypeVerifier::verify()` loops over outputs and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

`output.type_()` returns `Option<Script>`. When it is `Some(script)`, the `hash_type` byte of that type script is **never validated** against `ENABLED_SCRIPT_HASH_TYPE`.

The permitted set is:

```
{0 = Data, 1 = Type, 2 = Data1, 4 = Data2}
``` [2](#0-1) 

`ScriptHashTypeVerifier` is wired into `NonContextualTransactionVerifier`, which is the first gate applied to every incoming transaction: [3](#0-2) 

---

### Impact Explanation

A transaction sender can craft an output whose **type script** carries a `hash_type` byte outside `{0,1,2,4}` (e.g., `3` or `5`). That transaction passes `NonContextualTransactionVerifier` without error. It is then forwarded to contextual verification / script execution, where `TryInto::<ScriptHashType>::try_into()` on the invalid byte returns an error and the transaction is ultimately rejected.

Concrete consequences:

1. **Tx-pool resource waste**: The transaction is admitted past the cheap non-contextual gate and consumes CPU and memory during the more expensive contextual verification pass before being evicted.
2. **Incomplete enforcement of consensus rules at the earliest checkpoint**: The non-contextual verifier is the canonical place to enforce structural validity. Leaving the type script unchecked means the invariant "all script hash types in a transaction are within the permitted set" is not enforced at the earliest possible point, creating a gap that could widen if future code paths branch on the non-contextual result without re-running contextual checks.
3. **Inconsistency with `CellbaseVerifier`**: The cellbase verifier does check `output.lock().hash_type()` for cellbase outputs (and separately asserts `type_` is absent), so the design intent is clearly to validate hash types at the non-contextual stage. [4](#0-3) 

---

### Likelihood Explanation

Any unprivileged transaction sender reachable via the RPC (`send_transaction`) or P2P relay can submit a crafted transaction with an invalid type script `hash_type`. No key material, mining power, or privileged access is required. The entry path is fully open.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate `output.type_()` when it is `Some`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        check_hash_type(output.lock().hash_type())?;
        // missing type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the pattern already used in `CellbaseVerifier` and closes the gap between what the verifier's doc-comment promises and what it actually enforces.

---

### Proof of Concept

1. Construct a `TransactionView` with one output whose `lock` has `hash_type = 0` (valid) and whose `type_` has `hash_type = 3` (not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Call `NonContextualTransactionVerifier::new(&tx, &consensus).verify()`.
3. Observe `Ok(())` — the verifier returns success despite the invalid type script hash type.
4. The transaction is admitted to the tx pool and only rejected later during contextual script execution, confirming the non-contextual gate is incomplete. [5](#0-4) [6](#0-5)
