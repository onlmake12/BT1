### Title
Missing IBD State Guard in `GetBlocksProcess` Allows Peers to Bypass IBD Isolation - (File: `sync/src/synchronizer/get_blocks_process.rs`)

---

### Summary

The CKB synchronizer module explicitly documents that a node in Initial Block Download (IBD) mode must respond with `InIBD` to both `GetHeaders` **and** `GetBlocks` requests. `GetHeadersProcess` correctly implements this guard, but `GetBlocksProcess` is missing the equivalent IBD check entirely. Any unprivileged inbound peer can send `GetBlocks` messages to an IBD node and receive full block data in response, violating the IBD isolation policy and wasting node resources during the critical sync phase.

---

### Finding Description

The module-level documentation for the synchronizer explicitly states the intended behavior:

> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests" [1](#0-0) 

`GetHeadersProcess::execute()` correctly implements this guard: it checks `active_chain.is_initial_block_download()`, sends an `InIBD` response back to the peer, and returns `Status::ignored()`. [2](#0-1) 

`GetBlocksProcess::execute()` has **no such check**. It proceeds directly to serve block data to any requesting peer without ever consulting the IBD state: [3](#0-2) 

Both handlers are dispatched from the same `try_process` match arm in `Synchronizer`: [4](#0-3) 

The `Relayer` protocol also correctly gates all relay messages behind an IBD check: [5](#0-4) 

The IBD state itself is a monotonic flag: once the node exits IBD it never re-enters, and the check is cheap. [6](#0-5) 

---

### Impact Explanation

During IBD, a CKB node is designed to:
1. Sync exclusively with one selected outbound/whitelist peer.
2. Ignore or reject requests from arbitrary inbound peers that would divert resources.

Because `GetBlocksProcess` lacks the IBD guard, any inbound peer can send `GetBlocks` messages and receive full serialized block bodies in response. This:

- **Violates IBD isolation**: the node serves block data to arbitrary peers while it should be focused solely on its own chain download.
- **Wastes CPU and bandwidth**: block serialization and network I/O are non-trivial during IBD when the node is already under heavy load.
- **Enables resource exhaustion**: an attacker can send up to `MAX_HEADERS_LEN` (2000) block hashes per message, causing the node to look up and serialize up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` blocks per request, repeatedly. [7](#0-6) 

---

### Likelihood Explanation

- Any unprivileged peer that establishes a P2P connection can send a `GetBlocks` message over the Sync protocol.
- CKB nodes accept inbound connections during IBD.
- No special privileges, keys, or majority hashpower are required.
- The attacker only needs to know the hashes of valid blocks (which are public on-chain data) to trigger maximum block-serving work.

---

### Recommendation

Add an IBD guard at the start of `GetBlocksProcess::execute()`, mirroring the pattern in `GetHeadersProcess`:

```rust
pub fn execute(self) -> Status {
    let active_chain = self.synchronizer.shared.active_chain();

    // Add this guard, matching the documented behavior and GetHeadersProcess
    if active_chain.is_initial_block_download() {
        // Optionally send InIBD response here
        return Status::ignored();
    }

    let block_hashes = self.message.block_hashes();
    // ... rest of existing logic
}
```

This matches the documented contract in `sync/src/synchronizer/mod.rs` and the existing pattern in `GetHeadersProcess`. [2](#0-1) 

---

### Proof of Concept

1. Node A is in IBD mode (tip timestamp is more than `MAX_TIP_AGE` behind wall clock).
2. Attacker peer B connects to Node A as an inbound peer.
3. Attacker B sends a `SyncMessage::GetBlocks` containing up to 2000 valid block hashes.
4. `Synchronizer::try_process` dispatches to `GetBlocksProcess::execute()`.
5. `GetBlocksProcess::execute()` has no IBD check — it iterates the hashes, looks up each block via `active_chain.get_block()`, serializes them, and sends `SendBlock` responses back to B.
6. Node A serves full block bodies to the attacker while it should be ignoring such requests and focusing on its own IBD sync.
7. Contrast: if B had sent `GetHeaders` instead, Node A would have responded with `InIBD` and returned `Status::ignored()` immediately. [3](#0-2) [8](#0-7)

```rust
    pub fn execute(self) -> Status {
        let block_hashes = self.message.block_hashes();
        // use MAX_HEADERS_LEN as limit, we may increase the value of INIT_BLOCKS_IN_TRANSIT_PER_PEER in the future
        if block_hashes.len() > MAX_HEADERS_LEN {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "BlockHashes count({}) > MAX_HEADERS_LEN({})",
                block_hashes.len(),
                MAX_HEADERS_LEN,
            ));
        }
        let active_chain = self.synchronizer.shared.active_chain();

        let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);

        let mut dedup = HashSet::new();
        for block_hash in iter {
            debug!("get_blocks {} from peer {:?}", block_hash, self.peer);
            let block_hash = block_hash.to_entity();

            if block_hash == self.synchronizer.shared().consensus().genesis_hash() {
                return StatusCode::RequestGenesis.with_context("Request genesis block");
            }

            if !dedup.insert(block_hash.clone()) {
                return StatusCode::RequestDuplicate.with_context("Request duplicate block");
            }

            if !active_chain.contains_block_status(&block_hash, BlockStatus::BLOCK_VALID) {
                debug!(
                    "Ignoring get_block {} request from peer={} as it is not verified.",
                    block_hash, self.peer
                );
                continue;
            }

            if let Some(block) = active_chain.get_block(&block_hash) {
                debug!(
                    "respond_block {} {} to peer {:?}",
                    block.number(),
                    block.hash(),
                    self.peer,
                );
                let content = packed::SendBlock::new_builder().block(block.data()).build();
                let message = packed::SyncMessage::new_builder().set(content).build();

                let nc = Arc::clone(self.nc);
                self.synchronizer
                    .shared()
                    .shared()
                    .async_handle()
                    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
            } else {
                // TODO response not found
                // TODO add timeout check in synchronizer

                // We expect that `block_hashes` is sorted descending by height.
                // So if we cannot find the current one from local, we cannot find
                // the next either.
                debug!("Stopping getblocks, since {} is not found", block_hash);
                break;
            }
        }

        Status::ok()
    }
```

```rust
    pub fn is_initial_block_download(&self) -> bool {
        // Once this function has returned false, it must remain false.
        if self.ibd_finished.load(Ordering::Acquire) {
            false
        } else if unix_time_as_millis().saturating_sub(self.snapshot().tip_header().timestamp())
            > MAX_TIP_AGE
        {
            true
        } else {
            self.ibd_finished.store(true, Ordering::Release);
            false
        }
    }
```
