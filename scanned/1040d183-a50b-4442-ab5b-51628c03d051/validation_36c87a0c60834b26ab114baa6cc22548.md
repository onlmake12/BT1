### Title
`get_block_filter` RPC Panics Instead of Returning `null` on Missing Filter Hash — (`File: rpc/src/module/chain.rs`)

---

### Summary

The `get_block_filter` RPC implementation in `ChainRpcImpl` violates its own interface contract. The `ChainRpc` trait declares the method as returning `Result<Option<BlockFilter>>`, and the documented behavior specifies returning `null` when data is unavailable. However, the implementation contains a `.expect("stored filter hash")` call that will **panic the node process** if block filter data exists in the database but the corresponding filter hash record is absent — a condition reachable by any unprivileged RPC caller.

---

### Finding Description

The `ChainRpc` trait defines the interface contract: [1](#0-0) 

```rust
#[rpc(name = "get_block_filter")]
fn get_block_filter(&self, block_hash: H256) -> Result<Option<BlockFilter>>;
```

The return type `Result<Option<BlockFilter>>` signals that the function must either succeed with a value, succeed with `None` (null), or return a structured error. It must **not** panic.

The implementation in `ChainRpcImpl` is: [2](#0-1) 

```rust
fn get_block_filter(&self, block_hash: H256) -> Result<Option<BlockFilter>> {
    let store = self.shared.store();
    let block_hash = block_hash.into();
    if !store.is_main_chain(&block_hash) {
        return Ok(None);
    }
    Ok(store.get_block_filter(&block_hash).map(|data| {
        let hash = store
            .get_block_filter_hash(&block_hash)
            .expect("stored filter hash");   // <-- PANICS HERE
        BlockFilter {
            data: data.into(),
            hash: hash.into(),
        }
    }))
}
```

The logic correctly returns `Ok(None)` when the block is not in the main chain. However, when `store.get_block_filter()` returns `Some(data)` (filter data is present), the code immediately calls `store.get_block_filter_hash()` and uses `.expect("stored filter hash")`. If the filter hash record is absent — due to a partial write, a crash between the two separate DB write operations, or any storage inconsistency — the `.expect()` call **panics the entire node process**.

The two storage keys (filter data and filter hash) are distinct database entries. Any scenario where they become desynchronized (e.g., a node crash mid-write, a DB migration issue, or a bug in the block filter builder) creates a state where this panic is reachable via a normal RPC call.

---

### Impact Explanation

An unprivileged RPC caller who sends a `get_block_filter` request for a main-chain block whose filter data record exists but whose filter hash record is absent will trigger an unrecoverable node panic. In CKB, a panic in a service thread causes the node to shut down (the logger service captures panics and the process terminates). This is a **remote node crash** reachable by any entity with RPC access, which is typically exposed on localhost but also on nodes that expose their RPC publicly.

The interface contract violation also means that any downstream consumer of this RPC (light clients, block explorers, indexers) that relies on the `null`-on-missing guarantee will instead receive a connection error or a 500-class response, breaking their error-handling logic.

---

### Likelihood Explanation

The triggering condition — filter data present without a corresponding filter hash — can arise from:

1. **Node crash during filter write**: If the node crashes after writing filter data but before writing the filter hash, the database is left in an inconsistent state. On restart, any RPC call to `get_block_filter` for that block will panic the node again, creating a **persistent crash loop** for that block hash.
2. **Database migration or partial restore**: Any tooling that copies or migrates filter data without the hash column produces this state.

The likelihood is **medium**: it requires a prior storage inconsistency, but once that inconsistency exists, the crash is 100% reproducible and deterministic for any caller querying that block.

---

### Recommendation

Replace the `.expect()` with a graceful fallback that matches the interface contract. When filter data is present but the hash is missing, return `Ok(None)` or a structured `Err`:

```rust
Ok(store.get_block_filter(&block_hash).and_then(|data| {
    let hash = store.get_block_filter_hash(&block_hash)?;
    Some(BlockFilter {
        data: data.into(),
        hash: hash.into(),
    })
}))
```

Or, if the inconsistency should be surfaced as an error:

```rust
Ok(store.get_block_filter(&block_hash).map(|data| {
    let hash = store
        .get_block_filter_hash(&block_hash)
        .ok_or_else(|| RPCError::custom(RPCError::DatabaseIsCorrupt, "filter hash missing"))?;
    Ok(BlockFilter { data: data.into(), hash: hash.into() })
}).transpose()?)
```

Additionally, a unit test should be added that exercises `get_block_filter` when filter data is present but the hash record is absent, asserting that the function returns `Ok(None)` or a structured error rather than panicking.

---

### Proof of Concept

1. Start a CKB node with block filter enabled.
2. Allow it to build filters for several blocks.
3. Manually delete the filter hash DB key for one block (or simulate a partial write by patching the store).
4. Call `get_block_filter` via RPC with that block's hash.
5. **Observed**: The node process panics with `"stored filter hash"` and terminates.
6. **Expected per interface**: The RPC returns `null` (`Ok(None)`) or a structured error response.
