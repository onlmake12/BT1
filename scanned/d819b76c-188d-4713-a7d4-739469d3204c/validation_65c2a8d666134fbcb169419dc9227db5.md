### Title
Inbound Peer Can Inject Arbitrary Listen Port into Peer Store via Discovery GetNodes — (`network/src/protocols/discovery/mod.rs`)

### Summary

An unprivileged inbound peer can send a `GetNodes` message with an attacker-chosen `listen_port` value, causing the receiving node to rewrite the TCP port in the session's stored `RemoteAddress` and immediately persist the resulting unverified address into the peer store. This bypasses the explicit design invariant documented in the identify protocol that inbound addresses must not be added to the peer store unverified.

### Finding Description

**Session initialization** (`state.rs` lines 45–82):

When a new session is created, `SessionState::new` branches on session direction:
- **Outbound** sessions: `remote_addr = RemoteAddress::Listen(...)` (already resolved)
- **Inbound** sessions: `remote_addr = RemoteAddress::Init(peer_ephemeral_addr)` — the ephemeral source port of the TCP connection [1](#0-0) 

**GetNodes handler** (`mod.rs` lines 103–135):

When the node receives a `GetNodes` message (sent by the inbound peer), it unconditionally calls `update_port` with the attacker-supplied value: [2](#0-1) 

**`update_port`** (`state.rs` lines 153–167):

This method only acts on `RemoteAddress::Init` — exactly the variant used for inbound sessions. It replaces the TCP port component with the attacker-supplied `u16` and transitions the variant to `RemoteAddress::Listen`: [3](#0-2) 

After `update_port`, the `if let RemoteAddress::Listen(ref addr)` guard at line 128 is now satisfied, so `add_new_addr` is called with the rewritten address.

**`add_new_addrs` / `is_valid_addr`** (`mod.rs` lines 347–363, 332–341):

The only filter applied is `is_valid_addr`, which checks only whether the IP is publicly reachable — it does not validate the port: [4](#0-3) 

The address is then inserted into the peer store via `peer_store.add_addr` with `last_connected_at_ms = 0`: [5](#0-4) 

**Contrast with the identify protocol's explicit guard** (`identify/mod.rs` lines 415–419):

The identify protocol explicitly avoids adding inbound addresses to the peer store for exactly this reason: [6](#0-5) 

The discovery protocol has no equivalent guard.

### Impact Explanation

The polluted address (attacker's real IP, attacker-chosen port) is stored with `last_connected_at_ms = 0`.

- `fetch_random_addrs` requires `last_connected_at_ms > 0` (connected within 7 days), so the address is **not** broadcast to other peers via discovery.
- `fetch_addrs_to_feeler` returns addresses with `last_connected_at_ms = 0` (never connected), so the victim node **will** attempt feeler connections to the attacker-controlled port.
- `fetch_nat_addrs` also selects addresses with `last_connected_at_ms == 0`, so the address may be used for hole-punching attempts. [7](#0-6) 

Impact is local to the victim node: wasted feeler/NAT connection attempts, peer store slot consumption, and degraded future connection quality. The address is not propagated to the wider network.

### Likelihood Explanation

Any peer that can establish an inbound TCP connection to the victim node can trigger this. No authentication, PoW, or special privilege is required. The `GetNodes` message is a standard part of the discovery handshake and is expected from inbound peers. The `listen_port` field is optional (`PortOpt`) but freely settable to any `u16`. [8](#0-7) 

### Recommendation

Before calling `update_port` / `add_new_addr`, check that the session is inbound from the **local node's perspective** and skip the peer store insertion, mirroring the guard already present in the identify protocol. Alternatively, require that the reported `listen_port` be verified (e.g., via a successful feeler connection) before persisting it.

### Proof of Concept

1. Connect to the victim node as an inbound peer (victim accepts the connection).
2. After the discovery protocol `connected` callback fires, send a `GetNodes` message with `listen_port = Some(1)`.
3. Observe that `state.remote_addr` transitions from `RemoteAddress::Init(<attacker_ip>:<ephemeral_port>)` to `RemoteAddress::Listen(<attacker_ip>:1)`.
4. Assert that the victim's peer store now contains `<attacker_ip>:1` (retrievable via `fetch_addrs_to_feeler`).
5. Observe the victim node attempting a feeler connection to `<attacker_ip>:1`.

    pub(crate) fn update_port(&mut self, port: u16) {
        if let RemoteAddress::Init(addr) = self {
            let addr = addr
                .into_iter()
                .map(|proto| {
                    match proto {
                        // TODO: other transport, UDP for example
                        Protocol::Tcp(_) => Protocol::Tcp(port),
                        value => value,
                    }
                })
                .collect();
            *self = RemoteAddress::Listen(addr);
        }
    }

    pub fn fetch_addrs_to_feeler<F>(&mut self, count: usize, filter: F) -> Vec<AddrInfo>
    where
        F: Fn(&AddrInfo) -> bool,
    {
        // Get info:
        // 1. Not already connected
        // 2. Not already tried in a minute
        // 3. Not connected within 3 days

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let addr_expired_ms = now_ms.saturating_sub(ADDR_TRY_TIMEOUT_MS);
        let peers = &self.connected_peers;

        let filter = |peer_addr: &AddrInfo| {
            filter(peer_addr)
                && extract_peer_id(&peer_addr.addr)
                    .map(|peer_id| !peers.contains_key(&peer_id))
                    .unwrap_or_default()
                && !peer_addr.tried_in_last_minute(now_ms)
                && !peer_addr.connected(|t| t > addr_expired_ms)
        };

        self.addr_manager.fetch_random(count, filter)
    }
