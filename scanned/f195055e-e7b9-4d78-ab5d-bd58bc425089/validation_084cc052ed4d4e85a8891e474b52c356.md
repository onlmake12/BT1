### Title
`total_tx_size` Inflated by Evicted Transactions in `PoolMap::add_entry`, Causing Tx-Pool DoS вАФ (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new `total_tx_size` and `total_tx_cycles` are computed **before** ancestor-eviction occurs, then written back **after** eviction, silently discarding the decrements applied by the eviction. This inflates `total_tx_size` permanently, causing `limit_size` to over-evict legitimate transactions and eventually block all new tx-pool submissions.

---

### Finding Description

`PoolMap::add_entry` has a write-ordering bug:

```
// pool_map.rs lines 210-219
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//  ^^^ snapshot: total = X + new_size, computed BEFORE evictions

evicts = self.check_and_record_ancestors(&mut entry)?;
//  ^^^ may call remove_entry_and_descendants -> remove_entry
//      -> update_stat_for_remove_tx, which DECREMENTS self.total_tx_size
//         (self.total_tx_size is now X - evicted_size)

self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

self.total_tx_size = total_tx_size;   // OVERWRITES with X + new_size
self.total_tx_cycles = total_tx_cycles; // OVERWRITES with X + new_cycles
```

Correct post-state: `X - evicted_size + new_size`
Actual post-state: `X + new_size`

The evicted transactions' sizes are permanently lost from the accounting. Each trigger inflates `total_tx_size` by the sum of the sizes of all evicted transactions.

The eviction path inside `check_and_record_ancestors` is reached when:
1. The incoming transaction has more than `max_ancestors_count` (default 25) ancestors in the pool.
2. But removing the `cell_ref_parents` subset brings the count to вЙ§ 25.
3. The code evicts those `cell_ref_parents` entries via `remove_entry_and_descendants`. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict transactions:

```rust
// pool.rs line 298
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee-rate entries
}
``` [4](#0-3) 

When `total_tx_size` is inflated, `limit_size` evicts more transactions than the actual pool occupancy warrants. Repeated triggering accumulates inflation until `total_tx_size` permanently exceeds `max_tx_pool_size` (default 180 MB) even when the pool is nearly empty. At that point every new `submit_entry` call immediately triggers `limit_size`, which evicts the just-inserted transaction or other legitimate transactions, effectively blocking all tx-pool submissions and miner block assembly.

---

### Likelihood Explanation

The trigger condition вАФ a transaction whose ancestor count exceeds `max_ancestors_count` only because of `cell_ref_parents` вАФ is reachable by any unprivileged tx-pool submitter via the `send_raw_transaction` RPC or P2P relay. An attacker can:

1. Submit a base transaction `T0` whose outputs are used as **cell-deps** (not inputs) by a set of transactions `T1вА¶Tk`.
2. Submit `T1вА¶Tk` into the pool (they become `cell_ref_parents` of any future tx that spends `T0`'s output).
3. Submit a new transaction `Tnew` that spends `T0`'s output and has a long input-chain ancestry, pushing `ancestors_count` just above 25 while `ancestors_count - k вЙ§ 25`.
4. The eviction fires, inflating `total_tx_size` by `sum(size(T1вА¶Tk))`.
5. Repeat until `total_tx_size` exceeds `max_tx_pool_size`.

No privileged access, no majority hashpower, and no Sybil attack is required.

---

### Recommendation

Move the `total_tx_size` / `total_tx_cycles` write to **after** `check_and_record_ancestors` completes, and base it on the **current** (post-eviction) `self.total_tx_size` rather than the pre-computed snapshot:

```rust
// After check_and_record_ancestors, record_entry_edges, insert_entry, etc.
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .expect("overflow checked above");
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .expect("overflow checked above");
```

Alternatively, recompute `updated_stat_for_add_tx` after `check_and_record_ancestors` returns, so it reflects the post-eviction baseline.

---

### Proof of Concept

**Setup** (default config: `max_ancestors_count = 25`, `max_tx_pool_size = 180_000_000`):

1. Submit `T0` (large tx, ~1 MB serialized).
2. Submit 5 transactions `T1вА¶T5`, each using `T0`'s output as a **cell-dep** (not input). Each is ~1 MB.
3. Build a 25-deep input chain `C1 вЖТ C2 вЖТ вА¶ вЖТ C25` in the pool.
4. Submit `Tnew` that:
   - Spends `C25`'s output (giving it 25 input-chain ancestors).
   - Also uses `T0`'s output as a cell-dep (making `T1вА¶T5` become `cell_ref_parents`).
   - Total `ancestors_count = 26 > 25`, but `26 - 5 = 21 вЙ§ 25`, so the eviction branch fires.
5. `check_and_record_ancestors` evicts some of `T1вА¶T5` (say 2 of them, ~2 MB total).
6. `update_stat_for_remove_tx` decrements `self.total_tx_size` by ~2 MB.
7. Lines 218-219 overwrite `self.total_tx_size` with the pre-eviction value + `Tnew.size`, losing the 2 MB decrement.
8. Repeat steps 3-7 ~90 times вЖТ `total_tx_size` exceeds 180 MB while the actual pool holds far fewer bytes.
9. Every subsequent `send_raw_transaction` call triggers `limit_size`, which immediately evicts the submitted tx вЖТ **tx-pool DoS**. [5](#0-4) [6](#0-5) [7](#0-6)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
