### Title
Min Fee Rate Admission Check Uses Serialized Size While Actual Fee Rate Uses Weight (Size + Cycles) — (`tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission gate (`check_tx_fee`) validates the minimum fee rate using only the transaction's serialized byte size, while every downstream computation — block-assembly priority, eviction ordering, and fee-rate statistics — uses the true transaction **weight** (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). For a cycle-heavy transaction the weight can be many times larger than the serialized size, so a transaction can pass the admission check with a fee that is far below the configured minimum fee rate when measured against its actual resource consumption.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The actual transaction weight used everywhere else is defined as:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` (≈ `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`). [3](#0-2) 

This weight is used for:

- **Block-assembly priority** (`AncestorsScoreSortKey`): [4](#0-3) 
- **Eviction ordering** (`EvictKey`): [5](#0-4) 
- **Per-entry fee rate** (`fee_rate()`): [6](#0-5) 
- **Historical fee-rate statistics** (RPC): [7](#0-6) 

The admission check runs **before** script execution, so cycles are not yet known. After `verify_rtx` returns the actual cycles, the `TxEntry` is created with the real cycle count, but no second fee-rate check is performed against the true weight. [8](#0-7) 

---

### Impact Explanation

Consider a transaction with:
- `tx_size = 1 000 bytes`
- `cycles = 70 000 000` (near the per-tx cycle cap)
- `weight = max(1 000, 70 000 000 × 0.000_170_571_4) ≈ 11 940 bytes`

With `min_fee_rate = 1 000 shannons/KW`:

| Check | Formula | Required fee |
|---|---|---|
| Admission (`check_tx_fee`) | `1 000 × 1 000 / 1 000` | **1 000 shannons** |
| True weight-based rate | `1 000 × 11 940 / 1 000` | **11 940 shannons** |

A transaction paying exactly 1 000 shannons passes the gate but has an effective fee rate of only ~83 shannons/KW — roughly **12× below** the configured minimum. There is no subsequent check that catches this discrepancy.

An attacker can therefore flood the tx pool with cycle-heavy, size-small transactions at a fraction of the intended cost, displacing legitimate transactions and degrading block-assembly quality.

---

### Likelihood Explanation

The entry path is fully unprivileged: any peer or RPC caller can submit transactions via `send_transaction`. Crafting a transaction with high cycle consumption but small serialized size is straightforward (e.g., a script that loops many times with minimal witness/output data). No special keys, majority hashpower, or social engineering are required.

---

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee-rate check using the true weight:

```rust
let weight = get_transaction_weight(tx_size, cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

Alternatively, enforce a minimum fee based on both dimensions independently:
`fee >= min_fee_rate × tx_size` **and** `fee >= min_fee_rate × (cycles × DEFAULT_BYTES_PER_CYCLES)`.

---

### Proof of Concept

1. Construct a CKB transaction whose lock/type script executes a tight loop consuming ~70 000 000 cycles, with minimal outputs and witness (e.g., 500–1 000 serialized bytes).
2. Set the fee to exactly `min_fee_rate × tx_size` shannons (e.g., 1 000 shannons at the default 1 000 shannons/KW rate).
3. Submit via `send_transaction` RPC.
4. Observe the transaction is accepted into the pool (`check_tx_fee` passes).
5. Query `get_pool_transaction` or inspect `TxEntry::fee_rate()` — the reported fee rate will be `fee / weight ≪ min_fee_rate`, confirming the inconsistency between the admission check and the actual fee rate.

The inconsistency is structurally identical to the BunniToken finding: the **guard** (admission check) uses a smaller denominator (`tx_size`) than the **actual metric** (`weight = max(tx_size, cycles × factor)`), allowing resource consumption that the guard was designed to prevent.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
