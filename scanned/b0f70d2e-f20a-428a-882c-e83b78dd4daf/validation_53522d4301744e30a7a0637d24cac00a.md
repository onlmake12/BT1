## Title
Stale `total_tx_size`/`total_tx_cycles` Overwrite After Eviction in `add_entry` Inflates Pool Accounting - (File: tx-pool/src/component/pool_map.rs)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, those decrements to `self.total_tx_size`/`self.total_tx_cycles` are immediately overwritten by the stale pre-eviction locals at lines 218–219. The result is a permanent, cumulative inflation of the pool's size and cycle counters, causing false `Reject::Full` rejections and spurious evictions of legitimate transactions.

## Finding Description

**Root cause — three-step sequence in `add_entry` (lines 210–219):** [1](#0-0) 

Step 1 (lines 210–211): `updated_stat_for_add_tx` computes `total_tx_size = self.total_tx_size + entry.size` and stores it in a local. `self.total_tx_size` is not yet modified. [2](#0-1) 

Step 2 (line 213): `check_and_record_ancestors` may call `remove_entry_and_descendants` (line 618), which calls `remove_entry` for each evicted entry: [3](#0-2) 

`remove_entry` calls `update_stat_for_remove_tx`, which immediately decrements `self.total_tx_size` and `self.total_tx_cycles`: [4](#0-3) [5](#0-4) 

Step 3 (lines 218–219): The stale pre-eviction locals are written back, silently discarding all decrements applied during step 2.

**Eviction trigger path in `check_and_record_ancestors`:**

Eviction fires when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`: [6](#0-5) 

`cell_ref_parents` are pool entries whose cell-dep out-point matches an input's `previous_output` of the incoming transaction: [7](#0-6) 

**Why existing guards do not prevent the inflation:**

`recompute_total_stat` is only invoked inside `update_stat_for_remove_tx` on underflow. Because the stale overwrite always produces a value ≥ 0, no underflow is ever detected, so `recompute_total_stat` is never triggered to correct the inflated counter. The inflation accumulates indefinitely until node restart. [8](#0-7) 

## Impact Explanation

**Inflated `total_tx_size`/`total_tx_cycles` has two direct, persistent consequences:**

1. **False `Reject::Full`**: `updated_stat_for_add_tx` uses `self.total_tx_size` to guard against pool overflow. An inflated counter causes honest transactions to be rejected even when the pool has physical room.
2. **Spurious eviction of legitimate transactions**: `limit_size` loops while `self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts lowest-fee-rate entries. An inflated counter causes extra eviction iterations, removing honest transactions that should remain. [9](#0-8) 

Both effects are permanent and cumulative across repeated attacks. This matches the allowed CKB bounty impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The trigger requires the incoming transaction to have `ancestors_count > max_ancestors_count` with the excess attributable to `cell_ref_parents`. An attacker can construct this cheaply:

1. Submit `max_ancestors_count` independent transactions (e.g., 25) that all reference the same attacker-controlled output as a cell dep. These are accepted into the pool normally.
2. Submit one transaction whose input spends that output. This makes all 25 pool entries `cell_ref_parents`, pushing `ancestors_count` to 26 > 25.
3. `check_and_record_ancestors` evicts all 25 entries; `self.total_tx_size` is decremented 25 times; then the stale local overwrites it.

The 25 evicted transactions are never confirmed, so the attacker pays no on-chain fees for them. The only confirmed cost is the fee for the final triggering transaction. The attack is fully repeatable: each iteration inflates the counter by the aggregate size of the evicted batch, and the inflation never self-corrects.

No privileged access, key material, or majority hashpower is required.

## Recommendation

Move `updated_stat_for_add_tx` (or its assignment) to after `check_and_record_ancestors` returns, so the snapshot reflects post-eviction reality:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size now reflects post-eviction state
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, remove the local caching entirely and apply the addition in-place after all evictions, mirroring the saturating-arithmetic pattern already used in `update_stat_for_remove_tx`.

## Proof of Concept

**Setup:** Pool contains 25 independent entries `A₁…A₂₅` (each size=1000, cycles=1000), all referencing output `D` as a cell dep. `max_ancestors_count = 25`. `total_tx_size = 25000`.

**Attack:** Submit transaction `B` (size=500, cycles=500) whose input spends output `D`.

| Step | Action | `self.total_tx_size` |
|------|--------|----------------------|
| Before | — | 25000 |
| 1 | `updated_stat_for_add_tx(500,500)` → local `total_tx_size = 25500` | 25000 (unchanged) |
| 2 | `check_and_record_ancestors` evicts A₁…A₂₅; 25× `update_stat_for_remove_tx(1000,…)` | 0 |
| 3 | `self.total_tx_size = 25500` (stale local) | **25500** ← wrong |

**Expected:** `0 + 500 = 500` (only B remains).
**Actual:** `25500` — inflated by 25000.

Subsequent honest transactions of aggregate size ≥ `max_tx_pool_size − 25500` are falsely rejected with `Reject::Full`, or legitimate pool entries are spuriously evicted by `limit_size`. Repeating the attack accumulates further inflation with no self-correction mechanism.

A unit test can be written directly against `PoolMap` in `tx-pool/src/component/tests/` by constructing the above scenario and asserting `pool.total_tx_size == entry_B.size` after `add_entry` returns.
