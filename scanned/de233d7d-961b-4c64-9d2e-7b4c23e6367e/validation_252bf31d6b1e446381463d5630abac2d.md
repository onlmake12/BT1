### Title
Inefficient O(n) Linear Status-Filtered Scan in `next_evict_entry` Causes Increasing CPU Cost Per Transaction Submission - (File: `tx-pool/src/component/pool_map.rs`)

### Summary
`PoolMap::next_evict_entry` performs a full linear scan over all pool entries to find one matching a target status. This is called up to three times per eviction cycle inside `TxPool::limit_size`, which runs after every transaction submission. As the pool fills, each submission triggers an O(n) scan over the entire pool, causing CPU cost that grows linearly with pool size.

### Finding Description
`PoolMap` stores all entries — regardless of status (`Pending`, `Gap`, `Proposed`) — in a single `MultiIndexPoolEntryMap`. Status-filtered lookup is implemented as a full linear scan with a predicate filter: [1](#0-0) 

```rust
pub(crate) fn next_evict_entry(&self, status: Status) -> Option<ProposalShortId> {
    self.entries
        .iter_by_evict_key()
        .find(move |entry| entry.status == status)
        .map(|entry| entry.id.clone())
}
```

This is called from `TxPool::limit_size` in a cascade of three sequential O(n) scans — one per status — until an evictable entry is found: [2](#0-1) 

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    let next_evict_entry = || {
        self.pool_map.next_evict_entry(Status::Pending)
            .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
            .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
    };
    ...
}
```

In the worst case — where all pool entries are `Proposed` — finding an evictable entry requires exhausting all three scans, each traversing the full pool. The same O(n) pattern appears in `score_sorted_iter_by_status`, which filters the entire score-sorted index by status: [3](#0-2) 

This is used by `sorted_proposed_iter`, `get_proposals`, `get_ids`, and `get_all_entry_info` — all of which iterate over every entry regardless of status to collect only those matching a target status.

The `remove_expired` path in `pool.rs` also performs a full O(n) scan over all entries: [4](#0-3) 

### Impact Explanation
The pool is bounded by `max_tx_pool_size` (default 180 MB). With minimum-size transactions (~60 bytes each), the pool can hold on the order of millions of entries. Each transaction submission that triggers eviction causes up to three full O(n) scans. An attacker who keeps the pool near capacity by continuously submitting low-fee transactions forces every subsequent submission to pay the full O(n) scan cost. This degrades node responsiveness for transaction processing and block assembly proportionally to pool occupancy. Under sustained load, this can cause the tx-pool service thread to lag, delaying block template generation and transaction relay.

### Likelihood Explanation
Medium. Any unprivileged peer or RPC caller can submit transactions to the pool. Filling the pool to capacity requires only that submitted transactions meet the minimum fee rate. No special privilege, key, or majority hashpower is needed. The attacker simply needs to keep the pool near its byte limit with transactions of a single status (e.g., all `Proposed`) to maximize the scan cost of `next_evict_entry`.

### Recommendation
Maintain separate per-status indices or counters so that `next_evict_entry` can locate an evictable entry in O(1) or O(log n) rather than O(n). The `PoolMap` already tracks `pending_count`, `gap_count`, and `proposed_count`; extending this to per-status ordered eviction indices (e.g., a separate `BTreeMap<EvictKey, ProposalShortId>` per status) would eliminate the linear scan. Similarly, `score_sorted_iter_by_status` should use a per-status sorted index rather than filtering the global score index.

### Proof of Concept
1. Configure a CKB node with default `max_tx_pool_size` (180 MB).
2. Submit enough valid transactions (meeting minimum fee rate) to fill the pool to capacity, ensuring all entries reach `Proposed` status (by including their proposal IDs in a block's proposal zone).
3. Submit one additional transaction. This triggers `limit_size` at: [5](#0-4) 
4. `next_evict_entry(Status::Pending)` scans all N entries, finds none. `next_evict_entry(Status::Gap)` scans all N entries, finds none. `next_evict_entry(Status::Proposed)` scans all N entries, finds one. Total: 3N iterations per eviction cycle.
5. Repeat step 3 continuously. Each submission incurs 3N CPU iterations. With N in the millions, this measurably increases per-submission latency and can starve the tx-pool service loop.

            .pool_map
