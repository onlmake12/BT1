### Title
DAO Withdrawal `header_deps` Index Interpretation Discrepancy Between Rust Node and CKB-VM Enables Consensus Split — (File: `util/dao/src/lib.rs`)

### Summary
The Rust node's `DaoCalculator::transaction_maximum_withdraw` reads the `header_deps` index from the witness as a full `u64`, while the on-chain DAO C script running in CKB-VM reads only the **lowest byte** (u8) of the same 8-byte field. Any unprivileged transaction sender can craft a DAO withdrawal transaction with `header_deps_index > 255` (where the lowest byte differs from the full value) to exploit this discrepancy. The Rust node will reject a transaction that the CKB-VM would accept, creating a consensus split: a malicious miner can produce a block that the CKB-VM validates correctly but that every honest Rust node rejects.

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit-header index from the witness `input_type` field as a full little-endian `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and then uses it to index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
```

The on-chain DAO C script (`dao.c`) reads only the **lowest byte** of the same 8-byte field. This is explicitly documented in the test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs`:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

For a crafted transaction with `header_deps_index = 257` (little-endian bytes: `0x01 0x01 0x00 …`):

| Component | Index read | Header resolved | Block-number check |
|---|---|---|---|
| CKB-VM (dao.c) | 1 (lowest byte) | deposit block (number 100) | 100 == 100 → **PASS** |
| Rust `DaoCalculator` | 257 (full u64) | withdraw block (number 200) | 200 ≠ 100 → **FAIL** |

The Rust node's block-number guard at line 105 of `util/dao/src/lib.rs`:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
```

fires because the Rust node resolved the wrong header, causing it to reject a transaction that the CKB-VM would accept.

### Impact Explanation

A malicious miner can include a crafted DAO withdrawal transaction (with 258+ `header_deps` and `header_deps_index = 257`) in a block. The CKB-VM executes the DAO script correctly (reads index 1 → deposit block → passes all checks → valid withdrawal). Every honest Rust node, however, calls `DaoCalculator::transaction_fee` during block validation, resolves index 257 → wrong block → block-number mismatch → rejects the block. This produces a **consensus split**: the block is valid on-chain but rejected by all honest Rust nodes, allowing a minority miner to fork the chain or permanently stall honest nodes on a minority branch. Additionally, any legitimate DAO depositor who happens to construct a withdrawal with `header_deps_index > 255` (e.g., due to a large `header_deps` list) will have their withdrawal permanently rejected by the Rust node even though it is on-chain valid.

### Likelihood Explanation

The attacker must be a miner (to include the transaction in a block) or must convince a miner to do so. Constructing the transaction itself requires no special privilege: any transaction sender can craft a DAO withdrawal with 258 `header_deps` and set `input_type` to `257u64.to_le_bytes()`. The discrepancy is deterministic and reproducible. Likelihood is **medium** for the denial-of-withdrawal impact (any DAO user with a large `header_deps` list) and **low-to-medium** for the consensus-split impact (requires a cooperating miner).

### Recommendation

The Rust node's `transaction_maximum_withdraw` must interpret the `header_deps` index identically to the on-chain DAO C script. Either:

1. **Match the C script**: read only the lowest byte of the 8-byte `input_type` field (`header_deps_index_data[0] as usize`) instead of the full `u64`.
2. **Update the C script**: change `dao.c` to read the full `u64` index, and update the Rust node to match.

Both sides of the consensus boundary must agree on the same index-decoding logic.

### Proof of Concept

The existing test in `util/dao/src/tests.rs` already demonstrates the discrepancy:

```rust
// input_type = 257, lowest byte = 1
let witness = WitnessArgs::new_builder()
    .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
    .build();
// ...
// Rust resolves index 257 → withdraw block (number 200), but cell data
// says deposited at block 100. Block number check catches the mismatch.
assert!(result.is_err(), "expected Err, got {result:?}");
```

The test asserts `result.is_err()` — confirming the Rust node rejects the transaction — while the comment explicitly states the CKB-VM would resolve index 1 (the deposit block) and accept it. A malicious actor submits this transaction to a cooperating miner; the miner's block passes CKB-VM script execution but is rejected by every honest Rust node's `DaoCalculator`, splitting the chain.
