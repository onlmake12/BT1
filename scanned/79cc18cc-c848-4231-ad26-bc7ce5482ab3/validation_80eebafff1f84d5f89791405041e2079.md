Audit Report

## Title
Full CKB-VM Execution Before Cycle Mismatch Check Enables Asymmetric CPU DoS via Relay Protocol ŌĆö (`tx-pool/src/process.rs`)

## Summary

In `_process_tx`, full `ContextualTransactionVerifier` (CKB-VM script execution) runs unconditionally before the `declared_cycles != verified.cycles` check. Because the verify cache is only populated on the success path, every resubmission of the same transaction with a wrong-but-plausible `declared_cycles` forces a full VM execution. Combined with the `remove_from_known_txs` behavior on rejection, an attacker with one valid UTXO and multiple IPs can sustain repeated full-VM occupations of the victim's verify queue at negligible cost.

## Finding Description

The execution order in `_process_tx` is confirmed at [1](#0-0) :

1. `pre_check` (line 715) ŌĆö cheap input resolution and fee check
2. `fetch_tx_verify_cache` (line 719) ŌĆö cache lookup by `witness_hash`
3. `max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles())` (line 720)
4. **`verify_rtx(...)` with `max_cycles`** (lines 724ŌĆō732) ŌĆö full CKB-VM execution
5. **Only then**: `if declared != verified.cycles` ŌåÆ `Reject::DeclaredWrongCycles` (lines 736ŌĆō748)

The correct attack uses `declared_cycles = actual_cycles + 1`. With this value, `max_cycles` is set to `actual + 1`, which is sufficient for the VM to complete. The VM returns `verified.cycles = actual`, the mismatch is detected, and `DeclaredWrongCycles` is returned. (Using `declared = actual - 1` would instead cause `ExceededMaximumCycles` before the mismatch check ŌĆö the PoC's use of `C-1` is incorrect; the integration test at line 26 correctly uses `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1`.)

The cache update is placed **after** the early return for `DeclaredWrongCycles` and only on the success path: [2](#0-1) 

So the cache is never populated on a `DeclaredWrongCycles` rejection. Every resubmission of the same witness triggers full VM execution.

On rejection, `after_process` emits `TxVerificationResult::Reject`, which causes `remove_from_known_txs` to remove the tx_hash from the `tx_filter`: [3](#0-2) 

This means the same tx_hash can be re-announced after reconnection, since the filter no longer blocks it.

The only pre-VM guard in `TransactionsProcess::execute` is `declared_cycles > max_block_cycles` (immediate ban, no VM). A declared value of `actual_cycles + 1` passes this guard: [4](#0-3) 

The cache fast-path (cheap `TimeRelativeTransactionVerifier`) is only taken when `cache_entry` is `Some`: [5](#0-4) 

Since the cache is never populated for `DeclaredWrongCycles` rejections, this path is never taken for repeated submissions.

## Impact Explanation

Each relay submission with `declared_cycles = actual + 1` forces the victim node to run full `ContextualTransactionVerifier` including secp256k1 CKB-VM execution, occupying the verify queue for the full duration. The attacker spends O(1) CPU (send a P2P message); the victim spends O(secp256k1_cycles) CPU. With N IPs, the attacker can sustain N parallel verify-queue occupations, delaying legitimate transaction admission across the network.

This matches the **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

**Preconditions:**
- Attacker must own at least one valid UTXO to pass `pre_check`
- Attacker needs multiple IPs to bypass the 3-day ban per IP (triggered by `ban_malformed` if `DeclaredWrongCycles` is classified as malformed)

**Attack loop:**
1. Connect as relay peer
2. Send `RelayTransactionHashes([T.hash])`
3. Receive `GetRelayTransactions([T.hash])`
4. Send `RelayTransactions([{cycles: actual+1, transaction: T}])`
5. Victim runs full CKB-VM ŌåÆ detects mismatch ŌåÆ bans peer IP ŌåÆ removes T.hash from tx_filter
6. Reconnect from new IP ŌåÆ repeat (UTXO is never consumed; T.hash was removed from filter)

The UTXO is never consumed (tx is always rejected), so one UTXO suffices indefinitely. Cost scales with number of IPs, not CKB holdings. The existing integration test confirms the behavior is reachable: [6](#0-5) 

## Recommendation

Move the `declared_cycles` plausibility check **before** `verify_rtx`. Since `max_cycles` is already set to `declared_cycles`, the simplest fix is to also populate `txs_verify_cache` on a `DeclaredWrongCycles` rejection ŌĆö subsequent resubmissions of the same witness would then take the cheap `TimeRelativeTransactionVerifier` path. Alternatively, run the VM with a cycle cap equal to `declared_cycles` and fail fast if the actual cycles differ, or add a pre-execution estimate check before committing to full script execution.

## Proof of Concept

```
1. Attacker owns UTXO U, creates valid secp256k1-signed tx T (actual cycles = C)
2. Connect to victim node as relay peer
3. Send RelayTransactionHashes([T.hash])
4. Receive GetRelayTransactions([T.hash])
5. Send RelayTransactions([{cycles: C+1, transaction: T}])
   ŌåÆ victim sets max_cycles = C+1, runs full secp256k1 VM (C cycles of CPU)
   ŌåÆ verified.cycles = C, declared = C+1 ŌåÆ DeclaredWrongCycles
   ŌåÆ victim bans attacker IP, removes T.hash from tx_filter
   ŌåÆ cache NOT populated
6. Reconnect from new IP ŌåÆ goto step 3
   (UTXO U is still unspent; T.hash is no longer filtered)
```

Confirmed by integration test `DeclaredWrongCycles` which uses `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` (i.e., `actual + 1`) and asserts the tx is rejected after full verification. [7](#0-6)
