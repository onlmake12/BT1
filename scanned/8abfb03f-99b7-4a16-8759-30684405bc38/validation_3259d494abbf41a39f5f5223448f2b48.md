### Title
Missing Bounds Validation in `ExecV2` Syscall Silently Clamps Data Instead of Returning `SLICE_OUT_OF_BOUND` — (File: `script/src/syscalls/exec_v2.rs`)

---

### Summary

The `ExecV2` syscall handler (`script/src/syscalls/exec_v2.rs`) does not validate the `offset` and `length` parameters extracted from the `bounds` register against the actual size of the target data before enqueuing the `Message::ExecV2` message. Both the legacy `Exec` syscall (`exec.rs`) and the `Spawn` syscall (`spawn.rs`) perform explicit bounds checks and return `SLICE_OUT_OF_BOUND` (error code 3) to the calling script when parameters are out of range. `ExecV2` omits these checks entirely, causing the scheduler to silently clamp or truncate the data slice instead of surfacing the expected error code to the script.

---

### Finding Description

In `script/src/syscalls/exec_v2.rs`, the `ecall` handler extracts `offset` and `length` from the `bounds` register and immediately enqueues them without any validation:

```rust
// exec_v2.rs lines 46-66
let bounds = machine.registers()[A3].to_u64();
let offset = bounds >> 32;
let length = bounds as u32 as u64;
// ... no bounds check against actual data size ...
self.message_box.lock()...push(Message::ExecV2(
    self.id,
    ExecV2Args {
        location: DataLocation { data_piece_id, offset, length },
        argc, argv,
    },
));
``` [1](#0-0) 

Compare this to `spawn.rs`, which explicitly fetches the full data length and validates both `offset` and `length` before enqueuing:

```rust
// spawn.rs lines 107-132
let (_, full_length) = sc.load_data(&data_piece_id, 0, 0)?;
if offset >= full_length {
    machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
    return Ok(true);
}
if length > 0 {
    let end = offset.checked_add(length).ok_or(VMError::MemOutOfBound)?;
    if end > full_length {
        machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
        return Ok(true);
    }
}
``` [2](#0-1) 

And the legacy `exec.rs` also validates before slicing: [3](#0-2) 

When the scheduler later processes the `Message::ExecV2` message, it calls `sc.load_data(data_piece_id, offset, length)`: [4](#0-3) 

The underlying `SgData::load_data` implementation silently clamps out-of-bounds parameters rather than returning an error:

```rust
// types.rs lines 1084-1092
let offset = std::cmp::min(offset as usize, data.len());
let full_length = data.len() - offset;
let real_length = if length > 0 {
    std::cmp::min(full_length, length as usize)
} else { full_length };
(data.slice(offset..offset + real_length), full_length as u64)
``` [5](#0-4) 

---

### Impact Explanation

Two concrete misbehaviors result:

1. **`offset >= data_size`**: The clamped offset equals `data.len()`, producing an empty byte slice. `parse_elf` on an empty slice fails with an ELF parse error propagated as a `VMError`, not as `SLICE_OUT_OF_BOUND` (code 3). A script that checks the return value of `ExecV2` for code 3 to implement fallback logic will never see it; instead the entire VM terminates with an internal error, bypassing the script's error-handling branch.

2. **`offset + length > data_size` (partial overflow)**: The clamped slice is a non-empty but truncated prefix of the target cell data. If that truncated prefix is a structurally valid ELF binary (e.g., the ELF header and one loadable segment fit within the prefix), the scheduler loads and executes it as the replacement program. A script author can craft a cell whose first N bytes form a valid ELF with different logic than the full binary, then invoke `ExecV2` with `length = N` where `N < actual_size` — but because no validation occurs, they can also supply `offset + length > actual_size` and still get the truncated prefix loaded without any error signal.

In both cases the observable contract — that out-of-bounds `bounds` values return `SLICE_OUT_OF_BOUND` to the calling script — is violated, breaking any script that relies on this error code for control flow and potentially allowing execution of an unintended code slice.

---

### Likelihood Explanation

`ExecV2` is the exec syscall used in the multi-VM (Meepo hardfork) scheduler path. Any script deployed on CKB that issues syscall `2043` in the new VM context can supply arbitrary `bounds` values. A script author (an unprivileged participant who deploys a lock or type script) is the direct attacker. No special privilege, key material, or majority hashpower is required. The attacker controls the `bounds` register value entirely through their own script code.

---

### Recommendation

Add the same pre-enqueue bounds validation that `spawn.rs` performs. Before pushing `Message::ExecV2`, acquire the `snapshot2_context` lock, call `sc.load_data(&data_piece_id, 0, 0)` to obtain `full_length`, then check:

```rust
if offset >= full_length {
    machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
    return Ok(true);
}
if length > 0 {
    let end = offset.checked_add(length).ok_or(VMError::MemOutOfBound)?;
    if end > full_length {
        machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
        return Ok(true);
    }
}
```

This mirrors the pattern already used in `spawn.rs` lines 107–132 and makes `ExecV2` consistent with both `Spawn` and the legacy `Exec` syscall.

---

### Proof of Concept

A script running under the multi-VM scheduler issues:

```c
// bounds = (offset=0xFFFFFFFF << 32) | length=1
// offset far exceeds any real cell data size
uint64_t bounds = ((uint64_t)0xFFFFFFFF << 32) | 1;
syscall(2043, index, source, place, bounds, argc, argv);
// Expected: A0 = 3 (SLICE_OUT_OF_BOUND)
// Actual:   VM terminates with ELF parse error; A0 is never set to 3
```

Or for the truncation case:

```c
// Cell data is 1000 bytes; offset=0, length=600 (valid ELF header fits in 600 bytes)
// Attacker sets length=1001 (offset+length > data_size)
uint64_t bounds = (0ULL << 32) | 1001;
syscall(2043, index, source, place, bounds, argc, argv);
// Expected: A0 = 3 (SLICE_OUT_OF_BOUND)
// Actual:   scheduler loads data[0..600] (clamped), executes truncated ELF
```

Neither case returns `SLICE_OUT_OF_BOUND` to the calling script, confirming the missing validation. [6](#0-5) [2](#0-1) [7](#0-6) [5](#0-4)
