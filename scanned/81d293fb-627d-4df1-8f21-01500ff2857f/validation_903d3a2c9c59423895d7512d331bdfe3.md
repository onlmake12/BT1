### Title
Softfork Lock-In Threshold Uses Floor Division, Allowing Activation Below the Specified Minimum Ratio — (File: `spec/src/versionbits/mod.rs`)

### Summary
The `threshold_number` helper in the versionbits state machine computes the minimum number of signalling blocks required for a softfork to lock in using **integer (floor) division**. Because the result is rounded down, a softfork can transition to `LockedIn` with a signalling ratio that is strictly **less than** the configured `threshold`, contradicting the specification which states the threshold is the *minimum* required ratio.

### Finding Description
In `spec/src/versionbits/mod.rs`, the `threshold_number` function computes the required signal count as:

```
floor(total_blocks * numer / denom)
``` [1](#0-0) 

The result is then compared with `>=` in `get_state`:

```rust
let threshold_number = threshold_number(total, self.threshold())?;
if count >= threshold_number {
    next_state = ThresholdState::LockedIn;
``` [2](#0-1) 

The `Deployment.threshold` field is documented as **"Specifies the minimum ratio of block per `period`, which indicate the locked_in of the softfork during the `period`"**: [3](#0-2) 

The RPC-facing `DeploymentInfo` similarly describes it as **"the ratio of blocks with the version bit set required to activate the feature"**: [4](#0-3) 

When `total * numer` is not evenly divisible by `denom`, floor division produces a `threshold_number` that is one less than the true ceiling. The `count >= threshold_number` check then passes with a signalling ratio strictly below the configured minimum.

**Concrete example** (testnet threshold = 3/4):
- `total` = 101 blocks in a period
- `threshold_number` = `floor(101 × 3 / 4)` = `floor(75.75)` = **75**
- `count >= 75` passes with 75 signalling blocks → actual ratio = 75/101 ≈ **74.26% < 75%**

The mainnet `LightClient` deployment uses threshold `8/10`: [5](#0-4) 

For any epoch-period length not divisible by 10, the same rounding applies.

### Impact Explanation
A softfork (e.g., the `LightClient` protocol) can reach `LockedIn` and subsequently `Active` with a miner signalling ratio that is fractionally below the governance-mandated minimum. This undermines the consensus guarantee that a supermajority of miners have agreed to the rule change. Nodes that enforce the new rules will diverge from nodes that do not, creating a potential chain split if the activation is contested.

### Likelihood Explanation
CKB epoch lengths are dynamically adjusted by the difficulty algorithm and are not guaranteed to be multiples of the threshold denominator. The maximum under-count is `(denom − 1)` blocks per period (less than one full block's worth of signalling). For the mainnet `8/10` threshold this is at most 9 blocks short; for testnet `3/4` it is at most 3 blocks short. While the absolute deviation is small, it is deterministic and exploitable by any miner coalition that can coordinate signalling to exactly the floor-rounded count.

### Recommendation
Replace floor division with ceiling division in `threshold_number`:

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    // ceiling: (length * numer + denom - 1) / denom
    length
        .checked_mul(threshold.numer())
        .and_then(|product| product.checked_add(threshold.denom().saturating_sub(1)))
        .and_then(|ret| ret.checked_div(threshold.denom()))
}
```

This ensures `count >= threshold_number` is satisfied only when the true ratio `count / total >= numer / denom`, matching the specification's "minimum ratio" semantics.

### Proof of Concept

1. Deploy a softfork with `threshold = Ratio::new(3, 4)` and `period = 2` epochs.
2. Construct a period of 101 total blocks (achievable via difficulty adjustment producing a non-multiple-of-4 epoch length).
3. Signal with exactly 75 blocks (74.26% of 101).
4. Call `get_state` — `threshold_number(101, 3/4)` returns `75`; `75 >= 75` is `true`; state transitions to `LockedIn` despite the actual ratio being below 75%. [6](#0-5)
