### Title
`check_tx_fee` Uses Serialized Size Instead of Actual Weight, Allowing Min-Fee-Rate Bypass — (File: `tx-pool/src/util.rs`)

### Summary

The `check_tx_fee` function in the CKB tx-pool enforces the `min_fee_rate` threshold using only the transaction's serialized byte size as the weight proxy. However, the canonical transaction weight used everywhere else in the codebase — block assembly scoring, fee estimation, and eviction — is `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. An unprivileged transaction sender can craft a transaction with a small serialized size but very high script execution cycles, pay a fee that satisfies only the size-based minimum, and have the transaction admitted to the pool with an effective fee rate far below the configured `min_fee_rate`.

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

where `tx_size = tx.data().serialized_size_in_block()`. The code itself acknowledges the discrepancy:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
```

The actual transaction weight used for scoring and fee-rate calculation throughout the rest of the codebase is defined in `util/types/src/core/tx_pool.rs`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This same function is used in `TxEntry::fee_rate()`, `AncestorsScoreSortKey`, `EvictKey`, and both fee estimator algorithms (`confirmation_fraction` and `weight_units_flow`).

The fee check in `pre_check` runs **before** script execution in `verify_rtx`. At that point, actual cycles are unknown, so `tx_size` is used as a proxy. After verification, the actual cycles are known but no second fee-rate check is performed against the true weight.

### Impact Explanation

With default parameters (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):

- Maximum cycle-derived weight: `70,000,000 × 0.000_170_571_4 ≈ 11,940 bytes`
- A minimal transaction (~100 bytes serialized) with max cycles has actual weight ≈ 11,940 bytes
- Size-based min fee: `1000 × 100 / 1000 = 100 shannons`
- Actual effective fee rate: `100 × 1000 / 11,940 ≈ 8.4 shannons/KW`

This is a **~119× bypass** of the configured `min_fee_rate`. An attacker can flood the mempool with transactions whose effective fee rates are far below the threshold, degrading fee estimation accuracy, displacing legitimate transactions, and consuming mempool resources at a fraction of the intended cost.

### Likelihood Explanation

Any unprivileged transaction sender can exploit this. The attacker deploys a complex lock or type script (once, as a cell dep) and then submits many transactions referencing it. Each transaction has small serialized size but high script execution cycles. The only cost is the size-based minimum fee (e.g., 100 shannons per transaction) and having valid input cells to spend. This is reachable via the `send_transaction` RPC or P2P relay (`submit_remote_tx`).

### Recommendation

Replace the size-only proxy in `check_tx_fee` with the canonical weight function. Since actual cycles are not known at pre-check time, use the declared cycles (available for remote transactions) or a conservative upper bound (e.g., `max_tx_verify_cycles`) for local transactions:

```rust
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(max_tx_verify_cycles));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

Alternatively, perform a second fee-rate check after `verify_rtx` returns the actual cycles, rejecting transactions whose true effective fee rate falls below `min_fee_rate`.

### Proof of Concept

1. Deploy a lock script that consumes ~70,000,000 cycles (e.g., a tight computation loop) as a cell dep.
2. Construct a transaction spending a cell locked by that script. The transaction body (inputs, outputs, witnesses) is ~100 bytes serialized.
3. Set the transaction fee to exactly `ceil(min_fee_rate × tx_size / 1000) = 100 shannons` (for `min_fee_rate = 1000`, `tx_size = 100`).
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` passes: `fee (100) >= min_fee (100)`.
6. `verify_rtx` executes the script, consuming ~70,000,000 cycles.
7. The transaction is admitted to the pool with actual weight ≈ 11,940 and effective fee rate ≈ 8.4 shannons/KW — **119× below** the configured threshold.
8. Repeat with many input cells to fill the mempool with sub-threshold-fee-rate transactions.
