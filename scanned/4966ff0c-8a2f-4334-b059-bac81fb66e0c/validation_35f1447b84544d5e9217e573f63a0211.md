### Title
Unconditional `.expect()` on `BlockExt.txs_sizes` in `FeeRateCollector::statistics` panics on legacy/IBD-stored blocks — (`rpc/src/util/fee_rate.rs`)

### Summary

`FeeRateCollector::statistics` calls `.expect()` on `BlockExt.txs_sizes` without guarding against `None`. `txs_sizes` is legitimately `None` for any block stored via the old `BlockExt` schema (pre-hardfork) or via the IBD fast-sync path (`switch.disable_all()`). Any unprivileged caller invoking `get_fee_rate_statistics` with a target window that includes such a block triggers the panic.

---

### Finding Description

In `rpc/src/util/fee_rate.rs` line 93, the closure passed to `collect` unconditionally unwraps `txs_sizes`:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
``` [1](#0-0) 

`BlockExt.txs_sizes` is typed `Option<Vec<u64>>` and is `None` in two concrete, production-reachable situations:

**1. Old `BlockExt` schema (pre-hardfork blocks)**

The old molecule schema `BlockExt` has no `cycles` or `txs_sizes` fields. Its deserialization always produces `txs_sizes: None`:

```rust
impl<'r> Unpack<core::BlockExt> for packed::BlockExtReader<'r> {
    fn unpack(&self) -> core::BlockExt {
        core::BlockExt {
            ...
            cycles: None,
            txs_sizes: None,   // hardcoded None for old format
        }
    }
}
``` [2](#0-1) 

The two schemas are defined side-by-side in the molecule schema file — `BlockExt` (5 fields, no `txs_sizes`) vs `BlockExtV1` (7 fields, with `txs_sizes`): [3](#0-2) 

**2. IBD fast-sync path (`switch.disable_all()`)**

When the chain verifier runs with all checks disabled (IBD), it calls `insert_ok_ext` with explicit `None` for `txs_sizes`:

```rust
self.insert_ok_ext(&txn, &b.header().hash(), ext.clone(), None, None)?;
``` [4](#0-3) 

These blocks are on the canonical main chain and are returned by `get_block_ext_by_number`.

**3. The `collect` function does not filter `None` txs_sizes**

`collect` uses `filter_map` only to skip blocks missing from the DB entirely. A block that exists but has `txs_sizes: None` passes through to the closure:

```rust
let block_ext_iter =
    (start..=tip_number).filter_map(|number| self.get_block_ext_by_number(number));
block_ext_iter.fold(Vec::new(), f)
``` [5](#0-4) 

**4. The RPC endpoint is public and accepts arbitrary `target`**

`get_fee_rate_statistics` is exposed to any caller with no authentication:

```rust
fn get_fee_rate_statistics(&self, target: Option<Uint64>) -> Result<Option<FeeRateStatistics>> {
    Ok(FeeRateCollector::new(self.shared.snapshot().as_ref())
        .statistics(target.map(Into::into)))
}
``` [6](#0-5) 

---

### Impact Explanation

A Rust `panic!` in an async RPC task unwinds that task. Depending on the async runtime configuration, this either drops the connection (RPC call returns an error) or, if the panic propagates across a thread boundary without a catch, terminates the process. At minimum, the RPC call crashes. The node's RPC service is disrupted for the duration of the panic recovery.

---

### Likelihood Explanation

- Any node that synced its chain history via IBD will have blocks with `txs_sizes: None` in the database.
- Any node within 100 blocks of the hardfork activation height has old-format blocks in the default 101-block target window.
- The trigger requires only a single unauthenticated JSON-RPC call with no special privileges.

---

### Recommendation

Replace the `.expect()` with a graceful skip or early return:

```rust
let Some(txs_sizes) = txs_sizes else {
    return fee_rates; // skip blocks without size data
};
```

This matches the existing pattern for `cycles` (which is already handled with `if let Some(cycles) = cycles`). [7](#0-6) 

---

### Proof of Concept

Construct a `DummyFeeRateProvider` (already exists in the test suite) with one block whose `txs_sizes` is `None`, then call `statistics`:

```rust
let mut provider = DummyFeeRateProvider::new(101);
provider.append(1, BlockExt {
    received_at: 0,
    total_difficulty: 0u64.into(),
    total_uncles_count: 0,
    verified: Some(true),
    txs_fees: vec![Capacity::shannons(1000)],
    cycles: Some(vec![100]),
    txs_sizes: None,   // simulates pre-hardfork or IBD block
});
// panics: "expect txs_size's length >= 1"
let _ = FeeRateCollector::new(&provider).statistics(Some(3));
``` [8](#0-7)

use crate::util::{FeeRateCollector, FeeRateProvider};
use ckb_jsonrpc_types::FeeRateStatistics;
use ckb_types::core::{BlockExt, BlockNumber, Capacity};
use std::collections::HashMap;

struct DummyFeeRateProvider {
    tip_number: BlockNumber,
    block_exts: HashMap<BlockNumber, BlockExt>,
    max_target: u64,
}

impl DummyFeeRateProvider {
    pub fn new(max_target: u64) -> DummyFeeRateProvider {
        DummyFeeRateProvider {
            tip_number: 0,
            block_exts: HashMap::new(),
            max_target,
        }
    }

    pub fn append(&mut self, number: BlockNumber, ext: BlockExt) {
        if number > self.tip_number {
            self.tip_number = number;
        }
        self.block_exts.insert(number, ext);
    }

    pub fn set_max_target(&mut self, max_target: u64) {
        self.max_target = max_target
    }
}

impl FeeRateProvider for DummyFeeRateProvider {
    fn get_tip_number(&self) -> BlockNumber {
        self.tip_number
    }

    fn get_block_ext_by_number(&self, number: BlockNumber) -> Option<BlockExt> {
        self.block_exts.get(&number).cloned()
    }

    fn max_target(&self) -> u64 {
        self.max_target
    }
}
