### Title
Peer Store Permanently Blocked via Crafted /16-Subnet Address Flooding — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

The `check_purge` second eviction path in `PeerStore` uses a strict `> 4` threshold to select eviction candidates. An attacker who fills the addr store with exactly 4 connectable addresses per /16 subnet across 4096 distinct subnets (4096 × 4 = 16384 = `ADDR_COUNT_LIMIT`) causes every group to fail the threshold check, leaving `candidate_peers` empty and returning `Err(EvictionFailed)` — permanently blocking `add_addr` for any new peer.

---

### Finding Description

**Constants confirmed:**

- `ADDR_COUNT_LIMIT = 16384` [1](#0-0) 

- Network group for IPv4 is the first two octets (`[bits[0], bits[1]]`), i.e., the /16 subnet: [2](#0-1) 

**`add_addr` entry point (called from discovery P2P messages):**

`add_addr` calls `check_purge()` before inserting, and propagates any error: [3](#0-2) 

**`check_purge` first eviction pass** — removes non-connectable addresses: [4](#0-3) 

Freshly injected addresses (via `add_addr`) are created with `last_connected_at_ms=0`, `last_tried_at_ms=0`, `attempts_count=0`. The `is_connectable` check returns `true` for these because neither the retry limit (`attempts_count >= 3`) nor the failure limit (`attempts_count >= 10`) is reached: [5](#0-4) 

So the first pass yields nothing when all injected addresses are fresh.

**`check_purge` second eviction pass** — the critical flaw: [6](#0-5) 

The logic:
1. Groups all addresses by `/16` network group.
2. Sorts groups by descending size.
3. Takes the top `len / 2` groups.
4. For each group, evicts 2 random peers **only if `addrs.len() > 4`** (strict greater-than).
5. If `candidate_peers` is empty after this, returns `Err(PeerStoreError::EvictionFailed)`.

The strict `> 4` check is at: [7](#0-6) 

**Attack construction:**

- Attacker fills the store with exactly 4 addresses per /16 subnet across 4096 distinct subnets: 4096 × 4 = 16384 = `ADDR_COUNT_LIMIT`.
- All groups have `len == 4`. The condition `4 > 4` is `false` for every group.
- `flat_map` yields `None` for all 2048 groups examined (`take(len/2) = take(2048)`).
- `candidate_peers` is empty → `Err(EvictionFailed)` is returned.
- Every subsequent `add_addr` call fails permanently.

**P2P entry point confirmed:** The discovery protocol calls `add_addr` directly from received `Nodes` messages: [8](#0-7) 

A single malicious peer can relay 16384 fake addresses across 4096 /16 subnets in ~17 `Nodes` messages (`MAX_ADDR_TO_SEND = 1000` per message). No address reachability verification is performed before insertion.

---

### Impact Explanation

Once the store is poisoned, `add_addr` returns `Err(EvictionFailed)` for every new peer address. The node can no longer learn about or connect to new peers via discovery, identify, or DNS seeding. Existing connections are unaffected, but peer churn (disconnections) cannot be replenished. Over time the node becomes isolated from the network.

---

### Likelihood Explanation

The attack requires a single malicious peer connected via the discovery protocol. No PoW, no privileged access, no key material. The attacker only needs to send ~17 well-crafted `Nodes` messages with addresses spread across 4096 /16 subnets. IPv4 has 65536 possible /16 subnets; 4096 is 6.25% of the space and trivially constructable with synthetic addresses. There is no per-subnet insertion cap and no rate limit on `add_addr`.

---

### Recommendation

Replace the strict `> 4` threshold with `>= 4` (i.e., `addrs.len() >= 4`) so that groups of exactly 4 are also eligible for eviction:

```rust
// peer_store_impl.rs line 378
if addrs.len() >= 4 {   // was: > 4
```

Additionally, consider:
- Enforcing a per-/16-subnet cap at insertion time in `add_addr` (e.g., max 4 per group) so the store cannot be filled with a uniform distribution that defeats eviction.
- Returning `Ok(())` (silently dropping) instead of propagating `EvictionFailed` to callers, so a full store degrades gracefully rather than hard-erroring.

---

### Proof of Concept

```rust
#[test]
fn test_eviction_failed_with_uniform_groups() {
    let mut peer_store = PeerStore::default();
    // Fill 4096 /16 subnets × 4 addresses = 16384 = ADDR_COUNT_LIMIT
    for subnet in 0u16..4096 {
        let a = (subnet >> 8) as u8;
        let b = (subnet & 0xff) as u8;
        for host in 1u8..=4 {
            let addr: Multiaddr = format!("/ip4/{}.{}.1.{}/tcp/8115", a, b, host)
                .parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
        }
    }
    // Store is now at ADDR_COUNT_LIMIT with 4096 groups of exactly 4
    // Next add_addr must trigger check_purge → EvictionFailed
    let new_addr: Multiaddr = "/ip4/200.200.200.200/tcp/8115".parse().unwrap();
    let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
    assert!(result.is_err(), "Expected EvictionFailed but got Ok");
}
```
