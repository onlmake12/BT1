Audit Report

## Title
Stale Descendant Statistics in `remove_entry_and_descendants` Due to Premature Link Clearing - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` clears all parent/child links for every entry in the removal set before calling `remove_entry` on each. Because `update_ancestors_index_key` resolves ancestors through those same links, it always finds an empty ancestor set and never calls `sub_descendant_weight` on ancestors that remain in the pool. This leaves surviving ancestors with permanently inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`, corrupting the `EvictKey` ordering used for pool eviction.

## Finding Description
In `remove_entry_and_descendants` (`pool_map.rs` L252-265), the pre-pass loop calls `remove_entry_links(id)` for every entry in `removed_ids`. `remove_entry_links` (L418-430) does three things: removes the entry from each parent's children set, removes the entry from each child's parents set, and calls `self.links.remove(id)` — which deletes the entry's node entirely from `self.links.inner`.

When `remove_entry` is subsequently called (L235-250), it immediately invokes `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` (L242). That function calls `self.links.calc_ancestors(&child.proposal_short_id())` (L433-434). `calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)` — but the entry's node was already deleted by `remove_entry_links`, so `get` returns `None`, `direct` is empty, and the function returns an empty `HashSet`. The `for anc_id in &ancestors` loop never executes, so `sub_descendant_weight` is never called on any surviving ancestor.

The comment at L256 acknowledges the pre-clearing is intentional to suppress `update_descendants_index_key` (since all descendants are being removed), but it silently also suppresses `update_ancestors_index_key`, which must update ancestors that are **not** being removed.

The existing test `test_remove_entry_and_descendants` (score_key.rs L171-230) only asserts that removed entries are absent and that `calc_descendants` is empty on the surviving ancestor — it never checks `descendants_count`, `descendants_fee`, or `descendants_size` on `tx1` after removal, so the invariant violation is undetected.

## Impact Explanation
Every ancestor of the removed cluster that remains in the pool retains permanently inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`. These fields feed directly into `EvictKey` (entry.rs L234-247): `fee_rate` is computed as `max(descendants_feerate, feerate)`, where `descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight)`. An ancestor with ghost `descendants_fee` will have an artificially elevated `fee_rate` in its `EvictKey`, causing `next_evict_entry` (L380-385) to skip it in favor of legitimately higher-fee transactions. This corrupts pool eviction ordering and allows an unprivileged attacker to make low-fee transactions resistant to eviction, which can cause the pool to reject legitimate high-fee transactions when the pool is full — matching the **High** impact class: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation
The three callers of `remove_entry_and_descendants` — `resolve_conflict`, `resolve_conflict_header_dep`, and the ancestor-count eviction path in `check_and_record_ancestors` — are all reachable by any unprivileged node peer or RPC caller. The attacker-controlled path requires only two submitted transactions (a parent and a child) followed by one conflicting transaction. This is a zero-cost, repeatable operation. Each repetition permanently inflates one ancestor's descendant statistics. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation
Before the pre-clearing loop in `remove_entry_and_descendants`, compute the set of ancestors of the root entry that are **not** in `removed_ids` (i.e., ancestors that will survive). After all entries are removed, iterate over those surviving ancestors and call `sub_descendant_weight` for each removed entry that was a descendant of that ancestor. Alternatively, restructure `remove_entry` to accept an optional pre-computed ancestor snapshot, so that `update_ancestors_index_key` uses the snapshot taken before any links are cleared, rather than querying `self.links` at call time.
