Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Permanently Inflated by Stale Snapshot Overwrite After Cell-Ref Ancestor Eviction in `add_entry` — (File: tx-pool/src/component/pool_map.rs)

## Summary
In `PoolMap::add_entry`, a pre-eviction snapshot of `total_tx_size` and `total_tx_cycles` is computed before `check_and_record_ancestors` is called. When that function evicts cell-ref-parent transactions, it correctly decrements the counters via `update_stat_for_remove_tx`. However, `add_entry` then unconditionally overwrites those correctly-updated values with the stale snapshot, permanently inflating both counters by the total size and cycles of every evicted entry. An unprivileged attacker can repeat this to inflate the counters until the pool rejects all new transactions with `Reject::Full`, even though actual pool occupancy is near zero.

## Finding Description
`add_entry` (lines 200–221) executes:

1. **L210–211**: `(total_tx_size, total_tx_cycles) = self.updated_stat_for_add_tx(entry.size, entry.cycles)?` — snapshot computed as `self.total_tx_size + entry.size` (local variable, not yet written).
2. **L213**: `evicts = self.check_and_record_ancestors(&mut entry)?` — when `ancestors_count > max_ancestors_count` and `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (L603), the function calls `remove_entry_and_descendants` (L618), which calls `remove_entry` (L252→L235), which calls `update_stat_for_remove_tx` (L247), which **directly decrements** `self.total_tx_size` and `self.total_tx_cycles` in place (L738–740).
3. **L218–219**: `self.total_tx_size = total_tx_size; self.total_tx_cycles = total_tx_cycles;` — the stale snapshot (which does not include the eviction decrements) is written back, erasing the correct post-eviction values.

Net effect per triggering call: `total_tx_size` is inflated by the sum of sizes of all evicted entries; `total_tx_cycles` is inflated by their cycles. The inflation accumulates across repeated calls. No existing guard prevents this: `updated_stat_for_add_tx` only checks for overflow/pool-full at the moment of the snapshot, before evictions occur, and `update_stat_for_remove_tx`'s underflow fallback (`recompute_total_stat`) is never reached because the overwrite happens after it.

## Impact Explanation
`total_tx_size` is the sole gate for pool-full rejection (`limit_size`, pool.rs L298: `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`) and for new-entry admission (`updated_stat_for_add_tx`, pool_map.rs L716). An attacker who inflates `total_tx_size` to ≥ 180 MB (the default `max_tx_pool_size`) causes every subsequent `send_transaction` call to be rejected with `Reject::Full`, and causes `limit_size` to evict legitimately admitted transactions. The pool appears full to all admission logic while its actual byte occupancy may be near zero. This matches the **High** impact: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation
The eviction path is reachable by any unprivileged `send_transaction` RPC caller with no special privileges. The attacker pays only normal transaction fees. The attack requires constructing a transaction whose ancestor count exceeds `max_ancestors_count` (25) but whose non-cell-ref ancestor count does not, which is achievable with a 25-deep input chain plus one cell-dep reference to an ancestor. The attack is repeatable in a loop; each iteration inflates the counters by the sizes/cycles of the evicted chain. No Sybil attack, majority hashpower, or victim mistake is required.

## Recommendation
Move the counter assignment to **after** `check_and_record_ancestors` returns, computing the final values from the post-eviction state of `self`:

```rust
// In add_entry, replace lines 210-211 and 218-219:
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Note: the pool-full check in `updated_stat_for_add_tx` must still be performed before insertion; if the check must remain early, the alternative is to subtract evicted sizes/cycles from the pre-computed snapshot before assigning.

## Proof of Concept
Default config: `max_ancestors_count = 25`, `max_tx_pool_size = 180_000_000`.

1. Submit transactions `T0 → T1 → … → T24` linked by inputs (25-deep chain). All admitted; `total_tx_size = 25 * S`.
2. Submit `T25` whose input spends `T24`'s output **and** whose `cell_deps` reference `T0`'s output cell. `T25` has 26 ancestors; `26 - 1 = 25 ≤ 25`, so the eviction branch at L603 fires.
3. `check_and_record_ancestors` calls `remove_entry_and_descendants(T0)`, removing T0…T24 (25 entries). `self.total_tx_size` is correctly decremented to `0`.
4. `add_entry` L218 assigns `self.total_tx_size = 25*S + S_T25` (the stale snapshot), overwriting `0 + S_T25`.
5. Pool now reports `total_tx_size = 25*S + S_T25` while holding only `T25` (size `S_T25`).
6. Repeat from step 1. After `ceil(180_000_000 / (25*S))` iterations, `total_tx_size ≥ 180 MB` and all subsequent `send_transaction` calls return `Reject::Full`. A unit test asserting `pool_map.total_tx_size == actual_sum_of_entry_sizes` after the eviction path fires would reproduce the invariant violation directly.
