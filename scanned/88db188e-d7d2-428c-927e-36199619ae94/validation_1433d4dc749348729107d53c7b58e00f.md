Audit Report

## Title
`DownloadScheduler::decrease` Uses `task_count` Instead of `timeout_count` in Accumulation, Breaking Peer Penalty Throttling - (File: sync/src/types/mod.rs)

## Summary
In `sync/src/types/mod.rs` at line 522, `DownloadScheduler::decrease` assigns `self.task_count.saturating_add(num)` to `self.timeout_count` instead of `self.timeout_count.saturating_add(num)`. This discards any previously accumulated value of `timeout_count` on every call, making the intended "accumulate slow-response events before penalizing" threshold non-functional. The result is that `task_count` (the per-peer block download concurrency budget) is either decremented on every single slow response (Case A) or never decremented further once it reaches 1 (Case B), both deviating from the intended behavior.

## Finding Description
The confirmed code at line 522:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // uses task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

`INIT_BLOCKS_IN_TRANSIT_PER_PEER` is 32 and `MAX_BLOCKS_IN_TRANSIT_PER_PEER` is 128 (confirmed in `util/constant/src/sync.rs` lines 14–16). Since `task_count` starts at 32, `task_count + num` is always > 2 for any `num ≥ 1`, so the threshold check is trivially true on every call and `task_count` is decremented on every slow-block response — the intended 3-event accumulation gate never engages. Conversely, when `task_count` has been penalized down to 1 and `num = 1`, `1 + 1 = 2` which is not `> 2`, so no further decrement occurs and the peer is stuck at minimum budget indefinitely through this path.

`decrease` is called from `remove_by_block` (lines 801–810) for `NormalToUpper` and `UpperToMax` time quantiles, which are normal network conditions. The `should_punish` guard (line 786) only requires that the number of tracked peers exceeds `protect_num`, a routine condition during active sync.

## Impact Explanation
The bug causes suboptimal peer scheduling: legitimate peers experiencing transient slowness have their download concurrency budget (`task_count`) drained far faster than intended, degrading sync throughput. Peers penalized to `task_count = 1` are never further penalized via this path regardless of continued slow responses, wasting a download slot. This constitutes a concrete sync performance regression fitting **Low (501–2000 points): Any other important performance improvements for CKB**. The claimed eclipse/disconnection impact is overstated — `task_count` reaching 0 stops block requests to a peer but does not terminate the connection.

## Likelihood Explanation
Any peer whose block responses fall in the `NormalToUpper` or `UpperToMax` time quantile triggers `decrease`. This is a routine network condition requiring no special privileges. The `should_punish` guard is satisfied whenever the peer count exceeds the protected minimum, which is the normal operating state during sync. An adversarial peer can deliberately tune response latency to stay below `BLOCK_DOWNLOAD_TIMEOUT` (30s, line 48 of `sync.rs`) while remaining in the slow quantile, exploiting Case B to hold a persistent minimum-budget slot.

## Recommendation
Fix line 522 in `sync/src/types/mod.rs`:

```rust
// Before (buggy):
self.timeout_count = self.task_count.saturating_add(num);

// After (correct):
self.timeout_count = self.timeout_count.saturating_add(num);
```

## Proof of Concept
Starting from `task_count = 32, timeout_count = 0` with the buggy code, calling `decrease(1)` three times:

| Call | Buggy `timeout_count` | `task_count` after | Correct `timeout_count` | `task_count` after |
|---|---|---|---|---|
| `decrease(1)` | `32+1=33 > 2` → reset | **31** | `0+1=1 ≤ 2` | 32 |
| `decrease(1)` | `31+1=32 > 2` → reset | **30** | `1+1=2 ≤ 2` | 32 |
| `decrease(1)` | `30+1=31 > 2` → reset | **29** | `2+1=3 > 2` → reset | **31** |

A unit test can assert that after two `decrease(1)` calls, `task_count` remains unchanged, and only after the third call does it decrement by 1. With the current code, the test will fail because `task_count` decrements on every call. [1](#0-0) [2](#0-1) [3](#0-2)
