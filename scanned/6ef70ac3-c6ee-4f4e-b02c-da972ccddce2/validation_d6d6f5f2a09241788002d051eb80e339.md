### Title
Unchecked Integer Underflow in `serialized_size_without_uncle_proposals` Bypasses Block Size Accounting — (`util/gen-types/src/extension/serialized_size.rs`)

---

### Summary

`BlockBytesVerifier` calls `serialized_size_without_uncle_proposals()` on every non-genesis block received from the network. Inside that function, the per-uncle subtraction `x.proposals().as_slice().len() - molecule::NUMBER_SIZE` is performed without a lower-bound guard. If a remote peer sends a block whose uncle carries a malformed `proposals` field whose raw byte length is less than `molecule::NUMBER_SIZE` (4), the subtraction underflows. In Rust debug/test builds this is a guaranteed panic; in release builds it wraps, producing a wildly incorrect block-size value that is then compared against the consensus limit.

---

### Finding Description

In `util/gen-types/src/extension/serialized_size.rs` the function `BlockReader::serialized_size_without_uncle_proposals` computes:

```rust
pub fn serialized_size_without_uncle_proposals(&self) -> usize {
    let block_size = self.as_slice().len();
    let uncles_proposals_size = self
        .uncles()
        .iter()
        .map(|x| x.proposals().as_slice().len() - molecule::NUMBER_SIZE)  // ← unchecked subtraction
        .sum::<usize>();
    block_size - uncles_proposals_size   // ← second unchecked subtraction
}
``` [1](#0-0) 

`molecule::NUMBER_SIZE` is 4. A `ProposalShortIdVec` (fixvec) is valid only when its raw slice is ≥ 4 bytes. However, the molecule field accessor `x.proposals()` uses `new_unchecked` internally and does **not** verify the slice length before returning it:

```rust
// pattern from generated code – no length guard before returning the slice
ProposalShortIdVecReader::new_unchecked(&self.as_slice()[start..end])
```

A remote peer can craft a block whose uncle block's molecule encoding places the `proposals` field at an offset that yields a raw slice of 0–3 bytes. When `serialized_size_without_uncle_proposals` iterates over that uncle, `x.proposals().as_slice().len()` returns a value less than 4, and the subtraction underflows.

This function is called unconditionally by `BlockBytesVerifier::verify` **before** any molecule-level structural verification of uncle proposals is performed:

```rust
pub fn verify(&self, block: &BlockView) -> Result<(), Error> {
    if block.is_genesis() { return Ok(()); }
    let block_bytes = block.data().serialized_size_without_uncle_proposals() as u64;
    if block_bytes <= self.block_bytes_limit { Ok(()) }
    else { Err(BlockErrorKind::ExceededMaximumBlockBytes.into()) }
}
``` [2](#0-1) 

`BlockBytesVerifier` is the **second** check in `BlockVerifier::verify`, running before `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier`:

```rust
fn verify(&self, target: &BlockView) -> Result<(), Error> {
    BlockProposalsLimitVerifier::new(max_block_proposals_limit).verify(target)?;
    BlockBytesVerifier::new(max_block_bytes).verify(target)?;   // ← underflow here
    CellbaseVerifier::new().verify(target)?;
    ...
}
``` [3](#0-2) 

---

### Impact Explanation

**Debug / test builds (panic = crash):** Rust panics on integer underflow in debug mode. Any P2P peer that sends a single malformed block causes the receiving node to abort. This is a remotely-triggered DoS.

**Release builds (wrapping arithmetic = incorrect size accounting):** Rust wraps on underflow in release mode. With one uncle whose proposals slice is 0 bytes:

```
0usize - 4usize  →  usize::MAX - 3   (wrapping)
block_size - (usize::MAX - 3)  →  block_size + 4   (wrapping)
```

The computed "block size without uncle proposals" becomes `block_size + 4` instead of the correct value. The `BlockBytesVerifier` then compares this inflated value against `max_block_bytes`. A legitimately-sized block that is within 4 bytes of the limit would be incorrectly rejected with `ExceededMaximumBlockBytes`, causing a consensus split between nodes that processed the block before and after the malformed uncle was seen. With two malformed uncles the inflation is `+8`, and so on.

---

### Likelihood Explanation

Any unprivileged P2P peer (block relayer) can trigger this. The attacker only needs to:
1. Connect to a CKB node.
2. Send a `SendBlock` message containing a block whose uncle block's molecule encoding places the `proposals` fixvec field at an offset that yields a raw slice shorter than 4 bytes.

No special privileges, keys, or majority hash power are required. The malformed block is processed before molecule structural verification of uncle proposals occurs.

---

### Recommendation

Add a lower-bound guard before the subtraction in `serialized_size_without_uncle_proposals`:

```rust
.map(|x| {
    let proposals_len = x.proposals().as_slice().len();
    proposals_len.saturating_sub(molecule::NUMBER_SIZE)
})
```

Alternatively, enforce that molecule verification of the full block (including uncle proposals) is completed before any size-accounting function is called. The `BlockBytesVerifier` should run only after structural molecule verification has confirmed that every uncle's `proposals` field is at least `NUMBER_SIZE` bytes.

---

### Proof of Concept

Craft a molecule-encoded `Block` where one `UncleBlock`'s raw bytes are arranged so that the `proposals` field offset in the molecule header points to a region of 0–3 bytes. Submit this block to a CKB node via the P2P `SendBlock` message.

- **Debug build**: the node panics at `x.proposals().as_slice().len() - molecule::NUMBER_SIZE` inside `serialized_size_without_uncle_proposals`.
- **Release build**: the function returns `block_size + (4 - proposals_len)` instead of the correct value; `BlockBytesVerifier` compares this inflated figure against `max_block_bytes`, potentially rejecting a valid block or producing a different accept/reject decision than nodes that did not receive the malformed uncle.
