### Title
Unauthenticated Miner Notify HTTP Endpoint Allows Injection of Crafted Block Template, Redirecting Mining Rewards — (File: `miner/src/client.rs`)

### Summary

The `ckb-miner` process, when configured in notify mode, opens an HTTP server that accepts block template updates from any caller with no authentication. An attacker who can reach this port can POST a crafted `BlockTemplate` JSON payload with an attacker-controlled `cellbase` lock script, causing the miner to solve PoW for a block that pays mining rewards to the attacker's address. The CKB node's `submit_block` RPC does not validate that the cellbase matches the node's configured block assembler, so the block is accepted by the network and the reward is permanently diverted.

### Finding Description

**Root cause — `miner/src/client.rs`, `handle` function (lines 358–369):**

When the miner is started with a `listen` address configured, `listen_block_template_notify` binds a TCP listener and serves every incoming HTTP connection through the `handle` function:

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
```

There is no IP allowlist check, no token, no HTTP Basic Auth, and no HMAC verification on the incoming request. Any HTTP client that can reach the port is treated identically to the legitimate CKB node. [1](#0-0) 

`update_block_template` then accepts the injected template if its `work_id` differs from the currently stored one (or if the stored id is `0`), and immediately dispatches it to the PoW workers: [2](#0-1) 

The miner workers solve PoW against whatever `BlockTemplate` they receive and call `submit_block` with the solved block: [3](#0-2) 

**Why the node accepts the block:**

`submit_block` in `rpc/src/module/miner.rs` runs `HeaderVerifier` and `blocking_process_block`. Neither check validates that the cellbase lock script matches the node's configured `BlockAssemblerConfig`. Consensus rules only enforce that cellbase output capacity does not exceed the allowed reward — not that it goes to a specific address. A structurally valid block with an attacker-supplied cellbase is accepted. [4](#0-3) 

**The notify feature is documented and supported:**

The `BlockAssemblerConfig` explicitly supports a `notify` URL list, and the miner's `listen` option is a first-class configuration field. The feature is not experimental or hidden. [5](#0-4) [6](#0-5) 

**Contrast with the outgoing RPC client:**

The miner's outgoing RPC client (`Rpc::new`) does support HTTP Basic Auth via `parse_authorization` when credentials are embedded in the RPC URL. No equivalent protection exists on the incoming notify server. [7](#0-6) [8](#0-7) 

### Impact Explanation

An attacker who can reach the miner's notify HTTP port can:

1. POST a crafted `BlockTemplate` JSON with their own `cellbase` lock script (reward address) but otherwise valid fields (correct `parent_hash`, `compact_target`, `epoch`, `number`, `dao`, etc., copied from a legitimate template obtained via `get_block_template`).
2. The miner's PoW workers immediately switch to mining this template.
3. When a valid nonce is found, the miner submits the block to the CKB node.
4. The node accepts the block (all consensus rules pass).
5. The block reward is permanently paid to the attacker's address.

This is **direct, irreversible theft of mining rewards** with no recovery path once the block is confirmed.

### Likelihood Explanation

- The notify mode is a supported, documented feature. Miners using pool-style or automated setups are likely to enable it.
- If the miner binds `listen` to any non-loopback address (e.g., `0.0.0.0:8888`), the endpoint is reachable by any network peer — no special access required.
- Even on `127.0.0.1`, any local process (co-located services, compromised dependency, SSRF from another local service) can exploit this.
- The attacker needs only a single valid HTTP POST. No PoW, no key material, no privileged access.
- The `work_id` bypass is trivial: use `work_id=0` (always accepted when the stored id is `0` at startup) or any value differing from the current one. [9](#0-8) 

### Recommendation

1. **Authenticate the notify endpoint.** Add an HMAC-SHA256 or HTTP Basic Auth check in the `handle` function. The shared secret should be configured in `ckb-miner.toml` and matched against an `Authorization` or `X-Notify-Token` header on every incoming request.
2. **IP allowlist.** Reject connections from addresses not in a configured allowlist (e.g., only accept from the node's IP).
3. **Validate the template source.** After receiving a template via notify, optionally re-fetch and cross-check the template from the trusted RPC endpoint before dispatching to workers.

### Proof of Concept

**Precondition:** Miner is running with `listen = "0.0.0.0:8888"` (or any reachable address).

**Step 1:** Obtain a valid current block template from the node's public RPC (no auth needed if the RPC is accessible, or infer fields from chain state):
```
curl http://<node>:8114/ -d '{"id":1,"jsonrpc":"2.0","method":"get_block_template","params":[null,null,null]}'
```

**Step 2:** Modify the `cellbase.data.outputs[0].lock` to the attacker's lock script. Keep all other fields (`parent_hash`, `compact_target`, `epoch`, `number`, `dao`, `work_id` set to a value ≠ current) unchanged.

**Step 3:** POST the crafted template to the miner's notify port:
```
curl -X POST http://<miner>:8888/ \
  -H "Content-Type: application/json" \
  -d '<crafted_block_template_json>'
```

**Result:** The miner's workers immediately begin mining the attacker's template. The next solved block pays the block reward to the attacker's address. The CKB node accepts the block without error. [1](#0-0) [2](#0-1)

    async fn listen_block_template_notify(&self, addr: SocketAddr) {
        let listener = TcpListener::bind(addr).await.unwrap();
        let server = auto::Builder::new(TokioExecutor::new());
        let graceful = GracefulShutdown::new();
        let stop_rx: CancellationToken = new_tokio_exit_rx();

        loop {
            let client = self.clone();
            let handle = service_fn(move |req| handle(client.clone(), req));
            tokio::select! {
                conn = listener.accept() => {
                    let (stream, _) = match conn {
                        Ok(conn) => conn,
                        Err(e) => {
                            info!("accept error: {}", e);
                            tokio::time::sleep(std::time::Duration::from_secs(1)).await;
                            continue;
                        }
                    };
                    let stream = hyper_util::rt::TokioIo::new(Box::pin(stream));
                    let conn = server.serve_connection_with_upgrades(stream, handle);

                    let conn = graceful.watch(conn.into_owned());
                    tokio::spawn(async move {
                        if let Err(err) = conn.await {
                            info!("connection error: {}", err);
                        }
                    });
                },
                _ = stop_rx.cancelled() => {
                    info!("Miner client received exit signal. Exit now");
                    break;
                }
            }
        }
        drop(listener);
        graceful.shutdown().await;

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}
