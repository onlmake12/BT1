### Title
Any RPC Caller Can Remove Any Pending Transaction from the Pool Without Submitter Ownership Check — (File: `rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC method allows any caller with Pool RPC access to remove **any** pending transaction from the tx-pool by hash, with no check that the caller is the original submitter of that transaction. This is a direct structural analog to the ERC721F burn issue: a privileged action (pool removal) is gated only on module-level access, not on per-resource ownership.

---

### Finding Description

`remove_transaction` is implemented in `rpc/src/module/pool.rs` at lines 662–669:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();

    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [1](#0-0) 

The function accepts a bare `tx_hash: H256` and immediately forwards it to `remove_local_tx`. There is no check that:
- the caller submitted the transaction being removed, or
- the caller has any relationship to the inputs/outputs of that transaction.

The Pool RPC module is registered as a single unit in `service_builder.rs` via `pool_enable`:

```rust
set_rpc_module_methods!(self, "Pool", pool_enable, add_pool_rpc_methods, methods)
``` [2](#0-1) 

This means `remove_transaction` shares the exact same access gate as `send_transaction`. Any caller who can submit a transaction can also remove **any other user's** transaction from the pool.

The internal pool path `remove_tx` in `tx-pool/src/pool.rs` unconditionally removes the entry and all its descendants:

```rust
pub(crate) fn remove_tx(&mut self, id: &ProposalShortId) -> bool {
    let entries = self.pool_map.remove_entry_and_descendants(id);
    !entries.is_empty()
}
``` [3](#0-2) 

`remove_entry_and_descendants` cascades the removal to all dependent transactions: [4](#0-3) 

---

### Impact Explanation

An attacker with Pool RPC access can:

1. Observe any pending transaction hash (via `get_raw_tx_pool` or `send_transaction` response from another user).
2. Call `remove_transaction` with that hash.
3. The target transaction **and all its descendants** are permanently evicted from the pool.

Concrete consequences:
- **Targeted transaction censorship**: An attacker can repeatedly evict a victim's transaction before it is proposed/committed, preventing confirmation indefinitely.
- **Cascade eviction**: Because `remove_entry_and_descendants` is called, a single call can evict an entire chain of dependent transactions built by the victim.
- **Time-locked transaction griefing**: Transactions with `since`-based time locks that must be confirmed within a specific epoch/block window can be made to miss their window by repeated removal.

---

### Likelihood Explanation

The CKB RPC server binds to `127.0.0.1:8114` by default, so the immediate attacker surface is any local process on the node host. However:

- Operators commonly expose the RPC port to a LAN or the public internet for wallet/dApp integration.
- The Pool RPC module is enabled by default for any functional node.
- No authentication layer exists at the RPC level; `remove_transaction` is co-gated with `send_transaction`.
- The attack requires only knowing a target transaction hash, which is public information broadcast over the P2P network.

Likelihood is **medium**: trivially exploitable once RPC is reachable, and RPC exposure is a common real-world deployment pattern.

---

### Recommendation

Add a submitter-identity check before allowing removal. One approach: record the submitter identity (e.g., a session token or IP-based nonce) at `submit_local_tx` time and require the same identity to call `remove_local_tx`. Alternatively, restrict `remove_transaction` to a separate, explicitly privileged RPC module (analogous to `IntegrationTest` or `Miner`) that operators consciously enable only for trusted callers, rather than bundling it with the general-purpose Pool module.

---

### Proof of Concept

1. Alice submits a transaction via `send_transaction` → receives `tx_hash_A`.
2. Bob (any other RPC caller on the same node) calls:
   ```json
   {"method": "remove_transaction", "params": ["<tx_hash_A>"], ...}
   ```
3. The node returns `true`; Alice's transaction and all its descendants are evicted from the pool.
4. Alice's transaction never reaches a proposed or committed state unless she resubmits — at which point Bob can repeat step 2.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
