Audit Report

## Title
Missing Per-Peer Rate Limiter on `LightClientProtocol` Allows Unbounded I/O Exhaustion via `GetBlocksProof` — (`util/light-client-protocol-server/src/lib.rs`)

## Summary
`LightClientProtocol` implements `CKBProtocolHandler` with no per-peer rate-limiting state. An unprivileged remote peer can flood the node with `GetBlocksProof` messages each carrying up to 1000 block hashes, triggering up to 3000 synchronous RocksDB reads and one O(N log N) MMR proof generation per message with no throttle, ban, or backpressure applied.

## Finding Description
`LightClientProtocol` contains only `shared: Shared` — no rate-limiting field of any kind. [1](#0-0) 

`received` parses the message and immediately calls `try_process` with no guard: [2](#0-1) 

`try_process` dispatches directly to `GetBlocksProofProcess::execute`: [3](#0-2) 

Inside `execute`, the only check is `block_hashes().len() <= GET_BLOCKS_PROOF_LIMIT` (1000). For every hash on the main chain, three synchronous DB reads are issued: [4](#0-3) 

Then `reply_proof` calls `mmr.gen_proof(items_positions)` — an O(N log N) MMR traversal over up to 1000 leaf positions: [5](#0-4) 

`GET_BLOCKS_PROOF_LIMIT` caps only per-message size, not message rate: [6](#0-5) 

By contrast, `Relayer` wraps every dispatch with a `governor::RateLimiter` keyed by `(peer, message_type)` at 30 req/s, checked before any processing: [7](#0-6) 

A `grep` for `rate_limiter`, `RateLimiter`, and `governor` across the entire `light-client-protocol-server` crate returns zero matches, confirming no equivalent protection exists.

## Impact Explanation
A single attacker peer sustaining a tight loop of `GetBlocksProof(1000 valid hashes)` messages imposes per-message costs of 3000 RocksDB point-reads plus one MMR proof generation, at negligible network cost (small packed request messages). This can saturate disk I/O and CPU on the full node, degrading or blocking service for all honest peers. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points). It also overlaps with **High — Vulnerabilities which could easily crash a CKB node** if I/O saturation causes the node to become unresponsive.

## Likelihood Explanation
The LightClient protocol is enabled in production CKB nodes. Opening a LightClient session requires no privilege, no PoW, and no stake — any peer can do it. The attack is trivially scriptable in a tight loop. The attacker only needs a chain with ≥1000 blocks (trivially satisfied on mainnet). There is no existing ban, throttle, or backpressure mechanism to stop or slow the flood.

## Recommendation
Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, mirroring the pattern in `Relayer`: [8](#0-7) 

Check the rate limiter at the top of `received` (or `try_process`) before dispatching to any component handler, returning `StatusCode::TooManyRequests` (without banning) on excess. A quota of 10–30 req/s per peer per message type is consistent with the existing relayer policy.

## Proof of Concept
1. Start a CKB node with LightClient protocol enabled on a chain with ≥1000 blocks.
2. From a single peer, open a LightClient session and send a tight loop:
```rust
loop {
    let hashes = (1..=1000)
        .map(|n| snapshot.get_header_by_number(n).unwrap().hash())
        .collect::<Vec<_>>();
    let msg = packed::GetBlocksProof::new_builder()
        .last_hash(tip_hash.clone())
        .block_hashes(hashes)
        .build();
    send(msg);
}
```
3. Monitor node disk I/O (`iostat`) and CPU. Observe that honest-peer response latency degrades proportionally to flood rate, with zero ban or throttle applied to the attacker peer, confirming unbounded I/O exhaustion.

    /// Init relay protocol handle

    /// This is a runtime relay protocol shared state, and any relay messages will be processed and forwarded by it
    pub fn new(chain: ChainController, shared: Arc<SyncShared>) -> Self {
        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second
        // current max rps is 10 (ASK_FOR_TXS_TOKEN / TX_PROPOSAL_TOKEN), 30 is a flexible hard cap with buffer
        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
        let rate_limiter = RateLimiter::hashmap(quota);

        Relayer {
            chain,
            shared,
            rate_limiter,
        }
    }

    /// Get shared state
    pub fn shared(&self) -> &Arc<SyncShared> {
        &self.shared
    }

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate
            && self
                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
        }
