Audit Report

## Title
Unbounded `unknown_tx_hashes` Queue Growth via Multi-Peer `RelayTransactionHashes` Flooding — (`sync/src/types/mod.rs`)

## Summary
`SyncState::add_ask_for_txs` inserts up to 32,767 hashes per peer call **before** evaluating the soft-cap check. When the check fires it bans or ignores the current peer but performs no eviction of the already-inserted entries. With N peers each sending 32,767 unique hashes for non-existent transactions, the queue grows to N × 32,767 entries with no removal path, enabling heap exhaustion and node crash.

## Finding Description
The insertion loop runs unconditionally first: [1](#0-0) 

The size check is evaluated only **after** all hashes are already in the queue (the code comment on line 1506 even labels this explicitly): [2](#0-1) 

When the check fires it either returns `StatusCode::TooManyUnknownTransactions` (ban) or `Status::ignored()` — neither branch removes any entries. The inserted hashes remain in the queue permanently until `mark_as_known_txs` is called, which only happens when the node actually receives and processes the transaction: [3](#0-2) 

An attacker sending hashes for non-existent transactions ensures `mark_as_known_txs` is never called. `pop_ask_for_txs` re-inserts every entry it pops: [4](#0-3) 

The relevant constants confirm the arithmetic: [5](#0-4) 

Exploit sequence (mutex-serialized):
- **Peer 1** inserts 32,767 unique hashes → queue = 32,767 (< 50,000) → `Status::ok()`, no check fires.
- **Peer 2** inserts 32,767 unique hashes → queue = 65,534 (≥ 50,000) → check fires, peer 2 has 32,767 entries ≥ `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` → banned. Entries **remain**.
- **Peer N** → queue = N × 32,767. Each peer is banned after insertion; entries are never evicted.

The entry point `TransactionHashesProcess::execute` only validates the per-message count (≤ 32,767) before calling `add_ask_for_txs`: [6](#0-5) 

The second condition `self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` does not help: it fires at the same threshold for N peers and still performs no eviction.

## Impact Explanation
Default configuration sets `max_peers = 125`, `max_outbound_peers = 8`, giving up to 117 inbound slots: [7](#0-6) 

At 125 peers × 32,767 entries = ~4.1 M queue entries. Each `KeyedPriorityQueue` entry holds a `Byte32` key (32 bytes), an `UnknownTxHashPriority` (`Instant` + `Vec<PeerIndex>` + `bool`, ~50 bytes), and internal HashMap/heap bookkeeping (~100+ bytes), totalling ~200 bytes per entry → **~820 MB** of heap growth from inbound peers alone. On VPS or resource-constrained deployments this causes OOM and node crash. Even before OOM, the mutex on `unknown_tx_hashes` serializes all relay processing, causing severe latency degradation. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" (High, 10001–15000 points)**.

## Likelihood Explanation
The attacker needs only the ability to open inbound TCP connections to the target node and send valid `RelayTransactionHashes` protocol messages containing unique 32-byte hashes. No proof-of-work, no keys, no privileged access, and no victim mistakes are required. The attack is repeatable: after a peer is banned, a new connection from a different IP (or after the ban expires) contributes another 32,767 permanent entries. The existing integration test only validates the single-peer ban path and does not assert the global queue bound across multiple peers: [8](#0-7) 

## Recommendation
Move the size check **before** the insertion loop, or enforce a hard cap by refusing to insert new entries when `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE`. A pre-insertion guard such as:

```rust
if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE {
    // run the per-peer counter check and ban/ignore as today
    return ...;
}
```

ensures the queue is hard-bounded regardless of the number of peers. Alternatively, implement LRU/priority-based eviction of the lowest-priority entries when the cap is reached. The per-peer counter check can remain as the ban trigger, but the global queue must be bounded **before** any insertion occurs.

## Proof of Concept
```rust
// Invariant fuzz assertion — fails at peer_id=1 (queue=65534 > 50000)
let state = SyncState::new(...);
for peer_id in 0..10u32 {
    let hashes: Vec<Byte32> = (0..MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        .map(|i| unique_hash(peer_id, i as u32))
        .collect();
    state.add_ask_for_txs(PeerIndex::new(peer_id as usize), hashes);
    let len = state.unknown_tx_hashes().len();
    assert!(
        len <= MAX_UNKNOWN_TX_HASHES_SIZE,
        "Queue at {} after peer {}, exceeds bound {}",
        len, peer_id, MAX_UNKNOWN_TX_HASHES_SIZE
    );
    // FAILS: after peer_id=1, len=65534 > 50000
}
```

Manual steps: connect 2+ peers to a CKB node, have each send a `RelayTransactionHashes` message with 32,767 unique hashes for non-existent transactions, and observe via metrics or memory profiling that `unknown_tx_hashes` grows beyond `MAX_UNKNOWN_TX_HASHES_SIZE` and continues growing with each additional peer.

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "TxHashes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    relay_transaction_hashes.tx_hashes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
