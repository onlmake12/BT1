Audit Report

## Title
Incomplete `ScriptHashType` Filtering in `ScriptHashTypeVerifier` — Output Type Scripts Not Validated Against Enabled Set - (File: verification/src/transaction_verifier.rs)

## Summary
`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the optional **type** script. An unprivileged submitter can craft a transaction whose output carries a type script with an unactivated `ScriptHashType` (e.g., `Data3` = 6) and it will pass this pre-execution gate. The downstream rejection only fires during full VM execution in `extract_script_and_dep_index`, after the transaction has already occupied a tx-pool slot.

## Finding Description
`util/constant/src/consensus.rs` lines 7–12 define the permitted set as `{0, 1, 2, 4}` (Data, Type, Data1, Data2). [1](#0-0) 

`ScriptHashTypeVerifier::verify()` at lines 796–814 loops over outputs and checks only `output.lock().hash_type()`. There is no call to `output.type_()` anywhere in the method. [2](#0-1) 

The downstream guard in `script/src/types.rs` lines 854–859 does reject unactivated hash types, but only during full script execution — after the transaction has already been admitted through the pre-execution pipeline. [3](#0-2) 

The exploit path is: craft a `CellOutput` with a valid lock script (hash_type = 0) and a type script with `hash_type = 6` (Data3) → submit via `send_transaction` → `ScriptHashTypeVerifier` passes (only lock is checked) → transaction enters tx-pool → script execution rejects it at `extract_script_and_dep_index` → transaction is evicted, but the UTXO is not consumed → attacker resubmits. Because the failure occurs before VM execution, eviction is fast, enabling a rapid submit→fail→evict→resubmit cycle with a small number of UTXOs.

## Impact Explanation
This matches the **High** impact class: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs." An attacker with a small number of UTXOs can sustain continuous tx-pool pollution. Because the invalid type script hash type causes a pre-VM failure (fast path), the eviction cycle is rapid, allowing the attacker to keep tx-pool slots occupied with minimal on-chain cost. Legitimate transactions compete for the same pool capacity.

## Likelihood Explanation
Any unprivileged RPC caller can trigger this. Constructing a `packed::Script` with `hash_type = 6` requires a single byte change. No key material, mining power, or privileged access is needed. The `ScriptHashType` enum exposes `Data3` through `Data127` as valid Rust values via the `seq!` macro expansion. The existing test suite confirms the lock-script path is covered (`test_not_enabled_hash_type_output_lock`) but no analogous test exists for type scripts, confirming the gap is untested. [4](#0-3) 

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script against `ENABLED_SCRIPT_HASH_TYPE`, mirroring the existing lock-script check. After the lock check, add:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

Add a corresponding test `test_not_enabled_hash_type_output_type` analogous to the existing `test_not_enabled_hash_type_output_lock`.

## Proof of Concept
```rust
let malicious_type_script = Script::default()
    .as_builder()
    .hash_type(ScriptHashType::Data3) // value = 6, not in ENABLED_SCRIPT_HASH_TYPE
    .build();

let output = CellOutput::new_builder()
    .lock(Script::default())           // valid lock (Data = 0)
    .type_(Some(malicious_type_script).pack())
    .capacity(capacity_bytes!(100))
    .build();

let transaction = TransactionBuilder::default()
    .output(output)
    .output_data(Bytes::default())
    .build();

let verifier = ScriptHashTypeVerifier::new(&transaction);
assert!(verifier.verify().is_ok()); // passes — gap confirmed
```

The existing `test_not_enabled_hash_type_output_lock` at lines 100–122 of `verification/src/tests/transaction_verifier.rs` confirms the lock path is guarded; no equivalent test for type scripts exists, confirming the gap is live and untested. [4](#0-3)
