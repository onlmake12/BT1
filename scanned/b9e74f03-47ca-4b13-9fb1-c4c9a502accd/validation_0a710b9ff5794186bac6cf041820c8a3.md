Audit Report

## Title
Tx-Pool Minimum Fee Check Uses Only Serialized Size, Ignoring Cycles Weight — Allows High-Cycles Transactions to Bypass Minimum Fee Rate - (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size as the weight denominator. The actual transaction weight used everywhere else in the system is `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns the actual cycles, no second fee-rate check is performed using the true weight. An attacker can craft a high-cycles, small-size transaction, pay a fee that only satisfies the size-based minimum, and have the transaction accepted into the pool with an effective fee rate far below `min_fee_rate`, while consuming significant validator CPU during verification.

## Finding Description

In `tx-pool/src/util.rs` at line 45, `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code's own comment acknowledges the inconsistency. The weight denominator is `tx_size` (serialized bytes only).

However, everywhere else in the system — fee-rate sorting, block assembly, fee estimation — the weight is computed via `get_transaction_weight` in `util/types/src/core/tx_pool.rs` at lines 298–303:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This weight is used for `TxEntry::fee_rate()` at `tx-pool/src/component/entry.rs` lines 115–118 and for `AncestorsScoreSortKey` at lines 221–231.

The critical gap is in `_process_tx` at `tx-pool/src/process.rs` lines 724–751: after `verify_rtx` returns `verified` with the actual cycles, the code immediately creates `TxEntry::new(rtx, verified.cycles, fee, tx_size)` at line 751 with no second fee-rate check using the actual weight. The `check_tx_fee` call happens only in `pre_check` (line 289), before script execution.

The same inconsistency exists in `calculate_min_replace_fee` at `tx-pool/src/pool.rs` line 103:

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
```

This also uses `size` only, not `get_transaction_weight(size, cycles)`, meaning a high-cycles replacement transaction also underpays the RBF surcharge.

## Impact Explanation

An attacker submits a transaction with small serialized size (e.g., 200 bytes) and high cycles (up to `max_tx_verify_cycles`). The fee check passes because `min_fee_rate.fee(200) = 200 shannons` at the default 1,000 shannons/KB. The actual weight is `max(200, 70,000,000 × 0.000_170_571_4) ≈ 11,940 bytes`, giving an effective fee rate of approximately 16.7 shannons/KB — roughly 60× below the configured minimum.

Such transactions:
1. Pass `check_tx_fee` and trigger expensive script execution in `verify_rtx`
2. Enter the pool consuming significant validator CPU (up to 70M cycles per tx)
3. Are sorted by their true (low) fee rate, crowding out legitimate transactions
4. Can be submitted in bulk to exhaust pool capacity and validator resources at a fraction of the intended cost

This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

Any unprivileged RPC caller (`send_transaction`) or P2P transaction relayer can exploit this. No special role, key, or majority hashpower is required. The attacker deploys a CKB-VM script that consumes many cycles (e.g., a tight loop) stored as a confirmed cell dep (keeping the transaction's serialized size small), then submits transactions with fees just above `min_fee_rate × tx_size`. The attack is repeatable and low-cost relative to the computational burden imposed on validators.

## Recommendation

After `verify_rtx` returns the actual cycles in `_process_tx`, perform a second fee-rate check using the true weight before creating the `TxEntry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Apply the same fix to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, replacing `self.config.min_rbf_rate.fee(size as u64)` with `self.config.min_rbf_rate.fee(get_transaction_weight(size, cycles))`, where `cycles` is the cycles of the replacing transaction's `TxEntry`.

## Proof of Concept

1. Deploy a CKB-VM script that runs a tight loop consuming ~69,000,000 cycles. Store it in a confirmed cell (cell dep, not inline — keeping the transaction's serialized size small, e.g., ~300 bytes).
2. Construct a transaction spending any live cell, referencing the loop script as a type script, with outputs capacity = inputs capacity − 300 shannons (fee = 300 shannons).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` at `tx-pool/src/util.rs:45` computes `min_fee = min_fee_rate.fee(300) = 300 shannons` → passes.
5. `verify_rtx` executes the script, consuming ~69M cycles.
6. `TxEntry` is created at `tx-pool/src/process.rs:751` with `cycles ≈ 69,000,000`, giving actual weight `max(300, 69,000,000 × 0.000_170_571_4) ≈ 11,769 bytes` and effective fee rate `300 / 11,769 × 1,000 ≈ 25.5 shannons/KB` — far below the 1,000 shannons/KB minimum.
7. The transaction is accepted into the pool. Repeat to flood the pool with computationally expensive, underpaying transactions.
