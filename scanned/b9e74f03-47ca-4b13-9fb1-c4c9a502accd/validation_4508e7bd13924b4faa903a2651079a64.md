Audit Report

## Title
Unbounded `pending_compact_blocks` Growth via Trivially-Mined Compact Blocks — (`sync/src/relayer/compact_block_process.rs`)

## Summary
The `pending_compact_blocks` map accepts and stores compact block entries whenever reconstruction yields `Missing` or `Collided`, with no size cap, TTL, or per-peer limit. Because the `PowVerifier` validates PoW against the `compact_target` embedded in the header itself rather than the epoch's expected difficulty, an attacker can set `compact_target = 0x207fffff` (maximum target) and satisfy PoW with any nonce, generating arbitrarily many distinct valid-looking headers at negligible cost. Each such header that references a missing transaction inserts a permanent entry into the map, causing unbounded heap growth that can crash the node.

## Finding Description

**PoW uses the header's own `compact_target`, not the epoch's expected difficulty.**

In `pow/src/eaglesong.rs`, the verifier derives the target directly from the header field:

```rust
let (block_target, overflow) = compact_to_target(header.raw().compact_target().into());
``` [1](#0-0) 

Setting `compact_target = 0x207fffff` yields the maximum possible target, so the Eaglesong output of virtually any nonce satisfies the inequality `output <= block_target`. The `HeaderVerifier` used in `contextual_check` calls `PowVerifier` with this same logic and never cross-checks the header's `compact_target` against the epoch's expected difficulty: [2](#0-1) 

The `EpochVerifier` called from `HeaderVerifier::verify` only checks structural well-formedness and epoch succession — it performs no `compact_target` validation: [3](#0-2) 

**Height window is the only staleness guard, and it is wide.**

`non_contextual_check` rejects blocks only if `header.number() < tip - epoch_length` (~1000 blocks). An attacker targeting `tip + 1` always passes: [4](#0-3) 

**Unconditional insertion on `Missing` or `Collided`.**

`missing_or_collided_post_process` inserts into the map with no capacity check: [5](#0-4) 

**The map type is an unbounded `HashMap`.** [6](#0-5) 

Initialized as a plain `HashMap::default()` with no capacity limit: [7](#0-6) 

**The only eviction path requires full block reconstruction.**

Entries are removed only in the `ReconstructionResult::Block` branch. The `shrink_to_fit!` call on line 117 compacts allocation but does not remove entries. There is no TTL sweep, no periodic cleanup, and no per-peer cap anywhere: [8](#0-7) 

**Deduplication is per `(block_hash, peer)`, not per peer.**

The pending check only blocks the same hash from the same peer: [9](#0-8) 

Different nonces produce different block hashes, so a single peer sending N headers with distinct nonces inserts N independent entries.

## Impact Explanation

Each `pending_compact_blocks` entry holds a full `CompactBlock` (header + short IDs + prefilled transactions + proposals). With no eviction until a legitimate block is accepted, an attacker can accumulate thousands of entries consuming megabytes of heap. Additionally, `pending_compact_blocks` is protected by a `tokio::sync::Mutex`; a bloated map degrades every legitimate compact block relay that must acquire this lock. Sustained attack leads to OOM and node crash.

**Impact: High — Vulnerabilities which could easily crash a CKB node (10001–15000 points).**

## Likelihood Explanation

The attack requires no privileged access and no stake. The only cost is generating headers with `compact_target = 0x207fffff` against the current tip's parent hash — any nonce satisfies PoW at this target, so a single commodity machine produces thousands of valid headers per second. The attacker needs only one known valid parent block hash (the public chain tip) and can vary the nonce to produce arbitrarily many distinct block hashes. The attack is repeatable indefinitely and does not require victim mistakes.

## Recommendation

1. **Add a size cap** on `pending_compact_blocks`: reject insertion when the map exceeds a constant (e.g., 512 entries).
2. **Add a TTL sweep**: periodically evict entries older than a configurable timeout (e.g., 30 seconds) regardless of whether their transactions arrived.
3. **Add a per-peer entry limit**: track how many pending entries each peer has contributed and disconnect or ignore peers that exceed a threshold.
4. **Verify `compact_target` against the epoch** inside `contextual_check`, mirroring the check in `contextual_block_verifier.rs`'s `EpochVerifier`, so headers with an anomalously easy target are rejected before insertion.

## Proof of Concept

```
1. Connect to a CKB node as a relay peer.
2. Obtain the current tip block hash T and its epoch info.
3. For i in 1..1000:
     a. Build a header: parent_hash=T, number=tip+1,
        compact_target=0x207fffff, epoch=successor(tip.epoch),
        timestamp=tip.median_time+1, nonce=i
        (trivially satisfies PowVerifier since block_target is maximal).
     b. Wrap in a CompactBlock with one short_id referencing a
        transaction not in the node's mempool (guaranteed Missing).
     c. Send via the relay protocol.
4. Observe len(pending_compact_blocks) == 1000 and growing.
5. Repeat indefinitely; the map never shrinks until a legitimate block
   is accepted and its epoch evicts older entries.
```

                parent_hash: header.parent_hash(),

        EpochVerifier::new(parent_fields.epoch, header).verify()?;

            self.data_loader,
            header,
            self.consensus.median_time_block_count(),
        )
        .verify()?;
        Ok(())
    }
