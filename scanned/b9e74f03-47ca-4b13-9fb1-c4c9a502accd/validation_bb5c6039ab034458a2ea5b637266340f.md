Audit Report

## Title
Stale `evict_key` on Surviving Ancestors After Batch Removal — (`tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::remove_entry_and_descendants`, `remove_entry_links` is called for every entry in the batch before any `remove_entry` call. This destroys the link graph entries that `update_ancestors_index_key` depends on to locate surviving ancestors. As a result, when `remove_entry` runs and calls `update_ancestors_index_key`, `calc_ancestors` returns an empty set and the surviving parents of the removed root transaction never receive the `evict_key` correction that subtracts the removed descendants' fee-rate contribution. Those parents retain a permanently inflated `evict_key`, corrupting the pool's eviction-priority ordering.

## Finding Description

`remove_entry_and_descendants` (L252–265) first iterates all `removed_ids` and calls `remove_entry_links` on each:

```rust
for id in &removed_ids {
    self.remove_entry_links(id);   // tears down ALL link records first
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` (L418–430) does three things for the root entry:
1. Calls `self.links.remove_child(&parent, root_id)` for every surviving parent — removing root from each parent's children set.
2. Calls `self.links.remove_parent(&child, root_id)` for every descendant.
3. Calls `self.links.remove(root_id)` — **deletes the root's entire record from `self.links.inner`**.

Later, when `remove_entry` runs for the root and calls `update_ancestors_index_key` (L432–445):

```rust
let ancestors: HashSet<ProposalShortId> =
    self.links.calc_ancestors(&child.proposal_short_id());
```

`calc_ancestors` → `calc_relative_ids` → `self.inner.get(root_id)` returns `None` because the record was already deleted. The returned set is empty; the `for anc_id in &ancestors` loop is never entered. Surviving parents of the root — which are **not** in `removed_ids` — never have `sub_descendant_weight` called on them and never have their `evict_key` recomputed.

`EvictKey` is computed as `max(descendants_feerate, own_feerate)` with `descendants_count` (L234–247 of `entry.rs`). After the batch removal, each surviving parent `P` retains:
- `evict_key.fee_rate` = old `max(descendants_feerate, own_feerate)` instead of just `own_feerate`
- `evict_key.descendants_count` still inflated

`next_evict_entry` (L380–385) iterates `iter_by_evict_key()` ascending and picks the first entry matching the requested status. A stale high `evict_key` on `P` places it toward the high end of the eviction index, so it is skipped when the pool needs to shed low-value entries.

The comment at L256 acknowledges only half the intent: *"update links state for remove, so that we won't update_descendants_index_key in remove_entry"* — it prevents redundant updates to entries being removed, but simultaneously suppresses the necessary update to entries that are **not** being removed.

## Impact Explanation

This is a suboptimal implementation of CKB's transaction pool state management mechanism (**Medium, 2001–10000 points**). The eviction index is permanently corrupted for any surviving ancestor of a batch-removed subtree. Low-fee-rate transactions that should be first candidates for eviction are instead ranked as high-value entries and retained. Legitimate higher-fee transactions submitted when the pool is full may be rejected because the pool appears occupied by entries that should have already been evicted. The corruption persists for the lifetime of the affected entry in the pool.

## Likelihood Explanation

The trigger is any code path that calls `remove_entry_and_descendants` on a transaction that has at least one in-pool parent not included in the removal batch. This occurs naturally during normal node operation via `resolve_conflict` (L305–332) and `resolve_conflict_header_dep` (L267–292) — both called when a new block is connected. No special attacker capability is required; a chain of transactions where a child conflicts with a confirmed transaction is sufficient to trigger the bug. It is repeatable on every block that confirms a transaction whose pool-descendants are evicted while its pool-ancestors survive.

## Recommendation

Before calling `remove_entry_links` in the batch loop, first collect the surviving ancestors of the root and perform the `evict_key` update for each. Concretely, compute `calc_ancestors(root_id)` (which excludes the removed set) before any link teardown, then call `sub_descendant_weight` and recompute `evict_key` for each surviving ancestor for every entry being removed. Alternatively, restructure `remove_entry_and_descendants` to call `update_ancestors_index_key` for the root entry while the link graph is still intact, before the batch `remove_entry_links` loop runs.

Construct a unit test in `tx-pool/src/component/pool_map.rs`:

1. Insert three transactions: grandparent `G` (low fee rate), parent `P` (low fee rate, child of `G`), child `C` (high fee rate, child of `P`).
2. Record `P`'s `evict_key` — it should be inflated by `C`'s high fee rate.
3. Call `pool_map.remove_entry_and_descendants(&P.id)` — removes `P` and `C`.
4. Assert that `G`'s `evict_key` has been updated to reflect only `G`'s own fee rate (i.e., `evict_key.fee_rate == G.own_feerate` and `evict_key.descendants_count == 1`).
5. Without the fix, the assertion fails: `G.evict_key` still contains the inflated values from the now-removed `P` and `C`.
