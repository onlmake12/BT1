Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequest` Enables `pending_delivered` Cache Poisoning and Unbounded `try_nat_traversal` Task Spawning — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

`ConnectionRequestProcess::execute` accepts the `from` PeerId entirely from the wire message and never verifies it against the actual sending session's identity. When the victim node is the `to` target, it unconditionally writes attacker-supplied listen addresses into `pending_delivered[content.from]`. A subsequent `ConnectionSync` message (also unauthenticated, sendable by the same attacker) causes the victim to spawn unbounded `try_nat_traversal` async tasks against attacker-controlled endpoints, each retrying TCP connections for 30 seconds. With the per-session rate limit of 30 messages/second, an attacker sustains 900 concurrent retry tasks, exhausting file descriptors and async runtime resources sufficient to crash the node.

## Finding Description

**Root cause — no sender authentication on `from`:**

`RequestContent::try_from` parses `from` directly from the wire message with no check that it matches the actual session's peer identity. [1](#0-0) 

`self.peer` (the actual `PeerIndex` of the sending session) is stored in `ConnectionRequestProcess` but is never compared against `content.from` anywhere in `execute` or `respond_delivered`. [2](#0-1) 

**Cache write path:**

When `self_peer_id == &content.to`, `respond_delivered` is called with the unauthenticated `content.from` and attacker-supplied `content.listen_addrs`. [3](#0-2) 

Inside `respond_delivered`, the only guard is a 2-minute time-window check on the existing entry. There is no identity check. [4](#0-3) 

After a successful `send_message_to` back to the attacker's session, the poisoned entry is written unconditionally. [5](#0-4) 

`remote_listens` is filtered only for TCP/IP transport type — not for address ownership. [6](#0-5) 

**`pending_delivered` has no size bound:**

`pending_delivered` is a plain `HashMap<PeerId, PendingDeliveredInfo>` with no capacity limit. Cleanup only runs every 5 minutes via `notify()`, retaining entries newer than 5 minutes. [7](#0-6) [8](#0-7) 

**Consumption path — `ConnectionSync`:**

`ConnectionSyncProcess::execute` also accepts `content.from` from the wire message without authentication. When `self_peer_id == content.to` and `content.route` is empty, it looks up `pending_delivered[content.from]` and spawns `try_nat_traversal` for each stored address. [9](#0-8) [10](#0-9) 

**`try_nat_traversal` resource cost:**

Each spawned task retries TCP connections for up to 30 seconds at ~200 ms intervals, creating a new `TcpSocket` on every iteration. [11](#0-10) [12](#0-11) 

**Rate limiter bypass:**

The per-session `rate_limiter` allows 30 messages/second per `(session_id, msg.item_id())`. [13](#0-12) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1/second. [14](#0-13) 

Each new spoofed `from` PeerId is a distinct key, so the forward rate limiter is bypassed entirely. The attacker can send 30 `ConnectionRequest` + 30 `ConnectionSync` messages per second, each with a fresh spoofed `from`, poisoning 30 entries and spawning 30 `try_nat_traversal` tasks per second.

## Impact Explanation

After 30 seconds of sustained attack, 900 concurrent `try_nat_traversal` async tasks are running on the victim. Each task creates a new `TcpSocket` every ~400 ms. This produces ~2,250 socket creation attempts per second and sustains ~900 open file descriptors at all times solely from this attack. Combined with the victim's normal P2P connections, this exhausts the process file descriptor limit (commonly 1024 on default Linux configurations), causing `create_socket` to return `Err` and propagating failures throughout the node's networking layer. Additionally, `pending_delivered` grows at 30 entries/second with no bound, consuming unbounded heap memory (each entry holds up to 24 `Multiaddr` values). This constitutes **"Vulnerabilities which could easily crash a CKB node"** (High, 10001–15000 points).

## Likelihood Explanation

- Requires only a standard P2P connection to the victim — no special privileges, no proof-of-work, no key material.
- The victim's own peer ID (needed as `content.to`) is publicly observable from the P2P network.
- Spoofed `from` PeerIds need only be valid-format byte sequences; they do not need to correspond to real peers.
- The 2-minute re-entry window is trivially satisfied by using a fresh spoofed `from` PeerId for each message.
- The attack is fully repeatable and persistent with minimal bandwidth (30 small messages/second).
- A single attacker session is sufficient; multiple sessions multiply the effect linearly.

## Recommendation

In `respond_delivered`, resolve the actual sender's `PeerId` from the session registry and reject the message if `content.from` does not match:

```rust
let actual_sender_peer_id = self.protocol.network_state
    .peer_registry
    .read()
    .get_peer(self.peer)
    .map(|p| p.peer_id.clone());

if actual_sender_peer_id.as_ref() != Some(&from_peer_id) {
    return StatusCode::Ignore.with_context("from field does not match actual sender");
}
```

Additionally, enforce a maximum capacity on `pending_delivered` (e.g., bounded by the number of connected peers) to prevent unbounded memory growth from any residual unauthenticated path. [15](#0-14) 

## Proof of Concept

**Minimal state-transition sequence (manual steps):**

1. Attacker establishes a standard P2P session `S` to the victim node.
2. For `N = 1..30` each second, attacker sends over session `S`:
   ```
   ConnectionRequest {
     from = fresh_random_peer_id_N,
     to   = victim_peer_id,
     listen_addrs = [attacker_ip:attacker_port],
     route = [],
     max_hops = 6
   }
   ```
   Each message passes the per-session rate limiter (30/sec) and the forward rate limiter (new key per `from`). Victim writes `pending_delivered[peer_id_N] = ([attacker_ip:attacker_port], now)`.

3. For the same `N`, attacker immediately sends:
   ```
   ConnectionSync {
     from = fresh_random_peer_id_N,
     to   = victim_peer_id,
     route = []
   }
   ```
   Victim finds `pending_delivered[peer_id_N]`, spawns `try_nat_traversal(bind_addr, attacker_ip:attacker_port)`.

4. After 30 seconds: 900 concurrent `try_nat_traversal` tasks running on the victim, each creating a `TcpSocket` every ~400 ms. File descriptor exhaustion occurs; the node's networking layer begins failing on new socket allocations, crashing the node.

**Invariant test:** Assert that after processing 1000 `ConnectionRequest` messages from a single session with distinct spoofed `from` values, `pending_delivered.len()` remains bounded by the number of authenticated peers — this invariant currently fails. [16](#0-15)
