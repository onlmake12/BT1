### Title
Copy-paste bug in `script_exists_in_output` reads `row_lock` instead of `row_type` in second match arm, causing type-script rows to be incorrectly deleted from the script table during block rollback on PostgreSQL — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary
A copy-paste error at line 252 of `remove.rs` causes `script_exists_in_output` to evaluate the PostgreSQL `EXISTS` result for `lock_script_id` a second time instead of evaluating the result for `type_script_id`. During any block rollback on a PostgreSQL-backed rich-indexer node, scripts that are referenced only as type scripts in surviving outputs are incorrectly garbage-collected, permanently corrupting the `script` table until a full re-sync.

### Finding Description
`script_exists_in_output` is supposed to return `true` if `script_id` appears in any remaining output as either a lock script or a type script. It issues two queries and stores results in `row_lock` and `row_type`. The first match (line 223) correctly inspects `row_lock`. The second match (line 252) should inspect `row_type` but instead re-reads `row_lock`. On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds with `false` at this point (the `true` case was already handled by the early return), so the function unconditionally returns `Ok(false)` for any script not used as a lock script, regardless of whether it is still used as a type script.

### Impact Explanation
Any block rollback (reorg) on a PostgreSQL rich-indexer node will silently delete `script` table rows for scripts that are used only as type scripts in surviving outputs. All subsequent `get_cells` / `get_transactions` RPC calls that join on `script.id` will return empty results for those cells, causing incorrect cell liveness reporting. Wallets and dApps relying on the indexer will be unable to locate those cells, potentially preventing users from constructing valid transactions spending them.

### Likelihood Explanation
Chain reorganizations are a normal part of CKB network operation and require no attacker involvement. Any operator running the rich indexer with a PostgreSQL backend will encounter this bug during the first reorg that involves outputs with type scripts. The SQLite backend is not affected due to the type mismatch causing the fallback path to be taken.

### Recommendation
Change line 252 from:
```rust
match row_lock.try_get::<bool, _>(0) {
```
to:
```rust
match row_type.try_get::<bool, _>(0) {
```
This makes the second match arm correctly evaluate the `type_script_id` existence query result, consistent with the intent of the function and the SQLite fallback path on line 255.

### Proof of Concept
1. Start a CKB node with the rich indexer configured to use PostgreSQL.
2. Append a block containing a transaction with one output whose `type_script` is script `S` (and whose `lock_script` is a different script).
3. Roll back that block (trigger a reorg).
4. Query `SELECT COUNT(*) FROM script WHERE id = <id of S>` — on a correct implementation this returns 1 if `S` is still referenced by any surviving output; with the bug it returns 0 even if `S` is still referenced.
5. Issue a `get_cells` RPC for a cell whose type script is `S` — the result will be empty despite the cell existing in the chain.
