### Title
No Access Control on State-Mutating RPC Setters Generated by the `#[rpc]` Macro — (`rpc/src/server.rs`, `rpc/src/module/pool.rs`, `rpc/src/module/net.rs`)

---

### Summary

The CKB JSON-RPC server exposes multiple state-mutating methods (`clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`, `clear_banned_addresses`, `set_ban`, `set_network_active`) that are registered via the `#[rpc]` macro without any authentication or authorization guard. The RPC server itself (`rpc/src/server.rs`) contains no authentication middleware — no token check, no IP allowlist, no credential validation at the application layer. Any caller who can reach the HTTP/TCP/WebSocket RPC port can invoke these destructive setters unconditionally.

---

### Finding Description

**Root cause — no authentication in the RPC server:**

The `handle_jsonrpc` function in `rpc/src/server.rs` dispatches every incoming request directly to the `MetaIoHandler` with no authentication step:

```rust
async fn handle_jsonrpc<T: Default + Metadata>(
    Extension(io): Extension<Arc<MetaIoHandler<T>>>,
    req_body: Bytes,
) -> Response {
    // ... parse JSON ...
    let result = io.handle_call(call, T::default()).await;
    // No auth check anywhere
}
``` [1](#0-0) 

The server is built with `CorsLayer::permissive()`, accepting requests from any origin: [2](#0-1) 

The `RpcConfig` struct has no authentication fields — no token, no allowed-IP list, no credential option: [3](#0-2) 

**Root cause — macro-generated handlers with no access control:**

The `#[rpc(name = "...")]` macro registers each method as a publicly callable handler. The following state-mutating methods carry no guard:

- `clear_tx_pool` — wipes all pending transactions from the mempool: [4](#0-3) 

- `clear_tx_verify_queue` — wipes the entire verification queue: [5](#0-4) 

- `remove_transaction` — removes a specific transaction by hash: [6](#0-5) 

- `clear_banned_addresses` — erases the entire peer ban list: [7](#0-6) 

- `set_ban` — inserts or deletes IP/subnet bans: [8](#0-7) 

- `set_network_active` — disables or enables all P2P networking: [9](#0-8) 

- `update_main_logger` / `set_extra_logger` — modifies runtime logging configuration: [10](#0-9) 

None of these implementations contain any caller-identity check. The `#[rpc]` macro is the structural analog to the openbrush `accessors` macro: it generates externally-callable handlers with no modifier/guard capability.

**Attacker entry path:**

The default `listen_address = "127.0.0.1:8114"` is a configuration default, not an enforced security control. The configuration explicitly supports binding to non-localhost addresses (e.g., `0.0.0.0:8114`), which is a documented and supported deployment scenario for mining pools, exchanges, and dApp backends: [11](#0-10) 

When a node operator exposes the RPC port for legitimate remote access, there is no application-layer authentication mechanism available to them — not even an optional token. The only defense is external network controls (firewall, reverse proxy), which the CKB application itself cannot enforce or verify.

---

### Impact Explanation

An attacker who can reach the RPC port (network-accessible deployment, SSRF, misconfigured firewall, or shared hosting) can:

1. **`clear_tx_pool`**: Wipe all pending transactions from the mempool. For a mining node, this destroys the block assembly queue, causing revenue loss and transaction propagation failure. For a relay node, it disrupts the network's transaction propagation.

2. **`set_network_active(false)`**: Completely isolate the node from the P2P network. The node stops syncing, stops relaying blocks and transactions, and becomes a dead endpoint. This is a targeted node-isolation attack.

3. **`clear_banned_addresses`**: Erase all IP bans, immediately re-enabling previously-banned malicious peers (eclipse attackers, DoS sources) to reconnect.

4. **`set_ban` with `"0.0.0.0/0"`**: Ban the entire IP space, effectively cutting off all P2P connections.

5. **`remove_transaction`**: Selectively evict specific transactions from the pool, enabling targeted transaction censorship or fee-sniping attacks.

Severity: **High** — service unavailability, transaction loss, and network isolation are all achievable with a single unauthenticated HTTP POST.

---

### Likelihood Explanation

- Any node operator who exposes the RPC port beyond localhost (for remote mining, exchange integration, or dApp use) is immediately vulnerable.
- The attack requires only HTTP access to the RPC port — no credentials, no cryptographic material, no protocol knowledge beyond a single JSON-RPC call.
- The `CorsLayer::permissive()` setting means browser-based SSRF (cross-origin fetch) can also reach the endpoint if the victim's browser is on the same network.
- The RPC README itself acknowledges the risk but provides no mitigation within the software: *"Allowing arbitrary machines to access the JSON-RPC port is dangerous and strongly discouraged."* — this is documentation, not enforcement. [12](#0-11) 

---

### Recommendation

1. **Add an optional authentication token** to `RpcConfig` (e.g., `auth_token: Option<String>`). When set, the `handle_jsonrpc` middleware must validate a `Authorization: Bearer <token>` header before dispatching any call.
2. **Separate read-only and state-mutating method sets** at the module level, so operators can expose read methods publicly while restricting write methods to authenticated callers.
3. **Add an IP allowlist field** to `RpcConfig` enforced at the server layer, not just as a documentation recommendation.
4. At minimum, gate `clear_tx_pool`, `clear_tx_verify_queue`, `set_network_active`, `clear_banned_addresses`, and `set_ban` behind a configurable access control check within the handler, analogous to adding a modifier to the openbrush setter.

---

### Proof of Concept

**Precondition:** Target node has `listen_address = "0.0.0.0:8114"` (supported configuration) and the `Pool` and `Net` modules enabled (default).

**Step 1 — Wipe the mempool:**
```bash
curl -X POST http://TARGET:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}'
# Response: {"jsonrpc":"2.0","result":null,"id":1}
# All pending transactions are now gone.
```

**Step 2 — Isolate the node from P2P:**
```bash
curl -X POST http://TARGET:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":2}'
# Response: {"jsonrpc":"2.0","result":null,"id":2}
# Node is now disconnected from all peers and stops syncing.
```

**Step 3 — Clear the ban list (re-enable previously-banned attackers):**
```bash
curl -X POST http://TARGET:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_banned_addresses","params":[],"id":3}'
# Response: {"jsonrpc":"2.0","result":null,"id":3}
```

No authentication, no token, no prior knowledge required. The `handle_jsonrpc` dispatcher at `rpc/src/server.rs:258` executes the call unconditionally. [13](#0-12)

    fn set_ban(
        &self,
        address: String,
        command: String,
        ban_time: Option<Timestamp>,
        absolute: Option<bool>,
        reason: Option<String>,
    ) -> Result<()> {
        let ip_network = address.parse().map_err(|_| {
            RPCError::invalid_params(format!(
                "Expected `params[0]` to be a valid IP address, got {address}"
            ))
        })?;

        match command.as_ref() {
            "insert" => {
                let ban_until = if absolute.unwrap_or(false) {
                    ban_time.unwrap_or_default().into()
                } else {
                    unix_time_as_millis()
                        + ban_time
                            .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                            .value()
                };
                self.network_controller
                    .ban(ip_network, ban_until, reason.unwrap_or_default());
                Ok(())
            }
            "delete" => {
                self.network_controller.unban(&ip_network);
                Ok(())
            }
            _ => Err(RPCError::invalid_params(format!(
                "Expected `params[1]` to be in the list [insert, delete], got {address}"
            ))),
        }
    }
