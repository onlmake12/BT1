### Title
Unvalidated Remote-Controlled `observed_addr` in Identify Protocol Poisons Victim Node's Advertised Address List — (`network/src/protocols/identify/mod.rs`, `network/src/network.rs`)

---

### Summary

An unprivileged remote peer can send an Identify message containing an attacker-chosen `observed_addr` (any valid global IP Multiaddr). The victim node stores it without reachability validation and subsequently re-broadcasts it as its own listen address to every peer it connects to, causing network-wide address table pollution.

---

### Finding Description

**Entrypoint:** Any connected peer sends a P2P Identify message with a crafted `observed_addr`.

**Trace:**

`IdentifyProtocol::received()` decodes the message and calls `process_observed()`: [1](#0-0) 

`process_observed()` calls `IdentifyCallback::add_observed_addr()` with no prior validation: [2](#0-1) 

The only operation performed is appending the local peer ID if absent. No `is_reachable()` check, no IP class check, no format validation. The address is then stored unconditionally: [3](#0-2) 

**Propagation path 1 — Identify `connected()`:**

`IdentifyCallback::local_listen_addrs()` merges raw `observed_addrs` into the advertised list: [4](#0-3) 

When the victim opens a new session, `connected()` applies an `is_reachable()` filter before sending: [5](#0-4) 

This filter blocks loopback (127.0.0.1) and RFC1918 addresses. However, **any valid globally-routable IP** (e.g., `/ip4/1.2.3.4/tcp/8114`) passes `is_reachable()` and is broadcast verbatim to every subsequent peer.

**Propagation path 2 — HolePunching `ConnectionDelivered` (completely unfiltered):**

`connection_request.rs` directly includes `observed_addrs` in the HolePunching response with **no reachability filter at all**: [6](#0-5) 

This second path propagates even RFC1918/loopback addresses to peers participating in hole punching.

**Contrast with `process_listens()`:**

The listen-address processing path correctly applies `is_reachable()` before storing remote peers' addresses: [7](#0-6) 

The `observed_addr` path has no equivalent guard.

---

### Impact Explanation

- The victim node advertises attacker-controlled global IP addresses as its own listen addresses to every peer it connects to.
- Receiving peers store these addresses in their peer stores and may re-broadcast them further via the Discovery protocol, causing **network-wide address table pollution**.
- Nodes across the network waste connection attempts on addresses the victim does not own.
- A sophisticated attacker can redirect connection attempts to an attacker-controlled IP, enabling man-in-the-middle positioning or amplified DoS against a third-party target.
- The HolePunching path leaks even private/loopback addresses to hole-punching participants.

---

### Likelihood Explanation

- Requires only a standard P2P connection — no special privileges, no PoW, no key material.
- The Identify protocol runs on every session open; a single connected peer can poison the victim immediately.
- The attack is silent: no ban, no disconnect, `add_observed_addr` always returns `MisbehaveResult::Continue`. [8](#0-7) 

---

### Recommendation

1. In `IdentifyCallback::add_observed_addr()`, apply `is_reachable()` (and optionally `multiaddr_to_socketaddr()` existence check) before calling `NetworkState::add_observed_addr()`, mirroring the guard already present in `process_listens()`.
2. In `NetworkState::observed_addrs()`, filter out non-reachable addresses before returning them, so all consumers (Identify and HolePunching) are protected uniformly.
3. In `connection_request.rs` `respond_delivered()`, apply the same `is_reachable()` filter to `observed_addrs` before including them in the HolePunching response.

---

### Proof of Concept

```rust
// Unit test sketch (no network needed)
let network_state = build_test_network_state();
let session_id = SessionId::new(1);

// Attacker-controlled global IP — passes is_reachable()
let poison: Multiaddr = "/ip4/1.2.3.4/tcp/8114".parse().unwrap();
network_state.add_observed_addr(session_id, poison.clone());

// Simulate local_listen_addrs() with public_addrs below MAX_RETURN_LISTEN_ADDRS
let mut callback = build_test_identify_callback(network_state.clone());
let addrs = callback.local_listen_addrs();

// Assert: poisoned address IS present — demonstrating the bug
assert!(addrs.iter().any(|a| a.to_string().contains("1.2.3.4")),
    "Poisoned address must not appear in advertised listen addrs");
```

The test will pass (i.e., the poisoned address IS present), confirming the invariant is broken for globally-routable attacker-controlled addresses.
