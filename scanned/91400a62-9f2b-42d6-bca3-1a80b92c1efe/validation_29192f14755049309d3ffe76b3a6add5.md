### Title
`total_tx_size` / `total_tx_cycles` Updated Incorrectly When Ancestor-Eviction Occurs During `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the aggregate pool statistics `total_tx_size` and `total_tx_cycles` are computed **before** a potential in-pool eviction occurs, then unconditionally written back **after** the eviction has already modified those same fields. The evicted transactions' sizes and cycles are silently dropped from the accounting, leaving the totals permanently inflated. This is the direct CKB analog of the FrankenDAO H-1 bug: an aggregate counter is updated without checking whether intermediate state changes have already modified it.

---

### Finding Description

In `PoolMap::add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221):

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    ...
    // Step 1: snapshot new totals BEFORE any eviction
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

    // Step 2: may evict in-pool txs, which calls remove_entry →
    //         update_stat_for_remove_tx → MODIFIES self.total_tx_size / self.total_tx_cycles
    evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Step 3: OVERWRITE the already-modified fields with the stale pre-eviction values
    self.total_tx_size = total_tx_size;                             // line 218
    self.total_tx_cycles = total_tx_cycles;                         // line 219
    Ok((true, evicts))
}
```

The eviction path inside `check_and_record_ancestors` (lines 615–625) calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` for each evicted transaction. But lines 218–219 then overwrite those correctly-decremented values with the stale snapshot taken at line 210–211, which only added the new entry's size/cycles to the **pre-eviction** totals.

**Concrete arithmetic:**

| Step | `self.total_tx_size` | `self.total_tx_cycles` |
|---|---|---|
| Initial | 1000 | 5000 |
| After `updated_stat_for_add_tx` (local vars) | local=1200 | local=6000 |
| After eviction of tx (size=300, cycles=1500) | 700 | 3500 |
| After line 218-219 overwrite | **1200 (wrong)** | **6000 (wrong)** |

Correct values should be `900` and `4500`. The evicted transaction's contribution is never subtracted.

The eviction path is triggered inside `check_and_record_ancestors` when:
- `ancestors_count > max_ancestors_count`, AND
- `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`

This condition is reachable when a submitted transaction has inputs that consume outputs of in-pool transactions that are also referenced as cell deps by other in-pool transactions (`cell_ref_parents`). [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

`total_tx_size` is used in two critical places:

1. **`limit_size` over-eviction** (`tx-pool/src/pool.rs`, line 298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated `total_tx_size` causes the pool to evict additional legitimate transactions even though the real pool size is within limits. This can cause valid, fee-paying transactions to be permanently rejected with `Reject::Full`.

2. **`updated_stat_for_add_tx` false rejection** (`pool_map.rs`, lines 716–727): The next call to `add_entry` uses the inflated `total_tx_size` as the baseline. If the inflated value already exceeds `max_tx_pool_size`, the overflow check at line 716 will reject the next transaction with `Reject::Full` even though the pool has real space.

3. **Incorrect RPC reporting**: `tx_pool_info` (`tx-pool/src/service.rs`, line 1089) reads `tx_pool.pool_map.total_tx_size` directly, so the `total_tx_size` and `total_tx_cycles` fields returned to callers of the `tx_pool_info` RPC are wrong. [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` requires `cell_ref_parents` to be non-empty and the ancestor count to exceed `max_ancestors_count`. An unprivileged transaction submitter (via the `send_transaction` RPC or P2P relay) can craft this scenario by:

1. Submitting a chain of transactions where transaction A's output is used as a cell dep by transaction B already in the pool.
2. Submitting a new transaction C that spends A's output as an input, making B a `cell_ref_parent` of C.
3. If the ancestor count of C exceeds `max_ancestors_count` (default 25 on mainnet per `ckb.toml`), the eviction path fires, B is evicted, and the bug triggers.

This is a realistic scenario in a busy mempool with deep transaction chains and shared cell deps, which is a normal usage pattern in CKB's cell model. [7](#0-6) [8](#0-7) 

---

### Recommendation

Move the `total_tx_size` / `total_tx_cycles` assignment to **after** `check_and_record_ancestors` completes, and compute the new totals from the **post-eviction** state rather than the pre-eviction snapshot. The simplest correct fix:

```rust
// Remove lines 210-211 (the early snapshot)
// After check_and_record_ancestors, compute totals fresh:
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Now add the new entry's contribution to the post-eviction totals
self.total_tx_size = self.total_tx_size.checked_add(entry.size)
    .ok_or_else(|| Reject::Full(...))?;
self.total_tx_cycles = self.total_tx_cycles.checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(...))?;
```

Alternatively, the overflow check can be done before `check_and_record_ancestors` (as a pre-flight guard), but the actual field assignment must happen after all evictions are complete. [1](#0-0)

### Proof of Concept

**Setup:**
- `max_ancestors_count = 3` (for brevity)
- Pool contains: `tx_A` (size=100), `tx_B` (size=200, cell dep on `tx_A`'s output), `tx_C` (size=150, child of `tx_A`)
- `total_tx_size = 450`

**Attack:**
- Submit `tx_D` with: input spending `tx_A`'s output (making `tx_B` a `cell_ref_parent`), and ancestors `{tx_A, tx_C}` → `ancestors_count = 3 = max_ancestors_count`, so no eviction yet.
- Submit `tx_E` with: input spending `tx_C`'s output, cell dep on `tx_A` → `ancestors_count = 4 > 3`, `cell_ref_parents = {tx_B}`, `4 - 1 = 3 <= 3` → eviction path fires, `tx_B` (size=200) is evicted.

**During `add_entry` for `tx_E` (size=100):**
1. Line 210: `total_tx_size_local = 450 + 100 = 550`
2. `check_and_record_ancestors` evicts `tx_B`: `self.total_tx_size = 450 - 200 = 250`
3. Line 218: `self.total_tx_size = 550` ← **wrong**, should be `350`

**Result:** `total_tx_size` is inflated by 200 (the size of evicted `tx_B`). Subsequent calls to `limit_size` will over-evict, and `updated_stat_for_add_tx` will use the wrong baseline, potentially causing `Reject::Full` for transactions that would otherwise fit. [9](#0-8) [10](#0-9) [11](#0-10)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,

        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {

                self.total_tx_size, tx_size
            ))
        })?;

            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
