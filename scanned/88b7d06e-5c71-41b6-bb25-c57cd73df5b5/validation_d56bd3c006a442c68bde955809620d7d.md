Audit Report

## Title
O(n) Linear Scan Over `unknown_tx_hashes` Under Blocking Mutex on Every `RelayTransactionHashes` Message When Queue Is Full — (File: sync/src/types/mod.rs)

## Summary
`SyncState::add_ask_for_txs` performs a full O(n) linear scan over all entries in the `unknown_tx_hashes` priority queue while holding its blocking mutex whenever the queue reaches capacity (50,000 entries). Two unprivileged peers can cooperatively fill the queue without triggering the per-peer ban, then sustain continuous O(n) scans on every subsequent relay message, causing mutex contention that starves legitimate relay and tx-pool operations.

## Finding Description
The function `add_ask_for_txs` acquires the `unknown_tx_hashes` mutex at entry and holds it until return: [1](#0-0) 

After inserting hashes, when `len >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) or the peer-scaled threshold is met, the code performs a nested linear scan over every queue entry and every peer in each entry's `peers: Vec<PeerIndex>`: [2](#0-1) 

The `peers` field is an unbounded `Vec<PeerIndex>`, so if many peers announce the same hash, the inner loop grows proportionally: [3](#0-2) 

The ban threshold is `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767: [4](#0-3) 

With two peers each contributing ~25,000 distinct fake hashes, the queue reaches 50,000 entries while each peer's individual counter (25,000) stays below the 32,767 ban threshold. The threshold check at line 1507 fires, the O(n) scan runs, and the function returns `Status::ignored()` — no ban issued: [5](#0-4) 

Once saturated, every subsequent `RelayTransactionHashes` message from any peer re-enters the scan path. The entry point is reachable by any unauthenticated peer: [6](#0-5) 

## Impact Explanation
This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** While the mutex is held during the O(50,000) scan, `pop_ask_for_txs`, `mark_as_known_txs`, and `unknown_tx_hashes()` all contend on the same `Mutex`. An attacker sending relay messages at high frequency forces repeated O(n) scans under the mutex, degrading tx-request scheduling and relay throughput on the victim node. The cost to the attacker is minimal: two TCP connections and ~1.6 MB of fake hash data to saturate the queue, then cheap small messages to sustain the condition.

## Likelihood Explanation
Any unprivileged peer on the `RelayV3` protocol can participate. No authentication is required. The setup requires only two connections and two batches of 32-byte fake hashes (~800 KB each). The attacker avoids the per-peer ban by distributing hashes across peers. The queue can be kept saturated indefinitely by periodically re-announcing hashes before natural eviction, since the node cannot distinguish fake hashes from real ones at announcement time.

## Recommendation
1. **Maintain a per-peer counter map** (`HashMap<PeerIndex, usize>`) updated incrementally on insert/remove, replacing the O(n) scan with an O(1) lookup.
2. **Bound `UnknownTxHashPriority::peers`** to a small constant (e.g., 8) to cap inner-loop cost even if the outer loop cannot be eliminated.
3. **Apply the per-peer check before inserting** rather than after, so the queue never grows beyond the threshold due to a single peer's contribution.

## Proof of Concept
1. Attacker opens two peer connections (`peer_A`, `peer_B`) to the target node.
2. `peer_A` sends `RelayTransactionHashes` with 25,000 distinct fake tx hashes → queue grows to 25,000; no threshold crossed; no ban.
3. `peer_B` sends `RelayTransactionHashes` with 25,000 distinct fake tx hashes → queue reaches 50,000; threshold fires; O(50,000) scan runs; `peer_B` counter = 25,000 < 32,767 → `Status::ignored()`, not banned.
4. Attacker sends `RelayTransactionHashes` from `peer_A` (or any additional peer) at high rate → each message acquires the mutex, runs the O(50,000) scan, releases the mutex; `pop_ask_for_txs` and `mark_as_known_txs` callers are blocked waiting for the mutex.
5. Attacker periodically re-announces hashes to replenish evicted entries, sustaining the saturated condition and continuous O(n) scans indefinitely.
