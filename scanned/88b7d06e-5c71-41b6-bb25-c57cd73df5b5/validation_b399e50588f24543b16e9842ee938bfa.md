Audit Report

## Title
Unbounded `GROUP BY` Aggregation Before `LIMIT` in `get_tx_with_cells` Enables Remote DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary

When `group_by_transaction=true`, `get_transactions` constructs a three-level SQL query where `GROUP_CONCAT`/`ARRAY_AGG` aggregates every matching row across the entire indexed chain before the outer `LIMIT` is applied. An unprivileged RPC caller can trigger full-table aggregation with `limit=1`, causing unbounded memory consumption and potential OOM or extreme latency on any node with the rich indexer enabled.

## Finding Description

The query is built in `get_tx_with_cells` (L271–430) in two stages:

**Stage 1** — `build_tx_with_cell_union_sub_query` (L432–541) produces an unbounded `UNION ALL` subquery with no `LIMIT`: [1](#0-0) 

**Stage 2** — that subquery is wrapped in a middle subquery (L281–299, also no `LIMIT`), then the outer query applies `GROUP_CONCAT`/`ARRAY_AGG` and only then applies `LIMIT`: [2](#0-1) 

SQL semantics require the database engine to materialize and aggregate **every matching row** before it can emit the first `LIMIT`-bounded result. `LIMIT` cannot be pushed past a `GROUP BY` with an aggregate function.

**Contrast with the non-grouped path** (`get_tx_with_cell`, L145–269): there `LIMIT` is applied directly to the flat query, so the DB stops after `limit` rows: [3](#0-2) 

**The `request_limit` guard is ineffective**: it only rejects calls where the caller's `limit` parameter exceeds `self.request_limit`. It does not bound the number of rows the DB processes internally: [4](#0-3) 

**No default timeout**: `timeout_limit` defaults to `None`, so there is no server-side query timeout: [5](#0-4) 

## Impact Explanation

On a mainnet-scale node where a widely-used lock script (e.g., secp256k1) is indexed, a single RPC call with `group_by_transaction=true` and `limit=1` forces the SQLite/Postgres engine to aggregate millions of `(tx_id, io_type, io_index)` rows into `GROUP_CONCAT`/`ARRAY_AGG` in memory before returning any result. Peak RSS grows proportionally to total matching rows, not to `limit`. This can exhaust node memory and cause OOM or extreme latency, constituting a remote DoS against any node with the rich indexer enabled.

**Impact class**: High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.

## Likelihood Explanation

- The rich indexer is opt-in but documented and used in production deployments.
- The attack requires no authentication, no key, no PoW — just a valid JSON-RPC call.
- Empty `args` with `prefix` mode is the default search mode and is explicitly supported.
- No rate limiting, no per-query memory cap, and no default timeout exist in this code path.
- The attack is repeatable and can be sustained by a single attacker with minimal resources.

## Recommendation

Restructure the query to first select the top-`limit` distinct `tx_id` values (with `LIMIT` applied before aggregation), then join back to collect `io_pairs` only for those `tx_id`s. For example:

```sql
SELECT tx_id, block_number, tx_index, tx_hash,
  GROUP_CONCAT(io_type || ',' || io_index, '","') AS io_pairs
FROM (UNION ALL subquery) AS res
JOIN ckb_transaction ... JOIN block ...
WHERE tx_id IN (
  SELECT DISTINCT tx_id FROM (UNION ALL subquery) AS res2
  WHERE tx_id > ?
  ORDER BY tx_id ASC
  LIMIT <limit>
)
GROUP BY tx_id, block_number, tx_index, tx_hash
ORDER BY tx_id ASC
```

Additionally, enforce a server-side query timeout by making `timeout_limit` default to a safe non-`None` value.

## Proof of Concept

1. Run a CKB node with rich indexer enabled and sync a large portion of mainnet.
2. Record baseline RSS of the node process.
3. Issue the following RPC call:
   ```json
   {
     "method": "get_transactions",
     "params": [{
       "script": { "code_hash": "<secp256k1_type_code_hash>", "hash_type": "type", "args": "0x" },
       "script_type": "lock",
       "script_search_mode": "prefix",
       "group_by_transaction": true
     }, "asc", "0x1", null]
   }
   ```
4. Monitor peak RSS during query execution.
5. Assert: peak RSS grows proportionally to total indexed rows matching the script, not to `limit=1`. The node becomes unresponsive or OOMs during aggregation.
