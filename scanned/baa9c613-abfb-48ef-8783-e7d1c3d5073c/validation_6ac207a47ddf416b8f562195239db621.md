Audit Report

## Title
RPC Handler Panic via Unconditional `.expect()` on `txs_sizes: None` from Legacy `BlockExt` Records — (`rpc/src/util/fee_rate.rs`)

## Summary

`FeeRateCollector::statistics` unconditionally calls `.expect()` on `block_ext.txs_sizes` at line 93 of `rpc/src/util/fee_rate.rs`. The legacy on-disk `BlockExt` format (5-field molecule schema, pre-v0.106) always deserializes with `txs_sizes: None`. Any node whose database contains such blocks will panic in the RPC handler when `get_fee_rate_statistics` or `get_fee_rate_statics` is called with a target window covering those blocks. The release profile does not set `panic = "abort"`, so the default `unwind` strategy applies: the handler task unwinds and the caller receives an error, but the node process itself is not killed.

## Finding Description

**Panic site** — `rpc/src/util/fee_rate.rs`, line 93: [1](#0-0) 

The fold closure destructures every `BlockExt` returned by `collect` and calls `.expect()` on `txs_sizes` with no `None` guard.

**Source of `None`** — `util/types/src/conversion/storage.rs`, lines 139–150: the `Unpack<core::BlockExt>` impl for the legacy `packed::BlockExtReader` (5-field schema) unconditionally sets `txs_sizes: None`: [2](#0-1) 

**Store dispatch** — `store/src/store.rs`, lines 247–263: when the stored blob has `count_extra_fields() == 0` (old format), the store returns the legacy reader, yielding `txs_sizes: None`: [3](#0-2) 

**`collect` does not guard against `None`** — the `filter_map` in `FeeRateProvider::collect` only skips blocks absent from the database; it passes every found `BlockExt` (including those with `txs_sizes: None`) into the fold closure: [4](#0-3) 

**RPC entry points** — both `get_fee_rate_statics` and `get_fee_rate_statistics` call `FeeRateCollector::statistics` with no authentication requirement: [5](#0-4) 

**Panic strategy** — the `[profile.release]` section in `Cargo.toml` sets only `overflow-checks = true` and does **not** set `panic = "abort"`. The `[profile.prod]` profile inherits from release and also does not set `panic = "abort"`. Rust's default `unwind` strategy applies, meaning the panic unwinds the handler task and returns an error to the caller; the node process is not killed. [6](#0-5) 

## Impact Explanation

With the default `unwind` panic strategy confirmed in the release and prod profiles, a panic in the RPC handler task terminates only that task and returns an error response to the caller. The node process continues running. This is a **local RPC API crash** — matching the allowed impact: **Note (0 – 500 points): Any local RPC API crash**. The submitter's claim of a full node crash (High) is not supported because `panic = "abort"` is not set in any production build profile.

## Likelihood Explanation

Any mainnet/testnet node that synced blocks before v0.106 retains legacy `BlockExt` rows in `COLUMN_BLOCK_EXT`. The RPC is bound to `127.0.0.1:8114` by default and requires no authentication token. A caller (local process, script, or any entity with access to the RPC port) can trigger the panic repeatedly by calling `get_fee_rate_statistics` with a `target` window that covers legacy blocks. The condition is deterministic and repeatable. [7](#0-6) 

## Recommendation

Replace the `.expect()` with a graceful skip, consistent with the existing `cycles` `Option` handling at line 97:

```rust
let Some(txs_sizes) = txs_sizes else {
    return fee_rates; // skip legacy blocks lacking txs_sizes
};
```

This eliminates the panic entirely and matches the pattern already used for the `cycles` field in the same closure.

## Proof of Concept

```rust
// Add to rpc/src/tests/fee_rate.rs
#[test]
fn test_fee_rate_statistics_none_txs_sizes_panics() {
    let mut provider = DummyFeeRateProvider::new(3);
    provider.append(1, BlockExt {
        received_at: 0,
        total_difficulty: 0u64.into(),
        total_uncles_count: 0,
        verified: None,
        txs_fees: vec![Capacity::shannons(1000)],
        cycles: Some(vec![100]),
        txs_sizes: None,   // legacy block — triggers .expect() panic at line 93
    });
    // Panics on current code at rpc/src/util/fee_rate.rs:93
    let _ = FeeRateCollector::new(&provider).statistics(Some(3));
}
```

Running this test against the current code reproduces the panic. Alternatively, on a node with pre-v0.106 blocks on disk, call:

```
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_fee_rate_statistics","params":[null],"id":1}'
```
