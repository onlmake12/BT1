Looking at the key evidence in the codebase to validate this claim.

Audit Report

## Title
Missing Uniqueness Validation on `tx_hashes` Causes Unguarded Panic and Node Crash in Light Client Protocol Server - (File: `util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute()` accepts `GetTransactionsProof` messages without validating that `tx_hashes` are unique. Duplicate hashes cause the same transaction index to be pushed twice into the indices slice passed to `CBMT::build_merkle_proof`, which returns `None` for invalid (duplicate) indices. The result is unwrapped with `.expect()`, causing a panic. When a protocol handler panics, the network service layer calls `broadcast_exit_signals()`, crashing the entire CKB node process.

## Finding Description

In `execute()`, only two input checks are performed — non-empty and count limit — with no uniqueness check on `tx_hashes`:

```rust
if self.message.tx_hashes().is_empty() { ... }
if self.message.tx_hashes().len() > constant::GET_TRANSACTIONS_PROOF_LIMIT { ... }
``` [1](#0-0) 

When a duplicate hash `[H, H]` is sent, both copies pass the `partition` step (both resolve to the same on-chain transaction), and both are pushed into `txs_in_blocks` with the same `tx_info.index`:

```rust
txs_in_blocks
    .entry(tx_info.block_hash)
    .or_insert_with(Vec::new)
    .push((tx, tx_info.index));  // same index pushed twice
``` [2](#0-1) 

The resulting `txs_and_tx_indices` vector (containing `[(tx, idx), (tx, idx)]`) is then passed directly to `CBMT::build_merkle_proof` with duplicate indices, and the `Option` result is unwrapped with `.expect()`:

```rust
let merkle_proof = CBMT::build_merkle_proof(
    &block.transactions().iter().map(|tx| tx.hash()).collect::<Vec<_>>(),
    &txs_and_tx_indices.iter().map(|(_, index)| *index as u32).collect::<Vec<_>>(),
)
.expect("build proof with verified inputs should be OK");
``` [3](#0-2) 

`CBMT` is the `merkle_cbt` library's Complete Binary Merkle Tree, which returns `None` for semantically invalid index inputs (including duplicates). The `.expect()` converts this `None` into a panic. [4](#0-3) 

When the protocol handler panics, the network service layer catches it as `ProtocolHandleErrorKind::AbnormallyClosed` and unconditionally calls `broadcast_exit_signals()`, terminating the entire node process: [5](#0-4) 

By contrast, the RPC-side `get_tx_indices` helper correctly uses a `HashSet` to detect and reject duplicate transaction indices before building any proof: [6](#0-5) 

The light client protocol path has no equivalent guard.

## Impact Explanation

A remote peer can crash the entire CKB node process by sending a single `GetTransactionsProof` message with any known committed transaction hash repeated twice. The panic propagates to `broadcast_exit_signals()` in the network service layer, which terminates the whole node — not just the async task or the peer session. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The `GetTransactionsProof` message is part of the light client P2P protocol, reachable by any peer that connects to a node with `LightClient` support enabled. The required input — a duplicate committed transaction hash — is trivially constructable from any publicly visible on-chain transaction. No authentication, cryptographic capability, or special privilege is required. The attack is repeatable and requires sending a single crafted message.

## Recommendation

Add a uniqueness check on `tx_hashes` at the start of `execute()`, mirroring the pattern used in `get_tx_indices` on the RPC side:

```rust
let tx_hashes_set: HashSet<_> = self.message.tx_hashes().to_entity().into_iter().collect();
if tx_hashes_set.len() != self.message.tx_hashes().len() {
    return StatusCode::MalformedProtocolMessage.with_context("duplicate tx_hashes");
}
```

Alternatively, deduplicate `txs_and_tx_indices` by index before calling `build_merkle_proof`, or replace the `.expect()` with a graceful error return using `StatusCode::InternalError`.

## Proof of Concept

1. Connect a peer to a CKB node with `LightClient` protocol support enabled.
2. Obtain any committed transaction hash `H` (e.g., from the `get_transaction` RPC or by observing the chain).
3. Construct a `GetTransactionsProof` message with `tx_hashes = [H, H]` and a valid `last_hash` pointing to a main-chain block.
4. Send the message to the node.
5. The server finds both copies of `H` in the chain (both pass the `partition` check), pushes `(tx, idx)` twice into `txs_and_tx_indices` for the same block, calls `CBMT::build_merkle_proof(..., &[idx, idx])`, receives `None`, and panics at `.expect("build proof with verified inputs should be OK")`.
6. The network service layer catches the panic as `AbnormallyClosed` and calls `broadcast_exit_signals()`, crashing the entire CKB node process.
