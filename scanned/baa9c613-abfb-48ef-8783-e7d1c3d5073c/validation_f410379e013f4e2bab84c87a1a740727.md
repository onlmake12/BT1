Audit Report

## Title
`remove_entry_and_descendants` Silently Skips Ancestor `evict_key` Updates Due to Premature Link Removal — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::remove_entry_and_descendants`, all parent/child link records are erased for every entry in the removal set before `remove_entry` is called on each. Because `update_ancestors_index_key` relies on `self.links.calc_ancestors` to locate ancestors, and those links are already gone, the function becomes a no-op. Ancestors remaining in the pool retain stale `descendants_fee`/`descendants_count` values and inflated `evict_key` entries, permanently corrupting the eviction-priority index until those ancestors are themselves removed.

## Finding Description
`PoolEntry` carries an `evict_key: EvictKey` field used by `next_evict_entry` to select the lowest-priority transaction for eviction when the pool is full. [5](#0-4) 

`EvictKey` is derived from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` stored in `TxEntry`: [6](#0-5) 

When a child is added, `record_entry_descendants` calls `update_ancestors_index_key(entry, EntryOp::Add)` to increment each ancestor's `descendants_*` fields and refresh their `evict_key`: [7](#0-6) 

The symmetric removal path is `remove_entry_and_descendants`. It first calls `remove_entry_links` for **every** entry in the removal set (including the root), then calls `remove_entry` on each: [1](#0-0) 

`remove_entry_links` erases the entry's own link record and removes it from its parents' children sets: [2](#0-1) 

`remove_entry` then calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)`: [8](#0-7) 

But `update_ancestors_index_key` begins with `self.links.calc_ancestors(&child.proposal_short_id())`. Since `remove_entry_links` already erased the entry's link record, `calc_ancestors` returns an empty set. The inner loop never executes, and **no ancestor's `evict_key` is updated**: [3](#0-2) 

The code comment at L256 acknowledges the intentional early link removal ("so that we won't update_descendants_index_key in remove_entry"), but the same early removal also silences `update_ancestors_index_key`, which is the defect.

Reachable trigger paths:
- `resolve_conflict` — called on any double-spend submission
- `resolve_conflict_header_dep` — called when a block invalidates a header dep [9](#0-8) 

## Impact Explanation
`next_evict_entry` iterates `iter_by_evict_key()` ascending (lowest fee rate first) to find the transaction to evict when the pool is full: [10](#0-9) 

An ancestor whose `evict_key` still reflects removed high-fee descendants will have an inflated `fee_rate` field in its `EvictKey`, placing it later in the eviction order than it deserves. It will not be evicted even when the pool is full. This allows an attacker to fill the pool with stale-keyed low-fee ancestors, causing legitimate high-fee transactions to be rejected and degrading block-template quality across nodes. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Any unprivileged submitter can trigger this with two transactions per captured slot:
1. Submit T1 (low fee, parent).
2. Submit T2 (high fee, child of T1). T1's `evict_key` is updated to reflect T2's fee.
3. Submit T3 conflicting with T2. `resolve_conflict` → `remove_entry_and_descendants(T2)` → bug fires.
4. T2 is removed; T1's `evict_key` still counts T2's fee. T1 is never evicted.
5. Repeat to fill the pool.

No privileged access, no key material, and no majority hash power is required. The cost is two small transactions per slot captured, and the effect persists until T1 is mined or the node restarts.

## Recommendation
Collect the ancestors of the root entry **before** any links are torn down. After all entries are removed, iterate those surviving ancestors and recompute their `descendants_*` fields and `evict_key`. Concretely, in `remove_entry_and_descendants`:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Collect ancestors of the root BEFORE links are torn down
    let ancestors_to_update = self.links.calc_ancestors(id);

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Recompute evict_key for each surviving ancestor
    for anc_id in &ancestors_to_update {
        self.entries.modify_by_id(anc_id, |e| {
            e.evict_key = e.inner.as_evict_key();
        });
    }

    removed
}
```

Alternatively, restructure removal so that links for the root entry are torn down **last** (after `update_ancestors_index_key` has already run for it), preserving the existing incremental subtraction logic.

## Proof of Concept
```
Initial state:
  T1 (low fee) in pool, T2 (high fee, child of T1) in pool
  T1.inner.descendants_fee  = T1.fee + T2.fee   (high)
  T1.inner.descendants_count = 2
  T1.evict_key.fee_rate = high  ← reflects T2

Attacker submits T3 (spends same input as T2):
  resolve_conflict()
    edges.remove_input(T2_input) → T2_id
    remove_entry_and_descendants(T2_id):
      removed_ids = [T2_id]
      remove_entry_links(T2_id):
        links.remove_child(T1_id, T2_id)   ← T1 no longer has T2 as child
        links.remove(T2_id)                ← T2 link record gone
      remove_entry(T2_id):
        update_ancestors_index_key(T2, Remove):
          calc_ancestors(T2_id) → {} (link record gone)
          loop body never executes
          T1.evict_key NOT updated

Final state:
  T2 removed from pool
  T1 still in pool
  T1.inner.descendants_fee   = T1.fee + T2.fee  ← stale
  T1.inner.descendants_count = 2                ← stale
  T1.evict_key.fee_rate      = high             ← stale

Effect:
  next_evict_entry() ranks T1 as hard-to-evict.
  Pool fills; legitimate high-fee transactions are rejected.
  T1 (low fee) persists indefinitely.
```

A unit test can confirm this by constructing a two-transaction chain, triggering `resolve_conflict` with a conflicting transaction, and asserting that the surviving parent's `evict_key` equals `parent.inner.as_evict_key()` (i.e., reflects only its own fee, not the removed child's).
