### Title
Witness-Supplied `header_dep_index` Interpreted as Full u64 in Rust but Lowest Byte Only in `dao.c`, Enabling Consensus Split - (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` reads the DAO deposit header index from the transaction witness as a full **u64**, while the on-chain `dao.c` script running in CKB-VM reads only the **lowest byte** of the same field. A transaction sender can exploit this semantic discrepancy to craft a DAO withdrawal transaction that the C script accepts but the Rust node rejects during block verification, causing a consensus split.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads the `header_dep_index` from `WitnessArgs.input_type` as a full 8-byte little-endian u64: [1](#0-0) 

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
```

The on-chain `dao.c` script, however, reads only the **lowest byte** of this same 8-byte field when indexing into `header_deps`. This is explicitly documented in the test added to expose the discrepancy: [2](#0-1) 

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

The `FeeCalculator` wrapping `DaoCalculator::transaction_fee` is called inside `ContextualTransactionVerifier::verify`, which runs during both tx-pool admission and block verification: [3](#0-2) [4](#0-3) 

The `DaoCalculator` is also invoked directly in the tx-pool fee check: [5](#0-4) 

---

### Impact Explanation

An attacker crafts a DAO withdrawal transaction with:
- `header_dep_index = 257` (bytes: `0x01 0x01 0x00 0x00 0x00 0x00 0x00 0x00`) in `WitnessArgs.input_type`
- `header_deps[1]` = deposit block hash (block number 100)
- `header_deps[257]` = withdraw block hash (block number 200)
- Cell data = `100` (deposit block number)

**C script (dao.c)**: reads lowest byte of 257 → index 1 → deposit block (number 100) → block number check passes → **ACCEPTS**

**Rust `DaoCalculator`**: reads full u64 = 257 → index 257 → withdraw block (number 200) → block number check: `200 ≠ 100` → `DaoError::InvalidOutPoint` → **REJECTS** [6](#0-5) 

If a miner includes this transaction in a block, the C script accepts it (block is consensus-valid), but every Rust node rejects the block via `ContextualTransactionVerifier`. This produces a **chain split**: Rust nodes fork off the canonical chain.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` confirms the Rust rejection: [7](#0-6) 

---

### Likelihood Explanation

Any unprivileged transaction sender can submit a DAO withdrawal transaction. Constructing a transaction with 258 `header_deps` entries and a witness value of `257u64` is trivial. The attacker does not need any privileged access, leaked keys, or majority hashpower. The only prerequisite is having a live DAO cell to withdraw.

---

### Recommendation

Validate that the caller-supplied `header_dep_index` from the witness does not exceed `u8::MAX` (255), or equivalently, that the upper 7 bytes are all zero. This ensures the Rust `DaoCalculator` and the `dao.c` C script always resolve the same `header_deps` entry:

```rust
if header_dep_index > u8::MAX as u64 {
    return Err(DaoError::InvalidDaoFormat);
}
```

This check should be inserted immediately after the index is decoded at `util/dao/src/lib.rs` line 91, before the `header_deps().get(header_dep_index as usize)` call at line 96. [8](#0-7) 

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` directly demonstrates the discrepancy: [9](#0-8) 

- `header_dep_index = 257` is placed in the witness
- `header_deps[1]` = deposit block (number 100) — what `dao.c` resolves via `257 & 0xFF = 1`
- `header_deps[257]` = withdraw block (number 200) — what Rust resolves via full u64
- Cell data encodes deposit block number = 100
- Rust `DaoCalculator` resolves index 257 → withdraw block (number 200) → `200 ≠ 100` → `Err`
- The `dao.c` script resolves index 1 → deposit block (number 100) → passes → would accept the transaction

A miner submitting a block containing this transaction would produce a block accepted by the C script but rejected by all Rust nodes, splitting the chain.

fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
