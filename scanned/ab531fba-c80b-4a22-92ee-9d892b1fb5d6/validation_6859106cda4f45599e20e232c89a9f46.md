### Title
Malicious script author can replace `hash_type=type` dep cell code with always-fail binary, permanently locking all cells that reference it - (`script/src/types.rs`)

---

### Summary

CKB's `ScriptHashType::Type` ("upgradable script") mechanism resolves script code at transaction execution time by looking up the **current** data of a dep cell whose type script hash matches the lock/type script's `code_hash`. A script author who controls the dep cell can spend it and recreate it with the same type script identity but with always-failing bytecode. Because the old dep cell is now dead (spent), every user who locked funds with that type-hash-referenced script is forced to reference the new, always-failing dep cell. All such cells become permanently unspendable.

---

### Finding Description

In `script/src/types.rs`, `TxData::new` builds `binaries_by_type_hash` by iterating over `resolved_cell_deps` and mapping each dep cell's type script hash to the **current** binary data of that cell:

```rust
// script/src/types.rs lines 708-713
if let Some(t) = &cell_meta.cell_output.type_().to_opt() {
    binaries_by_type_hash
        .entry(t.calc_script_hash())
        .and_modify(|bin| bin.merge(&data_hash))
        .or_insert_with(|| Binaries::new(data_hash.to_owned(), i, lazy.to_owned()));
}
```

When a lock script uses `hash_type = Type`, `extract_script_and_dep_index` resolves the executable binary from this map:

```rust
// script/src/types.rs lines 843-852
ScriptHashType::Type => {
    if let Some(ref bin) = self.binaries_by_type_hash.get(&script.code_hash()) {
        match bin {
            Binaries::Unique(_, dep_index, lazy) => Ok((lazy, dep_index)),
            Binaries::Duplicate(_, dep_index, lazy) => Ok((lazy, dep_index)),
            Binaries::Multiple => Err(ScriptError::MultipleMatches),
        }
    } else {
        Err(ScriptError::ScriptNotFound(script.code_hash()))
    }
}
```

The type script hash is computed from the dep cell's **type script** (`t.calc_script_hash()`), not from the cell's data. The cell data (the actual RISC-V bytecode) is entirely independent and can be changed by the cell owner. The Type ID verification in `script/src/type_id.rs` only enforces uniqueness and creation-hash correctness — it places **no constraint on the cell data content** during an upgrade (one-input, one-output transfer):

```rust
// script/src/type_id.rs lines 42-75
if self.script_group.input_indices.len() > 1 || self.script_group.output_indices.len() > 1 {
    return Err(self.validation_failure(ERROR_TOO_MANY_CELLS));
}
// If there is one input and one output, verification passes unconditionally.
Ok(TYPE_ID_CYCLES)
```

Once the script author spends the original dep cell, it becomes `CellStatus::Dead`. The `resolve_transaction` function in `util/types/src/core/cell.rs` (lines 706-714) rejects dead cells with `OutPointError::Dead`. Users cannot reference the old (valid) dep cell; they are forced to reference the new (always-fail) one.

**Attack path:**

1. Script author deploys cell `C1` with type script `T` (a Type ID script) and valid lock bytecode `B1`.
2. Users create cells with lock scripts `Script { code_hash: hash(T), hash_type: Type }`, trusting `B1`.
3. Script author submits a transaction spending `C1` and creating `C2` with the same type script `T` but always-fail bytecode `B2` (e.g., a RISC-V binary that immediately exits with code 1).
4. `C1` is now dead. Any spend attempt by a victim must include `C2` as the dep cell.
5. `TxData::new` populates `binaries_by_type_hash[hash(T)] = B2`.
6. `extract_script_and_dep_index` returns `B2` for the lock script resolution.
7. The CKB-VM executes `B2`, which returns a non-zero exit code → `ScriptError::ValidationFailure`.
8. The transaction is rejected. The victim's funds are permanently locked.

---

### Impact Explanation

All CKB capacity (CKBytes) locked in cells whose lock script uses `hash_type = Type` pointing to the attacker-controlled dep cell becomes permanently unspendable. There is no recovery path: the script author can immediately re-upgrade back to always-fail code if the protocol attempts any remediation, and the Type ID mechanism provides no governance veto to third parties. The impact is **permanent, irrecoverable loss of user funds**.

---

### Likelihood Explanation

The attacker is a **script author** — an unprivileged role explicitly listed in the RESEARCHER.md attacker profiles. No keys, governance access, or majority hashpower are required. The only precondition is that the script author controls the lock on the dep cell (which they do, by construction, since they deployed it). The attack is a single on-chain transaction. Any user who adopted a third-party `hash_type=type` lock script without auditing the deployer's ongoing control of the dep cell is vulnerable. This pattern is actively encouraged by CKB tooling and documentation as the standard "upgradable script" deployment model, making the population of affected users potentially large.

---

### Recommendation

1. **Protocol-level:** Introduce an optional "freeze" flag or a governance-controlled upgrade path for Type ID cells used as lock scripts, so that the cell owner cannot unilaterally replace the bytecode once other cells depend on it.
2. **Node-level:** Emit a warning (or consensus rule) when a transaction upgrades a Type ID cell whose type hash is referenced as a lock script by live cells in the UTXO set.
3. **User-level guidance:** Document clearly that `hash_type=type` lock scripts carry an unconditional trust assumption in the dep cell owner; recommend `hash_type=data/data1/data2` for lock scripts where immutability is required.
