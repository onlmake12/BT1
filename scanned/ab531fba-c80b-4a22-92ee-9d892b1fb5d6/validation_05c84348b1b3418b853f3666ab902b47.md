### Title
Unsolicited Ping messages allow an inbound peer to trivially bypass eviction protection — (`network/src/protocols/ping.rs`, `network/src/peer_registry.rs`)

### Summary

An unprivileged remote inbound peer can send unsolicited `Ping` messages at any rate to continuously refresh `last_ping_protocol_message_received_at`. The eviction logic in `try_evict_inbound_peer()` uses this field to protect the 8 most-recently-active peers from eviction, but the field is trivially attacker-controlled, directly violating the stated invariant that protection criteria must be "hard to simulate or manipulate."

### Finding Description

`PingHandler::received()` accepts `PingPayload::Ping` from any connected peer without any rate limit, nonce challenge, or prior-Ping requirement. Each such message calls `ping_received()`, which unconditionally sets `peer.last_ping_protocol_message_received_at = Some(Instant::now())`.

`try_evict_inbound_peer()` runs three protection rounds before evicting a peer. Round 2 protects the 8 peers with the smallest elapsed time since `last_ping_protocol_message_received_at`. Because an attacker can refresh this timestamp at will (e.g., one Ping per 100 ms), they can guarantee they always appear as one of the 8 most-recently-active peers and are removed from the eviction candidate pool before any eviction decision is made.

Contrast this with round 1 (lowest `ping_rtt`), which is based on actual measured network latency and cannot be freely manipulated, and round 3 (longest `connected_time`), which is set at connection time and immutable. Round 2 provides no such resistance.

### Impact Explanation

- At max_inbound capacity, a new legitimate peer triggers `try_evict_inbound_peer()`.
- The attacker's peer is removed from the candidate pool in round 2 due to its artificially fresh timestamp.
- A legitimate long-lived peer is evicted instead.
- Repeated over time, an attacker with even a single connection can guarantee permanent residency, displacing honest peers.
- With multiple connections from distinct `/16` network groups (to survive the final group-based random eviction step), an attacker can occupy a disproportionate share of inbound slots at negligible cost (one small Ping message per 100 ms per connection).

### Likelihood Explanation

The attack requires only a standard TCP connection and the ability to send small Ping protocol messages. No PoW, no key material, no privileged access. The Ping message is tiny (a 4-byte nonce in a flatbuffer envelope). The attack is fully local-testable and requires no coordination.

### Recommendation

1. **Do not update `last_ping_protocol_message_received_at` on receipt of an unsolicited `Ping`.** Only update it on receipt of a valid `Pong` that matches a nonce the local node sent (i.e., inside `pong_received()`, which already validates `status.processing && nonce == status.nonce()`). This makes the field reflect genuine bidirectional liveness, not attacker-controlled activity.
2. Alternatively, add per-session rate limiting on inbound Ping messages so that flooding cannot refresh the timestamp faster than the eviction interval.
3. Consider replacing the `last_ping_protocol_message_received_at` eviction criterion with a metric that is harder to manipulate, such as the number of valid block/header announcements received from the peer.

### Proof of Concept

```
1. Connect max_inbound honest peers to a test node.
2. Connect one attacker peer.
3. Attacker sends a Ping message every 100 ms in a loop.
4. Attempt to connect a new legitimate peer repeatedly.
5. Observe: try_evict_inbound_peer() always protects the attacker in round 2
   (smallest elapsed since last_ping_protocol_message_received_at ≈ 0 s),
   evicting an honest peer instead.
6. Assert: attacker session_id is never selected for eviction;
   legitimate peer always succeeds by displacing an honest peer.
```

    F: FnMut(&T, &T) -> std::cmp::Ordering,
