Audit Report

## Title
Unauthenticated Amplification DoS via Genesis-Hash GetHeaders — (`sync/src/synchronizer/get_headers_process.rs`)

## Summary
Any peer can send a `GetHeaders` message with a single-entry locator containing the genesis hash. When the node is out of IBD, it unconditionally performs up to 2000 DB reads, serializes ~384 KB of headers, and dispatches them via a spawned async task — with no per-peer rate limit, no ban triggered, and negligible cost to the attacker. This creates a high-amplification request/response asymmetry exploitable as a sustained bandwidth and CPU DoS.

## Finding Description

**Entrypoint:** `GetHeadersProcess::execute()` in `sync/src/synchronizer/get_headers_process.rs`.

**Guard 1 — locator size** (L47–51): Rejects only if `locator_size > MAX_LOCATOR_SIZE (101)`. A single-entry locator (size = 1) passes unconditionally. [1](#0-0) 

**Guard 2 — IBD check** (L53–66): Skipped entirely when the node is out of IBD, which is the normal operating state of a synced node. [2](#0-1) 

**`locate_latest_common_block([genesis_hash])`** (L1866–1880 in `sync/src/types/mod.rs`): The function requires `locator.last() == genesis_hash`. With a single-entry locator `[genesis_hash]`, this check passes. The iterator then finds `(index=0, latest_common=Some(0))` — the genesis block is always in the DB. The early-return condition `index == 0 || latest_common == Some(0)` fires, returning `Some(0)` — a valid common ancestor, not `None`. [3](#0-2) 

**`get_locator_response(0, &zero_hash)`** (L1905–1921): Starting from block 1, iterates up to `tip_number`, capped at `MAX_HEADERS_LEN = 2000`. On a long chain this returns exactly 2000 headers (~384 KB serialized). [4](#0-3) 

**No rate limiting on incoming `GetHeaders`:** The `pending_get_headers` cache (L1929–1951) lives inside `send_getheaders_to_peer` — it throttles *outgoing* requests the node sends to peers, not *incoming* requests from peers. There is no equivalent guard in `execute()`. [5](#0-4) 

**No ban triggered:** `GetHeadersMissCommonAncestors` (which carries `SYNC_USELESS_BAN_TIME`) is only returned when `locate_latest_common_block` returns `None` (L94–96). Since genesis is always a valid common ancestor, this path is never reached for the genesis-hash attack. [6](#0-5) 

**`RequestGenesis` (status 417) is never used in this handler:** It exists in `sync/src/status.rs` (L107–108) and is used in `get_blocks_process.rs`, but is entirely absent from `get_headers_process.rs`. [7](#0-6) 

**Response dispatched with no backpressure** (L84–93): The serialized `SendHeaders` message is sent via a spawned async task with no queue depth check or flow control. [8](#0-7) 

## Impact Explanation

Each ~70-byte `GetHeaders` message causes the victim node to perform up to 2000 DB reads and transmit ~384 KB. An attacker with a single connection sending these in a tight loop achieves a ~5500× amplification ratio on bandwidth alone, plus proportional DB I/O and async task queue pressure. With multiple Sybil connections the effect multiplies linearly. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

The attack requires only a standard P2P connection and the genesis hash, which is hardcoded in the chain spec and trivially obtained from any block explorer or node. No proof-of-work, no keys, no privileged role. The attack is locally testable, repeatable indefinitely, and self-sustaining — no victim action required beyond the node being out of IBD.

## Recommendation

1. **Add per-peer rate limiting on incoming `GetHeaders`**: implement a token bucket or sliding-window counter per `PeerIndex` inside `execute()`, analogous to the existing `pending_get_headers` mechanism for outgoing requests.
2. **Detect and ban genesis-only locators**: if `locate_latest_common_block` resolves to block 0 and the peer is not in a legitimate sync state, return `RequestGenesis` (status 417, already defined) and apply a ban duration.
3. **Cap response work for low-progress peers**: refuse to serve the maximum 2000 headers to peers that have not demonstrated legitimate sync progress beyond block 0.

## Proof of Concept

```
1. Connect to a CKB node that is out of IBD with a chain of ≥2000 blocks.
2. Obtain the genesis hash from the chain spec or any public source.
3. In a tight loop, send:
       SyncMessage::GetHeaders {
           block_locator_hashes: [genesis_hash],
           hash_stop: 0x000...000
       }
4. Observe: each message triggers a ~384 KB SendHeaders response.
5. Assert: no ban is applied, no rate limit activates, no cost to attacker.
6. Measure: victim node outbound bandwidth and DB I/O scale linearly
   with message rate; no self-protection mechanism activates.
```

    pub fn get_locator_response(
        &self,
        block_number: BlockNumber,
        hash_stop: &Byte32,
    ) -> Vec<core::HeaderView> {
        let tip_number = self.tip_header().number();
        let Some(start_number) = block_number.checked_add(1) else {
            return Vec::new();
        };
        std::iter::successors(Some(start_number), |number| number.checked_add(1))
            .take_while(|number| *number <= tip_number)
            .take(MAX_HEADERS_LEN)
            .filter_map(|block_number| self.snapshot.get_block_hash(block_number))
            .take_while(|block_hash| block_hash != hash_stop)
            .filter_map(|block_hash| self.sync_shared.store().get_block_header(&block_hash))
            .collect()
    }
