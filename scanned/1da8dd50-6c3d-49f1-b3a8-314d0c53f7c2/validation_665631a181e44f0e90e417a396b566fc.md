Audit Report

## Title
Per-Peer `unknown_tx_hashes` Limit Bypass via Cyclic Batch Submission Below Global Threshold — (`sync/src/types/mod.rs`)

## Summary
`add_ask_for_txs` unconditionally inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32 767) hashes before checking any per-peer limit. The per-peer ban check only fires when the global queue length reaches `MAX_UNKNOWN_TX_HASHES_SIZE` (50 000) or `peers_count × 32 767`. Because `pop_ask_for_txs` permanently drops single-peer entries after the 30-second `RETRY_ASK_TX_TIMEOUT_INCREASE` window, a malicious peer can submit batches of 32 766 hashes indefinitely — always staying below the global threshold — causing the node to issue `GetRelayTransactions` for non-existent transactions without ever banning the attacker.

## Finding Description
**Phase 1 — unconditional insertion** (`sync/src/types/mod.rs`, lines 1486–1504): `add_ask_for_txs` iterates through the incoming hashes and inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` entries into `unknown_tx_hashes` with no per-peer guard.

**Phase 2 — conditional per-peer check** (lines 1506–1529): only after insertion does the code check whether `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50 000) or `>= peers_count × 32 767`. With a single connected peer and a batch of 32 766 hashes, the queue length is 32 766, which satisfies neither condition (32 766 < 50 000 and 32 766 < 1 × 32 767), so the per-peer counter scan and ban are never reached.

**Entry expiry** (`pop_ask_for_txs`, lines 1466–1474): when `next_request_peer()` is called on an entry whose `requested` flag is already `true` and `peers.len() == 1`, it returns `None` (lines 1277–1284). The entry is then **not** re-pushed to the queue and is permanently dropped. After the 30-second timeout all 32 766 entries are silently discarded, resetting the attacker's effective per-peer count to zero.

The existing integration test (`test/src/specs/relay/too_many_unknown_transactions.rs`) sends exactly 32 767 hashes, which triggers the global check (32 767 ≥ 1 × 32 767) and correctly bans the peer. Sending 32 766 hashes bypasses this test's coverage entirely.

## Impact Explanation
A single unprivileged peer causes the victim node to continuously emit `GetRelayTransactions` messages for non-existent transactions (~32 766 requests every ~31 seconds), wasting outbound bandwidth and CPU. The `unknown_tx_hashes` queue is kept perpetually occupied, crowding out legitimate transaction-hash announcements from honest peers and degrading transaction propagation. The attacker is never banned. This matches the allowed impact: **Low (501–2000 points) — important performance degradation for a CKB node**, with potential escalation toward High if the attack is applied simultaneously across many nodes to cause network-wide transaction-relay congestion.

## Likelihood Explanation
Any peer that can establish a relay connection can execute this attack. No special keys, hashpower, or Sybil capability is required. The rate limiter in `try_process` (30 messages/second per peer/message-type) is not a barrier: the attacker needs only one `RelayTransactionHashes` message per 31-second cycle. The attack is fully automated, repeatable indefinitely, and leaves no ban record on the attacker.

## Recommendation
Replace the post-insertion current-state scan with a **monotonic per-peer counter** stored in `PeerState`. Increment it by the number of newly inserted (Vacant) entries in Phase 1 of `add_ask_for_txs` and never decrement it (or decay it only over a long window). Reject or ban the peer as soon as its cumulative counter reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, regardless of whether earlier entries have already been consumed from the queue. This mirrors the fix pattern for analogous `balanceOf`-reset bugs: the accounting variable must be monotonic and independent of the mutable queue state.

## Proof of Concept
```
// Attacker pseudo-code (single peer, no special privileges):
loop {
    // 32766 < MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER (32767)
    // → Phase 1 inserts all 32766 unconditionally
    // → Phase 2 check: 32766 < 50000 AND 32766 < 1×32767 → skipped
    // → Status::ok() returned, no ban
    let hashes = generate_unique_hashes(32766);
    send_relay_tx_hashes(hashes);

    // Node sends GetRelayTransactions for all 32766 hashes.
    // Attacker ignores them.
    // After RETRY_ASK_TX_TIMEOUT_INCREASE (30s), pop_ask_for_txs drops
    // all entries: next_request_peer() returns None (single-peer, already requested).
    sleep(31_seconds);

    // unknown_tx_hashes is now empty for this peer → per-peer count = 0
    // → repeat indefinitely, attacker never banned
}
```
To reproduce: run the existing `TooManyUnknownTransactions` integration test modified to send 32 766 hashes instead of 32 767 and assert that the peer is **not** banned after 60 seconds — the assertion will pass, confirming the bypass.
