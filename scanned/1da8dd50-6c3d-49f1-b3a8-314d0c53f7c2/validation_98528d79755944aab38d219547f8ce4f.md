Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Malicious `GetBlockProposal` P2P Messages — (File: `sync/src/types/mod.rs`)

## Summary
The `SyncState::pending_get_block_proposals` DashMap has no size cap and is only cleared by a periodic drain timer. Any unprivileged peer can send `GetBlockProposal` messages containing up to 3,000 fake `ProposalShortId` values per message; all IDs absent from the local tx-pool are inserted unconditionally. With multiple Sybil peers, the map grows without bound between drain cycles, causing heap exhaustion and potential node crash.

## Finding Description
`SyncState` declares the map with no capacity bound: [1](#0-0) 

It is initialized with `DashMap::new()`: [2](#0-1) 

`GetBlockProposalProcess::execute()` enforces only a per-message count limit of `max_block_proposals_limit × max_uncles_num` (1,500 × 2 = 3,000): [3](#0-2) 

Every proposal ID not found in the tx-pool is passed to `insert_get_block_proposals` with no total-map size guard: [4](#0-3) 

`insert_get_block_proposals` inserts every ID unconditionally: [5](#0-4) 

The only relief valve is `drain_get_block_proposals`, which performs an O(n) full clone before clearing — amplifying latency when the map is large: [6](#0-5) 

The rate limiter is keyed by `(PeerIndex, message_item_id)` at 30 req/s per peer per message type: [7](#0-6) 

This is a per-peer cap; with `k` colluding peers the insertion rate is `k × 30 × 3,000 = 90,000k` entries/second. Additionally, `disconnected()` does not remove a peer's entries from `pending_get_block_proposals`: [8](#0-7) 

So even after a peer disconnects, its injected entries remain in the map until the next drain.

## Impact Explanation
This directly matches the **High** bounty impact: "Vulnerabilities which could easily crash a CKB node." With enough Sybil peers, the node's heap grows proportionally to `k × drain_interval × 90,000` entries. Each `ProposalShortId` key is 10 bytes; with `HashSet<PeerIndex>` overhead, sustained attack at 100 peers can accumulate hundreds of megabytes per drain cycle, leading to OOM termination or severe swap pressure that stalls the node. The O(n) clone in `drain_get_block_proposals` additionally causes a relay-timer latency spike when the map is large, delaying legitimate compact block and transaction processing.

## Likelihood Explanation
Any unprivileged peer with a TCP connection can send `GetBlockProposal` messages — no authentication, stake, or PoW is required. Generating 3,000 random 10-byte IDs per message is negligible CPU work. The per-peer rate limit (30 req/s) is easily saturated by a single connection and scales linearly with Sybil peer count. The attack is repeatable indefinitely and requires no valid transactions. The CHANGELOG records prior fixes for analogous memory-bloat issues in the same subsystem (`#3093`, `#3094`, `#3110`), confirming this class of bug is realistic and has been exploited before.

## Recommendation
1. **Cap `pending_get_block_proposals`**: Enforce a maximum total entry count (e.g., `max_block_proposals_limit × max_peers`). Reject insertions once the cap is reached.
2. **Per-peer sub-limit**: Track how many pending IDs each peer has contributed; evict or reject when a peer's quota is exceeded.
3. **Clean up on disconnect**: In `disconnected()`, remove all entries in `pending_get_block_proposals` where the peer's `PeerIndex` appears in the `HashSet<PeerIndex>` value, and remove the key entirely if the set becomes empty.
4. **Replace clone-then-clear with `std::mem::take`**: In `drain_get_block_proposals`, use `std::mem::take` or swap with a fresh `DashMap` to avoid the O(n) allocation spike.
5. **Reduce drain interval or make it event-driven**: Drain more frequently to shrink the accumulation window.

## Proof of Concept
```
1. Establish N TCP connections to the victim node (N Sybil peers).
2. For each peer, loop at ≤30 msg/s (to stay under per-peer rate limit):
   a. Generate 3,000 random 10-byte ProposalShortId values (none in victim's tx-pool).
   b. Build a GetBlockProposal message with any known tip block_hash and the 3,000 fake IDs.
   c. Send over the RelayV3 protocol.
3. All 3,000 IDs pass the per-message count check (≤3,000) and are inserted into
   pending_get_block_proposals on every message.
4. Between relay timer ticks, the DashMap grows by N × 30 × 3,000 entries per second.
5. Monitor victim RSS: it grows linearly with N and time.
6. When drain fires, the O(n) clone causes a measurable relay-timer latency spike.
7. At sufficient N and drain interval, victim node reaches OOM and crashes.
```
