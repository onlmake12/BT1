Audit Report

## Title
Non-Deterministic Orphan Expiry via Single-Entry HashMap Sampling in `need_clean` — (`chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` samples only the first entry from a `HashMap<packed::Byte32, LonelyBlockHash>` to decide whether all blocks under a given `parent_hash` should be evicted. Because Rust's `HashMap` iteration order is non-deterministic, a malicious P2P peer can inject a crafted block sharing a `parent_hash` with a legitimate orphan but carrying an expired `epoch_number`, causing either premature eviction of the legitimate orphan (sync disruption) or indefinite on-disk retention of the crafted block (storage bloat). No PoW is required because epoch continuity is only checked in the contextual `HeaderVerifier`, not in the non-contextual path that gates orphan pool insertion.

## Finding Description

**Root cause — `need_clean` samples one arbitrary entry:**

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← arbitrary first entry
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

The inner structure is `HashMap<ParentHash, HashMap<packed::Byte32, LonelyBlockHash>>`. Multiple blocks can share the same `parent_hash` (competing blocks at the same height). `map.iter().next()` returns an arbitrary entry; if that entry's `epoch_number` is expired the entire sibling group is evicted via `remove_blocks_by_parent`, and if it is not expired the entire group is retained.

**Why crafted blocks pass non-contextual verification:**

`asynchronous_process_block` calls `non_contextual_verify`, which runs only `BlockVerifier` and `NonContextualBlockTxsVerifier`. `BlockVerifier` checks cellbase structure, merkle roots, block bytes, proposals, and duplicates — it does **not** check PoW and does **not** check epoch continuity. PoW and epoch continuity are checked only in `HeaderVerifier`, which is contextual (requires the parent header from the chain store). Therefore a malicious peer can send a block with an arbitrary `epoch_number` (e.g., epoch 1, which is expired when `tip_epoch >= 7`) and the same `parent_hash` as a legitimate orphan block. It will pass `non_contextual_verify`, be written to disk via `insert_block`, and be inserted into the orphan pool.

**Full exploit path:**

1. Attacker observes a legitimate orphan block B with `parent_hash = P` and `epoch_number = N` (current epoch, not expired) in the P2P network.
2. Attacker crafts block A with `parent_hash = P`, `epoch_number = 1` (expired), valid merkle root, valid cellbase, no PoW needed.
3. Block A passes `non_contextual_verify` and is written to disk and inserted into the orphan pool under `blocks[P]`.
4. Now `blocks[P]` contains both A (epoch 1) and B (epoch N).
5. When `clean_expired_orphans` fires (every 60 seconds), `clean_expired_blocks` calls `need_clean(P, tip_epoch)`.
6. `map.iter().next()` returns either A or B non-deterministically:
   - If A: `need_clean` returns `true` → `remove_blocks_by_parent(P)` evicts both A and B. Block B's disk record is deleted, its header view and block status are stripped. Sync stalls until B is re-requested.
   - If B: `need_clean` returns `false` → neither block is evicted. Block A (and its on-disk record) is retained indefinitely.

**Existing guards are insufficient:**

`non_contextual_verify` explicitly does not run `HeaderVerifier`. The `epoch_number` stored in `LonelyBlockHash` is taken directly from `block.epoch().number()` — the block's self-reported value — with no validation against the chain state at insertion time.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The attacker is an unprivileged P2P sync peer requiring no PoW. The attack is repeatable and amplifiable: the peer can flood many nodes with crafted blocks under many different `parent_hash` values simultaneously. In the premature-eviction case, each targeted node must re-request the evicted legitimate orphan block, generating additional P2P traffic across the network. In the indefinite-retention case, crafted blocks accumulate on disk and in the in-memory `block_status_map` with no cleanup path. Both effects compound with scale, causing measurable network congestion and per-node resource exhaustion with negligible attacker cost.

## Likelihood Explanation

- Attacker is an unprivileged P2P sync peer; no key, privilege, or PoW is required.
- Crafting a valid block structure with an arbitrary `epoch_number` and a target `parent_hash` requires only constructing a valid `BlockBuilder` payload with consistent merkle roots and a well-formed cellbase.
- The target `parent_hash` values are observable from P2P gossip (orphan block announcements).
- The 60-second cleanup timer provides a reliable, predictable trigger window.
- The attack is repeatable indefinitely; each 60-second window is a new opportunity.

## Recommendation

Replace the single-entry sample with a check over **all** entries under the parent. The correct invariant is: evict only when **every** sibling block is expired:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            !map.is_empty()
                && map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

Alternatively, track the minimum `epoch_number` per `parent_hash` group at insertion time to avoid iterating all entries on every cleanup tick.

## Proof of Concept

```rust
// In chain/src/tests/orphan_block_pool.rs, add:
#[test]
fn test_need_clean_nondeterministic_with_mixed_epochs() {
    use crate::utils::orphan_block_pool::{EXPIRED_EPOCH, OrphanBlockPool};
    use crate::{LonelyBlock, LonelyBlockHash};
    use ckb_chain_spec::consensus::ConsensusBuilder;
    use ckb_systemtime::unix_time_as_millis;
    use ckb_types::core::{BlockBuilder, EpochNumberWithFraction};
    use std::sync::Arc;

    let consensus = ConsensusBuilder::default().build();
    let pool = OrphanBlockPool::with_capacity(10);
    let parent = consensus.genesis_block().header();
    let tip_epoch = 20_u64; // EXPIRED_EPOCH=6, so epoch<=13 is expired

    // Block A: epoch 1 (expired: 1 + 6 < 20)
    let block_a = BlockBuilder::default()
        .parent_hash(parent.hash())
        .timestamp(unix_time_as_millis())
        .number(parent.number() + 1)
        .epoch(EpochNumberWithFraction::new(1, 0, 1000))
        .nonce(1u128.pack())
        .build();

    // Block B: epoch 15 (NOT expired: 15 + 6 = 21, not < 20)
    let block_b = BlockBuilder::default()
        .parent_hash(parent.hash())
        .timestamp(unix_time_as_millis() + 1)
        .number(parent.number() + 1)
        .epoch(EpochNumberWithFraction::new(15, 0, 1000))
        .nonce(2u128.pack())
        .build();

    pool.insert(LonelyBlock { block: Arc::new(block_a), switch: None, verify_callback: None }.into());
    pool.insert(LonelyBlock { block: Arc::new(block_b.clone()), switch: None, verify_callback: None }.into());

    let evicted = pool.clean_expired_blocks(tip_epoch);

    // BUG: outcome is non-deterministic across HashMap seeds.
    // Either both blocks are evicted (block_b prematurely removed),
    // or neither is evicted (block_a retained indefinitely).
    // Correct behavior: neither should be evicted (block_b is not expired).
    // Run with RUSTFLAGS="-C target-cpu=native" across multiple seeds to observe both outcomes.
    let evicted_hashes: Vec<_> = evicted.iter().map(|b| b.hash()).collect();
    assert!(
        !evicted_hashes.contains(&block_b.hash()),
        "block_b (epoch 15) must not be evicted: non-deterministic eviction confirmed"
    );
}
```

To replicate the attacker path: send block A via P2P `SendBlock` message with `parent_hash` equal to the hash of a known orphan's parent, `epoch = EpochNumberWithFraction::new(1, 0, 1000)`, a valid merkle root, and a valid cellbase. The block passes `non_contextual_verify` and is inserted into the orphan pool alongside the legitimate orphan.
