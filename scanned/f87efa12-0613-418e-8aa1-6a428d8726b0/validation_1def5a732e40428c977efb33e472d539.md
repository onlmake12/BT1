Audit Report

## Title
`readd_detached_tx` Holds `tx_pool` Write Lock During Full CKB-VM Script Execution — (`tx-pool/src/process.rs`)

## Summary

`TxPoolService::update_tx_pool_for_reorg` acquires the `tx_pool` write lock at line 836 and holds it for the entire duration of `readd_detached_tx`, which calls `verify_rtx` with `command_rx = None` for each detached transaction not in the pre-fetched cache. This causes `verify_rtx` to fall into the `block_in_place` branch in `tx-pool/src/util.rs`, running full synchronous CKB-VM execution while the write lock remains held. All concurrent `tx_pool.read().await` and `tx_pool.write().await` callers block for the full re-verification window.

## Finding Description

In `tx-pool/src/process.rs`, the write lock is acquired at line 836 and is not released until line 851, spanning the entire `readd_detached_tx` call:

```rust
let mut tx_pool = self.tx_pool.write().await;   // line 836 — lock acquired
_update_tx_pool_for_reorg(&mut tx_pool, ...);
self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await;
// line 851 — lock released
```

Inside `readd_detached_tx` (lines 878–914), for each detached transaction, `verify_rtx` is called with `command_rx = None` (line 901). In `tx-pool/src/util.rs` lines 116–131, when `command_rx` is `None` and there is no cache hit, `verify_rtx` executes the `block_in_place` branch — full synchronous CKB-VM execution — while the write lock guard remains owned by the calling async task.

The correct pattern is demonstrated in `_process_tx` (lines 705–753): `verify_rtx` is called before acquiring any lock, and only the fast `submit_entry` call (which internally calls `with_tx_pool_write_lock`) holds the lock. The `readd_detached_tx` path inverts this ordering.

The pre-fetch at line 828 (`fetch_txs_verify_cache`) only populates the cache for transactions already known to the node. An attacker who mines a private block with transactions never broadcast to the network ensures the cache is cold for all detached transactions, maximizing lock hold time.

## Impact Explanation

While the write lock is held during re-verification:
- `submit_entry` → `with_tx_pool_write_lock` blocks (line 103).
- `pre_check` for incoming transactions → `with_tx_pool_read_lock` → `tx_pool.read().await` blocks (line 251); tokio's `RwLock` write lock excludes all readers.
- Block assembler `update_full`/`update_transactions` → `tx_pool.read().await` blocks (block_assembler/mod.rs line 191).

The stall duration scales as `N_detached_txs × max_tx_verify_cycles / CPU_speed`. For a block filled with max-cycle transactions (`max_block_cycles` ≈ 3.5 × 10⁹), this produces multi-second stalls per reorg event, during which the node cannot admit new transactions or update block templates. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points).

## Likelihood Explanation

The attacker must be a miner capable of producing at least one competing block at the current tip height. They mine a private block containing transactions consuming `max_tx_verify_cycles` cycles each, never broadcast to the network (bypassing `txs_verify_cache`). Submitting this block triggers a 1-block reorg. Any miner with nonzero hashpower can eventually achieve this given enough attempts. The attack is repeatable: each reorg event re-triggers the full lock-hold window.

## Recommendation

Move all `verify_rtx` calls in `readd_detached_tx` outside the `tx_pool` write lock, mirroring the `_process_tx` pattern: collect all `(rtx, status, fee, tx_size, verified)` tuples first without holding the lock, then acquire the write lock only for the fast `_submit_entry` calls. Alternatively, release and reacquire the lock between each transaction's verification, or pass a `command_rx` to allow interruption.

## Proof of Concept

1. Attacker mines a private block containing N transactions each consuming `max_tx_verify_cycles` cycles, none previously broadcast (victim's `txs_verify_cache` is cold for all).
2. Attacker submits the block to the victim node at the same tip height, triggering a 1-block reorg.
3. Victim node calls `update_tx_pool_for_reorg` → acquires `tx_pool.write()` at line 836 → calls `readd_detached_tx` → calls `verify_rtx(..., None)` for each of the N transactions via `block_in_place` while holding the lock.
4. Concurrently issue a `submit_remote_tx` (which calls `pre_check` → `with_tx_pool_read_lock`) and measure time-to-acquire.
5. Assert that the lock hold time exceeds an acceptable threshold (e.g., > 1 second for a block with many max-cycle transactions).
