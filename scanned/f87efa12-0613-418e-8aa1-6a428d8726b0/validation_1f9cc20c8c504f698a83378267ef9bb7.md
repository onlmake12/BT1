Audit Report

## Title
Global Orphan Pool Saturation via No Per-Peer Limit Enables Targeted Eviction of Legitimate Orphan Transactions — (`tx-pool/src/component/orphan.rs`)

## Summary

The `OrphanPool` enforces a single global cap of 100 entries with no per-peer sub-accounting. When the pool is full, `limit_size()` evicts entries in non-deterministic HashMap iteration order, never consulting the stored `peer` field. A single malicious P2P peer can flood all 100 slots with fabricated orphan transactions and maintain saturation indefinitely, causing every legitimate orphan from honest peers to be evicted before its parent arrives and silently dropped from automatic promotion.

## Finding Description

**Root cause — `tx-pool/src/component/orphan.rs`:**

The global cap is a compile-time constant: [1](#0-0) 

The pool stores all entries in a single flat `HashMap` with no per-peer sub-structure: [2](#0-1) 

Each `Entry` carries a `peer: PeerIndex` field: [3](#0-2) 

`add_orphan_tx` inserts the entry and immediately calls `limit_size()`: [4](#0-3) 

`limit_size()` evicts by taking `self.entries.keys().next()` — HashMap iteration order, effectively random — and never inspects the `peer` field of any entry: [5](#0-4) 

**Exploit flow:**

1. Attacker connects as a normal P2P peer.
2. Attacker sends 100 `RelayTransaction` messages, each a structurally valid transaction whose single input references a fabricated (random 32-byte) `OutPoint`. Each passes `non_contextual_verify` and, because `is_missing_input` returns true, is admitted via `add_orphan` in `TxPoolService`: [6](#0-5) 
3. Pool is now at capacity (100 attacker-controlled entries).
4. Any subsequent legitimate orphan from an honest peer triggers `limit_size()`, which randomly evicts one entry. The attacker immediately re-sends a replacement to maintain saturation.
5. When the honest peer's parent transaction arrives and `process_orphan_tx` is called, `find_by_previous` returns an empty list for the evicted child: [7](#0-6) 
6. The child is silently absent from the orphan pool and is never promoted to pending.

**Why existing mitigations are insufficient:**

The `TooManyUnknownTransactions` ban tracks unknown *hash announcements* in `RelayTransactionHashes` messages. An attacker sending 100 full `RelayTransaction` messages does not touch that code path and is never banned. [8](#0-7) 

## Impact Explanation

Under sustained attack, the attacker keeps all 100 orphan slots permanently occupied, causing every orphan transaction submitted by honest peers to be evicted before its parent arrives. Affected child transactions are never automatically promoted and must be re-relayed. This constitutes a low-cost, sustained disruption of CKB transaction relay — matching the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (10001–15000 points).

## Likelihood Explanation

The attack requires only a single P2P connection and 100 minimal transactions with fabricated parent hashes. No funds, no privileged access, and no special timing are needed. The attacker can sustain saturation indefinitely by re-sending one replacement orphan each time a slot opens. The 100-slot global limit makes initial saturation trivially achievable in a single burst.

## Recommendation

1. **Add per-peer orphan limits**: Track orphan count per `PeerIndex` inside `OrphanPool` and reject new submissions from a peer that already holds more than `DEFAULT_MAX_ORPHAN_TRANSACTIONS / N` slots.
2. **Evict from the most-represented peer**: Replace the random eviction in `limit_size()` with a policy that selects the entry belonging to the peer with the highest orphan count, protecting honest peers from attacker-controlled saturation.
3. **Increase or make the global limit configurable**: The current hard-coded limit of 100 is very small; even modest honest traffic can be disrupted without an active attacker.

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }
