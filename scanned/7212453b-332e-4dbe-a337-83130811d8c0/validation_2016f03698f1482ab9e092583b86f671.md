Audit Report

## Title
Incomplete Orphan Block Expiry Check Allows Unbounded Memory Growth via Single-Element `need_clean` — (`chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` uses `map.iter().next()` to sample a single arbitrary entry from a `HashMap` group of orphan blocks sharing a parent hash, then uses that one block's epoch to decide whether to evict the entire group. An attacker can insert two competing orphan blocks under the same parent hash — one with an old epoch (expired) and one with a far-future epoch (never expired) — ensuring that whenever the non-expired block is sampled, the entire group is retained indefinitely. Repeating this across many parent hashes causes unbounded growth of the orphan pool, `block_status_map`, and `header_view` caches, leading to node memory exhaustion.

## Finding Description

`InnerPool` stores orphan blocks as:
```
blocks: HashMap<ParentHash, HashMap<packed::Byte32, LonelyBlockHash>>
```
Multiple orphan blocks with the same parent hash (competing forks at the same height) are stored in the same inner `HashMap`.

The expiry check at `orphan_block_pool.rs` lines 112–122 is:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← only ONE element checked
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

`map.iter().next()` returns an **arbitrary** entry from the inner `HashMap` (Rust `HashMap` iteration order is non-deterministic). The epoch of that single block determines whether the **entire group** is evicted. If the sampled block is not expired, all blocks in the group — including genuinely expired ones — are retained.

`clean_expired_blocks` (lines 98–110) calls `need_clean` for every leader and removes the group only if it returns `true`. The cleanup timer fires every 60 seconds from `chain_service.rs` line 61–63.

The epoch field in a CKB block header is **not validated non-contextually**. `BlockVerifier::verify()` in `block_verifier.rs` (lines 39–47) calls only `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, and `MerkleRootVerifier` — no epoch check. The contextual `EpochVerifier` in `header_verifier.rs` (lines 123–148) requires the parent block's epoch (`is_successor_of(parent)`) and cannot be applied to orphan blocks whose parent is unknown. Therefore, an attacker can set the epoch field to any well-formed value and it will be accepted into the orphan pool.

Additionally, `compact_target` (which controls PoW difficulty) is not validated against the chain's expected difficulty in non-contextual verification. An attacker can set `compact_target` to the minimum difficulty, making PoW trivially solvable, while still passing `PowVerifier` (which only checks the nonce against the block's own declared target).

When expired orphans are not returned by `clean_expired_blocks`, `clean_expired_orphans` in `orphan_broker.rs` (lines 134–156) never calls `delete_block`, `remove_header_view`, or `remove_block_status` for them, leaving stale entries permanently in all three structures. The orphan pool has no enforced maximum size — the `with_capacity` argument in `orphan_block_pool.rs` line 30 sets only the initial allocation.

## Impact Explanation

Expired orphan blocks that are not pruned permanently occupy memory in `InnerPool::blocks`, `InnerPool::parents`, `InnerPool::leaders`, and the shared `block_status_map` and `header_view` caches. Because there is no enforced size cap on the orphan pool, an attacker can grow these structures without bound across repeated 60-second cleanup cycles. Each cycle that fails to evict a group leaves the stale entries permanently, since the expiry condition only becomes more true over time — but the non-expired competing block in the group will always prevent the check from firing for that group. Sustained attack leads to OOM and node crash.

**Impact: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

Any peer connected to a CKB node can relay blocks. The attacker crafts two blocks with the same parent hash `P` (unknown to the node), one with an old epoch and one with a far-future epoch. Both use a minimum `compact_target` so PoW is trivially solvable. Both pass `BlockVerifier` (no epoch or difficulty-against-chain check). Both are inserted into the orphan pool under parent `P`. The attack is cheap, repeatable with fresh parent hashes, and requires no privileged access or mining infrastructure beyond solving trivial PoW.

## Recommendation

Replace the single-element check with an all-elements check. Only clean the group if **every** block in it is expired (most conservative, avoids false eviction of groups containing a legitimately recent block):

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().all(|lonely_block| {
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

Alternatively, track the minimum epoch number per group at insertion time to avoid iterating the inner map on every cleanup cycle. Also consider enforcing a hard maximum size on the orphan pool with LRU or FIFO eviction of the oldest entries.

## Proof of Concept

1. Connect to a CKB node as a peer.
2. Choose a parent hash `P` not known to the node.
3. Craft **Block A**: `parent_hash = P`, `epoch = EpochNumberWithFraction::new(1, 0, 1)` (epoch 1, well-formed), `compact_target = minimum`, solve trivial PoW.
4. Craft **Block B**: `parent_hash = P`, `epoch = EpochNumberWithFraction::new(9999, 0, 1)` (epoch 9999, well-formed), `compact_target = minimum`, solve trivial PoW.
5. Send both blocks to the node. Both pass `BlockVerifier` (no epoch or difficulty-against-chain check) and are inserted into `blocks[P]` as `{hash_A: BlockA, hash_B: BlockB}`.
6. Wait for the chain to advance past epoch 7 (`tip_epoch > 1 + EXPIRED_EPOCH = 7`). `clean_expired_blocks` fires every 60 seconds.
7. `need_clean(P, tip_epoch=8)` calls `map.iter().next()`. If it returns Block B (epoch 9999): `9999 + 6 < 8` → false → group not cleaned. Block A (epoch 1, genuinely expired) remains permanently.
8. Repeat steps 2–5 with fresh parent hashes `P1, P2, …`. Roughly half of groups will not be cleaned per cycle (those where the non-expired block is sampled first). Sustained repetition causes unbounded memory growth with no cleanup path.

**Unit test plan**: Write a test that inserts two `LonelyBlockHash` entries under the same parent with epochs 1 and 9999, calls `clean_expired_blocks(8)`, and asserts that the returned `Vec` is empty when the HashMap happens to iterate Block B first — demonstrating the incomplete eviction. pub struct EpochVerifier<'a> {
    parent: EpochNumberWithFraction,
    header: &'a HeaderView,
}

impl<'a> EpochVerifier<'a> {
    pub fn new(parent: EpochNumberWithFraction, header: &'a HeaderView) -> Self {
        EpochVerifier { parent, header }
    }

    pub fn verify(&self) -> Result<(), Error> {
        if !self.header.epoch().is_well_formed() {
            return Err(EpochError::Malformed {
                value: self.header.epoch(),
            }
            .into());
        }
        if !self.parent.is_genesis() && !self.header.epoch().is_successor_of(self.parent) {
            return Err(EpochError::NonContinuous {
                current: self.header.epoch(),
                parent: self.parent,
            }
            .into());
        }
        Ok(())
    }
