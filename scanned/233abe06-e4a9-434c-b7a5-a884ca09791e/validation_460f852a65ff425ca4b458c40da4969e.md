### Title
Incorrect `ancestors_count` Decrement During Cell-Dep Conflict Eviction Causes Unnecessary Transaction Over-Eviction - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `check_and_record_ancestors`, when a new transaction's ancestor count exceeds `max_ancestors_count` and the excess is attributable to `cell_ref_parents` (pool transactions that use the same cell as a dep that the new tx wants to consume as an input), the code evicts those cell_ref_parents one at a time. However, each call to `remove_entry_and_descendants` removes the target **and all its descendants**, while `ancestors_count` is only decremented by a fixed `1` per iteration. This mirrors the Lido bug class exactly: an incorrect fixed-value adjustment to a limit/count when removing items from a collection, without accounting for the actual number of items removed.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, the function `check_and_record_ancestors` handles the case where a new transaction has too many ancestors. When the excess is due to `cell_ref_parents`, it enters an eviction loop:

```rust
// lines 615-625
let mut iter = evict_candidates.iter();
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        ancestors_count = ancestors_count.saturating_sub(1); // <-- BUG
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
```

`remove_entry_and_descendants(next_id)` removes `next_id` **and all its descendants** from the pool. If any of those descendants are also ancestors of the new transaction (e.g., `tx_A` is a cell_ref_parent, `tx_B` spends `tx_A`'s output, and the new tx spends `tx_B`'s output), then removing `tx_A` also removes `tx_B` from the ancestor set. But `ancestors_count` is only decremented by `1`, not by the actual number of ancestors removed.

This causes the while loop to run more iterations than necessary, evicting additional `cell_ref_parents` that did not need to be removed.

The `get_tx_ancenstors` function (lines 517–553) computes `ancestors` as the full transitive closure of all pool-resident parents. `ancestors_count = ancestors.len() + 1`. When `remove_entry_and_descendants` removes a chain of entries that are all in `ancestors`, the local counter `ancestors_count` falls behind the true count by the number of extra ancestors removed. ### Impact Explanation

Valid, higher-fee transactions already in the tx-pool are unnecessarily evicted. Specifically, when a new transaction is submitted that triggers the cell-dep conflict eviction path, the pool removes more `cell_ref_parents` (and their descendants) than the minimum required to bring the ancestor count below `max_ancestors_count`. This degrades tx-pool quality: fee-paying transactions are discarded without cause, reducing miner revenue and disrupting legitimate users whose transactions are silently dropped.

The assert at line 636 (`assert!(ancestors.len() < self.max_ancestors_count)`) does not catch this because the over-eviction makes the actual ancestor count *lower* than the limit, not higher — the assert passes while the damage (unnecessary evictions) has already occurred. [4](#0-3) 

---

### Likelihood Explanation

The trigger condition is:
1. A `cell_ref_parent` `tx_A` is in the pool.
2. A descendant of `tx_A` (e.g., `tx_B` spending `tx_A`'s output) is also in the pool and is an ancestor of the new transaction (the new tx spends `tx_B`'s output).
3. The new transaction's total ancestor count exceeds `max_ancestors_count`, but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (so the eviction branch at line 603 is taken, not the hard rejection at line 627).

This is a realistic scenario reachable by any unprivileged RPC caller submitting transactions via `send_transaction`. The existing integration test in `test/src/specs/tx_pool/limit.rs` (`TxPoolLimitAncestorCount`) already exercises the eviction path, confirming it is reachable in practice, though it does not test the multi-ancestor-per-cell_ref_parent case. [5](#0-4) [6](#0-5) 

---

### Recommendation

Replace the fixed `saturating_sub(1)` with a decrement equal to the number of entries actually removed that were members of the original `ancestors` set. One correct approach:

```rust
let removed = self.remove_entry_and_descendants(next_id);
// Count how many removed entries were in the ancestor set
let removed_ancestor_count = removed.iter()
    .filter(|e| ancestors.contains(&e.proposal_short_id()))
    .count();
ancestors_count = ancestors_count.saturating_sub(removed_ancestor_count.max(1));
parents.remove(next_id);
evicted.extend(removed);
```

Alternatively, after each eviction, recompute `ancestors_count` from the updated `parents` set using `calc_relation_ids`, which is already done after the loop anyway.

### Proof of Concept

**Setup:**
- `max_ancestors_count = 3`
- Submit `tx_A` to pool (uses `cell_X` as a dep).
- Submit `tx_B` spending `tx_A`'s output (so `tx_B` is a descendant of `tx_A`).
- Submit `tx_C` spending `tx_A`'s output (another cell_ref_parent, independent of `tx_B`).
- Submit `tx_NEW` that: (a) spends `tx_B`'s output (making `tx_A` and `tx_B` input-ancestors), and (b) consumes `cell_X` as an input (making `tx_A` a cell_ref_parent).

**State before `tx_NEW` submission:**
- `ancestors = {tx_A, tx_B}`, `ancestors_count = 3` (tx_A + tx_B + tx_NEW itself)
- `cell_ref_parents = {tx_A}` (tx_A uses cell_X as dep, tx_NEW consumes cell_X as input)
- `ancestors_count - cell_ref_parents.len() = 3 - 1 = 2 <= 3` → eviction branch taken

**Eviction loop iteration 1:**
- `ancestors_count (3) > max_ancestors_count (3)` → **false**, loop does not execute

Wait, 3 > 3 is false. Let me adjust: `max_ancestors_count = 2`.

- `ancestors = {tx_A, tx_B}`, `ancestors_count = 3`
- `cell_ref_parents = {tx_A, tx_C}` (both use cell_X as dep)
- `ancestors_count - cell_ref_parents.len() = 3 - 2 = 1 <= 2` → eviction branch taken
- `evict_candidates = [tx_C, tx_A]` (sorted by lowest fee first)

**Iteration 1:** evict `tx_C` (lowest fee). `remove_entry_and_descendants(tx_C)` removes only `tx_C`. `ancestors_count = 3 - 1 = 2`. Loop condition `2 > 2` → false. Loop exits.

**Iteration 1 (corrected scenario):** evict `tx_A` first (lowest fee). `remove_entry_and_descendants(tx_A)` removes `tx_A` AND `tx_B` (2 entries, both ancestors). But `ancestors_count = 3 - 1 = 2`. Loop condition `2 > 2` → false. Loop exits. **Correct behavior here.**

Let me construct the exact triggering case: `max_ancestors_count = 3`, `ancestors_count = 5`, `cell_ref_parents = {tx_A, tx_C}` where `tx_A` has descendant `tx_B` also in ancestors.

- `ancestors_count - cell_ref_parents.len() = 5 - 2 = 3 <= 3` → eviction branch
- Need to reduce from 5 to ≤ 3, so need to remove ≥ 2 ancestors.
- `evict_candidates = [tx_A, tx_C]` (tx_A lowest fee)
- **Iteration 1:** evict `tx_A`. `remove_entry_and_descendants(tx_A)` removes `tx_A` + `tx_B` (2 actual ancestors). `ancestors_count = 5 - 1 = 4`. Loop: `4 > 3` → true.
- **Iteration 2:** evict `tx_C`. `remove_entry_and_descendants(tx_C)` removes `tx_C`. `ancestors_count = 4 - 1 = 3`. Loop: `3 > 3` → false. Loop exits.

**Result:** `tx_C` is unnecessarily evicted. After removing `tx_A` and `tx_B`, the actual ancestor count was already `5 - 2 = 3 ≤ max_ancestors_count`, but the loop continued because `ancestors_count` was only decremented by 1 (to 4), not by 2 (to 3). `tx_C` — a valid, fee-paying transaction — is evicted without cause.     fn get_tx_ancenstors(
        &self,
        entry: &TransactionView,
    ) -> (
        HashSet<ProposalShortId>,
        HashSet<ProposalShortId>,
        HashSet<ProposalShortId>,
    ) {
        let mut parents: HashSet<ProposalShortId> =
            HashSet::with_capacity(entry.inputs().len() + entry.cell_deps().len());
        let mut cell_ref_parents: HashSet<ProposalShortId> = Default::default();

        for input in entry.inputs() {
            let input_pt = input.previous_output();
            if let Some(deps) = self.edges.deps.get(&input_pt) {
                cell_ref_parents.extend(deps.iter().cloned());
                parents.extend(deps.iter().cloned());
            }

            let id = ProposalShortId::from_tx_hash(&input_pt.tx_hash());
            if self.links.inner.contains_key(&id) {
                parents.insert(id);
            }
        }
        for cell_dep in entry.cell_deps() {
            let dep_pt = cell_dep.out_point();
            let id = ProposalShortId::from_tx_hash(&dep_pt.tx_hash());
            if self.links.inner.contains_key(&id) {
                parents.insert(id);
            }
        }

        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        (ancestors, parents, cell_ref_parents)
    }

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }
