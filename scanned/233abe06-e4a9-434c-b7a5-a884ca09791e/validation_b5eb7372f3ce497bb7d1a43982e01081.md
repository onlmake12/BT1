### Title
Missing Time-Relative Re-Validation When Promoting Transactions to Proposed Pool During Chain Reorg - (File: `tx-pool/src/process.rs`)

### Summary
During a chain reorganization, the tx-pool promotes pending transactions to `Gap` or `Proposed` status based solely on proposal-window membership, without re-running the `TimeRelativeTransactionVerifier` (`since` / cellbase-maturity checks) against the new chain tip. A transaction whose `since` absolute-block-number lock was satisfied at the old tip may no longer be satisfied after the reorg rolls the chain back, yet it is silently promoted into the proposed pool. The block assembler then includes it in a block template; the block is mined and broadcast, but rejected by every verifier node because `ContextualTransactionVerifier` re-checks `since` at commit time. This wastes miner resources and can be triggered by any tx-pool submitter who times a submission to coincide with a natural reorg.

### Finding Description
`_update_tx_pool_for_reorg` in `tx-pool/src/process.rs` iterates over pending entries and promotes them to `Proposed` or `Gap` status if their proposal short-id appears in the new snapshot's proposal window:

```rust
for (id, entry) in proposals {
    if let Err(e) = tx_pool.proposed_rtx(&id) { ... }
}
for (id, entry) in gaps {
    if let Err(e) = tx_pool.gap_rtx(&id) { ... }
}
``` [1](#0-0) 

`proposed_rtx` and `gap_rtx` only flip the pool-map status flag; neither calls `time_relative_verify`: [2](#0-1) [3](#0-2) 

The function that enforces both `MaturityVerifier` and `SinceVerifier` is `time_relative_verify` in `tx-pool/src/util.rs`: [4](#0-3) 

This function is correctly called in the concurrent-submission path (`submit_entry`) when the snapshot changes between resolve and submit: [5](#0-4) 

It is also correctly called in `readd_detached_tx` (for transactions that were committed in the detached blocks) via `verify_rtx`: [6](#0-5) 

But it is **not** called for transactions that were already in the pending pool and are being promoted to `Gap`/`Proposed` during the reorg.

The `SinceVerifier` evaluates absolute block-number locks against `tx_env.block_number(proposal_window)`. For a `TxStatus::Proposed` environment this resolves to approximately `tip_number + 1`: [7](#0-6) 

After a reorg that reduces `tip_number`, a transaction whose `since = absolute block N` was valid at the old tip (old `tip_number >= N - proposal_window`) may no longer be valid at the new tip (new `tip_number + 1 < N`), yet it is promoted to `Proposed` without this being checked.

### Impact Explanation
A transaction with an unsatisfied `since` absolute-block-number (or epoch, or timestamp) lock lands in the proposed pool. The block assembler (`get_block_template`) draws from the proposed pool and includes the transaction in the next block template. The miner expends proof-of-work on this template. When the mined block is submitted, `ContextualTransactionVerifier::verify` — which calls `self.time_relative.verify()` — rejects the block: [8](#0-7) 

The miner's work is wasted. Repeated natural reorgs (or an adversary who can induce short reorgs) can cause a targeted miner to repeatedly produce invalid blocks, degrading their effective hash-rate.

### Likelihood Explanation
Short reorgs (1–2 blocks) occur naturally on any proof-of-work chain without any attacker involvement. An unprivileged tx-pool submitter can craft a transaction with `since = absolute block N` and submit it when the chain is at block `N - proposal_window.0`, so it is accepted into the pending pool. If a natural reorg then rolls the chain back even one block, the condition `tip_number + 1 >= N` fails, but the promotion logic in `_update_tx_pool_for_reorg` does not detect this. No privileged access, leaked key, or majority hashpower is required to submit the transaction; the reorg is a normal network event.

### Recommendation
Before calling `tx_pool.proposed_rtx(&id)` or `tx_pool.gap_rtx(&id)` inside `_update_tx_pool_for_reorg`, construct a `TxVerifyEnv` from the new snapshot's tip header (using `TxStatus::Proposed` / `TxStatus::Gap` respectively) and call `time_relative_verify(snapshot, entry.rtx.clone(), tx_env)`. Reject and evict entries that fail this check, mirroring the guard already present in `submit_entry` for context-switch scenarios.

### Proof of Concept
1. Chain is at block `N = 100`, `proposal_window = (2, 10)`.
2. Attacker submits a transaction `T` with `since = absolute block 100` (accepted because `100 + 2 >= 100`).
3. `T` enters the pending pool.
4. A natural 3-block reorg occurs; the new tip is block 97.
5. `_update_tx_pool_for_reorg` runs. If `T`'s proposal short-id appears in the proposal window of the new snapshot, `proposed_rtx` is called — no `since` re-check is performed.
6. `T` is now in the proposed pool. The block assembler includes `T` in the next block template.
7. The miner mines the block and broadcasts it.
8. Every node's `ContextualTransactionVerifier` evaluates `SinceVerifier::verify_absolute_lock` with `block_number = 97 + 1 = 98 < 100` and returns `TransactionError::Immature { index }`, rejecting the block. [9](#0-8) [10](#0-9)

fn _update_tx_pool_for_reorg(
    tx_pool: &mut TxPool,
    attached: &LinkedHashSet<TransactionView>,
    detached_headers: &HashSet<Byte32>,
    detached_proposal_id: HashSet<ProposalShortId>,
    snapshot: Arc<Snapshot>,
    callbacks: &Callbacks,
    mine_mode: bool,
) {
    tx_pool.snapshot = Arc::clone(&snapshot);

    // NOTE: `remove_by_detached_proposal` will try to re-put the given expired/detached proposals into
    // pending-pool if they can be found within txpool. As for a transaction
    // which is both expired and committed at the one time(commit at its end of commit-window),
    // we should treat it as a committed and not re-put into pending-pool. So we should ensure
    // that involves `remove_committed_txs` before `remove_expired`.
    tx_pool.remove_committed_txs(attached.iter(), callbacks, detached_headers);
    tx_pool.remove_by_detached_proposal(detached_proposal_id.iter());

    // mine mode:
    // pending ---> gap ----> proposed
    // try move gap to proposed
    if mine_mode {
        let mut proposals = Vec::new();
        let mut gaps = Vec::new();

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) {
            let short_id = entry.inner.proposal_short_id();
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push((short_id, entry.inner.clone()));
            }
        }

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) {
            let short_id = entry.inner.proposal_short_id();
            let elem = (short_id.clone(), entry.inner.clone());
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push(elem);
            } else if snapshot.proposals().contains_gap(&short_id) {
                gaps.push(elem);
            }
        }

        for (id, entry) in proposals {
            debug!("begin to proposed: {:x}", id);
            if let Err(e) = tx_pool.proposed_rtx(&id) {
                debug!(
                    "Failed to add proposed tx {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e);
            } else {
                callbacks.call_proposed(&entry)
            }
        }

        for (id, entry) in gaps {
            debug!("begin to gap: {:x}", id);
            if let Err(e) = tx_pool.gap_rtx(&id) {
                debug!(
                    "Failed to add tx to gap {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e.clone());
            }
        }
    }

    // Remove expired transaction from pending
    tx_pool.remove_expired(callbacks);

    // Remove transactions from the pool until its size <= size_limit.
    let _ = tx_pool.limit_size(callbacks, None);

pub(crate) fn time_relative_verify(
    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: TxVerifyEnv,
) -> Result<(), Reject> {
    let consensus = snapshot.cloned_consensus();
    TimeRelativeTransactionVerifier::new(
        rtx,
        consensus,
        snapshot.as_data_loader(),
        Arc::new(tx_env),
    )
    .verify()
    .map_err(Reject::Verification)

    fn verify_absolute_lock(&self, index: usize, since: Since) -> Result<(), Error> {
        if since.is_absolute() {
            match since.extract_metric() {
                Some(SinceMetric::BlockNumber(block_number)) => {
                    let proposal_window = self.consensus.tx_proposal_window();
                    if self.tx_env.block_number(proposal_window) < block_number {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::EpochNumberWithFraction(epoch_number_with_fraction)) => {
                    if !epoch_number_with_fraction.is_well_formed_increment() {
                        return Err((TransactionError::InvalidSince { index }).into());
                    }
                    let a = self.tx_env.epoch().to_rational();
                    let b = epoch_number_with_fraction.normalize().to_rational();
                    if a < b {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                Some(SinceMetric::Timestamp(timestamp)) => {
                    let parent_hash = self.tx_env.parent_hash();
                    let tip_timestamp = self.block_median_time(&parent_hash);
                    if tip_timestamp < timestamp {
                        return Err((TransactionError::Immature { index }).into());
                    }
                }
                None => {
                    return Err((TransactionError::InvalidSince { index }).into());
                }
            }
        }
        Ok(())
    }
