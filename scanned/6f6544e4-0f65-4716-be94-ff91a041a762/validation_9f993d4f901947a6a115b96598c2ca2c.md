Audit Report

## Title
Unbounded CPU Exhaustion via `test_tx_pool_accept` RPC with No Rate Limiting or Deduplication — (`tx-pool/src/process.rs`)

## Summary

The `test_tx_pool_accept` RPC endpoint triggers a full CKB-VM script execution (up to `max_block_cycles` = 70,000,000 cycles) per call via `_test_accept_tx`, with `command_rx = None` (non-interruptible), in a freshly spawned tokio task. Because the transaction is never inserted into the pool, the same transaction (with varied witnesses to bypass the verify cache) can be resubmitted indefinitely at zero on-chain cost. There is no rate limiting, no concurrency cap, and no pool-full check on this path, allowing an attacker to saturate the node's tokio thread pool and starve legitimate transaction processing.

## Finding Description

**Entrypoint:** `rpc/src/module/pool.rs` L637–660 — `test_tx_pool_accept` sends a `Message::TestAcceptTx` to the tx-pool service channel with no rate limiting or quota check. [1](#0-0) 

**Service loop:** `tx-pool/src/service.rs` L616–632 — every received message is immediately dequeued and spawned as a new concurrent tokio task via `handle_clone.spawn(process(service_clone, message))`. The channel capacity of 512 (`DEFAULT_CHANNEL_SIZE`) does not limit concurrent executions — it only limits queued messages. [2](#0-1) [3](#0-2) 

**Message handler:** `tx-pool/src/service.rs` L835–843 — `TestAcceptTx` calls `service.test_accept_tx(tx).await` directly in the spawned task. [4](#0-3) 

**Deduplication gap:** `tx-pool/src/process.rs` L386–399 — `test_accept_tx` checks only `verify_queue_contains` and `orphan_contains`. Since `_test_accept_tx` never inserts the tx into either structure, the same transaction passes the duplicate check on every subsequent call. [5](#0-4) 

**Root cause:** `tx-pool/src/process.rs` L779–800 — `_test_accept_tx` calls `verify_rtx` with `max_cycles = self.consensus.max_block_cycles()` (70,000,000) and `command_rx = None`. Passing `None` means the CKB-VM execution runs to completion with no chunking or interruption, blocking the tokio worker thread for the full duration. [6](#0-5) 

**Contrast with `VerifyMgr`:** `tx-pool/src/verify_mgr.rs` L147–163 — the normal verification path passes `Some(&mut self.command_rx)`, enabling chunked/interruptible execution. `_test_accept_tx` deliberately omits this, making it non-interruptible. [7](#0-6) 

**Verify cache bypass:** The `txs_verify_cache` is keyed by `witness_hash`. An attacker trivially bypasses it by varying witness bytes across calls, ensuring each call performs a full VM execution.

## Impact Explanation

Each concurrent `test_tx_pool_accept` call with a near-max-cycle script pins a tokio worker thread for the duration of the VM execution. With N concurrent connections, N tokio threads are occupied. The node's default worker count is 3/4 of CPU cores; a small number of concurrent attacker connections can fully saturate available compute. [8](#0-7) 

All other tokio tasks — including `send_transaction`, block propagation, and peer communication — compete for the same thread pool. Sustained flooding renders the node effectively unresponsive to legitimate operations.

**Matched impact:** High (10001–15000 points) — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs." A secondary match is High — "Vulnerabilities which could easily crash a CKB node" (via resource exhaustion leading to unresponsiveness).

## Likelihood Explanation

- The Pool RPC module is enabled by default and commonly exposed for dApp use.
- The attacker only needs knowledge of any live cell on-chain (publicly visible) to craft a transaction that passes `pre_check`.
- Zero on-chain cost: the transaction is never broadcast or inserted into the pool.
- The same transaction with varied witnesses can be reused indefinitely, bypassing the verify cache.
- No special privileges, keys, or network position are required — any external HTTP client suffices.

## Recommendation

1. **Concurrency cap:** Enforce a semaphore (e.g., bounded to `max_tx_verify_workers`) on concurrent `TestAcceptTx` tasks to prevent thread pool saturation.
2. **Cycle-based rate limiting:** Before calling `_test_accept_tx`, apply a per-IP or global rate limit based on declared or estimated cycles.
3. **Route through VerifyQueue:** Subject `test_accept_tx` to the existing `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB) cap so it competes fairly with real submissions and is bounded.
4. **Pass a live `command_rx`:** Use an interruptible execution path (as `VerifyMgr` does) so the node can reclaim threads under load.

## Proof of Concept

```python
import requests, threading

def flood(nonce):
    # Build a tx referencing a known live cell with a near-max-cycle script.
    # Vary witness bytes by nonce to bypass txs_verify_cache (keyed by witness_hash).
    tx = build_near_max_cycle_tx(witness_nonce=nonce)
    while True:
        requests.post("http://target:8114", json={
            "jsonrpc": "2.0", "method": "test_tx_pool_accept",
            "params": [tx, "passthrough"], "id": 1
        }, timeout=120)
        nonce += 1000  # vary witness each iteration

# Saturate with N concurrent threads (N = number of tokio worker threads)
for i in range(16):
    threading.Thread(target=flood, args=(i,), daemon=True).start()

# Measure send_transaction latency from a separate client.
# Expected: >10x latency increase; node becomes unresponsive to legitimate calls.
```

Each iteration triggers a full 70M-cycle non-interruptible VM execution in a new tokio task. With no rate limiting, no deduplication, and no concurrency cap, the node's thread pool is saturated and legitimate pool operations are starved.

    pub(crate) async fn test_accept_tx(&self, tx: TransactionView) -> Result<Completed, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, None).await?;

        if self.verify_queue_contains(&tx).await {
            return Err(Reject::Duplicated(tx.hash()));
        }

        if self.orphan_contains(&tx).await {
            debug!("reject tx {} already in orphan pool", tx.hash());
            return Err(Reject::Duplicated(tx.hash()));
        }
        self._test_accept_tx(tx.clone()).await
    }
