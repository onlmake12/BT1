Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Unauthenticated Peer `GetBlockProposal` Messages - (File: sync/src/types/mod.rs)

## Summary
`SyncState.pending_get_block_proposals` is a bare `DashMap` with no capacity bound. Any peer that completes the CKB P2P handshake can flood `GetBlockProposal` messages containing up to 3,000 unique `ProposalShortId` values absent from the local tx pool per message, causing unbounded insertion into the map with no eviction until the next periodic timer tick. This enables memory exhaustion and OOM-kill of the victim node.

## Finding Description
`pending_get_block_proposals` is declared as a plain `DashMap` with no size limit: [1](#0-0) 

`insert_get_block_proposals` iterates over all supplied IDs and inserts them unconditionally with no size check: [2](#0-1) 

`GetBlockProposalProcess::execute` filters out IDs found in the tx pool and passes all remaining IDs directly to `insert_get_block_proposals`: [3](#0-2) 

The only per-message guard is a count ceiling of `max_block_proposals_limit * max_uncles_num` (mainnet: 1500 × 2 = 3,000): [4](#0-3) 

This guard limits IDs *per message*, not the total map size or per-peer contribution rate. The sole cleanup path is `drain_get_block_proposals`, called from `prune_tx_proposal_request` on a periodic timer: [5](#0-4) 

Between consecutive timer ticks, an attacker can insert an unbounded number of entries. There is no per-peer quota, no total-map cap, and no eviction policy.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

Each `ProposalShortId` is 10 bytes; with `DashMap` shard overhead and the `HashSet<PeerIndex>` value, each entry costs on the order of ~100–200 bytes. A single peer sending 3,000-ID messages at network speed can insert millions of entries per timer interval, exhausting available RAM and causing the node process to be OOM-killed or to stall under allocator pressure. This disrupts block relay, transaction propagation, and all other node services for the duration of the attack.

## Likelihood Explanation
**Medium.** Any peer that completes the standard CKB P2P handshake can send `GetBlockProposal` messages. No stake, bond, or privileged role is required. The attacker generates random 10-byte `ProposalShortId` values that will not be found in the victim's tx pool, ensuring every ID is cached. The cost to the attacker is negligible network bandwidth; the cost to the victim is unbounded memory growth.

## Recommendation
1. **Hard cap on `pending_get_block_proposals`**: In `insert_get_block_proposals`, reject insertions once `self.pending_get_block_proposals.len()` exceeds a configurable maximum (e.g., `max_block_proposals_limit * max_connected_peers`).
2. **Per-peer entry limit**: Track how many entries each peer has contributed and refuse further insertions from peers that exceed their quota, mirroring the per-peer guard already present in `add_ask_for_txs` for `unknown_tx_hashes`.
3. **Eviction on overflow**: When the cap is reached, evict entries from the most prolific peer rather than silently accepting new ones.

## Proof of Concept
```
Attacker (any connected CKB peer):
  loop:
    ids = [random_10_bytes() for _ in range(3000)]  # all absent from victim tx pool
    send GetBlockProposal { block_hash: any_valid_hash, proposals: ids }
    # Each iteration inserts up to 3000 new entries into pending_get_block_proposals
    # with no eviction until the next prune_tx_proposal_request timer tick
```

Exploit path:
1. Peer sends `GetBlockProposal` → `Relayer::try_process` dispatches to `GetBlockProposalProcess::execute`.
2. Per-message count check passes (≤ 3,000 IDs).
3. `tx_pool.fetch_txs(proposals)` returns empty (IDs are random/unknown).
4. All 3,000 IDs are passed to `insert_get_block_proposals` → inserted into `pending_get_block_proposals` with no size guard.
5. Repeat before the next `prune_tx_proposal_request` timer fires.

A unit test can confirm this by calling `insert_get_block_proposals` in a loop with fresh random IDs and asserting that `pending_get_block_proposals.len()` grows without bound, while `drain_get_block_proposals` is not called.
