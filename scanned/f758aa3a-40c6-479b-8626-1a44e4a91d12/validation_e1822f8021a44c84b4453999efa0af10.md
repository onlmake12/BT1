Audit Report

## Title
Missing Peer-Type Verification Allows Arbitrary Peers to Subscribe to Light Client State Broadcasts — (`util/light-client-protocol-server/src/components/get_last_state.rs`)

## Summary
`GetLastStateProcess::execute()` unconditionally sets `peer.if_lightclient_subscribed = true` on any peer that sends `GetLastState` with `subscribe=true`, without verifying the peer is a light-client-only peer. Because `ProtocolTypeCheckerService::opened_protocol_type()` only requires that a peer has opened the `Sync` protocol (the sole entry in `required_protocol_ids`), a peer that opens both `Sync` and `LightClient` passes all checks, subscribes, and receives `SendLastState` broadcasts on every new block. The actual CPU impact is lower than claimed (MMR root is computed once per block, not once per subscriber), but the bandwidth cost scales linearly with the number of malicious subscribers.

## Finding Description
In `util/light-client-protocol-server/src/components/get_last_state.rs` lines 29–38, `execute()` sets the subscription flag with no peer-type guard:

```rust
pub(crate) async fn execute(self) -> Status {
    let subscribe: bool = self.message.subscribe().into();
    if subscribe {
        self.nc.with_peer_mut(self.peer, Box::new(|peer| {
            peer.if_lightclient_subscribed = true;  // no type check
        }));
    }
    ...
}
```

The `if_lightclient_subscribed` field lives on the shared `Peer` struct (`network/src/peer.rs` lines 86–87), used for all peer types.

`ProtocolTypeCheckerService::opened_protocol_type()` (`network/src/services/protocol_type_checker.rs` lines 108–123) uses `.all()` against `fully_open_required_protocol_ids`. In `util/launcher/src/lib.rs` line 495, that list is `vec![SupportProtocols::Sync.protocol_id()]` — only `Sync`. A peer opening `Sync` + `LightClient` returns `Ok(ProtocolType::FullyOpen)` and is never disconnected.

On every new block, both `submit_block` (`rpc/src/module/miner.rs` lines 323–367) and `build_and_broadcast_compact_block` (`sync/src/relayer/mod.rs` lines 745–792) compute one MMR root, build a `VerifiableHeader`, and broadcast `SendLastState` over `SupportProtocols::LightClient` to every peer where `if_lightclient_subscribed == true`. The broadcast is a `filter_broadcast` that sends one message per subscribed peer, so bandwidth scales with subscriber count.

Note: the MMR root computation (`mmr.get_root()`) is performed **once per block**, not once per subscriber — the CPU impact claim in the submission is overstated. The real cost is one network send per subscribed peer per block.

## Impact Explanation
An attacker controlling `N` connections (bounded by `max_peers = 125` per the default config) causes the node to perform `N` extra network sends of `SendLastState` on every block arrival (~every 10 seconds). Each message carries a `VerifiableHeader` (header data + uncles hash + extension + MMR root). This is a bounded but real bandwidth drain that degrades service for legitimate light clients sharing the same connection slots and wastes outbound bandwidth on the victim node. This fits: **Low (501–2000 points) — Any other important performance improvements for CKB**.

## Likelihood Explanation
Any unprivileged external peer can execute this with minimal effort: connect, open `Sync` + `LightClient` protocols (both advertised in the default `support_protocols` config), and send one `GetLastState(subscribe=true)` message. No keys, privileges, or hashpower are required. The attack is repeatable up to the connection limit and persists for the lifetime of the connection.

## Recommendation
In `GetLastStateProcess::execute()`, before setting `if_lightclient_subscribed = true`, verify that the peer has **not** opened the full-node sync/relay protocols. Concretely, check `peer.protocols` for the absence of `SupportProtocols::Sync.protocol_id()` (and optionally `RelayV3`) before granting the subscription. Alternatively, extend `ProtocolTypeCheckerService` to recognize a `LightClientOnly` type and disconnect peers that open both full-node and light-client protocols simultaneously.

## Proof of Concept
1. Attacker connects to a CKB full node with `LightClient` in its `support_protocols`.
2. Attacker opens `Sync` (protocol ID 100) and `LightClient` (protocol ID 120) — satisfying `ProtocolTypeCheckerService` since only `Sync` is in `required_protocol_ids`.
3. Attacker sends a `LightClientMessage::GetLastState` with `subscribe = true` on the `LightClient` channel.
4. `GetLastStateProcess::execute()` sets `peer.if_lightclient_subscribed = true` with no type check.
5. On every subsequent block, the node sends `SendLastState` to the attacker's peer via `filter_broadcast`.
6. Repeating with up to `max_peers` (125) connections maximizes bandwidth drain per block.
