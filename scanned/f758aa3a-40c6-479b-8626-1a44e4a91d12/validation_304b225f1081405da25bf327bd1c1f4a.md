Audit Report

## Title
Unvalidated `observed_addr` Injection Poisons Victim Node's Advertised Listen Addresses — (File: `network/src/protocols/identify/mod.rs`)

## Summary
Any unprivileged remote peer can send an identify message whose `observed_addr` field contains an arbitrary globally-routable `Multiaddr`. Because no reachability or ownership validation is performed at any point in the call chain, the victim node stores the attacker-controlled address and subsequently broadcasts it as one of its own listen addresses to every new peer that connects. This violates the invariant that a node must only advertise its own verified listen addresses, and enables low-cost, cascading address-table pollution across the CKB P2P network.

## Finding Description
**Step 1 — `process_observed` accepts any address without validation.**
`process_observed` (lines 152–169) takes the raw `observed` `Multiaddr` from the incoming identify message and immediately calls `add_observed_addr`. There is no check that the address matches `session.address`, no IP reachability check at insertion time, and no size limit. The function unconditionally returns `MisbehaveResult::Continue`.

**Step 2 — `IdentifyCallback::add_observed_addr` stores the address verbatim.**
Lines 497–507: the only transformation is appending the victim's own peer ID if absent. The address is then written into `NetworkState::observed_addrs` keyed by `session_id`. Always returns `MisbehaveResult::Continue`.

**Step 3 — `NetworkState::add_observed_addr` performs no validation.**
Lines 513–516: the address is inserted into `observed_addrs: RwLock<HashMap<PeerIndex, Multiaddr>>` with no further checks.

**Step 4 — `local_listen_addrs` pads with attacker-controlled addresses.**
Lines 458–470: when the victim has fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses — the common case for NAT-ed nodes — `local_listen_addrs` fills the remainder from `observed_addrs`, directly including the attacker's injected address.

**Step 5 — `connected` broadcasts the poisoned list to every new peer.**
Lines 206–237: when any new peer connects, the victim immediately sends an identify message whose `listen_addrs` is built from `local_listen_addrs()`. The only filter applied is `is_reachable`, which merely checks that the IP is globally routable — a condition the attacker trivially satisfies by using any real public IP (e.g., `8.8.8.8`).

**Step 6 — Cleanup is too late.**
Lines 814–817: the `observed_addrs` entry is only removed on `SessionClose`. While the attacker remains connected, every new peer that connects to the victim receives the poisoned address.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

An attacker with a single TCP connection can inject a chosen address into the victim's advertised address set. The victim then propagates this poisoned address to every subsequent peer via the identify protocol. Those peers store it in their peer stores and may re-broadcast it via the discovery protocol, amplifying the pollution network-wide. At scale — an attacker connecting to many nodes simultaneously — this causes widespread address-table corruption across the CKB P2P network: peers repeatedly attempt connections to poisoned addresses, fail, and retry, generating abnormal connection churn and degrading effective network connectivity. NAT-ed nodes (the majority of CKB nodes) are fully exposed because they have zero configured public addresses and rely entirely on `observed_addrs` to fill their advertised list, meaning the attacker can control the entire advertised address set with a single connection.

## Likelihood Explanation
The attack requires only a standard P2P connection — no special privileges, no PoW, no key material. The identify protocol is mandatory for all peers. The precondition (victim has fewer than 10 public addresses) is the default state for any node behind NAT. The `is_reachable` filter in `connected()` is the only guard, and it is trivially bypassed by using any real public IP. A single attacker can maintain many simultaneous connections to different victim nodes, keeping all poisoned entries alive indefinitely.

## Recommendation
1. **Validate ownership of `observed_addr` at insertion time:** In `process_observed`, reject any `observed_addr` that does not match the actual transport address of the session (`session.address`). The observed address is supposed to tell the local node what its own external address looks like from the remote peer's perspective — it must correspond to the session's remote endpoint, not an arbitrary value supplied by the peer.
2. **Apply `is_reachable` at insertion time** in `add_observed_addr` / `NetworkState::add_observed_addr`, not only at broadcast time in `connected()`.
3. **Require quorum confirmation:** Only promote an observed address to the advertised list if multiple independent peers report the same address, preventing a single attacker from injecting an address.
