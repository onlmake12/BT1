Audit Report

## Title
Tx-Pool Minimum Fee Rate Admission Check Uses Serialized Size Instead of Transaction Weight, Allowing Below-Minimum Fee Rate Entries - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the transaction's serialized byte size to compute the minimum required fee. After `verify_rtx` returns the actual execution cycles, no second fee rate check is performed against the true transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). An unprivileged submitter can craft a transaction with a small serialized size but high script execution cycles that passes admission with an effective fee rate far below the configured minimum, consuming verification CPU and polluting the pool at reduced cost.

## Finding Description
`check_tx_fee` computes the minimum fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The inline comment acknowledges this is intentional as a "cheap check" because cycles are unknown at `pre_check` time. However, the code never performs a second check after cycles become known.

The full `_process_tx` flow is:
1. `pre_check` → `check_tx_fee(tx_pool, snapshot, rtx, tx_size)` — size-only gate, cycles unknown.
2. `verify_rtx(...)` — scripts execute, `verified.cycles` is now known.
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry constructed with actual cycles but fee already approved against size-only check.
4. `submit_entry(tip_hash, entry, status)` — no fee rate re-validation against actual weight.

`submit_entry` performs RBF checks, time-relative re-verification on tip change, and `limit_size` eviction, but contains no call to `check_tx_fee` or any equivalent weight-based fee rate gate.

Meanwhile, `TxEntry::fee_rate()` — used for eviction ordering and block assembly scoring — correctly computes:

```rust
let weight = get_transaction_weight(self.size, self.cycles);
FeeRate::calculate(self.fee, weight)
```

This creates a permanent split: admission uses `size`, but all post-admission accounting uses `max(size, cycles × 0.000_170_571_4)`. A transaction with `size=200`, `cycles=5_000_000` has actual weight ≈ 852, but is admitted if `fee ≥ min_fee_rate × 200 / 1000`, yielding an effective fee rate of `fee × 1000 / 852` — roughly 4× below the configured minimum.

## Impact Explanation
This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can submit a stream of transactions that each:
- Pass the size-based admission gate (paying only `min_fee_rate × size / 1000` shannons)
- Trigger full `ContextualTransactionVerifier` + RISC-V script execution consuming up to `max_tx_verify_cycles` cycles per transaction
- Are admitted to the pool with an effective fee rate well below `min_fee_rate`

The cost per unit of CPU consumed is reduced by the ratio `actual_weight / size`, which grows linearly with cycle count. At `cycles = 5_000_000` the attacker pays ~4× less than the minimum intended; at higher cycle counts the multiplier increases further. This allows sustained exhaustion of the node's verification worker threads (default: 3/4 of CPU cores) and pool slot consumption at a fraction of the intended cost, constituting network congestion achievable with few costs.

## Likelihood Explanation
The attack is reachable by any unprivileged caller of the `send_transaction` RPC or any P2P peer relaying a transaction. No special privileges, keys, or majority hashpower are required. Constructing a transaction with a small serialized body but a loop-heavy lock or type script is straightforward on CKB's RISC-V VM. The attacker needs valid UTXOs to spend, but the on-chain fee cost is materially lower than the weight-based minimum would require. The attack is repeatable as long as the attacker controls UTXOs, and the severity scales with cycle count.

## Recommendation
After `verify_rtx` returns `verified.cycles` in `_process_tx`, perform a second fee rate check using the actual transaction weight before calling `submit_entry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
```

This check belongs between lines 734 and 751 of `tx-pool/src/process.rs`, after `verified` is obtained and before `TxEntry::new` is called. Alternatively, pass an optional `declared_cycles` hint into `check_tx_fee` on the relay path (where cycles are declared) so the weight-based gate can be applied even before full verification.

## Proof of Concept
1. Configure a CKB node with `tx_pool.min_fee_rate = 1000` (shannons/KW, the default).
2. Construct a transaction:
   - One input cell with capacity `C` shannons.
   - One output cell with capacity `C − 200` shannons (fee = 200 shannons).
   - A lock script containing a tight RISC-V loop consuming ~5,000,000 cycles.
   - Serialized transaction size ≈ 200 bytes.
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; the transaction passes.
5. `verify_rtx` executes the script, consuming 5,000,000 cycles; actual weight = `max(200, 5_000_000 × 0.000_170_571_4)` ≈ 852.
6. `TxEntry::new(rtx, 5_000_000, 200_shannons, 200)` is created; `entry.fee_rate()` = `200 × 1000 / 852 ≈ 234 shannons/KW` — well below the 1000 shannons/KW minimum.
7. The entry is admitted. Repeat with many UTXOs to exhaust verification threads and fill the pool with below-minimum fee rate entries.

Relevant code path: `send_transaction` → `_process_tx` → `pre_check` / `check_tx_fee` (size-only gate, L45 of `tx-pool/src/util.rs`) → `verify_rtx` (cycles revealed) → `TxEntry::new` (L751 of `tx-pool/src/process.rs`, no re-check) → `submit_entry` (no weight-based fee gate).         &self,
        pre_resolve_tip: Byte32,
        entry: TxEntry,
        mut status: TxStatus,
    ) -> (Result<(), Reject>, Arc<Snapshot>) {
        let (ret, snapshot) = self
            .with_tx_pool_write_lock(move |tx_pool, snapshot| {
                // check_rbf must be invoked in `write` lock to avoid concurrent issues.
                let conflicts = if tx_pool.enable_rbf() {
                    tx_pool.check_rbf(&snapshot, &entry)?
                } else {
                    // RBF is disabled but we found conflicts, return error here
                    // after_process will put this tx into conflicts_pool
                    let conflicted_outpoint =
                        tx_pool.pool_map.find_conflict_outpoint(entry.transaction());
                    if let Some(outpoint) = conflicted_outpoint {
                        return Err(Reject::Resolve(OutPointError::Dead(outpoint)));
                    }
                    HashSet::new()
                };

                // if snapshot changed by context switch we need redo time_relative verify
                let tip_hash = snapshot.tip_hash();
                if pre_resolve_tip != tip_hash {
                    debug!(
                        "submit_entry {} context changed. previous:{} now:{}",
                        entry.proposal_short_id(),
                        pre_resolve_tip,
                        tip_hash
                    );

                    // destructuring assignments are not currently supported
                    status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;

                    let tip_header = snapshot.tip_header();
                    let tx_env = status.with_env(tip_header);
                    time_relative_verify(snapshot, Arc::clone(&entry.rtx), tx_env)?;
                }

                let may_recovered_txs = self.process_rbf(tx_pool, &entry, &conflicts);
                let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?;

                // in a corner case, a tx with lower fee rate may be rejected immediately
                // after inserting into pool, return proper reject error here
                for evict in evicted {
                    let reject = Reject::Invalidated(format!(
                        "invalidated by tx {}",
                        evict.transaction().hash()
                    ));
                    self.callbacks.call_reject(tx_pool, &evict, reject);
                }

                tx_pool.remove_conflict(&entry.proposal_short_id());
                tx_pool
                    .limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
                    .map_or(Ok(()), Err)?;

                if !may_recovered_txs.is_empty() {
                    let self_clone = self.clone();
                    tokio::spawn(async move {
                        // push the recovered txs back to verify queue, so that they can be verified and submitted again
                        let mut queue = self_clone.verify_queue.write().await;
                        for tx in may_recovered_txs {
                            debug!("recover back: {:?}", tx.proposal_short_id());
                            let _ = queue.add_tx(tx, false, None);
                        }
                    });
                }
                Ok(())
            })
            .await;

        (ret, snapshot)
    }
