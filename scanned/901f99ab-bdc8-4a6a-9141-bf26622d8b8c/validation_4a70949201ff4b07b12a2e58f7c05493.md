Audit Report

## Title
Unbounded Wait Loop in `read_from_json` Causes Permanent CLI Hang on Crafted Import File — (`util/instrument/src/import.rs`)

## Summary

`Import::read_from_json` unconditionally updates `largest_block_number` for every non-genesis block received from the channel, then enters a boundless polling loop waiting for `snapshot.get_block_hash(largest_block_number)` to return `Some`. A crafted JSONL file containing a block whose `parent_hash` is unknown to the chain causes that block to be silently parked in the orphan pool with its callback never invoked. Because `COLUMN_INDEX` is never written for that height, `get_block_hash` returns `None` on every iteration and the `ckb import` process never exits. The only recovery is `SIGKILL`.

## Finding Description

**Root cause — unconditional `largest_block_number` update.** [1](#0-0) 
`largest_block_number` is set to the maximum block number seen on the channel regardless of whether the chain will ever accept that block.

**Fire-and-forget dispatch.** [2](#0-1) 
`asynchronous_process_lonely_block` returns immediately; the caller has no success/failure signal.

**Block lands in the orphan pool with no callback.**
In `chain_service.rs`, after optional non-contextual verification and DB insertion, the block is handed to `orphan_broker.process_lonely_block`. [3](#0-2) 
Inside `process_lonely_block`, when the parent hash is unknown (not stored, not pending-verify, not invalid), the block is silently inserted into the orphan pool and the `verify_callback` is **never called**. [4](#0-3) 

**Wait loop has no exit condition for failure.** [5](#0-4) 
`get_block_hash` is a plain `COLUMN_INDEX` lookup: [6](#0-5) 
No block at the crafted height is ever written to `COLUMN_INDEX`, so the lookup returns `None` on every 1-second iteration indefinitely.

**Orphan cleanup does not unblock the loop.**
The orphan TTL timer fires every 60 seconds: [7](#0-6) 
`clean_expired_orphans` removes the orphan from the pool but does not call its callback and does not write a block hash to `COLUMN_INDEX`, so the wait condition remains unsatisfied after eviction. [8](#0-7) 

## Impact Explanation

The `ckb import` CLI process hangs permanently, consuming a thread sleeping in a 1-second loop. This matches the in-scope impact: **Note (0–500 points) — Any local command line crash/hang**. The impact is bounded to the local import tool; it does not affect a running node, the network, or consensus.

## Likelihood Explanation

Triggering the hang requires only the ability to supply a crafted JSONL file to `ckb import`. With `--skip-all-verify`, non-contextual checks are bypassed entirely. Without it, a structurally well-formed block with a bogus `parent_hash` still passes `BlockVerifier` and `NonContextualBlockTxsVerifier` (neither checks parent existence), proceeds through `insert_block`, and enters the orphan pool. No cryptographic material, PoW solution, or network access is required. The file can be as small as two lines.

## Recommendation

Replace the unconditional `largest_block_number` update with tracking only blocks that are successfully committed to the chain, **or** add a bounded timeout to the wait loop, **or** plumb a shared atomic/channel that the `verify_callback` writes to on permanent failure (orphan eviction, `BLOCK_INVALID`) so the wait loop can break early. A minimal safe fix is to add a deadline:

```rust
let deadline = std::time::Instant::now() + std::time::Duration::from_secs(MAX_WAIT_SECS);
while self.shared.snapshot().get_block_hash(largest_block_number).is_none() {
    if std::time::Instant::now() > deadline {
        return Err("import timed out waiting for block commitment".into());
    }
    std::thread::sleep(std::time::Duration::from_secs(1));
}
```
