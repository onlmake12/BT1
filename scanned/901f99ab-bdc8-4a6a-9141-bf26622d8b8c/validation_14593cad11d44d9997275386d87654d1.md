Audit Report

## Title
Off-by-One in Group Eviction Threshold Allows Remote Peer to Permanently Fill Peer Store via Crafted Discovery `Nodes` Messages — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge()` uses a strict `> 4` guard when evicting peers from network groups. An attacker connected via the discovery protocol can send crafted `Nodes` messages containing addresses from 4096 distinct /16 subnets with exactly 4 addresses each, filling the 16384-slot peer store. Both eviction phases then find nothing to remove, returning `Err(EvictionFailed)`, which is silently swallowed. The peer store remains at capacity, blocking all new legitimate address insertion until entries age out (up to 7 days).

## Finding Description
**Phase 1** (`peer_store_impl.rs` lines 341–355): collects addresses where `is_connectable()` returns `false`. Freshly discovered addresses are created via `AddrInfo::new(addr, 0, score, flags)` with `last_connected_at_ms = 0` and `attempts_count = 0`. In `is_connectable()` (`types.rs` lines 95–96), the condition `self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES` evaluates to `0 >= 3 = false`. Phase 1 finds nothing.

**Phase 2** (`peer_store_impl.rs` lines 358–401): groups addresses by `Group`, which is keyed on the first two IPv4 octets (`network_group.rs` lines 26–29). With 4096 groups of exactly 4 addresses:
- `len = 4096`, `len / 2 = 2048`
- The top 2048 groups each have `addrs.len() == 4`
- The guard `if addrs.len() > 4` evaluates to `4 > 4 = false` for every group
- `candidate_peers` is empty → `Err(PeerStoreError::EvictionFailed)` is returned

`add_addr()` propagates this error. In `add_new_addrs()` (`discovery/mod.rs` lines 354–361), the error is caught and logged only at `debug!` level. The peer store remains at `ADDR_COUNT_LIMIT = 16384` and no new addresses can be inserted.

## Impact Explanation
The peer store is the sole mechanism by which a CKB node discovers new peers. With it permanently at capacity, the node cannot learn about new peers until existing entries age out (up to 7 days via `ADDR_TIMEOUT_MS`, or after `ADDR_MAX_RETRIES`/`ADDR_MAX_FAILURES` failed dial attempts). An attacker targeting multiple nodes simultaneously can degrade the network's peer discovery capability at low cost, matching **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — the network's ability to route peer information is impaired across targeted nodes with minimal attacker resources.

## Likelihood Explanation
The attack is reachable from any unprivileged peer connected via the discovery protocol. Non-announce `Nodes` messages allow up to 1000 items × 3 addresses = 3000 addresses per session; approximately 6 concurrent sessions suffice to fill 16384 slots. The attacker controls the IP addresses in the `Nodes` payload and can trivially craft addresses from 4096 distinct /16 subnets (there are ~65536 possible /16 subnets in IPv4). Addresses must pass `is_valid_addr()` (globally routable), which is easily satisfied. The attack is repeatable: after entries age out, the attacker can refill the store.

## Recommendation
Change the strict `> 4` threshold to `>= 4` (or `> 1`) in the group eviction condition at `peer_store_impl.rs` line 378:

```rust
// Before:
if addrs.len() > 4 {

// After:
if addrs.len() >= 4 {
```

Additionally, add a fallback eviction path that always frees at least one slot (e.g., evict the lowest-scored or oldest address) when both phases fail, enforcing the invariant that `check_purge()` always succeeds in making room when the store is at capacity.
