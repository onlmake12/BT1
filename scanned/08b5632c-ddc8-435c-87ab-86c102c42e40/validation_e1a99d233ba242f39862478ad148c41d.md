### Title
Eviction Protection Bypass via Manipulable Ping Fields Enables Preferential Honest-Peer Eviction — (`network/src/peer_registry.rs`)

---

### Summary

`PeerRegistry::try_evict_inbound_peer` protects peers from eviction using fields that the code comments claim are "hard to simulate or manipulate." In practice, both `last_ping_protocol_message_received_at` and `ping_rtt` are directly attacker-controllable via the P2P ping protocol. An attacker with `EVICTION_PROTECT_PEERS` (8) connections sending unsolicited pings can reliably saturate the "most recently active" protection tier. Combined with a second-tier `as_secs()` granularity bug that degrades the ping-RTT protection to a random shuffle, the algorithm systematically leaves honest peers as the only eviction candidates.

---

### Finding Description

**`EVICTION_PROTECT_PEERS = 8`** is the constant governing both protection tiers. [1](#0-0) 

The eviction function builds a candidate list of all non-whitelist inbound peers, then applies three sequential protection passes:

**Pass 1 — Lowest ping (8 peers protected):** [2](#0-1) 

The comparator calls `.as_secs()` on the RTT duration. Since virtually all internet peers have sub-second RTT, every peer resolves to `ping_rtt_secs = 0`, making all comparisons return `Equal`. Rust's stable sort preserves HashMap iteration order for equal elements — which is effectively random. The "lowest ping" protection degrades to a random selection among all sub-second peers.

**Pass 2 — Most recently active (8 peers protected):** [3](#0-2) 

Again uses `.as_secs()`. The field `last_ping_protocol_message_received_at` is set in two places in the ping protocol: [4](#0-3) [5](#0-4) 

An attacker can send **unsolicited `Ping` messages** at any time. Each received ping sets `last_ping_protocol_message_received_at = Instant::now()`. If the attacker sends a ping on each of their 8 connections immediately before eviction is triggered, those connections get `duration_since = 0 seconds`. Honest peers whose last ping exchange was at the regular interval (e.g., 15 s ago) get `duration_since = 15`. The attacker's 8 connections are sorted to the protected end and removed from the candidate list.

**Pass 3 — Longest connection time (half of remaining protected):** [6](#0-5) 

`connected_time` is set at `Peer::new()` and is not attacker-controllable. [7](#0-6) 

**Result of all three passes:** With 8 attacker connections reliably filling Pass 2 (and probabilistically filling Pass 1), the remaining candidate pool after both passes consists entirely of honest peers. Pass 3 then protects half of those honest peers by connection age, and the final eviction randomly selects from the other half — all honest peers.

---

### Impact Explanation

Each eviction round removes one honest peer and admits one new attacker connection. Repeated over `max_inbound` rounds (where `max_inbound = max_peers − max_outbound_peers`), the attacker can fill all inbound slots with attacker-controlled peers, achieving a full eclipse. An eclipsed node receives only attacker-curated blocks and transactions, enabling:
- Consensus deviation (feeding a minority chain)
- Double-spend facilitation
- Transaction censorship

---

### Likelihood Explanation

- **Attacker cost**: 8 persistent inbound connections sending periodic pings — trivially achievable from a single host with multiple IPs or from a botnet.
- **No privileged access required**: Pure P2P inbound path, no authentication.
- **Deterministic for Pass 2**: Sending a ping message is a single packet; the attacker controls the exact timestamp.
- **Probabilistic for Pass 1**: Due to `as_secs()` degradation, the attacker cannot guarantee filling the ping tier, but with 8 connections they have a statistically significant chance of occupying several of the 8 random slots.
- **Time to eclipse**: Proportional to `max_inbound`, but each round requires only one new connection.

---

### Recommendation

1. **Fix `as_secs()` → use sub-second granularity** (`as_millis()` or `as_nanos()`) in both ping comparators so the lowest-RTT protection is meaningful.
2. **Do not use attacker-sendable messages as protection criteria.** `last_ping_protocol_message_received_at` is updated on receipt of a peer-initiated `Ping`. Replace or supplement this with a field the node controls, such as the timestamp of the last **node-initiated** ping that received a valid pong (i.e., only update on `pong_received`, not `ping_received`).
3. **Add IP subnet diversity enforcement** to the protection tiers so that multiple connections from the same /16 cannot all occupy protected slots.
4. **Rate-limit unsolicited inbound `Ping` messages** to prevent timestamp freshening attacks.

---

### Proof of Concept

```
Setup:
  max_inbound = N (e.g., 117)
  EVICTION_PROTECT_PEERS = 8

Step 1: Attacker opens 8 inbound connections (A1..A8).
        Each connection sends a Ping message every second.
        → last_ping_protocol_message_received_at[Ai] = Instant::now() always

Step 2: Attacker opens 8 more inbound connections (A9..A16).
        Each responds to pings immediately.
        → ping_rtt[Ai] ≈ 0 ms (as_secs() = 0, same as all honest peers)

Step 3: Honest peers H1..H(N-16) fill remaining slots.
        Their last_ping_protocol_message_received_at is ~15 s old (ping interval).

Step 4: Attacker opens connection A17.
        accept_peer() detects non_whitelist_inbound >= max_inbound.
        try_evict_inbound_peer() runs:
          Pass 1: All sub-second peers tie at 0; random 8 protected (attacker may get some).
          Pass 2: A1..A8 have duration=0; honest peers have duration=15.
                  A1..A8 are protected and removed from candidates.
          Pass 3: Half of remaining honest peers protected by connection age.
          Final pool: honest peers only.
          → One honest peer Hx is evicted.

Step 5: Repeat Step 4 (N-16) times.
        → All honest peers evicted; all N slots held by attacker.
        → Node is eclipsed.

Invariant violated:
  The comment "characteristics that an attacker hard to simulate or manipulate"
  is false for last_ping_protocol_message_received_at.
``` [8](#0-7) [4](#0-3)
