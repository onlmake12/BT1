### Title
Unbounded `pending_delivered` HashMap Growth via Spoofed `from` Field in `ConnectionRequest` — (`network/src/protocols/hole_punching/`)

### Summary

An unprivileged remote peer can exhaust node memory by sending a stream of `ConnectionRequest` messages where the `from` field is set to a fresh random `PeerId` on each message and `to` is set to the target node's `local_peer_id` (publicly derivable via the Identify protocol). Each message inserts a new entry into `pending_delivered` and a new key into `forward_rate_limiter`, neither of which is bounded per-peer. The only throttle is the outer per-session rate limiter (30 req/sec), which still allows ~9,000 `pending_delivered` inserts per session per 5-minute cleanup cycle, and unlimited `forward_rate_limiter` growth for the lifetime of the connection.

---

### Finding Description

**Entry point — `HolePunching::received`**

Every inbound `HolePunchingMessage` passes through a single outer rate limiter keyed by `(session_id, msg.item_id())`: [1](#0-0) 

For `ConnectionRequest`, `item_id()` is always `0`, so this allows **30 req/sec per session** — a fixed, non-zero throughput.

**`forward_rate_limiter` bypass**

Inside `ConnectionRequestProcess::execute`, the second rate limiter is keyed by `(content.from, content.to, msg_item_id)`: [2](#0-1) 

Because `content.from` is **attacker-controlled** and set to a fresh random `PeerId` on every message, the key is always new. The rate limiter allows every message through (quota: 1 req/sec per key, and each key is brand new). This limiter provides **zero protection** against the attack.

**`pending_delivered` insert — no per-peer cap**

When `self_peer_id == content.to`, `respond_delivered` is called. The only deduplication guard checks whether the **same** `from_peer_id` was seen within 2 minutes: [3](#0-2) 

Since every message uses a unique `from`, this guard is never triggered. After a successful `send_message_to` back to the attacker's session, the entry is unconditionally inserted: [4](#0-3) 

**`pending_delivered` is an unbounded `HashMap<PeerId, (Vec<Multiaddr>, u64)>`:** [5](#0-4) 

**Cleanup is only periodic (every 5 minutes):** [6](#0-5) 

**`forward_rate_limiter` also grows unboundedly during a connection**

`retain_recent()` on the `forward_rate_limiter` is only called on `disconnected()`: [7](#0-6) 

During a long-lived connection, the `HashMapStateStore` inside `forward_rate_limiter` accumulates one entry per unique `(from, to, item_id)` tuple — i.e., one per message — and is never pruned. [8](#0-7) 

---

### Impact Explanation

**Per-session, per-5-minute-window:**
- 30 req/sec × 300 sec = **9,000 `pending_delivered` entries**
- Each entry: `PeerId` (32 B key) + `Vec<Multiaddr>` (up to 24 × ~50 B) + `u64` ≈ **~1,240 bytes**
- Per session per window: ~11 MB

**With default `max_peers = 125` (117 inbound slots):** [9](#0-8) 

- 117 sessions × 9,000 entries = **~1,053,000 entries ≈ 1.3 GB** in `pending_delivered` alone
- `forward_rate_limiter` grows at 30 entries/sec per session and is **never pruned** during the connection; over 1 hour with 117 sessions: ~12.6 M entries

This is sufficient to exhaust memory and crash the node.

---

### Likelihood Explanation

- The attacker needs only a standard P2P connection — no special privileges, no PoW, no keys.
- The target node's `local_peer_id` is publicly broadcast via the Identify protocol.
- The `from` field in `ConnectionRequest` is never validated against the actual session's peer ID.
- The attack is executable from a single IP with a single connection (slower) or amplified with multiple inbound connections.
- The `HolePunching` protocol is listed in the default `support_protocols` config: [10](#0-9) 

---

### Recommendation

1. **Bind `from` to the actual session peer ID**: In `received()`, reject any `ConnectionRequest` where `content.from` does not match the peer ID of the sending session. This eliminates the spoofing vector entirely.
2. **Cap `pending_delivered` size**: Enforce a hard maximum (e.g., 1,000 entries) and evict oldest entries when the cap is reached.
3. **Periodically prune `forward_rate_limiter`**: Call `retain_recent()` in the `notify()` handler (every 5 minutes), not only on `disconnected()`.
4. **Add a per-session insert budget**: Track how many entries each session has contributed to `pending_delivered` and reject further inserts once a per-session cap is reached.

---

### Proof of Concept

```
1. Connect to target node (standard P2P handshake).
2. Obtain target's local_peer_id via Identify protocol exchange.
3. In a loop at ≤30 msg/sec:
     send ConnectionRequest {
         from:        PeerId::random(),   // fresh random each iteration
         to:          target_local_peer_id,
         max_hops:    1,
         route:       [],
         listen_addrs: ["/ip4/1.2.3.4/tcp/8115"]
     }
4. After 5 minutes, assert pending_delivered.len() ≈ 9,000 (single session).
5. Open 117 simultaneous inbound connections and repeat;
   assert total pending_delivered.len() ≈ 1,053,000 and node RSS grows by ~1.3 GB.
```
