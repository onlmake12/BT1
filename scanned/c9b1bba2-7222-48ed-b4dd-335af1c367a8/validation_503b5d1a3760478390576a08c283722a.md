### Title
Unprivileged RPC Caller Can Trigger Simultaneous Full Table Scans via `instr`-Based Partial Search on Unindexed Script and Output Tables — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

---

### Summary

When the SQLite backend is used, calling `get_transactions` with `script_search_mode=Partial` and `output_data_filter_mode=Partial` generates SQL using `instr(args, $N) > 0` on the `script` table and `instr(output.data, $N) > 0` on the `output` table. Neither column has any index. The `script` table has **no index whatsoever**, making the script subquery a guaranteed full table scan. Both scans are issued simultaneously via a `UNION ALL` (once for outputs, once for inputs). The `request_limit` guard bounds only the returned row count, not the scan work performed. An unprivileged caller can repeatedly trigger O(total_script_rows + total_output_rows) work per request.

---

### Finding Description

**Entrypoint**: The public `get_transactions` RPC method, reachable by any caller without authentication.

**Path through code**:

1. `get_transactions` (line 56) dispatches to `get_tx_with_cell` for the ungrouped path.
2. `get_tx_with_cell` calls `build_tx_with_cell_union_sub_query` (line 153), which calls `build_query_script_id_sql` twice (once for outputs, once for inputs) and `build_filter` twice.
3. In `build_query_script_id_sql` (`mod.rs` lines 145–154), when `script_search_mode = Partial` and `db_driver = Sqlite`, the generated SQL is:

   ```sql
   SELECT script.id FROM script
   WHERE code_hash = $1 AND hash_type = $2
   AND instr(args, $3) > 0
   ```

4. In `build_filter` (`get_transactions.rs` lines 608–619), when `output_data_filter_mode = Partial` and `db_driver = Sqlite`, the generated SQL appends:

   ```sql
   AND instr(output.data, $N) > 0
   ```

**Index audit** — `create_sqlite_index.sql` defines all SQLite indexes:

```sql
-- output table
CREATE INDEX IF NOT EXISTS "idx_output_table_tx_id_output_index" ON "output" ("tx_id", "output_index");
CREATE INDEX IF NOT EXISTS "idx_output_table_lock_script_id" ON "output" ("lock_script_id");
CREATE INDEX IF NOT EXISTS "idx_output_table_type_script_id" ON "output" ("type_script_id");
-- input table
CREATE INDEX IF NOT EXISTS "idx_input_table_consumed_tx_id" ON "input" ("consumed_tx_id");
```

There is **no index on the `script` table at all** — not on `code_hash`, `hash_type`, or `args`. There is **no index on `output.data`**.

Consequently:
- The script subquery `WHERE code_hash = $1 AND hash_type = $2 AND instr(args, $3) > 0` performs a **full table scan** of the entire `script` table.
- The output filter `instr(output.data, $N) > 0` scans every output row that passes the join, with no index to prune.
- Both scans are emitted twice (output branch + input branch of the `UNION ALL`), so the total work is 2× per call.

The `request_limit` check (lines 27–32 of `get_transactions`) only rejects oversized `limit` values; it does not bound the number of rows scanned internally.

---

### Impact Explanation

An attacker sends repeated `get_transactions` RPC calls with:
- `script_search_mode: "partial"`, `args: "0x00"` (1-byte pattern matching nearly all scripts)
- `output_data_filter_mode: "partial"`, `output_data: "0x00"`
- `limit: 1` (minimizes response size, maximizes scan-to-result ratio)

Each call forces SQLite to scan the entire `script` table and all matching `output` rows. On a mainnet-scale node with millions of scripts and outputs, each call consumes significant CPU and I/O. Concurrent calls from a single IP (or multiple IPs) can saturate the indexer's database connection pool and degrade or stall all indexer RPC responses, including those from legitimate users.

Impact is scoped to the rich-indexer component's availability — **important CKB performance degradation (Low 501–2000)**.

---

### Likelihood Explanation

- The RPC is publicly accessible with no authentication.
- The `Partial` search mode is a documented, supported feature.
- No rate limiting is applied at the RPC or DB layer.
- The attack requires only a standard JSON-RPC call; no special knowledge or privileges are needed.
- The effect scales linearly with chain history (more blocks → more scripts/outputs → worse scans).

---

### Recommendation

1. **Add indexes on the `script` table**: At minimum, a composite index on `(code_hash, hash_type)` so the equality predicates can prune before `instr` is applied. A covering index `(code_hash, hash_type, args)` would further reduce scan cost.
2. **Add an index on `output.data`**: A prefix index or, for SQLite, a generated column with a hash of `data` for exact/prefix lookups.
3. **Rate-limit or disable `Partial` mode on SQLite**: Since SQLite cannot use indexes for `instr`-based search, either reject `Partial` mode with an error when `db_driver = Sqlite`, or enforce a per-IP/per-connection request rate limit.
4. **Bound scan complexity independently of `limit`**: Introduce a maximum scan row count (e.g., via `LIMIT` on the inner subquery) so that even with `Partial` mode, the DB work is capped.

### Proof of Concept

```python
import requests, time

RPC = "http://127.0.0.1:8116"

payload = {
    "id": 1, "jsonrpc": "2.0", "method": "get_transactions",
    "params": [{
        "script": {
            "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
            "hash_type": "type",
            "args": "0x00"
        },
        "script_type": "lock",
        "script_search_mode": "partial",
        "filter": {
            "output_data": "0x00",
            "output_data_filter_mode": "partial"
        }
    }, "asc", "0x1", None]
}

start = time.time()
for _ in range(10):
    requests.post(RPC, json=payload)
elapsed = time.time() - start
print(f"10 calls took {elapsed:.2f}s — assert proportional to total row count, not result size")
```

With 100k+ outputs in the DB, query latency will be proportional to total table size, not to the 1-row result, confirming the invariant violation.
