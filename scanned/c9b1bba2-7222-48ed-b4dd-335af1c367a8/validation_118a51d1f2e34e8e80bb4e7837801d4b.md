### Title
Integer Underflow in `VerifyQueue::is_full` Causes Incorrect Full-Queue Detection — (File: `tx-pool/src/component/verify_queue.rs`)

---

### Summary

The `VerifyQueue::is_full` method in the CKB tx-pool performs an unchecked subtraction `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size`. If `self.total_tx_size` ever exceeds `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (due to a state inconsistency acknowledged in the codebase itself), this subtraction wraps around to a very large `usize` value, causing `is_full` to return `false` when the queue is actually over capacity. This allows an unbounded number of transactions to be admitted into the verify queue, bypassing the intended 256 MB size limit.

---

### Finding Description

In `tx-pool/src/component/verify_queue.rs`, the `is_full` check is:

```rust
pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}
```

`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is `256_000_000` (a `usize`). If `self.total_tx_size` is greater than `256_000_000`, the subtraction `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size` **wraps around** (Rust `usize` subtraction in release mode is wrapping/panicking depending on build, but in release builds this is a wrapping underflow producing a huge value). The result is that `add_tx_size >= <huge_number>` evaluates to `false`, so `is_full` returns `false` — the queue appears empty when it is actually over-full.

The codebase itself acknowledges that `total_tx_size` can become inconsistent: the `remove_tx` method already handles the underflow case with an error log and recomputation:

```rust
if let Some(total_tx_size) = self.total_tx_size.checked_sub(tx_size) {
    self.total_tx_size = total_tx_size;
} else if let Some(total_tx_size) = self.recompute_total_tx_size() {
    error!("verify_queue total_tx_size {} underflowed ...", ...);
    self.total_tx_size = total_tx_size;
} else {
    error!("verify_queue total_tx_size {} underflowed ..., and recomputing overflowed", ...);
}
```

This means the developers are aware that `total_tx_size` can drift. When it drifts above `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`, the `is_full` guard silently fails.

The `add_tx` path also has a second `checked_add` guard after `is_full`, but that only catches integer overflow of `total_tx_size` itself, not the case where `total_tx_size > DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`.

The analogous pattern to the external report: the Solidity contracts attempted to withdraw more than the available balance without a cap check, causing a revert. Here, CKB's `is_full` attempts to compute "remaining capacity" via subtraction without checking whether the subtrahend exceeds the minuend, causing the guard to silently invert.

---

### Impact Explanation

An unprivileged tx-pool submitter (RPC caller via `send_transaction`, or a P2P relay peer) can submit transactions to a node whose `verify_queue.total_tx_size` has drifted above `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`. In that state, `is_full` always returns `false`, so every submitted transaction bypasses the 256 MB verify-queue size limit. The verify queue grows without bound, consuming unbounded memory on the target node, leading to resource exhaustion (OOM) or severe performance degradation. This is reachable by any unprivileged peer or RPC caller submitting transactions.

---

### Likelihood Explanation

The `total_tx_size` drift is acknowledged by the existing error-handling code in `remove_tx`. The drift can occur when cycles/size accounting becomes inconsistent (e.g., a tx is removed with a size value that doesn't match what was added, or concurrent state transitions). Once `total_tx_size` exceeds `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`, the `is_full` guard permanently fails until the queue is cleared. An attacker who can observe this state (or trigger the drift) can then flood the verify queue.

---

### Recommendation

Replace the unchecked subtraction in `is_full` with a saturating or checked variant:

```rust
pub fn is_full(&self, add_tx_size: usize) -> bool {
    let remaining = DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE.saturating_sub(self.total_tx_size);
    add_tx_size >= remaining
}
```

This ensures that when `total_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`, `remaining` is `0`, and any `add_tx_size >= 0` (i.e., any non-zero transaction) correctly triggers `is_full` returning `true`.

---

### Proof of Concept

1. Cause `verify_queue.total_tx_size` to exceed `256_000_000` (e.g., via the acknowledged drift path in `remove_tx`, or by directly manipulating state in a test as the existing test harness does with `set_total_tx_size_for_test`).
2. Call `is_full(any_tx_size)`:
   - `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size` wraps to a large `usize` (e.g., `usize::MAX - delta`).
   - `any_tx_size >= usize::MAX - delta` → `false`.
   - `is_full` returns `false`.
3. `add_tx` proceeds past the `is_full` guard and inserts the transaction, growing the queue beyond its intended limit.
4. Repeat from any unprivileged RPC or P2P entry point to exhaust node memory.

The existing test `process_orphan_tx_keeps_high_cycle_orphan_when_verify_queue_is_full` sets `total_tx_size` to `256_000_000 - 1` to test the boundary, confirming the guard is the sole enforcement mechanism. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
