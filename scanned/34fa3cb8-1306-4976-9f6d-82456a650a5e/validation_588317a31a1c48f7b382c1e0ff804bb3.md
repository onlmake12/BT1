### Title
NervosDAO Accumulation Rate (`ar`) Floor-Division Truncation Causes Systematic Underpayment of Interest to Depositors — (File: `util/dao/src/lib.rs`)

---

### Summary
Every block, the NervosDAO accumulation rate (`ar`) is updated using integer (floor) division. The fractional remainder is silently discarded, causing `ar` to grow slightly slower than the mathematically exact value. Because depositor interest is proportional to `ar_withdraw / ar_deposit`, every NervosDAO depositor receives slightly less interest than they are owed. A second, independent floor-division truncation occurs at withdrawal time in `calculate_maximum_withdraw`, compounding the underpayment.

---

### Finding Description

**Root cause 1 — `ar_increase` truncation (per-block, cumulative)**

In `dao_field_with_current_epoch`, the accumulation-rate increment is:

```rust
// util/dao/src/lib.rs  lines 256-261
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
let current_ar = parent_ar
    .checked_add(ar_increase)
    .ok_or(DaoError::Overflow)?;
```

Integer division discards the fractional part. The exact value of `ar_increase` is `parent_ar × g2 / C`; the stored value is `⌊parent_ar × g2 / C⌋`. The per-block error is in `[0, 1)`. Over `N` blocks the accumulated deficit in `ar` is at most `N`.

**Root cause 2 — `calculate_maximum_withdraw` truncation (per-withdrawal)**

When a depositor withdraws, the entitled capacity is:

```rust
// util/dao/src/lib.rs  lines 152-154
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
```

Again, floor division discards the fractional part, shaving up to 1 shannon from the withdrawal amount independently of the `ar` deficit above.

**Combined effect**

The interest a depositor receives is:

```
interest = counted_capacity × (withdrawing_ar − deposit_ar) / deposit_ar
```

Because `withdrawing_ar` is systematically lower than its exact value (root cause 1) and the final division is also floored (root cause 2), the depositor always receives ≤ the mathematically correct interest.

---

### Impact Explanation

Every NervosDAO depositor is underpaid. The magnitude scales with deposit size and duration:

- `ar_0 = 10^16` (genesis)
- Mainnet `ar_increase` ≈ `10^8` per block (observed: block 1 delta = `104,789,669`)
- Per-block truncation error ∈ `[0, 1)` → accumulated deficit after `N` blocks ≤ `N`
- For a 1-year deposit (~2,190,000 blocks at ~14.4 s/block):
  - Accumulated `ar` deficit ≤ 2,190,000
  - For a 1,000 CKB deposit (10^11 shannons): interest shortfall ≤ `10^11 × 2,190,000 / 10^16 ≈ 22 shannons`
  - For a 1,000,000 CKB deposit (10^14 shannons): shortfall ≤ ~22,000 shannons (~0.00022 CKB)

The per-depositor loss is small in absolute terms but is systematic, deterministic, and affects every withdrawal. The protocol consistently favors itself over depositors by the truncated remainder.

---

### Likelihood Explanation

This is triggered unconditionally on every block and every NervosDAO withdrawal. No special conditions, privileges, or attacker actions are required. Any unprivileged user who deposits into NervosDAO and later withdraws will experience the underpayment. The entry path is:

1. User submits a NervosDAO deposit transaction (standard RPC/tx-pool path).
2. Each subsequent block, `dao_field_with_current_epoch` is called by the chain service, truncating `ar_increase`.
3. User submits a withdrawal transaction referencing the deposit and prepare headers.
4. `calculate_maximum_withdraw` is called during verification, applying a second truncation.
5. The user receives less than the mathematically correct interest.

---

### Recommendation

Use ceiling division for `ar_increase` to ensure depositors receive at least their entitled interest:

```rust
// Replace floor division with ceiling division for ar_increase
let ar_increase128 =
    (u128::from(parent_ar) * u128::from(current_g2.as_u64()) + u128::from(parent_c.as_u64()) - 1)
    / u128::from(parent_c.as_u64());
```

Similarly, apply ceiling division in `calculate_maximum_withdraw` for `withdraw_counted_capacity`. This mirrors the recommendation in the referenced report: use the finer-grained unit (per-block rather than per-epoch) and round in the depositor's favor to avoid systematic underpayment.

### Proof of Concept

Using mainnet genesis parameters (`ar_0 = 10^16`, `C_0 ≈ 3.36 × 10^18 shannons`, `g2_1 ≈ 1.45 × 10^11 shannons`):

```

                    ≈ 104,789,669.xxx

Stored ar_increase_1 = 104,789,669   (fractional .xxx discarded)
```

Confirmed by the on-chain value: `ar_1 = 10,000,000,104,789,669` (block 1 mainnet DAO field).

After `N = 2,190,000` blocks (≈ 1 year), the accumulated `ar` deficit is ≤ 2,190,000. For a depositor with `counted_capacity = 10^14` shannons (1,000,000 CKB):

```
Interest shortfall from ar truncation ≤ 10^14 × 2,190,000 / 10^16 = 21,900 shannons
Interest shortfall from withdrawal truncation ≤ 1 shannon
Total shortfall ≤ 21,901 shannons  (~0.000219 CKB)
```
