### Title
Integer Overflow in `GetLastStateProofProcess::execute` Bypasses Limit Check, Enabling O(chain_length) DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

A single unauthenticated P2P `GetLastStateProof` message with `last_n_blocks = 2^63` causes a `usize` multiplication overflow that silently zeroes the guard expression, bypassing the `GET_LAST_STATE_PROOF_LIMIT` check. The subsequent "not enough blocks" branch then collects every block number from genesis to tip into a `Vec`, and `complete_headers` performs one MMR root computation and two DB lookups per block — O(chain_length) work per request.

---

### Finding Description

**Step 1 — Overflow bypass of the limit guard** [1](#0-0) 

`last_n_blocks` is a `u64` read directly from the peer message. On a 64-bit target, `usize` is also 64 bits. With `last_n_blocks = 2^63`:

```
(2^63_u64 as usize) * 2  →  2^64 mod 2^64  =  0   (release-mode wrapping)
```

`0 + 0 > 1000` is `false`, so the guard returns without rejecting the message. Rust's release profile does **not** enable `overflow-checks` by default, and no `checked_mul` / `saturating_mul` is used here. [2](#0-1) 

**Step 2 — "Not enough blocks" branch allocates O(N) block numbers** [3](#0-2) 

With `start_block_number = 0` and `last_n_blocks = 2^63`, the condition `N - 0 <= 2^63` is true for any realistic chain length N. The branch collects `(0..N)` into a `Vec<BlockNumber>` — N 8-byte allocations.

**Step 3 — `complete_headers` performs O(N) expensive work** [4](#0-3) 

For each of the N block numbers, `complete_headers` calls:
- `snapshot.get_ancestor(last_hash, number)` — walks the chain
- `snapshot.get_block(hash)` — DB read
- `snapshot.chain_root_mmr(number - 1).get_root()` — MMR root computation

All three are non-trivial per-block operations. A chain of 10 million blocks means 10 million iterations per request.

**Step 4 — `reorg_last_n_numbers` is empty (no extra amplification)** [5](#0-4) 

With `start_block_number = 0`, the first branch is taken and `reorg_last_n_numbers = Vec::new()`, so there is no additional amplification from that path.

---

### Impact Explanation

A single malformed P2P message forces the light-client server to iterate every block in the chain. With a sufficiently long chain, this causes:
- **OOM**: the `Vec<BlockNumber>` and `Vec<VerifiableHeader>` both grow to O(N)
- **CPU exhaustion**: N MMR root computations and N DB reads block the async executor
- **Node crash or unresponsiveness**: other peers cannot be served while the handler runs

The attacker needs no credentials, no PoW, and no stake — only a valid tip hash (publicly observable) and a TCP connection to the light-client P2P port.

---

### Likelihood Explanation

The exploit is mechanically straightforward: craft one molecule-encoded `GetLastStateProof` message with `last_n_blocks = 0x8000000000000000`. The tip hash is observable from any synced peer. No fuzzing is required; the overflow boundary is exact and deterministic.

---

### Recommendation

Replace the unchecked multiplication with a saturating or checked variant:

```rust
// Before (vulnerable):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
let n_samples = (last_n_blocks as usize).saturating_mul(2);
if self.message.difficulties().len().saturating_add(n_samples)
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

Additionally, add a hard cap on `last_n_blocks` itself before any arithmetic:

```rust
if last_n_blocks > constant::GET_LAST_STATE_PROOF_LIMIT as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}
```

### Proof of Concept

```
last_n_blocks       = 0x8000000000000000  (2^63)
start_number        = 0
start_hash          = genesis_hash        (always known)
last_hash           = current tip hash    (observable from any peer)
difficulty_boundary = 1                   (any U256 >= 1)
difficulties        = []                  (empty list)
```

1. Guard: `[] .len() + (2^63 as usize) * 2 = 0 + 0 = 0 > 1000` → **false** → passes
2. `last_hash` is on main chain → proceeds
3. `start_block_number (0) <= last_block_number (N)` → proceeds
4. `reorg_last_n_numbers = []` (start == 0)
5. `N <= 2^63` → "not enough blocks" → `last_n_numbers = (0..N).collect()` → **N entries**
6. `complete_headers` loops N times → **O(N) MMR + DB work**
