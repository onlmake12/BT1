### Title
Off-by-One in `filter_broadcast` Gossip Counter Broadcasts to `isqrt(N)−1` Peers Instead of `isqrt(N)` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary
The `filter_broadcast` closure in `ConnectionRequestProcess::forward_message` decrements `total` **before** checking `total != 0`, causing the relay to forward a `ConnectionRequest` to `isqrt(N)−1` peers instead of the intended `isqrt(N)`. When the relay has 1–3 connected peers (`isqrt(N) == 1`), the broadcast reaches **zero** peers, silently dropping the hole-punching request.

### Finding Description

In `forward_message`, when the target peer is not directly connected, the relay gossips the request to `isqrt(peers.len())` random peers: [1](#0-0) 

The closure logic is:
```
total = isqrt(N)
for each candidate peer (excluding sender):
    total -= 1          ← decrement FIRST
    return total != 0   ← then check
```

Trace for `total = 1` (N = 1, 2, or 3):
- Peer 1: `total` becomes `0`, returns `0 != 0` → **false** → not sent

Trace for `total = 2` (N = 4–8):
- Peer 1: `total` becomes `1`, returns `true` → sent
- Peer 2: `total` becomes `0`, returns `false` → not sent → only 1 peer receives it instead of 2

The invariant "broadcast to `isqrt(N)` peers" is violated in every case; the actual count is `isqrt(N) − 1`. The identical bug exists in the initiator path in `notify`: [2](#0-1) 

### Impact Explanation

An unprivileged remote peer sends a well-formed `ConnectionRequest` with `to` set to any unknown peer ID. The relay enters the `None` branch of `forward_message` and calls `filter_broadcast` with the broken closure. With 1–3 connected peers the message is dropped entirely; with 4–8 peers only half the intended relays receive it. Hole-punching requests silently fail to propagate, preventing NAT traversal from completing. This degrades network connectivity for nodes behind NAT.

The claimed **consensus deviation** impact is not directly reachable from this bug alone. Hole-punching is an optimization path for NAT traversal; nodes can still connect through other relays or direct connections. The bug causes degraded connectivity and hole-punching failure, not consensus deviation.

### Likelihood Explanation

Any peer connected to a relay with 1–3 peers can trigger this with a single valid `ConnectionRequest` message. No privilege, key, or special state is required. The rate limiter (`forward_rate_limiter`) limits per `(from, to, item_id)` tuple but does not prevent the bug from firing on the first attempt.

### Recommendation

Check `total != 0` **before** decrementing, or initialize `total` to `isqrt(N) + 1`:

```rust
// Fix: check before decrement
TargetSession::Filter(Box::new(move |id| {
    if id == &sid {
        return false;
    }
    if total == 0 {
        return false;
    }
    total = total.saturating_sub(1);
    true
}))
```

Apply the same fix to the identical pattern in `notify` in `mod.rs`.

### Proof of Concept

```
peers.len() = 2  →  isqrt(2) = 1  →  total starts at 1
filter_broadcast iterates peer A (not sid):
  total = 1.saturating_sub(1) = 0
  return 0 != 0  →  false  →  peer A excluded
Result: 0 peers receive the broadcast. Hole-punching silently fails.
```

The finding is valid at a network-connectivity/availability severity, not at the "easily causes consensus deviation" Critical tier claimed in the question.
