### Title
Ambiguous Variable-Length Key Encoding in Indexer Enables Cross-Script Result Pollution — (`util/indexer/src/indexer.rs`)

---

### Summary

The `extract_raw_data` function concatenates the variable-length `args` field of a `Script` directly with fixed-length positional fields (`block_number`, `tx_index`, `io_index`) in the RocksDB key, without any length delimiter. This is the CKB analog of `abi.encodePacked` with multiple dynamic types: the resulting byte string is ambiguous, and a key for script A at position P can be byte-identical to a prefix of a key for script B at a different position. In the default prefix-scan mode used by `get_cells`, `get_transactions`, and `get_cells_capacity`, an unprivileged transaction sender can craft a script with specific `args` such that their cell, once included in a block at a predictable position, appears in indexer query results for a completely different target script.

---

### Finding Description

`extract_raw_data` in `util/indexer/src/indexer.rs` produces:

```
code_hash (32 bytes, fixed) || hash_type (1 byte, fixed) || args (variable, no length prefix)
```

`append_key` then appends fixed-size positional fields:

```
extract_raw_data(script) || block_number (8 BE) || tx_index (4 BE) || io_index (4 BE)
```

The full stored key for `CellLockScript` / `CellTypeScript` / `TxLockScript` / `TxTypeScript` is therefore:

```
[prefix_byte(1)] || code_hash(32) || hash_type(1) || args(N) || block_number(8) || tx_index(4) || io_index(4)
```

Because `args` has no length tag, the byte boundary between `args` and `block_number` is invisible in the raw key. The `build_query_options` function constructs a RocksDB prefix scan key as:

```
[prefix_byte] || code_hash || hash_type || args_of_search_script
```

and iterates all stored keys that `starts_with` this prefix. A stored key for **script A** (shorter `args`) at a specific position will `starts_with` the prefix for **script B** (longer `args`) whenever:

```
args_B == args_A || first_(len(args_B) - len(args_A))_bytes_of_(block_number_A || tx_index_A || io_index_A)
``` [1](#0-0) [2](#0-1) 

The `exact` mode applies a key-length guard that prevents this confusion, but **prefix mode is the default** and has no such guard: [3](#0-2) 

---

### Impact Explanation

An attacker who controls a cell's script `args` and can predict or influence the block/tx/output position of their cell can inject that cell into the indexer results for an arbitrary target script. Concretely:

- **`get_cells`** returns the attacker's cell as a live cell belonging to the target script.
- **`get_transactions`** returns the attacker's transaction as associated with the target script.
- **`get_cells_capacity`** inflates or deflates the reported capacity for the target script.

Wallets, dApps, and bridges that rely on the indexer RPC to enumerate UTXOs or verify balances receive incorrect data. This can cause incorrect balance displays, missed or phantom UTXOs, and — in automated systems — incorrect asset-transfer decisions. [4](#0-3) 

---

### Likelihood Explanation

The attacker controls:
- **`args`** entirely (any byte string, set at transaction construction time).
- **`output_index`** entirely (by adding outputs to their transaction).
- **`tx_index`** partially (by submitting multiple transactions to influence ordering within a block).
- **`block_number`** indirectly (by waiting for the right block height, or targeting early blocks that already exist).

**Concrete example:** Target script has `args = [0xAA, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]` (17 bytes). Attacker creates a script with `args = [0xAA]` (1 byte) and gets their cell included at `block_number=0, tx_index=0, output_index=0` (genesis block, always present). The stored key for the attacker's cell is:

```
[prefix] || code_hash || hash_type || [0xAA] || [0x00×8] || [0x00×4] || [0x00×4]
```

The search prefix for the target script is:

```
[prefix] || code_hash || hash_type || [0xAA, 0x00×16]
```

The stored key starts with the search prefix — the attacker's cell is returned. No privileged access, no majority hashpower, and no social engineering is required. [5](#0-4) 

---

### Recommendation

Include the length of `args` as a fixed-width prefix (e.g., 4-byte little-endian `u32`) in `extract_raw_data`, so that the boundary between `args` and the positional fields is unambiguous:

```rust
pub fn extract_raw_data(script: &Script) -> Vec<u8> {
    let args = script.args().raw_data();
    let args_len = (args.len() as u32).to_le_bytes();
    [
        script.code_hash().as_slice(),
        script.hash_type().as_slice(),
        &args_len,
        &args,
    ]
    .concat()
}
```

This change requires a full re-index of the indexer database (existing stored keys must be migrated or the index rebuilt), but it eliminates the ambiguity entirely. Alternatively, the `exact` mode length guard already present in `get_cells` should be applied unconditionally to all scan modes, though this is a weaker mitigation that does not fix the underlying key-space collision. [6](#0-5) 

---

### Proof of Concept

1. Deploy a cell on a CKB node (testnet or devnet) with lock script:
   - `code_hash = X`, `hash_type = Y`, `args = [0xAA]`
   - Ensure the cell is in block `N`, `tx_index = T`, `output_index = I`.

2. Compute the 16-byte suffix: `N.to_be_bytes() || T.to_be_bytes() || I.to_be_bytes()`.

3. Call `get_cells` via RPC with:
   ```json
   {
     "script": {
       "code_hash": "X",
       "hash_type": "Y",
       "args": "0xAA<N_BE_8bytes><T_BE_4bytes><I_BE_4bytes>"
     },
     "script_type": "lock",
     "script_search_mode": "prefix"
   }
   ```

4. Observe that the attacker's cell (which has `args = [0xAA]`, not the 17-byte args in the query) is returned in the result set, demonstrating cross-script result pollution. [3](#0-2) [2](#0-1)

const MAX_PREFIX_SEARCH_SIZE: usize = u16::MAX as usize;

// a helper fn to build query options from search parameters, returns prefix, from_key, direction and skip offset
fn build_query_options(
    search_key: &IndexerSearchKey,
    lock_prefix: KeyPrefix,
    type_prefix: KeyPrefix,
    order: IndexerOrder,
    after_cursor: Option<JsonBytes>,
) -> Result<(Vec<u8>, Vec<u8>, Direction, usize), Error> {
    let mut prefix = match search_key.script_type {
        IndexerScriptType::Lock => vec![lock_prefix as u8],
        IndexerScriptType::Type => vec![type_prefix as u8],
    };
    let script: packed::Script = search_key.script.clone().into();
    let args_len = script.args().len();
    if args_len > MAX_PREFIX_SEARCH_SIZE {
        return Err(Error::invalid_params(format!(
            "search_key.script.args len should be less than {MAX_PREFIX_SEARCH_SIZE}"
        )));
    }
    prefix.extend_from_slice(extract_raw_data(&script).as_slice());

    let (from_key, direction, skip) = match order {
        IndexerOrder::Asc => after_cursor.map_or_else(
            || (prefix.clone(), Direction::Forward, 0),
            |json_bytes| (json_bytes.as_bytes().into(), Direction::Forward, 1),
        ),
        IndexerOrder::Desc => after_cursor.map_or_else(
            || {
                (
                    [
                        prefix.clone(),
                        vec![0xff; MAX_PREFIX_SEARCH_SIZE - args_len],
                    ]
                    .concat(),
                    Direction::Reverse,
                    0,
                )
            },
            |json_bytes| (json_bytes.as_bytes().into(), Direction::Reverse, 1),
        ),
    };

    Ok((prefix, from_key, direction, skip))
