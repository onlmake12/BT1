### Title
Stale `ancestors_count` in `check_and_record_ancestors` Eviction Loop Causes Over-Eviction of Pool Transactions - (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `tx-pool/src/component/pool_map.rs`, the function `check_and_record_ancestors` maintains a local `ancestors_count` variable to decide how many `cell_ref_parent` transactions to evict. Each eviction calls `remove_entry_and_descendants`, which removes the target entry **plus all its descendants**. However, `ancestors_count` is only decremented by `1` per iteration regardless of how many of those descendants were also members of the `ancestors` set. This mirrors the Suzaku bug class exactly: a counter used to gate iterative removal decisions does not account for items already removed in the current pass, causing the loop to over-evict.

---

### Finding Description

`check_and_record_ancestors` is called from `add_entry` (line 213) every time a transaction is inserted into the pool. When the new transaction's ancestor count exceeds `max_ancestors_count` but the excess is entirely attributable to `cell_ref_parents` (transactions that hold a cell dep on an output the new tx is spending as an input), the function enters an eviction loop:

```
// tx-pool/src/component/pool_map.rs  lines 615-625

    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        ancestors_count = ancestors_count.saturating_sub(1);   // ŌåÉ always -1
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
```

`remove_entry_and_descendants` (lines 252-265) removes the named entry **and every descendant** reachable from it:


If any of those descendants are also members of the `ancestors` set of the incoming transaction (reachable because the new tx depends on them via a separate input or cell-dep path), the actual ancestor count drops by `1 + k` (where `k Ōēź 1` is the number of such descendants), but `ancestors_count` only drops by `1`. After the eviction, `ancestors_count` is `k` higher than the true remaining ancestor count. The loop therefore continues and evicts the next `cell_ref_parent` candidate even though the actual ancestor count is already at or below `max_ancestors_count`.

The post-loop assertion at line 636 (`assert!(ancestors.len() < self.max_ancestors_count)`) recalculates ancestors from scratch and will pass because the actual count is already below the limit ŌĆö but by then, extra transactions have already been permanently removed from the pool.

---

### Impact Explanation

Every unnecessary eviction permanently removes a valid, fee-paying transaction from the mempool. The eviction list is sorted by lowest fee-rate first (`iter_by_evict_key`), so the cheapest `cell_ref_parents` are removed first; however, if those cheap entries have descendants that are also ancestors of the new tx, the stale counter causes additional (potentially higher-fee) `cell_ref_parents` to be evicted as well. Affected transactions are rejected with `Reject::Full` or similar and must be resubmitted. In a busy mempool this constitutes a griefing / mempool-DoS primitive: a single crafted transaction can silently purge more pool entries than the protocol intends.

---

### Likelihood Explanation

The trigger condition ŌĆö a `cell_ref_parent` whose descendants are also ancestors of the incoming transaction ŌĆö is reachable without any privileged access. A transaction sender (RPC `send_transaction` or P2P relay) can construct this graph deliberately:

1. Submit `tx_A` (root).
2. Submit `tx_B` spending `tx_A`'s output (descendant of `tx_A`).
3. Submit `tx_C` that (a) spends `tx_B`'s output **and** (b) spends a separate output `O` that `tx_A` holds as a cell dep.

When `tx_C` is submitted, `tx_A` is a `cell_ref_parent` (it holds a cell dep on `O`), and `tx_B` is both a descendant of `tx_A` and an ancestor of `tx_C`. Evicting `tx_A` via `remove_entry_and_descendants` removes both `tx_A` and `tx_B`, but `ancestors_count` only decrements by 1. The loop then evicts the next `cell_ref_parent` unnecessarily.

---

### Recommendation

Replace the fixed `-1` decrement with the actual number of ancestors removed in each iteration:

```diff
 while ancestors_count > self.max_ancestors_count {
     if let Some(next_id) = iter.next() {
         let removed = self.remove_entry_and_descendants(next_id);
-        ancestors_count = ancestors_count.saturating_sub(1);
+        // Count how many of the removed entries were in the ancestors set
+        let removed_ancestor_count = removed
+            .iter()
+            .filter(|e| ancestors.contains(&e.proposal_short_id()))
+            .count();
+        ancestors_count = ancestors_count.saturating_sub(removed_ancestor_count.max(1));
         parents.remove(next_id);
         evicted.extend(removed);
     } else {
         break;
     }
 }
```

Alternatively, recompute `ancestors_count` from the live pool state after each eviction (analogous to the Suzaku fix of passing a signed offset), or break early once the recalculated ancestor set already satisfies the limit.

### Proof of Concept

**Setup (pool state before submitting the trigger tx):**

| Tx | Role | Relationship |
|----|------|--------------|
| `tx_root` | root | no pool parents |
| `tx_dep_holder` | cell_ref_parent | has cell dep on output `O`; spends `tx_root` output |
| `tx_mid` | descendant of `tx_dep_holder` AND ancestor of trigger | spends `tx_dep_holder` output |
| *(repeat `tx_dep_holder_i` / `tx_mid_i` pairs to exceed `max_ancestors_count`)* | | |

**Trigger transaction:** spends output `O` as an input (making all `tx_dep_holder_i` become `cell_ref_parents`) and spends `tx_mid_last`'s output (making `tx_mid_last` and its ancestors part of `ancestors`).

**Expected behaviour:** evict only enough `cell_ref_parents` to bring `ancestors_count` to `max_ancestors_count`.

**Actual behaviour:** because each eviction of `tx_dep_holder_i` also removes `tx_mid_i` (a member of `ancestors`) but `ancestors_count` only decrements by 1, the loop continues and evicts the next `tx_dep_holder_{i+1}` unnecessarily. All `cell_ref_parent` candidates are evicted even though the limit was satisfied after the first one or two evictions.

```rust
    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }
```
