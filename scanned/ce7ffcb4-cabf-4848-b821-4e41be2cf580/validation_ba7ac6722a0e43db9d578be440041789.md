### Title
RBF `replace_count` Double-Counts Shared Descendants, Enabling False Rejection of Legitimate RBF Transactions - (File: `tx-pool/src/pool.rs`)

---

### Summary

In `check_rbf`, the `replace_count` accumulator is incremented by `descendants.len() + 1` for each conflicting transaction in a loop, without deduplicating descendants that are shared across multiple conflicts. When two or more conflicting transactions share a common descendant (a diamond-shaped dependency graph), that descendant is counted multiple times, inflating `replace_count` beyond the true unique replacement count. This causes the `MAX_REPLACEMENT_CANDIDATES` guard to trigger prematurely, rejecting a legitimate RBF transaction that would otherwise satisfy the limit.

---

### Finding Description

`check_rbf` in `tx-pool/src/pool.rs` enforces Rule #5: the total number of transactions that would be replaced must not exceed `MAX_REPLACEMENT_CANDIDATES` (100). [1](#0-0) 

```rust
let mut replace_count: usize = 0;
let mut all_conflicted = conflicts.clone();
let ancestors = self.pool_map.calc_ancestors(&short_id);
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    replace_count += descendants.len() + 1;          // ← no deduplication
    if replace_count > MAX_REPLACEMENT_CANDIDATES {
        return Err(Reject::RBFRejected(format!(
            "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
            replace_count, MAX_REPLACEMENT_CANDIDATES,
        )));
    }
    ...
    all_conflicted.extend(entries);                   // ← duplicates accumulate
}
``` [2](#0-1) 

`calc_descendants` returns a `HashSet<ProposalShortId>` for a single conflict root, but the loop accumulates counts across all conflicts without a global deduplication set. If conflict `A` has descendants `{C, D}` and conflict `B` has descendants `{C, E}`, then:

- Actual unique replacements: `A, B, C, D, E` = 5
- Computed `replace_count`: `(2+1) + (2+1)` = 6 — `C` is counted twice

A diamond-shaped dependency graph in the pool (transaction `C` spending outputs from both `A` and `B`) is a realistic and valid pool state. The `Edges` map tracks inputs per outpoint, and a transaction with two inputs from two different pool transactions is fully supported. [3](#0-2) 

By contrast, `calculate_min_replace_fee` already handles this correctly by deduplicating via a `HashMap` keyed on `id`: [4](#0-3) 

The `replace_count` path has no equivalent guard.

---

### Impact Explanation

An unprivileged tx-pool submitter can pre-populate the pool with a diamond-shaped dependency graph (two transactions `A` and `B` that each conflict with a future RBF transaction, plus a child transaction `C` that spends outputs from both `A` and `B`). When a legitimate RBF transaction is later submitted that conflicts with both `A` and `B`, `replace_count` is inflated to `(descendants_of_A.len() + 1) + (descendants_of_B.len() + 1)`, double-counting `C`. If this inflated value exceeds 100, the RBF is rejected with `RBFRejected("Tx conflict with too many txs")` even though the true unique replacement count is within the limit.

This prevents legitimate RBF fee-bumping, which is the primary mechanism for unsticking stuck transactions. It also means `all_conflicted` accumulates duplicate entries, though `calculate_min_replace_fee` already deduplicates those via `HashMap`, so the fee threshold itself is computed correctly.

---

### Likelihood Explanation

The attack requires only that the pool contain a transaction with two inputs from two different conflicting transactions — a standard diamond dependency. This is a normal pool topology that arises organically (e.g., a consolidation transaction spending outputs from two pending parents). No privileged access, no majority hashpower, and no external dependency is required. The attacker only needs to submit transactions via the standard RPC.

---

### Recommendation

Accumulate descendants into a single global `HashSet` across all conflicts before computing `replace_count`, mirroring the deduplication already present in `calculate_min_replace_fee`:

```rust
let mut all_descendants: HashSet<ProposalShortId> = HashSet::new();
for conflict in conflicts.iter() {
    all_descendants.extend(self.pool_map.calc_descendants(&conflict.id));
}
// replace_count = unique descendants + number of direct conflicts
let replace_count = all_descendants.len() + conflicts.len();
if replace_count > MAX_REPLACEMENT_CANDIDATES {
    return Err(Reject::RBFRejected(...));
}
```

### Proof of Concept

1. Submit transaction `A` (spends cell `X`, has output `out_A`).
2. Submit transaction `B` (spends cell `Y`, has output `out_B`).
3. Submit transaction `C` (spends `out_A` and `out_B`). `C` is now a descendant of both `A` and `B`.
4. Repeat step 3 with 49 more child transactions of `A` and `B` (each spending one output from `A` and one from `B`), giving 50 shared descendants.
5. Submit RBF transaction `R` that spends both `X` and `Y` (conflicting with both `A` and `B`), with a higher fee.
6. In `check_rbf`, `conflicts = [A, B]`. For `A`: `descendants = {C, …}` (50 entries), `replace_count += 51`. For `B`: `descendants = {C, …}` (same 50 entries), `replace_count += 51`. Total `replace_count = 102 > 100`.
7. `R` is rejected with `"Tx conflict with too many txs, conflict txs count: 102, expect <= 100"` even though only 52 unique transactions would actually be replaced. [5](#0-4) [6](#0-5)

    pub(crate) fn check_rbf(
        &self,
        snapshot: &Snapshot,
        entry: &TxEntry,
    ) -> Result<HashSet<ProposalShortId>, Reject> {
        assert!(self.enable_rbf());
        let tx_inputs: Vec<OutPoint> = entry.transaction().input_pts_iter().collect();
        let conflict_ids = self.pool_map.find_conflict_tx(entry.transaction());

        if conflict_ids.is_empty() {
            return Ok(HashSet::new());
        }

        let short_id = entry.proposal_short_id();

        // Rule #1, the node has enabled RBF, which is checked by caller
        let conflicts = conflict_ids
            .iter()
            .filter_map(|id| self.get_pool_entry(id))
            .collect::<Vec<_>>();
        assert!(conflicts.len() == conflict_ids.len());

        // Rule #2, new tx don't contain any new unconfirmed inputs
        let mut inputs = HashSet::new();
        for c in conflicts.iter() {
            inputs.extend(c.inner.transaction().input_pts_iter());
        }

        if tx_inputs
            .iter()
            .any(|pt| !inputs.contains(pt) && !snapshot.transaction_exists(&pt.tx_hash()))
        {
            return Err(Reject::RBFRejected(
                "new Tx contains unconfirmed inputs".to_string(),
            ));
        }

        // Rule #5, the replaced tx's descendants can not more than 100
        // and the ancestor of the new tx don't have common set with the replaced tx's descendants
        let mut replace_count: usize = 0;
        let mut all_conflicted = conflicts.clone();
        let ancestors = self.pool_map.calc_ancestors(&short_id);
        for conflict in conflicts.iter() {
            let descendants = self.pool_map.calc_descendants(&conflict.id);
            replace_count += descendants.len() + 1;
            if replace_count > MAX_REPLACEMENT_CANDIDATES {
                return Err(Reject::RBFRejected(format!(
                    "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
                    replace_count, MAX_REPLACEMENT_CANDIDATES,
                )));
            }

            if !descendants.is_disjoint(&ancestors) {
                return Err(Reject::RBFRejected(
                    "Tx ancestors have common with conflict Tx descendants".to_string(),
                ));
            }

            let entries = descendants
                .iter()
                .filter_map(|id| self.get_pool_entry(id))
                .collect::<Vec<_>>();

            for entry in entries.iter() {
                let hash = entry.inner.transaction().hash();
                if tx_inputs.iter().any(|pt| pt.tx_hash() == hash) {
                    return Err(Reject::RBFRejected(
                        "new Tx contains inputs in descendants of to be replaced Tx".to_string(),
                    ));
                }
            }
            all_conflicted.extend(entries);
        }
