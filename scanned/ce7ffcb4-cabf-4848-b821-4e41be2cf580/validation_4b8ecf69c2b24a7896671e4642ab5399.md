### Title
Missing `hash_type` Validation for Output Type Scripts in `ScriptHashTypeVerifier` — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` is documented to verify that the `ScriptHashType` of **all** transaction output scripts is within the range permitted by current consensus rules. However, it only validates the **lock script** `hash_type` of each output and silently skips the **type script** `hash_type`. A transaction submitter can craft an output whose type script carries a structurally valid but consensus-prohibited `hash_type` (e.g., `Data3 = 6` before that VM version is activated), and the non-contextual verifier will accept it without complaint.

---

### Finding Description

`ScriptHashTypeVerifier` is included in `NonContextualTransactionVerifier` and is the designated early-rejection gate for invalid hash types:

```
// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> { ... }
``` [1](#0-0) 

Its `verify()` loop iterates every output but only inspects `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else {
            return Err(...InvalidScriptHashType...);
        }
    }
    Ok(())
}
``` [2](#0-1) 

`output.type_()` — the optional type script — is never read. A type script with `hash_type = Data3` (byte value `6`, structurally valid per `ScriptHashType::verify_value` since it is even) would pass this verifier even when `Data3` is not yet activated by consensus. [3](#0-2) 

By contrast, the lower-level `check_data` helper correctly validates **both** lock and type scripts:

```rust
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
``` [4](#0-3) 

But `check_data` only tests structural validity (`verify_value`), not consensus-level activation. `ScriptHashTypeVerifier` is the only place that enforces the `ENABLED_SCRIPT_HASH_TYPE` set, and it does so only for lock scripts. [5](#0-4) 

The `ScriptHashType` enum already defines many future variants (`Data3` through `Data127`) that are structurally valid but not yet activated: [6](#0-5) 

---

### Impact Explanation

A transaction with an output whose type script carries a non-permitted `hash_type` (e.g., `Data3`) will:

1. Pass `NonContextualTransactionVerifier` (including `ScriptHashTypeVerifier`) — the designated early-rejection gate.
2. Be admitted to the tx pool pending contextual verification.
3. Fail only during script execution (`select_version` / `extract_script_and_dep_index` return `InvalidScriptHashType`). [7](#0-6) 

The practical consequences are:
- **Tx-pool pollution**: Malformed transactions bypass the cheap non-contextual filter and consume tx-pool admission resources (script execution is expensive).
- **Block-template corruption**: A miner node that includes such a transaction in a block template will produce a block that fails script verification on all other nodes, wasting PoW work and causing a temporary fork.
- **Consensus inconsistency**: Nodes that apply `check_data` (relayer/synchronizer path) reject the transaction earlier than nodes that receive it via RPC, creating divergent tx-pool states. [8](#0-7) 

---

### Likelihood Explanation

Any unprivileged RPC caller or tx-pool submitter can craft such a transaction. The `hash_type` field is a single byte in the Molecule-encoded `Script` struct, and values like `6` (`Data3`) are structurally valid and will not be rejected by Molecule deserialization or `check_data`. No special privileges, keys, or majority hash power are required. [9](#0-8) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script `hash_type` when the type script is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        Self::check_script_hash_type(output.lock().hash_type())?;
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            Self::check_script_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the completeness already present in `check_data`, which validates both lock and type scripts. [4](#0-3) 

---

### Proof of Concept

1. Construct a transaction output with a type script whose `hash_type` byte is `6` (`Data3`, not yet activated).
2. Submit via RPC (`send_transaction`).
3. Observe that `NonContextualTransactionVerifier` (including `ScriptHashTypeVerifier`) accepts the transaction.
4. The transaction enters the tx pool and is only rejected later during contextual script verification — after consuming significantly more node resources than the early-rejection path.

The existing test `test_not_enabled_hash_type_output_lock` demonstrates the check works for lock scripts but has no counterpart for type scripts, confirming the gap: [10](#0-9)

pub struct ScriptHashTypeVerifier<'a> {

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}
