### Title
Expired Alert Replay Enables Re-Execution of Notification Actions Due to Missing Guard in `clear_expired_alerts()` — (File: `util/network-alert/src/notifier.rs`)

### Summary
The `clear_expired_alerts()` function in `util/network-alert/src/notifier.rs` removes expired alerts from `received_alerts` and `noticed_alerts` but does **not** record their IDs in `cancel_filter`. Because the sole idempotency guard (`has_received()`) checks only those two structures, any unprivileged connected peer can replay the raw bytes of a previously-valid, now-expired alert and cause the node to re-verify, re-broadcast, and re-execute the alert notification action — an exact structural analog of the reported "missing guard allows multiple executions" class.

---

### Finding Description

### Impact Explanation

An unprivileged peer that stored the raw bytes of any previously-broadcast, now-expired alert can replay those bytes to any reachable node. The node will:

1. Re-verify the alert (signatures still valid — no expiry check in `verify_signatures`).
2. Re-broadcast the alert to all its connected peers, causing a network-wide cascade.
3. Re-execute the operator-configured `network_alert_notify_script` (arbitrary shell command) for each replayed message.
4. Re-deliver the alert to all RPC/WebSocket subscribers.

Repeated replays cause unbounded re-execution of the notification script and network-wide alert spam. If the script performs side-effectful actions (e.g., paging on-call engineers, triggering automated responses), those actions fire repeatedly. The cascading re-broadcast amplifies the effect across the entire P2P network.

---

### Likelihood Explanation

- Every node that was online during the original alert broadcast stored the raw alert bytes (they are relayed verbatim).
- Any connected peer (zero privilege required) can send those bytes after the alert's `notice_until` timestamp passes.
- `clear_expired_alerts` is called on every `connected` and `notify` event in `AlertRelayer`, so the window opens reliably and predictably.
- No key material, mining power, or special role is needed — only a stored copy of the alert and a TCP connection to the target node.

---

### Recommendation

In `clear_expired_alerts`, insert each expired alert's ID into `cancel_filter` before removing it from `received_alerts`, so that `has_received` continues to return `true` for those IDs:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ()); // <-- add this
            false
        } else {
            true
        }
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

This mirrors the recommendation in the external report: add a guard at the entry point so that once an alert has been processed, no further actions can be triggered from it.

### Proof of Concept

1. Node A receives a valid alert (ID=`0x1`, `notice_until` = T+60s), signed by 2-of-4 CKB developer keys. It is stored in `received_alerts`, `noticed_alerts`, and broadcast to peers.
2. At time T+61s, `clear_expired_alerts` fires (triggered by any peer connection event). The alert is removed from `received_alerts` and `noticed_alerts`. `cancel_filter` is unchanged. `has_received(0x1)` now returns `false`.
3. Attacker (any connected peer) sends the original raw alert bytes to Node A.
4. `alert_relayer.received()` calls `has_received(0x1)` → `false`. Proceeds.
5. `verifier.verify_signatures()` → passes (signatures are still cryptographically valid).
6. Node A re-broadcasts the alert to all connected peers (step 6 in `alert_relayer.rs` lines 166–176).
7. `notifier.add()` is called. `noticed_alerts.contains(alert)` → `false` (cleared). `notify_controller.notify_network_alert()` fires again. `network_alert_notify_script` is re-executed.
8. Each peer that receives the re-broadcast repeats steps 2–7 if their own `clear_expired_alerts` has also fired, causing a network-wide re-execution cascade. [7](#0-6) [8](#0-7)
