### Title
Unbounded Per-Session Announce Flooding Allows Peer Store Saturation — (`network/src/protocols/discovery/mod.rs`)

### Summary

An unprivileged remote peer can send an unlimited number of `Nodes(announce=true)` messages per session. Because `announce=true` messages never set `received_nodes`, the one-time-receive guard does not apply. The only check is a per-message item count (`≤ ANNOUNCE_THRESHOLD=10` items, `≤ MAX_ADDRS=3` addresses each). With no per-session rate limit, an attacker can drive `addr_mgr.add_new_addrs` repeatedly, filling the peer store to its hard cap (`ADDR_COUNT_LIMIT=16384`) with attacker-controlled addresses. Once full, the eviction logic can be defeated by spreading addresses across many `/16` subnets (≤4 per group), causing `EvictionFailed` and silently rejecting all subsequent legitimate addresses.

---

### Finding Description

**Step 1 — No rate limit on `announce=true` messages.**

`verify_nodes_message` only enforces a per-message item count: [1](#0-0) 

There is no counter, timestamp, or token-bucket guard on how many `announce=true` messages a single session may send.

**Step 2 — `received_nodes` guard is bypassed for `announce=true`.**

The duplicate-message guard only fires for `announce=false`: [2](#0-1) 

For `announce=true`, `state.received_nodes` is never set, so the `else` branch (which calls `add_new_addrs`) executes on every single message.

**Step 3 — Every passing message writes directly to the peer store.** [3](#0-2) 

`add_new_addrs` iterates every address and calls `peer_store.add_addr` with no session-level quota: [4](#0-3) 

**Step 4 — Peer store hard cap is 16 384 entries.** [5](#0-4) 

**Step 5 — Eviction can be defeated.**

`check_purge` only evicts entries that are non-connectable (tried too many times) or belong to a `/16` group with more than 4 peers: [6](#0-5) 

An attacker who spreads ≤4 addresses across each of 4 096 distinct `/16` subnets fills all 16 384 slots. Neither eviction branch finds a candidate, `EvictionFailed` is returned, and every subsequent `add_addr` call silently fails: [7](#0-6) 

The failure is only logged at `debug` level and the address is dropped: [8](#0-7) 

---

### Impact Explanation

Once the peer store is saturated with attacker-controlled addresses:
- Legitimate peer addresses advertised by honest nodes are silently rejected.
- The victim node's feeler and outbound connection pools draw exclusively from attacker-controlled entries, degrading or eliminating organic peer discovery.
- This is a prerequisite step for an eclipse attack: the attacker does not need majority hashpower, only a single persistent P2P session.

---

### Likelihood Explanation

- Requires only a standard P2P TCP connection — no privilege, no key, no PoW.
- ~547 messages of 10 items × 3 addresses each saturates the store (16 384 / 30 ≈ 547). Each message is a few hundred bytes; the entire attack fits in a single TCP session in seconds.
- `is_valid_addr` filters non-routable IPs, but the attacker only needs to advertise syntactically valid routable addresses — they do not need to own them.
- The `addr_known` bloom filter is per-session and only prevents re-sending the same address to the attacker; it does not prevent the attacker from using fresh addresses each message.

---

### Recommendation

1. **Add a per-session rate limit on `announce=true` messages** — e.g., a counter in `SessionState` that disconnects or ignores the peer after N announce messages per time window.
2. **Cap the total number of addresses accepted per session** across all announce messages (e.g., ≤ `ANNOUNCE_THRESHOLD` addresses total per session lifetime, not per message).
3. **Improve eviction diversity** — consider source-session attribution so that addresses from a single session can be bulk-evicted if the session is later identified as malicious.

---

### Proof of Concept

```
1. Connect to victim node on the discovery protocol.
2. In a loop, send Nodes { announce: true, items: [10 × Node { addresses: [addr_i, addr_i+1, addr_i+2] }] }
   where each addr_i is a unique routable IP from a distinct /16 subnet.
3. After ~550 iterations the peer store reaches ADDR_COUNT_LIMIT=16384.
4. Assert: victim's peer store contains only attacker-supplied addresses.
5. Assert: subsequent honest Nodes messages from other peers are silently dropped (EvictionFailed).
6. Assert: victim's outbound feeler connections go exclusively to attacker-controlled IPs.
```

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                    .into_iter()
