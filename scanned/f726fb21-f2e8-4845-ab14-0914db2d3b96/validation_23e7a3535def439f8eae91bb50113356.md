### Title
JSON-RPC Batch Deserialization Before Size Check Enables Memory Amplification — (`File: rpc/src/server.rs`)

---

### Summary

In `rpc/src/server.rs`, the `handle_jsonrpc` handler calls `serde_json::from_str::<Request>(req)` to fully deserialize the entire JSON payload — including all batch items — into heap-allocated Rust structures **before** any batch size check is applied. The batch size limit (`rpc_batch_limit`) is disabled by default and, even when configured, the check occurs after deserialization is complete. Compounding this, the `max_request_body_size` configuration field is defined but **never applied** to the axum router as a middleware layer, meaning no HTTP body size limit is enforced at the transport level. An RPC caller can exploit these two gaps together to cause unbounded memory consumption via serde_json amplification.

---

### Finding Description

**Step 1 — `max_request_body_size` is never enforced.**

`RpcConfig::max_request_body_size` is declared in `util/app-config/src/configs/rpc.rs` and documented as "Default is 10MiB" in `resource/ckb.toml`, but it is never passed to the axum router. In `start_server`, the `Router` is built with only `CorsLayer`, `TimeoutLayer`, and two `Extension` layers — no `RequestBodyLimitLayer` or `DefaultBodyLimit::max()` is present:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())
    .layer(TimeoutLayer::with_status_code(...))
    .layer(Extension(stream_config));
```

The config field is read and stored but silently ignored at runtime. Axum's own default body limit (2 MiB in axum 0.7+) may apply, but the operator-configured value does not.

**Step 2 — Full batch deserialization before the batch size check.**

In `handle_jsonrpc`, `serde_json::from_str::<Request>(req)` at line 238 fully deserializes the entire payload — including every element of a batch array — into a `Vec<Call>` in heap memory. Only after this allocation does the code branch on `Request::Batch(calls)` and check `JSONRPC_BATCH_LIMIT`:

```rust
let req = serde_json::from_str::<Request>(req);   // full deserialization here
...
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()  // check is AFTER alloc
        && calls.len() > *batch_size { ... }
```

**Step 3 — Batch limit is `None` by default.**

`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only initialized when `config.rpc_batch_limit` is `Some`. In the shipped default config, `rpc_batch_limit` is commented out:

```toml
# By default, there is no limitation on the size of batch request size
# rpc_batch_limit = 2000
```

So `JSONRPC_BATCH_LIMIT.get()` always returns `None` in a default deployment, and the batch size check at line 275 is never reached.

**Memory amplification mechanism.**

`serde_json::from_str::<Request>` converts compact JSON into `jsonrpc_core::Call` enum variants, each containing `Params::Array(Vec<Value>)`. Each `serde_json::Value` carries:
- Strings → heap-allocated `String` (capacity overhead)
- Arrays → `Vec<Value>` with element boxing (24+ bytes per `Value` enum)
- Objects → `IndexMap<String, Value>` with hash table overhead

A compact JSON batch near axum's 2 MiB default limit can expand to 12–16 MiB in memory per request. With no per-connection concurrency limit on the HTTP handler, many concurrent connections multiply this linearly.

---

### Impact Explanation

An RPC caller (local or remote if the operator has exposed the RPC port) can send repeated large batch JSON payloads that are fully deserialized into memory before any size or count check is applied. With no enforced body size limit and no default batch count limit, each request can consume 6–8× its wire size in heap memory. Concurrent requests from the same or multiple callers can exhaust available memory, causing the CKB node process to be OOM-killed, resulting in a denial of service: the node stops processing blocks, relaying transactions, and responding to peers.

---

### Likelihood Explanation

The RPC port is bound to `127.0.0.1` by default, which limits remote exploitation. However:
- The TCP RPC endpoint (`tcp_listen_address`) and WebSocket endpoint (`ws_listen_address`) are optional but commonly enabled and may be bound to external interfaces.
- Any local process (including scripts, CI pipelines, or SSRF from a co-located service) qualifies as an RPC caller.
- The attack requires no authentication, no special privileges, and no knowledge of chain state — only the ability to POST to the RPC port.
- The default configuration ships with both protections disabled (no body limit enforcement, no batch count limit), making exploitation trivially reproducible on any default node that has exposed its RPC.

---

### Recommendation

1. **Enforce `max_request_body_size` in the axum router.** Add `DefaultBodyLimit::max(config.max_request_body_size)` as a layer in `start_server` so the configured limit is actually applied before the body reaches `handle_jsonrpc`.

2. **Check batch count before deserialization.** Parse only enough of the JSON to determine whether the top-level value is an array and count its elements (e.g., using a streaming JSON parser or a pre-check on the raw bytes) before calling `serde_json::from_str::<Request>`.

3. **Enable `rpc_batch_limit` by default.** Change the default config from a commented-out `# rpc_batch_limit = 2000` to an active, conservative default (e.g., `rpc_batch_limit = 100`) so that the post-deserialization guard at least limits damage in the common case.

4. **Add per-connection concurrency limiting** on the HTTP handler to bound the number of simultaneous large deserializations.

### Proof of Concept

```python
import socket, json, time

# Craft a batch of requests with large string params to maximize amplification
single = {"jsonrpc":"2.0","method":"get_tip_block_number","params":[],"id":1,
          "_pad": "A" * 900}  # pad each item toward the per-item limit
batch = [single] * 2000       # 2000 items, no default batch limit to stop this

payload = json.dumps(batch).encode()
print(f"Wire size: {len(payload)/1024:.1f} KB")
# serde_json amplification: ~6-8x → ~12-16x wire size in heap

for _ in range(20):           # 20 concurrent connections
    s = socket.socket()
    s.connect(("127.0.0.1", 8114))
    http = (
        f"POST / HTTP/1.1\r\nHost: localhost\r\n"
        f"Content-Type: application/json\r\n"
        f"Content-Length: {len(payload)}\r\n\r\n"
    ).encode() + payload
    s.sendall(http)
    # Do not read response — keep connection open to hold memory
    time.sleep(0.01)

print("Requests sent. Monitor node RSS for amplified allocation.")
```

**Execution path:**
1. HTTP layer accepts the body (no enforced size limit).
2. `handle_jsonrpc` → `serde_json::from_str::<Request>(req)` at line 238 allocates the full `Vec<Call>` (2000 elements) in heap.
3. `JSONRPC_BATCH_LIMIT.get()` returns `None` (default config) → no rejection.
4. Each concurrent connection holds its allocation while the stream processes calls sequentially.
5. Node RSS grows proportionally to concurrent connections × amplified batch size.
