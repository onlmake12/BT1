### Title
Division-by-Zero Panic in `EpochNumberWithFraction::to_rational()` via Unchecked Deserialization — (`File: util/types/src/conversion/blockchain.rs`, `util/types/src/core/extras.rs`, `verification/src/transaction_verifier.rs`)

---

### Summary

The `Unpack` trait implementation for `EpochNumberWithFraction` uses `from_full_value_unchecked`, which skips the zero-length normalization guard. Any attacker-controlled `u64` deserialized as an `EpochNumberWithFraction` (e.g., via the `since` field of a transaction input) can carry a zero epoch-length with a non-zero number or index. When `to_rational()` is subsequently called in the `SinceVerifier`, a division-by-zero panic occurs, crashing or aborting the verification thread.

---

### Finding Description

**Root cause — unchecked deserialization:**

`util/types/src/conversion/blockchain.rs` lines 21–31 define the canonical `Unpack` and `From` implementations for `EpochNumberWithFraction`:

```rust
impl<'r> Unpack<core::EpochNumberWithFraction> for packed::Uint64Reader<'r> {
    fn unpack(&self) -> core::EpochNumberWithFraction {
        core::EpochNumberWithFraction::from_full_value_unchecked(self.unpack())
    }
}
impl<'r> From<packed::Uint64Reader<'r>> for core::EpochNumberWithFraction {
    fn from(value: packed::Uint64Reader<'r>) -> core::EpochNumberWithFraction {
        core::EpochNumberWithFraction::from_full_value_unchecked(value.into())
    }
}
``` [1](#0-0) 

The safe constructor `from_full_value` calls `.normalize()` to rewrite a zero-length epoch to `(number, index=0, length=1)`:

```rust
pub fn from_full_value(value: u64) -> Self {
    Self::from_full_value_unchecked(value).normalize()
}
``` [2](#0-1) 

The unchecked variant carries an explicit warning:

> The `EpochNumberWithFraction` constructed by this method has a potential risk that when call `self.to_rational()` may lead to a panic if the user specifies a zero epoch length. [3](#0-2) 

**Root cause — division-by-zero in `to_rational()`:**

```rust
pub fn to_rational(self) -> RationalU256 {
    if self.0 == 0 {
        RationalU256::zero()
    } else {
        RationalU256::new(self.index().into(), self.length().into())
            + U256::from(self.number())
    }
}
``` [4](#0-3) 

The guard `if self.0 == 0` only handles the genesis sentinel (all bits zero). If an attacker crafts a value where the epoch-number or epoch-index bits are non-zero but the epoch-length bits are zero, `self.0 != 0` and execution falls into `RationalU256::new(index, 0)`. The subsequent `+` operation performs cross-multiplication and division by the denominator (length = 0), triggering a Rust integer division-by-zero panic.

**Reachable call sites in the verifier:**

`grep` confirms `to_rational()` is called **7 times** in `verification/src/transaction_verifier.rs`, and `from_full_value_unchecked` is called **once** in the same file — directly in the `SinceVerifier` path that processes the attacker-controlled `since` field of transaction inputs. [5](#0-4) 

The `since` field of a CKB transaction input is a raw `u64`. When the `SinceVerifier` interprets it as an epoch-based lock, it constructs an `EpochNumberWithFraction` from the attacker-supplied bits. Because the `Unpack` implementation (and the direct `from_full_value_unchecked` call in the verifier) skip normalization, a crafted `since` value with zero epoch-length bits but non-zero number/index bits passes through unchecked and reaches `to_rational()`.

---

### Impact Explanation

**Impact: High**

A Rust integer division-by-zero is an unrecoverable panic. In CKB's architecture, transaction verification runs inside the tx-pool service and the block-verification pipeline. A panic in either path:

- **Tx-pool**: crashes the tx-pool worker thread; depending on the thread-pool implementation, this can abort the entire node process or leave the pool in a broken state, blocking all subsequent transaction admission and block assembly.
- **Block verification**: crashes the chain-service thread, halting block processing and effectively DoS-ing the node.

The attacker does not need to spend any significant resources — a single dust transaction with a crafted `since` field suffices.

---

### Likelihood Explanation

**Likelihood: Medium**

Any unprivileged RPC caller or P2P transaction relayer can submit a transaction with an arbitrary `since` field. The crafted value is trivial to construct: set the epoch-type flag bits in the `since` field, set the epoch-number or epoch-index bits to any non-zero value, and set the epoch-length bits to zero. No hashpower, no privileged access, and no prior chain state is required.

---

### Recommendation

Replace `from_full_value_unchecked` with `from_full_value` (which calls `.normalize()`) in the `Unpack` and `From` implementations so that all wire-deserialized `EpochNumberWithFraction` values are safe before any arithmetic is performed:

```diff
-impl<'r> Unpack<core::EpochNumberWithFraction> for packed::Uint64Reader<'r> {
-    fn unpack(&self) -> core::EpochNumberWithFraction {
-        core::EpochNumberWithFraction::from_full_value_unchecked(self.unpack())
-    }
-}
+impl<'r> Unpack<core::EpochNumberWithFraction> for packed::Uint64Reader<'r> {
+    fn unpack(&self) -> core::EpochNumberWithFraction {
+        core::EpochNumberWithFraction::from_full_value(self.unpack())
+    }
+}
``` [6](#0-5) 

Additionally, add an explicit zero-length guard inside `to_rational()` as a defense-in-depth measure:

```diff
 pub fn to_rational(self) -> RationalU256 {
-    if self.0 == 0 {
+    if self.0 == 0 || self.length() == 0 {
         RationalU256::zero()
     } else {
``` [4](#0-3)

### Proof of Concept

1. Construct a `since` value with the epoch-type flag set, epoch-number = 1, epoch-index = 0, epoch-length = 0. In the packed 64-bit layout (`length[63:48] | index[47:32] | number[23:0]`), this is `0x0000_0000_0000_0001` with the epoch-type flag bits set — length field = 0, number field = 1.
2. Submit a transaction via RPC (`send_transaction`) with one input whose `since` field carries this value.
3. The tx-pool calls `SinceVerifier`, which calls `from_full_value_unchecked` on the `since` bits, producing an `EpochNumberWithFraction` with `length() == 0` and `number() == 1`.
4. `to_rational()` is called: `self.0 != 0`, so it executes `RationalU256::new(0, 0) + U256::from(1)`, dividing by zero and panicking.
5. The panic propagates through the tx-pool worker, crashing the verification thread and blocking all subsequent pool operations — analogous to the Ion pool DoS.
