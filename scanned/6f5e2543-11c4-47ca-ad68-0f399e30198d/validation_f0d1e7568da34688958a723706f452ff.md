Audit Report

## Title
Admission Fee-Rate Check Uses Size-Only Weight While Eviction/Scoring Uses Cycle-Adjusted Weight, Allowing High-Cycles Transactions to Bypass `min_fee_rate` — (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` validates the minimum fee rate using only the serialized transaction size as the weight denominator, while all downstream scoring and eviction logic uses `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns the true cycle count, `_process_tx` creates the `TxEntry` and submits it to the pool with no second fee-rate check against the actual weight. A transaction sender can craft a high-cycles transaction that passes the size-based admission gate but carries an effective fee rate far below `min_fee_rate`, causing it to be accepted into the pool, permanently deprioritized by miners, and never included in a block — soft-locking the sender's input cells until pool expiry or eviction.

## Finding Description
`check_tx_fee` in `tx-pool/src/util.rs` (L28–53) explicitly acknowledges the discrepancy:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The actual weight formula used everywhere else is `get_transaction_weight` in `util/types/src/core/tx_pool.rs` (L298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`.

The processing pipeline in `_process_tx` (`tx-pool/src/process.rs`, L705–777):

1. **L715**: `pre_check` → calls `check_tx_fee` with `tx_size` only (cycles unknown)
2. **L724–732**: `verify_rtx` → actual cycles now known
3. **L734**: `try_or_return_with_snapshot!(verified_ret, snapshot)` — cycles available here
4. **L751**: `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles
5. **L753**: `submit_entry` — entry inserted into pool

There is no second fee-rate check between steps 2 and 4. The fee accepted during `pre_check` is carried forward unchanged.

`TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs`, L115–118) and `AncestorsScoreSortKey` (L221–231) both use the correct weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
```

`txs_to_commit` in `tx-pool/src/component/tx_selector.rs` (L97–162) iterates via `sorted_proposed_iter()`, which sorts by `AncestorsScoreSortKey` — so a transaction with a low actual fee rate is sorted to the bottom and never selected for a block template.

## Impact Explanation
This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

Using default config (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):

| Parameter | Value |
|---|---|
| `tx_size` | 500 bytes |
| `cycles` | 70,000,000 |
| Size-based min fee (admission) | `1000 × 500 / 1000 = 500 shannons` |
| Actual weight | `max(500, 70,000,000 × 0.000_170_571_4) ≈ 11,940` |
| Actual fee rate at 500 shannons | `500 / 11,940 × 1000 ≈ 42 shannons/KW` |

A transaction paying 500 shannons passes `check_tx_fee` but has an actual fee rate ~24× below `min_fee_rate`. It enters the pool, is never selected by `txs_to_commit`, and occupies pool space until evicted or expired. At 500 shannons per transaction, an attacker can continuously submit such transactions at negligible cost, maintaining pool pressure. While the eviction mechanism (which also uses actual weight) will eventually remove these entries, the attacker can replenish them faster than they are evicted, degrading pool quality for legitimate users and consuming node verification resources (70M cycles per verification is expensive).

## Likelihood Explanation
The entry path is fully unprivileged via `send_transaction` RPC or P2P relay. Writing a CKB script that consumes ~70M cycles is within the capability of any script author — ZK verifiers and complex multi-sig schemes routinely approach this limit. No special privileges, leaked keys, or victim mistakes are required. The attack is repeatable and cheap.

## Recommendation
After `verify_rtx` returns the actual cycle count in `_process_tx` (`tx-pool/src/process.rs`, after L734), add a second fee-rate check using the true weight before creating the `TxEntry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Re-check fee rate against actual weight now that cycles are known
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee {
    return Some((
        Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64())),
        snapshot,
    ));
}

let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

Alternatively, `check_tx_fee` could accept an optional `cycles` parameter and use `get_transaction_weight` when cycles are available, unifying the admission and eviction criteria.

## Proof of Concept
1. Write a CKB lock script that loops to consume ~70,000,000 cycles (near `max_tx_verify_cycles`).
2. Create a transaction spending a cell locked by this script with `tx_size ≈ 500 bytes`.
3. Set the fee to exactly `ceil(min_fee_rate × tx_size / 1000) = 500 shannons`.
4. Submit via `send_transaction` RPC.
5. **Expected (admission)**: `check_tx_fee` computes `min_fee = 1000 × 500 / 1000 = 500`; fee ≥ min_fee → transaction accepted into pool.
6. **Observed (scoring)**: `TxEntry::fee_rate()` computes `weight = max(500, 11940) = 11940`; effective fee rate ≈ 42 shannons/KW. `txs_to_commit` never selects this transaction. It remains in the pool for up to `expiry_hours` (default 12 hours), soft-locking the input cell. Repeating step 4 with fresh UTXOs maintains continuous pool pressure at ~500 shannons per slot occupied.
