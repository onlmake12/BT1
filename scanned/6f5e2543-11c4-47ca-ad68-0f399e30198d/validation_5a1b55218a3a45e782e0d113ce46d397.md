Audit Report

## Title
Unbounded O(N×M) Linear Scan in `add_ask_for_txs` Triggered by Unprivileged Relay Peers — (File: sync/src/types/mod.rs)

## Summary

`SyncState::add_ask_for_txs` performs a full nested-loop scan over all entries in `unknown_tx_hashes` (up to 50,000) and all peers inside each entry's `peers` Vec while holding the `unknown_tx_hashes` mutex. Any unprivileged peer can fill the queue to capacity with fabricated tx hashes and then repeatedly trigger this scan, causing sustained CPU load and mutex starvation that degrades transaction relay and compact-block reconstruction on the targeted node.

## Finding Description

**Root cause — O(N×M) scan under mutex:**

`add_ask_for_txs` acquires the `unknown_tx_hashes` mutex at line 1484 and holds it for the entire function. After inserting hashes, it checks whether the queue has reached its soft cap. When it has, it iterates every entry and every peer inside each entry to count how many entries belong to the calling peer:

```rust
// sync/src/types/mod.rs lines 1506-1528
if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE          // 50,000
    || unknown_tx_hashes.len()
        >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
{
    let mut peer_unknown_counter = 0;
    for (_hash, priority) in unknown_tx_hashes.iter() {   // O(N)
        for peer in priority.peers.iter() {               // O(M) per entry
            if *peer == peer_index {
                peer_unknown_counter += 1;
            }
        }
    }
    ...
}
```

This entire scan executes while the mutex is held, blocking every other caller that needs the lock.

**Root cause — unbounded `peers` Vec per entry:**

`UnknownTxHashPriority::push_peer` appends without any cap. When P peers all announce the same hash, the `Occupied` branch calls `push_peer` for each, growing the inner `peers` Vec to length P. This makes each scan iteration O(P) for shared hashes.

**Existing guards are insufficient:**

- The rate limiter (30 msg/s per `(peer, message_type)`) only bounds per-peer rate; with K attacker peers it allows 30×K scans/second.
- The `MAX_RELAY_TXS_NUM_PER_BATCH = 32767` per-message cap does not prevent filling the queue — two peers each sending ~25,000 unique hashes reaches `MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`.
- The soft-cap check fires *after* insertion, not before, so the queue is already full when the scan begins.

**Call chain:**

```
RelayTransactionHashes P2P message
  → Relayer::try_process  (sync/src/relayer/mod.rs:143)
  → TransactionHashesProcess::execute  (sync/src/relayer/transaction_hashes_process.rs:49)
  → SyncState::add_ask_for_txs  (sync/src/types/mod.rs:1483)
      → mutex lock acquired (line 1484)
      → O(N×M) scan (lines 1517-1523)
      → mutex released
```

## Impact Explanation

The `unknown_tx_hashes` mutex is shared with `mark_as_known_txs`, `pop_ask_for_txs`, and `tx_filter`. Holding it for the duration of a 50,000-entry × K-peer scan at 30×K times/second starves these operations, directly degrading transaction relay throughput and compact-block reconstruction. Compact-block reconstruction stalls translate to delayed block propagation for the targeted node, which constitutes **CKB network congestion with few costs** — matching the High impact tier (10001–15000 points): *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

- Any peer can connect to a CKB node without authentication.
- `RelayTransactionHashes` with fabricated (non-existent) tx hashes requires no PoW, no fee, and no valid UTXO.
- Filling `unknown_tx_hashes` to 50,000 entries requires only two peers each sending one batch of ~25,000 hashes — well within `MAX_RELAY_TXS_NUM_PER_BATCH = 32,767`.
- The rate limiter (30/s per peer) does not prevent the attack; it only bounds per-peer rate.
- The existing integration test `test/src/specs/relay/too_many_unknown_transactions.rs` confirms the code path is reachable and exercised in CI.

## Recommendation

1. **Replace the O(N×M) scan with a per-peer counter map.** Maintain a `HashMap<PeerIndex, usize>` alongside `unknown_tx_hashes` that is updated incrementally on insert/remove, making the overflow check O(1) and eliminating the need to hold the mutex during a full scan.
2. **Cap the `peers` Vec per entry.** In `push_peer`, enforce a maximum number of peers per `UnknownTxHashPriority` entry (e.g., equal to the maximum number of connected peers) to bound the inner loop.
3. **Enforce a hard cap before insertion.** Reject new hashes from a peer that has already reached `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` before inserting, rather than scanning after the fact.

## Proof of Concept

```
1. Attacker opens K=10 inbound connections to the target node.

2. Peer 1 sends one RelayTransactionHashes message with 25,000 unique
   fabricated tx hashes; Peer 2 sends another 25,000 unique hashes.
   → unknown_tx_hashes reaches MAX_UNKNOWN_TX_HASHES_SIZE = 50,000.

3. All K peers also announce the same 1,000 shared hashes, causing
   push_peer to be called K times per shared entry → peers Vec length = K.

4. Each peer now sends RelayTransactionHashes at 30 msg/s (rate limit).
   → 300 scans/second, each iterating 50,000 entries × K peers per entry
     = 500,000 comparisons per scan → 150,000,000 comparisons/second,
     all while holding the unknown_tx_hashes mutex.

5. Observable effect: mark_as_known_txs, pop_ask_for_txs, and tx_filter
   are starved; relay latency spikes; compact-block reconstruction stalls;
   CPU pegged on the relay async task.

Verification: run the existing TooManyUnknownTransactions integration test
(test/src/specs/relay/too_many_unknown_transactions.rs) as a baseline,
then extend it with multiple peers and measure mutex wait time and relay
latency under the described load.
```
