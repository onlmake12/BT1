## Vulnerability Analysis

Let me trace through the exact code path to verify each step of the claim.

### Step 1: `Group::None` for Onion3 addresses

In `network_group.rs`, `From<&Multiaddr> for Group` calls `multiaddr_to_socketaddr`. For Onion3 multiaddrs, this returns `None`, so the function falls through to `Group::None`: [1](#0-0) 

### Step 2: Freshly added addresses are "connectable"

`add_addr` calls `AddrInfo::new(addr, 0, score, flags.bits())` — `last_connected_at_ms = 0`, `attempts_count = 0`. [2](#0-1) 

In `is_connectable`, with `last_connected_at_ms=0` and `attempts_count=0`:
- `tried_in_last_minute` → false
- `attempts_count >= ADDR_MAX_RETRIES (3)` → false (0 < 3)
- `attempts_count >= ADDR_MAX_FAILURES (10)` → false (0 < 10)

So freshly added addresses return `true` from `is_connectable`. Step 1 of `check_purge` (evict non-connectable) finds **zero candidates**. [3](#0-2) 

### Step 3: The `take(len/2)` integer-division bug

When `candidate_peers.is_empty()` after step 1, `check_purge` enters the network-group eviction path: [4](#0-3) 

With 16384 Onion3 addresses all in `Group::None`:
- `peers_by_network_group` has exactly **1 entry**
- `len = 1`
- `peers.into_iter().take(len / 2)` → `take(1 / 2)` → **`take(0)`** (Rust integer division)
- The iterator yields nothing; `candidate_peers` is empty
- Line 399–400: `return Err(PeerStoreError::EvictionFailed.into())`

### Step 4: Attacker reachability via discovery

`add_addr` is called from `network/src/protocols/discovery/mod.rs` when processing `AddressesMessage` from peers. There is no validation preventing Onion3 multiaddrs from being accepted. The `AddrManager::add` deduplicates by exact address, so the attacker needs 16384 **distinct** Onion3 addresses — trivially achievable given the 35-byte host + 2-byte port space. [5](#0-4) 

`ADDR_COUNT_LIMIT` is 16384: [6](#0-5)

### Title
`check_purge` integer-division bug allows permanent peer-store DoS via `Group::None` flooding — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
`check_purge` uses `take(len / 2)` where `len` is the number of distinct network groups. When all stored addresses map to `Group::None` (e.g., Onion3 multiaddrs), `len == 1` and integer division yields `take(0)`, selecting zero eviction candidates. The function returns `EvictionFailed`, permanently blocking new peer address insertion.

### Finding Description
In `check_purge` (line 376), after grouping addresses by `Group::from(&addr.addr)`, the code takes only the top `len / 2` groups for eviction candidates. When `len == 1` (all addresses share `Group::None`), `1 / 2 == 0` in Rust integer arithmetic, so no group is processed and no address is evicted. The subsequent `candidate_peers.is_empty()` check at line 399 then returns `Err(PeerStoreError::EvictionFailed)`.

The precondition — all addresses connectable — is satisfied by freshly discovered addresses: `AddrInfo::new` sets `last_connected_at_ms = 0` and `attempts_count = 0`, which passes all `is_connectable` checks until `ADDR_MAX_RETRIES` (3) or `ADDR_MAX_FAILURES` (10) are reached.

### Impact Explanation
Once the peer store is filled with 16384 `Group::None` addresses, every subsequent `add_addr` call returns `EvictionFailed`. The node cannot learn new peers from discovery, degrading its view of the network. This impairs block/transaction propagation, increases stale-tip risk, and can delay transaction confirmation — matching the stated economic harm scope.

### Likelihood Explanation
An attacker connected to the target node via the discovery protocol can send `AddressesMessage` packets containing distinct Onion3 multiaddrs. No PoW, no privileged role, and no Sybil majority is required. The 16384-address threshold is reachable through repeated discovery messages from one or a small number of peers.

### Recommendation
Replace `take(len / 2)` with `take((len + 1) / 2)` (ceiling division) or `take(len.max(1))` to ensure at least one group is always considered for eviction. Additionally, consider capping the number of `Group::None` addresses accepted (e.g., a separate sub-limit for ungroupable addresses) to prevent a single group from monopolizing the store.

    let addr: Multiaddr = std::iter::once(onion3).collect();
    // First 16383 succeed; the 16384th fills the store

let mut host = [0xffu8; 35];

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times

            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
