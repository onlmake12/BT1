### Title
Unsafe `u128`-to-`u64` Truncating Cast in DAO Withdrawal Calculation Silently Corrupts Withdrawal Amount - (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` performs a `u128`-to-`u64` narrowing cast (`withdraw_counted_capacity as u64`) without an overflow check. Every other analogous `u128`-to-`u64` conversion in the same file uses the safe `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. If the intermediate `u128` result exceeds `u64::MAX`, the value is silently truncated, producing an incorrect (lower) maximum-withdraw figure that propagates into fee calculation and transaction verification.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← unsafe cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently truncates any bits above bit 63. The same file performs three other `u128`-to-`u64` conversions using the safe checked pattern:

```rust
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

```rust
let miner_issuance = Capacity::shannons(
    u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [3](#0-2) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [4](#0-3) 

The inconsistency is the root cause: the `calculate_maximum_withdraw` path was not updated to use the safe pattern that was applied everywhere else in the same module.

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

Since `withdrawing_ar ≥ deposit_ar` (the accumulation rate `ar` is monotonically non-decreasing), the ratio is always ≥ 1. With `counted_capacity` up to ~3.36 × 10¹⁸ shannons (total CKB supply) and `u64::MAX ≈ 1.84 × 10¹⁹`, overflow occurs when the ratio exceeds ~5.47×.

### Impact Explanation

When `withdraw_counted_capacity` overflows `u64`, `Capacity::shannons(withdraw_counted_capacity as u64)` silently holds a truncated (much smaller) value. This corrupts the return value of `calculate_maximum_withdraw`, which is consumed by:

1. `transaction_maximum_withdraw` → `transaction_fee` — fee is computed as `maximum_withdraw - outputs_capacity`. A truncated maximum-withdraw makes the fee appear negative or zero, causing `safe_sub` to return `DaoError::Overflow` and reject a valid DAO withdrawal transaction.
2. Directly by the DAO type script verifier to validate the withdrawal amount — an incorrect figure could cause a legitimate withdrawal to be rejected or, depending on script logic, allow an incorrect amount to pass.

The net effect is denial-of-service for DAO depositors whose withdrawal transactions are permanently rejected, or incorrect capacity accounting in the DAO subsystem.

### Likelihood Explanation

The overflow requires the `ar` ratio to grow beyond ~5.47×. The initial `ar` is `10^16`; it grows proportionally to secondary issuance relative to total locked capacity. On mainnet this growth is slow (fractions of a percent per year), making the condition practically unreachable in the near term. However:

- The condition is **not** guarded by any assertion or checked cast, unlike every other `u128→u64` conversion in the same file.
- The inconsistency is a latent defect that becomes exploitable as the chain ages or if consensus parameters change.
- This mirrors exactly the acknowledged finding in the external report: "extremely low likelihood within the current system … however, if this contract is reused … there could be a heightened risk."

### Recommendation

Replace the unsafe cast with the same safe pattern used elsewhere in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This is consistent with lines 204, 245, and 258 of the same file and eliminates the silent truncation.

### Proof of Concept

1. Construct a DAO deposit cell with `counted_capacity` near `u64::MAX / 2` (achievable with a large CKB holder).
2. Wait (or simulate via test) until `withdrawing_ar / deposit_ar > 5.47`.
3. Submit a DAO withdrawal transaction referencing that cell.
4. `calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a `u128` value exceeding `u64::MAX`.
5. `withdraw_counted_capacity as u64` silently wraps to a small value.
6. `transaction_fee` calls `maximum_withdraw.safe_sub(outputs_capacity)` — since the truncated maximum is less than `outputs_capacity`, `safe_sub` returns `DaoError::Overflow`.
7. The withdrawal transaction is rejected despite being valid, permanently locking the depositor's funds. [5](#0-4)
