### Title
Legacy migration floor silently overrides `max_ancestors_count`, enforcing 1000 instead of operator-configured 25 — (`util/app-config/src/legacy/tx_pool.rs`)

### Summary

`CKBAppConfig::load_from_slice` always deserializes through the legacy struct, which applies `cmp::max(1_000, configured_value)` to `max_ancestors_count`. The production `ckb.toml` ships with `max_ancestors_count = 25`, but the enforced runtime limit is always 1000. Any unprivileged transaction submitter can build 1000-deep chains that the operator believes are capped at 25.

### Finding Description

`CKBAppConfig::load_from_slice` in `util/app-config/src/app_config.rs` line 266 always deserializes into `legacy::CKBAppConfig`, then converts via `From<legacy::CKBAppConfig> for CKBAppConfig`. [1](#0-0) 

That conversion calls `tx_pool.into()`, which invokes `From<legacy::TxPoolConfig> for crate::TxPoolConfig`. [2](#0-1) 

Inside that `From` impl, line 129 applies a hard floor:

```rust
max_ancestors_count: cmp::max(DEFAULT_MAX_ANCESTORS_COUNT, max_ancestors_count),
```

where `DEFAULT_MAX_ANCESTORS_COUNT = 1_000`. [3](#0-2) 

The production template sets `max_ancestors_count = 25`: [4](#0-3) 

After `cmp::max(1000, 25)`, the value stored in `crate::TxPoolConfig::max_ancestors_count` is **1000**, not 25. This value is passed directly to `PoolMap::new(config.max_ancestors_count)` in `TxPool::new`. [5](#0-4) 

`PoolMap` uses `self.max_ancestors_count` as the admission gate in `check_and_record_ancestors`. [6](#0-5) 

There is no other place where the configured value is re-applied. The floor is unconditional.

### Impact Explanation

- An attacker submitting transactions via the `send_transaction` RPC (or P2P relay) can build chains of up to 1000 transactions, 40× longer than the operator intended.
- Each additional ancestor in a chain adds to the `get_tx_ancestors` traversal cost on every subsequent submission in that chain.
- The pool can hold 40× more chained transactions per "chain slot" than the operator configured, consuming proportionally more memory and CPU during block assembly.
- The operator has no effective way to lower the limit below 1000 without modifying source code, because the floor is unconditional.

### Likelihood Explanation

- Reachable by any unprivileged caller via `send_transaction` RPC or P2P transaction relay.
- No special privileges, keys, or hashpower required.
- The discrepancy is permanent for all nodes running the current codebase with any `max_ancestors_count < 1000` in their config.
- The production default `ckb.toml` ships with 25, so every default-configured mainnet node is affected.

### Recommendation

Remove or condition the `cmp::max` floor in `From<TxPoolConfig> for crate::TxPoolConfig`. If backward-compatibility migration is needed (to avoid breaking nodes that relied on the old 1000 default), the floor should only apply when the field was absent from the config (i.e., use the serde `default` mechanism), not when the operator has explicitly set a lower value. The current `max_ancestors_count` field in the legacy struct has no `#[serde(default)]` attribute, so a present value of 25 is a deliberate operator choice and must not be silently raised.
