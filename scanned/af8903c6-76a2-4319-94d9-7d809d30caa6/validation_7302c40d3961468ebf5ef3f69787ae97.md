### Title
`check_tx_fee` Uses Raw `tx_size` Without Cycles Normalization, Allowing Sub-Minimum Fee Rate Admission — (File: `tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` enforces the minimum fee rate using only the serialized byte size of a transaction, ignoring the cycles dimension entirely. The actual transaction weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`, but the gate check uses only `tx_size`. A transaction sender or relayer can craft a compact (small-byte) but cycle-heavy transaction that passes the fee gate with a fee far below what the actual weight-based minimum fee rate requires, admitting it to the tx-pool at a steep discount.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee as:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
``` [1](#0-0) 

The actual transaction weight used everywhere else in the codebase is defined in `util/types/src/core/tx_pool.rs`:

```rust
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

`check_tx_fee` is called in `pre_check` with `tx_size` before cycles are known: [3](#0-2) 

After verification, the entry is constructed with the actual `verified.cycles` and submitted — with no second fee-rate check against the full weight: [4](#0-3) 

`submit_entry` performs no fee-rate re-validation: [5](#0-4) 

The `TxEntry::fee_rate()` method — used for sorting and eviction — correctly uses `get_transaction_weight`: [6](#0-5) 

This creates a split: the **admission gate** uses size-only weight, while the **eviction/sorting logic** uses the full weight. A transaction that passes the gate may have an effective fee rate orders of magnitude below `min_fee_rate`.

---

### Impact Explanation

**Concrete example** with `min_fee_rate = 1000 shannons/KW` (default):

| Parameter | Value |
|---|---|
| `tx_size` | 100 bytes |
| `cycles` | 70,000,000 (max block cycles) |
| Actual weight | `max(100, 70,000,000 × 0.000_170_571_4)` = **11,940** |
| Fee required by `check_tx_fee` | `1000 × 100 / 1000` = **100 shannons** |
| Fee required by actual weight | `1000 × 11,940 / 1000` = **11,940 shannons** |
| Effective fee rate admitted | `100 × 1000 / 11,940` ≈ **8 shannons/KW** |
| Underpayment factor | **~119×** |

Such a transaction is admitted to the pool and occupies cycle-budget space at 1/119th the intended cost. An attacker can flood the pool with cycle-heavy, byte-light transactions, displacing legitimate transactions that pay the correct fee rate. The pool's `limit_size` eviction uses the correct weight-based fee rate, so these transactions will eventually be evicted when the pool fills — but the attacker can continuously re-submit them faster than they are evicted, keeping the pool polluted. [7](#0-6) 

---

### Likelihood Explanation

- **Entry path**: Any unprivileged RPC caller (`send_transaction`) or P2P relayer (`submit_remote_tx`) can submit such a transaction. No special privilege is required.
- **Craft difficulty**: Low. The attacker only needs a transaction with a script that consumes many cycles (e.g., a loop-heavy lock script) while keeping the serialized transaction small. The CKB-VM cycle limit per block is 70,000,000, which is publicly documented.
- **Cost**: The attacker pays only `~8 shannons/KW` effective fee rate instead of the enforced `1000 shannons/KW`, making sustained pool flooding economically cheap.

---

### Recommendation

Replace the size-only weight in `check_tx_fee` with the full `get_transaction_weight` calculation. Since cycles are not yet known at `pre_check` time, use the declared cycles (for relayed transactions) or the configured `max_tx_verify_cycles` as a conservative upper bound:

```rust
// Use declared_cycles or max_tx_verify_cycles as upper bound
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(tx_pool.config.max_tx_verify_cycles));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

This ensures the fee gate is at least as strict as the actual weight-based minimum, closing the normalization gap between admission and eviction logic. [8](#0-7) 

---

### Proof of Concept

1. Craft a CKB transaction with:
   - A lock script that executes a tight loop consuming ~70,000,000 cycles
   - Minimal witnesses/outputs so `tx_size ≈ 100` bytes
   - Fee = `min_fee_rate × tx_size / 1000 = 1000 × 100 / 1000 = 100 shannons`

2. Submit via `send_transaction` RPC.

3. `check_tx_fee` computes `min_fee = 1000 × 100 / 1000 = 100 shannons`. Fee passes.

4. Verification runs, actual cycles = ~70,000,000. `TxEntry` is created with `fee=100, cycles=70_000_000, size=100`.

5. `entry.fee_rate()` = `FeeRate::calculate(100, get_transaction_weight(100, 70_000_000))` = `100 × 1000 / 11_940` ≈ **8 shannons/KW** — far below `min_fee_rate = 1000`.

6. Transaction is in the pool, consuming cycle-budget space at 1/119th the intended cost. Repeat to fill the pool. [1](#0-0) [9](#0-8) [6](#0-5)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

    )

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,

    pub(crate) async fn submit_entry(
        &self,
        pre_resolve_tip: Byte32,
        entry: TxEntry,
        mut status: TxStatus,
    ) -> (Result<(), Reject>, Arc<Snapshot>) {
        let (ret, snapshot) = self
            .with_tx_pool_write_lock(move |tx_pool, snapshot| {
                // check_rbf must be invoked in `write` lock to avoid concurrent issues.
                let conflicts = if tx_pool.enable_rbf() {
                    tx_pool.check_rbf(&snapshot, &entry)?
                } else {
                    // RBF is disabled but we found conflicts, return error here
                    // after_process will put this tx into conflicts_pool
                    let conflicted_outpoint =
                        tx_pool.pool_map.find_conflict_outpoint(entry.transaction());
                    if let Some(outpoint) = conflicted_outpoint {
                        return Err(Reject::Resolve(OutPointError::Dead(outpoint)));
                    }
                    HashSet::new()
                };

                // if snapshot changed by context switch we need redo time_relative verify
                let tip_hash = snapshot.tip_hash();
                if pre_resolve_tip != tip_hash {
                    debug!(
                        "submit_entry {} context changed. previous:{} now:{}",
                        entry.proposal_short_id(),
                        pre_resolve_tip,
                        tip_hash
                    );

                    // destructuring assignments are not currently supported
                    status = check_rtx(tx_pool, &snapshot, &entry.rtx)?;

                    let tip_header = snapshot.tip_header();
                    let tx_env = status.with_env(tip_header);
                    time_relative_verify(snapshot, Arc::clone(&entry.rtx), tx_env)?;
                }

                let may_recovered_txs = self.process_rbf(tx_pool, &entry, &conflicts);
                let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?;

                // in a corner case, a tx with lower fee rate may be rejected immediately
                // after inserting into pool, return proper reject error here
                for evict in evicted {
                    let reject = Reject::Invalidated(format!(
                        "invalidated by tx {}",
                        evict.transaction().hash()
                    ));
                    self.callbacks.call_reject(tx_pool, &evict, reject);
                }

                tx_pool.remove_conflict(&entry.proposal_short_id());
                tx_pool
                    .limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
                    .map_or(Ok(()), Err)?;

                if !may_recovered_txs.is_empty() {
                    let self_clone = self.clone();
                    tokio::spawn(async move {
                        // push the recovered txs back to verify queue, so that they can be verified and submitted again
                        let mut queue = self_clone.verify_queue.write().await;
                        for tx in may_recovered_txs {
                            debug!("recover back: {:?}", tx.proposal_short_id());
                            let _ = queue.add_tx(tx, false, None);
                        }
                    });
                }
                Ok(())
            })
            .await;

        (ret, snapshot)
    }
