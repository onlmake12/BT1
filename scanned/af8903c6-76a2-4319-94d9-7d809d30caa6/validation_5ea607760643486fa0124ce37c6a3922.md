### Title
Copy-Paste Bug in `script_exists_in_output` Causes Incorrect Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

A copy-paste error at line 252 of `script_exists_in_output` causes the function to re-read `row_lock` (the lock-script EXISTS result) instead of `row_type` (the type-script EXISTS result) when running against PostgreSQL. Because execution only reaches line 252 when `row_lock` is already `false`, the function unconditionally returns `false` for the type-script check on PostgreSQL. During `rollback_block`, this causes any script that is referenced **only** as a `type_script_id` in surviving outputs to be incorrectly deleted from the `script` table, permanently corrupting the rich-indexer's state for those cells.

### Finding Description

`script_exists_in_output` is supposed to return `true` if `script_id` is still referenced by any surviving output row — either as a `lock_script_id` or as a `type_script_id`. The function issues two separate SQL `EXISTS` queries and stores their results in `row_lock` and `row_type` respectively.

The first match block (lines 223–235) correctly short-circuits with `Ok(true)` if `row_lock` is true. If not, execution falls through to fetch `row_type`. The second match block at line 252 is supposed to evaluate `row_type`, but instead reads `row_lock` again:

```rust
// Correct first check (lines 222-235): reads row_lock
match row_lock.try_get::<bool, _>(0) { ... }

// row_type is fetched (lines 237-249)
let row_type = sqlx::query("SELECT EXISTS (... WHERE type_script_id = $1 ...)") ...

// BUG at line 252: reads row_lock again instead of row_type
match row_lock.try_get::<bool, _>(0) {   // <-- should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

**PostgreSQL path:** `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`). Since we only reach line 252 when `row_lock` is `false` (the early return at line 226 would have fired otherwise), this always returns `Ok(false)` — the type-script check is completely bypassed.

**SQLite path:** `row_lock.try_get::<bool, _>(0)` fails (SQLite `EXISTS` returns `BIGINT`), so the `Err(_)` branch executes `Ok(row_type.get::<i64, _>(0) == 1)`, which correctly reads `row_type`. SQLite is unaffected.

The caller in `rollback_block` (lines 33–37) uses the return value to decide whether to add `type_script_id` to the deletion list:

```rust
if let Some(type_script_id) = type_script_id
    && !script_exists_in_output(type_script_id, tx).await?
{
    script_id_list_to_remove.push(type_script_id);
}
```

On PostgreSQL, `script_exists_in_output(type_script_id)` always returns `false` when the script is not also a `lock_script_id` in any surviving output, so the script row is unconditionally added to the deletion list and removed from the `script` table even when surviving outputs still reference it via `type_script_id`.

### Impact Explanation

After the incorrect deletion, the `script` table no longer contains the row for the shared type-script. Any subsequent `get_cells` or `get_transactions` RPC call that joins against the `script` table using that `type_script_id` will produce NULL joins or missing script data for all surviving cells from prior blocks that reference that type-script. The rich-indexer's view of those cells is permanently broken until the indexer is fully re-synced from scratch.

This affects only PostgreSQL-backed rich-indexer deployments. SQLite deployments are not affected due to the type-mismatch fallback.

### Likelihood Explanation

Chain reorganizations are a normal part of CKB network operation and require no special privileges to trigger — any peer can relay a valid competing chain. The only structural requirement is that the rolled-back block contains an output whose `type_script_id` is shared with a surviving output in a prior block but whose `lock_script_id` is distinct (so the type-script is not also a lock-script in any surviving output). This is a routine scenario in practice (e.g., UDT token cells with a shared type-script but different owner lock-scripts across blocks).

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // was: row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

### Proof of Concept

1. Run a PostgreSQL-backed rich-indexer node.
2. Append block 0 containing output A with `type_script = T`, `lock_script = L1`.
3. Append block 1 containing output B with `type_script = T`, `lock_script = L2` (same T, different lock).
4. Trigger a reorg that rolls back block 1 (`rollback_block` executes).
5. After rollback, query the `script` table: the row for T is gone (incorrectly deleted).
6. Call `get_cells` with `script = T`: returns empty or broken results despite output A from block 0 surviving.
7. Expected: T's script row is preserved; `get_cells` returns output A. [1](#0-0) [2](#0-1) [3](#0-2)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }
