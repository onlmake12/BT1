### Title
Unbounded `GetHeaders` Message Processing Without Rate Limiting Enables CPU/IO Exhaustion DoS — (File: `sync/src/synchronizer/get_headers_process.rs`)

---

### Summary

The CKB Synchronizer protocol handler processes `GetHeaders` messages from any connected peer without any per-peer rate limiting. Each message triggers up to 2,101 synchronous RocksDB read operations and blocks a tokio worker thread via `block_in_place`. An unprivileged inbound peer can flood the node with `GetHeaders` messages to exhaust I/O bandwidth and CPU, causing a denial of service. This is structurally analogous to the HTTP/2 CONTINUATION frame attack: both exploit the absence of a sufficient limit on how much work a single peer can force the receiver to perform per unit of time.

---

### Finding Description

The `Relayer` protocol handler explicitly installs a per-peer, per-message-type rate limiter capped at 30 requests per second:

```rust
// sync/src/relayer/mod.rs lines 91-92
let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
let rate_limiter = RateLimiter::hashmap(quota);
``` [1](#0-0) 

The `Synchronizer` protocol handler has no equivalent rate limiter. Its `try_process` dispatches `GetHeaders` directly:

```rust
// sync/src/synchronizer/mod.rs lines 397-400
packed::SyncMessageUnionReader::GetHeaders(reader) => {
    tokio::task::block_in_place(|| {
        GetHeadersProcess::new(reader, self, peer, &nc).execute()
    })
}
``` [2](#0-1) 

Inside `GetHeadersProcess::execute()`, after a single size check (`locator_size > MAX_LOCATOR_SIZE = 101`), the handler performs two expensive operations for every message:

1. `locate_latest_common_block` — iterates through up to 101 locator hashes with DB lookups to find the common ancestor.
2. `get_locator_response` — reads up to `MAX_HEADERS_LEN = 2,000` block headers sequentially from RocksDB:

```rust
// sync/src/types/mod.rs lines 1914-1920
std::iter::successors(Some(start_number), |number| number.checked_add(1))
    .take_while(|number| *number <= tip_number)
    .take(MAX_HEADERS_LEN)
    .filter_map(|block_number| self.snapshot.get_block_hash(block_number))
    .take_while(|block_hash| block_hash != hash_stop)
    .filter_map(|block_hash| self.sync_shared.store().get_block_header(&block_hash))
    .collect()
``` [3](#0-2) 

The constants confirm the per-message work budget: [4](#0-3) [5](#0-4) 

Because `block_in_place` is used, each `GetHeaders` message blocks a tokio worker thread for the duration of all DB reads. Multiple peers sending concurrent floods can saturate the thread pool.

The size check only bounds the *locator* input, not the *rate* of messages: [6](#0-5) 

---

### Impact Explanation

A single malicious inbound peer sending `GetHeaders` messages at the maximum rate the TCP connection allows forces the victim node to perform up to ~2,101 RocksDB reads per message. With no rate limiter, the attacker can sustain this indefinitely. The combined effect is:

- **I/O exhaustion**: RocksDB read throughput is saturated, starving legitimate block/transaction processing.
- **CPU exhaustion**: `block_in_place` blocks tokio worker threads; concurrent floods from multiple peers can exhaust the thread pool, stalling all async tasks including block relay, chain processing, and RPC responses.
- **Node unavailability**: The node becomes unresponsive to legitimate peers and miners.

---

### Likelihood Explanation

Any peer that can establish a TCP connection to the CKB node can send `GetHeaders` messages. No authentication, no mining power, and no privileged role is required. The attack is trivially scriptable: craft a `GetHeaders` message with a 101-entry locator pointing to genesis (all hashes known from the public chain), and send it in a tight loop. The contrast with the Relayer's explicit 30 req/sec cap confirms the Synchronizer's exposure was not intentionally designed.

---

### Recommendation

Apply a per-peer rate limiter to the Synchronizer's `try_process` dispatch, mirroring the pattern already used in the Relayer. A limit of 10–30 `GetHeaders` messages per second per peer is sufficient for legitimate sync while eliminating the flood vector. Additionally, consider caching the `get_locator_response` result for a short TTL keyed by `(block_number, hash_stop)` to amortize repeated identical queries from different peers.

---

### Proof of Concept

1. Connect to a fully-synced CKB mainnet node as an inbound peer using the Sync protocol.
2. Construct a `GetHeaders` (`SyncMessage::GetHeaders`) molecule-encoded message with `block_locator_hashes` set to 101 copies of the genesis block hash and `hash_stop` set to `Byte32::zero()`.
3. Send this message in a tight loop at maximum TCP throughput.
4. **Expected result**: Each message causes the node to execute `locate_latest_common_block` (101 DB lookups) and `get_locator_response` (up to 2,000 DB reads), blocking a tokio worker thread via `block_in_place` for the duration. Sustained flooding saturates RocksDB I/O and the tokio thread pool, causing the node to stop processing blocks, relaying transactions, and responding to RPC calls.
