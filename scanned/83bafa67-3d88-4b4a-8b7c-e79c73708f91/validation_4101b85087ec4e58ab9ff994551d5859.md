### Title
Peer-Controlled Timestamp Causes Silent `i64` Truncation in `NetTimeProtocol::received`, Bypassing Clock-Skew Detection — (`File: sync/src/net_time_checker.rs`)

---

### Summary

In `NetTimeProtocol::received`, the time offset between the local clock and a peer-supplied timestamp is computed via an `i128` subtraction and then **silently truncated** to `i64` with an `as i64` cast. A malicious outbound peer can send a crafted `timestamp` value that makes the `i128` difference fall outside the `i64` range, causing the stored sample to wrap to an arbitrary (possibly small or zero) value. This defeats the `NetTimeChecker::check()` guard that is supposed to warn the operator when the node's clock is badly skewed.

---

### Finding Description

In `sync/src/net_time_checker.rs`, line 156:

```rust
let offset: i64 = (i128::from(now) - i128::from(timestamp)) as i64;
```

`now` and `timestamp` are both `u64` millisecond values. Their difference is computed in `i128` to avoid overflow, but the result is then **cast to `i64` with `as`**, which is a wrapping/truncating cast in Rust — it silently discards the high bits when the value is outside `[-2^63, 2^63-1]`.

A peer can send any `u64` timestamp it likes. If a peer sends `timestamp = 0` (or any value far in the future relative to `now`), the `i128` difference can be as large as `now` (up to ~`1.8 × 10^19`), which far exceeds `i64::MAX` (~`9.2 × 10^18`). The `as i64` cast wraps this into a small or negative value, so the sample stored in `NetTimeChecker` does not reflect the true offset.

The `check()` function at line 63 then calls `network_offset.unsigned_abs()` on the corrupted sample and compares it to `TOLERANT_OFFSET` (7,200,000 ms). Because the sample was silently wrapped, the check passes even though the true offset is enormous.

The analog to the report is exact: just as `unsafe_abs` removed the `require(x != MIN_64x64)` guard and allowed a silent wrap-around on negation, here the `as i64` cast removes the bounds check on the subtraction result, allowing a silent wrap-around that defeats the downstream safety check.

---

### Impact Explanation

The `NetTimeChecker` is the only mechanism CKB uses to warn operators that their node's local clock is significantly skewed relative to the network. A skewed clock can cause:

- The node to reject valid blocks whose timestamps appear "too new" relative to the local clock.
- The node to accept blocks with timestamps that are actually in the future.
- Disruption of time-locked transaction (`since`) verification.

By sending a crafted timestamp, a single malicious outbound peer can suppress the clock-skew warning entirely, leaving the operator unaware that their node is operating with a dangerously wrong clock.

---

### Likelihood Explanation

Any outbound peer the node connects to can send an arbitrary `timestamp` in the `Time` protocol message. No authentication or privilege is required. The `received` handler accepts the raw `u64` timestamp from the peer message without range validation before computing the offset. The condition is trivially triggerable by sending `timestamp = 0` or `timestamp = u64::MAX`.

---

### Recommendation

Replace the silent `as i64` cast with a checked conversion that clamps or rejects out-of-range values:

```rust
let diff: i128 = i128::from(now) - i128::from(timestamp);
let offset: i64 = diff.clamp(i64::MIN as i128, i64::MAX as i128) as i64;
// or: reject the sample entirely if |diff| > some reasonable bound
```

Alternatively, validate that the peer-supplied `timestamp` is within a plausible range before computing the offset, and discard (or ban) peers that send wildly out-of-range timestamps.

---
