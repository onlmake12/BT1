### Title
Ping Nonce Derived from Elapsed Seconds Enables Pong Replay by Malicious Peer - (File: `network/src/protocols/ping.rs`)

---

### Summary

The `nonce()` function in the CKB ping protocol computes the challenge nonce as elapsed **whole seconds** since node start. This is a non-unique, time-bounded value — directly analogous to using `block.timestamp` as a salt. If `ping_peers()` is invoked twice within the same wall-clock second (via the periodic timer and the control channel simultaneously), both invocations produce the identical nonce. A malicious connected peer can exploit this to replay a previously captured pong, satisfying the second ping challenge without actually being live, thereby evading timeout-based disconnection.

---

### Finding Description

In `network/src/protocols/ping.rs`, the nonce sent in every ping message is computed as:

```rust
fn nonce(t: &Instant, start_time: Instant) -> u32 {
    t.saturating_duration_since(start_time).as_secs() as u32
}
``` [1](#0-0) 

This value is the number of whole seconds elapsed since the `PingHandler` was constructed. It has **second-level granularity**, meaning any two calls to `ping_peers()` that occur within the same calendar second produce the same nonce.

`ping_peers()` is reachable from two independent code paths:

1. **`notify()` → `SEND_PING_TOKEN`** — the periodic timer fires every `ping_interval_secs` seconds.
2. **`poll()` → control channel** — the `ping_controller` sender can trigger an immediate ping at any time. [2](#0-1) 

The `ping_controller` (`Sender<()>`) is stored in `NetworkService` and is accessible to internal subsystems: [3](#0-2) 

When `ping_peers()` runs, it sets `ps.processing = true` and `ps.nonce = send_nonce` for every peer not already processing, then broadcasts the ping: [4](#0-3) 

The pong validation accepts any pong whose nonce matches the stored `ps.nonce`:

```rust
if let Some(status) = self.connected_session_ids.get_mut(&session.id)
    && (true, nonce) == (status.processing, status.nonce())
``` [5](#0-4) 

**Collision scenario:**

1. Timer fires at second `T` → `ping_peers()` → `nonce = T`, peer A: `processing=true`, `nonce=T`, `ping(T)` sent.
2. Peer A immediately sends `pong(T)` → `processing=false`.
3. Control channel fires within the same second `T` → `ping_peers()` → `nonce = T` (identical), peer A: `processing=true`, `nonce=T`, `ping(T)` sent again.
4. Peer A replays the already-sent `pong(T)` → check `(true, T) == (true, T)` passes → accepted.

The default production `ping_interval_secs = 120` means normal consecutive rounds differ by 120 nonce units, so no collision under steady-state. The collision window opens when the control channel fires within the same second as the periodic timer. [6](#0-5) 

---

### Impact Explanation

A malicious peer that pre-sends `pong(N)` immediately after receiving `ping(N)` can replay that same `pong(N)` to satisfy a subsequent `ping(N)` issued within the same second. This means:

- **Timeout evasion**: The timeout check disconnects peers where `ps.processing && ps.elapsed() >= timeout`. By replaying a pong, the peer resets `processing=false` and `last_ping_sent_at`, preventing the timeout from triggering. A peer that is actually unresponsive (e.g., a zombie connection) can maintain its slot indefinitely.
- **RTT corruption**: `peer.ping_rtt` is set to `now - last_ping_sent_at` of the second ping, but the pong was pre-prepared for the first ping, producing a falsely low RTT.
- **Peer slot exhaustion**: By keeping zombie connections alive, an attacker can occupy peer slots, degrading the node's ability to connect to honest peers.

---

### Likelihood Explanation

The collision requires the control channel to fire within the same second as the periodic timer. The `ping_controller` is held by `NetworkService` and is triggered by internal subsystems. Under load or during rapid peer churn, both paths can fire within the same second. Additionally, operators or test environments may configure `ping_interval_secs = 1` (as seen in `rpc/src/tests/setup.rs`), making consecutive timer-only rounds collide trivially. [7](#0-6) 

The attacker only needs to be a connected P2P peer — no privileges, keys, or majority hashpower required.

---

### Recommendation

Replace the time-derived nonce with a monotonically incrementing counter stored in `PingHandler`. A simple `u32` counter incremented on each call to `ping_peers()` guarantees uniqueness across all invocations regardless of timing:

```rust
// In PingHandler struct:
nonce_counter: u32,

// In ping_peers():
self.nonce_counter = self.nonce_counter.wrapping_add(1);
let send_nonce = self.nonce_counter;
```

This eliminates the second-granularity collision window entirely and is the direct analog of the fix recommended in the source report.

---

### Proof of Concept

1. Connect a malicious peer to a CKB node.
2. Receive `ping(N)` where `N = elapsed_secs_since_start`.
3. Immediately send `pong(N)` → node sets `processing=false`.
4. Trigger the `ping_controller` channel (or wait for a second timer tick within the same second) → node calls `ping_peers()` again with the same `N` → sends `ping(N)` again.
5. Send `pong(N)` again (replay) → node accepts it, resets `processing=false`, updates `last_ping_sent_at`.
6. Repeat step 4–5 indefinitely to prevent the timeout check from ever firing, keeping the zombie connection alive.

    async fn ping_peers(&mut self, context: &ProtocolContext) {
        let now = Instant::now();
        let send_nonce = nonce(&now, self.start_time);
        let peers: HashSet<SessionId> = self
            .connected_session_ids
            .iter_mut()
            .filter_map(|(session_id, ps)| {
                if ps.processing {
                    None
                } else {
                    ps.processing = true;
                    ps.last_ping_sent_at = now;
                    ps.nonce = send_nonce;
                    Some(*session_id)
                }
            })
            .collect();
        if !peers.is_empty() {
            debug!("start ping peers: {:?}", peers);
            let ping_msg = PingMessage::build_ping(send_nonce);
            let proto_id = context.proto_id;
            if context
                .filter_broadcast(
                    TargetSession::Multi(Box::new(peers.into_iter())),
                    proto_id,
                    ping_msg,
                )
                .await
                .is_err()
            {
                debug!("Failed to send message");
            }
        }
    }
```

    async fn notify(&mut self, context: &mut ProtocolContext, token: u64) {
        match token {
            SEND_PING_TOKEN => self.ping_peers(context).await,
            CHECK_TIMEOUT_TOKEN => {
                let timeout = self.timeout;
                for (id, _ps) in self
                    .connected_session_ids
                    .iter()
                    .filter(|(_id, ps)| ps.processing && ps.elapsed() >= timeout)
                {
                    debug!("Ping timeout, {:?}", id);
                    if let Err(err) =
                        async_disconnect_with_message(context.control(), *id, "ping timeout").await
                    {
                        debug!("Disconnect failed {:?}, error: {:?}", id, err);
                    }
                }
            }
            _ => panic!("unknown token {token}"),
        }
    }

    async fn poll(&mut self, context: &mut ProtocolContext) -> Option<()> {
        if self.control_receiver.next().await.is_some() {
            self.ping_peers(context).await;
            Some(())
        } else {
            None
        }
    }
