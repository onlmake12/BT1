### Title
Peer Store Eviction Deadlock via Phase-2 `addrs.len() > 4` Off-by-One — (`network/src/peer_store/peer_store_impl.rs`)

### Summary

The `check_purge` function in `PeerStore` contains a logic flaw in its phase-2 eviction path: the guard `addrs.len() > 4` uses strict greater-than, and the group selector uses integer-division `take(len / 2)`. An attacker who fills the peer store to `ADDR_COUNT_LIMIT` (16 384) with addresses distributed across ≥ 4 096 distinct network groups of **exactly 4 entries each** causes phase-2 to produce zero eviction candidates, returning `PeerStoreError::EvictionFailed` on every subsequent `add_addr` call. This permanently prevents the node from learning new peer addresses via P2P discovery or identify, causing progressive network isolation.

---

### Finding Description

**Root cause — `check_purge` phase-2:** [1](#0-0) 

The two compounding defects are:

1. **`take(len / 2)` integer truncation** — when `len` is even (e.g. 4 096), exactly half the groups are selected. All selected groups have the same size (4), so none pass the next guard.
2. **`addrs.len() > 4` strict inequality** — groups of exactly 4 are silently skipped. The comment in the code says "more than 4 peer", but the intended semantic is "4 or more". A group of exactly 4 never yields eviction candidates. [2](#0-1) 

When both conditions align, `candidate_peers` is empty and the function returns: [3](#0-2) 

**Why freshly-advertised addresses are all connectable:**

`add_addr` stores entries with `last_connected_at_ms = 0` and `attempts_count = 0`. [4](#0-3) 

`is_connectable` returns `true` for these because `attempts_count (0) < ADDR_MAX_RETRIES (3)`: [5](#0-4) 

So phase-1 finds nothing to evict, and phase-2 runs with all 16 384 entries intact.

**Attack entry point — P2P Discovery protocol:**

`DiscoveryAddressManager::add_new_addrs` calls `peer_store.add_addr` for every address received in a `Nodes` message: [6](#0-5) 

The `EvictionFailed` error is silently swallowed (debug-logged only). The same path exists in `IdentifyProtocol::add_remote_listen_addrs`: [7](#0-6) 

An attacker connects to the victim node and sends `Nodes` messages advertising addresses from 4 096+ distinct `/24` network groups, 4 addresses per group. `MAX_ADDR_TO_SEND = 1000` per message means ~17 messages suffice to fill the store. Once full, every subsequent `add_addr` call returns `Err(EvictionFailed)`. [8](#0-7) 

---

### Impact Explanation

Once the peer store is saturated in the adversarial configuration:
- All new peer addresses received via discovery or identify are silently dropped.
- The node cannot expand its peer set beyond currently connected peers.
- When existing connections drop (natural churn, restarts), the node cannot find replacements, leading to progressive network isolation.
- The node cannot sync new blocks or transactions from peers it has never connected to.

The node process itself does not crash — the error is swallowed — but its ability to participate in peer discovery is permanently disabled until the store is manually cleared or entries naturally expire (7-day `ADDR_TIMEOUT_MS` timeout, only if the attacker stops refreshing). [9](#0-8) 

---

### Likelihood Explanation

- **No special privilege required**: any peer that can establish a P2P connection can send `Nodes` messages.
- **Low bandwidth cost**: 16 384 addresses at ~50 bytes each ≈ 800 KB total, spread across ~17 messages.
- **Addresses need not be real**: the discovery protocol does not verify that advertised IPs are reachable before storing them.
- **Maintenance**: the attacker must periodically re-advertise addresses to prevent natural expiry, but this is trivial.
- **Partial mitigation**: the node's own feeler connections will eventually mark some entries non-connectable (after 3 failed attempts), which would allow phase-1 to evict them. However, the attacker can continuously re-fill those slots faster than the node can exhaust them, since feeler connections are rate-limited.

---

### Recommendation

Fix both defects in `check_purge` phase-2:

1. Change `addrs.len() > 4` to `addrs.len() >= 4` (or `> 3`) so groups of exactly 4 are eligible for eviction.
2. Change `take(len / 2)` to `take((len + 1) / 2)` (ceiling division) to avoid truncating odd-length group lists.
3. Add a fallback: if phase-2 still produces zero candidates (e.g. all groups have ≤ 3 entries), evict the oldest-tried entry unconditionally rather than returning `EvictionFailed`. [2](#0-1)

### Proof of Concept

```rust
#[test]
fn test_eviction_deadlock_4_per_group() {
    let mut peer_store = PeerStore::default();
    // Fill with 4096 groups × 4 addresses each = 16384 total
    // Each group is a distinct /24 (first two octets vary)
    let mut count = 0;
    'outer: for a in 1u8..=255 {
        for b in 0u8..=255 {
            for c in 1u8..=4 {
                if count >= 16384 { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/{}.{}.1.{}/tcp/8115/p2p/{}",
                    a, b, c, PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(peer_store.addr_manager().count(), 16384);

    // 16385th add_addr must fail with EvictionFailed
    let extra: Multiaddr = format!(
        "/ip4/10.0.0.1/tcp/8115/p2p/{}", PeerId::random().to_base58()
    ).parse().unwrap();
    let result = peer_store.add_addr(extra, Flags::COMPATIBILITY);
    assert!(result.is_err(), "Expected EvictionFailed, got Ok");
}
```
