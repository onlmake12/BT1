### Title
Incorrect Malformed-Transaction Classification for `MismatchedVersion` Allows Unbounded Submission of Permanently-Invalid Transactions Without Peer Ban - (File: `util/types/src/core/error.rs`)

---

### Summary

`TransactionError::MismatchedVersion` is incorrectly classified as a non-malformed transaction error in `TransactionError::is_malformed_tx()`. Because this classification drives the peer-banning decision in the tx-pool, a remote peer can relay an unlimited number of permanently-invalid (wrong-version) transactions without ever being banned.

---

### Finding Description

In `util/types/src/core/error.rs`, the `is_malformed_tx()` method partitions `TransactionError` variants into two groups: those that indicate a structurally malformed transaction (returns `true`) and those that do not (returns `false`).

`TransactionError::MismatchedVersion` is placed in the **non-malformed** group: [1](#0-0) 

However, a transaction with a wrong version number is **permanently and unconditionally invalid** — it can never become valid regardless of chain state, time, or epoch. This is structurally identical to `TransactionError::Empty`, `TransactionError::DuplicateCellDeps`, or `TransactionError::OutputsSumOverflow`, all of which are correctly classified as malformed.

By contrast, `TransactionError::Immature` and `TransactionError::CellbaseImmaturity` are correctly classified as non-malformed because those transactions *can* become valid later (they are time/epoch-gated).

`MismatchedVersion` is checked inside `NonContextualTransactionVerifier`, which is part of the non-contextual verification path: [2](#0-1) 

The `VersionVerifier` returns `MismatchedVersion` when the transaction version does not match `consensus.tx_version()` (currently `0`): [3](#0-2) 

In the tx-pool, `non_contextual_verify` calls `ban_malformed` only when `reject.is_malformed_tx()` returns `true`: [4](#0-3) 

Because `MismatchedVersion` returns `false` from `is_malformed_tx()`, the ban is never triggered. The same logic applies in `after_process`: [5](#0-4) 

Additionally, `is_allowed_relay()` returns `true` for non-malformed, non-`LowFeeRate` rejections: [6](#0-5) 

This means the node also sends unnecessary relay-rejection notifications back to the peer for each wrong-version transaction.

---

### Impact Explanation

A malicious peer can relay an unbounded stream of transactions with `version != 0`. Each transaction passes through `non_contextual_verify` (fast, but not free), is rejected with `MismatchedVersion`, and the peer is never banned. The peer can repeat this indefinitely, consuming CPU and network resources on the victim node. The incorrect classification also causes the node to emit spurious relay-rejection messages back to the attacker.

**Impact: 3 / 5** — Resource exhaustion (DoS) potential; no consensus or fund-safety impact.

---

### Likelihood Explanation

Exploiting this requires a peer to deliberately send wrong-version transactions. The relay protocol (`RelayTransactions`) is the natural entry point. No special privileges are needed — any connected peer can do this.

**Likelihood: 2 / 5** — Requires a deliberately malicious peer; the non-contextual check is fast, limiting per-transaction cost.

---

### Recommendation

Move `TransactionError::MismatchedVersion` from the `false` arm to the `true` arm in `is_malformed_tx()`:

```rust

### Proof of Concept

**Attack path:**

1. Attacker connects to a CKB node as a peer via the relay protocol.
2. Attacker sends `RelayTransactionHashes` for a transaction hash, triggering a `GetRelayTransactions` response.
3. Attacker replies with `RelayTransactions` containing a transaction with `version = 1` (or any value ≠ 0).
4. The node calls `non_contextual_verify` → `NonContextualTransactionVerifier::verify()` → `VersionVerifier::verify()`.
5. `TransactionError::MismatchedVersion` is returned, wrapped as `Reject::Verification(...)`.
6. `reject.is_malformed_tx()` returns `false` (incorrect classification).
7. `ban_malformed` is **not** called; the peer remains connected.
8. Attacker repeats from step 2 indefinitely.

The existing unit test in `util/types/src/core/tests/tx_pool.rs` confirms the current (incorrect) behavior — `MismatchedVersion` is tested and asserted to produce `is_malformed = false`: [8](#0-7)
