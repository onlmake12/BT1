### Title
`ScriptHashTypeVerifier` Only Validates Lock Script Hash Type, Skips Type Script Hash Type — (`verification/src/transaction_verifier.rs`)

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and checks only the **lock script's** `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. It never inspects the **type script's** `hash_type`. A transaction output carrying a type script with a non-enabled hash type (e.g., `Data3 = 6`) silently passes the non-contextual gate and is only rejected later during expensive contextual script verification.

### Finding Description

`ScriptHashTypeVerifier` is the non-contextual, cheap gate that enforces the consensus rule "only enabled `ScriptHashType` values are permitted in transaction outputs." Its implementation is:

```rust
// verification/src/transaction_verifier.rs  L796-L814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

Only `output.lock().hash_type()` is examined. `output.type_().to_opt()` is never consulted. The enabled set is: [2](#0-1) 

`Data3` (value `6`) is structurally valid (even byte, passes `check_data()`/`verify_value()`) but is not in `ENABLED_SCRIPT_HASH_TYPE`. A transaction output whose **type script** carries `hash_type = Data3` passes `ScriptHashTypeVerifier` without error. [3](#0-2) 

The analogous check for the lock script is correct; the type-script branch is simply absent — exactly the same pattern as the reported bug where only the ERC-721 single-parameter `balanceOf()` was called instead of also dispatching to the ERC-1155 two-parameter variant.

### Impact Explanation

Any unprivileged RPC caller (`send_transaction`) can craft a transaction whose lock script uses a valid hash type (e.g., `Type = 1`) while its type script uses a non-enabled hash type (e.g., `Data3 = 6`). The transaction:

1. Passes `NonContextualTransactionVerifier` (cheap, O(outputs) scan).
2. Enters the contextual verification pipeline, where `TransactionScriptsVerifier` calls `select_version()` per script group.
3. `select_version()` hits the catch-all arm and returns `InvalidScriptHashType`. [4](#0-3) 

The transaction is ultimately rejected, but only after the node has built the full `TxInfo` structure (iterating cell deps, constructing binary hash maps) and entered the contextual verifier — work that should have been avoided by the non-contextual gate. An attacker can flood the node with such transactions at zero cost (rejected transactions pay no fees), forcing repeated unnecessary contextual-verification overhead.

### Likelihood Explanation

The attack requires only a valid RPC endpoint and the ability to construct a transaction with a structurally well-formed but semantically non-enabled type script hash type. No privileged access, key material, or majority hashpower is needed. The structural validity check (`verify_value`) accepts any even byte, so `Data3 = 6` passes deserialization and `check_data()` cleanly.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type when a type script is present:

```rust
// After the lock-script check, add:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

This mirrors the existing lock-script branch and closes the gap at the cheap non-contextual layer.

### Proof of Concept

1. Build a transaction with one output:
   - Lock script: `hash_type = Type (1)` — passes `ScriptHashTypeVerifier`.
   - Type script: `hash_type = Data3 (6)` — **not** in `ENABLED_SCRIPT_HASH_TYPE`, but never checked.
2. Submit via `send_transaction` RPC.
3. Observe: `NonContextualTransactionVerifier::verify()` returns `Ok(())`.
4. Observe: `ContextualTransactionVerifier::verify()` → `ScriptVerifier` → `select_version()` returns `Err(InvalidScriptHashType("Data3"))`.
5. Repeat in a tight loop — each iteration forces contextual-verification work with no fee cost to the attacker. [5](#0-4) [6](#0-5)
