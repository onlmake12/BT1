### Title
Expired Network Alert Replay via Missing Expiry Check in P2P Handler — (`util/network-alert/src/notifier.rs`, `util/network-alert/src/alert_relayer.rs`)

---

### Summary

The CKB network alert system contains a replay vulnerability: when an alert expires and is cleared from the `received_alerts` map, the node loses its memory of that alert ID. Any P2P peer who captured the original signed alert bytes can re-send them after expiry. The P2P `received()` handler performs no `notice_until` freshness check, so the node re-accepts the expired alert, re-notifies operators, and re-broadcasts it network-wide.

---

### Finding Description

The `Notifier` struct tracks received alerts in two structures:

- `received_alerts: HashMap<u32, Alert>` — alerts indexed by ID
- `cancel_filter: LruCache<u32, ()>` — cancelled alert IDs (size 128) [1](#0-0) 

Deduplication is enforced by `has_received()`: [2](#0-1) 

When `clear_expired_alerts()` is called (triggered on every new peer connection in `connected()`), expired alerts are removed from `received_alerts` and `noticed_alerts`, but their IDs are **never added to `cancel_filter`**: [3](#0-2) 

After this cleanup, `has_received(expired_id)` returns `false`. The P2P `received()` handler in `AlertRelayer` then checks `has_received()` as its only deduplication guard, and performs **no independent check that `notice_until` is in the future**: [4](#0-3) 

The `notifier.add()` path also performs no expiry check before calling `notify_controller.notify_network_alert()` and pushing to `noticed_alerts`: [5](#0-4) 

The `clear_expired_alerts()` is only called in `connected()`, not in `received()`: [6](#0-5) 

A secondary issue: `cancel_filter` is an LRU cache capped at 128 entries. If more than 128 distinct alert IDs are cancelled over the node's lifetime, the oldest cancelled IDs are silently evicted, allowing replay of those cancelled alerts as well. [7](#0-6) 

---

### Impact Explanation

An attacker who captures a valid signed alert from the P2P network (public traffic) can, after the alert's `notice_until` timestamp passes and any new peer connects to the victim node (triggering `clear_expired_alerts`):

1. Re-send the original alert bytes to the victim node over the P2P alert protocol.
2. The node re-accepts the alert (signatures are still cryptographically valid), re-adds it to `received_alerts`, calls `notify_controller.notify_network_alert()` (re-alarming operators), and re-broadcasts it to all connected peers.
3. This propagates network-wide, causing every node that has also cleared the alert to re-accept and re-broadcast it.

The result is false/confusing emergency alerts displayed to node operators across the network, and unnecessary alert traffic. Because the `notice_until` field is part of the signed payload, the replayed alert will show an expired timestamp — but the node still processes and re-broadcasts it.

---

### Likelihood Explanation

- **Attacker preconditions**: Must be a P2P peer (no privileged access) and must have captured a valid alert from the network. CKB alerts are broadcast to all peers, so any connected peer can capture them.
- **Trigger**: Wait for the alert to expire, then connect to (or already be connected to) a target node so `clear_expired_alerts()` runs, then send the captured alert bytes.
- **Difficulty**: Low. Alert traffic is public. The attacker needs no keys, no special role, and no on-chain resources.

---

### Recommendation

1. **In `received()` in `alert_relayer.rs`**: Add an explicit check that `alert.raw().notice_until() > current_time_ms` before accepting any P2P alert. Reject and optionally ban peers that send expired alerts.
2. **In `clear_expired_alerts()` in `notifier.rs`**: When removing an expired alert from `received_alerts`, insert its ID into `cancel_filter` so `has_received()` continues to return `true` for that ID.
3. **`cancel_filter` size**: Replace the fixed-size LRU with an unbounded set, or use a time-based eviction that only removes entries older than the maximum possible alert lifetime, to prevent cancelled-alert replay via LRU eviction.

---

### Proof of Concept

```
1. Attacker connects to the CKB P2P network as a normal peer.
2. A valid alert (id=42, notice_until=T) is broadcast. Attacker captures the raw bytes.
3. Time advances past T (alert expires).
4. Any new peer connects to the victim node, triggering `clear_expired_alerts()`.
   → `received_alerts.remove(42)` is called; cancel_filter is NOT updated.
   → `has_received(42)` now returns false.
5. Attacker sends the captured alert bytes to the victim node via the alert P2P protocol.
6. In `received()`:
   - `has_received(42)` → false (no dedup guard)
   - `verify_signatures()` → Ok (signatures are still valid)
   - `notifier.add(&alert)` is called
7. In `add()`:
   - `notify_controller.notify_network_alert(alert)` fires → operators see the expired alert
   - Alert is re-added to `noticed_alerts` and `received_alerts`
   - Alert is re-broadcast to all connected peers
8. Each peer that has also cleared the alert repeats steps 6–7, causing network-wide re-propagation.
```

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;
        }
        self.notify_controller.notify_network_alert(alert.clone());
        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }

    pub fn clear_expired_alerts(&mut self, now: u64) {
        self.received_alerts.retain(|_id, alert| {
            let notice_until: u64 = alert.raw().notice_until().into();
            notice_until > now
        });
        self.noticed_alerts.retain(|a| {
            let notice_until: u64 = a.raw().notice_until().into();
            notice_until > now
        });
    }
