Looking at the actual code to verify the claim about missing rate limiting on the Filter protocol.

The overflow claim (`u64::MAX - 2000*2000`) is a red herring — `checked_add` at line 52 correctly breaks the loop on overflow. The real issue is confirmed by the code: **the `BlockFilter` protocol handler has no rate limiter at all**.

### Title
Missing Per-Peer Rate Limiting on Filter Protocol Allows I/O-Based Resource Exhaustion — (`sync/src/filter/mod.rs`)

### Summary
The `BlockFilter` protocol handler processes all three request types (`GetBlockFilters`, `GetBlockFilterHashes`, `GetBlockFilterCheckPoints`) without any per-peer rate limiting. An unprivileged peer can flood the node with requests, each triggering up to 4000 RocksDB reads, causing sustained I/O amplification and degraded node performance.

### Finding Description
The `BlockFilter::received` handler in `sync/src/filter/mod.rs` dispatches directly to `self.process()` with no rate check: [1](#0-0) 

The `BlockFilter` struct itself has no `rate_limiter` field: [2](#0-1) 

This contrasts directly with `Relayer`, which carries a `rate_limiter: RateLimiter<(PeerIndex, u32)>` and checks it before every message dispatch: [3](#0-2) [4](#0-3) 

And `HolePunching`, which also rate-limits at the `received` level: [5](#0-4) 

Per request, `GetBlockFilterCheckPoints::execute` performs up to `BATCH_SIZE` (2000) iterations, each doing two DB reads (`get_block_hash` + `get_block_filter_hash`) — up to 4000 DB reads per request: [6](#0-5) 

`GetBlockFilterHashes` has the same 2000-iteration, 2-reads-per-iteration pattern: [7](#0-6) 

### Impact Explanation
A single peer can send an unbounded stream of `GetBlockFilterCheckPoints(start_number=0)` (or `GetBlockFilterHashes`/`GetBlockFilters`) messages. Each message causes up to 4000 RocksDB reads plus a response serialization and send. With no rate gate, this saturates the node's I/O and async task queue, degrading sync and relay performance for all other peers.

### Likelihood Explanation
The attack requires only a valid P2P connection and knowledge of the Filter protocol message format (publicly documented). No keys, hashpower, or privileged access are needed. The `GetBlockFilterCheckPoints` message is a 9-byte struct (`start_number: Uint64`), trivial to construct at high frequency.

### Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`) and check it at the top of `try_process` before dispatching any of the three request handlers. A quota of 30 requests/second per (peer, message-type) — matching the `Relayer` configuration — would be a reasonable starting point. [8](#0-7) 

### Proof of Concept
1. Connect a peer to a full node with the Filter protocol enabled.
2. In a tight loop, send `GetBlockFilterCheckPoints { start_number: 0 }` as fast as the TCP connection allows.
3. Observe RocksDB read IOPS spike and async task queue depth grow on the target node, with no ban or throttle applied to the sender.
