### Title
Orphan Pool Cleanup Bypass via Crafted `epoch_number` in Block Header — (`chain/src/utils/orphan_block_pool.rs`)

### Summary

The `need_clean` predicate in `InnerPool` uses plain `u64` addition without overflow protection. An attacker can craft a block with `epoch_number` set to `u64::MAX - EXPIRED_EPOCH` (or any large value in the non-overflowing range far beyond the current tip epoch), causing the cleanup condition to evaluate permanently false. Because non-contextual verification never range-checks the epoch field and PoW is not verified before orphan pool insertion, such blocks enter the pool and are never evicted, leading to unbounded memory and disk growth.

### Finding Description

**Root cause — arithmetic in `need_clean`:** [1](#0-0) 

The condition is:
```rust
lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
```
where `EXPIRED_EPOCH = 6`. [2](#0-1) 

If `epoch_number = u64::MAX - 6`, then `(u64::MAX - 6) + 6 = u64::MAX` (no overflow), and `u64::MAX < tip_epoch` is **always false** since `tip_epoch` is a `u64` and can never exceed `u64::MAX`. The block is never evicted. Any value in the range `[tip_epoch - EXPIRED_EPOCH + 1, u64::MAX - EXPIRED_EPOCH]` achieves the same effect.

**Non-contextual verification does not check epoch:**

`ChainService::non_contextual_verify` calls only `BlockVerifier` and `NonContextualBlockTxsVerifier`: [3](#0-2) 

`BlockVerifier::verify` checks cellbase structure, block bytes, proposals limit, duplicates, and merkle root — **no epoch range check, no PoW check**: [4](#0-3) 

PoW verification requires the parent header (to compute expected difficulty) and is therefore deferred to contextual verification, which never runs for a block whose parent is absent.

**Orphan pool insertion path:**

`epoch_number` is read directly from the block header with no validation: [5](#0-4) 

A block with an unknown parent is inserted unconditionally into the orphan pool: [6](#0-5) 

**Cleanup timer:**

`clean_expired_orphans` runs every 60 seconds and calls `clean_expired_blocks(tip_epoch_number)`: [7](#0-6) [8](#0-7) 

Because `need_clean` returns false for the crafted blocks, they are never passed to `delete_block` or `remove_header_view`.

**No hard capacity limit:**

`OrphanBlockPool::with_capacity` sets only an initial `HashMap` hint, not a hard eviction ceiling: [9](#0-8) 

### Impact Explanation

Each crafted block occupies memory in the orphan pool's `blocks`, `parents`, and `leaders` maps, and is also written to the on-disk `ChainDB` as an unverified block. An attacker sending a sustained stream of such blocks causes unbounded memory and disk exhaustion, crashing or severely degrading the node.

### Likelihood Explanation

The attack requires only the ability to send P2P block messages — no privileged access, no valid PoW, no majority hashpower. A single attacker with a standard internet connection can generate and send structurally valid (non-contextually) blocks with arbitrary header fields at high rate. The 60-second cleanup timer provides no relief since the crafted blocks are immune to it.

### Recommendation

1. **Use saturating addition** in `need_clean`:
   ```rust
   lonely_block.epoch_number().saturating_add(EXPIRED_EPOCH) < tip_epoch
   ```
   This makes any `epoch_number` near `u64::MAX` evaluate as `u64::MAX < tip_epoch` → false, but also consider treating overflow as "expired" (use `checked_add` and return `true` on overflow, since a block claiming an epoch near `u64::MAX` is clearly invalid).

2. **Enforce a hard capacity limit** on the orphan pool with LRU or FIFO eviction when the limit is reached.

3. **Reject blocks with `epoch_number` implausibly far from the current tip** in `non_contextual_verify` or at the P2P ingress layer (e.g., reject if `epoch_number > tip_epoch + MAX_FUTURE_EPOCH_TOLERANCE`).

### Proof of Concept

```rust
// Craft a block with epoch_number = u64::MAX - EXPIRED_EPOCH (= u64::MAX - 6)
// Parent hash = random unknown hash
// All other fields satisfy non_contextual_verify (valid cellbase, merkle root, etc.)
// No valid PoW required.

// Send via P2P block relay to target node.
// After 60s, assert orphan pool still contains the block.
// Repeat to grow pool without bound.

// Arithmetic proof:
let epoch_number: u64 = u64::MAX - 6;
let expired_epoch: u64 = 6;
let tip_epoch: u64 = 100; // any realistic value
assert!(!((epoch_number + expired_epoch) < tip_epoch));
// (u64::MAX - 6) + 6 = u64::MAX; u64::MAX < 100 → false → block never cleaned
```

    pub(crate) fn clean_expired_orphans(&self) {
        debug!("clean expired orphans");
        let tip_epoch_number = self
            .shared
            .store()
            .get_tip_header()
            .expect("tip header")
            .epoch()
            .number();
        let expired_orphans = self
            .orphan_blocks_broker
            .clean_expired_blocks(tip_epoch_number);
        for expired_orphan in expired_orphans {
            self.delete_block(&expired_orphan);
            self.shared.remove_header_view(&expired_orphan.hash());
            self.shared.remove_block_status(&expired_orphan.hash());
            info!(
                "cleaned expired orphan: {}-{}",
                expired_orphan.number(),
                expired_orphan.hash()
            );
        }
    }
