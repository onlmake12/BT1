### Title
Unguarded `.expect()` in `PreloadUnverifiedBlocksChannel::load_full_unverified_block_by_hash` Causes Thread Panic and Permanent Block-Verification DoS — (`chain/src/preload_unverified_blocks_channel.rs`)

---

### Summary

A race condition between the `preload_unverified_block` thread and the `verify_blocks` thread allows an unprivileged P2P peer to permanently crash the `preload_unverified_block` thread. When a parent block P fails contextual verification and is deleted from the database, a child block B already queued in the preload channel will trigger an unguarded `.expect("parent header stored")` panic. Because the preload thread has no `catch_unwind` guard and is never restarted, this drops the `unverified_block_tx` sender, causing the `verify_blocks` thread to exit on the next receive error, permanently halting all block verification on the node.

---

### Finding Description

**Thread architecture** (from `chain/src/init.rs`):

Three threads are spawned once and never restarted:
- `ChainService` — single-threaded; runs `process_lonely_block` and `clean_expired_orphans`
- `preload_unverified_block` — reads `LonelyBlockHash` from `preload_unverified_rx`, loads full block + parent header from DB, forwards `UnverifiedBlock` to `unverified_block_tx`
- `verify_blocks` — reads `UnverifiedBlock` from `unverified_block_rx`, runs contextual verification, deletes failed blocks from DB [1](#0-0) [2](#0-1) 

**The unguarded panic site** in `load_full_unverified_block_by_hash`:

```rust
let parent_header = {
    self.shared
        .store()
        .get_block_header(&parent_hash)
        .expect("parent header stored")   // ← panics if P was deleted
};
``` [3](#0-2) 

The `preload_unverified_block` thread's `start()` loop has **no `catch_unwind`**: [4](#0-3) 

By contrast, the `verify_blocks` thread wraps its processor call in `catch_unwind` — the preload thread was simply never given the same protection: [5](#0-4) 

**Deletion path for the parent block P** — `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks` calls `delete_unverified_block` when P fails contextual verification, then removes P from `is_pending_verify`: [6](#0-5) [7](#0-6) 

**Race condition — step by step:**

1. Attacker sends block **P** (valid non-contextual structure, invalid PoW or script). `ChainService::asynchronous_process_block` passes non-contextual verification, calls `insert_block` (P stored in DB), then `process_lonely_block`. P's grandparent is on-chain, so P is sent to `preload_unverified_rx` and added to `is_pending_verify`. [8](#0-7) [9](#0-8) 

2. Attacker sends block **B** (child of P, valid non-contextual structure). `process_lonely_block` sees P is in `is_pending_verify` → B is sent to `preload_unverified_rx` and added to `is_pending_verify`. [10](#0-9) 

3. The `preload_unverified_block` thread dequeues P first, loads it successfully, forwards it to `unverified_block_tx`.

4. The `verify_blocks` thread dequeues P, contextual verification fails (e.g., invalid PoW), calls `delete_unverified_block(P)` — **P's header is now gone from the DB** — then removes P from `is_pending_verify`. [11](#0-10) 

5. The `preload_unverified_block` thread dequeues B, calls `load_full_unverified_block_by_hash`, reaches `get_block_header(&parent_hash).expect("parent header stored")` → **returns `None` → panic**. [12](#0-11) 

**Cascade after the panic:**

- The `preload_unverified_block` thread dies, dropping `unverified_block_tx` (the only sender).
- The `verify_blocks` thread's next `recv(self.unverified_block_rx)` returns `Err` → logs the error and returns, terminating the thread. [13](#0-12) 

- `ChainService` continues running but no block can ever be verified again. The node is permanently stuck.

**Note on `clean_expired_orphans`:** The question's proposed deletion path via `clean_expired_orphans` is not the actual trigger. `clean_expired_orphans` only removes blocks still in the in-memory orphan pool; once B is sent to `preload_unverified_rx` it is removed from the orphan pool and cannot be expired. The real deletion path is `ConsumeUnverifiedBlockProcessor::delete_unverified_block` as described above. [14](#0-13) 

---

### Impact Explanation

A single unprivileged P2P peer can permanently halt all block verification on a target node with two crafted messages. The node continues to accept P2P connections and relay headers but can never advance its chain tip, constituting a complete consensus-layer DoS. Recovery requires a manual node restart.

---

### Likelihood Explanation

The attack requires no special privileges, no majority hashpower, and no key material. The attacker only needs to:
- Craft a block P with valid non-contextual structure but invalid contextual content (e.g., wrong PoW solution — trivially constructable)
- Craft a child block B pointing to P
- Send P then B in rapid succession to any reachable node

The timing window is widened by the bounded channel depth (`preload_unverified_rx` holds up to `BLOCK_DOWNLOAD_WINDOW * 10` items; `unverified_block_tx` holds 128). Under any moderate load, B will sit in the preload queue long enough for P to be verified and deleted. The attack is deterministically reproducible in a local two-node setup. [1](#0-0) 

---

### Recommendation

1. **Immediate fix:** Replace the `.expect()` calls in `load_full_unverified_block_by_hash` with graceful error handling. If the block or parent header is missing, log an error and skip (or return an `Err`) rather than panicking.

2. **Defense in depth:** Wrap the body of `PreloadUnverifiedBlocksChannel::preload_unverified_channel` in `std::panic::catch_unwind` (mirroring the pattern already used in `ConsumeUnverifiedBlocks::start`) so a panic does not kill the thread.

3. **Structural fix:** Before sending B to the preload channel, verify that P's header is still present in the DB, or pass the parent header by value at enqueue time (when it is known to be present) rather than re-fetching it later.

---

### Proof of Concept

```
1. Start a CKB node (node A).
2. Connect a test peer (node B).
3. From node B, craft block P:
   - Valid parent = current tip of node A
   - Valid non-contextual structure (correct merkle roots, valid tx structure)
   - Invalid PoW nonce (fails contextual PoW check)
4. From node B, craft block C (child of P):
   - parent_hash = hash(P)
   - Valid non-contextual structure
5. Send P to node A via SendBlock P2P message.
6. Immediately send C to node A via SendBlock P2P message.
7. Observe:
   - Node A inserts P into DB, sends P to preload channel, adds P to is_pending_verify.
   - Node A sees C's parent P is in is_pending_verify, sends C to preload channel.
   - verify_blocks thread processes P, fails PoW check, deletes P from DB.
   - preload_unverified_block thread processes C, panics at .expect("parent header stored").
   - verify_blocks thread exits on channel error.
   - Node A's chain tip is permanently frozen; no further blocks are verified.
8. Assert: node A's tip does not advance even when valid blocks are sent.
```
