### Title
Attacker Can DOS the Orphan Transaction Pool via Costless Invalid-Input Transactions - (`tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` in CKB's tx-pool is capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries and evicts entries **randomly** when full. There is no per-peer limit and no economic cost enforced before a transaction is admitted to the orphan pool. An unprivileged P2P peer can flood the pool with 100 structurally-valid transactions whose inputs reference non-existent UTXOs, permanently displacing legitimate orphan transactions at zero cost.

---

### Finding Description

`tx-pool/src/component/orphan.rs` defines the orphan pool with a hard cap: [1](#0-0) 

When the pool exceeds this limit, `limit_size()` evicts entries by calling `self.entries.keys().next()` — effectively random HashMap iteration order — with no fairness guarantee across peers: [2](#0-1) 

`add_orphan_tx()` performs no per-peer accounting before inserting: [3](#0-2) 

The path from the P2P relay layer to the orphan pool is:

1. A peer announces transaction hashes via the relay protocol (`RelayTransactionHashes`).
2. The node requests them; the peer responds with transactions whose inputs reference non-existent UTXOs.
3. `TransactionsProcess::execute` in `sync/src/relayer/transactions_process.rs` calls `submit_remote_tx`: [4](#0-3) 

4. `submit_remote_tx` → `resumeble_process_tx` → `non_contextual_verify` (structural checks only) → `enqueue_verify_queue`.
5. The verify worker calls `_process_tx`; `pre_check` → `resolve_tx` fails with `OutPointError::Unknown` (missing input).
6. `after_process` detects `is_missing_input` and calls `add_orphan`: [5](#0-4) 

The attacker's transactions are structurally valid (pass `non_contextual_verify`) but reference UTXOs that do not exist. No CKB needs to be locked up. The attacker can repeat this for 100 distinct transaction hashes, filling the pool entirely.

---

### Impact Explanation

Once the orphan pool is saturated with attacker-controlled entries, any legitimate orphan transaction submitted by an honest peer is randomly evicted. When the legitimate orphan's parent transaction later arrives and `process_orphan_tx` is called, the orphan is no longer in the pool and is silently dropped: [6](#0-5) 

The honest user's transaction is lost from the node's view. The user must detect the failure (e.g., by polling `get_transaction`) and resubmit. In a sustained attack, every resubmission is again evicted, making the node's orphan pool permanently unusable for legitimate traffic. This disrupts transaction propagation and can delay or prevent confirmation of transactions that depend on in-flight parents.

---

### Likelihood Explanation

The attack requires only a standard P2P peer connection — no privileged access, no funds, no majority hashpower. The attacker crafts 100 structurally-valid transactions with fabricated input `OutPoint`s (non-existent tx hashes), announces their hashes, and delivers them on request. The total bandwidth cost is negligible (100 small transactions). The attack can be sustained indefinitely by rotating transaction hashes. There is no banning triggered because the transactions are not malformed — they merely reference missing inputs, which is the normal condition for orphan admission.

---

### Recommendation

1. **Per-peer orphan limit**: Track how many orphan entries each `PeerIndex` has contributed. Evict the contributing peer's oldest entry first when the pool is full, rather than evicting randomly across all peers.
2. **Prioritized eviction**: When evicting, prefer entries from peers that have contributed the most orphans, making the attack self-limiting.
3. **Increase the cap with expiry enforcement**: `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` is very small. Combine a larger cap with aggressive expiry of entries whose parents have not arrived within a short window.
4. **Announce-before-relay validation**: Before adding a relayed transaction to the orphan pool, verify that at least one of its missing inputs was recently announced on the network (i.e., is plausibly in-flight), rather than accepting any missing-input transaction unconditionally.

---

### Proof of Concept

```
Attacker (P2P peer) → CKB node:

1. Send RelayTransactionHashes { hashes: [H1, H2, ..., H100] }
   (H_i = sha256(random_bytes), each unique)

2. Node responds: GetRelayTransactions { tx_hashes: [H1..H100] }

3. Attacker responds with RelayTransactions containing 100 transactions:
   Each tx_i:
     - inputs:  [{ previous_output: { tx_hash: <random 32 bytes>, index: 0 } }]
     - outputs: [{ capacity: 100 CKB, lock: always_success }]
     - witnesses: [<valid witness>]
   (Structurally valid; inputs reference non-existent UTXOs)

4. Each tx_i passes non_contextual_verify, enters verify_queue,
   fails resolve_tx with OutPointError::Unknown (is_missing_input),
   and is inserted into OrphanPool via add_orphan_tx().

5. After 100 insertions, OrphanPool.len() == DEFAULT_MAX_ORPHAN_TRANSACTIONS.
   Any subsequent legitimate orphan tx is randomly evicted on insertion.

6. When the legitimate orphan's parent arrives and process_orphan_tx() runs,
   find_orphan_by_previous() returns empty — the orphan was evicted.
   The legitimate transaction is silently dropped.
```

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
