### Title
Missing Zero/Expired `ban_until` Validation in `set_ban` RPC Allows Silent Ineffective Ban Insertion — (File: `rpc/src/module/net.rs`)

---

### Summary

The `set_ban` RPC handler computes a `ban_until` timestamp of `0` when called with `absolute=true` and `ban_time=null` (or `ban_time=0x0`). This zero-valued timestamp is inserted into the `BanList` HashMap without any check that it is non-zero or in the future. The result is a phantom ban entry that appears in `get_banned_addresses()` but never actually blocks any peer, silently defeating the operator's intent to ban a malicious peer.

---

### Finding Description

In `rpc/src/module/net.rs`, the `set_ban` implementation computes `ban_until` as follows:

```rust
let ban_until = if absolute.unwrap_or(false) {
    ban_time.unwrap_or_default().into()   // Timestamp::default() == 0
} else {
    unix_time_as_millis() + ban_time.unwrap_or_else(|| DEFAULT_BAN_DURATION.into()).value()
};
self.network_controller.ban(ip_network, ban_until, reason.unwrap_or_default());
``` [1](#0-0) 

When `absolute=true` and `ban_time` is `null` (or explicitly `0x0`), `ban_time.unwrap_or_default()` resolves to `Timestamp::default()` which is `0`. The resulting `ban_until = 0` is passed directly to `BanList::ban()`.

`BanList::ban()` inserts the entry into its `inner: HashMap<IpNetwork, BannedAddr>` with no validation:

```rust
pub fn ban(&mut self, banned_addr: BannedAddr) {
    self.inner.insert(banned_addr.address, banned_addr);
    ...
}
``` [2](#0-1) 

The ban-check logic then uses `banned_addr.ban_until.gt(&now_ms)`:

```rust
fn is_ip_banned_until(&self, ip: IpAddr, now_ms: u64) -> bool {
    ...
    if let Some(banned_addr) = self.inner.get(&ip_network)
        && banned_addr.ban_until.gt(&now_ms)
    {
        return true;
    }
    ...
}
``` [3](#0-2) 

Since `0 > now_ms` is always `false`, the peer is never actually blocked. The entry persists in the ban list (visible via `get_banned_addresses()`) until the periodic `clear_expires()` sweep runs every 1024 inserts, creating a false sense of security.

The intermediate path through `ban_network()` also performs no zero-check: [4](#0-3) 

---

### Impact Explanation

A node operator who calls `set_ban("1.2.3.4", "insert", null, true, null)` (or `set_ban("1.2.3.4", "insert", "0x0", true, null)`) believes they have banned a malicious peer. In reality:

1. The ban entry is inserted with `ban_until = 0`.
2. `is_ip_banned()` always returns `false` for that entry.
3. The peer at `1.2.3.4` continues to connect and relay messages normally.
4. `get_banned_addresses()` shows the entry as banned, misleading the operator.

Any ongoing attack (e.g., eclipse attempt, malformed message flood, relay spam) from the "banned" peer continues unimpeded. The security control is silently bypassed.

---

### Likelihood Explanation

The trigger requires `absolute=true` combined with `ban_time=null` (or `ban_time=0x0`). This is a plausible operator mistake: the RPC documentation states `ban_time` is optional and defaults to 24h, but does not warn that `absolute=true` with a null/zero `ban_time` produces an immediately-expired ban. Any local RPC caller — including a CLI user or a script — can trigger this. The `set_ban` RPC is not authenticated beyond network-level access to the RPC port. [5](#0-4) 

---

### Recommendation

Add a validation check in `set_ban` (or in `BanList::ban()`) to reject or warn when `ban_until` is zero or already in the past:

```rust
"insert" => {
    let ban_until = if absolute.unwrap_or(false) {
        ban_time.unwrap_or_default().into()
    } else {
        unix_time_as_millis() + ban_time.unwrap_or_else(|| DEFAULT_BAN_DURATION.into()).value()
    };
    // Add: reject zero or already-expired ban_until
    if ban_until <= unix_time_as_millis() {
        return Err(RPCError::invalid_params(
            "ban_time must be a future timestamp when absolute=true"
        ));
    }
    self.network_controller.ban(ip_network, ban_until, reason.unwrap_or_default());
    Ok(())
}
```

Alternatively, `BanList::ban()` itself should skip insertion when `banned_addr.ban_until <= unix_time_as_millis()`.

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "set_ban",

- `get_banned_addresses()` returns `{"address": "1.2.3.4/32", "ban_until": "0x0", ...}` — appears banned.
- The peer at `1.2.3.4` can still connect; `is_ip_banned("1.2.3.4")` returns `false`.
