### Title
Tx-Pool Minimum Fee Rate Admission Check Uses Serialized Size Instead of Actual Transaction Weight, Allowing Cycle-Heavy Transactions to Bypass Fee Enforcement - (File: tx-pool/src/util.rs)

---

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` computes the minimum required fee using only the transaction's serialized byte size (`tx_size`), not the actual transaction weight (`max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`). For cycle-heavy transactions, the actual weight can be orders of magnitude larger than the serialized size. Because no second fee check is performed after script verification (when actual cycles are known), an unprivileged tx-pool submitter can admit cycle-heavy transactions with fees far below the minimum fee rate requirement.

---

### Finding Description

In `check_tx_fee`, the minimum fee threshold is computed as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The code itself acknowledges the discrepancy with a comment: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."*

However, the actual transaction weight used everywhere else in the system — for block packing, fee rate sorting, and eviction — is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. For a transaction with 200 bytes and 70,000,000 cycles (the configured maximum), the actual weight is `max(200, 11,940) = 11,940`, but the fee check uses only `200`. The minimum fee required is therefore ~60× lower than it should be.

The `check_tx_fee` function is called during `pre_check` (before script verification, when cycles are unknown):

```rust
let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
``` [3](#0-2) 

After `verify_rtx` completes and actual cycles are known, no second fee check is performed against the actual weight. The `TxEntry` stores the real cycles and exposes `fee_rate()` using actual weight for sorting/eviction purposes, but this is never used as an admission gate:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

This is directly analogous to the `initiateBorrow` bug in the external report: the fee check condition is too lenient because it omits a required component (cycles-derived weight), allowing the attacker to bypass the intended minimum fee rate.

---

### Impact Explanation

An unprivileged RPC caller submitting via `send_transaction` can craft a transaction with:
- Small serialized size (e.g., 200 bytes) — minimizing the size-based fee check
- High cycle consumption (up to `max_tx_verify_cycles = 70,000,000`) — maximizing actual weight

Such a transaction passes `check_tx_fee` with a fee of just `min_fee_rate * tx_size / 1000` shannons (e.g., 200 shannons at the default 1,000 shannons/KW rate), while the correct minimum fee based on actual weight would be ~11,940 shannons — a ~60× shortfall.

Consequences:
1. **Fee rate enforcement bypass**: Cycle-heavy transactions enter the tx-pool at a fraction of the intended cost.
2. **Resource exhaustion**: Each such transaction consumes significant CPU (script verification up to 70M cycles) and tx-pool memory, at a cost far below what the fee rate policy intends to impose.
3. **Miner revenue distortion**: Miners selecting transactions by actual fee rate will see these entries as extremely low-fee-rate and deprioritize or exclude them, but the node has already paid the verification cost.

---

### Likelihood Explanation

Any unprivileged peer with RPC access to `send_transaction` (the standard public pool submission endpoint) can trigger this. No special privilege, key, or majority hashpower is required. The attacker only needs to construct a transaction whose lock/type script consumes many cycles — a straightforward task for any script author. The default `max_tx_verify_cycles = 70,000,000` provides a large amplification factor. [5](#0-4) 

---

### Recommendation

After `verify_rtx` completes and actual cycles are known, perform a second fee check using the actual transaction weight:

```rust
let actual_weight = get_transaction_weight(tx_size, completed.cycles);
let min_fee_actual = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_actual {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors the correct pattern already used in `TxEntry::fee_rate()` and `FeeRateCollector::statistics()`. [6](#0-5) 

---

### Proof of Concept

1. Configure a CKB node with default `min_fee_rate = 1000` shannons/KW and `max_tx_verify_cycles = 70_000_000`.
2. Craft a transaction with:
   - Serialized size ≈ 200 bytes
   - A lock script that consumes ~70,000,000 cycles (e.g., a tight computation loop in CKB-VM)
   - Fee = 201 shannons (just above `1000 * 200 / 1000 = 200`)
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 * 200 / 1000 = 200`; fee 201 ≥ 200 → **passes**.
5. `verify_rtx` runs the script, consuming ~70M cycles; actual weight = `max(200, 11940) = 11940`.
6. Correct minimum fee should be `1000 * 11940 / 1000 = 11940` shannons — the submitted fee of 201 is ~59× too low.
7. The transaction is admitted to the tx-pool, consuming CPU and memory at ~1.7% of the intended fee cost. [7](#0-6) [8](#0-7)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
