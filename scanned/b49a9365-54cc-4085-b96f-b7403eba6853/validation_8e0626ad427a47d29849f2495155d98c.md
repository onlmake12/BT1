Audit Report

## Title
Unbounded `async_handle().spawn()` Task Accumulation via Unauthenticated `GetHeaders` Flood — (`sync/src/synchronizer/get_headers_process.rs`)

## Summary
Any unprivileged peer can flood a post-IBD CKB node with `GetHeaders` messages containing a trivially valid locator (any hash followed by the genesis hash). Each message unconditionally triggers `async_handle().spawn()` with a pre-built `SendHeaders` payload of up to 2,000 headers, with no rate limit, backpressure, or task cap. Tasks accumulate unboundedly in the shared Tokio runtime, leading to memory exhaustion and starvation of block processing, relay, and RPC subsystems.

## Finding Description

In `GetHeadersProcess::execute()`, after two cheap checks (locator size ≤ `MAX_LOCATOR_SIZE` = 101, and last locator entry == genesis hash), the code unconditionally builds a full `SendHeaders` message and spawns a fire-and-forget task:

```rust
// sync/src/synchronizer/get_headers_process.rs, lines 84–93

    .headers(headers.into_iter().map(|x| x.data()).collect::<Vec<_>>())
    .build();
let message = packed::SyncMessage::new_builder().set(content).build();
let nc = Arc::clone(self.nc);
self.synchronizer
    .shared()
    .shared()
    .async_handle()
    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
``` [1](#0-0) 

The `locate_latest_common_block` gate at lines 1866–1868 of `sync/src/types/mod.rs` only requires the last locator hash to equal the genesis hash. A locator of `[<any_unknown_hash>, <genesis_hash>]` passes this check and returns `Some(0)`, causing `get_locator_response(0, &hash_stop)` to return up to `MAX_HEADERS_LEN` = 2,000 headers: [2](#0-1) 

`Handle::spawn()` is a thin, unbounded wrapper over Tokio's `spawn` with no semaphore, queue depth limit, or backpressure: [3](#0-2) 

The `Synchronizer::try_process()` match arm for `GetHeaders` has no rate-limiting guard whatsoever: [4](#0-3) 

By contrast, the `Relayer` explicitly installs a `governor`-based rate limiter capped at 30 req/s per peer per message type, applied before any processing: [5](#0-4) [6](#0-5) 

The `Synchronizer` struct has no equivalent field or guard. `MAX_HEADERS_LEN` is 2,000 and `MAX_LOCATOR_SIZE` is 101: [7](#0-6) [8](#0-7) 

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Each spawned task holds a serialized `SendHeaders` message of up to 2,000 × ~192 bytes ≈ 384 KB, allocated synchronously before the spawn. At 10,000 queued tasks, heap pressure from message payloads alone reaches ~3.84 GB, independent of Tokio task metadata. The shared Tokio runtime services block processing, relay propagation, and RPC; unbounded task accumulation starves all these subsystems, causing OOM kill or livelock of the node.

## Likelihood Explanation

Preconditions are all reachable in normal mainnet operation with no privilege:
1. Establish a standard P2P connection (no key, no role, no PoW required).
2. Wait for the target node to exit IBD (normal state for any synced node).
3. Send `GetHeaders` in a tight loop with locator `[<random_hash>, <genesis_hash>]`.

The attack is asymmetric: the attacker sends small messages (~200 bytes each); the victim allocates and queues large `SendHeaders` payloads per message. A single peer connection is sufficient.

## Recommendation

Apply a per-peer rate limit to `GetHeaders` processing in the `Synchronizer`, mirroring the existing `governor`-based `RateLimiter` in the `Relayer` (`sync/src/relayer/mod.rs`, lines 63–92). A limit of 1–5 `GetHeaders` per second per peer is sufficient for legitimate sync behavior. Additionally, consider bounding total outstanding `SendHeaders` spawn tasks via a `tokio::sync::Semaphore` to enforce a hard cap on queued work regardless of peer count.

## Proof of Concept

1. Connect to a post-IBD CKB node via the Sync P2P protocol.
2. In a tight loop, send `SyncMessage::GetHeaders` with `block_locator_hashes = [<random_32_byte_hash>, <genesis_hash>]` and `hash_stop = Byte32::zero()`.
3. Monitor Tokio task count (via metrics or `/proc/<pid>/status` VmRSS) and heap allocation.
4. Assert: after N messages, task queue depth grows proportionally to N and heap grows proportionally to `N × MAX_HEADERS_LEN × header_size`.
5. Observe RPC latency and block relay degradation as the runtime is saturated, eventually leading to OOM kill or node unresponsiveness.
