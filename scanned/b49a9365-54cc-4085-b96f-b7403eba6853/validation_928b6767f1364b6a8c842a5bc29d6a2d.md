Audit Report

## Title
`remove_expired` Removes Parent Transactions Without Evicting Dependent Descendants, Leaving Zombie Entries in the Pool - (File: tx-pool/src/pool.rs)

## Summary
`TxPool::remove_expired` removes each expired entry individually via `pool_map.remove_entry`, which severs parent-child links but does not cascade removal to child transactions. Child transactions submitted after the parent remain in the pool in a permanently uncommittable state, holding pool size/cycle budget and locking their input outpoints in `edges.inputs` until their own individual timestamps cross the expiry threshold. This contradicts the function's own comment ("Expire all transaction **and their dependencies**") and is inconsistent with every other eviction path in the codebase.

## Finding Description
In `tx-pool/src/pool.rs` at L270–288, `remove_expired` collects all entries whose individual timestamp satisfies `self.expiry + entry.inner.timestamp < now_ms`, then calls `self.pool_map.remove_entry(&entry.proposal_short_id())` for each one.

`remove_entry` (pool_map.rs L235–250) does the following for the removed entry only:
- `remove_entry_edges` — releases the parent's own inputs from `edges.inputs`.
- `remove_entry_links` — severs the parent↔child link in `links`.
- Updates ancestor/descendant score keys for remaining entries.

It does **not** remove child transactions. After the parent is evicted:
1. Child transactions remain in `entries` with `Status::Pending/Gap/Proposed`.
2. Their inputs (referencing the now-gone parent's output outpoints) remain registered in `edges.inputs`.
3. Their `links.parents` is now empty (link severed), so `calc_ancestors` returns an empty set — the pool treats them as root transactions.
4. They can never be committed: the parent's outputs are neither in the pool nor on-chain.
5. They will not be re-evicted until their own timestamps also cross the expiry threshold.

This is confirmed by the existing test `test_remove_entry` (score_key.rs L157–167), which explicitly asserts that after `remove_entry(&tx1_id)`, `tx2` and `tx3` remain in the pool.

Contrast with every other eviction path:
- `limit_size` → `remove_entry_and_descendants` (pool.rs L307)
- `remove_tx` → `remove_entry_and_descendants` (pool.rs L359)
- `remove_by_detached_proposal` → `remove_entry_and_descendants` (pool.rs L343)
- `resolve_conflict` → `remove_entry_and_descendants` (pool_map.rs L310, L321)
- `resolve_conflict_header_dep` → `remove_entry_and_descendants` (pool_map.rs L285)

`remove_expired` is the sole eviction path using the single-entry variant.

**Exploit flow:**
1. Attacker submits parent `P` at time `T` with minimum fee (accepted, will expire at `T + expiry`).
2. Attacker submits `N` children `C₁…Cₙ` spending `P`'s outputs at time `T + δ` (where `δ` can be up to nearly `expiry_hours`).
3. At `T + expiry`, `remove_expired` removes `P` but leaves `C₁…Cₙ` in the pool.
4. `C₁…Cₙ` remain as zombies until `T + expiry + δ`, consuming pool size/cycle budget and holding their outpoints in `edges.inputs`.
5. Any new transaction attempting to spend the same outpoints as `C₁…Cₙ` is rejected as a double-spend, even though the parent was never mined.
6. Repeating this pattern with many chains keeps a significant fraction of the pool filled with permanently unspendable transactions.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The zombie entries consume `pool_map.total_tx_size` and `total_tx_cycles` budget, reducing available pool capacity for legitimate transactions. The double-spend rejection on zombie-held outpoints further blocks legitimate users from spending those outputs. With a default `expiry_hours` of 12 hours and `δ` approaching that value, an attacker can maintain persistent pool congestion across repeated cycles at the cost of minimum transaction fees.

## Likelihood Explanation
The attack requires only the ability to submit transactions — available to any unprivileged peer or RPC caller. No special privileges, leaked keys, or victim mistakes are required. The cost is minimum fees for parent and child transactions. The 12-hour default expiry window gives a long zombie persistence period. The attack is repeatable and can be automated.

## Recommendation
Replace `remove_entry` with `remove_entry_and_descendants` inside `remove_expired`, consistent with every other eviction path:

```rust
// tx-pool/src/pool.rs  remove_expired()
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
-   self.pool_map.remove_entry(&entry.proposal_short_id());
+   let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    let reject = Reject::Expiry(entry.timestamp);
-   callbacks.call_reject(self, &entry, reject);
+   for e in evicted {
+       callbacks.call_reject(self, &e, reject.clone());
+   }
}
```

This aligns the implementation with the existing comment ("and their dependencies") and with the behaviour of all other eviction paths.

## Proof of Concept
1. Submit parent transaction `P` with minimum fee at time `T`.
2. Submit child transaction `C` spending an output of `P` at time `T + δ` (e.g., `δ` = 11 hours with `expiry_hours` = 12).
3. Advance the node clock past `T + expiry_hours` but before `T + expiry_hours + δ`.
4. Observe via `tx_pool_info` that `C` remains in the pending pool even though `P` is gone.
5. Attempt to submit a new transaction `C'` spending the same output of `P` as `C` — it is rejected with a double-spend error, even though `P` is no longer in the pool and was never mined.
6. A unit test can be constructed analogously to `test_remove_entry` (score_key.rs L97–168): add a parent and child, call `remove_entry` on the parent, and assert that the child's input outpoint is still registered in `edges.inputs` — confirming the zombie state.
