Audit Report

## Title
`ScriptHashTypeVerifier` Omits Output Type Script Hash-Type Check, Allowing Unactivated VM Variants to Pollute the Tx-Pool — (`verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only on output **lock** scripts. Output **type** scripts are never inspected. An attacker can craft a transaction whose output type script carries an unactivated `ScriptHashType` value (e.g., `Data3 = 6`). The transaction passes all non-contextual checks and is admitted to the tx-pool, yet can never be committed to a block because `select_version()` returns `Err(InvalidScriptHashType)` for any variant outside `{Data, Data1, Data2, Type}`. The result is permanent tx-pool slot occupation with zero cost beyond a standard fee.

## Finding Description

**Allowlist definition** — `util/constant/src/consensus.rs` restricts activated hash types to `{0, 1, 2, 4}`: [1](#0-0) 

**Enum expansion** — `util/gen-types/src/core.rs` uses `seq_macro` to generate `Data3`–`Data127` (values 6–254). `verify_value()` accepts all even values and 1, so value 6 passes `check_data()` without error.

**The gap** — `ScriptHashTypeVerifier::verify()` iterates outputs and checks only `output.lock().hash_type()`. The `output.type_()` field is never read: [2](#0-1) 

This verifier is the sole `ENABLED_SCRIPT_HASH_TYPE` enforcement point inside `NonContextualTransactionVerifier::verify()`: [3](#0-2) 

**Execution-layer rejection** — `select_version()` in `script/src/types.rs` handles only `Data`, `Data1`, `Data2`, and `Type`; every other variant hits the catch-all and returns `Err(ScriptError::InvalidScriptHashType(...))`: [4](#0-3) 

**Cellbase comparison** — `CellbaseVerifier` in `block_verifier.rs` explicitly forbids type scripts on cellbase outputs entirely (`output.type_().is_some()` → `InvalidTypeScript`), so the gap only affects regular transactions: [5](#0-4) 

## Impact Explanation

Transactions with an unactivated type-script hash type occupy tx-pool slots indefinitely. They are valid enough to be relayed across the P2P network and to be selected by miners for block assembly, but every block containing them is rejected by all peers at contextual script execution. This maps directly to **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points): an attacker can flood the tx-pool and waste miner resources at negligible cost.

## Likelihood Explanation

Any unprivileged user can construct such a transaction. Value `6` is a structurally valid even byte accepted by `verify_value()`. No special key material, majority hashpower, or privileged access is required. The attack is repeatable and cheap.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the hash type of each output's type script against `ENABLED_SCRIPT_HASH_TYPE`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the pattern already used in `CellbaseVerifier` for lock scripts. [6](#0-5) 

## Proof of Concept

1. Build a transaction with one output whose type script sets `hash_type = 6` (`Data3`) and whose lock script uses a valid activated hash type (`Data` or `Type`).
2. Submit via `send_transaction` RPC. `NonContextualTransactionVerifier` runs `ScriptHashTypeVerifier`, which checks only the lock script → passes. Transaction is admitted to the tx-pool.
3. Attempt block assembly including this transaction. `select_version()` is invoked for the type script group → returns `Err(InvalidScriptHashType("Data3 has not been activated"))` → block is rejected by all peers.
4. The transaction remains in the tx-pool, occupying a slot and consuming relay bandwidth, with no path to eviction short of expiry or manual intervention.

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
