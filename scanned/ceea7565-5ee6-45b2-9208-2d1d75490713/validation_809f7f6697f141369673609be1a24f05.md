### Title
`validate_dao` Assumes Lock Script Identity from Args Length Alone, Silently Accepting Unwithdrawable DAO Deposits — (File: `rpc/src/module/pool.rs`)

---

### Summary

`WellKnownScriptsOnlyValidator::validate_dao` identifies a DAO deposit's lock script as `secp256k1_blake160_multisig_all` based solely on the lock script args being exactly 28 bytes long, without ever checking the lock script's `code_hash` or `hash_type`. Any lock script with 28-byte args whose last 8 bytes encode a valid absolute-epoch `since` value will silently pass the `well_known_scripts_only` validator — even if that lock script has no DAO-withdrawal capability. An RPC caller who submits such a transaction receives a success response, but the deposited CKB may be permanently unrecoverable.

---

### Finding Description

`validate_secp256k1_blake160_multisig_all` (lines 807–835) correctly gates its `since`-field check behind an explicit identity check:

```rust

    Err(DefaultOutputsValidatorError::HashType)
} else if script.code_hash()
    != self.consensus.secp256k1_blake160_multisig_all_type_hash()...
{
    Err(DefaultOutputsValidatorError::CodeHash)
} else if script.args().len() != BLAKE160_LEN {
    if script.args().len() == BLAKE160_LEN + SINCE_LEN {
        // only now interpret last 8 bytes as `since`
``` [1](#0-0) 

`validate_dao` (lines 853–879) performs **no such identity check** on the lock script. It branches on args length alone:

```rust
} else if output.lock().args().len() == BLAKE160_LEN + SINCE_LEN {
    // https://github.com/nervosnetwork/ckb/wiki/Common-Gotchas#nervos-dao
    let since =
        extract_since_from_secp256k1_blake160_multisig_all_args(&output.lock());
    match since.extract_metric() {
        Some(SinceMetric::EpochNumberWithFraction(_)) if since.is_absolute() => {
            Ok(())
        }
        _ => Err(DefaultOutputsValidatorError::ArgsSince),
    }
} else {
    Ok(())
}
``` [2](#0-1) 

The helper function name `extract_since_from_secp256k1_blake160_multisig_all_args` documents the assumption, but the caller never enforces it. Any lock script — regardless of `code_hash` or `hash_type` — whose args are exactly 28 bytes and whose last 8 bytes satisfy `since.is_absolute() && since.extract_metric() == Some(SinceMetric::EpochNumberWithFraction(_))` will return `Ok(())`. [3](#0-2) 

This is the direct CKB analog of using a concrete `ERC20` implementation instead of the `IERC20` interface: the code hard-codes an assumption about the internal layout of a specific lock script and applies that assumption to any lock script that matches only on argument length.

---

### Impact Explanation

An RPC caller who submits a DAO deposit transaction with `outputs_validator: "well_known_scripts_only"` and a non-standard lock script whose args are exactly 28 bytes and whose last 8 bytes encode a valid absolute-epoch `since` (e.g., `0x2000_0000_0000_0001`) receives a success response from `send_transaction`. [4](#0-3) 

The validator gives a false guarantee of safety. If the lock script in question does not implement the DAO withdrawal protocol — i.e., it cannot produce a valid witness for the DAO type script's withdrawal check — the deposited CKB is permanently locked in the DAO cell with no way to withdraw it. This is the same class of impact as the ERC20 report: assets deposited under an assumed-compatible implementation become irrecoverable.

---

### Likelihood Explanation

The `well_known_scripts_only` validator must be explicitly requested; the default is `passthrough`. [5](#0-4) 

However, wallets and tooling that advertise "safe mode" submission routinely set this flag. A script author or integrator who deploys a custom lock script with 28-byte args and tests DAO deposits in "safe mode" will observe no rejection and may proceed to production, locking real CKB. The condition for false acceptance (28-byte args + last 8 bytes = valid absolute epoch `since`) is easy to satisfy accidentally — for example, any lock script whose args happen to end in bytes that decode to epoch 0 (`0x2000_0000_0000_0000`) will pass.

---

### Recommendation

Add a lock-script identity check inside the 28-byte branch of `validate_dao`, mirroring the check already present in `validate_secp256k1_blake160_multisig_all`:

```rust
} else if output.lock().args().len() == BLAKE160_LEN + SINCE_LEN {
    // Only apply the since-field check when the lock script is actually
    // secp256k1_blake160_multisig_all; reject any other 28-byte lock script.
    let lock = output.lock();
    if !lock.is_hash_type_type()
        || lock.code_hash()
            != self
                .consensus
                .secp256k1_blake160_multisig_all_type_hash()
                .expect("No secp256k1_blake160_multisig_all system cell")
    {
        return Err(DefaultOutputsValidatorError::NotWellKnownLockScript);
    }
    let since = extract_since_from_secp256k1_blake160_multisig_all_args(&lock);
    match since.extract_metric() {
        Some(SinceMetric::EpochNumberWithFraction(_)) if since.is_absolute() => Ok(()),
        _ => Err(DefaultOutputsValidatorError::ArgsSince),
    }
``` [6](#0-5)

### Proof of Concept

1. Deploy a custom lock script (any bytecode, any `code_hash`) whose args are exactly 28 bytes. Set the last 8 bytes to `0x00 0x00 0x00 0x00 0x00 0x00 0x00 0x20` (little-endian `0x2000_0000_0000_0000` — absolute epoch 0, which satisfies `is_absolute()` and `extract_metric() == Some(EpochNumberWithFraction(...))`).

2. Construct a transaction output:
   - `type_`: DAO type script (`code_hash = dao_type_hash`, `hash_type = type`)
   - `lock`: the custom script above

3. Call `send_transaction` via RPC with `outputs_validator: "well_known_scripts_only"`.

4. `validate_dao` enters the 28-byte branch at line 863, extracts `since = 0x2000_0000_0000_0000`, finds `is_absolute() = true` and `extract_metric() = Some(EpochNumberWithFraction(epoch=0))`, and returns `Ok(())`. [7](#0-6) 

5. The transaction is accepted and committed on-chain. The CKB is now locked in the DAO under a lock script that has no DAO-withdrawal capability, making the funds permanently unrecoverable — the exact same outcome as depositing an incompatible ERC20 token into the Endaoment contracts.

fn extract_since_from_secp256k1_blake160_multisig_all_args(script: &packed::Script) -> Since {
