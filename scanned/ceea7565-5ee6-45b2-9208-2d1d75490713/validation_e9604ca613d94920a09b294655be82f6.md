### Title
Expired Alert Re-Acceptance via Missing `cancel_filter` Insertion in `clear_expired_alerts` — (`util/network-alert/src/notifier.rs`)

### Summary
`clear_expired_alerts()` removes expired alert IDs from `received_alerts` and `noticed_alerts` but never inserts them into `cancel_filter`. Because `has_received()` checks only those two structures, any P2P peer can replay the original (still-validly-signed) alert bytes after expiry, causing the node to re-notify subscribers and re-broadcast the alert to connected peers.

### Finding Description

`clear_expired_alerts` removes expired entries from both collections but leaves `cancel_filter` untouched: [1](#0-0) 

`has_received` then returns `false` for the evicted ID because neither `received_alerts` nor `cancel_filter` contains it: [2](#0-1) 

`add()` uses `has_received` as its sole deduplication gate. When it returns `false`, the alert is re-inserted and `notify_controller.notify_network_alert()` fires again: [3](#0-2) 

In `alert_relayer.rs`, `received()` checks `has_received` before signature verification. After expiry clears the ID, the check passes, the original signatures verify successfully (they are over the deterministic alert hash and remain cryptographically valid), and the alert is re-broadcast to connected peers **before** `add()` is even called: [4](#0-3) 

`clear_expired_alerts` is triggered on every new peer connection, making the window predictable: [5](#0-4) 

The existing test `test_clear_expired_alerts` explicitly asserts `has_received(1)` returns `false` after expiry, confirming the gap is observable and untested for re-relay: [6](#0-5) 

### Impact Explanation
An unprivileged P2P peer holding the original signed alert bytes (received during normal propagation) can replay them after expiry to: (1) re-fire `notify_controller.notify_network_alert()` on the victim node, causing spurious subscriber notifications about a stale alert; (2) re-insert the expired alert into `noticed_alerts` and `received_alerts`; (3) trigger re-broadcast to other connected peers whose `known_lists` entry for that alert ID may have been evicted from the LRU cache. Impact is **Low (501–2000)**: no funds at risk, no consensus impact, but repeated spurious notifications and unnecessary network churn.

### Likelihood Explanation
Any peer that observed the original alert broadcast retains the signed bytes. No key material or privileged access is required. The replay is trivially executable after any `connected()` event triggers `clear_expired_alerts`. Likelihood is **Medium**: requires timing awareness but no cryptographic capability.

### Recommendation
In `clear_expired_alerts`, insert each evicted alert ID into `cancel_filter` before removing it:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ()); // <-- add this
            false
        } else {
            true
        }
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

This ensures `has_received(id)` returns `true` for any previously-seen alert ID, regardless of whether it was cancelled or expired.
