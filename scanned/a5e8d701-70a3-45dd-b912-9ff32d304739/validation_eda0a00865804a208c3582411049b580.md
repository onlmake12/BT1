### Title
Division-by-Zero in DAO Secondary Reward Calculation When `parent_c` Is Zero — (File: `util/dao/src/lib.rs`)

### Summary

The `DaoCalculator` in `util/dao/src/lib.rs` performs integer division by `parent_c` (the total capacity field `C` from the parent block's DAO data) in three separate arithmetic expressions without any zero-check guard. If a block's DAO field encodes `C = 0`, the node panics with a division-by-zero at runtime. This is the direct CKB analog of the "broken inverse of zero" class: the code treats zero as a valid divisor and silently computes a mathematically undefined result (or crashes), just as the reported bug treats `0` as a valid field element for modular inversion.

---

### Finding Description

In `util/dao/src/lib.rs`, the function `dao_field_with_current_epoch` extracts four values from the parent block's DAO field:

```rust
let (parent_ar, parent_c, parent_s, parent_u) = extract_dao_data(parent.dao());
```

It then divides by `parent_c` in three places:

```rust
// Line 242-243
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());

// Line 257
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
```

And in `secondary_block_reward` (line 202-203):

```rust
let reward128 = u128::from(target_g2.as_u64()) * u128::from(target_parent_u.as_u64())
    / u128::from(target_parent_c.as_u64());
```

None of these three division sites check whether `parent_c` (or `target_parent_c`) is zero before dividing. In Rust, integer division by zero causes a **panic** (thread abort).

The DAO field is a 32-byte `Byte32` value stored in every block header. The `C` field occupies bytes `[0..8]` in little-endian. A block header with all-zero DAO bytes (or any crafted DAO bytes where `C = 0`) will cause `parent_c = 0`, triggering the panic.

The genesis-time helper `genesis_dao_data_with_satoshi_gift` does guard against `C = 0` at genesis:

```rust
if c == Capacity::zero() {
    return Err(DaoError::ZeroC);
}
```

However, this guard only applies at genesis construction. There is **no equivalent guard** in the per-block DAO field verification path (`DaoHeaderVerifier::verify` → `DaoCalculator::dao_field` → `dao_field_with_current_epoch`). A block with a crafted DAO field where `C = 0` passes the DAO bytes into `extract_dao_data` without validation, and the division-by-zero panic follows immediately.

---

### Impact Explanation

Any peer that relays a syntactically valid block (passes structural/PoW checks) with a DAO field encoding `C = 0` will cause the receiving full node to **panic and crash** during contextual block verification (`DaoHeaderVerifier::verify`). This is a remotely-triggerable node crash (denial of service). Because the panic occurs inside the block verification pipeline, it can crash the node process entirely, disrupting sync and block propagation.

---

### Likelihood Explanation

A block relayer or miner (even without majority hashpower) can craft a block with an arbitrary DAO field. The DAO field is a plain `Byte32` in the block header — it is not independently validated for internal consistency before the division is reached. The attacker only needs to relay one such block to a victim node. The crash is deterministic and reproducible.

---

### Recommendation

Add an explicit zero-check for `parent_c` (and `target_parent_c`) before each division in `dao_field_with_current_epoch` and `secondary_block_reward`, returning `Err(DaoError::ZeroC)` (which already exists in the error enum) if the value is zero. Additionally, the `DaoHeaderVerifier` should validate that the parent block's DAO field has a non-zero `C` before invoking the calculator.

pub fn extract_dao_data(dao: Byte32) -> (u64, Capacity, Capacity, Capacity) {
    let data = dao.raw_data();
    let c = Capacity::shannons(LittleEndian::read_u64(&data[0..8]));
    let ar = LittleEndian::read_u64(&data[8..16]);
    let s = Capacity::shannons(LittleEndian::read_u64(&data[16..24]));
    let u = Capacity::shannons(LittleEndian::read_u64(&data[24..32]));
    (ar, c, s, u)
}
