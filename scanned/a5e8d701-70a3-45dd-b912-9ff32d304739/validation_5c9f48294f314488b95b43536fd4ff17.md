### Title
Stale `total_tx_size`/`total_tx_cycles` Counter After Ancestor Eviction Inflates Pool Accounting, Causing Unnecessary Transaction Eviction — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's `total_tx_size` and `total_tx_cycles` counters are computed **before** potential ancestor evictions occur, then unconditionally written back **after** those evictions have already decremented the same counters. This overwrites the correctly-updated values with stale pre-eviction figures, permanently inflating the pool's size accounting. The inflated counter then causes `limit_size` to evict legitimate pending transactions that should not have been removed.

---

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```
1. total_tx_size = self.total_tx_size + entry.size   ← snapshot taken here
2. check_and_record_ancestors(entry)                  ← may evict txs, decrementing self.total_tx_size
3. self.total_tx_size = total_tx_size                 ← stale snapshot written back, evictions ignored
```

Step 1 calls `updated_stat_for_add_tx`, which snapshots `self.total_tx_size + entry.size` into a local variable.

Step 2 calls `check_and_record_ancestors`. When a new transaction has more ancestors than `max_ancestors_count` but some of those ancestors are cell-dep-only parents (`cell_ref_parents`), the function enters the eviction branch and calls `remove_entry_and_descendants` for each evicted transaction. Each removal calls `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`.

Step 3 then unconditionally overwrites `self.total_tx_size` with the local variable captured in step 1 — a value that does not account for the evictions performed in step 2.

**Concrete arithmetic:**

| Event | `self.total_tx_size` |
|---|---|
| Initial | 1000 |
| After `updated_stat_for_add_tx(+100)` (local var = 1100) | 1000 (unchanged) |
| After evicting 3 txs of sizes 200+300+400 | 100 |
| After `self.total_tx_size = total_tx_size` | **1100** (should be 200) |

The counter is inflated by the total size of all evicted transactions (900 in this example).

The inflated `total_tx_size` is then consumed by `limit_size`:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evicts legitimate pending transactions
}
```

Because the pool believes it holds far more data than it actually does, `limit_size` evicts additional legitimate transactions that were correctly admitted and should remain in the pool.

The same logic applies to `total_tx_cycles`.

---

### Impact Explanation

An unprivileged tx-pool submitter can permanently inflate `total_tx_size` by triggering the ancestor-eviction path. After inflation, every subsequent call to `limit_size` (which runs after every new submission) will evict legitimate pending transactions. Affected transactions are rejected with `Reject::Full` and recorded in the recent-reject database, preventing their resubmission for the configured TTL. This constitutes a targeted, repeatable denial-of-service against the transaction pool: honest users' transactions are silently dropped even when the pool has ample real capacity.

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is reachable by any unprivileged submitter. The attacker:

1. Submits N transactions (N > `max_ancestors_count`) that each reference a common in-pool cell dep (making them `cell_ref_parents`).
2. Submits one transaction that **consumes** that cell dep as an input. This transaction's ancestor count exceeds `max_ancestors_count`, but `ancestors_count - cell_ref_parents.len() ≤ max_ancestors_count`, so the eviction branch is entered.
3. The eviction branch removes the cell-ref transactions, decrementing `self.total_tx_size`, but `add_entry` then overwrites the counter with the stale pre-eviction snapshot.

This is a standard, documented use-case for the pool (the test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` exercises exactly this scenario). No special privilege, key, or majority hash power is required.

---

### Recommendation

Move the counter snapshot **after** all evictions complete, or recompute the delta by subtracting evicted entries' sizes/cycles from the pre-computed total before writing it back. For example, in `add_entry`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Recompute totals after evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, subtract the total size/cycles of all entries in `evicts` from `total_tx_size`/`total_tx_cycles` before the final assignment.

    ) -> Result<(bool, HashSet<TxEntry>), Reject> {

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }

            .dep_type(DepType::Code)

        // we can have more than config.max_ancestors_count number of txs using one cell ref
        let mut cell_ref_txs = vec![];
