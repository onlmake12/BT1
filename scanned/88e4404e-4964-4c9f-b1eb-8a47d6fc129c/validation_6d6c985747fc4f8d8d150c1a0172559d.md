### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Validation, Allowing Future/Disabled Hash Types to Bypass Non-Contextual Checks - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` only validates the `hash_type` field of each output's **lock script**, completely skipping the **type script**. This mirrors the SubExecutor bug exactly: a simple field check (`address == 0 → native, else → ERC20`) is used to distinguish processing paths, but the "else" branch is never validated. Here, the distinction between "enabled" and "disabled" `hash_type` values is broken for type scripts — any transaction output carrying a type script with a future/not-yet-enabled `hash_type` (e.g., `Data3` = 6) silently passes the non-contextual gate and enters the tx-pool verify queue, where it wastes script execution resources before being rejected.

---

### Finding Description

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and checks only `output.lock().hash_type()` against `ENABLED_SCRIPT_HASH_TYPE`:

```rust
// verification/src/transaction_verifier.rs
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
```

`output.type_()` is never inspected. The docstring of `NonContextualTransactionVerifier` even documents this gap explicitly: *"Check whether output **lock** hash type within enabled range"*.

`ENABLED_SCRIPT_HASH_TYPE` is `{0, 1, 2, 4}` (Data, Type, Data1, Data2). The `ScriptHashType` enum, however, defines `Data3 = 6`, `Data4 = 8`, … `Data127 = 254` via a macro expansion, and `ScriptHashType::verify_value(v)` accepts any even number or `1` — so values 6, 8, 10, … are structurally valid but not yet enabled.

The P2P-layer `check_data()` uses `verify_value()`, which accepts these future values for both lock and type scripts. So a transaction with a type script carrying `hash_type = 6` (Data3):

1. Passes `check_data()` (6 is even → `verify_value(6)` = true)
2. Passes `ScriptHashTypeVerifier` (only lock script is checked)
3. Enters the tx-pool verify queue
4. Fails during `ContextualTransactionVerifier` → `select_version()` → `ScriptError::InvalidScriptHashType`

The `select_version()` and `extract_script_and_dep_index()` functions both have a catch-all arm that returns `ScriptError::InvalidScriptHashType` for any `ScriptHashType` variant not explicitly handled (i.e., Data3 and above).

---

### Impact Explanation

- **Unnecessary script execution**: Transactions with invalid type script `hash_type` values bypass the cheap non-contextual gate and force the node to spin up the CKB-VM scheduler before being rejected. This wastes CPU cycles proportional to the number of such transactions submitted.
- **Tx-pool verify queue pollution**: These transactions occupy slots in the async verify queue (`enqueue_verify_queue`) before being evicted, degrading throughput for legitimate transactions.
- **Peer banning inconsistency**: For lock scripts with invalid `hash_type`, the peer is banned immediately at the non-contextual stage (`Reject::Verification(TransactionError::ScriptHashTypeNotPermitted)`). For type scripts with invalid `hash_type`, the peer is only banned after script execution completes (`Reject::Verification(ScriptError::InvalidScriptHashType)`). An attacker can exploit this asymmetry to force script execution before the ban takes effect.
- **Broken distinction**: The `ScriptHashTypeVerifier` is the canonical non-contextual gate for hash_type validity. Its failure to cover type scripts means the distinction between "valid" and "invalid" hash_types is incomplete, analogous to the SubExecutor bug where `erc20TokensValid` was set by a trivial address-zero check rather than proper token validation.

---

### Likelihood Explanation

Any unprivileged RPC caller (`send_transaction`) or P2P tx-pool submitter (relay) can craft a transaction with a type script `hash_type = 6` (Data3). The transaction is structurally valid (passes `check_data()`), requires no special privileges, and can be submitted repeatedly. The attacker only needs to pay the minimum fee for each transaction, making this a low-cost resource-exhaustion vector.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's type script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Check lock script hash_type (existing)
        check_hash_type(output.lock().hash_type())?;

        // Check type script hash_type (missing today)
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}

fn check_hash_type(hash_type: packed::Byte) -> Result<(), Error> {
    if let Ok(ht) = TryInto::<ScriptHashType>::try_into(hash_type) {
        let val: u8 = ht.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err(TransactionError::InvalidScriptHashType { hash_type }.into());
    }
    Ok(())
}
```

Also update the `NonContextualTransactionVerifier` docstring to say *"lock and type"* rather than just *"lock"*.

### Proof of Concept

**Objective:** Demonstrate that a transaction with a type script using `hash_type = Data3` (value 6, not in `ENABLED_SCRIPT_HASH_TYPE`) passes `ScriptHashTypeVerifier` without error.

**Procedure:**

```rust
#[test]
pub fn test_not_enabled_hash_type_output_type_script_bypasses_verifier() {
    use ckb_types::{
        core::{ScriptHashType, TransactionBuilder},
        packed::{CellOutput, Script, ScriptOpt},
        prelude::*,
    };
    use crate::transaction_verifier::ScriptHashTypeVerifier;

    // Build a transaction whose output has:
    //   - lock script: valid hash_type = Data (0)
    //   - type script: future hash_type = Data3 (6), NOT in ENABLED_SCRIPT_HASH_TYPE
    let valid_lock = Script::default(); // hash_type = Data = 0
    let invalid_type = Script::default()
        .as_builder()
        .hash_type(ScriptHashType::Data3) // value = 6, not enabled
        .build();

    let output = CellOutput::new_builder()
        .lock(valid_lock)
        .type_(ScriptOpt::new_builder().set(Some(invalid_type)).build())
        .build();

    let transaction = TransactionBuilder::default()
        .output(output)
        .build();

    let verifier = ScriptHashTypeVerifier::new(&transaction);

    // BUG: this should return Err(ScriptHashTypeNotPermitted { hash_type: 6 })
    // but it returns Ok(()) because type scripts are never checked
    assert!(verifier.verify().is_ok(), "type script hash_type bypass confirmed");
}
```
