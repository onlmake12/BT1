### Title
`ScriptHashTypeVerifier` Validates Lock Script `hash_type` But Omits Type Script `hash_type` — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces permitted `hash_type` values only on the **lock** script of each transaction output. The **type** script of the same output is never checked. An unprivileged transaction sender can craft an output whose type script carries a non-permitted or invalid `hash_type` byte, bypassing the non-contextual gate entirely and injecting a malformed script reference into the tx-pool and, if mined, into a block.

---

### Finding Description

`ScriptHashTypeVerifier` is one of the six sub-verifiers composing `NonContextualTransactionVerifier`. Its purpose is to reject transactions that reference script code via a `hash_type` value that the current consensus rules do not permit. [1](#0-0) 

The verifier iterates over every output and reads `output.lock().hash_type()`: [2](#0-1) 

The loop body calls `output.lock().hash_type()` and checks it against `ENABLED_SCRIPT_HASH_TYPE`. It never calls `output.type_()` at all. Because `type_()` returns an `Option<Script>`, a non-`None` type script with an arbitrary `hash_type` byte passes through `NonContextualTransactionVerifier::verify()` without any check: [3](#0-2) 

The parallel in the reference report is exact:

| Reference report | CKB analog |
|---|---|
| `_interactions` — validated by `validateOrder()` | `lock` script `hash_type` — checked by `ScriptHashTypeVerifier` |
| `_hooks` — **not** validated | `type` script `hash_type` — **not** checked |

The `check_data` path in the packed-type layer also does not fill this gap; it only validates `dep_type` on cell-deps and output field lengths: [4](#0-3) 

---

### Impact Explanation

A transaction output whose type script carries a non-permitted `hash_type` byte (e.g., a reserved value outside `{0,1,2,4}`) passes `NonContextualTransactionVerifier` and enters the tx-pool. If included in a block, every node must attempt to execute the type script during contextual verification. Depending on how the VM resolves the `hash_type`, this can cause:

- **Consensus split**: nodes running different software versions may accept or reject the block differently, forking the chain.
- **Panic / unhandled error path**: if the VM or script resolver does not gracefully handle an unknown `hash_type`, it may panic or return an internal error rather than a clean `TransactionError`, causing inconsistent block acceptance.
- **Tx-pool pollution**: malformed transactions occupy pool slots and propagate to peers before being rejected at the contextual stage, wasting network and CPU resources.

---

### Likelihood Explanation

The entry path requires only submitting a transaction via the `send_transaction` RPC or the P2P relay protocol — both are reachable by any unprivileged actor. No key, privilege, or majority hash-power is needed. Crafting such a transaction is trivial: set the `hash_type` byte of a type script to any value not in `ENABLED_SCRIPT_HASH_TYPE`. The `outputs_validator` parameter on `send_transaction` only validates well-known script *code hashes*, not `hash_type` values: [5](#0-4) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's type script when it is present, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        let lock_ht: u8 = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?
            .into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&lock_ht) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: lock_ht }.into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht: u8 = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?
                .into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&type_ht) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: type_ht }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Build a transaction with one output whose **lock** script uses `hash_type = 0x00` (valid `data`) and whose **type** script uses `hash_type = 0x03` (reserved, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC with `outputs_validator = "passthrough"`.
3. Observe that `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` returns `Ok(())` — the transaction enters the pool.
4. The lock script's `hash_type` would have been rejected if placed there, confirming the asymmetric validation gap. [6](#0-5) [7](#0-6)

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
