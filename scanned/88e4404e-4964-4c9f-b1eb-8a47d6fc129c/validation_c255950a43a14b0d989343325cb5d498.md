### Title
Inbound Peer Can Promote Ephemeral Source Port to Listen Address via `change_to_listen()` with `listen_port=None` — (`network/src/protocols/discovery/state.rs`, `network/src/protocols/discovery/mod.rs`)

---

### Summary

An unprivileged inbound peer that sends `GetNodes{version: 1, listen_port: None}` causes the victim node to unconditionally promote the peer's ephemeral TCP source port to a `RemoteAddress::Listen` entry. After the first 60-second notify tick, this ephemeral-port address passes all existing guards and is broadcast as a valid listen address to up to 3 other peers, who store it in their peer databases. The address is unreachable, polluting network-wide peer state.

---

### Finding Description

**Step 1 — Inbound session initialization**

For inbound sessions, `SessionState::new` sets:

```rust
RemoteAddress::Init(context.session.address.clone())
``` [1](#0-0) 

The stored address is the attacker's ephemeral TCP source port (e.g., `/ip4/1.2.3.4/tcp/54321`).

**Step 2 — GetNodes handler with `listen_port=None`**

When the attacker sends `GetNodes{version: 1, listen_port: None}`, the `if let Some(port) = listen_port` block is entirely skipped (no `update_port` call, no peer-store insertion). Execution falls through unconditionally to:

```rust
if version >= state::REUSE_PORT_VERSION {
    state.remote_addr.change_to_listen();
}
``` [2](#0-1) 

**Step 3 — `change_to_listen()` promotes the ephemeral address**

```rust
pub(crate) fn change_to_listen(&mut self) {
    if let RemoteAddress::Init(addr) = self {
        *self = RemoteAddress::Listen(addr.clone());
    }
}
``` [3](#0-2) 

Because `listen_port` was `None`, `remote_addr` is still `Init`. `change_to_listen()` clones the ephemeral address verbatim into `Listen` — no port rewriting, no validation.

**Step 4 — `check_timer()` returns the ephemeral address on the first tick**

`last_announce` is `None` at session creation, so `unwrap_or(true)` fires on the very first `notify()` call (60 seconds after connection):

```rust
.unwrap_or(true)
// ...
if let RemoteAddress::Listen(addr) = &self.remote_addr {
    Some(addr)
``` [4](#0-3) 

**Step 5 — Both guards in `notify()` are bypassable**

```rust
.filter(|addr| self.addr_mgr.is_valid_addr(addr))
&& let Some(flags) = self.addr_mgr.node_flags(*id)
``` [5](#0-4) 

- `is_valid_addr` only checks `is_reachable(ip)` — it filters private/loopback IPs, not ephemeral ports. A public attacker IP passes.
- `node_flags` requires identify info, which is populated during the normal P2P handshake. An attacker completing the standard handshake satisfies this.

**Step 6 — Broadcast to 3 peers and propagation into their peer stores**

The ephemeral address is pushed into `announce_list`, then queued into `announce_multiaddrs` for up to 3 sessions. `send_messages` delivers it as `Nodes{announce: true}`. Receiving peers call `add_new_addrs` → `peer_store.add_addr`, permanently storing the unreachable address. [6](#0-5)

### Impact Explanation

The victim node broadcasts the attacker's ephemeral TCP source port as a legitimate listen address to up to 3 peers per announcement cycle. Those peers store it and will attempt outbound connections to it, which will always fail (the ephemeral port is closed). With multiple attacker connections, the pollution scales linearly. Peer stores across the network accumulate unreachable entries, degrading peer discovery quality and wasting connection-slot resources.

---

### Likelihood Explanation

The preconditions are minimal: any node that accepts inbound connections is vulnerable. The attacker needs only to:
1. Establish a TCP connection (standard P2P)
2. Complete the identify handshake (normal protocol flow)
3. Send one `GetNodes{version: 1, listen_port: None}` message
4. Hold the connection for 60 seconds

No special privileges, no PoW, no key material required.

---

### Recommendation

The root cause is that `change_to_listen()` is called unconditionally when `version >= REUSE_PORT_VERSION`, even when `listen_port=None`. The SO_REUSEPORT semantic (where the outbound source port equals the listen port) only applies when the peer explicitly advertises a `listen_port`. The fix is to gate `change_to_listen()` on `listen_port` being `Some`:

```rust
// Only promote to Listen if the peer explicitly declared a listen port
if let Some(port) = listen_port {
    state.remote_addr.update_port(port);
    // ... existing add_new_addr logic ...
} else if version >= state::REUSE_PORT_VERSION {
    // Only valid for outbound sessions where SO_REUSEPORT is in use;
    // for inbound sessions, the source port is ephemeral — do not promote.
    if context.session.ty.is_outbound() {
        state.remote_addr.change_to_listen();
    }
}
```

Alternatively, `change_to_listen()` should be restricted to outbound sessions only, since for inbound sessions the remote's source port is never a stable listen address.

### Proof of Concept

```
1. Attacker at /ip4/1.2.3.4/tcp/54321 connects inbound to victim node.
   → victim: remote_addr = Init(/ip4/1.2.3.4/tcp/54321)

2. Attacker completes identify handshake (populates node_flags).

3. Attacker sends: GetNodes { version: 1, listen_port: None, count: 1, required_flags: 0 }
   → listen_port is None → update_port() skipped
   → version(1) >= REUSE_PORT_VERSION(1) → change_to_listen() called
   → remote_addr = Listen(/ip4/1.2.3.4/tcp/54321)   ← ephemeral port, no rewrite

4. After 60 seconds, notify() fires:
   → check_timer(): last_announce=None → unwrap_or(true) → returns Listen addr
   → is_valid_addr(1.2.3.4): is_reachable(1.2.3.4) = true → passes
   → node_flags(session_id): Some(...) → passes
   → announce_list = [(/ip4/1.2.3.4/tcp/54321, flags)]

5. announce_list broadcast to ≤3 peers as Nodes{announce:true}.
   → receiving peers: add_new_addrs → peer_store.add_addr(/ip4/1.2.3.4/tcp/54321)
   → future connection attempts to port 54321 fail (ephemeral, closed)
```
