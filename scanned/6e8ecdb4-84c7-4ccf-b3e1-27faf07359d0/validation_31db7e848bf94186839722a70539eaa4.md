### Title
Incorrect `CKB2021_START_EPOCH` Constant Causes Mainnet/Testnet Consensus Divergence — (File: `util/constant/src/hardfork/mainnet.rs`, `util/constant/src/hardfork/testnet.rs`)

---

### Summary

The production constant `CKB2021_START_EPOCH` in both `util/constant/src/hardfork/mainnet.rs` and `util/constant/src/hardfork/testnet.rs` has been silently changed from the historically correct activation epoch to `0`. The original correct values are preserved only as commented-out lines. This causes the node to apply CKB2021 consensus rules (RFC 0029, 0030, 0031, 0036, 0038) from genesis block instead of the correct activation epoch, producing a consensus split against the real mainnet and testnet chains.

---

### Finding Description

In `util/constant/src/hardfork/mainnet.rs`:

```rust
// pub const CKB2021_START_EPOCH: u64 = 5414;   // correct historical value
pub const CKB2021_START_EPOCH: u64 = 0;          // active value — wrong
```

In `util/constant/src/hardfork/testnet.rs`:

```rust
// pub const CKB2021_START_EPOCH: u64 = 3113;   // correct historical value
pub const CKB2021_START_EPOCH: u64 = 0;          // active value — wrong
```

The CHANGELOG explicitly records the correct values:
- `v0.103.0`: *"Activate ckb2021 in mainnet since epoch 5414"*
- `v0.101.0`: *"Activate ckb2021 in testnet since epoch 3113"*

These constants are consumed in two places:

**1. `spec/src/hardfork.rs` — `complete_mainnet()` / `complete_testnet()`:**

```rust
pub fn complete_mainnet(&self) -> Result<HardForks, String> {
    let mut ckb2021 = CKB2021::new_builder();
    ckb2021 = self.update_2021(
        ckb2021,
        mainnet::CKB2021_START_EPOCH,                        // = 0 (wrong)
        mainnet::RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH, // = 5414 (correct)
    )?;
    ...
}
```

**2. `util/types/src/core/hardfork/ckb2021.rs` — `CKB2021::new_mirana()`:**

```rust
pub fn new_mirana() -> Self {
    Self::new_builder()
        .rfc_0028(hardfork::mainnet::RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH) // 5414
        .rfc_0029(hardfork::mainnet::CKB2021_START_EPOCH)  // 0 — wrong
        .rfc_0030(hardfork::mainnet::CKB2021_START_EPOCH)  // 0 — wrong
        .rfc_0031(hardfork::mainnet::CKB2021_START_EPOCH)  // 0 — wrong
        .rfc_0032(hardfork::mainnet::RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH) // 5414
        .rfc_0036(hardfork::mainnet::CKB2021_START_EPOCH)  // 0 — wrong
        .rfc_0038(hardfork::mainnet::CKB2021_START_EPOCH)  // 0 — wrong
        .build().unwrap()
}
```

`HardForks::new_mirana()` is used as the default `hardfork_switch` in `ConsensusBuilder::new()` (`spec/src/consensus.rs` line 298), meaning every mainnet node built from this codebase inherits the wrong activation epoch.

The affected RFCs and their rule direction:
| RFC | Rule change | Effect of activating at epoch 0 |
|-----|-------------|----------------------------------|
| RFC 0029 | More permissive (multiple cell dep matches) | Accepts more txs in historical range |
| RFC 0030 | **Stricter** (since epoch index < length) | Rejects historical txs valid under pre-2021 rules |
| RFC 0031 | Structural (variable-length header field) | Misinterprets historical block structure |
| RFC 0036 | More permissive (remove header deps immature rule) | Accepts more txs in historical range |
| RFC 0038 | **Stricter** (dep expansion limit) | Rejects historical txs valid under pre-2021 rules |

---

### Impact Explanation

A node running this code applies RFC 0030 and RFC 0038 stricter validation rules to all historical blocks from genesis. Historical mainnet blocks in epochs 0–5413 were produced and validated under pre-2021 rules. Any such block containing a transaction with a `since` epoch field where `index >= length` (valid pre-2021, invalid under RFC 0030) or a transaction exceeding the dep expansion limit (valid pre-2021, invalid under RFC 0038) will be **rejected** by this node. This causes the node to fail to sync the canonical mainnet chain from genesis, producing a hard consensus split. The node cannot participate correctly in mainnet consensus.

---

### Likelihood Explanation

Any operator who builds and runs this node against mainnet or testnet will encounter the divergence as soon as the sync process reaches a historical block that exercises the affected validation paths. The root cause is a production constant file with the correct value commented out and replaced with `0`. There is no runtime guard, assertion, or test that cross-checks `CKB2021_START_EPOCH` against the known historical activation epoch — the exact gap the external report's vulnerability class describes.

---

### Recommendation

Restore the correct activation epoch values:

```rust
// util/constant/src/hardfork/mainnet.rs
pub const CKB2021_START_EPOCH: u64 = 5414;

// util/constant/src/hardfork/testnet.rs
pub const CKB2021_START_EPOCH: u64 = 3113;
```

Additionally, add compile-time or test-time assertions that cross-check `CKB2021_START_EPOCH` against `RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH` for mainnet (they must be equal per the historical activation), analogous to the cross-layer constant verification tests recommended in the external report.
