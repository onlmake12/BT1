Audit Report

## Title
Missing rate limiting for `GetHeaders` in `Synchronizer` allows CPU/IO amplification by any connected peer — (`sync/src/synchronizer/get_headers_process.rs`)

## Summary
The `Synchronizer` protocol handler has no per-peer rate limiter for `GetHeaders` messages, unlike the `Relayer` which enforces 30 req/s per `(PeerIndex, message_type)`. Any connected peer can flood `GetHeaders` at an unbounded rate post-IBD, triggering up to 2,101 RocksDB reads per message (101 in `locate_latest_common_block` + 2,000 in `get_locator_response`), degrading block sync throughput for all legitimate peers.

## Finding Description
`GetHeadersProcess::execute` in `sync/src/synchronizer/get_headers_process.rs` (L36–99) performs two expensive DB operations per message: `locate_latest_common_block` (up to 101 hash lookups via `snapshot.get_block_number`) and `get_locator_response` (up to 2,000 sequential `store().get_block_header` reads). The call to `self.synchronizer.peers().getheaders_received(self.peer)` at L77 is a confirmed no-op stub (`sync/src/types/mod.rs` L897–899: `// TODO:`). The `Synchronizer` struct (`sync/src/synchronizer/mod.rs` L357–362) has no `rate_limiter` field and `try_process` (L381–422) performs no rate check before dispatching `GetHeadersProcess`. By contrast, `Relayer::try_process` (`sync/src/relayer/mod.rs` L106–123) checks `self.rate_limiter.check_key(&(peer, message.item_id()))` before every dispatch. The only existing guard is the `MAX_LOCATOR_SIZE` (101) check and the IBD guard; neither limits message frequency. The `GetHeadersMissCommonAncestors` ban path is bypassed because an attacker includes the genesis hash as the last locator entry, causing `locate_latest_common_block` to return `Some(0)` (genesis block number) and proceed to `get_locator_response`, returning `Status::ok()` — no ban triggered.

## Impact Explanation
Sustained flooding saturates the RocksDB I/O path and the `tokio::task::block_in_place` thread pool used for sync message processing (`sync/src/synchronizer/mod.rs` L398), degrading block download throughput for all legitimate peers on the targeted node. Impact: **Low (501–2000 points) — measurable sync performance degradation**, matching the allowed CKB bounty impact "Any other important performance improvements for CKB."

## Likelihood Explanation
Any connected peer (inbound or outbound, no special flag, no PoW, no key) can execute this. The only required knowledge is the genesis hash, which is public. The attacker needs a single TCP connection and a tight send loop. The attack is repeatable and persistent with no self-limiting mechanism.

## Recommendation
Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to the `Synchronizer` struct mirroring the one in `Relayer` (30 req/s keyed by `(PeerIndex, message_item_id)`), and check it at the top of `try_process` before dispatching any message. Additionally, implement `getheaders_received` to track per-peer `GetHeaders` budget and enforce a hard cap.

## Proof of Concept
1. Connect a peer to a non-IBD CKB node (tip age < 24 hours).
2. In a tight loop, send `GetHeaders` with `block_locator_hashes = [<any valid tip hash>, <genesis hash>]` (2 entries, well within `MAX_LOCATOR_SIZE = 101`).
3. Each message passes the locator size check, bypasses the IBD guard, causes `locate_latest_common_block` to return `Some(0)` (genesis found), then calls `get_locator_response` (up to 2,000 DB reads), and spawns an async send task — returning `Status::ok()` with no ban.
4. Benchmark legitimate peer sync throughput (blocks/s) with and without the flood to assert measurable degradation.
