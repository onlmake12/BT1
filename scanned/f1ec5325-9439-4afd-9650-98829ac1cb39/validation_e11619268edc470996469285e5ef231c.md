Audit Report

## Title
`MaturityVerifier::verify()` Omits Cellbase Maturity Check on `resolved_dep_groups` — (`File: verification/src/transaction_verifier.rs`)

## Summary
`MaturityVerifier::verify()` enforces the cellbase maturity rule on `resolved_inputs` and `resolved_cell_deps` but never iterates `resolved_dep_groups`. When a `CellDep` with `dep_type = DepGroup` is resolved, the container cell is placed in `resolved_dep_groups`, not `resolved_cell_deps`. An immature cellbase used as a dep-group container therefore bypasses the maturity gate entirely, violating a core consensus invariant and allowing a miner to make their block reward cell usable as a live dependency before the maturity period elapses.

## Finding Description
`ResolvedTransaction` holds three distinct `Vec<CellMeta>` collections: [1](#0-0) 

`resolve_transaction_dep` routes cells based on `dep_type`: when `dep_type == DepGroup`, the container cell is pushed to `resolved_dep_groups` and the cells it points to are pushed to `resolved_cell_deps`: [2](#0-1) 

`MaturityVerifier::verify()` applies the `cellbase_immature` closure only to `resolved_inputs` and `resolved_cell_deps`, with no corresponding check on `resolved_dep_groups`: [3](#0-2) 

The closure logic itself is correct: [4](#0-3) 

The struct doc-comment explicitly states the intent covers all dep types ("If input or dep prev is cellbase, check that it's matured"), but the implementation leaves `resolved_dep_groups` unchecked: [5](#0-4) 

**Exploit path:**
1. A miner produces block N whose cellbase output (index 0) encodes a valid molecule `OutPointVec` pointing to any live, mature cells.
2. Before `cellbase_maturity` epochs elapse, an unprivileged user submits a transaction via RPC with `cell_deps = [{ out_point: <cellbase_outpoint>, dep_type: "dep_group" }]`.
3. `resolve_transaction_dep` places the immature cellbase cell into `resolved_dep_groups`; the mature sub-cells go into `resolved_cell_deps`.
4. `MaturityVerifier::verify()` iterates `resolved_inputs` (no cellbase) and `resolved_cell_deps` (mature — passes). `resolved_dep_groups` is never touched.
5. The transaction clears maturity verification and enters the tx-pool / gets committed.

Replacing `dep_type` with `code` causes the same cellbase to land in `resolved_cell_deps`, where the check fires and correctly rejects the transaction — confirming the gap is specific to the dep-group path.

## Impact Explanation
The cellbase maturity rule is a consensus-level invariant. Bypassing it for dep-group container cells allows an immature cellbase output to be referenced as a live dependency in committed blocks. Because all nodes run the same flawed `MaturityVerifier`, they all accept the transaction silently — no fork occurs, but the protocol rule is violated chain-wide. A miner can immediately make their block reward cell usable as a dep-group dependency, circumventing the economic delay the maturity rule is designed to enforce. This matches the allowed impact: **Vulnerabilities which could easily damage CKB economy (Critical, 15001–25000 points)**.

## Likelihood Explanation
Preconditions are minimal: the attacker needs one cellbase output (reachable by any miner, including on a private fork for testing) whose data field encodes a valid `OutPointVec`. The transaction sender is any unprivileged RPC caller. No privileged keys, insider access, or victim mistakes are required. The gap has existed since dep-group support was introduced and is not gated by any hardfork switch.

## Recommendation
Add a third maturity check in `MaturityVerifier::verify()` for `resolved_dep_groups`, mirroring the existing checks:

```rust
if let Some(index) = self
    .transaction
    .resolved_dep_groups
    .iter()
    .position(cellbase_immature)
{
    return Err(TransactionError::CellbaseImmaturity {
        inner: TransactionErrorSource::CellDeps,
        index,
    }
    .into());
}
```

This should be inserted between the `resolved_cell_deps` check and the final `Ok(())` in `verify()`. [6](#0-5) 

## Proof of Concept
1. Mine block N; craft the cellbase output so its `data` field is a valid molecule `OutPointVec` pointing to two mature live cells (e.g., genesis system cells).
2. Before `cellbase_maturity` epochs elapse, submit via RPC:
   ```json
   {
     "cell_deps": [{
       "out_point": { "tx_hash": "<cellbase_tx_hash_of_block_N>", "index": "0x0" },
       "dep_type": "dep_group"
     }],
     ...
   }
   ```
3. Observe: the transaction is accepted into the tx-pool and committed. `MaturityVerifier` does not return `CellbaseImmaturity` because `resolved_dep_groups` is never iterated.
4. Confirm: replacing `dep_type` with `code` causes the same cellbase to be placed in `resolved_cell_deps`, where the check fires and the transaction is correctly rejected.
5. A unit test can be written in `verification/src/tests/transaction_verifier.rs` mirroring the existing `MaturityVerifier` test cases but populating `resolved_dep_groups` with an immature cellbase `CellMeta` — the test will demonstrate that `verify()` currently returns `Ok(())` instead of `Err(CellbaseImmaturity)`.
