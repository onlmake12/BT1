Audit Report

## Title
RBF Extra-Fee Computed on Serialized Size Instead of Transaction Weight, Allowing Cycle-Heavy Replacements to Underpay — (`File: tx-pool/src/pool.rs`)

## Summary

`calculate_min_replace_fee` computes the mandatory RBF extra fee using only the raw serialized byte size of the incoming transaction, while every other fee-rate path in the tx-pool uses the transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). For a cycle-heavy replacement transaction, the weight can be orders of magnitude larger than the serialized size, causing the required extra fee to be severely underestimated and the RBF bump threshold to be effectively bypassed. An unprivileged attacker can exploit this to repeatedly replace mempool transactions at a fraction of the intended marginal cost.

## Finding Description

The weight-based fee metric is defined in `util/types/src/core/tx_pool.rs`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [1](#0-0) 

`TxEntry::fee_rate()` correctly uses this weight: [2](#0-1) 

`FeeRate::fee()` multiplies the stored rate by whatever `weight` argument is passed: [3](#0-2) 

However, `calculate_min_replace_fee` passes only `size as u64` — the raw serialized byte count — ignoring cycles entirely: [4](#0-3) 

The enforcement gate in `check_rbf` calls this function with `entry.size` only: [5](#0-4) 

The same flaw exists in the `min_replace_fee` public helper, which also passes only `tx.size`: [6](#0-5) 

For a transaction with 100 serialized bytes and 70,000,000 cycles:
- `weight = max(100, 70_000_000 × 0.000_170_571_4) ≈ 11,940`
- `extra_rbf_fee (actual) = min_rbf_rate × 100 / 1000`
- `extra_rbf_fee (correct) = min_rbf_rate × 11,940 / 1000` (~119× larger)

The replacement is admitted after paying only ~0.8% of the intended extra fee.

## Impact Explanation

This maps to the **High** bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The RBF extra-fee bump is the primary economic barrier against cheap mempool churn. By crafting cycle-heavy replacement transactions, an attacker can repeatedly displace existing pool transactions while paying a marginal cost ~119× below what the node operator configured. This enables sustained, low-cost mempool cycling that exhausts miner bandwidth, degrades mempool quality, and undermines the fee-rate ordering that miners rely on for block construction — all without any special privilege.

## Likelihood Explanation

Any caller with access to `send_transaction` (RPC or P2P) can trigger this. Producing a transaction with near-maximum cycle consumption and a small serialized body is straightforward for any script author. No leaked keys, admin access, or victim mistakes are required. The cycle count is observable before VM verification, so the attacker can precisely tune the size/cycle ratio to maximize the discount. The attack is repeatable at will.

## Recommendation

Add a `cycles: Cycle` parameter to `calculate_min_replace_fee` and replace the raw `size` argument with the transaction weight:

```rust
fn calculate_min_replace_fee(
    &self,
    conflicts: &[&PoolEntry],
    size: usize,
    cycles: Cycle,
) -> Option<Capacity> {
    let weight = get_transaction_weight(size, cycles);
    let extra_rbf_fee = self.config.min_rbf_rate.fee(weight);
    ...
}
```

Update both call sites to pass `entry.cycles` / `tx.cycles` alongside `entry.size` / `tx.size`. This aligns the RBF admission gate with the weight-based fee-rate metric used in `fee_rate()` and everywhere else in the pool. [7](#0-6) 

## Proof of Concept

1. Configure node with `min_rbf_rate = 1500` shannons/KW, `max_tx_verify_cycles = 70_000_000`.
2. Submit T1: 100-byte body, fee = 10,000 shannons → accepted.
3. Craft T2: same inputs, 100-byte body, 70,000,000 cycles, fee = 10,150 shannons.
   - Node computes: `extra_rbf_fee = 1500 × 100 / 1000 = 150` shannons → check passes.
   - Correct value: `1500 × 11,940 / 1000 = 17,910` shannons → T2 should be rejected.
4. T2 is accepted despite paying only 150 shannons extra instead of the required 17,910 shannons.
5. Repeat with T3 replacing T2 at the same marginal cost to demonstrate sustained cheap cycling.

The discrepancy is directly verifiable by reading `calculate_min_replace_fee` at `tx-pool/src/pool.rs:102–103` against `fee_rate()` at `tx-pool/src/component/entry.rs:115–117`. [8](#0-7) [9](#0-8)

    /// min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }
