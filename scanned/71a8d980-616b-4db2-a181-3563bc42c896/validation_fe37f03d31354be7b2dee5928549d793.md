Audit Report

## Title
Unbounded `pending_delivered` Growth and Uncapped `runtime::spawn` Task Flooding via Crafted Hole-Punching Messages — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
An unprivileged remote peer can exhaust memory and saturate the Tokio async runtime of a victim CKB node by sending crafted `ConnectionRequest` messages with distinct random `from` PeerIds, bypassing all per-`from` guards and filling `pending_delivered` without bound. A follow-up flood of `ConnectionSync` messages referencing those entries triggers an uncapped number of `runtime::spawn` tasks, each running up to 24 concurrent 30-second TCP connection loops, exhausting file descriptors and thread-pool capacity.

## Finding Description

**Phase 1 — Unbounded `pending_delivered` insertion**

`pending_delivered` is declared as a plain `HashMap<PeerId, PendingDeliveredInfo>` with no size cap. [1](#0-0) 

Insertion occurs in `ConnectionRequestProcess::respond_delivered()` after three guards: [2](#0-1) 

**Guard 1 — per-session rate limiter** (30 req/s per `(session_id, msg_item_id)`): this is the only real throttle and caps the insertion rate, not the total size. [3](#0-2) 

**Guard 2 — `forward_rate_limiter`** (1/s per `(from, to, msg_item_id)`): keyed on the attacker-controlled `from` field. Each distinct `from` PeerId creates a fresh key, so this guard is completely bypassed when the attacker uses a new random `from` per message. [4](#0-3) 

**Guard 3 — `HOLE_PUNCHING_INTERVAL` check** (2-minute dedup on `from_peer_id`): also keyed on `from`, so it is equally bypassed by distinct `from` PeerIds. [5](#0-4) 

The `from` field need only be a syntactically valid `PeerId` (e.g., `PeerId::random()`); the `to` field is the victim's public peer ID; `listen_addrs` need only contain one valid TCP/IPv4 address (e.g., an unreachable black-hole IP). At 30 req/s over the 5-minute `TIMEOUT` window, a single session inserts **30 × 300 = 9,000 entries** before `notify()` cleans them up. [6](#0-5) 

**Phase 2 — Uncapped `runtime::spawn` task flooding**

When a `ConnectionSync` message arrives with `to` = victim and `route` empty, and `from` matches a `pending_delivered` entry, the code unconditionally calls `runtime::spawn` with no cap on concurrent tasks: [7](#0-6) 

Each spawned task calls `select_ok(tasks)` where `tasks` contains up to 24 `try_nat_traversal` futures running concurrently. Each `try_nat_traversal` loops for up to 30 seconds, creating a new TCP socket and attempting a connection every ~200 ms: [8](#0-7) 

The `forward_rate_limiter` for `ConnectionSync` is also keyed by `(from, to, msg_item_id)`, so distinct `from` PeerIds bypass it identically: [9](#0-8) 

At 30 `ConnectionSync` messages/s, after 300 seconds there are **9,000 concurrent tasks**, each holding up to 24 TCP connection futures = **216,000 concurrent `try_nat_traversal` futures**, each creating a new TCP socket every ~200 ms for 30 seconds.

**Compounding factor — `forward_rate_limiter` unbounded growth**

`forward_rate_limiter` is a `HashMapStateStore<(PeerId, PeerId, u32)>` with no size cap. `retain_recent()` is called only on session disconnect, not during normal operation, so 9,000 distinct `from` PeerIds accumulate 9,000 entries in the rate limiter itself. [10](#0-9) 

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

- **File descriptor exhaustion**: 216,000 concurrent `try_nat_traversal` futures each create a new `TcpSocket` every ~200 ms. The OS file descriptor limit (typically 1,024–65,536) is exhausted within seconds, causing the node process to crash or fail to accept any new connections.
- **Memory exhaustion**: `pending_delivered` grows to ~9,000 entries (each holding up to 24 `Multiaddr` objects ≈ 2–3 KB/entry ≈ 18–27 MB), plus the `forward_rate_limiter` HashMap, plus heap allocations for 9,000 spawned task futures.
- **Async runtime saturation**: Tens of thousands of concurrent Tokio tasks starve the thread pool, blocking all legitimate protocol processing (sync, block propagation, transaction relay) across all sessions.
- **Scales linearly**: Multiple attacker sessions multiply the effect; `pending_delivered` and `forward_rate_limiter` are per-`HolePunching` instance (one per node), so the attack compounds.

## Likelihood Explanation

The attack requires only a standard P2P connection — no privileges, no PoW, no leaked keys. The `from` field is arbitrary valid bytes (`PeerId::random()`); the victim's peer ID is public information from the P2P handshake; `listen_addrs` need only be syntactically valid TCP addresses. The protobuf schema is published. The per-session rate limit of 30 req/s is the only real throttle and is insufficient to prevent the attack over the 5-minute cleanup window. The attack is repeatable and automatable with a simple script.

## Recommendation

1. **Cap `pending_delivered` size**: Reject insertion when `pending_delivered.len()` exceeds a small constant (e.g., 256 or `max_outbound`) in `respond_delivered()`.
2. **Cap spawned tasks**: Maintain an atomic counter of in-flight NAT traversal tasks; refuse to spawn new ones beyond a fixed limit (e.g., `max_outbound × ADDRS_COUNT_LIMIT`).
3. **Global rate limit on unique `from` PeerIds**: Add a counter or bloom filter to bound the number of distinct `from` PeerIds accepted per cleanup interval, independent of the per-`(from,to)` rate limiter.
4. **Periodic `retain_recent()` on `forward_rate_limiter`**: Call it in `notify()` rather than only on disconnect to prevent unbounded growth of the rate limiter itself.

## Proof of Concept

```
1. Attacker connects to victim (single P2P session).

2. For i in 1..9000:
     Send ConnectionRequest {
       from = PeerId::random(),        // distinct each time — bypasses forward_rate_limiter and HOLE_PUNCHING_INTERVAL
       to   = victim_peer_id,          // public, from handshake
       listen_addrs = ["/ip4/1.2.3.4/tcp/9999"],  // unreachable TCP addr, passes filter
       route = [],
       max_hops = 6,
     }
     // Passes per-session rate limiter (30/s)
     // Passes forward_rate_limiter (new key each time)
     // Passes HOLE_PUNCHING_INTERVAL check (new from each time)
     // Inserts into pending_delivered
     sleep(1/30 s)

3. Record all 9000 from PeerIds used above.

4. For each from_i in recorded_from_peerids:
     Send ConnectionSync {
       from  = from_i,
       to    = victim_peer_id,
       route = [],
     }
     // Passes forward_rate_limiter (distinct key)
     // Finds pending_delivered[from_i] -> spawns runtime::spawn task
     // Task runs 24 concurrent try_nat_traversal futures for 30s each,
     // creating TCP sockets to 1.2.3.4:9999 every ~200ms

5. Observe: victim file descriptors exhausted; Tokio thread pool saturated;
   legitimate sync/block messages stop being processed; node crashes or hangs.
```
