Audit Report

## Title
Stale Aggregate Counter Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the aggregate counters `total_tx_size` and `total_tx_cycles` are snapshotted into local variables before `check_and_record_ancestors` is called. When that function evicts transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in place. However, lines 218–219 then unconditionally overwrite those fields with the stale pre-eviction snapshot, permanently inflating the counters by the sum of all evicted transactions' sizes and cycles. This corrupts the pool's admission and eviction logic for the lifetime of the node process.

## Finding Description

**Root cause — stale snapshot written after in-place mutation:**

`add_entry` captures a local snapshot at lines 210–211: [1](#0-0) 

`check_and_record_ancestors` is then called at line 213: [2](#0-1) 

Inside `check_and_record_ancestors`, when `ancestors_count > max_ancestors_count` and the cell-ref eviction branch is taken (line 603), `remove_entry_and_descendants` is called in a loop: [3](#0-2) 

`remove_entry_and_descendants` calls `remove_entry` for each removed id: [4](#0-3) 

`remove_entry` calls `update_stat_for_remove_tx`, which **directly mutates** `self.total_tx_size` and `self.total_tx_cycles`: [5](#0-4) [6](#0-5) 

After `check_and_record_ancestors` returns, lines 218–219 overwrite the now-correct `self.total_tx_size` / `self.total_tx_cycles` with the stale pre-eviction snapshot: [7](#0-6) 

**Net arithmetic corruption:**
- Before `add_entry`: `self.total_tx_size = T`
- Snapshot at line 210: `total_tx_size = T + new_size`
- After evictions (correct): `self.total_tx_size = T − Σ(evicted_sizes)`
- After line 218 (stale overwrite): `self.total_tx_size = T + new_size`
- Inflation = `new_size + Σ(evicted_sizes)` instead of just `new_size`

**Existing guards are insufficient:** `update_stat_for_remove_tx` has an underflow guard that recomputes from scratch on underflow, but it is never reached here — the overwrite happens *after* the correct decrement, not during it. [8](#0-7) 

## Impact Explanation

The inflated `total_tx_size` directly drives two critical pool-management paths:

1. **Spurious eviction loop** — `limit_size` loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter causes the loop to evict additional valid, fee-paying transactions that should remain in the pool. [9](#0-8) 

2. **False `Reject::Full` for legitimate transactions** — `updated_stat_for_add_tx` uses `self.total_tx_size` to gate admission. An inflated counter causes new submissions to be rejected even when the real pool is well within capacity. [10](#0-9) 

An attacker who can submit transactions to a node (via `send_transaction` RPC or P2P relay) can repeatedly trigger this path, causing the node to permanently reject legitimate transactions and evict valid ones until it is restarted. Applied across multiple nodes, this degrades the network's ability to propagate and confirm transactions, matching the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs.*

## Likelihood Explanation

The eviction branch in `check_and_record_ancestors` fires whenever:
- A submitted transaction has more than `max_ancestors_count` ancestors reachable through cell-dep references, AND
- Removing the cell-ref parents would bring the count within the limit.

This requires no special privilege, key, or majority hashpower. Any unprivileged submitter can craft a transaction that uses a cell output already referenced as a `cell_dep` by many pooled transactions, then submits a new transaction that *spends* that same output. The integration test `TxPoolLimitAncestorCount` demonstrates this path is reachable with 2000 cell-ref transactions and a single spending transaction, evicting 1002 entries in one call. [11](#0-10) 

The corruption is permanent per node process and can be repeated to inflate counters further on each trigger.

## Recommendation

Move the snapshot computation to **after** `check_and_record_ancestors` completes, so it reads the already-correct (post-eviction) `self.total_tx_size`:

```rust
// In add_entry, replace lines 210-219 with:
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size already reflects evictions; add only the new entry
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Note that `updated_stat_for_add_tx` must also be moved after the eviction step so its overflow check uses the post-eviction baseline. [12](#0-11) 

## Proof of Concept

1. Configure a node with default `max_ancestors_count = 125`.
2. Submit a root transaction `tx_a` with one output used as a `cell_dep`.
3. Submit 200 transactions each referencing `tx_a`'s output as a `cell_dep` (cell-ref parents). Pool now has 201 entries; `total_tx_size = S`.
4. Submit a new transaction that **spends** `tx_a`'s output. Its ancestor count = 202 > 125, but `202 − 200 = 2 ≤ 125`, so the eviction branch fires and evicts 77 cell-ref transactions.
5. Each eviction correctly decrements `self.total_tx_size`. After evictions: `self.total_tx_size = S − Σ(77 evicted sizes)`.
6. Line 218 overwrites with stale snapshot `S + new_entry_size`, inflating by `Σ(77 evicted sizes)`.
7. Call `tx_pool_info` RPC — `total_tx_size` is visibly inflated.
8. Submit any new transaction — it is rejected with `Reject::Full` even though the real pool is under capacity.
9. Observe `limit_size` evicting additional valid transactions from the pool.

The existing integration test `TxPoolLimitAncestorCount` exercises the eviction path; adding an assertion that `pool_info.total_tx_size == Σ(actual entry sizes)` after the final submission would reproduce the invariant violation. [13](#0-12)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }
