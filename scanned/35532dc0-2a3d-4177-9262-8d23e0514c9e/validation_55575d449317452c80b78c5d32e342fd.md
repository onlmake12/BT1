### Title
`NetworkState::active` Flag Not Enforced in `CKBHandler::poll` — (`network/src/protocols/mod.rs`)

---

### Summary

The `NetworkState.active` flag, toggled via the `set_network_active` RPC, is checked in every `CKBHandler` callback (`connected`, `disconnected`, `received`, `notify`) **except** `poll`. Protocol handlers therefore continue executing their periodic polling tasks — and can send messages to peers — even after the network has been administratively deactivated.

---

### Finding Description

`NetworkState` holds an `AtomicBool` named `active` (initialized to `true`): [1](#0-0) 

The RPC `set_network_active(false)` stores `false` into this flag: [2](#0-1) 

The `CKBHandler` struct (the tentacle `ServiceProtocol` adapter) correctly gates four of its five callbacks behind `is_active()`:

| Callback | Guards with `is_active()`? |
|---|---|
| `connected` | ✅ line 327 |
| `disconnected` | ✅ line 351 |
| `received` | ✅ line 366 |
| `notify` | ✅ line 387 |
| **`poll`** | ❌ **missing** |

The `poll` implementation: [3](#0-2) 

There is no `if !self.network_state.is_active() { return None; }` guard. The tentacle service calls `poll` on every registered protocol handler on each service tick, so all protocol handlers (sync, relay, discovery, etc.) continue executing their periodic work — including sending messages to connected peers — regardless of the `active` flag.

Compare with the four guarded callbacks: [4](#0-3) [5](#0-4) 

---

### Impact Explanation

The documented intent of `set_network_active(false)` is to **"discard any received messages"** and halt all p2p network activity: [6](#0-5) 

Because `poll` is not gated, protocol handlers continue to:
- Request blocks from peers (sync protocol periodic poll)
- Broadcast pending transactions (relay protocol periodic poll)
- Discover and dial new peers (discovery protocol periodic poll)

The "disable networking" control plane operation is therefore **incomplete**: inbound message processing stops, but outbound activity driven by `poll` continues unimpeded. An operator who calls `set_network_active(false)` to isolate a node (e.g., during an incident response or emergency maintenance) cannot rely on the flag to actually stop the node from communicating with the network.

---

### Likelihood Explanation

Any local RPC caller with access to the `Net` RPC module can invoke `set_network_active`. The `Net` module is enabled by default. The inconsistency is structural — every service tick the tentacle runtime calls `poll` on every registered protocol, so the bypass is continuous and unconditional once the flag is set to `false`.

---

### Recommendation

Add the same `is_active()` guard to `CKBHandler::poll` that is already present in the other four callbacks:

```rust
async fn poll(&mut self, context: &mut ProtocolContext) -> Option<()> {
    if !self.network_state.is_active() {
        return None;
    }
    let nc = DefaultCKBProtocolContext { ... };
    self.handler.poll(Arc::new(nc)).await
}
``` [3](#0-2) 

---

### Proof of Concept

1. Start a CKB node with the `Net` RPC module enabled (default).
2. Call `set_network_active(false)` via RPC.
3. Observe via network capture or logging that the node **continues** to send outbound protocol messages (e.g., `GetBlocks`, `SendTransaction`, `GetAddr`) to connected peers on each service tick, driven by the unguarded `poll` callbacks.
4. Contrast with `received`: any inbound message is silently dropped, confirming the flag is only partially enforced.
