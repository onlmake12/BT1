Audit Report

## Title
Tx-Pool Minimum Fee Rate Enforced Only on Serialized Size, Not Actual Transaction Weight — (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` gates pool admission using only the transaction's serialized byte size to compute the minimum acceptable fee, while the protocol's own weight formula is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` resolves the actual cycle count, no weight-based fee rate re-check is performed. An attacker can submit script-heavy (high-cycle), byte-light transactions that pass the size-based gate while carrying an effective fee rate far below `min_fee_rate` when measured by weight, enabling cheap pool flooding.

## Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum fee using only `tx_size`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

The code's own comment acknowledges this is an approximation:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [1](#0-0) 

The actual weight formula, used for scoring and eviction but **not** for admission, is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

The `_process_tx` flow in `tx-pool/src/process.rs` is:

1. `pre_check` → `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` — size-only gate
2. `verify_rtx(...)` → returns `verified.cycles` (actual cycle count now known)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry built with real cycles
4. `submit_entry(...)` — no weight-based fee rate re-check [3](#0-2) 

After step 2, the actual weight is computable but never used to re-validate the fee rate. The `fee_rate()` method on `TxEntry` does use `get_transaction_weight`, but only for sorting and eviction priority — not for admission control. [4](#0-3) 

## Impact Explanation

An attacker can fill the pool's cycle budget with transactions that pay only the size-based minimum fee. For example:

- `tx_size = 500` bytes → `min_fee = 500 × (1000/1000) = 500 shannons` (passes admission)
- `cycles = 50,000,000` → `weight = max(500, 8528) = 8528`
- Weight-based minimum fee would be `8,528 shannons` — ~17× higher

Such transactions enter the pool legitimately, consume the pool's cycle quota cheaply, and are assigned the lowest eviction priority (lowest weight-based fee rate). An attacker can continuously submit them to keep the pool's cycle budget saturated, causing legitimate high-cycle transactions to be rejected or evicted. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

## Likelihood Explanation

Any unprivileged RPC caller with access to `send_transaction` can exploit this. Crafting a high-cycle, low-size transaction requires only a lock script that performs expensive computation (e.g., repeated hashing) with a minimal witness. No privileged access, key material, or majority hashpower is required. The entry path is `rpc/src/module/pool.rs` → `send_transaction` → `_process_tx` → `pre_check` → `check_tx_fee`. The attack is repeatable and cheap.

## Recommendation

After `verify_rtx` returns the actual cycle count in `_process_tx`, compute the true transaction weight and re-validate the fee rate before building the entry:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

This ensures the `min_fee_rate` policy is enforced against the actual transaction weight, not just the serialized size.

## Proof of Concept

1. Deploy a lock script that performs 50,000,000 cycles of computation but requires only a 1-byte witness (small serialized size).
2. Create a transaction spending a cell locked by this script. Serialized size ≈ 500 bytes.
3. Set fee to `min_fee_rate.fee(500) + 1` shannons (just above the size-based minimum, e.g., 501 shannons).
4. Submit via `send_transaction`. The call succeeds: `check_tx_fee` passes because it uses `tx_size = 500`.
5. `verify_rtx` runs the script, consuming 50,000,000 cycles. Actual weight ≈ 8,528.
6. The transaction enters the pool with an effective fee rate of ≈ `501/8528 ≈ 59 shannons/KB`, far below the configured `min_fee_rate = 1000 shannons/KB`.
7. Repeat with many such transactions. Each costs ~501 shannons but consumes 50M cycles of pool quota. The pool's cycle budget fills rapidly, blocking legitimate high-cycle transactions from entering.
