Audit Report

## Title
Unbounded Heap Allocation via Extreme Fee Rate in `do_estimate` — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`do_estimate` derives `max_bucket_index` directly from the highest fee rate among all pending/proposed transactions in the pool, then allocates two `Vec<u64>` of that size with no upper-bound guard. Because the tx-pool enforces only a minimum fee rate and `FeeRate::calculate` uses `saturating_mul`, an attacker can submit a valid transaction with an extreme fee to drive `max_bucket_index` into the hundreds of millions or higher, causing a multi-gigabyte or petabyte heap allocation and an OOM crash when `estimate_fee_rate` is called.

## Finding Description

**Root cause — `FeeRate::calculate` is unbounded.**
`FeeRate::calculate` computes `fee.as_u64().saturating_mul(KW) / weight`, capping only at `u64::MAX`. No maximum fee rate is enforced anywhere. [1](#0-0) 

**No maximum fee rate check in tx admission.**
`check_tx_fee` rejects only transactions whose fee is *below* the minimum; there is no upper bound on the fee a transaction may carry. [2](#0-1) 

**`max_bucket_index_by_fee_rate` is unbounded.**
For any fee rate above 2,000,000 shannons/KW the last match arm applies: `index = (x + t * 11_500) / (100 * t)`. For `x = 1.25 × 10^13` (25,000 CKB fee on a 200-weight tx) this yields `index ≈ 1.25 × 10^8`. For `x = u64::MAX` the index reaches `≈ 1.84 × 10^14`. There is no cap. [3](#0-2) 

**Two uncapped `Vec` allocations of that size.**
`do_estimate` allocates `current_weight_buckets` and `flow_speed_buckets`, both of length `max_bucket_index + 1`, with no guard between the index computation and the allocation. [4](#0-3) [5](#0-4) 

**The extreme fee rate flows directly from the attacker's transaction.**
`estimate_fee_rate` builds `sorted_current_txs` from the live pool state (pending + proposed), sorts descending by fee rate, and passes the slice to `do_estimate`. `max_fee_rate` is taken from the first element — the attacker's transaction. [6](#0-5) [7](#0-6) 

## Impact Explanation

A single RPC call to `estimate_fee_rate` after the attacker's transaction enters the pool causes the node process to attempt a heap allocation proportional to the fee rate. At 25,000 CKB fee on a minimal-weight transaction the allocation is ~1 GB; at 200,000 CKB it is ~8 GB. On any node with less available memory than the attempted allocation, the process is killed by the OS OOM killer. This directly matches the allowed High impact: **"Vulnerabilities which could easily crash a CKB node"** (10001–15000 points). [8](#0-7) 

## Likelihood Explanation

No privileged access is required. Any party that can submit a valid CKB transaction and reach the `estimate_fee_rate` RPC endpoint (available when the `Experiment` module is enabled) can trigger the crash. The cost is non-zero but finite: ~25,000 CKB is sufficient to cause a ~1 GB allocation. The attack is repeatable: after the node restarts, the attacker can resubmit the transaction and call the RPC again. No victim mistake or external dependency is required beyond the `Experiment` module being enabled. [2](#0-1) 

## Recommendation

Add a hard cap on `max_bucket_index` before any allocation in `do_estimate`:

```rust
const MAX_BUCKET_INDEX: usize = 2000; // covers fee rates up to ~200,000,000 shannons/KW
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_BUCKET_INDEX);
```

Alternatively, harden `max_bucket_index_by_fee_rate` itself to return a bounded value regardless of input, so the cap is enforced at the single point of computation rather than at every call site. [3](#0-2) 

## Proof of Concept

```rust
// Demonstrates that max_bucket_index_by_fee_rate is unbounded
use ckb_types::core::FeeRate;
use weight_units_flow::Algorithm;

// 25,000 CKB fee on a 200-weight transaction
let fee_rate = FeeRate::from_u64(12_500_000_000u64); // 1.25e10 shannons/KW
let index = Algorithm::max_bucket_index_by_fee_rate(fee_rate);
// index ≈ 125_000_000 → vec![0u64; 125_000_001] ≈ 1 GB allocation → OOM

assert!(index < 10_000, "index {} is unbounded and will OOM", index); // FAILS
```

Call sequence: `submit_tx` (high fee, small weight) → tx enters pool → `estimate_fee_rate` RPC → `get_all_entry_info` → `do_estimate` → `max_bucket_index_by_fee_rate(extreme_fee_rate)` → `vec![0u64; huge_index + 1]` → OOM crash. [9](#0-8)

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
