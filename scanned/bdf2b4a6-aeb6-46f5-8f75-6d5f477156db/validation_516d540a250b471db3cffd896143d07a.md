### Title
Unauthenticated Miner Notify HTTP Server Allows Any Caller to Redirect Mining Rewards — (File: `miner/src/client.rs`)

---

### Summary

When the CKB miner is configured in "notify mode," it starts an HTTP server that accepts block template pushes from the CKB node. The `handle` function that processes incoming HTTP requests performs no authentication whatsoever. Any attacker who can reach the miner's HTTP notify port can push a crafted `BlockTemplate` containing their own lock script in the cellbase witness, causing the miner to mine blocks whose future rewards are directed to the attacker's address.

---

### Finding Description

**Root cause — `miner/src/client.rs`, lines 358–369:**

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }
    Ok(Response::new(Empty::new()))
}
``` [1](#0-0) 

This `handle` function is the HTTP request handler for the miner's notify server, started in `listen_block_template_notify` when `config.listen` is set: [2](#0-1) 

The server binds to the configured `SocketAddr` and accepts connections from any source. Every incoming HTTP POST body is deserialized as a `BlockTemplate` and immediately passed to `update_block_template`, which replaces the current mining work: [3](#0-2) 

**Contrast with outgoing auth:** The miner's outgoing RPC client (`Rpc::new`) does support HTTP Basic Auth via `parse_authorization` when credentials are embedded in the RPC URL: [4](#0-3) 

There is no equivalent inbound authentication on the notify server. The asymmetry is structural: the miner authenticates itself to the node, but the miner's own HTTP server authenticates nobody.

**How rewards are redirected:**

The `BlockTemplate` contains a `cellbase` transaction. The cellbase witness holds the miner's lock script, which determines who receives the block reward when it is finalized `PROPOSAL_WINDOW.farthest + 1` blocks later. The node's `BlockAssembler` builds this witness from the configured `BlockAssemblerConfig`: [5](#0-4) 

The `RewardVerifier` enforces that the cellbase **output** lock matches the **target block's** previously committed witness lock — but it does not re-verify that the cellbase **witness** in the current block matches the node's own `BlockAssemblerConfig`: [6](#0-5) 

An attacker-supplied template with a different witness lock passes all consensus checks (the output lock is still correct for the target block), but the witness is stored and used to pay the reward in a future block — to the attacker's address.

---

### Impact Explanation

An attacker who can reach the miner's notify HTTP port can:

1. POST a crafted `BlockTemplate` JSON body with their own lock script embedded in the cellbase witness.
2. The miner immediately replaces its current work with the attacker's template.
3. The miner solves PoW and submits the block to the node.
4. The node accepts the block (cellbase output lock is consensus-valid; witness format is valid per `InvalidWitness` check).
5. `PROPOSAL_WINDOW.farthest + 1` blocks later, the block reward is paid to the attacker's lock script.

The attacker steals 100% of the mining reward for every block mined while the fake template is active. The legitimate miner receives nothing for those blocks.

---

### Likelihood Explanation

The notify mode is opt-in (the `listen` field in `ClientConfig` is `Option<SocketAddr>`, defaulting to `None`). However, mining pools and professional mining operations routinely run the CKB node and miner on separate machines, requiring a non-localhost listen address. The `SocketAddr` type accepts any address including `0.0.0.0:8888`. Any attacker with network access to the miner host (e.g., same data center, compromised network path, or misconfigured firewall) can exploit this with a single HTTP POST. No credentials, keys, or privileged access are required. [7](#0-6) 

---

### Recommendation

Add authentication to the miner's notify HTTP server. The simplest approach is to reuse the same HTTP Basic Auth credential already supported for outgoing RPC calls: extract the credential from the configured `rpc_url` and require the same credential on incoming notify requests. The `handle` function should reject requests that do not present a matching `Authorization` header before deserializing or acting on the body.

---

Precondition: miner is configured with `listen = "0.0.0.0:8888"` (or any reachable address).

    async fn listen_block_template_notify(&self, addr: SocketAddr) {
        let listener = TcpListener::bind(addr).await.unwrap();
        let server = auto::Builder::new(TokioExecutor::new());
        let graceful = GracefulShutdown::new();
        let stop_rx: CancellationToken = new_tokio_exit_rx();

        loop {
            let client = self.clone();
            let handle = service_fn(move |req| handle(client.clone(), req));
            tokio::select! {
                conn = listener.accept() => {
                    let (stream, _) = match conn {
                        Ok(conn) => conn,
                        Err(e) => {
                            info!("accept error: {}", e);
                            tokio::time::sleep(std::time::Duration::from_secs(1)).await;
                            continue;
                        }
                    };
                    let stream = hyper_util::rt::TokioIo::new(Box::pin(stream));
                    let conn = server.serve_connection_with_upgrades(stream, handle);

                    let conn = graceful.watch(conn.into_owned());
                    tokio::spawn(async move {
                        if let Err(err) = conn.await {
                            info!("connection error: {}", err);
                        }
                    });
                },
                _ = stop_rx.cancelled() => {
                    info!("Miner client received exit signal. Exit now");
                    break;
                }
            }
        }
        drop(listener);
        graceful.shutdown().await;
    }

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}

    pub(crate) fn build_cellbase_witness(
        config: &BlockAssemblerConfig,
        snapshot: &Snapshot,
    ) -> CellbaseWitness {
        let hash_type: ScriptHashType = config.hash_type.into();
        let cellbase_lock = Script::new_builder()
            .args(config.args.as_bytes())
            .code_hash(&config.code_hash)
            .hash_type(hash_type)
            .build();
        let tip = snapshot.tip_header();

        let mut message = vec![];
        if let Some(version) = snapshot.compute_versionbits(tip) {
            message.extend_from_slice(&version.to_le_bytes());
            message.extend_from_slice(b" ");
        }
        if config.use_binary_version_as_message_prefix {
            message.extend_from_slice(config.binary_version.as_bytes());
        }
        if !config.message.is_empty() {
            message.extend_from_slice(b" ");
            message.extend_from_slice(config.message.as_bytes());
        }

        CellbaseWitness::new_builder()
            .lock(cellbase_lock)
            .message(message)
            .build()
    }
