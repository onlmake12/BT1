### Title
Unbounded DB/CPU Load via `GetTransactionsProof` with No Rate Limiting in `LightClientProtocol` — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

`LightClientProtocol` has no rate limiter. Any unauthenticated peer can send repeated `GetTransactionsProof` messages containing up to 1000 tx hashes each spread across 1000 distinct blocks, forcing the server to perform 1000 full block DB loads, 1000 CBMT merkle proof computations, and one MMR proof generation over 1000 positions per message, with no per-connection or per-message-type throttle.

---

### Finding Description

`GET_TRANSACTIONS_PROOF_LIMIT` is set to 1000: [1](#0-0) 

`GetTransactionsProofProcess::execute` enforces only this count check before proceeding: [2](#0-1) 

For each found tx hash, the handler groups by block hash, then for every unique block it calls `snapshot.get_block()` (full block load from RocksDB), computes a CBMT merkle proof over all transactions in that block, and loads block uncles and extensions: [3](#0-2) 

It then calls `reply_proof`, which calls `mmr.gen_proof(items_positions)` with up to 1000 MMR positions — a logarithmic-in-chain-height DB traversal per position: [4](#0-3) 

The `LightClientProtocol` struct contains only `shared: Shared` — **no rate limiter field exists anywhere in the crate**: [5](#0-4) 

The `received` handler dispatches directly to `try_process` with zero throttling: [6](#0-5) 

Contrast this with `Relayer`, which has an explicit `rate_limiter: RateLimiter<(PeerIndex, u32)>` field and checks it at 30 req/sec per peer per message type before any processing: [7](#0-6) 

---

### Impact Explanation

A single attacker peer can saturate the CKB node's I/O and CPU by sending a continuous stream of `GetTransactionsProof` messages, each requesting 1000 tx hashes confirmed in 1000 distinct blocks. Each message triggers:
- 1000 `get_transaction_info` DB reads (partition step)
- Up to 1000 `get_transaction_with_info` DB reads
- Up to 1000 `get_block` full-block DB reads
- Up to 1000 CBMT merkle proof builds (O(N log N) in block tx count)
- Up to 1000 `get_block_uncles` + `get_block_extension` DB reads
- 1 `mmr.gen_proof(1000 positions)` call (O(1000 × log(chain_height)) DB reads)

The attacker's cost is a single TCP connection and a few hundred bytes per message. The server's cost is thousands of DB reads and CPU operations per message. This can stall block processing and sync on the same node.

---

### Likelihood Explanation

The light client protocol (`SupportProtocols::LightClient`) is a production protocol enabled on full nodes that serve light clients. Any peer that can connect to the node's P2P port can open this protocol session and send `GetTransactionsProof` messages without any authentication or prior state. The 5-minute ban only applies to malformed messages (parse failures), not to well-formed but expensive requests. The attack is trivially scriptable.

---

### Recommendation

Add a per-peer, per-message-type rate limiter to `LightClientProtocol` mirroring the pattern already used in `Relayer`: [8](#0-7) 

Additionally, consider reducing `GET_TRANSACTIONS_PROOF_LIMIT` or adding a secondary limit on the number of distinct blocks that may be referenced in a single request.

---

### Proof of Concept

1. Connect to a CKB node with the light client protocol enabled.
2. Collect 1000 confirmed transaction hashes, each from a different block.
3. In a tight loop, send `GetTransactionsProof` messages with `last_hash` set to the current tip and `tx_hashes` set to all 1000 hashes.
4. Observe that each message causes ~3000+ RocksDB reads and significant CPU usage on the server, while the attacker sends ~300 bytes per message.
5. Measure server CPU/IO saturation vs. attacker bandwidth cost to confirm the asymmetric amplification.

                item_name, peer, ban_time, status

            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }

    /// Init relay protocol handle
    ///
    /// This is a runtime relay protocol shared state, and any relay messages will be processed and forwarded by it
    pub fn new(chain: ChainController, shared: Arc<SyncShared>) -> Self {
        // setup a rate limiter keyed by peer and message type that lets through 30 requests per second
        // current max rps is 10 (ASK_FOR_TXS_TOKEN / TX_PROPOSAL_TOKEN), 30 is a flexible hard cap with buffer
        let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
        let rate_limiter = RateLimiter::hashmap(quota);

        Relayer {
            chain,
            shared,
            rate_limiter,
        }
    }

    /// Get shared state
    pub fn shared(&self) -> &Arc<SyncShared> {
        &self.shared
    }

    async fn try_process(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        message: packed::RelayMessageUnionReader<'_>,
    ) -> Status {
        // CompactBlock will be verified by POW, it's OK to skip rate limit checking.
        let should_check_rate =
            !matches!(message, packed::RelayMessageUnionReader::CompactBlock(_));

        if should_check_rate

                .rate_limiter
                .check_key(&(peer, message.item_id()))
                .is_err()
        {
            return StatusCode::TooManyRequests.with_context(message.item_name());
