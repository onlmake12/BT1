### Title
Unbounded DB Read Amplification via Unauthenticated `GetBlockFilters` Messages in Filter Protocol — (`sync/src/filter/get_block_filters_process.rs`)

---

### Summary

The Filter protocol handler (`sync/src/filter/mod.rs`) has no rate limiter, unlike the Relay protocol which enforces a governor-based 30 req/s per-peer limit. Any connected peer can send `GetBlockFilters(start_number=0)` at maximum rate, triggering up to 2,000 DB reads per message (1,000 × `get_block_hash` + 1,000 × `get_block_filter`), with no throttling, banning, or per-peer cost accounting.

---

### Finding Description

`GetBlockFiltersProcess::execute` loops up to `BATCH_SIZE = 1000` times per message, performing two DB reads per iteration: [1](#0-0) [2](#0-1) 

The 1.8MB size guard (added in #4972) only breaks the loop **after** both `get_block_hash` and `get_block_filter` have already been executed for that iteration — it limits response size, not DB read count: [3](#0-2) 

The `BlockFilter::received` handler dispatches directly to `execute` with zero rate limiting: [4](#0-3) 

A grep for `rate_limiter`, `governor`, `RateLimiter`, or `per_second` in `sync/src/filter/` returns **zero matches**, confirming no throttle exists anywhere in the Filter protocol stack.

By contrast, the Relay protocol explicitly constructs a governor-based limiter keyed by `(PeerIndex, message_type)` at 30 req/s: [5](#0-4) [6](#0-5) 

The same unbounded pattern also exists in `GetBlockFilterHashesProcess` (`BATCH_SIZE = 2000`, 2 DB reads/iteration) and `GetBlockFilterCheckPointsProcess` (`BATCH_SIZE = 2000`, 2 DB reads/iteration): [7](#0-6) [8](#0-7) 

---

### Impact Explanation

Each `GetBlockFilters(start_number=0)` message from a single peer causes up to 2,000 synchronous RocksDB reads. With N peers each sending at maximum rate, the node's I/O subsystem and async executor thread pool are saturated proportionally to N × message_rate × 2,000. This degrades block propagation, sync, and relay processing for all honest peers, constituting low-cost network congestion.

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no PoW, no stake, no privilege. The Filter protocol is a production protocol enabled on full nodes to serve light clients. The attacker controls `start_number` freely (it is a u64 field with no validation beyond `latest >= start_number`). The attack is locally reproducible with a minimal test harness.

---

### Recommendation

Apply a governor-based per-peer rate limiter to the `BlockFilter` handler, mirroring the existing pattern in `Relayer::new`:

```rust
// In BlockFilter struct
rate_limiter: RateLimiter<(PeerIndex, u32)>,

// In try_process, before dispatching:
if self.rate_limiter.check_key(&(peer, message.item_id())).is_err() {
    return StatusCode::TooManyRequests.with_context(message.item_name());
}
```

A quota of 1–2 req/s per peer per message type is sufficient for legitimate light-client use. Additionally, consider moving the 1.8MB size check to occur before the second DB read (`get_block_filter`) to avoid paying the read cost for entries that will be discarded.

---

### Proof of Concept

1. Connect N peers to a full node with the Filter protocol enabled.
2. Each peer sends `GetBlockFilters { start_number: 0 }` in a tight loop.
3. Each message triggers up to 1,000 calls to `active_chain.get_block_hash` and 1,000 calls to `active_chain.get_block_filter` — all DB reads, no PoW or auth required.
4. Monitor node CPU, RocksDB I/O, and block relay latency; observe saturation proportional to N × rate.
5. Compare against a patched node with a 2 req/s per-peer rate limiter — saturation disappears.
