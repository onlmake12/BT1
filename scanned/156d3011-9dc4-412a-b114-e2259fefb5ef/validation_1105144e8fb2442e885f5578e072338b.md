### Title
Full secp256k1 VM Execution Before Cycle Mismatch Detection Enables Asymmetric CPU DoS via Relay Protocol — (`tx-pool/src/process.rs`)

### Summary

In `_process_tx`, the full `ContextualTransactionVerifier` (CKB-VM secp256k1 script execution) runs unconditionally before the `declared_cycles != verified.cycles` check. An unprivileged relay peer can exploit this ordering to force a victim node to perform full script verification on every relay submission, at the cost of only a network connection and a valid UTXO.

### Finding Description

The execution order in `_process_tx` is:

1. `pre_check` (cheap — resolves inputs, checks fee)
2. `fetch_tx_verify_cache` — cache lookup by `witness_hash`
3. **`verify_rtx` — full CKB-VM execution** (secp256k1 script, ~1.7M cycles)
4. **Only then**: `if declared != verified.cycles` → `Reject::DeclaredWrongCycles` [1](#0-0) 

The cache update that would make subsequent attempts cheap is placed **after** the early return for `DeclaredWrongCycles`: [2](#0-1) 

So the cache is never populated on a `DeclaredWrongCycles` rejection. Every resubmission of the same witness triggers full VM execution.

After rejection, `after_process` calls `ban_malformed` (3-day ban) and also emits `TxVerificationResult::Reject`, which causes `remove_from_known_txs` to remove the tx_hash from the `tx_filter`: [3](#0-2) [4](#0-3) 

This means the same tx_hash can be re-announced after reconnection, since the filter no longer blocks it.

The only pre-VM guard in `TransactionsProcess::execute` is `declared_cycles > max_block_cycles` (immediate ban, no VM). A declared value of `actual_cycles ± 1` passes this guard and proceeds to full VM: [5](#0-4) 

### Impact Explanation

Each relay submission with a wrong-but-plausible `declared_cycles` forces the victim node to:
- Run full `ContextualTransactionVerifier` including secp256k1 CKB-VM execution
- Occupy the verify queue for the duration of that execution

The attacker spends O(1) CPU (send a P2P message); the victim spends O(secp256k1_cycles) CPU. This asymmetry can saturate the verify queue, delaying legitimate transaction admission across the network.

The existing integration test `DeclaredWrongCycles` confirms the behavior is reachable and that the tx is rejected only after full verification: [6](#0-5) 

### Likelihood Explanation

**Preconditions:**
- Attacker must own at least one valid UTXO (real CKB) to pass `pre_check`
- Attacker needs multiple IPs to bypass the 3-day ban per IP

**Attack loop:**
1. Connect as relay peer
2. Send `RelayTransactionHashes` → node sends `GetRelayTransactions`
3. Send `RelayTransactions` with `declared_cycles = actual ± 1`
4. Node runs full secp256k1 VM → detects mismatch → bans peer → removes tx_hash from filter
5. Reconnect with new IP → repeat (same UTXO, same tx, tx_hash was removed from filter)

The UTXO is never consumed (tx is rejected), so one UTXO suffices indefinitely. The cost is proportional to the number of IPs available, not to CKB holdings.

### Recommendation

Move the `declared_cycles` check **before** `verify_rtx`. If `declared_cycles` is provided, compare it against a lightweight upper-bound estimate (e.g., `declared > max_block_cycles` already exists; additionally, a pre-execution plausibility check or a short-circuit that runs VM with a cycle limit equal to `declared_cycles` and fails fast if exceeded) before committing to full script execution.

Alternatively, populate the `txs_verify_cache` even on `DeclaredWrongCycles` rejection so that subsequent resubmissions of the same witness use the cheap `TimeRelativeTransactionVerifier` path. [7](#0-6) 

### Proof of Concept

```
1. Attacker owns UTXO U, creates valid secp256k1-signed tx T (actual cycles = C)
2. Connect to victim node as relay peer
3. Send RelayTransactionHashes([T.hash])
4. Receive GetRelayTransactions([T.hash])
5. Send RelayTransactions([{cycles: C-1, transaction: T}])
   → victim runs full secp256k1 VM (C cycles of CPU)
   → victim detects C-1 != C → DeclaredWrongCycles → bans attacker IP
   → victim removes T.hash from tx_filter
6. Reconnect from new IP → goto step 3
   (UTXO U is still unspent; T.hash is no longer filtered)
```

CPU time consumed per rejection ≈ secp256k1 verification time (~tens of milliseconds). With N IPs, attacker can sustain N parallel verify-queue occupations, each costing the victim full VM execution while costing the attacker only a TCP connection.
