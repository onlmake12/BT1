### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Persistent Connection with Unique (from, to) PeerId Pairs — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

`HolePunching::forward_rate_limiter` uses `governor`'s `HashMapStateStore<(PeerId, PeerId, u32)>`. Its only cleanup call, `retain_recent()`, is placed exclusively in the `disconnected()` handler. The periodic `notify()` handler (every 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on `forward_rate_limiter`. An attacker who maintains a persistent connection and sends `ConnectionRequest` messages with unique `(from, to)` PeerId pairs can grow the HashMap without bound, exhausting heap memory.

---

### Finding Description

`forward_rate_limiter` is declared as:

```rust
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
``` [1](#0-0) 

It is backed by `governor::state::keyed::HashMapStateStore<(PeerId, PeerId, u32)>`, which accumulates one entry per unique key seen. The only place `retain_recent()` is called on it is:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    ...
}
``` [2](#0-1) 

The `notify()` handler, which fires every `CHECK_INTERVAL` (5 minutes), cleans up `pending_delivered` and `inflight_requests` but **never** calls `retain_recent()` on `forward_rate_limiter`: [3](#0-2) 

In `ConnectionRequestProcess::execute`, every message that passes the per-session `rate_limiter` check calls:

```rust
self.protocol
    .forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
``` [4](#0-3) 

`check_key` on a `HashMapStateStore` inserts a new entry for every previously-unseen key. The `from` and `to` fields are arbitrary attacker-controlled bytes parsed as `PeerId`. The per-session `rate_limiter` is keyed by `(session_id, msg_item_id)` — `msg_item_id` is a fixed constant per message type — so it allows 30 messages/sec from a single session regardless of how many unique `(from, to)` pairs are used. [5](#0-4) 

---

### Impact Explanation

At 30 messages/sec sustained, the attacker inserts 30 new `(PeerId, PeerId, u32)` entries per second into the HashMap. Each `PeerId` is 39 bytes; each entry consumes roughly ~100–120 bytes including HashMap overhead. Over 24 hours that is ~2.6 million entries and ~300 MB of heap growth from a single connection. Multiple concurrent attackers scale this linearly. The node eventually OOM-crashes, taking down block relay, transaction propagation, and consensus participation.

---

### Likelihood Explanation

The attack requires only a single persistent TCP connection to a victim node — no PoW, no stake, no privileged role. The attacker generates random PeerIds locally (trivial). The per-session rate limiter is the only throttle, and it does not bound total HashMap size. The attack is fully automatable and replicable across the network.

---

### Recommendation

Call `self.forward_rate_limiter.retain_recent()` (and `self.rate_limiter.retain_recent()`) inside the `notify()` handler, which already fires every 5 minutes, so stale entries are evicted on a time-based schedule independent of peer disconnection:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();          // add
    self.forward_rate_limiter.retain_recent();  // add
    let now = unix_time_as_millis();
    self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
    ...
}
``` [3](#0-2) 

---

### Proof of Concept

```rust
// Pseudocode: single persistent session, never disconnects
let session = connect_to_victim_node();
loop {
    // 30 unique (from, to) pairs per second — stays within per-session rate limit
    for _ in 0..30 {
        let from = PeerId::random();
        let to   = PeerId::random();
        session.send(ConnectionRequest { from, to, max_hops: 1, listen_addrs: [valid_addr], route: [] });
    }
    sleep(Duration::from_secs(1));
    // forward_rate_limiter HashMap grows by 30 entries/sec
    // retain_recent() is never called because session stays open
    // After 24h: ~2.6M entries, ~300MB heap consumed
}
```
