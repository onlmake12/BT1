### Title
Integer overflow in `GetLastStateProofProcess::execute` bypasses `GET_LAST_STATE_PROOF_LIMIT`, enabling unbounded server-side work per light-client request — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged light-client peer can send a `GetLastStateProof` message with `last_n_blocks = 2^63` and an empty `difficulties` list. The guard at line 201 computes `(2^63_usize) * 2`, which wraps to `0` in Rust release mode, making the check trivially false. The server then collects and processes every block from `start_block_number` to the chain tip, doing multiple expensive DB and MMR lookups per block, with no effective upper bound.

### Finding Description
The check intended to cap server work is:

```rust
// line 201
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
```

With `last_n_blocks = 2^63` (a valid `Uint64` molecule value):
- `last_n_blocks as usize = 2^63` (no-op on 64-bit)
- `2^63 * 2` wraps to `0` (Rust release-mode wrapping arithmetic)
- `0 + 0 > 1000` → `false` → guard skipped

Execution then reaches line 291. Because `last_block_number - start_block_number ≤ 2^63` is always true for any real chain, the server takes the "not enough blocks" branch and executes:

```rust
let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
```

This Vec has `last_block_number - start_block_number` entries — the full chain range — and `complete_headers` is called for every entry.

### Impact Explanation
Each iteration of `complete_headers` performs: `get_ancestor` (O(log H) header traversal), `get_block` (full block fetch), `calc_uncles_hash`, and `chain_root_mmr(...).get_root()`. For a chain of height H, a single malicious request causes O(H) expensive operations. On a long-running mainnet node (height in the millions), this can exhaust CPU for the duration of the request and allocate gigabytes of `VerifiableHeader` data in memory, causing OOM or a sustained denial-of-service against the light-client server thread.

### Likelihood Explanation
Any peer that speaks the light-client protocol can send this message. No PoW, no key, no privilege is required. The molecule encoding of `Uint64` accepts all 64-bit values, so crafting the message is trivial. The attack is repeatable and can be sent in a tight loop.

### Recommendation
Replace the unchecked multiplication with a saturating or checked variant before the comparison:

```rust
// Safe: saturating_mul prevents wrap-around
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, add an explicit early rejection if `last_n_blocks > GET_LAST_STATE_PROOF_LIMIT as u64` before any arithmetic.

### Proof of Concept
1. Connect to a CKB node as a light-client peer.
2. Obtain the current tip hash.
3. Send `GetLastStateProof { last_hash: tip_hash, start_hash: genesis_hash, start_number: 0, last_n_blocks: 9223372036854775808 /* 2^63 */, difficulty_boundary: U256::MAX, difficulties: [] }`.
4. Observe: the guard at line 201 passes (evaluates to `0 > 1000 = false`), and the server begins iterating over all blocks from 0 to tip, performing full DB lookups for each, until OOM or timeout.
5. A unit test can confirm: build a chain of height H > 1000, send the above message, assert that the server either rejects it or processes no more than 1000 block numbers — currently it will process all H.
