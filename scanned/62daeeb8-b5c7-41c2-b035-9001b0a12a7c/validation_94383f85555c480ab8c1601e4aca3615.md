### Title
Unbounded Full-Pool Iteration in `get_raw_tx_pool(verbose=true)` Enables RPC-Triggered Resource Exhaustion — (`rpc/src/module/pool.rs`)

---

### Summary

The `get_raw_tx_pool` RPC with `verbose=true` iterates over every entry in the tx-pool without pagination, result-count limits, or per-call rate limiting. An attacker who fills the pool with many minimum-fee transactions can hammer this endpoint to cause sustained CPU/memory pressure and prolonged tx-pool read-lock contention, degrading or stalling normal node operation.

---

### Finding Description

`get_all_entry_info()` in `tx-pool/src/pool.rs` performs three unbounded sequential scans:

1. All `Pending`+`Gap` entries, sorted by score
2. All `Proposed` entries, sorted by score
3. All entries in `conflicts_cache` (LRU, up to 10,000) [1](#0-0) 

This function is called directly from the `get_raw_tx_pool` RPC handler with no guard: [2](#0-1) 

The tx-pool is bounded by byte size (`max_tx_pool_size = 180 MB`), not by entry count: [3](#0-2) 

A minimal CKB transaction is ~200–300 bytes, so the pool can hold on the order of **600,000–900,000 entries** at the default limit. Each `get_raw_tx_pool(verbose=true)` call must iterate and JSON-serialize every one of them. The RPC server has only a batch-request count limit; there is no per-method rate limit and no pagination parameter for this endpoint: [4](#0-3) 

The tx-pool read lock is held for the entire duration of the scan. While held, transaction submission and block-processing callbacks that need the write lock are blocked.

---

### Impact Explanation

- **CPU exhaustion**: Serializing hundreds of thousands of `TxEntryInfo` structs to JSON on every call is CPU-intensive. A tight loop of calls from a single client saturates a CPU core.
- **Memory spike**: The full verbose response is materialized in memory before being sent. At pool capacity the response can exceed 100 MB per call; concurrent calls multiply this.
- **Lock contention**: The tx-pool read lock is held across the entire iteration. Concurrent write-lock waiters (e.g., `submit_transaction`, block-commit callbacks in `_update_tx_pool_for_reorg`) stall until the scan completes. [5](#0-4) 

Combined effect: a node with a full pool can be made to spend the majority of its time serving `get_raw_tx_pool` responses, starving block relay and transaction acceptance.

---

### Likelihood Explanation

- The RPC is bound to `127.0.0.1:8114` by default, making the attacker a local or co-located process. The scope explicitly includes "RPC caller" and "supported local CLI/RPC user" as valid unprivileged attackers. Many production deployments also expose the RPC publicly for dApp connectivity.
- Filling the pool requires paying minimum fees (1,000 shannons/KB), which is a real but low cost barrier — at current CKB prices, filling 180 MB costs a modest amount of CKB.
- No special privilege, key, or majority hashpower is required.
- The attack is repeatable and self-sustaining: once the pool is full, the attacker only needs to keep calling the RPC.

---

### Recommendation

1. **Add pagination** to `get_raw_tx_pool`: accept `offset` and `limit` parameters and slice the iterator accordingly.
2. **Add a per-method rate limit** (e.g., 1–2 calls/second) for `get_raw_tx_pool` in the RPC server middleware, analogous to the relay-protocol rate limiter already used for P2P messages. [6](#0-5) 

3. **Cap the verbose response** at a configurable maximum entry count and document the truncation.

---

### Proof of Concept

```python
import requests, json, threading

NODE = "http://127.0.0.1:8114"

# Step 1: flood the pool with many minimum-fee transactions
# (omitted for brevity — use send_transaction RPC in a loop)

# Step 2: hammer get_raw_tx_pool(verbose=true) concurrently
def spam():
    while True:
        requests.post(NODE, json={
            "id": 1, "jsonrpc": "2.0",
            "method": "get_raw_tx_pool",
            "params": [True]
        })

for _ in range(8):
    threading.Thread(target=spam, daemon=True).start()

# Observe: node CPU pegged, transaction submission latency spikes,
# block relay stalls as the tx-pool write lock is starved.
```

The root cause is the unbounded `score_sorted_iter_by_statuses` + `sorted_proposed_iter` loops inside `get_all_entry_info`, which have no upper bound on the number of entries they traverse per call. [7](#0-6)

    pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let conflicted = self
            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {
            pending,
            proposed,
            conflicted,
        }
    }

    fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
        let tx_pool = self.shared.tx_pool_controller();

        let raw = if verbose.unwrap_or(false) {
            let info = tx_pool
                .get_all_entry_info()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Verbose(info.into())
        } else {
            let ids = tx_pool
                .get_all_ids()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Ids(ids.into())
        };
        Ok(raw)
    }
