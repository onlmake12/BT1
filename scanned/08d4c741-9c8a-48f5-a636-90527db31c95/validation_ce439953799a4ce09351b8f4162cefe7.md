Audit Report

## Title
`ScriptHashTypeVerifier` Fails to Validate Type Script `hash_type` in Transaction Outputs — (`File: verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates `hash_type` only for the **lock script** of each output. The **type script** `hash_type` is never read or validated. Because `ENABLED_SCRIPT_HASH_TYPE` currently excludes value `3` (and any future additions), an unprivileged sender can embed a disallowed `hash_type` byte in an output's type script, pass `NonContextualTransactionVerifier`, and commit a permanently unspendable cell to the chain — locking capacity forever.

## Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as `{0, 1, 2, 4}` — value `3` is explicitly absent. [1](#0-0) 

`ScriptHashTypeVerifier::verify()` loops over outputs and checks only `output.lock().hash_type()`: [2](#0-1) 

`output.type_()` is never consulted. A CKB cell output carries two independent scripts; the type script has its own `hash_type` byte that is completely ignored here.

The verifier is registered as the final step of `NonContextualTransactionVerifier::verify()`: [3](#0-2) 

**Exploit path:**
1. Craft an output with `lock.hash_type = 0` (Data — passes the check) and `type.hash_type = 3` (excluded from `ENABLED_SCRIPT_HASH_TYPE` — never checked).
2. Submit via RPC or P2P relay. `ScriptHashTypeVerifier` reads only the lock script, finds `0 ∈ ENABLED_SCRIPT_HASH_TYPE`, returns `Ok(())`.
3. The transaction is accepted into the tx-pool and committed.
4. The resulting live cell carries `hash_type = 3` in its type script on-chain.
5. Any spending transaction triggers type-script execution; `ScriptHashType::try_from(3)` fails (no variant defined), causing the spending transaction to be rejected by every node uniformly — the cell is permanently unspendable.

No existing guard catches this: `NonContextualTransactionVerifier` is the only place hash-type gating is applied, and it is incomplete.

## Impact Explanation

The concrete, immediately triggerable impact is **permanent capacity lock**: an attacker can create cells with disallowed type-script `hash_type` values and transfer them to any address. In CKB's UTXO model recipients cannot refuse incoming cells. The locked capacity is irrecoverable, constituting direct economic damage to cell recipients and reducing the effective circulating supply of CKB. At scale this maps to **"Vulnerabilities which could easily damage CKB economy"** (Critical tier). A secondary impact — consensus deviation — becomes concrete the moment a new `ScriptHashType` variant is added to the enum but intentionally withheld from `ENABLED_SCRIPT_HASH_TYPE` pending a hardfork, since nodes at different upgrade states would resolve the hash type differently.

## Likelihood Explanation

The attack requires only the ability to submit a standard transaction — available to every RPC caller or P2P relayer. No special key, role, or victim mistake is needed. The attacker funds the cell capacity themselves (a one-time cost), then transfers the unspendable cell to a target. The bypass is deterministic and repeatable. Likelihood is **high** for the capacity-lock scenario at any time, and **medium-high** for the consensus-deviation scenario during any active hardfork window.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to validate the type script `hash_type` of every output in addition to the lock script `hash_type`. After validating the lock script (existing logic), add:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(ht) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = ht.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err(TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }.into());
    }
}
```

This closes the bypass symmetrically for both script fields.

## Proof of Concept

1. Confirm `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}` — value `3` is absent.
2. Build a `TransactionView` with one output:
   - `lock`: `{ code_hash: <any 32 bytes>, hash_type: 0 }` — passes the verifier.
   - `type`: `{ code_hash: <any 32 bytes>, hash_type: 3 }` — **never read by the verifier**.
3. Run `NonContextualTransactionVerifier::new(&tx, &consensus).verify()` — it returns `Ok(())`.
4. Submit the transaction; it is accepted and committed.
5. Attempt to spend the resulting cell in a follow-up transaction — script execution fails because `ScriptHashType::try_from(3)` returns `Err`, making the cell permanently unspendable.
6. Repeat targeting arbitrary recipient addresses to lock their capacity at will.
