### Title
Peer-Supplied `declared_cycles` Used as VM Cycle Cap Without Lower-Bound Validation Causes Tx Propagation DoS via Known-Filter Poisoning — (`File: sync/src/relayer/transactions_process.rs`, `tx-pool/src/process.rs`)

---

### Summary

The CKB relay protocol accepts a peer-supplied `declared_cycles` field alongside each relayed transaction. This value is used directly as the VM cycle cap (`max_cycles`) during script verification. When an attacker relays a valid transaction with an artificially low `declared_cycles` value, the VM aborts with a cycle-limit-exceeded error. This error is misclassified as a malformed transaction (`Reject::Verification` with `is_malformed_tx() = true`), which causes the tx hash to be permanently retained in the relay `tx_filter` (the known-tx bloom filter) without being cleared. All subsequent legitimate announcements of the same tx hash from honest peers are silently dropped by the filter, blocking the transaction from propagating until the filter TTL expires.

---

### Finding Description

**Entry path — relay protocol:**

In `sync/src/relayer/transactions_process.rs`, `TransactionsProcess::execute()` accepts a `RelayTransactions` P2P message. Each entry carries a `(TransactionView, Cycle)` pair where the `Cycle` is the peer-declared cycle count. The only validation performed at this layer is an upper-bound check:

```rust
if txs.iter().any(|(_, declared_cycles)| declared_cycles > &max_block_cycles) {
    self.nc.ban_peer(...);
    return Status::ok();
}
```

No lower-bound check exists. A value of `1` is accepted. [1](#0-0) 

The tx hashes are immediately marked as "known" in the relay filter before the tx is submitted to the pool:

```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
``` [2](#0-1) 

**Root cause — `declared_cycles` used as VM cap:**

In `tx-pool/src/process.rs`, `_process_tx()` uses `declared_cycles` directly as `max_cycles` for the VM:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
let verified_ret = verify_rtx(Arc::clone(&snapshot), Arc::clone(&rtx), tx_env,
                               &verify_cache, max_cycles, command_rx).await;
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
``` [3](#0-2) 

If `declared_cycles` is too low, `verify_rtx` returns a cycle-limit-exceeded error **before** reaching the `DeclaredWrongCycles` check:

```rust
if let Some(declared) = declared_cycles && declared != verified.cycles {
    return Some((Err(Reject::DeclaredWrongCycles(declared, verified.cycles)), snapshot));
}
``` [4](#0-3) 

The cycle-limit-exceeded error is of kind `Script`, so `is_malformed_from_verification` returns `true`:

```rust
ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
``` [5](#0-4) 

This makes `is_malformed_tx()` return `true` for the resulting `Reject::Verification`: [6](#0-5) 

**Consequence — tx hash locked in known filter:**

In `after_process`, when `is_malformed_tx()` is true, the peer is banned. Crucially, `is_allowed_relay()` returns `false` for malformed txs (except `DeclaredWrongCycles`):

```rust
pub fn is_allowed_relay(&self) -> bool {
    matches!(self, Reject::DeclaredWrongCycles(..))
        || (!matches!(self, Reject::LowFeeRate(..)) && !self.is_malformed_tx())
}
``` [7](#0-6) 

Because `is_allowed_relay()` is false, `TxVerificationResult::Reject` is never sent to the relayer, so `remove_from_known_txs` is never called: [8](#0-7) 

The tx hash therefore remains in `tx_filter` as "known." In `TransactionHashesProcess::execute()`, any subsequent announcement of the same tx hash from an honest peer is silently discarded:

```rust
.filter(|tx_hash| !tx_filter.contains(tx_hash))
``` [9](#0-8) 

---

### Impact Explanation

An attacker who connects to a target node and announces a tx hash before the legitimate originator can poison the relay filter for that tx hash. The target node will request the tx from the attacker, receive it with `declared_cycles = 1`, fail verification, ban the attacker, and retain the tx hash in the known filter. All subsequent honest announcements of the same hash are silently dropped. The transaction cannot propagate through the target node until the `TtlFilter` TTL expires. This delays or prevents the transaction from being proposed and committed, constituting a targeted transaction-relay denial of service.

---

### Likelihood Explanation

The attacker must be peered with the target node and must announce the tx hash before the legitimate peer. Both conditions are achievable for a well-connected attacker who monitors the P2P gossip layer. The attacker is banned after a single attempt (3-day ban), but the damage — the tx hash locked in the filter — persists for the full TTL duration regardless. The attack requires no special privilege, no key material, and no majority hashpower. [10](#0-9) 

---

### Recommendation

1. **Add a lower-bound validation on `declared_cycles`** in `TransactionsProcess::execute()`. Reject (and ban) any peer that declares `declared_cycles = 0` or below a configurable minimum (e.g., `TWO_IN_TWO_OUT_CYCLES`).

2. **Distinguish cycle-limit-exceeded from structurally malformed.** A cycle-limit-exceeded error means the declared value was wrong, not that the transaction is invalid. It should be treated as `DeclaredWrongCycles` (or a new `DeclaredInsufficientCycles` variant) so that `is_allowed_relay()` returns `true` and `remove_from_known_txs` is called, allowing honest peers to re-announce the tx.

3. **Do not mark the tx hash as known before pool admission succeeds.** Move `mark_as_known_txs` to after successful pool insertion, or ensure that any rejection path that is not the peer's fault clears the filter entry.

---

### Proof of Concept

1. Attacker connects to target node via RelayV3.
2. Attacker sends `RelayTransactionHashes` containing hash `H` of a valid pending transaction.
3. Target node adds `H` to `unknown_tx_hashes` and sends `GetRelayTransactions` back to attacker.
4. Attacker responds with `RelayTransactions` containing the valid transaction body but with `cycles = 1`.
5. `TransactionsProcess::execute()` passes the upper-bound check (`1 ≤ max_block_cycles`), marks `H` as known in `tx_filter`, and calls `submit_remote_tx(tx, 1, peer)`.
6. `_process_tx` sets `max_cycles = 1`, VM aborts immediately with cycle-limit-exceeded → `Reject::Verification(Script error)`.
7. `after_process`: `is_malformed_tx() = true` → `ban_malformed(peer)` called; `is_allowed_relay() = false` → `remove_from_known_txs` NOT called; `H` remains in `tx_filter`.
8. Legitimate peer later sends `RelayTransactionHashes` with `H`; `TransactionHashesProcess::execute()` filters it out (`tx_filter.contains(H) = true`); node never requests the tx from the honest peer.
9. Transaction is blocked from propagating through this node for the duration of the `TtlFilter` TTL.
