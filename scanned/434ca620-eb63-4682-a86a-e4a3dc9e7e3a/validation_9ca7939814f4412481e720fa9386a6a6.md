### Title
Fee Estimator State Inconsistency After Chain Reorganization — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary
The `ConfirmationFraction` fee estimator's `process_block` function silently skips all block processing when the incoming block height is not strictly greater than `best_height`. During a chain reorganization where the new canonical chain is shorter than the old one, `best_height` is never rolled back, causing the estimator to stop tracking transactions and stop recording confirmation statistics for an extended window of blocks. Transactions re-added to the pool from detached blocks are also silently dropped from tracking because `current_tip` diverges from `best_height`. The result is that `estimate_fee_rate` RPC responses are based on stale, incorrect data after any reorg.

### Finding Description

In `update_tx_pool_for_reorg` (`tx-pool/src/process.rs`), `fee_estimator.commit_block` is called only for `attached_blocks` — the new chain's blocks — and never for `detached_blocks`: [1](#0-0) 

Inside `commit_block`, `current_tip` is updated unconditionally, then `process_block` is called: [2](#0-1) 

`process_block` contains an explicit early-return guard: [3](#0-2) 

When a reorg replaces a chain of height `H_old` with a shorter chain of height `H_new < H_old`:

1. `best_height` remains at `H_old` (it is never decremented).
2. `current_tip` is updated to `H_new` via `commit_block`.
3. All subsequent `commit_block` calls for blocks at heights `H_new+1 … H_old` are silently skipped — no confirmation statistics are recorded, no decay is applied, no tracking window is advanced.

Additionally, `accept_tx` uses `current_tip` as the height for `track_tx`: [4](#0-3) 

`track_tx` rejects any transaction whose height does not exactly equal `best_height`: [5](#0-4) 

Because `current_tip` (`H_new`) ≠ `best_height` (`H_old`) for all blocks in the catch-up window, every transaction entering the pool during that window — including transactions re-added from detached blocks via `readd_detached_tx` — is silently discarded from tracking. The fee estimator accumulates no new samples until the chain surpasses `H_old`, which can take many blocks.

The `register_tx_pool_callback` wiring confirms that `fee_estimator.accept_tx` is the sole mechanism for tracking new pool entries: [6](#0-5) 

### Impact Explanation

After a reorg to a shorter chain, the `estimate_fee_rate` RPC (`tx-pool/src/process.rs` `estimate_fee_rate`) returns fee rate estimates derived from a frozen, stale statistical model. The estimator:

- Does not record confirmation times for transactions committed in the new chain's catch-up blocks.
- Does not track unconfirmed transactions entering the pool during the catch-up window.
- Does not apply the decay factor to old samples during the skipped blocks.

This produces systematically incorrect fee rate recommendations — either too high (stale high-fee samples dominate) or too low (unconfirmed backlog is invisible) — for the entire duration of the catch-up period, which scales with the depth of the reorg. [7](#0-6) 

### Likelihood Explanation

Any sync peer can trigger a chain reorganization by relaying a valid alternative chain with greater total difficulty. This is a normal, unprivileged network operation. Reorgs of 1–5 blocks are common on live networks. A deeper reorg (e.g., 10–20 blocks) would leave the fee estimator blind for a proportionally longer window. No special access, key material, or majority hashpower is required to trigger the state inconsistency — only the ability to relay a valid competing chain, which any peer can do.

### Recommendation

Reset `best_height` to the common ancestor's height at the start of `update_tx_pool_for_reorg` when detached blocks are present. Alternatively, clear the estimator state entirely on reorg (similar to how `update_ibd_state` calls `clear()`) and restart sampling from the new tip. The `process_block` guard comment — *"For simpfy, we assume chain reorg will not effect tx fee"* — documents the known simplification but does not prevent the divergence of `current_tip` and `best_height` from silently corrupting the tracking state. [8](#0-7) 

### Proof of Concept

1. Node is at chain tip height 100; `best_height = 100`, `current_tip = 100`.
2. A sync peer relays a valid alternative chain whose tip is at height 95 with greater total difficulty. A reorg occurs: blocks 96–100 are detached; blocks at heights 91–95 (new chain) are attached.
3. `update_tx_pool_for_reorg` calls `fee_estimator.commit_block` for each attached block (heights 91–95). Each call sets `current_tip` to 91, 92, …, 95 in sequence, but `process_block` returns immediately because `height (91–95) <= best_height (100)`.
4. After the reorg: `current_tip = 95`, `best_height = 100`.
5. Transactions from detached blocks are re-added to the pool via `readd_detached_tx` → `_submit_entry` → `callbacks.call_pending` → `fee_estimator.accept_tx(tx_hash, info)`. Inside `accept_tx`, `track_tx` is called with `height = current_tip = 95`. Since `95 ≠ best_height (100)`, every re-added transaction is silently dropped.
6. New blocks arrive at heights 96, 97, 98, 99, 100. Each `commit_block` call updates `current_tip` but `process_block` skips all of them. Transactions entering the pool during this window are also not tracked (`current_tip ≠ best_height`).
7. Only at block 101 does `process_block` resume. The fee estimator has produced incorrect estimates for 6 blocks, with no confirmation samples recorded and no unconfirmed transactions tracked.
8. An RPC caller invoking `estimate_fee_rate` during steps 4–7 receives a fee rate derived entirely from pre-reorg data, which may be significantly wrong.

    fn process_block(&mut self, height: u64, txs: impl Iterator<Item = Byte32>) {
        // For simpfy, we assume chain reorg will not effect tx fee.
        if height <= self.best_height {
            return;
        }
        self.best_height = height;
        // update tx confirm stat
        self.tx_confirm_stat.move_track_window(height);
        self.tx_confirm_stat.decay();
        let processed_txs = txs.filter(|tx| self.process_block_tx(height, tx)).count();
        if self.start_height == 0 && processed_txs > 0 {
            // start record
            self.start_height = self.best_height;
            ckb_logger::debug!("start recording at {}", self.start_height);
        }
    }

    fn track_tx(&mut self, tx_hash: Byte32, fee_rate: FeeRate, height: u64) {
        if self.tracked_txs.contains_key(&tx_hash) {
            // already in track
            return;
        }
        if height != self.best_height {
            // ignore wrong height txs
            return;
        }
        if let Some(bucket_index) = self.tx_confirm_stat.add_unconfirmed_tx(height, fee_rate) {
            self.tracked_txs.insert(
                tx_hash,
                TxRecord {
                    height,
                    bucket_index,
                    fee_rate,
                },
            );
        }
    }
