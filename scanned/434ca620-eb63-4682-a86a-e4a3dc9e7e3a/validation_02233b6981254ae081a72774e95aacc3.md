### Title
Unbounded Per-Peer DB Read Amplification via `GetBlockFilterHashes` — (`sync/src/filter/get_block_filter_hashes_process.rs`)

### Summary

The `GetBlockFilterHashesProcess::execute` handler performs up to 4,000 RocksDB reads per message (2 reads × `BATCH_SIZE=2000`) with no per-peer rate limiting. Any unprivileged P2P peer can flood the node with `GetBlockFilterHashes(start_number=0)` messages, forcing unbounded DB read amplification and degrading sync performance for all peers.

### Finding Description

`GetBlockFilterHashesProcess::execute` iterates up to `BATCH_SIZE = 2000` times, calling `get_block_hash(block_number)` and `get_block_filter_hash(&block_hash)` on each iteration — two RocksDB reads per block, totalling up to 4,000 reads per message. [1](#0-0) [2](#0-1) 

The `start_number` is taken directly from the peer-supplied message with no validation beyond `latest >= start_number`. With `start_number=0` on any chain with 2000+ blocks, the full 4,000-read path is always triggered. [3](#0-2) 

The `BlockFilter::received` handler dispatches directly to `process()` with no rate-limiting gate, no per-peer message counter, and no cooldown: [4](#0-3) 

The only ban path in the Filter handler is for malformed messages (400-series `StatusCode`). A well-formed `GetBlockFilterHashes` message returns `Status::ok()` or `Status::ignored()` — neither triggers a ban. [5](#0-4) 

By contrast, the Relay protocol (`sync/src/relayer/mod.rs`) has explicit rate-limiting logic (9 occurrences of `rate_limit`). The Filter protocol has none. The `TooManyRequests = 110` status code exists in the codebase but is never used in the Filter handler. [6](#0-5) 

### Impact Explanation

A single unprivileged peer can send a continuous stream of `GetBlockFilterHashes(start_number=0)` messages. Each message forces the node to execute up to 4,000 synchronous RocksDB reads before returning. This saturates RocksDB read bandwidth, stalls the async executor (since the handler is `async` but the DB reads are blocking), and degrades block sync and relay performance for all other peers — a node-wide DoS from a single connection.

### Likelihood Explanation

The attack requires only a valid P2P connection and a single well-formed message type. No PoW, no keys, no privileged role. The Filter protocol is enabled by default. The attack is trivially scriptable and locally testable.

### Recommendation

1. Add a per-peer rate limiter (token bucket or leaky bucket) in `BlockFilter::received` or `try_process`, using the existing `TooManyRequests` status code to reject excess requests.
2. Consider reducing `BATCH_SIZE` or adding a configurable cap.
3. Ensure DB reads in the loop are non-blocking (offloaded to a blocking thread pool) to avoid stalling the async executor.

### Proof of Concept

1. Spin up a CKB node with 2000+ blocks and the Filter protocol enabled.
2. Connect a custom P2P peer to `/ckb/filter`.
3. Send 100 `GetBlockFilterHashes(start_number=0)` messages in rapid succession from a single peer.
4. Measure node RocksDB read IOPS and sync throughput for other peers — both will degrade proportionally to the flood rate.
