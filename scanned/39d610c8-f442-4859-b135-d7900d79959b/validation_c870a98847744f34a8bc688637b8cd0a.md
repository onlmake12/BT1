The code is fully readable. Let me trace the exploit path precisely before rendering the verdict.

**Trace:**

1. `received()` in `mod.rs` dispatches a `ConnectionRequest` to `ConnectionRequestProcess::execute()`. The `peer` field is the session index of the actual connected peer (attacker C), but `content.from` is taken verbatim from the message bytes — no check that it matches the session's actual peer identity. [1](#0-0) 

2. `execute()` routes to `respond_delivered` when `self_peer_id == &content.to`. The only guard is a time-window check: if `pending_delivered[from_peer_id]` was written less than `HOLE_PUNCHING_INTERVAL` (2 minutes) ago, it returns `Ignore`. After 2 minutes, execution falls through unconditionally. [2](#0-1) 

3. `respond_delivered` then inserts `(attacker_addresses, now)` into `pending_delivered` keyed by the spoofed `from_peer_id` (A's PeerId). [3](#0-2) 

4. `ConnectionSyncProcess::execute()` later reads `pending_delivered.get(&content.from)` and passes those addresses directly to `try_nat_traversal`, which opens real TCP connections. [4](#0-3) 

5. The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` — i.e., the spoofed values — so it does not bind the rate limit to the actual session identity and resets after its window. [5](#0-4) 

6. The address validation only checks that any embedded peer ID in the multiaddr matches `content.from` (the spoofed A). The attacker can supply plain `ip4/tcp` addresses (no embedded peer ID), which then get A's PeerId appended — but the IP:port remains fully attacker-controlled. [6](#0-5) 

The invariant is broken: `pending_delivered` is supposed to record the addresses of the authentic originating peer, but any connected peer can overwrite any entry after the 2-minute window by spoofing `from`.

### Title
Unauthenticated `from` field in `ConnectionRequest` allows `pending_delivered` overwrite via peer ID spoofing — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary
`ConnectionRequestProcess::respond_delivered` writes attacker-supplied listen addresses into `pending_delivered` keyed by an arbitrary `from` peer ID taken from the message payload, with no verification that the sender's actual session identity matches `from`. Any connected peer can overwrite the `pending_delivered` entry for any other peer ID after the 2-minute `HOLE_PUNCHING_INTERVAL` expires.

### Finding Description
`HolePunching::pending_delivered` is a `HashMap<PeerId, (Vec<Multiaddr>, u64)>` that maps a requesting peer's ID to its listen addresses and a timestamp. It is populated in `respond_delivered` and consumed in `ConnectionSyncProcess::execute` to drive NAT traversal.

The `from` field of a `ConnectionRequest` message is a raw byte field in the molecule schema with no cryptographic binding to the sending session:

```
table ConnectionRequest {
    from: Bytes,   // claimed peer ID — not authenticated
    to:   Bytes,
    ...
    listen_addrs: AddressVec,
}
```

`respond_delivered` extracts `from_peer_id` directly from `content.from` and uses it as the map key:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

The only guard is a time-window check (`now - t < HOLE_PUNCHING_INTERVAL`). Once 2 minutes have elapsed since the last write for a given key, any peer can overwrite it. There is no check of the form "does `content.from` equal the peer ID associated with `self.peer`?"

### Impact Explanation
After overwriting `pending_delivered[A]` with attacker-controlled addresses, the next `ConnectionSync` message with `from=A` causes node B to:
1. Retrieve the attacker's addresses from `pending_delivered`.
2. Spawn async tasks calling `try_nat_traversal` to each address.
3. On success, call `control.raw_session(stream, addr, ...)` — establishing a raw P2P session to the attacker's endpoint.

Concrete impacts:
- **NAT traversal hijack**: B establishes a P2P session to an attacker-controlled node instead of A, allowing the attacker to impersonate A at the transport layer.
- **Disrupted peer connectivity**: The legitimate A–B hole-punch fails; B's `pending_delivered[A]` is poisoned until the 5-minute `TIMEOUT` expires.
- **Arbitrary outbound TCP**: The attacker can cause B to make TCP connections to any IP:port (SSRF-equivalent), potentially probing internal network services.
- **CKB economy damage**: Repeated poisoning of `pending_delivered` entries degrades NAT traversal success rates across the network, reducing peer connectivity for nodes behind NAT.

### Likelihood Explanation
- Requires only a standard P2P connection to the target node — no special privileges.
- The attacker needs to know A's PeerId (public information, observable from the network).
- The 2-minute wait is trivially satisfied; the attacker can maintain a persistent connection and send the spoofed message once per interval.
- The `forward_rate_limiter` is keyed on the spoofed `(from, to, item_id)` tuple, not the session, so it does not prevent repeated attacks from the same session using the same spoofed identity after the rate window resets.
- No PoW, no cryptographic material, no privileged role required.

### Recommendation
Verify that `content.from` matches the actual peer ID of the sending session before writing to `pending_delivered`. In `respond_delivered`, look up the peer ID associated with `self.peer` from the peer registry and reject the message if it does not equal `content.from`:

```rust
// Pseudocode
let actual_peer_id = self.protocol.network_state
    .peer_registry.read()
    .get_peer(self.peer)
    .map(|p| p.peer_id.clone());

if actual_peer_id.as_ref() != Some(&from_peer_id) {
    return StatusCode::InvalidFromPeerId
        .with_context("from peer id does not match session identity");
}
```

This ensures only the authentic originating peer can write its own entry into `pending_delivered`.

### Proof of Concept
1. Start three nodes: A (legitimate initiator), B (target, the victim), C (attacker). All are connected to B.
2. A sends a legitimate `ConnectionRequest` with `from=A_peer_id`, `to=B_peer_id`, `listen_addrs=[A_real_addr]`. B stores `pending_delivered[A_peer_id] = ([A_real_addr], t0)`.
3. Wait 2 minutes (or mock `unix_time_as_millis` in a test to advance time past `HOLE_PUNCHING_INTERVAL`).
4. C sends a `ConnectionRequest` with `from=A_peer_id`, `to=B_peer_id`, `listen_addrs=[C_attacker_addr]` over C's session to B.
5. B's `respond_delivered` passes the time check (elapsed ≥ 2 min), sends a `ConnectionRequestDelivered` back to C's session, and writes `pending_delivered[A_peer_id] = ([C_attacker_addr], t1)`.
6. Send a `ConnectionSync` with `from=A_peer_id`, `to=B_peer_id` to B (either from C spoofed, or wait for A's legitimate sync).
7. Observe B attempting TCP connections to `C_attacker_addr` instead of `A_real_addr`, confirming the hijack.
