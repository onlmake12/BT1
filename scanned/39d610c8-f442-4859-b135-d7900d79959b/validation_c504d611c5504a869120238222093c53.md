### Title
Fee Rate Check Uses Raw Size Instead of Weight, Allowing High-Cycle Transactions to Underpay Fees - (File: tx-pool/src/util.rs)

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` computes the minimum required fee using only the raw serialized transaction size (`tx_size`), not the true transaction weight (`get_transaction_weight(tx_size, cycles)`). Because CKB's block space is constrained by both bytes and cycles, the correct fee-rate denominator is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction sender can craft a computationally expensive transaction (high cycles, small byte size) that passes the fee gate while paying far less than the fee rate it actually demands from block space.

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The comment directly above this line acknowledges the problem:

```
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
``` [2](#0-1) 

However, the correct weight function used everywhere else in the codebase is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, meaning one cycle is worth ~0.00017 bytes of block space. The maximum block cycles is ~3.5 billion, so a cycle-heavy transaction can have a weight many times larger than its byte size. [4](#0-3) 

The `TxEntry::fee_rate()` method — used for block assembly sorting and eviction — correctly uses `get_transaction_weight`:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

This creates a split: admission uses `size` only, but post-admission sorting and eviction use `weight`. A transaction with `cycles >> size * (1/DEFAULT_BYTES_PER_CYCLES)` passes the fee gate paying only `min_fee_rate * size / 1000` shannons, but its true weight-based cost is `min_fee_rate * weight / 1000` shannons — potentially orders of magnitude higher.

The `pre_check` function is the sole admission gate for all transactions entering the pool (via RPC `send_transaction`, P2P relay, and re-add after reorg): [6](#0-5) 

### Impact Explanation

An unprivileged transaction sender can submit a transaction with a very small serialized size but extremely high cycle consumption (e.g., a script that runs a tight loop up to `max_tx_verify_cycles = 70,000,000` cycles). The byte size of such a transaction may be only a few hundred bytes, but its weight is `max(size, 70_000_000 * 0.000_170_571_4) ≈ 11,940` bytes. With `min_fee_rate = 1000 shannons/KW`, the correct minimum fee is ~11,940 shannons, but the check only requires ~size/1000 shannons (e.g., ~300 shannons for a 300-byte tx). The attacker pays ~40x less than the fee rate they actually consume from block space. This degrades miner revenue and allows mempool spam with cycle-heavy transactions at artificially low cost.

### Likelihood Explanation

The entry path is fully open: any RPC caller can invoke `send_transaction`, and any P2P peer can relay a transaction. Crafting a high-cycle, small-byte transaction requires only writing a CKB script that loops — a standard capability for any script author. The `max_tx_verify_cycles` cap (70M by default) bounds the maximum exploitation ratio but does not eliminate it. The discrepancy is acknowledged in a code comment, confirming the developers are aware it is an approximation. [7](#0-6) 

### Recommendation

Replace the size-only fee check in `check_tx_fee` with the true weight. Since cycles are not yet known at `pre_check` time (verification happens after), use the declared cycles (passed as `remote` hint) or the configured `max_tx_verify_cycles` as a conservative upper bound for the weight calculation. Alternatively, perform a second fee check after verification completes and the actual cycle count is known, rejecting the transaction at that point if the weight-based fee is insufficient.

### Proof of Concept

1. Craft a CKB transaction with a lock script that executes a tight loop consuming ~70,000,000 cycles. The serialized transaction size is ~300 bytes.
2. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
3. `check_tx_fee` computes `min_fee = 1000 * 300 / 1000 = 300 shannons`.
4. Set `outputs_capacity = inputs_capacity - 300` (fee = 300 shannons). The check passes.
5. After verification, the true weight = `max(300, 70_000_000 * 0.000_170_571_4)` ≈ 11,940. The correct minimum fee should be 11,940 shannons — the attacker paid only ~2.5% of the required fee.
6. The transaction enters the pool, is sorted by its true (low) weight-based fee rate, and consumes cycle budget from block assembly at a heavily subsidized fee. [8](#0-7) [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
