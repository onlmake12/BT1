### Title
Stale `descendants_*` stats after `remove_entry_and_descendants` enables flash-child eviction-order manipulation — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

`remove_entry_and_descendants` clears all parent/child links **before** calling `remove_entry` on each removed entry. Because `update_ancestors_index_key` resolves ancestors through those same links, it finds an empty set and never decrements the `descendants_fee / descendants_size / descendants_cycles / descendants_count` fields of entries that are **ancestors** of the evicted subtree. Those ancestors retain permanently inflated `descendants_*` stats, which inflates their `evict_key.fee_rate` and makes them artificially resistant to eviction. An unprivileged tx-pool submitter can exploit this with a two-step flash-child attack: submit a high-fee child to inflate a parent's stats, then evict the child via a conflicting transaction, leaving the parent with a stale, inflated eviction key.

---

### Finding Description

`remove_entry_and_descendants` is the function used by `resolve_conflict`, `resolve_conflict_header_dep`, and `check_and_record_ancestors` to atomically remove a transaction and all of its in-pool descendants. [1](#0-0) 

The implementation first strips every link for every entry in the removal set, then calls `remove_entry` on each:

```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
for id in &removed_ids {
    self.remove_entry_links(id);   // ← clears ALL parent/child links
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry` then calls both update helpers: [2](#0-1) 

`update_ancestors_index_key` resolves the ancestor set through `self.links.calc_ancestors(...)`: [3](#0-2) 

Because all links were already cleared, `calc_ancestors` returns an empty set. The loop body never executes, so **no ancestor's `descendants_*` fields are decremented**. The comment's stated intent ("so that we won't update_descendants_index_key") only needed to suppress the *descendant* direction (since all descendants are being removed), but the implementation also silences the *ancestor* direction, which is a bug.

The `EvictKey` for each remaining ancestor is computed from those stale fields: [4](#0-3) 

`fee_rate = max(descendants_feerate, own_feerate)`. With inflated `descendants_fee` and `descendants_size`, `descendants_feerate` is higher than reality, so `evict_key.fee_rate` is higher than reality, making the ancestor **harder to evict** than it should be.

The `AncestorsScoreSortKey` (block-template priority) is unaffected because it uses `ancestors_*` fields, not `descendants_*` fields. [5](#0-4) 

---

### Impact Explanation

When the tx-pool is near capacity and the node must evict entries, it selects the entry with the lowest `evict_key` (lowest `descendants_feerate`). A parent tx whose `descendants_*` stats are stale appears to have a higher `descendants_feerate` than it actually does, so it is skipped in favour of other entries. Legitimate transactions from other users — with correctly computed, genuinely higher fee rates — may be evicted instead. The stale state persists until the parent tx is itself removed (confirmed in a block, expired, or manually evicted), so the distortion is long-lived.

---

### Likelihood Explanation

The attack is reachable by any unprivileged `send_transaction` RPC caller or P2P relay peer. The entry path is:

1. Submit parent tx P (low fee rate) → accepted into pool.
2. Submit child tx C (spends P's output **and** an attacker-controlled UTXO X; high fee rate) → P's `descendants_*` stats are updated upward.
3. Submit conflicting tx C′ (spends UTXO X; meets RBF rules if RBF is enabled) → `resolve_conflict` → `remove_entry_and_descendants(C)` → P's `descendants_*` stats are **not** decremented. [6](#0-5) 

The incremental cost to the attacker is only `C′.fee − C.fee` (the RBF premium), which can be made arbitrarily small by choosing C with a tiny absolute fee but a high fee *rate* (small-size, high-fee-per-byte transaction). The attack is repeatable and does not require any privileged role, leaked key, or majority hash power.

---

### Recommendation

Update ancestors' `descendants_*` stats **before** clearing links in `remove_entry_and_descendants`. One approach:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // 1. Decrement ancestors' descendants_* stats while links are still intact.
    for rid in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(rid).map(|e| e.inner.clone()) {
            self.update_ancestors_index_key(&entry, EntryOp::Remove);
        }
    }

    // 2. Now clear links so that remove_entry skips the (already-done) ancestor update
    //    and also skips the descendant update (all descendants are being removed).
    for rid in &removed_ids {
        self.remove_entry_links(rid);
    }

    removed_ids.iter().filter_map(|rid| self.remove_entry(rid)).collect()
}
```

Alternatively, add a dedicated "bulk remove" path that does not rely on `remove_entry`'s per-entry link traversal.

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

impl AncestorsScoreSortKey {
    /// compare tx fee rate with ancestors fee rate and return the min one
    pub(crate) fn min_fee_and_weight(&self) -> (Capacity, u64) {
        // avoid division a_fee/a_weight > b_fee/b_weight
        let tx_weight = u128::from(self.fee.as_u64()) * u128::from(self.ancestors_weight);
        let ancestors_weight = u128::from(self.ancestors_fee.as_u64()) * u128::from(self.weight);

        if tx_weight < ancestors_weight {
            (self.fee, self.weight)
        } else {
            (self.ancestors_fee, self.ancestors_weight)
        }
    }
}
