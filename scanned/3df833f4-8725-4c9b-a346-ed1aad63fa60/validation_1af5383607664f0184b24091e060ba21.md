### Title
`CyclesEstimator` resolves cells against a fresh snapshot but verifies scripts against a stale captured snapshot — (`rpc/src/module/chain.rs`)

---

### Summary

`CyclesEstimator::run()` captures one snapshot (`snapshot`, line 2376) for `ScriptVerifier`, but delegates cell resolution to `self` as `CellProvider`, whose `cell()` method fetches a **new** `self.shared.snapshot()` on every invocation (line 2349). If the chain tip advances between those two points, the resolved `CellMeta` objects reference cells that exist only in the newer snapshot, while `snapshot.as_data_loader()` (line 2385) belongs to the older snapshot and cannot supply their data. This is the same class of bug as M-25: an identifier obtained from one context (the newer snapshot) is used in a different, incompatible context (the older snapshot's data loader).

---

**Exploit path**

1. Chain is at tip N. RPC caller invokes `estimate_cycles` with a transaction whose inputs reference cells created in block N.
2. `snapshot` (line 2376) is captured at tip N.
3. A new block N+1 is committed. `self.shared.snapshot()` now returns tip N+1.
4. `resolve_transaction` calls `self.cell()` → snapshot B = tip N+1 → cells are resolved as live (they exist in N+1's live-cell index).
5. `ScriptVerifier` is constructed with `snapshot.as_data_loader()` = snapshot A (tip N). Snapshot A's `COLUMN_CELL_DATA` does not contain data for cells created in block N+1 (they were added to the store only when N+1 was committed, after snapshot A was taken).
6. During script execution, `CKB_SYSCALL_LOAD_CELL_DATA` (or `load_cell_data_as_code`) calls `snapshot_A.get_cell_data(out_point)` → returns `None` → syscall returns `INDEX_OUT_OF_BOUND` or the verifier aborts with a data-load error.
7. `estimate_cycles` returns `TransactionFailedToVerify` for a transaction that is actually valid at the current tip.

The same mismatch applies to `header_deps`: a block hash that passes `check_valid` under snapshot B (tip N+1) may not be present in snapshot A's store, causing `load_header` to fail inside the VM.

---

### Impact Explanation

Any unprivileged RPC caller can trigger this by submitting a transaction that spends or depends on cells/headers committed in the most recent block, then racing the `estimate_cycles` call against the next block arrival. The result is a spurious `TransactionFailedToVerify` error from the public `estimate_cycles` RPC endpoint. Callers relying on this RPC for fee/cycle budgeting receive a false negative and cannot determine the correct fee for a valid transaction. Because the mismatch is structural (two independent snapshot acquisitions), it is reproducible whenever the chain advances during the RPC call window.

---

### Likelihood Explanation

CKB produces a block roughly every 8–10 seconds. The window between line 2376 and the first `cell()` call inside `resolve_transaction` is on the order of microseconds under normal load, but under high RPC concurrency or a slow disk the window widens. The condition is non-trivial to trigger deliberately but is a latent correctness hazard that any RPC caller can hit without any special privilege. The structural defect (two snapshot acquisitions) is always present in the code regardless of timing.

---

### Recommendation

Capture a single snapshot at the top of `run()` and implement `CellProvider` / `HeaderChecker` against that snapshot, mirroring the pattern used in `TxPool::resolve_tx_from_pool`:

```rust
pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
    let snapshot = self.shared.cloned_snapshot();   // one snapshot for everything
    let consensus = snapshot.cloned_consensus();

    // Build a cell provider backed by the same snapshot
    let cell_provider = SnapshotCellProvider::new(&snapshot);
    match resolve_transaction(tx.into_view(), &mut HashSet::new(),
                              &cell_provider, &snapshot) {
        Ok(resolved) => {
            let tip_header = snapshot.tip_header();
            let tx_env = TxVerifyEnv::new_submit(tip_header);
            ScriptVerifier::new(
                Arc::new(resolved),
                snapshot.as_data_loader(),   // same snapshot
                consensus,
                Arc::new(tx_env),
            ).verify(consensus.max_block_cycles)
            ...
        }
    }
}
```
