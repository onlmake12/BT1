### Title
Empty `GetBlockProposal` Message Proceeds Without Early Return, Causing Unnecessary Async Work and State Mutation - (File: `sync/src/relayer/get_block_proposal_process.rs`)

### Summary

`GetBlockProposalProcess::execute()` does not guard against an empty proposals list. When a peer sends a `GetBlockProposal` message with zero proposals, the handler passes all validation checks, makes an unnecessary async call into the tx-pool service (acquiring locks), mutates sync state with an empty vector, and returns `Status::ok()` — signaling success with no penalty or detection. This is the direct CKB analog of the `seizeInternal(0)` bug: a zero-value operation that consumes resources and touches state without producing any meaningful effect.

### Finding Description

In `sync/src/relayer/get_block_proposal_process.rs`, the `execute()` method processes incoming `GetBlockProposal` relay messages:

```rust
pub async fn execute(self) -> Status {
    let shared = self.relayer.shared();
    let message_len = self.message.proposals().len();
    {
        let limit = shared.consensus().max_block_proposals_limit()
            * (shared.consensus().max_uncles_num() as u64);
        if message_len as u64 > limit {          // passes when message_len == 0
            return StatusCode::ProtocolMessageIsMalformed...;
        }
    }

    let proposals: HashSet<packed::ProposalShortId> =
        self.message.proposals().to_entity().into_iter().collect();

    if proposals.len() != message_len {          // passes: 0 == 0
        return StatusCode::RequestDuplicate...;
    }

    // *** NO early-return for empty proposals ***

    let fetched_transactions = {
        let tx_pool = self.relayer.shared.shared().tx_pool_controller();
        let fetch_txs = tx_pool.fetch_txs(proposals.clone()).await;  // async call with empty set
        ...
    };

    let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
        .into_iter()
        .filter(...)
        .collect();                              // empty vec

    self.relayer
        .shared()
        .state()
        .insert_get_block_proposals(self.peer, not_exist_proposals);  // state mutation with empty vec

    // loop over empty fetched_transactions — nothing sent
    Status::ok()   // returns success, no penalty
}
``` [1](#0-0) 

When `proposals` is empty:

1. Both guards (`> limit`, `len != message_len`) pass trivially.
2. `tx_pool.fetch_txs(empty_set).await` is called — this sends a message through the tx-pool service channel and awaits a response, acquiring the tx-pool read lock for no purpose.
3. `state().insert_get_block_proposals(self.peer, empty_vec)` mutates the shared sync state with an empty vector.
4. The function returns `Status::ok()`, which means no warning is logged and no ban is applied to the peer.

Compare with `BlockProposalProcess::execute()`, which correctly guards against empty work:

```rust
if unknown_txs.is_empty() {
    return Status::ignored();
}
// ...
if asked_txs.is_empty() {
    return Status::ignored();
}
``` [2](#0-1) 

`GetBlockProposalProcess` lacks this pattern entirely.

### Impact Explanation

An unprivileged peer can repeatedly send `GetBlockProposal` messages with zero proposals. Each such message:

- Causes an async round-trip into the tx-pool service (channel send + lock acquisition + response), consuming CPU and lock time on the tx-pool write path.
- Mutates the shared sync state (`insert_get_block_proposals`) with a no-op write, adding unnecessary lock contention on the sync state.
- Returns `Status::ok()`, so the relay dispatcher never logs a warning, never applies a ban, and never applies backpressure — the peer is never penalized.

A malicious peer can sustain a high rate of such messages to degrade tx-pool responsiveness and sync-state throughput for legitimate operations (transaction submission, block assembly, proposal tracking), constituting a resource-exhaustion / availability impact.

### Likelihood Explanation

Any connected peer can send `GetBlockProposal` messages; no privilege is required. The `RelayV3` protocol handler dispatches these messages directly to `GetBlockProposalProcess::execute()`. The empty-proposals case is trivially constructable and requires no special knowledge of chain state. Because the node returns `Status::ok()` and applies no rate-limit or penalty specific to this message type, the attack can be sustained indefinitely from a single peer connection.

### Recommendation

Add an early-return guard immediately after the duplicate-check, mirroring the pattern used in `BlockProposalProcess`:

```rust
if proposals.is_empty() {
    return Status::ignored();
}
```

This should be inserted after line 52 (after the `proposals` `HashSet` is built and the duplicate check passes) and before the `tx_pool.fetch_txs` call, so that zero-proposal messages are rejected cheaply without touching the tx-pool or sync state. [3](#0-2) 

### Proof of Concept

1. Connect to a CKB node as a peer on the `RelayV3` protocol.
2. Construct a valid `GetBlockProposal` molecule message with an empty `proposals` field (a zero-length `ProposalShortIdVec`).
3. Send this message repeatedly at high frequency.
4. Observe: the node processes each message, makes an async `fetch_txs` call into the tx-pool, mutates sync state, and returns `Status::ok()` — no ban, no warning, no backpressure.
5. Monitor tx-pool lock contention and sync-state lock contention; both increase proportionally to the message rate, degrading legitimate operations such as transaction submission and block template generation.
