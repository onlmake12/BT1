Audit Report

## Title
Unbounded JSON-RPC Batch Requests Combined with Free Full-VM `estimate_cycles` Execution Enable Node CPU Exhaustion — (`rpc/src/server.rs`, `rpc/src/module/chain.rs`)

## Summary

The CKB JSON-RPC server's batch-request size limit (`JSONRPC_BATCH_LIMIT`) is opt-in and unset by default, meaning any caller can submit an arbitrarily large batch. The `estimate_cycles` RPC method, enabled in the default module list, invokes `ScriptVerifier::verify(max_block_cycles)` — full CKB-VM execution up to 10 billion cycles — with no fee, no authentication, and no per-caller rate limit. An attacker can combine these two facts to saturate the node's CPU with a stream of concurrent batch requests, rendering the RPC service unresponsive to miners and dApp users.

## Finding Description

**Root cause 1 — `JSONRPC_BATCH_LIMIT` is never initialized by default**

`JSONRPC_BATCH_LIMIT` is a `static OnceLock<usize>` that is only populated when `config.rpc_batch_limit` is `Some(...)`:

```rust
// rpc/src/server.rs L34
static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new();

// rpc/src/server.rs L53-55
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

The shipped default config leaves the option commented out:

```toml
# resource/ckb.toml L205-208
# By default, there is no limitation on the size of batch request size
# rpc_batch_limit = 2000
```

The batch handler only enforces the limit when the `OnceLock` is populated; when it returns `None` (the default), the guard is skipped and every call in the batch is processed unconditionally:

```rust
// rpc/src/server.rs L274-289
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    { return make_error_response(...); }
    // No limit → process every call
    let stream = stream::iter(calls)
        .then(move |call| { ... io.handle_call(call, T::default()).await })
        ...
    (... StreamBodyAs::json_array(stream) ...).into_response()
}
```

Critically, the response is a **streaming body** (`StreamBodyAs::json_array`). The `TimeoutLayer` applied at the axum router level (`Duration::from_secs(30)`) governs the time until response headers are sent, not the time until the stream is fully consumed. Because the streaming response begins immediately (headers are flushed before any batch item is processed), the 30-second wall-clock timeout does **not** bound the total CPU time spent processing the batch items. The stream continues until all calls are exhausted or the connection is dropped.

**Root cause 2 — `estimate_cycles` runs full VM execution for free**

`CyclesEstimator::run()` resolves the transaction and calls `ScriptVerifier::verify(max_cycles)` where `max_cycles = consensus.max_block_cycles` (10,000,000,000 on mainnet):

```rust
// rpc/src/module/chain.rs L2380-2389
let max_cycles = consensus.max_block_cycles;
...
ScriptVerifier::new(Arc::new(resolved), snapshot.as_data_loader(), consensus, Arc::new(tx_env))
    .verify(max_cycles)
```

This is synchronous, CPU-bound work executed inside the async Tokio runtime, blocking the worker thread for the duration of script execution. No fee is charged, no authentication is required, and no per-IP or per-method rate limit exists. The `estimate_cycles` method is registered in the `Chain` module, which is included in the default module list:

```toml
# resource/ckb.toml L190
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

**Combined attack path**

1. Attacker deploys a cell whose script is a RISC-V tight loop consuming close to `max_block_cycles` cycles.
2. Attacker opens multiple concurrent HTTP connections to the RPC endpoint and sends a JSON array of N `estimate_cycles` calls per connection.
3. Because `JSONRPC_BATCH_LIMIT` is `None`, the server processes all N calls sequentially per connection, each blocking a Tokio worker thread for up to the full VM execution budget.
4. Across concurrent connections, all Tokio worker threads are occupied with VM execution; legitimate RPC calls (miner `get_block_template`, transaction relay) queue indefinitely or time out.

## Impact Explanation

This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**. A targeted node's RPC service becomes fully unresponsive for the duration of the attack. Miners cannot fetch block templates (direct revenue loss), dApp backends and relayers cannot submit transactions, and the node cannot participate normally in the network. The attack requires zero CKB balance, zero cryptographic material, and only a single on-chain cell with a looping script.

## Likelihood Explanation

The default bind address is `127.0.0.1`, which limits exposure to processes on the same host. However, mining pools, public RPC endpoints, and dApp backends routinely rebind to `0.0.0.0` or a public interface. For any such node, the attack is trivially reachable by any external HTTP client. Even for localhost-only nodes, any compromised process on the same machine qualifies. The `rpc_batch_limit` protection exists in the codebase but is explicitly opt-in; the default config documents that no limit is applied, meaning most deployed nodes are unprotected. The attack is repeatable with no cooldown.

## Recommendation

1. **Set a safe default for `rpc_batch_limit`**: Change `resource/ckb.toml` to uncomment and set `rpc_batch_limit = 200` (or a similarly conservative value) so `JSONRPC_BATCH_LIMIT` is always initialized. The current opt-in model inverts the safe default.
2. **Add a configurable cycle cap to `estimate_cycles`**: Introduce a `max_estimate_cycles` config option (defaulting to a fraction of `max_block_cycles`, e.g., `70_000_000` matching `max_tx_verify_cycles`) and pass it instead of `consensus.max_block_cycles` in `CyclesEstimator::run()`.
3. **Move CPU-bound VM execution off the async runtime**: Spawn `ScriptVerifier::verify` onto a dedicated blocking thread pool (`tokio::task::spawn_blocking`) to prevent Tokio worker thread starvation.
4. **Apply per-IP or per-connection rate limiting** to expensive RPC methods (`estimate_cycles`, `dry_run_transaction`) at the server layer.

## Proof of Concept

```python
import json, requests, threading

# Attacker has deployed a cell whose script is a RISC-V tight loop
expensive_tx = { ... }  # valid cell deps referencing the looping script on-chain

batch = [
    {"jsonrpc": "2.0", "method": "estimate_cycles", "params": [expensive_tx], "id": i}
    for i in range(5000)
]

def flood():
    # Each POST starts a streaming response; the 30s TimeoutLayer does not
    # bound stream completion, so all 5000 calls are processed per connection.
    requests.post("http://<node-rpc>:8114", json=batch, timeout=None)

# Saturate all Tokio worker threads across concurrent connections
threads = [threading.Thread(target=flood) for _ in range(16)]
for t in threads: t.start()
# Node RPC becomes unresponsive to all other callers during processing
```

**Verification steps (unit level):**
1. Start a CKB node with default config (no `rpc_batch_limit`).
2. Deploy a cell with a script that loops for ~`max_block_cycles` cycles.
3. Send a batch of 1000 `estimate_cycles` calls referencing that cell.
4. Concurrently attempt `get_tip_block_number` from a second client — observe timeout/no response.

        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();

                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),

                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),

                        err,
                    )),
                }
            }
