Audit Report

## Title
Unbounded CPU Exhaustion via Repeated `test_tx_pool_accept` with Max-Cycle Script ‚Äî (`tx-pool/src/process.rs`, `rpc/src/module/pool.rs`)

## Summary
`_test_accept_tx` hardcodes `max_cycles = self.consensus.max_block_cycles()` and passes `command_rx = None` to `verify_rtx`, causing script verification to run synchronously via `block_in_place` with no suspension mechanism. Because the service loop spawns each `Message::TestAcceptTx` as an independent tokio task and the transaction is never inserted into the pool, N concurrent callers produce N concurrent uninterruptible max-cycle verifications that saturate all tokio worker threads, starving block template generation and relay processing.

## Finding Description
In `tx-pool/src/util.rs` (lines 85‚Äì131), `verify_rtx` branches on `command_rx`: when `Some`, it calls `verify_with_pause` (async, interruptible via `ChunkCommand`); when `None`, it falls into the `else` branch and calls `block_in_place(|| ContextualTransactionVerifier::new(...).verify(max_tx_verify_cycles, false))` ‚Äî a fully synchronous, blocking call that pins the tokio worker thread for the entire verification duration.

`_test_accept_tx` (`tx-pool/src/process.rs`, lines 779‚Äì800) always passes `None` as `command_rx` and always uses `self.consensus.max_block_cycles()` as the cycle limit. In contrast, the normal `_process_tx` path (`tx-pool/src/process.rs`, lines 705‚Äì732) receives a real `command_rx` from `VerifyMgr` (`tx-pool/src/verify_mgr.rs`, line 152: `Some(&mut self.command_rx)`), making it interruptible on block arrival.

The service loop (`tx-pool/src/service.rs`, lines 619‚Äì621) spawns each incoming message as a separate tokio task with `handle_clone.spawn(process(service_clone, message))`. There is no concurrency cap on `TestAcceptTx` messages. The deduplication check in `test_accept_tx` (`tx-pool/src/process.rs`, lines 390‚Äì397) only checks `verify_queue` and `orphan_pool`; since `_test_accept_tx` never inserts the transaction anywhere, the same transaction passes the dedup check on every call. No rate limiting exists at the RPC layer (confirmed: zero matches for rate limiting in `rpc/**/*.rs`).

**Exploit flow:**
1. Attacker constructs a transaction whose lock script consumes `max_block_cycles()` cycles.
2. Attacker opens N concurrent connections and calls `test_tx_pool_accept` in a tight loop on each.
3. Each call is dispatched as a new tokio task that calls `block_in_place` with a synchronous max-cycle verification.
4. When N ‚â• tokio worker thread count (default = CPU core count), all worker threads are pinned.
5. `get_block_template`, relay processing, and reorg handling ‚Äî all tokio tasks ‚Äî cannot be scheduled.

## Impact Explanation
Sustained CPU saturation across all tokio worker threads causes block template generation to stall, preventing the node from producing blocks and missing block intervals (8 seconds on mainnet). Relay and reorg processing share the same runtime and are equally starved. This constitutes **CKB network congestion with few costs** ‚Äî matching the **High (10001‚Äì15000 points)** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
Any node with the Pool RPC module enabled and the RPC endpoint reachable (common for mining pool backends and public RPC nodes) is directly exploitable. The attacker requires only one valid UTXO locked by a max-cycle script. The same transaction can be reused across all concurrent calls indefinitely. No hashpower, key material, or privileged access is required. The channel bound of 512 (`DEFAULT_CHANNEL_SIZE`) limits queuing but not concurrency, since tasks are spawned immediately upon dequeue.

## Recommendation
1. Pass a real `command_rx` to `verify_rtx` inside `_test_accept_tx`, consistent with the `_process_tx` path, so block arrival can interrupt in-progress verification.
2. Cap `max_cycles` for `_test_accept_tx` to a fraction of `max_block_cycles()` (e.g., `max_tx_verify_cycles` from pool config).
3. Apply a global or per-IP concurrency semaphore on `test_tx_pool_accept` at the RPC layer to bound the number of simultaneous verifications.

## Proof of Concept
1. Deploy a CKB devnet node with `[rpc] modules = ["Pool"]` and RPC bound to a reachable address.
2. Deploy a lock script that loops for `max_block_cycles()` cycles (e.g., an infinite loop capped by the cycle limit).
3. Create a transaction spending a cell locked by that script.
4. Spawn `N = CPU_CORE_COUNT` threads, each calling `test_tx_pool_accept` with that transaction in a tight loop.
5. Concurrently poll `get_block_template` and measure response latency.
6. Observe that `get_block_template` latency exceeds one block interval (8 seconds on mainnet) and that CPU utilization is pegged at 100% across all cores.
