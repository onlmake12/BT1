Audit Report

## Title
Cheap Block-Byte Stuffing via Oversized Low-Cycle Transactions — (`tx-pool/src/util.rs`, `util/types/src/core/tx_pool.rs`, `spec/src/consensus.rs`)

## Summary
The CKB tx-pool admits transactions up to `TRANSACTION_SIZE_LIMIT = 512,000 bytes`, which is ~86% of `MAX_BLOCK_BYTES = 597,000 bytes`. The minimum fee gate is computed solely from serialized byte size with no cycles-floor component, so a 512 KB transaction with a trivial lock script pays only ~512,000 shannons (~0.005 CKB) while consuming the vast majority of a block's byte budget and leaving the 3.5-billion-cycle budget almost entirely unused. Because there is no dynamic fee adjustment mechanism, this cost never rises, enabling sustained cheap block-byte stuffing by any unprivileged user.

## Finding Description
**Block limits and transaction size ceiling**

`MAX_BLOCK_BYTES` is defined as `TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1,000 = 597,000 bytes`. [1](#0-0) 

The tx-pool enforces a per-transaction ceiling of `TRANSACTION_SIZE_LIMIT = 512,000 bytes`. [2](#0-1) 

`non_contextual_verify` rejects anything above that limit but accepts anything below it. [3](#0-2) 

**Minimum fee is byte-proportional only**

`check_tx_fee` computes the minimum fee using raw serialized size with no cycles component:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [4](#0-3) 

The static default `min_fee_rate = 1,000 shannons/KB` means a 512 KB transaction requires only 512,000 shannons (~0.005 CKB). [5](#0-4) 

**Weight formula provides no surcharge for low-cycle large transactions**

`get_transaction_weight` returns `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For a trivial lock script consuming ~1,000,000 cycles: `1,000,000 × 0.000170571 ≈ 170 bytes`, so `weight = max(512,000, 170) = 512,000`. The weight collapses to the byte size, and no cycles-based fee premium applies. [6](#0-5) 

**Block assembler stops on byte exhaustion**

`TxSelector.txs_to_commit` breaks out of the selection loop as soon as either limit is exceeded. After a 512 KB transaction is selected, only ~83 KB of byte budget remains (597 KB − ~2 KB overhead − 512 KB), leaving room for at most ~138 typical 597-byte transactions. [7](#0-6) 

**No dynamic fee pressure**

`min_fee_rate` is a static node-local config value. Repeated byte-stuffed blocks do not raise the fee floor, so the attacker's cost per block never increases. [8](#0-7) 

**Byte limit is a hard consensus rule**

`BlockBytesVerifier` confirms the byte limit is enforced at consensus level, not merely as a miner preference. [9](#0-8) 

## Impact Explanation
This matches the allowed High-severity impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker submitting one 512 KB transaction per block fills ~86% of the byte budget while consuming ~0.03% of the cycles budget. Legitimate transactions are delayed indefinitely. Time-sensitive operations (DAO withdrawals, oracle updates, contract interactions with deadlines) are blocked. The cost is ~0.005 CKB per block — negligible at any realistic CKB price — and never increases due to the absence of a dynamic fee mechanism. [2](#0-1) 

## Likelihood Explanation
The attack requires no special privilege. Any account with a small CKB balance can submit transactions via the public `send_transaction` RPC. The technique is straightforward: craft a transaction with a large witness blob and a minimal always-success lock script. The static fee floor creates no economic feedback loop to deter the attacker. The gap between `TRANSACTION_SIZE_LIMIT` (512 KB) and `MAX_BLOCK_BYTES` (597 KB) is structural and exploitable on every block. [10](#0-9) 

## Recommendation
1. **Reduce `TRANSACTION_SIZE_LIMIT`** to a value that ensures no single transaction can dominate the block's byte budget (e.g., cap at 64 KB or 10% of `MAX_BLOCK_BYTES`).
2. **Introduce a cycles-floor per byte**: require that a transaction's declared cycles be at least proportional to its serialized size, so that a large-byte transaction must also consume a meaningful share of the cycles budget.
3. **Dynamic fee-rate adjustment**: implement a protocol-level mechanism analogous to EIP-1559 that raises `min_fee_rate` when recent blocks are consistently byte-full but cycles-empty, creating economic pressure against byte-stuffing. [11](#0-10) 

## Proof of Concept
1. Deploy an always-success lock script cell on a local devnet.
2. Construct a transaction spending one cell locked by that script, with a single output returning capacity to the same lock, and a witness field padded to ~511,900 bytes of arbitrary data.
3. Verify serialized size: ~512,000 bytes — below `TRANSACTION_SIZE_LIMIT`, accepted by the pool.
4. Pay minimum fee: `1,000 shannons/KB × 512 KB = 512,000 shannons`.
5. Submit via `send_transaction` RPC. The transaction enters the pool, is proposed in the next proposal window, and committed in the following block.
6. Observe: block byte usage ~514,000 / 597,000 bytes; cycles used ~1,000,000 / 3,500,000,000 (0.03% utilization).
7. Repeat every block. Cost: ~512,000 shannons/block ≈ 0.005 CKB/block.
8. Confirm that legitimate pending transactions are not committed while the attacker's transactions are continuously submitted. [4](#0-3)

/// The precision is set so that the difference between MAX_BLOCK_CYCLES * DEFAULT_BYTES_PER_CYCLES
/// and MAX_BLOCK_BYTES is less than 1.

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"
)]
pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
