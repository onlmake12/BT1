Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites Live `total_tx_size`/`total_tx_cycles` Accounting in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, aggregate statistics are snapshotted before `check_and_record_ancestors` runs. That call can evict conflicting pool entries via `remove_entry`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`. However, `add_entry` then unconditionally overwrites those live fields with the stale pre-eviction snapshot, erasing the decrements. The result is a permanently inflated `total_tx_size` that can drive `limit_size` into an infinite loop, hanging the node's tx-pool thread.

## Finding Description
`PoolMap` maintains two aggregate fields:

```rust
pub(crate) total_tx_size: usize,
pub(crate) total_tx_cycles: Cycle,
``` [3](#0-2) 

`add_entry` is the sole insertion path for all three pool states:

```rust
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;  // snapshot HERE
// ...
evicts = self.check_and_record_ancestors(&mut entry)?;         // may call remove_entry
// ...
self.total_tx_size = total_tx_size;   // stale snapshot OVERWRITES live field
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` reads `self.total_tx_size` at call time and adds the new entry's size: [4](#0-3) 

`remove_entry` (called inside `check_and_record_ancestors` for each evicted tx) correctly decrements the live field via `update_stat_for_remove_tx`: [5](#0-4) [6](#0-5) 

Because the snapshot is taken before evictions and written back after them, every eviction's decrement is silently discarded. With T1 (size S1) in the pool and T2 (size S2) being inserted while evicting T1:

- Snapshot: `total_tx_size = S1 + S2`
- After eviction of T1: `self.total_tx_size = 0`
- After write-back: `self.total_tx_size = S1 + S2` (should be `S2`)

The secondary consequence is in `limit_size`:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    if let Some(id) = next_evict_entry() {
        // evict
    }
    // no else — if pool is empty but total_tx_size is inflated, loops forever
}
``` [2](#0-1) 

If `S1 + S2 > max_tx_pool_size`, `limit_size` evicts T2, leaving the pool empty with `total_tx_size = S1`. If `S1 > max_tx_pool_size`, `next_evict_entry()` returns `None` on every iteration, the `if let Some` arm never fires, and the while loop spins forever — hanging the tx-pool processing thread permanently.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

An unprivileged attacker can permanently hang the CKB node's tx-pool processing thread by triggering the infinite loop in `limit_size`. Once hung, the node cannot process any further transactions (all `send_transaction` RPC calls block or time out), and the node must be restarted. Even without reaching the infinite-loop condition, `total_tx_size` is permanently inflated after each eviction-triggering insertion, causing `limit_size` to cascade-evict legitimate transactions and report `Reject::Full` to submitters.

## Likelihood Explanation
The trigger condition — a new transaction whose inputs consume a cell dep already referenced by an existing pool transaction — is reachable by any unprivileged `send_transaction` RPC caller or P2P relay sender. No special privilege is required. The attacker only needs to craft a transaction whose `cell_deps` overlap with those of existing mempool transactions. Each such submission permanently inflates `total_tx_size`. Repeated submissions compound the drift without bound until the node restarts. The eviction path is explicitly documented in the code comment at line 199. [7](#0-6) 

## Recommendation
Move the stat computation **after** `check_and_record_ancestors` completes, so it reflects the post-eviction pool state:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER evictions have already updated self.total_tx_*
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, accumulate evicted sizes during `check_and_record_ancestors` and subtract them from the pre-computed snapshot before the final assignment. Also add a guard in `limit_size` to break if `next_evict_entry()` returns `None` to prevent infinite loops from any future accounting drift.

## Proof of Concept
1. Configure a node with `max_tx_pool_size = N`.
2. Submit transaction **T1** (size `S1`, where `S1 < N`) with `cell_dep = OutPoint(H, 0)`. Pool accepts it; `total_tx_size = S1`.
3. Submit transaction **T2** (size `S2`, where `S1 + S2 > N`) whose input consumes `OutPoint(H, 0)`. Inside `add_entry`: snapshot = `S1 + S2`; `check_and_record_ancestors` evicts T1 via `remove_entry`, decrementing `self.total_tx_size` to `0`; write-back sets `self.total_tx_size = S1 + S2`.
4. `limit_size` is called. Since `S1 + S2 > N`, it evicts T2, setting `self.total_tx_size = S1`. If `S1 > N`, `next_evict_entry()` returns `None` on every subsequent iteration and the while loop never terminates — the tx-pool thread hangs indefinitely.
5. All subsequent `send_transaction` RPC calls block. The node must be restarted to recover.

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
