### Title
Attacker-Controlled `observed_addr` Injection into Victim's Advertised `listen_addrs` via Identify Protocol — (`network/src/protocols/identify/mod.rs`)

---

### Summary

An unprivileged remote peer can poison the victim node's advertised listen addresses by establishing inbound connections and sending crafted `observed_addr` values. Because `NetworkState::observed_addrs` has no content validation and is used as a fallback source for `local_listen_addrs`, the victim will re-advertise up to 10 attacker-controlled addresses as its own to every subsequent outbound peer.

### Impact Explanation

- Any node that does not have `MAX_RETURN_LISTEN_ADDRS` (10) entries in `public_addrs` — the default for nodes behind NAT with no `public_addresses` configured — will advertise up to 10 fully attacker-controlled addresses as its own listen addresses in every outgoing `IdentifyMessage`.
- Receiving peers store these addresses in their peer stores and propagate them via the Discovery protocol, causing network-wide address poisoning for the victim.
- Nodes attempting to reach the victim via the poisoned addresses will fail, facilitating eclipse attack setup against the victim.

---

### Likelihood Explanation

- Requires only the ability to open inbound TCP connections to the victim — no authentication, no PoW, no privileged role.
- The `is_reachable` filter is trivially bypassed by using any globally routable IP (e.g., `8.8.8.8`).
- The victim's `public_addrs` is empty for any node behind NAT without explicit `public_addresses` config, which is the common deployment scenario.
- The attacker only needs to hold connections open (within `max_inbound_peers`) to sustain the attack indefinitely.

---

### Recommendation

1. **Validate `observed_addr` against the session's actual remote address.** Only accept an `observed_addr` if it matches the IP of the session that sent it (i.e., the peer can only report what it actually observed, not an arbitrary IP).
2. **Do not include unverified `observed_addrs` in outgoing `listen_addrs`.** `observed_addrs` should only be used for self-discovery (e.g., NAT detection), not re-advertised to third parties.
3. **Require corroboration from multiple independent peers** before promoting an `observed_addr` to the advertised address set (similar to how libp2p's identify protocol handles external address confirmation).
