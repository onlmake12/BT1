### Title
Missing Upper-Bound Validation on DAO Witness `header_dep_index` Causes Rust/C-Script Consensus Divergence — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` reads `header_dep_index` from the witness as a raw `u64` and uses it directly to index into `header_deps`, with no check that the value fits within the range the on-chain C DAO script can handle. The C DAO script reads only the **lowest byte** of the 8-byte witness field as the index. This mismatch means the Rust node and the C script can resolve different deposit headers for the same transaction, causing the Rust node to reject valid DAO withdrawals (censorship) or to accept a transaction that the C script will reject (consensus split).

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads the deposit header index from the witness `input_type` field as a full little-endian `u64`: [1](#0-0) 

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // uses full u64
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

There is **no assertion** that `header_dep_index <= u8::MAX` (or any other maximum). The on-chain C DAO script, however, reads only the **lowest byte** of the same 8-byte witness field as the index. This is explicitly documented in the test suite: [2](#0-1) 

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
``` [3](#0-2) 

The test only covers the case where the Rust code rejects (because `header_deps[257]` is the wrong block). It does **not** cover the inverse: `header_deps[1]` = some unrelated block, `header_deps[257]` = correct deposit block. In that case:

- **Rust `DaoCalculator`**: resolves `header_deps[257]` → correct deposit block → block number matches → **accepts**
- **C DAO script**: resolves `header_deps[1]` → wrong block → block number mismatch → **rejects**

The `DaoCalculator::transaction_fee()` is called in two places in the verification pipeline:

1. **Tx-pool pre-check** (`tx-pool/src/util.rs`, `check_tx_fee`): calls `DaoCalculator` directly, independently of the C script. [4](#0-3) 

2. **Full contextual verification** (`verification/src/transaction_verifier.rs`, `FeeCalculator::transaction_fee`): called after `self.script.verify()`. [5](#0-4) 

The missing constraint is directly analogous to the reported issue: just as the Noir circuit sets `email_address.len = email_address_sequence.length` without checking `length <= MAX_EMAIL_ADDRESS_LENGTH`, the Rust node uses `header_dep_index as usize` without checking `header_dep_index <= u8::MAX`.

---

### Impact Explanation

**Scenario A — Valid withdrawal censored (false rejection):**

An attacker or a DAO depositor crafts a withdrawal transaction with:
- `witness.input_type` = `257u64` (LE bytes: `[0x01, 0x01, 0x00, …]`)
- `header_deps[1]` = correct deposit block hash
- `header_deps[257]` = some unrelated block hash

The C DAO script (running in CKB-VM) resolves index `1` → correct deposit block → **accepts**. The Rust `DaoCalculator` resolves index `257` → unrelated block → block number mismatch → **rejects** with `DaoError::InvalidOutPoint`. The transaction is dropped from the tx-pool and never included in a block, permanently censoring a valid DAO withdrawal.

**Scenario B — Invalid withdrawal accepted by Rust, rejected by C script (consensus split):**

Reverse the placement: `header_deps[257]` = correct deposit block, `header_deps[1]` = unrelated block. The Rust `DaoCalculator` accepts the transaction (block number matches at index 257). The C DAO script rejects it (wrong block at index 1). If the Rust node assembles a block containing this transaction, other nodes executing the C script will reject the block, causing a chain split.

---

### Likelihood Explanation

Any unprivileged transaction sender who holds a DAO cell can trigger this. Constructing a transaction with 258 `header_deps` entries (258 × 32 = 8,256 bytes) is well within the block size limit. The witness `input_type` field is fully attacker-controlled. No privileged access, key material, or majority hashpower is required.

---

### Recommendation

Add an explicit upper-bound check on `header_dep_index` immediately after reading it from the witness, before using it as an array index:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

should become:

```rust
let index = LittleEndian::read_u64(&header_deps_index_data.unwrap());
if index > u8::MAX as u64 {
    return Err(DaoError::InvalidDaoFormat);
}
Ok(index)
```

This aligns the Rust node's validation with the C DAO script's actual byte-reading behavior and eliminates the divergence.

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` already demonstrates the divergence exists. The inverse scenario (Rust accepts, C rejects) can be constructed by swapping the positions of the deposit and withdraw block hashes in that test: [6](#0-5) 

Modify the test so that:
- `header_deps[257]` = `deposit_block.hash()` (Rust resolves this → block number 100 → matches cell data → Rust accepts)
- `header_deps[1]` = `withdraw_block.hash()` (C script resolves this → block number 200 ≠ 100 → C rejects)

With `witness.input_type = 257u64`, `DaoCalculator::transaction_fee()` returns `Ok(...)` while the C DAO script would return an error code, demonstrating the consensus split.

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
