### Title
Off-by-One in `MAX_VMS_COUNT` Guard Allows 17 Simultaneous VMs — (`script/src/scheduler.rs`)

### Summary

The spawn guard in `process_message_box` uses a strict `>` comparison against `MAX_VMS_COUNT`, allowing exactly one extra VM to be created beyond the declared limit of 16.

### Finding Description

`MAX_VMS_COUNT` is defined as `16`: [1](#0-0) 

The guard in `Message::Spawn` handling reads:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
``` [2](#0-1) 

The condition is `> 16` (strictly greater than), not `>= 16`. When exactly 16 VMs already exist (e.g., root VM + 15 children), the check evaluates `16 > 16 = false`, the guard is skipped, and `boot_vm` is called at line 539, inserting a 17th VM into `self.instantiated` and `self.states`. [3](#0-2) 

`boot_vm` itself has no secondary count check — it unconditionally inserts the new VM: [4](#0-3) 

### Impact Explanation

The invariant "at most `MAX_VMS_COUNT` VMs may exist simultaneously" is violated by exactly **1**: a maximum of 17 VMs can exist instead of 16. The impact is **bounded**, not unbounded as the question claims. The question's assertion of "unbounded VM resource consumption leading to node memory exhaustion" is incorrect — after the 17th VM exists, the next spawn attempt evaluates `17 > 16 = true` and is correctly rejected.

The real impact is:
- A script can consume one extra VM's worth of memory and cycles beyond the consensus-defined limit.
- This is a consensus violation: nodes enforcing the spec strictly would reject the transaction, while nodes running this code would accept it, creating a potential chain split vector.
- Cycle accounting is not directly broken by this, but the extra VM's execution cycles are still charged normally.

### Likelihood Explanation

Any unprivileged script author targeting ScriptVersion::V2 (Meepo hardfork) can trigger this by writing a script that spawns exactly 16 child VMs. The 16th spawn call passes the guard and succeeds. This is trivially reproducible in a unit test.

### Recommendation

Change `>` to `>=` on line 532:

```rust
// Before (buggy):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
``` [5](#0-4) 

### Proof of Concept

A unit test would:
1. Boot a root VM that calls `spawn` in a loop.
2. Spawn 15 child VMs (total = 16 including root). Each spawn succeeds.
3. Attempt a 16th spawn (total would be 17). With the bug, this **succeeds** and returns `SUCCESS` instead of `MAX_VMS_SPAWNED`.
4. Assert that the 16th spawn returns `MAX_VMS_SPAWNED` — this assertion **fails** against the current code, confirming the bug.

The existing test infrastructure in `script/src/verify/tests/ckb_latest/features_since_v2023.rs` already exercises `MAX_VMS_COUNT` paths and can be extended to cover this boundary condition. [6](#0-5)
