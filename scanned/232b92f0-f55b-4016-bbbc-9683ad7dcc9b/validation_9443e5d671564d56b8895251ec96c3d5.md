Audit Report

## Title
Pool Size Limit (`max_tx_pool_size`) Not Enforced After Reorg Re-insertion — (File: `tx-pool/src/process.rs`)

## Summary

During a chain reorganization, `readd_detached_tx` re-inserts detached transactions into the tx-pool by calling `_submit_entry` directly, without the subsequent `limit_size` call that the normal `submit_entry` path enforces. This allows `total_tx_size` to exceed the operator-configured `max_tx_pool_size` invariant after any reorg, with the violation persisting until the next externally submitted transaction triggers eviction via the normal path.

## Finding Description

The two insertion paths diverge at the `limit_size` call:

**Normal path** (`submit_entry`, `tx-pool/src/process.rs` lines 137–152): after `_submit_entry` returns, `limit_size` is unconditionally called to evict low-fee-rate entries until `total_tx_size <= max_tx_pool_size`.

**Reorg path** (`readd_detached_tx`, `tx-pool/src/process.rs` lines 905–910): `_submit_entry` is called inside a `for` loop with no `limit_size` call anywhere in the function body or in its caller `update_tx_pool_for_reorg` (lines 802–858). The `grep` across all `tx-pool/src/*.rs` files confirms `limit_size` appears only twice in `process.rs`, both within the normal `submit_entry` closure.

`check_tx_fee` (called at line 889) only validates the minimum fee rate of the individual transaction; it does not consult or enforce the aggregate pool size. There is no other guard in `readd_detached_tx` that bounds total pool size.

`limit_size` in `tx-pool/src/pool.rs` (lines 292–329) is the sole enforcement point: it loops while `total_tx_size > max_tx_pool_size` and evicts the lowest-fee-rate entries. It is never reached from the reorg path.

**Exploit flow:**
1. Pool is at or near `max_tx_pool_size`.
2. A competing block at the same height arrives (natural one-block reorg or attacker-relayed block).
3. `update_tx_pool_for_reorg` → `readd_detached_tx` re-inserts all transactions from the detached block that are not in the new tip.
4. Each insertion via `_submit_entry` increments `pool_map.total_tx_size` with no eviction.
5. Pool's `total_tx_size` now exceeds `max_tx_pool_size` by up to the aggregate size of the detached block's transactions.
6. The violation persists in memory until the next externally submitted transaction reaches `submit_entry` and triggers `limit_size`.

## Impact Explanation

This is a concrete violation of the tx-pool's size-bounding invariant, which is the primary resource-management mechanism for the pending transaction storage. The pool's memory footprint can exceed the operator-configured bound after every reorg, with the excess bounded by the detached block's transaction payload. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**, as the tx-pool is the node's primary in-memory transaction state store and the size limit is its key resource constraint.

## Likelihood Explanation

One-block reorgs are routine in any PoW network and require no attacker involvement — they occur naturally when two miners find blocks at the same height near-simultaneously. Any peer acting as a block relayer can deliver a valid competing block to trigger the path. No special privileges, majority hashpower, or victim mistakes are required. The condition is repeatable on every reorg event.

## Recommendation

After the `for` loop in `readd_detached_tx` completes, call `limit_size` to restore the invariant:

```rust
// tx-pool/src/process.rs, after the for-loop in readd_detached_tx
tx_pool.limit_size(&self.callbacks, None);
```

Alternatively, refactor `readd_detached_tx` to call the full `submit_entry` method (which already includes the `limit_size` call) instead of the bare `_submit_entry` free function, consistent with the normal submission path.

## Proof of Concept

**Unit/integration test plan:**
1. Configure a `TxPool` with a small `max_tx_pool_size` (e.g., 10 KB).
2. Fill the pool to exactly `max_tx_pool_size` with pending transactions.
3. Simulate a one-block reorg: call `update_tx_pool_for_reorg` with a `detached_blocks` containing transactions totaling >1 KB and an `attached_blocks` with different transactions.
4. After the call returns, assert `tx_pool.pool_map.total_tx_size <= max_tx_pool_size`.
5. Without the fix, the assertion fails — `total_tx_size` exceeds the configured limit by the size of the re-added detached transactions.
