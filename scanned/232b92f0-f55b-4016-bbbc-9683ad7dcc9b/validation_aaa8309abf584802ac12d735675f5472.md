Audit Report

## Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Enables Unbounded CPU/Memory DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The guard at line 201 of `GetLastStateProofProcess::execute` computes `(last_n_blocks as usize) * 2` without overflow protection. In Rust release builds, setting `last_n_blocks = 0x8000000000000000` causes this multiplication to wrap to zero, bypassing the 1000-entry limit entirely. A subsequent unconditional range collection at line 296 then allocates a `Vec` containing every block number in the chain, followed by O(H · log H) MMR work per request, with no peer ban triggered.

## Finding Description

**Root cause — integer overflow in the guard:** [1](#0-0) 

`last_n_blocks` is a `u64` wire field cast to `usize` (64-bit on x86-64), then multiplied by 2. In Rust release builds, integer arithmetic wraps on overflow without panicking. With `last_n_blocks = 0x8000000000000000`:

```
(0x8000000000000000usize) * 2 = 0x10000000000000000  →  wraps to 0
```

The guard reduces to `difficulties.len() + 0 > 1000`, which passes for any `difficulties.len() ≤ 1000`.

**Limit constant being bypassed:** [2](#0-1) 

**Unbounded collection after bypass:** [3](#0-2) 

With `start_block_number = 0` and `last_n_blocks = 0x8000000000000000`, the condition `last_block_number - 0 <= 0x8000000000000000` is always true for any realistic chain height. This causes `last_n_numbers` to collect **all H block numbers** into a `Vec`, completely bypassing the 1000-entry limit.

**Per-entry work in `complete_headers`:** [4](#0-3) 

For each of the H entries: `get_ancestor` (chain traversal), `get_block` (DB read), and `chain_root_mmr(*number - 1).get_root()` (MMR root computation, O(log n)).

**O(H · log H) MMR proof generation:** [5](#0-4) 

`mmr.gen_proof(items_positions)` is called with all H positions.

**No ban triggered:** `should_ban()` only fires for 4xx status codes. The crafted message bypasses the `MalformedProtocolMessage` (400) return and results in either `Status::ok()` (200) or `InternalError` (500), neither of which bans the peer. [6](#0-5) [7](#0-6) 

## Impact Explanation

A single unauthenticated P2P peer can force a full node to allocate a `Vec<BlockNumber>` of size equal to the entire chain height, perform O(H · log H) MMR computations, and read every block from the database — all per request, with no per-peer throttling or concurrency cap. On a chain of height 100,000 this is ~1.7M MMR operations per message. Multiple concurrent peers amplify this to node-wide CPU and memory exhaustion. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The exploit requires only a valid P2P connection to a light-client-serving full node — no PoW, no keys, no privileged role. The value `0x8000000000000000` fits in the `Uint64` wire field. The attack is repeatable without ban because the crafted message passes the malformed-message check and returns a non-4xx status. Any node with the light-client protocol enabled is exposed.

## Recommendation

Replace the unchecked multiplication with a saturating variant before the comparison:

```rust
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, add an explicit upper-bound check on `last_n_blocks` before any arithmetic:

```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT / 2 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

assert_eq!(overflowed, 0); // wraps to 0 in release

// Craft the exploit message:
let content = packed::GetLastStateProof::new_builder()
    .last_hash(tip_header.hash())
    .start_hash(genesis_header.hash())
    .start_number(0u64.pack())
    .last_n_blocks(0x8000000000000000u64.pack()) // triggers overflow
    .difficulty_boundary(some_valid_boundary)
    .difficulties(packed::Uint256Vec::new_builder().build()) // len=0, passes guard
    .build();
// Send to a node with chain height H.
// Result: server allocates Vec of H entries and performs O(H * log H)
// MMR operations. No ban is issued. Attack is immediately repeatable.
