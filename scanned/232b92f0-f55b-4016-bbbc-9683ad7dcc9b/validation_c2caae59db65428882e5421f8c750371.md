Audit Report

## Title
Orphan Pool DoS via Random Eviction — Attacker Permanently Occupies All Orphan Slots - (File: tx-pool/src/component/orphan.rs)

## Summary
The `OrphanPool` is capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`. When the cap is exceeded, `limit_size()` evicts entries via `HashMap::keys().next()`, which is non-deterministic and has no relationship to fee rate, declared cycles, or transaction age. Any unprivileged P2P peer can flood all 100 slots with transactions referencing fabricated `OutPoint`s that will never resolve, causing legitimate orphan transactions to be evicted with ~99% probability and marked as rejected in the relay filter.

## Finding Description
**Root cause — `limit_size()` random eviction:** [1](#0-0) 

When `self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS`, the eviction loop picks `self.entries.keys().next()` — a non-deterministic `HashMap` iterator with no fee-rate or priority ordering. There is no comparison between the incoming transaction's fee rate and the lowest-fee-rate entry currently in the pool.

**The pool structure — plain `HashMap`, no ordering:** [2](#0-1) 

**Attacker entry path — `after_process` in `process.rs`:** [3](#0-2) 

When a remote peer relays a transaction whose inputs are not in the chain or pool, `is_missing_input` is true and `add_orphan` is called unconditionally with no per-peer rate limit.

**`add_orphan_tx` inserts then calls `limit_size()`:** [4](#0-3) 

**Evicted transactions receive `Reject` signal:** [5](#0-4) 

This causes the relay layer to mark the transaction as unknown/rejected in its bloom filter, preventing re-relay of the evicted transaction.

**Expiry window — attacker must refresh:** [6](#0-5) 

`ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL`. The attacker refreshes 100 fabricated-input transactions within this window to maintain the attack indefinitely.

**No fee-based eviction exists.** Unlike the main `PoolMap` which uses `EvictKey` sorted by fee rate, the `OrphanPool` has no such structure and no per-peer insertion cap.

## Impact Explanation
This is a **High** severity issue matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

An attacker with only 100 minimum-fee transactions can permanently deny any node's orphan pool to legitimate users. In CKB's two-phase proposal/commit model, child transactions legitimately arrive before their parents and must persist in the orphan pool until the parent is seen. Under this attack, those child transactions are evicted with ~99% probability and marked as rejected in the relay filter, preventing re-relay. The attack is self-sustaining at negligible cost and can be applied simultaneously to all reachable nodes on the network.

## Likelihood Explanation
- **Attacker capability**: Any unprivileged P2P peer; no special role required.
- **Cost**: 100 transactions with minimum fee rate, refreshed every `ORPHAN_TX_EXPIRE_TIME`. Attacker transactions reference fabricated `OutPoint`s and are never committed, so the cost is effectively zero beyond the minimum relay fee.
- **No rate limiting**: `add_orphan_tx` has no per-peer cap; a single peer can fill all 100 slots.
- **Persistence**: Self-sustaining as long as the attacker refreshes before expiry.
- **Scalability**: The attacker can target multiple nodes simultaneously with the same 100 transactions (or 100 per target node).

## Recommendation
1. **Fee-based eviction**: When the orphan pool is full, evict the entry with the lowest declared fee rate (using `entry.cycle` already stored in `Entry`) rather than a random entry. Only admit the incoming transaction if its fee rate exceeds the minimum in the pool.
2. **Per-peer orphan limit**: Cap the number of orphan transactions accepted from a single peer (e.g., 10 per peer) to prevent a single attacker from occupying all 100 slots.
3. **Incoming vs. evictee comparison**: Before inserting, compare the incoming transaction's declared cycles/size against the lowest-priority orphan currently in the pool and reject the incoming transaction if it is lower priority.

## Proof of Concept
1. Attacker generates 100 transactions, each spending a fabricated `OutPoint` (e.g., `tx_hash = [0xAA; 32], index = 0`) that does not exist on-chain or in the pool.
2. Attacker connects to the target CKB node as a P2P peer and relays all 100 transactions via the standard relay protocol. Each fails `resolve_tx` with `is_missing_input = true` and is placed in the orphan pool via `add_orphan`.
3. The orphan pool is now at capacity (100/100).
4. An honest user sends a child transaction (spending the output of a pending parent). It enters `add_orphan_tx`, is inserted (pool size = 101), and `limit_size()` is called. One of the 101 entries is randomly evicted — the honest transaction is evicted with probability 100/101 ≈ 99%. The evicted transaction receives `TxVerificationResult::Reject`, marking it as rejected in the relay filter.
5. The attacker refreshes their 100 orphan transactions before `ORPHAN_TX_EXPIRE_TIME` elapses.
6. The honest user's child transaction is never retained long enough for its parent to arrive and resolve it. The attack is verifiable by adding a counter to `limit_size()` and observing that non-attacker entries are evicted at the expected rate.

```rust
    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
```
