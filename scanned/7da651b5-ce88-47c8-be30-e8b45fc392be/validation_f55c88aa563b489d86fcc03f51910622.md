### Title
Verify Queue Size Limit Exhaustion Enables Sustained Transaction Submission DoS - (File: `tx-pool/src/component/verify_queue.rs`)

### Summary

The `VerifyQueue` in CKB's tx-pool enforces a hard 256 MB size ceiling (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) before any fee-rate or contextual validity check is performed. An unprivileged attacker — reachable via the RPC `send_transaction` endpoint or the P2P relay protocol — can flood the queue with structurally valid but contextually invalid transactions, exhausting the limit and causing every subsequent legitimate transaction submission to be rejected with `Reject::Full` for as long as the attacker sustains the flood.

### Finding Description

`VerifyQueue::add_tx` is the single admission gate for all incoming transactions before script execution. It enforces only a byte-size ceiling:

```rust
// 256mb for total_tx_size limit, default max_tx_pool_size is 180mb
const DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE: usize = 256_000_000;

pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}

pub fn add_tx(...) -> Result<bool, Reject> {
    ...
    if self.is_full(tx_size) {
        return Err(Reject::Full(...));
    }
    ...
}
``` [1](#0-0) [2](#0-1) [3](#0-2) 

Both the RPC path (`notify_tx`) and the P2P relay path (`submit_remote_tx`) converge on `resumeble_process_tx`, which performs only `non_contextual_verify` before calling `enqueue_verify_queue`. No fee-rate check, no cell-existence check, and no per-sender quota are applied before admission:

```rust
pub(crate) async fn resumeble_process_tx(...) -> Result<bool, Reject> {
    self.non_contextual_verify(&tx, remote).await?;   // structure only
    ...
    self.enqueue_verify_queue(tx, is_proposal_tx, remote).await  // enters queue
}
``` [4](#0-3) 

Fee-rate validation (`check_tx_fee`) and cell resolution (`resolve_tx`) happen only **after** a worker dequeues the entry — meaning the queue slot is occupied for the full worker-processing latency even for zero-fee or cell-missing transactions. [5](#0-4) 

The individual transaction size ceiling is `TRANSACTION_SIZE_LIMIT = 512 * 1_000` bytes: [6](#0-5) 

So the queue can be filled with as few as ⌈256 MB / 512 KB⌉ = **512 transactions**.

### Impact Explanation

Once `total_tx_size` reaches 256 MB, every call to `add_tx` — from any source — returns `Err(Reject::Full(...))`. The RPC layer surfaces this as `PoolIsFull (-1106)` and the P2P relay layer marks the transaction as rejected and notifies the originating peer. Legitimate users submitting valid, fee-paying transactions are indistinguishable from the attacker's flood at the admission gate and are equally rejected. The node's transaction-processing service is effectively unavailable for the duration of the attack.

### Likelihood Explanation

The attack requires no privileged access, no mining power, and no on-chain funds. An attacker can craft transactions that:

1. Reference non-existent `OutPoint`s (cell resolution fails fast inside the worker, freeing the slot, but the attacker re-submits immediately with a different transaction hash).
2. Have zero fee (fee check is post-dequeue).
3. Are structurally valid (pass `NonContextualTransactionVerifier`).

The RPC endpoint has no documented per-IP or per-connection rate limit. The P2P relay path has a 30 req/s per-peer-per-message-type governor on the `Relayer`, but a single peer can still inject up to 30 × 512 KB = ~15 MB/s into the queue, filling it in under 20 seconds. Multiple peers multiply this rate linearly.

### Recommendation

1. **Short term**: Enforce a minimum fee-rate check (or at least a non-zero fee check) inside `resumeble_process_tx` **before** calling `enqueue_verify_queue`, so zero-fee or below-threshold transactions are rejected at the gate without consuming queue space.
2. **Short term**: Add a per-peer (P2P session or RPC connection) quota on the number of bytes that can be in-flight in the verify queue simultaneously, analogous to how `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` caps per-peer unknown-tx accumulation.
3. **Long term**: Implement RPC-level rate limiting (e.g., per-IP connection throttle) for `send_transaction` to bound the submission rate from any single source.

### Proof of Concept

```
# Attacker loop (pseudocode):
for i in 0..512:
    tx = build_tx(
        inputs=[OutPoint(tx_hash=random_nonexistent_hash, index=0)],
        outputs=[Output(capacity=100_CKB)],
        witnesses=[random_512KB_blob],   # pads to TRANSACTION_SIZE_LIMIT
    )
    rpc.send_transaction(tx)   # passes non_contextual_verify, enters VerifyQueue

# After ~512 submissions the queue is at 256 MB.
# Legitimate user:
rpc.send_transaction(valid_tx)
# → PoolIsFull (-1106)

# Attacker re-submits as workers drain slots (each drain is a fast DB-miss):
# sustains the full state indefinitely at ~512 RPC calls per drain cycle.
```

    pub(crate) async fn non_contextual_verify(
        &self,
        tx: &TransactionView,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<(), Reject> {
        if let Err(reject) = non_contextual_verify(&self.consensus, tx) {
            if reject.is_malformed_tx()
                && let Some(remote) = remote
            {
                self.ban_malformed(remote.1, format!("reject {reject}"))
                    .await;
            }
            return Err(reject);
        }
        Ok(())
    }

    pub(crate) async fn resumeble_process_tx(

        tx: TransactionView,

        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, remote).await?;

        if self.orphan_contains(&tx).await {
            debug!("reject tx {} already in orphan pool", tx.hash());
            return Err(Reject::Duplicated(tx.hash()));

            return Err(Reject::Duplicated(tx.hash()));
        }
        self.enqueue_verify_queue(tx, is_proposal_tx, remote).await
    }
