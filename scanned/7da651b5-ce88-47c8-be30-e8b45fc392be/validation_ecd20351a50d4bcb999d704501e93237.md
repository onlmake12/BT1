### Title
Mismatch Between `max_tx_verify_cycles` and `max_block_cycles` Allows Local RPC Callers to Bypass Per-Transaction Cycle Limit — (`tx-pool/src/process.rs`)

---

### Summary

CKB has two separate cycle-limit parameters governing transaction verification: `TxPoolConfig.max_tx_verify_cycles` (a node-local, operator-configurable per-transaction cap) and `Consensus.max_block_cycles` (the consensus-level total-block cap). For transactions submitted via the local RPC (`send_transaction`), the verification cycle limit used is `max_block_cycles`, not `max_tx_verify_cycles`. This means a local RPC caller can force the node to spend up to 50× more CPU cycles verifying a single transaction than the operator intended, bypassing the DoS-mitigation purpose of `max_tx_verify_cycles`.

---

### Finding Description

`max_tx_verify_cycles` was introduced specifically to "reduce DDOS vulnerability" by capping how many cycles a single transaction can consume during pool admission. [1](#0-0) 

Its default value is `TWO_IN_TWO_OUT_CYCLES * 20 = 70,000,000`. [2](#0-1) 

The consensus-level `max_block_cycles` is `TWO_IN_TWO_OUT_CYCLES * 1000 = 3,500,000,000` — 50× larger. [3](#0-2) 

In `_process_tx`, the cycle limit passed to the script verifier is chosen as:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
``` [4](#0-3) 

`declared_cycles` is `Some(...)` only for **relayed** transactions (peer-originated). For **locally submitted** transactions (via RPC `send_transaction` or `notify_transaction`), `declared_cycles` is `None`, so `max_block_cycles` is used as the verification limit — not `max_tx_verify_cycles`.

The `max_tx_verify_cycles` config value is only consulted for **orphan routing** (deciding whether to use the async verify queue): [5](#0-4) 

And as a queue classification threshold in `VerifyQueue`: [6](#0-5) 

Neither of these is a hard rejection limit for local submissions. The actual script verifier is called with `max_block_cycles` for local txs: [7](#0-6) 

There is no programmatic enforcement that `max_tx_verify_cycles` and the limit used for local-tx verification are consistent — a direct analog to the OVM mismatch between `maxTransactionGasLimit` and the Execution Manager's gas limit.

---

### Impact Explanation

A local RPC caller (in scope per the bounty rules as "supported local CLI/RPC user") can craft a transaction whose lock/type script consumes close to `max_block_cycles` (3.5 billion) cycles. The node will spend up to 50× more CPU time verifying this single transaction than the operator intended via `max_tx_verify_cycles`. This can:

- Stall the tx-pool worker thread for the duration of the verification
- Delay block assembly and other pending transaction processing
- Cause the node to be unresponsive to other RPC calls during verification

The operator's explicit configuration of `max_tx_verify_cycles` to a lower value (e.g., the default 70M, or a custom lower value) is silently ignored for local submissions.

---

### Likelihood Explanation

The RPC endpoint is bound to `127.0.0.1:8114` by default, so exploitation requires local access. However, any process running on the same host with RPC access — including scripts, daemons, or compromised local software — qualifies. Crafting a high-cycle transaction requires only a valid script binary (e.g., a tight loop in RISC-V), which is straightforward for any script author. The attack is repeatable and requires no special privileges beyond RPC access.

---

### Recommendation

In `_process_tx`, when `declared_cycles` is `None` (local submission), use `min(max_block_cycles, max_tx_verify_cycles)` as the verification limit, or unconditionally cap `max_cycles` at `tx_pool_config.max_tx_verify_cycles`. This establishes a single source of truth for the per-transaction cycle cap regardless of submission path:

```rust
let max_cycles = declared_cycles
    .unwrap_or_else(|| self.consensus.max_block_cycles())
    .min(self.tx_pool_config.max_tx_verify_cycles);
```

Alternatively, document explicitly that `max_tx_verify_cycles` does not apply to local RPC submissions and add a separate config field for local submissions if different behavior is desired.

---

### Proof of Concept

1. Configure a CKB node with `max_tx_verify_cycles = 70_000_000` (the default).
2. Deploy a lock script that is a tight RISC-V loop consuming ~3,000,000,000 cycles before returning success.
3. Submit a transaction spending a cell locked by that script via `send_transaction` RPC (no `declared_cycles` field).
4. Observe that the node's tx-pool worker spends ~3 billion cycles (≈ 43× the configured limit) verifying the transaction, blocking other pool operations for the duration.

The root cause is at: [4](#0-3) 

where `self.tx_pool_config.max_tx_verify_cycles` is never consulted for local submissions, while the consensus `max_block_cycles` is used instead.
