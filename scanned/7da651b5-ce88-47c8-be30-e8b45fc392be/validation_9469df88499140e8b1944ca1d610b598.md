### Title
Shared `total_tx_size` Counter in `VerifyQueue` Allows Remote Peers to Block Local Transaction Submission — (File: `tx-pool/src/component/verify_queue.rs`)

### Summary
The `VerifyQueue` uses a single shared `total_tx_size` counter to track all transactions in the verification queue, regardless of whether they originate from remote peers or local sources. The hardcoded `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256MB` limit is enforced against this shared counter. An unprivileged remote peer can fill the verify queue to the 256MB limit, causing locally-submitted transactions (via `notify_tx`) to be rejected with `Reject::Full` — silently dropped with no error returned to the caller.

### Finding Description

In `VerifyQueue`, a single `total_tx_size: usize` field accumulates the serialized size of every transaction added to the queue, regardless of source: [1](#0-0) [2](#0-1) 

The `is_full()` check compares this shared counter against the hardcoded 256MB ceiling: [3](#0-2) 

The `add_tx()` method enforces this limit for every caller — both remote peer submissions and local proposal transactions: [4](#0-3) 

Two distinct code paths share this counter:

1. **Remote peer path**: `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue` → `add_tx` [5](#0-4) 

2. **Local path**: `notify_tx` → `resumeble_process_tx_and_notify_full_reject` → `enqueue_verify_queue` → `add_tx` [6](#0-5) 

Critically, when `notify_tx` fails with `Reject::Full`, the result is silently discarded at the service layer: [7](#0-6) 

This mirrors the Pepper bug exactly: `totalSupply()` (analogous to `total_tx_size`) includes tokens from `claim()` (analogous to remote peer transactions), causing the `MINTER_ROLE` cap (analogous to the 256MB limit) to block legitimate `mint()` calls (analogous to `notify_tx`).

### Impact Explanation

A remote peer that submits enough valid transactions to fill the 256MB verify queue will cause all subsequent `notify_tx` calls to fail silently. The local node's ability to enqueue locally-originated or proposal transactions into the verify pipeline is completely blocked for as long as the attacker sustains the flood. Because the failure is swallowed (`let _ret = service.notify_tx(tx).await`), the node operator receives no indication that transactions are being dropped. The verify queue workers drain the queue over time, but a sustained attacker can continuously refill it.

### Likelihood Explanation

The attacker needs only a single P2P connection and enough UTXOs to generate valid transactions. A typical serialized transaction is ~200–600 bytes; filling 256MB requires roughly 400,000–1,280,000 valid transactions. This is achievable by a well-funded attacker who pre-generates a large UTXO set. No privileged access, key compromise, or majority hashpower is required — only an unprivileged peer connection, which is the normal P2P entry point.

### Recommendation

Introduce a separate size tracker for remote-peer-originated transactions, analogous to the Pepper fix's `minterRoleMintedAmount`. Apply the 256MB cap only to the remote-peer sub-total, while allowing locally-originated transactions (`remote == None`) to bypass or use a separate, higher (or unlimited) budget:

```rust
// In VerifyQueue, add:
remote_tx_size: usize,

// In is_full(), check only remote_tx_size for remote txs:
pub fn is_full(&self, add_tx_size: usize, is_remote: bool) -> bool {
    if is_remote {
        add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.remote_tx_size
    } else {
        false // local txs are never blocked by the remote cap
    }
}
```

This ensures that remote peer flooding cannot starve locally-submitted transactions, while still protecting the node from remote DoS.

### Proof of Concept

1. Establish a P2P connection to the target node.
2. Pre-generate ~500,000 valid transactions spending distinct UTXOs.
3. Submit all transactions via the P2P relay protocol (`submit_remote_tx` path), filling `VerifyQueue::total_tx_size` to ≥ 256MB.
4. Observe that any subsequent call to `notify_tx` (e.g., triggered by a local `send_transaction` RPC) returns `Err(Reject::Full(...))` and is silently dropped.
5. Confirm via `tx_pool_info` RPC that `verify_queue_size` is at its maximum and locally-submitted transactions do not appear in the pool. [8](#0-7) [9](#0-8)

```rust
    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
```
