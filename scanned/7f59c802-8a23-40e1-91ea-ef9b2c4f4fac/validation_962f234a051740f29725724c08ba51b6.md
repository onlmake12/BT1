Audit Report

## Title
Incomplete Hash-Type Validation in `ScriptHashTypeVerifier` Checks Only Lock Script, Skips Type Script — (File: `verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` field of each output's **lock script** against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the **type script** `hash_type` field. Because `check_data` (which validates only that a hash_type byte is a valid enum variant, not that it is enabled) is absent from the tx-pool admission path, a transaction with an output carrying a not-yet-activated type script hash type (e.g., `Data3 = 6`) passes all non-contextual checks, enters the tx pool, and forces the contextual verification pipeline to process it before rejection.

## Finding Description

`ScriptHashTypeVerifier::verify()` at lines 796–814 exclusively reads `output.lock().hash_type()` in every branch: [1](#0-0) 

The `output.type_()` field — an `Option<Script>` with its own `hash_type` — is never examined. `ENABLED_SCRIPT_HASH_TYPE` contains only `{0, 1, 2, 4}` (Data, Type, Data1, Data2): [2](#0-1) 

Values such as `6` (Data3), `8` (Data4), etc. are valid `ScriptHashType` enum variants (generated by the `seq!` macro) and pass `verify_value` (which only checks `v.is_multiple_of(2) || v == 1`): [3](#0-2) 

`check_data` on `CellOutputReader` does check both lock and type scripts: [4](#0-3) 

However, `check_data` only validates that the hash_type byte is a valid enum variant — not that it is within the enabled range. Furthermore, `check_data` is confirmed absent from the tx-pool admission path (no matches in `tx-pool/**/*.rs`). The tx-pool calls only `NonContextualTransactionVerifier::verify()` via `non_contextual_verify`: [5](#0-4) 

which in turn calls `ScriptHashTypeVerifier::verify()`: [6](#0-5) 

A transaction with a type script whose `hash_type = 0x06` (Data3) therefore passes all non-contextual checks and proceeds to `verify_rtx`, which invokes `ContextualTransactionVerifier`. The contextual verifier attempts to resolve and execute the type script group, failing at the script resolution/hash-type dispatch stage with `ScriptError::InvalidScriptHashType`, classified under `ErrorKind::Script`: [7](#0-6) 

## Impact Explanation

The non-contextual verifier is designed as a cheap fast-path rejection gate. Bypassing it forces the node to perform contextual verification work (cell resolution, script group construction, script code resolution) for every malicious transaction before rejection. An attacker submitting a stream of such transactions from multiple IPs (before bans accumulate) can impose sustained unnecessary load on the node's verification pipeline. This matches the allowed impact: **High — bad designs which could cause CKB network congestion with few costs**. Note: the actual cost per transaction is less than full VM execution (the script verifier fails at hash-type dispatch before running the VM), but the bypass of the intended cheap-rejection gate is real and repeatable at scale.

## Likelihood Explanation

Exploitation requires only the ability to call `send_transaction` via RPC or the P2P relay path — no keys, privileges, or hash power are needed. Crafting a transaction with a type script whose `hash_type` byte is `0x06` is trivial. The attack is repeatable from many IP addresses before bans accumulate, and each malicious transaction imposes contextual verification overhead that the non-contextual gate was designed to prevent.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also inspect the type script hash type for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        Self::check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            Self::check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

Also correct the `ScriptHashTypeVerifier` struct comment (line 785) and the `NonContextualTransactionVerifier` doc comment (line 70) to accurately reflect that both lock and type script hash types are validated. [8](#0-7) [9](#0-8) 

## Proof of Concept

Submit via RPC `send_transaction`:

```json
{
  "outputs": [{
    "capacity": "0x...",
    "lock": { "code_hash": "0x...", "hash_type": "type", "args": "0x" },
    "type": {
      "code_hash": "0xdeadbeef...",
      "hash_type": "0x06",
      "args": "0x"
    }
  }],
  "outputs_data": ["0x"],
  ...
}
```

`hash_type = 0x06` (Data3) passes `check_data` (`verify_value(6)` → `6 % 2 == 0` → `true`) and passes `ScriptHashTypeVerifier` (only lock script is checked). The transaction enters the tx pool and proceeds to contextual verification, where it fails with `ScriptError::InvalidScriptHashType`. Repeating this from multiple IPs before bans accumulate demonstrates the resource-consumption bypass.

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

    }
