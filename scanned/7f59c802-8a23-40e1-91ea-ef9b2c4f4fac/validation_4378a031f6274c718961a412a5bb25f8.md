Looking at the full transaction processing pipeline to verify whether a post-verification fee rate check exists.

Audit Report

## Title
Minimum Fee Admission Check Uses Serialized Size Only, Bypassing Cycle-Derived Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` enforces `min_fee_rate` using only the transaction's serialized byte size, not the full weight `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Because cycles are unknown before script execution, the check is intentionally described as a "cheap" pre-filter, but no weight-corrected fee check is ever performed after `verify_rtx` returns the actual cycle count. An unprivileged submitter can craft a transaction with minimal serialized size and near-maximum cycle consumption, pay only the size-derived minimum fee, and be admitted to the pool at an effective fee rate orders of magnitude below the configured floor.

## Finding Description
The transaction admission pipeline in `_process_tx` (`tx-pool/src/process.rs`) proceeds as follows:

1. `pre_check` calls `check_tx_fee`, which computes `min_fee = min_fee_rate.fee(tx_size)` and rejects if `fee < min_fee`. The code comment explicitly acknowledges the limitation: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check."*
2. `verify_rtx` executes scripts and returns `Completed { cycles, fee }` with the actual cycle count.
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed with the real cycles.
4. `submit_entry` is called — **no weight-based fee rate check is performed at any point after step 1**.

The grep across all of `tx-pool/src/` finds zero occurrences of a post-verification `min_fee_rate` check against the full weight. `TxEntry::fee_rate()` does use `get_transaction_weight(self.size, self.cycles)`, but this is only consulted for eviction ordering and pool scoring, never for admission rejection.

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_tx_verify_cycles = 70_000_000`:

| Parameter | Value |
|---|---|
| `tx_size` | 100 bytes |
| `cycles` | 70,000,000 |
| True weight | `max(100, 11,940)` = **11,940** |
| Fee paid (size-only gate) | `1000 × 100 / 1000` = **100 shannons** |
| Effective fee rate | `100 × 1000 / 11,940` ≈ **8.4 shannons/KW** |

The transaction is admitted at ~119× below the configured 1000 shannons/KW minimum.

The same size-only pattern appears in `calculate_min_replace_fee` for RBF: `extra_rbf_fee = min_rbf_rate.fee(size)`, meaning the RBF extra-fee requirement is also undercharged for cycle-heavy replacing transactions.

## Impact Explanation
This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

An attacker can flood the tx-pool (up to `max_tx_pool_size = 180 MB`) with cycle-heavy, size-tiny transactions at ~1/119th the intended economic cost per unit of block resource. While the eviction mechanism correctly uses weight-based fee rate to evict low-rate entries first, the attacker can continuously resubmit to maintain pool pressure, displacing legitimate transactions and degrading miner revenue. The fee estimators (`confirmation_fraction.rs` and `weight_units_flow.rs`) both call `get_transaction_weight` with actual cycles when ingesting pool entries, so they will observe and propagate the artificially low effective fee rates to wallets.

## Likelihood Explanation
The attack requires only the ability to call `send_transaction` via RPC or P2P relay — both are fully unprivileged. Constructing a transaction with small serialized footprint (one input, one output, minimal witness) and a lock script that runs a tight CKB-VM loop consuming ~70M cycles is straightforward. No special privileges, leaked keys, or victim mistakes are required. The attack is repeatable and cheap.

## Recommendation
After `verify_rtx` returns the `Completed` entry containing the actual cycle count, recompute the minimum fee using the full weight and reject if the fee is insufficient:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

Apply the same fix to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, replacing `min_rbf_rate.fee(size)` with `min_rbf_rate.fee(get_transaction_weight(size, entry.cycles))`.

## Proof of Concept
1. Construct a CKB transaction: one input spending a live cell, one output (minimal capacity), a lock script executing a tight CKB-VM loop consuming ~70,000,000 cycles. Serialized size ≈ 100–200 bytes.
2. Set fee = `min_fee_rate × tx_size / 1000` (e.g., 100–200 shannons at default 1000 shannons/KW).
3. Submit via `send_transaction` RPC.
4. Observe acceptance: `check_tx_fee` passes because `fee ≥ min_fee_rate.fee(tx_size)`. No subsequent weight check occurs.
5. Query the pool entry; compute `entry.fee_rate()` = `fee × 1000 / get_transaction_weight(size, 70_000_000)` ≈ 8–17 shannons/KW — far below the 1000 shannons/KW floor.
6. Repeat to fill the pool with sub-minimum-fee-rate transactions, observing displacement of legitimate higher-fee entries via `get_raw_tx_pool`.
