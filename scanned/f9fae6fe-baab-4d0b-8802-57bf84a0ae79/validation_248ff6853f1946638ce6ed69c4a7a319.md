### Title
Surviving ancestors retain permanently inflated `descendants_count`/`descendants_fee` after `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

### Summary
`remove_entry_and_descendants` pre-clears all link entries before calling `remove_entry`. Because `update_ancestors_index_key` relies on the live links map to find surviving ancestors, it sees an empty ancestor set and never calls `sub_descendant_weight` on them. Surviving ancestors permanently carry stale, inflated descendant weights, corrupting their `EvictKey` and making them appear more valuable than they are.

### Finding Description
In `remove_entry_and_descendants` (lines 252–265), the loop at lines 257–259 calls `remove_entry_links` for every entry in `removed_ids` (the target and all its descendants). `remove_entry_links` removes the target from the links map entirely. When the subsequent loop calls `remove_entry` on each id, `remove_entry` calls `update_ancestors_index_key`, which calls `self.links.calc_ancestors(child_id)`. Since the child's link entry was already deleted, `calc_ancestors` returns `{}`. The loop over ancestors is a no-op, so no surviving ancestor (e.g., A in A→B→C when B is removed) ever has `sub_descendant_weight` called on it.

Concrete state corruption for chain A→B→C after `remove_entry_and_descendants(B)`:

| Field on A | Expected | Actual |
|---|---|---|
| `descendants_count` | 1 | 3 |
| `descendants_fee` | fee_A | fee_A + fee_B + fee_C |
| `descendants_size` | size_A | size_A + size_B + size_C |
| `EvictKey.fee_rate` | max(desc_rate_A, rate_A) | max(inflated_desc_rate, rate_A) |

### Impact Explanation
A's `EvictKey.fee_rate` is computed as `max(descendants_feerate, own_feerate)`. With inflated `descendants_fee`/`descendants_size`, `descendants_feerate` is artificially high, so A sorts toward the "never evict" end of `iter_by_evict_key()`. When the pool is full and `next_evict_entry` is called, A is skipped in favor of legitimately higher-fee transactions. An attacker can exploit this to keep low-fee transactions alive in a full pool, causing high-fee transactions to be rejected or evicted — directly harming miner revenue and distorting the fee market.

### Likelihood Explanation
Reachable by any unprivileged transaction submitter:
1. Submit tx A (low fee).
2. Submit tx B spending A's output.
3. Submit tx C spending B's output.
4. Submit tx D double-spending B's input → `resolve_conflict` → `remove_entry_and_descendants(B)` → A's stats are now permanently inflated.

No special privileges, PoW, or operator access required. The attacker pays fees for B and C (which are removed), but A survives with a corrupted eviction key. This can be repeated to fill the pool with low-fee transactions that appear high-fee.

### Recommendation
In `remove_entry_and_descendants`, call `update_ancestors_index_key` for each removed entry **before** clearing its links, or reconstruct the ancestor set from the pre-removal snapshot. One correct approach: collect the ancestor sets for all `removed_ids` before the `remove_entry_links` loop, then apply `sub_descendant_weight` to surviving ancestors after removal. Alternatively, restructure `remove_entry` so that `update_ancestors_index_key` is called with a pre-computed ancestor set passed in, rather than re-deriving it from the (already-mutated) links map.

### Proof of Concept
Invariant test: after `remove_entry_and_descendants(B)` on chain A→B→C, assert:
```
pool.get(A_id).descendants_count == 1  // only A itself
pool.get(A_id).descendants_fee   == fee_A
```
Both assertions fail with the current code: `descendants_count` is 3 and `descendants_fee` is `fee_A + fee_B + fee_C`. [1](#0-0) [4](#0-3)

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score

                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
