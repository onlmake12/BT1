### Title
Unauthenticated Block Template Injection via Miner Listen Endpoint — (`miner/src/client.rs`)

---

### Summary

When the CKB miner is configured with `listen = Some(addr)`, it binds an HTTP server that accepts `BlockTemplate` notifications. The `handle()` function performs **no authentication, no source IP check, and no token validation**. Any attacker who can reach the listen port can POST a crafted `BlockTemplate` JSON body and cause the miner to immediately mine on an attacker-controlled template. A `work_id=0` in the injected template guarantees the update is always accepted due to a logical condition in `update_block_template`.

---

### Finding Description

`listen_block_template_notify` binds a raw TCP listener on the configured `SocketAddr` and dispatches every accepted connection to `handle()`: [1](#0-0) 

The `handle()` function reads the HTTP body and deserializes it directly into a `BlockTemplate` with no authentication or source validation: [2](#0-1) 

On successful deserialization, `update_block_template` is called unconditionally. The only guard inside it is a `work_id` comparison: [3](#0-2) 

The closure condition is `if id != work_id || id == 0`. When an attacker sends `work_id = 0`, this simplifies to `(id != 0) || (id == 0)` — which is **always true**, bypassing the guard entirely and guaranteeing the injected template is dispatched to workers via `new_work_tx`.

The `listen` field is a documented, supported configuration option: [4](#0-3) [5](#0-4) 

The feature is explicitly intended for use when the miner is on a separate machine from the CKB node (a common mining farm topology), meaning the listen address is frequently bound to a non-loopback interface.

---

### Impact Explanation

An attacker who can reach the listen port can redirect 100% of the miner's hashrate to mine attacker-specified blocks. The attacker controls `compact_target`, `transactions_root`, `cellbase`, and all other template fields. Mined blocks are submitted to the CKB node via `submit_block`, which will reject them if they fail consensus validation — but the miner's hashrate is wasted on the attacker's work for the duration of the attack. In a pool or farm context this causes direct economic damage and potential consensus disruption.

---

### Likelihood Explanation

The `listen` feature is opt-in and the default example uses `127.0.0.1:8888`. However:
- Mining operations commonly separate the miner process from the CKB node, requiring a non-loopback bind address.
- The code provides **no authentication mechanism whatsoever** — there is no way for a security-conscious operator to restrict access at the application level; only external firewall rules can protect this endpoint.
- No documentation warns about the security implications of exposing this port.

---

### Recommendation

Add source IP allowlisting (only accept connections from the configured `rpc_url` host) or a shared-secret token that the CKB node includes in its notification requests and `handle()` validates before calling `update_block_template`. At minimum, document that the listen address must never be bound to a publicly reachable interface without firewall protection.

---

### Proof of Concept

```bash
# Miner configured with: listen = "0.0.0.0:8888"
# Attacker sends a crafted BlockTemplate with work_id=0
curl -s -X POST http://<miner-ip>:8888/ \
  -H "Content-Type: application/json" \
  -d '{
    "version": "0x0",
    "compact_target": "0x1a08a97e",
    "current_time": "0x...",
    "number": "0x1",
    "epoch": "0x...",
    "parent_hash": "0x...",
    "cycles_limit": "0x...",
    "bytes_limit": "0x...",
    "uncles_count_limit": "0x2",
    "uncles": [],
    "transactions": [],
    "proposals": [],
    "cellbase": { ... },
    "work_id": "0x0",
    "dao": "0x..."
  }'
# Result: miner immediately dispatches WorkerMessage::NewWork with attacker's pow_hash
# to all worker threads, which begin mining the attacker's block.
```
