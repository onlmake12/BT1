### Title
PoW Verified Before Parent Existence Check Enables Free CPU Exhaustion via Trivially-Mined Headers — (`verification/src/header_verifier.rs`, `sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderVerifier::verify` unconditionally runs the full eaglesong+blake2b PoW computation before checking whether the parent block exists. Because `compact_target` is taken directly from the attacker-controlled header and is never validated against chain difficulty prior to the PoW check, an attacker can mine headers at near-zero cost (using minimum difficulty), send batches of up to 2,000 such headers with unknown parents, and force the victim node to perform 2,000 cryptographic hash operations per message — with no disconnect, no ban, and no rate limit applied to the sync path.

**`EaglesongBlake2bPowEngine::verify`** uses the `compact_target` embedded in the header itself — it does not validate that the target is correct for the current chain epoch:

```rust
let (block_target, overflow) = compact_to_target(header.raw().compact_target().into());
// only checks: output <= block_target (attacker-supplied)
``` [2](#0-1) 

An attacker sets `compact_target = 0x207fffff` (minimum difficulty / maximum target). Mining a valid header at this difficulty is essentially instant on commodity hardware. The epoch/difficulty validation that would catch an invalid `compact_target` lives in `EpochVerifier`, which only runs after the parent lookup — i.e., after the PoW has already been computed.

**`HeaderAcceptor::accept`** call order (`headers_process.rs` lines 295–358):

1. `prev_block_check` — only rejects if parent has `BLOCK_INVALID` status; an unknown parent passes this check.
2. `non_contextual_check` → `HeaderVerifier::verify` → **PoW computed** → parent lookup fails → `UnknownParentError`.
3. `UnknownParentError` maps to `TemporaryInvalid` → `Status::ok()` — **no disconnect, no ban**. [3](#0-2) [4](#0-3) 

**`MAX_HEADERS_LEN = 2,000`** per message: [5](#0-4) 

The `Relayer` has a 30 req/s rate limiter, but it explicitly exempts `CompactBlock` and does not cover the `Synchronizer`'s `SendHeaders` path at all: [6](#0-5) 

There is no per-peer rate limit on `SendHeaders` messages in the synchronizer.

### Impact Explanation

An attacker with a single inbound connection can:
1. Mine 2,000 headers at minimum difficulty (trivially fast, near-zero cost).
2. Send them as a `SendHeaders` P2P message with all parent hashes set to unknown values.
3. Force the victim to run 2,000 × (eaglesong + blake2b) before discovering the parents are unknown.
4. Receive `Status::ok()` (no disconnect, no ban).
5. Repeat indefinitely.

The CPU cost per header is small (~microseconds), but with no rate limiting and no peer penalty, the attacker can sustain this at the maximum rate the network link allows, causing sustained unnecessary CPU load on the victim node.

### Likelihood Explanation

The attack requires only a standard P2P connection (inbound peers are accepted by default) and trivial mining work. No special privileges, no hashpower proportional to mainnet difficulty, and no coordination are needed. The `is_continuous` check only validates internal batch continuity, not that the first header's parent is known. [7](#0-6) 

### Recommendation

Move the parent existence check before the PoW check in `HeaderVerifier::verify`. The parent lookup is an O(1) hash-map read; PoW is a cryptographic computation. Cheap structural checks should always gate expensive cryptographic ones. Additionally, add per-peer rate limiting to the `SendHeaders` handler in the synchronizer, and consider returning a scored penalty (not just `TemporaryInvalid`) when a peer repeatedly sends headers with unknown parents.

### Proof of Concept

1. Spin up a CKB node with a private chain (genesis with `compact_target = 0x207fffff`).
2. Mine 2,000 headers at minimum difficulty; set each header's `parent_hash` to a random unknown 32-byte value.
3. Connect to a mainnet/testnet node and send a `SendHeaders` message containing all 2,000 headers.
4. Observe: the victim node performs 2,000 eaglesong+blake2b calls, then returns `Status::ok()` without disconnecting.
5. Benchmark: repeat in a tight loop and measure victim CPU utilization vs. a patched version that checks parent existence first.
