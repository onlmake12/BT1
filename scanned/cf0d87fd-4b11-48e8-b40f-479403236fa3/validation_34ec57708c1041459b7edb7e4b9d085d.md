### Title
Unnecessary Full Block Fetch in `complete_headers` Enables I/O Amplification via `GetLastStateProof` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
`BlockSampler::complete_headers` calls `snapshot.get_block()` for each sampled block number, which reads the full block body (all transactions) from `COLUMN_BLOCK_BODY`. Only `calc_uncles_hash()` and `extension()` are consumed from the result. The cheaper `get_block_uncles + get_block_extension` pattern used in `get_blocks_proof.rs` would suffice and avoids the body read entirely. Any unprivileged light-client peer can trigger this for up to 998 blocks per `GetLastStateProof` request.

### Finding Description
In `complete_headers` (lines 137–148 of `get_last_state_proof.rs`), for each block number in the `numbers` slice, the code:
1. Calls `snapshot.get_block(&ancestor_header.hash())` — which internally calls `get_block_body(h)`, iterating `COLUMN_BLOCK_BODY` to deserialize all transactions in the block, plus `get_block_uncles`, `get_block_proposal_txs_ids`, and `get_block_extension`.
2. Uses only `ancestor_block.calc_uncles_hash()` and `ancestor_block.extension()` from the result.

The transaction body data is read, deserialized, and allocated entirely for nothing. By contrast, `get_blocks_proof.rs` (lines 82–94) calls `get_block_header`, `get_block_uncles`, and `get_block_extension` separately — all of which are backed by LRU caches in `StoreCache` and do not touch `COLUMN_BLOCK_BODY`.

The limit check at line 201 permits `difficulties.len() + last_n_blocks * 2 ≤ 1000`. With `difficulties.len()=0` and `last_n_blocks=499`, the check passes and `complete_headers` is called with up to 998 block numbers.

### Impact Explanation
Each `get_block()` call reads all transactions for a block from RocksDB's `COLUMN_BLOCK_BODY`. For a block at the CKB block size limit, this can be hundreds of KB to several MB of data per block. With 998 blocks per request, a single peer can force the full node to read and deserialize up to ~998× the per-block transaction data, amplifying I/O and heap allocation compared to the header-only path. The `COLUMN_BLOCK_BODY` data is not LRU-cached at the store layer, so repeated requests bypass any warm-cache benefit. Multiple concurrent peers multiply the effect.

### Likelihood Explanation
The path is reachable by any unprivileged light-client peer with no PoW, no key, and no special role. The crafted message is trivially constructable: set `difficulties` to empty and `last_n_blocks` to 499. The limit check passes. No other guard prevents the full-block fetch loop from executing.

### Recommendation
Replace the `get_block()` call in `complete_headers` with the same pattern used in `get_blocks_proof.rs`:

```rust
// Instead of:
let ancestor_block = self.snapshot.get_block(&ancestor_header.hash())...;
let uncles_hash = ancestor_block.calc_uncles_hash();
let extension = ancestor_block.extension();

// Use:
let uncles = self.snapshot.get_block_uncles(&ancestor_header.hash())
    .ok_or_else(|| format!("failed to find uncles for header#{}", number))?;
let uncles_hash = uncles.calc_uncles_hash();
let extension = self.snapshot.get_block_extension(&ancestor_header.hash());
```

This eliminates the `COLUMN_BLOCK_BODY` read entirely and benefits from the existing `block_uncles` and `block_extensions` LRU caches.

### Proof of Concept
1. Connect a light-client peer to a full node.
2. Send a `GetLastStateProof` message with `difficulties = []`, `last_n_blocks = 499`, and a valid `last_hash` on the main chain with at least 499 blocks after `start_block_number`.
3. The limit check at line 201 passes: `0 + 499*2 = 998 ≤ 1000`.
4. `complete_headers` is called with up to 998 block numbers.
5. For each, `snapshot.get_block()` reads `COLUMN_BLOCK_BODY` from RocksDB.
6. Benchmark: measure RocksDB bytes read from `COLUMN_BLOCK_BODY` and heap allocations for this path vs. the `get_block_uncles + get_block_extension` path on a chain with large blocks. The `get_block()` path will read significantly more bytes and allocate more memory proportional to total transaction data across those 998 blocks.
