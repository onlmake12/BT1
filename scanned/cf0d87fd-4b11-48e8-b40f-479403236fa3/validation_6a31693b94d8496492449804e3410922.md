### Title
Orphan Pool Monopolization via Missing Per-Peer Quota in `add_orphan_tx` — (`tx-pool/src/component/orphan.rs`)

### Summary

The `OrphanPool` enforces a global cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` but has no per-peer slot quota. A single unprivileged relay peer can fill all 100 slots with its own orphan transactions and then continuously refresh them, causing any other peer's orphan entries to be evicted with ~99% probability on each insertion attempt.

### Finding Description

`OrphanPool::add_orphan_tx` inserts any orphan transaction unconditionally (subject only to a duplicate check), then calls `limit_size()` to enforce the global cap. [1](#0-0) 

`limit_size()` evicts by taking the first key from the `HashMap` — effectively random due to hash randomization — with no regard for which peer contributed the entry: [2](#0-1) 

There is no per-peer counter, no per-peer cap, and no fairness mechanism anywhere in `OrphanPool`. [3](#0-2) 

The relay path that feeds orphans into the pool is:

`TransactionsProcess::execute()` → `submit_remote_tx()` → `after_process()` → `add_orphan()` → `add_orphan_tx()` [4](#0-3) 

The `TransactionsProcess` does filter incoming transactions against `unknown_tx_hashes` (only accepting transactions the node previously requested), but this is easily satisfied by the normal relay flow: the attacker advertises orphan tx hashes via `RelayTransactionHashes`, the node requests them, and the attacker delivers them. [5](#0-4) 

The existing rate limiter in `Relayer` is keyed by `(PeerIndex, message_type)` at 30 req/s, which limits message frequency but not the number of transactions per message (up to `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`), so 100 orphans can be delivered in a single `RelayTransactions` message. [6](#0-5) 

### Impact Explanation

Once peer A holds all 100 slots, any new orphan from peer B causes the pool to reach 101 entries, then one random entry is evicted. With 100 of 101 entries belonging to peer A, peer B's entry is evicted with ~99% probability. Peer A can maintain near-100% occupancy by continuously advertising new orphan hashes. The practical effect is that legitimate orphan transactions from all other peers are evicted before their parents can arrive, breaking the orphan resolution mechanism network-wide for the victim node.

### Likelihood Explanation

The attack requires only a standard P2P connection and the ability to craft transactions with missing inputs (trivially done with any valid key pair and a fabricated outpoint). No PoW, no fees, no privileged access. The `ORPHAN_TX_EXPIRE_TIME` of `100 * MAX_BLOCK_INTERVAL` means evicted orphans do not naturally expire quickly, so the attacker does not need to race against expiry — only against other peers inserting new orphans. [7](#0-6) 

### Recommendation

Introduce a per-peer slot quota (e.g., `DEFAULT_MAX_ORPHAN_TRANSACTIONS / max_peers`). When the pool is full, prefer evicting entries from the peer with the most slots before evicting randomly. This is the approach used by Bitcoin Core's orphan pool since 2022.

### Proof of Concept

1. Attacker connects as peer A.
2. Peer A sends `RelayTransactionHashes` with 100 hashes of transactions whose inputs reference non-existent outpoints.
3. Victim node sends `GetRelayTransactions` for all 100 hashes.
4. Peer A responds with `RelayTransactions` containing all 100 orphan transactions in one message.
5. Each transaction fails `pre_check` with a missing-input error, triggering `add_orphan` for each. After 100 insertions, the orphan pool is 100% occupied by peer A's entries.
6. Peer B (honest) relays one orphan. `add_orphan_tx` inserts it (pool = 101), then `limit_size` evicts one random entry. Probability peer B's entry survives: 1/101 ≈ 1%.
7. Peer A repeats step 2–4 with fresh orphan hashes to replace any of its evicted entries, maintaining monopoly at the cost of one round-trip per refresh cycle.

```rust
    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
```

```rust
    pub(crate) async fn add_orphan(
        &self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) {
        let evicted_txs = self
            .orphan
            .write()
            .await
            .add_orphan_tx(tx, peer, declared_cycle);
        // for any evicted orphan tx, we should send reject to relayer
        // so that we mark it as `unknown` in filter
        for tx_hash in evicted_txs {
            self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash });
        }
    }
```
