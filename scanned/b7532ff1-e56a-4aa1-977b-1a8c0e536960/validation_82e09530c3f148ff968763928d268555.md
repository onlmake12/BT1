### Title
Pre-Allocation of Up to 8MB Per Malicious Compressed Frame Before Decompression Validation — (`network/src/compress.rs`)

### Summary

In `LengthDelimitedCodecWithCompress::decode` and `Message::decompress`, the node allocates a buffer of size `decompressed_bytes_len` (up to `MAX_UNCOMPRESSED_LEN = 8MB`) **before** calling `SnapDecoder::decompress`. An unprivileged remote peer can craft a snappy frame with a valid varint length header claiming ~8MB but an invalid body, causing the node to allocate ~8MB and then immediately free it on decompression failure. With `max_connection_number = 1024` simultaneous connections each sending one such frame, the node can be forced to allocate up to ~8GB before any error is detected.

---

### Finding Description

The vulnerable sequence is in `LengthDelimitedCodecWithCompress::decode`: [1](#0-0) 

1. `decompress_len(&data[1..])` reads only the snappy varint prefix from the compressed payload — it does **not** validate the rest of the stream.
2. If `decompressed_bytes_len <= MAX_UNCOMPRESSED_LEN` (8MB), the code immediately allocates: `let mut buf = BytesMut::zeroed(decompressed_bytes_len);`
3. Only then does `SnapDecoder::new().decompress(...)` run — and it can fail on a corrupted body.
4. On failure, `Err(io::ErrorKind::InvalidData)` is returned and the buffer is dropped — but the allocation already happened.

The same pattern exists in `Message::decompress`: [2](#0-1) 

The outer frame size check (`max_frame_length`) does **not** prevent this: the malicious compressed payload is only a few bytes (a 4-byte snappy varint claiming ~8MB + garbage), well within any protocol's `max_frame_length`: [3](#0-2) 

The service-level connection limit is 1024: [4](#0-3) 

`MAX_UNCOMPRESSED_LEN` is 8MB: [5](#0-4) 

---

### Impact Explanation

- Each malicious frame: ~10 bytes sent on the wire → ~8MB allocated on the heap.
- With 1024 simultaneous connections each sending one frame: up to **8GB** of heap allocation before any decompression error is detected.
- This can cause OOM, node crash, or severe memory pressure degrading sync/relay performance — matching the "network congestion" scope.
- A single attacker with 1024 TCP connections (no authentication required) can trigger this from any IP.

---

### Likelihood Explanation

- The P2P port is open to all peers by default.
- No PoW, no stake, no authentication is required to open a connection and send a compressed frame.
- The snappy varint format is trivially craftable: encode `8388607` (just under 8MB) as a 4-byte LEB128 varint, append garbage bytes, set the compress flag byte.
- The attack is repeatable: after disconnect, the attacker reconnects and repeats.
- A single machine with 1024 TCP connections can sustain the attack continuously.

---

### Recommendation

Validate the snappy stream integrity **before** allocating the output buffer. Options:
1. Use `snap::read::FrameDecoder` (streaming) instead of pre-allocating based on `decompress_len`.
2. Impose a tighter pre-allocation cap (e.g., refuse to allocate more than the compressed input size × a safe expansion ratio, e.g., 4×).
3. Validate that `decompressed_bytes_len` is consistent with the compressed input size before allocating (snappy's compression ratio is bounded; a 10-byte input cannot legitimately decompress to 8MB).

---

### Proof of Concept

```python
import socket, struct

# Snappy varint encoding of 8388607 (just under 8MB = 1<<23)
# 8388607 = 0x7FFFFF, varint: 0xFF 0xFF 0xFF 0x03
snappy_varint = b'\xff\xff\xff\x03'
garbage_body  = b'\xde\xad\xbe\xef' * 4  # invalid snappy body

# compress flag byte (0x80) + snappy payload
payload = b'\x80' + snappy_varint + garbage_body

# 4-byte big-endian length prefix (LengthDelimitedCodec uses 4-byte header by default)
frame = struct.pack('>I', len(payload)) + payload

# Connect to CKB P2P port and send after handshake (Sync or RelayV3 protocol)
# Each connection triggers BytesMut::zeroed(~8MB) before Err(InvalidData)
s = socket.create_connection(('target-node', 8115))
s.sendall(frame)
# Node allocates ~8MB, then disconnects — repeat with 1024 connections for ~8GB
```

Each connection causes `BytesMut::zeroed(decompressed_bytes_len)` at: [6](#0-5) 

before the error path is reached, violating the invariant that failed message parsing must not cause large memory allocations prior to validation.
