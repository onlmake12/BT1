### 1. Attacker Entry Point

A remote peer sends a `DiscoveryMessage::Nodes` message. The handler calls `add_new_addrs`, which calls `peer_store.add_addr()` for each address: [1](#0-0) 

### 2. `add_addr` always sets `last_connected_at_ms = 0`

The question claims the attacker sets `last_connected_at_ms` to a recent timestamp. This is incorrect — `add_addr()` hardcodes it to `0`: [2](#0-1) 

However, a fresh address with `last_connected_at_ms = 0` and `attempts_count = 0` still passes `is_connectable`: [3](#0-2) 

The three conditions that return `false` all require either `attempts_count >= 3` (never connected) or `attempts_count >= 10` (connected but stale). A fresh address with `attempts_count = 0` returns `true` regardless of `last_connected_at_ms`.

### 3. The Real Bug: `take(len / 2)` Integer Division

When the store reaches `ADDR_COUNT_LIMIT = 16384` and phase-1 eviction finds no non-connectable addresses, phase 2 runs: [4](#0-3) 

The critical line is:

```rust
let len = peers_by_network_group.len();
// ...
peers.into_iter().take(len / 2)
```

If all 16384 addresses belong to **exactly 1** `/16` network group (e.g., all `1.2.x.x`):
- `len = 1`
- `len / 2 = 0` (integer division)
- `take(0)` → empty iterator → `candidate_peers` is empty
- Returns `Err(PeerStoreError::EvictionFailed)`

The `/16` group is computed as: [5](#0-4) 

### 4. Feasibility of Filling the Store

- `MAX_ADDR_TO_SEND = 1000` addresses per non-announce Nodes message
- A single session can send 1000 addresses; 17 sessions fill 16384 slots
- Each session can also send announce messages (up to `ANNOUNCE_THRESHOLD = 10` items per 60s interval)
- The `is_valid_addr` check requires globally reachable IPs, but any real public `/16` block (e.g., `1.2.0.0/16`) provides 65536 IPs × many ports — far more than 16384 unique addresses [6](#0-5) 

### 5. Correction to the Question's Claim

The question says "≤4 distinct /16 groups" triggers the bug. This is wrong:
- With 2 groups: `take(2/2) = take(1)` → 1 group processed → eviction succeeds
- With 4 groups: `take(4/2) = take(2)` → 2 groups processed → eviction succeeds
- **Only `len == 1` (exactly 1 group) triggers `take(0)` → no eviction → `EvictionFailed`**

### 6. Impact Qualification

The impact is **not permanent**. Once the store is full with fake addresses:
- The victim tries to connect to them via `fetch_addrs_to_feeler`
- Failed attempts call `mark_tried`, incrementing `attempts_count`
- After 3 failures with `last_connected_at_ms == 0`, `is_connectable` returns `false`
- Phase-1 eviction then removes them, freeing space

The attacker must **continuously re-fill** the store to sustain the block. This is a sustained resource-exhaustion attack, not a one-shot permanent lock.

The error is silently swallowed in the discovery path (logged at `debug` level only): [7](#0-6)

## Verdict

### Title
Peer Store Phase-2 Eviction Permanently Fails When All Addresses Share One /16 Network Group Due to `take(len / 2)` Integer Division — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
`check_purge()` contains an integer division bug: when all stored addresses belong to exactly one `/16` network group, `take(len / 2)` = `take(0)` processes zero groups, produces no eviction candidates, and returns `Err(PeerStoreError::EvictionFailed)` on every subsequent `add_addr()` call.

### Finding Description
In `check_purge()`, phase-2 eviction groups addresses by `/16` network segment, then calls `.take(len / 2)` on the sorted groups. When `len == 1`, integer division yields `0`, so no groups are iterated, `candidate_peers` is empty, and the function returns `EvictionFailed`. An attacker who fills the 16384-slot store with addresses all in one `/16` block (e.g., `1.2.0.0/16`) triggers this path on every subsequent `add_addr()` call. [8](#0-7) 

### Impact Explanation
While the store is locked, the victim node cannot add newly discovered peer addresses from either the discovery protocol or the identify protocol. The attack must be sustained: as the victim attempts connections to fake addresses and marks them non-connectable, phase-1 eviction eventually frees slots. The attacker must continuously re-inject addresses to maintain the block. During the attack window, the victim's peer discovery is severely degraded.

### Likelihood Explanation
Requires ~17 inbound sessions (each sending 1000 addresses via a single non-announce Nodes message), all advertising addresses in the same `/16` block. This is well within the default inbound peer limit and requires no special privileges — any peer that can establish a P2P connection can execute this.

### Recommendation
Replace `take(len / 2)` with `take((len + 1) / 2)` or `take(len.max(1))` to ensure at least one group is processed when `len >= 1`. Additionally, consider capping the number of addresses accepted per session and per source `/16` group before they reach the peer store.

        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr

            return false;

        // consider addr is not connectable if failed too many times

        }
        true
    }
