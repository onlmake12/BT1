### Title
Stale Descendant Metrics on Ancestor Entries After `remove_entry_and_descendants` Due to Premature Link Removal - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link relationships for every entry in the batch before calling `remove_entry` on each one. Because `remove_entry` relies on `self.links.calc_ancestors(id)` to find which remaining pool entries need their descendant-weight metrics decremented, and those links are already gone by the time the call happens, ancestors of the removed chain that **remain in the pool** are never updated. They permanently retain inflated `descendants_count`, `descendants_size`, `descendants_cycles`, `descendants_fee`, and a stale `evict_key`. This is the CKB analog of the `removeToken` class of bug: a removal operation that does not properly unwind all dependent accounting state.

---

### Finding Description

`remove_entry_and_descendants` collects the root entry and all its descendants, then calls `remove_entry_links` for every one of them before calling `remove_entry`: [1](#0-0) 

`remove_entry_links` removes the entry from `self.links.inner` entirely (via `self.links.remove(id)`): [2](#0-1) 

When `remove_entry` is subsequently called for the root entry, it invokes `update_ancestors_index_key(&entry.inner, EntryOp::Remove)`: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())` to find which remaining pool entries to update: [4](#0-3) 

`calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)` on the links map: [5](#0-4) 

Because `remove_entry_links` already called `self.links.remove(id)` for the root, `self.inner.get(short_id)` returns `None`, `direct` is empty, and `calc_ancestors` returns an empty set. The loop in `update_ancestors_index_key` never executes. Ancestors of the root that are still in the pool never receive a call to `sub_descendant_weight`: [6](#0-5) 

Those ancestors' `descendants_count`, `descendants_size`, `descendants_cycles`, `descendants_fee`, and `evict_key` fields remain permanently inflated.

The `evict_key` is computed from these stale descendant fields: [7](#0-6) 

---

### Impact Explanation

The corrupted `evict_key` on surviving ancestors directly affects two critical pool operations:

1. **Eviction ordering** — `next_evict_entry` selects the entry with the lowest `evict_key` to drop when the pool exceeds `max_tx_pool_size`: [8](#0-7) 
   Ancestors with inflated descendant fee-rate appear more valuable than they are, so they are skipped during eviction. Lower-fee-rate transactions that should survive are evicted instead.

2. **Fee rate estimation** — `estimate_fee_rate` iterates entries by score. While the `score` key (based on ancestors) is not directly corrupted, the pool's overall accounting is inconsistent, degrading the reliability of fee estimates returned to users and miners.

3. **Ancestor-count enforcement** — `check_and_record_ancestors` enforces `max_ancestors_count`. Stale descendant counts on surviving ancestors do not directly affect this check, but the overall pool state is inconsistent with reality.

---

### Likelihood Explanation

The bug is triggered by every code path that calls `remove_entry_and_descendants` when the removed root has at least one ancestor still in the pool:

- **RBF (Replace-By-Fee)**: Any unprivileged transaction sender can submit a higher-fee transaction that conflicts with an existing pool transaction. `process_rbf` calls `remove_entry_and_descendants` for each conflicted entry: [9](#0-8) 

- **Block commitment / conflict resolution**: When a block is committed, `remove_committed_tx` calls `resolve_conflict`, which calls `remove_entry_and_descendants` for any pool transaction that spent the same input: [10](#0-9) 

- **Pool size limit enforcement**: `limit_size` calls `remove_entry_and_descendants` when the pool exceeds its size limit: [11](#0-10) 

- **`remove_tx` RPC**: A local RPC caller can directly trigger the path via `TxPool::remove_tx` → `remove_entry_and_descendants`: [12](#0-11) 

Any of these paths is reachable without privilege. The RBF path is reachable by any unprivileged peer or transaction submitter.

---

### Recommendation

In `remove_entry_and_descendants`, update the ancestors' descendant weights **before** removing links. Specifically, before the loop that calls `remove_entry_links` for all entries, call `update_ancestors_index_key` for the root entry `id` (and each descendant, in topological order) so that ancestors remaining in the pool have their `descendants_*` fields and `evict_key` correctly decremented. Alternatively, restructure `remove_entry` to accept a flag indicating whether link-based ancestor lookup should be skipped, and perform the ancestor update separately before link teardown.

---

### Proof of Concept

1. Submit Tx A to the pool (pending). A's `descendants_count = 1`.
2. Submit Tx B (child of A) to the pool. A's `descendants_count = 2`, `descendants_fee = fee_A + fee_B`.
3. Submit Tx C (child of B) to the pool. A's `descendants_count = 3`, `descendants_fee = fee_A + fee_B + fee_C`.
4. Submit Tx D that spends the same input as B, with a higher fee (RBF). `process_rbf` calls `remove_entry_and_descendants(B)`, removing B and C.
5. After removal: A's `descendants_count` is still `3`, `descendants_fee` still includes `fee_B + fee_C`, and A's `evict_key` is inflated.
6. When the pool reaches `max_tx_pool_size`, `next_evict_entry` iterates by `evict_key`. A appears to have a higher descendant fee-rate than it actually does, so it is skipped. A lower-fee-rate transaction that should survive is evicted instead.
7. The stale state persists for the lifetime of A in the pool, since there is no mechanism to recompute descendant metrics after the fact.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {

                self.links.remove_child(&parent, id);

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
