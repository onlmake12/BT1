### Title
Silent u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Accounting — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity using a u128 intermediate value, then casts it to u64 with a bare `as u64` (a silent truncating cast). Every other analogous u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When the u128 result exceeds `u64::MAX`, the truncated value is silently accepted as correct, causing downstream DAO state accounting (`withdrawed_interests`) to underflow and reject the containing block.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
// lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The formula is the direct analog of the reported borrow-rate formula: it scales `counted_capacity` by the ratio `withdrawing_ar / deposit_ar`. When that ratio is large enough that the u128 product exceeds `u64::MAX`, the `as u64` cast silently wraps to the lower 64 bits — a small, incorrect value — instead of returning `DaoError::Overflow`.

Every other u128→u64 narrowing in the same `impl` block uses the checked form:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The truncated value then propagates into `withdrawed_interests`:

```rust
// lines 316-332
let maximum_withdraws = rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
    self.transaction_maximum_withdraw(rtx)   // returns truncated small value
        .and_then(|c| capacities.safe_add(c).map_err(Into::into))
})?;
...
maximum_withdraws
    .safe_sub(input_capacities)   // underflows: truncated < actual deposited amount
    .map_err(Into::into)
``` [5](#0-4) 

`withdrawed_interests` returns an error, which propagates through `dao_field_with_current_epoch` and causes the block to be rejected. [6](#0-5) 

The existing overflow test (`check_withdraw_calculation_overflows`) only exercises the `safe_add` overflow path (where `withdraw_counted_capacity as u64` + `occupied_capacity` > `u64::MAX`), not the silent truncation path where `withdraw_counted_capacity as u128` > `u64::MAX` but the truncated value + `occupied_capacity` fits in u64. [7](#0-6) 

---

### Impact Explanation

When triggered, the `as u64` truncation produces a `withdraw_capacity` that is far smaller than the actual accrued interest. The `withdrawed_interests` subtraction then underflows (`maximum_withdraws < input_capacities`), causing `dao_field_with_current_epoch` to return an error. Any block containing such a DAO withdrawal is permanently rejected by all nodes running this code, even though the on-chain DAO script (running in CKB-VM with its own arithmetic) may accept the transaction. This creates a **consensus split**: the block is valid per the script layer but invalid per the node's DAO accounting layer. The affected depositor's funds become permanently unwithdrawable — a loss-of-funds outcome.

Additionally, the public RPC `calculate_dao_maximum_withdraw` calls `calculate_maximum_withdraw` directly and would silently return a grossly incorrect (truncated) capacity value to the caller instead of a `CKBInternalError` / `DaoError`, misleading wallet software. [8](#0-7) 

---

### Likelihood Explanation

For `withdraw_counted_capacity` (u128) to exceed `u64::MAX ≈ 1.844 × 10¹⁹`:

```
counted_capacity × withdrawing_ar / deposit_ar > 1.844 × 10¹⁹
```

With `counted_capacity` bounded by the total CKB supply (≈ 3.36 × 10¹⁸ shannons), the AR ratio must exceed ≈ 5.49×. Given the secondary issuance rate (`DEFAULT_SECONDARY_EPOCH_REWARD = 613_698_63013698` shannons/epoch) and the initial AR (`10¹⁶`), this requires approximately 42 years of continuous operation on mainnet. [9](#0-8) 

Likelihood on mainnet is therefore **very low in the near term**. However, the defect is real, is inconsistent with every other u128→u64 narrowing in the same file, and is exploitable on any chain with a higher secondary issuance rate or lower initial AR (e.g., a custom `dev.toml` chain spec). [10](#0-9) 

---

### Recommendation

Replace the silent `as u64` cast with the same checked conversion used everywhere else in the file:

```rust
// Before (line 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
``` [11](#0-10) 

Add a unit test that sets `withdrawing_ar` to a value large enough that `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX` but `+ occupied_capacity` would not overflow, verifying that `DaoError::Overflow` is returned rather than a silently truncated `Ok` value.

---

### Proof of Concept

Construct a scenario with:
- `counted_capacity = 3_360_000_000_000_000_000` (≈ 3.36 × 10¹⁸ shannons, near total CKB supply)
- `deposit_ar = 10_000_000_000_000_000` (10¹⁶, genesis AR)
- `withdrawing_ar = 60_000_000_000_000_000` (6 × 10¹⁶, AR after ~42 years)

```
withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 60_000_000_000_000_000
    / 10_000_000_000_000_000
  = 20_160_000_000_000_000_000   ← exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 20_160_000_000_000_000_000 mod 2^64
  = 1_713_255_926_290_448_384   ← small, incorrect value; safe_add succeeds
```

`withdrawed_interests` then computes:
```
maximum_withdraws = 1_713_255_926_290_448_384 + occupied_capacity  (small)
input_capacities  = 3_360_000_000_000_000_000 + occupied_capacity  (large)
safe_sub → underflow → DaoError
```

Block containing this withdrawal is rejected. The depositor cannot recover their CKB. [1](#0-0) [5](#0-4)
