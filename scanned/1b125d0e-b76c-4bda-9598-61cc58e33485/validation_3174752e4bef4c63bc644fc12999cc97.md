### Title
Unguarded `.expect()` on Peer-Influenced Index in `BlockTransactionsVerifier` - (`File: sync/src/relayer/block_transactions_verifier.rs`)

### Summary

`BlockTransactionsVerifier::verify` accesses `block_short_ids` using entries from `expected_transaction_indexes` with `.expect("should never outbound")` rather than a bounds-checked return. While the indexes are locally computed (not directly peer-supplied), they are derived from a peer-supplied compact block. The analog to the Bus report's missing `numPassengers` validation is the absence of an explicit upper-bound check on each index before the `.get()` call, relying entirely on an implicit invariant that is asserted rather than enforced.

### Finding Description

In `sync/src/relayer/block_transactions_verifier.rs`, `BlockTransactionsVerifier::verify` receives an `indexes: &[u32]` slice representing the positions of missing transactions within the compact block's transaction list: [1](#0-0) 

Each entry in `indexes` is used to index into `block_short_ids` (a `Vec<Option<ProposalShortId>>` of length `txs_len = prefilled_transactions.len() + short_ids.len()`). The code calls `.expect("should never outbound")` instead of returning a graceful error if any index is out of range.

The `indexes` originate from `expected_transaction_indexes` stored in `pending_compact_blocks`: [2](#0-1) 

These are computed from the peer-supplied compact block during `reconstruct_block`. The compact block's structure (specifically `prefilled_transactions` and `short_ids`) is peer-controlled and governs `txs_len`. The `PrefilledVerifier` validates only that the **last** prefilled index is `< txs_len` and that indices are strictly increasing: [3](#0-2) 

It does **not** independently validate that `txs_len` itself is within a safe range, nor does it guard against the `usize` addition `prefilled_transactions.len() + short_ids.len()` wrapping in release builds.
