### Title
Incorrect `row_lock` read in `script_exists_in_output` on PostgreSQL causes type_script deletion during rollback, corrupting rich-indexer DB — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary
In `script_exists_in_output`, line 252 reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, this causes the function to return the lock_script EXISTS result for the type_script check, leading to incorrect deletion of type_scripts that are still referenced by surviving outputs after a rollback.

### Finding Description
The function `script_exists_in_output` performs two SQL EXISTS queries: one checking `lock_script_id` (result in `row_lock`) and one checking `type_script_id` (result in `row_type`). The final match at line 252 mistakenly re-reads `row_lock` instead of `row_type`. On PostgreSQL, `try_get::<bool, _>(0)` always succeeds on `row_lock`, so the correct `row_type` path is never taken. The function returns the lock_script existence result for the type_script branch. [3](#0-2) 

### Impact Explanation
Any type_script whose `id` does not appear as a `lock_script_id` in any surviving output will be incorrectly deleted from the `script` table during rollback on PostgreSQL. Surviving outputs that reference this `type_script_id` now point to a deleted row. Subsequent `get_cells` or `get_transactions` RPC calls that JOIN on the `script` table will encounter DB errors, causing RPC crashes or error responses. The indexer DB is silently corrupted and requires manual repair or full re-sync.

### Likelihood Explanation
Reorgs are a normal part of CKB network operation and do not require any attacker. Any PostgreSQL-backed rich-indexer node that processes a reorg involving outputs with type_scripts not shared as lock_scripts will trigger this bug. This is a common real-world scenario (most type_scripts are distinct from lock_scripts).

### Recommendation
Change line 252 from:
```rust

```
to:
```rust
match row_type.try_get::<bool, _>(0) {
``` [1](#0-0) 

### Proof of Concept
1. Start a rich-indexer node with PostgreSQL backend.
2. Append a block containing an output with a `type_script` whose `code_hash` differs from all `lock_scripts` in the DB.
3. Trigger a rollback (reorg) of that block.
4. Call `get_cells` or `get_transactions` with a search key referencing any script that was in the rolled-back block.
5. Observe DB error due to missing `script` row.

Differential test: run the same sequence against SQLite — no error occurs, confirming the PG-specific divergence caused by the `row_lock`/`row_type` mix-up.
