Audit Report

## Title
Sustained O(n) CPU Scan Per `add_addr` Call When Peer Store Is Saturated With Connectable Diverse-Subnet Entries — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
When `AddrManager` holds exactly `ADDR_COUNT_LIMIT` (16 384) entries that are all connectable and spread across distinct `/16` network groups (no group exceeding 4 peers), every call to `add_addr` triggers two full O(n) scans plus an O(k log k) sort inside `check_purge`, returns `Err(EvictionFailed)`, and leaves the count unchanged at 16 384. Because the new address is never inserted, the store remains saturated and every subsequent call repeats the same work. This is reachable from any unprivileged peer via the discovery `Nodes` message, which can carry up to 3 000 `add_addr` calls per message with errors silently swallowed.

## Finding Description

**Entry point — `DiscoveryAddressManager::add_new_addrs`:**
`add_new_addrs` iterates over every address in a received `Nodes` message and calls `peer_store.add_addr()` for each one. Errors are swallowed at `debug!` level; the loop continues unconditionally. A single non-announce `Nodes` message may carry up to `MAX_ADDR_TO_SEND = 1000` nodes × `MAX_ADDRS = 3` addresses = **3 000 `add_addr` calls per message**.

**`add_addr` always calls `check_purge` before inserting:**
```rust
pub fn add_addr(&mut self, addr: Multiaddr, flags: Flags) -> Result<()> {
    if self.ban_list.is_addr_banned(&addr) { return Ok(()); }
    self.check_purge()?;   // <-- always called
    ...
    self.addr_manager.add(AddrInfo::new(addr, 0, score, flags.bits()));
    Ok(())
}
```

**`check_purge` when the store is full:**

- **Phase 1** — full `addrs_iter()` scan collecting non-connectable entries (O(n)). A freshly added `AddrInfo` has `last_connected_at_ms = 0` and `attempts_count = 0`; `is_connectable` returns `true` because `0 < ADDR_MAX_RETRIES (3)`. So Phase 1 finds nothing.

- **Phase 2** (entered only when Phase 1 finds nothing) — second full `addrs_iter()` scan, `HashMap<Group, Vec<_>>` construction, and `sort_unstable_by_key` (O(n) + O(k log k)). The eviction logic only removes peers from groups with `> 4` members. If every `/16` subnet has ≤ 4 peers (e.g., 16 384 entries each in a unique subnet), `candidate_peers` is empty.

- **`Err(EvictionFailed)` is returned**, propagated through `check_purge()?`, and `addr_manager.add()` is never reached. The count stays at 16 384.

Every subsequent `add_addr` call re-enters the same two-phase scan. There is no caching of the eviction-failed state, no early exit, and no rate-limiting of the caller.

## Impact Explanation
This is a **sustained CPU denial-of-service** against a CKB node reachable by any unprivileged peer. With n = 16 384 and k up to 16 384 distinct groups, each `add_addr` call performs two full table scans plus a sort. A single `Nodes` message triggers ~3 000 such calls (~100 M operations). The network-service thread — which handles all P2P protocol logic — is monopolised processing peer-store work, degrading or effectively denying the node's ability to participate in the network. An attacker sustaining this with a stream of `Nodes` messages can cause **CKB network congestion with few costs**, matching the High impact category (10 001–15 000 points).

## Likelihood Explanation
The precondition is reachable by an unprivileged attacker:
1. **Fill the store**: send 16 384 unique addresses spread across ≥ 4 097 distinct `/16` subnets (e.g., spoofed IPv4 or IPv6) so no group exceeds 4. Each entry is connectable by default (`attempts_count = 0`).
2. **Sustain the load**: keep sending `Nodes` messages with new addresses. Each message triggers 3 000 × O(n) calls. The error is logged at `debug!` only; no disconnect or rate-limit is applied to the sender.

The only practical mitigations are the node's inbound connection limit and the OS TCP stack, neither of which limits the rate of `Nodes` messages on an established session.

## Recommendation
1. **Cache the eviction-failed state**: after `check_purge` returns `Err(EvictionFailed)`, set a flag (e.g., `store_full_no_eviction: bool`) and return `Err` immediately on subsequent calls without re-scanning, until an entry is removed or expires.
2. **Skip Phase 2 when Phase 1 finds nothing**: if Phase 1 finds no non-connectable candidates, return `Err` immediately instead of performing the expensive group-scan and sort.
3. **Rate-limit `add_addr` calls per session**: track how many addresses a session has contributed and throttle or drop excess advertisements.

## Proof of Concept
```rust
// Fill AddrManager to ADDR_COUNT_LIMIT with connectable, diverse-subnet entries
let mut peer_store = PeerStore::default();
for i in 0u32..16384 {
    // Each address in a unique /16 subnet; attempts_count=0 → connectable
    let addr: Multiaddr = format!("/ip4/{}.{}.0.1/tcp/8115", i >> 8, i & 0xff)
        .parse().unwrap();
    peer_store.add_outbound_addr(addr, Flags::COMPATIBILITY);
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Measure CPU time per add_addr call when store is saturated
let new_addr: Multiaddr = "/ip4/200.200.200.1/tcp/8115".parse().unwrap();
let t0 = std::time::Instant::now();
for _ in 0..1000 {
    let _ = peer_store.add_addr(new_addr.clone(), Flags::COMPATIBILITY);
    // Each call: Phase1 O(n) scan + Phase2 O(n)+O(k log k) + Err; count stays 16384
}
println!("1000 saturated add_addr calls: {:?}", t0.elapsed());
// Expected: orders of magnitude slower than 1000 calls on a non-full store
```

```rust
    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;

        // consider addr is not connectable if failed too many times

            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
```
