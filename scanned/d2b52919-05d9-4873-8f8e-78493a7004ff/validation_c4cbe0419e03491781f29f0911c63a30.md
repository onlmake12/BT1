Audit Report

## Title
DAO Withdrawal Header-Dep Index Read as u64 Instead of u8, Diverging from On-Chain C Script — (`util/dao/src/lib.rs`)

## Summary
`DaoCalculator::transaction_maximum_withdraw` reads the witness `input_type` field as a full **u64** (line 91) to index into `header_deps`, while the on-chain C DAO script reads only the **lowest byte**. When a witness index exceeds 255, the two implementations resolve different `header_dep` entries, causing them to use different deposit headers and `ar` values. The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` (lines 475–537 of `tests.rs`) directly confirms the divergence: Rust rejects a transaction the C VM would accept.

## Finding Description
In `util/dao/src/lib.rs` line 91, the deposit-header index is parsed as:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and used at line 96:

```rust
rtx.transaction.header_deps().get(header_dep_index as usize)
```

The on-chain C DAO script (referenced in the test comments) reads only the lowest byte of the same 8-byte field. For a witness `input_type` value of 257 (LE bytes: `0x01 0x01 0x00 …`):
- **C VM** reads byte 0 → index **1**
- **Rust** reads full u64 → index **257**

The test at `util/dao/src/tests.rs` lines 489–495 explicitly constructs this scenario: `header_deps[1]` = deposit block (number 100), `header_deps[257]` = withdraw block (number 200), witness index = 257. Rust resolves index 257 → withdraw block (number 200), which fails the block-number guard at line 105 (`deposit_header.number() != deposited_block_number`), returning `DaoError::InvalidOutPoint`. The C VM resolves index 1 → deposit block (number 100), which passes. The test asserts `result.is_err()`, confirming the divergence.

For the silent `ar`-divergence variant: if `header_deps[257]` is a fork block at the same height as the deposit block but with a different `ar` value, the block-number check passes in both implementations, but `calculate_maximum_withdraw` (line 108) uses a different `deposit_ar` in each. This produces a different `withdrawed_interests` (line 222 via `withdrawed_interests()`), a different `current_s` in the packed DAO field, and a consensus split when other nodes recompute the DAO field independently.

## Impact Explanation
Two concrete impacts:

1. **Incorrect implementation of system scripts (High)**: Rust rejects DAO withdrawal transactions that the on-chain C DAO script accepts, breaking tx-pool admission and block verification for any DAO withdrawal with a witness index > 255.

2. **Consensus deviation (Critical)**: In the silent-divergence variant, a block producer includes a crafted DAO withdrawal where Rust and the C VM agree on acceptance but compute different `deposit_ar` values. The resulting DAO field (`current_s`) diverges between the block producer and verifying nodes, causing a chain split.

## Likelihood Explanation
No privileged role is required. Any transaction sender can submit a DAO withdrawal via RPC or P2P relay. Preconditions: (a) the transaction carries ≥ 258 `header_deps` entries (258 × 32 bytes ≈ 8.25 KB, within CKB block-size limits; no `MAX_HEADER_DEPS` limit was found in the codebase); (b) for the consensus-split variant, two blocks at the same height with different `ar` values must exist in the node's database, which is normal during any fork or reorg. The attack is repeatable and requires no social engineering or majority hashpower.

## Recommendation
Replace the full-u64 read with a single-byte read to match the C DAO script's behavior:

```rust
// Before (line 91):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))

// After — match the C VM's lowest-byte semantics:
Ok(u64::from(header_deps_index_data.unwrap()[0]))
```

Alternatively, explicitly reject any witness index whose upper 7 bytes are non-zero, ensuring Rust and the C VM are guaranteed to agree on the resolved `header_dep` entry.

## Proof of Concept
The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` (lines 475–537) is a direct PoC for the false-rejection case. It passes today, confirming the divergence.

For the consensus-split variant: modify the test so that `header_deps[257]` is a fork block at height 100 (same as the deposit block) but with a different `ar` value in its DAO field. Both the block-number check (line 105) and the `header_deps` membership check (line 97) pass in both implementations. `calculate_maximum_withdraw` then uses a different `deposit_ar` in Rust vs. the C VM, producing a different `withdrawed_interests` and a different packed DAO field — demonstrating the consensus split.
