Audit Report

## Title
`check_tx_fee` Admits Below-Minimum-Weight-Fee-Rate Transactions via Size-Only Gate — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` computes the minimum required fee using only the serialized transaction size, while the canonical fee-rate metric used for block selection, eviction, and fee estimation is `fee / weight` where `weight = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction with a small serialized size but near-maximum cycle consumption passes the size-based gate with a fee that is orders of magnitude below the true minimum weight-based fee rate. No second weight-based check is performed after `verify_rtx` returns the actual cycle count. An unprivileged attacker can exploit this to flood the mempool with transactions that are never mined, causing sustained pool churn and delaying legitimate transactions.

## Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` (lines 42–52) explicitly uses `tx_size` as the denominator for the minimum fee calculation, with a comment acknowledging this is intentionally a "cheap check":

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The canonical weight function used everywhere else is `get_transaction_weight` in `util/types/src/core/tx_pool.rs` (lines 298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

`TxEntry::fee_rate()` uses this weight-based calculation for all post-admission decisions: [3](#0-2) 

In `_process_tx` (lines 705–777 of `tx-pool/src/process.rs`), the flow is:
1. `pre_check` → `check_tx_fee` (size-based gate, cycles unknown)
2. `verify_rtx` → returns actual `verified.cycles`
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry constructed with actual cycles
4. `submit_entry` — no second fee-rate check against true weight [4](#0-3) 

The gap: after step 2, the actual weight is known but never checked against `min_fee_rate`.

## Impact Explanation

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_tx_verify_cycles = 70_000_000` (default per `resource/ckb.toml` line 215), a 200-byte transaction consuming 70 M cycles has:
- Size-based min fee (checked): `1000 × 200 / 1000 = 200 shannons` → passes
- Actual weight: `max(200, 11940) = 11940`
- Actual fee rate: `200 / 11940 × 1000 ≈ 16.7 shannons/KW` — ~60× below `min_fee_rate` [5](#0-4) 

Such transactions are never selected for block templates (miners rank by weight-based fee rate). The eviction mechanism in `limit_size` (`tx-pool/src/pool.rs` lines 290–329) uses `entry.fee_rate()` (weight-based), so attacker transactions are evicted first when the pool is full — legitimate transactions are not permanently displaced. However, the attacker can continuously re-submit evicted transactions (given sufficient UTXOs), maintaining sustained pool pressure: the pool remains perpetually at capacity, every legitimate transaction admission requires an eviction cycle, and fee estimation is polluted by the low-fee-rate entries. [6](#0-5) 

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. The per-transaction cost is just above `min_fee_rate × size` (≈200 shannons ≈ 0.000002 CKB), making sustained attacks economically viable.

## Likelihood Explanation

Any unprivileged user with access to `send_transaction` RPC or P2P relay can trigger this. Writing a CKB-VM script that loops to consume ~70 M cycles is straightforward. The attacker pays only the size-based minimum fee per transaction. The default pool size of 180 MB with 200-byte transactions allows ~900,000 attack transactions at a total cost of ~1.8 CKB. The 12-hour expiry window (`expiry_hours = 12`) means each UTXO can be used once per 12-hour window, but with enough UTXOs the attack is repeatable and sustainable. [7](#0-6) 

## Recommendation

After `verify_rtx` returns the actual cycle count and before `TxEntry::new` is constructed, perform a second fee-rate check using the true weight:

```rust
// In _process_tx, after: let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_fee_rate = FeeRate::calculate(fee, actual_weight);
if actual_fee_rate < tx_pool_config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, ...)), snapshot));
}
```

The existing size-based check in `check_tx_fee` can remain as a cheap early-exit for obviously under-fee transactions, but must not be the sole admission gate. [8](#0-7) 

## Proof of Concept

1. Deploy a CKB lock script that loops to consume exactly `max_tx_verify_cycles - 1` cycles (≈70 M cycles).
2. Create a UTXO locked by this script.
3. Build a transaction spending that UTXO. Serialized size ≈ 200 bytes.
4. Set output capacity so `fee = ceil(min_fee_rate × tx_size / 1000) = 200 shannons`.
5. Submit via `send_transaction` RPC.
6. Observe: transaction accepted into mempool (`check_tx_fee` passes: `fee(200) ≥ min_fee(200)`).
7. Observe: `entry.fee_rate()` ≈ 16.7 shannons/KW — far below `min_fee_rate = 1000`.
8. Observe: transaction never appears in block templates.
9. Repeat with additional UTXOs to fill the pool; observe sustained pool churn as the node continuously evicts and re-admits attacker transactions. [1](#0-0) [3](#0-2)

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
