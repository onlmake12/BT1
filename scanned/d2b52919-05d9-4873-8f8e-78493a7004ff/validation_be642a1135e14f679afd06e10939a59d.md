Audit Report

## Title
Missing Deduplication of `tx_hashes` Enables 2×N Redundant DB Reads via Light-Client Protocol — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` accepts up to 1000 `tx_hashes` from any unprivileged light-client peer but performs no deduplication before processing. Submitting N identical hashes for a committed on-chain transaction causes N calls to `get_transaction_info` (Stage 1) and N calls to `get_transaction_with_info` (Stage 2), totalling 2×N redundant RocksDB point-reads and N duplicate `(tx, index)` entries accumulated into `txs_in_blocks`, all from a single P2P message.

## Finding Description

The only input guard is a length check at lines 37–39:

```rust
if self.message.tx_hashes().len() > constant::GET_TRANSACTIONS_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many transactions");
}
```

No deduplication follows. The partition at lines 54–64 iterates the raw (possibly duplicate-filled) iterator and calls `snapshot.get_transaction_info(tx_hash)` once per element:

```rust
let (found, missing): (Vec<_>, Vec<_>) = self
    .message
    .tx_hashes()
    .to_entity()
    .into_iter()
    .partition(|tx_hash| {
        snapshot.get_transaction_info(tx_hash)  // called N times for N identical hashes
            ...
    });
```

The `found` loop at lines 67–75 then calls `snapshot.get_transaction_with_info` once per element in `found` (again without deduplication) and pushes N identical `(tx, tx_info.index)` tuples into `txs_in_blocks[block_hash]`:

```rust
for tx_hash in found {
    let (tx, tx_info) = snapshot.get_transaction_with_info(&tx_hash).expect("tx exists");
    txs_in_blocks.entry(tx_info.block_hash).or_insert_with(Vec::new).push((tx, tx_info.index));
}
```

Those N duplicate entries are then passed to `CBMT::build_merkle_proof` at lines 86–97 with N copies of the same index. The `.expect("build proof with verified inputs should be OK")` at line 97 would panic if the underlying `merkle_cbt` library returns `None` for duplicate indices (behavior depends on library version; this is the worst-case path). Even without a panic, the server serializes and transmits a `FilteredBlock` containing N duplicate transaction entries. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

## Impact Explanation

A single unprivileged peer can force the server to execute up to 2000 RocksDB point-reads (2 × 1000) for a single unique transaction, allocate and process a 1000-entry duplicate `Vec`, and serialize an oversized response — all from one P2P message. Multiple concurrent connections multiply the amplification linearly. This matches the **Low (501–2000): Any other important performance improvements for CKB** impact class. The panic path (if `merkle_cbt::build_merkle_proof` returns `None` for duplicate indices) would additionally crash the handler task, but this is not confirmed without inspecting the exact library version.

## Likelihood Explanation

The attack requires no privilege, no key material, and no proof-of-work. Any peer that can open a light-client protocol connection can send a `GetTransactionsProof` message. The message type is part of the standard `LightClientMessage` union and is reachable through the normal P2P receive path. The attacker only needs one valid committed `tx_hash` (publicly observable on-chain) and can repeat the attack continuously.

## Recommendation

Deduplicate `tx_hashes` immediately after the length check, before any DB access:

```rust
// After line 39:
let mut seen = std::collections::HashSet::new();
let tx_hashes: Vec<_> = self
    .message
    .tx_hashes()
    .to_entity()
    .into_iter()
    .filter(|h| seen.insert(h.clone()))
    .collect();
```

Then use this deduplicated `tx_hashes` vec for both the partition and the found loop. This reduces DB lookups to at most N unique hashes and eliminates duplicate entries in `txs_in_blocks`.
