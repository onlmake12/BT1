Audit Report

## Title
`total_tx_size` / `total_tx_cycles` Inflated by Stale Snapshot Overwrite After Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, the new pool-size totals are captured into local variables before `check_and_record_ancestors` runs. When that function evicts transactions via `remove_entry_and_descendants`, the in-place decrements to `self.total_tx_size` / `self.total_tx_cycles` are immediately overwritten by the stale pre-eviction snapshot. The counters are permanently inflated by the sum of all evicted transactions' sizes and cycles, and the inflation compounds with each subsequent eviction-triggering submission.

## Finding Description

In `add_entry` (L200–221), the sequence is:

```rust
// L210-211: snapshot taken BEFORE evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// L213: may call remove_entry_and_descendants → update_stat_for_remove_tx
evicts = self.check_and_record_ancestors(&mut entry)?;

// ... insert entry ...

// L218-219: stale snapshot overwrites correctly-decremented self fields
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (L711–729) computes `self.total_tx_size + entry.size` and returns it as a local. `check_and_record_ancestors` (L588–639) enters the eviction branch at L603 when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`, and calls `remove_entry_and_descendants` (L618) for each evicted entry. Each removal calls `update_stat_for_remove_tx` (L733–758), which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in-place (L738–740). The final assignments at L218–219 then overwrite those correct values with the stale pre-eviction snapshot, producing:

```
final total_tx_size = original_total + entry.size
correct value       = original_total − evicted_sizes + entry.size
inflation           = evicted_sizes
```

No existing guard prevents this: `updated_stat_for_add_tx` only checks for integer overflow, not for the ordering problem, and `update_stat_for_remove_tx` has no way to prevent its result from being overwritten.

## Impact Explanation

The inflated `total_tx_size` directly corrupts two critical control paths:

1. **`limit_size`** (pool.rs L292–328) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts legitimate, already-accepted transactions. An inflated counter causes honest transactions to be unnecessarily expelled from the mempool.
2. **`updated_stat_for_add_tx`** (L716–721) rejects new submissions with `Reject::Full` when the inflated counter would overflow. Valid transactions are refused even when the pool has real capacity.

Both effects are permanent until the node restarts, and the inflation compounds with each repeated attack submission. An attacker can systematically drain the mempool of honest transactions and block new honest submissions, constituting **network congestion with low cost** — matching the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

The eviction path is reachable by any unprivileged user via the `send_transaction` RPC. Required conditions:

- Submit a chain of N+1 transactions (N > `max_ancestors_count`, default 25) all referencing a shared cell dep `D`.
- Submit one more transaction spending the chain tip and referencing `D`, so `ancestors_count − cell_ref_parents.len() ≤ 25`, triggering the eviction branch.

No key material, privileged access, or majority hashpower is required. The attack is repeatable: each qualifying submission inflates the counter by the size of the evicted transactions. The attacker controls the eviction magnitude by choosing transaction sizes.

## Recommendation

Move `updated_stat_for_add_tx` to **after** `check_and_record_ancestors` completes, so the totals are computed from the already-updated (post-eviction) `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute AFTER evictions have already decremented self.total_tx_size
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

If the pre-flight overflow check must remain early, it should be a pure guard that does not capture a snapshot, and the final assignment should always be derived from the post-eviction state.

## Proof of Concept

1. Start a node with an empty pool (`total_tx_size = 0`, `max_ancestors_count = 25`).
2. Submit 26 transactions tx₁ → tx₂ → … → tx₂₆, each 200 bytes, all referencing cell dep `D`. Pool: 26 entries, `total_tx_size = 5200`.
3. Submit tx₂₇ spending tx₂₆ and referencing `D`. `ancestors_count = 27`; `cell_ref_parents = {tx₁…tx₂₆}`; `27 − 26 = 1 ≤ 25` → eviction branch triggers.
4. `updated_stat_for_add_tx` captures `total_tx_size_local = 5200 + 200 = 5400`.
5. `check_and_record_ancestors` evicts tx₁ (200 bytes): `self.total_tx_size` decremented to `5000`.
6. tx₂₇ inserted. L218 executes: `self.total_tx_size = 5400` (overwrites `5000`).
7. Pool holds 26 entries (tx₂…tx₂₇) = 5200 bytes actual; `total_tx_size = 5400` — inflated by 200 bytes.
8. Repeat steps 2–7 to accumulate inflation until `limit_size` begins evicting honest transactions or `updated_stat_for_add_tx` rejects valid submissions with `Reject::Full`.
