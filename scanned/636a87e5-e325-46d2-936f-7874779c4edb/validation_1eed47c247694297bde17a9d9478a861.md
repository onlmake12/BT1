Audit Report

## Title
Missing `request_limit` Guard in `get_cells_capacity` Allows Unbounded Aggregate Scan — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`)

## Summary

`AsyncRichIndexerHandle::get_cells_capacity` builds and executes an unbounded `SUM(output.capacity)` aggregate query with no row limit and no `request_limit` enforcement, while the sibling methods `get_cells` and `get_transactions` both enforce `self.request_limit` and emit a SQL `LIMIT` clause. When called with `script_search_mode=Partial` and empty `args` on SQLite, the condition `instr(args, '') > 0` evaluates to true for every row, causing a full-table scan aggregate with no early termination. An unprivileged RPC caller can exploit this to stall the SQLite connection pool and block all other indexer RPC responses.

## Finding Description

**Missing `request_limit` guard in `get_cells_capacity`:**

`get_cells` enforces the limit at lines 26–34 and appends `LIMIT` to the SQL at line 156: [1](#0-0) [2](#0-1) 

`get_transactions` does the same at lines 24–32: [3](#0-2) 

`get_cells_capacity` has no such check and no `LIMIT` clause. The entire function body builds and executes an unbounded aggregate: [4](#0-3) 

**`instr(args, '') > 0` is always true on SQLite:**

`build_query_script_id_sql` (used exclusively by `get_cells_capacity`) generates this condition for SQLite + Partial mode: [5](#0-4) 

When the caller passes empty `args` (`0x`), the bound parameter is `b""`. SQLite's `instr(X, '')` returns `1` for every row regardless of content, so the script subquery returns all `script` rows matching the given `code_hash`/`hash_type`. The outer query then aggregates `SUM(output.capacity)` over every live output joined to those scripts with no row limit.

**The `request_limit` field exists on the handle but is never consulted in `get_cells_capacity`:** [6](#0-5) 

## Impact Explanation

An attacker can make the indexer RPC unresponsive by repeatedly calling `get_cells_capacity` with a high-cardinality lock script and empty `args` in Partial mode. Each call forces a full sequential scan and CPU-bound aggregation with no early termination. Because SQLite serializes writes and limits concurrent readers, concurrent calls compound the effect and starve the connection pool, blocking all other indexer RPC responses. This matches the allowed impact: **Note (0–500 points) — Any local RPC API crash/hang**.

## Likelihood Explanation

The RPC requires no authentication. The attack payload is a single valid JSON-RPC call. The `code_hash` of the secp256k1 default lock is public knowledge and covers the majority of mainnet outputs. The `Partial` + empty `args` combination passes all input validation. The asymmetry between `get_cells_capacity` and its siblings is a straightforward oversight.

## Recommendation

Add the same `request_limit` guard that `get_cells` and `get_transactions` use. Since `get_cells_capacity` is an aggregate with no caller-supplied `limit` parameter, apply the guard at the SQL level — e.g., add a `LIMIT self.request_limit` to the script subquery inside `build_query_script_id_sql`, or add a `LIMIT` to the outer output query. Additionally, consider rejecting requests where `script_search_mode=Partial` is specified with empty `args`, or treating empty `args` as an error for that mode.

    "params": [{
      "script": {
        "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
        "hash_type": "type",
        "args": "0x"
      },
      "script_type": "lock",
      "script_search_mode": "partial"
    }],
    "id": 1
  }'

# SUM scans every live output with that code_hash/hash_type, no LIMIT.
# Repeat in a tight loop; compare wall-clock time to get_cells with the same
# key (which returns after request_limit rows). get_cells_capacity will be
# orders of magnitude slower and will not terminate early regardless of table size.
