Audit Report

## Title
`PoolMap::add_entry` Overwrites Eviction-Decremented `total_tx_size`/`total_tx_cycles` With Pre-Eviction Values, Inflating Pool Size Accounting - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` snapshots `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When that function evicts cell-ref ancestors, `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` in-place. However, `add_entry` then unconditionally overwrites `self.total_tx_size` with the stale pre-eviction local, permanently inflating the counter by the aggregate size of all evicted transactions. Any unprivileged caller who triggers the cell-ref eviction path can silently reduce the pool's effective capacity, causing legitimate transactions to be rejected or unnecessarily evicted.

## Finding Description
The exact code path in `tx-pool/src/component/pool_map.rs`:

```
L210-211: let (total_tx_size, total_tx_cycles) =
              self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
          // Captures: self.total_tx_size + entry.size  (pre-eviction snapshot)

L213:     evicts = self.check_and_record_ancestors(&mut entry)?;
          // Internally calls remove_entry_and_descendants → update_stat_for_remove_tx
          // which does: self.total_tx_size -= evicted_size  (in-place mutation)

L218-219: self.total_tx_size  = total_tx_size;   // OVERWRITES the decremented value
          self.total_tx_cycles = total_tx_cycles; // OVERWRITES the decremented value
```

`updated_stat_for_add_tx` (L716) reads `self.total_tx_size` and returns a new value as a local — it does not modify `self`. `update_stat_for_remove_tx` (L738-740) directly assigns to `self.total_tx_size`. Because the local snapshot was taken before evictions, the final write at L218 restores the pre-eviction value plus `entry.size`, erasing the decrements. The inflation per triggering submission equals the total serialized size of all evicted transactions.

The eviction path at L603-625 is reached whenever `ancestors_count > max_ancestors_count` but the excess is attributable to cell-ref parents. `remove_entry_and_descendants` is called in a loop (L618), each call invoking `update_stat_for_remove_tx`, all of whose effects are discarded by L218-219.

The inflated counter is then used directly in `limit_size` (pool.rs L298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, causing legitimate transactions to be evicted against a falsely elevated baseline.

## Impact Explanation
The inflated `total_tx_size` persists across all subsequent operations. `limit_size` will evict additional legitimate transactions to bring the falsely elevated counter below `max_tx_pool_size`, and `updated_stat_for_add_tx` will reject new incoming transactions with `Reject::Full` even though real pool occupancy is well below the configured limit. An attacker who repeats the triggering pattern can inflate the counter without bound, eventually making the pool reject all new transactions. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — the pool's effective capacity is silently destroyed, degrading transaction throughput for all users of the node, and the attack is cheap and repeatable.

## Likelihood Explanation
No special privilege is required. The attacker needs only the ability to submit transactions via the standard `send_transaction` RPC. The cell-ref eviction path is explicitly supported and integration-tested (`TxPoolLimitAncestorCount`). The attack is repeatable: each triggering submission inflates the counter further. The cost is proportional to the number of transactions submitted, which is bounded by normal transaction fees.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` returns, so the snapshot reflects the post-eviction baseline:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;  // evictions happen first
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Now snapshot reflects post-eviction self.total_tx_size
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

## Proof of Concept
1. Configure a node with default `max_ancestors_count = 1000` and `max_tx_pool_size = 2_000_000`.
2. Submit `tx_root` creating an output cell.
3. Submit 1,001 transactions each referencing `tx_root`'s output as a `cell_dep` (~200 bytes each). Pool `total_tx_size` ≈ 200,200.
4. Submit `tx_spend` spending `tx_root`'s output. `check_and_record_ancestors` evicts 2 cell-ref parents (~400 bytes). `update_stat_for_remove_tx` sets `self.total_tx_size = 199,800`. Then L218 writes back `200,200 + tx_spend.size`, inflating by 400 bytes.
5. Repeat steps 3–4. Each iteration inflates `total_tx_size` by the evicted bytes.
