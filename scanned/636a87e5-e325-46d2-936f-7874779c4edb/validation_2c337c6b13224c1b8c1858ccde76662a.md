Audit Report

## Title
Incomplete Panic Recovery in `ConsumeUnverifiedBlocks::start()` Leaves `unverified_tip` Unreset and Block Status Uncleaned — (File: `chain/src/verify.rs`)

## Summary
When `consume_unverified_blocks` panics, the `catch_unwind` recovery block in `ConsumeUnverifiedBlocks::start()` only removes the block hash from `is_pending_verify` but omits the two critical cleanup steps performed by the normal error path: resetting `unverified_tip` to the actual verified chain tip and marking the block as `BLOCK_INVALID`. This leaves `unverified_tip` frozen at the panicked block's number and the block's status stuck at `BLOCK_RECEIVED`, which together can permanently stall block synchronization.

## Finding Description

**Panic handler (incomplete cleanup):** [1](#0-0) 

The handler only calls `self.processor.is_pending_verify.remove(&block_hash)`. It does not call `set_unverified_tip` or `insert_block_status(..., BLOCK_INVALID)`.

**Normal error path (complete cleanup):** [2](#0-1) 

The `Err` branch resets `unverified_tip` to the actual verified tip and marks the block `BLOCK_INVALID` (unless it is an internal DB error).

**`unverified_tip` is advanced when a block is dispatched to the preload channel:** [3](#0-2) 

**`block_fetcher` uses `unverified_tip` as a hard gate:** [4](#0-3) 

If `get_unverified_tip().number() >= tip_number + BLOCK_DOWNLOAD_WINDOW * 9`, `fetch()` returns `None` unconditionally.

**Exploit chain:**
1. Block B at height N is dispatched to the preload channel → `unverified_tip = N`.
2. `consume_unverified_blocks` panics while processing B.
3. `catch_unwind` fires: only `is_pending_verify.remove(&B.hash())` executes. `unverified_tip` stays at N; B's status stays `BLOCK_RECEIVED`.
4. Subsequent queued blocks (B+1, B+2, …) attempt verification. Their parent (B) has status `BLOCK_RECEIVED` (not `BLOCK_STORED`, not `BLOCK_INVALID`), and if B's `BlockExt` was never committed, `get_block_ext(&B.hash())` returns `None`, causing each to return an `Err`. Each `Err` path resets `unverified_tip` to the verified tip M — **but only if there are subsequent blocks in the queue**. If B is the last dispatched block, no subsequent block resets `unverified_tip`, and it remains at N.
5. Even after `unverified_tip` is eventually reset, B's status is `BLOCK_RECEIVED`. `new_block_received` checks `BlockStatus::HEADER_VALID.eq(&status)` before accepting a re-delivered block: [5](#0-4) 
   Since B's status is `BLOCK_RECEIVED` ≠ `HEADER_VALID`, B cannot be re-accepted from any peer. The chain cannot advance past height N−1.

## Impact Explanation
The node's chain tip is permanently frozen at the last verified height before the panic. No further blocks can be verified because the panicked block occupies the chain slot with an unresolvable `BLOCK_RECEIVED` status, and its descendants cannot be verified without a valid parent. This constitutes a **persistent sync stall** for the affected node, matching the allowed impact: **High — Vulnerabilities which could easily crash a CKB node** (the node's chain advancement is equivalent to a crash from a sync perspective, requiring a restart to recover).

## Likelihood Explanation
The developers explicitly anticipated panics by wrapping `consume_unverified_blocks` in `catch_unwind` — this is production defensive code, not a theoretical construct. Panics in Rust arise from `unwrap()`/`expect()` on `None`, integer overflow in debug builds, out-of-bounds slice access, or explicit `panic!()` calls anywhere in the verification pipeline (script execution, capacity accounting, MMR operations, fork resolution). For an attacker to deliberately trigger this, they must submit a block that passes non-contextual checks including proof-of-work, which requires significant computational resources — this is not a zero-cost attack. However, the same stall can be triggered by any naturally occurring panic in the verification pipeline (e.g., a latent `unwrap()` bug hit by a valid mainnet block), making this a reliability concern independent of adversarial intent.

## Recommendation
In the panic handler inside `ConsumeUnverifiedBlocks::start()`, perform the same `unverified_tip` reset and block status update that the normal error path performs:

```rust
if let Err(payload) = catch_unwind(AssertUnwindSafe(|| {
    self.processor.consume_unverified_blocks(unverified_task);
})) {
    error!("consume unverified block {}-{} panicked: {}", ...);
    self.processor.is_pending_verify.remove(&block_hash);

    // Reset unverified_tip to the actual verified tip
    if let Some(tip) = self.processor.shared.store().get_tip_header() {
        if let Some(tip_ext) = self.processor.shared.store().get_block_ext(&tip.hash()) {
            self.processor.shared.set_unverified_tip(
                ckb_shared::HeaderIndex::new(tip.number(), tip.hash(), tip_ext.total_difficulty)
            );
        }
    }
    // Mark block invalid so it cannot re-enter the pipeline
    self.processor.shared.insert_block_status(block_hash, BlockStatus::BLOCK_INVALID);
}
```

## Proof of Concept
1. Run a CKB node in IBD mode.
2. Inject a temporary `panic!("test")` at the start of `consume_unverified_blocks` (simulating a verification pipeline panic).
3. Allow the node to receive and dispatch one block to the preload channel; observe `unverified_tip` advances to that block's number.
4. The panic fires; observe via logs that only `is_pending_verify.remove` executes.
5. Inspect `get_unverified_tip().number()` — it remains at the panicked block's number.
6. Inspect the block's status in `block_status_map` — it remains `BLOCK_RECEIVED`.
7. Attempt to re-submit the same block from a peer; observe `new_block_received` returns `false` due to status mismatch.
8. Observe the node's chain tip does not advance and `block_fetcher.fetch()` either returns `None` (if `unverified_tip >= tip + BLOCK_DOWNLOAD_WINDOW * 9`) or fetches blocks that all fail due to the unresolved parent, confirming the sync stall.
