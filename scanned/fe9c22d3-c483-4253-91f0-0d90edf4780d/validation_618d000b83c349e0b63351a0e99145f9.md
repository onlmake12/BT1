### Title
`ScriptHashTypeVerifier` Missing Enabled-Range Check for Output Type Scripts — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that output **lock** script `hash_type` values fall within the `ENABLED_SCRIPT_HASH_TYPE` allowlist, but performs no equivalent check on output **type** scripts. An unprivileged transaction sender can craft a transaction whose output carries a type script with a not-yet-activated `hash_type` (e.g., `Data3 = 6`), which passes every non-contextual gate and can be committed to a block, permanently locking the cell's CKB capacity.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the four currently activated hash-type values:

```
{0 = Data, 1 = Type, 2 = Data1, 4 = Data2}
```

Values `6` (`Data3`), `8` (`Data4`), … are structurally valid (they satisfy `verify_value`: even number or `1`) but are **not** activated.

`ScriptHashTypeVerifier::verify()` iterates over every output and checks only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  lines 796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else { ... }
    }
    Ok(())
}
```

`output.type_()` is never inspected. A type script carrying `hash_type = 6` (`Data3`) passes:

1. **`check_data`** — `verify_value(6)` is `true` (even number).
2. **`ScriptHashTypeVerifier`** — only the lock script is checked.
3. **tx-pool `non_contextual_verify`** — calls `NonContextualTransactionVerifier::verify()`, which delegates to `ScriptHashTypeVerifier`.

The transaction is accepted into the pool and can be mined into a block. The resulting live cell carries a type script whose `hash_type` is not in `ENABLED_SCRIPT_HASH_TYPE`. When any future transaction attempts to consume that cell, `select_version()` in `script/src/types.rs` hits the catch-all arm and returns `Err(ScriptError::InvalidScriptHashType(...))`, making the cell permanently unspendable.

---

### Impact Explanation

**Impact: Low**

Any CKB capacity locked in an output with an unsupported type-script `hash_type` is permanently frozen: the cell can never be consumed because script execution will always fail at `select_version`. The attacker burns their own funds and pollutes the live-cell set with provably unspendable UTXOs. There is no direct harm to other users' funds, but the inconsistency undermines the invariant that `ScriptHashTypeVerifier` is supposed to guarantee — that no cell with a non-activated hash type enters the chain — and creates a class of permanently dead cells that inflate the UTXO set.

---

### Likelihood Explanation

**Likelihood: High**

The attack requires no privileges, no special tooling, and no coordination. Any transaction sender who can construct a raw transaction (via RPC `send_raw_transaction` or the miner's `submit_block` path) can set `hash_type = 6` on a type script. The structural validity check (`verify_value`) passes, and the only guard that should catch it (`ScriptHashTypeVerifier`) does not inspect type scripts.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script:

```diff
 pub fn verify(&self) -> Result<(), Error> {
     for output in self.transaction.outputs() {
+        // Check lock script hash type
         if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
             let val: u8 = hash_type.into();
             if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                 return Err(
                     TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                 );
             }
         } else {
             return Err((TransactionError::InvalidScriptHashType {
                 hash_type: output.lock().hash_type(),
             }).into());
         }
+        // Check type script hash type (if present)
+        if let Some(type_script) = output.type_().to_opt() {
+            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
+                let val: u8 = hash_type.into();
+                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
+                    return Err(
+                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
+                    );
+                }
+            } else {
+                return Err((TransactionError::InvalidScriptHashType {
+                    hash_type: type_script.hash_type(),
+                }).into());
+            }
+        }
     }
     Ok(())
 }
```

### Proof of Concept

1. Build a transaction with one output whose **type script** has `hash_type = 6` (`Data3`) and whose lock script uses a standard `hash_type = 1` (`Type`).
2. Submit via `send_raw_transaction` RPC.
3. Observe the transaction is accepted by the tx-pool (no `ScriptHashTypeNotPermitted` rejection).
4. Mine the transaction into a block.
5. Attempt to spend the resulting cell in a subsequent transaction — the node rejects it with `InvalidScriptHashType` / `InvalidVmVersion`, confirming the cell is permanently unspendable.
