### Title
Unbounded Iteration over All Tx-Pool Entries on Every `estimate_fee_rate` RPC Call Enables CPU Exhaustion DoS — (File: `tx-pool/src/component/pool_map.rs`, `tx-pool/src/process.rs`, `rpc/src/module/experiment.rs`)

---

### Summary

The `estimate_fee_rate` RPC endpoint (and the `get_raw_tx_pool(verbose=true)` endpoint) triggers an unbounded linear-then-sort traversal of every entry in the tx-pool on each invocation. An unprivileged attacker who fills the pool with many low-fee transactions can then repeatedly call these endpoints to force O(N) to O(N log N) CPU work per call, with no rate limit or per-call entry cap, degrading or stalling the node's tx-pool service.

---

### Finding Description

**Vulnerability class:** Resource accounting / DoS via unbounded iteration over attacker-grown collection — the direct CKB analog of the NFT auction's `getBestBid()` looping over an unbounded `bids[]` array.

**Call chain for `estimate_fee_rate`:**

1. **RPC entry** — `ExperimentRpcImpl::estimate_fee_rate` in `rpc/src/module/experiment.rs` (lines 301–315) forwards the call to the tx-pool controller with no per-call guard. [1](#0-0) 

2. **Service dispatch** — `TxPoolController::estimate_fee_rate` in `tx-pool/src/service.rs` (lines 407–413) sends a synchronous `EstimateFeeRate` message to the pool actor. [2](#0-1) 

3. **Pool actor handler** — `TxPoolService::estimate_fee_rate` in `tx-pool/src/process.rs` (lines 945–970) first acquires the pool read-lock and calls `get_all_entry_info()`, which iterates **every** pending, gap, and proposed entry with no count cap. [3](#0-2) 

4. **Unbounded collection** — `TxPool::get_all_entry_info` in `tx-pool/src/pool.rs` (lines 464–487) calls `score_sorted_iter_by_statuses` and `sorted_proposed_iter`, both of which walk the entire multi-index map. [4](#0-3) 

5. **O(N log N) sort** — `weight_units_flow::Algorithm::estimate_fee_rate` in `util/fee-estimator/src/estimator/weight_units_flow.rs` (lines 173–184) collects all pending+proposed entries into a `Vec` and calls `sort_unstable`, making the per-call cost O(N log N) in pool size. [5](#0-4) 

6. **Fallback path — second full scan** — When the primary estimator returns `Err(NotReady)` (e.g., node just started), the code falls through to `PoolMap::estimate_fee_rate` in `tx-pool/src/component/pool_map.rs` (lines 334–358), which iterates **all** entries a second time in the same call, holding the read-lock throughout. [6](#0-5) 

**Second affected endpoint:** `get_raw_tx_pool(verbose=true)` in `rpc/src/module/pool.rs` (lines 703–718) also calls `get_all_entry_info()` and serializes every entry to JSON with no pagination or entry limit. [7](#0-6) 

**Why the pool can be grown by an attacker:** The pool is bounded only by `max_tx_pool_size` (a byte budget, not an entry count). With minimum-size transactions (~200 bytes each) and the default budget of ~180 MB, an attacker can place on the order of hundreds of thousands of entries. There is no per-sender entry cap analogous to the NFT fix of "one bid per bidder." [8](#0-7) 

---

### Impact Explanation

Every call to `estimate_fee_rate` (or `get_raw_tx_pool(verbose=true)`) while the pool is near capacity forces the tx-pool actor to:

- Traverse the entire multi-index map (O(N) with constant-factor overhead from the sorted index)
- Allocate and sort a `Vec` of N `TxStatus` structs (O(N log N))
- Optionally traverse the map a second time in the fallback path

Because the tx-pool actor processes messages sequentially (the `Message::EstimateFeeRate` arm in `tx-pool/src/service.rs` lines 1029–1039 is handled inline), a flood of concurrent `estimate_fee_rate` RPC calls queues up work in the actor, delaying all other pool operations including transaction submission, block assembly, and relay. At pool saturation this constitutes a sustained CPU and latency DoS against the node's tx-pool service reachable by any unprivileged RPC caller. [9](#0-8) 

---

### Likelihood Explanation

- The `estimate_fee_rate` RPC is publicly documented and enabled by default in the `Experiment` module; no authentication is required.
- Filling the pool requires paying transaction fees, but the attacker only needs to sustain pool occupancy, not continuously replace entries; a one-time cost suffices.
- The fallback path (`NotReady`) is active whenever the node has recently started or exited IBD, making the double-scan reachable without any special timing.
- No rate limiting, no per-call entry cap, and no pagination exist on either affected endpoint.

---

### Recommendation

1. **Cap per-call iteration** — Add a configurable `max_entries_per_fee_estimate` limit; truncate or sample the pool map before sorting rather than collecting all entries.
2. **Paginate `get_raw_tx_pool(verbose=true)`** — Accept `limit`/`offset` parameters and enforce a hard maximum (e.g., 1000 entries per call).
3. **Rate-limit expensive RPC methods** — Apply a per-IP or global call-rate limit to `estimate_fee_rate` and `get_raw_tx_pool` at the RPC layer.
4. **Decouple estimation from the pool actor** — Perform the sort and estimation on a snapshot copied outside the actor loop so that the actor's message queue is not blocked during the O(N log N) work.

---

### Proof of Concept

```
# Step 1: Fill the tx-pool with many small transactions from different addresses
# (each pays min_fee_rate; no per-sender limit prevents this)
for i in $(seq 1 50000); do
    ckb-cli tx send --tx <min_fee_tx_$i>
done

# Step 2: Flood the estimate_fee_rate RPC from any unprivileged client
while true; do
    curl -s -X POST http://<node>:8114 \
      -H 'Content-Type: application/json' \
      -d '{"id":1,"jsonrpc":"2.0","method":"estimate_fee_rate","params":[]}' &
done
```

With ~50 000 pool entries, each call forces a full O(N log N) sort. Concurrent calls queue in the pool actor, stalling transaction submission and block assembly responses for legitimate users. The `get_raw_tx_pool` variant additionally allocates a proportionally large JSON response, amplifying memory pressure.

            .sorted_proposed_iter()

            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {

        }
