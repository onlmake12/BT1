### Title
BlockFilter P2P Handler Silently Ignores Unsolicited Response Messages Without Banning Peer - (`File: sync/src/filter/mod.rs`)

### Summary
The `BlockFilter` P2P protocol handler in CKB's full node accepts unsolicited response-type messages (`BlockFilters`, `BlockFilterHashes`, `BlockFilterCheckPoints`) from any peer without banning or disconnecting them. The handler logs a warning and returns `Status::ignored()`, which carries status code `101` — outside the `4xx` range required to trigger a peer ban. An explicit `// TODO: ban remote peer` comment in the code acknowledges this is incomplete. Any unprivileged peer can repeatedly send these unsolicited messages to exhaust node resources without consequence.

### Finding Description

In `sync/src/filter/mod.rs`, the `BlockFilter::try_process` function dispatches incoming P2P messages. The three response-type variants — `BlockFilters`, `BlockFilterHashes`, and `BlockFilterCheckPoints` — are messages that a full node acting as a filter server should only ever *send*, never *receive*. When a peer sends one of these unsolicited, the handler:

1. Logs a `warn_target!` message
2. Returns `Status::ignored()` (code `101`) [1](#0-0) 

The `process()` wrapper then checks whether to ban the peer based on the returned status: [2](#0-1) 

`Status::should_ban()` only returns `Some(...)` for codes in the `400..500` range: [3](#0-2) 

`StatusCode::Ignored = 101` is in the `1xx` informational range, so `should_ban()` returns `None` and no ban is issued. The peer is free to send unlimited unsolicited filter response messages.

The code itself acknowledges this is wrong with an explicit comment: [4](#0-3) 

By contrast, the malformed-message path in the same `received()` function correctly bans the peer: [5](#0-4) 

### Impact Explanation

A peer that sends unsolicited `BlockFilters`, `BlockFilterHashes`, or `BlockFilterCheckPoints` messages forces the receiving node to:
- Deserialize and parse each message (molecule decoding)
- Execute the `try_process` match arm
- Emit a warning log entry

Because no ban is applied, the attacker can repeat this indefinitely at full link speed. This is a resource-exhaustion / griefing vector: CPU cycles for parsing, memory pressure from log buffering, and I/O for log writes are consumed with zero cost to the attacker. The `BAD_MESSAGE_BAN_TIME` constant used elsewhere would normally prevent this by disconnecting and temporarily blocking the peer's IP.

### Likelihood Explanation

Any unprivileged peer that establishes a connection to a CKB node with the Filter protocol enabled can trigger this. No special privileges, keys, or majority hashpower are required. The Filter protocol is a standard CKB protocol (`SupportProtocols::Filter`), so this is reachable from any connected peer.

### Recommendation

Replace `Status::ignored()` with a ban-triggering status code for the unsolicited response message arm, consistent with how other protocol handlers treat unexpected messages:

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    StatusCode::ProtocolMessageIsMalformed
        .with_context("received unsolicited block filter response")
}
```

`StatusCode::ProtocolMessageIsMalformed = 400` falls in the `4xx` range, causing `should_ban()` to return `Some(BAD_MESSAGE_BAN_TIME)` and triggering a peer ban via `nc.ban_peer(...)`. [6](#0-5) 

### Proof of Concept

1. Connect to a CKB full node that has the Filter protocol enabled.
2. Send a `BlockFilterMessage` containing a `BlockFilters` (or `BlockFilterHashes` / `BlockFilterCheckPoints`) variant — a response message the node never requested.
3. Observe: the node logs `"Received unexpected message from peer"` but does **not** disconnect or ban the peer.
4. Repeat step 2 in a tight loop. The node continues processing each message, consuming CPU and logging resources, while the attacker remains connected indefinitely.

    pub fn should_ban(&self) -> Option<Duration> {
        if !(400..500).contains(&(self.code as u16)) {
            return None;
        }
        if let Some(context) = &self.context {
            // TODO: it might be worthwhile to formalize all error texts
            // that won't be banned.
            if context.contains(ARGV_TOO_LONG_TEXT) {
                return None;
            }
        }
        match self.code {
            StatusCode::GetHeadersMissCommonAncestors => Some(SYNC_USELESS_BAN_TIME),
            _ => Some(BAD_MESSAGE_BAN_TIME),
        }
    }
