### Title
Sub-second RTT Truncation via `as_secs()` Allows Attacker to Monopolize Ping-Based Eviction Protection Slots — (`network/src/peer_registry.rs`)

---

### Summary

`try_evict_inbound_peer` uses `Duration::as_secs()` to compare peer ping RTTs for eviction protection. This truncates all sub-second RTTs to `0`, collapsing the entire [1ms, 999ms] range into a single bucket. An attacker controlling 8+ inbound connections can guarantee their peers occupy all `EVICTION_PROTECT_PEERS = 8` low-ping protection slots whenever honest peers have RTTs ≥ 1s, causing only honest peers to remain as eviction candidates.

---

### Finding Description

**Root cause — `as_secs()` precision loss:**

In `try_evict_inbound_peer`, the first protection pass sorts peers by ping RTT using `as_secs()`: [1](#0-0) 

`Duration::as_secs()` returns the whole-second component only, truncating any sub-second RTT to `0`. A peer with RTT = 1ms and a peer with RTT = 999ms are treated identically.

**How `sort_then_drop` works:** [2](#0-1) 

The comparator `peer2_ping.cmp(&peer1_ping)` sorts **descending** (highest ping first). `truncate(len - n)` then keeps the first `(len - n)` elements (highest ping) in the candidate list and **removes** the last `n` (lowest ping) from it — those last `n` are the **protected** peers.

**Attack mechanics:**

With the comparator sorting descending by `as_secs()`:
- Honest peers with RTT ≥ 1s → `as_secs()` = 1+ → sorted to the **front** → remain as eviction candidates
- Attacker peers with RTT ∈ [1ms, 999ms] → `as_secs()` = 0 → sorted to the **end** → fill the 8 protected slots

The attacker controls pong response timing (via `pong_received` which records `now.saturating_duration_since(last_ping)`) to produce any sub-second RTT at will: [3](#0-2) 

The same `as_secs()` truncation also affects the second protection criterion (most-recently-sent-messages): [4](#0-3) 

---

### Impact Explanation

Once all 8 ping-protection slots are filled by attacker peers, honest peers with RTT ≥ 1s are left unprotected and become the sole eviction candidates. The attacker can repeatedly trigger eviction (by connecting a new peer whenever the node is at `max_inbound`) to systematically displace all honest inbound peers. This enables an eclipse attack: the victim node's inbound peer set becomes entirely attacker-controlled, allowing network partition and consensus deviation.

---

### Likelihood Explanation

The attack requires honest peers to have RTT ≥ 1s, which is the binding constraint. This holds for:
- Cross-continental or satellite-connected peers (RTTs of 1–3s are common)
- Nodes in regions with high-latency infrastructure

The attacker needs only 8 inbound connections (a low bar) and the ability to respond to pings within <1s (trivially achievable). No privileged access, PoW, or key material is required. The attack is repeatable and persistent as long as the node remains at `max_inbound`.

---

### Recommendation

Replace `as_secs()` with `as_millis()` (or `as_nanos()`) in both protection sort comparators in `try_evict_inbound_peer`:

```rust
// Ping protection
.map(|p| p.as_millis())
.unwrap_or(u128::MAX)

// Last-message protection
.map(|t| now.saturating_duration_since(t).as_millis())
.unwrap_or(u128::MAX)
```

This restores millisecond-level discrimination, making it practically impossible for an attacker to guarantee their peers land in the lowest-ping bucket ahead of honest peers with typical sub-second RTTs.

---

### Proof of Concept

```rust
// 8 attacker peers: ping_rtt = 999ms → as_secs() = 0
// 8 honest peers:   ping_rtt = 1000ms → as_secs() = 1
// After descending sort by as_secs():
//   [honest(1), honest(1), ..., attacker(0), attacker(0), ...]
// truncate(len - 8) removes last 8 (all attacker peers) → protected
// All honest peers remain as candidates → always evicted
for _ in 0..1000 {
    let evicted = registry.try_evict_inbound_peer(&peer_store);
    assert!(honest_peer_ids.contains(&evicted.unwrap()),
        "honest peer must never be evicted — but it was");
}
```

The assertion would pass (honest peers are always evicted), confirming the invariant violation stated in the question.

fn sort_then_drop<T, F>(list: &mut Vec<T>, n: usize, compare: F)

    F: FnMut(&T, &T) -> std::cmp::Ordering,

    if list.len() > n {
        list.truncate(list.len() - n);
    }
}
