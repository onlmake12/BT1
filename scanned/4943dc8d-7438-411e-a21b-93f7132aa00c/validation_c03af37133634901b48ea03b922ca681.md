### Title
Unbounded Per-Message I/O Work with No Per-Peer Rate Limit in `GetBlocksProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

---

### Summary

An unprivileged remote light-client peer can continuously send structurally valid `GetBlocksProof` messages each containing up to `GET_BLOCKS_PROOF_LIMIT` (1000) valid main-chain block hashes. Each such message forces the server to perform 4 × 1000 synchronous DB reads plus one `mmr.gen_proof(1000 positions)` call. Because the message is valid, the handler returns `Status::ok()` (HTTP-class 200), which never triggers a ban. No per-peer rate limit or message-frequency guard exists anywhere in the light-client protocol server, so a single peer can sustain this at negligible cost.

---

### Finding Description

**Constant ceiling, no rate guard.** [1](#0-0) 

`GET_BLOCKS_PROOF_LIMIT = 1000` is the only bound. It caps the work per message but does not limit how many messages a peer may send per unit time.

**Per-message work inside `execute()`.**

For every hash in the `found` partition (up to 1000), the handler performs: [2](#0-1) 

- `snapshot.is_main_chain(block_hash)` — DB read (×1000)
- `snapshot.get_block_header(block_hash)` — DB read (×1000)
- `snapshot.get_block_uncles(block_hash)` — DB read (×1000)
- `snapshot.get_block_extension(block_hash)` — DB read (×1000)

Then `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1000 positions: [3](#0-2) 

**Valid messages never trigger a ban.**

`should_ban()` only returns `Some(ban_time)` for 4xx status codes: [4](#0-3) 

A well-formed request with valid hashes returns `StatusCode::OK` (200). The dispatch loop in `received()` only calls `nc.ban_peer` when `status.should_ban()` is `Some`: [5](#0-4) 

**No rate-limiting code exists in the light-client server.**

A grep for `rate_limit`, `throttle`, `quota`, `message_count`, `flood`, or `per_peer` across the entire `util/light-client-protocol-server/` tree returns zero matches. The `LightClientProtocol` struct holds only a `Shared`: [6](#0-5) 

There is no counter, token bucket, or timestamp guard on a per-peer basis.

---

### Impact Explanation

A single attacker peer can saturate the light-client server's RocksDB I/O and CPU by sending a continuous stream of `GetBlocksProof` messages each carrying 1000 valid main-chain hashes. This causes:

- **CPU saturation**: MMR proof generation over 1000 leaf positions is non-trivial per call.
- **I/O saturation**: 4000 synchronous DB reads per message compete with normal chain operations.
- **Network congestion**: The server must also serialize and send a large `SendBlocksProofV1` response for each message.

Because the server is single-threaded per protocol handler and processes messages sequentially, legitimate light-client peers are starved. The attacker pays only the cost of sending ~32 KB messages (1000 × 32-byte hashes + overhead).

---

### Likelihood Explanation

The attack requires only a valid P2P connection to a node running the light-client protocol server. No PoW, no key, no privileged role. The attacker needs to know 1000 valid main-chain block hashes, which are trivially obtained by syncing a few headers first. The existing test already confirms no ban is issued for valid requests: [7](#0-6) 

---

### Recommendation

1. **Per-peer message-rate limit**: Track the timestamp of the last `GetBlocksProof` message per peer and enforce a minimum inter-message interval (e.g., 1 request per second). Disconnect or ban peers that exceed it.
2. **Inflight request cap**: Allow at most one outstanding `GetBlocksProof` per peer; drop or ban if a second arrives before the first response is sent.
3. **Reduce the per-message ceiling**: `GET_BLOCKS_PROOF_LIMIT = 1000` is generous; a lower value (e.g., 100–200) reduces per-message cost while still serving legitimate clients.
4. **Apply the same fix to `GetTransactionsProof`** and `GetLastStateProof`, which share the same structural pattern and the same `GET_*_LIMIT = 1000` constant. [8](#0-7) 

---

### Proof of Concept

```
1. Connect to a CKB node with the light-client protocol enabled.
2. Sync enough headers to collect 1000 valid main-chain block hashes H[0..999].
3. In a tight loop:
     msg = GetBlocksProof { last_hash: tip_hash, block_hashes: H[0..999] }
     send(msg)
4. Observe: server processes each message (4000 DB reads + mmr.gen_proof(1000)),
   returns SendBlocksProofV1, issues no ban.
5. After K iterations the server's RocksDB I/O queue and async executor are saturated;
   legitimate peers receive no responses.
```

The existing unit test at `util/light-client-protocol-server/src/tests/components/get_blocks_proof.rs` line 78 already asserts `nc.not_banned(peer_index)` for a valid request, confirming the absence of any ban or throttle on the happy path.
