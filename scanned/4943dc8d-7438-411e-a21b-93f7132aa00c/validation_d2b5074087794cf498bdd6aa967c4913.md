### Title
Unbounded Unauthenticated DoS via `GetBlocksProof` with Mismatched `last_hash`/`block_hashes` Triggering Unpenalized `InternalError` — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`, `util/light-client-protocol-server/src/lib.rs`)

### Summary
An unprivileged remote peer can craft a `GetBlocksProof` message where `last_hash` references a low-numbered main-chain block (e.g., block 1) and `block_hashes` contains up to 1000 hashes of higher-numbered main-chain blocks (e.g., blocks 2–1001). Every validation check passes, the server performs ~3000 database reads, then `reply_proof` calls `mmr.gen_proof` with positions that are out-of-range for the MMR anchored at `last_block.number() - 1`, causing an `InternalError` (5xx). The 5xx code is explicitly excluded from the ban predicate, so the peer is never penalized and can repeat the attack indefinitely.

### Finding Description
In `GetBlocksProofProcess::execute`, the server validates that `last_hash` and each entry in `block_hashes` are individually on the main chain, but it does **not** validate that each requested block's number is ≤ `last_block.number()`. As a result, a peer can supply a `last_hash` pointing to block 1 and `block_hashes` pointing to blocks 2–1001, all of which pass `is_main_chain`.

The server then performs 3 DB reads per found block (header, uncles, extension) — 3000 reads total — before calling `reply_proof`. Inside `reply_proof`, the MMR is constructed as `chain_root_mmr(last_block.number() - 1)` = `chain_root_mmr(0)`, which covers only the genesis block. Calling `mmr.gen_proof` with positions for leaves 2–1001 on a size-1 MMR returns `Err`, and the function returns `StatusCode::InternalError` (500).

Back in `received`, `status.should_ban()` checks `(400..500).contains(&code)`. Since 500 is not in that range, no ban is issued. There is no rate limiter on the light-client protocol handler. The attacker can repeat this at will.

### Impact Explanation
Each malicious request forces ~3000 synchronous database reads and an MMR proof attempt before failing. With no ban and no rate limit, an attacker with a single persistent connection can saturate the server's I/O and CPU, degrading or denying service to legitimate light clients. The impact is a sustained, low-cost, high-amplification DoS against the light-client server.

### Likelihood Explanation
The attack requires only a valid P2P connection to a node running the light-client protocol server. No keys, no PoW, no privileged access. The attacker needs only to know two main-chain block hashes at different heights, which are publicly available. The attack is trivially scriptable and repeatable.

### Recommendation
1. **Add a number-range check in `execute`**: after resolving each found block's header, reject (with `MalformedProtocolMessage` / 4xx) any block whose `header.number() > last_block.number()`. This converts the attacker-controlled error path from 5xx to 4xx, triggering the existing ban logic.
2. **Add per-peer rate limiting** to `LightClientProtocol::received`, analogous to the rate limiter already present in the hole-punching and relay protocol handlers.

### Proof of Concept
