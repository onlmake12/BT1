### Title
Silent Truncating Cast in DAO Withdrawal Capacity Calculation Produces Incorrect Withdrawal Amounts — (File: `util/dao/src/lib.rs`)

### Summary

In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result of the DAO interest calculation is cast to `u64` using a silent truncating `as u64` cast. Every other analogous `u128 → u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, which propagates an error on overflow. The inconsistent `as u64` cast silently discards the high bits, producing a severely underestimated withdrawal capacity without any error signal.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a Rust wrapping/truncating cast. If `withdraw_counted_capacity` exceeds `u64::MAX`, the high bits are silently discarded and the result wraps to a small value. No error is returned.

Compare this to every other `u128 → u64` conversion in the same file, which all use checked conversion:

```rust
// Line 244-245 (miner_issuance):
u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?

// Line 258 (ar_increase):
u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?
``` [2](#0-1) [3](#0-2) 

The `calculate_maximum_withdraw` function is the only site that uses the unsafe cast. The `DaoError::Overflow` variant exists precisely for this purpose and is used everywhere else. [4](#0-3) 

### Impact Explanation

If `withdraw_counted_capacity` overflows `u64`, the truncated value is used as the withdrawal capacity. This has two downstream effects:

1. **Incorrect fee calculation**: `transaction_fee` calls `transaction_maximum_withdraw` → `calculate_maximum_withdraw`. A truncated (too-small) `maximum_withdraw` causes `maximum_withdraw.safe_sub(outputs_capacity)` to fail, making a valid DAO withdrawal transaction appear to have negative fee and be rejected from the tx-pool. [5](#0-4) 

2. **Inflated DAO savings pool**: `withdrawed_interests` calls `calculate_maximum_withdraw` and feeds the result into `dao_field_with_current_epoch`. A truncated (too-small) `withdrawed_interests` causes `current_s` to be overstated, inflating the NervosDAO savings pool and allowing future depositors to earn more interest than the protocol intends. [6](#0-5) 

The `DaoHeaderVerifier` uses the same buggy `DaoCalculator` to verify the DAO field in blocks, so a miner computing the DAO field with the truncated value would produce a block that passes consensus verification — the error is not caught at the consensus layer. [7](#0-6) 

### Likelihood Explanation

The overflow condition requires `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. Since `counted_capacity ≤ total_CKB_supply ≈ 3.36 × 10^18 shannons` and `u64::MAX ≈ 1.844 × 10^19 shannons`, the current total supply is approximately 18% of `u64::MAX`. With perpetual secondary issuance of ~1.344 billion CKB/year, the supply approaches `u64::MAX` on a multi-decade timescale. The likelihood is therefore **very low in the near term** but the defect is a latent protocol risk and represents a clear code inconsistency relative to every other overflow-guarded conversion in the same file.

### Recommendation

Replace the silent truncating cast with the same checked conversion pattern used everywhere else in the file:

```rust
let withdraw_capacity = Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?;
``` [8](#0-7) 

### Proof of Concept

Construct a DAO cell with `output_capacity` near `u64::MAX` and a `withdrawing_ar` sufficiently larger than `deposit_ar` such that `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. The `as u64` cast wraps the result to a small value (e.g., if the true result is `u64::MAX + 100`, the cast yields `99`). `Capacity::shannons(99).safe_add(occupied_capacity)` succeeds and returns a capacity of ~61 CKBytes instead of the correct ~184 billion CKB. The existing unit test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` demonstrates the overflow path but relies on `safe_add` catching the error after the truncation — it does not test the case where the truncated value is small enough that `safe_add` also succeeds, which is the silent-miscalculation scenario. [9](#0-8)

fn check_withdraw_calculation_overflows() {

        .number(100)

        .dao(pack_dao_data(

            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
