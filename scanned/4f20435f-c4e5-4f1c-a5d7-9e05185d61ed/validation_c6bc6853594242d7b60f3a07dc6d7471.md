### Title
Unverified `from` Peer Identity in Hole-Punching `ConnectionRequest` Enables NAT-Traversal Hijacking and Rate-Limiter Bypass â€” (`File: network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The `ConnectionRequest` handler in CKB's hole-punching protocol trusts the `from` field supplied inside the message payload without ever verifying it against the actual session peer identity. Any connected peer can claim to be any arbitrary `PeerId` in the `from` field. This attacker-controlled identity is then used as the key for the `pending_delivered` cache and the `forward_rate_limiter`, enabling NAT-traversal hijacking and rate-limiter bypass.

---

### Finding Description

In `ConnectionRequestProcess::execute()`, the `from` peer ID is parsed exclusively from the message payload:

```rust
// connection_request.rs line 111-113
let content = match RequestContent::try_from(&self.message) {
    Ok(content) => content,
    Err(status) => return status,
};
```

`RequestContent::from` is decoded from `value.from().raw_data()` â€” fully attacker-controlled. The actual session identity (`self.peer`, a `PeerIndex`/`SessionId`) is **never compared** to `content.from`.

This attacker-controlled `from` is then used in two security-sensitive operations:

**1. Forward rate-limiter keyed on `content.from`:**
```rust
// connection_request.rs lines 132-143
if self
    .protocol
    .forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
    .is_err()
{ ... }
```
By varying `content.from` across requests, an attacker bypasses the per-`(from, to)` rate limit entirely.

**2. `pending_delivered` cache keyed on `content.from`:**
```rust
// connection_request.rs lines 234-237
let now = unix_time_as_millis();
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```
An attacker sends a `ConnectionRequest` with `to = self_peer_id` (the target node's own ID) and `from = victim_peer_id`. The node calls `respond_delivered(content.from, ...)`, inserting `(victim_peer_id â†’ attacker_listen_addrs)` into `pending_delivered`.

Later, when a legitimate `ConnectionSync` arrives with `from = victim_peer_id`, the node looks up:
```rust
// connection_sync.rs lines 111-115
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```
It finds the **attacker's addresses** and initiates NAT traversal to them instead of the real peer's addresses.

The `pending_delivered` HashMap has no size bound and is only pruned every 5 minutes (`TIMEOUT = 5 * 60 * 1000 ms`), so an attacker can also exhaust memory by flooding with distinct fake `from` peer IDs.

---

### Impact Explanation

1. **NAT-traversal hijacking**: An attacker poisons `pending_delivered[victim_peer_id]` with their own listen addresses. When the victim's `ConnectionSync` arrives, the target node attempts TCP hole-punching to the attacker's endpoints instead of the victim's, redirecting the connection establishment.

2. **Forward rate-limiter bypass**: The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`. By cycling through arbitrary `from` values, a single connected peer can flood the network with forwarded `ConnectionRequest` broadcasts at unlimited rate, amplifying traffic across the gossip network.

3. **`pending_delivered` memory exhaustion**: Each unique fake `from` PeerId inserts a new entry. With no size cap and a 5-minute eviction window, a single peer can grow the HashMap to consume significant memory.

---

### Likelihood Explanation

Any peer connected to a CKB node over the HolePunching protocol can send a `ConnectionRequest` message. The outer `rate_limiter` (keyed by `(session_id, msg.item_id())`) limits to 30 messages/second per session, but this does not prevent the attack â€” 30 distinct fake `from` values per second is sufficient to poison the cache and bypass the forward rate limiter. No special privilege, key material, or majority hashpower is required.

---

### Recommendation

In `ConnectionRequestProcess::execute()`, resolve the actual `PeerId` of the sending session from the peer registry using `self.peer` (the `PeerIndex`/`SessionId`) and assert it equals `content.from` before proceeding. If they differ, return an error status (and optionally ban the session). This mirrors the correct pattern used when the node originates its own requests in `HolePunching::notify()`, where `from_peer_id` is always `self.network_state.local_peer_id()` â€” a trusted, locally-derived identity.

### Proof of Concept

1. Attacker peer `A` connects to victim node `V`.
2. `A` sends a `ConnectionRequest` with `from = peer_id_of_B` (any legitimate peer B's ID), `to = V.peer_id`, and `listen_addrs = [attacker_tcp_addr]`.
3. `V` receives the message. `self_peer_id == content.to`, so `respond_delivered(peer_id_of_B, ...)` is called.
4. `V` inserts `pending_delivered[peer_id_of_B] = ([attacker_tcp_addr], now)`.
5. Later, `B` legitimately initiates hole-punching to `V` and the `ConnectionSync` message (with `from = peer_id_of_B`) reaches `V`.
6. `V` looks up `pending_delivered[peer_id_of_B]`, finds `[attacker_tcp_addr]`, and calls `try_nat_traversal(ttl, [attacker_tcp_addr])` â€” connecting to `A` instead of `B`.
