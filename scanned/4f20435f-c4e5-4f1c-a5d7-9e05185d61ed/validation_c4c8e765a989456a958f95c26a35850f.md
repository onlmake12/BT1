### Title
`ScriptHashTypeVerifier` Fails to Validate Type Script `hash_type` on Transaction Outputs — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier`, the non-contextual allowlist guard for script hash types, only validates the `hash_type` field of **lock scripts** on transaction outputs. It never inspects **type scripts** on the same outputs. This mirrors the external report's pattern exactly: a facade that routes to an executor without enforcing an allowlist on the executor. Any unprivileged `send_transaction` RPC caller can submit a transaction whose output carries a type script with an unenabled `hash_type` (e.g., `Data3 = 6`). The transaction bypasses the cheap non-contextual gate, enters the tx-pool pipeline, and is only rejected later during the more expensive contextual/script-execution stage.

---

### Finding Description

**Root cause — `ScriptHashTypeVerifier::verify()` only reads `output.lock()`:**

```rust
// verification/src/transaction_verifier.rs  lines 796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())  // ← lock only
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())   // ← output.type_() is never examined
}
```

The `ENABLED_SCRIPT_HASH_TYPE` allowlist is `{0, 1, 2, 4}` (Data, Type, Data1, Data2). The `ScriptHashType` enum, generated by a `seq!` macro, defines values up to `Data127 = 254`; every even value ≥ 6 (e.g., `Data3 = 6`, `Data4 = 8`, …) is a structurally valid but **unenabled** hash type.

**Attack path:**

1. Attacker calls `send_transaction` (RPC, no privilege required) with `outputs_validator = "passthrough"` and a transaction whose output has:
   - a valid lock script (e.g., secp256k1, `hash_type = "type"`)
   - a type script with `hash_type = "data3"` (byte value `6`)
2. `NonContextualTransactionVerifier::verify()` runs `ScriptHashTypeVerifier::verify()`. Because only `output.lock()` is inspected, the check passes.
3. The transaction enters the tx-pool and proceeds to contextual verification (`verify_rtx`).
4. `ContextualTransactionVerifier::verify()` calls `TransactionScriptsVerifier`. The verifier calls `TxInfo::select_version()` for the type script group. `select_version` hits the catch-all arm and returns `ScriptError::InvalidScriptHashType("The ScriptHashType/Data3 has not been activated…")`.
5. The transaction is rejected — but only after consuming contextual-verification resources.

**The `NonContextualTransactionVerifier` doc comment itself exposes the gap:**

```
/// - Check whether output lock hash type within enabled range
```

Type scripts are not mentioned; the omission is structural, not accidental.

---

### Impact Explanation

The `ScriptHashTypeVerifier` is the designated cheap gate that is supposed to reject structurally invalid transactions before they reach the expensive script-execution stage. By not covering type scripts, it allows an attacker to force every submitted transaction through the full contextual-verification pipeline — including script-group construction, `select_version` dispatch, and error propagation — before rejection. Repeated at scale (each transaction requires only valid inputs and the minimum fee rate), this creates a resource-exhaustion vector against the tx-pool admission worker. Additionally, any future code path that relies on `NonContextualTransactionVerifier` passing as a sufficient pre-filter for hash-type correctness (e.g., caching, relay decisions) inherits the same blind spot.

---

### Likelihood Explanation

The entry point is the public `send_transaction` JSON-RPC method, reachable by any unprivileged peer or local caller. The attacker needs only: (a) a live UTXO to use as input, (b) the minimum fee rate, and (c) knowledge that `hash_type = "data3"` is accepted by the JSON schema but not by the allowlist. No keys, no operator access, no majority hashpower are required.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of type scripts when present on outputs:

```rust
for output in self.transaction.outputs() {
    // existing lock-script check …

    // add: type-script check
    if let Some(type_script) = output.type_().to_opt() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: type_script.hash_type(),
            }).into());
        }
    }
}
```

Update the `NonContextualTransactionVerifier` doc comment to reflect that both lock and type script hash types are checked.

        self.script_hash_type.verify()?;

pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
