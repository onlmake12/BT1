Audit Report

## Title
Stale `ancestors_count` in `check_and_record_ancestors` Eviction Loop Causes Over-Eviction of Pool Transactions - (File: tx-pool/src/component/pool_map.rs)

## Summary
In `check_and_record_ancestors`, the local `ancestors_count` variable is decremented by exactly `1` per loop iteration regardless of how many entries in the `ancestors` set are actually removed by `remove_entry_and_descendants`. When a `cell_ref_parent` has descendants that are also ancestors of the incoming transaction, a single eviction call removes multiple members of the `ancestors` set, but the counter only drops by one. The loop therefore continues past the point where the limit is already satisfied, evicting additional valid transactions from the mempool unnecessarily.

## Finding Description
`check_and_record_ancestors` (lines 588–640) computes `ancestors_count = ancestors.len() + 1` at line 595, where `ancestors` is the full transitive ancestor set of the incoming transaction as returned by `get_tx_ancenstors`. The eviction loop at lines 615–625 iterates over `evict_candidates` (the `cell_ref_parents` sorted by evict key) and calls `remove_entry_and_descendants(next_id)` for each candidate, then unconditionally executes `ancestors_count = ancestors_count.saturating_sub(1)`.

`remove_entry_and_descendants` (lines 252–265) removes the named entry **and every transitive descendant** via `calc_descendants`. If any of those descendants are also members of the `ancestors` set of the incoming transaction — which occurs when the new tx spends an output of a descendant of the `cell_ref_parent` — the true ancestor count drops by `1 + k` (where `k ≥ 1`), but `ancestors_count` only drops by `1`. The loop condition `ancestors_count > self.max_ancestors_count` therefore remains true for `k` extra iterations, causing `k` additional `cell_ref_parents` to be evicted.

The post-loop assertion at line 636 (`assert!(ancestors.len() < self.max_ancestors_count)`) recomputes ancestors from the live pool state and will pass because the actual count is already below the limit — but by then the extra evictions have already occurred.

The guard at line 603 (`ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`) only checks whether removing all direct `cell_ref_parents` would bring the count within limits; it does not account for the cascading removal of their descendants that are also ancestors of the new tx, so it does not prevent the over-eviction.

## Impact Explanation
Every unnecessary eviction permanently removes a valid, fee-paying transaction from the mempool for the duration until it is resubmitted. The eviction list is sorted by lowest fee-rate first (`iter_by_evict_key`), so the cheapest `cell_ref_parents` are targeted first; however, the stale counter causes additional (potentially higher-fee) `cell_ref_parents` to be evicted as well. This constitutes an important correctness defect in the mempool eviction accounting, matching **Low (501–2000 points): Any other important performance/correctness improvement for CKB**. The impact does not rise to High because evicted transactions can be resubmitted, the over-eviction is bounded by the size of `cell_ref_parents`, and the attacker must spend fees to construct the triggering graph.

## Likelihood Explanation
The trigger condition is reachable by any unprivileged user via `send_transaction` (RPC) or P2P relay. The attacker submits: (1) `tx_A` with some output `O` used as a cell dep, (2) `tx_B` spending `tx_A`'s output, and (3) a trigger transaction that both spends `O` as an input (making `tx_A` a `cell_ref_parent`) and spends `tx_B`'s output (making `tx_B` an ancestor). No privileged access, leaked keys, or victim mistakes are required. The scenario is repeatable and deliberate.

## Recommendation
Replace the fixed `-1` decrement with the actual count of removed entries that were members of the `ancestors` set:

```rust
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id);
        let removed_ancestor_count = removed
            .iter()
            .filter(|e| ancestors.contains(&e.proposal_short_id()))
            .count();
        ancestors_count = ancestors_count.saturating_sub(removed_ancestor_count.max(1));
        parents.remove(next_id);
        evicted.extend(removed);
    } else {
        break;
    }
}
```

Alternatively, recompute `ancestors_count` from the live pool state after each eviction (analogous to the post-loop recalculation already done at lines 631–633), or break early once the recalculated ancestor set already satisfies the limit.

## Proof of Concept
**Pool state before trigger:**

| Tx | Role |
|----|------|
| `tx_A` | root; holds output `O` as a cell dep in some in-pool tx |
| `tx_B` | spends `tx_A`'s output (descendant of `tx_A`) |

**Trigger transaction:** spends `O` as an input (making `tx_A` a `cell_ref_parent`) and spends `tx_B`'s output (making `tx_B` an ancestor). With `max_ancestors_count = 2`, `ancestors = {tx_A, tx_B}`, `ancestors_count = 3`, `cell_ref_parents = {tx_A}`.

- Guard at line 603: `3 - 1 = 2 <= 2` → enters eviction loop.
- Iteration 1: evict `tx_A` → removes `tx_A` and `tx_B`; true ancestor count drops to 0; but `ancestors_count` becomes `3 - 1 = 2 > 2` is false — wait, `2 > 2` is false, so the loop exits. In this minimal case the loop exits correctly.

To demonstrate over-eviction, extend to two `cell_ref_parent` pairs: `tx_A1 → tx_B1` and `tx_A2 → tx_B2`, with the trigger spending both `tx_B2`'s output and outputs `O1`, `O2`. `ancestors = {tx_A1, tx_B1, tx_A2, tx_B2}`, `ancestors_count = 5`, `cell_ref_parents = {tx_A1, tx_A2}`, `max_ancestors_count = 3`.

- Guard: `5 - 2 = 3 <= 3` → enters loop.
- Iteration 1: evict `tx_A1` → removes `tx_A1` and `tx_B1` (both ancestors); true count drops to `5 - 2 = 3`; but `ancestors_count = 5 - 1 = 4 > 3` → loop continues.
- Iteration 2: evict `tx_A2` → removes `tx_A2` and `tx_B2` unnecessarily; `ancestors_count = 4 - 1 = 3`, loop exits.

Expected: only `tx_A1` (and `tx_B1`) evicted. Actual: `tx_A2` and `tx_B2` also evicted. The post-loop assertion at line 636 passes because the true ancestor count is now 0, masking the over-eviction. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
