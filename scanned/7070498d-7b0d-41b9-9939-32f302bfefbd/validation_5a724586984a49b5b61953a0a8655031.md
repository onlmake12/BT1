Audit Report

## Title
`remove_expired` Omits Descendant Cascade, Leaving Child Transactions Permanently Stuck with Dangling Inputs — (`tx-pool/src/pool.rs`)

## Summary

`TxPool::remove_expired` calls `pool_map.remove_entry` on each expired transaction, which removes only that single entry. When a parent transaction expires, its in-pool children remain with inputs referencing outputs that no longer exist in the pool or on-chain. These children can never be mined and occupy pool space until their own expiry or until `limit_size` evicts them. An unprivileged attacker can exploit this to continuously fill the pool with unresolvable child transactions at near-zero cost.

## Finding Description

`remove_expired` collects all entries whose `timestamp + expiry < now_ms` and removes each individually via `pool_map.remove_entry`:

```rust
// tx-pool/src/pool.rs L284
self.pool_map.remove_entry(&entry.proposal_short_id());
```

`remove_entry` (pool_map.rs L235–250) cleans up the removed entry's own edges and links — including calling `remove_entry_links`, which strips the parent reference from each child's `links` entry. However, the children themselves are **not removed**. Their inputs, which reference the now-gone parent's outputs, remain registered in `edges.inputs` (pool_map.rs L10: `HashMap<OutPoint, ProposalShortId>`). Those outpoints no longer exist anywhere (parent was never mined; parent is no longer in pool), so the children's inputs are permanently unresolvable.

By contrast, `limit_size` (pool.rs L307) calls `pool_map.remove_entry_and_descendants`, which first collects the full descendant closure via `calc_descendants` (pool_map.rs L252–265) and removes all of them:

```rust
// tx-pool/src/component/pool_map.rs L252-264
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

The call chain is: new block → `_update_tx_pool_for_reorg` → `remove_expired` (process.rs L1110), followed immediately by `limit_size` (process.rs L1113). `limit_size` only evicts when `total_tx_size > max_tx_pool_size`, so stuck children below the size threshold are never cleaned up by it.

**Exploit path:**
1. Attacker submits parent `P` at time `T`.
2. Attacker waits until `T + expiry - ε`, then submits children `C1…CN` spending `P`'s outputs. Children get timestamps just before expiry.
3. At the next block after `T + expiry`, `remove_expired` fires. `P` is removed; `C1…CN` are not (their timestamps haven't crossed expiry yet).
4. `C1…CN` remain in the pool with inputs pointing to `P`'s outputs, which no longer exist. They cannot be included in any block.
5. Attacker repeats, accumulating stuck children. Each cycle costs only RPC calls (transactions are never mined, so no fees are paid).
6. When the pool fills, `limit_size` evicts legitimate transactions by fee-rate to make room, degrading service for honest users.

## Impact Explanation

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. The attacker pays no transaction fees (transactions are never mined) and can continuously inject unresolvable child transactions into the pool. Legitimate transactions are displaced by `limit_size` eviction. Pool metrics (`pending_count`, `gap_count`, `proposed_count`) are inflated, misleading the block assembler and operators.

## Likelihood Explanation

Any unprivileged user with RPC access (`send_transaction`) can trigger this. The timing condition — submitting children just before the parent's expiry window closes — is straightforward to achieve deliberately. `remove_expired` is invoked on every block via `_update_tx_pool_for_reorg`, so the trigger fires at every block interval. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation

Replace `pool_map.remove_entry` with `pool_map.remove_entry_and_descendants` inside `remove_expired`, matching the behavior of `limit_size`:

```rust
// tx-pool/src/pool.rs — remove_expired
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
    // was: self.pool_map.remove_entry(&entry.proposal_short_id());
    let descendants = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    for e in descendants {
        let reject = Reject::Expiry(e.timestamp);
        callbacks.call_reject(self, &e, reject);
    }
}
```

Care must be taken to skip entries already removed in a prior iteration of the loop (a child may have been collected in `removed` and also be a descendant of an earlier-processed parent).

## Proof of Concept

1. Configure a node with a short expiry (e.g., 1000 ms) for faster iteration.
2. Submit parent tx `P` via `send_transaction`. Record its pool `timestamp` as `T`.
3. Sleep until `T + expiry - 200 ms`.
4. Submit children `C1…C100`, each spending a distinct output of `P`. Their timestamps are `T + expiry - 200 ms`.
5. Sleep 300 ms (past `P`'s expiry, before children's expiry).
6. Mine a block (or wait for one). `remove_expired` fires: `P` is removed, `C1…C100` are not.
7. Call `get_transaction` on any `Ci`: status is `Pending`. Call `get_pool_info`: `pending_count` is inflated by 100.
8. Attempt to assemble a block: none of `C1…C100` can be included (inputs unresolvable).
