Audit Report

## Title
VerifyQueue Lacks Time-Based Expiry, Enabling Persistent Resource Exhaustion via P2P Relay — (`tx-pool/src/component/verify_queue.rs`)

## Summary
`VerifyQueue` has no time-based eviction mechanism. While `TxPool::remove_expired()` sweeps stale entries from `pool_map` on every block reorg, it is never called on `verify_queue`. Any P2P relay peer can fill the 256 MB verify queue with syntactically valid, high-cycle transactions that are never expired by time, blocking legitimate transaction relay indefinitely until the attacker's transactions are individually processed by the fixed worker pool.

## Finding Description
The `VerifyQueue` struct in `tx-pool/src/component/verify_queue.rs` records an `added_time` field on every `VerifyEntry` (line 43), but this field is used only for ordering — never for expiry. The struct has no TTL field and no `remove_expired` method. Its only cleanup paths are:
- `remove_tx` / `remove_txs` — explicit manual removal
- `remove_txs_by_peer` — triggered on peer disconnect
- `pop_front` — normal worker processing
- `clear` — full wipe

The sole guard against unbounded growth is a 256 MB size cap (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256_000_000` at line 18). Once reached, `add_tx` returns `Reject::Full("verify_queue total_tx_size exceeded")` for all subsequent submissions.

`_update_tx_pool_for_reorg` in `tx-pool/src/process.rs` (lines 1109–1113) calls `tx_pool.remove_expired(callbacks)` on `pool_map` and `queue.remove_txs(attached.iter()...)` (lines 855–857) to remove committed transactions from the verify queue — but performs no time-based sweep of the verify queue. Entries that have been waiting past the pool's configured `expiry` window remain in the queue indefinitely.

By contrast, `OrphanPool` in `tx-pool/src/component/orphan.rs` correctly implements time-based expiry via `ORPHAN_TX_EXPIRE_TIME` (line 15) and calls `limit_size()` on every insertion (line 158), which evicts expired entries before checking the count cap.

The exploit path:
1. A P2P relay peer calls `submit_remote_tx`, which calls `resumeble_process_tx` → `non_contextual_verify` (cheap format check) → `enqueue_verify_queue`.
2. The attacker submits many syntactically valid transactions with high declared cycle counts (so workers process them slowly).
3. `verify_queue.total_tx_size` grows toward 256 MB.
4. Once the cap is hit, all subsequent `add_tx` calls return `Reject::Full`.
5. On each new block, `_update_tx_pool_for_reorg` calls `tx_pool.remove_expired()` on `pool_map` only — the verify queue is not swept. The queue remains full.
6. Legitimate P2P relay transactions are blocked for the duration of the attack.

## Impact Explanation
This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The verify queue is the sole ingress path for P2P-relayed transactions. Filling it blocks all relay-based transaction propagation across the node without any on-chain cost to the attacker (invalid transactions are never confirmed and pay no fees). The node continues to function for consensus but is effectively isolated from the transaction relay network.

## Likelihood Explanation
The attack is reachable by any unprivileged P2P peer. No key material, special privilege, or majority hashpower is required. Filling 256 MB requires submitting hundreds to low thousands of maximum-size transactions that pass only non-contextual (format) verification. With high declared-cycle transactions stalling the two verify workers, the queue drains slowly, sustaining the DoS. The attack is repeatable: once the queue drains, it can be refilled immediately.

## Recommendation
Add a time-based expiry sweep to `VerifyQueue` mirroring `TxPool::remove_expired()`:

1. `VerifyEntry.added_time` already exists (line 43 of `verify_queue.rs`) — no new field needed.
2. Implement `VerifyQueue::remove_expired(expiry_ms: u64)` that evicts entries where `added_time + expiry_ms < unix_time_as_millis()`.
3. Call this method from `_update_tx_pool_for_reorg` in `tx-pool/src/process.rs` alongside the existing `tx_pool.remove_expired(callbacks)` call, using the same configured `expiry` duration from `TxPoolConfig`.

## Proof of Concept
1. Connect to a CKB node as a P2P peer.
2. Repeatedly relay syntactically valid transactions (passing non-contextual checks) with maximum serialized size and high declared cycle counts.
3. Observe `verify_queue.total_tx_size` growing toward 256 MB.
4. Once the cap is hit, all subsequent relay attempts return `Reject::Full("verify_queue total_tx_size exceeded")`.
5. Mine or wait for blocks: `_update_tx_pool_for_reorg` calls `tx_pool.remove_expired(callbacks)` (line 1110 of `process.rs`) but performs no expiry sweep on `verify_queue` — the queue remains full.
6. Legitimate relay peers cannot submit transactions until the attacker's transactions are individually processed by workers.
