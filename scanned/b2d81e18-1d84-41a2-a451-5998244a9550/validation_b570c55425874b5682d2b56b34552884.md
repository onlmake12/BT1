### Title
Check-After-Insert in `add_ask_for_txs` Allows `unknown_tx_hashes` Queue to Grow 84× Beyond Intended Limit with O(N) Mutex-Held Scan — (`sync/src/types/mod.rs`)

---

### Summary

`SyncState::add_ask_for_txs` inserts all peer-supplied transaction hashes into the `unknown_tx_hashes` `KeyedPriorityQueue` **before** checking the global size limit. When the global limit is exceeded but the per-peer count is still below `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, the function returns `Status::ignored()` — leaving the just-inserted hashes in the queue permanently. Simultaneously, the per-peer counting loop (lines 1516–1523) performs an O(N) full-queue scan **while holding the `unknown_tx_hashes` mutex**, creating a lock contention hotspot that worsens as the queue grows.

---

### Finding Description

**Entry point:** Any unauthenticated remote peer sends `RelayTransactionHashes` P2P messages.

**Call chain:**
```
received RelayTransactionHashes message
  → TransactionHashesProcess::execute()          [transaction_hashes_process.rs:25-50]
      → state.add_ask_for_txs(peer, tx_hashes)   [mod.rs:1483]
          → unknown_tx_hashes.lock()             [mod.rs:1484]
          → insert all hashes (up to 32767)      [mod.rs:1486-1503]
          → check global limit (AFTER insert)    [mod.rs:1507-1509]
          → O(N) per-peer scan (mutex held)      [mod.rs:1516-1523]
          → Status::ignored() if peer < limit    [mod.rs:1528]
```

**Root cause — check-after-insert:** [1](#0-0) 

All hashes are inserted unconditionally first. The global limit check only follows: [2](#0-1) 

When `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50 000) but the per-peer counter is below `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32 767), the function returns `Status::ignored()`. The inserted hashes **are not removed**. The peer is not banned. The queue retains the new entries.

**Constants driving the math:** [3](#0-2) 

- `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`
- `MAX_UNKNOWN_TX_HASHES_SIZE = 50000` (intended global cap)
- `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767`
- `MAX_RELAY_PEERS = 128` (from `sync/src/relayer/mod.rs:59`)

**Maximum reachable queue size:**

Each peer can contribute up to 32 767 unique entries before being banned. With 128 peers:

```
128 × 32767 = 4,194,176 entries  ≈ 84× the intended limit of 50,000
```

**O(N) mutex-held scan:** [4](#0-3) 

Every time a message arrives when the queue exceeds 50 000, this loop iterates over every entry in the queue — up to 4.2 M entries — while the `unknown_tx_hashes` mutex is held. All concurrent callers (other peer message handlers) block on this mutex for the duration of the scan.

**Rate limiter — mitigates but does not prevent:** [5](#0-4) 

`RelayTransactionHashes` is subject to the 30 req/sec per-(peer, message-type) rate limiter. However, each message carries up to 32 767 hashes, so a single peer can inject 32 767 entries in one message. With 128 peers at 30 req/sec, the queue fills rapidly and the O(N) scan fires at 3 840 times/sec against a 4.2 M-entry queue.

---

### Impact Explanation

1. **Memory bloat:** Queue grows to ~4.2 M entries. Each entry holds a `Byte32` key plus an `UnknownTxHashPriority` struct (containing a `Vec<PeerIndex>`, `Instant`, `bool`). Estimated heap usage: several hundred MB above the intended baseline.

2. **Lock contention / relay stall:** The `unknown_tx_hashes` mutex is held during the O(N) scan on every qualifying message. With many peers sending simultaneously, relay threads queue up waiting for the mutex. This delays `pop_ask_for_txs` (which also acquires the same mutex) and stalls the node's transaction-request scheduling, degrading transaction propagation across the network.

3. **Scope match:** Causes CKB node relay processing degradation achievable with low attacker cost (no PoW, no keys, only peer connections and unique hash generation).

---

### Likelihood Explanation

- Requires only establishing peer connections (up to `MAX_RELAY_PEERS = 128`) and sending `RelayTransactionHashes` messages with unique, previously-unseen transaction hashes.
- No cryptographic work, no stake, no privileged access required.
- The rate limiter (30 req/sec) slows but does not stop the attack; a single max-batch message already injects 32 767 entries per peer.
- The `Status::ignored()` return means the node does not disconnect or ban the peer, allowing sustained exploitation.

---

### Recommendation

1. **Check before insert:** Move the global and per-peer limit checks to **before** inserting hashes. Reject or truncate the batch if either limit would be exceeded.

2. **Maintain a per-peer counter separately:** Replace the O(N) full-queue scan with a `HashMap<PeerIndex, usize>` counter updated incrementally on insert/remove, making the check O(1).

3. **Return a ban status on global-limit breach:** When the global limit is reached, return `TooManyUnknownTransactions` (ban) rather than `Status::ignored()` to prevent repeated exploitation from the same peer.

---

### Proof of Concept

```
1. Attacker establishes P = 128 peer connections to a target CKB node.
2. Each peer generates 32767 unique Byte32 hashes (random bytes, no PoW needed).
3. Each peer sends one RelayTransactionHashes message with all 32767 hashes.
4. add_ask_for_txs inserts all 32767 hashes before checking limits.
5. Global limit (50000) is exceeded; per-peer count = 32767 = MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER.
   → Peer is banned on first message (per-peer == limit, not <).
   
   Revised: each peer sends 32766 hashes in the first message, then 1 hash per subsequent
   message. After the first message: queue = 32766, no limit triggered.
   After peer 2 first message: queue = 65532 > 50000, peer 2 per-peer = 32766 < 32767 → ignored.
   After peer 3: queue = 98298, peer 3 per-peer = 32766 < 32767 → ignored.
   ...
   After peer 128: queue ≈ 4.19M entries.

6. Each subsequent message from any peer (queue > 50000) triggers the O(4.19M) scan
   while holding the mutex, stalling all relay threads.

7. Measure: unknown_tx_hashes mutex wait time spikes; pop_ask_for_txs latency
   increases proportionally; transaction relay throughput drops.
```

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
