### Title
Zero `last_cursor` Returned on Empty `get_transactions` Result Causes Silent Pagination Restart from Offset 0 — (File: `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

---

### Summary

`AsyncRichIndexerHandle::get_transactions` initializes its pagination state variable to `0` and never updates it when the query returns no rows. The resulting `last_cursor` of all-zero bytes is silently accepted as a valid `after` parameter on the next call, causing the query to restart from the very beginning of the transaction table. This is the direct CKB analog of the reported `FindOldestPendingID()` bug: a zero default is returned when no items exist, and that zero is subsequently used as an offset, causing repeated full-table scans from offset 0.

---

### Finding Description

**Ungrouped mode** (`group_by_transaction = None | Some(false)`): [1](#0-0) 

`last_id` is initialized to `0` and `count` to `0`. When the inner query returns no rows the iterator body never executes, so both variables remain at their zero defaults. The function then serializes them unconditionally:

```rust
let mut last_cursor = last_id.to_le_bytes().to_vec();   // [0,0,0,0,0,0,0,0]
let mut offset = count.to_le_bytes().to_vec();           // [0,0,0,0]
last_cursor.append(&mut offset);                         // 12 zero bytes
```

and returns `last_cursor = JsonBytes::from_vec([0u8; 12])` inside a successful `IndexerPagination` response.

When a caller feeds this cursor back as `after`, the parsing path at lines 44–53 decodes it to `last = 0, offset = 0` and passes `Some((0, 0))` to `get_tx_with_cell`: [2](#0-1) 

This emits `WHERE tx_id >= 0 OFFSET 0`. Because auto-increment `tx_id` values start at 1, the predicate is always true and the query returns every transaction from the beginning of the table — identical to calling with `after = None`.

**Grouped mode** (`group_by_transaction = Some(true)`): [3](#0-2) 

`last_cursor` is initialized to `0i64`. On an empty result it is serialized as `[0u8; 8]`. In `get_tx_with_cells` the cursor is decoded and used as: [4](#0-3) 

`WHERE tx_id > 0` — again always true for ascending order, restarting from the first row.

**Contrast with `get_cells` (correct behavior):**

`get_cells` initializes `last_cursor` to `Vec::new()` and only populates it inside the row-mapping closure: [5](#0-4) 

An empty result therefore returns an empty byte slice. Additionally, `get_cells` uses strict `>` / `<` comparisons: [6](#0-5) 

Passing the empty cursor back would cause `decode_i64` to fail with a parse error, making the incorrect usage visible rather than silently restarting.

---

### Impact Explanation

Any RPC caller (wallet, dApp, block explorer, automated monitor) that paginates through `get_transactions` and naively feeds the returned `last_cursor` into the next call will silently restart from offset 0 whenever a page returns no results. Concretely:

- **Duplicate transaction processing**: an application tracking deposits or withdrawals re-processes every historical transaction, potentially crediting or debiting accounts multiple times.
- **Infinite loop / resource exhaustion**: a polling loop that continues while `objects` is non-empty will never terminate once the zero cursor causes the first page to be returned again.
- **Growing performance degradation**: as the transaction table grows, each spurious full-table scan takes longer, mirroring the original report's finding that "over time this can lead to performance degradation and increased processing time."

The server itself bears the cost of repeated full-table SQL scans triggered by any number of concurrent callers.

---

### Likelihood Explanation

- The `get_transactions` RPC is publicly documented and reachable by any unprivileged RPC caller.
- The API contract does not document that `last_cursor` is meaningless on an empty result; callers reasonably assume it is safe to use.
- The behavior is inconsistent with `get_cells`, which handles the empty case correctly, so developers who test with `get_cells` will not discover the issue until they switch to `get_transactions`.
- Filters with narrow `block_range` windows or rare scripts routinely return empty pages during normal operation, making accidental triggering highly likely.

---

### Recommendation

Mirror the `get_cells` pattern in both modes of `get_transactions`:

1. Initialize `last_cursor` / `last_id` to `None` or an empty `Vec`, not `0`.
2. Only populate the cursor inside the row-mapping closure, so an empty result produces an empty byte slice.
3. Switch the ungrouped mode's SQL predicate from `>=` to `>` (matching `get_cells` and the grouped mode) to eliminate the need for the offset-based deduplication that makes the zero-default dangerous.

---

### Proof of Concept

```
# 1. Call get_transactions with a script that has no indexed transactions
POST /rpc
{"method":"get_transactions","params":[{"script":{"code_hash":"0x00...00","hash_type":"data","args":"0xdeadbeef"},"script_type":"lock"},"asc","0x1",null]}

# Response: {"objects":[], "last_cursor":"0x000000000000000000000000"}  ← 12 zero bytes

# 2. Feed the zero cursor back as `after`
POST /rpc
{"method":"get_transactions","params":[{"script":{"code_hash":"0x00...00","hash_type":"data","args":"0xdeadbeef"},"script_type":"lock"},"asc","0x1","0x000000000000000000000000"]}

# Result: identical to calling with after=null — returns the first page of ALL
# transactions matching the script from tx_id=1, not an empty page or an error.
# Repeating step 2 indefinitely causes unbounded full-table scans.
```
