Audit Report

## Title
Gossip Amplification Cascade via Unknown-Target `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

When a `ConnectionRequest` names a `to` PeerId absent from the local peer registry, each receiving node independently fans out to `sqrt(N)` neighbors via `filter_broadcast`. The only cross-hop deduplication is a per-node `forward_rate_limiter` keyed on `(from, to, item_id)`, where `item_id` is the molecule union discriminant — a compile-time constant identical for every `ConnectionRequest`. An attacker rotating `(from, to)` pairs can sustain O(N · √N) network-wide messages per second from a single low-bandwidth connection, sufficient to congest the CKB P2P gossip layer.

## Finding Description

**Root cause — per-node deduplication with a constant key component**

The `forward_rate_limiter` is keyed on `(content.from, content.to, self.msg_item_id)`: [1](#0-0) 

`msg_item_id` is passed as `msg.item_id()` — the molecule union discriminant, a constant for all `ConnectionRequest` messages. The effective deduplication key is therefore `(from, to)` per node, with a quota of 1 forward per second per pair: [2](#0-1) 

**Fan-out on unknown target**

When `to` is not found in the peer registry, `forward_message` calls `filter_broadcast` to exactly `sqrt(total_peers)` neighbors: [3](#0-2) 

Each of those neighbors runs the same logic and, finding `to` unknown, fans out to another `sqrt(N)` peers.

**Route check does not prevent multi-path amplification**

`forward_request` appends the forwarding node to the route and decrements `max_hops`: [4](#0-3) 

The route check at line 128 only prevents a node from re-processing a message it already forwarded along the same path: [5](#0-4) 

When node C receives the same `(from, to)` via two different paths (B1→C with route `[R, B1]` and B2→C with route `[R, B2]`), C is absent from both routes. The first arrival is forwarded; the second is blocked by the rate limiter. But C still forwards once, contributing to the O(N · √N) total.

**Attacker throughput**

The receive-side `rate_limiter` allows 30 messages/second per `(session_id, item_id)`: [6](#0-5) 

The attacker can inject 30 unique `(from, to)` pairs per second from a single connection, each triggering a fresh cascade. `MAX_HOPS = 6` allows the cascade to propagate across the full network diameter: [7](#0-6) 

## Impact Explanation

Each node forwards a given `(from, to)` at most once per second; each forward sends to `sqrt(N)` peers. Total network-wide messages per pair: **O(N · √N)**. At 30 pairs/second from one connection:

| N | Messages/pair | Messages/second |
|---|---|---|
| 100 | ~1,000 | ~30,000 |
| 1,000 | ~31,623 | ~948,690 |
| 10,000 | ~1,000,000 | ~30,000,000 |

This is a **~31,000× amplification** at N=1,000 from a single low-bandwidth connection. The resulting message flood degrades block and transaction propagation across the CKB P2P gossip layer.

**Matched impact**: High — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation

- Requires only a standard P2P handshake — no keys, no PoW, no privileged role.
- The `to` PeerId can be any random 32-byte value guaranteed to be absent from all peer registries.
- The attacker needs one well-connected relay peer to start the cascade.
- Rotating `(from, to)` pairs is trivial and sustains the flood indefinitely within the 30 req/s per-connection cap.
- No operator intervention is possible short of disconnecting the attacker peer; the cascade is self-sustaining up to `MAX_HOPS = 6`.

## Recommendation

1. **Add a unique per-message nonce** (e.g., a random 8-byte field) to `ConnectionRequest`. Include it in the `forward_rate_limiter` key so deduplication is per-message-instance, not per `(from, to)` type.
2. **Global seen-set**: Maintain a bounded LRU set of recently-seen `(from, to, nonce)` tuples on each node; drop duplicates before forwarding regardless of arrival path.
3. **Cap unknown-target forwards per second** at the node level regardless of `(from, to)` diversity, to bound total amplification even if nonces are rotated.
4. **Reduce `MAX_HOPS`** or add exponential backoff on fan-out for unknown targets.

## Proof of Concept

```
1. Attacker connects to relay node R via standard P2P handshake.
2. Attacker sends ConnectionRequest{from=random_A, to=random_unknown, max_hops=6, listen_addrs=[valid_addr]}.
3. R: `to` not in peer_registry → filter_broadcast to sqrt(N) peers (B1..Bk).
4. Each Bi: `to` not in peer_registry, forward_rate_limiter allows (first time for (random_A, random_unknown)) → filter_broadcast to sqrt(N) more peers.
5. Repeat for 6 hops. Each node forwards at most once per (from, to), but N nodes each send sqrt(N) messages.
6. Total messages = O(N · sqrt(N)).
7. Attacker repeats with fresh (from, to) pairs up to 30 times/second.
8. Instrument a 100-node simulated network: inject 1 request, count delivered messages across all nodes → observe ~1,000 deliveries from 1 sent message.
```

The cascade is bounded per `(from, to)` by the `forward_rate_limiter`, but the attacker trivially rotates `(from, to)` pairs to sustain the flood indefinitely within the 30 req/s per-connection cap.

pub(crate) fn forward_request(
    request: packed::ConnectionRequestReader<'_>,
    current_id: &PeerId,
) -> packed::ConnectionRequest {
    let max_hops: u8 = request.max_hops().into();
    let message = request.to_entity();
    let new_route = message
        .route()
        .as_builder()
        .push(current_id.as_bytes())
        .build();
    message
        .as_builder()
        .max_hops(max_hops.saturating_sub(1))
        .route(new_route)
        .build()
}
