Audit Report

## Title
`check_tx_fee` Enforces `min_fee_rate` Using Serialized Size Instead of Transaction Weight, Allowing Cycle-Heavy Transactions to Bypass the Fee Rate Floor — (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` gates admission using only the transaction's serialized byte size as the weight argument to `FeeRate::fee()`, while the rest of the fee subsystem (fee estimation, pool ordering, eviction) uses `get_transaction_weight(size, cycles)`. A transaction with near-maximum cycles but tiny serialized size pays a fee that satisfies the size-based gate while its true weight-based fee rate is up to ~119× below `min_fee_rate`. No compensating weight-based gate exists after cycle verification completes, and the code comment at the check site explicitly acknowledges the mismatch as a known limitation.

## Finding Description

**Root cause — size used as weight in the admission gate**

In `_process_tx` (`tx-pool/src/process.rs` L715–751), the flow is:

1. `pre_check` → calls `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` — cycles are unknown at this point.
2. `verify_rtx` → VM runs, actual cycles are measured.
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles.
4. `submit_entry` — entry inserted into pool.

No weight-based fee check is performed between steps 2 and 4.

Inside `check_tx_fee` (`tx-pool/src/util.rs` L42–52):

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

`FeeRate::fee(weight)` computes `fee_rate * weight / 1000`. Passing `tx_size` (bytes) instead of `get_transaction_weight(tx_size, cycles)` means the gate is `min_fee_rate × size / 1000`, not `min_fee_rate × max(size, cycles × DEFAULT_BYTES_PER_CYCLES) / 1000`.

`get_transaction_weight` (`util/types/src/core/tx_pool.rs` L298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`.

For a crafted transaction with `tx_size = 100 bytes`, `cycles = 70_000_000`:
- True weight ≈ `max(100, 70_000_000 × 0.000_170_571_4)` ≈ **11,940**
- Fee required by `check_tx_fee`: `1_000 × 100 / 1_000 = 100 shannons`
- Fee required by weight-based gate: `1_000 × 11_940 / 1_000 = 11,940 shannons`
- Ratio: **~119× below the intended floor**

**Fee estimation pollution**

`weight_units_flow::Algorithm::accept_tx` (`util/fee-estimator/src/estimator/weight_units_flow.rs` L153–161) is called for every admitted transaction and records the entry using `get_transaction_weight` internally via `TxStatus::new_from_entry_info`. Admitted cycle-heavy, size-tiny transactions appear in the estimator's bucket distribution with very low weight-based fee rates, biasing `estimate_fee_rate` RPC output downward.

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can flood the tx pool (up to 180 MB) with cycle-heavy transactions at near-zero cost (fees ~119× below the intended floor). Each such transaction also consumes VM verification resources (up to 70M cycles per tx). Sustained submission pollutes the `weight_units_flow` fee estimator, causing honest users who follow the `estimate_fee_rate` RPC to underpay and have their transactions stall or be evicted. Pool eviction is weight-based, so these low-weight-fee-rate transactions are evicted first — but only after consuming pool space and verification queue capacity, and after being recorded in the fee estimator.

## Likelihood Explanation

The attack requires only standard `send_transaction` RPC access — no privilege, no key material, no victim interaction. The attacker writes a CKB-VM script that burns cycles in a tight loop, keeps the serialized transaction small, and pays the size-based minimum fee. The `max_tx_verify_cycles = 70_000_000` cap bounds the per-transaction amplification to ~119× but does not prevent the attack. Submitting many such transactions is cheap because the fee per transaction is near-zero relative to the true weight.

## Recommendation

After `verify_rtx` returns the actual cycle count and before `submit_entry`, add a weight-based fee rate gate:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

This check belongs in `_process_tx` (`tx-pool/src/process.rs`) between the `verify_rtx` call (L724–732) and the `TxEntry::new` construction (L751), where both `tx_size` and `verified.cycles` are available. The existing size-based check in `check_tx_fee` can remain as the cheap pre-verification gate.

## Proof of Concept

1. Write a CKB lock script that executes a tight arithmetic loop consuming ≈ 70,000,000 cycles.
2. Construct a transaction using that script; keep serialized size ≤ 200 bytes.
3. Set output capacity so that `inputs_capacity − outputs_capacity = 200 shannons` (fee).
4. Submit via `send_transaction` RPC.
5. **`check_tx_fee` gate**: `min_fee = 1_000 × 200 / 1_000 = 200 shannons ≤ 200` → **passes**.
6. VM verifies the script; cycles ≈ 70,000,000; `TxEntry` is created with real cycles.
7. `TxEntry::fee_rate()` = `200 × 1_000 / max(200, 11_940)` ≈ **16 shannons/KW** — 62× below `min_fee_rate = 1_000`.
8. No rejection occurs; the entry is admitted and fed to the fee estimator via `accept_tx`.
9. Repeat with many such transactions to shift the estimator's bucket distribution downward, causing `estimate_fee_rate` RPC to return values below what miners actually require.
