Audit Report

## Title
Missing Block-Number Bounds Check Before MMR Proof Generation Allows Unbanned Repeated InternalError — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary
`GetBlocksProofProcess::execute()` partitions requested block hashes solely by `is_main_chain()` without verifying that each block's number is ≤ `last_block.number()`. When block hashes with higher numbers are supplied, their MMR leaf positions exceed the size of the MMR constructed for `last_block`, causing `mmr.gen_proof()` to return `Err`, which is mapped to `StatusCode::InternalError` (500). Because `should_ban()` only triggers for 4xx codes, the peer is never banned and can repeat this indefinitely at zero cost.

## Finding Description
**Root cause:** In `get_blocks_proof.rs` L72-85, found block hashes are partitioned by `snapshot.is_main_chain()` and their positions are computed directly from `header.number()` via `leaf_index_to_pos(header.number())`. There is no check that `header.number() <= last_block.number()`.

**Exploit flow:**
1. Attacker sends `GetBlocksProof` with `last_hash` = hash of block at height N, and `block_hashes` = hashes of valid main-chain blocks at heights > N.
2. All four existing guards pass: non-empty check (L34-36), count limit (L38-40), `last_hash` on main chain (L45-50), no duplicates (L62-70).
3. All attacker-supplied hashes pass `is_main_chain()` and land in `found` (L72-74).
4. `leaf_index_to_pos(header.number())` produces positions beyond the MMR's range (L85).
5. In `reply_proof`, the MMR is built for `last_block.number() - 1` (L199), so `mmr.gen_proof(items_positions)` returns `Err` (L210-214), mapped to `StatusCode::InternalError`.
6. `should_ban()` checks `!(400..500).contains(&code)` — code 500 returns `None`, no ban (L95-101). The `received` handler only logs a warning (L87-88).

**Why existing checks are insufficient:** All guards validate message structure and chain membership, but none enforce the semantic constraint that requested blocks must fall within the MMR window defined by `last_hash`.

## Impact Explanation
An attacker with a valid P2P connection can repeatedly send crafted `GetBlocksProof` messages (up to 1000 hashes each) that force the server to perform up to 1000 block header lookups, MMR position computations, and a failed `gen_proof` call per message — all with no ban penalty. This constitutes a low-cost, sustained degradation of light-client serving capacity, matching the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The attack requires only a valid P2P connection and any two main-chain block hashes where one has a higher number than the other — trivially obtainable from public chain data. No special privileges, keys, or hashpower are needed. The attacker is never banned, so the attack can be sustained indefinitely from a single connection.

## Recommendation
In `GetBlocksProofProcess::execute()`, after resolving each found block header, validate that `header.number() <= last_block.number()`. Treat any violation as a protocol error:

```rust
// After: let header = snapshot.get_block_header(&block_hash)...
if header.number() > last_block.number() {
    return StatusCode::MalformedProtocolMessage
        .with_context("block hash exceeds last_block boundary");
}
```

This returns a 4xx code, which `should_ban()` will correctly convert into a peer ban.

## Proof of Concept
1. Mine a chain to height 20.
2. Connect as a light-client peer.
3. Send `GetBlocksProof` with `last_hash = block_10.hash()` and `block_hashes = [block_11.hash(), ..., block_20.hash()]`.
4. Observe: server hits the `InternalError` path in `reply_proof` (L212-214), logs `"failed to generate a proof since ..."`, and does **not** ban the peer.
5. Repeat step 3 in a tight loop — peer remains connected and unbanned indefinitely.
6. Confirm via node logs: repeated warning-level messages with no ban action taken.
