### Title
`CapacityVerifier` Skips Capacity Overflow Check for Entire Transaction When Any Input Uses DAO Type Script, Allowing Non-DAO Cells to Carry Excess Capacity — (`verification/src/transaction_verifier.rs`)

---

### Summary

The `CapacityVerifier::verify()` method unconditionally skips the `inputs_sum >= outputs_sum` check for the **entire transaction** whenever `valid_dao_withdraw_transaction()` returns `true`. That predicate fires if **any single input** carries the DAO type script. The DAO type script itself only validates DAO-specific cells (deposit/withdrawal pair and interest arithmetic); it does not enforce the global capacity balance of the transaction. An unprivileged attacker who owns a DAO withdrawal cell can therefore include arbitrary non-DAO inputs and outputs in the same transaction, with the non-DAO outputs carrying more capacity than the non-DAO inputs, and the node will accept the transaction — minting CKB capacity from thin air.

---

The guard predicate is:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
``` [2](#0-1) 

`cell_uses_dao_type_script` matches on `ScriptHashType::Type` and the consensus-level DAO code hash: [3](#0-2) 

The predicate is satisfied by the presence of **any** DAO input, regardless of how many non-DAO inputs and outputs also appear in the transaction. Once it returns `true`, the entire `inputs_sum < outputs_sum` guard is bypassed for the whole transaction.

The code comment reads:

> `// DAO withdraw transaction is verified via the type script of DAO cells` [4](#0-3) 

This comment assumes the DAO type script enforces the global capacity balance. Per RFC-0023, the DAO type script only validates:
1. That the input is a withdrawal cell (non-zero data).
2. That the output capacity equals `deposit_capacity × (AR_m / AR_0)`.

It does **not** inspect or constrain non-DAO cells in the same transaction. The assumption in the comment is therefore incorrect, and the global capacity invariant is left unguarded for any transaction that contains at least one DAO input.

The only check that still runs unconditionally is the per-output minimum-capacity check (lines 496–512), which ensures each output can hold its own data — but this does not prevent outputs from collectively exceeding inputs. [5](#0-4) 

---

### Impact Explanation

An attacker can mint CKB capacity from thin air. By constructing a DAO withdrawal transaction that also includes regular (non-DAO) inputs and outputs where the regular outputs exceed the regular inputs, the attacker gains the difference as newly created capacity. Because the `CapacityVerifier` is part of both the tx-pool admission path (`ContextualTransactionVerifier::verify`) and the block-commit verification path, a crafted transaction that passes this check will be relayed, pooled, and ultimately committed to the chain, permanently inflating the CKB supply. [6](#0-5) 

---

### Likelihood Explanation

The attacker must own a DAO withdrawal cell, which requires a prior DAO deposit. This is a low barrier: any CKB holder can deposit into the DAO, wait the required unlock period, complete step-1 withdrawal, and then craft the malicious step-2 transaction. No privileged access, no leaked keys, no majority hashpower, and no social engineering are required. The attack is fully self-contained and submittable via the standard `send_transaction` RPC.

---

### Recommendation

Replace the coarse `valid_dao_withdraw_transaction()` bypass with a cell-level accounting approach:

1. Compute `dao_inputs_capacity` = sum of capacities of inputs that carry the DAO type script.
2. Compute `dao_outputs_capacity` = sum of capacities of outputs that carry the DAO type script.
3. Compute `non_dao_inputs_capacity` = total inputs capacity − dao_inputs_capacity.
4. Compute `non_dao_outputs_capacity` = total outputs capacity − dao_outputs_capacity.
5. Enforce `non_dao_inputs_capacity >= non_dao_outputs_capacity` unconditionally.
6. Allow `dao_outputs_capacity > dao_inputs_capacity` (DAO interest), but only up to the amount validated by the DAO type script.

Alternatively, the DAO type script itself should be upgraded to assert that the sum of all non-DAO outputs does not exceed the sum of all non-DAO inputs.

### Proof of Concept

```
// Setup (prior transactions):
// TX_A: Attacker deposits 100 CKB into DAO → creates DAO deposit cell D.
// TX_B: Attacker initiates withdrawal → creates DAO withdrawal cell W (100 CKB, non-zero data).

// Malicious TX_C (DAO withdrawal step 2):
Inputs:
  [0] W  — DAO withdrawal cell, 100 CKB  (DAO type script runs, validates interest)
  [1] R  — Regular cell,        10 CKB   (attacker's own lock script passes)

Outputs:
  [0] — Regular cell, 100 + interest CKB  (DAO type script validates this output)
  [1] — Regular cell, 50 CKB              (no script validates this vs. input[1])

// valid_dao_withdraw_transaction() → true  (input[0] uses DAO type script)
// CapacityVerifier skips inputs_sum >= outputs_sum check entirely.
// DAO type script validates output[0] = 100 + interest. ✓
// No check on output[1] = 50 vs. input[1] = 10.
// Net gain: 40 CKB created from thin air.
// TX_C is accepted by tx-pool and committed to chain.
```

fn cell_uses_dao_type_script(cell_output: &CellOutput, dao_type_hash: &Byte32) -> bool {
    cell_output
        .type_()
        .to_opt()
        .map(|t| {
            Into::<u8>::into(t.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
                && &t.code_hash() == dao_type_hash
        })
        .unwrap_or(false)
}
