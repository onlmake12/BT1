### Title
Silent u128→u64 Truncating Cast in `calculate_maximum_withdraw` Causes Incorrect DAO Withdrawal Capacity - (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the maximum CKB a depositor can withdraw from NervosDAO using a u128 intermediate product, then silently truncates it to u64 with `as u64`. Every other u128→u64 conversion in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern. If the intermediate product exceeds u64::MAX, the result wraps silently to a drastically smaller value, producing a wrong withdrawal capacity with no error signal.

---

### Finding Description

In `calculate_maximum_withdraw`, the withdrawal amount is computed as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The three other u128→u64 conversions in the same file all use checked conversion:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

The variables involved:
- `counted_capacity` = `output_capacity - occupied_capacity` (u64, shannons)
- `withdrawing_ar` = accumulation rate at withdrawal block (u64, starts at `10_000_000_000_000_000`)
- `deposit_ar` = accumulation rate at deposit block (u64) [3](#0-2) 

The genesis `ar` is `10^16`. It grows monotonically. Since `withdrawing_ar > deposit_ar` always holds for a valid withdrawal, the product `counted_capacity * withdrawing_ar / deposit_ar` is always strictly greater than `counted_capacity`. If this product exceeds u64::MAX, `as u64` wraps it silently to a tiny value — no `DaoError::Overflow` is returned.

The existing test `check_withdraw_calculation_overflows` uses a near-u64::MAX capacity and expects `result.is_err()`, but the error it catches comes from the subsequent `safe_add(occupied_capacity)` overflowing, not from the `as u64` truncation itself. The truncation path is untested and unguarded. [4](#0-3) 

---

### Impact Explanation

**Two reachable call sites:**

1. **`transaction_fee` → `transaction_maximum_withdraw` → `calculate_maximum_withdraw`**: Used during block reward finalization to compute per-transaction fees for proposer rewards. A silently truncated `maximum_withdraw` causes `fee = maximum_withdraw - outputs_capacity` to underflow via `safe_sub`, rejecting a valid DAO withdrawal transaction, or to produce a wildly wrong fee that corrupts the proposer reward accounting. [5](#0-4) 

2. **RPC `calculate_dao_maximum_withdraw`**: Directly calls `calculate_maximum_withdraw` and returns the result to the caller. A truncated value causes the RPC to silently report a drastically underestimated withdrawal amount, causing wallets and users to construct incorrect withdrawal transactions. [6](#0-5) 

If truncation occurs, `withdraw_counted_capacity as u64` wraps to a value near zero (e.g., if the true u128 result is `u64::MAX + 229`, the cast yields `228`). The final `withdraw_capacity` becomes `228 + occupied_capacity` — orders of magnitude smaller than the correct value — with no error returned to the caller.

---

### Likelihood Explanation

For `withdraw_counted_capacity` to exceed u64::MAX:

```
counted_capacity × (withdrawing_ar / deposit_ar) > 18_446_744_073_709_551_615
```

The total CKB supply is ~3.36 × 10^18 shannons, so `counted_capacity ≤ 3.36 × 10^18`. Overflow requires:

```
withdrawing_ar / deposit_ar > 18.44 × 10^18 / 3.36 × 10^18 ≈ 5.49
```

This means the NervosDAO accumulation rate must grow by 5.49× from deposit to withdrawal. Given the current CKB issuance schedule, this is not reachable on mainnet in the near term. However:

- The code is **inconsistent** with every other u128→u64 conversion in the same file — all of which are checked.
- The silent `as u64` cast provides **no error signal** if it ever fires, making the failure mode invisible.
- The existing overflow test does **not** exercise this code path.
- Any future change to issuance parameters or a long-lived deposit could bring this closer to reachable.

---

### Recommendation

Replace the silent cast with the same checked conversion pattern used everywhere else in the file:

```rust
// Before (line 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
``` [7](#0-6) 

Add a dedicated test that sets `withdrawing_ar` large enough relative to `deposit_ar` and `counted_capacity` to make `withdraw_counted_capacity` exceed u64::MAX, and asserts `result == Err(DaoError::Overflow)`.

---

### Proof of Concept

**Triggering the silent truncation via the RPC path:**

1. Deposit a DAO cell with `output_capacity = 3_360_000_000_000_000_000` shannons (3.36 × 10^18, within total supply).
2. Wait (or craft a test chain) until `withdrawing_ar / deposit_ar ≥ 5.5`, i.e., `withdrawing_ar ≥ 5.5 × deposit_ar`.
3. Call `calculate_dao_maximum_withdraw` via RPC.

**Arithmetic demonstration:**

```
counted_capacity  = 3_360_000_000_000_000_000  (u64, fits)
deposit_ar        = 10_000_000_000_000_000      (genesis ar)
withdrawing_ar    = 55_000_000_000_000_000      (5.5× growth)

withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 55_000_000_000_000_000
    / 10_000_000_000_000_000
  = 18_480_000_000_000_000_000   ← exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_480_000_000_000_000_000 mod 2^64
  = 33_255_926_290_448_385       ← silently wrong, no error returned
```

The RPC returns `33_255_926_290_448_385` shannons instead of the correct `18_480_000_000_000_000_000`-equivalent, with no `DaoError::Overflow` raised. [8](#0-7) [9](#0-8)

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
