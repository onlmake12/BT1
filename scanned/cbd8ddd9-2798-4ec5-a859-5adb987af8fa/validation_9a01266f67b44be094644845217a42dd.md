### Title
Stale `EvictKey` on surviving ancestors after `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

### Summary
`remove_entry_and_descendants` pre-clears all link entries for the entire removed subtree before calling `remove_entry` on each node. Because `remove_entry` calls `update_ancestors_index_key`, which resolves ancestors via the now-cleared links, it finds an empty ancestor set and performs no updates. Surviving ancestors of the removed root retain permanently inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` fields, and a correspondingly stale `evict_key`.

### Finding Description

The comment at line 256 reveals the intent:

```
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
```

The goal was to skip redundant `update_descendants_index_key` calls for nodes within the subtree (all being removed anyway). But the implementation also silently skips `update_ancestors_index_key` for **surviving ancestors** of the removed root, because `remove_entry_links(tx_root)` removes `tx_root`'s own entry from `self.links.inner` before `remove_entry(tx_root)` calls `calc_ancestors(tx_root)`. [1](#0-0) 

`remove_entry_links` removes the node from `self.links.inner` entirely: [2](#0-1) 

`calc_ancestors` then finds no entry and returns empty: [3](#0-2) 

So `update_ancestors_index_key` iterates over an empty set and never calls `sub_descendant_weight` or refreshes `evict_key`: [4](#0-3) 

The `EvictKey` is computed from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`: [5](#0-4) 

All four fields remain inflated on surviving ancestors after the subtree is removed.

### Impact Explanation

`next_evict_entry` selects the eviction candidate by iterating `iter_by_evict_key()` in ascending order: [6](#0-5) 

`EvictKey` ordering is: lowest `fee_rate` first, then lowest `descendants_count`, then latest `timestamp`: [7](#0-6) 

A surviving ancestor whose removed descendants had high fees will retain an inflated `fee_rate` in its `evict_key`, making it appear more valuable than it is and suppressing its eviction. An attacker can exploit this to park a low-fee transaction in the pool indefinitely:

1. Submit `ancestor_tx` (low fee rate).
2. Submit `tx_root` + a large descendant chain (all high fee rate, spending `ancestor_tx`'s output).
3. `ancestor_tx.evict_key` now reflects the high-fee descendants.
4. Submit `conflict_tx` spending the same input as `tx_root` → triggers `resolve_conflict` → `remove_entry_and_descendants(tx_root)`.
5. `ancestor_tx.evict_key` is now permanently stale with inflated fee rate.
6. When the pool is full, `ancestor_tx` is never selected for eviction despite its true low fee rate.

The stale state persists for the entire remaining lifetime of `ancestor_tx` in the pool.

### Likelihood Explanation

The attack requires only standard P2P transaction relay — no privileged access, no PoW, no key material. The attacker needs to submit two sets of transactions (the chain and the conflict), both of which are ordinary valid transactions. The pool size limit bounds the descendant chain length but does not prevent the attack.

### Recommendation

In `remove_entry_and_descendants`, update surviving ancestors **before** clearing links. Specifically, for the root node `id`, call `update_ancestors_index_key` (with `EntryOp::Remove`) for each node in the subtree while links are still intact, or alternatively collect the ancestor set of `id` before any link removal and apply `sub_descendant_weight` to each surviving ancestor for every removed node.

### Proof of Concept

Invariant test (as suggested in the question): after `remove_entry_and_descendants(tx_root)`, for every remaining entry `e` that was an ancestor of `tx_root`:

```
assert_eq!(e.descendants_count, 1 + calc_descendants(e.id).len())
```

This assertion will fail because `e.descendants_count` still includes the removed subtree nodes, while `calc_descendants` (operating on the now-correct links) returns the true current set. [8](#0-7) [9](#0-8)
