### Title
Stale Inflated Descendant-Weight Accounting After `remove_entry_and_descendants` - (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all transaction links are torn down **before** `remove_entry` is called for each removed entry. Because `remove_entry` relies on `links.calc_ancestors` to find which surviving pool entries need their `descendants_*` fields decremented, pre-removing the links causes those updates to be silently skipped. Any ancestor of the removed chain that remains in the pool retains permanently inflated `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` values.

---

### Finding Description

`remove_entry_and_descendants` is the primary removal path used when a conflict is detected, a header dep becomes invalid, the pool exceeds its size limit, or a proposal is detached:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← ALL links torn down here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links are gone
        .collect()
}
``` [1](#0-0) 

Inside `remove_entry`, the ancestor update path is:

```rust
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
```

which calls:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [2](#0-1) 

Because `remove_entry_links` already removed the entry from `self.links`, `calc_ancestors` returns an empty set. No ancestor ever receives a `sub_descendant_weight` call. The comment in `remove_entry_and_descendants` acknowledges the intent to suppress `update_descendants_index_key` (to avoid updating entries that are themselves being removed), but the same suppression incorrectly also kills `update_ancestors_index_key` for entries that are **not** being removed.

`sub_descendant_weight` decrements all four descendant counters using the individual entry's own `size`, `cycles`, and `fee`:

```rust
pub fn sub_descendant_weight(&mut self, entry: &TxEntry) {
    self.descendants_count = self.descendants_count.saturating_sub(1);
    self.descendants_size = self.descendants_size.saturating_sub(entry.size);
    self.descendants_cycles = self.descendants_cycles.saturating_sub(entry.cycles);
    self.descendants_fee = Capacity::shannons(
        self.descendants_fee.as_u64().saturating_sub(entry.fee.as_u64()),
    );
}
``` [3](#0-2) 

These calls never happen for surviving ancestors, so their `descendants_*` fields remain permanently inflated for the lifetime of those entries in the pool.

---

### Impact Explanation

The `descendants_*` fields directly drive two pool-management decisions:

1. **Eviction ordering** — `EvictKey` is computed as `max(descendants_feerate, own_feerate)`. An ancestor with artificially inflated `descendants_fee` / `descendants_size` will show a higher `descendants_feerate` than it actually has, making it appear more valuable and **less likely to be evicted** when the pool is full. [4](#0-3) 

2. **Score sorting** — `AncestorsScoreSortKey` uses `ancestors_fee` / `ancestors_weight` for mining priority. Descendants of the stale ancestor will inherit incorrect ancestor-fee values, distorting block-template construction. [5](#0-4) 

The net result is that a transaction with stale inflated descendant stats occupies pool space it should have been evicted from, displacing legitimate higher-fee transactions.

---

### Likelihood Explanation

`remove_entry_and_descendants` is triggered by every common pool-management event:

- `resolve_conflict` — fires whenever a committed block contains a transaction that conflicts with a pool entry (routine on every block).
- `resolve_conflict_header_dep` — fires on every new block that invalidates a header dep.
- `limit_size` — fires whenever the pool exceeds `max_tx_pool_size`.
- `remove_by_detached_proposal` — fires on every reorg. [1](#0-0) [6](#0-5) 

Any unprivileged transaction submitter can deliberately trigger this path: submit a parent transaction P and a child transaction C, then submit a second transaction that spends the same input as C. When the conflict is resolved, `remove_entry_and_descendants(C)` is called, P's `descendants_*` fields are never decremented, and P retains inflated eviction resistance for the rest of its time in the pool.

---

### Recommendation

Move the ancestor-weight update **before** the links are torn down, or perform a targeted update of surviving ancestors separately. One approach:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update surviving ancestors' descendant weights BEFORE removing links
    for rid in &removed_ids {
        if let Some(entry) = self.get(rid).cloned() {
            self.update_ancestors_index_key(&entry, EntryOp::Remove);
        }
    }

    // Now safe to remove links (descendants being removed don't need updates)
    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry_without_ancestor_update(id))
        .collect()
}
```

Alternatively, suppress only `update_descendants_index_key` inside `remove_entry` when called from a bulk-removal context, while preserving `update_ancestors_index_key`.

---

### Proof of Concept

Consider three pool transactions: **A** (no parents) → **B** (child of A) → **C** (child of B).

After insertion, A's state includes:
- `descendants_count = 3`, `descendants_fee = fee_A + fee_B + fee_C`

Call `remove_entry_and_descendants(B)` (removes B and C):

1. `remove_entry_links(B)` — removes B from links; A no longer has B as a child.
2. `remove_entry_links(C)` — removes C from links.
3. `remove_entry(B)` — calls `update_ancestors_index_key(B, Remove)` → `calc_ancestors(B)` returns `{}` (links gone) → **A is never updated**.
4. `remove_entry(C)` — same; A still not updated.

After removal, A still shows `descendants_count = 3` and `descendants_fee = fee_A + fee_B + fee_C`, even though B and C are gone. A's `EvictKey` is computed using the stale inflated `descendants_feerate`, making A appear more valuable than it is and preventing correct eviction. [7](#0-6) [8](#0-7)

        let ancestors: HashSet<ProposalShortId> =

        for anc_id in &ancestors {
            // update parent score
