### Title
Pre-Decompression Buffer Amplification via Attacker-Controlled Snappy Header in `LengthDelimitedCodecWithCompress::decode` — (`network/src/compress.rs`)

---

### Summary

In `LengthDelimitedCodecWithCompress::decode`, the node allocates a zeroed buffer sized by the **attacker-controlled** snappy header varint before verifying that the actual compressed payload can produce that many bytes. An unprivileged remote peer can send a tiny wire frame (~10 bytes) that causes the victim node to allocate up to ~8 MB per message. With many concurrent sessions, this produces a large memory amplification attack.

---

### Finding Description

The decode path in `LengthDelimitedCodecWithCompress::decode` is:

1. The `length_delimited` inner codec reads the wire frame and enforces `max_frame_length` on the **compressed** wire bytes.
2. `decompress_len(&data[1..])` reads the uncompressed-length varint from the snappy stream header — a value entirely under attacker control.
3. The result is checked against `MAX_UNCOMPRESSED_LEN` with a **strict greater-than** (`>`), so any value up to `8MB - 1` passes.
4. `BytesMut::zeroed(decompressed_bytes_len)` allocates up to ~8 MB **before** any decompression is attempted.
5. `SnapDecoder::new().decompress(...)` is called; if the actual payload does not match the claimed size, it returns an error and the buffer is dropped — but the allocation already occurred. [1](#0-0) 

The `MAX_UNCOMPRESSED_LEN` constant is `1 << 23` (8,388,608 bytes = 8 MB): [2](#0-1) 

The guard uses `>` not `>=`, so a header claiming exactly `8,388,607` bytes passes: [3](#0-2) 

The allocation at line 242 is unconditional on the header value: [4](#0-3) 

The `max_frame_length` check enforced by the inner codec applies only to the **compressed wire frame**, not the claimed decompressed size: [5](#0-4) 

Per-protocol wire limits (the attacker's tiny frame easily fits within all of them): [6](#0-5) 

---

### Impact Explanation

**Amplification ratio:** A valid snappy stream with a header varint claiming 8,388,607 bytes requires only ~4–6 bytes for the varint plus a minimal snappy literal block — roughly 10–20 bytes on the wire. The node allocates ~8 MB in response. That is a ~400,000× memory amplification per message.

**Multi-session scaling:** With `max_connection_number = 1024` concurrent sessions (each running in its own tokio task, potentially executing on separate OS threads simultaneously), and each session continuously sending such frames, the peak concurrent allocation can reach `1024 × 8 MB ≈ 8 GB`, causing OOM and node crash.

**No authentication required:** Any peer that completes the P2P handshake (no PoW, no stake, no key) can open a Sync or RelayV3 session and immediately send crafted frames.

---

### Likelihood Explanation

- The attack requires only a standard TCP connection and knowledge of the snappy framing format — both trivially available.
- The snappy varint header is not authenticated or bounded relative to the compressed payload size anywhere before the allocation.
- The strict `>` comparison (instead of `>=`) means the maximum allowed allocation is `MAX_UNCOMPRESSED_LEN - 1`, not zero — the off-by-one is not protective.
- The `Message::decompress` path (used elsewhere) has the same pattern with `vec![0; decompressed_bytes_len]`: [7](#0-6) 

---

### Recommendation

1. **Bound allocation relative to wire size:** Before allocating, verify that `decompressed_bytes_len` is plausible given the compressed input size. Snappy's maximum compression ratio is ~6×, so `decompressed_bytes_len > compressed_len * 8` (with some slack) should be rejected.
2. **Use `>=` instead of `>`** in the guard: `if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN` to close the off-by-one.
3. **Consider per-connection message rate limiting** to bound the rate at which a single peer can trigger allocations.
4. Apply the same fix to `Message::decompress` at line 81.

### Proof of Concept

```python
import socket, struct

# Snappy stream: varint claiming 8,388,607 decompressed bytes,
# followed by a minimal literal block (e.g., 1 byte literal "A")
# Varint encoding of 8388607 = 0xFF 0xFF 0xFF 0x03 (little-endian base-128)
snappy_header = bytes([0xFF, 0xFF, 0xFF, 0x03])  # varint = 8388607
# Minimal snappy literal: tag byte 0x00 (literal, length 1) + 'A'
snappy_body = bytes([0x00, 0x41])
snappy_payload = snappy_header + snappy_body

# CKB compress frame: flag byte 0x80 (COMPRESS_FLAG) + snappy payload
ckb_frame = bytes([0x80]) + snappy_payload

# Length-delimited framing: 4-byte big-endian length prefix
wire = struct.pack(">I", len(ckb_frame)) + ckb_frame

# Send to target node on any protocol port (e.g., Sync/RelayV3)
s = socket.create_connection(("TARGET_NODE", 8115))
# ... complete tentacle handshake, then:
for _ in range(10000):  # flood with crafted frames
    s.sendall(wire)
# Each frame causes ~8MB allocation on the victim; measure RSS growth.
```

Each received frame causes `BytesMut::zeroed(8_388_607)` to execute on the victim node before the decompression failure is detected and the buffer is freed. With 1024 concurrent connections each sending at line rate, peak RSS reaches ~8 GB.
