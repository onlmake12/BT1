### Title
Unbounded Dead-Cell NOT IN Clause Exceeds SQLite Bind-Parameter Limit in `get_cells` â€” (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

### Summary
When `index_tx_pool = true` and the SQLite backend is used, `get_cells` (and `get_cells_capacity`) dynamically construct a `NOT IN ($1, $2, â€¦, $N)` clause with one bind parameter per dead cell in the pool. SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is 999. If the pool contains â‰Ą 1000 distinct dead cells, every subsequent call to `get_cells` returns a DB-level error to the caller until the pool drains.

### Finding Description
In `get_cells`, all dead cells are collected from the pool without any count bound:

```rust
dead_cells = pool.dead_cells()
    .map(|out_point| { â€¦ })
    .collect::<Vec<(_, u32)>>()
```

Then one `$N` placeholder is emitted per entry:

```rust
let placeholder = format!("(${}, {})", param_index, output_index);
param_index += 1;
```

And one `.bind(tx_hash)` call is made per entry. When `dead_cells.len() > 999`, SQLite rejects the query with `SQLITE_RANGE` (too many bind variables). The error propagates as `Error::DB(â€¦)` to the RPC caller.

The codebase already recognises this limit in `insert.rs` (`BATCH_SIZE_THRESHOLD = 1_000` with an explicit comment about database bind limits), but the same guard was never applied to the query path.

### Impact Explanation
Any caller of the `get_cells` or `get_cells_capacity` RPC receives a DB error for the duration that the pool holds â‰Ą 1000 dead cells. The node itself does not crash, but the indexer RPC endpoints are effectively non-functional during that window.

### Likelihood Explanation
Requires: (1) operator has enabled `RichIndexer` module and `index_tx_pool = true` (non-default but a documented production feature); (2) attacker controls â‰Ą 1000 live cells on-chain and submits transactions spending them. The 180 MB pool size limit imposes no transaction-count ceiling that would prevent this. On a testnet the cost is negligible; on mainnet it requires real CKB but is not prohibitive.

### Recommendation
Before constructing the `NOT IN` clause, chunk `dead_cells` into batches of at most 999 (matching `BATCH_SIZE_THRESHOLD`) and either issue multiple queries or use a temporary table / CTE approach. Apply the same fix to `get_cells_capacity`.

### Proof of Concept
Populate the indexer `Pool` with 1000 synthetic `OutPoint` entries, call `get_cells` against a SQLite-backed `AsyncRichIndexerHandle`, and assert the call returns `Error::DB(â€¦)` rather than a successful result. The existing `BATCH_SIZE_THRESHOLD` constant and its comment in `insert.rs` confirm the developers are aware of the limit; the omission in the query path is the root cause.
