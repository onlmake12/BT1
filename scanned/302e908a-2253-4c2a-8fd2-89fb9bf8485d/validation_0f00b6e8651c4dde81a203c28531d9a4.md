### Title
Missing `BLOCK_INVALID` Status Check in Block Fetcher Allows Repeated Re-Download of Persistently Invalid Blocks — (File: `sync/src/synchronizer/block_fetcher.rs`)

---

### Summary

The `BlockFetcher::fetch()` function in the sync module does not check for `BlockStatus::BLOCK_INVALID` when deciding whether to request a block from a peer. As a result, a block that has already been downloaded, stored, and found to be permanently invalid (e.g., failed script execution) will be repeatedly re-requested and re-verified on every fetch cycle, wasting bandwidth and CPU resources indefinitely.

---

### Finding Description

In `sync/src/synchronizer/block_fetcher.rs`, the inner loop that selects which blocks to download checks only two statuses before adding a block to the in-flight download queue:

```rust
if status.contains(BlockStatus::BLOCK_STORED) {
    // already on chain, skip
    break;
} else if status.contains(BlockStatus::BLOCK_RECEIVED) {
    // already in-flight or orphan pool, skip
} else if (matches!(self.ibd, IBDState::In)
    || state.compare_with_pending_compact(&hash, now))
    && state
        .write_inflight_blocks()
        .insert(self.peer, (header.number(), hash).into())
{
    fetch.push(header)   // ← block is queued for GetBlocks
}
``` [1](#0-0) 

`BlockStatus::BLOCK_INVALID` is defined as `1 << 12`, a completely disjoint bit from `BLOCK_STORED` (`0b111`) and `BLOCK_RECEIVED` (`0b11`): [2](#0-1) 

Therefore, a block whose status is `BLOCK_INVALID` satisfies **neither** guard condition and falls through to the `else if` branch, causing it to be inserted into `InflightBlocks` and included in a `GetBlocks` request.

The full block verification pipeline marks a block `BLOCK_INVALID` in `block_status_map` when contextual verification fails (e.g., invalid script execution, bad cellbase reward). The block body remains stored in the database. On the next `BlockFetcher` poll, `get_block_status` returns `BLOCK_INVALID` from the map (or from `block_ext.verified = Some(false)` in the DB after a restart), but the fetcher ignores this and re-queues the block. [3](#0-2) 

A related `FIXME` comment in `headers_process.rs` acknowledges that `BLOCK_INVALID` is not checked in the header acceptance path either, indicating awareness of the gap: [4](#0-3) 

---

### Impact Explanation

An attacker who connects as a peer and advertises a chain containing a block with a **valid header** (passes PoW and header checks) but an **invalid body** (e.g., a script that always fails) causes the victim node to:

1. Accept the header and set `best_known_header` to the attacker's chain tip.
2. Download the block body, store it in the DB, and fail contextual verification → `BLOCK_INVALID`.
3. On the next `BlockFetcher` poll (every ~30 seconds via `BLOCK_DOWNLOAD_TIMEOUT`), re-queue the same block for download because `BLOCK_INVALID` is not filtered.
4. Repeat indefinitely, consuming bandwidth (re-downloading the block body) and CPU (re-running CKB-VM script verification).

If the attacker's chain contains multiple such blocks, or if multiple attacker peers advertise the same invalid chain, the wasted resources multiply. The `InflightBlocks` limit (`MAX_BLOCKS_IN_TRANSIT_PER_PEER = 128`) bounds the concurrency but not the repetition over time. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The entry path is fully reachable by any unprivileged P2P peer:

- The attacker only needs to connect as a standard sync peer and send `SendHeaders` followed by `SendBlock`.
- Crafting a block with a valid header but invalid body (e.g., a lock script that always aborts) requires no special privilege — only the ability to mine a block, which is the normal PoW process.
- The attacker peer is banned after the first invalid block is verified, but the `block_status_map` entry and the stored block body persist. Any subsequent peer (or a reconnecting attacker with a different identity) that advertises the same chain tip will trigger the same re-download loop, because the `BlockFetcher` does not consult `BLOCK_INVALID` status.
- The attack is low-cost for the attacker (one block mined once) and continuous for the victim.

---

### Recommendation

Add an explicit `BLOCK_INVALID` guard in the `BlockFetcher::fetch` inner loop, analogous to the existing `BLOCK_STORED` and `BLOCK_RECEIVED` guards:

```rust
if status.contains(BlockStatus::BLOCK_STORED) {
    // ...
    break;
} else if status.contains(BlockStatus::BLOCK_RECEIVED) {
    // Do not download repeatedly
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    // Do not re-download persistently invalid blocks
    break;
} else if ... {
    fetch.push(header)
}
```

This mirrors the check already present in `compact_block_process.rs` and `headers_process.rs`, which both return early on `BLOCK_INVALID`: [7](#0-6) 

---

### Proof of Concept

1. Attacker mines a block `B` at height `N` with a valid header (correct PoW, correct parent hash, valid timestamp) but an invalid body (e.g., a lock script cell that always exits with code 1).
2. Attacker connects to victim node as a sync peer and sends `SendHeaders` for the chain `[B, B+1, ...]` where all headers are structurally valid.
3. Victim's `HeadersProcess::accept()` validates the headers and sets `best_known_header` to the attacker's chain tip.
4. Victim's `BlockFetcher::fetch()` traverses from `best_known` backwards, finds block `B` with status `BLOCK_UNKNOWN` (not yet downloaded), inserts it into `InflightBlocks`, and sends `GetBlocks`.
5. Attacker responds with block `B`. Victim stores it in the DB (`BLOCK_STORED` transiently), then `ConsumeUnverifiedBlocks` runs the CKB-VM and fails → `BLOCK_INVALID` written to `block_status_map`. Attacker peer is banned.
6. On the next `BlockFetcher` poll (after `BLOCK_DOWNLOAD_TIMEOUT` = 30 s), `get_block_status(B)` returns `BLOCK_INVALID`. The fetcher checks `BLOCK_STORED` → false; `BLOCK_RECEIVED` → false; falls through to `write_inflight_blocks().insert(...)` → true. Block `B` is re-queued.
7. A `GetBlocks` message is sent to any peer that claims the same chain. If the attacker reconnects (or another peer has the block), the block is re-downloaded and re-verified, failing again.
8. Steps 6–7 repeat every ~30 seconds indefinitely. [8](#0-7) [9](#0-8)

    pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
        match self.block_status_map().get(block_hash) {
            Some(status_ref) => *status_ref.value(),
            None => {
                if self.header_map().contains_key(block_hash) {
                    BlockStatus::HEADER_VALID
                } else {
                    let verified = self
                        .snapshot()
                        .get_block_ext(block_hash)
                        .map(|block_ext| block_ext.verified);
                    match verified {
                        None => BlockStatus::UNKNOWN,
                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,
                    }
                }
            }
        }

    pub fn prune(&mut self, tip: BlockNumber) -> HashSet<PeerIndex> {
        let now = unix_time_as_millis();
        let mut disconnect_list = HashSet::new();
        // Since statistics are currently disturbed by the processing block time, when the number
        // of transactions increases, the node will be accidentally evicted.
        //
        // Especially on machines with poor CPU performance, the node connection will be frequently
        // disconnected due to statistics.
        //
        // In order to protect the decentralization of the network and ensure the survival of low-performance
        // nodes, the penalty mechanism will be closed when the number of download nodes is less than the number of protected nodes
        let should_punish = self.download_schedulers.len() > self.protect_num;
        let adjustment = self.adjustment;

        let trace = &mut self.trace_number;
        let download_schedulers = &mut self.download_schedulers;
        let states = &mut self.inflight_states;

        let mut remove_key = Vec::new();
        // Since this is a btreemap, with the data already sorted,
        // we don't have to worry about missing points, and we don't need to
        // iterate through all the data each time, just check within tip + 20,
        // with the checkpoint marking possible blocking points, it's enough
        let end = tip + 20;
        for (key, value) in states.iter() {
            if key.number > end {
                break;
            }
            if value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now {
                if let Some(set) = download_schedulers.get_mut(&value.peer) {
                    set.hashes.remove(key);
                    if should_punish && adjustment {
                        set.punish(2);
                    }
                };
                if !trace.is_empty() {
                    trace.remove(key);
                }
                remove_key.push(key.clone());
                debug!(
                    "prune: remove InflightState: remove {}-{} from {}",
                    key.number, key.hash, value.peer
                );

                if let Some(metrics) = ckb_metrics::handle() {
                    metrics.ckb_inflight_timeout_count.inc();
                }
            }
        }

        for key in remove_key {
            states.remove(&key);
        }

        download_schedulers.retain(|k, v| {
            // task number zero means this peer's response is very slow
            if v.task_count == 0 {
                disconnect_list.insert(*k);
                false
            } else {
                true
            }
        });
