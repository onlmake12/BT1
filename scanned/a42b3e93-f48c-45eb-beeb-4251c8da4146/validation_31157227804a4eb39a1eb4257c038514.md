Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Post-Eviction `total_tx_size`/`total_tx_cycles` in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the new pool-size totals are captured into local variables before `check_and_record_ancestors` runs. That inner call can evict transactions and correctly decrements `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`, but the stale pre-eviction locals are then unconditionally written back, permanently inflating the counters by exactly the size of the evicted transactions. This causes `limit_size` to spuriously evict valid transactions and causes future submissions to be prematurely rejected with `Reject::Full`.

## Finding Description
The exact sequence in `add_entry` (lines 210–219 of `tx-pool/src/component/pool_map.rs`):

```rust
// Step 1: capture S + new_size into locals, before any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // locals = S + new_size

// Step 2: may evict cell_ref_parent txs; each eviction calls
//         update_stat_for_remove_tx → self.total_tx_size -= evicted_size
//         so self.total_tx_size is now S - evicted_size
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3: overwrite with stale pre-eviction value
self.total_tx_size  = total_tx_size;   // written as S + new_size  ← BUG
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) simply reads `self.total_tx_size`, adds the new entry's size, and returns the result as a local — it does not mutate state. `check_and_record_ancestors` (lines 615–625) calls `remove_entry_and_descendants` when ancestor count exceeds `max_ancestors_count`, which calls `remove_entry` (line 247), which calls `update_stat_for_remove_tx` — correctly decrementing `self.total_tx_size`. Step 3 then blindly overwrites that decremented value with the stale local, inflating the counter by `evicted_size`. The correct post-operation value should be `S + new_size - evicted_size`; the actual value is `S + new_size`.

Existing guards do not prevent this: `updated_stat_for_add_tx` only checks for integer overflow, not for the ordering problem. There is no re-read of `self.total_tx_size` after evictions complete.

## Impact Explanation
`total_tx_size` is the sole counter driving `limit_size` (pool.rs lines 298–328), which loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts the lowest-fee-rate transactions. An inflated counter causes two concrete effects:

1. **Spurious evictions**: `limit_size` is called immediately after `add_entry` returns. It sees an inflated `total_tx_size > max_tx_pool_size` and evicts additional valid, fee-paying transactions that should have remained in the pool.
2. **Premature rejection**: Subsequent calls to `updated_stat_for_add_tx` start from an already-inflated baseline, causing valid transactions to be rejected with `Reject::Full` even when the pool has real capacity.

An attacker who can repeatedly trigger the eviction path can continuously starve the mempool of legitimate transactions at low cost, matching the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The eviction path in `check_and_record_ancestors` is triggered when a new transaction has `cell_ref_parents` (in-pool transactions that use one of its cell deps) and the total ancestor count including those parents exceeds `max_ancestors_count` (default 25). This condition is reachable by any unprivileged `send_transaction` RPC caller or relay peer. An attacker can observe mempool state via `get_raw_tx_pool`, construct a chain of 24 transactions sharing a cell dep, submit a cell-dep-referencing transaction `R`, then submit a new transaction that pushes the ancestor count over 25, reliably triggering the bug. The attack is repeatable and requires no special privileges.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so eviction-driven decrements are already reflected in `self.total_tx_size` before the new entry's contribution is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Re-read post-eviction totals, then add only the new entry's contribution
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, remove the local variables entirely and apply the increment in-place after all mutations complete.

## Proof of Concept
1. Fill the pool with a chain of 24 transactions `T1 → T2 → … → T24` where each spends the previous output.
2. Submit transaction `R` that takes `T1`'s output as a **cell dep** (making `R` a `cell_ref_parent`).
3. Submit `T_new` that spends `T1`'s output. `T_new` now has 24 ancestors + `R` as a cell-ref parent, exceeding `max_ancestors_count = 25`.
4. `check_and_record_ancestors` evicts `R` (size `S_R`) via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by `S_R`.
5. Step 3 of `add_entry` overwrites `self.total_tx_size` with the pre-eviction value, inflating it by `S_R`.
6. Verify via `get_tip_tx_pool_info` that `total_tx_size` exceeds the sum of actual entry sizes, and observe that `limit_size` subsequently evicts a valid transaction that fits within `max_tx_pool_size`.
7. A unit test can assert `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum()` after the above sequence to confirm the invariant is violated.
