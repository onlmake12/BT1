Audit Report

## Title
Unbounded `GetBlocksProof` Request Rate with O(k·log(N)) MMR DB Reads Enables DoS on Light Client Servers — (`util/light-client-protocol-server/src/lib.rs`, `util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

`LightClientProtocol` contains no per-peer rate limiter, unlike `Relayer` which explicitly enforces one. Any peer can repeatedly send `GetBlocksProof` messages containing up to 1000 valid block hashes, each triggering O(k·log(N)) RocksDB reads via `mmr.gen_proof(items_positions)`. At sufficient request rate from even a single peer, this saturates the node's storage I/O and degrades or halts normal chain processing.

## Finding Description

**Struct definition — no rate limiter field:**
`LightClientProtocol` holds only `shared: Shared`. [1](#0-0) 

**`received` dispatches directly to `try_process` with no rate check:** [2](#0-1) 

**Per-request limit is 1000 hashes:** [3](#0-2) 

The limit check in `GetBlocksProofProcess::execute()` rejects messages exceeding 1000 hashes but imposes no rate constraint on how frequently valid requests arrive: [4](#0-3) 

After validation, `execute()` calls `reply_proof` with up to 1000 MMR leaf positions: [5](#0-4) 

Inside `reply_proof`, `mmr.gen_proof(items_positions)` is called unconditionally: [6](#0-5) 

The `MMRStore` implementation for `&Snapshot` issues one RocksDB read per MMR node accessed: [7](#0-6) 

Standard MMR proof generation for k leaves in a tree of height N requires O(k·log(N)) node reads. At chain height ~10M, log₂(10M) ≈ 23, yielding ~23,000 RocksDB reads per max-size request.

**Contrast with `Relayer`**, which explicitly adds a per-peer, per-message-type rate limiter capped at 30 req/s: [8](#0-7) 

The rate check is enforced before any processing in `Relayer::try_process`: [9](#0-8) 

No equivalent guard exists anywhere in `LightClientProtocol::try_process` or `received`. [10](#0-9) 

## Impact Explanation

A single attacker peer sending `GetBlocksProof` at even 10 req/s with 1000 spread-out hashes forces ~230,000 RocksDB reads/second. This is sufficient to saturate I/O on a typical server, causing the node to become unresponsive or crash. The cost grows with chain height (log factor), making the attack progressively cheaper as the chain matures. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node (10001–15000 points)**.

## Likelihood Explanation

The attack requires only a valid P2P connection to a node running the light client server and knowledge of 1000 main-chain block hashes, which are trivially obtainable from any public block explorer or by syncing headers. No PoW, no keys, no privileged access is needed. The attack is repeatable indefinitely from a single peer.

## Recommendation

Add a per-peer rate limiter to `LightClientProtocol` analogous to the one in `Relayer`:

```rust
type RateLimiter<T> = governor::RateLimiter<
    T,
    governor::state::keyed::HashMapStateStore<T>,
    governor::clock::DefaultClock,
>;

pub struct LightClientProtocol {
    pub shared: Shared,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Enforce it at the top of `try_process` before dispatching to any handler. Additionally, consider reducing `GET_BLOCKS_PROOF_LIMIT` from 1000 or adding a global request queue with backpressure for the light client protocol.

## Proof of Concept

1. Connect to a mainnet CKB node with the light client server enabled via P2P (`/ckb/lightclient`).
2. Obtain 1000 valid main-chain block hashes spread across the full chain height range (e.g., blocks 0, 10000, 20000, …, 9,990,000) from a public block explorer.
3. In a tight loop, send `GetBlocksProof { last_hash: <current tip>, block_hashes: [1000 hashes] }`.
4. Monitor server RocksDB I/O (e.g., via `iostat` or RocksDB metrics); observe saturation.
5. To confirm O(k·log(N)) scaling, benchmark `gen_proof(1000 positions)` on chains of height 100, 10,000, and 1,000,000 and verify cost grows with log(N).
