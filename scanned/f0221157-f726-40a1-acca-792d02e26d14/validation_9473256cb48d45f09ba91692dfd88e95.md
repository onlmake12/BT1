Audit Report

## Title
RBF Replacement Always Resets Pool Entry Timestamp, Enabling Indefinite Expiry Bypass - (`tx-pool/src/process.rs`, `tx-pool/src/pool.rs`, `tx-pool/src/component/entry.rs`)

## Summary
When a transaction is replaced via RBF, the new `TxEntry` is constructed with `TxEntry::new(...)` which unconditionally stamps the current wall-clock time as the entry's `timestamp`. No mechanism preserves or inherits the original entry's timestamp. An attacker can repeatedly replace their own transaction just before the `expiry_hours` window closes, resetting the expiry clock indefinitely at near-zero marginal cost, permanently defeating the `remove_expired` eviction mechanism and occupying pool slots and UTXOs indefinitely.

## Finding Description
`TxEntry::new` always calls `unix_time_as_millis()` for the timestamp:
```rust
// tx-pool/src/component/entry.rs L48-50
pub fn new(rtx: Arc<ResolvedTransaction>, cycles: Cycle, fee: Capacity, size: usize) -> Self {
    Self::new_with_timestamp(rtx, cycles, fee, size, unix_time_as_millis())
}
```
In `_process_tx` (`process.rs:751`), the new entry is always created via `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — no timestamp from the conflicting entry is retrieved or forwarded. `process_rbf` (`process.rs:190-235`) removes the old entries but discards their timestamps entirely. `_submit_entry` then inserts the fresh entry. The `remove_expired` check in `pool.rs:277` is `self.expiry + entry.inner.timestamp < now_ms`, so the new entry's deadline is always `now + expiry_hours`, regardless of how long the replaced entry had already been in the pool.

The minimum replacement fee is `sum(replaced_txs.fee) + min_rbf_rate.fee(size)` (`pool.rs:103-113`). With `min_rbf_rate = 1500 shannons/KB` and a ~200-byte tx, each replacement costs ~300 shannons (~0.000003 CKB). Fees grow linearly across replacements, but remain negligible in absolute terms (e.g., 100 replacements covering ~50 days costs ~0.015 CKB total).

`check_rbf` (`pool.rs:574-679`) validates fee rules and descendant limits but imposes no constraint on the original entry's age or timestamp — it does not carry the old timestamp forward.

## Impact Explanation
This maps to **High: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker can occupy tx-pool slots indefinitely at near-zero cost, defeating the expiry eviction mechanism designed to reclaim pool resources. Combined with the pool size limit, an attacker running multiple such chains can crowd out legitimate transactions from the pool. Additionally, any UTXO consumed by the attacker's transaction is kept locked in the pool indefinitely, requiring any third party to continuously outbid the attacker — who can always respond with another cheap replacement.

## Likelihood Explanation
- Triggerable by any unprivileged user via the `send_transaction` RPC or P2P relay.
- Requires only knowledge of the publicly documented default `expiry_hours = 12` and `min_rbf_rate`.
- RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is an opt-in node configuration, but nodes that enable it are fully exposed.
- Cost is ~300 shannons per 12-hour cycle, growing linearly but remaining negligible indefinitely.
- No special knowledge, victim mistakes, or external dependencies required.

## Recommendation
Preserve the original entry's timestamp when a transaction is replaced via RBF. In `process_rbf` or `submit_entry`, retrieve the minimum (oldest) `timestamp` among all conflicting entries being evicted, then pass it to `TxEntry::new_with_timestamp(...)` for the replacement entry. The `new_with_timestamp` constructor already exists (`entry.rs:52-75`) and accepts an explicit timestamp — it simply needs to be called with the inherited timestamp instead of `TxEntry::new` in the RBF replacement path (`process.rs:751`).

## Proof of Concept
1. Configure node: `min_fee_rate = 1000`, `min_rbf_rate = 1500`, `expiry_hours = 12`.
2. Submit **T1** spending UTXO `[A]` with fee 1000 shannons/KB. Pool records `T1.timestamp = T0`.
3. At `T0 + 11h55m`, submit **T2** spending the same UTXO `[A]` with fee ≥ 1500 shannons/KB. `check_rbf` passes; `process_rbf` removes T1; `_submit_entry` inserts T2 with `T2.timestamp = T0 + 11h55m`.
4. `remove_expired` at `T0 + 12h`: T1 is already gone; T2's deadline is `T0 + 11h55m + 12h` — safely in the future.
5. Repeat step 3 every ~11.9 hours. UTXO `[A]` remains locked in the pool indefinitely. The attacker's total cost after N cycles is approximately `N*(N+1)/2 * 300` shannons — negligible for any practical duration.
6. Verification: query pool state via `get_pool_tx_detail_info` RPC after each replacement to confirm the timestamp resets and the entry never expires.
