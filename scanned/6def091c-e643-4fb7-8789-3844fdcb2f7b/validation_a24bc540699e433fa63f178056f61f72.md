### Title
DAO Withdrawal Bypasses Capacity Conservation for Non-DAO Outputs, Enabling CKB Creation from Thin Air - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::verify()` unconditionally skips the `OutputsSumOverflow` check for any transaction that contains at least one DAO-type input. The DAO type script only enforces the capacity of DAO-specific output cells. Non-DAO outputs in the same transaction are never checked against non-DAO inputs. An attacker who holds a DAO deposit can craft a phase-2 withdrawal transaction that includes regular inputs and outputs where outputs exceed inputs, creating CKB capacity from thin air.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `CapacityVerifier::verify()` skips the global `OutputsSumOverflow` check whenever `valid_dao_withdraw_transaction()` returns `true`: [1](#0-0) 

`valid_dao_withdraw_transaction()` uses `.any()` — it returns `true` if **any single input** carries the DAO type script: [2](#0-1) 

The rationale is that the DAO type script itself enforces the correct output capacity for DAO cells (interest calculation). However, the DAO type script is a **type script scoped to DAO cells only**. It verifies that the DAO output capacity equals the calculated maximum withdrawal. It does not verify the total transaction balance, and it does not verify that non-DAO outputs are covered by non-DAO inputs.

The per-output minimum-capacity check (lines 496–512) still runs, but it only ensures each individual output cell holds enough capacity for its own data — it does not enforce `sum(inputs) >= sum(outputs)`. [3](#0-2) 

**Attack construction:**

| Cell | Capacity |
|---|---|
| DAO withdrawing input | 100 CKB (deposited earlier) |
| Regular input (attacker-owned) | 1 CKB |
| DAO output | 110 CKB (correct, verified by DAO script) |
| Regular output (attacker-controlled) | 1,000,000 CKB |

- `CapacityVerifier` skips the sum check because `valid_dao_withdraw_transaction()` is `true`.
- The DAO type script runs and approves the 110 CKB DAO output.
- The 1,000,000 CKB regular output is never checked against the 1 CKB regular input.
- The transaction is accepted, and the attacker gains 999,999 CKB from thin air.

---

### Impact Explanation

An attacker can inflate their CKB balance arbitrarily by mixing a legitimate DAO withdrawal with crafted non-DAO outputs. This breaks the fundamental CKB invariant that `sum(outputs capacity) <= sum(inputs capacity)` for non-cellbase transactions. The attacker can repeat this with any DAO deposit they control, minting unbounded CKB. This is a critical consensus-level accounting failure.

---

### Likelihood Explanation

Any user who has ever deposited CKB into the Nervos DAO (a common, encouraged operation) can trigger this. The attack requires no privileged access, no leaked keys, and no majority hashpower. The attacker submits a crafted transaction via the standard RPC (`send_transaction`) or P2P relay. The entry path is fully reachable by an unprivileged tx-pool submitter.

---

### Recommendation

Replace the `.any()` guard with a per-cell approach: do not skip the global `OutputsSumOverflow` check. Instead, allow the DAO output capacity to exceed the corresponding DAO input capacity by the calculated interest amount, while still enforcing that the total transaction balance (including non-DAO cells) is conserved. Alternatively, compute `non_dao_inputs_sum` and `non_dao_outputs_sum` separately and enforce `non_dao_outputs_sum <= non_dao_inputs_sum` even when a DAO withdrawal is present.

---

### Proof of Concept

1. Deposit 100 CKB into the Nervos DAO (phase 1 deposit).
2. Mine enough blocks to accrue interest (e.g., 10 CKB interest → 110 CKB withdrawal).
3. Execute DAO withdrawal phase 1 (create withdrawing cell).
4. Craft phase 2 withdrawal transaction:
   - **Input 0**: DAO withdrawing cell (100 CKB, DAO type script present → triggers `valid_dao_withdraw_transaction()`)
   - **Input 1**: Any regular live cell (1 CKB)
   - **Output 0**: DAO output cell (110 CKB, passes DAO type script verification)
   - **Output 1**: Regular output cell (1,000,000 CKB, lock = attacker's lock)
5. Submit via `send_transaction` RPC.
6. `CapacityVerifier::verify()` skips `OutputsSumOverflow` check (line 483). DAO script approves output 0. Output 1 is never balance-checked.
7. Transaction is accepted into the tx-pool and committed to a block. Attacker now holds 1,000,000 CKB. [4](#0-3) [5](#0-4)

    pub fn verify(&self) -> Result<(), Error> {
        // skip OutputsSumOverflow verification for resolved cellbase and DAO
        // withdraw transactions.
        // cellbase's outputs are verified by RewardVerifier
        // DAO withdraw transaction is verified via the type script of DAO cells
        if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
            let inputs_sum = self.resolved_transaction.inputs_capacity()?;
            let outputs_sum = self.resolved_transaction.outputs_capacity()?;

            if inputs_sum < outputs_sum {
                return Err((TransactionError::OutputsSumOverflow {
                    inputs_sum,
                    outputs_sum,
                })
                .into());
            }
        }
