Audit Report

## Title
Remote Panic via `get_block_uncles` on Frozen-Block Transactions in `GetTransactionsProof` Handler — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

An unprivileged light-client peer can crash a CKB node by sending a `GetTransactionsProof` message containing a `tx_hash` from a frozen block. The freezer migration deletes `COLUMN_BLOCK_UNCLE` for frozen blocks but preserves `COLUMN_TRANSACTION_INFO`, so the handler's `.expect("block uncles must be stored")` at line 121 panics when it calls `get_block_uncles()` on a frozen block hash. The panic is unrecoverable and terminates the node process.

## Finding Description

**Exploit flow:**

1. The attacker sends `GetTransactionsProof` with a `tx_hash` belonging to a frozen block.
2. At lines 59–64, `get_transaction_info(tx_hash)` succeeds because `COLUMN_TRANSACTION_INFO` is **not** deleted by `delete_block_body()` ( [1](#0-0) ). `is_main_chain(&tx_info.block_hash)` also returns true because `COLUMN_INDEX` is not deleted either. The tx_hash enters the `found` partition. ( [2](#0-1) )
3. At lines 83–85, `get_block(&block_hash)` **succeeds** for the frozen block via the early-return path in `store.rs` that reads from the freezer directly and returns before ever calling `get_block_uncles()`. ( [3](#0-2) )
4. At lines 119–121, `get_block_uncles(&block_hash)` is called. This function reads only from the in-memory cache and `COLUMN_BLOCK_UNCLE` — it has no freezer fallback. ( [4](#0-3) )
5. `COLUMN_BLOCK_UNCLE` was deleted during freezer migration by `delete_block_body()`. ( [5](#0-4) ) The function returns `None`, and `.expect("block uncles must be stored")` panics. ( [6](#0-5) )

**Why existing guards fail:**

- The `GET_TRANSACTIONS_PROOF_LIMIT` check only limits count, not block age.
- The `is_main_chain` check passes for frozen blocks (their `COLUMN_INDEX` entry is preserved).
- The `get_block()` call at line 83 does not panic for frozen blocks because it takes the freezer early-return path, masking the missing `COLUMN_BLOCK_UNCLE`. ( [7](#0-6) )

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

A single crafted `GetTransactionsProof` message causes an unrecoverable `panic!` in the node process, terminating it entirely. No authentication or privilege is required. The attack is repeatable and can prevent the node from staying online.

## Likelihood Explanation

- All historical `tx_hash` values are publicly available via block explorers; no special knowledge is needed.
- The freezer activates automatically on any long-running CKB node once the chain grows past the freezer threshold — this is a standard configuration.
- The light-client protocol server requires no authentication from peers.
- The attack requires sending a single well-formed protocol message.

## Recommendation

At lines 119–121, replace the separate `get_block_uncles()` call with `block.uncles()`, using the `BlockView` already retrieved at line 83. For frozen blocks, `get_block()` reconstructs the full `BlockView` (including uncles) from the freezer's raw block data, so `block.uncles()` is correct and always available:

```rust
// Replace:
let uncles = snapshot
    .get_block_uncles(&block_hash)
    .expect("block uncles must be stored");

// With:
let uncles = block.uncles();
```

This is consistent with how `get_block()` itself handles frozen blocks and eliminates the dependency on the deleted `COLUMN_BLOCK_UNCLE` entry.

## Proof of Concept

1. Run a CKB node with a freezer configured until blocks are frozen (or use an existing mainnet/testnet node past the freezer threshold).
2. Obtain any `tx_hash` from a frozen block (block number < `freezer.number()`) from a public block explorer.
3. Connect as a light-client peer to the node.
4. Send:
   ```
   GetTransactionsProof {
       last_hash: <any recent main-chain block hash>,
       tx_hashes: [<tx_hash from frozen block>],
   }
   ```
5. The node panics at `get_transactions_proof.rs:121` with message `"block uncles must be stored"` and the process terminates.
