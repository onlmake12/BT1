Audit Report

## Title
Unbounded `vec!` Allocation in `WeightUnitsFlow::do_estimate` via High-Fee-Rate Transaction — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`Algorithm::max_bucket_index_by_fee_rate` maps a `FeeRate` to a bucket index with no upper bound. For fee rates produced by a high-fee transaction (achievable with ~10,000 CKB), the returned index reaches ~1.67×10⁸, causing `do_estimate` to attempt two allocations of ~1.34 GB each via `vec![0u64; max_bucket_index + 1]`. The OS OOM killer terminates the node process. Any node with `algorithm = "WeightUnitsFlow"` in `[fee_estimator]` is affected.

## Finding Description

**Root cause 1 — unbounded `max_bucket_index_by_fee_rate`:**

The final match arm applies a linear formula with no ceiling:

```rust
x => (x + t * 11_500) / (100 * t),   // t = 1000
``` [1](#0-0) 

For `x ≈ 1.67×10¹³` (fee rate from a 10,000 CKB fee / 60-byte tx), this yields `≈ 1.67×10⁸`. No cap or guard exists.

**Root cause 2 — unchecked allocation in `do_estimate`:**

```rust
let mut buckets = vec![0u64; max_bucket_index + 1];   // line 219
...
let mut buckets = vec![0u64; max_bucket_index + 1];   // line 248
``` [2](#0-1) [3](#0-2) 

Both allocations use the raw index with no bounds check.

**Root cause 3 — `FeeRate::calculate` uses `saturating_mul`:**

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
``` [4](#0-3) 

For `fee = 10,000 CKB = 10¹² shannons`, `weight = 60`: `fee_rate = 10¹² × 1000 / 60 ≈ 1.67×10¹³`. For `fee ≥ ~1.844×10¹⁶ shannons`, `saturating_mul` caps at `u64::MAX`, yielding `fee_rate ≈ 3.07×10¹⁷`.

**Root cause 4 — no fee-rate cap in tx-pool admission:** A grep across `tx-pool/src/` for any fee-rate cap returns no matches. Any valid transaction with a high fee is admitted.

**Exploit path:**

1. Attacker submits a valid transaction with fee ≥ 10,000 CKB and minimum weight (60 bytes).
2. `TxStatus::new_from_entry_info` computes `fee_rate ≈ 1.67×10¹³`.
3. Attacker (or any caller) invokes the `estimate_fee_rate` RPC.
4. `process.rs::estimate_fee_rate` calls `get_all_entry_info` then `FeeEstimator::estimate_fee_rate` → `do_estimate`.
5. `max_bucket_index_by_fee_rate` returns `≈ 1.67×10⁸`; `vec![0u64; 1.67×10⁸+1]` is attempted twice (~2.68 GB total).
6. OOM killer terminates the node. [5](#0-4) 

**`WeightUnitsFlow` is opt-in:** [6](#0-5) [7](#0-6) 

## Impact Explanation

On any node with `WeightUnitsFlow` configured, a single `estimate_fee_rate` RPC call while a high-fee-rate transaction is in the mempool causes the tx-pool service thread to attempt a multi-GB heap allocation, crashing the node process via OOM. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**.

## Likelihood Explanation

`WeightUnitsFlow` requires explicit opt-in, limiting the affected population. For nodes that do enable it: the attacker needs ~10,000 CKB temporarily (not permanently burned if the node crashes before block confirmation). The `estimate_fee_rate` RPC is unauthenticated and publicly accessible. The attack is repeatable within each ~28-second block interval. The economic barrier is low relative to the impact.

## Recommendation

Cap the result of `max_bucket_index_by_fee_rate` to a safe constant before use:

```rust
const MAX_BUCKET_INDEX: usize = 1000;

fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
    let t = FEE_RATE_UNIT;
    let index = match fee_rate.as_u64() {
        // ... existing arms ...
        x => (x + t * 11_500) / (100 * t),
    };
    (index as usize).min(MAX_BUCKET_INDEX)
}
```

Alternatively, add a guard in `do_estimate` before both `vec!` allocations:

```rust
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_BUCKET_INDEX);
```

## Proof of Concept

```rust
#[test]
fn test_no_oom_on_high_fee_rate() {
    // fee = 10_000 CKB = 10^12 shannons, weight = 60
    // fee_rate = 10^12 * 1000 / 60 ≈ 1.67e13
    // max_bucket_index ≈ 1.67e8 → ~1.34 GB per vec → OOM
    let fee_rate = FeeRate::calculate(
        ckb_types::core::Capacity::shannons(1_000_000_000_000u64),
        60,
    );
    let idx = Algorithm::max_bucket_index_by_fee_rate(fee_rate);
    assert!(idx < 10_000, "bucket index {} is unbounded", idx);
}
```

Running this test on an unpatched node will either OOM-kill the test process or confirm the index is ~167,000,000, demonstrating the allocation would be ~1.34 GB per `vec!` call.
