Audit Report

## Title
RPC Batch Request Limit Disabled by Default Allows Unbounded Resource Exhaustion - (File: rpc/src/server.rs)

## Summary
The CKB JSON-RPC server's batch request size limit is opt-in and disabled by default. `JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is never initialized unless the operator explicitly sets `rpc_batch_limit` in the config, meaning the batch size guard is entirely skipped for default deployments. An attacker with access to the RPC port can submit a single HTTP POST containing thousands of expensive calls, exhausting node memory and CPU.

## Finding Description
In `rpc/src/server.rs` lines 53–55, `JSONRPC_BATCH_LIMIT` is only initialized when `config.rpc_batch_limit` is `Some(...)`:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

In `handle_jsonrpc` lines 274–282, the guard is conditional on `JSONRPC_BATCH_LIMIT.get()` returning `Some`. When the `OnceLock` is uninitialized (the default), the check is skipped entirely and all calls in the batch are processed:

```rust
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    { ... }
    // falls through with no limit when OnceLock is None
```

The config field `rpc_batch_limit: Option<usize>` has no `#[serde(default)]` providing a safe fallback, and the production template explicitly leaves it commented out (`resource/ckb.toml` lines 205–208). The 10 MiB `max_request_body_size` allows ~174,000 lightweight call objects to fit in a single request. Calls are processed sequentially via `stream::iter(calls).then(...)` (lines 284–289). The `TimeoutLayer` of 30 seconds (lines 125–128) limits request duration but does not prevent memory exhaustion during that window: each `get_raw_tx_pool(verbose=true)` call acquires the tx-pool lock and serializes up to 180 MB of pool state (`max_tx_pool_size = 180_000_000`). Even a few hundred such calls within the 30-second window can exhaust available memory and crash the node process.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.** Repeated serialization of up to 180 MB of tx-pool state per batch call causes process-level OOM, crashing the entire node (not just the RPC handler). The RPC runtime shares the same process memory as the P2P, sync, and consensus subsystems. A node crash halts block propagation and mining for that operator.

## Likelihood Explanation
The RPC is bound to `127.0.0.1:8114` by default, but the bounty scope explicitly includes "RPC caller" and "supported local CLI/RPC user" as valid attacker roles. Many production deployments expose the RPC to a local network or behind a reverse proxy. The attack requires a single HTTP POST with a crafted JSON array — no authentication, no special privileges, no prior chain knowledge. The config comment itself acknowledges the risk is unmitigated by default. The attack is trivially repeatable.

## Recommendation
- Change `rpc_batch_limit` from `Option<usize>` to a field with a hardcoded safe default (e.g., 200), enforced unconditionally by initializing `JSONRPC_BATCH_LIMIT` at startup regardless of config.
- Alternatively, initialize `JSONRPC_BATCH_LIMIT` with a safe fallback when `config.rpc_batch_limit` is `None`, rather than skipping the check entirely.
- Emit a startup warning when no batch limit is configured.

## Proof of Concept
```bash
# Build a batch of 500 get_raw_tx_pool(verbose=true) calls (fits well within 10 MiB)
python3 -c "
import json
calls = [{'id': i, 'jsonrpc': '2.0', 'method': 'get_raw_tx_pool', 'params': [True]} for i in range(500)]
print(json.dumps(calls))
" > batch.json

# Send to a node with no rpc_batch_limit configured (default)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data @batch.json
# Each call serializes up to 180 MB of tx-pool state; 500 calls = up to 90 GB of
# serialization work within the 30-second timeout window, causing OOM and node crash.
```
