### Title
Integer Overflow in `GetLastStateProofProcess::execute` Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Remote DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The guard at line 201 of `get_last_state_proof.rs` is intended to bound total work per `GetLastStateProof` request to `GET_LAST_STATE_PROOF_LIMIT = 1000`. Due to a `usize` multiplication overflow in Rust release mode, an attacker can supply `last_n_blocks = u64::MAX/2 + 1` to make the check evaluate to `0 > 1000 = false`, bypassing the limit entirely. The server then allocates a `Vec` of up to N `u64` block numbers (where N is the chain height) and performs N MMR root computations and N DB reads — O(N) work per request from a single unauthenticated peer.

---

### Finding Description

**The overflow:** [1](#0-0) 

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

On a 64-bit system, `usize` is 64 bits. With `last_n_blocks = u64::MAX/2 + 1 = 9_223_372_036_854_775_808`:

- `last_n_blocks as usize = 9_223_372_036_854_775_808` (fits in 64-bit `usize`)
- `9_223_372_036_854_775_808usize * 2` wraps to `0` in Rust release mode (no overflow panics)
- `0 + 0 > 1000` → `false` → the guard is bypassed

The limit constant is: [2](#0-1) 

**The unbounded allocation after bypass:** [3](#0-2) 

```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    (sampled_numbers, last_n_numbers)
```

With `last_n_blocks = u64::MAX/2 + 1` and `start_block_number = 0`, the condition `last_block_number <= u64::MAX/2 + 1` is trivially true for any realistic chain. The server collects ALL block numbers from 0 to `last_block_number` into a `Vec`.

**The O(N) work in `complete_headers`:** [4](#0-3) 

For each of the N block numbers, `complete_headers` calls:
- `snapshot.get_ancestor(...)` — DB read
- `snapshot.get_block(...)` — DB read
- `snapshot.chain_root_mmr(*number - 1).get_root()` — MMR computation

**The `GetLastStateProof` message schema confirms `last_n_blocks` is an unconstrained `Uint64`:** [5](#0-4) 

---

### Impact Explanation

A single unauthenticated P2P peer can send one crafted `GetLastStateProof` message to force the full node to:
1. Allocate a `Vec<u64>` of up to N entries (e.g., 10M entries × 8 bytes = 80 MB per request)
2. Perform N sequential DB reads and N MMR root computations

Repeated requests from one or more peers can exhaust memory and saturate I/O, crashing or severely degrading the node and all peers it serves. The invariant that `GET_LAST_STATE_PROOF_LIMIT = 1000` bounds work per request is completely violated.

---

### Likelihood Explanation

The light client protocol server is a production feature. The attack requires only:
- Knowledge of any valid main-chain block hash (publicly observable)
- Crafting a single P2P message with `last_n_blocks = u64::MAX/2 + 1`, `difficulties = []`, `start_number = 0`

No PoW, no keys, no privileged access. The overflow is deterministic in release builds.

---

### Recommendation

Replace the plain multiplication with a saturating or checked variant:

```rust
// Before (vulnerable):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

`saturating_mul` returns `usize::MAX` on overflow, which is always `> 1000`, so the guard correctly rejects the malicious request.

```rust
// Attacker constructs:
let msg = GetLastStateProof::new_builder()
    .last_hash(any_valid_main_chain_block_hash)
    .start_hash(Byte32::default())
    .start_number(0u64.pack())
    .last_n_blocks((u64::MAX / 2 + 1).pack())  // triggers overflow
    .difficulty_boundary(U256::one().pack())
    .difficulties(Uint256Vec::default())        // empty
    .build();

// (u64::MAX/2 + 1) as usize = 9_223_372_036_854_775_808
// 9_223_372_036_854_775_808usize * 2 = 0  (wraps)
// 0 + 0 > 1000 → false → guard bypassed
```
