### Title
Unvalidated Attacker-Controlled `listen_addrs` in Hole-Punching Protocol Enables SSRF and Internal Network Port Scanning — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

An unprivileged peer with a standard P2P connection can cause the victim node to make repeated outbound TCP connections to arbitrary attacker-specified IP:port combinations — including private/internal network addresses — by sending a crafted `ConnectionRequest` followed by a `ConnectionSync` message. No IP-range validation exists anywhere in the hole-punching code path.

---

### Finding Description

The hole-punching protocol processes `ConnectionRequest` messages in `ConnectionRequestProcess::execute()`. When the victim node is the `to` target, it calls `respond_delivered()`, which filters the attacker-supplied `listen_addrs` only by transport type (TCP + IPv4/IPv6 presence): [1](#0-0) 

After this filter, the addresses — which may be any routable or private IPv4/IPv6 TCP address — are stored verbatim in `pending_delivered`, keyed by `content.from` (a message field, **not** the actual session peer ID): [2](#0-1) 

When a `ConnectionSync` message subsequently arrives with a matching `from` field, `ConnectionSyncProcess::execute()` retrieves those stored addresses and spawns an async task calling `try_nat_traversal` on each one: [3](#0-2) 

`try_nat_traversal` makes real TCP connection attempts in a retry loop for up to **30 seconds** (~150 attempts per address at 200ms intervals) with no IP-range filtering: [4](#0-3) 

There is no check that `content.from` in the `ConnectionSync` matches the actual session peer ID of the sender, and no check that the sender of `ConnectionSync` is the same peer that sent the original `ConnectionRequest`. The `from` field is purely message-level and fully attacker-controlled. [5](#0-4) 

---

### Impact Explanation

- **SSRF**: The victim node makes TCP connections to attacker-specified internal addresses (e.g., `10.0.0.1:8545`, `192.168.1.1:22`, `127.0.0.1:8114`). This can interact with internal services (RPC endpoints, databases, cloud metadata APIs at `169.254.169.254`).
- **Internal port scanning**: By varying the address across multiple `ConnectionRequest` messages (using different `from` peer IDs to bypass the 2-minute `HOLE_PUNCHING_INTERVAL` per-key cooldown), an attacker can systematically probe the victim's internal network.
- **Resource exhaustion**: Each triggered `try_nat_traversal` spawns a 30-second async task. Multiple attackers or rapid rotation of `from` peer IDs can exhaust goroutine/task resources.

---

### Likelihood Explanation

The preconditions are minimal: the attacker only needs a standard P2P connection to the victim (no special privileges) and knowledge of the victim's peer ID (publicly discoverable via the P2P network). The two-message sequence (`ConnectionRequest` then `ConnectionSync`) is straightforward to craft. The `ADDRS_COUNT_LIMIT` of 24 addresses per request means up to 24 internal hosts can be probed per exchange. [6](#0-5) 

---

### Recommendation

1. **Reject private/loopback/link-local addresses** in `respond_delivered()` before storing into `pending_delivered`. Add a filter that rejects `Protocol::Ip4(addr)` where `addr.is_private() || addr.is_loopback() || addr.is_link_local()` and similarly for IPv6.
2. **Bind `pending_delivered` to the actual session peer ID**, not the message-level `content.from` field, to prevent spoofing.
3. **Verify that the `ConnectionSync` sender matches the original `ConnectionRequest` sender** (by session peer ID, not message field).
4. Apply the same IP-range filter inside `try_nat_traversal` as a defense-in-depth measure.

---

### Proof of Concept

```
1. Attacker connects to victim via standard P2P.
2. Attacker sends:
     ConnectionRequest {
       from: <any_valid_peer_id>,
       to:   <victim_peer_id>,
       max_hops: 6,
       route: [],
       listen_addrs: [/ip4/192.168.1.100/tcp/8114, /ip4/169.254.169.254/tcp/80, ...]
     }
3. Victim's respond_delivered() stores those addresses in pending_delivered[any_valid_peer_id].
4. Attacker sends:
     ConnectionSync {
       from: <any_valid_peer_id>,
       to:   <victim_peer_id>,
       route: []
     }
5. Victim's ConnectionSyncProcess::execute() retrieves the stored addresses and spawns
   try_nat_traversal tasks → victim makes TCP SYN packets to 192.168.1.100:8114,
   169.254.169.254:80, etc. for 30 seconds each.
```

The `from` field in both messages is not verified against the actual session peer ID, so the attacker can use any syntactically valid peer ID bytes and the lookup will succeed.

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
