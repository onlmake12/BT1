### Title
Proxy Credentials and Tor Controller Password Exposed via `Debug` Derive and Error Messages — (File: `util/app-config/src/configs/network.rs`, `network/src/proxy.rs`)

---

### Summary

`ProxyConfig` and `OnionConfig` both derive `Debug` while holding sensitive credentials (`proxy_url` with embedded `username:password`, and `tor_password`). Both are embedded in the top-level `NetworkConfig`, which also derives `Debug`. Additionally, `check_proxy_url` constructs error strings that include the full proxy URL verbatim. With Sentry crash-reporting integrated into the codebase, any panic or captured error that formats these structs via `{:?}` will transmit credentials to a third-party service.

---

### Finding Description

**1. `ProxyConfig` derives `Debug` and stores credentials in plaintext**

`ProxyConfig` is declared with `#[derive(Clone, Debug, Serialize, Deserialize, Default)]`. Its `proxy_url` field holds a SOCKS5 URL of the form `socks5://username:password@host:port`. [1](#0-0) 

**2. `OnionConfig` derives `Debug` and stores `tor_password` in plaintext**

`OnionConfig` is declared with `#[derive(Clone, Debug, Serialize, Deserialize, Default)]`. Its `tor_password` field holds the Tor controller authentication password. [2](#0-1) 

**3. Top-level `NetworkConfig` derives `Debug` and embeds both**

The parent `Config` struct also derives `Debug` and contains both `ProxyConfig` and `OnionConfig` as fields, meaning any `{:?}` formatting of the top-level config recursively exposes all nested credentials. [3](#0-2) 

**4. `check_proxy_url` leaks the full URL (with credentials) in error messages**

When proxy URL validation fails (e.g., wrong scheme, missing host, missing port), the error string returned by `check_proxy_url` includes the raw `proxy_url` value verbatim — credentials and all. [4](#0-3) 

This error is propagated at startup via: [5](#0-4) 

**5. Sentry integration is present in the network module**

The network module conditionally imports and uses Sentry's `capture_message` and `with_scope` APIs. Any error or panic that formats the config or propagates the `check_proxy_url` error string can be captured and transmitted to Sentry (a third-party service). [6](#0-5) 

---

### Impact Explanation

An operator who configures a SOCKS5 proxy with credentials (`proxy_url = "socks5://user:secret@proxy.example.com:1080"`) or a Tor controller password (`tor_password = "mypassword"`) will have those credentials exposed in:

- Any log line that formats `NetworkConfig`, `ProxyConfig`, or `OnionConfig` with `{:?}`
- Error messages returned by `check_proxy_url` when the URL is malformed (e.g., wrong scheme), which include the full URL verbatim
- Sentry crash reports, which are transmitted to a third-party service

This mirrors the exact exploit scenario in the reference report: a service crash or startup error causes credential material to be exported to a third-party analytics/crash service.

---

### Likelihood Explanation

Medium. The Sentry feature is opt-in but documented and used in production deployments. Any misconfigured proxy URL (wrong scheme, missing port) triggers the `check_proxy_url` error path, which returns the full URL in the error string. This error propagates to the caller and can be logged or Sentry-captured. The `Debug` derive issue is always present regardless of Sentry.

---

### Recommendation

- Implement a custom `Debug` for `ProxyConfig` and `OnionConfig` that redacts sensitive fields (e.g., replace password with `[REDACTED]`).
- In `check_proxy_url`, sanitize the URL before including it in error messages — strip the `username:password@` component before formatting.
- Consider storing credentials outside the config struct (e.g., fetched at runtime from a secret store) so they are never present in the in-memory config object that gets formatted.

---

### Proof of Concept

1. Configure `ckb.toml` with:
   ```toml
   [network.proxy]
   proxy_url = "http://user:secret@proxy.example.com:1080"
   ```
2. Start `ckb run`. The `check_proxy_url` call at startup fails because the scheme is `http` (not `socks5`).
3. The returned error string is:
   ```
   CKB doesn't support proxy scheme: http  (full url: http://user:secret@proxy.example.com:1080)
   ``` [7](#0-6) 
4. This error propagates up through `NetworkState::from_config` and is logged or captured by Sentry.
5. With Sentry enabled, the error — including the credential-bearing URL — is transmitted to the third-party Sentry service.

pub(crate) fn check_proxy_url(proxy_url: &str) -> Result<(), String> {
    let parsed_url = Url::parse(proxy_url).map_err(|e| e.to_string())?;
    if parsed_url.host_str().is_none() {
        return Err(format!("missing host in proxy url: {}", proxy_url));
    }
    let scheme = parsed_url.scheme();
    if scheme.ne("socks5") {
        return Err(format!("CKB doesn't support proxy scheme: {}", scheme));
    }
    if parsed_url.port().is_none() {
        return Err(format!("missing port in proxy url: {}", proxy_url));
    }
    Ok(())
}
