Audit Report

## Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Validation, Enabling Permanently Unspendable Cells — (File: `verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify()` iterates over every transaction output and validates the `hash_type` of the **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the optional **type** script. Any unprivileged user can submit a transaction whose output carries a type script with a structurally valid but not-yet-activated `hash_type` (e.g., `6` = `Data3`). The transaction passes all non-contextual gates, is accepted into the tx-pool, and can be mined into a block. The resulting live cell is permanently unspendable because `select_version()` will always reject the unsupported hash type at execution time.

## Finding Description

`ENABLED_SCRIPT_HASH_TYPE` defines the four activated values `{0, 1, 2, 4}`: [1](#0-0) 

`ScriptHashTypeVerifier::verify()` loops over outputs and checks only `output.lock().hash_type()`: [2](#0-1) 

`output.type_()` is never read inside this function. The struct's own documentation in `NonContextualTransactionVerifier` explicitly states only the lock hash type is checked: [3](#0-2) 

A type script with `hash_type = 6` passes `ScriptHashTypeVerifier` because only the lock is inspected. When any future transaction attempts to consume the resulting cell, `select_version()` hits the catch-all arm: [4](#0-3) 

This returns `Err(ScriptError::InvalidScriptHashType(...))` unconditionally, making the cell permanently unspendable. There is no upgrade path or recovery mechanism.

**Exploit flow:**
1. Attacker constructs a transaction with a standard lock script (`hash_type = 1`) and a type script with `hash_type = 6`.
2. `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`, which only checks the lock — passes.
3. Transaction is accepted into the tx-pool and mined into a block.
4. The live cell now carries a type script with `hash_type = 6` on-chain.
5. Any attempt to spend the cell triggers `select_version()` on the type script, which returns `InvalidScriptHashType` — the cell is permanently frozen.

## Impact Explanation

The concrete impact is **permanent freezing of CKB capacity** in provably unspendable live cells, inflating the live-cell (UTXO) set with dead entries that can never be pruned. This constitutes a **suboptimal implementation of the CKB state storage mechanism** (Medium, 2001–10000 points): the live-cell set is a critical shared resource, and cells that can never be consumed represent permanent, irrecoverable state bloat. While the attacker burns their own funds, the dead cells persist in every node's cell store indefinitely, and the invariant that `ScriptHashTypeVerifier` is supposed to enforce — no cell with a non-activated hash type enters the chain — is silently violated for type scripts.

## Likelihood Explanation

Likelihood is high. No privileges, special tooling, or coordination are required. Any user who can call `send_raw_transaction` via RPC can craft the malicious transaction. The structural validity check (`TryInto::<ScriptHashType>`) passes for `hash_type = 6` because it is a recognized enum variant. The only guard that should catch it (`ScriptHashTypeVerifier`) does not inspect type scripts. The attack is repeatable at will and costs only the minimum cell capacity.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

Also update the `NonContextualTransactionVerifier` doc comment to reflect that both lock and type script hash types are checked.

## Proof of Concept

1. Construct a raw transaction with one output: lock script `hash_type = 1` (Type), type script `hash_type = 6` (Data3), capacity at minimum occupied capacity.
2. Submit via `send_raw_transaction` RPC.
3. Observe: no `ScriptHashTypeNotPermitted` error — transaction is accepted into the pool.
4. Mine the transaction into a block (or use a dev chain with `submit_block`).
5. Attempt to spend the resulting cell in a new transaction.
6. Observe: node rejects with `InvalidScriptHashType` / `InvalidVmVersion`, confirming the cell is permanently unspendable.
7. Confirm the cell remains in the live-cell set indefinitely with no mechanism to remove it.
