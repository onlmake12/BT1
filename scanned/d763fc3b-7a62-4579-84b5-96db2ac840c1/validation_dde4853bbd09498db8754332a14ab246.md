### Title
u64 Overflow in `set_ban` Relative Ban-Time Computation — (`rpc/src/module/net.rs`)

### Summary
The `set_ban` RPC handler performs an unchecked `u64` addition of the current wall-clock milliseconds and the caller-supplied `ban_time`. Supplying `ban_time = u64::MAX` with `absolute = false` overflows the `u64` type, causing a panic in debug builds and a silent wrap-around (producing an already-expired ban entry) in release builds.

### Finding Description
In `NetRpcImpl::set_ban`, the relative-time branch computes:

```rust
unix_time_as_millis() + ban_time.unwrap_or_else(|| DEFAULT_BAN_DURATION.into()).value()
``` [5](#0-4) 

Both operands are `u64`. No overflow guard (`checked_add`, `saturating_add`) is applied. A caller supplying `ban_time = 0xffffffffffffffff` triggers overflow.

### Impact Explanation
- **Debug builds**: the RPC handler thread panics, crashing the RPC service for that request and potentially the thread.
- **Release builds** (without `overflow-checks = true`): `ban_until` wraps to a value in the past; the ban entry is silently inserted but is immediately expired, so the intended peer is never actually banned — a ban-bypass.

### Likelihood Explanation
The RPC endpoint is unauthenticated and bound to localhost by default. Any local process or user with access to port 8114 can trigger this with a single JSON-RPC call. No special privileges are required.

### Recommendation
Replace the plain `+` with `checked_add` and return an `RPCError::invalid_params` if overflow would occur:

```rust
let ban_until = unix_time_as_millis()
    .checked_add(ban_time.unwrap_or_else(|| DEFAULT_BAN_DURATION.into()).value())
    .ok_or_else(|| RPCError::invalid_params("ban_time overflows u64"))?;
```

### Proof of Concept
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{
    "id": 1, "jsonrpc": "2.0", "method": "set_ban",
    "params": ["192.168.0.1", "insert", "0xffffffffffffffff", false, null]
  }'
```

In a debug build this panics the RPC handler. In a release build it silently inserts an already-expired ban entry, verifiable by calling `get_banned_addresses` and observing `ban_until` is in the past.
