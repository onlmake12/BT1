### Title
Missing Upper Bound on `genesis_epoch_length` Silently Corrupts `EpochNumberWithFraction` Packed Encoding in Release Builds - (File: spec/src/lib.rs, spec/src/consensus.rs)

### Summary
The `genesis_epoch_length` chain spec parameter is accepted as an unchecked `u64`, but `EpochNumberWithFraction` packs the epoch length into only 16 bits (max 65,535). In release builds, `debug_assert!` bounds checks are compiled out, so any value ≥ 65,536 silently overflows the length field, corrupting every block header's epoch encoding and causing division-by-zero panics or wrong epoch comparisons across the node.

### Finding Description
`EpochNumberWithFraction` encodes three sub-fields into a single `u64`:

- bits 0–23 (24 bits): epoch number
- bits 24–39 (16 bits): epoch index
- bits 40–55 (16 bits): epoch length (`LENGTH_MAXIMUM_VALUE = 2^16 = 65,536`) [1](#0-0) 

The constructor `EpochNumberWithFraction::new()` guards these invariants only with `debug_assert!`, which is a no-op in release builds: [2](#0-1) 

The chain spec `Params` struct exposes `genesis_epoch_length` as a plain `Option<BlockNumber>` (i.e., `u64`) with no upper bound: [3](#0-2) 

`build_consensus()` passes this value directly to `build_genesis_epoch_ext()` without any range check: [4](#0-3) 

`build_genesis_epoch_ext()` stores the raw value in `EpochExt.length` (a full `u64`), so no overflow there: [5](#0-4) 

The corruption surfaces when `number_with_fraction()` is called to stamp every block header's epoch field: [6](#0-5) 

This calls `EpochNumberWithFraction::new(epoch_number, index, length)`. In release builds the unchecked path executes:

```
(65536 << 40) | (index << 24) | number
```

When the length is read back via `(self.0 >> 40) & 0xFFFF`, the result is `65536 & 65535 = 0`. Every block header in the chain carries a length of 0 in its packed epoch field.

### Impact Explanation
A length of 0 in `EpochNumberWithFraction` causes:

1. **Division-by-zero panic** in `to_rational()` for any non-genesis block, because it computes `RationalU256::new(index, length)` with `length = 0`: [7](#0-6) 

2. **`is_well_formed()` always returns false**, breaking `since`-field epoch lock verification in `SinceVerifier`: [8](#0-7) 

3. **`EpochStartBlockNumber` syscall panics** because `header.number().checked_sub(epoch.index())` is called on a header whose epoch index is also corrupted: [9](#0-8) 

The node crashes or produces invalid blocks from genesis, making the chain completely non-functional.

### Likelihood Explanation
Mainnet and testnet use the fixed default `GENESIS_EPOCH_LENGTH = 1_000`, so they are unaffected. However, CKB explicitly supports custom devnets and private chains via user-supplied chain spec TOML files. Any operator or developer who sets `genesis_epoch_length` to a value ≥ 65,536 in a custom spec triggers this silently in a release binary. There is no validation at spec load time, no error message, and no runtime guard in release mode. The `MAX_EPOCH_LENGTH` constant (1,800) is only enforced by the dynamic difficulty adjustment algorithm at runtime, not at spec parsing time. [10](#0-9) 

### Recommendation
Add an explicit upper bound check in `build_consensus()` or `Params::genesis_epoch_length()` before the value is used:

```rust
pub fn genesis_epoch_length(&self) -> BlockNumber {
    let length = self.genesis_epoch_length
        .unwrap_or_else(default_params::genesis_epoch_length);
    assert!(
        length < EpochNumberWithFraction::LENGTH_MAXIMUM_VALUE,
        "genesis_epoch_length {} must be less than {}",
        length,
        EpochNumberWithFraction::LENGTH_MAXIMUM_VALUE
    );
    length
}
```

Alternatively, replace `debug_assert!` with hard `assert!` in `EpochNumberWithFraction::new()` so the invariant is enforced in all build profiles.

### Proof of Concept
Set in a custom chain spec TOML:

```toml
[params]
genesis_epoch_length = 65536
```

In a release build, `build_genesis_epoch_ext` stores `length = 65536` in `EpochExt`. When the first non-genesis block is built, `epoch.number_with_fraction(1)` calls `EpochNumberWithFraction::new(0, 1, 65536)`. The packed value becomes `(65536 << 40) | (1 << 24)`. Reading back `length()` yields `(packed >> 40) & 0xFFFF = 0`. Any subsequent call to `to_rational()` on this epoch value (e.g., during `since` verification or block reward calculation) executes `RationalU256::new(1u64.into(), 0u64.into())`, which panics with a division-by-zero, crashing the node.

```rust
/// Build the epoch information of genesis block
pub fn build_genesis_epoch_ext(
    epoch_reward: Capacity,
    compact_target: u32,
    genesis_epoch_length: BlockNumber,
    epoch_duration_target: u64,
    genesis_orphan_rate: (u32, u32),
) -> EpochExt {
    let block_reward = Capacity::shannons(epoch_reward.as_u64() / genesis_epoch_length);
    let remainder_reward = Capacity::shannons(epoch_reward.as_u64() % genesis_epoch_length);

    let genesis_orphan_count =
        genesis_epoch_length * genesis_orphan_rate.0 as u64 / genesis_orphan_rate.1 as u64;
    let genesis_hash_rate = compact_to_difficulty(compact_target)
        * (genesis_epoch_length + genesis_orphan_count)
        / epoch_duration_target;

    EpochExt::new_builder()
        .number(0)
        .base_block_reward(block_reward)
        .remainder_reward(remainder_reward)
        .previous_epoch_hash_rate(genesis_hash_rate)
        .last_block_hash_in_previous_epoch(Byte32::zero())
        .start_number(0)
        .length(genesis_epoch_length)
        .compact_target(compact_target)
        .build()
}
```
