### Title
Attacker-Controlled Snappy Varint Causes 8 MB Pre-Decompression Allocation per P2P Message — (File: `network/src/compress.rs`)

---

### Summary

`LengthDelimitedCodecWithCompress::decode()` in `network/src/compress.rs` calls `snap::raw::decompress_len()` to read the claimed uncompressed length from the snappy stream varint header, then immediately allocates a zero-initialized buffer of that size (up to `MAX_UNCOMPRESSED_LEN = 8 MB`) **before** attempting decompression. Because the varint is fully attacker-controlled and is not cross-validated against the compressed payload size, any unauthenticated peer can send a ~9-byte crafted message that forces the node to allocate and zero-initialize 8 MB. Sending such messages in a tight loop from one or more connections causes sustained memory pressure and CPU burn proportional to the message rate, with no authentication prerequisite.

### Impact Explanation

Each crafted 9-byte message forces the node to:
1. Allocate 8 MB of heap memory (`BytesMut::zeroed` / `vec![0; n]`)
2. Zero-initialize all 8 MB (O(n) CPU time, memory-bandwidth bound)
3. Attempt decompression (fails immediately)
4. Free the buffer

At 1,000 such messages per second (easily achievable over a single TCP connection), the node performs 8 GB/s of zero-initialization. This saturates memory bandwidth, starves the Tokio async runtime of CPU time, and causes all other protocol handlers (block sync, transaction relay, consensus) to stall. Sustained attack across multiple connections can cause OOM or complete node unresponsiveness, preventing the node from confirming new transactions — matching the "network not able to confirm new transactions" impact tier.

---

### Likelihood Explanation

- **No authentication required.** The codec runs at the TCP framing layer, before any CKB-level handshake or peer scoring.
- **No prior network membership required.** Any TCP client that speaks the tentacle framing format can trigger this.
- **Trivially scriptable.** The crafted message is 9 bytes and requires no cryptographic material.
- **Affects all compression-enabled protocols simultaneously.** An attacker can open connections to Sync, RelayV3, and LightClient ports concurrently.
- **No rate limit** on the allocation path itself; the only bound is the TCP connection throughput.

---

### Recommendation

1. **Validate the varint against the compressed payload size before allocating.** Snappy's maximum compression ratio is bounded (~250× for pathological inputs, typically much less). Reject messages where `decompressed_bytes_len > compressed_payload_len * MAX_SNAPPY_RATIO` for a conservative constant (e.g., 256).

2. **Lower `MAX_UNCOMPRESSED_LEN` to match the actual protocol `max_frame_length`.** Currently `MAX_UNCOMPRESSED_LEN` (8 MB) exceeds the Sync protocol's `max_frame_length` (2 MB). The cap should be `min(MAX_UNCOMPRESSED_LEN, protocol_max_frame_length)`.

3. **Allocate lazily or use a pooled buffer.** Avoid zero-initializing a large buffer before the decompressor has validated the stream header.

### Proof of Concept

```python
import socket, struct

# Snappy raw format: varint(8_388_608) = [0x80, 0x80, 0x80, 0x04]
# CKB compress flag byte: 0x80

# LengthDelimitedCodec: 4-byte big-endian length prefix
frame = struct.pack(">I", len(payload)) + payload  # 9 bytes total

s = socket.create_connection(("TARGET_CKB_NODE", 8115))
# Send thousands of 9-byte frames; each forces an 8 MB zeroed allocation
for _ in range(10_000):
    s.sendall(frame)
s.close()
```
