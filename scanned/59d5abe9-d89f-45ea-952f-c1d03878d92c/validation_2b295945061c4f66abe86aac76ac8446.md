### Title
Whitelisted Peer Cannot Be Removed at Runtime, Permanently Bypassing Ban and Score Reporting — (`network/src/network.rs`, `network/src/peer_registry.rs`)

---

### Summary

Once a peer is added to `whitelist_peers` via node config, there is no runtime mechanism to remove it. Both `ban_session` and `report_session` unconditionally skip whitelisted peers. The `OutboundPeerService` continuously re-dials all whitelist peers. A compromised or misbehaving whitelisted peer therefore cannot be banned, penalized, or durably disconnected without a full node restart.

---

### Finding Description

`PeerRegistry` stores the whitelist as a `HashSet<PeerId>` populated once at construction: [1](#0-0) [2](#0-1) 

No method on `PeerRegistry` mutates `whitelist_peers` after construction. There is no `remove_whitelist_peer`, no runtime RPC, and no CLI command to modify the set.

When any protocol handler calls `ban_session`, the function silently exits without banning if the peer is whitelisted: [3](#0-2) 

The same guard is applied in `report_session`, which drives the reputation/score system: [4](#0-3) 

Even if the operator manually disconnects the peer via RPC, `OutboundPeerService::try_dial_whitelist` runs on every tick and unconditionally re-dials every address in `config.whitelist_peers()`: [5](#0-4) 

The combination means: **ban is silently dropped → score penalty is silently dropped → peer is automatically reconnected → cycle repeats indefinitely**.

---

### Impact Explanation

A compromised whitelisted peer can:

- Send malformed sync/relay messages continuously without triggering a ban or score penalty.
- Occupy connection slots that bypass `max_inbound`/`max_outbound` limits, since whitelist peers are excluded from the connection-count checks in `accept_peer`.
- Force the local node into a persistent degraded state (invalid block relay, flooding) with no runtime escape path short of a full node restart.

The node operator has no RPC or CLI command to revoke whitelist status at runtime. The only remediation is editing the config file and restarting the node, which introduces downtime and a window of continued exposure. [6](#0-5) 

---

### Likelihood Explanation

Whitelisted peers are a standard production deployment pattern in CKB (used for trusted relay nodes, mining pool peers, and IBD bootstrap peers). A whitelisted peer becoming compromised — through key theft, infrastructure compromise, or a supply-chain attack on the peer's software — is a realistic operational scenario. The inability to respond at runtime without downtime makes the exposure window operator-controlled but unbounded.

---

### Recommendation

1. Add a `remove_whitelist_peer(peer_id: PeerId)` method to `PeerRegistry` that removes the entry from `whitelist_peers` and sets `is_whitelist = false` on any currently connected session for that peer.
2. Expose a `set_whitelist_peer(address, status: bool)` RPC endpoint analogous to `set_ban`, so operators can revoke whitelist status at runtime without restarting.
3. In `try_dial_whitelist`, skip peers whose `PeerId` is no longer present in `whitelist_peers`.

```rust
// Suggested addition to PeerRegistry
pub fn remove_whitelist_peer(&mut self, peer_id: &PeerId) -> bool {
    if self.whitelist_peers.remove(peer_id) {
        // Downgrade any live session for this peer
        for peer in self.peers.values_mut() {
            if extract_peer_id(&peer.connected_addr).as_ref() == Some(peer_id) {
                peer.is_whitelist = false;
            }
        }
        return true;
    }
    false
}
```

---

### Proof of Concept

1. Configure a CKB node with `whitelist_peers = ["/ip4/1.2.3.4/tcp/8115/p2p/<PEER_ID>"]`.
2. Start the node; the peer connects and `is_whitelist = true` is set on its session.
3. The peer begins sending malformed flatbuffer sync messages.
4. The sync protocol calls `ban_peer(session_id, duration, reason)` → `ban_session` → hits `.filter(|peer| !peer.is_whitelist)` → returns `None` → ban is silently dropped.
5. `report_session` is similarly skipped; no score penalty accumulates.
6. If the operator disconnects the peer via `disconnect_peer` RPC, `OutboundPeerService::try_dial_whitelist` reconnects it on the next tick (default interval).
7. The only escape is `kill -SIGTERM <ckb>`, edit `ckb.toml` to remove the peer, restart — introducing downtime. [7](#0-6) [8](#0-7)

    pub(crate) fn ban_session(
        &self,
        p2p_control: &ServiceControl,
        session_id: SessionId,
        duration: Duration,
        reason: String,
    ) {
        if let Some(addr) = self.with_peer_registry(|reg| {
            reg.get_peer(session_id)
                .filter(|peer| !peer.is_whitelist)
                .map(|peer| peer.connected_addr.clone())
        }) {
            info!(
                "Ban peer {:?} for {} seconds, reason: {}",
                addr,
                duration.as_secs(),
                reason
            );
            if let Some(metrics) = ckb_metrics::handle() {
                metrics.ckb_network_ban_peer.inc();
            }
            if let Some(peer) = self.with_peer_registry_mut(|reg| reg.remove_peer(session_id)) {
                let message = format!("Ban for {} seconds, reason: {}", duration.as_secs(), reason);
                self.peer_store.lock().ban_addr(
                    &peer.connected_addr,
                    duration.as_millis() as u64,
                    reason,
                );
                if let Err(err) =
                    disconnect_with_message(p2p_control, peer.session_id, message.as_str())
                {
                    debug!("Disconnect failed {:?}, error: {:?}", peer.session_id, err);
                }
            }
        } else {
            debug!(
                "Ban session({}) failed: not found in peer registry or it is on the whitelist",
                session_id
            );
        }
    }
