### Title
Unconstrained Code Execution via Witness-Sourced `exec` Syscall Enables Lock Script Authorization Bypass — (File: `script/src/syscalls/exec.rs`)

---

### Summary

The CKB `exec` syscall (V1) allows a running lock or type script to replace itself with arbitrary executable code loaded directly from a transaction **witness**. Witnesses are fully controlled by the transaction submitter. There is no hash verification of the witness-sourced code against the script's `code_hash`. Any lock script that calls `exec` with `place=1` (Witness) hands full control of its authorization decision to the transaction submitter, who can supply any RISC-V ELF binary that returns exit code `0` to unconditionally satisfy the lock.

---

### Finding Description

In `script/src/syscalls/exec.rs`, the `Exec::ecall` function dispatches on the `place` register (A2). When `place == 1`, the `Place::Witness` branch is taken:

```rust
Place::Witness => {
    let witness = self.fetch_witness(source, index as usize);
    ...
    witness.raw_data()   // raw bytes from the transaction witness
}
```

The raw witness bytes are then loaded directly into the VM as an ELF binary:

```rust
machine.reset(max_cycles);
machine.set_cycles(cycles);
match machine.load_elf(&data, true) { ... }
```

The `data` here is `witness.raw_data()` — bytes supplied by the transaction submitter with **zero integrity verification** against the script's `code_hash` or any other committed hash. The machine is fully reset and the new program runs in place of the original script. Its exit code becomes the authorization result for the lock.

This is structurally identical to Marmo's `delegatecall` to a caller-chosen `_implementation`:

| Marmo | CKB |
|---|---|
| `relay` accepts `_implementation` address from relayer | `exec` accepts `(index, source, place=Witness)` from tx submitter |
| `delegatecall` runs implementation in wallet storage context | `exec` runs witness ELF in the same VM with same cycle budget and tx context |
| No code hash verification of `_implementation` | No hash verification of witness bytes against `code_hash` |
| Malicious impl can overwrite signer, drain ETH, selfdestruct | Malicious witness can return 0, bypassing the lock unconditionally |

The `fetch_witness` helper resolves the witness from `Source::Transaction(Input)`, `Source::Transaction(Output)`, `Source::Group(Input)`, or `Source::Group(Output)` — all of which are attacker-supplied fields in the transaction.

The `Place` enum and its parsing confirm `1` is the valid Witness variant:

```rust
fn parse_from_u64(i: u64) -> Result<Place, Error> {
    match i {
        0 => Ok(Place::CellData),
        1 => Ok(Place::Witness),
        _ => Err(...)
    }
}
```

The syscall is registered unconditionally for `ScriptVersion::V1` in `generator.rs`:

```rust
if script_version == &ScriptVersion::V1 {
    syscalls.push(Box::new(Exec::new(sg_data)));
}
```

And for `ScriptVersion::V2` the `ExecV2` variant (`exec_v2.rs`) also accepts a `DataPieceId` that can resolve to a witness via `DataPieceId::try_from((source, index, place))`, preserving the same attack surface.

---

### Impact Explanation

A lock script that calls `exec` with `place=1` (Witness) delegates its entire authorization decision to the transaction submitter. The submitter provides a RISC-V ELF binary in the witness that:

- Returns exit code `0` → lock is satisfied, cell is spent regardless of the intended authorization logic (signature check, multisig, timelock, etc.)
- Reads transaction fields via syscalls and performs arbitrary computation within the cycle budget

The cell owner loses control of their cell. Any CKB capacity (CKByte) locked by such a script can be stolen by any transaction submitter who knows the script uses this pattern. This is the direct CKB analog of Marmo's "overwrite the signer / transfer ETH to an attacker-controlled address."

---

### Likelihood Explanation

The `exec`-from-witness pattern is an explicitly supported, documented, and tested feature of CKB Script V1/V2. The test `check_exec_from_witness` in `script/src/verify/tests/ckb_latest/features_since_v2021.rs` demonstrates it working correctly. Any script author who uses `exec` with `place=Witness` to implement a "pluggable verifier" or "upgradeable logic" pattern — a natural design choice given the feature exists — creates a cell that any transaction submitter can spend. The risk is non-obvious because the script's `code_hash` appears to commit to specific code, but the actual code that runs at authorization time is not the dep-cell binary — it is the witness.

---

### Recommendation

1. **Documentation**: The CKB RFC and syscall documentation should prominently warn that `exec` with `place=Witness` transfers full authorization control to the transaction submitter. The risk is equivalent to `delegatecall` to a caller-chosen address.
2. **Script-level mitigation**: Lock scripts that use `exec` from a witness should hash the witness bytes and compare against a committed hash in `script.args` before calling `exec`, analogous to Marmo's proposed `EXTCODEHASH` whitelist.
3. **Protocol-level consideration**: The CKB VM could optionally expose a "verified exec" variant that requires the caller to supply an expected hash, which the syscall checks before loading the ELF.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Victim deploys a lock script (e.g., a "pluggable verifier") whose binary calls `exec(0, CKB_SOURCE_INPUT, 1 /*Witness*/, 0, 0, NULL)` — loading code from witness index 0 of the spending transaction.
2. Victim creates a cell locked by this script.
3. Attacker constructs a transaction spending the victim's cell, with witness[0] set to a minimal RISC-V ELF that executes `li a0, 0; ecall` (exit 0).
4. The CKB node processes the transaction: `Exec::ecall` takes the `Place::Witness` branch at line 128–136 of `exec.rs`, loads the attacker's ELF via `machine.load_elf(&data, true)` at line 180, and the machine exits with code 0.
5. The lock is satisfied. The cell is spent. The victim's CKB capacity is transferred to the attacker's output.
