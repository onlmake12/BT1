### Title
Silent `u128 → u64` Truncation in DAO Maximum Withdrawal Calculation — (File: `util/dao/src/lib.rs`)

### Summary
`DaoCalculator::calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a `u128` intermediate value but casts it to `u64` with a bare `as u64` — a silent truncating cast that produces a drastically wrong (small) result if the value exceeds `u64::MAX`. Every other analogous `u128 → u64` narrowing in the same file uses `u64::try_from(…).map_err(|_| DaoError::Overflow)?`, making this a clear inconsistency with a concrete arithmetic-safety impact.

### Finding Description
In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the maximum CKB a DAO depositor may withdraw:

```rust
// lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← bare truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

`withdraw_counted_capacity` is a `u128`. The `as u64` cast silently keeps only the low 64 bits. If the value exceeds `u64::MAX ≈ 1.84 × 10¹⁹`, the result wraps to a much smaller number — exactly the same class of bug as the reference report's `uint256(-amount)`.

Every other `u128 → u64` narrowing in the same file uses the safe pattern:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

`calculate_maximum_withdraw` is the sole exception.

### Impact Explanation
When `withdraw_counted_capacity` overflows `u64`, the truncated value is silently used as the computed maximum withdrawal capacity. Two downstream consequences follow:

1. **DAO withdrawal denial** — The DAO type script calls `calculate_maximum_withdraw` to bound the amount a depositor may claim. A truncated (too-small) result causes a legitimate withdrawal to be rejected as exceeding the computed maximum, permanently locking the depositor's funds in the cell until the bug is patched.

2. **DAO accounting corruption** — `withdrawed_interests` (called from `dao_field_with_current_epoch`) sums results of `transaction_maximum_withdraw`, which itself calls `calculate_maximum_withdraw`. A truncated interest figure causes the DAO `s` (secondary issuance accumulator) field written into every subsequent block header to be inflated, corrupting the global DAO state for all future depositors. [5](#0-4) [6](#0-5) 

### Likelihood Explanation
The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX  ≈  1.84 × 10¹⁹
```

`counted_capacity` is bounded by the total CKB supply (≈ 3.36 × 10¹⁸ shannons), so the ratio `withdrawing_ar / deposit_ar` must exceed ≈ 5.5 for overflow to occur. The genesis `ar` is `10¹⁶`; mainnet block 5892 shows `ar ≈ 10,000,616,071,298,000`, a growth of ≈ 104,564 per block. [7](#0-6) 

At that rate (which decreases as secondary issuance halves every 4 years), reaching `5.5 × 10¹⁶` would take many decades of continuous chain operation. Likelihood is therefore **low** but non-zero for a chain designed to run indefinitely, and the fix is a one-line change already demonstrated elsewhere in the same file.

### Recommendation
Replace the bare cast on line 156 with the same checked conversion used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with lines 204, 245, 258):
Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?
``` [8](#0-7) 

### Proof of Concept
Construct a scenario where `counted_capacity × withdrawing_ar / deposit_ar` exceeds `u64::MAX`:

```
deposit_ar        = 10_000_000_000_000_000   (genesis ar, u64)
withdrawing_ar    = 55_000_000_000_000_000   (ar after sufficient time, u64)
counted_capacity  = 3_360_000_000_000_000_000 shannons  (≈ total CKB supply)

withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 55_000_000_000_000_000
    / 10_000_000_000_000_000
  = 18_480_000_000_000_000_000   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_480_000_000_000_000_000 mod 2^64
  = 33_255_926_290_448_385       ← silently truncated, ~550× too small
```

A depositor who locked the full CKB supply at genesis `ar` and attempts to withdraw at `withdrawing_ar = 5.5 × 10¹⁶` would have their maximum withdrawal computed as ≈ 33 quadrillion shannons instead of ≈ 18.48 quintillion shannons. The DAO type script would reject the withdrawal as exceeding this fabricated ceiling, permanently denying the depositor their funds.

        let input_capacities = rtxs.try_fold(Capacity::zero(), |capacities, rtx| {

                },
            )?;

        })?;
