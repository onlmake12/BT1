### Title
PoW Difficulty Not Enforced Against Epoch Consensus in Compact Block Relay Header Verification — (`pow/src/eaglesong_blake2b.rs`, `verification/src/header_verifier.rs`)

---

### Summary

The `HeaderVerifier` used in the compact block relay path verifies PoW only against the header's **self-declared** `compact_target`, without checking that `compact_target` matches the epoch's consensus-mandated difficulty. An unprivileged peer can craft a compact block with `compact_target=0x20ffffff` (minimum difficulty), find a valid nonce in microseconds, and relay it via P2P. The block passes all relay-path header checks and is inserted into `pending_compact_blocks`, enabling cheap resource exhaustion.

---

### Finding Description

**Step 1 — `compact_to_target(0x20ffffff)` behavior**

`compact_to_target` in `util/types/src/utilities/difficulty.rs`:

```rust
let exponent = compact >> 24;          // 0x20 = 32
let mantissa = compact & 0x00ff_ffff;  // 0xffffff
// exponent > 3, so:
ret = mantissa << (8 * (exponent - 3)); // 0xffffff << 232
let overflow = !mantissa.is_zero() && (exponent > 32); // 32 > 32 = false
``` [1](#0-0) 

The test suite confirms `target_to_compact(U256::max_value()) == 0x20ffffff`, meaning `compact_to_target(0x20ffffff)` returns `(0xffffff << 232, false)`. The target is not zero and overflow is false, so the guard in `EaglesongBlake2bPowEngine::verify` passes. [2](#0-1) 

**Step 2 — PoW check is trivially satisfiable**

`EaglesongBlake2bPowEngine::verify` checks `output <= block_target`. With `block_target = 0xffffff << 232`, any 256-bit hash whose top 24 bits are `≤ 0xffffff` passes. Since `0xffffff` is the maximum 24-bit value, approximately `(2^24 − 1)/2^24 ≈ 99.9999940%` of all possible hashes satisfy this condition. Nonce=0 passes with overwhelming probability. [3](#0-2) 

**Step 3 — `HeaderVerifier` in the relay path does not check `compact_target` against the epoch**

`CompactBlockProcess::execute` calls `contextual_check`, which constructs a `HeaderVerifier` and calls `verify`: [4](#0-3) 

`HeaderVerifier::verify` runs four sub-verifiers: [5](#0-4) 

The `EpochVerifier` used here (from `verification/src/header_verifier.rs`) only checks epoch number continuity (`is_successor_of`), **not** `compact_target`: [6](#0-5) 

**Step 4 — The compact_target check exists only in the contextual full-block verifier**

The contextual `EpochVerifier` in `verification/contextual/src/contextual_block_verifier.rs` does enforce `compact_target == epoch.compact_target()`: [7](#0-6) 

But this verifier is only invoked when a fully-reconstructed block is submitted to the chain — it is never reached for a compact block that is still pending (missing transactions). The relay path's `HeaderVerifier` is the only gate applied before insertion into `pending_compact_blocks`.

**Step 5 — Insertion into `pending_compact_blocks`**

After `contextual_check` returns `Status::ok()`, the compact block is inserted into the pending pool via `missing_or_collided_post_process`: [8](#0-7) 

---

### Impact Explanation

An attacker with no hashpower can:
1. Craft a compact block with `compact_target=0x20ffffff` and a valid epoch number matching the current tip's parent epoch.
2. Find a satisfying nonce in microseconds (any nonce works with ~99.99% probability).
3. Relay the compact block to any CKB node via P2P.
4. The block passes all relay-path checks and enters `pending_compact_blocks`.
5. By repeating with different block hashes (varying nonce, timestamp, or parent), the attacker can flood the pending pool at essentially zero cost, exhausting memory and CPU on victim nodes.

---

### Likelihood Explanation

The attack requires only a P2P connection to a CKB node and trivial computation. No hashpower, no privileged access, and no special knowledge is required. The crafted compact block must have a valid parent (known block hash) and a plausible epoch number, both of which are publicly observable from the chain.

---

### Recommendation

In `HeaderVerifier::verify` (or in `contextual_check` before inserting into `pending_compact_blocks`), add a check that the header's `compact_target` matches the epoch's consensus-mandated `compact_target`. This requires passing the `EpochExt` (or at minimum its `compact_target`) through `HeaderFields`, or performing the check inline in `contextual_check` after resolving the parent epoch.

Alternatively, mirror the check already present in the contextual `EpochVerifier`:

```rust
if self.epoch.compact_target() != actual_compact_target {
    return Err(EpochError::TargetMismatch { ... }.into());
}
``` [7](#0-6) 

---

### Proof of Concept

1. Connect to a CKB node via P2P.
2. Observe the current tip block hash and its epoch (`EpochNumberWithFraction`).
3. Construct a `RawHeader` with:
   - `parent_hash` = current tip hash
   - `number` = tip.number + 1
   - `epoch` = valid successor epoch fraction
   - `compact_target` = `0x20ffffff`
   - `timestamp` = current time (> median of last 37 blocks)
4. Set `nonce = 0`. Compute `eaglesong(blake2b(pow_message(pow_hash, nonce)))`. With ~99.99% probability the output is `≤ 0xffffff << 232` and the block is valid.
5. Wrap in a `CompactBlock` message with a prefilled cellbase transaction.
6. Send via the relay protocol.
7. Assert the node does not return `CompactBlockHasInvalidHeader` and the block hash appears in `pending_compact_blocks`.
8. Repeat with different nonces/timestamps to fill the pool.

pub fn compact_to_target(compact: u32) -> (U256, bool) {
    let exponent = compact >> 24;

        ret <<= 8 * (exponent - 3);
    }

    let overflow = !mantissa.is_zero() && (exponent > 32);
    (ret, overflow)
