### Title
Fee Admission Check Uses Only Serialized Size, Not Cycle-Weighted Cost, Enabling Cheap Script-Execution DoS — (File: `tx-pool/src/util.rs`)

---

### Summary

The tx-pool's pre-admission fee check (`check_tx_fee`) computes the minimum required fee using only the transaction's serialized byte size, not its actual computational weight (which is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because script execution happens *after* this check, and no second weight-based fee check is performed after cycles are known, an attacker can submit transactions with tiny serialized size but near-maximum cycle consumption while paying only the size-based minimum fee. This forces every receiving node to execute up to `max_block_cycles` VM cycles per transaction for a fee that is orders of magnitude below what the actual resource cost demands.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

This check runs inside `pre_check`, which is called before `verify_rtx` (script execution):

```rust
let (ret, snapshot) = self.pre_check(&tx).await;
// ...
let verified_ret = verify_rtx(...).await;
``` [2](#0-1) 

After `verify_rtx` returns the actual consumed cycles, a `TxEntry` is created with those cycles, but **no second fee check** is performed against the cycle-weighted cost:

```rust
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
``` [3](#0-2) 

The actual fee rate stored in the pool entry uses the correct weight formula:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

where:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [5](#0-4) 

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This weight is used for eviction scoring and fee-rate statistics, but **never enforced as a gate before or after script execution**.

---

### Impact Explanation

An attacker crafts a transaction with:
- Serialized size ≈ 200 bytes (small enough to pass the size-based fee check cheaply)
- A lock script that consumes near-maximum cycles (e.g., 70,000,000 cycles per `max_tx_verify_cycles` default)
- Fee = `min_fee_rate × tx_size / 1000 = 1000 × 200 / 1000 = 200 shannons`

The size-based pre-check passes (`fee ≥ min_fee_rate × size`). The node then runs 70M VM cycles of script verification. The actual weight would be `max(200, 70_000_000 × 0.000_170_571_4) ≈ 11,940` bytes, meaning the transaction is ~60× underpriced relative to its true resource cost. The transaction is accepted into the pool (fee satisfies the size-only gate), and the attacker can repeat this at near-zero cost to saturate node CPU across the entire network. [6](#0-5) 

---

### Likelihood Explanation

The attack is reachable by any unprivileged actor via the `send_transaction` RPC endpoint or P2P transaction relay. No special role, key, or hashpower is required. The `process_tx` and `resumeble_process_tx` code paths both call `non_contextual_verify` then `pre_check` (size-only fee gate) then `verify_rtx` (full script execution) with no cycle-weighted fee enforcement at any point. [7](#0-6) [8](#0-7) 

The default `min_fee_rate = 1000 shannons/KB` and `max_tx_verify_cycles = 70,000,000` are public consensus parameters, making the attack fully predictable and repeatable. [9](#0-8) 

---

### Recommendation

After `verify_rtx` returns the actual consumed cycles, perform a second fee check using the cycle-weighted transaction weight before accepting the entry into the pool:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors how `TxEntry::fee_rate()` already computes the effective fee rate and ensures that the admission gate is consistent with the actual resource cost of script execution. [10](#0-9) 

---

### Proof of Concept

1. Deploy a lock script cell whose script body loops for exactly `N` cycles (e.g., `N = 69,000,000`).
2. Construct a transaction spending a cell locked by that script:
   - Serialized size ≈ 200 bytes
   - Fee = `ceil(1000 × 200 / 1000)` = 200 shannons (satisfies size-only gate)
3. Submit via `send_transaction` RPC.
4. Observe: `pre_check` passes (200 shannons ≥ size-based minimum). Node runs 69M VM cycles. Transaction is accepted into the pool.
5. Actual fee rate = `200 / max(200, 69_000_000 × 0.000_170_571_4) × 1000 ≈ 17 shannons/KW` — far below `min_fee_rate = 1000 shannons/KW`.
6. Repeat in a tight loop from multiple connections to saturate node verification workers. [11](#0-10) [12](#0-11)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

pub(crate) async fn verify_rtx(
    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: Arc<TxVerifyEnv>,
    cache_entry: &Option<CacheEntry>,
    max_tx_verify_cycles: Cycle,
    command_rx: Option<&mut watch::Receiver<ChunkCommand>>,
) -> Result<Completed, Reject> {
    let consensus = snapshot.cloned_consensus();
    let data_loader = snapshot.as_data_loader();

    if let Some(completed) = cache_entry {
        TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
            .verify()
            .map(|_| *completed)
            .map_err(Reject::Verification)
    } else if let Some(command_rx) = command_rx {
        ContextualTransactionVerifier::new(
            Arc::clone(&rtx),
            consensus,
            data_loader,
            Arc::clone(&tx_env),
        )
        .verify_with_pause(max_tx_verify_cycles, command_rx)
        .await
        .and_then(|result| {
            DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
                .verify()?;
            Ok(result)
        })
        .map_err(Reject::Verification)
    } else {
        block_in_place(|| {
            ContextualTransactionVerifier::new(Arc::clone(&rtx), consensus, data_loader, tx_env)
                .verify(max_tx_verify_cycles, false)
                .and_then(|result| {
                    DaoScriptSizeVerifier::new(
                        rtx,
                        snapshot.cloned_consensus(),
                        snapshot.as_data_loader(),
                    )
                    .verify()?;
                    Ok(result)
                })
                .map_err(Reject::Verification)
        })
    }
}

        let (ret, snapshot) = self.pre_check(&tx).await;

        let (tip_hash, rtx, status, fee, tx_size) = try_or_return_with_snapshot!(ret, snapshot);

            Arc::clone(&rtx),

            max_cycles,
            command_rx,

        if let Some(declared) = declared_cycles

                snapshot,

#[deprecated(

pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
