### Title
Unbounded `pending_get_block_proposals` DashMap Growth via Peer-Supplied `GetBlockProposal` Messages — (File: `sync/src/types/mod.rs`)

---

### Summary

The `insert_get_block_proposals` function inserts peer-supplied `ProposalShortId` values into the shared `pending_get_block_proposals: DashMap` without any hard size cap. An unprivileged P2P peer can flood this map with fabricated proposal IDs that do not exist in the tx pool, causing unbounded memory growth between periodic drain cycles. The drain itself clones the entire map, temporarily doubling peak memory. Sustained flooding from multiple peers can exhaust node memory and crash the process.

---

### Finding Description

**Root cause — `insert_get_block_proposals` has no size guard:**

`GetBlockProposalProcess::execute()` in `sync/src/relayer/get_block_proposal_process.rs` validates only the per-message proposal count against `max_block_proposals_limit * max_uncles_num` (≤ 3 000 on mainnet). Proposals absent from the tx pool are then forwarded unconditionally to `insert_get_block_proposals`:

```rust
// sync/src/relayer/get_block_proposal_process.rs  lines 68-77
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

self.relayer
    .shared()
    .state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [1](#0-0) 

`insert_get_block_proposals` performs no size check before inserting into the shared `DashMap`:

```rust
// sync/src/types/mod.rs  lines 1594-1600
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [2](#0-1) 

The map is declared as an unbounded `DashMap::new()` with no capacity argument:

```rust
// sync/src/types/mod.rs  line 1021
pending_get_block_proposals: DashMap::new(),
``` [3](#0-2) 

The only relief valve is `drain_get_block_proposals`, called from `prune_tx_proposal_request` on the relayer timer (≈10 Hz):

```rust
// sync/src/types/mod.rs  lines 1586-1592
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();   // full clone
    self.pending_get_block_proposals.clear();
    ret
}
``` [4](#0-3) 

The clone is O(n) in both time and memory — it **doubles** peak memory usage at every drain tick when the map is large.

**Attacker-controlled entry path:**

The `GetBlockProposal` relay message is accepted from any connected peer. The relayer rate-limits it to 30 req/s per peer:

```rust
// sync/src/relayer/mod.rs  lines 116-123
if should_check_rate
    && self
        .rate_limiter
        .check_key(&(peer, message.item_id()))
        .is_err()
{
    return StatusCode::TooManyRequests.with_context(message.item_name());
}
``` [5](#0-4) 

With 30 req/s × 3 000 proposals/req × 10 peers = **900 000 entries inserted per second**. The drain fires 10 times/s, so the steady-state map size is ≈ 90 000 entries per peer. Each entry is a `ProposalShortId` (10 B) plus `HashSet<PeerIndex>` overhead (~50 B). With 10 attacker peers the map holds ~9 M entries ≈ **540 MB** between drains; the clone at drain time spikes to **~1 GB**.

The three previously fixed analogous issues confirm this class of bug is real in this codebase:

- `#3093: Resolve inflight proposals memory bloat issue`
- `#3094: Fix inflight block potential memory bloat issues`
- `#3110: Fix pending compact block memory bloat on abnormal flow` [6](#0-5) 

`pending_get_block_proposals` received no equivalent fix.

---

### Impact Explanation

- **Memory exhaustion → OOM kill** of the CKB node process, equivalent to a complete local node shutdown.
- The `drain_get_block_proposals` clone creates a transient memory spike that can trigger OOM even before steady-state is reached.
- CPU time spent cloning and clearing the large DashMap delays the relayer timer, allowing further accumulation — a positive feedback loop.
- Impact matches the target scope: network not being able to confirm new transactions (node offline).

---

### Likelihood Explanation

- Requires only a standard P2P connection — no authentication, no stake, no miner role.
- Fabricating `ProposalShortId` values absent from the tx pool is trivial (random 10-byte values).
- The rate limiter (30 req/s per peer) slows but does not prevent the attack; multiple Sybil connections multiply the effect linearly.
- No existing hard cap or eviction policy guards `pending_get_block_proposals`.

---

### Recommendation

Add a hard cap inside `insert_get_block_proposals`:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    const MAX_PENDING_PROPOSALS: usize = 10_000;
    if self.pending_get_block_proposals.len() >= MAX_PENDING_PROPOSALS {
        return; // or return an error / ban the peer
    }
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

Additionally, replace the full-clone drain with an atomic swap to avoid the transient memory doubling.

---

### Proof of Concept

1. Connect to a victim CKB node as a standard P2P peer.
2. Repeatedly send `GetBlockProposal` relay messages containing 3 000 random `ProposalShortId` values (none present in the victim's tx pool) at 30 messages/second.
3. Observe `pending_get_block_proposals` growing without bound between drain ticks.
4. With ≥ 10 such connections, peak RSS of the victim process climbs by hundreds of MB per second until OOM.
5. The victim node crashes; it cannot relay or confirm transactions.
