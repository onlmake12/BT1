### Title
Wrong Row Variable in `script_exists_in_output` Causes Type-Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

A copy-paste error at line 252 of `script_exists_in_output` causes the function to re-evaluate `row_lock` (the lock-script EXISTS result) instead of `row_type` (the type-script EXISTS result) when deciding whether a script is still referenced. On PostgreSQL this silently returns `false` for any script that is a type-script-only reference, causing it to be deleted from the `script` table during every rollback. The corruption is permanent and affects all subsequent type-script-based indexer queries.

### Finding Description

`script_exists_in_output` in [1](#0-0)  performs two `SELECT EXISTS` queries:

- `row_lock` — checks whether `script_id` appears as a `lock_script_id` in any surviving output (lines 208–220).
- `row_type` — checks whether `script_id` appears as a `type_script_id` in any surviving output (lines 237–249).

The early-return guard at lines 223–235 means that if execution reaches line 252, `row_lock` is **definitively `false`**. The final match at line 252 should evaluate `row_type`, but instead reads:

```rust
// pg type is BOOLEAN
match row_lock.try_get::<bool, _>(0) {   // ← BUG: should be row_type
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [2](#0-1) 

**PostgreSQL path:** `EXISTS` returns a SQL `BOOLEAN`. `row_lock.try_get::<bool, _>(0)` succeeds with `Ok(false)` (because `row_lock` is already known to be false at this point). The function returns `Ok(false)` regardless of what `row_type` contains — `row_type` is never consulted.

**SQLite path:** `EXISTS` returns a `BIGINT`. `row_lock.try_get::<bool, _>(0)` fails with a type error, falling into the `Err(_)` arm, which correctly reads `row_type.get::<i64, _>(0) == 1`. SQLite is **not** affected.

The caller in `rollback_block` uses the return value to gate deletion:

```rust
if let Some(type_script_id) = type_script_id
    && !script_exists_in_output(type_script_id, tx).await?
{
    script_id_list_to_remove.push(type_script_id);
}
``` [3](#0-2) 

On PostgreSQL, `script_exists_in_output(type_script_id, …)` always returns `false` when the script is not also a lock-script, so every type-script ID from the rolled-back block is unconditionally pushed into `script_id_list_to_remove` and deleted from the `script` table via `remove_batch_by_blobs("script", …)`. [4](#0-3) 

### Impact Explanation

After the deletion, any surviving output whose `type_script_id` foreign key pointed to the now-deleted row is left with a dangling reference. All indexer query paths that join on `type_script_id` — `get_cells`, `get_cells_capacity`, `get_transactions` — will return empty or incorrect results for those scripts. The corruption is persistent (written to the DB inside the same transaction that commits the rollback) and accumulates across every subsequent reorg.

### Likelihood Explanation

Chain reorganizations are a routine part of CKB operation; they require no attacker. Any natural reorg on a PostgreSQL-backed rich-indexer node triggers the bug. The existing rollback test suite uses only SQLite (`connect_sqlite(MEMORY_DB)`) and therefore does not exercise the PostgreSQL code path where the bug manifests. [5](#0-4) [6](#0-5) 

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ← fix
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Add a PostgreSQL-backed rollback integration test that inserts two blocks sharing a type script, rolls back the second block, and asserts the shared script row still exists in the `script` table.

### Proof of Concept

1. Start a PostgreSQL-backed rich-indexer node.
2. Index block A containing output O1 with `type_script_id = S`.
3. Index block B (tip) containing output O2 with the same `type_script_id = S`.
4. Trigger a rollback of block B (reorg).
5. `rollback_block` calls `script_exists_in_output(S, tx)`.
6. `row_lock` query: `S` is not a `lock_script_id` → `false`.
7. Early-return guard not taken.
8. `row_type` query: `S` IS a `type_script_id` (O1 still exists) → `true`.
9. Line 252: `row_lock.try_get::<bool, _>(0)` → `Ok(false)` (PostgreSQL BOOLEAN, value is false).
10. Function returns `Ok(false)` — script incorrectly flagged for deletion.
11. Script row `S` is deleted from the `script` table.
12. Query `get_cells` with type-script filter for `S` → empty result, even though O1 still exists.

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN

        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
