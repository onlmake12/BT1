### Title
ELF Parsing and Memory Allocation Performed Before Cycle-Limit Enforcement in VM Boot Path — (`script/src/scheduler.rs`)

---

### Summary

In CKB's `Scheduler`, when a new VM is booted (either for the root script or via a `ckb_spawn` syscall), the expensive operations of ELF parsing and program loading into VM memory are performed **before** any cycle-limit check is applied to those costs. An unprivileged transaction submitter can craft a transaction that forces the node to perform this expensive work and then immediately fail the cycle check, making DoS attacks cheaper than the cycle accounting implies.

---

### Finding Description

The call chain for booting any VM is:

`run()` → `boot_root_vm_if_needed()` → `boot_vm()` → `load_vm_program()`

and for spawned VMs:

`iterate_inner()` → `iterate_process_results()` → `process_message_box()` → `boot_vm()` → `load_vm_program()`

Inside `boot_vm`, the following expensive operations execute with **no cycle pre-check**:

1. `create_dummy_vm()` — allocates a new RISC-V machine with all syscall handlers attached
2. `sc.load_data(...)` — reads the full ELF binary from cell data
3. `load_vm_program()`:
   - `parse_elf::<u64>(...)` — parses the ELF binary structure
   - `machine.load_program_with_metadata(...)` — maps ELF segments into VM memory (heap allocation proportional to program size)
   - `sc.mark_program(...)` — registers the program in the snapshot context

Only **after** all of the above does the cycle charge occur:

```rust
machine
    .inner_mut()
    .add_cycles_no_checking(transferred_byte_cycles(bytes))?;
``` [1](#0-0) 

This charge is applied to the new machine's internal counter via `add_cycles_no_checking`, which does **not** enforce the limit — it merely accumulates. The actual limit enforcement only happens in the **next** scheduler iteration when `set_max_cycles(limit_cycles)` is called on the machine before it runs:

```rust
vm.inner_mut().set_max_cycles(limit_cycles);
...
let result = vm.run();
``` [2](#0-1) 

If the machine's pre-loaded cycle count (from `transferred_byte_cycles`) already exceeds `limit_cycles`, the VM fails immediately on its first instruction — but all the expensive setup work has already been completed.

The `SPAWN_EXTRA_CYCLES_BASE` (100,000) and `SPAWN_YIELD_CYCLES_BASE` (800) charged in the spawn syscall itself are flat fees that do not cover the variable cost of loading a large ELF: [3](#0-2) 

For a 1 MB ELF binary, `transferred_byte_cycles(1_048_576) = 262_144` cycles (at `BYTES_PER_CYCLE = 4`). An attacker sets the transaction cycle limit to just above `SPAWN_EXTRA_CYCLES_BASE + SPAWN_YIELD_CYCLES_BASE = 100_800` but below `100_800 + 262_144 = 362_944`, causing full ELF parsing and memory allocation to occur before the cycle check fires. [4](#0-3) 

The same pattern applies to the root VM boot path in `boot_root_vm_if_needed`, where the root script's ELF is fully parsed and loaded before any cycle limit is enforced: [5](#0-4) 

---

### Impact Explanation

A transaction submitter (unprivileged) can submit many transactions that each trigger full ELF parsing and VM memory allocation before being rejected for cycle exhaustion. The node performs O(ELF size) work — heap allocation, ELF segment mapping — for each such transaction, but the attacker only pays for the flat spawn fee cycles. This makes the attack cheaper than the cycle model intends, enabling sustained CPU and memory pressure on tx-pool verification workers and block verification, slowing chain processing.

---

### Likelihood Explanation

The attack requires only the ability to submit transactions to the tx-pool, which is open to any network peer. The attacker needs to craft a transaction with a minimal lock script that calls `ckb_spawn` pointing to a large cell dep ELF, with a cycle limit tuned to the gap between the flat spawn fee and the ELF loading cost. No special privileges, keys, or majority hashpower are required. The attack is repeatable and cheap relative to the node work induced.

---

### Recommendation

Before calling `boot_vm` (or at the start of `load_vm_program`), check whether the remaining cycle budget is sufficient to cover at least the minimum expected ELF loading cost. Concretely:

- In `process_message_box` for `Message::Spawn`, compute the expected `transferred_byte_cycles` from the known program size (already fetched via `sc.load_data(&data_piece_id, 0, 0)` in the spawn syscall for bounds validation) and verify the remaining cycle budget covers it before proceeding to `boot_vm`.
- Alternatively, charge `transferred_byte_cycles` to `iteration_cycles` (the scheduler-level counter that is checked against the limit in `iterate_outer`) rather than to the new machine's unchecked counter, so the cost is enforced in the same iteration that triggers the spawn. [6](#0-5) 

---

### Proof of Concept

1. Deploy a large cell dep containing a valid ELF binary of size ~1 MB.
2. Submit a transaction whose lock script is a minimal program that immediately calls `ckb_spawn(cell_dep_index, ...)` targeting the large ELF.
3. Set `max_cycles` on the transaction to `~150_000` (above `SPAWN_EXTRA_CYCLES_BASE + SPAWN_YIELD_CYCLES_BASE = 100_800`, below `100_800 + transferred_byte_cycles(1_048_576) = 362_944`).
4. The node executes the spawning script, issues the spawn syscall (charges 100,800 cycles, yields), then enters `process_message_box` → `boot_vm`:
   - `create_dummy_vm` allocates the machine
   - `load_data` reads the 1 MB ELF
   - `parse_elf` parses it
   - `load_program_with_metadata` maps it into VM memory
   - `add_cycles_no_checking(262_144)` charges to the new machine
5. Next iteration: `set_max_cycles(remaining ~49_200)` is called; the machine already has 262,144 cycles accumulated; it fails immediately with `CyclesExceeded`.
6. Transaction is rejected, but the node has performed full ELF parsing and memory allocation for the spawned VM.
7. Repeat with many such transactions to sustain CPU/memory pressure on the node. [7](#0-6) [8](#0-7)

    fn boot_vm(&mut self, location: &DataLocation, args: VmArgs) -> Result<VmId, Error> {
        let id = self.next_vm_id;
        self.next_vm_id += 1;
        let (context, mut machine) = self.create_dummy_vm(&id)?;
        let (program, _) = {
            let sc = context.snapshot2_context.lock().expect("lock");
            sc.load_data(&location.data_piece_id, location.offset, location.length)?
        };
        self.load_vm_program(&context, &mut machine, location, program, args)?;
        // Newly booted VM will be instantiated by default
        while self.instantiated.len() >= MAX_INSTANTIATED_VMS {
            // Instantiated is a BTreeMap, first_entry will maintain key order
            let id = *self
                .instantiated
                .first_entry()
                .ok_or_else(|| Error::Unexpected("Map should not be empty".to_string()))?
                .key();
            self.suspend_vm(&id)?;
        }

        self.instantiated.insert(id, (context, machine));
        self.states.insert(id, VmState::Runnable);

        Ok(id)
    }

    fn load_vm_program(
        &mut self,
        context: &VmContext<DL>,
        machine: &mut M,
        location: &DataLocation,
        program: Bytes,
        args: VmArgs,
    ) -> Result<u64, Error> {
        let metadata = parse_elf::<u64>(&program, machine.inner_mut().version())?;
        let bytes = match args {
            VmArgs::Reader { vm_id, argc, argv } => {
                let (_, machine_from) = self.ensure_get_instantiated(&vm_id)?;
                let argc = Self::u64_to_reg(argc);
                let argv = Self::u64_to_reg(argv);
                let argv =
                    FlattenedArgsReader::new(machine_from.inner_mut().memory_mut(), argc, argv);
                machine.load_program_with_metadata(&program, &metadata, argv)?
            }
            VmArgs::Vector(data) => {
                machine.load_program_with_metadata(&program, &metadata, data.into_iter().map(Ok))?
            }
        };
        let mut sc = context.snapshot2_context.lock().expect("lock");
        sc.mark_program(
            machine.inner_mut(),
            &metadata,
            &location.data_piece_id,
            location.offset,
        )?;
        machine
            .inner_mut()
            .add_cycles_no_checking(transferred_byte_cycles(bytes))?;
        Ok(bytes)
    }
