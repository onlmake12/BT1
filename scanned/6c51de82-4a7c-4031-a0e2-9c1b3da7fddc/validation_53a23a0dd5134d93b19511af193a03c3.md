### Title
Orphan Transaction Pool Admission Bypasses Fee Check, Enabling Free Griefing DoS — (`tx-pool/src/component/orphan.rs`, `tx-pool/src/process.rs`)

---

### Summary

The CKB orphan transaction pool (`OrphanPool`) accepts transactions with unresolvable inputs without performing any fee-rate check. Because `check_tx_fee` is only reached when `resolve_tx` succeeds, a remote peer can flood the 100-slot orphan pool with zero-fee, structurally-valid transactions that reference fabricated (non-existent) outpoints. This evicts legitimate orphan transactions at essentially zero cost, preventing them from being automatically promoted when their parent arrives.

---

### Finding Description

The tx-pool admission pipeline in `_process_tx` runs as follows:

1. `pre_check` is called, which calls `resolve_tx`. [1](#0-0) 
2. `check_tx_fee` is only invoked on the `Ok` branch of `resolve_tx`. If the inputs are unknown, `resolve_tx` returns `Err(Reject::Resolve(OutPointError::Unknown(...)))` and `check_tx_fee` is **never called**. [2](#0-1) 
3. `_process_tx` returns early with the error. `after_process` then checks `is_missing_input`, and if true, calls `add_orphan` unconditionally — with no fee gate. [3](#0-2) 

```rust
// tx-pool/src/util.rs
pub(crate) fn is_missing_input(reject: &Reject) -> bool {
    matches!(reject, Reject::Resolve(out_point_err) if out_point_err.is_unknown())
}
``` [4](#0-3) 

The orphan pool is capped at exactly **100 entries** (`DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`). When full, it evicts entries **randomly** (by taking the first key from the `HashMap` iterator): [5](#0-4) 

```rust
// tx-pool/src/component/orphan.rs
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
    let id = self.entries.keys().next().cloned().expect("bound checked");
    if let Some(entry) = self.remove_orphan_tx(&id) {
        evicted_txs.push(entry.tx.hash());
    }
}
``` [6](#0-5) 

An attacker can craft 100+ structurally-valid transactions (passing `non_contextual_verify`) that reference fabricated outpoints. These are relayed via the P2P relay protocol, admitted to the orphan pool without any fee check, and fill all 100 slots. Legitimate orphan transactions are randomly evicted.

---

### Impact Explanation

When a legitimate child transaction is evicted from the orphan pool, the node's `process_orphan_tx` callback — which automatically promotes orphans when their parent arrives — will not find the child. The child transaction is silently lost from the node's perspective. The user must detect this and resubmit. Repeated flooding keeps the orphan pool permanently saturated, preventing any orphan-based transaction chaining from working reliably on the targeted node.

---

### Likelihood Explanation

The attack is reachable by any unprivileged P2P peer. The cost is near-zero: the attacker only needs to construct and sign transactions with fake inputs — no on-chain fees are paid because these transactions are never mined. The orphan pool is small (100 slots), so filling it requires only 100 messages. The attack can be sustained continuously. The integration test template explicitly sets `min_fee_rate = 0`, and even with `min_fee_rate = 1000`, orphan admission still bypasses the check entirely. [7](#0-6) 

---

### Recommendation

Apply a fee-rate check before admitting a transaction to the orphan pool. One approach: perform a lightweight capacity check on the declared outputs vs. inputs (using only the resolvable portion) before calling `add_orphan`. Alternatively, require a minimum declared fee density (using the `declared_cycle` and transaction size already available at orphan admission time) and reject orphans that do not meet the threshold. A per-peer orphan admission rate limit would also raise the cost of sustained flooding.

---

### Proof of Concept

1. Attacker connects to a CKB node as a P2P peer via the relay protocol.
2. Attacker generates 100+ transactions, each spending a fabricated `OutPoint` (random tx hash, index 0). Each transaction is structurally valid (passes `non_contextual_verify`): correct molecule encoding, non-cellbase, within size limits.
3. Attacker relays each transaction with any `declared_cycle` value via `RelayTransactionHashes` / `GetRelayTransactions`.
4. Each transaction enters `_process_tx` → `pre_check` → `resolve_tx` fails with `OutPointError::Unknown` → `after_process` → `is_missing_input` → `add_orphan`. No fee check occurs. [8](#0-7) 
5. After 100 such transactions, `limit_size` in `OrphanPool::add_orphan_tx` randomly evicts legitimate orphan entries. [9](#0-8) 
6. When a legitimate parent transaction arrives, `process_orphan_tx` finds no matching child in the orphan pool. The child is lost and must be resubmitted by the user.

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
