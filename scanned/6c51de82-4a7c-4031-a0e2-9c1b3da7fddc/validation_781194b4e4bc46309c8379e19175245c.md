### Title
Incomplete `ScriptHashType` Version Constraint Allows Unactivated VM Versions in Output Type Scripts to Bypass Non-Contextual Verification — (`verification/src/transaction_verifier.rs`)

### Summary

The `ScriptHashTypeVerifier` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only on **output lock scripts**, leaving output **type scripts** unchecked. Because the `ScriptHashType` enum is defined to include future/unactivated variants `Data3`–`Data127` (values 6–254), a transaction sender can craft a transaction whose output type script carries an unactivated hash type (e.g., `Data3 = 6`). Such a transaction passes all non-contextual checks and is admitted to the tx-pool, but is permanently unexecutable: every attempt to include it in a block fails during contextual script execution with `InvalidScriptHashType`. This is the direct CKB analog of the Solidity `^0.8.19` pragma issue — an unconstrained version range is accepted at one layer but breaks at the execution layer.

---

### Finding Description

**Root cause — `ENABLED_SCRIPT_HASH_TYPE` is not applied to type scripts:**

`util/constant/src/consensus.rs` defines the currently activated hash types:

```rust
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // Data
    1u8, // Type
    2u8, // Data1
    4u8, // Data2
};
``` [1](#0-0) 

`util/gen-types/src/core.rs` expands the enum to include `Data3`–`Data127` (values 6–254) via `seq_macro`, and `TryFrom<u8>` accepts all even values and 1:

```rust
seq!(N in 3..=127 {
    pub enum ScriptHashType {
        Data = 0, Type = 1, Data1 = 2, Data2 = 4,
        #( Data~N = N << 1, )*
    }
});
``` [2](#0-1) 

`ScriptHashType::verify_value()` accepts all even values and 1, so `check_data()` passes for `Data3` (value 6): [3](#0-2) [4](#0-3) 

**The verifier only checks lock scripts:**

`ScriptHashTypeVerifier::verify()` iterates over `self.transaction.outputs()` and checks only `output.lock().hash_type()`. It never inspects `output.type_()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { ... }
        } ...
    }
    Ok(())
}
``` [5](#0-4) 

This verifier is the only place `ENABLED_SCRIPT_HASH_TYPE` is enforced for transactions: [6](#0-5) 

**Execution layer rejects the unactivated version:**

`select_version()` in `script/src/types.rs` handles only `Data`, `Data1`, `Data2`, and `Type`. Any other variant (e.g., `Data3`) falls through to the catch-all arm and returns `Err(ScriptError::InvalidScriptHashType(...))`:

```rust
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated, and is not permitted for use.",
        hash_type
    )));
}
``` [7](#0-6) 

---

### Impact Explanation

A transaction with an output whose **type script** uses `Data3` (value 6):

1. Passes `check_data()` (6 is even → `verify_value` returns `true`)
2. Passes `ScriptHashTypeVerifier` (only lock scripts are checked)
3. Is admitted to the tx-pool
4. Can never be committed to a block — every block containing it fails contextual script execution with `InvalidScriptHashType`

Consequences:
- **Tx-pool pollution**: Permanently unexecutable transactions occupy tx-pool slots, displacing legitimate transactions.
- **Miner resource waste**: A miner that includes such a transaction assembles a block that all peers reject, forfeiting the block reward.
- **Propagation overhead**: The transaction is relayed across the P2P network before being discovered as unexecutable.

---

### Likelihood Explanation

Any unprivileged transaction sender can craft such a transaction with zero cost beyond the standard fee. The `Data3` value (6) is a structurally valid even byte that passes all parsing and non-contextual checks. No special access, key material, or majority hashpower is required.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also check the hash type of each output's **type script** against `ENABLED_SCRIPT_HASH_TYPE`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        check_hash_type(output.lock().hash_type())?;
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the approach already used in `CellbaseVerifier`, which checks both the cellbase witness lock and each cellbase output lock against `ENABLED_SCRIPT_HASH_TYPE`. [8](#0-7) 

---

### Proof of Concept

1. Construct a transaction with one output whose **type script** has `hash_type = 6` (`Data3`):

```rust
let type_script = Script::new_builder()
    .code_hash(some_code_hash)
    .hash_type(ScriptHashType::Data3)  // value = 6, not in ENABLED_SCRIPT_HASH_TYPE
    .build();
let output = CellOutput::new_builder()
    .lock(valid_lock_script)           // lock uses Data or Type — passes verifier
    .type_(Some(type_script))
    .capacity(...)
    .build();
let tx = TransactionBuilder::default()
    .input(valid_input)
    .output(output)
    .output_data(Bytes::new())
    .build();
```

2. Submit via RPC `send_transaction`. The node runs `NonContextualTransactionVerifier`:
   - `ScriptHashTypeVerifier` checks only the lock script → passes.
   - Transaction is admitted to the tx-pool.

3. Attempt to include in a block. `select_version` is called for the type script → returns `Err(InvalidScriptHashType("Data3 has not been activated"))` → block rejected by all peers.

4. The transaction remains in the tx-pool indefinitely, occupying a slot.
