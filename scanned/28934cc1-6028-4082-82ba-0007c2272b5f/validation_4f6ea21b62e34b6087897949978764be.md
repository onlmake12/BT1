### Title
`packed::BlockReader::extension()` Panics on Malformed Extra Field in Peer-Supplied Blocks — (`util/gen-types/src/extension/shortcut.rs`)

---

### Summary

`packed::BlockReader::extension()` unconditionally calls `.unwrap()` when deserializing the block's first extra field as `packed::Bytes`. If a peer sends a block whose extra field is present but not a valid `packed::Bytes` encoding, this method panics. The owned counterpart `packed::Block::extension()` handles the same case gracefully with `.ok()`. The sync protocol's pre-filter only rejects blocks with more than one extra field, allowing a single malformed extra field to reach the panicking code path.

---

### Finding Description

In `util/gen-types/src/extension/shortcut.rs`, two methods expose the same optional block extension field with inconsistent error handling:

**`packed::Block::extension()` (owned, safe):**
```rust
pub fn extension(&self) -> Option<packed::Bytes> {
    self.extra_field(0)
        .and_then(|data| packed::Bytes::from_slice(&data).ok())  // graceful None
}
```

**`packed::BlockReader::extension()` (reader, panics):**
```rust
/// # Panics
/// Panics if the first extra field exists but not a valid BytesReader.
pub fn extension(&self) -> Option<packed::BytesReader<'_>> {
    self.extra_field(0)
        .map(|data| packed::BytesReader::from_slice(data).unwrap())  // PANIC
}
```

The block extension field is optional per the CKB protocol (RFC-0031). Blocks without it are valid; blocks with it must encode it as `packed::Bytes`. However, the molecule wire format does not enforce this at the deserialization layer — `from_compatible_slice` accepts any bytes in the extra field position. The sync receiver at `sync/src/synchronizer/mod.rs` only rejects blocks with **more than one** extra field:

```rust
if reader.has_extra_fields() || reader.block().count_extra_fields() > 1 {
    // ban peer
}
```

A block with exactly one extra field that is **not** valid `packed::Bytes` passes this guard. If `BlockReader::extension()` is subsequently called on such a block (e.g., during hash calculation in `util/gen-types/src/extension/calc_hash.rs`, which has a confirmed call site), the node panics.

---

### Impact Explanation

A Rust `panic!` in a non-`catch_unwind` context terminates the thread. In CKB's async runtime, an unhandled panic in a task causes that task to abort. Depending on whether the block-processing task is supervised, this can crash the node process entirely or silently drop the block-processing worker, halting chain progress. Either outcome is a remote denial-of-service against any full node reachable by the attacker.

---

### Likelihood Explanation

The attacker only needs to be a connected P2P peer. Crafting a block with a single extra field that is not valid `packed::Bytes` is trivial — any byte sequence that fails `packed::BytesReader::from_slice` suffices (e.g., a 4-byte field whose declared length does not match). The sync guard explicitly permits exactly one extra field, so the malformed block is not rejected at the message-parsing stage. The likelihood is **medium**: it requires a valid PoW header to pass header verification before the extension is inspected, but the PoW requirement is the only meaningful barrier.

---

### Recommendation

Replace `.unwrap()` with `.ok()` in `packed::BlockReader::extension()` to match the behavior of the owned `packed::Block::extension()`:

```rust
pub fn extension(&self) -> Option<packed::BytesReader<'_>> {
    self.extra_field(0)
        .and_then(|data| packed::BytesReader::from_slice(data).ok())
}
```

This makes the reader and owned versions consistent: both return `None` when the extra field is absent or not a valid `packed::Bytes` encoding, eliminating the panic path entirely.

---

### Proof of Concept

1. Craft a `SendBlock` message whose embedded block has exactly one extra field. The extra field bytes must fail `packed::BytesReader::from_slice` — for example, set the 4-byte molecule length prefix to a value larger than the actual byte slice.
2. Send the message to a target full node over the CKB sync protocol.
3. The sync receiver at `sync/src/synchronizer/mod.rs` passes the block (one extra field, not more than one).
4. `block_process.rs` forwards the block for processing.
5. When `packed::BlockReader::extension()` is called during hash calculation or extension verification, `packed::BytesReader::from_slice(data).unwrap()` panics.
6. The block-processing task aborts; the node halts chain progress or crashes. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    pub fn verify(&self, block: &BlockView) -> Result<(), Error> {
        let extra_fields_count = block.data().count_extra_fields();

        let mmr_active = self
            .context
            .consensus
            .rfc0044_active(self.parent.epoch().number());
        match extra_fields_count {
            0 => {
                if mmr_active {
                    return Err(BlockErrorKind::NoBlockExtension.into());
                }
            }
            1 => {
                let extension = if let Some(data) = block.extension() {
                    data
                } else {
                    return Err(BlockErrorKind::UnknownFields.into());
                };
                if extension.is_empty() {
                    return Err(BlockErrorKind::EmptyBlockExtension.into());
                }
                if extension.len() > 96 {
                    return Err(BlockErrorKind::ExceededMaximumBlockExtensionBytes.into());
                }
                if mmr_active {
                    if extension.len() < 32 {
                        return Err(BlockErrorKind::InvalidBlockExtension.into());
                    }

                    let chain_root = self
                        .chain_root_mmr
                        .get_root()
                        .map_err(|e| InternalErrorKind::MMR.other(e))?;
                    let actual_root_hash = chain_root.calc_mmr_hash();
                    let expected_root_hash =
                        Byte32::new_unchecked(extension.raw_data().slice(..32));
                    if actual_root_hash != expected_root_hash {
                        return Err(BlockErrorKind::InvalidChainRoot.into());
                    }
                }
            }
            _ => {
                return Err(BlockErrorKind::UnknownFields.into());
            }
        }

        let actual_extra_hash = block.calc_extra_hash().extra_hash();
        if actual_extra_hash != block.extra_hash() {
            return Err(BlockErrorKind::InvalidExtraHash.into());
        }
        Ok(())
    }
