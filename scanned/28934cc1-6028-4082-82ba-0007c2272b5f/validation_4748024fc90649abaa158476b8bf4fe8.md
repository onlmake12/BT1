### Title
Out-of-Bounds Panic in `ConfirmationFraction` Fee Estimator at `MAX_CONFIRM_BLOCKS` Boundary via Long-Lived Dropped Transaction - (File: `util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

`remove_unconfirmed_tx` in the `ConfirmationFraction` fee estimator unconditionally indexes into `confirm_blocks_to_failed_txs` using `tx_age - 1` even when `tx_age > MAX_CONFIRM_BLOCKS`. Since `confirm_blocks_to_failed_txs` has exactly `MAX_CONFIRM_BLOCKS = 1000` entries (valid indices 0–999), any `tx_age >= 1001` produces an out-of-bounds index, causing a Rust panic. An unprivileged tx-pool submitter can trigger this by submitting a transaction that remains in the pool for more than 1000 blocks and then having it dropped.

---

### Finding Description

`TxConfirmStat` is initialized with two parallel arrays of the same outer length `MAX_CONFIRM_BLOCKS = 1000`:

- `block_unconfirmed_txs: Vec<Vec<usize>>` — circular ring buffer, length 1000
- `confirm_blocks_to_failed_txs: Vec<Vec<f64>>` — failure counter, length 1000 (valid indices 0–999) [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, the code correctly branches on `tx_age >= block_unconfirmed_txs.len()` (i.e., `tx_age >= 1000`) to decide whether to decrement the ring buffer or `old_unconfirmed_txs`. However, the `count_failure` write to `confirm_blocks_to_failed_txs` is **outside** that branch and is not bounds-checked:

```rust
if tx_age >= self.block_unconfirmed_txs.len() {
    self.bucket_stats[bucket_index].old_unconfirmed_txs -= 1;
} else {
    let block_index = (entry_height % self.block_unconfirmed_txs.len() as u64) as usize;
    self.block_unconfirmed_txs[block_index][bucket_index] -= 1;
}
if count_failure {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64; // ← no bounds check
}
``` [3](#0-2) 

When `tx_age = 1000`, `tx_age - 1 = 999` — the last valid index, so no panic. When `tx_age = 1001`, `tx_age - 1 = 1000` — **one past the end of a length-1000 Vec** — Rust panics with an index-out-of-bounds.

The `count_failure = true` path is reached via `drop_tx` → `drop_tx_inner(tx_hash, true)`: [4](#0-3) [5](#0-4) 

`drop_tx_inner` passes `self.best_height` as `tip_height`, so `tx_age = best_height - entry_height`. If a tracked transaction entered the pool at height H and `best_height` is H + 1001 or more when `reject_tx` is called, the panic fires.

---

### Impact Explanation

The panic unwinds the fee-estimator write lock held by the `ConfirmationFraction` algorithm. Depending on Rust's panic strategy (unwind vs. abort) and how the tx-pool service handles the panic, this results in either:

- **Node crash** (abort mode or unhandled unwind propagating to the async runtime), or
- **Poisoned RwLock**, making all subsequent fee-estimator calls fail permanently until the node is restarted.

Either outcome is a **denial-of-service** against the node operator. The fee estimator is invoked on every `estimate_fee_rate` RPC call and on every block commit, so the poisoned state would be immediately visible.

**Impact: High** (node crash / permanent DoS of fee estimation subsystem).

---

### Likelihood Explanation

The `ConfirmationFraction` algorithm must be the configured estimator. A transaction must:

1. Be submitted to the tx pool (standard `send_transaction` RPC — no privilege required).
2. Remain unconfirmed for > 1000 blocks. At CKB's ~8 s/block target, 1001 blocks ≈ 2.2 hours. A transaction with a fee rate just above `min_fee_rate` but below the current market rate will sit in the pool indefinitely.
3. Be evicted or rejected from the pool after block 1001. The adversary can force this by submitting a conflicting RBF transaction, or by waiting for pool-capacity eviction.

No special privilege, hashpower, or Sybil capability is required. The entry path is the public `send_transaction` RPC.

**Likelihood: Medium** (requires patience but is straightforward to execute deliberately).

---

### Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` write in `remove_unconfirmed_tx`:

```rust
if count_failure {
    let fail_index = tx_age.saturating_sub(1);
    if fail_index < self.confirm_blocks_to_failed_txs.len() {
        self.confirm_blocks_to_failed_txs[fail_index][bucket_index] += 1f64;
    }
}
```

This mirrors the existing guard on `block_unconfirmed_txs` and silently discards failure samples for transactions older than `MAX_CONFIRM_BLOCKS`, which is consistent with the intent of the `old_unconfirmed_txs` overflow path.

Additionally, the `TODO` comment at line 247 (`// TODO do we need decay the old unconfirmed?`) should be resolved — `old_unconfirmed_txs` currently accumulates without decay, which can inflate the denominator in `estimate_median` over time. [6](#0-5) 

---

### Proof of Concept

1. Configure the CKB node to use the `ConfirmationFraction` fee estimator.
2. Submit a transaction T with fee rate just above `min_fee_rate` at block height H. The fee estimator calls `accept_tx` → `track_tx` → `add_unconfirmed_tx`, recording T at height H.
3. Wait for 1001+ blocks to be committed. Each block calls `commit_block` → `process_block` → `decay()` + `move_track_window`. At block H + 1000, `move_track_window` moves T's slot into `old_unconfirmed_txs`. `best_height` is now H + 1001.
4. Submit a conflicting RBF transaction to evict T. The tx-pool calls `fee_estimator.reject_tx(&T.hash)`.
5. `reject_tx` → `drop_tx` → `drop_tx_inner(T.hash, true)` → `remove_unconfirmed_tx(H, H+1001, bucket_index, true)`.
6. `tx_age = 1001 >= 1000` → enters the `old_unconfirmed_txs` branch (correct), then unconditionally executes `confirm_blocks_to_failed_txs[1000][bucket_index] += 1f64`.
7. `confirm_blocks_to_failed_txs` has length 1000; index 1000 is out of bounds → **panic**. [7](#0-6) [8](#0-7) [9](#0-8)

        &mut self,
        entry_height: u64,
        tip_height: u64,
        bucket_index: usize,
        count_failure: bool,
    ) {
        let tx_age = tip_height.saturating_sub(entry_height) as usize;

            return;

        }

            self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
        }
    }
