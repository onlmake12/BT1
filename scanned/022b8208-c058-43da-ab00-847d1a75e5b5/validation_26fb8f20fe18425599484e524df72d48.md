Audit Report

## Title
Missing Per-Peer Rate Limiting in BlockFilter Protocol Enables RocksDB I/O Saturation — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

## Summary
The `BlockFilter` protocol handler processes every incoming `GetBlockFilters`, `GetBlockFilterHashes`, and `GetBlockFilterCheckPoints` message with no per-peer rate limiting. Each such message triggers up to 1000–2000 synchronous RocksDB point reads. An unprivileged remote peer can flood the serving node at maximum TCP rate, saturating RocksDB I/O and starving block validation and sync threads, potentially stalling the node's tip advancement.

## Finding Description
**Entry point:** Any peer connecting on the Filter protocol — no authentication, no PoW, no stake required.

**Root cause:** `BlockFilter` has no `rate_limiter` field and no throttle check anywhere in its dispatch path:

```
received() → process() → try_process() → GetBlockFiltersProcess::execute()
```

`BlockFilter::received` parses the message and immediately calls `self.process(nc, peer_index, msg).await` with no rate check. `try_process` dispatches directly to the handler. There is no guard between message receipt and RocksDB access.

**Per-message work:**
- `GetBlockFiltersProcess::execute`: loops up to `BATCH_SIZE = 1000`, calling `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` per iteration — 2 RocksDB reads per iteration, up to 2000 reads per message. The only guard is `if latest >= start_number`; with `start_number=0` and any built filter present, this is always satisfied.
- `GetBlockFilterHashesProcess::execute`: same pattern, `BATCH_SIZE = 2000`, up to 4000 RocksDB reads per message (hash lookup + filter hash lookup per iteration).
- `GetBlockFilterCheckPointsProcess::execute`: same pattern, `BATCH_SIZE = 2000`.

**Contrast with `Relayer`:** `Relayer` carries a `RateLimiter<(PeerIndex, u32)>` initialized at 30 req/s per `(peer, message_type)` pair. `try_process` checks `self.rate_limiter.check_key(&(peer, message.item_id()))` before dispatching any message and returns `StatusCode::TooManyRequests` on violation. `BlockFilter` has no equivalent mechanism.

**Why existing checks are insufficient:** The 1.8 MB response size cap in `GetBlockFiltersProcess` limits response size per message but does not limit the rate at which messages are processed. An attacker sending `start_number=0` repeatedly will trigger the full loop on every message regardless of the cap.

## Impact Explanation
Each `GetBlockFilters(start_number=0)` message causes up to 2000 synchronous RocksDB reads. At 100 msg/s per peer, a single attacker connection generates 200,000 RocksDB reads/s. Multiple connections multiply this linearly. RocksDB's read IOPS budget is shared with block validation and chain sync; saturation directly degrades block processing latency and can stall the node's tip advancement. This matches the allowed CKB bounty impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points). If multiple nodes are targeted simultaneously, network-wide tip advancement can be impaired.

## Likelihood Explanation
- Requires only a standard P2P connection on the Filter protocol — no credentials, no PoW, no stake.
- The precondition (at least one built filter, i.e., `get_latest_built_filter_block_number() >= 0`) is the normal operating state of any synced node with filters enabled.
- The attack is trivially scriptable: open N connections, loop `send(GetBlockFilters{start_number: 0})`.
- No special knowledge of the target node is required.

## Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`) and check it at the top of `BlockFilter::try_process` before dispatching `GetBlockFilters`, `GetBlockFilterHashes`, and `GetBlockFilterCheckPoints`. A quota of 10–30 req/s per `(peer, message_type)` is consistent with the existing Relayer policy. Call `self.rate_limiter.retain_recent()` in `disconnected` to bound memory usage.

## Proof of Concept
```
1. Ensure the target node has filters enabled and is synced
   (get_latest_built_filter_block_number() >= 0).
2. Open N TCP connections to the target node's Filter protocol endpoint.
3. Each connection sends in a tight loop:
       GetBlockFilters { start_number: 0 }
4. Observe via RocksDB metrics or `perf` that get_block_hash /
   get_block_filter read IOPS scale linearly with N.
5. Observe that block validation latency (time from compact block
   receipt to BLOCK_STORED) degrades proportionally to N.
6. Confirm that adding a 30 req/s RateLimiter (as in Relayer) caps
   IOPS regardless of N.
```
