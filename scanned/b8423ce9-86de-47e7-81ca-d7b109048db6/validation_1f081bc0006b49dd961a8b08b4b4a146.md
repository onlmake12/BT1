### Title
`declared_cycles` Used as Both Script Execution Limit and Exact Cycle Validator, Causing Permanent Relay Suppression - (File: tx-pool/src/process.rs)

---

### Summary

In `tx-pool/src/process.rs`, the `_process_tx` function reuses the peer-supplied `declared_cycles` value for two semantically distinct purposes: (1) as the hard execution-cycle cap passed to `verify_rtx`, and (2) as the exact expected cycle count compared against the verifier's output. When `declared_cycles` is smaller than the transaction's actual cycle consumption, the script verifier aborts with a cycles-exceeded error rather than completing and producing a `DeclaredWrongCycles` rejection. The cycles-exceeded error is classified as a malformed-transaction error, which suppresses re-relay of the transaction hash to other peers. A valid transaction can therefore be permanently silenced on a target node by any unprivileged peer that relays it with an artificially low cycle declaration.

---

### Finding Description

**Root cause — `tx-pool/src/process.rs`, `_process_tx`:**

```rust
// line 720
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());

let verified_ret = verify_rtx(
    Arc::clone(&snapshot),
    Arc::clone(&rtx),
    tx_env,
    &verify_cache,
    max_cycles,          // ← declared_cycles used as execution cap
    command_rx,
)
.await;

let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

if let Some(declared) = declared_cycles
    && declared != verified.cycles   // ← declared_cycles used as exact expected value
{
    return Some((
        Err(Reject::DeclaredWrongCycles(declared, verified.cycles)),
        snapshot,
    ));
}
``` [1](#0-0) 

The two uses are semantically incompatible:

| Use | Correct limit | What is used |
|---|---|---|
| Script execution cap | `max_block_cycles` (consensus) | `declared_cycles` (peer-supplied) |
| Exact cycle validation | `declared_cycles` | `declared_cycles` ✓ |

When `declared_cycles < actual_cycles`, `verify_rtx` → `ContextualTransactionVerifier::verify(max_cycles)` → `TransactionScriptsVerifier::verify(max_cycles)` terminates early with a cycles-exceeded error. That error surfaces as `Reject::Verification(ErrorKind::Script)`.

`is_malformed_from_verification` classifies all `ErrorKind::Script` errors (except `ARGV_TOO_LONG_TEXT`) as malformed:

```rust
ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
``` [2](#0-1) 

`is_allowed_relay` then returns `false` for this error, so the node never sends `TxVerificationResult::Reject` to the relayer:

```rust
pub fn is_allowed_relay(&self) -> bool {
    matches!(self, Reject::DeclaredWrongCycles(..))
        || (!matches!(self, Reject::LowFeeRate(..)) && !self.is_malformed_tx())
}
``` [3](#0-2) 

Contrast with the correct path: if `declared_cycles > actual_cycles`, the script completes, the exact-match check fires, and `DeclaredWrongCycles` is returned. `is_allowed_relay` returns `true` for `DeclaredWrongCycles`, so the relayer is notified and can re-request the transaction from other peers with the correct cycle count.

The asymmetry is confirmed by `_test_accept_tx`, which correctly uses `max_block_cycles` as the execution cap and never passes `declared_cycles` to `verify_rtx`:

```rust
let max_cycles = self.consensus.max_block_cycles();
``` [4](#0-3) 

**Relay suppression mechanism:**

In `TransactionsProcess::execute`, the transaction hash is marked as known *before* pool submission:

```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
``` [5](#0-4) 

Because `is_allowed_relay` returns `false` for the script error, `after_process` never calls `send_result_to_relayer(TxVerificationResult::Reject{..})`, so the hash is never removed from the known-tx filter. The node will not request this transaction from any other peer for the duration of the filter's TTL.

---

### Impact Explanation

An unprivileged peer can permanently suppress relay of any valid transaction on a target node for the filter's TTL window by relaying it with `declared_cycles = 1` (or any value below the transaction's actual cycle cost). The transaction is:

1. Marked as known (will not be re-requested from other peers).
2. Rejected as malformed (script cycles exceeded).
3. Not eligible for re-relay (`is_allowed_relay` = false).

The attacker is banned after the attempt, but the damage — suppression of that transaction hash on the target node — persists. If the attacker targets multiple nodes simultaneously (using multiple IPs or Sybil connections), a valid transaction can be prevented from propagating across a significant portion of the network, delaying or blocking confirmation.

---

### Likelihood Explanation

The attack requires only a standard P2P relay message (`RelayTransactions`) with a crafted `cycles` field set below the transaction's actual cost. No privileged access, no key material, and no majority hash power is needed. The only existing guard in `TransactionsProcess::execute` rejects `declared_cycles > max_block_cycles`; there is no lower-bound check:

```rust
if txs.iter().any(|(_, declared_cycles)| declared_cycles > &max_block_cycles) {
    self.nc.ban_peer(...);
    return Status::ok();
}
``` [6](#0-5) 

Any peer that knows a target transaction's hash (e.g., from observing the mempool or a pending proposal) can execute this attack.

---

### Recommendation

Use `max_block_cycles` (or `max_tx_verify_cycles`) as the execution cap in `_process_tx`, independent of `declared_cycles`. Only use `declared_cycles` for the post-verification exact-match check:

```rust
// Use consensus limit for execution, not the peer-declared value
let max_cycles = self.consensus.max_block_cycles();
// (or self.tx_pool_config.max_tx_verify_cycles for pool-level cap)

let verified_ret = verify_rtx(..., max_cycles, ...).await;
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Only now compare against declared_cycles
if let Some(declared) = declared_cycles
    && declared != verified.cycles
{
    return Some((Err(Reject::DeclaredWrongCycles(declared, verified.cycles)), snapshot));
}
```

This matches the behavior of `_test_accept_tx` and ensures that a peer declaring too-low cycles receives `DeclaredWrongCycles` (re-relay allowed) rather than a script error (re-relay suppressed).

---

### Proof of Concept

1. Node A has `max_block_cycles = 3_500_000_000` and `max_tx_verify_cycles = 70_000_000`.
2. A valid transaction `T` requires `537` cycles (e.g., `always_success` script).
3. Attacker peer connects to Node A and sends a `RelayTransactions` message containing `T` with `cycles = 1`.
4. `TransactionsProcess::execute`: `1 <= max_block_cycles` → no ban at this stage; tx hash marked as known; `submit_remote_tx(T, declared_cycles=1, peer)` called.
5. `_process_tx`: `max_cycles = declared_cycles = 1`; `verify_rtx` runs the script with cap `1`; script aborts with cycles-exceeded (`ErrorKind::Script`).
6. `after_process`: `is_malformed_tx()` = true → attacker peer banned; `is_allowed_relay()` = false → no `TxVerificationResult::Reject` sent to relayer.
7. `T`'s hash remains in Node A's known-tx filter; Node A will not request `T` from any other peer.
8. Legitimate peers relaying `T` with correct cycles `537` are ignored by Node A (tx already "known").

The existing integration test `DeclaredWrongCyclesChunk` (which sets `max_tx_verify_cycles = 500 < 537`) inadvertently exercises this path and confirms the transaction is silently dropped rather than re-requestable. [7](#0-6)

fn is_malformed_from_verification(error: &Error) -> bool {
    match error.kind() {
        ErrorKind::Transaction => error
            .downcast_ref::<TransactionError>()
            .expect("error kind checked")
            .is_malformed_tx(),
        ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
        ErrorKind::Internal => {
            error
                .downcast_ref::<InternalError>()
                .expect("error kind checked")
                .kind()
                == InternalErrorKind::CapacityOverflow
        }
        _ => false,
    }
}
