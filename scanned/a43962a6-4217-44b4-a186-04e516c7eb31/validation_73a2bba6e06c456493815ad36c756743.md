Audit Report

## Title
Cycle Over-Charging via Caller-Controlled `memory_size` in `load_data_as_code` Syscall - (File: `script/src/syscalls/load_cell_data.rs`)

## Summary

The `load_data_as_code` function charges cycles based on the caller-supplied `memory_size` parameter rather than the actual bytes loaded from the cell (`content_size`). A script author can set `memory_size` far larger than `content_size`, causing the VM to charge cycles for zero-padding that was never read from the cell. This is inconsistent with the `load_data` path, which correctly charges only `wrote_size` cycles, and constitutes incorrect cycle accounting behavior in CKB-VM.

## Finding Description

In `load_data_as_code` (`script/src/syscalls/load_cell_data.rs`, lines 101–166), the function reads six register arguments: `addr` (A0), `memory_size` (A1, caller-controlled), `content_offset` (A2), `content_size` (A3), `index` (A4), `source` (A5).

The validation block at lines 148–154 enforces only:
- `content_offset < cell.len()`
- `content_end <= cell.len()`
- `content_size <= memory_size`

There is no upper bound on `memory_size` beyond the VM address space limit. After validation, `init_pages` maps `memory_size` bytes — writing `content_size` bytes from the cell and zero-padding the remainder — and then cycles are charged as:

```rust
machine.add_cycles_no_checking(transferred_byte_cycles(memory_size))?;
``` [1](#0-0) 

The actual data transferred from the cell is only `content_size` bytes (the slice `content_offset..content_end`). The remaining `memory_size - content_size` bytes are zero-padding added by `init_pages`, not loaded from the cell. The cycle charge is therefore `ceil(memory_size / 4)` instead of `ceil(content_size / 4)`. [2](#0-1) 

By contrast, the `load_data` path at line 87 correctly charges only `transferred_byte_cycles(wrote_size)` — the actual bytes written: [3](#0-2) 

`transferred_byte_cycles` is defined as `bytes.div_ceil(4)`: [4](#0-3) 

With `content_size = 1` and `memory_size = 4 MiB` (the VM address space ceiling), the syscall charges `ceil(4194304 / 4) = 1,048,576` cycles for loading a single byte — a 1,048,576× inflation ratio. The only enforced relationship between the two values is `content_size <= memory_size` (line 150), which permits this extreme disparity. [5](#0-4) 

## Impact Explanation

This is an incorrect implementation of CKB-VM syscall behavior. Cycles are the canonical resource unit governing block inclusion and transaction weight. Charging cycles for zero-padding that was never read from the cell means the cycle counter no longer accurately represents the computational work performed. A single transaction can consume up to ~1,048,576 cycles (~1.5% of the 70M block cycle limit) via this syscall while loading only 1 byte of actual cell data. Repeated across multiple transactions or calls, this allows an attacker to crowd out legitimate transactions from the block cycle budget at a cost disproportionate to the actual node computation incurred. This matches the allowed impact: **Incorrect implementation or behavior of CKB-VM or system scripts (High, 10001–15000 points)**.

## Likelihood Explanation

The exploit path requires no special privileges. Any script author can deploy a cell containing a script that calls syscall `2091` (`LOAD_CELL_DATA_AS_CODE_SYSCALL_NUMBER`) with an arbitrarily large `memory_size` and a minimal `content_size`. Any node processing a transaction that references such a script will execute it and incur the inflated cycle charge. The syscall is a standard, documented interface available to all CKB scripts. The condition is trivially reproducible and repeatable.

## Recommendation

Charge cycles based on `content_size` (actual bytes loaded from the cell) rather than `memory_size` (the memory region size, which includes zero-padding):

```rust
// In load_data_as_code, replace line 163:
machine.add_cycles_no_checking(transferred_byte_cycles(memory_size))?;
// With:
machine.add_cycles_no_checking(transferred_byte_cycles(content_size))?;
```

This aligns `load_data_as_code` with the `load_data` path, which charges only `transferred_byte_cycles(wrote_size)`. If charging for the full memory mapping operation (including zero-initialization) is intentional, it must be explicitly documented and `memory_size` must be bounded to a small multiple of `content_size` or to the VM page size granularity to prevent unbounded inflation.

## Proof of Concept

A minimal RISC-V CKB script:

```c
#include "ckb_syscalls.h"
int main() {
    uint8_t buf[1];
    // Load 1 byte from cell dep index 0, but map 4 MiB
    // Cycles charged: ceil(4194304 / 4) = 1,048,576
    // Cycles for actual data: ceil(1 / 4) = 1
    ckb_load_cell_data_as_code(buf, 4 * 1024 * 1024, 0, 1, 0, CKB_SOURCE_CELL_DEP);
    return 0;
}
```

Deploy this as a lock script. Any transaction spending a cell locked by this script will execute it and charge 1,048,576 cycles for a 1-byte data load. Verification: instrument the VM cycle counter before and after the syscall and confirm the delta is `ceil(memory_size / 4)`, not `ceil(content_size / 4)`. The discrepancy is directly observable in the cycle counter output of `ckb-debugger` or any CKB-VM test harness.
