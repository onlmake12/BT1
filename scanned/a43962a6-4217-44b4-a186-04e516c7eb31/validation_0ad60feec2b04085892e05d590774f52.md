Audit Report

## Title
`RecentReject::put` Never Increments `self.total_keys_num`, Making `count_limit` Permanently Unenforced — (`tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put()`, the incremented counter is bound to a local variable `total_keys_num` via `let Some(total_keys_num) = self.total_keys_num.checked_add(1)` but is never written back to `self.total_keys_num`. The struct field stays at its initial estimated value (0 on a fresh node) for the entire process lifetime. Because the limit check `total_keys_num > self.count_limit` evaluates as `1 > 10_000_000` on every call, `shrink()` is never triggered, and the RocksDB-with-TTL store grows without bound. An unprivileged attacker submitting a stream of unique low-fee transactions can exhaust the node's disk, causing RocksDB write failures and a node crash.

## Finding Description
**Root cause** — `tx-pool/src/component/recent_reject.rs`, lines 62–69:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

`total_keys_num` here is a fresh local binding. `self.total_keys_num` is never assigned. The only place the field is mutated is inside `shrink()` (lines 110–111), which re-estimates the count from RocksDB — but `shrink()` is never reached because the guard condition is always false.

**Exploit path:**
1. Attacker calls `send_transaction` RPC with unique, structurally valid transactions whose fees are below `min_fee_rate`.
2. Each transaction reaches `tx-pool/src/util.rs` fee-rate check and is rejected with `Reject::LowFeeRate`.
3. `process.rs` line 548 calls `should_recorded()` → returns `true` (only `Reject::Duplicated` is excluded, per `util/types/src/core/tx_pool.rs` line 101).
4. `put_recent_reject` calls `RecentReject::put()`, which writes to RocksDB and then evaluates `(0+1) > 10_000_000` → false. No shrink. No eviction.
5. Repeat indefinitely. The DB grows at the rate of attacker submissions.

**Why existing checks fail:** The `count_limit` guard (default 10,000,000, `util/app-config/src/legacy/tx_pool.rs` line 58) and the TTL (default 7 days, line 54) are the only defenses. TTL-based expiry is passive and slow; the count guard is permanently disabled by the missing assignment. The existing unit test (`tests/recent_reject.rs` line 39) asserts `total_keys_num < 100` after 160 puts with `limit=100` — this passes vacuously because the counter is always 0, not because shrinking functioned correctly.

## Impact Explanation
Disk exhaustion causes RocksDB write failures, which crash or severely degrade the CKB node. Each rejected transaction writes ~32 bytes (key) + serialized reject reason to RocksDB. With no eviction, sustained submission at modest rates (e.g., 1,000 tx/s) accumulates hundreds of GB within the 7-day TTL window. This maps to: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
The attack requires only the ability to call the `send_transaction` RPC endpoint with structurally valid transactions. No signing keys, no CKB balance, and no special privileges are required for the `LowFeeRate` path — the transactions are rejected before any state change. There is no rate limiting on the RPC endpoint. The bug is present from process start on any fresh or recently-shrunk node and is fully reproducible.

## Recommendation
In `RecentReject::put()`, assign the incremented value back to `self.total_keys_num` before the limit check:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ... write to db ...
    if let Some(new_total) = self.total_keys_num.checked_add(1) {
        self.total_keys_num = new_total;  // missing assignment
        if self.total_keys_num > self.count_limit {
            self.shrink()?;
        }
    } else {
        self.shrink()?;
    }
    Ok(())
}
```

Additionally, update the unit test to assert the actual RocksDB entry count (via `estimate_total_keys_num`) rather than the in-memory field, so the shrink behavior is genuinely verified.

## Proof of Concept

let tmp = tempfile::tempdir().unwrap();
let limit = 5u64;

for i in 0..20u64 {
