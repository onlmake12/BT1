Audit Report

## Title
Shared `VerifyQueue` Has No Per-Sender Quota, Allowing a Single Actor to Exhaust the 256 MB Global Limit and Block All Legitimate Transaction Submissions — (`tx-pool/src/component/verify_queue.rs`)

## Summary

The `VerifyQueue` enforces a single global 256 MB size cap (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) with no per-peer or per-RPC-caller quota. A single connected P2P peer or local RPC caller can submit ~512 structurally valid but contextually invalid transactions (each up to 512 KB) to fill the entire queue. Once full, every subsequent transaction submission from any source is rejected with `Reject::Full` until workers drain the attacker's entries, and the attacker can immediately re-fill to sustain the denial.

## Finding Description

The admission check in `add_tx` is a flat global comparison with no per-origin tracking: [1](#0-0) 

When `is_full` returns `true`, `add_tx` immediately rejects with `Reject::Full`: [2](#0-1) 

Every submission path — `submit_remote_tx`, `notify_tx`, and local RPC — converges on `resumeble_process_tx`, which calls `enqueue_verify_queue` after only a structural pre-check: [3](#0-2) 

The pre-queue check `non_contextual_verify` validates structure, enforces the 512 KB per-transaction size ceiling, and rejects cellbase transactions — but does **not** verify that referenced input `OutPoint`s exist on-chain: [4](#0-3) 

An attacker therefore crafts transactions with fabricated (non-existent) input `OutPoint`s. These pass `non_contextual_verify` and are admitted into the queue, occupying space until a worker processes and evicts them via contextual resolution failure. The only per-peer mechanism, `remove_txs_by_peer`, is invoked only on peer disconnect — not as an admission control: [5](#0-4) 

The `VerifyQueue` struct holds no per-peer accounting field whatsoever: [6](#0-5) 

## Impact Explanation

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** A single malicious P2P peer (no local access required) can exhaust the 256 MB `VerifyQueue` of any targeted node, preventing that node from accepting any new transactions — blocking its participation in transaction relay and block assembly. The attack is cheap to sustain: each re-submission costs only the CPU of `non_contextual_verify` (a structural check), and the attacker can continuously re-fill as workers drain entries. Applied to multiple nodes simultaneously, this causes network-wide transaction relay congestion at negligible cost.

## Likelihood Explanation

**High for P2P attacker.** Any connected peer can relay transactions via the relay protocol through `submit_remote_tx`: [7](#0-6) 

Filling 256 MB requires approximately `256 MB / 512 KB = 512` transactions. Each transaction is trivial to construct (one fabricated input `OutPoint`, one output, a large witness blob). There is no connection-level rate limiting on transaction relay. The attacker can reconnect after disconnect to repeat. **Medium for local RPC attacker**, since the RPC is bound to `127.0.0.1:8114` by default, requiring co-location. [8](#0-7) 

## Recommendation

1. **Introduce per-peer quota in `VerifyQueue`**: Add a `HashMap<PeerIndex, usize>` tracking each peer's contribution to `total_tx_size`. Cap each peer's share (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / max_peers`). The existing `remove_txs_by_peer` infrastructure already iterates by peer — extend it to enforce admission limits.

2. **Per-peer eviction when full**: Instead of a flat `Reject::Full`, evict the lowest-priority transaction from the peer holding the largest queue share, analogous to how `limit_size` evicts by fee rate in the main pool.

3. **Apply minimum fee-rate check before queue admission**: Move the fee-rate floor check ahead of `enqueue_verify_queue` so zero-fee or dust transactions cannot cheaply fill the queue.

## Proof of Concept

1. Construct ~512 transactions, each with a witness blob approaching 512 KB and a fabricated (non-existent) input `OutPoint`. Each passes `non_contextual_verify` (valid structure, size ≤ `TRANSACTION_SIZE_LIMIT`, not cellbase).
2. Submit all 512 via `send_transaction` RPC or P2P relay in rapid succession. Each reaches `enqueue_verify_queue` and is accepted, accumulating ~256 MB of `total_tx_size`.
3. The 513th `send_transaction` from any caller returns `PoolIsFull: verify_queue total_tx_size exceeded`.
4. Workers process the attacker's transactions; each fails contextual resolution (`OutPointError::Unknown`) and is evicted. The attacker immediately re-submits to refill.
5. Legitimate transactions are continuously rejected for as long as the attacker maintains the flood. [1](#0-0) [2](#0-1)

```rust
pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
```
