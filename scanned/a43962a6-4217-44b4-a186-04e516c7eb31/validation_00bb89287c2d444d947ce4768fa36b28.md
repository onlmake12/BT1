Audit Report

## Title
`RecentReject::put()` Never Persists Incremented `total_keys_num`, Disabling `shrink()` and Allowing Unbounded DB Growth — (`tx-pool/src/component/recent_reject.rs`)

## Summary

In `RecentReject::put()`, the result of `self.total_keys_num.checked_add(1)` is bound to a local variable `total_keys_num` but is never written back to `self.total_keys_num`. Because the field never advances past its initialization value (0 on a fresh DB), the guard condition `total_keys_num > self.count_limit` evaluates to `1 > 10_000_000` on every call and is always false, so `shrink()` is never invoked. The reject DB therefore grows without bound for the lifetime of the node.

## Finding Description

The root cause is a single missing assignment in `put()`:

```rust
// tx-pool/src/component/recent_reject.rs, lines 62-69
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // self.total_keys_num is NEVER updated here
} else {
    self.shrink()?;
}
``` [1](#0-0) 

`self.total_keys_num` has exactly two write sites:

1. **Construction** (`build()`): set from `estimate_num_keys_cf`, which returns 0 for a fresh DB. [2](#0-1) 

2. **`shrink()`**: reset to the post-drop estimate after dropping a shard. [3](#0-2) 

Because neither write occurs in the normal `put()` path, `self.total_keys_num` stays at 0 indefinitely. Every invocation of `put()` computes `0 + 1 = 1`, compares `1 > 10_000_000` (the default `count_limit`), finds it false, and returns without calling `shrink()`. The underlying `DBWithTTL` (RocksDB) accumulates entries without any eviction beyond TTL-driven compaction.

The call path from an unauthenticated remote peer to `put()` is:

- P2P relay / `send_transaction` RPC → `process_tx` → `after_process` → `put_recent_reject` → `RecentReject::put()` [4](#0-3) [5](#0-4) 

Any transaction that fails validation and satisfies `reject.should_recorded()` writes a `PoolTransactionReject` JSON record to the DB.

## Impact Explanation

The reject DB grows without bound. Each entry is a JSON-serialized `PoolTransactionReject` record. With no `shrink()` ever firing, disk space is consumed monotonically. RocksDB's TTL (default 7 days) only reclaims space during compaction, not on write; an attacker can write entries faster than compaction reclaims them. Full disk exhaustion prevents the node from writing to any RocksDB instance (chain DB, tx-pool DB), halting block processing and sync entirely — a complete node crash.

This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The exploit requires no privileges, no PoW, no key material, and no victim interaction. Any peer can submit transactions that fail validation (double-spend, malformed script, insufficient fee, etc.) via the public P2P relay protocol or the `send_transaction` RPC. The bug is deterministic and reproducible on every fresh node. The attacker controls the submission rate and can sustain it indefinitely.

## Recommendation

Add the missing assignment in the non-shrink branch of `put()`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // ← add this line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

No change is needed in the `shrink()` branch or in `shrink()` itself, as `self.total_keys_num` is already re-estimated there. [6](#0-5) 

## Proof of Concept

```rust
let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
let count_limit = 10u64;
let mut rr = RecentReject::build(tmp_dir.path(), 2, count_limit, -1).unwrap();

for i in 0..100u64 {
    let key = Byte32::new(blake2b_256(i.to_le_bytes()));
    rr.put(&key, Reject::Malformed(i.to_string(), Default::default())).unwrap();
}

// total_keys_num never advanced; shrink() was never called
assert_eq!(rr.total_keys_num, 0);
// DB holds ~100 entries despite count_limit = 10
```

After 100 insertions with `count_limit = 10`, `total_keys_num` remains 0 and `shrink()` has never been invoked. Scaling this to millions of rejected transactions exhausts disk space.

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        self.total_keys_num = total_keys_num;
