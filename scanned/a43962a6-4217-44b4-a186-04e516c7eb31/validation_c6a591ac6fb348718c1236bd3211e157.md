Audit Report

## Title
VerifyQueue Admits Conflicting Transactions Without Input Locking, Enabling `total_tx_size` Inflation and DoS — (`tx-pool/src/component/verify_queue.rs`, `tx-pool/src/process.rs`)

## Summary
The `VerifyQueue` deduplicates entries solely by `proposal_short_id` (derived from tx hash), never by input `OutPoint`. An attacker holding one confirmed cell can craft ~500 structurally valid transactions that all spend the same input but produce distinct outputs (yielding distinct hashes), flood the verify queue, exhaust the 256 MB `total_tx_size` cap, and cause every subsequent legitimate `send_transaction` call to return `Reject::Full` until workers drain the queue — which the attacker can continuously counteract by re-submitting.

## Finding Description

**Root cause — hash-only deduplication in `add_tx`:**
`VerifyQueue::add_tx` checks `self.contains_key(&tx.proposal_short_id())` before inserting. [1](#0-0) 
Two transactions spending the same `OutPoint` but with different outputs have different hashes and different `proposal_short_id` values, so both pass this check unconditionally.

**Admission path — no input-conflict gate before enqueue:**
`resumeble_process_tx` runs `non_contextual_verify`, then checks `orphan_contains` and `verify_queue_contains` (both keyed on tx hash), then calls `enqueue_verify_queue`. There is no step that inspects whether any input of the incoming transaction is already claimed by a queued entry. [2](#0-1) 

**`non_contextual_verify` is input-conflict-blind:**
It verifies version, size, empty inputs/outputs, duplicate deps, outputs data, and script hash type — nothing cross-transaction. [3](#0-2) 

**Fee and conflict checks are deferred to the worker:**
`check_tx_fee` and `resolve_tx` (which detects dead outpoints) are called inside `pre_check`, which runs only after a worker has already dequeued the transaction from the verify queue. [4](#0-3) 

**`total_tx_size` hard cap is 256 MB; per-tx limit is 512 KB:** [5](#0-4) [6](#0-5) 
So ≥ 512 conflicting transactions saturate the queue. Once full, `is_full` returns true and every new `add_tx` call returns `Reject::Full`. [7](#0-6) 

**`Edges.insert_input` enforces input uniqueness only for the committed pool:**
This protection is never reached for transactions still sitting in the verify queue. [8](#0-7) 

## Impact Explanation
Once `total_tx_size` reaches 256 MB, every call to `resumeble_process_tx` — whether via RPC `send_transaction` or P2P relay — returns `Reject::Full` for all legitimate transactions. Workers drain conflicting entries one by one (all but the first fail at `pre_check` with `OutPointError::Dead`), but the attacker re-submits a fresh batch to maintain saturation. This constitutes a sustained, low-cost DoS of the tx-pool admission pipeline, matching the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack requires only: (1) one confirmed live cell, trivially obtained on mainnet; (2) the ability to call `send_transaction` via RPC or relay via P2P, both unprivileged and externally reachable; (3) generating ~512 structurally valid transactions with the same input and distinct outputs, trivial with any wallet library. No hashpower, no special privileges, no victim mistakes. The attack is cheap and indefinitely repeatable.

## Recommendation
Maintain a per-`OutPoint` index inside `VerifyQueue` (mirroring `Edges.inputs` in the main pool). Before inserting in `add_tx`, iterate the incoming transaction's inputs and reject with a new `Reject::ConflictInVerifyQueue` variant if any input is already claimed by a queued entry. Remove the corresponding entries from this index in `remove_tx`. This mirrors the existing `Edges::insert_input` enforcement and closes the gap between the verify queue and the committed pool.

## Proof of Concept
```
1. Obtain one confirmed live cell C at outpoint OP (e.g., from a coinbase or transfer).
2. Construct transactions T1..T512:
   - Each Ti: input = OP, output = distinct capacity or lock args (ensuring distinct tx hash)
   - Each Ti passes NonContextualTransactionVerifier (valid version, non-empty, no dup deps, size < 512 KB)
3. Submit T1..T512 via RPC send_transaction (or P2P relay).
4. For each Ti, resumeble_process_tx:
   - non_contextual_verify: OK
   - orphan_contains(Ti): false (distinct hashes)
   - verify_queue_contains(Ti): false (distinct hashes)
   - enqueue_verify_queue: OK → total_tx_size += size(Ti)
5. After ~512 submissions, total_tx_size >= 256 MB.
6. Any subsequent send_transaction from a legitimate user returns:
   Reject::Full("verify_queue total_tx_size exceeded, failed to add tx: 0x...")
7. Workers drain Ti one by one; T1 may succeed, T2..T512 fail with Dead outpoint.
   Attacker re-submits a fresh batch to maintain saturation indefinitely.
```

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
    }

    pub(crate) async fn resumeble_process_tx(
        &self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, remote).await?;

        if self.orphan_contains(&tx).await {
            debug!("reject tx {} already in orphan pool", tx.hash());
            return Err(Reject::Duplicated(tx.hash()));
        }

        if self.verify_queue_contains(&tx).await {
            return Err(Reject::Duplicated(tx.hash()));
        }
        self.enqueue_verify_queue(tx, is_proposal_tx, remote).await
    }

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}

    pub(crate) fn insert_input(
        &mut self,
        out_point: OutPoint,
        txid: ProposalShortId,
    ) -> Result<(), Reject> {
        // inputs is occupied means double speanding happened here
        match self.inputs.entry(out_point.clone()) {
            Entry::Occupied(occupied) => {
                let msg = format!(
                    "txpool unexpected double-spending out_point: {:?} old_tx: {:?} new_tx: {:?}",
                    out_point,
                    occupied.get(),
                    txid
                );
                Err(Reject::RBFRejected(msg))
            }
            Entry::Vacant(vacant) => {
                vacant.insert(txid);
                Ok(())
            }
        }
    }
