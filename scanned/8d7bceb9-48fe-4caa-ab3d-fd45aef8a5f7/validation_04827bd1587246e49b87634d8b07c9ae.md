Audit Report

## Title
TOCTOU Race in `process_fetch_cmd`: `assume_valid_targets` Cleared Between Independent Lock Acquisitions Causes Unconditional Panic in BlockDownload Thread — (`sync/src/synchronizer/mod.rs`)

## Summary
`process_fetch_cmd` calls `can_start()` (which acquires and releases the `assume_valid_targets` mutex), receives `CanStart::AssumeValidNotFound`, then performs a second independent lock acquisition and calls `.expect("assume valid target must exist")` unconditionally. Between these two acquisitions, the chain verification thread can call `assume_valid_targets.take()` setting the value to `None`, causing the `.expect()` to panic and terminating the `BlockDownload` thread, which permanently halts IBD block fetching until the node is restarted.

## Finding Description
In `process_fetch_cmd` (`sync/src/synchronizer/mod.rs`), the match at line 121 calls `self.can_start()`. Inside `can_start()`, the closure `assume_valid_target_find` acquires `shared.assume_valid_targets()` as a local `MutexGuard`, observes `Some([last_target])` with the target absent from `header_map` and the time condition not met, drops the guard (releasing the mutex), and returns `CanStart::AssumeValidNotFound`.

`process_fetch_cmd` then enters the `AssumeValidNotFound` match arm (line 138) and at lines 143–148 performs a **second, independent** call to `shared.assume_valid_targets()`, acquiring a new `MutexGuard`. The `.expect("assume valid target must exist")` at line 148 is unconditional — it executes on every entry into this branch regardless of whether the logging condition at line 150 is true.

Concurrently, the chain verification thread in `verify_block()` (`chain/src/verify.rs`, lines 226–227) acquires the same mutex, removes the last target entry, and calls `assume_valid_targets.take()`, setting the `Option` to `None`.

If this chain-thread operation occurs between the two lock acquisitions in `process_fetch_cmd`, the second acquisition returns `None`. The chain `.as_ref().and_then(|targets| targets.first())` evaluates to `None`, and `.expect()` panics, terminating the `BlockDownload` thread.

The `else` branch at line 311–312 of `can_start()` (`*flag = CanStart::Ready`) only guards against `None` being observed **inside** `can_start()` itself. It provides no protection for the second, separate lock acquisition in `process_fetch_cmd`.

## Impact Explanation
The panic terminates the `BlockDownload` thread (the `BlockFetchCMD` run loop, lines 240–250). This thread is solely responsible for IBD block fetching. Its termination permanently halts all IBD block downloads until the node process is restarted. A node caught in this state cannot make further chain progress and stalls at the point where the last assume_valid_target was being processed. This matches the allowed impact: **Vulnerabilities which could easily crash a CKB node** (High, 10001–15000 points).

## Likelihood Explanation
No attacker capability is required beyond being a normal sync peer providing valid mainnet blocks. The race condition is triggered naturally during normal mainnet IBD when the chain thread processes the last hardcoded assume_valid_target block concurrently with the sync thread's logging path. The default target list contains 20 mainnet checkpoints; the race applies specifically at the last one. The race window is narrow (two sequential mutex acquisitions with no atomic guarantee), but the condition is deterministically reached on every mainnet IBD and requires no special peer behavior.

## Recommendation
Eliminate the second lock acquisition in the `AssumeValidNotFound` branch of `process_fetch_cmd`. The `assume_valid_target` value needed for logging should be captured **inside** `can_start()` while the mutex is still held and returned alongside the `CanStart` variant (e.g., `AssumeValidNotFound(Byte32)`). Alternatively, the `AssumeValidNotFound` branch should re-check for `None` after the second acquisition and gracefully transition to `CanStart::Ready` instead of calling `.expect()`.

## Proof of Concept
Spawn two threads sharing the same `Shared` instance initialized with `assume_valid_targets = Some([target_hash])` where `target_hash` is absent from `header_map` and the time condition is not met. Thread A calls `can_start()` (returns `AssumeValidNotFound`), then parks. Thread B calls `verify_block()` with the target block, triggering `assume_valid_targets.take()` → `None`. Thread A resumes and enters the `AssumeValidNotFound` match arm, executing the second `shared.assume_valid_targets()` acquisition and calling `.expect()` on the now-`None` guard — assert panic occurs, thread terminates.
