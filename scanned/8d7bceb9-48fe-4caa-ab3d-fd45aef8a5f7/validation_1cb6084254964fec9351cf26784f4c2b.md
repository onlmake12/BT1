Audit Report

## Title
O(N¬≤) Ancestor BFS Re-traversal in `remove_by_detached_proposal` During Reorg ‚Äî (`tx-pool/src/pool.rs`, `tx-pool/src/component/pool_map.rs`)

## Summary
`TxPool::remove_by_detached_proposal` removes a proposed transaction chain and re-inserts each entry into pending via `add_pending`. Each `add_pending` call triggers two full ancestor BFS traversals (`calc_relation_ids` and `calc_ancestors`). For a linear chain of N transactions re-inserted in ancestor-count order, this produces O(N¬≤) total BFS work while holding the tx-pool write lock, blocking all concurrent tx-pool operations for the duration of the reorg handler.

## Finding Description

**Root cause ‚Äî `remove_by_detached_proposal` (pool.rs L333‚Äì356):**
The function removes the chain atomically, sorts entries by `ancestors_count` (root first), then calls `add_pending(entry)` for each of the N entries one at a time. Because entries are re-inserted in topological order, when inserting the k-th entry, exactly k‚àí1 ancestors are already present in the pool.

**Traversal 1 ‚Äî `check_and_record_ancestors` ‚Üí `get_tx_ancenstors` ‚Üí `calc_relation_ids` (pool_map.rs L588‚Äì600, L549‚Äì553; links.rs L52‚Äì72):**
`get_tx_ancenstors` calls `self.links.calc_relation_ids(parents.clone(), Relation::Parents)`, which is an iterative BFS over all ancestors currently in the pool. For the k-th insertion, this visits k‚àí1 nodes.

**Traversal 2 ‚Äî `record_entry_descendants` ‚Üí `update_ancestors_index_key` ‚Üí `calc_ancestors` (pool_map.rs L509‚Äì512, L432‚Äì444):**
`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`, which calls `calc_relative_ids` ‚Üí `calc_relation_ids` ‚Äî another full BFS over all ancestors. For the k-th insertion, this again visits k‚àí1 nodes.

**Quadratic accumulation:**
```
Total BFS node visits = 2 √ó (0 + 1 + 2 + ... + (N-1)) = N(N-1) = O(N¬≤)
```
For N = `max_ancestors_count` (configurable, up to 1000): ~1,000,000 HashSet BFS operations in a single lock-holding call.

**Lock scope ‚Äî `_update_tx_pool_for_reorg` (process.rs L836‚Äì851, L1055‚Äì1056):**
The tx-pool write lock is acquired at L836 and held across the entire call to `_update_tx_pool_for_reorg`, which calls `remove_by_detached_proposal` at L1056. All concurrent tx-pool operations (transaction submission, block template generation, peer sync) are blocked for the full duration.

No existing guard limits the number of entries processed per call or bounds the per-insertion BFS depth during re-insertion.

## Impact Explanation
While the tx-pool write lock is held, all concurrent operations are blocked. A 1000-tx chain causes ~1,000,000 HashSet BFS operations in a single lock-holding call during reorg processing. This stalls transaction submission and block template generation across all nodes that received the chain, simultaneously, during any reorg event. Because natural reorgs affect all network participants at once, a single attacker-submitted chain can cause coordinated stalls across many nodes.

**Impact class: High ‚Äî Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Requirements for the attacker:
1. Submit a chain of N transactions (N ‚â§ `max_ancestors_count`) via standard RPC ‚Äî no privilege required.
2. Wait for miners to include the proposal short IDs in a block's proposal zone ‚Äî miners do this automatically.
3. Wait for any natural 1-block reorg ‚Äî these occur regularly on mainnet without attacker involvement.

Cost: transaction fees for N transactions. No hashpower, no privileged access, no victim mistakes required. The attack is repeatable after every reorg event.

## Recommendation
Replace per-insertion ancestor BFS with a single-pass bulk re-insertion:

1. After `remove_entry_and_descendants`, re-link all entries in topological order (already sorted by `ancestors_count`).
2. For each re-inserted entry, compute ancestors incrementally from the already-inserted parent set using O(1) parent lookups rather than a full BFS ‚Äî since parents are guaranteed to be inserted before children in the sorted order.
3. Defer `update_ancestors_index_key` and `_record_ancestors` until the full chain is re-linked, then perform a single pass to update descendant weights, reducing total complexity from O(N¬≤) to O(N¬∑E) where E is the average out-degree (typically 1 for a chain).

## Proof of Concept
1. Build `tx0` (coinbase spend), `tx1` (spends `tx0` output), ..., `tx999` (spends `tx998` output).
2. Submit all 1000 transactions to the node via RPC.
3. Wait for a block that includes all 1000 proposal short IDs in its proposal zone.
4. Observe a natural 1-block reorg (or broadcast a competing block at the same height).
5. `_update_tx_pool_for_reorg` calls `remove_by_detached_proposal` with 1000 IDs.
6. Benchmark `remove_by_detached_proposal` with chains of length 100, 200, 500, 1000 ‚Äî wall-clock time should scale quadratically (10√ó more transactions ‚Üí ~100√ó longer execution).
7. During the stall, confirm that concurrent RPC calls for transaction submission and `get_block_template` are blocked.
