Looking at the actual code to trace the full execution path.

**`finalize` in `util/proposal-table/src/lib.rs`:**

`removed_ids` is computed exclusively as:
```rust

    origin.set().difference(&new_ids).cloned().collect();
``` [1](#0-0) 

This is a difference of `origin.set()` (the "proposed" window) against `new_ids`. There is no corresponding computation for `origin.gap().difference(&new_gap)`. Proposals that were only in the gap window and fall out of it during a reorg are silently dropped from `removed_ids`. [2](#0-1)

**Concrete scenario with `ProposalWindow(2, 10)` (closest=2, farthest=10):**

- Tip N: `candidate_number = N+1`, `proposal_end = N-1`, `gap = blocks (N-1, N]`. Proposal P at block N → in `gap`.
- Reorg to tip N-1: `candidate_number = N`, `proposal_end = N-2`, `gap = blocks (N-2, N-1]`. P at block N → NOT in gap or set.
- `origin.set()` at tip N = proposals from `[N-9, N-1]`. P is NOT there.
- `removed_ids = origin.set().difference(&new_ids)` → P is absent → `removed_ids` does not contain P.
- `remove_by_detached_proposal` is never called for P.
- P remains in `Status::Gap` in the tx-pool indefinitely (until `limit_size` evicts it under pool pressure).

The only safety net is `limit_size`, which evicts Gap entries only when the pool exceeds its size limit — not proactively on reorg. [9](#0-8)

### Title
Gap-status tx-pool entries not demoted to Pending after reorgs that remove proposals from the gap window — (`util/proposal-table/src/lib.rs`, `tx-pool/src/process.rs`)

### Summary
`ProposalTable::finalize` only computes `removed_ids` as proposals leaving `origin.set()` (the commit window). Proposals that were exclusively in `origin.gap()` and fall out of the gap window during a reorg are never included in `detached_proposal_id`. Consequently, `_update_tx_pool_for_reorg` never calls `remove_by_detached_proposal` for them, and those transactions remain permanently stuck in `Status::Gap` until pool eviction pressure forces removal.

### Finding Description
The invariant that every tx-pool entry's status must reflect the current `ProposalView` after each tip change is violated. After a reorg that rolls the tip back so that a previously gap-eligible block is no longer within the gap window, the corresponding Gap-status entries in the tx-pool are not demoted to Pending. The mine_mode promotion loop only upgrades Gap→Proposed; there is no corresponding demotion path for Gap→Pending when the proposal leaves the gap.

### Impact Explanation
Stale Gap-status entries accumulate in the tx-pool. They cannot be committed (not in the proposed set) and are not re-proposed (not in Pending). Miners running in mine mode will not re-propose these transactions, causing them to be effectively invisible to the block-assembly path until pool pressure evicts them. This is suboptimal state storage: the pool's view of which transactions are "gap-proposed" diverges from the canonical chain's `ProposalView`.

### Likelihood Explanation
Any unprivileged peer can trigger this by relaying a valid competing chain that causes a reorg rolling the tip back by even one block, which is a normal P2P operation requiring no special privileges or majority hashpower.

### Recommendation
In `finalize`, also compute and return `removed_gap_ids = origin.gap().difference(&new_gap)`. In `_update_tx_pool_for_reorg`, call `remove_by_detached_proposal` for both `detached_proposal_id` (set-level) and the new gap-level diff, so Gap-status entries whose proposals are no longer in the current gap window are properly demoted to Pending.

### Proof of Concept
```rust
// ProposalWindow(2, 10): closest=2, farthest=10
let window = ProposalWindow(2, 10);
let mut table = ProposalTable::new(window);
let p = ProposalShortId::new([1u8; 10]);
table.insert(5, iter::once(p.clone()).collect()); // P proposed at block 5

// finalize to tip 5: P enters gap (block 5 is in gap window (3,5])
let (removed, view_at_5) = table.finalize(&ProposalView::default(), 5);
assert!(view_at_5.contains_gap(&p));   // P is in gap
assert!(!view_at_5.contains_proposed(&p));

// reorg: finalize to tip 4: P should leave gap (block 5 > tip 4)
let (removed_ids, view_at_4) = table.finalize(&view_at_5, 4);
assert!(!view_at_4.contains_gap(&p));      // P not in new gap ✓
assert!(!view_at_4.contains_proposed(&p)); // P not in new set ✓
// BUG: removed_ids is empty — P was only in origin.gap(), not origin.set()
assert!(removed_ids.is_empty()); // passes — P is never reported as detached
// Therefore remove_by_detached_proposal is never called for P,
// and the tx-pool retains P as Status::Gap indefinitely.

    pub fn finalize(
        &mut self,
        origin: &ProposalView,
        number: BlockNumber,
    ) -> (HashSet<ProposalShortId>, ProposalView) {
        let candidate_number = number + 1;
        let proposal_start = candidate_number.saturating_sub(self.proposal_window.farthest());
        let proposal_end = candidate_number.saturating_sub(self.proposal_window.closest());

        if proposal_start > 1 {
            self.table = self.table.split_off(&proposal_start);
        }

        ckb_logger::trace!("[proposal_finalize] table {:?}", self.table);

        // - if candidate_number <= self.proposal_window.closest()
        //      new_ids = []
        //      gap = [1..candidate_number]
        // - else
        //      new_ids = [candidate_number- farthest..= candidate_number- closest]
        //      gap = [candidate_number- closest + 1..candidate_number]
        // - end
        let (new_ids, gap) = if candidate_number <= self.proposal_window.closest() {
            (
                HashSet::new(),
                self.table
                    .range((Bound::Unbounded, Bound::Included(&number)))
                    .flat_map(|pair| pair.1)
                    .cloned()
                    .collect(),
            )
        } else {
            (
                self.table
                    .range((
                        Bound::Included(&proposal_start),
                        Bound::Included(&proposal_end),
                    ))
                    .flat_map(|pair| pair.1)
                    .cloned()
                    .collect(),
                self.table
                    .range((Bound::Excluded(&proposal_end), Bound::Included(&number)))
                    .flat_map(|pair| pair.1)
                    .cloned()
                    .collect(),
            )
        };

        let removed_ids: HashSet<ProposalShortId> =
            origin.set().difference(&new_ids).cloned().collect();
        ckb_logger::trace!(
            "[proposal_finalize] number {} proposal_start {}----proposal_end {}",
            number,
            proposal_start,
            proposal_end
        );
        ckb_logger::trace!(
            "[proposal_finalize] number {} new_ids {:?}----removed_ids {:?}",
            number,
            new_ids,
            removed_ids
        );
        (removed_ids, ProposalView::new(gap, new_ids))
    }

    pub(crate) fn remove_by_detached_proposal<'a>(
        &mut self,
        ids: impl Iterator<Item = &'a ProposalShortId>,
    ) {
        for id in ids {
            if let Some(e) = self.pool_map.get_by_id(id) {
                let status = e.status;
                if status == Status::Pending {
                    continue;
                }
                let mut entries = self.pool_map.remove_entry_and_descendants(id);
                entries.sort_unstable_by_key(|entry| entry.ancestors_count);
                for mut entry in entries {
                    let tx_hash = entry.transaction().hash();
                    entry.reset_statistic_state();
                    let ret = self.add_pending(entry);
                    debug!(
                        "remove_by_detached_proposal from {:?} {} add_pending {:?}",
                        status, tx_hash, ret
                    );
                }
            }
        }
    }
