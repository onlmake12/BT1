### Title
Hardcoded `DEFAULT_BYTES_PER_CYCLES` in `get_transaction_weight` Ignores Runtime Consensus Parameters, Causing Incorrect Fee Rate Accounting - (File: `util/types/src/core/tx_pool.rs`)

---

### Summary

`get_transaction_weight()` uses a compile-time constant `DEFAULT_BYTES_PER_CYCLES` derived from mainnet defaults instead of the actual runtime consensus ratio `max_block_bytes / max_block_cycles`. When a node runs with non-default consensus parameters (a supported configuration via `Params`), every fee rate calculation, tx-pool ordering decision, and fee estimate is computed against the wrong scale factor — directly analogous to OmoOracle assuming all tokens have 18 decimals.

---

### Finding Description

`get_transaction_weight` in `util/types/src/core/tx_pool.rs` converts a transaction's cycle count into a byte-equivalent weight using a hardcoded floating-point constant:

```rust
// util/types/src/core/tx_pool.rs
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;
// comment: "Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec."

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

This constant is derived from the compile-time defaults:

```
MAX_BLOCK_BYTES  = 597 * 1_000 = 597_000
MAX_BLOCK_CYCLES = 3_500_000 * 1_000 = 3_500_000_000
DEFAULT_BYTES_PER_CYCLES = 597_000 / 3_500_000_000 ≈ 0.000_170_571_4
``` [2](#0-1) 

However, `max_block_bytes` and `max_block_cycles` are **runtime-configurable** fields of the `Consensus` struct, settable via the chain spec `Params`: [3](#0-2) [4](#0-3) 

`get_transaction_weight` is never passed the `Consensus` object and cannot read the actual runtime ratio. It always uses the hardcoded mainnet constant regardless of what the node is actually configured with.

The same problem appears in `weight_units_flow.rs`, which imports the compile-time constant `MAX_BLOCK_BYTES` directly and uses it to compute how much weight each block removes:

```rust
use ckb_chain_spec::consensus::MAX_BLOCK_BYTES;
// ...
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
``` [5](#0-4) [6](#0-5) 

---

### Impact Explanation

`get_transaction_weight` is the single function used for all of the following:

1. **Tx-pool fee rate ordering** — `TxEntry::fee_rate()`, `AncestorsScoreSortKey`, `EvictKey` all call `get_transaction_weight` to rank transactions. [7](#0-6) [8](#0-7) 

2. **Fee estimation RPC** — both `ConfirmationFraction` and `WeightUnitsFlow` estimators call `get_transaction_weight` to compute per-transaction fee rates. [9](#0-8) 

3. **Historical fee rate statistics** — `FeeRateCollector::statistics` calls `get_transaction_weight` on stored block data. [10](#0-9) 

**Concrete miscalculation example** (chain with `max_block_cycles` halved to `1_750_000_000`):

- Actual `bytes_per_cycle` = `597_000 / 1_750_000_000 ≈ 0.000_341`
- Hardcoded `DEFAULT_BYTES_PER_CYCLES` = `0.000_170_571_4`
- For a tx with `size=100`, `cycles=1_000_000`:
  - Correct weight = `max(100, 341)` = **341**
  - Computed weight = `max(100, 170)` = **170** (underestimated by ~50%)
- The tx's fee rate is **overestimated by ~2×**, causing it to jump ahead of legitimately higher-fee-rate transactions in the tx-pool.

A tx-pool submitter can exploit this by crafting cycles-heavy transactions that appear to have a higher fee rate than they actually do, displacing legitimately higher-paying transactions from block inclusion.

---

### Likelihood Explanation

**Medium.** The bug is silent on mainnet (where the constants match exactly). It activates on any node running a chain spec with non-default `max_block_bytes` or `max_block_cycles` — a fully supported configuration via the `Params` struct and `build_consensus()`. Devnet and custom testnet deployments routinely adjust these parameters. Any unprivileged tx-pool submitter on such a network can trigger the miscalculation simply by submitting a cycles-heavy transaction.

---

### Recommendation

Pass the runtime `Consensus` (or its `max_block_bytes`/`max_block_cycles` fields) into `get_transaction_weight` and compute the ratio dynamically:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64, max_block_bytes: u64, max_block_cycles: u64) -> u64 {
    let bytes_per_cycle = max_block_bytes as f64 / max_block_cycles as f64;
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * bytes_per_cycle) as u64,
    )
}
```

Similarly, replace the compile-time `MAX_BLOCK_BYTES` import in `weight_units_flow.rs` with the runtime value from the `Consensus` object passed through the estimator.

---

### Proof of Concept

1. Start a CKB devnet node with a custom chain spec setting `max_block_cycles` to `1_750_000_000` (half the default).
2. Submit two transactions:
   - **Tx A**: `size=200`, `cycles=1_000_000`, `fee=200 shannons` → actual weight=341, actual fee rate≈586 shannons/KW
   - **Tx B**: `size=200`, `cycles=100`, `fee=150 shannons` → actual weight=200, actual fee rate=750 shannons/KW
3. Observe that `get_transaction_weight` computes:
   - Tx A weight = `max(200, 170)` = 200 → computed fee rate = 1000 shannons/KW (overestimated)
   - Tx B weight = `max(200, 17)` = 200 → computed fee rate = 750 shannons/KW
4. Tx A is ranked **above** Tx B in the tx-pool despite having a lower actual fee rate, causing Tx B (the higher actual payer) to be displaced from block inclusion.
