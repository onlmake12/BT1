### Title
`RecentReject::put()` silently destroys just-written entry when `shrink()` randomly drops the same shard — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put()` writes an entry to shard X, then conditionally calls `shrink()`, which randomly drops and recreates one of the `shard_num` (default 5) column families. If `shrink()` selects shard X, the entry is permanently destroyed, yet `put()` returns `Ok(())`. The caller has no way to detect the loss. A subsequent `get()` returns `None`, causing `get_transaction` RPC to report `TxStatus::Unknown` instead of `TxStatus::Rejected`, and the recent-reject guard is silently bypassed.

---

### Finding Description

**Root cause — missing counter update in `put()`:** [1](#0-0) 

`self.total_keys_num` is **never incremented** inside `put()`. The local binding `total_keys_num` (line 62) is computed only for the comparison; `self.total_keys_num` stays frozen at whatever value it held at initialization or after the last `shrink()`. This means `shrink()` fires only when `self.total_keys_num` was already `>= count_limit` at the time of the call — concretely, on the first `put()` after a node restart whose DB already held `>= count_limit` entries, or after a previous `shrink()` left the estimate at or above the limit.

**The invariant-breaking sequence:**

1. `put()` writes entry to shard X via `self.db.put(&shard, ...)` (line 60).
2. `self.total_keys_num.checked_add(1) > count_limit` → `shrink()` is called (line 64).
3. `shrink()` picks a random shard via `thread_rng()` (line 106). [2](#0-1) 

4. If the randomly chosen shard equals shard X, `drop_cf(X)` destroys all data in X (line 107), then `create_cf_with_ttl(X, ttl)` recreates it empty (line 108).
5. `put()` returns `Ok(())` — the caller observes success.
6. A subsequent `get()` on the same hash returns `None` because the CF is now empty.

**Shard assignment is deterministic and attacker-observable:** [3](#0-2) 

Shard = `u32::from_le_bytes(hash[0..4]) % shard_num`. An attacker can craft a transaction whose hash maps to any desired shard. With `DEFAULT_SHARDS = 5`, the probability that `shrink()` drops that exact shard is 1/5 per triggering `put()`.

---

### Impact Explanation

- `get_transaction` RPC returns `TxStatus::Unknown` instead of `TxStatus::Rejected` for the affected transaction.
- The recent-reject guard in `after_process` / `put_recent_reject` is bypassed: [4](#0-3) 

- The transaction can be re-relayed by the submitter without the node recognizing it as previously rejected, causing repeated re-verification (CPU waste / DoS amplification).
- The `register_reject` callback in `shared_builder.rs` also calls `recent_reject.put()` and silently swallows the loss via an `error!` log: [5](#0-4) 

---

### Likelihood Explanation

The trigger condition (`self.total_keys_num >= count_limit`) is reached on any node that has accumulated `>= keep_rejected_tx_hashes_count` rejected transactions and then restarts — a routine operational event. The default `keep_rejected_tx_hashes_count` is a finite configured value. Once triggered, the probability of destroying the just-written entry is 1/`shard_num` = 20% with the default of 5 shards. An attacker who can submit many rejected transactions (trivially achievable via P2P relay) can repeatedly trigger this condition.

---

### Recommendation

1. **Increment `self.total_keys_num` in `put()`** after a successful write, so the counter tracks actual inserts:
   ```rust
   self.total_keys_num = self.total_keys_num.saturating_add(1);
   ```
2. **Call `shrink()` before writing**, not after, so the just-written entry is never in the shard that may be dropped.
3. Alternatively, **exclude the just-written shard from `shrink()`'s random selection** by passing it as a parameter and retrying if the random pick matches.

---

### Proof of Concept

```rust
// shard_num=2, count_limit=1, ttl=-1
let mut rr = RecentReject::build(tmp, 2, 1, -1).unwrap();

// Manually set total_keys_num to count_limit so shrink fires on next put
rr.total_keys_num = rr.count_limit; // simulate DB at limit

// Craft a hash that maps to shard 0 (first 4 bytes little-endian % 2 == 0)
let hash = Byte32::new([0u8; 32]); // 0 % 2 == 0 → shard "0"

// Instrument shrink() to always drop shard "0" (or just run in a loop until it does)
rr.put(&hash, Reject::Malformed("x".into(), Default::default())).unwrap();

// With 50% probability (shard_num=2), get() returns None immediately after Ok put
assert!(rr.get(&hash).unwrap().is_none()); // invariant violated
```

With `shard_num = 2` the probability of hitting the bug is 50% per attempt; with the default of 5 it is 20%. Repeated submissions make reliable triggering straightforward.

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        let total_keys_num = self.estimate_total_keys_num()?;
        self.total_keys_num = total_keys_num;
        Ok(total_keys_num)
    }
