### Title
Ancestors' `descendants_*` weight fields not decremented when `remove_entry_and_descendants` clears links before calling `remove_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `pool_map.rs`, `remove_entry_and_descendants` first clears all parent/child links for every tx in the removed subtree, then calls `remove_entry` for each. Inside `remove_entry`, `update_ancestors_index_key` relies on those same links to locate the surviving ancestors and decrement their `descendants_*` accounting fields. Because the links are already gone, `calc_ancestors` returns an empty set, the decrement is silently skipped, and every surviving ancestor of the removed subtree permanently carries inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` values.

---

### Finding Description

`remove_entry_and_descendants` executes in three phases:

```
1. removed_ids = [root] + calc_descendants(root)
2. for id in removed_ids { remove_entry_links(id) }   // clears ALL links
3. for id in removed_ids { remove_entry(id) }
``` [1](#0-0) 

Inside `remove_entry`, the two weight-update helpers are called:

```rust
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
``` [2](#0-1) 

`update_ancestors_index_key` resolves the ancestor set via `self.links.calc_ancestors(...)`:

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [3](#0-2) 

Because phase 2 already called `remove_entry_links` for every removed ID, `calc_ancestors` returns an empty set for every tx in the subtree. The `sub_descendant_weight` call is never reached for any surviving ancestor. The developer comment acknowledges the intent ("so that we won't update_descendants_index_key in remove_entry"), but the same link-clearing also silently suppresses the necessary ancestor-side decrement. [1](#0-0) 

By contrast, when a single tx is removed via `remove_entry` directly (links still intact), `calc_ancestors` returns the correct set and the decrement is applied correctly. [2](#0-1) 

The `descendants_*` fields are initialized correctly when a tx is added:

```rust
self.update_ancestors_index_key(entry, EntryOp::Add);
``` [4](#0-3) 

So the add path increments the ancestor's `descendants_*` fields, but the subtree-remove path never decrements them — a permanent one-sided accounting error.

---

### Impact Explanation

The `descendants_*` fields feed directly into `EvictKey`:

```rust
let descendants_weight =
    get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
let feerate = FeeRate::calculate(entry.fee, weight);
EvictKey {
    fee_rate: descendants_feerate.max(feerate),
    ...
    descendants_count: entry.descendants_count,
}
``` [5](#0-4) 

`EvictKey` is the sole criterion used by `next_evict_entry` and `limit_size` to decide which tx to drop when the pool is full: [6](#0-5) [7](#0-6) 

A surviving ancestor with inflated `descendants_fee` appears to carry a high-fee descendant cluster that no longer exists. Its `EvictKey.fee_rate` is overstated, so it is ranked as more valuable than it truly is and is systematically skipped during eviction. Legitimate higher-fee-rate transactions may be evicted in its place.

The `score` key (used for mining priority) is based only on `ancestors_*` fields and is unaffected. [8](#0-7) 

---

### Likelihood Explanation

`remove_entry_and_descendants` is called from multiple reachable code paths triggered by ordinary, unprivileged tx-pool activity:

- **`resolve_conflict`** — invoked on every committed block to evict pool txs whose inputs are now spent. Any tx that has a surviving parent in the pool will leave that parent with inflated fields.
- **`check_and_record_ancestors`** — invoked when a new tx is admitted and cell-ref ancestors must be evicted to satisfy the ancestor-count limit.
- **`remove_by_detached_proposal`** — invoked during every reorg.
- **`limit_size`** — invoked after every pool insertion. [1](#0-0) [9](#0-8) [10](#0-9) 

An unprivileged submitter can deliberately trigger the bug:

1. Submit low-fee Tx A (parent).
2. Submit high-fee Tx B and Tx C as children of Tx A.
3. Submit a conflicting replacement for Tx B (RBF). `resolve_conflict` calls `remove_entry_and_descendants(Tx B)`, removing Tx B and Tx C.
4. Tx A's `descendants_*` fields remain inflated with Tx B's and Tx C's fee/size/cycles.
5. Tx A is now effectively immune to eviction despite its low individual fee rate.

---

### Recommendation

Before clearing links in `remove_entry_and_descendants`, snapshot the ancestors of the root tx and the full weight of the subtree being removed, then apply the decrement to those ancestors after the removal. For example:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

+   // Capture ancestors of the root BEFORE links are cleared
+   let root_ancestors = self.links.calc_ancestors(id);

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

+   // Decrement each surviving ancestor's descendants_* fields
+   for removed_entry in &removed {
+       for anc_id in &root_ancestors {
+           self.entries.modify_by_id(anc_id, |e| {
+               e.inner.sub_descendant_weight(removed_entry);
+               e.evict_key = e.inner.as_evict_key();
+           });
+       }
+   }

    removed
}
```

### Proof of Concept

**State before:**

| Tx | In pool | `descendants_count` | `descendants_fee` |
|----|---------|---------------------|-------------------|
| A  | yes     | 1                   | fee_A             |

**After adding Tx B (child of A) and Tx C (child of B):**

| Tx | `descendants_count` | `descendants_fee`         |
|----|---------------------|---------------------------|
| A  | 3                   | fee_A + fee_B + fee_C     |
| B  | 2                   | fee_B + fee_C             |

**After `remove_entry_and_descendants(B)` (removes B and C):**

Expected:

| Tx | `descendants_count` | `descendants_fee` |
|----|---------------------|-------------------|
| A  | 1                   | fee_A             |

Actual (due to bug):

| Tx | `descendants_count` | `descendants_fee`         |
|----|---------------------|---------------------------|
| A  | 3 (**inflated**)    | fee_A + fee_B + fee_C (**inflated**) |

Tx A's `EvictKey.fee_rate` is computed from the inflated `descendants_fee` / `descendants_size`, making it appear far more valuable than it is. `limit_size` will skip Tx A and evict other transactions instead. [1](#0-0) [3](#0-2) [11](#0-10) [5](#0-4)

    fn record_entry_descendants(&mut self, entry: &TxEntry) {
        let tx_short_id: ProposalShortId = entry.proposal_short_id();
        let outputs = entry.transaction().output_pts();
        let mut children = HashSet::new();

        // collect children
        for o in outputs {
            if let Some(ids) = self.edges.get_deps_ref(&o).cloned() {
                children.extend(ids);
            }
            if let Some(id) = self.edges.get_input_ref(&o).cloned() {
                children.insert(id);
            }
        }
        // update children
        if !children.is_empty() {
            for child in &children {
                self.links.add_parent(child, tx_short_id.clone());

            if let Some(links) = self.links.inner.get_mut(&tx_short_id) {
                links.children.extend(children);
            }
            self.update_descendants_index_key(entry, EntryOp::Add);
        }
        // update ancestor's index key for adding new entry
        self.update_ancestors_index_key(entry, EntryOp::Add);

        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
