### Title
Stale `dead_cells` Secondary Index in Indexer Pool Overlay Not Properly Maintained After RBF Replacement — (File: `util/indexer-sync/src/pool.rs`)

---

### Summary

The `Pool` struct in `util/indexer-sync/src/pool.rs` maintains a `dead_cells: HashSet<OutPoint>` as a secondary index tracking which cells are consumed by pending transactions in the tx-pool. This secondary index is updated via asynchronous notifications (`new_transaction`, `reject_transaction`) processed by `tokio::select!` on separate channels. When a transaction is replaced via RBF, the `new_transaction` notification for the incoming transaction and the `reject_transaction` notification for the evicted transaction are dispatched on **different channels** and can be consumed in non-deterministic order by `tokio::select!`. If `new_transaction(new_tx)` is processed before `reject_transaction(old_tx)`, the shared input cell is removed from `dead_cells` even though the new transaction still consumes it — leaving the secondary index permanently stale until the next block commit.

---

### Finding Description

The `Pool` struct is an in-memory overlay used by the indexer to track which cells are consumed by pending transactions: [1](#0-0) 

It is updated via two notification paths: [2](#0-1) 

The `PoolService::index_tx_pool()` function subscribes to `new_transaction_receiver` and `reject_transaction_receiver` as **separate channels**, processed inside a single `tokio::select!` loop: [3](#0-2) 

The root cause is that `dead_cells` is a plain `HashSet<OutPoint>` with no reference counting. In the RBF scenario:

1. **Tx A** is in the pool consuming cell **X** → `dead_cells = {X}`
2. **Tx B** is submitted, replacing Tx A (both consume cell X)
3. The RBF path in `process_rbf` first calls `call_reject` for Tx A, then adds Tx B to the pool: [4](#0-3) 

4. This sends `reject_transaction(Tx A)` to `reject_transaction_receiver` and `new_transaction(Tx B)` to `new_transaction_receiver` — on **different channels**.
5. When the `index_tx_pool` task wakes up with both messages ready, `tokio::select!` may pick `new_transaction_receiver` first (pseudo-random branch selection to avoid starvation):
   - `new_transaction(Tx B)` → `dead_cells.insert(X)` → no change (X already present)
   - `reject_transaction(Tx A)` → `dead_cells.remove(X)` → **X removed**
6. Result: `dead_cells = {}`, but Tx B is still in the pool consuming X.

The `HashSet` has no way to know that X is now "owned" by Tx B rather than Tx A. Removing it because Tx A was rejected is incorrect when Tx B still holds the same input.

---

### Impact Explanation

The `dead_cells` set is queried via `is_consumed_by_pool_tx()` to filter cells from indexer `get_cells` RPC responses: [5](#0-4) 

When `dead_cells` is stale (missing cell X), `get_cells` returns cell X as a live, spendable cell even though it is already consumed by a pending transaction (Tx B). Any client that acts on this response and attempts to spend X will have its transaction rejected by the tx-pool with a "dead cell" error. This breaks the contract between the indexer and its consumers: the indexer's `get_cells` output is supposed to reflect the real-time pool state, but it silently returns incorrect data. The stale state persists until the next block is committed and `transactions_committed()` is called, which clears the entry via the block-append path.

This is a direct analog to the `RdpxDecayingBonds` bug: a secondary data structure (`dead_cells`) tracking ownership/consumption state is not correctly updated when the primary state (tx-pool membership) changes via RBF replacement.

---

### Likelihood Explanation

- RBF is an opt-in feature enabled via `min_rbf_rate` config, but it is supported in production CKB nodes.
- The indexer pool overlay is enabled when `index_tx_pool = true` in the indexer config.
- The race window is narrow but real: `tokio::select!` uses pseudo-random branch selection when multiple branches are ready simultaneously, which is exactly the case when both notifications arrive before the task wakes up.
- Any unprivileged tx-pool submitter can trigger this by submitting a higher-fee replacement transaction (RBF) for any existing pending transaction.
- The stale state is self-healing on the next block commit, but the window between an RBF event and the next block (potentially minutes) is long enough to affect clients polling `get_cells`.

---

### Recommendation

Replace `dead_cells: HashSet<OutPoint>` with a reference-counted map `dead_cells: HashMap<OutPoint, usize>`. Increment the count on `new_transaction` and decrement (removing the entry when count reaches zero) on `transaction_rejected` and `transaction_committed`. This makes the secondary index robust against out-of-order notification delivery and correctly handles the case where multiple transactions in the pool reference the same cell (e.g., during RBF transitions).

```rust
// Instead of:
dead_cells: HashSet<OutPoint>

// Use:
dead_cells: HashMap<OutPoint, usize>

// new_transaction:
*self.dead_cells.entry(out_point).or_insert(0) += 1;

// transaction_rejected / transaction_committed:
if let Entry::Occupied(mut e) = self.dead_cells.entry(out_point) {
    if *e.get() <= 1 { e.remove(); } else { *e.get_mut() -= 1; }
}
```

### Proof of Concept

```
1. Enable indexer with index_tx_pool = true and min_rbf_rate > 0.
2. Submit Tx_A spending cell X (fee = F).
   → Pool.dead_cells = {X}
3. Submit Tx_B spending cell X (fee = F + delta, replacing Tx_A via RBF).
   → process_rbf fires:
       call_reject(Tx_A) → reject_transaction channel ← message 1
       add Tx_B to pool  → new_transaction channel    ← message 2
4. index_tx_pool task wakes up; tokio::select! picks new_transaction first:
       new_transaction(Tx_B): dead_cells.insert(X) → {X} (no change)
5. tokio::select! picks reject_transaction next:
       reject_transaction(Tx_A): dead_cells.remove(X) → {}
6. Pool.dead_cells = {} but Tx_B is still pending and consuming X.
7. Call get_cells filtering by X's lock script:
   → is_consumed_by_pool_tx(X) returns false (stale)
   → X is returned as a live cell — incorrect.
``` [6](#0-5)

#[derive(Default)]
pub struct Pool {
    dead_cells: HashSet<OutPoint>,
}

impl Pool {
    /// the tx has been committed in a block, it should be removed from pending dead cells
    pub fn transaction_committed(&mut self, tx: &TransactionView) {
        for input in tx.inputs() {
            self.dead_cells.remove(&input.previous_output());
        }
    }

    /// the tx has been rejected for some reason, it should be removed from pending dead cells
    pub fn transaction_rejected(&mut self, tx: &TransactionView) {
        for input in tx.inputs() {
            self.dead_cells.remove(&input.previous_output());
        }
    }

    /// a new tx is submitted to the pool, mark its inputs as dead cells
    pub fn new_transaction(&mut self, tx: &TransactionView) {
        for input in tx.inputs() {
            self.dead_cells.insert(input.previous_output());
        }
    }

    /// Return weather out_point referred cell consumed by pooled transaction
    pub fn is_consumed_by_pool_tx(&self, out_point: &OutPoint) -> bool {
        self.dead_cells.contains(out_point)
    }

    /// the txs has been committed in a block, it should be removed from pending dead cells
    pub fn transactions_committed(&mut self, txs: &[TransactionView]) {
        for tx in txs {
            self.transaction_committed(tx);
        }
    }

    /// return all dead cells
    pub fn dead_cells(&self) -> impl Iterator<Item = &OutPoint> {
        self.dead_cells.iter()
    }
}
