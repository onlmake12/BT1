### Title
Peer Store Permanently Blocked by Group::None Collapse via Onion3 Address Flooding — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

The `check_purge` phase-2 eviction path in `PeerStore` uses `take(len / 2)` where `len` is the number of distinct network groups. When all 16384 entries map to `Group::None` (as Onion3 addresses do), `len = 1`, integer division yields `take(0)`, no candidates are produced, and `EvictionFailed` is returned permanently. Every subsequent `add_addr` call propagates this error, making the peer store permanently full.

---

### Finding Description

**Step 1 — Onion3 addresses are accepted and stored.**

`base_addr` strips only `Ws`, `Wss`, `Memory`, and `Tls` protocols; Onion3 passes through unchanged. [1](#0-0) 

`add_addr` adds any non-banned address after calling `check_purge`: [2](#0-1) 

**Step 2 — Onion3 addresses are connectable (survive phase-1).**

A freshly added address has `last_connected_at_ms = 0`, `attempts_count = 0`, `last_tried_at_ms = 0`. None of the three failure conditions in `is_connectable` fire, so every Onion3 address is considered connectable and is NOT evicted in phase-1. [3](#0-2) 

**Step 3 — Onion3 addresses all map to `Group::None`.**

`Group::from(&Multiaddr)` calls `multiaddr_to_socketaddr`, which only succeeds for IP4/IP6. Onion3 addresses fall through to the final `Group::None` return. [4](#0-3) 

**Step 4 — The `take(len / 2)` integer-division bug.**

In phase-2, `len` is the number of distinct groups in `peers_by_network_group`. With all 16384 entries in `Group::None`, `len = 1`. Rust integer division: `1 / 2 = 0`. `take(0)` produces an empty iterator, `candidate_peers` is empty, and `EvictionFailed` is returned. [5](#0-4) 

The secondary guard `addrs.len() > 4` is never even reached because `take(0)` yields nothing.

**Step 5 — Discovery protocol is the attacker's entry point.**

The discovery protocol explicitly handles Onion3 addresses and includes them in `fetch_random` results, confirming they are a first-class supported address type that flows through `add_addr`: [6](#0-5) 

The discovery protocol caps a single announcement at `MAX_ADDR_TO_SEND = 1000`, but an attacker needs only 17 such messages (across one or more connections) to fill 16384 slots. [7](#0-6) 

---

### Impact Explanation

Once the peer store is full with 16384 `Group::None` entries, every call to `add_addr` returns `Err(PeerStoreError::EvictionFailed)`. The node can no longer learn about new peers from discovery, DNS seeding, or identify protocol. Existing connections are unaffected, but the node's ability to recover from peer churn or network partitions is permanently impaired until the node is restarted and the peer store is cleared.

---

### Likelihood Explanation

- No special node configuration is required; Onion3 addresses are accepted by default.
- The attacker needs only a single TCP connection to the victim to send discovery messages.
- 16384 addresses at 1000 per message requires ~17 discovery announcements — trivially achievable.
- The condition is permanent until node restart; there is no self-healing path.

---

### Recommendation

Replace `take(len / 2)` with a minimum-of-1 guard:

```rust
let take_count = (len / 2).max(1);
peers.into_iter().take(take_count)
```

Additionally, consider assigning Onion3 addresses to a dedicated `Group::Onion3([u8; N])` variant keyed on the first N bytes of the onion address, so they are distributed across multiple groups and subject to normal per-group eviction. [8](#0-7) [9](#0-8)

    let mut host = [0u8; 35];
