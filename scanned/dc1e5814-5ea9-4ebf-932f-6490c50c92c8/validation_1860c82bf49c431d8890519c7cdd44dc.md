Audit Report

## Title
Ancestor Descendant-Stats Not Updated in `remove_entry_and_descendants`, Leading to Stale Eviction Keys - (File: tx-pool/src/component/pool_map.rs)

## Summary

In `PoolMap::remove_entry_and_descendants`, all parent/child links for every entry in the removed subtree are stripped via `remove_entry_links` before `remove_entry` is called on each. Because `update_ancestors_index_key` resolves ancestors through `self.links`, it finds an empty ancestor set and never calls `sub_descendant_weight` on any surviving ancestor of the removed subtree root. Those ancestors permanently retain inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`, which are the sole inputs to `EvictKey`. An unprivileged user can exploit this via repeated RBF replacements to make a low-fee-rate parent transaction immune to pool eviction, causing legitimate transactions to be evicted in its place.

## Finding Description

`remove_entry_and_descendants` (L252–265) first collects the root and all its descendants, then calls `remove_entry_links` on every collected entry before calling `remove_entry` on each:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    for id in &removed_ids {
        self.remove_entry_links(id);   // strips ALL links first
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

`remove_entry_links` (L418–430) removes the entry from its parents' children sets, removes the parents from the entry's parent set, and then calls `self.links.remove(id)`, fully erasing the entry from `self.links`. When `remove_entry` (L235–250) subsequently calls `update_ancestors_index_key` (L432–445), it calls `self.links.calc_ancestors(&child.proposal_short_id())`. Because the entry's record in `self.links` was already deleted, `calc_ancestors` returns an empty set. The loop body — which calls `e.inner.sub_descendant_weight(child)` and recomputes `e.evict_key` — never executes for any surviving ancestor.

By contrast, when `remove_entry` is called in isolation (not through `remove_entry_and_descendants`), links are still intact at the time `update_ancestors_index_key` runs, so ancestor stats are correctly decremented. The asymmetry is exclusive to the batch-removal path.

The stale fields feed directly into `EvictKey` (entry.rs L234–247): `fee_rate` is set to `max(descendants_feerate, feerate)`. A surviving ancestor with inflated `descendants_fee` will have an artificially high `fee_rate` in its `EvictKey`. `next_evict_entry` (L380–385) iterates `iter_by_evict_key()` in ascending order, so the inflated ancestor is sorted toward the back and skipped during eviction.

`remove_entry_and_descendants` is called from all major removal paths: `limit_size` (pool.rs L307), `remove_by_detached_proposal` (pool.rs L343), `remove_tx` (pool.rs L359), `resolve_conflict_header_dep` (pool_map.rs L285), and `check_and_record_ancestors` (pool_map.rs L618).

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can submit a parent transaction with a very low fee rate and a child transaction with a high fee rate, then repeatedly replace the child via RBF. Each replacement calls `remove_entry_and_descendants` on the old child without decrementing the parent's descendant stats, then `add_descendant_weight` is called again when the replacement child is inserted. After N replacements, `tx_parent.descendants_fee` equals N × (old child fee) + (replacement child fee), making `descendants_feerate` — and therefore `EvictKey.fee_rate` — grow unboundedly. The parent becomes effectively immune to eviction. When the pool is full and `limit_size` runs, legitimate high-fee-rate transactions are evicted instead of the stale-keyed low-fee-rate parent, degrading pool quality and enabling sustained pool congestion at low cost to the attacker.

## Likelihood Explanation

The bug is triggered by any call to `remove_entry_and_descendants` on a transaction whose parent remains in the pool — a normal operational condition. An unprivileged submitter can deliberately engineer this state: submit a parent–child chain, then trigger child removal through repeated RBF replacements. Each RBF requires only a marginally higher fee for the replacement child (the minimum RBF fee bump), so the cost to inflate the parent's eviction key by a large factor is low. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation

Before stripping any links in `remove_entry_and_descendants`, resolve the ancestors of the subtree root while links are still intact and update their descendant stats. Concretely:

1. Before the `remove_entry_links` loop, call `self.links.calc_ancestors(id)` to obtain the set of surviving ancestors.
2. For each entry being removed, call `sub_descendant_weight` on each surviving ancestor and recompute `evict_key`.

Alternatively, restructure the function so that `update_ancestors_index_key` for the root entry is called before any `remove_entry_links` invocation, mirroring the symmetric behavior of the single-entry `remove_entry` path.

## Proof of Concept

1. Submit `tx_parent` (low fee rate, e.g. 1 shannon/byte) spending an on-chain cell.
2. Submit `tx_child` (high fee rate, e.g. 1000 shannons/byte) spending `tx_parent`'s output. At this point `tx_parent.descendants_fee = tx_child.fee`, `tx_parent.descendants_count = 2`.
3. Submit `tx_child_v2` (RBF replacement of `tx_child`, slightly higher fee) spending the same input as `tx_child`. This triggers `remove_entry_and_descendants(tx_child_id)`.
4. Because `remove_entry_links` is called before `update_ancestors_index_key`, `tx_parent.descendants_fee` is **not** decremented. Then `tx_child_v2` is inserted and `add_descendant_weight` increments `tx_parent.descendants_fee` again.
5. After step 4: `tx_parent.descendants_fee = tx_child.fee + tx_child_v2.fee` instead of `tx_child_v2.fee`.
6. Repeat steps 3–5 N times. `tx_parent.descendants_fee` grows by N × `tx_child.fee`.
7. Verify: call `next_evict_entry` on a full pool. `tx_parent` is never selected despite its intrinsic fee rate of 1 shannon/byte, while legitimate transactions with higher intrinsic fee rates are evicted instead.

A unit test can be written directly against `PoolMap` in `tx-pool/src/component/tests/pending.rs` following the pattern of `test_pool_min_descendants_evict`, asserting that after N RBF cycles `tx_parent.descendants_fee == tx_child_vN.fee` (not N × old fee) and that `next_evict_entry` returns `tx_parent` before any higher-fee-rate transaction.
