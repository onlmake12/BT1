Audit Report

## Title
Unbounded `last_n_numbers` allocation bypasses `GET_LAST_STATE_PROOF_LIMIT` in `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The limit check in `execute()` only bounds `difficulties.len() + last_n_blocks * 2`, but with `last_n_blocks=0` and `difficulties=[]` this evaluates to zero and passes unconditionally. In the `else` branch, `last_n_numbers` is built as `(difficulty_boundary_block_number..last_block_number)`, which with a crafted `difficulty_boundary=1` collapses to `(start_block_number..last_block_number)` — a range of arbitrary length proportional to chain height. No secondary bound check exists before `complete_headers` iterates every entry, performing two DB reads and one heap allocation per block, causing OOM on a synced node.

## Finding Description

**Limit check (lines 201–205):** The guard only covers `difficulties.len() + last_n_blocks * 2`. With `last_n_blocks=0` and `difficulties=[]`, the expression is `0`, which is not greater than `GET_LAST_STATE_PROOF_LIMIT` (1000), so the check passes. [1](#0-0) 

**Branch selection (lines 291–292):** The `if` branch requires `last_block_number - start_block_number <= last_n_blocks`. With `last_n_blocks=0` and a chain tip far ahead of `start_block_number`, the condition is false and the `else` branch is taken. [2](#0-1) 

**`difficulty_boundary_block_number` resolution (lines 29–36):** `get_first_block_total_difficulty_is_not_less_than` checks whether `start_total_difficulty >= min_total_difficulty` at the very first block. With `difficulty_boundary=U256::from(1)`, every block's cumulative difficulty satisfies this, so the function immediately returns `Some((start_block_number, …))`, setting `difficulty_boundary_block_number = start_block_number`. [3](#0-2) 

**Adjustment guard (line 313):** The guard `if last_block_number - difficulty_boundary_block_number < last_n_blocks` with `last_n_blocks=0` evaluates to `(H - start) < 0`, which is false for unsigned arithmetic when `H > start`, so `difficulty_boundary_block_number` is not adjusted. [4](#0-3) 

**Unbounded `last_n_numbers` (lines 318–319):** The resulting range is `(start_block_number..last_block_number)` — potentially millions of entries on mainnet. [5](#0-4) 

**No bound check before `complete_headers` (lines 350–366):** `block_numbers` is assembled and passed directly to `complete_headers` with no size assertion. `complete_headers` iterates every entry, calling `get_ancestor`, fetching the full block, calling `chain_root_mmr(n-1).get_root()`, and pushing a `VerifiableHeader` onto the heap — O(H) DB reads and allocations in a single request. [6](#0-5) [7](#0-6) 

**All input validation checks are bypassed:** The `difficulties` monotonicity check, the `difficulty_boundary > max(difficulties)` check, and the start-difficulty check all short-circuit on empty `difficulties`, leaving no rejection path. [8](#0-7) 

## Impact Explanation
A single unauthenticated P2P peer can crash the light-client-protocol-server process via OOM by sending one crafted `GetLastStateProof` message. On CKB mainnet (height ~millions), the attack allocates O(H) `VerifiableHeader` objects and issues O(H) DB reads in a single synchronous request handler. This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points). [9](#0-8) 

## Likelihood Explanation
The attack requires only a TCP connection to a node with the light-client protocol server enabled and a single crafted message. No authentication, PoW, or stake is required. CKB mainnet has been live since 2019 with block height well into the millions, making the O(H) allocation immediately catastrophic. The attack is trivially repeatable and requires no victim interaction. [10](#0-9) 

## Recommendation
After assembling `block_numbers` and before calling `complete_headers`, assert that the total count does not exceed `GET_LAST_STATE_PROOF_LIMIT`:

```rust
let block_numbers = reorg_last_n_numbers
    .into_iter()
    .chain(sampled_numbers)
    .chain(last_n_numbers)
    .collect::<Vec<_>>();

if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage
        .with_context("too many blocks to prove");
}
```

This ensures `|block_numbers| ≤ GET_LAST_STATE_PROOF_LIMIT` regardless of how `difficulty_boundary` is placed or what `last_n_blocks` is set to. [6](#0-5) 

## Proof of Concept

```
Preconditions:
  - Node synced to chain height H >> 1 (e.g., H = 2_000_000)
  - Light-client protocol server enabled

Craft GetLastStateProof:
  last_n_blocks       = 0
  difficulties        = []
  difficulty_boundary = U256::from(1)
  start_number        = 1
  start_hash          = hash(block #1)   // valid main-chain block
  last_hash           = hash(tip block)  // valid main-chain tip

Execution path:
  1. Limit check: 0 + 0*2 = 0 ≤ 1000                          → passes
  2. is_main_chain(last_hash)                                   → true
  3. get_ancestor(last_hash, 1).hash() == start_hash            → reorg_last_n_numbers = []
  4. last_block_number - start_block_number = ~2M > 0           → else branch
  5. get_first_block_total_difficulty_is_not_less_than(1, 2M, 1)
       → td[1] ≥ 1 → returns Some((1, td[1]))
       → difficulty_boundary_block_number = 1
  6. 2M - 1 < 0 is false (unsigned)                            → no adjustment
  7. last_n_numbers = (1..2_000_000)                           → Vec of ~2M entries
  8. block_numbers.len() = ~2M, no bound check
  9. complete_headers iterates ~2M times                        → OOM crash
``` [11](#0-10)
