Audit Report

## Title
Wrong Variable Reference in `script_exists_in_output()` Causes Premature Script Deletion During Reorg ‚Äî (File: util/rich-indexer/src/indexer/remove.rs)

## Summary
`script_exists_in_output` at line 252 re-reads `row_lock` (the lock-script EXISTS result) instead of `row_type` (the type-script EXISTS result) in its second `match` block. On PostgreSQL, where `try_get::<bool, _>(0)` succeeds, the function returns the lock-script existence result for both checks, causing `rollback_block` to incorrectly delete script rows that are still referenced as `type_script_id` in surviving outputs, silently corrupting the rich-indexer database on every reorg involving type-scripted cells.

## Finding Description
`script_exists_in_output` (lines 204‚Äì257) runs two SQL `EXISTS` queries: one against `lock_script_id` (stored in `row_lock`) and one against `type_script_id` (stored in `row_type`). The first `match` block (lines 223‚Äì235) correctly uses `row_lock` and short-circuits with `Ok(true)` if the lock query matched. The second `match` block (lines 252‚Äì256) is supposed to evaluate `row_type`, but instead reads `row_lock` again:

```rust
// line 252 ‚Äî BUG: row_lock used instead of row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

On PostgreSQL, `row_lock.try_get::<bool, _>(0)` always succeeds (PostgreSQL returns `BOOLEAN`), so the `Ok(r)` arm is taken and `row_type` is never consulted. The function returns the lock-script result for both the lock and type checks. On SQLite, `try_get::<bool, _>` fails (SQLite returns `BIGINT`), so the `Err` arm falls through to `row_type.get::<i64, _>(0)`, which is correct ‚Äî SQLite is unaffected.

`rollback_block` (lines 29‚Äì38) calls `script_exists_in_output` for each `type_script_id` and pushes the ID into `script_id_list_to_remove` when the function returns `false`. Because the function returns the lock-script result for the type-script check on PostgreSQL, any script used only as a type script (not simultaneously as a lock script in any surviving output) is incorrectly reported as absent and deleted from the `script` table by `remove_batch_by_blobs("script", "id", &script_id_list_to_remove, tx)` at line 39. Subsequent RPC queries that join on `script_id` return incomplete or missing results for any cell whose type script was incorrectly purged.

## Impact Explanation
This is a concrete implementation bug in CKB's rich-indexer state storage mechanism, causing silent, persistent data corruption on PostgreSQL. It maps to **Medium (2001‚Äì10000 points): Suboptimal implementation of CKB state storage mechanism**. The rich-indexer is CKB's supplementary indexing layer; its corruption causes incorrect RPC results (`get_cells`, `get_transactions`) for type-scripted cells (UDT, NFT, DAO, etc.) after any reorg, without any error being surfaced to the operator.

## Likelihood Explanation
Chain reorganizations are routine and externally triggerable by any peer relaying a competing chain of sufficient work ‚Äî no special privilege is required. Type scripts are ubiquitous on CKB. Any PostgreSQL-backed rich-indexer node will silently corrupt its database on every reorg that involves outputs with type scripts not simultaneously used as lock scripts in surviving outputs. The condition is common and repeatable.

## Recommendation
Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ‚Üê was row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

## Proof of Concept
1. Run a PostgreSQL-backed rich-indexer node.
2. Index a block containing an output with a type script (e.g., a UDT cell) whose `type_script_id` differs from any `lock_script_id` in surviving outputs.
3. Trigger a chain reorganization that rolls back that block (e.g., by feeding a competing chain of equal or greater work).
4. `rollback_block` calls `script_exists_in_output(type_script_id, tx)`. Because line 252 reads `row_lock` instead of `row_type`, the function returns the lock-script EXISTS result (`false` if the script is not a lock script in any surviving output).
5. The type script ID is pushed into `script_id_list_to_remove` and deleted from the `script` table.
6. Query `get_cells` or `get_transactions` for an address using that type script ‚Äî results are missing or incomplete, confirming database corruption.
