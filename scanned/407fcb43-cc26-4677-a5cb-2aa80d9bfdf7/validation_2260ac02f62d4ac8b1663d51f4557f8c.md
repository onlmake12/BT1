### Title
Silent u128-to-u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Capacity — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate u128 result (`withdraw_counted_capacity`) and then casts it to u64 with a bare `as u64` — a **silent truncating cast**. If the u128 value exceeds `u64::MAX`, the high bits are silently discarded, producing a wrong (smaller) capacity. Every other u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`, making this omission a clear inconsistency and a latent arithmetic correctness bug.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity:

```rust
// util/dao/src/lib.rs  lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The same file performs two other u128→u64 conversions with proper checked casts:

```rust
// line 244-245
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

The `as u64` cast at line 156 is the only place in the file that bypasses this pattern.

The truncated value propagates through two critical paths:

**Path 1 — `withdrawed_interests`:**

```rust
// lines 316-332
let maximum_withdraws = rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
    self.transaction_maximum_withdraw(rtx)          // calls calculate_maximum_withdraw
        .and_then(|c| capacities.safe_add(c).map_err(Into::into))
})?;
// ...
maximum_withdraws.safe_sub(input_capacities).map_err(Into::into)
``` [4](#0-3) 

If truncation makes `maximum_withdraws` smaller than `input_capacities`, `safe_sub` returns `DaoError::Overflow`, causing `dao_field_with_current_epoch` to fail.

**Path 2 — `transaction_fee`:**

```rust
// lines 30-36
pub fn transaction_fee(&self, rtx: &ResolvedTransaction) -> Result<Capacity, DaoError> {
    let maximum_withdraw = self.transaction_maximum_withdraw(rtx)?;
    rtx.transaction
        .outputs_capacity()
        .and_then(|y| maximum_withdraw.safe_sub(y))
        .map_err(Into::into)
}
``` [5](#0-4) 

A truncated `maximum_withdraw` that falls below `outputs_capacity` causes `safe_sub` to fail, making the fee calculation return an error for a valid DAO withdrawal transaction.

---

### Impact Explanation

`dao_field_with_current_epoch` is called during block verification to compute the DAO accumulator field. If it returns an error because `withdrawed_interests` fails due to the truncated capacity, the node rejects a **valid block** containing DAO withdrawal transactions. This is a consensus-level denial-of-service: a correctly constructed DAO withdrawal block would be permanently unacceptable to the affected node, causing it to diverge from the canonical chain.

Additionally, the RPC `calculate_dao_maximum_withdraw` (via `experiment.rs`) would silently return a wrong (smaller) withdrawal amount to users, misleading them about their actual entitlement. [6](#0-5) 

---

### Likelihood Explanation

The truncation fires when:

```
withdraw_counted_capacity = counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

- `counted_capacity` is bounded by the total CKB supply (~3.36 × 10¹⁸ shannons)
- `u64::MAX ≈ 1.844 × 10¹⁹`
- The ratio `withdrawing_ar / deposit_ar` must therefore exceed **~5.49×**

Given NervosDAO's secondary issuance rate (~4% per year compounded), reaching a 5.49× AR ratio from any deposit point would take on the order of 42 years. Under current network parameters this is not practically reachable. The likelihood is therefore **low**, but the bug is structurally real, inconsistent with the rest of the file, and requires only a one-line fix.

---

### Recommendation

Replace the silent cast with the same checked pattern used elsewhere in the file:

```rust
// Before (line 156):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After:
Capacity::shannons(
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
).safe_add(occupied_capacity)?
```

This makes overflow explicit and consistent with `miner_issuance128` and `ar_increase128` handling in `dao_field_with_current_epoch`. [7](#0-6) 

---

### Proof of Concept

1. Deposit a DAO cell with `output_capacity` close to the total CKB supply (e.g., `counted_capacity ≈ 3.36 × 10¹⁸` shannons).
2. Wait until `withdrawing_ar / deposit_ar > u64::MAX / counted_capacity ≈ 5.49`.
3. Construct a valid DAO withdrawal transaction (Phase 2) spending that cell.
4. Submit the block containing this transaction.
5. The node calls `calculate_maximum_withdraw`, computes `withdraw_counted_capacity > u64::MAX`, silently truncates it to a small value via `as u64`, and then `withdrawed_interests` calls `safe_sub(input_capacities)` where `input_capacities > truncated_maximum_withdraws`, returning `DaoError::Overflow`.
6. `dao_field_with_current_epoch` propagates the error; block verification fails; the node rejects the valid block and diverges from peers that correctly accept it.
