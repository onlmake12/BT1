### Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Eviction Values, Inflating Pool Size Accounting - (File: `tx-pool/src/component/pool_map.rs`)

### Summary

In `PoolMap::add_entry`, the new `total_tx_size` and `total_tx_cycles` are computed as local variables **before** `check_and_record_ancestors` is called. That call can evict existing pool entries (removing them via `remove_entry` → `update_stat_for_remove_tx`, which immediately decrements `self.total_tx_size`/`self.total_tx_cycles`). Afterwards, lines 218–219 unconditionally overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale pre-eviction snapshot, erasing the correct decrements and inflating the pool's size accounting.

### Finding Description

`add_entry` follows this sequence:

```rust
// Step 1: snapshot computed BEFORE evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//   = self.total_tx_size + tx_size  (stale local)

// Step 2: may remove evicted entries from pool,
//         calling remove_entry → update_stat_for_remove_tx,
//         which IMMEDIATELY decrements self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Step 3: unconditionally overwrite with the stale snapshot,
//         erasing the eviction decrements
self.total_tx_size = total_tx_size;    // should be post-eviction + tx_size
self.total_tx_cycles = total_tx_cycles; // should be post-eviction + cycles
``` [1](#0-0) 

`updated_stat_for_add_tx` computes the new totals as `self.total_tx_size + tx_size` and stores them in **local** variables without touching `self`. [2](#0-1) 

`update_stat_for_remove_tx` (called inside `remove_entry`) **immediately** decrements `self.total_tx_size` and `self.total_tx_cycles`. [3](#0-2) 

`remove_entry` calls `update_stat_for_remove_tx` as part of every removal. [4](#0-3) 

The eviction path is confirmed by the doc comment on `add_entry`: *"evicts: is the evicted transactions before inserting this TxEntry, Currently, evicts when inserting is only due to referring cell dep will be consumed by this new transaction."* The test-only `add_proposed` wrapper discards the returned evicts entirely, which is only safe if the evictions are already applied inside `check_and_record_ancestors`. [5](#0-4) 

**Net effect:** after `add_entry` returns, `self.total_tx_size` equals `old_total + tx_size` instead of the correct `old_total − Σ(evicted_sizes) + tx_size`. The evicted entries' sizes/cycles are silently added back into the pool totals.

### Impact Explanation

`limit_size` enforces the pool capacity by comparing `self.pool_map.total_tx_size` against `self.config.max_tx_pool_size`. [6](#0-5) 

With an inflated `total_tx_size`, the pool believes it is larger than it actually is. This causes `limit_size` to evict additional legitimate pending/proposed transactions that would otherwise fit, degrading tx-pool throughput and potentially causing valid user transactions to be dropped. The accounting error compounds with each eviction-triggering insertion.

### Likelihood Explanation

The trigger condition — a new transaction that consumes a cell dep already referenced by existing pool entries — is reachable by any unprivileged tx-pool submitter via the standard `send_transaction` RPC or P2P relay. No special privilege is required. A malicious actor can craft such a transaction deliberately to repeatedly trigger the accounting drift.

### Recommendation

Compute `total_tx_size` and `total_tx_cycles` **after** `check_and_record_ancestors` completes (so eviction decrements are already reflected in `self.total_tx_size`/`self.total_tx_cycles`), or re-read `self.total_tx_size`/`self.total_tx_cycles` at the point of the final assignment:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;

// Recompute AFTER evictions have updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

### Proof of Concept

1. Pool starts empty; `max_tx_pool_size = 1000`.
2. Submit `tx_A` (size=400) with `cell_dep_X`. Pool: `total_tx_size = 400`.
3. Submit `tx_B` (size=400) with `cell_dep_X`. Pool: `total_tx_size = 800`.
4. Submit `tx_C` (size=100) that **consumes** `cell_dep_X` as an input, triggering eviction of `tx_A` and `tx_B`.
   - `updated_stat_for_add_tx` snapshots `total_tx_size = 800 + 100 = 900` into a local.
   - `check_and_record_ancestors` removes `tx_A` and `tx_B`: `self.total_tx_size = 800 − 800 = 0`.
   - Lines 218–219 overwrite: `self.total_tx_size = 900` (stale).
5. Pool now reports `total_tx_size = 900` but actually contains only `tx_C` (size=100).
6. `limit_size` sees `900 > max_tx_pool_size` and begins evicting `tx_C` unnecessarily, or future submissions of size ≤ 100 are incorrectly rejected as "pool full."

    pub(crate) fn add_entry(
        &mut self,
        mut entry: TxEntry,
        status: Status,
    ) -> Result<(bool, HashSet<TxEntry>), Reject> {
        let tx_short_id = entry.proposal_short_id();
        let mut evicts = Default::default();
        if self.entries.get_by_id(&tx_short_id).is_some() {
            return Ok((false, evicts));
        }
        let (total_tx_size, total_tx_cycles) =
            self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
        trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
        evicts = self.check_and_record_ancestors(&mut entry)?;
        self.record_entry_edges(&entry)?;
        self.insert_entry(&entry, status);
        self.record_entry_descendants(&entry);
        self.track_entry_statics(None, Some(status));
        self.total_tx_size = total_tx_size;
        self.total_tx_cycles = total_tx_cycles;
        Ok((true, evicts))
    }

                self.total_tx_size, tx_size

                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
