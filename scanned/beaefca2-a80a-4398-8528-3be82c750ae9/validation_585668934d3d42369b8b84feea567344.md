### Title
`CapacityVerifier::verify` Skips `OutputsSumOverflow` for Entire Transaction When Any Input Uses DAO Type Script, Enabling Capacity Inflation — (`verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::verify` unconditionally skips the `OutputsSumOverflow` check for any transaction that contains **at least one** DAO-type-script input. The DAO type script only validates the DAO-specific cells' withdrawal amount; it does not enforce the overall transaction capacity balance. Any CKB holder who has deposited into the Nervos DAO can exploit this to produce non-DAO outputs whose total capacity exceeds the non-DAO inputs, creating CKB out of thin air.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `CapacityVerifier::verify` guards the `OutputsSumOverflow` check with:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    let inputs_sum  = self.resolved_transaction.inputs_capacity()?;
    let outputs_sum = self.resolved_transaction.outputs_capacity()?;
    if inputs_sum < outputs_sum {
        return Err(TransactionError::OutputsSumOverflow { ... }.into());
    }
}
``` [1](#0-0) 

`valid_dao_withdraw_transaction` returns `true` if **any** resolved input carries the DAO type script:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
``` [2](#0-1) 

The code comment states: *"DAO withdraw transaction is verified via the type script of DAO cells."* The DAO type script runs as a type script on the DAO cells and enforces that the DAO output capacity equals the deposit plus accrued interest. It does **not** constrain non-DAO outputs in the same transaction. The only remaining check after the early skip is the per-output `InsufficientCellCapacity` guard, which only verifies that each individual output's `capacity` field is large enough to store its own data — it does not enforce the global `inputs_sum >= outputs_sum` invariant. [3](#0-2) 

This is structurally identical to the ERC1155Fuse bug: a condition check (`oldOwner == to` / `valid_dao_withdraw_transaction()`) causes a critical validation to be silently skipped, allowing an unauthorized state change to pass without error.

---

### Impact Explanation

An attacker who owns a DAO deposit cell can construct a transaction such as:

| Side | Cell | Capacity |
|---|---|---|
| Input | DAO cell (attacker's deposit) | 100 CKB |
| Input | Regular cell (attacker's) | 10 CKB |
| Output | DAO withdrawal cell | 105 CKB (deposit + interest, enforced by DAO type script) |
| Output | Regular cell (attacker's) | **1 000 CKB** (unchecked) |

Total inputs: 110 CKB. Total outputs: 1 105 CKB. The `OutputsSumOverflow` check is skipped because the DAO input is present. The DAO type script validates only the 105 CKB DAO output. The 1 000 CKB regular output is validated only for per-cell minimum occupancy. The transaction passes all verifiers, and the attacker receives 895 CKB that did not exist before — a direct inflation of the CKB supply.

---

### Likelihood Explanation

Any CKB holder who has ever deposited into the Nervos DAO (a common, encouraged action) can trigger this. No privileged access, leaked keys, or majority hashpower is required. The attacker submits a single crafted transaction through the normal RPC/tx-pool path. The entry point is fully reachable by an unprivileged tx-pool submitter.

---

### Recommendation

Replace the blanket skip with a check that only excludes the **DAO-specific portion** of the capacity delta from the overflow check, or verify that the DAO type script is actually responsible for enforcing the full transaction capacity balance (and document that invariant explicitly). At minimum, the `OutputsSumOverflow` guard should still run for the non-DAO inputs and outputs even when a DAO withdrawal is present:

```rust
// Pseudocode: compute non-DAO input/output sums separately
let dao_input_sum  = /* sum of DAO inputs */;
let dao_output_sum = /* sum of DAO outputs */;
let non_dao_inputs_sum  = inputs_sum  - dao_input_sum;
let non_dao_outputs_sum = outputs_sum - dao_output_sum;
if non_dao_inputs_sum < non_dao_outputs_sum {
    return Err(TransactionError::OutputsSumOverflow { ... }.into());
}
```

### Proof of Concept

1. Deposit 100 CKB into the Nervos DAO, obtaining a DAO cell.
2. Construct a withdrawal transaction:
   - **Input 0**: the DAO cell (100 CKB, DAO type script present → `valid_dao_withdraw_transaction()` returns `true`)
   - **Input 1**: any regular cell owned by the attacker (10 CKB)
   - **Output 0**: DAO withdrawal cell (105 CKB — passes DAO type script check)
   - **Output 1**: regular cell owned by the attacker (1 000 CKB — only checked for per-cell minimum occupancy)
3. Submit via RPC.
4. `CapacityVerifier::verify` enters the `if !(... || valid_dao_withdraw_transaction())` branch as `false`, skipping the `OutputsSumOverflow` check entirely.
5. The per-output loop passes because each output individually has sufficient capacity for its data.
6. The transaction is accepted; the attacker's regular cell now holds 1 000 CKB, 890 CKB of which were created from nothing. [4](#0-3) [5](#0-4)

fn cell_uses_dao_type_script(cell_output: &CellOutput, dao_type_hash: &Byte32) -> bool {
    cell_output
        .type_()
        .to_opt()
        .map(|t| {
            Into::<u8>::into(t.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
                && &t.code_hash() == dao_type_hash
        })
        .unwrap_or(false)
}
