### Title
`reject_ill_transactions` Defaults to `false` via `#[serde(default)]`, Allowing Known-Buggy Scripts to Enter the Tx-Pool — (File: `util/app-config/src/configs/rpc.rs`)

---

### Summary

The `reject_ill_transactions` field in CKB's RPC configuration struct uses `#[serde(default)]`, which resolves to Rust's `bool` default of `false`. Any node operator who creates a custom `ckb.toml` without explicitly setting this field will silently run with the protective check disabled. An unprivileged RPC caller can then submit transactions whose output cells contain scripts with known VM bugs, bypassing the `IllTransactionChecker` gate entirely and landing those cells on-chain.

---

### Finding Description

In `util/app-config/src/configs/rpc.rs`, the `Config` struct declares:

```rust
/// Rejects txs with scripts that might trigger known bugs
#[serde(default)]
pub reject_ill_transactions: bool,
```

`#[serde(default)]` on a bare `bool` field resolves to `false` when the key is absent from the TOML file. The bundled template `resource/ckb.toml` does set `reject_ill_transactions = true`, but that template is only a starting point. Any operator who writes a minimal or custom config omitting this key will have the field silently deserialize to `false`.

The RPC pool module exposes a dedicated error code `PoolRejectedTransactionByIllTransactionChecker = -1103` and the integration test `SendDefectedBinary` explicitly exercises both states (`reject_ill_transactions: true` and `reject_ill_transactions: false`), confirming that when the flag is `false` a transaction carrying a `defected_binary` payload is accepted by `send_transaction` and subsequently committed to the chain.

The protective check is therefore gated entirely on this flag. When the flag is `false` (the struct-level default), the gate is open and any RPC caller can submit transactions with output cells containing scripts that trigger known CKB-VM bugs.

---

### Impact Explanation

Scripts with known VM bugs can be deployed on-chain through any node whose operator did not explicitly opt in to the protection. Once such a cell is live on-chain, any transaction that references it as a dep or input will execute the buggy script path. The CKB changelog entry for this feature states: "those bugs, especially VM bugs might lead to surprising unexpected behaviors." Because the consensus layer does not reject these transactions, the buggy cells persist permanently until a hardfork removes them.

---

### Likelihood Explanation

The default config template sets the flag to `true`, but the Rust struct default is `false`. Operators who generate configs programmatically, copy a minimal config, or upgrade from an older version that lacked this field will silently run with the check disabled. The entry point is the standard `send_transaction` JSON-RPC method, reachable by any local or network-accessible RPC caller with no authentication required beyond network access to port 8114.

---

### Recommendation

Change the struct-level default to `true` by replacing `#[serde(default)]` with a named default function:

```rust
fn default_reject_ill_transactions() -> bool { true }

#[serde(default = "default_reject_ill_transactions")]
pub reject_ill_transactions: bool,
```

This ensures that any config file missing the key is treated as if `reject_ill_transactions = true` were present, matching the intent expressed in the bundled template and the feature's design goal.

---

### Proof of Concept

1. Create a `ckb.toml` that omits the `reject_ill_transactions` key under `[rpc]`.
2. Start `ckb run`. The field deserializes to `false`.
3. Call `send_transaction` via JSON-RPC with a transaction whose output cell data is the `defected_binary` test payload (or any script matching a known-bug pattern).
4. The `IllTransactionChecker` gate is skipped; the transaction enters the tx-pool.
5. Mine blocks; the transaction is committed. The integration test `SendDefectedBinary::new("send_defected_binary_do_not_reject_known_bugs", false)` in `test/src/specs/tx_pool/send_defected_binary.rs` asserts `is_transaction_committed(node, &tx)` — confirming the cell lands on-chain.
