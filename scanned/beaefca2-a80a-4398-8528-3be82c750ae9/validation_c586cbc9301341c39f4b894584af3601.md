### Title
Silent Truncating Cast in `calculate_maximum_withdraw` Produces Wrong Withdrawal Amounts for Large DAO Cells - (File: util/dao/src/lib.rs)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` uses a silent truncating `as u64` cast on a `u128` intermediate result instead of a checked conversion. For DAO cells holding large capacities after a sufficient deposit period, the intermediate value `withdraw_counted_capacity` can exceed `u64::MAX`, causing the high bits to be silently discarded. This produces a drastically wrong (too-small) withdrawal amount, which in turn causes valid DAO withdrawal transactions to be rejected by the fee verifier, or causes the `calculate_dao_maximum_withdraw` RPC to return a wrong value that misleads users into constructing transactions that forfeit interest.

---

### Finding Description

In `calculate_maximum_withdraw`:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The cast `withdraw_counted_capacity as u64` is a **silent truncating cast**. If `withdraw_counted_capacity > u64::MAX`, the upper bits are discarded without any error, producing a value that is `withdraw_counted_capacity mod 2^64` — potentially orders of magnitude smaller than the correct result.

Every other analogous computation in the same file uses the correct pattern `u64::try_from(...).map_err(|_| DaoError::Overflow)?`:

```rust
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

```rust
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [4](#0-3) 

The `calculate_maximum_withdraw` function is the sole exception that uses the unsafe cast.

**Overflow condition:**

`withdraw_counted_capacity = counted_capacity × withdrawing_ar / deposit_ar`

- `deposit_ar` starts at `DEFAULT_GENESIS_ACCUMULATE_RATE = 10^16` and only increases.
- `withdrawing_ar > deposit_ar` for any non-zero deposit period.
- `counted_capacity` can be up to the total CKB supply in shannons (~3.36 × 10^18).

For overflow: `counted_capacity × withdrawing_ar > u64::MAX × deposit_ar ≈ 1.844 × 10^35`.

With `counted_capacity = 3.36 × 10^18` and `deposit_ar = 10^16`, overflow occurs when `withdrawing_ar > 5.49 × 10^16`. The accumulate rate grows by approximately `ar × g2 / c ≈ 2.22 × 10^11` per block. Reaching the overflow threshold requires roughly **200,000 blocks (~18.5 days)** for a cell holding the maximum practical capacity. Smaller cells require proportionally longer deposit periods but the condition remains reachable within the protocol's lifetime. [5](#0-4) 

---

### Impact Explanation

**Path 1 — Transaction rejection (tx-pool / block verification):**

`transaction_fee` calls `transaction_maximum_withdraw` → `calculate_maximum_withdraw`. If the returned `maximum_withdraw` is silently truncated to a small value, then:

```rust
maximum_withdraw.safe_sub(outputs_capacity)
``` [6](#0-5) 

underflows and returns `DaoError::Overflow`, causing the transaction to be rejected. A valid DAO withdrawal transaction is permanently unprocessable — the user's funds are locked.

**Path 2 — Wrong RPC result:**

`calculate_dao_maximum_withdraw` calls `calculate_maximum_withdraw` directly:

```rust
match calculator.calculate_maximum_withdraw(
    &output,
    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
    &deposit_header_hash,
    &withdrawing_header_hash.into(),
) {
    Ok(capacity) => Ok(capacity.into()),
``` [7](#0-6) 

The RPC returns a silently wrong (too-small) capacity. A user relying on this value to construct their withdrawal transaction would set their output capacity too low, forfeiting accrued interest.

**Path 3 — Block assembly:**

`calc_dao` in the block assembler calls `dao_field_with_current_epoch`, which calls `transaction_maximum_withdraw` → `calculate_maximum_withdraw` for DAO inputs: [8](#0-7) 

A wrong fee calculation can cause valid DAO withdrawal transactions to be silently dropped from block templates.

---

### Likelihood Explanation

- The overflow requires a large-capacity DAO cell and a sufficient deposit period. Both conditions are realistic for institutional or long-term depositors.
- The `calculate_dao_maximum_withdraw` RPC is a public endpoint callable by any RPC user, making Path 2 reachable without any special privilege.
- The condition is deterministic and reproducible: any cell exceeding the threshold capacity deposited for the threshold duration will trigger it.
- No malicious miner, Sybil attack, or key compromise is required.

---

### Recommendation

Replace the silent truncating cast with a checked conversion, consistent with every other arithmetic result in the same file:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

This makes the function return `DaoError::Overflow` instead of silently producing a wrong result, consistent with the existing error-handling pattern throughout `DaoCalculator`.

---

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` demonstrates that the function can fail for large capacities, but it only catches the `safe_add` overflow path (when the truncated value happens to still be large). The silent truncation path — where the truncated value is small enough that `safe_add` succeeds but the result is wrong — is not tested.

Concrete trigger values:
- `deposit_ar = 10_000_000_000_000_000` (genesis value)
- `withdrawing_ar = 55_000_000_000_000_000` (reachable after ~200,000 blocks)
- `counted_capacity = 3_360_000_000_000_000_000` (near total supply)
- `withdraw_counted_capacity = 3_360_000_000_000_000_000 × 55_000_000_000_000_000 / 10_000_000_000_000_000 = 18_480_000_000_000_000_000`
- `18_480_000_000_000_000_000 > u64::MAX (18_446_744_073_709_551_615)` → truncates to `33_255_926_290_448_384`

The returned withdrawal capacity would be ~33 quadrillion shannons instead of ~18.48 quintillion shannons — a factor of ~556× too small — causing `transaction_fee` to underflow and reject the withdrawal. [9](#0-8) [10](#0-9)

    /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }

pub fn extract_dao_data(dao: Byte32) -> (u64, Capacity, Capacity, Capacity) {
    let data = dao.raw_data();
    let c = Capacity::shannons(LittleEndian::read_u64(&data[0..8]));
    let ar = LittleEndian::read_u64(&data[8..16]);
    let s = Capacity::shannons(LittleEndian::read_u64(&data[16..24]));
    let u = Capacity::shannons(LittleEndian::read_u64(&data[24..32]));
    (ar, c, s, u)
}
