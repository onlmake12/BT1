### Title
Targeted GCS Filter False Positive Attack via Fixed SipHash Seed (0, 0) — (`util/types/src/utilities/block_filter.rs`)

---

### Summary

`build_gcs_filter` hardcodes `SipHasher24Builder::new(0, 0)`, making the GCS filter's hash function fully public and constant across all blocks. An unprivileged attacker can precompute lock/type script hashes that deterministically collide with any target address's GCS bucket, then deploy transactions containing those hashes to force targeted false positives against light clients monitoring that address.

---

### Finding Description

The sole call site for GCS filter construction is:

```rust
fn build_gcs_filter(out: &mut dyn Write) -> GCSFilterWriter<'_, SipHasher24Builder> {
    GCSFilterWriter::new(out, SipHasher24Builder::new(0, 0), M, P)
}
``` [1](#0-0) 

This is called unconditionally for every block:

```rust
let mut filter = build_gcs_filter(&mut filter_writer);
``` [2](#0-1) 

The filter adds `calc_lock_hash()` and `calc_script_hash()` for every input cell and output cell in every non-coinbase transaction: [3](#0-2) 

The GCS filter maps each element `e` to a bucket value `SipHash(k0=0, k1=0, e) mod (N * M)`, where `N` is the number of elements and `M` is the false-positive denominator (imported from the `golomb-coded-set` crate, version 0.2.1, which uses `M = 784931` per BIP-158). [4](#0-3) 

Because `k0 = k1 = 0` is fixed and public, the mapping from any 32-byte script hash to its GCS bucket is fully deterministic and precomputable by anyone, with no per-block randomness.

**Contrast with BIP-158:** Bitcoin's compact block filters derive the SipHash key from the block hash (`k0 = block_hash[0..8]`, `k1 = block_hash[8..16]`). Even though the block hash is public after mining, it is unknown before the block is assembled, creating a circular dependency that prevents pre-mining collision crafting. CKB's fixed `(0, 0)` seed removes this protection entirely.

---

### Impact Explanation

An attacker who knows a target light client's monitored address `A` can:

1. Compute `target_bucket = SipHash(0, 0, A) mod (N_B * M)` for a block `B` they intend to submit a transaction to (N_B is estimable from mempool/block template).
2. Iterate over candidate lock script args until finding a script hash `S` such that `SipHash(0, 0, S) mod (N_B * M) == target_bucket`. With `M ≈ 784,931`, expected search cost is ~M SipHash evaluations — microseconds on commodity hardware.
3. Submit a transaction to block `B` with an output whose lock script hashes to `S`.
4. When block `B` is mined, its GCS filter contains a bucket matching `A`'s query.
5. Every light client monitoring address `A` receives a false positive for block `B` and downloads the full block unnecessarily.

By repeating this for every block (one crafted transaction per block), the attacker can force light clients monitoring `A` to download every block in a sustained range, amplifying their bandwidth and CPU cost by the false positive rate. The attack is targeted (specific address), repeatable, and requires only standard transaction submission — no privileged access.

---

### Likelihood Explanation

- **Entry point:** Standard transaction submission (unprivileged, production path).
- **Computational cost to attacker:** ~784,931 SipHash-2-4 evaluations per false positive — trivially fast.
- **Economic cost:** One transaction fee per false positive induced. On CKB, fees are low.
- **Detectability:** The crafted script hashes look like ordinary lock scripts; no consensus rule rejects them.
- **Prerequisite knowledge:** Only the target address (public) and the approximate block element count (estimable from block template RPC).

Likelihood is **medium-high** for a motivated attacker targeting specific high-value light client wallets.

---

### Recommendation

Derive the SipHash key from the block hash, matching BIP-158 semantics:

```rust
fn build_gcs_filter<'a>(out: &'a mut dyn Write, block_hash: &[u8; 32]) -> GCSFilterWriter<'a, SipHasher24Builder> {
    let k0 = u64::from_le_bytes(block_hash[0..8].try_into().unwrap());
    let k1 = u64::from_le_bytes(block_hash[8..16].try_into().unwrap());
    GCSFilterWriter::new(out, SipHasher24Builder::new(k0, k1), M, P)
}
```

This makes the hash function unpredictable to anyone who does not yet know the block hash, breaking the precomputation step. The block hash must be passed through `build_filter_data` and `build_filter_data_for_block`. [5](#0-4)

### Proof of Concept

```rust
use siphasher::sip::SipHasher24;
use std::hash::Hasher;

const M: u64 = 784_931;

fn gcs_bucket(k0: u64, k1: u64, data: &[u8], n: u64) -> u64 {
    let mut h = SipHasher24::new_with_keys(k0, k1);
    h.write(data);
    // GCS maps to [0, N*M)
    let hash = h.finish();
    // full 64-bit reduction as golomb-coded-set does
    ((hash as u128 * (n as u128 * M as u128)) >> 64) as u64
}

fn main() {
    // Target: address A the light client monitors
    let target_address_hash: [u8; 32] = /* victim's lock script hash */ [0xAA; 32];
    let n: u64 = 10; // estimated elements in target block

    let target_bucket = gcs_bucket(0, 0, &target_address_hash, n);

    // Find a colliding script hash S
    for i in 0u64.. {
        let mut candidate = [0u8; 32];
        candidate[..8].copy_from_slice(&i.to_le_bytes());
        if gcs_bucket(0, 0, &candidate, n) == target_bucket {
            println!("Collision found: {:?}", candidate);
            // Deploy a transaction whose lock script hashes to `candidate`
            break;
        }
    }
}
```

Expected iterations to first collision: ~M ≈ 784,931. At ~1 billion SipHash ops/sec, this completes in under 1 millisecond. The resulting script hash, when included in any block as a lock script, causes every light client querying for `target_address_hash` to receive a false positive for that block. [1](#0-0)
