### Title
`CapacityVerifier` Skips Total Capacity Conservation for Non-DAO Cells in Mixed DAO Transactions — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::verify()` unconditionally skips the `OutputsSumOverflow` check for the **entire transaction** whenever any single input uses the DAO type script. The on-chain DAO type script only validates DAO cells. Non-DAO cells in the same transaction are left without any consensus-level capacity conservation check. An attacker who controls a DAO cell can include non-DAO cells in the same withdrawal transaction and create non-DAO outputs with more capacity than the non-DAO inputs provide, minting CKB out of thin air.

---

### Finding Description

`CapacityVerifier::verify()` contains the following guard:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    let inputs_sum = self.resolved_transaction.inputs_capacity()?;
    let outputs_sum = self.resolved_transaction.outputs_capacity()?;
    if inputs_sum < outputs_sum {
        return Err(TransactionError::OutputsSumOverflow { ... }.into());
    }
}
``` [1](#0-0) 

The predicate `valid_dao_withdraw_transaction()` is:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
``` [2](#0-1) 

This returns `true` if **any** input carries the DAO type script. The moment that condition is met, the entire `OutputsSumOverflow` check is skipped for **all** cells in the transaction — including non-DAO cells.

The code comment justifies the skip with:

> *"DAO withdraw transaction is verified via the type script of DAO cells"* [3](#0-2) 

This is correct for DAO cells: the on-chain DAO RISC-V type script enforces that each DAO output capacity equals `input_capacity × (withdrawing_ar / deposit_ar)`. However, the DAO type script only runs for cells that carry it as their type script. It does not inspect or constrain non-DAO cells in the same transaction. No other verifier fills this gap.

`CapacityVerifier` is part of `ContextualTransactionVerifier`, which is invoked during both tx-pool admission (`tx-pool/src/util.rs`) and consensus block verification (`verification/contextual/src/contextual_block_verifier.rs`). The bypass therefore affects consensus validation, not just mempool policy. [4](#0-3) 

The analog to the OmnichainStaking report is direct:

| OmnichainStaking | CKB |
|---|---|
| Power = f(amount, duration) — a derived aggregate value | Presence of any DAO input — a coarse boolean derived from the input set |
| Power from NFT A authorizes unstaking NFT B (different underlying value) | DAO cell's presence bypasses capacity check for non-DAO cells (different underlying value) |
| Attacker substitutes a cheaper NFT to claim a more valuable one | Attacker uses a DAO cell to exempt non-DAO cells from the conservation invariant |

---

### Impact Explanation

An attacker can mint CKB capacity out of thin air:

1. Attacker deposits a small amount into the DAO (creates a DAO input cell, capacity `X`).
2. Attacker also controls a non-DAO cell with capacity `Y`.
3. Attacker constructs a DAO phase-2 withdrawal transaction:
   - **Input 1**: DAO cell (capacity `X`) → triggers `valid_dao_withdraw_transaction() = true`
   - **Input 2**: non-DAO cell (capacity `Y`)
   - **Output 1**: DAO cell (capacity `X + interest`) → validated correctly by the DAO type script
   - **Output 2**: non-DAO cell (capacity `Y + Z`, where `Z > 0`) → validated by **no one**
4. `CapacityVerifier` skips the overflow check because Input 1 is a DAO cell.
5. The DAO type script validates Output 1 and ignores Output 2.
6. `Z` CKB is created out of thin air, violating CKB's fixed-supply invariant.

This constitutes direct inflation of the CKB supply, which is equivalent to theft from all existing CKB holders. A sufficiently large `Z` could drain the entire DAO interest pool or destabilize the network's economic model.

---

### Likelihood Explanation

- Any unprivileged user can open a DAO deposit (no special role required).
- The attack requires only a standard DAO withdrawal transaction with an additional non-DAO input — a transaction structure that is syntactically valid and passes all non-contextual checks.
- The attacker controls the transaction entirely and can set `Z` to any value up to the block capacity limit.
- No Sybil attack, hashpower majority, or social engineering is required.

---

### Recommendation

The `OutputsSumOverflow` check should not be skipped wholesale for the entire transaction. Instead, the verifier should:

1. Compute `inputs_sum` and `outputs_sum` for **non-DAO cells only**.
2. Enforce `non_dao_inputs_sum >= non_dao_outputs_sum` unconditionally.
3. Continue to delegate DAO cell capacity verification to the DAO type script.

Alternatively, the check can remain transaction-wide but must account for the DAO interest by adding the expected maximum DAO withdrawal to the inputs side before comparing, rather than skipping the check entirely.

        .capacity(capacity_bytes!(50))           // 50 CKB non-DAO cell
        .build(),
    ..

    .capacity(capacity_bytes!(100) + interest)  // validated by DAO type script
    .type_(Some(dao_type_script).pack())
    .build();

    .capacity(capacity_bytes!(200))             // 150 CKB more than non_dao_input — UNCHECKED
    .build();

// valid_dao_withdraw_transaction() == true  →  OutputsSumOverflow check skipped
// DAO type script validates dao_output only
// non_dao_output capacity overflow is never caught at consensus level
// Net result: 150 CKB minted out of thin air
