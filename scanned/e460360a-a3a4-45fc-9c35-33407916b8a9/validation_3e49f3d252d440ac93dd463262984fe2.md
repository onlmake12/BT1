### Title
Unbounded `OrphanBlockPool::insert()` Allows Memory and Disk Exhaustion via Orphan Block Spam — (File: `chain/src/utils/orphan_block_pool.rs`)

---

### Summary

`OrphanBlockPool::insert()` imposes no size limit on the in-memory orphan block pool. Any peer that can produce valid-PoW block headers (i.e., any miner, regardless of hashrate fraction) can continuously push blocks with unknown parents into the pool, causing unbounded memory growth and unbounded RocksDB disk writes until the node crashes or becomes unresponsive.

---

### Finding Description

`InnerPool::insert()` unconditionally inserts a `LonelyBlockHash` into the internal `HashMap` structures with no count or byte-size check:

```rust
fn insert(&mut self, lonely_block: LonelyBlockHash) {
    let hash = lonely_block.hash();
    let parent_hash = lonely_block.parent_hash();
    self.blocks
        .entry(parent_hash.clone())
        .or_default()
        .insert(hash.clone(), lonely_block);
    ...
    self.parents.insert(hash, parent_hash);
}
``` [1](#0-0) 

The public `OrphanBlockPool::insert()` wrapper simply acquires the write lock and delegates, with no limit enforcement:

```rust
pub fn insert(&self, lonely_block: LonelyBlockHash) {
    self.inner.write().insert(lonely_block);
}
``` [2](#0-1) 

The `with_capacity(capacity)` constructor only sets the initial `HashMap` allocation hint — it is **not** a hard cap: [3](#0-2) 

The only eviction mechanism is epoch-based expiry (`EXPIRED_EPOCH = 6`, approximately 24 hours), triggered by a 60-second timer in `ChainService::start_process_block`: [4](#0-3) 

The expiry check only removes blocks whose epoch is more than 6 epochs behind the current tip: [5](#0-4) 

**Contrast with the tx-pool orphan pool**, which has a hard count limit (`DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100`) and calls `limit_size()` after every insertion, evicting entries immediately when the cap is exceeded: [6](#0-5) [7](#0-6) 

The block orphan pool has no equivalent protection.

---

### Impact Explanation

Each block that enters the orphan pool also triggers `insert_block()` in `ChainService::asynchronous_process_block`, which writes the full block body to RocksDB before the orphan pool insertion: [8](#0-7) [9](#0-8) 

A sustained attack therefore causes:
- **Unbounded in-memory growth** of `InnerPool.blocks`, `InnerPool.parents`, and `InnerPool.leaders` HashMaps.
- **Unbounded RocksDB disk writes** — every orphan block body is persisted before the orphan pool check.
- **Node crash or OOM kill** once system memory is exhausted.
- **Disk exhaustion** on nodes with limited storage.

The node becomes unable to process legitimate blocks or serve RPC requests, constituting a complete denial of service.

---

### Likelihood Explanation

The attack entry path is the standard P2P Sync protocol (`SupportProtocols::Sync`, `SendBlock` message), handled by `BlockProcess::execute()`: [10](#0-9) 

`asynchronous_process_remote_block` only accepts a block body if the corresponding header already has `HEADER_VALID` status: [11](#0-10) 

This means the attacker must first send valid headers (which requires solving PoW via Eaglesong). However, this does **not** require 51% hashrate — any miner with any fraction of hashrate can produce valid blocks at a rate proportional to their hashrate. The attacker mines a chain of blocks whose root parent is unknown to the victim (e.g., a fork from a very old or fabricated ancestor), sends the headers to get `HEADER_VALID` status, then sends the block bodies. All blocks pass `non_contextual_verify` (structural checks, PoW validity) and are inserted into the orphan pool without limit.

The 6-epoch expiry window (~24 hours) means an attacker can accumulate a large pool before any cleanup occurs. A miner with even 1% of network hashrate produces roughly one block every ~13 minutes, yielding thousands of orphan entries per day.

---

### Recommendation

Add a hard size limit to `OrphanBlockPool`, mirroring the tx-pool orphan pool design:

1. Store a `max_capacity: usize` in `InnerPool` (set at construction time from a configurable parameter).
2. After each `insert()`, if `self.parents.len() > max_capacity`, evict the oldest or a random entry (similar to `OrphanPool::limit_size()` in `tx-pool/src/component/orphan.rs`).
3. Return a boolean or `Result` from `insert()` to allow the caller to ban or penalize the sending peer when the pool is full and the new block is evicted.

---

### Proof of Concept

**Attacker steps:**

1. Connect to a victim CKB node as a P2P peer via the Sync protocol.
2. Mine a sequence of N valid-PoW blocks (`B1, B2, ..., BN`) where `B1.parent_hash` is a fabricated hash unknown to the victim.
3. Send `SendHeaders(B1..BN)` to the victim. The victim validates PoW and sets `HEADER_VALID` for each.
4. The victim sends `GetBlocks` requesting the block bodies.
5. Send `SendBlock(Bi)` for each block. Each block:
   - Passes `non_contextual_verify` (valid structure and PoW).
   - Is written to RocksDB via `insert_block()`.
   - Enters `OrphanBroker::process_lonely_block()` → `self.orphan_blocks_broker.insert(lonely_block)` with no size check.
6. Repeat from step 2 with a new fabricated root parent hash.
7. The victim's `OrphanBlockPool` and RocksDB grow without bound until OOM or disk exhaustion.

The 60-second cleanup timer only removes blocks from epochs 6+ epochs old, providing no protection during an active attack within the current epoch window.

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }

    pub fn asynchronous_process_remote_block(&self, remote_block: RemoteBlock) {
        let block_hash = remote_block.block.hash();
        let status = self.shared.active_chain().get_block_status(&block_hash);
        // NOTE: Filtering `BLOCK_STORED` but not `BLOCK_RECEIVED`, is for avoiding
        // stopping synchronization even when orphan_pool maintains dirty items by bugs.
        if status.contains(BlockStatus::BLOCK_STORED) {
            error!("Block {} already stored", block_hash);
        } else if status.contains(BlockStatus::HEADER_VALID) {
            self.shared.accept_remote_block(&self.chain, remote_block);
        } else {
            debug!(
                "Synchronizer process_new_block unexpected status {:?} {}",
                status, block_hash,
            );
            // TODO which error should we return?
        }
    }
