### Title
Missing Per-Peer Rate Limiter in `LightClientProtocol` Enables Amplified DB and MMR Load via `GetBlocksProof` — (`util/light-client-protocol-server/src/lib.rs`, `src/components/get_blocks_proof.rs`)

### Summary
`LightClientProtocol` has no rate limiter, unlike `Relayer` (30 req/s keyed by peer+message) and `HolePunching` (30 req/s keyed by session+message). An unprivileged remote peer can send a continuous stream of `GetBlocksProof` messages each containing exactly 1000 unique on-chain block hashes and a valid `last_hash`, forcing the server to perform up to 3000 DB reads plus an MMR proof generation over 1000 positions per message, with no throttling.

### Finding Description
`LightClientProtocol::received` parses the message and calls `try_process`, which dispatches to `GetBlocksProofProcess::execute`. That function:

1. Accepts up to `GET_BLOCKS_PROOF_LIMIT = 1000` block hashes per message.
2. For each hash found on the main chain, performs `snapshot.get_block_header`, `snapshot.get_block_uncles`, and `snapshot.get_block_extension` — three synchronous DB reads per hash.
3. Passes all positions to `reply_proof`, which calls `mmr.gen_proof(items_positions)` — an O(N log N) MMR computation over up to 1000 positions.

No per-peer, per-message, or global rate limiter exists anywhere in the light client protocol handler. The contrast with `Relayer` and `HolePunching` — both of which explicitly construct a `governor::RateLimiter` at 30 req/s — confirms this is an unintentional omission, not a design choice.

### Impact Explanation
A single attacker peer can saturate the node's RocksDB read I/O and async executor threads with max-cost requests at line rate. With multiple peers (each allowed by the P2P connection manager), the amplification multiplies. The DB snapshot reads and MMR proof generation compete directly with the chain processing loop's own DB access and block validation work, degrading block validation throughput. The light client protocol is a production feature enabled on full nodes that serve light clients.

### Likelihood Explanation
The exploit requires only a valid P2P connection and knowledge of 1000 on-chain block hashes (trivially obtained from any block explorer or by syncing). No PoW, no key, no privileged role. The attacker sends well-formed, protocol-valid messages at maximum rate. The server has no mechanism to detect or throttle this.

### Recommendation
Add a per-peer rate limiter to `LightClientProtocol`, mirroring the pattern already used in `Relayer` and `HolePunching`:

```rust
// In LightClientProtocol struct:
rate_limiter: RateLimiter<(PeerIndex, u32)>,

// In LightClientProtocol::new:
let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
let rate_limiter = RateLimiter::hashmap(quota);

// At the top of try_process:
if self.rate_limiter.check_key(&(peer_index, message.item_id())).is_err() {
    return StatusCode::TooManyRequests.into();
}
```

Additionally, consider a tighter per-message limit for `GetBlocksProof` specifically (e.g., 100 hashes) given its high per-item DB cost.

### Proof of Concept
1. Connect to a CKB full node with the light client protocol enabled.
2. Obtain 1000 valid on-chain block hashes and the current tip hash.
3. In a tight loop, send `GetBlocksProof` messages each containing all 1000 hashes and the valid `last_hash`.
4. Measure: the server performs 3000 DB reads + `mmr.gen_proof(1000)` per iteration with no throttle.
5. Assert: concurrent block validation throughput (measured via `ckb_chain_process_block_duration` metrics) degrades below baseline under sustained load from a single peer.

        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
