### Title
Deterministic Peer Eviction in Bootnode Mode Enables Biased Connection Survival - (File: `network/src/network.rs`)

### Summary
In bootnode mode, `EventHandler::inbound_eviction` selects which inbound peers to disconnect using a deterministic index-parity pattern (`index & 0x1 != 0`) over `HashMap::values()` iteration order, rather than using a cryptographically random selection. This is directly analogous to the reported vulnerability: a value that should be random is replaced by a predictable, structured substitute, making the outcome manipulable by an attacker who can influence or observe the selection state.

### Finding Description
When a CKB node runs in bootnode mode and its inbound connection count approaches capacity (within 10 of `max_inbound`), every new `SessionOpen` event triggers `inbound_eviction`:

```rust
// network/src/network.rs, line 580-604
fn inbound_eviction(&self) -> Vec<PeerIndex> {
    if self.network_state.config.bootnode_mode {
        let status = self.network_state.connection_status();
        if status.max_inbound <= status.non_whitelist_inbound.saturating_add(10) {
            self.network_state
                .with_peer_registry(|registry| {
                    registry
                        .peers()
                        .values()
                        .filter(|peer| peer.is_inbound() && !peer.is_whitelist)
                        .map(|peer| peer.session_id)
                        .collect::<Vec<SessionId>>()
                })
                .into_iter()
                .enumerate()
                .filter_map(|(index, peer)| if index & 0x1 != 0 { Some(peer) } else { None })
                .collect()
        } else {
            Vec::new()
        }
    } else {
        Vec::new()
    }
}
``` [1](#0-0) 

The function collects all non-whitelist inbound peers from `registry.peers().values()` — a `HashMap` iterator — and evicts every peer at an odd index. The iteration order of `HashMap::values()` is determined by the HashMap's internal hash state. While Rust's `RandomState` seeds the hasher randomly at process startup, the seed is fixed for the entire process lifetime. Once the seed is fixed, the iteration order for any given set of session IDs is deterministic.

Session IDs are assigned sequentially by the tentacle networking library. An attacker who connects multiple times and observes which of their connections survive eviction can reconstruct the effective ordering and predict future eviction outcomes. By timing reconnections to land at even-indexed positions, the attacker ensures their connections are never evicted.

This contrasts sharply with the proper eviction path used in `try_evict_inbound_peer`, which explicitly uses `thread_rng()` for random selection:

```rust
// network/src/peer_registry.rs, line 205-210
// randomly evict a peer
let mut rng = thread_rng();
evict_group.choose(&mut rng).map(|peer| {
    debug!("Disconnect inbound peer {:?}", peer.connected_addr);
    peer.session_id
})
``` [2](#0-1) 

The `inbound_eviction` path has no equivalent randomness. The "user-provided random number" — i.e., the unpredictable selection that should prevent manipulation — is entirely absent.

### Impact Explanation
A bootnode is a critical peer-discovery entry point. New nodes bootstrapping onto the CKB network connect to bootnodes to obtain their initial peer list. If an attacker can guarantee their connections survive eviction on a bootnode, they can:

1. Flood the bootnode with attacker-controlled connections until capacity is reached.
2. Exploit the deterministic eviction to ensure attacker connections occupy the surviving even-indexed slots.
3. Cause the bootnode to serve only attacker-controlled peer addresses to bootstrapping nodes.
4. Eclipse newly joining nodes by poisoning their initial peer discovery.

This is a P2P-layer eclipse attack vector enabled by the absence of randomness in the eviction path.

### Likelihood Explanation
The attack is reachable by any unprivileged peer that can open TCP connections to a bootnode. No special privileges, keys, or majority hashpower are required. The attacker needs only to:
- Open enough connections to push the bootnode near its `max_inbound` limit.
- Observe which connections survive one eviction round to calibrate the HashMap ordering.
- Reconnect strategically to land at even-indexed positions.

Bootnode mode is an explicitly supported operational configuration (`bootnode_mode: true` in network config), making this a production-reachable path.

### Recommendation
Replace the deterministic `index & 0x1 != 0` filter with a random selection using `rand::thread_rng()`, consistent with the approach already used in `try_evict_inbound_peer`. For example, randomly shuffle the candidate list and evict the first half, or use `choose_multiple` from the `rand` crate. This ensures that no attacker-observable state can be used to predict or bias which connections survive.

### Proof of Concept
1. Configure a CKB node with `bootnode_mode = true` and a small `max_inbound` (e.g., 20).
2. Open 15 connections from attacker-controlled IPs (approaching the `max_inbound - 10` threshold).
3. Open one more connection to trigger `inbound_eviction`.
4. Observe which connections are disconnected (odd-indexed in HashMap iteration order).
5. Reconnect the evicted connections; repeat step 3.
6. After 2–3 rounds, the attacker has sufficient information to predict the HashMap ordering for the current session ID range.
7. Time subsequent reconnections to land at even-indexed positions, guaranteeing survival.
8. Once all surviving slots are attacker-controlled, the bootnode serves only attacker addresses to new bootstrapping nodes.
