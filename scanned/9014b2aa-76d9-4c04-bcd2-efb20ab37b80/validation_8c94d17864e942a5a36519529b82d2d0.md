### Title
Any RPC Caller Can Remove Arbitrary Transactions from the Tx Pool Without Authorization — (`rpc/src/module/pool.rs`)

### Summary

The `remove_transaction` RPC method in the Pool module accepts only a `tx_hash` and removes the matching transaction (and all its descendants) from the tx pool with no check that the caller is the original submitter or has any ownership over the transaction. Any unprivileged RPC caller who can reach the Pool RPC endpoint can silently evict any pending or proposed transaction submitted by any other user, causing a targeted denial-of-service against transaction submitters.

### Finding Description

The `remove_transaction` RPC handler is defined in `rpc/src/module/pool.rs`:

```rust
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
```

Its implementation at lines 662–669 of the same file:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

This delegates directly to `TxPoolController::remove_local_tx`, which reaches `TxPoolService::remove_tx` in `tx-pool/src/process.rs` (lines 440–456):

```rust
pub(crate) async fn remove_tx(&self, tx_hash: Byte32) -> bool {
    let id = ProposalShortId::from_tx_hash(&tx_hash);
    // checks verify_queue, orphan pool, then main pool
    let mut tx_pool = self.tx_pool.write().await;
    tx_pool.remove_tx(&id)
}
```

At no point in this call chain is there any check that the caller owns, submitted, or is otherwise authorized to remove the identified transaction. The only parameter is the hash of the target transaction, which is public information (visible via `get_raw_tx_pool`).

The Pool RPC module is registered in `rpc/src/service_builder.rs` (lines 65–77) via the standard `pool_enable` config flag, which is enabled by default in production CKB nodes. There is no per-method authentication or IP allowlist enforced at the method level.

### Impact Explanation

**Impact: High**

1. **Targeted transaction DoS**: An attacker who observes the mempool (via `get_raw_tx_pool`) can immediately call `remove_transaction` on any pending or proposed transaction, preventing it from being mined. This is especially damaging for time-sensitive transactions (e.g., those using `since` locks near expiry).

2. **Descendant chain eviction**: `remove_entry_and_descendants` in `tx-pool/src/component/pool_map.rs` (lines 252–265) removes the target transaction *and all its descendants*. A single call can evict an entire chain of dependent transactions, amplifying the DoS impact.

3. **Fee griefing / MEV interference**: An attacker can selectively remove high-fee transactions to manipulate block composition or grief competing users.

### Likelihood Explanation

**Likelihood: High**

- The Pool RPC is enabled by default on CKB nodes.
- The target transaction hash is publicly observable via `get_raw_tx_pool`.
- No special privilege, key, or network position is required — any client that can reach the RPC port can exploit this.
- The attack is trivially scriptable: poll `get_raw_tx_pool`, call `remove_transaction` on any hash of interest.

### Recommendation

Add an authorization check before removing the transaction. The most direct fix is to record the submitter identity (e.g., source IP or a caller token) at submission time and verify it matches at removal time. At minimum, restrict `remove_transaction` to a separate privileged RPC module (e.g., alongside `clear_tx_pool`) that is disabled by default and only accessible from localhost or an authenticated endpoint, consistent with how other destructive pool operations should be gated.

### Proof of Concept

1. Alice submits a valid transaction `T` via `send_transaction`. Its hash `H` is now visible in `get_raw_tx_pool`.
2. Attacker Bob (any RPC client) calls:
   ```json
   {
     "id": 1,
     "jsonrpc": "2.0",
     "method": "remove_transaction",
     "params": ["<H>"]
   }
   ```
3. The node responds `{"result": true}` — `T` and all its descendants are evicted from the pool.
4. Alice's transaction never gets mined. Bob can repeat this indefinitely for any transaction in the pool, requiring zero privilege and only knowledge of the public mempool contents. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```rust
    pub(crate) async fn remove_tx(&self, tx_hash: Byte32) -> bool {
        let id = ProposalShortId::from_tx_hash(&tx_hash);
        {
            let mut queue = self.verify_queue.write().await;
            if queue.remove_tx(&id).is_some() {
                return true;
            }
        }
        {
            let mut orphan = self.orphan.write().await;
            if orphan.remove_orphan_tx(&id).is_some() {
                return true;
            }
        }
        let mut tx_pool = self.tx_pool.write().await;
        tx_pool.remove_tx(&id)
    }
```
