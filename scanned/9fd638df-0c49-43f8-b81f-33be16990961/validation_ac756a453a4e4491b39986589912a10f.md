Audit Report

## Title
Batch Transaction Relay Aborts Entirely When Any Single Transaction Has Excessive Declared Cycles â€” (`File: sync/src/relayer/transactions_process.rs`)

## Summary

In `TransactionsProcess::execute()`, a single transaction with `declared_cycles > max_block_cycles` causes the entire `RelayTransactions` batch to be silently dropped via an early `return Status::ok()` at line 73. The valid co-batched transactions are never passed to `submit_remote_tx` and are never removed from `unknown_tx_hashes` via `mark_as_known_txs`. An unprivileged peer can exploit this to suppress delivery of specific transactions to a target node by bundling them with one transaction carrying an inflated `declared_cycles` field.

## Finding Description

`TransactionsProcess::execute()` first filters the incoming batch to only those transactions the node actually requested from this specific peer (lines 49â€“55), then performs a batch-level cycles check:

```rust
if txs
    .iter()
    .any(|(_, declared_cycles)| declared_cycles > &max_block_cycles)
{
    self.nc.ban_peer(...);
    return Status::ok();   // entire batch dropped
}
``` [1](#0-0) 

The `return` at line 73 bypasses both `mark_as_known_txs` (line 76) and the `submit_remote_tx` loop (lines 85â€“92). [2](#0-1) 

Because `mark_as_known_txs` is skipped, the valid transactions remain in `unknown_tx_hashes`. However, `UnknownTxHashPriority::next_request_peer()` only advances to a fallback peer if `peers.len() > 1`:

```rust
pub fn next_request_peer(&mut self) -> Option<PeerIndex> {
    if self.requested {
        if self.peers.len() > 1 {
            self.peers.swap_remove(0);
            self.peers.first().cloned()
        } else {
            None   // no fallback peer â€” entry is silently dropped from the queue
        }
    } else { ... }
}
``` [3](#0-2) 

In `pop_ask_for_txs`, when `next_request_peer()` returns `None`, the entry is popped and **not re-inserted**, permanently removing the hash from the request queue: [4](#0-3) 

If the attacker is the only peer that announced the target transaction hashes, those hashes are permanently lost from the node's request queue after the attack. The node will only recover if another peer independently announces the same hashes later.

## Impact Explanation

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. An attacker operating multiple sybil peers can systematically suppress transaction propagation to targeted nodes across the network. Each attack attempt costs only a P2P connection and a single crafted message; the 3-day ban is per-IP and trivially bypassed with rotating IPs. At scale, this degrades mempool liveness network-wide, delays transaction confirmation, and can be used to selectively censor specific transactions from reaching specific miners or nodes.

## Likelihood Explanation

The attack requires only an unprivileged P2P connection. The attacker:
1. Connects to the target node.
2. Sends `RelayTransactionHashes` for the target tx hashes â€” this is the standard gossip flow and raises no suspicion.
3. Waits for the node to issue `GetRelayTransactions` (triggered by `pop_ask_for_txs`).
4. Responds with a `RelayTransactions` batch containing the requested valid transactions plus one entry with `declared_cycles` set to `max_block_cycles + 1`.

The `declared_cycles` field is a plain integer in the wire format with no cryptographic binding to the transaction body at the relay layer. [5](#0-4) 

The attack is fully repeatable with fresh IPs and requires no privileged access.

## Recommendation

Replace the batch-aborting `.any()` guard with a per-transaction filter. Ban the peer when any offending transaction is detected, but continue processing the remaining valid transactions:

```rust
let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
let has_excessive = txs.iter().any(|(_, c)| c > &max_block_cycles);
if has_excessive {
    self.nc.ban_peer(self.peer, DEFAULT_BAN_TIME,
        String::from("relay declared cycles greater than max_block_cycles"));
}
let txs: Vec<_> = txs.into_iter()
    .filter(|(_, c)| c <= &max_block_cycles)
    .collect();
if txs.is_empty() {
    return Status::ok();
}
// proceed to mark_as_known_txs and submit_remote_tx
```

## Proof of Concept

1. Attacker peer connects to a CKB node.
2. Attacker sends `RelayTransactionHashes([H_valid_1, H_valid_2, H_bad])`.
3. Node calls `add_ask_for_txs`, inserting all three hashes into `unknown_tx_hashes` with `requested: false`.
4. Node's periodic `pop_ask_for_txs` fires, calls `next_request_peer()` (sets `requested: true`), and sends `GetRelayTransactions([H_valid_1, H_valid_2, H_bad])` to the attacker.
5. Attacker responds with `RelayTransactions` containing `tx_valid_1` (correct cycles), `tx_valid_2` (correct cycles), `tx_bad` (any body, `declared_cycles = max_block_cycles + 1`).
6. The filter at lines 49â€“55 passes all three (all were requested from this peer).
7. `.any()` at line 66 returns `true` for `tx_bad`; peer is banned; `return Status::ok()` fires.
8. `mark_as_known_txs` and `submit_remote_tx` are never called for `tx_valid_1` or `tx_valid_2`.
9. If the attacker was the sole announcer of `H_valid_1`/`H_valid_2`, `next_request_peer()` returns `None` on the next `pop_ask_for_txs` tick, permanently evicting those hashes from the queue.
