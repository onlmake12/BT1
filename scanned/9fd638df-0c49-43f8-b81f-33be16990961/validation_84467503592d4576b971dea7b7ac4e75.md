Audit Report

## Title
Off-by-one in spawn guard allows 17 concurrent VMs instead of the defined maximum of 16 — (`script/src/scheduler.rs`)

## Summary
`MAX_VMS_COUNT` is defined as `16` at line 34, but the spawn guard at line 532 uses strict `>` instead of `>=`. When exactly 16 VMs are alive (`suspended + instantiated == 16`), the expression `16 > 16` evaluates to `false`, the guard is skipped, and `boot_vm` is called, producing a 17th concurrent VM. This violates the protocol-defined VM count limit in CKB-VM.

## Finding Description
`MAX_VMS_COUNT` is declared as:

```rust
// script/src/scheduler.rs, line 34
pub const MAX_VMS_COUNT: u64 = 16;
``` [1](#0-0) 

The spawn guard in `process_message_box` is:

```rust
// script/src/scheduler.rs, line 532
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
``` [2](#0-1) 

`instantiated` (`BTreeMap<VmId, (VmContext<DL>, M)>`) and `suspended` (`BTreeMap<VmId, Snapshot2<DataPieceId>>`) together represent all live VMs, including the root VM. [3](#0-2) 

When the root VM plus 15 children are alive, `suspended.len() + instantiated.len() == 16`. The condition `16 > 16` is `false`, so the guard does not fire and `boot_vm` is called at line 539, creating a 17th VM. The guard only triggers at 17 existing VMs (`17 > 16 == true`), meaning the effective cap is 17, not 16. [4](#0-3) 

## Impact Explanation
This is an incorrect implementation of CKB-VM behavior: the protocol-defined constant `MAX_VMS_COUNT = 16` is not enforced. A script can instantiate 17 concurrent VMs instead of 16. This fits the allowed impact: **High — Incorrect implementation or behavior of CKB-VM or system scripts**.

## Likelihood Explanation
The path is fully unprivileged. Any user can submit a transaction via RPC or P2P relay containing a V2 ELF script that spawns 16 children (each blocking on `wait`) and then spawns a 17th. The condition is deterministic and locally reproducible with no special access, keys, or PoW required.

## Recommendation
Change the guard from strict greater-than to greater-than-or-equal:

```rust
// Before (off-by-one):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

## Proof of Concept
1. Write a V2 RISC-V ELF that calls `spawn` in a loop 16 times; each child calls `wait(999)` (non-existent pid) so it remains alive in suspended/instantiated state.
2. The root VM then calls `spawn` a 17th time.
3. With the current code (`>`), the 17th `spawn` returns `SUCCESS` and `scheduler.instantiated.len() + scheduler.suspended.len()` reaches 17.
4. With the fix (`>=`), the 17th `spawn` returns `MAX_VMS_SPAWNED`, correctly enforcing the limit.
