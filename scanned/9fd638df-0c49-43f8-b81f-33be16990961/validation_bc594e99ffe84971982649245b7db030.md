Audit Report

## Title
Under-constrained `genesis_epoch_length`, `orphan_rate_target`, and `epoch_duration_target` Consensus Parameters Allow Node Crash via Division-by-Zero - (File: spec/src/consensus.rs)

## Summary

`build_genesis_epoch_ext` in `spec/src/consensus.rs` performs integer division by `genesis_epoch_length`, `genesis_orphan_rate.1`, and `epoch_duration_target` with no zero-checks. All three values flow directly from the user-controlled chain spec TOML `[params]` section through `ChainSpec::build_consensus` without any validation layer. Setting any of them to zero causes an immediate integer division-by-zero panic that crashes the node process at startup. The only guard present (`debug_assert!` for `epoch_duration_target`) is stripped in release builds.

## Finding Description

**Root cause 1 — `genesis_epoch_length = 0`**

`ChainSpec::build_consensus` (`spec/src/lib.rs:562-568`) calls `build_genesis_epoch_ext` passing `self.params.genesis_epoch_length()` directly. Inside `build_genesis_epoch_ext`:

```rust
// spec/src/consensus.rs:222-223
let block_reward = Capacity::shannons(epoch_reward.as_u64() / genesis_epoch_length);
let remainder_reward = Capacity::shannons(epoch_reward.as_u64() % genesis_epoch_length);
```

If `genesis_epoch_length = 0`, both lines panic with integer division by zero at node startup. No validation exists between TOML deserialization and this call.

**Root cause 2 — `orphan_rate_target = [N, 0]`**

```rust
// spec/src/consensus.rs:225-226
let genesis_orphan_count =
    genesis_epoch_length * genesis_orphan_rate.0 as u64 / genesis_orphan_rate.1 as u64;
```

A zero denominator panics here. Additionally, `ConsensusBuilder::orphan_rate_target` stores the pair via `RationalU256::new_raw`, explicitly bypassing the zero-denominator guard in `RationalU256::new`:

```rust
// spec/src/consensus.rs:387-392
pub fn orphan_rate_target(mut self, orphan_rate_target: (u32, u32)) -> Self {
    self.inner.orphan_rate_target = RationalU256::new_raw(
        U256::from(orphan_rate_target.0),
        U256::from(orphan_rate_target.1),
    );
    self
}
```

This zero-denominator `RationalU256` is then used in `next_epoch_ext` (`spec/src/consensus.rs:887-896`) during epoch boundary processing, where arithmetic on `orphan_rate_target` eventually calls `into_u256()` → `self.numer / self.denom` — a second division-by-zero panic path reachable on every epoch transition when `epoch_uncles_count != 0`.

**Root cause 3 — `epoch_duration_target = 0`**

```rust
// spec/src/consensus.rs:227-229
let genesis_hash_rate = compact_to_difficulty(compact_target)
    * (genesis_epoch_length + genesis_orphan_count)
    / epoch_duration_target;
```

The only guard is:

```rust
// spec/src/consensus.rs:342-345
debug_assert!(
    self.inner.epoch_duration_target() != 0,
    "epoch_duration_target must be non-zero"
);
```

This `debug_assert!` is compiled away in release builds, providing zero protection in production binaries. The panic at line 229 is reached before the assert in `ConsensusBuilder::build` anyway, since `build_genesis_epoch_ext` is called earlier in `build_consensus`.

**No validation layer exists:** `Params::genesis_epoch_length()`, `Params::epoch_duration_target()`, and `Params::orphan_rate_target()` (`spec/src/lib.rs:303-318`) are simple `unwrap_or_else` accessors with no bounds checking. The `Params` struct (`spec/src/lib.rs:185-251`) accepts all three as raw `Option<T>` with no serde validation constraints.

## Impact Explanation

A node operator running a dev or private CKB network who sets `genesis_epoch_length = 0`, `epoch_duration_target = 0`, or `orphan_rate_target = [1, 0]` in their local chain spec TOML will experience an immediate, unrecoverable node process crash at startup with no meaningful error message — only a panic. This constitutes a **local command line crash** (Note, 0–500 points). The `orphan_rate_target` zero-denominator path additionally defers the crash to the first epoch boundary if startup happens to survive (e.g., numerator also zero), crashing the running node process at that point.

## Likelihood Explanation

The `[params]` section of the chain spec TOML is explicitly documented as user-configurable for dev and private networks. All three fields are `Option<T>` with no validation between deserialization and use. A misconfiguration — accidental or otherwise — by any node operator running a dev or private chain triggers the crash deterministically. The `orphan_rate_target` denominator path is particularly subtle because `new_raw` is used instead of `new`, silently bypassing the only existing guard and deferring the panic to epoch transition time.

## Recommendation

1. **Validate all divisor parameters before use** in `build_genesis_epoch_ext` and `build_consensus`. Return a `Result` error (not a panic) when `genesis_epoch_length == 0`, `epoch_duration_target == 0`, or `orphan_rate_target.1 == 0`.
2. **Replace `debug_assert!` with `assert!`** (or proper `Result`-returning validation) for `epoch_duration_target != 0` and `genesis_epoch_length != 0` in `ConsensusBuilder::build` so the check is not stripped in release builds.
3. **Use `RationalU256::new` instead of `RationalU256::new_raw`** in `ConsensusBuilder::orphan_rate_target` so the zero-denominator is caught at construction time with a clear error rather than silently stored and panicking later.
4. Add minimum-value constraints in `Params` accessors: `genesis_epoch_length >= 1`, `epoch_duration_target >= 1`, `orphan_rate_target.1 >= 1`.
