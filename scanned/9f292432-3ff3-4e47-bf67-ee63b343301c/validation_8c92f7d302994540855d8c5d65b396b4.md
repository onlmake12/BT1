### Title
`declared_cycles` from Remote Peers Bypasses `max_tx_verify_cycles` Limit in `_process_tx` - (File: `tx-pool/src/process.rs`)

### Summary

When a remote peer relays a transaction, it supplies a `declared_cycles` value that is used directly as the script-verification cycle cap inside `_process_tx`. The only upstream guard checks `declared_cycles <= max_block_cycles` (the consensus-wide limit), but never checks against `max_tx_verify_cycles` (the node's per-transaction DoS-protection limit). A malicious peer can therefore force the node to run script verification with a cycle budget up to `max_block_cycles` — orders of magnitude larger than the operator-configured `max_tx_verify_cycles` — bypassing the intended resource cap entirely.

### Finding Description

**Root cause — conditional bounds check**

In `TransactionsProcess::execute()` the only validation on the peer-supplied `declared_cycles` is:

```rust
// sync/src/relayer/transactions_process.rs  lines 63-74
let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
if txs.iter().any(|(_, declared_cycles)| declared_cycles > &max_block_cycles) {
    self.nc.ban_peer(..., "relay declared cycles greater than max_block_cycles");
    return Status::ok();
}
```

Any value in `[1, max_block_cycles]` passes this check and is forwarded to the tx-pool as `declared_cycles`.

Inside `_process_tx` the cycle cap for the actual script verifier is set unconditionally from the peer-supplied value:

```rust
// tx-pool/src/process.rs  line 720
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```

`max_cycles` is then passed straight into `verify_rtx`:

```rust
// tx-pool/src/process.rs  lines 724-732
let verified_ret = verify_rtx(
    Arc::clone(&snapshot),
    Arc::clone(&rtx),
    tx_env,
    &verify_cache,
    max_cycles,          // ← peer-controlled value, not max_tx_verify_cycles
    command_rx,
).await;
```

`verify_rtx` in turn calls `ContextualTransactionVerifier::verify(max_tx_verify_cycles, …)` or `verify_with_pause(max_tx_verify_cycles, …)`, where `max_tx_verify_cycles` is the argument it received — i.e., the peer-supplied `declared_cycles`, not the node's configured `TxPoolConfig::max_tx_verify_cycles`.

**The missing check**

`TxPoolConfig::max_tx_verify_cycles` (default 70 000 000 cycles) is used only for routing decisions — deciding whether a tx goes to the async verify queue — but is never enforced as an upper bound on the actual cycle limit passed to the script verifier when `declared_cycles` is present.

**Attack path**

1. Attacker crafts a transaction whose lock/type script consumes close to `max_block_cycles` cycles (e.g., a tight loop).
2. Attacker relays it via the P2P relay protocol with `declared_cycles = max_block_cycles` (e.g., 3 500 000 000 on mainnet).
3. `TransactionsProcess::execute()` accepts it (passes the `> max_block_cycles` check).
4. `_process_tx` sets `max_cycles = 3_500_000_000` and runs the script verifier with that budget.
5. The node spends up to ~50× more CPU time per transaction than `max_tx_verify_cycles` was designed to allow.
6. Repeating with many such transactions causes sustained CPU exhaustion / node slowdown.

### Impact Explanation

The `max_tx_verify_cycles` configuration exists specifically to cap per-transaction verification cost and prevent DoS. By supplying a large `declared_cycles`, an unprivileged P2P peer can silently override this cap, causing the node to spend up to `max_block_cycles / max_tx_verify_cycles` times more CPU per transaction than the operator intended. On mainnet defaults this ratio is `3 500 000 000 / 70 000 000 = 50×`. Sustained relay of such transactions can degrade or stall transaction processing.

### Likelihood Explanation

Any unprivileged P2P peer can send `RelayTransactions` messages with arbitrary `declared_cycles` values up to `max_block_cycles`. No special privilege, key, or majority hash-power is required. The attacker only needs a valid (resolvable) transaction whose script actually consumes many cycles.

### Recommendation

In `_process_tx`, clamp the effective cycle limit to the node's configured `max_tx_verify_cycles` when `declared_cycles` is provided:

```rust
// tx-pool/src/process.rs
let max_cycles = declared_cycles
    .map(|d| d.min(self.tx_pool_config.max_tx_verify_cycles))
    .unwrap_or_else(|| self.consensus.max_block_cycles());
```

Alternatively, enforce the check in `TransactionsProcess::execute()` against `max_tx_verify_cycles` in addition to `max_block_cycles`, and ban peers that exceed it.

### Proof of Concept

1. Configure a CKB node with `max_tx_verify_cycles = 70_000_000`.
2. Craft a transaction whose lock script loops for ~3 000 000 000 cycles before returning 0 (valid exit).
3. Relay it via the P2P relay protocol with `cycles = 3_000_000_000` (below `max_block_cycles`, so the peer is not banned).
4. Observe that `_process_tx` sets `max_cycles = 3_000_000_000` and the script verifier runs for ~3B cycles instead of stopping at 70M.
5. Repeat with many transactions to sustain CPU pressure on the target node.

pub(crate) async fn verify_rtx(
    snapshot: Arc<Snapshot>,
    rtx: Arc<ResolvedTransaction>,
    tx_env: Arc<TxVerifyEnv>,
    cache_entry: &Option<CacheEntry>,
    max_tx_verify_cycles: Cycle,
    command_rx: Option<&mut watch::Receiver<ChunkCommand>>,
) -> Result<Completed, Reject> {
    let consensus = snapshot.cloned_consensus();
    let data_loader = snapshot.as_data_loader();

    if let Some(completed) = cache_entry {
        TimeRelativeTransactionVerifier::new(rtx, consensus, data_loader, tx_env)
            .verify()
            .map(|_| *completed)
            .map_err(Reject::Verification)
    } else if let Some(command_rx) = command_rx {
        ContextualTransactionVerifier::new(
            Arc::clone(&rtx),
            consensus,
            data_loader,
            Arc::clone(&tx_env),
        )
        .verify_with_pause(max_tx_verify_cycles, command_rx)
        .await
        .and_then(|result| {
            DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
                .verify()?;
            Ok(result)
        })
        .map_err(Reject::Verification)
    } else {
        block_in_place(|| {
            ContextualTransactionVerifier::new(Arc::clone(&rtx), consensus, data_loader, tx_env)
                .verify(max_tx_verify_cycles, false)
                .and_then(|result| {
                    DaoScriptSizeVerifier::new(
                        rtx,
                        snapshot.cloned_consensus(),
                        snapshot.as_data_loader(),
                    )
                    .verify()?;
                    Ok(result)
                })
                .map_err(Reject::Verification)
        })
    }
