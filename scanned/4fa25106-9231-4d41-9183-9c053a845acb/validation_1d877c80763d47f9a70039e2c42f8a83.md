Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Stale Snapshot Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` snapshots the prospective new `total_tx_size`/`total_tx_cycles` before calling `check_and_record_ancestors`, which may evict entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`. The decrements applied during eviction are then silently overwritten when `add_entry` unconditionally restores the pre-eviction snapshot at lines 218–219. Each trigger permanently inflates `total_tx_size` by the byte-size of the evicted entries, causing `limit_size` to over-evict legitimate transactions from the pool.

## Finding Description

In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`):

```
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // snapshot: T + entry.size
evicts = self.check_and_record_ancestors(&mut entry)?;          // may decrement self.total_tx_size by E
// ...
self.total_tx_size = total_tx_size;    // writes T + entry.size, not T - E + entry.size
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) reads `self.total_tx_size` at call time and returns `self.total_tx_size + entry.size` as a local variable. It does not mutate `self`.

`check_and_record_ancestors` (lines 588–640) enters the eviction branch when `ancestors_count > max_ancestors_count` and `cell_ref_parents` are present (lines 603–625). It calls `remove_entry_and_descendants` (lines 252–265), which calls `remove_entry` (lines 235–250) for each removed entry. `remove_entry` calls `update_stat_for_remove_tx` (lines 733–758) at line 247, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in place.

After `check_and_record_ancestors` returns, `self.total_tx_size` correctly equals `T − E` (where `E` is the total size of evicted entries). Lines 218–219 then unconditionally overwrite this with the stale snapshot `T + entry.size`, producing a final value of `T + entry.size` instead of the correct `T − E + entry.size`. The inflation per trigger is exactly `E`.

`limit_size` (lines 292–329 of `tx-pool/src/pool.rs`) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts real entries. With a sufficiently inflated phantom total, it evicts all real pool contents even though the actual byte count is within limits. No guard or recomputation is triggered on the inflation path; `recompute_total_stat` is only called on underflow.

## Impact Explanation

An attacker can repeatedly inflate `total_tx_size` until it exceeds `max_tx_pool_size`. At that point `limit_size` evicts every legitimate pending/proposed transaction from the pool. The node's mempool becomes permanently empty and re-submitted honest transactions are immediately evicted again on the next `limit_size` call. This constitutes a sustained tx-pool denial-of-service against the node, preventing it from relaying or mining any user transactions. Targeting multiple nodes simultaneously causes CKB network-wide transaction congestion with low cost to the attacker.

**Impact: High** — matches "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation

The eviction path requires `ancestors_count > max_ancestors_count` (default 25) with at least one `cell_ref_parent` that can be evicted to bring the count back within limits. An unprivileged caller via `send_transaction` RPC can construct this: build a 24-deep chain, add a `cell_dep` transaction referencing the chain tip's output, then submit a transaction consuming that output as an input. This pushes `ancestors_count` to 26, triggering eviction of the `cell_dep` tx. The attacker needs ~26 funded outputs per trigger cycle, which is a low barrier. The attack is repeatable indefinitely with fresh outputs.

**Likelihood: Medium** — deliberate construction required, no privileged access needed.

## Recommendation

Move the `updated_stat_for_add_tx` call to after `check_and_record_ancestors` completes, so the base value of `self.total_tx_size` already reflects any evictions:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size now reflects evictions; compute new totals from current state
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, add a debug-build invariant assertion after every `add_entry` that `total_tx_size == entries.iter().map(|e| e.size).sum()`.

## Proof of Concept

1. Configure a node with `max_ancestors_count = 25`, `max_tx_pool_size = 20 MB`.
2. Submit `tx_0` creating `cell_0`.
3. Submit `tx_1…tx_24` as a chain, each spending the previous output.
4. Submit `tx_dep` using `tx_24`'s output as a **cell dep** (not consuming it). `tx_dep` becomes a `cell_ref_parent`.
5. Submit `tx_25` spending `tx_24`'s output as an **input**.
   - `ancestors_count = 26 > 25`; `cell_ref_parents = {tx_dep}`; eviction branch fires.
   - `remove_entry_and_descendants(tx_dep)` → `update_stat_for_remove_tx` decrements `self.total_tx_size` by `size(tx_dep)`.
   - `add_entry` writes back stale snapshot: `self.total_tx_size += size(tx_dep)` net inflation.
6. Repeat steps 3–5 with fresh outputs. Each iteration inflates `total_tx_size` by `size(tx_dep)`.
7. After `ceil(max_tx_pool_size / size(tx_dep))` iterations, `limit_size` fires and evicts all real pool contents.
8. Verify: `pool_map.entries` is empty while `pool_map.total_tx_size > max_tx_pool_size` — the invariant is broken.

A unit test can reproduce this by directly calling `pool_map.add_entry` in a loop with the described transaction structure and asserting `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.inner.size).sum::<usize>()` after each insertion.
