Audit Report

## Title
Unbounded Pre-Aggregation in `get_tx_with_cells` Grouped Mode Allows Unprivileged RPC DoS ‚Äî (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary

When `group_by_transaction=true`, the `get_transactions` RPC in the rich-indexer constructs a three-level nested SQL query where `GROUP BY` with aggregation functions (`GROUP_CONCAT`/`ARRAY_AGG`) is applied before `LIMIT`. Both SQLite and PostgreSQL must fully materialize all matching rows before emitting any grouped result, regardless of the requested `limit`. The rich-indexer applies no query timeout (unlike the original indexer), so a single crafted RPC call with a broad prefix search can force unbounded database work, making the indexer RPC unresponsive.

## Finding Description

`get_tx_with_cells` in `get_transactions.rs` constructs the query in three stages:

**Level 1 (innermost):** `build_tx_with_cell_union_sub_query` (lines 432‚Äì541) produces a `UNION ALL` of all matching outputs and inputs with no `LIMIT`.

**Level 2 (middle):** Lines 281‚Äì299 join the union result with `ckb_transaction` and `block`, apply an optional `block_range` filter, and wrap it as a subquery ‚Äî still with no `LIMIT`.

**Level 3 (outer):** Lines 320‚Äì332 apply `WHERE tx_id > ?after`, then `GROUP BY tx_id, block_number, tx_index, tx_hash` with `GROUP_CONCAT`/`ARRAY_AGG`, then `ORDER BY tx_id`, then `LIMIT limit`.

Because `GROUP BY` with aggregation is a blocking operator, the database engine must consume every row from the inner subquery before it can return even one grouped result. `LIMIT` is applied only after all groups are formed.

The `request_limit` guard at lines 27‚Äì31 only rejects caller-supplied `limit` values that exceed the configured cap; it does not bound the number of rows the inner subquery scans and aggregates.

Critically, the rich-indexer service does not propagate or enforce `timeout_limit` at all. In `service.rs` line 51, only `request_limit` is stored and forwarded to `AsyncRichIndexerHandle`; `timeout_limit` (which the original indexer does enforce) is absent from the rich-indexer's service construction and handle. This means queries run without any time bound.

The default `request_limit` is `usize::MAX` when unconfigured, so by default there is no cap on either query duration or result size.

## Impact Explanation

This matches **Note (0‚Äì500 points): Any local RPC API crash**. A single valid JSON-RPC call with `group_by_transaction=true`, `script_search_mode: "prefix"`, and a broad `args` prefix forces the indexer database to scan, join, and aggregate an unbounded number of rows before returning. With no timeout enforcement in the rich-indexer, the query runs to completion regardless of duration, blocking the database connection pool and making the indexer RPC unresponsive for the duration of the attack. The claimed "High" severity (easily crash a CKB node or cause network congestion) is not supported: the rich-indexer is an optional, separate service; its degradation does not affect CKB consensus, block propagation, or the core node process.

## Likelihood Explanation

The rich-indexer must be explicitly enabled (`--rich-indexer`). The RPC binds to `127.0.0.1:8114` by default, but operators running public indexer services (block explorers, wallet backends) routinely expose it. For such deployments, any network-accessible caller can issue the attack payload ‚Äî a single valid JSON-RPC POST with no authentication. The stress test script in the repository (`get_transactions_prefix.lua`) demonstrates the exact call pattern with `group_by_transaction: true` and `script_search_mode: "prefix"`, confirming the developers themselves identified this as a high-load path. The attack is repeatable with no rate limiting in the RPC layer.

## Recommendation

Restructure the grouped query to apply `LIMIT` on distinct `tx_id` values before aggregation, using a CTE or subquery to first select the top-N `tx_id` values (with `ORDER BY` and `LIMIT`), then join back to aggregate only those N transactions' cells. Additionally, enforce `timeout_limit` in the rich-indexer service the same way the original indexer does, and document that `request_limit` does not bound query scan cost.

## Proof of Concept

1. Enable the rich-indexer on a node with a populated database containing many transactions for a common lock script.
2. Send the following RPC call:
```json
{
  "id": 1, "jsonrpc": "2.0", "method": "get_transactions",
  "params": [
    {
      "script": {
        "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
        "hash_type": "type", "args": "0x"
      },
      "script_type": "lock",
      "script_search_mode": "prefix",
      "group_by_transaction": true
    },
    "asc", "0x1"
  ]
}
```
3. Run `EXPLAIN QUERY PLAN` on the generated SQL against the SQLite database; the plan will show `USE TEMP B-TREE FOR GROUP BY` with no early termination, confirming full materialization before `LIMIT 1` is applied.
4. Benchmark grouped (`group_by_transaction: true`) vs. ungrouped latency on a dataset with 10,000+ matching transactions to observe O(N) vs. O(limit) response time difference.
5. Confirm that no query timeout fires by observing the rich-indexer service construction in `service.rs` ‚Äî `timeout_limit` is never read or forwarded, unlike in `util/indexer/src/service.rs`.
