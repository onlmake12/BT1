### Title
Tampered CompactBlockV1 Extension Bytes Cause Permanent BLOCK_INVALID Marking — (`sync/src/relayer/mod.rs`, `chain/src/chain_service.rs`, `chain/src/verify.rs`)

---

### Summary

An unprivileged relay peer can send a `CompactBlockV1` whose extension field has been mutated by one or more bits. The node reconstructs a full `BlockV1` using the tampered extension but the original (valid-PoW) header, passes it through `accept_block`, and eventually fails `BlockExtensionVerifier` because the computed `extra_hash` of the tampered extension does not match the `extra_hash` committed in the header. The failure path in `ConsumeUnverifiedBlockProcessor` permanently writes `BLOCK_INVALID` for the block hash. Because the block hash is derived solely from the header (not the extension), the same hash is used for the legitimate block, which is then permanently rejected by the victim node.

---

### Finding Description

**Step 1 — Attacker-controlled entrypoint**

The attacker connects as a relay peer and sends a `CompactBlockV1` (molecule union tag 6) with:
- The original, untampered header (valid PoW, valid `extra_hash` committed to the real extension)
- Extension bytes mutated by at least one bit

**Step 2 — Compact block processing passes all pre-reconstruction checks**

In `CompactBlockProcess::execute()`:
- `non_contextual_check` only validates uncle count, proposal count, and block height. [1](#0-0) 
- `contextual_check` runs `HeaderVerifier` (PoW, timestamp, epoch) against the untampered header — passes. [2](#0-1) 
- `CompactBlockVerifier::verify` only checks prefilled-tx structure and short-id duplicates — no extension check. [3](#0-2) 

**Step 3 — `reconstruct_block` builds a poisoned `BlockV1`**

`reconstruct_block` blindly uses `compact_block.extension()` to build the `BlockV1`:

```rust
let block = if let Some(extension) = compact_block.extension() {
    packed::BlockV1::new_builder()
        .header(compact_block.header())   // original header, valid extra_hash
        .extension(extension)             // ← TAMPERED bytes
        .build()
        .as_v0()
``` [4](#0-3) 

The only post-reconstruction check is `transactions_root` — there is no extension-vs-extra_hash check here. [5](#0-4) 

**Step 4 — `accept_block` submits the poisoned block to the chain pipeline**

`accept_block` calls `accept_remote_block`, which routes to `asynchronous_process_block` in `ChainService`. [6](#0-5) 

**Step 5 — Contextual verification fails and BLOCK_INVALID is written**

`BlockExtensionVerifier::verify` computes `actual_extra_hash` from the tampered extension bytes and compares it to `block.extra_hash()` (the value committed in the original header):

```rust
let actual_extra_hash = block.calc_extra_hash().extra_hash();
if actual_extra_hash != block.extra_hash() {
    return Err(BlockErrorKind::InvalidExtraHash.into());
}
``` [7](#0-6) 

On this error, `ConsumeUnverifiedBlockProcessor` permanently writes:

```rust
self.shared.insert_block_status(block_hash.clone(), BlockStatus::BLOCK_INVALID);
``` [8](#0-7) 

**Step 6 — Legitimate block is permanently suppressed**

The block hash is computed from the header only, not the extension. The tampered compact block and the legitimate block share the same hash. When the legitimate block arrives (via `SendBlock` or another compact block relay), `contextual_check` immediately returns:

```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
``` [9](#0-8) 

The node will never accept the legitimate block again without a restart that clears the in-memory `block_status_map`.

---

### Impact Explanation

- **Block suppression**: A targeted miner's block can be permanently hidden from any victim node the attacker is peered with.
- **Chain fork amplification**: If the attacker is positioned between a miner and a significant fraction of the network, the miner's block is orphaned on those nodes, causing mining revenue loss and potential chain instability.
- **No recovery without restart**: `block_status_map` is in-memory only; the BLOCK_INVALID entry persists for the lifetime of the process. [10](#0-9) 

---

### Likelihood Explanation

- Requires only a standard P2P relay connection — no keys, no hashpower, no Sybil attack.
- The attacker must receive or anticipate the compact block before the victim does, which is trivially achievable for a man-in-the-middle relay peer.
- The mutation is a single-bit flip in the extension bytes — trivial to implement.
- The attack is silent: the victim node logs an error but does not ban the peer for this specific failure path (the `ReconstructionResult::Error` arm in `compact_block_process.rs` returns a status code but does not trigger peer punishment). [11](#0-10) 

---

### Recommendation

Before calling `accept_block` with the reconstructed block, verify that the extension bytes in the compact block are consistent with the `extra_hash` committed in the header:

```rust
// After reconstruction, before accept_block:
let actual_extra_hash = block.calc_extra_hash().extra_hash();
if actual_extra_hash != block.extra_hash() {
    // Reject the compact block; do NOT mark BLOCK_INVALID
    // (the header itself may be valid; only the compact block is malformed)
    return StatusCode::ProtocolMessageIsMalformed
        .with_context("compact block extension does not match header extra_hash");
}
```

This check must **not** set `BLOCK_INVALID` — only the compact block message is malformed, not the block itself. The peer should be penalized/banned instead.

---

### Proof of Concept

1. Observe a valid mined `BlockV1` with hash `H` being relayed as a `CompactBlockV1`.
2. Intercept the compact block message before it reaches the victim node.
3. Flip one bit in the `extension` field of the `CompactBlockV1`.
4. Forward the mutated compact block to the victim node.
5. Observe that the victim node logs `InvalidExtraHash` and sets `BLOCK_INVALID` for hash `H`.
6. Send the legitimate `BlockV1` (or the original compact block) to the victim node via `SendBlock`.
7. Observe that the victim node rejects it with `BlockIsInvalid` and never adds it to its chain.

The `build_from_block` function in `util/types/src/extension.rs` confirms that the extension is copied verbatim from the block into the compact block, and `reconstruct_block` copies it verbatim back — with no integrity check against the header's `extra_hash` at any point in the relay pipeline. [12](#0-11)
