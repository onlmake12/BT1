### Title
`ScriptHashTypeVerifier` Omits Type-Script Hash-Type Validation on Transaction Outputs — (`verification/src/transaction_verifier.rs`)

### Summary

`NonContextualTransactionVerifier` includes a `ScriptHashTypeVerifier` step that enforces `ENABLED_SCRIPT_HASH_TYPE` only on the **lock** script of each output. The **type** script of each output is never checked against the same permitted set. A transaction submitter can therefore create an output whose type script carries a disallowed or reserved `hash_type` byte, bypassing the non-contextual gate entirely. Any capacity locked in such a cell becomes permanently unspendable, because every future attempt to consume the cell must execute the type script, which the VM will reject.

### Finding Description

`ScriptHashTypeVerifier::verify()` iterates over `self.transaction.outputs()` and validates only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  ~line 796
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(…ScriptHashTypeNotPermitted…);
            }
        } else {
            return Err(…InvalidScriptHashType…);
        }
    }
    Ok(())
}
```

`output.type_()` — the optional type script — is never inspected here. The verifier is composed into `NonContextualTransactionVerifier::verify()` as the final step, and `NonContextualTransactionVerifier` is the gate used by both the tx-pool (`pre_check`) and the block-level `NonContextualBlockTxsVerifier`.

The structural parallel to the Olympus finding is exact: the check covers one category of scripts (lock) but silently skips an equally critical category (type), mirroring how `_isHighRiskProposal` covered `UpgradeModule`/`InstallPolicy`/`DeactivatePolicy` but not `MigrateKernel` or `_setModuleRiskLevel`.

The `check_data` path in `util/gen-types` does validate both lock and type hash types (as confirmed by the test in `util/gen-types/src/extension/tests/check_data.rs` lines 133–167, where `output_error2` with an invalid type-script hash type causes `check_data` to return `false`). However, `check_data` is a P2P relay-layer guard; it is not called inside `NonContextualTransactionVerifier::verify()` or the tx-pool `pre_check` path, leaving a gap for transactions submitted via the local RPC (`send_transaction`) or injected through other non-relay paths.

### Impact Explanation

An attacker submits a transaction via RPC that creates an output with a syntactically valid but consensus-disallowed type-script `hash_type` (e.g., the reserved value `3`). The transaction passes `NonContextualTransactionVerifier`, enters the tx-pool, and is mined into a block. The resulting live cell is permanently unspendable: every spend attempt triggers type-script execution, which the VM rejects for the invalid hash type. Any CKB capacity placed in that output is irrecoverably locked. The invariant that only permitted hash types appear in chain state is violated.

### Likelihood Explanation

Any unprivileged user with RPC access to a node (the default for local nodes and many hosted nodes) can submit such a transaction. No special keys, hashpower, or network position are required. The crafted transaction is cheap to construct and costs only the minimum cell capacity.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for every output that carries one:

```rust
for output in self.transaction.outputs() {
    // existing lock check …
    if let Some(type_script) = output.type_().to_opt() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: type_script.hash_type(),
            }.into());
        }
    }
}
```

This mirrors the existing lock-script check and closes the gap without touching the `check_data` relay-layer guard.

### Proof of Concept

1. Construct a `TransactionView` whose first output has a valid lock script and a type script with `hash_type = 3` (reserved/invalid).
2. Submit via `send_transaction` RPC.
3. Observe that `NonContextualTransactionVerifier::verify()` returns `Ok(())` — the type-script hash type is never inspected.
4. The transaction enters the tx-pool and is eligible for mining.
5. Once mined, the output cell is live but permanently unspendable: any spend triggers type-script execution, which the VM rejects.

/// Context-independent verification checks for transaction
///
/// Basic checks that don't depend on any context
/// Contains:
/// - Check for version
/// - Check for size
/// - Check inputs and output empty
/// - Check for duplicate deps
/// - Check for whether outputs match data
/// - Check whether output lock hash type within enabled range
pub struct NonContextualTransactionVerifier<'a> {
    pub(crate) version: VersionVerifier<'a>,
    pub(crate) size: SizeVerifier<'a>,
    pub(crate) empty: EmptyVerifier<'a>,
    pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,
    pub(crate) outputs_data_verifier: OutputsDataVerifier<'a>,
    pub(crate) script_hash_type: ScriptHashTypeVerifier<'a>,
}

impl<'a> NonContextualTransactionVerifier<'a> {
    /// Creates a new NonContextualTransactionVerifier
    pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
        NonContextualTransactionVerifier {
            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),
            duplicate_deps: DuplicateDepsVerifier::new(tx),
            outputs_data_verifier: OutputsDataVerifier::new(tx),
            script_hash_type: ScriptHashTypeVerifier::new(tx),
        }
    }

    /// Perform context-independent verification
    pub fn verify(&self) -> Result<(), Error> {
        self.version.verify()?;
        self.size.verify()?;
        self.empty.verify()?;
        self.duplicate_deps.verify()?;
        self.outputs_data_verifier.verify()?;
        self.script_hash_type.verify()?;
        Ok(())
    }
