Let me check the `public_addrs` function and the `is_reachable` filter behavior to fully confirm the exploit path.

### Title
Unauthenticated Inbound Peer Can Inject Arbitrary Public IP into Victim's Self-Advertised Listen Addresses — (`network/src/protocols/identify/mod.rs`)

---

### Summary

Any inbound peer can send a crafted `observed_addr` containing an arbitrary globally-routable IP address. The victim node stores it without validation and includes it in the `listen_addrs` field of every subsequent `IdentifyMessage` it sends to new peers, poisoning network-wide peer discovery.

---

### Finding Description

The `process_observed` function passes the attacker-supplied address directly to `add_observed_addr` with no IP validation: [1](#0-0) 

`add_observed_addr` only appends the local peer-id if absent, then unconditionally stores the address: [2](#0-1) 

`NetworkState::add_observed_addr` inserts it into the `observed_addrs` map keyed by `session_id` with no further checks: [3](#0-2) 

`local_listen_addrs` then appends `observed_addrs` directly to the list returned to every newly-connecting peer: [4](#0-3) 

The only filter applied before transmission is the `is_reachable` check in `connected`: [5](#0-4) 

`is_reachable` only rejects private/loopback ranges. A public attacker-controlled IP (e.g., `1.2.3.4`) passes this filter and is broadcast verbatim.

**Contrast with `process_listens`**, which explicitly applies the same `is_reachable` filter before storing remote peers' addresses: [6](#0-5) 

`process_observed` has no equivalent guard. There is also no check that the `observed_addr` IP matches the actual remote address of the session.

---

### Impact Explanation

- The victim node broadcasts the injected address (with its own peer-id appended) to every peer that connects after the attacker.
- Receiving peers add this address to their peer stores and may attempt to connect to the attacker-controlled endpoint.
- This poisons peer discovery network-wide for as long as the attacker maintains the connection.
- The injected address also appears in the `hole_punching` protocol's `listen_addrs` broadcast: [7](#0-6) 

---

### Likelihood Explanation

- Requires only an inbound TCP connection — no special privileges, no PoW, no key material.
- The `observed_addrs` map is keyed by `session_id`; each connected attacker session injects one address. Multiple simultaneous attacker sessions inject multiple addresses.
- The entry is removed on disconnect (`observed_addrs.write().remove(&session_context.id)` at line 814–817 of `network.rs`), but the window is open for the entire session duration. [8](#0-7) 

---

### Recommendation

1. In `process_observed`, validate that the IP in `observed_addr` matches the actual remote IP of the session (`context.session.address`), or at minimum apply the same `is_reachable` filter used in `process_listens`.
2. Optionally, reject `observed_addr` values whose IP does not match the session's remote IP, since a peer can only legitimately observe the address it is actually connecting from.

---

### Proof of Concept

```
1. Attacker connects inbound to victim node (any valid CKB peer handshake).
2. Attacker sends IdentifyMessage with:
     observed_addr = /ip4/1.2.3.4/tcp/8114   (attacker-controlled public IP)
     listen_addrs  = []
     identify      = valid network identify bytes
3. Victim calls add_observed_addr(/ip4/1.2.3.4/tcp/8114, session_id),
   which appends /p2p/<victim_peer_id> and stores it in observed_addrs.
4. A second legitimate peer connects to the victim.
5. Victim's connected() calls local_listen_addrs(), which returns
   [... public_addrs ..., /ip4/1.2.3.4/tcp/8114/p2p/<victim_peer_id>].
6. The second peer receives the injected address in the victim's IdentifyMessage
   listen_addrs field and stores it in its peer store.
7. The second peer (and peers it gossips to) may attempt to connect to 1.2.3.4:8114.
```
