### Title
Unbounded DB Read Amplification via Crafted `GetLastStateProof` Difficulties Array — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged light-client peer can send a `GetLastStateProof` message with exactly `GET_LAST_STATE_PROOF_LIMIT` (1000) difficulty entries and `last_n_blocks=0`, bypassing the size guard, and force up to 1000 × O(log H) DB reads per request. Because a well-formed request returns `OK` (200), the peer is never banned and can repeat indefinitely.

### Finding Description
`GetLastStateProofProcess::execute` enforces a combined sample limit with:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // 1000
```

With `last_n_blocks=0` and `difficulties.len()=1000`, the expression is `1000 > 1000` = false, so the check passes. Each of the 1000 difficulty values is then processed by `get_block_numbers_via_difficulties`, which calls `get_first_block_total_difficulty_is_not_less_than` — a binary search that performs O(log H) calls to `get_block_total_difficulty`, each requiring two RocksDB reads (`get_block_hash` + `get_block_ext`). The `LightClientProtocol` handler has no rate limiter, and a successful response returns `Status::ok()` (code 200), which never triggers `should_ban()`.

### Impact Explanation
With a chain of height H = 10,000,000 (log₂ ≈ 23), a single request causes approximately 1000 × 23 × 2 ≈ 46,000 RocksDB point reads, plus additional reads in `complete_headers`. An attacker with a single persistent connection can flood the node with back-to-back requests, saturating disk I/O and starving the main chain processing loop, leading to node unresponsiveness or OOM from queued async tasks.

### Likelihood Explanation
The attack requires only a valid light-client connection and knowledge of the chain's total difficulty range (public on-chain data). No PoW, no key, no privileged role. The crafted message is a structurally valid `GetLastStateProof` that passes all validation checks.

### Recommendation
1. Change `>` to `>=` in the limit check, or count each difficulty entry as costing more than 1 unit (e.g., `log2(H)` units) toward the limit.
2. Add a per-peer rate limiter to `LightClientProtocol` analogous to the one in `Relayer` (`governor::RateLimiter` keyed by `(PeerIndex, message_item_id)`).
3. Consider capping the binary search range or caching total-difficulty lookups within a single request.

### Proof of Concept
```
1. Connect to a CKB node with light-client protocol enabled.
2. Obtain chain tip hash T, genesis hash G, and chain total difficulty D_max.
3. Build GetLastStateProof:
     last_hash        = T
     start_hash       = G
     start_number     = 0
     last_n_blocks    = 0
     difficulty_boundary = D_max + 1
     difficulties     = [D_max/1001, 2*D_max/1001, ..., 1000*D_max/1001]  // 1000 strictly increasing values
4. Send in a tight loop. Each iteration triggers ~46,000 DB reads and returns SendLastStateProof (OK). Peer is never banned.
```
