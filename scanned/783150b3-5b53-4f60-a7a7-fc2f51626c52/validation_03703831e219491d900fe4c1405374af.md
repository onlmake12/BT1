### Title
IBD Stall via TOCTOU Race Between `n_sync_started` Increment and Peer State Update in `start_sync_headers` — (File: sync/src/synchronizer/mod.rs)

---

### Summary

In `start_sync_headers`, the global `n_sync_started` counter is atomically incremented **before** the selected peer's state is updated to `Started`. If a peer disconnects in the window between these two non-atomic operations, `Peers::disconnected` sees the peer in `SyncProtocolConnected` (not `Started`) and skips the decrement, while `start_sync` is never called because the peer entry is already gone. The counter is permanently stuck at 1. In IBD mode, this prevents the node from ever starting header sync with any other peer.

---

### Finding Description

`start_sync_headers` in `sync/src/synchronizer/mod.rs` performs two logically coupled operations without holding any lock that spans both:

**Step 1 — atomic counter increment** (lines 675–682):
```rust
if self
    .shared()
    .state()
    .n_sync_started()
    .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
        if ibd && x != 0 { None } else { Some(x + 1) }
    })
    .is_err()
{
    break;
}
```

**Step 2 — peer state update** (lines 683–687):
```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
}
``` [1](#0-0) 

Between Step 1 and Step 2, a peer disconnect can be processed concurrently by `Peers::disconnected`:

```rust
pub fn disconnected(&self, peer: PeerIndex) {
    if let Some(peer_state) = self.state.remove(&peer)... {
        if peer_state.sync_started() {   // ← false: peer is SyncProtocolConnected, not Started
            self.n_sync_started.fetch_sub(1, ...);  // ← NOT reached
        }
    }
}
``` [2](#0-1) 

Because the peer is still in `SyncProtocolConnected` state (not `Started`) at the moment `disconnected` runs, `sync_started()` returns `false` and `n_sync_started` is **not** decremented. [3](#0-2) 

Back in `start_sync_headers`, `get_mut(&peer)` returns `None` (peer already removed), so `start_sync` is never called. The counter is now 1 with zero peers in `Started` state.

The only places `n_sync_started` is decremented are `suspend_sync`, `tip_synced`, and `disconnected` — all of which guard the decrement behind `peer_state.sync_started()`: [4](#0-3) 

Since no peer ever reaches `Started` state, the counter can never return to 0. There is no timeout or recovery path.

The IBD guard in `start_sync_headers` enforces exactly one sync peer:

```rust
if ibd && x != 0 { None } else { Some(x + 1) }
```

With `n_sync_started == 1` and IBD active, every subsequent call to `start_sync_headers` hits `is_err()` and `break`s immediately. The node is permanently unable to start header sync. [5](#0-4) 

The `HeadersSyncState` machine and the `Peers` struct confirm the state is never self-healing: [6](#0-5) [7](#0-6) 

The CKB CHANGELOG itself acknowledges a related race: *"#2894: There may be competition between header sync and eviction"*, and the test suite contains the comment *"There may be competition between header sync and eviction, it will cause assert panic"* — confirming the developers recognise this class of race in the sync state machine. [8](#0-7) 

---

### Impact Explanation

A malicious peer that connects and immediately disconnects at the right moment causes `n_sync_started` to be permanently stuck at 1 while no peer is in `Started` state. In IBD mode the node cannot start header sync with any peer. The node is isolated: it cannot download blocks, cannot advance its chain tip, and cannot leave IBD. The condition persists until the process is restarted. This is a complete denial of the node's ability to synchronise during Initial Block Download.

---

### Likelihood Explanation

The race window is narrow (a few instructions between the atomic `fetch_update` and the DashMap `get_mut`), but CKB uses a multi-threaded tokio runtime where the network disconnect handler and the sync timer genuinely run on separate threads concurrently. A malicious peer can repeatedly connect and disconnect to increase the probability of hitting the window. Because the stuck state is permanent (no recovery path), a single successful race is sufficient to stall the node indefinitely. The attack requires only an unprivileged TCP connection — no keys, no stake, no special role.

---

### Recommendation

After `get_mut` returns `None` (peer already gone), decrement `n_sync_started` to restore consistency:

```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
} else {
    // Peer disconnected between counter increment and state update; undo the increment.
    self.shared().state().n_sync_started().fetch_sub(1, Ordering::AcqRel);
    continue;
}
```

Alternatively, hold the DashMap entry lock across both the counter increment and the state update so that `disconnected` cannot interleave.

---

### Proof of Concept

1. Identify a CKB node in IBD (`is_initial_block_download() == true`).
2. Connect a malicious peer and complete the sync protocol handshake so the peer is inserted into `Peers::state` with `HeadersSyncState::SyncProtocolConnected`.
3. Wait for the sync timer to fire `start_sync_headers`. The peer satisfies `can_start_sync` and is selected. `n_sync_started.fetch_update` succeeds, incrementing the counter from 0 to 1.
4. Immediately disconnect the peer. `Peers::disconnected` removes the entry; `sync_started()` is `false`; `n_sync_started` is **not** decremented.
5. `get_mut(&peer)` in `start_sync_headers` returns `None`; `start_sync` is never called.
6. `n_sync_started` is now 1 with no peer in `Started` state.
7. Every subsequent `start_sync_headers` invocation evaluates `ibd && 1 != 0 → None → is_err() → break` and exits without starting sync.
8. The node remains in IBD indefinitely, unable to download blocks, until the process is restarted.
