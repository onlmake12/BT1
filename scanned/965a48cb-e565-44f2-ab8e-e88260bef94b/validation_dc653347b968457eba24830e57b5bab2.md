### Title
Unbounded Per-Peer RocksDB Read Amplification in `BlockFilter` Protocol Due to Missing Rate Limiter — (`sync/src/filter/get_block_filter_hashes_process.rs`)

---

### Summary

The `BlockFilter` protocol handler has no per-peer rate limiter. Any unprivileged remote peer can send `GetBlockFilterHashes` messages at maximum rate, each triggering up to ~4000 RocksDB reads. The `Relayer` protocol explicitly implements this protection; `BlockFilter` does not.

---

### Finding Description

**`BlockFilter` struct has no rate limiter field:** [1](#0-0) 

The struct holds only `shared: Arc<SyncShared>`. Compare with `Relayer`, which carries a `rate_limiter: RateLimiter<(PeerIndex, u32)>` and checks it at the top of `try_process` before dispatching any message: [2](#0-1) [3](#0-2) 

`BlockFilter::try_process` has no equivalent guard — it dispatches directly to the handler: [4](#0-3) 

**`GetBlockFilterHashesProcess::execute` performs up to ~4000 RocksDB reads per message:**

`BATCH_SIZE = 2000`. For each of up to 2000 blocks, the loop calls `get_block_hash` (one RocksDB read via `snapshot`) and `get_block_filter_hash` (one RocksDB read via `store`), totalling up to 4000 reads. With `start_number > 0`, two additional reads are made for the parent: [5](#0-4) [6](#0-5) 

Both `get_block_hash` and `get_block_filter_hash` on `ActiveChain` are direct storage reads with no caching layer that would absorb repeated identical queries: [7](#0-6) [8](#0-7) 

**The `received` handler calls `process` with no throttle:** [9](#0-8) 

The same gap applies to `GetBlockFilters` (BATCH_SIZE=1000, 2 reads/block) and `GetBlockFilterCheckPoints` (BATCH_SIZE=2000): [10](#0-9) [11](#0-10) 

---

### Impact Explanation

An attacker with N connections, each sending `GetBlockFilterHashes(start_number=0)` at maximum TCP throughput, generates `N × 4000` RocksDB reads per message cycle. RocksDB uses a shared I/O path for all column families. Saturating it with filter-hash reads starves concurrent block validation and sync reads, degrading block import latency proportionally to N. The node becomes unable to keep up with the chain tip, effectively causing local node degradation and sync stall.

The "whole-network congestion" framing in the question is overstated — the impact is local to the targeted node, not the entire network. However, targeted node degradation (sync stall, block validation latency) is a concrete, measurable impact.

---

### Likelihood Explanation

- The Filter protocol is an opt-in feature (`block_filter` must be enabled in config), which limits the exposed population to nodes that have enabled it.
- Any unprivileged peer can connect and send these messages; no authentication or PoW is required.
- The attack is trivially scriptable: open N connections, loop sending a 9-byte `GetBlockFilterHashes` message.
- The contrast with `Relayer`'s explicit 30 req/s cap per `(PeerIndex, item_id)` confirms this is an oversight, not an intentional design choice.

---

### Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`'s implementation) and check it at the top of `BlockFilter::try_process` before dispatching to any of the three handler processes. A cap of 30 req/s per `(peer, message_type)` pair, consistent with `Relayer`'s quota, would bound per-peer I/O cost.

---

### Proof of Concept

1. Enable `block_filter` on a CKB node and let it build filter data for ≥2000 blocks.
2. Open N TCP connections to the node's P2P port, each performing the Filter protocol handshake.
3. From each connection, send `GetBlockFilterHashes { start_number: 0 }` in a tight loop.
4. Monitor RocksDB read IOPS (via `ckb_metrics` or `rocksdb.stats`) and block import latency.
5. Assert: read IOPS scales linearly with N; block import latency degrades proportionally; the node falls behind the chain tip.
