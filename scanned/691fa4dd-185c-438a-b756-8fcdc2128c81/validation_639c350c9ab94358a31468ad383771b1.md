### Title
Missing Type Script `hash_type` Validation in `ScriptHashTypeVerifier` — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier`, the non-contextual check that enforces consensus-permitted `ScriptHashType` values, validates only the **lock script** of each output. It never inspects the **type script** `hash_type`. A transaction carrying an output whose type script uses a disallowed or unparseable hash-type byte passes the cheap, early rejection gate and is forwarded into the expensive contextual/script-execution pipeline before being rejected there — without the submitting peer being classified as malformed or banned.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `ScriptHashTypeVerifier::verify()` iterates over every output and checks only `output.lock().hash_type()`:

```rust
// lines 796–815
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

`output.type_()` is never consulted. The same gap exists in the block-level path: `NonContextualBlockTxsVerifier::verify()` calls `NonContextualTransactionVerifier` for every transaction in a block, which in turn calls this same `ScriptHashTypeVerifier`. [1](#0-0) 

The `NonContextualTransactionVerifier` comment explicitly documents the check as "Check whether output **lock** hash type within enabled range", confirming the type-script side is unguarded. [2](#0-1) 

`NonContextualBlockTxsVerifier` propagates the same gap to the block-processing path. [3](#0-2) 

---

### Impact Explanation

An unprivileged transaction sender submits a transaction whose output carries a type script with a hash-type byte that is either (a) a valid `ScriptHashType` variant not yet enabled by consensus, or (b) an unrecognised byte value. The transaction clears `NonContextualTransactionVerifier` (the cheap gate), is admitted to the tx-pool verify queue, and only fails during full contextual/script execution — the most CPU-intensive stage. Because the failure is not classified as `is_malformed_tx()`, the submitting peer is **not banned** (`after_process` only calls `ban_malformed` on malformed errors). The attacker's inputs are never consumed (the transaction is rejected, not committed), so the same UTXO set can be reused to flood the verify queue indefinitely, exhausting CPU in the script-execution pipeline. [4](#0-3) 

---

### Likelihood Explanation

Any node reachable via RPC (`send_transaction`) or the P2P relay protocol accepts transactions from unprivileged callers. Crafting an output with an arbitrary type-script `hash_type` byte requires no special privilege, no key material, and no majority hash power. The attacker retains their CKB (inputs are not spent on rejection), making repeated submission essentially free. [5](#0-4) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate `output.type_().to_opt()` when a type script is present, applying the same two-branch check (parse failure → `InvalidScriptHashType`, valid-but-not-enabled → `ScriptHashTypeNotPermitted`) that is already applied to lock scripts. This mirrors the existing pattern and ensures the consensus-permitted hash-type invariant is enforced uniformly at the cheapest possible verification stage. [6](#0-5) 

---

### Proof of Concept

1. Construct a `TransactionView` with at least one output whose **type script** `hash_type` field is set to a byte value absent from `ENABLED_SCRIPT_HASH_TYPE` (e.g., `0x05` if that value is not yet enabled).
2. Set the lock script to a valid, enabled hash type so `ScriptHashTypeVerifier` passes.
3. Submit via RPC `send_transaction` or inject via P2P relay.
4. Observe: `NonContextualTransactionVerifier::verify()` returns `Ok(())` — the transaction enters the verify queue.
5. Observe: the transaction is only rejected during `verify_rtx` (script execution stage), and the submitting peer receives no ban.
6. Repeat with the same UTXOs; the node re-executes scripts on each submission without banning the source. [7](#0-6)
