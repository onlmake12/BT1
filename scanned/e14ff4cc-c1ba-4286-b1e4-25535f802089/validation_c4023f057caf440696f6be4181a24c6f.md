Audit Report

## Title
Per-Session Rate Limiter Bypass via Reconnection — (`sync/src/relayer/mod.rs`, `network/src/protocols/hole_punching/mod.rs`)

## Summary
Both the `Relayer` and `HolePunching` protocol handlers maintain in-process rate limiters keyed by `(PeerIndex, u32)`, where `PeerIndex` is a session-scoped integer alias for `SessionId`. Because a reconnecting peer receives a new `SessionId`, its rate-limit bucket resets entirely on each reconnection. Since `TooManyRequests` (status code 110, a 1xx informational code) never triggers a ban, an attacker can cycle TCP connections to sustain relay message throughput far above the intended 30 req/sec cap with no IP-level consequence.

## Finding Description

**Root cause — session-keyed rate limiter:**

`PeerIndex` is defined as a type alias for `SessionId` in `network/src/protocols/mod.rs` (L33). Both protocol handlers initialize their rate limiters keyed on this per-session value:

- `Relayer`: `rate_limiter: RateLimiter<(PeerIndex, u32)>`, initialized with `RateLimiter::hashmap(quota)` in `Relayer::new()` (`sync/src/relayer/mod.rs` L81, L92).
- `HolePunching`: same pattern (`network/src/protocols/hole_punching/mod.rs` L45, L252).

**Rate-limit check does not ban:**

In `Relayer::try_process()` (`sync/src/relayer/mod.rs` L116–122), a rate-limit failure returns `StatusCode::TooManyRequests` (value 110). In `sync/src/status.rs` L165–180, `should_ban()` only returns `Some` for codes in the 400–499 range. `TooManyRequests = 110` is a 1xx code and therefore **never triggers a ban**. The same logic applies in `network/src/protocols/hole_punching/status.rs` L99–106.

**Disconnect does not remove the peer's bucket:**

On disconnect, both handlers call only `self.rate_limiter.retain_recent()` (`sync/src/relayer/mod.rs` L934; `network/src/protocols/hole_punching/mod.rs` L67–68). This prunes globally stale entries but does not remove the disconnecting peer's specific bucket. More importantly, the **new** session after reconnection starts with a completely fresh, full token bucket.

**Ban check is IP-keyed and never reached:**

The IP-level ban check in `network/src/peer_registry.rs` L109–111 (`peer_store.is_addr_banned(&remote_addr)`) is only triggered by explicit `ban_peer()` / `ban_session()` calls, which require a 4xx status. Rate-limit violations never reach this path.

**Exploit flow:**

The `governor` crate's token bucket starts full (30 tokens) on first use for a new key. An attacker connects (new `SessionId` → new key), immediately sends 30 relay messages consuming all tokens, gets silently dropped on the 31st, disconnects, and reconnects (new `SessionId` → fresh 30-token bucket). With TCP + secio handshake overhead of ~50 ms, the effective sustained throughput is approximately 30 messages per 50 ms = **~600 relay messages/sec**, versus the intended 30/sec cap — a ~20× bypass.

## Impact Explanation

Sustained relay message flooding at 20× the intended rate forces the victim node to perform repeated tx-pool lookups, message deserialization, and proposal/transaction processing for every message. This constitutes a low-cost, high-throughput resource exhaustion attack against a single CKB node, matching the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (resource exhaustion via CPU saturation and memory pressure from queued work), and *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (a flooded node falls behind on block relay, degrading its participation in the network).

## Likelihood Explanation

Any unprivileged peer reachable over TCP can exploit this. No special keys, hashpower, or protocol knowledge beyond standard CKB P2P is required. The default CKB configuration (`max_peers = 125`, `listen_addresses = ["/ip4/0.0.0.0/tcp/8115"]`) accepts inbound peers from the public internet. The attacker only needs to generate a fresh key pair per connection (trivial) to avoid the `PeerIdExists` check in `accept_peer()`. The attack is fully repeatable and requires no victim interaction.

## Recommendation

- Re-key the rate limiter by the peer's **IP address** (extracted from `connected_addr` / `session_context.address`) rather than by `PeerIndex`/`SessionId`. This mirrors how the ban list works and survives reconnection.
- Alternatively, on `disconnected`, explicitly remove the disconnecting peer's entries from the rate limiter hashmap using the known `PeerIndex` before the new session can claim a fresh bucket, and maintain a separate IP-keyed counter that persists across reconnections.
- Consider escalating repeated rate-limit violations from the same IP to a temporary IP-level ban via `peer_store.ban_addr()`, analogous to how malformed messages are handled.

## Proof of Concept

```
1. Attacker generates keypair_1, connects to victim (SessionId=7, PeerIndex=7).
2. Sends 30 GetRelayTransactions in rapid succession → all accepted (fresh bucket).
3. Sends 31st → StatusCode::TooManyRequests returned, no ban issued.
4. Attacker disconnects. victim calls retain_recent() only.
5. Attacker generates keypair_2, reconnects (SessionId=8, PeerIndex=8).
6. rate_limiter has no entry for (PeerIndex=8, item_id) → fresh 30-token bucket.
7. Sends 30 more GetRelayTransactions → all accepted. ✅
8. Repeat from step 3 indefinitely.

Effective rate: ~600 relay messages/sec vs. intended 30/sec cap.
```

To reproduce in a test: instantiate a `Relayer`, call `try_process` 31 times with `PeerIndex(1)` to exhaust the bucket, assert `TooManyRequests`, then call `disconnected` with `PeerIndex(1)`, then call `try_process` with `PeerIndex(2)` and assert the first call succeeds — demonstrating the fresh-bucket bypass.
