Audit Report

## Title
`valid_dao_withdraw_transaction()` Fires on Deposit Cells, Bypassing Capacity Check and Enabling CKB Inflation — (File: `verification/src/transaction_verifier.rs`)

## Summary

`CapacityVerifier::verify()` skips the `OutputsSumOverflow` check whenever any input carries the DAO type script, including phase-1 deposit cells (data = `[0u8; 8]`). Because `valid_dao_withdraw_transaction()` never inspects cell data, it returns `true` for deposit→prepare transactions, which are not withdrawals. The DAO type script (`dao.c`) only enforces capacity equality for the DAO cell pair and does not verify the overall transaction balance. An attacker who mixes a DAO deposit cell with regular cells in one transaction can inflate regular outputs beyond their inputs, creating CKB from nothing.

## Impact Explanation

When the bypass fires on a deposit→prepare transaction, the `OutputsSumOverflow` guard is skipped entirely. The DAO type script (`dao.c`) runs and verifies only that the output DAO cell carries the same capacity as the input DAO cell. Any regular (non-DAO) cells in the same transaction are completely unchecked for capacity conservation. An attacker can create CKB from nothing, repeatable at will. This is a direct, consensus-level CKB token inflation vulnerability.

**Impact: Critical — Vulnerabilities which could easily damage CKB economy.**

## Likelihood Explanation

Any unprivileged transaction sender can trigger this. The only prerequisite is owning a DAO deposit cell, which anyone can create by sending a standard DAO deposit transaction. No special keys, miner collusion, or Sybil attack is required. The crafted transaction is structurally valid and passes all other verifiers (script execution, since-field, etc.). Likelihood is **high**.

## Recommendation

`valid_dao_withdraw_transaction()` must be narrowed to return `true` only when at least one input is a phase-2 (prepare) DAO cell — i.e., a DAO type-script cell whose 8-byte little-endian data encodes a non-zero `deposited_block_number`. This mirrors the logic already present in `transaction_maximum_withdraw` in `util/dao/src/lib.rs`:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| {
            if !cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash) {
                return false;
            }
            // Only phase-2 cells (deposited_block_number > 0) are true withdrawal inputs.
            matches!(
                cell_meta.mem_cell_data.as_deref(),
                Some(d) if d.len() == 8 && LittleEndian::read_u64(d) > 0
            )
        })
}
```

Alternatively, pass a data loader into `CapacityVerifier` (as is already done in `DaoCalculator`) and reuse the same `deposited_block_number > 0` guard from `transaction_maximum_withdraw`.

## Proof of Concept

1. **Setup**: Attacker owns a DAO deposit cell `D` (100 CKB, DAO type script, data = `[0u8; 8]`) and a regular live cell `R` (50 CKB, no type script).

2. **Craft transaction**:
   - `inputs[0]` = `D` (DAO deposit cell, 100 CKB)
   - `inputs[1]` = `R` (regular cell, 50 CKB)
   - `outputs[0]` = DAO prepare cell (100 CKB, DAO type script, data = current block number) — satisfies `dao.c`
   - `outputs[1]` = regular cell (100 CKB, attacker's lock) — **50 CKB inflated**
   - Total inputs: 150 CKB; Total outputs: 200 CKB

3. **Verification path**:
   - `valid_dao_withdraw_transaction()` → `true` (inputs[0] has DAO type script; no data check)
   - `OutputsSumOverflow` guard → **skipped**
   - `dao.c` type script → verifies `outputs[0].capacity == inputs[0].capacity` (100 == 100) ✓
   - `DaoScriptSizeVerifier` → checks lock script sizes only, passes ✓
   - No verifier checks `outputs[1]` vs `inputs[1]`

4. **Result**: Transaction accepted by consensus. Attacker gains 50 CKB from nothing. Repeatable indefinitely.

A unit test can be written mirroring the existing `WithdrawDAOWithOverflowCapacity` integration test structure, but using a deposit cell (data = `[0u8; 8]`) as input alongside a regular cell, with inflated regular output — asserting the transaction is **rejected** (currently it would be accepted, demonstrating the bug).             .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))

                continue;

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,

            if input_data.into_iter().any(|b| b != 0) {
