Audit Report

## Title
Unauthenticated Block Template Injection via Miner Notify HTTP Server — (`miner/src/client.rs`)

## Summary
The miner's notify-mode HTTP server, started by `listen_block_template_notify`, accepts `BlockTemplate` payloads from any TCP client with no authentication, IP allowlisting, or source verification. Any attacker who can reach the configured listen port can POST a crafted `BlockTemplate` with an attacker-controlled coinbase lock script, causing all mining workers to immediately switch to mining the attacker's template and, upon finding a valid nonce, submit a block that pays the block reward to the attacker.

## Finding Description
`listen_block_template_notify` (`miner/src/client.rs`, L234–271) binds to the configured `SocketAddr` and dispatches every accepted TCP connection to the `handle` function. The peer address returned by `listener.accept()` is explicitly discarded at L245 (`let (stream, _) = match conn`), so no IP-based filtering is possible at the accept stage.

The `handle` function (L358–369) reads the request body, attempts to deserialize it as `BlockTemplate`, and unconditionally calls `client.update_block_template(template)` on success — with no `Authorization` header check, no shared-secret validation, and no peer-address comparison.

`update_block_template` (L293–312) contains a single guard:
```rust
let updated = |id| {
    if id != work_id || id == 0 {
        Some(work_id)
    } else {
        None
    }
};
```
This only rejects a new template whose `work_id` equals the current one (and is non-zero). Because `work_id` is a field in the attacker-supplied JSON, the attacker trivially bypasses it by supplying any value that differs from the current one. On success, `Works::New(work)` is sent to all worker threads via `new_work_tx`, immediately redirecting all hashpower.

`parse_authorization` (L380–394) exists in the same file but is used exclusively for **outgoing** RPC requests to the CKB node (L77), never for **incoming** requests on the notify server.

When a worker finds a valid nonce, `submit_nonce` in `miner/src/miner.rs` (L140–188) assembles the block from the injected template and submits it to the CKB node via `submit_block` — with the attacker's coinbase intact.

## Impact Explanation
A successfully mined block built from the injected template pays the full coinbase reward to the attacker's lock script. This constitutes direct, concrete theft of block rewards from legitimate miners and falls squarely within **"Vulnerabilities which could easily damage CKB economy" (Critical, 15001–25000 points)**. The attack requires no privileges, no keys, and no victim interaction beyond the miner being reachable on the network.

## Likelihood Explanation
Notify mode is a documented, production-supported feature. Operators running the CKB node and miner on separate hosts **must** bind the listen address to a non-loopback interface, making the port network-reachable. The `BlockTemplate` JSON schema is fully public. The exploit requires a single unauthenticated HTTP POST with no credentials, no reverse engineering, and no race condition. The attacker can re-inject continuously to keep the miner permanently redirected.

## Recommendation
Inside `handle`, verify the request originates from the configured CKB node address before calling `update_block_template`. The simplest correct fix is to pass the allowed origin IP (derived from `config.rpc_url`) into `handle` and reject connections whose peer address does not match. Alternatively, require a configurable bearer token in the `Authorization` header, mirroring the `parse_authorization` pattern already used for outgoing RPC calls. The peer address is available from `listener.accept()` at L244–245 and must be threaded through to `handle` rather than discarded.

## Proof of Concept
**Preconditions:** Miner running in notify mode with `listen` bound to a network-reachable address (required for any distributed node+miner setup).

```python
import requests

attacker_lock = {
    "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
    "hash_type": "type",
    "args": "0x<ATTACKER_PUBKEY_HASH>"
}

# Copy all fields from a real get_block_template response;
# replace cellbase output lock with attacker_lock;
# set work_id to any value != current (attacker controls this field).
fake_template = { ... }  # as described in the report

r = requests.post("http://<miner-listen-addr>/", json=fake_template)
assert r.status_code == 200
# Miner workers now mine attacker's template; next valid nonce submits
# a block paying block reward to attacker_lock.
```

**Verification:** After the POST, inspect the miner's worker log — `WorkerMessage::NewWork` will carry the injected template's `pow_hash`. The next `submit_block` RPC call will include a `cellbase` transaction paying the attacker's lock script.
