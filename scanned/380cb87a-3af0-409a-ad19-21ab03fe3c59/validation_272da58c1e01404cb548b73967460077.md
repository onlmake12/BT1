Audit Report

## Title
Insufficient RBF Fee Validation Before Expensive Script Execution Enables CPU Exhaustion - (File: `tx-pool/src/process.rs`)

## Summary

When a transaction's input resolves as `Dead` (conflicting with a pool tx), `pre_check` only validates `fee >= min_fee_rate` before returning `Ok`, allowing the transaction to proceed to full `ContextualTransactionVerifier` script execution. The actual RBF authorization check — including the much stricter `min_replace_fee` threshold — is deferred to `submit_entry` under write lock, which runs only after script verification completes. An unprivileged attacker can repeatedly submit low-fee RBF candidates that pass `pre_check`, consume full CKB-VM cycles, and are then rejected by `check_rbf`, at negligible cost to the attacker.

## Finding Description

In `_process_tx`, the execution order is:

1. `pre_check` (read lock) — L715
2. `verify_rtx` (full script execution) — L724–732
3. `submit_entry` → `check_rbf` (write lock) — L753, L106

In `pre_check`, the RBF branch (triggered by `OutPointError::Dead`) performs only two checks before returning `Ok`:

```rust
// process.rs L292-309
Err(Reject::Resolve(OutPointError::Dead(out))) => {
    let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
    let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;  // only min_fee_rate
    let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
    if conflicts.is_none() { ... return Err(...); }
    Ok((tip_hash, rtx, status, fee, tx_size))  // proceeds to script verification
}
```

`check_tx_fee` only enforces `fee >= min_fee_rate` (util.rs L45–52). After `pre_check` returns `Ok`, `verify_rtx` runs the full `ContextualTransactionVerifier` including `ScriptVerifier` (CKB-VM execution). Only after this expensive step does `submit_entry` call `check_rbf`, which enforces the real RBF rules:

- **Rule #2** (pool.rs L596–609): no new unconfirmed inputs
- **Rule #5** (pool.rs L611–645): ≤100 replacement candidates
- **Rule #3/#4** (pool.rs L664–676): `fee >= min_replace_fee = sum(all_replaced_txs.fee) + extra_rbf_fee`

The `min_replace_fee` threshold is categorically different from `min_fee_rate`. A transaction paying just above `min_fee_rate` (e.g., 363 shannons) but far below `min_replace_fee` (e.g., 10 CKB + extra) passes `pre_check`, triggers full VM execution, and is then rejected by `check_rbf` with `RBFRejected`. The node bears the full CPU cost; the attacker pays only the RPC/P2P submission cost.

## Impact Explanation

This is a **High** severity finding matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

The attacker forces the node to execute CKB-VM scripts (potentially millions of cycles, bounded only by `max_tx_verify_cycles`) for transactions that are guaranteed to be rejected. The attacker's cost is submitting messages with fee just above `min_fee_rate`; the node's cost is full script verification per transaction. This can be sustained to saturate the verification worker pool, degrading or halting legitimate transaction processing on the targeted node.

## Likelihood Explanation

- RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is a node operator configuration choice, not an attacker-controlled precondition.
- The attacker only needs to observe a transaction in the pool, which is publicly visible via the `get_raw_tx_pool` RPC or P2P relay.
- No privileged access is required — any RPC caller or P2P peer can submit transactions.
- The attack is repeatable: each iteration uses a fresh `tx_B` variant (different witness or output) to avoid `check_txid_collision` and the `txs_verify_cache` keyed on `witness_hash`.
- The verify queue size limit (`Reject::Full`) provides partial rate-limiting but does not prevent the attack — the attacker simply waits for the queue to drain and resubmits.

## Recommendation

Move the core RBF fee adequacy check (`min_replace_fee` comparison) and Rule #2 (no new unconfirmed inputs) into `pre_check` under the read lock, before the transaction is enqueued for script verification. These checks do not require write-lock atomicity — they only need a consistent snapshot read. The final conflict resolution and pool mutation must remain in `submit_entry` under write lock, but the fee-adequacy and input-validity checks can be evaluated cheaply upfront to gate access to the script verifier.

Specifically, in the `Err(Reject::Resolve(OutPointError::Dead(out)))` branch of `pre_check`, after `find_conflict_outpoint` confirms a conflict exists, compute the set of conflicting entries and verify `fee >= calculate_min_replace_fee(conflicts, tx_size)` before returning `Ok`.

## Proof of Concept

1. Submit `tx_A` spending `cell_X` with fee = 10 CKB. It enters the pending pool.
2. Observe `tx_A` via `get_raw_tx_pool` RPC.
3. Craft `tx_B` spending the same `cell_X`, with fee = `min_fee_rate * size` (e.g., 363 shannons — far below `min_replace_fee` of ~10 CKB + extra).
4. Submit `tx_B` via `send_transaction` RPC.
5. **`pre_check`**: `resolve_tx(..., rbf=true)` succeeds; `check_tx_fee` passes (363 ≥ min_fee); `find_conflict_outpoint` finds `tx_A` → returns `Ok`.
6. **`verify_rtx`**: Full `ContextualTransactionVerifier` runs CKB-VM script execution on `tx_B` (consuming up to `max_tx_verify_cycles`).
7. **`submit_entry` → `check_rbf`**: Rule #3/#4: `363 < 10_CKB + extra` → `Err(RBFRejected)`.
8. `tx_B` is placed in the conflicts pool; node wasted full VM cycles.
9. Repeat steps 3–8 with fresh `tx_B` variants (vary witness bytes to change `witness_hash` and bypass cache) until the verify queue is saturated, then wait and repeat.
