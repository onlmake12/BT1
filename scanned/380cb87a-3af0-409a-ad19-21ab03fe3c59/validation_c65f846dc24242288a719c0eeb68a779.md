Audit Report

## Title
Truncated 4-Byte Genesis Hash in `identify_name()` Allows Rogue Nodes to Pass Network Identity Check — (File: `spec/src/consensus.rs`)

## Summary
`identify_name()` constructs the P2P network identifier using only the first 8 hex characters (4 bytes) of the 32-byte genesis hash. Because `Identify::verify()` performs no other chain-membership check, an attacker who brute-forces a genesis block sharing the same 4-byte prefix can connect rogue nodes to mainnet, pass the identify handshake, and occupy connection slots — enabling targeted connection-slot exhaustion and network congestion.

## Finding Description
`identify_name()` in `spec/src/consensus.rs` at line 967 slices only the first 8 hex characters of the genesis hash:

```rust
format!("/{}/{}", self.id, &genesis_hash[..8])
```

The full 32-byte `genesis_hash` field is stored in `Consensus` but 28 bytes are discarded. The resulting string (e.g., `/ckb/92b197aa`) is the sole network-membership token passed to `IdentifyCallback`.

`Identify::verify()` in `network/src/protocols/identify/mod.rs` at lines 541–561 performs exactly one check — string equality of this truncated name:

```rust
if self.name != name {
    return None;
}
```

No other chain-identity check exists. If `verify()` returns `Some`, `received_identify()` at lines 384–454 accepts the peer unconditionally and opens all non-feeler protocols (sync, relay, block-filter, light-client) via `open_protocols`.

An attacker constructs a rogue spec with `name = "ckb"` (freely settable, as in `resource/specs/mainnet.toml`) and varies any genesis field (e.g., `genesis_cell.message`) until `blake2b_256(genesis_block)` shares the same leading 4 bytes as the mainnet genesis hash. This is a preimage search over 2^32 ≈ 4 billion candidates — trivially achievable on a GPU in seconds. The crafted spec is a one-time artifact; the attacker then runs arbitrarily many rogue nodes using it. Each rogue node passes the identify handshake against any mainnet node and occupies an inbound or outbound slot in `PeerRegistry` (bounded by `max_inbound`/`max_outbound`). Flooding these slots prevents legitimate peers from connecting.

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker running a botnet of rogue nodes (all sharing the same crafted genesis spec) can saturate the connection slots of targeted mainnet nodes, partitioning them from the honest network. Sync, relay, and block-filter protocols are opened against rogue peers, wasting CPU and memory on structurally valid but consensus-invalid messages before they are rejected downstream. The connection-slot exhaustion directly causes network congestion and can isolate individual nodes.

## Likelihood Explanation
The attacker requires no privileged access, no key material, and no majority hashpower. The only prerequisite is a one-time offline GPU computation (~2^32 blake2b operations, seconds to minutes). The crafted spec is reusable indefinitely. The attack is fully automatable, repeatable, and requires only commodity hardware and a network connection.

## Recommendation
Use the full 64-character genesis hash in `identify_name()`:

```rust
pub fn identify_name(&self) -> String {
    let genesis_hash = format!("{:x}", Into::<H256>::into(&self.genesis_hash));
    format!("/{}/{}", self.id, &genesis_hash)  // full 32-byte hash
}
```

This raises the preimage search to 2^256, making brute-force computationally infeasible and making the network identifier cryptographically unique per chain.

## Proof of Concept
1. Read the mainnet genesis hash from `docs/hashes.toml` (first 8 hex chars, e.g., `92b197aa`).
2. Write a script that iterates over `genesis.genesis_cell.message` values in a custom spec with `name = "ckb"`, computes `blake2b_256` of the resulting genesis block, and checks whether the first 4 bytes match `0x92b197aa`.
3. With GPU acceleration, this search completes in seconds (~2^32 blake2b operations).
4. Start a CKB node with the crafted spec. Connect it to a mainnet node (e.g., via `ckb run --config-file crafted.toml`).
5. Observe: the identify protocol exchange succeeds (`self.name == name` in `verify()`), `received_identify()` returns `MisbehaveResult::Continue`, and sync/relay protocols are opened — despite the node operating on a completely different chain.
6. Repeat with many rogue nodes to exhaust `max_inbound`/`max_outbound` slots on victim nodes, preventing legitimate peers from connecting.
