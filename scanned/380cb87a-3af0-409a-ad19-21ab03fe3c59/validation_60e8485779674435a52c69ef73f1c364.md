Audit Report

## Title
Unbounded Full-Pool Iteration in `get_raw_tx_pool` Stalls Tx-Pool Service — (`File: tx-pool/src/service.rs`, `rpc/src/module/pool.rs`)

## Summary
The `get_raw_tx_pool` RPC endpoint dispatches to `get_all_entry_info()` or `get_ids()`, both of which unconditionally iterate every entry in the tx-pool while holding the async read lock on the pool. Because the tx-pool service message loop is sequential, a large pool causes the service to stall for the full duration of the iteration, delaying all concurrently queued operations. The pool is bounded by bytes (default 180 MB), not transaction count, so the number of entries can reach into the hundreds of thousands.

## Finding Description
`get_raw_tx_pool` in `rpc/src/module/pool.rs` (L703–718) calls either `tx_pool.get_all_entry_info()` (verbose) or `tx_pool.get_all_ids()` (non-verbose) with no limit or pagination parameter. The RPC signature at L394–395 accepts only `verbose: Option<bool>`.

The service message handler at `tx-pool/src/service.rs` (L1000–1006) acquires `service.tx_pool.read().await` and then calls `tx_pool.get_all_entry_info()` synchronously before releasing the lock. `get_all_entry_info()` at `tx-pool/src/pool.rs` (L464–487) calls `score_sorted_iter_by_statuses` and `sorted_proposed_iter`, which traverse the entire multi-index map at `tx-pool/src/component/pool_map.rs` (L408–416) with no bound. The same pattern applies to `get_ids()` (L448–462) and to `get_tx_detail()` (L682–711), which calls `get_ids()` internally and then performs a second linear scan to compute `rank_in_pending`.

The default `max_tx_pool_size` is 180 MB (`resource/ckb.toml` L211). At the minimum fee rate of 1000 shannons/KB, filling 180 MB costs approximately 1.8 CKB, and at a minimum viable transaction size of ~200 bytes the pool can hold ~900,000 entries. Iterating and serializing all of them in a single synchronous call while holding the read lock blocks the service message loop for the entire duration.

## Impact Explanation
The impact is a temporary (or sustained if repeated) stall of the tx-pool service on a single node: `send_transaction`, `get_block_template`, and reorg-processing messages queue behind the long-running `GetAllEntryInfo` message. The node does not crash and the network is not affected; the service resumes normal processing once the iteration completes. This matches **Low (501–2000 points): Any other important performance improvements for CKB**. It does not meet the threshold for High (node crash or network congestion) because the node recovers automatically and the effect is local.

## Likelihood Explanation
By default the RPC binds to `127.0.0.1:8114` (localhost only, per `resource/ckb.toml` L182), so remote exploitation requires the operator to have exposed the RPC port. Locally, any process on the same machine can trigger it. Filling the pool to near capacity is cheap (~1.8 CKB at default fee rate). Repeated back-to-back calls can sustain the stall. The `get_max_update_time` helper (`tx-pool/src/component/pool_map.rs` L130–136), called on every `tx_pool_info` RPC, also iterates all entries, compounding the effect under load.

## Recommendation
1. Add `limit` and `offset` (or a cursor) parameters to `get_raw_tx_pool` and enforce a server-side cap on the number of entries returned per call.
2. Snapshot the pool IDs under the read lock, release the lock, then serialize outside it to avoid holding the lock during the full O(N) work.
3. For `get_tx_detail`, avoid calling `get_ids()` (full O(N) scan) just to compute `rank_in_pending`; maintain a rank index or restrict this endpoint to debug builds.
4. For `get_max_update_time`, maintain a running maximum updated on insert/remove rather than scanning all entries on every call.

## Proof of Concept
1. Configure a node with `max_tx_pool_size = 180_000_000` and `min_fee_rate = 1_000` (defaults).
2. Submit enough chained transactions via `send_transaction` to fill the pool to near capacity.
3. In a tight loop, call `get_raw_tx_pool(verbose=true)` from any RPC client with access to port 8114.
4. Concurrently call `send_transaction` and `get_block_template` and observe significantly increased response latency, because the tx-pool service message loop is occupied processing the unbounded `GetAllEntryInfo` iteration for each call.
