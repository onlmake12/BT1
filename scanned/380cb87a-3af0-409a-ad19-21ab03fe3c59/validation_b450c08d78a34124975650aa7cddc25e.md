Audit Report

## Title
Uncharged O(N) `header_deps` Linear Scan in `load_header` Syscall Breaks Cycle-to-CPU-Work Invariant — (`script/src/syscalls/load_header.rs`)

## Summary
The `load_header` syscall performs a full linear scan of `header_deps` in host Rust code on every invocation for `Source::Transaction(Input)` and `Source::Transaction(CellDep)`, but this scan is never charged VM cycles. When the scan exhausts all `header_deps` without a match, the syscall returns `ITEM_MISSING` via an early return that bypasses `add_cycles_no_checking` entirely. A malicious script can call the syscall in a tight loop, causing O(N × loop_count) host CPU work while consuming only O(loop_count) VM cycles, where N is the number of `header_deps`, breaking the invariant that the cycle limit bounds node CPU work.

## Finding Description

**Root cause — uncharged linear scan:**

In `load_header.rs`, the private `load_header` method performs a membership check by iterating the entire `header_deps` vector on every call: [1](#0-0) 

The comment at the top of the file explicitly acknowledges this is a linear-scan design: [2](#0-1) 

**Root cause — early return skips cycle charging:**

In the `ecall` handler, when `fetch_header` returns any error (including `ITEM_MISSING` produced when the linear scan finds no match), the handler sets the error register and returns immediately: [3](#0-2) 

The `add_cycles_no_checking` call that charges for transferred bytes is only reached on the success path, after the header is successfully fetched: [4](#0-3) 

The cost model charges only for bytes written to VM memory, not for host-side scan work: [5](#0-4) 

**No limit on `header_deps` count:**

A grep search for `MAX_HEADER_DEPS` or any `header_deps` count limit returns no matches anywhere in the codebase. The only enforced limit is the block byte limit: [6](#0-5) 

The `SizeVerifier` enforces this per-transaction byte limit but does not independently cap the number of `header_deps`: [7](#0-6) 

With `MAX_BLOCK_BYTES = 597 × 1000 = 597,000 bytes` and each `header_dep` being a 32-byte hash, a single transaction can carry approximately 18,000 `header_deps` before hitting the byte limit.

**Exploit flow:**

1. Attacker constructs a transaction with ~18,000 `header_deps` (valid on-chain block hashes) and one input whose `transaction_info.block_hash` is deliberately absent from `header_deps`.
2. A lock script calls `load_header(Source::Transaction(Input), 0)` in a tight loop until cycles are exhausted.
3. Each call: the VM executes a small fixed number of RISC-V instructions (charged cycles), then the host performs a full 18,000-element hash scan (uncharged).
4. With a 3.5 billion cycle budget and ~5 cycles per syscall, the script makes ~700 million calls, each triggering an 18,000 × 32-byte scan in host code — entirely invisible to the cycle counter.

## Impact Explanation

This breaks the fundamental invariant that the block cycle limit bounds the CPU time required to verify a block. Nodes with slower hardware will take disproportionately longer to verify such a block than the cycle limit implies. This can cause slower nodes to fall behind the chain tip or fail to validate blocks within their expected window, producing **network congestion with few costs** (High) and potentially **consensus deviation** (Critical) if nodes diverge on block acceptance timing. The attacker pays only normal transaction fees and PoW for a block; the amplification factor is O(N) per syscall invocation.

## Likelihood Explanation

The attack requires no privileged access. Any user can:
1. Submit a transaction via the standard `send_transaction` RPC or P2P relay.
2. Write a trivial RISC-V lock script that loops calling `load_header`.
3. Fill `header_deps` with valid on-chain block hashes (publicly available from any block explorer).
4. Ensure the input's `block_hash` is not among the `header_deps` to guarantee the ITEM_MISSING path is always taken.

All preconditions are attacker-controlled and reachable through normal transaction submission. The attack is repeatable across blocks.

## Recommendation

Charge cycles proportional to the number of `header_deps` scanned on every `load_header` invocation, regardless of whether the lookup succeeds — add a cycle charge of `transferred_byte_cycles(header_deps.len() * 32)` before the early return on the ITEM_MISSING path. Alternatively, replace the per-call linear scan with a `HashSet<Byte32>` built once per script group execution (charged upfront at O(N) cost), making each subsequent lookup O(1) with no additional uncharged work.

## Proof of Concept

1. Mine or obtain ~18,000 distinct valid block hashes `H[0..17999]` from the chain.
2. Construct a transaction with:
   - `header_deps = [H[0], ..., H[17999]]` (~576,000 bytes, within the 597,000-byte limit).
   - One input cell whose `transaction_info.block_hash` is a hash **not** in `header_deps`.
   - A lock script that executes: `loop { ckb_load_header(buf, &mut len, 0, 0, CKB_SOURCE_INPUT); }` until cycles are exhausted.
3. Submit the transaction and measure wall-clock verification time on a reference node.
4. Compare against a baseline transaction with 0 `header_deps` and the same cycle consumption.
5. Assert that wall-clock time for the attack transaction far exceeds the baseline, demonstrating that the cycle limit does not bound actual CPU time — the ratio should approach O(N) = ~18,000×.
