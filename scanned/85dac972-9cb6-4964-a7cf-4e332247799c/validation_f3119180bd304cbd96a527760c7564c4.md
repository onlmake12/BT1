### Title
Stale Descendant-Weight Tracking in `remove_entry_and_descendants` Corrupts Eviction Ordering — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` pre-clears all link relationships before calling `remove_entry` on each removed transaction. This causes `update_ancestors_index_key` — the function responsible for decrementing an ancestor's `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` — to silently no-op for every ancestor that remains in the pool. The result is that surviving ancestor entries carry permanently inflated descendant-weight fields, corrupting the `EvictKey` used to select which transactions are evicted when the pool is full.

---

### Finding Description

`remove_entry_and_descendants` is the bulk-removal path used during size-limit eviction, RBF conflict resolution, header-dep invalidation, and detached-proposal cleanup:

```rust
// tx-pool/src/component/pool_map.rs  lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← clears ALL link data for every removed id
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
``` [1](#0-0) 

The comment explains the intent: pre-clearing links prevents `update_descendants_index_key` from trying to update entries that are themselves being removed. However, it also silences `update_ancestors_index_key`, which is supposed to update entries that are **not** being removed.

Inside `remove_entry`:

```rust
// tx-pool/src/component/pool_map.rs  lines 235-250
pub(crate) fn remove_entry(&mut self, id: &ProposalShortId) -> Option<TxEntry> {
    self.entries.remove_by_id(id).map(|entry| {
        self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);  // ← no-ops: links already gone
        self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
        self.remove_entry_edges(&entry.inner);
        self.remove_entry_links(id);
        self.track_entry_statics(Some(entry.status), None);
        self.update_stat_for_remove_tx(entry.inner.size, entry.inner.cycles);
        entry.inner
    })
}
``` [2](#0-1) 

`update_ancestors_index_key` walks `self.links.calc_ancestors(short_id)` to find surviving ancestors and calls `sub_descendant_weight` on each:

```rust
// tx-pool/src/component/pool_map.rs  lines 432-445
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                EntryOp::Add    => e.inner.add_descendant_weight(child),
            };
            e.evict_key = e.inner.as_evict_key();   // ← EvictKey recomputed here
        });
    }
}
``` [3](#0-2) 

Because `remove_entry_links` already erased the root entry's link record, `calc_ancestors` returns an empty set and the loop body never executes. Every ancestor of the removed subtree root retains stale, inflated values for `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`.

By contrast, the single-entry removal path `remove_entry` (called directly, not via `remove_entry_and_descendants`) does perform the ancestor update correctly, because links are still intact at call time.

---

### Impact Explanation

The `EvictKey` for each pool entry is computed from the stale descendant fields:

```rust
// tx-pool/src/component/entry.rs  lines 234-247
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),   // ← inflated by stale data
            timestamp: entry.timestamp,
            descendants_count: entry.descendants_count,   // ← inflated by stale data
        }
    }
}
``` [4](#0-3) 

`EvictKey` ordering is ascending: the entry with the **smallest** key is evicted first.

```rust
// tx-pool/src/component/sort_key.rs  lines 92-103
impl Ord for EvictKey {
    fn cmp(&self, other: &Self) -> Ordering {
        if self.fee_rate == other.fee_rate {
            if self.descendants_count == other.descendants_count {
                self.timestamp.cmp(&other.timestamp)
            } else {
                self.descendants_count.cmp(&other.descendants_count)
            }
        } else {
            self.fee_rate.cmp(&other.fee_rate)
        }
    }
}
``` [5](#0-4) 

A surviving ancestor whose descendants were removed via `remove_entry_and_descendants` retains an artificially high `fee_rate` (from the removed descendants' fees) and an artificially high `descendants_count`. Both make its `EvictKey` larger than it should be, so it is evicted **later** than warranted. A low-fee ancestor transaction that should be the first candidate for eviction is instead protected, occupying pool space that should be available to legitimate higher-fee transactions.

`limit_size` — the function that enforces the pool size cap — relies entirely on `next_evict_entry` / `EvictKey` ordering:

```rust
// tx-pool/src/pool.rs  lines 292-328

            let removed = self.pool_map.remove_entry_and_descendants(&id);
            ...
        }
    }
}
``` [6](#0-5) 

Corrupted eviction ordering means the pool can be kept artificially full of low-fee transactions, degrading throughput for honest users and enabling a sustained resource-exhaustion attack.

---

### Likelihood Explanation

The trigger path is fully reachable by an unprivileged transaction sender:

1. Submit parent transaction **A** with a low fee rate.
2. Submit child transaction **B** (spending an output of A) with a high fee rate. This inflates A's `descendants_fee`, `descendants_cycles`, and `descendants_count`, raising A's `EvictKey`.
3. Submit a conflicting transaction **B′** that spends the same input as B with a fee high enough to satisfy RBF rules. `resolve_conflict` calls `remove_entry_and_descendants(&B_id)`, removing B but leaving A's descendant fields stale.
4. A now has an `EvictKey` that reflects B's high fee rate even though B is gone. A is effectively shielded from eviction.
5. Repeat steps 2–4 with new children to keep A's `EvictKey` perpetually inflated.

RBF is enabled whenever `min_rbf_rate > min_fee_rate` in the node configuration, which is the recommended production setting. The same stale-state condition is also triggered by `resolve_conflict_header_dep`, `remove_by_detached_proposal`, and `remove_tx`, all of which call `remove_entry_and_descendants`. [7](#0-6) 

---

### Recommendation

Before pre-clearing links, snapshot the direct parents of the root entry (those that will survive the removal), then explicitly call `sub_descendant_weight` on each survivor for every entry in `removed_ids`:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Collect surviving ancestors of the root BEFORE clearing links
    let surviving_ancestors: HashSet<ProposalShortId> = self
        .links
        .calc_ancestors(id)
        .into_iter()
        .filter(|anc| !removed_ids.contains(anc))
        .collect();

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Update surviving ancestors' descendant weights
    for removed_entry in &removed {
        for anc_id in &surviving_ancestors {
            self.entries.modify_by_id(anc_id, |e| {
                e.inner.sub_descendant_weight(removed_entry);
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    removed
}
```

Alternatively, defer `remove_entry_links` to inside `remove_entry` (its original position) and instead guard `update_descendants_index_key` with a membership check against `removed_ids` to avoid updating entries that are themselves being removed.

### Proof of Concept

**Setup:** pool with `max_tx_pool_size` near capacity; `min_rbf_rate > min_fee_rate`.

1. Submit tx **A** (low fee, e.g. 1 shannon/byte). A enters pool; `A.descendants_count = 1`, `A.EvictKey.fee_rate = low`.
2. Submit tx **B** spending A's output (high fee, e.g. 1000 shannon/byte). `pool_map.add_entry` calls `record_entry_descendants` → `update_ancestors_index_key(B, Add)` → `A.descendants_count = 2`, `A.descendants_fee += B.fee`, `A.EvictKey.fee_rate = high`.
3. Submit tx **B′** conflicting with B, fee > B.fee (satisfies RBF). `resolve_conflict` → `remove_entry_and_descendants(&B_id)`:
   - `remove_entry_links(B_id)` clears B's link record.
   - `remove_entry(B_id)` → `update_ancestors_index_key(B, Remove)` → `calc_ancestors(B_id)` returns `{}` (links gone) → **A is not updated**.
   - A still has `descendants_count = 2`, `descendants_fee = A.fee + B.fee`, `EvictKey.fee_rate = high`.
4. Pool fills up. `limit_size` calls `next_evict_entry` → iterates `iter_by_evict_key()` ascending. A's inflated `EvictKey` places it after genuinely high-fee transactions. A is not evicted despite having a low own fee rate.
5. Repeat steps 2–3 indefinitely to keep A's `EvictKey` perpetually inflated at negligible net cost (attacker pays only the RBF fee delta). [1](#0-0) [3](#0-2) [4](#0-3) [6](#0-5)
