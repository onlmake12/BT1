### Title
Unbounded GetHeaders Response Amplification via Genesis-Hash Locator — (`sync/src/synchronizer/get_headers_process.rs`)

### Summary
An unprivileged remote peer can send a `GetHeaders` message containing a single-entry locator equal to the genesis hash. The `Synchronizer` has no per-peer rate limiting on incoming `GetHeaders` messages. Each 32-byte request reliably elicits a `SendHeaders` response of up to 2,000 headers (~384 KB), yielding a ~12,000× bandwidth amplification ratio that can be sustained in a tight loop.

### Finding Description
`GetHeadersProcess::execute()` validates only that `locator_size ≤ MAX_LOCATOR_SIZE` (101) and that the node is not in IBD. It then calls `locate_latest_common_block`, which requires the last locator entry to equal the genesis hash. With a single-entry locator `[genesis_hash]`, the function immediately hits the `index == 0` early-return path and returns `Some(0)`. `get_locator_response(0, hash_stop)` then iterates from block 1 up to `min(tip, MAX_HEADERS_LEN=2000)` and returns all headers. The resulting `SendHeaders` message is sent unconditionally with no throttle.

The `Relayer` protocol has a 30 req/sec rate limiter keyed by `(peer, message_type)`, but the `Synchronizer` protocol — which handles `GetHeaders` — has no equivalent guard. The `pending_get_headers` LRU cache only throttles the node's own *outgoing* `GetHeaders` requests; it is never consulted when processing *incoming* ones.

### Impact Explanation
- **Bandwidth exhaustion**: A single attacker connection can generate ~384 KB of outbound traffic per message. At network line speed, a single peer can saturate the victim's uplink.
- **CPU/DB amplification**: Each request triggers up to 2,000 `get_block_hash` + `get_block_header` store lookups.
- **Multi-peer scaling**: An attacker controlling multiple IPs (or exploiting the inbound connection limit) multiplies the effect linearly.
- **No PoW or stake required**: The genesis hash is public knowledge; no block production is needed.

### Likelihood Explanation
The attack is trivially executable: connect to any non-IBD CKB node, send `GetHeaders([genesis_hash])` in a loop. The genesis hash is hardcoded in the chain spec and publicly known. No authentication, no PoW, no special peer status is required.

### Recommendation
1. Add a per-peer rate limiter to `Synchronizer::try_process()` for `GetHeaders` messages, analogous to the existing `Relayer` rate limiter.
2. Alternatively, track the last `GetHeaders` response timestamp per peer and enforce a minimum inter-response interval (e.g., 1 second).
3. Consider whether a locator consisting solely of the genesis hash should be treated as a protocol violation (it conveys no useful sync information from a well-behaved peer).
