### Title
DAO Withdrawal `header_dep_index` Interpreted as Full `u64` in Rust but as Lowest Byte in On-Chain C Script — (`File: util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the `header_dep_index` stored in the witness `input_type` field as a full 8-byte `u64`, while the on-chain DAO C script reads only the **lowest byte** of the same field. For any DAO withdrawal transaction where the witness encodes an index value greater than 255, the two components resolve to **different deposit header hashes**, producing a semantic mismatch between the Rust node's fee/capacity accounting and the authoritative on-chain script enforcement.

---

### Finding Description

In `util/dao/src/lib.rs`, `DaoCalculator::transaction_maximum_withdraw` extracts the deposit header index from the witness as follows:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The full 8-byte little-endian `u64` is read and used directly as the `header_deps` array index. [1](#0-0) 

The on-chain DAO C script, however, reads only the **lowest byte** of the same `input_type` witness field as the index. This is explicitly documented in the test suite:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
// input_type = 257, lowest byte = 1
``` [2](#0-1) 

For a witness encoding index `257` (little-endian `u64`):
- **Rust node** resolves `header_deps[257]`
- **C script** resolves `header_deps[1]` (lowest byte of `257` = `0x01`)

These are different entries, so the two components compute different deposit headers, different AR ratios, and therefore different maximum withdrawal capacities.

The `CapacityVerifier` explicitly **skips** the `OutputsSumOverflow` check for DAO withdrawal transactions, delegating enforcement entirely to the DAO type script:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // ... OutputsSumOverflow check ...
}
``` [3](#0-2) 

Meanwhile, the tx-pool's `check_tx_fee` uses `DaoCalculator::transaction_fee` (which calls `transaction_maximum_withdraw`) for fee validation:

```rust
let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
    .transaction_fee(rtx)
    .map_err(|err| {
        Reject::Malformed(format!("{err}"), "expect (outputs capacity) <= (inputs capacity)".to_owned())
    })?;
``` [4](#0-3) 

---

### Impact Explanation

When `header_dep_index > 255`, the Rust `DaoCalculator` and the C script resolve to different deposit headers. Two concrete consequences follow:

**1. Valid DAO withdrawal transactions rejected by the tx-pool.**
If the Rust-computed `max_withdraw` is lower than the C-script-computed `max_withdraw`, a user who sets their output capacity to the C-script-allowed amount will have their transaction rejected by `check_tx_fee` with a `DaoError::Overflow` (negative fee). The transaction is valid on-chain but the Rust node refuses to admit it. The user's DAO deposit becomes inaccessible through the normal submission path.

**2. Incorrect fee accounting for admitted transactions.**
If the Rust-computed `max_withdraw` is higher than the C-script-computed `max_withdraw`, the Rust node calculates an inflated fee. The transaction may be admitted and included in a block, but the fee recorded by the node does not reflect the actual economic transfer. This corrupts the fee-rate estimation and block reward accounting.

---

### Likelihood Explanation

Exploiting this requires a DAO withdrawal transaction with `header_dep_index > 255`, meaning the transaction must reference more than 255 header deps. While unusual in typical usage, it is protocol-valid and reachable by any unprivileged transaction sender via the `send_transaction` RPC. The discrepancy is already documented in the test suite (`check_dao_withdraw_header_dep_index_exceeds_u8`), confirming the developers are aware the two interpretations diverge. A user with a large multi-input DAO withdrawal, or an attacker deliberately constructing such a transaction to grief a target miner or block a victim's withdrawal, can trigger this path.

---

### Recommendation

Align the Rust `DaoCalculator` with the on-chain C script's interpretation. If the C script reads only the lowest byte of the `input_type` witness field as the index, the Rust code should do the same:

```rust
// Read only the lowest byte, matching the C script's behavior
let index_byte = header_deps_index_data.unwrap()[0];
Ok(index_byte as u64)
```

Alternatively, if the intent is to support full `u64` indices, the on-chain C script must be updated to match. The two components must agree on the same interpretation of the witness field. Add a consensus-level bound check rejecting any `header_dep_index` that would cause the two interpretations to diverge (i.e., reject indices where `full_u64 != lowest_byte_u64`).

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` directly demonstrates the discrepancy:

1. Construct a DAO withdrawal transaction with 258 `header_deps`.
2. Place the deposit block hash at `header_deps[1]` (what the C script resolves for index `257`).
3. Place the withdraw block hash at `header_deps[257]` (what Rust resolves for index `257`).
4. Set witness `input_type` to `257u64` in little-endian.
5. Call `DaoCalculator::transaction_fee`.

Rust resolves `header_deps[257]` (the withdraw block) as the deposit header, which mismatches the cell data's recorded deposit block number, causing `DaoError::InvalidOutPoint`. The C script would resolve `header_deps[1]` (the actual deposit block) and accept the transaction. A legitimate user in this configuration cannot submit their withdrawal through the Rust node's tx-pool. [5](#0-4)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
