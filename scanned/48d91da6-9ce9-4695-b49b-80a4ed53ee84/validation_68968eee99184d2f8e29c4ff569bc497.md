Audit Report

## Title
VerifyQueue Enforces Hardcoded 256 MB Ceiling Independent of Operator-Configured `max_tx_pool_size`, Enabling Unbounded Memory Consumption — (`tx-pool/src/component/verify_queue.rs`)

## Summary
The CKB tx-pool two-stage admission pipeline stages transactions in a `VerifyQueue` before moving them to `pool_map`. The verify queue enforces a hardcoded 256 MB ceiling (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) that is entirely independent of the operator-configured `max_tx_pool_size`. The main pool's `limit_size()` eviction loop has no awareness of the verify queue, and `TxPoolInfo.total_tx_size` excludes verify-queue bytes. Any unprivileged RPC caller or P2P peer can flood the verify queue up to 256 MB regardless of the operator's memory budget, potentially causing OOM on constrained nodes.

## Finding Description

**Hardcoded ceiling:**
`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is a compile-time constant of 256 MB, with a comment acknowledging the default `max_tx_pool_size` is 180 MB: [1](#0-0) 

**`is_full()` checks only the hardcoded constant:** [2](#0-1) 

**`add_tx()` gates on `is_full()` alone:** [3](#0-2) 

**`limit_size()` only iterates `pool_map`, never the verify queue:** [4](#0-3) 

**`TxPoolInfo` reports verify-queue size as a count only, not bytes; `total_tx_size` is `pool_map`-only:** [5](#0-4) 

**`TRANSACTION_SIZE_LIMIT` is 512 KB per transaction:** [6](#0-5) 

**`non_contextual_verify` (the only pre-queue gate) checks only size ≤ 512 KB and non-cellbase:** [7](#0-6) 

The combined memory budget at any moment is: `max_tx_pool_size` (operator-controlled) + 256 MB (hardcoded verify queue) + orphan pool. If an operator sets `max_tx_pool_size = 10 MB` for a constrained node, the verify queue alone can consume 256 MB — 25× the intended budget — with no eviction mechanism to drain it.

## Impact Explanation

This maps to **High: Vulnerabilities which could easily crash a CKB node (10001–15000 points)**. On a memory-constrained node configured with a small `max_tx_pool_size`, an attacker can force RSS growth of up to 256 MB via the verify queue alone. Because script verification is CPU-bound and asynchronous, transactions linger in the queue while consuming memory. On constrained nodes this triggers OOM-kill, halting block production and P2P connectivity. The operator-visible `total_tx_size` metric shows no alarm signal because verify-queue bytes are excluded from it. [1](#0-0) 

## Likelihood Explanation

No privilege is required. Any RPC caller (`send_transaction`) or P2P peer can submit transactions. The only pre-queue check is `non_contextual_verify`, which passes any syntactically valid transaction ≤ 512 KB that is not a cellbase. An attacker needs ~500 transactions of ~512 KB each (256 MB / 512 KB) to saturate the queue. This requires no keys, no hashpower, and no Sybil capability — only the ability to craft large valid transactions and submit them in rapid succession. [8](#0-7) [6](#0-5) 

## Recommendation

1. Replace the hardcoded `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` with a value derived from `TxPoolConfig::max_tx_pool_size` (e.g., `max_tx_pool_size` or `max_tx_pool_size * 1.5`), so the verify queue respects the operator's memory budget.
2. Include `verify_queue.total_tx_size` in the combined size accounting so that `limit_size()` can account for in-flight transactions when deciding whether to accept new ones.
3. Alternatively, enforce a per-peer rate limit on verify-queue submissions to bound the rate at which any single source can fill the staging area. [9](#0-8) 

## Proof of Concept

1. Configure a CKB node with `max_tx_pool_size = 10_000_000` (10 MB) in `ckb.toml`.
2. Generate ~500 syntactically valid transactions each close to 512 KB (large witness data, valid but slow-to-verify lock scripts).
3. Submit all transactions via `send_transaction` RPC in rapid succession.
4. Query `tx_pool_info`: observe `verify_queue_size` grows to ~500 while `total_tx_size` remains near 0 (transactions are still being verified).
5. Monitor node RSS: it grows by ~256 MB — 25× the configured `max_tx_pool_size` — while the operator-visible `total_tx_size` metric shows no alarm signal.
6. On a node with ≤ 512 MB total RAM, this triggers OOM-kill. [2](#0-1) [5](#0-4)

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
