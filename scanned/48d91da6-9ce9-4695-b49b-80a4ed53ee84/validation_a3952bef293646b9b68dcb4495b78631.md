Audit Report

## Title
Unbounded Full-Pool Iteration in `get_raw_tx_pool` Stalls Tx-Pool Service Message Loop — (`File: rpc/src/module/pool.rs`)

## Summary
The `get_raw_tx_pool(verbose=true)` RPC handler unconditionally iterates every entry in the tx-pool and serializes it into a single allocation before returning. Because the tx-pool service is a sequential async message loop, processing this message holds the service busy for the entire duration of the iteration, delaying all other queued operations including transaction submission, block template assembly, and reorg handling. No pagination, server-side cap, or rate limit exists on this endpoint.

## Finding Description
`get_raw_tx_pool` at `rpc/src/module/pool.rs:703-718` dispatches to `TxPoolController::get_all_entry_info()` (verbose) or `get_all_ids()` (non-verbose). Both paths are unconditional full-pool scans.

The `Message::GetAllEntryInfo` handler at `tx-pool/src/service.rs:1000-1006` acquires the async read lock and calls `tx_pool.get_all_entry_info()` synchronously before releasing it. The tx-pool service is a single sequential message loop — while this handler runs, no other message (including `SubmitLocalTx`, `GetBlockTemplate`, `SubmitRemoteTx`) can be dequeued and processed.

`TxPool::get_all_entry_info()` at `tx-pool/src/pool.rs:464-487` calls `score_sorted_iter_by_statuses` and `sorted_proposed_iter`, both of which traverse the entire multi-index map at `tx-pool/src/component/pool_map.rs:408-416` with no limit. All results are materialized into a `HashMap`/`Vec` before returning.

The pool is bounded by `max_tx_pool_size` in bytes (`util/app-config/src/configs/tx_pool.rs:13`), not by transaction count. At minimum viable transaction sizes, the pool can hold a very large number of entries. No rate limiting, pagination parameter, or server-side cap exists anywhere in the call chain — confirmed by the absence of any `rpc_rate_limit`, `RateLimit`, or `throttle` pattern in the codebase.

A secondary instance exists in `get_tx_detail` at `tx-pool/src/pool.rs:682-711`, which calls `get_ids()` (full O(N) scan) just to compute `rank_in_pending`, then performs a second linear scan of all pending IDs, plus unbounded `calc_descendants` and `calc_ancestors` traversals.

The `get_max_update_time` helper at `tx-pool/src/component/pool_map.rs:130-136`, called on every `tx_pool_info` RPC, also iterates all entries unconditionally.

## Impact Explanation
This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. An attacker with access to the RPC port (which operators commonly expose for wallet and tooling integrations) can issue back-to-back `get_raw_tx_pool(verbose=true)` calls at zero marginal cost per call. Each call occupies the tx-pool service's message loop for the full duration of the iteration and JSON serialization. With a sufficiently large pool (which can be grown by organic network activity or by the attacker submitting minimum-fee transactions), this keeps the service continuously busy, delaying block template assembly on mining nodes and stalling transaction relay processing, which degrades the node's participation in the network.

## Likelihood Explanation
The `get_raw_tx_pool` RPC is a standard documented endpoint. No authentication is required. Any caller with TCP access to the RPC port can trigger this. The pool can be grown by organic network activity without attacker involvement. The attacker's only cost is network access and the RPC call itself, which is free. The attack is trivially repeatable in a tight loop.

## Recommendation
1. Add `limit` and `offset` (or a cursor) parameters to `get_raw_tx_pool` and enforce a server-side maximum entries per call.
2. Move full-pool serialization off the service's main message loop: snapshot only the IDs under the read lock, release the lock, then serialize outside it.
3. For `get_tx_detail`, avoid calling `get_ids()` (full O(N) scan) to compute `rank_in_pending`; maintain a rank index or restrict this endpoint to debug builds only.
4. Replace `get_max_update_time`'s full iteration with a maintained high-watermark timestamp updated incrementally on entry insertion/removal.

## Proof of Concept
1. Connect to a CKB node's RPC port (default 8114).
2. Submit many low-fee transactions via `send_transaction` to grow the tx-pool toward `max_tx_pool_size`.
3. In a tight loop, call `get_raw_tx_pool` with `verbose: true`.
4. Concurrently call `get_block_template` and `send_transaction` and measure response latency.
5. Observe that concurrent calls experience significantly increased latency proportional to pool size and call frequency, because the tx-pool service message loop is occupied processing each unbounded `GetAllEntryInfo` message before it can dequeue the next message.

        Ok(raw)
    }

    pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let conflicted = self
            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {
            pending,
            proposed,
            conflicted,
        }
    }

        if let Some(entry) = self.pool_map.get_by_id(id) {
            let ids = self.get_ids();
            let rank_in_pending = if entry.status == Status::Proposed {
                0
            } else {
                let tx_hash = entry.inner.transaction().hash();
                ids.pending
                    .iter()
                    .enumerate()
                    .find(|(_, hash)| &tx_hash == *hash)
                    .map(|r| r.0)
                    .unwrap_or_default()
                    + 1
            };
            let res = PoolTxDetailInfo {
                timestamp: entry.inner.timestamp,
                entry_status: entry.status.to_string(),
                pending_count: self.pool_map.pending_size(),
                rank_in_pending,
                proposed_count: ids.proposed.len(),
                descendants_count: self.pool_map.calc_descendants(id).len(),
                ancestors_count: self.pool_map.calc_ancestors(id).len(),
                score_sortkey: entry.inner.as_score_key().into(),
            };
            Some(res)

            None
        }
    }
