### Title
Unweighted Arithmetic Mean Used Instead of Weight-Proportional Mean in `get_fee_rate_statistics` RPC — (File: `rpc/src/util/fee_rate.rs`)

---

### Summary

The `get_fee_rate_statistics` (and its deprecated alias `get_fee_rate_statics`) RPC computes the `mean` fee rate using a simple arithmetic mean of per-transaction fee rates, dividing by transaction **count** instead of by total transaction **weight**. This is the same class of wrong-variable substitution as the reported `avgDepositPrice` bug: the wrong denominator is used in an averaging formula, producing a systematically skewed result that any unprivileged tx-pool submitter can manipulate.

---

### Finding Description

In `rpc/src/util/fee_rate.rs`, the `statistics()` method collects per-transaction fee rates and then computes the mean:

```rust
fn mean(numbers: &[u64]) -> u64 {
    let sum: u128 = numbers.iter().map(|number| u128::from(*number)).sum();
    (sum / numbers.len() as u128) as u64   // denominator = transaction COUNT
}
``` [1](#0-0) 

Each element pushed into `fee_rates` is already a per-transaction fee rate:

```rust
let weight = get_transaction_weight(*size as usize, cycles);
if weight > 0 {
    fee_rates.push(FeeRate::calculate(fee, weight).as_u64());
}
``` [2](#0-1) 

So the returned `mean` is:

```
mean = Σ( fee_i × 1000 / weight_i ) / N
```

The correct weight-proportional mean fee rate — the one that actually represents the cost to fill block space — is:

```
correct_mean = Σ(fee_i) × 1000 / Σ(weight_i)
```

The wrong variable is `N` (transaction count) in the denominator where `Σ(weight_i)` (total weight) should be used. This is structurally identical to the reported bug: `sendAmt` (ZeUSD) used where `ethAmt` (ETH) should have been used.

The result is exposed directly to any RPC caller via `get_fee_rate_statistics`:

```rust
fn get_fee_rate_statistics(&self, target: Option<Uint64>) -> Result<Option<FeeRateStatistics>> {
    Ok(FeeRateCollector::new(self.shared.snapshot().as_ref())
        .statistics(target.map(Into::into)))
}
``` [3](#0-2) 

---

### Impact Explanation

The `mean` field of `FeeRateStatistics` is documented as the mean fee rate in shannons per kilo-weight, used by wallets and users to calibrate transaction fees. [4](#0-3) 

Because the formula weights every transaction equally regardless of its byte-size or cycle cost, a single tiny 1-byte transaction with a fee rate of 1,000,000 shannons/KW has the same influence on the reported mean as a 10 KB transaction with a fee rate of 1,000 shannons/KW. An attacker who gets many small high-fee-rate transactions confirmed can inflate the reported mean arbitrarily, causing honest users to overpay fees. Conversely, an attacker flooding with large zero-fee-rate transactions deflates the mean, causing users to underpay and have transactions stuck.

---

### Likelihood Explanation

The attack entry path is fully unprivileged: any node participant can submit transactions to the tx-pool via `send_transaction` RPC, have them confirmed in blocks, and thereby influence the historical data consumed by `FeeRateCollector`. The cost of the attack is bounded only by the fees paid for the manipulating transactions themselves, which for tiny transactions is negligible. The RPC is enabled by default on all CKB nodes.

---

### Recommendation

Replace the simple arithmetic mean with a weight-proportional mean. Accumulate total fees and total weight across all transactions, then compute:

```rust
// Accumulate (total_fee_shannons, total_weight) instead of a Vec<fee_rate>
let mean_fee_rate = total_fee_shannons.saturating_mul(1000) / total_weight;
```

This mirrors `FeeRate::calculate` semantics and is immune to manipulation by small-transaction flooding. [5](#0-4) 

---

### Proof of Concept

**Setup:** Two blocks, each with two transactions:

| Tx | fee (shannons) | weight (bytes) | fee_rate (shannons/KW) |
|----|---------------|----------------|------------------------|
| A  | 1,000         | 1,000          | 1,000                  |
| B  | 100           | 1              | 100,000                |

**Current (buggy) mean:**
```
(1,000 + 100,000) / 2 = 50,500 shannons/KW
```

**Correct weight-proportional mean:**
```
(1,000 + 100) × 1000 / (1,000 + 1) ≈ 1,099 shannons/KW
```

The reported mean is **~46× higher** than the true cost of block space. A wallet using the RPC's `mean` field to set fees would overpay by a factor of 46. An attacker who controls transaction B (a 1-byte transaction) can tune this ratio arbitrarily by adjusting the fee on their tiny transaction, with no privileged access required. [6](#0-5)
