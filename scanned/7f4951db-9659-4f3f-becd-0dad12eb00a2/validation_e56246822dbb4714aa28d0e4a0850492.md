Audit Report

## Title
Missing `request_limit` Guard in `get_cells_capacity` Enables Unbounded Full-Table Aggregation Scan — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`)

## Summary
`AsyncRichIndexerHandle::get_cells_capacity` performs an unbounded `SUM(output.capacity)` aggregation over the entire live-cell set with no row-count or limit guard, while the sibling methods `get_cells` and `get_transactions` both enforce `self.request_limit` before executing any query. An unprivileged caller can repeatedly invoke this RPC with `script_search_mode=prefix` and empty `args`, triggering a full-table join+aggregation scan that saturates the SQLite/PostgreSQL backend and degrades or stalls all concurrent indexer RPC responses on the affected node.

## Finding Description
`AsyncRichIndexerHandle` carries a `request_limit` field: [1](#0-0) 

`get_cells` enforces this limit before executing any query: [2](#0-1) 

`get_transactions` applies the same guard: [3](#0-2) 

`get_cells_capacity` has no such guard. It immediately builds and executes an unbounded aggregation: [4](#0-3) 

No `LIMIT` clause is added anywhere in the function before the query is executed: [5](#0-4) 

When `args` is empty (`0x`), `get_binary_upper_boundary` returns `[0xFF; 32]`: [6](#0-5) 

This makes the script subquery match every script row sharing the given `code_hash`/`hash_type`, and the outer query aggregates every matching live cell with no bound. The generated SQL is a full join + `SUM` aggregation over the entire `output` table filtered only by `is_spent = 0`, with no `LIMIT` applied. [7](#0-6) 

## Impact Explanation
Each call forces the database engine to perform a full join and aggregation scan over the entire live-cell set. On a large indexed chain, this saturates I/O and CPU on the SQLite/PostgreSQL backend, blocking all concurrent indexer queries for the duration of the scan. Repeated calls in a tight loop cause sustained unresponsiveness of the local node's RPC layer for all methods backed by the rich-indexer. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**, with potential escalation to **High** if the node operator's RPC becomes fully unresponsive under sustained load.

## Likelihood Explanation
The rich-indexer RPC endpoint is publicly reachable by any unprivileged caller when the node operator enables it. No authentication, proof-of-work, or rate-limiting is required. The exploit requires a single standard JSON-RPC POST with a valid `IndexerSearchKey`. The `request_limit` configuration is already wired into the handle but is simply never consulted by `get_cells_capacity`. The attack is trivially repeatable in a tight loop.

## Recommendation
Add the same `request_limit` guard at the top of `get_cells_capacity`, before any query is built:

```rust
// In get_cells_capacity, before building the query:
// Mirror the pattern from get_cells and get_transactions.
// Since this is an aggregation (no caller-supplied limit), enforce a
// configurable maximum via a LIMIT on the inner script subquery, or
// reject requests that would scan more than request_limit rows.
```

At minimum, apply a `LIMIT self.request_limit` to the inner script subquery (`build_query_script_id_sql`) so the aggregation is bounded, or add a pre-flight count check and return `Error::invalid_params` if the estimated row count exceeds `self.request_limit`.

## Proof of Concept
```bash
# On a node with rich-indexer enabled and a large indexed chain:
while true; do
  curl -s -X POST http://localhost:8114 \
    -H 'Content-Type: application/json' \
    -d '{
      "jsonrpc":"2.0","id":1,
      "method":"get_cells_capacity",
      "params":[{
        "script":{
          "code_hash":"0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
          "hash_type":"type",
          "args":"0x"
        },
        "script_type":"lock",
        "script_search_mode":"prefix"
      }]
    }'
done
```

Each iteration issues a `SUM(output.capacity)` scan over all live cells matching the given `code_hash`/`hash_type` with empty-prefix args (matching all args values). A concurrent `get_cells` call with `limit=request_limit` returns in bounded time, confirming the asymmetry. Wall-clock time for `get_cells_capacity` grows with the number of indexed outputs while `get_cells` remains bounded.
