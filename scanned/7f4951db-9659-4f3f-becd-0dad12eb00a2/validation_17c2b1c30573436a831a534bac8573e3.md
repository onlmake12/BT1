Audit Report

## Title
Stale Pre-Eviction Snapshot Overwrites In-Place Accounting in `add_entry`, Inflating `total_tx_size`/`total_tx_cycles` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures a pre-eviction snapshot of `total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When that function evicts cell-ref-parent transactions via `remove_entry_and_descendants → remove_entry → update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size` in-place. The final two lines of `add_entry` then unconditionally overwrite those in-place decrements with the stale snapshot, permanently inflating both counters by the size and cycles of every evicted transaction. An unprivileged attacker can exploit this to make `limit_size` evict legitimate pending transactions from the mempool.

## Finding Description

The exact sequence in `add_entry` (lines 200–221):

```rust
// Step 1: snapshot — reads self.total_tx_size, does NOT write back
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;  // L210-211

// Step 2: may evict cell-ref-parents, calling update_stat_for_remove_tx
//         which WRITES self.total_tx_size -= evicted_size in-place
evicts = self.check_and_record_ancestors(&mut entry)?;         // L213

// Step 3: overwrites the in-place decrement with the stale snapshot
self.total_tx_size = total_tx_size;                            // L218
self.total_tx_cycles = total_tx_cycles;                        // L219
```

`updated_stat_for_add_tx` (L711–729) is a `&self` method — it only reads `self.total_tx_size` and returns a new value; it never writes back.

`check_and_record_ancestors` (L588–640) triggers the eviction path at L603 when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count`. It calls `remove_entry_and_descendants` (L618), which calls `remove_entry` (L235–250), which calls `update_stat_for_remove_tx` (L247), which writes `self.total_tx_size -= evicted_size` at L739.

The final assignment at L218 then clobbers that decrement with the stale local variable, producing:

| Step | `self.total_tx_size` | local `total_tx_size` |
|---|---|---|
| Initial | `X` | — |
| After `updated_stat_for_add_tx` | `X` (unchanged) | `X + entry_size` |
| After eviction of `evicted_size` | `X − evicted_size` | `X + entry_size` (stale) |
| After `self.total_tx_size = total_tx_size` | `X + entry_size` **(wrong)** | — |

Correct value: `X − evicted_size + entry_size`. Inflation per eviction: `evicted_size`.

No existing guard prevents this: `updated_stat_for_add_tx` only checks for integer overflow, not for the ordering problem. `update_stat_for_remove_tx` has an underflow fallback that recomputes from scratch, but it is never reached here because the overwrite happens after it has already successfully decremented.

## Impact Explanation

`limit_size` (pool.rs L292–329) uses `self.pool_map.total_tx_size > self.config.max_tx_pool_size` as its sole eviction guard (L298). Because `total_tx_size` is inflated by the size of every cell-ref-parent eviction, `limit_size` will see the pool as over-capacity when it is not, and will evict additional valid pending/proposed transactions via `remove_entry_and_descendants` with a `Reject::Full` callback. Those transactions are permanently removed from the pool and must be resubmitted. An attacker can accumulate inflation across repeated submissions until `limit_size` fires continuously, effectively draining the mempool of honest transactions with minimal cost.

**Allowed impact: High — vulnerability or bad design that could cause CKB network congestion with few costs.** The attacker pays only normal transaction fees for the submitted transactions; the cost of each inflation step is bounded by the fee of the evicted cell-ref-parent, which the attacker controls.

## Likelihood Explanation

The trigger condition is reachable by any unprivileged `send_raw_transaction` caller:

1. Submit two transactions `tx_A`, `tx_B` that both reference the same cell dep output `O` (making them cell-ref-parents of any future consumer of `O`).
2. Submit `tx_C` spending `O`, so `ancestors_count = 3 > max_ancestors_count = 2`, but `3 − 2 = 1 ≤ 2`, triggering the eviction branch.
3. Each such round inflates `total_tx_size` by the size of the evicted transaction.
4. Repeat with fresh cell deps until accumulated inflation causes `limit_size` to fire.

No privileged access, key material, or majority hashpower is required. The attacker's own transactions are the ones evicted in the cell-ref-parent step, so the cost is only the fee paid to get them into the pool initially.

## Recommendation

Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so it incorporates any in-place decrements made by evictions:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Compute AFTER evictions have already modified self.total_tx_size
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, replace the local-variable pattern with direct in-place increments (`self.total_tx_size += entry.size`) after all mutations are complete, mirroring the pattern already used in `update_stat_for_remove_tx`.

## Proof of Concept

```
Setup: max_tx_pool_size = 200_000, max_ancestors_count = 2
       pool starts with 180_000 bytes of honest transactions

Round 1:
  Submit tx_A (5_000 bytes) and tx_B (5_000 bytes), both referencing cell dep O.
  → total_tx_size = 190_000 (correct)

  Submit tx_C (1_000 bytes) spending O.
  → updated_stat_for_add_tx snapshots local = 191_000
  → check_and_record_ancestors evicts tx_A:
       update_stat_for_remove_tx(5_000) → self.total_tx_size = 185_000
  → add_entry overwrites: self.total_tx_size = 191_000  ← BUG
  → Correct value: 186_000. Inflation: +5_000.

Repeat with fresh cell deps. After N rounds:
  Reported total_tx_size = 190_000 + N * 5_000
  Actual pool contents    = 190_000 − N * 4_000  (net: evict 5k, add 1k)

  When N = 2: reported = 200_000, actual = 182_000
  → limit_size fires, evicts honest transactions from the 182_000-byte pool.
```

The assertion will fail on the current code.
