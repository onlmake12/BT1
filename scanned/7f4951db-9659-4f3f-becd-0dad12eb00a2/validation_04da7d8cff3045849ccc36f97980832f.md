Audit Report

## Title
Tx-Pool Minimum Fee Rate Check Uses Serialized Size Instead of Actual Transaction Weight, Allowing Below-Minimum Fee Rate Admission - (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size, while the actual weight used for pool sorting, eviction, and fee rate reporting is `get_transaction_weight(size, cycles) = max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. Because cycles are unknown at `pre_check` time and no second fee rate check is performed after `verify_rtx` reveals the actual cycles, an attacker can craft a transaction with a small serialized size but high script execution cycles that is admitted with an effective fee rate far below the configured minimum.

## Finding Description

`check_tx_fee` computes the minimum fee using only `tx_size`:

```rust
// tx-pool/src/util.rs L42-45
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The actual weight function used everywhere else in the pool is:

```rust
// util/types/src/core/tx_pool.rs L298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

In `_process_tx`, `pre_check` (which calls `check_tx_fee`) runs before `verify_rtx`. After `verify_rtx` returns `verified.cycles`, `TxEntry` is constructed and passed directly to `submit_entry` with no second weight-based fee rate check:

```rust
// tx-pool/src/process.rs L751-753
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
``` [4](#0-3) 

`TxEntry::fee_rate()` correctly uses the full weight for sorting and eviction:

```rust
// tx-pool/src/component/entry.rs L114-118
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

This creates a gap: an entry can be admitted satisfying `fee >= min_fee_rate * size` while its actual `fee_rate()` — used for eviction and sorting — is far below `min_fee_rate`.

## Impact Explanation

The most severe concrete impact is **CPU exhaustion of verification threads at drastically reduced cost**. With `min_fee_rate = 1000 shannons/KW`, a transaction with `size = 200 bytes` and `cycles ≈ max_block_cycles (~3.5B)`:

- Size-based min fee paid: `1000 × 200 / 1000 = 200 shannons`
- Actual weight: `max(200, 3.5e9 × 0.000_170_571_4) ≈ 597,000`
- Weight-based min fee that should be required: `597,000 shannons`
- **Cost reduction factor: ~3,000×**

An attacker can flood the node with near-max-cycle transactions at ~3,000× below the intended minimum cost, exhausting `ContextualTransactionVerifier` + script execution threads and causing sustained verification congestion. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

Reachable by any unprivileged caller of the `send_transaction` RPC or any P2P relay peer. No special privileges or keys are required. Crafting a transaction with a small serialized body but a loop-heavy RISC-V lock/type script consuming near-max cycles is straightforward. The cost discount grows with cycle count, making the attack more severe for compute-intensive scripts. The gap is explicitly acknowledged in the codebase comment but no compensating post-verification check exists.

## Recommendation

After `verify_rtx` returns `verified.cycles` in `_process_tx`, perform a second fee rate check using the actual transaction weight before calling `submit_entry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
```

This check belongs in `_process_tx` at `tx-pool/src/process.rs` between lines 750 and 751, after `verified.cycles` is known and before `TxEntry::new` is called. [6](#0-5) 

## Proof of Concept

1. Configure a CKB node with `tx_pool.min_fee_rate = 1000` (shannons/KW).
2. Construct a transaction: `size ≈ 200 bytes`, lock script consuming ~5,000,000 cycles (tight RISC-V loop), fee = 200 shannons.
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons` → passes.
5. `verify_rtx` runs the script, consuming 5,000,000 cycles; actual weight = `max(200, 852) = 852`.
6. `TxEntry` is created with `fee = 200`, `cycles = 5_000_000`; `fee_rate()` returns `≈ 234 shannons/KW` — well below the 1000 shannons/KW minimum.
7. Repeat with many such transactions (each paying only 200 shannons instead of 852) to exhaust verification threads and pollute the pool.

Code path: `send_transaction` → `submit_local_tx` → `_process_tx` → `pre_check`/`check_tx_fee` (size-only gate, L289) → `verify_rtx` (cycles revealed, L724–732) → `TxEntry::new` (no re-check, L751) → `submit_entry`. [7](#0-6)
