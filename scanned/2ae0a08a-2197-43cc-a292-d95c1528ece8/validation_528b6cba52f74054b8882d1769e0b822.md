### Title
`BLOCK_INVALID` Status Not Checked Before `HEADER_VALID` in `HeaderAcceptor::accept()`, Allowing Invalid Block Status Reset — (File: `sync/src/synchronizer/headers_process.rs`)

### Summary

`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` that skips an early-return guard for `BLOCK_INVALID` blocks. Because `BlockStatus::BLOCK_INVALID` (bit 12) and `BlockStatus::HEADER_VALID` (bit 0) are disjoint bits, a block previously marked invalid passes the `status.contains(HEADER_VALID)` check as `false`, falls through all header validation steps, and — if those steps pass — has `insert_valid_header` called on it, which can overwrite the `BLOCK_INVALID` entry in `block_status_map` with `HEADER_VALID`. An unprivileged remote peer can trigger this path by relaying the header of any block the local node has already rejected.

---

### Finding Description

`BlockStatus` is defined as a bitflag set: [1](#0-0) 

`BLOCK_INVALID = 1 << 12 = 4096` is completely disjoint from the cumulative chain `HEADER_VALID (1) → BLOCK_RECEIVED (3) → BLOCK_STORED (7) → BLOCK_VALID (15)`.

`get_block_status()` returns whatever value is stored in `block_status_map` first: [2](#0-1) 

In `HeaderAcceptor::accept()`, the developer left an explicit `FIXME` noting that `BLOCK_INVALID` is not guarded: [3](#0-2) 

Because `4096 & 1 == 0`, `status.contains(BlockStatus::HEADER_VALID)` is `false` when status is `BLOCK_INVALID`. The function does **not** return early; it falls through to `prev_block_check`, `non_contextual_check`, and `version_check`. If the header itself is structurally valid (e.g., the block was rejected for body/script reasons, not header reasons), all three checks pass and `insert_valid_header` is called: [4](#0-3) 

`insert_valid_header` writes to `block_status_map` with `HEADER_VALID`, overwriting the `BLOCK_INVALID` entry. The block is now treated as having a valid header by the rest of the sync subsystem.

---

### Impact Explanation

1. **Status cache poisoning**: A block the node correctly rejected (e.g., invalid script, bad cellbase reward, capacity overflow) has its `BLOCK_INVALID` cache entry silently replaced with `HEADER_VALID`. Subsequent calls to `get_block_status()` return `HEADER_VALID` instead of `BLOCK_INVALID`.

2. **Re-download loop**: The sync subsystem uses `HEADER_VALID` to decide whether to fetch block bodies. After the reset, the node may re-request and re-process the same invalid block body, wasting bandwidth and CPU.

3. **Peer best-known-header pollution**: `may_set_best_known_header` is called with the attacker-controlled header index, skewing the node's view of the peer's chain tip and potentially distorting fork-choice or IBD decisions. [5](#0-4) 

---

### Likelihood Explanation

Any unprivileged peer connected over P2P can send a `SendHeaders` message. The attacker only needs to know the hash of a block the target node has previously rejected — which is observable from the node's behavior (e.g., the node requested the block, then stopped). The header for such a block is freely available on the network. No special privilege, key, or majority hashpower is required.

---

### Recommendation

Add an explicit early-return guard for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, resolving the `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

// Guard: reject immediately if this block is already known-invalid.
if status.contains(BlockStatus::BLOCK_INVALID) {
    state.invalid(Some(ValidationError::InvalidBlock));
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return for already-valid headers
}
```

This mirrors the fix recommended in the Seaport report: introduce an early abort when the accumulated error state is non-zero before any further processing occurs.

---

### Proof of Concept

1. Connect a malicious peer to a CKB node.
2. Relay a block whose **body** is invalid (e.g., a transaction with `OutputsSumOverflow`). The node processes it, fails body verification, and calls `insert_block_status(hash, BlockStatus::BLOCK_INVALID)`.
3. The malicious peer sends a `SendHeaders` message containing the **header** of that same block.
4. `HeadersProcess::execute()` → `HeaderAcceptor::accept()` is called.
5. `get_block_status()` returns `BLOCK_INVALID` (4096). `status.contains(HEADER_VALID)` → `4096 & 1 == 0` → `false`. No early return.
6. `prev_block_check` passes (parent is valid). `non_contextual_check` passes (header PoW and structure are valid). `version_check` passes.
7. `insert_valid_header` is called, overwriting `block_status_map[hash]` from `BLOCK_INVALID` to `HEADER_VALID`.
8. The node now considers the block's header valid and may re-fetch its body, re-run verification, and loop indefinitely. [6](#0-5)

            None => {

                        .get_block_ext(block_hash)

                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,
