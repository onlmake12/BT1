### Title
Observed-Address Peer-ID Poisoning via Identify Protocol — (`network/src/protocols/identify/mod.rs`)

### Summary

`IdentifyCallback::add_observed_addr` only checks whether the remote-supplied `observed_addr` contains *any* peer-id component, not whether that peer-id is the *local* node's peer-id. An unprivileged remote peer can therefore supply an address that already embeds an arbitrary foreign peer-id, which is stored verbatim in `NetworkState::observed_addrs` and subsequently broadcast to every third peer the victim connects to as one of the victim's own listen addresses.

### Finding Description

The guard in `add_observed_addr` is:

```rust
// network/src/protocols/identify/mod.rs  lines 497-507
fn add_observed_addr(&mut self, mut addr: Multiaddr, session_id: SessionId) -> MisbehaveResult {
    if extract_peer_id(&addr).is_none() {          // ← only tests presence, not identity
        addr.push(Protocol::P2P(Cow::Borrowed(
            self.network_state.local_peer_id().as_bytes(),
        )))
    }
    self.network_state.add_observed_addr(session_id, addr);   // stored as-is
    MisbehaveResult::Continue
}
``` [1](#0-0) 

When the attacker sends `observed_addr = /ip4/1.2.3.4/tcp/8114/p2p/<attacker-peer-id>`, `extract_peer_id(&addr)` returns `Some(attacker_peer_id)`, so the `is_none()` branch is skipped and the local peer-id is never appended. The address with the attacker's peer-id is passed directly to:

```rust
// network/src/network.rs  lines 513-516
pub(crate) fn add_observed_addr(&self, session_id: SessionId, addr: Multiaddr) {
    let mut pending_observed_addrs = self.observed_addrs.write();
    pending_observed_addrs.insert(session_id, addr);   // poisoned entry stored
}
``` [2](#0-1) 

The poisoned entry then surfaces in `local_listen_addrs`, which is called every time the victim opens a new identify session with a third peer:

```rust

That list is encoded into the identify handshake message and sent to the third peer:

```rust

let _ = context.quick_send_message(data).await ...;

The third peer stores the poisoned address in its peer store via `add_remote_listen_addrs` and may later attempt to dial it. The TLS/secio handshake will reject the connection because the peer-id embedded in the multiaddr does not match the actual node's cryptographic identity. [5](#0-4) 

### Impact Explanation

Any peer that receives the victim's identify message will store `/ip4/<victim-ip>/tcp/<port>/p2p/<attacker-peer-id>` as a reachable address for the victim. Every subsequent dial attempt to that address will fail at the secio/TLS layer with a peer-id mismatch, degrading the victim's apparent reachability in the network. The victim's real `public_addrs` are still advertised, but the poisoned entry competes with them in the `MAX_RETURN_LISTEN_ADDRS = 10` slot budget. [6](#0-5) 

### Likelihood Explanation

- Requires only a single, unprivileged inbound or outbound P2P connection — no special role or key material needed.
- The attacker controls the `observed_addr` field of the identify message entirely; no PoW or consensus bypass is required.
- The poisoned entry persists in `observed_addrs` for the lifetime of the session and is removed only on disconnect. [7](#0-6) 

### Recommendation

Replace the existence check with an identity check:

```rust
fn add_observed_addr(&mut self, mut addr: Multiaddr, session_id: SessionId) -> MisbehaveResult {
    match extract_peer_id(&addr) {
        None => addr.push(Protocol::P2P(Cow::Borrowed(
            self.network_state.local_peer_id().as_bytes(),
        ))),
        Some(ref id) if id != self.network_state.local_peer_id() => {
            // foreign peer-id injected — reject silently or misbehave
            return MisbehaveResult::Continue;
        }
        Some(_) => {} // already has the correct local peer-id
    }
    self.network_state.add_observed_addr(session_id, addr);
    MisbehaveResult::Continue
}
``` [1](#0-0) 

### Proof of Concept

1. Connect to the victim node and complete the identify handshake, sending `observed_addr = /ip4/1.2.3.4/tcp/8114/p2p/<attacker-peer-id>`.
2. While the session is open, have a third node connect to the victim.
3. Inspect the identify message the victim sends to the third node; it will contain `/ip4/1.2.3.4/tcp/8114/p2p/<attacker-peer-id>` in its `listen_addrs`.
4. Have the third node attempt to dial that address; the connection will be rejected at the secio/TLS layer with a peer-id mismatch.
