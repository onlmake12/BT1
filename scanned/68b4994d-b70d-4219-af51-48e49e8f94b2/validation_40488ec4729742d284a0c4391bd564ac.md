Audit Report

## Title
DB Read Amplification via Crafted `difficulties` Array in `GetLastStateProof` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The size guard in `GetLastStateProofProcess::execute` bounds the number of difficulty entries to 1000, but each entry triggers an independent binary-search DB scan via `get_first_block_total_difficulty_is_not_less_than`. An unprivileged remote peer can craft 1000 strictly-increasing difficulty values that each map to a consecutive block near the start of the range, forcing ~1000 × log₂(N) DB reads per request instead of the ~1000 the guard was designed to allow. All three in-code mitigations (`take_while` filter, `current_difficulty` deduplication, `start_block_number` advancement) are individually bypassable by an attacker with knowledge of the chain's public difficulty distribution.

## Finding Description
**Root cause — guard counts entries, not reads:** [1](#0-0) 
`GET_LAST_STATE_PROOF_LIMIT = 1000` caps the array length, not the number of DB operations it generates. [2](#0-1) 

**Per-entry cost — binary search:**
Each surviving difficulty entry calls `get_first_block_total_difficulty_is_not_less_than`, which reads `start_block_number`, `end_block_number - 1`, then performs a binary search loop — O(log(end − start)) reads total. [3](#0-2) 

**Bypass of `current_difficulty` deduplication:**
The skip at line 82 is bypassed by sending `difficulty[i] = total_difficulty(block_i) + 1`, which is always strictly greater than `current_difficulty = total_difficulty(block_i)` set in the previous iteration. [4](#0-3) 

**Bypass of `start_block_number` advancement:**
After finding block `num`, `start_block_number` is set to `num - 1`. If crafted difficulties map to consecutive blocks 2, 3, 4, …, the search range shrinks by only 1 per iteration, keeping each binary search over nearly the full [i, N) range. [5](#0-4) 

**Bypass of `take_while` filter:**
The filter removes difficulties above `total_difficulty(difficulty_boundary_block_number - 1)`. An attacker who sets `difficulty_boundary` just above `total_difficulty(1000)` ensures all 1000 entries survive. [6](#0-5) 

## Impact Explanation
Each malicious request causes ~1000 × log₂(N) DB reads instead of ~1000. For the current CKB chain (≈13M blocks), log₂(13,000,000) ≈ 23, yielding ~23,000 DB reads per request. With no per-peer rate limiting visible in the handler, an attacker can sustain a stream of such requests, amplifying disk I/O on the light-client server node. This maps to **Low (501–2000 points): important performance improvement for CKB** — the amplification is real and attacker-controlled, but a single request does not by itself crash the node or cause network-wide congestion.

## Likelihood Explanation
- Entrypoint is the standard unauthenticated P2P `GetLastStateProof` message.
- Required knowledge (per-block total difficulty) is public and queryable from any synced node.
- No rate limiting or per-peer request quota is present in the handler.
- The attack is fully repeatable in a tight loop from a single peer.

## Recommendation
Bound the total number of DB reads, not just the number of difficulty entries. Concretely: introduce a per-request read counter inside `get_block_numbers_via_difficulties` (or inside `get_first_block_total_difficulty_is_not_less_than`) and abort with an error once it exceeds `GET_LAST_STATE_PROOF_LIMIT`. Alternatively, cap the number of difficulties that actually enter the binary-search path (i.e., those that pass the `current_difficulty` check) to a constant well below 1000.

## Proof of Concept
On a chain of N blocks (e.g., N = 1,000,000):

1. Query the node to obtain `total_difficulty(i)` for i = 1 … 1000.
2. Build `difficulties = [total_difficulty(1)+1, total_difficulty(2)+1, …, total_difficulty(1000)+1]` — strictly increasing, each just above the actual total difficulty of block i.
3. Set `start_number = 0`, `last_n_blocks = 0`, `difficulty_boundary` just above `total_difficulty(1000)`, `last_hash` = current tip.
4. Send the `GetLastStateProof` P2P message.

**Expected behavior:** `get_block_numbers_via_difficulties` calls `get_first_block_total_difficulty_is_not_less_than` 1000 times. Iteration i searches range [i, N) performing ~log₂(N) reads each. Total DB reads ≈ 1000 × log₂(N) ≈ 20,000 for N = 1M, versus the ~1000 the guard was designed to allow. Repeating this in a loop amplifies node I/O proportionally.
