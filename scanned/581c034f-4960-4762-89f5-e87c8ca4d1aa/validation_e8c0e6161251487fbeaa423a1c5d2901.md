### Title
Missing Per-Peer Rate Limit on `GetLastState` Enables O(log H) DB Read Amplification — (`util/light-client-protocol-server/src/lib.rs`, `util/light-client-protocol-server/src/components/get_last_state.rs`)

### Summary

The `LightClientProtocol` handler processes every incoming `GetLastState` message by unconditionally calling `get_verifiable_tip_header()`, which performs O(log H) RocksDB reads via `chain_root_mmr(tip-1).get_root()`. Unlike the `Relayer` protocol — which enforces a 30 req/sec per-peer rate limiter — the light client protocol has **no rate limiting whatsoever**. A single unprivileged peer can flood the server with `GetLastState{subscribe=false}` messages at network speed, driving sustained DB read amplification.

### Finding Description

**Confirmed call chain:**

`GetLastStateProcess::execute()` (line 40) calls `self.protocol.get_verifiable_tip_header()` unconditionally on every message: [1](#0-0) 

`get_verifiable_tip_header()` calls `snapshot.chain_root_mmr(tip_block.number() - 1).get_root()`, performing O(log H) DB reads: [2](#0-1) 

**No rate limiting in the light client protocol.** The entire `util/light-client-protocol-server/` directory contains zero references to `RateLimiter`, `rate_limit`, or `TooManyRequests`. The `try_process()` dispatcher applies no throttle before dispatching to `GetLastStateProcess`: [3](#0-2) 

**Contrast with the Relayer**, which explicitly rate-limits at 30 req/sec per `(peer, message_type)` key before processing any message: [4](#0-3) 

The `received()` handler only bans peers for malformed messages; valid `GetLastState` messages never trigger a ban regardless of frequency: [5](#0-4) 

### Impact Explanation

At mainnet height H ≈ 10,000,000, `get_root()` reads approximately log₂(H) ≈ 23 MMR nodes from RocksDB per message. A single peer sending 10,000 messages/sec drives ~230,000 DB reads/sec from one connection alone. This causes I/O saturation on the node's database, degrading block processing, transaction relay, and overall node responsiveness. The node cannot distinguish this from legitimate traffic and will not disconnect or ban the peer.

### Likelihood Explanation

The light client protocol is opt-in (enabled via `support_protocols.contains(&SupportProtocol::LightClient)` in the launcher): [6](#0-5) 

Nodes that explicitly serve light clients are the target population. Any peer that connects to such a node can execute this attack with a trivial loop — no special privileges, no PoW, no keys required.

### Recommendation

Add a per-peer rate limiter to `LightClientProtocol::try_process()` mirroring the `Relayer` implementation — keyed on `(peer, message.item_id())` with a configurable quota (e.g., 10–30 req/sec). Additionally, consider caching the result of `get_verifiable_tip_header()` with a short TTL (e.g., one block interval) so repeated calls within the same block epoch hit an in-memory cache rather than the DB.

### Proof of Concept

1. Connect a peer to a CKB node with `SupportProtocol::LightClient` enabled.
2. In a tight loop, send `LightClientMessage{ GetLastState{ subscribe: false } }` as fast as the TCP connection allows.
3. Monitor RocksDB read counters (e.g., via `rocksdb.block.cache.miss` or `iostat`).
4. Observe DB read rate scaling linearly with message rate, with no server-side throttle or disconnect.

            warn!("process {} from {}; result is {}", item_name, peer, status);
