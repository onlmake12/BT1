### Title
`ScriptHashTypeVerifier` Only Checks Lock Script `hash_type`, Not Type Script `hash_type`, Allowing Transactions With Disallowed Type Script Hash Types to Bypass Non-Contextual Verification — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` is documented as checking "whether output lock hash type [is] within enabled range" for all transaction outputs. In practice it only iterates over `output.lock().hash_type()` and never inspects `output.type_().hash_type()`. A transaction whose outputs carry a type script with a not-yet-enabled hash type (e.g. `Data3 = 6`) passes `NonContextualTransactionVerifier` and enters the tx-pool verify queue, where it is only rejected later during expensive contextual script execution. Because the rejection originates from a `ScriptError` rather than a `TransactionError`, the submitting peer is not classified as malformed and is not banned, allowing the attacker to repeat the submission indefinitely.

---

### Finding Description

**Root cause — `ScriptHashTypeVerifier` iterates only over lock scripts**

`verification/src/transaction_verifier.rs` lines 796–814:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The loop body calls `output.lock().hash_type()` exclusively. `output.type_()` is never read. The struct comment in `NonContextualTransactionVerifier` even documents this gap explicitly:

> `Check whether output lock hash type within enabled range` [2](#0-1) 

**Contrast with `check_data`, which covers both scripts**

The lower-level structural check in `util/gen-types/src/extension/check_data.rs` correctly validates both:

```rust
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
``` [3](#0-2) 

`check_data` only verifies that the raw byte is a structurally valid enum discriminant (multiple-of-2 or 1). It does **not** check membership in `ENABLED_SCRIPT_HASH_TYPE`. That second, stricter check is the sole responsibility of `ScriptHashTypeVerifier`, which omits type scripts entirely.

**Attack path through the tx-pool**

1. Attacker calls `send_transaction` RPC with a transaction whose output has a valid lock script but a type script with `hash_type = Data3` (value `6`), which is structurally valid but not in `ENABLED_SCRIPT_HASH_TYPE`.
2. `non_contextual_verify` → `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` passes (only lock script is checked). [4](#0-3) 

3. `pre_check` resolves the transaction and checks the fee — passes if the attacker provides sufficient fee.
4. `verify_rtx` runs `ContextualTransactionVerifier`, which invokes `ScriptVerifier`. Inside `select_version`, the invalid hash type is caught:

```rust
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated…",
        hash_type
    )));
}
``` [5](#0-4) 

5. The rejection is `Reject::Verification(script_error)`. In `after_process`, `reject.is_malformed_tx()` is evaluated. `TransactionError::InvalidScriptHashType` and `TransactionError::ScriptHashTypeNotPermitted` are classified as malformed: [6](#0-5) 

However, the error here is a `ScriptError` (not a `TransactionError`), so `is_malformed_tx()` returns `false` and **the peer is not banned**. [7](#0-6) 

6. The attacker reuses the same UTXOs (inputs are not spent on rejection) and repeats from step 1.

---

### Impact Explanation

- Transactions with not-yet-enabled type script hash types bypass the cheap non-contextual filter and consume contextual verification resources on every submission.
- The submitting peer is never banned because the rejection error is a `ScriptError`, not a `TransactionError`, so `is_malformed_tx()` returns `false`.
- An attacker with a modest number of live UTXOs can sustain a continuous stream of such submissions, forcing the node to repeatedly run contextual verification (including script group construction and `select_version` dispatch) without penalty.
- The `NonContextualBlockTxsVerifier` path has the same gap: a block relayed with transactions carrying invalid type script hash types also passes non-contextual block verification before being caught at contextual script execution. [8](#0-7) 

---

### Likelihood Explanation

The entry point is the public `send_transaction` RPC, reachable by any unprivileged caller. No special role, key, or majority hash power is required. The attacker only needs a small number of live cells to fund the minimum fee. Because the peer is never banned, the attack can be sustained indefinitely at low cost. [9](#0-8) 

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type when a type script is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        let lock_ht = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?;
        let val: u8 = lock_ht.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?;
            let val: u8 = type_ht.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        }
    }
    Ok(())
}
```

This mirrors the existing lock-script logic, ensures the `TransactionError` variant is raised (triggering peer banning), and is consistent with `check_data`'s symmetric treatment of both script fields. [10](#0-9)

### Proof of Concept

1. Construct a transaction with one output whose lock script uses `hash_type = Data` (0, valid and enabled) and whose type script uses `hash_type = Data3` (6, structurally valid but not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Provide sufficient capacity and fee from a live UTXO.
3. Submit via `send_transaction` RPC.
4. Observe: the call returns a `TransactionFailedToVerify` error (from script execution), **not** `PoolRejectedTransactionByOutputsValidator`.
5. Observe: the node does not ban the submitting peer.
6. Resubmit with the same UTXO (inputs are not consumed). The node repeats full contextual verification each time.

The test suite already demonstrates that `ScriptHashTypeVerifier` only tests the lock script path: [11](#0-10) 

No analogous test exists for an invalid type script hash type, confirming the gap is untested and unguarded.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
