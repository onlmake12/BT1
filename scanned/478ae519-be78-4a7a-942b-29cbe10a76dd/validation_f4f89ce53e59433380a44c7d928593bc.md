### Title
Unauthenticated Block Template Injection via Miner HTTP Notification Endpoint Allows Mining Reward Theft - (File: miner/src/client.rs)

### Summary

The CKB miner process exposes an HTTP notification endpoint that accepts `BlockTemplate` pushes from the node. The handler at `miner/src/client.rs` performs no authentication whatsoever before accepting and applying an incoming block template. Any local process (or remote client if the port is exposed) can POST a crafted `BlockTemplate` containing an attacker-controlled `cellbase` reward address, causing the miner to mine blocks that pay all block rewards to the attacker. This is a direct analog to the Uniswap `createIncentive` bug: just as any caller could replace the `incentiveConfig` by contributing a small reward, any HTTP caller can replace the miner's active block template — including its reward-destination cellbase — without any ownership or identity check.

### Finding Description

The miner's HTTP notification server handler is defined at `miner/src/client.rs` lines 358–369:

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
``` [1](#0-0) 

The handler:
1. Reads the raw HTTP body unconditionally.
2. Attempts to deserialize it as a `BlockTemplate`.
3. On success, immediately calls `client.update_block_template(template)` — replacing the miner's active template with the attacker-supplied one.
4. Returns HTTP 200 regardless.

There is no `Authorization` header check, no shared secret, no source-IP allowlist, and no validation that the incoming template's `cellbase` matches the node's configured `block_assembler`. The `parse_authorization` function that exists in the same file is used only for the miner's *outbound* RPC calls to the node, not for the miner's *inbound* HTTP server: [2](#0-1) 

The `BlockTemplate` type carries the full `cellbase` transaction, which encodes the miner's reward lock script (i.e., the address that receives block rewards). The node's block validation enforces consensus rules on the cellbase (correct reward amount, correct structure) but does **not** verify that the cellbase lock script matches the node's configured `block_assembler`. Therefore, a block mined from an attacker-supplied template with a different cellbase will be accepted by the node and committed to the chain, with the reward going to the attacker's address.

The miner polls the node for templates via `get_block_template` RPC: [3](#0-2) 

But it also accepts pushed templates via this unauthenticated HTTP endpoint. Whichever template arrives last wins, because `update_block_template` unconditionally overwrites the current template.

### Impact Explanation

An attacker who can send an HTTP POST to the miner's notification port can:

1. Craft a `BlockTemplate` JSON with a `cellbase` transaction whose output lock script points to the attacker's own address.
2. POST it to the miner's HTTP notification endpoint.
3. The miner immediately begins mining on the attacker's template.
4. When a valid proof-of-work is found, the miner submits the block to the node.
5. The node validates and accepts the block (cellbase is consensus-valid).
6. All block rewards (primary + secondary issuance) for that block are paid to the attacker.

Impact: **direct, permanent theft of mining rewards** from the legitimate miner operator. Every block mined while the attacker's template is active transfers the full block reward to the attacker. The legitimate miner expends real hashpower and electricity with zero return.

### Likelihood Explanation

The miner's HTTP notification port is configurable. If it binds to `127.0.0.1` (localhost), the attacker must be a local process on the same machine — realistic in shared hosting, cloud VMs with multiple tenants, or any environment where a malicious co-located process exists. If the operator misconfigures the listen address to `0.0.0.0` or a public IP (not uncommon in pool/farm setups), the attack is trivially reachable from the internet with a single HTTP POST. No credentials, no keys, no privileged access are required — only network reachability to the miner's HTTP port. The attack is silent: the miner continues operating normally and shows no error; only the reward destination changes.

### Recommendation

Add a shared-secret authentication mechanism to the miner's HTTP notification endpoint. The node and miner should be configured with a common token (e.g., a random 256-bit secret in `ckb-miner.toml`). The node includes this token in an `Authorization: Bearer <token>` header when pushing templates. The `handle` function must reject any request that does not present the correct token before calling `update_block_template`. Additionally, the miner should validate that the incoming template's `cellbase` output lock script matches the locally configured `block_assembler` before accepting the template.

### Proof of Concept

**Preconditions:** Attacker can send HTTP requests to the miner's notification port (e.g., `127.0.0.1:18114`).

**Steps:**

1. Obtain a valid `BlockTemplate` JSON from the node (via `get_block_template` RPC — public, unauthenticated by default).
2. Replace the `cellbase.outputs[0].lock` field with the attacker's own lock script (attacker's public key hash).
3. POST the modified template to the miner's HTTP notification endpoint:
   ```
   POST http://127.0.0.1:18114/ HTTP/1.1
   Content-Type: application/json

   { ...modified BlockTemplate with attacker cellbase... }
   ```
4. The `handle` function deserializes the body and calls `client.update_block_template(template)` with no authentication check.
5. The miner's workers now mine on the attacker's template.
6. The next block found is submitted to the node, accepted (cellbase is consensus-valid), and the block reward is credited to the attacker's address.

**Expected result:** Mining reward for every block mined after the injection is transferred to the attacker. The legitimate miner operator receives nothing.

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}
