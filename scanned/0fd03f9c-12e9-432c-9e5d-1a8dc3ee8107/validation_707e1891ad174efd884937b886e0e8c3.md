Audit Report

## Title
Unvalidated `observed_addr` Injection Poisons Node's Own Address Advertisement - (File: `network/src/protocols/identify/mod.rs`)

## Summary
Any peer that completes a P2P handshake with a victim node can inject an arbitrary globally-routable `Multiaddr` into the victim's `observed_addrs` store by sending a crafted `IdentifyMessage`. The victim then unconditionally re-advertises this attacker-controlled address as its own listen address to every subsequent peer it connects to, propagating the poisoned entry into those peers' persistent peer stores and into hole-punching gossip broadcasts.

## Finding Description
**Root cause — no IP validation in `process_observed`:**

`process_listens` (L123–150) applies a `global_ip_only` reachability filter to addresses received from remote peers. `process_observed` (L152–169) applies **no such filter** — it passes the attacker-supplied `Multiaddr` directly to `add_observed_addr`:

```rust
// L167
self.callback.add_observed_addr(observed, info.session.id);
```

`IdentifyCallback::add_observed_addr` (L497–507) appends the local peer-id if absent and writes the address unconditionally into `NetworkState::observed_addrs` keyed by `SessionId`:

```rust
self.network_state.add_observed_addr(session_id, addr);
```

`NetworkState::add_observed_addr` (network.rs L512–516) performs a direct map insert with no validation:

```rust
pending_observed_addrs.insert(session_id, addr);
```

**Propagation path:**

`local_listen_addrs` (L457–470) merges `observed_addrs` into the node's own listen addresses. In `connected` (L206–236), this merged list is filtered only by `is_reachable()` (which passes any globally-routable IP) and then sent as the node's own `IdentifyMessage` to every new peer. The same `observed_addrs` are included in hole-punching `ConnectionRequest` broadcasts (hole_punching/mod.rs L187–198).

**Why existing guards are insufficient:**

- `check_duplicate` (L107–121) prevents a second `IdentifyMessage` per session but one message is sufficient.
- The `global_ip_only` / `is_reachable` filter in `connected` (L217–224) only excludes private/loopback IPs; any globally-routable attacker IP (e.g., `1.2.3.4`) passes through.
- There is no cross-check between the `observed_addr` IP component and `context.session.address` (the transport-layer source IP of the session).

**Persistence:** Entries in `observed_addrs` are keyed by `SessionId`. Even if the entry is removed on session close, the poisoned address has already been written into the peer stores of every peer the victim connected to during the session via `add_remote_listen_addrs` (L472–495), where it persists independently.

## Impact Explanation
With a single inbound or outbound connection, an attacker can redirect all future connection attempts targeting the victim node to an attacker-controlled endpoint. At scale — or when combined with hole-punching gossip amplification — this degrades network-wide peer discovery, causes widespread failed dials, and can partition subsets of the CKB P2P network. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as the attacker's cost is one TCP connection and one crafted message, while the effect propagates to O(N) peers via the identify and hole-punching protocols.

## Likelihood Explanation
- **Entry path**: Any node that can complete a TCP + secio handshake (inbound or outbound) qualifies. No authentication beyond a valid P2P handshake is required.
- **Ease**: One connection, one message. The `IdentifyMessage` is sent immediately on protocol open, so the attack window is the entire session lifetime.
- **Amplification**: Via hole-punching gossip, the poisoned address reaches `sqrt(total_connections)` additional peers per notify interval without further attacker action.
- **Persistence**: The poisoned address survives session teardown in the peer stores of all peers that received it.

## Recommendation
- **Short term**: In `process_observed` / `add_observed_addr`, extract the IP component from the supplied `observed_addr` and compare it against the transport-layer source IP of the session (`context.session.address`). Reject (return `MisbehaveResult::Disconnect`) any `observed_addr` whose IP does not match the session's actual remote IP.
- **Long term**: Treat `observed_addrs` as unconfirmed hints. Require corroboration from at least two independent peers reporting the same external address before including it in outbound `listen_addrs`. This is the approach used by Bitcoin Core's address manager and libp2p's identify-push extension.

## Proof of Concept
1. Attacker Eve establishes a TCP connection to victim node V (inbound or outbound) and completes the secio handshake.
2. Eve sends an `IdentifyMessage` with `observed_addr = /ip4/1.2.3.4/tcp/8114` (attacker-controlled, globally routable IP).
3. V's `received` handler calls `process_observed` → `add_observed_addr` → `NetworkState::add_observed_addr`, storing `/ip4/1.2.3.4/tcp/8114/p2p/<V-peer-id>` in `observed_addrs[eve_session_id]`.
4. V subsequently connects to peer P. `connected` calls `local_listen_addrs()`, which appends the poisoned address; `is_reachable(1.2.3.4)` returns `true`, so it passes the filter and is sent to P.
5. P's `add_remote_listen_addrs` writes `/ip4/1.2.3.4/tcp/8114/p2p/<V-peer-id>` into its peer store under V's peer-id.
6. P later dials `1.2.3.4:8114` expecting V, but reaches Eve's server.
7. Repeat for every peer V connects to during Eve's session; hole-punching gossip further propagates the address without additional attacker action.

**Minimal test**: Write a unit test that creates a mock `IdentifyProtocol`, calls `received` with a crafted `IdentifyMessage` containing a mismatched `observed_addr`, then asserts that `NetworkState::observed_addrs()` does not contain the injected address and that the session is disconnected.
