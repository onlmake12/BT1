Audit Report

## Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Bypasses Sample Cap, Enabling Unbounded MMR Work per Request — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The guard at line 201 of `GetLastStateProofProcess::execute` computes `(last_n_blocks as usize) * 2`, which wraps to `0` in release-mode Rust when `last_n_blocks = 2^63`. With an empty `difficulties` list, the check becomes `0 + 0 > 1000 = false`, silently bypassing the cap. The server then collects every block number from `start_block_number` to `last_block_number` into a `Vec` and calls `chain_root_mmr(*number - 1).get_root()` for each entry — an expensive MMR operation — with no bound on iterations. A single crafted message from any peer can exhaust server CPU and memory proportional to full chain length.

## Finding Description

**Root cause — overflow in the guard** (`get_last_state_proof.rs`, lines 199–205):

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
``` [1](#0-0) 

On 64-bit targets `usize` is 64 bits, so `last_n_blocks as usize` is lossless. With `last_n_blocks = 0x8000000000000000` (`2^63`), the multiplication `(2^63) * 2 = 2^64` wraps to `0` in release mode (Rust only panics on overflow in debug builds). With `difficulties = []`, the expression evaluates to `0 + 0 > 1000 = false`, and the guard is bypassed.

**Unbounded allocation after bypass** (`get_last_state_proof.rs`, lines 291–297):

```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [2](#0-1) 

With `last_n_blocks = 2^63`, the condition `last_block_number - start_block_number <= 2^63` is always true for any real chain. The server takes the "not enough blocks" branch and allocates a `Vec` containing every block number from `start_block_number` to `last_block_number` — potentially millions of entries.

**Expensive per-entry work** (`get_last_state_proof.rs`, lines 150–163):

```rust
let mmr = self.snapshot.chain_root_mmr(*number - 1);
match mmr.get_root() { ... }
``` [3](#0-2) 

`complete_headers` iterates every collected block number and calls `chain_root_mmr(*number - 1).get_root()` for each, a non-trivial MMR computation. CPU and memory scale linearly with chain length, not with the intended limit of 1000.

**The bypassed constant** (`constant.rs`, line 6): [4](#0-3) 

**Precondition check on `start_block_number = 0`**: With `start_block_number = 0`, the reorg branch at line 237 short-circuits to `Vec::new()` without validating `start_hash`, so the attacker does not need a valid start hash. [5](#0-4) 

The only non-trivial precondition is a valid `last_hash` (line 210), which is obtainable from normal chain sync. [6](#0-5) 

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

A single `GetLastStateProof` message causes the server to allocate a `Vec` proportional to full chain length and perform one MMR root computation per block. On a chain of 500,000 blocks this means 500,000 MMR operations per request. Repeated requests from multiple peers exhaust server memory and CPU, causing the full node to crash or become unresponsive. This directly matches the allowed impact class "Vulnerabilities which could easily crash a CKB node."

## Likelihood Explanation

Preconditions are trivially satisfiable by any unprivileged peer:
- A valid `last_hash` is available from normal chain sync (no keys, no PoW, no privilege required).
- `last_n_blocks = 0x8000000000000000` is a single `u64` field in the protobuf message.
- `difficulties = []` and `start_number = 0` require no special knowledge.

The overflow is deterministic and reproducible on any 64-bit host running a release build. The attack is repeatable from multiple peers simultaneously.

## Recommendation

Add an explicit upper-bound check on `last_n_blocks` before any arithmetic:

```rust
if last_n_blocks > constant::GET_LAST_STATE_PROOF_LIMIT as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}
```

Replace the unchecked multiplication with overflow-safe operations:

```rust
let last_n_blocks_doubled = (last_n_blocks as usize)
    .checked_mul(2)
    .unwrap_or(usize::MAX);
if self.message.difficulties().len().saturating_add(last_n_blocks_doubled)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

## Proof of Concept

1. Sync a node to height N (e.g., N = 500,000) and obtain `tip_hash` via normal sync.
2. Craft a `GetLastStateProof` message:
   - `last_n_blocks = 0x8000000000000000`
   - `difficulties = []`
   - `start_number = 0`
   - `last_hash = tip_hash`
   - `difficulty_boundary = U256::MAX`
   - `start_hash` = any value (not checked when `start_number = 0`)
3. Send to the server peer.
4. Observe: guard evaluates `0 + 0 > 1000 = false`; server allocates `Vec` of N entries and calls `chain_root_mmr` N times.
5. Assert `block_numbers.len() == N` (not bounded by 1000) and that server RSS grows proportionally to N.
6. Repeat from multiple peers to exhaust memory and crash the node.
