### Title
Unbounded Async Task Spawning via `ConnectionSync` with Attacker-Controlled `from` PeerId Bypassing Per-Tuple Rate Limiter — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

An unprivileged remote peer can trigger unbounded `runtime::spawn` calls in `ConnectionSyncProcess::execute` by exploiting a rate-limiter key design flaw. The `forward_rate_limiter` is keyed on `(content.from, content.to, item_id)`, but `content.from` is fully attacker-controlled. By cycling through distinct `from` PeerIds — each pre-seeded into `pending_delivered` via a prior `ConnectionRequest` flood — the attacker bypasses the per-tuple 1/sec limit and is constrained only by the outer 30/sec session limiter. Each spawned task holds up to 24 concurrent TCP connections open for up to 30 seconds, yielding O(N × 24) concurrent connections and O(N) live tokio tasks.

---

### Finding Description

**Phase 1 — Populate `pending_delivered` (prerequisite)**

The attacker sends `ConnectionRequest` messages with `content.to = victim_peer_id` and a fresh `content.from = attacker_peer_id_N` for each message. [1](#0-0) 

The `forward_rate_limiter` check uses `(content.from, content.to, item_id)` as the key. Each new `from` PeerId creates a fresh bucket, so the only binding limit is the outer `rate_limiter` at 30 messages/sec per session. [2](#0-1) 

When `self_peer_id == content.to`, `respond_delivered` is called, which inserts `from_peer_id → (remote_listens, now)` into `pending_delivered` — with no global cap on the map size. [3](#0-2) 

The only per-`from` guard is a 2-minute re-insertion check, which is irrelevant when using distinct `from` PeerIds. [4](#0-3) 

**Phase 2 — Trigger unbounded task spawning via `ConnectionSync`**

The attacker sends `ConnectionSync` messages with `content.to = victim_peer_id` and `content.from = attacker_peer_id_N` (matching the seeded entries). The same `forward_rate_limiter` bypass applies. [5](#0-4) 

When `self_peer_id == content.to` and `pending_delivered` contains `content.from`, the code unconditionally calls `runtime::spawn` with no deduplication, no task counter, and no cap: [6](#0-5) 

Each spawned task runs `select_ok(tasks)` over up to `ADDRS_COUNT_LIMIT` (24) concurrent `try_nat_traversal` futures, each of which loops for up to 30 seconds: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

- **Outer rate limiter**: 30 `ConnectionSync` messages/sec per session.
- **`forward_rate_limiter`**: 1/sec per `(from, to, item_id)` — fully bypassed by rotating `from` PeerIds.
- **Task lifetime**: up to 30 seconds per task.
- **Connections per task**: up to 24 (`ADDRS_COUNT_LIMIT`).

Steady-state after 30 seconds: **30 tasks/sec × 30 sec = 900 live tokio tasks**, each holding up to **24 outbound TCP connections** = **21,600 concurrent TCP connections** from a single attacker session. This exhausts file descriptors, tokio worker threads, and network resources, causing denial of service. [9](#0-8) 

---

### Likelihood Explanation

The attack requires only a single P2P connection to the victim. Both phases use standard, well-formed protocol messages. No PoW, no privileged role, no leaked keys. The `pending_delivered` seeding (Phase 1) and the task-spawning trigger (Phase 2) can be executed by the same session within seconds.

---

### Recommendation

1. **Cap `pending_delivered` map size** — enforce a hard upper bound (e.g., 64 entries) and evict oldest entries when full.
2. **Track live NAT traversal task count** — maintain an atomic counter; reject new `ConnectionSync` triggers when the count exceeds a threshold (e.g., 10).
3. **Fix the rate-limiter key** — key the `forward_rate_limiter` on `(sender_session_id, item_id)` rather than `(content.from, content.to, item_id)` so the attacker cannot bypass it by rotating the attacker-controlled `from` field.
4. **Deduplicate in-flight traversals** — track `(from, to)` pairs currently being traversed and skip re-spawning if one is already live.

---

### Proof of Concept

```
1. Connect to victim via P2P (single session).
2. For i in 0..900:
     Send ConnectionRequest { from: fresh_peer_id_i, to: victim_peer_id,
                              listen_addrs: [24 × attacker_TCP_addr], ... }
     // victim inserts fresh_peer_id_i → ([24 addrs], now) into pending_delivered
3. For i in 0..900:
     Send ConnectionSync { from: fresh_peer_id_i, to: victim_peer_id, route: [] }
     // victim spawns runtime::spawn(select_ok([24 try_nat_traversal futures]))
     // each task lives 30s, each holds 24 TCP connections
4. Assert: victim has ~900 live tokio tasks and ~21,600 open TCP connections.
   Monitor with /proc/<pid>/fd count or tokio-console task count.
```
