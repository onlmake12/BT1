### Title
Per-removal `shrink_to_fit!` on `MultiIndexVerifyEntryMap` causes O(N) reallocation on every `pop_front`, degrading tx verification throughput — (`tx-pool/src/component/verify_queue.rs`)

---

### Summary

`VerifyQueue::remove_tx` unconditionally calls `self.shrink_to_fit()` on every successful removal. The `shrink_to_fit!` macro guard (`capacity > len + SHRINK_THRESHOLD`) does **not** prevent repeated O(N) reallocations — it actually guarantees them once the queue exceeds ~700 entries, because after each shrink the new capacity is approximately `1.14 × len` (hashbrown's 7/8 load factor), which immediately satisfies the condition on the very next pop. An unprivileged attacker who fills the verify queue with structurally valid transactions (no UTXO lookup required) causes every subsequent verification to pay O(N) allocator cost, degrading throughput for all peers.

---

### Finding Description

**Root cause — `remove_tx` calls `shrink_to_fit` unconditionally:** [1](#0-0) 

Every successful removal ends with `self.shrink_to_fit()` at line 146.

**The macro guard:** [2](#0-1) 

The guard fires when `capacity > len + SHRINK_THRESHOLD` where `SHRINK_THRESHOLD = 100`. [3](#0-2) 

**Why the guard does not help — the self-reinforcing cycle:**

After `shrink_to_fit()` executes, the backing store is reallocated to the minimum capacity needed for the current `len`. For hashbrown (Rust's `HashMap` backend, load factor 7/8), this is approximately `capacity_after_shrink ≈ len × 8/7 ≈ 1.14 × len`.

On the very next pop: `capacity = 1.14 × N`, `len = N − 1`.

Condition check: `1.14N > (N − 1) + 100` → `0.14N > 99` → **N > 707**.

So for any queue with more than ~707 entries, **every single pop triggers another O(N) shrink**, which resets capacity to `1.14 × (N−1)`, which again satisfies the condition on the next pop. The threshold of 100 is structurally too small relative to the post-shrink capacity surplus introduced by the load factor.

**The `pop_front` path that the verify worker uses:** [4](#0-3) 

`pop_front` calls `remove_tx`, so every normal verification triggers this.

**The attacker's entry path:** [5](#0-4) 

`submit_remote_tx` → `resumeble_process_tx` → `non_contextual_verify` → `enqueue_verify_queue`. The `non_contextual_verify` check requires only structural validity — no UTXO lookup, no script execution, no real CKB tokens spent.

**The queue size limit:** [6](#0-5) 

256 MB of total serialized transaction data. With minimum CKB transaction sizes (~60–100 bytes), this accommodates hundreds of thousands of entries, far above the 707-entry threshold.

---

### Impact Explanation

Once the queue exceeds ~707 entries, every call to `pop_front` (the verify worker's normal operation) triggers an O(N) `MultiIndexMap` reallocation. The `verify_queue` write lock is held for the duration, blocking all concurrent queue operations. The total cost to drain a queue of N entries becomes O(N²) instead of O(N). This directly degrades tx verification throughput for all peers sharing the node, constituting tx-pool throughput degradation and contributing to network congestion.

---

### Likelihood Explanation

The attack requires only:
1. Submitting transactions that pass `non_contextual_verify` (structural validity only — no real CKB tokens, no valid UTXOs required).
2. Maintaining the queue above ~707 entries (well below the 256 MB cap).
3. Sustaining the submission rate to keep the queue populated.

This is reachable via the standard P2P `RelayTransactionHashes`/`SendRelayTransactions` path or the `send_transaction` RPC, both of which are unprivileged. The cost is CPU and bandwidth only.

---

### Recommendation

Remove the `shrink_to_fit` call from `remove_tx`. Shrinking should be done periodically (e.g., every K removals, or only when the queue transitions from non-empty to empty, as is already done in `clear`). Alternatively, raise the threshold to account for the post-shrink capacity surplus: `SHRINK_THRESHOLD` must exceed `len / 7` (for hashbrown's 7/8 load factor) to avoid the self-reinforcing cycle, which is impractical as a static constant. The correct fix is to decouple shrinking from the hot removal path entirely.
