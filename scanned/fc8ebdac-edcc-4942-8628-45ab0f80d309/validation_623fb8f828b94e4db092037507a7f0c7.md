### Title
Copy-Paste Bug in `script_exists_in_output` Causes Incorrect Script Deletion on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary
`script_exists_in_output` in `remove.rs` contains a copy-paste error: the second `match` block (which should evaluate the **type-script** query result `row_type`) instead re-reads `row_lock`. On PostgreSQL, this causes the function to return the lock-script existence result for both checks, so a script used only as a `type_script` in a surviving block's output is incorrectly reported as unreferenced and deleted during rollback.

### Finding Description

In `script_exists_in_output` (lines 204–257), the function issues two SQL `EXISTS` queries:

1. `row_lock` — checks `WHERE lock_script_id = $1`
2. `row_type` — checks `WHERE type_script_id = $1`

After the second query, the final `match` block at line 252 reads:

```rust
match row_lock.try_get::<bool, _>(0) {   // ← BUG: should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

On **PostgreSQL**, `row_lock.try_get::<bool, _>(0)` always succeeds (`Ok(r)`), so the function returns the lock-script query result (`r`) instead of the type-script query result. On **SQLite**, `try_get::<bool, _>` fails and the fallback correctly reads `row_type`, so SQLite is unaffected.

The rollback path in `rollback_block` (lines 7–51) first removes the rolled-back block's outputs, then calls `script_exists_in_output` for each script referenced by those outputs: [2](#0-1) 

If a script `S` is shared between the rolled-back block (block 2) and a surviving block (block 1), and block 1's output uses `S` only as a `type_script` (not `lock_script`):

- After block 2's outputs are removed, `WHERE lock_script_id = S.id` → `false`
- `WHERE type_script_id = S.id` → `true` (block 1's output still references it)
- Bug: the function returns `false` (the lock result) → `S` is added to `script_id_list_to_remove` and deleted

Block 1's `output` rows still hold `type_script_id = S.id`, but the `script` row no longer exists. All RPC queries that join `output` to `script` for that script hash return empty results.

### Impact Explanation
On PostgreSQL deployments, any chain reorganization (reorg) where a rolled-back block shares a `type_script` with a surviving block causes that script to be permanently deleted from the `script` table. All subsequent RPC queries for live cells by that script (e.g., `get_cells`, `get_cells_capacity`) return empty results, permanently corrupting the indexer's view of live cells for any application relying on type-script-based cell queries.

### Likelihood Explanation
Reorgs are a normal part of CKB chain operation and do not require attacker action. The scope rules explicitly list "database restart/reorg state" as a valid entry point. Any PostgreSQL-backed rich-indexer deployment that experiences a reorg where the rolled-back block shares a type_script with a surviving block will silently corrupt its script table. The condition (shared type_script across a reorg boundary) is common in practice, e.g., any two blocks whose transactions use the same type script (UDT, NFT, etc.).

### Recommendation
Fix line 252 in `remove.rs` to read `row_type` instead of `row_lock`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ← was row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

### Proof of Concept
1. Run a PostgreSQL-backed rich-indexer node.
2. Append block 1 containing a transaction with an output whose `type_script` is `S` (and whose `lock_script` is some other script `L ≠ S`).
3. Append block 2 containing a transaction with an output also using `S` as `type_script`. `bulk_insert_script_table` silently skips re-inserting `S` via `ON CONFLICT DO NOTHING`.
4. Trigger rollback of block 2 (simulate a reorg).
5. Assert: `SELECT * FROM script WHERE id = <S.id>` returns a row (it should still exist).
6. **Observed**: the row is gone. Block 1's output still has `type_script_id = <S.id>`, but the script row was deleted.
7. Assert: RPC `get_cells` filtered by script `S` returns block 1's cell. **Observed**: returns empty. [3](#0-2)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
