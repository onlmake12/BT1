### Title
Order-Dependent State Update in `add_entry` Inflates `total_tx_size`/`total_tx_cycles` After Ancestor Eviction — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new transaction's aggregate size and cycle totals are pre-computed into local variables **before** `check_and_record_ancestors` runs. When that function evicts existing pool entries (via `remove_entry_and_descendants`), those evictions correctly decrement `self.total_tx_size` and `self.total_tx_cycles`. However, the stale pre-eviction local values are then unconditionally written back, overwriting the correct post-eviction state and permanently inflating the pool's accounting.

---

### Finding Description

The root cause is in `PoolMap::add_entry`: [1](#0-0) 

The sequence of operations is:

**Step 1 (lines 210–211):** `updated_stat_for_add_tx` computes `total_tx_size = self.total_tx_size + entry.size` and `total_tx_cycles = self.total_tx_cycles + entry.cycles` into **local variables** — they are not yet written to `self`. [2](#0-1) 

**Step 2 (line 213):** `check_and_record_ancestors` is called. When the incoming transaction's ancestor count exceeds `max_ancestors_count` but can be reduced by evicting `cell_ref_parents`, it calls `remove_entry_and_descendants` in a loop: [3](#0-2) 

Each call to `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` **correctly decrements** `self.total_tx_size` and `self.total_tx_cycles` by the evicted entry's size and cycles: [4](#0-3) [5](#0-4) 

**Step 3 (lines 218–219):** The stale local variables — computed **before** any evictions occurred — are unconditionally written back to `self`: [6](#0-5) 

This overwrites the correctly-decremented `self.total_tx_size` and `self.total_tx_cycles` with values that do not account for the evictions. For every evicted entry with size `s` and cycles `c`, the pool's totals are permanently inflated by `s` and `c` respectively.

---

### Impact Explanation

**1. Spurious eviction of legitimate transactions.**
`limit_size` uses `self.pool_map.total_tx_size` as the sole guard to decide whether to evict more transactions: [7](#0-6) 

An inflated `total_tx_size` causes `limit_size` to evict additional legitimate transactions that would otherwise fit within `max_tx_pool_size`. This is the direct analog to the M-02 report: an intermediate state where the pool's accounting is misaligned causes downstream operations (eviction decisions) to produce incorrect results unfavorable to legitimate users.

**2. Incorrect RPC-reported pool state.**
`total_tx_size` and `total_tx_cycles` are surfaced directly via the `tx_pool_info` RPC endpoint as `TxPoolInfo.total_tx_size` and `TxPoolInfo.total_tx_cycles`: [8](#0-7) 

Callers (wallets, monitoring tools, fee estimators) receive permanently incorrect values.

**3. No self-correction.** The `recompute_total_stat` fallback in `update_stat_for_remove_tx` is only triggered on **underflow**, not overflow. The inflation persists until the pool is cleared or the node restarts. [9](#0-8) 

---

### Likelihood Explanation

The eviction path inside `check_and_record_ancestors` is reached when all three conditions hold:

1. A submitted transaction T has inputs or cell deps whose outputs are already in the pool (making those pool transactions `cell_ref_parents` of T).
2. The total ancestor count for T exceeds `max_ancestors_count` (default 1000).
3. The condition `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` is satisfied, meaning evicting the cell-ref parents brings the count within limits. [10](#0-9) 

An unprivileged tx-pool submitter (via P2P relay or the `send_transaction` RPC) who controls multiple transactions can craft a chain that satisfies these conditions. No privileged access, 51% hashpower, or Sybil attack is required. The attacker entry path is: submit a crafted transaction via `send_transaction` RPC or P2P relay → `add_entry` is called → eviction path fires → totals are inflated.

---

### Recommendation

Move the stat computation to **after** `check_and_record_ancestors` completes, so it uses the already-updated `self.total_tx_size` as the base:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Pre-check for overflow only (do not store result yet)
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    evicts = self.check_and_record_ancestors(&mut entry)?;

    // Re-compute AFTER evictions have already updated self.total_tx_size/cycles
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

### Proof of Concept

**Initial state:** Pool contains transactions A and B (both `cell_ref_parents`), total size `S`, total cycles `C`. `max_ancestors_count = 1000`, current ancestor count for the incoming tx = 1001.

**Attacker submits transaction T** such that T's inputs reference outputs of A and B, and A+B are `cell_ref_parents` that can be evicted to bring the ancestor count to ≤ 1000.

**Execution trace in `add_entry`:**

| Step | Operation | `self.total_tx_size` | local `total_tx_size` |
|------|-----------|---------------------|----------------------|
| 1 | `updated_stat_for_add_tx(T.size, T.cycles)` | `S` (unchanged) | `S + T.size` |
| 2a | evict A via `remove_entry_and_descendants` | `S - A.size` | `S + T.size` (stale) |
| 2b | evict B via `remove_entry_and_descendants` | `S - A.size - B.size` | `S + T.size` (stale) |
| 3 | `self.total_tx_size = total_tx_size` | `S + T.size` ← **WRONG** | — |

**Correct value:** `S - A.size - B.size + T.size`
**Actual value:** `S + T.size`
**Inflation:** `A.size + B.size`

If `S + T.size > max_tx_pool_size`, `limit_size` will now evict legitimate pending transactions to compensate for the phantom inflation, even though the pool's true occupancy is `S - A.size - B.size + T.size` which may be well within limits.
