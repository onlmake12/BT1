### Title
Tight Infinite Loop on Persistent `indexer.tip()` or `indexer.append()` Error in `try_loop_sync` — (`util/indexer-sync/src/lib.rs`)

---

### Summary

`IndexerSyncService::try_loop_sync` contains two structurally identical instances of the same bug class as the Allora M-15 report: when `indexer.tip()` returns `Err`, or when `indexer.append()` returns `Err`, the loop body falls through to the next iteration with no `break`, no sleep, and no backoff. If either error persists, the function spins in a tight infinite loop, permanently consuming the `spawn_blocking` thread it runs on.

---

### Finding Description

`try_loop_sync` is a `loop {}` that drives the indexer forward one block at a time.

**Case 1 — `indexer.tip()` failure (lines 195–197):**

```rust
Err(e) => {
    error!("Failed to get tip: {}", e);
    // no break, no sleep — falls through to next iteration
}
```

When `indexer.tip()` returns `Err`, the match arm logs the error and exits. Control returns to the top of the `loop {}`. The tip cursor has not advanced, the error condition has not changed, and there is no sleep or backoff. The loop immediately calls `indexer.tip()` again, producing the same error, forever.

**Case 2 — `indexer.append()` failure (lines 160–162 and 186–188):**

```rust
if let Err(e) = indexer.append(&block) {
    error!("Failed to append block: {}. Will attempt to retry.", e);
    // no break, no sleep — falls through to next iteration
}
```

When `append` fails, the indexer tip has not advanced. The next iteration calls `indexer.tip()` again, gets the same tip, fetches the same block, and calls `append` again. The comment "Will attempt to retry" confirms the intent is retry, but there is no delay, making it a tight spin loop.

Both errors originate from RocksDB (`Error::DB` wrapping `rocksdb::Error`), as defined in `util/indexer-sync/src/error.rs` lines 6–13.

The function is dispatched via `spawn_blocking` (lines 214–216 and 240–244 in `spawn_poll`), which draws from Tokio's blocking thread pool. A spinning `spawn_blocking` task permanently occupies one thread. Because `spawn_poll` re-invokes `try_loop_sync` on every new-block notification and on every polling tick, each subsequent call queues another `spawn_blocking` task. If the error is persistent, the blocking thread pool is progressively exhausted.

---

### Impact Explanation

A persistent RocksDB error in the indexer causes `try_loop_sync` to spin at 100% CPU on a blocking thread indefinitely. Repeated new-block notifications (one per block relayed by any peer) each spawn an additional blocking task that also spins. The Tokio blocking thread pool is exhausted, stalling all other `spawn_blocking` work in the node (chain verification, DB reads, etc.), effectively halting the node's ability to process blocks.

---

### Likelihood Explanation

The most realistic external trigger is disk exhaustion: a peer relays a sustained stream of valid, fee-paying transactions that fill the node's disk. Once disk is full, RocksDB write operations fail, causing `indexer.tip()` or `indexer.append()` to return `Error::DB`. From that point the loop spins. The attacker does not need any privileged access; submitting valid transactions via the P2P tx-relay or RPC `send_transaction` endpoint is sufficient to drive disk usage. The indexer is enabled by default in CKB node configurations that expose the indexer RPC module.

---

### Recommendation

**For the `tip()` error arm:** replace the fall-through with a `break` (or at minimum a bounded sleep + break after N retries):

```rust
Err(e) => {
    error!("Failed to get tip: {}", e);
    break;
}
```

**For the `append()` error arm:** either break out of the loop or introduce an exponential backoff with a maximum retry count before breaking:

```rust
if let Err(e) = indexer.append(&block) {
    error!("Failed to append block: {}", e);
    break; // or: sleep + bounded retry counter
}
```

---

### Proof of Concept

1. Start a CKB node with the indexer enabled.
2. Fill the node's disk to capacity (e.g., by submitting a large volume of valid transactions via `send_transaction` RPC, or by relaying many valid blocks).
3. Once disk is full, RocksDB returns `rocksdb::Error` on any write or metadata read.
4. The next new-block notification triggers `spawn_poll` → `spawn_blocking(try_loop_sync)`.
5. Inside `try_loop_sync`, `indexer.tip()` calls into RocksDB and returns `Err(Error::DB(...))`.
6. The `Err(e)` arm at line 195–197 logs the error and falls through; the loop immediately retries.
7. Each subsequent new-block notification (one per block from any peer) spawns another `spawn_blocking` task that also spins.
8. The Tokio blocking thread pool fills up; `spawn_blocking` calls from chain verification and other subsystems begin to queue and stall.
9. The node stops processing new blocks.
