Audit Report

## Title
Stale `total_difficulty` Reference Allows Outbound Peers to Escape Eviction Indefinitely — (`File: sync/src/synchronizer/mod.rs`)

## Summary
In `Synchronizer::eviction()`, when Branch C fires and extends `timeout` to `now + EVICTION_HEADERS_RESPONSE_TIME`, it updates only `sent_getheaders` and `timeout` but leaves `work_header` and `total_difficulty` at their original values (D₁). If the local chain advances to D₂ > D₁ during that window and the peer sends headers only up to D₁, the next eviction tick evaluates Branch B's condition `peer.best_known_difficulty >= state.chain_sync.total_difficulty` as true (D₁ >= D₁), resetting the entire timeout cycle with a fresh `CHAIN_SYNC_TIMEOUT` and `sent_getheaders = false`. The peer permanently escapes eviction without ever responding to the GetHeaders challenge.

## Finding Description
`ChainSyncState` in `sync/src/types/mod.rs` (L70–77) holds four coupled fields: `timeout`, `work_header`, `total_difficulty`, and `sent_getheaders`. The `eviction()` function in `sync/src/synchronizer/mod.rs` processes outbound peers through three mutually exclusive branches:

- **Branch A** (L590–600): peer's best-known difficulty ≥ current local tip → clear all four fields.
- **Branch B** (L601–614): `timeout == 0` OR `peer.best_known_difficulty >= state.chain_sync.total_difficulty` → set all four fields to current tip, fresh `CHAIN_SYNC_TIMEOUT`.
- **Branch C** (L615–641): `timeout > 0 && now > timeout` → if `sent_getheaders`, evict; else extend deadline.

Branch C's else path (L627–639) writes:
```rust
state.chain_sync.sent_getheaders = true;
state.chain_sync.timeout = now + EVICTION_HEADERS_RESPONSE_TIME;
// work_header and total_difficulty are NOT refreshed
```

After this, `total_difficulty` still holds D₁. If the local chain mines new blocks advancing to D₂ > D₁, and the peer sends headers reaching exactly D₁, the next eviction call finds:
- Branch A: false (`peer.best_known_difficulty` D₁ < D₂)
- Branch B: true (`peer.best_known_difficulty` D₁ >= `state.chain_sync.total_difficulty` D₁)

Branch B fires, resetting `timeout = now + CHAIN_SYNC_TIMEOUT`, `sent_getheaders = false`, and updating the reference to D₂. The peer is not evicted and receives a completely fresh timeout window. This cycle repeats indefinitely.

## Impact Explanation
An outbound connection slot is permanently occupied by a peer that never genuinely responds to the GetHeaders challenge. CKB limits the number of outbound peers; exhausting these slots degrades the node's ability to sync from honest peers and weakens eclipse-attack resistance. This matches **Low (501–2000 points): Any other important performance/security improvements for CKB**, with potential escalation toward Medium if combined with Sybil occupation of all outbound slots to fully eclipse a node.

## Likelihood Explanation
The precondition — local chain advancing during `EVICTION_HEADERS_RESPONSE_TIME` — is the normal operating condition of any live syncing node. An attacker needs only one outbound connection slot. The attacker can observe the local tip at the time the first timeout fires (via public RPC or block announcements) and trickle headers up to that stale D₁ reference before the second deadline. No privileged access, no majority hash power, and no Sybil attack is required for the basic slot-occupation exploit.

## Recommendation
In Branch C's else path, refresh `work_header` and `total_difficulty` to the current local tip alongside the timeout extension:

```rust
state.chain_sync.sent_getheaders = true;
state.chain_sync.timeout = now + EVICTION_HEADERS_RESPONSE_TIME;
// Refresh reference so Branch B cannot fire on a stale baseline:
state.chain_sync.work_header = Some(tip_header.clone());
state.chain_sync.total_difficulty = Some(local_total_difficulty.clone());
active_chain.send_getheaders_to_peer(
    nc, *peer,
    state.chain_sync.work_header.as_ref().expect("work_header be assigned").into(),
);
```

This ensures that after the second deadline, Branch B's comparison always uses the current tip, so a peer that only caught up to the old D₁ cannot trigger a reset.

## Proof of Concept
1. Attacker connects as an outbound peer; announces best-known header with difficulty D₀ < D₁ (local tip).
2. Branch B fires: `timeout = T + CHAIN_SYNC_TIMEOUT`, `total_difficulty = D₁`, `sent_getheaders = false`.
3. At `T + CHAIN_SYNC_TIMEOUT + ε`, Branch C fires: `timeout = T + CHAIN_SYNC_TIMEOUT + EVICTION_HEADERS_RESPONSE_TIME`, `total_difficulty` remains D₁, `sent_getheaders = true`.
4. Local node mines new blocks; tip advances to D₂ > D₁.
5. Attacker sends headers up to exactly D₁ (not D₂) before the second deadline.
6. Next `eviction()` call: `peer.best_known_difficulty (D₁) >= state.chain_sync.total_difficulty (D₁)` → Branch B fires → `timeout = now + CHAIN_SYNC_TIMEOUT`, `sent_getheaders = false`. Peer is not evicted.
7. Repeat from step 3 indefinitely.

A unit test can be written against `Synchronizer::eviction()` by: (a) setting up a peer state with `sent_getheaders = true`, `total_difficulty = D₁`, `timeout = now - 1`; (b) advancing the mock chain tip to D₂; (c) setting `peer.best_known_difficulty = D₁`; (d) calling `eviction()` and asserting the peer is added to the eviction list rather than having its timeout reset.
