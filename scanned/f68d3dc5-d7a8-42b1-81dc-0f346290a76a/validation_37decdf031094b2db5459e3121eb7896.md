Audit Report

## Title
`DownloadScheduler::decrease` Overwrites `timeout_count` With `task_count` Instead of Accumulating — (File: sync/src/types/mod.rs)

## Summary
In `DownloadScheduler::decrease`, line 522 reads `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count = self.timeout_count.saturating_add(num)`. Because `task_count` starts at 32 and `num` is 1 or 2, the right-hand side always evaluates to a value well above 2, so the `if self.timeout_count > 2` guard fires on every single call. This causes `task_count` to be decremented on every slow-block event rather than every third one, exhausting a peer's download concurrency budget ~3× faster than the protocol intends.

## Finding Description
`DownloadScheduler` ( [1](#0-0) ) tracks per-peer concurrent block download capacity. `task_count` is initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) and `timeout_count` to 0. The `decrease` function is intended to accumulate timeout events and only reduce `task_count` after more than 2 have been seen:

```rust
// sync/src/types/mod.rs L521-527
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // BUG: uses task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [2](#0-1) 

Because `self.task_count` (not `self.timeout_count`) is used on the right-hand side, `timeout_count` is always set to `task_count + num`. With `task_count ≥ 2` and `num ≥ 1`, this is always `> 2`, so the branch fires unconditionally — decrementing `task_count` by 1 and resetting `timeout_count` to 0 on every call.

`decrease` is called from `remove_by_block` when a block arrives in the `NormalToUpper` or `UpperToMax` time quantile and `should_punish` is true: [3](#0-2) 

`should_punish` is set when the number of connected peers exceeds `protect_num`: [4](#0-3) 

Once `task_count` reaches 0, `can_fetch()` returns 0: [5](#0-4) 

and the block fetcher skips that peer entirely.

## Impact Explanation
This is a sync performance degradation. A peer's download budget is drained after ~32 slow-block events instead of ~96, causing the node to stop requesting blocks from that peer prematurely. This maps to **Low (501–2000 points): Any other important performance improvements for CKB** — the bug measurably degrades IBD and normal sync throughput, particularly when the node has a small peer set.

## Likelihood Explanation
The conditions are met during normal operation: the node has more than `protect_num` connected peers (typical during IBD), any peer delivers blocks in the upper latency quantiles (common under load or network jitter), and `adjustment` is `true` (the default). No special privileges are required. A peer that deliberately delays block responses can trigger this path repeatedly, but even without a malicious actor, ordinary network latency will cause the budget to drain 3× faster than intended.

## Recommendation
Fix line 522 to accumulate into `self.timeout_count` rather than overwrite it with `self.task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // fix: was self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

## Proof of Concept
Confirmed by reading the code at [2](#0-1) . A unit test can be added to `sync/src/tests/inflight_blocks.rs` alongside the existing `punish` test at line 117:

1. Construct a `DownloadScheduler` with default state (`task_count = 32`, `timeout_count = 0`).
2. Call `decrease(1)` three times.
3. **With the bug**: `task_count` is 29 after 3 calls (decremented each time).
4. **With the fix**: `task_count` is still 32 after 2 calls; only after the 3rd call does `timeout_count` exceed 2 and `task_count` become 31.

The existing test at line 117 (`assert_eq!(inflight_blocks.peer_can_fetch_count(2.into()), 32 >> 4)`) confirms `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` and that `punish` uses right-shift, validating the starting value used in the PoC. [6](#0-5)

impl DownloadScheduler {
    fn inflight_count(&self) -> usize {
        self.hashes.len()
    }

    fn can_fetch(&self) -> usize {
        self.task_count.saturating_sub(self.hashes.len())
    }

    pub(crate) const fn task_count(&self) -> usize {
        self.task_count
    }

    fn increase(&mut self, num: usize) {
        if self.task_count < MAX_BLOCKS_IN_TRANSIT_PER_PEER {
            self.task_count = ::std::cmp::min(
                self.task_count.saturating_add(num),
                MAX_BLOCKS_IN_TRANSIT_PER_PEER,
            )
        }
    }

    fn decrease(&mut self, num: usize) {
        self.timeout_count = self.task_count.saturating_add(num);
        if self.timeout_count > 2 {
            self.task_count = self.task_count.saturating_sub(1);
            self.timeout_count = 0;
        }
    }

    fn punish(&mut self, exp: usize) {
        self.task_count >>= exp
    }
}
