Audit Report

## Title
`load_data_as_code` Charges Cycles on `memory_size` Instead of `content_size`, Enabling Low-Cost Verify Worker Saturation — (`script/src/syscalls/load_cell_data.rs`)

## Summary
The `load_data_as_code` function (syscall 2091) charges `transferred_byte_cycles(memory_size)` after a successful `init_pages` call, regardless of how many bytes of actual cell data (`content_size`) were loaded. An attacker can set `memory_size` to a value far exceeding `content_size` — bounded only by the VM's 128 MB address space — and consume tens of millions of cycles per syscall invocation with a 1-byte cell dep, forcing every tx-pool verify worker that picks up the transaction to run near the cycle limit before rejection.

## Finding Description
In `load_data_as_code` (`script/src/syscalls/load_cell_data.rs`, lines 148–163), the only pre-charge guard is `content_size > memory_size`, which returns `SLICE_OUT_OF_BOUND`. There is no upper bound on `memory_size` itself:

```rust
// lines 148–154
if content_offset >= cell.len() as u64
    || content_end > cell.len() as u64
    || content_size > memory_size   // only guard: content_size ≤ memory_size
{
    machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
    return Ok(());
}
// lines 155–163
machine.memory_mut().init_pages(
    addr, memory_size, FLAG_EXECUTABLE | FLAG_FREEZED,
    Some(cell.slice(content_offset..content_end)), 0,
)?;
sc.track_pages(machine, addr, memory_size, &data_piece_id, content_offset)?;
machine.add_cycles_no_checking(transferred_byte_cycles(memory_size))?; // ← memory_size, not content_size
```

With `content_size = 1` (a 1-byte cell dep) and `memory_size = 64 MB`:
- The bounds check passes (`1 ≤ 64 MB`)
- `init_pages` initializes 64 MB of VM address space (real page-initialization work)
- Cycles charged: `transferred_byte_cycles(64 MB) = 64×1024×1024 / 4 = 16,777,216`

Four such calls per transaction consume `4 × 16,777,216 = 67,108,864` cycles, approximately 96% of `DEFAULT_MAX_TX_VERIFY_CYCLES` (`TWO_IN_TWO_OUT_CYCLES × 20 = 3,500,000 × 20 = 70,000,000`). The transaction is then rejected, but the verify worker has already performed the full `init_pages` work for each call.

Existing checks are insufficient: the `content_size > memory_size` guard only prevents the data slice from exceeding the mapped region; it does not cap `memory_size` itself. No fee-per-cycle mechanism exists in the tx pool — fees are based on transaction byte size only (`min_fee_rate = 1000 shannons/KB`).

## Impact Explanation
Submitting `N ≥ max_tx_verify_workers` (default: `max(num_cpus * 3/4, 1)`) such transactions simultaneously pins all verify workers at near-cycle-limit execution. Legitimate transactions queue behind them indefinitely. This concretely matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. The attack degrades tx-pool admission throughput for all honest users without requiring any consensus or PoW bypass.

## Likelihood Explanation
- **Unprivileged**: Any user can submit transactions via RPC or P2P relay.
- **Cheap**: A 1-byte cell dep (minimum ~61 CKB capacity, one-time cost, reusable across unlimited transactions) plus ~300 shannons per transaction (~300-byte tx at 1000 shannons/KB).
- **Repeatable**: The same cell dep can be referenced by arbitrarily many transactions. The attacker continuously floods the verify queue.
- **No special knowledge required**: The syscall number (2091), register convention, and VM address layout are all publicly documented.

## Recommendation
Charge cycles based on `content_size` (the actual bytes transferred into the VM), not `memory_size` (the virtual address range reserved). The zero-padding of `memory_size - content_size` bytes is a VM-internal page-initialization cost that should not be exposed as a user-controllable cycle multiplier:

```rust
// script/src/syscalls/load_cell_data.rs, line 163
// Change:
machine.add_cycles_no_checking(transferred_byte_cycles(memory_size))?;
// To:
machine.add_cycles_no_checking(transferred_byte_cycles(content_size))?;
```

Alternatively, add an explicit cap on `memory_size` (e.g., `content_size` rounded up to the next page boundary) before the `init_pages` call.

## Proof of Concept
1. Deploy a cell dep with 1 byte of data (`0x01`).
2. Write a minimal RISC-V ELF script that:
   - Sets `A0 = 0x20000` (addr, free VM space above ELF load address at `0x10000`)
   - Sets `A1 = 64 * 1024 * 1024` (`memory_size = 64 MB`)
   - Sets `A2 = 0` (`content_offset`)
   - Sets `A3 = 1` (`content_size = 1 byte`)
   - Sets `A4 = 0` (index)
   - Sets `A5 = source` (CellDep)
   - Sets `A7 = 2091` (`LOAD_CELL_DATA_AS_CODE_SYSCALL_NUMBER`)
   - Issues `ecall`; repeats 4 times (4 × 16,777,216 = 67,108,864 cycles ≈ 96% of budget)
3. Submit `N ≥ max_tx_verify_workers` such transactions simultaneously.
4. Observe: all verify workers are pinned at near-cycle-limit; the verify queue grows; legitimate transactions stall.

Expected cycle measurement per call: `transferred_byte_cycles(64 MB) = 16,777,216`, confirmed by `script/src/cost_model.rs` lines 10–12. Each transaction is rejected after hitting `DEFAULT_MAX_TX_VERIFY_CYCLES = 70,000,000`, but only after the full `init_pages` work has been performed.
