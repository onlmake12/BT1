Audit Report

## Title
`Scheduler::resume()` Unconditionally Zeroes `iteration_cycles`, Discarding Pre-Suspension IO Cycles and Understating Total Consumed Cycles — (File: `script/src/scheduler.rs`)

## Summary

`Scheduler::resume()` sets `iteration_cycles = 0` after calling `ensure_vms_instantiated`, but this also silently discards the non-zero `iteration_cycles` value that was faithfully saved by `suspend()`. Those cycles — charged by `ensure_vms_instantiated` calls inside `process_io()` during the final iteration before suspension — are not yet reflected in `total_cycles`. Discarding them permanently understates the scheduler's total consumed cycles, allowing a script to execute beyond the enforced cycle limit across each suspend/resume boundary.

## Finding Description

**Cycle accounting invariant.** The scheduler tracks consumed cycles across three accumulators whose sum equals total consumed cycles at any instant: `total_cycles`, `iteration_cycles`, and `machine.cycles()`. The code and comments explicitly acknowledge this: [1](#0-0) 

**`iterate_outer` flow.** In `iterate_outer`, after `iterate_inner` returns (storing any `Err(Pause)` in `iterate_return`), the function commits `iteration_cycles` to `total_cycles` via `consume_cycles`, resets `iteration_cycles = 0`, then calls `process_io()`. `process_io()` calls `ensure_vms_instantiated` for read/write VM pairs, which charges cycles directly into `iteration_cycles`. Only after that does `iterate_return?` propagate the `Err(Pause)`: [2](#0-1) 

**`suspend()` correctly preserves non-zero `iteration_cycles`.** The code and `FullSuspendedState` documentation both acknowledge that `iteration_cycles` will not always be zero at suspension boundaries due to this exact behavior, and explicitly preserve it: [3](#0-2) [4](#0-3) 

**`resume()` discards the preserved value.** `resume()` initially copies `full.iteration_cycles` into the new scheduler (line 216), then calls `ensure_vms_instantiated`, then unconditionally zeroes `iteration_cycles` at line 242. The stated intent is to avoid charging cycles for resume overhead — but the reset also destroys the pre-suspension `iteration_cycles` that represent real script execution work (IO data-transfer cycles charged during `process_io`): [5](#0-4) 

The cycles lost are not in `total_cycles` (they were charged after the `consume_cycles` call and the `iteration_cycles = 0` reset in `iterate_outer`). They vanish entirely.

**`chunk_run` suspend/resume path.** When `run()` propagates `Err(Pause)`, `chunk_run` calls `scheduler.suspend()` and returns `ChunkState::Suspended`. On the next chunk, `resume_scheduler` calls `Scheduler::resume()`, which zeroes the saved `iteration_cycles`: [6](#0-5) 

## Impact Explanation

After resume, `total_cycles` is understated by the magnitude of the discarded `iteration_cycles`. The per-chunk cycle budget (`limit_cycles`) is not reduced to compensate, so the script receives extra execution budget equal to the lost IO cycles on every resume. Across N suspend/resume cycles (the tx-pool calls `resume_from_state` in a loop for chunked verification), the undercount compounds to `N × (IO cycles per boundary)`. The total cycle count reported on script completion is therefore understated, allowing a transaction that genuinely exceeds the declared cycle limit to be accepted by the tx-pool. If a miner includes such a transaction, the consensus-layer non-resumable re-verification will reject the block, wasting mining effort and causing chain-tip instability. This matches **"Incorrect implementation or behavior of CKB-VM or system scripts" (High, 10001–15000 points)** and potentially **"Vulnerabilities which could easily damage CKB economy" (Critical)** if the bypass is large enough to be systematically exploited.

## Likelihood Explanation

No privileged access is required. Any transaction submitter can craft a script using `spawn` and `pipe` syscalls so that pending IO operations are in flight when the tx-pool's `ChunkCommand::Suspend` signal fires. At that moment `process_io()` will call `ensure_vms_instantiated` for the read/write VM pair, charging cycles into `iteration_cycles`. On the next chunk, `resume()` discards those cycles. The attack is repeatable across every suspend/resume boundary and requires only submitting a transaction.

## Recommendation

In `resume()`, save the pre-suspension `iteration_cycles` before calling `ensure_vms_instantiated`, then restore it afterward. Only the cycles added by `ensure_vms_instantiated` itself (resume overhead) should be discarded:

```rust
let saved_iteration_cycles = full.iteration_cycles;
// ... build scheduler with iteration_cycles: full.iteration_cycles ...
scheduler.ensure_vms_instantiated(&full.instantiated_ids).unwrap();
// Discard only resume-overhead cycles; restore pre-suspension IO cycles.
scheduler.iteration_cycles = saved_iteration_cycles;
```

Alternatively, `suspend()` should flush `iteration_cycles` into `total_cycles` before saving state, ensuring `iteration_cycles` is always zero at suspension boundaries and making the reset in `resume()` safe. The latter approach also requires updating the `current cycles` syscall behavior, which the existing comments flag as a known concern for future hardfork versions.

## Proof of Concept

1. Submit a transaction containing a script that spawns a child VM and opens a pipe between them.
2. The child VM writes data; the parent VM reads. Both are in IO-wait state at the end of an iteration.
3. The tx-pool issues `ChunkCommand::Suspend`, triggering `Err(Pause)` from `iterate_inner`.
4. `iterate_outer` resets `iteration_cycles = 0`, then calls `process_io()`.
5. `process_io()` calls `ensure_vms_instantiated(&[read_vm_id, write_vm_id])`, charging, e.g., 50 cycles into `iteration_cycles`.
6. `iterate_outer` propagates `Err(Pause)`; `chunk_run` calls `scheduler.suspend()`.
7. `suspend()` saves: `total_cycles = T`, `iteration_cycles = 50`.
8. `resume()` restores `total_cycles = T`, then sets `iteration_cycles = 0`. The 50 cycles are permanently lost.
9. Next chunk runs with full `max_cycles` budget; the scheduler believes only `T` cycles have been consumed, actual is `T + 50`.
10. Repeat across N chunks: `N × 50` cycles of bypass accumulate. A script declared at limit `L` can actually consume `L + N×50` cycles while the tx-pool reports `L`. [7](#0-6) [8](#0-7)

    fn iterate_outer(
        &mut self,
        pause: &Pause,
        limit_cycles: Cycle,
    ) -> Result<(VmId, Cycle), Error> {
        let iterate_return = self.iterate_inner(pause.clone(), limit_cycles);
        self.consume_cycles(self.iteration_cycles)?;
        let remaining_cycles = limit_cycles
            .checked_sub(self.iteration_cycles)
            .ok_or(Error::CyclesExceeded)?;
        // Clear iteration cycles intentionally after each run
        self.iteration_cycles = 0;
        // Process all pending VM reads & writes. Notice ideally, this invocation
        // should be put at the end of `iterate_inner` function. However, 2 things
        // prevent this:
        //
        // * In earlier implementation of the Meepo hardfork version, `self.process_io`
        // was put at the very start of +iterate_prepare_machine+ method. Meaning we used
        // to process IO syscalls at the very start of a new iteration.
        // * Earlier implementation contains a bug that cycles consumed by suspending / resuming
        // VMs are not updated in the subsequent VM's `current cycles` syscalls.
        //
        // To make ckb-script package suitable for outside usage, we want IOs processed at
        // the end of each iteration, not at the start of the next iteration. We also need
        // to replicate the exact same runtime behavior of Meepo hardfork. This means the only
        // viable change will be:
        //
        // * Move `self.process_io` call to the very end of `iterate_outer` method, which is
        // exactly current location
        // * For now we have to live with the fact that `iteration_cycles` will not always be
        // zero at iteration boundaries, and also preserve its value in `FullSuspendedState`.
        //
        // One expected change is that +process_io+ is now called once more
        // after the whole scheduler terminates, and not called at the very beginning
        // when no VM is executing. But since no VMs will be in IO states at this 2 timeslot,
        // we should be fine here.
        self.process_io()?;
        let id = iterate_return?;
        Ok((id, remaining_cycles))
    }

    fn chunk_run(
        &self,
        script_group: &ScriptGroup,
        max_cycles: Cycle,
        state: &Option<FullSuspendedState>,
    ) -> Result<ChunkState, ScriptError> {
        let mut scheduler = if let Some(state) = state {
            self.resume_scheduler(script_group, state)
        } else {
            self.create_scheduler(script_group)
        }?;
        let previous_cycles = scheduler.consumed_cycles();
        let res = scheduler.run(RunMode::LimitCycles(max_cycles));
        match res {
            Ok(TerminatedResult {
                exit_code,
                consumed_cycles: cycles,
            }) => {
                if exit_code == 0 {
                    Ok(ChunkState::Completed(
                        cycles,
                        scheduler.consumed_cycles() - previous_cycles,
                    ))
                } else {
                    Err(ScriptError::validation_failure(
                        &script_group.script,
                        exit_code,
                    ))
                }
            }
            Err(error) => match error {
                VMInternalError::CyclesExceeded | VMInternalError::Pause => {
                    let snapshot = scheduler
                        .suspend()
                        .map_err(|err| self.map_vm_internal_error(err, max_cycles))?;
                    Ok(ChunkState::suspended(snapshot))
                }
                _ => Err(self.map_vm_internal_error(error, max_cycles)),
            },
        }
    }
