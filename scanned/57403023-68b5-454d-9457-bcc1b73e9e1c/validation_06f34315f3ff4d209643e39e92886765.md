### Title
Unauthenticated `from` Field in `ConnectionRequest` Allows `pending_delivered` Poisoning — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

An unprivileged remote peer can poison the `pending_delivered` map on any target node by sending a `ConnectionRequest` whose `from` field is set to an arbitrary victim `PeerId`. Because the protocol never verifies that `content.from` matches the actual session's `PeerId`, the attacker can (a) block the victim's legitimate hole-punching attempt via the interval guard, and (b) cause the target node to attempt NAT traversal toward attacker-controlled addresses when a subsequent `ConnectionSync` arrives.

---

### Finding Description

**Root cause — no binding between session identity and `content.from`.**

In `mod.rs`, the `ConnectionRequest` handler is constructed with `context.session.id` (a `PeerIndex`, i.e. the transport-layer session handle), but `content.from` is parsed purely from the message payload: [1](#0-0) 

There is no code anywhere in the hole-punching stack that looks up the actual `PeerId` of the session and compares it to `content.from`. The `peer` field stored in `ConnectionRequestProcess` is only used to send the reply back to the same session: [2](#0-1) 

**The poisoning write.**

`respond_delivered()` unconditionally inserts the attacker-supplied addresses under the attacker-supplied `from_peer_id`: [3](#0-2) 

**The only guard is a time-window check, not an identity check.**

The sole protection is a 2-minute cooldown keyed on `from_peer_id`: [4](#0-3) 

This guard is itself exploitable in two directions:

1. **Pre-poisoning (DoS):** If the attacker sends the spoofed request *before* any legitimate entry exists for the victim, the entry is created immediately. When the victim's real `ConnectionRequest` later arrives, the interval guard fires and the target node returns `Ignore` *without* sending `ConnectionRequestDelivered` — the victim's hole-punching attempt silently fails.

2. **Post-poisoning (misdirection):** After the 2-minute window elapses on a legitimate entry, the attacker overwrites it. Any subsequent `ConnectionSync{from=victim_peer_id}` causes the target node to attempt NAT traversal toward the attacker's address: [5](#0-4) 

**The `forward_rate_limiter` does not help.**

The forward rate limiter is keyed by `(content.from, content.to, msg_item_id)` — all attacker-controlled values — and allows 1 request per second per tuple: [6](#0-5) 

The attacker only needs to send a single request, so this limiter provides no meaningful protection.

**`listen_addrs` validation does not prevent the attack.**

The address filter only rejects non-TCP and non-IP transports; a valid `attacker_ip:port` TCP address passes through: [7](#0-6) 

---

### Impact Explanation

- **Denial of service on hole punching:** The attacker can permanently suppress hole-punching for any victim `PeerId` on any target node the attacker is connected to, by refreshing the poisoned entry every 2 minutes. The victim can never complete NAT traversal through that target node.
- **Misdirected NAT traversal:** The target node opens outbound TCP connections toward the attacker's address, leaking the target's IP/port and potentially allowing the attacker to observe or interfere with the connection attempt.
- **Scope:** Confined to the hole-punching protocol. Consensus, funds, and block relay are unaffected.

---

### Likelihood Explanation

- Requires only a standard P2P connection to the target node — no privilege, no key material, no hashpower.
- The victim's `PeerId` is public (advertised via the discovery protocol).
- The attack is repeatable every 2 minutes with a single message per cycle.
- Locally testable without any mainnet interaction.

---

### Recommendation

In `respond_delivered()`, verify that `from_peer_id` matches the actual `PeerId` of the session (`self.peer`) when the message arrives with an empty `route` (i.e., it was sent directly, not forwarded). For forwarded messages, the intermediate node's session identity cannot be used, but the route itself provides a chain of accountability that can be checked. At minimum, when `content.route` is empty and `self_peer_id == &content.to`, require that the session's resolved `PeerId` equals `content.from` before writing to `pending_delivered`.

### Proof of Concept

```
1. Attacker (A) establishes a normal P2P connection to target node B.
2. A learns victim V's PeerId (from discovery).
3. A sends ConnectionRequest { from=V_peer_id, to=B_peer_id,
       listen_addrs=[attacker_ip:port], max_hops=6, route=[] }.
4. B: self_peer_id == content.to → calls respond_delivered(V_peer_id, ...).
5. B: pending_delivered has no entry for V → interval check passes.
6. B: sends ConnectionRequestDelivered back to A's session (harmless).
7. B: pending_delivered.insert(V_peer_id, ([attacker_ip:port], now)).

DoS path:
8. V sends a legitimate ConnectionRequest { from=V, to=B } through the network.
9. B: respond_delivered(V_peer_id) → interval check fires (< 2 min) → returns Ignore.
10. B never sends ConnectionRequestDelivered to V; V's hole-punch fails.

Misdirection path (after 2 min):
8. A repeats step 3 after HOLE_PUNCHING_INTERVAL elapses.
9. B overwrites: pending_delivered[V] = ([attacker_ip:port], now).
10. A legitimate ConnectionSync { from=V, to=B } arrives at B.
11. B: pending_delivered.get(V) → [attacker_ip:port].
12. B spawns try_nat_traversal(bind_addr, attacker_ip:port) → connects to attacker.
```

State assertion: after step 7, `pending_delivered[V_peer_id].0 == [attacker_ip:port]`, violating the invariant that entries must correspond to authenticated peer addresses.
