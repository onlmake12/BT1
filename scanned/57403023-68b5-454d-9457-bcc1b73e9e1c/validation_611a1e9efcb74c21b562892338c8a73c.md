### Title
Silent Truncating Cast in NervosDAO Withdrawal Capacity Calculation Produces Incorrect Withdrawal Amount - (File: util/dao/src/lib.rs)

### Summary

In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result of the accumulate-rate-scaled capacity is cast to `u64` using a silent truncating `as u64` cast. If the product overflows `u64::MAX`, the cast silently discards the high bits and returns a drastically incorrect (too-small) withdrawal capacity instead of an error. This is inconsistent with every other analogous arithmetic operation in the same file, which all use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The existing overflow test even documents the expected behavior as returning an error, but the implementation silently truncates instead.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the scaled withdrawal capacity using the NervosDAO accumulate rate (`ar`) ratio — the direct CKB analog of the Fuji Protocol's index:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The cast `withdraw_counted_capacity as u64` is a **silent truncating cast**. In Rust, `as` casts on integers never panic and never return errors — they silently discard the upper bits when the value exceeds `u64::MAX`. If `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`, the result is silently wrapped to a small incorrect value, and `safe_add(occupied_capacity)` then succeeds, returning a valid but wrong `Capacity`.

Every other analogous `u128 → u64` narrowing in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern:

```rust
// dao_field_with_current_epoch — correct pattern:
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

```rust
// secondary_block_reward — correct pattern:
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The existing unit test `check_withdraw_calculation_overflows` explicitly documents the expected behavior as returning an error on overflow:

```rust
assert!(result.is_err());
``` [4](#0-3) 

But because `as u64` silently truncates, the function returns `Ok(truncated_small_value)` instead of `Err(DaoError::Overflow)`, meaning the test documents the correct behavior but the implementation does not match it.

---

### Impact Explanation

`calculate_maximum_withdraw` is called in two critical paths:

1. **Transaction verification** (`transaction_maximum_withdraw` → `withdrawed_interests` → `dao_field_with_current_epoch`): The DAO state field embedded in each block header is computed using this function. A silent truncation here causes the on-chain `ar`/`s` accounting to diverge from the correct value, corrupting the global DAO state for all subsequent depositors. [5](#0-4) 

2. **RPC `calculate_dao_maximum_withdraw`**: Users and wallets call this to determine how much CKB to claim in their withdrawal transaction. A truncated return value causes the user to construct a transaction claiming far less than they are owed, resulting in direct fund loss. [6](#0-5) 

The depositor loses the difference between the correct withdrawal amount and the silently truncated amount, with no error or warning.

---

### Likelihood Explanation

The overflow requires `counted_capacity * withdrawing_ar > u64::MAX * deposit_ar`. Since `deposit_ar` starts at `10_000_000_000_000_000` (10^16) and grows slowly, and the total CKB supply (~33.6 billion CKB = ~3.36 × 10^18 shannons) is well below `u64::MAX` (~1.844 × 10^19 shannons), triggering this in practice requires a cell holding a very large fraction of the total supply AND a very long deposit period. Under current mainnet parameters this is extremely unlikely. However, the bug is a real correctness defect: the implementation contradicts its own test specification, and the silent truncation path is reachable by any RPC caller or transaction submitter who provides a sufficiently large DAO cell.

---

### Recommendation

Replace the silent truncating cast with the same checked conversion used everywhere else in the file:

```rust
// Before (incorrect):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (correct):
let withdraw_counted = u64::try_from(withdraw_counted_capacity)
    .map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted).safe_add(occupied_capacity)?
```

This makes the overflow behavior consistent with `ar_increase` and `miner_issuance` calculations in the same file, and makes the existing `check_withdraw_calculation_overflows` test pass as intended.

---

### Proof of Concept

The existing test at `util/dao/src/lib.rs` already demonstrates the discrepancy. With:
- `output.capacity = 18_446_744_073_709_550_000` shannons
- `deposit_ar = 10_000_000_000_123_456`, `withdrawing_ar = 10_000_000_001_123_456`

The intermediate value `counted_capacity * withdrawing_ar / deposit_ar` exceeds `u64::MAX`. The `as u64` cast silently truncates it to a small value (the lower 64 bits of the u128 result), and `safe_add(occupied_capacity)` succeeds, returning `Ok(truncated_small_value)`. The test asserts `result.is_err()` — which fails — proving the implementation is incorrect. [4](#0-3)
