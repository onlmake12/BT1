### Title
Systematic Integer Truncation in `ar_increase` Causes Permanent NervosDAO Interest Deficit — (File: `util/dao/src/lib.rs`)

---

### Summary

Every block, the NervosDAO accumulation rate (`ar`) is updated via integer division that silently truncates the result. Because `ar` grows slightly slower than the true continuous rate, NervosDAO depositors receive slightly less interest than they are entitled to on every withdrawal. The truncated remainder accumulates permanently in the `s` field (the NervosDAO secondary-issuance accumulator) and can never be withdrawn by any depositor, representing a systematic, compounding loss of value to all NervosDAO participants.

---

### Finding Description

In `dao_field_with_current_epoch` (`util/dao/src/lib.rs`), the accumulation rate is updated each block:

```rust
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
let current_ar = parent_ar
    .checked_add(ar_increase)
    .ok_or(DaoError::Overflow)?;
``` [1](#0-0) 

The integer division `parent_ar * current_g2 / parent_c` truncates the fractional part. The true `ar_increase` is `parent_ar * current_g2 / parent_c`, but the stored value is `floor(parent_ar * current_g2 / parent_c)`. The truncation error per block is in the range `[0, 1)` units of `ar`.

`ar` starts at `DEFAULT_GENESIS_ACCUMULATE_RATE = 10_000_000_000_000_000` (10^16): [2](#0-1) 

When a depositor withdraws, the payout is computed as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
``` [3](#0-2) 

Because `ar` at withdrawal time (`withdrawing_ar`) has accumulated more truncation error than `ar` at deposit time (`deposit_ar`), the ratio `withdrawing_ar / deposit_ar` is systematically less than the true ratio. The depositor receives less interest than they are owed.

The `s` field is updated as:

```rust
let current_s = parent_s
    .safe_add(nervosdao_issuance)
    .and_then(|s| s.safe_sub(withdrawed_interests))?;
``` [4](#0-3) 

Because `withdrawed_interests` is computed using the underestimated `ar`, it is slightly less than the true interest owed. The difference accumulates in `s` and is never distributed to any party — it is permanently stranded.

A secondary truncation in `miner_issuance` compounds this:

```rust
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
``` [5](#0-4) 

This floors the miner's share, routing the remainder into `nervosdao_issuance` and thus into `s`. Combined with the `ar_increase` truncation, `s` accumulates more than depositors can ever withdraw.

Notably, the block-reward distribution explicitly handles its remainder via `remainder_reward`:

```rust
let block_reward = Capacity::shannons(epoch_reward.as_u64() / genesis_epoch_length);
let remainder_reward = Capacity::shannons(epoch_reward.as_u64() % genesis_epoch_length);
``` [6](#0-5) 

The `ar_increase` path has no analogous remainder-recovery mechanism, confirming the omission is unintentional.

---

### Impact Explanation

Every NervosDAO depositor receives slightly less interest than the protocol intends. The deficit per depositor per block is bounded by `counted_capacity / deposit_ar` (at most 1 shannon for typical deposit sizes), but it is additive across all depositors and all blocks. The stranded value in `s` grows monotonically and is irrecoverable under the current design. For large deposits held over long periods, the cumulative shortfall is measurable in shannons and grows without bound over the chain's lifetime.

**Severity: Low — Impact: Medium — Likelihood: High (deterministic, every block)**

---

### Likelihood Explanation

This truncation fires unconditionally on every block that processes the DAO field update. It requires no special attacker action; any NervosDAO depositor who deposits and later withdraws will observe the deficit. The entry path is the standard tx-pool submission of a DAO deposit transaction followed by a withdrawal transaction — both are fully accessible to any unprivileged user.

---

### Recommendation

Carry the fractional remainder of `ar_increase` forward across blocks, similar to how `remainder_reward` is tracked for block rewards. One approach:

```rust
// Store ar_remainder in the DAO field or a separate accumulator
let ar_numerator = u128::from(parent_ar) * u128::from(current_g2.as_u64())
    + u128::from(carried_remainder);
let ar_increase = ar_numerator / u128::from(parent_c.as_u64());
let new_remainder = ar_numerator % u128::from(parent_c.as_u64());
```

Alternatively, increase the precision of `ar` (e.g., use a 128-bit field) so that the truncation error is negligible over the chain's lifetime. At minimum, document the known truncation and its upper bound per block so that economic analyses account for it.

---

### Proof of Concept

Using mainnet genesis data from `util/dao/utils/src/lib.rs` test vectors:

- Block 0: `ar = 10_000_000_000_000_000`, `C = 3_360_000_145_238_488_200`
- Block 1: `ar = 10_000_000_104_789_669`

`ar_increase` for block 1 = `104_789_669`. The true (untruncated) value is `10_000_000_000_000_000 * g2 / 3_360_000_145_238_488_200`. The truncation discards the fractional part — at most 1 unit of `ar` per block. [7](#0-6) 

Over ~3,942,000 blocks (1 year at ~10,800 blocks/day), the accumulated `ar` deficit is at most 3,942,000 units. The relative error in `ar` is ≤ 3.94 × 10⁻¹⁰. For a 1,000,000 CKB deposit (10¹⁴ shannons), the interest shortfall per year is at most ~39,400 shannons (~0.000394 CKB). Across all depositors and over the full chain lifetime, the stranded `s` balance grows without bound and is permanently unwithdrawable.
