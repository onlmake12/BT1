### Title
Unbounded `JsonBytes` Input in Indexer RPC `filter.output_data` Enables RPC-Caller DoS via Memory and CPU Exhaustion — (`util/indexer/src/service.rs`, `util/jsonrpc-types/src/bytes.rs`)

---

### Summary

The CKB indexer RPC endpoints (`get_cells`, `get_cells_capacity`, `get_transactions`) accept a `filter.output_data` field typed as `JsonBytes` with no upper-bound size check. Unlike `search_key.script.args` and `filter.script.args`, which are both bounded by `MAX_PREFIX_SEARCH_SIZE` (65535 bytes), `filter.output_data` is accepted at arbitrary size, decoded into a heap allocation proportional to the input, and then applied as a filter against every cell in the database. An unprivileged RPC caller can supply a multi-megabyte hex string to force O(n) memory allocation, O(n) hex decoding, and O(n × cells) filter work before any rejection.

---

### Finding Description

**`JsonBytes` deserialization has no length bound.**

`BytesVisitor::visit_str` in `util/jsonrpc-types/src/bytes.rs` checks only for a `0x` prefix and even length, then immediately allocates `vec![0; bytes.len() >> 1]` and calls `hex_decode` on the full input: [1](#0-0) 

There is no maximum length guard anywhere in this path.

**`filter.output_data` is accepted without a size check, unlike `script.args`.**

In `build_query_options`, the primary script's `args` length is validated: [2](#0-1) 

In `TryInto<FilterOptions>`, the filter script's `args` length is also validated against `MAX_PREFIX_SEARCH_SIZE`: [3](#0-2) 

But `filter.output_data` is accepted and stored with no analogous check: [4](#0-3) 

The `IndexerSearchKeyFilter` struct declares `output_data` as `Option<JsonBytes>` with no size constraint: [5](#0-4) 

**The unbounded data is then applied per-cell during iteration.**

For `Partial` mode, `memmem::find` is called against every cell's output data in the database: [6](#0-5) 

The same pattern appears in `get_cells_capacity`: [7](#0-6) 

---

### Impact Explanation

An unprivileged RPC caller sends a `get_cells` or `get_cells_capacity` request with a `filter.output_data` value that is an arbitrarily large hex string (e.g., `"0x" + "aa" * 50_000_000` = 100 MB JSON string). The node will:

1. Allocate ~50 MB of heap memory during `JsonBytes` deserialization.
2. Spend proportional CPU time in `hex_decode`.
3. Apply the oversized pattern against every live cell in the database using `memmem::find`, `starts_with`, or equality comparison.

This can be repeated concurrently by multiple callers. The result is sustained memory pressure and CPU exhaustion on the CKB node process, degrading or halting indexer RPC availability. The node's core consensus and P2P functions share the same process, so severe memory pressure can affect overall node stability.

---

### Likelihood Explanation

The indexer RPC is exposed to any local or remote RPC caller (depending on node configuration). No authentication or privileged role is required. The attack requires only a single crafted JSON-RPC request. The asymmetry is clear: `script.args` is bounded but `output_data` is not, making this an oversight rather than an intentional design choice. The `partial` search mode, which performs `memmem::find` per cell, amplifies the cost.

---

### Recommendation

Add a size check for `filter.output_data` in `TryInto<FilterOptions>` (in `util/indexer/src/service.rs`) analogous to the existing check for `filter.script.args`:

```rust
let output_data = filter.output_data.map(|data| {
    if data.len() > MAX_PREFIX_SEARCH_SIZE {
        return Err(Error::invalid_params(format!(
            "search_key.filter.output_data len should be less than {MAX_PREFIX_SEARCH_SIZE}"
        )));
    }
    let mode = filter
        .output_data_filter_mode
        .unwrap_or(IndexerSearchMode::Prefix);
    Ok((data.as_bytes().to_vec(), mode))
}).transpose()?;
```

Apply the same fix in the rich-indexer's `AsyncRichIndexerHandle` path. Additionally, consider enforcing a global JSON-RPC request body size limit at the HTTP server layer to prevent oversized payloads from reaching any handler.

---

### Proof of Concept

```bash
# Send a get_cells request with a 10 MB output_data filter
python3 -c "
import json, socket

data = '0x' + 'aa' * 5_000_000  # 10 MB hex string
payload = json.dumps({
    'id': 1,
    'jsonrpc': '2.0',
    'method': 'get_cells',
    'params': [
        {
            'script': {
                'code_hash': '0x' + '00' * 32,
                'hash_type': 'data',
                'args': '0x'
            },
            'script_type': 'lock',
            'filter': {
                'output_data': data,
                'output_data_filter_mode': 'partial'
            }
        },
        'asc',
        '0x64',
        None
    ]
})
print(f'Payload size: {len(payload) / 1e6:.1f} MB')
# Send to CKB node RPC port (default 8114)
"
```

The node will allocate ~5 MB per request for the decoded buffer, spend CPU on `hex_decode`, and then execute `memmem::find` against every indexed cell. Sending this in a tight loop from multiple connections will exhaust node memory and CPU.

    fn visit_str<E>(self, v: &str) -> Result<Self::Value, E>
    where
        E: serde::de::Error,
    {
        if v.len() < 2 || &v.as_bytes()[0..2] != b"0x" {
            return Err(E::invalid_value(serde::de::Unexpected::Str(v), &self));
        }

        if v.len() & 1 != 0 {
            return Err(E::invalid_length(v.len(), &"even length"));
        }

        let bytes = &v.as_bytes()[2..];
        if bytes.is_empty() {
            return Ok(JsonBytes::default());
