### Title
Zero-Threshold Bypass in `verify_m_of_n` Allows Signature-Free Alert Acceptance and Relay ŌĆö (`util/multisig/src/secp256k1.rs`)

### Summary

`verify_m_of_n` performs no lower-bound check on `m_threshold`. When `signatures_threshold = 0` is set in `NetworkAlertConfig` (no validation prevents it), the function returns `Ok(())` with zero signatures, allowing any caller to inject and relay network alerts without any valid cryptographic authorization.

### Finding Description

The `verify_m_of_n` function in `util/multisig/src/secp256k1.rs` is the sole signature-verification primitive used by the network alert system. Its two guard conditions are:

```rust
if m_threshold > sigs.len() {          // line 23
    return Err(ErrorKind::SigNotEnough.into());
}
// ...
if verified_sig_count < m_threshold {  // line 47
    return Err(ErrorKind::Threshold { ŌĆ” }.into());
}
```

When `m_threshold = 0`:
- `0 > sigs.len()` is always `false` ŌåÆ first guard passes
- `verified_sig_count < 0` is always `false` (unsigned) ŌåÆ second guard passes
- The function returns `Ok(())` unconditionally, regardless of the signatures slice

The threshold value flows directly from `NetworkAlertConfig.signatures_threshold` (a plain `usize` with no minimum-value validation) into `Verifier::verify_signatures`, which passes it unchanged to `verify_m_of_n`:

```rust
// util/network-alert/src/verifier.rs line 56-61
verify_m_of_n(
    &message,
    self.config.signatures_threshold,   // ŌåÉ unchecked, can be 0
    &signatures,
    &self.pubkeys,
)
```

`NetworkAlertConfig` is deserialized from `ckb.toml` via the `alert_signature` field of `CKBAppConfig` with no post-deserialization validation:

```rust
// util/app-config/src/app_config.rs line 88
pub alert_signature: Option<NetworkAlertConfig>,
```

Additionally, `signatures_threshold` is entirely independent of `public_keys.len()` ŌĆö there is no enforcement that the threshold is at most the number of registered public keys, mirroring the Lido quorum/guardians decoupling exactly.

### Impact Explanation

A node running with `signatures_threshold = 0` in its `ckb.toml`:

1. Accepts any `send_alert` RPC call from a local RPC user with zero valid signatures.
2. Relays the fabricated alert to all connected P2P peers via `AlertRelayer::received` ŌåÆ `nc.quick_filter_broadcast`.
3. Peers that also accept the alert (same misconfiguration, or nodes that trust the relayed message) propagate it further.

The result is arbitrary network-wide alert injection without possession of any Nervos Foundation private key, bypassing the documented 2-of-4 multi-signature requirement entirely.

### Likelihood Explanation

The `alert_signature` section is an optional, user-editable block in `ckb.toml`. There is no startup validation that rejects `signatures_threshold = 0` or that enforces `signatures_threshold <= public_keys.len()`. A node operator who sets this value to `0` (e.g., for local testing, or by mistake) silently degrades the security of the entire alert subsystem. The `send_alert` RPC endpoint is then reachable by any local RPC caller without any key material.

### Recommendation

1. **Add a lower-bound check in `verify_m_of_n`**: reject `m_threshold == 0` with a dedicated `ErrorKind::ZeroThreshold` variant before any other logic.
2. **Validate `NetworkAlertConfig` at startup**: assert `signatures_threshold >= 1` and `signatures_threshold <= public_keys.len()` when the config is loaded; return `ExitCode::Config` on violation (consistent with how other config errors are handled in `Launcher`).
3. **Tie threshold to key count**: consider deriving the threshold as a fraction of `public_keys.len()` (e.g., Ōēź 2/3) rather than storing it as a free-standing integer, directly addressing the decoupling that is the root cause of the Lido class of bugs.
