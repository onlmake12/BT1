Audit Report

## Title
Missing `BLOCK_INVALID` Status Check in `HeaderAcceptor::accept()` Enables Repeated Re-Validation of Rejected Headers - (File: sync/src/synchronizer/headers_process.rs)

## Summary
`HeaderAcceptor::accept()` in `sync/src/synchronizer/headers_process.rs` fetches block status but only short-circuits on `HEADER_VALID`, not on `BLOCK_INVALID`. A developer `FIXME` comment at line 301 explicitly acknowledges this gap. Any unprivileged P2P peer can repeatedly deliver headers already marked `BLOCK_INVALID`, forcing the node to re-execute `prev_block_check`, `non_contextual_check` (which includes PoW verification via `HeaderVerifier::verify`), and `version_check` on every delivery, with no mechanism to short-circuit the work.

## Finding Description
In `HeaderAcceptor::accept()` (lines 295–358 of `sync/src/synchronizer/headers_process.rs`), the status check reads:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer state and return early
    return result;
}
``` [1](#0-0) 

When `status` is `BLOCK_INVALID` (bit `1 << 12`, defined in `shared/src/block_status.rs`), the condition `status.contains(BlockStatus::HEADER_VALID)` is false, so execution falls through to the full validation pipeline: [2](#0-1) 

1. `prev_block_check` — checks parent block status
2. `non_contextual_check` — calls `self.verifier.verify(self.header)`, which includes PoW verification
3. `version_check` — checks header version [3](#0-2) 

Each of these runs to completion before the header is re-marked `BLOCK_INVALID` and discarded. The analogous guard is correctly present in `sync/src/relayer/compact_block_process.rs`:

```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
``` [4](#0-3) 

The `execute()` function in `HeadersProcess` processes up to `MAX_HEADERS_LEN` headers per message, each going through `HeaderAcceptor::accept()` independently, multiplying the per-message cost. [5](#0-4) 

## Impact Explanation
The attack maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. An attacker with multiple standard P2P connections can flood target nodes with `SendHeaders` messages containing previously-rejected headers. Each message triggers repeated PoW verification (the most CPU-intensive part of `non_contextual_check`) for up to `MAX_HEADERS_LEN` headers. At scale — multiple attacker peers, each sending maximum-size `SendHeaders` messages in a tight loop — the sync thread's CPU budget is consumed re-validating permanently-rejected headers, degrading or halting legitimate header synchronization across targeted nodes. If enough nodes are targeted simultaneously, the network's ability to propagate new blocks is impaired, constituting network congestion with minimal attacker cost (only standard P2P connections required).

## Likelihood Explanation
The attack requires only a standard P2P connection — no keys, stake, or privileged role. The attacker first sends one malformed header (e.g., wrong parent hash) to cause the node to store `BLOCK_INVALID` for that hash, then replays it indefinitely. The `FIXME` comment at line 301 confirms the developers are aware the check is missing. The attack is fully repeatable, stateless on the attacker side, and amplifiable by opening multiple peer connections or using multiple invalid headers per message.

## Recommendation
Add an early-return guard for `BLOCK_INVALID` immediately after fetching block status in `HeaderAcceptor::accept()`, mirroring the pattern in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // existing early-return logic
}
```

This eliminates all downstream validation work for permanently-rejected headers and resolves the acknowledged `FIXME`. [1](#0-0) 

## Proof of Concept
1. Attacker connects to a victim CKB node as a standard P2P peer.
2. Attacker sends a `SendHeaders` message with a syntactically valid but semantically invalid header (e.g., parent hash pointing to an unknown block that itself has `BLOCK_INVALID` status). The node runs `prev_block_check`, fails, and calls `shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID)`.
3. Attacker enters a tight loop sending `SendHeaders` messages containing the same header hash (now `BLOCK_INVALID`).
4. On each delivery, `accept()` fetches status (`BLOCK_INVALID`), finds `HEADER_VALID` is not set, and falls through to re-execute `prev_block_check`, `non_contextual_check` (PoW), and `version_check`.
5. To amplify: pack `MAX_HEADERS_LEN` distinct `BLOCK_INVALID` headers per message; open multiple peer connections to multiply throughput.
6. Observable effect: elevated CPU on the sync thread, increasing latency for processing legitimate headers from honest peers.

The root-cause evidence is the `FIXME` comment at line 301 of `sync/src/synchronizer/headers_process.rs`, confirmed by direct code inspection showing no `BLOCK_INVALID` guard exists in the `accept()` function. [6](#0-5)
