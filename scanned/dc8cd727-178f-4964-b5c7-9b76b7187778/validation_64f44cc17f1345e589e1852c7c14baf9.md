Audit Report

## Title
Copy-Paste Bug in `script_exists_in_output` Reads `row_lock` Instead of `row_type` on PostgreSQL, Causing Incorrect Script Deletion During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
In `script_exists_in_output` at line 252 of `util/rich-indexer/src/indexer/remove.rs`, the second `match` block reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. On PostgreSQL, where `EXISTS` returns `BOOLEAN`, this causes the function to always return `false` for the type-script check (since `row_lock` is already `false` at that point — otherwise the early return at line 226 would have fired). During `rollback_block`, this causes any script referenced only as a `type_script_id` in surviving outputs to be incorrectly deleted from the `script` table, permanently corrupting the rich-indexer's indexed state for those cells.

## Finding Description
`script_exists_in_output` issues two `SELECT EXISTS` queries: one checking `lock_script_id` (result in `row_lock`) and one checking `type_script_id` (result in `row_type`). The first match block at lines 222–235 correctly short-circuits with `Ok(true)` if `row_lock` is true. Execution only reaches line 252 when `row_lock` is `false`. The second match block at line 252 is supposed to evaluate `row_type`, but instead reads `row_lock` again:

```rust
// line 252 — BUG: should be row_type.try_get::<bool, _>(0)
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),           // PostgreSQL: r is always false here
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // SQLite: correctly reads row_type
}
```

On **PostgreSQL**, `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL `EXISTS` returns `BOOLEAN`) and returns `false` unconditionally, completely bypassing the `row_type` result. On **SQLite**, `try_get::<bool, _>(0)` fails (SQLite `EXISTS` returns `BIGINT`), so the `Err(_)` branch correctly reads `row_type`.

The caller in `rollback_block` at lines 33–37 uses this return value to decide whether to push `type_script_id` into the deletion list. On PostgreSQL, `script_exists_in_output(type_script_id)` always returns `false` when the script is not also a `lock_script_id` in any surviving output, so the script row is unconditionally added to `script_id_list_to_remove` and deleted from the `script` table even when surviving outputs from prior blocks still reference it via `type_script_id`.

The existing rollback tests in `util/rich-indexer/src/tests/rollback.rs` all use SQLite (`connect_sqlite`), so this PostgreSQL-specific regression is not caught by the test suite.

## Impact Explanation
This is a correctness bug in the CKB rich-indexer's state storage mechanism. After the incorrect deletion, the `script` table is missing the row for the shared type-script. Any subsequent `get_cells` or `get_transactions` RPC call that joins against the `script` table using that `type_script_id` will return NULL joins or missing script data for all surviving cells from prior blocks that reference that type-script. The rich-indexer's indexed state is permanently broken for those cells until a full re-sync from scratch. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
Chain reorganizations are a normal part of CKB network operation and require no special privileges to trigger — any peer can relay a valid competing chain. The only structural requirement is that the rolled-back block contains an output whose `type_script_id` is shared with a surviving output in a prior block but whose `lock_script_id` is not present in any surviving output. This is a routine scenario (e.g., UDT token cells with a shared type-script but different owner lock-scripts across blocks). The bug affects all PostgreSQL-backed rich-indexer deployments and is silently triggered on every qualifying reorg.

## Recommendation
Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // was: row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Additionally, add a PostgreSQL-backed rollback test that verifies a type-script shared across two blocks is preserved after rolling back the second block.

## Proof of Concept
1. Start a PostgreSQL-backed rich-indexer node.
2. Append block 0 containing output A with `type_script = T`, `lock_script = L1`.
3. Append block 1 containing output B with `type_script = T`, `lock_script = L2` (same `T`, different lock).
4. Trigger a reorg that rolls back block 1 (`rollback_block` executes, removing output B from the `output` table before `script_exists_in_output` is called).
5. `script_exists_in_output(T)` is called: `row_lock` query returns `false` (T is not a `lock_script_id` in any surviving output), `row_type` query returns `true` (T is still a `type_script_id` in output A), but line 252 reads `row_lock` again and returns `false`.
6. T is pushed to `script_id_list_to_remove` and deleted from the `script` table.
7. Query the `script` table: the row for T is gone.
8. Call `get_cells` with `script = T`: returns empty or broken results despite output A from block 0 surviving.
9. **Expected**: T's script row is preserved; `get_cells` returns output A.
