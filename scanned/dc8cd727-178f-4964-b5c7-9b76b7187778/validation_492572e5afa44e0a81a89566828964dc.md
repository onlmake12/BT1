Audit Report

## Title
Silent Truncating `u128 → u64` Cast in `calculate_maximum_withdraw` Returns Wrong Withdrawal Capacity — (`File: util/dao/src/lib.rs`)

## Summary
In `calculate_maximum_withdraw`, the intermediate `u128` product `withdraw_counted_capacity` is narrowed to `u64` via an unchecked `as u64` cast at line 156. If the product exceeds `u64::MAX`, the upper 64 bits are silently discarded and the function returns `Ok(wrong_capacity)` instead of `Err(DaoError::Overflow)`. Every other `u128 → u64` narrowing in the same file (lines 204, 245, 258) uses the safe `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern, making this an isolated, confirmed inconsistency.

## Finding Description

**Root cause — line 156:**
```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The three safe conversions in the same file for comparison: [2](#0-1) [3](#0-2) [4](#0-3) 

**Overflow condition:** `withdraw_counted_capacity = counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. Since `counted_capacity` is bounded by the total CKB supply (~3.36 × 10¹⁸ shannons) and `u64::MAX ≈ 1.84 × 10¹⁹`, overflow requires `withdrawing_ar / deposit_ar > ~5.48`. AR starts at 10¹⁰ at genesis and grows slowly; a 5.48× ratio requires an extremely long deposit duration.

**Exploit path:**
1. A depositor locks a large DAO cell (near total-supply capacity).
2. After sufficient chain time such that `withdrawing_ar / deposit_ar > 5.48`, they initiate withdrawal.
3. `calculate_maximum_withdraw` computes `withdraw_counted_capacity > u64::MAX` in u128, then silently truncates via `as u64`, producing a value that is `withdraw_counted_capacity mod 2^64` — orders of magnitude smaller than correct.
4. The function returns `Ok(wrong_capacity)` with no error.

**Why existing guards fail:** The existing test `check_withdraw_calculation_overflows` uses a near-`u64::MAX` output capacity with a tiny AR ratio (~1.0000000001). In that test, `withdraw_counted_capacity` stays below `u64::MAX` (the truncation does not trigger); the error is caught later by `safe_add` overflowing when `occupied_capacity` is added back. The `as u64` truncation path is never exercised. [5](#0-4) 

## Impact Explanation

**Path 1 — DAO savings field (`S`) corruption:** `withdrawed_interests` sums the (wrong, too-small) maximum-withdraw values for all DAO withdrawal transactions in a block and subtracts from `parent_s`:
```rust
let current_s = parent_s
    .safe_add(nervosdao_issuance)
    .and_then(|s| s.safe_sub(withdrawed_interests))?;
``` [6](#0-5) 
A truncated `withdrawed_interests` inflates `current_s` in every subsequent block header. Note: the AR accumulation rate is computed from `parent_ar`, `current_g2`, and `parent_c` — not from `S` — so the claim that AR itself is corrupted is overstated. However, the `S` field is stored in every block header's DAO data and is used by the DAO system script for on-chain verification of withdrawals; a persistently wrong `S` constitutes corruption of the DAO accounting state.

**Path 2 — Fee miscalculation / valid tx rejection:** `transaction_fee` computes `maximum_withdraw - outputs_capacity`: [7](#0-6) 
A truncated (too-small) `maximum_withdraw` causes `safe_sub` to underflow, returning `Err`, which causes the tx-pool to reject a valid DAO withdrawal transaction. The depositor cannot withdraw their funds through the normal tx-pool path.

This maps to **damage to CKB economy**: legitimate DAO depositors are unable to withdraw, and the DAO savings accounting in block headers is permanently corrupted for any block containing such a withdrawal.

## Likelihood Explanation

Overflow requires `withdrawing_ar / deposit_ar > ~5.48`. Under current mainnet secondary issuance parameters, AR grows on the order of 10⁻¹⁰ per block; reaching a 5.48× ratio would take an extremely long time. The bug is latent and grows more likely as the chain ages. A single large depositor (holding a significant fraction of total supply) who holds for a very long duration is the triggering actor. No special privileges are required — any unprivileged user with a sufficiently large, sufficiently old DAO deposit can trigger it. The condition is not easily met today but is not permanently impossible.

## Recommendation

Replace the unchecked cast with the same safe pattern used at lines 204, 245, and 258:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [8](#0-7) 

## Proof of Concept

The existing test infrastructure in `util/dao/src/tests.rs` already provides all necessary scaffolding. A new test that would directly trigger the `as u64` truncation (rather than the `safe_add` overflow caught by the existing test) would:

1. Set `deposit_ar` to `10_000_000_000` (genesis value).
2. Set `withdrawing_ar` to `55_000_000_000` (5.5× growth).
3. Set `output_capacity` to `3_000_000_000_000_000_000` shannons (~3×10¹⁸, within total supply).
4. Compute expected: `3_000_000_000_000_000_000 × 55_000_000_000 / 10_000_000_000 = 16_500_000_000_000_000_000` — within u64 range, so no overflow here. Adjust: use `output_capacity = 4_000_000_000_000_000_000` → `4×10¹⁸ × 5.5 = 2.2×10¹⁹ > u64::MAX`. With the current `as u64` cast, the function returns `Ok(2_200_000_000_000_000_000 mod 2^64)` — a wrong, small value — instead of `Err(DaoError::Overflow)`.
5. Assert `result.is_err()` — this assertion **fails** with the current code, proving the silent truncation.

fn check_withdraw_calculation_overflows() {
