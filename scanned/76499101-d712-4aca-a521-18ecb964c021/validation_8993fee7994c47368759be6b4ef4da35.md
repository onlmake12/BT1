Audit Report

## Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect Withdrawal Capacity — (`File: util/dao/src/lib.rs`)

## Summary

`DaoCalculator::calculate_maximum_withdraw` uses an unchecked `as u64` cast on a u128 intermediate product at line 156, silently truncating the result when it exceeds `u64::MAX`. Every other u128→u64 narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When triggered, the truncated value propagates through both the tx-pool fee check and the DAO field computation used in block headers, producing incorrect results in both paths.

## Finding Description

At `util/dao/src/lib.rs` lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)  // ← unchecked
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The three analogous conversions in the same file all use the checked form:
- Line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- Line 245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- Line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [2](#0-1) [3](#0-2) [4](#0-3) 

The overflow condition is `counted_capacity × withdrawing_ar / deposit_ar > u64::MAX`. Since `ar` is monotonically increasing from genesis (`10^16`), and the total CKB supply is ~3.36×10¹⁸ shannons, a cell holding capacity near the supply ceiling overflows when `withdrawing_ar / deposit_ar ≳ 5.5`.

The corrupted value propagates through two paths:

1. **tx-pool path**: `calculate_maximum_withdraw` → `transaction_maximum_withdraw` → `transaction_fee` → `check_tx_fee` in `tx-pool/src/util.rs` lines 34–41. A truncated `maximum_withdraw` causes either a spurious `safe_sub` error (rejecting a valid withdrawal) or an understated fee (triggering `LowFeeRate` rejection). [5](#0-4) 

2. **consensus path**: `calculate_maximum_withdraw` → `transaction_maximum_withdraw` → `withdrawed_interests` → `dao_field_with_current_epoch`. The `current_s` field of the DAO data is computed as `parent_s + nervosdao_issuance - withdrawed_interests`. A truncated `withdrawed_interests` produces a wrong `current_s`, meaning the DAO field packed into the block header diverges from the correct value. [6](#0-5) [7](#0-6) 

## Impact Explanation

The consensus path impact maps to **"Vulnerabilities which could cause consensus deviation"**. Nodes that have applied a fix would compute a different DAO field than unfixed nodes when processing a block containing an overflowing DAO withdrawal, causing a chain split. The tx-pool path additionally causes valid DAO withdrawal transactions to be rejected by affected nodes. Both impacts are caused entirely within this codebase, not by an external dependency.

## Likelihood Explanation

The overflow requires `withdrawing_ar / deposit_ar ≳ 5.5`. Starting from the genesis accumulate rate of `10^16` and growing at approximately 4% annually from secondary issuance, this ratio is reached after roughly 43 years of continuous network operation. No special privilege is required — any user submitting a DAO withdrawal transaction is the unprivileged entry point. The condition is not triggerable today, but the defect is a present code inconsistency that will silently corrupt results rather than surface an error when eventually reached. The existing test `check_withdraw_calculation_overflows` at line 296 demonstrates that the developers explicitly test overflow behavior in this function area, making the omission of a checked cast here a clear oversight. [8](#0-7) 

## Recommendation

Replace the unchecked cast with the same checked pattern used in the three sibling conversions:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This makes `calculate_maximum_withdraw` consistent with `secondary_block_reward` and `dao_field_with_current_epoch`, and ensures the function returns `DaoError::Overflow` rather than a silently wrong value.

## Proof of Concept

Using the values from the claim:
- `deposit_ar = 10_000_000_000_000_000` (genesis)
- `withdrawing_ar = 55_000_000_000_000_000` (5.5× growth)
- `counted_capacity = 18_446_744_073_709_551_615` (u64::MAX)

```
withdraw_counted_capacity (u128)
  = 18_446_744_073_709_551_615 × 55_000_000_000_000_000
    / 10_000_000_000_000_000
  = 101_457_092_405_402_533_882   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 101_457_092_405_402_533_882 mod 2^64
  = 83_010_348_331_692_982_266    ← silently truncated wrong value
```

A unit test extending `check_withdraw_calculation_overflows` with `withdrawing_ar` set to `55_000_000_000_000_000` and `deposit_ar` set to `10_000_000_000_000_000` on a near-max-capacity cell would demonstrate that the current code returns a wrong `Capacity` value instead of `DaoError::Overflow`, while the fixed code returns `DaoError::Overflow`.
