Audit Report

## Title
Tx-Pool Minimum Fee Admission Check Omits Cycles Component of Transaction Weight, Allowing Underpayment - (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the serialized transaction size (`tx_size`) as the weight denominator, while CKB's actual transaction weight is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction with high cycle consumption but small serialized size passes the admission check by paying only the size-based minimum fee — orders of magnitude less than the fee required by the true weight. No second weight-based fee check is performed after `verify_rtx` returns the actual cycle count, so the transaction is admitted to the pool and is eligible for mining.

## Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` (lines 42–52) explicitly uses only `tx_size` for the minimum fee calculation, with a comment acknowledging this is intentionally a "cheap check":

```rust
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The correct weight function `get_transaction_weight` in `util/types/src/core/tx_pool.rs` (lines 298–303) computes `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`: [2](#0-1) 

All internal fee-rate calculations — pool scoring, eviction — use this correct weight. `fee_rate()` in `tx-pool/src/component/entry.rs` (lines 115–118) uses `get_transaction_weight(self.size, self.cycles)`: [3](#0-2) 

The admission flow in `process.rs` (lines 276–316) calls `check_tx_fee` during `pre_check` (before script execution), then calls `verify_rtx` to obtain actual cycles, then calls `submit_entry` — with no second weight-based fee check at any point: [4](#0-3) 

The same omission exists in `calculate_min_replace_fee` in `tx-pool/src/pool.rs` (lines 102–103), where the extra RBF fee is computed from `size` alone: [5](#0-4) 

## Impact Explanation

With default config (`min_fee_rate = 1000` shannons/KW, `max_tx_verify_cycles = 70,000,000`): [6](#0-5) 

A transaction with `tx_size = 100` bytes and `cycles = 70,000,000` has actual weight ≈ 11,940. The admission check requires only `1000 × 100 / 1000 = 100 shannons`, while the weight-based minimum is `1000 × 11,940 / 1000 = 11,940 shannons` — a ~119× underpayment. The transaction enters the pool, consumes nearly the full block cycle budget, and is eligible for mining. This constitutes a **bad design which could cause CKB network congestion with few costs** (High impact), as an attacker can repeatedly submit cycle-heavy, fee-cheap transactions that force full script execution on every node while paying negligible fees, and can fill blocks with transactions that consume the cycle budget at far below the enforced minimum fee rate.

## Likelihood Explanation

The attack is trivially reachable by any unprivileged user via the `send_transaction` RPC or P2P relay. The attacker only needs to craft a transaction whose lock or type script consumes near-maximum cycles (e.g., a tight RISC-V computation loop) while keeping the serialized transaction small. No special privilege, key, or majority hashpower is required. The `max_tx_verify_cycles` limit bounds the maximum underpayment factor but does not prevent the attack. The attack is repeatable at low cost.

## Recommendation

After `verify_rtx` returns the actual cycle count and before constructing `TxEntry`, perform a second fee check using the true weight:

```rust
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64()));
}
```

Apply the same fix to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, replacing `size` with `get_transaction_weight(size, cycles)` for the `extra_rbf_fee` computation. [7](#0-6) 

## Proof of Concept

1. Craft a CKB transaction with a lock script that runs a tight RISC-V loop consuming ~70,000,000 cycles, a single input and single output (serialized size ≈ 100 bytes), and fee = 100 shannons.
2. Submit via `send_transaction` RPC.
3. `check_tx_fee` passes: `fee (100) >= min_fee_rate.fee(tx_size=100) = 100`. [8](#0-7) 
4. `verify_rtx` executes the script and returns `cycles ≈ 70,000,000`.
5. `TxEntry` is created with `cycles = 70,000,000`; `fee_rate()` computes `FeeRate::calculate(100, 11940) ≈ 8 shannons/KW` — far below `min_fee_rate = 1000`. No rejection occurs. [3](#0-2) 
6. The transaction is admitted to the pool and is eligible for mining, consuming ~11,940 weight units of block capacity while paying only 100 shannons — a ~119× underpayment relative to the enforced minimum fee rate.
