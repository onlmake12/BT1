Audit Report

## Title
Unbounded `by_out_point` Index Growth in `OrphanPool` Allows Memory Amplification via High-Input Orphan Transactions — (`tx-pool/src/component/orphan.rs`)

## Summary
`OrphanPool` enforces a 100-transaction count limit via `DEFAULT_MAX_ORPHAN_TRANSACTIONS`, but the secondary index `by_out_point` is bounded only by the total number of input `OutPoint`s across all orphan transactions. An unprivileged remote peer can submit 100 transactions each carrying the maximum number of inputs (~11,636 given the 512KB tx size limit), causing `by_out_point` to grow to ~1.16M entries while `entries.len() == 100`, resulting in severe heap memory amplification that can crash a CKB node.

## Finding Description
In `add_orphan_tx` (`orphan.rs` L150–155), every input `OutPoint` of the transaction is unconditionally inserted into `by_out_point`:

```rust
for out_point in tx.input_pts_iter() {
    self.by_out_point
        .entry(out_point)
        .or_default()
        .insert(tx.proposal_short_id());
}
``` [1](#0-0) 

`limit_size()` is called immediately after, but it only enforces the transaction count:

```rust
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
``` [2](#0-1) 

`self.len()` returns `self.entries.len()` — the number of transactions, not the number of `by_out_point` entries: [3](#0-2) 

The `by_out_point` map is never directly bounded. The upstream guard `non_contextual_verify` enforces `TRANSACTION_SIZE_LIMIT = 512 * 1_000` bytes per transaction: [4](#0-3) 

This limits inputs per transaction to approximately 11,636 (a `CellInput` is ~44 bytes: 36-byte `OutPoint` + 8-byte `since`), but does not prevent the amplification across 100 orphan transactions. With 100 orphan transactions each at maximum inputs, `by_out_point` accumulates ~1.16M entries while `entries.len() == 100`.

## Impact Explanation
Each `OutPoint` key is 36 bytes; each `HashSet<ProposalShortId>` value adds ~50–100 bytes of overhead. At ~1.16M entries, `by_out_point` alone consumes ~100–150MB of heap memory, far exceeding what the 100-tx count limit implies. This is a memory amplification DoS: the attacker uses 100 relay messages to force ~1.16M map entries. Sustained or repeated attacks can exhaust node heap memory and crash the process. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node."

## Likelihood Explanation
The attack path is fully reachable by an unprivileged remote peer via P2P relay. The attacker crafts 100 transactions each referencing ~11,636 unknown `OutPoint`s (causing them to be classified as orphans), each within the 512KB size limit. No PoW, no privileged access, and no special conditions are required. The attack is repeatable: after eviction by expiry or replacement, the attacker can re-submit. The only upstream guard (`TRANSACTION_SIZE_LIMIT`) limits per-transaction input count but does not prevent the amplification.

## Recommendation
Replace the transaction-count-based limit with one that accounts for total `by_out_point` entries or total serialized size:
- Cap `by_out_point.len()` directly (e.g., at `DEFAULT_MAX_ORPHAN_TRANSACTIONS * MAX_INPUTS_PER_TX`), evicting transactions when the cap is exceeded.
- Track total input count across all orphan transactions and reject or evict when it exceeds a threshold.
- Apply a per-peer orphan input quota to prevent a single peer from monopolizing the index.

## Proof of Concept
```rust
let mut pool = OrphanPool::new();
for i in 0..100u32 {
    // Build a tx with ~11_000 distinct unknown OutPoints as inputs (within 512KB)
    let tx = build_tx_with_n_inputs(11_000, i);
    pool.add_orphan_tx(tx, PeerIndex::new(0), 1000);
}
assert_eq!(pool.entries.len(), 100);           // capped correctly by limit_size()
assert!(pool.by_out_point.len() > 1_000_000); // NOT capped — amplification confirmed
``` [5](#0-4)
