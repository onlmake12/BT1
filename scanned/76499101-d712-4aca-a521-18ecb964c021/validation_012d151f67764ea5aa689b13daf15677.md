## Title
Missing `Flags::DISCOVERY` Removal When Discovery Protocol Is Disabled ŌĆö (`util/launcher/src/lib.rs`)

## Summary
In `util/launcher/src/lib.rs`, `flags` is initialized to `Flags::all()` and individual bits are cleared via `else` branches for `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT` when those protocols are absent from `support_protocols`. No equivalent `else` branch exists for `Flags::DISCOVERY`, so any node configured without the Discovery protocol still broadcasts `Flags::DISCOVERY` in every Identify handshake. Peers receiving this message store the address as Discovery-capable and propagate it through the network, causing repeated failed Discovery sessions to that node.

## Finding Description
At `util/launcher/src/lib.rs:429`, `flags = Flags::all()` sets all capability bits including `DISCOVERY`. The code then conditionally removes `RELAY` (line 440), `BLOCK_FILTER` (line 455), and `LIGHT_CLIENT` (line 475) when those protocols are not in `support_protocols`, but there is no analogous branch for `DISCOVERY`. Meanwhile, in `network/src/network.rs:896ŌĆō914`, the Discovery protocol handler is only registered when `config.support_protocols.contains(&SupportProtocol::Discovery)`. The resulting `flags` value (with `DISCOVERY` still set) is passed directly to `NetworkService::new()` at `util/launcher/src/lib.rs:504`. When a peer connects, the Identify handler at `network/src/protocols/identify/mod.rs:422` calls `peer_store.add_outbound_addr(address, flags)`, persisting the incorrect flags. Both `fetch_addrs_to_attempt` (`peer_store_impl.rs:208`) and `fetch_random_addrs` (`peer_store_impl.rs:277`) use `required_flags_filter` to select peers, so the falsely-tagged address is returned to any caller requesting `Flags::DISCOVERY`-capable peers, including the Discovery `Nodes` response path.

## Impact Explanation
Nodes configured without Discovery will have their addresses propagated network-wide as Discovery-capable. Any node seeking Discovery peers will dial them, open a session (Sync succeeds), but Discovery never opens ŌĆö wasting an outbound connection slot. This degrades peer discovery efficiency across the network for operators who disable Discovery. This matches the **Low (501ŌĆō2000 points)** impact class: an important performance/reliability improvement for CKB, as it causes systematic misdirection of outbound connection attempts and peer store pollution for any deployment that omits Discovery from `support_protocols`.

## Likelihood Explanation
The default `support_protocols` in `resource/ckb.toml` includes Discovery, so the bug is dormant in standard deployments. It is triggered by any operator who removes Discovery from `support_protocols` (e.g., relay-only nodes, light-client-only nodes). The configuration explicitly documents Discovery as optional. Once triggered, the false flag is set at startup and persists for the entire process lifetime, affecting every peer that connects.

## Recommendation
Add the missing `else` branch in `util/launcher/src/lib.rs` immediately after the Discovery-related block, mirroring the pattern used for `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT`:

```rust
if support_protocols.contains(&SupportProtocol::Discovery) {
    // Discovery is registered inside NetworkService::new()
} else {
    flags.remove(Flags::DISCOVERY);
}
```
