### Title
Missing `from != to` Identity Check in `ConnectionRequestProcess::execute` Allows `pending_delivered` Poisoning and Forced NAT Traversal to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

`ConnectionRequestProcess::execute` contains no guard enforcing `from != to`. An unprivileged connected peer can send a `ConnectionRequest` with both `from` and `to` set to the victim's own `PeerId`, causing the victim to insert attacker-controlled `listen_addrs` into `pending_delivered` under its own key. A subsequent `ConnectionSync` (also crafted with `from == to == victim_peer_id`) then causes the victim to spawn a NAT traversal task to attacker-controlled TCP addresses and, on success, register a raw inbound session to the attacker.

---

### Finding Description

**Step 1 — `ConnectionRequest` with `from == to == victim_peer_id`**

In `ConnectionRequestProcess::execute`, after parsing and rate-limit checks, the only routing branch is:

```rust
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs).await
``` [1](#0-0) 

There is no check that `content.from != content.to`. When the attacker sets both fields to the victim's own `PeerId`, the condition `self_peer_id == &content.to` is satisfied and `respond_delivered` is called with `from_peer_id = victim_peer_id` and `remote_listens = attacker_controlled_addrs`.

**Step 2 — `pending_delivered` poisoned**

Inside `respond_delivered`, after filtering addresses and sending the `ConnectionRequestDelivered` reply, the state is written unconditionally:

```rust
self.protocol.pending_delivered.insert(from_peer_id, (remote_listens, now));
``` [2](#0-1) 

`pending_delivered` now maps `victim_peer_id → ([attacker_addr], timestamp)`.

**Step 3 — `ConnectionSync` with `from == to == victim_peer_id` triggers NAT traversal**

In `ConnectionSyncProcess::execute`, when `route` is empty and `self_peer_id == &content.to`, the handler reads back the poisoned entry:

```rust
let listens_info = self.protocol.pending_delivered.get(&content.from)
    .map(|info| info.0.clone());
``` [3](#0-2) 

`content.from` is `victim_peer_id`, so it retrieves the attacker-controlled addresses. It then spawns NAT traversal tasks to those addresses:

```rust
.map(|listen_addr| Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
``` [4](#0-3) 

On success, a raw session is registered to the attacker:

```rust
let _ignore = control.raw_session(stream, addr, RawSessionInfo::inbound(listen_addr)).await;
``` [5](#0-4) 

Neither `ConnectionRequest` parsing nor `ConnectionSync` processing contains a `from != to` guard anywhere in the pipeline. [6](#0-5) 

---

### Impact Explanation

The attacker forces the victim to:
1. Corrupt `pending_delivered` with attacker-controlled addresses keyed under the victim's own `PeerId`
2. Initiate outbound TCP connections to attacker-controlled endpoints
3. Register a `raw_session` (bypassing normal peer scoring/selection) to the attacker if the TCP handshake succeeds

This gives the attacker a persistent, protocol-level foothold as a peer of the victim. Repeated use (within the `HOLE_PUNCHING_INTERVAL` window per key, but the key is `victim_peer_id` which the attacker controls) can saturate the victim's peer slots with attacker-controlled connections, enabling an eclipse attack. An eclipsed node receives only attacker-curated blocks and transactions, directly enabling consensus deviation (e.g., feeding a longer private chain or withholding blocks).

---

### Likelihood Explanation

The attack requires only a single established P2P connection to the victim — no special privileges, no PoW, no key material. The two crafted messages (`ConnectionRequest` then `ConnectionSync`) are valid wire-format messages that pass all existing checks. The `from` field is never verified against the sender's actual session `PeerId`, so spoofing it to the victim's own ID is trivially possible.

---

### Recommendation

Add an explicit identity check immediately after parsing in `ConnectionRequestProcess::execute`:

```rust
if content.from == content.to {
    return StatusCode::InvalidFromPeerId
        .with_context("from and to must be distinct peers");
}
``` [7](#0-6) 

Apply the same guard in `ConnectionSyncProcess::execute` and `ConnectionRequestDeliveredProcess::execute`. Additionally, validate that `content.from` matches the sender's actual session `PeerId` (available via `context.session`) to prevent identity spoofing entirely.

---

### Proof of Concept

```
1. Attacker peer A establishes a P2P connection to victim V.
2. A sends ConnectionRequest {
       from: V.peer_id,
       to:   V.peer_id,
       listen_addrs: [/ip4/attacker_ip/tcp/attacker_port/p2p/V.peer_id],
       max_hops: 1,
       route: [],
   }
3. V processes it: self_peer_id == content.to → respond_delivered() →
   pending_delivered[V.peer_id] = ([attacker_addr], now)
4. A sends ConnectionSync {
       from: V.peer_id,
       to:   V.peer_id,
       route: [],
   }
5. V processes it: self_peer_id == content.to, route empty →
   pending_delivered.get(V.peer_id) → [attacker_addr] →
   try_nat_traversal(bind_addr, attacker_addr) →
   raw_session(stream, attacker_addr, inbound) registered.
6. Assert: pending_delivered contains V.peer_id → attacker_addr (should be empty).
           V has initiated a TCP connection to attacker_ip:attacker_port.
```

    fn try_from(value: &packed::ConnectionRequestReader<'_>) -> Result<Self, Self::Error> {
        let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
            StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
        })?;
        let to = PeerId::from_bytes(value.to().raw_data().to_vec())
            .map_err(|_| StatusCode::InvalidToPeerId.with_context("the to peer id is invalid"))?;
        let listen_addrs: Vec<Multiaddr> = value
            .listen_addrs()
            .iter()
            .map(
                |raw| match Multiaddr::try_from(raw.bytes().raw_data().to_vec()) {
                    Ok(mut addr) => {
                        if let Some(peer_id) = extract_peer_id(&addr) {
                            if peer_id != from {
                                return Err(StatusCode::InvalidListenAddrLen
                                    .with_context("peer id in listen address is invalid"));
                            }
                        } else {
                            addr.push(Protocol::P2P(Cow::Borrowed(from.as_bytes())));
                        }
                        Ok(addr)
                    }
                    Err(_) => Err(StatusCode::InvalidListenAddrLen

        let max_hops: u8 = value.max_hops().into();

        Ok(Self {
            from,
            to,
            listen_addrs,

        })
