### Title
Attacker-Controlled `last_ping_protocol_message_received_at` Bypasses Inbound Peer Eviction Protection — (`network/src/protocols/ping.rs`, `network/src/peer_registry.rs`)

---

### Summary

An inbound peer can permanently protect itself from eviction by flooding the node with unsolicited Ping messages. `ping_received()` unconditionally refreshes `last_ping_protocol_message_received_at` on every incoming Ping, and `try_evict_inbound_peer()` uses that same field to protect the `EVICTION_PROTECT_PEERS` (8) most recently active peers. Because there is no rate limit or authenticity check on incoming Pings, the attacker trivially keeps their timestamp at `Instant::now()` and is never selected for eviction.

---

### Finding Description

**Entry point — `ping_received()` (ping.rs lines 62–69):**

When the `received()` handler dispatches a `PingPayload::Ping`, it calls `ping_received()` unconditionally:

```rust
PingPayload::Ping(nonce) => {
    self.ping_received(session.id);   // ← no rate limit
    ...
}
```

`ping_received()` writes `Instant::now()` into the peer's registry entry with no guard:

```rust
fn ping_received(&mut self, id: SessionId) {
    self.network_state.with_peer_registry_mut(|reg| {
        if let Some(peer) = reg.get_peer_mut(id) {
            peer.last_ping_protocol_message_received_at = Some(Instant::now());
        }
    });
}

**Eviction logic — `try_evict_inbound_peer()` (peer_registry.rs lines 149–183):**

The code comment explicitly states the intent:

```rust
// Protect peers based on characteristics that an attacker hard to simulate or manipulate
```

The second protection round sorts remaining candidates by `last_ping_protocol_message_received_at` and removes the 8 most recently active from the eviction pool:

```rust
sort_then_drop(
    &mut candidate_peers,
    EVICTION_PROTECT_PEERS,
    |peer1, peer2| {
        let peer1_last_message = peer1
            .last_ping_protocol_message_received_at
            .map(|t| now.saturating_duration_since(t).as_secs())
            .unwrap_or_else(|| u64::MAX);
        ...
        peer2_last_message.cmp(&peer1_last_message)
    },
);
``` [3](#0-2) 

`sort_then_drop` sorts descending (oldest first) and truncates to keep the last `n` elements — i.e., the peers with the smallest elapsed duration (most recent activity). An attacker who sends a Ping message moments before eviction is evaluated will have elapsed ≈ 0 seconds and will always land in the protected tail. [4](#0-3) 

**Why the first protection round does not save legitimate peers:**

The first round protects 8 peers with the lowest `ping_rtt`. The attacker deliberately avoids responding to the server's Ping (so `ping_rtt` stays `None` → `u64::MAX`), meaning the attacker is NOT protected in round 1 and remains a candidate — but is then shielded in round 2 via the manipulated timestamp. [5](#0-4) 

**`Peer` struct confirms the field is attacker-writable:**

```rust
/// Ping/Pong message last received time
pub last_ping_protocol_message_received_at: Option<Instant>,
``` [6](#0-5) 

---

### Impact Explanation

A single attacker-controlled inbound peer can permanently occupy one inbound slot. When `max_inbound` is reached and a legitimate peer attempts to connect, `try_evict_inbound_peer()` will never select the attacker — it will always evict a legitimate peer instead. With multiple attacker-controlled connections (each independently flooding Pings), all `EVICTION_PROTECT_PEERS` "recently active" slots can be monopolised, systematically displacing honest peers and degrading the node's inbound peer diversity and network topology.

---

### Likelihood Explanation

The attack requires only a standard TCP connection to the node's P2P port and the ability to send well-formed Ping messages in a loop. No cryptographic material, hashpower, or privileged access is needed. The Ping message format is public and trivially constructable. There is no rate limiting, no nonce validation for incoming Pings (only for Pong), and no cost imposed on the sender.

---

### Recommendation

1. **Restrict what updates `last_ping_protocol_message_received_at`** — only a valid Pong (with matching nonce) should refresh this field, not an unsolicited incoming Ping. `pong_received()` already does this correctly; `ping_received()` should not touch the field at all, or should use a separate, non-eviction-relevant counter.
2. **Rate-limit incoming Ping messages** per session to prevent flooding.
3. **Rename/separate the field** to make the semantic clear: "last genuine protocol activity" should only be set on authenticated round-trips (Pong with correct nonce), not on one-way unsolicited messages.

---

### Proof of Concept

```
1. Connect to a CKB node as an inbound peer (standard P2P handshake).
2. Fill max_inbound slots with attacker-controlled peers.
3. Each attacker peer sends a Ping message every ~1 ms in a tight loop.
4. Each Ping triggers ping_received() → last_ping_protocol_message_received_at = Instant::now().
5. When a legitimate peer attempts to connect, try_evict_inbound_peer() is called.
6. Round 2 of sort_then_drop protects the 8 most recently active peers.
7. All attacker peers have elapsed ≈ 0 s; legitimate peers have elapsed > 0 s.
8. Attacker peers are always in the protected tail; a legitimate peer is evicted instead.
9. Assert: legitimate peer receives PeerError::ReachMaxInboundLimit or is immediately
   evicted, while no attacker peer is ever disconnected.
```

fn sort_then_drop<T, F>(list: &mut Vec<T>, n: usize, compare: F)
where
    F: FnMut(&T, &T) -> std::cmp::Ordering,
{
    list.sort_by(compare);
    if list.len() > n {
        list.truncate(list.len() - n);
    }
}
