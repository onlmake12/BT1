### Title
Unbounded O(pool\_size) Full-Pool Scan on Every Block in `_update_tx_pool_for_reorg` Delays Mining - (`tx-pool/src/process.rs`)

---

### Summary

`_update_tx_pool_for_reorg` is invoked on every new best block and unconditionally scans **all** pending entries, all gap entries, and all pool entries (via `remove_expired`). These O(pool\_size) loops hold the exclusive `tx_pool.write()` lock for their entire duration, blocking `get_block_template` and new transaction submissions. Because any unprivileged tx submitter can inflate the pool toward its size limit, the per-block work grows proportionally with attacker-controlled pool occupancy, degrading mining throughput and, at sufficient pool size, causing miners to miss block-time windows.

---

### Finding Description

`_update_tx_pool_for_reorg` in `tx-pool/src/process.rs` is the CKB analog of the Allora `SafeApplyFuncOnAllActiveEpochEndingTopics` pattern: a function that iterates over an unbounded, attacker-influenced collection on every block in the critical processing path.

**Call chain:**

1. `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks` (block verification thread) calls `tx_pool_controller.update_tx_pool_for_reorg(...)` on every new best block. [1](#0-0) 

2. `TxPoolController::update_tx_pool_for_reorg` sends a notification to the dedicated reorg channel. [2](#0-1) 

3. The reorg task receives the notification and calls `service.update_tx_pool_for_reorg(...)`, which acquires `tx_pool.write()` and calls `_update_tx_pool_for_reorg`. [3](#0-2) 

**Inside `_update_tx_pool_for_reorg`, three unbounded scans occur on every block:**

**Scan 1 & 2 — Full gap and pending iteration (mine mode, active on mining nodes):**
```rust
for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) { ... }
for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) { ... }
```
Every gap entry and every pending entry is visited to check proposal promotion. No early exit exists; the entire collection is always scanned. [4](#0-3) 

**Scan 3 — Full pool iteration in `remove_expired`:**
```rust
let removed: Vec<_> = self.pool_map.iter()
    .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
    ...
    .collect();
```
`remove_expired` iterates over **every entry in the pool** on every block, even when no transactions are expired. [5](#0-4) 

**Scan 4 — `readd_detached_tx` re-verifies all detached transactions (reorg path):**

During any reorg, `readd_detached_tx` calls `verify_rtx` (full CKB-VM script execution) for every transaction from detached blocks that is not in the new chain. This is O(detached\_txs × script\_cost). [6](#0-5) 

**The write lock is held for the entire duration of all four operations:**
```rust
let mut tx_pool = self.tx_pool.write().await;
_update_tx_pool_for_reorg(...);
self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await;
``` [7](#0-6) 

While the lock is held, `get_block_template` (used by miners to assemble the next block) is blocked, because it also requires the pool lock.

---

### Impact Explanation

The pool size is bounded in bytes by `max_tx_pool_size` (configurable, default is large). An attacker who fills the pool with many small valid transactions maximizes the number of entries, making each per-block scan proportionally slower. On every block:

- Miners cannot obtain a new block template until the scan completes.
- New transaction submissions are blocked.
- If the scan takes longer than the block interval (10 seconds on mainnet), miners consistently miss their window, degrading block production throughput.
- In the worst case (very large pool + expensive `readd_detached_tx` during reorgs), block production can stall entirely.

This matches the Allora report's scenario 2: `topicPageLimit * maxTopicPages >= totalTopicNumbers` — all items are processed, but the computational cost is high enough to slow or halt production.

---

### Likelihood Explanation

- Any unprivileged tx submitter can submit valid transactions to fill the pool toward its size limit.
- The pool accepts transactions from RPC (`send_transaction`) and from P2P relay, both reachable without any privilege.
- Mining nodes (which run in `mine_mode`) are the most affected, as scans 1 and 2 only execute in mine mode.
- No special access, leaked keys, or majority hashpower is required.

---

### Recommendation

1. **Replace full-pool scans with index-driven lookups.** The `ProposalView` already stores proposed/gap IDs in `HashSet`s. Instead of iterating all pending entries and checking each against the proposal set, iterate the (much smaller) proposal set and look up entries by ID in O(1).

2. **Amortize `remove_expired` across blocks.** Instead of scanning all entries on every block, maintain a time-ordered index (e.g., a `BTreeMap<timestamp, ProposalShortId>`) so only entries near the expiry boundary are checked.

3. **Release the write lock before `readd_detached_tx`.** Re-verification of detached transactions does not require the pool write lock for its entire duration; it can be done with finer-grained locking.

---

### Proof of Concept

**Step 1:** Fill the tx pool with many small valid transactions via `send_transaction` RPC until `total_tx_size` approaches `max_tx_pool_size`.

**Step 2:** Observe that on every new block, the reorg task acquires `tx_pool.write()` and iterates over all pool entries (scans 1–3 above).

**Step 3:** Measure `get_block_template` latency before and after pool inflation. With a large pool, `get_block_template` is blocked for the duration of the full-pool scan on every block, increasing block template latency proportionally to pool size.

The relevant code paths are:

- Full pending scan: [8](#0-7)

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
