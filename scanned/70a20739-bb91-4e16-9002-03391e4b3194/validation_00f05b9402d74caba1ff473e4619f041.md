### Title
`check_purge` `take(len/2)` integer truncation allows single-group peer store flood to permanently block new peer admission — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
When all entries in the peer store belong to the same `/16` network group, the second-pass eviction in `check_purge` computes `take(1 / 2) == take(0)`, evicts nothing, and returns `EvictionFailed`, blocking all subsequent `add_addr` calls.

### Finding Description
`check_purge` groups stored addresses by `Group` (first two IPv4 octets) and attempts to evict from the top `len/2` groups. When `len == 1` (all addresses share one group), integer division yields `take(0)`, producing an empty candidate list. The function then returns `Err(PeerStoreError::EvictionFailed)` even though 16,384 addresses in a single oversized group are trivially evictable. [5](#0-4) 

The `Group::IP4` key is derived from only the first two octets of an IPv4 address, so all of `1.2.0.0/16` (65,535 distinct IPs × many ports) maps to `Group::IP4([1, 2])`. [6](#0-5) 

### Impact Explanation
`add_addr` returns `Err` to its caller for every subsequent discovery-advertised address. The node's peer store is saturated with attacker-controlled addresses; legitimate peer addresses cannot be stored. Peer discovery is effectively halted until the fake addresses age out through failed connection attempts (slow feeler rate × 16,384 entries).

### Likelihood Explanation
Any peer connected to the victim can send discovery `GetNodes`/`Nodes` responses. No authentication or per-group insertion cap exists. Filling 16,384 slots requires sending addresses in bulk over the discovery protocol, which is a normal operation — only the subnet homogeneity is adversarial.

### Recommendation
Replace `take(len / 2)` with `take(len.saturating_add(1) / 2)` (ceiling division) so that a single group is still selected when `len == 1`. Additionally, enforce a per-group insertion cap at `add_addr` time to prevent any single `/16` from consuming a disproportionate share of the store.
