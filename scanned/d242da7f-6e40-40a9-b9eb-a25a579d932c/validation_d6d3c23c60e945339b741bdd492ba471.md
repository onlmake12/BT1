### Title
Peer Store Permanently Deadlocked via Crafted Discovery Addresses — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
`check_purge` contains two compounding logic flaws: `take(len/2)` examines only half of all network groups, and the `addrs.len() > 4` guard excludes groups of exactly 4. An attacker who fills the store with 4096 groups of exactly 4 connectable addresses (totaling exactly `ADDR_COUNT_LIMIT = 16384`) causes every subsequent `add_addr` call to permanently return `Err(EvictionFailed)`, halting peer discovery.

### Finding Description
In `check_purge` (`peer_store_impl.rs`, lines 327–404), when the store is full and no non-connectable peers exist, the fallback eviction path:

1. Groups all addresses by `/16` network segment.
2. Sorts groups by descending size.
3. Calls `.take(len / 2)` — examining only the **top half** of groups by count.
4. For each examined group, evicts 2 random peers only `if addrs.len() > 4` — strictly greater than 4.

An attacker who advertises exactly 4096 distinct `/16` subnets with exactly 4 addresses each satisfies both failure conditions simultaneously: `take(4096/2) = take(2048)` processes 2048 groups, but every group has `len == 4`, which fails `> 4`. `candidate_peers` remains empty and `Err(PeerStoreError::EvictionFailed)` is returned unconditionally on every future `add_addr` call.

`add_addr` always inserts with `last_connected_at_ms=0` and `attempts_count=0` (the defaults in `AddrInfo::new`), so all attacker-supplied addresses are permanently connectable (`attempts_count=0 < ADDR_MAX_RETRIES=3`), blocking Step 1 eviction as well.

### Impact Explanation
Once the store is deadlocked, the victim node cannot accept any new peer addresses from the discovery protocol. The node's peer set becomes frozen at whatever peers it already knows, preventing it from learning about new honest peers. Over time (as existing peers go offline), the node's connectivity degrades and it risks being isolated from the honest chain tip. This is whole-network peer discovery starvation for any node an attacker can reach via a single P2P connection.

### Likelihood Explanation
The attacker needs only one P2P connection to the victim to advertise 16384 addresses. The discovery protocol is designed to relay peer addresses, and there is no rate-limiting or per-source quota visible in `add_addr`. The exact parameters (4096 groups × 4 peers) are trivially constructable. The attack is deterministic and requires no timing, no PoW, and no privileged access.

### Recommendation
Fix both flaws in the Step 2 eviction path:
1. Change `take(len / 2)` to `take(len)` (or remove the `take` entirely) so all groups are considered for eviction.
2. Change `addrs.len() > 4` to `addrs.len() >= 4` (or lower the threshold to `>= 2`) so groups at the boundary are also eligible for eviction.
Additionally, consider enforcing a per-source limit on how many addresses a single discovery peer can contribute to the store.
