### Title
Fee Rate Admission Check Uses `tx_size` (bytes) Instead of `weight` (cycle-adjusted bytes), Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` — (`tx-pool/src/util.rs`)

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee by calling `min_fee_rate.fee(tx_size as u64)`, passing the raw serialized byte size as the weight argument. However, `FeeRate` is defined as shannons per kilo-**weight**, where weight is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, weight can be orders of magnitude larger than `tx_size`, so the size-based check requires a fee far below what the actual fee rate policy demands. Any unprivileged transaction submitter can craft a small-but-cycle-heavy transaction that passes pool admission with an effective fee rate far below `min_fee_rate`.

### Finding Description

`FeeRate` is defined as shannons per kilo-weight (shannons/KW): [1](#0-0) 

The `fee()` method computes `rate * weight / 1000`: [2](#0-1) 

Transaction weight is defined as `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`: [3](#0-2) 

In `check_tx_fee`, the minimum fee is computed using `tx_size` (raw bytes) as the weight argument, not the actual weight: [4](#0-3) 

The code comment acknowledges the mismatch ("Theoretically we cannot use size as weight directly") but treats it as an intentional "cheap check." The result is that the admission gate uses a different unit than the fee rate policy specifies.

**Concrete example with `min_fee_rate = 1000` shannons/KW and `max_tx_verify_cycles = 70_000_000`:**

| Metric | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 |
| `weight` = `max(200, 70M × 0.000_170_571_4)` | **11,940** |
| `min_fee` via size-based check | `1000 × 200 / 1000` = **200 shannons** |
| `min_fee` via correct weight-based check | `1000 × 11940 / 1000` = **11,940 shannons** |
| Effective fee rate if fee = 200 shannons | `200 × 1000 / 11940 ≈ 16.7 shannons/KW` |

A transaction paying only 200 shannons passes `check_tx_fee` despite having an effective fee rate of ~16.7 shannons/KW — 60× below the configured `min_fee_rate` of 1000 shannons/KW.

The pool's `limit_size` eviction uses the correct weight-based fee rate, but only triggers when the pool is full: [5](#0-4) 

Until eviction, the transaction occupies pool memory, consumes verification cycles, and is relayed to peers.

### Impact Explanation

An unprivileged transaction submitter can inject cycle-heavy transactions into the mempool with effective fee rates far below `min_fee_rate`. This:

1. **Bypasses the fee rate filter** — the primary economic spam-prevention mechanism for the tx-pool.
2. **Consumes pool resources** — memory (`total_tx_size`) and verification cycles (`max_tx_verify_cycles`) are consumed before any eviction occurs.
3. **Propagates to peers** — relayed transactions force peer nodes to also verify and temporarily store them, amplifying the resource cost across the network.

The bypass factor scales with `cycles / tx_size`. At `max_tx_verify_cycles = 70M` and a minimal transaction, the effective fee rate can be ~60× below the configured minimum.

### Likelihood Explanation

Any transaction submitter (no privilege required) can craft a transaction with a small serialized body but high cycle consumption (e.g., a script that loops near the cycle limit). This is a standard, reachable entry point via the `send_transaction` RPC or P2P relay. The condition is deterministic and reproducible.

### Recommendation

Replace the size-based minimum fee computation in `check_tx_fee` with the weight-based computation:

```rust
// Instead of:
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);

// Use the declared cycles from the cache entry if available, or fall back to size:
let weight = get_transaction_weight(tx_size, declared_cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

If cycles are not yet known at the admission stage, the check should be deferred until after script verification, or the comment should be updated to document the known bypass and a compensating control (e.g., a cycle-based fee floor) should be added.

### Proof of Concept

1. Craft a transaction with a lock script that consumes ~70M cycles but has a small serialized size (~200 bytes).
2. Set the transaction fee to 200 shannons (just above the size-based minimum of `1000 × 200 / 1000 = 200`).
3. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
4. Observe the transaction is accepted into the pool despite having an effective fee rate of ~16.7 shannons/KW — 60× below `min_fee_rate`.
5. Repeat to fill the pool with below-minimum-fee-rate transactions, forcing legitimate transactions to compete with economically underpriced ones. [6](#0-5) [7](#0-6)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
