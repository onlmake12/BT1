### Title
Peer store permanently DoS'd via `Group::None` flooding — `take(len/2)` integer truncation evicts zero entries when all peers share one group — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
When all 16384 `ADDR_COUNT_LIMIT` slots are occupied by addresses that resolve to `Group::None` (e.g., Onion3), `check_purge`'s second-pass grouping computes `take(len / 2)` where `len = 1`, yielding `take(0)`. No eviction occurs, `EvictionFailed` is returned permanently, and the node can never add legitimate peer addresses again.

### Finding Description
`check_purge` in `peer_store_impl.rs` has two eviction passes. The second pass groups all stored addresses by `Group`, sorts by group size, then takes the top `len / 2` groups for eviction. When all entries map to `Group::None` (because `multiaddr_to_socketaddr` returns `None` for Onion3 or similar addresses), there is exactly one group (`len = 1`). Integer division gives `1 / 2 = 0`, so `.take(0)` processes no groups, produces no candidates, and the function returns `Err(PeerStoreError::EvictionFailed)`. Since `add_addr` calls `check_purge` before every insertion, the peer store is permanently locked against new entries.

The first pass also fails to help: entries added via `add_addr` always have `last_connected_at_ms = 0` and `attempts_count = 0`, so `is_connectable` returns `true` for all of them (neither non-connectable condition is met with `attempts_count < 3`).

### Impact Explanation
After the attack, the victim node's peer store is full of useless Onion3 entries. It cannot add any new legitimate IPv4/IPv6 peer addresses. Peer discovery is permanently broken for the lifetime of the process (or until the store is manually cleared). The node cannot reconnect to honest peers after disconnections and cannot expand its peer set. This degrades network connectivity without crashing the node or affecting consensus directly. Impact: **Medium (2001–10000)**.

### Likelihood Explanation
Any peer participating in the CKB discovery protocol can send `GetNodes` responses containing arbitrary `Multiaddr` entries. A single attacker peer can send 16384 unique Onion3 addresses across multiple responses. No PoW, no privileged access, no Sybil attack is required — one malicious peer is sufficient. The attack is cheap and deterministic.

### Recommendation
Fix the `take(len / 2)` expression to handle the single-group case. Options:
1. Use `take(len.saturating_add(1) / 2).max(1)` to always process at least one group.
2. Remove the `take(len / 2)` limit entirely and evict from all groups with `> 4` peers.
3. Treat `Group::None` entries as non-connectable (or cap the number of `Group::None` entries accepted) so the first pass can evict them.
