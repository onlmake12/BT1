### Title
Reorg Notification Silently Dropped via `try_send` on Bounded Channel Leaves Tx-Pool Permanently Inconsistent with Chain State — (`tx-pool/src/service.rs`)

---

### Summary

`TxPoolController::update_tx_pool_for_reorg` dispatches the reorg reconciliation work via a non-blocking `try_send` on a bounded Tokio MPSC channel (capacity `DEFAULT_CHANNEL_SIZE = 32`). If the channel is full the send fails, the error is only logged, and the tx-pool is **never reconciled** for that reorg. Because the chain's canonical state (RocksDB + shared snapshot) was already committed before the send, the tx-pool's internal snapshot permanently diverges from the chain, with no recovery path.

---

### Finding Description

**Root cause — `tx-pool/src/service.rs`**

The reorg channel is created with a fixed capacity of 32:

```rust
let (reorg_sender, reorg_receiver) = mpsc::channel(DEFAULT_CHANNEL_SIZE);
``` [1](#0-0) 

`DEFAULT_CHANNEL_SIZE` is 32: [2](#0-1) 

`update_tx_pool_for_reorg` on the controller uses `try_send` (non-blocking):

```rust
self.reorg_sender.try_send(notify).map_err(|e| {
    let (_m, e) = handle_try_send_error(e);
    e.into()
})
``` [3](#0-2) 

**Call site — `chain/src/verify.rs`**

After committing the new canonical tip to RocksDB and storing the new snapshot in `shared`, the chain service calls `update_tx_pool_for_reorg` and only logs the error on failure:

```rust
self.shared.store_snapshot(Arc::clone(&new_snapshot));   // ← chain state committed

if let Err(e) = tx_pool_controller.update_tx_pool_for_reorg(...) {
    error!("[verify block] notify update_tx_pool_for_reorg error {}", e);
    // ← no retry, no recovery, no pool reset
}
``` [4](#0-3) 

**What the dropped notification skips**

The reorg consumer task processes one message at a time and calls `TxPoolService::update_tx_pool_for_reorg`, which:

1. Updates the pool's internal snapshot (`tx_pool.snapshot = Arc::clone(&snapshot)`)
2. Removes committed transactions from the pool (`remove_committed_txs`)
3. Re-adds detached-block transactions (`readd_detached_tx`)
4. Handles detached proposals (`remove_by_detached_proposal`)
5. Cleans up the verify queue for newly attached transactions [5](#0-4) [6](#0-5) 

If the notification is dropped, **none of these steps execute**. The pool's internal snapshot stays at the pre-reorg tip while `shared` (used by RPC and the rest of the node) reflects the post-reorg tip.

**Why the channel can fill up**

The reorg consumer is a single sequential async loop: [7](#0-6) 

`readd_detached_tx` re-verifies every detached transaction inline (script execution), which can be slow for blocks with many transactions. During this time, the chain service continues processing new blocks. If 32 more reorg-triggering blocks arrive before the consumer finishes one reorg, the channel fills and subsequent `try_send` calls fail.

---

### Impact Explanation

When a reorg notification is dropped:

- **Stale pool snapshot**: The pool validates new incoming transactions against the pre-reorg UTXO set. Transactions spending cells that are live on the old fork but dead on the new fork are accepted into the pool; transactions spending cells that are live on the new fork but were spent on the old fork are incorrectly rejected.
- **Detached-block transactions permanently lost from mempool**: Transactions that were confirmed on the detached chain and should be re-queued for inclusion are silently discarded. Users must resubmit them manually with no indication that they were lost.
- **Committed transactions not evicted**: Transactions committed in the newly attached blocks remain in the pool. They will be re-broadcast to peers and re-proposed by miners, wasting network and CPU resources, until they fail validation at block assembly.
- **No recovery path**: There is no mechanism to force a pool reset or re-sync after a dropped notification. The inconsistency persists until the node is restarted.

---

### Likelihood Explanation

The channel capacity is 32. The reorg consumer processes one reorg at a time. During IBD or when a node receives a long competing fork (e.g., a 50-block reorg), the consumer may be occupied re-verifying hundreds of transactions while the chain service rapidly processes the remaining blocks of the new fork, each potentially triggering another reorg notification. An adversarial block relayer can craft a sequence of competing chains to maximize the number of reorg events delivered in rapid succession. No privileged access is required; any peer that can relay valid blocks can trigger this path.

---

### Recommendation

Replace `try_send` with a blocking or back-pressured send for reorg notifications, since a missed reorg is a correctness-critical event. Options:

1. Use an **unbounded channel** for the reorg sender so notifications are never dropped.
2. Use a **blocking send** (`send(...).await`) so the chain service waits until the pool is ready.
3. On `try_send` failure, **trigger a full pool reset** (`clear_pool`) to bring the pool back to a consistent (if empty) state rather than leaving it with a stale snapshot.

---

### Proof of Concept

```
1. Node is syncing a long fork (e.g., 100 blocks) that causes repeated reorgs.
2. Each block processed by the chain service calls update_tx_pool_for_reorg via try_send.
3. The reorg consumer is busy re-verifying transactions from the first reorg.
4. After 32 more reorg-triggering blocks arrive, the channel is full.
5. The 33rd try_send returns Err(Full), which is only logged.
6. The chain's shared snapshot now reflects block N+33's fork.
7. The pool's internal snapshot still reflects block N's fork.
8. New transactions submitted via RPC are validated against the wrong UTXO set.
9. Transactions from detached blocks N+1..N+32 are permanently absent from the mempool.
10. There is no API or automatic mechanism to recover the pool's consistency.
```

    pub(crate) async fn update_tx_pool_for_reorg(
        &self,
        detached_blocks: VecDeque<BlockView>,
        attached_blocks: VecDeque<BlockView>,
        detached_proposal_id: HashSet<ProposalShortId>,
        snapshot: Arc<Snapshot>,
    ) {
        let mine_mode = self.block_assembler.is_some();
        let mut detached = LinkedHashSet::default();
        let mut attached = LinkedHashSet::default();

        let detached_headers: HashSet<Byte32> = detached_blocks
            .iter()
            .map(|blk| blk.header().hash())
            .collect();

        for blk in detached_blocks {
            detached.extend(blk.transactions().into_iter().skip(1))
        }

        for blk in attached_blocks {
            self.fee_estimator.commit_block(&blk);
            attached.extend(blk.transactions().into_iter().skip(1));
        }
        let retain: Vec<TransactionView> = detached.difference(&attached).cloned().collect();

        let fetched_cache = self.fetch_txs_verify_cache(retain.iter()).await;

        // If there are any transactions requires re-process, return them.
        //
        // At present, there is only one situation:
        // - If the hardfork was happened, then re-process all transactions.
        {
            // This closure is used to limit the lifetime of mutable tx_pool.
            let mut tx_pool = self.tx_pool.write().await;

            _update_tx_pool_for_reorg(
                &mut tx_pool,
                &attached,
                &detached_headers,
                detached_proposal_id,
                snapshot,
                &self.callbacks,
                mine_mode,
            );

            // notice: readd_detached_tx don't update cache
            self.readd_detached_tx(&mut tx_pool, retain, fetched_cache)
                .await;
        }

        self.remove_orphan_txs_by_attach(&attached).await;
        {
            let mut queue = self.verify_queue.write().await;
            queue.remove_txs(attached.iter().map(|tx| tx.proposal_short_id()));
        }
    }

fn _update_tx_pool_for_reorg(
    tx_pool: &mut TxPool,
    attached: &LinkedHashSet<TransactionView>,
    detached_headers: &HashSet<Byte32>,
    detached_proposal_id: HashSet<ProposalShortId>,
    snapshot: Arc<Snapshot>,
    callbacks: &Callbacks,
    mine_mode: bool,
) {
    tx_pool.snapshot = Arc::clone(&snapshot);

    // NOTE: `remove_by_detached_proposal` will try to re-put the given expired/detached proposals into
    // pending-pool if they can be found within txpool. As for a transaction
    // which is both expired and committed at the one time(commit at its end of commit-window),
    // we should treat it as a committed and not re-put into pending-pool. So we should ensure
    // that involves `remove_committed_txs` before `remove_expired`.
    tx_pool.remove_committed_txs(attached.iter(), callbacks, detached_headers);
    tx_pool.remove_by_detached_proposal(detached_proposal_id.iter());
