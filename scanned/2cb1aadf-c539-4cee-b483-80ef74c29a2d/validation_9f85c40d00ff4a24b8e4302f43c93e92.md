Audit Report

## Title
`load_data_as_code` Charges Cycles on `memory_size` Instead of `content_size`, Enabling Cheap Verify-Worker Saturation — (`script/src/syscalls/load_cell_data.rs`)

## Summary

`load_data_as_code` (syscall 2091) charges `transferred_byte_cycles(memory_size)` after a successful `init_pages` call, regardless of how many bytes of actual cell data (`content_size`) were loaded into the VM. An unprivileged script author can set `memory_size` far above `content_size` — bounded only by the VM's usable address space (~124 MB) — and consume tens of millions of cycles with a single syscall, forcing every tx-pool verify worker that picks up the transaction to run near the cycle limit before rejection.

## Finding Description

In `load_data_as_code`, the only pre-charge guard is `content_size > memory_size` returning `SLICE_OUT_OF_BOUND`. There is no upper bound on `memory_size` itself:

```rust
// script/src/syscalls/load_cell_data.rs  lines 148–163
if content_offset >= cell.len() as u64
    || content_end > cell.len() as u64
    || content_size > memory_size          // only guard: content_size ≤ memory_size
{
    machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
    return Ok(());
}
machine.memory_mut().init_pages(addr, memory_size, FLAG_EXECUTABLE | FLAG_FREEZED,
    Some(cell.slice(content_offset..content_end)), 0)?;
sc.track_pages(machine, addr, memory_size, &data_piece_id, content_offset)?;
machine.add_cycles_no_checking(transferred_byte_cycles(memory_size))?;  // memory_size, not content_size
```

The cycle formula is:

```rust
// script/src/cost_model.rs  lines 10–12
pub fn transferred_byte_cycles(bytes: u64) -> u64 {
    bytes.div_ceil(BYTES_PER_CYCLE)   // BYTES_PER_CYCLE = 4
}
```

Cycles are only charged after `init_pages` returns `Ok`, so the attacker must choose `addr` and `memory_size` such that `addr + memory_size` stays within the VM's 128 MB address space and does not overlap already-initialized pages. With `RISCV_MAX_MEMORY = 128 MB`, a 4 MB stack at the top, and the ELF loaded at ~64 KB, the usable free window is ~124 MB. A single call with `memory_size = 60 MB` charges `transferred_byte_cycles(60 MB) = 15,728,640` cycles. Two such non-overlapping calls charge ~31.5 M cycles (~45% of the 70 M budget). Four calls with 30 MB each charge ~31.5 M cycles. The attacker can tune `memory_size` and call count to approach the full `DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES × 20 = 70,000,000` limit before the script is rejected.

Note: the report's specific claim of "two calls with `memory_size = RISCV_MAX_MEMORY`" is slightly overstated — `init_pages` would fail if `addr + memory_size` exceeds the VM boundary — but the core vulnerability is unaffected. The attacker can still consume the majority of the cycle budget per transaction with a handful of calls.

## Impact Explanation

This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

Each crafted transaction forces a verify worker to execute `init_pages` (real page-initialization work proportional to `memory_size`) and then be charged near-limit cycles before rejection. The verify manager spawns `max_tx_verify_workers = max(num_cpus * 3/4, 1)` workers, each processing one transaction at a time. Submitting `N ≥ max_tx_verify_workers` such transactions simultaneously pins all workers at near-cycle-limit execution, draining verify queue throughput. Legitimate transactions queue behind them indefinitely as long as the attacker keeps flooding.

## Likelihood Explanation

- **Unprivileged**: Any user can submit transactions via RPC or P2P relay.
- **Cheap**: A 1-byte cell dep (minimum ~61 CKB capacity, reusable across arbitrarily many transactions) plus `min_fee_rate × tx_size ≈ 300 shannons` per transaction. No fee-per-cycle mechanism exists in the tx-pool.
- **Repeatable**: The same cell dep can be referenced by unlimited transactions. The attacker can continuously flood the verify queue.
- **No PoW or consensus bypass needed**: The attack operates entirely within the tx-pool admission path.

## Recommendation

Charge cycles based on `content_size` (the actual bytes transferred into the VM), not `memory_size` (the virtual address range reserved). The zero-padding of the remaining `memory_size - content_size` bytes is a VM-internal page-initialization cost that should not be exposed as a user-controllable cycle multiplier.

```rust
// Proposed fix: charge for content_size, not memory_size
machine.add_cycles_no_checking(transferred_byte_cycles(content_size))?;
```

Alternatively, cap `memory_size` at `content_size` rounded up to the next page boundary before passing it to `init_pages`.

## Proof of Concept

1. Deploy a cell dep with 1 byte of data (`0x01`).
2. Write a minimal RISC-V ELF script that:
   - Sets `A0 = 0x20000` (addr, in free VM space above ELF load address)
   - Sets `A1 = 60 * 1024 * 1024` (memory_size = 60 MB)
   - Sets `A2 = 0` (content_offset)
   - Sets `A3 = 1` (content_size = 1 byte)
   - Sets `A4 = 0` (index)
   - Sets `A5 = source` (CellDep)
   - Sets `A7 = 2091` (`LOAD_CELL_DATA_AS_CODE_SYSCALL_NUMBER`)
   - Issues `ecall`; repeats with `A0 = 0x3C20000` (second non-overlapping region); continues until cycle budget is exhausted
3. Submit `N ≥ max_tx_verify_workers` such transactions simultaneously.
4. Observe: all verify workers are pinned at near-cycle-limit; `verify_queue_size` grows; legitimate transactions stall.

Expected cycle measurement per call: `transferred_byte_cycles(60 MB) = 60 × 1024 × 1024 / 4 = 15,728,640`, confirmed by the formula at `script/src/cost_model.rs:10–12`. The `DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES × 20 = 70,000,000` is confirmed at `spec/src/consensus.rs:70` and `util/app-config/src/legacy/tx_pool.rs:14`.
