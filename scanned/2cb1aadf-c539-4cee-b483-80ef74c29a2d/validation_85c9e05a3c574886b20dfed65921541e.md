Audit Report

## Title
Unbounded `peers` Vec Growth via Repeated `RelayTransactionHashes` with Phantom Hashes — (`sync/src/types/mod.rs`, `sync/src/relayer/transaction_hashes_process.rs`)

## Summary
`push_peer()` appends a `PeerIndex` to the `peers` Vec of an `UnknownTxHashPriority` entry without any deduplication or size cap. The overflow guard in `add_ask_for_txs` only fires when the number of **unique hash keys** in the queue exceeds `MAX_UNKNOWN_TX_HASHES_SIZE` (50 000); it never inspects the size of individual `peers` Vecs. An attacker who repeatedly announces the same small set of phantom transaction hashes (hashes for transactions that never exist) causes the `peers` Vec for each entry to grow by one per repetition, indefinitely, leading to OOM and node crash.

## Finding Description
**Entry point** — `TransactionHashesProcess::execute()` filters announced hashes through `tx_filter`, then calls `state.add_ask_for_txs(self.peer, tx_hashes)`. [1](#0-0)  Phantom hashes (for transactions that are never delivered) are never passed to `mark_as_known_txs`, so they are never inserted into `tx_filter`. [2](#0-1)  Every repeated announcement of the same phantom hashes passes the filter and reaches `add_ask_for_txs`.

**Unconditional append** — For a hash already present in `unknown_tx_hashes`, `add_ask_for_txs` clones the existing `UnknownTxHashPriority`, calls `push_peer(peer_index)`, and writes it back. [3](#0-2)  `push_peer` has no deduplication and no cap: [4](#0-3) 

**Overflow guard is bypassed** — The guard fires only when `unknown_tx_hashes.len()` (the count of unique hash keys) reaches `MAX_UNKNOWN_TX_HASHES_SIZE` (50 000) or the per-peer product. [5](#0-4)  An attacker using N < 50 000 phantom hashes keeps `unknown_tx_hashes.len()` fixed at N; the guard never fires, and the per-peer counter check inside it is never reached. The `peers` Vec for each of the N entries grows by 1 on every repetition.

**Rate limiter** — 30 messages/second per `(PeerIndex, message_type)`. [6](#0-5)  `MAX_RELAY_TXS_NUM_PER_BATCH` = 32 767 hashes per message. [7](#0-6)  This gives up to ~983 000 `PeerIndex` appends per second.

**Drain rate is negligible** — `pop_ask_for_txs` runs every 100 ms and removes at most one peer per hash per `RETRY_ASK_TX_TIMEOUT_INCREASE` (30 s). [8](#0-7) [9](#0-8)  Net growth rate is effectively the full injection rate.

## Impact Explanation
Each `PeerIndex` is a `usize` (8 bytes on 64-bit). At 30 messages/s × 100 phantom hashes per message, the `peers` Vecs accumulate ~983 000 entries/second. After a few minutes the process RSS exceeds available RAM and the OS OOM-kills the node. This is a complete denial of service of a single CKB node, matching the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10 001–15 000 points).

## Likelihood Explanation
The attack requires only a single connected peer with no special privileges, no PoW, and no keys. The attacker generates phantom 32-byte hashes (for transactions that do not exist), so the victim never fetches the actual transactions and the hashes are never added to `tx_filter`. The attack sustains indefinitely within the 30 req/s rate limit. No victim mistake is required.

## Recommendation
1. **Deduplicate in `push_peer`**: check `self.peers.contains(&peer_index)` before appending, or replace `Vec<PeerIndex>` with `HashSet<PeerIndex>` for O(1) dedup. [4](#0-3) 
2. **Cap `peers` Vec length**: enforce a hard maximum equal to the number of connected peers or a small constant (e.g., 8).
3. **Move the per-peer counter check outside the overflow guard**: count per-peer `peers` Vec entries unconditionally on every insertion, not only when the unique-key threshold is exceeded. [5](#0-4) 
4. **Per-peer recently-announced hash tracking**: maintain a per-peer LRU set of recently announced hashes and drop duplicates before calling `add_ask_for_txs`. [1](#0-0) 

## Proof of Concept
1. Connect to a CKB node as a relay peer.
2. Generate 100 random 32-byte values as phantom tx hashes (no corresponding transactions exist).
3. In a loop at ≤30 iterations/second: send a `RelayTransactionHashes` message containing all 100 hashes.
4. After 1 000 iterations, inspect `unknown_tx_hashes` (via instrumentation or debug RPC): each of the 100 entries will have `peers.len() == 1000`. The queue length stays at 100 (well below `MAX_UNKNOWN_TX_HASHES_SIZE` = 50 000), confirming the guard never fires.
5. Continue until the CKB process RSS grows proportionally and the node is OOM-killed.
