### Title
Unsafe `u128 as u64` Truncating Cast in DAO Withdrawal Capacity Calculation - (File: `util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` in `util/dao/src/lib.rs` performs an unsafe `u128 as u64` truncating cast on the computed withdrawal capacity. If the intermediate `u128` result exceeds `u64::MAX`, the value is silently truncated, producing an incorrect (too-small) maximum withdrawal amount. This is inconsistent with every other `u128→u64` narrowing conversion in the same file, all of which use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`.

### Finding Description

In `calculate_maximum_withdraw` at lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← unsafe truncating cast
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The arithmetic is correctly widened to `u128` to avoid overflow during multiplication, but the final narrowing back to `u64` is done with the bare `as u64` cast, which silently discards the upper 64 bits if the value exceeds `u64::MAX`.

Every other analogous `u128→u64` conversion in the same file uses the safe checked form:

```rust
// line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

The inconsistency is the root cause: the developer applied safe conversion discipline everywhere except this one site.

### Impact Explanation

`calculate_maximum_withdraw` is called from two paths:

1. **`transaction_fee`** (line 30–35) → `FeeCalculator::transaction_fee` in `verification/src/transaction_verifier.rs` (line 265–273). A truncated `maximum_withdraw` that is smaller than `outputs_capacity` causes a `DaoError::Overflow` on `safe_sub`, making the node reject a valid DAO withdrawal transaction. The depositor cannot reclaim their funds through this node.

2. **`withdrawed_interests`** (line 312–333) → `dao_field_with_current_epoch` (line 209–263). A wrong `withdrawed_interests` value produces an incorrect DAO field (`current_s`), which is then verified by `DaoHeaderVerifier`. Nodes that compute different values for the DAO field would disagree on block validity, creating a consensus split. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX  ≈ 1.84 × 10¹⁹
```

`counted_capacity` is bounded by the total CKB supply (~3.36 × 10¹⁸ shannons). The accumulation rate (`ar`) starts at `10¹⁶` and grows slowly via secondary issuance. For the ratio `withdrawing_ar / deposit_ar` to exceed ~5.5 (the threshold needed to overflow), the AR would need to grow by more than 5.5× from deposit to withdrawal — a process that would take centuries at current issuance rates on mainnet.

Likelihood on mainnet is therefore **extremely low** in the near term. However, the bug is a latent correctness defect that is already inconsistent with the rest of the codebase, and it would be silently triggered (no panic, no error) if the condition is ever met.

### Recommendation

Replace the unsafe cast with the same checked pattern used elsewhere in the file:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
``` [7](#0-6) 

### Proof of Concept

The truncation is purely arithmetic. Given:

- `counted_capacity = u64::MAX` (maximum possible cell capacity, ~1.84 × 10¹⁹ shannons)
- `withdrawing_ar = 2 × deposit_ar` (AR has doubled since deposit)

Then:

```
withdraw_counted_capacity = u64::MAX × 2 / 1 = 3.68 × 10¹⁹  (fits in u128)
withdraw_counted_capacity as u64 = 3.68×10¹⁹ mod 2⁶⁴ ≈ 1.84×10¹⁹ - 1  (truncated)
```

The node would compute a `withdraw_capacity` that is ~`u64::MAX` shannons less than the correct value, causing either a spurious `DaoError::Overflow` on `safe_sub` in `transaction_fee`, or an incorrect DAO field in block header verification.
