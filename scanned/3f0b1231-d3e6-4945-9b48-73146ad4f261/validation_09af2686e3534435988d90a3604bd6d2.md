### Title
`RecentReject::put` Never Increments `total_keys_num`, Allowing Unbounded Disk Growth via Rejected-Transaction Spam — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject` is a sharded `DBWithTTL`-backed store that records recently rejected transactions so that RPC callers can query rejection reasons. It is supposed to enforce a configurable `count_limit` (default 10,000,000 entries). However, the `put()` method computes a local incremented value but never writes it back to `self.total_keys_num`, so the limit check always evaluates against the stale initial estimate. The shrink path is therefore never triggered during a node's lifetime unless the database was already at or above the limit at startup. Any unprivileged transaction sender can fill the disk by submitting a stream of transactions that are rejected with a recordable rejection reason.

---

### Finding Description

In `tx-pool/src/component/recent_reject.rs`, the `put` method is:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ... write to DB ...
    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
        if total_keys_num > self.count_limit {
            self.shrink()?;
        }
    } else {
        self.shrink()?;
    }
    Ok(())
}
```

`total_keys_num` is a **local variable** holding `self.total_keys_num + 1`. The field `self.total_keys_num` is **never assigned** inside `put()`. After every call, `self.total_keys_num` remains at whatever value it held when the node started (the RocksDB key-count estimate from construction). The guard `total_keys_num > self.count_limit` therefore evaluates `(initial_estimate + 1) > count_limit` on every single call — a value that is almost always `false` for a freshly started node.

`self.total_keys_num` is only updated inside `shrink()`:

```rust
fn shrink(&mut self) -> Result<u64, AnyError> {
    // drop and recreate a random shard
    let total_keys_num = self.estimate_total_keys_num()?;
    self.total_keys_num = total_keys_num;   // only update site
    Ok(total_keys_num)
}
```

Because `shrink()` is never reached, the database grows without bound for the entire duration of the node's uptime, limited only by the TTL expiry window (default 7 days).

The correct fix is to assign `self.total_keys_num = total_keys_num;` immediately after the `checked_add(1)` succeeds, before the limit check.

---

### Impact Explanation

Every rejected transaction whose `Reject::should_recorded()` returns `true` is written to the `recent_reject` database via `put_recent_reject`. Because the count limit is never enforced, an attacker who continuously submits transactions that are rejected for a recordable reason (e.g., fee-too-low, malformed, RBF-rejected) will cause the on-disk database to grow at the rate of one entry per rejected transaction, bounded only by the 7-day TTL window. At the default limit of 10,000,000 entries, each entry serialised as a JSON string is on the order of hundreds of bytes, so sustained spam can exhaust gigabytes of disk space. A full disk causes RocksDB writes to fail, which propagates as errors through the tx-pool and can halt block processing or crash the node.

---

### Likelihood Explanation

The entry path requires no privilege: any peer can broadcast transactions over the P2P relay protocol or submit them via the `send_transaction` RPC. Transactions that fail fee-rate checks, capacity checks, or script verification are all candidates for `should_recorded()`. The cost per entry to the attacker is a single low-fee (or zero-fee, if the node's minimum fee rate is zero in dev/test mode) transaction broadcast. On mainnet the cost is non-trivial per transaction, but the attacker only needs to sustain a rate sufficient to fill the disk before TTL expiry clears old entries. Given that the default TTL is 7 days and the default limit is 10 million entries, the effective cap is absent for any realistic attack window.

---

### Recommendation

In `RecentReject::put`, assign the incremented value back to `self.total_keys_num` before the limit check:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;   // ← add this line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

Additionally, consider using `>=` instead of `>` so that the limit is enforced at exactly `count_limit` entries rather than `count_limit + 1`.

### Proof of Concept

1. Start a CKB node with default tx-pool config (`keep_rejected_tx_hashes_count = 10_000_000`, `keep_rejected_tx_hashes_days = 7`).
2. Observe that `RecentReject` is constructed with `total_keys_num = 0` (empty database).
3. Submit transactions that fail fee-rate validation in a tight loop via `send_transaction` RPC or P2P relay.
4. Each call to `put_recent_reject` → `RecentReject::put` writes one entry to RocksDB and then evaluates `0 + 1 > 10_000_000` → `false`, so `shrink()` is never called and `self.total_keys_num` stays `0`.
5. After N iterations, the database contains N entries with no eviction. Disk usage grows linearly with N.
6. Confirm by inspecting `recent_reject.get_estimate_total_keys_num()` — it will always return `0` regardless of how many entries have been written.
