### Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` After In-Flight Eviction, Causing Permanently Inflated Pool-Size Accounting — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::add_entry` snapshots the new pool-size totals **before** calling `check_and_record_ancestors`, which may itself evict entries and correctly decrement `self.total_tx_size` / `self.total_tx_cycles`. The function then unconditionally overwrites those fields with the pre-eviction snapshot, silently discarding every decrement that happened during eviction. The result is a permanently inflated `total_tx_size` that causes `limit_size` to over-evict legitimate transactions from the pool.

---

### Finding Description

In `add_entry` the sequence is:

1. `updated_stat_for_add_tx` (takes `&self`, returns new totals as **local variables** `total_tx_size`, `total_tx_cycles`).
2. `check_and_record_ancestors` — when `ancestors_count > max_ancestors_count` and `cell_ref_parents` exist, it calls `remove_entry_and_descendants`, which calls `update_stat_for_remove_tx`, which **mutates `self.total_tx_size` and `self.total_tx_cycles`** in place.
3. `self.total_tx_size = total_tx_size` — unconditionally **overwrites** the field with the stale pre-eviction snapshot, erasing every decrement from step 2.

```
// step 1 – snapshot BEFORE eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// step 2 – may call remove_entry_and_descendants → update_stat_for_remove_tx
//           which mutates self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

// step 3 – OVERWRITES the mutated self.total_tx_size with the stale snapshot
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` is a pure read (`&self`) that returns new values without touching `self`: [2](#0-1) 

`update_stat_for_remove_tx` (called inside `remove_entry`) **does** mutate `self.total_tx_size`: [3](#0-2) 

`remove_entry` is called from `remove_entry_and_descendants`, which is called from `check_and_record_ancestors`: [4](#0-3) [5](#0-4) 

**Concrete example:**

| Step | `self.total_tx_size` | local `total_tx_size` |
|---|---|---|
| Initial | 100 | — |
| After `updated_stat_for_add_tx(+10)` | 100 | **110** |
| After eviction of 30 bytes inside `check_and_record_ancestors` | **70** | 110 |
| After `self.total_tx_size = total_tx_size` | **110** ← wrong | — |

Correct value: 70 + 10 = **80**. Actual value: **110** (overcounted by 30).

---

### Impact Explanation

`total_tx_size` is the sole gate for pool-size enforcement:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict entries
}
``` [6](#0-5) 

An inflated `total_tx_size` causes `limit_size` to evict additional legitimate transactions that would otherwise fit within the configured `max_tx_pool_size`. The overcounting is **permanent** for the lifetime of the pool (until a node restart or `clear_tx_pool`): every subsequent `add_entry` that triggers the eviction path adds more phantom bytes to the total. Over time, the pool can be driven to a state where it continuously evicts valid transactions even though the real occupancy is well below the limit, effectively denying service to honest users.

Additionally, `tx_pool_info` RPC reports the inflated `total_tx_size` to callers, giving a false picture of pool occupancy. [7](#0-6) 

---

### Likelihood Explanation

The trigger condition requires:

1. A submitted transaction whose in-pool ancestor count exceeds `max_ancestors_count` (default 125).
2. At least one of those ancestors is a `cell_ref_parent` — i.e., a pool transaction that references one of the new transaction's **outputs** as a cell dep.
3. Evicting those `cell_ref_parents` brings the ancestor count back within the limit.

An attacker can deliberately construct this scenario:

- Submit a long ancestor chain (≥ 125 transactions deep).
- Submit a transaction B that uses an output of a mid-chain transaction A as a cell dep.
- Submit a new transaction C that spends A's output. C's ancestor count exceeds the limit; B is a `cell_ref_parent` and gets evicted. `total_tx_size` is now overcounted by B's serialized size.

Repeating this pattern accumulates phantom bytes. Because `send_transaction` is open to any unprivileged RPC caller, no special privilege is required.

---

### Recommendation

Capture the post-eviction totals rather than the pre-eviction snapshot. The simplest fix is to move the stat update to **after** `check_and_record_ancestors` completes:

```rust
// Remove the pre-computation of total_tx_size / total_tx_cycles.
// After check_and_record_ancestors (which may evict), add the new entry's contribution:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

Alternatively, re-read `self.total_tx_size` after `check_and_record_ancestors` and add only the new entry's contribution, rather than using the stale snapshot.

---

### Proof of Concept

```
Initial pool: total_tx_size = 90 MB (max = 100 MB)

1. Attacker builds a 125-deep ancestor chain (tx0 → tx1 → … → tx124).
   Each tx is ~1 KB. Chain is submitted; pool grows to ~90.125 MB.

2. Attacker submits tx_dep: spends tx60's output as a cell dep.
   tx_dep is accepted; pool grows slightly.

3. Attacker submits tx_new: spends tx60's output directly.
   - tx_new's ancestor count = 65 (tx0..tx60 + tx_dep) > 125? No, but
     adjust chain length so that tx_new's ancestor count = 126.
   - cell_ref_parents = {tx_dep}
   - check_and_record_ancestors evicts tx_dep (say 1 KB):
       self.total_tx_size decremented by 1 KB → 89.125 MB
   - add_entry then writes back total_tx_size = 90.125 MB + 1 KB = 90.126 MB
     (the decrement is lost; pool is overcounted by 1 KB)

4. Repeat step 3 with fresh inputs. Each iteration adds ~1 KB of phantom bytes.
   After ~10 000 iterations, total_tx_size reads ~100 MB even though real
   occupancy is ~90 MB. limit_size now continuously evicts honest transactions.

        mut entry: TxEntry,

        let mut evicts = Default::default();
        if self.entries.get_by_id(&tx_short_id).is_some() {

        let (total_tx_size, total_tx_cycles) =

        self.record_entry_edges(&entry)?;
        self.insert_entry(&entry, status);
        self.record_entry_descendants(&entry);
        self.track_entry_statics(None, Some(status));
        self.total_tx_size = total_tx_size;
        self.total_tx_cycles = total_tx_cycles;
        Ok((true, evicts))

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    /// Calculate size and cycles statistics for adding a tx.
    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
