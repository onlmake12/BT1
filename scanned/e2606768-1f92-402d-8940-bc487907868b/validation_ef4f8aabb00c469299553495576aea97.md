### Title
`ScriptHashTypeVerifier` Omits Type Script Hash Type Validation, Allowing Permanently Unspendable Outputs — (File: `verification/src/transaction_verifier.rs`)

### Summary

The `ScriptHashTypeVerifier` in `verification/src/transaction_verifier.rs` only validates the `hash_type` field of **lock scripts** in transaction outputs, but completely omits the same check for **type scripts**. This allows any transaction sender to create on-chain outputs whose type script carries a consensus-disabled `hash_type`. Such outputs are permanently unspendable: every future attempt to spend them will fail at script execution time with `InvalidVmVersion` or `InvalidScriptHashType`, causing permanent loss of the CKB capacity locked inside.

---

### Finding Description

The verifier's own doc-comment states its intent:

> *"Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules."*

The implementation, however, only iterates over `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  lines 796-815
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
```

`output.type_().hash_type()` is never inspected. A type script carrying `ScriptHashType::Data2` (or any future hash type) before the corresponding hardfork epoch is silently accepted.

At spend time, `select_version` in `script/src/types.rs` enforces the hardfork gate:

```rust
// script/src/types.rs  lines 914-919
ScriptHashType::Data2 => {
    if is_vm_version_2_and_syscalls_3_enabled {
        Ok(ScriptVersion::V2)
    } else {
        Err(ScriptError::InvalidVmVersion(2))
    }
}
```

Because the creation-time check is absent, the output is committed to the chain. Because the spend-time check is present, every subsequent spend attempt fails. The cell is frozen on-chain with no recovery path.

The lower-level `check_data` in `util/gen-types/src/extension/check_data.rs` does cover both lock and type scripts, but it only verifies that the raw byte is a *known* `ScriptHashType` variant — it does not enforce the consensus-activation gate that `ENABLED_SCRIPT_HASH_TYPE` represents. The two checks are complementary; the gap is exclusively in `ScriptHashTypeVerifier`.

---

### Impact Explanation

Any CKB capacity locked in an output whose type script carries a consensus-disabled `hash_type` is irrecoverable. The output passes all admission checks (non-contextual and contextual), is committed to the canonical chain, and then becomes permanently unspendable. This is a direct, on-chain permanent loss of funds for the output owner, analogous to a token vault being frozen by an external freeze authority.

---

### Likelihood Explanation

The `send_transaction` RPC is reachable by any unprivileged user. The `NonContextualTransactionVerifier` — which calls `ScriptHashTypeVerifier::verify()` — runs on every submitted transaction before pool admission. A user or a higher-level protocol (e.g., a DEX, a bridge, or an issuance contract) that programmatically constructs outputs and sets a type script `hash_type` to a future value (e.g., `Data2 = 0x02` before the CKB2023 epoch) will have the transaction accepted and committed, with no warning. The mistake is easy to make when porting scripts across network versions or when a protocol does not re-validate the `hash_type` of user-supplied type scripts before forwarding them to outputs.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_hash_type: u8 = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?
            .into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&lock_hash_type) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: lock_hash_type }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_hash_type: u8 = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?
                .into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&type_hash_type) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: type_hash_type }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Before the CKB2023 hardfork epoch (i.e., while `Data2` is not yet in `ENABLED_SCRIPT_HASH_TYPE`), submit via `send_transaction`:

```json
{
  "outputs": [{
    "capacity": "0x2540be400",
    "lock": { "code_hash": "<secp256k1_type_hash>", "hash_type": "type", "args": "<20-byte-pubkey-hash>" },
    "type": { "code_hash": "<any_code_hash>", "hash_type": "data2", "args": "0x" }
  }],
  "outputs_data": ["0x"],
  ...
}
```

2. `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`. The lock script's `hash_type = type` passes. The type script's `hash_type = data2` is **never checked**. The transaction is accepted.

3. The transaction is proposed and committed. The output is now a live cell on-chain.

4. Any attempt to spend this cell triggers `select_version` on the type script, which returns `Err(ScriptError::InvalidVmVersion(2))` because `is_vm_version_2_and_syscalls_3_enabled` is `false`. The spend fails.

5. The capacity in the output is permanently inaccessible.

                }
