### Title
TOCTOU Race in `HeaderMapKernel::limit_memory` Causes Ghost Sled Entries and Stale Header Resurrection — (`shared/src/types/header_map/kernel_lru.rs`)

### Summary
A race between `limit_memory`'s non-atomic snapshot-then-write sequence and a concurrent `remove` call allows a header to be written to the sled backend after its removal, creating a ghost entry that is never cleaned up during the node's lifetime.

### Finding Description
`limit_memory` operates in three non-atomic steps:
1. `memory.front_n` snapshots keys S under a read lock, then **releases the lock**.
2. `backend.insert_batch(&values)` writes S to sled.
3. `memory.remove_batch(...)` removes S from memory.

`remove` uses `backend.is_empty()` as a guard before calling `backend.remove_no_return`. This check is not atomic with `insert_batch`. If `remove(hash_k)` executes between steps 1 and 2 — after `front_n` releases the read lock but before `insert_batch` runs — it sees an empty backend, skips the sled deletion, and removes `hash_k` from memory. Then `insert_batch` writes `hash_k` to sled, and `remove_batch` finds it already gone from memory (no-op). `hash_k` is now a ghost in sled.

### Impact Explanation
- **Stale header resurrection:** Any subsequent `get(hash_k)` finds the ghost in sled, removes it from sled, re-inserts it into memory, and returns the stale `HeaderIndexView`. Headers that were explicitly removed (e.g., from orphaned forks) can be resurrected and influence sync decisions.
- **Disk space growth:** Ghost entries accumulate in the sled `TempDir` for the duration of the node's lifetime. Under sustained header flooding, this can grow to hundreds of MB.
- **Incorrect `contains_key` results:** `contains_key` returns `true` for removed headers, corrupting the node's view of which headers are known.

### Likelihood Explanation
The race window is the time between `front_n` returning and `insert_batch` completing (including `tokio::task::block_in_place`). Block verification routinely calls `remove` concurrently. An attacker keeping memory near `memory_limit` via header relay maximizes `limit_memory` frequency (every 5 s) and the probability of hitting the window. No privileges are required beyond sending valid P2P headers.

### Recommendation
Eliminate the non-atomic snapshot-then-write by holding the memory write lock across the `front_n` + `remove_batch` steps, or by re-checking membership in memory before inserting into the backend. Alternatively, replace the `backend.is_empty()` fast-path in `remove` with an unconditional `backend.remove_no_return` call so that any entry written to sled after a concurrent `remove` is immediately cleaned up on the next `remove` call. A simpler fix: in `limit_memory`, after `insert_batch`, re-check each key against memory before calling `remove_batch`, and issue a `backend.remove_no_return` for any key that is no longer in memory.

### Proof of Concept
```
Thread A (limit_memory):                Thread B (remove hash_k):
front_n() → snapshots S, lock released
                                        memory.remove(hash_k)  ← succeeds
                                        backend.is_empty() → true → skip backend removal
backend.insert_batch(S)  ← writes hash_k to sled
memory.remove_batch(S)   ← hash_k already gone, no-op

State: hash_k absent from memory, present in sled (ghost)

Later: get(hash_k) → misses memory → finds ghost in sled → resurrects into memory → returns stale view
```
