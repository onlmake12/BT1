Audit Report

## Title
Global `unknown_tx_hashes` Soft Limit Bypassed by Post-Insertion Check in `add_ask_for_txs` - (`File: sync/src/types/mod.rs`)

## Summary

`SyncState::add_ask_for_txs` unconditionally inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) entries per peer into the global `unknown_tx_hashes` map before evaluating the global soft limit (`MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000). With N concurrent peers each sending disjoint sets of novel tx hashes, the map grows to N × 32,767 entries — far beyond the intended cap. The overflow also triggers an O(map_size) full-map scan while holding the mutex, blocking the relay subsystem's `pop_ask_for_txs` timer and degrading transaction propagation.

## Finding Description

**Root cause:** In `sync/src/types/mod.rs`, `add_ask_for_txs` (line 1483) first iterates and inserts all incoming hashes (lines 1486–1504), then checks the global limit (lines 1507–1509). Even when the check fires and returns `TooManyUnknownTransactions` or `Status::ignored()`, the entries are already committed to the map.

**Constants mismatch** (`util/constant/src/sync.rs`, lines 68–72):
- `MAX_RELAY_TXS_NUM_PER_BATCH` = `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767
- `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000
- The global cap (50,000) is smaller than 2 × per-peer quota (65,534), so just two peers suffice to exceed it.

**Exploit flow:**
1. Peer A sends one `RelayTransactionHashes` message with 32,767 unique novel hashes. `add_ask_for_txs` inserts all of them (map: 32,767 < 50,000 → limit check not triggered). Returns `Status::ok()`.
2. Peer B sends 32,767 different novel hashes. All inserted (map: 65,534). Post-insertion check fires (65,534 ≥ 50,000). Per-peer counter for Peer B = 32,767 ≥ `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` → returns `TooManyUnknownTransactions`. **Entries remain in the map.**
3. Peer C sends 32,767 more. All inserted (map: 98,301). Check fires; Peer C's counter = 32,767 ≥ threshold → `TooManyUnknownTransactions`. **Entries remain.**
4. With N peers (up to `MAX_RELAY_PEERS` = 128): map grows to N × 32,767 ≈ 4.2M entries.

**Why existing guards fail:**
- The per-peer rate limiter in `sync/src/relayer/mod.rs` (lines 116–123) limits message frequency (30 req/s per peer per message type), not cumulative map size. One message per peer is sufficient.
- `TransactionHashesProcess::execute` (`sync/src/relayer/transaction_hashes_process.rs`, lines 25–49) only filters hashes already in `tx_filter`; novel fake hashes pass through unconditionally.
- The second condition `self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` scales with peer count, so it never triggers before the first condition and does not prevent growth.

**O(N) mutex scan:** Once the global threshold is exceeded, every subsequent call to `add_ask_for_txs` executes the full-map iteration at lines 1516–1523 while holding `unknown_tx_hashes.lock()`. This is O(map_size × avg_peers_per_entry). With 4M entries, this blocks `pop_ask_for_txs` (called every 100ms via `ASK_FOR_TXS_TOKEN`) and `mark_as_known_txs`, stalling transaction relay.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

With `MAX_RELAY_PEERS` = 128, the map can reach ~4.2M entries. Each entry holds a `Byte32` (32 bytes) plus a `Vec<PeerIndex>`, yielding hundreds of MB of heap growth. More critically, the O(map_size) scan while holding the mutex blocks the 100ms relay timer (`ASK_FOR_TXS_TOKEN`) that dispatches `GetRelayTransactions` requests to peers. Sustained mutex contention stalls transaction propagation across the node, causing relay degradation and contributing to network-wide congestion. The attack requires no special privilege and minimal bandwidth (one P2P message per peer).

## Likelihood Explanation

**Medium-High.** Any peer that completes the P2P handshake can send `RelayTransactionHashes`. Only two peers with disjoint sets of fake (non-existent) tx hashes are needed to exceed the 50,000 global limit. The hashes need not correspond to real transactions — the only filter is `tx_filter` (known transactions), which novel hashes bypass. No key, hashpower, or elevated privilege is required. The rate limiter does not prevent this since only one message per peer is needed.

## Recommendation

Check the global limit **before** inserting, not after. At the start of `add_ask_for_txs`, before the insertion loop, check whether `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE`. If so, perform the per-peer count check first and reject if the peer is already at quota, or drop the request entirely. This mirrors the correct pattern: enforce the global cap pre-insertion, then enforce per-peer caps.

Additionally, maintain a separate `HashMap<PeerIndex, usize>` tracking per-peer entry counts to replace the O(map_size) scan at lines 1516–1523 with an O(1) lookup, eliminating the mutex contention amplification.

## Proof of Concept

1. Start a CKB node and connect two test peers (Peer A, Peer B) via the relay protocol.
2. Peer A sends one `RelayTransactionHashes` message containing 32,767 unique `Byte32` values not present in `tx_filter`. Observe `unknown_tx_hashes.len()` = 32,767; no limit check triggered.
3. Peer B sends one `RelayTransactionHashes` message containing 32,767 **different** unique `Byte32` values. Observe `unknown_tx_hashes.len()` = 65,534 after the call returns `TooManyUnknownTransactions` — entries are already inserted, global cap bypassed.
4. Add a third peer (Peer C) with another 32,767 disjoint hashes. Observe map grows to 98,301.

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)

                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })

            }
        }
