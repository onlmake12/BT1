Audit Report

## Title
Minimum Fee Check Uses Transaction Size Instead of Weight, Allowing High-Cycle Transactions Below Effective Fee Rate — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the serialized transaction size to compute the minimum required fee, not the transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because cycles are unknown before script execution, no weight-based fee check is performed either before or after `verify_rtx`. An attacker can submit a transaction with near-maximum cycles but minimal serialized size, pay only the size-based minimum fee, force the node to execute up to 70M cycles of script verification, and have the transaction admitted to the pool at an actual fee rate ~50× below the configured minimum.

## Finding Description

`FeeRate` is defined as shannons per kilo-**weight**, where weight = `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` (`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`). However, `check_tx_fee` computes the minimum fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code comment itself acknowledges this is a known approximation. The full admission flow in `_process_tx` is:

1. `pre_check` → `check_tx_fee(tx_size)` — size-only gate, cycles unknown
2. `verify_rtx` — executes scripts, returns actual `verified.cycles`
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles
4. `submit_entry` — no post-execution weight-based fee re-check

After `verify_rtx` returns the actual cycle count, there is no second check comparing `fee` against `min_fee_rate.fee(get_transaction_weight(tx_size, verified.cycles))`. The entry is admitted directly. The weight-based fee rate is only used later for pool ordering and eviction via `TxEntry::fee_rate()` and `AncestorsScoreSortKey`.

## Impact Explanation

With `min_fee_rate = 1,000 shannons/KW`, `size = 242 bytes`, `cycles = 70,000,000`:

- Correct weight: `max(242, 70_000_000 × 0.000_170_571_4)` = **11,940**
- Correct minimum fee: **11,940 shannons**
- Actual size-based check: **242 shannons**
- Discount: **~49×**

An attacker pays 242 shannons per transaction, forces 70M cycles of script execution per submission, and fills the pool with entries whose actual fee rate is ~20 shannons/KW — far below the configured minimum. This constitutes a **High** impact: a bad design that allows CKB network congestion with few costs. The pool's total size limit bounds the number of entries but does not prevent continuous resubmission, each time forcing maximum-cycle verification at a fraction of the intended cost.

## Likelihood Explanation

Any unprivileged caller with RPC access can trigger this via `send_transaction`. Crafting a CKB transaction whose lock or type script runs a tight computation loop consuming ~70M cycles while keeping the serialized body small (~242 bytes) is straightforward using CKB-VM. The required fee (242 shannons at default `min_fee_rate`) is publicly documented and trivially computable. No special privileges, key material, or hashpower are required. The attack is repeatable as long as the attacker can pay the small size-based fee per submission.

## Recommendation

After `verify_rtx` returns the actual cycle count, perform a second weight-based fee check before admitting the entry to the pool. In `_process_tx`, between lines 734 and 751:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool_config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

This mirrors how `TxEntry::fee_rate()` and `AncestorsScoreSortKey` already correctly apply `get_transaction_weight` for ordering and eviction.

## Proof of Concept

1. Deploy a CKB lock script that runs a tight computation loop consuming ~70,000,000 cycles.
2. Construct a transaction spending a cell locked by that script, with minimal witnesses and outputs (~242 bytes serialized).
3. Set the transaction fee to 242 shannons (exactly `min_fee_rate.fee(242)` at 1,000 shannons/KW).
4. Submit via `send_transaction` RPC.
5. Observe: the node executes 70M cycles of script verification, `check_tx_fee` passes (242 ≥ 242), and the transaction is admitted. `TxEntry::fee_rate()` will report ~20 shannons/KW — ~50× below the configured minimum.
6. Repeat continuously. Each submission forces 70M cycles of node computation at 242 shannons cost, filling the pool with low-effective-fee-rate entries and displacing legitimate transactions.
