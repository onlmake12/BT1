Audit Report

## Title
Stale Pre-Eviction Stat Snapshot Overwrites Correct Post-Eviction Totals in `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures a snapshot of `total_tx_size` and `total_tx_cycles` as local variables before `check_and_record_ancestors` runs. When that function evicts entries via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, the decrements are applied directly to `self.total_tx_size`/`self.total_tx_cycles` in-place. The stale local snapshot is then unconditionally written back, permanently re-inflating both counters by the aggregate size and cycles of all evicted entries. Repeated exploitation causes the pool to reject valid transactions with `Reject::Full` even when real capacity exists.

## Finding Description

The exact sequence in `add_entry` (lines 200–221):

```rust
// Step 1: snapshot BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//  = self.total_tx_size + entry.size  (stale)

// Step 2: may evict N entries; each calls update_stat_for_remove_tx,
//         which decrements self.total_tx_size / self.total_tx_cycles in-place
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3: stale snapshot overwrites the correctly-decremented values
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` is a pure read that returns `self.total_tx_size + tx_size` and `self.total_tx_cycles + cycles` without modifying `self`. [2](#0-1) 

`check_and_record_ancestors` enters the eviction branch (line 603) when `ancestors_count > max_ancestors_count` but the excess is covered by `cell_ref_parents`. It calls `remove_entry_and_descendants` for each candidate. [3](#0-2) 

`update_stat_for_remove_tx` directly mutates `self.total_tx_size` and `self.total_tx_cycles` in-place. [4](#0-3) 

After `check_and_record_ancestors` returns, the stale local variables (which do not reflect any eviction decrements) are written back unconditionally at lines 218–219, erasing all correct in-place decrements performed during eviction. [5](#0-4) 

## Impact Explanation

`total_tx_size` is the primary guard for the `max_tx_pool_size` limit. When permanently inflated, the pool reports itself as fuller than it actually is, causing subsequent valid transactions to be rejected with `Reject::Full`. `total_tx_cycles` is similarly used for cycle-limit enforcement and is exposed via the `tx_pool_info` RPC. An attacker who repeatedly triggers the eviction path can drive both counters to their limits without the pool actually being full, effectively making the node's mempool non-functional for all unprivileged submitters. If applied to multiple nodes simultaneously, this constitutes **CKB network congestion with low cost**, matching the High impact class.

## Likelihood Explanation

The eviction path is reachable by any unprivileged user who can submit transactions to the RPC. The attacker must construct a chain of ≥125 transactions where each uses the prior transaction's output as a cell dep (not an input), making them `cell_ref_parents`. Submitting a new transaction that references those cells as deps triggers the eviction branch. Each such submission permanently inflates the counters. The attack is repeatable and compounds: after enough iterations, `total_tx_size >= max_tx_pool_size` even though the pool holds far fewer bytes.

## Recommendation

Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so the snapshot reflects the post-eviction state:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals only after all evictions are done
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, eliminate the local-variable pattern entirely and apply the increment directly to `self.total_tx_size`/`self.total_tx_cycles` after all mutations complete.

## Proof of Concept

1. Submit a chain of 125 transactions `T1 → T2 → … → T125` where each `Ti` uses `T(i-1)`'s output as a **cell dep** (not an input), making all 125 appear as `cell_ref_parents`.
2. Submit `T_new` that also references `T1`'s output as a cell dep. Its ancestor count is 126, exceeding `max_ancestors_count = 125`.
3. `check_and_record_ancestors` enters the eviction branch at line 603, evicts `T125` (and descendants), calling `update_stat_for_remove_tx(T125.size, T125.cycles)`, which correctly decrements `self.total_tx_size`.
4. `add_entry` then writes back the stale `total_tx_size` (computed before eviction at line 211), re-inflating it by `T125.size`.
5. Repeat steps 1–4. `total_tx_size` grows without bound relative to actual pool contents.
6. Eventually `total_tx_size >= max_tx_pool_size` and all subsequent `send_transaction` RPC calls are rejected with `Reject::Full`. [6](#0-5)

        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {

                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }
