Audit Report

## Title
Unbounded `pending_delivered` Growth via Attacker-Controlled `from` PeerIds Bypassing `forward_rate_limiter` — (`network/src/protocols/hole_punching/mod.rs`, `network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

An unprivileged remote peer with a single valid P2P connection can flood a victim node with `ConnectionRequest` messages bearing distinct, attacker-generated `from` PeerIds and `to = victim_peer_id`. Because `forward_rate_limiter` is keyed by the attacker-controlled `(from, to, item_id)` tuple, every unique `from` is a fresh key that passes the 1/sec limit. Each such message inserts a new unbounded entry into `pending_delivered`, which is only cleaned up every 5 minutes, enabling heap exhaustion and node crash.

## Finding Description

**Data structures** (`mod.rs` lines 44–46): `pending_delivered` is an unbounded `HashMap<PeerId, PendingDeliveredInfo>`. The `forward_rate_limiter` is keyed by `(PeerId, PeerId, u32)` — i.e., `(from, to, item_type_id)`. [1](#0-0) 

**Outer rate limiter** (`mod.rs` lines 95–107): keyed by `(session_id, item_id)` at 30/sec. This is the only per-session cap and correctly bounds throughput to 30 msgs/sec per attacker session. [2](#0-1) 

**Forward rate limiter bypass** (`connection_request.rs` lines 132–143): the check uses `content.from.clone()` as part of the key. Since `from` is a raw attacker-supplied field in the message payload, the attacker generates a distinct random `PeerId` per message. Each distinct `from` is a brand-new key — the 1/sec quota applies per key, so N distinct keys allow N msgs/sec total, completely defeating the forward limiter's intent. [3](#0-2) 

**Unbounded insertion** (`connection_request.rs` lines 161–167, 234–237): the only deduplication guard is `pending_delivered.get(&from_peer_id)`, which only blocks re-insertion for the *same* `from`. With a distinct `from` per message, every message inserts a new `(PeerId → (Vec<Multiaddr>, u64))` entry with no size cap. [4](#0-3) [5](#0-4) 

**Cleanup only in `notify()`** (`mod.rs` lines 172–175): entries with age < `TIMEOUT = 5 minutes` are retained. All entries inserted during a 5-minute window survive until the next tick. [6](#0-5) 

**Secondary growth — `forward_rate_limiter` state** (`mod.rs` lines 66–70): `retain_recent()` on `forward_rate_limiter` is only called in `disconnected()`, never in `notify()`. While the attacker holds the connection open, the limiter's `HashMapStateStore` accumulates one entry per unique `(from, to, item_id)` key, also growing unboundedly. [7](#0-6) 

## Impact Explanation

Per session: 30 msgs/sec × 300 sec = **9,000 entries** per 5-minute window. Each entry holds a `PeerId` key (~39 bytes) plus a `Vec<Multiaddr>` of up to `ADDRS_COUNT_LIMIT = 24` addresses (~50 bytes each ≈ 1.2 KB) plus a `u64` timestamp — roughly **~11 MB per session per window**. [8](#0-7) 

CKB nodes accept up to ~125 inbound peers by default. With multiple attacker-controlled sessions, aggregate heap growth reaches **~1.4 GB** before the first cleanup tick, causing OOM and node crash.

This matches the allowed bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

- Requires only a single valid P2P connection — no privilege, no PoW, no key material.
- `from` PeerIds are arbitrary bytes in the message payload; the attacker generates them locally in a tight loop.
- The attacker must supply at least one valid TCP IPv4/IPv6 listen address (trivially satisfied with any real address).
- `send_message_to` back to the attacker's session succeeds since the attacker is connected.
- No banning is triggered: `TooManyRequests` and `Ignore` return codes do not call `should_ban()`. [9](#0-8) 

## Recommendation

1. **Key `forward_rate_limiter` by `(session_id, to, item_id)` instead of `(from, to, item_id)`** — the session is the only attacker-uncontrollable identity. This makes the forward limiter equivalent to the outer limiter and eliminates the bypass.
2. **Cap `pending_delivered` size** with an LRU eviction policy or a hard `HashMap::len()` guard before insertion in `respond_delivered`.
3. **Periodically call `forward_rate_limiter.retain_recent()`** inside `notify()` alongside the existing `pending_delivered` cleanup, to prevent unbounded limiter state growth while a connection is held open.
4. **Reduce `CHECK_INTERVAL`** or add an intermediate size-based eviction trigger to shorten the worst-case accumulation window.

## Proof of Concept

```rust
// Pseudocode unit test
let mut protocol = HolePunching::new(network_state);
let session_id = PeerIndex::new(1);
let victim_peer_id = protocol.network_state.local_peer_id().clone();

for i in 0..9000 {
    let from = PeerId::random(); // distinct each iteration
    let msg = build_connection_request(
        from,
        victim_peer_id.clone(),
        vec![valid_tcp_addr()], // at least one valid TCP addr required
    );
    // outer rate_limiter: (session_id=1, item_id=0) — 30/sec, passes within burst
    // forward_rate_limiter: (from_i, victim, 0) — new key each iteration, always passes
    // respond_delivered: pending_delivered.get(&from_i) → None → inserts unconditionally
    protocol.received(ctx_with_session(session_id), msg).await;
}

// pending_delivered grows to 9000 entries; no cleanup until notify() fires after 5 min
assert_eq!(protocol.pending_delivered.len(), 9000);
```
