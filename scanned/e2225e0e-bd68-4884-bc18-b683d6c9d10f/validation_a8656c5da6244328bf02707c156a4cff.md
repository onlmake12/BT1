Looking at the full code path carefully:

**Step 1: `ConnectionRequest` → `pending_delivered` population**

In `connection_request.rs`, `respond_delivered()` is called when `self_peer_id == &content.to`. The `from` field is taken directly from the message bytes with no check that it matches the actual sender's peer ID: [1](#0-0) 

The `remote_listens` stored are filtered to TCP+IPv4/IPv6 only, but the IP addresses themselves are fully attacker-controlled: [2](#0-1) 

**Step 2: `ConnectionSync` → `try_nat_traversal` trigger**

In `connection_sync.rs`, when `route` is empty and `self_peer_id == content.to`, the handler looks up `pending_delivered` by `content.from` (again, fully attacker-controlled) and spawns NAT traversal tasks: [3](#0-2) 

**Step 3: `try_nat_traversal` behavior**

The function makes repeated TCP `connect()` calls every ~200ms for up to 30 seconds to the attacker-supplied address: [4](#0-3) 

**Rate limiter analysis**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. Since `from` is attacker-controlled and can be any arbitrary `PeerId`, the attacker can use distinct `from` values to bypass the per-key rate limit. The `HOLE_PUNCHING_INTERVAL` (2 min) check in `respond_delivered` is also per-`from_peer_id`: [5](#0-4) 

**Missing authentication invariant**

There is no check anywhere that `content.from` in either `ConnectionRequest` or `ConnectionSync` matches the actual session peer ID. The `peer` field (session ID) is only used to route the `ConnectionRequestDelivered` response back — it is never compared against `content.from`: [6](#0-5)

### Title
Unauthenticated `from` field in hole-punching protocol allows any connected peer to trigger outbound TCP connections to attacker-controlled addresses — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
Any directly connected P2P peer can cause the victim node to make repeated outbound TCP connections to arbitrary attacker-controlled IP:port combinations by sending two crafted hole-punching messages with a spoofed `from` peer ID.

### Finding Description
The hole-punching protocol's `ConnectionRequest` and `ConnectionSync` handlers accept a `from` field from message content without verifying it matches the actual sender's peer ID. An attacker who is a connected P2P peer can:

1. Send a `ConnectionRequest` with `from=<arbitrary_peer_id>`, `to=<victim_peer_id>`, `listen_addrs=[attacker_ip:port]`. Since the victim is the `to` target, `respond_delivered()` stores `pending_delivered[arbitrary_peer_id] = ([attacker_ip:port], now)` with no origin authentication.
2. Send a `ConnectionSync` with `from=<same_arbitrary_peer_id>`, `to=<victim_peer_id>`, `route=[]`. The handler finds the entry in `pending_delivered` and spawns `try_nat_traversal(bind_addr, attacker_ip:port)`.

`try_nat_traversal` then makes repeated TCP `connect()` calls (every ~200ms, up to 30 seconds) to the attacker-supplied address. By cycling through distinct `from` peer IDs, the attacker bypasses both the `forward_rate_limiter` (keyed by `(from, to, msg_item_id)`) and the `HOLE_PUNCHING_INTERVAL` deduplication guard.

### Impact Explanation
- **Port scanning / SSRF**: The victim node makes outbound TCP SYNs to arbitrary IP:port, including internal network addresses (RFC-1918), localhost services, or third-party hosts.
- **Connection exhaustion**: Each `try_nat_traversal` task runs for up to 30 seconds, creating a new TCP socket every ~200ms (~150 sockets per task). With many distinct spoofed `from` IDs, the attacker can exhaust file descriptors or ephemeral ports on the victim.
- **Third-party harassment**: The victim's IP appears as the source of port-scan traffic against third-party hosts.

### Likelihood Explanation
The attacker only needs a single standard P2P connection to the victim — no special privileges, no PoW, no key material. The two required messages are structurally valid and pass all existing format checks. The rate limiter is trivially bypassed by varying the `from` peer ID bytes.

### Recommendation
1. **Authenticate `from`**: In `ConnectionRequestProcess`, verify that `content.from` equals the peer ID of the actual session (`context.session.id` resolved to a `PeerId` via the peer registry). Reject messages where `from` does not match the sender.
2. **Bound concurrent NAT traversal tasks**: Maintain a counter of in-flight `try_nat_traversal` tasks and refuse to spawn new ones beyond a small limit (e.g., 8).
3. **Validate addresses against a blocklist**: Reject RFC-1918, loopback, and link-local addresses in `listen_addrs` before storing them in `pending_delivered`.

### Proof of Concept
```
1. Attacker connects to victim via normal P2P handshake.
2. Attacker sends HolePunchingMessage::ConnectionRequest {
       from: <random_peer_id_bytes>,
       to: <victim_peer_id>,
       listen_addrs: [/ip4/192.168.1.1/tcp/22],  // internal SSH
       route: [],
       max_hops: 6
   }
   → victim stores pending_delivered[random_peer_id] = ([/ip4/192.168.1.1/tcp/22], now)
3. Attacker sends HolePunchingMessage::ConnectionSync {
       from: <same_random_peer_id_bytes>,
       to: <victim_peer_id>,
       route: []
   }
   → victim spawns try_nat_traversal(bind_addr, /ip4/192.168.1.1/tcp/22)
   → victim makes ~150 TCP SYN packets to 192.168.1.1:22 over 30 seconds
4. Repeat step 2–3 with fresh random_peer_id values to bypass rate limiting.
```
Assert: tcpdump on attacker-controlled host (or 192.168.1.1) shows TCP SYNs originating from the victim node's IP.

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
