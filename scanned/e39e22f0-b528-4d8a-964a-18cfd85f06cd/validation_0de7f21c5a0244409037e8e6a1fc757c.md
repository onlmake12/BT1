### Title
Missing Chain Domain Separation in Alert Signature Verification Enables Cross-Network Alert Replay — (`util/network-alert/src/verifier.rs`)

### Summary
The `RawAlert` message hash used for multi-signature verification contains no chain or network identifier. Because the same developer keys are configured for both CKB mainnet and testnet, a valid alert signed for one network is cryptographically valid on any other CKB network. Any unprivileged P2P peer can relay a testnet alert verbatim to mainnet nodes and have it accepted, broadcast, and displayed.

### Finding Description

**Root cause — no domain separation in `calc_alert_hash`**

`calc_alert_hash` is implemented as a plain `blake2b_256` over the raw Molecule-encoded bytes of `RawAlert`:

```
impl<'r> packed::RawAlertReader<'r> {
    pub fn calc_alert_hash(&self) -> packed::Byte32 {
        self.calc_hash()   // → blake2b_256(self.as_slice())
    }
}
``` [1](#0-0) 

The `RawAlert` schema contains only:

```
table RawAlert {
    notice_until:   Uint64,
    id:             Uint32,
    cancel:         Uint32,
    priority:       Uint32,
    message:        Bytes,
    min_version:    BytesOpt,
    max_version:    BytesOpt,
}
``` [2](#0-1) 

There is no `chain_id`, genesis hash, or network name in the signed payload. The `CalcHash` blanket implementation hashes only `self.as_slice()` with no domain prefix: [3](#0-2) 

**Verifier uses this hash directly**

`verify_signatures` derives the ECDSA message from `calc_alert_hash()` and checks it against the hardcoded developer public keys: [4](#0-3) 

**Same keys on all networks**

The default `alert_signature.toml` lists 4 public keys with no per-network distinction: [5](#0-4) 

**Relay path — no expiry check on P2P receive**

`alert_relayer.received()` accepts an alert from any peer, verifies signatures, and immediately broadcasts it to all connected peers. There is no `notice_until` check at this layer — only the RPC `send_alert` path checks expiry: [6](#0-5) 

**Exploit flow**

1. Attacker observes a legitimately signed alert on CKB testnet (e.g., "upgrade to v0.105.x").
2. Attacker connects to CKB mainnet as a normal P2P peer.
3. Attacker sends the identical raw `Alert` bytes to mainnet nodes over the Alert protocol.
4. Each mainnet node calls `verify_signatures` → `calc_alert_hash` produces the same 32-byte digest (no chain binding) → the same developer signatures pass → the alert is accepted.
5. The alert is broadcast to all mainnet peers and displayed to users.

The reverse (mainnet alert replayed on testnet) is equally trivial.

### Impact Explanation
The alert system is the sole mechanism by which the Nervos Foundation communicates critical bug warnings to all nodes. A replayed cross-network alert causes mainnet nodes to display a false or out-of-context emergency message (e.g., a testnet-only upgrade notice). Users may perform unnecessary emergency upgrades, or a carefully crafted replay could suppress or drown out a genuine mainnet alert by exhausting the alert ID space. The integrity of the only out-of-band emergency channel is broken.

### Likelihood Explanation
Any P2P peer — no keys, no stake, no privilege required — can execute this by simply forwarding bytes observed on one network to nodes on another. Both mainnet and testnet are public and permissionless. The attacker needs only to be connected to both networks simultaneously, which is trivially achievable.

### Recommendation
Include a network domain separator in the signed payload. The simplest approach is to add the genesis block hash (already available as `consensus.genesis_hash`) to the `RawAlert` structure, or to prepend it during hash computation in `calc_alert_hash`, analogous to EIP-712's `domainSeparator`. This ensures a signature produced for mainnet is cryptographically invalid on testnet and vice versa.

### Proof of Concept

```
# Step 1: capture a valid testnet alert (e.g., via RPC get_blockchain_info → alerts field)
TESTNET_ALERT_HEX="<raw hex of packed::Alert from testnet>"

# Step 2: connect to a mainnet node and send the Alert protocol message
# (using any CKB P2P client or netcat with the Tentacle framing)
send_p2p_message --network mainnet \
                 --protocol Alert \
                 --data $TESTNET_ALERT_HEX

# Result: mainnet node logs "ReceiveD alert <id> from peer <x>",
# passes verify_signatures (same hash, same keys, same sig),
# broadcasts to all its peers, and adds to notifier.
# All connected mainnet nodes display the testnet alert.
```

The hash identity is deterministic: `blake2b_256(raw_alert_bytes)` is identical on both networks for the same `RawAlert` content, so no cryptographic work is required. [7](#0-6) [8](#0-7)

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }
