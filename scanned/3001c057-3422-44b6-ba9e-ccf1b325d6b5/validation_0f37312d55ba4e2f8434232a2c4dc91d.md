### Title
`total_tx_size` and `total_tx_cycles` Over-Counted When Cell-Ref-Parent Evictions Occur During `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new `total_tx_size` and `total_tx_cycles` values are pre-computed **before** `check_and_record_ancestors` runs. When that function evicts transactions (via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`), it correctly decrements `self.total_tx_size` and `self.total_tx_cycles`. However, at the end of `add_entry`, both counters are unconditionally overwritten with the stale pre-eviction values, permanently over-counting pool occupancy by the sum of all evicted transaction sizes and cycles.

---

### Finding Description

In `pool_map.rs`, `add_entry` follows this sequence:

```rust
// Step 1: pre-compute new totals (snapshot of current state + entry)
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Step 2: may evict transactions, calling update_stat_for_remove_tx
//         which MODIFIES self.total_tx_size and self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Step 3: OVERWRITES the post-eviction values with the stale pre-eviction snapshot
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

The eviction path inside `check_and_record_ancestors` is reached when a new transaction's ancestor count exceeds `max_ancestors_count`, but the excess ancestors are all *cell-ref parents* (in-pool transactions that reference the same cell as a dep as the new transaction's inputs). The code evicts those cell-ref parents to bring the ancestor count within limits: [2](#0-1) 

Each eviction calls `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`: [3](#0-2) [4](#0-3) 

But those decrements are immediately discarded when `add_entry` writes the stale snapshot back. The correct post-eviction values should be:

```
total_tx_size  = original + entry.size  − Σ(evicted_tx.size)
total_tx_cycles = original + entry.cycles − Σ(evicted_tx.cycles)
```

What is actually written:

```
total_tx_size  = original + entry.size   ← evictions ignored
total_tx_cycles = original + entry.cycles ← evictions ignored
```

The over-count equals the aggregate size and cycles of all evicted transactions and **persists indefinitely**: `recompute_total_stat` is only invoked on underflow inside `update_stat_for_remove_tx`, which never triggers in an over-count scenario. [5](#0-4) 

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further transactions: [6](#0-5) 

And it is the basis for the overflow check that rejects incoming transactions: [7](#0-6) 

An inflated `total_tx_size` causes the pool to believe it is fuller than it actually is. Consequences:

1. **Spurious evictions**: `limit_size` will evict legitimate pending/proposed transactions that would otherwise fit, degrading miner revenue and user experience.
2. **Spurious rejections**: `updated_stat_for_add_tx` will reject new transactions with `Reject::Full` even when actual pool occupancy is well below `max_tx_pool_size`.
3. **Incorrect RPC reporting**: `total_tx_size` and `total_tx_cycles` surfaced via `tx_pool_info` are wrong, misleading operators and fee-estimation callers. [8](#0-7) 

---

### Likelihood Explanation

The trigger condition is reachable by any unprivileged RPC/P2P transaction submitter:

1. An attacker publishes a live cell `C` on-chain.
2. The attacker submits `N > max_ancestors_count` transactions `T1…TN` to the pool, each referencing `C` as a `cell_dep`. These become cell-ref parents for any future transaction that spends `C`.
3. The attacker submits transaction `TX` that spends `C` as an input. `get_tx_ancenstors` discovers `T1…TN` as `cell_ref_parents` and `ancestors`.
4. Because `ancestors_count > max_ancestors_count` but `ancestors_count − |cell_ref_parents| ≤ max_ancestors_count`, the eviction branch fires, removing some `Ti` from the pool.
5. `add_entry` completes, writing the stale totals and permanently inflating `total_tx_size`/`total_tx_cycles`. [9](#0-8) 

No privileged access, no majority hash power, and no Sybil attack is required. The default `max_ancestors_count` is 125; an attacker needs only slightly more than 125 cell-dep transactions referencing the same live cell.

---

### Recommendation

Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` completes (so it reads the already-adjusted `self.total_tx_size`/`self.total_tx_cycles`), or accumulate the evicted sizes/cycles from the returned `evicts` set and subtract them from the pre-computed totals before writing:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Now compute totals on the post-eviction state:
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// ... rest of add_entry ...
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

### Proof of Concept

```
Setup:
  max_ancestors_count = 125
  pool is empty, total_tx_size = 0

Step 1: Submit 130 transactions T1…T130, each with cell_dep = C (on-chain cell).
        Each Ti has size = 1000 bytes.
        pool: 130 entries, total_tx_size = 130_000

Step 2: Submit TX that spends C as an input.
        get_tx_ancenstors: ancestors = {T1…T130}, cell_ref_parents = {T1…T130}
        ancestors_count = 131 > 125
        131 − 130 = 1 ≤ 125  → eviction branch fires

        updated_stat_for_add_tx called:
          total_tx_size  (snapshot) = 130_000 + TX.size
          total_tx_cycles (snapshot) = ... + TX.cycles

        check_and_record_ancestors evicts 6 Ti entries:
          update_stat_for_remove_tx called 6×:
          self.total_tx_size  = 130_000 − 6_000 = 124_000
          self.total_tx_cycles adjusted accordingly

        add_entry writes stale snapshot:
          self.total_tx_size  = 130_000 + TX.size   ← 6_000 bytes phantom
          self.total_tx_cycles = ...                 ← phantom cycles

Result:
  Actual pool occupancy ≈ 124_000 + TX.size bytes
  Reported total_tx_size = 130_000 + TX.size bytes  (6_000 bytes phantom)

  If max_tx_pool_size = 130_500, limit_size now fires and evicts legitimate
  transactions even though the pool has ~6 KB of free space.
  Repeating the attack accumulates phantom bytes until the pool permanently
  rejects all new submissions.
```

                cell_ref_parents.extend(deps.iter().cloned());
                parents.extend(deps.iter().cloned());

            let id = ProposalShortId::from_tx_hash(&input_pt.tx_hash());
            if self.links.inner.contains_key(&id) {
                parents.insert(id);
            }

        for cell_dep in entry.cell_deps() {
            let dep_pt = cell_dep.out_point();
            let id = ProposalShortId::from_tx_hash(&dep_pt.tx_hash());
            if self.links.inner.contains_key(&id) {
                parents.insert(id);
            }
        }

        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        (ancestors, parents, cell_ref_parents)

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
