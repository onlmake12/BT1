Audit Report

## Title
Unverified `from` Peer Identity in Hole-Punching `ConnectionRequest` Enables NAT-Traversal Hijacking and Rate-Limiter Bypass — (File: network/src/protocols/hole_punching/component/connection_request.rs)

## Summary
`ConnectionRequestProcess::execute()` decodes the `from` field exclusively from the attacker-controlled message payload and never cross-checks it against the actual session identity (`self.peer`). This attacker-controlled `from` is used as the key for both the `forward_rate_limiter` and the `pending_delivered` cache. An attacker can poison `pending_delivered[victim_peer_id]` with their own listen addresses and bypass the forward rate limiter by cycling through arbitrary `from` values, enabling NAT-traversal hijacking and unbounded gossip amplification.

## Finding Description

**Root cause — unverified `from` field:**

In `connection_request.rs` lines 36–38, `from` is decoded entirely from the raw message bytes with no comparison against the actual session's `PeerIndex`/`PeerId`:

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
    StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
})?;
```

**Attack path 1 — `pending_delivered` cache poisoning and NAT-traversal hijacking:**

When `self_peer_id == content.to` (lines 145–146 of `connection_request.rs`), `respond_delivered(content.from, ...)` is called. Inside `respond_delivered` (lines 234–237), the attacker-controlled `from_peer_id` is inserted as the cache key:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

The only guard (lines 161–166) is a 2-minute deduplication check on the same `from_peer_id` — it does not prevent the initial insertion with a fake identity.

Later, `ConnectionSyncProcess::execute()` (lines 111–115 of `connection_sync.rs`) looks up the cache using the equally unverified `content.from` from the sync message:

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```

The attacker can then send a `ConnectionSync` with `from = peer_id_of_B`, `to = V.peer_id`, and an empty route. `V` finds the poisoned entry and calls `try_nat_traversal(ttl, [attacker_tcp_addr])`, connecting to the attacker instead of the legitimate peer B.

**Attack path 2 — forward rate-limiter bypass:**

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` (lines 132–143 of `connection_request.rs`). The outer `rate_limiter` (keyed by `(session_id, msg.item_id())`, capped at 30/s per session) does not prevent this: 30 distinct fake `from` values per second each produce a unique key, bypassing the per-`(from, to)` forward limit entirely. Each bypassed message is gossip-broadcast to `sqrt(total_peers)` nodes for up to `MAX_HOPS = 6` hops, producing multiplicative amplification across the network.

**Attack path 3 — `pending_delivered` memory exhaustion:**

`pending_delivered` is an unbounded `HashMap` (declared in `mod.rs` line 44) pruned only every 5 minutes (`TIMEOUT = 5 * 60 * 1000 ms`, `CHECK_INTERVAL = 5 * 60s`). At 30 messages/second, an attacker can insert up to 9,000 entries per eviction window, each holding a `PeerId` and up to 24 `Multiaddr` values.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The forward rate-limiter bypass is the primary network-level impact. A single connected peer sending 30 `ConnectionRequest` messages/second with distinct fake `from` values bypasses the `forward_rate_limiter` at every relay node. Each message is broadcast to `sqrt(N)` peers and forwarded up to 6 hops. With N=100 peers, this yields up to `30 × 10^6` message deliveries per second at the deepest hop, saturating the gossip layer. The `route` deduplication loop-prevention check (lines 128–130) and `max_hops` bound limit but do not eliminate this amplification. The NAT-traversal hijacking additionally causes victim nodes to spawn 30-second TCP connection tasks to attacker-controlled endpoints, wasting file descriptors and CPU.

## Likelihood Explanation

Any peer connected to a CKB node over the HolePunching protocol can trigger this. No special privilege, key material, or hashpower is required. The attacker only needs one active session. The outer `rate_limiter` (30 msg/s per session) is the only throttle, and it is insufficient: 30 distinct fake `from` values per second is enough to sustain both the cache poisoning and the rate-limiter bypass indefinitely. The attack is repeatable and requires no victim interaction.

## Recommendation

In `ConnectionRequestProcess::execute()`, resolve the actual `PeerId` of the sending session from the peer registry using `self.peer` (the `PeerIndex`) before accepting `content.from`. Assert that the resolved `PeerId` equals `content.from`; if they differ, return an error status and optionally ban the session. Apply the same check in `ConnectionSyncProcess::execute()` for `content.from`. Additionally, cap the size of `pending_delivered` (e.g., with an LRU eviction policy) to bound memory consumption independent of the rate limiter.

## Proof of Concept

**Cache poisoning + NAT hijacking:**
1. Attacker peer `A` connects to victim node `V` over the HolePunching protocol.
2. `A` sends a `ConnectionRequest` with `from = peer_id_of_B` (any known peer), `to = V.peer_id`, `listen_addrs = [A_tcp_addr]`, `max_hops = 6`, empty `route`.
3. `V` receives it: `self_peer_id == content.to`, so `respond_delivered(peer_id_of_B, ...)` is called. `pending_delivered[peer_id_of_B] = ([A_tcp_addr], now)` is inserted.
4. `A` immediately sends a `ConnectionSync` with `from = peer_id_of_B`, `to = V.peer_id`, empty `route`.
5. `V` receives it: `self_peer_id == content.to`, route is empty, looks up `pending_delivered[peer_id_of_B]`, finds `[A_tcp_addr]`, spawns `try_nat_traversal(ttl, [A_tcp_addr])` — connecting to `A` instead of `B`.

**Forward rate-limiter bypass:**
1. `A` connects to `V`.
2. `A` sends 30 `ConnectionRequest` messages/second, each with a distinct randomly-generated `from` PeerId, `to = any_peer_id`, valid `listen_addrs`, `max_hops = 6`.
3. Each message passes the outer `rate_limiter` (30/s per session) and the `forward_rate_limiter` (unique key per message), and is broadcast to `sqrt(N)` peers for up to 6 hops, amplifying traffic across the gossip network.
