Audit Report

## Title
Incomplete `ScriptHashType` Filtering in `ScriptHashTypeVerifier` — Output Type Scripts Not Validated Against Enabled Set - (File: verification/src/transaction_verifier.rs)

## Summary
`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the output's optional **type** script. An unprivileged attacker can craft transactions whose outputs carry type scripts with unactivated `ScriptHashType` values (e.g., `Data3` = 6) that pass this pre-execution gate, forcing the node to enter the full script-execution pipeline before the error is caught, enabling a cheap resource-exhaustion vector.

## Finding Description
`util/constant/src/consensus.rs` defines the permitted set as `{0, 1, 2, 4}`: [1](#0-0) 

`ScriptHashTypeVerifier::verify()` at `verification/src/transaction_verifier.rs` lines 796–814 loops over outputs and checks only `output.lock().hash_type()`: [2](#0-1) 

There is no call to `output.type_()` anywhere in this method. A grep across all of `verification/src/transaction_verifier.rs` for `type_\.to_opt` or `output\.type_` returns zero matches inside `ScriptHashTypeVerifier`. The downstream guard that does catch unactivated hash types lives in `script/src/types.rs`: [3](#0-2) 

This guard fires inside `extract_script_and_dep_index` during the script-execution phase — after the pre-execution pipeline has already accepted the transaction. The attacker therefore forces the node to traverse the full verification pipeline (transaction parsing, structural checks, script-group construction, dep resolution) before the error is raised, with zero mining cost per attempt.

## Impact Explanation
Maps to **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. An unprivileged RPC caller can submit a continuous stream of structurally valid transactions (valid lock scripts, valid capacity, valid molecule encoding) that each pass `ScriptHashTypeVerifier` and proceed into the script-execution pipeline before being rejected. Because no mining power or key material is required, the attacker can sustain this at negligible cost, causing repeated unnecessary work on every receiving node.

## Likelihood Explanation
Any unprivileged user with access to `send_transaction` can trigger this. Constructing a `packed::Script` with `hash_type = 6` (`Data3`) requires a single byte change. The `ScriptHashType` enum exposes `Data3` through `Data127` as valid Rust values via the `seq!` macro expansion. No special privileges, majority hashpower, or victim interaction is needed. The attack is trivially repeatable.

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script immediately after the lock-script check:

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

Add a corresponding test analogous to the existing `test_not_enabled_hash_type_output_lock`: [4](#0-3) 

## Proof of Concept
The existing test `test_not_enabled_hash_type_output_lock` confirms the lock-script path is guarded. No analogous test exists for type scripts. The following unit test demonstrates the gap:

```rust
#[test]
pub fn test_not_enabled_hash_type_output_type() {
    let malicious_type_script = Script::default()
        .as_builder()
        .hash_type(ScriptHashType::Data3) // value = 6, not in ENABLED_SCRIPT_HASH_TYPE
        .build();
    let transaction = TransactionBuilder::default()
        .output(
            CellOutput::new_builder()
                .lock(Script::default()) // valid lock (Data = 0)
                .type_(Some(malicious_type_script).pack())
                .build(),
        )
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);
    // Currently passes — gap confirmed
    assert!(verifier.verify().is_ok());
}
```
