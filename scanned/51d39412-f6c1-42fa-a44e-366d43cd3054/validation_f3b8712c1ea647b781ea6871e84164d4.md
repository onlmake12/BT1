### Title
Unbounded CPU Exhaustion via `estimate_cycles` RPC — Script Execution Without Cycle-Budget Compensation or Rate Limiting — (`rpc/src/module/chain.rs`)

---

### Summary

`CyclesEstimator::run`, the implementation backing both the `estimate_cycles` (Chain module) and the deprecated `dry_run_transaction` (Experiment module) RPC methods, executes attacker-supplied scripts up to `consensus.max_block_cycles` with no fee requirement, no rate limiting, and no authentication. Any unprivileged RPC caller can submit a transaction whose lock/type script runs for the full block-cycle budget, forcing the node to expend maximum CPU resources per call with zero compensation — a direct structural analog to the gas-griefing pattern in the reference report.

---

### Finding Description

`CyclesEstimator::run` in `rpc/src/module/chain.rs` (lines 2375–2405) resolves the caller-supplied transaction and immediately invokes `ScriptVerifier::verify(max_block_cycles)`:

```rust
pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
    let snapshot = self.shared.cloned_snapshot();
    let consensus = snapshot.cloned_consensus();
    match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
        Ok(resolved) => {
            let max_cycles = consensus.max_block_cycles;   // ← full block budget
            ...
            match ScriptVerifier::new(Arc::new(resolved), ...).verify(max_cycles) {
                Ok(cycles) => Ok(EstimateCycles { cycles: cycles.into() }),
                Err(err)   => Err(RPCError::custom_with_error(...)),
            }
        }
        ...
    }
}
``` [1](#0-0) 

The `CellProvider` implementation for `CyclesEstimator` deliberately treats every referenced cell as live regardless of its actual on-chain state:

```rust
fn cell(&self, out_point: &packed::OutPoint, eager_load: bool) -> CellStatus {
    snapshot.get_cell(out_point)
        .map(|cell_meta| CellStatus::live_cell(cell_meta))  // live regardless of dead/spent
        .unwrap_or(CellStatus::Unknown)
}
``` [2](#0-1) 

This means an attacker does not need to own or control any live cells. They can reference any historical cell (including spent ones) as a `cell_dep` to load a maximally expensive script binary, then craft a lock script that loops for exactly `max_block_cycles` cycles. The RPC documentation explicitly confirms no validity check is performed:

> "This method will not check the transaction validity, but only run the lock script and type script and then return the execution cycles." [3](#0-2) 

The `ExperimentRpcImpl::dry_run_transaction` implementation delegates to the same `CyclesEstimator::run` path:

```rust
fn dry_run_transaction(&self, tx: Transaction) -> Result<EstimateCycles> {
    let tx: packed::Transaction = tx.into();
    CyclesEstimator::new(&self.shared).run(tx)
}
``` [4](#0-3) 

The normal tx-pool path does enforce a fee check (`check_tx_fee`) in `pre_check` before script execution, and caps execution at `max_tx_verify_cycles`: [5](#0-4) [6](#0-5) 

`estimate_cycles` bypasses both of these guards entirely.

---

### Impact Explanation

`consensus.max_block_cycles` is the maximum cycle budget for an entire block. A single `estimate_cycles` call with a maximally expensive script forces the node to execute that full budget synchronously on the RPC handler thread. Because there is no rate limiting, a single attacker sending a stream of such requests can saturate the node's CPU, delaying or blocking block processing, tx-pool operations, and all other RPC responses. The node bears the full computational cost with no on-chain fee collected and no mechanism to attribute or penalize the caller.

---

### Likelihood Explanation

The `estimate_cycles` method is part of the default `Chain` RPC module, enabled in standard node configurations. The RPC port is typically exposed on localhost but is often forwarded or exposed in hosted/cloud deployments. No authentication, API key, or proof-of-work is required. An attacker needs only a single TCP connection and the ability to construct a transaction referencing any existing cell containing a high-cycle script (e.g., the standard secp256k1 lock, which is a permanent on-chain cell). The attack is trivially scriptable and requires no privileged access.

---

### Recommendation

1. **Short term:** Introduce a configurable per-call cycle cap for `estimate_cycles` (e.g., `max_tx_verify_cycles` rather than `max_block_cycles`) and add RPC-level rate limiting (requests per second per IP or per connection).
2. **Long term:** Require callers to supply a declared cycle limit that is enforced as the VM budget, so callers self-limit their own resource consumption. Consider moving `estimate_cycles` behind an opt-in configuration flag so node operators can disable it on public-facing nodes.

---

### Proof of Concept

1. Deploy or reference any existing on-chain cell containing a RISC-V binary that loops for `N` cycles (e.g., a script that spins in a tight loop; `N` can be set to `max_block_cycles`).
2. Construct a `Transaction` JSON with that cell as a `cell_dep` and a lock script pointing to it.
3. Call `estimate_cycles` in a tight loop:
   ```bash
   while true; do
     curl -s -X POST http://localhost:8114 \
       -H 'Content-Type: application/json' \
       -d '{"id":1,"jsonrpc":"2.0","method":"estimate_cycles","params":[<TX_JSON>]}'
   done
   ```
4. Observe node CPU pinned at 100%; block-template generation and peer sync stall.

The `CyclesEstimator` will accept the transaction even if the referenced input cell is spent, because its `CellProvider` returns `CellStatus::live_cell` unconditionally for any cell found in the snapshot, eliminating the need for the attacker to control any live UTXO. [7](#0-6) [8](#0-7)

        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {

                        cycles: cycles.into(),

                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
                RPCError::TransactionFailedToResolve,
                err,
            )),
        }

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
    }

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
