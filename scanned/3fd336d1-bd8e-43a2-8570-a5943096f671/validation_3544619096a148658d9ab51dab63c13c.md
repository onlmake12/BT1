### Title
PostgreSQL-only wrong-variable copy-paste in `script_exists_in_output` causes premature script deletion during rollback, corrupting indexer state — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary
A copy-paste error at line 252 of `remove.rs` reads `row_lock` instead of `row_type` in the final `match` of `script_exists_in_output`. On PostgreSQL (where `EXISTS` returns `BOOLEAN`), this causes the function to return the lock_script existence result as the answer for type_script existence. Scripts used only as `type_script` are incorrectly reported as absent and deleted, even when still referenced by surviving outputs in earlier blocks.

### Finding Description
`script_exists_in_output` performs two sequential `SELECT EXISTS` queries: one for `lock_script_id` (result: `row_lock`) and one for `type_script_id` (result: `row_type`). The final `match` at line 252 mistakenly re-reads `row_lock` instead of `row_type`. On PostgreSQL, `try_get::<bool, _>(0)` succeeds on `row_lock`, so the function returns the lock_script result as if it were the type_script result. On SQLite, `try_get::<bool, _>(0)` fails on `row_lock` (BIGINT type), so the `Err` branch correctly falls through to `row_type`. The bug is therefore PostgreSQL-exclusive.

During `rollback_block`, after the tip block's outputs are deleted, `script_exists_in_output` is called for each script referenced by those outputs. For a script S used only as `type_script` in a surviving earlier block's output, the function returns `false` (because S is not a `lock_script` anywhere), and S is added to `script_id_list_to_remove` and deleted. The surviving output's `type_script_id` now points to a non-existent script row.

### Impact Explanation
The `output` table has no foreign key constraint on `type_script_id`, so the deletion succeeds silently. Subsequent `get_cells`, `get_cells_capacity`, and `get_transactions` RPC calls that JOIN `output` with `script` on `type_script_id = script.id` return empty or incomplete results for affected outputs. This is silent, persistent data corruption in the indexer — not a process crash, but a functional failure of the indexer RPC surface for any script that is exclusively a type_script and shared across blocks.

### Likelihood Explanation
Any chain reorganization (reorg) on a PostgreSQL-backed rich-indexer node triggers this path. Reorgs are a normal part of CKB network operation and require no attacker. The precondition (a script used as `type_script` in outputs across multiple blocks) is common in practice (e.g., UDT type scripts, DAO scripts). The bug is deterministic and reproducible.

### Recommendation
At line 252, replace `row_lock.try_get::<bool, _>(0)` with `row_type.try_get::<bool, _>(0)`:

```rust
// line 252 — was: match row_lock.try_get::<bool, _>(0) {
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Alternatively, combine both checks into a single SQL query: `WHERE lock_script_id = $1 OR type_script_id = $1`.

### Proof of Concept
1. Start a CKB node with the rich-indexer configured to use PostgreSQL.
2. Submit two blocks: Block N with output O_prev (type_script = S, lock_script = L1), Block N+1 with output O_tip (type_script = S, lock_script = L2). S is never used as a lock_script.
3. Trigger a rollback of Block N+1 (via a reorg or direct indexer rollback call).
4. Query `SELECT * FROM script WHERE id = <S_id>` — the row is gone.
5. Query `SELECT * FROM output WHERE id = <O_prev_id>` — the row exists with `type_script_id = <S_id>`.
6. Call `get_cells` with `script_type: "type"` and script = S — returns empty results despite O_prev existing.
7. Repeat with SQLite backend — O_prev is correctly returned, confirming the PostgreSQL-specific regression.
