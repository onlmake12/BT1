Audit Report

## Title
Peer Store Exhaustion via Crafted Discovery Addresses Causes Permanent `EvictionFailed` — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
An unprivileged remote peer can permanently exhaust the CKB peer store by advertising 16,384 fake addresses distributed as exactly 4 per /16 subnet via the Discovery protocol. Once full, `check_purge` cannot evict any entry because the eviction condition requires strictly more than 4 peers per group, and all fresh addresses are considered connectable. Every subsequent `add_addr` call returns `PeerStoreError::EvictionFailed`, silently preventing the victim from learning about any honest peers.

## Finding Description
`PeerStore::check_purge` has two eviction strategies:

**Strategy 1 — Non-connectable eviction:** `is_connectable` returns `false` only when `attempts_count >= ADDR_MAX_RETRIES` (3) with `last_connected_at_ms == 0`, or when `attempts_count >= ADDR_MAX_FAILURES` (10) after timeout. Fresh addresses inserted via `add_addr` have `attempts_count = 0` and `last_connected_at_ms = 0`, so they always pass `is_connectable` and are never evicted by this pass. [1](#0-0) 

**Strategy 2 — Network-group eviction:** Groups addresses by /16 subnet (first two octets for IPv4), sorts by group size descending, takes the top `len/2` groups, and evicts 2 random peers — but **only from groups with strictly more than 4 members** (`addrs.len() > 4`). [2](#0-1) 

The network group uses only the first two octets: [3](#0-2) 

**Exploit path:** An attacker fills all 16,384 slots (`ADDR_COUNT_LIMIT`) with exactly 4 addresses per /16 subnet across 4,096 distinct subnets (4,096 × 4 = 16,384). Strategy 1 finds no candidates (all fresh). Strategy 2 evaluates 4,096 groups each of size 4; `4 > 4` is `false` for every group, so `candidate_peers` is empty and `Err(PeerStoreError::EvictionFailed)` is returned. [4](#0-3) [5](#0-4) 

The error is silently dropped in `add_new_addrs` with only a `debug!` log, so the victim node permanently stops accepting new peer addresses: [6](#0-5) 

## Impact Explanation
The victim node's peer store is permanently poisoned. It cannot learn about honest peers via Discovery. When existing connections drop (e.g., after a restart or natural churn), the node attempts outbound connections exclusively to attacker-controlled fake addresses, all of which fail. Over time this isolates the node from the honest network — a prerequisite for an eclipse attack enabling consensus deviation or transaction censorship. This matches the **High** impact class: a vulnerability that could cause CKB network congestion/node isolation with minimal attacker cost.

## Likelihood Explanation
The attacker requires only **one P2P connection** to the victim. The Discovery protocol accepts up to 1,000 nodes per message with up to 3 addresses each (3,000 addresses/message), so approximately 6 messages suffice to fill 16,384 slots. Addresses are never verified before storage — `add_addr` is explicitly for untrusted addresses. No proof-of-work, no privileged role, and no Sybil majority is required. The attack is repeatable and cheap. [7](#0-6) 

## Recommendation
Change the eviction threshold from strict `> 4` to `>= 4` (i.e., `> 3`) at line 378 of `peer_store_impl.rs` so that groups of exactly 4 are also eligible for eviction. Additionally consider:
- Capping the number of addresses accepted per session/connection to limit the rate of store filling
- Requiring at least one successful connection attempt before an address occupies a permanent slot
- Treating repeated `EvictionFailed` results as a misbehavior signal against the advertising peer (triggering disconnect or ban) rather than silently discarding the error

## Proof of Concept
```rust
// Fill peer store with 4096 subnets × 4 addrs = 16384 entries
for a in 0u8..=255 {
    for b in 0u8..=15 { // 256 * 16 = 4096 subnets
        for c in 0u8..4 {
            let addr: Multiaddr = format!("/ip4/{}.{}.1.{}/tcp/8115", a, b, c).parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
        }
    }
}
// Now try to add an honest peer address — returns Err(EvictionFailed)
let honest: Multiaddr = "/ip4/203.0.113.1/tcp/8115".parse().unwrap();
assert!(peer_store.add_addr(honest, Flags::COMPATIBILITY).is_err());
```

The `add_addr_fuzz` method (gated behind `#[cfg(feature = "fuzz")]`) exposes the same path and can be used to write a deterministic unit test or fuzz harness directly against `check_purge`. [8](#0-7)

            return Ok(());
        }
