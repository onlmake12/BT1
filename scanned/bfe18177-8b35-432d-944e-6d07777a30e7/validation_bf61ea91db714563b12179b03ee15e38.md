Audit Report

## Title
Unbounded JSON-RPC Batch Request Causes RPC Server Resource Exhaustion (DoS) — (`File: rpc/src/server.rs`)

## Summary
The CKB JSON-RPC server's batch request handler checks `JSONRPC_BATCH_LIMIT.get()` before processing a batch, but this `OnceLock<usize>` is only initialized when `config.rpc_batch_limit` is explicitly set. The shipped default configuration leaves `rpc_batch_limit` commented out, so the guard is never entered and batch requests are processed with no upper bound on call count. Any local process can send a single HTTP POST with a JSON array of tens of thousands of calls (within the 10 MiB body limit) and force the server to execute them all sequentially, making the RPC interface unresponsive to legitimate callers.

## Finding Description
In `rpc/src/server.rs`, `JSONRPC_BATCH_LIMIT` is declared as `static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new()` (line 34). It is initialized only inside the conditional block at lines 53–55:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

When `rpc_batch_limit` is absent from the config (the default), `OnceLock::get()` returns `None`. In `handle_jsonrpc` at lines 274–282, the guard `if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get() && calls.len() > *batch_size` is therefore never entered. The batch is then dispatched via `.then()` at lines 284–289, which processes each call sequentially — each call must complete before the next begins. The field `rpc_batch_limit` in `util/app-config/src/configs/rpc.rs` (line 44) is typed `Option<usize>` with no `#[serde(default)]` forcing a value, and `resource/ckb.toml` lines 205–208 explicitly comment out the option with a note acknowledging the risk. The only body-size constraint is `max_request_body_size = 10485760` (10 MiB, line 187 of `ckb.toml`). A minimal JSON-RPC call body (~50–80 bytes) allows packing roughly 130,000–200,000 calls into a single request. The `TimeoutLayer` at lines 125–128 of `server.rs` applies a 30-second per-request timeout, which limits individual request duration but does not prevent sustained flooding via multiple concurrent connections.

## Impact Explanation
The RPC server becomes unresponsive to all legitimate callers (miners, wallets, monitoring tools) for the duration of the attack. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**. The RPC endpoint defaults to `127.0.0.1:8114` (localhost only), so the attacker must be a local process. The 30-second `TimeoutLayer` bounds individual request duration, and the sequential `.then()` combinator processes calls lazily without pre-allocating the full batch in memory, making OOM from a single batch speculative rather than demonstrated. The primary concrete impact is RPC unresponsiveness, not node process termination.

## Likelihood Explanation
Any local process — a compromised application, a malicious dependency, or a script run by the node operator — can reach `127.0.0.1:8114` without credentials. The attack requires only a single HTTP POST with a crafted JSON array; no special protocol knowledge beyond JSON-RPC 2.0 batch specification is needed. The default configuration ships with the protection disabled, and the config comment itself acknowledges the risk. The attack is repeatable and can be sustained with concurrent connections to maintain pressure across the 30-second timeout window.

## Recommendation
Set a safe hard-coded default for `JSONRPC_BATCH_LIMIT` that applies even when `rpc_batch_limit` is absent from the configuration:

```rust
const DEFAULT_BATCH_LIMIT: usize = 2000;
let limit = config.rpc_batch_limit.unwrap_or(DEFAULT_BATCH_LIMIT);
let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| limit);
```

This ensures the guard at lines 275–282 of `server.rs` is always active. Additionally, consider processing batch calls with bounded concurrency (e.g., `buffer_unordered`) rather than purely sequentially via `.then()`, so that a single slow call does not block the entire batch response stream.

## Proof of Concept
```python
import json, requests

batch = [
    {"jsonrpc": "2.0", "method": "get_tip_block_number", "params": [], "id": i}
    for i in range(100_000)
]
payload = json.dumps(batch)
# ~5.5 MiB — within the 10 MiB max_request_body_size limit

resp = requests.post(
    "http://127.0.0.1:8114",
    data=payload,
    headers={"Content-Type": "application/json"},
    timeout=120,
)
print(resp.status_code)
```

Run multiple concurrent instances to sustain pressure across the 30-second timeout window. Substituting `get_block_template` amplifies impact by contending on the tx-pool read lock.
