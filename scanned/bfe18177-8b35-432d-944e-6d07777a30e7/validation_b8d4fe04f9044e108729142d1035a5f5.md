Audit Report

## Title
Unbounded CPU Exhaustion via `estimate_cycles` RPC Using Full Block Cycle Budget Without Fee or Rate Limiting — (`rpc/src/module/chain.rs`)

## Summary
`CyclesEstimator::run` in `rpc/src/module/chain.rs` executes attacker-supplied scripts up to `consensus.max_block_cycles` — the full block cycle budget — with no fee requirement, no rate limiting, and no authentication. Its `CellProvider` implementation unconditionally treats any cell found in the snapshot as live, meaning an attacker does not need to control any live UTXO. Any caller with RPC access can repeatedly force the node to expend maximum CPU per call at zero cost, degrading or blocking block processing and all other RPC responses.

## Finding Description
`CyclesEstimator::run` (lines 2375–2405) resolves the caller-supplied transaction and immediately invokes `ScriptVerifier::verify(max_block_cycles)`:

```rust
let max_cycles = consensus.max_block_cycles;   // full block budget, ~3.5B cycles on mainnet
...
ScriptVerifier::new(Arc::new(resolved), ...).verify(max_cycles)
```

The `CellProvider` implementation for `CyclesEstimator` (lines 2347–2362) returns `CellStatus::live_cell` for any cell found in the snapshot, with the comment "treat as live cell, regardless of live or dead":

```rust
snapshot.get_cell(out_point)
    .map(|mut cell_meta| { ... CellStatus::live_cell(cell_meta) })
    .unwrap_or(CellStatus::Unknown)
```

This means an attacker can reference any historical (including spent) cell containing a high-cycle script binary as a `cell_dep`, craft a lock script that loops for `max_block_cycles` cycles, and submit it to `estimate_cycles` repeatedly. The tx-pool path enforces `check_tx_fee` and caps execution at `max_tx_verify_cycles` (70,000,000 by default) in `pre_check`, but `estimate_cycles` bypasses both guards entirely. The RPC server (`rpc/src/server.rs`) has no per-method or per-IP rate limiting — rate limiters exist only for P2P protocols (relayer, hole punching). The deprecated `dry_run_transaction` in `experiment.rs` delegates to the same `CyclesEstimator::run` path.

## Impact Explanation
`consensus.max_block_cycles` is approximately 3,500,000,000 cycles on mainnet — roughly 50× the `max_tx_verify_cycles` cap enforced by the tx-pool. A single `estimate_cycles` call with a maximally expensive script forces the node to execute that full budget synchronously on the RPC handler thread. A stream of such requests from a single attacker saturates the node's CPU thread pool, delaying or blocking block-template generation, peer sync, tx-pool operations, and all other RPC responses. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and **High — Vulnerabilities which could easily crash a CKB node** (render it unresponsive).

## Likelihood Explanation
The `Chain` module (which includes `estimate_cycles`) is enabled in the default node configuration. While the RPC binds to `127.0.0.1:8114` by default, it is routinely forwarded or exposed in hosted, cloud, and infrastructure deployments. No authentication, API key, or proof-of-work is required. The attacker needs only TCP access to the RPC port and the ability to reference any existing on-chain cell (e.g., the permanent secp256k1 system cell) containing a high-cycle script. The attack is trivially scriptable with a single `curl` loop and requires no privileged access or live UTXO ownership.

## Recommendation
1. **Short term:** Cap `estimate_cycles` at `max_tx_verify_cycles` rather than `max_block_cycles`. Add per-connection or per-IP rate limiting at the RPC server layer for computationally expensive methods.
2. **Long term:** Allow callers to supply an explicit cycle limit that is enforced as the VM budget, so callers self-limit resource consumption. Consider placing `estimate_cycles` behind an opt-in configuration flag so operators can disable it on public-facing nodes.

## Proof of Concept
1. Identify any on-chain cell containing a RISC-V binary that loops for `N` cycles (e.g., a script that spins in a tight loop with `N` set to `max_block_cycles`; the permanent secp256k1 system cell is a convenient `cell_dep` to load a script binary).
2. Construct a `Transaction` JSON with that cell as a `cell_dep` and a lock script pointing to the looping binary. The referenced input cell may be spent — `CyclesEstimator`'s `CellProvider` will accept it as live.
3. Call `estimate_cycles` in a tight loop:
   ```bash
   while true; do
     curl -s -X POST http://localhost:8114 \
       -H 'Content-Type: application/json' \
       -d '{"id":1,"jsonrpc":"2.0","method":"estimate_cycles","params":[<TX_JSON>]}'
   done
   ```
4. Observe node CPU pinned at 100%; block-template generation (`get_block_template`) and peer sync stall measurably. The node bears the full `max_block_cycles` computational cost per request with zero on-chain fee collected and no mechanism to attribute or penalize the caller.
