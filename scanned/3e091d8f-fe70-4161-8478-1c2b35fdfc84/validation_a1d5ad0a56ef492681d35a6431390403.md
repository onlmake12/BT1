Audit Report

## Title
`check_tx_fee` Uses Only Serialized Size as Weight, Allowing Cycle-Heavy Transactions to Bypass the Minimum Fee Rate Check - (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size as the weight, while the canonical weight used for pool sorting, eviction, and fee rate statistics is `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. No second fee check using the actual cycles is performed after `verify_rtx` returns in `_process_tx`. This allows an attacker to submit cycle-dominated transactions at a fee up to ~119× below the configured `min_fee_rate`, flooding the tx-pool and forcing expensive script execution on all nodes that relay the transaction.

## Finding Description

**Root cause — `check_tx_fee` (tx-pool/src/util.rs, L42–52):**

The function computes the minimum required fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The comment explicitly acknowledges the limitation but treats it as acceptable. `FeeRate::fee(weight)` computes `fee_rate * weight / 1000`, so the minimum fee is proportional only to byte count. [2](#0-1) 

**No post-verification fee check — `_process_tx` (tx-pool/src/process.rs, L715–753):**

The flow is:
1. `pre_check` → `check_tx_fee` (size-only, L715–717)
2. `verify_rtx` → returns `verified.cycles` (L724–734)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` → submitted directly (L751–753)

There is no second fee check between steps 2 and 3 using `get_transaction_weight(tx_size, verified.cycles)`. [3](#0-2) 

**Canonical weight used everywhere else — `TxEntry::fee_rate()` (tx-pool/src/component/entry.rs, L115–118):**

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

Pool sorting, eviction (`EvictKey`), and fee statistics all use `get_transaction_weight`. [5](#0-4) 

**`get_transaction_weight` definition (util/types/src/core/tx_pool.rs, L298–303):** [6](#0-5) 

**Exploit path:**
1. Attacker crafts a transaction with `tx_size ≈ 100 bytes` and a lock script consuming ~70,000,000 cycles.
2. Sets fee = 100 shannons. Size-based check passes: `1000 × 100 / 1000 = 100 shannons`.
3. `verify_rtx` executes the script (70M cycles of CPU work on the node).
4. No post-verification fee check; `TxEntry` is created and submitted.
5. Effective fee rate by canonical weight: `1000 × 100 / 11940 ≈ 8 shannons/KW` — ~119× below `min_fee_rate`.
6. The transaction is relayed to peers via P2P; each peer repeats the same expensive verification.

**Why existing guards are insufficient:**
- Pool eviction (`limit_size`) uses the canonical weight and will eventually evict these entries, but only after they have already consumed 70M cycles of script execution per node.
- The `declared_cycles` mismatch check (L736–748) only applies when `declared_cycles` is `Some`; for local RPC (`send_transaction`), `declared_cycles` is `None` (L414: `remote.map(|r| r.0)`), so it is skipped entirely. [7](#0-6) [8](#0-7) 

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Each malicious transaction forces every receiving node to execute up to 70,000,000 VM cycles before the transaction can be accepted or evicted. At 100 shannons per transaction (the size-based minimum), an attacker can sustain a continuous CPU-exhaustion attack against the entire relay network at a cost ~119× below what the fee market intends to charge. Secondary effects include fee rate statistic corruption (accepted entries with sub-`min_fee_rate` effective rates pollute fee estimation) and miner revenue loss for any such transaction that is mined.

## Likelihood Explanation

The attack requires only an unprivileged `send_transaction` RPC call. No special keys, majority hashpower, or victim mistakes are needed. A lock script with a tight loop consuming ~70M cycles is straightforward to construct on CKB-VM. The discrepancy is deterministic and reproducible on mainnet with default configuration (`max_tx_verify_cycles = 70,000,000`, `min_fee_rate = 1000 shannons/KW`). The attack is repeatable at negligible cost and scales linearly with the number of nodes in the relay graph.

## Recommendation

After `verify_rtx` returns `verified.cycles` in `_process_tx`, perform a second fee check using the canonical weight before calling `submit_entry`:

```rust
// After verify_rtx returns verified.cycles (tx-pool/src/process.rs, after L734):
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool_config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

This closes the gap completely without requiring any changes to the pre-check stage, since actual cycles are available at that point. The size-only pre-check in `check_tx_fee` can remain as a cheap early filter.

## Proof of Concept

1. Write a CKB lock script that executes a tight loop consuming exactly 69,999,999 cycles (just under `max_tx_verify_cycles`). Compile and deploy it.
2. Craft a transaction spending a cell locked by that script. Keep witness data minimal so `tx_size ≈ 100 bytes`.
3. Set the transaction fee to 100 shannons.
4. Submit via `ckb_sendTransaction` RPC to a node with default config (`min_fee_rate = 1000`).
5. Observe: the transaction is accepted (size check: `1000 × 100 / 1000 = 100 ✓`).
6. Query the pool entry via `get_pool_tx_detail_info`; confirm the effective fee rate by weight is `≈ 8 shannons/KW`, far below `min_fee_rate = 1000`.
7. Repeat in a loop. Each iteration forces ~70M cycles of CKB-VM execution on the node and all relay peers, at a cost of 100 shannons per iteration.
