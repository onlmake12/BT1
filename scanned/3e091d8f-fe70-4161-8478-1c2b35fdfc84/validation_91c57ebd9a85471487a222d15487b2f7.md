Audit Report

## Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` — (`sync/src/types/mod.rs`)

## Summary
`push_peer()` appends a `PeerIndex` to the `peers` Vec unconditionally with no deduplication or size cap. The overflow guard in `add_ask_for_txs` only checks the number of unique hash keys in `unknown_tx_hashes`, not the cumulative size of individual `peers` Vecs. A single remote peer can repeatedly announce the same phantom transaction hashes, causing the `peers` Vec for each entry to grow without bound, ultimately exhausting node memory.

## Finding Description
**Root cause — `push_peer` has no deduplication:** [1](#0-0) 

**Entry point — `TransactionHashesProcess::execute`** filters hashes via `tx_filter`, but phantom hashes (for transactions that never arrive) are never added to `tx_filter` (that only happens in `mark_as_known_txs` upon actual receipt). So the same phantom hashes pass the filter on every repeated announcement: [2](#0-1) 

**Vulnerable insertion path — `add_ask_for_txs`:** For a hash already present in `unknown_tx_hashes`, the code clones the priority, calls `push_peer`, and writes it back — one unbounded append per repetition: [3](#0-2) 

**The overflow guard is bypassed:** It only fires when `unknown_tx_hashes.len()` (the count of unique hash keys) exceeds `MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) or the per-peer threshold. If the attacker uses N < 50,000 phantom hashes, the queue length stays at N and the guard never triggers: [4](#0-3) 

The secondary `peer_unknown_counter` check inside the guard is also unreachable for small N, since the outer `if` condition is never satisfied.

**Rate limiter** allows 30 messages/second per `(PeerIndex, message_type)`: [5](#0-4) 

**`MAX_RELAY_TXS_NUM_PER_BATCH` = 32,767** hashes per message: [6](#0-5) 

At 30 msg/s × 32,767 hashes/msg = ~983,010 `PeerIndex` entries injected per second. Each `PeerIndex` is a `usize` (8 bytes on 64-bit), yielding ~7.5 MB/s of unbounded heap growth. The drain path (`pop_ask_for_txs`) removes at most one peer entry per hash per `RETRY_ASK_TX_TIMEOUT_INCREASE` (30 s): [7](#0-6) 

Net growth rate is effectively the full injection rate.

## Impact Explanation
A single connected peer can exhaust all available RAM on the victim node within minutes (~4.5 GB after 10 minutes at maximum rate), causing an OOM kill. This results in complete denial of service: no block production, no transaction relay, no RPC responses. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
- Requires only a single connected peer with no special privileges, no PoW, no keys.
- The attacker announces phantom tx hashes (hashes for transactions that do not exist). The victim never fetches the actual transactions, so the hashes are never added to `tx_filter` and the attack sustains indefinitely.
- The rate limiter (30 req/s) slows but does not stop the attack; a single peer can exhaust gigabytes of RAM within minutes.
- No victim interaction or mistake required.

## Recommendation
1. **Deduplicate in `push_peer`**: Before appending, check `if !self.peers.contains(&peer_index)`.
2. **Cap `peers` Vec length**: Enforce a hard maximum equal to the number of connected peers or a small constant (e.g., 8).
3. **Alternatively, use `HashSet<PeerIndex>`** instead of `Vec<PeerIndex>` for O(1) automatic deduplication.
4. **Add per-peer announcement deduplication**: Track a per-peer LRU set or bloom filter of recently announced hashes and drop duplicates before calling `add_ask_for_txs`.

## Proof of Concept
1. Connect to a CKB node as a relay peer.
2. Generate 32,767 random 32-byte values as phantom tx hashes (transactions that do not exist on the network).
3. In a loop at ≤30 iterations/second (within the rate limit):
   - Send a `RelayTransactionHashes` message containing all 32,767 hashes.
4. After 1,000 iterations, observe that for each of the 32,767 entries in `unknown_tx_hashes`, `peers.len() == 1000`.
5. Monitor the RSS of the `ckb` process: it grows by ~7.5 MB per second.
6. Continue until OOM kill (expected within ~10 minutes for a node with 4 GB RAM).

The `peers.len()` grows by exactly 1 per repetition per hash because `push_peer` appends unconditionally and the overflow guard never fires (`unknown_tx_hashes.len()` stays at 32,767, below `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000). [8](#0-7)
