Audit Report

## Title
Bypassed `request_limit` Guard via `usize::MAX` Default Enables RPC-Triggered Node OOM Crash — (`File: util/rich-indexer/src/service.rs`)

## Summary
`RichIndexerService` initializes `request_limit` to `usize::MAX` when the operator has not set it. The guard in `get_cells` and `get_transactions` compares a `Uint32` limit value (max `4,294,967,295`) against `usize::MAX` (`18,446,744,073,709,551,615`), making the check permanently false. Any unauthenticated RPC caller can submit `limit = 0xFFFFFFFF` to force unbounded database row fetching and JSON serialization, exhausting node memory and causing an OOM crash. The same default flaw exists in the built-in RocksDB indexer.

## Finding Description
`RichIndexerService::new()` sets `request_limit: config.request_limit.unwrap_or(usize::MAX)` (line 51, `util/rich-indexer/src/service.rs`). The guard in `get_cells` is:

```rust
if limit as usize > self.request_limit { ... }
```

With `limit = 0xFFFFFFFF` (max `Uint32`, value `4,294,967,295`) and `self.request_limit = usize::MAX` (`18,446,744,073,709,551,615`), the condition evaluates to `false` unconditionally. The guard is a no-op. The SQL query is issued with `LIMIT 4294967295`, and `fetch_all()` attempts to collect all matching rows into a `Vec<IndexerCell>` in memory. The identical pattern exists in `get_transactions`. For `get_cells_capacity`, there is no limit parameter at all — it executes an unbounded `SUM(output.capacity)` aggregation with no guard. The configuration comment at lines 286–291 of `resource/ckb.toml` explicitly acknowledges that serde JSON serialization consumes ~10× raw data size and recommends `400` as a safe limit, but leaves it opt-in. The built-in indexer (`util/indexer/src/service.rs` line 98) has the identical `unwrap_or(usize::MAX)` default.

## Impact Explanation
A single RPC call with `limit = 0xFFFFFFFF` against a node with millions of indexed cells causes `fetch_all()` to attempt collecting up to 4.29 billion rows into heap memory, followed by serde JSON serialization at ~10× raw size. This exhausts available RAM and triggers an OOM kill of the node process, taking down all services including consensus-critical ones. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The RPC is reachable by any local or network-accessible caller with no authentication required. The vulnerable default (`usize::MAX`) is the out-of-the-box behavior — operators must explicitly opt in to the safe value. A single request is sufficient. No chain state knowledge is needed; an empty `args` with `prefix` search mode maximizes the result set. The attack is trivially repeatable.

## Recommendation
1. Replace `usize::MAX` with a safe hard default in both indexers:
   ```rust
   // util/rich-indexer/src/service.rs and util/indexer/src/service.rs
   request_limit: config.request_limit.unwrap_or(400),
   ```
2. Add a `request_limit` check to `get_cells_capacity` in `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`, either by enforcing a maximum matching-row count via a subquery or adding a configurable query timeout.
3. Consider making `request_limit` a required config field with no implicit default, forcing operators to make an explicit choice.
