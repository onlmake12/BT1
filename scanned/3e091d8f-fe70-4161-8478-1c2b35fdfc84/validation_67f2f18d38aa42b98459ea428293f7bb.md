Audit Report

## Title
TOCTOU Race in `HeaderMapKernel::get` Causes Sled Entry Loss and Node Panic — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary

`HeaderMapKernel::get` performs a non-atomic check-then-remove sequence: it checks memory under a write lock, releases that lock, then calls `SledBackend::remove` with no synchronization. Two concurrent callers looking up the same sled-resident parent hash can both observe a memory miss and both call `backend.remove`; sled's atomic remove gives `Some` to one and `None` to the other. The losing thread propagates `None` through `get_header_index_view` into `insert_valid_header`, which panics unconditionally, crashing the node.

## Finding Description

**Root cause — `HeaderMapKernel::get`** (`shared/src/types/header_map/kernel_lru.rs`, lines 109–142):

```rust
pub(crate) fn get(&self, hash: &Byte32) -> Option<HeaderIndexView> {
    if let Some(view) = self.memory.get_refresh(hash) {  // write-lock acquired + released
        return Some(view);
    }
    // lock is now dropped; no synchronization below
    if self.backend.is_empty() { return None; }
    if let Some(view) = self.backend.remove(hash) {      // no lock held
        self.memory.insert(view.clone());
        Some(view)
    } else {
        None   // losing thread returns None here
    }
}
```

`MemoryMap::get_refresh` acquires and immediately drops a write lock (`memory.rs` lines 87–93). After the lock is released, any number of concurrent threads can observe the same key absent from memory and proceed to `backend.remove`. Sled's `remove` is atomic at the DB level, so exactly one caller receives `Some`; all others receive `None`. [1](#0-0) [2](#0-1) [3](#0-2) 

**Panic site — `insert_valid_header`** (`sync/src/types/mod.rs`, lines 1101–1103):

```rust
let parent_header_index = self
    .get_header_index_view(&parent_hash, store_first)
    .expect("parent should be verified");
```

`get_header_index_view` (lines 1143–1167) calls `self.shared.header_map().get(hash)` and falls back to the chain store. During IBD, `store_first = false` (because `tip_number < header.number()`), so the header map is queried first. If the race causes `None` to be returned and the parent is not yet committed to the chain store (the normal IBD state), `get_header_index_view` returns `None` and the `.expect` panics. [4](#0-3) [5](#0-4) 

**Reachable from two production P2P paths with no outer lock:**

- `headers_process.rs` line 356: `sync_shared.insert_valid_header(self.peer, self.header)` — the `HEADER_VALID` status check at line 304 is not atomic with this call; two threads processing sibling headers (or the same header from two peers) can both pass the check and both call `insert_valid_header` concurrently.
- `compact_block_process.rs` line 78: `shared.insert_valid_header(self.peer, &header)` — same issue. [6](#0-5) [7](#0-6) 

`HeaderMap::get` at `mod.rs` lines 92–100 adds no additional synchronization; it simply delegates to `self.inner.get(hash)`. [8](#0-7) 

## Impact Explanation

A panic in `insert_valid_header` terminates the node process. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.** The crash is unconditional (`.expect`, not a recoverable error), affects the entire node process, and is triggerable remotely by any unprivileged peer.

## Likelihood Explanation

Preconditions routinely met during IBD:

1. The memory map exceeds `memory_limit`, causing `limit_memory()` (triggered every 5 seconds) to evict entries to sled — the designed behavior during long-chain sync.
2. Two peers (honest or attacker-controlled) send headers whose parent H is sled-resident. This triggers concurrent `insert_valid_header` calls that both look up H.
3. The race window between `memory.get_refresh` returning `None` and `backend.remove` completing is small but non-zero and repeatable under load.

No PoW, no privileged access, and no majority hashpower is required. An attacker needs only two connections to the victim node and knowledge of a header whose parent has been evicted to sled.

## Recommendation

Make the check-and-promote sequence atomic. Options in increasing robustness:

1. **Re-check memory after a failed `backend.remove`**: if `backend.remove` returns `None`, call `memory.get_refresh` again before returning `None` — the winning thread will have re-inserted the entry.
2. **Hold the memory write lock across the entire sequence**: extend `get_refresh` to also perform the backend lookup and re-insertion under the same lock guard.
3. **Use a per-key entry API or `DashMap`-style occupied/vacant entry**: prevents two threads from ever racing on the same key's promote path.

Additionally, replace the unconditional `.expect("parent should be verified")` in `insert_valid_header` with a recoverable error path to limit blast radius from any future similar issue.

## Proof of Concept

```
Thread A (peer 1, child C1 with parent H)     Thread B (peer 2, child C2 with parent H)
──────────────────────────────────────────────────────────────────────────────────────
insert_valid_header(C1)                        insert_valid_header(C2)
  get_header_index_view(&H.hash, false)          get_header_index_view(&H.hash, false)
    header_map().get(&H.hash)                      header_map().get(&H.hash)
      memory.get_refresh(&H.hash) → None             memory.get_refresh(&H.hash) → None
      backend.remove(&H.hash) → Some(v)              backend.remove(&H.hash) → None  ← H lost
      memory.insert(v)                               return None
      return Some(v)                               get_header_index_view → None
                                               .expect("parent should be verified") → PANIC
```

**Minimal reproduction steps:**

1. Start a CKB node with a `memory_limit` small enough that IBD evicts entries to sled (or reduce the limit in config).
2. Connect two peers that both send a batch of headers sharing a common parent H at a height where H has been evicted.
3. Observe the node panic with `"parent should be verified"` in the thread handling one of the two peers.

A targeted integration test can reproduce this deterministically by: inserting a parent header H into the `HeaderMapKernel`, calling `limit_memory()` to force eviction to sled, then spawning two threads that concurrently call `kernel.get(&H.hash)` and asserting both return `Some`.
