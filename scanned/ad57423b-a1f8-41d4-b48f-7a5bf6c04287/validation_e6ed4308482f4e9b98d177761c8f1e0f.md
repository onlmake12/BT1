Audit Report

## Title
Missing `BLOCK_INVALID` Guard in `HeaderAcceptor::accept()` Enables Repeated Re-acceptance of Rejected Headers — (`File: sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` noting that a `BLOCK_INVALID` early-return is missing. Because `BLOCK_INVALID` (`1 << 12`) shares no bits with `HEADER_VALID` (`1`), the existing guard does not fire, and a header for a fully-verified-and-rejected block passes all three header-level sub-checks, causing `insert_valid_header` to be called. The block fetcher subsequently does not check `BLOCK_INVALID` before scheduling downloads, so the same invalid block is re-requested from the peer, wasting bandwidth and CPU in a repeatable loop.

## Finding Description

**Root cause — missing bit-flag guard:**

`BLOCK_INVALID = 1 << 12 = 4096` and `HEADER_VALID = 1`. These flags share no bits. [1](#0-0) 

In `accept()`, the only early-return checks `status.contains(BlockStatus::HEADER_VALID)`. When status is `BLOCK_INVALID`, this condition is false and execution falls through — exactly as the `FIXME` comment acknowledges: [2](#0-1) 

**Why the three sub-checks do not catch it:**

- `prev_block_check` (lines 244–253) checks whether the *parent* has `BLOCK_INVALID` — not the block itself. A block whose body fails script execution has a valid parent. [3](#0-2) 
- `non_contextual_check` runs `HeaderVerifier::verify()` — PoW and structural checks only. A block can be `BLOCK_INVALID` due to invalid transactions while its header passes PoW. [4](#0-3) 
- `version_check` only tests `header.version() != 0`. [5](#0-4) 

All three pass → `insert_valid_header` is called at line 356, inserting the header into the header map and updating the peer's best-known header. [6](#0-5) 

**`insert_valid_header` does not set `HEADER_VALID` in `block_status_map`:**

The function inserts into `header_map` and calls `may_set_best_known_header` / `may_set_shared_best_header`, but never calls `insert_block_status`. The `BLOCK_INVALID` flag therefore remains in `block_status_map`, meaning the next `SendHeaders` with the same header will again fall through the same path — the loop is unbounded. [7](#0-6) 

**Block fetcher does not guard on `BLOCK_INVALID`:**

`block_fetcher.rs` skips blocks with `BLOCK_STORED` or `BLOCK_RECEIVED`, but has no `BLOCK_INVALID` branch. Since `BLOCK_INVALID` (4096) shares no bits with either flag, the fetcher falls into the else-if and schedules a `GetBlocks` request: [8](#0-7) 

**Full cycle:**
1. Block B fails full verification → `BLOCK_INVALID` written to `block_status_map`.
2. Attacker sends `SendHeaders` containing B's header.
3. `accept()` reads `BLOCK_INVALID`, skips `HEADER_VALID` branch, passes all three sub-checks, calls `insert_valid_header`.
4. B's header is now in `header_map`; peer's best-known header points to B.
5. Block fetcher traverses from peer's best-known header, finds B, sees neither `BLOCK_STORED` nor `BLOCK_RECEIVED`, schedules download.
6. B is downloaded, fails verification again, `BLOCK_INVALID` is re-written.
7. Because `insert_valid_header` never set `HEADER_VALID`, the next `SendHeaders` repeats from step 3.

## Impact Explanation

**High — network congestion with few costs.**

Each iteration of the loop causes: (a) CPU work inside `insert_valid_header` (skip-pointer construction, peer-state update, shared-best-header update); (b) a `GetBlocks` network round-trip; (c) full block download and re-verification. An attacker operating multiple peers can sustain this against many nodes simultaneously with negligible cost (only `SendHeaders` messages are needed to re-trigger the cycle after each peer is disconnected). This matches the allowed impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

Any unprivileged remote peer can send `SendHeaders`. The attacker only needs the hash of a block that the target node has already rejected — observable by monitoring the network or by deliberately crafting a block whose header passes PoW but whose body contains an invalid transaction. No keys, privileges, or majority hashpower are required. The attack is repeatable across reconnections or different peer identities.

## Recommendation

Add an explicit `BLOCK_INVALID` guard at the top of `accept()`, before the `HEADER_VALID` check, resolving the existing `FIXME`:

```rust
pub fn accept(&self) -> ValidationResult {
    let mut result = ValidationResult::default();
    let sync_shared = self.active_chain.sync_shared();
    let state = self.active_chain.state();
    let shared = sync_shared.shared();

    let status = self.active_chain.get_block_status(&self.header.hash());

    // Resolve FIXME: reject headers for blocks already known to be invalid
    if status.contains(BlockStatus::BLOCK_INVALID) {
        result.invalid(Some(ValidationError::InvalidParent));
        return result;
    }

    if status.contains(BlockStatus::HEADER_VALID) {
        // ... existing logic ...
    }
    // ... rest of function ...
}
```

Additionally, the block fetcher should be hardened to skip `BLOCK_INVALID` blocks in its download loop, analogous to the guard already present in `compact_block_process.rs`: [9](#0-8) 

## Proof of Concept

1. Run a CKB node (node A). Let it sync several blocks so it has a known-good chain tip.
2. Craft block B: valid header (correct PoW, parent = chain tip, version = 0), but body contains an invalid transaction (e.g., double-spend or script failure).
3. Submit B to node A via `BlockProcess` (or via a peer). Node A stores B, runs full verification, marks it `BLOCK_INVALID` in `block_status_map`.
4. From a test peer, send a `SendHeaders` message containing only B's header.
5. Observe that `HeaderAcceptor::accept()` calls `insert_valid_header` (add a log or breakpoint at line 356) — confirming the missing guard.
6. Observe that the block fetcher subsequently issues a `GetBlocks` request for B (log at `block_fetcher.rs` line 283).
7. Repeat step 4 from the same or a different peer; confirm the cycle repeats without bound.

An integration test analogous to `test/src/specs/sync/invalid_block.rs` can be extended to assert that after a block is marked `BLOCK_INVALID`, a subsequent `SendHeaders` containing that header does **not** result in a `GetBlocks` request. [10](#0-9)

pub struct ChainContainsInvalidBlock;

impl Spec for ChainContainsInvalidBlock {
    crate::setup!(num_nodes: 3);

    // Case:
    //   1. `bad_node` generate a long chain `CN` contains an invalid block
    //      B[i] is an invalid block.
    //   2. `good_node` synchronizes from `bad_node`. We expect that `good_node`
    //      end synchronizing at B[i-1].
    //   3. `good_node` mines a new block B[i]', i < `CN.length`.
    //   4. `fresh_node` synchronizes from `bad_node` and `good_node`. We expect
    //      that `fresh_node` synchronizes the valid chain.
    fn run(&self, nodes: &mut Vec<Node>) {
        let bad_node = nodes.pop().unwrap();
        let good_node = nodes.pop().unwrap();
        let fresh_node = nodes.pop().unwrap();

        // wait a few seconds to let find_unverified_block thread finish
        #[cfg(target_os = "macos")]
        std::thread::sleep(Duration::from_secs(3));

        // Build invalid chain on bad_node
        bad_node.mine(3);
        let invalid_block = bad_node
            .new_block_builder(None, None, None)
            .transaction(TransactionBuilder::default().build())
            .build();
        let invalid_number = invalid_block.header().number();
        let invalid_hash = bad_node.process_block_without_verify(&invalid_block, false);
        bad_node.mine(3);

        // Start good_node and let it synchronize from bad_node
        good_node.mine(1);
        fresh_node.connect(&good_node);
        good_node.connect_and_wait_ban(&bad_node);
        fresh_node.connect_and_wait_ban(&bad_node);
        assert!(
            wait_until(5, || good_node.get_tip_block_number() >= invalid_number - 1),
            "good_node should synchronize from bad_node 1~{}",
            invalid_number - 1,
        );
        assert!(
            !wait_until(5, || good_node
                .rpc_client()
                .get_block(invalid_hash.clone())
                .is_some()),
            "good_node should not synchronize invalid block {invalid_number} from bad_node"
        );

        // good_node mine the next block
        good_node.mine(1);
        let valid_hash = good_node.get_tip_block().header().hash();
        let valid_number = invalid_number + 1;

        assert!(
            !wait_until(5, || fresh_node.get_tip_block_number() > valid_number),
            "fresh_node should synchronize the valid blocks only",
        );
        assert!(
            wait_until(5, || fresh_node.get_tip_block().header().hash()
                == valid_hash),
            "fresh_node should synchronize the valid blocks only",
        );
    }
