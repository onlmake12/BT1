Audit Report

## Title
Unbounded Cell Iteration in `get_cells_capacity()` Enables RPC Thread Exhaustion — (File: `util/indexer/src/service.rs`)

## Summary
`get_cells_capacity()` accepts no `limit` parameter and iterates every RocksDB cell matching a caller-supplied prefix with no count-based bound. The only guard is a `TimeoutIterator` that blocks the calling thread for up to the full `timeout_limit` (default 10 seconds) before returning an error. Concurrent calls from an unprivileged attacker can saturate the synchronous RPC thread pool, making the node's RPC interface unresponsive.

## Finding Description
`get_cells()` (L193) and `get_transactions()` (L381) both accept a `limit: Uint32` parameter, validate it against `self.request_limit` (L212–221, L388–397), and terminate iteration early with `.take(limit)` (L371). `get_cells_capacity()` (L686) accepts only `search_key` with no limit parameter, constructs an unbounded `TimeoutIterator` (L720), and iterates with only a prefix `take_while` guard (L728) — no `.take(N)` count bound exists anywhere in the function body (L726–836). For each matching key, it performs a secondary `snapshot.get(Key::OutPoint(...))` lookup (L748–752). The `TimeoutIterator::next()` checks wall-clock elapsed time on every call (L55–61); until the timeout fires, the thread is fully occupied. The default `timeout_limit` is 10 seconds (L99). The RPC trait exposes this as a public, unauthenticated endpoint (L879–883).

## Impact Explanation
Each call to `get_cells_capacity` with a broad prefix occupies one synchronous RPC handler thread for up to 10 seconds. Concurrent calls exhaust the thread pool, making the node's entire RPC interface unresponsive. This matches the allowed impact: **Note (0–500 points) — Any local RPC API crash**. The impact is bounded to a single node's RPC surface; it does not crash the node process, cause consensus deviation, or affect the P2P network, so it does not rise to High.

## Likelihood Explanation
The endpoint requires no authentication and is publicly documented. An attacker needs only a standard CKB wallet to create cells under a shared script prefix over time (normal on-chain activity), then issue repeated concurrent `get_cells_capacity` calls with that prefix. No special privilege, key material, or majority hashpower is required. The attack is low-cost and repeatable.

## Recommendation
Apply the same limit enforcement already used by `get_cells()` and `get_transactions()`:
1. Add a `limit: Uint32` parameter to `get_cells_capacity()` in both `rpc/src/module/indexer.rs` (L880–883) and `util/indexer/src/service.rs` (L686–689).
2. Reject requests where `limit == 0` or `limit > self.request_limit`.
3. Add `.take(limit)` to the iterator chain at L726 and return a partial sum with a cursor for pagination.

Alternatively, enforce a hard cap of `self.request_limit` cells scanned and return an error if the result is truncated.

## Proof of Concept
1. Submit N transactions (e.g., N = 100,000) each creating one cell with `lock.args = 0x` (empty args).
2. Wait for the indexer to sync.
3. Send M concurrent JSON-RPC calls to `get_cells_capacity` with `search_key.script.args = "0x"` and `script_search_mode = "prefix"`, where M equals the RPC thread pool size.
4. Observe each call blocks for the full `timeout_limit` (10 s default) before returning `"Indexer request timeout"`.
5. During those 10 seconds, all other RPC calls (e.g., `get_tip_block_number`) queue behind the exhausted thread pool and time out from the caller's perspective.
6. Contrast: the same search key sent to `get_cells` with `limit: 1` returns immediately after one cell is found.
