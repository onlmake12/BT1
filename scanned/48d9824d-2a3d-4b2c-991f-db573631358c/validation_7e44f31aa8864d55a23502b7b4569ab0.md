Audit Report

## Title
Unbounded RocksDB Read Amplification via `GetLastStateProof` Binary Search — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary

The `GET_LAST_STATE_PROOF_LIMIT` guard in `GetLastStateProofProcess::execute` bounds only the *count* of difficulty entries (≤ 1000), not the number of RocksDB reads those entries trigger. Each difficulty entry causes `get_first_block_total_difficulty_is_not_less_than` to run a binary search over the full block range, issuing 2 RocksDB reads per step and O(log H) steps per entry. The `LightClientProtocol` handler has no per-peer rate limiter, unlike `Relayer` and `HolePunching` which both use `governor::RateLimiter`. A sustained flood of crafted `GetLastStateProof` messages can saturate RocksDB read I/O and crash or severely degrade the node.

## Finding Description

**Guard that is supposed to bound work** — `execute()` rejects messages where `difficulties.len() + last_n_blocks * 2 > 1000`: [1](#0-0) 

This only limits N (the number of difficulty entries). It does not limit the number of storage reads those N entries trigger.

**Binary search loop** — `get_first_block_total_difficulty_is_not_less_than` iterates O(log H) times over `[start_block_number, end_block_number)`: [2](#0-1) 

Each iteration calls `get_block_total_difficulty`, which issues two RocksDB reads: [3](#0-2) 

**`start_block_number` advancement does not eliminate worst-case amplification** — after finding block `num` for difficulty `d_i`, `start_block_number` is set to `num - 1`: [4](#0-3) 

An attacker who crafts difficulties resolving to blocks 1, 2, 3, …, N keeps `start_block_number` near 0 for every iteration, maintaining a search range of approximately `[0, H)` throughout, yielding O(N × log H) total reads.

**Unconditional `difficulty_boundary` binary search** — an additional O(log H) binary search is always executed before the per-difficulty loop: [5](#0-4) 

**No rate limiting on the light client handler** — `LightClientProtocol` has no `rate_limiter` field and `received` dispatches directly to `try_process` with no per-peer throttle: [6](#0-5) [7](#0-6) 

By contrast, `Relayer` and `HolePunching` both carry a `governor::RateLimiter` keyed by peer and message type: [8](#0-7) [9](#0-8) 

The `governor` crate is not even a dependency of `util/light-client-protocol-server`.

## Impact Explanation

**Reads per request (worst case):**
- N = 1000 difficulties (`last_n_blocks = 0`)
- H ≈ 10⁶ (mainnet chain height), log₂(H) ≈ 20
- Per-difficulty binary search: 1000 × 20 × 2 = **40,000 RocksDB reads**
- `difficulty_boundary` search: +40 reads
- `complete_headers` (`get_ancestor` + `get_block` per sampled block): additional O(N × log H) reads

**Flood scenario:** 100 concurrent connections × 40,000 reads = **4,000,000 RocksDB reads per second**, sustained indefinitely. This saturates disk I/O and degrades block sync, transaction relay, and all other peers sharing the same node.

This matches the allowed CKB bounty impact: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points). The `GET_LAST_STATE_PROOF_LIMIT = 1000` constant creates a false sense of safety: it bounds output size but not I/O cost. [10](#0-9) 

## Likelihood Explanation

- No PoW, no stake, no authentication required — any peer can connect to the light client protocol endpoint.
- The default `max_peers = 125` allows up to 125 simultaneous inbound connections.
- The attack is trivially scriptable: open connections, send crafted `GetLastStateProof` in a tight loop.
- The attacker needs only the chain's total difficulty range (publicly available from any synced node or block explorer) to craft valid, maximally-amplifying difficulty lists.
- The attack is repeatable and sustained; there is no mechanism to detect or throttle it.

## Recommendation

1. **Bound total binary-search steps, not just difficulty count.** Replace the current guard with one that accounts for the O(log H) factor, e.g., limit `difficulties.len() ≤ GET_LAST_STATE_PROOF_LIMIT / log2(chain_height_estimate)`.
2. **Add per-peer rate limiting** on `GetLastStateProof` messages, analogous to the `governor::RateLimiter` used in `Relayer` and `HolePunching`.
3. **Cap the binary search range** by enforcing a maximum `end_block_number - start_block_number` in `get_first_block_total_difficulty_is_not_less_than`.

## Proof of Concept

```
1. Connect to a CKB full node running the LightClient protocol (default port 8115).
2. Obtain total_diff(block #1) and total_diff(tip) from any public source.
3. Generate 1000 monotonically increasing U256 difficulty values d_1 < d_2 < ... < d_1000,
   each chosen to fall within [total_diff(block 1), total_diff(block N)],
   spaced so that each d_i resolves to a block near the start of the chain.
4. Craft a GetLastStateProof message:
     last_hash            = current tip hash
     start_hash           = genesis hash
     start_number         = 0
     last_n_blocks        = 0
     difficulty_boundary  = total_diff(tip) + 1
     difficulties         = [d_1, ..., d_1000]
5. Send 100 such messages concurrently from 100 connections in a tight loop.
6. Monitor RocksDB read counters (via /proc/<pid>/io or RocksDB statistics).
7. Assert: observed reads/sec >> 1000 (the nominal GET_LAST_STATE_PROOF_LIMIT),
   confirming O(N × log H) amplification per request.
```

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
