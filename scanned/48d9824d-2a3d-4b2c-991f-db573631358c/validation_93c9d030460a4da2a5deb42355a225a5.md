Audit Report

## Title
Tautological Deduplication Guard Allows Unbounded Channel Growth Leading to Miner OOM — (`miner/src/client.rs`)

## Summary
In `update_block_template`, the `fetch_update` closure evaluates `id != work_id || id == 0`. When an attacker supplies `work_id = 0`, this becomes `(id != 0) || (id == 0)`, which is unconditionally true for every stored value of `id`. Combined with an unbounded crossbeam channel and an unauthenticated, rate-unlimited HTTP notify endpoint, any caller with TCP access to the miner's listen address can flood the channel with `Works::New` entries faster than the miner consumes them, growing heap usage until the process is OOM-killed.

## Finding Description
**Root cause — tautological closure:** [1](#0-0) 

When `work_id = 0` is supplied, the condition `id != 0 || id == 0` is a tautology. `fetch_update` always returns `Ok`, so the send on line 308 executes unconditionally on every call.

**Unbounded channel:** [2](#0-1) 

No capacity limit exists; the channel grows without bound.

**Unauthenticated notify endpoint — no validation in handler:** [3](#0-2) 

**No connection cap or rate limit in accept loop:** [4](#0-3) 

Exploit flow: attacker opens TCP connections to the miner's notify address → POSTs valid `BlockTemplate` JSON with `work_id = "0x0"` in a tight loop → each request bypasses deduplication → one `Works::New(Work)` (full block template struct) enqueued per request → consumer (`Miner::run`) cannot drain at HTTP flood rate → heap grows monotonically → OOM kill.

## Impact Explanation
The miner process is crashed via heap exhaustion. This maps to **Note (0–500 points): Any local RPC API crash**. The miner is a separate process from the CKB node; its crash does not directly affect node consensus or network stability, but it halts all mining activity for the operator.

## Likelihood Explanation
The notify listen address is operator-configured. The documented example in the log message at line 216 shows `http://{addr}`, and if bound to any non-loopback interface, any network-adjacent host can trigger this with no credentials, no PoW, and no special privileges — only the ability to open a TCP connection and send HTTP POST. Even on loopback, any unprivileged local process suffices. The condition is opt-in (requires `config.listen` to be set), but when enabled, exploitation is trivial and repeatable.

## Recommendation
Two independent fixes are needed:

1. **Fix the tautology**: The intent is to force an update when the *stored* ID is zero (initial state). The correct guard is `id != work_id || id == 0` where `id == 0` refers to the stored value being uninitialized — but when `work_id` is also `0`, the clause fires spuriously. Replace with a strict inequality: `if id != work_id { Some(work_id) } else { None }`. The initial-state bootstrap is unnecessary because `work_id = 0` from the node is a valid first update that `id != work_id` already handles (stored starts at `0`, first real `work_id` will differ).

2. **Bound the channel**: Replace `unbounded()` with `bounded(N)` (e.g., `bounded(4)`) so a slow consumer applies back-pressure and excess sends are dropped rather than queued indefinitely. [2](#0-1)
