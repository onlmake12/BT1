Audit Report

## Title
TxEntry Timestamp Reset in `readd_detached_tx` Bypasses Expiry Enforcement During Chain Reorganization - (File: tx-pool/src/process.rs)

## Summary
During a chain reorganization, `readd_detached_tx` re-admits detached transactions by constructing a brand-new `TxEntry` via `TxEntry::new()`, which stamps the entry with the current wall-clock time. Because `remove_expired` and `limit_size` are both called inside `_update_tx_pool_for_reorg` before `readd_detached_tx` runs, the freshly re-stamped entries bypass the expiry check entirely until the next block. A transaction near its expiry deadline survives the reorg with a fully reset expiry window, and the pool can transiently exceed `max_tx_pool_size` with no immediate enforcement.

## Finding Description
`TxEntry::new()` unconditionally stamps the entry with `unix_time_as_millis()`: [1](#0-0) 

`readd_detached_tx` calls this constructor for every recovered transaction: [2](#0-1) 

By contrast, `remove_by_detached_proposal` reuses the existing `TxEntry` object and only calls `reset_statistic_state()`, preserving the original `timestamp`: [3](#0-2) 

The call sequence in `update_tx_pool_for_reorg` is:
1. `_update_tx_pool_for_reorg` (contains `remove_expired` and `limit_size`)
2. `readd_detached_tx` (re-admits with fresh timestamps) [4](#0-3) 

`remove_expired` filters using `self.expiry + entry.inner.timestamp < now_ms`: [5](#0-4) 

Because `remove_expired` runs before `readd_detached_tx`, the re-admitted entries with fresh timestamps are invisible to the expiry check until the next block event. A transaction that was 1 hour from expiry before the reorg re-enters the pool with a full `expiry_hours` window.

The `EvictKey` ordering uses `timestamp` as a tiebreaker for same-fee-rate entries: [6](#0-5) 

The comment states "select the latest timestamp" for eviction, meaning the eviction cursor targets the newest same-fee-rate entry first. A re-admitted transaction with `timestamp = now` (the largest possible value) therefore becomes the primary eviction candidate among same-fee-rate peers, which is the inverse of the report's eviction-inversion claim. The eviction ordering impact as described in the submission is directionally incorrect — re-admitted entries are actually evicted *sooner*, not later. However, this does not negate the expiry bypass.

The pool overflow is real but transient: `limit_size` runs before `readd_detached_tx`, so the pool can exceed `max_tx_pool_size` after re-admission with no immediate enforcement until the next block. [7](#0-6) 

## Impact Explanation
The confirmed impact is **expiry bypass**: a low-fee transaction near its deadline survives a reorg with a fully reset expiry clock. This is a suboptimal implementation of the CKB tx-pool state management mechanism, matching the **Medium (2001–10000 points)** impact class: *Suboptimal implementation of CKB state storage mechanism*. The pool can also transiently exceed its size limit after a reorg, though this self-corrects on the next block. The eviction-order-inversion claim in the submission is not confirmed by the code — the `EvictKey` ordering evicts the *newest* same-fee-rate entry first, so re-admitted entries are not deprioritized.

## Likelihood Explanation
Natural one- or two-block reorgs occur on mainnet without any attacker involvement. Every such reorg triggers `update_tx_pool_for_reorg` → `readd_detached_tx`. No special privileges, hashpower, or key material are required. The expiry bypass is triggered by the standard block-processing pipeline reachable by any peer relaying a valid competing chain tip. Indefinite survival requires repeated reorgs of the same block, which is unrealistic in practice; a single reorg grants at most one additional `expiry_hours` window.

## Recommendation
Preserve the original pool-entry timestamp when re-admitting detached transactions, mirroring `remove_by_detached_proposal`. Record the `(tx_hash → timestamp)` mapping before removing committed transactions in `_update_tx_pool_for_reorg`, pass it into `readd_detached_tx`, and use `TxEntry::new_with_timestamp` with the recovered value when available, falling back to `unix_time_as_millis()` only for transactions with no prior pool history. [8](#0-7) 

## Proof of Concept
1. Configure a node with `expiry_hours = 12`.
2. Submit transaction `T` with fee-rate just above `min_fee_rate`; record pool timestamp `t0`.
3. Mine a block that includes `T`; `T` is removed from the pool via `remove_committed_tx`.
4. Set `t1 = t0 + 11h` (1 hour before expiry). Arrange a one-block reorg at this moment (e.g., via a competing chain of equal work).
5. `readd_detached_tx` re-admits `T` with `timestamp = now ≈ t0 + 11h`.
6. Query `get_transaction` RPC: `time_added_to_pool` has jumped forward; `T` now has a full 12-hour expiry window from `t1`.
7. Confirm via `remove_expired` logic: `expiry + t1 > now`, so `T` is not expired.
8. Without the reorg, `T` would have expired at `t0 + 12h`; with the reorg it survives until `t1 + 12h = t0 + 23h`.

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
