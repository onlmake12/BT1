### Title
Minimum Fee Check Uses Only Serialized Size, Ignoring Cycles — Allows Sub-Minimum Fee-Rate Transactions Into the Tx-Pool - (File: `tx-pool/src/util.rs`)

### Summary
The `check_tx_fee` function in the tx-pool admission path computes the minimum required fee using only the transaction's serialized byte size as weight. However, the actual fee rate of a pool entry is computed using `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. A transaction with a small serialized size but high cycle consumption passes the size-only gate yet enters the pool with an effective fee rate far below the configured minimum.

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee rate using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { ... }
``` [1](#0-0) 

The actual fee rate of a `TxEntry` is computed with the full weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

Where `get_transaction_weight` is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

The admission flow in `_process_tx` calls `pre_check` (which invokes `check_tx_fee`) **before** `verify_rtx`, so cycles are not yet known at the fee-check point: [4](#0-3) 

The `pre_check` function passes `tx_size` to `check_tx_fee` and the fee check passes before cycles are determined: [5](#0-4) 

After verification, the `TxEntry` is created with the real cycles, but no second fee-rate check is performed against the full weight: [6](#0-5) 

### Impact Explanation

An attacker submits a transaction with:
- Small serialized size (e.g., 100 bytes) — passes `min_fee_rate.fee(100)`
- High cycle consumption (e.g., 10,000,000 cycles) — actual weight ≈ `max(100, 10_000_000 × 0.000_170_571) ≈ 1,705`
- Fee just above `min_fee_rate × 100` but far below `min_fee_rate × 1,705`

The transaction enters the pool with an effective fee rate ~17× below the minimum. This:
1. **Pollutes the tx-pool** with transactions that miners have little incentive to include, causing them to linger and consume pool capacity.
2. **Displaces legitimate transactions**: when `limit_size` evicts entries, it uses the real fee rate, so these sub-minimum entries may survive longer than expected while higher-fee-rate transactions are evicted.
3. **Wastes node verification resources**: the full cycle execution in `verify_rtx` runs before the entry is admitted, consuming CPU for transactions that should have been rejected.

### Likelihood Explanation

The entry path is fully reachable by any unprivileged RPC caller via `send_transaction` or by a P2P peer relaying transactions. The code comment explicitly acknowledges the approximation ("Theoretically we cannot use size as weight directly"), confirming the gap is known but unmitigated. Crafting a small transaction with high cycle consumption is straightforward for any script author. [7](#0-6) 

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee-rate check using the full weight before calling `submit_entry`:

```rust
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
// Re-check fee rate with actual weight (size + cycles)
let actual_fee_rate = entry.fee_rate();
if actual_fee_rate < tx_pool_config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

This mirrors the pattern already used for eviction and sorting, and closes the gap between the admission check and the actual resource cost.

### Proof of Concept

1. Craft a CKB transaction with a lock script that loops for ~10,000,000 cycles but whose serialized size is ~100 bytes.
2. Set the transaction fee to exactly `min_fee_rate.fee(100)` (e.g., 1,000 shannons at 1,000 shannons/KB).
3. Submit via `send_transaction` RPC.
4. Observe: `check_tx_fee` passes (fee ≥ size-based minimum).
5. After `verify_rtx`, the entry's actual weight is `max(100, 10_000_000 × 0.000_170_571) ≈ 1,705`.
6. The entry's effective fee rate is `1,000 / 1,705 × 1,000 ≈ 586 shannons/KW`, far below the 1,000 shannons/KW minimum.
7. The transaction is now in the pool with a sub-minimum fee rate, confirmed by `get_pool_tx_detail_info` RPC showing `score_sortkey.weight` >> `score_sortkey.fee / min_fee_rate`.
