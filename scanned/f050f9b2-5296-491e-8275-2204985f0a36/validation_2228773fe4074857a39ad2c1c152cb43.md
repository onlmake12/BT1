### Title
Global Orphan Transaction Pool Saturation via No Per-Peer Limit Allows Targeted Eviction of Legitimate Orphan Transactions — (`tx-pool/src/component/orphan.rs`)

### Summary

The `OrphanPool` in CKB's transaction pool enforces a single global limit of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` with no per-peer accounting. When the pool is full, entries are evicted in HashMap iteration order (effectively random), with no preference for entries from honest peers. A single malicious P2P peer can flood the pool with 100 crafted orphan transactions referencing non-existent parents, continuously evicting legitimate orphan transactions from honest peers. When the legitimate parent transactions later arrive, their children are absent from the orphan pool and are not automatically promoted, requiring re-relay.

### Finding Description

**Root cause — `tx-pool/src/component/orphan.rs`:**

The `OrphanPool` struct tracks all orphan transactions in a single global `HashMap<ProposalShortId, Entry>` with no per-peer sub-accounting: [1](#0-0) 

The global cap is a compile-time constant: [2](#0-1) 

When `add_orphan_tx` is called, it inserts the entry and then calls `limit_size()`: [3](#0-2) 

`limit_size()` evicts entries in HashMap iteration order (non-deterministic, effectively random) until the pool is at or below 100: [4](#0-3) 

The `peer: PeerIndex` field stored in each `Entry` is never consulted during eviction: [5](#0-4) 

**Attack path:**

1. Attacker connects as a P2P peer via the relay protocol.
2. Attacker sends 100 `RelayTransaction` messages, each referencing a fabricated (non-existent) parent `OutPoint`. Each transaction passes the declared-cycle check and is admitted to the orphan pool via `add_orphan` in `TxPoolService`: [6](#0-5) 

3. The pool is now at capacity (100 entries, all attacker-controlled).
4. Any subsequent legitimate orphan transaction from an honest peer triggers `limit_size()`, which randomly evicts one entry. Because the attacker can immediately re-send a replacement orphan, the attacker maintains pool saturation.
5. When the legitimate parent transaction arrives and `process_orphan_tx` is called, `find_by_previous` returns nothing for the evicted child: [7](#0-6) 

6. The child transaction is silently dropped from the orphan pool and is not promoted to pending. The original sender must re-relay it.

**Why existing mitigations are insufficient:**

The `TooManyUnknownTransactions` ban (triggered by `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767` unknown hashes in a `RelayTransactionHashes` message) operates on a different code path — it tracks unknown *hash announcements*, not orphan *transaction submissions*. An attacker sending 100 full orphan transactions via `RelayTransaction` messages does not trigger this ban. [8](#0-7) 

### Impact Explanation

Legitimate orphan transactions are evicted from the pool before their parents arrive. The child transactions are not automatically promoted when the parent is later received, causing transaction propagation failures that require re-relay by the original sender. Under sustained attack, an attacker can keep the orphan pool permanently saturated, causing all orphan transactions from honest peers to be continuously evicted. This is a service-availability degradation for transaction relay, analogous to the VotingEscrow `MAX_DELEGATES` saturation that blocks legitimate delegations.

### Likelihood Explanation

The attack requires only a single P2P connection and 100 small transactions with fabricated parent hashes. No funds, no privileged access, and no special timing are required. The attacker can sustain the attack indefinitely at negligible cost by continuously re-sending replacement orphans as slots open. The 100-slot global limit makes saturation trivially achievable.

### Recommendation

1. **Add per-peer orphan limits**: Track orphan count per `PeerIndex` and reject or evict preferentially from the peer with the most orphans when the global limit is reached.
2. **Prefer evicting from the most-represented peer**: Replace the random eviction in `limit_size()` with a policy that evicts from the peer holding the most orphan slots, protecting honest peers from attacker-controlled saturation.
3. **Increase the global limit or make it configurable**: The current limit of 100 is very small; even modest honest traffic can be disrupted.

### Proof of Concept

A malicious peer connects and sends 100 `RelayTransaction` messages, each a minimal valid transaction whose single input references a fabricated `OutPoint` (random 32-byte tx hash, index 0). Each transaction is structurally valid (passes size/cycle checks) but has an unknown parent, so it is admitted to the orphan pool. After 100 such messages, the pool is full. Any subsequent legitimate orphan from an honest peer triggers random eviction. The attacker immediately re-sends a replacement orphan to maintain saturation. When the honest peer's parent transaction arrives, `find_by_previous` returns an empty list for the evicted child, and the child is silently lost from the pool.

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
