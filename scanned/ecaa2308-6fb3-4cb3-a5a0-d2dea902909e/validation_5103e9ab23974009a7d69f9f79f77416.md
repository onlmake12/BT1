Audit Report

## Title
Peer Store Permanently Blocked by Single-/16-Group Flood via Discovery Nodes — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge`'s second-pass eviction uses integer division `len / 2` to determine how many network groups to process. When all 16384 peer store entries share the same /16 network group (`len == 1`), `1 / 2 == 0` groups are taken, no eviction occurs, and `EvictionFailed` is returned. Because `add_addr` is called unconditionally for every address received via the Discovery protocol and errors are only logged at `debug` level, an attacker who floods the store with 16384 addresses from a single /16 permanently blocks all future peer discovery until the node is restarted.

## Finding Description

**Root cause — integer division in `check_purge` pass 2:** [1](#0-0) 

When `peers_by_network_group.len() == 1`, `.take(len / 2)` becomes `.take(0)`, yielding an empty iterator. `candidate_peers` is empty, and the function returns `Err(PeerStoreError::EvictionFailed)`. [2](#0-1) 

**Why pass 1 also fails for attacker-injected entries:**

`add_addr` always creates entries with `last_connected_at_ms = 0` and `attempts_count = 0`: [3](#0-2) 

`is_connectable` returns `true` for such entries (neither `attempts_count >= 3` nor `attempts_count >= 10` is satisfied), so pass 1 finds zero eviction candidates: [4](#0-3) 

**Network group is /16, not /24:** [5](#0-4) 

A single /16 block provides 65,536 unique IPs, more than enough to fill all 16,384 slots.

**Error is silently swallowed by the Discovery handler:** [6](#0-5) 

**Store capacity:** [7](#0-6) 

## Impact Explanation

Once the store is saturated, every subsequent `add_addr` call returns `EvictionFailed`. The node can no longer learn about new peers via Discovery or Identify. As existing connections churn naturally, the node cannot replace them and will progressively lose connectivity until it is fully isolated from the network. This constitutes a **High** impact: a remotely triggerable condition that can effectively crash a CKB node's networking layer with no recovery path short of a restart with peer store deletion.

## Likelihood Explanation

A single malicious peer already connected via P2P can send repeated Discovery `Nodes` messages. Each message can carry many addresses. All addresses need only share the same /16 (e.g., `1.2.x.x`) with distinct peer IDs. The `is_valid_addr` / `is_reachable` filter is satisfied by any public /16 range. No privileged access is required; any peer that completes the P2P handshake can execute this attack. The 16,384-entry limit is reachable with sustained but modest message volume.

## Recommendation

Replace `len / 2` with `(len + 1) / 2` (ceiling division) so that at least one group is processed when `len >= 1`:

```rust
.take((len + 1) / 2)
```

Additionally, enforce a per-/16 admission cap inside `add_addr` to prevent any single network group from monopolizing the store.

## Proof of Concept

```rust
let mut peer_store = PeerStore::default();
// Fill with 16384 entries all from 1.2.x.x (/16 group IP4([1, 2]))
for i in 0u32..16384 {
    let ip = std::net::Ipv4Addr::new(1, 2, (i >> 8) as u8, i as u8);
    let peer_id = PeerId::random();
    let addr: Multiaddr = format!("/ip4/{}/tcp/8114/p2p/{}", ip, peer_id.to_base58())
        .parse().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Store is now full with a single /16 group (len == 1 in check_purge pass 2)
let new_peer_id = PeerId::random();
let new_addr: Multiaddr = format!("/ip4/8.8.8.8/tcp/8114/p2p/{}", new_peer_id.to_base58())
    .parse().unwrap();
// Returns EvictionFailed — peer store is permanently blocked
assert!(peer_store.add_addr(new_addr, Flags::COMPATIBILITY).is_err());
```
