Audit Report

## Title
Unbounded Per-Message DB Amplification in `GetTransactionsProof` Handler with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

The `GetTransactionsProofProcess::execute` handler accepts up to 1000 tx hashes per message and performs O(6000 + 1000×log(chain_length)) RocksDB reads per message with no per-peer rate limiting. Unlike the relay protocol, `LightClientProtocol` carries no `RateLimiter`, and successful handler execution returns `Status::ok()` (code 200), which never triggers a ban. An unprivileged remote peer can flood the server with maximum-size requests continuously, saturating RocksDB I/O and degrading node performance for all peers.

## Finding Description

`GET_TRANSACTIONS_PROOF_LIMIT` is set to 1000 in `constant.rs`: [1](#0-0) 

The check in `execute()` uses strict `>`, so exactly 1000 hashes pass through: [2](#0-1) 

For each of the 1000 hashes, `get_transaction_info()` is called in the partition step: [3](#0-2) 

For each found tx, `get_transaction_with_info()` is called: [4](#0-3) 

For each unique block (up to 1000), `get_block()`, `CBMT::build_merkle_proof()`, `get_block_uncles()`, and `get_block_extension()` are all called: [5](#0-4) 

`reply_proof()` then calls `mmr.get_root()` and `mmr.gen_proof(items_positions)` where `items_positions` can hold up to 1000 entries, costing O(k × log N) MMR node reads: [6](#0-5) 

On successful completion, `reply_proof()` returns `Status::ok()`. `should_ban()` only fires on 4xx codes, so no ban is ever triggered for a well-formed max-size request: [7](#0-6) 

`LightClientProtocol` carries no rate limiter — the struct holds only `shared: Shared`: [8](#0-7) 

The `received()` handler dispatches directly to `try_process()` with no throttle: [9](#0-8) 

By contrast, the relay protocol's `Relayer` struct carries an explicit `RateLimiter<(PeerIndex, u32)>` keyed by peer and message type, initialized at 30 req/s: [10](#0-9) 

## Impact Explanation

A single peer can continuously send `GetTransactionsProof` messages with 1000 tx hashes each, forcing ~6000 direct DB reads plus O(1000 × log(chain_length)) MMR node reads per message, with no ban and no throttle. On a 1M-block chain this is ~26,000 RocksDB operations per message. Sustained flooding saturates RocksDB I/O, degrading block processing and peer service for the entire node. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The attack requires only: (1) a P2P connection to a node with the light client protocol enabled (no authentication), (2) 1000 valid on-chain tx hashes in distinct blocks (trivially obtained from public chain data, e.g., cellbase transactions), and (3) the current tip hash (publicly observable). No PoW, no key, and no privileged role is needed. The attack is fully repeatable with no cooldown.

## Recommendation

1. Add per-peer rate limiting to `LightClientProtocol::received()` mirroring the `RateLimiter<(PeerIndex, u32)>` used in `sync/src/relayer/mod.rs` lines 81–98.
2. Reduce `GET_TRANSACTIONS_PROOF_LIMIT` to a value that bounds worst-case DB work to an acceptable level (e.g., 100–200).
3. Consider banning peers that repeatedly send maximum-size requests in rapid succession by tracking request rate per peer and returning a 4xx status code to trigger the existing ban logic in `should_ban()`.

## Proof of Concept

```
1. Run a CKB full node with light client protocol enabled on a chain with ≥1000 blocks.
2. Collect 1000 tx hashes, one from each of 1000 distinct blocks (e.g., cellbase txs from
   blocks 1–1000, trivially obtained from the public chain).
3. Obtain the current tip hash (publicly observable).
4. Construct a GetTransactionsProof message:
     last_hash = <tip_hash>
     tx_hashes = [hash_0, hash_1, ..., hash_999]   // 1000 entries, 1000 distinct blocks
5. Send the message repeatedly in a tight loop from a single peer connection.
6. Monitor RocksDB read counters (rocksdb.number.db.get / iterator ops) — each message
   produces O(6000 + 1000×log(chain_len)) reads with no ban and no throttle.
7. Observe node CPU/IO saturation and degraded response times for other peers.
```
