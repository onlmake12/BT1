Audit Report

## Title
Off-by-one in `ConnectionRequestProcess::execute` allows attacker to cause legitimate relay nodes to be banned by their peers — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The two route-length guards in `execute()` both use strict `>` against `MAX_HOPS (6)`, so a crafted message with `route.len() = 6` and `max_hops = 6` passes both checks. `forward_request` then appends the relay node's own ID to the route, producing `route.len() = 7`. The next hop receives a message that fails its own `route.len() > 6` check, returns `InvalidRoute` (status 413), and bans the forwarding relay node for 24 hours. An attacker with a single P2P connection to a relay node can systematically cause all of that node's forwarding peers to ban it, effectively isolating it from the network.

## Finding Description
In `execute()` (`connection_request.rs` L120–124), the guards are:
```rust
if content.max_hops > MAX_HOPS { ... }              // 6 > 6 = false → passes
if content.route.len() > MAX_HOPS as usize { ... }  // 6 > 6 = false → passes
```
Since `max_hops != 0`, execution reaches `forward_message` → `forward_request` (`component/mod.rs` L171–187), which unconditionally pushes `current_id` onto the route (now length 7) and decrements `max_hops` to 5. The forwarded message violates the same `route.len() > MAX_HOPS` check at the next hop, which returns `InvalidRoute` (413). Because 413 is in the 4xx range, `should_ban()` returns `Some(BAD_MESSAGE_BAN_TIME)` (`status.rs` L99–105), and `ban_session` is called against the forwarding relay's session ID (`mod.rs` L150–155). The invariant `route.len() + max_hops <= MAX_HOPS` is never enforced; a legitimately-initialized message always satisfies it (starts at `route=[], max_hops=6`), but an attacker can violate it freely. The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` — all attacker-controlled — so it does not prevent repeated attacks using varied `(from, to)` pairs.

## Impact Explanation
An attacker can cause every peer that a relay node forwards to (Node B, C, D, …) to ban that relay node (Node A) for 24 hours. By repeating this across multiple relay nodes simultaneously, the attacker can fragment the CKB P2P topology, severing relay paths and degrading network connectivity. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (10001–15000 points). A single attacker connection per target relay is sufficient; no PoW, keys, or privileged role is required.

## Likelihood Explanation
The attacker needs only one standard P2P connection to the target relay node. The crafted message is trivially constructable: set `route` to 6 arbitrary peer IDs (none matching Node A's ID, to pass the loop-detection check at L128) and `max_hops = 6`. The attack is repeatable by cycling through different `(from, to)` pairs to bypass the `forward_rate_limiter`. The 24-hour ban duration means a single successful trigger per `(relay, peer)` pair is sufficient to sever that connection for a full day.

## Recommendation
Change the route-length guard to account for the pending append before forwarding:
```rust
// route will grow by 1 in forward_request; reject if it would exceed MAX_HOPS
if content.route.len() >= MAX_HOPS as usize {
    return StatusCode::InvalidRoute.with_context("route too long to forward");
}
```
This ensures that after `forward_request` appends `current_id`, the resulting `route.len()` is at most `MAX_HOPS`, which passes the next hop's check. Optionally, also enforce the combined invariant `content.route.len() + content.max_hops as usize <= MAX_HOPS as usize` to reject any message that violates the protocol's structural guarantee regardless of individual field values.

## Proof of Concept
```rust
// Craft a ConnectionRequest with route.len() = MAX_HOPS and max_hops = MAX_HOPS
let route: Vec<PeerId> = (0..MAX_HOPS as usize).map(|_| PeerId::random()).collect();
// Ensure none of the route IDs match Node A's local peer ID (trivially satisfied with random IDs)
let msg = packed::ConnectionRequest::new_builder()
    .from(attacker_id.as_bytes())
    .to(target_id.as_bytes())   // any peer ID not directly connected to Node A
    .max_hops(MAX_HOPS.into())
    .route(/* 6 random peer IDs */)
    .listen_addrs(/* ≥1 valid TCP listen addr */)
    .build();

// Send to Node A over the HolePunching protocol.
// Node A: 6 > 6 = false → passes; 6 > 6 = false → passes; max_hops != 0 → forward_message
// forward_request produces route.len()=7, max_hops=5
// Node B receives: route.len()=7 > MAX_HOPS=6 → InvalidRoute (413) → bans Node A for 24h

// Verify the forwarded message is structurally invalid:
let forwarded = forward_request(msg.as_reader(), &node_a_id);
assert!(forwarded.route().len() > MAX_HOPS as usize); // 7 > 6 — next hop will ban Node A
```
Repeat with different `(from, to)` pairs to exhaust all of Node A's forwarding peers within the rate-limiter window.
