Let me analyze the vulnerability class from the external report and search for analogous patterns in the CKB codebase.

The core vulnerability is: a **global accumulator is incremented** while a **per-entity state is overwritten (set)**, causing the global counter to become permanently inflated beyond what can be decremented, leading to a permanent DOS.

Let me look at the `start_sync_headers` function and the `n_sync_started` counter more carefully, and also check the relay protocol connection handler.

### Title
Non-Atomic `n_sync_started` Increment Causes Permanent IBD Stall — (`sync/src/synchronizer/mod.rs`)

### Summary

In `start_sync_headers`, the global atomic counter `n_sync_started` is incremented **before** the per-peer state flag `sync_started()` is set. If a peer disconnects in the window between these two non-atomic operations, `n_sync_started` is permanently inflated by 1. In IBD mode, this permanently prevents any new peer from starting header sync, stalling the node's Initial Block Download indefinitely.

### Finding Description

In `sync/src/synchronizer/mod.rs`, `start_sync_headers` performs two separate, non-atomic operations:

1. Atomically increment the global counter `n_sync_started`
2. Separately call `peer_state.start_sync()` to set the per-peer `headers_sync_state` to `Started` [1](#0-0) 

The decrement of `n_sync_started` only happens in `Peers::disconnected()` and `SyncState::suspend_sync()` / `SyncState::tip_synced()`, and only when `peer_state.sync_started()` returns `true`: [2](#0-1) 

`sync_started()` returns `true` only when `headers_sync_state == HeadersSyncState::Started`, which is set exclusively by `peer_state.start_sync()`: [3](#0-2) 

**Race window:** Between the `fetch_update` (step 1) and the `get_mut` + `start_sync` call (step 2), a peer disconnect event can fire concurrently. The network event handler calls `Peers::disconnected()`, which removes the peer from `self.state` and checks `sync_started()` — which is still `false` because `start_sync` has not yet been called. Therefore `n_sync_started` is **not decremented**. When control returns to `start_sync_headers`, `self.peers().state.get_mut(&peer)` returns `None` (peer already removed), so `start_sync` is never called. `n_sync_started` is now permanently inflated by 1. [4](#0-3) 

The global counter `n_sync_started` is used as a gate in IBD mode:

```rust
if ibd && x != 0 { None } else { Some(x + 1) }
``` [5](#0-4) 

Once `n_sync_started` is stuck at ≥ 1, the `fetch_update` always returns `Err` in IBD mode, and the loop immediately `break`s. No new peer can ever start header sync.

### Impact Explanation

When `n_sync_started` is permanently inflated to 1 (or higher) during IBD:

- The IBD guard `if ibd && x != 0 { None }` causes every subsequent `fetch_update` to fail.
- `start_sync_headers` breaks immediately on every invocation.
- No peer can ever be selected for header sync.
- The node is permanently stuck in IBD, unable to advance its chain tip.
- All downstream functionality gated on IBD completion (block relay, tx-pool, RPC chain state) is degraded.

The `n_sync_started` counter has no recovery path: there is no administrative reset, no periodic recomputation, and no fallback. Once inflated, it stays inflated for the lifetime of the process.

### Likelihood Explanation

The race window is the time between two consecutive lines of Rust code executing in an async task. While small, it is non-zero and exploitable:

- An unprivileged peer can connect to the node (triggering `sync_connected`) and immediately disconnect (triggering `disconnected`).
- The disconnect event is handled by the network layer's async event loop, which can interleave with the sync protocol's `start_sync_headers` task.
- `DashMap` operations are individually thread-safe but the two-step sequence is not atomic.
- An attacker can repeatedly connect and disconnect to maximize the probability of hitting the race window. With enough attempts, the race is won.

No special privileges, keys, or majority hashpower are required — only the ability to make and drop TCP connections to the node's P2P port.

### Recommendation

Make the increment of `n_sync_started` and the update of `peer_state.sync_started()` atomic with respect to peer removal. One approach:

```rust
// Acquire a write lock on the peer state FIRST, verify the peer still exists,
// then increment n_sync_started and call start_sync atomically within the same lock scope.
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    if self
        .shared()
        .state()
        .n_sync_started()
        .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
            if ibd && x != 0 { None } else { Some(x + 1) }
        })
        .is_err()
    {
        break;
    }
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
} else {
    // Peer already gone, do not increment n_sync_started
    continue;
}
```

This ensures that if the peer is already removed, `n_sync_started` is never incremented. Alternatively, `disconnected()` could unconditionally decrement `n_sync_started` if the peer was in the candidate list, but the lock-first approach is cleaner.

### Proof of Concept

**Scenario:**

1. Node is in IBD mode (`n_sync_started == 0`).
2. Attacker peer connects; `sync_connected` is called, peer is added to `Peers::state`.
3. `start_sync_headers` runs: peer passes `can_start_sync`, is added to the `peers` list.
4. `n_sync_started.fetch_update(...)` succeeds → `n_sync_started = 1`.
5. **Race:** Attacker peer disconnects. `Peers::disconnected()` fires concurrently:
   - Peer is removed from `self.state`.
   - `peer_state.sync_started()` → `false` (start_sync not yet called).
   - `n_sync_started` is **not** decremented. Still `1`.
6. Back in `start_sync_headers`: `self.peers().state.get_mut(&peer)` → `None`. `start_sync` is never called.
7. `n_sync_started` is permanently `1`.
8. All future calls to `start_sync_headers` in IBD mode: `fetch_update` returns `Err` immediately, loop breaks. No peer ever starts syncing.
9. Node is permanently stuck in IBD. [6](#0-5) [1](#0-0)
