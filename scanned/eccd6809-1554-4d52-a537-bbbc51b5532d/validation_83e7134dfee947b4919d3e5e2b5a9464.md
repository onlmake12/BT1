### Title
Expired Network Alert Signature Replay via P2P — (`util/network-alert/src/alert_relayer.rs`, `util/network-alert/src/notifier.rs`)

### Summary

CKB's Network Alert system signs alert content (id, message, `notice_until`, etc.) without any nonce. The deduplication guard (`has_received`) relies on an in-memory HashMap that is cleared when alerts expire. After expiry, the same signed alert bytes — captured by any P2P peer — can be replayed through the P2P `received` handler, which performs no `notice_until` check. The replayed alert passes signature verification, is re-accepted, and is re-broadcast across the entire P2P network.

### Finding Description

**Root cause — no nonce in signed data:**

`Verifier::verify_signatures` hashes only the `RawAlert` fields (id, cancel, message, notice_until, priority, min/max version) and verifies m-of-n ECDSA signatures against that hash. There is no nonce, sequence number, or one-time token in the signed payload. [1](#0-0) 

**Root cause — deduplication guard is cleared on expiry:**

`Notifier` tracks received alerts in a plain `HashMap<u32, Alert>`. `clear_expired_alerts` removes entries from this map (and from `noticed_alerts`) when `notice_until` passes. It does **not** add the expired IDs to `cancel_filter`. After clearing, `has_received(id)` returns `false` for the expired alert's ID. [2](#0-1) 

**Root cause — P2P handler skips `notice_until` check:**

The P2P `received` handler in `AlertRelayer` only checks `has_received(alert_id)` and then verifies signatures. It does **not** check whether `notice_until` is in the future. The RPC `send_alert` path does perform this check, but the P2P path does not. [3](#0-2)

**Exploit flow:**

1. A legitimate alert (ID=X, `notice_until=T`) is broadcast on the P2P network. Any connected peer captures the raw bytes (they are publicly relayed).
2. Time advances past `T`. `clear_expired_alerts()` is triggered on the next peer connection event, removing alert X from `received_alerts`. `has_received(X)` now returns `false`.
3. The attacker (any former peer that stored the bytes) reconnects and sends the original alert bytes via the Alert P2P protocol.
4. `has_received(X)` → `false` → proceeds to `verify_signatures` → **passes** (same bytes, same valid signatures, no nonce).
5. The alert is re-added to `received_alerts` and `noticed_alerts`, and re-broadcast to all connected peers via `quick_filter_broadcast`.
6. All nodes on the network re-display the expired alert message. [5](#0-4) 

### Impact Explanation

An unprivileged P2P peer who observed the original alert broadcast can replay it indefinitely after expiry. Each replay causes the expired alert to be re-displayed to all CKB node operators and re-broadcast network-wide. This can:

- Cause false alarms and confusion among node operators (e.g., replaying a "critical bug" alert after the bug is fixed).
- Be used to manipulate operator behavior (e.g., inducing unnecessary node restarts or upgrades).
- Trigger any configured `network_alert_notify_script` on every affected node. [6](#0-5) 

### Likelihood Explanation

Likelihood is **medium**. The attacker preconditions are minimal: any node that was connected during the original alert broadcast has the bytes. The alert bytes are public P2P messages. The only timing requirement is waiting for the alert to expire and for `clear_expired_alerts` to be called (triggered by any new peer connection). No privileged keys, no cryptographic break, and no special network position are required.

### Recommendation

1. **Add a `notice_until` check in the P2P `received` handler**, mirroring the check already present in `send_alert`:
   ```rust
   let notice_until: u64 = alert.as_reader().raw().notice_until().into();
   let now_ms = ckb_systemtime::unix_time_as_millis();
   if notice_until < now_ms {
       // drop silently or ban peer
       return;
   }
   ```
2. **Retain expired IDs in a persistent seen-set** (not just the LRU `cancel_filter`) so that `has_received` continues to return `true` for expired alert IDs after `clear_expired_alerts` runs.
3. Optionally, include a nonce or monotonic counter in the signed `RawAlert` payload to prevent any form of replay across alert ID reuse.

### Proof of Concept

```
# Setup: attacker is a CKB P2P peer that received alert ID=42 during its active period.
# The alert bytes are stored: alert_bytes = <raw packed::Alert bytes from P2P>

# Step 1: Wait for notice_until to pass (alert expires).
# Step 2: Connect to a target CKB node via the Alert P2P protocol (SupportProtocols::Alert).
# Step 3: Send alert_bytes directly.

# In alert_relayer.rs::received():
#   alert_id = 42
#   has_received(42) -> false  (cleared by clear_expired_alerts)
#   verify_signatures(&alert) -> Ok(())  (same valid signatures, no nonce)
#   -> alert re-broadcast to all peers
#   -> notifier.add(&alert) -> noticed_alerts updated, notify_network_alert fired
#   -> all connected nodes re-display the expired alert
```

The attacker needs only a standard CKB P2P connection and the previously observed alert bytes — both trivially available to any former network participant.

    pub fn clear_expired_alerts(&mut self, now: u64) {
        self.received_alerts.retain(|_id, alert| {

            notice_until > now
        });
