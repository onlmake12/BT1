### Title
Unauthenticated notify endpoint accepts zero `compact_target`, causing miner to spin indefinitely — (`miner/src/miner.rs`)

### Summary
The miner's HTTP notify listener accepts `BlockTemplate` payloads from any unauthenticated sender. When a template with `compact_target = 0x00000000` is injected, `compact_to_target(0)` returns `(U256::zero(), false)`. The `notify_new_work` function discards the overflow flag and performs no zero-target check before dispatching `WorkerMessage::NewWork { target: U256::zero() }` to all worker threads. The `EaglesongSimple` worker's `solve()` loop then spins forever because `output <= U256::zero()` is never satisfied by any real hash output.

### Finding Description
`notify_new_work` in `miner/src/miner.rs` calls `compact_to_target` and silently discards both the overflow flag and any zero-target result:

```rust
let (target, _) = compact_to_target(work.block.header().raw().compact_target().into());
self.notify_workers(WorkerMessage::NewWork { pow_hash, work, target });
```

The `_` means a zero or overflowing target is forwarded directly to workers. The `EaglesongPowEngine::verify()` in `pow/src/eaglesong.rs` has the correct guard (`if block_target.is_zero() || overflow { return false; }`), but this guard was never replicated in the miner's work-dispatch path.

### Impact Explanation
Workers enter an infinite busy-loop hashing against an impossible target. No nonce is ever found, no block is ever submitted, and the miner produces zero output for as long as the attacker keeps injecting malicious templates. Because `update_block_template` accepts any template whose `work_id` differs from the current one (or is 0), the attacker can continuously re-inject to prevent recovery even when the CKB node sends a legitimate notification.

### Likelihood Explanation
Requires notify mode to be enabled (`listen` in `ckb-miner.toml`) and the listen socket to be reachable. The default config has `listen` commented out, but the feature is documented and intended for production mining pools. Any process or host that can reach the bound address — including other processes on the same machine or any host on the LAN if bound to `0.0.0.0` — can trigger this with a single HTTP POST.

### Recommendation
In `notify_new_work`, mirror the guard already present in `EaglesongPowEngine::verify()`:

```rust
fn notify_new_work(&mut self, work: Work) {
    let parent_hash = work.block.header().into_view().parent_hash();
    if !self.legacy_work.contains(&parent_hash) {
        let pow_hash = work.block.header().calc_pow_hash();
        let (target, overflow) =
            compact_to_target(work.block.header().raw().compact_target().into());
        if target.is_zero() || overflow {
            error!("received work with invalid compact_target, ignoring");
            return;
        }
        self.notify_workers(WorkerMessage::NewWork { pow_hash, work, target });
    }
}
```

Additionally, consider adding IP allowlist or shared-secret authentication to the notify HTTP listener.

pub fn compact_to_target(compact: u32) -> (U256, bool) {
    let exponent = compact >> 24;
    let mut mantissa = U256::from(compact & 0x00ff_ffff);

    let mut ret;
    if exponent <= 3 {
        mantissa >>= 8 * (3 - exponent);
        ret = mantissa.clone();
    } else {
        ret = mantissa.clone();
        ret <<= 8 * (exponent - 3);
    }

    let overflow = !mantissa.is_zero() && (exponent > 32);
    (ret, overflow)
}
