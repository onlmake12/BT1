### Title
Minimum Fee Threshold Ignores Cycles-Based Weight Dimension, Allowing Under-Priced High-Cycle Transactions Into the Tx-Pool — (File: `tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` calculates the minimum required fee using only the transaction's serialized byte size, ignoring the cycles dimension of the transaction weight. Because CKB's actual weight is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`, a transaction with small serialized size but high cycle consumption can pass the fee gate while paying far less than the true minimum fee rate requires. No subsequent fee check using the actual weight is performed after cycles are determined.

---

### Finding Description

`FeeRate::fee` computes a minimum fee as `fee_rate * weight / 1000`: [1](#0-0) 

The canonical transaction weight is defined as: [2](#0-1) 

`DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`, so a transaction consuming 70,000,000 cycles (the default `max_tx_verify_cycles`) has a cycle-derived weight of approximately **11,940 bytes**, regardless of its serialized size.

`check_tx_fee` computes the minimum fee using only `tx_size`, not `get_transaction_weight(tx_size, cycles)`: [3](#0-2) 

The code comment explicitly acknowledges this is an approximation ("here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly"), but there is **no subsequent enforcement** using the actual weight after cycles are known from `verify_rtx`. The fee value returned from `check_tx_fee` is accepted as final: [4](#0-3) 

---

### Impact Explanation

An attacker can craft a transaction with:
- Serialized size: ~200 bytes (small)
- Cycles consumed: ~70,000,000 (near `max_tx_verify_cycles`)
- Fee paid: `min_fee_rate * 200 / 1000` = **200 shannons** (at default 1,000 shannons/KB)

The actual minimum fee required by the weight formula would be:
- `min_fee_rate * max(200, 70_000_000 * 0.0001705714) / 1000` = `1000 * 11940 / 1000` = **11,940 shannons**

The attacker pays ~**60× less** than the true minimum. Such transactions:
1. Pass `check_tx_fee` and enter the pool
2. Consume maximum verification resources (cycles) at minimal cost
3. Are relayed to peers, propagating the under-priced load across the network
4. Can crowd out legitimately-priced transactions from the bounded pool

---

### Likelihood Explanation

The attack is reachable by any unprivileged RPC caller via `send_transaction` or `test_tx_pool_accept`. No special role, key, or configuration is required. The default `min_fee_rate = 1_000` shannons/KB and `max_tx_verify_cycles = 70_000_000` are both set in the shipped default config: [5](#0-4) 

The discrepancy is maximized when cycles dominate weight, which is achievable with any script that runs close to the cycle limit.

---

### Recommendation

After `verify_rtx` returns the actual consumed cycles, perform a second fee rate check using the true weight:

```rust
let actual_weight = get_transaction_weight(tx_size, completed.cycles);
let actual_min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This mirrors the pattern used in fee rate statistics collection: [6](#0-5) 

---

### Proof of Concept

1. Construct a CKB transaction with a lock script that loops near 70,000,000 cycles but has minimal witness/output data (serialized size ~200 bytes).
2. Set the output capacity so that `inputs_capacity - outputs_capacity = 200 shannons` (fee = 200 shannons).
3. Submit via `send_transaction` RPC to a node with default `min_fee_rate = 1000`.
4. `check_tx_fee` computes `min_fee = 1000 * 200 / 1000 = 200 shannons`; the fee equals the threshold, so the transaction is accepted.
5. The actual weight is `max(200, 70_000_000 * 0.0001705714) ≈ 11,940`; the true minimum fee should be `11,940 shannons`, meaning the transaction is accepted at ~60× below the intended minimum fee rate.
6. The node expends full cycle-verification resources and relays the transaction to peers.
