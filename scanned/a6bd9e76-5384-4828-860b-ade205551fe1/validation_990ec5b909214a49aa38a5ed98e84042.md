### Title
Systematic Truncation Bias in NervosDAO Accumulate-Rate (`ar`) Computation Causes Depositors to Receive Less Interest Than Entitled — (`File: util/dao/src/lib.rs`)

---

### Summary

Every block, the NervosDAO accumulate rate (`ar`) is updated using integer floor division. Because Rust's `/` operator on integers always truncates toward zero, each block discards the fractional remainder of the division. This introduces a systematic, always-downward bias that compounds across every block of the chain. When a depositor withdraws from NervosDAO, their interest is computed as `counted_capacity * withdrawing_ar / deposit_ar`; a suppressed `withdrawing_ar` means every depositor receives strictly less interest than the mathematically correct amount. The bias is one-directional (always downward) and grows with chain age and deposit duration, directly analogous to the DSMath "round half up" systematic bias described in the external report.

---

### Finding Description

The NervosDAO accumulate rate `ar` is stored in the DAO field of every block header and is updated in `dao_field_with_current_epoch` inside `util/dao/src/lib.rs`:

```rust
// util/dao/src/lib.rs lines 256-258
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [1](#0-0) 

The division `parent_ar * current_g2 / parent_c` is integer floor division. The exact mathematical result is `(parent_ar × current_g2) / parent_c`, but the fractional part `(parent_ar × current_g2) mod parent_c` is silently discarded every block. The genesis `ar` is initialized to `10^16`:

```rust
// util/dao/utils/src/lib.rs line 17
const DEFAULT_GENESIS_ACCUMULATE_RATE: u64 = 10_000_000_000_000_000;
``` [2](#0-1) 

A second truncation occurs at withdrawal time in `calculate_maximum_withdraw`:

```rust
// util/dao/src/lib.rs lines 152-154
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
``` [3](#0-2) 

Both truncations always round downward. There is no compensating upward rounding anywhere in the pipeline. The `miner_issuance128` computation in the same function suffers the same pattern:

```rust
// util/dao/src/lib.rs lines 242-243
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
``` [4](#0-3) 

The `safe_mul_ratio` helper used for proposer reward splits also truncates:

``` [5](#0-4) 

The reward calculator explicitly notes the rounding asymmetry but does not correct it:

### Impact Explanation

The `ar` accumulate rate is the sole parameter governing NervosDAO interest. Every block's truncation discards up to `parent_c - 1` units of the numerator. With `parent_c` (total CKB capacity) on the order of `3.36 × 10^27` shannons and `ar` on the order of `10^16`, the per-block truncation error in `ar_increase128` is bounded by 1 unit in the u128 space. Over N blocks, the accumulated downward error in `ar` is at most N units.

For a depositor with `counted_capacity = D` shannons depositing for N blocks:

```
interest_error ≤ D × N / deposit_ar
               = D × N / 10^16
```

For D = 10^17 shannons (1 billion CKB) and N = 10^6 blocks (~3 years):

```
interest_error ≤ 10^17 × 10^6 / 10^16 = 10^7 shannons ≈ 0.1 CKB
```

This is a systematic, one-directional loss: depositors always receive less than the mathematically correct interest. The error is proportional to both deposit size and duration, and it never self-corrects because every block truncates in the same direction. The `withdraw_counted_capacity` truncation at withdrawal time adds a further one-time downward error of up to 1 shannon per withdrawal.

The `ar` value is consensus-critical: it is stored in the DAO field of every block header and must be reproduced identically by all nodes. Any change to the rounding strategy would require a hard fork.

---

### Likelihood Explanation

This affects every NervosDAO depositor on every withdrawal, unconditionally. The entry path requires only submitting a standard NervosDAO deposit transaction (an unprivileged transaction sender action) and later withdrawing. The `calculate_dao_maximum_withdraw` RPC endpoint exposes the truncated result directly to any RPC caller. The bias is not probabilistic — it fires on every block where `(parent_ar × current_g2) mod parent_c ≠ 0`, which is the overwhelming majority of blocks given the large, non-round values of `parent_c`.

---

### Recommendation

Replace the single-step floor division with a rounding-aware alternative. The simplest correct fix is **round-half-up** for `ar_increase128`:

```rust
// Round half up: (a * b + c/2) / c
let ar_increase128 = (u128::from(parent_ar) * u128::from(current_g2.as_u64())
    + u128::from(parent_c.as_u64()) / 2)
    / u128::from(parent_c.as_u64());
```

Alternatively, adopt **banker's rounding** (round-half-to-even) as recommended in the external report, which eliminates systematic bias when the distribution of remainders is roughly uniform. Apply the same correction to `withdraw_counted_capacity` and `miner_issuance128`. Because `ar` is consensus-critical state, any change requires a coordinated hard fork with a defined activation epoch.

---

### Proof of Concept

**Concrete numeric example for `ar_increase128`:**

Suppose at some block:
- `parent_ar = 10_000_000_000_000_000` (genesis value, `10^16`)
- `current_g2 = 1_000_000` shannons (secondary issuance for this block)
- `parent_c = 3_000_000_000_000_000_000` shannons (3 × 10^18, plausible total capacity)

Exact result: `10^16 × 10^6 / (3 × 10^18) = 10^22 / (3 × 10^18) = 3333.333...`

Truncated result: `3333`

Error per block: `0.333...` units of `ar`.

After 1,000,000 blocks: accumulated `ar` error ≈ `333,333` units.

For a depositor with `counted_capacity = 10^17` shannons (1 billion CKB):

```
interest_error = 10^17 × 333_333 / 10^16 = 3_333_330 shannons ≈ 0.033 CKB
```

This depositor receives ~0.033 CKB less than entitled, with no recourse. The error scales linearly with deposit size and duration. A whale depositing 100 billion CKB for 3 years would lose ~3.3 CKB to systematic truncation bias alone — before the additional `withdraw_counted_capacity` truncation at withdrawal time.
