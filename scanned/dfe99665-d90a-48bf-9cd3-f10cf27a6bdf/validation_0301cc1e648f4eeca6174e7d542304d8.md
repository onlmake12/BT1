Looking at the specific code paths involved in this claim, I need to trace: (1) whether `add_addr` validates peer_id, (2) whether `fetch_addrs_to_feeler` truly silences addresses without peer_id, (3) whether the discovery protocol allows injecting such addresses, and (4) whether eviction prevents the attack.

Based on my thorough code analysis, here is my assessment:

---

### Title
Peer Store Pollution via Peer-ID-less Addresses Injected Through Discovery — (`network/src/peer_store/peer_store_impl.rs`, `network/src/protocols/discovery/mod.rs`)

### Summary

`add_addr` accepts any `Multiaddr` without validating the presence of a `/p2p/` component. The discovery protocol's `add_new_addrs` path also performs no such check. Meanwhile, every fetch function (`fetch_addrs_to_feeler`, `fetch_addrs_to_attempt`) uses `extract_peer_id(...).unwrap_or_default()` which silently returns `false` for peer-ID-less addresses, permanently excluding them from dial candidates. Because these addresses are connectable by `is_connectable` (attempts_count = 0 < ADDR_MAX_RETRIES = 3) they are never evicted by `check_purge`. An unprivileged remote peer can therefore fill all 16 384 slots with permanently-inert addresses, starving the store of dialable peers.

### Finding Description

**`add_addr` — no peer-ID validation:** [1](#0-0) 

The function only checks the ban list and calls `check_purge`. There is no guard requiring the address to contain a `/p2p/` component.

**`fetch_addrs_to_feeler` — silent exclusion via `unwrap_or_default`:** [2](#0-1) 

`extract_peer_id` returns `None` for addresses without `/p2p/`. `unwrap_or_default()` on `Option<bool>` yields `false`, making the entire filter predicate `false`. The address is silently skipped — not an error, not logged, just dropped from results.

The identical pattern appears in `fetch_addrs_to_attempt`: [3](#0-2) 

**Discovery protocol — no peer-ID validation in the receive path:** [4](#0-3) 

`is_valid_addr` only checks IP reachability, not peer-ID presence. The raw `Multiaddr` bytes from the remote peer are decoded and passed directly to `add_addr`: [5](#0-4) 

No validation of `/p2p/` presence occurs during decode.

**`check_purge` — eviction does not remove peer-ID-less addresses:** [6](#0-5) 

Pass 1 evicts addresses where `is_connectable` returns `false`. A freshly injected address has `attempts_count = 0 < ADDR_MAX_RETRIES (3)` and `last_connected_at_ms = 0`, so `is_connectable` returns `true` — it is never evicted. Pass 2 evicts from network groups with >4 peers; an attacker using addresses from many distinct /16 subnets defeats this. The result is `Err(PeerStoreError::EvictionFailed)`, blocking all future legitimate `add_addr` calls. [7](#0-6) 

**`ADDR_COUNT_LIMIT`:** [8](#0-7) 

The limit is 16 384. A single non-announce `Nodes` message carries up to 1 000 items × 3 addresses = 3 000 addresses. Six connections suffice to exceed the limit.

**`base_addr` does not strip `/p2p/`:** [9](#0-8) 

Only `Ws`, `Wss`, `Memory`, and `Tls` are stripped. Addresses without `/p2p/` are stored verbatim and keyed as-is in `addr_to_id`.

### Impact Explanation

Once the store is saturated with peer-ID-less addresses:
- `fetch_addrs_to_feeler` returns an empty list — the feeler service dials nothing.
- `fetch_addrs_to_attempt` returns an empty list — no new outbound connections are initiated.
- `fetch_random_addrs` returns nothing useful (these addresses were never connected to, so `last_connected_at_ms = 0`).
- `check_purge` returns `Err` on every subsequent `add_addr` call, so no legitimate address can enter the store.
- The peer store is persisted to disk (`load_from_dir_or_default`), making the condition survive node restarts.

The node retains inbound connections and hardcoded bootnodes but loses all autonomous outbound peer discovery.

### Likelihood Explanation

The attack requires only a standard P2P connection and the ability to send a crafted `Nodes` discovery message — no authentication, no special role. Six connections each sending one non-announce `Nodes` message (3 000 addresses each) saturate the store. The `received_nodes` flag prevents a second non-announce message per session, but six separate connections are trivially established. The attack is persistent across restarts.

### Recommendation

1. **In `add_addr`**: reject any `Multiaddr` for which `extract_peer_id` returns `None`.
2. **In `add_new_addrs` (discovery)**: filter out addresses without a `/p2p/` component before calling `add_addr`.
3. **In `check_purge`**: treat peer-ID-less addresses as non-connectable so they are evicted in pass 1.

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
        Ok(())
    }

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
