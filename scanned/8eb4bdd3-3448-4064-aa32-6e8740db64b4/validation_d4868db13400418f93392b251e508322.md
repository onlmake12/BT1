## Title
Premature `add_anchors` Before Identify Completes Allows Attacker-Controlled Address Persistence in `anchors.db` — (`network/src/peer_registry.rs`)

## Summary
`accept_peer` unconditionally calls `peer_store.add_anchors` at TCP session-open time, before the identify protocol runs. No code path exists to remove an address from the in-memory `Anchors` set after it is added. On shutdown, anchors are flushed to `anchors.db`; on restart, every anchored address is dialed unconditionally. An attacker who triggers a `SessionOpen` event while the victim's regular outbound slots are full can permanently occupy one or both block-relay-only anchor slots across every subsequent node restart.

## Finding Description

**Root cause — `add_anchors` at session open, before identify:**
In `accept_peer`, when `non_whitelist_outbound >= max_outbound` and a block-relay-only slot is available, line 130 calls `peer_store.add_anchors(remote_addr.clone())` and sets `session_type = SessionType::BlockRelayOnly`. This executes inside the `ServiceEvent::SessionOpen` handler — the identify protocol has not yet run. [1](#0-0) 

**No removal path — `Anchors` has no `remove` method:**
The `Anchors` struct exposes only `add`, `count`, `dump_iter`, `drain`, and `contains`. A codebase-wide search for any call matching `remove.*anchor` or `anchors.remove` returns zero results. Once an address is inserted, it can only leave via `drain()` (called once at startup to consume the set for dialing). [2](#0-1) 

**`SessionClose` does not touch anchors:**
The `SessionClose` handler calls `peer_store.remove_disconnected_peer`, which only removes the peer from the `connected_peers: HashMap<PeerId, PeerInfo>`. The `anchors` field is untouched. [3](#0-2) [4](#0-3) 

**`ban_addr` also does not remove from anchors:**
When identify fails (wrong network, flag mismatch), `ban_addr` is called. It removes the address from `addr_manager` but leaves `anchors` intact. [5](#0-4) 

**Anchors flushed to disk on shutdown:**
`dump_to_dir` serializes the in-memory `Anchors` (up to `MAX_OUTBOUND_BLOCK_RELAY = 2` entries) to `anchors.db`. [6](#0-5) 

**Anchors dialed unconditionally on restart:**
`NetworkService::start` drains the loaded anchors and dials every address via `dial_identify` before any handshake validation. [7](#0-6) 

## Impact Explanation
An attacker who establishes a single TCP connection that reaches `SessionOpen` while the victim's regular outbound slots are full gets their address written into `anchors.db`. Even if the connection is immediately dropped (identify timeout, flag mismatch, or deliberate disconnect), the anchor entry survives in memory and is flushed to disk on shutdown. On every subsequent restart, the victim unconditionally dials the attacker's address. With `MAX_OUTBOUND_BLOCK_RELAY = 2`, an attacker controlling two addresses can fill both anchor slots, persistently occupying the victim's block-relay-only outbound connections across restarts. This constitutes a persistent forced-reconnection primitive that contributes to a long-term eclipse of the victim node, matching the allowed impact: **vulnerabilities or bad designs which could cause CKB network congestion / eclipse with few costs (High, 10001–15000 points)**. [8](#0-7) 

## Likelihood Explanation
Preconditions are realistic on mainnet: a victim node with all regular outbound slots occupied is common under normal load. The attacker only needs to be discoverable via standard P2P gossip and reachable at the TCP level. No privileged access, no majority hashpower, and no social engineering is required. A single TCP handshake that reaches `SessionOpen` is sufficient to plant the anchor. The attack is repeatable across restarts with zero ongoing cost once the anchor is written.

## Recommendation
1. Add a `remove(&mut self, addr: &Multiaddr)` method to `Anchors` in `network/src/peer_store/anchors.rs`.
2. In the `SessionClose` handler (or in `remove_disconnected_peer`), call `anchors.remove(&addr)` for any peer whose session closes before identify completes (i.e., before `peer.identify_info` is set).
3. Alternatively, defer `add_anchors` until `received_identify` succeeds and the peer is confirmed to be on the correct network with the required flags. This is the cleaner fix as it aligns anchor promotion with actual peer validation.

## Proof of Concept
```rust
// Minimal unit test demonstrating the invariant violation
let mut registry = PeerRegistry::new(2, 2, false, vec![], false);
let mut peer_store = PeerStore::default();

// Fill regular outbound slots
for i in 0..2u8 {
    registry.accept_peer(
        make_outbound_addr(i), SessionId::new(i as usize),
        RawSessionType::Outbound, &mut peer_store
    ).unwrap();
}

// Attacker triggers SessionOpen → add_anchors fires
let attacker_addr = make_outbound_addr(99);
registry.accept_peer(
    attacker_addr.clone(), SessionId::new(99),
    RawSessionType::Outbound, &mut peer_store
).unwrap();

// Identify fails → session closed (simulated)
registry.remove_peer(SessionId::new(99));
peer_store.remove_disconnected_peer(&attacker_addr);

// Anchor still present — no removal path exists
assert!(peer_store.anchors().contains(&attacker_addr)); // PASSES

// On shutdown: anchors.db written with attacker_addr
// On restart:  attacker_addr dialed unconditionally via dial_identify
```
The `assert!` passes because `remove_disconnected_peer` only touches `connected_peers`, and `Anchors` has no `remove` method, confirming the anchor persists through session close.
