Audit Report

## Title
Unbounded O(n×m) Nested Loop in `add_ask_for_txs` Triggered by Malicious Peer Flooding `RelayTransactionHashes` - (File: sync/src/types/mod.rs)

## Summary

In `sync/src/types/mod.rs`, the function `add_ask_for_txs` contains a nested loop that scans every entry in the `unknown_tx_hashes` queue and every peer stored per entry whenever the queue reaches capacity. The `push_peer` method appends peer indices without deduplication, so a single attacker peer can grow each entry's `peers` vector without bound by repeatedly re-announcing the same hashes. This causes the nested scan to become progressively more expensive on every subsequent message, blocking the `unknown_tx_hashes` mutex and degrading relay throughput for all peers.

## Finding Description

**Root cause — unbounded `peers` vector growth:**

`UnknownTxHashPriority::push_peer` unconditionally appends without deduplication:

```rust
pub fn push_peer(&mut self, peer_index: PeerIndex) {
    self.peers.push(peer_index);   // no contains() check, no dedup
}
``` [1](#0-0) 

The `Occupied` branch in `add_ask_for_txs` calls this on every re-announcement of an already-queued hash: [2](#0-1) 

The `tx_filter` in `TransactionHashesProcess::execute` only filters *known* (confirmed) transactions; hashes sitting in `unknown_tx_hashes` are not in `tx_filter`, so the same hash can be re-announced indefinitely and the attacker's `PeerIndex` is pushed again on every message. [3](#0-2) 

**Trigger — nested scan under capacity condition:**

Once `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000) or `>= peers * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, every call to `add_ask_for_txs` executes:

```rust
for (_hash, priority) in unknown_tx_hashes.iter() {   // O(50 000)
    for peer in priority.peers.iter() {                // O(k) — grows per message
        if *peer == peer_index {
            peer_unknown_counter += 1;
        }
    }
}
``` [4](#0-3) 

After `k` attacker messages re-announcing the same 32,767 hashes, each of those entries has `k` copies of the attacker's `PeerIndex`. The scan cost grows as O(50,000 × k) — unbounded over time, not merely O(n × connected_peers) as the claim states.

**Constants:**
- `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000
- `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = `MAX_RELAY_TXS_NUM_PER_BATCH` = 32,767 [5](#0-4) 

**Exploit flow:**
1. Attacker connects as a normal relay peer (no privilege required).
2. Sends two `RelayTransactionHashes` messages each containing 32,767 fresh hashes → queue reaches 50,000 entries, capacity condition becomes permanently true.
3. Attacker repeatedly re-sends the same 32,767 hashes. Each message: (a) pushes attacker's `PeerIndex` onto 32,767 entries' `peers` vectors via the `Occupied` branch, then (b) triggers the full nested scan, which now iterates over 50,000 × k entries.
4. The scan holds the `unknown_tx_hashes` mutex for its entire duration, blocking `pop_ask_for_txs`, `mark_as_known_txs`, and all concurrent relay message processing.

**Existing guard is insufficient:** The `TooManyUnknownTransactions` ban at line 1524–1526 only fires after the scan completes and only if the per-peer counter exceeds the threshold. The scan itself is the attack surface; banning the peer afterward does not undo the CPU time already consumed, and the attacker can reconnect. [6](#0-5) 

## Impact Explanation

**Impact: High — CKB network congestion with few costs.**

The `unknown_tx_hashes` mutex is shared across all relay message processing. Sustained O(n×k) scans on every incoming `RelayTransactionHashes` message stall compact block processing, transaction relay, and block propagation for all peers connected to the targeted node. A node that cannot process compact blocks in time falls back to full block download, increasing block propagation latency across the network. A single attacker with one connection and cheap 32-byte hash messages can cause this degradation, matching the "CKB network congestion with few costs" High-severity criterion.

## Likelihood Explanation

Any unprivileged peer can connect and send `RelayTransactionHashes` messages. No keys, hashpower, or special knowledge are required. The queue fills in two messages. Thereafter, each re-announcement message is ~1 KB (32,767 × 32 bytes) and forces an increasingly expensive mutex-held scan. The attacker can sustain the attack indefinitely from a single connection at a moderate send rate.

## Recommendation

1. **Deduplicate on insert:** In `push_peer`, check `self.peers.contains(&peer_index)` before pushing, or use a `HashSet<PeerIndex>` instead of `Vec<PeerIndex>`. This bounds the inner vector to the number of distinct connected peers and eliminates unbounded growth.
2. **Replace the linear scan with an O(1) side-map:** Maintain a `HashMap<PeerIndex, usize>` counter alongside `unknown_tx_hashes`, incremented on insert and decremented on eviction. The capacity check then becomes a single map lookup, eliminating the nested loop entirely.
3. **Move the capacity check before insertion:** Check and reject before inserting new hashes so the mutex is not held during a full scan.

## Proof of Concept

1. Connect to a CKB node as a `RelayV3` peer.
2. Send two `RelayTransactionHashes` messages each containing 32,767 distinct random 32-byte hashes to fill the queue to `MAX_UNKNOWN_TX_HASHES_SIZE`.
3. Repeatedly re-send the same 32,767 hashes. After `k` repetitions, each queue entry's `peers` vector contains `k` copies of the attacker's `PeerIndex`.
4. Measure relay message processing latency (e.g., compact block round-trip time) — it increases proportionally to `k`.
5. The existing integration test at `test/src/specs/relay/too_many_unknown_transactions.rs` confirms the queue-filling path; extend it to re-send the same hashes in a loop and observe the growing scan time. [7](#0-6)

use crate::util::cell::gen_spendable;
use crate::util::transaction::always_success_transaction;
use crate::utils::{build_relay_tx_hashes, since_from_absolute_timestamp, wait_until};
use crate::{Net, Node, Spec};
use ckb_constant::sync::{MAX_RELAY_TXS_NUM_PER_BATCH, MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER};
use ckb_network::SupportProtocols;
use ckb_types::packed::CellInput;

pub struct TooManyUnknownTransactions;

impl Spec for TooManyUnknownTransactions {
    fn run(&self, nodes: &mut Vec<Node>) {
        let node0 = &nodes[0];
        let mut net = Net::new(
            self.name(),
            node0.consensus(),
            vec![SupportProtocols::Sync, SupportProtocols::RelayV3],
        );
        net.connect(node0);

        // Send `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` transactions with a same input
        let input = gen_spendable(node0, 1)[0].to_owned();
        let tx_template = always_success_transaction(node0, &input);
        let txs = {
            (0..MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER).map(|i| {
                let since = since_from_absolute_timestamp(i as u64);
                tx_template
                    .as_advanced_builder()
                    .set_inputs(vec![CellInput::new(input.out_point.clone(), since)])
                    .build()
            })
        };
        let tx_hashes = txs.map(|tx| tx.hash()).collect::<Vec<_>>();
        assert!(MAX_RELAY_TXS_NUM_PER_BATCH >= tx_hashes.len());
        net.send(
            node0,
            SupportProtocols::RelayV3,
            build_relay_tx_hashes(&tx_hashes),
        );

        let banned = wait_until(60, || node0.rpc_client().get_banned_addresses().len() == 1);
        assert!(
            banned,
            "NetController should be banned cause TooManyUnknownTransactions"
        );
    }
