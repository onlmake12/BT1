### Title
Under-constrained `genesis_epoch_length` and `orphan_rate_target` Consensus Parameters Allow Node Crash via Division-by-Zero - (File: spec/src/consensus.rs)

### Summary

The `build_genesis_epoch_ext` function in `spec/src/consensus.rs` performs integer division by `genesis_epoch_length` and by `genesis_orphan_rate.1` (the denominator of `orphan_rate_target`) without any zero-check. Both values flow directly from the user-controlled chain spec TOML file (`[params]` section) with no validation before use. Setting either to zero causes an immediate integer division-by-zero panic that crashes the node process at startup. Additionally, `orphan_rate_target` is stored as a `RationalU256::new_raw` (bypassing the zero-denominator check in `RationalU256::new`) and is later used as a divisor in `next_epoch_ext`, which can produce a divide-by-zero panic during live epoch transitions.

### Finding Description

**Root cause 1 — `genesis_epoch_length = 0` in chain spec TOML**

`build_genesis_epoch_ext` is called from `ChainSpec::build_consensus` with the value returned by `self.params.genesis_epoch_length()`, which is the raw TOML-supplied value with no lower-bound check:

```rust
// spec/src/consensus.rs:222-223
let block_reward = Capacity::shannons(epoch_reward.as_u64() / genesis_epoch_length);
let remainder_reward = Capacity::shannons(epoch_reward.as_u64() % genesis_epoch_length);
```

If `genesis_epoch_length = 0` is written in the spec file, both lines panic with integer division by zero at node startup.

**Root cause 2 — `orphan_rate_target = [N, 0]` in chain spec TOML**

The denominator of `orphan_rate_target` is used as a divisor in `build_genesis_epoch_ext`:

```rust
// spec/src/consensus.rs:225-226
let genesis_orphan_count =
    genesis_epoch_length * genesis_orphan_rate.0 as u64 / genesis_orphan_rate.1 as u64;
```

A zero denominator causes a panic here. Furthermore, `ConsensusBuilder::orphan_rate_target` stores the pair via `RationalU256::new_raw`, which explicitly skips the zero-denominator guard present in `RationalU256::new`:

```rust
// spec/src/consensus.rs:387-392
pub fn orphan_rate_target(mut self, orphan_rate_target: (u32, u32)) -> Self {
    self.inner.orphan_rate_target = RationalU256::new_raw(
        U256::from(orphan_rate_target.0),
        U256::from(orphan_rate_target.1),
    );
    self
}
```

This zero-denominator `RationalU256` is then used as a multiplicative factor in `next_epoch_ext` during epoch boundary processing, where `RationalU256::into_u256` calls `self.numer / self.denom` — a second division-by-zero panic path reachable on every epoch transition.

**Root cause 3 — `epoch_duration_target = 0`**

`build_genesis_epoch_ext` also divides by `epoch_duration_target`:

```rust
// spec/src/consensus.rs:227-229
let genesis_hash_rate = compact_to_difficulty(compact_target)
    * (genesis_epoch_length + genesis_orphan_count)
    / epoch_duration_target;
```

The only guard for this is a `debug_assert!` in `ConsensusBuilder::build`, which is compiled away in release builds:

```rust
// spec/src/consensus.rs:342-345
debug_assert!(
    self.inner.epoch_duration_target() != 0,
    "epoch_duration_target must be non-zero"
);
```

In a release build, `epoch_duration_target = 0` causes a panic in `build_genesis_epoch_ext` before the assert is even reached.

**Entry path:** A node operator (supported local CLI/RPC user) writes a custom chain spec TOML for a dev/private network and sets any of `genesis_epoch_length = 0`, `orphan_rate_target = [1, 0]`, or `epoch_duration_target = 0`. The node crashes immediately on startup. For `orphan_rate_target` with a zero denominator that survives startup (e.g., numerator also zero making the genesis calculation not divide), the panic is deferred to the first epoch boundary, crashing the running node.

### Impact Explanation

- **Crash / DoS of the node process** at startup or at the first epoch boundary.
- Any operator of a dev or private CKB network who misconfigures these parameters will have a non-functional node with no clear error message — only a panic.
- The `genesis_epoch_length` and `epoch_duration_target` panics are immediate and deterministic. The `orphan_rate_target` zero-denominator panic is deferred to the first epoch transition.
- The `debug_assert!` guards are stripped in release builds, providing no protection in production binaries.

### Likelihood Explanation

The `[params]` section of the chain spec TOML is explicitly documented as user-configurable. The `genesis_epoch_length`, `epoch_duration_target`, and `orphan_rate_target` fields are all `Option<T>` with no validation layer between deserialization and use. A misconfiguration — accidental or otherwise — by any node operator running a dev or private chain triggers the crash. The `orphan_rate_target` denominator path is particularly subtle because `new_raw` is used instead of `new`, silently bypassing the only existing guard.

### Recommendation

1. **Validate all divisor parameters before use** in `build_genesis_epoch_ext` and `build_consensus`. Return a `Result` error (not a panic) when `genesis_epoch_length == 0`, `epoch_duration_target == 0`, or `orphan_rate_target.1 == 0`.
2. **Replace `debug_assert!` with `assert!`** (or proper `Result`-returning validation) for `epoch_duration_target != 0` and `genesis_epoch_length != 0` in `ConsensusBuilder::build` so the check is not stripped in release builds.
3. **Use `RationalU256::new` instead of `RationalU256::new_raw`** in `ConsensusBuilder::orphan_rate_target` so the zero-denominator is caught at construction time with a clear error rather than silently stored and panicking later.
4. Add minimum-value constraints: `genesis_epoch_length >= 1`, `epoch_duration_target >= 1`, `orphan_rate_target.1 >= 1`.
