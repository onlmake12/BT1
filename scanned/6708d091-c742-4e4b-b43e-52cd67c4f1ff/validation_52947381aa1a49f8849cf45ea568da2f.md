### Title
Premature `mark_as_known_txs` State Update Before `submit_remote_tx` Enables Transaction Propagation Suppression - (File: sync/src/relayer/transactions_process.rs)

---

### Summary

In `TransactionsProcess::execute()`, `mark_as_known_txs` is called at line 76 to mark all incoming relay transactions as "known" **before** the actual `submit_remote_tx` calls are dispatched in a spawned async task. This premature state update removes each transaction hash from `unknown_tx_hashes` and inserts it into `tx_filter` before any submission succeeds. When submission fails for certain rejection types, the node is left in an inconsistent state: the transaction is no longer tracked for re-request, yet it never entered the pool.

---

### Finding Description

In `TransactionsProcess::execute()`, the processing of a `RelayTransactions` P2P message follows this sequence:

**Step 1 — Filter (correct):** Transactions are collected only if they are not already in `tx_filter` and are present in `unknown_tx_hashes` with `requesting_peer == self.peer`.

**Step 2 — Premature state update (the bug):** At line 76, `mark_as_known_txs` is called on the entire batch:

```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
```

`mark_as_known_txs` does two things atomically (lines 1443–1451 of `sync/src/types/mod.rs`):
- Removes each hash from `unknown_tx_hashes` (the proactive re-request queue)
- Inserts each hash into `tx_filter` (the TTL-based "already seen" filter)

**Step 3 — Deferred action:** The actual submissions are spawned as a separate async task:

```rust
self.relayer.shared.shared().async_handle().spawn(async move {
    for (tx, declared_cycles) in txs {
        if let Err(e) = tx_pool.submit_remote_tx(tx.clone(), declared_cycles, peer).await {
            error!("submit_tx error {}", e);
        }
    }
});
```

The state update in Step 2 is unconditional and happens before any submission result is known. If `submit_remote_tx` fails:

- For `Reject::Full`: `resumeble_process_tx_and_notify_full_reject` sends `TxVerificationResult::Reject`, which causes `send_bulk_of_tx_hashes` to call `remove_from_known_txs` — removing the hash from `tx_filter` only. The hash is **permanently gone** from `unknown_tx_hashes`.
- For `Reject::LowFeeRate` or malformed rejections: `is_allowed_relay()` returns `false`, so `TxVerificationResult::Reject` is **never sent**, and `remove_from_known_txs` is **never called**. The hash stays in `tx_filter` until TTL expiry and is permanently absent from `unknown_tx_hashes`.
- For `Reject::DeclaredWrongCycles`: `TxVerificationResult::Reject` is sent and `remove_from_known_txs` is called, but the hash is still permanently removed from `unknown_tx_hashes`.

In all failure cases, the node loses its proactive re-request record for the transaction.

---

### Impact Explanation

An unprivileged P2P peer can exploit this to suppress a target transaction `T` from a victim node's pool:

1. Peer announces hash of `T` to victim via `RelayTransactionHashes`.
2. Victim adds `T` to `unknown_tx_hashes` and sends `GetRelayTransactions`.
3. Peer responds with `T` but with a mismatched `declared_cycles` value.
4. `mark_as_known_txs([T])` fires — `T` is removed from `unknown_tx_hashes` and added to `tx_filter`.
5. Verification fails with `Reject::DeclaredWrongCycles`; `remove_from_known_txs` removes `T` from `tx_filter`.
6. `T` is now absent from both `unknown_tx_hashes` and `tx_filter`.
7. The victim node will **not** proactively re-request `T` from any peer.
8. The attacker peer is banned, but a new peer connection can repeat the cycle.

With multiple peer connections, an attacker can continuously prevent `T` from being proactively fetched by the victim, delaying or preventing its entry into the pool and its eventual inclusion in a block. This disrupts transaction propagation for targeted transactions.

---

### Likelihood Explanation

Medium. The attacker must be connected to the victim as a P2P peer (no special privilege required — CKB accepts inbound connections from the public). The attacker must know the hash of the transaction they wish to suppress (observable from the public mempool or by monitoring `RelayTransactionHashes` messages). Multiple peer connections are needed to sustain the suppression after each ban, which is feasible with modest resources. The attack is most effective when the victim has few alternative peers that could re-announce the transaction.

---

### Recommendation

Move `mark_as_known_txs` inside the spawned async task, called only after a successful `submit_remote_tx`. This ensures the state update is coupled to the actual outcome:

```rust
self.relayer.shared.shared().async_handle().spawn(async move {
    for (tx, declared_cycles) in txs {
        match tx_pool.submit_remote_tx(tx.clone(), declared_cycles, peer).await {
            Ok(_) => {
                shared_state.mark_as_known_tx(tx.hash());
            }
            Err(e) => {
                error!("submit_tx error {}", e);
            }
        }
    }
});
```

Alternatively, if early deduplication is required, ensure that on any submission failure the hash is re-inserted into `unknown_tx_hashes` so the node can re-request it from other peers.

---

### Proof of Concept

1. Connect to a CKB node as a P2P relay peer.
2. Observe a pending transaction hash `T` from the public mempool.
3. Send a `RelayTransactionHashes` message advertising `T` to the victim node.
4. When the victim sends `GetRelayTransactions` for `T`, respond with a `RelayTransactions` message containing `T` but with `declared_cycles` set to an incorrect value (e.g., `actual_cycles + 1`).
5. The victim calls `mark_as_known_txs([T])` (line 76 of `transactions_process.rs`), removing `T` from `unknown_tx_hashes`.
6. Verification fails with `DeclaredWrongCycles`; `remove_from_known_txs` removes `T` from `tx_filter`.
7. Confirm via debug logging or state inspection that `T` is absent from both `unknown_tx_hashes` and `tx_filter`.
8. Confirm the victim does not proactively request `T` from any other connected peer.
9. Repeat from step 3 using a new peer connection (after the ban expires or from a different IP) to sustain suppression.

    pub fn mark_as_known_txs(&self, hashes: impl Iterator<Item = Byte32> + std::clone::Clone) {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();
        let mut tx_filter = self.tx_filter.lock();
