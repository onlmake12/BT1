### Title
Permanent Ancestor Descendant-Weight Corruption via Pre-Cleared Links in `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

### Summary

`remove_entry_and_descendants` pre-clears all link entries for every tx in the removal set **before** calling `remove_entry` on each. Because `update_ancestors_index_key` inside `remove_entry` resolves ancestors via `calc_ancestors`, which reads from the now-empty `TxLinksMap`, it returns an empty set for every removed tx. As a result, the surviving ancestor P's `descendants_count / descendants_fee / descendants_size / descendants_cycles` fields are never decremented, permanently inflating P's `EvictKey`.

---

### Finding Description

**Execution trace for chain P → C1 → C2, calling `remove_entry_and_descendants(C1)`:**

**Phase 1 — pre-clear loop** [1](#0-0) 

```
removed_ids = [C1, C2]
remove_entry_links(C1)  →  removes C1 from P.children, removes C1 from C2.parents, deletes C1's link node
remove_entry_links(C2)  →  C2.parents already empty; deletes C2's link node
```

After this loop: P's link node still exists but `P.children = {}`. C1 and C2 have **no link nodes at all**.

**Phase 2 — `remove_entry(C1)`** [2](#0-1) 

```
update_ancestors_index_key(C1, Remove)
  → calc_ancestors(C1)          // C1's link node is gone
  → returns {}                  // empty — no ancestors found
  → P.sub_descendant_weight(C1) is NEVER called
``` [3](#0-2) 

**Phase 2 — `remove_entry(C2)`**

```
update_ancestors_index_key(C2, Remove)
  → calc_ancestors(C2)          // C2's link node is gone
  → returns {}
  → P.sub_descendant_weight(C2) is NEVER called
```

**State of P after the call:**

| Field | Expected | Actual |
|---|---|---|
| `descendants_count` | 1 | 3 |
| `descendants_fee` | P.fee | P.fee + C1.fee + C2.fee |
| `descendants_size` | P.size | P.size + C1.size + C2.size |
| `descendants_cycles` | P.cycles | P.cycles + C1.cycles + C2.cycles |

The comment on line 256–258 reveals the intent was only to suppress `update_descendants_index_key` (to avoid decrementing C2's ancestor weights when C1 is removed), but the pre-clear also silences `update_ancestors_index_key`, which is the bug. [4](#0-3) 

---

### Impact Explanation

`EvictKey.fee_rate` is computed as `max(descendants_feerate, feerate)`. [5](#0-4) 

`EvictKey` ordering is ascending by `fee_rate` — the **lowest** fee_rate entry is evicted first. [6](#0-5) 

With inflated `descendants_fee` and `descendants_size/cycles`, P's `descendants_feerate` is higher than reality, so `fee_rate` is inflated. P is never selected by `next_evict_entry` when the pool is full. [7](#0-6) 

Concrete effects:
- P permanently occupies a pool slot with a falsely high evict priority
- When the pool is full, P is skipped during eviction; lower-fee transactions that should be evicted remain
- Higher-fee incoming transactions are rejected as the pool stays full
- The corruption is **permanent** — there is no reconciliation path; P's descendant fields are only updated via `update_ancestors_index_key` / `update_descendants_index_key`, both of which are now no-ops for P's ghost descendants

---

### Likelihood Explanation

The attacker path is fully unprivileged and reachable via standard P2P transaction submission:

1. Submit P, C1, C2 (a valid dependency chain) to the target node's tx-pool
2. Submit a new transaction T that double-spends one of C1's inputs

Step 2 triggers `resolve_conflict` → `remove_entry_and_descendants(&C1_id)`. [8](#0-7) 

The same trigger fires via `resolve_conflict_header_dep` and via the ancestor-count eviction path inside `check_and_record_ancestors`. [9](#0-8) [10](#0-9) 

No special privileges, no PoW, no key material required. The attack is repeatable and cheap.

---

### Recommendation

Remove the pre-clear loop entirely. Without it, `remove_entry` processes each tx in topological order and `update_ancestors_index_key` correctly decrements P's descendant weights via the still-intact link graph. The concern about double-decrementing C2's ancestor weights is moot because C2 is also being removed in the same pass — its `entries` record is deleted before any further use.

Alternatively, if the pre-clear is kept for the `update_descendants_index_key` suppression, the fix must manually call `update_ancestors_index_key` for each removed tx **before** its link node is erased, or collect and apply the ancestor weight deltas in a separate pass.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
