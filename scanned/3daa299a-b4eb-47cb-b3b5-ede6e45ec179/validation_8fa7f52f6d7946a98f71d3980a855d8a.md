### Title
Stale `committed_txs_hash_cache` Secondary-Index Lookup Causes Compact Block Reconstruction Collision â€” (`tx-pool/src/pool.rs`)

### Summary

`TxPool::get_tx_from_pool_or_store` resolves compact-block short IDs against a secondary LRU cache keyed by the truncated `ProposalShortId` (first 10 bytes of the tx hash). When a previously committed transaction occupies a cache slot whose `ProposalShortId` collides with a different transaction in an incoming compact block, the node silently substitutes the stale committed transaction, causing block reconstruction to fail with a `Collided` result and forcing the node to re-request every non-prefilled transaction in the block rather than only the genuinely missing one.

### Finding Description

`ProposalShortId` is defined as the first 10 bytes of a transaction hash: [1](#0-0) 

`TxPool` maintains a secondary cache mapping this truncated identifier to the full committed transaction hash: [2](#0-1) 

When a block is committed, every transaction's `ProposalShortId` is written into this cache: [3](#0-2) 

During compact block reconstruction, `get_tx_from_pool_or_store` consults this cache as a fallback: [4](#0-3) 

This function is called from `get_tx_for_compact_block`, which supplies transactions to `reconstruct_block`: [5](#0-4) 

The check performed is purely by `ProposalShortId` â€” a 10-byte prefix. If a previously committed transaction `A` shares the same `ProposalShortId` as a different transaction `B` referenced in an incoming compact block, the cache lookup returns `A`'s full hash, the node fetches `A` from the snapshot, and inserts it into the reconstruction map in place of `B`. The reconstructed block's `transactions_root` then diverges from the compact block header's declared root: [6](#0-5) 

This triggers `ReconstructionResult::Collided`, which causes the node to request **all** non-prefilled transactions from the peer rather than only the missing one: [7](#0-6) 

The structural parallel to the Aave Lens bug is exact: a secondary mapping (handleâ†’profileId / `ProposalShortId`â†’txHash) is consulted to resolve an entity, and a stale or colliding entry causes the wrong entity to be returned and acted upon.

### Impact Explanation

An attacker who can commit a transaction with a crafted `ProposalShortId` that collides with a transaction in a future compact block causes every receiving node to:

1. Retrieve the wrong (stale committed) transaction from the cache.
2. Fail the `transactions_root` check and classify the event as a short-ID collision.
3. Re-request **all** non-prefilled transactions in the block instead of only the one that was genuinely absent.

For a block with many transactions this is a significant bandwidth amplification. It also delays block propagation by one full round-trip for every affected node. Consensus safety is not broken because the `transactions_root` mismatch is detected before `accept_block` is called, but block relay latency and node bandwidth are materially degraded.

### Likelihood Explanation

`ProposalShortId` is 10 bytes (80 bits). Finding a transaction whose first 10 hash bytes match a target requires approximately 2^40 hash evaluations â€” feasible for a motivated attacker with commodity hardware in hours to days. The attacker must also pay the on-chain fee to commit the crafted transaction. Once committed, the cache entry persists for up to 100,000 slots: [8](#0-7) 

A targeted variant is possible: observe a high-value transaction `B` in the mempool, brute-force a transaction `A` sharing `B`'s `ProposalShortId`, commit `A` (spending different inputs), and the cache entry for that short ID is overwritten. When the next compact block containing `B` arrives, every node that has `A` in its committed cache will misfire.

### Recommendation

When retrieving a transaction from `committed_txs_hash_cache` for compact block reconstruction, verify that the returned transaction's `ProposalShortId` actually matches the short ID requested **and** that the full transaction hash matches what the compact block expects (i.e., cross-check against the compact block's `transactions_root` or the short ID derivation). Alternatively, do not use the committed cache as a source for compact block reconstruction at all â€” its purpose is performance optimisation, and the collision risk it introduces outweighs the benefit when a genuine missing-transaction request is the correct fallback.

A minimal fix in `get_tx_from_pool_or_store` would be to return `None` from the committed-cache branch unconditionally when called in the compact-block reconstruction context, or to add a caller-supplied expected full hash parameter so the cache hit can be validated before use.

### Proof of Concept

1. Observe transaction `B` (short_id = `S`) in the mempool of a target node.
2. Brute-force transaction `A` such that `ProposalShortId::from_tx_hash(hash(A)) == S` but `hash(A) != hash(B)`. (~2^40 SHA2 evaluations.)
3. Submit and mine `A` (spending any live cell). The target node's `committed_txs_hash_cache` now stores `S â†’ hash(A)`.
4. A miner produces a compact block containing `B` (identified by short_id `S`). The target node is not pre-seeded with `B`.
5. `reconstruct_block` calls `get_tx_for_compact_block({S})`. `get_tx_from_pool_or_store` finds `S` in the committed cache, fetches `A` from the snapshot, and returns it.
6. The reconstructed block uses `A` in place of `B`. `transactions_root` mismatches â†’ `ReconstructionResult::Collided`.
7. The node sends `GetBlockTransactions` requesting **all** non-prefilled transaction indexes instead of only index of `B`, amplifying bandwidth by a factor proportional to block size.
