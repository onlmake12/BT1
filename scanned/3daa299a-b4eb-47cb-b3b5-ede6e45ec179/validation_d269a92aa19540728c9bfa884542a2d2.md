### Title
`get_block_template` RPC Always Returns Stub `required`/`depends` Fields in `TransactionTemplate`, Breaking Miner Transaction Selection — (`tx-pool/src/block_assembler/mod.rs`)

---

### Summary

The `tx_entry_to_template` function in the block assembler unconditionally hardcodes `required: false` and `depends: None` for every `TransactionTemplate` entry in the block template, despite both fields being documented as meaningful protocol-level hints that miners must rely on to safely select transactions. This is a direct analog to the CNft.sol `safeTransferFrom` deviation: a standard interface is advertised as functional but silently returns stub values, breaking any caller that depends on the interface contract.

---

### Finding Description

In `tx-pool/src/block_assembler/mod.rs`, the conversion function `tx_entry_to_template` is the sole path by which every `TxEntry` in the tx-pool becomes a `TransactionTemplate` in the `get_block_template` RPC response:

```rust
pub(crate) fn tx_entry_to_template(entry: &TxEntry) -> TransactionTemplate {
    TransactionTemplate {
        hash: entry.transaction().hash().into(),
        required: false, // unimplemented
        cycles: Some(entry.cycles.into()),
        depends: None, // unimplemented
        data: entry.transaction().data().into(),
    }
}
``` [1](#0-0) 

This function is called for every transaction in the template during `From<&BlockTemplate> for JsonBlockTemplate`: [2](#0-1) 

The `TransactionTemplate` type in `util/jsonrpc-types/src/block_template.rs` documents both fields as protocol-level contracts:

- `required: bool` — "Whether miner must include this transaction in the new block."
- `depends: Option<Vec<Uint64>>` — "Transaction dependencies. This is a hint to help miners selecting transactions. **This transaction can only be committed if its dependencies are also committed in the new block.**" [3](#0-2) 

The `BlockTemplate` struct itself states: "Miners must include the transactions marked as `required` in the assembled new block." The RPC is explicitly designed to "allow miners to remove transactions." [4](#0-3) 

The tx-pool's `TxSelector` does correctly package chained (parent→child) transactions together with their ancestors when building the template: [5](#0-4) 

So the template can and does contain chains of dependent transactions (e.g., tx B spends an output of tx A, both in the same template). The `depends` field exists precisely to expose this relationship to miners — but it is never populated.

---

### Impact Explanation

A miner or mining pool that calls `get_block_template` and attempts to exercise the protocol-guaranteed right to remove transactions from the template has no way to know which transactions are ancestors of others. If the miner removes tx A while retaining tx B (which spends A's output), the assembled block will fail full consensus validation at `submit_block` because B's input cell is unresolvable. The miner's PoW work is wasted.

The `submit_block` path passes the block through `blocking_process_block`, which performs full transaction resolution and verification: [6](#0-5) 

This is confirmed by the existing test suite, which explicitly verifies that a block committing a child transaction without its parent is rejected: [7](#0-6) 

Additionally, `required: false` being hardcoded means that if the protocol ever marks a transaction as mandatory (e.g., for protocol-level enforcement), miners will silently omit it, producing invalid blocks without any indication from the RPC.

---

### Likelihood Explanation

Any miner or mining pool software that implements transaction selection logic — a standard and explicitly supported use case per the RPC documentation — will be affected. The `get_block_template` RPC is the primary interface for all CKB miners. The tx-pool regularly packages chained transactions (parent-child chains are a core CKB feature, tested extensively), so the template will routinely contain dependent transactions with `depends: None`. A miner that removes any transaction from a non-trivial template risks submitting an invalid block.

---

### Recommendation

In `tx_entry_to_template`, populate `depends` by computing the indices of ancestor transactions that are also present in the current template's transaction list. The block assembler already tracks ancestor relationships via `pool_map.calc_ancestors` (used in `TxSelector`); this information should be threaded through to the template serialization step. Similarly, `required` should reflect any future protocol-level mandatory transaction designation rather than being hardcoded to `false`.

---

### Proof of Concept

1. Call `get_block_template` on a node with a tx-pool containing a chain: tx A (parent) → tx B (child spends A's output). Both are in the proposed pool.
2. Observe the response: both `TransactionTemplate` entries have `"required": false` and `"depends": null`.
3. Remove tx A from the template's `transactions` array, keeping tx B.
4. Solve PoW and call `submit_block` with the modified block.
5. The node rejects the block because tx B's input (A's output cell) cannot be resolved — the miner's PoW work is wasted.

/// There are 3 transactions, A, B and C:
///   - A was already committed before;
///   - B spends A;
///   - A is one of C's cell-deps.
///
/// A block, which commits B and C in order, should be invalid because that C's cell-dep A is dead
/// (as C spends A, A is dead).
///
/// The difference between case `CellBeingSpentThenCellDepInSameBlockTestSubmitBlock` is the order
/// of committed transactions. This case commits `[B, C]`.
pub struct CellBeingSpentThenCellDepInSameBlockTestSubmitBlock;

impl Spec for CellBeingSpentThenCellDepInSameBlockTestSubmitBlock {
    fn run(&self, nodes: &mut Vec<Node>) {
        let node0 = &nodes[0];

        let initial_inputs = gen_spendable(node0, 2);
        let input_a = &initial_inputs[0];
        let input_c = &initial_inputs[1];

        // Commit transaction A
        let tx_a = {
            let tx_a = always_success_transaction(node0, input_a);
            node0.submit_transaction(&tx_a);
            node0.mine_until_bool(|| is_transaction_committed(node0, &tx_a));
            tx_a
        };

        // Create transaction B which spends A
        let tx_b = {
            let input =
                CellMetaBuilder::from_cell_output(tx_a.output(0).unwrap(), Default::default())
                    .out_point(OutPoint::new(tx_a.hash(), 0))
                    .build();
            always_success_transaction(node0, &input)
        };

        // Create transaction C which depends A
        let tx_c = {
            let tx = always_success_transaction(node0, input_c);
            let cell_dep_to_tx_a = CellDepBuilder::default()
                .dep_type(DepType::Code)
                .out_point(OutPoint::new(tx_a.hash(), 0))
                .build();
            tx.as_advanced_builder().cell_dep(cell_dep_to_tx_a).build()
        };

        // Propose B and C, to prepare testing
        let block = node0
            .new_block_builder(None, None, None)
            .proposal(tx_b.proposal_short_id())
            .proposal(tx_c.proposal_short_id())
            .build();
        node0.submit_block(&block);
        node0.mine(node0.consensus().tx_proposal_window().closest());

        // Create block commits B and C in order
        let block = node0
            .new_block_builder(None, None, None)
            .transactions(vec![tx_b, tx_c])
            .build();

        let ret = node0
            .rpc_client()
            .submit_block("".to_owned(), block.data().into());
        assert!(
            ret.is_err(),
            "a block commits transactions [B, C] should be invalid, ret: {ret:?}"
        );
