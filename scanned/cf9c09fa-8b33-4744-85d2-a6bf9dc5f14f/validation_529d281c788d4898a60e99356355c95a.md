Audit Report

## Title
Surviving ancestors retain permanently inflated `descendants_count`/`descendants_fee` after `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-clears all link entries via `remove_entry_links` before calling `remove_entry` on each removed transaction. Because `update_ancestors_index_key` re-derives the ancestor set from the (already-mutated) links map, it finds an empty ancestor set and never calls `sub_descendant_weight` on surviving ancestors. Those ancestors permanently carry stale, inflated descendant weights, corrupting their `EvictKey` and making them appear more fee-valuable than they are, allowing low-fee transactions to evade eviction indefinitely.

## Finding Description
In `remove_entry_and_descendants` (lines 252–265), the first loop (lines 257–259) calls `remove_entry_links` for every id in `removed_ids`. `remove_entry_links` (lines 418–430) removes the target from its parents' children sets, removes the target from its children's parents sets, and then calls `self.links.remove(id)` — fully deleting the entry from the `TxLinksMap`.

The second loop (lines 261–264) then calls `remove_entry` on each id. `remove_entry` (lines 235–250) calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` (line 242). `update_ancestors_index_key` (lines 432–445) calls `self.links.calc_ancestors(&child.proposal_short_id())` (line 434). `calc_ancestors` delegates to `calc_relative_ids` (links.rs line 37–50), which does `self.inner.get(short_id)` — but the entry was already deleted from `self.links.inner` by the first loop. The result is `.unwrap_or_default()` → empty set. The ancestor loop is a no-op; no surviving ancestor ever receives `sub_descendant_weight`.

The code comment at line 256 states the intent: "update links state for remove, so that we won't update_descendants_index_key in remove_entry." The intent was only to suppress `update_descendants_index_key` (which would otherwise try to update descendants that are themselves being removed), but the side effect is that `update_ancestors_index_key` is also neutered for the same reason.

Concrete state corruption for chain A→B→C after `remove_entry_and_descendants(B)`:

| Field on A | Expected | Actual |
|---|---|---|
| `descendants_count` | 1 | 3 |
| `descendants_fee` | fee_A | fee_A + fee_B + fee_C |
| `descendants_size` | size_A | size_A + size_B + size_C |
| `evict_key.fee_rate` | max(desc_rate_A, rate_A) | max(inflated_desc_rate, rate_A) |

`EvictKey` is built from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` (entry.rs lines 234–247). `Ord for EvictKey` sorts primarily by `fee_rate` ascending, then `descendants_count` ascending (sort_key.rs lines 92–103). With inflated values, A sorts toward the "never evict" end of `iter_by_evict_key()`, used by `next_evict_entry` (pool_map.rs line 380–385).

## Impact Explanation
An attacker can keep low-fee transactions alive in a full mempool indefinitely. When the pool is full and `next_evict_entry` is called, A is skipped in favor of legitimately higher-fee transactions. High-fee transactions submitted by honest users are rejected or evicted while low-fee attacker transactions survive. This directly harms miner revenue and distorts the CKB fee market. This matches the allowed impact: **"Vulnerabilities which could easily damage CKB economy" — Critical (15001–25000 points)**.

## Likelihood Explanation
Reachable by any unprivileged transaction submitter with no special access:
1. Submit tx A (low fee).
2. Submit tx B spending A's output.
3. Submit tx C spending B's output.
4. Submit tx D double-spending B's input → triggers `resolve_conflict` → `remove_entry_and_descendants(B)` → A's stats are permanently inflated.

The attacker pays fees for B and C (which are removed), but A survives with a corrupted eviction key. The attack is repeatable and cheap relative to the damage caused. No PoW, operator access, or victim mistakes are required.

## Recommendation
In `remove_entry_and_descendants`, collect the ancestor sets for all `removed_ids` **before** the `remove_entry_links` loop, then apply `sub_descendant_weight` to surviving ancestors (those not in `removed_ids`) after removal. One correct approach:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Snapshot ancestor sets BEFORE clearing links
    let ancestor_map: Vec<(ProposalShortId, HashSet<ProposalShortId>)> = removed_ids
        .iter()
        .map(|rid| (rid.clone(), self.links.calc_ancestors(rid)))
        .collect();

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed_entries: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Now update surviving ancestors
    for (removed_id, ancestors) in &ancestor_map {
        if let Some(removed_entry) = removed_entries.iter().find(|e| &e.proposal_short_id() == removed_id) {
            for anc_id in ancestors {
                if !removed_set.contains(anc_id) {
                    self.entries.modify_by_id(anc_id, |e| {
                        e.inner.sub_descendant_weight(removed_entry);
                        e.evict_key = e.inner.as_evict_key();
                    });
                }
            }
        }
    }

    removed_entries
}
```

Alternatively, restructure `remove_entry` to accept a pre-computed ancestor set rather than re-deriving it from the already-mutated links map.

## Proof of Concept
Invariant test: after `remove_entry_and_descendants(B)` on chain A→B→C, assert:
```rust
let a = pool.get(&a_id).unwrap();
assert_eq!(a.descendants_count, 1);   // fails: actual is 3
assert_eq!(a.descendants_fee, fee_a); // fails: actual is fee_a + fee_b + fee_c
```

This can be added as a unit test in `tx-pool/src/component/tests/score_key.rs` alongside the existing `test_remove_entry` test (which only tests `remove_entry`, not `remove_entry_and_descendants`). Build chain A→B→C, call `pool.remove_entry_and_descendants(&b_id)`, then assert A's descendant fields equal their solo values.
