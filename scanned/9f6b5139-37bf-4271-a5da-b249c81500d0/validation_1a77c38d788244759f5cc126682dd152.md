Audit Report

## Title
Peer-Supplied `declared_cycles` Used as VM Verification Cap Enables Permanent Known-TX Filter Poisoning â€” (`tx-pool/src/process.rs`, `sync/src/relayer/transactions_process.rs`)

## Summary

A remote peer relaying a transaction via `RelayTransactions` supplies a `declared_cycles` value that is used verbatim as the `max_cycles` cap for CKB-VM script verification. When a peer deliberately under-declares cycles (e.g., `declared_cycles = 1`), the VM halts with `ExceededMaximumCycles`, producing `Reject::Verification(Script error)`. This reject variant is classified as `is_malformed_tx() = true` / `is_allowed_relay() = false`, so `TxVerificationResult::Reject` is never sent to the relayer, and the tx hash is never removed from the node's known-tx filter. Because the hash was inserted into the filter before verification, all subsequent honest relay attempts for the same transaction are silently dropped, permanently suppressing the transaction from the victim node's tx-pool for the lifetime of the filter entry.

**Step 1 â€” Hash inserted before verification** (`sync/src/relayer/transactions_process.rs` L76):

After the only guard (banning `declared_cycles > max_block_cycles`), the tx hash is unconditionally inserted into the known-tx filter: [1](#0-0) 

A value of `declared_cycles = 1` passes the `> max_block_cycles` check trivially.

**Step 2 â€” Attacker-controlled cap used for VM verification** (`tx-pool/src/process.rs` L720):

`_process_tx` uses the peer-supplied value directly as `max_cycles`: [2](#0-1) 

With `max_cycles = 1` and actual script cost of 537 cycles, `verify_rtx` returns `ExceededMaximumCycles`. The `try_or_return_with_snapshot!` macro at L734 returns early with `Reject::Verification(Script error)` â€” the `DeclaredWrongCycles` check at L736â€“748 is never reached.

**Step 3 â€” Misclassification of the error** (`util/types/src/core/tx_pool.rs`):

`is_malformed_from_verification` returns `true` for any `ErrorKind::Script` error that does not contain `ARGV_TOO_LONG_TEXT`: [3](#0-2) 

`ExceededMaximumCycles` does not contain that string, so `is_malformed_tx()` returns `true` and `is_allowed_relay()` returns `false`: [4](#0-3) 

**Step 4 â€” `TxVerificationResult::Reject` never sent** (`tx-pool/src/process.rs` L514â€“521):

Because `is_allowed_relay()` is `false`, the `send_result_to_relayer(TxVerificationResult::Reject {...})` branch is skipped: [5](#0-4) 

**Step 5 â€” Hash never removed from filter** (`sync/src/relayer/mod.rs` L673â€“675):

`remove_from_known_txs` is only called when `TxVerificationResult::Reject` is received. Since it is never sent, the hash persists in the filter: [6](#0-5) 

**Step 6 â€” Honest peers silently dropped** (`sync/src/relayer/transactions_process.rs` L49â€“55):

Any subsequent honest peer relaying the same tx with correct cycles is filtered out: [7](#0-6) 

**Contrast with over-declaration**: When `declared_cycles > actual`, `verify_rtx` succeeds and the `DeclaredWrongCycles` check at L736â€“748 fires, producing `Reject::DeclaredWrongCycles`, for which `is_allowed_relay()` returns `true` and the hash is correctly cleared. [8](#0-7) 

## Impact Explanation

An attacker with only a standard P2P connection can suppress arbitrary valid transactions from propagating through targeted nodes. Applied at scale across many nodes and many transactions, this constitutes **CKB network congestion with few costs** â€” matching the High (10001â€“15000 points) bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The attacker pays only the cost of P2P connections and a 3-day ban per IP, while the suppression effect persists for the full filter entry lifetime.

## Likelihood Explanation

The attack requires no privileged access, no key material, and no hashpower. The attacker only needs to know a target transaction hash (observable from any node's mempool broadcast) and send it with `declared_cycles = 1` via `RelayV3`. The 3-day IP ban is the only friction; it is trivially bypassed by rotating IPs or waiting for expiry. The `tx_filter.remove_expired()` call does eventually expire entries, but the attacker can re-inject before expiry to maintain suppression indefinitely.

## Recommendation

1. **Do not use `declared_cycles` as `max_cycles` for VM verification.** Always verify with `consensus.max_block_cycles()` (or `max_tx_verify_cycles`) as the actual cap. Use `declared_cycles` only as a hint for queue prioritization.
2. **Treat `ExceededMaximumCycles` caused by a peer-supplied cap as `DeclaredWrongCycles`**: detect when `verify_rtx` fails with `ExceededMaximumCycles` and `declared_cycles` was the binding cap, then return `Reject::DeclaredWrongCycles` so `is_allowed_relay()` returns `true` and the hash is cleared.
3. **Alternatively, mark the tx hash as known only after successful verification**, or unconditionally remove it from the filter on any non-`DeclaredWrongCycles` failure.

## Proof of Concept

1. Observe a valid pending transaction `T` with actual script cost of 537 cycles (standard secp256k1 lock) on the network.
2. Connect to victim node as a peer via `RelayV3`.
3. Announce `T`'s hash via `RelayTransactionHashes`; wait for `GetRelayTransactions`.
4. Send `RelayTransactions` with `T` and `cycles = 1`.
5. Victim node: inserts `T`'s hash into known-tx filter (`mark_as_known_txs`), runs VM with `max_cycles = 1`, gets `ExceededMaximumCycles` â†’ `Reject::Verification(Script error)`, bans attacker peer, but does **not** send `TxVerificationResult::Reject` â†’ hash stays in filter.
6. Honest peer B now tries to relay `T` with correct `cycles = 537`. The filter check `!tx_filter.contains(&T.hash())` returns `false` â†’ `T` is silently dropped at `transactions_process.rs` L49â€“55.
7. `T` never enters the victim's tx-pool.
8. Attacker reconnects (after ban expiry or via new IP) and repeats step 4 to refresh the filter entry before it expires.

The existing integration test `DeclaredWrongCyclesChunk` (`test/src/specs/tx_pool/declared_wrong_cycles.rs`) covers only the over-declaration path; the under-declaration path (cycles below actual) produces a different error code that bypasses the re-relay mechanism and is not tested.
