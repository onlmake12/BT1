Audit Report

## Title
`ScriptHashTypeVerifier` Omits Type Script Hash-Type Validation, Allowing Cheap Bypass of the Non-Contextual DoS Filter — (`verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the hash type of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the optional **type** script. A transaction carrying a valid lock script but a type script with hash type `Data3` (byte `6`) or any other non-enabled value passes `NonContextualTransactionVerifier`, is admitted to the tx-pool, and is only rejected during the more expensive contextual verification step at `select_version`. This undermines the two-stage admission design and allows any RPC caller with a live UTXO to repeatedly trigger contextual verification at negligible cost.

## Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is `{0, 1, 2, 4}` (Data, Type, Data1, Data2): [1](#0-0) 

`ScriptHashTypeVerifier::verify()` loops over outputs and checks only `output.lock().hash_type()`. The `output.type_()` field is never read: [2](#0-1) 

`ScriptHashTypeVerifier` is a sub-verifier of `NonContextualTransactionVerifier`, which is the first and only cheap gate called by `non_contextual_verify` in the tx-pool: [3](#0-2) [4](#0-3) 

The downstream contextual path calls `select_version`, which **does** check the type script hash type and returns `InvalidScriptHashType` for any value outside the four enabled variants: [5](#0-4) 

The gap: a transaction whose outputs have a valid lock script but a type script with `hash_type = 6` (`Data3`) passes `ScriptHashTypeVerifier` (only lock is checked), passes `non_contextual_verify`, enters the tx-pool, and is rejected only during contextual verification — after DB lookups, transaction resolution, and verifier setup have already been paid.

The existing test suite confirms the lock-only scope and the absence of any analogous test for type scripts: [6](#0-5) 

## Impact Explanation

This directly matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. The non-contextual check is the intended cheap DoS barrier before the expensive contextual check. By bypassing it for type scripts, an attacker forces the node to perform transaction resolution (DB reads) and contextual verifier setup for every crafted submission. With a single UTXO, after each eviction the UTXO remains unspent and the attacker can immediately resubmit a new variant (different outputs/amounts) to avoid any recently-rejected cache, sustaining a continuous stream of forced contextual verification cycles. This degrades node throughput and can crowd out legitimate transactions from the tx-pool.

## Likelihood Explanation

The attacker requires only a single live UTXO and access to the `send_transaction` RPC endpoint, which is publicly reachable. No special privileges, leaked keys, or victim mistakes are needed. The attack is repeatable indefinitely: each eviction leaves the UTXO unspent, and varying the output set produces a distinct transaction hash that bypasses any recently-rejected deduplication. The construction is trivial — set `type_` on any output to a script with `hash_type` byte `6`.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the hash type of each output's optional type script, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

A corresponding test analogous to `test_not_enabled_hash_type_output_lock` should be added for the type script path.

## Proof of Concept

1. Obtain any live UTXO on-chain.
2. Construct a transaction spending that UTXO with one output whose lock uses `ScriptHashType::Type` (valid) and whose `type_` is set to any script with `hash_type` byte `6` (`Data3`).
3. Submit via `send_transaction` RPC.
4. Observe: `non_contextual_verify` returns `Ok` — `ScriptHashTypeVerifier` only checked the lock.
5. Observe: contextual verification fails at `select_version` with `InvalidScriptHashType`.
6. The UTXO remains unspent after eviction; repeat with a different output value to produce a new tx hash and bypass recently-rejected deduplication.
7. Add a unit test mirroring `test_not_enabled_hash_type_output_lock` but setting `Data3` on the type script instead of the lock script — it will pass `ScriptHashTypeVerifier::verify()` without the fix, confirming the gap.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
