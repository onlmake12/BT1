Audit Report

## Title
`add_entry` Overwrites Correctly-Updated Pool Totals with Pre-Eviction Snapshot, Inflating `total_tx_size`/`total_tx_cycles` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the new aggregate totals are computed **before** ancestor-eviction runs, stored in local variables, and then unconditionally written back to `self.total_tx_size` / `self.total_tx_cycles` **after** eviction has already correctly decremented those fields. Each eviction cycle therefore re-inflates the pool's accounting by the sizes and cycles of the evicted transactions. An attacker who repeatedly triggers the cell-ref-parent eviction path can drive the perceived pool size toward `max_tx_pool_size`, causing legitimate transactions to be rejected.

## Finding Description

**Exact code path** (`tx-pool/src/component/pool_map.rs`, lines 200–221):

```rust
pub(crate) fn add_entry(...) -> Result<(bool, HashSet<TxEntry>), Reject> {
    ...
    // Step 1 – snapshot computed BEFORE any eviction
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211

    // Step 2 – may evict entries; each removal calls update_stat_for_remove_tx,
    //           which correctly subtracts from self.total_tx_size / self.total_tx_cycles
    evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

    ...
    // Step 3 – stale snapshot OVERWRITES the correctly-decremented fields
    self.total_tx_size  = total_tx_size;                            // line 218
    self.total_tx_cycles = total_tx_cycles;                         // line 219
    Ok((true, evicts))
}
```

`updated_stat_for_add_tx` (lines 711–729) computes:
```
total_tx_size  = self.total_tx_size  + entry.size
total_tx_cycles = self.total_tx_cycles + entry.cycles
```
at the moment of the call, before evictions.

`check_and_record_ancestors` (lines 588–640) enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618), which calls `remove_entry` (line 247), which calls `update_stat_for_remove_tx` — correctly subtracting the evicted entries from `self.total_tx_size` and `self.total_tx_cycles`.

Then lines 218–219 overwrite those correctly-decremented fields with the pre-eviction snapshot, effectively **adding back** the sizes/cycles of every evicted transaction. The net result after one such `add_entry` call:

```
self.total_tx_size  = (old + entry.size)          // evicted entries NOT subtracted
self.total_tx_cycles = (old + entry.cycles)        // evicted entries NOT subtracted
```

**Why existing checks fail:** `update_stat_for_remove_tx` (lines 733–758) correctly handles the subtraction and even has an underflow-recovery path via `recompute_total_stat`. However, its correct result is immediately discarded when lines 218–219 restore the stale snapshot. The `track_entry_statics` assertion (line 681–684) only validates entry counts, not size/cycle totals, so it does not catch this divergence.

## Impact Explanation

`total_tx_size` is compared against `max_tx_pool_size` (confirmed match in `tx-pool/src/pool.rs`) to enforce pool capacity limits. An inflated `total_tx_size` causes the pool to believe it is fuller than it actually is, leading to premature `Reject::Full` responses for legitimate transactions. Repeated triggering accumulates inflation monotonically (it is never corrected except by `clear()` or a full recompute path that is only reached on underflow). This maps to:

**High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A node whose mempool is artificially saturated stops relaying and accepting new transactions, degrading the network's ability to process user transactions.

## Likelihood Explanation

The eviction branch requires crafting a transaction whose ancestor count exceeds `max_ancestors_count` but where enough `cell_ref_parents` exist to bring it back within limits. This is a non-default but fully reachable code path requiring no special privileges — any unprivileged user can submit transactions to the mempool. The attacker pays only standard transaction fees. The inflation per iteration equals the total size/cycles of evicted transactions; with chains of moderate depth, a few hundred iterations can exhaust a 180 MB pool budget.

## Recommendation

Move the stat update to **after** all evictions complete, so it reflects the final pool state:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute and assign AFTER evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, eliminate the local variables entirely and call `update_stat_for_add_tx` as a direct mutating method (mirroring `update_stat_for_remove_tx`) after evictions.

## Proof of Concept

1. Set `max_ancestors_count = 25` (default).
2. Submit 25 transactions `T1 → T2 → … → T25` where each `Ti` uses the output of `T(i-1)` as an input, and `T1`–`T5` all use the same live cell as a `cell_dep`.
3. Submit a new transaction `T_new` that also uses that same cell as a `cell_dep` and spends an output of `T25`. Its ancestor count is 26 (> 25), but `cell_ref_parents = {T1..T5}`, so `26 - 5 = 21 ≤ 25` — the eviction branch fires.
4. `T1`–`T5` (and their descendants) are evicted; `update_stat_for_remove_tx` correctly subtracts their sizes. Then lines 218–219 restore the pre-eviction snapshot, re-inflating `total_tx_size` by the sum of evicted sizes.
5. Repeat steps 2–4. Each iteration inflates `total_tx_size` by ~5 × `tx_size`. After enough iterations, `total_tx_size ≥ max_tx_pool_size` and all subsequent `add_entry` calls return `Reject::Full`.

A unit test asserting `pool_map.total_tx_size == pool_map.recompute_total_stat().0` after each `add_entry` call that triggers eviction will reproduce the divergence.
