Audit Report

## Title
Wrong Variable in `DownloadScheduler::decrease` Bypasses Timeout Damping, Causing Overly Aggressive Peer Penalization — (File: `sync/src/types/mod.rs`)

## Summary
`DownloadScheduler::decrease` at line 522 assigns `self.task_count.saturating_add(num)` to `self.timeout_count` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` is initialized to 32 and `num` is always 1 or 2, `timeout_count` is always set to at least 33, making the `> 2` guard permanently true. As a result, `task_count` is decremented and `timeout_count` is reset to 0 on every single slow-response event, completely bypassing the intended 3-event damping threshold and causing peers to be disconnected 3× faster than designed.

## Finding Description
`DownloadScheduler` tracks per-peer block-download concurrency via `task_count` (init: `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`) and accumulated slow-response events via `timeout_count` (init: `0`). The `decrease` method is intended to increment `timeout_count` and only reduce `task_count` once more than 2 slow events have accumulated:

```rust
// sync/src/types/mod.rs line 521-527
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: reads task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

Since `task_count` starts at 32 and `num ∈ {1, 2}`, `timeout_count` is always set to ≥ 33. The guard `self.timeout_count > 2` is always true, so `task_count` is decremented and `timeout_count` reset to 0 on every invocation. The accumulation state is never preserved.

`decrease` is called from `remove_by_block` when `adjustment` is true and `should_punish` is true, for `TimeQuantile::NormalToUpper` (passes `num=1`) and `TimeQuantile::UpperToMax` (passes `num=2`). When `task_count` reaches 0, the peer is added to `disconnect_list` and removed from `download_schedulers`.

## Impact Explanation
This matches **Low (501–2000 points): Any other important performance improvements for CKB**. The bug causes peers to be disconnected after 32 slow responses instead of the intended ~96, degrading IBD sync throughput and causing unnecessary peer churn. It does not crash the node, cause consensus deviation, or damage the economy, but it concretely degrades sync performance in a measurable and reproducible way.

## Likelihood Explanation
The code path is exercised continuously during IBD whenever a peer's block response latency exceeds `TimeAnalyzer`'s `normal_time` threshold (~1250 ms). No attacker capability is required — ordinary network jitter, CPU-heavy block validation, or a moderately loaded remote peer triggers it. The only preconditions are `adjustment = true` (the default during IBD) and `should_punish = true` (requires more than `MAX_OUTBOUND_PEERS_TO_PROTECT_FROM_DISCONNECT = 4` connected download peers, a normal IBD condition).

## Recommendation
Change line 522 in `sync/src/types/mod.rs` from:
```rust
self.timeout_count = self.task_count.saturating_add(num);
```
to:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```
This restores the intended damping: `task_count` is only decremented after `timeout_count` accumulates more than 2 slow-response events, and `timeout_count` is reset to 0 only at that point.

## Proof of Concept
**Initial state:** `task_count = 32`, `timeout_count = 0`

| Call | Buggy result | Correct result |
|------|-------------|----------------|
| `decrease(1)` | `timeout_count = 33 > 2` → `task_count = 31`, reset | `timeout_count = 1`, no change |
| `decrease(1)` | `timeout_count = 32 > 2` → `task_count = 30`, reset | `timeout_count = 2`, no change |
| `decrease(1)` | `timeout_count = 31 > 2` → `task_count = 29`, reset | `timeout_count = 3 > 2` → `task_count = 31`, reset |

After 32 slow responses, the buggy code reaches `task_count = 0` and disconnects the peer. The correct code requires ~96 slow responses for the same outcome. A unit test constructing a `DownloadScheduler`, calling `decrease(1)` three times, and asserting `task_count == 31` (not 29) would reproduce the bug.
