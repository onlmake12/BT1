### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Expunge Any User's Pending Transaction — (File: rpc/src/module/pool.rs)

---

### Summary

The `remove_transaction` JSON-RPC endpoint in `rpc/src/module/pool.rs` has no access control and accepts an arbitrary `tx_hash` parameter supplied by the caller. Any RPC-reachable client — including any local process or, if the RPC listen address is non-loopback, any network peer — can silently evict any user's pending or proposed transaction (and every descendant) from the tx-pool. This is structurally identical to the reported `USDToken::burn` pattern: no caller restriction, arbitrary victim identifier, destructive effect on another user's in-flight asset operation.

---

### Finding Description

**Root cause — `rpc/src/module/pool.rs`, lines 662–669:**

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();

    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [1](#0-0) 

There is no authentication token, no ownership check, and no restriction on which `tx_hash` values are permitted. The caller supplies an arbitrary 32-byte hash and the node unconditionally forwards it to `remove_local_tx`.

**Propagation — `tx-pool/src/service.rs`, line 273:**

```rust
pub fn remove_local_tx(&self, tx_hash: Byte32) -> Result<bool, AnyError> {
    send_message!(self, RemoveLocalTx, tx_hash)
}
``` [2](#0-1) 

**Effect — `tx-pool/src/pool.rs`, lines 358–361:**

```rust
pub(crate) fn remove_tx(&mut self, id: &ProposalShortId) -> bool {
    let entries = self.pool_map.remove_entry_and_descendants(id);
    !entries.is_empty()
}
``` [3](#0-2) 

`remove_entry_and_descendants` walks the full descendant graph and evicts every child transaction as well. [4](#0-3) 

**Exploit path:**

1. Victim submits a transaction `T` (e.g., a time-locked CKB transfer, a UDT issuance, or the first leg of an atomic swap) via `send_transaction`. `T` enters the pending or proposed pool.
2. Attacker calls `remove_transaction` with `T`'s hash. No credential is required.
3. `T` and all its descendants are silently removed from the pool.
4. If `T` carries a `since`-based absolute time constraint, or if the proposal window for `T` has already opened and is about to close, the victim's transaction may become permanently unconfirmable without re-creating the inputs — effectively destroying the in-flight operation.

---

### Impact Explanation

- **Direct:** Any RPC-reachable caller can remove any pending or proposed transaction belonging to any other user, including the entire descendant chain, with a single unauthenticated call.
- **Severe scenario:** A transaction that has already entered the proposal window (`Status::Proposed`) and whose `since` absolute-epoch or absolute-timestamp deadline is imminent can be evicted and rendered permanently invalid before the victim can resubmit and re-propose it. This is functionally equivalent to burning the capacity locked in that transaction's inputs for the duration of the attack.
- **Protocol disruption:** Multi-party protocols (payment channels, atomic swaps, DAO withdrawal chains) depend on specific transactions being confirmed in specific windows. An attacker who can evict any transaction can unilaterally abort such protocols, causing the victim to lose time-locked funds or miss withdrawal deadlines.

---

### Likelihood Explanation

- The CKB RPC listen address is operator-configurable. Nodes that bind to `0.0.0.0` (common in data-center or cloud deployments) expose this endpoint to any network peer.
- Even on a default localhost binding, any local process — including a malicious co-tenant in a shared environment — qualifies as a "supported local CLI/RPC user," which is explicitly listed as an in-scope attacker profile.
- No special knowledge beyond the victim's transaction hash is required; transaction hashes are publicly observable on the P2P relay network.

---

### Recommendation

1. **Restrict the endpoint to privileged callers only.** Add an authentication token or API key requirement to `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` — the three destructive pool-management RPCs — consistent with how node operators already gate administrative operations in other blockchain implementations.
2. **Alternatively, remove the endpoint from the public RPC surface** and expose it only via a separate, authenticated administrative interface (e.g., a Unix-domain socket or a separate port with mandatory HTTP Basic Auth).
3. **At minimum, document the security boundary** so operators know to firewall the RPC port and never bind it to a public interface.

### Proof of Concept

```bash
# Step 1: Victim submits a time-sensitive transaction
curl -X POST http://<node>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<victim_tx>, "passthrough"]}'
# Returns: {"result": "0xVICTIM_TX_HASH"}

# Step 2: Attacker (no credentials needed) removes it immediately
curl -X POST http://<node>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"remove_transaction","params":["0xVICTIM_TX_HASH"]}'
# Returns: {"result": true}

# Step 3: Victim's transaction and all descendants are gone from the pool.
# If the proposal window or since-deadline has passed, the transaction
# cannot be re-proposed and the victim's in-flight operation is permanently lost.
```

The `remove_transaction` call requires no authentication token, no proof of ownership, and no privileged role — matching the exact vulnerability pattern of `USDToken::burn`: unrestricted caller, arbitrary victim identifier, destructive outcome on another user's assets.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
