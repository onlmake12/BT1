### Title
Silent `u128`→`u64` Truncation in NervosDAO Withdrawal Capacity Calculation Produces Wrong Withdrawal Amount — (`File: util/dao/src/lib.rs`)

---

### Summary

In `DaoCalculator::calculate_maximum_withdraw`, the intermediate u128 result `withdraw_counted_capacity` is cast to `u64` with a bare `as u64` (a silent truncating cast). If the u128 value exceeds `u64::MAX`, the high bits are silently discarded, producing a drastically wrong (much smaller) withdrawal capacity. The subsequent `safe_add` may succeed without error, meaning the user's withdrawal transaction is accepted on-chain but they receive far less CKB than they deposited — a permanent loss of funds. This is inconsistent with every other analogous calculation in the same codebase, which uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The multiplication is correctly widened to `u128` to avoid overflow there. However, the final result is cast back to `u64` with `as u64` — a **silent truncating cast** in Rust. If `withdraw_counted_capacity > u64::MAX`, the value wraps modulo 2⁶⁴ with no error.

The rest of the DAO calculation code consistently uses checked conversions:

```rust
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The `calculate_maximum_withdraw` function is the sole exception. The genesis accumulate rate `ar` starts at `10_000_000_000_000_000` (10¹⁶): [4](#0-3) 

and grows monotonically each block. `ar` is a `u64` stored in the DAO field: [5](#0-4) 

**Overflow condition:** `withdraw_counted_capacity > u64::MAX` requires:

```
counted_capacity × (withdrawing_ar / deposit_ar) > 18_446_744_073_709_551_615
```

With the total CKB supply capped at ~33.6 billion CKB = ~3.36×10¹⁸ shannons, the ratio `withdrawing_ar / deposit_ar` must exceed ~5.49. Given the secondary issuance rate of ~4% per year, `ar` grows by 5.49× in roughly 43 years. This is a long-term latent defect, but the code path is live and the defect is structurally identical to the Celo report.

**Silent loss scenario:** When `withdraw_counted_capacity = u64::MAX + X` (X > 0), the truncated value is `X − 1`. Then:

```rust
Capacity::shannons(X - 1).safe_add(occupied_capacity)
```

succeeds without error (since `X − 1 + occupied_capacity ≤ u64::MAX` for all realistic `occupied_capacity` values of a few billion shannons). The user's withdrawal transaction is accepted on-chain, but they receive `X − 1 + occupied_capacity` shannons instead of the correct amount — a permanent loss of the deposited principal plus interest.

The existing test `check_withdraw_calculation_overflows` only catches the case where the truncated value + `occupied_capacity` itself overflows `u64` (triggering `safe_add`'s error path), not the silent truncation case: [6](#0-5) 

---

### Impact Explanation

A NervosDAO depositor who withdraws a sufficiently large cell (or whose deposit spans a period where `ar` has grown by ≥5.49×) will have their withdrawal transaction accepted by the network, but will receive a drastically wrong (near-zero) capacity. The deposited CKB is permanently lost — it cannot be re-withdrawn because the cell has already been consumed. This is a **permanent, silent loss of user funds** with no on-chain error signal.

The `calculate_maximum_withdraw` function is called in two places:
1. During transaction verification via `transaction_maximum_withdraw` → `transaction_fee` (consensus-critical path)
2. Via the `calculate_dao_maximum_withdraw` RPC endpoint [7](#0-6) 

---

### Likelihood Explanation

**Near-term (current mainnet):** Very low. The `ar` ratio would need to grow by 5.49× from genesis, requiring ~43 years at the current secondary issuance rate. The total CKB supply constraint makes this infeasible today.

**Long-term:** The defect becomes exploitable as `ar` grows. Any user depositing a large fraction of the total CKB supply into NervosDAO and withdrawing after sufficient time is at risk. The risk increases monotonically as `ar` grows, analogous to the Celo report's note that "as the inflation factor increases over time, this issue becomes slightly more likely."

The defect is also reachable today via the `calculate_dao_maximum_withdraw` RPC with crafted header data (e.g., in a test network or via a malicious block producer manipulating the DAO field), making it a realistic attack surface in non-mainnet environments.

---

### Recommendation

Replace the silent cast with a checked conversion, consistent with the rest of the codebase:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After (safe, consistent with rest of codebase):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

Add a test case that specifically exercises the silent truncation path (where `withdraw_counted_capacity > u64::MAX` but the truncated value + `occupied_capacity ≤ u64::MAX`) to confirm the fix catches it.

---

### Proof of Concept

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar > u64::MAX
```

With `counted_capacity = 3_360_000_000_000_000_000` (3.36×10¹⁸ shannons, ~33.6 billion CKB — the full supply), `deposit_ar = 10_000_000_000_000_000` (genesis), and `withdrawing_ar = 55_000_000_000_000_000` (5.5× genesis, reachable after ~43 years):

```
3_360_000_000_000_000_000 × 55_000_000_000_000_000 / 10_000_000_000_000_000
= 3_360_000_000_000_000_000 × 5.5
= 18_480_000_000_000_000_000
> u64::MAX (18_446_744_073_709_551_615)
```

`withdraw_counted_capacity as u64` = `18_480_000_000_000_000_000 mod 2^64` = `33_255_926_290_448_384`

`safe_add(occupied_capacity)` succeeds (no error), returning ~33.2 billion shannons (~332 CKB) instead of ~18.48×10¹⁸ shannons (~184.8 billion CKB). The user loses virtually their entire deposit silently. [8](#0-7)

    /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }

pub fn extract_dao_data(dao: Byte32) -> (u64, Capacity, Capacity, Capacity) {
    let data = dao.raw_data();
    let c = Capacity::shannons(LittleEndian::read_u64(&data[0..8]));
    let ar = LittleEndian::read_u64(&data[8..16]);
    let s = Capacity::shannons(LittleEndian::read_u64(&data[16..24]));
    let u = Capacity::shannons(LittleEndian::read_u64(&data[24..32]));
    (ar, c, s, u)
}

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
            DaoWithdrawingCalculationKind::WithdrawingOutPoint(withdrawing_out_point) => {
                let (_tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                let withdrawing_out_point: packed::OutPoint = withdrawing_out_point.into();
                let (withdrawing_tx, withdrawing_header_hash) = snapshot
                    .get_transaction(&withdrawing_out_point.tx_hash())
