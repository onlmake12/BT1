### Title
Miner Main Thread Hangs Indefinitely Due to Missing Timeout in `blocking_fetch_block_template` — (`miner/src/client.rs`)

### Summary
`blocking_fetch_block_template` blocks the miner's main thread with no timeout. A network-level attacker who can stall the TCP connection to the RPC node causes the miner to freeze permanently after finding a valid nonce.

### Finding Description
`Client::blocking_fetch_block_template` [1](#0-0)  calls `self.handle.block_on(self.fetch_block_template())`. This synchronously blocks the calling OS thread until the future resolves. The future resolves only when the HTTP response to `get_block_template` is received. The `hyper` HTTP client is constructed with no connect timeout, no read timeout, and no request timeout: [2](#0-1) 

The `Rpc::request` future awaits a `oneshot` channel that is only resolved when the HTTP response body is fully collected: [3](#0-2)  There is no `tokio::time::timeout` wrapping anywhere in this call chain.

`blocking_fetch_block_template` is called unconditionally from `submit_nonce` at `miner.rs:186`: [4](#0-3) 

`submit_nonce` is invoked directly inside the `run()` select loop: [5](#0-4)  Because `submit_nonce` is synchronous and called on the loop's OS thread, the entire `run()` loop — including `nonce_rx` and `work_rx` processing and the `stop_rx` signal — is frozen for the duration of the `block_on` call.

### Impact Explanation
After the miner finds a valid nonce, it calls `submit_nonce` → `blocking_fetch_block_template`. If the attacker stalls the TCP connection at this point, the miner's main thread never returns. Workers are stopped (`WorkerMessage::Stop` at line 141) and `WorkerMessage::Start` at line 187 is never sent. All mining workers remain idle indefinitely. The miner cannot process stop signals, new work, or any further nonces. All hashpower is wasted for the duration of the stall.

### Likelihood Explanation
Any attacker on the network path between the miner and its configured RPC node can exploit this. Miners commonly connect to remote RPC endpoints over the internet. A BGP hijack, compromised router, or ISP-level interception can establish a TCP connection that never delivers data. No credentials, keys, or privileged access to the CKB node are required. The precondition (`blocking_fetch_block_template` called after submit) is hardcoded and always true in production.

### Recommendation
Wrap the `fetch_block_template` future with `tokio::time::timeout` before passing it to `block_on`:

```rust
pub(crate) fn blocking_fetch_block_template(&self) {
    let fut = tokio::time::timeout(
        std::time::Duration::from_secs(30),
        self.fetch_block_template(),
    );
    if self.handle.block_on(fut).is_err() {
        error!("get_block_template timed out");
    }
}
```

Additionally, configure connect and read timeouts on the `HttpClient` builder at construction time. [2](#0-1) 

### Proof of Concept
1. Start a TCP server on the RPC port that accepts connections but never sends any bytes.
2. Configure the miner to point at this server.
3. Call `blocking_fetch_block_template()`.
4. Assert the call returns within 60 seconds — it will not, confirming the indefinite block.
