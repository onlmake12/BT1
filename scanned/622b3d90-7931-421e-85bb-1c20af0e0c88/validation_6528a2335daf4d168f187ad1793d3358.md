### Title
`check_tx_fee` Enforces Minimum Fee Rate Using Serialized Size Instead of Weight, Allowing Below-Minimum Fee Rate Transactions Into the Pool — (File: `tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission gate `check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the rest of the system (scoring, eviction, block assembly, fee estimation) computes fee rate using `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. An unprivileged RPC caller or P2P peer can craft a transaction with high VM cycles and small serialized size that passes the size-only admission check but carries an actual fee rate far below `min_fee_rate`, bypassing the anti-spam fee floor.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code itself acknowledges the mismatch with an inline comment:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [1](#0-0) 

The rest of the system uses `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

`TxEntry::fee_rate()` — used for pool scoring, eviction, and block assembly — uses the weight-based formula:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [4](#0-3) 

The admission check in `pre_check` calls `check_tx_fee` with only `tx_size` before script execution (cycles are unknown at that point): [5](#0-4) 

After `check_tx_fee` passes, `verify_rtx` executes the scripts and records the actual cycles. The entry is then inserted into the pool with those cycles, but no second fee-rate check against the weight-based formula is performed. [6](#0-5) 

---

### Impact Explanation

**Concrete bypass example** with default `min_fee_rate = 1000 shannons/KW` and `max_tx_verify_cycles = 70,000,000`:

| Parameter | Value |
|---|---|
| Transaction serialized size | 200 bytes |
| Transaction cycles | 70,000,000 |
| `weight = max(200, 70_000_000 × 0.000_170_571_4)` | **11,940** |
| Admission check: `min_fee = 1000 × 200 / 1000` | **200 shannons** |
| Fee paid | 200 shannons (passes admission) |
| Actual fee rate: `200 × 1000 / 11940` | **~16 shannons/KW** |
| Ratio below minimum | **~60×** |

Such a transaction:
1. Is admitted to the tx-pool, consuming memory and CPU.
2. Is relayed to all connected peers via the P2P relay path (`submit_remote_tx`), which uses the same `check_tx_fee` path on the receiving side.
3. Occupies a pool slot at a fee rate ~60× below the anti-spam floor, degrading the pool's fee-rate ordering and eviction accuracy.
4. Distorts the `estimate_fee_rate` RPC output, which uses the actual weight-based fee rate of pool entries. [7](#0-6) 

---

### Likelihood Explanation

The entry path requires only an unprivileged RPC call to `send_transaction` or a P2P relay message. The attacker must control a script that consumes a large number of cycles relative to its serialized size — straightforward with any compute-heavy RISC-V script (e.g., a tight loop). No privileged access, key material, or majority hashpower is required. The default `min_fee_rate = 1000 shannons/KW` and `max_tx_verify_cycles = 70,000,000` make the maximum bypass ratio approximately 60×. [8](#0-7) 

---

### Recommendation

Replace the size-only check in `check_tx_fee` with a weight-based check. Since cycles are not yet known at `pre_check` time, one of the following approaches should be used:

1. **Defer the fee-rate check to after script execution**: Move the `min_fee_rate` enforcement into `verify_rtx` / `submit_entry`, after cycles are known, using `get_transaction_weight(size, cycles)`.
2. **Use a conservative upper-bound weight at pre-check**: Compute `weight = get_transaction_weight(tx_size, max_tx_verify_cycles)` as a worst-case weight for the pre-check, then re-verify with actual cycles post-execution.
3. **Add a post-execution fee-rate gate**: After `verify_rtx` returns the `Completed { cycles, fee }` result, check `FeeRate::calculate(fee, get_transaction_weight(size, cycles)) >= min_fee_rate` before inserting the entry.

---

### Proof of Concept

1. Deploy a RISC-V lock script that executes a tight loop consuming ~70,000,000 cycles but compiles to a small binary (e.g., ~200 serialized bytes in the transaction).
2. Construct a transaction spending a cell locked by that script, with `outputs_capacity = inputs_capacity - 200` (fee = 200 shannons).
3. Submit via `send_transaction` RPC.
4. **Expected (broken) behavior**: `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`, the fee equals `min_fee`, the transaction is admitted.
5. **Actual fee rate**: After script execution, `cycles ≈ 70,000,000`, `weight = max(200, 11940) = 11940`, `actual_fee_rate = 200 × 1000 / 11940 ≈ 16 shannons/KW` — approximately 60× below `min_fee_rate`.
6. The transaction is now in the pool and relayed to peers, all of which admit it via the same flawed check. [9](#0-8) [10](#0-9)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
