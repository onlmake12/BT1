### Title
Peer store monopolization via identify-message address flooding with diverse /16 subnets — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
An unprivileged remote peer can fill the peer store to `ADDR_COUNT_LIMIT` (16384) by cycling short-lived connections, each advertising up to 10 addresses from distinct /16 subnets. Once full, `check_purge` enters the network-group eviction path but finds no group with >4 peers, returns `EvictionFailed`, and `add_addr` silently drops all subsequent legitimate addresses.

### Finding Description
`IdentifyCallback::add_remote_listen_addrs` unconditionally calls `peer_store.add_addr()` for every address in an identify message (up to `MAX_ADDRS=10`), with no per-peer or per-connection rate limit. `add_addr` calls `check_purge` before each insert. When the store reaches `ADDR_COUNT_LIMIT=16384`:

- **First eviction pass** removes only non-connectable addresses. Freshly injected addresses have `last_connected_at_ms=0` and `attempts_count=0`, so `is_connectable` returns `true` — they survive.
- **Second eviction pass** groups addresses by `/16` prefix (`Group::IP4([octet0, octet1])`), sorts groups by size, takes the top half, and evicts 2 random entries only from groups with `>4` peers. If the attacker placed ≤4 addresses per /16, no group qualifies, `candidate_peers` is empty, and `Err(EvictionFailed)` is returned.
- `add_addr` propagates the error via `?`, silently dropping the new address. The error is only logged in `add_remote_listen_addrs`.

### Impact Explanation
The peer store is monopolized by attacker-controlled addresses. New legitimate addresses discovered via identify or discovery protocols cannot be inserted. The node's feeler and outbound connection pools are seeded exclusively from attacker-controlled entries, degrading peer diversity and potentially isolating the node from honest peers.

### Likelihood Explanation
The attack requires cycling ~1638+ connections (achievable in minutes), each advertising 10 addresses from distinct /16 subnets. IPv4 has 65536 /16 blocks; filling 16384 slots requires only 16384 distinct /16s, well within reach. No privileged access, PoW, or key material is needed — only the ability to make inbound TCP connections to the victim node.

### Recommendation
1. **Rate-limit address ingestion per peer/session**: cap the number of addresses accepted from a single peer within a time window.
2. **Relax the eviction threshold**: remove or lower the `>4` guard so that even singleton-group entries can be evicted (e.g., always evict the oldest/lowest-scored entry from the largest group).
3. **Prefer evicting never-connected addresses first**: addresses with `last_connected_at_ms==0` should be prioritized for eviction over addresses with a successful connection history.

### Proof of Concept
```
1. Fill peer store: for i in 0..1638:
     connect to victim, send identify with 10 addrs from 10 distinct /16s (e.g., i*10.0.0.1 .. i*10.0.0.10), disconnect.
2. Assert addr_manager.count() == 16384.
3. Attempt to add one legitimate address via add_addr().
4. Observe: check_purge() → candidate_peers (first pass) = [] (all freshly added, connectable)
            → network-group pass: all groups have size 1 ≤ 4 → candidate_peers = []
            → returns Err(EvictionFailed).
5. Assert the legitimate address is NOT in the peer store.
```

    pub fn add_addr(&mut self, addr: Multiaddr, flags: Flags) -> Result<()> {
        if self.ban_list.is_addr_banned(&addr) {
            return Ok(());
        }
        self.check_purge()?;
        let score = self.score_config.default_score;
        self.addr_manager
            .add(AddrInfo::new(addr, 0, score, flags.bits()));
        Ok(())
    }

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
