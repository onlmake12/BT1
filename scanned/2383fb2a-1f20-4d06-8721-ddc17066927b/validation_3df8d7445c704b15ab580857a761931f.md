Audit Report

## Title
Missing Per-Peer Rate Limit in BlockFilter Protocol Handler Enables Unbounded DB Read Amplification — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

## Summary
The `BlockFilter` protocol handler contains no per-peer rate limiter. Any peer connected via the Filter protocol can send `GetBlockFilters`, `GetBlockFilterHashes`, or `GetBlockFilterCheckPoints` messages in a tight loop, each triggering up to 2000 RocksDB reads and a large molecule-encoded response, with no throttle applied. The `Relayer` and `HolePunching` handlers both have explicit `governor`-based rate limiters; the `BlockFilter` handler has none, creating a directly exploitable asymmetry.

## Finding Description
The `BlockFilter` struct holds only `shared: Arc<SyncShared>` with no rate limiter field: [1](#0-0) 

Its `try_process` dispatches directly to all three process handlers with no rate check at any point: [2](#0-1) 

`GetBlockFiltersProcess::execute()` iterates up to `BATCH_SIZE = 1000` blocks, performing two DB reads per block (`get_block_hash` + `get_block_filter`) and accumulating up to 1.8 MB of response data per request: [3](#0-2) [4](#0-3) 

`GetBlockFilterHashesProcess::execute()` and `GetBlockFilterCheckPointsProcess::execute()` are similarly unguarded, each with `BATCH_SIZE = 2000` and two DB reads per iteration: [5](#0-4) [6](#0-5) 

By contrast, `Relayer` explicitly installs a `governor` rate limiter keyed by `(PeerIndex, message_item_id)` at 30 req/s and checks it before every dispatch: [7](#0-6) [8](#0-7) 

`HolePunching` does the same at the `received` entry point: [9](#0-8) 

No equivalent guard exists anywhere in `sync/src/filter/`.


## Impact Explanation
A single attacker peer connecting via the Filter protocol can saturate the node's RocksDB read throughput and async task queue. `GetBlockFilters` costs up to 2000 DB reads and ~1.8 MB of encoding per request; `GetBlockFilterHashes` and `GetBlockFilterCheckPoints` each cost up to 4000 DB reads per request. At even modest send rates this monopolizes the I/O path shared with consensus-critical Relay and Sync protocol messages, causing processing delays or drops for those messages. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
The Filter protocol is optional but is enabled on nodes that serve light clients. The attack requires only a standard P2P connection — no credentials, no PoW, no stake. The attacker sends a fixed 9-byte `GetBlockFilters(start_number=0)` message in a tight loop. The gap is directly analogous to the one the `Relayer` rate limiter was added to close, and the attack surface is wider because three message types are unprotected instead of one.

## Recommendation
Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer::rate_limiter` at `sync/src/relayer/mod.rs` L81–98) and check it at the top of `try_process` before dispatching any of the three request message types (`GetBlockFilters`, `GetBlockFilterHashes`, `GetBlockFilterCheckPoints`), returning `StatusCode::TooManyRequests` on failure. Call `rate_limiter.retain_recent()` in the `disconnected` handler to bound memory growth.

## Proof of Concept
1. Enable block filter service on a test node; mine ≥1000 blocks so filters are built.
2. Connect a test peer using `SupportProtocols::Filter`.
3. Send `GetBlockFilters { start_number: 0 }` in a tight loop (no sleep).
4. Observe: each message triggers up to 2000 RocksDB reads and a ~1.8 MB `async_send_message_to` call with no throttle.
5. Repeat with `GetBlockFilterHashes { start_number: 0 }` (up to 4000 DB reads per message) and `GetBlockFilterCheckPoints { start_number: 0 }` (up to 4000 DB reads per message).
6. Measure latency of concurrent Relay/Sync messages — assert they are delayed proportionally to the flood rate.
7. Assert no `TooManyRequests` status is ever returned by the Filter handler (confirmed by code: no such check exists in `sync/src/filter/mod.rs` or any of its sub-processors).
