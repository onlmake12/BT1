Looking at the exact arithmetic and all downstream uses of `last_n_blocks` after the guard.

### Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Allows Limit Bypass — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

The guard at line 201 uses `(last_n_blocks as usize) * 2` without overflow protection. In a release build, Rust wraps on integer overflow, so a peer sending `last_n_blocks = 2^63` causes the multiplication to produce `0`, silently bypassing the 1000-sample limit. The server then processes up to `last_block_number - start_block_number` blocks (the entire chain range), each requiring multiple expensive DB operations.

### Finding Description

`GetLastStateProofProcess::execute` reads `last_n_blocks` from the peer message and immediately uses it in an unchecked arithmetic expression:

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

On a 64-bit target `usize = u64`. When `last_n_blocks = 2^63`, `(2^63 as usize) * 2` wraps to `0` in release mode. The check becomes `difficulties.len() + 0 > 1000`, which is `false` for any normally-sized `difficulties` list. The guard returns no error and execution continues with the original huge `last_n_blocks` value.

Downstream, the branch at line 291 evaluates:

```rust
if last_block_number - start_block_number <= last_n_blocks {
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [2](#0-1) 

Because no real chain can have `2^63` blocks, this condition is always `true` for the overflow value, so the server collects every block number from `start_block_number` to `last_block_number`. For each collected block number, `complete_headers` performs `get_ancestor`, `get_block`, and `chain_root_mmr` — all DB-backed operations: [3](#0-2)

### Impact Explanation

An unprivileged remote light-client peer can force the server to perform O(chain_height) DB reads and MMR computations per message instead of the intended O(1000). On a chain with millions of blocks, this is orders of magnitude more work than the limit intends to allow. Multiple peers sending such messages concurrently can exhaust I/O, CPU, and memory, constituting a practical resource-exhaustion DoS against the light-client protocol server.

### Likelihood Explanation

The exploit requires only crafting a single P2P `GetLastStateProof` message with `last_n_blocks` set to any value `V` where `(V as usize).wrapping_mul(2)` produces a result ≤ `1000 - difficulties.len()`. The simplest value is `2^63` (producing `0`). No authentication, no hashpower, and no privileged access is required. Release builds are the standard deployment mode, so the wrapping behavior is always present in production.

### Recommendation

Replace the unchecked multiplication with a saturating or checked variant:

```rust
// Option A: saturating_mul — safe, idiomatic
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: reject immediately if last_n_blocks alone exceeds the limit
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT / 2 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Either approach closes the bypass without changing the intended semantics.
