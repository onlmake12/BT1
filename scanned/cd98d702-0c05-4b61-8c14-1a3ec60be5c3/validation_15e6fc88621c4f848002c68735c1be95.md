### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Unique `(from, to)` PeerId Pairs in `ConnectionRequestDelivered` — (`network/src/protocols/hole_punching/mod.rs`)

### Summary

The `HolePunching` protocol's `forward_rate_limiter` uses a `governor::RateLimiter` backed by a `HashMapStateStore<(PeerId, PeerId, u32)>`. Every `ConnectionRequestDelivered` message with a novel `(from, to)` pair inserts a new entry into this map. The map is only pruned via `retain_recent()` in the `disconnected` handler. A long-lived session that never disconnects can grow this map without bound, causing OOM.

### Finding Description

**Rate limiter types and keys:**

`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` with a quota of 1 req/s: [1](#0-0) [2](#0-1) 

**The check that inserts entries:**

In `ConnectionRequestDeliveredProcess::execute()`, the `forward_rate_limiter` is checked with the attacker-supplied `content.from` and `content.to` values: [3](#0-2) 

`governor`'s `check_key` inserts a new bucket into the `HashMapStateStore` on first access for any key. Since `from` and `to` are parsed from raw message bytes with no constraint beyond being valid `PeerId` multihashes, an attacker can supply an unlimited number of distinct pairs.

**No validation of `from`/`to` against known peers:** [4](#0-3) 

The only check is that the bytes decode as a valid `PeerId`. There is no requirement that `from` or `to` correspond to any connected or known peer.

**`retain_recent()` is only called on disconnect:** [5](#0-4) 

**The `notify()` timer (every 5 minutes) does NOT call `retain_recent()`:** [6](#0-5) 

`pending_delivered` and `inflight_requests` are pruned on the timer, but neither rate limiter is.

**The outer per-session rate limiter is the only throttle:** [7](#0-6) 

This limiter is keyed by `(session_id, msg_item_id)` and allows 30 msg/s. It does not prevent unique `(from, to)` pairs — it only caps the total message rate per session.

### Impact Explanation

At 30 msg/s, an attacker inserts 30 new `(PeerId, PeerId, u32)` entries per second into `forward_rate_limiter`. A `PeerId` is a 39-byte multihash. Each map entry carries the key (~82 bytes) plus `governor`'s internal `AtomicU64` state plus `HashMap` overhead (~100–150 bytes total). Over 24 hours of a single persistent connection: ~2.6 million entries ≈ hundreds of MB. Multiple concurrent attacker sessions multiply this linearly. The node has no mechanism to reclaim this memory until the session disconnects, which the attacker controls.

### Likelihood Explanation

Any unprivileged peer that can establish a P2P connection can execute this. No PoW, no key material, no privileged role is required. The attacker only needs to maintain a TCP connection and send well-formed `ConnectionRequestDelivered` messages with fresh random `PeerId` bytes in the `from`/`to` fields. This is trivially scriptable.

### Recommendation

1. **Periodic cleanup**: Call `self.forward_rate_limiter.retain_recent()` inside the `notify()` handler (every 5 minutes) rather than only on disconnect.
2. **Bound the map size**: Add a hard cap on the number of entries in `forward_rate_limiter` (e.g., reject or evict when the map exceeds a threshold like 10,000 entries).
3. **Validate `from`/`to` against known peers**: Before inserting into the rate limiter, check that `from` and `to` correspond to peers in the peer registry. Unknown peer IDs should be rejected early, before any state is allocated.

### Proof of Concept

```
1. Connect a single TCP session to the victim CKB node.
2. In a loop, construct ConnectionRequestDelivered messages where:
   - `from` = random valid PeerId (39 random bytes formatted as a valid multihash)
   - `to`   = random valid PeerId (distinct from `from`)
   - `listen_addrs` = one valid address (to pass the length check)
   - `route` = empty (to pass the hop check)
3. Send at ~30 msg/s (within the per-session rate_limiter quota).
4. Never disconnect.
5. After N seconds, the forward_rate_limiter internal HashMap contains N*30 entries.
6. After ~1 hour: ~108,000 entries. After ~24 hours: ~2.6M entries.
7. Monitor victim RSS; it grows monotonically without bound.
```

The invariant test described in the question (send 10^6 messages with unique `(from, to)` pairs over a single session without disconnecting and assert map size is bounded) would fail against the current implementation.
