### Title
Unbounded O(N) Tx Pool Read Lock Hold in `FetchTxsWithCycles` via Cache-Miss `GetRelayTransactions` — (`sync/src/relayer/get_transactions_process.rs`)

---

### Summary

An unprivileged remote peer can send `GetRelayTransactions` messages containing up to 32,767 random tx hashes absent from the pool. The tx pool service handler acquires a read lock and iterates over every requested hash with no early termination, performing O(N) hash map lookups while holding the lock, then returns an empty result. With a modest number of peers operating at the per-peer rate limit, the single-threaded service loop saturates, delaying all other tx pool operations including block template generation and transaction submission.

---

### Finding Description

**Entry point — `GetTransactionsProcess::execute`:**

The only guard on message size is a ceiling check against `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`. Any message at or below that limit is forwarded unconditionally to `fetch_txs_with_cycles`. [1](#0-0) [2](#0-1) 

**Service handler — `FetchTxsWithCycles`:**

The handler acquires the tx pool read lock, then unconditionally iterates over every requested `short_id`, calling `get_tx_with_cycles` (which calls `pool_map.get_by_id`) for each one. When all hashes are absent (cache-miss), the iterator produces nothing, but all N lookups still execute before the lock is released. [3](#0-2) [4](#0-3) 

**Rate limiter — present but insufficient:**

A per-`(peer, message_type)` rate limiter caps each peer at 30 `GetRelayTransactions` messages per second. [5](#0-4) 

This does not prevent the amplification: 30 req/s × 32,767 hashes = ~983,000 hash map lookups per second **per peer**. At ~100 ns per lookup, one peer consumes ~98 ms of service CPU per second (~10% utilization). Ten peers saturate the service.

**Service is single-threaded with a bounded channel:**

The tx pool service processes all `Message` variants sequentially in one async task. The channel capacity is `DEFAULT_CHANNEL_SIZE = 512`. [6](#0-5) 

When the channel fills, callers receive send errors. But before that point, every queued `FetchTxsWithCycles` message holds the read lock for its full O(N) iteration, serializing all other service operations behind it.

---

### Impact Explanation

- **Tx pool read lock contention**: The read lock is held for the entire O(32,767) iteration. Write-lock waiters (e.g., `SubmitRemoteTx`, chain reorg) are blocked for each such iteration.
- **Service queue saturation**: With 10–20 peers each at 30 req/s, the service loop is fully occupied processing empty-result iterations, starving `BlockTemplate`, `SubmitLocalTx`, and `SubmitRemoteTx` messages.
- **Zero attacker cost**: The attacker sends random 32-byte hashes; no valid transaction, PoW, or fee is required.
- **No benefit required**: The attack is pure resource consumption — the attacker gains nothing from the empty responses.

---

### Likelihood Explanation

Establishing 10–20 peer connections from distinct IPs is trivially achievable by any motivated attacker and does not require a Sybil attack, majority hashpower, or privileged access. The `GetRelayTransactions` message type is a standard relay protocol message accepted from any connected peer.

---

### Recommendation

1. **Add a per-message hash count limit lower than 32,767** for `GetRelayTransactions` specifically (e.g., 256 or 512), separate from the `RelayTransactionHashes` broadcast limit.
2. **Bound the work inside `FetchTxsWithCycles`**: process at most N hashes per call (e.g., 256), or return early once a configurable lookup budget is exhausted.
3. **Charge the rate limiter per hash, not per message**: count each requested hash against the peer's quota so that a 32,767-hash message costs 32,767 tokens.
4. **Consider a stricter per-peer service-queue depth limit** for `FetchTxsWithCycles` messages specifically.

---

### Proof of Concept

```
1. Connect to a CKB node as a relay peer (SupportProtocols::RelayV3).
2. In a tight loop (≤30 iterations/second per peer), send:
     GetRelayTransactions { tx_hashes: [random_hash_1, ..., random_hash_32767] }
   where all hashes are random 32-byte values not present in the pool.
3. Repeat from 10 concurrent peer connections.
4. Observe: block template RPC latency increases, SubmitLocalTx latency increases,
   tx pool service channel approaches DEFAULT_CHANNEL_SIZE=512 depth.
5. Assert: tx pool read lock hold time scales linearly with the number of requested
   hashes, independently of how many are found.
```

The root cause is confirmed at: [7](#0-6) 

— the read lock is acquired once and held across the full `into_iter().filter_map(...)` chain with no early exit or per-hash budget.
