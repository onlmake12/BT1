### Title
Legacy `revealWithCallback` Path Permanently Traps Entropy Requests When Consumer Callback Reverts — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, `revealWithCallback` has two execution paths gated by `req.gasLimit10k`. The **new path** (`gasLimit10k != 0`) uses `excessivelySafeCall` and a `CALLBACK_FAILED` recovery state. The **legacy path** (`gasLimit10k == 0`, triggered when a provider has `defaultGasLimit == 0`) clears the request *before* calling `_entropyCallback` with no error handling. If the callback reverts, the entire transaction reverts (undoing the clear), leaving the request permanently stuck with no protocol-level recovery mechanism.

---

### Finding Description

`revealWithCallback` branches at line 574:

```solidity
if (
    req.gasLimit10k != 0 &&
    req.callbackStatus == EntropyStatusConstants.CALLBACK_NOT_STARTED
) {
    // NEW PATH: excessivelySafeCall + CALLBACK_FAILED state
} else {
    // LEGACY PATH: clearRequest first, then bare callback call
    clearRequest(provider, sequenceNumber);          // line 666
    // ...
    IEntropyConsumer(callAddress)._entropyCallback(  // line 676
        sequenceNumber, provider, randomNumber
    );
    // no try/catch — revert propagates, undoing clearRequest
}
``` [1](#0-0) [2](#0-1) 

The legacy path is entered whenever `req.gasLimit10k == 0`. This is set at request time when the provider's `defaultGasLimit` is zero:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0;  // opts out of new callback failure state flow
}
``` [3](#0-2) 

A user can call `requestWithCallback` against any such legacy provider. The `callbackStatus` is set to `CALLBACK_NOT_STARTED`, and `gasLimit10k` is 0. When the provider later calls `revealWithCallback`:

1. `clearRequest` executes (line 666).
2. `_entropyCallback` is called on the consumer contract (line 676) with no `try/catch`.
3. If the consumer reverts, the entire transaction reverts — including the `clearRequest`.
4. The request returns to `CALLBACK_NOT_STARTED` with `gasLimit10k == 0`.
5. Every subsequent `revealWithCallback` attempt repeats the same revert.

There is no admin function, no `CALLBACK_FAILED` state, and no force-clear path for legacy requests. The request is permanently stuck.

The new path explicitly documents this design split:

```solidity
// Requests that haven't been invoked yet will be invoked safely (catching reverts),
// and any reverts will be reported as an event. Any failing requests move to a failure state
// at which point they can be recovered.
``` [4](#0-3) 

The legacy path has no equivalent protection.

---

### Impact Explanation

- **Permanently stuck requests**: Any request made to a legacy provider (`defaultGasLimit == 0`) whose consumer callback always reverts is irrecoverable. The request occupies a storage slot indefinitely.
- **Provider DoS**: The `allocRequest` function uses a fixed-size circular buffer (`NUM_REQUESTS` slots). An attacker who fills all slots with stuck legacy requests prevents any new requests from being allocated for that provider, effectively DoS-ing the provider's randomness service.
- **Fee loss**: The user's fee is credited to the provider at request time (`providerInfo.accruedFeesInWei += providerFee`), so the user loses their fee with no randomness delivered and no refund path. [5](#0-4) 

---

### Likelihood Explanation

- Any unprivileged user can trigger this: deploy a consumer contract whose `_entropyCallback` always reverts, then call `requestWithCallback` against a legacy provider.
- Legacy providers (`defaultGasLimit == 0`) are still fully supported by the contract and may be in active use.
- The attack cost is `NUM_REQUESTS × fee`. On low-fee chains this is negligible.
- No privileged access, no key compromise, and no external oracle manipulation is required.

---

### Recommendation

1. **Wrap the legacy callback in a `try/catch`** and transition to a `CALLBACK_FAILED`-equivalent state, mirroring the new path's behavior.
2. **Add a protocol-level force-clear function** (callable by admin or the provider) to recover permanently stuck legacy requests.
3. **Deprecate the legacy path**: require all providers to set a non-zero `defaultGasLimit` before accepting new `requestWithCallback` calls, so all new requests use the safe path.

---

### Proof of Concept

1. Provider registers with `defaultGasLimit == 0` (legacy mode). `req.gasLimit10k` will be 0 for all their requests.
2. Attacker deploys `MaliciousConsumer` whose `_entropyCallback` always reverts.
3. Attacker calls `requestWithCallback(provider, userCommitment)` from `MaliciousConsumer` — repeats `NUM_REQUESTS` times, filling all storage slots.
4. Provider calls `revealWithCallback(provider, seqN, userContrib, providerContrib)` for each request.
5. Each call hits the legacy `else` branch: `clearRequest` runs, then `_entropyCallback` reverts, rolling back `clearRequest`.
6. All `NUM_REQUESTS` slots remain permanently occupied in `CALLBACK_NOT_STARTED` state.
7. Any new `requestWithCallback` call to this provider reverts with `OutOfRandomness` or slot exhaustion — the provider's service is DoS'd. [2](#0-1) [6](#0-5)
