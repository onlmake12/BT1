### Title
No Maximum Fee Safeguard in Entropy Request Functions Allows Provider Front-Running to Capture Excess User ETH - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `requestHelper` function accepts any `msg.value ≥ requiredFee` but provides no mechanism for callers to specify a maximum acceptable fee. All ETH beyond `providerFee` is permanently credited to Pyth's internal fee pool — it is never refunded. Because `setProviderFee` takes effect immediately with no timelock, a provider (or its fee manager) can front-run a pending `requestV2` transaction to raise the fee to exactly the amount the user sent, capturing the user's entire buffer as provider revenue.

---

### Finding Description

`requestHelper` in `Entropy.sol` performs the following fee accounting:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

Every wei above `providerFee` is swept into `accruedPythFeesInWei`. The NatSpec on every `requestV2` variant explicitly warns: *"excess value is not refunded to the caller."* [2](#0-1) 

`setProviderFee` is an unrestricted, immediately-effective call — no timelock, no delay:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
}
``` [3](#0-2) 

`getFeeV2` computes the total fee as `getProviderFee(provider, gasLimit) + _state.pythFeeInWei`: [4](#0-3) 

None of the four `requestV2` overloads or `requestWithCallback` accept a `maxFee` parameter: [5](#0-4) 

Because the Fortuna keeper (`apps/fortuna/src/keeper/fee.rs`) automatically adjusts provider fees in response to gas price changes, fee fluctuations are a normal, frequent occurrence. Callers who send a buffer above the queried fee to avoid reverts are structurally exposed. [6](#0-5) 

---

### Impact Explanation

A malicious provider (or its fee manager) can:

1. Observe a pending `requestV2` transaction in the mempool with `msg.value = F + buffer`.
2. Front-run it with `setProviderFee(F + buffer - pythFee)`.
3. The user's transaction executes: `requiredFee = F + buffer`, `msg.value = F + buffer`.
4. `providerInfo.accruedFeesInWei += F + buffer - pythFee` — the provider captures the entire buffer.
5. `accruedPythFeesInWei += pythFee` — Pyth receives only its normal cut.
6. The user loses the buffer with no recourse.

Even without deliberate front-running, any legitimate fee increase that occurs between a user's `getFeeV2` call and their `requestV2` execution causes the user to overpay with no refund, because the contract silently accepts any `msg.value ≥ requiredFee` and retains the surplus.

---

### Likelihood Explanation

- Providers are explicitly listed in the Immunefi scope as potential attackers ("Entropy user/provider").
- `setProviderFee` requires no privileged role beyond being the registered provider or its fee manager — both are reachable by the attacker.
- The Fortuna keeper adjusts fees automatically and frequently, so users routinely add a buffer to avoid reverts. This buffer is the attack surface.
- No timelock or delay on `setProviderFee` means same-block front-running is trivially achievable on any EVM chain where Entropy is deployed.

---

### Recommendation

1. **Add a `maxFee` parameter** to all `requestV2` and `requestWithCallback` variants. Revert if `requiredFee > maxFee`.
2. **Refund excess ETH** to `msg.sender` when `msg.value > requiredFee`, replacing the current pattern of sweeping the surplus into `accruedPythFeesInWei`.
3. Alternatively, introduce a **fee-change timelock** on `setProviderFee` so users can rely on the fee they queried remaining valid for at least one block.

---

### Proof of Concept

```
Setup:
  pythFee = 100 wei
  provider.feeInWei = 900 wei
  getFeeV2(provider, 0) = 1000 wei

Step 1: User calls getFeeV2 → 1000 wei.
Step 2: User submits requestV2{value: 1500}(provider, userRandom, 0)
        (500 wei buffer to avoid InsufficientFee reverts from fee changes).

Step 3: Provider front-runs with setProviderFee(1400).
        New getFeeV2 = 1400 + 100 = 1500 wei.

Step 4: User's tx executes:
        requiredFee = 1500, msg.value = 1500 → passes check.
        providerInfo.accruedFeesInWei += 1400   // provider captures 1400
        accruedPythFeesInWei += 1500 - 1400 = 100

Result: User paid 1500 instead of 1000.
        Provider gained an extra 500 wei.
        User has no recourse; no refund is issued.
```

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[

        }
        uint128 oldFeeInWei = provider.feeInWei;
