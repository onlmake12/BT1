### Title
Unauthenticated Privileged RPC Methods with Permissive CORS Allow Cross-Origin Pool Manipulation — (File: `rpc/src/server.rs`, `rpc/src/module/pool.rs`)

---

### Summary

The CKB JSON-RPC server is built with `CorsLayer::permissive()` and no authentication middleware, while the `Pool` module exposes destructive, privileged state-changing methods (`clear_tx_pool`, `remove_transaction`, `clear_tx_verify_queue`) with zero caller authentication. Any web page loaded in a browser on the same machine as a running CKB node can issue cross-origin `POST` requests to `http://127.0.0.1:8114` and invoke these methods, wiping the entire transaction pool or selectively evicting specific transactions without any credential or token. This is a direct analog to the reported vulnerability class: privileged actions reachable without authentication.

---

### Finding Description

**Root cause — no authentication layer on the RPC server:**

`rpc/src/server.rs` builds the Axum router with `CorsLayer::permissive()` and no authentication middleware:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())          // ← allows any origin
    .layer(TimeoutLayer::with_status_code(
        StatusCode::REQUEST_TIMEOUT,
        Duration::from_secs(30),
    ))
    .layer(Extension(stream_config));
```

There is no `Authorization` header check, no token validation, and no IP allowlist enforced at the handler level. Every request that reaches the socket is dispatched directly to the RPC method.

**Privileged methods with no authentication:**

`rpc/src/module/pool.rs` exposes `clear_tx_pool`, `remove_transaction`, and `clear_tx_verify_queue` as plain synchronous RPC methods. Their implementations perform immediate, irreversible state mutations with no caller identity check:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}

fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

**Exploit path — CSRF via permissive CORS:**

The default `listen_address = "127.0.0.1:8114"` binds to loopback. `CorsLayer::permissive()` sets `Access-Control-Allow-Origin: *` and allows cross-origin `POST` with `Content-Type: application/json`. A browser will send a cross-origin `POST` to `http://127.0.0.1:8114` from any origin and receive the response, because the server explicitly permits it. No preflight rejection occurs.

An attacker hosts a page with:

```javascript
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    id: 1, jsonrpc: "2.0",
    method: "clear_tx_pool", params: []
  })
});
```

When a CKB node operator visits any attacker-controlled page (ad network, compromised site, etc.), this request executes silently and the entire mempool is wiped. The same technique applies to `remove_transaction` (targeted eviction of a specific tx by hash) and `clear_tx_verify_queue`.

---

### Impact Explanation

- **`clear_tx_pool`**: Atomically removes every pending and proposed transaction from the mempool. All user-submitted transactions are lost and must be rebroadcast. A loop calling this every few seconds constitutes a sustained denial-of-service against transaction confirmation for the node's users.
- **`remove_transaction`**: Allows targeted eviction of a specific transaction (and all its descendants) by hash. An attacker who knows a victim's pending transaction hash (observable via `get_raw_tx_pool`, also unauthenticated) can repeatedly evict it, preventing confirmation.
- **`clear_tx_verify_queue`**: Drains the verification queue, stalling the pipeline that admits new transactions.

Impact: **Service unavailability / severe degradation** — persistent mempool disruption for any node whose operator visits a malicious page. For a mining node, this also disrupts block assembly.

---

### Likelihood Explanation

- The RPC binds to `127.0.0.1` by default, but `CorsLayer::permissive()` makes it reachable from any browser tab on the same machine.
- Node operators routinely browse the web on the same machine running their node.
- The `Pool` module is enabled by default in the production config (`modules = ["Net", "Pool", "Miner", "Chain", ...]`).
- No credential, token, or secret is required — the attack payload is a single unauthenticated JSON-RPC call.
- The attacker needs only to serve a page that the operator loads; no leaked keys, no privileged access, no 51% hashpower.

Likelihood: **High** — standard CSRF against a localhost service with permissive CORS, requiring only that the operator opens a browser.

---

### Recommendation

1. **Add an authentication layer to the RPC server.** Require a configurable bearer token or HTTP Basic credential on all state-mutating methods. The miner client already implements `parse_authorization` for outbound calls (`miner/src/client.rs:380`); the server side has no equivalent enforcement.
2. **Replace `CorsLayer::permissive()` with a restrictive CORS policy** that only allows `null` or `127.0.0.1` origins, or disables CORS entirely for the HTTP listener.
3. **Separate privileged methods** (`clear_tx_pool`, `remove_transaction`, `clear_tx_verify_queue`) behind an explicit authorization gate, independent of module-level enable/disable.
