### Title
Unprivileged Peer Can Trigger Unbounded NAT Traversal Task Spawning via Crafted `ConnectionRequestDelivered` — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

An unprivileged connected peer can craft a `ConnectionRequestDelivered` message that passes all validation checks and reaches the `try_nat_traversal` branch, spawning up to 24 concurrent long-lived TCP connection tasks per exploited `inflight_requests` entry. The attacker learns valid `to` peer IDs by observing the victim's own `ConnectionRequest` gossip broadcasts, then spoofs the `from` field with the victim's public peer ID. The `inflight_requests` map is consumed entry-by-entry via `remove()`, bounding per-cycle exploitation to K entries, but the attack repeats every 5-minute `notify()` cycle. Critically, the attacker fully controls the `listen_addrs` target IPs, turning the victim into a TCP connection amplifier toward arbitrary hosts.

---

### Finding Description

**Step 1 — Victim populates `inflight_requests`.**

Every 5 minutes, `HolePunching::notify()` calls `fetch_nat_addrs()`, builds `ConnectionRequest` messages, and gossip-broadcasts them to `sqrt(total)` peers. For each broadcast, the target `to_peer_id` is inserted into `inflight_requests`. [1](#0-0) 

**Step 2 — Attacker learns `to` peer IDs.**

Because the attacker is a connected peer, they are among the `sqrt(total)` recipients of the `ConnectionRequest` gossip. The `to` field in that message is the exact key stored in `inflight_requests`.

**Step 3 — Attacker crafts a `ConnectionRequestDelivered` message.**

The attacker constructs a message with:
- `from` = victim's own public peer ID (known from P2P handshake/identify)
- `to` = observed peer ID from the `ConnectionRequest` broadcast
- `route` = `[]` (empty, so routing falls through to the terminal branch)
- `listen_addrs` = up to 24 attacker-controlled TCP addresses

**Step 4 — All validation gates pass.**

| Check | Outcome |
|---|---|
| Malformed message ban | Passes — attacker constructs valid molecule encoding |
| `rate_limiter` (per session, per msg type) | Passes — 30 req/s budget, one message suffices |
| `listen_addrs.len() <= 24` | Passes — attacker sets exactly 24 |
| `route.len() <= MAX_HOPS` | Passes — empty route |
| `forward_rate_limiter` keyed `(from, to, item_id)` | Passes — 1/s, one message suffices |
| `route.last() == None` | Passes — empty route |
| `self_peer_id == content.from` | Passes — attacker sets `from` = victim's own peer ID |
| `inflight_requests.remove(&content.to)` | Returns `Some(start)` — attacker used observed peer ID | [2](#0-1) 

**Step 5 — `try_nat_traversal` spawns tasks.**

`try_nat_traversal` filters the 24 addresses to TCP+IP ones and calls `runtime::spawn` with a future that:
1. Sleeps for `ttl/2` milliseconds
2. Runs `select_ok(24 tasks)` — 24 concurrent `try_nat_traversal` futures [3](#0-2) 

**Step 6 — Each spawned task loops for 30 seconds.**

Each `try_nat_traversal` future retries TCP connects every ~200ms for a hard 30-second timeout, creating a new `TcpSocket` on every iteration. [4](#0-3) 

---

### Impact Explanation

Per triggered NAT traversal (one consumed `inflight_requests` entry):
- 24 concurrent async tasks spawned, each running up to 30 seconds
- Each task creates a new `TcpSocket` every ~200ms → ~150 socket allocations per task over 30 s
- 24 tasks × 150 attempts = **~3,600 outbound TCP SYN packets** toward attacker-controlled IPs

With K inflight entries per 5-minute cycle (bounded by `max_outbound - non_whitelist_outbound`, typically 8–10):
- **K × 24 tasks** spawned per cycle
- **K × 3,600 TCP SYN packets** sent to arbitrary attacker-chosen hosts

The attacker fully controls `listen_addrs`, so the victim becomes a **TCP SYN amplifier** toward arbitrary third-party hosts. The claimed "exhausting OS fd limits" is overstated for typical K values (~10), but the unauthorized outbound connection storm and DDoS amplification primitive are concrete and repeatable every 5 minutes.

---

### Likelihood Explanation

- Requires only a single connected peer — no special privilege, no PoW, no key material
- Victim's peer ID is public (exchanged during identify handshake)
- `ConnectionRequest` gossip is received by `sqrt(total)` peers; attacker has a non-trivial probability of being in that set
- `inflight_requests.remove()` limits each entry to one exploitation per cycle, but the cycle repeats every 5 minutes indefinitely
- No authentication or signature on `ConnectionRequestDelivered` messages verifying they traversed the legitimate relay path

---

### Recommendation

1. **Authenticate the delivery path**: Include a relay-chain signature or nonce in `ConnectionRequestDelivered` that the originating node can verify was produced by the actual relay chain, not injected directly by a connected peer.
2. **Verify sender session**: The terminal branch (`self_peer_id == content.from`) should only be reachable from a session that is known to be a legitimate relay, not from any arbitrary connected peer.
3. **Cap concurrent NAT traversal tasks**: Maintain a global semaphore or task counter to bound the total number of simultaneously live NAT traversal tasks regardless of how many `inflight_requests` entries are consumed.
4. **Restrict `listen_addrs` to addresses associated with the `to` peer**: Validate that the IP addresses in `listen_addrs` are consistent with previously observed addresses for `content.to` in the peer store, preventing use of arbitrary third-party IPs.

    fn try_nat_traversal(&self, ttl: u64, remote_addrs: Vec<Multiaddr>) {
        let tasks = remote_addrs
            .into_iter()
            .filter_map(|listen_addr| match find_type(&listen_addr) {
                TransportType::Tcp => {
                    if listen_addr
                        .iter()
                        .any(|p| matches!(p, Protocol::Ip4(_) | Protocol::Ip6(_)))
                    {
                        Some(Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
                    } else {
                        None
                    }
                }
                TransportType::Memory
                | TransportType::Onion
                | TransportType::Ws
                | TransportType::Wss
                | TransportType::Tls => None,
            })
            .collect::<Vec<_>>();

        if tasks.is_empty() {
            return;
        }

        debug!("start NAT traversal");

        let control = self.p2p_control.clone();

        runtime::spawn(async move {
            runtime::delay_for(std::time::Duration::from_millis(ttl / 2)).await;
            if let Ok(((stream, addr), _)) = select_ok(tasks).await {
                debug!("NAT traversal success, addr: {:?}", addr);
                if let Some(metrics) = ckb_metrics::handle() {
                    metrics.ckb_hole_punching_active_success_count.inc();
                }
                let _ignore = control
                    .raw_session(
                        stream,
                        addr,
                        RawSessionInfo::outbound(TargetProtocol::Single(
                            SupportProtocols::Identify.protocol_id(),
                        )),
                    )
                    .await;
            }
        });
    }
