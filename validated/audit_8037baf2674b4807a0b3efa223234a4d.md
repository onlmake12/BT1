### Title
Griefing Attack on Entropy Provider Reveals via Unbounded `numHashes` Loop in `constructProviderCommitment` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `Entropy.sol` contract allows providers to register with `feeInWei = 0` and imposes no minimum fee floor. An unprivileged attacker can flood the contract with minimum-cost requests against such a provider. Each request increments the provider's `sequenceNumber` and increases the `numHashes` stored in every subsequent request. When the provider attempts to fulfill any request via `revealWithCallback` or `reveal`, the internal `constructProviderCommitment` function iterates `numHashes` times in an unbounded `while` loop. With enough outstanding requests, the gas cost of any reveal transaction exceeds the block gas limit, permanently DoS-ing the provider's ability to fulfill randomness.

---

### Finding Description

**Root cause 1 — No minimum fee in `register()`:**

The `register()` function accepts any `feeInWei` value including zero, with no lower-bound validation:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    ...
    provider.feeInWei = feeInWei;   // no minimum check
``` [1](#0-0) 

Similarly, `setProviderFee` allows reducing the fee to zero at any time with no floor check. [2](#0-1) 

**Root cause 2 — `numHashes` grows with outstanding requests:**

In `requestHelper`, each new request records `numHashes` as the distance from the provider's last revealed commitment to the current sequence number:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
``` [3](#0-2) 

Every unfulfilled request increases this distance for the next request. With N outstanding requests, the N-th request has `numHashes = N`.

**Root cause 3 — Unbounded loop in `constructProviderCommitment`:**

During `revealHelper` (called by both `reveal` and `revealWithCallback`), the contract calls:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [4](#0-3) 

Each `keccak256` costs ~30 gas. With `numHashes = 300,000`, this loop alone consumes ~9M gas, approaching or exceeding the block gas limit on many chains.

**Root cause 4 — `maxNumHashes` protection is disabled by default:**

The only on-chain guard is `maxNumHashes`, which defaults to `0` (disabled). When `maxNumHashes == 0`, the check is skipped entirely:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [5](#0-4) 

Providers who do not explicitly call `setMaxNumHashes` are fully exposed.

The Fortuna keeper has an off-chain `update_commitments_loop` that calls `advanceProviderCommitment` when outstanding requests exceed a threshold, but this is a reactive off-chain mitigation, not an on-chain guarantee. [6](#0-5) 

---

### Impact Explanation

An attacker who floods a zero-fee (or low-fee) provider with requests causes `numHashes` to grow until `revealWithCallback` / `reveal` transactions run out of gas. The provider becomes permanently unable to fulfill any pending randomness request. All users who requested randomness from that provider are stuck with unresolvable callbacks. This is a complete DoS of the Entropy service for the targeted provider.

---

### Likelihood Explanation

The attack cost is `N × pythFeeInWei` (the protocol flat fee). In tests this is set to 7 wei; on live deployments it is a small fixed amount. A provider with `feeInWei = 0` (which is explicitly allowed) makes the attack cost only the Pyth protocol fee per request. On low-fee chains (e.g., BNB, Polygon), an attacker can create tens of thousands of requests for a few dollars. The `maxNumHashes` protection is opt-in and off by default, so any provider that does not explicitly configure it is vulnerable. The Fortuna keeper's `advanceProviderCommitment` mitigation is reactive and can be outpaced by a sustained flood.

---

### Recommendation

1. **Enforce a minimum `feeInWei` in `register()` and `setProviderFee()`** — require `feeInWei >= pythFeeInWei` or a governance-set floor, analogous to the minimum deposit threshold recommended in the original report.

2. **Enable `maxNumHashes` by default** — set a non-zero default in `_initialize` so that all providers are protected without requiring explicit configuration.

3. **Enforce `maxNumHashes` unconditionally** — remove the `providerInfo.maxNumHashes != 0` guard, or set a protocol-level hard cap on `numHashes` regardless of provider configuration.

---

### Proof of Concept

1. Attacker calls `register()` with `feeInWei = 0` to become a provider (or targets an existing provider with a low fee).
2. Attacker calls `request(provider, ...)` in a loop, paying only `pythFeeInWei` per call. After `N` calls, `providerInfo.sequenceNumber` has advanced by `N` and `currentCommitmentSequenceNumber` is still at its initial value.
3. The N-th request has `req.numHashes = N`.
4. Provider calls `revealWithCallback(provider, seqN, ...)`. Inside `revealHelper`, `constructProviderCommitment(N, revelation)` loops `N` times. At `N ≈ 300,000`, gas consumption exceeds 9M gas and the transaction reverts out-of-gas.
5. The provider cannot fulfill any request. All pending randomness callbacks are permanently stuck.
