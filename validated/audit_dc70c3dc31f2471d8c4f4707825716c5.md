### Title
Excess ETH Permanently Credited to Pyth Fee Pool Instead of Refunded to Caller — (`Entropy.sol`, `requestHelper`)

---

### Summary

In `Entropy.sol`, the `requestHelper` function credits `msg.value - providerFee` to `_state.accruedPythFeesInWei` rather than the correct Pyth fee portion (`requiredFee - providerFee`). When any unprivileged caller overpays (sends `msg.value > requiredFee`), the excess ETH is permanently absorbed into the Pyth fee pool with no refund path for the user.

---

### Finding Description

The vulnerability class from the reference report is **fee/refund accounting with wrong value passed to an accounting update** — analogous to `_paymentAH()` passing a stale/zeroed value to `decreaseEpochLienCount()` after deletion.

In `Entropy.sol`, `requestHelper` performs the following fee split:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The correct Pyth fee is `requiredFee - providerFee`. Instead, the code uses `msg.value - providerFee`. When `msg.value > requiredFee`, the delta `(msg.value - requiredFee)` is silently credited to `_state.accruedPythFeesInWei` — funds that belong to the caller.

There is no refund path for the caller. The only withdrawal functions are:
- `withdraw()` / `withdrawAsFeeManager()` — for registered providers only
- `withdrawFee()` — for the Pyth admin/owner only [2](#0-1) 

The caller has no mechanism to recover the excess ETH.

---

### Impact Explanation

Any user who calls `request()` or `requestWithCallback()` with `msg.value > getFeeV2(provider, callbackGasLimit)` permanently loses the excess ETH. The funds are credited to `accruedPythFeesInWei` and are only withdrawable by the Pyth admin. This is a direct, irreversible loss of user funds with no protocol-level recovery mechanism.

---

### Likelihood Explanation

Overpayment is a realistic scenario:
- Front-end tooling and gas estimation libraries commonly add a buffer to `msg.value` to avoid reverts.
- Callers using `callbackGasLimit` may round up their fee estimate.
- The contract enforces only a minimum (`msg.value < requiredFee` reverts), not an exact match.

The Pyth Lazer contract explicitly refunds excess fees (confirmed in test: `assertEq(alice.balance, 1 ether - fee - fee)` after overpaying with `0.5 ether`), demonstrating that the Pyth codebase treats refunds as expected behavior in analogous fee-collection flows. [3](#0-2) 

The absence of a refund in Entropy is therefore an inconsistency, not an intentional design choice.

---

### Recommendation

Replace `msg.value` with `requiredFee` in the Pyth fee accounting line, and refund any excess to the caller:

```solidity
uint128 pythFee = requiredFee - providerFee;
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += pythFee;

// Refund excess
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

---

### Proof of Concept

1. Provider registers with `feeInWei = 100 wei`. Pyth fee = 50 wei. `requiredFee = 150 wei`.
2. User calls `random.request{value: 200 wei}(provider, commitment, false)`.
3. `msg.value (200) >= requiredFee (150)` — no revert.
4. `providerInfo.accruedFeesInWei += 100` (correct).
5. `_state.accruedPythFeesInWei += (200 - 100) = 100` — **should be 50**.
6. User's 50 wei excess is permanently locked in the Pyth fee pool.
7. User has no function to call to recover it. Only the Pyth admin can withdraw via `withdrawFee`. [1](#0-0)
