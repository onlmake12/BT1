### Title
Hermes WebSocket `/ws` Endpoint Has No Connection-Level Rate Limiting, Enabling Resource Exhaustion DoS — (File: `apps/hermes/server/src/api/ws.rs`)

### Summary

The Hermes WebSocket endpoint accepts an unlimited number of concurrent connections per IP address. The only rate limiting present is a bytes-per-second throughput limit applied **after** a connection is established and data is being sent. An unprivileged attacker can open thousands of WebSocket connections from a single IP, exhausting server-side resources (Tokio tasks, broadcast channel receivers, memory, file descriptors) without any connection-level gating.

### Finding Description

In `apps/hermes/server/src/api/ws.rs`, the `ws_route_handler` function handles incoming WebSocket upgrade requests. It extracts the requester IP and API token, then immediately upgrades the connection with no rate limiting check at the connection establishment phase: [1](#0-0) 

The `websocket_handler` function then allocates a `Subscriber` struct, subscribes to the aggregation broadcast channel, and spawns a long-running async task — all before any rate limiting is applied: [2](#0-1) 

The `WsState` rate limiter is a `DefaultKeyedRateLimiter<IpAddr>` configured at `BYTES_LIMIT_PER_IP_PER_SECOND = 256 KiB`: [3](#0-2) [4](#0-3) 

This rate limiter is **only checked inside `handle_price_feeds_update`**, when the server is about to send data to the client: [5](#0-4) 

There is no check on the number of concurrent connections per IP, and no rate limit on the rate of new connection attempts per IP. The `subscriber_counter` is only an incrementing ID, not a cap: [6](#0-5) 

The SSE endpoint `price_stream_sse_handler` at `/v2/updates/price/stream` has the same absence of connection-level rate limiting: [7](#0-6) 

### Impact Explanation

Each accepted WebSocket connection before any rate limiting consumes:
- A Tokio async task (OS thread-equivalent resource)
- A `broadcast::Receiver` slot from the aggregation channel (`Aggregates::subscribe`)
- A `Subscriber` struct with a `HashMap` for price feed configs
- A file descriptor

An attacker opening tens of thousands of connections from a single IP (or a small set of IPs) can exhaust these resources, degrading or denying service to legitimate Hermes users. The public Hermes endpoint (`hermes.pyth.network`) is the primary price update delivery mechanism for Pyth integrators, so its unavailability directly impacts DeFi protocols relying on Pyth price feeds.

### Likelihood Explanation

The `/ws` endpoint is publicly accessible with no authentication required (authentication is optional, keyed by token suffix for metrics only). Any unprivileged actor can open WebSocket connections. The attack requires only a standard WebSocket client and a loop — no special knowledge or privileges. The documentation confirms the endpoint is intentionally public. [8](#0-7) 

### Recommendation

Add a per-IP concurrent connection limit and/or a per-IP connection-rate limit at the `ws_route_handler` level, before the WebSocket upgrade is accepted. This mirrors the fix recommended in the reference report: rate limiting should be applied at the point of the connection attempt, not only after the connection is established and data is flowing. A secondary `DefaultKeyedRateLimiter<IpAddr>` tracking connection counts or connection attempt rates per IP should be checked in `ws_route_handler` before calling `ws.on_upgrade`.

### Proof of Concept

```python
import asyncio
import websockets

async def flood():
    conns = []
    for _ in range(10000):
        try:
            ws = await websockets.connect("wss://hermes.pyth.network/ws")
            conns.append(ws)
        except Exception:
            pass
    await asyncio.sleep(300)  # hold connections open

asyncio.run(flood())
```

Each connection is accepted, a Tokio task and broadcast receiver are allocated, and the bytes rate limiter is never triggered because no data is being sent. Server resources are exhausted proportionally to the number of open connections, with no per-IP connection cap enforced anywhere in `apps/hermes/server/src/api/ws.rs`. [9](#0-8)

#[tracing::instrument(skip(stream, state, subscriber_ip, token_suffix))]
async fn websocket_handler<S>(
    stream: WebSocket,
    state: ApiState<S>,
    subscriber_ip: Option<IpAddr>,
    token_suffix: String,
) where
    S: Aggregates,
    S: Send,
{
    let ws_state = state.ws.clone();

    // Retain the recent rate limit data for the IP addresses to
    // prevent the rate limiter size from growing indefinitely.
    ws_state.rate_limiter.retain_recent();

    let id = ws_state.subscriber_counter.fetch_add(1, Ordering::SeqCst);

    tracing::debug!(id, ?subscriber_ip, "New Websocket Connection");
    ws_state
        .metrics
        .interactions
        .get_or_create(&Labels {
            interaction: Interaction::NewConnection,
            status: Status::Success,
            token_suffix: token_suffix.clone(),
        })
        .inc();

    let notify_receiver = Aggregates::subscribe(&*state.state);
    let (sender, receiver) = stream.split();

    state
        .metrics
        .stream_active_connections
        .get_or_create(&stream_protocol_label("ws"))
        .inc();

    let mut subscriber = Subscriber::new(
        id,
        subscriber_ip,
        token_suffix,
        state.state.clone(),
        state.ws.clone(),
        state.metrics.clone(),
        state.streaming.disconnect_slow_consumers,
        state.streaming.ws_send_timeout,
        notify_receiver,
        receiver,
        sender,
    );

    subscriber.run().await;
}
