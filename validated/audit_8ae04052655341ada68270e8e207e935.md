### Title
Integer Overflow in `GetLastStateProofProcess::execute` Limit Check Allows Unbounded Vec Allocation — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged light-client peer can send a crafted `GetLastStateProof` message with `last_n_blocks = u64::MAX/2 + 1` and an empty `difficulties` list. In release mode, the limit-check expression `difficulties.len() + (last_n_blocks as usize) * 2` wraps to `0`, bypassing `GET_LAST_STATE_PROOF_LIMIT = 1000`. The server then unconditionally collects every block number from `start_block_number` to `last_block_number` into a `Vec`, followed by expensive per-block DB lookups, proportional to the full chain length.

---

### Finding Description

**Overflow in the guard:** [1](#0-0) 

```
last_n_blocks = 0x8000_0000_0000_0000  (u64::MAX/2 + 1)
last_n_blocks as usize                 = 0x8000_0000_0000_0000  (64-bit: no truncation)
(last_n_blocks as usize) * 2           = 0x1_0000_0000_0000_0000 → wraps to 0 (release mode)
difficulties.len() + 0 = 0  >  1000   → false  →  guard bypassed
```

**Unbounded allocation after bypass:** [2](#0-1) 

Because `last_n_blocks = 0x8000_0000_0000_0000`, the condition `last_block_number - start_block_number <= last_n_blocks` is always true for any real chain. The server falls into the "not enough blocks" branch and executes:

```rust
let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
```

With `start_block_number = 0` and `last_hash` set to the current tip (publicly observable via `SendLastState`), this allocates a `Vec<u64>` with `chain_length` entries — unbounded by any attacker-independent cap.

**Expensive DB work follows:** [3](#0-2) 

`complete_headers` is then called with the full `block_numbers` slice, performing `get_ancestor` + `get_block` + MMR root lookups for every entry.

**The limit constant itself:** [4](#0-3)

### Impact Explanation

On a long-running CKB mainnet node (~12 M blocks as of mid-2025):

- `Vec<u64>` allocation: ~96 MB per request.
- `complete_headers` DB I/O: O(chain_length) random reads per request.
- Multiple concurrent crafted requests → memory exhaustion / OOM kill of the node process.

The impact is scoped to the light-client protocol server process; it does not affect consensus or funds directly, but it crashes or severely degrades a running full node that serves light clients.

---

### Likelihood Explanation

- Any peer that connects to the light-client protocol port can send this message — no authentication required.
- The current tip hash needed for `last_hash` is broadcast proactively by the server via `SendLastState`.
- The attack requires a single malformed packet; no hashpower, keys, or privileged access needed.
- Release builds (the only production deployment mode) use wrapping arithmetic, making the overflow deterministic.

---

### Recommendation

Replace the unchecked arithmetic with a saturating or checked operation:

```rust
// Option A: saturating_add / saturating_mul
if (last_n_blocks as usize)
    .saturating_mul(2)
    .saturating_add(self.message.difficulties().len())
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: checked_mul, reject on overflow
let n = (last_n_blocks as usize)
    .checked_mul(2)
    .and_then(|v| v.checked_add(self.message.difficulties().len()));
if n.map_or(true, |v| v > constant::GET_LAST_STATE_PROOF_LIMIT) {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Either form ensures that any `last_n_blocks` value large enough to overflow is rejected before any allocation occurs.

### Proof of Concept

```rust
#[test]
fn test_limit_check_overflow() {
    // Simulates the guard expression in release mode (wrapping arithmetic)
    let last_n_blocks: u64 = u64::MAX / 2 + 1;          // 0x8000_0000_0000_0000
    let difficulties_len: usize = 0;

    // This is what the current code computes:
    let check = difficulties_len.wrapping_add(
        (last_n_blocks as usize).wrapping_mul(2)
    );
    // check == 0, so the guard `0 > 1000` is false → bypassed
    assert_eq!(check, 0, "overflow wraps to 0, guard is bypassed");

    // The fix: saturating arithmetic fires correctly
    let safe_check = (last_n_blocks as usize)
        .saturating_mul(2)
        .saturating_add(difficulties_len);
    assert!(safe_check > 1000, "saturating check correctly rejects the request");
}
```
