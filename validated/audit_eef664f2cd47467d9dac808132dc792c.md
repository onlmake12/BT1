### Title
Subscription Manager Cannot Reduce Active Price Feed Count When Balance Falls Below New Minimum — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler::updateSubscription` enforces the minimum balance requirement against the **new** (post-update) feed count unconditionally whenever the subscription remains active. When a subscription manager tries to **reduce** the number of tracked price feeds — a cost-cutting action — the same minimum balance gate blocks them if their balance has been depleted below `newFeedCount × minimumBalancePerFeed`. The manager is trapped: they cannot reduce their ongoing spend rate, and their only escape is full deactivation.

---

### Finding Description

In `updateSubscription`, after crediting any `msg.value` sent with the call, the contract performs this check:

```solidity
// Check minimum balance if subscription remains active
if (willBeActive) {
    uint256 minimumBalance = this.getMinimumBalance(
        uint8(newParams.priceIds.length)   // ← always the NEW feed count
    );
    if (currentStatus.balanceInWei < minimumBalance) {
        revert SchedulerErrors.InsufficientBalance();
    }
}
``` [1](#0-0) 

`getMinimumBalance` is purely linear in feed count:

```solidity
return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
``` [2](#0-1) 

The check is applied identically whether the manager is **adding** feeds (increasing cost) or **removing** feeds (decreasing cost). There is no branch that relaxes or skips the check for a pure reduction in feed count.

The existing test suite explicitly documents and asserts this behavior under the label **"Scenario 4: Reducing number of feeds still checks minimum balance"**:

```
// Action 4: Update should fail due to insufficient balance
vm.expectRevert(
    abi.encodeWithSelector(SchedulerErrors.InsufficientBalance.selector)
);
scheduler.updateSubscription(subId_reduce, newParams_reduce);
``` [3](#0-2) 

The test confirms: a subscription with 2 feeds whose balance has been depleted below `1 × minimumBalancePerFeed` cannot be reduced to 1 feed while remaining active.

---

### Impact Explanation

A subscription manager whose balance has been naturally drained by keeper updates reaches a state where:

- Their balance is below `newFeedCount × minimumBalancePerFeed` for any reduced feed count.
- They **cannot** reduce the number of feeds to slow the depletion rate.
- Their only option is full deactivation (`isActive = false`), which stops all price updates entirely — a far more disruptive outcome than a partial reduction.
- Meanwhile, the subscription continues to be updated at the **higher** (unreduced) feed count, draining the remaining balance faster.

The manager is denied the ability to reduce their exposure to ongoing costs, directly analogous to a trader being denied the ability to reduce a losing position.

---

### Likelihood Explanation

Any subscription manager whose balance is depleted below the minimum for a smaller feed count triggers this. This is a natural lifecycle event: subscriptions are funded to a minimum at creation and balances erode with each keeper update. No special permissions, leaked keys, or external oracle manipulation are required — the manager simply calls `updateSubscription` with fewer `priceIds` while active.

---

### Recommendation

When the new feed count is strictly less than the current feed count (a pure reduction), skip or relax the minimum balance check. For example:

```solidity
if (willBeActive) {
    bool isReducingFeeds = newParams.priceIds.length < currentParams.priceIds.length;
    if (!isReducingFeeds) {
        uint256 minimumBalance = this.getMinimumBalance(
            uint8(newParams.priceIds.length)
        );
        if (currentStatus.balanceInWei < minimumBalance) {
            revert SchedulerErrors.InsufficientBalance();
        }
    }
}
```

This mirrors the fix applied in the referenced Zaros report: when a user is reducing their exposure, only the lower (maintenance-equivalent) threshold — or no threshold at all — should apply.

---

### Proof of Concept

The existing test `testUpdateSubscriptionEnforcesMinimumBalance` in `PulseScheduler.t.sol` already demonstrates the issue end-to-end:

1. Create a subscription with 2 feeds, funded exactly to the 2-feed minimum (`2 × minimumBalancePerFeed`).
2. Drain the balance below `1 × minimumBalancePerFeed` by triggering 50 keeper updates.
3. Attempt to reduce from 2 feeds to 1 feed while keeping the subscription active.
4. The call reverts with `InsufficientBalance`. [4](#0-3) 

The test then shows the only workaround is to inject additional funds — i.e., the manager must **add** ETH in order to **reduce** their subscription, which is counterintuitive and harmful to managers with depleted balances.
