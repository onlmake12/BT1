### Title
EIP-7702 Breaks `extcodesize`-Based EOA Detection in `revealWithCallback`, Permanently Blocking Legacy Entropy Fulfillment — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `revealWithCallback()` uses an inline `extcodesize` assembly check to distinguish EOA requesters from contract requesters. Under EIP-7702 (Ethereum Pectra), an EOA can temporarily attach code, causing `extcodesize` to return non-zero. The legacy fulfillment path then unconditionally calls `IEntropyConsumer._entropyCallback()` on the EOA with no try/catch. If the delegated code does not implement `_entropyCallback`, the call reverts, reverting the entire `revealWithCallback` transaction and permanently blocking fulfillment of that request.

---

### Finding Description

In `revealWithCallback()`, when `req.gasLimit10k == 0` (the legacy provider path), the contract takes the `else` branch:

```solidity
// Check if the requester is a contract account.
uint len;
assembly {
    len := extcodesize(callAddress)
}
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
``` [1](#0-0) 

The design intent is: if `extcodesize == 0`, the requester is an EOA and no callback is made; if `extcodesize > 0`, the requester is a contract and `_entropyCallback` is invoked.

Under EIP-7702, an EOA can set a code delegation (via a signed authorization tuple) so that `extcodesize` returns non-zero at runtime. If the EOA's delegated code does not implement `IEntropyConsumer._entropyCallback`, the call at line 676 reverts. Because there is no `try/catch`, the revert propagates and unwinds the entire `revealWithCallback` transaction — including the `clearRequest` at line 666 — leaving the request permanently stuck. [2](#0-1) 

The legacy path is activated when the provider's `defaultGasLimit` is `0` (i.e., the provider has not opted into the new callback-failure-state flow), causing `requestHelper` to set `req.gasLimit10k = 0`: [3](#0-2) 

---

### Impact Explanation

- The provider cannot fulfill the stuck request via `revealWithCallback` for as long as the EOA has EIP-7702 code attached that does not implement `_entropyCallback`.
- The request fee was already collected at request time; the user never receives their random number.
- Any downstream protocol logic waiting on the callback (e.g., a game, lottery, or DeFi protocol that the EOA controls) is permanently blocked.
- The provider's randomness chain is not advanced for that sequence number, potentially causing gas overhead on future requests (`numHashes` grows).

---

### Likelihood Explanation

- Post-Pectra (EIP-7702 live on mainnet), any EOA can attach arbitrary code via a signed authorization.
- An EOA requester may attach EIP-7702 code for entirely unrelated reasons (e.g., using a smart wallet delegation) and inadvertently break their pending Entropy request.
- A malicious EOA can deliberately attach non-`IEntropyConsumer` code after requesting randomness to grief the provider or block a dependent protocol.
- No privileged access is required; the entry path is fully permissionless (`requestWithCallback` → EIP-7702 delegation → `revealWithCallback` reverts).

---

### Recommendation

Replace the bare `extcodesize` + direct call pattern in the legacy path with a `try/catch` guard, mirroring the robustness of the new `excessivelySafeCall` path:

```solidity
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    ) {} catch {}
}
```

Alternatively, unconditionally attempt the callback (removing the `extcodesize` gate entirely) and wrap it in `try/catch`, so that both EOAs and contracts are handled safely regardless of EIP-7702 state.

---

### Proof of Concept

1. Deploy a legacy Entropy provider with `defaultGasLimit = 0`.
2. EOA Alice calls `requestWithCallback(provider, userContribution)`. `req.gasLimit10k` is set to `0` (legacy path).
3. Post-Pectra: Alice signs an EIP-7702 authorization delegating her address to a contract that does **not** implement `_entropyCallback` (e.g., a simple ERC-20 token contract).
4. Provider calls `revealWithCallback(provider, seqNum, userContribution, providerContribution)`.
5. Execution reaches the `else` branch. `extcodesize(alice) > 0` due to EIP-7702 delegation.
6. `IEntropyConsumer(alice)._entropyCallback(...)` is called. The delegated code has no such selector → reverts.
7. The entire transaction reverts. `clearRequest` is undone. The request remains active.
8. Steps 4–7 repeat indefinitely as long as Alice's EIP-7702 delegation is active. The provider cannot fulfill the request; Alice never receives her random number.
