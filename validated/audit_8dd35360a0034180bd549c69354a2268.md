Audit Report

## Title
Stale `descendants_fee` After Subtree Removal Enables Eviction Immunity and Pool-Filling Attack - (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` strips all link records for the entire removed subtree before calling `remove_entry` on each member. When `remove_entry` subsequently calls `update_ancestors_index_key`, the `calc_ancestors` lookup finds no link record for the already-removed entry and returns an empty set, so `sub_descendant_weight` is never called on surviving ancestors. Their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` are permanently inflated. An unprivileged attacker can exploit this via repeated RBF replacements to make a low-fee parent immune to eviction and fill the pool, causing legitimate transactions to be rejected with `Reject::Full`.

## Finding Description

**Root cause — pre-removal of links before ancestor accounting:**

`remove_entry_and_descendants` collects the root and all its descendants, then calls `remove_entry_links` for every ID in that set before any `remove_entry` call: [1](#0-0) 

`remove_entry_links` calls `self.links.remove(id)`, deleting the entry from `TxLinksMap::inner`: [2](#0-1) 

When `remove_entry` subsequently runs, it calls `update_ancestors_index_key`: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

`calc_ancestors` delegates to `calc_relative_ids`, which does `self.inner.get(short_id)`. Because `remove_entry_links` already called `self.links.remove(id)` for the root, this lookup returns `None`, `direct` is empty, and `calc_relation_ids` returns an empty set: [5](#0-4) 

Consequently, the `sub_descendant_weight` call inside `update_ancestors_index_key` is never reached for any surviving ancestor. Their `descendants_fee` (and related fields) remain permanently inflated.

**Exploit flow (RBF path):**

`resolve_conflict` — reachable by any caller of `send_transaction` — calls `remove_entry_and_descendants` for each conflicting transaction: [6](#0-5) 

Setup: parent P (low fee) with child C1 (high fee). After insertion, `P.descendants_fee = P.fee + C1.fee`.

1. Attacker submits C1′ (higher fee, same input as C1) → RBF triggers `resolve_conflict(C1_id)`.
2. `remove_entry_links(C1_id)` severs the P→C1 link and removes C1's link record.
3. `remove_entry(C1_id)` → `update_ancestors_index_key(C1, Remove)` → `calc_ancestors(C1_id)` returns `{}` → P's `descendants_fee` is **not** decremented.
4. C1′ is added as a new child of P → `P.descendants_fee += C1′.fee`.
5. After one replacement: `P.descendants_fee = P.fee + C1.fee (stale) + C1′.fee`.
6. After N replacements: `P.descendants_fee = P.fee + Σ(C1..CN fees) + CN′.fee` — grows without bound.

**No correction path exists for `descendants_fee`:**

`sub_descendant_weight` uses `saturating_sub`, so underflow silently zeroes the field rather than panicking: [7](#0-6) 

The pool-level `recompute_total_stat` corrects only `total_tx_size` and `total_tx_cycles`; there is no equivalent recomputation for per-entry `descendants_fee`: [8](#0-7) 

**Eviction immunity:**

The inflated `descendants_fee` feeds directly into `EvictKey` via `descendants_feerate.max(feerate)`: [9](#0-8) 

`EvictKey` is ordered ascending by `fee_rate`, so `next_evict_entry` (which takes the first/smallest) never selects P. The same stale accounting affects the size-limit eviction loop: [10](#0-9) 

## Impact Explanation

An attacker can permanently inflate the `EvictKey` of a low-fee parent transaction, making it immune to pool eviction. By repeating this across many parent transactions, the attacker fills the pool, causing all subsequent legitimate transaction submissions to be rejected with `Reject::Full`. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs.*

## Likelihood Explanation

`send_transaction` is an unprivileged RPC endpoint. RBF is active whenever `min_rbf_rate > min_fee_rate`, which is the standard production configuration. The attacker pays incrementally higher fees per replacement (the RBF requirement), but each replacement permanently inflates the parent's `descendants_fee` by the replaced child's fee. After N replacements the inflation is `Σ(child_fees[0..N-1])`, which grows without bound. No key material, privileged access, or majority hashpower is required.

## Recommendation

Before calling `remove_entry_links` for any member of the removed subtree, compute the surviving ancestors of the subtree root and apply `sub_descendant_weight` to each of them. Concretely, in `remove_entry_and_descendants`, before the link-removal loop, call `self.links.calc_ancestors(id)` for the root to obtain the set of surviving ancestors, then iterate over those IDs and call `sub_descendant_weight` with the root entry's weight (and similarly for each removed node that has surviving ancestors outside the removed set). This mirrors the logic already present in `update_ancestors_index_key` but executed before links are torn down.

Alternatively, restructure `remove_entry_and_descendants` so that `remove_entry` (including its ancestor accounting) is called before `remove_entry_links` for the subtree root, preserving the link graph at the time ancestor weights are updated.
