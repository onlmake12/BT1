Audit Report

## Title
Orphan Transaction Pool Flooding Allows Attacker to Evict Legitimate Orphan Transactions - (File: tx-pool/src/component/orphan.rs)

## Summary
The `OrphanPool` enforces a global hard cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` with no per-peer accounting. An unprivileged relay peer can saturate the pool with 100 fake orphan transactions referencing non-existent inputs, causing legitimate in-flight child transactions to be randomly evicted and silently dropped. The attack is low-cost, repeatable, and requires only a standard peer connection.

## Finding Description

`DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` is the sole global cap on the orphan pool, with no per-peer quota: [1](#0-0) 

`limit_size()` evicts via non-deterministic `HashMap` iteration — `self.entries.keys().next()` — with no peer-weighted fairness: [2](#0-1) 

`add_orphan_tx` performs no per-peer accounting before insertion; it inserts unconditionally and then calls `limit_size()`: [3](#0-2) 

The `is_missing_input` path in `process.rs` unconditionally calls `add_orphan` for any transaction whose inputs are unresolved, including attacker-crafted transactions spending non-existent `OutPoint`s: [4](#0-3) 

`add_orphan` propagates evicted tx hashes as `TxVerificationResult::Reject`, which causes `remove_from_known_txs` to be called — silently dropping the tx with no re-request: [5](#0-4) [6](#0-5) 

The upstream guard `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767` is orders of magnitude larger than the 100-entry orphan pool cap and provides no meaningful protection: [7](#0-6) 

**Exploit flow:**
1. Attacker connects as a relay peer and sends `RelayTransactionHashes` with 100 fake tx hashes.
2. Victim issues `GetRelayTransactions`; attacker responds with 100 structurally valid transactions each spending a non-existent `OutPoint`.
3. Each transaction fails `_process_tx` with `is_missing_input` → `add_orphan_tx` is called → pool reaches capacity (100 entries).
4. A legitimate user's child transaction (whose parent is in-flight) arrives and triggers `limit_size()`, which evicts a random entry via non-deterministic HashMap iteration.
5. The evicted child's hash is sent as `TxVerificationResult::Reject` → `remove_from_known_txs` → silently dropped.
6. When the legitimate parent arrives, `process_orphan_tx` finds no children and the child is permanently lost from this node.
7. Attacker re-sends fake orphans every `ORPHAN_TX_EXPIRE_TIME` (`100 * MAX_BLOCK_INTERVAL`) to maintain saturation.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** A single attacker can continuously disrupt orphan transaction propagation across any reachable node with only 100 small transactions per attack cycle. Applied simultaneously to multiple nodes, this degrades the network's ability to propagate transaction chains, causing silent propagation failures for legitimate users without any error surfaced to the submitting peer.

## Likelihood Explanation

The attack requires only a standard peer connection and the ability to send `RelayTransactionHashes` followed by `RelayTransactions` containing structurally valid but semantically invalid transactions (non-existent inputs). No proof-of-work, no privileged keys, no Sybil attack is required. The cost is 100 small transactions per attack cycle, and the cycle can be sustained indefinitely. The `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` cap is small enough that a single peer saturates it in one request-response round trip.

## Recommendation

1. **Add per-peer orphan pool accounting**: Track how many orphan entries each peer has contributed. Reject new entries from peers that have already filled their quota (e.g., `DEFAULT_MAX_ORPHAN_TRANSACTIONS / expected_peers`).
2. **Replace random eviction with peer-weighted eviction**: When the pool is full, prefer to evict entries from the peer that has contributed the most orphans rather than a random entry.
3. **Increase or make configurable the orphan pool cap**: The current cap of 100 is very small for a network with many concurrent in-flight transaction chains.

## Proof of Concept

```
1. Attacker connects to victim CKB node as a relay peer.

2. Attacker sends RelayTransactionHashes with 100 fake tx hashes:
   [hash_1, hash_2, ..., hash_100]

3. Victim node adds them to unknown_tx_hashes and sends GetRelayTransactions
   back to the attacker.

4. Attacker responds with RelayTransactions containing 100 transactions,
   each spending a non-existent OutPoint (random_bytes(32), index=0).

5. Each transaction fails _process_tx with is_missing_input → added to
   OrphanPool via add_orphan_tx. Pool is now at capacity (100 entries).

6. Legitimate user's child_tx (parent is in-flight) arrives via relay.
   add_orphan_tx is called → limit_size() evicts a random entry via
   self.entries.keys().next(). child_tx may be the evicted entry.

7. Evicted child_tx hash is sent as TxVerificationResult::Reject →
   remove_from_known_txs. Node silently drops child_tx with no re-request.

8. Legitimate parent_tx arrives → process_orphan_tx finds no children
   for parent_tx's outputs. child_tx is permanently lost from this node.

9. Attacker repeats steps 2-4 every ORPHAN_TX_EXPIRE_TIME seconds
   (100 * MAX_BLOCK_INTERVAL) to maintain pool saturation.

Unit test plan: construct an OrphanPool, insert 100 entries from a
single mock peer with non-existent OutPoints, then insert a 101st
entry from a different peer and assert it may be evicted (or that
the pool contains no entry from the second peer after limit_size()).
```

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
