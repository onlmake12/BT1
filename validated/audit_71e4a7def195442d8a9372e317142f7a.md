### Title
`ConsensusBuilder::build()` Relies on `debug_assert!` for Critical Consensus Parameter Validation, Silently Bypassed in Release Builds — (`spec/src/consensus.rs`)

---

### Summary

`ConsensusBuilder::build()` in `spec/src/consensus.rs` uses `debug_assert!` — not `assert!` — to enforce that critical consensus parameters such as `epoch_duration_target`, `initial_primary_epoch_reward`, and genesis difficulty are non-zero. In Rust, `debug_assert!` is compiled out entirely in release builds (`--release`), meaning these invariants are **never checked in production**. The individual setter methods (`initial_primary_epoch_reward()`, `epoch_duration_target()`, `orphan_rate_target()`, `max_block_cycles()`, `max_block_bytes()`, `tx_proposal_window()`) accept any value with no bounds or zero-value checks. A node operator (supported local CLI user) who configures a custom chain spec with a zero `epoch_duration_target` or zero-denominator `orphan_rate_target` will start the node without error in release mode; the node then panics (division by zero) when the difficulty adjustment algorithm runs during block processing from the network.

---

### Finding Description

In `spec/src/consensus.rs`, `ConsensusBuilder::build()` contains five `debug_assert!` guards:

```rust
// spec/src/consensus.rs lines 319–353
debug_assert!(
    self.inner.genesis_block.difficulty() > U256::zero(),
    "genesis difficulty should greater than zero"
);
debug_assert!(
    self.inner.initial_primary_epoch_reward != Capacity::zero(),
    "initial_primary_epoch_reward must be non-zero"
);
debug_assert!(
    self.inner.epoch_duration_target() != 0,
    "epoch_duration_target must be non-zero"
);
``` [1](#0-0) 

`debug_assert!` expands to nothing when `debug_assertions` is disabled (the default for `cargo build --release`). The production binary therefore performs **zero validation** of these parameters.

The setter methods themselves are unconditional assignments:

```rust
pub fn initial_primary_epoch_reward(mut self, initial_primary_epoch_reward: Capacity) -> Self {
    self.inner.initial_primary_epoch_reward = initial_primary_epoch_reward;
    self
}
pub fn epoch_duration_target(mut self, target: u64) -> Self {
    self.inner.epoch_duration_target = target;
    self
}
pub fn orphan_rate_target(mut self, orphan_rate_target: (u32, u32)) -> Self {
    self.inner.orphan_rate_target = RationalU256::new_raw(
        U256::from(orphan_rate_target.0),
        U256::from(orphan_rate_target.1),  // denominator — zero accepted silently
    );
    self
}
``` [2](#0-1) 

These setters are called directly from `ChainSpec::build_consensus()` using values deserialized from the TOML chain spec file, with no intermediate validation layer: [3](#0-2) 

`ProposalWindow` also has no invariant enforcement that `closest < farthest`. Its `length()` method computes `self.1 - self.0 + 1`, which underflows (wraps or panics) if `closest >= farthest`: [4](#0-3) 

---

### Impact Explanation

- **`epoch_duration_target = 0`**: Used as a divisor in the difficulty adjustment algorithm (`next_epoch_ext`). A zero value causes an integer division-by-zero panic, crashing the node the first time it processes a new epoch boundary block received from the network.
- **`orphan_rate_target = (n, 0)`**: Creates a `RationalU256` with a zero denominator. Any arithmetic using this rational (difficulty adjustment) panics.
- **`initial_primary_epoch_reward = 0`**: Causes reward calculation to produce zero, breaking miner incentives and potentially causing arithmetic errors in DAO accounting.
- **`ProposalWindow` with `closest >= farthest`**: `length()` underflows; `TwoPhaseCommitVerifier` and `ProposalTable::finalize()` produce incorrect results, causing valid transactions to be permanently rejected or invalid ones accepted. [5](#0-4) [6](#0-5) 

The node crash is triggered by **network-delivered blocks** (peers), not by the misconfiguration alone. The operator starts the node successfully; the panic occurs during block processing.

---

### Likelihood Explanation

CKB explicitly supports custom chain specs for dev chains, private networks, and testnets. The `ckb init` CLI command and the `[chain]` config section allow operators to supply arbitrary spec files. A developer building a private chain or integration test environment who sets `epoch_duration_target = 0` (e.g., to disable difficulty adjustment) or `orphan_rate_target = [1, 0]` will receive no error at startup in a release build. The node will crash only when the first epoch boundary is reached — potentially after hours of operation — making the root cause difficult to diagnose. The `debug_assert!` pattern gives a false sense of safety to developers who test in debug mode and then deploy in release mode.

---

### Recommendation

1. Replace all `debug_assert!` calls in `ConsensusBuilder::build()` with `assert!` (or return `Result<Consensus, Error>` and propagate errors).
2. Add explicit zero/range checks in each setter method:
   - `epoch_duration_target`: reject `0`.
   - `initial_primary_epoch_reward`: reject `Capacity::zero()`.
   - `orphan_rate_target`: reject denominator `0`.
   - `tx_proposal_window`: enforce `closest >= 1` and `farthest > closest`.
   - `max_block_cycles`, `max_block_bytes`: reject `0`.
3. Add a `ChainSpec::validate()` step in `build_consensus()` that runs unconditionally before constructing the `Consensus` object.

---

### Proof of Concept

Configure a dev chain spec with:
```toml
[params]
epoch_duration_target = 0
```

Build and run in release mode:
```sh
cargo build --release
./target/release/ckb init --chain dev
# edit specs/dev.toml: epoch_duration_target = 0
./target/release/ckb run
```

The node starts without error. When a peer delivers a block that crosses an epoch boundary, `next_epoch_ext` divides by `epoch_duration_target` (0), producing a panic and crashing the node. In debug mode (`cargo run`), the `debug_assert!` at line 342–345 would have caught this at startup. [7](#0-6) [8](#0-7)
