Audit Report

## Title
Unbounded `pending_compact_blocks` HashMap Enables Memory Exhaustion via Incomplete Compact Block Relay - (File: `sync/src/types/mod.rs`, `sync/src/relayer/compact_block_process.rs`)

## Summary
`PendingCompactBlockMap` is a plain, unbounded `HashMap` with no size cap, per-peer limit, or timeout-based eviction. Entries are inserted whenever compact block reconstruction fails due to missing transactions and are only removed upon successful block reconstruction. An attacker with mining capability can continuously insert entries that are never cleaned up, causing unbounded memory growth and eventual node crash.

## Finding Description
`PendingCompactBlockMap` is defined as a plain `HashMap` with no capacity bound: [1](#0-0) 

It is initialized without any bound in `SyncState`: [2](#0-1) 

When compact block reconstruction returns `ReconstructionResult::Missing` or `ReconstructionResult::Collided`, `missing_or_collided_post_process` unconditionally inserts an entry via `or_insert_with`: [3](#0-2) 

Cleanup only occurs in the `ReconstructionResult::Block` branch — i.e., only when a block is successfully reconstructed. The `retain` call that removes entries from previous epochs fires exclusively here: [4](#0-3) 

If an attacker never responds to `GetBlockTransactions`, this branch is never reached. The same pattern holds in `block_transactions_process.rs` — `pending.remove()` is called only on `ReconstructionResult::Block`: [5](#0-4) 

The `u64` timestamp stored in each entry is used only by `compare_with_pending_compact` to gate sync task creation (2-second window); it is never used for eviction: [6](#0-5) 

The `CompactBlockIsAlreadyPending` guard only prevents the same `(peer, block_hash)` pair from being re-inserted; different block hashes from the same peer are all stored: [7](#0-6) 

By contrast, the orphan block pool has a 60-second cleanup timer (`clean_expired_orphans`), but no equivalent exists for `pending_compact_blocks`: [8](#0-7) 

## Impact Explanation
Each entry stores a full `packed::CompactBlock` (header + short IDs + prefilled transactions + proposals + uncles). With no eviction mechanism, entries accumulate indefinitely within an epoch. Sufficient accumulation exhausts the node's memory, leading to OOM conditions, process crash, or severe performance degradation. This matches **High: "Vulnerabilities which could easily crash a CKB node."**

## Likelihood Explanation
The attacker must produce compact blocks with valid proof-of-work headers that pass `non_contextual_check` (valid height, uncle/proposal counts) and `contextual_check` (valid PoW, known parent, not already stored). This requires real mining capability. However, a mining pool with even a small fraction of network hashrate can continuously mine valid blocks, relay them as compact blocks with all short IDs listed but zero prefilled transactions, and then ignore all `GetBlockTransactions` responses. Each mined block creates one persistent entry. The rate of accumulation scales with the attacker's hashrate. The `retain` call only cleans entries from previous epochs, so within a single epoch (~1800 blocks, ~4 hours) entries accumulate without bound.

## Recommendation
1. **Bound the map**: Cap `pending_compact_blocks` at a maximum number of entries (e.g., 512), evicting the oldest entry (by stored timestamp) when the cap is reached.
2. **Timeout-based eviction**: Add a periodic background task analogous to `clean_expired_orphans` that removes entries whose timestamp is older than a threshold (e.g., 30 seconds).
3. **Per-peer limit**: Track how many pending entries each peer has contributed and reject new entries from peers that exceed the limit.
4. **Disconnect non-responding peers**: If a peer never responds to `GetBlockTransactions` for a pending entry, disconnect and remove their entries.

## Proof of Concept
```
1. Attacker mines valid CKB blocks (requires hashpower).
2. For each mined block, attacker sends a CompactBlock message to the target node
   with all short_ids listed but zero prefilled transactions.
3. The node calls reconstruct_block(), gets ReconstructionResult::Missing,
   and calls missing_or_collided_post_process() which inserts into pending_compact_blocks.
4. Attacker ignores all GetBlockTransactions responses — never sends the missing txs.
5. The entry remains in pending_compact_blocks indefinitely (no timeout eviction).
6. Repeat for each newly mined block within the epoch. Memory grows without bound.
```

`non_contextual_check` passes because the block header is valid and height is within epoch range. `contextual_check` passes because the PoW is valid and the parent is known. The entry is inserted and never cleaned up until a successful reconstruction triggers the `retain` call, which never fires if the attacker withholds transactions.
