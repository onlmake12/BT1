### Title
`advance` and `advance_delegation_record` Bypass Staking Protocol Freeze — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The Pyth staking program (`staking`) exposes a `GlobalConfig.freeze` boolean flag (error code 6017: `Frozen`) that is intended to halt all protocol operations during an emergency. However, the integrity pool program's two permissionless reward-distribution instructions — `advance` (epoch advancement + reward minting) and `advance_delegation_record` (per-staker reward transfer) — do not include the staking program's `config_account` in their account lists and therefore cannot check the `freeze` flag. Any unprivileged caller can continue to advance epochs and claim rewards even while the staking protocol is frozen.

---

### Finding Description

The staking program's `GlobalConfig` struct contains a `freeze` field:

```json
{ "name": "freeze", "type": "bool" }
```

When `freeze = true`, the staking program reverts with error 6017 (`Frozen`) on guarded instructions. Instructions that make a CPI into the staking program — such as `delegate` in the integrity pool — inherit this protection because they pass `config_account` (the staking program's `GlobalConfig` PDA) and the staking program enforces the check on its side.

However, the two reward-distribution instructions in the integrity pool program are **purely internal** and never CPI into the staking program:

**`advance` accounts** (IDL lines 147–207):
- `signer`, `pool_data`, `publisher_caps`, `pool_config`, `pool_reward_custody`
- No `config_account` from the staking program.

**`advance_delegation_record` accounts** (IDL lines 209–346):
- `payer`, `stake_account_positions`, `pool_data`, `pool_config`, `pool_reward_custody`, `stake_account_custody`, `publisher`, `publisher_stake_account_positions`, `publisher_stake_account_custody`, `delegation_record`, `token_program`, `system_program`
- No `config_account` from the staking program.

By contrast, the `delegate` instruction (IDL lines ~427–600) **does** include `config_account` (staking program's GlobalConfig PDA) because it CPIs into the staking program to create a position, and the staking program enforces the freeze check there.

This creates an asymmetry: staking/unstaking operations are blocked when frozen, but epoch advancement and reward distribution are not.

---

### Impact Explanation

When governance freezes the staking protocol (e.g., in response to a discovered accounting bug or exploit), the intent is to halt all state-changing operations. Despite the freeze:

1. Any caller can invoke `advance` to advance the epoch and compute/mint new rewards into `pool_reward_custody`.
2. Any caller can invoke `advance_delegation_record` to transfer tokens from `pool_reward_custody` to individual stakers' `stake_account_custody`.

This means:
- Rewards continue to accrue and be claimable during the freeze window, potentially based on corrupted or disputed position state.
- Stakers who are aware of the freeze can front-run the freeze by claiming rewards before governance can react, while other protocol operations remain blocked.
- The freeze period is not accounted for in reward calculations, so the reward rate `y` applies uniformly across frozen and non-frozen time, inflating effective yield during the incident.

---

### Likelihood Explanation

- Both `advance` and `advance_delegation_record` are **permissionless** — any transaction sender can call them with no privileged key.
- The entry path is direct: construct a transaction with the correct accounts (all derivable from public PDAs) and submit it.
- A freeze event is a realistic scenario (governance has the mechanism and the error code exists in production IDL).
- The reward rate `y` is currently 0 per OP-PIP-103, which reduces immediate financial impact, but the code path is live and `y` can be updated by `reward_program_authority` at any time via `update_y`.

---

### Recommendation

Add the staking program's `config_account` (GlobalConfig PDA) as a read-only account to both `advance` and `advance_delegation_record` in the integrity pool program, and add an explicit check:

```rust
require!(!config_account.freeze, IntegrityPoolError::ProtocolFrozen);
```

This mirrors the protection already present in `delegate` via the staking program CPI.

---

### Proof of Concept

1. Governance calls `update_config` on the staking program to set `freeze = true`.
2. Attempt to call `delegate` → reverts with error 6017 (`Frozen`). ✓ Protected.
3. Attempt to call `advance` with accounts `[signer, pool_data, publisher_caps, pool_config, pool_reward_custody]` → **succeeds**, epoch advances, rewards minted. ✗ Not protected.
4. Attempt to call `advance_delegation_record` with the correct PDA accounts → **succeeds**, tokens transferred to staker custody. ✗ Not protected.
