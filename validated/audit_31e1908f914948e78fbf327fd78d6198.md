### Title
`DownloadScheduler::decrease` Uses Wrong Variable for `timeout_count` Accumulation, Causing Overly Aggressive Peer Throttling and Disconnection — (`File: sync/src/types/mod.rs`)

### Summary

In `sync/src/types/mod.rs`, the `DownloadScheduler::decrease` function contains a wrong-variable bug: it assigns `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count = self.timeout_count.saturating_add(num)`. Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32), the condition `self.timeout_count > 2` is always true on every call, so `task_count` is decremented on every single slow block response instead of every third. This causes the node to throttle and disconnect slow-responding peers far more aggressively than the protocol intends, degrading sync performance and reducing peer connectivity.

### Finding Description

`DownloadScheduler` is the per-peer scheduler that controls how many concurrent block download requests (`task_count`) a peer is allowed. When a block response arrives slowly, `remove_by_block` calls `set.decrease(1)` or `set.decrease(2)` to gradually reduce `task_count`. The design intent is to accumulate `timeout_count` across multiple slow responses and only decrement `task_count` after three slow events (i.e., `timeout_count > 2`).

The actual code is:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

Because `task_count` initializes to `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`, the expression `self.task_count.saturating_add(num)` always evaluates to ≥ 33, which is always `> 2`. Therefore:

- `task_count` is decremented on **every** call to `decrease`, not every third.
- `timeout_count` is immediately reset to 0 after each call, so it never accumulates.
- The `timeout_count` field is rendered completely inert.

The correct line should be:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

The call path is:

1. A remote peer sends a block response.
2. `InflightBlocks::remove_by_block` is called in `sync/src/types/mod.rs`.
3. `time_analyzer.push_time(elapsed)` classifies the response as `NormalToUpper` or `UpperToMax`.
4. `set.decrease(1)` or `set.decrease(2)` is called.
5. Due to the bug, `task_count` is decremented immediately every time.
6. In `prune()`, any peer whose `task_count == 0` is added to `disconnect_list` and disconnected.

### Impact Explanation

- **Overly aggressive throttling**: A peer that responds slowly even once has its concurrent block request quota decremented immediately. The intended "tolerate up to 2 slow responses before penalizing" grace period is completely bypassed.
- **Premature peer disconnection**: Since `task_count` reaches 0 three times faster than intended, legitimate peers with slightly elevated latency are disconnected prematurely. This reduces the node's peer count and degrades sync throughput.
- **Sync stall risk**: In environments with high-latency peers (e.g., cross-region connections), the node may disconnect from all available peers, stalling block synchronization entirely.
- **Eclipse attack surface widening**: Aggressive disconnection of legitimate peers leaves the node with fewer connections, making it easier for an attacker to fill the remaining slots with malicious peers.

### Likelihood Explanation

Any unprivileged peer that responds to block requests with latency classified as `NormalToUpper` or `UpperToMax` by the `TimeAnalyzer` triggers this path. This is a normal network condition on any heterogeneous network. No special privileges, keys, or coordination are required. The bug fires on every slow block response, making it continuously active during normal operation.

### Recommendation

Fix the wrong variable reference in `DownloadScheduler::decrease`:

```rust
fn decrease(&mut self, num: usize) {
    // Was: self.timeout_count = self.task_count.saturating_add(num);
    self.timeout_count = self.timeout_count.saturating_add(num);
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

### Proof of Concept

Given `task_count = 32` (the default `INIT_BLOCKS_IN_TRANSIT_PER_PEER`) and `timeout_count = 0`:

**Buggy behavior** — calling `decrease(1)` once:
- `self.timeout_count = 32 + 1 = 33`
- `33 > 2` → true → `task_count` becomes 31, `timeout_count` reset to 0
- Result: `task_count` decremented on the **first** slow response.

**Intended behavior** — calling `decrease(1)` three times:
- Call 1: `timeout_count = 0 + 1 = 1`, `1 > 2` → false, no decrement.
- Call 2: `timeout_count = 1 + 1 = 2`, `2 > 2` → false, no decrement.
- Call 3: `timeout_count = 2 + 1 = 3`, `3 > 2` → true → `task_count` decremented once.
- Result: `task_count` decremented only after the **third** slow response.

A peer that consistently responds in the `NormalToUpper` quantile will have its `task_count` reach 0 in 32 slow responses under the bug, versus 96 slow responses under the correct logic — a 3× acceleration of disconnection. Any unprivileged sync peer can trigger this by responding to `GetBlocks` messages with latency above the `normal_time` threshold. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
