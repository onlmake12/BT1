### Title
Unverified `IEntropyConsumer` Interface on Requester Causes Permanent Request DoS and Fee Loss in Legacy Callback Path — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.revealWithCallback` contains a legacy code path (when `req.gasLimit10k == 0`) that calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly on the requester contract with **no error catching and no prior interface verification**. Any contract that calls `requestWithCallback` or `requestV2` without implementing `IEntropyConsumer._entropyCallback` will cause every subsequent `revealWithCallback` attempt to revert, permanently locking the request and the user's paid fees.

---

### Finding Description

`Entropy.requestWithCallback` and `requestV2` accept `msg.sender` as the requester and store it in `req.requester` without verifying that the caller implements `IEntropyConsumer._entropyCallback`. [1](#0-0) [2](#0-1) 

When `revealWithCallback` is later called, it branches on `req.gasLimit10k`:

- **Safe path** (`gasLimit10k != 0`): uses `excessivelySafeCall` which catches reverts and records `CALLBACK_FAILED`.
- **Unsafe legacy path** (`gasLimit10k == 0`): directly calls `_entropyCallback` with no error catching. [3](#0-2) [4](#0-3) 

The legacy path (`gasLimit10k == 0`) is entered when the provider has `defaultGasLimit == 0` (legacy providers) or when `requestWithCallback` is used (which always passes `gasLimit = 0`). [5](#0-4) 

If the requester is a contract that does **not** implement `_entropyCallback` (e.g., a contract that calls `requestWithCallback` but does not inherit `IEntropyConsumer`), the call at line 676 reverts. Because the entire transaction reverts, `clearRequest` at line 666 is also rolled back. The request remains permanently active but unfulfillable — every future `revealWithCallback` attempt will also revert. [6](#0-5) 

The `IEntropyConsumer` abstract contract itself does not enforce EIP-165 or any on-chain interface check: [7](#0-6) 

---

### Impact Explanation

1. **Permanent loss of user fees**: Fees are credited to the provider and Pyth at request time inside `requestHelper` (lines 237–239). Even though `revealWithCallback` reverts, the fees credited during `requestWithCallback` are not refunded. The user paid for a service that can never be delivered.
2. **Permanent DoS on the request slot**: The request remains active indefinitely. The provider's keeper wastes gas on every failed fulfillment attempt.
3. **Provider sequence number consumed**: The provider's `sequenceNumber` was incremented at request time and is not recoverable. [8](#0-7) 

---

### Likelihood Explanation

This is a realistic mistake. The `requestWithCallback` NatSpec says "The address calling this function **should be** a contract that inherits from the `IEntropyConsumer` interface" — this is advisory only, not enforced. A developer integrating Entropy who calls `requestWithCallback` from a contract that does not properly inherit `IEntropyConsumer` (e.g., a proxy, a multisig, or a contract that implements a differently-named callback) will silently create an unfulfillable request. The protocol provides no on-chain guard. [9](#0-8) 

---

### Recommendation

At request time in `requestWithCallback` / `requestV2`, verify that `msg.sender` exposes the `_entropyCallback` selector before storing it as the requester. This can be done via EIP-165 `supportsInterface` or a low-level static call to check the selector exists:

```solidity
// At request time, verify the caller implements IEntropyConsumer
if (isRequestWithCallback && msg.sender.code.length > 0) {
    (bool ok, ) = msg.sender.staticcall(
        abi.encodeWithSelector(IEntropyConsumer._entropyCallback.selector, uint64(0), address(0), bytes32(0))
    );
    // A revert here means the interface is missing
    require(ok || /* selector exists check */, "Requester must implement IEntropyConsumer");
}
```

Alternatively, add error catching in the legacy path (lines 675–681) analogous to the `excessivelySafeCall` used in the gas-limit path, and emit a `CallbackFailed` event so the request can be cleared rather than permanently stuck.

---

### Proof of Concept

1. Deploy a contract `BadRequester` that calls `requestWithCallback` but does **not** inherit `IEntropyConsumer` and has no `_entropyCallback` function.
2. `BadRequester` calls `entropy.requestWithCallback{value: fee}(provider, userRandomNumber)` — succeeds, fees paid, request stored with `req.requester = address(BadRequester)`, `req.gasLimit10k = 0`.
3. Provider's keeper calls `entropy.revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
4. Execution reaches line 675: `extcodesize(BadRequester) != 0`, so it enters the `if` block.
5. Line 676: `IEntropyConsumer(BadRequester)._entropyCallback(...)` — `BadRequester` has no such function, the call reverts.
6. The entire `revealWithCallback` transaction reverts. The request is not cleared.
7. Every subsequent `revealWithCallback` attempt reverts identically. The request is permanently stuck. User fees are permanently lost. [10](#0-9)

// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

abstract contract IEntropyConsumer {
    // This method is called by Entropy to provide the random number to the consumer.
    // It asserts that the msg.sender is the Entropy contract. It is not meant to be
    // override by the consumer.
    function _entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) external {
        address entropy = getEntropy();
        require(entropy != address(0), "Entropy address not set");
        require(msg.sender == entropy, "Only Entropy can call this function");

        entropyCallback(sequence, provider, randomNumber);
    }

    // getEntropy returns Entropy contract address. The method is being used to check that the
    // callback is indeed from Entropy contract. The consumer is expected to implement this method.
    // Entropy address can be found here - https://docs.pyth.network/entropy/contract-addresses
    function getEntropy() internal view virtual returns (address);

    // This method is expected to be implemented by the consumer to handle the random number.
    // It will be called by _entropyCallback after _entropyCallback ensures that the call is
    // indeed from Entropy contract.
    function entropyCallback(
        uint64 sequence,
        address provider,
        bytes32 randomNumber
    ) internal virtual;
}
