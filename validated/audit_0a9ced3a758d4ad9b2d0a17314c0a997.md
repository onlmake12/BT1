### Title
`TxPool::remove_expired` Removes Parent Transactions Without Removing Dependent Children, Leaving Pool in Inconsistent State - (File: tx-pool/src/pool.rs)

### Summary
The `remove_expired` function in CKB's transaction pool removes expired transactions by calling `pool_map.remove_entry` (single-entry removal) instead of `pool_map.remove_entry_and_descendants`. When a parent transaction expires and is removed, its child transactions remain in the pool in an inconsistent state: their inputs reference outputs that no longer exist in the pool or on-chain, yet those inputs remain tracked in `edges.inputs`. These zombie children can never be committed, consume pool capacity, and block re-submission of the parent transaction.

### Finding Description

**Root cause — `remove_expired` uses single-entry removal:** [1](#0-0) 

```rust
pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
    ...
    for entry in removed {
        self.pool_map.remove_entry(&entry.proposal_short_id());   // ← single-entry only
        ...
    }
}
```

The correct variant that also removes all descendants exists and is used elsewhere: [2](#0-1) 

`remove_entry_and_descendants` first collects all descendant IDs, strips all links, then calls `remove_entry` for each. The RPC-triggered `remove_tx` path correctly uses this variant: [3](#0-2) 

**What `remove_entry` does NOT do when called alone:** [4](#0-3) 

`remove_entry_edges` removes the *parent's* inputs from `edges.inputs` and cleans up its dep records. `remove_entry_links` removes the parent from children's parent-sets. However, the children's own input records — which point to the now-deleted parent's outputs — remain registered in `edges.inputs`: [5](#0-4) 

**Trigger path — `remove_expired` is called on every new block:** [6](#0-5) 

`_update_tx_pool_for_reorg` → `tx_pool.remove_expired(callbacks)` is invoked for every block attachment. No special privilege is required to trigger it; any block arrival suffices.

**Exploit scenario:**

1. An unprivileged tx-pool submitter sends `parent_tx` (spending an on-chain UTXO, producing `out_B`).
2. The same submitter sends `child_tx` (consuming `out_B`). `child_tx` has a later timestamp than `parent_tx`.
3. After `expiry` milliseconds (default 12 hours), `parent_tx` expires. On the next block, `remove_expired` removes `parent_tx` via `remove_entry`.
4. `child_tx` has not yet expired (its timestamp is later). It remains in the pool.
5. `child_tx`'s input (`out_B`) is still registered in `edges.inputs`. `out_B` does not exist on-chain or in the pool.
6. Any attempt to re-submit `parent_tx` will succeed at the input-conflict check (its inputs were cleaned up), but the pool now contains both `parent_tx` and `child_tx` with `child_tx`'s input still pointing to `out_B` — creating a duplicate edge entry and corrupting `total_tx_size`/`total_tx_cycles` accounting when `child_tx` is eventually removed.
7. Until `child_tx` expires, it occupies pool capacity and its stale `edges.inputs` entry blocks any new transaction that tries to consume `out_B`.

**Accounting drift:** `total_tx_size` and `total_tx_cycles` are decremented only for the parent when it is removed. The child's size and cycles remain counted. If the child is later removed via a path that calls `update_stat_for_remove_tx` with the child's own values, the subtraction may underflow, triggering the error-recovery recompute path: [7](#0-6) 

### Impact Explanation

**Impact: Medium.**

- Child transactions in the pool with no valid parent can never be committed to a block. They occupy pool space (counted against `max_tx_pool_size`) until their own expiry fires.
- The stale `edges.inputs` entry for the child's input (`out_B`) causes any new transaction spending `out_B` to be rejected as a conflict, even though `out_B` does not exist. This blocks re-submission of the parent for the duration of the child's remaining lifetime.
- `total_tx_size` / `total_tx_cycles` pool statistics become transiently incorrect, potentially causing premature rejection of legitimate transactions that would otherwise fit within pool limits.
- The inconsistency is bounded in time (child expires eventually) and in space (pool size limits apply), so it does not cause permanent loss of funds or consensus failure. Severity is medium.

### Likelihood Explanation

**Likelihood: Medium.**

- The condition (parent expires before child) is a normal operational scenario: a child transaction is always added after its parent, so its timestamp is always later, meaning it expires later.
- `remove_expired` fires on every block, so the window opens automatically without any attacker action beyond submitting a parent-child transaction pair and waiting.
- No privileged access, key material, or majority hashpower is required.

### Recommendation

Replace the `remove_entry` call inside `remove_expired` with `remove_entry_and_descendants`, and invoke `callbacks.call_reject` for each removed descendant:

```rust
pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
    let now_ms = ckb_systemtime::unix_time_as_millis();
    let expired_ids: Vec<_> = self
        .pool_map
        .iter()
        .filter(|entry| self.expiry + entry.inner.timestamp < now_ms)
        .map(|entry| entry.inner.proposal_short_id())
        .collect();

    for id in expired_ids {
        // Capture the timestamp before removal for the reject reason
        let timestamp = self.pool_map.get_by_id(&id)
            .map(|e| e.inner.timestamp)
            .unwrap_or(0);
        let entries = self.pool_map.remove_entry_and_descendants(&id);
        for entry in entries {
            let reject = Reject::Expiry(timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
}
```

This mirrors the pattern used by `remove_tx` and `remove_by_detached_proposal`, ensuring no child transaction is left in an inconsistent state after its parent is evicted.

### Proof of Concept

```
1. Node starts with default expiry (e.g., 12 hours).
2. Submit parent_tx spending on-chain UTXO_A, producing out_B.
3. Submit child_tx spending out_B (child_tx.timestamp > parent_tx.timestamp).
4. Both are in the pending pool.
5. Advance system time past parent_tx.timestamp + expiry.
6. Mine a block (triggers _update_tx_pool_for_reorg → remove_expired).
7. Observe: parent_tx is gone; child_tx remains in pool.
8. Query pool: child_tx is still listed as pending.
9. Attempt to submit parent_tx again: accepted (its inputs are free),
   but edges.inputs now has a duplicate entry for out_B.
10. Query total_tx_size: inflated by child_tx's size even though
    child_tx can never be committed.
11. Submit new_tx spending out_B: rejected with "Dead" conflict error,
    even though out_B does not exist on-chain or in the pool.
```

```rust
    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
```

```rust
    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
```

```rust
fn _update_tx_pool_for_reorg(
    tx_pool: &mut TxPool,
    attached: &LinkedHashSet<TransactionView>,
    detached_headers: &HashSet<Byte32>,
    detached_proposal_id: HashSet<ProposalShortId>,
    snapshot: Arc<Snapshot>,
    callbacks: &Callbacks,
    mine_mode: bool,
) {
    tx_pool.snapshot = Arc::clone(&snapshot);

    // NOTE: `remove_by_detached_proposal` will try to re-put the given expired/detached proposals into
    // pending-pool if they can be found within txpool. As for a transaction
    // which is both expired and committed at the one time(commit at its end of commit-window),
    // we should treat it as a committed and not re-put into pending-pool. So we should ensure
    // that involves `remove_committed_txs` before `remove_expired`.
    tx_pool.remove_committed_txs(attached.iter(), callbacks, detached_headers);
    tx_pool.remove_by_detached_proposal(detached_proposal_id.iter());

    // mine mode:
    // pending ---> gap ----> proposed
    // try move gap to proposed
    if mine_mode {
        let mut proposals = Vec::new();
        let mut gaps = Vec::new();

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) {
            let short_id = entry.inner.proposal_short_id();
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push((short_id, entry.inner.clone()));
            }
        }

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) {
            let short_id = entry.inner.proposal_short_id();
            let elem = (short_id.clone(), entry.inner.clone());
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push(elem);
            } else if snapshot.proposals().contains_gap(&short_id) {
                gaps.push(elem);
            }
        }

        for (id, entry) in proposals {
            debug!("begin to proposed: {:x}", id);
            if let Err(e) = tx_pool.proposed_rtx(&id) {
                debug!(
                    "Failed to add proposed tx {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e);
            } else {
                callbacks.call_proposed(&entry)
            }
        }

        for (id, entry) in gaps {
            debug!("begin to gap: {:x}", id);
            if let Err(e) = tx_pool.gap_rtx(&id) {
                debug!(
                    "Failed to add tx to gap {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e.clone());
            }
        }
    }

    // Remove expired transaction from pending
    tx_pool.remove_expired(callbacks);

    // Remove transactions from the pool until its size <= size_limit.
    let _ = tx_pool.limit_size(callbacks, None);
}
```
