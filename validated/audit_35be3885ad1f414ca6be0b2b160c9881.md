Audit Report

## Title
Unbounded Attacker-Controlled `OFFSET` in `get_tx_with_cell` SQL Query Enables Indexer RPC DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC handler (ungrouped path) decodes a 4-byte offset from the caller-supplied `after` cursor and passes it verbatim as a SQL `OFFSET` with no upper bound. Because `AsyncRichIndexerHandle` has no query timeout, a single crafted request encoding `i32::MAX` as the offset forces the database to scan and discard every matching row, rendering the indexer RPC unresponsive for the duration of the scan.

## Finding Description
**Cursor parsing — no offset validation:**
`get_transactions` splits the 12-byte `after` blob and decodes the last 4 bytes via `decode_i32`: [1](#0-0) 

`decode_i32` only validates that the slice is exactly 4 bytes; it accepts every value in `[i32::MIN, i32::MAX]` without any range check: [2](#0-1) 

**Offset injected directly into SQL:**
`get_tx_with_cell` applies the decoded offset to the query builder with no cap: [3](#0-2) 

**No timeout on rich-indexer SQL queries:**
`AsyncRichIndexerHandle` carries only `store`, `pool`, and `request_limit` — there is no `timeout_limit` field: [4](#0-3) 

**Existing `request_limit` guard is insufficient:**
The guard at L27-32 bounds `limit`, not `offset`, providing no protection against this attack vector: [5](#0-4) 

## Impact Explanation
On SQLite (the default backend), a query with `OFFSET 2147483647` forces the engine to scan and discard every row in the result set before returning. This holds the single writer connection for the full scan duration, blocking every concurrent indexer read and write. On PostgreSQL, a small number of concurrent crafted requests exhausts the connection pool. In both cases the indexer RPC becomes unresponsive with no server-side timeout to bound the damage. This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
The `after` parameter is a plain `JsonBytes` field accepted over the public JSON-RPC interface with no authentication. The crafted cursor is a fixed 12-byte value requiring no chain state, PoW, or privileged access. Any caller who can reach the RPC port can trigger this. The RPC port defaults to localhost but is commonly exposed in production deployments.

## Recommendation
1. **Clamp the offset at parse time.** After `decode_i32`, reject any value that exceeds a small practical bound (e.g., the configured `request_limit`):
   ```rust
   if offset < 0 || offset as usize > self.request_limit {
       return Err(Error::Params("invalid cursor offset".to_string()));
   }
   ```
2. **Add a query timeout to the rich-indexer.** Thread a `timeout_limit` through `AsyncRichIndexerHandle` and wrap SQL futures with `tokio::time::timeout`.
3. **Consider keyset pagination instead of OFFSET.** The `last` (`tx_id`) field already provides a keyset anchor; bounding the offset to the maximum number of cells per transaction eliminates the attack surface entirely.

## Proof of Concept

# Craft after = tx_id=1 (LE i64) || offset=i32::MAX (LE i32)
after_bytes = struct.pack("<q", 1) + struct.pack("<i", 2147483647)
after_hex = "0x" + after_bytes.hex()

payload = json.dumps({
    "jsonrpc": "2.0", "id": 1,
    "method": "get_transactions",
    "params": [
        {"script": {"code_hash": "0x" + "00"*32,
                    "hash_type": "data", "args": "0x"},
         "script_type": "lock"},
        "asc", "0x1",
        after_hex   # bytes 8-11 = 0x7FFFFFFF
    ]
})
# Send to RPC port and measure wall-clock time.
# On a node with a non-trivial indexer DB the call will hang
# far beyond any reasonable per-request budget, making the

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
