### Title
Unbounded `cells` Vec Growth in `IndexerHandle::get_transactions` with `group_by_transaction=true` Violates `limit` Memory Bound — (`util/indexer/src/service.rs`)

---

### Summary

When `group_by_transaction=true`, the break condition in `IndexerHandle::get_transactions` only fires when `tx_with_cells.len() == limit` **and** the next entry's `tx_hash` differs. For the last (or only) matching transaction in the result window, all of its indexed cells are unconditionally appended to the `cells` Vec regardless of how large that Vec grows. A caller with `limit=1` against a transaction with N outputs sharing one lock script causes O(N) allocation in a single RPC call.

---

### Finding Description

The grouped path of `get_transactions` in `util/indexer/src/service.rs` iterates over RocksDB keys and accumulates results into `tx_with_cells: Vec<IndexerTxWithCells>`. The loop's only exit condition is:

```rust
if tx_with_cells.len() == limit
    && tx_with_cells.last_mut().unwrap().tx_hash != tx_hash
{
    break;
}
``` [1](#0-0) 

When `limit=1` and the current key belongs to the **same** transaction as the last accumulated entry, the second predicate (`tx_hash != tx_hash`) is always `false`, so the loop never breaks. Every matching cell for that transaction is pushed into `last.cells`:

```rust
last.cells.push((io_type.clone(), io_index.into()));
``` [2](#0-1) 

The `request_limit` guard only validates that the caller-supplied `limit` value does not exceed a configured ceiling; it says nothing about the number of cells per transaction:

```rust
if limit > self.request_limit {
    return Err(Error::invalid_params(...));
}
``` [3](#0-2) 

The `TimeoutIterator` (default 10 s) is the only other guard, but iterating thousands of RocksDB keys completes in milliseconds, so the timeout does not prevent the unbounded accumulation. [4](#0-3) 

---

### Impact Explanation

A single RPC call allocates memory proportional to the number of outputs in the last matching transaction, completely bypassing the `limit` parameter's intended role as a response-size bound. With concurrent requests, an attacker can drive the indexer process toward OOM, causing node instability or crash. The serialized JSON response also grows proportionally, consuming additional CPU and network bandwidth.

---

### Likelihood Explanation

The RPC endpoint is unauthenticated by default. The precondition — a mined transaction with many outputs under one lock script — is realistic: batch-payment or airdrop transactions routinely have hundreds to thousands of outputs. CKB's block size (~597 KB) permits roughly 7,000–9,000 minimal outputs per transaction. The attacker does not need to create such a transaction themselves; any existing one in the indexed chain suffices. Once the precondition is met, the exploit is a single cheap HTTP call.

---

### Recommendation

Cap the inner `cells` Vec per transaction. After the `limit` check, add a per-transaction cell limit (e.g., equal to `limit` or a separate configurable cap), and break or skip once it is reached. Alternatively, restructure the loop to use `.take(limit)` on the outer transaction count and a separate `.take(cell_limit)` on the inner cell accumulation, mirroring the `get_cells` path which correctly uses `.take(limit)`: [5](#0-4) 

---

### Proof of Concept

1. Index a block containing a transaction `T` with 8,000 outputs all using the same lock script `L`.
2. Call the RPC:
   ```json
   {
     "method": "get_transactions",
     "params": [
       {"script": <L>, "script_type": "lock", "group_by_transaction": true},
       "asc", "0x1", null
     ]
   }
   ```
3. Observe that the single returned `TxWithCells` object contains 8,000 entries in its `cells` array, and that peak RSS of the indexer process spikes by tens of MB per concurrent call — confirming that `limit=1` does not bound memory usage to one cell's worth of data. [6](#0-5)
