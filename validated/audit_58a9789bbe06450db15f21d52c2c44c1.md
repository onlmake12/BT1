### Title
Unauthenticated `from` Field in `ConnectionRequestDelivered` Allows Any Connected Peer to Consume Arbitrary `inflight_requests` Entries — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

`ConnectionRequestDeliveredProcess::execute` trusts the `from` field from the message payload without verifying it matches the actual sender's peer ID. Any connected peer can craft a `ConnectionRequestDelivered` with `from = local_peer_id` and `to = victim_peer_id` (empty `route`), causing the node to call `inflight_requests.remove(&content.to)` and consume the in-flight entry. The legitimate delivery then finds `None` and returns `StatusCode::Ignore`, silently aborting NAT traversal for that cycle.

---

### Finding Description

`HolePunching` tracks outgoing hole-punch attempts in `inflight_requests: HashMap<PeerId, u64>`, keyed by the `to` peer ID. [1](#0-0) 

Entries are inserted in `notify()`: [2](#0-1) 

In `execute()`, when `content.route` is empty and `content.from == local_peer_id`, the code unconditionally removes the entry: [3](#0-2) 

The branch condition at line 151 only checks whether `content.from` (a field the attacker fully controls) equals the local peer ID — it never verifies that the actual network sender (`self.peer`) is the peer whose ID appears in `content.from`: [4](#0-3) 

The `self.peer` field (the real sender's `PeerIndex`) is present in the struct but is only used later in `respond_sync` to route the sync reply back — it is never cross-checked against `content.from`. [5](#0-4) 

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` and allows 1 request/second per tuple: [6](#0-5) 

Since the hole-punch cycle fires every 5 minutes (`CHECK_INTERVAL`), the attacker needs only one spoofed message per cycle — well within the rate limit. [7](#0-6) 

---

### Impact Explanation

An attacker who is a connected peer can permanently suppress hole-punching to any target peer ID they know. Each `notify` cycle inserts a fresh entry; the attacker races one spoofed `ConnectionRequestDelivered` per cycle to consume it. The legitimate delivery always arrives to find `None` and returns `StatusCode::Ignore`, so NAT traversal is never attempted. The attacker can learn target peer IDs by observing `ConnectionRequest` gossip broadcasts (sent to `sqrt(total)` peers, of which the attacker is one).

---

### Likelihood Explanation

- The attacker only needs a standard P2P connection — no special privileges.
- `local_peer_id` is public information in the P2P network.
- `to` peer IDs are observable from `ConnectionRequest` gossip.
- The spoofed message passes all existing validation: valid `listen_addrs` (≥1), empty `route`, and the rate limiter allows it.
- The attack is repeatable every 5-minute cycle with a single message.

---

### Recommendation

Before entering the `self_peer_id == content.from` branch, verify that the actual sender's peer ID matches `content.from`. Concretely, resolve `self.peer` (the `PeerIndex`) to its `PeerId` via the peer registry and assert it equals `content.from`. If it does not match, return `StatusCode::Ignore` (or ban the peer for sending a forged `from` field).

---

### Proof of Concept

```
1. Node A has local_peer_id = A, and has an inflight_requests entry: { B -> timestamp }.
2. Attacker E is a connected peer to A.
3. E sends ConnectionRequestDelivered {
       from: A,          // spoofed to equal A's own peer ID
       to: B,            // known from observing ConnectionRequest gossip
       route: [],        // empty → triggers the "current peer is target" branch
       listen_addrs: [<any valid TCP multiaddr>],
       sync_route: [],
   }
4. A processes the message:
   - route.last() == None  → enters the None branch
   - self_peer_id (A) == content.from (A)  → enters the else branch
   - inflight_requests.remove(&B)  → returns Some(timestamp), entry is consumed
   - respond_sync sends ConnectionSync back to E (harmless)
   - try_nat_traversal with E's fake addr (fails harmlessly)
5. Legitimate ConnectionRequestDelivered{from:A, to:B} arrives from the relay chain.
   - inflight_requests.remove(&B)  → returns None
   - Returns StatusCode::Ignore — NAT traversal is never attempted.
6. Repeat each 5-minute notify cycle to permanently block hole-punching to B.
```
