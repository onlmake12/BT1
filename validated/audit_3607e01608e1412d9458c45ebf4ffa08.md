Audit Report

## Title
`get_block_filter` RPC `.expect()` Panics on Missing Filter Hash in Inconsistent DB State — (File: rpc/src/module/chain.rs)

## Summary
`ChainRpcImpl::get_block_filter` calls `.expect("stored filter hash")` when filter data is present but the corresponding filter hash record is absent. The normal write path is atomic, but the `AddBlockFilterHash` migration writes hashes in batches of 10,000 per RocksDB transaction; an interrupted migration leaves blocks with filter data but no filter hash. A subsequent `get_block_filter` RPC call for such a block triggers the panic, causing a local RPC handler crash.

## Finding Description
The panic is at `rpc/src/module/chain.rs` lines 1738–1741: [1](#0-0) 

`store.get_block_filter_hash(&block_hash).expect("stored filter hash")` fires unconditionally when `get_block_filter` returns `Some`, with no fallback for a missing hash.

The normal write path in `store/src/transaction.rs` `insert_block_filter` writes both `COLUMN_BLOCK_FILTER` and `COLUMN_BLOCK_FILTER_HASH` atomically within the same `StoreTransaction`: [2](#0-1) 

This is committed as a single unit in `block-filter/src/filter.rs`: [3](#0-2) 

A crash during normal filter building cannot produce the desynchronized state. The realistic path is the `AddBlockFilterHash` migration at `util/migrate/src/migrations/add_block_filter_hash.rs`, which writes filter hashes in batches of 10,000 per RocksDB transaction: [4](#0-3) 

If the migration process is killed mid-batch, the in-progress batch is rolled back, leaving those blocks with `COLUMN_BLOCK_FILTER` data (written by a prior step) but no `COLUMN_BLOCK_FILTER_HASH`. The two columns are distinct column families (`"17"` and `"18"`): [5](#0-4) 

If the node subsequently starts in this partially-migrated state (e.g., by manually setting the migration version key to bypass re-run) and an RPC caller queries one of those block hashes, the `.expect()` fires.

## Impact Explanation
**Note (0–500 points): Local RPC API crash.** The panic terminates the RPC handler for that request. The precondition — filter data present without a corresponding filter hash — is not achievable by an unprivileged RPC caller through normal RPC calls alone; it requires a prior interrupted DB migration or manual DB manipulation. The impact is scoped to a local RPC crash, not a remotely-induced node shutdown.

## Likelihood Explanation
Low. The triggering condition requires the `AddBlockFilterHash` migration to have been interrupted mid-batch and the node to have started in that partially-migrated state while bypassing the migration re-run (e.g., by manually setting `MIGRATION_VERSION_KEY`). Manual DB manipulation requires local filesystem access. The condition is not reachable by a remote unprivileged caller under normal operating conditions.

## Recommendation
Replace `.expect()` with a graceful fallback:

```rust
Ok(store.get_block_filter(&block_hash).and_then(|data| {
    let hash = store.get_block_filter_hash(&block_hash)?;
    Some(BlockFilter { data: data.into(), hash: hash.into() })
}))
```

This returns `Ok(None)` when filter data is present but the hash is absent, matching the `Result<Option<BlockFilter>>` contract.

## Proof of Concept
1. Run a CKB node with block filters enabled until several blocks have filter data.
2. Interrupt the `AddBlockFilterHash` migration mid-run (e.g., `kill -9` during migration).
3. Start the node in the partially-migrated state by manually setting the `MIGRATION_VERSION_KEY` to `"20230206163640"` to bypass migration re-run.
4. Call `get_block_filter` via RPC with the hash of a block whose filter hash was in the rolled-back batch.
5. **Observed**: `.expect("stored filter hash")` panics.
6. **Expected**: `Ok(None)` or a structured error response.
