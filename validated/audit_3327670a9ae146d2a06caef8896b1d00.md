### Title
Division-by-Zero in `calculateTwap` When Start and End `publishSlot` Are Equal — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`Pyth.sol`'s `validateTwapPriceInfo` only rejects the case where `startSlot > endSlot`, but silently allows `startSlot == endSlot`. When both slots are equal, `slotDiff` is `0` and the three subsequent divisions in `calculateTwap` trigger a Solidity panic (division-by-zero), causing `parseTwapPriceFeedUpdates` to revert with an opaque panic instead of a clean protocol error.

---

### Finding Description

`validateTwapPriceInfo` enforces slot ordering with a strict-greater-than check:

```solidity
// Pyth.sol L604-606
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This passes when `startSlot == endSlot`. Control then flows to `calculateTwap`:

```solidity
// Pyth.sol L722-723
uint64 slotDiff = twapPriceInfoEnd.publishSlot -
    twapPriceInfoStart.publishSlot;   // == 0

// L731-732
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic: div/0
uint128 twapConf  = confDiff / uint128(slotDiff);           // panic: div/0

// L748
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: div/0
```

Solidity 0.8 reverts with panic code `0x12` on integer division by zero. The transaction reverts, but with an unhandled panic rather than a defined `PythErrors` selector.

**Contrast with the Solana receiver**, which correctly uses a strict-greater-than guard that explicitly rejects equal slots before reaching the arithmetic:

```rust
// pyth-solana-receiver/src/lib.rs L540-543
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots   // clean, named error
);
```

In Rust, `checked_div(0)` also returns `None` → `TwapCalculationOverflow`, providing a second safety net. The Ethereum path has neither guard.

---

### Impact Explanation

Any caller of the public payable function `parseTwapPriceFeedUpdates` who supplies two Wormhole-attested TWAP messages sharing the same `publishSlot` will receive an opaque panic revert. Because Pythnet can legitimately emit two TWAP snapshots within the same slot (e.g., two publishers at the same slot boundary), valid on-chain data can trigger this path. The result is a silent, unrecoverable failure of the TWAP query for that slot pair — the caller cannot distinguish a protocol invariant violation from a contract bug, and integrators relying on `try/catch` with specific error selectors will silently swallow the panic.

---

### Likelihood Explanation

The entry point is fully unprivileged: `parseTwapPriceFeedUpdates` is `external payable` with no access control. The only prerequisite is two valid Wormhole-guardian-signed TWAP accumulator messages whose `publishSlot` fields are identical — a realistic edge case at slot boundaries on Pythnet. No key compromise, governance majority, or Sybil attack is required.

---

### Recommendation

Change the slot-ordering check in `validateTwapPriceInfo` from strict-greater-than to greater-than-or-equal, mirroring the Solana receiver's semantics:

```solidity
// Before (L604):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After:
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This ensures `slotDiff` is always `≥ 1` before reaching `calculateTwap`, eliminating the division-by-zero path and producing a clean, named error consistent with the rest of the validation logic.

---

### Proof of Concept

1. Obtain two valid Wormhole-signed TWAP accumulator messages for the same price feed where both have `publishSlot = S` (e.g., `S = 1000`).
2. Call `parseTwapPriceFeedUpdates{value: fee}([startUpdate, endUpdate], [priceId])`.
3. `validateTwapPriceInfo` passes: `1000 > 1000` is `false`, so no revert.
4. `calculateTwap` computes `slotDiff = 1000 - 1000 = 0`.
5. `priceDiff / int128(uint128(0))` → Solidity panic `0x12` (division by zero).
6. Transaction reverts with raw panic bytes instead of `InvalidTwapUpdateDataSet`.
