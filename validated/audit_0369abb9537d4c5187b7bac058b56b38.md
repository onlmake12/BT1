Looking at the actual code in detail:

### Title
ANNOUNCE_INTERVAL Rate-Limit Bypass via Connection Cycling — (`network/src/protocols/discovery/state.rs`)

---

### Summary

`SessionState::check_timer` unconditionally fires on the first `notify()` tick for every new session because `last_announce` is initialized to `None` and the guard uses `.unwrap_or(true)`. An attacker who cycles inbound connections can reset this state on demand, causing the victim node to re-announce the attacker's address to 3 peers every ~60 s instead of every 24 h — a 1440× rate amplification.

---

### Finding Description

Every `SessionState` is created with `last_announce: None`: [1](#0-0) 

`check_timer` treats `None` as "interval has elapsed" via `.unwrap_or(true)`: [2](#0-1) 

The function returns `Some(addr)` only when `remote_addr` is `RemoteAddress::Listen`. For an inbound connection, `remote_addr` starts as `Init`, but the attacker trivially upgrades it to `Listen` by sending a `GetNodes` message with a `listen_port` field: [3](#0-2) 

Or by sending `version >= REUSE_PORT_VERSION`: [4](#0-3) 

Once `remote_addr = Listen(attacker_addr)`, the very next `notify()` tick (fired every `ANNOUNCE_CHECK_INTERVAL = 60 s`) calls `check_timer`, which returns `Some(attacker_addr)` and queues it for broadcast to up to 3 randomly chosen peer sessions: [5](#0-4) [6](#0-5) [7](#0-6) 

On disconnect, the entire `SessionState` — including `last_announce` — is destroyed: [8](#0-7) 

Reconnecting creates a fresh `SessionState` with `last_announce: None` again, resetting the 24 h clock to zero. The attacker repeats: connect → send `GetNodes` with `listen_port` → wait ≤60 s → get announced → disconnect → reconnect.

---

### Impact Explanation

The victim node announces the attacker's address to 3 of its peers every ~60 s. Those peers store the address via `add_new_addrs` → `peer_store.add_addr`. Over time, the attacker's address saturates the peer stores of nodes reachable from the victim, enabling a targeted eclipse-attack setup at 1440× the rate the protocol designers intended to allow.

---

### Likelihood Explanation

The attacker needs only a public IP (required by `is_valid_addr`) and the ability to open TCP connections to the victim — standard for any internet-connected node. No keys, no hashpower, no privileged access. The cycling loop is trivially scriptable.

---

### Recommendation

Initialize `last_announce` to `Some(Instant::now())` at session creation, or use `Some(Instant::now() - interval + small_jitter)` to allow a single early announcement without enabling full bypass. Alternatively, track announced addresses globally (not per-session) so that disconnecting and reconnecting does not reset the rate-limit state for a given remote address.

---

### Proof of Concept

```
1. Victim has sessions S1, S2, S3 already connected.
2. Attacker opens inbound connection → new SessionState { last_announce: None, remote_addr: Init(A) }.
3. Attacker sends GetNodes { listen_port: Some(P), version: 1 }.
   → remote_addr becomes Listen(A:P).
4. ~60 s later, notify() fires:
   check_timer(now, 24h) → unwrap_or(true) → true → returns Some(A:P).
   is_valid_addr(A:P) → true (public IP).
   node_flags(attacker_id) → Some(flags).
   A:P pushed to announce_multiaddrs of S1, S2, S3.
5. send_messages() delivers Nodes{announce:true, items:[A:P]} to S1, S2, S3.
   Each peer stores A:P in its peer_store.
6. Attacker disconnects → SessionState dropped, last_announce gone.
7. Attacker reconnects → goto step 2.
   Effective announcement rate: once per 60 s vs. intended once per 24 h.
```

```rust
    pub(crate) fn check_timer(&mut self, now: Instant, interval: Duration) -> Option<&Multiaddr> {
        if self
            .last_announce
            .map(|time| now.saturating_duration_since(time) > interval)
            .unwrap_or(true)
        {
            self.last_announce = Some(now);
            if let RemoteAddress::Listen(addr) = &self.remote_addr {
                Some(addr)
            } else {
                None
            }
        } else {
            None
        }
    }
```
