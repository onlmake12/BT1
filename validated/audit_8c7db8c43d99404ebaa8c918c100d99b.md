### Title
Unbounded `firstUnfulfilledSeq` Advancement Loop Outside Try-Catch in `Echo.executeCallback` Can Permanently Lock User Funds - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

`Echo.sol`'s `executeCallback` function runs a while loop that advances `_state.firstUnfulfilledSeq` past all inactive requests **before** the try-catch block that wraps the user callback. An unprivileged attacker can create many requests after a target request and pre-fulfill them all, causing the while loop to exhaust gas when the target request is later fulfilled. This permanently locks the target user's fee in the contract and prevents the request from ever being cleared.

---

### Finding Description

In `Echo.executeCallback`, the execution order is:

1. Credit the provider's accrued fee (`accruedFeesInWei +=`)
2. `clearRequest(sequenceNumber)` — marks the request inactive
3. **While loop** — advances `firstUnfulfilledSeq` past all consecutive inactive requests (each iteration calls `findRequest`, a storage lookup)
4. `try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` — the only protected block

Steps 1–3 are **outside** the try-catch. The code itself contains an explicit TODO acknowledging the hazard:

> "TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."

And a second TODO on the loop:

> "TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."

The while loop iterates from `firstUnfulfilledSeq` through every consecutive inactive sequence number up to `currentSequenceNumber`. Each iteration calls `findRequest`, which performs at least one SLOAD. If the loop runs out of gas, the entire transaction reverts atomically — undoing `clearRequest` and the fee credit — leaving the request permanently active with no way to clear it. [1](#0-0) 

---

### Impact Explanation

- The target user's fee (paid at request time) is permanently locked in the contract with no refund path.
- The provider cannot collect their fee for fulfilling the request.
- The request slot is permanently occupied; `firstUnfulfilledSeq` can never advance past it.
- The attack is **self-sustaining**: because the while loop always starts from `firstUnfulfilledSeq` (which stays at K since the tx always reverts), the N inactive requests K+1…K+N remain in place and the loop always exhausts gas on every retry.

---

### Likelihood Explanation

The attacker entry path is fully unprivileged:

- `requestPriceUpdatesWithCallback` is open to any address.
- `executeCallback` is open to any address after the exclusivity period.

Attack cost: the attacker pays fees for N requests and gas for N `executeCallback` calls. Each of those N calls is cheap because `firstUnfulfilledSeq` is stuck at K (the target's active request), so the while loop exits immediately. Only the victim's call pays the full N-iteration cost. With a 30 M gas block limit and ~2 100 gas per cold SLOAD, N ≈ 14 000 requests suffices; with warm SLOADs (100 gas) N ≈ 300 000. At low fee settings the economic cost is modest relative to permanently locking a user's funds. [2](#0-1) 

---

### Recommendation

1. **Move the `firstUnfulfilledSeq` advancement loop inside the try-catch block**, or after the callback, so that a gas-exhaustion revert in the loop does not roll back `clearRequest` and the fee credit.
2. **Replace the linear scan with a gas-bounded or O(1) structure** — the TODO already suggests a doubly-linked list of active requests.
3. Alternatively, cap the number of iterations in the while loop per call (e.g., advance at most M steps per `executeCallback` invocation).
