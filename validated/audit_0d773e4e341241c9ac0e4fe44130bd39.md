### Title
Scheduler `getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed `publishTime` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions in `Scheduler.sol` enforce staleness using a single subscription-level timestamp (`priceLastUpdatedAt`), which is set to the **maximum** `publishTime` across all feeds in the subscription. When a subscription contains mixed asset types (e.g., crypto + equities), a closed-market feed's individual `publishTime` can be days old while the subscription-level timestamp is fresh. A reader calling `getPricesNoOlderThan` with a tight age window receives stale per-feed prices without any revert, violating the freshness guarantee the function name and documentation imply.

---

### Finding Description

In `_validateShouldUpdatePrices`, the `updateTimestamp` used to set `priceLastUpdatedAt` is computed as the **maximum** `publishTime` across all feeds:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

This maximum is then stored as the subscription's last-updated timestamp:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The staleness check in `getPricesNoOlderThan` then compares `block.timestamp` against this single max value:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();
prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [3](#0-2) 

The same pattern applies to `getEmaPricesNoOlderThan`: [4](#0-3) 

The Scheduler README and constants explicitly acknowledge that closed-market feeds (e.g., equities) will carry a `publishTime` from their last trading period, which can be days in the past: [5](#0-4) 

The `parsePriceFeedUpdatesWithConfig` call uses `minAllowedPublishTime = 0` to allow these stale per-feed timestamps through: [6](#0-5) 

The result: `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` passes the staleness check (because `priceLastUpdatedAt` reflects the freshest feed, e.g., ETH/USD updated 1 second ago), but returns an AAPL price that is days old.

---

### Impact Explanation

Any protocol or reader integrating Pyth Pulse and calling `getPricesNoOlderThan` with a tight `age_seconds` window to enforce freshness will silently receive stale per-feed prices for closed-market assets. This breaks the freshness guarantee the function name and the SDK documentation promise:

> "Readers are recommended to use the SDK's functions `get(Ema)PricesNoOlderThan`, which wrap the contract's `get(Ema)PricesUnsafe` functions and validate that the price is recent." [7](#0-6) 

Downstream protocols relying on this guarantee for liquidation triggers, collateral valuation, or settlement pricing may act on prices that are hours or days stale, leading to incorrect liquidations, mispriced positions, or loss of funds for users.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support mixed-asset subscriptions (crypto + equities). The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour` constant and the `minAllowedPublishTime = 0` design choice confirm that stale per-feed timestamps are expected and accepted. Any subscription containing at least one non-crypto feed (e.g., AAPL, TSLA) alongside a crypto feed will exhibit this behavior whenever the equity market is closed. This is a routine, predictable condition — not an edge case.

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each individual feed's `publishTime`:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

Alternatively, clearly document that `getPricesNoOlderThan` guarantees only that the **subscription was updated** within `age_seconds`, not that each individual feed's price is within `age_seconds`, and provide a separate per-feed staleness function.

---

### Proof of Concept

1. Create a Scheduler subscription with two price feeds: `ETH/USD` (crypto, always active) and `AAPL/USD` (equity, market closed on weekends).
2. A keeper calls `updatePriceFeeds` on a Saturday. The ETH/USD feed has `publishTime = now`, AAPL/USD has `publishTime = Friday 4pm` (18+ hours ago). `priceLastUpdatedAt` is set to `now` (the max).
3. A reader calls `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` expecting a price no older than 60 seconds.
4. The check `distance(block.timestamp, priceLastUpdatedAt) = 0 < 60` passes.
5. The function returns the AAPL price with `publishTime` from 18+ hours ago — no revert occurs.
6. A protocol using this price for collateral valuation or liquidation acts on an 18-hour-old price, potentially causing incorrect liquidations or mispriced positions.
