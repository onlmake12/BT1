Audit Report

## Title
Unbounded `GetBlocks` Replay Allows Unauthenticated Peer to Exhaust Node Resources — (`sync/src/synchronizer/get_blocks_process.rs`)

## Summary
`GetBlocksProcess::execute()` allocates a fresh `HashSet` per invocation, preventing only intra-message duplicate hashes. The `Synchronizer` dispatches `GetBlocks` with no rate limiting and no cross-message per-peer tracking. An unprivileged peer can flood the node with repeated `GetBlocks` messages each containing 32 distinct valid block hashes, causing the node to spawn up to 32 unbounded async `SendBlock` tasks per message indefinitely, exhausting the async task pool, CPU, and outbound bandwidth.

## Finding Description
In `sync/src/synchronizer/get_blocks_process.rs`, `execute()` allocates a fresh `HashSet` on every call: [1](#0-0) 

The `take(INIT_BLOCKS_IN_TRANSIT_PER_PEER)` caps processing at 32 hashes per message, but the `dedup` guard only prevents duplicates *within* a single message. Sending N messages each with 32 distinct hashes bypasses it entirely — `StatusCode::RequestDuplicate` (code 418) is never triggered across messages, and since the normal path returns `Status::ok()` (code 100, outside the 4xx ban range), the peer is never banned.

Each valid block hash causes an async task to be unconditionally spawned: [2](#0-1) 

The `Synchronizer::try_process` dispatches `GetBlocks` with no rate-limiting wrapper: [3](#0-2) 

By contrast, the `Relayer` protocol has an explicit `RateLimiter<(PeerIndex, u32)>` keyed by peer and message type: [4](#0-3) 

Applied to all non-`CompactBlock` messages before dispatch: [5](#0-4) 

No equivalent guard exists in `Synchronizer`. The `should_ban()` logic in `sync/src/status.rs` only bans on 4xx status codes: [6](#0-5) 

Since `GetBlocksProcess::execute()` returns `Status::ok()` (code 100) for well-formed messages with distinct hashes, `post_sync_process` never bans the peer. The constants confirm the per-message cap: [7](#0-6) 

A message with up to 2,000 hashes passes the length check; `take(32)` limits processing to 32 per message, but with no rate limit, N messages yield N×32 spawned tasks.

Additionally, unlike `GetHeadersProcess` which checks `is_initial_block_download()` and sends `InIBD`, `GetBlocksProcess` contains no IBD guard, meaning the attack surface exists regardless of node sync state. [8](#0-7) 

## Impact Explanation
Each spawned task reads a full block from storage, serializes it, and sends it over the network. An attacker with a single TCP connection can exhaust the async task pool via unbounded `spawn` calls, saturate outbound bandwidth (~1 KB request → MB-scale responses per message), and consume CPU for repeated block serialization and DB reads. This constitutes a denial-of-service that can crash or severely degrade a CKB node. Impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node** and/or **cause CKB network congestion with few costs**.

## Likelihood Explanation
The attack requires only a valid P2P connection and knowledge of 32 valid block hashes, trivially obtained from any public block explorer or by first syncing headers. No PoW, no key, no privileged role is needed. The `Relayer` protocol already demonstrates the developers are aware rate limiting is necessary — its absence in `Synchronizer` for `GetBlocks` is a concrete, exploitable gap. The attack is repeatable indefinitely with no cost to the attacker beyond maintaining a connection and sending small messages.

## Recommendation
Add a per-peer rate limiter to `Synchronizer` for `GetBlocks` messages, mirroring the `RateLimiter<(PeerIndex, u32)>` pattern already used in `Relayer` (`sync/src/relayer/mod.rs` lines 63–98). Apply the check in `try_process` before dispatching to `GetBlocksProcess`. Additionally, consider maintaining a per-peer LRU cache of recently served block hashes to reject re-requests within a time window, and/or ban peers that exceed a configurable request threshold within a sliding window. An IBD guard (mirroring `GetHeadersProcess`) should also be added to `GetBlocksProcess`.

## Proof of Concept
1. Connect a peer to the node's sync port.
2. Obtain 32 valid block hashes from the chain (e.g., blocks 1–32).
3. In a tight loop, send N `SyncMessage::GetBlocks` messages each containing those same 32 hashes (all distinct within each message, so `StatusCode::RequestDuplicate` is never triggered).
4. Observe: N×32 `SendBlock` async tasks are spawned; the peer is never banned; node CPU and outbound bandwidth spike proportionally to N.

State-transition unit test sketch:
```rust
// Send N identical GetBlocks messages with 32 unique valid hashes
for _ in 0..N {
    let process = GetBlocksProcess::new(msg.as_reader(), &synchronizer, peer, &nc);
    assert_eq!(process.execute(), Status::ok()); // never banned
}
// Assert nc.sent_messages.len() == N * 32
```
