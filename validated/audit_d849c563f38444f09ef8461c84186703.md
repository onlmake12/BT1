### Title
Unbounded `last_n_numbers` allocation in `GetLastStateProofProcess::execute` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged light-client peer can craft a `GetLastStateProof` message that causes the server node to allocate a `Vec` of up to `chain_height` entries and perform a proportional number of database reads, leading to OOM crash or severe resource exhaustion. The existing `GET_LAST_STATE_PROOF_LIMIT` guard does not bound the actual number of blocks processed.

---

### Finding Description

The limit check in `execute()` only validates the sum of `difficulties.len()` and `last_n_blocks * 2`: [1](#0-0) 

With `difficulties=[]` and `last_n_blocks=0`, this evaluates to `0 + 0 = 0`, which trivially passes the `GET_LAST_STATE_PROOF_LIMIT = 1000` guard: [2](#0-1) 

After the limit check passes, the code enters the `else` branch (because `last_block_number - start_block_number > 0 = last_n_blocks`): [3](#0-2) 

Inside the `else` branch, `get_first_block_total_difficulty_is_not_less_than` is called with the attacker-supplied `difficulty_boundary`. If `difficulty_boundary` is set to `U256::from(1)` (or any value ≤ total difficulty at `start_block_number`), the binary search returns `start_block_number` immediately: [4](#0-3) 

So `difficulty_boundary_block_number = start_block_number`. The adjustment guard at line 313 only fires when `last_block_number - difficulty_boundary_block_number < last_n_blocks`; with `last_n_blocks=0` this is never true. Then: [5](#0-4) 

This collects `last_block_number - start_block_number` entries — potentially millions — into a `Vec<BlockNumber>` with no size check. This vector is then passed directly to `complete_headers` with no guard: [6](#0-5) 

`complete_headers` performs two database reads per entry (`get_ancestor` + `chain_root_mmr(n-1).get_root()`) and pushes a `packed::VerifiableHeader` per entry: [7](#0-6) 

---

### Impact Explanation

On a node with chain height ≥ 1M, a single crafted P2P message causes O(chain_height) database reads and O(chain_height) heap allocation of `VerifiableHeader` structs. This exhausts memory and/or I/O, crashing the node process (OOM kill or unrecoverable stall). Impact: **local node crash / denial of service** via a single unauthenticated P2P message.

---

### Likelihood Explanation

The light-client protocol is reachable from any peer without authentication. The crafted message is trivial to construct: set `last_n_blocks=0`, `difficulties=[]`, `difficulty_boundary=U256::from(1)`, a valid main-chain `last_hash` at the tip, and `start_number=0`. No PoW, no key, no special role required.

---

### Recommendation

After computing `last_n_numbers` (and `sampled_numbers`, `reorg_last_n_numbers`), assert that the total `block_numbers` length does not exceed `GET_LAST_STATE_PROOF_LIMIT` before calling `complete_headers`. Specifically, add a check after line 354:

```rust
if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many blocks");
}
```

The existing limit check at lines 201–205 is insufficient because it only bounds `difficulties.len() + last_n_blocks * 2` and does not account for the `last_n_numbers` range derived from `difficulty_boundary_block_number`.

---

### Proof of Concept

```
chain height: 1_000_001 blocks
message fields:
  last_hash        = hash of block #1_000_000 (valid main-chain tip)
  start_hash       = hash of block #0
  start_number     = 0
  last_n_blocks    = 0
  difficulty_boundary = U256::from(1)   // <= total_difficulty[0]
  difficulties     = []

Trace:
  limit check: 0 + 0*2 = 0 <= 1000  → passes
  is_main_chain(last_hash)            → passes
  start_number(0) <= last_number(1M)  → passes
  else branch entered (1M - 0 > 0)
  get_first_block_total_difficulty_is_not_less_than(0, 1M, 1)
    → total_difficulty[0] >= 1 → returns Some((0, ...))
  difficulty_boundary_block_number = 0
  adjustment: 1M - 0 < 0 → false, no change
  last_n_numbers = (0..1_000_000) → Vec with 1,000,000 entries
  complete_headers loops 1,000,000 times → OOM
```
