### Title
`FeeRate::fee()` Rounds Down Minimum Fee, Allowing Transactions Below Configured Minimum Fee Rate — (File: `util/types/src/core/fee_rate.rs`)

---

### Summary

`FeeRate::fee()` computes the minimum required fee using integer floor division. This means the enforced minimum fee is always rounded **down**, allowing a transaction sender to submit a transaction whose actual fee rate is measurably below the configured `min_fee_rate` or `min_rbf_rate` threshold. This is the direct CKB analog of the AMM rounding-direction bug: just as AMM buy prices should round up to prevent value leakage, the minimum fee floor should round up to prevent below-threshold transactions from entering the pool.

---

### Finding Description

`FeeRate::fee()` is the single function used to convert a fee-rate threshold into a concrete minimum shannon amount:

```rust
// util/types/src/core/fee_rate.rs
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000, integer floor
    Capacity::shannons(fee)
}
``` [1](#0-0) 

This result is used in two admission-control paths:

**1. Tx-pool minimum fee check** (`tx-pool/src/util.rs`):

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [2](#0-1) 

**2. RBF minimum replacement fee** (`tx-pool/src/pool.rs`):

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [3](#0-2) 

Because `fee = floor(fee_rate × weight / 1000)`, a transaction paying exactly `min_fee` shannons has an actual fee rate of:

```
actual_rate = floor(fee_rate × weight / 1000) × 1000 / weight
            = fee_rate − (fee_rate × weight mod 1000) / weight
```

This is **strictly less than** `fee_rate` whenever `fee_rate × weight` is not divisible by 1000.

**Concrete example with the default `min_rbf_rate = 1500` shannons/KW** (from `util/app-config/src/legacy/tx_pool.rs`): [4](#0-3) 

- `tx_size = 101` bytes  
- `min_fee = floor(1500 × 101 / 1000) = floor(151.5) = 151` shannons  
- A transaction paying exactly 151 shannons **passes** the check  
- Its actual fee rate = `151 × 1000 / 101 ≈ 1495 shannons/KW` — **5 shannons/KW below the 1500 threshold**

For the default `min_fee_rate = 1000` shannons/KW, since `KW = 1000`, the division is always exact and no rounding occurs. The issue is therefore most impactful for `min_rbf_rate = 1500` (default) and any operator-configured non-round fee rates.

The `safe_mul_ratio` used in reward splitting also floors:

```rust
pub fn safe_mul_ratio(self, ratio: Ratio) -> Result<Self> {
    self.0
        .checked_mul(ratio.numer())
        .and_then(|ret| ret.checked_div(ratio.denom()))  // floor
        .map(Capacity::shannons)
        .ok_or(Error::Overflow)
}
``` [5](#0-4) 

This is used in `RewardCalculator::proposal_reward()` to compute the proposer's 40% share:

```rust
reward = reward.safe_add(tx_fee.safe_mul_ratio(proposer_ratio)?)?;
``` [6](#0-5) 

The proposer consistently receives `floor(tx_fee × 40/100)` rather than the exact 40%, leaking fractional shannons to the committer miner on every transaction with a fee not divisible by 5. The code comment at line 136 acknowledges the rounding asymmetry but does not address the direction: [7](#0-6)

### Impact Explanation

**Primary impact — tx-pool admission bypass via `FeeRate::fee()` floor:**  
Any RPC caller (`send_transaction`) or peer relayer can submit a transaction whose actual fee rate is up to `floor(min_rbf_rate / weight)` shannons/KW below the configured threshold. With `min_rbf_rate = 1500` and a 101-byte transaction, the bypass is ~5 shannons/KW. This allows:
- Below-threshold transactions to enter the pool and be relayed
- RBF replacements to succeed with a slightly lower-than-required fee rate, undermining the RBF fee-bump guarantee

**Secondary impact — proposer reward leakage via `safe_mul_ratio()` floor:**  
On every transaction whose fee is not divisible by the proposer ratio denominator (5), the proposer receives 1 shannon less than their entitled 40%. Over millions of transactions this represents a systematic, small but consistent value leak from proposers to committer miners.

---

### Likelihood Explanation

The primary impact is reachable by any unprivileged transaction sender via the standard `send_transaction` RPC. No special privileges, keys, or majority hashpower are required. The attacker only needs to craft a transaction of a specific byte size such that `min_rbf_rate × size` is not divisible by 1000. With `min_rbf_rate = 1500`, this is true for any odd-sized transaction, which is the common case. The secondary (proposer reward) impact occurs automatically on every block containing transactions with fees not divisible by 5.

---

### Recommendation

Replace floor division with ceiling division in `FeeRate::fee()`:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    // Ceiling: (a + b - 1) / b
    let fee = (self.0.saturating_mul(weight) + KW - 1) / KW;
    Capacity::shannons(fee)
}
```

This ensures the minimum fee enforced is always at least `min_fee_rate × weight / 1000`, so no transaction with a fee rate below the threshold can pass admission. The same ceiling rounding should be applied in `safe_mul_ratio` when computing proposer rewards to prevent systematic leakage.

---

### Proof of Concept

With `min_rbf_rate = 1500` shannons/KW (the default):

```
tx_size = 101 bytes
min_fee = FeeRate(1500).fee(101)
        = 1500 * 101 / 1000          (integer floor)
        = 151500 / 1000
        = 151 shannons

Actual fee rate of a tx paying 151 shannons:
  = 151 * 1000 / 101
  = 1495 shannons/KW

1495 < 1500  →  tx passes RBF check despite being below min_rbf_rate
```

A transaction sender submits via `send_transaction` RPC a transaction of 101 serialized bytes with fee = 151 shannons. `check_tx_fee` computes `min_fee = 151` (floored), the check `fee < min_fee` is `151 < 151 = false`, so the transaction is accepted. Its true fee rate of 1495 shannons/KW is below the configured 1500 shannons/KW `min_rbf_rate` threshold.
