### Title
Tor/Onion3 Peers Bypass Ban Mechanism — (`network/src/peer_store/peer_store_impl.rs`, `network/src/peer_store/ban_list.rs`)

### Summary
`ban_addr` silently no-ops the BanList insertion for onion addresses because `multiaddr_to_socketaddr` returns `None` for them. Symmetrically, `is_addr_banned` always returns `false` for onion addresses. A misbehaving Tor peer can be "banned" and immediately reconnect with a new circuit, bypassing the ban entirely.

### Finding Description
`PeerStore::ban_addr` conditionally inserts into the BanList only when `multiaddr_to_socketaddr` succeeds:

```rust
pub(crate) fn ban_addr(&mut self, addr: &Multiaddr, timeout_ms: u64, ban_reason: String) {
    if let Some(addr) = multiaddr_to_socketaddr(addr) {   // None for onion → skipped
        let network = ip_to_network(addr.ip());
        self.ban_network(network, timeout_ms, ban_reason)  // never called
    }
    self.addr_manager.remove(addr);  // only this runs
}
```

`BanList::is_addr_banned` has the same blind spot:

```rust
pub fn is_addr_banned(&self, addr: &Multiaddr) -> bool {
    multiaddr_to_socketaddr(addr)
        .map(|socket_addr| self.is_ip_banned(&socket_addr.ip()))
        .unwrap_or_default()  // returns false for onion
}
```

The `BanList` is keyed on `IpNetwork` (`HashMap<IpNetwork, BannedAddr>`), which has no representation for onion addresses. There is no fallback mechanism (e.g., storing banned onion hostnames or peer IDs). [5](#0-4) 

### Impact Explanation
An attacker operating over Tor can:
1. Connect to a CKB node with onion support enabled.
2. Trigger a ban (e.g., via protocol misbehavior that calls `ban_session` → `ban_addr`).
3. Immediately reconnect with a new Tor circuit (new source address, trivially new peer key pair).
4. Repeat indefinitely.

This enables unlimited inbound slot exhaustion against nodes with onion support, and allows repeated protocol-level misbehavior (e.g., sending invalid headers/blocks) with no enforcement consequence.

### Likelihood Explanation
- Onion support is a production-supported configuration in CKB (confirmed by 13 references in `network.rs`).
- Tor circuits are freely obtainable; no privileged access is required.
- The attacker only needs to trigger any ban-worthy behavior (score drops below `ban_score` via `report()`).
- The bug is structural: both the write path (`ban_addr`) and the read path (`is_addr_banned`) are broken for onion addresses.

### Recommendation
The BanList must be extended to store banned onion addresses (e.g., as their raw byte representation or as a separate `HashSet<OnionAddr>`). `ban_addr` should detect onion protocol components and insert them into this separate structure. `is_addr_banned` must check this structure for onion Multiaddrs. Alternatively, ban by peer ID (stored in the Multiaddr) as a fallback when no IP is extractable.

### Proof of Concept
