Audit Report

## Title
HolePunching Per-Peer Rate Limit Bypassed via Session ID Reset on Reconnect ŌĆö (File: network/src/protocols/hole_punching/mod.rs)

## Summary
The `HolePunching` protocol's primary rate limiter is keyed by `(PeerIndex, u32)` where `PeerIndex` is a type alias for `SessionId` ŌĆö an ephemeral integer reassigned on every new TCP connection. An attacker can exhaust their 30 msg/s quota, disconnect, and immediately reconnect to receive a fresh `SessionId` with an empty bucket, bypassing the rate limit entirely. The `disconnected` handler only calls `retain_recent()` and does not remove the disconnecting peer's entry, leaving the old exhausted bucket in the map while the new session starts unconstrained.

## Finding Description
`PeerIndex` is a pure type alias for `SessionId`:
```rust
// network/src/protocols/mod.rs, line 33
pub type PeerIndex = SessionId;
```
The `HolePunching` struct holds two rate limiters with different key types:
```rust
// hole_punching/mod.rs, lines 45-46
rate_limiter: RateLimiter<(PeerIndex, u32)>,          // session-scoped, bypassable
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>, // identity-scoped, not bypassable
```
Every incoming message is gated by `rate_limiter` using the ephemeral `session_id`:
```rust
// hole_punching/mod.rs, lines 95-107
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() { return; }
```
On disconnect, no targeted removal occurs ŌĆö only a global stale-entry sweep:
```rust
// hole_punching/mod.rs, lines 66-70
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
}
```
When the attacker reconnects, the p2p layer assigns a new `SessionId` (e.g., N+1). The `rate_limiter` has no entry for `(N+1, item_id)`, so the token bucket starts completely full. The bypass is repeatable at will: each TCP reconnect cycle resets the quota, making the effective message rate `30 ├Ś reconnect_rate/s` with no upper bound enforced by the protocol.

The `forward_rate_limiter` is keyed by `(PeerId, PeerId, u32)` ŌĆö cryptographic identity ŌĆö and is not affected.

## Impact Explanation
An unprivileged remote attacker can flood a CKB node with `ConnectionRequest`, `ConnectionRequestDelivered`, and `ConnectionSync` HolePunching messages at rates far exceeding the intended 30/s cap. Processing each message consumes CPU (deserialization, state lookups, forwarding logic). Sustained flooding can exhaust CPU and network bandwidth on the victim node, degrading or halting normal block and transaction relay. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** The cost to the attacker is only TCP handshake overhead per reconnect cycle, which is negligible.

## Likelihood Explanation
Any peer that can establish a TCP connection to a CKB node can exploit this. No special privileges, cryptographic keys, or majority hashpower are required. The HolePunching protocol is enabled by default and reachable from the public internet. The reconnect cycle is cheap (single TCP handshake) and can be automated trivially. The bypass is deterministic and repeatable indefinitely.

## Recommendation
Key the primary rate limiter by `PeerId` (stable cryptographic identity) instead of `PeerIndex` (ephemeral session ID), matching the approach already used by `forward_rate_limiter`:
```rust
// Change:
rate_limiter: RateLimiter<(PeerIndex, u32)>,
// To:
rate_limiter: RateLimiter<(PeerId, u32)>,
```
In the `received` handler, extract the `PeerId` from the session context (available via `context.session.address` using `extract_peer_id`) and use it as the rate-limit key. This ensures reconnection does not reset the bucket, as the bucket is tied to cryptographic identity rather than the transient session handle.

## Proof of Concept
```
1. Attacker connects to victim CKB node.
   ŌåÆ Assigned SessionId = N (PeerIndex = N).

2. Attacker sends 30 ConnectionRequest messages within 1 second.
   ŌåÆ rate_limiter[(N, item_id)] bucket exhausted.
   ŌåÆ 31st message is dropped.

3. Attacker closes TCP connection.
   ŌåÆ disconnected() fires; retain_recent() runs but entry (N, item_id)
     is still "recent" and remains in the hashmap.

4. Attacker immediately reconnects.
   ŌåÆ Assigned SessionId = N+1.
   ŌåÆ rate_limiter has NO entry for (N+1, item_id); bucket is full.

5. Attacker sends 30 more ConnectionRequest messages within 1 second.
   ŌåÆ All 30 pass the rate check.

6. Repeat from step 3 indefinitely.
   ŌåÆ Effective throughput: 30 ├Ś reconnect_rate messages/second,
     unbounded by the intended 30/second cap.
```
