### Title
Unauthenticated `from` Field in `ConnectionRequestDelivered` Allows Any Peer to Drain `inflight_requests` — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

Any connected peer can send a crafted `ConnectionRequestDelivered` message with `from` set to the local node's own peer ID and `route=[]`. The handler unconditionally calls `inflight_requests.remove(&content.to)` without verifying that `content.from` matches the actual sender's peer ID. This silently deletes legitimate inflight hole-punching entries, causing NAT traversal to be permanently skipped for those peers.

---

### Finding Description

In `ConnectionRequestDeliveredProcess::execute`, the routing decision is:

```
route is empty → check if self_peer_id == content.from → if yes, call inflight_requests.remove(&content.to)
``` [1](#0-0) 

The `from` field is parsed directly from the attacker-supplied message bytes with no cryptographic verification that the sender actually owns that peer ID: [2](#0-1) 

The only guard before the `remove` is the `forward_rate_limiter`, keyed on `(content.from, content.to, msg_item_id)`: [3](#0-2) 

This rate limiter allows 1 request per second per `(from, to)` pair. Since the attacker controls both fields, they can target every known peer ID as a distinct `to` value, each with its own independent rate-limit bucket — effectively cancelling all inflight entries within one second of their creation.

`inflight_requests` is populated in `notify()`: [4](#0-3) 

When the legitimate `ConnectionRequestDelivered` later arrives, `inflight_requests.remove` returns `None` and the handler returns `StatusCode::Ignore`, skipping NAT traversal entirely: [5](#0-4) 

---

### Impact Explanation

Every hole-punching attempt initiated by the local node can be cancelled. The attacker pre-empts the legitimate delivered message by removing the inflight entry first. The local node never attempts NAT traversal for any peer whose entry was removed, permanently blocking new outbound connections through NAT for the duration of the attack. [6](#0-5) 

---

### Likelihood Explanation

- The local node's peer ID is public (broadcast via the Identify protocol).
- Peer IDs of NAT-traversal targets are discoverable via the peer store / discovery protocol.
- The attacker only needs a single ordinary P2P connection to the victim node.
- No special privileges, keys, or timing precision are required; the attacker can continuously re-send spoofed messages to cancel entries as fast as they are created.

---

### Recommendation

Before entering the `else` branch that calls `inflight_requests.remove`, verify that `content.from` matches the peer ID of the actual session sender (`self.peer`). The session's peer ID is already available via `self.protocol.network_state.peer_registry`. Reject (and ban) any message where `content.from` does not match the authenticated sender.

---

### Proof of Concept

1. Node A (victim) has `inflight_requests` populated with an entry for peer X (inserted during `notify()`).
2. Attacker (peer B, connected to A) sends:
   ```
   ConnectionRequestDelivered {
       from: <A's peer ID>,   // spoofed
       to:   <X's peer ID>,
       route: [],
       sync_route: [],
       listen_addrs: [<any valid TCP multiaddr>],
   }
   ```
3. In `execute()`: `route.last()` is `None`; `self_peer_id == content.from` is `true`; `inflight_requests.remove(&X)` fires and returns `Some(start)`.
4. `respond_sync` is called back to peer B (harmless), and `try_nat_traversal` runs with attacker-supplied addresses (also harmless or separately exploitable).
5. The legitimate `ConnectionRequestDelivered` from the real relay path arrives; `inflight_requests.remove(&X)` now returns `None`; the handler returns `StatusCode::Ignore`; NAT traversal is never attempted for peer X.
6. Repeat for every peer ID in A's peer store to suppress all hole-punching. [7](#0-6)
