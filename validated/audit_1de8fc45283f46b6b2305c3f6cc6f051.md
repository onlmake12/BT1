### Title
Echo Provider Gas Fee Charged on Full `callbackGasLimit` Instead of Actual Gas Consumed - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the fee charged to a requester at request time includes a gas component calculated as `callbackGasLimit * feePerGasInWei` — the theoretical maximum gas cost. However, the actual gas consumed by the callback is always ≤ `callbackGasLimit`. There is no refund mechanism for unused gas. The entire upfront fee is unconditionally credited to the provider upon `executeCallback`, regardless of how much gas the callback actually consumed.

---

### Finding Description

`getFee` computes the total fee as:

```
gasFee = callbackGasLimit * providerFeeInWei
feeAmount = pythFeeInWei + providerBaseFee + providerFeedFee + gasFee
``` [1](#0-0) 

At request time, the full fee (including the maximum gas component) is collected from the requester and stored:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

At `executeCallback`, the entire stored `req.fee` is credited to the provider with no adjustment for actual gas used:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [3](#0-2) 

The callback is invoked with exactly `req.callbackGasLimit` gas forwarded:

```solidity
IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(sequenceNumber, priceFeeds)
``` [4](#0-3) 

There is no code path that measures actual gas consumed by the callback and refunds the difference to the requester. The provider earns `(callbackGasLimit - actualGasUsed) * feePerGasInWei` as unearned profit on every fulfilled request.

---

### Impact Explanation

Every requester systematically overpays for callbacks. The gas fee component is priced at the worst-case gas limit, but real callbacks almost always consume a fraction of that limit. The excess fee — `(callbackGasLimit - actualGasUsed) * feePerGasInWei` — is permanently transferred to the provider with no recourse for the requester. For high gas limits (e.g., 1,000,000 gas) with a non-trivial `feePerGasInWei`, this overpayment can be substantial per request and compounds across all users of the protocol.

---

### Likelihood Explanation

This affects every single `requestPriceUpdatesWithCallback` call where the callback uses less than `callbackGasLimit` gas — which is the normal case, since users set limits conservatively to avoid out-of-gas failures. The entry path is fully unprivileged: any user calling `requestPriceUpdatesWithCallback` is affected. No special conditions are required.

---

### Recommendation

Measure actual gas consumed during the callback execution and refund the unused portion to the requester:

```solidity
uint256 gasStart = gasleft();
// ... execute callback ...
uint256 gasUsed = gasStart - gasleft();
uint256 unusedGasFee = (req.callbackGasLimit - gasUsed) * providerFeePerGas;
// Refund unusedGasFee to req.requester
// Credit only actualGasFee to provider
```

Alternatively, adopt a post-execution fee settlement model where the provider fee for gas is calculated from actual gas used rather than the limit, similar to how `Scheduler.sol` measures `startGas - gasleft()` for keeper compensation. [5](#0-4) 

---

### Proof of Concept

1. Provider registers with `feePerGasInWei = 1 wei`.
2. Alice calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 1,000,000`. She pays `1,000,000 wei` in gas fees (plus base/feed fees).
3. The callback is executed and consumes only 50,000 gas.
4. Provider is credited the full `req.fee` including `1,000,000 wei` gas component.
5. Alice receives no refund. The provider earned `950,000 wei` for gas it never spent.
6. Across 1,000 such requests, the provider earns `950,000,000 wei` (~0.00095 ETH) in unearned gas fees at the requester's expense. [6](#0-5)
