### Title
Cross-Chain Signature Replay in `PythLazer.verifyUpdate` Due to Missing Chain ID in Signing Hash — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `PythLazer.verifyUpdate` function on EVM computes the signing hash as `keccak256(payload)` where `payload` contains no chain ID, contract address, or any chain-specific domain separator. Because the same Lazer trusted signer key is registered across multiple EVM deployments, a valid signed price update from one chain is cryptographically valid on every other EVM chain where the same signer is trusted. An unprivileged relayer can replay a signed update from chain A onto chain B to inject a stale or cross-chain-divergent price.

---

### Finding Description

In `PythLazer.verifyUpdate`, the verification hash is computed as:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` bytes are structured as `PAYLOAD_FORMAT_MAGIC (4 bytes) | timestamp (8 bytes) | channel (1 byte) | feedsLen (1 byte) | feeds...` — confirmed by `parsePayloadHeader`:

```solidity
uint32 FORMAT_MAGIC = 2479346549;
pos = 0;
uint32 magic = _readBytes4(update, pos);
pos += 4;
timestamp = _readBytes8(update, pos);
pos += 8;
channel = PythLazerStructs.Channel(_readBytes1(update, pos));
``` [2](#0-1) 

There is no `chainId`, no contract address, and no EIP-712 domain separator anywhere in the payload or in the hash construction. The `EVM_FORMAT_MAGIC = 706910618` is a protocol-level constant, not chain-specific. [3](#0-2) 

The same trusted signer address is registered on every EVM deployment of `PythLazer` via `updateTrustedSigner`. Validity is checked only against the signer's expiry:

```solidity
function isValidSigner(address signer) public view returns (bool) {
    return block.timestamp < trustedSignerToExpiresAtMapping[signer];
}
``` [4](#0-3) 

Because `hash = keccak256(payload)` is identical on every EVM chain for the same payload bytes, a signature `(v, r, s)` produced by the trusted signer for chain A is also a valid signature on chain B, C, etc.

The same structural issue exists in the Sui contract, where `secp256k1_ecrecover(signature, payload, 0)` is called directly on the raw payload with no chain-specific prefix: [5](#0-4) 

---

### Impact Explanation

An attacker acting as an unprivileged relayer can:

1. Observe a signed Lazer price update on chain A (e.g., Ethereum) where ETH/USD = $3,000.
2. Submit that exact `update` bytes to `PythLazer.verifyUpdate` on chain B (e.g., Arbitrum) where the current price is $2,800.
3. The contract on chain B accepts the signature as valid (same signer, same hash, no chain binding).
4. Any consumer contract on chain B that calls `verifyUpdate` and uses the returned `payload` will receive the stale/wrong price from chain A.

Consumer contracts that use the returned price for lending, derivatives, or liquidation decisions can be exploited to borrow against inflated collateral, avoid liquidation, or trigger incorrect liquidations. The `verifyUpdate` function returns the `payload` and `signer` directly to the caller with no further chain-binding check. [6](#0-5) 

---

### Likelihood Explanation

- Pyth Lazer is deployed on multiple EVM chains (Ethereum, Arbitrum, Polygon, etc.) with the same trusted signer key.
- The attacker only needs to monitor public on-chain calldata or the Lazer WebSocket stream to obtain a valid signed update.
- No privileged access, leaked key, or brute force is required — the attacker simply submits a legitimately signed update to the wrong chain.
- Price divergence between chains is common during volatile market conditions, making the attack window realistic and recurring.

---

### Recommendation

Include the chain ID (and optionally the contract address) in the signed digest. For example:

```solidity
bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
```

Alternatively, adopt EIP-712 with a proper domain separator:

```solidity
bytes32 domainSeparator = keccak256(abi.encode(
    keccak256("EIP712Domain(string name,uint256 chainId,address verifyingContract)"),
    keccak256("PythLazer"),
    block.chainid,
    address(this)
));
bytes32 hash = keccak256(abi.encodePacked("\x19\x01", domainSeparator, keccak256(payload)));
```

The Lazer signing infrastructure must be updated to include the chain ID when producing signatures, and the on-chain verifier must enforce it. The same fix should be applied to the Sui contract (`verify_le_ecdsa_message`) by prepending a chain-specific prefix to the payload before hashing.

---

### Proof of Concept

1. Deploy `PythLazer` on two testnets (chain A = Sepolia, chain B = Arbitrum Sepolia) with the same trusted signer.
2. On chain A, call `verifyUpdate{value: fee}(update)` with a valid signed update — it succeeds and returns `payload`.
3. On chain B, call `verifyUpdate{value: fee}(update)` with the **identical** `update` bytes — it also succeeds and returns the same `payload`, despite the update being signed for chain A.
4. Confirm that `signer` returned on chain B matches the trusted signer registered on chain B, and that `isValidSigner(signer)` returns `true`.

The attack requires no special tooling beyond a standard Ethereum RPC client. The signed `update` bytes are publicly observable from any transaction that calls `verifyUpdate` on chain A.

        bytes calldata update

        // Require fee and refund excess

            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
