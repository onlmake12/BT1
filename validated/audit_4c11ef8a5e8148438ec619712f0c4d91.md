### Title
Unbounded RPC Batch Requests by Default Enable Resource Exhaustion DoS — (`File: rpc/src/server.rs`)

---

### Summary

The CKB JSON-RPC server enforces no rate limiting on individual RPC calls and, critically, the batch request size limit (`rpc_batch_limit`) is **disabled by default**. Any RPC caller can submit a single HTTP POST containing an arbitrarily large JSON-RPC batch of expensive operations (e.g., `get_block_template`, `send_transaction`), exhausting the tx-pool service channel, CPU, and memory, making the node unresponsive to legitimate requests.

---

### Finding Description

In `rpc/src/server.rs`, the `handle_jsonrpc` function processes batch requests by iterating over all calls sequentially with no upper bound unless `JSONRPC_BATCH_LIMIT` is set:

```rust
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    {
        return make_error_response(...);
    }
    let stream = stream::iter(calls)
        .then(move |call| { ... io.handle_call(call, T::default()).await })
        ...
}
``` [1](#0-0) 

`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only initialized if `config.rpc_batch_limit` is `Some(...)`: [2](#0-1) 

The `rpc_batch_limit` field in `RpcConfig` is `Option<usize>` and defaults to `None`: [3](#0-2) 

The production config explicitly documents this gap and leaves the limit commented out: [4](#0-3) 

Each call in the batch is dispatched to the tx-pool service via a bounded channel of size 512 (`DEFAULT_CHANNEL_SIZE`): [5](#0-4) 

The `send_message!` macro uses `try_send`, which will return an error if the channel is full, but the batch loop continues processing remaining calls, and each successful call blocks the RPC worker thread waiting for a response from the single-threaded tx-pool service actor: [6](#0-5) 

Additionally, the TCP RPC server (`start_tcp_server`) accepts an unbounded number of persistent connections with no connection cap, spawning a new Tokio task per connection: [7](#0-6) 

By contrast, the P2P relay protocol does implement per-peer rate limiting (30 req/sec), showing the pattern is known and intentionally applied elsewhere but absent from the RPC layer: [8](#0-7) 

---

### Impact Explanation

An attacker (or buggy local application) submitting a single HTTP POST with tens of thousands of `get_block_template` or `send_transaction` calls will:

1. Saturate the tx-pool service's 512-slot channel, causing all subsequent legitimate tx submissions and block template requests to fail with channel-full errors.
2. Hold the RPC worker thread for up to 30 seconds (the `TimeoutLayer` limit) processing the batch sequentially, consuming CPU proportional to batch size.
3. Exhaust memory with pending response objects for each call in the batch.
4. Render the node unable to serve miners (`get_block_template`) or relay transactions, effectively halting participation in the network.

---

### Likelihood Explanation

The RPC is documented as localhost-only by default, but:
- Operators routinely expose it to internal networks or behind proxies.
- The TCP/WS endpoints (`tcp_listen_address`, `ws_listen_address`) can be bound to external interfaces.
- Even a local process (buggy miner software, a script in a loop) can trigger this without any network access.
- The config itself warns that a large batch "may cost a lot of memory or makes the RPC server slow" but ships with the limit disabled, meaning most deployments are vulnerable by default.

The bounty scope explicitly includes "RPC caller" and "miner/block-template caller" as valid attacker roles.

---

### Recommendation

1. **Enable `rpc_batch_limit` by default** in the production `ckb.toml` (e.g., `rpc_batch_limit = 100`) rather than leaving it commented out.
2. **Add per-IP or per-connection request rate limiting** at the HTTP/TCP server layer (e.g., using `tower`'s `RateLimitLayer` or `governor`).
3. **Cap concurrent TCP RPC connections** in `start_tcp_server` using a semaphore, analogous to how the P2P layer caps inbound peers via `PeerRegistry`.

---

### Proof of Concept

```bash
# Generate a batch of 50,000 get_block_template calls
python3 -c "
import json, sys
batch = [{'jsonrpc':'2.0','method':'get_block_template','params':[None,None,None],'id':i} for i in range(50000)]
print(json.dumps(batch))
" > big_batch.json

# Send to default RPC endpoint (no rpc_batch_limit set)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data @big_batch.json > /dev/null &

# Immediately try a legitimate call — will time out or fail
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_tip_block_number","params":[],"id":1}'
```

The second call will either time out (30 s `TimeoutLayer`) or receive a channel-full error while the batch monopolizes the tx-pool service, demonstrating denial of service to legitimate callers.
