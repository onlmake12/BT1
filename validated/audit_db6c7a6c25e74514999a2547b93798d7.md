### Title
Permanent Subscription Funds Permanently Locked With No Recovery Path - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, once a subscription manager sets `isPermanent = true`, all ETH deposited into that subscription is permanently locked. The `withdrawFunds` function unconditionally reverts for permanent subscriptions, the `isPermanent` flag itself is irreversible, and there is no admin/governance escape hatch to recover the funds. This is a direct analog to the escrow "no arbiter → no dispute → no recovery" pattern.

---

### Finding Description

The `withdrawFunds` function in `Scheduler.sol` contains an unconditional guard that blocks all withdrawals from permanent subscriptions:

```solidity
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount
) external override onlyManager(subscriptionId) {
    ...
    // Prevent withdrawals from permanent subscriptions
    if (params.isPermanent) {
        revert SchedulerErrors.CannotUpdatePermanentSubscription();
    }
    ...
}
``` [1](#0-0) 

The `isPermanent` flag is also irreversible. Any call to `updateSubscription` on a permanent subscription reverts immediately, including attempts to set `isPermanent = false` or `isActive = false`:

```solidity
if (currentParams.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [2](#0-1) 

This means once `isPermanent = true` is set, the subscription can never be deactivated, never have its parameters changed, and the manager can never withdraw any remaining balance. The only way ETH leaves the subscription is through keeper fee deductions via `updatePriceFeeds`.

A compounding issue: `addFunds` has no access control — **anyone** can add ETH to a permanent subscription (up to `MAX_DEPOSIT_LIMIT = 100 ETH` per transaction), and those funds are also permanently locked:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    ...
    if (!params.isActive) { revert SchedulerErrors.InactiveSubscription(); }
    if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
    status.balanceInWei += msg.value;
    ...
}
``` [3](#0-2) 

The `MAX_DEPOSIT_LIMIT` constant is 100 ETH: [4](#0-3) 

---

### Impact Explanation

A subscription manager who sets `isPermanent = true` permanently surrenders all remaining ETH in the subscription. If the subscribed price feeds are later deprecated, if the keeper network stops servicing the subscription (e.g., due to insufficient balance to cover keeper fees), or if the manager simply no longer needs the subscription, the remaining balance is irrecoverable. There is no admin/governance function in `SchedulerGovernance.sol` or elsewhere that can rescue these funds. Additionally, any third party who calls `addFunds` on a permanent subscription also permanently loses their ETH.

**Impact:** Direct, permanent loss of user ETH with no recovery path.

---

### Likelihood Explanation

The `isPermanent` flag is a user-controlled setting that any subscription manager can set via `updateSubscription`. A manager may set it intending to signal long-term commitment, not realizing the flag is irreversible and that all future balance is permanently locked. The scenario where a subscription outlives its usefulness (deprecated price feeds, changed business needs) is realistic and common. Likelihood is **medium** — the action is self-inflicted but the irreversibility and lack of any escape hatch is not clearly communicated.

---

### Recommendation

1. Add an emergency withdrawal function that allows the subscription manager to recover remaining funds from a permanent subscription after a time-delay (e.g., 30 days), analogous to the escrow report's recommendation.
2. Alternatively, allow governance/admin to recover funds from permanent subscriptions in exceptional circumstances.
3. At minimum, restrict `addFunds` to the subscription manager for permanent subscriptions, so third parties cannot accidentally lock their own ETH.

---

### Proof of Concept

1. Manager calls `createSubscription{value: 50 ether}(params)` with `params.isPermanent = true`. Subscription ID `N` is created with 50 ETH balance.
2. Manager later decides the price feeds are no longer needed and calls `withdrawFunds(N, 50 ether)`.
3. Transaction reverts with `CannotUpdatePermanentSubscription`.
4. Manager tries `updateSubscription(N, params_with_isPermanent_false)` — also reverts with `CannotUpdatePermanentSubscription`. [2](#0-1) 
5. Manager tries `updateSubscription(N, params_with_isActive_false)` — also reverts. [2](#0-1) 
6. The 50 ETH is permanently locked. The only exit is keeper fee deductions, which stop once the balance falls below the minimum required to trigger an update. [5](#0-4)

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
