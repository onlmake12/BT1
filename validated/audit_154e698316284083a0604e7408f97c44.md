Audit Report

## Title
`calculate_min_replace_fee` Uses Raw `size` Instead of `get_transaction_weight(size, cycles)` for RBF Fee Enforcement — (File: `tx-pool/src/pool.rs`)

## Summary
`calculate_min_replace_fee` computes the minimum fee required to replace conflicting transactions using only the serialized byte `size` of the incoming transaction, ignoring cycles. Every other fee-rate-sensitive path in the tx-pool uses `get_transaction_weight(size, cycles)`, which returns `max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For any transaction where cycles dominate, the required RBF fee increment is underestimated, allowing an attacker to replace pool transactions at a fraction of the cost the node operator's `min_rbf_rate` policy intends.

## Finding Description

`get_transaction_weight` is defined in `util/types/src/core/tx_pool.rs` at L298–303 and returns `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. [1](#0-0) 

All fee-rate-sensitive paths use it correctly:
- `fee_rate()` in `TxEntry` at `tx-pool/src/component/entry.rs` L115–117 [2](#0-1) 
- `AncestorsScoreSortKey::from` at L221–231 [3](#0-2) 
- `EvictKey::from` at L234–247 [4](#0-3) 

`calculate_min_replace_fee` does not — it passes raw `size as u64` directly to `min_rbf_rate.fee()`: [5](#0-4) 

The caller `check_rbf` at L665 passes only `entry.size`, never `entry.cycles`: [6](#0-5) 

The analogous `check_tx_fee` path in `tx-pool/src/util.rs` carries an explicit comment acknowledging the size-only approach as an intentional "cheap check" for `min_fee_rate`: [7](#0-6) 

No such acknowledgment exists in `calculate_min_replace_fee`, confirming the divergence is unintentional. `FeeRate::fee(weight)` computes `fee_rate * weight / KW`: [8](#0-7) 

For a transaction where `cycles * DEFAULT_BYTES_PER_CYCLES >> size`, the weight returned by `get_transaction_weight` is much larger than `size`, so the actual required fee increment is far higher than what `calculate_min_replace_fee` enforces.

**Concrete example** (`min_rbf_rate = 1000 shannons/KW`, `size = 100`, `cycles = 10,000,000`):
- `get_transaction_weight(100, 10_000_000) = max(100, ~1705) = 1705`
- Expected extra fee: `1000 × 1705 / 1000 = 1705 shannons`
- Actual extra fee enforced: `1000 × 100 / 1000 = 100 shannons`
- Attacker pays ~17× less than policy intends.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The `min_rbf_rate` threshold exists specifically to make RBF replacement economically non-trivial and to prevent low-cost replacement spam. An attacker can repeatedly replace pool transactions at ~17× (or more) lower cost than the node operator's `min_rbf_rate` is designed to enforce, enabling sustained low-cost churn of the mempool at a fraction of the intended economic barrier.

## Likelihood Explanation

RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is a supported and documented node configuration confirmed at `tx-pool/src/pool.rs` L81–83. [9](#0-8) 

Any unprivileged RPC caller can submit transactions. CKB scripts can be written to consume near-maximum cycles with minimal serialized witness/output data (e.g., a tight loop in a script). The attacker controls both the victim transaction (T1) and the replacement (T2), so no external cooperation is needed. The condition is fully self-contained and repeatable.

## Recommendation

Replace the raw `size` argument in `calculate_min_replace_fee` with `get_transaction_weight(size, cycles)`. The caller `check_rbf` already has access to `entry.cycles`:

```rust
// In check_rbf (pool.rs L665):
if let Some(min_replace_fee) = self.calculate_min_replace_fee(&all_conflicted, entry.size, entry.cycles) {

// In calculate_min_replace_fee:
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize, cycles: Cycle) -> Option<Capacity> {
    let weight = get_transaction_weight(size, cycles);
    let extra_rbf_fee = self.config.min_rbf_rate.fee(weight);
    ...
}
```

Also update the `min_replace_fee` public method at L86–99 which calls `calculate_min_replace_fee` with only `tx.size` and has the same defect. [10](#0-9) 

## Proof of Concept

1. Configure a CKB node with `min_rbf_rate > min_fee_rate` (RBF enabled).
2. Submit victim transaction T1 with a moderate fee.
3. Craft replacement T2 that:
   - Spends the same inputs as T1
   - Has small serialized size (e.g., 100 bytes)
   - Executes a script consuming near-maximum cycles (e.g., 10,000,000)
   - Pays `T1.fee + min_rbf_rate.fee(100)` — only the size-based increment
4. Submit T2 via `send_transaction` RPC.
5. Observe T2 is accepted: `calculate_min_replace_fee` checks only `size = 100`, requiring 100 shannons extra, while the weight-correct check would require ~1705 shannons.
6. Repeat steps 2–4 to demonstrate sustained low-cost mempool churn.

Unit test: construct a `TxEntry` with `size = 100` and `cycles = 10_000_000`, assert that the returned `min_replace_fee` equals `min_rbf_rate.fee(100)` (demonstrating the bug), then apply the fix and assert it equals `min_rbf_rate.fee(get_transaction_weight(100, 10_000_000))`.
