### Title
Alert Signature Hash Lacks Chain-Context Binding, Enabling Cross-Network Alert Replay — (File: `util/network-alert/src/verifier.rs`)

---

### Summary

The CKB network alert system signs and verifies alerts using a hash of the raw alert payload (`RawAlert`). This hash does not include any chain-specific context (genesis block hash, network ID, etc.). Because the same hard-coded signing keys are compiled into every CKB binary regardless of network, a validly-signed alert from one CKB network (e.g., testnet) can be replayed verbatim on another network (e.g., mainnet) by any unprivileged P2P peer or RPC caller, and it will pass signature verification.

---

### Finding Description

`verify_signatures()` in `util/network-alert/src/verifier.rs` constructs the message to verify as:

```rust
let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
```

`calc_alert_hash()` in `util/gen-types/src/extension/calc_hash.rs` is:

```rust
pub fn calc_alert_hash(&self) -> packed::Byte32 {
    self.raw().calc_alert_hash()   // = blake2b(raw_alert_bytes)
}
```

The `RawAlert` molecule struct contains only: `id`, `cancel`, `min_version`, `max_version`, `priority`, `notice_until`, `message`. It contains **no chain-specific field** — no genesis block hash, no network identifier, no chain ID.

The default `NetworkAlertConfig` is compiled from `util/app-config/src/configs/alert_signature.toml`, which hard-codes 4 public keys with a 2-of-4 threshold. This same config is the default for every CKB binary, meaning mainnet and testnet nodes share the same verifier key set.

An unprivileged actor who obtains a validly-signed alert from testnet (e.g., by observing the testnet P2P network or the testnet `send_alert` RPC) can submit the identical `Alert` bytes to a mainnet node via:
- The P2P Alert protocol handler (`alert_relayer.rs` `received()`)
- The `send_alert` JSON-RPC endpoint (`rpc/src/module/alert.rs`)

The mainnet node will call `verify_signatures()`, compute `blake2b(raw_alert_bytes)`, verify the signatures against the same hard-coded keys, and accept the alert — because the hash is identical on both networks for the same payload.

---

### Impact Explanation

**Impact: Medium**

Every mainnet CKB node that receives the replayed alert will:
1. Accept it as valid (signature check passes)
2. Broadcast it to all connected peers via `quick_filter_broadcast`
3. Display the alert message to users via the notifier

A testnet-specific alert (e.g., "CKB v0.X has a critical bug, stop transacting") replayed on mainnet would be shown to all mainnet users, causing unnecessary panic, halted activity, or confusion. The alert system is the only out-of-band emergency communication channel to all nodes; corrupting it with cross-network replays undermines its trustworthiness.

---

### Likelihood Explanation

**Likelihood: Low**

The attack requires:
1. The Nervos foundation to have legitimately signed an alert for one network (a normal operational event)
2. An attacker to capture those signed bytes (trivially done by any peer on that network)
3. The attacker to relay them to the other network (requires connectivity to both, or use of the `send_alert` RPC on a mainnet node)

No keys are needed by the attacker. The only prerequisite is that a valid alert has been issued on any CKB network sharing the same signing keys.

---

### Recommendation

Include a chain-specific domain separator in the alert hash computation. The genesis block hash is the canonical chain identifier already available in the node. The fix is to include it when computing the alert hash:

```rust
// In calc_alert_hash or in verify_signatures:
let mut blake2b = new_blake2b();
blake2b.update(genesis_hash.as_slice());   // chain-specific context
blake2b.update(raw_alert.as_slice());
```

This is the direct analog of adding `address(this)` to the hash in the referenced report: binding the commitment to the specific chain context so it cannot be replayed on a different network.

**Replay steps:**

1. Connect to CKB testnet as a peer; observe a broadcast `Alert` message (or call `get_blockchain_info` to retrieve active alerts).
2. Extract the raw `Alert` bytes (already contains valid 2-of-4 signatures from the Nervos foundation keys).
3. Connect to a CKB mainnet node and call `send_alert` RPC with the identical alert payload, or relay the bytes directly via the P2P Alert protocol.
4. `verify_signatures()` computes `blake2b(raw_alert_bytes)` — identical on both networks — and verifies against the same hard-coded keys. Verification passes.
5. The mainnet node accepts the alert, broadcasts it to all its peers, and displays the testnet-specific message to all mainnet users.

    fn send_alert(&self, alert: Alert) -> Result<()> {
        let alert: packed::Alert = alert.into();
        let now_ms = ckb_systemtime::unix_time_as_millis();
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until < now_ms {
            return Err(RPCError::invalid_params(format!(
                "Expected `params[0].notice_until` in the future (> {now_ms}), got {notice_until}",
            )));
        }

        let result = self.verifier.verify_signatures(&alert);

        match result {
            Ok(()) => {
                // set self node notifier
                self.notifier.lock().add(&alert);

                self.network_controller.broadcast_with_handle(
                    SupportProtocols::Alert.protocol_id(),
                    alert.as_bytes(),
                    &self.handle,
                );
                Ok(())
            }
            Err(e) => Err(RPCError::custom_with_error(
                RPCError::AlertFailedToVerifySignatures,
                e,
            )),
        }
    }
