### Title
Incorrect `int32`→`int8` Narrowing Cast in `decimals()` Causes DOS for Positive or Out-of-Range Exponents — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3.decimals()` silently truncates `price.expo` (an `int32`) to `int8` before negating and casting to `uint8`. This causes an arithmetic revert in Solidity 0.8 for any price feed whose exponent is positive, equals `-128`, or is less than `-128`. Any Chainlink-compatible protocol that calls `decimals()` to normalize prices is permanently DOSed for that feed.

---

### Finding Description

The `decimals()` function performs:

```solidity
function decimals() public view virtual returns (uint8) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return uint8(-1 * int8(price.expo));   // line 42
}
```

`price.expo` is declared as `int32` in `PythStructs.Price`:

```solidity
struct Price {
    int64 price;
    uint64 conf;
    int32 expo;       // full int32 range
    uint publishTime;
}
```

The cast chain `uint8(-1 * int8(price.expo))` has three failure modes under Solidity ≥0.8 checked arithmetic:

| `expo` value | `int8(expo)` | `-1 * int8(expo)` | `uint8(...)` | Result |
|---|---|---|---|---|
| `-8` (typical) | `-8` | `8` | `8` | ✓ OK |
| `5` (positive) | `5` | `-5` | — | **REVERT** (negative → uint) |
| `-128` | `-128` | `128` | — | **REVERT** (int8 overflow) |
| `-200` | `56` (truncated) | `-56` | — | **REVERT** (negative → uint) |

Solidity 0.8 explicit narrowing casts (`int8(int32_value)`) silently truncate without reverting, but casting a negative `int8` to `uint8` reverts with an arithmetic overflow panic.

The Pyth protocol defines `expo` as `int32`, and positive exponents are a valid part of the data model (e.g., large-integer commodity prices). The on-chain price store accepts any exponent that arrives in a valid Wormhole-signed accumulator update.

---

### Impact Explanation

Any Chainlink-compatible protocol that calls `PythAggregatorV3.decimals()` to scale prices — a standard pattern in Chainlink integrations — will revert and be permanently DOSed for any feed whose `expo` is positive, equals `-128`, or is below `-128`. This breaks price normalization, liquidation checks, collateral valuation, and any other function that depends on `decimals()` returning a valid value. The DOS persists as long as the on-chain price retains the problematic exponent.

---

### Likelihood Explanation

Current Pyth feeds use exponents in the range `-8` to `-18`, so the bug is not triggered today. However:

1. The `int32` type explicitly permits positive exponents; the protocol makes no guarantee they will never appear.
2. Anyone can call `updatePriceFeeds()` with a valid Wormhole-signed update containing a positive exponent. If Pyth's network ever publishes such a feed (e.g., for a new asset class), the adapter breaks immediately.
3. The root cause is entirely within Pyth's own adapter code — the incorrect narrowing cast — not in any external system.

---

### Recommendation

Perform all arithmetic in `int32`, validate the range, and only then convert:

```solidity
function decimals() public view virtual returns (uint8) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    require(price.expo <= 0 && price.expo >= -255, "PythAggregatorV3: invalid exponent");
    return uint8(uint32(-price.expo));
}
```

This mirrors the safe pattern already used in `PythUtils.convertToUint`, which accepts `int32 expo` and validates it before use.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a Pyth contract and a `priceId`.
2. Submit a valid price update (via `updatePriceFeeds`) for that `priceId` with `expo = 5`.
3. Call `decimals()` on the adapter.
4. The call reverts with `Panic(0x11)` (arithmetic overflow) because `uint8(-1 * int8(5))` = `uint8(-5)` is invalid.
5. Any downstream protocol (e.g., a lending protocol using this adapter) that calls `decimals()` before computing a normalized price is now permanently DOSed for this feed.
