### Title
`GetBlocksProcess::execute` Missing IBD State Gate Allows Block Serving During Initial Block Download — (`sync/src/synchronizer/get_blocks_process.rs`)

---

### Summary

The synchronizer module explicitly documents that during IBD the node must respond with `InIBD` to **both** `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess::execute` enforces this gate; `GetBlocksProcess::execute` does not. Any unprivileged peer can therefore force an IBD node to read and transmit verified blocks from disk, consuming I/O, CPU, and upload bandwidth that the node needs for its own block download and verification pipeline.

---

### Finding Description

`sync/src/synchronizer/mod.rs` line 4 states the invariant explicitly:

> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` **and** `GetBlocks` requests"

`GetHeadersProcess::execute` honours this:

```rust
// sync/src/synchronizer/get_headers_process.rs:53-66
if active_chain.is_initial_block_download() {
    info!("Ignoring getheaders from peer={} ...", self.peer);
    self.send_in_ibd();
    ...
    return Status::ignored();
}
```

`GetBlocksProcess::execute` has no such check. It proceeds directly to block lookup and transmission:

```rust
// sync/src/synchronizer/get_blocks_process.rs:33-96
pub fn execute(self) -> Status {
    let block_hashes = self.message.block_hashes();
    // ... only checks count limit and dedup ...
    for block_hash in iter {
        if let Some(block) = active_chain.get_block(&block_hash) {
            // sends block unconditionally — no IBD gate
            ...spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
        }
    }
    Status::ok()
}
```

The synchronizer's `received` handler (`mod.rs:890`) has no global IBD guard (unlike the relayer's `received` at `relayer/mod.rs:816` which returns immediately when IBD is active). Each message type is dispatched individually through `try_process`, so the per-handler gate in `GetHeadersProcess` is the only protection — and `GetBlocksProcess` lacks it entirely.

---

### Impact Explanation

During IBD an attacker-controlled peer can repeatedly send `GetBlocks` messages (up to `MAX_HEADERS_LEN` = 500 hashes per message, iterated `INIT_BLOCKS_IN_TRANSIT_PER_PEER` at a time). For each hash that maps to a `BLOCK_VALID` block the node reads the full block from RocksDB and serialises it over the network. Consequences:

1. **Upload bandwidth exhaustion** — the IBD node's uplink is consumed serving the attacker instead of receiving new blocks.
2. **Disk I/O saturation** — repeated random block reads compete with the IBD pipeline's sequential writes and reads.
3. **CPU overhead** — block serialisation and async task spawning per block.
4. **IBD slowdown** — all three resources are on the critical path of the IBD pipeline; degrading them extends the time the node spends in IBD, during which it is already operating in a restricted mode.

The stated invariant ("respond InIBD to GetHeaders and GetBlocks") is violated, and the node's IBD completion time becomes attacker-influenced.

---

### Likelihood Explanation

- Any peer that can establish a TCP connection to the node can send `GetBlocks` messages; no authentication or privilege is required.
- IBD is the normal startup state for every new or restarted node.
- The attack requires only a single connection and a loop of `GetBlocks` messages; no special tooling is needed.
- There is no per-message rate limit in `GetBlocksProcess` or in the synchronizer's dispatch path.

---

### Recommendation

Add the same IBD gate to `GetBlocksProcess::execute` that already exists in `GetHeadersProcess::execute`:

```diff
 pub fn execute(self) -> Status {
     let block_hashes = self.message.block_hashes();
     if block_hashes.len() > MAX_HEADERS_LEN { ... }
+
+    let active_chain = self.synchronizer.shared.active_chain();
+    if active_chain.is_initial_block_download() {
+        // send InIBD and return, consistent with GetHeadersProcess
+        self.send_in_ibd();
+        return Status::ignored();
+    }
+
-    let active_chain = self.synchronizer.shared.active_chain();
     let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);
     ...
 }
```

A helper `send_in_ibd` analogous to the one in `GetHeadersProcess` should be added to `GetBlocksProcess`.

---

### Proof of Concept

1. Start a CKB node from genesis (or with a stale tip) so it enters IBD (`is_initial_block_download()` returns `true`).
2. Connect a custom peer that speaks the Sync protocol.
3. Send a `SyncMessage::GetBlocks` containing hashes of blocks the IBD node has already verified (obtainable from any public explorer or another node).
4. Observe that the IBD node responds with `SendBlock` messages for each requested block instead of `InIBD`.
5. Repeat in a tight loop; monitor the IBD node's disk I/O, CPU, and network utilisation to confirm resource consumption and IBD pipeline degradation.

```rust
    pub fn execute(self) -> Status {
        let block_hashes = self.message.block_hashes();
        // use MAX_HEADERS_LEN as limit, we may increase the value of INIT_BLOCKS_IN_TRANSIT_PER_PEER in the future
        if block_hashes.len() > MAX_HEADERS_LEN {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "BlockHashes count({}) > MAX_HEADERS_LEN({})",
                block_hashes.len(),
                MAX_HEADERS_LEN,
            ));
        }
        let active_chain = self.synchronizer.shared.active_chain();

        let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);

        let mut dedup = HashSet::new();
        for block_hash in iter {
            debug!("get_blocks {} from peer {:?}", block_hash, self.peer);
            let block_hash = block_hash.to_entity();

            if block_hash == self.synchronizer.shared().consensus().genesis_hash() {
                return StatusCode::RequestGenesis.with_context("Request genesis block");
            }

            if !dedup.insert(block_hash.clone()) {
                return StatusCode::RequestDuplicate.with_context("Request duplicate block");
            }

            if !active_chain.contains_block_status(&block_hash, BlockStatus::BLOCK_VALID) {
                debug!(
                    "Ignoring get_block {} request from peer={} as it is not verified.",
                    block_hash, self.peer
                );
                continue;
            }

            if let Some(block) = active_chain.get_block(&block_hash) {
                debug!(
                    "respond_block {} {} to peer {:?}",
                    block.number(),
                    block.hash(),
                    self.peer,
                );
                let content = packed::SendBlock::new_builder().block(block.data()).build();
                let message = packed::SyncMessage::new_builder().set(content).build();

                let nc = Arc::clone(self.nc);
                self.synchronizer
                    .shared()
                    .shared()
                    .async_handle()
                    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
            } else {
                // TODO response not found
                // TODO add timeout check in synchronizer

                // We expect that `block_hashes` is sorted descending by height.
                // So if we cannot find the current one from local, we cannot find
                // the next either.
                debug!("Stopping getblocks, since {} is not found", block_hash);
                break;
            }
        }

        Status::ok()
    }
```
