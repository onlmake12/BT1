### Title
Incorrect `MAX_DEPOSIT_LIMIT` Check in `addFunds()` Allows Permanent Subscription Balance to Exceed Cap - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
`Scheduler.addFunds()` enforces `MAX_DEPOSIT_LIMIT` by checking only the incoming `msg.value` against the cap, without accounting for the subscription's existing balance. Because permanent subscriptions prohibit withdrawals, this allows any caller to permanently lock an unbounded amount of ETH into a single permanent subscription, defeating the purpose of the deposit cap.

### Finding Description
In `Scheduler.sol`, `addFunds()` performs the following check for permanent subscriptions:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
``` [1](#0-0) 

The check validates only the new deposit (`msg.value`) against `MAX_DEPOSIT_LIMIT` (100 ETH), not the post-addition balance (`status.balanceInWei + msg.value`). Since `addFunds()` has no access control — any address may call it — an attacker can call it repeatedly with `msg.value = MAX_DEPOSIT_LIMIT` each time. Each call passes the guard (`100 ETH > 100 ETH` is false), and the balance grows without bound.

`MAX_DEPOSIT_LIMIT` is defined as 100 ETH: [2](#0-1) 

`createSubscription()` has the same structural check, but it is correct there because the balance starts at zero — `msg.value` alone equals the post-creation balance: [3](#0-2) 

`withdrawFunds()` is permanently blocked for permanent subscriptions: [4](#0-3) 

This means any ETH added beyond the cap is irrecoverably locked.

### Impact Explanation
`MAX_DEPOSIT_LIMIT` exists specifically to bound the amount of ETH that can be permanently locked in a single permanent subscription. By bypassing it, an attacker (or even the subscription manager) can lock an arbitrary amount of ETH — far exceeding 100 ETH — into a permanent subscription from which no withdrawal is ever possible. This results in permanent, irrecoverable loss of ETH for whoever sends the funds, and breaks the protocol's intended economic invariant for permanent subscriptions.

### Likelihood Explanation
The entry point is fully unpermissioned: `addFunds(uint256 subscriptionId)` has no `onlyManager` or similar modifier. Any externally-owned account can call it for any active permanent subscription. The attack requires only ETH and knowledge of a permanent subscription ID (which is publicly observable on-chain). No privileged access, leaked keys, or governance majority is needed.

### Recommendation
Replace the check in `addFunds()` to validate the **post-addition cumulative balance** against the cap:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [5](#0-4) 

### Proof of Concept
1. Deploy `SchedulerUpgradeable` (as in the test suite).
2. Create a permanent subscription with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). This succeeds; `status.balanceInWei = 100 ETH`.
3. Call `addFunds(subscriptionId)` with `msg.value = MAX_DEPOSIT_LIMIT` (100 ETH). The check `100 ETH > 100 ETH` is **false** → no revert. `status.balanceInWei` becomes **200 ETH**.
4. Repeat step 3 arbitrarily. Each call passes the guard and adds another 100 ETH permanently.
5. Since `withdrawFunds` reverts with `CannotUpdatePermanentSubscription` for permanent subscriptions, all deposited ETH above the intended cap is permanently locked and unrecoverable. [6](#0-5)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();

                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
