### Title
`setFeeManager` Direct Overwrite Enables Front-Running Fee Drain by Outgoing Fee Manager — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `setFeeManager` directly overwrites the `feeManager` address with no atomicity protection. An outgoing `feeManager` who monitors the mempool can front-run the provider's `setFeeManager` call and drain all of the provider's accrued fees via `withdrawAsFeeManager` before the replacement takes effect — an exact structural analog to the ERC-20 `approve` front-run.

---

### Finding Description

`setFeeManager` performs a direct, single-step overwrite of `provider.feeManager`:

```solidity
// Entropy.sol lines 876–893
function setFeeManager(address manager) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();

    address oldFeeManager = provider.feeManager;
    provider.feeManager = manager;          // ← direct overwrite, no atomicity
    ...
}
``` [1](#0-0) 

The `IEntropy.sol` interface comment even acknowledges the pattern:

> "Only one address can be the fee manager for a provider at a time — calling this function again with a new value will override the previous value." [2](#0-1) 

The `feeManager` role grants the ability to drain the provider's entire `accruedFeesInWei` balance via `withdrawAsFeeManager`:

```solidity
// Entropy.sol lines 175–209
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    ...
    if (providerInfo.feeManager != msg.sender) revert EntropyErrors.Unauthorized();
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    ...
}
``` [3](#0-2) 

---

### Impact Explanation

A provider who calls `setFeeManager(newManager)` to rotate or revoke a fee manager loses all accrued ETH fees if the outgoing `feeManager` is malicious. The old manager front-runs the rotation transaction and calls `withdrawAsFeeManager(provider, accruedFeesInWei)`, draining the full balance before the overwrite lands. The new manager inherits an empty balance. This is a direct, irreversible loss of ETH for the provider.

---

### Likelihood Explanation

EVM chains with public mempools (Ethereum mainnet, L2s) make mempool monitoring trivial. A malicious or compromised `feeManager` has a clear financial incentive to execute this attack at the moment of rotation. The provider has no on-chain mechanism to atomically revoke and drain simultaneously. Likelihood is **medium**: it requires the outgoing feeManager to be adversarial, but that is precisely the scenario that motivates a rotation in the first place.

---

### Recommendation

Adopt the same pattern recommended for ERC-20 allowance changes:

1. **Two-step rotation**: Require the new manager to accept the role before the old one is removed (analogous to `increaseAllowance`/`decreaseAllowance`).
2. **Atomic revoke-and-drain**: Allow the provider to atomically call `setFeeManager(address(0))` and `withdraw(balance)` in a single transaction, or add a `revokeAndWithdraw` helper.
3. **Pending-manager pattern**: Store a `pendingFeeManager` and require the new address to call `acceptFeeManager()` before the old one loses access.

---

### Proof of Concept

1. Provider `P` has `accruedFeesInWei = 10 ETH` and `feeManager = Alice`.
2. `P` submits `setFeeManager(Bob)` to replace Alice with Bob.
3. Alice monitors the mempool, sees the pending transaction.
4. Alice front-runs with `withdrawAsFeeManager(P, 10 ETH)` at a higher gas price.
   - Check: `providerInfo.feeManager == Alice` ✓ (not yet overwritten)
   - `providerInfo.accruedFeesInWei` decremented to 0; 10 ETH sent to Alice.
5. `P`'s `setFeeManager(Bob)` lands; Bob is now feeManager but `accruedFeesInWei == 0`.
6. Provider `P` has lost 10 ETH; Alice has stolen it. [4](#0-3) [5](#0-4)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
