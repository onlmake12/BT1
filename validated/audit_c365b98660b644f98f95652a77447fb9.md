### Title
Fee Oracle Manipulation via Temporary Pool State Inflation — (File: `util/fee-estimator/src/estimator/weight_units_flow.rs`, `tx-pool/src/component/pool_map.rs`)

### Summary
Both CKB fee estimation algorithms (`WeightUnitsFlow` and the fallback `pool_map::estimate_fee_rate`) read the **live tx-pool state** directly when computing the fee estimate. An unprivileged tx-pool submitter can temporarily flood the pool with large, high-fee transactions to inflate the `estimate_fee_rate` RPC response, causing other users to overpay. The attacker then uses RBF to replace their transactions at minimum cost. In the `WeightUnitsFlow` algorithm, the historical `flow_speed` component is also persistently inflated because `reject_tx` is a no-op for that algorithm, meaning the manipulation outlasts the attacker's pool presence by up to `2 × target_blocks` blocks.

### Finding Description

**Two vulnerable code paths:**

**Path 1 — `weight_units_flow::Algorithm::estimate_fee_rate`** [1](#0-0) 

This function takes `all_entry_info: TxPoolEntryInfo` — a live snapshot of every pending and proposed transaction — and builds `current_weight_buckets` from it. The decision formula is: [2](#0-1) 

```
final_weight = current_weight + added_weight - removed_weight
passed       = final_weight <= 0
```

`current_weight` is read directly from the live pool. Inflating it by submitting many large, high-fee transactions causes more buckets to fail the `passed` check, pushing the returned fee rate upward.

Additionally, every transaction submitted to the pool triggers `accept_tx`, which appends to the historical `self.txs` map used to compute `flow_speed`: [3](#0-2) 

Critically, `reject_tx` is a **no-op** for `WeightUnitsFlow`: [4](#0-3) 

When the attacker replaces their transactions via RBF, the replacement transactions also call `accept_tx`, but the evicted originals are never removed from `self.txs`. The historical `flow_speed` therefore remains inflated for `historical_blocks = 2 × target_blocks` blocks after the attacker has exited the pool.

**Path 2 — Fallback `pool_map::estimate_fee_rate`** [5](#0-4) 

This fallback (used when the primary estimator lacks data) iterates entries sorted by score from highest to lowest, simulating block-filling. Injecting many large, high-fee transactions causes the simulated blocks to fill quickly with the attacker's entries, and the returned fee rate is the fee rate of the last entry that fills the target block — i.e., the attacker's high fee rate.

**How the estimate reaches users:** [6](#0-5) 

The `estimate_fee_rate` async handler reads the live pool state (`get_all_entry_info`) and passes it directly to the estimator. There is no smoothing, outlier filtering, or snapshot isolation.

**RBF exit cost:** [7](#0-6) 

The minimum cost to replace a transaction is `sum(replaced_fees) + min_rbf_rate × size`. For a single attacker transaction, this is a small incremental cost, making the attack economically feasible.

**Attack steps:**
1. Attacker submits N large transactions with fee rate >> market rate (e.g., 1,000,000 shannons/KB).
2. `accept_tx` is called for each → `current_weight` in high-fee buckets inflates; historical `flow_speed` inflates.
3. Victim calls `estimate_fee_rate` → receives inflated estimate.
4. Victim submits their transaction with the inflated fee.
5. Attacker uses RBF to replace their N transactions with minimum-fee transactions (cost: `N × min_rbf_rate × size`).
6. `current_weight` drops immediately, but historical `flow_speed` remains inflated for `2 × target_blocks` more blocks, keeping the estimate elevated for subsequent victims.

### Impact Explanation
Users who rely on the `estimate_fee_rate` RPC (exposed in the default `Experiment` module) to set their transaction fees will systematically overpay. The excess fees flow to miners. The attacker's direct cost is only the RBF fee increment. For the `WeightUnitsFlow` algorithm, the inflation of `flow_speed` persists for up to `2 × target_blocks` blocks (e.g., 60 blocks for the 30-block target), meaning a single attack round affects many subsequent fee estimate queries without the attacker maintaining any pool presence.

### Likelihood Explanation
The `Experiment` RPC module is enabled by default on mainnet nodes: [8](#0-7) 

RBF is enabled by default (`min_rbf_rate = 1500 > min_fee_rate = 1000`): [9](#0-8) 

Any unprivileged RPC caller with enough CKB to cover the RBF fee increment can execute this attack. No special access, leaked keys, or majority hashpower is required. Wallets and dApps that call `estimate_fee_rate` to auto-set fees are the natural victims.

### Recommendation
1. **Decouple current pool state from the fee estimate.** Use a time-weighted or block-averaged snapshot of pool weight rather than the instantaneous live state.
2. **Honor `reject_tx` in `WeightUnitsFlow`.** When a transaction is evicted (including via RBF), remove it from `self.txs` so the historical flow speed is not permanently inflated by ephemeral attacker transactions.
3. **Apply outlier filtering.** Transactions with fee rates significantly above the moving average should be down-weighted or excluded from the `current_weight` and `flow_speed` calculations.
4. **Rate-limit or authenticate the `estimate_fee_rate` RPC** if it is intended only for trusted callers.

### Proof of Concept

```
Initial state: pool is empty, market fee rate ≈ 1,000 shannons/KB

Step 1: Attacker submits 100 transactions, each 500 KB, fee rate = 1,000,000 shannons/KB
        → pool current_weight in bucket[115] (≈1,000,000 shannons/KB) = 100 × 500,000 bytes
        → accept_tx called 100 times → flow_speed[115] inflated

Step 2: Victim calls estimate_fee_rate(NoPriority)
        → weight_units_flow sees current_weight[115] >> removed_weight for all lower buckets
        → returns FeeRate ≈ 1,000,000 shannons/KB  (1000× the real market rate)

Step 3: Victim submits transaction with fee = 1,000,000 shannons/KB × tx_size

Step 4: Attacker replaces all 100 transactions via RBF
        → cost: 100 × (original_fee + min_rbf_rate × 500,000) ≈ small incremental cost
        → current_weight drops to 0
        → BUT self.txs still contains 200 entries (100 original + 100 replacements)
        → flow_speed remains inflated for next ~60 blocks

Step 5: Any subsequent victim calling estimate_fee_rate during those 60 blocks
        still receives an inflated estimate, with no attacker transactions in the pool.
``` [10](#0-9) [11](#0-10)
