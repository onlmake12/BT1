### Title
Unbounded `numHashes` Loop in `constructProviderCommitment` Enables Gas-Based DoS on Entropy Provider Reveals — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any unprivileged user can call `requestV2` / `requestWithCallback` to create entropy requests. Each request stores `numHashes = assignedSequenceNumber − currentCommitmentSequenceNumber` at creation time. The `constructProviderCommitment` function loops exactly `numHashes` times on every reveal. When a provider's `maxNumHashes` is `0` (the default when not explicitly configured), there is no on-chain cap on this value. An attacker who creates a large burst of requests can force a specific request's stored `numHashes` to exceed the block gas limit, permanently preventing that request from being revealed and locking the requester's fee in the contract.

---

### Finding Description

**Root cause — three cooperating code paths:**

**1. `numHashes` is computed and frozen at request creation time** [1](#0-0) 

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

When `maxNumHashes == 0` the guard is entirely skipped. The value stored in `req.numHashes` is immutable after this point — no subsequent call to `advanceProviderCommitment` can reduce it.

**2. `constructProviderCommitment` loops `numHashes` times on every reveal** [2](#0-1) 

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

This is called unconditionally inside `revealHelper`, which is called by both `reveal` and `revealWithCallback`. [3](#0-2) 

**3. The default Fortuna configuration leaves `maxNumHashes` at 0** [4](#0-3) 

```rust
sync_max_num_hashes(
    &contract,
    &provider_info,
    chain_config.max_num_hashes.unwrap_or(0),  // defaults to 0
)
```

Any provider that does not explicitly set `max_num_hashes` in its chain config — or any permissionlessly registered third-party provider — will have `maxNumHashes == 0` on-chain.

**Attack flow:**

1. Provider is registered with `maxNumHashes == 0` and `currentCommitmentSequenceNumber == C`.
2. Attacker calls `requestV2` (paying the required fee) N times in rapid succession.
3. The Nth request stores `numHashes = (C + N) − C = N`.
4. The off-chain Fortuna keeper's `update_commitments_loop` runs every 30 seconds; if the burst is large enough it cannot advance the commitment before the requests are created.
5. When the provider (or anyone) calls `revealWithCallback` for the Nth request, `constructProviderCommitment` must execute N keccak256 iterations. At ~30 gas each, N ≈ 1,000,000 iterations consumes ~30 M gas — at or above the block gas limit on most EVM chains.
6. The transaction reverts with out-of-gas. The request is permanently unrevealable; the requester's fee is locked in the contract forever.

The `advanceProviderCommitment` function cannot retroactively reduce `numHashes` for already-created requests because `numHashes` is stored in the request struct at creation time. [5](#0-4) 

---

### Impact Explanation

- **Permanent fund lock**: Any request whose stored `numHashes` exceeds `blockGasLimit / 30` can never be revealed. The requester's fee (paid in native token) is permanently trapped in the contract with no recovery path.
- **Provider service disruption**: If the attacker targets the default provider, legitimate users' requests that happen to land in the same sequence range are also affected.
- **Quadratic gas cost**: Even below the block-gas-limit threshold, the total gas cost to reveal N sequentially created requests is O(1 + 2 + … + N) = O(N²), making the provider's keeper increasingly expensive to operate as outstanding requests accumulate.

---

### Likelihood Explanation

- **Permissionless entry**: `requestV2` / `requestWithCallback` require no special role — any EOA or contract can call them.
- **Fee-gated but asymmetric**: The attacker pays a linear cost (N × fee) while the provider bears a quadratic gas cost (O(N²)) and the victim's request is permanently bricked.
- **Low-fee chains are directly in scope**: The Pyth deployment scripts target multiple chains (`script/input/{blockchainId}/*.json`). On chains such as Polygon, BNB Chain, or Arbitrum where fees are fractions of a cent, creating 1,000,000 requests costs tens to hundreds of dollars — well within reach of a motivated attacker.
- **Default configuration is vulnerable**: The `unwrap_or(0)` default means any provider that does not explicitly configure `max_num_hashes` is exposed without any additional attacker effort.

---

### Recommendation

1. **Enforce a non-zero `maxNumHashes` at the protocol level**: Add a global minimum (e.g., `require(providerInfo.maxNumHashes > 0)`) in `requestHelper`, or set a protocol-wide hard cap (e.g., `MAX_NUM_HASHES = 10_000`) that applies even when the provider's value is 0.

2. **Change the Fortuna default**: Replace `chain_config.max_num_hashes.unwrap_or(0)` with a safe non-zero default (e.g., `unwrap_or(10_000)`) so that providers that omit the field are protected automatically.

3. **Bound the loop in `constructProviderCommitment`**: Add an explicit revert if `numHashes` exceeds a safe constant before entering the loop, providing a last-resort safety net.

---

### Proof of Concept

```solidity
// Assume provider is registered with maxNumHashes == 0
// and currentCommitmentSequenceNumber == 0.

uint256 N = 1_100_000; // ~33 M gas at 30 gas/keccak256 > block gas limit

for (uint256 i = 0; i < N; i++) {
    // Attacker pays the required fee each time.
    entropy.requestV2{value: fee}(provider, 0);
}

// Request with sequenceNumber == N has numHashes == N stored immutably.
// Any attempt to reveal it will loop 1,100,000 times and revert OOG.
// The requester's fee for that request is permanently locked.
entropy.revealWithCallback(provider, N, userContrib, providerContrib);
// ↑ reverts: out of gas
```

The `constructProviderCommitment` loop at lines 991–995 of `Entropy.sol` is the necessary vulnerable step; no privileged access is required beyond paying the per-request fee.
