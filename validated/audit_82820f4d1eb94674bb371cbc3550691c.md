Audit Report

## Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Corrupts Ancestor `EvictKey` — (`tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::remove_entry_and_descendants` tears down all link records for every entry in the removal set before calling `remove_entry` on each one. Because `update_ancestors_index_key` resolves ancestors through those same link records, it finds an empty ancestor set and never calls `sub_descendant_weight` on ancestor entries that remain in the pool. Those ancestors permanently retain inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` fields, corrupting the `EvictKey` used to decide which transactions to evict when the pool is full.

## Finding Description

`remove_entry_and_descendants` (lines 252–265) first tears down all link records for every entry in the removal set, then iterates and calls `remove_entry` on each:

```rust
// tx-pool/src/component/pool_map.rs lines 252-265
for id in &removed_ids {
    self.remove_entry_links(id);   // ALL links torn down here
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

The inline comment reads: *"update links state for remove, so that we won't update_descendants_index_key in remove_entry"* — confirming the intent was to skip redundant descendant-score updates. However, this also silences the necessary ancestor-score update. [1](#0-0) 

`remove_entry_links` (lines 418–430) removes the entry from `self.links.inner` entirely via `self.links.remove(id)`: [2](#0-1) 

Inside `remove_entry` (lines 235–250), `update_ancestors_index_key` is called first: [3](#0-2) 

`update_ancestors_index_key` (lines 432–445) resolves ancestors via `self.links.calc_ancestors`: [4](#0-3) 

`calc_ancestors` calls `calc_relative_ids` (links.rs lines 37–50), which does `self.inner.get(short_id).map(...).unwrap_or_default()`. Since the entry was already removed from `self.links.inner`, this returns an empty `HashSet`. The loop over ancestors never executes, so `sub_descendant_weight` is never called on any ancestor that remains in the pool. [5](#0-4) [6](#0-5) 

The existing test `test_remove_entry_and_descendants` (lines 170–230 of `score_key.rs`) only asserts that the removed entries are gone and that the ancestor's descendants set is empty. It does not assert the ancestor's `descendants_fee`, `descendants_size`, `descendants_cycles`, or `descendants_count` fields, leaving the accounting bug undetected. [7](#0-6) 

By contrast, `test_remove_entry` (lines 157–167) calls `remove_entry` directly (links still intact) and correctly verifies that `ancestors_count` is decremented — confirming the update path works when links are present. [8](#0-7) 

`remove_entry_and_descendants` is called from three reachable production paths: `resolve_conflict` (lines 310, 321), `resolve_conflict_header_dep` (line 285), and `check_and_record_ancestors` (line 618) — all triggerable by an unprivileged transaction submitter or block relayer. [9](#0-8) [10](#0-9) 

## Impact Explanation

`EvictKey` is computed directly from the stale fields (entry.rs lines 234–247):

```rust
let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
let feerate = FeeRate::calculate(entry.fee, weight);
EvictKey { fee_rate: descendants_feerate.max(feerate), ... }
``` [11](#0-10) 

An ancestor whose removed descendants had a higher fee rate than itself retains an inflated `descendants_feerate`, making its `EvictKey.fee_rate` appear higher than it truly is. `next_evict_entry` (lines 380–385) selects the lowest-`EvictKey` entry to drop when the pool is full. [12](#0-11) 

The ancestor with the stale inflated key is ranked as more eviction-resistant than it deserves, so other legitimate transactions are evicted in its place. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The trigger requires no special privileges. An attacker submits a low-fee root transaction X, then high-fee descendants A → B spending X's output, then a conflicting transaction spending the same input as A. `resolve_conflict` calls `remove_entry_and_descendants(A)`, leaving X with permanently inflated descendant stats. The only cost is the fee for the conflicting transaction. The attack is repeatable: each round inflates one more ancestor's `EvictKey`, progressively degrading eviction ordering across the pool.

## Recommendation

Collect and apply ancestor `sub_descendant_weight` updates **before** tearing down links, skipping entries that are themselves in the removal set:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();
    for removed_id in &removed_ids {
        if let Some(entry) = self.get(removed_id).cloned() {
            let ancestors = self.links.calc_ancestors(removed_id);
            for anc_id in ancestors.difference(&removed_set) {
                self.entries.modify_by_id(anc_id, |e| {
                    e.inner.sub_descendant_weight(&entry);
                    e.evict_key = e.inner.as_evict_key();
                });
            }
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

Add a regression test asserting that after `remove_entry_and_descendants` on a child chain, the surviving ancestor's `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` equal the ancestor's own values only.

## Proof of Concept

**Setup**: pool contains chain `X → A → B`.

| Tx | fee | size | cycles |
|---|---|---|---|
| X | 100 | 100 | 100 |
| A | 300 | 200 | 200 |
| B | 200 | 200 | 200 |

After insertion, X's tracked state: `descendants_fee=600, descendants_size=500, descendants_cycles=500, descendants_count=3`.

**Trigger**: submit tx A′ spending the same input as A. `resolve_conflict` calls `remove_entry_and_descendants(A)`.

**Expected state of X**: `descendants_fee=100, descendants_size=100, descendants_cycles=100, descendants_count=1`.

**Actual state of X (bug)**: `descendants_fee=600, descendants_size=500, descendants_cycles=500, descendants_count=3`.

X's `EvictKey.fee_rate` is computed from the inflated `descendants_feerate = 600/500 = 1.2 shannons/KW` instead of the correct `100/100 = 1.0 shannons/KW`. Any transaction with a true fee rate between 1.0 and 1.2 is evicted before X, even though X is less valuable. This can be confirmed by extending `test_remove_entry_and_descendants` to assert `tx1_entry.descendants_count == 1` after the call — the assertion will fail against the current code. [13](#0-12)
