### Title
`DownloadScheduler::decrease` Bypasses Its Own Accumulation Threshold, Over-Aggressively Disconnecting Sync Peers — (`File: sync/src/types/mod.rs`)

---

### Summary

`DownloadScheduler::decrease` contains a wrong-variable assignment that makes its internal accumulation counter (`timeout_count`) always exceed the guard threshold of `2`, causing `task_count` to be decremented on **every** slow block-download response instead of only after accumulating 2–3 slow responses. When `task_count` reaches zero the peer is disconnected. The intended soft-penalty buffer is completely bypassed, making the scheduler 2–3× more aggressive than designed.

---

### Finding Description

In `sync/src/types/mod.rs`, `DownloadScheduler` tracks how many concurrent block-download tasks a peer may have in flight (`task_count`). When a completed download falls in the slow time quantiles (`NormalToUpper` or `UpperToMax`), `remove_by_block` calls `decrease(1)` or `decrease(2)` respectively.

The `decrease` function is:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);   // ← BUG
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [1](#0-0) 

The right-hand side of the assignment reads `self.task_count` instead of `self.timeout_count`. Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` and `num` is 1 or 2, the expression `self.task_count.saturating_add(num)` always evaluates to ≥ 3, so the guard `if self.timeout_count > 2` is **always true**. `task_count` is decremented and `timeout_count` is reset to 0 on every single call — the accumulation buffer never actually accumulates anything.

The intended logic (accumulate slow responses, only penalize after the count exceeds 2) is completely inoperative. The correct line should be:

```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

`decrease` is called from `remove_by_block` whenever `adjustment` is enabled (default: `true`) and `should_punish` is true (more peers than `protect_num = 4`): [2](#0-1) 

When `task_count` reaches zero, `prune` adds the peer to the disconnect list:

```rust
download_schedulers.retain(|k, v| {
    if v.task_count == 0 {
        disconnect_list.insert(*k);
        false
    } else {
        true
    }
});
``` [3](#0-2) 

---

### Impact Explanation

Starting from `task_count = 32`, a peer is disconnected after exactly **32 slow responses** (one decrement per slow block). With the correct accumulation logic, the same peer would need **~64–96 slow responses** before disconnection (one decrement per 2–3 slow blocks). The scheduler is 2–3× more aggressive than designed.

Legitimate peers experiencing transient network slowness — whose response times fall in the `NormalToUpper` band — are penalized as harshly as peers in the `UpperToMax` band. This causes unnecessary disconnections from honest, reachable peers, degrading the local node's sync connectivity. In a network segment with elevated latency (e.g., cross-region peers), a node can shed all non-protected outbound peers far faster than intended, narrowing its view of the chain and increasing susceptibility to eclipse-style isolation. [4](#0-3) 

---

### Likelihood Explanation

The bug is active in the default configuration (`adjustment = true`, `protect_num = 4`). Any sync peer whose block-delivery latency lands in the `NormalToUpper` or `UpperToMax` quantile — a normal occurrence under real-world network jitter — triggers the broken path. No special privileges, crafted messages, or malicious intent are required. The `TimeAnalyzer` thresholds are dynamically computed from observed samples, so even a modest increase in network latency can push many peers into the penalty zone simultaneously. [5](#0-4) 

---

### Recommendation

Fix the assignment in `decrease` to accumulate into `self.timeout_count` rather than overwrite it with `self.task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was: self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

This restores the intended soft-penalty buffer: `task_count` is only decremented after 2–3 slow responses have accumulated, matching the documented design intent ("slowly eliminated").

---

### Proof of Concept

**Step 1 — Setup:** Connect a sync peer to a CKB node with more than 4 outbound peers (`download_schedulers.len() > protect_num`) and `adjustment = true` (default).

**Step 2 — Trigger:** The peer delivers blocks whose round-trip time consistently falls in the `NormalToUpper` quantile (between `normal_time` ≈ 1250 ms and `low_time` ≈ 1500 ms by default). Each completed download calls `remove_by_block`, which calls `decrease(1)`.

**Step 3 — Observe the bug:** Each call to `decrease(1)` executes:
```
timeout_count = task_count + 1   // e.g., 32 + 1 = 33  (always > 2)
task_count    = task_count - 1   // 32 → 31 → 30 → … → 0
timeout_count = 0
```

**Step 4 — Disconnection:** After 32 such responses, `task_count` reaches 0. The next `prune` call inserts the peer into `disconnect_list` and disconnects it — despite the peer never timing out and never being in the harshest penalty band.

**Expected (correct) behavior:** With `self.timeout_count.saturating_add(num)`, `task_count` would only decrement after every 3rd slow response, requiring ~96 slow responses before disconnection. [6](#0-5) [7](#0-6)

    pub fn prune(&mut self, tip: BlockNumber) -> HashSet<PeerIndex> {
        let now = unix_time_as_millis();
        let mut disconnect_list = HashSet::new();
        // Since statistics are currently disturbed by the processing block time, when the number
        // of transactions increases, the node will be accidentally evicted.
        //
        // Especially on machines with poor CPU performance, the node connection will be frequently
        // disconnected due to statistics.
        //
        // In order to protect the decentralization of the network and ensure the survival of low-performance
        // nodes, the penalty mechanism will be closed when the number of download nodes is less than the number of protected nodes
        let should_punish = self.download_schedulers.len() > self.protect_num;
        let adjustment = self.adjustment;

        let trace = &mut self.trace_number;
        let download_schedulers = &mut self.download_schedulers;
        let states = &mut self.inflight_states;

        let mut remove_key = Vec::new();
        // Since this is a btreemap, with the data already sorted,
        // we don't have to worry about missing points, and we don't need to
        // iterate through all the data each time, just check within tip + 20,
        // with the checkpoint marking possible blocking points, it's enough
        let end = tip + 20;
        for (key, value) in states.iter() {
            if key.number > end {
                break;
            }
            if value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now {
                if let Some(set) = download_schedulers.get_mut(&value.peer) {
                    set.hashes.remove(key);
                    if should_punish && adjustment {
                        set.punish(2);
                    }
                };
                if !trace.is_empty() {
                    trace.remove(key);
                }
                remove_key.push(key.clone());
                debug!(
                    "prune: remove InflightState: remove {}-{} from {}",
                    key.number, key.hash, value.peer
                );

                if let Some(metrics) = ckb_metrics::handle() {
                    metrics.ckb_inflight_timeout_count.inc();
                }
            }
        }

        for key in remove_key {
            states.remove(&key);
        }

        download_schedulers.retain(|k, v| {
            // task number zero means this peer's response is very slow
            if v.task_count == 0 {
                disconnect_list.insert(*k);
                false
            } else {
                true
            }
        });
