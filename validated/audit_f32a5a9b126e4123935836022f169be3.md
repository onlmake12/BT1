Audit Report

## Title
Inbound session from feeler-listed peer bypasses ban checks, inbound limit, and peer registry — (`network/src/network.rs`, `network/src/peer_registry.rs`)

## Summary
`is_feeler` matches on peer ID alone with no direction check. When a victim dials peer X as a feeler, any inbound session from X during the 300-second dial window causes `handle_event`'s `SessionOpen` branch to take the feeler fast-path, skipping `accept_peer` entirely. The session is live at the transport layer but absent from `PeerRegistry.peers`, bypassing ban enforcement, inbound connection limits, and all monitoring and eviction logic.

## Finding Description
`dial_feeler` (network.rs:498–510) issues the outbound dial and on success calls `reg.add_feeler(&addr)`, inserting the peer ID into `feeler_peers`. The race window is bounded by `DIAL_HANG_TIMEOUT = 300s` (network.rs:71).

`is_feeler` (peer_registry.rs:244–248) checks only `feeler_peers.contains_key(&peer_id)`. It does not inspect `session_context.ty` to confirm the session is outbound. A grep for `session_context.ty` in network.rs returns exactly one match, and it is not in the `SessionOpen` handler.

In `handle_event`'s `SessionOpen` branch (network.rs:746–793), when `is_feeler` returns `true`, the code logs `"Feeler connected"` and falls through — no `disconnect_with_message`, no `accept_peer`. The `else` branch containing `accept_peer` is never reached. Consequently:

- `peer_store.is_addr_banned` (peer_registry.rs:109) is never checked.
- `connection_status.non_whitelist_inbound >= self.max_inbound` (peer_registry.rs:116) is never checked.
- `peer_store.add_connected_peer` (peer_registry.rs:135) is never called.
- `self.peers.insert(session_id, peer)` (peer_registry.rs:137) is never called.

`connection_status` (peer_registry.rs:294–316) derives all counts from `self.peers`, so the hidden session is invisible to limit enforcement, `connected_peers()`, and eviction logic.

## Impact Explanation
**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

1. **Ban bypass**: A banned peer whose address is in `feeler_peers` can maintain an active inbound session. `peer_store.is_addr_banned` is only called inside `accept_peer`, which is skipped.
2. **Inbound limit bypass**: `connection_status().non_whitelist_inbound` is derived from `self.peers`. The hidden session is not counted, so `max_inbound` is never enforced for it. An attacker can hold multiple such sessions simultaneously (one per feeler dial window), consuming real inbound capacity while honest peers are rejected.
3. **Invisible session**: Protocol handlers dispatch on `session_id` independently of `PeerRegistry.peers`, so the hidden session can exchange protocol messages while remaining undetectable to monitoring and eviction.

The combination of ban bypass and inbound limit bypass with an active protocol-level session constitutes a concrete, exploitable path toward network-level disruption and eclipse attack prerequisites.

## Likelihood Explanation
No privileged access is required. The attacker only needs to be present in the victim's peer store, which is achievable via the discovery/gossip protocol. `OutboundPeerService`/`Feeler` periodically selects addresses from the peer store and calls `dial_feeler`. Once dialed, the attacker has up to 300 seconds to open the inbound connection. The attack is repeatable: each new feeler dial to the attacker reopens the window. No hashpower, key material, or operator access is needed.

## Recommendation
In `handle_event`'s `SessionOpen` branch, gate the feeler fast-path on the session being outbound:

```rust
if session_context.ty == RawSessionType::Outbound
    && self.network_state.with_peer_registry(|reg| reg.is_feeler(&session_context.address))
{
    // feeler outbound path — log only
} else {
    match self.network_state.accept_peer(&session_context) { ... }
}
```

Alternatively, modify `is_feeler` to accept the session type as a parameter and require `Outbound`, or key `feeler_peers` on `(PeerId, SessionId)` so only the specific outbound session matches.

## Proof of Concept
State-level test (no network required):

```rust
let mut registry = PeerRegistry::new(2, 8, false, vec![], false);
let attacker_addr: Multiaddr = "/ip4/1.2.3.4/tcp/8115/p2p/QmAttacker".parse().unwrap();

// Victim dials attacker as feeler — adds to feeler_peers
registry.add_feeler(&attacker_addr);
assert!(registry.is_feeler(&attacker_addr)); // true — peer ID match only, no direction check

// Attacker opens inbound session during the window.
// handle_event takes the feeler branch: accept_peer is NOT called.
// Simulate: do NOT call registry.accept_peer(...)

// Assert: session absent from registry
let session_id = SessionId::new(42);
assert!(registry.get_peer(session_id).is_none());

// Assert: connection_status does not count the hidden session
let status = registry.connection_status();
assert_eq!(status.non_whitelist_inbound, 0);
assert_eq!(status.total, 0);
// Session is live at transport layer; registry is unaware.
```
