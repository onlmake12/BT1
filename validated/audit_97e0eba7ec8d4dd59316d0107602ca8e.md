### Title
Missing Caller Authorization Check in `set_publisher_stake_account` Allows Any Signer to Redirect a Publisher's Stake Account — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `set_publisher_stake_account` instruction in the Integrity Pool program accepts an arbitrary `signer` and does not constrain that signer to be the publisher or any authorized governance account. The `publisher` account is only verified for existence in `pool_data`, not for ownership by the caller. Any unprivileged user can invoke this instruction for any registered publisher, redirecting the publisher's stake account to an account they control and capturing the publisher's staking rewards.

---

### Finding Description

The `set_publisher_stake_account` instruction's account layout, as reflected in the deployed program's IDL, is:

```
signer          – signer: true  (unconstrained; any keypair)
publisher       – CHECK: "The publisher will be checked against data in the pool_data"
pool_data       – writable
pool_config     – PDA
newStakeAccountPositionsOption    – optional
currentStakeAccountPositionsOption – optional
```

The `publisher` account is **not** marked as a signer and carries only an existence check against `pool_data`. There is no constraint of the form `signer == publisher` or `signer == pool_config.authority`. This is structurally identical to the M-13 pattern: a privileged state-mutation function (add/remove strategy on a vault) that is missing the check tying the caller to the resource being modified.

In contrast, the `create_slash_event` instruction in the same program uses an `owner` account that is constrained against `pool_config`, demonstrating that the program does enforce caller identity in other instructions. The absence of any such constraint on `set_publisher_stake_account`'s `signer` is the root cause.

The `xc_admin_cli` command `set-publisher-stake-account` passes the governance vault PDA as the signer, confirming the intended design is governance-gated. However, the on-chain program does not enforce this restriction.

---

### Impact Explanation

An attacker who calls `set_publisher_stake_account` with:
- `signer` = attacker's keypair
- `publisher` = any registered publisher key
- `newStakeAccountPositionsOption` = attacker-controlled stake account

…will overwrite the publisher's registered stake account in `pool_data`. Subsequent reward distributions (advance delegation record, slash accounting) will credit the attacker's stake account instead of the legitimate publisher's. This constitutes unauthorized redirection of staking rewards — a direct financial loss to the publisher and integrity of the pool's reward accounting.

---

### Likelihood Explanation

The instruction is permissionless at the program level: any funded Solana account can sign and submit it. The publisher key is publicly observable on-chain from `pool_data`. No privileged access, leaked keys, or governance majority is required. The attacker only needs to know a registered publisher's public key, which is public information.

---

### Recommendation

Add an explicit authorization constraint requiring the signer to be either the publisher itself or the pool authority:

```rust
require!(
    ctx.accounts.signer.key() == ctx.accounts.publisher.key()
        || ctx.accounts.signer.key() == ctx.accounts.pool_config.authority,
    IntegrityPoolError::Unauthorized
);
```

This mirrors the fix applied in the M-13 yAxis case: adding the missing `allowedVaults` check to the add/remove strategy functions.

---

### Proof of Concept

1. Attacker observes `pool_data` on-chain and identifies a registered publisher key `P`.
2. Attacker creates a stake account `A` under their own control.
3. Attacker submits `set_publisher_stake_account` with:
   - `signer` = attacker wallet
   - `publisher` = `P`
   - `newStakeAccountPositionsOption` = `A`
4. `pool_data` now records `A` as publisher `P`'s stake account.
5. On the next `advance_delegation_record` call for publisher `P`, rewards are credited to `A` (attacker-controlled) instead of the legitimate publisher's account.
6. Publisher `P` loses their staking rewards with no recourse.

    publisher: PublicKey,
    stakeAccountPositions: PublicKey,
    newStakeAccountPositions: PublicKey | undefined,
  ) {
    const instruction = await this.integrityPoolProgram.methods
      .setPublisherStakeAccount()
      .accounts({
        currentStakeAccountPositionsOption: stakeAccountPositions,
        // eslint-disable-next-line unicorn/no-null
        newStakeAccountPositionsOption: newStakeAccountPositions ?? null,
        publisher,
      })
      .instruction();

    await sendTransaction([instruction], this.connection, this.wallet);
    return;
  }
