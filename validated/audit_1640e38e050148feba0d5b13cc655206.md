Audit Report

## Title
Off-by-one in `check_purge()` eviction condition allows remote peer to permanently freeze peer discovery — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge()` in `peer_store_impl.rs` uses `addrs.len() > 4` to decide whether to evict peers from a network group, but this condition is never satisfied when every group contains exactly 4 addresses. A single malicious peer can advertise 4096 × 4 fake addresses across distinct `/16` subnets, filling `ADDR_COUNT_LIMIT=16384` and causing every subsequent `add_addr()` call to silently return `Err(EvictionFailed)`, permanently blocking peer discovery until node restart.

## Finding Description
`check_purge()` runs two eviction passes when `addr_manager.count() >= ADDR_COUNT_LIMIT (16384)`:

**Pass 1** collects addresses where `!is_connectable(now_ms)`. Newly inserted addresses have `last_connected_at_ms=0` and `attempts_count=0`. The two `false`-return branches in `is_connectable()` require either `attempts_count >= ADDR_MAX_RETRIES (3)` or `attempts_count >= ADDR_MAX_FAILURES (10)`, neither of which holds for fresh entries. Pass 1 yields zero candidates. [1](#0-0) [2](#0-1) 

**Pass 2** is entered only when Pass 1 is empty. It groups all addresses by `/16` IPv4 prefix (`Group::IP4([bits[0], bits[1]])`), sorts groups by descending size, takes the top `len/2` groups, and for each group applies:
```rust
if addrs.len() > 4 { Some(...evict 2...) } else { None }
``` [3](#0-2) [4](#0-3) 

When every group has exactly 4 entries, `4 > 4` is `false` for all groups, so `candidate_peers` is empty and the function returns `Err(PeerStoreError::EvictionFailed)`. [5](#0-4) 

`add_addr()` propagates this error via `?`, and `DiscoveryAddressManager::add_new_addrs()` catches it with only a `debug!` log — no ban, no disconnect, no alert. [6](#0-5) [7](#0-6) 

## Impact Explanation
Peer discovery is permanently frozen for the targeted node. The node cannot learn new peer addresses from the network until restarted. Because `is_valid_addr()` only checks `is_reachable()` (not actual connectivity), fake addresses pass ingestion freely. If an attacker targets multiple nodes simultaneously, the network's ability to propagate peer information degrades, matching the allowed impact: **High — vulnerabilities or bad designs which could cause CKB network congestion with few costs**. [8](#0-7) 

## Likelihood Explanation
Requires only one P2P connection. The attacker sends approximately 6 `Nodes` messages (each carrying up to `MAX_ADDR_TO_SEND=1000` items × `MAX_ADDRS=3` addresses = 3000 addresses per message) containing fake addresses from 4096 different `/16` subnets. IPv4 provides 65536 possible `/16` subnets, so 4096 distinct ones are trivially available. No proof-of-work, no privileged access, and no Sybil attack is required. [9](#0-8) [10](#0-9) 

## Recommendation
Change the eviction condition at line 378 of `network/src/peer_store/peer_store_impl.rs` from:
```rust
if addrs.len() > 4 {
```
to:
```rust
if addrs.len() >= 4 {
``` [11](#0-10) 

Additionally, consider rate-limiting the number of addresses accepted per session and per source `/16` subnet during ingestion in `add_new_addrs()` to reduce the attack surface further. [12](#0-11) 

## Proof of Concept
```rust
// Fill store with 4096 groups × 4 addresses each (all connectable, attempts_count=0)
for subnet in 0u16..4096 {
    for host in 1u8..=4 {
        let ip = format!("{}.{}.1.{}", subnet >> 8, subnet & 0xff, host);
        let addr = format!("/ip4/{}/tcp/8115/p2p/Qm...", ip).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// 16385th add_addr must fail with EvictionFailed
let extra = "/ip4/200.0.1.1/tcp/8115/p2p/Qm...".parse().unwrap();
let result = peer_store.add_addr(extra, Flags::COMPATIBILITY);
assert!(matches!(result, Err(Error::PeerStore(PeerStoreError::EvictionFailed))));
assert_eq!(peer_store.addr_manager().count(), 16384); // store frozen
```
