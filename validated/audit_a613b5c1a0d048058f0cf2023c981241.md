### Title
Wrong Column Traversal in `migrate_transaction_info` Leaves `COLUMN_TRANSACTION_INFO` Unmigrated, Causing Panic on Any Transaction Lookup — (`util/migrate/src/migrations/table_to_struct.rs`)

### Summary

`ChangeMoleculeTableToStruct::migrate_transaction_info()` traverses `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO`. This leaves all pre-migration (68-byte table-format) transaction info entries untouched. After the migration completes and bumps the DB version, any call to `get_transaction_info()` reads a 68-byte entry and passes it to `from_slice_should_be_ok`, which panics because `TransactionInfoReader::verify` requires exactly 52 bytes.

### Finding Description

**The bug — line 93:** [1](#0-0) 

```rust
let (_count, nk) =
    db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
//              ^^^^^^^^^^^^^ BUG: should be COLUMN_TRANSACTION_INFO
```

Compare with the correct pattern used in every other sub-migration: [2](#0-1) [3](#0-2) 

**What happens during migration:**

1. `migrate_uncles()` runs first (line 153) and correctly rewrites uncle `HeaderView` entries from 240 bytes → 228 bytes.
2. `migrate_transaction_info()` then traverses `COLUMN_UNCLES` (now 228-byte entries). Since `228 != TRANSACTION_INFO_SIZE (52)`, every uncle entry passes the guard and the closure writes `uncle_value[16:]` (212 bytes) into `COLUMN_TRANSACTION_INFO` under the uncle's hash key.
3. The actual `COLUMN_TRANSACTION_INFO` entries (68-byte old table format) are **never read or rewritten**.
4. Migration returns `Ok(db)` and the DB version is bumped to `"20200703124523"`. [4](#0-3) 

**The panic path after restart:**

`get_transaction_info` calls `from_slice_should_be_ok` on whatever bytes are stored: [5](#0-4) 

`from_slice_should_be_ok` is defined to **unconditionally panic** on any verification error: [6](#0-5) 

`TransactionInfoReader::verify` rejects any slice that is not exactly 52 bytes: [7](#0-6) 

The old table-format entries are 68 bytes (`52 + 16` bytes of molecule table overhead). `verify` returns `TotalSizeNotMatch(52, 68)`, and `from_slice_should_be_ok` panics with:

```
failed to convert from slice: reason: ...; data: 0x<68 bytes of old tx info>
```

### Impact Explanation

Any node that ran `ckb migrate` while upgrading from a pre-v0.35.0 database will have an unmigrated `COLUMN_TRANSACTION_INFO`. After the upgrade, the node crashes (panics) on the first call to `get_transaction_info`, which is triggered by:
- RPC calls: `get_transaction`, `get_transaction_verbosity1/2`
- Chain sync and block attachment paths that look up committed transactions
- Any tx-pool or script verification path that checks whether a transaction is already committed

The node is rendered permanently inoperable until the DB is manually repaired or re-synced from scratch.

### Likelihood Explanation

The precondition is trivially met on any real mainnet or testnet node: uncle blocks exist in every epoch, and their serialized `HeaderView` size (228 bytes post-migration) is never 52 bytes. Every node operator who ran `ckb migrate` for the v0.35.0 upgrade would have hit this.

### Recommendation

Fix line 93 to traverse `COLUMN_TRANSACTION_INFO` instead of `COLUMN_UNCLES`:

```rust
let (_count, nk) =
    db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```

Add a migration version bump or a separate repair migration for nodes that already ran the broken migration.

### Proof of Concept

1. Start a CKB node on a pre-v0.35.0 DB that has at least one uncle block.
2. Run `ckb migrate`.
3. Observe migration completes with no error.
4. Restart the node and call `ckb rpc get_transaction --hash <any_committed_tx_hash>`.
5. Node panics: `failed to convert from slice: reason: TotalSizeNotMatch(52, 68); data: 0x...`
6. Verify by dumping `COLUMN_TRANSACTION_INFO` directly: entries still have 68-byte values (old table format), plus spurious entries with uncle hashes as keys containing 212-byte uncle data slices.

    fn migrate_transaction_info(&self, db: &RocksDB) -> Result<()> {
        const TRANSACTION_INFO_SIZE: usize = 52;
        let mut next_key = vec![0];
        while !next_key.is_empty() {
            let mut wb = db.new_write_batch();
            let mut transaction_info_migration = |key: &[u8], value: &[u8]| -> Result<()> {
                // (1 total size field + 3 fields) * 4 byte per field
                if value.len() != TRANSACTION_INFO_SIZE {
                    wb.put(COLUMN_TRANSACTION_INFO, key, &value[16..])?;
                }
                Ok(())
            };

            let mode = self.mode(&next_key);

            let (_count, nk) =
                db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
            next_key = nk;

            if !wb.is_empty() {
                db.write(&wb)?;
                wb.clear()?;
            }
        }
        Ok(())
    }
