### Title
Entropy `revealWithCallback` Legacy Path Propagates Consumer Callback Revert, Permanently Locking User Fees — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function has two execution paths. The **new path** (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` to catch consumer callback reverts and transitions the request to a `CALLBACK_FAILED` state. The **legacy path** (when `req.gasLimit10k == 0`) calls `_entropyCallback` directly with no error handling. If the consumer's callback reverts, the entire `revealWithCallback` transaction reverts, the request remains permanently active but unfulfillable, and the user's fee is locked in the contract.

---

### Finding Description

`revealWithCallback` is a permissionless function callable by anyone (typically the Fortuna provider service) to fulfill an entropy request. Inside, two branches exist:

**New path** (`req.gasLimit10k != 0`, lines 574–660): uses `excessivelySafeCall` to catch reverts, emits `CallbackFailed`, and sets `CALLBACK_FAILED` state — the transaction succeeds regardless of consumer behavior.

**Legacy path** (`req.gasLimit10k == 0`, lines 661–702): calls `_entropyCallback` directly without any try/catch:

```solidity
clearRequest(provider, sequenceNumber);
// ...
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
``` [1](#0-0) 

If `_entropyCallback` reverts, the entire transaction reverts (including the `clearRequest` call), leaving the request active. However, every subsequent call to `revealWithCallback` will also revert for the same reason. The request is permanently stuck: it cannot be fulfilled, and no recovery path exists for the legacy flow.

The legacy path is triggered when the provider has `defaultGasLimit == 0` (the default for providers that have never called `setDefaultGasLimit`). In `requestHelper`:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    req.gasLimit10k = 0; // legacy path
} else {
    req.gasLimit10k = roundTo10kGas(...);
}
``` [2](#0-1) 

The user's fee is credited to the provider during the **request** phase (in `requestHelper`), not the reveal phase. So the fee is already transferred to the provider's `accruedFeesInWei` at request time. The user paid for a random number they can never receive. [3](#0-2) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms this behavior — when the consumer reverts and `defaultGasLimit == 0`, `revealWithCallback` reverts: [4](#0-3) 

---

### Impact Explanation

- **User fee permanently locked**: The user paid `getFee(provider)` for a random number. The fee is credited to the provider at request time. The request can never be fulfilled, so the user receives nothing for their payment.
- **Request permanently stuck**: No recovery mechanism exists in the legacy path. Unlike the new path (which has `CALLBACK_FAILED` → retry), the legacy path has no state machine for failed callbacks.
- **Provider Fortuna service disrupted**: The Fortuna service repeatedly submits `revealWithCallback` transactions that all revert, wasting gas and blocking the fulfillment queue.

---

### Likelihood Explanation

- Any provider that has not called `setDefaultGasLimit` has `defaultGasLimit == 0`, placing all their requests on the legacy path.
- Any consumer contract with a buggy or intentionally reverting `_entropyCallback` (e.g., one that calls back into the Entropy contract and hits a reentrancy guard, or one that runs out of gas) will trigger this.
- An unprivileged user can trigger this by deploying a consumer contract that reverts in `_entropyCallback` and requesting from a legacy provider. No privileged access is required.

---

### Recommendation

Wrap the legacy-path callback invocation in a try/catch block, mirroring the new path's behavior:

```solidity
// Legacy path: wrap callback to prevent revert propagation
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    ) {} catch {
        emit CallbackFailed(provider, callAddress, sequenceNumber, ...);
    }
}
```

Alternatively, migrate all providers to the new path by requiring `defaultGasLimit > 0` for new registrations, and providing a migration path for existing legacy providers.

---

### Proof of Concept

1. Deploy a consumer contract whose `_entropyCallback` always reverts.
2. Call `requestWithCallback` on a provider with `defaultGasLimit == 0` (legacy provider). Fee is paid and credited to provider.
3. Provider's Fortuna service calls `revealWithCallback`. Transaction reverts because `_entropyCallback` reverts.
4. Repeat step 3 — every attempt reverts. The request is permanently active but unfulfillable.
5. User's fee (paid in step 2) is permanently locked — the provider holds it but the user receives no random number.
