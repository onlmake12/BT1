### Title
Unauthenticated `ConnectionSync` Bypasses Three-Way Handshake, Triggering Attacker-Directed NAT Traversal (SSRF) — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
An unprivileged peer directly connected to a victim CKB node can cause the victim to make repeated outbound TCP connection attempts to arbitrary attacker-controlled IPv4/IPv6 addresses by sending two crafted hole-punching protocol messages in sequence: a `ConnectionRequest` (to populate `pending_delivered`) followed immediately by a `ConnectionSync` (to consume it), bypassing the intended three-way handshake entirely.

### Finding Description

The hole-punching protocol is designed as a three-phase handshake:
1. `ConnectionRequest` (from → relay → to): `to` stores attacker's `listen_addrs` in `pending_delivered[from]` and sends back `ConnectionRequestDelivered`.
2. `ConnectionRequestDelivered` (to → relay → from): `from` receives delivery confirmation and sends `ConnectionSync`.
3. `ConnectionSync` (from → relay → to): `to` reads `pending_delivered[from]` and initiates NAT traversal.

**Root cause — no ordering enforcement and no sender authentication:**

**Step 1:** When the victim receives `ConnectionRequest(from=A, to=victim, listen_addrs=[attacker_ip:port])`, `respond_delivered()` stores the attacker-supplied addresses without verifying that `from=A` matches the actual sender's peer ID: [1](#0-0) 

**Step 2:** When the victim then receives `ConnectionSync(from=A, to=victim, route=[])`, `ConnectionSyncProcess::execute()` checks only `self_peer_id == &content.to`, then directly reads `pending_delivered.get(&content.from)` — with no check that a `ConnectionRequestDelivered` round-trip ever occurred: [2](#0-1) 

**Step 3:** The retrieved addresses are passed directly to `try_nat_traversal`, which spawns an async task making real outbound TCP connections for up to 30 seconds: [3](#0-2) 

The `try_nat_traversal` function retries TCP connections with jitter for a full 30-second window: [4](#0-3) 

The `from` field in `ConnectionRequest` is parsed from message content with no check that it matches the actual sending session's peer ID: [5](#0-4) 

### Impact Explanation

The victim node makes repeated outbound TCP connection attempts (200ms timeout, ~30s total, with jitter) to any IPv4/IPv6 address the attacker specifies. The address filter in `respond_delivered` only excludes Memory/Onion/Ws/Wss/Tls transports — all routable TCP addresses are accepted: [6](#0-5) 

If a connection succeeds, it is registered as a raw inbound P2P session to the attacker-controlled address: [7](#0-6) 

This constitutes SSRF: the victim can be weaponized to probe and connect to arbitrary internal or external TCP endpoints, and the three-way handshake invariant — that `ConnectionSync` is only reachable after a legitimate `ConnectionRequestDelivered` round-trip — is completely broken.

### Likelihood Explanation

Any peer with a direct P2P connection to the victim can trigger this. No special privileges, leaked keys, or majority hashpower are required. The attacker sends two well-formed protocol messages. The rate limiter (`forward_rate_limiter`) keys on `(from, to, msg_item_id)` but `msg_item_id` differs between `ConnectionRequest` (id=0) and `ConnectionSync` (id=2), so the limiter does not block the sequence. [8](#0-7) 

### Recommendation

1. **Authenticate `from`**: In `respond_delivered`, verify that `content.from` matches the actual peer ID of the sending session (looked up via `peer_registry` using `self.peer`). Reject messages where `from` does not match the sender.
2. **Enforce ordering**: In `ConnectionSyncProcess::execute()`, require that a corresponding entry exists in a separate `pending_sync` map that is only populated after a legitimate `ConnectionRequestDelivered` is processed — not by `ConnectionRequest` alone. `pending_delivered` should not be directly readable by `ConnectionSync`.
3. **Remove entry on consumption**: `pending_delivered` entries should be removed (not just read) when consumed by `ConnectionSync` to prevent replay.

### Proof of Concept

```
1. Attacker (peer B, directly connected to victim V) sends:
   ConnectionRequest { from=B, to=V, listen_addrs=[192.168.1.100:8080], max_hops=3, route=[] }

2. Victim V processes ConnectionRequestProcess::execute():
   - self_peer_id == content.to → calls respond_delivered()
   - pending_delivered.insert(B, ([192.168.1.100:8080], now))  ← attacker controls this address

3. Attacker immediately sends (no relay, no ConnectionRequestDelivered):
   ConnectionSync { from=B, to=V, route=[] }

4. Victim V processes ConnectionSyncProcess::execute():
   - self_peer_id == content.to
   - pending_delivered.get(&B) → Some([192.168.1.100:8080])
   - runtime::spawn(try_nat_traversal(bind_addr, 192.168.1.100:8080))
   → Victim makes repeated TCP SYN packets to 192.168.1.100:8080 for 30 seconds
```

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
