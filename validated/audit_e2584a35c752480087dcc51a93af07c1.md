### Title
Provider Earns Fees Without Delivering Callback via EIP-150 Gas Griefing - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, `executeCallback` credits the provider's fees and clears the request **before** invoking the consumer callback. The callback is wrapped in a bare `try/catch` with no gas-sufficiency check. Due to EIP-150 (at most 63/64 of gas is forwarded to a sub-call), a caller can supply just enough gas to complete all pre-callback logic but not enough to actually forward `req.callbackGasLimit` gas to `_echoCallback`. The callback reverts out-of-gas, the `catch` block executes using the residual 1/64 gas, and the provider retains their credited fees while the consumer's callback is silently dropped and the request is permanently cleared.

---

### Finding Description

`executeCallback` in `Echo.sol` follows this execution order:

1. Validates the request and parses price feeds.
2. **Credits provider fees** (`accruedFeesInWei += ...`) — line 161–162.
3. **Clears the request** (`clearRequest(sequenceNumber)`) — line 164.
4. Calls `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` inside a `try/catch` — lines 176–201.

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // fees credited first

clearRequest(sequenceNumber);                       // request cleared second

try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(...);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(..., reason);
} catch {
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
``` [1](#0-0) 

Because EIP-150 caps the gas forwarded to a sub-call at 63/64 of the current context's gas, if the caller provides only `callbackGasLimit * 64/63 + ε` gas at the point of the `try` statement, the sub-call receives exactly `callbackGasLimit` gas (or less), runs out of gas inside `_echoCallback`, and reverts. The remaining 1/64 gas in the outer context is sufficient to execute the `catch` block and emit `PriceUpdateCallbackFailed`. The function then returns successfully — with the provider already credited and the request already cleared.

There is **no check** that the outer context held enough gas to guarantee the full `callbackGasLimit` was actually forwarded. Compare this to `Entropy.sol`, which explicitly measures `startingGas` before the call and reverts with `InsufficientGas()` when `(startingGas * 31) / 32 <= gasLimit`:

```solidity
uint256 startingGas = gasleft();
(success, ret) = req.requester.excessivelySafeCall(...);
...
} else if ((startingGas * 31) / 32 > uint256(req.gasLimit10k) * TEN_THOUSAND) {
    // sufficient gas — genuine callback failure
    req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;
} else {
    revert EntropyErrors.InsufficientGas();  // insufficient gas — revert
}
``` [2](#0-1) 

`Echo.sol` has no equivalent protection.

---

### Impact Explanation

- The consumer paid a fee that includes gas costs for the callback (`callbackGasLimit * feePerGasInWei`), but the callback is never executed.
- The request is permanently cleared (`clearRequest` was called before the callback), so there is no retry path.
- The provider collects the full fee without performing the service.
- This constitutes a direct financial loss to the consumer: they paid for a callback delivery that was deliberately withheld. [3](#0-2) 

---

### Likelihood Explanation

- `executeCallback` is callable by **any address** once the exclusivity period expires (and by the assigned provider at any time).
- The attack requires only crafting a transaction with a specific gas limit — no privileged access, no leaked keys, no external oracle manipulation.
- The attacker is the direct beneficiary (they receive `accruedFeesInWei`), giving a clear economic incentive.
- The exclusivity period is configurable and can be zero, making the attack immediately available to any unprivileged caller. [4](#0-3) 

---

### Recommendation

Before the `try` block, record `gasleft()` and verify that the outer context holds enough gas to guarantee `callbackGasLimit` is forwarded, mirroring the pattern in `Entropy.sol`:

```solidity
uint256 startingGas = gasleft();
// EIP-150: at most 63/64 of gas is forwarded; use 31/32 as a safety margin
require(
    (startingGas * 31) / 32 >= uint256(req.callbackGasLimit),
    "Insufficient gas to forward callbackGasLimit"
);
```

Alternatively, adopt `excessivelySafeCall` (already used in `Entropy.sol`) and apply the same `(startingGas * 31) / 32 > gasLimit` guard before treating a failed call as a genuine consumer-side revert rather than a gas-griefing attack. [5](#0-4) 

---

### Proof of Concept

1. Alice (requester) calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 200_000`, paying the full fee including `200_000 * feePerGasInWei`.
2. Bob (provider or any caller after exclusivity) calls `executeCallback` with a transaction gas limit of approximately `200_000 * 64/63 + overhead` — enough to complete all pre-callback logic but not enough to guarantee 200,000 gas is forwarded.
3. Inside `executeCallback`:
   - Bob's `accruedFeesInWei` is incremented by the full fee amount (line 161–162).
   - The request is cleared (line 164).
   - `_echoCallback{gas: 200_000}` is called; due to EIP-150, fewer than 200,000 gas are actually forwarded; the callback reverts out-of-gas.
   - The `catch` block emits `PriceUpdateCallbackFailed` using the residual 1/64 gas.
4. `executeCallback` returns successfully. Bob withdraws his accrued fees. Alice's consumer contract never received the price update callback and has no recourse — the request is gone. [6](#0-5)

                // than the indicated gas limit. (See CALL opcode docs here https://www.evm.codes/?fork=cancun#f1)

                // was truly provided with a sufficient amount of gas.
                uint256(req.gasLimit10k) * TEN_THOUSAND,

                    sequenceNumber,
                    provider,
                    randomNumber
                )

            uint32 gasUsed = SafeCast.toUint32(startingGas - gasleft());
            // Reset status to not started here in case the transaction reverts.
            req.callbackStatus = EntropyStatusConstants.CALLBACK_NOT_STARTED;

            if (success) {
                emit RevealedWithCallback(

                    userContribution,

                    randomNumber

                // catch out-of-gas errors and revert with a different error.
                // In this case, ensure that the callback was provided with sufficient gas. Technically, 63/64ths of the startingGas is forwarded,

                    userContribution,

                );
                req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;

                // in calls (see the comment on the call above).
