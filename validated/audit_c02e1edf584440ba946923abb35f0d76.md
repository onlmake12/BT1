### Title
Tx-Pool Minimum Fee Rate Check Uses Serialized Size Instead of Actual Weight, Allowing Cycle-Heavy Transactions to Bypass Fee Protection - (File: tx-pool/src/util.rs)

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` enforces the `min_fee_rate` threshold by computing the minimum fee against the transaction's **serialized byte size** (`tx_size`) rather than its true **weight** (`max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`). This is explicitly acknowledged in a code comment as a "cheap check." Because no subsequent re-check is performed after actual cycle counts are known, a transaction sender can craft a cycle-heavy transaction that passes the admission gate while its real effective fee rate is a small fraction of `min_fee_rate`.

### Finding Description

In `check_tx_fee`, the minimum fee is computed as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code comment directly acknowledges the discrepancy:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly"

The actual transaction weight used everywhere else in the codebase (sorting, eviction, fee estimation) is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. For a transaction consuming the maximum allowed `70_000_000` cycles, the cycle-derived weight component is `70_000_000 × 0.000_170_571_4 ≈ 11,940`. A small transaction of, say, 200 bytes would have an actual weight of `max(200, 11940) = 11,940`, but `check_tx_fee` uses only `200` as the weight.

After `check_tx_fee` passes, `verify_rtx` runs and produces the actual cycle count, which is stored in `TxEntry::new(rtx, verified.cycles, fee, tx_size)`. However, there is **no second fee-rate check** against the actual weight after cycles are known. The `TxEntry::fee_rate()` method computes the real fee rate using `get_transaction_weight`, but this is only used for pool ordering and eviction — not for admission enforcement.

**Concrete example with `min_fee_rate = 1000 shannons/KW`:**
- Transaction: 200 bytes, 70,000,000 cycles
- Actual weight: 11,940
- Size-based minimum fee (what `check_tx_fee` requires): `1000 × 200 / 1000 = 200 shannons`
- Actual minimum fee that should be required: `1000 × 11940 / 1000 = 11,940 shannons`
- A transaction paying 201 shannons passes admission but has an effective fee rate of `201 × 1000 / 11940 ≈ 16.8 shannons/KW` — only **1.68%** of the required `min_fee_rate`.

### Impact Explanation

An unprivileged transaction sender (RPC caller via `send_transaction`, or a P2P peer relaying a transaction) can submit cycle-heavy transactions that:
1. Pass the `min_fee_rate` admission check by paying only the size-based minimum fee
2. Consume significant block cycle budget at a tiny fraction of the intended cost
3. Flood the tx-pool with economically underpriced transactions, degrading the fee market
4. Cause miners to include transactions that pay far less fee per unit of computational work than the node operator's `min_fee_rate` policy intends

### Likelihood Explanation

The attack is trivially reachable by any RPC caller or P2P peer. Constructing a cycle-heavy, small-byte transaction requires only writing a script that loops extensively (e.g., a lock script with a tight computation loop). No special privileges, keys, or majority hashpower are required. The discrepancy is largest when cycles dominate the weight, which is achievable up to `max_tx_verify_cycles`.

### Recommendation

After `verify_rtx` returns the actual cycle count, re-validate the fee rate against the true weight before admitting the transaction to the pool:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This second check should be performed in the post-verification submission path (e.g., in `submit_entry` or the verification result handler in `process.rs`) where `verified.cycles` is available.

### Proof of Concept

1. Craft a CKB transaction with a lock script that consumes close to `max_tx_verify_cycles` (70,000,000) cycles but has a small serialized size (~200 bytes).
2. Set the transaction fee to just above `min_fee_rate × tx_size / 1000` (e.g., 201 shannons for a 200-byte tx with `min_fee_rate = 1000`).
3. Submit via `send_transaction` RPC.
4. Observe: `check_tx_fee` passes because `fee (201) >= min_fee (200)`.
5. After verification, `TxEntry::fee_rate()` returns `≈ 16.8 shannons/KW`, far below `min_fee_rate = 1000 shannons/KW`.
6. The transaction is admitted to the pool and eligible for mining despite violating the intended fee rate floor.
