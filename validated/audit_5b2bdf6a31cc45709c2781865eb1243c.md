### Title
Unbounded Growth of `peers` Vec in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` and Missing Cleanup on Peer Disconnect — (`sync/src/types/mod.rs`)

---

### Summary

The `add_ask_for_txs` function in `sync/src/types/mod.rs` allows any connected peer to repeatedly push their `PeerIndex` into the `peers: Vec<PeerIndex>` field of `UnknownTxHashPriority` without deduplication or a per-peer size cap. A malicious peer can send repeated `RelayTransactionHashes` messages containing the same tx hashes, causing unbounded memory growth. Compounding this, `SyncState::disconnected` never removes stale peer indices from `unknown_tx_hashes`, leaving the map in an incorrect state after a peer disconnects — an exact analog to the "incomplete state reset" described in the reference report.

---

### Finding Description

**Root cause 1 — Unbounded `peers` Vec growth:**

`UnknownTxHashPriority` holds a `Vec<PeerIndex>` with no capacity bound: [1](#0-0) 

`push_peer` appends unconditionally: [2](#0-1) 

In `add_ask_for_txs`, when a tx hash is already present in the map (the `Occupied` branch), the calling peer's index is pushed again with no deduplication check: [3](#0-2) 

The per-call limit `take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` only caps the number of hashes processed per message, not the number of times the same peer can push into the same entry: [4](#0-3) 

The overflow guard that triggers `TooManyUnknownTransactions` only fires when the total number of *unique* hashes in the map reaches `MAX_UNKNOWN_TX_HASHES_SIZE = 50 000`: [5](#0-4) 

A peer that sends 32 767 unique hashes keeps the map at 32 767 entries — below the 50 000 threshold — so the guard never fires. The peer can then resend the same 32 767 hashes indefinitely, pushing its index into every entry's `peers` Vec on each round trip.

**Root cause 2 — Incomplete cleanup on peer disconnect:**

`SyncState::disconnected` removes the peer from `inflight_blocks` and from `peers.state`, but never touches `unknown_tx_hashes`: [6](#0-5) 

After the peer disconnects, every `UnknownTxHashPriority` entry that the peer populated still contains the stale `PeerIndex`. `pop_ask_for_txs` will later call `next_request_peer`, which returns the stale index and attempts to send a `GetRelayTransactions` message to the now-disconnected peer: [7](#0-6) 

This is the direct analog of the reference report's "does not delete the array / methods still think `claimed == true`" flaw.

The constants involved: [8](#0-7) 

---

### Impact Explanation

**Memory exhaustion.** Each round of 32 767 hashes adds one `PeerIndex` (≥ 4 bytes) to each of the 32 767 entries. After *k* rounds: `32 767 × k × 4` bytes of heap growth. At *k* = 10 000 rounds this exceeds 1.3 GB from a single peer, with no internal bound to stop it.

**CPU spike under lock.** When the map does reach 50 000 entries (e.g., via multiple peers), every subsequent call to `add_ask_for_txs` performs an O(n × p) nested scan while holding the `unknown_tx_hashes` mutex, where *n* = 50 000 and *p* = the (now unbounded) length of each `peers` Vec: [9](#0-8) 

**Stale-state incorrect behavior.** After a peer disconnects, the node wastes outbound bandwidth attempting to fetch transactions from a peer that no longer exists, and the `peers` Vec is never reclaimed, permanently leaking memory for the lifetime of those map entries.

Impact rating: **Medium** — no funds are at risk, but node availability and memory integrity are degraded.

---

### Likelihood Explanation

Any unprivileged inbound or outbound peer that speaks the Relay protocol can trigger this. The attacker needs only to:

1. Connect to the victim node.
2. Repeatedly send `RelayTransactionHashes` messages containing the same ≤ 32 767 tx hashes (hashes for transactions that will never be delivered, so they are never removed from the map via `mark_as_known_txs`).
3. Optionally disconnect to leave stale indices behind.

No special privileges, no majority hashpower, no Sybil attack required. The `RelayTransactionHashes` message path has no per-peer rate limit in the reviewed code.

Likelihood rating: **Medium** — not triggered in normal operation, but straightforward for a motivated attacker.

---

### Recommendation

1. **Deduplicate `peers` in `add_ask_for_txs`**: Before calling `push_peer`, check whether `peer_index` is already present in `priority.peers`. Alternatively, replace `Vec<PeerIndex>` with a `HashSet<PeerIndex>` or a bounded `SmallVec`.

2. **Cap `peers` Vec size**: Enforce a maximum length (e.g., equal to `MAX_OUTBOUND_PEERS_TO_PROTECT_FROM_DISCONNECT` or the total peer count) inside `push_peer` or `add_ask_for_txs`.

3. **Clean up `unknown_tx_hashes` on peer disconnect**: In `SyncState::disconnected`, iterate over `unknown_tx_hashes` and remove the disconnecting peer's index from every `UnknownTxHashPriority.peers` Vec, or remove entries whose `peers` Vec becomes empty.

4. **Move the per-peer counter check before insertion**: The `TooManyUnknownTransactions` guard currently runs *after* the peer's index has already been pushed into the map. The check should precede the `push_peer` call.

---

### Proof of Concept

```
1. Attacker connects to a CKB node via the Relay protocol.

2. Attacker constructs a RelayTransactionHashes message containing
   MAX_RELAY_TXS_NUM_PER_BATCH (32 767) hashes for transactions
   that do not exist on the network (so they are never delivered
   and never removed from unknown_tx_hashes via mark_as_known_txs).

3. Attacker sends this message in a tight loop.

4. On each iteration, add_ask_for_txs is called:
   - The 32 767 hashes are already Occupied in the map.
   - push_peer(attacker_peer_index) is called 32 767 times.
   - The map size remains 32 767 < MAX_UNKNOWN_TX_HASHES_SIZE (50 000),
     so the overflow guard never fires.

5. After k iterations:
   - Each of the 32 767 entries has a peers Vec of length k.
   - Heap growth ≈ 32 767 × k × sizeof(PeerIndex) bytes.
   - At k = 10 000: ~1.3 GB consumed by a single peer.

6. Attacker disconnects.
   - SyncState::disconnected does NOT purge the attacker's PeerIndex
     from unknown_tx_hashes.
   - pop_ask_for_txs continues to return the stale PeerIndex and
     attempts to send GetRelayTransactions to the disconnected peer.
```

                self.peers.swap_remove(0);
                self.peers.first().cloned()
