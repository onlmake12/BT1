### Title
Excess Native Token Sent to Entropy Request Functions Is Not Refunded to Caller — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.requestHelper()` is the shared internal handler for all payable randomness request entry points (`request`, `requestWithCallback`, `requestV2`). It enforces a minimum fee (`msg.value >= requiredFee`) but silently credits any excess `msg.value` beyond the provider fee directly into `_state.accruedPythFeesInWei` rather than refunding it to the caller. Users who overpay lose the excess permanently to Pyth's fee pool.

---

### Finding Description

`requestHelper()` performs the following fee accounting:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The expression `SafeCast.toUint128(msg.value) - providerFee` is assigned to `_state.accruedPythFeesInWei`. When `msg.value > requiredFee`, the surplus (`msg.value - requiredFee`) is silently absorbed into Pyth's accrued fee balance instead of being returned to `msg.sender`. The contract's own NatSpec comments acknowledge this explicitly:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

All three public payable entry points delegate to `requestHelper()` and are therefore affected:

- `request()` [4](#0-3) 
- `requestWithCallback()` [5](#0-4) 
- `requestV2(address, bytes32, uint32)` [6](#0-5) 

---

### Impact Explanation

Any caller who sends `msg.value` greater than `getFeeV2(provider, gasLimit)` permanently loses the excess ETH (or native token on non-ETH chains). The excess is credited to `_state.accruedPythFeesInWei`, which is controlled by Pyth governance and withdrawable only by Pyth — not by the original sender. The user has no mechanism to recover the overpayment. This constitutes a direct, irreversible loss of user funds.

---

### Likelihood Explanation

Overpayment is a realistic and common occurrence:

1. **Stale fee estimates**: A caller queries `getFeeV2()` off-chain, then the fee decreases before the transaction is mined. The caller pays the old (higher) fee; the excess is lost.
2. **Round-number payments**: Integrators or end-users frequently send a round ETH amount (e.g., `0.001 ether`) rather than the exact fee returned by `getFeeV2()`.
3. **Wrapper contracts with fee buffers**: Consumer contracts (e.g., `IEntropyConsumer` implementations) often add a safety margin to the fee to avoid reverts, as shown in the SDK documentation examples. Any margin above the exact fee is silently taken.
4. **Multi-call batching**: Contracts that forward `msg.value` from an outer call without exact fee calculation will routinely overpay.

The vulnerability requires no special privileges — any unprivileged Entropy user triggers it by calling a standard public payable function.

---

### Recommendation

Add a refund of excess `msg.value` at the end of `requestHelper()`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // use requiredFee, not msg.value

// Refund excess
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

This mirrors the fix applied to `BreederTimeLockPool.crossChainClaimReward()` and is consistent with how `PythAggregatorV3` already handles excess value after calling `updatePriceFeeds`. [7](#0-6) 

---

### Proof of Concept

1. Provider is registered with `getFeeV2(provider, 0) = 1000 wei`.
2. User calls `entropy.requestV2{value: 2000}(provider, userRandom, 0)`.
3. Inside `requestHelper`:
   - `requiredFee = 1000`
   - `msg.value (2000) >= requiredFee (1000)` → no revert
   - `providerInfo.accruedFeesInWei += providerFee` (e.g., 800)
   - `_state.accruedPythFeesInWei += (2000 - 800) = 1200` ← 200 wei excess absorbed by Pyth
4. User receives their sequence number but permanently loses 1000 wei of excess.
5. The 200 wei surplus (above the normal Pyth fee of 200) is now in Pyth's fee pool, withdrawable only by Pyth governance via `EntropyGovernance`. [8](#0-7)
