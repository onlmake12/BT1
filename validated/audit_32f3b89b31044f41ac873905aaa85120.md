### Title
Zero-Value Price Silently Treated as "Missing" via Undocumented Implicit Assumption in `parseUpdateFromPayload` — (`lazer/contracts/evm/src/PythLazerLib.sol`)

---

### Summary

`PythLazerLib.parseUpdateFromPayload` uses `value != 0` as the sole criterion to distinguish a **present** price from an **applicable-but-missing** price when populating the `triStateMap` bit-field. This is an undocumented implicit assumption: a legitimately signed price of exactly `0` is silently demoted to `ApplicableButMissing`, causing `hasPrice()` to return `false` and `getPrice()` to revert. The same flaw applies to `BestBidPrice`, `BestAskPrice`, `Confidence`, `PublisherCount`, `EmaPrice`, and `EmaConfidence`.

---

### Finding Description

In `parseUpdateFromPayload`, after reading each price field from the verified payload, the code decides the tri-state as follows:

```solidity
// lazer/contracts/evm/src/PythLazerLib.sol lines 252-264
(feed._price, pos) = parseFeedValueInt64(payload, pos);
if (feed._price != 0) {
    _setPresent(feed, uint8(PriceFeedProperty.Price));
} else {
    _setApplicableButMissing(feed, uint8(PriceFeedProperty.Price));
}
```

The same `!= 0` sentinel pattern is repeated for `BestBidPrice` (line 274), `BestAskPrice` (line 298), `PublisherCount` (line 323), `Confidence` (line 361), `EmaPrice` (line 486), and `EmaConfidence` (line 505). [1](#0-0) 

This conflates two distinct protocol states:
- **"The Lazer network has a price and it is 0"** → should be `Present`
- **"The Lazer network has no price available at this timestamp"** → should be `ApplicableButMissing`

The `triStateMap` encoding in `PythLazerStructs.Feed` documents three states (`00 NotApplicable`, `01 ApplicableButMissing`, `10 Present`, `11 Reserved`) but nowhere documents the invariant that zero is not a valid price value. [2](#0-1) 

By contrast, the `FundingRate`, `FundingTimestamp`, `FundingRateInterval`, and `FeedUpdateTimestamp` properties correctly use an **explicit `exists` flag** (a separate `uint8` byte in the payload) to determine presence, independent of the value:

```solidity
// lines 377-397
uint8 exists;
(exists, pos) = parseFeedValueUint8(payload, pos);
if (exists != 0) {
    (feed._fundingRate, pos) = parseFeedValueInt64(payload, pos);
    _setPresent(...);
} else {
    _setApplicableButMissing(...);
}
``` [3](#0-2) 

The `getPrice()` safe getter enforces both checks, so it will revert for any zero price: [4](#0-3) 

---

### Impact Explanation

Any consumer contract that calls `hasPrice(feed)` or `getPrice(feed)` on a Lazer update containing a legitimately zero price will:

1. Receive `false` from `hasPrice()` — incorrect state, the price **was** present in the signed payload.
2. Have `getPrice()` revert with `"Price is not present for the timestamp"` — a silent, misleading failure.
3. Potentially fall back to a stale cached price rather than the current zero price.

For DeFi protocols (lending, perpetuals, liquidation engines) integrated with Pyth Lazer, this means critical operations that depend on a zero price (e.g., triggering liquidations when a collateral asset crashes to zero, or settling expired derivatives) would silently fail or use incorrect data.

---

### Likelihood Explanation

For major assets (BTC, ETH) a price of exactly zero is extremely unlikely. However:

- Synthetic instruments, newly listed tokens, or assets undergoing a terminal event can legitimately reach zero.
- The Lazer protocol itself may use zero as a valid signal (e.g., market closed, asset delisted, or insufficient publishers producing a zero aggregate).
- The bug is **latent and silent**: no error is raised at parse time; the consumer simply sees the price as "missing" and may proceed with stale data without any indication that a zero price was actually delivered.

The `FundingRate` family of properties already demonstrates that the Lazer payload format supports explicit `exists` flags precisely because zero is a valid value for those fields — confirming the protocol designers are aware of this distinction, making the omission for price fields a concrete inconsistency.

---

### Recommendation

Replace the `value != 0` sentinel check for `Price`, `BestBidPrice`, `BestAskPrice`, `Confidence`, `PublisherCount`, `EmaPrice`, and `EmaConfidence` with an explicit `exists` flag in the payload encoding, consistent with how `FundingRate` and related properties are handled. If the protocol guarantees that zero is never a valid price at the data-source level, this invariant must be explicitly documented in `PythLazerStructs.Feed` and enforced at the parser level with a `require(feed._price != 0, "price cannot be zero")` rather than silently demoting it to `ApplicableButMissing`.

---

### Proof of Concept

1. A trusted Lazer signer produces a valid, correctly-signed payload for feed ID `X` with `Price = 0` (e.g., a synthetic asset that has crashed).
2. A consumer contract calls `pythLazer.verifyUpdate{value: fee}(update)` — this **succeeds** (signature is valid).
3. The consumer calls `PythLazerLib.parseUpdateFromPayload(payload)`.
4. Inside `parseUpdateFromPayload`, line 253: `parseFeedValueInt64` reads `0` from the payload.
5. Line 254: `if (feed._price != 0)` → `false`.
6. Lines 260–263: `_setApplicableButMissing()` sets `triStateMap` bits for `Price` to `01`.
7. Consumer calls `PythLazerLib.hasPrice(feed)` → returns `false` (incorrect; price was present and equal to 0).
8. Consumer calls `PythLazerLib.getPrice(feed)` → reverts: `"Price is not present for the timestamp"`.
9. Consumer falls back to a stale price or halts, while the actual current price is zero — leading to incorrect financial decisions (e.g., no liquidation triggered when collateral is worthless). [5](#0-4) [6](#0-5)
