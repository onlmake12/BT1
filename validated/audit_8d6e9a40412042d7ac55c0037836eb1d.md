EIP-150 1/64 Gas Rule Allows Silent Callback Failure with Permanent Request Loss - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` forwards exactly `req.callbackGasLimit` gas to the consumer callback via `{gas: req.callbackGasLimit}`, but performs no pre-call or post-call check that the calling context actually had enough gas to honour that limit. Due to EIP-150's 1/64 rule, the EVM can only forward at most `gasleft() * 63/64` gas to a sub-call. A malicious provider (or any caller after the exclusivity period) can supply a transaction gas limit that is just sufficient to execute `executeCallback`'s own logic but not sufficient to forward the full `callbackGasLimit` to the consumer. The callback silently fails, is caught by the bare `catch` block, and the request is **permanently cleared** — the user's callback is irrecoverably lost while the provider is already paid.

---

### Finding Description

In `Echo.executeCallback`, the request is cleared and provider fees are credited **before** the callback is attempted:

```solidity
// Echo.sol lines 161–164
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

clearRequest(sequenceNumber);   // ← request permanently gone
```

Then the callback is issued with a hard gas cap:

```solidity
// Echo.sol lines 176–179
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
```

The EVM's CALL opcode (EIP-150) caps the gas actually forwarded at `min(req.callbackGasLimit, gasleft() * 63/64)`. If the caller supplies a transaction gas limit such that, at the point of the sub-call, `gasleft() * 63/64 < req.callbackGasLimit`, the callback receives less gas than the user paid for. The callback runs out of gas and reverts. The bare `catch` block absorbs the revert silently:

```solidity
// Echo.sol lines 192–200
} catch {
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
```

There is **no post-call check** analogous to the one in `Entropy.revealWithCallback` (which uses `(startingGas * 31) / 32 > gasLimit` to detect under-gassed calls and revert the whole transaction). Echo has no such guard.

---

### Impact Explanation

- The consumer's `_echoCallback` is never executed despite the user having paid `callbackGasLimit * feePerGasInWei` for it.
- `clearRequest` has already run; the sequence number is gone. There is no retry path.
- Provider fees are already credited; the provider suffers no penalty.
- The user permanently loses the price-update callback they paid for. Any application state that depends on the callback (e.g., a lottery draw, a liquidation trigger, a DeFi settlement) is silently skipped.
- After the exclusivity period, **any** address can call `executeCallback`, so the attacker does not need to be the assigned provider.

---

### Likelihood Explanation

- `executeCallback` is a public, permissionless function (any caller is valid after the exclusivity window).
- The attacker only needs to craft a transaction with a specific gas limit — a trivial on-chain action.
- The gas arithmetic is deterministic and predictable off-chain, so the attack can be executed reliably.
- The economic incentive exists: a competing provider could grief a rival's reputation, or an attacker could cause application-level failures (e.g., stuck lotteries) for extortion.

---

### Recommendation

Add a post-call gas check mirroring the pattern already used in `Entropy.revealWithCallback`:

```solidity
uint256 startingGas = gasleft();
try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...) {
    ...
} catch {
    // Only treat as a legitimate callback failure if enough gas was forwarded.
    // Using 31/32 (more conservative than 63/64) as a safety margin.
    if ((startingGas * 31) / 32 <= req.callbackGasLimit) {
        // Caller did not supply enough gas — revert so the request is not lost.
        revert InsufficientGas();
    }
    emit PriceUpdateCallbackFailed(...);
}
```

This ensures that if the callback failed because the caller under-supplied gas, the entire transaction reverts (preserving the request and the user's funds), rather than silently clearing the request.

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 1_000_000`. The request is stored and the user pays `1_000_000 * feePerGasInWei`.

2. After the exclusivity period, attacker calls:
   ```solidity
   echo.executeCallback{gas: G}(provider, sequenceNumber, updateData, priceIds);
   ```
   where `G` is chosen so that at the point of the sub-call, `gasleft() ≈ 1_032_258` (just above `1_000_000 * 64/63`), meaning `gasleft() * 63/64 ≈ 1_015_748` — still above `1_000_000`, so the check would pass if one existed. But the attacker sets `G` slightly lower, e.g. so `gasleft() ≈ 1_000_100` at the sub-call site, yielding `gasleft() * 63/64 ≈ 984,473` — about 15,527 gas short of the required `1_000_000`.

3. The callback receives only ~984,473 gas, runs out of gas, and reverts.
