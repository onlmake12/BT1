### Title
Hardcoded `DEFAULT_BYTES_PER_CYCLES` and `MAX_BLOCK_BYTES` Constants Used in Weight and Fee-Rate Calculations Instead of Dynamic Consensus Values — (`util/types/src/core/tx_pool.rs`, `util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

`get_transaction_weight` uses a compile-time constant `DEFAULT_BYTES_PER_CYCLES` to convert VM cycles into byte-equivalent weight. Separately, the `weight_units_flow` fee estimator uses the compile-time constant `MAX_BLOCK_BYTES` to compute how much weight is drained per block. Both values are hardcoded at build time and never queried from the live consensus object, even though `max_block_bytes` and `max_block_cycles` are configurable per chain spec and can diverge from the defaults. This is the direct CKB analog of the Smart Vault bug where USDC/USDs were assumed to always be worth $1.

---

### Finding Description

**Root cause 1 — `get_transaction_weight` / `DEFAULT_BYTES_PER_CYCLES`**

`util/types/src/core/tx_pool.rs` defines:

```rust
/// Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

The constant `0.000_170_571_4` is the ratio `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES` computed from the **default** consensus constants:

```rust
pub const MAX_BLOCK_BYTES: u64 = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT; // 597 * 1000 = 597,000
pub const MAX_BLOCK_CYCLES: u64 = TWO_IN_TWO_OUT_CYCLES * TWO_IN_TWO_OUT_COUNT; // 3_500_000 * 1000
``` [2](#0-1) 

However, `max_block_bytes` and `max_block_cycles` are **runtime consensus fields** that are independently configurable per chain spec:

```rust
pub struct Consensus {
    pub max_block_cycles: Cycle,
    pub max_block_bytes: u64,
    ...
}
``` [3](#0-2) 

For example, the dev chain spec sets `max_block_cycles = 10_000_000_000` — nearly 3× the default `3_500_000_000` — while `max_block_bytes` stays at the default `597,000`: [4](#0-3) 

On that chain the true ratio is `597,000 / 10,000,000,000 ≈ 0.0000000597`, roughly **2,857× smaller** than `DEFAULT_BYTES_PER_CYCLES`. Every call to `get_transaction_weight` for a cycle-heavy transaction returns a wildly inflated weight, making the computed fee rate appear far lower than it actually is.

`get_transaction_weight` is called in every fee-admission check, every tx-pool sort/eviction decision, and every fee-rate statistic: [5](#0-4) [6](#0-5) [7](#0-6) 

---

**Root cause 2 — `weight_units_flow` fee estimator uses `MAX_BLOCK_BYTES` constant**

The `weight_units_flow` fee estimator imports the compile-time constant `MAX_BLOCK_BYTES` and uses it to compute how much weight is removed from the mempool per block:

```rust
use ckb_chain_spec::consensus::MAX_BLOCK_BYTES;
...
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
``` [8](#0-7) [9](#0-8) 

The actual consensus `max_block_bytes` is never consulted. If the live chain's `max_block_bytes` differs from the compile-time constant, the estimator's model of how fast the mempool drains is wrong, producing systematically biased fee-rate recommendations.

---

### Impact Explanation

| Scenario | Effect |
|---|---|
| `DEFAULT_BYTES_PER_CYCLES` too **high** (actual `max_block_cycles` larger than default) | Cycle-heavy txs get inflated weight → computed fee rate appears **lower** than actual → legitimate txs rejected by `check_tx_fee` even when they pay sufficient fees |
| `DEFAULT_BYTES_PER_CYCLES` too **low** (actual `max_block_cycles` smaller than default) | Cycle-heavy txs get deflated weight → computed fee rate appears **higher** than actual → txs with genuinely insufficient fees pass the min-fee-rate gate, enabling cheap mempool spam |
| `MAX_BLOCK_BYTES` wrong in fee estimator | `estimate_fee_rate` RPC returns incorrect fee recommendations; users systematically overpay or underpay, and underpaying txs stall |

The second row is the security-relevant case: an attacker on a chain where `DEFAULT_BYTES_PER_CYCLES` overestimates the true ratio can submit cycle-heavy transactions whose real fee rate is below `min_fee_rate` but whose computed weight makes them appear compliant, bypassing the admission gate in `check_tx_fee`.

---

### Likelihood Explanation

On mainnet the default constants happen to match the deployed consensus parameters, so the discrepancy is zero. The risk is **real and immediate** on:

- The dev chain (`max_block_cycles = 10_000_000_000` vs default `3_500_000_000`).
- Any custom or staging chain with non-default block limits.
- Any future hardfork that adjusts `max_block_bytes` or `max_block_cycles` without updating the hardcoded constant — exactly the scenario the external report describes for stablecoin de-pegging.

An unprivileged tx-pool submitter or RPC caller is the entry point; no special privilege is required.

---

### Recommendation

1. **Remove `DEFAULT_BYTES_PER_CYCLES` as a compile-time constant.** Pass the live `Consensus` object (or its `max_block_bytes` / `max_block_cycles` fields) into `get_transaction_weight` and compute the ratio dynamically:
   ```rust
   pub fn get_transaction_weight(tx_size: usize, cycles: u64, consensus: &Consensus) -> u64 {
       let bytes_per_cycle = consensus.max_block_bytes() as f64
           / consensus.max_block_cycles() as f64;
       std::cmp::max(tx_size as u64, (cycles as f64 * bytes_per_cycle) as u64)
   }
   ```

2. **In `weight_units_flow`**, accept the live `max_block_bytes` from the consensus snapshot rather than importing the compile-time `MAX_BLOCK_BYTES` constant, mirroring how `block_assembler` already queries `consensus.max_block_bytes()` at runtime. [10](#0-9) 

---

### Proof of Concept

1. Start a CKB node with the dev chain spec (`max_block_cycles = 10_000_000_000`).
2. Construct a transaction whose actual fee rate (fee / true weight) is just below `min_fee_rate`, but whose **computed** weight via `get_transaction_weight` (using the inflated `DEFAULT_BYTES_PER_CYCLES`) makes the apparent fee rate exceed `min_fee_rate`.
3. Submit via `send_transaction` RPC. `check_tx_fee` calls `get_transaction_weight` with the wrong constant, computes an inflated weight, derives a fee rate that passes the threshold, and admits the transaction.
4. Repeat to fill the mempool with transactions that pay less than the true minimum fee rate, degrading pool quality without paying the required cost.
