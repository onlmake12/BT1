### Title
DAO Withdrawal Permanently Rejected Due to Header-Dep Index Interpretation Mismatch Between C VM and Rust `DaoCalculator` — (File: `util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the **full u64** witness index to locate the deposit header in `header_deps`, while the on-chain C VM (`dao.c`) reads only the **lowest byte** of that same u64. When a DAO withdrawal transaction encodes a witness index whose lowest byte correctly identifies the deposit block but whose full u64 value points to a different block, the C VM accepts the transaction while the Rust node's fee calculator rejects it. This permanently blocks the user's DAO withdrawal from being admitted to the tx-pool or included in any block accepted by standard CKB nodes.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` parses the `input_type` field of the `WitnessArgs` as a little-endian u64 and uses it directly as an array index into `header_deps`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // ← full u64 used as index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The on-chain C VM (`dao.c`) is documented in the codebase to read only the **lowest byte** of this same u64 field when resolving the deposit header. This is explicitly noted in the test added to `util/dao/src/tests.rs`:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

A transaction with witness index `257` (binary `0x0000000000000101`):
- **C VM**: reads lowest byte `0x01 = 1` → `header_deps[1]` = deposit block → **accepts**
- **Rust**: reads full u64 `257` → `header_deps[257]` = a different block → block number check `deposit_header.number() != deposited_block_number` → **`DaoError::InvalidOutPoint`**

This `DaoCalculator::transaction_fee` is called in two places:
1. `tx-pool/src/util.rs` `check_tx_fee` — tx-pool admission
2. `verification/src/transaction_verifier.rs` `FeeCalculator::transaction_fee` — contextual block verification

Both paths reject the transaction, so it cannot be submitted to the tx-pool and any block containing it is rejected by all standard CKB nodes, even though the on-chain script would accept it.

---

### Impact Explanation

A DAO depositor who constructs a withdrawal transaction with ≥ 258 `header_deps` and a witness index whose lowest byte correctly identifies the deposit block (but whose full u64 value does not) will find their withdrawal **permanently rejected** by every standard CKB node. The funds remain locked in the DAO cell with no path to recovery through the standard node software. This is a direct analog to H-04: the accounting layer (Rust `DaoCalculator`) and the custody layer (on-chain C VM script) disagree on which header to use, causing the claim to fail.

**Impact: High** — permanent loss of access to DAO-deposited CKB capacity.

---

### Likelihood Explanation

Constructing such a transaction requires placing ≥ 258 entries in `header_deps` and choosing a witness index > 255 whose lowest byte maps to the deposit block. Normal DAO withdrawals use exactly 2 header deps (deposit block + withdraw block) and index 0 or 1, so this does not occur in the common case. However, a user building a complex multi-input DAO withdrawal with many header deps, or one following a non-standard construction path, could trigger this. An adversary could also self-craft such a transaction to permanently lock their own funds (e.g., to demonstrate the bug).

**Likelihood: Low-Medium**

---

### Recommendation

In `util/dao/src/lib.rs`, mask the parsed index to its lowest byte before using it as an array index, matching the C VM's behavior:

```rust
.and_then(|header_dep_index| {
    let effective_index = (header_dep_index & 0xFF) as usize;  // match C VM
    rtx.transaction
        .header_deps()
        .get(effective_index)
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

Alternatively, enforce that the witness index must be ≤ 255 and return `DaoError::InvalidDaoFormat` if it exceeds that bound, so the rejection reason is explicit and consistent with the C VM's constraint.

fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
