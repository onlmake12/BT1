### Title
Off-by-One in RFC0044 MMR Activation Check Causes First Activation-Epoch Block to Lack Verified Chain Root — (`tx-pool/src/block_assembler/mod.rs`)

---

### Summary

`BlockAssembler::build_extension` checks whether RFC0044 (MMR for light clients) is active using the **tip header's** epoch number instead of the **new block's** epoch number. This off-by-one causes the first block of the RFC0044 activation epoch to be assembled and accepted by full nodes without a verified MMR chain root in its extension field, creating a one-block gap in the MMR chain. The same off-by-one is present in `BlockExtensionVerifier`, making the behavior consistent but permanently incorrect at the activation boundary. The code itself acknowledges this as a bug.

---

### Finding Description

In `build_extension` (`tx-pool/src/block_assembler/mod.rs:564–581`), the MMR activation check reads:

```rust
let tip_header = snapshot.tip_header();
// The use of the epoch number of the tip here leads to an off-by-one bug,
// so be careful, it needs to be preserved for consistency reasons and not fixed directly.
let mmr_activate = snapshot
    .consensus()
    .rfc0044_active(tip_header.epoch().number());
``` [1](#0-0) 

The block being assembled is `tip + 1`. The correct epoch to test is the **new block's** epoch, not the tip's. `rfc0044_active` simply checks `target >= rfc0044_active_epoch`: [2](#0-1) 

With `RFC0044_ACTIVE_EPOCH = 8651` on mainnet: [3](#0-2) 

When the tip is the **last block of epoch 8650**, the new block is the **first block of epoch 8651**. The check evaluates `rfc0044_active(8650)` → `8650 >= 8651` → **false**. `build_extension` returns `None`, so the block template carries no MMR extension. The same off-by-one is mirrored in `BlockExtensionVerifier::verify`, which also gates on `self.parent.epoch().number()`: [4](#0-3) 

Because both assembler and verifier share the same off-by-one, full nodes reach consensus on a chain where the **first block of epoch 8651 carries no verified MMR chain root**, even though RFC0044 is nominally active from that epoch.

The verifier's `match extra_fields_count` branch for `1` (extension present) only validates the MMR hash when `mmr_active` is true. When `mmr_active` is false, any 1–96 byte extension is accepted without MMR verification: [5](#0-4) 

This means a miner who mines the first block of epoch 8651 can embed an **arbitrary or invalid MMR hash** in the extension field, and every full node will accept the block.

The analog to the external report is direct: in the Rubicon bug, the last borrowed asset acquired inside the loop was never collateralized because the collateralization step was only performed at the start of each loop iteration, not after the final one. Here, the MMR extension is only required/verified starting from the **second** block of the activation epoch, not the first — the "last step" of the activation transition is skipped.

---

### Impact Explanation

1. **One-block MMR gap**: The first block of epoch 8651 (mainnet) is permanently committed to the canonical chain without a verified MMR chain root. The MMR chain effectively starts one block later than RFC0044 intends.
2. **Unverified extension accepted**: A miner can include an arbitrary 1–96 byte payload (including a crafted, invalid MMR hash) in the extension of the first activation-epoch block. Full nodes accept it without checking the MMR root.
3. **Light-client confusion**: Light clients implementing RFC0044 that do not replicate the exact same off-by-one (e.g., those that check `>= activation_epoch` rather than `> activation_epoch`) will encounter a block whose extension claims an MMR root that was never validated by the network. `VerifiableHeader::is_valid` uses a strict `>` comparison to work around this: [6](#0-5) 

Light clients that do not replicate this exact workaround are vulnerable to accepting a forged chain root at the activation boundary.

---

### Likelihood Explanation

The bug is **deterministic and inevitable** at the RFC0044 activation epoch boundary. On mainnet it fires at epoch 8651; on testnet at epoch 5711. Any miner who mines the first block of that epoch can exploit it. No special privilege beyond normal mining (PoW) is required. The attacker role is "miner/block-template caller," which is explicitly in scope.

---

### Recommendation

The correct fix is to evaluate the activation check against the **new block's** epoch number, not the tip's. The assembler already computes `current_epoch` via `next_epoch_ext`: [7](#0-6) 

`build_extension` should accept `current_epoch` as a parameter and check `rfc0044_active(current_epoch.number())`. The same correction is needed in `BlockExtensionVerifier`. Because both sites must change atomically to preserve consensus, this requires a coordinated hard fork. Until then, the gap should be explicitly documented in the RFC0044 specification so that all light-client implementors replicate the strict `>` boundary check used in `VerifiableHeader::is_valid`.

---

### Proof of Concept

```
Mainnet RFC0044_ACTIVE_EPOCH = 8651

Tip = last block of epoch 8650
  tip.epoch().number() = 8650
  rfc0044_active(8650) = (8650 >= 8651) = false
  build_extension() → None   ← no MMR extension in template

New block = first block of epoch 8651
  Correct check: rfc0044_active(8651) = (8651 >= 8651) = true → extension required
  Actual check:  rfc0044_active(8650) = false → extension NOT required

A miner submits block at epoch 8651[0] with extension = <arbitrary 32 bytes>:
  BlockExtensionVerifier: mmr_active = rfc0044_active(parent.epoch().number())
                        = rfc0044_active(8650) = false
  → extension length/emptiness checked, MMR hash NOT verified → block ACCEPTED

Result: canonical chain contains one block with an unverified (potentially forged)
        MMR chain root hash at the RFC0044 activation boundary.
```

    pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
        let tip_header = snapshot.tip_header();
        // The use of the epoch number of the tip here leads to an off-by-one bug,
        // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
        let mmr_activate = snapshot
            .consensus()
            .rfc0044_active(tip_header.epoch().number());
        if mmr_activate {
            let chain_root = snapshot
                .chain_root_mmr(tip_header.number())
                .get_root()
                .map_err(|e| InternalErrorKind::MMR.other(e))?;
            let bytes = chain_root.calc_mmr_hash().as_bytes().into();
            Ok(Some(bytes))
        } else {
            Ok(None)
        }
    }
