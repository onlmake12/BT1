### Title
Unbounded Memory Growth in `pending_get_block_proposals` DashMap via Repeated `GetBlockProposal` P2P Messages — (`File: sync/src/types/mod.rs`)

---

### Summary

A malicious peer can cause unbounded memory growth in a CKB full node by repeatedly sending `GetBlockProposal` relay messages containing unique, non-existent proposal IDs. These IDs are inserted into the `pending_get_block_proposals` `DashMap` in `SyncState` without any total-size cap, accumulating between timer-driven drain cycles and causing a Denial-of-Service condition.

---

### Finding Description

The `SyncState` struct holds a `DashMap` field:

```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
``` [1](#0-0) 

This map is populated by `insert_get_block_proposals`, which is called from `GetBlockProposalProcess::execute` for every proposal ID that is **not found** in the local tx pool:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [2](#0-1) 

The only per-message guard in `GetBlockProposalProcess::execute` is a count check against `max_block_proposals_limit * max_uncles_num` (typically 1500 × 2 = 3000):

```rust
let limit = shared.consensus().max_block_proposals_limit()
    * (shared.consensus().max_uncles_num() as u64);
if message_len as u64 > limit {
    return StatusCode::ProtocolMessageIsMalformed...
}
``` [3](#0-2) 

After this check, proposals absent from the tx pool are inserted into the DashMap with **no check on the DashMap's total accumulated size**:

```rust
let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
    .into_iter()
    .filter(|short_id| !fetched_transactions.contains_key(short_id))
    .collect();

self.relayer.shared().state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [4](#0-3) 

The map is only drained in `prune_tx_proposal_request`, which is called from the relayer's periodic notify handler:

```rust
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
``` [5](#0-4) 

Between two consecutive timer ticks, a peer can send an unbounded number of `GetBlockProposal` messages, each carrying up to 3000 unique, attacker-crafted `ProposalShortId` values (10 bytes each) that do not exist in the tx pool. Every such ID is inserted into the DashMap, growing it without limit until the next drain.

There is no per-peer rate limit on `GetBlockProposal` messages in the relayer's `received` handler: [6](#0-5) 

The CKB project's own CHANGELOG acknowledges this class of issue: analogous memory bloat bugs in `inflight_proposals` (#3093) and `pending_compact_blocks` (#3110) were fixed in v0.101.0, but `pending_get_block_proposals` was not addressed. [7](#0-6) 

---

### Impact Explanation

An attacker controlling one or more peers can continuously send `GetBlockProposal` messages with unique, non-existent `ProposalShortId` values. Each message inserts up to 3000 entries (each ~50 bytes including the `HashSet` overhead) into the `pending_get_block_proposals` DashMap. With no total-size cap and no per-peer rate limit on this message type, the DashMap grows without bound between timer ticks, consuming heap memory at a rate proportional to the attacker's message throughput. This can exhaust available memory, causing the node process to be killed by the OS OOM killer or become unresponsive — a full Denial-of-Service.

---

### Likelihood Explanation

The attack requires only a single connected peer (inbound or outbound). No privileged role, key material, or majority hashpower is needed. `ProposalShortId` is 10 bytes and can be set to arbitrary values; the attacker does not need to know any real proposal IDs. The per-message limit of 3000 is generous, and there is no rate limit on how many `GetBlockProposal` messages a peer may send per second. The attack is straightforward to automate and sustain.

---

### Recommendation

1. **Add a total-size cap** to `pending_get_block_proposals`: reject or drop insertions once the DashMap exceeds a configurable maximum entry count (e.g., `max_block_proposals_limit * max_uncles_num * MAX_PEERS`).
2. **Add per-peer rate limiting** for `GetBlockProposal` messages in the relayer's `received` handler, analogous to the rate limiter already present in the hole-punching protocol.
3. **Ban peers** that repeatedly send proposals not found in the tx pool, consistent with the existing `BAD_MESSAGE_BAN_TIME` pattern used elsewhere in the sync/relay protocols.

---

### Proof of Concept

```
Attacker peer connects to victim CKB node (inbound or outbound).

Loop indefinitely:
  Craft a RelayMessage::GetBlockProposal with 3000 unique, random ProposalShortId values
  (10 random bytes each, guaranteed not to exist in the victim's tx pool).
  Send the message over the established P2P session.

Effect:
  Each iteration inserts up to 3000 entries into pending_get_block_proposals DashMap.
  The DashMap is only drained on the relayer's notify timer tick.
  Between ticks, memory grows at ~150 KB per message sent.
  Sustained at network speed, this exhausts the node's heap memory,
  triggering OOM or severe performance degradation (DoS).
```

The entry path is:

`P2P receive` → `Relayer::received` → `GetBlockProposalProcess::execute` → `insert_get_block_proposals` → unbounded `pending_get_block_proposals` DashMap growth. [8](#0-7) [2](#0-1)

    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        let message_len = self.message.proposals().len();
        {
            // The block proposal request is separate from uncles,
            // so here the limit is only used to calculate the maximum value of uncles
            let limit = shared.consensus().max_block_proposals_limit()
                * (shared.consensus().max_uncles_num() as u64);
            if message_len as u64 > limit {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "GetBlockProposal proposals count({message_len}) > consensus max_block_proposals_limit({limit})"
                ));
            }
        }

        let proposals: HashSet<packed::ProposalShortId> =
            self.message.proposals().to_entity().into_iter().collect();

        if proposals.len() != message_len {
            return StatusCode::RequestDuplicate.with_context("Request duplicate proposal");
        }

        let fetched_transactions = {
            let tx_pool = self.relayer.shared.shared().tx_pool_controller();
            let fetch_txs = tx_pool.fetch_txs(proposals.clone()).await;
            if let Err(e) = fetch_txs {
                debug_target!(
                    crate::LOG_TARGET_RELAY,
                    "relayer tx_pool_controller send fetch_txs error: {:?}",
                    e
                );
                return Status::ok();
            }
            fetch_txs.unwrap()
        };
        // Transactions that do not exist on this node
        let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
            .into_iter()
            .filter(|short_id| !fetched_transactions.contains_key(short_id))
            .collect();

        // Cache request, try process on timer
        self.relayer
            .shared()
            .state()
            .insert_get_block_proposals(self.peer, not_exist_proposals);
