### Title
Unbounded `read_to_end` on character/block device in `read_secret_key` causes OOM crash — (`util/app-config/src/configs/network.rs`)

### Summary
`read_secret_key` opens any path supplied via `--secret-path` and calls `read_to_end` into an unbounded heap buffer with no file-type check and no size limit. Passing a character device such as `/dev/urandom` or `/dev/zero` causes the process to allocate memory until it is killed by the OOM killer, rather than returning an error.

### Finding Description
In `util/app-config/src/configs/network.rs`, `read_secret_key` (line 288) opens the caller-supplied path with `fs::File::open`, performs only a Unix permission-mode warning, then executes:

```rust
let mut buf = Vec::new();
file.read_to_end(&mut buf)...
```

No check is made on `file.metadata()?.file_type()` to confirm the path is a regular file, and no maximum read size is enforced. For a character device, `read_to_end` loops indefinitely, growing `buf` until virtual memory is exhausted.

The CLI entry point `Setup::peer_id` in `ckb-bin/src/setup.rs` (line 527) passes the raw `--secret-path` string directly to this function with no pre-validation.

### Impact Explanation
An unprivileged local user can crash the `ckb` process (OOM kill) with a single command. Under a tight `ulimit -v`, the crash is near-instant. Without a limit, the process will consume all available memory before being killed, potentially starving co-located processes.

### Likelihood Explanation
The attack requires only local shell access and knowledge of the `ckb peer-id from-secret` subcommand. No privileges, no keys, and no configuration are needed. The path is fully attacker-controlled.

### Recommendation
Before calling `read_to_end`, validate that the opened file is a regular file and that its reported size is exactly 32 bytes:

```rust
let meta = file.metadata()?;
if !meta.file_type().is_file() || meta.len() != 32 {
    return Err(Error::new(ErrorKind::InvalidData, "invalid secret key data"));
}
```

Alternatively, replace `read_to_end` with a fixed-size read (`file.read_exact(&mut [0u8; 32])`) so the buffer is stack-allocated and bounded by construction.
