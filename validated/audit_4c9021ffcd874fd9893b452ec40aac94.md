Audit Report

## Title
Missing Count Sanity Check on `RelayTransactions` Message Allows Unbounded Deserialization Work — (`sync/src/relayer/transactions_process.rs`)

## Summary

`TransactionsProcess::execute()` performs full molecule deserialization (`.to_entity().into_view()`, including hash computation) on every transaction in a `RelayTransactions` message before any filtering occurs, with no upper-bound check on transaction count. Every sibling relay handler enforces `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) before doing any work. Any peer completing the standard P2P handshake can exploit this to force O(n) full deserialization work per message at the rate-limiter ceiling, degrading node availability.

## Finding Description

In `sync/src/relayer/mod.rs`, the `RelayTransactions` branch passes molecule-validated data directly to `TransactionsProcess::new(...).execute()` with no count guard: [1](#0-0) 

Inside `TransactionsProcess::execute()`, the iterator chain calls `.map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))` — full deserialization including hash computation — on every transaction before `.filter()` is evaluated. Because Rust iterators are lazy and driven element-by-element by `.collect()`, `to_entity().into_view()` is invoked for every transaction in the message regardless of whether it passes the filter: [2](#0-1) 

There is no count check anywhere in `execute()`. The constant is defined at: [3](#0-2) 

The rate limiter is set to 30 req/sec per `(peer, message_type)` key: [4](#0-3) 

All sibling handlers enforce the count bound before any work. `TransactionHashesProcess` checks at: [5](#0-4) 

`GetTransactionsProcess` checks at: [6](#0-5) 

`GetBlockTransactionsProcess` checks at: [7](#0-6) 

`BlockProposalProcess` checks at: [8](#0-7) 

`TransactionsProcess` is the sole handler in this dispatch loop with no such guard. [9](#0-8) 

## Impact Explanation

A single connected peer sending 30 `RelayTransactions` messages per second, each containing up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767) minimal transactions, forces ~983,010 full transaction deserializations (including hash computation) per second on the receiving node. The filter discards nearly all of them — the deserialization work is entirely wasted. With multiple peers this scales linearly. The sustained CPU pressure degrades the node's ability to process legitimate blocks and transactions. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — any peer that completes the P2P handshake can trigger this at zero marginal cost beyond the rate-limiter ceiling.

## Likelihood Explanation

No privilege, hashpower, or social engineering is required. Any peer completing the standard P2P handshake can send `RelayTransactions` messages. Constructing a molecule-valid message with maximal minimal-size transactions is trivially scriptable. The rate limiter at 30 req/sec provides partial mitigation but does not eliminate the attack surface; multiple peers multiply the effect linearly.

## Recommendation

Add a count check at the top of `TransactionsProcess::execute()`, consistent with all sibling handlers:

```rust
pub fn execute(self) -> Status {
    let tx_count = self.message.transactions().len();
    if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
        return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
            "Transactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})"
        ));
    }
    // ... existing logic
}
```

Import `MAX_RELAY_TXS_NUM_PER_BATCH` from `crate::relayer` as the sibling handlers do.

## Proof of Concept

1. Complete the CKB P2P handshake with a target node to obtain a valid peer connection.
2. Construct a molecule-valid `RelayTransactions` message containing 32,767 `RelayTransaction` entries, each with a minimal valid transaction body (raw bytes, no valid signature required — `check_data()` only validates molecule encoding, not transaction validity).
3. Send this message at 30 messages/second (the rate-limiter ceiling for this `(peer, message_type)` key).
4. The target node executes `check_data()` (O(n) molecule validation) followed by `TransactionsProcess::execute()` which calls `to_entity().into_view()` on all 32,767 entries before the filter discards them.
5. Observe elevated CPU usage on the target node proportional to the transaction count per message. With multiple attacker peers the CPU load scales linearly with no additional protocol cost.

```rust
    pub fn execute(self) -> Status {
        let shared_state = self.relayer.shared().state();
        let txs: Vec<(TransactionView, Cycle)> = {
            // ignore the tx if it's already known or it has never been requested before
            let mut tx_filter = shared_state.tx_filter();
            tx_filter.remove_expired();
            let unknown_tx_hashes = shared_state.unknown_tx_hashes();

            self.message
                .transactions()
                .iter()
                .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
                .filter(|(tx, _)| {
                    !tx_filter.contains(&tx.hash())
                        && unknown_tx_hashes
                            .get_priority(&tx.hash())
                            .map(|priority| priority.requesting_peer() == Some(self.peer))
                            .unwrap_or_default()
                })
                .collect()
        };
```
