### Title
Unbounded Script Execution in `estimate_cycles` RPC Enables CPU Exhaustion Without Fee Payment — (File: `rpc/src/module/chain.rs`)

---

### Summary

The `estimate_cycles` RPC method (and its deprecated alias `dry_run_transaction`) executes CKB-VM scripts synchronously on the RPC handler thread up to `consensus.max_block_cycles` — the block-level cycle ceiling — without requiring any on-chain fee, without a per-call cycle cap lower than the block maximum, and without any per-method rate limiting. An unprivileged RPC caller can repeatedly invoke this endpoint with maximum-cycle scripts to exhaust node CPU, blocking the RPC thread pool and degrading or denying service to all other RPC consumers.

---

### Finding Description

`CyclesEstimator::run` in `rpc/src/module/chain.rs` resolves and fully executes the supplied transaction's lock and type scripts using `ScriptVerifier::verify(max_cycles)`, where `max_cycles` is set to `consensus.max_block_cycles` — the consensus-level block cycle limit, which is far larger than the per-transaction pool cap `max_tx_verify_cycles` (default 70 000 000 cycles, configured in `ckb.toml`).

```rust
// rpc/src/module/chain.rs  ~line 2375-2389
pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
    let snapshot = self.shared.cloned_snapshot();
    let consensus = snapshot.cloned_consensus();
    match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
        Ok(resolved) => {
            let max_cycles = consensus.max_block_cycles;   // ← block-level limit, not tx-pool limit
            ...
            ScriptVerifier::new(Arc::new(resolved), snapshot.as_data_loader(),
                                consensus, Arc::new(tx_env))
                .verify(max_cycles)                        // ← blocking, synchronous
        }
    }
}
```

The call is **synchronous and blocking** on the RPC handler thread. No fee is deducted, no proof-of-work is required, and no per-call rate limiter exists for this method. The `rpc_batch_limit` configuration option is commented out and disabled by default:

```toml
# resource/ckb.toml ~line 207-208
# By default there is no limitation on the size of batch request size
# rpc_batch_limit = 2000
```

This means a single HTTP request can carry an unbounded JSON-RPC batch of `estimate_cycles` calls, each independently running up to `max_block_cycles` cycles of VM execution.

The tx-pool path, by contrast, enforces `max_tx_verify_cycles` (70 M cycles) as an upper bound per transaction before script execution begins, and requires the caller to pay `min_fee_rate` (1 000 shannons/KB). `estimate_cycles` bypasses both guards entirely.

The `estimate_cycles` RPC is part of the `Chain` module, which is enabled in the default configuration:

```toml
# resource/ckb.toml ~line 190
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

The default listen address is `127.0.0.1:8114`, placing this squarely within the "supported local CLI/RPC user" attacker role listed in the bounty scope.

---

### Impact Explanation

An attacker with RPC access (local user, or any caller if the operator exposes the RPC publicly) can:

1. Craft a transaction whose lock/type script loops for the maximum allowed cycles.
2. Send a batch of `estimate_cycles` calls in a single HTTP request (no `rpc_batch_limit` by default).
3. Each call blocks an RPC handler thread for the full duration of VM execution.
4. With enough concurrent requests, all RPC handler threads are saturated.
5. Legitimate RPC calls (`send_transaction`, `get_block`, `get_transaction`, etc.) queue up and time out.
6. The node becomes effectively unresponsive to all RPC consumers, including the operator's own tooling and monitoring.

The cost asymmetry mirrors the Across vulnerability exactly: the attacker pays **zero on-chain fee** and sends a trivially small HTTP payload, while the node performs **orders of magnitude more CPU work** (up to `max_block_cycles` per call, vs. the 70 M cycle tx-pool cap).

---

### Likelihood Explanation

- The `Chain` RPC module is enabled by default and `estimate_cycles` is a documented, stable API.
- Any process running on the same host as the node (a common deployment pattern for miners, indexers, and dApps) qualifies as an unprivileged RPC caller.
- Operators who expose the RPC on a non-loopback address (e.g., for remote tooling) expose this to the network.
- No special knowledge beyond the public RPC documentation is required to exploit this.
- The attack requires no on-chain funds, no mining power, and no peer connection.

---

### Recommendation

1. **Cap `estimate_cycles` at `max_tx_verify_cycles`** instead of `consensus.max_block_cycles`. This aligns the RPC's cycle budget with the tx-pool's own per-transaction limit and eliminates the asymmetry.
2. **Run script execution off the RPC handler thread** (e.g., via `tokio::task::spawn_blocking` or a dedicated thread pool) so that a slow `estimate_cycles` call does not block other RPC methods.
3. **Enable `rpc_batch_limit` by default** (e.g., 100) to prevent batch amplification.
4. **Add per-IP or per-connection rate limiting** for computationally expensive RPC methods.

        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();

                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,

                        cycles: cycles.into(),

                        err,
                    )),
                }
            }
