Audit Report

## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Computed Values After In-Flight Evictions — (`File: tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` snapshots the post-add totals before calling `check_and_record_ancestors`, which can evict existing pool entries and correctly decrement `self.total_tx_size` via `update_stat_for_remove_tx`. The final two lines of `add_entry` then unconditionally overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale pre-eviction snapshot, permanently inflating the pool-size tracker. The inflated tracker causes `limit_size` to evict legitimate transactions that should remain in the pool, and the condition is repeatably triggerable by any unprivileged caller.

## Finding Description

`PoolMap` maintains running totals at [1](#0-0) 

`add_entry` computes the post-add totals via `updated_stat_for_add_tx` and stores them in locals before any mutations occur: [2](#0-1) 

`updated_stat_for_add_tx` simply adds the new entry's size/cycles to the current running totals and returns the result — it does not modify `self`: [3](#0-2) 

`check_and_record_ancestors` is then called and, when `ancestors_count > max_ancestors_count` but reducible by evicting cell-dep parents, calls `remove_entry_and_descendants` in a loop: [4](#0-3) 

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` for each evicted entry: [5](#0-4) [6](#0-5) 

After `check_and_record_ancestors` returns, `add_entry` unconditionally overwrites `self.total_tx_size` and `self.total_tx_cycles` with the stale locals: [7](#0-6) 

The stale locals were computed before any evictions, so they do not reflect the decrements applied during step B. The result is:

```
self.total_tx_size  == old_total + new_entry_size          (stale)
actual pool size    == old_total + new_entry_size - Σ(evicted_sizes)
```

No existing guard prevents this: `update_stat_for_remove_tx` correctly updates `self.total_tx_size` during eviction, but that update is immediately discarded by the overwrite.

## Impact Explanation

`total_tx_size` is the sole metric used by `limit_size` to enforce `max_tx_pool_size`: [8](#0-7) 

With an inflated tracker, `limit_size` believes the pool is over-capacity when it is not, and evicts legitimate high-fee-rate transactions. Each eviction correctly decrements the tracker, so the inflation is consumed — but only after an excess number of valid transactions have been expelled. An attacker can repeat the trigger to keep the tracker perpetually inflated, effectively preventing honest transactions from remaining in the pool. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**.

## Likelihood Explanation

The trigger is reachable by any unprivileged `send_transaction` caller with no special privileges or victim interaction required:

1. Submit N > `max_ancestors_count` (default 25, configurable up to 1000) transactions each referencing the same in-pool cell dep output.
2. Submit one transaction that **consumes** that cell dep output.

Step 2 enters `check_and_record_ancestors`, finds `ancestors_count > max_ancestors_count`, enters the eviction branch at line 603, and removes `(N − max_ancestors_count + 1)` entries — inflating `total_tx_size` by their combined sizes. The integration test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` demonstrates this exact flow is a supported, exercised code path with 2000 cell-dep transactions and 1002 evictions. The attack is cheap (small transactions suffice), repeatable, and compounds across rounds.

## Recommendation

Move the stat-update commit to **after** `check_and_record_ancestors` completes, incorporating any evictions. Replace the pre-computed snapshot pattern with a post-eviction delta:

```rust
// After check_and_record_ancestors returns `evicts`:
let evicted_size:   usize = evicts.iter().map(|e| e.size).sum();
let evicted_cycles: Cycle = evicts.iter().map(|e| e.cycles).sum();
// Now apply the net change: +new_entry, -evicted
self.total_tx_size = self.total_tx_size
    .saturating_add(entry.size)
    .saturating_sub(evicted_size);
self.total_tx_cycles = self.total_tx_cycles
    .saturating_add(entry.cycles)
    .saturating_sub(evicted_cycles);
```

Remove the pre-computed `(total_tx_size, total_tx_cycles)` locals and the `updated_stat_for_add_tx` call from the top of `add_entry` (overflow checking can be re-added inline or via a separate guard before insertion). Alternatively, call `recompute_total_stat` after `check_and_record_ancestors`, consistent with the fallback already used in `update_stat_for_remove_tx`. [9](#0-8) 

## Proof of Concept

Using `max_ancestors_count = 25`, `max_tx_pool_size = 180 MB`, entry size S:

1. Submit 30 transactions T₁…T₃₀, each referencing cell dep output `X` (an in-pool UTXO). `self.total_tx_size = 30S`.
2. Submit T_consume spending output `X`.
3. `add_entry(T_consume)` executes:
   - **(A)** `total_tx_size_local = 30S + S_consume` (snapshot, line 210)
   - **(B)** `check_and_record_ancestors` finds 31 ancestors > 25; evicts T₁…T₆ via `remove_entry_and_descendants`. Each calls `update_stat_for_remove_tx` → `self.total_tx_size = 24S` (line 247)
   - **(C)** `self.total_tx_size = total_tx_size_local = 30S + S_consume` (overwrite, line 218)
4. Actual pool size = `24S + S_consume`. Tracker = `30S + S_consume`. **Inflation = 6S**.
5. `limit_size` is called; it evicts 6S worth of legitimate transactions that should have remained.
6. Repeat from step 1 to accumulate further inflation.

This matches the flow exercised by `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` (lines 70–127), which submits 2000 cell-dep transactions and verifies 1002 are evicted — confirming the eviction path is reachable and functional.
