### Title
Bounceable Response Message to User-Supplied `target_address` Permanently Locks Excess TON on Bounce — (File: `target_chains/ton/contracts/contracts/Pyth.fc`)

---

### Summary

`send_price_feeds_response` in `Pyth.fc` sends the price-feed callback to a caller-supplied `target_address` using the bounceable flag (`0x18`). Per TON protocol, a bounceable message sent to a non-existent (uninitialized) address bounces back. The Pyth contract's `recv_internal` unconditionally discards every bounced message, so the excess TON forwarded via `SENDMODE::CARRY_ALL_BALANCE` is permanently absorbed into the contract balance with no refund path.

---

### Finding Description

**TON message flag `0x18` decoded (6-bit prefix):**

```
0x18 = 011000
       │││
       ││└─ bounced  = 0
       │└── bounce   = 1  ← BOUNCEABLE
       └─── ihr_disabled = 1
```

`0x18` is **bounceable**. The inline comment `";; nobounce"` in `error_handling.fc` is incorrect and misleading.

**Vulnerable call site — `send_price_feeds_response`:** [1](#0-0) 

```
raw_reserve(update_fee, RESERVE_MODE::INCREASE_BY_ORIGINAL_BALANCE);  // keeps fee
send_raw_message(
    begin_cell()
        .store_uint(0x18, 6)          // ← bounceable
        .store_slice(target_address)  // ← fully attacker-controlled
        .store_coins(0)
        ...
    .end_cell(),
    SENDMODE::CARRY_ALL_BALANCE);     // sends all balance above reserve
```

`target_address` is read directly from the incoming message body with no existence check: [2](#0-1) [3](#0-2) 

**Bounce handler — `recv_internal`:** [4](#0-3) 

```
if (flags & 1) { ;; ignore all bounced messages
    return ();
}
```

Every bounced message is silently dropped. There is no refund logic, no event emission, and no state rollback.

**`RESERVE_MODE::INCREASE_BY_ORIGINAL_BALANCE = 4`** means the contract retains `original_balance + update_fee`; everything above that is forwarded to `target_address` via `CARRY_ALL_BALANCE`. When the message bounces, the returned value is simply added to the contract balance and the `return ()` path discards it. [5](#0-4) 

---

### Impact Explanation

Any user who calls `OP_PARSE_PRICE_FEED_UPDATES` or `OP_PARSE_UNIQUE_PRICE_FEED_UPDATES` with a `target_address` that is uninitialized (zero balance, no code) will:

1. Pay the update fee (correctly reserved).
2. Have all excess TON forwarded to `target_address` via a bounceable message.
3. Receive a bounce back to the Pyth contract.
4. Have that bounced value silently discarded — **permanently lost**.
5. Never receive the price-feed response.

The funds accumulate in the Pyth contract with no on-chain recovery mechanism. This is a direct, permanent loss of user funds triggered by a single unprivileged transaction.

---

### Likelihood Explanation

The scenario is realistic and common:

- Integrators routinely call `OP_PARSE_PRICE_FEED_UPDATES` from a freshly deployed contract before it has been initialized or funded.
- The `target_address` field is explicitly designed to be a different address from the sender (e.g., a callback contract), making uninitialized targets a normal use case.
- No privileged access is required; any wallet can trigger this with a standard internal message.
- The developer comment at lines 355–357 of `Pyth.fc` acknowledges the bounce risk but incorrectly frames it as an integrator responsibility rather than a protocol-level fund-loss bug. [6](#0-5) 

---

### Recommendation

1. **Change `0x18` to `0x10`** in `send_price_feeds_response` to send a non-bounceable message to `target_address`. Non-bounceable messages to non-existent addresses do not bounce; the TON is used to create the account, which is the correct TON-native behavior for callback delivery.

2. **Alternatively**, implement a proper bounce handler in `recv_internal` that identifies the bounced op-code and refunds the original `sender_address` stored in the response cell.

3. **Fix the misleading comment** `";; nobounce"` in `error_handling.fc` — `0x18` is bounceable, not non-bounceable. [7](#0-6) 

---

### Proof of Concept

1. Deploy a fresh TON wallet `W` (sender).
2. Deploy a new contract address `T` that has **not yet been initialized** (zero balance, no code deployed).
3. From `W`, send an internal message to the Pyth contract with:
   - `op = OP_PARSE_PRICE_FEED_UPDATES`
   - valid `update_data` for one price feed
   - `target_address = T`
   - `msg_value` = update fee + 1 TON excess
4. Observe:
   - Pyth contract processes the update, reserves `update_fee`, sends `~1 TON` to `T` with bounceable flag.
   - TON validator bounces the message back to Pyth (destination `T` does not exist).
   - Pyth `recv_internal` hits `flags & 1` → `return ()`.
   - `W` never receives a refund. `T` is never created. The `~1 TON` is absorbed into Pyth's balance.
5. Repeat to drain arbitrary amounts of user TON into the Pyth contract. [8](#0-7) [4](#0-3)
