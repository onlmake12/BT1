### Title
Denial of Service via O(n) Linear Search Over `trusted_signers` Vector in `verify_message` - (File: `lazer/contracts/aptos/sources/pyth_lazer.move`)

### Summary

The `pyth_lazer::verify_message` function (and its entry-function wrapper `verify_message_with_funder`) performs an O(n) linear scan over a `vector<TrustedSignerInfo>` to locate a matching trusted signer pubkey. As the number of trusted signers grows, the gas cost of every user-facing call to `verify_message` grows proportionally. If the list grows large enough, the Aptos gas limit will be reached, permanently bricking the function for all callers.

### Finding Description

`Storage.trusted_signers` is declared as a plain Move `vector`: [1](#0-0) 

In `verify_message`, the code iterates over every element of this vector until it finds a pubkey match: [2](#0-1) 

`update_trusted_signer` performs the same O(n) scan before inserting, updating, or removing a signer: [3](#0-2) 

Both `verify_message` (public, callable by anyone with a fee coin) and `verify_message_with_funder` (public entry, callable by any account) are unprivileged entry points: [4](#0-3) [5](#0-4) 

The worst-case scan occurs when the caller supplies a `trusted_signer` pubkey that is not in the list (or is the last element), forcing a full traversal of all N entries.

### Impact Explanation

Every Lazer price update on Aptos goes through `verify_message`. As the number of registered trusted signers grows (a normal operational action by the admin), the per-call gas cost of `verify_message` increases linearly. Once the list is large enough, the Aptos gas limit is hit and **no user can successfully call `verify_message` or `verify_message_with_funder`**, making Pyth Lazer price updates on Aptos completely unavailable. This is a permanent DoS of the core update path until a contract upgrade is performed.

### Likelihood Explanation

The admin controls the size of `trusted_signers`. However:
- Adding trusted signers is a routine operational action (key rotation, adding new data providers).
- The vector is never pruned automatically; expired signers remain in the list and still cost gas to scan.
- A caller supplying an invalid/unknown pubkey always pays the full O(n) scan cost with no short-circuit, making every failed verification maximally expensive.
- The Aptos gas model charges per-instruction; a vector scan of even a few hundred entries can become prohibitively expensive.

### Recommendation

Replace `trusted_signers: vector<TrustedSignerInfo>` with `aptos_std::table::Table<vector<u8>, TrustedSignerInfo>` (keyed by pubkey). Table lookups on Aptos are O(1) and do not grow in cost as the number of entries increases. This is the same fix applied in the referenced Blocto audit (replacing `SimpleMap` with `Table`).

```move
// Before
struct Storage has key {
    treasury: address,
    single_update_fee: u64,
    trusted_signers: vector<TrustedSignerInfo>
}

// After
use aptos_std::table::{Self, Table};

struct Storage has key {
    treasury: address,
    single_update_fee: u64,
    trusted_signers: Table<vector<u8>, TrustedSignerInfo>  // keyed by pubkey
}
```

All three linear scans in `verify_message` and `update_trusted_signer` would then become O(1) `table::contains` / `table::borrow` / `table::add` / `table::remove` calls.

### Proof of Concept

1. Admin calls `update_trusted_signer` repeatedly to register N trusted signers (legitimate operational use, e.g. key rotation over time, with expired entries never removed).
2. Any user calls `verify_message_with_funder` with a valid or invalid `trusted_signer` pubkey.
3. The Move VM executes the `while (i < storage.trusted_signers.length())` loop N times.
4. At sufficiently large N, the transaction aborts with an out-of-gas error.
5. All subsequent calls to `verify_message` / `verify_message_with_funder` also abort — Pyth Lazer price updates on Aptos are permanently unavailable until a contract upgrade.

The expired-signer accumulation path is particularly realistic: `update_trusted_signer` with `expires_at > 0` pushes to the vector but never removes old entries whose `expires_at` has passed, so the list grows monotonically over time even under normal key-rotation operations. [6](#0-5)

    public fun verify_message(
        message: vector<u8>,
        signature: vector<u8>,
        trusted_signer: vector<u8>,
        fee: coin::Coin<AptosCoin>
    ) acquires Storage {
        let storage = borrow_global<Storage>(@pyth_lazer);

        // Verify fee amount
        assert!(coin::value(&fee) >= storage.single_update_fee, EINSUFFICIENT_FEE);

        // Transfer fee to treasury
        coin::deposit(storage.treasury, fee);

        // Verify signer is trusted and not expired
        let i = 0;
        let valid = false;
        while (i < storage.trusted_signers.length()) {
            let signer_info = vector::borrow(&storage.trusted_signers, (i as u64));
            if (&signer_info.pubkey == &trusted_signer
                && signer_info.expires_at > timestamp::now_seconds()) {
                valid = true;
                break
            };
            i = i + 1;
        };
        assert!(valid, EINVALID_SIGNER);

        // Verify signature
        let sig = ed25519::new_signature_from_bytes(signature);
        let pk = ed25519::new_unvalidated_public_key_from_bytes(trusted_signer);
        assert!(
            ed25519::signature_verify_strict(&sig, &pk, message),
            EINVALID_SIGNATURE
        );
    }
