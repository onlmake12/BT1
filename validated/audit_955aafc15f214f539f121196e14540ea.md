Audit Report

## Title
Unbounded JSON-RPC Batch Requests Allow Local RPC DoS via `estimate_cycles` CPU Exhaustion — (`rpc/src/server.rs`, `rpc/src/module/chain.rs`)

## Summary
`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is never initialized under the default configuration, so the batch-size guard is silently skipped. `estimate_cycles` invokes `ScriptVerifier::verify(max_block_cycles)` synchronously on a tokio worker thread with no fee, authentication, or rate limit. A caller with access to the RPC endpoint can submit an arbitrarily large batch of `estimate_cycles` calls and saturate the node's CPU, rendering its RPC unresponsive.

## Finding Description

**Root cause 1 — No default batch limit**

`JSONRPC_BATCH_LIMIT` is declared as an empty `OnceLock` and is only populated when `config.rpc_batch_limit` is `Some(...)`: [1](#0-0) [2](#0-1) 

The shipped default config leaves `rpc_batch_limit` commented out, so `JSONRPC_BATCH_LIMIT` is never set: [3](#0-2) 

The batch handler's guard is a short-circuit that only fires when `JSONRPC_BATCH_LIMIT.get()` returns `Some`. When it returns `None` (the default), the entire guard is skipped and all calls are processed unconditionally: [4](#0-3) 

**Root cause 2 — `estimate_cycles` runs full VM execution synchronously**

`CyclesEstimator::run()` uses `consensus.max_block_cycles` as the cycle cap and calls `ScriptVerifier::verify(max_cycles)` with no fee, no authentication, and no per-caller rate limit. There is no `spawn_blocking` call — the CPU-bound verifier runs directly on the tokio worker thread, blocking it for the duration: [5](#0-4) 

The `Chain` module (which includes `estimate_cycles`) is enabled by default: [6](#0-5) 

**Why the `TimeoutLayer` does not mitigate this**

A 30-second `TimeoutLayer` is applied to the HTTP server: [7](#0-6) 

For batch requests, the handler constructs a lazy `StreamBodyAs::json_array(stream)` and returns the `Response` object immediately — no batch calls are executed before the function returns: [8](#0-7) 

The `TimeoutLayer` wraps the `Service::call` future (i.e., the `handle_jsonrpc` async fn), which completes near-instantly for batch requests because the `Response` is constructed before any stream items are polled. The actual VM execution happens when the HTTP layer polls the response body, which is outside the timeout scope.

## Impact Explanation

**Note (0–500 points): Any local RPC API crash.**

The default bind address is `127.0.0.1`, so the attack surface is limited to processes on the same machine. The impact is a single-node local RPC DoS — the node's RPC becomes unresponsive for the duration of batch processing. This does not constitute network-wide congestion (the "High" impact class), as other nodes continue to operate normally. The "High" impact claim in the submission is not supported: it requires either operator misconfiguration (public RPC exposure) or a compromised local process, neither of which is a direct consequence of this vulnerability. The correct impact class is **Note**.

## Likelihood Explanation

Any local process on the same machine can trigger this with a single HTTP POST. The attacker must have previously deployed a tight-loop RISC-V script as a cell on-chain (a one-time CKB cost), after which the attack is free to repeat indefinitely. No cryptographic material or chain knowledge beyond the cell dep outpoint is required.

## Recommendation

1. **Enable `rpc_batch_limit` by default**: Set `rpc_batch_limit = 200` (or similar) in `resource/ckb.toml` so `JSONRPC_BATCH_LIMIT` is always initialized. The current opt-in model inverts the safe default.
2. **Cap `estimate_cycles` cycle budget**: Introduce a configurable `max_estimate_cycles` lower than `max_block_cycles` (e.g., matching `max_tx_verify_cycles = 70_000_000` already present in the tx pool config) to bound per-call CPU cost.
3. **Offload VM execution to a blocking thread pool**: Wrap `CyclesEstimator::run()` in `tokio::task::spawn_blocking` so that CPU-bound script verification does not block tokio worker threads.
4. **Apply timeout to streaming body**: Wrap the batch stream with a wall-clock deadline so that the `TimeoutLayer` actually bounds total batch processing time.

## Proof of Concept

1. Start a CKB node with the default `ckb.toml` (no `rpc_batch_limit`).
2. Deploy a tight-loop RISC-V script as a cell on-chain.
3. Send the following batch POST:

```python
import json, requests

expensive_tx = { ... }  # references the deployed tight-loop script cell dep

batch = [
    {"jsonrpc": "2.0", "method": "estimate_cycles", "params": [expensive_tx], "id": i}
    for i in range(5000)
]

r = requests.post("http://127.0.0.1:8114", json=batch, timeout=None)
```

4. Observe that concurrent legitimate RPC calls (`get_block_template`, `send_transaction`) time out or receive no response for the duration of batch processing, confirming tokio worker thread saturation.

        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),
                    Err(err) => Err(RPCError::custom_with_error(
                        RPCError::TransactionFailedToVerify,
                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
