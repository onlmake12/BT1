Audit Report

## Title
TOCTOU Race in `process_lonely_block` / `process_descendant` Causes Preload Thread Panic and Node Block-Processing Halt — (`chain/src/orphan_broker.rs`, `chain/src/preload_unverified_blocks_channel.rs`)

## Summary
A time-of-check/time-of-use race between the `ChainService` thread (`OrphanBroker`) and the `verify_blocks` thread (`ConsumeUnverifiedBlockProcessor`) allows a child block C to be dispatched to the preload thread after its parent P's `COLUMN_BLOCK_HEADER` entry has been deleted due to P's failed contextual verification. The preload thread calls `.expect("parent header stored")` on the now-missing header, panics unconditionally, and terminates. Because the preload thread owns the `unverified_block_tx` sender, its termination causes the verify thread to also exit, permanently halting all block processing on the node until restart.

## Finding Description

**TOCTOU window in `process_lonely_block`.** In `chain/src/orphan_broker.rs` at lines 111–118, the `ChainService` thread reads `is_pending_verify` and then unconditionally forwards C to the preload channel with no lock held across the two operations:

```rust
let parent_is_pending_verify = self.is_pending_verify.contains(&parent_hash);  // line 111
let parent_status = self.shared.get_block_status(&parent_hash);                 // line 112
if parent_is_pending_verify || parent_status.contains(BlockStatus::BLOCK_STORED) {
    self.process_descendant(lonely_block);  // line 118 — sends C to preload channel
```

`process_descendant` (lines 199–204) inserts C into `is_pending_verify` and calls `send_unverified_block`, which sends C to the preload channel via `preload_unverified_tx.send(lonely_block)` (line 168).

**Concurrent failure path on the verify thread.** When P fails contextual verification, `consume_unverified_blocks` in `chain/src/verify.rs` (lines 153–193) does, in order:
1. Calls `self.delete_unverified_block(&block)` at line 173, which calls `delete_unverified_block` in `chain/src/lib.rs` (lines 189–231) → `db_txn.delete_block(&block)` → `db_txn.commit()`, committing a RocksDB transaction that deletes `COLUMN_BLOCK_HEADER` for P's hash (`store/src/transaction.rs` line 216).
2. Calls `self.is_pending_verify.remove(&block_hash)` at line 193.

**The panic site.** The preload thread's `load_full_unverified_block_by_hash` in `chain/src/preload_unverified_blocks_channel.rs` (lines 91–96) reads P's header:

```rust
let parent_header = {
    self.shared
        .store()
        .get_block_header(&parent_hash)
        .expect("parent header stored")   // ← panics if P was deleted
};
```

`get_block_header` in `store/src/store.rs` (lines 73–91) checks an LRU cache first, then falls back to `COLUMN_BLOCK_HEADER`. The cache (`store/src/cache.rs` line 13: `headers: Mutex<LruCache<Byte32, HeaderView>>`) is size-bounded and evicts entries; `delete_block` does not invalidate cache entries. A cache miss — under load, after eviction, or on first access for C's parent — causes the DB read to return `None`, triggering the `.expect` panic.

**No `catch_unwind` in the preload thread.** The `start()` loop in `PreloadUnverifiedBlocksChannel` (`chain/src/preload_unverified_blocks_channel.rs` lines 33–51) calls `self.preload_unverified_channel(preload_unverified_block_task)` with no `catch_unwind` wrapper. Contrast: the `verify_blocks` thread wraps `consume_unverified_blocks` in `catch_unwind` at `chain/src/verify.rs` lines 86–96.

**Cascade.** When the preload thread panics, `PreloadUnverifiedBlocksChannel` is dropped, which drops `unverified_block_tx`. The verify thread's `unverified_block_rx` then returns `Err`, hitting the branch at `chain/src/verify.rs` lines 103–106:

```rust
Err(err) => {
    error!("unverified_block_rx err: {}", err);
    return;
},
```

This terminates the verify thread. The `ChainService` thread continues running but `send_unverified_block` silently discards all future blocks once the preload receiver is gone (lines 175–178 in `orphan_broker.rs`). No further blocks are processed.

## Impact Explanation

The node permanently stops processing new blocks. P2P connections remain active and RPC responds normally, making the freeze non-obvious. Recovery requires a node restart. This constitutes a complete block-processing denial-of-service targeting a single CKB node, matching **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The race window is wide: P's contextual verification (CKB-VM script execution, transaction resolution) can take hundreds of milliseconds to seconds for a block with complex scripts, while the `ChainService` thread dispatches C to the preload channel in microseconds. An attacker who mines a single PoW-valid block P containing a contextually invalid transaction (e.g., a script that always aborts, a double-spend, or an invalid capacity) and immediately sends child C can reliably hit this window. The LRU cache eviction condition is realistic under normal node load; the attacker can also force it by sending many blocks to fill the bounded header cache before the attack. Mining a single block is a real cost but does not require majority hashpower, and the attack can be attempted repeatedly against any reachable node.

## Recommendation

1. **Add `catch_unwind` in the preload thread loop** (mirroring the pattern already used in `verify_blocks` at `chain/src/verify.rs` lines 86–96) so a panic on one task does not kill the thread.
2. **Replace `.expect("parent header stored")` with graceful error handling** in `load_full_unverified_block_by_hash` (`chain/src/preload_unverified_blocks_channel.rs` lines 91–96) — if the parent header is absent, log an error and drop the task rather than panicking.
3. **Re-check parent validity before dispatch** — after the `is_pending_verify.contains` check in `process_lonely_block`, immediately re-check `get_block_status(&parent_hash)` for `BLOCK_INVALID` before calling `process_descendant`, so a concurrently-failed parent is detected before C is forwarded.

## Proof of Concept

```
1. Attacker mines block P (valid PoW, invalid contextual content — e.g., script always aborts).
2. Attacker sends P to the victim node via P2P SendBlock.
   → P passes non-contextual checks, stored in DB, inserted into is_pending_verify,
     sent to preload channel, then to verify channel.
3. Attacker immediately sends child C (parent_hash = P.hash) via P2P SendBlock.
   → C passes non-contextual checks, stored in DB.
   → OrphanBroker.process_lonely_block(C): is_pending_verify.contains(P) == true
     → process_descendant(C) → C sent to preload channel.
4. Concurrently, verify_blocks thread finishes verifying P:
   → P fails → delete_unverified_block(P) commits → COLUMN_BLOCK_HEADER[P.hash] deleted
   → is_pending_verify.remove(P)
5. Preload thread dequeues C (after any cache eviction of P's header):
   → load_full_unverified_block_by_hash(C)
   → get_block_header(P.hash) → None (cache miss + DB entry deleted)
   → .expect("parent header stored") → PANIC
6. Preload thread terminates → unverified_block_tx dropped → verify thread receives Err → returns.
7. Node tip is frozen; no further blocks are processed until restart.
```
