### Title
`PythLazer.verifyUpdate` Reverts for Contract Callers Without `receive()` When Overpaying — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` uses `transfer` to refund excess ETH to `msg.sender`. If `msg.sender` is a contract without a payable `receive()` or `fallback()` function, the `transfer` reverts, causing the entire `verifyUpdate` call to fail. This mirrors the external report's root cause exactly: a contract that forwards ETH to another contract cannot receive the refund back.

---

### Finding Description

In `PythLazer.sol`, `verifyUpdate` accepts `msg.value >= verification_fee` and refunds any excess:

```solidity
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

`transfer` forwards only 2300 gas and reverts on failure. If `msg.sender` is a smart contract (e.g., a keeper, relayer, or on-chain integration contract) that lacks a payable `receive()` or `fallback()`, the `transfer` reverts unconditionally, bubbling up to revert the entire `verifyUpdate` call.

The grep search across all production Solidity files confirms that neither `Scheduler.sol`, `Echo.sol`, `EchoUpgradeable.sol`, nor `SchedulerUpgradeable.sol` define a `receive()` or `fallback()` — only `Implementation.sol`, `ReceiverImplementation.sol`, and `Executor.sol` do (and those explicitly revert on receipt). Any of these contracts, or any third-party integration contract, that calls `verifyUpdate` with `msg.value > verification_fee` will have the call revert.

---

### Impact Explanation

A Lazer updater or relayer implemented as a smart contract (without a payable `receive()`) that sends more ETH than `verification_fee` — for example, to avoid a separate `getUpdateFee` call or to handle fee changes — will have every `verifyUpdate` call revert. This blocks Lazer price-feed verification for that caller entirely, preventing on-chain Lazer price updates from being processed.

---

### Likelihood Explanation

Lazer updaters and relayers are frequently implemented as smart contracts (keeper bots, on-chain relayer contracts). The `verification_fee` can be updated by the owner at any time; a caller that cached the fee and sends a slightly higher value will trigger the refund path. No privileged access is required — any contract caller that overpays hits this path.

---

### Recommendation

Replace `transfer` with a low-level `call` and either check the return value or silently absorb the failure (leaving excess ETH in the contract for later recovery):

```solidity
if (msg.value > verification_fee) {
    (bool ok, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    // optionally: require(ok, "refund failed");
}
```

Alternatively, require callers to send the exact fee (`require(msg.value == verification_fee, ...)`), eliminating the refund path entirely.

---

### Proof of Concept

1. Deploy a contract `RelayerContract` with no `receive()` function.
2. `RelayerContract` calls `pythLazer.verifyUpdate{value: verification_fee + 1}(update)`.
3. Inside `verifyUpdate`, the condition `msg.value > verification_fee` is true.
4. `payable(msg.sender).transfer(1)` is called — `msg.sender` is `RelayerContract` with no `receive()`.
5. `transfer` reverts → entire `verifyUpdate` reverts → Lazer price update is not processed.

This is confirmed by the existing test at `lazer/contracts/evm/test/PythLazer.t.sol` lines 65–68, which tests the overpay-and-refund path only for an EOA (`alice`). No test covers a contract caller without `receive()`. [2](#0-1) [3](#0-2)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {

    }
