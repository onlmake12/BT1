Audit Report

## Title
Unbounded Cell Iteration in `get_cells_capacity` Enables RPC-Triggered Indexer Exhaustion — (File: util/indexer/src/service.rs)

## Summary
`IndexerHandle::get_cells_capacity` accepts no `limit` parameter and never consults `self.request_limit`, relying solely on a 10-second `TimeoutIterator` as its only execution bound. In contrast, `get_cells` and `get_transactions` both enforce `self.request_limit` and terminate iteration with `.take(limit)`. An unprivileged caller can issue concurrent broad-prefix requests to hold the indexer busy for the full 10-second window per request, rendering the indexer RPC non-functional for legitimate callers during the attack.

## Finding Description
`get_cells` validates `limit > self.request_limit` at lines 216–221 and caps iteration with `.take(limit)` at line 371. `get_transactions` applies the same guard at lines 392–397 and line 674. `get_cells_capacity` (lines 686–854) takes no `limit` argument and never reads `self.request_limit` (line 170). It constructs a `TimeoutIterator` at line 720 and chains `.take_while(|(key, _value)| key.starts_with(&prefix))` at line 728 with no `.take(n)` count cap. For every matching key it performs a RocksDB point-lookup (`Key::OutPoint`) at lines 748–752, deserializes the full cell value, and applies all filter checks. `TimeoutIterator::next` (lines 55–61) only checks elapsed wall-clock time; it does not bound the number of reads or CPU work per request. With `timeout_limit` defaulting to 10 seconds (line 99), each request can hold the indexer busy for the full window. The rich-indexer variant (`AsyncRichIndexerHandle::get_cells_capacity`, lines 26–27 of `get_cells_capacity.rs`) issues `SUM(capacity)` with no `LIMIT` clause, forcing a full matching-row scan.

## Impact Explanation
The indexer RPC service becomes non-functional for legitimate callers during the attack window. A small number of concurrent `get_cells_capacity` requests with a broad prefix (e.g., secp256k1 `code_hash` with empty `args`) saturates the indexer's I/O and CPU for up to 10 seconds per request. This matches **Note (0–500 points): Any local RPC API crash**, as the indexer RPC is rendered non-functional for the duration. The node's consensus and P2P layers are unaffected.

## Likelihood Explanation
The indexer RPC is accessible on any CKB node with the indexer module enabled. No authentication is required. The secp256k1 lock `code_hash` is publicly documented and matches every standard wallet cell on mainnet. The attack requires a single well-formed JSON-RPC call with a known parameter value and is trivially repeatable from any network-connected client.

## Recommendation
1. Add a `limit: Option<Uint32>` parameter to `get_cells_capacity` and enforce `self.request_limit` as the maximum, consistent with `get_cells` and `get_transactions`.
2. Alternatively, add a hard internal `.take(self.request_limit)` cap on the iterator chain inside `get_cells_capacity` regardless of caller input, returning an error if the cap is reached.
3. For the rich-indexer variant, add a `LIMIT` clause to the SQL aggregate query or enforce a row-count guard before aggregation.
4. Retain `TimeoutIterator` as a secondary defense only.

## Proof of Concept
Send the following to a CKB node with the indexer enabled:

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [{
    "script": {
      "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
      "hash_type": "type",
      "args": "0x"
    },
    "script_type": "lock",
    "script_search_mode": "prefix"
  }]
}
```

With `args: "0x"`, the `take_while` loop at line 728 of `util/indexer/src/service.rs` iterates every cell whose lock script begins with the secp256k1 `code_hash`. Each iteration performs a RocksDB point-lookup at lines 748–752. Issuing this request from multiple concurrent connections sustains exhaustion for the full 10-second timeout window per request. The missing `.take(self.request_limit)` guard — present in `get_cells` at line 371 but absent in `get_cells_capacity` — is the root cause.
