### Title
`remove_entry_and_descendants` Pre-Deletes Links Before Ancestor Score Update, Leaving Surviving Ancestors with Permanently Inflated Descendant Weights - (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`remove_entry_and_descendants` in the CKB tx-pool pre-removes all parent/child links for every entry in the removal batch before calling `remove_entry` on each. `remove_entry` then calls `update_ancestors_index_key`, which uses those same links to find surviving ancestors and decrement their descendant-weight fields. Because the links are already gone, `calc_ancestors` returns an empty set, and no ancestor score is ever decremented. Any transaction that is a parent of the removed subtree but is **not** itself being removed retains permanently inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles` fields for the rest of its lifetime in the pool.

---

### Finding Description

`remove_entry_and_descendants` is the function used to evict a transaction and all of its in-pool descendants at once (e.g., during conflict resolution, RBF, header-dep invalidation, or detached-proposal handling). [1](#0-0) 

The function first pre-removes all links for every entry in the batch:

```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
for id in &removed_ids {
    self.remove_entry_links(id);   // ← deletes ALL link entries, including the root's link to its external parent
}
```

`remove_entry_links` for the root entry (`tx1` in the example below) does three things:

1. Removes `tx1` from its parent's (`tx0`'s) children set.
2. Removes `tx1` from its children's parents sets.
3. **Deletes `tx1`'s own link entry entirely.** [2](#0-1) 

After the pre-deletion loop, `remove_entry` is called for each id. Inside `remove_entry`, `update_ancestors_index_key` is invoked: [3](#0-2) 

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

`calc_ancestors` traverses the `links` map starting from the given id: [5](#0-4) 

Because `tx1`'s link entry was already deleted in the pre-deletion loop, `calc_ancestors(tx1)` returns an **empty set**. The loop body never executes. `tx0`'s `sub_descendant_weight` is never called.

**Concrete trace** for chain `tx0 → tx1 → tx2 → tx3`, calling `remove_entry_and_descendants(tx1)`:

| Step | Action | Effect on tx0 |
|---|---|---|
| Pre-delete loop | `remove_entry_links(tx1)` deletes tx1's link entry | tx0's children set loses tx1, but tx0's `descendants_*` fields unchanged |
| `remove_entry(tx1)` | `calc_ancestors(tx1)` → `{}` (link gone) | tx0's `descendants_count/fee/size/cycles` **not decremented** |
| `remove_entry(tx2,tx3)` | Same — links already gone | No further updates |

After the call, `tx0` still carries the descendant weight of `tx1 + tx2 + tx3` even though all three are gone from the pool.

This is the direct analog of the Solidity bug: in Solidity, a storage pointer `w` is obtained, the underlying slot is `delete`d, and `w.requestId` silently reads `0`. Here, the `links` map entry is deleted before `update_ancestors_index_key` reads it, so the function silently operates on an empty ancestor set instead of the real one.

---

### Impact Explanation

`tx0`'s `EvictKey` is computed from its `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`: [6](#0-5) 

An inflated `descendants_feerate` makes `tx0` appear more valuable than it actually is. Consequences:

1. **Incorrect eviction ordering**: When the pool is full and the eviction loop calls `next_evict_entry`, `tx0` is ranked as if it still has high-value descendants. It will be evicted later than it should be, displacing legitimately higher-value transactions.
2. **Stale RPC data**: `get_pool_tx_verbose` and similar RPC calls return the inflated `descendants_*` fields to callers, giving miners and fee-estimators incorrect data.
3. **Persistent corruption**: The inflated fields are never corrected. They remain until `tx0` itself is removed from the pool (committed, expired, or evicted).

---

### Likelihood Explanation

Every call path that removes a non-root transaction with an in-pool parent triggers this bug:

- `resolve_conflict` — triggered by any new transaction that double-spends an output already consumed by a child of `tx0`. Any unprivileged transaction sender can do this.
- `resolve_conflict_header_dep` — triggered by a block that invalidates a header dep used by a descendant of `tx0`. Any block relayer can do this.
- `remove_by_detached_proposal` — triggered during any chain reorganization.
- `check_and_record_ancestors` — triggered when a new transaction evicts a cell-ref parent. [7](#0-6) [8](#0-7) 

Transaction chains of depth ≥ 2 are common and permitted by the protocol. The bug fires on every such removal.

---

### Recommendation

Before the pre-deletion loop, capture the set of **external** ancestors of the root entry (ancestors not in `removed_ids`) and their current descendant weights. After all entries are removed, apply the decrements directly to those surviving ancestors.

A minimal fix:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));
    let removed_set: HashSet<_> = removed_ids.iter().cloned().collect();

    // Capture external ancestors BEFORE any links are deleted
    let external_ancestors: HashSet<ProposalShortId> = self
        .links
        .calc_ancestors(id)
        .into_iter()
        .filter(|a| !removed_set.contains(a))
        .collect();

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed_entries: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Decrement surviving ancestors' descendant weights
    for removed in &removed_entries {
        for anc_id in &external_ancestors {
            self.entries.modify_by_id(anc_id, |e| {
                e.inner.sub_descendant_weight(removed);
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    removed_entries
}
```

### Proof of Concept

Given pool state: `tx0 → tx1 → tx2` (tx0 is root, tx1 and tx2 are descendants).

After `add_entry` for all three, `tx0.descendants_count == 3`, `tx0.descendants_fee == fee0 + fee1 + fee2`.

Call `remove_entry_and_descendants(tx1)`:

1. `removed_ids = [tx1, tx2]`
2. Pre-delete loop removes links for tx1 and tx2 (including tx1's link to tx0).
3. `remove_entry(tx1)` → `calc_ancestors(tx1)` → `{}` → tx0 not updated.
4. `remove_entry(tx2)` → `calc_ancestors(tx2)` → `{}` → tx0 not updated.

After the call:
- `tx0.descendants_count` is still `3` (should be `1`).
- `tx0.descendants_fee` is still `fee0 + fee1 + fee2` (should be `fee0`).
- `tx0.evict_key` reflects the inflated descendant feerate.

The existing test `test_remove_entry_and_descendants` only checks that `tx2` and `tx3` are absent from the pool and that `tx1`'s descendants set is empty — it does **not** assert that `tx0`'s descendant weight fields are correctly decremented, so the bug is not caught. [9](#0-8)

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }
