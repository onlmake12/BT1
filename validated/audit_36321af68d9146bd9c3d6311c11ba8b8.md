Audit Report

## Title
Non-Blocking `try_send` for Reorg Notifications Silently Drops Updates, Leaving Tx-Pool Inconsistent - (File: `tx-pool/src/service.rs`, `chain/src/verify.rs`)

## Summary
`TxPoolController::update_tx_pool_for_reorg` uses `reorg_sender.try_send(notify)` on a bounded channel of size 512. When the channel is full, the reorg notification is silently discarded — the dropped payload is bound to `_m` and never retried. The chain store snapshot is already committed at this point, so the tx-pool permanently diverges from chain state until node restart.

## Finding Description
In `chain/src/verify.rs` at lines 383–394, `store_snapshot` commits the new chain state unconditionally, then `update_tx_pool_for_reorg` is called. On error, only an `error!` log is emitted; execution continues with no retry or rollback.

In `tx-pool/src/service.rs` at line 254, `self.reorg_sender.try_send(notify)` is the non-blocking send. On `TrySendError::Full`, `handle_try_send_error` (`tx-pool/src/error.rs` lines 38–44) extracts the full notification into `_m` and discards it, returning only an error string. The notification is permanently lost.

The channel is created at `tx-pool/src/service.rs` line 513 with `DEFAULT_CHANNEL_SIZE = 512`. The reorg receiver loop (`tx-pool/src/service.rs` lines 697–731) processes one reorg at a time sequentially. Each iteration calls `readd_detached_tx` (`tx-pool/src/process.rs` lines 878–914), which runs full CKB-VM script verification per detached transaction — making each iteration potentially slow. If 512 notifications queue before the receiver drains, all subsequent `try_send` calls fail silently.

When a notification is dropped: `remove_committed_txs` is never called for attached blocks (committed transactions remain in the pending pool); `readd_detached_tx` is never called for detached blocks (orphaned transactions are permanently lost from the mempool); the tx-pool snapshot remains stale, causing all downstream operations (block assembly, fee estimation, RBF) to operate on incorrect state.

## Impact Explanation
This is a suboptimal implementation of the CKB tx-pool state management mechanism. The tx-pool drives block template construction, transaction relay, and fee estimation. When desynchronized from committed chain state, block templates include already-committed transactions (producing invalid templates) and user transactions from detached blocks are silently dropped from the mempool. This matches the **Medium** impact category: *Suboptimal implementation of CKB state storage mechanism* (2001–10000 points).

## Likelihood Explanation
During normal operation (~8s block time) with a lightly loaded tx-pool, the receiver drains quickly and the channel rarely fills. However, during IBD catch-up (blocks committed at maximum throughput) or during repeated shallow reorgs with a loaded mempool (slow `readd_detached_tx` due to script execution), the receiver can fall behind. An adversary with moderate hashpower can deliberately trigger repeated shallow reorgs to keep the receiver occupied while the chain service continues committing blocks, eventually exhausting the 512-slot channel. The condition is reachable by an unprivileged block relayer given sufficient hashpower.

## Recommendation
Replace `reorg_sender.try_send(notify)` with a blocking `reorg_sender.send(notify).await` (or equivalent async send) so the chain service applies backpressure rather than dropping notifications. If blocking the chain service is unacceptable, use an unbounded channel for reorg notifications (reorgs are rare relative to normal block production and bounded by fork-choice rules). At minimum, implement a retry loop with a timeout and a hard failure if the tx-pool cannot be updated, rather than silently continuing with stale state.

## Proof of Concept
1. Start a CKB node with a tx-pool containing many pending transactions (to slow `readd_detached_tx` via script execution).
2. Trigger rapid block production or repeated shallow reorgs so that `update_tx_pool_for_reorg` is called faster than the reorg receiver can drain the channel.
3. After 512 queued notifications, observe `[verify block] notify update_tx_pool_for_reorg error` logged at `chain/src/verify.rs:393`.
4. Query the tx-pool via RPC: transactions from detached blocks are absent; transactions from attached blocks are still present.
5. Call `get_block_template`: the returned template contains already-committed transactions, confirming the inconsistency.
