### Title
Unprivileged Rent Theft via `close_signature_set` — Missing Payer Authority Check - (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary

The `close_signature_set` instruction allows any signer to designate themselves as `sol_destination` and receive the rent lamports from a `SignatureSet` account they did not create or fund. The original relayer/payer who funded the account receives nothing.

### Finding Description

In `close_signature_set.rs`, the `CloseSignatureSet` account context requires only that `sol_destination` is a signer — there is no constraint tying it to the original payer of the `signature_set`:

```rust
#[account(mut)]
sol_destination: Signer<'info>,
// ...
#[account(
    mut,
    close = sol_destination   // rent goes to whoever signs, not the original payer
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,
``` [1](#0-0) 

The `SignatureSet` state struct stores only `sig_verify_successes`, `message_hash`, and `guardian_set_index` — the original payer's pubkey is never recorded: [2](#0-1) 

In `verify_signatures.rs`, the `payer` funds the `signature_set` via `init_if_needed, payer = payer`, but this relationship is never persisted into the account data: [3](#0-2) 

Because `posted_vaa` and `signature_set` are both public on-chain accounts (no signer required to read them), any attacker can construct a valid `close_signature_set` call with their own key as `sol_destination`.

### Impact Explanation

An attacker monitors the chain for `PostedVaaV1` accounts. For each one, they call `close_signature_set` with themselves as `sol_destination`. The `signature_set` is closed and its rent (proportional to guardian set size — typically thousands of lamports) is transferred to the attacker. The honest relayer who paid for the account receives nothing. This is a direct, repeatable theft of SOL from every relayer using the legacy VAA verification path.

### Likelihood Explanation

The attack is fully permissionless: no special role, no leaked key, no governance access. The attacker only needs to sign a transaction. All required account addresses (`posted_vaa`, `signature_set`) are discoverable on-chain. The attack can be automated to front-run or follow every VAA post.

### Recommendation

Store the original payer's pubkey in the `SignatureSet` account at creation time, and add a constraint in `CloseSignatureSet` requiring `sol_destination` to match that stored payer:

```rust
// In SignatureSet state:
pub payer: Pubkey,

// In CloseSignatureSet constraints:
#[account(
    mut,
    close = sol_destination,
    has_one = payer @ CoreBridgeError::InvalidFeeRecipient
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,

#[account(mut, address = signature_set.payer)]
sol_destination: Signer<'info>,
```

### Proof of Concept

1. Relayer A calls `verify_signatures` (funding `signature_set` S) then `post_vaa` (creating `posted_vaa` P referencing S).
2. Attacker B observes P and S on-chain.
3. B calls `close_signature_set` with accounts `{ sol_destination: B, posted_vaa: P, signature_set: S }`.
4. The instruction passes all checks: B is a signer, P has `has_one = S`, S is a valid `AccountVariant<SignatureSet>`.
5. Anchor's `close = sol_destination` macro transfers S's rent to B.
6. Assert: B's balance increased by S's rent; A received nothing. [4](#0-3)
