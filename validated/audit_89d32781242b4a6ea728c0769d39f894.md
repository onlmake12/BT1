Having read all the relevant files, I can now trace the full attack path precisely.

**Key facts established from the code:**

1. `post_vaa_v1` is fully permissionless — `payer` is any `Signer`, and `encoded_vaa` only requires `status == ProcessingStatus::Verified` with no ownership check. [1](#0-0) 

2. `post_vaa_v1` always writes `signature_set: Default::default()` (zero pubkey) into the `PostedVaaV1` account. [2](#0-1) 

3. The `posted_vaa` PDA seed is `[PostedVaaV1::SEED_PREFIX, keccak::hash(body.as_ref())]` — deterministic from VAA body content. [3](#0-2) 

4. The legacy `post_vaa` uses the **same PDA seeds** (`[PostedVaaV1::SEED_PREFIX, signature_set.message_hash.as_ref()]`), where `message_hash` is `keccak::hashv` over the same body fields — equivalent to `keccak::hash(body.as_ref())`. [4](#0-3) 

5. `close_signature_set` enforces `has_one = signature_set` on `posted_vaa`, meaning the `signature_set` account's pubkey must equal `posted_vaa.signature_set`. If that field is the zero pubkey, the only valid `signature_set` account would be the system program — which cannot be deserialized as `Account<'info, AccountVariant<SignatureSet>>`. [5](#0-4) 

6. The comment in `close_signature_set` itself acknowledges the zero-pubkey case but provides no mechanism to handle it. [6](#0-5)

### Title
Front-running `post_vaa_v1` with zero `signature_set` permanently freezes relayer's `SignatureSet` lamports — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/post_vaa_v1.rs`)

### Summary
An unprivileged attacker can front-run a relayer's `post_vaa_v1` (or legacy `post_vaa`) call by submitting the same VAA through `post_vaa_v1` first. Because `post_vaa_v1` always writes `signature_set = Default::default()` (zero pubkey) into the `PostedVaaV1` account, and `close_signature_set` enforces `has_one = signature_set`, the relayer's pre-paid `SignatureSet` account becomes permanently unclaimable.

### Finding Description
The attack proceeds as follows:

1. **Relayer** calls legacy `verify_signatures`, paying to create a `SignatureSet` account (costs rent-exempt lamports proportional to the guardian set size).
2. **Attacker** observes the VAA content (VAAs are public; the `SignatureSet.message_hash` reveals the body hash). The attacker independently creates an `EncodedVaa` for the same VAA via `init_encoded_vaa` → `write_encoded_vaa` → `verify_encoded_vaa_v1`.
3. **Attacker** calls `post_vaa_v1` with their `EncodedVaa`. This succeeds and creates the `PostedVaaV1` PDA (seeds: `[b"PostedVAA", keccak(body)]`) with `signature_set = Pubkey::default()` (all zeros).
4. **Relayer** calls legacy `post_vaa`. This fails with an account-already-exists error because the PDA was already initialized in step 3.
5. **Relayer** attempts `close_signature_set`. This fails because `posted_vaa.signature_set == Pubkey::default()`, so `has_one = signature_set` requires passing an account at the zero pubkey — which is the system program and cannot be deserialized as `AccountVariant<SignatureSet>`. The instruction is permanently uncallable for this VAA.
6. The relayer's `SignatureSet` lamports are frozen indefinitely with no recovery path.

The root cause is the combination of:
- `post_vaa_v1` being permissionless with no ownership check on `encoded_vaa`
- `post_vaa_v1` unconditionally writing `signature_set: Default::default()` regardless of whether a `SignatureSet` was used
- `close_signature_set` using `has_one = signature_set` with no special handling for the zero-pubkey case

### Impact Explanation
The relayer's `SignatureSet` lamports (rent-exempt deposit for a guardian-set-sized account) are permanently frozen. The `SignatureSet` account cannot be closed via `close_signature_set` and no other instruction closes it. This is a targeted griefing/fund-freeze attack. While the per-attack lamport value is modest (~0.01–0.05 SOL depending on guardian set size), the attack can be repeated for every VAA a relayer processes, compounding losses over time.

### Likelihood Explanation
- VAA content is fully public (broadcast by Wormhole guardians and observable on-chain).
- Solana has no private mempool; `verify_signatures` transactions are visible before `post_vaa` is submitted.
- The attacker's cost is the rent for one `EncodedVaa` account plus transaction fees — recoverable by closing their own `EncodedVaa` afterward.
- No privileged access, leaked keys, or oracle manipulation is required.

### Recommendation
Two complementary fixes:

1. **In `close_signature_set`**: Add a guard that if `posted_vaa.signature_set == Pubkey::default()`, the instruction should either be a no-op or require no `signature_set` account at all (e.g., use `Option<Account<...>>` or a separate instruction path).

2. **In `post_vaa_v1`**: Accept an optional `signature_set` account and, if provided and valid, write its pubkey into `PostedVaaV1.signature_set` instead of `Default::default()`. This preserves the link between the `PostedVaaV1` and the `SignatureSet` that was used to verify the VAA, allowing `close_signature_set` to function correctly.

### Proof of Concept
```

// For any valid VAA V:
//   1. relayer calls verify_signatures(V) -> SignatureSet S (relayer pays rent)
//   2. attacker calls init_encoded_vaa / write / verify_encoded_vaa_v1(V) -> EncodedVaa E
//   3. attacker calls post_vaa_v1(encoded_vaa=E) -> PostedVaaV1 P with P.signature_set = 0x000...
//   4. relayer calls post_vaa(signature_set=S) -> FAILS (P already exists)
//   5. relayer calls close_signature_set(posted_vaa=P, signature_set=S) -> FAILS
```

The invariant "a relayer who paid to create a `SignatureSet` can always reclaim those lamports via `close_signature_set`" is broken by this attack path.

```rust
#[derive(Accounts)]
pub struct PostVaaV1<'info> {
    /// Payer to create the posted VAA account. This instruction allows anyone with an encoded VAA
    /// to create a posted VAA account.
    #[account(mut)]
    payer: Signer<'info>,

    /// Encoded VAA, whose body will be serialized into the posted VAA account.
    ///
    /// NOTE: This instruction handler only exists to support integrators that still rely on posted
    /// VAA accounts. While we encourage integrators to use the encoded VAA account instead, we
    /// allow a pathway to convert the encoded VAA into a posted VAA. However, the payload is
    /// restricted to 9.5KB, which is much larger than what was possible with the old implementation
    /// using the legacy post vaa instruction. The Core Bridge program will not support posting VAAs
    /// larger than this payload size.
    #[account(
        constraint = encoded_vaa.status == ProcessingStatus::Verified @ CoreBridgeError::UnverifiedVaa
    )]
    encoded_vaa: Account<'info, EncodedVaa>,

    #[account(
        init,
        payer = payer,
        space = PostedVaaV1::compute_size(
            encoded_vaa
                .as_vaa()?
                .to_v1()?
                .body()
                .payload()
                .as_ref()
                .len()
        ),
        seeds = [
            PostedVaaV1::SEED_PREFIX,
            solana_program::keccak::hash(
                encoded_vaa
                    .as_vaa()?
                    .to_v1()?
                    .body().as_ref()
            ).as_ref()
        ],
        bump,
    )]
    posted_vaa: Account<'info, LegacyAnchorized<PostedVaaV1>>,

    system_program: Program<'info, System>,
}
```
