### Title
`HeaderAcceptor::accept()` Missing `BLOCK_INVALID` Guard Allows Untrusted Peer to Overwrite `shared_best_header` with a Known-Invalid Block's Header — (`File: sync/src/synchronizer/headers_process.rs`)

---

### Summary

`HeaderAcceptor::accept()` in `sync/src/synchronizer/headers_process.rs` contains an acknowledged `FIXME` comment noting that it does not return early when the block status is `BLOCK_INVALID`. An unprivileged P2P peer can send a `SendHeaders` message containing the header of a block that the local node has already marked `BLOCK_INVALID` (e.g., a block whose body failed contextual verification). Because the header still carries valid PoW, it passes all checks inside `accept()` and causes `insert_valid_header` to be called, which unconditionally overwrites the node's global `shared_best_header` if the header's total difficulty is higher. This corrupts the sync-state machine without the attacker needing any privileged access.

---

### Finding Description

**Root cause — missing guard in `HeaderAcceptor::accept()`** [1](#0-0) 

The function reads the block status and returns early only when the status contains `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    ...
    return result;
}
```

`BLOCK_INVALID` is a completely separate bit (`1 << 12`) from `HEADER_VALID` (`1`): [2](#0-1) 

So when `status == BLOCK_INVALID`, the early-return is **not** taken. Execution falls through to:

1. `prev_block_check` — checks only whether the **parent** is `BLOCK_INVALID`, not the block itself. [3](#0-2) 
2. `non_contextual_check` — runs `HeaderVerifier::verify()`, which includes PoW, number, epoch, and timestamp checks. A header for a block that was rejected at the **body** level (e.g., invalid transactions) still carries valid PoW and passes all of these. [4](#0-3) 
3. `version_check` — trivially passes for any version-0 header.
4. `sync_shared.insert_valid_header(self.peer, self.header)` is called. [5](#0-4) 

**State overwrite inside `insert_valid_header`**

`insert_valid_header` unconditionally inserts the header into `header_map` and then calls `may_set_shared_best_header`: [6](#0-5) 

`may_set_shared_best_header` overwrites the global `shared_best_header` whenever the incoming header's total difficulty exceeds the current value: [7](#0-6) 

`shared_best_header` is the node-wide view of the best known chain and is used to gate block-download decisions: [8](#0-7) 

**Exploit path**

1. A legitimately-mined block B is broadcast. The local node receives it, verifies the header (PoW passes), but rejects the body (e.g., a script fails). `insert_block_status(B.hash, BLOCK_INVALID)` is called. [9](#0-8) 
2. An unprivileged peer (anyone who received block B) sends a `SendHeaders` P2P message containing B's header.
3. `HeadersProcess::execute()` calls `HeaderAcceptor::accept()` for B's header. [10](#0-9) 
4. The `BLOCK_INVALID` guard is absent; all sub-checks pass; `insert_valid_header` is called.
5. `shared_best_header` is overwritten to point to B (an invalid block) if B's total difficulty exceeds the current value.

---

### Impact Explanation

- **Sync-state corruption**: `shared_best_header` now points to a known-invalid block. `min_chain_work_ready()` may return `true` prematurely, causing the node to transition out of the `MinWorkNotReach` state and begin fetching blocks along the invalid chain. [8](#0-7) 
- **Resource exhaustion / DoS**: The node wastes bandwidth and CPU requesting and re-verifying blocks on the invalid chain. A peer can repeat this continuously by re-sending the same header.
- **Incorrect RPC reporting**: `sync_state()` reports `best_known_block_number` and `best_known_block_timestamp` derived from `shared_best_header`, misleading operators and downstream tooling. [11](#0-10) 

The node's canonical chain is not corrupted (the block remains `BLOCK_INVALID` in `block_status_map`), but the sync-state machine is driven by incorrect data, which can stall or misdirect synchronization.

---

### Likelihood Explanation

- **No privilege required**: Any connected P2P peer can send a `SendHeaders` message.
- **No new mining required**: The attacker only needs the header of any block that was previously rejected at the body-verification stage on the target node. Such blocks exist naturally on the network (e.g., blocks with invalid scripts, double-spend attempts, or malformed transactions that passed header PoW).
- **Repeatable**: The attack can be repeated indefinitely because the `BLOCK_INVALID` status in `block_status_map` is never cleared by this path, but `shared_best_header` is overwritten each time.

---

### Recommendation

In `HeaderAcceptor::accept()`, add an explicit early-return for `BLOCK_INVALID` before the existing `HEADER_VALID` check, resolving the acknowledged `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
// Resolve FIXME: reject headers for blocks already known to be invalid.
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    ...
    return result;
}
```

This mirrors the guard already present in `compact_block_process.rs`: [12](#0-11) 

---

### Proof of Concept

1. Connect a custom peer to a CKB node.
2. Submit a block whose header has valid Eaglesong PoW but whose body contains an invalid transaction (e.g., a script that always fails). The node marks the block `BLOCK_INVALID`.
3. Send a `SendHeaders` P2P message containing only that block's header.
4. Observe via `sync_state` RPC that `best_known_block_number` advances to the invalid block's height, and `min_chain_work_reached` flips to `true` if the block's total difficulty exceeds the configured threshold — despite the block being permanently invalid.
