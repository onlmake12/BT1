Audit Report

## Title
Missing Sender-Identity Verification in `ConnectionRequestDeliveredProcess::execute` Allows Forged Delivery to Cancel Inflight Hole-Punch Requests and Redirect NAT Traversal to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary

`ConnectionRequestDeliveredProcess::execute` determines whether the local node is the originator of a hole-punch request by comparing the attacker-controlled `content.from` field against `self_peer_id`, without verifying that the actual delivering session (`self.peer`) corresponds to `content.to`. Any connected peer can forge a `ConnectionRequestDelivered` with `from` set to the victim's own peer ID, `to` set to any key currently in `inflight_requests`, an empty `route`, and `listen_addrs` pointing to attacker-controlled IP:port pairs. This causes the victim to silently remove the legitimate inflight entry and spawn outbound TCP connection attempts toward attacker-controlled addresses, with a successful connection establishing an unauthorized P2P session.

## Finding Description

**Root cause — no sender-identity check in the `None` branch (lines 147–179):**

When `content.route` is empty, the only guard is `self_peer_id != &content.from` (line 151). Setting `content.from = self_peer_id` bypasses this check and enters the `else` branch. There is no assertion that `self.peer` (the `PeerIndex` of the actual delivering session) equals the session ID corresponding to `content.to`. The `peer_registry` already exposes `get_key_by_peer_id` for exactly this lookup and is used in `forward_delivered` (lines 183–188), but it is never consulted here.

**Inflight entry removal (line 160):**

`inflight_requests.remove(&content.to)` permanently removes the entry keyed by the attacker-supplied `content.to`. If the entry exists, execution continues; if not, `StatusCode::Ignore` is returned — so the attacker must supply a valid key, which is trivially observable from broadcast `ConnectionRequest` messages.

**`respond_sync` sends to attacker's session (lines 226–229):**

`send_message_to(self.peer, ...)` sends the `ConnectionSync` response to the attacker's session, not to the legitimate `content.to` peer.

**`try_nat_traversal` with attacker-controlled addresses (lines 237–284):**

`listen_addrs` validation (lines 51–71) only checks that any embedded peer ID matches `content.to`; if no peer ID is present, `content.to`'s peer ID is appended. The IP:port is unrestricted. On a successful TCP connection, `control.raw_session(stream, addr, RawSessionInfo::outbound(...))` establishes a full P2P session running the Identify protocol with the attacker's server.

**Rate limiter does not prevent the attack (lines 134–145):**

`forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`. With `content.from` fixed to the victim's own peer ID and `content.to` fixed to the target key, this permits exactly 1 forged delivery per second per `(from, to)` pair — sufficient to cancel any newly inserted inflight entry as soon as it appears.

**Inflight keys are observable (mod.rs lines 224–241):**

`ConnectionRequest` messages are broadcast to a gossip subset of all connected peers. The `to` field of each broadcast is exactly the key inserted into `inflight_requests`. An attacker who is a connected peer receives these broadcasts and trivially learns valid keys.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker connected to many nodes can, at a cost of one message per second per target, continuously cancel their inflight hole-punch entries and force each victim to spawn repeated outbound TCP connection tasks toward attacker-controlled addresses. This wastes connection-slot budget, generates spurious TCP traffic, and degrades the NAT traversal layer that nodes behind NAT depend on for peer connectivity. A successful `raw_session` call also allows the attacker to establish unauthorized P2P sessions that bypass normal peer-selection and connection policies.

## Likelihood Explanation

The attacker requires only a standard inbound or outbound P2P connection to the victim. The victim's peer ID is publicly advertised. Valid `inflight_requests` keys are directly observable from broadcast `ConnectionRequest` messages received over the same connection. No privileged access, key material, or hash power is required. The attack is repeatable at 1 Hz per `(from, to)` pair and is trivially parallelizable across many victims simultaneously.

## Recommendation

Before entering the "local node is the originator" branch, verify that the actual session delivering the message (`self.peer`) corresponds to `content.to`. Use `peer_registry.get_key_by_peer_id(&content.to)` and assert the returned `SessionId` equals `self.peer`, mirroring the pattern already used in `forward_delivered` (lines 183–188). If the check fails, return `StatusCode::Ignore` (or a ban-worthy status) rather than proceeding to remove the inflight entry or spawn NAT traversal tasks.

## Proof of Concept

```
1. Victim node V has peer ID P_V and has an inflight_requests entry for peer ID P_T
   (learned by attacker A from a broadcast ConnectionRequest).
2. A is connected to V with session S_A.
3. A sends a ConnectionRequestDelivered to V with:
     from         = P_V   (victim's own peer ID)
     to           = P_T   (key in inflight_requests)
     route        = []    (empty)
     sync_route   = []
     listen_addrs = [attacker_ip:attacker_port]  (no embedded peer ID; code appends P_T)
4. V's execute():
     content.route.last() == None  → enters the None branch
     self_peer_id == content.from  → enters the else branch (line 154)
     inflight_requests.remove(&P_T) → Some(start)  (entry consumed)
     respond_sync(P_V) → sends ConnectionSync back to S_A
     try_nat_traversal(ttl, [attacker_ip:attacker_port/p2p/P_T]) → spawned
5. V makes outbound TCP connections to attacker_ip:attacker_port.
   If attacker's server accepts, raw_session is called → unauthorized P2P session established.
   The legitimate hole-punch to P_T is permanently cancelled.
6. Repeat once per second to cancel any newly inserted inflight entry for P_T.
```
