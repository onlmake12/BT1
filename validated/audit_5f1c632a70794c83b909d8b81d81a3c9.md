### Title
DAO Withdraw Deposit-Header Resolved from Wrong Index by Rust vs. C VM — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` reads the header-dep index from the witness as a full `u64`, while the on-chain DAO C script resolves the same index using only the **lowest byte** of that 8-byte little-endian value. This is a direct analog to the reported "wrong destination" class: the Rust tx-pool and the C VM resolve different deposit headers for the same transaction, causing the tx-pool to accept DAO withdraw transactions that the C VM will reject, which forces miners to produce invalid blocks.

---

### Finding Description

In `transaction_maximum_withdraw`, the witness `input_type` field is decoded as a full `u64` little-endian integer and used directly as the index into `header_deps`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // full u64 cast to usize
``` [1](#0-0) 

The on-chain DAO C script, however, reads the same 8-byte field but only uses the **lowest byte** as the index. This discrepancy is explicitly documented in the test suite:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
``` [2](#0-1) 

The test `check_dao_withdraw_header_dep_index_exceeds_u8` constructs a transaction with 258 `header_deps`, places the actual deposit block at index 1 (where the C VM resolves it via `257 & 0xFF = 1`) and the withdraw block at index 257 (where Rust resolves it via the full `u64`). The test asserts the Rust code **rejects** the transaction because it resolves the wrong header and the block-number check fails — but the C VM would **accept** it. [3](#0-2) 

The reverse scenario — where Rust **accepts** but the C VM **rejects** — is equally constructible:

| Position | Content | Resolved by |
|---|---|---|
| `header_deps[0]` | arbitrary block (number ≠ deposit block number) | C VM (lowest byte of index `256` = `0`) |
| `header_deps[256]` | actual deposit block | Rust (full `u64` index `256`) |

Witness index = `256` (`0x0000000000000100` LE).

- **Rust path**: resolves `header_deps[256]` = actual deposit block → block-number check passes → `transaction_fee` returns `Ok` → tx-pool admits the transaction.
- **C VM path**: resolves `header_deps[0]` = arbitrary block → block-number check fails → script execution fails → block is invalid.

The `CapacityVerifier` explicitly skips `OutputsSumOverflow` for DAO withdraw transactions, delegating capacity enforcement entirely to the DAO type script: [4](#0-3) 

This means there is no Rust-side capacity guard that would catch the mismatch before the transaction reaches a miner.

---

### Impact Explanation

A transaction sender who holds any DAO cell can craft a phase-2 DAO withdraw transaction with ≥257 `header_deps` and a witness index whose lowest byte points to a different block than the full `u64` index. The tx-pool (`check_tx_fee` → `DaoCalculator::transaction_fee`) accepts the transaction because the Rust code resolves the correct deposit header. When a miner includes the transaction, the C VM resolves a different header, the block-number check fails, script execution aborts, and the entire block is rejected by the network. The miner loses the block reward. The attacker can repeat this to continuously waste miner resources and stall block production. [5](#0-4) 

---

### Likelihood Explanation

The attacker only needs:
1. A DAO cell (requires depositing their own CKB — no privileged access).
2. A transaction with ≥257 `header_deps`, each referencing a valid canonical-chain block hash. No size limit on `header_deps` count is enforced beyond the overall transaction size limit.
3. Knowledge of the index-resolution discrepancy (documented in the codebase itself).

The attack is fully reachable via the standard `send_transaction` RPC by any unprivileged transaction sender.

---

### Recommendation

Align the Rust index resolution with the C VM behavior. In `transaction_maximum_withdraw`, truncate the decoded `u64` index to its lowest byte before indexing into `header_deps`:

```rust
// Before:
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// After (matching C VM behavior):
Ok(u64::from(header_deps_index_data.unwrap()[0]))
```

Alternatively, if the C VM is to be updated to use the full `u64`, the Rust code is already correct and the C script must be patched and redeployed. Either way, the two must agree on the same index-resolution semantics.

---

### Proof of Concept

1. Deposit CKB into NervosDAO; wait for the prepare phase.
2. Build a phase-2 withdraw transaction:
   - `header_deps`: 258 entries; `header_deps[0]` = any canonical block whose number ≠ the deposit block number; `header_deps[256]` = the actual deposit block.
   - Witness `input_type` = `256u64` encoded as 8-byte little-endian (`[0x00, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`).
   - Output capacity = value returned by `calculate_dao_maximum_withdraw` (Rust-side correct value).
3. Submit via `send_transaction` RPC — the tx-pool accepts it (Rust resolves `header_deps[256]` = deposit block, block-number check passes).
4. A miner picks up the transaction and assembles a block.
5. During block validation, the C VM resolves `header_deps[0]` (lowest byte of `256` = `0`) = wrong block → block-number check fails → script aborts → block rejected by the network.
6. The miner's block reward is lost. Repeat to sustain the attack.

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
