### Title
`verifyUpdate` Excess Fee Refund via `transfer()` Blocks Smart Contract Callers - (`lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate` uses Solidity's `transfer()` to refund excess ETH to `msg.sender`. Because `transfer()` forwards only 2300 gas, any smart contract caller whose `receive()` or `fallback()` function consumes more than 2300 gas will have the entire `verifyUpdate` call revert, permanently blocking those integrators from using Pyth Lazer.

### Finding Description

In `PythLazer.sol`, the `verifyUpdate` function accepts a fee and refunds any overpayment:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);  // <-- vulnerable
    }
    ...
}
``` [1](#0-0) 

`transfer()` hard-caps the gas forwarded to the recipient at 2300. This is insufficient for any smart contract whose `receive()` or `fallback()` function:
- emits an event (costs ~375+ gas per topic),
- writes to storage,
- calls another contract, or
- performs any non-trivial logic.

Since EIP-1884 (Istanbul), even a simple `SLOAD` costs 2100 gas, making 2300 gas routinely insufficient for real-world contracts.

The `verifyUpdate` function is the **core entry point** for all Pyth Lazer consumers on EVM chains. It is permissionless and intended to be called by any integrating contract. The `verification_fee` is initialized to `1 wei`: [2](#0-1) 

Any caller that sends `msg.value > 1 wei` (e.g., to avoid a separate fee-estimation step, or due to rounding in a wrapper contract) will trigger the refund path and revert if their contract cannot accept ETH with 2300 gas.

### Impact Explanation

Smart contract integrators of Pyth Lazer are permanently blocked from calling `verifyUpdate` whenever they overpay the fee and their contract has a non-trivial `receive()` function. This is the **only** on-chain verification function for Lazer price updates on EVM. A blocked `verifyUpdate` means the integrating protocol cannot consume Lazer price data at all, breaking its core functionality. The funds (excess ETH) are also permanently locked in the caller's transaction revert — no partial state change occurs, but the operation cannot be completed.

### Likelihood Explanation

Moderate-to-high. Many DeFi protocols use smart contracts with `receive()` functions that emit events or update accounting state (e.g., tracking received ETH). Any such protocol that sends even 2 wei to `verifyUpdate` will be blocked. The `verification_fee` of 1 wei makes overpayment trivially easy. The Fortuna keeper and other automated callers may also send rounded gas amounts that exceed the exact fee.

### Recommendation

Replace `transfer()` with a low-level `call` for the refund, which forwards all available gas:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

This is the same pattern already used correctly in `Entropy.sol`'s `withdraw()` and `withdrawAsFeeManager()` functions. [3](#0-2) 

### Proof of Concept

1. Deploy a smart contract `Consumer` with a `receive()` function that emits an event (costs >2300 gas).
2. `Consumer` calls `PythLazer.verifyUpdate{value: 2 wei}(update)`.
3. Since `msg.value (2) > verification_fee (1)`, the contract attempts `payable(Consumer).transfer(1 wei)`.
4. The `transfer()` call reverts because `Consumer.receive()` exceeds 2300 gas.
5. The entire `verifyUpdate` call reverts — `Consumer` cannot verify any Lazer price update. [4](#0-3)
