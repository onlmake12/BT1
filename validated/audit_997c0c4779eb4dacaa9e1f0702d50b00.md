### Title
Unpermissioned `set_publisher_stake_account` Allows Any Signer to Hijack Any Publisher's Stake Account Association — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `set_publisher_stake_account` instruction in the Integrity Pool program accepts an arbitrary `signer` with no constraint tying it to the `publisher` account being modified. Any unprivileged transaction sender can call this instruction, pass any registered publisher's pubkey as `publisher`, and overwrite that publisher's stake account pointer in `pool_data` to an attacker-controlled account.

---

### Finding Description

The IDL for `set_publisher_stake_account` defines the following accounts:

```
signer                          – signer: true  (no relations, no constraint)
publisher                       – CHECK: checked against pool_data (existence only)
pool_data                       – writable
pool_config                     – PDA
new_stake_account_positions_option     – optional
current_stake_account_positions_option – optional
```

The `signer` account carries **no `relations` constraint** linking it to the `publisher` or to any governance authority stored in `pool_config`. The `publisher` account comment reads only `"CHECK : The publisher will be checked against data in the pool_data"` — meaning the program verifies the publisher exists in the pool, not that the signer owns or controls that publisher's stake account. [1](#0-0) 

The SDK's `setPublisherStakeAccount` method passes `this.wallet.publicKey` implicitly as `signer` and accepts any caller-supplied `publisher` pubkey: [2](#0-1) 

This method is also surfaced through the public staking dashboard API as `reassignPublisherAccount`, bound to the connected wallet: [3](#0-2) 

The `current_stake_account_positions_option` provides no meaningful protection because all `pool_data` is public on-chain; an attacker trivially reads the current value before crafting the transaction.

---

### Impact Explanation

`pool_data.publisherStakeAccounts[publisher_index]` is the authoritative record used to distinguish a publisher's **self-delegation** from external delegator stake. By overwriting this field:

1. **Reward theft** — The publisher's self-stake rewards (`R^p_p`) are credited to the attacker-controlled stake account instead of the legitimate publisher account.
2. **Self-stake erasure** — The publisher's real stake account is no longer recognized as self-stake; the publisher loses eligibility for the self-stake reward component and their pool cap contribution is distorted.
3. **Pool accounting corruption** — `selfDelState` and `selfDelegationDelta` tracked in `pool_data` become inconsistent, corrupting per-epoch reward snapshots for all delegators in that pool. [4](#0-3) 

---

### Likelihood Explanation

- The instruction is reachable by any Solana wallet with no special role.
- The staking dashboard exposes `reassignPublisherAccount` to any connected user.
- All required inputs (`publisher` pubkey, current stake account address) are publicly readable from `pool_data` on-chain.
- No economic barrier exists; the attacker only pays transaction fees.

---

### Recommendation

Add an explicit ownership constraint to `set_publisher_stake_account` requiring the `signer` to be either:

1. The owner of the `current_stake_account_positions_option` (i.e., `stake_account_metadata.owner == signer`), enforced via a CPI into the staking program, **or**
2. The `governance_authority` stored in `pool_config`.

This mirrors the fix applied in the referenced ERC-4337 report: replace the "operate on any account" pattern with "caller can only update their own record."

---

### Proof of Concept

```
1. Attacker reads pool_data on-chain → finds publisher P with stake account S_current.
2. Attacker creates their own stake account S_attacker.
3. Attacker submits set_publisher_stake_account with:
     signer                              = attacker wallet (signs)
     publisher                           = P
     current_stake_account_positions_option = S_current
     new_stake_account_positions_option  = S_attacker
4. pool_data.publisherStakeAccounts[P_index] is now S_attacker.
5. Next advance_delegation_record for publisher P credits self-stake
   rewards to S_attacker's custody account instead of P's.
6. Publisher P's self-stake is no longer counted; their pool cap
   and reward ratio are permanently distorted until governance intervenes.
``` [5](#0-4) [6](#0-5)

  async setPublisherStakeAccount(
    publisher: PublicKey,
    stakeAccountPositions: PublicKey,
    newStakeAccountPositions: PublicKey | undefined,
  ) {
    const instruction = await this.integrityPoolProgram.methods
      .setPublisherStakeAccount()
      .accounts({
        currentStakeAccountPositionsOption: stakeAccountPositions,
        // eslint-disable-next-line unicorn/no-null
        newStakeAccountPositionsOption: newStakeAccountPositions ?? null,
        publisher,
      })
      .instruction();

    await sendTransaction([instruction], this.connection, this.wallet);
    return;
  }
