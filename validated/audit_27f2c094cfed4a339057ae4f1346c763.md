Audit Report

## Title
Surviving Ancestors Retain Stale `evict_key` After Batch Descendant Removal — (File: tx-pool/src/component/pool_map.rs)

## Summary
`remove_entry_and_descendants` tears down all link-graph entries in a first pass before calling `remove_entry` on any of them. Because `update_ancestors_index_key` depends on the link graph to locate surviving ancestors, it finds an empty ancestor set for every removed entry and silently skips the `evict_key` update. Every pool entry that is a surviving ancestor of the removed subtree is left with a stale, inflated `evict_key`, corrupting the eviction-priority ordering of the pool.

## Finding Description

`remove_entry_and_descendants` (pool_map.rs L252–265) performs two sequential passes over `removed_ids`:

```rust
for id in &removed_ids {
    self.remove_entry_links(id);   // tears down ALL link records first
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
``` [1](#0-0) 

`remove_entry_links` (L418–430) removes the entry from its parents' children sets, from its children's parents sets, and then calls `self.links.remove(id)`, which deletes the record from `TxLinksMap::inner`: [2](#0-1) [3](#0-2) 

After the first loop, no link record for any removed entry exists. When `remove_entry` subsequently runs, it calls `update_ancestors_index_key` (L432–445): [4](#0-3) 

`update_ancestors_index_key` calls `self.links.calc_ancestors`, which calls `calc_relative_ids`. That function does `self.inner.get(short_id)` — since the link record was already deleted, it returns `None`, `direct` is an empty set, and `calc_relation_ids` returns an empty set: [5](#0-4) 

The loop body in `update_ancestors_index_key` never executes. Surviving ancestors (those not in `removed_ids`) never receive `sub_descendant_weight` or an `evict_key` recomputation. Their `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` fields remain inflated.

The `evict_key` is computed from these stale fields: [6](#0-5) 

The `EvictKey` ordering sorts ascending by `fee_rate`, then `descendants_count`, then `timestamp`: [7](#0-6) 

The comment in `remove_entry_and_descendants` — *"update links state for remove, so that we won't update_descendants_index_key in remove_entry"* — confirms the intent was only to suppress descendant updates (correct, since all descendants are being removed), but the implementation also suppresses ancestor updates (incorrect, since ancestors survive).

The reachable trigger is `resolve_conflict` (L305–332), called whenever a new transaction double-spends an input already consumed by a pool transaction — an unprivileged, externally-initiated operation: [8](#0-7) 

## Impact Explanation

`next_evict_entry` (L380–385) iterates `iter_by_evict_key()` in ascending order and picks the first entry matching the requested status: [9](#0-8) 

A stale, inflated `evict_key` places a surviving ancestor toward the high end of the eviction index, making it appear more fee-valuable than it actually is. When the pool is full and eviction is needed, that ancestor is skipped in favour of entries that should have been retained. A low-fee-rate transaction can persist in the pool indefinitely while higher-fee-rate transactions are incorrectly evicted. This is a concrete corruption of the pool's state-management invariants, matching the **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism** impact class.

## Likelihood Explanation

Any unprivileged user can trigger this by submitting a transaction chain A→B→C (where A has a low own fee rate but B/C have high fee rates, inflating A's `evict_key`), then submitting a transaction D that double-spends B's input. `resolve_conflict` calls `remove_entry_and_descendants(&B_id)`, removing B and C while leaving A with a stale `evict_key`. The attack is repeatable at the cost of transaction fees for A, B, C, and D, and the stale state persists until A is itself removed.

## Recommendation

In `remove_entry_and_descendants`, update surviving ancestors' `evict_key` before tearing down the link graph. One correct approach: for the root entry only, call `update_ancestors_index_key` while the link graph is still intact, then proceed with `remove_entry_links` for all entries and `remove_entry` for each (skipping the redundant ancestor update inside `remove_entry` for the batch case). Alternatively, restructure so that `remove_entry_links` is called as the last step inside `remove_entry`, consistent with the single-entry removal path.

## Proof of Concept

1. Insert ancestor `A` (low own fee rate) into the pool.
2. Insert child `B` of `A` (high fee rate); `A.evict_key.fee_rate` is now elevated.
3. Insert grandchild `C` of `B` (high fee rate); `A.evict_key` is further elevated.
4. Assert `A.evict_key.descendants_count == 3` and `A.evict_key.fee_rate` reflects B/C's fee rate.
5. Call `pool_map.remove_entry_and_descendants(&B.id)` (simulating `resolve_conflict`).
6. Assert `A` is still present in the pool.
7. Assert `A.evict_key.descendants_count` — it will still be `3` instead of the correct `1`.
8. Assert `A.evict_key.fee_rate` — it will still reflect B/C's fee rate instead of A's own fee rate.
9. Fill the pool to capacity with medium-fee-rate entries; call `next_evict_entry` and observe that `A` is not selected for eviction despite having the lowest genuine fee rate.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {

        }

    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
