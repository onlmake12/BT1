Audit Report

## Title
Expired Alert Replay via P2P Enables Repeated Network-Wide Re-broadcast — (`util/network-alert/src/notifier.rs`, `util/network-alert/src/alert_relayer.rs`)

## Summary
`Notifier::clear_expired_alerts` evicts expired alert IDs from `received_alerts` without inserting them into `cancel_filter`, so `has_received` subsequently returns `false` for those IDs. The P2P `received` handler has no independent `notice_until` check, so any unprivileged peer can replay the raw bytes of a previously-expired, legitimately-signed alert. The node re-accepts it, re-notifies users, and re-broadcasts it to all connected peers. Because the attacker is never banned and can reconnect to re-trigger eviction, the cycle repeats indefinitely at negligible cost.

## Finding Description
`Notifier::clear_expired_alerts` (notifier.rs L135–144) uses `HashMap::retain` to drop expired entries from `received_alerts` and `Vec::retain` on `noticed_alerts`, but never calls `self.cancel_filter.put(id, ())` for the removed IDs. `has_received` (notifier.rs L147–149) returns `true` only if the ID is in `received_alerts` OR `cancel_filter`; after eviction it is in neither, so it returns `false`.

`AlertRelayer::connected` (alert_relayer.rs L87) calls `clear_expired_alerts` on every new peer connection. The P2P `received` handler (alert_relayer.rs L144–179) then checks only `has_received(alert_id)` before proceeding to signature verification — there is no check of `notice_until` against the current time. The RPC `send_alert` path (alert.rs L104–110) does perform this check, creating an asymmetry.

Exploit loop:
1. Attacker saves raw bytes of a legitimately-signed alert (id=N, notice_until=T) observed on the network.
2. After time T, attacker connects to a target node → `clear_expired_alerts` evicts id=N from `received_alerts`.
3. Attacker sends saved bytes → `has_received(N)` = `false` → `verify_signatures` passes (bytes unchanged) → `notifier.add` re-inserts id=N, calls `notify_network_alert`, and `quick_filter_broadcast` re-propagates to all connected peers.
4. Attacker disconnects and reconnects → eviction fires again → repeat indefinitely.

If the replayed alert carries `cancel > 0` targeting a currently-active alert, `notifier.add` calls `self.cancel(alert_cancel)` (notifier.rs L100–102), which inserts the active alert's ID into `cancel_filter` and removes it from `received_alerts` and `noticed_alerts`, silently suppressing a live warning on every node that processes the replay.

## Impact Explanation
An unprivileged attacker can drive repeated network-wide re-broadcasts of an expired alert at negligible cost (one TCP connection per cycle, no ban triggered). This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** Each replay cycle fans out to all peers connected on the Alert protocol, and those peers re-broadcast further, amplifying the load. Additionally, if the replayed alert's `cancel` field targets an active alert, it suppresses a live network warning across all reachable nodes, undermining the integrity of the alert system.

## Likelihood Explanation
- No privilege is required; any P2P peer on the Alert protocol can send arbitrary alert bytes.
- Alerts are broadcast to all nodes, so capturing raw bytes is trivial for any network observer.
- `clear_expired_alerts` fires on every new peer connection, so the dedup state resets continuously in a live network.
- The attacker is never banned (valid signatures pass the ban check), enabling indefinite repetition.
- The only precondition is that at least one legitimate alert has been issued and has since expired — a condition that will be met in any long-running network.

## Recommendation
1. **In `Notifier::clear_expired_alerts`**: insert evicted IDs into `cancel_filter` before removing them from `received_alerts`, so `has_received` continues to return `true` for expired IDs:
```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    let expired_ids: Vec<u32> = self.received_alerts
        .iter()
        .filter(|(_, alert)| {
            let notice_until: u64 = alert.raw().notice_until().into();
            notice_until <= now
        })
        .map(|(id, _)| *id)
        .collect();
    for id in expired_ids {
        self.cancel_filter.put(id, ()); // prevent replay
        self.received_alerts.remove(&id);
    }
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```
2. **In the P2P `received` handler**: add an explicit `notice_until < now_ms` check mirroring the RPC path, rejecting and banning peers that send already-expired alerts before signature verification.

## Proof of Concept
1. Run two CKB nodes (A and B) connected via the Alert protocol.
2. Use `send_alert` RPC on node A to broadcast a valid alert (id=1, notice_until = now + 5 seconds). Capture the raw P2P bytes.
3. Wait 6 seconds for the alert to expire.
4. Connect a test peer to node A on the Alert protocol, triggering `clear_expired_alerts` (id=1 evicted from `received_alerts`, not added to `cancel_filter`).
5. Send the captured raw bytes from the test peer.
6. Observe: node A's `has_received(1)` returns `false`; signatures pass; `notify_network_alert` fires; the alert is re-broadcast to node B.
7. Disconnect and reconnect the test peer; repeat step 5. Observe the cycle repeats without limit.
8. To test the cancel-suppression variant: issue a second active alert (id=2) and replay an expired alert whose `cancel` field is set to 2; observe id=2 is removed from `noticed_alerts` on all reachable nodes.
