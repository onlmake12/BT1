### Title
Ping-RTT Eviction Protection Neutralized by `as_secs()` Truncation — (`network/src/peer_registry.rs`)

### Summary

`try_evict_inbound_peer` uses `Duration::as_secs()` to compare peer ping RTTs for eviction protection. Because `as_secs()` truncates sub-second values to `0`, every peer with a ping RTT in `[0ms, 999ms]` is treated as identical. Any attacker who can respond to pings within one second — trivially achievable from any co-located or well-connected vantage point — is indistinguishable from a legitimate low-latency peer, neutralizing the entire first protection round.

### Finding Description

In `try_evict_inbound_peer`, the first protection pass is intended to retain the 8 peers with the lowest measured ping RTT: [1](#0-0) 

The comparator extracts `p.as_secs()` from each peer's `ping_rtt`. `Duration::as_secs()` returns the whole-second component only, truncating everything below 1 second to `0u64`. In practice, `pong_received` stores the raw measured RTT: [2](#0-1) 

For any peer reachable within one second — which covers virtually all real-world peers — `ping_rtt` will be a sub-second `Duration`. After `.as_secs()`, every such peer maps to `0`. The sort comparator sees all values as equal, so Rust's stable sort preserves the original `HashMap::values()` iteration order, which is non-deterministic. The 8 "protected" peers are therefore selected pseudo-randomly from the pool of all sub-second RTT peers, not by actual latency rank.

The second protection round (most-recently-messaged peers, lines 168–183) has the identical defect: it also calls `.as_secs()` on the elapsed duration since the last ping message, collapsing all peers that sent a message within the last second to the same bucket. [3](#0-2) 

The existing test only sets `ping_rtt = Some(Duration::from_secs(0))` (whole seconds), so it never exercises the sub-second case and does not catch this regression: [4](#0-3) 

### Impact Explanation

An attacker who establishes ≥8 inbound connections and ensures each responds to pings within 1 second (no special infrastructure required — any peer within a continent can achieve this) gains equal standing with legitimate low-latency peers in the first two protection rounds. The code comment explicitly states these rounds protect "characteristics that an attacker [is] hard to simulate or manipulate," but with `as_secs()` granularity, sub-second RTT is trivially simulatable. The attacker's connections survive eviction rounds at the same rate as legitimate peers, increasing their ability to maintain a persistent inbound foothold and contributing to eclipse attack conditions over time.

### Likelihood Explanation

The precondition is minimal: the attacker needs ≥8 inbound TCP connections to the target node with sub-second ping response times. This is achievable from any geographically proximate server or VPS fleet. No privileged access, leaked keys, or majority hashpower is required. The attack is fully reachable via the standard P2P connection path (`accept_peer` → `try_evict_inbound_peer`). [5](#0-4) 

### Recommendation

Replace `p.as_secs()` with `p.as_millis()` (returning `u128`) in both the ping-RTT sort and the last-message-received sort. This restores millisecond-granularity discrimination between peers and makes the protection rounds meaningful against sub-second RTT manipulation.

```rust
// ping RTT round
.map(|p| p.as_millis())
.unwrap_or(u128::MAX)

// last-message round
.map(|t| now.saturating_duration_since(t).as_millis())
.unwrap_or(u128::MAX)
```

The test should also be updated to set `ping_rtt` values in milliseconds (e.g., `Duration::from_millis(50)` vs `Duration::from_millis(500)`) to verify that the sort correctly distinguishes sub-second peers.

### Proof of Concept

```
Scenario: max_inbound = 9, 8 attacker peers + 8 legitimate peers already connected.

1. Attacker opens 8 inbound connections from diverse IPs.
2. Each attacker peer responds to every ping within 500ms.
   → peer.ping_rtt = Some(Duration::from_millis(500))
   → .as_secs() = 0

3. 8 legitimate peers have ping_rtt = Some(Duration::from_millis(50))
   → .as_secs() = 0

4. try_evict_inbound_peer() runs. All 16 peers map to ping key = 0.
   Sort is a no-op (all equal). sort_then_drop keeps last 8 by HashMap order.
   → 8 "protected" peers are chosen non-deterministically, ~50% attacker peers.

5. Repeat across many eviction rounds: attacker peers survive at the same
   rate as legitimate peers. Over time, attacker maintains persistent
   inbound presence, weakening eclipse attack resistance.

Differential test: with as_millis(), attacker peers (500ms) sort strictly
worse than legitimate peers (50ms), and all 8 legitimate peers are protected.
With as_secs(), both groups are indistinguishable.
``` [6](#0-5)
