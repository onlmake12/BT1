Audit Report

## Title
`consecutive_failed` counts total failures instead of consecutive failures, enabling premature block-template truncation — (`tx-pool/src/component/tx_selector.rs`)

## Summary
In `TxSelector::txs_to_commit`, the variable `consecutive_failed` is incremented on two failure paths but is never reset to zero when a transaction is successfully packaged. This means the counter accumulates across the entire loop rather than tracking a streak of consecutive failures, violating the semantic contract of both the variable name and `MAX_CONSECUTIVE_FAILURES`. An attacker can submit 4001 parent-in-`Gap` / child-in-`Proposed` transaction pairs to exhaust the counter and force the loop to exit early, excluding valid fee-paying transactions from the block template.

## Finding Description
`consecutive_failed` is initialized to `0` at line 104. [1](#0-0) 

It is incremented at line 153 when `next_cycles > cycles_limit || next_size > size_limit`: [2](#0-1) 

It is incremented again at line 184 when any ancestor of the candidate transaction is not in `Status::Proposed`: [3](#0-2) 

The success path (lines 191–220) contains no `consecutive_failed = 0` reset anywhere: [4](#0-3) 

`has_proposed` returns false for any transaction not in `Status::Proposed`, including `Status::Gap`: [5](#0-4) 

**Exploit flow:**
1. Attacker submits parent transactions A₁…A₄₀₀₁ to the P2P network; they are proposed in block N.
2. Block N+1 is produced without committing the parents; they transition to `Status::Gap`. Children C₁…C₄₀₀₁ (each spending a respective parent) are proposed in block N+1 and enter `Status::Proposed`.
3. When block N+2's template is assembled via `txs_to_commit`, `sorted_proposed_iter()` yields each Cᵢ (it is `Proposed`). For each Cᵢ, `calc_ancestors` returns {Aᵢ}, and `has_proposed(Aᵢ)` returns false because Aᵢ is `Status::Gap`. The ancestor check at line 176–178 fails, incrementing `consecutive_failed`.
4. After 4001 such failures, `consecutive_failed > MAX_CONSECUTIVE_FAILURES` (4000) is true and the loop breaks, even if valid high-fee transactions remain in the iterator.

The `failed_txs` set does not mitigate this: when `using_modified` is false (transactions from the main `sorted_proposed_iter()`), the failing transaction is not inserted into `failed_txs`, so the guard in `skip_proposed_entry` does not apply. The iterator simply advances, and each new Cᵢ triggers a fresh increment. [6](#0-5) 

## Impact Explanation
Block templates are truncated prematurely. Miners produce blocks with fewer transactions than the size/cycle limits allow, directly reducing fee revenue per block. This is a concrete, measurable performance degradation in block template generation, fitting **Low (501–2000 points): Any other important performance improvements for CKB**. The impact does not rise to High (network congestion) because the attacker must pay fees for approximately 8002 on-chain transactions, making sustained attack costly.

## Likelihood Explanation
An unprivileged attacker uses only the standard P2P transaction submission interface. The required pool state — parent in `Gap`, child in `Proposed` — is a normal, reachable transition achievable by proposing parents in block N (without committing them) and children in block N+1. Producing 4001 such pairs requires approximately 8002 transactions. This is feasible on mainnet at moderate cost and is repeatable every few blocks as long as the attacker continues funding the attack.

## Recommendation
Add `consecutive_failed = 0;` at the end of the success path, immediately after `self.update_modified_entries(&ancestors)` at line 220:

```rust
self.update_modified_entries(&ancestors);
consecutive_failed = 0; // reset: a successful selection breaks any failure streak
``` [7](#0-6) 

## Proof of Concept
1. Construct a `PoolMap` with 4001 transactions Cᵢ each having one ancestor Aᵢ in `Status::Gap` and Cᵢ itself in `Status::Proposed`. Insert one additional valid transaction V (all ancestors `Proposed`, fits within limits) at a sort position after all Cᵢ.
2. Call `TxSelector::new(&pool_map).txs_to_commit(size_limit, cycles_limit)`.
3. Assert: V is **absent** from the returned `entries` vector despite fitting within size and cycle limits — `consecutive_failed` reached 4001 before V was evaluated.
4. Apply the one-line fix (`consecutive_failed = 0` after `update_modified_entries`).
5. Re-run; assert V is now **present** in `entries`.
