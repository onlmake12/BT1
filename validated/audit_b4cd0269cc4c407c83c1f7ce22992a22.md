### Title
Incomplete Block Status Guard in `HeaderAcceptor::accept()` Allows Re-Processing of Already-Invalid Headers — (File: `sync/src/synchronizer/headers_process.rs`)

### Summary
`HeaderAcceptor::accept()` checks whether a block header is already known-valid (`HEADER_VALID`) to short-circuit processing, but contains an acknowledged `FIXME` that it does **not** check for the `BLOCK_INVALID` terminal state. A remote peer can send headers for blocks already confirmed invalid, bypassing the guard, causing the node to re-run verification and potentially overwrite the in-memory `BLOCK_INVALID` status with `HEADER_VALID`.

### Finding Description

In `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept()` begins with a status check:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // update peer's best known header and return early
    return result;
}
``` [1](#0-0) 

The guard only short-circuits on `HEADER_VALID`. It does **not** short-circuit on `BLOCK_INVALID`. The `FIXME` comment explicitly acknowledges this gap.

`get_block_status()` derives `BLOCK_INVALID` from two sources: the in-memory `block_status_map`, or from the persisted `block_ext.verified == Some(false)` in the database:

```rust
pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
    match self.block_status_map().get(block_hash) {
        Some(status_ref) => *status_ref.value(),
        None => {
            if self.header_map().contains_key(block_hash) {
                BlockStatus::HEADER_VALID
            } else {
                let verified = self.snapshot().get_block_ext(block_hash)
                    .map(|block_ext| block_ext.verified);
                match verified {
                    Some(Some(false)) => BlockStatus::BLOCK_INVALID,
                    ...
                }
            }
        }
    }
}
``` [2](#0-1) 

When a block fails **full contextual verification** (e.g., invalid transactions), its status is persisted to the DB as `block_ext.verified = Some(false)`, but it may **not** be inserted into the in-memory `block_status_map`. In that case, `get_block_status` returns `BLOCK_INVALID` by reading the DB.

If a peer then sends the header of this already-invalid block:
1. `status = BLOCK_INVALID` — does **not** contain `HEADER_VALID`, so the early-return is skipped.
2. `prev_block_check` — checks the *parent's* status, not the block itself; passes if parent is valid.
3. `non_contextual_check` — verifies PoW and timestamp only; passes if the block was invalid for contextual reasons (e.g., bad transactions).
4. `version_check` — passes if version == 0.
5. `sync_shared.insert_valid_header(self.peer, self.header)` — **inserts the block into `header_map`**.

After step 5, `get_block_status` now returns `HEADER_VALID` (because `header_map` is checked before the DB), effectively **overwriting** the `BLOCK_INVALID` status in the node's live view. [3](#0-2) 

### Impact Explanation

- **State inconsistency**: A block confirmed invalid (persisted in DB) has its in-memory status silently upgraded to `HEADER_VALID`. The block fetcher (`BlockFetcher`) consults `get_block_status` to decide what to download; it may now treat this block as a candidate for re-download and re-processing.
- **Resource exhaustion (DoS)**: Any peer can repeatedly send headers for previously-rejected blocks, forcing the node to re-run `non_contextual_check` (PoW verification) and re-insert into `header_map` on every delivery, with no rate-limiting from the status guard.
- **Peer best-known-header pollution**: The peer's `best_known_header` may be set to a chain tip that descends from an invalid block, distorting sync decisions.

### Likelihood Explanation

Any unprivileged remote peer can send a `SendHeaders` message containing arbitrary headers, including headers for blocks the node has already fully verified and rejected. No special privilege or key is required. The attack is trivially repeatable.

### Recommendation

Add an explicit early-return for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, resolving the existing `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    state.invalid(Some(ValidationError::InvalidParent));
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // existing logic ...
    return result;
}
```

This mirrors the pattern already used in `prev_block_check`, which correctly rejects headers whose *parent* is `BLOCK_INVALID`. [4](#0-3) 

### Proof of Concept

1. Node A fully processes block B; B fails contextual verification (e.g., a transaction spends a non-existent cell). `block_ext.verified = Some(false)` is written to DB. `block_status_map` has no entry for B.
2. Attacker peer sends `SendHeaders([B.header])` to Node A.
3. `HeaderAcceptor::accept()` is called. `get_block_status(B)` reads DB → `BLOCK_INVALID`. The `HEADER_VALID` guard does not fire.
4. `prev_block_check` passes (B's parent is valid). `non_contextual_check` passes (B's PoW and timestamp are valid). `version_check` passes.
5. `insert_valid_header` inserts B into `header_map`.
6. Node A now reports `get_block_status(B) == HEADER_VALID`. The block fetcher may schedule B for re-download. Repeating step 2 continuously forces repeated re-verification work with no guard.
