Audit Report

## Title
Unvalidated Ordering of Peer-Supplied `block_locator_hashes` Breaks Early-Exit Invariant in `insert_peer_unknown_header_list`, Disrupting IBD Block Fetching — (`sync/src/types/mod.rs`)

## Summary
During IBD, `GetHeadersProcess::execute` passes peer-supplied `block_locator_hashes` verbatim to `insert_peer_unknown_header_list`, which assumes the list is ordered highest-to-lowest to correctly populate `unknown_header_list` and eventually set `best_known_header`. A malicious outbound peer can send hashes in reversed (lowest-to-highest) order, causing `BlockFetcher::fetch` to immediately stall on the first `take_unknown_last` call and never advance `best_known_header`, so the victim node never issues `GetBlocks` requests to that peer for the duration of IBD.

## Finding Description
`GetHeadersProcess::execute` collects `block_locator_hashes` from the raw P2P message with no ordering validation and passes them directly to `insert_peer_unknown_header_list` when the sender is an outbound/whitelist/protect peer during IBD. [1](#0-0) 

`insert_peer_unknown_header_list` iterates the list and pushes unknown hashes via `insert_unknown_header_hash` (`Vec::push`), breaking only when a known hash is found. The comment documents the ordering assumption but nothing enforces it. [2](#0-1) 

`insert_unknown_header_hash` appends to the tail of the `Vec`. [3](#0-2) 

`take_unknown_last` uses `Vec::pop`, yielding the tail element. [4](#0-3) 

**Normal case (highest-to-lowest):** hashes are pushed as `[h_n, ..., h_1]` (highest at index 0, lowest at tail). `pop` returns `h_1` (lowest unknown). As IBD progresses and lower blocks are downloaded, `h_1` becomes known, `best_known_header` advances, and the loop continues upward.

**Reversed case (lowest-to-highest, attacker-controlled):** hashes are pushed as `[h_1, ..., h_n]` (lowest at index 0, highest at tail). `pop` returns `h_n` (highest unknown). During IBD `h_n` is still unknown, so the `else` branch fires immediately — the hash is pushed back and the loop breaks. [5](#0-4) 

`best_known_header` is never set, and `peer_best_known_header()` returns `None`, causing `fetch` to return `None` at the guard check. [6](#0-5) 

The `unknown_header_list_is_empty` guard means the list is populated only once per peer connection, so the corrupted state persists for the entire connection lifetime. [7](#0-6) 

## Impact Explanation
The impact is IBD performance degradation for individual nodes. An attacker occupying a fraction of a victim's outbound slots reduces the effective block-download bandwidth proportionally, prolonging IBD. The node is not crashed, no consensus deviation occurs, and no economic damage results. This fits **Low (501–2000 points): Any other important performance improvements for CKB**, as the fix (enforcing sort order) is a correctness/performance improvement to the IBD sync mechanism.

## Likelihood Explanation
The attack requires only running a reachable CKB node and waiting for victim nodes to open outbound connections — a low-cost, realistic condition. The guard `flag.is_outbound || flag.is_whitelist || flag.is_protect` is satisfied by any normal outbound peer. [8](#0-7) 

The malformed message is a single P2P packet; no key material or hash power is required.

## Recommendation
In `insert_peer_unknown_header_list`, after collecting hashes into `unknown_header_list`, sort the list by block number in ascending order (so `Vec::pop` yields the lowest unknown hash first) rather than trusting the peer-supplied ordering. Alternatively, look up each hash's block number from the header map or store and insert into a sorted structure (e.g., `BTreeMap` keyed by block number) to guarantee correct consumption order regardless of peer input. [2](#0-1) 

## Proof of Concept
1. Attacker runs a CKB node; victim opens an outbound connection during IBD.
2. Attacker sends `SyncMessage::GetHeaders` with `block_locator_hashes` ordered lowest-to-highest (reversed from the expected highest-to-lowest).
3. `GetHeadersProcess::execute` calls `insert_peer_unknown_header_list(peer, reversed_hashes)`.
4. All hashes are unknown (IBD ongoing); they are pushed in reversed order — lowest at index 0, highest at tail. No known hash is found, so `best_known_header` is not set here.
5. `BlockFetcher::fetch` calls `take_unknown_last`, popping the highest hash. That hash is unknown → pushed back, loop breaks immediately.
6. `peer_best_known_header()` returns `None`; `fetch` returns `None`; no `GetBlocks` is ever sent to this peer.
7. Repeating across multiple outbound slots degrades IBD throughput proportionally to the fraction of slots controlled.

    pub fn insert_peer_unknown_header_list(&self, pi: PeerIndex, header_list: Vec<Byte32>) {
        // update peer's unknown_header_list only once
        if self.state().peers.unknown_header_list_is_empty(pi) {
            // header list is an ordered list, sorted from highest to lowest,
            // so here you discard and exit early
            for hash in header_list {
                if let Some(header) = self.shared().header_map().get(&hash) {
                    self.state()
                        .peers
                        .may_set_best_known_header(pi, header.as_header_index());
                    break;
                } else {
                    self.state().peers.insert_unknown_header_hash(pi, hash)
                }
            }
        }
    }
