### Title
Silent Negative-to-Zero Price Corruption via Unsafe `i64`→`u64` Cast in Lazer Bulk-Trade Pusher - (File: `apps/pyth-lazer-pusher/bulk-trade-pusher/src/pusher.rs`)

---

### Summary

In `apps/pyth-lazer-pusher/bulk-trade-pusher/src/pusher.rs`, the `run_push_loop` function converts a Lazer price mantissa from its native signed `i64` type to an unsigned `u64` using `u64::try_from(...).unwrap_or(0)`. When the Lazer price is negative — a documented, valid state for derivative products such as funding rates — the conversion fails silently and substitutes `0` as the price. The corrupted zero-price is then signed and pushed to Bulk Trade validators as if it were a legitimate oracle update.

---

### Finding Description

The Pyth Lazer protocol explicitly represents prices as signed 64-bit integers (`i64`). The payload reference documents the `price` field as `optional non-zero i64` and the Sui SDK documentation explicitly warns that "Pyth prices can be negative (e.g., for certain derivative products)."

In the bulk-trade pusher's push loop, each cached Lazer price is mapped to a `PythOraclePrice` struct for transmission to Bulk Trade validators:

```rust
// apps/pyth-lazer-pusher/bulk-trade-pusher/src/pusher.rs, line 107-119
let oracles: Vec<PythOraclePrice> = prices
    .iter()
    .filter_map(|cached| {
        let price = cached.data.price.as_ref()?;
        let exponent = cached.data.exponent?;
        Some(PythOraclePrice {
            timestamp: cached.timestamp_ms,
            feed_index: u64::from(cached.feed_id.0),
            price: u64::try_from(price.mantissa_i64()).unwrap_or(0),  // ← BUG
            exponent,
        })
    })
    .collect();
```

`price.mantissa_i64()` returns an `i64`. `u64::try_from(negative_i64)` returns `Err` for any negative value. `.unwrap_or(0)` silently replaces the actual negative price with `0`. No log, no metric, no skip — the entry is included in the batch with `price = 0` and forwarded to validators. [1](#0-0) 

This is structurally identical to the reported vulnerability: a signed value (`bytes32` / `i64`) is cast to an unsigned type (`uint256` / `u64`) without a validity check, producing a semantically wrong result that is then used downstream as if it were correct.

---

### Impact Explanation

Any Lazer feed whose current aggregate price is negative (e.g., a perpetual funding rate feed) will be transmitted to Bulk Trade validators with `price = 0`. Validators that consume this data for settlement, margin calculation, or liquidation logic will act on a completely wrong price. The severity depends on how Bulk Trade uses the oracle price, but a zero price for a non-zero asset is a maximal corruption of the price signal and can cause:

- Incorrect mark prices used for P&L or margin calculations.
- Erroneous liquidations or failure to liquidate when required.
- Mispriced derivatives settled at zero.

---

### Likelihood Explanation

Pyth Lazer's own documentation and SDK explicitly state that prices can be negative for derivative products. The `price` field type is `i64` throughout the protocol. Any feed that legitimately reports a negative price — without any attacker involvement — will trigger this path. The condition is reachable in normal operation whenever a funding-rate or similar derivative feed goes negative. [2](#0-1) 

---

### Recommendation

Replace the silent fallback with an explicit sign check and skip the entry (or log and skip) when the price is negative, consistent with how `PythUtils.convertToUint` handles negative prices in the EVM SDK:

```rust
// Option A: skip negative prices with a warning
.filter_map(|cached| {
    let price = cached.data.price.as_ref()?;
    let mantissa = price.mantissa_i64();
    if mantissa < 0 {
        warn!(feed_id = ?cached.feed_id, mantissa, "skipping negative price");
        return None;
    }
    let exponent = cached.data.exponent?;
    Some(PythOraclePrice {
        timestamp: cached.timestamp_ms,
        feed_index: u64::from(cached.feed_id.0),
        price: mantissa as u64,
        exponent,
    })
})
```

If Bulk Trade's protocol supports signed prices, the `PythOraclePrice` struct should be updated to use `i64` for the `price` field instead of `u64`. [3](#0-2) 

---

### Proof of Concept

1. A Lazer feed (e.g., a perpetual funding rate) reports a price of `-50000000` (mantissa, with exponent `-8`, representing `-0.5`).
2. The Lazer receiver caches this as `CachedPrice { data: { price: Some(Price { mantissa: -50000000 }), exponent: Some(-8) }, ... }`.
3. `run_push_loop` calls `price.mantissa_i64()` → returns `-50000000i64`.
4. `u64::try_from(-50000000i64)` → `Err(TryFromIntError(...))`.
5. `.unwrap_or(0)` → `0u64`.
6. `PythOraclePrice { price: 0, exponent: -8, ... }` is signed and pushed to Bulk Trade validators.
7. Validators receive a price of `0 × 10^-8 = 0.0` for a feed whose true price is `-0.5`.

No privileged access, no key compromise, and no attacker action is required. The corruption occurs automatically for any feed with a negative price. [2](#0-1)
