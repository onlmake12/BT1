### Title
Crafted `SignatureSet` Account Bypasses Guardian Signature Verification in `post_vaa` — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_vaa.rs`)

---

### Summary

`post_vaa` accepts any `SignatureSet` account owned by the core-bridge program with a valid Anchor discriminator. Because `SignatureSet` is a free keypair account (not a PDA), an attacker can create a raw account, write a crafted layout with `sig_verify_successes=[true; N]` and an arbitrary `message_hash`, assign it to the core-bridge program, and call `post_vaa` directly — completely bypassing the secp256k1 guardian signature verification enforced by `verify_signatures`. The `is_initialized()` flaw in `verify_signatures` is a secondary amplifier, but the primary root cause is that `post_vaa` places no constraint tying the `SignatureSet` to the `verify_signatures` flow.

---

### Finding Description

**Root cause 1 — `post_vaa` has no provenance constraint on `signature_set`:** [1](#0-0) 

The account is loaded with no `seeds`, no `has_one`, and no `init` — only Anchor's standard owner + discriminator check. Any account owned by the program with the correct 8-byte discriminator is accepted.

**Root cause 2 — `is_initialized()` uses a weak sentinel:** [2](#0-1) 

"Initialized" means *any* `sig_verify_successes` entry is `true`. A pre-seeded account with all entries `true` is treated as fully initialized, and the `else` branch that resets the array to `[false; N]` is never reached. [3](#0-2) 

**Root cause 3 — `signature_set` in `verify_signatures` is a free keypair, not a PDA:** [4](#0-3) 

No `seeds` constraint means the attacker fully controls which account is passed.

**`post_vaa` quorum check trusts the account data at face value:** [5](#0-4) 

It only checks `num_verified() >= quorum` and that the message hash matches the recomputed hash from the caller-supplied VAA body — both of which the attacker controls.

---

### Impact Explanation

An unprivileged attacker can post any fraudulent VAA as verified. Concretely, they can craft a governance VAA with `emitter_chain = GOVERNANCE_CHAIN` and `emitter_address = GOVERNANCE_EMITTER` (both hard-coded constants the attacker can read from the program), a `transfer_fees` payload pointing to their own wallet, and an arbitrary sequence number. After `post_vaa` succeeds, `transfer_fees` passes all checks in `require_valid_governance_vaa`: [6](#0-5) 

The fee collector is drained to the attacker's address. [7](#0-6) 

---

### Likelihood Explanation

The attack requires no privileged access, no guardian keys, and no brute force. The steps are:
1. Compute `sha256("account:SignatureSet")[0..8]` (the Anchor discriminator — deterministic and public).
2. Allocate a system account of size `SignatureSet::compute_size(N)`.
3. Write: discriminator (8 B) + vec length (4 B, LE) + `[0x01; N]` + target `message_hash` (32 B) + `guardian_set_index` (4 B).
4. `SystemInstruction::Assign` the account to the core-bridge program ID.
5. Call `post_vaa` with this account and a matching VAA body.
6. Call `transfer_fees`.

All steps are executable in a single Solana session by any wallet. The blocklist of 16 hardcoded keys in `post_vaa` is trivially bypassed by using a fresh keypair. [8](#0-7) 

---

### Recommendation

1. **Make `signature_set` a PDA in `post_vaa`** derived from the message hash (e.g., seeds `[b"SignatureSet", message_hash]`), so only accounts created through `verify_signatures` with the correct seeds are accepted.
2. **Alternatively, add an `init` or `has_one` constraint** in `post_vaa` that verifies the `signature_set` was created by the program's own `verify_signatures` instruction (e.g., store the payer/creator pubkey in the account and verify it).
3. **Fix `is_initialized()`** to use a dedicated boolean flag or a non-zero discriminator field rather than scanning `sig_verify_successes`, so a pre-seeded account cannot masquerade as initialized.
4. **Prefer the new `init_encoded_vaa` / `verify_encoded_vaa_v1` path**, which does not rely on the `SignatureSet` account pattern at all and performs inline secp256k1 recovery with a quorum check. [9](#0-8) 

---

### Proof of Concept

```rust
// 1. Compute discriminator offline: sha256("account:SignatureSet")[0..8]
let discriminator = anchor_discriminator("SignatureSet"); // 8 bytes

// 2. Craft fraudulent governance VAA body
let fraudulent_body = VaaBody {
    emitter_chain: GOVERNANCE_CHAIN,       // hard-coded constant
    emitter_address: GOVERNANCE_EMITTER,   // hard-coded constant
    sequence: 999999,                      // unused sequence
    payload: transfer_fees_payload(attacker_wallet, amount),
    ..
};
let message_hash = keccak(fraudulent_body.serialize());

// 3. Build SignatureSet account data
let mut data = Vec::new();
data.extend_from_slice(&discriminator);
data.extend_from_slice(&(N as u32).to_le_bytes());  // vec length
data.extend(vec![0x01u8; N]);                        // all true
data.extend_from_slice(message_hash.as_ref());
data.extend_from_slice(&current_guardian_set_index.to_le_bytes());

// 4. Create, write, and assign account to core-bridge program
let sig_set_kp = Keypair::new();
create_account(sig_set_kp.pubkey(), data.len(), payer);
write_data(sig_set_kp.pubkey(), &data);
assign(sig_set_kp.pubkey(), CORE_BRIDGE_PROGRAM_ID);

// 5. Call post_vaa — succeeds because num_verified()=N >= quorum
post_vaa(sig_set_kp.pubkey(), fraudulent_body);

// 6. Call transfer_fees — drains fee_collector to attacker_wallet
transfer_fees(posted_vaa_pda, attacker_wallet);
```
