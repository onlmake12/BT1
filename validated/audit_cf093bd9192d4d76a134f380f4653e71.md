Audit Report

## Title
Unbounded Block Iteration via `difficulty_boundary=0` + `difficulties=[]` Bypasses Size Guard — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
A remote peer can send a crafted `GetLastStateProof` message with `difficulties=[]`, `difficulty_boundary=U256::zero()`, and `last_n_blocks=0`. The upfront size guard evaluates to zero and passes. Because `U256` is unsigned, every block satisfies `total_difficulty >= 0`, causing `get_first_block_total_difficulty_is_not_less_than` to return `start_block_number` immediately and setting `last_n_numbers` to span the entire chain. The server then performs O(N) database lookups, MMR root computations, and memory allocations with no secondary cap, enabling a single unauthenticated peer to exhaust node resources.

## Finding Description

**Size guard bypass (L201–205):** [1](#0-0) 

With `difficulties=[]` (len=0) and `last_n_blocks=0`, the expression evaluates to `0 + 0 = 0 ≤ 1000`. The check passes unconditionally.

**Validation checks all skipped on empty `difficulties` (L254–288):** [2](#0-1) 

- `difficulties.windows(2).any(...)` — empty slice, `any()` returns `false`.
- `difficulties.last().map(...).unwrap_or(false)` — `None`, returns `false`. Boundary check skipped.
- `difficulties.first()` — `None`, start-difficulty check skipped entirely.

**Branch selection (L291–292):** [3](#0-2) 

With `last_n_blocks=0` and any chain with at least one block, `last_block_number - start_block_number > 0`, so the `else` branch is taken.

**Zero boundary causes immediate return of `start_block_number` (L30–33):** [4](#0-3) 

`U256` is unsigned, so `start_total_difficulty >= U256::zero()` is always true. The function returns `Some((start_block_number, ...))` on the very first check, setting `difficulty_boundary_block_number = start_block_number`.

**Adjustment guard does not fire (L313–316):** [5](#0-4) 

`last_block_number - start_block_number < 0` is false for any non-empty chain, so `difficulty_boundary_block_number` remains at `start_block_number`.

**`last_n_numbers` spans the entire chain (L318–319):** [6](#0-5) 

`(start_block_number..last_block_number)` collects every block number — potentially millions on mainnet.

**`complete_headers` iterates all of them with no cap (L132–163):** [7](#0-6) 

For each block number: `snapshot.get_ancestor`, `snapshot.get_block`, and `snapshot.chain_root_mmr(...).get_root()` are called. No secondary size check exists before or inside this loop.

## Impact Explanation

A single unauthenticated peer message forces the server to perform O(N) disk reads (header + block + MMR root per block), O(N) memory allocation for the resulting `Vec<VerifiableHeader>`, and O(N) serialization. On CKB mainnet with millions of blocks, this exhausts CPU, memory, and I/O on the server process. Multiple concurrent requests from different peer connections amplify the effect, making node crash or sustained unresponsiveness easily achievable. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation

The attack requires no credentials, no proof-of-work, and no prior chain state beyond knowing a valid tip hash, which is publicly observable from any block explorer or peer. The crafted message is minimal (empty difficulties array, zero boundary scalar, zero `last_n_blocks`). It is constructable by any peer that can open a connection to the light-client server. The attack is repeatable and can be parallelized across multiple connections.

## Recommendation

1. **Add a secondary size cap on `last_n_numbers` before calling `complete_headers`**: after computing `last_n_numbers`, check `last_n_numbers.len() + sampled_numbers.len() + reorg_last_n_numbers.len() > GET_LAST_STATE_PROOF_LIMIT` and return `StatusCode::MalformedProtocolMessage` if exceeded.
2. **Reject `difficulty_boundary = U256::zero()` explicitly** when `last_block_number - start_block_number > last_n_blocks`, since a zero boundary is semantically meaningless and only enables this bypass.
3. **Update the upfront size guard** to account for the actual computed `last_n_numbers` length rather than only the raw request parameters.

## Proof of Concept

```
Setup: CKB node with light-client server enabled, chain of N blocks (e.g., N=10,000).

Craft GetLastStateProof message:
  last_hash           = current tip hash (publicly observable)
  start_hash          = genesis hash
  start_number        = 0
  last_n_blocks       = 0
  difficulty_boundary = U256::zero()  (0x00...00)
  difficulties        = []            (empty)

Send from any unauthenticated peer connection.

Observed behavior:
  - Size guard: 0 + 0*2 = 0 ≤ 1000 → passes
  - Validation: all checks skipped (empty difficulties)
  - get_first_block_total_difficulty_is_not_less_than(0, N, 0) → returns Some((0, genesis_diff))
  - difficulty_boundary_block_number = 0
  - last_n_numbers = (0..N) → N entries
  - complete_headers called with N block numbers
  - N × (get_ancestor + get_block + chain_root_mmr.get_root) executed
  - N-entry VerifiableHeader Vec allocated and serialized

Expected (correct): server rejects with InvalidRequest or caps response at GET_LAST_STATE_PROOF_LIMIT.
Actual: server processes all N blocks from a single unauthenticated message.
```
