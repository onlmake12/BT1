### Title
Unguarded `set_parameters` Initialization Allows Any Signer to Hijack Stake Cap Authority and Parameters — (`File: pythnet/stake_caps_parameters/programs/stake_caps_parameters/src/lib.rs`)

---

### Summary

The `set_parameters` instruction in the `stake_caps_parameters` program uses `init_if_needed` on the `parameters` PDA and permits any signer to execute the instruction when `current_authority == Pubkey::default()` (i.e., when the account is uninitialized). There is no hard-coded or admin-gated constraint on who the first caller may be. An attacker who calls `set_parameters` before the legitimate deployer can set arbitrary `m`, `z`, and `current_authority` values, permanently seizing control of the stake cap parameters.

---

### Finding Description

The `set_parameters` handler performs the following authority check:

```rust
require!(
    ctx.accounts.signer.key() == stored_parameters.current_authority
        || stored_parameters.current_authority == Pubkey::default(),
    ErrorCode::WrongAuthority
);
**stored_parameters = parameters;
``` [1](#0-0) 

The `parameters` account is declared with `init_if_needed`:

```rust
#[account(
    init_if_needed,
    seeds = ["parameters".as_bytes()],
    bump,
    payer = signer,
    space = Parameters::LEN
)]
pub parameters: Account<'info, Parameters>,
``` [2](#0-1) 

When the PDA has never been written to, Anchor zero-initializes it, making `current_authority == Pubkey::default()`. The second branch of the `require!` then passes unconditionally for **any** signer. The attacker supplies their own pubkey as `current_authority` in the `parameters` argument, and the full struct is written atomically:

```rust
**stored_parameters = parameters;
``` [3](#0-2) 

After this single transaction the attacker is the permanent authority and can set `m` and `z` to any value at will.

The analog to the original report is direct: just as `LiquidateSolvent` accepted any `executor: Signer` without verifying it against an admin-controlled address, `set_parameters` accepts any `signer: Signer` without verifying it against a hard-coded or governance-controlled address when the account is uninitialized. [4](#0-3) 

---

### Impact Explanation

`m` and `z` are the two free parameters of the publisher stake-cap formula consumed by the Pyth staking application:

```typescript
const [publishers, walletAmount, poolConfig, currentEpoch, parameters] =
  await Promise.all([...
    pythnetClient.getStakeCapParameters(),
  ]);
// ...
m: parameters.m,
z: parameters.z,
``` [5](#0-4) 

An attacker who controls `current_authority` can:
- Set `m = 0` or `z = 0` to collapse all publisher stake caps to zero, freezing delegations.
- Set `m` or `z` to `u64::MAX` to cause arithmetic overflow in downstream cap calculations, breaking the staking accounting.
- Permanently lock out the legitimate Pyth governance authority from ever updating these parameters.

This constitutes a **critical staking accounting manipulation** reachable without any privileged access.

---

### Likelihood Explanation

The vulnerability is exploitable on any deployment where the `parameters` PDA has not yet been initialized — including fresh PythNet deployments, devnet/testnet instances, and any future redeployment. The attacker entry path requires only:
1. Monitoring the chain for the PDA to be absent (trivially detectable via RPC).
2. Submitting a single `set_parameters` transaction before the legitimate deployer.

No leaked keys, governance majority, or trusted-role access is required. The attack is a straightforward front-run of the initialization transaction.

---

### Recommendation

Replace the `init_if_needed` + `Pubkey::default()` pattern with a hard-coded upgrade authority constraint, mirroring the fix described in the original report. For example, use Anchor's `address` constraint on the `signer` account:

```rust
#[account(
    mut,
    address = EXPECTED_ADMIN_PUBKEY @ ErrorCode::WrongAuthority
)]
pub signer: Signer<'info>,
```

Alternatively, split initialization into a separate `initialize` instruction that is callable only once and only by a hard-coded governance key, and restrict `set_parameters` to the stored `current_authority` with no `Pubkey::default()` bypass.

---

### Proof of Concept

```rust
// Attacker keypair: attacker_kp
// Target PDA: PARAMETERS_ADDRESS (uninitialized on fresh deployment)

let attacker_params = Parameters {
    current_authority: attacker_kp.pubkey(), // attacker takes authority
    m: 0,                                    // collapses all stake caps
    z: 0,
};

let ix = Instruction::new_with_bytes(
    stake_caps_parameters::ID,
    &stake_caps_parameters::instruction::SetParameters {
        parameters: attacker_params,
    }.data(),
    stake_caps_parameters::accounts::SetParameters {
        signer: attacker_kp.pubkey(),
        parameters: PARAMETERS_ADDRESS,
        system_program: solana_sdk::system_program::ID,
    }.to_account_metas(None),
);

// Single transaction, no special privileges required.
// After execution: current_authority == attacker, m == 0, z == 0.
// Legitimate deployer can no longer update parameters.
```
