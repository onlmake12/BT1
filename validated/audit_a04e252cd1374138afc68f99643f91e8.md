### Title
Excess ETH Paid as Entropy Request Fee Is Permanently Absorbed Into Protocol Pool With No Refund Mechanism - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

In `Entropy.sol`, the `requestHelper` function accepts `msg.value` for fee payment but does not refund any excess ETH beyond the required fee. The entire surplus is silently credited to `_state.accruedPythFeesInWei` — the Pyth protocol's fee pool — with no mechanism for users to recover overpaid amounts. This is structurally analogous to the VaultBooster issue where depositing unsupported tokens results in permanently stuck funds.

---

### Finding Description

In `requestHelper`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The expression `msg.value - providerFee` is assigned entirely to `accruedPythFeesInWei`. When `msg.value > requiredFee`, the surplus (`msg.value - requiredFee`) is silently absorbed into the Pyth protocol fee pool rather than being refunded. The NatSpec on every public entry point (`request`, `requestWithCallback`, `requestV2`) explicitly acknowledges this:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

The `accruedPythFeesInWei` pool is only withdrawable by the Pyth admin via governance — there is no function that allows a user to reclaim their overpaid ETH. [4](#0-3) 

The existing test suite confirms this behavior explicitly — overpaying by 10,000 wei results in that amount going to `accruedPythFees`, not back to the caller: [5](#0-4) 

---

### Impact Explanation

Any user who calls `request`, `requestWithCallback`, or `requestV2` with `msg.value > getFeeV2(provider, gasLimit)` permanently loses the excess ETH. The funds are absorbed into the Pyth protocol fee pool and can only be withdrawn by the Pyth admin — not returned to the original payer. There is no `withdrawExcess` or equivalent function for users.

This is a direct analog to the VaultBooster finding: in both cases, a user sends assets to a contract without a supported recovery path, and those assets become permanently inaccessible to the user.

---

### Likelihood Explanation

Overpayment is a realistic and common occurrence:

1. **Fee changes between estimation and execution**: A provider can call `setProviderFee` at any time. If a user estimates the fee off-chain and the fee decreases before their transaction lands, they overpay.
2. **Integrating contracts**: Smart contracts that wrap Entropy (e.g., the `CoinFlip` example in the developer docs) often forward `msg.value` directly. If the caller sends more than needed, the surplus is lost.
3. **Deliberate overpayment for safety**: Users and integrators commonly add a buffer to ensure transactions succeed, especially on volatile-fee chains. [6](#0-5) 

---

### Recommendation

Refund excess ETH to the caller at the end of `requestHelper`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
uint128 pythFee = requiredFee - providerFee;
_state.accruedPythFeesInWei += pythFee;

// Refund excess
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

Alternatively, enforce `msg.value == requiredFee` exactly (revert on overpayment) so users are never silently overcharged.

---

### Proof of Concept

1. Provider registers with `feeInWei = 100`. Pyth fee is `50`. Total required fee = `150`.
2. User calls `random.request{value: 200}(provider, commitment, false)`.
3. `requestHelper` executes: `providerFee = 100`, `accruedPythFeesInWei += (200 - 100) = 100` (should be `50`).
4. The extra `50 wei` is permanently absorbed into `accruedPythFeesInWei`.
5. User has no function to call to recover the `50 wei`.
6. Only the Pyth admin can withdraw from `accruedPythFeesInWei` via `withdrawFees`, and it goes to an admin-chosen address, not back to the user. [7](#0-6)

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
