### Title
Lazer Price Update Signed Payload Lacks Chain ID Binding, Enabling Cross-Chain Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

In `PythLazer.sol`, the `verifyUpdate` function verifies a signature over `keccak256(payload)` where the payload contains only price data (magic, timestamp, channel, feed data) with **no chain ID or contract address**. Because the same trusted signer key is registered across multiple EVM deployments, an unprivileged relayer can take a valid signed Lazer update from one EVM chain and replay it verbatim on any other EVM chain where the same signer is trusted.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs the following steps:

1. Extracts `payload = update[71 : 71 + payload_len]`
2. Computes `hash = keccak256(payload)`
3. Recovers `signer` from the ECDSA signature `(r, s, v)` over `hash`
4. Checks `isValidSigner(signer)` [1](#0-0) 

The payload structure parsed by `PythLazerLib.parsePayloadHeader` is:

```
PAYLOAD_FORMAT_MAGIC (4 bytes)
timestamp            (8 bytes)
channel              (1 byte)
feedsLen             (1 byte)
... feed data ...
``` [2](#0-1) 

There is **no chain ID, no contract address, and no destination binding** anywhere in the signed payload. The same signed bytes are cryptographically valid on every EVM chain that trusts the same signer key.

The same structural absence exists in the Sui contract's `verify_le_ecdsa_message`, which recovers the public key from `secp256k1_ecrecover(signature, payload, 0)` over a payload with the same chain-agnostic format: [3](#0-2) 

And in the Aptos contract's `verify_message`, which calls `ed25519::signature_verify_strict(&sig, &pk, message)` over a message with no chain binding: [4](#0-3) 

The deployment tooling (`deploy_evm_lazer_contracts.ts`) calls `updateTrustedSigner` with the same signer address across chains, confirming that the same key is trusted on all EVM deployments: [5](#0-4) 

---

### Impact Explanation

An attacker who observes a valid signed Lazer update on chain A (e.g., Ethereum) can submit the identical bytes to `verifyUpdate` on chain B (e.g., Polygon, BNB Chain, Arbitrum). The contract on chain B will:

- Accept the `EVM_FORMAT_MAGIC` check (same constant everywhere)
- Recover the same signer address from the same `(r, s, v)` over the same `keccak256(payload)`
- Pass `isValidSigner` because the same key is registered

The returned `payload` is then consumed by downstream consumer contracts. Consumer contracts that do not independently enforce a freshness window on the embedded `timestamp` will accept a stale or cross-chain-mismatched price. This enables:

- **Price manipulation**: submitting a price from a period of high volatility on chain A to chain B at a moment when the true price on chain B differs
- **Forced incorrect liquidations or under-collateralization** in lending protocols consuming Lazer prices
- **Arbitrage extraction** against DEXes or derivatives protocols using Lazer prices

The `verifyUpdate` function itself performs **no timestamp check**, so the entire freshness responsibility falls on consumer contracts, many of which may not implement it correctly.

---

### Likelihood Explanation

- The attacker entry point is an unprivileged relayer or MEV bot — no special role required.
- The same trusted signer key is deployed across all EVM Lazer contracts (confirmed by deployment scripts).
- The attack requires only observing a mempool or confirmed transaction on one chain and submitting the same calldata on another — trivially achievable with standard cross-chain monitoring infrastructure.
- The attack window is bounded only by the signer's key expiry (`expiresAt`), not by any per-message nonce or chain binding.

---

### Recommendation

Include the destination chain ID and/or contract address in the signed payload so that a signature produced for chain A is cryptographically invalid on chain B. For example, the payload should commit to:

```
keccak256(abi.encode(block.chainid, address(this), payload_bytes))
```

Alternatively, adopt EIP-712 domain separation, which binds signatures to `chainId` and `verifyingContract` by construction.

---

### Proof of Concept

1. Attacker monitors Ethereum for a call to `PythLazer.verifyUpdate(update)` that succeeds.
2. Attacker copies the exact `update` bytes.
3. Attacker submits the same `update` bytes to `PythLazer.verifyUpdate` on Polygon (or any other EVM chain with the same trusted signer).
4. `verifyUpdate` on Polygon recovers the same signer, passes `isValidSigner`, and returns the payload — which carries Ethereum's price at the time of the original signature, not Polygon's current price.
5. Any consumer contract on Polygon that calls `verifyUpdate` with this replayed `update` and does not check `payload.timestamp` against `block.timestamp` will act on the stale/cross-chain price, enabling liquidation manipulation or mispriced trades. [6](#0-5)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }

public(package) fun verify_le_ecdsa_message(
    state: &State,
    clock: &Clock,
    signature: &vector<u8>,
    payload: &vector<u8>,
) {
    let current_cap = state.current_cap();

    // 0 stands for keccak256 hash
    let pubkey = secp256k1_ecrecover(signature, payload, 0);

    // Check if the recovered pubkey is in the trusted signers list
    let trusted_signers = state.trusted_signers(&current_cap);
    let mut maybe_idx = trusted_signers.find_index!(|signer|
        signer.public_key() == &pubkey
    );

    assert!(maybe_idx.is_some(), ESignerNotTrusted);
    let idx = maybe_idx.extract();
    let expires_at_ms = trusted_signers[idx].expires_at_ms();
    assert!(clock.timestamp_ms() < expires_at_ms, ESignerExpired);
}
