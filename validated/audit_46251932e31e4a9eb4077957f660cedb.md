### Title
Missing Request Cancellation/Refund Mechanism Permanently Locks User-Paid Provider Fees in Unfulfilled Requests — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`'s `requestPriceUpdatesWithCallback`, a user pays `msg.value` which is split into a Pyth protocol fee (tracked in `_state.accruedFeesInWei`, withdrawable by admin) and a provider fee stored directly in `req.fee`. If the assigned provider never calls `executeCallback`, the provider fee portion (`req.fee = msg.value - _state.pythFeeInWei`) is permanently locked in the contract. There is no `cancelRequest`, `refundRequest`, or timeout-based recovery mechanism anywhere in the Echo contract or its interface.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the fee accounting is:

```solidity
// Echo.sol line 84
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

// Echo.sol line 99
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

The full `msg.value` enters the contract. Of this:
- `_state.pythFeeInWei` is credited to `_state.accruedFeesInWei` — recoverable by admin via `withdrawFees`.
- `req.fee = msg.value - _state.pythFeeInWei` is stored in the request struct — **only recoverable when `executeCallback` is called**.

In `executeCallback`, the provider is credited:

```solidity
// Echo.sol line 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

If `executeCallback` is **never called**, `req.fee` is never moved into any withdrawable accounting variable. The ETH sits in the contract balance with no path out.

The complete `IEcho` interface confirms there is no cancel or refund function: [3](#0-2) 

The only withdrawal functions are:
- `withdrawFees(uint128 amount)` — admin only, drains `_state.accruedFeesInWei` only.
- `withdrawAsFeeManager(address provider, uint128 amount)` — fee manager only, drains `provider.accruedFeesInWei` only. [4](#0-3) 

Neither function can recover `req.fee` for an unfulfilled request. The developers themselves noted this risk in a TODO comment:

```solidity
// Echo.sol line 155-156
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
``` [5](#0-4) 

---

### Impact Explanation

For every unfulfilled request, `req.fee = msg.value - _state.pythFeeInWei` worth of ETH is permanently locked in the Echo contract. The admin can only recover the Pyth protocol fee slice (`_state.pythFeeInWei`). The user receives no refund and no callback. At scale — e.g., if a provider goes offline with many pending requests — the locked amount is proportional to the number of outstanding requests multiplied by the provider fee per request. There is no governance or admin override to drain these stranded funds.

---

### Likelihood Explanation

The entry path requires only an unprivileged transaction sender calling `requestPriceUpdatesWithCallback`. The trigger — a provider failing to call `executeCallback` — is realistic: providers are external off-chain services that can go offline due to infrastructure failure, key loss, or deliberate abandonment. The exclusivity period (`_state.exclusivityPeriodSeconds`) means that during that window only the assigned provider can fulfill the request; if they are offline during that window, the window closes and any provider can fulfill it, but if all providers are unresponsive or the request is simply abandoned, the funds remain locked indefinitely. No malicious intent is required. [6](#0-5) 

---

### Recommendation

Add a `cancelRequest(uint64 sequenceNumber)` function that:
1. Requires either the original requester calling after a configurable timeout (e.g., `block.timestamp > req.publishTime + CANCEL_TIMEOUT`), or the admin at any time.
2. Refunds `req.fee` to `req.requester`.
3. Refunds `_state.pythFeeInWei` from `_state.accruedFeesInWei` back to `req.requester` (or retains it as a cancellation fee).
4. Calls `clearRequest(sequenceNumber)`.

---

### Proof of Concept

1. Provider registers via `registerProvider(baseFee, feedFee, gasRate)`.
2. User calls `requestPriceUpdatesWithCallback{value: 1 ether}(provider, publishTime, priceIds, gasLimit)`.
   - `_state.accruedFeesInWei += _state.pythFeeInWei` (e.g., 0.001 ETH).
   - `req.fee = 0.999 ETH` stored in request struct.
3. Provider goes offline; `executeCallback` is never called.
4. Admin calls `withdrawFees(0.001 ether)` — succeeds, recovers only the Pyth fee.
5. The remaining `0.999 ETH` (`req.fee`) is permanently locked:
   - `withdrawFees` cannot access it (only tracks `_state.accruedFeesInWei`).
   - `withdrawAsFeeManager` cannot access it (only tracks `provider.accruedFeesInWei`, which was never incremented).
   - No cancel/refund function exists.
6. User's `0.999 ETH` is irrecoverable. [7](#0-6)

interface IEcho is EchoEvents {
    // Core functions
    /**
     * @notice Requests price updates with a callback
     * @dev The msg.value must be equal to getFee(callbackGasLimit)
     * @param provider The provider to fulfill the request
     * @param publishTime The minimum publish time for price updates, it should be less than or equal to block.timestamp + 60
     * @param priceIds The price feed IDs to update. Maximum 10 price feeds per request.
     *        Requests requiring more feeds should be split into multiple calls.
     * @param callbackGasLimit The amount of gas allocated for the callback execution
     * @return sequenceNumber The sequence number assigned to this request
     * @dev Security note: The 60-second future limit on publishTime prevents a DoS vector where
     *      attackers could submit many low-fee requests for far-future updates when gas prices
     *      are low, forcing executors to fulfill them later when gas prices might be much higher.
     *      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
     *      the fee estimation unreliable.
     */
    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable returns (uint64 sequenceNumber);

    /**
     * @notice Executes the callback for a price update request
     * @dev Requires 1.5x the callback gas limit to account for cross-contract call overhead
     * For example, if callbackGasLimit is 1M, the transaction needs at least 1.5M gas + some gas for some other operations in the function before the callback
     * @param providerToCredit The provider to credit for fulfilling the request. This may not be the provider that submitted the request (if the exclusivity period has elapsed).
     * @param sequenceNumber The sequence number of the request
     * @param updateData The raw price update data from Pyth
     * @param priceIds The price feed IDs to update, must match the request
     */
    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable;

    // Getters
    /**
     * @notice Gets the base fee charged by Pyth protocol
     * @dev This is a fixed fee per request that goes to the Pyth protocol, separate from gas costs
     * @return pythFeeInWei The base fee in wei that every request must pay
     */

    /**
