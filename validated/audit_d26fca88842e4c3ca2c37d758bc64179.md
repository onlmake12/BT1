Audit Report

## Title
Unbounded O(n) Pool Scan in `estimate_fee_rate` Blocks Service Loop, Degrading Transaction Throughput — (`tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::estimate_fee_rate` iterates every entry in the pool with no bound. Because the tx-pool service loop `await`s this call before processing the next message, an unprivileged attacker who fills the pool and calls the RPC in a tight loop serialises all subsequent pool operations — transaction admission, block assembly — behind each O(n) scan. No authentication, rate-limit, or iteration cap exists.

## Finding Description
**Root cause — unbounded scan:**
`PoolMap::estimate_fee_rate` (L342–358) iterates the full score-sorted set with no cap:

```rust
let iter = self.entries.iter_by_score().rev();   // no bound
for entry in iter {
    current_block_bytes += entry.inner.size;
    current_block_cycles += entry.inner.cycles;
    if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
        target_blocks -= 1;
        if target_blocks == 0 { return entry.inner.fee_rate(); }
        ...
    }
}
min_fee_rate   // reached after visiting every entry
``` [1](#0-0) 

With minimum-size (~100 B) transactions and `max_block_bytes` ~500 KB, the threshold is crossed only after ~5,000 entries per `target_blocks` increment. When the pool is packed with such transactions the loop falls through to `return min_fee_rate` after visiting every entry.

**Service loop serialisation:**
The `EstimateFeeRate` message handler in the service loop `.await`s the call before processing the next message:

```rust
Message::EstimateFeeRate(Request { responder, arguments: (estimate_mode, enable_fallback) }) => {
    let fee_estimates_result = service
        .estimate_fee_rate(estimate_mode, enable_fallback)
        .await;
    ...
}
``` [2](#0-1) 

While this `.await` is pending, no other message — `submit_transaction`, block commit, pool eviction — is dequeued. Every concurrent RPC call queues behind the in-flight scan. The read lock is also held during the scan (consistent with all other handlers, e.g. `GetAllEntryInfo` at L1001), blocking any concurrent writer. [3](#0-2) 

**No guards on the RPC entry point:**
`enable_fallback` defaults to `true` with no override required: [4](#0-3) 

A search for `rate_limit`, `ratelimit`, and `throttle` in `rpc/src/` returns zero matches — no per-IP quota or per-call entry cap exists anywhere in the RPC layer.

## Impact Explanation
Matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Sustained concurrent calls to `estimate_fee_rate` stall the service loop, delaying `submit_transaction` and `get_block_template`. A node unable to assemble blocks on schedule or relay transactions contributes to network-level propagation delays. The attacker's cost is one-time pool fill (minimum-fee valid transactions) plus essentially free RPC calls.

## Likelihood Explanation
- The endpoint is publicly documented, enabled by default, and requires no credentials.
- `enable_fallback = true` is the default; no special parameter is needed.
- Filling the pool with minimum-fee transactions is cheap; transactions need not be mined.
- The attack is stateless and repeatable: re-call the RPC in a loop with no per-call cost beyond network round-trip.
- No existing guard (authentication, rate-limit, iteration cap) prevents this.

## Recommendation
1. **Cap the scan** in `estimate_fee_rate` with an explicit `take` bound, e.g.:
   ```rust
   let max_entries = target_blocks * MAX_BLOCK_PROPOSALS_LIMIT;
   for entry in iter.take(max_entries) { ... }
   ```
2. **Rate-limit the RPC layer** with a per-IP token bucket for `estimate_fee_rate` and `get_raw_tx_pool`.
3. **Paginate `get_all_entry_info`** (pool.rs L464–487) with optional `limit`/`cursor` parameters to prevent unbounded heap allocation under a single read lock. [5](#0-4) 

## Proof of Concept
1. Populate a `PoolMap` with `MAX_TX_POOL_SIZE / MIN_TX_SIZE` entries of minimum size and minimum fee rate (as done in the existing unit test at `tx-pool/src/component/tests/estimate.rs`). [6](#0-5) 
2. Time `pool_map.estimate_fee_rate(target_blocks, max_block_bytes, max_block_cycles, min_fee_rate)` to confirm O(n) growth with pool size.
3. In an integration test: fill the pool via RPC, then call `estimate_fee_rate` in a tight loop from one client while measuring `submit_transaction` latency from a second client. Observe increasing latency or timeouts on the write path as the service loop is saturated.
4. Confirm `get_block_template` latency degrades proportionally during the attack.
