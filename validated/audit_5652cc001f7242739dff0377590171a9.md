### Title
Missing Chain ID in Lazer Update Signature Hash Enables Cross-Chain Replay of Price Updates - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.sol`'s `verifyUpdate` function computes the signature hash as `keccak256(payload)` with no chain ID, contract address, or nonce bound into the signed message. Because Pyth deploys `PythLazer` on many EVM chains with the same trusted signer key, a valid signed Lazer update captured on one chain can be replayed verbatim on any other EVM chain where that signer is registered, bypassing the intent that each update is chain-specific.

### Finding Description

In `verifyUpdate`, the only data committed to the signature is the raw payload bytes:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The payload itself contains only `FORMAT_MAGIC | timestamp | channel | feedsLen | feed_data` — no `block.chainid`, no contract address, no nonce:

```solidity
uint32 FORMAT_MAGIC = 2479346549;
// ...
timestamp = _readBytes8(update, pos);   // pos 4..12
channel   = _readBytes1(update, pos);   // pos 12
feedsLen  = _readBytes1(update, pos);   // pos 13
``` [2](#0-1) 

Because the hash is chain-agnostic, the identical `(update_bytes, signature)` pair that passes `verifyUpdate` on Ethereum mainnet will also pass on Arbitrum, Base, Optimism, or any other EVM chain where the same trusted signer address is registered.

The same-chain replay is also unrestricted: the contract stores no record of seen hashes, so the same `update` bytes can be submitted to `verifyUpdate` an unlimited number of times on the same chain. [3](#0-2) 

### Impact Explanation

An attacker who monitors Lazer price updates on chain A can replay a recently signed update on chain B within the same timestamp window. Because the payload timestamp is recent, any downstream consumer that guards only on `_timestamp > lastTimestamp` will accept the replayed data. This allows the attacker to inject chain-A prices into chain-B consumers, enabling manipulation of DeFi protocols (lending, perpetuals, options) that rely on Pyth Lazer prices on chain B. The impact is price oracle manipulation across all EVM deployments sharing a trusted signer.

### Likelihood Explanation

Pyth deploys `PythLazer` on many EVM chains and manages trusted signers via governance, making it highly likely the same signer key is registered on multiple chains simultaneously. The attack requires only the ability to observe on-chain or off-chain Lazer update bytes (publicly available) and submit a transaction on the target chain — no privileged access is needed. Lazer updates are published at high frequency (200 ms channels), giving an attacker a continuous stream of replayable material.

### Recommendation

Bind the chain ID and contract address into the signed hash, following EIP-712 domain separator conventions:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

This ensures a signature produced for one chain/contract deployment is cryptographically invalid on any other. Optionally, a per-update sequence number or nonce can be added to prevent same-chain replay within the timestamp window.

### Proof of Concept

1. Deploy `PythLazer` on two EVM chains (e.g., Ethereum and Arbitrum) with the same `trustedSigner`.
2. On Ethereum, call `verifyUpdate{value: fee}(update)` with a freshly signed Lazer update — it succeeds and returns `payload`.
3. Without any modification, submit the identical `update` bytes to `verifyUpdate` on Arbitrum.
4. The call succeeds: `keccak256(payload)` is identical, `ECDSA.tryRecover` returns the same `signer`, and `isValidSigner(signer)` returns `true` because the same key is registered.
5. The returned `payload` (with its original timestamp) is accepted by any downstream consumer that checks only `_timestamp > lastTimestamp`. [4](#0-3) [5](#0-4)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
