### Title
Onion3 Address Bypass in `fetch_random` Allows Permanent Feeler Slot Exhaustion — (`network/src/peer_store/addr_manager.rs`)

### Summary

The `None` branch of `AddrManager::fetch_random` unconditionally returns any Onion3 address that passes the caller's filter, regardless of whether `is_connectable` returns `false`. Because `fetch_addrs_to_feeler` adds no independent `is_connectable` guard, an attacker who advertises Onion3 multiaddrs via the discovery protocol can permanently occupy all `FEELER_CONNECTION_COUNT = 10` feeler dial slots per tick, preventing legitimate peer discovery.

---

### Finding Description

**The bypass — `addr_manager.rs` lines 74–88:**

In `fetch_random`, addresses that do not resolve to a socket address (the `None` branch) are subject to a weakened check:

```rust
None => {
    if filter(&addr_info) {
        if addr_info.is_connectable(now_ms)
            || addr_info
                .addr
                .iter()
                .any(|p| matches!(p, Protocol::Onion3(_)))   // ← bypass
        {
            addr_infos.push(addr_info);
        }
    }
}
``` [1](#0-0) 

An Onion3 address is returned whenever `filter` passes, even when `is_connectable` returns `false`.

**When does `is_connectable` return `false`?**

```rust
if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
    return false;
}
``` [2](#0-1) 

A freshly-advertised, never-connected Onion3 address becomes permanently non-connectable after `ADDR_MAX_RETRIES` failed dial attempts — but the Onion3 branch ignores this.

**`fetch_addrs_to_feeler` adds no compensating guard:**

```rust
let filter = |peer_addr: &AddrInfo| {
    filter(peer_addr)
        && ...
        && !peer_addr.tried_in_last_minute(now_ms)
        && !peer_addr.connected(|t| t > addr_expired_ms)
};
self.addr_manager.fetch_random(count, filter)
``` [3](#0-2) 

The only rate-limit is `tried_in_last_minute` (60-second cooldown). After 60 seconds, the same permanently-unreachable Onion3 address is eligible again.

**`dial_feeler` marks and dials without eviction:**

```rust
let paddrs = peer_store.fetch_addrs_to_feeler(FEELER_CONNECTION_COUNT, filter);
for paddr in paddrs.iter() {
    if let Some(paddr) = peer_store.mut_addr_manager().get_mut(&paddr.addr) {
        paddr.mark_tried(now_ms);
    }
}
``` [4](#0-3) 

`mark_tried` increments `attempts_count` and sets `last_tried_at_ms`, but never removes the address.

**`check_purge` only runs when the store is full, and re-advertisement resets `attempts_count`:**

`check_purge` uses `is_connectable` directly (no Onion3 bypass), so it *would* evict these addresses — but only when `addr_manager.count() >= ADDR_COUNT_LIMIT`. More critically, `AddrManager::add` replaces an existing entry when `addr_info.last_connected_at_ms >= exist_last_connected_at_ms`. Since both are `0`, re-advertising the same Onion3 address resets `attempts_count` to `0`, restarting the cycle. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

An attacker with a single P2P connection can send `Nodes` discovery messages containing ≥10 Onion3 multiaddrs. These fill the feeler pool. Every tick, `dial_feeler` exhausts its entire `FEELER_CONNECTION_COUNT = 10` budget on unreachable Onion3 addresses. [7](#0-6) 

Legitimate feeler connections — which are the primary mechanism for discovering and validating new peers — are starved. The victim node cannot expand its peer set, degrading its view of the network and its ability to propagate blocks and transactions.

---

### Likelihood Explanation

- **Entry point:** Standard P2P discovery (`Nodes` message) — no privilege required.
- **Cost:** Sending a handful of multiaddr strings; no PoW, no stake, no key material.
- **Persistence:** The 60-second cooldown means 10 addresses suffice to saturate 10 feeler slots indefinitely. Re-advertising resets `attempts_count`, preventing natural eviction even when the store is full.
- **Victim precondition:** Onion transport disabled or Tor unreachable (common in production deployments).

---

### Recommendation

1. **Remove the unconditional Onion3 bypass.** Apply `is_connectable` to Onion3 addresses the same as all others. If Onion3 addresses need special handling (e.g., no socket-addr conversion), introduce a separate, bounded Onion3 address pool with its own retry cap.
2. **Evict non-connectable addresses proactively**, not only when the store is full. A background sweep or eviction on `mark_tried` when `attempts_count >= ADDR_MAX_RETRIES && last_connected_at_ms == 0` would close the gap.
3. **Rate-limit per-source address injection** in the discovery handler to bound how many addresses a single peer can contribute.

---

### Proof of Concept

```rust
// Pseudocode unit test
let mut peer_store = PeerStore::default();
let onion3_addr: Multiaddr = "/onion3/...".parse().unwrap();

// Attacker advertises via discovery
peer_store.add_addr(onion3_addr.clone(), Flags::COMPATIBILITY).unwrap();

// Simulate ADDR_MAX_RETRIES failed dials
let now_ms = unix_time_as_millis();
for _ in 0..5 {
    if let Some(info) = peer_store.mut_addr_manager().get_mut(&onion3_addr) {
        info.mark_tried(now_ms - 120_000); // older than 1 minute
    }
}

// addr_info.is_connectable() is now false (attempts_count=5 >= ADDR_MAX_RETRIES, last_connected=0)
// But fetch_addrs_to_feeler still returns it due to Onion3 bypass:
let results = peer_store.fetch_addrs_to_feeler(10, |_| true);
assert!(!results.is_empty(), "Onion3 addr returned despite being permanently unreachable");
```
