Audit Report

## Title
Attacker-Controlled Snappy Varint Enables Up-to-8MB Pre-Decompression Heap Allocation Per Connection — (`network/src/compress.rs`)

## Summary
In `LengthDelimitedCodecWithCompress::decode`, `BytesMut::zeroed(decompressed_bytes_len)` is allocated unconditionally based on the attacker-supplied snappy varint header before any validation of the compressed payload. An unprivileged remote peer can trigger a ~7.9MB allocation with an ~11-byte wire frame. With default max inbound peers (~117), simultaneous crafted frames produce ~924MB peak RSS; sustained reconnect-and-repeat causes continuous allocator churn that can crash or severely degrade the node.

## Finding Description
In `network/src/compress.rs` line 233, `decompress_len(&data[1..])` reads the uncompressed length from the snappy stream varint — a value entirely under attacker control. The only guard is the `MAX_UNCOMPRESSED_LEN` (8MB) ceiling at line 235. If the claimed length passes that check, line 242 executes `BytesMut::zeroed(decompressed_bytes_len)` unconditionally. Only after this allocation does line 243 attempt actual decompression, which fails on a garbage payload, returning `Err` and closing the connection — but the allocation has already occurred.

The `max_frame_length` guard (2MB for Sync, 4MB for RelayV3, per `support_protocols.rs` lines 129–130) is enforced by the `length_delimited` codec on the **compressed wire frame size**, not on the claimed decompressed size. A crafted frame of ~11 bytes (1 flag byte + 4-byte snappy varint claiming 7.9MB + 3 garbage bytes) trivially passes this check.

When `decode` returns `Err`, tentacle closes the connection at the transport layer. The application-layer `ban_peer` path (defined in `protocols/mod.rs` line 684) is never reached for codec-level errors, so the attacker can reconnect immediately and repeat.

## Impact Explanation
This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

- Per-connection: one ~11-byte frame → `BytesMut::zeroed(7,864,320)` → decompression error → disconnect → memory freed. Peak RSS spike per connection.
- Simultaneous: with ~117 default max inbound connections all sending simultaneously: 117 × 7.9MB ≈ **924MB peak RSS**. On nodes with ≤1–2GB RAM this is sufficient to trigger OOM.
- Sustained: no codec-layer ban prevents immediate reconnect. Continuous allocator churn degrades all P2P protocol processing (Sync, RelayV3, LightClient, Filter) and can cause the node process to be killed by the OS OOM killer.

## Likelihood Explanation
The attack requires only a TCP connection to the default P2P port 8115, which is publicly exposed. No authentication, no PoW, no key material, no prior relationship with the target node is needed. The crafted frame is trivial to construct (a few bytes). The default `max_peers = 125` provides no meaningful barrier — an attacker can open 117 inbound connections from a single host or distributed IPs. The attack is fully repeatable with no per-attempt cost beyond a TCP handshake.

## Recommendation
Move the allocation **after** a lightweight structural pre-validation of the compressed payload (e.g., verify the snappy stream identifier bytes before trusting the varint), or replace `BytesMut::zeroed(decompressed_bytes_len)` with a streaming/lazy decompressor that does not pre-allocate the full claimed size. At minimum, add a codec-layer connection ban or exponential backoff when `decode` returns `Err` for a compressed frame, to prevent immediate reconnect-and-repeat. The same issue exists in `Message::decompress` at line 81 (`vec![0; decompressed_bytes_len]`) and should be fixed in both paths.

## Proof of Concept
```python
import socket, struct, time

# Snappy varint encoding of 7,864,320 (just under 8MB MAX_UNCOMPRESSED_LEN)
val = 7_864_320
varint = b''
while val > 0x7F:
    varint += bytes([(val & 0x7F) | 0x80])
    val >>= 7
varint += bytes([val])

# Frame: COMPRESS_FLAG=0x80, snappy varint, garbage payload
payload = bytes([0x80]) + varint + b'\xff\xff\xff'

# LengthDelimited 4-byte big-endian length prefix
frame = struct.pack('>I', len(payload)) + payload

# After tentacle handshake on Sync/RelayV3 protocol channel:
# Each send triggers BytesMut::zeroed(7,864,320) on the target node.
# Open 117 simultaneous connections, each sending this frame once
# → ~924MB peak RSS. Reconnect immediately after disconnect and repeat.
for _ in range(117):
    s = socket.create_connection(('TARGET_IP', 8115))
    # [complete tentacle/yamux/secio handshake]
    s.sendall(frame)
    time.sleep(0.01)
```

The minimal unit test: write a Rust test that calls `LengthDelimitedCodecWithCompress::decode` directly with the crafted bytes and measures allocator behavior before and after — the allocation of ~7.9MB followed by an `Err` return is directly observable.
