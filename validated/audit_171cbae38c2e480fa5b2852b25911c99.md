### Title
Unauthorized `write_authority` Hijack in `init_encoded_vaa` Allows Theft of Relayer Rent Lamports — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/init_encoded_vaa.rs`)

### Summary
`init_encoded_vaa` accepts any signer as `write_authority` with no binding to the `encoded_vaa` account's creator. An attacker who observes a freshly-created, zeroed account owned by the core-bridge program can call `init_encoded_vaa` first, set themselves as `write_authority`, and then drain all lamports via `close_encoded_vaa`.

### Finding Description

The `InitEncodedVaa` account struct requires only three things of the `encoded_vaa` account: [1](#0-0) 

1. It is owned by `crate::ID`
2. Its data length exceeds `EncodedVaa::VAA_START`
3. Its header bytes are all zero [2](#0-1) 

There is **no constraint** that `write_authority` has any relationship to the account's creator or funder. The `write_authority` field is simply a `Signer` — any keypair that signs the transaction qualifies. [3](#0-2) 

On success, `write_authority` is written directly into the account header: [4](#0-3) 

`close_encoded_vaa` then enforces only that the caller's key matches the stored `write_authority`, and transfers all lamports to that caller: [5](#0-4) [6](#0-5) 

The `close_account` utility unconditionally moves all lamports from the account to `sol_destination` (the `write_authority`): [7](#0-6) 

The design explicitly anticipates that account creation happens outside the program: [8](#0-7) 

This means the window between `create_account` and `init_encoded_vaa` is an intended part of the protocol flow, not an edge case.

### Impact Explanation
The attacker steals the full rent-exempt lamport deposit that the relayer placed into the `encoded_vaa` account. On mainnet, a typical VAA buffer account holds several thousand lamports. At scale (many VAAs), this compounds. The relayer's VAA processing is also disrupted, causing a denial-of-service on the relay path.

### Likelihood Explanation
The attack does not require traditional Ethereum-style mempool front-running. The attacker only needs to:
1. Monitor on-chain for accounts owned by the core-bridge program ID with zeroed header bytes (trivially done via `getProgramAccounts` with a `memcmp` filter on the first `VAA_START` bytes).
2. Call `init_encoded_vaa` on any such account before the legitimate relayer does.

No privileged access, leaked keys, or governance majority is required. The attacker is fully unprivileged.

### Recommendation
Bind `write_authority` to the `encoded_vaa` account at creation time. The standard Solana pattern is to derive the `encoded_vaa` account as a PDA seeded by `write_authority.key()` (and a nonce), so only the holder of `write_authority` can produce the correct PDA address. Alternatively, add an explicit constraint requiring that `write_authority` is the account's current lamport owner/funder, verified via a stored field set during `create_account`.

### Proof of Concept

```
1. Relayer: system_program::create_account(
       from=relayer, to=K, lamports=rent_exempt, space=VAA_SIZE, owner=core_bridge_id
   )
   // K is now owned by core_bridge, data all zeros, funded by relayer

2. Attacker (observing K on-chain via getProgramAccounts):
   init_encoded_vaa(
       write_authority=attacker,   // attacker signs
       encoded_vaa=K               // account created by relayer
   )
   // Passes: owner=core_bridge ✓, zeroed ✓, size ✓
   // Writes write_authority=attacker into K's header

3. Relayer: init_encoded_vaa(write_authority=relayer, encoded_vaa=K)
   // Fails: AccountNotZeroed (header no longer zero)

4. Attacker: close_encoded_vaa(write_authority=attacker, encoded_vaa=K)
   // write_authority_unsafe(K) == attacker ✓
   // All lamports transferred to attacker
```

pub(crate) fn close_account(info: &AccountInfo, sol_destination: &AccountInfo) -> Result<()> {
    // Transfer tokens from the account to the sol_destination.
    let dest_starting_lamports = sol_destination.lamports();
    **sol_destination.lamports.borrow_mut() =
        dest_starting_lamports.checked_add(info.lamports()).unwrap();
    **info.lamports.borrow_mut() = 0;

    info.assign(&system_program::ID);
    info.resize(0).map_err(Into::into)
}
