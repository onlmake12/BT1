### Title
Fee Consumed Without Price Update on Stale VAA — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

`update_single_price_feed` unconditionally deposits the caller's fee into the `PriceInfoObject` **before** performing the freshness check. If the submitted price update is stale (its `publish_time` ≤ the cached timestamp), the fee is silently consumed with no price update and no event emitted. Because no withdrawal mechanism exists anywhere in the Sui contracts, the deposited SUI is permanently locked inside the `PriceInfoObject`. An adversary can deliberately trigger this condition for any victim by front-running with a fresher VAA.

---

### Finding Description

The execution order in `update_single_price_feed` is:

1. **Line 274** — assert `base_update_fee <= coin::value(fee)` (fee sufficiency check passes)
2. **Line 277** — `price_info::deposit_fee_coins(price_info_object, fee)` — fee is irrevocably joined into the `PriceInfoObject`'s `FEE_STORAGE_KEY` dynamic field
3. **Line 287** — `update_cache(...)` is called, which internally calls `is_fresh_update` [1](#0-0) 

Inside `update_cache`, if `is_fresh_update` returns `false` (i.e., `update_timestamp <= cached_timestamp`), the function body is a no-op — no revert, no refund, no event: [2](#0-1) 

`is_fresh_update` uses strict greater-than, so an equal or older timestamp silently passes through: [3](#0-2) 

`deposit_fee_coins` performs a `coin::join` into the object's dynamic field with no corresponding withdrawal path: [4](#0-3) 

A search across all Sui Move sources confirms there is no `withdraw_fee_coins`, `collect_fees`, or equivalent function. The `set_fee_recipient.move` module itself carries the comment: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore"* — confirming the fee-collection path was intentionally removed, leaving deposited SUI with no egress: [5](#0-4) 

---

### Impact Explanation

Any SUI deposited via a stale `update_single_price_feed` call is permanently locked inside the target `PriceInfoObject`. The `fee_recipient_address` field in `State` exists but is never read by any fee-transfer logic: [6](#0-5) 

The attacker gains no direct financial benefit (this is a griefing/DoS-of-funds attack), but the victim suffers an irreversible loss of their update fee. At scale — e.g., an integrator calling `update_single_price_feed` in a loop across many feeds — the cumulative locked SUI can be significant.

---

### Likelihood Explanation

The attack requires only:

1. **A valid, newer VAA** — freely obtainable from Pyth's public Hermes price service at any time.
2. **Transaction ordering** — on Sui, an adversary submits their fresher-VAA transaction with higher gas priority before the victim's transaction lands. Sui's parallel execution model does not prevent this; both transactions touch the same shared `PriceInfoObject`, so they are sequenced, and the adversary's can be ordered first.

No privileged role, leaked key, or governance majority is required. The entrypoint (`update_single_price_feed`) is fully `public`. [7](#0-6) 

---

### Recommendation

Move the freshness check **before** the fee deposit, or refund the fee when `is_fresh_update` returns `false`. The cleanest fix is:

```move
// 1. Check freshness first (abort early, no fee consumed)
let cur_price_info = /* find matching entry */;
assert!(is_fresh_update(cur_price_info, price_info_object), E_NO_FRESH_UPDATE);

// 2. Only then deposit fee
price_info::deposit_fee_coins(price_info_object, fee);

// 3. Apply update
update_cache(latest_only, cur_price_info, price_info_object, clock);
```

Alternatively, restore a fee-withdrawal path so that locked SUI can be recovered by the fee recipient, mitigating the permanent-lock impact even if the ordering is not changed.

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {
        // Get the timestamp of the update's current price
        let price_feed = price_info::get_price_feed(update);
        let update_timestamp = price::get_timestamp(&price_feed::get_price(price_feed));

        // Get the timestamp of the cached data for the price identifier
        let cached_price_info = price_info::get_price_info_from_price_info_object(price_info_object);
        let cached_price_feed =  price_info::get_price_feed(&cached_price_info);
        let cached_timestamp = price::get_timestamp(&price_feed::get_price(cached_price_feed));

        update_timestamp > cached_timestamp
    }

    public fun get_balance(price_info_object: &PriceInfoObject): u64 {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);
        coin::value(fee)
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
