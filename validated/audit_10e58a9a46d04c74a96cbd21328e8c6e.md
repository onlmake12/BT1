### Title
Inconsistent `ScriptHashType` Consensus Validation in `ScriptHashTypeVerifier` — Only Lock Script Checked, Type Script Omitted - (File: verification/src/transaction_verifier.rs)

### Summary

`ScriptHashTypeVerifier::verify()` is documented as verifying that the `ScriptHashType` of transaction outputs is within the range permitted by current consensus rules. However, it only validates the **lock script** hash type of each output against `ENABLED_SCRIPT_HASH_TYPE`, while the **type script** hash type of the same output is never checked. This is directly analogous to the reported `balanceOfNFT` inconsistency: a protection is applied to one sibling field but silently omitted from the other. Any RPC caller or tx-pool submitter can craft a transaction whose outputs carry a type script with a non-permitted hash type, bypass this non-contextual gate, and inject the transaction into the tx-pool.

### Finding Description

In `verification/src/transaction_verifier.rs`, the `ScriptHashTypeVerifier` struct is constructed as part of `NonContextualTransactionVerifier` and its `verify()` method is called unconditionally during non-contextual transaction verification:

```rust
// verification/src/transaction_verifier.rs  lines 94–101
pub fn verify(&self) -> Result<(), Error> {
    self.version.verify()?;
    self.size.verify()?;
    self.empty.verify()?;
    self.duplicate_deps.verify()?;
    self.outputs_data_verifier.verify()?;
    self.script_hash_type.verify()?;   // <-- ScriptHashTypeVerifier
    Ok(())
}
```

The `ScriptHashTypeVerifier::verify()` implementation iterates over every output and checks only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  lines 796–814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
```

`output.type_()` — the optional type script attached to the same output — is never inspected here. A `CellOutput` carries both a mandatory lock script and an optional type script; both have a `hash_type` field that must conform to the same consensus-gated `ENABLED_SCRIPT_HASH_TYPE` set. The omission means a transaction whose outputs contain a type script with a hash type value outside the currently enabled set (e.g., a future or reserved hash type not yet activated by a hardfork) will pass `NonContextualTransactionVerifier` entirely.

By contrast, the lower-level structural check in `util/gen-types/src/extension/check_data.rs` is symmetric — it validates both scripts:

```rust
// util/gen-types/src/extension/check_data.rs  lines 24–28
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
```

`check_data` validates binary well-formedness of both scripts, but it does **not** enforce the consensus-level `ENABLED_SCRIPT_HASH_TYPE` gate. That gate is exclusively the responsibility of `ScriptHashTypeVerifier`, which is where the asymmetry lives.

The non-contextual verification path is invoked for every transaction entering the tx-pool, including those submitted via the JSON-RPC `send_transaction` endpoint and those relayed by peers through `sync/src/relayer/transactions_process.rs`. Both paths call `non_contextual_verify` (tx-pool/src/process.rs lines 318–333), which in turn calls `NonContextualTransactionVerifier::verify()`.

### Impact Explanation

A transaction with an output whose type script carries a non-permitted `hash_type` value bypasses the only non-contextual consensus gate designed to reject it. Such a transaction is admitted to the pending pool. When the block assembler selects it and the `TransactionScriptsVerifier` attempts to resolve the type script's code cell using the unsupported hash type, execution will fail — but only at the contextual/script-execution stage, not at admission time. The practical consequences are:

- **Tx-pool pollution**: Malformed transactions occupy pool slots and consume memory until eviction.
- **Block assembly interference**: If the assembler selects such a transaction before the contextual verifier rejects it, the assembly cycle is wasted and the transaction must be re-evicted.
- **Inconsistent peer behavior**: Nodes that apply the check only to lock scripts will accept and relay transactions that a stricter implementation would reject at the gate, creating divergent mempool states across the network.

**Impact: Medium** — no direct fund loss, but consensus-level invariants are violated at the admission layer, and the tx-pool can be polluted by any unprivileged submitter.

### Likelihood Explanation

The entry path requires no privilege: any user with access to the `send_transaction` RPC endpoint or any peer that can relay transactions can craft a `CellOutput` with an arbitrary `hash_type` byte in its type script. Constructing such a transaction is trivial with the molecule serialization library. The check is missing silently — there is no error, no log, and no rejection — so the attacker receives no feedback that the gate was bypassed.

**Likelihood: Medium** — trivially reachable by any tx-pool submitter or relaying peer; requires only knowledge of the molecule encoding format.

### Recommendation

In `ScriptHashTypeVerifier::verify()`, extend the loop body to also validate `output.type_()` when it is present, mirroring the existing lock-script check:

```rust
// After checking output.lock(), also check output.type_():
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

This makes the non-contextual gate symmetric for both script fields of every output, consistent with the intent expressed in the verifier's own doc comment and with the symmetric treatment already present in `check_data`.

### Proof of Concept

1. Construct a `Transaction` with one output whose lock script uses a valid, enabled `hash_type` (e.g., `0x00` = `Data`) and whose type script uses a reserved or future `hash_type` value not present in `ENABLED_SCRIPT_HASH_TYPE` (e.g., `0x04` or any value outside the enabled set).
2. Serialize the transaction using the molecule encoding.
3. Submit via `send_transaction` RPC or inject via the relay protocol.
4. Observe that `NonContextualTransactionVerifier::verify()` returns `Ok(())` — the transaction is accepted into the pending pool without error.
5. Confirm that an equivalent transaction whose **lock** script carries the same invalid `hash_type` is correctly rejected with `TransactionError::ScriptHashTypeNotPermitted`.

The asymmetric acceptance/rejection behavior between lock and type scripts at the same non-contextual gate demonstrates the inconsistency.
