### Title
Permissionless Sequence-Skip in `execute_posted_vaa` Permanently Blocks Any Skipped Governance VAA — (`governance/remote_executor/programs/remote-executor/src/lib.rs`)

---

### Summary

`execute_posted_vaa` is callable by any unprivileged signer. Its only replay guard is a strictly-greater-than sequence check (`posted_vaa.sequence > claim_record.sequence`). Because the handler immediately writes the executed sequence into `claim_record.sequence`, any attacker who executes VAA seq=N+1 before seq=N advances the cursor past N, making seq=N permanently unexecutable. Any governance action encoded in seq=N — including reward distribution — is frozen forever.

---

### Finding Description

In `execute_posted_vaa`:

```rust
// lib.rs line 32
claim_record.sequence = posted_vaa.sequence;
```

The account constraint on line 65:

```rust
constraint = posted_vaa.sequence > claim_record.sequence
    @ ExecutorError::NonIncreasingSequence
```

`ClaimRecord` defaults to `sequence: u64 = 0` (via `#[derive(Default)]`). [1](#0-0) 

The `payer` field is the only signer and carries no authority check — any wallet can call this instruction:

```rust
#[account(mut)]
pub payer: Signer<'info>,
``` [2](#0-1) 

The `claim_record` PDA is keyed only on `emitter_address`, so it is shared across all sequences from a given emitter:

```rust
seeds = [CLAIM_RECORD_SEED.as_bytes(), &posted_vaa.emitter_address], bump
``` [3](#0-2) 

**Attack path:**

1. Governance emitter posts VAA seq=N (e.g., reward distribution) and VAA seq=N+1 (any valid governance action). Both are verified by Wormhole guardians and visible on-chain.
2. Attacker calls `execute_posted_vaa` with the seq=N+1 account. Guard: `N+1 > 0` → passes. `claim_record.sequence` is set to `N+1`.
3. Legitimate relayer (or anyone) attempts `execute_posted_vaa` with seq=N. Guard: `N > N+1` → **false** → `NonIncreasingSequence` error. Permanently blocked. [4](#0-3) 

The existing test suite already demonstrates that a lower-sequence VAA is permanently rejected after a higher one executes: [5](#0-4) 

---

### Impact Explanation

Any governance VAA whose sequence is skipped is permanently unexecutable. If that VAA encodes OIS reward parameters or a token transfer to the reward custody, those tokens/parameters are frozen forever with no recovery path — the `claim_record` cannot be reset and there is no admin override in the program.

---

### Likelihood Explanation

- Both VAAs must be posted on-chain (Wormhole-verified) before the attack. This is a normal operational state whenever governance issues consecutive instructions.
- The attacker needs only to submit one permissionless transaction before the legitimate relayer. No special privilege, no key, no Sybil attack required.
- The `crank_pythnet_relayer` processes VAAs sequentially; a brief front-run window exists between posting and execution. [6](#0-5) 

---

### Recommendation

Replace the strictly-greater-than check with a strictly-equal-to check (`posted_vaa.sequence == claim_record.sequence + 1`), enforcing gapless sequential execution. This prevents any sequence from being skipped while still blocking replays. Alternatively, use per-VAA PDA claim accounts (keyed on `[emitter, chain, sequence]`) as the Wormhole core bridge itself does, so each VAA is independently claimable without a shared cursor. [7](#0-6) 

---

### Proof of Concept

```rust
// Pseudocode unit test
let emitter = Keypair::new();
let vaa_n   = post_vaa(&sim, &emitter, sequence = N,   payload = reward_distribution_ix);
let vaa_n1  = post_vaa(&sim, &emitter, sequence = N+1, payload = noop_ix);

// Attacker executes N+1 first
sim.execute_posted_vaa(&vaa_n1, &[], ExecutorAttack::None).await.unwrap();
assert_eq!(sim.get_claim_record(emitter.pubkey()).await.sequence, N+1);

// Legitimate relayer now tries N — permanently blocked
let err = sim.execute_posted_vaa(&vaa_n, &[], ExecutorAttack::None)
    .await.unwrap_err().unwrap();
assert_eq!(err, ExecutorError::NonIncreasingSequence.into());

// Reward custody balance never changes
assert_eq!(sim.get_account(reward_custody).await.unwrap().lamports, initial_balance);
```

    pub fn execute_posted_vaa(ctx: Context<ExecutePostedVaa>) -> Result<()> {
        let posted_vaa = &ctx.accounts.posted_vaa;
        let claim_record = &mut ctx.accounts.claim_record;
        claim_record.sequence = posted_vaa.sequence;

        let payload = ExecutorPayload::try_from_slice(&posted_vaa.payload)?;
        payload.check_header()?;

        let (_, bump) = Pubkey::find_program_address(
            &[EXECUTOR_KEY_SEED.as_bytes(), &posted_vaa.emitter_address],
            &id(),
        );

        for instruction in payload.instructions.iter().map(Instruction::from) {
            // TO DO: We currently pass `remaining_accounts` down to the CPIs, is there a more efficient way to do it?
            invoke_signed(
                &instruction,
                ctx.remaining_accounts,
                &[&[
                    EXECUTOR_KEY_SEED.as_bytes(),
                    &posted_vaa.emitter_address,
                    &[bump],
                ]],
            )?;
        }
        Ok(())
    }
