### Title
Provider Fee Portion of Unfulfilled Requests Permanently Locked — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, when a user calls `requestPriceUpdatesWithCallback`, the total `msg.value` is split: the Pyth protocol fee is immediately credited to `_state.accruedFeesInWei`, while the provider fee is stored in `req.fee` inside the request struct. If `executeCallback` is never called for a request, the provider fee portion is permanently locked in the contract with no recovery path — no cancellation, no timeout, and no admin rescue function covers `req.fee` balances.

---

### Finding Description

In `requestPriceUpdatesWithCallback`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);   // line 84
_state.accruedFeesInWei += _state.pythFeeInWei;                  // line 99
```

The Pyth fee is immediately tracked in `_state.accruedFeesInWei`. The provider fee is stored only in `req.fee` inside the request struct.

The only path that ever credits `req.fee` to a withdrawable balance is `executeCallback`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);   // lines 161-162
```

The two withdrawal functions are:

- `withdrawFees(amount)` — drains `_state.accruedFeesInWei` (Pyth fees only)
- `withdrawAsFeeManager(provider, amount)` — drains `_state.providers[provider].accruedFeesInWei` (provider fees from *completed* callbacks only)

Neither function can access `req.fee` stored in pending/unfulfilled requests. There is no `cancelRequest`, no user refund path, and no timeout mechanism. The contract's own TODO comment at lines 155–160 acknowledges this:

```
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
// TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
```

The contract balance invariant is therefore broken: `address(echo).balance > _state.accruedFeesInWei + Σ provider.accruedFeesInWei` whenever any request is pending, and the gap (Σ `req.fee` for unfulfilled requests) is permanently inaccessible.

---

### Impact Explanation

Any ETH paid as the provider fee portion for requests that are never fulfilled is permanently locked in the Echo contract. Neither the user (no refund), the provider (no credit until callback), nor the admin (no rescue function) can recover these funds. Over time, as providers stop operating or abandon requests, the locked amount grows monotonically with no recovery mechanism.

---

### Likelihood Explanation

**High likelihood of occurrence:**

1. **Permissionless provider registration** — any address can call `registerProvider` and begin accepting fee-paying requests.
2. **No fulfillment enforcement** — there is no on-chain penalty, timeout, or slashing for providers who never call `executeCallback`.
3. **Griefing/abandonment** — a malicious actor registers as a provider, attracts user requests, and simply never fulfills them. Each unfulfilled request permanently locks `req.fee` wei.
4. **Honest provider failure** — even a legitimate provider that goes offline leaves all in-flight `req.fee` balances permanently inaccessible.

---

### Recommendation

1. **Add a user cancellation / refund path**: After a configurable timeout (e.g., `block.timestamp > req.publishTime + maxFulfillmentDelay`), allow the original requester to call a `cancelRequest` function that refunds `req.fee` to the requester and clears the request.
2. **Track pending fees separately**: Maintain a `pendingProviderFees` accumulator so the invariant `address(this).balance == accruedFeesInWei + Σ provider.accruedFeesInWei + pendingProviderFees` is always checkable.
3. **Admin rescue**: Add a governance-controlled function to sweep unclaimed fees from requests older than a threshold into `accruedFeesInWei`.

---

### Proof of Concept

```
1. Attacker calls registerProvider(baseFee, feedFee, gasRate) → becomes a registered provider.
2. User calls requestPriceUpdatesWithCallback{value: F}(attacker, ...) where F = pythFeeInWei + providerFee.
   - _state.accruedFeesInWei += pythFeeInWei   ✓ (tracked)
   - req.fee = F - pythFeeInWei                 ✓ (stored in request, NOT in any withdrawable slot)
   - address(echo).balance increases by F
3. Attacker never calls executeCallback.
4. Admin calls withdrawFees(accruedFeesInWei) → withdraws only pythFeeInWei.
5. address(echo).balance still holds providerFee (= req.fee) with no function able to withdraw it.
6. Repeat step 2 N times → N * providerFee is permanently locked.
