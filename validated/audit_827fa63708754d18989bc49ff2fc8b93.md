### Title
Tx-Pool Admission Uses Size-Only Fee Check While Mining Uses Weight (Size+Cycles), Eliminating Miner Inclusion Incentive for Cycle-Heavy Transactions - (File: tx-pool/src/util.rs)

### Summary

The tx-pool admission gate checks the minimum fee rate using only the serialized transaction **size**, but the actual mining weight used for block selection is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. When a transaction's cycle cost dominates, its actual weight is significantly larger than its size. A transaction can therefore pass the pool admission check while having an actual fee rate far below the minimum, giving miners no incentive to include it. This is the direct CKB analog to the Opyn report: the "liquidation incentive" (miner inclusion incentive) disappears under adverse conditions (high-cycle scripts), leaving the pool holding obligations it cannot fulfill.

---

### Finding Description

**Root cause — `check_tx_fee` in `tx-pool/src/util.rs`:**

```rust

    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(...).transaction_fee(rtx)...;

**The gap:** For a transaction with `size = S` and `cycles = C` where `C × 0.000_170_571_4 > S`:

| Check | Formula | Threshold |
|---|---|---|
| Pool admission | `fee ≥ min_fee_rate × S` | Lower (size only) |
| Miner inclusion | `fee ≥ min_fee_rate × max(S, C × 0.000_170_571_4)` | Higher (weight) |

Any transaction with fee in the range `[min_fee_rate × S, min_fee_rate × weight)` passes admission but is never included by miners.

**Concrete example** with `min_fee_rate = 1000 shannons/KW` (default):
- `size = 200 bytes`, `cycles = 5,000,000`
- `weight = max(200, 5_000_000 × 0.000_170_571_4) = max(200, 852) = 852`
- Admission threshold: `fee ≥ 1000 × 200 / 1000 = 200 shannons`
- Mining threshold: `fee ≥ 1000 × 852 / 1000 = 852 shannons`
- A transaction with `fee = 300 shannons` passes admission but has actual fee rate `300 × 1000 / 852 ≈ 352 shannons/KW` — well below the 1000 shannons/KW minimum [6](#0-5) 

---

### Impact Explanation

1. **Pool resource exhaustion**: An attacker submits many cycle-heavy, size-small transactions just above the size-based admission threshold. Each transaction is accepted, consumes pool memory, and is never mined.
2. **Legitimate transaction displacement**: When the pool reaches `max_tx_pool_size` (default 180 MB), `limit_size` evicts by lowest weight-based fee rate. The attacker's transactions have low actual fee rates and are evicted first — but during the window before eviction, they displace legitimate transactions and consume verification resources.
3. **Wasted script verification**: The node fully executes the attacker's scripts (computing cycles) before discovering the actual fee rate is below the mining threshold. This CPU cost is borne by the node with no compensation.
4. **Sustained pool pressure**: The attacker can continuously resubmit at minimal cost (fee just above `min_fee_rate × size`), maintaining pool pressure indefinitely. [7](#0-6) 

---

### Likelihood Explanation

- **Attacker entry**: Any unprivileged user with RPC access to `send_transaction`. No special privileges required.
- **Cost**: The attacker pays `min_fee_rate × size` per transaction — the minimum possible fee. For a 200-byte transaction at 1000 shannons/KW, this is 200 shannons (~0.000_000_002 CKB) per transaction.
- **Amplification**: The ratio `weight / size` can be up to `max_block_cycles × DEFAULT_BYTES_PER_CYCLES / max_block_bytes ≈ 1` at the block level, but for individual transactions with concentrated cycle usage, the ratio can be much higher (e.g., 4× in the example above).
- **Detectability**: The discrepancy is only visible after script execution, which happens inside the node. The attacker's transactions look legitimate at the RPC layer.

---

### Recommendation

**Short term**: After script verification completes and cycles are known, re-validate the fee rate using the actual weight (`get_transaction_weight(size, cycles)`) before admitting the transaction to the pool. Reject with `Reject::LowFeeRate` if `fee < min_fee_rate × weight`.

**Long term**: Consider making the minimum fee rate check weight-aware from the start. Since cycles are declared by the submitter (and verified), a preliminary weight-based check using the declared cycles can be performed before full script execution, with a final check after execution confirms the declared cycle count.

---

### Proof of Concept

1. Craft a CKB transaction with a lock script that consumes ~5,000,000 cycles but has a serialized size of ~200 bytes (e.g., a tight loop in a RISC-V script stored in a dep cell).
2. Set the transaction fee to 300 shannons (above the size-based minimum of 200 shannons, below the weight-based minimum of 852 shannons).
3. Submit via `send_transaction` RPC. The node accepts it (`fee 300 ≥ min_fee 200`).
4. Observe that the transaction is never proposed or committed by miners (actual fee rate ≈ 352 shannons/KW < 1000 shannons/KW minimum).
5. Repeat in a loop. Each submission costs ~300 shannons and forces the node to execute the full script. The pool fills with permanently-stalled transactions until the 12-hour expiry (`expiry_hours = 12`) or pool eviction. [8](#0-7) [9](#0-8)

    // Remove transactions from the pool until total size <= size_limit.
    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
