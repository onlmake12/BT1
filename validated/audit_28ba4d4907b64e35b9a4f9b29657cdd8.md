### Title
Zero-Price Bypass of Deviation Check Causes Stale Price Feeds in Deviation-Only Subscriptions - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
In `Scheduler.sol`, the `_validateShouldUpdatePrices` function silently skips the deviation check for any price feed where either the current or stored previous price is zero. When a subscription uses `updateOnDeviation` as its sole update criterion, this causes the update to be rejected when prices crash to zero or recover from zero, leaving consumers with permanently stale price data.

### Finding Description
The analog vulnerability class is: **a zero/null value causes a required check to be bypassed**. In the Quadrata report, a null `businessAttribute` caused `isBusiness` to become `false`, bypassing the EntityType check. In Pyth's `Scheduler.sol`, a zero price causes the deviation check to be bypassed via `continue`.

In `_validateShouldUpdatePrices`, the deviation loop reads:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [1](#0-0) 

For a subscription configured with `updateOnDeviation = true` and `updateOnHeartbeat = false`, if every price feed in the batch has a zero current or previous price, the loop completes without ever returning `updateTimestamp`, and execution falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

The full deviation block is: [3](#0-2) 

**Scenario A — price crashes to zero:**
- Stored `previousPrice = 100`, incoming `currentPrice = 0`
- `continue` fires; deviation (effectively infinite) is never computed
- No other feed meets the threshold → `revert UpdateConditionsNotMet`
- Stored price remains `100` (stale) even though the real price is `0`

**Scenario B — price recovers from zero:**
- Stored `previousPrice = 0`, incoming `currentPrice = 100`
- `continue` fires; deviation (effectively infinite) is never computed
- No other feed meets the threshold → `revert UpdateConditionsNotMet`
- Stored price remains `0` (stale) even though the real price is `100`

The `_storePriceUpdates` function stores whatever Pyth returns with no zero-price guard: [4](#0-3) 

`parsePriceFeedUpdatesWithConfig` is called with `minValidTime = 0`, so a zero-price feed is accepted and stored: [5](#0-4) 

### Impact Explanation
Subscriptions configured with `updateOnDeviation` only permanently fail to update when any price feed transitions to or from zero. DeFi protocols consuming prices via `getPricesUnsafe` or `getPricesNoOlderThan` receive stale data, enabling incorrect liquidations, mispriced collateral, or other financial losses. The stored price is frozen at the pre-crash or pre-recovery value indefinitely until a heartbeat-based update (which the subscription does not have) or a manual subscription reconfiguration occurs.

### Likelihood Explanation
A price of zero is a valid `int64` in the Pyth data model and can occur during market halts, token collapses, or publisher data anomalies. No attacker action is required — the flaw is triggered automatically by any keeper submitting a legitimate Pyth update that contains a zero price. The entry path is `updatePriceFeeds(subscriptionId, updateData)`, callable by any unprivileged keeper.

### Recommendation
Treat a transition to or from zero as an infinite deviation, always triggering an update:

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // Price going to or from zero is an infinite deviation — always update
    return updateTimestamp;
}
```

This mirrors the existing first-update logic (`previousFeed.id == bytes32(0)` → `return updateTimestamp`) and is consistent with the intent that any significant price change should trigger an update.

### Proof of Concept
1. Create a subscription: `updateOnDeviation = true`, `updateOnHeartbeat = false`, `deviationThresholdBps = 100`
2. First update: submit all feeds with `price = 100` → stored successfully (first update path via `previousFeed.id == bytes32(0)`)
3. Second update: submit all feeds with `price = 0` (valid Pyth data, e.g., market halt)
4. In `_validateShouldUpdatePrices`: `previousPrice = 100`, `currentPrice = 0` → `continue` for every feed
5. Loop exits without returning → `revert UpdateConditionsNotMet()`
6. Stored price remains `100`; consumers read a stale non-zero price while the real price is `0`
