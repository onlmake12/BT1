### Title
`total_keys_num` Never Incremented in `RecentReject::put()` Disables Size Guard, Allowing Unbounded DB Growth — (File: `tx-pool/src/component/recent_reject.rs`)

### Summary
`RecentReject::put()` computes `self.total_keys_num.checked_add(1)` into a local variable but never writes the result back to `self.total_keys_num`. The size-based shrink guard is therefore permanently disabled: the `recent_reject` RocksDB instance grows without bound regardless of `count_limit`, bounded only by RocksDB's TTL-based compaction schedule.

### Finding Description
`RecentReject` is the tx-pool component that records every rejected transaction so that re-submissions can be answered quickly. It maintains a `total_keys_num: u64` counter and a `count_limit` threshold; when the counter exceeds the limit, `shrink()` is called to prune the DB.

In `put()`:

```rust
// tx-pool/src/component/recent_reject.rs  L55-L70
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    ...
    self.db.put(&shard, hash_slice, json_string)?;

    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
        if total_keys_num > self.count_limit {
            self.shrink()?;
        }
    } else {
        // overflow occurred, try shrink
        self.shrink()?;
    }
    Ok(())
}
```

`total_keys_num` is a **new local binding** holding `self.total_keys_num + 1`. `self.total_keys_num` is never assigned the new value. On every subsequent call to `put`, the same stale initial estimate is used. Because the node typically starts with an empty (or small) `recent_reject` DB, `self.total_keys_num` is initialised to a small value and stays there permanently. The `shrink()` branch is never reached, and the DB grows without bound.

The struct is initialised in `build()`:

```rust
// tx-pool/src/component/recent_reject.rs  L28-L53
let total_keys_num = Self::checked_estimate_sum(&estimate_keys_num)?;
Ok(RecentReject { ..., total_keys_num })
```

After that point, `total_keys_num` is never mutated by `put()`.

### Impact Explanation
An unprivileged tx-pool submitter or RPC caller can flood the node with transactions that are guaranteed to be rejected (invalid scripts, double-spend attempts, fee-rate violations, etc.). Each rejection writes one entry to the `recent_reject` DB. Because the size guard is permanently disabled, the DB grows until disk space is exhausted. Disk exhaustion prevents the node from writing new blocks, transactions, or any other persistent state, causing a full node outage. The TTL-based compaction provides a soft ceiling, but compaction is not guaranteed to run promptly; under sustained attack the DB can grow far beyond the intended `count_limit` before compaction reclaims space.

### Likelihood Explanation
The attack requires no special privileges, keys, or majority hashpower. Invalid transactions can be submitted via the `send_transaction` JSON-RPC endpoint or the P2P relay protocol by any external caller. The cost per entry is negligible (rejected transactions are not executed by CKB-VM). A sustained stream of cheap invalid transactions (e.g., transactions with a zero-byte lock script that fails immediately) is sufficient to trigger the condition.

### Recommendation
Assign the incremented value back to `self.total_keys_num` before the threshold check:

```rust
if let Some(new_total) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = new_total;   // ← missing assignment
    if self.total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

### Proof of Concept
1. Start a CKB node with default configuration (non-empty `count_limit`, e.g. the default 100 000).
2. In a tight loop, submit transactions via `send_transaction` RPC that are guaranteed to be rejected (e.g. a transaction whose lock script is a single `0x00` byte — it will fail script execution immediately and be recorded in `recent_reject`).
3. Observe that `self.total_keys_num` never advances past its initial value (verifiable via `get_estimate_total_keys_num()`), `shrink()` is never called, and the on-disk size of the `recent_reject` directory grows continuously, eventually exhausting available disk space and crashing the node.
