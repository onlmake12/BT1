Audit Report

## Title
Unbounded Per-Request Resource Exhaustion via Missing Rate Limit in `GetBlocksProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary
Any unprivileged P2P peer can send a structurally valid `GetBlocksProof` message with up to 1000 distinct main-chain block hashes. The server unconditionally performs 3 synchronous DB reads per hash (up to 3,000 total) plus an O(N × log M) MMR proof generation, with no per-peer rate limit. Because every valid request returns `Status::ok()` (HTTP 200), the banning logic never fires, allowing a peer to repeat the attack indefinitely and exhaust the node's RocksDB I/O and CPU.

## Finding Description
`GET_BLOCKS_PROOF_LIMIT` is set to 1000 with no accompanying rate-limiting mechanism. [1](#0-0) 

`execute` enforces only structural validity: non-empty, ≤ 1000 hashes, no duplicates. Any request satisfying these checks proceeds unconditionally to DB work. [2](#0-1) 

For every hash that passes `is_main_chain`, three synchronous DB reads are issued: `get_block_header`, `get_block_uncles`, and `get_block_extension`. [3](#0-2) 

All collected positions are then passed to `mmr.gen_proof`, which reads O(N × log M) MMR nodes from storage. [4](#0-3) 

A successful request returns `Status::ok()` (code 200). The banning logic in `received` only fires when `should_ban()` returns `Some`, which only occurs for 4xx status codes. [5](#0-4) [6](#0-5) 

A grep across the entire `light-client-protocol-server` crate for any rate-limiting, token-bucket, quota, or per-peer counter returns zero matches, confirming no such mechanism exists.


## Impact Explanation
A single attacker peer sending a continuous stream of max-size `GetBlocksProof` messages forces up to 3,000 synchronous RocksDB reads plus O(1000 × log N) MMR node reads per request. Because the peer is never banned, this can be sustained indefinitely. The shared RocksDB instance is also used by the consensus reactor for block relay and IBD; saturating its I/O budget stalls or crashes the node's participation in consensus. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **High: Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires only a valid P2P connection and 1000 main-chain block hashes, both trivially obtainable from any public block explorer or by syncing headers. No proof-of-work, private key, or privileged role is needed. The attacker can open multiple simultaneous connections to multiply the effect. The peer is never penalized, so the attack can run indefinitely at zero ongoing cost.

## Recommendation
1. **Per-peer rate limit**: enforce a maximum number of `GetBlocksProof` requests per peer per time window (e.g., 1 request/second) using a token-bucket or leaky-bucket counter stored in peer state.
2. **Return a 4xx status on rate-limit breach**: returning `StatusCode::MalformedProtocolMessage` or a new `RateLimitExceeded` (4xx) code when a peer exceeds the limit will trigger `should_ban()` and cause `nc.ban_peer` to fire, cutting off the attacker automatically.
3. **Reduce the effective limit or add cost proportional to request size**: lower `GET_BLOCKS_PROOF_LIMIT` or require the caller to pay a cost (e.g., a challenge-response) before the server performs DB work.

## Proof of Concept
1. Connect to a CKB node with the light-client protocol enabled.
2. Obtain 1000 distinct main-chain block hashes (e.g., blocks 1–1000) from any block explorer.
3. In a tight loop, send `GetBlocksProof { last_hash: <tip>, block_hashes: [h1..h1000] }`.
4. Observe: RocksDB read IOPS spike; block relay and IBD latency increase; the node's consensus participation degrades or stalls.
5. Confirm the peer is never banned by monitoring peer state — every response carries `Status::ok()` (200), so `should_ban()` always returns `None`.
