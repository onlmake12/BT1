### Title
Division-by-Zero in `calculateTwap` When `publishSlot` Values Are Equal Due to Missing `>=` Guard in `validateTwapPriceInfo` — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary
`validateTwapPriceInfo` rejects only the case where `start.publishSlot > end.publishSlot`, but permits `start.publishSlot == end.publishSlot`. When equal slots reach `calculateTwap`, `slotDiff` is `0` and three unchecked integer divisions panic, causing `parseTwapPriceFeedUpdates` to revert with a division-by-zero instead of a clean, descriptive error.

### Finding Description
In `validateTwapPriceInfo`:

```solidity
// Pyth.sol line 604
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

The guard is `>`, not `>=`. Equal slots are silently allowed through.

In `calculateTwap`:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic: div by 0
uint128 twapConf  = confDiff / uint128(slotDiff);           // panic: div by 0
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: div by 0
```

No `require(slotDiff > 0)` or `checked_div` exists. The Solana counterpart explicitly requires `end_msg.publish_slot > start_msg.publish_slot` (strict), rejecting equal slots before any arithmetic.

### Impact Explanation
Any caller of `parseTwapPriceFeedUpdates` supplying two Wormhole-attested TWAP VAAs that share the same `publishSlot` will receive an EVM panic revert rather than a clean protocol error. The call always reverts; no TWAP result is ever returned for that slot pair. If the only available attested VAAs for a given window share a slot (e.g., two messages published in the same Pythnet slot), the TWAP feed is effectively inaccessible for that window.

### Likelihood Explanation
The attacker does not forge VAAs — they only need two legitimately guardian-signed TWAP messages whose `publishSlot` fields are identical. This can occur naturally (same Pythnet slot, two messages) or an adversarial relayer can deliberately select such a pair from the set of valid attested messages. No privileged access is required; `parseTwapPriceFeedUpdates` is a public payable function.

### Recommendation
Change the slot ordering check in `validateTwapPriceInfo` from strict `>` to `>=`:

```solidity
// Before (line 604):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After:
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This matches the Solana reference implementation's `end_msg.publish_slot > start_msg.publish_slot` invariant and eliminates the division-by-zero path entirely.
