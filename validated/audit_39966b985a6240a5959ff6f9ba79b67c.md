### Title
Stale `total_tx_size` / `total_tx_cycles` Accumulator After Ancestor-Eviction in `add_entry` Overwrites Correct Intermediate State — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new entry's contribution to `total_tx_size` and `total_tx_cycles` is pre-computed before any mutations occur, but the pre-computed value is committed **after** `check_and_record_ancestors` has already correctly decremented those counters for any evicted transactions. The final assignment silently discards the correct intermediate state, leaving `total_tx_size` and `total_tx_cycles` permanently inflated by the aggregate size/cycles of every evicted entry. An unprivileged tx-pool submitter can repeatedly trigger this path to inflate the counters until the pool refuses all further submissions.

---

### Finding Description

`add_entry` in `pool_map.rs` follows this sequence:

```rust
// Step 1: pre-compute new totals (not yet committed)
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Step 2: may evict existing entries → calls remove_entry_and_descendants
//         → calls remove_entry → calls update_stat_for_remove_tx
//         which correctly decrements self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Step 3: overwrite with the stale pre-eviction snapshot + new entry size
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` into a local variable **before** any evictions occur. [2](#0-1) 

`check_and_record_ancestors` can call `remove_entry_and_descendants` when the ancestor count exceeds `max_ancestors_count` but some ancestors are `cell_ref_parents` that can be evicted to bring the count within bounds. [3](#0-2) 

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` and `self.total_tx_cycles` for each evicted entry. [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, the final two lines overwrite the correctly-decremented `self.total_tx_size` / `self.total_tx_cycles` with the stale pre-eviction snapshot plus the new entry's size. The evicted entries' sizes are effectively "added back" to the running totals.

**Concrete arithmetic example:**

| Step | `self.total_tx_size` | `total_tx_size` (local) |
|---|---|---|
| Initial | 100 | — |
| After `updated_stat_for_add_tx(+10)` | 100 | 110 |
| After eviction of 20-byte tx via `update_stat_for_remove_tx` | 80 | 110 |
| After `self.total_tx_size = total_tx_size` | **110** (wrong) | — |

Correct value: 100 − 20 + 10 = **90**. Tracked value: **110**. Inflation per eviction cycle: size of each evicted entry.

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further entries, and by `updated_stat_for_add_tx` to reject new submissions with `Reject::Full`. [6](#0-5) [7](#0-6) 

Each successful trigger of the ancestor-eviction path inflates `total_tx_size` by the aggregate byte-size of all evicted entries. After enough iterations the counter exceeds `max_tx_pool_size` (default 180 MB) even though the pool is nearly empty, causing every subsequent `add_entry` call to return `Reject::Full`. This is a **tx-pool DoS**: the node stops accepting any new transactions from any peer or RPC caller until the node is restarted.

`total_tx_cycles` is inflated by the same mechanism, which can additionally corrupt fee-rate estimation. [8](#0-7) 

---

### Likelihood Explanation

The eviction branch inside `check_and_record_ancestors` is reached when:

1. The new transaction's ancestor count exceeds `max_ancestors_count` (default 125), **and**
2. Removing `cell_ref_parents` (transactions that reference the same cell deps as the new tx's inputs) would bring the count within bounds.

An unprivileged submitter can engineer this deliberately:

- Build a chain of ~126 transactions (A₁ → A₂ → … → A₁₂₆) and submit them to the pool.
- Submit a new transaction B whose inputs reference an output of A₁₂₆ **and** whose cell deps reference an output of A₁ (making A₁ a `cell_ref_parent`).
- B's ancestor count is 127 > 125, but removing A₁ (and its descendants) brings it to ≤ 125, so the eviction path fires.
- Each such round inflates `total_tx_size` by the aggregate size of the evicted chain.
- Repeat with fresh UTXOs.

No special privilege, hashpower, or Sybil capability is required; only the ability to submit transactions to the RPC or P2P relay.

---

### Recommendation

Compute the pre-eviction snapshot **after** `check_and_record_ancestors` returns, or update the local `total_tx_size`/`total_tx_cycles` variables to account for evictions before committing them. The simplest fix is to move the stat commitment to after all mutations:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Recompute from current self.total_tx_size (already decremented by evictions)
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .ok_or_else(|| Reject::Full(...))?;
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(...))?;
```

Alternatively, call `updated_stat_for_add_tx` after `check_and_record_ancestors` so the snapshot reflects the post-eviction state.

---

### Proof of Concept

1. Connect to a CKB node's RPC.
2. Fund a wallet with enough CKB to create a 126-deep transaction chain.
3. Submit A₁ → A₂ → … → A₁₂₆ to the pool (each spending the previous output).
4. Craft transaction B:
   - Input: spends A₁₂₆'s output.
   - Cell dep: references A₁'s output (making A₁ a `cell_ref_parent`).
5. Submit B. The pool evicts A₁ and its descendants (A₂…A₁₂₆) to satisfy the ancestor limit, then inserts B. `total_tx_size` is now inflated by the sum of sizes of A₁…A₁₂₆.
6. Repeat steps 2–5 with fresh UTXOs. After enough iterations, `total_tx_size` exceeds `max_tx_pool_size`.
7. Observe that all subsequent `send_transaction` RPC calls return `Reject::Full` even though the pool contains only a handful of entries. [9](#0-8) [1](#0-0)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

                self.total_tx_size, tx_size
            ))
        })?;

            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    // Remove transactions from the pool until total size <= size_limit.
    // Return a `Reject` for current inserting entry if it's removed
    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
