### Title
`payable(msg.sender).transfer()` Used for Excess-Fee Refund Causes DoS for Smart Contract Callers - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` refunds excess native ETH to `msg.sender` using the Solidity `.transfer()` primitive, which forwards only 2300 gas. Any smart contract caller whose `receive()` or `fallback()` function consumes more than 2300 gas will have the refund revert, causing the entire `verifyUpdate()` call to revert. This permanently blocks those callers from using the Lazer verification service whenever they overpay.

### Finding Description
In `PythLazer.verifyUpdate()`, after confirming the fee is sufficient, the contract attempts to refund the excess:

```solidity
// lazer/contracts/evm/src/PythLazer.sol  lines 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

`address.transfer()` hard-caps the forwarded gas at 2300 stipend. If `msg.sender` is a smart contract — for example, a DeFi protocol, a multisig wallet, a proxy, or any contract whose `receive()`/`fallback()` emits an event, writes to storage, or performs any non-trivial logic — the refund call will run out of gas and revert. Because the refund is not wrapped in a try/catch, the revert propagates and the entire `verifyUpdate()` transaction fails.

The function is `external payable` with no access control:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
``` [2](#0-1) 

Any unprivileged Lazer updater or integrating contract is the entry point.

### Impact Explanation
Smart contract callers that overpay `verification_fee` by even 1 wei are permanently unable to call `verifyUpdate()` if their fallback/receive logic exceeds 2300 gas. This is a realistic scenario for:
- Multisig wallets (Gnosis Safe, etc.) whose fallback emits events
- Proxy contracts with non-trivial fallback routing
- Any DeFi protocol that wraps Lazer verification in a contract

The result is a complete DoS of the Lazer price-verification path for those callers. Since `verification_fee` can change (owner-controlled), callers who previously computed the exact fee may inadvertently overpay after a fee decrease, triggering the bug retroactively.

### Likelihood Explanation
Moderate. The Lazer `verifyUpdate()` function is designed to be called by on-chain integrators (smart contracts), not just EOAs. Any such integrator that sends `msg.value > verification_fee` — whether due to fee changes, conservative fee estimation, or deliberate overpayment — will be affected. The pattern is well-known and commonly triggered in production.

### Recommendation
Replace `.transfer()` with a low-level `.call{value: ...}("")` and check the return value, consistent with the pattern already used in `Entropy.sol`:

```solidity
// Entropy.sol line 199 — correct pattern already used elsewhere
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "refund failed");
``` [3](#0-2) 

Applied to `PythLazer.sol`:

```solidity
if (msg.value > verification_fee) {
    (bool refunded, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(refunded, "refund failed");
}
```

### Proof of Concept
1. Deploy a contract `MaliciousReceiver` whose `receive()` function writes to storage (costs >2300 gas).
2. From `MaliciousReceiver`, call `pythLazer.verifyUpdate{value: verification_fee + 1}(validUpdate)`.
3. The signature verification passes, but `payable(msg.sender).transfer(1)` forwards only 2300 gas to `MaliciousReceiver.receive()`, which runs out of gas.
4. The entire transaction reverts with an out-of-gas error.
5. `MaliciousReceiver` can never successfully call `verifyUpdate()` while overpaying, regardless of how valid the update data is. [4](#0-3)
