Audit Report

## Title
Unbounded Per-Peer Submissions to `VerifyQueue` Enable FIFO-Based Processing Delay for Legitimate Transactions — (File: `tx-pool/src/component/verify_queue.rs`)

## Summary

`VerifyQueue::add_tx` enforces only a single global 256 MB cap with no per-peer byte quota. Workers drain the queue strictly oldest-first via `iter_by_added_time().next()`, so an attacker who floods the queue before a legitimate user always gets processed first. The fee-rate check (`check_tx_fee`) is deferred until after a worker dequeues the entry inside `_process_tx → pre_check`, meaning sub-minimum-fee transactions temporarily occupy queue space and consume worker time.

## Finding Description

**No per-peer guard in `add_tx`:**

`VerifyQueue::add_tx` performs exactly one admission check: `self.is_full(tx_size)`, comparing the new transaction's size against `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256_000_000` minus `self.total_tx_size`. There is no `peer_tx_count`, no per-`PeerIndex` byte counter, and no minimum-fee floor enforced at this layer. [1](#0-0) [2](#0-1) [3](#0-2) 

**Fee check deferred until after dequeue:**

`resumeble_process_tx` calls `non_contextual_verify` (format/size/version checks only) and then immediately calls `enqueue_verify_queue`. The fee-rate check (`check_tx_fee`) is invoked only inside `_process_tx → pre_check`, which runs on a worker thread after `pop_front`. Transactions with zero or sub-minimum fees enter the queue and occupy space until a worker dequeues and rejects them. [4](#0-3) [5](#0-4) [6](#0-5) 

**FIFO drain confirmed:**

`Worker::process_inner` calls `tasks.pop_front(self.role == WorkerRole::OnlySmallCycleTx)`, which resolves to `peek → iter_by_added_time().next()` — strictly oldest-first. An attacker whose entries carry earlier `added_time` values is always dequeued before a later legitimate submission. [7](#0-6) [8](#0-7) 

**`remove_txs_by_peer` is insufficient:**

`remove_txs_by_peer` exists on `VerifyQueue` but is only called from `ban_malformed`, which triggers only on structurally malformed transactions — not on valid minimum-fee or zero-fee spam. [9](#0-8) [10](#0-9) 

**Relay path rate limit is insufficient:**

`MAX_RELAY_TXS_BYTES_PER_BATCH = 1024 * 1024` (1 MB) at 30 messages/second per `(PeerIndex, message_type)` yields 30 MB/second of inbound transaction data per peer — enough to saturate the 256 MB queue in under 10 seconds. [11](#0-10) [12](#0-11) [13](#0-12) 

## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

A single external P2P peer or RPC caller can saturate the 256 MB verify queue, causing two concrete effects:

1. **`Reject::Full` for legitimate users** — any `send_transaction` call or relayed transaction arriving after the queue is full is immediately rejected.
2. **Unbounded FIFO delay** — even before saturation, the attacker's earlier-timestamped entries are always dequeued first, introducing latency proportional to the attacker's queue depth for every honest user's transaction.

The attack degrades transaction propagation and confirmation latency across the node, constituting network congestion.

## Likelihood Explanation

- **Fully unprivileged:** Any P2P peer can execute the relay path; any host with RPC access can execute the RPC path.
- **Economic cost is low:** The attacker pays `min_fee_rate` (1,000 shannons/KB by default) on valid transactions. Sub-minimum-fee transactions also enter the queue (fee check is deferred), reducing cost further for temporary queue occupancy.
- **Continuously repeatable:** As workers drain attacker entries, the attacker replenishes. The relay rate limiter (30 msg/sec × 1 MB/msg) provides 30 MB/sec of sustained throughput — far exceeding typical honest traffic.
- **No special privilege or victim mistake required.**

## Recommendation

1. **Add a per-peer byte quota to `VerifyQueue`.** Track `HashMap<PeerIndex, usize>` of bytes currently held per peer inside `VerifyQueue`. Reject `add_tx` when a single peer's share exceeds a configurable fraction (e.g., 25%) of `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`.
2. **Enforce minimum fee rate at enqueue time.** Move a lightweight fee-rate pre-check (using declared cycles for remote transactions) into `resumeble_process_tx` before `enqueue_verify_queue`, so sub-minimum-fee transactions never enter the queue.
3. **Add RPC-level rate limiting.** Apply a per-connection or per-IP token-bucket limiter to `send_transaction`, analogous to the relay rate limiter in `Relayer::try_process`.
4. **Consider fair-queuing.** Replace strict FIFO with a round-robin or weighted-fair drain across `PeerIndex` values to prevent any single peer from monopolizing worker time even within their quota.

## Proof of Concept

**P2P relay path (external, unprivileged):**

1. Connect to the target node as a P2P peer.
2. Send `RelayTransactionHashes` messages at 30/second, each carrying 32,767 hashes of pre-built minimum-fee transactions (or zero-fee transactions to reduce cost).
3. The node sends `GetRelayTransactions`; respond with `RelayTransactions` messages (up to 1 MB each, 30/second).
4. Each transaction passes `non_contextual_verify` and enters `VerifyQueue` via `add_tx` without fee checking.
5. After ~8–10 seconds, `total_tx_size` approaches 256 MB; subsequent legitimate `send_transaction` calls return `Reject::Full`.
6. Workers drain attacker entries first (FIFO by `added_time`); attacker replenishes continuously.

**RPC path (open or local RPC endpoint):**

```python
for utxo in attacker_utxos:
    rpc.send_transaction(build_min_fee_tx(utxo))  # no rate limit
```

50,000 × 200 B = 10 MB occupies the queue; legitimate submissions are queued behind all attacker entries or rejected when the cap is reached.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
