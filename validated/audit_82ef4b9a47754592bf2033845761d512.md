### Title
Lack of Access Control on `providerToCredit` in `executeCallback` Enables Fee Theft and Misleading `PriceUpdateExecuted` Events — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` enforces that `providerToCredit == req.provider` only during the exclusivity window. After that window expires, the parameter is completely unconstrained: any caller may pass any address as `providerToCredit`, redirecting the entire request fee to an attacker-controlled address and emitting a `PriceUpdateExecuted` event that falsely attributes the execution to the attacker.

---

### Finding Description

`executeCallback` in `Echo.sol` contains a conditional access check:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
``` [1](#0-0) 

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, the check is skipped entirely. The function then unconditionally credits fees to the caller-supplied `providerToCredit`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

There is no check that `providerToCredit` is a registered provider, nor that it matches `req.provider`. Immediately after, the event is emitted with `msg.sender` (the attacker) as the executor:

```solidity
emit PriceUpdateExecuted(
    sequenceNumber,
    msg.sender,
    priceIds, prices, conf, expos, publishTimes
);
``` [3](#0-2) 

`registerProvider` is permissionless:

```solidity
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    ...
    provider.isRegistered = true;
``` [4](#0-3) 

`setFeeManager` is also permissionless for any registered provider:

```solidity
function setFeeManager(address manager) external override {
    require(
        _state.providers[msg.sender].isRegistered,
        "Provider not registered"
    );
    ...
    _state.providers[msg.sender].feeManager = manager;
``` [5](#0-4) 

`withdrawAsFeeManager` allows the fee manager to drain the accrued balance:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
    _state.providers[provider].accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [6](#0-5) 

---

### Impact Explanation

**Fee theft**: The original provider (`req.provider`) loses the entire request fee (`req.fee`) to the attacker. The attacker registers as a provider, sets themselves as fee manager, calls `executeCallback` with `providerToCredit = attacker_address` after the exclusivity period, and then calls `withdrawAsFeeManager` to drain the balance.

**Event authenticity degradation**: The `PriceUpdateExecuted` event records `msg.sender` (the attacker) as the executor, not the original provider. Off-chain indexers, dashboards, and reputation systems that rely on this event to track which provider fulfilled a request will receive false attribution data — a direct analog to the FR-2 `notifyFeeReceived` event authenticity issue.

---

### Likelihood Explanation

- `registerProvider` and `setFeeManager` are permissionless — no barrier to attacker setup.
- The attacker only needs to wait for `exclusivityPeriodSeconds` to elapse after `req.publishTime`, then front-run or simply call `executeCallback` before the legitimate provider does.
- The attack is profitable whenever `req.fee > gas cost`, which is the normal operating condition.
- No privileged access, leaked keys, or oracle manipulation is required.

---

### Recommendation

After the exclusivity period, restrict `providerToCredit` to registered providers only, or remove the parameter entirely and always credit `req.provider`. At minimum, add:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
```

Additionally, consider whether `providerToCredit` should ever differ from `req.provider` after the exclusivity period, or whether the design intent is simply to allow any registered provider to step in (in which case the event should record `providerToCredit`, not `msg.sender`, to preserve authenticity).

---

### Proof of Concept

1. Attacker calls `registerProvider(0, 0, 0)` — registers with zero fees.
2. Attacker calls `setFeeManager(attacker_address)` — sets self as fee manager.
3. A legitimate user calls `requestPriceUpdatesWithCallback(originalProvider, publishTime, priceIds, gasLimit)` paying `req.fee`.
4. Attacker waits until `block.timestamp >= publishTime + exclusivityPeriodSeconds`.
5. Attacker calls `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)`.
   - The exclusivity check is skipped.
   - `_state.providers[attacker_address].accruedFeesInWei += req.fee - pythFee`.
   - `PriceUpdateExecuted(sequenceNumber, attacker_address, ...)` is emitted — original provider is not mentioned.
6. Attacker calls `withdrawAsFeeManager(attacker_address, stolen_amount)` — receives the stolen fees.

The original provider receives nothing; the on-chain event log permanently records the attacker as the executor.
