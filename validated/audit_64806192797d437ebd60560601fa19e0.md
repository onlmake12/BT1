### Title
Unbounded `firstUnfulfilledSeq` Scan in `executeCallback` Enables Permanent Fund Lock — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by scanning every sequence number between it and `_state.currentSequenceNumber`. An unprivileged attacker can inflate this gap by creating many requests and fulfilling all but the oldest one, causing the while loop to exhaust gas when anyone attempts to fulfill the oldest request, permanently locking the victim's funds.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at least one cold `SLOAD` against the fixed-size `Request[32]` array and potentially a second `SLOAD` against the `requestsOverflow` mapping. [2](#0-1) 

The developers themselves flagged this as a known concern:

> `// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.` [3](#0-2) 

There is **no cap** on how large the gap between `firstUnfulfilledSeq` and `currentSequenceNumber` can grow. The fixed array holds only 32 slots, but the `requestsOverflow` mapping is unbounded, so an attacker can create arbitrarily many requests (paying the required fee for each). [4](#0-3) 

---

### Impact Explanation

**Attack scenario:**

1. Victim creates request with sequence number `S` (e.g., `S = 1`).
2. Attacker creates `N` additional requests (`S+1` through `S+N`), paying the required fee for each.
3. Attacker calls `executeCallback` for all requests `S+1` through `S+N`, fulfilling them. Each fulfillment advances `currentSequenceNumber` but leaves `firstUnfulfilledSeq = S`.
4. Now the gap is `N`. When the provider (or anyone) attempts to call `executeCallback` for request `S`, the while loop must iterate `N` times, each costing ~2,100 gas (cold `SLOAD`).
5. At the EVM 30M gas block limit, approximately **7,000 cold iterations** exhaust the block gas. With a larger gas limit chain, the threshold is higher but still finite.
6. The victim's request can never be fulfilled. The fee locked in the request (`req.fee`) is permanently stuck in the contract — there is no recovery path once `executeCallback` reverts on gas exhaustion.

The `executeCallback` function also uses a static `callbackGasLimit` forwarded to the consumer callback: [5](#0-4) 

This means the gas consumed by the while loop is drawn from the *outer* transaction's gas budget, not the callback's budget, making the exhaustion unavoidable regardless of the callback's gas limit.

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: `requestPriceUpdatesWithCallback` and `executeCallback` are both `external` with no access control beyond fee payment.
- **Cost to attack**: The attacker must pay the Echo fee for each of the `N` requests. On chains with low gas prices and low `pythFeeInWei`, this cost can be minimal relative to the value locked in the victim's request.
- **No cap exists**: Unlike `MAX_PRICE_IDS = 10` which bounds per-request iteration, there is no analogous cap on the sequence number gap. [6](#0-5) 

---

### Recommendation

Introduce a maximum cap on the number of concurrent unfulfilled requests (analogous to `MAX_NFT_APPROVALS` in the reference report). When `currentSequenceNumber - firstUnfulfilledSeq` would exceed this cap, revert in `requestPriceUpdatesWithCallback`. Alternatively, replace the linear scan with a doubly-linked list of active requests (as the in-code TODO already suggests), making `firstUnfulfilledSeq` advancement O(1).

```diff
// In requestPriceUpdatesWithCallback, before allocating:
+ uint64 MAX_CONCURRENT_REQUESTS = 1000; // tune appropriately
+ if (_state.currentSequenceNumber - _state.firstUnfulfilledSeq >= MAX_CONCURRENT_REQUESTS) {
+     revert TooManyConcurrentRequests();
+ }
```

---

### Proof of Concept

1. Deploy Echo with a registered provider.
2. Victim calls `requestPriceUpdatesWithCallback` → gets `sequenceNumber = 1`.
3. Attacker calls `requestPriceUpdatesWithCallback` in a loop `N` times → gets sequence numbers `2..N+1`.
4. Attacker calls `executeCallback` for each of sequences `2..N+1` (fulfilling them all).
5. At this point: `firstUnfulfilledSeq = 1`, `currentSequenceNumber = N+2`.
6. Provider calls `executeCallback` for sequence `1`. The while loop at lines 169–174 iterates `N` times, each doing `findRequest` (2 SLOADs). For `N ≈ 7000` on a 30M-gas chain, the transaction reverts with out-of-gas.
7. Victim's fee is permanently locked; no recovery function exists. [7](#0-6) [8](#0-7)
