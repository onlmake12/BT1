Audit Report

## Title
`remove_expired` Omits Descendant Cascade, Leaving Child Transactions Permanently Stuck with Dangling Inputs — (`tx-pool/src/pool.rs`)

## Summary

`TxPool::remove_expired` removes each expired entry individually via `pool_map.remove_entry`, which cleans up only the expired transaction's own edges and links. When a parent expires but its children do not (because they were submitted just before the parent's expiry window closed), the children remain in the pool with inputs referencing outputs that no longer exist anywhere. These children can never be mined and occupy pool space until their own expiry or until `limit_size` evicts them under pool pressure.

## Finding Description

`remove_expired` collects all entries satisfying `expiry + entry.timestamp < now_ms` and removes each via `pool_map.remove_entry`: [1](#0-0) 

`remove_entry` calls `remove_entry_edges`, which removes only the **removed transaction's own inputs** from `edges.inputs`: [2](#0-1) 

It also calls `remove_entry_links`, which strips the parent reference from each child's `links` entry — but does **not** remove the children from the pool: [3](#0-2) 

The children's own inputs — registered in `edges.inputs` as `{parent_output_outpoint → child_short_id}` — remain intact: [4](#0-3) 

Since the parent was never mined and is no longer in the pool, those outpoints are permanently unresolvable. The children cannot be included in any block.

By contrast, `limit_size` correctly uses `remove_entry_and_descendants`: [5](#0-4) 

Which collects the full descendant closure before removal: [6](#0-5) 

`limit_size` is called immediately after `remove_expired` in the reorg handler, but only evicts when `total_tx_size > max_tx_pool_size`: [7](#0-6) 

Stuck children below the size threshold are never cleaned up by it.

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** An attacker pays zero transaction fees (transactions are never mined) and can continuously inject unresolvable child transactions. When the pool fills, `limit_size` evicts legitimate transactions by fee-rate, degrading service for honest users. Pool metrics (`pending_count`, `gap_count`, `proposed_count`) are inflated, misleading the block assembler and operators.

## Likelihood Explanation

Any unprivileged user with `send_transaction` RPC access can trigger this. The timing condition — submitting children just before the parent's expiry window closes — is straightforward to achieve deliberately. `remove_expired` fires on every block via `_update_tx_pool_for_reorg`. No special privileges, leaked keys, or victim mistakes are required. The attack is repeatable indefinitely at near-zero cost.

## Recommendation

Replace `pool_map.remove_entry` with `pool_map.remove_entry_and_descendants` inside `remove_expired`, matching the behavior of `limit_size`. Care must be taken to skip entries already removed in a prior iteration (a child may appear in `removed` and also be a descendant of an earlier-processed parent — check `pool_map.get_by_id` before processing):

```rust
pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
    let now_ms = ckb_systemtime::unix_time_as_millis();
    let removed: Vec<_> = self
        .pool_map
        .iter()
        .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
        .map(|entry| entry.inner.clone())
        .collect();

    for entry in removed {
        let id = entry.proposal_short_id();
        if self.pool_map.get_by_id(&id).is_none() {
            continue; // already removed as a descendant of a prior entry
        }
        let tx_hash = entry.transaction().hash();
        debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
        let descendants = self.pool_map.remove_entry_and_descendants(&id);
        for e in descendants {
            let reject = Reject::Expiry(e.timestamp);
            callbacks.call_reject(self, &e, reject);
        }
    }
}
```

## Proof of Concept

1. Configure a node with a short `expiry_hours` equivalent (e.g., patch to milliseconds for testing).
2. Submit parent tx `P` via `send_transaction`. Record its pool `timestamp` as `T`.
3. Sleep until `T + expiry - 200 ms`.
4. Submit children `C1…C100`, each spending a distinct output of `P`. Their timestamps are `≈ T + expiry - 200 ms`.
5. Sleep 300 ms (past `P`'s expiry, before children's expiry).
6. Mine a block (or wait for one). `remove_expired` fires: `P` is removed; `C1…C100` are not.
7. Call `get_transaction` on any `Ci`: status is `Pending`. Call `get_pool_info`: `pending_count` is inflated by 100.
8. Attempt to assemble a block: none of `C1…C100` can be included (inputs unresolvable — parent output does not exist on-chain or in pool).
9. Repeat steps 2–8 until `pending_count` reaches the pool limit; observe legitimate transactions being rejected by `limit_size`.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {

                self.links.remove_child(&parent, id);
            }

                self.links.remove_parent(&child, id);
