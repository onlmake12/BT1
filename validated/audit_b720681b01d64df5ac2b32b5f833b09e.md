### Title
Missing Pre-Subtraction Check in `executeCallback` Fee Accounting Can Cause Revert and Lock User Funds - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
In `Echo.executeCallback`, the expression `(req.fee + msg.value) - pythFee` at line 162 performs an unchecked subtraction. There is no prior guard ensuring `pythFee <= req.fee + msg.value`. If the actual Pyth contract fee exceeds the stored request fee plus any additional ETH sent by the executor, the transaction reverts in Solidity 0.8+, preventing the callback from ever being executed and effectively locking the user's prepaid funds in the contract.

### Finding Description
When a user calls `requestPriceUpdatesWithCallback`, the stored `req.fee` is computed as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

Here `_state.pythFeeInWei` is the Echo protocol's own flat fee — it is **not** the actual Pyth contract fee for the update data. The provider is expected (but not enforced) to set their fees high enough to cover the Pyth contract fee.

Later, in `executeCallback`, the actual Pyth contract fee is computed dynamically from the executor-supplied `updateData`:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(...);

_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

There is no check that `pythFee <= req.fee + msg.value` before the subtraction. If `pythFee > req.fee + msg.value`, Solidity 0.8 arithmetic overflow protection causes a revert. Because `clearRequest` is called **after** this line (line 164), the request remains active but cannot be fulfilled. [3](#0-2) 

### Impact Explanation
Two concrete impact paths exist:

1. **Permanent fund locking via Pyth fee increase**: If the Pyth governance increases `pythFeeInWei` after a request is submitted, the actual `pythFee` at execution time can exceed `req.fee`. No executor can fulfill the callback without subsidizing the difference out-of-pocket. If no executor is willing to do so, the user's prepaid request fee is permanently locked in the Echo contract with no refund mechanism.

2. **Executor-controlled DoS**: `executeCallback` is callable by any unprivileged address after the exclusivity period. An executor controls `updateData`, and `pyth.getUpdateFee(updateData)` scales with the number of price updates in the blob. An executor can supply bloated `updateData` (containing many extra valid price feed updates beyond the required `priceIds`) to inflate `pythFee` arbitrarily, causing the subtraction to revert. `parsePriceFeedUpdates` still succeeds because the required `priceIds` are present; only the fee accounting line reverts. This blocks the assigned provider from being credited and delays callback execution.

### Likelihood Explanation
- The Pyth protocol fee is governance-controlled and has been adjusted historically. Any upward adjustment after a request is submitted triggers the permanent locking path.
- The executor-controlled DoS path requires no special privilege — anyone can call `executeCallback` after the exclusivity period with crafted `updateData`.
- The Echo contract is a production Pyth contract with real user funds at stake.

### Recommendation
Add a pre-subtraction guard before the fee accounting line:

```solidity
uint256 totalAvailable = uint256(req.fee) + msg.value;
require(pythFee <= totalAvailable, "Insufficient funds to cover Pyth fee");
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128(totalAvailable - pythFee);
```

Additionally, consider adding a refund path so users can reclaim their funds if the callback becomes permanently unexecutable due to fee increases.

### Proof of Concept

1. Pyth governance increases `pythFeeInWei` from 1 wei to 1000 wei.
2. A user previously called `requestPriceUpdatesWithCallback` paying `msg.value = 500 wei` with `_state.pythFeeInWei = 1 wei` at that time, so `req.fee = 499 wei`.
3. An executor calls `executeCallback` with `msg.value = 0`.
4. `pythFee = pyth.getUpdateFee(updateData) = 1000 wei` (new fee).
5. `parsePriceFeedUpdates{value: 1000}(...)` succeeds (contract has ETH from prior fees).
6. `(req.fee + msg.value) - pythFee = (499 + 0) - 1000` → Solidity 0.8 underflow → **revert**.
7. The request remains active. The user's 500 wei is locked. No executor can fulfill the callback without sending at least 501 wei of their own funds to cover the gap. [4](#0-3) [5](#0-4)
