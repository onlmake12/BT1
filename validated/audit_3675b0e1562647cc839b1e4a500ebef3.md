### Title
Entropy Provider Can Permanently Lock User Fees by Withholding Reveal — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract credits the provider's fee immediately at request time, before the provider has revealed their secret. There is no timeout, cancellation, or refund mechanism. A malicious provider can register permissionlessly, accept user requests, collect fees, and then permanently withhold the reveal — leaving user fees locked in the contract forever with no on-chain recourse.

---

### Finding Description

In `requestHelper`, the provider fee is credited to `providerInfo.accruedFeesInWei` at the moment the user submits their request, before any reveal has occurred: [1](#0-0) 

Fulfillment via `revealWithCallback` requires the caller to supply the provider's secret (`providerContribution`), which only the provider knows: [2](#0-1) 

The `reveal` path (non-callback) additionally restricts the caller to the original requester, but still requires the provider's secret: [3](#0-2) 

There is no `cancelRequest`, no timeout check, and no refund path anywhere in the contract. The provider registration is fully permissionless: [4](#0-3) 

Once a provider withholds their secret, the in-flight request can never be cleared, the user's fee is permanently credited to the provider's balance, and the provider can immediately withdraw it: [5](#0-4) 

---

### Impact Explanation

- User fees paid to a malicious provider are permanently locked/stolen — the provider collects them via `withdraw()` while the user's callback is never delivered.
- For callback-based consumers (e.g., NFT minting, lottery, gaming contracts), the downstream logic is permanently blocked with no recovery path.
- This constitutes **permanent freezing / theft of user funds** at the smart contract level.

---

### Likelihood Explanation

- Provider registration is fully permissionless — any address can call `register()`.
- A malicious actor can set an attractive fee, attract users (especially those not using the default Pyth provider), collect fees at request time, and never reveal.
- Users who explicitly pass a non-default `provider` address to `requestV2(provider, ...)` are directly exposed.
- The default Pyth provider (Fortuna) is trusted, but the protocol design allows and encourages use of alternative providers.

---

### Recommendation

Add a block-number-based timeout to each request. After `N` blocks (e.g., 256), allow the original requester to call a `cancelRequest(provider, sequenceNumber)` function that:
1. Clears the request from storage.
2. Refunds the user's fee (deducting from `providerInfo.accruedFeesInWei` and transferring back to the requester).

This mirrors the recommendation in the reference report: allow users to act unilaterally after sufficient time has passed, removing the privileged role's ability to permanently block them.

---

### Proof of Concept

```
1. Attacker deploys a wallet and calls:
   entropy.register(highFeeInWei, commitment, metadata, chainLength, uri)
   // Attacker is now a registered provider

2. Victim calls:
   entropy.requestV2{value: highFeeInWei + pythFee}(
       attackerAddress, userContribution, gasLimit
   )
   // Inside requestHelper:
   //   providerInfo.accruedFeesInWei += highFeeInWei  ← fee credited NOW
   //   request stored with callbackStatus = CALLBACK_NOT_STARTED

3. Attacker does nothing — never calls revealWithCallback.
   // No timeout exists. Request stays in-flight forever.
   // Victim's callback contract never fires.

4. Attacker calls:
   entropy.withdraw(highFeeInWei)
   // Fee is transferred to attacker. Victim has no recourse.
```

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    // Calling this function will transfer `amount` wei to the caller (provided that they have accrued a sufficient

    function withdraw(uint128 amount) public override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            msg.sender
        ];

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(msg.sender, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            msg.sender,
            msg.sender,
            amount,
            bytes("")
        );
    }

    function revealWithCallback(
        address provider,
        uint64 sequenceNumber,
        bytes32 userContribution,
        bytes32 providerContribution
    ) public override {
        EntropyStructsV2.Request storage req = findActiveRequest(
            provider,
            sequenceNumber
        );

        if (
            !(req.callbackStatus ==
                EntropyStatusConstants.CALLBACK_NOT_STARTED ||
                req.callbackStatus == EntropyStatusConstants.CALLBACK_FAILED)
        ) {
            revert EntropyErrors.InvalidRevealCall();
        }

        bytes32 randomNumber;
        (randomNumber, ) = revealHelper(
            req,
            userContribution,
            providerContribution
        );
