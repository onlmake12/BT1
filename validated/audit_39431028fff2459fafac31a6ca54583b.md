### Title
Epoch-Boundary Stake Timing Allows Full-Epoch OIS Rewards With Near-Zero Warmup Exposure - (`governance/pyth_staking_sdk/src/utils/position.ts`, `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The Oracle Integrity Staking (OIS) program distributes rewards on a per-epoch basis using a binary `LOCKED`/not-`LOCKED` position state. Because the warmup period is defined as "the remainder of the current epoch," a user who stakes at the very end of epoch N incurs a warmup of near-zero wall-clock time, yet their position becomes fully `LOCKED` for the entirety of epoch N+1 and earns the same full-epoch reward as a user who staked at the very beginning of epoch N. This is a direct on-chain analog to the off-chain snapshot timing attack described in the reference report.

---

### Finding Description

The `Position` struct stores two epoch-indexed fields: `activationEpoch` and `unlockingStart`. [1](#0-0) 

When `create_position` is called, `activationEpoch` is set to `currentEpoch + 1`. The position is in `LOCKING` (warmup) state for the remainder of the current epoch — which can be as short as a single Solana slot if the transaction lands one slot before the epoch boundary (every Thursday 00:00 UTC). [2](#0-1) 

At the epoch boundary the position transitions to `LOCKED` with no further action required. The `advance_delegation_record` instruction then distributes rewards proportional to the user's `LOCKED` delegation for that epoch. [3](#0-2) 

The reward calculation uses `total_delegation` and `delta_delegation` from the pool's `DelegationState`, which are snapshotted at epoch boundaries — not time-weighted within an epoch. [4](#0-3) 

**Attack sequence:**

1. Monitor the epoch clock (`epochToDate(currentEpoch + 1n)`) and submit `create_position` (delegate to a publisher) in the final slot of epoch N. Warmup ≈ 0 seconds.
2. At epoch N+1 start, position is `LOCKED`. The user earns the full epoch N+1 reward `R = y · min(S_p, C_p)` — identical to a user who staked at the very beginning of epoch N.
3. At epoch N+2 start, call `undelegate`. Position enters `PREUNLOCKING` (cooldown epoch N+2) then `UNLOCKING` (epoch N+3), then `UNLOCKED`.
4. Call `advance_delegation_record` to claim epoch N+1 rewards. [5](#0-4) 

The user earns a full epoch's rewards while their tokens were only in warmup for a negligible duration.

---

### Impact Explanation

A rational actor can repeatedly cycle stake → earn full epoch reward → unstake across every epoch boundary, capturing OIS rewards while minimizing the time their capital is committed. Because the epoch is deterministic and public (every Thursday 00:00 UTC), the timing is trivially automatable. The economic gain per cycle is `y · min(S_p, C_p)` for one epoch, minus two Solana transaction fees. At any non-zero reward rate `y`, this is profitable for sufficiently large `S_p`.

The cooldown period (two epochs of `PREUNLOCKING`/`UNLOCKING`) means tokens remain subject to slashing for epochs N+2 and N+3, which partially mitigates the attack — but does not eliminate the reward-capture asymmetry, since the user earns rewards only for epoch N+1 while bearing slashing risk for three epochs total. A sophisticated attacker can price this risk and still profit when `y` is large enough.

**Note:** Per OP-PIP-103, `y = 0` currently, so no rewards are distributed at present. The vulnerability is structural and becomes exploitable the moment governance sets `y > 0`. [6](#0-5) 

---

### Likelihood Explanation

- The epoch boundary is fully deterministic and publicly known (every Thursday 00:00 UTC).
- No privileged access is required; any wallet holder can call `create_position` / `undelegate`.
- The attack is automatable with a simple cron job or on-chain keeper.
- The only cost is two transaction fees per epoch cycle plus slashing risk during cooldown.
- Likelihood is **high** once `y > 0` is re-enabled by governance. [7](#0-6) 

---

### Recommendation

1. **Time-weight the reward calculation**: Instead of awarding a full epoch's reward to any position that was `LOCKED` at the epoch boundary, weight rewards by the number of epochs the position has been continuously `LOCKED`. A position that activated in epoch N+1 should earn a fraction of the epoch N+1 reward proportional to `(epochs_locked / total_epochs)`.

2. **Minimum lock duration**: Require a position to have been `LOCKED` for at least one full epoch before it is eligible for rewards (i.e., `activationEpoch + 1 <= rewardEpoch`).

3. **Extend warmup to a full epoch**: Change `activationEpoch = currentEpoch + 2` so that a position staked at the end of epoch N does not become `LOCKED` until epoch N+2, ensuring at least one full epoch of warmup regardless of when within the epoch the stake was placed.

### Proof of Concept

```
Epoch N   |████████████████████████████████████████████|  stake at last slot
Epoch N+1 |████████████████████████████████████████████|  LOCKED → full reward R
Epoch N+2 |████████████████████████████████████████████|  PREUNLOCKING (cooldown 1)
Epoch N+3 |████████████████████████████████████████████|  UNLOCKING (cooldown 2)
Epoch N+4 |                                             |  UNLOCKED, tokens free

User earns: R = y · min(S_p, C_p)   (full epoch N+1 reward)
Warmup time: ~0 seconds (one slot at end of epoch N)
```

Relevant on-chain entry points (unprivileged):
- `create_position` (staking program) — sets `activationEpoch = currentEpoch + 1`
- `undelegate` (integrity pool program) — sets `unlockingStart = currentEpoch + 1`
- `advance_delegation_record` (integrity pool program) — distributes rewards for elapsed epochs [8](#0-7) [9](#0-8)

export const getPositionState = (
  position: Position,
  currentEpoch: bigint,
): PositionState => {
  if (currentEpoch < position.activationEpoch) {
    return PositionState.LOCKING;
  }
  if (!position.unlockingStart) {
    return PositionState.LOCKED;
  }
  const hasActivated = position.activationEpoch <= currentEpoch;
  const unlockStarted = position.unlockingStart <= currentEpoch;
  const unlockEnded = position.unlockingStart + 1n <= currentEpoch;

  if (hasActivated && !unlockStarted) {
    return PositionState.PREUNLOCKING;
  } else if (unlockStarted && !unlockEnded) {
    return PositionState.UNLOCKING;
  } else {
    return PositionState.UNLOCKED;
  }
};

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }
