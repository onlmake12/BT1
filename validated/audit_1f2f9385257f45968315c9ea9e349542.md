### Title
Permanent Freezing of All Protocol Update Fees — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every SUI fee paid via `update_single_price_feed` is deposited into the calling `PriceInfoObject`'s `FEE_STORAGE_KEY` dynamic object field and can never be removed. No public, friend, or governance function in the entire Sui contract suite withdraws or transfers those coins. The `fee_recipient_address` stored in `State` is vestigial and is never used to receive funds.

### Finding Description
`pyth::update_single_price_feed` collects a `Coin<SUI>` fee from the caller and immediately calls `price_info::deposit_fee_coins`, which joins the coin into a `Coin<SUI>` stored as a dynamic object field (`FEE_STORAGE_KEY`) on the shared `PriceInfoObject`. [1](#0-0) 

`deposit_fee_coins` has two branches: add a new `Coin<SUI>` field if absent, or `coin::join` into the existing one. Neither branch ever removes coins. [2](#0-1) 

`PriceInfoObject` is a shared object (`transfer::public_share_object`). On Sui, dynamic object fields of a shared object are only accessible through the module that owns the type — and no function in `price_info.move` or any other module in the package removes or borrows-and-transfers the `FEE_STORAGE_KEY` coin. [7](#0-6) 

The governance action `set_fee_recipient` only updates the `fee_recipient_address` field in `State`; it does not sweep accumulated coins from any `PriceInfoObject`. [8](#0-7) 

### Impact Explanation
All SUI tokens paid as update fees are permanently locked inside shared `PriceInfoObject` objects. The protocol can never collect its revenue. The `fee_recipient_address` governance parameter is entirely inert with respect to actual coin flow. This matches the stated scope: **permanent freezing of unclaimed yield**.

### Likelihood Explanation
This is triggered by every single call to `update_single_price_feed` — the primary price-update entry point used by all integrators. It is 100% reproducible with no special preconditions. The `set_fee_recipient` module's own comment confirms the fee-forwarding mechanism was intentionally removed and never replaced.

### Recommendation
Add a privileged `withdraw_fees` function (gated on `LatestOnly` or a governance capability) to `price_info.move` that removes the `FEE_STORAGE_KEY` coin from a `PriceInfoObject` and transfers it to `state::get_fee_recipient(pyth_state)`. Alternatively, change `update_single_price_feed` to immediately `transfer::public_transfer` the fee coin to `state::get_fee_recipient(pyth_state)` instead of depositing it into the `PriceInfoObject`.
