### Title
Fan-out `ckb_wait` Fails for Second Waiter Due to Premature `terminated_vms` Eviction — (`script/src/scheduler.rs`)

---

### Summary

`process_message_box` removes a terminated VM's exit-code record from `terminated_vms` after the **first** successful `ckb_wait` consumer. Any subsequent VM that calls `ckb_wait` on the same already-terminated target finds it absent from both `terminated_vms` and `states`, and unconditionally receives `WAIT_FAILURE` instead of `SUCCESS`.

---

### Finding Description

**Root cause — `scheduler.rs` line 575:** [1](#0-0) 

When `Message::Wait(vm_id, args)` is processed and the target is already in `terminated_vms`, the handler:
1. Writes the exit code and sets `A0 = SUCCESS` for the waiting VM.
2. Calls `self.terminated_vms.retain(|id, _| id != &args.target_id)` — **permanently deleting** the entry.

The fallthrough check at line 578 then gates on `self.states.contains_key(&args.target_id)`: [2](#0-1) 

When C terminates, it is removed from `states` at line 402: [3](#0-2) 

So after the first waiter consumes the `terminated_vms` entry, C is present in **neither** map. Any second waiter hits the `WAIT_FAILURE` branch.

**Why the "same-iteration" framing in the question is incorrect:**

The scheduler is single-threaded and runs exactly one VM per `iterate_inner` call. Each `Wait::ecall` pushes one message and immediately returns `Err(VMError::Yield)`: [4](#0-3) 

`iterate_process_results` calls `process_message_box` before doing anything else, draining the box after every single VM yield: [5](#0-4) 

Two Wait messages can therefore **never** be in the box simultaneously. The bug does not require that — it triggers across two separate iterations.

**Concrete reachable sequence:**

1. Root VM spawns C, A, and B (all Runnable).
2. C runs and terminates → `terminated_vms.insert(C, code)`, C removed from `states`.
3. A runs, calls `ckb_wait(C)` → message processed → A gets `SUCCESS`, **C evicted from `terminated_vms`**.
4. B runs, calls `ckb_wait(C)` → message processed → C absent from `terminated_vms` and `states` → B gets `WAIT_FAILURE`.

This is fully deterministic and requires no race condition.

---

### Impact Explanation

Any CKB script using fan-out wait semantics — where more than one spawned VM calls `ckb_wait` on the same child — will have the second (and all subsequent) waiters receive `WAIT_FAILURE`. This causes incorrect script validation: a legitimately correct multi-VM script will fail on-chain. Conversely, a script that is supposed to fail may pass if the author exploits the ordering to suppress an expected error code delivery.

---

### Likelihood Explanation

The pattern is reachable by any unprivileged script author submitting a transaction for validation. No special privileges, keys, or majority hashpower are required. The trigger is purely a function of spawn/wait ordering within a single script execution, which the script author fully controls.

---

### Recommendation

Replace the destructive `retain` with a non-destructive read. The `terminated_vms` entry for C should only be removed once **no living VM can ever issue `ckb_wait(C)` again** (i.e., when all VMs that could reference C have themselves terminated or have already successfully waited). The simplest fix is to remove the `retain` call entirely and instead track, per terminated VM, how many waiters have consumed it — or simply never evict from `terminated_vms` until the scheduler itself terminates.

---
