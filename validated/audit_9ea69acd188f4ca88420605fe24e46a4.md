### Title
O(difficulties Г— log N) Unbounded DB Reads per `GetLastStateProof` Enables I/O Exhaustion вЂ” (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged remote peer can send a `GetLastStateProof` message with up to 998 attacker-controlled difficulty values. Each value triggers a binary search over the full chain height, performing 2 DB reads per iteration. The only guard is a flat count check that does not account for the O(log N) cost per difficulty. The `LightClientProtocol` handler has no per-peer rate limiter, unlike the `Relayer` protocol. On a tall chain (e.g., 10M blocks), a single request causes ~46,000 DB reads, and repeated requests cause I/O exhaustion.

---

### Finding Description

**Entry point вЂ” limit check passes with 998 difficulties:**

The guard at `execute()` is:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // 1000
```

With `difficulties.len()=998` and `last_n_blocks=1`: `998 + 2 = 1000`, which is **not greater than** 1000, so the check passes. [1](#0-0) [2](#0-1) 

**Binary search per difficulty вЂ” O(log N) iterations each:**

`get_block_numbers_via_difficulties` iterates over all 998 difficulties. For each, it calls `get_first_block_total_difficulty_is_not_less_than`, which performs a binary search over `[start_block_number, end_block_number]`. [3](#0-2) 

The `start_block_number` is updated to `num - 1` after each found block. An attacker crafts difficulties so each found block is near the start of the range (e.g., blocks 1, 2, 3, вЂ¦, 998), keeping `start_block_number` near 0 and the search range near [0, N] for all 998 iterations. [4](#0-3) 

**2 DB reads per binary search step:**

Each call to `get_block_total_difficulty` performs `get_block_hash(number)` followed by `get_block_ext(&block_hash)` вЂ” two sequential DB reads. [5](#0-4) 

**No rate limiter on the light client protocol:**

`LightClientProtocol` has no `rate_limiter` field and `try_process` dispatches directly to handlers with no per-peer throttling. This contrasts with `Relayer`, which has an explicit `RateLimiter` checked before every message dispatch. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

On a chain of height N=10,000,000 (mainnet scale):
- Binary search depth per difficulty: вЊ€logв‚‚(10M)вЊ‰ = 23 iterations
- DB reads per request: 998 Г— 23 Г— 2 = **~46,000 DB reads**
- An attacker sending requests at even modest rates (e.g., 10/s) drives ~460,000 DB reads/second from a single peer connection, saturating RocksDB I/O and degrading or halting normal node operation.

---

### Likelihood Explanation

Any peer that can connect to a node with the light client protocol enabled can send this message. No authentication, PoW, or privileged role is required. The attacker only needs to know the chain's total difficulty values for a few low-numbered blocks (public information from the chain), or can simply send 998 monotonically increasing difficulty values within the valid range. The attack is repeatable indefinitely with no ban triggered (a well-formed request returns `Status::ok()`).

---

### Recommendation

1. **Add a per-peer rate limiter** to `LightClientProtocol` mirroring the `governor`-based `RateLimiter` already used in `Relayer`.
2. **Bound computational cost, not just count**: Replace the flat `GET_LAST_STATE_PROOF_LIMIT` check with a limit that accounts for the O(log N) factor, e.g., cap `difficulties.len()` to a much smaller constant (e.g., 64 or 128).
3. **Amortize the binary search**: Since difficulties are monotonically increasing and `start_block_number` advances, enforce that each new difficulty must correspond to a block strictly after the previous result, preventing the attacker from keeping the search range large.

---

### Proof of Concept

1. Connect as an unprivileged peer to a full node with `--light-client` enabled.
2. Obtain the total difficulty of block #1 (public chain data).
3. Craft a `GetLastStateProof` with:
   - `last_hash` = current tip hash
   - `start_number` = 0
   - `last_n_blocks` = 1
   - `difficulties` = 998 monotonically increasing U256 values, each equal to the total difficulty of blocks 1 through 998
4. Send the message. The server executes 998 binary searches each over [0, N], performing ~46,000 DB reads.
5. Repeat in a tight loop. Observe RocksDB read IOPS spike and node responsiveness degrade proportionally to chain height.
6. Confirm the invariant violation: DB read count grows as O(difficulties.len() Г— log(chain_height)), not O(1).
