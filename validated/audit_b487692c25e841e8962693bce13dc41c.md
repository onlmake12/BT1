Audit Report

## Title
Missing Per-Peer Rate Limiter in `BlockFilter` Enables Head-of-Line Blocking DoS — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
The `BlockFilter` protocol handler processes all incoming messages from all peers sequentially in a single async task with no per-peer rate limiting. A single unprivileged peer can flood `GetBlockFilterCheckPoints` messages, each triggering up to 4,000 synchronous DB reads, monopolizing the handler and starving legitimate light-client peers of `GetBlockFilterHashes` and `GetBlockFilters` responses. The root cause is a concrete, verifiable absence of a rate limiter that is present in comparable handlers (`Relayer`, `HolePunching`).

## Finding Description

**Sequential dispatch model:** `CKBHandler::received` in `network/src/protocols/mod.rs` (L365–384) calls `self.handler.received(...).await` on a `Box<dyn CKBProtocolHandler>` that takes `&mut self`. Tentacle's `ServiceProtocol` contract means all messages for a given protocol instance are dispatched through the same single async context, one at a time. [1](#0-0) 

**Full await before next message:** `BlockFilter::received` at L152 calls `self.process(nc, peer_index, msg).await` and does not return until the call completes, meaning no other peer's message can begin processing until the current one finishes. [2](#0-1) 

**Per-message cost:** `GetBlockFilterCheckPointsProcess::execute` runs a loop of up to `BATCH_SIZE = 2000` iterations (L43), each performing two synchronous DB reads — `get_block_hash` followed by `get_block_filter_hash` (L44–46) — with no `.await` yield point inside the loop. This totals up to 4,000 blocking DB reads per message. [3](#0-2) [4](#0-3) 

**No rate limiter in `BlockFilter`:** The struct contains only `shared: Arc<SyncShared>` — no `rate_limiter` field, no per-peer quota check anywhere in `try_process`. [5](#0-4) 

**Contrast with `Relayer`:** `Relayer` has `rate_limiter: RateLimiter<(PeerIndex, u32)>` enforcing 30 req/sec per peer per message type, checked at the top of `try_process` before any dispatch. [6](#0-5) 

**Contrast with `HolePunching`:** `HolePunching` has both `rate_limiter` and `forward_rate_limiter`, checked before any processing in `received`. [7](#0-6) [8](#0-7) 

The exploit path is: attacker opens a single TCP connection on `SupportProtocols::Filter`, sends a continuous stream of `GetBlockFilterCheckPoints` messages (e.g., `start_number = 0`). Each message occupies the handler for the duration of up to 4,000 synchronous DB reads. All other peers' messages — including `GetBlockFilterHashes` and `GetBlockFilters` from legitimate light clients — queue behind the flood and are delayed proportionally.

## Impact Explanation

This is a **High** severity finding matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

The Filter protocol is the primary data-serving infrastructure for CKB light clients. A single attacker peer with one TCP connection can continuously saturate the handler, rendering the filter service non-functional for all connected light clients on the targeted node. The cost to the attacker is negligible (one connection, small messages); the cost to the network is degraded light-client infrastructure, which is a core part of CKB's scalability design.

## Likelihood Explanation

The attack requires only a single TCP connection to a CKB node with the Filter protocol enabled. No special privileges, keys, or hashpower are needed. The `start_number` field is attacker-controlled and unchecked. The absence of rate limiting is a concrete, verifiable code gap — not a configuration issue or edge case. The attack is continuously repeatable and self-sustaining.

## Recommendation

Add a per-peer, per-message-type rate limiter to `BlockFilter`, mirroring the pattern in `Relayer`:

```rust
pub struct BlockFilter {
    shared: Arc<SyncShared>,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Check the limiter at the top of `try_process` before dispatching to any `*Process::execute`. A quota of 5–10 `GetBlockFilterCheckPoints` requests per second per peer is sufficient for legitimate light-client use, as checkpoint data changes only with new blocks.

## Proof of Concept

1. Start a CKB node with the Filter protocol enabled and a populated block filter DB.
2. Connect two mock peers (peer A and peer B) to the node on `SupportProtocols::Filter`.
3. Have peer A send `GetBlockFilterCheckPoints` messages in a tight loop with `start_number = 0`.
4. Have peer B send `GetBlockFilterHashes` messages at a steady rate (e.g., 1/sec).
5. Record peer B's response latency with peer A idle vs. peer A flooding.
6. Assert that peer B's latency increases proportionally to peer A's flood rate, confirming head-of-line blocking.

    pub async fn execute(self) -> Status {
        let active_chain = self.filter.shared.active_chain();
        let start_number: BlockNumber = self.message.to_entity().start_number().into();
        let latest: BlockNumber = active_chain.get_latest_built_filter_block_number();

        let mut block_filter_hashes = Vec::new();

        if latest >= start_number {
            let mut block_number = start_number;
            for _ in 0..BATCH_SIZE {
                if let Some(block_filter_hash) = active_chain
                    .get_block_hash(block_number)
                    .and_then(|block_hash| active_chain.get_block_filter_hash(&block_hash))
                {
                    block_filter_hashes.push(block_filter_hash);
                } else {
                    break;
                }
                let Some(next_block_number) = block_number.checked_add(CHECK_POINT_INTERVAL) else {
                    break;
                };
                block_number = next_block_number;
            }
            let content = packed::BlockFilterCheckPoints::new_builder()
                .start_number(start_number)
                .block_filter_hashes(block_filter_hashes)
                .build();

            let message = packed::BlockFilterMessage::new_builder()
                .set(content)
                .build();
            async_send_message_to(&self.nc, self.peer, &message).await
        } else {
            Status::ignored()
        }
    }
