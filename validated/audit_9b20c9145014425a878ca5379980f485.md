### Title
Unprivileged Caller Can Steal Rent Lamports from Any `SignatureSet` Account via `close_signature_set` - (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

---

### Summary

The `close_signature_set` instruction closes a `SignatureSet` account and transfers its lamports to `sol_destination`. The only constraint on `sol_destination` is that it must be a `Signer`. There is no check that `sol_destination` is the original payer who funded the `SignatureSet`. Any unprivileged attacker who can sign a transaction can redirect the rent lamports of any `SignatureSet` account to themselves.

---

### Finding Description

The `CloseSignatureSet` account context is: [1](#0-0) 

`sol_destination` is declared only as `#[account(mut)]` + `Signer` â€” no `address`, `constraint`, or `has_one` ties it to the original funder of the `SignatureSet`.

The `SignatureSet` is created in `verify_signatures` with `payer = payer` (the relayer's keypair): [2](#0-1) 

The `payer` pubkey is **never stored** in the `SignatureSet` account. The account only stores `sig_verify_successes`, `message_hash`, and `guardian_set_index`: [3](#0-2) 

After the relayer calls `post_vaa`, the `PostedVaaV1` account stores the `signature_set` pubkey in its `info.signature_set` field: [4](#0-3) 

This pubkey is publicly readable on-chain. An attacker reads any `PostedVaaV1` account, extracts `info.signature_set`, and calls `close_signature_set` with their own keypair as `sol_destination`. The `has_one = signature_set` constraint on `posted_vaa` is satisfied (the attacker supplies the correct `signature_set` pubkey), and `close = sol_destination` transfers all lamports to the attacker. [5](#0-4) 

---

### Impact Explanation

Every relayer who uses the legacy `verify_signatures` + `post_vaa` flow funds a `SignatureSet` keypair account with rent (~0.00089 SOL for a 19-guardian set). After `post_vaa` succeeds, the `SignatureSet` is no longer needed for VAA validity but still holds rent. An attacker can drain every such account on-chain. At scale (high-frequency Pyth price update relaying), this constitutes systematic theft of SOL from legitimate relayers.

---

### Likelihood Explanation

- The attack requires no privileged access â€” only the ability to sign a transaction.
- The target `signature_set` pubkey is trivially discoverable by scanning `PostedVaaV1` accounts.
- The attack is executable immediately after `post_vaa` completes, before the relayer can reclaim rent themselves.
- No race condition or timing constraint beyond normal transaction ordering.

---

### Recommendation

Add a constraint that `sol_destination` must equal the original payer of the `SignatureSet`. Since the payer is not currently stored in `SignatureSet`, the fix requires one of:

1. **Store the payer** in `SignatureSet` during `verify_signatures` and add `constraint = sol_destination.key() == signature_set.payer @ CoreBridgeError::InvalidSolDestination` in `CloseSignatureSet`.
2. **Require the `SignatureSet` to be a signer** (since it is a keypair account, only the original relayer who holds the private key can sign for it), adding `#[account(mut, signer, close = sol_destination)]` on `signature_set`.

Option 2 is the minimal change: the `SignatureSet` is already a keypair account whose private key is held by the relayer, so requiring it as a signer enforces that only the relayer can authorize closure.

---

### Proof of Concept

```rust
// Setup: victim_relayer calls verify_signatures (funds signature_set) then post_vaa
// Result: PostedVaaV1 at pda stores info.signature_set = signature_set_keypair.pubkey()

// Attack:
let ix = close_signature_set(
    attacker_keypair.pubkey(),   // sol_destination â€” only needs to be a signer
    posted_vaa_pda,              // posted_vaa â€” valid, has_one = signature_set satisfied
    signature_set_keypair.pubkey(), // signature_set â€” read from posted_vaa.info.signature_set
);
let tx = Transaction::new_signed_with_payer(
    &[ix],
    Some(&attacker_keypair.pubkey()),
    &[&attacker_keypair],        // attacker signs; no victim key needed
    recent_blockhash,
);
banks_client.process_transaction(tx).await.unwrap();

// Assert: attacker_keypair lamports increased by SignatureSet rent amount
// Assert: signature_set account no longer exists
```
