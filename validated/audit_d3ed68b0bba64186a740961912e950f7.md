### Title
Unauthenticated `ConnectionRequestDelivered` Allows Hole-Punch Session Hijacking via Forged `from` Field — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

The `ConnectionRequestDeliveredProcess::execute()` function determines whether the local node is the "originator" of a hole-punch session by comparing `self_peer_id` to the attacker-controlled `content.from` field. Because a node's peer ID is public information and there is no cryptographic binding between the message and the session that legitimately relayed it, any connected peer can forge a `ConnectionRequestDelivered` with `content.from` set to the victim's own peer ID, causing the victim to (1) evict a legitimate `inflight_requests` entry, (2) send a `ConnectionSync` back to the attacker's session, and (3) spawn outbound TCP NAT-traversal tasks toward attacker-controlled addresses, potentially establishing a full P2P session with the attacker.

---

### Finding Description

The routing logic in `execute()` uses an empty `content.route` as the signal that the message has reached its final destination, then branches on whether `self_peer_id == &content.from` to decide if the local node is the originator: [1](#0-0) 

When `content.route` is empty and `content.from` equals the local peer ID, the node unconditionally:

1. Removes `content.to` from `inflight_requests`: [2](#0-1) 

2. Sends a `ConnectionSync` back to `self.peer` (the session that delivered the message — the attacker): [3](#0-2) 

3. Spawns NAT traversal tasks toward `content.listen_addrs` (attacker-controlled), and on success calls `raw_session()` to register a full outbound P2P session: [4](#0-3) 

The `content.from` field is parsed directly from the wire message with no authentication: [5](#0-4) 

The only guards present are structural validity (well-formed peer ID bytes), address count limits, and a per-`(from, to, msg_item_id)` rate limiter — none of which bind the message to the session that legitimately originated the hole-punch request. [6](#0-5) 

The `inflight_requests` map is keyed by `PeerId` and populated in `notify()` when the victim broadcasts `ConnectionRequest` messages: [7](#0-6) 

Those `ConnectionRequest` broadcasts are sent to `sqrt(total_peers)` connected peers via gossip, so the attacker (being a connected peer) directly observes the `to` field and learns which peer IDs are currently in `inflight_requests`: [8](#0-7) 

---

### Impact Explanation

- **Hole-punch session eviction (DoS):** The legitimate `inflight_requests` entry for the target peer is removed. The victim will not attempt NAT traversal toward the real peer.
- **`ConnectionSync` sent to attacker:** The attacker's session receives a `ConnectionSync` message intended for the originating peer, leaking hole-punch state.
- **Outbound TCP connections to attacker-controlled addresses:** `try_nat_traversal` retries TCP connections to attacker-supplied IP:port for up to 30 seconds. If any connection succeeds, `raw_session()` promotes it to a full P2P session, effectively injecting the attacker as a peer under the identity of the legitimate target.

---

### Likelihood Explanation

The attacker needs only a standard P2P connection to the victim (no special privileges). The victim's peer ID is public (exchanged during the Identify handshake). The target peer IDs in `inflight_requests` are directly observable from the gossip `ConnectionRequest` broadcasts the victim sends. The forged message requires no cryptographic material. The attack is repeatable and can be executed immediately after observing a `ConnectionRequest`.

---

### Recommendation

The `from` field in `ConnectionRequestDelivered` must not be trusted as proof of origin. The fix should verify that the session delivering the message (`self.peer`) is actually the peer whose peer ID matches `content.from`, by looking up `self.peer` in the peer registry and comparing its peer ID:

```rust
// In the None-route, self-is-originator branch, before removing inflight_requests:
let sender_peer_id = self.protocol.network_state
    .peer_registry.read()
    .get_peer(self.peer)
    .map(|p| p.peer_id.clone());

if sender_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::Ignore.with_context("sender peer id does not match content.from");
}
```

This ensures the message was delivered by the peer that claims to be `content.from`, not by an arbitrary connected peer.

### Proof of Concept

1. Attacker A establishes a standard P2P connection to victim V.
2. A waits to receive a `ConnectionRequest` broadcast from V (gossip). The message contains `from=V_peer_id`, `to=T_peer_id`. V now has `inflight_requests[T_peer_id] = timestamp`.
3. A constructs a `ConnectionRequestDelivered` with:
   - `from` = `V_peer_id` (V's own public peer ID)
   - `to` = `T_peer_id` (observed from step 2)
   - `route` = `[]` (empty, so `route.last()` is `None`)
   - `sync_route` = `[]`
   - `listen_addrs` = `[/ip4/<attacker_ip>/tcp/<attacker_port>/p2p/<T_peer_id>]`
4. A sends this message to V over the HolePunching protocol.
5. V's `execute()` evaluates: `route.last()` → `None`; `self_peer_id == content.from` → `true` (since `from = V_peer_id`).
6. V calls `inflight_requests.remove(&T_peer_id)` — entry evicted. ✓
7. V calls `respond_sync(...)` sending `ConnectionSync` to A's session. ✓
8. V calls `try_nat_traversal(ttl, [attacker_addr])` — spawns 30-second TCP retry loop toward attacker's IP:port. ✓
9. If A's server accepts the TCP connection, V calls `raw_session()` and registers A as a peer under `T_peer_id`'s address. ✓
