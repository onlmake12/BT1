### Title
`UpgradeContract` Governance Action Is a Silent No-Op — (`File: target_chains/stylus/contracts/pyth-receiver/src/governance.rs`)

---

### Summary

The Stylus Pyth receiver contract silently ignores `UpgradeContract` governance instructions. The governance VAA is fully verified and its sequence number is permanently consumed, but no upgrade is ever performed. The contract is therefore permanently un-upgradeable via governance on the Stylus/Arbitrum target chain.

---

### Finding Description

In `execute_governance_instruction_internal`, after the VAA is verified and the governance sequence is advanced (making the VAA non-replayable), the action dispatch match arm for `UpgradeContract` is an empty block:

```rust
// target_chains/stylus/contracts/pyth-receiver/src/governance.rs, line 62
GovernancePayload::UpgradeContract(_payload) => {}
``` [1](#0-0) 

The sequence number is consumed unconditionally inside `verify_governance_vm` **before** the match dispatch:

```rust
// governance.rs line 277-279
receiver
    .last_executed_governance_sequence
    .set(U64::from(current_sequence));
``` [2](#0-1) 

A private `upgrade_contract` method exists but is never called from the governance handler and panics with `unimplemented!()`:

```rust
fn upgrade_contract(&self, _new_implementation: FixedBytes<32>) {
    unimplemented!("Upgrade contract not yet implemented");
}
``` [3](#0-2) 

The `GovernanceAction::UpgradeContract` variant is fully parsed and accepted (action byte `0`): [4](#0-3) 

And the payload is correctly deserialized into an `UpgradeContract` struct with `new_implementation`: [5](#0-4) 

So the full pipeline — VAA verification, sequence consumption, payload parsing — succeeds, but the upgrade itself is never executed.

---

### Impact Explanation

- The Pyth Stylus contract **cannot be upgraded** via governance. Any security patch, bug fix, or feature upgrade is permanently blocked on this chain.
- Every governance upgrade VAA submitted advances `last_executed_governance_sequence`, permanently consuming that sequence slot. The VAA cannot be replayed after the sequence is consumed.
- The contract is frozen at its current implementation with no on-chain recovery path through the governance mechanism.

---

### Likelihood Explanation

Governance will eventually need to upgrade the Stylus contract (e.g., to apply a security fix or add a feature). When that happens, the upgrade VAA will be submitted, silently consumed, and the upgrade will not occur. The deployer/governance team may not immediately notice the failure since the transaction succeeds with no revert. This is a latent but certain failure when an upgrade is attempted.

---

### Recommendation

Implement the `UpgradeContract` arm to call the actual upgrade logic, analogous to how EVM handles it in `PythUpgradable.upgradeUpgradableContract`. At minimum, the empty arm should revert with an explicit error (e.g., `return Err(PythReceiverError::UpgradeNotImplemented)`) so that the governance sequence is **not** consumed and the VAA can be retried after the implementation is added. The current behavior silently discards the VAA and advances the sequence, which is the worst possible outcome.

---

### Proof of Concept

1. Governance signs and publishes a Wormhole VAA with action byte `0` (`UpgradeContract`) targeting the Stylus chain, containing a valid 32-byte `new_implementation` address.
2. Any relayer calls `execute_governance_instruction(vaa_bytes)` on the Stylus Pyth contract.
3. `verify_governance_vm` succeeds: emitter chain/address match, sequence is greater than `last_executed_governance_sequence`. The sequence is updated to the VAA's sequence number.
4. `parse_instruction` succeeds: magic, module, action byte `0`, and 32-byte payload are all valid. Returns `GovernancePayload::UpgradeContract(UpgradeContract { new_implementation: ... })`.
5. The match arm `GovernancePayload::UpgradeContract(_payload) => {}` executes — doing nothing.
6. `execute_governance_instruction_internal` returns `Ok(())`.
7. The contract implementation is unchanged. The governance sequence has advanced. The VAA is permanently consumed and cannot be replayed.
