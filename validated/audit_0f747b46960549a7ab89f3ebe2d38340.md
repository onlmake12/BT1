### Title
Stale Descendant Statistics in Tx-Pool After `remove_entry_and_descendants` Corrupts Eviction Ordering — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all parent-child links are torn down **before** `remove_entry` is called on each removed entry. Because `update_ancestors_index_key` resolves ancestors through those same links, it always sees an empty ancestor set and never subtracts the removed entries' weight from the surviving ancestors' `descendants_*` fields. Any transaction that is an ancestor of the removed subtree but is **not** itself removed retains permanently inflated (or deflated) `descendants_count / descendants_size / descendants_cycles / descendants_fee`, and therefore a permanently wrong `EvictKey`. This corrupts the pool's eviction ordering for the lifetime of those ancestor entries.

---

### Finding Description

`remove_entry_and_descendants` in `tx-pool/src/component/pool_map.rs` first strips all link records for every entry in the removed subtree, then calls `remove_entry` on each:

```rust

pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← ALL links torn down here

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links gone
        .collect()
}

### Impact Explanation

**Scenario A — ancestor appears less evictable than it is (pool bloat):**

1. Attacker submits low-fee parent `A` and high-fee children `B`, `C`.
2. `A`'s `descendants_feerate` is inflated by `B` and `C`'s fees → `A`'s `EvictKey` is high → `A` is not evicted.
3. Attacker submits a conflicting transaction that triggers `remove_entry_and_descendants(B)`, removing `B` and `C`.
4. `A`'s `descendants_*` stats are never decremented. `A` retains the inflated `EvictKey`.
5. When the pool is full, `limit_size` evicts other legitimate high-fee transactions instead of `A`.
6. `A` (low-fee) occupies pool space indefinitely, displacing higher-value transactions.

**Scenario B — ancestor appears more evictable than it is (premature eviction):**

1. Attacker submits high-fee parent `A` and low-fee children `B`, `C`.
2. `A`'s `descendants_feerate` is deflated → `A`'s `EvictKey` is low.
3. `B` and `C` are removed via conflict; `A`'s stats are not updated.
4. `A` retains the deflated `EvictKey` and is evicted before lower-fee entries.

Both scenarios allow an unprivileged tx-pool submitter to manipulate which transactions survive pool eviction, undermining fee-market fairness and enabling targeted displacement of competing transactions.

---

### Likelihood Explanation

The trigger is straightforward: submit a parent transaction, attach children, then submit a conflicting transaction that removes the children. This is achievable by any node that can submit transactions to the RPC (`send_transaction`) or relay them over the P2P network. No privileged access, key material, or majority hashpower is required. The `resolve_conflict` path is exercised on every RBF submission and every block commit that removes pool transactions, making accidental triggering common and deliberate exploitation trivial.

---

### Recommendation

In `remove_entry_and_descendants`, update surviving ancestors' descendant statistics **before** tearing down the links. Specifically, for the root entry being removed (not its descendants, which are also being removed), call `update_ancestors_index_key` while the links are still intact, or collect the ancestor set before calling `remove_entry_links` and apply the weight subtraction afterward:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Collect ancestors of the root (still reachable) before links are torn down,
    // then update their descendant stats for each entry being removed.
    for removed_id in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(removed_id) {
            let entry_clone = entry.inner.clone();
            let ancestors = self.links.calc_ancestors(removed_id);
            for anc_id in &ancestors {
                if !removed_ids.contains(anc_id) {
                    self.entries.modify_by_id(anc_id, |e| {
                        e.inner.sub_descendant_weight(&entry_clone);
                        e.evict_key = e.inner.as_evict_key();
                    });
                }
            }
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

### Proof of Concept

```
Pool state:
  A (fee=100, size=100) → B (fee=1000, size=100) → C (fee=1000, size=100)

After add_entry for A, B, C:
  A.descendants_fee   = 2100  (A+B+C)
  A.descendants_size  = 300
  A.EvictKey.fee_rate = feerate(2100, 300) = 7.0 shannons/byte  [high → not evicted]

Attacker submits tx D that conflicts with B's input.
resolve_conflict calls remove_entry_and_descendants(B):
  - remove_entry_links(B), remove_entry_links(C)   ← links gone
  - remove_entry(B): update_ancestors_index_key(B) → calc_ancestors(B) = {} → A NOT updated
  - remove_entry(C): update_ancestors_index_key(C) → calc_ancestors(C) = {} → A NOT updated

Pool state after:
  A.descendants_fee   = 2100  ← STALE (should be 100)
  A.descendants_size  = 300   ← STALE (should be 100)
  A.EvictKey.fee_rate = 7.0   ← STALE (should be 1.0 shannons/byte)

Pool fills up. limit_size evicts by lowest EvictKey.
A's stale EvictKey=7.0 causes legitimate 5.0 shannons/byte transactions to be evicted first.
A (actual feerate=1.0) survives indefinitely.
```
