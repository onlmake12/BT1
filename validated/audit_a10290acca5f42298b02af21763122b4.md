Audit Report

## Title
Orphan Block Pool Memory Exhaustion via `need_clean` Single-Child Sampling and Missing Size Cap — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::need_clean` samples only the first HashMap-iteration child of a sibling group to decide whether to evict the entire group. An attacker can insert one crafted orphan block per unique unknown parent hash with an arbitrarily large `epoch_number` (accepted verbatim from the header, with no PoW or epoch validation in the non-contextual path). Because each leader group then has exactly one child, `need_clean` deterministically returns `false` for every such group, and `clean_expired_blocks` never removes them. With no hard size cap on `OrphanBlockPool`, the pool grows without bound, exhausting node memory.

## Finding Description

**Root cause — `need_clean` checks only one arbitrary child:**

`InnerPool::need_clean` calls `map.iter().next()` on the inner `HashMap<packed::Byte32, LonelyBlockHash>` to pick a single representative child and decides whether to evict the whole group based solely on that child's `epoch_number`:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
``` [1](#0-0) 

If the attacker inserts exactly one block per unique parent hash, `map.iter().next()` always returns that single block. Setting its `epoch_number` to any value satisfying `epoch_number + EXPIRED_EPOCH >= tip_epoch` makes `need_clean` return `false` deterministically — no HashMap iteration lottery is required.

**`epoch_number` is taken verbatim from the attacker-supplied block header:**

The conversion from `LonelyBlock` to `LonelyBlockHash` reads `epoch_number` directly from the block header with no validation: [2](#0-1) 

**Non-contextual block verification does not check PoW or epoch correctness:**

`ChainService::non_contextual_verify` runs only `BlockVerifier` and `NonContextualBlockTxsVerifier`. `BlockVerifier` covers only cellbase, block bytes, extension, proposals limit, duplicates, and merkle root: [3](#0-2) [4](#0-3) 

`HeaderVerifier` — which runs `PowVerifier` and `EpochVerifier` — requires the parent header and is never invoked for orphan blocks: [5](#0-4) 

**Exploit path confirmed in `OrphanBroker::process_lonely_block`:**

A block whose parent is neither stored nor invalid is unconditionally inserted into the orphan pool: [6](#0-5) 

**No hard cap on orphan block pool size:**

`OrphanBlockPool::insert` has no size guard; `with_capacity` is only a HashMap pre-allocation hint: [7](#0-6) 

Compare with the tx orphan pool, which enforces `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` and actively evicts on overflow: [8](#0-7) [9](#0-8) 

**Cleanup timer fires every 60 seconds:** [10](#0-9) 

## Impact Explanation

An attacker can grow the orphan block pool without bound, consuming all available memory and crashing the targeted CKB node. This matches the allowed bounty impact: **High (10001–15000 points) — "Vulnerabilities which could easily crash a CKB node."**

## Likelihood Explanation

- **Zero mining work required.** `non_contextual_verify` does not invoke `PowVerifier` for orphan blocks, so any nonce is accepted.
- **No privileged access.** The attack path is the standard P2P block-relay path: `Synchronizer::received` → `asynchronous_process_block` → `process_lonely_block` → `orphan_blocks_broker.insert`.
- **Deterministic, not probabilistic.** Inserting exactly one block per unique unknown parent hash means `map.iter().next()` always returns the attacker's crafted block. Setting `epoch_number` to any value where `epoch_number + EXPIRED_EPOCH >= tip_epoch` guarantees `need_clean` always returns `false` — no iteration-order luck needed.
- **Repeatable at disk-write speed.** Each iteration inserts one entry into the pool with no cleanup ever occurring. The only throttle is the `insert_block` DB write, which is still fast enough to exhaust memory before any operator intervention.

## Recommendation

1. **Fix `need_clean` to check all children:** return `true` if *any* child satisfies `epoch_number + EXPIRED_EPOCH < tip_epoch`, or evict individual expired children rather than the whole group atomically.
2. **Enforce a hard cap on orphan block pool size** analogous to `DEFAULT_MAX_ORPHAN_TRANSACTIONS` in the tx orphan pool, with LRU or oldest-epoch eviction when the cap is reached.
3. **Add a minimum `compact_target` check in non-contextual verification** to reject blocks whose declared difficulty is implausibly low relative to the genesis target, preventing zero-work orphan spam even if the cleanup logic is fixed.

## Proof of Concept

```rust
// Mirrors the existing test harness in chain/src/tests/orphan_block_pool.rs
let pool = OrphanBlockPool::with_capacity(10);
let tip_epoch: EpochNumber = 20;

// Each iteration: one block, one unique unknown parent, far-future epoch.
// need_clean always returns false; pool grows by 1 per iteration, never cleaned.
loop {
    let parent = random_byte32(); // unknown to victim node
    let b = make_lonely_block_hash(
        /*parent*/ parent,
        /*epoch_number*/ tip_epoch + EXPIRED_EPOCH + 1, // any value >= tip_epoch - EXPIRED_EPOCH
    );
    pool.insert(b);
    // pool.len() grows unboundedly; clean_expired_blocks(tip_epoch) removes nothing
}

// Minimal single-group invariant violation:
let parent = random_byte32();
let b1 = make_lonely_block_hash(parent.clone(), tip_epoch + EXPIRED_EPOCH + 1);
pool.insert(b1);
let cleaned = pool.clean_expired_blocks(tip_epoch);
assert_eq!(cleaned.len(), 0); // always holds — group is never cleaned
```
