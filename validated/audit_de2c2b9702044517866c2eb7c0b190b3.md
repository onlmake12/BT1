### Title
`PriceUpdateExecuted` Event Emits Keeper Address Instead of Credited Provider — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the internal `emitPriceUpdate` function emits the `PriceUpdateExecuted` event with `msg.sender` as the `provider` field. However, `msg.sender` at that point is the **keeper/executor** who called `executeCallback`, not the actual provider being credited (`providerToCredit`). Off-chain systems and integrators that index `PriceUpdateExecuted` to track which provider fulfilled a request will receive a systematically wrong address.

---

### Finding Description

`executeCallback` accepts an explicit `providerToCredit` parameter — the provider address that should receive credit and fees for fulfilling the request:

```solidity
function executeCallback(
    address providerToCredit,   // <-- actual provider to credit
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
    ...
    _state.providers[providerToCredit].accruedFeesInWei += ...;
    ...
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
}
```

`emitPriceUpdate` is then called without `providerToCredit` being passed in:

```solidity
function emitPriceUpdate(
    uint64 sequenceNumber,
    bytes32[] memory priceIds,
    PythStructs.PriceFeed[] memory priceFeeds
) internal {
    ...
    emit PriceUpdateExecuted(
        sequenceNumber,
        msg.sender,   // BUG: this is the keeper, not providerToCredit
        priceIds,
        prices,
        conf,
        expos,
        publishTimes
    );
}
```

The `PriceUpdateExecuted` event definition names the second parameter `provider` (indexed):

```solidity
event PriceUpdateExecuted(
    uint64 indexed sequenceNumber,
    address indexed provider,   // intended to be the credited provider
    ...
);
```

`msg.sender` inside `emitPriceUpdate` resolves to whoever called `executeCallback` — the keeper — not `providerToCredit`. The keeper and the credited provider are explicitly allowed to differ (after the exclusivity period elapses, any address may call `executeCallback` and credit a different provider).

---

### Impact Explanation

Any off-chain system — indexers, dashboards, provider analytics, keeper bots, or user-facing UIs — that listens to `PriceUpdateExecuted` to determine which provider fulfilled a request will receive the **keeper's address** instead of the **credited provider's address**. This breaks:

- Provider reputation/performance tracking systems.
- Fee accounting dashboards that reconcile on-chain accruals with emitted events.
- Any consumer contract or off-chain bot that uses the event's `provider` field to route logic or verify fulfillment.

The state mutation (`accruedFeesInWei` credited to `providerToCredit`) is correct; only the event is wrong. This is a data-integrity issue for all event consumers, which is the primary interface for off-chain integrations.

---

### Likelihood Explanation

This is triggered on every successful `executeCallback` invocation. Any unprivileged keeper can call `executeCallback`. After the exclusivity period, the keeper and `providerToCredit` will routinely differ (a third-party keeper credits the originally assigned provider). The bug fires on every such fulfillment, making it high-frequency and immediately observable by any event subscriber.

---

### Recommendation

Pass `providerToCredit` into `emitPriceUpdate` and use it in the event emission instead of `msg.sender`:

```solidity
function emitPriceUpdate(
    uint64 sequenceNumber,
    address provider,           // add this parameter
    bytes32[] memory priceIds,
    PythStructs.PriceFeed[] memory priceFeeds
) internal {
    ...
    emit PriceUpdateExecuted(
        sequenceNumber,
        provider,               // use providerToCredit, not msg.sender
        ...
    );
}
```

And update the call site in `executeCallback`:

```solidity
emitPriceUpdate(sequenceNumber, providerToCredit, priceIds, priceFeeds);
```

---

### Proof of Concept

1. Provider A registers and is assigned as the provider for a request.
2. The exclusivity period elapses.
3. Keeper K (a different address) calls `executeCallback(providerA, sequenceNumber, updateData, priceIds)`.
4. `_state.providers[providerA].accruedFeesInWei` is correctly incremented.
5. `emitPriceUpdate` fires `PriceUpdateExecuted(sequenceNumber, K_address, ...)` — emitting the keeper's address.
6. An off-chain indexer records that keeper K (not provider A) fulfilled the request.
7. Provider A's fulfillment history is invisible to all event-based consumers. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function emitPriceUpdate(
        uint64 sequenceNumber,
        bytes32[] memory priceIds,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal {
        int64[] memory prices = new int64[](priceFeeds.length);
        uint64[] memory conf = new uint64[](priceFeeds.length);
        int32[] memory expos = new int32[](priceFeeds.length);
        uint64[] memory publishTimes = new uint64[](priceFeeds.length);

        for (uint i = 0; i < priceFeeds.length; i++) {
            prices[i] = priceFeeds[i].price.price;
            conf[i] = priceFeeds[i].price.conf;
            expos[i] = priceFeeds[i].price.expo;
            // Safe cast because this is a unix timestamp in seconds.
            publishTimes[i] = SafeCast.toUint64(
                priceFeeds[i].price.publishTime
            );
        }

        emit PriceUpdateExecuted(
            sequenceNumber,
            msg.sender,
            priceIds,
            prices,
            conf,
            expos,
            publishTimes
        );
    }
```
