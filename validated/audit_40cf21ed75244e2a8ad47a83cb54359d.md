Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequestDelivered` Enables Relay-Assisted Message Injection and Forced NAT Traversal to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
`ConnectionRequestDeliveredProcess::execute()` never verifies that `content.from` matches the authenticated peer ID of the session that delivered the message. An attacker connected to any relay node can forge `from` to any peer ID and set `route = []`, causing the relay to unconditionally forward the message to the forged peer. If the target has a matching `inflight_requests` entry, it spawns a background task making repeated TCP connection attempts to fully attacker-controlled addresses for up to 30 seconds. The forward rate limiter is keyed entirely on attacker-controlled fields and is trivially bypassed.

## Finding Description

**Root cause — no session-to-`from` binding:**
`DeliverdContent::try_from` validates only that `from` bytes decode to a syntactically valid `PeerId`; it never checks that `from` equals the peer ID of the authenticated TCP session (`self.peer`). [1](#0-0) 

**Relay forwarding path — unconditional on empty route:**
When `content.route` is empty and `self_peer_id != content.from`, `execute()` calls `self.forward_delivered(&content.from)` without any further validation. `forward_delivered` resolves the forged peer ID against the live peer registry and delivers the message verbatim to that session. [2](#0-1) [3](#0-2) 

**Terminal branch — forced outbound TCP connections:**
When the forged message reaches target node C (whose peer ID was placed in `from`), C's `execute()` enters the `else` branch. If `content.to` matches an entry in `inflight_requests`, C calls `try_nat_traversal(ttl, content.listen_addrs)`, spawning a background task that makes repeated TCP connection attempts to the fully attacker-controlled `listen_addrs` for up to 30 seconds (~150 SYN packets at ~200ms intervals). [4](#0-3) [5](#0-4) 

**`listen_addrs` validation does not prevent attacker-controlled IP:port:**
The address validation only checks that the embedded peer ID (if present) matches `content.to`; if no peer ID is present, it appends `to`'s peer ID automatically. The IP:port is entirely attacker-supplied. [6](#0-5) 

**Rate limiter is ineffective:**
The forward rate limiter is keyed on `(content.from, content.to, self.msg_item_id)` — all three fields are fully attacker-controlled. Varying any one of them (e.g., incrementing `msg_item_id`) bypasses the limiter entirely. [7](#0-6) 

**`inflight_requests` entries are observable:**
`inflight_requests` is populated in the `notify` function when a node initiates a `ConnectionRequest`, which is gossip-broadcast to `sqrt(total_peers)` nodes. Any connected peer can observe these broadcasts and learn which peer IDs are in a target node's `inflight_requests`. [8](#0-7) 

**`forward_delivered` free function passes message verbatim:**
The helper only rebuilds the (already-empty) `route` field; all attacker-supplied fields (`from`, `listen_addrs`, `sync_route`) are forwarded unchanged. [9](#0-8) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker with a single standard P2P connection to any relay node can force many target nodes to simultaneously spawn background tasks making high-frequency TCP SYN packets (~150 per 30-second window per task) to arbitrary external addresses. By targeting multiple nodes simultaneously (one crafted message per target, rate limiter bypassed by varying `msg_item_id`), and repeating after `inflight_requests` entries are repopulated by the periodic `notify` timer, the attacker can sustain aggregate outbound TCP connection floods across the network at negligible cost. [10](#0-9) 

## Likelihood Explanation

- Requires only a standard P2P connection to a relay node; no keys, no privileges, no majority hashpower.
- `ConnectionRequest` gossip broadcasts to `sqrt(total_peers)` nodes make `inflight_requests` entries observable to any connected peer. [11](#0-10) 
- The per-session rate limiter (`rate_limiter` keyed on `(session_id, msg.item_id())`) limits messages per session but does not prevent the attack since one message per target is sufficient. [12](#0-11) 
- The attack is repeatable: `inflight_requests` entries are repopulated each `notify` cycle, and the forward rate limiter is bypassed by varying `msg_item_id`.
- Scalable across all peers connected to the relay simultaneously.

## Recommendation

1. **Authenticate `from` against the session.** At the start of `execute()`, retrieve the peer ID associated with `self.peer` from the peer registry and assert it equals `content.from`. Reject with a ban-worthy status code if they differ.
2. **Key the rate limiter on the actual session.** Replace `content.from` in the rate-limiter key with `self.peer` (the authenticated session index), which cannot be forged.
3. **Validate `from` at relay hops.** When forwarding with a non-empty route, verify that `content.from` is not the local peer and that the message arrived from the expected previous hop.

## Proof of Concept

**Setup:** Three nodes A (attacker), B (relay), C (target) — all connected to each other.

1. A connects to B over the HolePunching protocol.
2. A observes C's `ConnectionRequest` gossip broadcast to learn a peer ID `D` currently in C's `inflight_requests`.
3. A crafts a `ConnectionRequestDelivered` message:
   - `from` = C's peer ID
   - `to` = D (from step 2)
   - `route` = `[]`
   - `listen_addrs` = `[attacker_IP:port]` (valid TCP multiaddr, peer ID of D appended automatically)
   - `sync_route` = `[]`
4. A sends the message to B.
5. **Assert:** B's `execute()` finds `route.last() == None`, `B_peer_id != C_peer_id`, calls `forward_delivered(&C_peer_id)`, resolves C's session ID from the peer registry, and delivers the message to C. [2](#0-1) 
6. **Assert:** C's `execute()` enters the `else` branch (`C_peer_id == content.from`). Since `D` is in C's `inflight_requests`, C calls `try_nat_traversal` and spawns a background task making outbound TCP connections to `attacker_IP:port` for 30 seconds. [13](#0-12) 
7. **Observe:** TCP SYN packets arriving at the attacker's listener — confirming forced outbound connection from C to an attacker-controlled endpoint.
8. **Repeat** with varying `msg_item_id` values and different target nodes to bypass the rate limiter and trigger additional `try_nat_traversal` tasks across the network.

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}

pub(crate) fn forward_delivered(
    delivered: packed::ConnectionRequestDeliveredReader<'_>,
) -> packed::ConnectionRequestDelivered {
    let route = delivered.route();
    let message = delivered.to_entity();
    let new_route = if route.is_empty() {
        packed::BytesVec::new_builder().build()
    } else {
        packed::BytesVec::new_builder()
            .extend(
                message
                    .route()
                    .into_iter()
                    .take(route.len().saturating_sub(1)),
            )
            .build()
    };
    message.as_builder().route(new_route).build()
}
