Audit Report

## Title
`check_tx_fee` Uses Raw Byte Size Instead of Transaction Weight for `min_fee_rate` Admission Check, Allowing Cycle-Heavy Transactions to Bypass Fee Rate Enforcement — (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using raw serialized byte size (`tx_size`) as the weight argument to `FeeRate::fee()`, even though `FeeRate` is defined as shannons per kilo-weight where weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, the actual weight can be orders of magnitude larger than `tx_size`, so the size-based check admits transactions whose true fee rate is far below `min_fee_rate`. No weight-based fee rate re-check is performed after `verify_rtx` returns the actual cycle count, making this the only admission gate.

## Finding Description

`FeeRate` is defined as shannons per kilo-weight in `util/types/src/core/fee_rate.rs`: [1](#0-0) 

`FeeRate::fee()` computes `self.0 * weight / 1000`: [2](#0-1) 

Transaction weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [3](#0-2) 

`check_tx_fee` passes raw byte size as the weight argument, with a comment explicitly acknowledging the mismatch: [4](#0-3) 

The full admission path in `_process_tx` (`tx-pool/src/process.rs`):
1. `pre_check` → calls `check_tx_fee` with `tx_size` (size-based gate only)
2. `verify_rtx` → returns actual `verified.cycles`
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed
4. `submit_entry` is called

There is no weight-based fee rate re-check between steps 2 and 4: [5](#0-4) 

`TxEntry::fee_rate()` correctly uses `get_transaction_weight(self.size, self.cycles)`, but this is only used for pool sorting and eviction, not for admission: [6](#0-5) 

**Concrete arithmetic:**

For a transaction with `tx_size = 100 bytes`, `cycles = 69,999,999`, `min_fee_rate = 1,000 shannons/KW`:
- `weight = max(100, 11,940) = 11,940`
- `min_fee` (as coded) = `1,000 × 100 / 1,000 = 100 shannons` → **passes**
- Actual fee rate = `100 / 11,940 × 1,000 ≈ 8.4 shannons/KW` — **~119× below `min_fee_rate`**

## Impact Explanation

An unprivileged attacker can flood the mempool with cycle-heavy, low-fee transactions that pass the admission check. Each such transaction consumes significant node CPU during `verify_rtx` script execution (~70M cycles per transaction), occupies pool capacity (`max_tx_pool_size`), and is relayed to peers via the relay protocol, propagating the spam network-wide. The pool eviction mechanism (`limit_size`) eventually removes low-fee-rate entries when full, but the attacker can continuously re-submit at negligible cost (~100 shannons ≈ 0.000001 CKB per transaction), maintaining sustained pool pressure and CPU load. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs (10001–15000 points)**.

## Likelihood Explanation

The attack requires only access to the `send_transaction` RPC (default-enabled, no authentication), a lock script that consumes near-maximum cycles with minimal serialized size (straightforward in CKB-VM RISC-V using a tight computation loop stored in a cell dep), and a fee of `min_fee_rate × tx_size / 1,000` shannons — as low as 100 shannons per transaction. No privileged access, majority hashpower, or social engineering is required. The attack is repeatable and economically trivial.

## Recommendation

After `verify_rtx` returns the actual cycle count, perform a weight-based fee rate re-check before calling `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Alternatively, use declared cycles (available for remote transactions) in `check_tx_fee` as an upper bound for the weight computation, deferring the precise check until after `verify_rtx`. This mirrors the correct pattern already used in `TxEntry::fee_rate()`.

## Proof of Concept

1. Construct a CKB transaction whose lock script executes a tight RISC-V loop consuming ~69,000,000 cycles. Store the script binary in a cell dep to keep the transaction's serialized size small (~200 bytes).
2. Set the transaction fee to `ceil(1000 × 200 / 1000) = 200 shannons`.
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons` — the transaction passes the size-based gate. [7](#0-6) 
5. `verify_rtx` executes the script; cycles ≈ 69,000,000 < 70,000,000 limit — the transaction passes.
6. No weight-based re-check occurs; the transaction is admitted with actual fee rate ≈ `200 / max(200, 11,769) × 1000 ≈ 17 shannons/KW`, far below the 1,000 shannons/KW floor. [8](#0-7) 
7. Repeat in a loop to fill the pool with such transactions at negligible cost (~0.000002 CKB per transaction), causing sustained CPU load and network relay spam.

    note = "Please use the get_transaction_weight instead"
)]
pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
