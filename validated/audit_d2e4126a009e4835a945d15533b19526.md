### Title
Fortuna Keeper `fulfilled_requests_cache` Never Evicts Failed Sequence Numbers, Causing Entropy Requests to Be Permanently Stuck Within a Service Run — (File: `apps/fortuna/src/keeper/block.rs`)

---

### Summary

The Fortuna keeper uses an in-memory `fulfilled_requests_cache` (`HashSet<u64>`) to deduplicate entropy request processing. When `process_event_with_backoff` exhausts its 5-minute retry window and fails, the sequence number remains in the cache permanently. All subsequent block-range rescans see the event but skip it because `newly_inserted` is `false`. The on-chain request stays active and unfulfilled indefinitely within that service run — a direct analog to the "presumptive edge tracker never confirmed" pattern.

---

### Finding Description

In `process_single_block_batch`, every observed `RequestedV2Event` is inserted into `fulfilled_requests_cache` before spawning the processing task:

```rust
let newly_inserted = process_params
    .fulfilled_requests_cache
    .write()
    .await
    .insert(event.sequence_number);
if newly_inserted {
    spawn(
        process_event_with_backoff(event.clone(), process_params.clone())
            .in_current_span(),
    );
}
``` [1](#0-0) 

The cache is an `Arc<RwLock<HashSet<u64>>>` that is **never cleared** during the service lifetime. Entries are only added, never removed. [2](#0-1) 

`process_event_with_backoff` retries for at most 5 minutes via `ExponentialBackoff`: [3](#0-2) 

The `error_mapper` only converts `NoSuchRequest` reverts into a permanent error after 5 retries. All other failure modes — `GasLimitExceeded`, `GasPriceEstimateError`, `SubmissionError`, `ConfirmationTimeout` — remain transient and are retried until the 5-minute window expires, then the function returns `Err(...)`. [4](#0-3) 

After failure, the function records `RequestEntryState::Failed` in history and returns `Ok(())`. The sequence number is **not removed** from `fulfilled_requests_cache`. [5](#0-4) 

`watch_blocks` continuously rescans the last `RETRY_PREVIOUS_BLOCKS = 100` blocks: [6](#0-5) 

Each rescan re-emits the same `RequestedV2Event`, but `insert()` returns `false` for the already-cached sequence number, so the event is silently dropped. The on-chain request remains active with `callbackStatus == CALLBACK_NOT_STARTED`, but the keeper will never attempt it again. [7](#0-6) 

---

### Impact Explanation

Any entropy request whose `process_event_with_backoff` task exhausts its retry budget (due to sustained RPC errors, gas price spikes, or a `GasLimitExceeded` condition) is permanently abandoned by the keeper within that service run. The user's `entropyCallback` is never invoked. The request remains on-chain in `CALLBACK_NOT_STARTED` state, consuming a slot in the provider's active-request array. The user receives no random number and no error — the protocol silently stalls.

The `RequestEntryState::Failed` entry in the history DB confirms the keeper gave up, but the on-chain state is inconsistent: the contract still holds the request as active. [8](#0-7) 

---

### Likelihood Explanation

Any unprivileged user who calls `requestV2` on an Entropy contract triggers this path. The failure condition (5-minute retry exhaustion) is reachable under:

- Sustained RPC degradation on the keeper's configured endpoint
- Gas price spikes that cause `GasPriceEstimateError` or `ConfirmationTimeout` for the full 5-minute window
- A `GasLimitExceeded` condition if the keeper's configured gas limit is lower than the gas required to execute `revealWithCallback` for a particular request

All of these are realistic operational conditions, not exotic attacks. The keeper is a long-running service; a restart clears the cache and allows reprocessing via `process_backlog`, but the window between failure and restart can be arbitrarily long. [9](#0-8) 

---

### Recommendation

**Short term:** When `process_event_with_backoff` returns an error (or when the on-chain request is confirmed still active after failure), remove the sequence number from `fulfilled_requests_cache` so the next block-range rescan will re-attempt it.

**Long term:** Separate the semantics of "currently being processed" (deduplication guard) from "permanently fulfilled" (skip guard). Only promote a sequence number to the permanent-skip set after a confirmed on-chain reveal (`clearRequest` has been called). Document the expected state transitions and the conditions under which each transition fires.

---

### Proof of Concept

1. User calls `entropy.requestV2{value: fee}()` on an EVM chain — sequence number `N` is emitted in a `RequestedWithCallback` event.
2. Fortuna keeper's `process_single_block_batch` sees the event, inserts `N` into `fulfilled_requests_cache` (`newly_inserted = true`), and spawns `process_event_with_backoff`.
3. The keeper's RPC endpoint experiences degradation. Every `submit_tx` attempt returns `ConfirmationTimeout` or `GasPriceEstimateError`. After 5 minutes, `submit_tx_with_backoff` returns `Err(...)`.
4. `process_event_with_backoff` records `RequestEntryState::Failed` in history and returns `Ok(())`. `N` remains in `fulfilled_requests_cache`.
5. `watch_blocks` rescans the last 100 blocks every 2 seconds. The same `RequestedWithCallback` event for `N` is returned repeatedly. Each time, `insert(N)` returns `false` → the event is skipped.
6. On-chain: `getRequestV2(provider, N)` still returns an active request with `callbackStatus == CALLBACK_NOT_STARTED`. The user's contract never receives `entropyCallback`. The request is stuck until the Fortuna service is restarted. [10](#0-9) [11](#0-10)

        if let backoff::Error::Transient {
            err: SubmitTxError::GasUsageEstimateError(tx, ContractError::Revert(revert)),
            ..
        } = &e
        {
            if let Ok(PythRandomErrorsErrors::NoSuchRequest(_)) =
                PythRandomErrorsErrors::decode(revert)
            {
                let err = SubmitTxError::GasUsageEstimateError(
                    tx.clone(),
                    ContractError::Revert(revert.clone()),
                );
                // Slow down the retries if the request is not found.
                // This probably means that the request is already fulfilled via another process.
                // After 5 retries, we return the error permanently.
                if num_retries >= 5 {
                    return backoff::Error::Permanent(err);
                }
                let retry_after_seconds = match num_retries {
                    0 => 5,
                    1 => 10,
                    _ => 60,
                };
                return backoff::Error::Transient {
                    err,
                    retry_after: Some(Duration::from_secs(retry_after_seconds)),
                };
            }
        }
        e
    };
