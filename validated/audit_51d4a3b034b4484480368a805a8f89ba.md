### Title
Missing Replay Protection in Signed Lazer Update Verification — (`File: lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.sol`'s `verifyUpdate` function performs ECDSA signature verification against a trusted signer but maintains **no on-chain state about previously consumed update payloads**. Any valid signed update can be submitted an unlimited number of times by any caller. This is the direct analog of the Harpie M-9 finding: signed data without a nonce or sequence number to prevent replay.

---

### Finding Description

The `verifyUpdate` function in `PythLazer.sol` verifies that the ECDSA signature over the payload was produced by a trusted, non-expired signer, then returns the raw payload bytes to the caller. It performs no replay protection:

- No mapping of consumed payload hashes.
- No sequence number or nonce in the signed data that the contract enforces.
- No check that the payload timestamp is within an acceptable window of `block.timestamp`.

The contract's own test suite confirms this: the identical `update` bytes are submitted twice in the same test and both calls succeed. [1](#0-0) 

The payload does contain a `uint64 timestamp` field (parsed by `PythLazerLib.parsePayloadHeader`), but `verifyUpdate` never reads or validates it. [2](#0-1) 

The test explicitly demonstrates double-submission of the same signed bytes succeeding: [3](#0-2) 

The same pattern exists on Sui (`parse_and_verify_le_ecdsa_update_v2`) and Aptos (`verify_message`), neither of which tracks consumed payloads. [4](#0-3) 

---

### Impact Explanation

An attacker who captures any previously broadcast, validly-signed Lazer update (e.g., from a public mempool, an event log, or a historical RPC call) can replay it at will against any consumer contract that calls `verifyUpdate`. If the consumer contract does not independently enforce `_timestamp > lastAcceptedTimestamp`, the attacker can feed it an arbitrarily old price. In DeFi contexts (lending, perpetuals, options) this enables:

- Borrowing against an inflated collateral price from a stale update.
- Liquidating positions using a stale price that no longer reflects market reality.
- Arbitraging any on-chain price-sensitive state that was updated with a replayed payload.

The `verifyUpdate` function is the trust anchor for the entire Lazer EVM integration. Consumers are directed to call it and trust its return value. The official integration guide shows the timestamp guard (`if (feedId == 2 && _timestamp > timestamp)`) as a consumer-side recommendation, not a contract-enforced invariant. [5](#0-4) 

---

### Likelihood Explanation

- The signed update bytes are broadcast on-chain by relayers and are trivially observable.
- No privileged access is required; any EOA can call `verifyUpdate` with a fee of 1 wei.
- The attack requires only capturing one historical update and replaying it — no key material, no Sybil attack, no miner collusion.
- Consumer contracts that omit the timestamp guard (a common integration mistake, especially given the low-level parsing API) are immediately exploitable.

---

### Recommendation

1. **Contract-level freshness enforcement**: Inside `verifyUpdate`, parse the `uint64 timestamp` from the payload header and require it to be within a configurable staleness window of `block.timestamp` (e.g., `require(block.timestamp - timestamp/1e6 <= MAX_AGE)`).

2. **Consumed-payload registry**: Maintain a `mapping(bytes32 => bool) usedPayloadHashes` and revert if `keccak256(payload)` has been seen before, analogous to Wormhole's `ConsumedVAAs` pattern already present in the same repository. [6](#0-5)

```solidity
pragma solidity ^0.8.13;

import {PythLazer} from "lazer/contracts/evm/src/PythLazer.sol";

contract ReplayAttack {
    PythLazer public pythLazer;

    constructor(address _pythLazer) {
        pythLazer = PythLazer(_pythLazer);
    }

    // `oldUpdate` is any previously broadcast, validly-signed Lazer update
    // captured from on-chain history (e.g., past tx calldata).
    function replayOldPrice(bytes calldata oldUpdate) external payable {
        uint256 fee = pythLazer.verification_fee();

        // First submission — succeeds, consumer accepts stale price T1
        (bytes calldata payload1, ) = pythLazer.verifyUpdate{value: fee}(oldUpdate);

        // Second submission in the same block — also succeeds.
        // PythLazer.sol stores no state; no revert occurs.
        (bytes calldata payload2, ) = pythLazer.verifyUpdate{value: fee}(oldUpdate);

        // payload1 == payload2: same stale timestamp, same stale price.
        // A consumer that does not check `_timestamp > lastTimestamp` will
        // accept this as a fresh update.
        assert(keccak256(payload1) == keccak256(payload2));
    }
}
```

```text
public fun parse_and_verify_le_ecdsa_update_v2(s: &State, clock: &Clock, update: vector<u8>): Update {
    let mut cursor = bcs::new(update);

    // Parse and validate message magic
    let magic = cursor.peel_u32();
    assert!(magic == UPDATE_MESSAGE_MAGIC, EInvalidUpdateMagic);

    // Parse signature
    let mut signature = vector::empty<u8>();
    let mut sig_i = 0;
    while (sig_i < SECP256K1_SIG_LEN) {
        signature.push_back(cursor.peel_u8());
        sig_i = sig_i + 1;
    };

    // Parse expected payload length and get remaining bytes as payload
    let payload_len = cursor.peel_u16();
    let payload = cursor.into_remainder_bytes();

    // Validate expected payload length
    assert!(payload_len as u64 == payload.length(), EInvalidPayloadLength);

    // Parse payload
    let mut payload_cursor = bcs::new(payload);
```
