### Title
Indexer `get_cells` / `get_transactions` Accept Pruned `block_range` Without Bounds Validation — (`File: util/indexer/src/service.rs`)

### Summary

The CKB Indexer's `get_cells` and `get_transactions` RPC methods accept a caller-supplied `block_range` filter with no validation against the indexer's internal pruned-data boundary. The indexer's `prune()` routine periodically deletes `ConsumedOutPoint`, `Header`, and `TxHash` records for blocks older than `keep_num` (hardcoded to 100). After pruning, an RPC caller who queries a `block_range` that falls entirely within the pruned window receives a silently empty result set — indistinguishable from "no data ever existed" — with no error and no indication that the requested range has been garbage-collected.

### Finding Description

`IndexerHandle::get_cells` and `IndexerHandle::get_transactions` both accept an optional `filter.block_range: [u64; 2]` parameter. [1](#0-0) 

Neither function reads the indexer's current pruned boundary before executing the query. The only input validation performed is on `limit` and `script_search_mode`.

The `prune()` function in `Indexer` deletes three categories of records for blocks older than `tip_number - keep_num`: [2](#0-1) 

`keep_num` is hardcoded to 100 in `IndexerService::get_indexer()`: [3](#0-2) 

After pruning, `ConsumedOutPoint` (dead-cell data), `Header` (block→tx list), and `TxHash` (tx→inputs) records for old blocks are permanently deleted. The `get_transactions` path iterates `TxLockScript`/`TxTypeScript` keys (which are **not** pruned), so output-side entries survive. However, the `ConsumedOutPoint` records that back the input-side cell data for those old blocks are gone. A caller who supplies `block_range: [0, 50]` on a node whose tip is at block 200 will receive an empty or incomplete result with no error, no warning, and no indication of the pruned boundary — exactly the same response as if no transactions had ever existed in that range.

The `block_range` filter check in `get_cells` is a pure post-fetch filter with no pre-check: [4](#0-3) 

The same pattern appears in `get_transactions`: [5](#0-4) 

Neither path exposes the pruned-boundary value to the caller.

### Impact Explanation

An RPC caller (wallet, block explorer, dApp) that queries `get_transactions` or `get_cells` with a `block_range` entirely within the pruned window receives `{"objects": [], "last_cursor": "0x"}` — an empty success response. The caller cannot distinguish this from "no matching cells/transactions existed in that range." A wallet reconstructing historical balance history, or a dApp auditing past cell ownership, will silently produce incorrect results (missing transactions, incorrect capacity totals) without any protocol-level signal that the data was pruned.

### Likelihood Explanation

Any unprivileged JSON-RPC caller can trigger this by supplying an old `block_range`. On a long-running mainnet node the pruned window grows continuously (every block beyond `keep_num = 100`). The condition is trivially reachable by any RPC user who queries historical ranges — a common pattern for wallets and indexer clients.

### Recommendation

1. Expose the current pruned boundary (lowest retained block number) via `get_indexer_tip` or a dedicated RPC field.
2. In `get_cells` and `get_transactions`, validate that `block_range.start` is not below the pruned boundary and return an explicit error (e.g., `"block_range start N is below the pruned boundary M"`) rather than a silent empty result.
3. Alternatively, document the pruned boundary clearly in the RPC response so callers can detect the condition.

### Proof of Concept

1. Start a CKB node with the indexer enabled.
2. Wait for the chain to advance past block 100 (so `prune()` fires and deletes records for blocks 0–(tip−101)).
3. Call:
```json
{
  "method": "get_transactions",
  "params": [{
    "script": { "code_hash": "0x...", "hash_type": "type", "args": "0x..." },
    "script_type": "lock",
    "filter": { "block_range": ["0x0", "0x5"] }
  }, "asc", "0x64", null]
}
```
4. Observe: response is `{"objects": [], "last_cursor": "0x"}` — no error, no indication that blocks 0–4 have been pruned. The same response is returned whether the address had transactions in those blocks or not. [2](#0-1) [1](#0-0) [3](#0-2)

    /// Prune useless data
    pub(crate) fn prune(&self) -> Result<(), Error> {
        let (tip_number, _tip_hash) = self.tip()?.expect("stored tip");
        let prune_number = self.keep_num + 1;
        if tip_number > prune_number {
            let prune_to_block = tip_number - prune_number;
            let mut batch = self.store.batch()?;
            // prune ConsumedOutPoint => Cell
            let key_prefix_consumed_out_point = vec![KeyPrefix::ConsumedOutPoint as u8];
            let iter = self
                .store
                .iter(&key_prefix_consumed_out_point, IteratorDirection::Forward)?
                .take_while(|(key, _value)| key.starts_with(&key_prefix_consumed_out_point));
            let mut min_block_number = BlockNumber::MAX;
            for (block_number, key) in iter
                .map(|(key, _value)| {
                    (
                        BlockNumber::from_be_bytes(
                            key[1..9].try_into().expect("stored block_number"),
                        ),
                        key,
                    )
                })
                .take_while(|(block_number, _key)| prune_to_block.gt(block_number))
            {
                if min_block_number == BlockNumber::MAX {
                    min_block_number = block_number;
                }
                batch.delete(key)?;
            }

            // prune Header => Transactions and TxHash => TransactionInputs
            let mut key_prefix_header = vec![KeyPrefix::Header as u8];
            key_prefix_header.extend_from_slice(&min_block_number.to_be_bytes());
            let iter = self
                .store
                .iter(&key_prefix_header, IteratorDirection::Forward)?
                .take_while(|(key, _value)| {
                    key.starts_with(&[KeyPrefix::Header as u8])
                        && BlockNumber::from_be_bytes(
                            key[1..9].try_into().expect("stored block_number"),
                        ) <= prune_to_block
                });
            for (txs, header_key) in iter.map(|(header_key, value)| {
                (
                    Value::parse_transactions_value(&value, header_key.len() == 42),
                    header_key,
                )
            }) {
                for (tx_hash, _outputs_len, _tx_index) in txs {
                    batch.delete(Key::TxHash(&tx_hash).into_vec())?;
                }
                batch.delete(header_key)?;
            }

            batch.commit()?;
        }
        Ok(())
    }
