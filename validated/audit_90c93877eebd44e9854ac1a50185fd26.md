### Title
Any RPC Caller Can Remove Any Pending/Proposed Transaction From the Pool Without Authorization - (File: `rpc/src/module/pool.rs`)

### Summary
The `remove_transaction` RPC in the Pool module allows any caller to remove any transaction from the tx-pool by hash, with no check that the caller is the original submitter or has any authorization over the targeted transaction. This is the direct CKB analog of STK-2: an action that should be scoped to the actor's own state can be performed against any arbitrary address's pending transaction, with compounding impact when combined with CKB's mandatory two-step proposal/commit window (the protocol-level "warmup period").

### Finding Description
`rpc/src/module/pool.rs` exposes `remove_transaction` in the `PoolRpc` trait, enabled by default under the `Pool` module:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
```

The implementation at lines 662–669 unconditionally forwards the call to the tx-pool service:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

No caller identity, no ownership check, no proof that the caller submitted the transaction. The service-layer handler at `tx-pool/src/process.rs:440–455` then removes the transaction from every sub-pool (verify queue, orphan pool, pending pool, proposed pool) without any authorization gate.

The `Pool` module is in the default-enabled module list (`resource/ckb.toml` line 190: `modules = ["Net", "Pool", "Miner", ...]`), so this RPC is live on every standard node.

### Impact Explanation
CKB enforces a mandatory two-step confirmation mechanism: a transaction must first be **proposed** in a block (entering the proposal window), then **committed** in a later block. This proposal window is the protocol-level analog of the "warmup period" cited in STK-2.

An attacker with access to the RPC endpoint can:
1. Observe any pending or proposed transaction via `get_raw_tx_pool`.
2. Call `remove_transaction(tx_hash)` to evict it from the pool — including transactions already in the **proposed** state that have already waited through the proposal window.
3. Force the victim to resubmit and restart the entire proposal window.
4. Repeat indefinitely, permanently preventing any targeted transaction from reaching the committed state.

This constitutes a targeted, persistent denial-of-service against specific transactions or addresses, with no cost to the attacker beyond RPC access.

### Likelihood Explanation
The RPC binds to `127.0.0.1:8114` by default. The bounty scope explicitly lists "supported local CLI/RPC user" and "RPC caller" as valid unprivileged attacker profiles. In any multi-tenant deployment, shared infrastructure, or node exposed beyond localhost (a common operator misconfiguration), any reachable caller can exercise this path. No key, no signature, no proof of transaction ownership is required — only the transaction hash, which is public information visible in `get_raw_tx_pool`.

### Recommendation
Add an ownership or authorization model to `remove_transaction`. At minimum, track the submitter identity (e.g., by source: local RPC vs. network relay) and reject removal requests that do not originate from the same source that submitted the transaction. Alternatively, restrict `remove_transaction` to a privileged-only sub-module (e.g., `Debug`) that is disabled by default, consistent with how other destructive operations are gated.

### Proof of Concept
```
# Step 1: User A submits a transaction
curl -X POST http://127.0.0.1:8114 -d '{"jsonrpc":"2.0","method":"send_transaction","params":[<tx_A>, "passthrough"],"id":1}'
# Returns: {"result": "0xTX_HASH_A"}

# Step 2: Wait for tx to enter proposed state (proposal window begins)
# Observe via get_raw_tx_pool that TX_HASH_A is now in "proposed"

# Step 3: Attacker (any local RPC caller) removes the proposed transaction
curl -X POST http://127.0.0.1:8114 -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xTX_HASH_A"],"id":2}'
# Returns: {"result": true}

# TX_HASH_A is now gone from the pool. User A must resubmit and restart
# the entire proposal window. Attacker repeats step 3 on each resubmission.
```

    pub(crate) fn remove_by_detached_proposal<'a>(
        &mut self,

    ) {
        for id in ids {
            if let Some(e) = self.pool_map.get_by_id(id) {
                let status = e.status;
                if status == Status::Pending {
                    continue;
                }
                let mut entries = self.pool_map.remove_entry_and_descendants(id);
                entries.sort_unstable_by_key(|entry| entry.ancestors_count);
                for mut entry in entries {
                    let tx_hash = entry.transaction().hash();
                    entry.reset_statistic_state();
                    let ret = self.add_pending(entry);
                    debug!(
                        "remove_by_detached_proposal from {:?} {} add_pending {:?}",
                        status, tx_hash, ret
                    );
                }
            }
        }
    }
