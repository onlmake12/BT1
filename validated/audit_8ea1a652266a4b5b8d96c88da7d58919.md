### Title
Peer store permanently fillable via crafted Nodes messages causing silent `EvictionFailed` — eclipse attack — (`network/src/peer_store/peer_store_impl.rs`, `network/src/protocols/discovery/mod.rs`)

### Summary
An unprivileged remote peer can permanently fill the CKB peer store to `ADDR_COUNT_LIMIT` (16384) by sending `Nodes` messages containing addresses spread across ≥4096 distinct /24 network groups (≤4 addresses each). Once full, `check_purge()` returns `Err(EvictionFailed)` because (a) all injected addresses are "connectable" (fresh, never tried), and (b) no network group exceeds the eviction threshold of 4. This error is silently swallowed in `add_new_addrs()`, permanently preventing the victim node from learning about honest peers.

### Finding Description
The vulnerability is a combination of three independent weaknesses:

1. **Silent error discard in `add_new_addrs`** (`mod.rs:355-360`): `peer_store.add_addr()` returning `Err` is only logged at `debug!` and discarded. The loop continues but every subsequent call also fails.

2. **`check_purge` first-pass bypass** (`peer_store_impl.rs:341-355`): Addresses inserted via `add_addr` are created with `last_connected_at_ms=0` and `attempts_count=0`. `is_connectable()` returns `true` for these (the `attempts_count >= ADDR_MAX_RETRIES` guard requires ≥3 failed attempts), so none are evicted in the first pass.

3. **`check_purge` second-pass bypass** (`peer_store_impl.rs:378`): The network-group eviction only removes peers from groups where `addrs.len() > 4` (strictly greater than 4). An attacker who places exactly 4 addresses per /24 group across 4096 groups satisfies `ADDR_COUNT_LIMIT` while keeping every group at the threshold, triggering `Err(PeerStoreError::EvictionFailed)`.

### Impact Explanation
Once the peer store is full with attacker-controlled addresses, all calls to `add_addr` for legitimate peer addresses return `Err(EvictionFailed)` and are silently dropped. The victim node's peer store is permanently poisoned: it cannot learn about honest peers via discovery, cannot dial honest peers (since `fetch_addrs_to_feeler` / `fetch_addrs_to_attempt` only return attacker addresses), and is effectively eclipsed from the honest network. This enables consensus deviation, transaction censorship, and double-spend facilitation against the victim.

### Likelihood Explanation
The attacker needs only ~6 concurrent P2P sessions (well within normal peer limits) and the ability to advertise ~16384 globally-routable IP addresses from ≥4096 distinct /24 subnets. Since these are gossip addresses (not actual connections), the attacker does not need to control those IPs — they only need to be globally routable to pass `is_reachable()`. IP address space in diverse /24 subnets is trivially available. The attack is persistent across node restarts if the peer store is saved to disk.

### Recommendation
1. **Do not silently discard `EvictionFailed`**: In `add_new_addrs`, treat `Err(EvictionFailed)` as a signal to stop processing the batch, or at minimum log at `warn!` level and trigger a more aggressive purge.
2. **Fix the eviction threshold**: Change `if addrs.len() > 4` to `if addrs.len() >= 4` (or lower), so groups at exactly the threshold are also eligible for eviction.
3. **Evict unverified addresses preferentially**: In `check_purge`, prioritize eviction of addresses with `last_connected_at_ms == 0` (never successfully connected) before attempting network-group eviction.
4. **Rate-limit address injection per session**: Cap the number of new addresses accepted per session per time window to limit how quickly the store can be filled.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
