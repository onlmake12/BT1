### Title
Indexer Storage Key Missing Length Delimiter on Variable-Length `args` Causes Prefix Search False Positives — (`util/indexer/src/indexer.rs`)

### Summary

The `extract_raw_data` function concatenates `code_hash || hash_type || args` without encoding the length of the variable-length `args` field. When this raw concatenation is used as a database iteration prefix, a query for script `(H, T, args=[A, B])` will also match stored index entries for the distinct script `(H, T, args=[A])` at block numbers whose leading byte equals `B`. This is a direct analog to the Rocket Pool missing-delimiter bug: two logically different keys produce the same byte prefix, causing namespace collision in the indexer's key space.

---

### Finding Description

`extract_raw_data` in `util/indexer/src/indexer.rs` produces:

```
code_hash (32 bytes, fixed) || hash_type (1 byte, fixed) || args (variable, no length prefix)
``` [1](#0-0) 

This blob is appended directly after the one-byte `KeyPrefix` to form storage keys. For `CellLockScript`, the full key layout is:

```
[KeyPrefix (1)] [code_hash (32)] [hash_type (1)] [args (variable)] [block_number (8)] [tx_index (4)] [output_index (4)]
``` [2](#0-1) 

Because `args` has no length tag, the byte boundary between `args` and `block_number` is invisible in the serialized key. When `build_query_options` constructs a search prefix it simply appends `extract_raw_data(script)` to the `KeyPrefix` byte:

```
prefix = [KeyPrefix] || code_hash || hash_type || search_args
``` [3](#0-2) 

The iterator then accepts every stored key that `starts_with(&prefix)`: [4](#0-3) 

**Collision scenario.** Consider two distinct scripts sharing the same `code_hash` H and `hash_type` T:

| Script | `args` | Block number |
|--------|--------|--------------|
| S1 | `[0xAB]` (1 byte) | `0xCD_00_00_00_00_00_00_00` |
| S2 | `[0xAB, 0xCD]` (2 bytes) | any |

Stored key for S1's cell:
```
[0x40] [H] [T] [0xAB] [0xCD 0x00 0x00 0x00 0x00 0x00 0x00 0x00] [tx_idx] [out_idx]
```

Search prefix for S2 (`args = [0xAB, 0xCD]`):
```
[0x40] [H] [T] [0xAB] [0xCD]
```

The stored key for S1 **starts with** the search prefix for S2. The iterator therefore returns S1's cell as a result of a query for S2, even though S1's `args = [0xAB]` does not start with `[0xAB, 0xCD]`.

The same flaw applies to `CellTypeScript`, `TxLockScript`, and `TxTypeScript` key families, all of which use `append_key` / `extract_raw_data` with the same layout. [5](#0-4) 

The `script_search_exact` length check (`key.len() != prefix.len() + 16`) does filter out this false positive in exact mode, but the **default prefix mode** — used by the vast majority of callers — does not. [6](#0-5) 

---

### Impact Explanation

Any application or wallet that calls `get_cells`, `get_transactions`, or `get_cells_capacity` via the CKB Indexer RPC in the default prefix-search mode may receive spurious results. Concretely:

- **Incorrect balance / UTXO set**: A wallet querying for cells locked by script `(H, T, args=[A, B])` silently includes cells locked by the unrelated script `(H, T, args=[A])` that happen to sit at block numbers whose first byte is `B`. The wallet over-counts spendable capacity.
- **Incorrect transaction history**: `get_transactions` returns transactions that do not involve the queried script.
- **Application-level confusion**: Any dApp that uses the indexer to gate access or verify ownership based on script args can be fooled by an on-chain actor who deploys cells with a shorter-args script at a strategically chosen block height.

The impact is data-integrity corruption of indexer query results, reachable without any privileged access.

---

### Likelihood Explanation

- The attacker-controlled entry path is the public `get_cells` / `get_transactions` RPC endpoint, reachable by any unprivileged peer or RPC caller.
- An on-chain actor who deploys cells with script `(H, T, args=[A])` does not control block numbers, but block numbers grow monotonically and cover all 8-byte values over time; for any target byte `B`, block number `B * 2^56` will eventually be reached on a live chain.
- Alternatively, the false positive is triggered passively: any existing cell with a shorter-args script whose block number happens to share a leading byte with the query args will silently pollute results.
- No special privileges, hash-power, or Sybil capability is required.

---

### Recommendation

Encode the length of `args` before appending it in `extract_raw_data` (or equivalently in `append_key`). A fixed-width little-endian or big-endian 4-byte length prefix is sufficient:

```rust
pub fn extract_raw_data(script: &Script) -> Vec<u8> {
    let args = script.args().raw_data();
    let args_len = (args.len() as u32).to_le_bytes();
    [
        script.code_hash().as_slice(),
        script.hash_type().as_slice(),
        &args_len,   // 4-byte length tag
        &args,
    ]
    .concat()
}
```

This is a breaking change to the stored key format and requires a database migration, but it eliminates the ambiguity entirely. Alternatively, the `script_search_exact` length guard should be applied in prefix mode as well, rejecting any key whose length is inconsistent with the expected `args` length derived from the prefix.

### Proof of Concept

1. Deploy a cell on CKB with lock script `(code_hash=H, hash_type=T, args=0xAB)` (1-byte args). Suppose it is confirmed at block number `0xCD00000000000000`.

2. Call the CKB Indexer RPC:
   ```json
   {
     "method": "get_cells",
     "params": [{
       "script": { "code_hash": "H", "hash_type": "T", "args": "0xabcd" },
       "script_type": "lock"
     }, "asc", "0x64", null]
   }
   ```
   (`args = 0xabcd` = 2 bytes `[0xAB, 0xCD]`)

3. **Expected**: no results (no cell has lock script with args starting with `[0xAB, 0xCD]`).

4. **Actual**: the cell from step 1 is returned, because its stored key:
   ```
   [0x40][H][T][0xAB][0xCD 00 00 00 00 00 00 00][tx_idx][out_idx]
   ```
   starts with the search prefix `[0x40][H][T][0xAB][0xCD]`, satisfying `key.starts_with(&prefix)`. [1](#0-0) [7](#0-6) [2](#0-1)
