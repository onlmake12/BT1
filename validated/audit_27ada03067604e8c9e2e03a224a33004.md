### Title
Governance Sequence Gap Allows Unprivileged Relayer to Permanently Suppress Any Governance Action — (`target_chains/sui/contracts/sources/governance/governance.move`)

### Summary
The governance sequence guard uses `sequence > last_executed_governance_sequence` instead of `sequence == last_executed_governance_sequence + 1`. An unprivileged relayer who holds two valid guardian-signed governance VAAs can submit the higher-sequence one first, permanently making the lower-sequence one unexecutable.

### Finding Description
In `execute_governance_instruction`, the only ordering check is:

```move
assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
``` [1](#0-0) 

After passing, the state is immediately updated:

```move
state::set_last_executed_governance_sequence(&latest_only, pyth_state, sequence);
``` [2](#0-1) 

The identical flaw exists in the contract upgrade path:

```move
assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
state::set_last_executed_governance_sequence_unchecked(pyth_state, sequence);
``` [3](#0-2) 

There is no `consumed_vaas` call anywhere in the governance execution path — the sequence number is the sole replay and ordering guard. [4](#0-3) 

`verify_vaa` only checks that the VAA originates from the registered governance emitter; it imposes no ordering constraint. [5](#0-4) 

`last_executed_governance_sequence` is initialized to `0` at deployment. [6](#0-5) 

**Attack trace (concrete):**

| Step | Action | State after |
|------|--------|-------------|
| 1 | Governance approves action A (seq=N+1, e.g. critical upgrade) and action B (seq=N+2) | `last_executed = N` |
| 2 | Attacker relays VAA seq=N+2 → `N+2 > N` passes | `last_executed = N+2` |
| 3 | Anyone tries to relay VAA seq=N+1 → `N+1 > N+2` **fails**, aborts with `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER` | Permanent |

No privileged access is required. `verify_vaa` and `execute_governance_instruction` are both `public` functions callable by any transaction sender. [7](#0-6) [8](#0-7) 

### Impact Explanation
A voted governance action — including a `contract_upgrade` VAA that patches a live vulnerability — can be permanently suppressed by any unprivileged relayer. The voted outcome is never executed, directly matching the stated scope: *"governance voting result manipulation that changes execution away from the voted outcome."*

### Likelihood Explanation
The window requires two guardian-signed governance VAAs to exist simultaneously (i.e., governance approved two actions before either is relayed on Sui). This is a realistic operational condition: governance proposals are batched or approved in quick succession regularly. Any observer of the Wormhole guardian network can detect pending VAAs and race to submit the higher-sequence one first.

### Recommendation
Replace the `>` guard with a strict sequential check in both `governance.move` and `contract_upgrade.move`:

```move
assert!(
    sequence == state::get_last_executed_governance_sequence(pyth_state) + 1,
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER
);
```

This ensures every approved governance action must be executed in the exact order it was approved, with no gaps skippable by a relayer.

### Proof of Concept
Unit test sketch (Move pseudocode):

```move
// Setup: last_executed_governance_sequence = 4
let vaa_seq5 = make_governance_vaa(sequence: 5, action: set_fee);
let vaa_seq4 = make_governance_vaa(sequence: 4, action: contract_upgrade_patch);

// Attacker submits seq=5 first
let receipt5 = governance::verify_vaa(&state, vaa_seq5);
governance::execute_governance_instruction(&mut state, receipt5);
// last_executed_governance_sequence is now 5

// Victim attempts seq=4 (the critical patch)
let receipt4 = governance::verify_vaa(&state, vaa_seq4);
governance::execute_governance_instruction(&mut state, receipt4);
// ABORTS: 4 > 5 is false → E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER
// The contract upgrade patch is permanently suppressed.
```
