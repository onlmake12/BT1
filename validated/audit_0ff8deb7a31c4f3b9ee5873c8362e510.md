### Title
Unrestricted `public` `deposit_fee_coins` with no withdrawal path permanently locks SUI in shared `PriceInfoObject` — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
`price_info::deposit_fee_coins` is declared `public` instead of `public(friend)`, and `PriceInfoObject` is a shared Sui object. Any unprivileged address can call this function directly in a PTB, depositing arbitrary `Coin<SUI>` into any `PriceInfoObject`. Because no withdrawal function exists anywhere in the module or protocol, all deposited SUI — including legitimate protocol update fees — is permanently locked with no recovery path.

### Finding Description
In `price_info.move` line 105, `deposit_fee_coins` is `public`:

```move
public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
```

`PriceInfoObject` has `key, store` abilities and is shared via `transfer::public_share_object` in `pyth.move` line 186. On Sui, shared objects can be passed as `&mut` by any transaction without ownership. The function joins the caller-supplied coin into the `FEE_STORAGE_KEY` dynamic object field unconditionally.

There is no `withdraw_fee_coins`, no admin extraction function, and no governance path to recover these coins anywhere in the contract sources. The `#[test_only]` `destroy` function at line 170 would abort at runtime if the dynamic field exists, since it calls `object::delete` without first removing the child object.

This means:
- Any user can call `deposit_fee_coins` directly (bypassing `update_single_price_feed`) and permanently lock their own SUI.
- All legitimately collected protocol fees deposited via `update_single_price_feed` → `price_info::deposit_fee_coins` are also permanently locked — the protocol has no mechanism to ever recover them.

### Impact Explanation
Permanent, irrecoverable locking of SUI inside shared `PriceInfoObject` objects. An attacker can lock their own funds (or any funds they control) in any `PriceInfoObject`. More critically, the protocol's own fee revenue is also permanently locked — there is no admin or governance function to withdraw accumulated fees. This matches the scope: "permanent locking of user funds or funds staked on Pyth." [5](#0-4) 

### Likelihood Explanation
The attack path requires no privilege, no leaked key, and no governance majority. Any Sui address can construct a PTB calling `price_info::deposit_fee_coins` with a shared `PriceInfoObject` object ID (all are publicly discoverable on-chain) and any `Coin<SUI>`. The function is callable today on any deployed version of this contract.

### Recommendation
1. Change `deposit_fee_coins` from `public` to `public(friend)` so only `pyth::pyth` (the declared friend) can call it.
2. Implement a privileged `withdraw_fee_coins` function (gated by a governance/admin capability) to allow recovery of accumulated fees from `PriceInfoObject`.
3. Consider whether fees should be stored in the central `PythState` object rather than distributed across individual `PriceInfoObject` instances, which would simplify fee accounting and recovery.
