### Title
Unbounded O(N) DB Amplification via `GetTransactionsProof` with No Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

### Summary

An unprivileged remote light-client peer can send a single `GetTransactionsProof` message containing up to `GET_TRANSACTIONS_PROOF_LIMIT` (1000) distinct tx_hashes, each from a different block. The server performs O(N) full block DB reads, CBMT proof computations, uncle reads, and extension reads — then generates an MMR proof over up to 1000 positions — all with no per-peer rate limiting and no ban on valid requests. The attacker can repeat this indefinitely.

### Finding Description

`GET_TRANSACTIONS_PROOF_LIMIT` is set to 1000. [1](#0-0) 

The `execute()` function rejects requests with `> 1000` tx_hashes (returning `MalformedProtocolMessage`, a 4xx code that triggers a 5-minute ban). However, a request with **exactly 1000** tx_hashes is fully processed. [2](#0-1) 

For each unique block containing a found tx, the server unconditionally calls:
- `snapshot.get_block(&block_hash)` — full block deserialization from DB
- `CBMT::build_merkle_proof(...)` — CPU-bound proof construction over all block transactions
- `snapshot.get_block_uncles(&block_hash)` — additional DB read
- `snapshot.get_block_extension(&block_hash)` — additional DB read [3](#0-2) 

After the loop, `reply_proof` is called with up to 1000 MMR positions, triggering `mmr.gen_proof(items_positions)` — an O(N log N) MMR traversal. [4](#0-3) 

A valid request returns `Status::ok()` (200). The ban logic only fires for 4xx codes. [5](#0-4) [6](#0-5) 

No rate limiting, per-peer request quota, or cooldown exists anywhere in the light-client-protocol-server. [7](#0-6) 

### Impact Explanation

Each maximum-size request causes ≥4000 DB operations (4 per block × 1000 blocks) plus CPU-intensive CBMT and MMR proof generation. The attacker can pipeline these requests continuously from a single connection (or multiple connections) with no back-pressure. This saturates disk I/O and CPU on the serving node, degrading or crashing it for all light clients.

### Likelihood Explanation

All required inputs are publicly observable on mainnet: tx_hashes are in block data, and `last_hash` is the chain tip. No key material, privileged access, or hashpower is needed. Any peer that can open a light-client protocol connection can execute this attack.

### Recommendation

1. **Add per-peer rate limiting** on `GetTransactionsProof` (and `GetBlocksProof`) — e.g., a token-bucket or sliding-window counter that bans peers exceeding a threshold of requests per second.
2. **Reduce the effective work per request** by capping the number of *distinct blocks* (not just tx_hashes) that will be processed in a single response, since the amplification factor is driven by block diversity, not raw tx count.
3. **Consider banning peers** that repeatedly send maximum-size requests within a short window, even if each individual request is technically valid.

### Proof of Concept

```
1. Sync a CKB node with >= 1000 blocks, each containing at least one transaction.
2. Collect 1000 tx_hashes from 1000 distinct blocks (publicly available via RPC/chain scan).
3. Obtain the current chain tip hash as last_hash.
4. Craft a GetTransactionsProof message with all 1000 tx_hashes and the valid last_hash.
5. Connect as a light-client peer and send the message in a loop.
6. Observe: each iteration triggers 4000+ DB reads + CBMT/MMR computation on the server,
   with no ban or rate limit applied, causing sustained CPU/IO exhaustion.
```

    async fn received(
        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
}
