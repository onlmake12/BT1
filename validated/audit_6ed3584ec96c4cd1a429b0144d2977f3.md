### Title
Delegators Cannot Unstake PYTH Tokens From a Removed Publisher — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

---

### Summary

The Oracle Integrity Staking (OIS) integrity pool program gates both `advance_delegation_record` and `undelegate` on the publisher being present in `PoolData`. If a publisher is removed from `PoolData` (e.g., via the `advance` instruction after their publisher caps are revoked), delegators who previously staked to that publisher can no longer advance their delegation record or undelegate their positions. Their PYTH tokens become permanently locked in the staking contract.

---

### Finding Description

The `PoolData` account stores a fixed array of up to 1024 publisher pubkeys. Both the `advance_delegation_record` and `undelegate` instructions require the `publisher` account to be validated against `pool_data` — the IDL comment on both instructions reads: `"CHECK : The publisher will be checked against data in the pool_data"`. The integrity pool program exposes a dedicated `PublisherNotFound` error (code 6000) that is returned when this check fails. [1](#0-0) 

The `advance` instruction updates `PoolData` based on the latest verified publisher caps. When a publisher is removed from the publisher caps feed (their pubkey slot is zeroed to `PublicKey::default`), `extractPublisherData` filters them out: [2](#0-1) 

After removal, two blocking conditions arise for any delegator who had staked to that publisher:

1. **`advance_delegation_record` fails**: The instruction requires the publisher to exist in `pool_data`. Once removed, this call reverts with `PublisherNotFound`, making it impossible to bring the delegator's accounting up to date.

2. **`undelegate` fails**: The instruction also checks the publisher against `pool_data`. It reverts with `PublisherNotFound` (6000) or `OutdatedDelegatorAccounting` (6008) if the delegation record was never advanced. [3](#0-2) [4](#0-3) 

The SDK's `getAdvanceDelegationRecordInstructions` only iterates over publishers currently present in `pool_data` (via `extractPublisherData`), so it silently skips removed publishers — meaning the delegator's accounting is never advanced and the stale positions can never be closed: [5](#0-4) 

The `undelegate` instruction accounts confirm the publisher check dependency: [6](#0-5) 

---

### Impact Explanation

A delegator who staked PYTH tokens to a publisher that is subsequently removed from the pool cannot:
- Call `advance_delegation_record` for that publisher (publisher not found in pool_data)
- Call `undelegate` to close their position (same check fails)
- Withdraw their tokens from the staking custody (tokens remain locked)

The staked PYTH tokens are permanently frozen in the staking contract with no recovery path available to the user. This is a direct loss of user funds.

---

### Likelihood Explanation

Publisher removal is a realistic governance action — publishers can be removed from the Pyth oracle network for poor performance, misconduct, or voluntary exit. The `advance` instruction is designed to update pool state based on publisher caps, and a publisher with zero or revoked caps would naturally be removed. Any delegator who had not yet undelegated before the removal occurs would be affected. Given that the staking warmup/cooldown cycle spans multiple epochs, there is a realistic window where delegators are still staked to a publisher at the time of removal.

---

### Recommendation

1. Allow `undelegate` to proceed even when the publisher is no longer present in `pool_data`, treating the position as if the publisher had zero delegation power (no rewards, immediate cooldown entry).
2. Alternatively, add a dedicated `force_undelegate` instruction that bypasses the publisher-in-pool-data check, callable only when the publisher is absent from `pool_data`.
3. Ensure `advance_delegation_record` can be called (or is automatically finalized) for a publisher at the moment they are removed from `pool_data`, so delegator accounting is never left in a permanently stale state.

---

### Proof of Concept

1. Alice stakes 1000 PYTH to Publisher P via `delegate`. Her `stakeAccountPositions` now contains a position with `targetWithParameters.integrityPool.publisher = P`.
2. Publisher P is removed from the publisher caps feed. The `advance` instruction is called, zeroing P's slot in `PoolData.publishers`.
3. Alice calls `undelegate` → transaction reverts with `PublisherNotFound` (6000).
4. Alice calls `advance_delegation_record` for P → transaction reverts with `PublisherNotFound` (6000).
5. Alice calls `unstakeFromAllPublishers` via the SDK → the SDK's `getAdvanceDelegationRecordInstructions` skips P (not in `extractPublisherData` output), so no advance instruction is generated; the subsequent `undelegate` call for P still reverts with `OutdatedDelegatorAccounting` (6008) or `PublisherNotFound` (6000).
6. Alice's 1000 PYTH remain locked with no recovery path. [7](#0-6) [8](#0-7)

  public async unstakeFromAllPublishers(
    stakeAccountPositions: PublicKey,
    positionStates: (PositionState.LOCKED | PositionState.LOCKING)[],
  ) {
    const [stakeAccountPositionsData, currentEpoch] = await Promise.all([
      this.getStakeAccountPositions(stakeAccountPositions),
      getCurrentEpoch(this.connection),
    ]);

    const instructions = await Promise.all(
      stakeAccountPositionsData.data.positions
        .map((position, index) => {
          const publisher =
            position.targetWithParameters.integrityPool?.publisher;
          return publisher === undefined
            ? undefined
            : { index, position, publisher };
        })
        // By separating this filter from the next, typescript can narrow the
        // type and automatically infer that there will be no `undefined` values
        // in the array after this line.  If we combine those filters,
        // typescript won't narrow properly.
        .filter((positionInfo) => positionInfo !== undefined)
        .filter(({ position }) =>
          (positionStates as PositionState[]).includes(
            getPositionState(position, currentEpoch),
          ),
        )
        .reverse()
        .map(({ position, index, publisher }) =>
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({ publisher, stakeAccountPositions })
            .instruction(),
        ),
    );

    return sendTransaction(instructions, this.connection, this.wallet);
  }
