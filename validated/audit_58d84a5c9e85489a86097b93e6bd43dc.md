Audit Report

## Title
Wrong Column Traversal in `migrate_transaction_info` Causes Post-Migration Node Crash - (File: util/migrate/src/migrations/table_to_struct.rs)

## Summary
The `migrate_transaction_info` function in `ChangeMoleculeTableToStruct` traverses `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO` at line 93. This leaves all pre-migration `TransactionInfo` records in the old 52-byte table format. After migration, any call to `get_transaction_info` on a pre-migration transaction hash invokes `from_slice_should_be_ok` on a 52-byte blob that fails molecule struct verification, triggering an unconditional `panic!` that crashes the node process.

## Finding Description
**Root cause:** In `util/migrate/src/migrations/table_to_struct.rs` line 93, the `db.traverse` call reads from `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO`:

```rust
// line 93 — BUG
db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
//          ^^^^^^^^^^^^^ should be COLUMN_TRANSACTION_INFO
``` [1](#0-0) 

**Two concrete consequences:**

1. **Real `TransactionInfo` records are never migrated.** The old 52-byte table-format entries remain in `COLUMN_TRANSACTION_INFO` untouched. The migration closure's guard `if value.len() != TRANSACTION_INFO_SIZE` (52) is `false` for these entries, so they are never rewritten to the new 36-byte struct format.

2. **Uncle data is injected into `COLUMN_TRANSACTION_INFO`.** Uncle header entries are 240 bytes. Since `240 != 52`, the closure fires for every uncle, writing `uncle_data[16..]` (224 bytes) into `COLUMN_TRANSACTION_INFO` keyed by uncle hashes. [2](#0-1) 

**Why existing checks fail — the panic path:**

`get_transaction_info` calls `packed::TransactionInfoReader::from_slice_should_be_ok` on whatever bytes are stored. `from_slice_should_be_ok` is implemented as:

```rust
fn from_slice_should_be_ok(slice: &'r [u8]) -> Self {
    match Self::from_slice(slice) {
        Ok(ret) => ret,
        Err(err) => panic!(
            "failed to convert from slice: reason: {}; data: 0x{}.",
            err, hex_string(slice)
        ),
    }
}
``` [3](#0-2) 

- A 52-byte old-format blob fed to a 36-byte struct reader → `from_slice` returns `Err` → **unconditional panic → node crash**.
- A 224-byte uncle payload fed to a 36-byte struct reader → same panic path. [4](#0-3) 

The panic propagates through `get_transaction_with_info` → `get_transaction` → any RPC or internal path that looks up a pre-migration transaction. [5](#0-4) 

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

After `ckb migrate` completes, the node starts with a permanently corrupted `COLUMN_TRANSACTION_INFO`. The first call to `get_transaction_info` for any pre-migration transaction hash (e.g., via `get_transaction`, `transaction_exists` returning a result that is then dereferenced, or the `calculate_dao_maximum_withdraw` RPC) triggers the `panic!` in `from_slice_should_be_ok`, crashing the node process. The corruption is permanent — re-running the migration does not help because the version stamp is already updated.

## Likelihood Explanation
Any node operator upgrading from a pre-v0.35.0 database to v0.35.0+ and running `ckb migrate` or `ckb migrate --force` triggers this path. `ChangeMoleculeTableToStruct` is registered unconditionally in the migration sequence: [6](#0-5) 

No attacker interaction is required. The operator follows the standard documented upgrade procedure and their node becomes permanently crash-prone on any transaction lookup touching pre-migration data.

## Recommendation
In `migrate_transaction_info`, replace `COLUMN_UNCLES` with `COLUMN_TRANSACTION_INFO` in the `db.traverse` call:

```rust
let (_count, nk) =
    db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```

Additionally, add per-function unit tests asserting that each `migrate_*` helper reads from and writes to the same column, and add a post-migration sanity check verifying that `COLUMN_TRANSACTION_INFO` entries are all exactly 36 bytes.

## Proof of Concept
The bug is statically visible and the crash path is deterministic:

1. Start a CKB node on a pre-v0.35.0 database that has at least one committed transaction and at least one uncle block.
2. Run `ckb migrate`.
3. Start the upgraded node.
4. Call any RPC that invokes `get_transaction_info` on a pre-migration transaction hash (e.g., `get_transaction` via JSON-RPC, or `calculate_dao_maximum_withdraw`).
5. The node panics: `"failed to convert from slice: reason: <molecule error>; data: 0x<52-byte hex>"` and crashes.

Static proof (no runtime needed):
- `migrate_transaction_info` traverses `COLUMN_UNCLES` (line 93), not `COLUMN_TRANSACTION_INFO` — confirmed in source.
- `from_slice_should_be_ok` unconditionally panics on verification failure — confirmed in `util/gen-types/src/prelude.rs` lines 47–52.
- `get_transaction_info` calls `from_slice_should_be_ok` on raw `COLUMN_TRANSACTION_INFO` bytes — confirmed in `store/src/store.rs` lines 308–312.
