### Title
Import Process Infinite Busy-Wait on Rejected Last Block — (`util/instrument/src/import.rs`)

### Summary
`read_from_json` uses the highest block number seen in the JSONL input as its termination sentinel. If that block fails `non_contextual_verify` it is never written to the canonical chain index, so `get_block_hash(largest_block_number)` returns `None` forever and the unbounded `while … is_none() { sleep(1s) }` loop never exits.

### Finding Description

**Step 1 — sentinel is set unconditionally.**

In `read_from_json`, every non-genesis block received from the producer thread updates the sentinel before the block is dispatched for verification: [1](#0-0) 

`largest_block_number` is set to `max(largest_block_number, block.number())` regardless of whether the block will pass verification.

**Step 2 — invalid block is silently dropped, never stored.**

`asynchronous_process_lonely_block` → `ChainService::asynchronous_process_block` runs `non_contextual_verify`. On failure it marks the block `BLOCK_INVALID`, fires the error callback, and **returns early** — `insert_block` is never called: [2](#0-1) [3](#0-2) 

**Step 3 — `get_block_hash` queries the canonical chain index only.**

`snapshot.get_block_hash(number)` reads `COLUMN_INDEX`, which is only written when a block is committed to the main chain: [4](#0-3) 

A block that never passed `insert_block` is never in `COLUMN_INDEX`.

**Step 4 — the termination loop has no timeout and no error exit.** [5](#0-4) 

There is no timeout, no channel to signal rejection, and no break condition other than `get_block_hash` returning `Some`. Because the rejected block is never committed, the condition is permanently `None` and the process hangs indefinitely.

### Impact Explanation
The `ckb import` process never returns. The operator's node cannot complete chain bootstrap or re-import. The only recovery is a `SIGKILL`. Any JSONL file whose last line contains a block with an invalid PoW nonce (or any other non-contextual failure) triggers this hang deterministically.

### Likelihood Explanation
The attack surface is the `ckb import <file>` CLI path, which is a documented, production-supported bootstrap mechanism. An attacker who can influence the JSONL file the operator imports — via a malicious export server, a tampered download, or a man-in-the-middle on an unverified source — can trigger the hang with a single crafted line appended to an otherwise valid export. No privileged access, key material, or majority hashpower is required.

### Recommendation
Replace the unconditional sentinel update with a mechanism that tracks only blocks that were successfully committed, or add a bounded retry with a hard timeout and an error return. The simplest correct fix is to record the `largest_block_number` only after the verify callback confirms `Ok(true)`, or to add a deadline to the wait loop that returns `Err` if the sentinel block never appears in the canonical index within a reasonable window.

### Proof of Concept
1. Export N valid blocks from a running devnet node to `valid.jsonl`.
2. Append one line: a JSON block with `block_number = N+1`, valid parent hash, but `nonce = 0x0` (guaranteed PoW failure on mainnet/devnet difficulty).
3. Run `ckb import valid.jsonl` (no skip flags).
4. Observe: after all N+1 blocks are dispatched, the process prints `"Error verifying block: …"` for the last block and then stalls indefinitely — `get_block_hash(N+1)` always returns `null`.
5. After 60 seconds the process has not exited; `kill -9` is required to terminate it.
