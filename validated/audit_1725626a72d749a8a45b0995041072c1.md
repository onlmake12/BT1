### Title
`getPricesNoOlderThan` / `getEmaPricesNoOlderThan` Always Revert When Whitelist Is Enabled Due to `address(this)` as `msg.sender` in Internal `this.` Calls — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` in `Scheduler.sol` delegate to `this.getPricesUnsafe()` and `this.getEmaPricesUnsafe()` via external `this.` calls. Because Solidity external calls reset `msg.sender` to `address(this)`, the `onlyWhitelistedReader` modifier re-executes with the Scheduler contract as the caller. When `whitelistEnabled` is `true`, the contract itself is neither the subscription manager nor a whitelisted address, so the inner call always reverts — making both `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` permanently broken for every whitelist-enabled subscription.

---

### Finding Description

`getPricesNoOlderThan` (line 554) and `getEmaPricesNoOlderThan` (line 597) both delegate to their `Unsafe` counterparts via `this.`:

```solidity
// Scheduler.sol line 554
prices = this.getPricesUnsafe(subscriptionId, priceIds);

// Scheduler.sol line 597
prices = this.getEmaPricesUnsafe(subscriptionId, priceIds);
```

`this.f()` is an external call; Solidity sets `msg.sender = address(this)` for the callee. Both `getPricesUnsafe` and `getEmaPricesUnsafe` carry the `onlyWhitelistedReader(subscriptionId)` modifier:

```solidity
modifier onlyWhitelistedReader(uint256 subscriptionId) {
    // Manager is always allowed
    if (_state.subscriptionManager[subscriptionId] == msg.sender) { _; return; }

    // If whitelist is not used, allow any reader
    if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) { _; return; }

    // Check if caller is in whitelist
    ...
    if (!isWhitelisted) {
        revert SchedulerErrors.Unauthorized();   // ← always hit when whitelist is on
    }
    _;
}
```

When `whitelistEnabled == true`:

1. `address(this)` is not the subscription manager (managers are EOAs/external contracts).
2. `address(this)` is not in `readerWhitelist` (no code path ever adds the Scheduler itself).
3. The modifier reverts with `Unauthorized`.

The outer function (`getPricesNoOlderThan`) already passed the same modifier for the real caller, but that check is irrelevant — the inner `this.` call re-runs the modifier with a different `msg.sender`.

---

### Impact Explanation

Every subscription that sets `whitelistEnabled = true` loses access to `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` entirely. These are the only functions that enforce a freshness guarantee (`age_seconds`). Protocols that depend on freshness-bounded price reads (e.g., liquidation engines, lending protocols) cannot use the Scheduler's whitelist feature and freshness guarantee simultaneously. The DoS is permanent and unconditional for any whitelist-enabled subscription — no workaround exists short of disabling the whitelist (which removes the access-control guarantee).

---

### Likelihood Explanation

Any subscription manager who enables the whitelist to restrict price-feed access triggers the bug. Enabling the whitelist is a documented, first-class feature of the Scheduler. The bug is deterministic: every call to `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` on a whitelist-enabled subscription reverts, 100% of the time.

---

### Recommendation

Replace the external `this.` delegation with a direct call to the shared internal helper `_getPricesInternal`, which already exists and carries no access-control modifier:

```solidity
// Before (broken when whitelistEnabled == true)
prices = this.getPricesUnsafe(subscriptionId, priceIds);

// After (correct)
PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
prices = new PythStructs.Price[](priceFeeds.length);
for (uint i = 0; i < priceFeeds.length; i++) {
    prices[i] = priceFeeds[i].price;
}
```

Apply the same fix to `getEmaPricesNoOlderThan`. The access-control check is already enforced by the `onlyWhitelistedReader` modifier on the outer function; there is no need to re-check it via an external self-call.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with `whitelistEnabled = true`, adding `alice` to `readerWhitelist`.
2. Call `updatePriceFeeds` to populate price data.
3. As `alice` (a legitimately whitelisted address), call `getPricesNoOlderThan(subscriptionId, priceIds, 60)`.
4. Execution flow:
   - `onlyWhitelistedReader` passes for `alice` ✓
   - `this.getPricesUnsafe(subscriptionId, priceIds)` is dispatched as an external call; `msg.sender` inside `getPricesUnsafe` is `address(this)`.
   - `onlyWhitelistedReader` runs again: `address(this) != manager`, `whitelistEnabled == true`, `address(this)` not in whitelist → **reverts with `Unauthorized`**.
5. `alice`'s call reverts despite being whitelisted. `getPricesNoOlderThan` is permanently broken for this subscription.
