### Title
Unconditional Fee Deposit Before Freshness Check Permanently Locks SUI Coins in `PriceInfoObject` With No Withdrawal Path — (`target_chains/sui/contracts/sources/pyth.move`, `price_info.move`)

### Summary
`update_single_price_feed` deposits the caller's `Coin<SUI>` fee into `PriceInfoObject` **before** the freshness check. When the submitted update is stale (`publish_time <= cached_timestamp`), `is_fresh_update` returns `false`, the cache is not updated, but the fee is already permanently locked inside the object. No withdrawal function exists anywhere in the contract suite.

### Finding Description

**Step 1 — Fee deposited unconditionally before freshness check.**

In `pyth.move`, `update_single_price_feed` calls `deposit_fee_coins` at line 277, then calls `update_cache` (which calls `is_fresh_update`) at line 287: [1](#0-0) 

The deposit happens regardless of whether the update will actually freshen the cache.

**Step 2 — `is_fresh_update` silently skips the update for stale data.** [2](#0-1) 

If `update_timestamp <= cached_timestamp`, the function returns `false`, `update_price_info_object` is never called, and no event is emitted — but the fee is already gone.

**Step 3 — `deposit_fee_coins` has no corresponding withdrawal.**

The fee is stored as a `Coin<SUI>` dynamic object field under `FEE_STORAGE_KEY` inside `PriceInfoObject`: [3](#0-2) 

A search of the entire `target_chains/sui/contracts/sources/` tree finds **zero** functions named `withdraw_fee`, `collect_fee`, or any equivalent. `get_balance` is read-only. No governance action touches the `PriceInfoObject` balance.

**Step 4 — The fee recipient mechanism was explicitly abandoned.**

The comment at the top of `set_fee_recipient.move` confirms the design regression: [4](#0-3) 

`fee_recipient_address` is stored in `State` but is never used to drain `PriceInfoObject`. The governance action only updates the stored address field; it does not transfer any coins. [5](#0-4) 

### Impact Explanation

Every `Coin<SUI>` paid as an update fee — whether the update is fresh or stale — is permanently locked inside the corresponding `PriceInfoObject` with no recovery path. For stale updates specifically, the caller loses their fee and receives zero benefit. Because `PriceInfoObject` is a shared Sui object with `key + store` but no `drop` ability and no destructor that handles the embedded coin, the funds cannot be recovered by any on-chain path in the current deployment.

### Likelihood Explanation

This is trivially reachable by any unprivileged relayer. In a competitive multi-relayer environment, a second relayer submitting the same (or older) accumulator message will always trigger the stale path. The attacker does not need any special privilege — only the ability to call `update_single_price_feed` with a valid VAA from an authorized data source and a `publish_time` not exceeding the cached timestamp.

### Recommendation

1. **Refund on stale update**: Check `is_fresh_update` before depositing the fee. If the update is stale, return the `Coin<SUI>` to the caller (or abort with a clear error).
2. **Add a fee withdrawal path**: Implement a privileged `collect_fees(price_info_object, recipient)` function (gated by governance or the fee recipient) that drains the accumulated `Coin<SUI>` from `PriceInfoObject` to `fee_recipient_address`.
3. **Correct ordering**: Move `deposit_fee_coins` to after the freshness check, or restructure so the fee is only consumed on a successful cache update.

### Proof of Concept

```
1. Deploy with base_update_fee = 1000 MIST.
2. Submit a fresh accumulator update for feed F → cache timestamp = T.
3. Submit the same (or any older) accumulator update for feed F again
   (publish_time = T or T-1) via update_single_price_feed, paying 1000 MIST.
4. Observe: price_info::get_balance(price_info_object) increases by 1000 MIST.
5. Observe: no emit_price_feed_update event is emitted (is_fresh_update = false).
6. Attempt any on-chain call to recover the 1000 MIST → no such function exists.
7. Repeat N times: balance grows by N×1000 MIST, all permanently locked.
```

The invariant `user funds must not be frozen without a recovery path` is broken. This is a concrete, locally testable, bounty-grade finding.

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {
        // Get the timestamp of the update's current price
        let price_feed = price_info::get_price_feed(update);
        let update_timestamp = price::get_timestamp(&price_feed::get_price(price_feed));

        // Get the timestamp of the cached data for the price identifier
        let cached_price_info = price_info::get_price_info_from_price_info_object(price_info_object);
        let cached_price_feed =  price_info::get_price_feed(&cached_price_info);
        let cached_timestamp = price::get_timestamp(&price_feed::get_price(cached_price_feed));

        update_timestamp > cached_timestamp
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
