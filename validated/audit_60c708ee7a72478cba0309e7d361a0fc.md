### Title
`RecentReject::put()` Never Increments `self.total_keys_num`, Rendering `count_limit` Unenforceable — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put()` computes a local variable `total_keys_num` via `self.total_keys_num.checked_add(1)` but **never writes the result back to `self.total_keys_num`**. Because the field stays at its initial value (0 on a fresh DB), the `count_limit` guard is never triggered, and the underlying `DBWithTTL` (RocksDB) grows without bound.

---

### Finding Description

In `put()`:

```rust
// tx-pool/src/component/recent_reject.rs  lines 62-69
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

`total_keys_num` is a **shadow local variable**. `self.total_keys_num` is never assigned the incremented value. On a fresh DB, `self.total_keys_num` is initialised to 0 from `estimate_num_keys_cf` at construction time. [1](#0-0) 

Every subsequent call to `put()` evaluates `0 + 1 = 1`, checks `1 > 10_000_000` → false, and returns without calling `shrink()` or updating the field. [2](#0-1) 

`shrink()` is the only place that re-estimates and writes back `self.total_keys_num`, but it is unreachable under this logic. [3](#0-2) 

---

### Impact Explanation

An attacker who floods the node with low-fee-rate transactions causes unbounded growth of the `recent_reject` RocksDB column families. With the default `keep_rejected_tx_hashes_count = 10_000_000` the operator believes disk usage is capped; it is not. Sustained flooding exhausts disk space, crashing the node or making it unresponsive.

---

### Likelihood Explanation

The attack path is fully reachable by an unprivileged remote peer:

1. Peer sends structurally valid transactions whose fee is below `min_fee_rate`.
2. `check_tx_fee()` returns `Reject::LowFeeRate` early, before expensive script execution. [4](#0-3) 
3. `should_recorded()` returns `true` for `LowFeeRate` (only `Duplicated` is excluded). [5](#0-4) 
4. `after_process()` calls `put_recent_reject()` → `put()`. [6](#0-5) 
5. `self.total_keys_num` stays 0; no shrink; DB entry count grows without bound.

Each rejected transaction requires only a unique input UTXO (or even a structurally valid but unresolvable tx), making the cost per entry very low. The TTL only helps after the attacker stops; during an active flood it provides no protection.

---

### Recommendation

After writing to the DB, assign the incremented value back to `self.total_keys_num`:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ... db.put ...
    match self.total_keys_num.checked_add(1) {
        Some(n) => {
            self.total_keys_num = n;          // ← missing assignment
            if n > self.count_limit {
                self.shrink()?;
            }
        }
        None => {
            self.shrink()?;
        }
    }
    Ok(())
}
``` [7](#0-6)

### Proof of Concept

```rust
// Pseudocode – mirrors the existing test structure
let mut rr = RecentReject::build(tmp, 2, /*count_limit=*/10, -1).unwrap();
assert_eq!(rr.total_keys_num, 0);

for i in 0u64..15 {
    let key = Byte32::new(blake2b_256(i.to_le_bytes()));
    rr.put(&key, Reject::LowFeeRate(Default::default(), 100, 0)).unwrap();
}

// Bug: total_keys_num is still 0, shrink was never called
assert_eq!(rr.total_keys_num, 0);          // passes (demonstrates bug)
// DB actually holds 15 entries, exceeding count_limit=10
let real_count: u64 = (0..2)
    .map(|s| rr.db.estimate_num_keys_cf(&s.to_string()).unwrap().unwrap_or(0))
    .sum();
assert!(real_count > 10);                  // passes (demonstrates unbounded growth)
```

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
