### Title
ERC20 Fees Permanently Locked in Starknet Pyth Contract — No Withdrawal Mechanism - (File: `target_chains/starknet/contracts/src/pyth.cairo`)

---

### Summary

The Starknet Pyth contract collects ERC20 fees (ETH or STRK) from every caller of `update_price_feeds` via `transferFrom` into the contract address. However, the Starknet governance module defines no `WithdrawFee` action, and no other withdrawal path exists anywhere in the Starknet Cairo contracts. All collected fees are permanently locked.

---

### Finding Description

Every call to `update_price_feeds` (and related functions) on the Starknet Pyth contract triggers `transfer_fee`, which calls `transferFrom` to pull ERC20 tokens from the caller into the contract: [1](#0-0) 

The tokens land at `contract` (the Pyth contract address): [2](#0-1) 

The `execute_governance_instruction` handler dispatches over all defined `GovernancePayload` variants: [3](#0-2) 

The `GovernanceAction` enum in the Starknet governance module contains only these variants — `WithdrawFee` is entirely absent: [4](#0-3) 

A grep across all Starknet Cairo source files confirms zero occurrences of `withdraw`, `WithdrawFee`, or `withdraw_fee`. There is no admin function, no fallback, and no governance path to recover the accumulated ERC20 balance.

By contrast, the EVM Pyth contract implements `WithdrawFee` as governance action 9: [5](#0-4) 

And the Stylus (Arbitrum) Pyth receiver also implements `withdraw_fee` via governance: [6](#0-5) 

The Starknet deployment is the only chain missing this capability.

---

### Impact Explanation

All ERC20 fees (ETH or STRK tokens) paid by every user of the Starknet Pyth contract accumulate in the contract with no recovery path. The Pyth Data Association cannot retrieve protocol revenue from the Starknet deployment. The loss is permanent and grows with usage. This is a direct financial loss of protocol-owned assets.

**Impact: Medium** — funds are locked, not stolen; the contract continues to function, but all fee revenue is irrecoverable.

---

### Likelihood Explanation

The Starknet Pyth contract is live and actively used. Every `update_price_feeds` call pays a fee. The condition is triggered by normal, unprivileged user activity with no special preconditions.

**Likelihood: High** — occurs on every fee-paying interaction.

---

### Recommendation

Add a `WithdrawFee` variant to the `GovernanceAction` enum and `GovernancePayload` enum in `target_chains/starknet/contracts/src/pyth/governance.cairo`, parse it in `parse_instruction`, and handle it in `execute_governance_instruction` by calling `IERC20CamelDispatcher::transfer` to send the accumulated token balance to a governance-specified recipient address — mirroring the EVM implementation in `PythGovernanceInstructions.sol` and `PythGovernance.sol`.

---

### Proof of Concept

1. User calls `pyth.update_price_feeds(data)` on Starknet with sufficient ERC20 allowance.
2. `update_price_feeds_internal` calls `transfer_fee`, which calls `fee_contract.transferFrom(caller, contract, total_fee)` — tokens move into the Pyth contract.
3. Governance submits a VAA with action `WithdrawFee` (action byte `9`, as used on EVM). `parse_instruction` in the Starknet module hits the `_ => Option::None` branch and panics with `InvalidGovernanceMessage` — no such action is recognized.
4. No other entry point exists. Tokens remain locked indefinitely. [7](#0-6) [8](#0-7)

    fn withdraw_fee(
        &mut self,
        value: u64,
        expo: u64,
        target_address: Address,
    ) -> Result<(), PythReceiverError> {
        let fee_to_withdraw =
            U256::from(value).saturating_mul(U256::from(10).pow(U256::from(expo)));
        let current_balance = self.vm().balance(self.vm().contract_address());

        if current_balance < fee_to_withdraw {
            return Err(PythReceiverError::InsufficientFee);
        }

        self.vm()
            .transfer_eth(target_address, fee_to_withdraw)
            .map_err(|_| PythReceiverError::InsufficientFee)?;

        log(
            self.vm(),
            crate::FeeWithdrawn {
                target_address,
                fee_amount: fee_to_withdraw,
            },
        );

        Ok(())
    }
