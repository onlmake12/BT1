### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any User's Pending Transaction — (File: `rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC endpoint in `PoolRpcImpl` performs a destructive pool operation — evicting a named transaction and all its dependents — with no check that the caller is the original submitter or is otherwise authorized to act on that transaction. Any unprivileged RPC caller who knows (or can observe) a transaction hash can silently remove another user's pending transaction from the mempool, including its entire descendant chain.

---

### Finding Description

**Root cause — missing authorization check in `remove_transaction`:**

`rpc/src/module/pool.rs` lines 662–669 implement the handler:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

There is no check that `msg.sender` (the RPC caller) is the original submitter of the transaction, holds any key associated with the transaction's inputs, or has been granted any form of permission. The only input is the `tx_hash`, which is public information visible in `get_raw_tx_pool`.

The trait declaration at line 254–255 confirms the function signature accepts only a hash:

```rust
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
```

The documentation at lines 220–228 states the function "Removes a transaction **and all transactions which depend on it**" — meaning a single call can cascade-evict an entire chain of dependent transactions built by the victim.

**Exploit flow:**

1. Attacker calls `get_raw_tx_pool` (unauthenticated read) to enumerate all pending transaction hashes.
2. Attacker identifies a target transaction hash (e.g., a high-value DAO withdrawal or a time-sensitive competitive transaction).
3. Attacker calls `remove_transaction(target_tx_hash)` — no credentials required.
4. The target transaction and all its descendants are silently removed from the pool.
5. The victim's transaction is not confirmed; the victim must resubmit.
6. Attacker repeats step 3 each time the victim resubmits, creating a sustained eviction loop.

**Analog to the reference vulnerability:**

| Reference (`claimHA`) | CKB (`remove_transaction`) |
|---|---|
| Any caller can act on behalf of perpetual owner | Any caller can act against any transaction submitter |
| Caller specifies governance token amount (attacker-controlled parameter) | Caller specifies `tx_hash` (attacker-controlled parameter) |
| No check that caller is owner or approved | No check that caller is original submitter |
| Victim's claim gets worse terms | Victim's transaction gets evicted |

---

### Impact Explanation

- **Targeted transaction censorship:** An attacker can prevent a specific user's transaction from ever being confirmed by repeatedly evicting it on resubmission.
- **Descendant chain amplification:** Because the function removes all dependent transactions, a single call can evict an entire CPFP chain or a sequence of pre-signed transactions.
- **Time-sensitive financial harm:** DAO withdrawal transactions, time-locked operations, or competitive transactions (e.g., arbitrage, NFT mints) that miss their window due to repeated eviction result in direct financial loss to the victim.
- **No cryptographic protection:** Unlike transaction *submission* (which requires a valid lock-script witness), *removal* requires only knowledge of the hash, which is public.

---

### Likelihood Explanation

- The RPC endpoint is reachable by any process with network access to the RPC port. While the default bind address is localhost, many node operators expose the RPC to serve wallets, dApps, or indexers — a common and documented deployment pattern.
- The target transaction hash is trivially obtained from `get_raw_tx_pool`, which is also unauthenticated.
- No special privileges, keys, or majority hashpower are required.
- The attack is cheap: a single JSON-RPC call per eviction, repeatable indefinitely.

---

### Recommendation

Add a submitter-identity check before allowing removal. Since CKB transactions are not "owned" by an RPC identity, the practical mitigations are:

1. **Restrict `remove_transaction` to a privileged/authenticated RPC module** (e.g., behind the same access-control flag used for `clear_tx_pool` and `clear_tx_verify_queue`), so only the node operator — not arbitrary external callers — can invoke it.
2. **Alternatively, remove the endpoint from the public RPC surface entirely** and expose it only via a local admin socket or a separate privileged port.
3. At minimum, document clearly that this endpoint must not be exposed on a public-facing port, and enforce this with a configuration-time warning or bind-address restriction.

### Proof of Concept

```
# Step 1: Observe the pool (unauthenticated)
curl -X POST http://<node-rpc>/  \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false],"id":1}'
# Returns list of tx hashes including victim's 0xVICTIM_TX_HASH

# Step 2: Evict victim's transaction (unauthenticated, no key required)

  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["0xVICTIM_TX_HASH"],"id":2}'
# Returns: {"result": true}
