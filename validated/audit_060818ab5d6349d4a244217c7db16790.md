### Title
GAS_OVERHEAD Double-Counted Per Call When Keeper Batches Multiple `updatePriceFeeds()` Calls in One Transaction — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_processFeesAndPayKeeper` function in `Scheduler.sol` adds a fixed `GAS_OVERHEAD = 30,000` gas constant to every `updatePriceFeeds()` call to compensate for per-transaction costs (base fee, calldata). When a keeper batches N calls to `updatePriceFeeds()` across N different subscriptions in a single transaction, each call independently charges `GAS_OVERHEAD` to its subscription, while the actual transaction-level overhead is only paid once. The keeper is overpaid by `(N−1) × GAS_OVERHEAD × tx.gasprice` at the direct expense of subscription balances.

---

### Finding Description

`updatePriceFeeds()` captures `startGas = gasleft()` at entry and later calls `_processFeesAndPayKeeper`, which computes:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
``` [1](#0-0) [2](#0-1) 

`GAS_OVERHEAD = 30,000` is documented as a "rough estimate of the tx overhead for a keeper to call `updatePriceFeeds`": [3](#0-2) 

This constant is intended to cover the one-time per-transaction costs (EVM base fee of ~21,000 gas, calldata encoding, etc.) that are invisible to `gasleft()` inside the function. However, when a keeper batches N calls to `updatePriceFeeds()` for N different subscriptions in a single transaction, each of the N calls independently adds `GAS_OVERHEAD` to the keeper payment:

- **Actual transaction overhead paid by keeper:** ~21,000 gas (once)
- **GAS_OVERHEAD charged to subscriptions:** N × 30,000 gas

The keeper is overpaid by `(N−1) × 30,000 × tx.gasprice` wei, extracted from subscription balances.

The `_processFeesAndPayKeeper` function transfers this inflated amount directly to `msg.sender`: [4](#0-3) 

There is no per-transaction call counter or cap on how many subscriptions can be updated in a single transaction.

---

### Impact Explanation

A keeper batching 50 subscription updates in one transaction at 10 gwei gas price extracts an extra `49 × 30,000 × 10e9 = 0.0147 ETH` from subscription balances beyond the actual transaction cost. Subscription balances are drained faster than their managers expect, causing subscriptions to go underfunded and stop receiving price updates. The keeper profits at the direct expense of subscription managers who deposited funds expecting fair cost reimbursement.

---

### Likelihood Explanation

Batching multiple `updatePriceFeeds()` calls in a single transaction is the natural and economically rational behavior for any keeper — it reduces their own gas overhead. The `getActiveSubscriptions` function is explicitly designed to let keepers discover all subscriptions needing updates: [5](#0-4) 

The keeper network is permissionless and anyone can run a keeper node: [6](#0-5) 

A malicious keeper can trivially exploit this by batching as many subscription updates as possible per transaction.

---

### Recommendation

Replace the fixed `GAS_OVERHEAD` with a per-transaction overhead that is only charged once, regardless of how many subscriptions are updated in a single transaction. One approach: require keepers to call a batch entry point that captures `startGas` once and distributes the overhead across all subscriptions updated in that batch. Alternatively, reduce `GAS_OVERHEAD` to zero and rely solely on the measured `startGas - gasleft()` gas accounting, accepting that calldata costs are not reimbursed.

---

### Proof of Concept

1. Keeper identifies N = 100 active subscriptions all due for a heartbeat update via `getActiveSubscriptions()`.
2. Keeper constructs a single transaction that calls `updatePriceFeeds(subId_i, updateData_i)` for each of the 100 subscriptions (via a multicall wrapper or direct sequential calls).
3. Each of the 100 calls executes `_processFeesAndPayKeeper`, each adding `GAS_OVERHEAD = 30,000` to the keeper payment.
4. At `tx.gasprice = 10 gwei`, the keeper receives an extra `99 × 30,000 × 10e9 = 29,700,000,000,000 wei ≈ 0.0297 ETH` beyond the actual transaction overhead, extracted from the 100 subscription balances.
5. Repeated across many transactions, this systematically drains subscription balances faster than managers expect, causing subscriptions to deactivate prematurely. [7](#0-6)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
