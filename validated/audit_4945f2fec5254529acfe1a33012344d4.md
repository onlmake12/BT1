### Title
`GetBlocksProcess` Missing IBD State Check Allows Peers to Drain Node Resources During Initial Block Download - (`File: sync/src/synchronizer/get_blocks_process.rs`)

### Summary

The CKB synchronizer module explicitly documents that during Initial Block Download (IBD), the node should respond with `InIBD` to both `GetHeaders` **and** `GetBlocks` requests. However, while `GetHeadersProcess` correctly enforces this IBD gate, `GetBlocksProcess` has no IBD check at all. Any connected peer can send `GetBlocks` messages to a node in IBD mode and receive full block data in response, consuming disk I/O and CPU that should be dedicated to the node's own sync progress.

### Finding Description

The module-level comment in `sync/src/synchronizer/mod.rs` states the design contract explicitly:

> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests" [1](#0-0) 

`GetHeadersProcess::execute()` correctly enforces this: it checks `is_initial_block_download()` and returns early with an `InIBD` response when the node is syncing. [2](#0-1) 

`GetBlocksProcess::execute()`, however, contains **no IBD check**. It immediately proceeds to look up and serve blocks from the local store to any requesting peer, regardless of whether the node is in IBD: [3](#0-2) 

The `Synchronizer::received` handler dispatches both message types through `try_process` with no top-level IBD guard: [4](#0-3) 

The `Relayer::received` does have an IBD guard, but `GetBlocks` is a **Sync protocol** message, not a Relay protocol message, so that guard is irrelevant here. [5](#0-4) 

### Impact Explanation

During IBD, a node is supposed to be exclusively focused on downloading and verifying the chain. By sending `GetBlocks` messages, a malicious peer forces the IBD node to:

1. Perform `BlockStatus::BLOCK_VALID` lookups in RocksDB for each requested hash.
2. Retrieve full block bodies from the chain store and serialize them.
3. Send the block data back over the network.

An attacker can send up to `MAX_HEADERS_LEN` (2000) block hashes per message, and the node processes up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) of them per call. Repeated flooding with `GetBlocks` messages degrades IBD throughput by competing for disk I/O and CPU with the node's own block download pipeline. This directly contradicts the stated design intent and can materially slow down a node's ability to reach chain tip. [6](#0-5) 

### Likelihood Explanation

Any peer that establishes a connection and opens the Sync protocol can send `GetBlocks` messages. No special privilege, key, or majority hashpower is required. During IBD, the node accepts inbound connections and processes Sync messages normally. The attack requires only a TCP connection and knowledge of valid block hashes (which are public on-chain data).

### Recommendation

Add an IBD check at the start of `GetBlocksProcess::execute()`, mirroring the pattern in `GetHeadersProcess`. When the node is in IBD, it should respond with `InIBD` and return early, consistent with the documented design intent:

```rust
pub fn execute(self) -> Status {
    let active_chain = self.synchronizer.shared.active_chain();

    if active_chain.is_initial_block_download() {
        // Send InIBD response and return, matching GetHeadersProcess behavior
        self.send_in_ibd();
        return Status::ignored();
    }

    let block_hashes = self.message.block_hashes();
    // ... rest of existing logic
}
```

### Proof of Concept

1. Start a CKB node from genesis (it will be in IBD mode; `is_initial_block_download()` returns `true` because `unix_time_as_millis() - tip_header.timestamp() > MAX_TIP_AGE`).
2. Connect a custom peer that opens the Sync protocol.
3. Send a `GetBlocks` message containing hashes of known genesis-adjacent blocks.
4. Observe that the node responds with `SendBlock` messages containing full block data, rather than an `InIBD` response.
5. Contrast with sending a `GetHeaders` message to the same node, which correctly returns `InIBD` and logs: *"Ignoring getheaders from peer=X because the node is in initial block download stage."* [7](#0-6) [8](#0-7)

    pub fn execute(self) -> Status {
        let block_hashes = self.message.block_hashes();
        // use MAX_HEADERS_LEN as limit, we may increase the value of INIT_BLOCKS_IN_TRANSIT_PER_PEER in the future
        if block_hashes.len() > MAX_HEADERS_LEN {
            return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                "BlockHashes count({}) > MAX_HEADERS_LEN({})",
                block_hashes.len(),
                MAX_HEADERS_LEN,
            ));
        }
        let active_chain = self.synchronizer.shared.active_chain();

        let iter = block_hashes.iter().take(INIT_BLOCKS_IN_TRANSIT_PER_PEER);

        let mut dedup = HashSet::new();
        for block_hash in iter {
            debug!("get_blocks {} from peer {:?}", block_hash, self.peer);
            let block_hash = block_hash.to_entity();

            if block_hash == self.synchronizer.shared().consensus().genesis_hash() {
                return StatusCode::RequestGenesis.with_context("Request genesis block");
            }

            if !dedup.insert(block_hash.clone()) {
                return StatusCode::RequestDuplicate.with_context("Request duplicate block");
            }

            if !active_chain.contains_block_status(&block_hash, BlockStatus::BLOCK_VALID) {
                debug!(
                    "Ignoring get_block {} request from peer={} as it is not verified.",
                    block_hash, self.peer
                );
                continue;
            }

            if let Some(block) = active_chain.get_block(&block_hash) {
                debug!(
                    "respond_block {} {} to peer {:?}",
                    block.number(),
                    block.hash(),
                    self.peer,
                );
                let content = packed::SendBlock::new_builder().block(block.data()).build();
                let message = packed::SyncMessage::new_builder().set(content).build();

                let nc = Arc::clone(self.nc);
                self.synchronizer
                    .shared()
                    .shared()
                    .async_handle()
                    .spawn(async move { async_send_message_to(&nc, self.peer, &message).await });
            } else {
                // TODO response not found
                // TODO add timeout check in synchronizer

                // We expect that `block_hashes` is sorted descending by height.
                // So if we cannot find the current one from local, we cannot find
                // the next either.
                debug!("Stopping getblocks, since {} is not found", block_hash);
                break;
            }
        }

        Status::ok()
    }
