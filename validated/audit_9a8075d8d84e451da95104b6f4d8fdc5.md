Audit Report

## Title
RPC Thread-Pool Exhaustion via Unbounded `block_in_place` in `get_overview(refresh=4)` — (`rpc/src/module/terminal.rs`)

## Summary
The `send_message!` macro in `tx-pool/src/service.rs` calls `block_in_place(|| response.recv())` using a synchronous `std::sync::mpsc::Receiver`. Each `get_overview` request with `refresh=4` triggers three sequential blocking calls on the same tokio worker thread. With no concurrency cap on the endpoint, an attacker can exhaust all worker threads, preventing the tx-pool event loop from being scheduled, causing a self-reinforcing deadlock that renders the entire RPC server unresponsive for up to 30 seconds per wave.

## Finding Description
`ckb_channel::oneshot` is implemented as `std::sync::mpsc::sync_channel(1)`, making `response.recv()` a hard blocking call. [1](#0-0) 

The `send_message!` macro wraps this with `block_in_place(|| response.recv())`, occupying the calling tokio worker thread for the full duration of the tx-pool service round-trip. [2](#0-1) 

`TxPoolController::get_tx_pool_info` and `TxPoolController::get_block_template` both expand this macro. [3](#0-2) [4](#0-3) 

`TerminalRpcImpl::get_tx_pool_info` issues three sequential `block_in_place` calls per invocation: `tx_pool.get_tx_pool_info()` (line 582), `self.shared.get_block_template(...)` (line 590–600), and `tx_pool.get_total_recent_reject_num()` (line 602) — all on the same worker thread, with no async yield between them. [5](#0-4) 

`RefreshKind::TX_POOL_INFO = 0b00000100 = 4`, so `refresh=4` unconditionally bypasses the cache and enters this path. [6](#0-5) 

`get_overview` is a synchronous `fn` with no concurrency guard; it calls `get_tx_pool_info` directly. [7](#0-6) 

The axum server applies only a 30-second `TimeoutLayer` — no per-method semaphore, no concurrency limiter, no authentication middleware. [8](#0-7) 

**Deadlock amplification**: The tx-pool event loop is an async task on the same shared tokio runtime. `block_in_place` signals the runtime to migrate other tasks away from the blocked thread, but once all worker threads are simultaneously in `block_in_place`, there are no threads left to schedule the tx-pool event loop. The queued `GetTxPoolInfo` and `BlockTemplate` messages are never dequeued, so `recv()` never returns, extending the blockage until the 30-second timeout fires across all in-flight requests simultaneously. The attacker can immediately re-flood after each wave.

## Impact Explanation
All tokio worker threads become occupied. The tx-pool event loop starvation extends each blocking window to the full 30-second timeout. During this window, every RPC method — including `get_tip_block_number`, `send_transaction`, and `get_block_template` for miners — times out or queues indefinitely. This constitutes **High: Vulnerabilities which could easily crash a CKB node** (RPC layer rendered fully unresponsive, node appears offline to clients and miners).

## Likelihood Explanation
The Terminal module requires operator enablement, but it is a documented production monitoring feature. Once enabled, the endpoint is entirely unauthenticated — no auth middleware exists in the server stack. The attacker needs only HTTP access to the RPC port and the ability to issue concurrent requests. No keys, no PoW, no peer relationship are required. The number of concurrent requests needed equals the tokio worker thread count (typically `num_cpus`, e.g., 8–16), well within the 64-request PoC. The attack is repeatable indefinitely after each 30-second timeout wave.

## Recommendation
1. Replace `block_in_place(|| response.recv())` in the `send_message!` macro with a proper async oneshot channel (`tokio::sync::oneshot`) and `.await`, so the tokio thread yields rather than blocks.
2. Add a `tokio::sync::Semaphore` on `get_overview` to cap simultaneous in-flight requests.
3. Apply a shorter per-call timeout specifically on the `get_block_template` sub-call inside `get_tx_pool_info`.
4. Consider running the Terminal RPC handler on a dedicated `spawn_blocking` thread pool to isolate thread exhaustion from the main RPC runtime.

## Proof of Concept
```python
import asyncio, aiohttp, json

TARGET = "http://127.0.0.1:8114"

async def flood(session, i):
    payload = {"jsonrpc":"2.0","method":"get_overview","params":[4],"id":i}
    async with session.post(TARGET, json=payload) as r:
        return await r.text()

async def check_liveness(session):
    payload = {"jsonrpc":"2.0","method":"get_tip_block_number","params":[],"id":9999}
    try:
        async with session.post(TARGET, json=payload,
                                timeout=aiohttp.ClientTimeout(total=3)) as r:
            return r.status == 200
    except Exception:
        return False

async def main():
    async with aiohttp.ClientSession() as session:
        tasks = [flood(session, i) for i in range(64)]
        asyncio.gather(*tasks)
        await asyncio.sleep(1)
        alive = await check_liveness(session)
        assert not alive, "RPC should be unresponsive"
        print("CONFIRMED: RPC unresponsive during flood")

asyncio.run(main())
```

Send 64 concurrent `get_overview` requests with `refresh=4` to a node with the Terminal module enabled. After 1 second, a `get_tip_block_number` call with a 3-second timeout should fail, confirming full RPC unresponsiveness.
