Audit Report

## Title
RBF Extra-Fee Computed on Serialized Size Instead of Transaction Weight, Allowing Cycle-Heavy Replacements to Underpay — (`File: tx-pool/src/pool.rs`)

## Summary

`calculate_min_replace_fee` computes the mandatory RBF extra fee using only the raw serialized byte size of the incoming transaction, while every other fee-rate path in the tx-pool uses the transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). For a cycle-heavy replacement transaction, the weight can be orders of magnitude larger than the serialized size, causing the required extra fee to be severely underestimated and the RBF bump threshold to be effectively bypassed.

## Finding Description

`get_transaction_weight` at [1](#0-0)  defines the canonical weight metric as `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`.

`TxEntry::fee_rate()` correctly uses this weight: [2](#0-1) 

`FeeRate::fee()` multiplies the stored rate by whatever `weight` argument is passed: [3](#0-2) 

However, `calculate_min_replace_fee` passes only `size as u64` — the raw serialized byte count — ignoring cycles entirely: [4](#0-3) 

The enforcement gate in `check_rbf` calls this function with `entry.size` only: [5](#0-4) 

The same flaw exists in the `min_replace_fee` public helper, which also passes only `tx.size`: [6](#0-5) 

For a transaction with 100 serialized bytes and 70,000,000 cycles, `weight ≈ 11,940`, but the extra fee is computed as if weight = 100 — approximately 119× too low. The replacement is admitted after paying only ~0.8% of the intended extra fee.

## Impact Explanation

This matches the **High** bounty impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* The RBF extra-fee bump is the primary economic barrier against cheap mempool churn. By crafting cycle-heavy replacement transactions, an attacker can repeatedly displace existing pool transactions while paying a marginal cost ~119× below what the node operator configured. This enables sustained, low-cost mempool cycling that exhausts miner bandwidth, degrades mempool quality, and undermines the fee-rate ordering that miners rely on for block construction.

## Likelihood Explanation

Any caller with access to `send_transaction` (RPC or P2P) can trigger this. Producing a transaction with near-maximum cycle consumption and a small serialized body is straightforward for any script author. No leaked keys, admin access, or victim mistakes are required. The cycle count is observable before VM verification, so the attacker can precisely tune the size/cycle ratio to maximize the discount. The attack is repeatable at will.

## Recommendation

Add a `cycles: Cycle` parameter to `calculate_min_replace_fee` and replace the raw `size` argument with the transaction weight:

```rust
fn calculate_min_replace_fee(
    &self,
    conflicts: &[&PoolEntry],
    size: usize,
    cycles: Cycle,
) -> Option<Capacity> {
    let weight = get_transaction_weight(size, cycles);
    let extra_rbf_fee = self.config.min_rbf_rate.fee(weight);
    ...
}
```

Update both call sites (`check_rbf` at line 665 and `min_replace_fee` at line 98) to pass `entry.cycles` / `tx.cycles` alongside `entry.size` / `tx.size`. This aligns the RBF admission gate with the weight-based fee-rate metric used in `fee_rate()`.

## Proof of Concept

1. Configure node with `min_rbf_rate = 1500` shannons/KW, `max_tx_verify_cycles = 70_000_000`.
2. Submit T1: 100-byte body, fee = 10,000 shannons → accepted.
3. Craft T2: same inputs, 100-byte body, 70,000,000 cycles, fee = 10,150 shannons.
   - Node computes: `extra_rbf_fee = 1500 × 100 / 1000 = 150` shannons → check passes.
   - Correct value: `1500 × 11,940 / 1000 = 17,910` shannons → T2 should be rejected.
4. T2 is accepted despite paying only 150 shannons extra instead of the required 17,910 shannons.
5. Repeat with T3 replacing T2 at the same marginal cost to demonstrate sustained cheap cycling.

The discrepancy is directly verifiable by reading `calculate_min_replace_fee` at [4](#0-3)  against `fee_rate()` at [2](#0-1) .
