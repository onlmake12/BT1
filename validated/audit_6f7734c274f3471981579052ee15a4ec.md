### Title
Fees Permanently Locked in `PriceInfoObject` with No Withdrawal Path — (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

### Summary

Every call to `update_single_price_feed` deposits the caller's SUI fee into a dynamic object field on the `PriceInfoObject` via `deposit_fee_coins`. There is no function anywhere in the contract — including governance — that withdraws those coins. The `fee_recipient_address` stored in `State` is explicitly documented as no longer used. All fees paid since deployment are permanently locked.

### Finding Description

**Step 1 — Fee deposit before freshness check.**

`update_single_price_feed` deposits the fee unconditionally at line 277, before the freshness check at line 316 inside `update_cache`: [1](#0-0) 

Even when `is_fresh_update` returns `false` (same or older timestamp), the fee is already locked.

**Step 2 — Unbounded accumulation with no withdrawal.**

`deposit_fee_coins` merges every incoming coin into a single `Coin<SUI>` stored under `FEE_STORAGE_KEY` on the `PriceInfoObject`'s `UID`: [2](#0-1) 

The entire `price_info.move` module exposes only `deposit_fee_coins` (write) and `get_balance` (read) for this field. There is no `withdraw_fee_coins`, no `take_fee`, and no `transfer` call referencing `FEE_STORAGE_KEY` anywhere in the contract sources.

**Step 3 — Governance has no fee-withdrawal action.**

The complete set of governance actions is: [3](#0-2) 

None of these actions touch the `FEE_STORAGE_KEY` field on any `PriceInfoObject`. `execute_governance_instruction` dispatches only to `set_governance_data_source`, `set_data_sources`, `set_update_fee`, `set_stale_price_threshold`, and `set_fee_recipient`: [4](#0-3) 

**Step 4 — `fee_recipient_address` is explicitly deprecated.**

The `set_fee_recipient` module carries this comment: [5](#0-4) 

`set_fee_recipient::execute` only writes to `state.fee_recipient_address`: [6](#0-5) 

That address is never read back to transfer coins anywhere in the codebase.

### Impact Explanation

Every SUI fee ever paid to `update_single_price_feed` is permanently locked inside the corresponding `PriceInfoObject` with no on-chain path to recover it — not by the user, not by the fee recipient, not by governance. This constitutes:

- **Permanent freezing of funds**: all accumulated fees are irrecoverable.
- **Protocol insolvency**: the protocol's intended revenue mechanism is entirely non-functional; the fee recipient receives nothing.
- **Fee charged without service**: when a data source stops publishing, callers pay the fee but `is_fresh_update` returns `false`, so the price is not updated — the user loses SUI with no benefit.

### Likelihood Explanation

This affects 100% of fee payments from the moment of deployment. No special attacker action is required; normal protocol usage triggers it. The stale-feed scenario amplifies the harm (fee paid, no update delivered), but the locking is universal.

### Recommendation

1. Add a `withdraw_fee_coins` function (friend-gated to `pyth::pyth` or a governance module) that removes the `FEE_STORAGE_KEY` dynamic field and transfers the coin to `fee_recipient_address`.
2. Call this withdrawal inside `update_single_price_feed` (or a dedicated governance action) so fees flow to the recipient immediately rather than accumulating.
3. Consider reverting the fee deposit when `is_fresh_update` returns `false`, or at minimum document that the fee is non-refundable even for no-op updates.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
