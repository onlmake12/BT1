### Title
Remote Peer Can Inflate `task_count` to 128 and Lock 128 Block Slots in `InflightBlocks` for 30 Seconds — (`sync/src/types/mod.rs`)

### Summary

An unprivileged remote sync peer can manipulate `DownloadScheduler::task_count` to `MAX_BLOCKS_IN_TRANSIT_PER_PEER=128` by responding to block requests faster than the `fast_time` threshold, then stop responding, causing 128 specific `BlockNumberAndHash` slots to be locked in `InflightBlocks::inflight_states` for up to `BLOCK_DOWNLOAD_TIMEOUT=30s`, during which no other peer can be asked for those same blocks.

### Finding Description

**Entrypoint:** Any unprivileged P2P sync peer that advertises a `best_known_header` ahead of the victim's tip causes the victim's `BlockFetcher` to request blocks from it.

**Inflation path:**

`remove_by_block` is called each time a block is received from a peer: [1](#0-0) 

When the elapsed response time falls below `time_analyzer.fast_time` (default 750 ms), `TimeQuantile::MinToFast` is returned and `set.increase(2)` is called unconditionally (no `should_punish` guard on the increase branch): [2](#0-1) 

Starting from `INIT_BLOCKS_IN_TRANSIT_PER_PEER=32`, only 48 fast-responded blocks are needed to reach `MAX_BLOCKS_IN_TRANSIT_PER_PEER=128`. [3](#0-2) 

**Slot locking:**

`InflightBlocks::insert` uses a global `BTreeMap<BlockNumberAndHash, InflightState>`. If a slot is already occupied, it returns `false` and the block is not re-requested from any other peer: [4](#0-3) 

`BlockFetcher::fetch` skips blocks whose `insert` returns false: [5](#0-4) 

**Timeout:**

`prune` only evicts timed-out inflight entries after `BLOCK_DOWNLOAD_TIMEOUT=30s`: [6](#0-5) [7](#0-6) 

**`can_fetch` calculation:**

`can_fetch = task_count - hashes.len()`. With `task_count=128` and 128 hashes inflight, `can_fetch=0`, so the victim stops requesting new blocks from this peer too: [8](#0-7) 

### Impact Explanation

For 30 seconds, up to 128 specific `BlockNumberAndHash` entries are locked in `inflight_states`. Since the key includes the block hash (not just number), this only blocks those exact hashes — but because the attacker must be on the same chain as the victim to have valid blocks to respond with during the inflation phase, those hashes are the same ones honest peers would be asked for. Block synchronization for those slots stalls for the full timeout window. Multiple coordinated malicious peers can amplify the effect across more block ranges.

### Likelihood Explanation

The attacker only needs a standard P2P sync connection and the ability to serve valid blocks quickly (e.g., they are a well-resourced node on the main chain). No PoW, no key material, no privileged access is required. The inflation requires ~48 fast block responses, which is trivially achievable. The `time_analyzer` is shared across all peers, so a single fast attacker can shift the `fast_time` baseline, but the initial default of 750 ms is easy to beat on a local or low-latency network.

### Recommendation

1. Allow re-requesting a block from a different peer before the full 30-second timeout if the assigned peer has been unresponsive for a shorter interval.
2. Cap how much `task_count` can grow per peer within a single checkpoint window, or require `should_punish`-style conditions before allowing increases beyond `INIT_BLOCKS_IN_TRANSIT_PER_PEER`.
3. Consider making `time_analyzer` per-peer rather than global so one fast peer cannot skew the quantile thresholds for all peers.

### Proof of Concept

1. Connect a malicious peer to the victim node and advertise a `best_known_header` ahead of the victim's tip.
2. Respond to every `GetBlocks` request within <750 ms with valid blocks. After 48 responses, `task_count` reaches 128.
3. Once the victim has 128 blocks inflight to the malicious peer, stop responding entirely.
4. Assert that honest peers cannot fetch those same 128 block hashes for the next 30 seconds — `BlockFetcher::fetch` will skip them because `inflight_states` shows them as occupied, and `insert` returns `false`.
5. After 30 seconds, `prune` evicts the stale entries and honest peers can resume fetching.
