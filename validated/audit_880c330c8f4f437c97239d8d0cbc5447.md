### Title
Incomplete `ScriptHashType` Validation Skips Type Script Hash Type Check, Enabling Cheap DoS via Expensive Contextual Verification Bypass — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` validates that output **lock** script hash types are within the currently-enabled set (`ENABLED_SCRIPT_HASH_TYPE = {0,1,2,4}`), but entirely omits the same check for output **type** scripts. Meanwhile, the lower-level structural check (`check_data` / `verify_value`) accepts any even byte or `1` — covering future-reserved values like `Data3` (6), `Data4` (8), …, `Data127` (254). An unprivileged transaction sender can craft outputs whose type script carries a non-enabled hash type, bypass the cheap non-contextual gate, and force the node into expensive contextual (script-execution) verification before the transaction is finally rejected.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the four currently-permitted hash type bytes:

```
{0 = Data, 1 = Type, 2 = Data1, 4 = Data2}
```

The structural deserialisation guard `ScriptHashType::verify_value()` in `util/gen-types/src/core.rs` accepts **any** even byte or `1` — i.e., the full future-reserved range `{0,1,2,4,6,8,…,254}`. Both lock and type scripts pass this gate:

```rust
// check_data.rs
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()  // verify_value for both
    }
}
```

`ScriptHashTypeVerifier::verify()` is supposed to enforce the narrower consensus-level set, but it only iterates over the **lock** script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Only output.lock() is checked — output.type_() is never inspected
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...);
            }
        } else {
            return Err(...);
        }
    }
    Ok(())
}
```

The type script's hash type is never tested against `ENABLED_SCRIPT_HASH_TYPE`. A transaction output with `type_script.hash_type = 6` (`Data3`) therefore:

1. Passes `check_data()` — `verify_value(6)` is `true` (6 is even).
2. Passes `ScriptHashTypeVerifier` — only the lock script is examined.
3. Passes the full `NonContextualTransactionVerifier` pipeline.
4. Only fails at `select_version()` inside the CKB-VM script executor, which returns `InvalidScriptHashType` for any hash type not yet activated.

The comment directly above the struct even states the intent: *"Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules"* — but the implementation only covers half of each output.

---

### Impact Explanation

Every transaction submitted to the node (via RPC `send_transaction`, P2P relay, or compact-block reconstruction) passes through `NonContextualTransactionVerifier` first (cheap) and then `ContextualTransactionVerifier` / script execution (expensive). By crafting outputs with a non-enabled type script hash type, an attacker forces the node to reach the expensive contextual stage before the transaction is rejected. This is a resource-exhaustion / DoS vector:

- Script execution is cycle-bounded but still involves VM setup, cell-dep resolution, and scheduler overhead per transaction.
- The tx-pool will reject the transaction after contextual verification, but the CPU and memory cost has already been paid.
- A flood of such transactions from many peers can degrade node throughput without any fee cost to the attacker (the transactions never enter the pool and are never mined).

Additionally, if a miner assembles a block containing such a transaction (e.g., a malicious or buggy miner), the block passes `BlockVerifier` (which calls `NonContextualTransactionVerifier` per tx) and only fails at the contextual block-processing stage, wasting chain-processing resources on all validating nodes.

---

### Likelihood Explanation

The attack requires no privilege, no key, no majority hashpower, and no social engineering. Any RPC caller or P2P peer can submit a syntactically valid transaction with a type script whose `hash_type` byte is `6` (or any other even value outside `{0,2,4}`). The crafted transaction is indistinguishable from a legitimate one until contextual verification. The attack is trivially scriptable and repeatable.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's hash type for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_val: u8 = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?
            .into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&lock_val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: lock_val }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_val: u8 = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?
                .into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&type_val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: type_val }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Build a transaction with one output whose lock script uses `ScriptHashType::Type` (value `1`, enabled) and whose type script uses hash type byte `6` (`Data3`, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC or relay over P2P.
3. Observe that `NonContextualTransactionVerifier` accepts the transaction (lock script passes, type script is never checked).
4. Observe that the transaction is only rejected later, inside `ContextualTransactionVerifier`, when `select_version()` returns `ScriptError::InvalidScriptHashType` for the type script.
5. Repeat at high rate to exhaust node contextual-verification resources.
