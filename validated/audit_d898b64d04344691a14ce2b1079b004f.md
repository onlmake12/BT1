Audit Report

## Title
Wrong Accounting of Ancestor `descendants_*` Fields in `PoolMap::remove_entry_and_descendants` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` strips all link-map entries for the target and its descendants before calling `remove_entry` on any of them. Because `update_ancestors_index_key` resolves ancestors through those same links, it receives an empty ancestor set for every removed entry and never calls `sub_descendant_weight` on any ancestor that remains in the pool. The `descendants_size`, `descendants_cycles`, `descendants_fee`, and `descendants_count` fields of surviving ancestors are permanently inflated, corrupting their `EvictKey` and distorting pool eviction ordering.

## Finding Description
`remove_entry_and_descendants` collects the target and all its descendants, then calls `remove_entry_links` for every id in that set before calling `remove_entry` on any of them:

```rust
// pool_map.rs L252-L265
for id in &removed_ids {
    self.remove_entry_links(id);   // ALL links gone before any remove_entry call
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` removes the id's own entry from `self.links.inner` and unlinks it from all parents and children. When `remove_entry` is subsequently called, it invokes `update_ancestors_index_key`, which calls `self.links.calc_ancestors(&child.proposal_short_id())`. `calc_ancestors` calls `calc_relative_ids`, which looks up `self.inner.get(short_id)` — already deleted — and returns an empty `HashSet`. The loop over ancestors never executes, so `sub_descendant_weight` is never called on any ancestor that remains in the pool.

The comment in the function ("so that we won't `update_descendants_index_key` in `remove_entry`") shows the intent was only to suppress descendant-side updates for entries that are themselves being removed. The ancestor-side update (`update_ancestors_index_key`) was collaterally disabled, leaving surviving ancestors with stale inflated `descendants_*` fields for the remainder of their time in the pool.

Existing guards do not mitigate this: `remove_entry` does call `remove_entry_links` again at line 245, but since the links were already removed, this is a no-op. There is no compensating recalculation of `descendants_*` anywhere in the removal path.

## Impact Explanation
The `descendants_*` fields feed directly into `EvictKey` (entry.rs lines 234–247), which is the sole ordering used by `next_evict_entry` and therefore by `limit_size` (pool-full eviction). An ancestor whose `descendants_*` are inflated has an overstated `descendants_feerate` and `descendants_count` in its `EvictKey`, causing it to rank lower in eviction order (evicted later than it should be). When the pool is full, legitimate high-fee transactions with no descendants may be evicted in its place. This constitutes a suboptimal implementation of the CKB tx-pool state management mechanism, matching the **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism** impact class.

## Likelihood Explanation
`remove_entry_and_descendants` is called from three externally reachable paths: `resolve_conflict` (triggered by any committed block containing a tx whose inputs overlap with pool entries), `check_and_record_ancestors` (triggered when a new submission exceeds `max_ancestors_count`), and `resolve_conflict_header_dep` (triggered on fork/reorg). The most accessible path is via block confirmation: an unprivileged user submits a parent tx (tx0) and a child tx (tx1); when tx1's input is spent in a confirmed block, `remove_entry_and_descendants(tx1)` is called while tx0 remains in the pool with permanently inflated stats. This is repeatable with no special privileges and low cost.

## Recommendation
Before stripping links in `remove_entry_and_descendants`, identify the set of external ancestors (ancestors of the root that are not in `removed_ids`) and call `sub_descendant_weight` on each of them for every entry being removed. Alternatively, restructure the loop so that `update_ancestors_index_key` is called for each entry before `remove_entry_links` is called for that entry, preserving link integrity at the time of the ancestor walk. The simplest targeted fix is to add a pre-pass that collects external ancestors and decrements their `descendants_*` fields before any link removal occurs.

## Proof of Concept
```
Pool state: tx0 (low fee, no parents) → tx1 (high fee) → tx2 (high fee)

After add_entry for all three:
  tx0.descendants_count = 3
  tx0.descendants_fee   = fee0 + fee1 + fee2

Step 1: tx1's input is spent in a confirmed block.
        resolve_conflict calls remove_entry_and_descendants(tx1).

Step 2: remove_entry_links called for tx1 and tx2 — all links gone.

Step 3: remove_entry(tx1):
        update_ancestors_index_key(tx1, Remove)
          → calc_ancestors(tx1) = {} (links already gone)
          → tx0.sub_descendant_weight(tx1) NEVER CALLED

Step 4: remove_entry(tx2):
        update_ancestors_index_key(tx2, Remove)
          → calc_ancestors(tx2) = {} (links already gone)
          → tx0.sub_descendant_weight(tx2) NEVER CALLED

Result:
  tx0.descendants_count = 3   (should be 1)
  tx0.descendants_fee   = fee0 + fee1 + fee2  (should be fee0)

tx0's EvictKey reflects a falsely high descendants_feerate and descendants_count.
When the pool is full, tx0 is ranked as if it still has two high-fee children,
and legitimate transactions are evicted in its place.
```

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
