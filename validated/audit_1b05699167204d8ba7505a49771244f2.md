Audit Report

## Title
Unconditional `broadcast_exit_signals()` on Peer-Triggered Protocol Handler Panic Causes Node Shutdown - (File: `network/src/network.rs`)

## Summary

In `EventHandler::handle_error`, the `ServiceError::ProtocolHandleError` arm unconditionally calls `broadcast_exit_signals()` at line 717, regardless of whether the panic originated from a per-session handler (`opt_session_id = Some(id)`) or a global handler (`opt_session_id = None`). The `if let Some(id)` guard at line 693 only wraps `ban_session`; the shutdown call is outside it. Any remote peer that causes a session-scoped protocol handler to panic will trigger a full graceful shutdown of the local CKB node process.

## Finding Description

In `network/src/network.rs` lines 688–719:

```rust
ServiceError::ProtocolHandleError { proto_id, error } => {
    let ProtocolHandleErrorKind::AbnormallyClosed(opt_session_id) = error;
    {
        if let Some(id) = opt_session_id {
            self.network_state.ban_session(...);  // guarded
        }
        // ... logging ...
        broadcast_exit_signals();  // UNCONDITIONAL — line 717
    }
}
``` [1](#0-0) 

The `if let Some(id)` block closes at line 700. `broadcast_exit_signals()` at line 717 is inside the outer bare `{ }` block introduced at line 692, not inside the `if let`. It executes for both `Some(session_id)` (per-peer panic) and `None` (global handler panic).

`broadcast_exit_signals()` sets `RECEIVED_STOP_SIGNAL`, cancels the global `TOKIO_EXIT` `CancellationToken`, and sends on all registered crossbeam exit channels — the identical path taken on OS `SIGTERM`/`SIGINT`: [2](#0-1) 

Tentacle wraps each session-scoped handler invocation (`received`, `connected`, `disconnected`) in `catch_unwind`. A panic in any such handler is caught and surfaced as `ProtocolHandleError { error: AbnormallyClosed(Some(session_id)) }`. This is the `Some` branch — the per-peer case — yet it still reaches `broadcast_exit_signals()`.

## Impact Explanation

A single unprivileged remote peer that can cause any registered protocol handler to panic (via a crafted inbound message) will trigger a full graceful shutdown of the target CKB node. The attacker's session is banned for 5 minutes, but the node process exits. This is a remote single-peer, single-message denial-of-service against a CKB node, matching the allowed bounty impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**.

## Likelihood Explanation

The precondition is a reachable panic site in any session-scoped protocol handler (sync, relay, discovery, etc.) triggered by a crafted inbound message. CKB's protocol handlers are non-trivial and use `unwrap()`/`expect()` in various places. Any such site — present today or introduced by a future change — is immediately weaponizable because the shutdown path is structural and unconditional. The attacker needs only an inbound connection and the ability to send a message on the relevant protocol.

## Recommendation

Gate `broadcast_exit_signals()` on `opt_session_id` being `None`. A per-session handler panic should result only in banning the offending peer, not shutting down the node:

```rust
ServiceError::ProtocolHandleError { proto_id, error } => {
    let ProtocolHandleErrorKind::AbnormallyClosed(opt_session_id) = error;
    if let Some(id) = opt_session_id {
        self.network_state.ban_session(...);
        // log and continue — do NOT exit for a per-peer panic
    } else {
        // Global handler panic: no session to attribute, safe to exit
        broadcast_exit_signals();
    }
}
```

## Proof of Concept

1. Implement a `ServiceProtocol` whose `received` method unconditionally panics (e.g., `panic!("test")`).
2. Register it with the CKB network service and start the node.
3. Open an inbound session and send any message on that protocol.
4. Tentacle's `catch_unwind` catches the panic and invokes `handle_error` with `ProtocolHandleError { error: AbnormallyClosed(Some(session_id)) }`.
5. Observe: `ban_session` is called for the peer, then `broadcast_exit_signals()` is called unconditionally at line 717 — `TOKIO_EXIT` is cancelled, all crossbeam exit receivers fire, and the node process performs a graceful shutdown and exits. [3](#0-2) [4](#0-3)

```rust
pub fn broadcast_exit_signals() {
    debug!("Received exit signal; broadcasting exit signal to all threads");
    RECEIVED_STOP_SIGNAL.store(true, std::sync::atomic::Ordering::SeqCst);
    TOKIO_EXIT.cancel();
    CROSSBEAM_EXIT_SENDERS
        .lock()
        .iter()
        .for_each(|tx| match tx.try_send(()) {
            Ok(_) => {}
            Err(TrySendError::Full(_)) => info!("Ckb process has received exit signal"),
            Err(TrySendError::Disconnected(_)) => {
                debug!("broadcast thread: channel is disconnected")
            }
        });
}
