Audit Report

## Title
Stale Pre-Eviction Stat Snapshot Overwrites Decremented Counters in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` computes and snapshots `self.total_tx_size + entry.size` into a local variable before `check_and_record_ancestors` runs. When that call evicts cell-dep-referencing ancestors via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, the decrements are applied directly to `self.total_tx_size`/`self.total_tx_cycles`. The stale pre-eviction snapshot is then unconditionally written back at lines 218–219, permanently discarding every decrement. Repeated triggering monotonically inflates the counters, causing `limit_size` to evict legitimate transactions and `updated_stat_for_add_tx` to reject honest submissions with `Reject::Full` even when the pool has real capacity.

## Finding Description
**Root cause — `add_entry` (lines 200–221):**

```rust
// (1) snapshot T + new_size BEFORE any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // lines 210–211

// (2) may call remove_entry → update_stat_for_remove_tx,
//     which DECREMENTS self.total_tx_size / self.total_tx_cycles in place
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// (3) OVERWRITES the decremented live values with the stale snapshot
self.total_tx_size = total_tx_size;                             // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
```

**Eviction path — `check_and_record_ancestors` (lines 603–625):**
When `ancestors_count > max_ancestors_count` but `ancestors_count.saturating_sub(cell_ref_parents.len()) <= max_ancestors_count`, the lowest-fee cell-dep-referencing parents are removed via `remove_entry_and_descendants` (line 618).

**Decrement site — `remove_entry` (line 247):**
`remove_entry_and_descendants` (line 263) calls `remove_entry`, which calls `update_stat_for_remove_tx` (line 247), decrementing `self.total_tx_size` and `self.total_tx_cycles` directly on `self`.

**`update_stat_for_remove_tx` (lines 733–758):**
Performs a checked subtraction and writes the result back to `self.total_tx_size` / `self.total_tx_cycles` in place. These in-place writes are then silently overwritten by the stale snapshot at lines 218–219.

**`cell_ref_parents` population — `get_tx_ancenstors` (lines 529–534):**
`cell_ref_parents` is populated when `self.edges.deps.get(&input_pt)` is non-empty, i.e., when a pool transaction uses one of T_new's **direct** inputs as a cell dep. The eviction path is reachable whenever such a transaction exists and pushes `ancestors_count` above `max_ancestors_count`.

**Why existing checks fail:**
`updated_stat_for_add_tx` only checks for integer overflow on the pre-eviction baseline; it does not account for the fact that evictions will later reduce the true pool size. There is no guard preventing the stale snapshot from overwriting the live, eviction-adjusted counters.

## Impact Explanation
**Pool-size enforcement is broken:** `limit_size` (`pool.rs` line 298) loops while `self.pool_map.total_tx_size > self.config.max_tx_pool_size`. With an inflated counter, this loop fires when the pool has real capacity, evicting legitimate pending transactions.

**New-transaction admission is broken:** `updated_stat_for_add_tx` (lines 716–728) rejects any incoming transaction whose addition would overflow the inflated baseline, causing honest transactions to be rejected with `Reject::Full` even when the pool is not actually full.

**RPC reporting is incorrect:** Monitoring tools, wallets, and fee estimators receive inflated `total_tx_size`/`total_tx_cycles` values.

This matches **High** impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."* An attacker can repeatedly submit cheap crafted transactions to compound the inflation monotonically, progressively degrading mempool admission for all honest users across the network.

## Likelihood Explanation
The trigger requires no privileged access. Any unprivileged user can craft the required transaction structure. The key requirement is that a pool transaction uses one of T_new's direct inputs as a cell dep (populating `cell_ref_parents`), and the total ancestor count exceeds `max_ancestors_count` by exactly the number of such cell-dep-referencing parents. This is a normal CKB transaction pattern (cell dep referencing a live pool output). The attack is cheap, repeatable, and compounds monotonically with no majority hashpower, no leaked keys, and no victim mistakes required.

## Recommendation
Move the stat computation to **after** all evictions have completed, so it reads the already-decremented baseline:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Compute correct final totals against the post-eviction baseline
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, drop the local snapshot entirely and apply only the net delta after evictions:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size has already been decremented by evictions;
// add only the new entry's contribution
self.total_tx_size = self.total_tx_size.checked_add(entry.size)
    .ok_or_else(|| Reject::Full(...))?;
self.total_tx_cycles = self.total_tx_cycles.checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(...))?;
```

## Proof of Concept
1. Configure a node with default `max_ancestors_count = 25`.
2. Submit T1 → T2 → … → T24 as a linear input chain (each spends the previous output). All 24 are accepted.
3. Submit T_dep: uses **T24's output** as a **cell dep** (not input). T_dep is accepted with T1…T24 as ancestors (`ancestors_count = 25`). At this point `self.edges.deps[T24_output] = {T_dep}`.
4. Submit T_new: spends T24's output as input AND lists T_dep's output as a cell dep.
   - `get_tx_ancenstors`: for input T24's output, `self.edges.deps.get(&T24_output)` = `{T_dep}` → `cell_ref_parents = {T_dep}`.
   - Ancestors = T1…T24 + T_dep = 25; `ancestors_count = 26 > 25`.
   - `26 - 1 = 25 ≤ 25` → eviction branch at line 603 taken.
   - `remove_entry_and_descendants(T_dep)` → `remove_entry` → `update_stat_for_remove_tx` decrements `self.total_tx_size` by `T_dep.size`.
   - `add_entry` line 218 overwrites `self.total_tx_size` with the pre-eviction snapshot, erasing the decrement.
5. Query `tx_pool_info` via RPC: `total_tx_size` is inflated by `T_dep.size`.
6. Repeat steps 2–5 N times: `total_tx_size` grows by `N × T_dep.size` while the actual pool shrinks.
7. Once `total_tx_size` exceeds `max_tx_pool_size`, `limit_size` begins evicting legitimate transactions and new honest submissions are rejected with `Reject::Full`.
