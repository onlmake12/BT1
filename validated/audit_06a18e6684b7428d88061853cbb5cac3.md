Audit Report

## Title
Cycle Accounting Undercount in `FullSuspendedState::size()` for `inherited_fd` — (File: script/src/types.rs)

## Summary

`FullSuspendedState::size()` miscalculates the byte cost of the `inherited_fd` field (`Vec<(VmId, Vec<Fd>)>`). The formula charges only `inherited_fd.len() * size_of::<Fd>()` — counting outer tuple entries, not the total `Fd` values across all inner `Vec<Fd>`s — and omits the `VmId` per entry entirely. This causes a systematic undercount of up to 64× in the cycle charge applied during suspend/resume cycles.

## Finding Description

The field declaration at line 511:

```rust
pub inherited_fd: Vec<(VmId, Vec<Fd>)>,
```

The `size()` implementation at line 546:

```rust
+ (self.inherited_fd.len() * (size_of::<Fd>())) as u64
```

`self.inherited_fd.len()` returns the number of outer `(VmId, Vec<Fd>)` tuples, not the sum of all `Fd` values across all inner `Vec<Fd>`s. Additionally, the `VmId` component of each tuple is not counted at all. With 16 spawned VMs each holding 64 inherited FDs, the formula charges `16 × size_of::<Fd>() = 128` bytes, while the actual data is `16 × 64 × size_of::<Fd>() = 8192` bytes — a 64× undercount. The method's own doc comment confirms it "should be used to derive cycles charged for suspending / resuming," and `FullSuspendedState` is actively referenced in both `script/src/scheduler.rs` and `script/src/verify.rs`.

## Impact Explanation

This is an incorrect implementation of CKB-VM cycle metering, matching the allowed impact: **"Incorrect implementation or behavior of CKB-VM or system scripts" (High, 10001–15000 points)**. An attacker can obtain up to 64× more suspend/resume computation than they are charged for in cycles, violating the invariant that all CKB-VM computation must be fully cycle-metered. This can be used to perform work that would otherwise exhaust the cycle budget, potentially enabling network congestion at reduced cost.

## Likelihood Explanation

The exploit path is fully unprivileged: submit a `ScriptVersion::V2` transaction with a crafted lock script. No special keys, hashpower, or operator access are required. The parameters (`MAX_VMS_COUNT`, `MAX_FDS`) are VM-enforced bounds, so the maximum undercount is bounded at 64× but is consistently achievable. The bug is in production code and is reachable through normal transaction submission.

## Recommendation

Replace the flat `len()` multiplication with a fold that sums the actual `Fd` count across all inner `Vec<Fd>`s and accounts for the `VmId` per entry:

```rust
+ self.inherited_fd.iter().fold(0usize, |acc, (_, fds)| {
    acc + size_of::<VmId>() + fds.len() * size_of::<Fd>()
}) as u64
```

## Proof of Concept

1. Construct a `ScriptVersion::V2` lock script that creates 64 pipes via `ckb_pipe` and spawns 16 child VMs, passing all 64 pipe FDs as inherited FDs to each via `ckb_spawn`.
2. Force a scheduler suspend by approaching the cycle limit within the script execution.
3. Resume execution and invoke `FullSuspendedState::size()`.
4. Assert: `size()` returns a value where the `inherited_fd` contribution is `16 × size_of::<Fd>() = 128` bytes, while the actual serialized byte count is `16 × 64 × size_of::<Fd>() = 8192` bytes — confirming the 64× undercount. A unit test can construct a `FullSuspendedState` with 16 entries each containing 64 `Fd` values and assert the discrepancy directly. [1](#0-0) [2](#0-1) [3](#0-2)
