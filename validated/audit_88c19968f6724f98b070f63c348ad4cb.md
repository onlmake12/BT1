### Title
Missing Upper Limit on Provider `feeInWei` Causes `getProviderFee()` Arithmetic Overflow DoS — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any registered Entropy provider can call `setProviderFee(type(uint128).max)` with no upper-bound validation. When a user subsequently calls `getFeeV2(provider, gasLimit)` with a `gasLimit` exceeding the provider's `defaultGasLimit`, the internal `getProviderFee()` function performs an unchecked `uint128` multiplication that overflows, causing a Solidity 0.8+ panic revert. This permanently bricks all requests to that provider for any `gasLimit > defaultGasLimit`.

---

### Finding Description

`setProviderFee()` accepts any `uint128` value with no ceiling check:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;   // no upper bound
    ...
}
``` [1](#0-0) 

The downstream `getProviderFee()` then computes an additional fee for requests whose `gasLimit` exceeds the provider's `defaultGasLimit`:

```solidity
uint128 additionalFee = ((roundedGasLimit -
    provider.defaultGasLimit) * provider.feeInWei) /
    provider.defaultGasLimit;
``` [2](#0-1) 

Both operands are widened to `uint128` before the multiply. With `feeInWei = type(uint128).max` and any `roundedGasLimit - defaultGasLimit ≥ 2`, the product exceeds `type(uint128).max` and Solidity 0.8 reverts. `getFeeV2(provider, gasLimit)` therefore always reverts for that provider, and so does every call to `requestHelper` / `requestV2` that routes through it. [3](#0-2) 

The existing test suite explicitly acknowledges this revert path:

```solidity
vm.prank(provider1);
random.register(MAX_UINT128, ...);
vm.expectRevert();
random.getFee(provider1);
``` [4](#0-3) 

The same class of issue exists in `Echo.sol`. `setProviderFee()` accepts unbounded `uint96` fee components, and `getFee()` computes:

```solidity
uint256 gasFee = callbackGasLimit * providerFeeInWei;
feeAmount = ... + SafeCast.toUint96(gasFee);
``` [5](#0-4) 

With `feePerGasInWei` set near `type(uint96).max` and `callbackGasLimit ≥ 2`, `SafeCast.toUint96(gasFee)` reverts, blocking all `requestPriceUpdatesWithCallback` calls to that provider. [6](#0-5) 

---

### Impact Explanation

Any registered provider (an unprivileged, self-registering role) can permanently disable their own endpoint for all users requesting with `gasLimit > defaultGasLimit`. If the affected provider is the **default provider**, `getFeeV2()` (no-argument form) also reverts, breaking the primary user-facing entry point for the entire Entropy deployment on that chain. Users have no recourse except to switch providers, which is not always possible (e.g., single-provider deployments or integrations that hardcode the default provider address).

---

### Likelihood Explanation

Any address that has called `register()` can trigger this. No governance approval, key leak, or Sybil attack is required — a single `setProviderFee(type(uint128).max)` transaction suffices. The likelihood is low for the Pyth-operated default provider but non-negligible for third-party providers or in single-provider deployments.

---

### Recommendation

Add a reasonable upper-bound check in both `setProviderFee` functions and in `Echo.setProviderFee`:

```solidity
// Entropy.sol
uint128 constant MAX_FEE_IN_WEI = 1 ether; // or a governance-controlled cap
require(newFeeInWei <= MAX_FEE_IN_WEI, "fee exceeds maximum");
```

Alternatively, use `SafeMath`/checked arithmetic with a meaningful error message in `getProviderFee()` so the failure mode is explicit rather than a generic panic.

---

### Proof of Concept

1. Provider registers: `random.register(1, commitment, meta, 1000, uri)`
2. Provider sets `defaultGasLimit` to e.g. `100_000`
3. Provider calls `random.setProviderFee(type(uint128).max)`
4. User calls `random.getFeeV2(provider, 200_000)` → `getProviderFee` computes `(200_000 - 100_000) * type(uint128).max` → overflows `uint128` → **revert**
5. User calls `random.requestV2{value: ...}(provider, userRandom, 200_000)` → same revert path via `requestHelper` → **all requests to this provider are permanently bricked**

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }
