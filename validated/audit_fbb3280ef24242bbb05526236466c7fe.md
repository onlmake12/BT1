### Title
Stale `parents` Set After `remove_entry_and_descendants` Causes `panic!("inconsistent pool")` Node Crash — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `check_and_record_ancestors`, when a `cell_ref_parent` C is evicted via `remove_entry_and_descendants`, that call also removes all of C's in-pool descendants (e.g., D). However, the cleanup on line 620 only calls `parents.remove(next_id)` for C itself — it never removes D or any other descendant of C from `parents`. The subsequent `calc_relation_ids(parents, Parents)` at line 631 unconditionally inserts every ID from `parents` into the returned `ancestors` set, even IDs that no longer exist in `links.inner`. When `_record_ancestors` then calls `get_by_id_checked` on the stale ID D, the `.expect("inconsistent pool")` fires and crashes the node.

### Finding Description

**Exact topology to trigger (with `max_ancestors_count = 3`):**

| Tx | Role |
|----|------|
| A | In pool, no parents |
| C | In pool; uses A's `output[0]` as a **cell dep** → A is C's parent |
| D | In pool; spends C's `output[0]` as an **input** → C is D's parent |
| NewTx | Spends D's `output[0]` AND A's `output[0]` as inputs |

### Impact Explanation

An unprivileged attacker submits four transactions (A, C, D, NewTx) in the described topology via the standard `send_transaction` RPC or P2P relay. The fourth submission causes the CKB node process to panic and terminate. This is a remotely triggerable, zero-privilege **Denial of Service** that crashes the node.

---

### Likelihood Explanation

The attack requires only valid, well-formed transactions. No PoW, no key material, no privileged access. The topology (cell-dep chain + descendant overlap) is a natural pattern in CKB's UTXO+cell-dep model. Any attacker who can submit transactions to a node — which is the default for any peer — can trigger this.

---

### Recommendation

In the eviction loop, after `remove_entry_and_descendants(next_id)`, remove **all** returned entries from `parents`, not just `next_id`:

```rust
let removed = self.remove_entry_and_descendants(next_id);
ancestors_count = ancestors_count.saturating_sub(1);
parents.remove(next_id);
// FIX: also remove any descendants that were evicted
for evicted_entry in &removed {
    parents.remove(&evicted_entry.proposal_short_id());
}
evicted.extend(removed);
```

Alternatively, after the eviction loop, filter `parents` to only IDs still present in `links.inner` before passing to `calc_relation_ids`.

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
