### Title
Cross-Chain Signature Replay in `PythLazer.verifyUpdate` — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` computes the verification hash as `keccak256(payload)` with no chain ID or contract address bound into the signed data. Because `PythLazer` is intentionally deployed at the **same deterministic address** (`0xACeA761c27A909d4D3895128EBe6370FDE2dF481`) on 20+ EVM chains, a signed payload that passes verification on one chain passes verification on every other chain without modification.

---

### Finding Description

In `verifyUpdate`, the signature hash is constructed as:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
```

The `payload` bytes (parsed by `PythLazerLib`) contain only: a format magic, a timestamp, a channel identifier, and feed price data. Neither `block.chainid` nor `address(this)` is included in the signed material.

The deployment script explicitly uses CREATE3 to produce the same proxy address on every supported EVM network:

> "Having deterministic addresses make it easier for users to access our contracts"

`EvmLazerContracts.json` confirms `0xACeA761c27A909d4D3895128EBe6370FDE2dF481` is live on Ethereum, Base, Arbitrum, Polygon, BSC, Monad, Berachain, HyperEVM, Soneium, MegaETH, and more.

Because the contract address is identical and the signed hash contains no chain discriminator, a valid `(r, s, v, payload)` tuple accepted by `verifyUpdate` on Chain A is byte-for-byte accepted by `verifyUpdate` on every other chain.

---

### Impact Explanation

Consumer contracts call `verifyUpdate` and trust the returned `payload` as a legitimately fresh, chain-appropriate price update. An attacker who captures a signed update on one chain can submit it to any other chain where:

1. **Prices diverge between chains at the moment of capture** — e.g., a price spike on Arbitrum that has not yet propagated to Ethereum. The attacker replays the Arbitrum-captured update on Ethereum to feed a stale or manipulated price to a lending protocol or derivatives contract.
2. **Signer key rotation is staggered** — if a trusted signer is revoked on Chain A but still valid on Chain B, payloads signed by the revoked key remain acceptable on Chain B via replay.
3. **Chain-specific payload variants are introduced** — any future differentiation of payload content by chain (e.g., chain-specific feed IDs or metadata) immediately becomes exploitable.

The `verifyUpdate` return value (`payload`, `signer`) is the sole trust anchor for all downstream consumer contracts. There is no secondary on-chain check that binds the payload to the chain it was submitted on.

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: any EOA can call `verifyUpdate` with a captured update from another chain.
- **Signed updates are publicly observable** on-chain (emitted or passed as calldata) on every chain where Lazer is integrated.
- **The same signer key and same contract address** are present on all chains, so no privileged access is needed to construct a valid cross-chain replay.
- Likelihood is **medium**: requires price divergence between chains at the moment of capture, which occurs naturally during high-volatility periods or network congestion on one chain.

---

### Recommendation

Bind the signed hash to the specific chain and contract instance by including `block.chainid` and `address(this)` in the hashed data:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

This is the EVM analog of the WaveContract fix (including `address(this)` in the signed data). The Lazer backend must correspondingly include these fields when constructing the payload or the outer signed envelope.

---

### Proof of Concept

1. Deploy a consumer contract on Arbitrum that calls `pythLazer.verifyUpdate(update)` and records the returned price.
2. Observe a signed Lazer update on Arbitrum at time T when ETH/USD = $3,100 (Arbitrum price).
3. At the same time T, Ethereum's ETH/USD = $3,000 (price has not yet moved on Ethereum).
4. Submit the Arbitrum-captured `update` bytes to `verifyUpdate` on Ethereum (`0xACeA761c27A909d4D3895128EBe6370FDE2dF481`).
5. `keccak256(payload)` is identical; the signer is the same trusted key; `isValidSigner` passes.
6. The Ethereum consumer contract receives `$3,100` as the verified ETH/USD price, a $100 deviation from the true Ethereum price, enabling profitable manipulation of any protocol consuming this price.

```text
    function parsePayloadHeader(
        bytes memory update
    )
        public
        pure
        returns (
            uint64 timestamp,
            PythLazerStructs.Channel channel,
            uint8 feedsLen,
            uint16 pos
        )
    {
        uint32 FORMAT_MAGIC = 2479346549;

        pos = 0;
        uint32 magic = _readBytes4(update, pos);
        pos += 4;
        if (magic != FORMAT_MAGIC) {
            revert("invalid magic");
        }
        timestamp = _readBytes8(update, pos);
        pos += 8;
        channel = PythLazerStructs.Channel(_readBytes1(update, pos));
        pos += 1;
        feedsLen = uint8(_readBytes1(update, pos));
        pos += 1;
    }
```
