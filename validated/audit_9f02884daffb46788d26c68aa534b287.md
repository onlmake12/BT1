Audit Report

## Title
Fee Admission Check Uses Serialized Size Instead of Weight, Allowing High-Cycle Transactions to Bypass the Effective Minimum Fee Rate — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` gates admission using only serialized transaction size, not the actual resource weight (`max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because cycles are unknown before script execution, and no weight-based re-check is performed after `verify_rtx` returns the real cycle count, an attacker can submit transactions with near-maximum cycle consumption but minimal serialized size, paying only the size-based minimum fee while forcing the node to execute up to 70 million CKB-VM cycles per transaction — approximately 60× below the intended cost.

## Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` (L42–53) explicitly uses only `tx_size` for the fee gate, with a comment acknowledging this is a "cheap check":

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The actual weight formula in `util/types/src/core/tx_pool.rs` (L298–303) is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

The `_process_tx` flow in `tx-pool/src/process.rs` confirms no weight-based re-check exists after verification:

1. **L715–717**: `pre_check` → `check_tx_fee` (size-based only, cycles unknown). [4](#0-3) 
2. **L724–734**: `verify_rtx` executes scripts; actual cycles now known. [5](#0-4) 
3. **L751–754**: `TxEntry::new` + `submit_entry` — **no weight-based fee re-check**. [6](#0-5) 

`TxEntry::fee_rate()` computes the effective fee rate using weight but is only used for display/sorting, never as an admission gate: [7](#0-6) 

## Impact Explanation

With `DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES × 20 = 70,000,000` and `DEFAULT_BYTES_PER_CYCLES ≈ 0.000_170_571_4`:

- Max weight from cycles: `70,000,000 × 0.000_170_571_4 ≈ 11,940` weight-bytes
- Minimal tx serialized size: ~200 bytes
- Weight-based minimum fee at `min_fee_rate = 1,000 shannons/KB`: **~11,940 shannons**
- Size-based minimum fee (what is actually checked): **~200 shannons**

An attacker pays ~200 shannons but forces the node to execute 70 million CKB-VM cycles — a ~60× underpayment. By submitting many such transactions, the attacker saturates the `max_tx_verify_workers` thread pool with expensive script executions at negligible cost.

**Impact class: High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attack requires only a valid UTXO controlled by the attacker and a lock script consuming close to `max_tx_verify_cycles` (trivially constructable with a tight loop in CKB-VM). Submission is via the public `send_transaction` RPC or P2P relay. No privileged access, no key leakage, no majority hashpower is required. The economic cost is ~200 shannons per transaction, making sustained attacks cheap and fully repeatable as long as the attacker controls UTXOs.

## Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee rate check against the true weight before `submit_entry` in `_process_tx`:

```rust
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
```

The existing size-based check in `check_tx_fee` remains as the cheap pre-verification gate; the weight-based check is added as a post-verification gate in `_process_tx` before `submit_entry`. [6](#0-5) 

## Proof of Concept

1. Deploy a lock script that executes a tight loop consuming ~69,000,000 CKB-VM cycles (just under `max_tx_verify_cycles = 70,000,000`).
2. Create a transaction spending a cell locked by this script. Serialized size ≈ 200 bytes. Set `output capacity = input capacity − 200 shannons` (satisfying the size-based fee check: `1,000 × 200 / 1,000 = 200 shannons`).
3. Submit via `send_transaction` RPC.
4. `pre_check` calls `check_tx_fee` with `tx_size = 200`: `min_fee = 200 shannons`, `fee = 200 shannons` → **passes**. [8](#0-7) 
5. `verify_rtx` executes the lock script for ~69,000,000 cycles. [5](#0-4) 
6. No weight-based re-check occurs. Entry is admitted with `fee_rate() ≈ 17 shannons/KW` — far below `min_fee_rate = 1,000 shannons/KW`. [6](#0-5) 
7. Repeat with many UTXOs. Each submission consumes a full verification worker for the duration of the script execution at a cost of only 200 shannons per transaction.
