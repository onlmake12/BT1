### Title
`getPricesNoOlderThan` Uses Subscription-Level Max Timestamp Instead of Per-Feed Timestamp for Staleness Check — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` perform their staleness check against `status.priceLastUpdatedAt`, which is the **maximum** publish timestamp across all feeds in a subscription. However, individual feeds within a subscription can have significantly older timestamps (e.g., closed-market equity feeds). The staleness check therefore passes for a stale individual feed as long as any other feed in the same subscription has a recent timestamp, silently returning stale price data to consumers who explicitly requested freshness guarantees.

---

### Finding Description

In `_validateShouldUpdatePrices`, the subscription-level `priceLastUpdatedAt` is set to the **maximum** publish time across all feeds:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
``` [1](#0-0) 

This max value is then stored as the subscription's last-updated timestamp:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

The same code explicitly acknowledges that closed-market feeds will carry stale timestamps from their last trading session:

> *"Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period."* [3](#0-2) 

Yet `getPricesNoOlderThan` checks staleness only against this subscription-level max timestamp, not against the individual feed's own `publishTime`:

```solidity
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [4](#0-3) 

The same flaw exists in `getEmaPricesNoOlderThan`: [5](#0-4) 

The individual price data is stored and retrieved per-feed:

```solidity
_state.priceUpdates[subscriptionId][priceFeeds[i].id] = priceFeeds[i];
``` [6](#0-5) 

Each stored feed carries its own `publishTime`, but this per-feed timestamp is never consulted during the staleness check in `getPricesNoOlderThan`.

---

### Impact Explanation

A consumer calling `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` expects a guarantee that the returned AAPL price is no older than 60 seconds. In a subscription that also contains BTC (with a current timestamp), `priceLastUpdatedAt` equals BTC's recent timestamp. The staleness check passes, and the consumer receives AAPL's price from the last trading session (potentially days old) without any error. Protocols using this function for safety-critical decisions (liquidations, collateral valuation) would operate on silently stale data.

---

### Likelihood Explanation

Pyth supports equity price feeds alongside crypto feeds. Mixed subscriptions (e.g., BTC + AAPL) are a natural and documented use case. Equity markets are closed on weekends and after hours. A keeper running during off-hours would submit a valid update where BTC has a current timestamp and AAPL carries a timestamp from the last trading session. This is not an edge case — it is the explicitly designed behavior of the Scheduler, making the staleness bypass a routine occurrence for any mixed subscription.

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each individual feed's `publishTime`:

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId) returns (PythStructs.Price[] memory prices) {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
-       // OLD: uses subscription-level max timestamp
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

This mirrors how the core Pyth contract's `getPriceNoOlderThan` works — checking the individual feed's own `publishTime` rather than a shared subscription-level timestamp.

---

### Proof of Concept

1. Create a subscription with two price IDs: `BTC_ID` and `AAPL_ID`.
2. On a Saturday, call `updatePriceFeeds` with valid update data. BTC's `publishTime` = `block.timestamp` (e.g., 1_000_000). AAPL's `publishTime` = Friday close (e.g., 1_000_000 - 86_400 = 913_600). The slot check passes (same Pythnet slot). `priceLastUpdatedAt` is set to `1_000_000` (BTC's timestamp).
3. Call `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)`.
4. The check: `distance(1_000_000, 1_000_000) = 0 ≤ 60` → **passes**.
5. The returned AAPL price has `publishTime = 913_600`, which is 86,400 seconds (24 hours) old — far outside the requested 60-second freshness window.
6. The consumer receives a day-old AAPL price with no error, violating the `NoOlderThan` guarantee.
