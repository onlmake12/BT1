Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Eviction-Adjusted `total_tx_size`/`total_tx_cycles` in `PoolMap::add_entry` — (`tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` computes new aggregate totals into local variables before `check_and_record_ancestors` runs. When that function evicts conflicting transactions, `update_stat_for_remove_tx` correctly decrements `self.total_tx_size`/`self.total_tx_cycles` in-place. However, the stale pre-eviction locals are then unconditionally written back to `self`, silently discarding all eviction-driven decrements. The result is a permanently overstated pool size counter, causing legitimate transactions to be spuriously evicted and new submissions to be rejected with `Reject::Full`.

## Finding Description

The exact sequence in `add_entry` (lines 200–221):

1. **Line 210–211**: `updated_stat_for_add_tx` takes `&self` (immutable borrow) and returns new totals as local variables `total_tx_size` / `total_tx_cycles`. `self.total_tx_size` and `self.total_tx_cycles` are **not yet modified**. [1](#0-0) 

2. **Line 213**: `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (lines 603–625), it calls `remove_entry_and_descendants` for each eviction candidate, which calls `remove_entry` (line 247), which calls `update_stat_for_remove_tx` — **directly mutating `self.total_tx_size` and `self.total_tx_cycles`** (lines 739–740). [2](#0-1) [3](#0-2) [4](#0-3) 

3. **Lines 218–219**: The stale pre-eviction locals are unconditionally written back, overwriting the correctly decremented `self.total_tx_size` / `self.total_tx_cycles`. [5](#0-4) 

The eviction trigger condition in `check_and_record_ancestors` is confirmed at lines 603–625: eviction occurs when the ancestor count exceeds `max_ancestors_count` but can be brought within the limit by removing `cell_ref_parents`. [6](#0-5) 

No guard exists between lines 213 and 218–219 to reconcile the in-place mutations with the stale locals. The `update_stat_for_remove_tx` comment at line 731–733 itself acknowledges accounting inaccuracy, confirming this is a known fragile area. [7](#0-6) 

## Impact Explanation

`total_tx_size` is the sole counter driving two critical admission paths:

- **`limit_size` (pool.rs line 298)**: evicts transactions in a loop while `self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An overstated counter causes this loop to evict legitimate pending/proposed transactions that would otherwise fit. [8](#0-7) 

- **`updated_stat_for_add_tx` (pool_map.rs lines 716–721)**: rejects incoming transactions with `Reject::Full` when `checked_add` overflows. An overstated baseline causes premature overflow rejections. [9](#0-8) 

The overstatement is permanent and cumulative: each triggering of the eviction path adds the aggregate size/cycles of all evicted transactions to the phantom surplus. This constitutes a sustained, escalating DoS against the transaction pool, matching the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The eviction path is reachable by any unprivileged peer via `send_transaction` RPC or P2P relay. The attacker must:

1. Submit a set of transactions that share a cell dep, populating `cell_ref_parents` in the pool.
2. Submit a new transaction whose ancestor count exceeds `max_ancestors_count` (default 25) but whose `cell_ref_parents` overlap is large enough to satisfy the eviction condition at line 603.

This requires no privileged access, no majority hashpower, and no victim mistakes. The condition is constructible with a modest number of crafted transactions. The effect is repeatable: each successful trigger permanently inflates the counter further, compounding the DoS.

## Recommendation

Move the accounting update to **after** `check_and_record_ancestors` returns, so eviction-driven decrements are already reflected in `self.total_tx_size`/`self.total_tx_cycles` before the new entry's contribution is added. Replace lines 210–211 and 218–219 with a post-eviction addition:

```rust
// Remove pre-computation of total_tx_size/total_tx_cycles before check_and_record_ancestors.
// After check_and_record_ancestors returns (evictions already applied to self):
evicts = self.check_and_record_ancestors(&mut entry)?;
// Now add the new entry's contribution on top of the already-adjusted self values:
self.total_tx_size = self.total_tx_size.checked_add(entry.size).ok_or_else(|| {
    Reject::Full(format!("tx-pool total_tx_size {} overflows by add {}", self.total_tx_size, entry.size))
})?;
self.total_tx_cycles = self.total_tx_cycles.checked_add(entry.cycles).ok_or_else(|| {
    Reject::Full(format!("tx-pool total_tx_cycles {} overflows by add {}", self.total_tx_cycles, entry.cycles))
})?;
```

This ensures the overflow check still rejects entries that would genuinely overflow, while correctly accounting for any evictions that occurred during ancestor recording.

## Proof of Concept

**Concrete accounting trace** (matches the code path exactly):

| Step | Event | `self.total_tx_size` |
|---|---|---|
| Initial | Pool has 3 txs, total size 300 | 300 |
| `updated_stat_for_add_tx(50, …)` | Returns local `total_tx_size = 350`; `self` unchanged | 300 |
| `check_and_record_ancestors` evicts 2 txs (80 bytes each) | `update_stat_for_remove_tx` called twice, mutates `self` | 140 |
| `self.total_tx_size = total_tx_size` (line 218) | Stale local written back | **350** (should be 190) |

**Minimal reproduction steps:**
1. Build a pool with transactions T1, T2, T3 sharing a cell dep (so they appear in `cell_ref_parents`).
2. Construct transaction T4 with > 25 ancestors, where removing T1+T2 from `cell_ref_parents` brings the count within `max_ancestors_count`.
3. Call `add_entry(T4, Pending)`.
4. Assert `pool_map.total_tx_size == actual_sum_of_entry_sizes()` — the assertion will fail, with `total_tx_size` overstated by 160 (the combined size of T1+T2).
5. Repeat to compound the overstatement until `limit_size` begins evicting legitimate transactions or `updated_stat_for_add_tx` rejects valid submissions.
