### Title
Unprivileged Front-Running of `init_message_v1` Allows Permanent Theft of Victim's Rent Lamports — (`target_chains/solana/programs/core-bridge/src/processor/post_message/init_message_v1.rs`)

---

### Summary

`init_message_v1` accepts any `draft_message` account that is owned by `crate::ID` and fully zeroed, with no check binding the account to the calling `emitter_authority`. An attacker who observes a freshly created, zeroed, core-bridge-owned account can call `init_message_v1` first with their own keypair as `emitter_authority`, permanently locking the victim's rent lamports under attacker control. The attacker then calls `close_message_v1` to drain those lamports to an arbitrary destination.

---

### Finding Description

The `InitMessageV1` account struct imposes only two constraints on `draft_message`:

```
owner = crate::ID
```

and the `constraints()` function verifies the header bytes are all zero. [1](#0-0) 

There is no constraint linking `draft_message` to `emitter_authority` — no PDA derivation, no stored creator pubkey, no signer-of-creation check. Any signer can pass any zeroed, core-bridge-owned account as `draft_message`.

After a successful `init_message_v1`, the account is written with:
- Discriminator `b"msg\x00"`
- `emitter_authority` = the caller's pubkey
- `status` = `MessageStatus::Writing` [2](#0-1) 

`close_message_v1` then only checks that the signing `emitter_authority` matches the stored field and that `status == Writing`: [3](#0-2) 

`require_draft_message` enforces exactly this — discriminator present, `emitter_authority` matches signer, status is `Writing`: [4](#0-3) 

`close_account_destination` is fully unchecked — lamports go wherever the attacker specifies: [5](#0-4) 

---

### Impact Explanation

The victim permanently loses the rent lamports deposited into the `draft_message` account. The attacker gains them via `close_message_v1` to an arbitrary destination. The victim's subsequent `init_message_v1` call fails with `AccountNotZeroed` because the header is no longer zeroed, and there is no recovery path — the account is now controlled by the attacker's `emitter_authority`.

---

### Likelihood Explanation

The documented and intended usage pattern for large messages is to create the account in one transaction and call `init_message_v1` in a subsequent transaction (the code comment explicitly states: *"This account will have been created using the system program outside of the Core Bridge"*). [6](#0-5) 

This creates a real inter-transaction window. An attacker does not need traditional mempool access — they only need to poll for accounts owned by `crate::ID` with all-zero data (trivially done via `getProgramAccounts` with a `memcmp` filter on the header bytes). Once spotted, the attacker submits `init_message_v1` with their own keypair. On Solana's ~400ms slot cadence, this is a realistic race.

---

### Recommendation

Bind `draft_message` to `emitter_authority` at account-creation time. The cleanest fix is to derive `draft_message` as a PDA seeded by `emitter_authority` (and a nonce/salt), so only the holder of `emitter_authority` can create and initialize the matching account. Alternatively, store the intended `emitter_authority` pubkey in the account at creation time (e.g., as the first 32 bytes before the discriminator is written) and verify it in `constraints()`.

---

#[derive(Accounts)]
pub struct InitMessageV1<'info> {
    /// This authority is the only one who can write to the draft message account.
    emitter_authority: Signer<'info>,

    /// CHECK: This account will have been created using the system program outside of the Core
    /// Bridge.
    #[account(
        mut,
        owner = crate::ID
    )]
    draft_message: UncheckedAccount<'info>,
}

impl<'info> InitMessageV1<'info> {
    fn constraints(ctx: &Context<Self>) -> Result<()> {
        // Infer the expected message length given the size of the created account.
        let data_len = ctx.accounts.draft_message.data_len();
        require!(
            data_len > PostedMessageV1::PAYLOAD_START,
            CoreBridgeError::InvalidCreatedAccountSize
        );

        // This message length cannot exceed the maximum message length.
        require!(
            data_len - PostedMessageV1::PAYLOAD_START <= MAX_MESSAGE_PAYLOAD_SIZE,
            CoreBridgeError::ExceedsMaxPayloadSize
        );

        // Check that the message account header is completely zeroed out. By doing this, we make
        // the assumption that no other Core Bridge account that is currently used will have all
        // zeros. Ideally all of the Core Bridge accounts should have a discriminator so we do not
        // have to mess around like this. But here we are.
        let msg_acc_data: &[_] = &ctx.accounts.draft_message.try_borrow_data()?;
        require!(
            msg_acc_data[..PostedMessageV1::PAYLOAD_START] == [0; PostedMessageV1::PAYLOAD_START],
            CoreBridgeError::AccountNotZeroed
        );

        // Done.
        Ok(())
    }

    pub(crate) fn require_draft_message(
        acc_info: &AccountInfo,
        emitter_authority: &Signer,
    ) -> Result<bool> {
        let data = acc_info.try_borrow_data()?;
        require!(
            data.len() > 4 && data[..4] == *Self::LEGACY_DISCRIMINATOR,
            ErrorCode::AccountDidNotDeserialize
        );

        require_keys_eq!(
            Self::emitter_authority_unsafe(&data),
            emitter_authority.key(),
            CoreBridgeError::EmitterAuthorityMismatch
        );

        require!(
            Self::status_unsafe(&data) == MessageStatus::Writing,
            CoreBridgeError::NotInWritingStatus
        );

        Ok(true)
    }
