### Title
Unbounded `HashMapStateStore` Growth in `forward_rate_limiter` via Attacker-Controlled `(from, to)` Pairs — (`network/src/protocols/hole_punching/mod.rs`)

### Summary
The `forward_rate_limiter` in `HolePunching` uses a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`. Its keys are derived from attacker-controlled message fields (`content.from`, `content.to`). The only call to `retain_recent()` on this limiter is inside `disconnected()`. The periodic `notify()` handler (fired every 5 minutes) never calls `retain_recent()` on `forward_rate_limiter`. A connected peer can therefore grow the store without bound for the lifetime of the connection.

### Finding Description

**Type alias and field:** [1](#0-0) 

The `forward_rate_limiter` is keyed by `(PeerId, PeerId, u32)`. The key is constructed from message content: [2](#0-1) 

`msg_item_id` for `ConnectionRequestDelivered` is always the constant `1`: [3](#0-2) 

So the effective key space is `(content.from, content.to)` — both are arbitrary bytes from the wire message, with no validation that they correspond to real connected peers before the `check_key` call.

**`retain_recent()` is only called on disconnect:** [4](#0-3) 

**`notify()` (fires every 5 minutes) never calls `retain_recent()` on `forward_rate_limiter`:** [5](#0-4) 

The `notify()` handler cleans up `pending_delivered` and `inflight_requests` but has no corresponding cleanup for `forward_rate_limiter`. The `governor` crate's `HashMapStateStore` has no internal eviction — it is a plain `HashMap` that only shrinks when `retain_recent()` is explicitly called.

### Impact Explanation
An attacker maintains a single P2P connection and sends `ConnectionRequestDelivered` messages at the outer rate limit of 30/second, each with a fresh unique `(from, to)` pair. Each call to `check_key` inserts a new entry into the `HashMapStateStore`. A `PeerId` is ~39 bytes; each map entry costs roughly 150–200 bytes (key + governor state + `HashMap` overhead). At 30 entries/second over 24 hours from a single connection: ~388 MB. With multiple connections the growth scales linearly. The node never reclaims this memory until the attacker disconnects, causing progressive memory exhaustion on the local node.

### Likelihood Explanation
The path requires only a standard P2P connection — no privilege, no PoW, no key material. The `from` and `to` fields are parsed from the message body with no cross-check against the connected peer registry before `check_key` is invoked. The outer per-session rate limiter (30 rps) is the only throttle, and it does not prevent unbounded key diversity. The attack is trivially automatable.

### Recommendation
Call `self.forward_rate_limiter.retain_recent()` inside the `notify()` handler in `network/src/protocols/hole_punching/mod.rs`, mirroring the pattern already used in `disconnected()`. The existing 5-minute `CHECK_INTERVAL` is a reasonable cleanup cadence. The same pattern should be verified for `rate_limiter` in `sync/src/relayer/mod.rs`, which also only calls `retain_recent()` on disconnect. [4](#0-3) 

### Proof of Concept
```rust
// Unit test sketch (no network required)
use governor::{Quota, RateLimiter, state::keyed::HashMapStateStore, clock::DefaultClock};
use std::num::NonZeroU32;

let quota = Quota::per_second(NonZeroU32::new(1).unwrap());
let limiter: RateLimiter<(Vec<u8>, Vec<u8>, u32), HashMapStateStore<_>, DefaultClock> =
    RateLimiter::hashmap(quota);

for i in 0u64..100_000 {
    let from = i.to_le_bytes().to_vec();
    let to = (i + 1).to_le_bytes().to_vec();
    let _ = limiter.check_key(&(from, to, 1u32));
}

// retain_recent() is NOT called here (simulating no disconnect)
// Measure: the internal HashMap now holds 100,000 entries.
// Each entry ~150 bytes → ~15 MB from a single connection in ~55 minutes at 30 rps.
// retain_recent() would drop all of them (all buckets refilled after 1 second).
```

The test confirms that without a `retain_recent()` call, all 100,000 entries persist in the `HashMapStateStore`, and that a single call to `retain_recent()` after the fact would reclaim them all — demonstrating the fix is trivial and the growth is real.
