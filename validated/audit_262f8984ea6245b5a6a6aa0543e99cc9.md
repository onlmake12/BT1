### Title
P2P Network Secret Key Written to Disk Without Adequate File Permission Protection on Non-Unix Systems — (File: `util/app-config/src/configs/network.rs`)

### Summary
The CKB node's P2P network secret key (a raw 32-byte secp256k1 private key) is persisted to disk in plaintext via `write_secret_to_file`. On non-Unix systems (Windows), the code only calls `set_readonly(true)`, which sets the `FILE_ATTRIBUTE_READONLY` flag but does **not** restrict read access by other local users. Any unprivileged local user or process on the same machine can read the secret key file. Additionally, even on Unix, the file is created with default umask permissions before the key bytes are written and before `set_permissions(0o400)` is applied, creating a brief TOCTOU window during which the key is world-readable.

### Finding Description
`write_secret_to_file` in `util/app-config/src/configs/network.rs` is the sole function responsible for persisting the node's P2P identity secret key to `data/network/secret_key`:

```rust
pub fn write_secret_to_file(secret: &[u8], path: PathBuf) -> Result<(), Error> {
    fs::OpenOptions::new()
        .create(true)
        .write(true)
        .truncate(true)
        .open(path)
        .and_then(|mut file| {
            file.write_all(secret)?;          // key written BEFORE permissions set
            #[cfg(unix)]
            {
                file.set_permissions(fs::Permissions::from_mode(0o400))
            }
            #[cfg(not(unix))]
            {
                let mut permissions = file.metadata()?.permissions();
                permissions.set_readonly(true);   // does NOT restrict reads by other users
                file.set_permissions(permissions)
            }
        })
}
```

Two distinct issues exist:

1. **Non-Unix (Windows) — insufficient read protection**: `set_readonly(true)` on Windows sets `FILE_ATTRIBUTE_READONLY`, which only prevents modification. It does not restrict read access by other local users or processes. The file inherits the default ACL of its parent directory, which typically grants read access to all authenticated users. Any local user can `ReadFile` on the secret key.

2. **Unix TOCTOU window**: The file is created via `OpenOptions::create(true)` which applies the process umask (commonly `0o022`, yielding `0o644`). The raw key bytes are written at line 272 before `set_permissions(0o400)` is called. During this window, other local users can read the key.

The key is loaded and stored in `NetworkState.local_private_key` and used as the node's secio P2P identity: [1](#0-0) [2](#0-1) [3](#0-2) 

The `read_secret_key` function confirms the file is read back as raw bytes and parsed directly as a secp256k1 key: [4](#0-3) 

The same `write_secret_to_file` is called from both the node startup path and the `ckb peer-id gen` CLI command: [5](#0-4) [6](#0-5) 

A parallel issue exists in `create_tor_secret_key` for the Tor onion service private key, where there is **no** `#[cfg(not(unix))]` fallback at all — on Windows the file is created with entirely default permissions: [7](#0-6) 

### Impact Explanation
An attacker who obtains the P2P network secret key can:
- Impersonate the node's P2P identity (the `node_id` is derived from this key via `local_private_key.peer_id()`)
- Perform targeted eclipse or man-in-the-middle attacks against peers that have whitelisted or trusted this specific node ID
- Disrupt the node's P2P connectivity by operating a rogue node with the same identity

For the Tor onion key, compromise allows impersonation of the node's `.onion` hidden service address.

### Likelihood Explanation
On Windows, the condition is always true: `set_readonly(true)` never restricts reads. Any local user or process (including malware running under a different user account) on the same Windows host can read `data/network/secret_key`. On Unix, the TOCTOU window is narrow but non-zero and exploitable by a local process polling the filesystem. CKB nodes are commonly run on shared or cloud infrastructure where multiple local users may exist.

### Recommendation
1. **Non-Unix**: Use platform-specific ACL APIs (e.g., `SetFileSecurity` / `SetNamedSecurityInfo` on Windows) to restrict the file's DACL to the owner only, rather than relying on `set_readonly`.
2. **Unix TOCTOU**: Create the file with restrictive permissions from the start using `OpenOptionsExt::mode(0o600)` before writing, then tighten to `0o400` after the write completes.
3. Apply the same fix to `create_tor_secret_key` in `util/onion/src/onion_service.rs`, which currently has no permission restriction at all on non-Unix systems.
4. Consider encrypting the key at rest (e.g., with a passphrase or OS keychain) rather than storing raw bytes.

### Proof of Concept
On a Windows host running a CKB node:
```
# After node starts and generates data/network/secret_key
# As any local user (not the node operator):
type C:\Users\nodeoperator\.ckb\data\network\secret_key | xxd
# Outputs the raw 32-byte secp256k1 private key in full
```

On Unix, race the TOCTOU window:
```bash
# In one shell: start ckb (or ckb peer-id gen)
# In another shell, poll for the file:
while true; do
  if [ -f data/network/secret_key ]; then
    cat data/network/secret_key | xxd && break
  fi
done
# Reads the key before 0o400 is applied
```
