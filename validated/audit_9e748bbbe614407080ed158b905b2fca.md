### Title
Frontrunning `executeCallback` After Exclusivity Period Allows Fee Theft - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, the `executeCallback` function accepts a caller-controlled `providerToCredit` address and enforces provider exclusivity only during a configurable window. After that window expires, any unprivileged actor can frontrun the legitimate provider's pending `executeCallback` transaction, substituting their own address as `providerToCredit` and stealing the entire request fee.

### Finding Description
`executeCallback` takes `providerToCredit` as a free parameter from the caller. During the exclusivity window it enforces `providerToCredit == req.provider`, but once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds` that guard is gone entirely:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
// ... no further check on providerToCredit ...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no validation that `providerToCredit` is the registered provider for the request, nor even that it is a registered provider at all. The fee is unconditionally credited to whatever address the caller supplies. [1](#0-0) [2](#0-1) 

### Impact Explanation
The legitimate provider who submitted the price update data loses the fee they were entitled to. The attacker gains that fee. Because `registerProvider` is permissionless and `setFeeManager` lets a provider designate themselves as their own fee manager, the attacker can fully extract the stolen balance via `withdrawAsFeeManager`. The user's callback still executes (the request is fulfilled), so the consumer contract is unaffected, but the economic incentive for honest providers is undermined: any provider who delays past the exclusivity window can have their fee stolen by a mempool observer. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
The attack requires no special privilege. Any address can register as a provider (permissionless, zero fees allowed), set itself as fee manager, and then watch the public mempool. On any EVM chain with a public mempool the attacker simply copies `updateData` and `priceIds` from the victim's pending transaction and resubmits with a higher gas price and `providerToCredit = attackerAddress`. The exclusivity period only delays the window; it does not prevent the attack once elapsed. Requests that the assigned provider fails to fulfill within the exclusivity window are the primary target, but even a provider who submits just after exclusivity expires is vulnerable. [5](#0-4) 

### Recommendation
1. **Bind `providerToCredit` to the caller**: replace the free parameter with `msg.sender` and require `msg.sender` to be a registered provider, or
2. **Validate `providerToCredit` is registered**: add `require(_state.providers[providerToCredit].isRegistered, "Provider not registered")` before crediting fees, and
3. **After exclusivity, still restrict `providerToCredit` to `req.provider`** unless a penalty/redistribution mechanism is intentionally desired. If open fulfillment is desired, credit `msg.sender` (the actual fulfiller) rather than an arbitrary address parameter.

### Proof of Concept
1. Attacker calls `registerProvider(0, 0, 0)` → attacker is now a registered provider.
2. Attacker calls `setFeeManager(attackerAddress)` → attacker is their own fee manager.
3. User calls `requestPriceUpdatesWithCallback(legitimateProvider, publishTime, priceIds, gasLimit)` paying fee `F`.
4. Exclusivity period (`publishTime + exclusivityPeriodSeconds`) elapses without fulfillment.
5. Legitimate provider broadcasts `executeCallback(legitimateProvider, seqNum, updateData, priceIds)`.
6. Attacker sees this in the mempool, copies `updateData` and `priceIds`, and broadcasts `executeCallback(attackerAddress, seqNum, updateData, priceIds)` with higher gas price.
7. Attacker's transaction lands first: `_state.providers[attackerAddress].accruedFeesInWei += F - pythFee`.
8. Legitimate provider's transaction reverts with `NoSuchRequest`.
9. Attacker calls `withdrawAsFeeManager(attackerAddress, F - pythFee)` and receives the stolen fee. [6](#0-5) [7](#0-6)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
    }
