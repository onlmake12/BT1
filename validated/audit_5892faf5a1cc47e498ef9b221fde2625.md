Audit Report

## Title
RBF `replace_count` Double-Counts Shared Descendants in Diamond Dependency Graphs, Causing False Rejection of Legitimate RBF Transactions - (File: `tx-pool/src/pool.rs`)

## Summary
In `check_rbf`, the `replace_count` variable accumulates `descendants.len() + 1` per conflicting transaction without a global deduplication set. When two conflicting transactions share a common descendant (diamond dependency), that descendant is counted once per conflict, inflating `replace_count` above `MAX_REPLACEMENT_CANDIDATES` and causing a false `RBFRejected` error for a transaction whose true unique replacement set is within the limit.

## Finding Description
At `tx-pool/src/pool.rs` lines 613–624, the Rule #5 loop calls `self.pool_map.calc_descendants(&conflict.id)` independently for each conflict and adds `descendants.len() + 1` to `replace_count`. `calc_descendants` (links.rs lines 82–84) performs a fresh BFS from a single root each call, returning a per-root `HashSet` with no awareness of prior calls. A diamond topology — transaction C spending one output from A and one from B — is fully supported: `record_entry_descendants` (pool_map.rs lines 487–513) iterates all outputs of C and calls `add_parent` for every parent whose output C spends, recording C as a child of both A and B in `TxLinksMap`. Consequently, `calc_descendants(A)` returns `{C, …}` and `calc_descendants(B)` also returns `{C, …}`, and C is counted twice. With 50 shared children, `replace_count` reaches 102 while the true unique replacement set has size 52. The fix pattern already exists in `calculate_min_replace_fee` (lines 104–108), which deduplicates via a `HashMap` keyed on `id`, but the `replace_count` path has no equivalent guard.

## Impact Explanation
A legitimate RBF transaction is rejected by the local node's RPC with `RBFRejected("Tx conflict with too many txs, conflict txs count: 102, expect <= 100")` even though only 52 unique transactions would be replaced. This is an incorrect local RPC API result triggered by a normal user action, matching the **Note (0–500 points): Any local RPC API crash** impact class.

## Likelihood Explanation
Requires only standard RPC access. The attacker submits two transactions A (with ≥50 outputs) and B (with ≥50 outputs), then 50 child transactions each spending one distinct output from A and one distinct output from B. No privileged access, no majority hashpower, and no external dependency is required. Diamond dependency graphs arise organically from consolidation transactions and are fully valid pool topologies. The condition is deterministically reproducible.

## Recommendation
Accumulate all descendants into a single global `HashSet` before computing `replace_count`, mirroring the deduplication already present in `calculate_min_replace_fee`:

```rust
let mut all_descendants: HashSet<ProposalShortId> = HashSet::new();
for conflict in conflicts.iter() {
    all_descendants.extend(self.pool_map.calc_descendants(&conflict.id));
}
let replace_count = all_descendants.len() + conflicts.len();
if replace_count > MAX_REPLACEMENT_CANDIDATES {
    return Err(Reject::RBFRejected(format!(
        "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
        replace_count, MAX_REPLACEMENT_CANDIDATES,
    )));
}
```

## Proof of Concept
1. Submit tx A with 50 outputs, spending cell X.
2. Submit tx B with 50 outputs, spending cell Y.
3. Submit 50 child transactions C₁…C₅₀, where Cᵢ spends output i from A and output i from B.
4. Submit RBF tx R spending both X and Y with fee exceeding the minimum replace fee.
5. In `check_rbf`: `conflicts = [A, B]`. For A: `descendants = {C₁…C₅₀}`, `replace_count += 51`. For B: `descendants = {C₁…C₅₀}`, `replace_count += 51`. Total `replace_count = 102 > 100`.
6. R is rejected with `"Tx conflict with too many txs, conflict txs count: 102, expect <= 100"` even though only 52 unique transactions (`A, B, C₁…C₅₀`) would be replaced.

        &self,
        short_id: &ProposalShortId,
        relation: Relation,

        let direct = self
            .inner

            .cloned()
            .unwrap_or_default();

        self.calc_relation_ids(direct, relation)

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }

    fn record_entry_descendants(&mut self, entry: &TxEntry) {
        let tx_short_id: ProposalShortId = entry.proposal_short_id();
        let outputs = entry.transaction().output_pts();
        let mut children = HashSet::new();

        // collect children
        for o in outputs {
            if let Some(ids) = self.edges.get_deps_ref(&o).cloned() {
                children.extend(ids);
            }
            if let Some(id) = self.edges.get_input_ref(&o).cloned() {
                children.insert(id);
            }
        }
        // update children
        if !children.is_empty() {
            for child in &children {
                self.links.add_parent(child, tx_short_id.clone());
            }
            if let Some(links) = self.links.inner.get_mut(&tx_short_id) {
                links.children.extend(children);
            }
            self.update_descendants_index_key(entry, EntryOp::Add);
        }
        // update ancestor's index key for adding new entry
        self.update_ancestors_index_key(entry, EntryOp::Add);
    }
