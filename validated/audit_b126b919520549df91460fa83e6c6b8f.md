### Title
Peer Store Monopolization via `attempts_count` Reset in `AddrManager::add` Bypasses `check_purge` Phase-1 Eviction — (`network/src/peer_store/peer_store_impl.rs`, `network/src/peer_store/addr_manager.rs`)

---

### Summary

The two-phase `check_purge` logic preferentially evicts non-connectable addresses in phase 1, only falling through to group-diversity eviction in phase 2 when phase 1 finds nothing. A critical design flaw in `AddrManager::add` allows an attacker to silently reset `attempts_count` to zero on any never-connected address by re-announcing it via the discovery protocol. This keeps attacker addresses perpetually connectable, ensuring they survive phase 1 while honest peers — whose feeler attempts accumulate — are continuously evicted. Phase 2 (group-based diversity enforcement) is never reached.

---

### Finding Description

**Root cause 1 — `AddrManager::add` resets `attempts_count` on re-announcement**

`add_addr` always constructs `AddrInfo::new(addr, 0, score, flags)`, setting `last_connected_at_ms = 0` and `attempts_count = 0`. [1](#0-0) 

Inside `AddrManager::add`, when the address already exists, the update guard is:

```rust
if addr_info.last_connected_at_ms >= exist_last_connected_at_ms {
    addr_info.random_id_pos = random_id_pos;
    self.id_to_info.insert(id, addr_info);  // full replacement
}
``` [2](#0-1) 

For a never-connected address, `exist_last_connected_at_ms = 0`. The incoming `addr_info.last_connected_at_ms` is also `0`. The condition `0 >= 0` is **always true**, so the entire record is replaced — including resetting `attempts_count` to `0`. An attacker who controls a connected peer can re-announce the same addresses via discovery `Nodes` messages at any time to reset this counter.

**Root cause 2 — `is_connectable` depends entirely on `attempts_count`**

A never-connected address becomes non-connectable only when `attempts_count >= ADDR_MAX_RETRIES (3)`:

```rust
if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
    return false;
}
``` [3](#0-2) 

Because the attacker can reset `attempts_count` to 0 at will, their addresses never cross this threshold.

**Root cause 3 — `check_purge` phase 1 short-circuits phase 2**

Phase 1 collects all `!is_connectable` addresses and removes them. If any are found, the function returns `Ok(())` immediately — phase 2 (group-diversity eviction) is skipped entirely:

```rust
if candidate_peers.is_empty() {
    // phase 2: group eviction
    ...
}
Ok(())
``` [4](#0-3) 

As long as at least one honest peer has accumulated 3 failed feeler attempts, phase 1 will always find something to evict, and phase 2 — which would enforce /16-group diversity — is never reached.

**Attacker entry point — discovery protocol**

`DiscoveryAddressManager::add_new_addrs` calls `peer_store.add_addr` for every address in a received `Nodes` message, with no per-sender rate limit: [5](#0-4) 

A non-announce `Nodes` message carries up to `MAX_ADDR_TO_SEND = 1000` items; announce messages carry up to `ANNOUNCE_THRESHOLD = 10` items × `MAX_ADDRS = 3` addresses = 30 per message. With `ADDR_COUNT_LIMIT = 16384`, the attacker fills the store in ~17 non-announce messages (sent once per session) and maintains it via periodic announce messages. [6](#0-5) 

**Phase 2 group eviction is also defeatable**

Phase 2 only evicts from groups with `> 4` peers, and only from the top `len/2` groups by size. If the attacker spreads 16384 addresses across ≥ 4097 distinct /16 subnets (trivially achievable with a /8 range), every group has ≤ 4 members, and phase 2 evicts nothing even when it does trigger. [7](#0-6) 

The `Group` key for IPv4 is only the first two octets: [8](#0-7)

### Impact Explanation

The victim node's peer store (capacity 16384) is monopolized by attacker-controlled addresses. The feeler connection pool (`fetch_addrs_to_feeler`) draws exclusively from never-connected addresses, so the node's feeler probes go only to attacker nodes. Successful feeler connections promote attacker addresses into the "attempt" pool (`fetch_addrs_to_attempt`), biasing all outbound connection selection toward attacker peers. Honest peers added via discovery are continuously evicted in phase 1 as their feeler attempts accumulate. This is a concrete step toward an eclipse attack: the node's view of the network is progressively replaced by attacker-controlled topology.

---

### Likelihood Explanation

The attacker needs only a single inbound or outbound P2P connection to the victim to send discovery `Nodes` messages. No special privileges, no PoW, no key material. The `attempts_count` reset is unconditional for never-connected addresses. The attack is self-sustaining: periodic announce messages (which the protocol sends every 24 hours, but the attacker can send more frequently) maintain the reset. Honest peers naturally accumulate failed feeler attempts over time, making phase 1 eviction of honest peers inevitable.

---

### Recommendation

1. **Fix `AddrManager::add` to preserve `attempts_count`**: When updating an existing entry, copy `attempts_count` and `last_tried_at_ms` from the existing record rather than replacing the full entry. The update should only refresh `last_connected_at_ms` and `flags` from the incoming announcement.

2. **Decouple phase 1 from phase 2**: After phase 1 eviction, still check whether the store remains at or above the limit and run phase 2 if needed. Currently, any phase 1 eviction — even of a single address — suppresses group-diversity enforcement entirely.

3. **Bound per-sender address injection**: Track how many addresses each session has contributed and cap the total, preventing a single peer from filling the store.

---

### Proof of Concept

```
1. Attacker connects to victim (one P2P session).
2. Attacker sends Nodes(announce=false) messages totaling 16384 addresses,
   each from a distinct /16 subnet (e.g., 1.0.x.x through 64.255.x.x).
   → peer store is full; all entries: last_connected_at_ms=0, attempts_count=0.
3. Victim's feeler logic dials attacker addresses; all fail.
   → attempts_count increments toward 3 for each attacker address.
4. Before any attacker address reaches attempts_count=3, attacker sends
   Nodes(announce=true) messages re-announcing the same addresses.
   → AddrManager::add: 0 >= 0 → full replacement → attempts_count reset to 0.
5. Meanwhile, honest peers announce their addresses via discovery.
   → check_purge runs; phase 1 finds honest peers with attempts_count >= 3
     (they were feeler-dialed and failed); evicts them; returns Ok.
   → honest address is added, but will itself be evicted in the next cycle.
6. Assert: after N cycles, peer_store.addr_manager.addrs_iter() contains
   zero honest addresses and 16384 attacker addresses.
```
