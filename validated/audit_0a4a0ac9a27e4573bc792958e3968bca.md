### Title
Stale `total_tx_size`/`total_tx_cycles` Accumulator Overwrite After In-Flight Eviction in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new aggregate statistics (`total_tx_size`, `total_tx_cycles`) are computed **before** any in-flight evictions occur, then unconditionally written back **after** those evictions have already correctly decremented the live counters. This overwrites the correct post-eviction state with a stale, inflated value, causing `total_tx_size` and `total_tx_cycles` to diverge upward from reality. The inflated counters then drive `limit_size` to evict additional legitimate transactions from the pool, enabling an unprivileged tx-pool submitter to repeatedly shrink the effective pool below its configured maximum.

---

### Finding Description

`PoolMap::add_entry` follows this sequence:

```
// Step 1 – snapshot pre-eviction totals + new entry
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//   = (S0 + new_size,  C0 + new_cycles)

// Step 2 – may evict N entries via remove_entry_and_descendants,
//           each call to update_stat_for_remove_tx correctly writes:
//           self.total_tx_size  -= evicted_size_i
//           self.total_tx_cycles -= evicted_cycles_i
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3 – OVERWRITES the correctly updated post-eviction counters
self.total_tx_size  = total_tx_size;   // S0 + new_size  (NOT S0 - Σevicted + new_size)
self.total_tx_cycles = total_tx_cycles; // C0 + new_cycles
``` [1](#0-0) 

After Step 2, `self.total_tx_size` holds the correct value `S0 − Σevicted_size + new_size`. Step 3 blindly overwrites it with `S0 + new_size`, inflating the counter by exactly `Σevicted_size`.

The eviction in Step 2 is triggered inside `check_and_record_ancestors` when the incoming transaction's ancestor count exceeds `max_ancestors_count` but can be reduced to within the limit by evicting `cell_ref_parents` — transactions that reference the same cell dep as the new transaction: [2](#0-1) 

Each evicted entry is removed via `remove_entry_and_descendants`, which calls `update_stat_for_remove_tx` and correctly decrements the live counters: [3](#0-2) 

But those decrements are then silently discarded by the unconditional assignment at lines 218–219.

This is structurally identical to the `sUSDz` bug: `totalAssets()` computed `pooledUSDz + vestedAmount` and returned it without persisting the result, so the next operation used the stale `pooledUSDz`. Here, `updated_stat_for_add_tx` computes `S0 + new_size` and stores it in a local, while intermediate mutations correctly update `self.total_tx_size`; the final write discards those mutations.

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further entries:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee entries
}
``` [4](#0-3) 

After each crafted submission that triggers the eviction path, `total_tx_size` is inflated by the byte-size of the evicted entries. `limit_size` then sees a pool that appears larger than `max_tx_pool_size` and evicts additional legitimate pending/proposed transactions. The attacker can repeat this to continuously drain the pool of honest transactions, degrading transaction propagation and block assembly for the targeted node.

Additionally, `total_tx_size` and `total_tx_cycles` are surfaced directly through the `get_tx_pool_info` RPC, so operators and monitoring tools will observe incorrect pool statistics. [5](#0-4) 

---

### Likelihood Explanation

The trigger condition requires:

1. The pool already contains a transaction chain of depth ≥ `max_ancestors_count` (default 25) that shares a cell dep with the attacker's new transaction (`cell_ref_parents` is non-empty).
2. After removing those `cell_ref_parents`, the ancestor count falls at or below `max_ancestors_count`.

An unprivileged tx-pool submitter can construct this scenario by first submitting a chain of 25+ transactions that all reference a common cell dep, then submitting a new transaction that also references that cell dep. No special privilege, key material, or majority hash power is required — only the ability to submit transactions to the node's RPC or P2P interface.

---

### Recommendation

Compute the new aggregate statistics **after** all evictions have completed, not before. Replace the pre-eviction snapshot with a post-eviction increment:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Increment from the already-correct post-eviction counters:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

Alternatively, keep the pre-eviction snapshot approach but subtract the total evicted size/cycles from `total_tx_size`/`total_tx_cycles` before writing back, or recompute via `recompute_total_stat` (which already exists for the underflow recovery path). [6](#0-5) 

---

### Proof of Concept

1. Submit 26 transactions `T1 → T2 → … → T26` where each `Ti` spends an output of `T(i-1)`, and all 26 also reference a shared live cell dep `D` (so all are `cell_ref_parents` of any future transaction that also references `D`).
2. Submit a new transaction `T_new` that spends an output of `T26` **and** references `D` as a cell dep.
3. Inside `add_entry` → `check_and_record_ancestors`:
   - `ancestors_count = 27 > max_ancestors_count (25)`
   - `cell_ref_parents` contains `T1…T26`; `27 - 26 = 1 ≤ 25`, so the eviction branch is taken.
   - One or more `Ti` are evicted; `update_stat_for_remove_tx` correctly decrements `self.total_tx_size`.
4. Lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, inflating it by the size of the evicted entries.
5. Call `get_tx_pool_info` — observe `total_tx_size` exceeds the sum of all remaining entries' sizes.
6. Any subsequent `add_pending` / `add_proposed` call triggers `limit_size`, which evicts legitimate transactions to bring the (falsely inflated) counter below `max_tx_pool_size`.

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
