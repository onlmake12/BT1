### Title
No Per-Peer Admission Limit in OrphanPool Enables Single-Peer Monopolization via Random Eviction - (`tx-pool/src/component/orphan.rs`)

---

### Summary

`OrphanPool::add_orphan_tx` imposes no per-peer quota. A single remote peer can fill all 100 slots of the orphan pool, causing every subsequent orphan submission from any other peer to be evicted with ~99% probability via the pool's random eviction in `limit_size`.

---

### Finding Description

`DEFAULT_MAX_ORPHAN_TRANSACTIONS` is set to 100 and `ORPHAN_TX_EXPIRE_TIME` is `100 * MAX_BLOCK_INTERVAL` — a very long window. [1](#0-0) 

`add_orphan_tx` inserts any structurally valid orphan tx (missing inputs) without checking how many entries the submitting peer already has in the pool: [2](#0-1) 

The `Entry` struct does record the `peer` field, but it is never consulted for admission control — only stored: [3](#0-2) 

When the pool exceeds 100 entries, `limit_size` evicts by taking `self.entries.keys().next()` — the first key in a `HashMap`, which is non-deterministic (hash-order dependent) and provides no per-peer fairness: [4](#0-3) 

The P2P entry path is: `submit_remote_tx` → `resumeble_process_tx` → `after_process` → `add_orphan` → `add_orphan_tx`. The `add_orphan` wrapper in `process.rs` passes the peer index through but performs no per-peer quota check before calling into the pool: [5](#0-4) 

The orphan path is triggered whenever `is_missing_input(reject)` is true after tx verification: [6](#0-5) 

---

### Impact Explanation

An attacker operating a single unprivileged P2P peer submits 100 crafted orphan transactions (structurally valid, referencing non-existent inputs). The pool is now at capacity. Every subsequent orphan tx from any other peer causes the pool to reach 101 entries; `limit_size` then evicts one entry at random. With 100 of 101 entries belonging to the attacker, the victim's tx is evicted with ~99% probability. The attacker can continuously re-fill any evicted slot to maintain monopoly.

Consequences:
- **CPFP chain starvation**: legitimate child-pays-for-parent chains whose parent has not yet been confirmed are permanently excluded from the orphan pool, so they can never be promoted to the pending pool when the parent confirms.
- **Fee-market manipulation**: an attacker can selectively prevent competing high-fee CPFP chains from resolving, giving their own transactions priority in block templates.
- **Targeted tx-pool congestion**: the attack is cheap — 100 minimal-size orphan txs with fabricated input `OutPoint`s suffice, and they never expire during the long `ORPHAN_TX_EXPIRE_TIME` window.

---

### Likelihood Explanation

The attack requires only a standard P2P connection and the ability to craft transactions with missing inputs — no keys, no PoW, no privileged access. The cost is negligible (100 small transactions). The long expiry window means the attacker does not need to continuously re-submit. This is concretely reachable on mainnet by any node operator.

---

### Recommendation

Add a per-peer cap inside `add_orphan_tx`. Before inserting, count how many entries in `self.entries` already belong to `peer` and reject (or evict the oldest from that peer) if the count exceeds `DEFAULT_MAX_ORPHAN_TRANSACTIONS / MAX_PEERS_PER_ORPHAN_SLOT` (e.g., 10–25). Alternatively, during `limit_size` eviction, prefer to evict from the peer with the most entries rather than evicting at random.

---

### Proof of Concept

1. Connect peer A to a CKB node.
2. Peer A submits 100 orphan transactions: each is a structurally valid CKB transaction whose inputs reference non-existent `OutPoint`s (random tx-hash + index). Each passes `non_contextual_verify` but fails resolution with `is_missing_input`.
3. After 100 submissions, `OrphanPool::len()` == 100 (pool is full, no eviction triggered since `len > 100` is false).
4. Connect peer B. Peer B submits 1 orphan transaction (same construction).
5. `add_orphan_tx` inserts it → `len()` == 101 → `limit_size` evicts one entry at random.
6. Assert: peer B's tx hash appears in the `evicted_txs` return value of `add_orphan_tx` with ~99% probability; all 100 of peer A's txs remain.
7. Repeat step 4–6 to confirm peer B is permanently starved.

The `peer` field stored in each `Entry` can be used to verify ownership of each remaining slot after eviction. [7](#0-6)

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }

    pub(crate) async fn add_orphan(
        &self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) {
        let evicted_txs = self
            .orphan
            .write()
            .await
            .add_orphan_tx(tx, peer, declared_cycle);
        // for any evicted orphan tx, we should send reject to relayer
        // so that we mark it as `unknown` in filter
        for tx_hash in evicted_txs {
            self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash });
        }
    }
