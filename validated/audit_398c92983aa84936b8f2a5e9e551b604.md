Audit Report

## Title
Stale `evict_key` on Ancestor Entries After `remove_entry_and_descendants` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` strips all parent/child links for every entry in the removed subtree before calling `remove_entry` on each. Because `update_ancestors_index_key` resolves ancestors through those same links, the surviving ancestors of the root entry never have their `descendants_*` fields or `evict_key` decremented. This leaves those ancestors with stale, inflated eviction keys, corrupting the pool's eviction ordering so that low-fee ancestors survive pool pressure they should not survive while genuinely higher-fee transactions are rejected.

## Finding Description
`remove_entry_and_descendants` (lines 252–265) operates in two phases:

**Phase 1** — `remove_entry_links` is called for every entry in `removed_ids` (root + all descendants). For each entry, `remove_entry_links` (lines 418–430) removes the entry from its parents' children sets, removes it from its children's parents sets, and then calls `self.links.remove(id)` which deletes the entry's own record from `self.links.inner`.

**Phase 2** — `remove_entry` (lines 235–250) is called for each entry. The first substantive call inside `remove_entry` is `self.update_ancestors_index_key(&entry.inner, EntryOp::Remove)` (line 242).

`update_ancestors_index_key` (lines 432–445) calls `self.links.calc_ancestors(&child.proposal_short_id())`. `calc_ancestors` (links.rs line 78) calls `calc_relative_ids`, which calls `self.inner.get(short_id)` to obtain the direct parents. Because Phase 1 already called `self.links.remove(id)` for the root entry, `self.inner.get(root_id)` returns `None`, `direct` is an empty set, and `calc_relation_ids` returns an empty set. The loop in `update_ancestors_index_key` never executes.

Consequently, for every ancestor of the root entry that remains in the pool, `sub_descendant_weight` is never called, so `descendants_count`, `descendants_size`, `descendants_cycles`, `descendants_fee`, and `evict_key` all remain at their pre-removal values.

The comment in Phase 1 acknowledges the intent is to suppress `update_descendants_index_key` (correct — all descendants are being removed). But it silently also suppresses `update_ancestors_index_key`, which is incorrect because the ancestors remain in the pool.

The single-entry `remove_entry` path (lines 235–250) does not have this bug: it calls both `update_ancestors_index_key` and `update_descendants_index_key` before stripping links, so ancestors are correctly updated in that path.

## Impact Explanation
`EvictKey` is computed from `descendants_fee` and `descendants_size/cycles` (entry.rs lines 234–247). A stale, inflated `descendants_fee` makes the ancestor's `evict_key` appear higher (more valuable) than it actually is.

`next_evict_entry` iterates entries in ascending `evict_key` order and picks the first match for a given status (pool_map.rs lines 380–385). `limit_size` calls `next_evict_entry` in a loop until `total_tx_size ≤ max_tx_pool_size` (pool.rs lines 292–329), calling `remove_entry_and_descendants` on each selected entry.

With a stale, inflated `evict_key`, the ancestor is ranked as harder to evict than it deserves. `limit_size` skips it and instead evicts a genuinely higher-fee transaction, issuing `Reject::Full` to that transaction's submitter. An attacker can exploit this to keep a low-fee transaction alive in the pool indefinitely under pool pressure while causing legitimate high-fee transactions from other users to be rejected.

This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Any unprivileged `send_transaction` RPC caller can reach this path:

1. Submit parent tx `P` with a low fee rate. `P` is accepted; `P.descendants_fee = P.fee`.
2. Submit child tx `C1` with a high fee rate spending `P`'s output. `P.descendants_fee` is boosted; `P.evict_key` is inflated.
3. Submit a conflicting tx `C1'` that spends the same input as `C1` with a fee rate exceeding `C1`'s (RBF). `resolve_conflict` calls `remove_entry_and_descendants(&C1_id)`. `P`'s links to `C1` are stripped in Phase 1; `update_ancestors_index_key` for `C1` finds no ancestors; `P.descendants_fee` is never decremented. `P.evict_key` remains inflated.
4. Wait for or cause the pool to fill to `max_tx_pool_size`. `limit_size` runs, skips `P` (inflated `evict_key`), and evicts a higher-fee transaction instead.

RBF is enabled when `min_rbf_rate > min_fee_rate` (pool.rs lines 80–83), a common node configuration. The marginal cost of the attack is the fee premium paid for `C1'` over `C1`, which can be made arbitrarily small (just above the RBF minimum increment). The attack is repeatable and requires no special privileges.

## Recommendation
Before stripping links in `remove_entry_and_descendants`, explicitly update the ancestors of the root entry. The root's ancestors remain in the pool and must have their `descendants_*` fields and `evict_key` decremented. Descendants of the root do not need this treatment because their ancestors are also being removed.

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // NEW: update ancestors of the root entry BEFORE links are torn down,
    // because remove_entry_links will make calc_ancestors return empty.
    if let Some(root_entry) = self.entries.get_by_id(id) {
        let root_inner = root_entry.inner.clone();
        self.update_ancestors_index_key(&root_inner, EntryOp::Remove);
    }

    // strip links so that remove_entry won't re-run update_descendants_index_key
    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

This mirrors the pattern used in the single-entry `remove_entry` path, which correctly calls `update_ancestors_index_key` before removing links.

## Proof of Concept
**Setup:** Insert three entries in a chain `tx1 → tx2 → tx3` (tx2 spends tx1's output, tx3 spends tx2's output), each with `fee=100, size=100`. After insertion, `tx1.descendants_fee = 300` and `tx1.evict_key` reflects a high descendants fee rate.

**Trigger:** Call `pool_map.remove_entry_and_descendants(&tx2_id)` (removes tx2 and tx3).

**Expected:** `tx1.descendants_fee` drops to `100` (only itself); `tx1.evict_key` is recomputed to reflect its actual lone fee rate.

**Actual:** `tx1.descendants_fee` remains `300`; `tx1.evict_key` is unchanged. `tx1` is ranked as if it still has two high-fee descendants.

**Consequence:** When `limit_size` runs and the pool is full, `tx1` is not selected for eviction even though it has the lowest actual fee rate among all pending entries. A different, genuinely higher-fee transaction is evicted instead and its submitter receives `Reject::Full`.

A targeted unit test should be added to `tx-pool/src/component/tests/score_key.rs` asserting that after `remove_entry_and_descendants(&tx2_id)`, `pool_map.get(&tx1_id).unwrap().descendants_fee == tx1.fee` and `pool_map.get_by_id(&tx1_id).unwrap().evict_key == tx1_entry.as_evict_key()`.
