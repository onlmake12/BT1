### Title
Excess ETH Sent to Entropy Request Functions Is Permanently Captured as Protocol Fees Instead of Being Refunded — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` internal function in `Entropy.sol`, which backs all public entry points (`request`, `requestWithCallback`, `requestV2`), credits the **entire** `msg.value - providerFee` to `_state.accruedPythFeesInWei` rather than only the Pyth protocol fee portion. Any ETH sent above the required fee is permanently captured by the contract and never returned to the caller.

---

### Finding Description

In `requestHelper` (lines 233–239 of `Entropy.sol`), fee accounting is performed as follows:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

The expression `SafeCast.toUint128(msg.value) - providerFee` is assigned to `accruedPythFeesInWei`. This means:

- If `msg.value == requiredFee`, the Pyth fee is correctly `requiredFee - providerFee`.
- If `msg.value > requiredFee` (overpayment), the **entire surplus** is silently absorbed into `accruedPythFeesInWei` rather than being refunded to `msg.sender`.

There is no refund path anywhere in `requestHelper`, `request`, `requestWithCallback`, or any `requestV2` overload. The contract's own NatSpec comments acknowledge this explicitly at lines 321 and 345:

> *"Note that excess value is* **not** *refunded to the caller."*

This is the direct analog of the CoverProtocol `_transferRem` omission: a missing "transfer remaining balance" step after each external payable entry point.

---

### Impact Explanation

Any caller who sends more ETH than `getFeeV2(provider, gasLimit)` permanently loses the surplus. The surplus is not held in a recoverable user balance — it is immediately credited to `_state.accruedPythFeesInWei`, which is only withdrawable by the Pyth admin. The user has no mechanism to reclaim it.

Affected entry points:
- `request(address, bytes32, bool)` — line 322
- `requestWithCallback(address, bytes32)` — line 346
- `requestV2()` / `requestV2(uint32)` / `requestV2(address, uint32)` / `requestV2(address, bytes32, uint32)` — lines 286–390

**Impact:** Direct, permanent loss of user ETH. Severity scales with overpayment amount. For integrating contracts that send a conservative buffer (e.g., `fee * 2`) to avoid `InsufficientFee` reverts, the loss is systematic and repeatable across every request.

---

### Likelihood Explanation

- Smart contract integrators commonly send a fee buffer (e.g., `getFee() * 2`) to guard against fee increases between estimation and execution.
- Frontend applications may compute fees with rounding or stale data, causing routine overpayment.
- Users calling directly may send a round ETH amount (e.g., `0.01 ether`) when the actual fee is smaller.
- The behavior is not surfaced as an error — the transaction succeeds silently, making the loss invisible to the caller.

---

### Recommendation

Replace the current fee accounting in `requestHelper` with:

```solidity
uint128 pythFee = requiredFee - providerFee;
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += pythFee;

// Refund excess ETH to caller
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

This mirrors the `_transferRem` pattern recommended in the original report: after every external payable entry point, transfer any remaining balance back to the caller.

---

### Proof of Concept

1. Provider is registered with `feeInWei = 0.001 ether`. Pyth fee is `0.0001 ether`. `getFeeV2(provider, 0)` returns `0.0011 ether`.
2. An integrating contract calls `requestWithCallback{value: 0.01 ether}(provider, userContribution)` — sending 10× the required fee as a safety buffer.
3. Inside `requestHelper`:
   - `requiredFee = 0.0011 ether`
   - `providerFee = 0.001 ether` → credited to provider
   - `accruedPythFeesInWei += 0.01 ether - 0.001 ether = 0.009 ether` ← **entire surplus absorbed**
4. The integrating contract receives no refund. The `0.0089 ether` surplus above the true Pyth fee (`0.0001 ether`) is permanently captured.
5. Repeated across N requests, the integrating contract loses `N × 0.0089 ether` with no recourse. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function requestV2(
        address provider,
        bytes32 userContribution,
        uint32 gasLimit
    ) public payable override returns (uint64) {
        EntropyStructsV2.Request storage req = requestHelper(
            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed
            // random number. Hence, useBlockHash is set to false.
            false,
            true,
            gasLimit
        );

        emit RequestedWithCallback(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );
        emit EntropyEventsV2.Requested(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }
