Audit Report

## Title
Rich-Indexer `insert_transaction` Unconditional `break` Leaves Subsequent Inputs Unspent When Cell Filter Is Active — (`util/rich-indexer/src/indexer/mod.rs`)

## Summary
In `insert_transaction`, when a `cell_filter` is configured, the input-processing loop calls `spend_cell` and unconditionally `break`s if it returns `false`. Because `spend_cell` returns `false` for any cell absent from the `output` table — the normal case for cells excluded by the filter — any multi-input transaction whose first input references a non-indexed cell causes all subsequent inputs to be silently skipped. Indexed cells consumed by those later inputs retain `is_spent = 0` in the persistent DB, causing them to appear as live cells indefinitely.

## Finding Description
In `util/rich-indexer/src/indexer/mod.rs` at lines 228–248, the input loop is:

```rust
if tx_index != 0 {
    for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
        let out_point = input.previous_output();
        if !spend_cell(&out_point, tx).await? {
            break;   // line 232
        }
        // ... build_input_rows, is_tx_matched
    }
}
```

`spend_cell` in `util/rich-indexer/src/indexer/insert.rs` (lines 403–427) issues `UPDATE output SET is_spent = 1 WHERE ...` and returns `Ok(rows_affected > 0)`. It returns `false` whenever the referenced cell is not present in the `output` table — which is the expected outcome for any cell excluded by the `cell_filter`. The `break` then terminates the entire loop. All inputs after the first non-indexed one are never visited, so their corresponding `output` rows (if indexed) are never updated to `is_spent = 1`. This state is committed to the persistent SQL database. No existing guard prevents this: the `cell_filter` path at line 234 is only reached after `spend_cell` succeeds, and the `is_tx_matched` / early-return logic at lines 251–257 is irrelevant to the `is_spent` column already written (or not written) for prior outputs.

## Impact Explanation
This is a correctness defect in CKB's rich-indexer state storage mechanism. The `output.is_spent` column is the sole authoritative signal used by `get_cells` (`output.is_spent = 0` at `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs` line 107) and `get_cells_capacity` to distinguish live from spent cells. Cells whose `is_spent` was never flipped will be returned as live indefinitely — surviving node restarts because the corruption is written to the persistent DB. This constitutes a suboptimal (incorrect) implementation of CKB's state storage mechanism, matching the **Medium (2001–10000 points)** impact category: *Suboptimal implementation of CKB state storage mechanism*.

## Likelihood Explanation
The precondition is that `cell_filter` is set — a documented, supported operator configuration. Once active, any valid block containing a multi-input transaction where the first input's cell is not indexed (e.g., created before the indexer started, or belonging to a different script type than the filter) triggers the bug. No attacker capability beyond submitting or relaying a valid block is required; normal P2P block propagation is sufficient. The condition is easily and repeatedly met in production deployments.

## Recommendation
Replace `break` with `continue` at line 232 of `util/rich-indexer/src/indexer/mod.rs`:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // cell not in indexer — skip, but process remaining inputs
}
```

This preserves existing semantics (non-indexed cells are silently skipped) while ensuring all inputs are visited so that indexed cells are correctly marked spent.

## Proof of Concept
1. Start a CKB node with the rich-indexer and a `cell_filter` matching only cells with script type `S`.
2. Index a block containing `T1` that creates `output[0]` with script `S` (indexed, `is_spent=0`).
3. Construct `T2` with two inputs: `input[0]` spending a cell with script `X` (not in `output` table), `input[1]` spending `T1:output[0]` (indexed).
4. Submit a valid block containing `T2`.
5. Query the DB: `SELECT is_spent FROM output WHERE ...` for `T1:output[0]`.
6. **Expected**: `is_spent = 1`. **Actual**: `is_spent = 0`.
7. Call `get_cells` for script `S`; `T1:output[0]` is returned as a live cell despite being consumed.

This can be exercised as a unit test extending the existing `with_custom_cell_filter` test in `util/rich-indexer/src/tests/insert.rs` by adding a second transaction whose first input references an unindexed cell and whose second input references an indexed cell, then asserting `is_spent = 1` on the indexed output.
