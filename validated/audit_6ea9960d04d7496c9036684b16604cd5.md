### Title
Silent Truncating Cast in `DaoCalculator::calculate_maximum_withdraw` Produces Wrong Withdrawal Capacity Рђћ (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes a `u128` intermediate result and then casts it to `u64` using the silent truncating `as u64` operator instead of the checked `u64::try_from(...)` used everywhere else in the same file. For large DAO deposits, this silently produces a drastically wrong (truncated) withdrawal capacity rather than returning a `DaoError::Overflow`, corrupting both the RPC response and the consensus-critical fee verification path.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO cell:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // Рєљ silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The multiplication is correctly widened to `u128`, but the final `as u64` cast silently discards the upper 64 bits when the result exceeds `u64::MAX`. Every other analogous computation in the same file uses the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern:

- `secondary_block_reward`: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` (miner issuance): `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` (AR increase): `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [2](#0-1) [3](#0-2) [4](#0-3) 

`calculate_maximum_withdraw` is the sole outlier. The overflow condition is:

```
counted_capacity ├Ќ withdrawing_ar / deposit_ar  >  u64::MAX
```

Since `withdrawing_ar` always grows above `deposit_ar` (the accumulation rate is monotonically increasing), any cell whose `counted_capacity` is large enough will trigger this. With `counted_capacity` near `u64::MAX` (РЅѕ 18.4 ├Ќ 10┬╣РЂИ shannons), even a tiny AR increase (e.g., `withdrawing_ar = deposit_ar + 1`) causes the product to exceed `u64::MAX` before division.

The existing unit test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` constructs exactly this scenario (capacity = `18_446_744_073_709_550_000` shannons) and asserts `result.is_err()`, but the production code silently returns a wrong small value instead of an error, meaning the test expectation is inconsistent with the actual code behavior. [5](#0-4) 

---

### Impact Explanation

**Two affected paths:**

**1. RPC path (`calculate_dao_maximum_withdraw`):**
The RPC handler in `rpc/src/module/experiment.rs` calls `calculator.calculate_maximum_withdraw(...)` directly and returns the result to the caller. [6](#0-5) 

For a large DAO deposit, the RPC silently returns a drastically wrong (truncated) capacity value. A user relying on this to construct a withdrawal transaction will build a transaction with an incorrect output capacity.

**2. Consensus-critical verification path:**
`calculate_maximum_withdraw` is called inside `transaction_maximum_withdraw`, which is called by `transaction_fee` in `DaoCalculator`. The `transaction_fee` result is used during transaction verification in `verification/src/transaction_verifier.rs`. [7](#0-6) 

When the truncated `withdraw_counted_capacity` is much smaller than the actual output capacity, `maximum_withdraw.safe_sub(outputs_capacity)` underflows, causing the verifier to reject a valid DAO withdrawal transaction. Conversely, if the truncated value happens to be larger than the output capacity, a transaction that should have been rejected (because the real withdrawal amount is insufficient) may pass fee validation.

---

### Likelihood Explanation

The overflow requires `counted_capacity ├Ќ withdrawing_ar / deposit_ar > u64::MAX`. The AR starts at `10_000_000_000_000_000` (10┬╣РЂХ) and grows with every block. For a cell holding the maximum practical capacity (total CKB supply РЅѕ 3.36 ├Ќ 10┬╣РЂИ shannons), the overflow threshold is reached when `withdrawing_ar > ~5.5 ├Ќ 10┬╣РЂХ`, i.e., when the AR has grown by a factor of ~5.5├Ќ. This requires a very long chain history, making it low-likelihood in the near term but a real risk over the long-term life of the chain. For cells with capacity near `u64::MAX` (which the protocol does not prevent), the overflow occurs with any nonzero AR increase, making it immediately triggerable.

---

### Recommendation

Replace the silent truncating cast with the checked conversion used everywhere else in the file:

```rust
// Before (silent truncation):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (checked, consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [8](#0-7)

### Proof of Concept

**Overflow condition:**

```
counted_capacity  = 18_446_744_073_709_550_000  (near u64::MAX)
deposit_ar        = 10_000_000_000_123_456
withdrawing_ar    = 10_000_000_001_123_456       (AR grew by 1_000_000)

withdraw_counted_capacity (u128)
  = 18_446_744_073_709_550_000 ├Ќ 10_000_000_001_123_456
    / 10_000_000_000_123_456
  РЅѕ 18_446_744_075_554_224_407                   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_446_744_075_554_224_407 mod 2^64
  = 1_844_672_791                                (silently truncated to ~1.8 billion shannons)
```

The returned `withdraw_capacity` is `~1.8 billion shannons` instead of the correct `~18.4 ├Ќ 10┬╣РЂИ shannons`. The `safe_add(occupied_capacity)` at the end does not catch this because the truncated value is well within `u64` range.

The existing test at `util/dao/src/tests.rs:296РђЊ349` already encodes this exact scenario and asserts `result.is_err()`, confirming the intended behavior is to return `DaoError::Overflow` Рђћ which the current `as u64` cast fails to do. [1](#0-0) [5](#0-4)
