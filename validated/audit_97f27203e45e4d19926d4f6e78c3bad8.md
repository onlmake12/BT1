### Title
`updatePriceFeeds` Approaches Block Gas Limit Due to Unbounded Cold SSTORE Writes for Max-Size Subscriptions — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `updatePriceFeeds` function in `Scheduler.sol` calls `_storePriceUpdates`, which writes a full `PriceFeed` struct (5 storage slots) for every subscribed price feed in a single transaction. With `MAX_PRICE_IDS_PER_SUBSCRIPTION = 255`, the first update of a max-size subscription requires approximately 1,275 cold SSTORE operations (~25.5 M gas), which, combined with Pyth parsing overhead, approaches or exceeds the block gas limit on Ethereum mainnet and definitively exceeds it on many other supported chains.

---

### Finding Description

`MAX_PRICE_IDS_PER_SUBSCRIPTION` is set to 255. [1](#0-0) 

Any unprivileged keeper calls `updatePriceFeeds`, which unconditionally calls `_storePriceUpdates` for every price feed in the subscription: [2](#0-1) 

`_storePriceUpdates` writes the entire `PriceFeed` struct to a nested mapping for each feed: [3](#0-2) 

The `PriceFeed` struct stored in `mapping(uint256 => mapping(bytes32 => PythStructs.PriceFeed)) priceUpdates` occupies **5 storage slots** per feed: [4](#0-3) 

- Slot 0: `id` (bytes32)
- Slot 1: `price.price` + `price.conf` + `price.expo` (packed, 20 bytes)
- Slot 2: `price.publishTime` (uint256, cannot pack)
- Slot 3: `emaPrice.price` + `emaPrice.conf` + `emaPrice.expo` (packed)
- Slot 4: `emaPrice.publishTime` (uint256)

**Gas arithmetic for the first update of a 255-feed subscription:**

| Component | Cost |
|---|---|
| Cold SSTOREs: 255 feeds × 5 slots × 20,000 gas | ~25,500,000 gas |
| `parsePriceFeedUpdatesWithConfig` (Pyth parsing + Merkle proofs + calldata) | ~3,000,000 gas |
| `_validateShouldUpdatePrices` (early-exit on first update) | ~2,100 gas |
| Overhead, event emission, keeper payment | ~200,000 gas |
| **Total** | **~28,700,000 gas** |

Ethereum mainnet block gas limit is ~30 M gas. The margin is ~1.3 M gas — easily consumed by calldata for 255 price feeds with Merkle proofs. On chains with lower block gas limits (many EVM-compatible chains Pyth targets), this transaction is **impossible to include in any block**.

There is no batching mechanism; `updatePriceFeeds` must write all feeds atomically in one call: [5](#0-4) 

---

### Impact Explanation

A subscription owner legitimately creates a subscription with 255 price feeds (the protocol-advertised maximum). No keeper can ever successfully call `updatePriceFeeds` for that subscription on gas-constrained chains, because the transaction cannot fit in a block. The subscription's balance is effectively locked: the subscription is active (preventing withdrawal below minimum balance), yet it can never be serviced. The subscription owner must deactivate and withdraw to recover funds, but the protocol's core promise — automated price feed updates — is permanently broken for max-size subscriptions.

---

### Likelihood Explanation

`MAX_PRICE_IDS_PER_SUBSCRIPTION = 255` is the protocol-advertised limit. Any user who creates a subscription near this limit (e.g., a DeFi protocol tracking a large basket of assets) triggers the condition. No adversarial intent is required; this is a normal, expected use case. The condition is deterministic and reproducible. [1](#0-0) 

---

### Recommendation

1. **Reduce `MAX_PRICE_IDS_PER_SUBSCRIPTION`** to a value that keeps worst-case gas well below the minimum block gas limit across all supported chains (e.g., ≤ 50 feeds keeps cold-write cost under ~5 M gas).
2. **Implement partial/batched updates**: allow `updatePriceFeeds` to accept a subset of the subscription's price IDs per call, storing partial results and only emitting `PricesUpdated` once all feeds are refreshed.
3. **Optimize `PriceFeed` storage layout**: pack `id`, `price`, and `emaPrice` more aggressively, or store only the fields that change (price, conf, publishTime) rather than the full struct, reducing slots per feed from 5 to 3.

---

```solidity

bytes32[] memory priceIds = new bytes32[](255);
for (uint i = 0; i < 255; i++) {
    priceIds[i] = bytes32(uint256(i + 1));
}
SchedulerStructs.SubscriptionParams memory params = SchedulerStructs.SubscriptionParams({
    priceIds: priceIds,
    readerWhitelist: new address[](0),
    whitelistEnabled: false,
    isActive: true,
    isPermanent: false,
    updateCriteria: SchedulerStructs.UpdateCriteria({
        updateOnHeartbeat: true,
        heartbeatSeconds: 60,
        updateOnDeviation: false,
        deviationThresholdBps: 0
    })
});
uint256 minBalance = scheduler.getMinimumBalance(255);
uint256 subId = scheduler.createSubscription{value: minBalance}(params);

// 2. Attempt updatePriceFeeds with valid update data for all 255 feeds.
//    On chains with block gas limit < ~29M, the transaction cannot be mined.
//    On Ethereum mainnet, the transaction consumes ~28.7M gas, leaving
//    virtually no margin and failing if calldata pushes it over 30M.
scheduler.updatePriceFeeds{gas: 30_000_000}(subId, updateData); // reverts OOG
```

The root cause is `_storePriceUpdates` performing 255 × 5 = 1,275 cold SSTOREs unconditionally in a single transaction, with no mechanism to split the work across multiple calls. [6](#0-5)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }
