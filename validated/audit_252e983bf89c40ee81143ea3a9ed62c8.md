### Title
Wrong `total_tx_size`/`total_tx_cycles` Accounting in `PoolMap::add_entry` Inflates Pool Statistics After Ancestor-Limit Evictions — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's size and cycle statistics are snapshotted **before** ancestor-limit evictions occur, then unconditionally written back **after** those evictions have already decremented the running totals. This causes `total_tx_size` and `total_tx_cycles` to be permanently inflated by the sizes/cycles of every transaction evicted during the ancestor-limit check, making the pool appear fuller than it actually is and enabling an unprivileged attacker to trigger spurious evictions of legitimate transactions or cause `Reject::Full` errors.

---

### Finding Description

In `PoolMap::add_entry` the following sequence occurs: [1](#0-0) 

**Step 1 — snapshot computed before evictions:**
```rust
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
```
`updated_stat_for_add_tx` simply returns `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` — a snapshot of the pre-eviction totals plus the new entry. [2](#0-1) 

**Step 2 — evictions decrement the live counters:**
```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
```
Inside `check_and_record_ancestors`, when the new transaction's ancestor count exceeds `max_ancestors_count` but some ancestors are "cell-ref parents" (pool transactions that reference a cell the new tx will consume), those ancestors are removed via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which **decrements** `self.total_tx_size` and `self.total_tx_cycles` for each evicted entry. [3](#0-2) [4](#0-3) [5](#0-4) 

**Step 3 — snapshot overwrites the correctly-decremented live counters:**
```rust
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```
This unconditional overwrite discards the decrements performed in Step 2. The final value is `T_before + entry.size` instead of the correct `T_before − evicted_sizes + entry.size`. [6](#0-5) 

**Concrete inflation formula:**

| Variable | Expected | Actual |
|---|---|---|
| `total_tx_size` after add | `T − Σ(evicted.size) + entry.size` | `T + entry.size` |
| Inflation per trigger | — | `Σ(evicted.size)` |

The inflation is permanent until the pool is cleared or the node restarts, because `recompute_total_stat()` is only invoked on underflow, never on inflation. [7](#0-6) 

---

### Impact Explanation

`limit_size()` drives eviction decisions using `self.pool_map.total_tx_size > self.config.max_tx_pool_size`: [8](#0-7) 

With an inflated `total_tx_size`, the pool believes it is over-capacity when it is not. Consequences:

1. **Spurious eviction of legitimate transactions** — `limit_size()` evicts valid pending/proposed transactions that would otherwise remain.
2. **`Reject::Full` for new submissions** — `updated_stat_for_add_tx` returns `Reject::Full` when the (inflated) total would overflow, blocking honest users from submitting transactions.
3. **Persistent degradation** — each attack iteration adds more inflation; the attacker can repeat cheaply to push `total_tx_size` arbitrarily above `max_tx_pool_size`, effectively freezing the pool.

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is reachable by any unprivileged tx-pool submitter (via `send_transaction` RPC or P2P relay). The attacker needs to:

1. Create a live cell `C` on-chain.
2. Submit `N > max_ancestors_count` (default 25) independent transactions to the pool that each reference `C` as a **cell dep** — these become `cell_ref_parents` in the pool's edge map.
3. Submit a new transaction `T_new` that **spends** `C` as an input. The pool counts the 25 cell-ref-parent transactions as ancestors of `T_new`, exceeding the limit, and evicts them.
4. After eviction, `total_tx_size` is inflated by the sum of the 25 evicted transactions' sizes.
5. Repeat from step 2 with fresh transactions to accumulate inflation.

No privileged access, no majority hash power, and no social engineering is required. The only cost is on-chain CKB to create cell `C` and transaction fees for the submitted transactions.

---

### Recommendation

Move the stat snapshot to **after** `check_and_record_ancestors` completes, so it reads the post-eviction `self.total_tx_size`:

```rust
// In PoolMap::add_entry — compute stats AFTER evictions, not before
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Now self.total_tx_size already reflects evictions; add only the new entry
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, after evictions, read `self.total_tx_size` directly and add `entry.size` without using the pre-computed snapshot. A defensive `debug_assert_eq!(self.pool_map.total_tx_size, self.pool_map.recompute_total_stat().map(|(s,_)| s).unwrap_or(0))` after `add_entry` would catch regressions.

---

### Proof of Concept

```
1. Attacker creates cell C on-chain (any live cell they control).

2. Attacker submits 26 independent transactions T1…T26 to the pool,
   each with cell_dep = C.  These are accepted (no ancestor relationship
   among them; each has ancestors_count = 1 ≤ 25).

3. Attacker submits T_new spending C as an input.
   - get_tx_ancenstors(T_new) finds T1…T26 in edges.deps[C] →
     cell_ref_parents = {T1…T26}, ancestors_count = 27.
   - 27 > max_ancestors_count (25), but
     27 − 26 = 1 ≤ 25, so eviction branch is taken.
   - T1…T26 are removed via remove_entry_and_descendants;
     update_stat_for_remove_tx decrements self.total_tx_size by
     Σ size(Ti) ≈ 26 × ~300 bytes ≈ 7 800 bytes.
   - self.total_tx_size = total_tx_size (snapshot) restores the
     pre-eviction value + size(T_new).
   - Net inflation: ~7 800 bytes per iteration.

4. Repeat steps 2–3 with fresh transactions.
   After ~23 000 iterations (180 MB / 7 800 B), total_tx_size
   overflows max_tx_pool_size; all subsequent send_transaction
   calls return Reject::Full.
```

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {

                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
