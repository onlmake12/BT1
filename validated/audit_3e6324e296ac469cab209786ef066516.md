### Title
Missing Post-Verification Fee Rate Re-Check Allows `min_fee_rate` Policy Bypass via High-Cycle Transactions — (File: tx-pool/src/util.rs)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` validates the minimum fee rate using only the serialized transaction size (`tx_size`) as a proxy for weight. After `verify_rtx` in `_process_tx` determines the actual consumed cycles, there is no re-check of the fee rate against the true transaction weight (`max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`). An unprivileged tx-pool submitter can craft a transaction with high cycles but small serialized size to bypass the `min_fee_rate` admission policy by a factor proportional to the cycles-to-size ratio — directly analogous to the missing `minAmountOut` validation in the reported swap adapter.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

The comment explicitly acknowledges this is a "cheap check." [1](#0-0) 

In `tx-pool/src/process.rs`, `_process_tx` calls `check_tx_fee` (via `pre_check`) before running the expensive `verify_rtx`. After `verify_rtx` returns `verified.cycles`, the code immediately creates the entry and submits it — with **no re-check** of the fee rate against the actual weight:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// ... DeclaredWrongCycles check only if declared_cycles was provided ...
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
``` [2](#0-1) 

The entry's actual fee rate — used for sorting and eviction — is computed as:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [3](#0-2) 

where `get_transaction_weight` is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [4](#0-3) 

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [5](#0-4) 

The admission check (size-based) and the eviction/sorting check (weight-based) use fundamentally different metrics. When cycles dominate the weight, the effective fee rate of an admitted transaction can be far below `min_fee_rate`.

---

### Impact Explanation

An attacker submits a transaction with:
- Small serialized size (e.g., 100 bytes) — passes the size-based `check_tx_fee`
- High cycles (e.g., 70,000,000 — the `max_tx_verify_cycles` ceiling)
- Fee set to exactly `min_fee_rate.fee(tx_size)` = 100 shannons (at `min_fee_rate = 1000 shannons/KW`)

After verification:
- Actual weight = `max(100, 70,000,000 × 0.000170571)` ≈ **11,940**
- Actual fee rate = `100 × 1000 / 11,940` ≈ **8 shannons/KW** — roughly **125× below** `min_fee_rate`

The transaction is admitted to the pool despite its effective fee rate being far below the configured minimum. Such transactions occupy pool space, can displace legitimate transactions during eviction (eviction is by actual fee rate, so these are evicted first — but only after they have already consumed pool resources), and can be relayed to peers that also apply `min_fee_rate` enforcement, causing inconsistent pool state across the network.

---

### Likelihood Explanation

The attack is reachable by any unprivileged actor via the `send_transaction` RPC or by a peer relaying a transaction with a declared cycle count. No privileged access, key material, or majority hashpower is required. Crafting a transaction with high cycles but small size is straightforward — a script that loops for many cycles but has a compact body suffices. The bypass factor scales linearly with cycles, making it easy to tune.

---

### Recommendation

After `verify_rtx` returns actual cycles in `_process_tx`, re-validate the fee rate against the true transaction weight before submitting the entry:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Re-check fee rate against actual weight now that cycles are known
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Some((
        Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, actual_min_fee.as_u64(), fee.as_u64())),
        snapshot,
    ));
}

let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

This mirrors the pattern already used for eviction and sorting (`TxEntry::fee_rate`, `AncestorsScoreSortKey`) and closes the gap between the admission check and the actual weight metric.

---

### Proof of Concept

1. Construct a CKB transaction whose lock or type script loops for ~70,000,000 cycles but whose serialized body is ~100 bytes.
2. Set the output capacity so that `inputs_capacity − outputs_capacity = 100 shannons` (fee = 100 shannons).
3. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000 shannons/KW`.
4. `check_tx_fee` computes `min_fee = 1000 × 100 / 1000 = 100 shannons`; the fee equals the minimum — **admission passes**.
5. `verify_rtx` runs the script and returns `verified.cycles ≈ 70,000,000`.
6. No re-check occurs. `TxEntry::new(rtx, 70_000_000, 100_shannons, 100)` is created.
7. The entry's actual fee rate: `weight = max(100, 11940) = 11940`; `fee_rate = 100×1000/11940 ≈ 8 shannons/KW`.
8. The transaction is now in the pool with an effective fee rate **125× below** `min_fee_rate`, bypassing the policy that `check_tx_fee` was intended to enforce.
