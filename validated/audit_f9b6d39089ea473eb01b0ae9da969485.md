### Title
Missing Whitelist Peer Check in `disconnect_peers_in_ip_range` Allows Forced Disconnection of Whitelisted Peers via `set_ban` RPC - (File: `network/src/network.rs`)

### Summary

`NetworkController::disconnect_peers_in_ip_range` iterates all connected peers and disconnects any whose IP falls within a given range, but does not check `peer.is_whitelist` before disconnecting. The two analogous peer-penalty functions in the same file — `ban_session` and `report_session` — both guard against acting on whitelisted peers. Because `disconnect_peers_in_ip_range` is called unconditionally from `NetworkController::ban`, any local RPC caller who invokes `set_ban` with an IP range that covers a whitelisted peer's address will forcibly disconnect that peer, bypassing the whitelist protection the node operator configured.

### Finding Description

`ban_session` and `report_session` both filter out whitelisted peers before taking action:

```rust
// network/src/network.rs:248-251
if let Some(addr) = self.with_peer_registry(|reg| {
    reg.get_peer(session_id)
        .filter(|peer| !peer.is_whitelist)   // ← whitelist guard
        .map(|peer| peer.connected_addr.clone())
})
``` [1](#0-0) [2](#0-1) 

`disconnect_peers_in_ip_range`, called directly from `ban`, has no such guard:

```rust
// network/src/network.rs:1456-1470
fn disconnect_peers_in_ip_range(&self, address: IpNetwork, reason: &str) {
    self.network_state.with_peer_registry(|reg| {
        reg.peers().iter().for_each(|(peer_index, peer)| {
            if let Some(addr) = multiaddr_to_socketaddr(&peer.connected_addr)
                && address.contains(addr.ip())
            {
                let _ = disconnect_with_message(   // ← no is_whitelist check
                    &self.p2p_control,
                    *peer_index,
                    ...
                );
            }
        })
    });
}
``` [3](#0-2) 

`ban` calls this function before writing to the peer store:

```rust

pub fn ban(&self, address: IpNetwork, ban_until: u64, ban_reason: String) {
    self.disconnect_peers_in_ip_range(address, &ban_reason);
    self.network_state.peer_store.lock()
        .ban_network(address, ban_until, ban_reason)
}
``` [4](#0-3) 

The `set_ban` RPC method is the public entry point that calls `ban`: [5](#0-4) 

The `Peer` struct carries the `is_whitelist` flag that is set at connection time and is already used correctly in `ban_session` and `report_session`: [6](#0-5) 

### Impact Explanation

A local RPC caller who issues `set_ban` with an IP range covering a whitelisted peer's address will immediately disconnect that peer. In `whitelist_only` mode, whitelisted peers are the **only** peers the node can connect to; disconnecting them leaves the node fully isolated from the network. Even outside `whitelist_only` mode, the operator's explicit trust relationship with the whitelisted peer is violated. The ban entry written to the peer store does not prevent the whitelisted peer from reconnecting (because `accept_peer` skips the ban check for whitelisted peers), but the repeated forced disconnection can be used to sustain the isolation. [7](#0-6) 

### Likelihood Explanation

The `set_ban` RPC is available to any process with access to the RPC endpoint (default: `127.0.0.1:8114`). Any local RPC user — including scripts, tools, or a compromised local process — can trigger this path without any privileged key. The condition is simply that the supplied IP range overlaps with a whitelisted peer's IP, which is trivially achievable by supplying a broad subnet (e.g., `0.0.0.0/0`).

### Recommendation

Add a `!peer.is_whitelist` guard inside `disconnect_peers_in_ip_range`, consistent with the existing pattern in `ban_session` and `report_session`:

```rust
fn disconnect_peers_in_ip_range(&self, address: IpNetwork, reason: &str) {
    self.network_state.with_peer_registry(|reg| {
        reg.peers().iter().for_each(|(peer_index, peer)| {
            if peer.is_whitelist {
                return; // never disconnect whitelisted peers
            }
            if let Some(addr) = multiaddr_to_socketaddr(&peer.connected_addr)
                && address.contains(addr.ip())
            {
                let _ = disconnect_with_message(...);
            }
        })
    });
}
```

### Proof of Concept

1. Start a CKB node with a whitelisted peer configured:
   ```toml
   [network]
   whitelist_peers = ["/ip4/192.168.1.10/tcp/8115/p2p/<peer_id>"]
   ```
2. Confirm the whitelisted peer is connected via `get_peers`.
3. Call `set_ban` with a range covering the whitelisted peer's IP:
   ```json
   {"method": "set_ban", "params": ["192.168.1.10", "insert", null, null, "test"]}
   ```
4. Observe that the whitelisted peer is immediately disconnected — `get_peers` returns an empty list — despite the whitelist protection that `ban_session` and `report_session` correctly enforce.
5. In `whitelist_only` mode, the node is now fully network-isolated. [3](#0-2) [8](#0-7)
