### Title
Wrong Cell Source Used in DAO Maximum Withdraw RPC Calculation — (`rpc/src/module/experiment.rs`)

### Summary

The `calculate_dao_maximum_withdraw` RPC handler, when called with the `WithdrawingOutPoint` variant, correctly fetches the deposit transaction to obtain `deposit_header_hash` but then **discards the deposit transaction** and instead reads `output` and `output_data` from the **withdrawing (phase-1 prepare) transaction**. This is the direct CKB analog of H-15: the code computes one quantity from source A (deposit tx → `deposit_header_hash`) but then performs the capacity calculation using source B (withdrawing tx → `output`, `output_data`), producing an incorrect maximum-withdraw result for every caller of this RPC path.

---

### Finding Description

In `rpc/src/module/experiment.rs`, the `calculate_dao_maximum_withdraw` implementation handles two variants. The `WithdrawingHeaderHash` branch (lines 246–267) correctly reads both `output` and `output_data` from the **deposit** transaction (`tx` bound to `out_point.tx_hash()`):

```rust
let (tx, deposit_header_hash) = snapshot
    .get_transaction(&out_point.tx_hash()) ...;
let output     = tx.outputs()     .get(out_point.index().into()) ...;
let output_data = tx.outputs_data().get(out_point.index().into()) ...;
```

The `WithdrawingOutPoint` branch (lines 269–297) fetches the same deposit transaction but **immediately discards it** (`_tx`), then reads `output` and `output_data` from the **withdrawing** transaction instead:

```rust
let (_tx, deposit_header_hash) = snapshot          // deposit tx discarded
    .get_transaction(&out_point.tx_hash()) ...;

let (withdrawing_tx, withdrawing_header_hash) = snapshot
    .get_transaction(&withdrawing_out_point.tx_hash()) ...;

let output      = withdrawing_tx.outputs()     .get(withdrawing_out_point.index().into()) ...;
let output_data = withdrawing_tx.outputs_data().get(withdrawing_out_point.index().into()) ...;

calculator.calculate_maximum_withdraw(
    &output,                                        // ← from WITHDRAWING tx, not deposit tx
    core::Capacity::bytes(output_data.len()) ...,   // ← from WITHDRAWING tx, not deposit tx
    &deposit_header_hash,
    &withdrawing_header_hash,
)
``` [1](#0-0) 

Inside `calculate_maximum_withdraw`, the `output` and `output_data_capacity` parameters are used to compute `occupied_capacity` and `counted_capacity`:

```rust
let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
let output_capacity: Capacity = output.capacity().into();
let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [2](#0-1) 

The function is designed to receive the **deposit** cell's output and data. The deposit cell typically carries 0 bytes of data, while the phase-1 prepare cell carries exactly 8 bytes (the deposit block number stored as a little-endian `u64`). Passing the prepare cell's `output_data` inflates `output_data_capacity` by 8 bytes, which inflates `occupied_capacity` by 8 shannons, deflates `counted_capacity` by 8 shannons, and therefore deflates the returned maximum-withdraw amount.

---

### Impact Explanation

Any wallet or client that calls `calculate_dao_maximum_withdraw` with the `WithdrawingOutPoint` kind receives a **systematically underestimated** maximum withdrawal capacity. The wallet then constructs a phase-2 withdrawal transaction whose output capacity is set too low, causing the user to permanently lose the difference (the unrecoverable shannons are left as miner fee). For a standard deposit (0-byte deposit data), the error is `8 × (withdrawing_ar / deposit_ar)` shannons — small per cell but compounding across many deposits or large DAO positions. Additionally, if a wallet sets the output capacity exactly equal to the (wrong) RPC result, the on-chain DAO type script may reject the transaction because the actual maximum withdraw is higher, causing a confusing failure.

---

### Likelihood Explanation

The `WithdrawingOutPoint` variant is the **recommended** path for wallets that have already broadcast a phase-1 prepare transaction, because it does not require the caller to know the withdrawing block hash in advance. Any RPC caller (unprivileged, no special role required) who uses this variant is affected. The entry path is a standard JSON-RPC call to a running CKB node.

---

### Recommendation

In the `WithdrawingOutPoint` branch, read `output` and `output_data` from the **deposit** transaction (`_tx`), not from `withdrawing_tx`. Rename `_tx` to `tx` and index it with `out_point`:

```rust
DaoWithdrawingCalculationKind::WithdrawingOutPoint(withdrawing_out_point) => {
    let (tx, deposit_header_hash) = snapshot          // keep deposit tx
        .get_transaction(&out_point.tx_hash()) ...;

    let output      = tx.outputs()     .get(out_point.index().into()) ...;  // deposit cell
    let output_data = tx.outputs_data().get(out_point.index().into()) ...;  // deposit cell

    let withdrawing_out_point: packed::OutPoint = withdrawing_out_point.into();
    let (_withdrawing_tx, withdrawing_header_hash) = snapshot
        .get_transaction(&withdrawing_out_point.tx_hash()) ...;

    calculator.calculate_maximum_withdraw(
        &output,
        core::Capacity::bytes(output_data.len()) ...,
        &deposit_header_hash,
        &withdrawing_header_hash,
    )
}
```

This makes the `WithdrawingOutPoint` branch consistent with the `WithdrawingHeaderHash` branch, which already correctly uses the deposit cell's output. [3](#0-2) 

---

### Proof of Concept

1. Deposit 1,000,000 CKB into NervosDAO (deposit cell has 0 bytes of data).
2. After some blocks, broadcast a phase-1 prepare transaction. The prepare cell output has the same capacity but 8 bytes of data (the deposit block number).
3. Call `calculate_dao_maximum_withdraw` with `WithdrawingOutPoint` pointing to the prepare cell's out-point.
4. The RPC reads `output_data.len() = 8` from the prepare cell, computes `occupied_capacity` as 8 shannons higher than correct, and returns a maximum-withdraw value that is `8 × (withdrawing_ar / deposit_ar)` shannons lower than the true value.
5. Call `calculate_dao_maximum_withdraw` with `WithdrawingHeaderHash` using the same prepare block hash. This branch reads `output_data.len() = 0` from the deposit cell and returns the correct (higher) value.
6. The two results differ, confirming the inconsistency and the underestimation in the `WithdrawingOutPoint` path. [4](#0-3) [5](#0-4)

    /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }
