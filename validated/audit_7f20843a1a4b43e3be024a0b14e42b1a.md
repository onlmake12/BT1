### Title
Missing Target-Chain Check in `guardian_set_update` Allows Cross-Chain Guardian Set Replay — (`target_chains/solana/programs/core-bridge/src/legacy/processor/governance/guardian_set_update.rs`)

### Summary
The `guardian_set_update` governance handler does not verify that `decree.chain() == SOLANA_CHAIN`, unlike every other governance handler (`upgrade_contract`, `set_message_fee`, `transfer_fees`). A relayer can submit a legitimately-signed guardian set update VAA that was issued for a different Wormhole chain (e.g., Ethereum, chain=2) and execute it on Solana, replacing Solana's active guardian set with one intended for another network.

### Finding Description
`require_valid_governance_vaa` validates only the VAA emitter (must be `GOVERNANCE_CHAIN=1`, `GOVERNANCE_EMITTER=[0..0,4]`) and that the latest guardian set signed it. [6](#0-5) 

It does **not** inspect `decree.chain()`. The three other handlers compensate by adding their own `require_eq!(decree.chain(), SOLANA_CHAIN, ...)` guard in their `constraints` function. The `guardian_set_update` constraints function skips this guard entirely:

```rust
// guardian_set_update.rs – constraints()
fn constraints(ctx: &Context<Self>) -> Result<()> {
    let config = &ctx.accounts.config;
    let vaa = VaaAccount::load(&ctx.accounts.vaa)?;
    let gov_payload = super::require_valid_governance_vaa(config, &vaa)?;

    require_eq!(
        gov_payload.guardian_set_update().unwrap().new_index(),
        config.guardian_set_index + 1,
        CoreBridgeError::InvalidGuardianSetIndex
    );
    // ← NO decree.chain() == SOLANA_CHAIN check here
    Ok(())
}
``` [4](#0-3) 

`SOLANA_CHAIN` is defined as `1`, the same value as `GOVERNANCE_CHAIN`, so the constants are available and the pattern is established. [7](#0-6) 

### Impact Explanation
If a guardian set update VAA is ever issued targeting a specific non-Solana chain (e.g., `decree.chain() = 2` for Ethereum) with a guardian set that differs from the intended Solana set, any relayer can replay it on Solana. The result is that Solana's Core Bridge `Config.guardian_set_index` is incremented and a new `GuardianSet` account is created with the wrong keys. All subsequent VAA verifications on Solana would use this incorrect guardian set, potentially allowing an attacker who controls the new guardian keys to forge arbitrary VAAs — including price updates, further governance actions, and bridge messages.

### Likelihood Explanation
Requires a legitimately-signed guardian set update VAA to exist that (a) targets a non-Solana chain and (b) encodes a guardian set different from Solana's intended one. In practice, most guardian set updates are global, but the Wormhole governance spec permits chain-specific updates. The attack is permissionless once such a VAA exists — any relayer can submit it. The missing check is a clear deviation from the pattern enforced in all sibling handlers.

### Recommendation
Add the same chain guard to `guardian_set_update`'s `constraints` function, immediately after parsing the decree:

```rust
require_eq!(
    gov_payload.guardian_set_update().unwrap().chain(),
    SOLANA_CHAIN,
    CoreBridgeError::GovernanceForAnotherChain
);
```

This mirrors the pattern in `upgrade_contract`, `set_message_fee`, and `transfer_fees`. [8](#0-7) 

### Proof of Concept
1. Obtain (or observe on-chain) a legitimately guardian-signed VAA with `emitter_chain=1`, `emitter_address=[0..0,4]`, action=`GuardianSetUpdate`, `decree.chain()=2` (Ethereum), `new_index = current_solana_index + 1`, and a different guardian key set.
2. Post the VAA to Solana via `post_vaa` / `verify_encoded_vaa_v1`.
3. Call `guardian_set_update` with the posted VAA account.
4. `constraints` passes: emitter is valid, `new_index == current + 1`, no chain check fires.
5. Solana's `Config.guardian_set_index` is incremented; a new `GuardianSet` account is created with the Ethereum-intended keys.
6. All future VAA verifications on Solana now require signatures from the wrong guardian set.

```rust
pub fn require_valid_governance_vaa<'ctx>(
    config: &'ctx Config,
    vaa: &'ctx VaaAccount<'ctx>,
) -> Result<CoreBridgeDecree<'ctx>> {
    // Make sure the VAA was attested for by the latest guardian set.
    let guardian_set_index = match vaa {
        VaaAccount::EncodedVaa(inner) => match inner.as_vaa()? {
            VaaVersion::V1(vaa) => vaa.guardian_set_index(),
        },
        VaaAccount::PostedVaaV1(inner) => inner.guardian_set_index(),
    };
    require_eq!(
        config.guardian_set_index,
        guardian_set_index,
        CoreBridgeError::LatestGuardianSetRequired
    );

    // The emitter must be the hard-coded governance emitter.
    let emitter = vaa.try_emitter_info()?;
    require!(
        emitter.chain == crate::constants::GOVERNANCE_CHAIN
            && emitter.address == crate::constants::GOVERNANCE_EMITTER,
        CoreBridgeError::InvalidGovernanceEmitter
    );

    // Finally attempt to parse the governance decree.
    CoreBridgeGovPayload::try_from(vaa.try_payload().unwrap())
        .map(|msg| msg.decree())
        .map_err(|_| error!(CoreBridgeError::InvalidGovernanceVaa))
}
```
