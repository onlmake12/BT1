Audit Report

## Title
`consecutive_failed` Never Reset on Successful Package Inclusion Causes Premature Block-Template Loop Termination — (`tx-pool/src/component/tx_selector.rs`)

## Summary

In `TxSelector::txs_to_commit`, the `consecutive_failed` counter is initialized once at line 104 and incremented on every size/cycles or ancestor-check failure, but is never reset to zero after a successful package inclusion (lines 207–220). The counter therefore accumulates across successes, causing the selection loop to terminate after 4001 total failures rather than 4001 *consecutive* failures. An attacker flooding the proposed pool with transactions whose `ancestors_size` is just below `size_limit` can exhaust the counter and cause block templates to be severely under-filled, leaving legitimate transactions unselected.

## Finding Description

`consecutive_failed` is initialized once at the top of `txs_to_commit` (line 104) and is never reset. The initial filter (lines 109–111) passes attacker transactions whose `ancestors_size` is just below `size_limit` (e.g., 599,950 ≤ 600,000). After even one legitimate transaction is committed (adding, say, 100 bytes to `size`), each attacker transaction fails the cumulative check at line 152 (`next_size = 100 + 599,950 = 600,050 > 600,000`), incrementing `consecutive_failed`. The ancestor-not-proposed path at lines 184–188 also increments the counter. The success path at lines 207–220 adds entries to the block and calls `update_modified_entries`, but contains no `consecutive_failed = 0` reset. Once `consecutive_failed > MAX_CONSECUTIVE_FAILURES` (4000), the loop breaks unconditionally at lines 158–159, skipping all remaining proposed transactions regardless of whether they individually fit the remaining block space. A new `TxSelector` is constructed on each `package_txs` call (line 542 of `pool.rs`), resetting `failed_txs` but not preventing the attack from repeating each block. The attacker's transactions are never committed (they fail the cumulative size check), so they remain in the proposed pool and continue to trigger the counter on every block template refresh.

## Impact Explanation

Every miner calling `get_block_template` receives a systematically under-filled block template. In the PoC scenario, a 600,000-byte block is filled with only 100 bytes (0.02% utilization) while legitimate transactions remain unselected. Repeated each block, this constitutes a sustained, low-cost mechanism to reduce network throughput and cause transaction backlog. This matches the allowed High impact: **Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

Triggerable by any unprivileged user via the standard `send_transaction` RPC. CKB's two-phase commit means attacker transactions move to the proposed pool after the proposal window (~2 blocks); miners naturally include cheap short-ID proposals. The attacker needs ~4001 transactions with `ancestors_size` in `(size_limit - S, size_limit]`, where `S` is observable from the mempool. The attacker's transactions are never committed, so they pay minimum fees but occupy no block space — cost is low relative to impact. The attack repeats each block at constant cost.

## Recommendation

Add `consecutive_failed = 0;` immediately after the `update_modified_entries` call at line 220 in `tx-pool/src/component/tx_selector.rs`, restoring the intended semantics that the counter tracks *consecutive* failures:

```rust
self.update_modified_entries(&ancestors);
consecutive_failed = 0; // reset: successful inclusion means block is not yet "full"
```

Proposed pool (sorted by fee rate, descending):
  [0]       tx_A (fee_rate=1000, ancestors_size=100)    → COMMITTED; size=100
