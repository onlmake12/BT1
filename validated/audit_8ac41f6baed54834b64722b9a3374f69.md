Based on my analysis of the code, I can trace the full exploit path.

**Tracing the attack path:**

**Step 1: Inbound session setup**

When an inbound peer connects, `SessionState::new()` sets `remote_addr = RemoteAddress::Init(context.session.address.clone())` — the actual TCP address of the connecting peer. [1](#0-0) 

**Step 2: GetNodes handler — no port validation**

When the inbound peer sends `GetNodes{listen_port=Some(80)}`, the handler calls `update_port(80)` unconditionally with no range or well-known-port check: [2](#0-1) 

**Step 3: `update_port()` blindly substitutes the port**

`update_port()` replaces the TCP component of the session's multiaddr with the attacker-supplied value. There is zero validation of the port number: [3](#0-2) 

**Step 4: `is_valid_addr()` only checks IP reachability**

The only guard before storing the address is `is_valid_addr()`, which calls `is_reachable(socket_addr.ip())` — it checks only the IP address, not the port: [4](#0-3) 

**Step 5: Address stored and gossiped**

The constructed address (e.g., `/ip4/1.2.3.4/tcp/80/p2p/...`) passes `is_valid_addr()` and is stored via `peer_store.add_addr()`. It is then returned by `fetch_random_addrs()` and included in `Nodes` messages sent to other peers. The announce path also only filters by `is_valid_addr()`: [5](#0-4) [6](#0-5) 

**Important scope limitation on the IP:**

The IP used is `context.session.address` — the attacker's **actual** TCP connection IP, not an arbitrary spoofed IP. The attacker cannot redirect traffic to a third-party IP they don't control (TCP-level IP spoofing is not feasible for established connections). The attack is limited to the attacker's own IP.

**Contrast with DNS seeding**, which does validate `port != 0` but still does not check for well-known ports: [7](#0-6) 

The discovery protocol has no equivalent check at all.

### Title
Attacker-Controlled `listen_port` in GetNodes Bypasses Port Validation, Storing and Gossiping Arbitrary Ports — (`network/src/protocols/discovery/state.rs`, `network/src/protocols/discovery/mod.rs`)

### Summary
An inbound peer can send `GetNodes{listen_port=Some(N)}` for any port N (including 0, 80, 443, 22, 53). `update_port()` substitutes N into the peer's multiaddr with no validation. `is_valid_addr()` only checks IP reachability, not the port. The resulting address is stored in the peer store and gossiped to other nodes, which then attempt TCP connections to that port on the attacker's IP.

### Finding Description
`RemoteAddress::update_port()` performs a blind substitution of the attacker-supplied `u16` port into the session multiaddr. The subsequent `is_valid_addr()` guard checks only `is_reachable(ip)` — it is entirely port-agnostic. No minimum port check (e.g., `> 1024`), no well-known-port blocklist, and no CKB-specific port range enforcement exists anywhere in the discovery pipeline.

### Impact Explanation
- Other CKB nodes waste outbound connection slots attempting TCP connections to port 80/443/22/53 on the attacker's IP.
- If the attacker's IP hosts a legitimate service on that port (e.g., an HTTP server on port 80), those services receive unsolicited TCP SYN traffic from every CKB node that receives the gossiped address.
- The peer store becomes polluted with non-CKB addresses, degrading the quality of address discovery for the entire network.
- The attacker can also supply `listen_port=Some(0)`, which `is_valid_addr()` also passes (port 0 is non-zero check is absent in discovery, unlike DNS seeding).

### Likelihood Explanation
The attack requires only an inbound TCP connection to any CKB node — no authentication, no stake, no PoW. A single attacker can poison the address tables of all reachable nodes by connecting to a few well-connected peers. The IP is constrained to the attacker's own address, limiting third-party collateral impact, but the connection-slot exhaustion and peer-store pollution effects are fully reachable.

### Recommendation
Add port validation in `update_port()` or in `add_new_addrs()` before storing:
- Reject `port == 0`.
- Reject ports `< 1024` (well-known/privileged range) unless explicitly allowlisted.
- Optionally enforce that the port matches the node's configured CKB listen port range.

### Proof of Concept
1. Connect to a CKB node as an inbound peer from IP `A.B.C.D`.
2. Send `DiscoveryMessage::GetNodes { listen_port: Some(80), count: 1000, version: 0, required_flags: 0 }`.
3. The victim node calls `update_port(80)`, constructing `/ip4/A.B.C.D/tcp/80/p2p/<peer_id>`.
4. `is_valid_addr()` returns `true` (IP is public/reachable).
5. `peer_store.add_addr(/ip4/A.B.C.D/tcp/80/p2p/<peer_id>)` succeeds.
6. `fetch_random_addrs()` returns this address; it is included in `Nodes` messages sent to third peers.
7. Third peers attempt `connect(/ip4/A.B.C.D/tcp/80/p2p/<peer_id>)` — hitting port 80 on `A.B.C.D`.
