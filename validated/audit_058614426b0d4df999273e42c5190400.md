### Title
Missing `BLOCK_INVALID` Status Check in `HeaderAcceptor` Allows Re-processing of Invalid Block Headers - (File: `sync/src/synchronizer/headers_process.rs`)

### Summary

The `HeaderAcceptor::accept()` function in the sync module does not check whether a received header's block has already been marked `BLOCK_INVALID` before proceeding with full header validation. This gap is explicitly acknowledged by a developer `FIXME` comment in the code. An unprivileged peer can exploit this by first causing a block to be marked invalid, then re-sending its header to force the node to re-process it, update the peer's best-known-header state, and trigger unnecessary block download requests — wasting CPU and bandwidth in a repeatable loop.

### Finding Description

`BlockStatus::BLOCK_INVALID` is a one-way, irreversible state flag (bit 12) set when a block fails contextual or script verification. Once set, it is never cleared for that block hash. The `get_block_status` function checks `block_status_map` first, then the database, and returns `BLOCK_INVALID` persistently. [1](#0-0) 

In `HeaderAcceptor::accept()`, the code retrieves the current status of the incoming header's hash and immediately checks for `BLOCK_INVALID` — but only via a `FIXME` comment that acknowledges the check is missing: [2](#0-1) 

The comment at line 301–302 reads:
```
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
```

Because no early return is implemented for `BLOCK_INVALID`, the function falls through to `prev_block_check` (which only checks the *parent's* status, not the header itself), `non_contextual_check` (PoW, number, epoch, timestamp), and `version_check`. A header belonging to a block that failed *contextual* or *script* verification — but whose header fields are individually valid — will pass all three of these checks. The function then calls `sync_shared.insert_valid_header(self.peer, self.header)`, recording the invalid block's header as the peer's best-known header. [3](#0-2) 

The `BLOCK_INVALID` flag is set in `ConsumeUnverifiedBlockProcessor` when block verification fails: [4](#0-3) 

And it is also set during non-contextual block verification in `asynchronous_process_block`: [5](#0-4) 

### Impact Explanation

After `insert_valid_header` is called for an invalid block's header, the node records the peer's best-known header as pointing to a block that will always fail verification. The sync engine may then:

1. Believe the peer has a higher-difficulty chain tip.
2. Issue `GetBlocks` requests for that block and its ancestors.
3. Receive the block, re-verify it, and mark it `BLOCK_INVALID` again.
4. Repeat indefinitely as the attacker re-sends the header.

This constitutes a sustained, repeatable resource-exhaustion loop (CPU for header and block verification, bandwidth for block downloads) triggered by a single unprivileged peer. Additionally, the corrupted best-known-header state for the peer can skew the node's sync-peer selection logic, potentially delaying honest chain synchronization.

### Likelihood Explanation

Any peer that can connect to the node (no privilege required) can:
1. Relay a crafted block whose header is valid (correct PoW, number, epoch, timestamp) but whose body fails contextual verification (e.g., a script that exits non-zero, or an invalid cell reference). The block is marked `BLOCK_INVALID`.
2. Immediately re-send the same block's header via the `SendHeaders` message.
3. The node processes the header through `HeaderAcceptor::accept()`, misses the `BLOCK_INVALID` check, and calls `insert_valid_header`.
4. Steps 2–3 can be repeated at will.

The attacker does not need any keys, stake, or privileged access. The only prerequisite is a valid PoW solution for the crafted block, which is the normal cost of block production on the network.

### Recommendation

**Short term:** Add an explicit early-return guard for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, immediately after retrieving `status`, resolving the documented `FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) { ... }
```

**Long term:** Introduce a dedicated `ValidationError::BlockPreviouslyInvalid` variant so the error is distinguishable from `InvalidParent`, and add a regression test that verifies headers for `BLOCK_INVALID` blocks are rejected without calling `insert_valid_header`.

### Proof of Concept

1. Attacker peer connects to the target node.
2. Attacker mines a block `B` with valid PoW but an invalid body (e.g., a lock script that returns exit code 1). Sends `B` via `SendBlock`.
3. Node verifies `B`, script fails, node sets `block_status_map[B.hash] = BLOCK_INVALID`.
4. Attacker sends `SendHeaders([B.header])`.
5. Node calls `HeaderAcceptor::accept()` for `B.header`:
   - `get_block_status(B.hash)` → `BLOCK_INVALID`
   - Missing guard: does **not** return early
   - `prev_block_check`: parent is valid → passes
   - `non_contextual_check`: PoW/number/epoch/timestamp valid → passes
   - `version_check`: version == 0 → passes
   - `insert_valid_header(peer, B.header)` is called
6. Node now believes the attacker peer's best-known header is `B.header`.
7. Sync engine issues `GetBlocks([B.hash])` to the attacker.
8. Attacker re-sends `B`; node re-verifies, re-marks `BLOCK_INVALID`.
9. Attacker repeats step 4 → loop continues indefinitely. [6](#0-5) [7](#0-6)

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

    pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
        match self.block_status_map().get(block_hash) {
            Some(status_ref) => *status_ref.value(),
            None => {
                if self.header_map().contains_key(block_hash) {
                    BlockStatus::HEADER_VALID
                } else {
                    let verified = self
                        .snapshot()
                        .get_block_ext(block_hash)

                    match verified {
                        None => BlockStatus::UNKNOWN,
                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,
                    }
                }
            }
        }
