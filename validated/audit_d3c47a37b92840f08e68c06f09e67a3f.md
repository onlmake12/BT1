Audit Report

## Title
Unbounded Iteration in `get_raw_tx_pool` RPC Causes Node DoS via Lock Starvation and Memory Exhaustion - (File: `rpc/src/module/pool.rs`)

## Summary
`get_raw_tx_pool` with `verbose=true` unconditionally collects every entry in the tx pool into an unbounded `HashMap` while holding the tx-pool service's async read lock for the entire duration. On a pool near its 180 MB capacity this blocks the tx-pool service message loop, starving `send_transaction` and `get_block_template`, and can exhaust process memory through JSON serialization amplification.

## Finding Description
`get_raw_tx_pool` at `rpc/src/module/pool.rs` L703–718 calls `get_all_entry_info()` or `get_ids()` with no size guard, limit, or pagination. [1](#0-0) 

Both methods in `tx-pool/src/pool.rs` L448–487 call `score_sorted_iter_by_statuses` and `sorted_proposed_iter` and `.collect()` the full result into unbounded `Vec` or `HashMap` structures. [2](#0-1) 

In `tx-pool/src/service.rs` L1000–1013, the async read lock (`service.tx_pool.read().await`) is acquired before the call and is held across the entire synchronous `get_all_entry_info()` collection. Because the tx-pool service processes messages sequentially in a single loop, this synchronous blocking work prevents any other message (including `send_transaction` and `get_block_template`) from being processed until the full collection completes. [3](#0-2) 

The result types `TxPoolIds` and `TxPoolEntryInfo` in `util/types/src/core/tx_pool.rs` L158–176 are plain unbounded `Vec<Byte32>` and `HashMap<Byte32, TxEntryInfo>` with no capacity cap. [4](#0-3) 

The pool is bounded by total serialized byte size (`max_tx_pool_size`, default 180 MB) rather than entry count. [5](#0-4)  With minimum-size transactions the pool can hold hundreds of thousands of entries. JSON serialization (hex-encoding binary fields, adding field-name strings, HashMap overhead) amplifies raw pool size by a large factor, potentially pushing heap allocation into the gigabyte range. No pagination, rate limit, or per-call cap exists anywhere in this call chain; the indexer RPC (`rpc/src/module/indexer.rs`) uses `IndexerPagination` but the pool RPC has no equivalent. [6](#0-5) 

## Impact Explanation
**High – Vulnerabilities which could easily crash a CKB node.** The tx-pool service message loop is blocked for the full duration of each call, directly stalling transaction relay and block assembly. Concurrent calls compound memory pressure. On a memory-constrained node the OS OOM killer may terminate the process. Both effects (service starvation and OOM) constitute a node crash or functional halt.

## Likelihood Explanation
The RPC binds to `127.0.0.1:8114` by default, but production operators (mining pools, exchanges, public RPC providers) routinely expose this port. Filling 180 MB at `min_fee_rate = 1,000` shannons/KB costs approximately 1.8 CKB, an economically trivial precondition. A busy mainnet node may reach a large pool state organically with no attacker-controlled transactions. The attack requires only standard JSON-RPC access, is stateless, and is trivially repeatable.

## Recommendation
Add pagination parameters (`limit` and `after` cursor) to `get_raw_tx_pool`, following the `IndexerPagination` pattern already used in `rpc/src/module/indexer.rs`. Enforce a hard per-call entry cap (e.g., matching the `request_limit = 400` pattern documented for the indexer in `ckb.toml`). Consider offloading the collection to a blocking thread pool (e.g., `tokio::task::spawn_blocking`) so the async service loop is not stalled during the scan.

## Proof of Concept
1. Configure a CKB node with RPC accessible (localhost or exposed port).
2. Submit minimum-size transactions until the pool approaches `max_tx_pool_size` (180 MB). At `min_fee_rate = 1,000` shannons/KB this costs ~1.8 CKB.
3. Issue concurrent JSON-RPC calls: `{"method":"get_raw_tx_pool","params":[true],"id":1,"jsonrpc":"2.0"}`.
4. Observe: node memory climbs as `TxEntryInfo` HashMaps are allocated; concurrent `send_transaction` and `get_block_template` calls stall or time out because the tx-pool service message loop is blocked; the node process may be killed by the OOM killer.
5. Unit-level reproduction: construct a `TxPool` with a large number of mock entries, call `get_all_entry_info()` while a concurrent writer attempts to acquire the write lock, and measure lock-hold duration and allocation size.

        let tx_pool = self.shared.tx_pool_controller();

        let raw = if verbose.unwrap_or(false) {
            let info = tx_pool

            RawTxPool::Verbose(info.into())

            let ids = tx_pool

                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Ids(ids.into())
        };
        Ok(raw)
    }

    pub(crate) fn get_ids(&self) -> TxPoolIds {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| entry.transaction().hash())
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| entry.transaction().hash())
            .collect();

        TxPoolIds { pending, proposed }
    }

    pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let conflicted = self
            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {
            pending,
            proposed,
            conflicted,
        }
    }
