### Title
`pipe()` Syscall Inserts 2 FDs After a Single-Count Guard, Allowing `MAX_FDS + 1` Open Descriptors — (`script/src/scheduler.rs`)

---

### Summary

In `Scheduler::process_message_box`, the `Message::Pipe` handler checks `self.fds.len() >= MAX_FDS` **before** atomically inserting two file descriptors. Because `close()` removes one fd at a time (making the live count odd), a script can reach `fds.len() == 63`, pass the guard, and end with 65 open descriptors — one beyond the intended ceiling of `MAX_FDS = 64`. This is the direct CKB analog of the Venus `_ensureMaxLoops` bug: a limit designed to cap one unit of resource (one fd) is applied before an operation that produces two units, so the actual count can exceed the limit by one.

---

### Finding Description

`MAX_FDS` is declared as 64 and is the sole guard against unbounded file-descriptor creation. [1](#0-0) 

The `Message::Pipe` handler checks the current count, then unconditionally inserts **two** entries: [2](#0-1) 

Each `Fd::create` call produces a read-end and a write-end as a pair: [3](#0-2) 

The `Message::Close` handler removes **one** fd at a time: [4](#0-3) 

Because close is one-at-a-time and pipe is two-at-a-time, the live fd count can become odd. When it reaches exactly 63, the guard `63 >= 64` evaluates to `false`, the pipe call proceeds, and two more fds are inserted — leaving `fds.len() == 65`.

---

### Impact Explanation

A script author can hold 65 simultaneously open file descriptors instead of the protocol-intended 64. If `MAX_FDS` is part of the consensus-enforced script execution rules (as it is for `MAX_VMS_COUNT`), a transaction whose script exploits this path would be accepted by this node but rejected by any node that correctly enforces the limit, producing a **consensus split**. Even absent a consensus split, the resource-accounting invariant `fds.len() ≤ MAX_FDS` is violated, undermining the DoS-protection rationale for the limit.

---

### Likelihood Explanation

The trigger is fully under script-author control and requires only three deterministic steps: create 32 pipes, close one end of any pipe, create one more pipe. No privileged access, no external dependency, and no probabilistic condition is required. Any script submitted via the standard `send_transaction` RPC path can exercise this path.

---

### Recommendation

Replace the single-count pre-check with one that accounts for the two-fd insertion:

```rust
// Before (allows fds.len() to reach MAX_FDS + 1):
if self.fds.len() as u64 >= MAX_FDS { … }

// After (ensures fds.len() never exceeds MAX_FDS after insertion):
if self.fds.len() as u64 + 2 > MAX_FDS { … }
```

This mirrors the correct fix for the Venus bug: the guard must be evaluated against the **post-operation** count, not the pre-operation count.

---

### Proof of Concept

```
Step 1: call pipe() × 32  →  fds.len() = 64   (32 read-ends + 32 write-ends)
Step 2: call close(fd[0]) →  fds.len() = 63   (count is now odd)
Step 3: call pipe()        →  guard: 63 >= 64? NO → insert p1, p2
                           →  fds.len() = 65   (exceeds MAX_FDS = 64)
```

The root cause is identical in structure to the Venus finding: the limit `MAX_FDS` was sized for one unit per operation, but the operation produces two units, so the guard fires one step too late. [2](#0-1) [5](#0-4)
