Audit Report

## Title
Rich-Indexer `insert_transaction` input loop aborts on first filtered-out cell, leaving subsequent indexed cells permanently marked `is_spent=0` — (File: util/rich-indexer/src/indexer/mod.rs)

## Summary
In `AsyncRichIndexer::insert_transaction`, the loop over a transaction's inputs calls `spend_cell` and issues a `break` when it returns `false`. `spend_cell` returns `false` whenever the referenced output is absent from the DB (filtered out by `cell_filter`). Any inputs after the first filtered-out one are never visited, so their `is_spent` flag is never set to `1`, leaving stale live-cell records in the indexer's `output` table permanently.

## Finding Description
`spend_cell` (`insert.rs` L403–427) executes `UPDATE output SET is_spent = 1 WHERE ...` and returns `Ok(rows_affected > 0)`. When the output was excluded by `cell_filter` and is therefore absent from the DB, `rows_affected == 0` and the function returns `Ok(false)`.

The input loop (`mod.rs` L229–248):
```rust
for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
    let out_point = input.previous_output();
    if !spend_cell(&out_point, tx).await? {
        break;   // exits the entire loop
    }
    // build_input_rows for indexed inputs — never reached for i+1, i+2, …
}
```
When input `i` is a filtered-out cell, `break` exits the loop. Inputs `i+1, i+2, …` are never processed. Their corresponding `output` rows retain `is_spent = 0` indefinitely. The corruption survives node restart because it is written to the persistent SQLite/PostgreSQL database.

No existing guard compensates: the `is_cell_filter_enabled` branch after the `break` is unreachable for the skipped inputs, and there is no post-loop reconciliation pass.

## Impact Explanation
The rich-indexer is CKB's optional but widely-used state storage mechanism for UTXO queries. Incorrect `is_spent` state causes `get_cells` and related RPCs to return already-spent cells as live, constituting an incorrect implementation of the CKB state storage mechanism. This maps to **Medium (2001–10000 points): Suboptimal/incorrect implementation of CKB state storage mechanism**.

## Likelihood Explanation
Triggering requires only a standard, valid CKB transaction whose input list contains at least one filtered-out cell before at least one indexed cell — e.g., `inputs = [plain-lock cell (filtered), type-script cell (indexed)]`. No miner privilege, no PoW, no special network position is needed. Any unprivileged user who holds both filtered and indexed cells can trigger this with a routine spend transaction.

## Recommendation
Replace `break` with `continue` so that filtered-out inputs are skipped without aborting processing of subsequent inputs:
```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // was: break
}
```

## Proof of Concept
1. Configure rich-indexer with a `cell_filter` that indexes only cells with a type script.
2. Create Cell A (lock-only, no type script — filtered) and Cell B (has type script — indexed).
3. Submit a transaction: `inputs = [Cell A, Cell B]`, spending both.
4. Wait for the block to be indexed.
5. Query the `output` table for Cell B: `is_spent = 0`.
6. Call `get_cells` for Cell B's type script: it is returned as a live cell despite being spent on-chain.
