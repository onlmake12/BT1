### Title
Stale Orphaned Entries in `conflicts_outputs_cache` After LRU Eviction from `conflicts_cache` — (`tx-pool/src/pool.rs`)

---

### Summary

`TxPool` maintains two coupled caches for rejected conflicting transactions: `conflicts_cache` (capacity 10,000) and `conflicts_outputs_cache` (capacity 30,000). Data is written to **both** caches in `record_conflict`, and `remove_conflict` correctly cleans **both**. However, when the LRU evicts an entry from `conflicts_cache` due to capacity pressure, the corresponding `OutPoint → ProposalShortId` entries in `conflicts_outputs_cache` are **never removed**. This is the direct analog to the NFT burn issue: data registered during a lifecycle event (conflict recording) is not cleaned up during a different lifecycle event (LRU eviction).

---

### Finding Description

`record_conflict` writes to both caches atomically:

```rust
pub(crate) fn record_conflict(&mut self, tx: TransactionView) {
    let short_id = tx.proposal_short_id();
    for inputs in tx.input_pts_iter() {
        self.conflicts_outputs_cache.put(inputs, short_id.clone());
    }
    self.conflicts_cache.put(short_id.clone(), tx);
}
``` [1](#0-0) 

`remove_conflict` correctly cleans both caches when called explicitly:

```rust
pub(crate) fn remove_conflict(&mut self, short_id: &ProposalShortId) {
    if let Some(tx) = self.conflicts_cache.pop(short_id) {
        for inputs in tx.input_pts_iter() {
            self.conflicts_outputs_cache.pop(&inputs);
        }
    }
}
``` [2](#0-1) 

But `conflicts_cache` is an `lru::LruCache` with a hard capacity of 10,000:

```rust
conflicts_cache: LruCache::new(CONFLICTES_CACHE_SIZE),          // 10_000
conflicts_outputs_cache: lru::LruCache::new(CONFLICTES_INPUTS_CACHE_SIZE), // 30_000
``` [3](#0-2) 

When `conflicts_cache` silently evicts an old entry via LRU, **no cleanup hook fires** for `conflicts_outputs_cache`. The `OutPoint → ProposalShortId` mappings for the evicted transaction's inputs remain in `conflicts_outputs_cache` indefinitely, pointing to a `ProposalShortId` that no longer exists in `conflicts_cache`.

The consumer of these caches is:

```rust
pub(crate) fn get_conflicted_txs_from_inputs(
    &self,
    inputs: impl Iterator<Item = OutPoint>,
) -> Vec<TransactionView> {
    inputs
        .filter_map(|input| {
            self.conflicts_outputs_cache
                .peek(&input)
                .and_then(|id| self.conflicts_cache.peek(id).cloned())
        })
        .collect()
}
``` [4](#0-3) 

A stale entry in `conflicts_outputs_cache` causes `peek(id)` on `conflicts_cache` to return `None`, silently dropping the conflict from the result.

---

### Impact Explanation

**Impact: Medium.**

1. **Stale data accumulation**: Each evicted `conflicts_cache` entry leaves behind N stale entries in `conflicts_outputs_cache` (one per input of the evicted tx). A tx with many inputs amplifies this.
2. **Premature eviction of valid entries**: `conflicts_outputs_cache` (capacity 30,000) fills with stale entries, causing valid, live conflict mappings to be evicted prematurely. This silently breaks `get_conflicted_txs_from_inputs` for those inputs.
3. **Incorrect RBF/conflict detection**: `get_conflicted_txs_from_inputs` is used to detect whether an incoming transaction conflicts with a recently-rejected one. Stale entries cause false negatives (missed conflicts) and stale entries cause valid conflicts to be evicted, causing false negatives in the other direction.
4. **No crash or consensus failure**: The pool's primary integrity (`pool_map`) is unaffected. The impact is confined to the conflict-cache layer used for RBF and conflict reporting.

---

### Likelihood Explanation

**Likelihood: Medium.**

An unprivileged tx-pool submitter can trigger this by submitting a stream of transactions that are rejected as conflicting with pool entries. Each rejection calls `record_conflict`, filling `conflicts_cache`. Once 10,000 entries are recorded, every new rejection silently evicts an old entry from `conflicts_cache` while its `conflicts_outputs_cache` entries remain. A moderately active network or a targeted attacker submitting conflicting transactions can reach this threshold. [5](#0-4) 

---

### Recommendation

Override the LRU eviction callback (or use a wrapper) to clean `conflicts_outputs_cache` whenever an entry is evicted from `conflicts_cache`. Alternatively, before inserting a new entry into `conflicts_cache` via `record_conflict`, check if the LRU is at capacity and manually remove the would-be-evicted entry's inputs from `conflicts_outputs_cache` first. The simplest fix is to call `remove_conflict` before `record_conflict` re-inserts a `short_id` that already exists, and to hook eviction.

A minimal approach: replace the bare `lru::LruCache` for `conflicts_cache` with a custom wrapper that calls `conflicts_outputs_cache.pop` for each input of the evicted tx on every `put` that triggers eviction.

---

### Proof of Concept

1. Fill `conflicts_cache` to capacity (10,000 entries) by submitting 10,000 transactions that conflict with pool entries. Each call to `record_conflict` inserts into both caches.
2. Submit one more conflicting transaction. `conflicts_cache.put(...)` silently evicts the oldest entry (e.g., `short_id_0`). `conflicts_outputs_cache` still contains all inputs of `short_id_0` mapped to `short_id_0`.
3. Submit a new transaction whose inputs overlap with the inputs of the evicted `short_id_0` tx. Call `get_conflicted_txs_from_inputs` with those inputs.
4. `conflicts_outputs_cache.peek(&input)` returns `Some(&short_id_0)`. `conflicts_cache.peek(&short_id_0)` returns `None` (evicted). The conflict is silently dropped from the result.
5. Repeat until `conflicts_outputs_cache` (30,000 slots) fills with stale entries, evicting valid live conflict mappings and causing `get_conflicted_txs_from_inputs` to return empty results for inputs that have active conflicts. [6](#0-5)
