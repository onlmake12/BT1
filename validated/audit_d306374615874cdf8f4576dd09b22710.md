### Title
Truncated Price ID Validation in Echo Allows Malicious Fulfiller to Substitute a Different Price Feed — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol` stores only the first 8 bytes (`bytes8`) of each requested price ID to save gas. When `executeCallback` is called, the contract validates the caller-supplied price IDs only against these 8-byte prefixes. After the exclusivity period expires, any unprivileged actor may call `executeCallback`. If two Pyth price feeds share the same 8-byte prefix, a malicious fulfiller can satisfy a request for feed A with valid Pyth update data for feed B, causing the consumer contract to receive and act on incorrect price data.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the full 32-byte `priceIds` supplied by the requester are truncated to 8-byte prefixes before storage:

```solidity
// Copy only the first 8 bytes of each price ID to storage
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

The `EchoState.Request` struct confirms only `bytes8[] priceIdPrefixes` is persisted — the full 32-byte IDs are discarded: [2](#0-1) 

In `executeCallback`, the validation compares only the first 8 bytes of the caller-supplied `priceIds` against the stored prefixes:

```solidity
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
``` [3](#0-2) 

After the exclusivity period (`block.timestamp >= req.publishTime + _state.exclusivityPeriodSeconds`), the provider restriction is lifted and **any address** may call `executeCallback`: [4](#0-3) 

The Pyth contract call then uses the caller-supplied full `priceIds` (not the stored prefixes) to parse update data: [5](#0-4) 

This means: if price feed A and price feed B share the same first 8 bytes, a malicious fulfiller can call `executeCallback` with feed B's full ID and valid Hermes update data for feed B. The 8-byte prefix check passes, the Pyth contract validates feed B's update data against feed B's full ID, and the consumer's `_echoCallback` receives price data for feed B instead of the originally requested feed A.

---

### Impact Explanation

A consumer contract that requested price feed A (e.g., BTC/USD) receives price data for feed B (a different asset whose ID shares the same 8-byte prefix). Any financial logic in the callback — liquidations, swaps, collateral checks — executes against the wrong price. This constitutes incorrect price data delivery to an on-chain consumer, with potential for direct financial loss.

---

### Likelihood Explanation

Exploitation requires two Pyth price feeds to share the same first 8 bytes of their 32-byte IDs. With ~500 current Pyth feeds, the birthday-collision probability over a 64-bit prefix space is approximately `500² / (2 × 2⁶⁴) ≈ 7 × 10⁻¹⁵` — negligible today. However:

- The design flaw is permanent and the risk grows as Pyth adds more price feeds.
- An attacker does not need to find a collision among existing feeds; a future Pyth feed addition could create one.
- After the exclusivity period, the entry path requires no privilege — any EOA can call `executeCallback` with arbitrary `priceIds`.

Likelihood is **low** but non-zero and increasing over time.

---

### Recommendation

Store the full 32-byte price IDs instead of 8-byte prefixes. The gas savings from truncation do not justify the security trade-off. If gas cost is a concern, use a `bytes32[]` stored in a mapping keyed by sequence number rather than inline in the `Request` struct, or use a Merkle commitment over the full price ID set.

---

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback(provider, t, [feedA], gasLimit)`. The contract stores `priceIdPrefixes[0] = feedA[0:8]`.
2. Suppose `feedB[0:8] == feedA[0:8]` for some other Pyth price feed B.
3. After `exclusivityPeriodSeconds` elapses, attacker calls:
   ```solidity
   echo.executeCallback(
       attacker,          // providerToCredit — attacker earns the fee
       sequenceNumber,
       hermesUpdateDataForFeedB,
       [feedB]            // full 32-byte ID of feed B, prefix matches feed A
   );
   ```
4. The prefix check at line 137 passes (`feedB[0:8] == feedA[0:8]`).
5. `parsePriceFeedUpdates` validates feed B's update data against feed B's full ID — succeeds.
6. Alice's consumer `_echoCallback` is invoked with feed B's price instead of feed A's price.
