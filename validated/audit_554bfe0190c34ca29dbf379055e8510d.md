### Title
Cycle Undercounting via `iteration_cycles` Discard in `Scheduler::resume` — (`script/src/scheduler.rs`)

### Summary

`Scheduler::resume` unconditionally resets `iteration_cycles` to `0` after restoring it from `FullSuspendedState`, silently discarding cycles that were legitimately charged by `process_io` (via `suspend_vm`/`resume_vm` VM-eviction) during the iteration that triggered the suspension. A script that forces VM eviction during `process_io` and is then suspended at `CyclesExceeded` will, upon resumption, have those eviction-cost cycles erased from the running total. A node that verifies the same script in one uninterrupted pass counts those cycles; a node that uses the chunked/resumable path does not. The two nodes therefore disagree on the script's total cycle consumption, which is a consensus-breaking invariant violation.

---

### Finding Description

**Three-variable cycle accounting**

The scheduler tracks consumed cycles across three variables whose sum equals total consumption at any instant: `total_cycles` (committed), `iteration_cycles` (accumulated within the current iteration), and `machine.cycles()` (live VM counter). [1](#0-0) 

**`iterate_outer` runs `process_io` even on error**

`iterate_outer` stores the result of `iterate_inner` in a local variable, then unconditionally calls `consume_cycles(self.iteration_cycles)`, clears `iteration_cycles = 0`, and calls `self.process_io()` before propagating the error via `iterate_return?`. [2](#0-1) 

`process_io` calls `ensure_get_instantiated`, which calls `ensure_vms_instantiated`, which calls `suspend_vm`/`resume_vm` when more than `MAX_INSTANTIATED_VMS` (4) VMs are live. Each such call adds `SPAWN_EXTRA_CYCLES_BASE` to `iteration_cycles`. [3](#0-2) 

So after `iterate_outer` returns `Err(CyclesExceeded)`, `iteration_cycles` is **non-zero** (it holds the eviction costs charged by `process_io`).

**`suspend()` faithfully saves the non-zero `iteration_cycles`**

```
FullSuspendedState {
    total_cycles: self.total_cycles.load(Ordering::Acquire),  // does NOT include iteration_cycles
    iteration_cycles: self.iteration_cycles,                   // non-zero eviction cost
    ...
}
``` [4](#0-3) 

The `FullSuspendedState` struct even documents this as a known quirk that must be preserved: [5](#0-4) 

**`resume()` discards the saved `iteration_cycles`**

```rust
let mut scheduler = Self {
    total_cycles: Arc::new(AtomicU64::new(full.total_cycles)),
    iteration_cycles: full.iteration_cycles,   // line 216: restored
    ...
};
scheduler.ensure_vms_instantiated(&full.instantiated_ids).unwrap();
// "We should not charge cycles for them."
scheduler.iteration_cycles = 0;               // line 242: RESET — discards full.iteration_cycles too
``` [6](#0-5) 

The comment's intent is to discard cycles added by `ensure_vms_instantiated` during the resume operation itself (scheduler-internal, non-consensus). But the blanket reset to `0` also throws away `full.iteration_cycles`, which represents real VM-eviction work done during the *previous* execution chunk.

**`consumed_cycles()` only reads `total_cycles`**

```rust
pub fn consumed_cycles(&self) -> Cycle {
    self.total_cycles.load(Ordering::Acquire)
}
``` [7](#0-6) 

The discarded `iteration_cycles` are never added to `total_cycles`, so they are permanently lost from the reported cycle count.

**`chunk_run` is the production resumable path**

```rust
Err(VMInternalError::CyclesExceeded | VMInternalError::Pause) => {
    let snapshot = scheduler.suspend()...;
    Ok(ChunkState::suspended(snapshot))
}
``` [8](#0-7) 

---

### Impact Explanation

Let `Y = SPAWN_EXTRA_CYCLES_BASE × (evictions during process_io)`.

| Verification path | Reported `consumed_cycles` |
|---|---|
| Single uninterrupted pass | `X + Y + Z` |
| Chunked (suspend at `CyclesExceeded`, then resume) | `X + Z` (missing `Y`) |

If the block cycle limit `L` satisfies `X + Z ≤ L < X + Y + Z`, a node using the chunked path accepts the block while a node using the direct path rejects it. This is a **consensus split**: two honest, fully-synced nodes disagree on block validity purely because of their internal verification scheduling.

A script author can reliably engineer `Y` to be large enough to straddle a cycle limit by spawning ≥ 5 VMs (exceeding `MAX_INSTANTIATED_VMS = 4`) so that `process_io` always evicts at least one VM, charging `SPAWN_EXTRA_CYCLES_BASE` per eviction into `iteration_cycles` after the `CyclesExceeded` boundary.

---

### Likelihood Explanation

- Requires only a valid CKB-VM V2 script (unprivileged, submittable via RPC).
- Spawning 5+ VMs is a normal, documented feature of the Meepo hardfork.
- The chunked verification path (`chunk_run`) is the standard path for tx-pool background verification and resumable block validation.
- No privileged access, no key material, no majority hashpower needed.
- The discrepancy is deterministic and reproducible given the same script and the same suspension point.

---

### Recommendation

In `Scheduler::resume`, separate the two concerns:

1. Save `full.iteration_cycles` before calling `ensure_vms_instantiated`.
2. After `ensure_vms_instantiated`, restore `iteration_cycles` to the saved value (not `0`), so that only the cycles added by the resume-time VM instantiation are discarded, while the cycles legitimately accumulated before suspension are preserved.

```rust
let saved_iteration_cycles = full.iteration_cycles;
scheduler.ensure_vms_instantiated(&full.instantiated_ids).unwrap();
// Discard only the cycles charged by ensure_vms_instantiated (scheduler internals),
// but restore the cycles that were pending from the previous chunk's process_io.
scheduler.iteration_cycles = saved_iteration_cycles;
```

---

### Proof of Concept

Differential test (pseudocode):

```
script S:
  spawn VMs 1..5 (exceeds MAX_INSTANTIATED_VMS=4, forces eviction in process_io)
  each VM does pipe I/O to trigger WaitForRead/WaitForWrite states
  root VM loops until near cycle limit

Run A (straight through):
  verifier.run(S, limit = BLOCK_CYCLE_LIMIT)
  → consumed_cycles = X + Y + Z

Run B (forced suspend/resume at midpoint):
  chunk_run(S, limit = midpoint)  → CyclesExceeded → suspend() → FullSuspendedState{total=X, iter=Y}
  chunk_run(S, limit = remainder) → resume() → iteration_cycles reset to 0 → run to completion
  → consumed_cycles = X + Z   (Y is missing)

assert_eq!(Run_A.consumed_cycles, Run_B.consumed_cycles)  // FAILS
```

Set `BLOCK_CYCLE_LIMIT` between `X + Z` and `X + Y + Z` to demonstrate that one path accepts and the other rejects the same transaction.

    fn iterate_outer(
        &mut self,
        pause: &Pause,
        limit_cycles: Cycle,
    ) -> Result<(VmId, Cycle), Error> {
        let iterate_return = self.iterate_inner(pause.clone(), limit_cycles);
        self.consume_cycles(self.iteration_cycles)?;
        let remaining_cycles = limit_cycles
            .checked_sub(self.iteration_cycles)
            .ok_or(Error::CyclesExceeded)?;
        // Clear iteration cycles intentionally after each run
        self.iteration_cycles = 0;
        // Process all pending VM reads & writes. Notice ideally, this invocation
        // should be put at the end of `iterate_inner` function. However, 2 things
        // prevent this:
        //
        // * In earlier implementation of the Meepo hardfork version, `self.process_io`
        // was put at the very start of +iterate_prepare_machine+ method. Meaning we used
        // to process IO syscalls at the very start of a new iteration.
        // * Earlier implementation contains a bug that cycles consumed by suspending / resuming
        // VMs are not updated in the subsequent VM's `current cycles` syscalls.
        //
        // To make ckb-script package suitable for outside usage, we want IOs processed at
        // the end of each iteration, not at the start of the next iteration. We also need
        // to replicate the exact same runtime behavior of Meepo hardfork. This means the only
        // viable change will be:
        //
        // * Move `self.process_io` call to the very end of `iterate_outer` method, which is
        // exactly current location
        // * For now we have to live with the fact that `iteration_cycles` will not always be
        // zero at iteration boundaries, and also preserve its value in `FullSuspendedState`.
        //
        // One expected change is that +process_io+ is now called once more
        // after the whole scheduler terminates, and not called at the very beginning
        // when no VM is executing. But since no VMs will be in IO states at this 2 timeslot,
        // we should be fine here.
        self.process_io()?;
        let id = iterate_return?;
        Ok((id, remaining_cycles))
    }

    // Resume a suspended VM
    fn resume_vm(&mut self, id: &VmId) -> Result<(), Error> {
        if !self.suspended.contains_key(id) {
            return Err(Error::Unexpected(format!("VM {:?} is not suspended!", id)));
        }
        let snapshot = &self.suspended[id];
        self.iteration_cycles = self
            .iteration_cycles
            .checked_add(SPAWN_EXTRA_CYCLES_BASE)
            .ok_or(Error::CyclesExceeded)?;
        let (context, mut machine) = self.create_dummy_vm(id)?;
        {
            let mut sc = context.snapshot2_context.lock().expect("lock");
            sc.resume(machine.inner_mut(), snapshot)?;
        }
        self.instantiated.insert(*id, (context, machine));
        self.suspended.remove(id);
        Ok(())
    }

    // Suspend an instantiated VM
    fn suspend_vm(&mut self, id: &VmId) -> Result<(), Error> {
        if !self.instantiated.contains_key(id) {
            return Err(Error::Unexpected(format!(
                "VM {:?} is not instantiated!",
                id
            )));
        }
        self.iteration_cycles = self
            .iteration_cycles
            .checked_add(SPAWN_EXTRA_CYCLES_BASE)
            .ok_or(Error::CyclesExceeded)?;
        let (context, machine) = self
            .instantiated
            .get_mut(id)
            .ok_or_else(|| Error::Unexpected("Unable to find VM Id".to_string()))?;
        let snapshot = {
            let sc = context.snapshot2_context.lock().expect("lock");
            sc.make_snapshot(machine.inner_mut())?
        };
        self.suspended.insert(*id, snapshot);
        self.instantiated.remove(id);
        Ok(())
    }
