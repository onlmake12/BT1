### Title
Excess `msg.value` Silently Credited to Provider Instead of Refunded to Caller - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.requestPriceUpdatesWithCallback` is a `payable` function that only enforces a minimum fee (`msg.value >= requiredFee`) but never refunds excess ETH to the caller. Any overpayment is silently credited to the provider's accrued fee balance, permanently transferring the user's excess ETH to the provider with no recourse.

### Finding Description
In `Echo.sol`, `requestPriceUpdatesWithCallback` computes a `requiredFee` and checks only that `msg.value` meets the minimum:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
// ...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [1](#0-0) 

`req.fee` stores the entire `msg.value - pythFeeInWei`, not just the provider's fair share. At callback execution time, the provider is credited the full stored fee:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

If a user sends `msg.value = requiredFee + X` (overpayment of `X` wei), the excess `X` is absorbed into `req.fee` and ultimately credited to the provider's `accruedFeesInWei`. There is no refund path and no documentation of this behavior in the contract.

This contrasts with `PythLazer.verifyUpdate`, which explicitly refunds excess:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [3](#0-2) 

The `Entropy.sol` request functions have the same excess-not-refunded pattern but explicitly document it in the interface (`IEntropyV2.sol`):

> "excess value is *not* refunded to the caller" [4](#0-3) 

`Echo.sol` has no such documentation, making this an undisclosed fund-loss behavior.

### Impact Explanation
Any user who overpays when calling `requestPriceUpdatesWithCallback` — due to fee estimation error, a race condition where the provider raises fees between estimation and submission, or a simple mistake — permanently loses the excess ETH to the provider. The provider can then withdraw it via `withdrawAsFeeManager`. The user has no mechanism to recover the funds. [5](#0-4) 

### Likelihood Explanation
Fee estimation is done off-chain before the transaction is submitted. In a volatile gas/fee environment, or when a provider updates their fee between the user's `getFee` call and the transaction landing on-chain, the user's `msg.value` can exceed the actual `requiredFee`. This is a realistic scenario for any active Echo user.

### Recommendation
Add a refund of excess ETH at the end of `requestPriceUpdatesWithCallback`, analogous to `PythLazer.verifyUpdate`:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();

// Refund excess
if (msg.value > requiredFee) {
    (bool refunded, ) = payable(msg.sender).call{value: msg.value - requiredFee}("");
    require(refunded, "Refund failed");
}

// Store only the exact provider fee
req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);
```

Alternatively, document the behavior explicitly (as Entropy does) so integrators are aware.

### Proof of Concept
1. Provider registers with `baseFeeInWei = 0.001 ether`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`.
2. `pythFeeInWei = 1 wei`. `requiredFee = 1 wei + 0.001 ether`.
3. User calls `getFee(provider, gasLimit, priceIds)` off-chain → gets `requiredFee`.
4. Provider raises `baseFeeInWei` to `0.002 ether` in a front-running tx.
5. User's tx lands with `msg.value = 0.001001 ether` (old estimate), which still satisfies the new `requiredFee = 0.002001 ether`? No — in this case the tx reverts. But if the user defensively sends `msg.value = 0.005 ether` to avoid reverts, the excess `~0.003 ether` is silently credited to the provider.
6. `req.fee = 0.005 ether - 1 wei` is stored. At callback, provider receives `req.fee - pythFee_actual`, capturing the user's overpayment. [6](#0-5)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;
