The vulnerability claim is valid. Here is the full analysis:

---

### Title
Unbounded DB Read Amplification via Crafted `difficulty_boundary` in `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The limit guard at line 201 checks `difficulties.len() + last_n_blocks * 2`, but the actual number of blocks processed is `last_block_number - difficulty_boundary_block_number`. An attacker can set `difficulty_boundary` to the total difficulty of block 1 and `last_n_blocks=1`, causing `last_n_numbers` to span nearly the entire chain (N-1 entries), while the guard sees only `2`. Each entry in `last_n_numbers` triggers multiple DB reads including an O(log N) MMR root computation, yielding unbounded read amplification per single unauthenticated P2P request.

---

### Finding Description

**Guard (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `difficulties=[]` and `last_n_blocks=1`, this evaluates to `0 + 2 = 2 ≤ 1000`. The guard passes. [1](#0-0) 

**`difficulty_boundary_block_number` resolution (lines 299–311):**

The binary search `get_first_block_total_difficulty_is_not_less_than(start=0, end=N, min=total_difficulty[1])` returns block 1, so `difficulty_boundary_block_number = 1`. [2](#0-1) 

**Adjustment check (lines 313–315):**

```rust
if last_block_number - difficulty_boundary_block_number < last_n_blocks {
    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
```

With `difficulty_boundary_block_number=1`, `last_block_number=N`, `last_n_blocks=1`: the condition is `N-1 < 1`, which is **false** for any chain with more than 2 blocks. The adjustment is skipped. [3](#0-2) 

**`last_n_numbers` construction (lines 318–319):**

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
```

This produces `(1..N)` — a vector of **N-1 entries**, completely unconstrained by `last_n_blocks` or `GET_LAST_STATE_PROOF_LIMIT`. [4](#0-3) 

**`complete_headers` per-entry cost (lines 132–163):**

For each of the N-1 block numbers, the server performs:
1. `snapshot.get_ancestor(last_hash, number)` — O(log N) or O(N) ancestor traversal
2. `snapshot.get_block(&ancestor_header.hash())` — one DB read
3. `snapshot.chain_root_mmr(number - 1).get_root()` — O(log N) MMR DB reads [5](#0-4) 

---

### Impact Explanation

A single malformed P2P message causes the server to perform O(N · log N) DB reads, where N is the chain height. On mainnet (millions of blocks), this saturates I/O and stalls the node's light-client handler thread, effectively causing a denial of service against any peer running the light-client protocol server. No authentication, PoW, or privileged role is required.

---

### Likelihood Explanation

The attack requires only a valid `last_hash` on the main chain (trivially obtained by syncing one block) and knowledge of `total_difficulty[1]` (public chain data). The crafted message is small and passes all structural validation. Any unprivileged peer can trigger this repeatedly.

---

### Recommendation

Replace the guard with a check on the **actual computed output size**, not the message field `last_n_blocks`. After computing `difficulty_boundary_block_number`, assert:

```rust
let last_n_count = last_block_number - difficulty_boundary_block_number;
if difficulties.len() + last_n_count as usize > GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, cap `difficulty_boundary_block_number` from below so that `last_block_number - difficulty_boundary_block_number ≤ last_n_blocks` always holds before building `last_n_numbers`. [6](#0-5) 

---

### Proof of Concept

```
1. Sync to the chain tip to obtain last_hash and last_block_number = N.
2. Read total_difficulty[1] from the chain (public).
3. Send GetLastStateProof {
       last_hash      = tip_hash,
       start_hash     = genesis_hash,
       start_number   = 0,
       difficulty_boundary = total_difficulty[1],
       difficulties   = [],
       last_n_blocks  = 1,
   }
4. Guard check: 0 + 1*2 = 2 ≤ 1000 → passes.
5. difficulty_boundary_block_number = 1 (binary search).
6. Adjustment: N-1 < 1 → false → skipped.
7. last_n_numbers = (1..N), size = N-1.
8. complete_headers iterates N-1 entries, each doing O(log N) MMR DB reads.
9. Differential: compare response latency for difficulty_boundary=total_difficulty[1]
   vs difficulty_boundary=total_difficulty[N-1]; the former is O(N log N),
   the latter is O(log N).
```
