### Title
Weak On-Chain PRNG Using Manipulable Block Variables as Sole Entropy Source Allows Validator+Provider Collusion to Bias Random Outputs - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `random()` function in `Entropy.sol` is used as the user's contribution to the commit-reveal protocol when callers invoke the no-`userRandomNumber` variants of `requestV2`. It derives its output exclusively from `block.timestamp`, `block.prevrandao`, `msg.sender`, and a publicly readable on-chain `_state.seed`. All four inputs are either publicly observable or validator-influenceable, making the PRNG output predictable and biasable. This is the direct Pyth analog to the `HighEntropyRandomGenerator`/`std::random_device` finding: a function presented as a secure entropy source that does not actually provide cryptographic unpredictability.

---

### Finding Description

`Entropy.sol` exposes three `requestV2` overloads that silently substitute the user's secret contribution with the output of an internal `random()` call:

```solidity
// Entropy.sol lines 286-310
function requestV2() external payable override returns (uint64) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), 0);
}
function requestV2(uint32 gasLimit) external payable override returns (uint64) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), gasLimit);
}
function requestV2(address provider, uint32 gasLimit) external payable override returns (uint64) {
    assignedSequenceNumber = requestV2(provider, random(), gasLimit);
}
``` [1](#0-0) 

The `random()` function itself is:

```solidity
// Entropy.sol lines 1079-1089
function random() internal returns (bytes32) {
    _state.seed = keccak256(
        abi.encodePacked(
            block.timestamp,
            block.prevrandao,
            msg.sender,
            _state.seed
        )
    );
    return _state.seed;
}
``` [2](#0-1) 

Every input to this function is either publicly known or validator-influenceable at the time the transaction is included in a block:

- **`block.prevrandao`**: Post-merge Ethereum RANDAO value. Validators can bias it by choosing to skip their slot (sacrificing the block reward), allowing them to select among two possible RANDAO values for any given slot.
- **`block.timestamp`**: Validators have a small window (~1 second) of discretion over the timestamp they set.
- **`msg.sender`**: Fully known to the attacker before the transaction is included.
- **`_state.seed`**: Stored in contract state (`EntropyState.sol` line 41), publicly readable via `eth_getStorageAt` at any time. [3](#0-2) 

The commit-reveal protocol's security guarantee is that the final random number `r = hash(x_provider, x_user)` is unbiasable as long as **either** the provider or the user is honest. When `random()` replaces the user's secret contribution, the user's side of the protocol is entirely under validator control. The security guarantee then collapses to: the result is unbiasable only if the provider is honest **and** the validator is honest — a conjunction, not a disjunction. A single colluding validator+provider pair breaks the protocol. [4](#0-3) 

---

### Impact Explanation

Any application that calls `requestV2()`, `requestV2(uint32)`, or `requestV2(address, uint32)` — the three most ergonomic and most-documented API entry points — receives a random number whose user-side contribution was generated from fully observable, validator-influenceable state. A colluding validator+provider can:

1. Read `_state.seed` from chain state before the block is proposed.
2. Compute `random()` for all candidate `(block.timestamp, block.prevrandao)` values they can produce.
3. Compute the final random number `r = hash(x_provider, x_user_predicted)` off-chain.
4. Selectively include or exclude the user's transaction, or choose which RANDAO value to reveal, to steer `r` toward a favorable outcome.

Affected downstream applications include NFT mints (trait assignment), on-chain games (dice/loot), and lotteries — all explicitly listed as Pyth Entropy use cases. Users of these applications can have outcomes manipulated without any on-chain evidence of wrongdoing. [5](#0-4) 

---

### Likelihood Explanation

The three no-`userRandomNumber` `requestV2` overloads are the default, zero-argument API variants promoted in all quickstart documentation and the `IEntropyV2` interface. Most integrators will use them. The attack requires a validator who is also (or is colluding with) a registered Entropy provider — a realistic scenario given that staking infrastructure operators and DeFi protocols routinely run both validator nodes and on-chain services. The `block.prevrandao` bias attack has a known, quantified cost (one skipped block reward) and has been demonstrated in practice against other on-chain RNG schemes. [6](#0-5) 

---

### Recommendation

**Short term**: Remove the no-`userRandomNumber` `requestV2` overloads, or replace the `random()` call within them with a commitment to a future blockhash (e.g., `blockhash(block.number + N)`) so that the user's contribution is not known at request time. Alternatively, require callers to always supply a `userRandomNumber`, eliminating the in-contract PRNG path entirely.

**Long term**: If an in-contract PRNG is retained for UX reasons, its inputs must include at least one value that is secret at the time of the request and verifiably unbiasable — for example, a VRF output or a future blockhash committed at request time and revealed later. `block.prevrandao` alone does not satisfy this requirement on any PoS chain where validators can skip slots.

### Proof of Concept

```
1. Deploy Entropy on a testnet. Note the current _state.seed via eth_getStorageAt.

2. As a colluding validator+provider:
   a. Read _state.seed = S from storage.
   b. For the next block you propose, you know block.timestamp = T (your choice ± 1s)
      and you can choose between two block.prevrandao values R0 (reveal) or R1 (skip slot).
   c. Compute candidate_user_contribution_0 = keccak256(T, R0, victim_address, S)
      and candidate_user_contribution_1 = keccak256(T, R1, victim_address, S).
   d. Compute r0 = keccak256(x_provider, candidate_user_contribution_0)
      and r1 = keccak256(x_provider, candidate_user_contribution_1).
   e. Choose whichever of r0/r1 is favorable (e.g., wins the lottery).
   f. Propose the block with the corresponding RANDAO reveal (or skip to get R1).

3. The victim's requestV2() call is included in that block.
   The Entropy contract calls random(), producing the pre-computed user contribution.
   The provider reveals x_provider in the next transaction.
   The final random number matches the attacker's pre-selected outcome.

// Entropy implements a secure 2-party random number generation procedure. The protocol
// is an extension of a simple commit/reveal protocol. The original version has the following steps:
//
// 1. Two parties A and B each randomly sample a contribution x_{A,B} to the random number
// 2. A commits to their number by sharing h_A = hash(x_A)
// 3. B reveals x_B
// 4. A reveals x_A
// 5. B verifies that hash(x_{A}) == h_A
// 6. The random number r = hash(x_A, x_B)
//
// This protocol has the property that the result is random as long as either A or B are honest.
// Honesty means that (1) they draw their value at random, and (2) for A, they keep x_A a secret until
// step 4. Thus, neither party needs to trust the other -- as long as they are themselves honest, they can
