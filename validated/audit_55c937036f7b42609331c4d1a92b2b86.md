### Title
Unvalidated `observed_addr` in IdentifyMessage Enables Network-Wide Address Table Poisoning — (`network/src/protocols/identify/mod.rs`)

### Summary

An unprivileged remote peer can send a crafted `IdentifyMessage` with an attacker-controlled IP in `observed_addr`. The victim stores it without IP validation and re-broadcasts it to every subsequent inbound peer as its own listen address, poisoning peer stores network-wide.

### Finding Description

**Step 1 — Asymmetric validation between `process_listens` and `process_observed`**

`process_listens` applies an `is_reachable` filter before storing remote listen addresses: [1](#0-0) 

`process_observed` applies **no such filter** — it passes the attacker-supplied address directly to `add_observed_addr`: [2](#0-1) 

**Step 2 — `IdentifyCallback::add_observed_addr` stores without IP validation**

The only operation performed is appending the local peer ID if absent. The address is then stored in `NetworkState::observed_addrs` keyed by `SessionId`: [3](#0-2) 

There is no check that the IP in `observed_addr` matches the victim's actual external IP, or that it is reachable, or that it belongs to the victim at all.

**Step 3 — `local_listen_addrs` includes poisoned observed addresses in outgoing identify messages**

When the victim connects to a new peer, `local_listen_addrs()` is called. If the victim has fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses, it pads the list with `observed_addrs`: [4](#0-3) 

**Step 4 — The `connected` handler sends this list to every new peer**

The `is_reachable` filter applied in `connected` does NOT save us — it only filters out private/loopback IPs. A globally routable attacker IP passes through: [5](#0-4) 

**Step 5 — Receiving peers add the poisoned address to their peer store**

`add_remote_listen_addrs` calls `peer_store.add_addr(addr, flags)` for every address in the received listen list, including the poisoned one: [6](#0-5) 

### Impact Explanation

The attacker causes the victim to advertise the attacker's IP (associated with the victim's peer ID) to all subsequent inbound peers. Those peers add the attacker's endpoint to their peer store with `SYNC|RELAY` flags. Nodes attempting to sync with the victim will dial the attacker's endpoint instead of the real one, degrading sync peer availability. At scale, this partitions honest nodes from each other and can cause consensus deviation by preventing block propagation.

### Likelihood Explanation

- Preconditions are minimal: establish a valid TLS/secio session with correct network name and non-zero flags (standard peer connection).
- The victim only needs fewer than 10 public addresses for observed_addrs to be included — common for nodes behind NAT.
- The attacker's globally routable IP passes the `is_reachable` filter.
- No rate limiting or deduplication prevents repeated poisoning across reconnects.

### Recommendation

In `process_observed` (or in `IdentifyCallback::add_observed_addr`), validate that the IP component of the reported `observed_addr` matches the session's actual remote socket address (`session.address`). This is the only address the remote peer can legitimately observe the victim at. Reject or ignore any `observed_addr` whose IP differs from the session's remote IP.

Additionally, apply the same `is_reachable` filter to `observed_addr` that is already applied to `listen_addrs` in `process_listens`.

### Proof of Concept

1. Attacker A connects to victim V with a valid identify handshake (correct network name, non-zero flags).
2. A sends `IdentifyMessage { observed_addr: /ip4/<attacker-ip>/tcp/8115/p2p/<victim-peer-id>, listen_addrs: [], identify: <valid> }`.
3. V stores `/ip4/<attacker-ip>/tcp/8115/p2p/<victim-peer-id>` in `observed_addrs`.
4. Honest node C connects to V. V's `local_listen_addrs()` returns the poisoned address (since V has < 10 public addrs). V sends it to C in the identify message.
5. C's `add_remote_listen_addrs` calls `peer_store.add_addr(/ip4/<attacker-ip>/tcp/8115/p2p/<victim-peer-id>, SYNC|RELAY)`.
6. C subsequently dials the attacker's endpoint when attempting to sync with V.

        let mut listens = self.listen_addrs();

                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
