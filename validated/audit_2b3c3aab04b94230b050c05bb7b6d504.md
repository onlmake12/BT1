### Title
Unauthenticated `from` Field in `ConnectionRequest` Enables State Poisoning and Forced Outbound Connections — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The CKB hole punching protocol's `ConnectionRequestProcess` and `ConnectionSyncProcess` decode a `from` peer ID from the message payload but never verify it against the actual session's peer ID. Any connected peer can claim to be any other peer, poison the `pending_delivered` state map with attacker-controlled listen addresses, and then force the target node to make outbound TCP connections to arbitrary attacker-controlled addresses.

---

### Finding Description

The `ConnectionRequest` molecule message carries a `from` field (the initiating peer's ID). In `ConnectionRequestProcess::execute()`, when the receiving node is the `to` target, it calls `respond_delivered(content.from, &content.to, content.listen_addrs)`: [1](#0-0) 

Inside `respond_delivered`, the node stores the attacker-supplied `from_peer_id` and `remote_listens` (also attacker-supplied) into the `pending_delivered` map: [2](#0-1) 

There is **no check** that `content.from` equals the peer ID of the actual session (`self.peer`) that delivered the message. The `from` field is decoded purely from the message payload: [3](#0-2) 

The same flaw exists in `ConnectionSyncProcess::execute()`. When the node is the `to` target and `content.route` is empty, it looks up `pending_delivered.get(&content.from)` to obtain listen addresses for NAT traversal: [4](#0-3) 

Again, `content.from` is decoded from the message with no verification against the actual session's peer ID: [5](#0-4) 

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`, so spoofing `from` also bypasses per-pair rate limiting for legitimate peers: [6](#0-5) 

The `HolePunching` struct's `pending_delivered` map is defined as `HashMap<PeerId, (Vec<Multiaddr>, u64)>`, keyed by the `from` peer ID: [7](#0-6) 

---

### Impact Explanation

An attacker who is a connected peer can:

1. **Poison `pending_delivered`**: Associate any victim peer ID with attacker-controlled listen addresses in the target node's `pending_delivered` map.
2. **Force outbound TCP connections (SSRF-like)**: By sending a follow-up `ConnectionSync` with `from = victim_peer_id`, the attacker causes the target node to spawn a task making outbound TCP connections to the attacker's addresses via `try_nat_traversal`.
3. **Rate-limit bypass**: By spoofing `from`, the attacker exhausts the `forward_rate_limiter` quota for any legitimate `(from, to)` pair, preventing real hole punching requests from being processed.

The NAT traversal task is spawned unconditionally once valid addresses are found in `pending_delivered`: [8](#0-7) 

---

### Likelihood Explanation

**High.** Any unprivileged peer connected to the target node can send two crafted messages (`ConnectionRequest` then `ConnectionSync`) with arbitrary `from` fields. No special privileges, keys, or majority hashpower are required. The hole punching protocol is enabled by default and reachable from any inbound or outbound P2P session. [9](#0-8) 

---

### Recommendation

In `ConnectionRequestProcess::execute()`, after decoding `content.from`, verify it matches the actual session's peer ID:

```rust
// Resolve the actual peer ID from the session
let actual_peer_id = self.protocol.network_state
    .with_peer_registry(|reg| reg.get_peer(self.peer).map(|p| p.peer_id().clone()));

if actual_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId
        .with_context("from field does not match the actual session peer id");
}
```

Apply the same check in `ConnectionSyncProcess::execute()` for `content.from`.

---

### Proof of Concept

1. Attacker peer **A** establishes a P2P connection to target node **T** (T's peer ID is public).
2. **A** sends a `ConnectionRequest` message with:
   - `from = V` (any arbitrary peer ID, e.g., a well-known node)
   - `to = T` (target node's own peer ID)
   - `listen_addrs = [attacker_controlled_ip:port]`
3. **T** receives the message, sees `self_peer_id == content.to`, and calls `respond_delivered(V, T, [attacker_ip:port])`.
4. **T** stores `V → ([attacker_ip:port], now)` in `pending_delivered` — no session verification occurs.
5. **A** immediately sends a `ConnectionSync` message with `from = V`, `to = T`, empty `route`.
6. **T** receives it, sees `self_peer_id == content.to`, looks up `pending_delivered[V]` = `[attacker_ip:port]`.
7. **T** spawns `try_nat_traversal(self.bind_addr, attacker_ip:port)` — an outbound TCP connection to the attacker's address. [10](#0-9) [11](#0-10)

    pub(crate) async fn execute(mut self) -> Status {
        let content = match RequestContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }
