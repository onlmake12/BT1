### Title
Fee Rate Admission Bypass via Unit Mismatch Between `tx_size` and `weight` in `check_tx_fee` — (File: `tx-pool/src/util.rs`)

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` computes the minimum required fee using raw transaction byte size (`tx_size`) as the weight argument to `FeeRate::fee()`. However, `FeeRate` is defined as **shannons per kilo-weight**, where `weight = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, `weight` can be orders of magnitude larger than `tx_size`, meaning the actual fee rate (in shannons/KW) of an admitted transaction can be far below `min_fee_rate`. This is a direct analog to the reported decimal-unit discrepancy: two quantities with different effective denominators are combined without normalization.

---

### Finding Description

`FeeRate` is defined as shannons per kilo-weight (KW = 1000):

```rust
// util/types/src/core/fee_rate.rs
/// shannons per kilo-weight
pub struct FeeRate(pub u64);

pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;
    Capacity::shannons(fee)
}
```

`weight` is computed by `get_transaction_weight`:

```rust
// util/types/src/core/tx_pool.rs
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

For a cycle-heavy transaction with `cycles = 70_000_000` (the default `max_tx_verify_cycles`) and `tx_size = 200` bytes:
- `weight = max(200, 70_000_000 × 0.000_170_571_4) = max(200, 11_940) = 11_940`

The pool admission check in `check_tx_fee` uses `tx_size` directly instead of `weight`:

```rust
// tx-pool/src/util.rs
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

With `min_fee_rate = 1000` shannons/KW and `tx_size = 200`:
- **Admission threshold**: `1000 × 200 / 1000 = 200 shannons`
- **Correct threshold** (using weight): `1000 × 11_940 / 1000 = 11_940 shannons`

A transaction paying only 200 shannons passes the check, but its actual fee rate is `200 × 1000 / 11_940 ≈ 16.7 shannons/KW` — roughly **60× below** `min_fee_rate`.

The `TxPoolInfo` struct further exposes the unit confusion: it documents `min_fee_rate` as "Shannons per 1000 bytes transaction serialization size" while `FeeRate` is "shannons per kilo-weight" — two different denominators for the same field.

---

### Impact Explanation

An unprivileged tx-pool submitter can craft cycle-heavy transactions (large `cycles`, small serialized `tx_size`) that:

1. **Bypass the `min_fee_rate` admission gate** — the fee check passes because it uses `tx_size` (small) instead of `weight` (large).
2. **Enter the pool with actual fee rates far below `min_fee_rate`** — by a factor of up to ~60–120× for transactions near `max_tx_verify_cycles`.
3. **Pollute the mempool** — these transactions occupy pool slots and are relayed to peers (relay also uses `check_tx_fee`), spreading the pollution network-wide.
4. **Displace legitimate transactions** — pool eviction is based on actual fee rate (using weight), so these low-actual-fee-rate transactions will eventually be evicted, but only after consuming pool resources and relay bandwidth.

The `min_fee_rate` anti-spam mechanism is the primary defense against mempool flooding. Bypassing it undermines DoS resistance for the tx-pool and relay subsystem.

---

### Likelihood Explanation

- Reachable by any unprivileged user via `send_transaction` RPC or P2P relay.
- No special privileges, keys, or majority hashpower required.
- The exploit is deterministic: craft a transaction with high declared cycles and small serialized size, pay fee just above `min_fee_rate × tx_size / 1000`.
- The `max_tx_verify_cycles` limit (70M by default) bounds the maximum bypass ratio but does not prevent the attack.
- The attack is cheap: the attacker pays ~60–120× less than the intended minimum fee per unit of block resource consumed.

---

### Recommendation

Replace the `tx_size`-based cheap check in `check_tx_fee` with the actual `weight`-based check:

```rust
// tx-pool/src/util.rs
let weight = get_transaction_weight(tx_size, cycles); // cycles must be passed in
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

If cycles are not yet available at the point of `check_tx_fee`, the function signature should be updated to accept `cycles` as a parameter, or the check should be deferred until cycles are known. The comment acknowledging the approximation should be removed once the correct check is in place.

Additionally, align the documentation of `min_fee_rate` across `TxPoolInfo`, config comments, and the `estimate_fee_rate` RPC to consistently use "shannons per kilo-weight" rather than "shannons per kilobyte."

---

### Proof of Concept

**Setup**: Node with default config (`min_fee_rate = 1000` shannons/KW, `max_tx_verify_cycles = 70_000_000`).

**Craft transaction**:
- Serialized size: ~200 bytes (minimal transaction)
- Declared cycles: 11_940_000 (so `weight = max(200, 11_940_000 × 0.000_170_571_4) = max(200, 2_036) = 2_036`)
- Fee: 201 shannons (just above `1000 × 200 / 1000 = 200`)

**Admission check** (`check_tx_fee`):
- `min_fee = 1000 × 200 / 1000 = 200 shannons`
- `fee = 201 >= 200` → **PASSES**

**Actual fee rate**:
- `FeeRate::calculate(201, 2036) = 201 × 1000 / 2036 ≈ 98 shannons/KW`
- This is ~10× below `min_fee_rate = 1000 shannons/KW`

**At maximum cycles** (`cycles = 70_000_000`, `weight = 11_940`):
- Required fee to pass check: 201 shannons
- Actual fee rate: `201 × 1000 / 11_940 ≈ 16.8 shannons/KW` — ~60× below `min_fee_rate`

use crate::core::Capacity;

/// shannons per kilo-weight
#[derive(Clone, Copy, Default, Debug, PartialEq, Eq, PartialOrd, Ord)]
pub struct FeeRate(pub u64);

const KW: u64 = 1000;

impl FeeRate {
    /// Calculates the fee rate from a total fee and weight.
    pub fn calculate(fee: Capacity, weight: u64) -> Self {
        if weight == 0 {
            return FeeRate::zero();
        }
        FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
    }

    /// Creates a fee rate from shannons per kilo-weight.
    pub const fn from_u64(fee_per_kw: u64) -> Self {
        FeeRate(fee_per_kw)
    }

    /// Returns the fee rate as shannons per kilo-weight.
    pub const fn as_u64(self) -> u64 {
        self.0
    }

    /// Creates a zero fee rate.
    pub const fn zero() -> Self {
        Self::from_u64(0)
    }

    /// Calculates the fee for a given weight.
    pub fn fee(self, weight: u64) -> Capacity {
        let fee = self.0.saturating_mul(weight) / KW;
        Capacity::shannons(fee)
    }
