### Title
Unprivileged First-Caller Can Permanently Brick Publisher Registration via Arbitrary Authority in `Initialize` â€” (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

---

### Summary

The `Initialize` instruction accepts an arbitrary `authority` pubkey from instruction data with no validation, and has no access control beyond requiring the payer to be a signer. Any unprivileged account can race to call `Initialize` before the legitimate operator, storing an unusable authority (e.g., a PDA of another program) in the singleton config account. Because the config is write-once and there is no `UpdateAuthority` instruction, `InitializePublisher` is permanently bricked.

---

### Finding Description

`initialize` in `processor/initialize.rs` calls `validate_payer`, which only checks `is_signer` and `is_writable` â€” no privileged key is required: [1](#0-0) 

The `args.authority` field is a raw `[u8; 32]` passed directly from instruction data and stored without any validation: [2](#0-1) 

`accounts::config::create` only guards against re-initialization (non-zero `format`), and blindly writes whatever bytes are supplied as `authority`: [3](#0-2) 

The config account is a singleton PDA. Once `format` is set to `FORMAT`, any subsequent call returns `AlreadyInitialized`: [4](#0-3) 

`validate_authority`, called by every `InitializePublisher` execution, unconditionally requires `is_signer` on the stored authority key: [5](#0-4) 

A PDA (off-curve point, no private key) can never satisfy `is_signer` in a direct transaction. CPI signing is only possible from the PDA's owning program, and `pyth-price-store` has no such CPI path. There is no `UpdateAuthority`, `Reinitialize`, or any other instruction that can overwrite the stored authority. [6](#0-5) 

---

### Impact Explanation

The entire publisher registration subsystem is permanently disabled. No publisher config account can ever be created, so no publisher can ever submit prices through this program instance. The only remediation would be redeploying the program (if upgrade authority is retained) and re-initializing.

---

### Likelihood Explanation

The attack window is the gap between program deployment and the operator's `Initialize` call. On Solana, all pending transactions are visible; an attacker can monitor the mempool for the deployment transaction and immediately submit a front-running `Initialize` with a PDA authority. The attacker needs no funds beyond transaction fees and no privileged access.

---

### Recommendation

Add an access-control check to `Initialize` so only a designated privileged key (e.g., the program's upgrade authority, or a hardcoded pubkey) can call it. Additionally, validate that `args.authority` is an on-curve point (i.e., a valid Ed25519 pubkey) using `Pubkey::is_on_curve()` before storing it, to prevent PDA bytes from being stored as the authority.

pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)

pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);
    }
    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
    if data.format != 0 {
        return Err(ReadAccountError::AlreadyInitialized);
    }
    data.format = FORMAT;
    data.authority = authority;
    Ok(data)
