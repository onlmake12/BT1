### Title
Keeper Can Drain Subscription Balances by Inflating `tx.gasprice` in `_processFeesAndPayKeeper` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.sol` (Pyth Pulse) contract reimburses keepers for gas costs using the live `tx.gasprice` opcode, including a fixed `GAS_OVERHEAD` multiplier. Because `tx.gasprice` is entirely attacker-controlled, a malicious keeper can submit an `updatePriceFeeds` transaction with an artificially inflated gas price to extract `GAS_OVERHEAD * tx.gasprice` from a subscription's balance in a single call, draining it far beyond the actual gas cost incurred.

---

### Finding Description

In `_processFeesAndPayKeeper`, the keeper fee is computed as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

The entire `totalKeeperFee` is then transferred to `msg.sender`:

```solidity
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [2](#0-1) 

The keeper's actual network cost is `(startGas - gasleft()) * tx.gasprice`. The contract additionally reimburses `GAS_OVERHEAD * tx.gasprice`. Since `tx.gasprice` is set by the transaction sender (the keeper), the keeper's net extraction from the subscription per call is:

```
profit = GAS_OVERHEAD * tx.gasprice + keeperSpecificFee
```

There is no cap on `tx.gasprice` in the contract. The only guard is:

```solidity
if (status.balanceInWei < totalKeeperFee) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [3](#0-2) 

So the keeper can set `tx.gasprice` to exactly drain the entire remaining subscription balance in one transaction.

The `updatePriceFeeds` entry point is permissionless — anyone can call it:

> "Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`." [4](#0-3) 

---

### Impact Explanation

A malicious keeper can drain the entire ETH balance of any active subscription in a single `updatePriceFeeds` call by setting `tx.gasprice` to `subscription.balanceInWei / (gasUsed + GAS_OVERHEAD)`. The subscription manager loses all deposited funds. On EIP-1559 chains, the attacker can set an arbitrarily high `maxPriorityFeePerGas`; if the attacker is also the block proposer (MEV), they capture the tip themselves and net the full `GAS_OVERHEAD * tx.gasprice` from the subscription. Even without being the block proposer, the `GAS_OVERHEAD` portion is pure profit extracted from the subscription.

---

### Likelihood Explanation

The `updatePriceFeeds` function is permissionless and publicly callable. Any actor who can submit a valid price update (i.e., any keeper) can exploit this. The attack requires no special privileges, no flashloan, and no coordination — only the ability to set a high gas price in a transaction, which is a standard EVM capability. The attack is profitable whenever `GAS_OVERHEAD * tx.gasprice > actual_gas_cost_of_inflated_tip`, which is trivially achievable.

---

### Recommendation

Cap the gas price used in the reimbursement calculation to a governance-controlled maximum, or use a time-weighted average gas price (e.g., from a TWAP oracle or EIP-1559 base fee) rather than the raw `tx.gasprice`. For example:

```solidity
uint256 effectiveGasPrice = tx.gasprice < _state.maxGasPriceInWei
    ? tx.gasprice
    : _state.maxGasPriceInWei;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

Alternatively, use `block.basefee` (EIP-1559 base fee) as the gas price reference, since it cannot be manipulated by the transaction sender.

---

### Proof of Concept

1. Attacker identifies an active subscription with balance `B` wei.
2. Attacker calls `scheduler.updatePriceFeeds(subscriptionId, validUpdateData)` with `tx.gasprice = P` where `P` is chosen so that `(gasUsed + GAS_OVERHEAD) * P ≈ B`.
3. Inside `_processFeesAndPayKeeper`, `gasCost = (gasUsed + GAS_OVERHEAD) * P ≈ B`.
4. The contract transfers `≈ B` wei to `msg.sender` (the attacker).
5. The attacker's actual network cost was only `gasUsed * P` (paid to the block proposer/burned). The attacker profits `GAS_OVERHEAD * P` from the subscription.
6. The subscription balance is fully drained. The legitimate subscription manager loses all deposited funds. [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
