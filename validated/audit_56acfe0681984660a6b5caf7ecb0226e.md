### Title
`MAX_DEPOSIT_LIMIT` Bypass for Permanent Subscriptions via Incremental Deposits — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` checks only the single-deposit amount (`msg.value`) against `MAX_DEPOSIT_LIMIT` rather than the resulting cumulative balance (`status.balanceInWei + msg.value`). This allows any user with a permanent subscription to bypass the cap by making multiple deposits, each individually at or below the limit, accumulating a total balance far exceeding `MAX_DEPOSIT_LIMIT`.

---

### Finding Description

The vulnerability class from the reference report is a **cap/limit bypass due to a stale or incomplete balance check before an accounting update**. In the original report, `totalTokenXBalance()` was read before the transfer completed, so the check used a stale value. In Pyth's `Scheduler.sol`, the analogous flaw is that the cap check reads only `msg.value` (the incremental deposit) rather than `status.balanceInWei + msg.value` (the post-deposit total).

In `createSubscription`, the initial deposit is bounded correctly in isolation:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
...
status.balanceInWei = msg.value;
``` [1](#0-0) [2](#0-1) 

In `addFunds`, the same per-deposit check is applied, but the balance is **cumulative**:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
``` [3](#0-2) 

`MAX_DEPOSIT_LIMIT` is defined as `100 ether`: [4](#0-3) 

Because the check is `msg.value > MAX_DEPOSIT_LIMIT` (not `status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT`), a user can:

1. Call `createSubscription` with `msg.value = 100 ether` → `status.balanceInWei = 100 ether` ✓
2. Call `addFunds` with `msg.value = 100 ether` → check: `100 ether > 100 ether` = **false** → passes → `status.balanceInWei = 200 ether`
3. Repeat indefinitely.

The `withdrawFunds` function explicitly blocks withdrawals from permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [5](#0-4) 

So any ETH deposited beyond the cap is **permanently locked** in the contract with no recovery path.

---

### Impact Explanation

A user can lock an unbounded amount of ETH into a permanent subscription, far exceeding the protocol's intended `MAX_DEPOSIT_LIMIT` of 100 ETH. Since permanent subscriptions prohibit withdrawals, the excess ETH is permanently trapped. This undermines the deposit cap's purpose (protecting users from accidentally locking large sums) and can result in irreversible loss of user funds beyond the intended limit.

---

### Likelihood Explanation

High. The exploit requires no special privileges — any user who creates a permanent subscription can call `addFunds` repeatedly. There is no reentrancy required; the flaw is a straightforward logic error in the cap check. The test suite confirms the per-deposit check exists but does not test the cumulative bypass scenario. [6](#0-5) 

---

### Recommendation

Change the `addFunds` check from a per-deposit check to a post-deposit cumulative check:

```diff
- if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
+ if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
      revert SchedulerErrors.MaxDepositLimitExceeded();
  }
```

Apply the same fix to `createSubscription` for consistency (though the initial deposit sets `balanceInWei = msg.value`, so the current check there is equivalent — but using the cumulative form is clearer and future-proof).

---

### Proof of Concept

```solidity
// Attacker creates a permanent subscription with MAX_DEPOSIT_LIMIT
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);

// Each addFunds call passes the check (msg.value == MAX_DEPOSIT_LIMIT, not >)
scheduler.addFunds{value: 100 ether}(subId); // balance = 200 ETH
scheduler.addFunds{value: 100 ether}(subId); // balance = 300 ETH
// ... repeat N times

// Funds are permanently locked; withdrawFunds reverts for permanent subscriptions
scheduler.withdrawFunds(subId, 1 ether); // reverts: CannotUpdatePermanentSubscription
``` [7](#0-6) [8](#0-7)

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
```
