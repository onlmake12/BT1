### Title
Division-by-Zero DoS in `calculateTwap` Due to Missing Strict Slot Ordering Validation — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

The EVM `Pyth.sol` implementation of `parseTwapPriceFeedUpdates` contains a missing strict inequality check on `publishSlot` in `validateTwapPriceInfo`. When a caller submits two valid Wormhole-signed `TwapPriceInfo` messages with identical `publishSlot` values, `calculateTwap` computes `slotDiff = 0` and then performs three integer divisions by zero, causing an unconditional `Panic(0x12)` revert. This is a denial-of-service on the TWAP price feed path for any message pair sharing a slot.

---

### Finding Description

**Root cause — missing `>=` guard in `validateTwapPriceInfo`:**

```solidity
// Pyth.sol L604
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

The check only rejects `startSlot > endSlot`. It silently allows `startSlot == endSlot`. [1](#0-0) 

**Consequence in `calculateTwap`:**

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // Panic(0x12)
uint128 twapConf  = confDiff  / uint128(slotDiff);          // Panic(0x12)
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // Panic(0x12)
```

All three divisions by zero cause an EVM `Panic(0x12)` revert in Solidity ≥ 0.8. [2](#0-1) 

**Contrast with the Solana receiver, which correctly uses strict `>`:**

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The Solana implementation rejects equal slots; the EVM implementation does not. This is a clear implementation inconsistency.

---

### Impact Explanation

Any caller of `parseTwapPriceFeedUpdates` who supplies two valid Wormhole-signed `TwapMessage` accumulator updates sharing the same `publishSlot` will receive an unconditional revert. Because the fee is collected before the slot check (`getTwapUpdateFee` is called and `msg.value` is consumed before `validateTwapPriceInfo`), the caller also loses their fee. [5](#0-4) 

Downstream integrators relying on `parseTwapPriceFeedUpdates` for on-chain TWAP prices (e.g., lending protocols, derivatives) would be unable to consume TWAP data for any feed whose legitimate Wormhole-signed snapshots happen to share a slot, permanently blocking that feed's TWAP path on EVM chains.

---

### Likelihood Explanation

The `updateData` passed to `parseTwapPriceFeedUpdates` must be valid Wormhole Merkle accumulator updates signed by Wormhole guardians; an attacker cannot forge arbitrary `publishSlot` values. However:

1. Pythnet can legitimately emit two `TwapMessage` snapshots with the same `publishSlot` if the oracle program is called twice within a single slot (e.g., during a slot boundary edge case or a Pythnet upgrade).
2. Any such pair of real, guardian-signed messages would be freely available from Hermes. An unprivileged caller can then submit them to `parseTwapPriceFeedUpdates` and trigger the panic revert.
3. The Solana receiver's stricter check (`>`) confirms the protocol intent is to require strictly increasing slots; the EVM omission is an implementation defect, not a design choice.

---

### Recommendation

Change the slot ordering check in `validateTwapPriceInfo` from `>` to `>=` to match the Solana receiver's behavior and prevent `slotDiff == 0`:

```solidity
// Before (allows equal slots → slotDiff = 0 → division by zero)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (rejects equal slots, consistent with Solana receiver)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

---

### Proof of Concept

1. Obtain two valid Wormhole-signed `TwapMessage` accumulator updates for the same feed where both messages carry `publishSlot = S` (same slot), `prevPublishTime < publishTime` (each individually valid), and `startPublishTime ≤ endPublishTime`.
2. Call `parseTwapPriceFeedUpdates{value: fee}(updateData, priceIds)` on the EVM Pyth contract.
3. `validateTwapPriceInfo` passes (condition `startSlot > endSlot` is false when they are equal).
4. `calculateTwap` computes `slotDiff = S - S = 0`.
5. `priceDiff / int128(uint128(0))` triggers `Panic(0x12)` — transaction reverts, fee is lost, TWAP is unavailable.
