Audit Report

## Title
Redundant Repeated `block_median_time` Store Lookups in `SinceVerifier` Per-Input Loop - (File: `verification/src/transaction_verifier.rs`)

## Summary
`SinceVerifier::verify()` calls `block_median_time(&parent_hash)` inside the per-input loop, where `parent_hash` is derived from `self.tx_env.parent_hash()` — a value that is constant for all inputs in the same transaction. `block_median_time` performs up to 37 sequential `get_header_fields` calls per invocation. This results in O(N × 37) redundant store traversals for a transaction with N timestamp-`since` inputs, when a single call before the loop would suffice.

## Finding Description
**Outer loop — `SinceVerifier::verify()` (`verification/src/transaction_verifier.rs` L735–758):**
The loop iterates over every resolved input and calls both `verify_absolute_lock` and `verify_relative_lock` per input unconditionally.

**Absolute timestamp branch — `verify_absolute_lock` (L651–657):**
```rust
Some(SinceMetric::Timestamp(timestamp)) => {
    let parent_hash = self.tx_env.parent_hash();          // constant per tx
    let tip_timestamp = self.block_median_time(&parent_hash); // recomputed every input
    ...
}
```

**Relative timestamp branch — `verify_relative_lock` (L704–719):**
```rust
let parent_hash = self.tx_env.parent_hash();              // constant per tx
...
let current_median_time = self.block_median_time(&parent_hash); // recomputed every input
```

**`block_median_time` cost (`traits/src/header_provider.rs` L32–50):**
Each call walks up to 37 ancestors via `get_header_fields`, which in the tx-pool path resolves through `DataLoaderWrapper::get_header_fields` → `ChainStore::get_block_header`. The `StoreCache` holds an LRU header cache (`store/src/cache.rs` L13), so after the first call the 37 headers are warm in the LRU. Subsequent calls within the same transaction verification hit the LRU rather than RocksDB directly, but still incur O(37) mutex-locked LRU lookups and full traversal per input. The claim of "499,500 raw RocksDB reads" is partially mitigated by the LRU cache, but O(N × 37) redundant in-memory traversals with mutex contention remain real.

**`MEDIAN_TIME_BLOCK_COUNT = 37` (`spec/src/consensus.rs` L55)** is the fixed walk depth.

The `SinceVerifier` is invoked in both the tx-pool admission path (`verify_rtx` in `tx-pool/src/util.rs` L85–132) and the block verification path (`BlockTxsVerifier` in `verification/contextual/src/contextual_block_verifier.rs`). No guard or cache exists at the `SinceVerifier` level to deduplicate the `block_median_time(&parent_hash)` result across inputs.

## Impact Explanation
This is a **suboptimal implementation of the CKB state storage/verification mechanism** (Medium, 2001–10000 points). A transaction with many timestamp-`since` inputs causes O(N × 37) redundant traversals of the header chain during `SinceVerifier::verify()`, where a single pre-loop computation would produce the identical result. Under sustained submission of large transactions with timestamp-`since` inputs, this amplifies CPU and lock-contention cost in the tx-pool worker and block-verification threads. The LRU cache prevents the worst-case RocksDB I/O amplification claimed, but the redundant computation and mutex overhead remain and are proportional to input count.

## Likelihood Explanation
Any unprivileged user with access to the `send_transaction` RPC or P2P relay can submit a transaction with many inputs carrying timestamp-`since` values. The transaction is rejected as `Immature` after verification, but the full `SinceVerifier` cost is paid before rejection. Because the inputs are not consumed on rejection, the same transaction can be resubmitted repeatedly at negligible marginal cost. The capital requirement (live cells) is a one-time setup cost.

## Recommendation
Compute `block_median_time(self.tx_env.parent_hash())` once before the input loop and pass the cached result into `verify_absolute_lock` and `verify_relative_lock`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    let mut cached_median_time: Option<u64> = None;
    for (index, (cell_meta, input)) in ... {
        ...
        self.verify_absolute_lock(index, since, &mut cached_median_time)?;
        self.verify_relative_lock(index, since, cell_meta, &mut cached_median_time)?;
    }
    Ok(())
}
```

`parent_median_time(&info.block_hash)` in the relative branch varies per input (depends on the cell's block hash) and cannot be cached globally, but `current_median_time` from `self.tx_env.parent_hash()` is always invariant within a single transaction verification and should be computed exactly once.

## Proof of Concept
1. On a devnet, create N live cells (e.g., N = 1000).
2. Construct a transaction spending all N cells, setting each input's `since` field to an absolute timestamp value in the future (e.g., `0x4000_0000_FFFF_FFFF`).
3. Submit via `send_transaction` RPC.
4. Observe that `SinceVerifier::verify()` calls `block_median_time(&parent_hash)` N times (once per input in `verify_absolute_lock`), each traversing 37 headers via `get_header_fields`.
5. The transaction is rejected as `Immature`, but the full O(N × 37) traversal cost was paid.
6. Instrument `block_median_time` with a counter to confirm N invocations with the identical `parent_hash` argument.
7. Repeat submission in a loop to sustain load on the tx-pool worker thread.
