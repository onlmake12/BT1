### Title
Deviation-Only Subscriptions Permanently Blocked From Updating When All Prices Are Zero — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary
In `Scheduler.sol`'s `_validateShouldUpdatePrices`, when a subscription is configured with `updateOnDeviation=true` and `updateOnHeartbeat=false`, and all stored price feeds carry a zero price value, the deviation-check loop unconditionally `continue`s over every feed and never returns a valid timestamp. The function falls through to `revert SchedulerErrors.UpdateConditionsNotMet()` on every call, permanently preventing any further price update for that subscription.

---

### Finding Description

`_validateShouldUpdatePrices` is the sole gate that decides whether `updatePriceFeeds` may proceed. Its structure mirrors the ZeroLendToken pattern exactly: two independent criteria blocks, each of which may `return` early to allow the update; if neither returns, the function reverts.

```solidity
// Scheduler.sol lines 399-453
if (params.updateCriteria.updateOnHeartbeat) {
    ...
    if (lastUpdateTime == 0 ||
        updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds) {
        return updateTimestamp;   // ← heartbeat path
    }
}

if (params.updateCriteria.updateOnDeviation) {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        PythStructs.PriceFeed storage previousFeed = _state
            .priceUpdates[subscriptionId][priceFeeds[i].id];

        if (previousFeed.id == bytes32(0)) {
            return updateTimestamp;   // ← first-update escape hatch
        }

        int64 currentPrice  = priceFeeds[i].price.price;
        int64 previousPrice = previousFeed.price.price;

        if (previousPrice == 0 || currentPrice == 0) {
            continue;               // ← skips the feed entirely
        }
        ...
        if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
            return updateTimestamp;
        }
    }
}

revert SchedulerErrors.UpdateConditionsNotMet();   // ← always reached when all prices are 0
```

When `updateOnHeartbeat=false` the first block is skipped entirely. When `updateOnDeviation=true` but every feed's `price.price` field is `0` (either the incoming update or the stored previous value), the `continue` fires for every iteration. The loop exits without ever returning, and the unconditional revert at the bottom fires on every call.

The first update is protected by the `previousFeed.id == bytes32(0)` escape hatch, so the subscription initialises correctly. All subsequent updates are permanently blocked.

The analogous ZeroLendToken bug: `require(!paused && !whitelisted[from], "paused")` was supposed to let whitelisted users bypass the pause but instead blocked them always. Here, the deviation block is supposed to let subscriptions update when prices move, but the `continue`-then-revert path blocks them permanently when prices are zero.

---

### Impact Explanation

A subscription with `updateOnDeviation=true` / `updateOnHeartbeat=false` whose price feeds report `price = 0` (e.g. a newly listed asset, a market that closed at zero, or a feed that has not yet been populated) is permanently unable to call `updatePriceFeeds` after the first successful update. The subscription's stored price data becomes permanently stale. The subscription balance is locked in the contract because there is no `withdrawFunds` path independent of an active subscription — the manager must deactivate via `updateSubscription`, but the price data is still frozen. This matches the Immunefi impact categories of **temporary/permanent freezing of funds** and **griefing**.

---

### Likelihood Explanation

The condition is triggered whenever `previousPrice == 0 || currentPrice == 0` for **all** feeds in the subscription. This is realistic for:

- Single-feed subscriptions tracking an asset whose price is reported as zero (new listing, halted market, misconfigured feed).
- Multi-feed subscriptions where every feed simultaneously reports zero (e.g. a batch of newly registered feeds before any publisher posts a non-zero value).

Any unprivileged updater (keeper / relayer) who submits `updatePriceFeeds` with zero-price data after the first update will trigger the permanent block. No special access is required.

---

### Recommendation

Treat a zero-price feed as a non-blocking case rather than a skipped case. Either:

1. **Allow the update when all feeds are zero** — after the loop, if every feed was skipped due to zero prices, return `updateTimestamp` instead of falling through to the revert.
2. **Count skipped feeds** — if `skippedCount == priceFeeds.length`, treat it as a first-update scenario and return `updateTimestamp`.

```solidity
if (params.updateCriteria.updateOnDeviation) {
    uint8 skipped = 0;
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        ...
        if (previousPrice == 0 || currentPrice == 0) {
            skipped++;
            continue;
        }
        if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
            return updateTimestamp;
        }
    }
    // All feeds had zero prices — allow the update rather than blocking forever
    if (skipped == priceFeeds.length) {
        return updateTimestamp;
    }
}
```

1. Deploy `SchedulerUpgradeable` with a valid Pyth mock.
2. Create a subscription with `updateOnHeartbeat=false`, `updateOnDeviation=true`, `deviationThresholdBps=100`.
3. Call `updatePriceFeeds` with `price.price = 1000` for all feeds → succeeds (first update, `previousFeed.id == bytes32(0)` path).
4. Call `updatePriceFeeds` again with `price.price = 0` for all feeds → reverts `UpdateConditionsNotMet` (current price is zero, `continue` fires for every feed).

Correct PoC path for permanent block:

4. Call `updatePriceFeeds` with `price.price = 0` for all feeds → succeeds only if `previousFeed.id == bytes32(0)` is still true... actually no, after step 3 the previous feed is stored with `price=1000`. So step 4 with `price=0`: `currentPrice=0`, `continue` fires, loop exits, revert.
5. Now the stored price is still `1000` (step 4 reverted). Call `updatePriceFeeds` with `price.price = 0` again → same revert.
6. The subscription can never be updated with a zero current price, even with a large deviation from the stored non-zero price.

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

        // If updateOnDeviation is enabled, check if any price has deviated enough
        if (params.updateCriteria.updateOnDeviation) {
            for (uint8 i = 0; i < priceFeeds.length; i++) {
                // Get the previous price feed for this price ID using subscriptionId
                PythStructs.PriceFeed storage previousFeed = _state
                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice
                );
                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator
                );

                // If deviation exceeds threshold, trigger update
                if (
                    deviationBps >= params.updateCriteria.deviationThresholdBps
                ) {
                    return updateTimestamp;
                }
            }
        }

        revert SchedulerErrors.UpdateConditionsNotMet();
