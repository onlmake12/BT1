### Title
Provider Can Request Randomness From Themselves, Fully Controlling the Random Output - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Entropy contract allows anyone to permissionlessly register as a randomness provider and also make requests to that same provider address. Because the two-party commit-reveal security model requires that the provider and requester be distinct honest parties, a single entity controlling both roles can predict and manipulate the random number output before committing to it.

### Finding Description

The Entropy protocol's security guarantee is stated explicitly in the contract:

> "This protocol has the same security properties as the 2-party randomness protocol above: as long as either the provider or user is honest, the number r is random."

This guarantee collapses entirely when the same address is both the provider and the requester.

**Root cause — missing `msg.sender != provider` check in `requestHelper`:**

In `requestHelper`, the requester is recorded as `msg.sender` with no validation that `msg.sender` is not the provider being requested from:

```solidity
// Entropy.sol line 260
req.requester = msg.sender;
``` [1](#0-0) 

The `request` and `requestV2` entry points both accept an explicit `provider` address and an explicit `userCommitment`/`userContribution`, with no check that `provider != msg.sender`:

```solidity
function request(
    address provider,
    bytes32 userCommitment,
    bool useBlockHash
) public payable override returns (uint64 assignedSequenceNumber) {
    EntropyStructsV2.Request storage req = requestHelper(
        provider, userCommitment, useBlockHash, false, 0
    );
``` [2](#0-1) 

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit

    EntropyStructsV2.Request storage req = requestHelper(
        provider,

Provider registration is fully permissionless:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
``` [4](#0-3) 

The random number is computed as:

```solidity
randomNumber = combineRandomValues(
    userContribution,
    providerContribution,
    blockHash
);
``` [5](#0-4) 

Because the attacker knows their own `providerContribution` (x_i) from their pre-committed hash chain before making the request, they can freely choose `userContribution` (x_U) such that `combineRandomValues(x_U, x_i, 0)` equals any desired target value. They commit to `hash(x_U)` in the request, then reveal x_U in the fulfill step — the contract accepts this as a valid reveal.

### Impact Explanation

A malicious actor who registers as a provider and requests from themselves gains **full, deterministic control** over the random number delivered to any downstream `IEntropyConsumer` callback. Any protocol that:

- Accepts a caller-specified provider address, or
- Uses a provider that the attacker controls

is vulnerable to complete randomness manipulation. Concrete downstream impacts include: rigged NFT trait assignment, guaranteed lottery wins, manipulated DeFi liquidation ordering, and predictable gaming outcomes — all of which represent direct financial loss to other users of the consuming protocol.

### Likelihood Explanation

Provider registration is permissionless and costs only gas. The attacker needs no privileged access, no leaked keys, and no governance majority. The only prerequisite is paying the Pyth protocol fee per request (a small ETH amount). The attack is repeatable for every sequence number in the attacker's hash chain. Any downstream protocol that does not hard-code the default Pyth provider address is immediately exploitable.

### Recommendation

Add a check in `requestHelper` (or in each public entry point) that prevents the requester from being the same address as the provider:

```solidity
if (msg.sender == provider) revert EntropyErrors.SelfRequest();
``` [6](#0-5) 

This mirrors the fix applied in the BountyV1 analog: directly disallowing the conflicted role at the point of submission.

### Proof of Concept

1. **Attacker registers as provider.** Attacker calls `register(fee, commitment, metadata, chainLength, uri)` from address `A`. They pre-compute their full hash chain `x_{N-1}, ..., x_0` and know every `x_i`.

2. **Attacker identifies target random value.** For sequence number `i`, the attacker knows `x_i` (their provider contribution). They compute a `userContribution` value `x_U` such that `combineRandomValues(x_U, x_i, 0)` equals their desired random output (e.g., the winning lottery ticket index).

3. **Attacker submits request to themselves.** From address `A`, attacker calls:
   ```solidity
   entropy.request{value: fee}(
       A,                              // provider = attacker
       constructUserCommitment(x_U),   // chosen to yield desired r
       false
   );
   ```
   `requestHelper` records `req.requester = A` and `req.provider = A` with no rejection. [7](#0-6) 

4. **Attacker fulfills the request.** Attacker calls `reveal(A, seqNum, x_U, x_i)`. The `reveal` function checks `req.requester == msg.sender` — this passes because both are `A`. The random number `r = combineRandomValues(x_U, x_i, 0)` is the pre-chosen value. [8](#0-7) 

5. **For callback-based consumers**, the attacker calls `revealWithCallback(A, seqNum, x_U, x_i)` instead. The callback fires on the consuming contract with the attacker-chosen random number. [9](#0-8)
