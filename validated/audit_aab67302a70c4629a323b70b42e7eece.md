### Title
Arithmetic Overflow in `getFeeV2` Blocks All Entropy Requests for a Provider — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `getFeeV2` function in `Entropy.sol` performs an unchecked `uint128` addition of the provider fee and the Pyth protocol fee. Because there is no upper bound enforced on a provider's `feeInWei`, a provider can register (or update) with `feeInWei = type(uint128).max`. When `pythFeeInWei > 0`, the addition overflows and reverts. Since `requestHelper` calls `getFeeV2` unconditionally, every user attempting to request randomness from that provider is permanently blocked.

### Finding Description

`getFeeV2(address provider, uint32 gasLimit)` is defined as:

```solidity
function getFeeV2(
    address provider,
    uint32 gasLimit
) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [1](#0-0) 

`getProviderFee` returns `provider.feeInWei` (or a scaled value) as `uint128`. When `feeInWei = type(uint128).max` and `_state.pythFeeInWei > 0`, the addition `type(uint128).max + pythFeeInWei` overflows and reverts in Solidity ≥ 0.8.

There is no upper-bound validation on `feeInWei` in either `register` or `setProviderFee`:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    ...
}
``` [2](#0-1) 

`requestHelper` calls `getFeeV2` unconditionally:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [3](#0-2) 

The existing test suite explicitly acknowledges this overflow causes a revert but treats it as accepted behavior rather than fixing it:

```solidity
// Check that overflowing the fee arithmetic causes the transaction to revert.
vm.prank(provider1);
random.register(MAX_UINT128, ...);
vm.expectRevert();
random.getFee(provider1);
``` [4](#0-3) 

### Impact Explanation

Any user calling `request`, `requestWithCallback`, or `requestV2` targeting the affected provider will have their transaction revert at the `getFeeV2` call inside `requestHelper`. The Entropy service becomes completely unusable for that provider. If the affected provider is the protocol's default provider (set by governance), all calls to the no-argument `requestV2()` and `getFeeV2()` variants are also blocked, since they route through `getDefaultProvider()`. [5](#0-4) 

### Likelihood Explanation

Any unprivileged actor can register as an Entropy provider (permissionless) and immediately set `feeInWei = type(uint128).max`. The `setProviderFee` function is also callable by any registered provider at any time with no bounds check. The Pyth protocol fee (`pythFeeInWei`) is always > 0 in production deployments, making the overflow deterministic once the provider fee is set to `type(uint128).max`. The existing test confirms the revert is reproducible.

### Recommendation

- **Short term**: Add an upper-bound check on `feeInWei` in both `register` and `setProviderFee` (e.g., `require(newFeeInWei <= MAX_PROVIDER_FEE)`), where `MAX_PROVIDER_FEE = type(uint128).max - _state.pythFeeInWei`.
- **Long term**: Use `SafeCast` or explicit overflow-safe addition (`unchecked` with a prior bounds check) in `getFeeV2` so that an extreme provider fee produces a graceful error rather than a panic revert that propagates to all callers.

### Proof of Concept

1. Deploy Entropy with `pythFeeInWei = 1`.
2. Call `register(type(uint128).max, ...)` as a new provider — succeeds with no revert.
3. Call `getFeeV2(provider, 0)` — reverts with arithmetic overflow (`type(uint128).max + 1`).
4. Call `requestV2(provider, userRandomNumber, 0){value: ...}` — reverts at the `getFeeV2` call inside `requestHelper`, permanently blocking all randomness requests to that provider.

The test at `Entropy.t.sol:612-622` already demonstrates steps 2–3 and confirms the revert. [6](#0-5)
