### Title
Hardcoded Price Returned for `BASE_CURRENCY` Asset Without Oracle Validation — (`File: target_chains/ethereum/contracts/contracts/aave/PythPriceOracleGetter.sol`)

---

### Summary

`PythPriceOracleGetter.getAssetPrice()` returns a hardcoded `BASE_CURRENCY_UNIT` value for any asset whose address matches `BASE_CURRENCY`, bypassing the Pyth oracle entirely. This is structurally identical to the umee `GetExchangeRateBase` bug: both short-circuit oracle lookup and return a fixed "1 USD" equivalent based on an identity check rather than a live price feed. If `BASE_CURRENCY` is set to a real token address (e.g., USDC), a depeg event is invisible to the system.

---

### Finding Description

In `getAssetPrice()`, the check on line 57 fires before any oracle call:

```solidity
function getAssetPrice(address asset) external view override returns (uint256) {
    bytes32 priceId = _registryState.assetsPriceIds[asset];
    if (asset == BASE_CURRENCY) {
        return BASE_CURRENCY_UNIT;   // ← hardcoded, no oracle
    }
    if (priceId == 0) {
        revert PythErrors.PriceFeedNotFound();
    }
    PythStructs.Price memory price = pyth().getPriceNoOlderThan(
        priceId,
        PythAssetRegistry.validTimePeriodSeconds()
    );
    // ...
}
``` [1](#0-0)

The `IPriceOracleGetter` interface comment says "Address 0x0 is reserved for USD as base currency," but this is not enforced in code: [3](#0-2) 

`setAssetsSources` also allows registering `BASE_CURRENCY` as a normal asset with a valid `priceId`, but the early-return on line 57–59 silently ignores that `priceId` and returns the hardcoded unit: [4](#0-3) 

---

### Impact Explanation

If `BASE_CURRENCY` is set to a real stablecoin token address (e.g., USDC at `0xA0b86991…`):

- `getAssetPrice(USDC)` always returns `BASE_CURRENCY_UNIT` (e.g., `1e8` = $1.00).
- A depeg event (USDC drops to $0.90) is completely invisible to the Aave pool using this oracle.
- Borrowers can post depegged USDC as collateral at the hardcoded $1.00 price and borrow against it, creating undercollateralized positions.
- The pool cannot liquidate correctly because the oracle still reports $1.00.

This matches the umee exploit scenario 2 exactly: stablecoin loses peg, leverage module fails to detect the change, attacker draws funds from the system.

---

### Likelihood Explanation

The `PythPriceOracleGetter` constructor accepts any `baseCurrency` address. Deployments targeting chains where USD is represented by a real token contract (e.g., USDC on Ethereum, axlUSDC on Cosmos-EVM chains) would naturally pass that token's address as `baseCurrency`. The Aave documentation pattern encourages this for non-ETH-denominated pools. The code provides no guard, no warning, and no fallback oracle call for the base currency asset.

---

### Recommendation

1. **Short term**: Add a constructor check enforcing `baseCurrency == address(0)` (the only safe value per the interface spec), or add a Pyth oracle lookup for `BASE_CURRENCY` when it is a non-zero address.
2. **Long term**: Remove the unconditional early-return for `BASE_CURRENCY`. Instead, if a `priceId` is registered for `BASE_CURRENCY`, use the oracle; only fall back to `BASE_CURRENCY_UNIT` when no `priceId` is registered and `asset == address(0)`.

---
