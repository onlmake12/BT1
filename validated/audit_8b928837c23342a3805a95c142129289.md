### Title
Echo Callback Permanently Lost Due to `clearRequest` Before `req.requester` Read — (`contracts/echo/Echo.sol`)

---

### Summary

In `Echo.executeCallback`, the request storage slot is zeroed via `clearRequest(sequenceNumber)` **before** the callback is dispatched using `req.requester` and `req.callbackGasLimit`. Because `req` is a `storage` pointer, both fields read as zero after the clear. The callback is sent to `address(0)` with `0` gas, silently fails, and the requester permanently loses their price-update delivery despite having paid the full fee.

---

### Finding Description

`Echo.executeCallback` follows this sequence:

1. Credits the provider's accrued fees (line 161–162).
2. Calls `clearRequest(sequenceNumber)` (line 164), which zeroes the storage slot that `req` points to.
3. Advances `firstUnfulfilledSeq` (lines 169–174).
4. Attempts the callback using the now-zeroed `req.requester` and `req.callbackGasLimit` (lines 176–179). [1](#0-0) 

Because `req` is declared as `Request storage req = findActiveRequest(sequenceNumber)`, it is a pointer into the contract's storage. After `clearRequest` zeroes that slot, `req.requester` evaluates to `address(0)` and `req.callbackGasLimit` evaluates to `0`. The `try` call therefore becomes:

```solidity
IEchoConsumer(address(0))._echoCallback{gas: 0}(sequenceNumber, priceFeeds)
```

A call with `gas: 0` to `address(0)` immediately runs out of gas. The `catch` block fires and emits `PriceUpdateCallbackFailed`. The request has already been cleared and can never be retried.

The Entropy contract, which shares the same codebase and the same `clearRequest` pattern, explicitly guards against this. In the legacy `revealWithCallback` path, the developers save `req.requester` to a stack variable **before** clearing and add a prominent warning comment: [2](#0-1) 

Echo omits this precaution entirely.

---

### Impact Explanation

Every call to `Echo.executeCallback` results in:

- The provider being credited `req.fee + msg.value - pythFee` (line 161–162) — payment is transferred correctly.
- The request being permanently deleted (line 164).
- The requester's `_echoCallback` **never being invoked** — the price update is lost.
- No retry path exists; the sequence number is gone.

The requester paid the full fee (base + per-feed + gas) and receives nothing. This is a complete loss of service and funds for every user of the Echo protocol. [3](#0-2) 

---

### Likelihood Explanation

This is triggered on **every** `executeCallback` invocation — there is no code path that avoids it. Any provider (or any unprivileged third party, since `executeCallback` is `external` with no access control) calling the function to collect fees will silently break the callback for the requester. The bug is deterministic and 100% reproducible. [4](#0-3) 

---

### Recommendation

Save `req.requester` and `req.callbackGasLimit` to memory variables **before** calling `clearRequest`, mirroring the pattern already used in `Entropy.revealWithCallback`:

```solidity
// Save before clearing
address callAddress = req.requester;
uint32 gasLimit = req.callbackGasLimit;

clearRequest(sequenceNumber);
// WARNING: DO NOT USE req BELOW HERE AS ITS CONTENTS HAS BEEN CLEARED

try
    IEchoConsumer(callAddress)._echoCallback{gas: gasLimit}(sequenceNumber, priceFeeds)
{ ... }
``` [5](#0-4) 

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback{value: fee}(provider, publishTime, priceIds, gasLimit)`. Request stored at slot `S`; `req.requester = user`, `req.callbackGasLimit = gasLimit`.
2. Provider calls `executeCallback(provider, sequenceNumber, updateData, priceIds)`.
3. Line 161–162: `_state.providers[provider].accruedFeesInWei += fee` — provider is paid.
4. Line 164: `clearRequest(sequenceNumber)` — slot `S` is zeroed. `req.requester` now reads `address(0)`, `req.callbackGasLimit` now reads `0`.
5. Line 176–179: `IEchoConsumer(address(0))._echoCallback{gas: 0}(...)` — call fails with out-of-gas.
6. Line 192–200: `catch` emits `PriceUpdateCallbackFailed`. Request is gone. User receives no callback and cannot recover funds. [6](#0-5)
