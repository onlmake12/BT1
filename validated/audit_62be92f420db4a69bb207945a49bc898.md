### Title
Ignored Return Value of `async_send_message_to` in Compact Block Reconstruction Permanently Stalls Block Propagation Per-Peer - (File: `sync/src/relayer/block_transactions_process.rs`)

### Summary
When a compact block cannot be fully reconstructed from a `BlockTransactions` message, the node sends a follow-up `GetBlockTransactions` request to the peer. The return value of this send is explicitly discarded with `let _ignore = ...`. If the send fails, the node still updates its internal expected-transaction state and returns `CompactBlockRequiresFreshTransactions`, leaving the compact block permanently stalled for that peer — the peer never receives the request and will never send the missing transactions.

### Finding Description
In `block_transactions_process.rs`, after a `ReconstructionResult::Missing` or `ReconstructionResult::Collided` outcome, the code builds a `GetBlockTransactions` message and sends it:

```rust
let _ignore = async_send_message_to(&self.nc, self.peer, &message).await;

let _ignore_prev_value =
    mem::replace(expected_transaction_indexes, missing_transactions);
let _ignore_prev_value = mem::replace(expected_uncle_indexes, missing_uncles);
``` [1](#0-0) 

`async_send_message_to` returns a `Status` value that encodes success or a `StatusCode::Network` error. The caller discards this `Status` unconditionally. Immediately after, `expected_transaction_indexes` and `expected_uncle_indexes` are overwritten with the new missing lists, and the function returns `StatusCode::CompactBlockRequiresFreshTransactions`. [2](#0-1) 

The analogous send helper in `sync/src/utils.rs` already logs the error internally but still returns a non-ok `Status` to the caller: [3](#0-2) 

Because the caller ignores that `Status`, the node's `pending_compact_blocks` entry for this peer is left in a state where it expects transactions that were never actually requested. The peer, having received no request, will never send them. The compact block reconstruction for this peer is permanently stalled until the entry is evicted or the peer disconnects.

### Impact Explanation
A malicious relay peer can:
1. Send a compact block that is missing one or more transactions.
2. Arrange for the TCP/P2P send channel to be full or closed at the moment the node tries to send `GetBlockTransactions` (e.g., by flooding the channel or disconnecting immediately after sending the compact block).
3. The node updates its state as if the request was sent, but the peer never receives it.
4. The compact block reconstruction for that peer is permanently stalled.

If the attacker is the only peer advertising the new block (e.g., in a targeted eclipse or low-connectivity scenario), the victim node cannot advance its chain tip via compact-block relay from that peer. This is a liveness impact: the node falls behind the canonical chain tip, which is a prerequisite for more serious attacks (double-spend, eclipse).

### Likelihood Explanation
Any unprivileged relay peer can trigger this by sending a compact block and then causing a send failure. Send failures occur naturally when the P2P send buffer is full (a condition an attacker can induce by flooding) or when the peer disconnects. The entry point is the standard relay protocol, reachable by any connected peer.

### Recommendation
Check the return value of `async_send_message_to`. If the send fails, do **not** update `expected_transaction_indexes` / `expected_uncle_indexes`, so the node retains the previous request state and can retry or fall back to requesting the full block from another peer. For example:

```rust
if async_send_message_to(&self.nc, self.peer, &message).await.is_ok() {
    *expected_transaction_indexes = missing_transactions;
    *expected_uncle_indexes = missing_uncles;
} else {
    // send failed; leave state unchanged so a retry or peer fallback can occur
    return StatusCode::Network.with_context(block_hash);
}
```

### Proof of Concept
1. Connect a malicious peer to a CKB node.
2. Send a `CompactBlock` message with short IDs for transactions not in the node's tx-pool.
3. The node will call `reconstruct_block`, get `ReconstructionResult::Missing`, and attempt to send `GetBlockTransactions`.
4. Before or during that send, close the P2P write channel (or fill the send buffer) so `async_send_message_to` returns `StatusCode::Network`.
5. Observe that `expected_transaction_indexes` is updated to the missing list, but the peer never receives the request.
6. The `pending_compact_blocks` entry for this block/peer is now permanently stalled: the node waits for a `BlockTransactions` response that will never arrive, and the block cannot be reconstructed via this peer. [4](#0-3)

```rust
pub(crate) async fn async_send_message<Message: Entity>(
    protocol_id: ProtocolId,
    nc: &Arc<dyn CKBProtocolContext + Sync>,
    peer_index: PeerIndex,
    message: &Message,
) -> Status {
    // ignore Error return, only happens on shutdown case
    if let Err(err) = nc
        .async_send_message(protocol_id, peer_index, message.as_bytes())
        .await
    {
        let name = message_name(protocol_id, message);
        let error_message = format!("nc.send_message {name}, error: {err:?}");
        ckb_logger::error!("{}", error_message);
        return StatusCode::Network.with_context(error_message);
    }

    let bytes = message.as_bytes().len() as u64;
    let item_name = item_name(protocol_id, message);
    let protocol_name = protocol_name(protocol_id);
    metric_ckb_message_bytes(
        MetricDirection::Out,
        &protocol_name,
        &item_name,
        None,
        bytes,
    );

    Status::ok()
}
