Audit Report

## Title
Tx-Pool DoS via Unbounded O(n) Descendant Graph Traversal on Transaction Removal — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry` in `pool_map.rs` unconditionally calls `update_descendants_index_key`, which performs an unbounded BFS over the entire descendant graph of the removed transaction before links are stripped. Both `remove_committed_tx` and `remove_expired` invoke `remove_entry` directly — not `remove_entry_and_descendants` — so the full descendant set is traversed under the tx-pool write lock for every committed or expired transaction. Because `max_ancestors_count` caps ancestor depth but places no bound on the number of children per transaction, an attacker can construct an arbitrarily wide dependency tree and trigger O(n) CPU work per removal, stalling all concurrent pool operations.

## Finding Description
`remove_entry` (L235–250) calls `update_descendants_index_key` (L447–460) before stripping links. `update_descendants_index_key` calls `self.links.calc_descendants`, which delegates to `calc_relation_ids` (L52–72) — an iterative BFS that visits every reachable child node and inserts each into a `HashSet`. After the BFS, `modify_by_id` is called on every descendant to update ancestor-weight scores. All of this executes while the tx-pool write lock is held.

The safe path, `remove_entry_and_descendants` (L252–265), explicitly calls `remove_entry_links` for every node in the set *before* calling `remove_entry`, so when `update_descendants_index_key` runs, `calc_descendants` finds an empty child set and returns immediately. The comment at L256 confirms this is intentional: *"update links state for remove, so that we won't update_descendants_index_key in remove_entry."* Neither `remove_committed_tx` (L253–268) nor `remove_expired` (L271–288) uses this pattern — both call `remove_entry` directly with links intact.

`max_ancestors_count` (default 25) limits how many ancestors a single transaction may have, but no symmetric `max_descendants_count` exists anywhere in the codebase. A transaction with K outputs can have K direct children; each child can have K children; the tree can be 25 levels deep, yielding up to K^25 descendants bounded only by the 180 MB pool size limit.

Trigger path 1 (committed transaction): attacker submits a wide dependency tree rooted at T0, then T0 is included in any block. `_update_tx_pool_for_reorg` (L1039–1056) calls `remove_committed_txs`, which calls `remove_committed_tx` → `pool_map.remove_entry(T0)` → full BFS over all descendants, under the write lock.

Trigger path 2 (expiry): same tree, attacker waits 12 hours. `remove_expired` iterates all expired entries and calls `pool_map.remove_entry` for each, again with links intact.

## Impact Explanation
The tx-pool write lock serializes all pool operations: `send_transaction` RPC, P2P relay submissions, and block assembly. While the BFS traverses n descendants and issues n hash-map writes, every concurrent operation is blocked. With the 180 MB pool limit and ~100-byte minimum transaction size, n can approach ~1.8 million entries. At that scale the traversal can consume hundreds of milliseconds per committed transaction. An attacker who pre-fills the pool and triggers repeated root-transaction commits can sustain this stall, degrading or denying service to legitimate users across the network.

This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack entry point is the standard `send_transaction` RPC or P2P relay — no special privilege is required. The minimum fee rate is 1,000 shannons/KB; a ~100-byte transaction costs ~100 shannons. Filling the pool with 1.8 million transactions costs roughly 180 billion shannons (~1.8 CKB in fees). CKB capacity for outputs adds capital cost, but the fee cost alone is low. Having the root transaction committed requires only that a miner includes it — standard behavior for any fee-paying transaction. The attack is repeatable: after T0 is committed, subsequent levels of the tree become new roots and can be committed in later blocks, each triggering another O(n) traversal. The attack is realistic for a motivated adversary.

## Recommendation
1. **Cap descendants per transaction** analogously to `max_ancestors_count`. During transaction submission, reject any transaction that would push any ancestor's descendant count above a configurable limit (e.g., 25).
2. **Strip links before calling `remove_entry` in `remove_committed_tx` and `remove_expired`**, mirroring the pattern in `remove_entry_and_descendants`. This ensures `calc_descendants` returns an empty set and reduces per-removal cost to O(1).
3. **Lazy descendant-weight updates**: store a generation counter on each entry and recompute ancestor weights on demand rather than eagerly propagating on every removal.

## Proof of Concept
```
1. Submit T0 with K outputs via send_transaction RPC.
2. Submit K transactions T1_1..T1_K, each spending one output of T0.
3. Submit K² transactions T2_1..T2_K², each spending one output of a T1_i.
4. Continue for up to 25 levels until pool approaches 180 MB.
5. T0 is included in a block by any miner.
6. Node calls _update_tx_pool_for_reorg
     → remove_committed_txs
       → remove_committed_tx(T0)
         → pool_map.remove_entry(T0)
           → update_descendants_index_key(T0)
             → links.calc_descendants(T0)  [BFS over ~K^d nodes, links still intact]
             → modify_by_id(desc_id) for each descendant
   All executed under the tx-pool write lock.
7. All concurrent send_transaction calls stall for the duration.
8. Repeat as subsequent levels of the tree are committed in later blocks.
```

A unit test can be written in `tx-pool/src/component/tests/` that constructs a pool with a wide dependency tree (e.g., depth 5, fan-out 10 = 100,000 descendants), calls `remove_entry` on the root, and measures wall-clock time to confirm super-linear growth relative to a shallow tree.
