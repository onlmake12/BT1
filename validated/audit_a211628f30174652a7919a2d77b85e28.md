### Title
Off-by-One in RFC0044 MMR Activation Epoch Check Causes First Activation-Epoch Block to Omit Chain Root — (`tx-pool/src/block_assembler/mod.rs`, `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

`build_extension()` in the block assembler checks RFC0044 (MMR) activation using the **tip/parent block's epoch number** instead of the **new block's epoch number**. The verifier in `BlockExtensionVerifier::verify()` makes the identical off-by-one check. As a result, the very first block of the RFC0044 activation epoch is assembled and accepted **without** the mandatory MMR chain root extension, violating the protocol specification and breaking light-client verifiability for that block. The bug is explicitly acknowledged in a code comment but marked as intentionally preserved for consensus consistency.

---

### Finding Description

`rfc0044_active()` returns `true` when `target >= RFC0044_ACTIVE_EPOCH`: [1](#0-0) 

In `build_extension()`, the activation check is performed against the **tip header's epoch number** — i.e., the parent of the block being assembled: [2](#0-1) 

The comment on line 566–567 explicitly states:

> "The use of the epoch number of the tip here leads to an off-by-one bug, so be careful, it needs to be preserved for consistency reasons and not fixed directly."

The block verifier makes the same off-by-one check, using `self.parent.epoch().number()` instead of the new block's epoch: [3](#0-2) 

**The off-by-one mechanics:**

- RFC0044 activates at epoch `N` (e.g., mainnet epoch 8651).
- The last block of epoch `N-1` has `parent.epoch().number() = N-1`.
- When assembling the **first block of epoch N**, `build_extension()` evaluates `rfc0044_active(N-1)` → `false`.
- Therefore, the first block of epoch N is assembled **without** the MMR chain root extension.
- The verifier also evaluates `rfc0044_active(N-1)` → `false` and accepts the block without the extension.
- Only from the **second block of epoch N** onward does the check return `true` and the extension is included.

This is confirmed by CHANGELOG entry `#4255: No chain_root in 1st block of the MMR activated epoch`.

---

### Impact Explanation

The first block of the RFC0044 activation epoch lacks the MMR chain root in its block extension. Light clients implementing the RFC0044 light-client protocol rely on the MMR chain root to verify block headers. For that specific block, no valid MMR proof can be constructed or verified, breaking the light-client protocol's continuity at the activation boundary. Any light client that requests a proof anchored at or spanning the activation epoch's first block will receive an invalid or missing proof, undermining the security guarantee of the light-client protocol for that block.

---

### Likelihood Explanation

This is a **certain** and **already-triggered** bug on mainnet and testnet. The activation epochs are hardcoded (`RFC0044_ACTIVE_EPOCH = 8651` on mainnet, `5711` on testnet). The first block of those epochs was already produced and accepted without the chain root. The bug is acknowledged in the source code comment and in the CHANGELOG. Any light client querying proofs that span the activation epoch boundary is affected. [4](#0-3) [5](#0-4) 

---

### Recommendation

The correct epoch to use for the activation check is the epoch of the **block being assembled/verified**, not the parent's epoch. For the block assembler, this requires computing the new block's epoch from the current epoch extension. For the verifier, the new block's epoch is available directly from `block.header().epoch().number()`.

However, since both the assembler and verifier share the same off-by-one, any fix must be coordinated as a consensus-breaking change (hard fork), ensuring both sides are updated atomically. A soft migration path would be to introduce a new activation epoch constant that accounts for the off-by-one, or to fix both sides in the same hard fork.

---

### Proof of Concept

1. Observe that `RFC0044_ACTIVE_EPOCH = 8651` on mainnet.
2. At the last block of epoch 8650, `tip_header.epoch().number() = 8650`.
3. `build_extension()` calls `rfc0044_active(8650)` → `8650 >= 8651` → `false` → no extension built.
4. The first block of epoch 8651 is assembled and submitted without the MMR chain root.
5. `BlockExtensionVerifier::verify()` calls `rfc0044_active(8650)` → `false` → accepts the block without extension.
6. A light client requesting an MMR proof for block at epoch 8651, index 0 finds no chain root in the extension, breaking RFC0044 verification for that block. [2](#0-1) [6](#0-5) [1](#0-0)

    pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
        let tip_header = snapshot.tip_header();
        // The use of the epoch number of the tip here leads to an off-by-one bug,
        // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
        let mmr_activate = snapshot
            .consensus()
            .rfc0044_active(tip_header.epoch().number());
        if mmr_activate {
            let chain_root = snapshot
                .chain_root_mmr(tip_header.number())
                .get_root()
                .map_err(|e| InternalErrorKind::MMR.other(e))?;
            let bytes = chain_root.calc_mmr_hash().as_bytes().into();
            Ok(Some(bytes))
        } else {
            Ok(None)
        }
    }

    pub fn verify(&self, block: &BlockView) -> Result<(), Error> {
        let extra_fields_count = block.data().count_extra_fields();

        let mmr_active = self
            .context
            .consensus
            .rfc0044_active(self.parent.epoch().number());
        match extra_fields_count {
            0 => {
                if mmr_active {
                    return Err(BlockErrorKind::NoBlockExtension.into());
                }
            }
            1 => {
                let extension = if let Some(data) = block.extension() {
                    data
                } else {
                    return Err(BlockErrorKind::UnknownFields.into());
                };
                if extension.is_empty() {
                    return Err(BlockErrorKind::EmptyBlockExtension.into());
                }
                if extension.len() > 96 {
                    return Err(BlockErrorKind::ExceededMaximumBlockExtensionBytes.into());
                }
                if mmr_active {
                    if extension.len() < 32 {
                        return Err(BlockErrorKind::InvalidBlockExtension.into());
                    }

                    let chain_root = self
                        .chain_root_mmr
                        .get_root()
                        .map_err(|e| InternalErrorKind::MMR.other(e))?;
                    let actual_root_hash = chain_root.calc_mmr_hash();
                    let expected_root_hash =
                        Byte32::new_unchecked(extension.raw_data().slice(..32));
                    if actual_root_hash != expected_root_hash {
                        return Err(BlockErrorKind::InvalidChainRoot.into());
                    }
                }
            }
            _ => {
                return Err(BlockErrorKind::UnknownFields.into());
            }
        }

        let actual_extra_hash = block.calc_extra_hash().extra_hash();
        if actual_extra_hash != block.extra_hash() {
            return Err(BlockErrorKind::InvalidExtraHash.into());
        }
        Ok(())
    }
