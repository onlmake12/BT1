### Title
`PythAggregatorV3` Does Not Enforce Staleness Check on Pyth Price Data — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

The official Pyth Chainlink-compatibility adapter `PythAggregatorV3` calls `pyth.getPriceUnsafe()` in every price-reading function without performing any staleness check on the returned `publishTime`. Any protocol that uses this adapter as a drop-in Chainlink replacement and calls `latestAnswer()` or `latestRoundData()` will silently consume arbitrarily stale prices, with no on-chain revert or warning.

---

### Finding Description

`PythAggregatorV3` is the official Pyth SDK contract for migrating Chainlink-dependent protocols to Pyth price feeds. It implements the `AggregatorV3Interface` and is actively promoted in Pyth's migration documentation.

Every price-reading function in the contract calls `pyth.getPriceUnsafe(priceId)`: [1](#0-0) [2](#0-1) 

`getPriceUnsafe` is explicitly documented to return prices from arbitrarily far in the past with no recency guarantee: [3](#0-2) 

The `AbstractPyth` contract provides `getPriceNoOlderThan()` which enforces a staleness threshold and reverts on stale data: [4](#0-3) 

`PythAggregatorV3` never calls this safe variant. There is no configurable staleness threshold, no revert on stale data, and no warning emitted. The `latestAnswer()` function is the most dangerous: it returns only the raw price integer with zero timestamp information, making it impossible for callers to independently verify freshness: [1](#0-0) 

While `latestRoundData()` does return `updatedAt = price.publishTime`, the contract itself enforces no threshold, and protocols migrating from Chainlink that do not add their own staleness check (the exact pattern in the external report) are fully exposed.

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink feed replacement and reads prices via `latestAnswer()` or `latestRoundData()` without independently checking `updatedAt` will operate on stale prices. This can lead to:

- Incorrect collateral valuations in lending protocols, enabling unfair liquidations or under-collateralized borrows.
- Mispriced trades in derivative or perpetual protocols.
- Incorrect fee or settlement calculations in any price-sensitive protocol.

The impact is identical in class to the external report: stale price data used in financial logic without any on-chain guard.

---

### Likelihood Explanation

`PythAggregatorV3` is the official, documented migration path from Chainlink to Pyth: [5](#0-4) 

Protocols migrating from Chainlink expect the adapter to behave safely. Pyth is a pull oracle — prices are only updated when someone submits an update transaction. If no keeper submits an update, the on-chain price can become arbitrarily stale. The `PythAggregatorV3` contract provides no protection against this scenario. Any unprivileged user can exploit the window between the last on-chain update and the current block.

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan`, using a configurable staleness threshold set at construction time. Add a `maxAge` state variable and revert in `latestAnswer()` and `latestRoundData()` if `block.timestamp - price.publishTime > maxAge`. Different feeds have different heartbeats, so the threshold should be configurable per deployment.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a Pyth contract and a price feed ID.
2. Do not call `updateFeeds()` for a period exceeding any reasonable staleness threshold (e.g., 1 hour).
3. Call `latestAnswer()` — it returns the price from 1+ hour ago with no revert.
4. Call `latestRoundData()` — it returns `updatedAt = price.publishTime` from 1+ hour ago with no revert.
5. A downstream protocol consuming this adapter without its own staleness check will use the stale price in all financial operations. [6](#0-5)

        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
