### Title
Tx-Pool `total_tx_size`/`total_tx_cycles` Inflated After Cell-Dep Eviction in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the global `total_tx_size` and `total_tx_cycles` counters are computed into local variables **before** `check_and_record_ancestors` runs. When that function evicts cell-dep-parent transactions via `remove_entry_and_descendants`, each eviction correctly decrements `self.total_tx_size`/`self.total_tx_cycles` through `update_stat_for_remove_tx`. However, immediately afterward, `add_entry` **overwrites** those decremented values with the stale pre-eviction snapshot. The result is that the pool's size/cycle accounting is permanently inflated by the aggregate size and cycles of every evicted transaction, causing `limit_size` to over-evict valid transactions from the pool.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, `add_entry` executes in this order:

```rust
// Step 1: snapshot pre-eviction totals + new entry
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//  ^ local vars = self.total_tx_size + entry.size
//                 self.total_tx_cycles + entry.cycles

// Step 2: may evict N transactions via remove_entry_and_descendants,
//         each calling update_stat_for_remove_tx → decrements self.total_tx_size/cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3: OVERWRITES the post-eviction self.total_tx_size/cycles
//         with the stale pre-eviction snapshot
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

The eviction path inside `check_and_record_ancestors` calls `remove_entry_and_descendants` for each `cell_ref_parent` that must be expelled to satisfy `max_ancestors_count`: [2](#0-1) 

Each `remove_entry_and_descendants` → `remove_entry` call invokes `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`: [3](#0-2) [4](#0-3) 

But those decrements are immediately discarded when `add_entry` writes back the stale local variables at lines 218–219. The comment on `update_stat_for_remove_tx` even acknowledges the accounting is fragile: *"cycles overflow is possible, currently obtaining cycles is not accurate"*. [5](#0-4) 

**Concrete example:**
- `self.total_tx_size = 1000`, new entry size = 100 → `total_tx_size` (local) = 1100
- `check_and_record_ancestors` evicts 3 transactions of size 200 each → `self.total_tx_size` becomes 400
- Line 218 writes `self.total_tx_size = 1100` (should be 500)
- Pool now believes it holds 1100 bytes instead of 500 — inflated by 600

---

### Impact Explanation

`limit_size` uses `total_tx_size` as the sole trigger for eviction:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee pending transactions
}
``` [6](#0-5) 

With an inflated counter, `limit_size` will evict additional valid, potentially high-fee transactions that should remain in the pool. This:

1. Degrades miner revenue (valid high-fee transactions are expelled).
2. Causes legitimate user transactions to be silently dropped and rejected with `Reject::Full`.
3. Can be exploited repeatedly to keep the pool artificially depleted.

The `total_tx_cycles` inflation similarly distorts fee-rate estimation via `estimate_fee_rate`, which iterates over pool entries and uses per-entry cycle counts — but the pool-level counter is also used for admission checks. [7](#0-6) 

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is triggered whenever:

1. A transaction is submitted whose input consumes a cell output that is currently referenced as a `cell_dep` by ≥ 1 pool transactions, AND
2. The resulting ancestor count exceeds `max_ancestors_count` (default 1000). [8](#0-7) 

This is a realistic scenario: popular lock/type scripts (e.g., secp256k1, UDT type scripts) are widely used as cell deps. An attacker can:

1. Submit many transactions referencing a specific cell dep (all accepted normally).
2. Submit a transaction that **spends** that cell dep's output as an input, triggering mass eviction of the cell-dep-parent transactions.
3. The counter inflation persists for the lifetime of the pool until the next `clear()` or reorg.

The attack requires only standard RPC access (`send_transaction`) — no privileged role, no key material, no majority hashpower.

---

### Recommendation

Compute the updated totals **after** `check_and_record_ancestors` completes, so that evictions are already reflected in `self.total_tx_size`/`self.total_tx_cycles` before the new entry's contribution is added:

```rust
// Remove the pre-eviction snapshot entirely.
// check_and_record_ancestors handles its own decrements via update_stat_for_remove_tx.
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Add only the new entry's contribution on top of the already-decremented totals.
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

Alternatively, move `updated_stat_for_add_tx` to after `check_and_record_ancestors` so the snapshot is taken from the post-eviction state.

---

### Proof of Concept

Setup (pseudocode using CKB RPC):

```
1. Mine enough blocks to get spendable cells.
2. Create tx_root spending cell_A, producing output_X (a script cell).
3. Submit tx_root → pool.
4. Create 1001 transactions tx_1..tx_1001, each spending independent inputs
   but with cell_dep pointing to output_X (tx_root output 0).
5. Submit tx_1..tx_1001 → pool. All accepted.
   pool.total_tx_size ≈ 1001 * avg_size.
6. Create tx_consume spending output_X as an INPUT (not cell_dep).
7. Submit tx_consume → pool.
   - check_and_record_ancestors detects 1001 cell_ref_parents.
   - Evicts enough of them (e.g., 2 transactions) via remove_entry_and_descendants.
   - update_stat_for_remove_tx decrements self.total_tx_size by 2 * avg_size.
   - add_entry then OVERWRITES self.total_tx_size with the pre-eviction snapshot + tx_consume.size.
   - Net inflation: 2 * avg_size bytes.
8. Query get_tx_pool_info → total_tx_size is inflated.
9. Repeat step 6-7 with fresh transactions to accumulate inflation until
   total_tx_size > max_tx_pool_size, triggering limit_size to evict valid transactions.
``` [9](#0-8) [10](#0-9) [11](#0-10)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
