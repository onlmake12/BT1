Audit Report

## Title
`debug_assert!`-Only Guards Allow Divide-by-Zero Panic in Release Builds via Crafted Chain Spec — (`spec/src/consensus.rs`)

## Summary
`ConsensusBuilder::build()` uses `debug_assert!` exclusively to guard the invariants that `genesis_epoch_length` and `epoch_duration_target` are non-zero. These macros expand to nothing in release builds. A local operator who supplies a chain spec with either value set to `0` will trigger an integer divide-by-zero panic in `build_genesis_epoch_ext()` at node startup in a release binary, with no graceful error or rejection.

## Finding Description
In `spec/src/consensus.rs` lines 337–345, the only guards for `initial_primary_epoch_reward` and `epoch_duration_target` are `debug_assert!` macros:

```rust
debug_assert!(
    self.inner.initial_primary_epoch_reward != Capacity::zero(), ...
);
debug_assert!(
    self.inner.epoch_duration_target() != 0, ...
);
``` [1](#0-0) 

There is no corresponding guard for `genesis_epoch_length` at all. In release builds, all four `debug_assert!` calls in `build()` are compiled out entirely.

`Params::genesis_epoch_length()` and `Params::epoch_duration_target()` in `spec/src/lib.rs` perform no zero-check — they return the raw `unwrap_or_else(default)` value, so an explicitly-set `0` passes through unchanged: [2](#0-1) 

`build_consensus()` at lines 562–568 passes these values directly into `build_genesis_epoch_ext()` without any intermediate validation: [3](#0-2) 

`build_genesis_epoch_ext()` then performs integer division by both values with no zero-guard:

```rust
let block_reward = Capacity::shannons(epoch_reward.as_u64() / genesis_epoch_length); // div/0
let remainder_reward = Capacity::shannons(epoch_reward.as_u64() % genesis_epoch_length); // div/0
...
let genesis_hash_rate = ... / epoch_duration_target; // div/0
``` [4](#0-3) 

The full call chain — spec parse → `Params` accessor → `build_consensus()` → `build_genesis_epoch_ext()` — contains no zero-check that survives a release build.

## Impact Explanation
This is a **local command line crash** (Note, 0–500 points). The crash is triggered entirely by the local operator supplying a crafted or misconfigured spec file. It does not affect other nodes on the network, cannot be triggered remotely, and does not cause consensus deviation on the main CKB network whose spec is hardcoded. The impact matches the allowed bounty impact: *Any local command line crash*.

## Likelihood Explanation
Requires a local user to explicitly set `genesis_epoch_length = 0` or `epoch_duration_target = 0` in a custom spec file and run `ckb init --import-spec` followed by `ckb run`. Omitting the field is safe (defaults are non-zero). Only affects custom/dev chain configurations; mainnet and testnet specs are not user-supplied. Accidental misconfiguration is plausible; deliberate self-inflicted crash is trivially reproducible.

## Recommendation
Replace the four `debug_assert!` calls in `ConsensusBuilder::build()` with `assert!`, or convert `build()` to return `Result<Consensus, SpecError>` so invariants are enforced in release builds. Add explicit zero-checks in `build_genesis_epoch_ext()` before each division. Add range validation in `Params::genesis_epoch_length()`, `Params::epoch_duration_target()`, and `Params::orphan_rate_target()` to reject zero denominators at parse time.

## Proof of Concept
1. Create `bad_spec.toml` identical to `resource/specs/dev.toml` but add `genesis_epoch_length = 0` under `[params]`.
2. Run `ckb init --chain dev --import-spec bad_spec.toml --force`.
3. Run the release binary: `ckb run`.
4. The process panics immediately in `build_genesis_epoch_ext` at line 222 (`epoch_reward.as_u64() / genesis_epoch_length`) because the `debug_assert!` at lines 337–345 was compiled out.
5. Repeat with `epoch_duration_target = 0` to trigger the division at line 229.
