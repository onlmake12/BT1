### Title
Missing Payload Timestamp Staleness Validation in `PythLazer.verifyUpdate()` Enables Stale Price Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` verifies only the ECDSA signature and signer key expiry. It never compares the payload's embedded `timestamp` field against `block.timestamp`. Any unprivileged caller can replay an arbitrarily old but validly-signed Lazer payload, and the contract will accept it as valid. The `PythLazerLib` parsing layer also provides no staleness-checking getter, leaving consumer contracts with zero protocol-level protection against stale price data.

---

### Finding Description

`PythLazer.verifyUpdate()` performs two checks before returning the raw payload:

1. ECDSA signature recovery — the payload hash must be signed by a known key.
2. Signer expiry — `block.timestamp < trustedSignerToExpiresAtMapping[signer]`. [1](#0-0) 

Neither check involves the `timestamp` field embedded inside the payload itself. The payload header encodes a `uint64 timestamp` (microseconds since epoch), parsed by `PythLazerLib.parsePayloadHeader()` and stored in `Update.timestamp`. [2](#0-1) [3](#0-2) 

After `verifyUpdate()` returns, `PythLazerLib.parseUpdateFromPayload()` parses the timestamp but performs no comparison against `block.timestamp`. The safe getter `getPrice()` also performs no staleness check — it only asserts that the property was requested and is present: [4](#0-3) 

By contrast, Pyth Core provides `getPriceNoOlderThan(priceId, age)` which enforces a configurable maximum age at the protocol level. Pyth Lazer EVM has no equivalent.

The official integration guide example shows only a monotonicity check (`_timestamp > timestamp`) with no comparison to `block.timestamp`: [5](#0-4) 

---

### Impact Explanation

An unprivileged relayer who observed a favorable price in the past (e.g., a price significantly higher or lower than the current market price) can:

1. Retain the raw signed `update` bytes from the Lazer websocket stream.
2. Call a consumer contract's `updatePrice(update)` at any future time.
3. The consumer calls `pythLazer.verifyUpdate{value: fee}(update)` — this passes because the signature is valid and the signer key has not yet expired.
4. The consumer parses the payload and stores the stale price.
5. The attacker exploits the price discrepancy (e.g., undercollateralized borrow, incorrect liquidation, mispriced derivative).

The window of exploitability is bounded only by the signer key's `expiresAt` value, which is set far in the future (e.g., `3000000000000000` in tests). [6](#0-5) 

---

### Likelihood Explanation

- **Entry path is fully unprivileged**: any `msg.sender` can call `verifyUpdate()` with a fee of 1 wei.
- **Old signed payloads are publicly observable**: the Lazer websocket stream is accessible to all subscribers; historical payloads are not invalidated on-chain.
- **Integration guide does not show a staleness check**: consumer contracts following the official example will be vulnerable by default.
- **No protocol-level mitigation exists** in the Lazer EVM stack (unlike Pyth Core's `getPriceNoOlderThan`).

---

### Recommendation

Add a configurable `maxAge` parameter to `verifyUpdate()` (or a new `verifyUpdateNoOlderThan(bytes calldata update, uint64 maxAge)` variant) that parses the payload timestamp and enforces:

```solidity
uint64 payloadTimestampSec = payloadTimestampUs / 1_000_000;
require(
    block.timestamp <= payloadTimestampSec + maxAge,
    "Stale Lazer payload"
);
require(payloadTimestampSec <= block.timestamp + FUTURE_TOLERANCE, "Future timestamp");
```

Additionally, add a `getPriceNoOlderThan(Feed memory feed, Update memory update, uint64 maxAgeSec)` helper to `PythLazerLib` that mirrors the Pyth Core staleness API, so consumers have a safe, audited path to follow.

---

### Proof of Concept

```solidity
// Attacker replays a 10-minute-old signed payload to a vulnerable consumer

// 1. At T=0: attacker records a favorable signed update from the Lazer stream
bytes memory oldUpdate = <signed_payload_from_10_minutes_ago>;

// 2. At T=600s: attacker calls the consumer contract
// Consumer internally calls:
(bytes calldata payload, ) = pythLazer.verifyUpdate{value: 1 wei}(oldUpdate);
// ✅ passes — signature valid, signer not expired

PythLazerStructs.Update memory parsed = PythLazerLib.parseUpdateFromPayload(payload);
// parsed.timestamp == T-600s (stale), but no revert occurs

int64 price = PythLazerLib.getPrice(parsed.feeds[0]);
// ✅ returns the 10-minute-old price — no staleness check

// 3. Consumer uses stale price for collateral/liquidation logic
// Attacker profits from the price discrepancy
```

The root cause is confirmed at:
- `PythLazer.verifyUpdate()` — no `block.timestamp` vs payload timestamp comparison. [1](#0-0) 
- `PythLazerLib.parseUpdateFromPayload()` — timestamp parsed but never validated against wall clock. [2](#0-1) 
- `PythLazerLib.getPrice()` — no staleness guard. [4](#0-3)
