Audit Report

## Title
`unknown_tx_hashes` Queue Grows Unbounded Beyond `MAX_UNKNOWN_TX_HASHES_SIZE` Due to Post-Insertion Check and Missing Disconnect Cleanup — (`sync/src/types/mod.rs`)

## Summary

`SyncState::add_ask_for_txs` inserts all peer-supplied hashes into the global `unknown_tx_hashes` `KeyedPriorityQueue` before checking the size limit. When the limit is exceeded the offending peer is banned, but the already-inserted entries are never removed — neither at ban time nor on disconnect. An attacker controlling N relay connections can grow the queue to N × 32,767 entries, exhausting heap memory and crashing the node.

## Finding Description

The insertion loop in `add_ask_for_txs` unconditionally inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) new entries per call: [1](#0-0) 

Only after the loop completes does the function test whether the queue has grown past `MAX_UNKNOWN_TX_HASHES_SIZE` (50,000): [2](#0-1) 

When the threshold is exceeded and the current peer's contribution reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, the function returns `StatusCode::TooManyUnknownTransactions`, banning the peer. However, the entries that peer just inserted are **not removed**. The `disconnected` handler that fires when the ban takes effect also performs no cleanup of `unknown_tx_hashes`: [3](#0-2) 

Concrete growth sequence with distinct hashes per peer:
- Peer 1: inserts 32,767 → queue = 32,767 (< 50,000, check does not fire, `Status::ok()`)
- Peer 2: inserts 32,767 → queue = 65,534 (≥ 50,000, peer 2 banned, **entries remain**)
- Peer 3: inserts 32,767 → queue = 98,301 (peer 3 banned, **entries remain**)
- …

The constants governing the limits are: [4](#0-3) 

The entry point `TransactionHashesProcess::execute` calls `add_ask_for_txs` directly after only a per-message count check (≤ 32,767) and a `tx_filter` dedup pass — no pre-insertion size guard: [5](#0-4) 

## Impact Explanation

With N attacker-controlled relay connections each sending one `RelayTransactionHashes` message of 32,767 distinct hashes, the queue grows to N × 32,767 entries. Each entry holds a 32-byte `Byte32` key plus an `UnknownTxHashPriority` (an `Instant`, a `Vec<PeerIndex>`, and a `bool`). At 128 peers (`MAX_RELAY_PEERS`) the queue reaches ~4.2 M entries (~420 MB). The queue is protected by a single `Mutex`, so lock contention also degrades throughput during the attack. At the upper bound the node OOM-crashes or becomes unresponsive.

This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" — High (10,001–15,000 points)**.

## Likelihood Explanation

The attacker only needs to open multiple inbound or outbound relay connections to a post-IBD node and send one maximum-size `RelayTransactionHashes` message per connection. No proof-of-work, no keys, no privileged access is required. The per-peer rate limiter (30 msg/s) does not prevent a single large batch from being delivered. The existing integration test confirms the ban path fires but tests only a single peer and does not assert that the queue length stays ≤ 50,000 after the ban: [6](#0-5) 

## Recommendation

1. **Pre-insertion guard**: before inserting any hashes, check `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE`; if so, reject immediately without inserting.
2. **Cleanup on disconnect/ban**: in `SyncState::disconnected`, iterate `unknown_tx_hashes` and remove entries whose `peers` list becomes empty after removing the disconnected peer index.
3. **Tighten the per-peer limit**: enforce the per-peer cap (`MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`) as a pre-check rather than a post-check.

## Proof of Concept

Extend the existing test at `test/src/specs/relay/too_many_unknown_transactions.rs`:

```rust
// Connect 10 peers, each sending 32_767 distinct hashes
for peer_id in 0..10 {
    let net = connect_new_relay_peer(node0);
    net.send(node0, SupportProtocols::RelayV3, build_relay_tx_hashes(&unique_hashes(32767)));
    // wait for ban
    wait_until(60, || node0.rpc_client().get_banned_addresses().len() == peer_id + 1);
}
// Read internal queue length via a debug RPC or instrumented build
// Currently this assertion FAILS, confirming the invariant is broken:
assert!(node0.unknown_tx_hashes_len() <= MAX_UNKNOWN_TX_HASHES_SIZE);
```

The queue will contain ~327,670 entries (>> 50,000), confirming unbounded growth.

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "TxHashes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    relay_transaction_hashes.tx_hashes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
