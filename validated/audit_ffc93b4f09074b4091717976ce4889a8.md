Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Stale Snapshot Overwrite After Ancestor-Eviction in `add_entry` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` snapshots prospective new totals via `updated_stat_for_add_tx` before calling `check_and_record_ancestors`, which may evict entries and correctly decrement `self.total_tx_size` in place. Lines 218–219 then unconditionally overwrite `self.total_tx_size` with the pre-eviction snapshot, permanently inflating it by the byte-size of evicted entries per trigger. Repeated triggering inflates `total_tx_size` past `max_tx_pool_size`, causing `limit_size` to evict all legitimate pool entries and subsequently spin in an infinite loop when the pool is empty but the phantom total remains elevated.

## Finding Description

**Root cause — stale snapshot overwrite:**

`add_entry` (lines 210–219) snapshots `total_tx_size` before any eviction occurs:

```rust
// line 210-211: snapshot taken while self.total_tx_size == T
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// line 213: may evict, decrementing self.total_tx_size to T - E
evicts = self.check_and_record_ancestors(&mut entry)?;
// ...
// lines 218-219: unconditionally overwrites with stale T + entry.size
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) takes `&self` (immutable borrow) and returns `self.total_tx_size + tx_size` as a local — it does not mutate `self`.

`check_and_record_ancestors` (lines 603–625) enters the eviction branch when `ancestors_count > max_ancestors_count` and `cell_ref_parents` can reduce the count. It calls `remove_entry_and_descendants` (line 618), which calls `remove_entry` (lines 235–250), which calls `update_stat_for_remove_tx` at line 247. `update_stat_for_remove_tx` (lines 733–758) correctly decrements `self.total_tx_size` in place.

After `check_and_record_ancestors` returns, `self.total_tx_size` correctly equals `T − E`. Lines 218–219 then overwrite this with `T + entry.size`. The net inflation per trigger is exactly `E` (total byte-size of evicted entries).

**Secondary consequence — infinite loop in `limit_size`:**

`limit_size` (lines 292–329 of `pool.rs`) loops:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    if let Some(id) = next_evict_entry() {
        // evict and decrement total_tx_size
    }
    // no else { break; }
}
```

Once the pool is emptied by eviction but `total_tx_size` still exceeds `max_tx_pool_size` due to phantom inflation, `next_evict_entry()` returns `None`, the `if let` body is skipped, and the while condition remains true forever. The thread processing the transaction hangs indefinitely.

**Why existing guards are insufficient:**

`recompute_total_stat` (lines 698–708) is only invoked on underflow inside `update_stat_for_remove_tx` — it is never called on the inflation path. There is no post-`add_entry` invariant check. The overflow guard in `updated_stat_for_add_tx` only rejects if `usize` wraps, not if the value is logically stale.

## Impact Explanation

Two concrete impacts, both High:

1. **CKB network congestion with few costs**: Repeated triggering inflates `total_tx_size` until `limit_size` evicts all legitimate pending/proposed transactions. Re-submitted honest transactions are immediately evicted on the next `limit_size` call. Targeting multiple nodes causes network-wide transaction congestion.

2. **CKB node crash/hang**: Once phantom inflation exceeds `max_tx_pool_size` and the pool is drained, `limit_size` enters an infinite loop, permanently hanging the tx-pool processing thread. The node becomes unresponsive to new transactions.

Both match the High bounty tier (10001–15000 points).

## Likelihood Explanation

An unprivileged caller via the `send_transaction` RPC can construct the trigger:

- Build a 24-deep chain (each tx spending the previous output): `ancestors_count` reaches 25 = `max_ancestors_count`.
- Submit `tx_dep` using the chain-tip output as a **cell dep** (not consuming it). `tx_dep` becomes a `cell_ref_parent` tracked in `edges.deps`.
- Submit `tx_25` spending the chain-tip output as an **input**. `ancestors_count` becomes 26 > 25; `cell_ref_parents = {tx_dep}`; condition at line 603 is satisfied; eviction fires.
- `remove_entry_and_descendants(tx_dep)` decrements `self.total_tx_size` by `size(tx_dep)`, then lines 218–219 restore the stale snapshot, inflating by `size(tx_dep)`.

The attack requires ~26 funded outputs per cycle and is repeatable indefinitely with fresh outputs. No privileged access is needed. After `ceil(max_tx_pool_size / size(tx_dep))` cycles the node hangs.

## Recommendation

Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` so the base value already reflects any evictions:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size now reflects evictions; compute new totals from current state
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Additionally, add an `else { break; }` branch in `limit_size` after the `if let Some(id)` block to prevent the infinite loop when the pool is empty but `total_tx_size` is still inflated. Add a debug-build invariant assertion after every `add_entry` that `total_tx_size == entries.iter().map(|e| e.inner.size).sum::<usize>()`.

## Proof of Concept

1. Configure a node with `max_ancestors_count = 25`, `max_tx_pool_size = 20 MB`.
2. Submit `tx_0` creating `cell_0`.
3. Submit `tx_1…tx_24` as a chain, each spending the previous output. Pool now has 25 entries; `ancestors_count` for the chain tip = 25.
4. Submit `tx_dep` using `tx_24`'s output as a **cell dep**. `tx_dep` is recorded in `edges.deps` as a `cell_ref_parent`.
5. Submit `tx_25` spending `tx_24`'s output as an **input**.
   - `get_tx_ancenstors` returns `ancestors_count = 26`, `cell_ref_parents = {tx_dep}`.
   - Line 603 condition: `26 - 1 = 25 <= 25` — eviction branch fires.
   - `remove_entry_and_descendants(tx_dep)` → `update_stat_for_remove_tx` decrements `self.total_tx_size` by `size(tx_dep)`.
   - Lines 218–219 restore stale snapshot: net inflation = `size(tx_dep)`.
6. Assert: `pool_map.total_tx_size > pool_map.entries.iter().map(|e| e.inner.size).sum::<usize>()` — invariant broken.
7. Repeat steps 2–5 with fresh outputs. Each iteration inflates `total_tx_size` by `size(tx_dep)`.
8. After `ceil(20_000_000 / size(tx_dep))` iterations, `limit_size` fires, drains the pool, then spins forever — node hangs.

A unit test can reproduce this by directly calling `pool_map.add_entry` in a loop with the described structure and asserting the size invariant after each insertion.
