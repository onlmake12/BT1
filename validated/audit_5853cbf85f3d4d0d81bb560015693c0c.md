### Title
Governance VAA Reordering Allows Permanent Skipping of Legitimate Governance Actions — (`target_chains/ethereum/contracts/contracts/pyth/PythGovernance.sol`)

---

### Summary

`executeGovernanceInstruction` is an unrestricted `public` function. The only sequence guard rejects VAAs with sequence ≤ `lastExecutedGovernanceSequence`, but does **not** enforce that VAAs are executed in strict ascending order (N, N+1, N+2, …). Any actor who possesses a valid, guardian-signed governance VAA with a higher sequence number can submit it first, permanently making any lower-sequence VAA unexecutable.

---

### Finding Description

`verifyGovernanceVM` performs three checks before accepting a governance VAA:

1. Wormhole guardian signature validity
2. Emitter matches the registered governance data source
3. `vm.sequence > lastExecutedGovernanceSequence` (strictly greater) [1](#0-0) 

After passing, it immediately writes the new sequence: [2](#0-1) 

There is no check that `vm.sequence == lastExecutedGovernanceSequence + 1`. Submitting sequence N+k (k > 1) is accepted and advances the watermark, permanently invalidating all sequences in the gap.

The entry point is fully public with no caller restriction: [3](#0-2) 

Wormhole VAAs are published on the Wormhole Guardian network and are publicly observable. In Pyth's multi-chain deployment, governance routinely emits multiple VAAs in sequence — some chain-specific (`targetChainId == chainId()`), some global (`targetChainId == 0`). Both pass the target check: [4](#0-3) 

An attacker who monitors the Wormhole network can obtain VAA N+1 (already signed by guardians, already public) and submit it on-chain before VAA N is relayed, advancing `lastExecutedGovernanceSequence` past N.

---

### Impact Explanation

VAA N is permanently unexecutable — `vm.sequence <= lastExecutedGovernanceSequence()` will always be true for it. If VAA N was a fee reduction, data source update, or valid period change, that governance action is silently dropped. The contract remains in its pre-N state indefinitely, with no recovery path short of issuing a new governance VAA (which itself could be reordered again). This matches the stated scope: griefing without profit motive, contract left in a degraded state. [5](#0-4) 

---

### Likelihood Explanation

- The attacker needs no special privilege — only the ability to monitor the public Wormhole Guardian API and submit an Ethereum transaction.
- The precondition (multiple signed governance VAAs in flight simultaneously) is realistic: Pyth governance targets dozens of chains; global VAAs (`targetChainId == 0`) are common and apply everywhere.
- Front-running is straightforward on any EVM chain with a public mempool or via direct RPC submission before a relayer's transaction lands.

---

### Recommendation

Enforce strict sequential execution by requiring:

```solidity
if (vm.sequence != lastExecutedGovernanceSequence() + 1)
    revert PythErrors.InvalidGovernanceMessage();
```

Alternatively, restrict `executeGovernanceInstruction` to a trusted relayer role, or implement a commit-reveal / time-lock scheme so that a pending VAA cannot be displaced by a later one before it is processed.

---

### Proof of Concept

State-transition test (pseudocode):

```solidity
// Setup: lastExecutedGovernanceSequence = 5
// VAA_6 = valid governance VAA, sequence=6, action=SetFee(low)
// VAA_7 = valid governance VAA, sequence=7, action=SetFee(high)

// Attacker front-runs: submit VAA_7 first
pyth.executeGovernanceInstruction(VAA_7);
// lastExecutedGovernanceSequence is now 7

// Legitimate relayer tries to submit VAA_6
vm.expectRevert(PythErrors.OldGovernanceMessage.selector);
pyth.executeGovernanceInstruction(VAA_6);
// VAA_6 (fee reduction) is permanently blocked
// Contract fee remains at the high value set by VAA_7
```

Both VAAs are valid Wormhole-signed messages from the registered governance emitter; the attacker forges nothing — they only reorder publicly available VAAs.
