### Title
Any RPC Caller Can Remove Another User's Pending Transaction from the Tx-Pool Without Ownership Verification — (`rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC method in `rpc/src/module/pool.rs` accepts only a `tx_hash` and unconditionally removes the matching transaction (and all its descendants) from the tx-pool. There is no check that the caller is the same party who submitted the transaction. Any RPC caller who can reach the JSON-RPC port can silently evict any other user's pending transaction from the pool, enabling targeted transaction censorship or full pool-clearing attacks.

---

### Finding Description

**Root cause**

`PoolRpcImpl::remove_transaction` at `rpc/src/module/pool.rs:662–669` takes a bare `tx_hash: H256` and immediately forwards it to the tx-pool controller with no caller-identity or ownership check:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [1](#0-0) 

The call chain proceeds without any authorization gate:

1. `TxPoolController::remove_local_tx` (service.rs:273–275) sends a `RemoveLocalTx` message. [2](#0-1) 

2. `TxPoolService::remove_tx` (process.rs:440–455) searches the verify-queue, orphan pool, and main pool and removes the entry — no submitter identity is stored or checked. [3](#0-2) 

3. `TxPool::remove_tx` (pool.rs:358–361) calls `pool_map.remove_entry_and_descendants`, which also removes every child transaction that depends on the evicted one. [4](#0-3) 

**Attacker-controlled entry path**

The `remove_transaction` method is declared in the `PoolRpc` trait and registered under the `Pool` module:

```rust
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
``` [5](#0-4) 

The `Pool` module is enabled by default in the production configuration:

```toml
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
``` [6](#0-5) 

The RPC server binds to `127.0.0.1:8114` by default, making it reachable by any local process or any co-located service (e.g., a web server on the same host that can be reached via SSRF). The scope explicitly lists "RPC caller" and "supported local CLI/RPC user" as valid unprivileged attacker profiles.

**Exploit flow**

1. Victim submits transaction `T` via `send_transaction`; it enters the pending pool.
2. Attacker calls `get_raw_tx_pool` (also in the `Pool` module, no auth) to enumerate all pending tx hashes.
3. Attacker calls `remove_transaction(T.hash)`.
4. `T` and all its descendants are silently removed from the pool. The victim's transaction is gone with no on-chain trace and no error returned to the victim.
5. Attacker can repeat continuously to prevent any transaction from ever being mined.

---

### Impact Explanation

- **Transaction censorship**: An attacker can target specific transactions (e.g., a DAO withdrawal, a time-sensitive HTLC unlock, or a high-fee replacement) and evict them before they are committed to a block.
- **Full pool drain**: By iterating `get_raw_tx_pool` and calling `remove_transaction` in a loop, an attacker can keep the pool empty, preventing all users' transactions from being confirmed on that node.
- **Descendant cascade**: Because `remove_entry_and_descendants` is called, removing one transaction also removes every child transaction that spends its outputs, amplifying the damage.
- **No on-chain footprint**: The removal is silent — the victim receives no error and must re-submit, only to be evicted again.

**Severity: High** — direct, unauthorized state mutation (pool contents) with concrete financial impact (transaction delay/censorship) and no privilege required beyond RPC port access.

---

### Likelihood Explanation

**High.** The `Pool` module is on by default. The only prerequisite is access to the RPC port, which is satisfied by any local process, any co-located service, or any operator who has (commonly) exposed the port to a LAN or a reverse proxy. The tx hashes needed to target a victim are freely enumerable via `get_raw_tx_pool`. No key material, no signature, and no special role is required.

---

### Recommendation

Record the submitter identity (e.g., a `local` flag already tracked in `TxEntry`, or the peer index for remote submissions) at insertion time and enforce that `remove_transaction` can only evict transactions that were submitted by the same caller context (local RPC vs. remote peer). At minimum, restrict `remove_transaction` to a separate, non-default RPC module (e.g., `Debug` or `Admin`) that operators must explicitly enable, so it is not exposed alongside the standard `Pool` module.

### Proof of Concept

```python
import requests, json

NODE = "http://127.0.0.1:8114"

def rpc(method, params):
    return requests.post(NODE, json={
        "id": 1, "jsonrpc": "2.0", "method": method, "params": params
    }).json()

# Step 1: enumerate all pending tx hashes (attacker has no special role)
pool = rpc("get_raw_tx_pool", [False])
pending_hashes = pool["result"]["pending"]

# Step 2: remove every pending transaction — including those submitted by other users
for tx_hash in pending_hashes:
    result = rpc("remove_transaction", [tx_hash])
    print(f"Removed {tx_hash}: {result['result']}")

# All victim transactions are now gone from the pool with no on-chain record.
# Victims must re-submit; attacker can loop to maintain a permanently empty pool.
```
