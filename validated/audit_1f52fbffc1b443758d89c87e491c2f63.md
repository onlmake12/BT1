### Title
`Entropy.revealWithCallback()` CALLBACK_FAILED Recovery Path Publicly Leaks `randomNumber` Before Callback Execution, Enabling Predictable Randomness Exploitation — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a `revealWithCallback` call fails due to the consumer's callback reverting (gas limit exceeded or logic error), the contract emits the final `randomNumber` in the public `CallbackFailed` event and transitions the request to `CALLBACK_FAILED` state. The request remains active and callable by anyone. In the subsequent recovery invocation, the callback receives the same `randomNumber` that was already publicly broadcast on-chain. This breaks the randomness guarantee: the consumer contract's `_entropyCallback` is executed with a value that any observer already knows, enabling front-running and state manipulation attacks before the retry is triggered.

---

### Finding Description

**Normal first-attempt flow (gasLimit10k != 0, CALLBACK_NOT_STARTED):**

`revealWithCallback` uses `excessivelySafeCall` to catch the callback revert. On failure, it emits `CallbackFailed` and `EntropyEventsV2.Revealed` — both containing the fully-computed `randomNumber` — and sets `callbackStatus = CALLBACK_FAILED`. [1](#0-0) 

The `CallbackFailed` event signature publicly broadcasts `userRandomNumber`, `providerRevelation`, and `randomNumber`: [2](#0-1) 

**Recovery path (CALLBACK_FAILED):**

On the next call, the condition `req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED` is `false`, so execution falls into the `else` branch. Here `clearRequest` is called first, then `_entropyCallback` is invoked **directly** (no error catching) with the same `randomNumber`: [3](#0-2) 

The `randomNumber` passed to `_entropyCallback` in the recovery path is **identical** to the one already emitted in `CallbackFailed`. Any on-chain observer has known this value since the first failed attempt.

**Anyone can trigger the retry:**

`revealWithCallback` is `public` with no `msg.sender` restriction: [4](#0-3) 

The guard only checks `callbackStatus == CALLBACK_NOT_STARTED || callbackStatus == CALLBACK_FAILED`: [5](#0-4) 

This means any unprivileged actor can call `revealWithCallback` with the already-known `providerContribution` and `userContribution` (both emitted in `CallbackFailed`) at any block of their choosing.

---

### Impact Explanation

Consumer contracts that set a `gasLimit` (via `requestV2` or `setDefaultGasLimit`) and whose callback temporarily fails enter a state where:

1. The `randomNumber` is publicly known on-chain.
2. The callback has not yet been applied to the consumer's state.
3. Any actor can choose **when** to trigger the retry.

An attacker who observes the `CallbackFailed` event can:
- Determine the exact outcome the consumer contract will produce when `_entropyCallback` is called with the known `randomNumber`.
- Manipulate the consumer contract's state (e.g., buy lottery tickets, adjust positions, change game state) to exploit the known outcome.
- Then trigger the retry at the optimal moment.

This is a direct analog to the H-04 replay attack: the "failed callback" is open to being retried by anyone under attacker-chosen conditions, with the random value already known. The randomness guarantee — the core security property of Entropy — is broken for any request that enters `CALLBACK_FAILED`.

---

### Likelihood Explanation

- Any provider that calls `setDefaultGasLimit` (a common configuration) causes all requests to use the gas-limit flow.
- Consumer callbacks can fail transiently (e.g., due to a dependency contract being paused, insufficient internal balance, or a logic bug that is later fixed).
- The `CallbackFailed` event is indexed and easily monitored by MEV bots and on-chain observers.
- The retry is permissionless — no privileged access is required.

---

### Recommendation

1. **Do not emit `randomNumber` in `CallbackFailed`**: Emit only the `sequenceNumber` and error data. The `randomNumber` can be recomputed by the requester off-chain from their own `userContribution` and the provider's `providerContribution` (which they already know).

2. **Alternatively, invalidate the request on callback failure**: Instead of keeping the request active in `CALLBACK_FAILED` state, clear the request and require the user to submit a new one. This eliminates the window between `randomNumber` revelation and callback execution.

3. **If the recovery path is kept**, document explicitly that consumer contracts in `CALLBACK_FAILED` state do **not** receive unpredictable randomness, and that any state-dependent logic in `_entropyCallback` is exploitable.

---

### Proof of Concept

**Setup:**
- Provider has `defaultGasLimit = 100_000`.
- Consumer contract `LotteryConsumer` accepts ticket purchases and uses `_entropyCallback` to pick a winner based on `randomNumber % numTickets`.
- Ticket purchases are open until `_entropyCallback` is called.

**Attack:**
1. User requests randomness via `requestV2`. Provider calls `revealWithCallback`.
2. `LotteryConsumer._entropyCallback` reverts (e.g., due to a transient bug or gas misconfiguration). `CallbackFailed` is emitted with `randomNumber = R`.
3. Attacker reads `R` from the `CallbackFailed` event.
4. Attacker computes `winner_index = R % numTickets` for any future ticket count.
5. Attacker purchases tickets until they hold ticket index `R % (currentTickets + attackerTickets)`.
6. Attacker calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)` — all values are public from the original events.
7. `LotteryConsumer._entropyCallback` executes with `randomNumber = R`, and the attacker wins.

The attacker controlled both the timing of the retry and the state of the consumer contract, exploiting the known `randomNumber`. [6](#0-5) [7](#0-6)
