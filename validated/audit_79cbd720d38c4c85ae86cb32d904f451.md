Audit Report

## Title
`pending_delivered` Entry Not Consumed After `ConnectionSync` Processing Allows Unbounded NAT Traversal Task Spawning - (File: `network/src/protocols/hole_punching/component/connection_sync.rs`)

## Summary
In `ConnectionSyncProcess::execute`, when the target node processes a `ConnectionSync` message, the `pending_delivered` map entry for `content.from` is read via `.get()` and cloned but never removed or cleared. The `forward_rate_limiter` permits 1 `ConnectionSync` per second per `(from, to, item_id)` tuple, and the `pending_delivered` entry persists for the full 5-minute `TIMEOUT` window, allowing an attacker to spawn up to 300 concurrent NAT traversal tasks — each internally running up to 24 concurrent `try_nat_traversal` futures for 30 seconds — exhausting file descriptors and outbound connection capacity on the victim node.

## Finding Description

`HolePunching` maintains `pending_delivered: HashMap<PeerId, PendingDeliveredInfo>` where `PendingDeliveredInfo = (Vec<Multiaddr>, u64)`. [1](#0-0) 

In `connection_request.rs`, `respond_delivered` inserts the entry with the attacker's listen addresses and a timestamp when the target processes a `ConnectionRequest` addressed to itself: [2](#0-1) 

A cooldown guard in `respond_delivered` prevents re-insertion within `HOLE_PUNCHING_INTERVAL` (2 minutes), but once inserted the entry persists for `TIMEOUT` = 5 minutes: [3](#0-2) 

In `connection_sync.rs`, when the target node is the `to` peer, it reads the entry with `.get()` and clones the listen addresses — **the entry is never removed or cleared after use**: [4](#0-3) 

This is structurally enforced: `ConnectionSyncProcess` holds only `protocol: &'a HolePunching` (an immutable reference), making it impossible to call `.remove()` or `.get_mut()` on `pending_delivered` within this handler: [5](#0-4) 

The only protection against repeated triggering is the `forward_rate_limiter`, keyed by `(content.from, content.to, self.msg_item_id)` at 1 request/second. Since `msg_item_id` is the fixed union discriminant for `ConnectionSync`, the effective key is constant for a given `(from, to)` pair: [6](#0-5) [7](#0-6) 

The `pending_delivered` entry only expires via the periodic `notify()` cleanup after `TIMEOUT` = 5 minutes from the **insertion** timestamp — not from each `ConnectionSync` invocation: [8](#0-7) 

Each `ConnectionSync` spawns a **single** async task that runs all `try_nat_traversal` futures concurrently via `select_ok`. With up to `ADDRS_COUNT_LIMIT = 24` attacker-supplied addresses, each spawned task runs up to 24 concurrent `try_nat_traversal` futures, each making a TCP connection attempt every ~200ms for 30 seconds (~150 attempts per future): [9](#0-8) [10](#0-9) 

## Impact Explanation

An attacker can spawn up to 300 tasks (1/second × 300 seconds per 5-minute window). Each task runs up to 24 concurrent `try_nat_traversal` futures for 30 seconds. At peak (~30 tasks active simultaneously), this yields up to 720 concurrent `try_nat_traversal` futures, each making ~150 TCP connection attempts — totalling up to ~108,000 outbound TCP attempts simultaneously. This exhausts the victim node's file descriptors, outbound connection slots, and CPU, causing a node crash or severe degradation.

**Impact class: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a direct P2P connection to the target node and the ability to send `HolePunching` protocol messages — both available to any unprivileged peer. The `ConnectionRequest` step is trivially satisfied since the attacker sets `to = target_peer_id` and is directly connected. No special privileges, keys, or majority hashpower are needed. The attack is repeatable every 2 minutes (re-insertion cooldown) with the `pending_delivered` entry remaining exploitable for the full 5-minute window.

## Recommendation

The root cause is that `ConnectionSyncProcess` holds only an immutable reference to `HolePunching`, preventing it from mutating `pending_delivered`. The fix requires either:

1. Change `ConnectionSyncProcess` to hold `protocol: &'a mut HolePunching` and remove or clear the entry after spawning NAT traversal:
```rust
// In connection_sync.rs, after runtime::spawn(...):
if let Some(entry) = self.protocol.pending_delivered.get_mut(&content.from) {
    entry.0.clear(); // Consume the listen addresses; keep timestamp for cooldown
}
```

2. Alternatively, maintain a separate `HashSet<PeerId>` (e.g., `nat_traversal_initiated`) tracking peers for which NAT traversal has already been initiated within the current session, and skip re-triggering if the peer is already present. This set would need to be cleared on the same `TIMEOUT` schedule as `pending_delivered`.

## Proof of Concept

```
Attacker (peer A) → directly connected to Target (peer T)

1. A sends: ConnectionRequest { from=A, to=T, listen_addrs=[addr1..addr24], max_hops=6 }
   T processes: respond_delivered(A, T, [addr1..addr24])
     → pending_delivered[A] = ([addr1..addr24], now)
     → sends ConnectionRequestDelivered back to A

2. A sends: ConnectionSync { from=A, to=T, route=[] }  ← 1 per second
   T processes: ConnectionSyncProcess::execute()
     → forward_rate_limiter.check_key((A, T, SYNC_ID)) → OK (1/sec)
     → pending_delivered.get(A) → Some([addr1..addr24])  ← NOT removed
     → runtime::spawn(select_ok([try_nat_traversal(addr1), ..., try_nat_traversal(addr24)]))
        ← 24 concurrent TCP loops per spawned task
     → returns Status::ok()

3. Repeat step 2 every second for 5 minutes:
   → 300 spawned tasks × 24 concurrent futures = 7,200 concurrent try_nat_traversal futures
   → Peak (~30 tasks active): 720 concurrent futures × ~150 TCP attempts = ~108,000 TCP attempts
   → File descriptor exhaustion → node crash
```
