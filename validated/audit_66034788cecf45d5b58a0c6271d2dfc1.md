### Title
Deprecated `.transfer()` in `verifyUpdate()` Excess-Fee Refund Causes Revert for Smart Contract Callers - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` uses `payable(msg.sender).transfer(...)` to refund excess ETH to callers who overpay the verification fee. The fixed 2,300 gas stipend imposed by `.transfer()` is insufficient for smart contract callers whose `receive()` or `fallback()` functions perform any non-trivial work, causing the entire `verifyUpdate()` call to revert and permanently blocking those contracts from consuming Pyth Lazer price updates.

### Finding Description
In `PythLazer.verifyUpdate()`, when `msg.value` exceeds `verification_fee`, the contract refunds the difference using the deprecated `.transfer()` primitive:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

`address.transfer()` forwards exactly 2,300 gas to the recipient. Since EIP-1884 (Istanbul hard fork), many common operations — including a single `SSTORE`, emitting an event, or calling another contract — cost more than 2,300 gas. Any smart contract integrator whose `receive()` or `fallback()` function performs such operations will have the refund revert, which bubbles up and reverts the entire `verifyUpdate()` call.

The official integration pattern documented for Pyth Lazer consumers is to call `verifyUpdate` from within a smart contract:

```solidity
function updatePrice(bytes calldata priceUpdate) public payable {
    uint256 verification_fee = pythLazer.verification_fee();
    (bytes calldata payload, ) = verifyUpdate{ value: verification_fee }(update);
``` [2](#0-1) 

If such a consumer contract passes `msg.value` directly (a common pattern to avoid computing the exact fee off-chain), and its `receive()` function does anything beyond a bare ETH acceptance, every call to `verifyUpdate()` will revert.

### Impact Explanation
Smart contract Lazer consumers that overpay `verifyUpdate()` and have a non-trivial `receive()` or `fallback()` function are completely unable to verify Lazer price updates. The revert is deterministic and unrecoverable without a contract upgrade on the consumer side. This constitutes a denial-of-service against an unprivileged Lazer updater/consumer — a class of user explicitly in scope.

### Likelihood Explanation
The Pyth Lazer EVM SDK is designed to be integrated into smart contracts. It is common practice for integrating contracts to forward `msg.value` directly rather than querying `verification_fee()` on every call. Contracts with storage-writing `receive()` functions (e.g., accounting for received ETH, emitting events) are standard. The combination is realistic and likely to occur in production integrations.

### Recommendation
Replace `.transfer()` with a low-level `.call()` and check the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

This forwards all available gas to the recipient, eliminating the 2,300 gas stipend restriction while still reverting on failure.

### Proof of Concept
1. Deploy a consumer contract with a `receive()` function that writes to storage (costs >2,300 gas):
   ```solidity
   contract Consumer {
       uint256 public received;
       receive() external payable { received += msg.value; } // SSTORE > 2300 gas
       function update(PythLazer lazer, bytes calldata data) external payable {
           lazer.verifyUpdate{value: msg.value}(data); // overpays
       }
   }
   ```
2. Call `Consumer.update()` with `msg.value > verification_fee` and a valid Lazer update payload.
3. Observe that `verifyUpdate()` reverts at line 76 of `PythLazer.sol` because the `.transfer()` to `Consumer` exceeds the 2,300 gas stipend, even though the signature and fee are both valid. [3](#0-2)
