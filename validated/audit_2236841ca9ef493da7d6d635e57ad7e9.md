### Title
Unbounded Loop in `constructProviderCommitment` Can Cause OOG Revert in `revealWithCallback` / `reveal` When Provider Has Not Set `maxNumHashes` — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol` contains an unbounded `while` loop in `constructProviderCommitment` that iterates `numHashes` times. `numHashes` is derived from the gap between the request's assigned sequence number and the provider's `currentCommitmentSequenceNumber`. When a provider has not configured `maxNumHashes` (which defaults to `0`, bypassing the cap check), this gap can grow arbitrarily large, causing `revealWithCallback` and `reveal` to revert out-of-gas.

This check is **skipped entirely when `maxNumHashes == 0`**, which is the default for every provider that has not explicitly called `setMaxNumHashes`. A provider who registers with a large `chainLength` but never calls `advanceProviderCommitment` or `setMaxNumHashes` will accumulate an ever-growing gap. Any request made against such a provider stores a large `numHashes` in the request struct. When `revealWithCallback` (callable by anyone) or `reveal` (callable by the requester) later invokes `revealHelper` → `constructProviderCommitment`, the loop runs `numHashes` iterations of `keccak256`, consuming gas proportional to the gap. At ~30 gas per `keccak256`, a gap of ~1 million sequence numbers exceeds a 30 M gas block limit, permanently bricking fulfillment of those requests.

`revealWithCallback` is explicitly open to any caller: [4](#0-3) 

`revealHelper` is the shared path that calls `constructProviderCommitment`: [5](#0-4) 

---

### Impact Explanation

Any in-flight randomness request whose stored `numHashes` exceeds the gas budget of a block can never be fulfilled. `revealWithCallback` and `reveal` will always revert OOG for that request. The user's funds (fee paid at request time) are permanently locked in the provider's accrued balance, and the callback is never delivered. This breaks the liveness guarantee of the Entropy protocol for affected requests.

---

### Likelihood Explanation

The condition is triggered whenever:
1. A provider registers with a large `chainLength` but does not call `setMaxNumHashes` (the default state for all providers).
2. The provider does not call `advanceProviderCommitment` frequently enough to keep the gap small.

Both conditions are realistic: `maxNumHashes` is an opt-in protection that many providers will not configure, and `advanceProviderCommitment` is an optional maintenance call. An unprivileged user can accelerate the gap by making many cheap requests (each request increments `providerInfo.sequenceNumber` by 1, widening the gap for subsequent requests). No privileged access is required.

---

### Recommendation

1. **Enforce a hard cap on `numHashes` unconditionally**, not only when `maxNumHashes != 0`. Either revert in `requestHelper` if the gap exceeds a protocol-level maximum, or default `maxNumHashes` to a safe value (e.g., `1000`) during `register`.
2. **Require providers to set `maxNumHashes` during registration** rather than making it optional.
3. Alternatively, replace the `while` loop in `constructProviderCommitment` with a check that reverts if `numHashes` exceeds a hard constant before iterating.

---

### Proof of Concept

1. Provider P calls `register(fee, commitment, meta, 2**32, uri)` — large chain, `maxNumHashes` defaults to `0`.
2. Attacker (or organic users) submits N = 500,000 `requestV2` calls against provider P. Each call increments `providerInfo.sequenceNumber`. Provider P never calls `advanceProviderCommitment`.
3. For the last request, `req.numHashes = 500,000 - 0 = 500,000`. The cap check is skipped because `providerInfo.maxNumHashes == 0`.
4. Provider P (or anyone) calls `revealWithCallback(P, lastSeqNum, userContrib, providerContrib)`.
5. `revealHelper` calls `constructProviderCommitment(500_000, providerContrib)`, which loops 500,000 × `keccak256` ≈ 15 M gas — approaching or exceeding the block gas limit depending on the chain.
6. The transaction reverts OOG. The request is permanently stuck; the user's fee is unrecoverable.

    function constructProviderCommitment(
        uint64 numHashes,
        bytes32 revelation
    ) internal pure returns (bytes32 currentHash) {
        currentHash = revelation;
        while (numHashes > 0) {
            currentHash = keccak256(bytes.concat(currentHash));
            numHashes -= 1;
        }
    }
