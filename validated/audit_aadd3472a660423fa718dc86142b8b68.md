Audit Report

## Title
`forward_rate_limiter` Bypassed via Attacker-Controlled `from`/`to` Fields — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`, where both `content.from` and `content.to` are deserialized directly from the attacker-supplied message body with no verification against the actual sending session. An attacker can trivially bypass this inner rate limiter by cycling through distinct fabricated `(from, to)` peer ID pairs, causing the node to forward up to 30 messages per second (the outer per-session cap) to other peers — all of which pass the inner limiter unchecked. The `forward_rate_limiter` hashmap also grows without bound during a session, as `retain_recent()` is only called on disconnect.

## Finding Description
In `network/src/protocols/hole_punching/mod.rs`, two rate limiters are initialized:

- **Outer limiter** (`rate_limiter`): keyed by `(session_id, msg.item_id())` at 30 req/s — uses the real, non-spoofable session ID.
- **Inner limiter** (`forward_rate_limiter`): keyed by `(content.from, content.to, msg_item_id)` at 1 req/s — uses attacker-supplied fields from the message body. [1](#0-0) 

The outer limiter is checked first using the real session: [2](#0-1) 

Then, in `ConnectionRequestProcess::execute()`, the inner limiter is checked using `content.from` and `content.to` — both deserialized from the message payload, with no assertion that `content.from` matches the peer ID of the actual sending session: [3](#0-2) 

The same pattern is present in `ConnectionRequestDeliveredProcess::execute()`: [4](#0-3) 

And in `ConnectionSyncProcess::execute()`: [5](#0-4) 

**Exploit path:**
1. Attacker connects as a normal P2P peer.
2. Attacker sends up to 30 `ConnectionRequest` messages/second (outer limiter allows this).
3. Each message carries a distinct fabricated `(from, to)` pair.
4. Each unique pair creates a fresh bucket in `forward_rate_limiter`, so all 30 pass the inner check.
5. The node forwards each message to `sqrt(total_peers)` other peers — the inner limiter provides zero protection.
6. The `forward_rate_limiter` hashmap grows by 30 entries/second per attacker connection; `retain_recent()` is only called on disconnect, not periodically. [6](#0-5) 

Note: The routing-loop claim in the submission is incorrect — the code does check `content.route.contains(self_peer_id)` and ignores already-routed messages, preventing loops. [7](#0-6) 

## Impact Explanation
This matches **High (10001–15000 points): "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

Each forwarded message is broadcast to `sqrt(N)` peers. At 30 req/s per attacker connection with N=100 peers, the node issues ~300 outbound forwarding sends/second. With multiple attacker connections, this multiplies linearly. Additionally, the unbounded hashmap growth (30 new entries/second/connection, never pruned mid-session) constitutes a slow memory exhaustion vector — at 30 req/s for 24 hours, a single connection produces ~2.6M entries consuming hundreds of MB.

## Likelihood Explanation
Only a standard P2P connection is required — no keys, stake, or privileged role. Crafting molecule-encoded messages with varying `from`/`to` byte sequences is straightforward with any P2P client. The inner rate limiter is never triggered, so no warning is logged and no ban is applied. The attack is repeatable and sustainable indefinitely.

## Recommendation
1. **Verify `content.from` against the actual session**: After parsing, assert that `content.from` equals the peer ID of the sending session. Reject (and ban) messages where they do not match.
2. **Re-key `forward_rate_limiter` by session**: Use `(peer_index, msg_item_id)` — the same key as the outer limiter — so the forwarding budget is tied to the real sender.
3. **Periodically prune the hashmap**: Call `forward_rate_limiter.retain_recent()` on the `CHECK_INTERVAL` timer in `notify()`, not only on disconnect.

## Proof of Concept
```
1. Connect to a CKB node on SupportProtocols::HolePunching.
2. In a loop at ≤30 iterations/second:
   a. Generate a fresh random PeerId for `from` (or increment a counter).
   b. Build a valid ConnectionRequest molecule message: from=<fresh_id>, to=<any_id>,
      max_hops=6, listen_addrs=[valid multiaddr with from's peer ID], route=[].
   c. Send to the node.
3. Observe: the node forwards each message to sqrt(N) peers (no TooManyRequests returned).
4. After T seconds: forward_rate_limiter hashmap contains T*30 entries;
   T*30*sqrt(N) forwarding sends have been issued to other peers.
```
