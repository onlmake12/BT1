### Title
`deviationThresholdBps` Has No Upper-Bound Check in `_validateSubscriptionParams`, Allowing Deviation Trigger to Be Permanently Disabled - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `_validateSubscriptionParams` function in the Pyth Pulse `Scheduler` contract validates `deviationThresholdBps` only for a lower bound (`!= 0`) when `updateOnDeviation` is `true`, but imposes no upper-bound check. Because `deviationThresholdBps` is a `uint32`, a subscriber can set it to `type(uint32).max` (4,294,967,295 bps ≈ 42,949,673%), making the deviation-based update condition permanently impossible to satisfy. This is the direct structural analog to the reported Notional vulnerability, where `oracleSlippagePercentOrLimit` was only bounded in one conditional branch.

### Finding Description

In `_validateSubscriptionParams`:

```solidity
if (
    params.updateCriteria.updateOnDeviation &&
    params.updateCriteria.deviationThresholdBps == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
```

The check only rejects `deviationThresholdBps == 0`. Any value from `1` to `type(uint32).max` is accepted without further validation.

The deviation comparison in `_validateShouldUpdatePrices` is:

```solidity
uint256 deviationBps = Math.mulDiv(
    numerator,
    10_000,
    denominator
);

if (
    deviationBps >= params.updateCriteria.deviationThresholdBps
) {
    return updateTimestamp;
}
```

The system uses `10_000` as the basis (10,000 bps = 100% price move). For any realistic price move, `deviationBps` is bounded by the magnitude of the move. Setting `deviationThresholdBps` to, e.g., `1_000_000` (10,000%) or `type(uint32).max` makes the condition unreachable in practice, silently disabling deviation-triggered updates while the subscription appears correctly configured with `updateOnDeviation = true`.

### Impact Explanation

A subscription created with `updateOnDeviation = true` and `deviationThresholdBps = type(uint32).max` will never satisfy the deviation condition. If `updateOnHeartbeat` is also `false` (deviation-only subscription), `updatePriceFeeds` will always revert with `UpdateConditionsNotMet`, meaning the subscription's stored prices are never refreshed. Any downstream consumer (DeFi protocol, liquidation engine, etc.) reading prices from this subscription via `getPricesNoOlderThan` or `getPricesUnsafe` will receive permanently stale data, which can lead to incorrect liquidations, mispriced collateral, or other financial losses for end users of those protocols.

### Likelihood Explanation

`createSubscription` is callable by any unprivileged address. There is no access control on who can create a subscription. A user can trivially pass `deviationThresholdBps = type(uint32).max` alongside `updateOnDeviation = true` and `updateOnHeartbeat = false`. The contract accepts this without revert. The misconfiguration is invisible to downstream readers of the subscription — the `SubscriptionParams` struct shows `updateOnDeviation = true`, implying active deviation monitoring, while in reality no update will ever occur.

### Recommendation

Add an upper-bound check on `deviationThresholdBps` in `_validateSubscriptionParams`, analogous to how `heartbeatSeconds` is bounded by the system's validity period. A reasonable maximum is `10_000` bps (100%), or at minimum a documented constant such as `MAX_DEVIATION_THRESHOLD_BPS`:

```diff
if (
    params.updateCriteria.updateOnDeviation &&
    params.updateCriteria.deviationThresholdBps == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
+if (
+    params.updateCriteria.updateOnDeviation &&
+    params.updateCriteria.deviationThresholdBps > MAX_DEVIATION_THRESHOLD_BPS
+) {
+    revert SchedulerErrors.InvalidUpdateCriteria();
+}
```

### Proof of Concept

1. Deploy `Scheduler` (or use a live instance).
2. Call `createSubscription` with:
   - `updateOnHeartbeat = false`, `heartbeatSeconds = 0`
   - `updateOnDeviation = true`, `deviationThresholdBps = type(uint32).max`
   - Valid `priceIds`, sufficient `msg.value`
3. Observe: `createSubscription` succeeds — no revert.
4. Call `updatePriceFeeds` with valid price data showing any realistic price move.
5. Observe: always reverts with `UpdateConditionsNotMet` because `deviationBps` (max ~10,000 for a 100% move) can never reach `type(uint32).max`.
6. The subscription's stored prices are never updated; downstream readers receive stale data indefinitely.
