Audit Report

## Title
Redundant Per-Input `block_median_time` Calls in `SinceVerifier::verify()` - (File: `verification/src/transaction_verifier.rs`)

## Summary
`SinceVerifier::verify()` calls `block_median_time(self.tx_env.parent_hash())` once per input inside the per-input loop in both `verify_absolute_lock` and `verify_relative_lock`. Because `self.tx_env.parent_hash()` is constant for all inputs of the same transaction, this results in O(N × 37) redundant header-chain traversals via `get_header_fields` for a transaction with N timestamp-`since` inputs, where a single pre-loop call would produce the identical result.

## Finding Description
**Root cause:** `SinceVerifier::verify()` (`verification/src/transaction_verifier.rs` L735–758) iterates over every resolved input and calls `verify_absolute_lock` and `verify_relative_lock` per input. Both functions, in their `SinceMetric::Timestamp` branches, independently call `self.tx_env.parent_hash()` (which returns the same value for every input in the transaction) and then `self.block_median_time(&parent_hash)`.

`SinceVerifier::block_median_time` (`verification/src/transaction_verifier.rs` L626–630) delegates to `HeaderFieldsProvider::block_median_time` (`traits/src/header_provider.rs` L32–50), which walks up to `MEDIAN_TIME_BLOCK_COUNT = 37` ancestors via sequential `get_header_fields` calls on every invocation. No result is cached at the `SinceVerifier` level between inputs.

**Why existing mitigations are insufficient:** The `StoreCache` LRU header cache prevents repeated RocksDB reads after the first call, but every subsequent call still performs a full O(37) traversal of the in-memory LRU with mutex acquisition per lookup. For a transaction with N timestamp-`since` inputs, this is O(N × 37) mutex-locked LRU lookups that are entirely redundant — the result is identical for every input because the argument (`parent_hash`) never changes within a single transaction verification.

**Exploit flow:**
1. Attacker creates N live cells (e.g., N = 1000).
2. Constructs a transaction spending all N cells with each input's `since` field set to an absolute timestamp value in the future (e.g., `0x4000_0000_FFFF_FFFF`).
3. Submits via `send_transaction` RPC.
4. `SinceVerifier::verify()` calls `block_median_time(&parent_hash)` N times (once per input in `verify_absolute_lock`), each traversing 37 headers.
5. Transaction is rejected as `Immature`, but the full O(N × 37) traversal cost is paid before rejection.
6. Because no cells are consumed on rejection, the same transaction can be resubmitted in a loop at negligible marginal cost.

## Impact Explanation
**Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism.** The header storage access path (`get_header_fields` → `DataLoaderWrapper` → `ChainStore`) is traversed O(N × 37) times per transaction verification where O(37) would suffice. Under sustained submission of large transactions with timestamp-`since` inputs, this amplifies CPU and mutex-contention cost in the tx-pool worker thread proportionally to input count. The LRU cache prevents RocksDB I/O amplification but does not eliminate the redundant traversal overhead.

## Likelihood Explanation
Any unprivileged user with access to the `send_transaction` RPC or P2P relay can trigger this. The one-time setup cost is acquiring N live cells. Because the transaction is rejected without consuming the cells, the same transaction can be resubmitted indefinitely. No special privileges, victim interaction, or external dependencies are required.

## Recommendation
Compute `block_median_time(self.tx_env.parent_hash())` once before the input loop and pass the cached result into `verify_absolute_lock` and `verify_relative_lock`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    let cached_median_time = self.block_median_time(&self.tx_env.parent_hash());
    for (index, (cell_meta, input)) in ... {
        ...
        self.verify_absolute_lock(index, since, cached_median_time)?;
        self.verify_relative_lock(index, since, cell_meta, cached_median_time)?;
    }
    Ok(())
}
```

`parent_median_time(&info.block_hash)` in the relative branch varies per input (depends on the cell's block hash) and cannot be cached globally, but `current_median_time` from `self.tx_env.parent_hash()` is invariant within a single transaction verification and should be computed exactly once.

## Proof of Concept
1. On a devnet, create N live cells (e.g., N = 1000).
2. Construct a transaction spending all N cells with each input's `since` field set to `0x4000_0000_FFFF_FFFF` (absolute timestamp, far future).
3. Submit via `send_transaction` RPC.
4. Instrument `SinceVerifier::block_median_time` with an invocation counter and log the `block_hash` argument.
5. Observe N invocations with the identical `parent_hash` argument before the `Immature` rejection is returned.
6. Repeat submission in a loop; confirm sustained CPU load on the tx-pool worker thread proportional to N.

    fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
        let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
        let mut block_hash = block_hash.clone();
        for _ in 0..median_block_count {
            let header_fields = self
                .get_header_fields(&block_hash)
                .expect("parent header exist");
            timestamps.push(header_fields.timestamp);
            block_hash = header_fields.parent_hash;

            if header_fields.number == 0 {
                break;
            }
        }

        // return greater one if count is even.
        timestamps.sort_unstable();
        timestamps[timestamps.len() >> 1]
    }
