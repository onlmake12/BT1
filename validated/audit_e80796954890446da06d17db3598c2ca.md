### Title
Synchronous RocksDB Write Under `tx_pool` Write Lock in `put_recent_reject` Enables Rejection-Flood Lock Starvation — (`tx-pool/src/process.rs`, `tx-pool/src/component/recent_reject.rs`)

---

### Summary

An unprivileged remote peer can flood the node with transactions that are rejected for any reason except `Duplicated`. Each such rejection causes `put_recent_reject()` to acquire the `tx_pool` Tokio async write lock and perform a **synchronous (blocking) RocksDB write** while holding it. Under a sustained rejection flood this serializes all concurrent tx-pool operations — block assembly, new-tx admission, RBF checks — against repeated blocking I/O, degrading throughput and block-assembly latency.

---

### Finding Description

**Entry point — `process_tx` → `after_process`**

`process_tx()` is the handler for both local and remote (relay) transaction submissions. [1](#0-0) 

After `submit_entry()` releases the write lock, `after_process()` is called. For every remote rejection that is not `Duplicated`, it calls `put_recent_reject()`: [2](#0-1) 

**`should_recorded()` admits almost every rejection type**

`should_recorded()` returns `true` for all variants except `Duplicated` — meaning `LowFeeRate`, `Resolve`, `Verification`, `Malformed`, `Full`, `RBFRejected`, `Expiry`, etc. are all recorded: [3](#0-2) 

**`put_recent_reject` holds the write lock across blocking I/O**

```rust
pub(crate) async fn put_recent_reject(&self, tx_hash: &Byte32, reject: &Reject) {
    let mut tx_pool = self.tx_pool.write().await;   // ← write lock acquired
    if let Some(ref mut recent_reject) = tx_pool.recent_reject
        && let Err(e) = recent_reject.put(tx_hash, reject.clone())  // ← blocking RocksDB write
    { ... }
}   // ← write lock released only here
``` [4](#0-3) 

`RecentReject::put()` performs a synchronous `self.db.put()` (RocksDB `PutCF`) while the write lock is held: [5](#0-4) 

**`shrink()` amplification — real but bounded by a counter bug**

`shrink()` additionally calls `drop_cf()`, `create_cf_with_ttl()`, and five `estimate_num_keys_cf()` calls — all blocking RocksDB operations: [6](#0-5) 

However, `self.total_keys_num` is **never incremented** inside `put()` — the local variable `total_keys_num` is computed but never written back to `self`. This means `shrink()` is only triggered when the node restarts with a DB already at or above `count_limit`, not continuously during a live flood. The question's claim of "shrink on every put" under a fresh flood is therefore inaccurate. [7](#0-6) 

The **base-case issue** — synchronous RocksDB write under the write lock — is present on every single rejection regardless of shrink.

---

### Impact Explanation

The `tx_pool` field is a `TokioRwLock<TxPool>`. Holding a Tokio async write lock while executing blocking I/O starves the async runtime: all concurrent tasks waiting for the read or write lock (block assembly via `get_block_template`, new-tx admission via `submit_entry`, RBF checks) are blocked for the duration of each RocksDB write. At 10 k rejections/second and ~100 µs per RocksDB write, the write lock is effectively held ~100% of the time, collapsing tx-pool throughput and increasing block-assembly latency.

---

### Likelihood Explanation

The attack requires no privilege. A remote peer can send transactions that are trivially rejected (e.g., `LowFeeRate` — craft a zero-fee transaction) via the standard P2P relay path. No PoW, no key material, no special configuration is needed. The only rate-limiting factor is the node's own network ingress and the verify-queue depth, both of which can be saturated with minimal effort.

---

### Recommendation

1. **Decouple the write from the lock**: move `recent_reject.put()` outside the `tx_pool` write lock. `RecentReject` owns its own `DBWithTTL` and does not need the pool lock for correctness; it can be protected by its own dedicated `Mutex` or called without any pool lock.
2. **Fix the counter bug**: increment `self.total_keys_num` in `put()` so that `shrink()` is triggered at the correct cadence and the counter reflects reality.
3. **Use `spawn_blocking`**: if the RocksDB write must remain inside an async context, wrap it in `tokio::task::spawn_blocking` to avoid blocking the async runtime thread.

---

### Proof of Concept

```
1. Connect to a CKB node as a P2P peer.
2. Relay a stream of transactions with fee = 0 (guaranteed LowFeeRate rejection,
   should_recorded() = true).
3. Observe: each rejection triggers put_recent_reject() → tx_pool.write() →
   db.put() (blocking RocksDB write) → lock release.
4. Concurrently call get_block_template RPC or submit a valid transaction.
5. Measure latency of get_block_template under 0 vs 10k rejection/s load.
6. Assert: p99 latency of get_block_template increases proportionally with
   rejection rate, confirming write-lock starvation.
```

    pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
        let hash_slice = hash.as_slice();
        let shard = self.get_shard(hash_slice).to_string();
        let reject: ckb_jsonrpc_types::PoolTransactionReject = reject.into();
        let json_string = serde_json::to_string(&reject)?;
        self.db.put(&shard, hash_slice, json_string)?;

        if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
            if total_keys_num > self.count_limit {
                self.shrink()?;
            }
        } else {
            // overflow occurred, try shrink
            self.shrink()?;
        }
        Ok(())
    }

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        let total_keys_num = self.estimate_total_keys_num()?;
        self.total_keys_num = total_keys_num;
        Ok(total_keys_num)
    }
