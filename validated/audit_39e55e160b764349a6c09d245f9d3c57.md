### Title
Inconsistent Underflow Guard in `ema_price_no_older_than` vs `price_no_older_than` Causes Unexpected Revert on Fuel - (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

---

### Summary

The Fuel Pyth contract applies a saturating-subtraction guard in `price_no_older_than` to prevent arithmetic underflow when a Pythnet timestamp is slightly ahead of the Fuel chain's block timestamp. The identical guard is absent from the parallel function `ema_price_no_older_than`. When `price.publish_time > current_time`, calling any EMA price function panics with an arithmetic underflow revert instead of returning a clean staleness error, making EMA prices temporarily unavailable to all consumers.

---

### Finding Description

In `target_chains/fuel/contracts/pyth-contract/src/main.sw`, the private helper `price_no_older_than` explicitly guards against underflow with a comment explaining the intent:

```sway
// Mimicking saturating subtraction to avoid underflow
let time_difference = if current_time > price.publish_time {
    current_time - price.publish_time
} else {
    0
};
require(time_difference <= time_period, PythError::OutdatedPrice);
``` [1](#0-0) 

The parallel private helper `ema_price_no_older_than` performs the same subtraction without any guard:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,
        PythError::OutdatedPrice,
    );
    price
}
``` [2](#0-1) 

Both `ema_price_no_older_than` and `ema_price` are part of the public `PythCore` ABI and are directly callable by any user: [3](#0-2) 

The public `ema_price` entry point delegates directly to the vulnerable helper: [4](#0-3) 

In Sway, unsigned integer subtraction that underflows causes an immediate VM panic (revert). When `price.publish_time > current_time` — a condition explicitly acknowledged elsewhere in the Pyth codebase as possible because "Pythnet timestamps may be slightly ahead of this chain" — the expression `current_time - price.publish_time` underflows and the transaction reverts with an arithmetic trap rather than the expected `PythError::OutdatedPrice`. The `price()` / `price_no_older_than()` path handles this correctly and returns a clean error.

---

### Impact Explanation

Any unprivileged caller invoking `ema_price()` or `ema_price_no_older_than()` during the window where a freshly-pushed Pythnet price has a `publish_time` slightly ahead of the Fuel chain's `block::timestamp()` receives an opaque arithmetic-underflow revert instead of a clean staleness error. This makes EMA price data temporarily unavailable to all consumers on Fuel, while the spot-price equivalents (`price()`, `price_no_older_than()`) continue to work correctly in the same scenario. Protocols that rely on EMA prices for liquidation guards, TWAP checks, or confidence-interval logic are silently broken during this window.

**Impact: Medium** — temporary, condition-dependent DoS of EMA price reads; no fund loss directly, but downstream protocols relying on EMA prices can be blocked from executing critical operations.

---

### Likelihood Explanation

The Pyth codebase itself acknowledges that Pythnet timestamps can be slightly ahead of target-chain block timestamps. The Scheduler contract comment reads: *"Use distance (absolute difference) since pythnet timestamps may be slightly ahead of this chain."* [5](#0-4) 

This is therefore a routine, expected condition — not a rare edge case. Any price update pushed to Fuel when Pythnet is even one second ahead of the Fuel node's clock triggers the underflow. No special privileges or attack setup are required; any user calling `ema_price()` in that window is affected.

**Likelihood: Medium** — the triggering condition is a normal operational state, but it is transient and self-resolving once the Fuel clock catches up.

---

### Recommendation

Apply the same saturating-subtraction guard to `ema_price_no_older_than` that already exists in `price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

This keeps the two functions consistent and ensures that a future-timestamped price is treated as fresh (time difference = 0) rather than causing a panic.

---

### Proof of Concept

1. A price update is submitted to the Fuel Pyth contract where `price.publish_time = block::timestamp() + 2` (Pythnet 2 seconds ahead of Fuel).
2. A consumer calls `ema_price(price_feed_id)`.
3. Internally, `ema_price_no_older_than(valid_time_period(), price_feed_id)` is called.
4. `current_time - price.publish_time` evaluates as `u64` underflow → VM arithmetic panic → transaction reverts.
5. The same consumer calls `price(price_feed_id)` — this succeeds because `price_no_older_than` uses the saturating guard, computes `time_difference = 0`, and the `require` passes.

The inconsistency is directly observable: identical inputs cause `price()` to succeed and `ema_price()` to revert.
