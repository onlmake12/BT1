Audit Report

## Title
`check_purge` Permanently Returns `EvictionFailed` When Peer Store Is Filled With Exactly 4 Peers Per Network Group — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

`check_purge` contains two compounding logic flaws: a strict `> 4` guard (instead of `>= 4`) and a `take(len/2)` that only examines half the network groups during eviction. An attacker who fills the peer store with exactly 16,384 addresses distributed across 4,096 distinct /16 subnets (4 per subnet) causes `check_purge` to permanently return `PeerStoreError::EvictionFailed` on every subsequent `add_addr` call, permanently blocking peer discovery for the victim node.

## Finding Description

**Root cause — two compounding flaws in `check_purge`:**

**Step 1** (`peer_store_impl.rs` L341–355): Collects non-connectable peers for eviction. Because `add_addr` always inserts with `last_connected_at_ms=0` and `attempts_count=0` [1](#0-0) 
and `AddrInfo::new` initializes `last_tried_at_ms=0`, `attempts_count=0` [2](#0-1) 
`is_connectable` returns `true` for any entry with `attempts_count < ADDR_MAX_RETRIES (3)`: [3](#0-2) 
Every adversarially-inserted address is connectable. Step 1 produces an empty `candidate_peers` and removes nothing.

**Step 2** (`peer_store_impl.rs` L358–401): Groups addresses by network segment (IPv4 /16 via `Group::IP4([bits[0], bits[1]])` [4](#0-3) 
), sorts by group size descending, then applies two flawed filters:
- `take(len / 2)` at L376 — only examines half the groups
- `if addrs.len() > 4` at L378 — strict greater-than guard [5](#0-4) 

With 4,096 groups of exactly 4 peers each:
- `len = 4096`, `take(2048)` — only 2,048 groups examined
- For every group: `4 > 4` → `false` → `None` returned
- `candidate_peers` remains empty
- L399–400: `return Err(PeerStoreError::EvictionFailed.into())` fires unconditionally [6](#0-5) 

**Persistence**: The poisoned state is written to `addr_manager.db` on shutdown and reloaded on startup, making the condition survive restarts. [7](#0-6) 

## Impact Explanation

Once the peer store is saturated in this configuration, every call to `add_addr` — the sole entry point for all P2P-discovered peer addresses — returns `Err(EvictionFailed)`. The node can no longer learn new peer addresses from the network. If existing connections drop (peers restart, network churn), the node cannot discover replacements and becomes progressively isolated from the honest network. A fully isolated node cannot receive new blocks, cannot relay transactions, and falls behind the chain tip. Applied at scale across multiple victim nodes, this constitutes **High — vulnerabilities or bad designs which could cause CKB network degradation with few costs**, as the attacker requires only standard P2P peer acceptance to execute the attack.

## Likelihood Explanation

The attack is reachable via the standard CKB discovery protocol. A single adversarial peer (or small set of colluding peers) can advertise up to 16,384 addresses across 4,096 /16 subnets via `GetNodes`/`Nodes` discovery messages. No special privilege, PoW, or key material is required — only acceptance as a P2P peer. The fill condition (4,096 × 4 = 16,384) is exact and deterministic. [8](#0-7) 
The state persists across restarts via `addr_manager.db`, meaning a one-time attack has permanent effect until the database is manually deleted.

## Recommendation

1. **Fix the off-by-one**: Change `addrs.len() > 4` to `addrs.len() >= 4` at `peer_store_impl.rs` L378, so groups of exactly 4 are eligible for eviction.
2. **Remove or justify `take(len/2)`**: There is no documented reason to examine only half the groups under eviction pressure. Consider examining all groups sorted by size.
3. **Add a guaranteed fallback**: If `candidate_peers` is still empty after both steps, fall back to evicting the oldest/least-recently-seen entries regardless of group size, to guarantee the invariant that the store can always accept new addresses when at capacity.

## Proof of Concept

```rust
#[test]
fn test_eviction_failed_with_4_per_group() {
    use crate::peer_store::{PeerStore, ADDR_COUNT_LIMIT};
    use crate::{Flags, PeerId};
    use p2p::multiaddr::Multiaddr;

    let mut peer_store = PeerStore::default();
    let mut count = 0;
    'outer: for a in 0u8..=255 {
        for b in 0u8..=255 {
            for port in 1u16..=4 {
                if count >= ADDR_COUNT_LIMIT { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/{}.{}.0.1/tcp/{}/p2p/{}",
                    a, b, port, PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(peer_store.addr_manager().count(), ADDR_COUNT_LIMIT);

    // Any new add_addr must now fail with EvictionFailed
    let new_addr: Multiaddr = format!(
        "/ip4/200.200.200.1/tcp/9999/p2p/{}",
        PeerId::random().to_base58()
    ).parse().unwrap();
    let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
    assert!(result.is_err(), "Expected EvictionFailed, got Ok");
}
```

The test fills the store with 4,096 groups × 4 connectable peers each (all `last_connected_at_ms=0`, `attempts_count=0`). The final `add_addr` triggers `check_purge`, which finds no non-connectable peers (step 1), then finds no groups with `len > 4` (step 2), and returns `Err(EvictionFailed)`. [9](#0-8)
