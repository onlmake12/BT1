### Title
Rich-Indexer Script Table Corruption on PostgreSQL During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

### Summary

A copy-paste bug in `script_exists_in_output` causes the type-script existence check to be silently skipped on PostgreSQL. During any chain reorg, `rollback_block` calls this function to decide whether to garbage-collect scripts from the `script` table. On PostgreSQL, the function always reads `row_lock` (the lock-script query result) a second time instead of reading `row_type` (the type-script query result), so the type-script check is never evaluated. Any script that is still referenced only as a `type_script_id` in remaining outputs is incorrectly deleted, permanently corrupting the rich indexer's `script` table and all subsequent type-script-based cell queries.

---

### Finding Description

`script_exists_in_output` executes two SQL `EXISTS` queries:

1. `row_lock` — checks `WHERE lock_script_id = $1`
2. `row_type` — checks `WHERE type_script_id = $1`

The function is supposed to return `true` if either query is true. The early-return after the first check is correct. The second match block, however, reads from the wrong variable:

```rust
// line 252 — BUG: row_lock used instead of row_type
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),                          // always Ok(false) on PostgreSQL
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),  // correct, but unreachable on PG
}
``` [1](#0-0) 

On **PostgreSQL**, `try_get::<bool, _>(0)` succeeds (PostgreSQL returns `BOOLEAN`). Because execution only reaches line 252 when the lock-script check already returned `false`, `row_lock` holds `false` at this point. The function therefore returns `Ok(false)` unconditionally, ignoring `row_type` entirely.

On **SQLite**, `try_get::<bool, _>(0)` fails (SQLite returns `BIGINT`), so the `Err(_)` branch executes `row_type.get::<i64, _>(0) == 1`, which is correct. SQLite is unaffected.

The caller in `rollback_block` uses the return value to decide whether to push the script ID into `script_id_list_to_remove`:

```rust
if let Some(type_script_id) = type_script_id
    && !script_exists_in_output(type_script_id, tx).await?
{
    script_id_list_to_remove.push(type_script_id);
}
``` [2](#0-1) 

On PostgreSQL, `script_exists_in_output` always returns `false` for any script that is not a lock script, so every `type_script_id` from the rolled-back block is unconditionally added to `script_id_list_to_remove` and deleted — even if it is still referenced by outputs in the remaining chain.

---

### Impact Explanation

After a reorg on a PostgreSQL-backed rich-indexer node:

- Scripts that are still referenced as `type_script_id` in surviving outputs are deleted from the `script` table.
- All subsequent RPC calls that join against the `script` table for type-script lookups (`get_cells`, `get_transactions`, `get_cells_capacity` with `script_type = "type"`) will silently return empty or incomplete results for those cells.
- The corruption is permanent until the indexer is fully rebuilt; there is no self-healing path.

The claimed "consensus deviation" label is inaccurate — this does not affect CKB consensus or block validation. The actual impact is **rich-indexer data corruption** on PostgreSQL deployments, causing incorrect query results for all type-script-based cell queries after any reorg.

---

### Likelihood Explanation

- Chain reorgs are a normal, unprivileged event: any peer can relay a valid competing chain with more cumulative work, triggering `rollback_block`.
- PostgreSQL is a documented, officially supported backend for the rich indexer.
- The bug fires on every reorg on every PostgreSQL deployment; no special conditions are required.

---

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0)

### Proof of Concept

Differential test on a PostgreSQL-backed rich-indexer node:

1. Index a block containing an output whose `type_script_id` is `S` and whose `lock_script_id` is a different script `L`. Another output in the same or a prior block also uses `S` as its `type_script_id`.
2. Trigger a reorg that rolls back the first block but keeps the second output (so `S` is still referenced).
3. After rollback, query `SELECT id FROM script WHERE id = <S>`.
4. **Expected**: row exists (script still referenced).
5. **Actual on PostgreSQL**: row is absent — `script_exists_in_output` returned `false` for `S` because `row_lock` (lock-script check, already `false`) was re-read instead of `row_type`.
