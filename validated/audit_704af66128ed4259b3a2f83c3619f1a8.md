Audit Report

## Title
Unauthenticated `from` PeerId in Hole-Punching Protocol Enables Attacker-Directed Outbound TCP Connection Storms — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
An attacker with a single direct P2P connection to a victim CKB node can forge the `from` field in `ConnectionRequest` and `ConnectionSync` messages to inject arbitrary TCP addresses into the victim's `pending_delivered` map and trigger passive NAT traversal tasks that make outbound TCP connections to attacker-controlled addresses. Because the attacker controls `from` and can rotate it freely, both the `forward_rate_limiter` and the `HOLE_PUNCHING_INTERVAL` deduplication guard are trivially bypassed. The only binding constraint is the per-session `rate_limiter` at 30 req/sec, allowing up to 720 outbound TCP connection attempts per second from the victim.

## Finding Description

**Root cause — no binding of `from` to the actual session sender.**

In `connection_request.rs`, `respond_delivered()` is entered when `self_peer_id == &content.to`. The `from_peer_id` argument is taken directly from the message's `from` field with no check that it equals the PeerId of the actual connected session (`self.peer`): [1](#0-0) 

The deduplication guard at the top of `respond_delivered` is keyed by `from_peer_id`: [2](#0-1) 

Because the attacker supplies a fresh random `from` PeerId each time, this guard is never hit. After filtering to TCP/IP addresses (which attacker-controlled addresses pass), the forged `from_peer_id` and attacker addresses are inserted into `pending_delivered` unconditionally: [3](#0-2) 

**Step 2 — `ConnectionSync` triggers NAT traversal using the poisoned entry.**

In `connection_sync.rs`, when `route` is empty and `self_peer_id == &content.to`, the handler looks up `pending_delivered` using `content.from` — again with no check that `content.from` matches the actual session sender: [4](#0-3) 

For each stored address it spawns a `runtime::spawn` task calling `try_nat_traversal`: [5](#0-4) [6](#0-5) 

**Rate-limit bypass.**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`: [7](#0-6) 

Its quota is 1 req/sec per key. A fresh random `from` PeerId creates a new key with a fresh bucket, so the limiter provides zero protection against an attacker rotating `from`. The only real constraint is the per-session `rate_limiter` keyed by `(session_id, msg.item_id())` at 30 req/sec: [8](#0-7) 

**Unbounded `pending_delivered` growth.**

`pending_delivered` is an unbounded `HashMap`. The `notify` cleanup only removes entries older than `TIMEOUT` (5 minutes). At 30 insertions/sec with unique `from` PeerIds, the map grows by up to 9,000 entries per 5-minute window before cleanup, consuming unbounded memory. [9](#0-8) [10](#0-9) 

## Impact Explanation

Each `ConnectionRequest`/`ConnectionSync` pair causes the victim to spawn one async task attempting outbound TCP connections to up to `ADDRS_COUNT_LIMIT` (24) attacker-controlled addresses. At the per-session cap of 30 pairs/sec, a single attacker session drives up to **720 outbound TCP connection attempts per second** to arbitrary third-party targets. This exhausts the victim node's file descriptors, OS connection-tracking state, and network bandwidth, crashing the node or rendering it unable to maintain legitimate peer connections. The victim's IP is the apparent source of the flood, enabling third-party amplification abuse.

This matches: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node** and **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attacker requires only a single standard, unprivileged P2P connection to the victim — the normal peer connection any node accepts. No key material, PoW, or special role is needed. The `HolePunching` protocol is enabled by default. The two-message sequence is trivially constructable from the public `packed::ConnectionRequest` / `packed::ConnectionSync` schema. The attack is fully repeatable and sustainable for the duration of the session.

## Recommendation

1. **Authenticate `from` in `respond_delivered`**: Assert that `from_peer_id` equals the PeerId of the actual session. In `ConnectionRequestProcess`, resolve the PeerId for `self.peer` from the peer registry and reject any message where `content.from` does not match.
2. **Authenticate `from` in `ConnectionSyncProcess`**: Before looking up `pending_delivered`, verify `content.from` against the actual session's authenticated PeerId.
3. **Cap `pending_delivered` size**: Enforce a maximum entry count (e.g., equal to `max_outbound`) to prevent memory exhaustion from forged `from` PeerIds even if authentication is added.

## Proof of Concept

```rust
// Step 1: send ConnectionRequest to victim with forged `from` and attacker TCP addr
let forged_from = PeerId::random();
let attacker_addr: Multiaddr = "/ip4/1.2.3.4/tcp/9999".parse().unwrap();
let req = packed::ConnectionRequest::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())          // victim's own PeerId
    .listen_addrs(vec![attacker_addr.to_vec()].pack())
    .max_hops(6u8.pack())
    .route(vec![].pack())               // empty route → victim is `to`
    .build();
attacker_session.send(req);

// Step 2: send ConnectionSync with same forged `from`
let sync = packed::ConnectionSync::new_builder()
    .from(forged_from.pack())
    .to(victim_peer_id.pack())
    .route(vec![].pack())               // empty route → victim is `to`
    .build();
attacker_session.send(sync);

// Victim spawns runtime::spawn task calling try_nat_traversal(bind_addr, "/ip4/1.2.3.4/tcp/9999")
// Repeat with fresh forged_from PeerIds at 30/sec → 720 TCP connection attempts/sec to 1.2.3.4:9999
```

The integration test can assert that after the two-message sequence, the victim node makes an outbound TCP connection attempt to `1.2.3.4:9999`, and that repeating with fresh `forged_from` values at 30/sec is not blocked by any existing guard.
