Audit Report

## Title
Min-Fee-Rate Check Applied to Serialized Size Instead of Actual Weight, Allowing Cycle-Heavy Transactions to Bypass the Fee Floor - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` enforces `min_fee_rate` using only the transaction's serialized byte size before cycles are known, but the actual effective weight used for block-packing and eviction is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Because `_process_tx` performs no weight-based re-check after `verify_rtx` returns the real cycle count, an unprivileged sender can submit cycle-heavy transactions whose effective fee rate is far below the configured floor while forcing full script execution.

## Finding Description
In `tx-pool/src/util.rs` lines 42–45, `check_tx_fee` computes the minimum required fee against `tx_size` only, with a comment explicitly acknowledging this is a "cheap check":

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The actual weight used everywhere else is computed by `get_transaction_weight` in `util/types/src/core/tx_pool.rs` lines 298–303:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

The admission flow in `_process_tx` (`tx-pool/src/process.rs` lines 715–751) is:
1. Line 715: `pre_check` → calls `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` — fee checked against `tx_size` only, before cycles are known.
2. Lines 724–732: `verify_rtx` — actual cycles determined (up to `max_block_cycles`).
3. Line 751: `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles; **no re-check of fee rate against actual weight**.

`TxEntry::fee_rate()` (`entry.rs` lines 114–118) and `EvictKey` (`entry.rs` lines 234–247) both use `get_transaction_weight` with real cycles, confirming the admission check is the sole gap.

With `min_fee_rate = 1000 shannons/KW`, `tx_size = 200 bytes`, `cycles = 70,000,000`:
- `weight = max(200, 70_000_000 × 0.000_170_571_4) ≈ 11,940`
- `min_fee` checked: `1000 × 200 / 1000 = 200 shannons` → **passes**
- Effective fee rate: `200 × 1000 / 11,940 ≈ 16.7 shannons/KW` — ~60× below the configured minimum

## Impact Explanation
This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** An attacker can continuously submit transactions that pass the size-based fee gate paying only the size-minimum fee (e.g., 200 shannons), force the node to execute up to `max_block_cycles` of script verification per transaction, get admitted to the pool consuming memory and relay bandwidth, and be evicted only when the pool fills up — after which the attacker repeats with fresh UTXOs. The economic gate (`min_fee_rate`) is the primary spam-prevention mechanism for tx-pool admission; bypassing it for cycle-heavy transactions breaks the intended resource-pricing guarantee and enables sustained CPU and pool exhaustion at a fraction of the intended cost.

## Likelihood Explanation
Any unprivileged transaction sender can exploit this via the `send_transaction` RPC or by relaying to a peer. Crafting a transaction with high cycle consumption but small serialized size is straightforward: a lock script running a tight computation loop (deployed as a cell dep, not inline) produces exactly this profile — the transaction body stays small while cycles consumed during verification are large. No special privileges, keys, or majority hash power are required. The attack is repeatable and cheap to automate.

## Recommendation
After `verify_rtx` returns the actual cycle count in `_process_tx` (after line 734 in `process.rs`), add a weight-based re-check before constructing `TxEntry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate,
        min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
```

Alternatively, replace the size-only pre-check in `check_tx_fee` with a weight-based check once cycles are known, or move the authoritative fee rate check into `submit_entry` after cycles are available.

## Proof of Concept
1. Deploy a lock script that consumes ~70,000,000 cycles (e.g., a tight hash loop) as a cell dep; the script cell itself is not part of the transaction's serialized size.
2. Craft a transaction referencing that lock script with serialized size ~200 bytes.
3. Set output capacity so that `inputs_capacity − outputs_capacity = 200 shannons` (fee = 200 shannons).
4. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
5. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; the check passes.
6. `verify_rtx` runs the script for ~70,000,000 cycles; `verified.cycles ≈ 70,000,000`.
7. `TxEntry::new` is called with no weight-based re-check; the transaction enters the pool.
8. Confirm via `get_transaction` RPC that the transaction is in `pending` state.
9. Observe via `get_pool_info` that pool CPU is saturated while the attacker pays only 200 shannons per submission cycle.
10. Repeat with fresh UTXOs to sustain the attack indefinitely.
