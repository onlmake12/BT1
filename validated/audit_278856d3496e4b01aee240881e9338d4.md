### Title
Excess `msg.value` Overpayment Permanently Absorbed Into Protocol Fee Pool With No Refund Mechanism - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper()` function absorbs the **entire** `msg.value` minus only the provider's fee into `_state.accruedPythFeesInWei`. Any amount a user overpays above the required fee is permanently and irrecoverably credited to the Pyth DAO fee pool, with no mechanism for the user to recover the excess. This is the direct analog to the external report's vulnerability class: a fee/refund accounting assumption that permanently denies users access to funds they are owed.

---

### Finding Description

In `requestHelper()`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The correct Pyth protocol fee is `_state.pythFeeInWei`, and `requiredFee = providerFee + pythFeeInWei`. When `msg.value == requiredFee`, the accounting is correct: `accruedPythFeesInWei += pythFeeInWei`. However, when `msg.value > requiredFee` (overpayment by `X` wei), the formula becomes `accruedPythFeesInWei += pythFeeInWei + X`, silently absorbing the excess `X` into the Pyth DAO pool. There is no refund path, no recovery function, and no event indicating the loss.

This behavior is explicitly documented in the interface:

> "Further note that excess value is *not* refunded to the caller." [2](#0-1) 

The same pattern applies to `request()`, `requestWithCallback()`, and all `requestV2()` variants, all of which route through `requestHelper()`. [3](#0-2) 

---

### Impact Explanation

Any user who sends `msg.value` exceeding `getFeeV2(provider, gasLimit)` permanently loses the excess wei. The funds are irrecoverably credited to `accruedPythFeesInWei`, which is only withdrawable by the Pyth admin/governance ŌĆö not the original payer. [4](#0-3) 

This is a direct analog to the external report: just as setting `refundAmountInUSDT[user][project] = true` on a partial withdrawal permanently blocks the user from their remaining funds, the Entropy contract's fee accounting permanently blocks the user from recovering any overpaid wei by routing it into a pool they cannot access.

---

### Likelihood Explanation

Overpayment is a realistic, common occurrence:

1. **Fee changes between query and execution**: A provider can call `setProviderFee()` at any time. If a user queries `getFeeV2()` and the provider raises their fee before the user's transaction is mined, the user's transaction would revert ŌĆö so integrators routinely add a small buffer (e.g., `fee * 110 / 100`), which becomes permanently lost. [5](#0-4) 

2. **Smart contract integrators passing `msg.value` directly**: Consumer contracts that forward `msg.value` from their own callers without exact-fee enforcement will silently lose any excess on every request.

3. **Hardcoded or stale fee values**: The interface warns against hardcoding fees, but integrators who do so will overpay whenever fees decrease, losing the difference permanently. [6](#0-5) 

---

### Recommendation

Replace the current accounting with exact-fee enforcement:

```solidity
// Only credit the configured pythFeeInWei, not the entire msg.value surplus
_state.accruedPythFeesInWei += _state.pythFeeInWei;
// Refund any excess to the caller
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

Alternatively, revert if `msg.value > requiredFee` (strict exact-fee enforcement), which is simpler and eliminates the ambiguity entirely.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1000 wei`; Pyth fee is `500 wei`; `requiredFee = 1500 wei`.
2. User calls `getFeeV2(provider, 0)` ŌåÆ receives `1500`.
3. User calls `requestV2{value: 2000}(provider, 0)` (500 wei buffer to avoid revert risk).
4. Inside `requestHelper`:
   - `providerInfo.accruedFeesInWei += 1000` Ō£ō
   - `accruedPythFeesInWei += (2000 - 1000) = 1000` ŌåÉ should be `500`, absorbs `500` extra
5. User's 500 wei overpayment is now in `accruedPythFeesInWei`.
6. Only the Pyth admin can call `withdrawFee()` to extract from that pool ŌĆö the user has no recovery path. [7](#0-6)
