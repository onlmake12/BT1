### Title
Delegators' Staked Tokens and Rewards Permanently Locked When Publisher Removed from `pool_data` with Active Delegations — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary
The Oracle Integrity Staking (OIS) integrity pool program allows publishers to be removed from `pool_data.publishers` (set to `PublicKey.default`). However, if a publisher is removed while delegators still have active staked positions targeting that publisher, those delegators permanently lose their pending rewards and cannot undelegate their tokens. The program does not check for outstanding delegations before removing a publisher, and there is no recovery path once the publisher is absent from `pool_data`.

---

### Finding Description

The `pool_data` account stores a fixed-size array of 1024 publisher pubkeys:

```
pool_data.publishers: [pubkey; 1024]
pool_data.del_state:  [DelegationState; 1024]   // per-publisher delegation totals
pool_data.self_del_state: [DelegationState; 1024]
```

Publishers are considered "removed" when their slot is set to `PublicKey.default` (the zero address). The SDK's `extractPublisherData` explicitly filters these out:

```typescript
return poolData.publishers
    .filter((publisher) => !publisher.equals(PublicKey.default))
``` [1](#0-0) 

The `advance_delegation_record` instruction requires the `publisher` account and validates it against `pool_data`. If the publisher is no longer present in `pool_data.publishers`, this instruction fails — the publisher's index cannot be resolved, so no reward calculation or `delegation_record` update can occur. [2](#0-1) 

The `undelegate` instruction enforces that the delegation record is up to date before allowing a position to be closed. This is enforced via the `OutdatedPublisherAccounting` error (code 6003):

```json
{ "code": 6003, "name": "OutdatedPublisherAccounting" }
``` [3](#0-2) 

Because `advance_delegation_record` fails when the publisher is absent from `pool_data`, the `undelegate` instruction subsequently fails with `OutdatedPublisherAccounting`. The delegator's positions cannot be closed, and their tokens remain locked in the staking custody account with no recovery path.

The `merge_delegation_positions` instruction also requires the publisher to be resolvable in `pool_data`, compounding the issue. [4](#0-3) 

---

### Impact Explanation

Delegators who have active `LOCKED` or `LOCKING` positions targeting a removed publisher:

1. **Cannot call `advance_delegation_record`** — publisher lookup in `pool_data` fails; all accrued rewards for that publisher are permanently unclaimable.
2. **Cannot call `undelegate`** — blocked by `OutdatedPublisherAccounting`; staked PYTH tokens are permanently locked in the custody account.
3. **Cannot call `merge_delegation_positions`** — same publisher resolution failure.

The `delegation_record` PDA (keyed by `[publisher, stake_account_positions]`) retains stale state with no mechanism to drain it. [5](#0-4) 

---

### Likelihood Explanation

Publishers are removed from `pool_data` via the `update_publisher_caps` instruction, which syncs the on-chain pool state with an attested `publisher_caps` account (Wormhole-signed). When a publisher is dropped from the attested caps (e.g., they stop publishing, are penalized, or are rotated out), `update_publisher_caps` can be called by anyone to apply the update. At the time of removal, delegators with active positions have no warning and no grace period to undelegate first. [6](#0-5) 

Given that publishers are regularly added and removed from the Pyth network, and that delegators may not monitor for publisher removal events, this scenario is realistically reachable.

---

### Recommendation

Before removing a publisher from `pool_data.publishers` (setting the slot to `PublicKey.default`), the program should verify that the publisher's `del_state.totalDelegation` and `self_del_state.totalDelegation` are both zero. If outstanding delegations exist, the removal should be blocked or deferred until all delegators have undelegated.

Alternatively, a grace-period mechanism should be introduced: mark the publisher as "pending removal" for at least one epoch, allowing delegators to call `advance_delegation_record` and `undelegate` before the publisher slot is cleared.

---

### Proof of Concept

1. Delegator Alice stakes 1000 PYTH to Publisher P via `delegate`. Her position enters `LOCKING` state and transitions to `LOCKED` in the next epoch.
2. Publisher P is dropped from the attested `publisher_caps` (e.g., governance decision or publisher inactivity).
3. Anyone calls `update_publisher_caps` with the new caps — Publisher P's slot in `pool_data.publishers` is set to `PublicKey.default`.
4. Alice calls `advance_delegation_record` for Publisher P → **fails**: publisher not found in `pool_data`.
5. Alice calls `undelegate` for Publisher P → **fails** with `OutdatedPublisherAccounting` (error 6003) because step 4 is a prerequisite.
6. Alice's 1000 PYTH remain locked in the staking custody account permanently, and all accrued rewards are lost. [7](#0-6) [8](#0-7)

export const extractPublisherData = (
  poolData: PoolDataAccount,
): PublisherData => {
  return poolData.publishers
    .filter((publisher) => !publisher.equals(PublicKey.default))
    .map((publisher, index) => ({
      apyHistory: poolData.events
        .filter((event) => event.epoch > 0n)
        .map((event) => ({
          apy:
            convertEpochYieldToApy(
              (event.y * (event.eventData[index]?.otherRewardRatio ?? 0n)) /
                FRACTION_PRECISION_N,
            ) *
            computeDelegatorRewardPercentage(
              poolData.delegationFees[index] ?? 0n,
            ),
          epoch: event.epoch,
          selfApy: convertEpochYieldToApy(
            (event.y * (event.eventData[index]?.selfRewardRatio ?? 0n)) /
              FRACTION_PRECISION_N,
          ),
        }))
        .sort((a, b) => Number(a.epoch) - Number(b.epoch)),
      delegationFee: poolData.delegationFees[index] ?? 0n,
      pubkey: publisher,
      selfDelegation: poolData.selfDelState[index]?.totalDelegation ?? 0n,
      selfDelegationDelta: poolData.selfDelState[index]?.deltaDelegation ?? 0n,
      stakeAccount:
        poolData.publisherStakeAccounts[index] === undefined ||
        poolData.publisherStakeAccounts[index].equals(PublicKey.default)
          ? null // eslint-disable-line unicorn/no-null
          : poolData.publisherStakeAccounts[index],
      totalDelegation:
        (poolData.delState[index]?.totalDelegation ?? 0n) +
        (poolData.selfDelState[index]?.totalDelegation ?? 0n),
      totalDelegationDelta:
        (poolData.delState[index]?.deltaDelegation ?? 0n) +
        (poolData.selfDelState[index]?.deltaDelegation ?? 0n),
    }));
