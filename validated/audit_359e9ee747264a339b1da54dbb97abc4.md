Audit Report

## Title
Unauthenticated Miner Notify Port with `work_id=0` Bypass Enables Cellbase Witness Hijacking and Block Reward Theft — (`miner/src/client.rs`)

## Summary
When notify mode is enabled, `listen_block_template_notify` accepts `BlockTemplate` POSTs from any source with no authentication. A crafted template with `work_id=0` exploits a tautology in `update_block_template`'s deduplication guard, guaranteeing the injected template is dispatched to workers. Because `CellbaseVerifier` never validates the cellbase witness lock against the operator's configured `block_assembler` lock, an attacker can embed their own lock script in the witness. When that block's reward is finalized, `RewardCalculator` reads the injected witness lock as `target_lock` and the reward is paid to the attacker's address.

## Finding Description

**1. No authentication on the notify port**

`listen_block_template_notify` binds a raw TCP listener and accepts every connection without any authentication, IP allowlist, or token check:

```rust
async fn listen_block_template_notify(&self, addr: SocketAddr) {
    let listener = TcpListener::bind(addr).await.unwrap();
    let server = auto::Builder::new(TokioExecutor::new());
    let graceful = GracefulShutdown::new();
    let stop_rx: CancellationToken = new_tokio_exit_rx();
``` [1](#0-0) 

The `handle` function deserializes any valid JSON body as a `BlockTemplate` and calls `update_block_template` unconditionally:

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();
    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }
    Ok(Response::new(Empty::new()))
}
``` [2](#0-1) 

**2. `work_id=0` is a tautological bypass**

In `update_block_template`, the deduplication closure is:

```rust
let updated = |id| {
    if id != work_id || id == 0 {
        Some(work_id)
    } else {
        None
    }
};
``` [8](#0-7) 

When the attacker sends `work_id=0`, the predicate becomes `id != 0 || id == 0`, which is always `true` regardless of the current stored `id`. `fetch_update` therefore always succeeds and the injected template is always forwarded to workers. Additionally, `current_work_id` is reset to `0`, so every subsequent legitimate template (with any non-zero `work_id`) is also accepted because `id == 0` is true — the attacker's template races with the legitimate one on every cycle. [9](#0-8) 

**3. Cellbase witness lock is never validated against operator config**

`CellbaseVerifier` only checks that the witness deserializes as a valid `CellbaseWitness` and that the lock's `hash_type` is in `ENABLED_SCRIPT_HASH_TYPE`. It performs no check that `CellbaseWitness.lock` matches the operator's configured `block_assembler` lock: [4](#0-3) 

**4. Cellbase output lock IS enforced (important nuance)**

`RewardVerifier` enforces that the cellbase output lock equals `target_lock`, which is derived from the *target block's* cellbase witness (a past, already-finalized block). An attacker who changes the output lock is rejected with `InvalidRewardTarget`: [6](#0-5) 

The attacker therefore keeps the output lock correct (easily obtained from public chain data) and only modifies the **witness** lock. The injected block passes all consensus checks.

**5. Reward flow reads the injected witness lock as `target_lock`**

`RewardCalculator::block_reward_internal` reads the cellbase witness of the stored target block and uses its `.lock()` as `target_lock` for the future reward output:

```rust
let target_lock = CellbaseWitness::from_slice(
    &self
        .store
        .get_cellbase(&target.hash())
        .expect("target cellbase exist")
        .witnesses()
        .get(0)
        .expect("target witness exist")
        .raw_data(),
)
.expect("cellbase loaded from store should has non-empty witness")
.lock();
``` [5](#0-4) 

When the injected block becomes the target block `PROPOSAL_WINDOW.farthest + 1` blocks later, the node's block assembler reads the attacker's lock from the stored witness and uses it as the cellbase output lock for the reward block. The reward is paid to the attacker's address.

## Impact Explanation

The attacker achieves complete redirection of the miner's block reward to an attacker-controlled address for every block mined on an injected template. The operator loses 100% of the block reward per affected block. If the attack is sustained or applied against multiple miners, this constitutes concrete, measurable damage to CKB mining economy. This matches the allowed impact: **Critical — Vulnerabilities which could easily damage CKB economy** (15001–25000 points).

## Likelihood Explanation

- **Precondition**: The notify port must be reachable. The default config has `listen` commented out. [7](#0-6)  However, operators running the miner on a machine separate from the CKB node (a common production setup) must bind to a non-loopback address to receive notifications from the node, making the port reachable from the network.
- **No credentials required**: Zero authentication on the port.
- **Trivial to exploit**: The attacker fetches the legitimate template from the public `get_block_template` RPC, replaces `cellbase.witnesses[0]` with a `CellbaseWitness` encoding their own lock, sets `work_id=0x0`, and POSTs it. The `work_id=0` tautology guarantees acceptance regardless of the current work state.
- **Persistent**: The attacker can continuously re-inject on every new template cycle.

## Recommendation

1. **Bind the notify port to loopback only** (`127.0.0.1`) and document this as a hard security requirement, not a suggestion.
2. **Add a shared secret / HMAC token** to the notify endpoint; the CKB node's block assembler should include it in an `Authorization` header when POSTing, and the miner should reject requests missing or failing the token check.
3. **Validate the cellbase witness lock** in `update_block_template` against the locally configured `block_assembler` lock before dispatching the template to workers.
4. **Remove or invert the `id == 0` special case** in `update_block_template`: treat `work_id=0` as invalid/rejected rather than always-accepted, or remove the special case entirely so the standard deduplication logic applies.

## Proof of Concept

```
1. Operator runs ckb-miner with listen = "0.0.0.0:8888" (miner on separate machine from node).

2. Attacker fetches the current template:
     curl http://<node>:8114/ \
       -d '{"id":1,"jsonrpc":"2.0","method":"get_block_template","params":[]}'

3. Attacker replaces cellbase.witnesses[0] with a CellbaseWitness encoding
   attacker_lock (any valid secp256k1 address they control).
   Keeps cellbase.outputs[0].lock unchanged (passes RewardVerifier).
   Sets work_id = "0x0".

4. Attacker POSTs the modified template to http://<miner>:8888/

5. update_block_template accepts it unconditionally (work_id=0 tautology in
   the fetch_update closure at miner/src/client.rs:296).

6. Miner mines the block; submit_block succeeds (all consensus checks pass —
   CellbaseVerifier only validates hash_type, not the lock identity;
   RewardVerifier validates the output lock against the past target block,
   which the attacker left correct).

7. PROPOSAL_WINDOW.farthest + 1 blocks later, RewardCalculator reads the
   injected block's cellbase witness → attacker_lock becomes target_lock →
   the block reward is paid to attacker_lock, not the operator's address.

8. Attacker repeats step 4 on every new template to sustain the theft.
```
