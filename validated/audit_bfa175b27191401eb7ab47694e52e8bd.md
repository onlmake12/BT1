### Title
Snappy Decompression Buffer Over-Allocation in `LengthDelimitedCodecWithCompress::decode` Allows 2× Memory Amplification per Peer — (`network/src/compress.rs`)

### Summary
An unprivileged remote peer on the RelayV3 protocol (protocol_id=101, `max_frame_length`=4 MB) can send a snappy-compressed frame whose wire size is just under 4 MB but whose `decompress_len()` header reports exactly `MAX_UNCOMPRESSED_LEN` = 8 MB. The decoder allocates an 8 MB `BytesMut::zeroed` buffer **before** attempting decompression. With N concurrent peers doing this simultaneously, the node allocates N × 8 MB, violating the invariant that per-peer memory is bounded by `max_frame_length`.

### Finding Description

`LengthDelimitedCodecWithCompress::decode` first delegates to the inner `LengthDelimitedCodec` which enforces the 4 MB wire-size cap. Then, for compressed frames, it reads the snappy varint header via `decompress_len()` and checks:

```rust
// compress.rs:235-242
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {   // strict >, not >=
    return Err(io::ErrorKind::InvalidData.into());
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len); // allocated BEFORE decompress
```

Two compounding issues:

1. **Off-by-one in the guard**: `> MAX_UNCOMPRESSED_LEN` permits `decompressed_bytes_len == 8388608` exactly, causing a full 8 MB allocation.
2. **Global constant decoupled from per-protocol limit**: `MAX_UNCOMPRESSED_LEN = 8 MB` is not derived from `max_frame_length`. For RelayV3 (4 MB), this is a 2× amplification. For smaller protocols (e.g., Ping at 1 KB), snappy's compression ratio physically limits the decompressed size, but the guard still permits up to 8 MB if a payload could be crafted to fit.

The allocation at line 242 is unconditional — it happens regardless of whether `SnapDecoder::decompress` subsequently succeeds or fails. [6](#0-5) 

### Impact Explanation

- Each malicious peer causes a transient 8 MB heap allocation during the decode call.
- With the default CKB peer limit (~125 peers), simultaneous crafted messages from all peers spike RSS by ~1 GB.
- On nodes with 1–2 GB RAM (common VPS deployments), this triggers OOM-killer termination, crashing the node.
- The allocation is temporary (freed after decode returns), but async I/O means many decode calls can be in-flight concurrently across sessions.

### Likelihood Explanation

- No authentication or PoW is required to open a RelayV3 session.
- Crafting the payload is trivial: snappy-compress 8 MB of zeros (compresses to ~32 KB, well under 4 MB), set the snappy varint header to exactly 8388608.
- The attack is repeatable — after the connection is dropped (decode returns `InvalidData`), the attacker reconnects and repeats.

### Recommendation

1. Change the guard from `>` to `>=`:
   ```rust
   if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN { ... }
   ```
2. Better: derive the decompressed limit from the per-protocol `max_frame_length` stored in `LengthDelimitedCodecWithCompress`, e.g., `max_frame_length * 2`, so smaller protocols are also protected proportionally.
3. Consider allocating the decompression buffer lazily or using a pre-allocated per-connection buffer with a hard cap.

                    match decompress_len(&data[1..]) {

                                debug!(

                                    Err(io::ErrorKind::InvalidData.into())
                                }
                            }
                        }

                            debug!("snappy decompress_len error: {:?}", e);
                            Err(io::ErrorKind::InvalidData.into())
                        }
                    }
                } else {

                }
            }

        }
