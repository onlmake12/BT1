### Title
Provider Accrued Fees Permanently Locked When No Fee Manager Is Set - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` accumulates provider fees in `_state.providers[providerToCredit].accruedFeesInWei` on every `executeCallback` call. The **only** withdrawal path for those fees is `withdrawAsFeeManager`, which gates on `msg.sender == _state.providers[provider].feeManager`. Because `feeManager` defaults to `address(0)` and `msg.sender` can never equal `address(0)`, any provider that has not explicitly called `setFeeManager` cannot withdraw its accrued fees. If the provider is a smart contract that lacks the ability to call `setFeeManager`, those fees are permanently locked in the contract.

### Finding Description

When a provider registers via `registerProvider`, the `ProviderInfo.feeManager` field is left at its zero-value (`address(0)`): [1](#0-0) 

Every fulfilled callback credits the provider: [2](#0-1) 

The sole withdrawal function for provider fees is: [3](#0-2) 

The check `msg.sender == _state.providers[provider].feeManager` will always fail when `feeManager == address(0)`, because no transaction can originate from the zero address. There is **no** alternative `withdraw(amount)` function that lets a provider withdraw directly, unlike the Entropy contract which exposes both `withdraw` (provider-direct) and `withdrawAsFeeManager`: [4](#0-3) 

The `ProviderInfo` struct confirms `feeManager` is a plain `address` with no default: [5](#0-4) 

### Impact Explanation

Any provider that is a smart contract without an explicit `setFeeManager` call path (e.g., a minimal keeper contract, a multisig, or any contract deployed before the operator understood the fee-manager requirement) will have its entire accrued fee balance permanently locked inside `Echo.sol`. The ETH is tracked in accounting but unreachable: `withdrawAsFeeManager` reverts unconditionally because `feeManager` is `address(0)`.

**Impact:** Permanent, irrecoverable loss of provider fee revenue. The ETH sits in the contract balance but can never be transferred out for the affected provider.

### Likelihood Explanation

- Any provider that is a smart contract and does not call `setFeeManager` before fees accrue is affected.
- The `registerProvider` function gives no indication that a separate `setFeeManager` call is required to enable withdrawals.
- The test suite only tests the fee-manager path (`testSetAndWithdrawAsFeeManager`) and never tests a provider withdrawing without a fee manager, meaning the gap is not surfaced by existing coverage. [6](#0-5) 

Likelihood is **medium**: EOA providers can self-rescue by calling `setFeeManager(self)`, but contract-based providers (e.g., automated keeper contracts) may have no such escape path.

### Recommendation

Add a direct `withdraw(uint128 amount)` function callable by the provider itself, mirroring the pattern already present in `Entropy.sol` / Fortuna:

```solidity
function withdraw(uint128 amount) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(provider.isRegistered, "Provider not registered");
    require(provider.accruedFeesInWei >= amount, "Insufficient balance");
    provider.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "Failed to send fees");
    emit FeesWithdrawn(msg.sender, amount);
}
```

This ensures providers always have a self-sovereign withdrawal path regardless of whether they have configured a fee manager.

### Proof of Concept

1. Deploy `EchoUpgradeable` with a provider that is a smart contract (e.g., a minimal keeper contract with no `setFeeManager` entrypoint).
2. The keeper contract calls `registerProvider(baseFee, feedFee, gasFee)` â€” `feeManager` is now `address(0)`.
3. Users submit requests; the keeper calls `executeCallback`, causing `_state.providers[keeper].accruedFeesInWei` to grow.
4. The keeper attempts to withdraw: there is no `withdraw()` function to call.
5. Any call to `withdrawAsFeeManager(keeper, amount)` reverts with `"Only fee manager"` because `_state.providers[keeper].feeManager == address(0)` and `msg.sender != address(0)` always.
6. The keeper contract has no `setFeeManager` function, so it cannot unblock itself.
7. All accrued fees are permanently locked. [3](#0-2) [7](#0-6)
