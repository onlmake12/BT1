### Title
Lazer Update Signatures Lack Chain ID Binding, Enabling Cross-Chain Replay — (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` signs and verifies only `keccak256(payload)`, where the payload contains only a magic number, timestamp, channel, and feed data. No chain ID or contract address is included in the signed data. Because the same trusted signers are registered across all EVM deployments of `PythLazer`, a valid signed update captured from one chain passes signature verification on every other EVM chain.

### Finding Description
In `PythLazer.sol`, `verifyUpdate()` recovers the signer from a raw `keccak256` of the payload bytes:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(hash, uint8(update[68]) + 27, ...);
``` [1](#0-0) 

The payload structure parsed by `PythLazerLib.parsePayloadHeader` is:

```
[FORMAT_MAGIC (4 bytes)] [timestamp (8 bytes)] [channel (1 byte)] [feedsLen (1 byte)] [feed data...]
``` [2](#0-1) 

There is no chain ID, no EVM chain ID (`block.chainid`), and no contract address anywhere in the signed payload. The same trusted signer set is shared across all EVM deployments of `PythLazer` (Ethereum, Arbitrum, Base, etc.), managed by Pyth governance via `updateTrustedSigner`. [3](#0-2) 

The same pattern exists in the Sui contract, where `verify_le_ecdsa_message` recovers the public key from `secp256k1_ecrecover(signature, payload, 0)` with no chain-specific context in the payload: [4](#0-3) 

### Impact Explanation
Any party who observes a valid Lazer update transaction on one EVM chain (e.g., from a public mempool or block explorer) can extract the `update` bytes and submit them to `verifyUpdate()` on any other EVM chain. The signature check passes because:
1. The signed hash is identical across chains (same payload bytes, no chain binding).
2. The same trusted signers are valid on all chains.

Concrete impacts:
- **Subscription bypass**: Pyth Lazer is a paid subscription service. An attacker can monitor on-chain Lazer updates on any chain and replay them on other chains without holding a subscription, circumventing the access-control model.
- **Cross-chain timing manipulation**: An attacker can selectively replay a slightly older (but still timestamp-valid) update to a target chain to prevent a more recent update from being the first applied, or to front-run legitimate updaters on chains where the price hasn't yet been refreshed.
- **Fee drain on target chains**: Each `verifyUpdate` call collects a `verification_fee`. Replayed calls consume this fee from the caller, but the attacker controls the timing and target chain.

### Likelihood Explanation
Lazer is deployed on multiple EVM chains simultaneously. All on-chain transactions are public. Any observer can extract the `update` bytes from a `verifyUpdate` call on chain A and resubmit them on chain B within the same timestamp window. No special access, leaked keys, or privileged role is required — only the ability to read public blockchain data and submit a transaction.

### Recommendation
Include `block.chainid` and `address(this)` (the contract address) in the data that is hashed and signed, analogous to EIP-712 domain separation:

```solidity
bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
```

This ensures a signature produced for chain A is cryptographically invalid on chain B. The same fix should be applied to the Sui and other chain implementations by including a chain-specific identifier in the signed payload.

### Proof of Concept
1. Deploy `PythLazer` on two EVM testnets (chain A and chain B) with the same trusted signer.
2. On chain A, call `verifyUpdate{value: fee}(update)` with a valid signed update — it succeeds.
3. Copy the exact same `update` bytes.
4. On chain B, call `verifyUpdate{value: fee}(update)` with the copied bytes — it also succeeds, because `keccak256(payload)` is identical and the trusted signer is the same.

The `verifyUpdate` function on chain B returns the same `payload` and `signer` as on chain A, confirming the cross-chain replay. [5](#0-4)

        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {

public(package) fun verify_le_ecdsa_message(
    state: &State,
    clock: &Clock,
    signature: &vector<u8>,
    payload: &vector<u8>,
) {
    let current_cap = state.current_cap();

    // 0 stands for keccak256 hash
    let pubkey = secp256k1_ecrecover(signature, payload, 0);

    // Check if the recovered pubkey is in the trusted signers list
    let trusted_signers = state.trusted_signers(&current_cap);
    let mut maybe_idx = trusted_signers.find_index!(|signer|
        signer.public_key() == &pubkey
    );

    assert!(maybe_idx.is_some(), ESignerNotTrusted);
    let idx = maybe_idx.extract();
    let expires_at_ms = trusted_signers[idx].expires_at_ms();
    assert!(clock.timestamp_ms() < expires_at_ms, ESignerExpired);
