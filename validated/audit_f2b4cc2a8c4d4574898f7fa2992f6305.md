Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Repeated `GetBlockProposal` P2P Messages — (File: sync/src/types/mod.rs)

## Summary
`SyncState.pending_get_block_proposals` is an unbounded `DashMap` with no total-size cap. Any connected peer can repeatedly send `GetBlockProposal` relay messages containing unique, non-existent `ProposalShortId` values, causing the map to grow without bound between periodic drain intervals. This can exhaust node heap memory and crash the CKB node.

## Finding Description
`SyncState` declares `pending_get_block_proposals` as a plain, unbounded `DashMap`: [1](#0-0) 

It is initialized with no capacity limit: [2](#0-1) 

`insert_get_block_proposals` performs no total-size check before inserting: [3](#0-2) 

This function is called from `GetBlockProposalProcess::execute` after filtering out proposals already in the tx-pool. Proposals not found in the tx-pool — guaranteed for random IDs — are inserted unconditionally: [4](#0-3) 

The only guard is a per-message count check: [5](#0-4) 

This limits entries **per message**, not the **total accumulated** across many messages. The map is only emptied by `drain_get_block_proposals`, called inside the periodic `prune_tx_proposal_request` timer: [6](#0-5) 

The drain clones and clears the map but provides no back-pressure: [7](#0-6) 

The contrast with `pending_get_headers`, which uses a bounded `LruCache`, confirms this is an oversight: [8](#0-7) 

## Impact Explanation
A single connected peer sending `GetBlockProposal` messages in a tight loop with random 10-byte `ProposalShortId` values (guaranteed absent from the tx-pool) causes `pending_get_block_proposals` to grow without bound between timer ticks. With `max_block_proposals_limit = 1500` and `max_uncles_num = 2`, each message inserts up to 3,000 new entries. Sustained flooding exhausts heap memory, triggering OOM termination or severe memory pressure that degrades all node operations. This matches the **High** impact: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation
The attack requires only a single standard P2P connection — no PoW, no valid transactions, no privileged role. The attacker generates random 10-byte IDs (2^80 unique values, effectively unlimited) and sends messages in a tight loop. The per-message cap of ~3,000 entries is not a meaningful barrier when hundreds of messages per second can be sent over a persistent TCP connection. The drain timer cannot keep pace with a sustained flood.

## Recommendation
Add a total-size cap to `pending_get_block_proposals`. In `insert_get_block_proposals`, check `self.pending_get_block_proposals.len()` against a configurable maximum (e.g., `MAX_PENDING_GET_BLOCK_PROPOSALS`) before inserting, and drop or penalize the peer if the limit is exceeded. Alternatively, replace the plain `DashMap` with a bounded LRU structure, mirroring the pattern already used for `pending_get_headers`: [8](#0-7) 

## Proof of Concept
1. Connect to a CKB node as a relay peer via standard P2P handshake.
2. In a tight loop, send `RelayMessage / GetBlockProposal` messages, each containing up to `max_block_proposals_limit × max_uncles_num` randomly generated `ProposalShortId` values (10 random bytes each), guaranteed not to exist in the target node's tx-pool.
3. `GetBlockProposalProcess::execute` calls `insert_get_block_proposals` with all IDs not found in the tx-pool — every random ID is inserted into `pending_get_block_proposals`.
4. The `drain_get_block_proposals` timer fires periodically but cannot keep pace with the flood; the `DashMap` grows without bound.
5. Monitor the node's RSS/heap; it grows linearly with message rate until OOM termination or severe performance degradation affecting all peers and RPC callers.
