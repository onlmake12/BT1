### Title
`getMinimumBalance()` Uses Static `minimumBalancePerFeed` But `updatePriceFeeds()` Charges Dynamic Gas + Pyth Fees — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`getMinimumBalance()` returns a static value derived solely from `minimumBalancePerFeed`, but `updatePriceFeeds()` deducts two dynamic fee components — the Pyth oracle fee and the gas-cost-based keeper fee — that are not reflected in `getMinimumBalance()`. A subscription funded at exactly the value returned by `getMinimumBalance()` can fail to execute `updatePriceFeeds()` during elevated gas prices or when the Pyth fee is non-trivial, causing price feeds to go stale.

---

### Finding Description

`getMinimumBalance()` is the canonical view function that subscribers, keepers, and integrators use to determine the minimum balance an active subscription must hold:

```solidity
function getMinimumBalance(
    uint8 numPriceFeeds
) external view override returns (uint256 minimumBalanceInWei) {
    return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
}
``` [1](#0-0) 

`getMinimumBalancePerFeed()` returns the static admin-configured `_state.minimumBalancePerFeed`: [2](#0-1) 

However, `updatePriceFeeds()` deducts **two separate dynamic fees** from the subscription balance:

**Step 1 — Pyth oracle fee (dynamic, depends on update data):**
```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
if (status.balanceInWei < pythFee) revert SchedulerErrors.InsufficientBalance();
status.balanceInWei -= pythFee;
``` [3](#0-2) 

**Step 2 — Keeper fee (dynamic, depends on `tx.gasprice` and actual gas consumed):**
```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
if (status.balanceInWei < totalKeeperFee) revert SchedulerErrors.InsufficientBalance();
``` [4](#0-3) 

The total balance consumed per update is therefore:

```
pythFee  +  (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice  +  singleUpdateKeeperFeeInWei * numPriceIds
```

`getMinimumBalance()` only accounts for `numPriceFeeds * minimumBalancePerFeed` — a static snapshot that does not include the Pyth fee and does not adapt to `tx.gasprice`. The `minimumBalancePerFeed` is set at initialization and updated only by admin governance: [5](#0-4) 

This same `getMinimumBalance()` value is used as the enforcement threshold in `createSubscription()`, `addFunds()`, `withdrawFunds()`, and `updateSubscription()`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A subscription funded at exactly `getMinimumBalance()` — the value the contract itself enforces as sufficient — will revert in `updatePriceFeeds()` whenever:

- `tx.gasprice` is elevated (e.g., during network congestion), causing `gasCost` to exceed the margin embedded in `minimumBalancePerFeed`; or
- The Pyth oracle fee (`pythFee`) is non-zero and not fully covered by `minimumBalancePerFeed`

The result is that price feeds for the subscription go stale even though the subscription appears "active" and "sufficiently funded" according to the contract's own view function. Downstream consumers calling `getPricesNoOlderThan()` will receive `StalePrice()` reverts: [8](#0-7) 

---

### Likelihood Explanation

EVM gas prices are highly volatile. The `minimumBalancePerFeed` is a static governance parameter that cannot self-adjust to real-time gas prices. Any period of gas price elevation above the level assumed when `minimumBalancePerFeed` was last set will cause subscriptions funded at the minimum to fail silently (the subscription remains active, the keeper call reverts, and no update occurs). This is a normal, recurring condition on all EVM chains.

---

### Recommendation

`getMinimumBalance()` should incorporate the Pyth fee estimate and a gas cost estimate at current `tx.gasprice` (or a configurable gas price ceiling), so that the value it returns is consistent with what `updatePriceFeeds()` will actually require. For example:

```solidity
function getMinimumBalance(uint8 numPriceFeeds) external view returns (uint256) {
    uint256 keeperFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceFeeds;
    uint256 estimatedGasCost = GAS_OVERHEAD * tx.gasprice; // or a configurable ceiling
    uint256 pythFeeEstimate = /* pyth.getUpdateFee for a typical update */;
    return keeperFee + estimatedGasCost + pythFeeEstimate;
}
```

Alternatively, document clearly that `getMinimumBalance()` is a floor that does not guarantee a successful update, and require subscribers to maintain a buffer above it.

---

### Proof of Concept

1. Admin deploys Scheduler with `minimumBalancePerFeed = 0.001 ether` and `singleUpdateKeeperFeeInWei = 0.0005 ether`.
2. Subscriber calls `getMinimumBalance(2)` → returns `0.002 ether`.
3. Subscriber calls `createSubscription{value: 0.002 ether}(params)` — succeeds.
4. Gas price spikes to 200 gwei. `updatePriceFeeds()` gas usage ≈ 300,000 gas → `gasCost = 300,000 * 200 gwei = 0.06 ether`.
5. Keeper calls `updatePriceFeeds(subscriptionId, updateData)`.
6. `pythFee` is deducted first (e.g., 0.0001 ether), leaving ~0.0019 ether.
7. `_processFeesAndPayKeeper` computes `totalKeeperFee = 0.06 ether + 0.001 ether = 0.061 ether`.
8. `status.balanceInWei (0.0019 ether) < totalKeeperFee (0.061 ether)` → **reverts with `InsufficientBalance`**.
9. Price feeds are not updated despite the subscription being "active" and funded at the contract-enforced minimum.
