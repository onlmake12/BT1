### Title
`SeedRecord::decode()` Does Not Verify Recovered Public Key Against Trusted Authority - (`File: network/src/services/dns_seeding/seed_record.rs`)

### Summary
`SeedRecord::decode()` performs ECDSA public-key recovery from a DNS TXT record signature but never checks whether the recovered public key belongs to a trusted/authorized signer. The companion `check()` method also omits this verification. This is the direct analog of the GearToken `signatory` not-checked vulnerability: a recovered identity is accepted without being compared to an expected authority.

### Finding Description
In `network/src/services/dns_seeding/seed_record.rs`, `SeedRecord::decode()` parses a DNS TXT seed record, recovers a `PublicKey` via `SECP256K1.recover_ecdsa()`, and returns a fully constructed `SeedRecord` — without ever asserting that the recovered key matches any trusted/authorized key: [1](#0-0) 

The `check()` method, which is the natural follow-up validation call, only verifies IP reachability, port non-zero, and timestamp freshness — it contains no pubkey authorization check at all: [2](#0-1) 

A safe wrapper `decode_with_pubkey()` does exist and does compare the recovered key: [3](#0-2) 

However, `decode()` and `check()` are both `pub`, and the test suite demonstrates the dangerous `decode()` + `check()` pattern as if it constitutes full validation: [4](#0-3) 

The production `seeding()` function currently uses `decode_with_pubkey()` correctly, but DNS seeding is gated behind an empty `TXT_VERIFY_PUBKEY` constant marked `// FIXME: should replace this later`, signaling that re-enablement is planned: [5](#0-4) 

When DNS seeding is re-enabled, any code path that calls `decode()` + `check()` — rather than `decode_with_pubkey()` — will accept seed records signed by **any** arbitrary key, not just the authorized Nervos Foundation key.

### Impact Explanation
An attacker who can inject or spoof DNS TXT records (DNS cache poisoning, BGP hijack of the authoritative DNS zone, or a compromised DNS provider) can craft syntactically valid seed records signed with their own arbitrary key. Because `decode()` accepts any recoverable signature and `check()` does not verify the signer, the attacker's records pass all validation in the `decode()` + `check()` pattern. The attacker-controlled addresses are then inserted into the node's peer store via `peer_store.add_addr()`: [6](#0-5) 

This enables **peer-store poisoning**: a bootstrapping node with no existing outbound peers will dial attacker-controlled addresses, enabling eclipse-attack primitives (transaction censorship, double-spend facilitation, block withholding).

### Likelihood Explanation
- DNS seeding is currently disabled (`TXT_VERIFY_PUBKEY = ""`), so the attack surface is not live today.
- The FIXME comment and the `with_dns_seeding` feature flag confirm re-enablement is planned.
- The dangerous `decode()` + `check()` pattern is already codified in the test suite, making it likely to be replicated in future code.
- DNS cache poisoning is a well-known, realistic attack vector requiring no privileged access.

### Recommendation
1. **Remove the `pub` visibility from `SeedRecord::decode()`** or rename it to make clear it performs no authority check (e.g., `decode_unchecked`). Force all callers through `decode_with_pubkey()`.
2. **Move the pubkey check into `check()`** by accepting an expected `PublicKey` parameter, so that `check()` alone constitutes a complete validation.
3. **Before re-enabling DNS seeding**, ensure `TXT_VERIFY_PUBKEY` is set to the real Nervos Foundation public key and that no call site uses `decode()` + `check()` without the pubkey comparison.

### Proof of Concept
An attacker generates an arbitrary secp256k1 keypair and crafts a valid DNS TXT record:

```
// Attacker-controlled key
let (attacker_sk, _attacker_pk) = Generator::random_keypair();
let malicious_ip: IpAddr = "1.2.3.4".parse().unwrap(); // attacker's node
let valid_until = now_ts() + 9999999;
let record = SeedRecord::new(malicious_ip, 8115, None, valid_until, attacker_pk);
let txt = record.encode(&attacker_sk).unwrap();

// Victim node calls decode() + check() (the dangerous pattern):
let parsed = SeedRecord::decode(&txt).unwrap();
assert!(parsed.check().is_ok()); // passes — no pubkey check performed
// parsed.address() now points to the attacker's node
```

The record passes `decode()` and `check()` despite being signed by an unauthorized key. When injected via DNS, the victim node dials the attacker's address and adds it to its peer store, completing the eclipse-attack setup.
