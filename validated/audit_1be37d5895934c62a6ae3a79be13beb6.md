### Title
Per-Transaction Integer Division in Fee Split Accumulates Precision Loss, Systematically Underpaying Proposers — (`util/reward-calculator/src/lib.rs`)

---

### Summary

`RewardCalculator::txs_fees` and `RewardCalculator::proposal_reward` both apply `safe_mul_ratio(proposer_reward_ratio)` once per transaction rather than once over the summed total. Because `safe_mul_ratio` performs integer (floor) division on each individual fee, the truncation error accumulates across every transaction in the block. The proposer consistently receives less than the intended 40 % share, while the committer receives the remainder, meaning the split is systematically wrong whenever any individual fee is not an exact multiple of the ratio denominator.

---

### Finding Description

`safe_mul_ratio` is defined as:

```rust
// util/occupied-capacity/core/src/units.rs:149-155
pub fn safe_mul_ratio(self, ratio: Ratio) -> Result<Self> {
    self.0
        .checked_mul(ratio.numer())
        .and_then(|ret| ret.checked_div(ratio.denom()))
        .map(Capacity::shannons)
        .ok_or(Error::Overflow)
}
```

It computes `⌊self × numer / denom⌋`, discarding the fractional remainder. The `proposer_reward_ratio` on mainnet is `Ratio(4, 10)`.

**`txs_fees`** (the committer's 60 % share) iterates over every stored per-transaction fee and applies the ratio individually:

```rust
// util/reward-calculator/src/lib.rs:144-155
target_ext
    .txs_fees
    .iter()
    .try_fold(Capacity::zero(), |acc, tx_fee| {
        tx_fee
            .safe_mul_ratio(consensus.proposer_reward_ratio())   // ← division per tx
            .and_then(|proposer| {
                tx_fee
                    .safe_sub(proposer)
                    .and_then(|miner| acc.safe_add(miner))
            })
    })
```

**`proposal_reward`** (the proposer's 40 % share) does the same:

```rust
// util/reward-calculator/src/lib.rs:233
reward = reward.safe_add(tx_fee.safe_mul_ratio(proposer_ratio)?)?;
// util/reward-calculator/src/lib.rs:266
reward = reward.safe_add(tx_fee.safe_mul_ratio(proposer_ratio)?)?;
```

The unit test in `util/reward-calculator/src/tests.rs` hard-codes the same per-transaction formula, confirming this is the intended (but imprecise) implementation:

```rust
// util/reward-calculator/src/tests.rs:94-97
let expected: u32 = [100u32, 20u32, 33u32, 34u32]
    .iter()
    .map(|x| x - x * 4 / 10)   // ← per-tx floor division
    .sum();
```

**Concrete precision-loss example.**  
Three transactions each paying a fee of `1` shannon:

| Approach | Proposer receives | Committer receives |
|---|---|---|
| Per-transaction (current) | `⌊1×4/10⌋ + ⌊1×4/10⌋ + ⌊1×4/10⌋ = 0` | `3` |
| Single division after sum (correct) | `⌊3×4/10⌋ = 1` | `2` |

The proposer is underpaid by 1 shannon. With N transactions the maximum accumulated error is N shannons. The total paid out is always `sum(tx_fees)`, so the committer silently absorbs the proposer's lost shannons.

The comment on line 135–136 acknowledges a related rounding concern (`tx_fee − 40 %` ≠ `60 %` of `tx_fee`) but does not address the cross-transaction accumulation problem.

---

### Impact Explanation

The proposer's reward (`proposal_reward`) is understated by up to N shannons per finalized block, where N is the number of transactions whose fees are not exact multiples of 5 (the effective granularity of `Ratio(4,10)`). The committer's reward (`txs_fees`) is correspondingly overstated by the same amount. Because these two rewards go to potentially different miners, the proposer miner is systematically underpaid relative to the protocol specification. The error is bounded but non-zero on every real block and is deterministic given the transaction set, meaning it is reproducible and verifiable on-chain.

---

### Likelihood Explanation

On mainnet, virtually every block contains transactions whose fees are not exact multiples of 5 shannons. The `proposer_reward_ratio` is a fixed consensus parameter (`Ratio(4, 10)`). The code path is exercised for every block finalization via `block_reward_to_finalize` → `block_reward_internal` → `txs_fees` / `proposal_reward`. No special conditions are required; any unprivileged transaction sender who submits transactions with fees of 1–4 shannons (or any non-multiple-of-5 amount) triggers the imprecision.

---

### Recommendation

Collect the raw fee totals first, then apply the ratio a single time:

**`txs_fees`:**
```rust
let total_fees: Capacity = target_ext
    .txs_fees
    .iter()
    .try_fold(Capacity::zero(), |acc, fee| acc.safe_add(*fee))?;
let proposer_share = total_fees.safe_mul_ratio(consensus.proposer_reward_ratio())?;
total_fees.safe_sub(proposer_share)
```

**`proposal_reward`:** accumulate raw `tx_fee` values into a running sum, then call `safe_mul_ratio` once at the end before returning.

This mirrors the fix recommended in the reference report: defer the single division until after all addends have been summed.

### Proof of Concept

Using the existing test harness in `util/reward-calculator/src/tests.rs`, replace the fee vector with `[1u32, 1u32, 1u32]`:

```
Per-tx result  : (1 - ⌊1×4/10⌋) + (1 - ⌊1×4/10⌋) + (1 - ⌊1×4/10⌋)
               = (1-0) + (1-0) + (1-0) = 3  (committer gets 3, proposer gets 0)

Correct result : 3 - ⌊3×4/10⌋ = 3 - 1 = 2  (committer gets 2, proposer gets 1)
```

The proposer is underpaid by 1 shannon. Scaling to a full block of 1 000 such transactions yields a 1 000-shannon (10 µCKB) systematic underpayment to the proposer per block, with the committer receiving the surplus.
