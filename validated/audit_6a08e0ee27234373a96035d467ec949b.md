### Title
Griefing DoS on `post_vaa` via Pre-funding the `posted_vaa` PDA — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_vaa.rs`)

---

### Summary

An unprivileged attacker can permanently block the posting of any specific VAA by transferring lamports to the deterministically-derivable `posted_vaa` PDA address before `post_vaa` is called. Anchor's `init` constraint will then fail unconditionally because the account already has lamports, and there is no fallback path in the legacy `post_vaa` instruction.

---

### Finding Description

The `posted_vaa` account in `PostVaa` is declared with a strict `init` constraint: [1](#0-0) 

The PDA is seeded with `[PostedVaaV1::SEED_PREFIX, signature_set.message_hash.as_ref()]`. The `message_hash` is a public field on the `SignatureSet` account: [2](#0-1) 

Because `SignatureSet` is a public on-chain account, any observer can read `message_hash` and deterministically compute the `posted_vaa` PDA address. On Solana, the System Program's `transfer` instruction allows sending lamports to any address without the recipient signing. Transferring even 1 lamport to the PDA address creates a system-owned account at that address. Anchor's `init` constraint internally calls `system_program::create_account`, which fails with `AccountAlreadyInUse` if the target account already has lamports > 0.

The code comment at lines 53–55 explicitly acknowledges that the **previous** implementation handled this case gracefully (bailing out with success), but the current implementation removed that protection: [3](#0-2) 

There is no recovery mechanism: the core-bridge program has no instruction to drain lamports from a pre-funded PDA it does not yet own, so the DoS is permanent for that specific VAA.

The alternative `post_vaa_v1` path uses a different seed (`keccak::hash(body.as_ref())` vs. `message_hash`) and requires an `EncodedVaa` account, not a `SignatureSet`, so it does not serve as a fallback for the legacy flow: [4](#0-3) 

---

### Impact Explanation

Permanent, per-VAA DoS of the legacy `post_vaa` instruction. Any downstream Pyth price update consumer that depends on a `PostedVaaV1` account for a specific VAA will be permanently blocked from receiving that update. The attacker spends only ~1 lamport + transaction fee per VAA griefed.

---

### Likelihood Explanation

The `SignatureSet` account and its `message_hash` are publicly readable on-chain. The PDA derivation is deterministic and requires no privileged access. The attacker only needs to observe a new `SignatureSet` account (or monitor the mempool for `verify_signatures` transactions) and front-run the subsequent `post_vaa` call with a lamport transfer. This is straightforward on Solana.

---

### Recommendation

Replace the strict `init` constraint with `init_if_needed` and add an explicit check that the account is uninitialized (e.g., discriminator is zeroed) before writing. Alternatively, restore the previous behavior of bailing out with success if the account already exists and its content matches the expected VAA data. A third option is to use `realloc` / `realloc_if_needed` with an ownership check so that a pre-funded system account can be claimed and resized by the program.

---
