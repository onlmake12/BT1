### Title
Fee Discontinuity in `getProviderFee()` Allows Users to Underpay for Callback Gas When `defaultGasLimit` Is Not a Multiple of 10,000 — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`getProviderFee()` in `Entropy.sol` contains a threshold-boundary fee inconsistency. When a user passes `callbackGasLimit = 0`, the rounded gas limit evaluates to `0`, which is never greater than `defaultGasLimit`, so the fee collapses to the flat base `feeInWei`. However, the request is stored with `req.gasLimit10k = roundTo10kGas(defaultGasLimit)`, meaning the callback actually executes with `ceil(defaultGasLimit / 10000) * 10000` gas — which exceeds `defaultGasLimit` whenever `defaultGasLimit` is not an exact multiple of 10,000. A user who requests `callbackGasLimit = defaultGasLimit` pays the proportionally scaled fee for the rounded-up gas, while a user who requests `callbackGasLimit = 0` pays only `feeInWei` for the identical callback gas allocation. This is a direct analog to the reported abrupt fee drop at a threshold boundary.

---

### Finding Description

**Root cause — `getProviderFee()` lines 780–793:**

```solidity
uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
if (
    provider.defaultGasLimit > 0 &&
    roundedGasLimit > provider.defaultGasLimit
) {
    uint128 additionalFee = ((roundedGasLimit -
        provider.defaultGasLimit) * provider.feeInWei) /
        provider.defaultGasLimit;
    return provider.feeInWei + additionalFee;
} else {
    return provider.feeInWei;   // ← reached when gasLimit == 0
}
``` [1](#0-0) 

When `gasLimit = 0`: `roundTo10kGas(0) = 0`, so `roundedGasLimit = 0`. The condition `0 > defaultGasLimit` is always false, so the function returns `feeInWei`.

**Stored gas limit — `requestHelper()` lines 278–282:**

```solidity
req.gasLimit10k = roundTo10kGas(
    callbackGasLimit < providerInfo.defaultGasLimit
        ? providerInfo.defaultGasLimit
        : callbackGasLimit
);
``` [2](#0-1) 

When `callbackGasLimit = 0 < defaultGasLimit`, the stored gas limit becomes `roundTo10kGas(defaultGasLimit)`. If `defaultGasLimit = 15000`, this is `2`, meaning the callback receives **20,000 gas**.

**Concrete fee comparison** (provider: `defaultGasLimit = 15000`, `feeInWei = F`):

| `callbackGasLimit` | Fee charged | Callback gas received |
|---|---|---|
| `0` | `F` | 20,000 |
| `15000` | `F × 20000/15000 ≈ 1.33F` | 20,000 |

Both users receive identical callback gas, but the user requesting `0` pays 25% less.

**Fee validation — `requestHelper()` lines 234–239:**

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [3](#0-2) 

The fee check uses `callbackGasLimit` directly, not the effective gas limit that will be stored in the request. When `callbackGasLimit = 0`, `requiredFee = feeInWei + pythFee`, which is less than what a user requesting `defaultGasLimit` gas would pay, despite both receiving the same callback gas.

**`roundTo10kGas()` — lines 962–973:** [4](#0-3) 

This function rounds up to the nearest 10,000. For `gas = 0` it returns `0`; for `gas = 15000` it returns `2` (i.e., 20,000). The asymmetry between the fee path (uses raw `gasLimit`) and the storage path (uses `max(gasLimit, defaultGasLimit)`) is the root cause.

---

### Impact Explanation

Any user can call `requestV2(provider, userRandomNumber, 0)` and pay only `feeInWei + pythFee` while receiving a callback with `roundTo10kGas(defaultGasLimit) * 10000` gas — the same gas as a user who pays the full proportional fee. The provider accrues less revenue per request than their fee structure intends whenever `defaultGasLimit` is not a multiple of 10,000. For `defaultGasLimit = 50000`, the underpayment is 20%; for `defaultGasLimit = 15000`, it is 33%. Excess `msg.value` is explicitly not refunded to callers, so users who naively pass `callbackGasLimit = defaultGasLimit` are overcharged relative to users who pass `0`.

---

### Likelihood Explanation

The condition is triggered whenever a provider sets a `defaultGasLimit` that is not an exact multiple of 10,000. `setDefaultGasLimit` only validates that the value does not exceed `MAX_GAS_LIMIT`; it does not enforce alignment to 10,000-gas boundaries. [5](#0-4) 

Any unprivileged Entropy user can exploit this by simply passing `gasLimit = 0` to `requestV2`. No special knowledge, privileged role, or key is required. The behavior is consistent and deterministic.

---

### Recommendation

In `getProviderFee()`, compute the fee against the **effective** gas limit (i.e., `max(roundedGasLimit, roundTo10kGas(defaultGasLimit) * TEN_THOUSAND)`) rather than the raw `roundedGasLimit`. This ensures that a user requesting 0 gas pays the same fee as a user requesting `defaultGasLimit` gas, since both receive the same callback gas allocation:

```solidity
uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
uint32 effectiveGasLimit = provider.defaultGasLimit > 0
    ? (roundedGasLimit > provider.defaultGasLimit ? roundedGasLimit : uint32(roundTo10kGas(provider.defaultGasLimit)) * TEN_THOUSAND)
    : roundedGasLimit;
// use effectiveGasLimit for fee calculation
```

Alternatively, enforce that `defaultGasLimit` must be a multiple of 10,000 in `setDefaultGasLimit`.

---

### Proof of Concept

1. Provider registers with `defaultGasLimit = 15000` and `feeInWei = 1e15 wei` (0.001 ETH).
2. **User A** calls `requestV2(provider, randomNum, 0)` — pays `1e15 + pythFee` wei, callback receives 20,000 gas.
3. **User B** calls `requestV2(provider, randomNum, 15000)` — pays `1e15 * 20000/15000 + pythFee ≈ 1.33e15 + pythFee` wei, callback also receives 20,000 gas.
4. User A and User B receive identical service. User A paid ~25% less.
5. Repeat step 2 at scale: provider accrues `1e15` per request instead of `1.33e15`, losing `3.3e14 wei` (~$0.001 at current ETH prices) per request. At 10,000 requests/day this is ~$10/day in lost provider revenue.

    function roundTo10kGas(uint32 gas) internal pure returns (uint16) {
        if (gas > MAX_GAS_LIMIT) {
            revert EntropyErrors.MaxGasLimitExceeded();
        }

        uint32 gas10k = gas / TEN_THOUSAND;
        if (gas10k * TEN_THOUSAND < gas) {
            gas10k += 1;
        }
        // Note: safe cast here should never revert due to the if statement above.
        return SafeCast.toUint16(gas10k);
    }
