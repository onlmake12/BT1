### Title
Unbounded RocksDB I/O via unauthenticated repeated `GetBlockFilters` requests — (`sync/src/filter/get_block_filters_process.rs`)

### Summary
The `BlockFilter` P2P protocol handler has no per-peer rate limiting. Any unauthenticated peer can send `GetBlockFilters(start_number=0)` in a tight loop, causing the node to perform up to 2,000 RocksDB reads and serialize up to 1.8 MB of data per message, with no throttle or ban applied.

### Finding Description
`GetBlockFiltersProcess::execute()` reads up to `BATCH_SIZE=1000` block hashes and filter entries from RocksDB on every invocation. The `BlockFilter` handler in `sync/src/filter/mod.rs` dispatches directly to this function with no rate check. The `Relayer` and `HolePunching` protocols both use a `governor`-based `RateLimiter<(PeerIndex, u32)>` keyed by peer and message type, but `BlockFilter` has no such field or check. The 1.8 MB frame-size cap added in v0.203.0 prevents oversized responses but does nothing to limit request frequency.

### Impact Explanation
A single attacker peer can saturate the victim node's RocksDB I/O and CPU by flooding `GetBlockFilters(start_number=0)` messages. Each message is cheap to send (~10 bytes) but expensive to serve (~2,000 RocksDB point-reads + up to 1.8 MB serialization). This degrades block propagation and sync for legitimate peers, causing CKB network congestion.

### Likelihood Explanation
The attack requires only a standard P2P connection — no keys, no stake, no PoW. The `Filter` protocol is enabled by default on full nodes. The attacker needs to send one valid 10-byte message repeatedly. The node will never ban the peer for this behavior because `Status::ok()` is returned on every successful response.

### Recommendation
Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` to the `BlockFilter` struct (mirroring the pattern in `Relayer::new()`) and check it at the top of `try_process()` before dispatching to `GetBlockFiltersProcess`. A limit of ~10–30 requests/second per peer is consistent with the existing protocol guards.

### Proof of Concept
1. Connect to a CKB full node that has built filters (any synced mainnet node).
2. In a loop, send `BlockFilterMessage { GetBlockFilters { start_number: 0 } }` as fast as the TCP connection allows.
3. Observe node CPU and RocksDB read IOPS spike proportionally to message rate, with no disconnection or ban triggered.
