### Title
Unbounded `pending_get_block_proposals` DashMap Growth via Unauthenticated `GetBlockProposal` Messages — (`sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

### Summary

`insert_get_block_proposals` has no size cap. Any peer can send `GetBlockProposal` messages with proposal IDs absent from the tx-pool, causing unbounded accumulation in `pending_get_block_proposals` until the timer fires.

### Finding Description

`GetBlockProposalProcess::execute` validates only that the per-message count does not exceed `max_block_proposals_limit * max_uncles_num`: [1](#0-0) 

IDs not found in the tx-pool are collected and unconditionally inserted: [2](#0-1) 

`insert_get_block_proposals` performs no per-peer or global size check: [3](#0-2) 

The map is only cleared when `drain_get_block_proposals` is called by the timer: [4](#0-3) 

The drain happens inside `prune_tx_proposal_request`, which is timer-driven: [5](#0-4) 

**Critically**, a search for any rate-limiter (`RateLimiter`, `rate_limit`, `MAX_RELAY_PEERS`) returned **no matches** in the codebase. The question's assumed "30 rps" guard does not appear to exist, making the attack surface larger than described.

The DashMap key is `ProposalShortId` (10 bytes). Multiple peers requesting the **same** ID merge into one entry (the value is `HashSet<PeerIndex>`), providing natural deduplication. However, an attacker trivially generates distinct IDs, so deduplication provides no real bound. Between two timer ticks, `N_peers × msgs_per_second × IDs_per_msg` unique entries accumulate with no eviction.

### Impact Explanation

Memory exhaustion → OOM kill → node goes offline → consensus participation lost. The node cannot validate blocks or relay transactions until restarted, causing it to fall behind the chain tip.

### Likelihood Explanation

- Entrypoint is a standard P2P message requiring no authentication or PoW.
- No rate limiter was found in the codebase.
- No per-peer or global cap exists in `insert_get_block_proposals`.
- The only relief valve is a periodic timer whose interval determines the worst-case map size.

### Recommendation

1. Add a global cap to `insert_get_block_proposals`: reject or drop insertions once `pending_get_block_proposals.len()` exceeds a constant (e.g., `max_block_proposals_limit * MAX_PEERS`).
2. Add a per-peer cap: track how many pending entries each peer has contributed and evict or reject when exceeded.
3. Implement a per-peer, per-message-type rate limiter for `GetBlockProposal` messages at the protocol handler level.

### Proof of Concept

    loop:
        ids = [random_unique_10_byte_id() for _ in range(max_block_proposals_limit * max_uncles_num)]
