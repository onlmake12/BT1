### Title
Accumulated Update Fees Permanently Locked in `PriceInfoObject` — No Withdrawal Path Exists - (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `update_single_price_feed` deposits `base_update_fee` SUI into a `PriceInfoObject`'s dynamic object field under `FEE_STORAGE_KEY`. No function anywhere in the production contract set removes or transfers those coins. All protocol fee revenue is permanently frozen.

### Finding Description

`update_single_price_feed` in `pyth.move` enforces a fee and immediately deposits it into the `PriceInfoObject`: [1](#0-0) 

`deposit_fee_coins` in `price_info.move` either creates the dynamic object field or joins coins into it — there is no corresponding remove path: [2](#0-1) 

`get_balance` is read-only: [3](#0-2) 

A grep across the entire `sources/` directory for `dynamic_object_field::remove`, `withdraw`, `sweep`, `collect.*fee`, or `transfer.*fee` returns **zero matches** in production code. The only `coin::split` calls are inside `#[test_only]` blocks.

The `fee_recipient_address` field exists in `State`: [4](#0-3) 

but it is never used to collect coins from any `PriceInfoObject`. The governance module `set_fee_recipient.move` even carries an explicit admission: [5](#0-4) 

> "The previous version of the contract sent the fees to a recipient address but **this state is not used anymore**."

The fee-collection logic was removed and never replaced with the current `deposit_fee_coins` design.

### Impact Explanation
Every `update_single_price_feed` call across every price feed permanently locks `base_update_fee` SUI inside the corresponding `PriceInfoObject`. Because `PriceInfoObject` is a shared object with no privileged owner and no withdrawal entry point, the coins are irrecoverable by any party — including the `fee_recipient_address` stored in `State`. The total locked amount grows monotonically with protocol usage. All protocol fee revenue is destroyed.

### Likelihood Explanation
This triggers on every normal price-feed update. No special attacker is required; the loss occurs automatically for every legitimate user who calls `update_single_price_feed`. It is already happening on any live deployment.

### Recommendation
Add a `public(friend)` (or governance-gated) function in `price_info` that calls `dynamic_object_field::remove<vector<u8>, Coin<SUI>>` on `FEE_STORAGE_KEY` and returns the coin, then add a sweep entry point in `pyth.move` that transfers the extracted coin to `state::get_fee_recipient(pyth_state)`. Alternatively, transfer the fee coin directly to the recipient inside `update_single_price_feed` instead of depositing it into the `PriceInfoObject`.

### Proof of Concept
1. Deploy the contract with `base_update_fee = 50`.
2. Call `update_single_price_feed` N times, each time passing a `Coin<SUI>` of value 50.
3. After each call, assert `price_info::get_balance(price_info_object) == 50 * call_count` — balance grows monotonically.
4. Attempt to find any callable function that reduces the balance — none exists.
5. Confirm `fee_recipient_address` never receives any SUI.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
