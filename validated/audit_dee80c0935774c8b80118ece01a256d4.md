### Title
Consensus Split in DAO Withdrawal `header_deps` Index Resolution: Rust Node Reads Full u64, C VM Reads Lowest Byte — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw()` in `util/dao/src/lib.rs` resolves the deposit-header index from the witness as a full 8-byte little-endian `u64`. The on-chain C DAO type script running inside CKB-VM resolves the same field using only the **lowest byte** (effectively a `u8`). For any DAO withdrawal transaction whose witness `input_type` index exceeds 255, the two evaluators silently resolve **different** entries in `header_deps` as the deposit header, producing a consensus split. An unprivileged transaction sender can exploit this split in either direction: to make the Rust node reject a C-VM-valid transaction (liveness), or to make the Rust node accept a C-VM-invalid transaction (invalid block production / miner griefing).

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` parses the deposit-header index from the witness `input_type` field:

```rust
// dao contract stores header deps index as u64 in the input_type field of WitnessArgs
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
``` [1](#0-0) 

It then uses that full `u64` value to index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
``` [2](#0-1) 

The C DAO type script (referenced at `https://github.com/nervosnetwork/ckb-system-scripts/blob/1fd4cd3e2ab7e5ffbafce1f60119b95937b3c6eb/c/dao.c`) reads the same 8-byte field but interprets only the **lowest byte** as the index. This is explicitly documented in the test suite:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
// input_type = 257, lowest byte = 1
``` [3](#0-2) 

The test `check_dao_withdraw_header_dep_index_exceeds_u8` constructs a transaction with 258 `header_deps` entries, places the deposit block at index 1 and the withdraw block at index 257, and sets the witness index to `257`. The Rust `DaoCalculator` resolves index 257 → withdraw block (number 200), which does not match the cell data block number (100), so it returns `Err`. The C VM resolves the lowest byte `1` → deposit block (number 100), which matches, so it would return success. [4](#0-3) 

`DaoCalculator` is called in both the transaction-verification path and the tx-pool admission path: [5](#0-4) [6](#0-5)

### Impact Explanation

**Direction A — C VM accepts, Rust rejects (liveness):**
A legitimate user crafts a DAO withdrawal with `header_deps_index = 257`, placing the correct deposit block at position 1 (lowest byte). The C VM accepts the transaction. The Rust node's `DaoCalculator` resolves index 257 to a different block, fails the block-number cross-check, and rejects the transaction from the tx-pool. The user's valid withdrawal is permanently stuck.

**Direction B — Rust accepts, C VM rejects (invalid block / miner griefing):**
An attacker crafts a DAO withdrawal with `header_deps_index = 257`, placing the correct deposit block at position **257** (full u64) and a wrong block at position **1** (lowest byte). The Rust node's `DaoCalculator` resolves index 257 → correct deposit block → block-number check passes → transaction admitted to tx-pool. The C VM resolves lowest byte 1 → wrong block → block-number check fails → script execution aborts. Any miner who includes this transaction produces an invalid block that every other node rejects, wasting the miner's PoW work and block reward.

---

### Likelihood Explanation

The attack requires:
1. Controlling a live DAO withdrawal cell (phase-1 prepare cell) — achievable by any CKB holder.
2. Constructing a transaction with ≥ 258 `header_deps` entries — the protocol imposes no hard cap below this range.
3. Submitting via the standard `send_transaction` JSON-RPC endpoint — no privileged access required.

The discrepancy is already documented and tested in the production codebase, confirming the developers are aware of the behavioral difference. The attack is deterministic and reproducible.

---

### Recommendation

Align the Rust index-resolution logic with the C DAO type script. The simplest fix is to truncate the parsed `u64` to `u8` before using it as the `header_deps` index, matching the C VM's behavior:

```rust
// Match C VM: use only the lowest byte of the 8-byte witness index
let header_dep_index = LittleEndian::read_u64(&header_deps_index_data.unwrap()) as u8;
```

Alternatively, if the intent is to support indices > 255 in the future, the C DAO type script must be updated to read the full `u64` and a consensus-breaking upgrade must be coordinated. Until alignment is confirmed, the node should reject any DAO withdrawal whose witness index exceeds 255 with an explicit `InvalidDaoFormat` error rather than silently resolving a different header.

---

### Proof of Concept

The existing test in `util/dao/src/tests.rs` already demonstrates Direction A (C VM accepts, Rust rejects):

```rust
// input_type = 257, lowest byte = 1
let witness = WitnessArgs::new_builder()
    .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
    .build();
// ...
// Rust resolves index 257 → withdraw block (number 200), but cell data
// says deposited at block 100. Block number check catches the mismatch.
assert!(result.is_err(), "expected Err, got {result:?}");
``` [7](#0-6) 

Direction B (Rust accepts, C VM rejects — miner griefing) is constructed by swapping the positions:

```
header_deps[1]   = dummy_block   (number ≠ 100)  ← C VM resolves here, check fails
header_deps[257] = deposit_block (number = 100)   ← Rust resolves here, check passes
cell_data        = 100
witness index    = 257
```

The Rust `DaoCalculator` returns `Ok`, the transaction enters the tx-pool, a miner includes it, and the resulting block is rejected by every peer whose C VM correctly executes the DAO type script.

fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
