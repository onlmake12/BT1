### Title
Funds Permanently Locked in Permanent Scheduler Subscriptions With No Recovery Path — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `Scheduler` contract's `withdrawFunds()` function unconditionally reverts for any subscription where `isPermanent == true`, and no admin escape hatch exists. Critically, `addFunds()` is permissionless — any address can deposit ETH into a permanent subscription. Once deposited, those funds are irrecoverably locked: the depositor cannot withdraw them, the subscription manager cannot withdraw them, and no admin function exists to rescue them. The only way funds ever leave is through keeper fee deductions over time.

### Finding Description

`withdrawFunds()` hard-blocks all withdrawals from permanent subscriptions: [1](#0-0) 

```solidity
function withdrawFunds(uint256 subscriptionId, uint256 amount)
    external override onlyManager(subscriptionId)
{
    ...
    // Prevent withdrawals from permanent subscriptions
    if (params.isPermanent) {
        revert SchedulerErrors.CannotUpdatePermanentSubscription();
    }
    ...
}
```

`updateSubscription()` also unconditionally reverts for permanent subscriptions — including attempts to deactivate them — so the manager cannot flip `isActive = false` to unlock the withdrawal path: [2](#0-1) 

Meanwhile, `addFunds()` is fully permissionless and accepts ETH from **any caller**, not just the subscription manager: [3](#0-2) 

The test suite explicitly confirms this: `"Anyone can add funds (not just manager)"`: [4](#0-3) 

There is no `emergencyWithdraw`, no admin rescue function, and no governance path to recover funds from a permanent subscription. The `MAX_DEPOSIT_LIMIT` of 100 ETH per `addFunds()` call caps individual deposits but does not prevent repeated calls: [5](#0-4) 

### Impact Explanation

Any ETH sent to a permanent subscription via `addFunds()` is permanently locked in the contract. The depositor — whether a third party, an integrating protocol, or the subscription manager themselves — has zero recourse. The funds can only be consumed by keeper fee deductions over time, which may be negligible relative to the locked amount. This is a direct loss of user funds with no recovery path, matching the severity of the USDM analog.

### Likelihood Explanation

The entry path is fully unprivileged: any EOA or contract can call `addFunds(subscriptionId)` with ETH on any active permanent subscription. Permanent subscriptions are publicly discoverable via `getActiveSubscriptions()`. A user who mistakenly sends ETH to a permanent subscription (e.g., via a UI that does not clearly distinguish permanent from non-permanent subscriptions, or a protocol that programmatically tops up subscriptions) permanently loses those funds. The `isPermanent` flag is also irreversible once set by the manager, so the attack surface grows monotonically over time.

### Recommendation

1. Restrict `addFunds()` to `onlyManager(subscriptionId)` for permanent subscriptions, or at minimum emit a clear warning and require explicit acknowledgment.
2. Add an admin-controlled emergency withdrawal function that can recover funds from permanent subscriptions to the subscription manager's address.
3. Alternatively, allow permanent subscriptions to be deactivated (but not otherwise modified), so the standard `withdrawFunds()` path becomes available after deactivation.

### Proof of Concept

1. Alice creates a subscription and sets `isPermanent = true` via `updateSubscription()`.
2. Bob (any address) calls `addFunds{value: 50 ether}(subscriptionId)`.
3. Bob's 50 ETH is credited to `status.balanceInWei`.
4. Bob calls `withdrawFunds(subscriptionId, 50 ether)` — reverts with `CannotUpdatePermanentSubscription`.
5. Alice (the manager) calls `withdrawFunds(subscriptionId, 50 ether)` — same revert.
6. Alice calls `updateSubscription(subscriptionId, params_with_isActive_false)` — same revert.
7. No admin function exists to recover the 50 ETH. Funds are permanently locked. [6](#0-5) [2](#0-1)

```text
    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```

```text
    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
```
