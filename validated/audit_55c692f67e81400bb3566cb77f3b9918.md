Audit Report

## Title
`tx_size` Used Instead of `weight` in Fee-Rate Admission and RBF Checks, Allowing Cycle-Heavy Transactions to Bypass Minimum Fee Rate — (`tx-pool/src/util.rs`, `tx-pool/src/pool.rs`)

## Summary
`check_tx_fee()` passes raw serialized byte size to `FeeRate::fee()`, which is documented as operating on weight (`max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`). The code comment explicitly acknowledges this as a "cheap check," but no weight-based fee-rate re-check is performed after `verify_rtx` returns the actual cycle count. `calculate_min_replace_fee()` has the same flaw for RBF. An attacker can submit cycle-heavy transactions at up to ~60× below the configured minimum fee rate, forcing full CKB-VM script execution at negligible cost.

## Finding Description

**Root cause — `FeeRate` unit mismatch:**

`FeeRate::fee()` is documented and named as taking a `weight` parameter (shannons per kilo-weight): [1](#0-0) 

Weight is defined as `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [2](#0-1) 

**Location 1 — `check_tx_fee` (`tx-pool/src/util.rs:45`):**

`tx_size as u64` is passed directly to `FeeRate::fee()`. The comment explicitly acknowledges the mismatch but labels it a "cheap check": [3](#0-2) 

In `_process_tx`, `check_tx_fee` is called during `pre_check` at lines 289/294, then `verify_rtx` runs and returns actual cycles, and then `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is created at line 751 — **no weight-based fee-rate re-check is performed after cycles are known**: [4](#0-3) 

**Location 2 — `calculate_min_replace_fee` (`tx-pool/src/pool.rs:103`):**

`size as u64` (raw bytes) is used where `get_transaction_weight(size, cycles)` is required. The `TxEntry` already carries `cycles`, so the correct weight is available: [5](#0-4) 

This is called from `check_rbf` to enforce RBF Rules #3/#4: [6](#0-5) 

**Why existing checks fail:**

The eviction key does use actual weight (via `TxEntry::fee_rate()` which calls `get_transaction_weight`), so these transactions are evicted first when the pool is full — but the attacker can continuously resubmit them, each time forcing a full CKB-VM script execution at minimal fee cost. The admission gate (the only place that could reject the transaction) uses only byte size.

## Impact Explanation

**High — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

With default config (`min_fee_rate = 1000`, `max_tx_verify_cycles = 70,000,000`, `DEFAULT_BYTES_PER_CYCLES ≈ 0.000_170_571_4`):

- A 200-byte transaction consuming 70,000,000 cycles has weight = max(200, 11,940) = **11,940**
- Correct minimum fee: `1000 × 11,940 / 1000 = 11,940 shannons`
- Actual admission check: `1000 × 200 / 1000 = 200 shannons`
- **Attacker pays ~200 shannons instead of ~11,940 — a ~60× reduction**

The attacker continuously floods the tx-pool with cycle-heavy transactions at 1/60th the intended cost, each forcing a full CKB-VM execution. Since these transactions are evicted first (correct weight-based eviction), the attacker simply resubmits, creating a sustained DoS against the node's script verification pipeline at negligible fee cost.

## Likelihood Explanation

Reachable by any unprivileged user via the `send_transaction` JSON-RPC endpoint. No special role, key, or configuration is required. The attacker only needs to craft a transaction with a lock script that loops for ~70,000,000 cycles while keeping the serialized transaction size small (~200 bytes). This is straightforward to construct using CKB-VM. The attack is repeatable indefinitely since evicted transactions can be immediately resubmitted.

## Recommendation

**For `check_tx_fee`:** Since cycles are not yet known at pre-check time, use `max_tx_verify_cycles` as a conservative upper bound: `get_transaction_weight(tx_size, tx_pool.config.max_tx_verify_cycles)`. Alternatively, restructure the flow to re-check the fee rate after `verify_rtx` returns the actual cycle count.

**For `calculate_min_replace_fee`:** The replacement entry's actual cycles are already known. Replace `size as u64` with `get_transaction_weight(size, entry_cycles)` where `entry_cycles` is passed in or derived from the `TxEntry`.

## Proof of Concept

1. Craft a CKB transaction with a lock script that loops for ~70,000,000 cycles and a serialized size of ~200 bytes.
2. Set the transaction fee to 200 shannons (satisfying `min_fee_rate.fee(200) = 200`).
3. Submit via `send_transaction` RPC.
4. The transaction passes `check_tx_fee` (200 ≥ 200), proceeds to `verify_rtx`, consumes ~70M cycles, and is admitted to the pool.
5. The actual fee rate is `FeeRate::calculate(200, 11940) ≈ 16 shannons/kW`, far below the configured minimum of 1000.
6. The transaction is evicted when the pool fills (correct weight-based eviction). Resubmit immediately.
7. Repeat to sustain a continuous stream of full CKB-VM executions at ~200 shannons each instead of the intended ~11,940 shannons each.

For the RBF variant: submit a legitimate transaction, then submit a cycle-heavy replacement with fee = `sum(replaced_fees) + min_rbf_rate.fee(size)` instead of the correct `sum(replaced_fees) + min_rbf_rate.fee(weight)`, bypassing Rules #3/#4 of the RBF check.
