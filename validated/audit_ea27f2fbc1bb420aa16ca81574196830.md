### Title
`total_tx_size` / `total_tx_cycles` Not Decremented for Evicted Entries in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the `total_tx_size` and `total_tx_cycles` accounting variables are computed as a pre-eviction snapshot and then unconditionally written back **after** `check_and_record_ancestors` has already correctly decremented those same variables for any evicted entries. The evicted entries' sizes and cycles are silently lost from the accounting, permanently inflating both counters.

---

### Finding Description

`add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```
// Step 1 — snapshot taken BEFORE any evictions
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//   = (self.total_tx_size + entry.size,
//      self.total_tx_cycles + entry.cycles)

// Step 2 — may call remove_entry_and_descendants → remove_entry
//           → update_stat_for_remove_tx, which CORRECTLY decrements
//             self.total_tx_size and self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3 — snapshot from Step 1 OVERWRITES the correctly-decremented values
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`check_and_record_ancestors` enters the eviction branch when the incoming transaction's ancestor count exceeds `max_ancestors_count` but some of those ancestors are cell-dep-referencing parents that can be removed to bring the count within limits: [2](#0-1) 

Inside that branch, `remove_entry_and_descendants` is called, which calls `remove_entry`, which calls `update_stat_for_remove_tx` — correctly decrementing `self.total_tx_size` and `self.total_tx_cycles`: [3](#0-2) [4](#0-3) 

But immediately after `check_and_record_ancestors` returns, lines 218–219 overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale snapshot from Step 1, which was computed before any evictions occurred. The net effect:

| State | `total_tx_size` |
|---|---|
| Before call | `T` |
| After Step 2 (correct) | `T − E` (E = sum of evicted sizes) |
| After Step 3 (bug) | `T + new_size` (evicted sizes never subtracted) |

Correct value should be `T − E + new_size`. The counters are inflated by `E` (the combined size/cycles of all evicted entries) on every such insertion. [5](#0-4) 

---

### Impact Explanation

**Impact: High**

`total_tx_size` is the sole guard used by `limit_size()` to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee entries
}
``` [6](#0-5) 

An inflated `total_tx_size` causes `limit_size()` to evict legitimate, fee-paying transactions that would otherwise fit within the pool. Each subsequent insertion that triggers the eviction path compounds the inflation. The `tx_pool_info` RPC also returns the corrupted values: [7](#0-6) 

Concrete consequences:
- Valid transactions are evicted from the pool even though actual pool size is within limits.
- The pool's effective capacity is silently reduced, degrading throughput for all users.
- `total_tx_size` reported via `tx_pool_info` RPC is incorrect, misleading operators and tooling.

---

### Likelihood Explanation

**Likelihood: Medium**

The eviction path in `check_and_record_ancestors` is triggered when:
1. The new transaction's ancestor count exceeds `max_ancestors_count` (default 25), AND
2. Some of those ancestors are cell-dep-referencing parents (`cell_ref_parents`) whose removal would bring the count within limits. [8](#0-7) 

This requires a moderately deep transaction chain where a new transaction uses a cell-dep pointing to an existing pool transaction. This is a standard CKB transaction feature (cell deps are a first-class construct) accessible to any unprivileged transaction sender. An attacker can deliberately craft such chains to repeatedly trigger the inflation, or it can occur organically in DeFi/contract-heavy workloads.

---

### Recommendation

Compute the pre-eviction snapshot **after** `check_and_record_ancestors` completes, or recompute the delta accounting for evictions. The simplest fix is to move the stat update to after evictions are known:

```diff
-    let (total_tx_size, total_tx_cycles) =
-        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
     evicts = self.check_and_record_ancestors(&mut entry)?;
+    let (total_tx_size, total_tx_cycles) =
+        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
     self.record_entry_edges(&entry)?;
     self.insert_entry(&entry, status);
     self.record_entry_descendants(&entry);
     self.track_entry_statics(None, Some(status));
     self.total_tx_size = total_tx_size;
     self.total_tx_cycles = total_tx_cycles;
```

At this point `self.total_tx_size` already reflects the post-eviction state (decremented by `update_stat_for_remove_tx` inside `remove_entry`), so adding `entry.size` on top of it produces the correct final value. [9](#0-8) 

---

### Proof of Concept

1. Fill the pool with a chain of 24 transactions `T1 → T2 → … → T24`, where `T2` uses `T1`'s output as a **cell dep** (not a consumed input).
2. Submit transaction `T25` that spends `T24`'s output and also uses `T1`'s output as a cell dep. Now `ancestors_count = 25 = max_ancestors_count`, so the check at line 598 passes and no eviction occurs yet.
3. Submit `T26` that spends `T25`'s output and uses `T1` as a cell dep. `ancestors_count = 26 > 25`, and `cell_ref_parents = {T1}`, so `26 − 1 = 25 ≤ 25` → eviction branch entered. `T1` (and its descendants `T2…T24`) are removed via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by their combined size `E`.
4. Lines 218–219 then overwrite `self.total_tx_size` with the pre-eviction snapshot `old_total + T26.size`, ignoring the `−E` correction.
5. Query `tx_pool_info`: `total_tx_size` is inflated by `E` (the size of 24 evicted transactions) even though the pool now contains only `T26`.
6. Subsequent `limit_size()` calls will evict legitimate transactions based on this inflated counter. [10](#0-9) [11](#0-10)

        self.total_tx_cycles = total_tx_cycles;

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
