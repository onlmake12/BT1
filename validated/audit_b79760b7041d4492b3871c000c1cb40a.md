### Title
Unauthenticated `estimate_cycles` RPC Executes Scripts Up to Full Block Cycle Limit Without Rate Limiting — (`File: rpc/src/module/chain.rs`)

---

### Summary

The `estimate_cycles` (and deprecated `dry_run_transaction`) RPC endpoint executes arbitrary transaction scripts up to `consensus.max_block_cycles` on every call, with no rate limiting, no authentication, and no per-request cycle cap. An unprivileged remote caller can repeatedly submit transactions whose scripts consume the full block cycle budget, saturating the node's CPU and preventing it from processing legitimate transactions and blocks.

---

### Finding Description

`CyclesEstimator::run()` in `rpc/src/module/chain.rs` resolves and fully executes a caller-supplied transaction using `ScriptVerifier::verify(max_cycles)`, where `max_cycles` is set directly to `consensus.max_block_cycles`:

```rust
let max_cycles = consensus.max_block_cycles;
// ...
ScriptVerifier::new(...).verify(max_cycles)
``` [1](#0-0) 

`consensus.max_block_cycles` is the maximum cycle budget for an entire block — on mainnet this is 3,500,000,000 cycles. The RPC handler performs no rate limiting, no authentication, and imposes no per-call cycle ceiling below the full block budget. [2](#0-1) 

The deprecated `dry_run_transaction` in the Experiment module calls the identical `CyclesEstimator::new(&self.shared).run(tx)` path: [3](#0-2) 

The RPC layer has no rate-limiting middleware — a grep across all `rpc/**/*.rs` files finds zero occurrences of any rate-limit, throttle, or concurrency-limit mechanism.

The `ScriptVerifier::verify` loop runs every script group sequentially, consuming real CPU cycles up to the provided limit before returning: [4](#0-3) 

---

### Impact Explanation

An attacker who can reach the node's RPC port (default: `localhost:8114`, but commonly exposed) can:

1. Deploy a script that spins in a tight loop consuming close to `max_block_cycles` cycles.
2. Flood `estimate_cycles` with transactions referencing that script.
3. Each call forces the node to execute the script for up to 3.5 billion cycles of real CPU work.
4. Concurrent or rapid sequential calls saturate the node's CPU, starving the block-processing pipeline, the tx-pool, and the P2P sync threads of CPU time.

The result is a denial-of-service against the node: it cannot assemble block templates, validate incoming blocks, or relay transactions within the expected time window.

---

### Likelihood Explanation

- The `estimate_cycles` endpoint is part of the default-enabled Chain RPC module.
- No authentication or API key is required.
- No rate limiting exists anywhere in the RPC stack.
- The attacker only needs network access to the RPC port and a one-time script deployment on-chain.
- The attack is cheap to sustain: each call costs the attacker only an HTTP request; the node pays the full CPU cost.

---

### Recommendation

1. **Introduce a per-call cycle cap** for `estimate_cycles` that is significantly lower than `max_block_cycles` (e.g., a configurable `max_estimate_cycles` defaulting to a fraction of the block limit).
2. **Add RPC rate limiting** (per-IP or global) for compute-heavy endpoints such as `estimate_cycles` and `dry_run_transaction`.
3. **Run script execution for RPC calls in a dedicated thread pool** with bounded concurrency so that RPC-triggered script execution cannot starve the block-processing pipeline.
4. Consider requiring the caller to declare an expected cycle count (analogous to `declared_cycles` in `_process_tx`) and reject calls that exceed it.

---

### Proof of Concept

```
1. Deploy a CKB script (lock script) that executes a tight loop consuming ~3,499,000,000 cycles.
   Store it in a live cell on-chain (one-time cost).

2. Construct a transaction T:
   - One input cell locked by the above script.
   - cell_dep pointing to the script cell.
   - A witness that causes the script to run its full loop.

3. In a loop, send HTTP POST to the node's RPC port:
   POST http://<node>:8114/
   {"id":1,"jsonrpc":"2.0","method":"estimate_cycles","params":[<T>]}

4. Each call causes CyclesEstimator::run() → ScriptVerifier::verify(3_500_000_000)
   to execute the full loop, consuming seconds of CPU per call.

5. With even a modest request rate (e.g., 10 concurrent callers), the node's
   CPU is fully saturated, block template generation stalls, and the node
   falls behind on block validation.
``` [1](#0-0) [4](#0-3)

    pub(crate) fn run(&self, tx: packed::Transaction) -> Result<EstimateCycles> {
        let snapshot = self.shared.cloned_snapshot();
        let consensus = snapshot.cloned_consensus();
        match resolve_transaction(tx.into_view(), &mut HashSet::new(), self, self) {
            Ok(resolved) => {
                let max_cycles = consensus.max_block_cycles;
                let tip_header = snapshot.tip_header();
                let tx_env = TxVerifyEnv::new_submit(tip_header);
                match ScriptVerifier::new(
                    Arc::new(resolved),
                    snapshot.as_data_loader(),
                    consensus,
                    Arc::new(tx_env),
                )
                .verify(max_cycles)
                {
                    Ok(cycles) => Ok(EstimateCycles {
                        cycles: cycles.into(),
                    }),
                    Err(err) => Err(RPCError::custom_with_error(
                        RPCError::TransactionFailedToVerify,
                        err,
                    )),
                }
            }
            Err(err) => Err(RPCError::custom_with_error(
                RPCError::TransactionFailedToResolve,
                err,
            )),
        }
    }

    pub fn verify(&self, max_cycles: Cycle) -> Result<Cycle, Error> {
        let mut cycles: Cycle = 0;

        // Now run each script group
        for (_hash, group) in self.groups() {
            // max_cycles must reduce by each group exec
            let used_cycles = self
                .verify_script_group(group, max_cycles - cycles)
                .map_err(|e| {
                    #[cfg(feature = "logging")]
                    logging::on_script_error(_hash, &self.hash(), &e);
                    e.source(group)
                })?;

            cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
        }
        Ok(cycles)
    }
