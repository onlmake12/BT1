### Title
Missing `provider` Address in `echoCallback` Prevents Consumer Authentication of Fulfilling Provider - (File: `target_chains/ethereum/contracts/contracts/echo/IEcho.sol`)

---

### Summary

The `IEchoConsumer.echoCallback` interface omits the `provider` address from its callback signature. Combined with the fact that the request record is deleted from storage **before** the callback is invoked, consumer contracts have no on-chain mechanism to verify which provider fulfilled their price update request. This is a direct analog to the Teleporter Token Bridge's missing `originSenderAddress`/`originBridgeAddress` issue.

---

### Finding Description

`IEchoConsumer` defines the callback as:

```solidity
function echoCallback(
    uint64 sequenceNumber,
    PythStructs.PriceFeed[] memory priceFeeds
) internal virtual;
``` [1](#0-0) 

The analogous Entropy interface explicitly includes the `provider`:

```solidity
function entropyCallback(
    uint64 sequence,
    address provider,
    bytes32 randomNumber
) internal virtual;
``` [2](#0-1) 

In `Echo.sol`, `clearRequest(sequenceNumber)` is called at line 164 **before** the callback dispatch at line 177:

```solidity
clearRequest(sequenceNumber);          // line 164 — request deleted here
...
IEchoConsumer(req.requester)._echoCallback{  // line 177 — callback fires after deletion
    gas: req.callbackGasLimit
}(sequenceNumber, priceFeeds)
``` [3](#0-2) 

Because the request is cleared before the callback, the consumer cannot call `getRequest(sequenceNumber)` to recover the `provider` field retroactively. The `Request` struct does store `provider`: [4](#0-3) 

After the exclusivity period expires, **any registered provider** may call `executeCallback` and credit themselves: [5](#0-4) 

---

### Impact Explanation

A consumer contract that needs to authenticate which provider fulfilled its request — for example, to enforce that only a specific trusted provider's fulfillments trigger sensitive downstream logic — cannot do so. The callback receives no provider identity, and the storage record is already gone. Any registered provider can fulfill after the exclusivity window, and the consumer is blind to this substitution at callback time.

---

### Likelihood Explanation

Any Echo consumer that implements provider-specific access control or conditional logic based on the fulfilling provider is affected. The exclusivity period is configurable and can be set to zero, making the window for provider substitution immediate. The entry path is fully unprivileged: any registered provider can call `executeCallback` after the exclusivity period.

---

### Recommendation

Add the `provider` address to both `_echoCallback` and `echoCallback` in `IEcho.sol`, mirroring the `IEntropyConsumer` pattern:

```solidity
function _echoCallback(
    uint64 sequenceNumber,
    address provider,
    PythStructs.PriceFeed[] memory priceFeeds
) external { ... }

function echoCallback(
    uint64 sequenceNumber,
    address provider,
    PythStructs.PriceFeed[] memory priceFeeds
) internal virtual;
```

Pass `providerToCredit` (the actual fulfilling provider) from `executeCallback` into the callback invocation.

---

### Proof of Concept

1. Consumer contract `C` calls `requestPriceUpdatesWithCallback(trustedProvider, ...)`, storing `req.provider = trustedProvider`.
2. The exclusivity period elapses.
3. Attacker (a registered but untrusted provider `P2`) calls `executeCallback(P2, sequenceNumber, updateData, priceIds)`.
4. Echo validates the price data, credits `P2`, calls `clearRequest(sequenceNumber)` (request deleted), then calls `C._echoCallback(sequenceNumber, priceFeeds)`.
5. Inside `C.echoCallback`, the consumer receives only `sequenceNumber` and `priceFeeds`. It cannot determine the fulfilling provider was `P2` rather than `trustedProvider`. Calling `getRequest(sequenceNumber)` returns an empty/zeroed struct because the request was already cleared.
6. Any provider-gated logic in `C` is bypassed.
