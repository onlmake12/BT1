Audit Report

## Title
Duplicate Transaction Inputs Not Rejected, Enabling Capacity Double-Counting - (`verification/src/transaction_verifier.rs`, `util/types/src/core/cell.rs`)

## Summary
`NonContextualTransactionVerifier` checks for duplicate `cell_deps` and `header_deps` but has no equivalent check for duplicate inputs. The `ResolvedTransaction::check` function silently accepts a repeated input `OutPoint` via the `checked_cells` early-return path designed for dep deduplication. As a result, `CapacityVerifier::verify` double-counts the capacity of the repeated cell, allowing a transaction sender to create outputs whose total capacity exceeds the capacity of the cells they actually own.

## Finding Description

**Step 1 — No duplicate-input check in `NonContextualTransactionVerifier`.**

`NonContextualTransactionVerifier` runs six sub-verifiers: `version`, `size`, `empty`, `duplicate_deps`, `outputs_data_verifier`, `script_hash_type`. `DuplicateDepsVerifier` explicitly deduplicates `cell_deps` and `header_deps` using a `HashSet`, but there is no corresponding verifier for inputs. [1](#0-0) 

A transaction whose `inputs` list contains the same `OutPoint` twice passes `NonContextualTransactionVerifier::verify` without error.

**Step 2 — `check` in `cell.rs` silently accepts the second occurrence.**

`ResolvedTransaction::check` uses a single `check_cell` closure for both inputs and cell deps, sharing a `checked_cells: HashSet<OutPoint>`. When the first occurrence of an input `OutPoint` is processed, it passes `is_live` and is inserted into `checked_cells`. When the second occurrence is processed, it hits the `checked_cells.contains` branch and returns `Ok(())` without error. [2](#0-1) 

The `seen_inputs` parameter (the cross-transaction double-spend guard) provides no within-transaction protection because it is only populated with the current transaction's inputs *after* all checks complete. [3](#0-2) 

**Step 3 — `CapacityVerifier` double-counts capacity.**

`inputs_capacity` iterates over `resolved_inputs`, which is a `Vec<CellMeta>` built one-to-one from the transaction's input list. A transaction with the same `OutPoint` twice produces two `CellMeta` entries for the same cell, and `inputs_capacity` sums all entries without deduplication. [4](#0-3) 

`CapacityVerifier::verify` then computes `inputs_sum = 2 × cell_capacity` and allows `outputs_sum` up to the same inflated value. [5](#0-4) 

## Impact Explanation

An unprivileged transaction sender can craft a transaction listing the same live cell `OutPoint` N times in `inputs`. `CapacityVerifier` computes `inputs_sum = N × cell_capacity`, permitting `outputs_sum` up to the same inflated value. The attacker effectively creates CKB capacity out of thin air — up to `(N-1) × cell_capacity` per transaction — permanently inflating the circulating supply upon block commitment. This matches the allowed Critical impact: **Vulnerabilities which could easily damage CKB economy**.

## Likelihood Explanation

The attack requires only the ability to submit a transaction via the standard RPC (`send_transaction`) or P2P relay. No privileged access, key compromise, or majority hashpower is needed. The attacker needs only one live cell they control. The crafted transaction passes all existing non-contextual and contextual checks as described. Likelihood is **high** for any actor motivated by financial gain.

## Recommendation

Add a `DuplicateInputsVerifier` mirroring `DuplicateDepsVerifier` inside `NonContextualTransactionVerifier`:

```rust
// In NonContextualTransactionVerifier::verify():
let mut seen = HashSet::with_capacity(tx.inputs().len());
if let Some(input) = tx.inputs_iter().find_map(|i| seen.replace(i.previous_output())) {
    return Err(TransactionError::DuplicateInputs { out_point: input }.into());
}
```

Alternatively, inside `ResolvedTransaction::check`, use a separate `HashSet` for inputs distinct from `checked_cells`, so the early-return path for dep deduplication does not apply to inputs. [6](#0-5) 

## Proof of Concept

1. Attacker owns cell `C` with capacity 100 CKB at `OutPoint { tx_hash: H, index: 0 }`.
2. Attacker builds a transaction:
   - `inputs`: `[OutPoint(H,0), OutPoint(H,0)]` — same cell listed twice
   - `outputs`: one cell with capacity 200 CKB (attacker's lock)
3. `NonContextualTransactionVerifier::verify()` passes — no duplicate-input check exists.
4. `ResolvedTransaction::check()` passes — second input hits `checked_cells` early return at line 321-323.
5. `CapacityVerifier::verify()` computes `inputs_sum = 200 CKB`, `outputs_sum = 200 CKB` → passes.
6. Lock script executes; attacker controls the key, so it passes.
7. Transaction is committed; attacker now holds a 200 CKB cell having spent only 100 CKB.

        let mut seen_cells = HashSet::with_capacity(self.transaction.cell_deps().len());

            .find_map(|hash| seen_headers.replace(hash))
        {
            return Err(TransactionError::DuplicateHeaderDeps { hash }.into());
        }
        Ok(())
    }
