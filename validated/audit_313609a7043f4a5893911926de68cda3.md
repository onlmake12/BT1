### Title
Provider Acting as Requester Breaks Randomness Guarantee — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract allows a registered randomness provider to also act as the requester of their own randomness. Because the provider knows their entire hash chain in advance, they can choose a `userContribution` that, combined with their known `x_i`, produces a predetermined favorable random number. This collapses the 2-party security model into a single-party model, completely breaking the randomness guarantee.

---

### Finding Description

The Entropy protocol's security guarantee is stated explicitly in the contract:

> "This protocol has the property that the result is random as long as either the provider or user is honest."

This guarantee requires two distinct, independent parties. In `requestHelper`, the requester is recorded as `msg.sender` with no check that `msg.sender != provider`: [1](#0-0) 

A registered provider can call `requestV2(their_own_address, userContribution, gasLimit)`, making themselves both the provider and the requester. Since the provider pre-computes their entire hash chain before registering: [2](#0-1) 

they know `x_i` for every future sequence number. When they are also the requester, they control `userContribution` as well. The final random number is:

```
r = hash(x_i, userContribution)
```

Both inputs are known to the attacker before the request is submitted. They can brute-force or directly compute a `userContribution` that yields a favorable `r`.

The `revealWithCallback` path has no restriction on who calls it, so the provider-requester can fulfill their own request: [3](#0-2) 

The `requestV2` entry points that accept a `provider` address parameter perform no identity check between `msg.sender` and `provider`: [4](#0-3) 

---

### Impact Explanation

Any application using Pyth Entropy for high-value randomness (NFT minting, on-chain lotteries, gaming outcomes, random selection) is vulnerable to complete outcome manipulation by a malicious provider who self-requests. The attacker receives the random number they choose, not a random one. The protocol's core security property — "random as long as either party is honest" — is void when both parties are the same address.

---

### Likelihood Explanation

Provider registration is permissionless: [5](#0-4) 

Any attacker can register as a provider with a crafted hash chain, then convince or wait for a target application to use them (or use the default provider path). The attack requires no privileged access beyond being a registered provider, which is open to anyone. The attacker only needs to pay the required fee (which they also accrue back to themselves as the provider).

---

### Recommendation

Add a check in `requestHelper` that the requester is not the provider:

```solidity
if (msg.sender == provider) revert EntropyErrors.ProviderCannotBeRequester();
```

This mirrors the fix applied in the referenced external report (commit `714e99e`), which added checks ensuring the operator address cannot equal the buyer or seller address in the same transaction.

---

### Proof of Concept

1. Attacker calls `register(feeInWei, commitment, ..., chainLength, uri)` as `address(attacker)`, pre-computing the full hash chain `[x_{N-1}, ..., x_0]`.
2. Target application is configured to use `address(attacker)` as provider (or attacker waits for a user to call `requestV2(attacker, ...)`).
3. Attacker calls `requestV2(address(attacker), userContribution, 0)` where `userContribution` is chosen so that `hash(constructUserCommitment(userContribution), x_i)` matches the stored commitment and `combineRandomValues(userContribution, x_i, 0)` equals the attacker's desired outcome. [6](#0-5) 

4. Attacker calls `revealWithCallback(address(attacker), sequenceNumber, userContribution, x_i)`.
5. The callback fires with the attacker-chosen `randomNumber`. The attacker also withdraws the accrued provider fee via `withdraw()`. [7](#0-6)

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    function requestV2(
        address provider,
        bytes32 userContribution,
        uint32 gasLimit
    ) public payable override returns (uint64) {
        EntropyStructsV2.Request storage req = requestHelper(
            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed
            // random number. Hence, useBlockHash is set to false.
            false,
            true,
            gasLimit
        );

        emit RequestedWithCallback(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );
        emit EntropyEventsV2.Requested(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }
