### Title
Stale Descendant Weight Tracking in `remove_entry_and_descendants` Causes Incorrect Eviction Priority — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all transaction links are torn down **before** `remove_entry` is called for each removed entry. This prevents `update_ancestors_index_key` from locating and updating the ancestor entries' descendant-weight fields (`descendants_fee`, `descendants_size`, `descendants_cycles`, `descendants_count`). Ancestors that remain in the pool after a conflict-driven removal retain permanently inflated descendant statistics, corrupting their `EvictKey` and causing the pool to keep low-value transactions that should be evicted while rejecting legitimate high-fee submissions.

---

### Finding Description

`remove_entry_and_descendants` is the code path used whenever a conflicting transaction (or a dep-consuming transaction) is removed from the pool along with its entire descendant subtree:

```rust
// tx-pool/src/component/pool_map.rs  lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ← ALL links removed here
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← remove_entry called after links are gone
        .collect()
}
``` [1](#0-0) 

The comment explains the intent: pre-removing links prevents `update_descendants_index_key` from walking into already-removed entries. However, `remove_entry` also calls `update_ancestors_index_key`, which is supposed to walk **up** to surviving ancestors and decrement their descendant-weight fields:

```rust
// tx-pool/src/component/pool_map.rs  lines 432-445
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());  // ← returns ∅ because links already removed
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),  // ← never reached
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
``` [2](#0-1) 

Because `remove_entry_links` has already erased the entry from the links map, `calc_ancestors` returns an empty set. The surviving ancestor's `sub_descendant_weight` is never called, so its `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` remain at their previously-incremented values indefinitely.

This is triggered every time `resolve_conflict` removes a conflicting transaction:

```rust
// tx-pool/src/component/pool_map.rs  lines 305-332
pub(crate) fn resolve_conflict(&mut self, tx: &TransactionView) -> Vec<ConflictEntry> {
    for i in tx.input_pts_iter() {
        if let Some(id) = self.edges.remove_input(&i) {
            let entries = self.remove_entry_and_descendants(&id);  // ← ancestor not updated
            ...
        }
        if let Some(x) = self.edges.remove_deps(&i) {
            for id in x {
                let entries = self.remove_entry_and_descendants(&id);  // ← same
                ...
            }
        }
    }
}
``` [3](#0-2) 

The same path is also triggered from `check_and_record_ancestors` when cell-ref-parent eviction occurs. [4](#0-3) 

---

### Impact Explanation

The corrupted fields feed directly into `EvictKey` computation:

```rust
// tx-pool/src/component/entry.rs  lines 234-247
impl From<&TxEntry> for EvictKey {
    fn from(entry: &TxEntry) -> Self {
        let descendants_weight =
            get_transaction_weight(entry.descendants_size, entry.descendants_cycles);
        let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
        let feerate = FeeRate::calculate(entry.fee, weight);
        EvictKey {
            fee_rate: descendants_feerate.max(feerate),
            timestamp: entry.timestamp,
            descendants_count: entry.descendants_count,
        }
    }
}
``` [5](#0-4) 

A surviving ancestor with stale-high `descendants_fee` and `descendants_count` will have an artificially elevated `EvictKey::fee_rate`. `next_evict_entry` selects the entry with the **lowest** evict key, so the inflated ancestor is skipped during eviction. The pool fills with these phantom-valuable entries, and `limit_size` rejects legitimate high-fee transactions with `Reject::Full`. [6](#0-5) 

---

### Likelihood Explanation

Any unprivileged RPC caller (`send_transaction`) can trigger this:

1. Submit parent transaction **A** (low fee).
2. Submit child transaction **B** spending A's output (high fee, inflating A's `descendants_fee`).
3. Submit a conflicting transaction **C** that double-spends B's input (valid RBF or simple conflict).
4. `resolve_conflict` removes B via `remove_entry_and_descendants`; A's descendant tracking is not decremented.
5. Repeat to accumulate many such "zombie-ancestor" entries.

No special privilege, key, or majority hashpower is required. The attack is cheap (only transaction fees for A and C; B is evicted so its fee is lost, but A's fee can be minimal).

---

### Recommendation

In `remove_entry_and_descendants`, before tearing down links, walk up to each surviving ancestor and call `sub_descendant_weight` for every entry being removed. Alternatively, restructure the function so that `update_ancestors_index_key` is called while the links are still intact (before `remove_entry_links`), and then suppress only the `update_descendants_index_key` call inside `remove_entry` for the batch-removal case.

---

### Proof of Concept

1. Pool contains: `A` (fee=1 shannon, size=100) → `B` (fee=1000 shannons, size=100).
2. After adding B, A's state: `descendants_fee=1001`, `descendants_size=200`, `descendants_count=2`.
3. Submit `C` conflicting with B. `resolve_conflict` calls `remove_entry_and_descendants(B)`.
4. Links for B are removed. `remove_entry(B)` calls `update_ancestors_index_key(B, Remove)` → `calc_ancestors(B)` returns `∅` → A is not updated.
5. A now has `descendants_fee=1001`, `descendants_size=200`, `descendants_count=2` even though it has zero descendants.
6. A's `EvictKey::fee_rate = FeeRate(1001 / weight(200, cycles))` — far above its actual `FeeRate(1 / weight(100, cycles))`.
7. When the pool is full, A is never selected by `next_evict_entry`, blocking admission of legitimate transactions.

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }
