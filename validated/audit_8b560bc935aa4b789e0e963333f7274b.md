### Title
Deprecated `parse_and_verify_le_ecdsa_update` aborts on valid channel-4 updates, permanently freezing funds in non-migrated downstream protocols — (`lazer/contracts/sui/sources/channel.move`, `pyth_lazer.move`)

### Summary
The v1 entry point `parse_and_verify_le_ecdsa_update` is still a live `public fun`. When the Lazer network issues a legitimately signed update with channel byte `4` (`FixedRate1000ms`), the v2 parser accepts it, but the v1 compatibility shim `channel::from_v2` aborts with `EInvalidChannel` because it only maps channels 1–3. Any downstream DeFi protocol that has not migrated to `parse_and_verify_le_ecdsa_update_v2` will have every price-gated transaction abort, effectively freezing all funds behind those checks.

### Finding Description
`channel_v2::from_u8` accepts channel values 1–4. [4](#0-3) 

`channel::from_v2` (used by the v1 path) only handles 1–3 and aborts otherwise: [3](#0-2) 

`parse_and_verify_le_ecdsa_update` delegates to v2 parsing (which succeeds), then immediately calls `update_v1::from_v2`, which calls `channel::from_v2`, which aborts: [1](#0-0) 

The Lazer network is already producing channel-4 updates in production (the test data is from a real subscription). [8](#0-7) 

### Impact Explanation
Any consuming protocol that calls `parse_and_verify_le_ecdsa_update` (v1) will have every transaction abort with `EInvalidChannel` once the Lazer network exclusively or predominantly issues channel-4 updates. Since the abort happens inside the price-verification call, all logic gated on a successful price read becomes permanently unreachable, freezing any funds controlled by those code paths. The JS SDK has already migrated to v2, but on-chain Move contracts that integrated against the v1 ABI are silently broken. [9](#0-8) 

### Likelihood Explanation
- The Lazer network is already issuing channel-4 updates (confirmed by real test data). [10](#0-9) 
- The v1 entry is still `public` and callable by any on-chain contract. [11](#0-10) 
- No privileged access is required; any relayer submitting valid signed updates triggers the abort.
- The existing test suite explicitly documents and confirms the abort path. [12](#0-11) 

### Recommendation
1. **Remove or gate the v1 entry**: Either remove `parse_and_verify_le_ecdsa_update` entirely (breaking change, but safe) or have it return an `Option<update_v1::Update>` / emit an error type that callers can handle gracefully instead of aborting.
2. **Alternatively**, extend `channel::from_v2` to map channel 4 to a new `FixedRate1000ms` variant in the v1 `Channel` enum, preserving backward compatibility without aborting.
3. **Communicate urgency**: Since channel-4 updates are already live, any protocol still on the v1 ABI is already broken or will be imminently.

### Proof of Concept
The existing test `test_parse_and_verify_le_ecdsa_incompatible_update` is the proof of concept: it uses a real signed Lazer payload with channel 4, calls the v1 entry, and asserts `EInvalidChannel`. The companion test `test_parse_and_verify_le_ecdsa_update_v2` calls the v2 entry on the identical payload and succeeds. [7](#0-6)
