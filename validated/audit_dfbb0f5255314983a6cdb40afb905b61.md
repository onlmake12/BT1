### Title
Stale Price Returned Without Staleness Check in `PythAggregatorV3.latestRoundData()` and Related Functions - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink-compatible adapter contract, published as part of `@pythnetwork/pyth-sdk-solidity`. Every price-reading function in this contract calls `pyth.getPriceUnsafe()` — which explicitly makes no staleness guarantee — without any subsequent check on `publishTime`. Any protocol that integrates this adapter as a drop-in Chainlink replacement will silently consume arbitrarily stale prices.

---

### Finding Description

`PythAggregatorV3` implements the Chainlink `AggregatorV3Interface` to allow existing Chainlink-integrated protocols to migrate to Pyth. All five price-reading functions — `latestRoundData()`, `getRoundData()`, `latestAnswer()`, `latestTimestamp()`, and `decimals()` — call `pyth.getPriceUnsafe(priceId)` with no staleness validation:

```solidity
// PythAggregatorV3.sol lines 99–119
function latestRoundData()
    external
    view
    returns (
        uint80 roundId,
        int256 answer,
        uint256 startedAt,
        uint256 updatedAt,
        uint80 answeredInRound
    )
{
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (
        roundId,
        int256(price.price),
        price.publishTime,
        price.publishTime,
        roundId
    );
}
``` [1](#0-0) 

The same pattern applies to `getRoundData()`, `latestAnswer()`, `latestTimestamp()`, and `decimals()`: [2](#0-1) 

The `IPyth` interface documents `getPriceUnsafe` explicitly as returning a price "from arbitrarily far in the past": [3](#0-2) 

The safe alternative, `getPriceNoOlderThan`, is available in `AbstractPyth` and enforces a staleness check: [4](#0-3) 

The contract's own NatSpec acknowledges the absence of roundId tracking but does not warn about the missing staleness check: [5](#0-4) 

---

### Impact Explanation

Any DeFi protocol (lending, derivatives, AMM) that deploys `PythAggregatorV3` as a Chainlink feed replacement and calls `latestRoundData()` or `latestAnswer()` will receive a price that may be hours or days old with no on-chain revert. This is the exact analog of the reported Chainlink `latestRoundData` staleness issue:

- A lending protocol using this adapter for collateral valuation will accept stale collateral prices, enabling under-collateralized borrows or blocking valid liquidations.
- A derivatives protocol will settle positions at outdated prices, allowing profitable exploitation of known price divergence.
- The `updatedAt` field returned equals `publishTime` from the stale cached data, so downstream callers that check `updatedAt` against `block.timestamp` will also see a stale timestamp — but only if they implement that check themselves. The adapter provides no protection.

---

### Likelihood Explanation

This is a production SDK contract distributed via `@pythnetwork/pyth-sdk-solidity` and explicitly documented as the migration path for Chainlink users. The Chainlink migration guide in the repository directs users to deploy this contract as a direct replacement: [6](#0-5) 

Pyth is a pull oracle — prices are only updated when a caller submits fresh `updateData`. If no keeper or user submits an update, the cached price ages indefinitely. Any protocol that relies on `PythAggregatorV3` without implementing its own staleness check on the returned `updatedAt` is exposed. The likelihood is high because the contract is the official migration artifact and its interface matches Chainlink's exactly, giving integrators no signal that staleness protection is absent.

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan` using a configurable `maxAge` parameter set at construction time. Alternatively, add an explicit staleness check after each `getPriceUnsafe` call:

```solidity
constructor(address _pyth, bytes32 _priceId, uint _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // ...
}
```

At minimum, add a NatSpec warning on every price-reading function that the returned price may be arbitrarily stale and that callers must validate `updatedAt` against `block.timestamp`.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a valid `priceId`.
2. Do not call `updateFeeds()` for an extended period (e.g., 24 hours).
3. Call `latestRoundData()`. The returned `answer` is the price from 24 hours ago; `updatedAt` is the stale `publishTime`. No revert occurs.
4. A lending protocol consuming this adapter will value collateral at the 24-hour-old price, enabling profitable exploitation of any price movement that occurred in the interim.

The root cause is entirely within Pyth's own production SDK file: [7](#0-6)
