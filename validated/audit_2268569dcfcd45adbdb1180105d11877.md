### Summary

The guard check at line 201 is designed to cap total work per `GetLastStateProof` message, but it incorrectly assumes `last_n_numbers.len() â‰¤ last_n_blocks`. An attacker can set `difficulty_boundary` to a value just above the total difficulty at `start_block_number`, forcing `difficulty_boundary_block_number` to equal `start_block_number` and making `last_n_numbers` span the entire range `[start_block_number, last_block_number)` â€” potentially O(chain_height) entries â€” while the guard check still passes.

The `* 2` factor is intended to account for both `reorg_last_n_numbers` (â‰¤ `last_n_blocks`) and `last_n_numbers` (assumed â‰¤ `last_n_blocks`). This assumption is wrong.

`last_n_numbers` is built as:

```rust

    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
``` [2](#0-1) 

The only adjustment to `difficulty_boundary_block_number` is upward (to guarantee at least `last_n_blocks` entries):

```rust

    difficulty_boundary_block_number = last_block_number - last_n_blocks;
}
``` [3](#0-2) 

There is no downward clamp. If `difficulty_boundary_block_number` is already small (e.g., equal to `start_block_number`), `last_n_numbers` has `last_block_number - start_block_number` entries â€” unbounded by `last_n_blocks`.

`difficulty_boundary_block_number` is determined by a binary search for the first block whose cumulative difficulty â‰¥ the attacker-supplied `difficulty_boundary`: [4](#0-3) 

By setting `difficulty_boundary` to a value just above the total difficulty at `start_block_number`, the attacker forces `difficulty_boundary_block_number = start_block_number`, making `last_n_numbers.len() = last_block_number - start_block_number`.

For each entry in the combined `block_numbers`, `complete_headers` calls `chain_root_mmr(number - 1).get_root()` â€” a synchronous RocksDB MMR traversal: [5](#0-4) 

This entire loop runs synchronously inside the `async fn execute`, blocking the Tokio worker thread. [6](#0-5)

### Impact Explanation

**Concrete worst-case scenario:**

| Field | Value |
|---|---|
| `last_n_blocks` | `1` |
| `difficulties` | `[]` (empty) |
| `start_block_number` | `0` |
| `difficulty_boundary` | `1` (just above genesis cumulative difficulty) |
| `last_block_number` | `10 000` |

Guard check: `0 + 1Ã—2 = 2 â‰¤ 1000` â†’ **passes**.

Resulting `last_n_numbers`: `(0..10000)` = **10 000 entries**.

Each entry triggers one `chain_root_mmr(n-1).get_root()` call, which reads O(log N) MMR nodes from RocksDB. At chain height 10 000, that is â‰ˆ14 reads/entry Ã— 10 000 entries = **~140 000 synchronous RocksDB reads** in a single async handler invocation.

Because `complete_headers` is not `async` and contains no `.await` points, it holds the Tokio worker thread for the entire duration. Multiple concurrent crafted messages from different peers can exhaust the thread pool, causing async runtime starvation and degrading node responsiveness for all peers. [7](#0-6) 

---

### Likelihood Explanation

- The light-client protocol is a supported production feature, reachable by any unprivileged peer.
- The attack requires only a single well-crafted P2P message; no PoW, no key, no privileged role.
- The `difficulty_boundary` field is fully attacker-controlled with no server-side upper bound check beyond the `difficulties` array consistency checks (which are bypassed by using an empty `difficulties` array).
- The `start_block_number = 0` path skips the "first difficulty > previous total difficulty" check entirely. [8](#0-7) 

---

### Recommendation

Replace the guard with a check that bounds the **actual** size of `last_n_numbers` rather than assuming it equals `last_n_blocks`. The simplest fix is to add a post-computation check before calling `complete_headers`:

```rust
if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many block numbers");
}
```

Alternatively, clamp `difficulty_boundary_block_number` from below so that `last_n_numbers.len()` is always â‰¤ `last_n_blocks`:

```rust
// Ensure last_n_numbers never exceeds last_n_blocks entries
let lower_bound = last_block_number.saturating_sub(last_n_blocks);
if difficulty_boundary_block_number < lower_bound {
    difficulty_boundary_block_number = lower_bound;
}
``` [9](#0-8)

2. Craft a `GetLastStateProof` message:

   - `last_n_blocks`: `1`
   - `difficulty_boundary`: `U256::from(1u64)` (just above genesis cumulative difficulty)
   - `difficulties`: empty
3. Send the message from an unprivileged peer.
4. Observe: guard check passes (`0 + 1Ã—2 = 2 â‰¤ 1000`); `last_n_numbers` = 10 000 entries; `complete_headers` performs ~140 000 synchronous RocksDB reads on the Tokio worker thread.
5. Repeat from multiple peers concurrently; measure wall-clock latency of other protocol handlers to confirm starvation.
