### Title
Verify Queue Has No Eviction Policy, Enabling Peer-Triggered DOS of Transaction Relay — (`tx-pool/src/component/verify_queue.rs`)

---

### Summary

The `VerifyQueue` in CKB's tx-pool has a hard 256 MB size cap (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) with **no eviction mechanism**. When the cap is reached, every subsequent attempt to enqueue a transaction — including legitimate relayed transactions from honest peers — is rejected with `Reject::Full`. A single malicious peer can fill the queue with syntactically valid but contextually invalid large transactions, permanently blocking the node's ability to accept relayed transactions until the queue drains naturally.

---

### Finding Description

`VerifyQueue::add_tx()` enforces a hard cap via `is_full()`:

```rust
// tx-pool/src/component/verify_queue.rs, line 104-106
pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}
```

When full, `add_tx` immediately returns an error with no eviction:

```rust
// tx-pool/src/component/verify_queue.rs, lines 215-219
if self.is_full(tx_size) {
    return Err(Reject::Full(format!(
        "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
        tx.hash()
    )));
}
``` [1](#0-0) [2](#0-1) [3](#0-2) 

This contrasts sharply with the main `pool_map`, which has an explicit `limit_size()` eviction loop that removes lowest-fee-rate entries when the pool exceeds `max_tx_pool_size`:

```rust
// tx-pool/src/pool.rs, lines 292-326
pub(crate) fn limit_size(...) {
    while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
        // evict lowest-fee-rate entry
    }
}
``` [4](#0-3) 

The verify queue has **no equivalent eviction**. There is also no per-peer quota on how many bytes a single peer may occupy in the queue.

The attacker-controlled entry path is `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue`:

```rust
// tx-pool/src/process.rs, lines 335-352
pub(crate) async fn resumeble_process_tx(...) -> Result<bool, Reject> {
    self.non_contextual_verify(&tx, remote).await?;   // cheap structural check only
    // ... duplicate checks ...
    self.enqueue_verify_queue(tx, is_proposal_tx, remote).await
}
``` [5](#0-4) [6](#0-5) 

Transactions are added to the verify queue **before** contextual verification (cell resolution, script execution). A peer can relay transactions that pass `non_contextual_verify` (structural checks only) but fail contextual checks. These occupy queue space until a worker dequeues and rejects them.

The maximum transaction size is 512 KB (`TRANSACTION_SIZE_LIMIT`): [7](#0-6) 

So `256 MB / 512 KB = 500` max-size transactions saturate the queue.

---

### Impact Explanation

Once the verify queue is full, every transaction arriving via P2P relay is rejected with `Reject::Full`. The node stops accepting any relayed transactions from any peer, disrupting its participation in the CKB mempool relay network. Transactions submitted via the local `send_transaction` RPC are **not** affected because that path calls `process_tx` → `_process_tx` directly, bypassing the verify queue entirely:

```rust
// tx-pool/src/service.rs, lines 805-812
Message::SubmitLocalTx(Request { responder, arguments: tx }) => {
    let result = service.process_tx(tx, None).await.map(|_| ());
    ...
}
``` [8](#0-7) 

The impact is therefore: **the node cannot propagate or receive relayed transactions from the P2P network**, effectively isolating it from the mempool relay layer. Miners running such a node would miss transactions from the broader network, and the node would fail to relay legitimate user transactions to peers.

---

### Likelihood Explanation

The attack requires only a single peer connection — no privileged access, no key material, no majority hashpower. The attacker needs to:

1. Connect to the target node as a P2P peer (publicly reachable).
2. Relay ~500 transactions of ~512 KB each that pass `non_contextual_verify` (structural validity only — no live cells required).
3. Maintain the connection to prevent `remove_txs_by_peer` cleanup on disconnect.

The 256 MB threshold is large but reachable in a sustained relay burst. There is no per-peer byte quota in the verify queue, no rate limiting on relay message processing, and no eviction to reclaim space. The attack is repeatable: once the queue drains (workers reject the invalid transactions), the attacker can refill it immediately.

---

### Recommendation

1. **Add per-peer quota enforcement** in `VerifyQueue`: track bytes per `PeerIndex` and reject entries from a peer that exceeds a configurable per-peer limit (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / max_peers`).

2. **Add eviction by lowest-priority entry** (analogous to `pool_map.limit_size()`): when the queue is full, evict the oldest or lowest-declared-cycle entry from the peer with the most bytes in the queue, rather than hard-rejecting the incoming transaction.

3. **Apply a tighter per-peer relay rate limit** at the network layer before transactions reach `resumeble_process_tx`.

---

### Proof of Concept

**Entry path** (production code only):

```
P2P peer sends RelayTransactions message
  → sync/relayer receives it
  → TxPoolController::submit_remote_tx(tx, declared_cycles, peer)
  → TxPoolService::submit_remote_tx(tx, declared_cycles, peer)   [process.rs:371]
  → resumeble_process_tx(tx, false, Some((declared_cycles, peer))) [process.rs:377]
  → non_contextual_verify(&tx, remote)   // passes: tx is structurally valid
  → enqueue_verify_queue(tx, false, remote)  [process.rs:352]
  → VerifyQueue::add_tx(tx, ...)
      → is_full(tx_size) == false  (queue not yet full)
      → insert into queue, total_tx_size += tx_size
```

Repeat with ~500 transactions of 512 KB each:

```
→ total_tx_size reaches 256_000_000
→ is_full(next_tx_size) == true
→ Err(Reject::Full("verify_queue total_tx_size exceeded ..."))
```

All subsequent relay transactions from any peer now receive `Reject::Full`. The node's relay participation is blocked until workers drain the queue by processing (and rejecting) the attacker's invalid transactions. [9](#0-8) [10](#0-9)

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
