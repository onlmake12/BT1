### Title
`transaction_maximum_withdraw` Reads Full u64 Witness Index While On-Chain DAO C Script Reads Only the Lowest Byte, Causing Incorrect Deposit Header (`ar`) Resolution in DAO Interest Calculation — (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` resolves the deposit block header by reading the full 8-byte (u64) index from the witness `input_type` field. The on-chain DAO C script (`dao.c`) reads only the **lowest byte** of that same 8-byte field. When a transaction encodes a witness index whose value exceeds 255 (high bytes non-zero), the two interpretations diverge: Rust resolves `header_deps[full_u64]` while the C VM resolves `header_deps[low_byte]`. This means the `ar` (accumulation rate) extracted from the deposit header — the exact analog of the "rate" in the external report — is sourced from a **different block** in Rust than in the C VM, producing incorrect DAO interest accounting and blocking valid DAO withdrawals from the tx-pool.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` decodes the deposit header index from the witness as a full `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and then uses it directly to index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
```

The on-chain DAO C script at `dao.c` reads only the **lowest byte** of the same 8-byte `input_type` field (documented in the test comment at `util/dao/src/tests.rs:490`):

> `Position 1: correct deposit block (what C VM resolves via lowest byte).`
> `Position 257: withdraw block (wrong — Rust resolves this with full u64).`

When a transaction sender encodes a witness index of `257` (`0x0101_0000_0000_0000` in little-endian):
- **C VM** reads byte `0x01` → index `1` → `header_deps[1]`
- **Rust** reads full u64 `257` → index `257` → `header_deps[257]`

These resolve to **different block headers**, and therefore different `ar` values. The `ar` field is the per-block accumulation rate used to compute DAO interest:

```rust
let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());
// ...
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
```

This is the direct analog of the external report's bug: the `rate` (here `deposit_ar`) is sourced from the wrong state before the interest calculation is performed.

The same `transaction_maximum_withdraw` is called from `withdrawed_interests`, which feeds into `dao_field_with_current_epoch` during block assembly in `tx-pool/src/block_assembler/mod.rs`:

```rust
let dao = DaoCalculator::new(consensus, &snapshot.borrow_as_data_loader())
    .dao_field_with_current_epoch(entries_iter, tip_header, current_epoch)?;
```

---

### Impact Explanation

**Impact 1 — Valid DAO withdrawal permanently blocked from tx-pool (Scenario A):**

Craft a transaction where:
- `header_deps[1]` = correct deposit block (block number = `deposited_block_number`) — C VM resolves this
- `header_deps[257]` = a different block (e.g., the withdraw block, number ≠ `deposited_block_number`) — Rust resolves this

C VM accepts the transaction (correct deposit block found at index 1). Rust's `transaction_maximum_withdraw` resolves index 257 → wrong block → `deposit_header.number() != deposited_block_number` → `DaoError::InvalidOutPoint`. The transaction is rejected from the tx-pool fee check even though it is on-chain valid. The DAO depositor cannot withdraw their funds through the normal RPC path.

**Impact 2 — Incorrect `S_i` (secondary issuance) in block DAO field (Scenario B):**

Craft a transaction where:
- `header_deps[257]` = correct deposit block — Rust resolves this (passes block number check)
- `header_deps[1]` = a different block with a different `ar` — C VM resolves this

Rust admits the transaction to the tx-pool (fee check passes). During block assembly, `dao_field_with_current_epoch` calls `withdrawed_interests` → `transaction_maximum_withdraw` using Rust's index 257 → computes `maximum_withdraw` based on the wrong `deposit_ar` → wrong `withdrawed_interests` → wrong `current_s` in the DAO field. The block template carries a corrupted `S_i` field. When validators check the block, they compute the same wrong `S_i` (same Rust code), so the block is not rejected on that basis — but the DAO secondary issuance accounting is silently corrupted, affecting future DAO reward distributions.

---

### Likelihood Explanation

The trigger requires a transaction with 258+ `header_deps` entries and a witness index > 255. While unusual in normal usage, this is fully attacker-controllable: any transaction sender can craft a DAO withdrawal phase-2 transaction with an arbitrarily large `header_deps` list and encode any u64 index in the witness. No privileged access, key material, or majority hashpower is required. The entry path is the standard `send_transaction` RPC.

---

### Recommendation

In `transaction_maximum_withdraw`, mask the decoded index to its lowest byte before indexing into `header_deps`, matching the on-chain DAO C script behavior:

```rust
// Before:
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// After:
Ok(header_deps_index_data.unwrap()[0] as u64)
```

Alternatively, reject any witness index whose high 7 bytes are non-zero with `DaoError::InvalidDaoFormat`, since the C VM would resolve it differently and the transaction would be invalid on-chain.
