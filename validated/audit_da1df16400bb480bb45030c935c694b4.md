Audit Report

## Title
Pre-Eviction Snapshot Overwrites Post-Eviction Counters in `add_entry`, Inflating `total_tx_size`/`total_tx_cycles` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::add_entry` snapshots `total_tx_size` and `total_tx_cycles` into locals before calling `check_and_record_ancestors`, which may evict transactions and correctly decrement those same struct fields via `update_stat_for_remove_tx`. After all evictions complete, lines 218–219 unconditionally overwrite the struct fields with the pre-eviction snapshot, permanently discarding every decrement. The result is that `total_tx_size` is inflated by the aggregate size of all evicted transactions, causing `limit_size` to expel legitimate pending transactions from the pool even when real occupancy is within bounds.

## Finding Description
In `add_entry` (L210–219), the sequence is:

1. **L210–211**: `updated_stat_for_add_tx` reads `self.total_tx_size + entry.size` into local variables `total_tx_size` and `total_tx_cycles`. No evictions have occurred yet.
2. **L213**: `check_and_record_ancestors` is called. When `ancestors_count > max_ancestors_count` and the excess is attributable to cell-ref parents, it enters the eviction loop (L615–625), calling `remove_entry_and_descendants` for each candidate. Each removal calls `update_stat_for_remove_tx` (L733–741), which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in place.
3. **L218–219**: `self.total_tx_size = total_tx_size` and `self.total_tx_cycles = total_tx_cycles` restore the pre-eviction snapshot, erasing every decrement applied in step 2.

Net effect: `self.total_tx_size` ends up as `(pre_eviction_total + entry.size)` instead of the correct `(pre_eviction_total − evicted_sizes + entry.size)`. The inflation equals the sum of all evicted transactions' sizes and persists until the node restarts or the pool is cleared. The underflow guard in `update_stat_for_remove_tx` is irrelevant — the decrements succeed correctly but are then silently overwritten. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
`total_tx_size` is the sole counter driving `TxPool::limit_size` (L298). An inflated counter causes `limit_size` to believe the pool is over-capacity and evict additional legitimate pending/gap/proposed transactions that should have been retained. This constitutes a **tx-pool denial-of-service**: honest users' transactions are expelled from the pool even though real occupancy is within the configured limit. Repeated triggering accumulates inflation monotonically until the node restarts. This matches the allowed High impact: **Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. [4](#0-3) 

## Likelihood Explanation
The eviction path is triggered when a submitted transaction has more than `max_ancestors_count` (default 1000) cell-ref ancestors. An unprivileged submitter can engineer this deliberately by submitting ≥1001 transactions each carrying a `cell_dep` pointing to a specific live cell output, then submitting one transaction that spends that cell as an input. Step 2 triggers eviction of the lowest-fee cell-ref parents. The attacker pays fees for the submitted transactions, but the inflation is permanent per invocation and the attack is repeatable. The existing integration test `TxPoolLimitAncestorCount` demonstrates exactly this scenario with 2000 cell-ref transactions and confirms 1002 are evicted.

## Recommendation
Remove the pre-eviction snapshot entirely. After `check_and_record_ancestors` returns, `self.total_tx_size` and `self.total_tx_cycles` already reflect all evictions. Add the new entry's contribution afterward:

```rust
// Delete lines 210-211 (the pre-eviction snapshot).
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Add entry weight post-eviction:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

If the overflow-check semantics of `updated_stat_for_add_tx` must be preserved, call it **after** `check_and_record_ancestors` returns so it operates on the post-eviction baseline.

## Proof of Concept
Using the existing `TxPoolLimitAncestorCount` test structure:

1. Pool starts empty; `total_tx_size = 0`.
2. Submit 2000 transactions each with `cell_dep` → cell `C`, each ~200 bytes. `total_tx_size ≈ 400_000`.
3. Submit `T_consume` spending cell `C` as input (~200 bytes).
   - `check_and_record_ancestors` detects 2002 ancestors > 1000; eviction path taken.
   - 1002 cell-ref parents are evicted; each calls `update_stat_for_remove_tx`, decrementing `self.total_tx_size` by ~200_400 bytes. After evictions: `self.total_tx_size ≈ 199_600`.
   - Pre-eviction snapshot was `400_000 + 200 = 400_200`.
   - Lines 218–219 restore: `self.total_tx_size = 400_200`.
   - **Inflation = ~200_400 bytes** (the evicted transactions are double-counted).
4. Repeat steps 2–3 with fresh cells. Each iteration inflates `total_tx_size` by ~200 KB.
5. After ~900 iterations, `total_tx_size` exceeds `max_tx_pool_size` (180 MB default) despite the real pool being well within limits. `limit_size` begins evicting honest transactions on every subsequent submission.
