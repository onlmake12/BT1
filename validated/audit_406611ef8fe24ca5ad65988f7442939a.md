Audit Report

## Title
`DaoCalculator::transaction_maximum_withdraw` Uses Zero as Sentinel to Misclassify Genesis-Block Phase-1 Withdrawal Cells as Deposit Cells — (File: util/dao/src/lib.rs)

## Summary

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads the 8-byte cell data of a DAO-typed input and interprets it as a u64 deposit block number, then guards the interest-calculation branch with `if deposited_block_number > 0`. When a DAO cell was deposited at block 0 (genesis), the phase-1 withdrawal cell's data is `0x0000000000000000` — identical to a deposit cell — so the guard fails, `maximum_withdraw` is set to `output.capacity()` (principal only), and the subsequent `safe_sub(outputs_capacity)` underflows, causing `check_tx_fee` to permanently reject the valid phase-2 withdrawal with `Reject::Malformed`.

## Finding Description

In `util/dao/src/lib.rs` lines 61–116, `transaction_maximum_withdraw` iterates over resolved inputs. For each DAO-typed cell it loads the 8-byte cell data and decodes it as a little-endian u64:

```rust
let deposited_block_number =
    match self.data_loader.load_cell_data(cell_meta) {
        Some(data) if data.len() == 8 => LittleEndian::read_u64(&data),
        _ => 0,
    };
if deposited_block_number > 0 {
    // withdrawing cell: calculate interest
    ...
} else {
    Ok(output.capacity().into())   // deposit cell: return principal only
}
```

The CKB DAO protocol encodes two states in cell data:
- **Deposit cell**: `0x0000000000000000` (8 zero bytes)
- **Phase-1 withdrawal cell**: deposit block number as u64 LE

When the deposit block is block 0, the phase-1 withdrawal cell data is also `0x0000000000000000`. `LittleEndian::read_u64` returns 0, the `> 0` guard fails, and the cell is treated as a deposit cell. `maximum_withdraw` is set to `output.capacity()` (principal only).

`transaction_fee` then computes:

```rust
maximum_withdraw.safe_sub(outputs_capacity)
```

A valid phase-2 withdrawal outputs `principal + interest`, so `outputs_capacity > maximum_withdraw`, causing `safe_sub` to return `CapacityError::Overflow`, which propagates as `DaoError::Overflow` → `Reject::Malformed` in `check_tx_fee` (`tx-pool/src/util.rs` lines 34–41).

The same zero-as-sentinel pattern appears in `test/src/specs/dao/dao_verifier.rs` lines 254–257:

```rust
let deposited_number = LittleEndian::read_u64(&input_data.raw_data()[0..8]);
if deposited_number == 0 {
    return false;
}
```

This confirms the pattern is load-bearing throughout the DAO accounting logic and that the test verifier would also misclassify a genesis-block withdrawal.

## Impact Explanation

A user holding a DAO cell deposited at block 0 cannot submit a valid phase-2 withdrawal transaction through the tx pool. The node's `check_tx_fee` permanently rejects it with `Reject::Malformed("Overflow", "expect (outputs capacity) <= (inputs capacity)")`. The user's DAO interest is inaccessible via the standard `send_transaction` RPC path. This constitutes a local RPC API failure for a specific class of DAO depositors, matching the **Note (0–500 points): Any local RPC API crash** impact category.

## Likelihood Explanation

On mainnet CKB, the genesis block does not contain user-depositable DAO cells, so mainnet is not affected. However, `genesis_dao_data_with_satoshi_gift` in `util/dao/utils/src/lib.rs` explicitly supports genesis blocks containing DAO cells, and custom chains (devnets, testnets, alternative deployments) can configure genesis-block DAO cells. Any such chain is fully affected. The entry path requires only a standard `send_transaction` RPC call — no privileged access, no key compromise.

## Recommendation

Replace the `deposited_block_number > 0` sentinel check with an explicit byte-level check distinguishing deposit cells (all-zero data) from phase-1 withdrawal cells (non-zero data encoding the deposit block number):

```rust
let cell_data = self.data_loader.load_cell_data(cell_meta);
let is_withdrawing_cell = matches!(&cell_data, Some(data) if data.len() == 8 && data.iter().any(|b| *b != 0));
if is_withdrawing_cell {
    let deposited_block_number = LittleEndian::read_u64(&cell_data.unwrap());
    // ... interest calculation
} else {
    Ok(output.capacity().into())
}
```

Apply the same fix to `is_dao_prepare_input` in `test/src/specs/dao/dao_verifier.rs` (lines 254–257).

## Proof of Concept

1. Configure a chain spec with a genesis block containing a DAO cell (using `genesis_dao_data_with_satoshi_gift`). The cell has data `0x0000000000000000` and the DAO type script.
2. Submit a phase-1 withdrawal transaction consuming the genesis DAO cell. The output cell has data `0x0000000000000000` (block 0 in LE) and the DAO type script.
3. Submit a phase-2 withdrawal transaction consuming the phase-1 cell, with `outputs_capacity = principal + interest`.
4. The node calls `check_tx_fee` → `DaoCalculator::transaction_fee` → `transaction_maximum_withdraw`.
5. `deposited_block_number = LittleEndian::read_u64([0,0,0,0,0,0,0,0]) = 0`.
6. The `if deposited_block_number > 0` branch is skipped; `maximum_withdraw = output.capacity()` (principal only).
7. `transaction_fee = maximum_withdraw.safe_sub(outputs_capacity)` underflows → `DaoError::Overflow` → `Reject::Malformed`.
