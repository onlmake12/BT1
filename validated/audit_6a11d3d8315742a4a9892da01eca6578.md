Audit Report

## Title
Unbounded `pending_get_block_proposals` Growth Enables Memory Exhaustion and Relay Timer Stall — (`File: sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

## Summary

`SyncState::pending_get_block_proposals` is an uncapped `DashMap` that any connected P2P peer can grow without bound by sending `GetBlockProposal` relay messages containing fabricated `ProposalShortId` values absent from the local tx pool. When the relay timer fires `prune_tx_proposal_request`, it clones the entire map before iterating it, causing O(N) memory allocation and O(N) CPU work proportional to the number of accumulated entries. Sustained flooding can exhaust heap memory or stall the relay timer task, degrading block propagation for all peers.

## Finding Description

`SyncState` declares the map with no capacity bound:

```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
``` [1](#0-0) 

`insert_get_block_proposals` inserts every supplied ID unconditionally:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [2](#0-1) 

`GetBlockProposalProcess::execute()` only enforces a per-message count limit (`max_block_proposals_limit * max_uncles_num`), then calls `insert_get_block_proposals` for every proposal ID not found in the tx pool, with no check on the total accumulated map size and no rate limiting: [3](#0-2) 

`drain_get_block_proposals` performs a full O(N) clone before clearing:

```rust
pub fn drain_get_block_proposals(&self) -> DashMap<...> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
``` [4](#0-3) 

`prune_tx_proposal_request` then iterates the entire cloned map on the async relay timer task: [5](#0-4) 

A `RateLimiter` type is defined in the relayer but is not applied to `GetBlockProposal` message handling. [6](#0-5) 

## Impact Explanation

A single malicious peer can continuously send `GetBlockProposal` messages, each carrying up to ~3,000 unique fabricated `ProposalShortId` values (10 bytes each). Between two timer ticks, the map grows without bound. When the timer fires: (1) `drain_get_block_proposals` allocates a full clone of the map — O(N) heap allocation — and (2) `prune_tx_proposal_request` iterates the entire map on the async relay timer task, blocking other relay operations. Sustained flooding leads to heap exhaustion (OOM crash) or relay timer stalling that degrades block propagation for all peers. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node."

## Likelihood Explanation

Any peer that completes the standard P2P handshake via the `RelayV3` protocol can send `GetBlockProposal` messages. No authentication, stake, or special role is required. The only guard is a per-message proposal count check; there is no per-peer rate limit and no cap on the total accumulated map size. The CHANGELOG confirms this exact class of unbounded-collection bug has been exploited in practice in the same codebase (`#3093: Resolve inflight proposals memory bloat issue`, `#3094: Fix inflight block potential memory bloat issues`, `#3110: Fix pending compact block memory bloat on abnormal flow`). [7](#0-6) 

## Recommendation

1. **Cap the map size**: in `insert_get_block_proposals`, reject new entries when `pending_get_block_proposals.len()` exceeds a configurable limit (e.g., `max_block_proposals_limit * max_uncles_num`).
2. **Per-peer quota**: track per-peer entry counts and disconnect or ignore peers that exceed the quota.
3. **Avoid full clone on drain**: replace `clone()` + `clear()` in `drain_get_block_proposals` with `std::mem::take` or swap with an empty `DashMap` to eliminate the O(N) allocation on every timer tick.

## Proof of Concept

1. Connect a peer to a CKB node via the standard `RelayV3` P2P handshake.
2. In a tight loop, send `GetBlockProposal` relay messages where `proposals` contains ~3,000 random 10-byte `ProposalShortId` values that do not correspond to any real transaction in the node's tx pool.
3. Observe `pending_get_block_proposals.len()` growing without bound between timer ticks (monitorable via metrics or a debug build).
4. When the relay timer fires `prune_tx_proposal_request`, observe O(N) heap allocation (from `clone()`) and O(N) CPU iteration on the relay timer task, proportional to accumulated entries.
5. Sustained flooding causes OOM or relay timer stall, confirmed by node crash or block propagation degradation.

    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        let message_len = self.message.proposals().len();
        {
            // The block proposal request is separate from uncles,
            // so here the limit is only used to calculate the maximum value of uncles
            let limit = shared.consensus().max_block_proposals_limit()
                * (shared.consensus().max_uncles_num() as u64);
            if message_len as u64 > limit {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "GetBlockProposal proposals count({message_len}) > consensus max_block_proposals_limit({limit})"
                ));
            }
        }

        let proposals: HashSet<packed::ProposalShortId> =
            self.message.proposals().to_entity().into_iter().collect();

        if proposals.len() != message_len {
            return StatusCode::RequestDuplicate.with_context("Request duplicate proposal");
        }

        let fetched_transactions = {
            let tx_pool = self.relayer.shared.shared().tx_pool_controller();
            let fetch_txs = tx_pool.fetch_txs(proposals.clone()).await;
            if let Err(e) = fetch_txs {
                debug_target!(
                    crate::LOG_TARGET_RELAY,
                    "relayer tx_pool_controller send fetch_txs error: {:?}",
                    e
                );
                return Status::ok();
            }
            fetch_txs.unwrap()
        };
        // Transactions that do not exist on this node
        let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
            .into_iter()
            .filter(|short_id| !fetched_transactions.contains_key(short_id))
            .collect();

        // Cache request, try process on timer
        self.relayer
            .shared()
            .state()
            .insert_get_block_proposals(self.peer, not_exist_proposals);
