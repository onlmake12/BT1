Audit Report

## Title
VerifyQueue Lacks Per-Peer Admission Limit, Enabling Large-Cycle Transaction Flood to DOS Tx-Pool Admission — (`tx-pool/src/component/verify_queue.rs`)

## Summary
`VerifyQueue.add_tx` enforces only a single global 256 MB byte-size cap with no per-peer quota. Any unprivileged P2P peer can flood the queue with large-cycle transactions, saturating it and causing all subsequent `submit_remote_tx`, `notify_tx`, and `process_orphan_tx` calls to return `Reject::Full`, blocking tx-pool admission for all peers and local RPC users for as long as the attacker sustains the flood.

## Finding Description
The `VerifyQueue` struct holds only a single `total_tx_size: usize` counter with no per-peer map: [1](#0-0) 

`add_tx` checks only the global counter against `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB) with no per-peer gate: [2](#0-1) [3](#0-2) 

Transactions with declared cycles above `large_cycle_threshold` are tagged `is_large_cycle = true`: [4](#0-3) 

Worker 0 is assigned `OnlySmallCycleTx` and skips all large-cycle entries via `peek`, meaning large-cycle transactions are only drained by the remaining `SubmitTimeFirst` workers through full script execution — a slow path: [5](#0-4) [6](#0-5) 

All three admission paths funnel through `enqueue_verify_queue` → `add_tx` and fail with `Reject::Full` when the queue is saturated: [7](#0-6) [8](#0-7) [9](#0-8) 

`remove_txs_by_peer` exists but is only invoked on peer ban, not on queue saturation: [10](#0-9) 

## Impact Explanation
When the queue is saturated, the target node's tx-pool admission is completely blocked: remote relay fails, local RPC `send_transaction` returns `PoolIsFull`, orphan transactions that become resolvable remain stranded, and miner proposal transactions cannot enter the queue. Applied across multiple nodes simultaneously (feasible with bounded cost), this constitutes sustained disruption of CKB transaction propagation. This matches the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The attack is reachable by any unprivileged P2P peer via the standard `RelayV3` protocol. Transactions pass `non_contextual_verify` (structural check only) before enqueuing; script validation happens inside the worker after dequeue. With `TRANSACTION_SIZE_LIMIT = 512 * 1_000` bytes, filling 256 MB requires approximately 512 transactions. A single attacker peer can consume the entire quota because `add_tx` has no per-peer gate. The attacker must hold enough CKB for valid UTXOs, but the cost is bounded and the disruption is sustained as long as replacement transactions are submitted faster than workers drain the queue.

## Recommendation
1. Add a `total_tx_size_by_peer: HashMap<PeerIndex, usize>` field to `VerifyQueue` and enforce a per-peer cap (e.g., `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE / MAX_PEERS`) inside `add_tx` before the global check.
2. When the queue approaches capacity, preferentially evict large-cycle entries from the highest-contributing peer using the existing `remove_txs_by_peer` infrastructure.
3. Apply a per-peer rate limit at the relay layer (before `submit_remote_tx`) analogous to the rate limiter already present in the hole-punching protocol.

## Proof of Concept
The existing unit test already confirms the `Reject::Full` outcome: [11](#0-10) 

Manual reproduction:
1. Attacker connects to target node as a P2P peer via RelayV3.
2. Attacker holds ~512 valid UTXOs and constructs 512 transactions, each ~512 KB serialized, with scripts consuming cycles just above `pool_config.max_tx_verify_cycles`.
3. Attacker relays all 512 transactions via `RelayTransactions` messages, declaring `cycles = max_tx_verify_cycles + 1` for each.
4. Each transaction passes `non_contextual_verify` and enters `VerifyQueue::add_tx`, tagged `is_large_cycle = true`. Worker 0 (`OnlySmallCycleTx`) skips them all. `SubmitTimeFirst` workers verify them slowly via full script execution.
5. `total_tx_size` approaches `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB).
6. All subsequent `submit_remote_tx`, `notify_tx`, and `process_orphan_tx` calls return `Reject::Full`.
7. Attacker continuously submits replacement transactions as old ones drain, maintaining the saturated state indefinitely.
