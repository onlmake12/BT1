### Title
`PythAggregatorV3` Exposes Unbounded Oracle Drift Window via Unconditional `getPriceUnsafe` in All Price-Reading Functions - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter contract, calls `getPriceUnsafe` in every single price-reading function — `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `decimals()` — with no staleness enforcement. This creates an unbounded oracle drift window: the on-chain price can lag the true market price by an arbitrary amount, and any downstream protocol that uses this adapter without independently checking `updatedAt` is exposed to the same oracle-drift arbitrage described in the reference report.

---

### Finding Description

Every price-reading function in `PythAggregatorV3` delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestAnswer — line 54
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
``` [1](#0-0) [2](#0-1) 

`getPriceUnsafe` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee: [3](#0-2) 

The `latestRoundData()` return value sets `updatedAt = price.publishTime`, which is the Pyth publish timestamp. Downstream Chainlink-compatible protocols that do not validate `updatedAt` against `block.timestamp` will silently consume a stale price. [4](#0-3) 

The Pyth documentation explicitly warns against using `getPriceUnsafe` in production DeFi: [5](#0-4) 

Yet the official Chainlink migration guide directs integrators to deploy `PythAggregatorV3` as a drop-in replacement for Chainlink feeds: [6](#0-5) 

---

### Impact Explanation

The analog to the Liquity oracle drift scenario is direct:

| Liquity | Pyth (`PythAggregatorV3`) |
|---|---|
| Deviation threshold > redemption fee → stale price window | No staleness check → unbounded stale price window |
| Arbitrageur exploits gap between oracle price and market price | Arbitrageur exploits gap between `getPriceUnsafe` result and true market price |
| Borrowers forced to sell collateral at under-market price | Users of downstream lending/derivatives/redemption protocols suffer mispriced liquidations or redemptions |

Any protocol that:
1. Uses `PythAggregatorV3` as a Chainlink-compatible feed
2. Does not independently validate `updatedAt` against a staleness threshold

is vulnerable to oracle-drift arbitrage. The attacker does not need to manipulate any price — they only need to wait for the on-chain price to drift from the market price (e.g., during a period of low update activity or network congestion), then interact with the downstream protocol at the stale price.

---

### Likelihood Explanation

- `PythAggregatorV3` is the official Pyth-provided migration path for Chainlink integrators, actively promoted in documentation.
- Chainlink-compatible protocols commonly do not check `updatedAt` staleness, as Chainlink's own push oracle historically kept prices fresh by design.
- The Pyth pull model means prices are only updated when someone calls `updateFeeds` or `updatePriceFeeds`. If no keeper is running, the on-chain price can become arbitrarily stale.
- The Scheduler (Pulse) contract's `deviationThresholdBps` creates a known, bounded drift window for push-style subscribers, but `PythAggregatorV3` has no such bound at all. [7](#0-6) 

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan` using a configurable `maxAge` parameter set at construction time:

```solidity
uint256 public maxAge; // e.g., 3600 seconds

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    ...
}
```

This mirrors the safe pattern already used in `AbstractPyth.sol`: [8](#0-7) 

---

### Proof of Concept

1. Deploy `PythAggregatorV3` wrapping a Pyth price feed (e.g., ETH/USD).
2. Deploy a downstream lending protocol that calls `latestRoundData()` for collateral valuation without checking `updatedAt`.
3. Allow the on-chain Pyth price to become stale (no `updateFeeds` call for N minutes).
4. True market price of ETH drops 3% while on-chain price remains at the old value.
5. Attacker calls the lending protocol to borrow against ETH collateral at the inflated stale price, receiving more than the collateral is worth at market.
6. Attacker sells the borrowed assets at market price, pocketing the 3% drift as profit.
7. The lending protocol is left with under-collateralized positions.

The root cause is entirely within `PythAggregatorV3.sol` lines 41, 54, 59, 89, and 110 — all of which call `getPriceUnsafe` with no staleness guard. [9](#0-8) [10](#0-9)

    function getPriceNoOlderThan(
        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
