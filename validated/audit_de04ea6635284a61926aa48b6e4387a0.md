### Title
Non-Deterministic Proposal Selection for Equal-Fee-Rate Transactions Under `max_block_proposals_limit` — (File: `tx-pool/src/component/sort_key.rs`)

### Summary
`AncestorsScoreSortKey::cmp` has no timestamp tie-breaker. When two pending transactions share the same effective fee rate and the same `ancestors_weight`, the comparison returns `Ordering::Equal`, leaving their relative position in the sorted pool index undefined. When the pool holds more pending transactions than `max_block_proposals_limit`, the selection of which transactions get proposed is non-deterministic for equal-scored entries. A transaction submitted earlier may be skipped in favour of a later one, mirroring the external report's pattern of a limited resource being distributed in an undefined order that disadvantages earlier participants.

### Finding Description

`AncestorsScoreSortKey` is the primary sort key used to rank pending transactions for both proposal selection and block-template assembly. [1](#0-0) 

The `cmp` implementation cross-multiplies to avoid division, then falls back to `ancestors_weight` as a secondary key. When both the effective fee rate **and** `ancestors_weight` are equal, the function returns `Ordering::Equal` with no further disambiguation: [2](#0-1) 

`PoolEntry` registers this key as an `ordered_non_unique` index in the `MultiIndexMap`: [3](#0-2) 

`score_sorted_iter_by_status` iterates over that index in reverse order. For entries whose keys compare `Equal`, the iteration order is implementation-defined (BTreeMap insertion order for equal keys is not guaranteed to be stable across mutations): [4](#0-3) 

`get_proposals` takes the first `limit` entries from that iterator and collects them into a `HashSet`, which itself has non-deterministic iteration order: [5](#0-4) 

`package_proposals` calls `get_proposals` with `max_block_proposals_limit` as the cap: [6](#0-5) 

The resulting `HashSet` is then converted to a `Vec` via `Vec::from_iter`, whose order is non-deterministic: [7](#0-6) 

The same non-determinism affects `package_txs` → `TxSelector::txs_to_commit`, which also iterates `sorted_proposed_iter()`: [8](#0-7) 

By contrast, the **eviction** path (`EvictKey`) does include a timestamp tie-breaker, explicitly preferring to evict the most recently submitted transaction last: [9](#0-8) 

The asymmetry is clear: eviction respects submission time; selection does not.

### Impact Explanation

When the pool contains more pending transactions than `max_block_proposals_limit` (consensus-enforced per block), the node must choose a subset to propose. For any group of transactions that share the same effective fee rate and `ancestors_weight`, the subset chosen is non-deterministic. A transaction submitted earlier — which should have priority under the same "first come, first served" principle already encoded in `EvictKey` — may be omitted from the proposal set in favour of a later transaction. Because CKB's two-phase commit requires a transaction to be proposed before it can be committed, repeated omission delays finality. If the pool remains saturated with equal-fee-rate transactions, an earlier submitter's transaction can be indefinitely deferred without any protocol-level recourse.

### Likelihood Explanation

Equal fee rates are common in practice: many wallets default to the minimum fee rate (`min_fee_rate`). During periods of moderate congestion — when the pending pool exceeds `max_block_proposals_limit` but is not so full that eviction fires — the non-deterministic selection path is exercised on every block template update. Any unprivileged tx-pool submitter can trigger this condition by submitting transactions at the prevailing minimum fee rate.

### Recommendation

Add a timestamp tie-breaker to `AncestorsScoreSortKey::cmp`, consistent with the existing `EvictKey` design. When both the effective fee rate and `ancestors_weight` are equal, prefer the entry with the lower (earlier) timestamp:

```rust
if self_weight == other_weight {
    if self.ancestors_weight == other.ancestors_weight {
        // tie-break by submission time: earlier first
        other.timestamp.cmp(&self.timestamp)  // requires adding timestamp field
    } else {
        self.ancestors_weight.cmp(&other.ancestors_weight)
    }
} else {
    self_weight.cmp(&other_weight)
}
```

Alternatively, include the `ProposalShortId` as a final deterministic tie-breaker to guarantee a total order without requiring a new field.

### Proof of Concept

1. Configure a node with a small `max_block_proposals_limit` (e.g., 10).
2. Submit 20 transactions, all with identical fee rates and sizes (so `AncestorsScoreSortKey` values are equal), recording submission timestamps.
3. Call `get_block_template` repeatedly.
4. Observe that the 10 proposals selected vary across calls — transactions submitted first are not consistently included.
5. Confirm that the `EvictKey` ordering (which does use timestamp) would have selected the earlier transactions, demonstrating the inconsistency. [1](#0-0) [5](#0-4) [6](#0-5) [10](#0-9)

impl Ord for AncestorsScoreSortKey {
    fn cmp(&self, other: &Self) -> Ordering {
        // avoid division a_fee/a_weight > b_fee/b_weight
        let (fee, weight) = self.min_fee_and_weight();
        let (other_fee, other_weight) = other.min_fee_and_weight();
        let self_weight = u128::from(fee.as_u64()) * u128::from(other_weight);
        let other_weight = u128::from(other_fee.as_u64()) * u128::from(weight);
        if self_weight == other_weight {
            // if fee rate weight is same, then compare with ancestor weight
            self.ancestors_weight.cmp(&other.ancestors_weight)
        } else {
            self_weight.cmp(&other_weight)
        }
    }
}
