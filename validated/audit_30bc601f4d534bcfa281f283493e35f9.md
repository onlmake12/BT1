Audit Report

## Title
IPv4-Mapped IPv6 Address Bypasses Peer Ban List - (`File: network/src/peer_store/ban_list.rs`)

## Summary
`BanList::is_ip_banned_until` does not normalize IPv4-mapped IPv6 addresses (e.g., `::ffff:1.2.3.4`) to their IPv4 equivalents before performing ban lookups. A peer banned under its IPv4 address can immediately reconnect using the IPv4-mapped IPv6 form of the same address and bypass the ban entirely. The inverse bypass also holds.

## Finding Description
**Ban storage path:** `ban_addr` in `peer_store_impl.rs` calls `ip_to_network(addr.ip())`, which preserves the address family verbatim:

```rust
// network/src/peer_store/types.rs L152-156
pub fn ip_to_network(ip: IpAddr) -> IpNetwork {
    match ip {
        IpAddr::V4(ipv4) => IpNetwork::V4(ipv4.into()),
        IpAddr::V6(ipv6) => IpNetwork::V6(ipv6.into()),
    }
}
```

A peer connecting as `/ip4/1.2.3.4/tcp/...` is stored as `IpNetwork::V4(1.2.3.4/32)`.

**Ban check path:** `is_ip_banned_until` in `ban_list.rs` (L48-58) performs two checks, both of which fail for cross-family comparisons:

1. **HashMap key lookup** (L49-54): `ip_to_network(::ffff:1.2.3.4)` produces `IpNetwork::V6(::ffff:1.2.3.4/128)`, which does not match the stored key `IpNetwork::V4(1.2.3.4/32)` — miss.
2. **Linear scan** (L56-58): `IpNetwork::V4(1.2.3.4/32).contains(IpAddr::V6(::ffff:1.2.3.4))` — the `ipnetwork` crate returns `false` for cross-family `contains` calls — miss.

Both checks fail, so `is_ip_banned_until` returns `false`, and `accept_peer` in `peer_registry.rs` (L109-111) accepts the connection.

**Contrast with `network_group.rs` (L31-35):** The `Group` implementation explicitly normalizes IPv4-mapped IPv6 to `Group::IP4` for peer diversity accounting, demonstrating developer awareness of the aliasing issue — but this normalization was never applied to the ban list. The `grep` search confirms `to_ipv4()` is called only in `network_group.rs` and nowhere in the ban list or `types.rs`.

## Impact Explanation
A banned peer can immediately reconnect by presenting the IPv4-mapped IPv6 form of its address (or vice versa), completely nullifying the ban. This allows a misbehaving peer to continuously reconnect after each ban, causing repeated processing overhead, resource consumption, and sync disruption on the targeted node. This maps to **Low (501–2000 points): Any other important performance/security improvement for CKB**, as the ban mechanism — the node's primary defense against actively misbehaving peers — is rendered ineffective against any peer that uses this trivial address-family switch.

## Likelihood Explanation
The attacker is an unprivileged peer with no special access. IPv4-mapped IPv6 is supported on all major operating systems and requires no new IP address, no cryptographic material, and no privileged access. A peer that has been banned has a direct incentive to reconnect. The bypass requires only changing the address family in the multiaddr used to connect. It is repeatable indefinitely.

## Recommendation
In `is_ip_banned_until` (`ban_list.rs`), normalize the incoming `IpAddr` before any lookup. Mirror the logic already present in `network_group.rs`:

```rust
fn normalize_ip(ip: IpAddr) -> IpAddr {
    if let IpAddr::V6(v6) = ip {
        if let Some(v4) = v6.to_ipv4() {
            return IpAddr::V4(v4);
        }
    }
    ip
}
```

Call `normalize_ip(ip)` at the top of `is_ip_banned_until` before the `ip_to_network` call. Optionally apply the same normalization inside `ip_to_network` or `ban_addr` to also normalize addresses at ban-storage time.

## Proof of Concept
1. Node A bans peer B (IPv4 `1.2.3.4`) for sending a malformed message. `ban_addr` stores `IpNetwork::V4(1.2.3.4/32)` in the ban list.
2. Peer B reconnects immediately using multiaddr `/ip6/::ffff:1.2.3.4/tcp/<port>/p2p/<peer_id>`.
3. `accept_peer` calls `peer_store.is_addr_banned(&remote_addr)`. `multiaddr_to_socketaddr` returns `SocketAddr::V6([::ffff:1.2.3.4]:port)`. `socket_addr.ip()` returns `IpAddr::V6(::ffff:1.2.3.4)`.
4. `is_ip_banned_until` calls `ip_to_network(IpAddr::V6(::ffff:1.2.3.4))` → `IpNetwork::V6(::ffff:1.2.3.4/128)`. HashMap `get` finds no entry (key family mismatch). The `iter().any(|net| net.contains(ip))` check evaluates `IpNetwork::V4(1.2.3.4/32).contains(IpAddr::V6(::ffff:1.2.3.4))` → `false`.
5. `is_addr_banned` returns `false`. Peer B is accepted despite being banned.

A minimal unit test can reproduce this by inserting a `BannedAddr` with `IpNetwork::V4(1.2.3.4/32)` into a `BanList` and asserting that `is_ip_banned(&IpAddr::V6("::ffff:1.2.3.4".parse().unwrap()))` returns `true` — it currently returns `false`.
