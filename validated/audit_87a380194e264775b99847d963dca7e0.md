### Title
Entropy `requestHelper` Lacks Caller-Side Maximum Fee Bound; Excess `msg.value` Is Never Refunded — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.requestHelper()` consumes the entire `msg.value` without refunding any excess and provides no `maxFee` parameter. Because a provider can update their fee at any time via `setProviderFee()`, a user who sends a buffer above the quoted fee (or whose transaction is front-run by a provider fee increase) will silently overpay, with the surplus permanently credited to `accruedPythFeesInWei` rather than returned to the caller.

---

### Finding Description

Every `requestV2` variant routes through `requestHelper`:

```solidity
// Entropy.sol L234-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is split between the provider and Pyth's treasury. There is no refund path. The interface documentation explicitly acknowledges this:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation… Further note that excess value is *not* refunded to the caller." [2](#0-1) 

A provider can change their fee at any time without restriction:

```solidity
// Entropy.sol L810-820
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
``` [3](#0-2) 

There is no `maxFee` or `maxProviderFee` parameter in any `requestV2` variant: [4](#0-3) 

The same pattern exists in `Echo.requestPriceUpdatesWithCallback`, where `req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei)` stores the entire excess above the Pyth flat fee as the provider's reward, with no refund: [5](#0-4) 

---

### Impact Explanation

A user who:
1. Calls `getFeeV2(provider, gasLimit)` off-chain to obtain the current fee, then
2. Submits `requestV2{value: quotedFee + buffer}()` with a safety buffer,

can have their buffer silently consumed if the provider calls `setProviderFee()` with a higher value between the user's quote and their transaction's execution. The provider receives the higher `providerFee`; the remainder goes to `accruedPythFeesInWei`. Neither is returned to the user. The user has no on-chain mechanism to enforce a maximum acceptable fee.

---

### Likelihood Explanation

Any registered Entropy provider (registration is permissionless) can call `setProviderFee()` at any time. Users who send a buffer to avoid `InsufficientFee` reverts are the most exposed. The Fortuna keeper itself is documented to adjust fees dynamically based on gas prices, meaning fee volatility is an expected and frequent condition, not a theoretical edge case. [6](#0-5) 

---

### Recommendation

Add an optional `maxFee` parameter to `requestV2` (and `requestPriceUpdatesWithCallback` in Echo). Inside `requestHelper`, revert if `requiredFee > maxFee`. Additionally, refund `msg.value - requiredFee` to `msg.sender` when `msg.value > requiredFee`, rather than silently crediting the surplus to `accruedPythFeesInWei`.

---

### Proof of Concept

```solidity
// 1. Provider registers with fee = 1 wei
vm.prank(provider);
entropy.register(1, commitment, meta, 100, uri);

// 2. User queries fee off-chain: quotedFee = 1 + pythFeeInWei
uint128 quotedFee = entropy.getFeeV2(provider, 0);

// 3. User prepares tx with a 1000-wei buffer for safety
uint128 sentValue = quotedFee + 1000;

// 4. Provider front-runs: raises fee to 500 wei
vm.prank(provider);
entropy.setProviderFee(500);

// 5. User's tx executes: requiredFee = 500 + pythFeeInWei <= sentValue → succeeds
vm.deal(user, sentValue);
vm.prank(user);
entropy.requestV2{value: sentValue}(provider, bytes32(0), 0);

// 6. Provider accrued 500 wei (not 1 wei as user expected)
// 7. User paid sentValue; no refund; no revert; no on-chain protection
assertEq(entropy.getProviderInfoV2(provider).accruedFeesInWei, 500);
``` [7](#0-6)

        bool useBlockhash,
        bool isRequestWithCallback,
        uint32 callbackGasLimit

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (_state.providers[provider].sequenceNumber == 0)
            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
        if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.OutOfRandomness();
        providerInfo.sequenceNumber += 1;

        // Check that fees were paid and increment the pyth / provider balances.

        if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();

        providerInfo.accruedFeesInWei += providerFee;
        _state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
