### Title
Unvalidated `providerToCredit` in `executeCallback` Enables Fee Theft via Pre-Registration Credit Preservation — (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

---

### Summary

In `Echo.sol`, the `executeCallback` function allows any caller to specify an arbitrary `providerToCredit` address once the exclusivity period expires. Because `registerProvider()` does not reset `accruedFeesInWei`, an attacker can credit fees to their own address before registering, then register and withdraw those fees — bypassing the intended fee-routing to the original provider.

---

### Finding Description

`executeCallback` enforces `providerToCredit == req.provider` only during the exclusivity window:

```solidity
// Echo.sol lines 113–121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

After that window, `providerToCredit` is completely unconstrained. Fees are credited unconditionally:

```solidity
// Echo.sol lines 161–162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

`registerProvider()` only sets `isRegistered = true` and fee parameters — it never zeroes `accruedFeesInWei`:

```solidity
// Echo.sol lines 381–393
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    ...
}
```

`withdrawAsFeeManager` has no registration check — it only requires `msg.sender == feeManager`:

```solidity
// Echo.sol lines 360–379
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    require(
        _state.providers[provider].accruedFeesInWei >= amount,
        "Insufficient balance"
    );
    _state.providers[provider].accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    ...
}
```

**Attack path:**

1. A user calls `requestPriceUpdatesWithCallback(legitimateProvider, ...)` paying the required fee. `req.fee = msg.value - pythFeeInWei` is stored.
2. After `exclusivityPeriodSeconds` elapses, the attacker fetches valid price update data from the public Hermes API.
3. Attacker calls `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)`. Because the exclusivity check is skipped, `_state.providers[attacker_address].accruedFeesInWei` is credited with `req.fee + msg.value - pythFee`.
4. Attacker calls `registerProvider(0, 0, 0)`. `accruedFeesInWei` is **not** reset.
5. Attacker calls `setFeeManager(attacker_address)`, setting themselves as their own fee manager.
6. Attacker calls `withdrawAsFeeManager(attacker_address, amount)` and receives the stolen funds.

The original provider receives nothing for fulfilling the request.

---

### Impact Explanation

The original provider loses all fees earned for a fulfilled request. The attacker — who performed no legitimate oracle work — receives those fees. The Pyth protocol's own `accruedFeesInWei` is unaffected, but provider-level fee accounting is broken. Any pending request whose exclusivity period has expired is exploitable. In a high-volume deployment, this can drain all provider fee balances.

**Impact: Medium** — provider funds are stolen; protocol-level Pyth fees are intact.

---

### Likelihood Explanation

**Likelihood: High.** Registration is permissionless. Valid price update data is publicly available from Hermes. The attacker only needs to wait for the exclusivity period (a configurable number of seconds) to expire. No privileged access, leaked key, or governance majority is required. The attack is repeatable for every unfulfilled request.

---

### Recommendation

1. **Validate `providerToCredit` unconditionally** — restrict it to `req.provider` regardless of the exclusivity period, or at minimum to a registered provider address.
2. **Reset `accruedFeesInWei` on registration** — `registerProvider()` should zero out any pre-existing `accruedFeesInWei` for the registering address to prevent pre-registration credit accumulation.
3. **Add a registration check to `withdrawAsFeeManager`** — mirror the `sequenceNumber != 0` guard present in `Entropy.sol`'s `withdrawAsFeeManager`.

---
