### Title
`script_exists_in_output` Re-Checks `row_lock` Instead of `row_type`, Silently Skipping the Type-Script Existence Check on PostgreSQL During Block Rollback — (`File: util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

In `util/rich-indexer/src/indexer/remove.rs`, the function `script_exists_in_output` is supposed to determine whether a given script ID is still referenced by any remaining output row — either as a `lock_script_id` **or** as a `type_script_id`. Due to a copy-paste error at line 252, the final `match` block re-reads `row_lock` (the lock-script query result) instead of `row_type` (the type-script query result). On PostgreSQL, this causes the type-script existence check to be silently skipped, making the function always return the lock-script result for both checks. Scripts that are exclusively used as type scripts are therefore incorrectly reported as absent, and are deleted from the `script` table during block rollback, permanently corrupting the rich indexer's relational database.

---

### Finding Description

`script_exists_in_output` executes two SQL `EXISTS` queries and is supposed to return `true` if the script is found in either column:

```rust
// Step 1 — query lock_script_id
let row_lock = sqlx::query("SELECT EXISTS (SELECT 1 FROM output WHERE lock_script_id = $1)")
    .bind(script_id).fetch_one(tx.as_mut()).await...;

// pg type is BOOLEAN
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => { if r { return Ok(true); } }   // early-return if found as lock
    Err(_) => { if row_lock.get::<i64, _>(0) == 1 { return Ok(true); } }
}

// Step 2 — query type_script_id
let row_type = sqlx::query("SELECT EXISTS (SELECT 1 FROM output WHERE type_script_id = $1)")
    .bind(script_id).fetch_one(tx.as_mut()).await...;

// BUG: row_lock is used again instead of row_type
match row_lock.try_get::<bool, _>(0) {   // ← should be row_type
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [1](#0-0) 

On **PostgreSQL**, `try_get::<bool, _>(0)` succeeds (the column is `BOOLEAN`), so the `Ok(r)` branch is always taken. The second `match` therefore returns the lock-script query result a second time; `row_type` is fetched but its value is never read. On **SQLite**, `try_get::<bool, _>` fails (the column is `BIGINT`), the `Err(_)` branch is taken, and `row_type.get::<i64, _>(0)` is correctly used — so SQLite is unaffected.

The correct fix mirrors the external report's recommended mitigation:

```rust
// correct
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

### Impact Explanation

`script_exists_in_output` is called inside `rollback_block` to decide which script rows are safe to delete:

```rust
for (_, lock_script_id, type_script_id) in output_lock_type_list {
    if !script_exists_in_output(lock_script_id, tx).await? {
        script_id_list_to_remove.push(lock_script_id);
    }
    if let Some(type_script_id) = type_script_id
        && !script_exists_in_output(type_script_id, tx).await?
    {
        script_id_list_to_remove.push(type_script_id);
    }
}
remove_batch_by_blobs("script", "id", &script_id_list_to_remove, tx).await?;
``` [2](#0-1) 

On PostgreSQL, any script that is referenced **only** as a `type_script_id` (and not as a `lock_script_id`) in the remaining outputs will have `script_exists_in_output` return `false`. That script ID is pushed into `script_id_list_to_remove` and deleted from the `script` table, even though it is still actively referenced by live outputs. This produces a dangling foreign-key situation and corrupts the rich indexer's relational state. Subsequent RPC calls that search by type script (`get_cells`, `get_transactions`, `get_cells_capacity`) will return empty or incorrect results for those scripts, and the corruption persists until a full re-index.

---

### Likelihood Explanation

The trigger is a **chain reorganization**, which is a normal, expected event in CKB. Any unprivileged block-relayer peer can submit a valid competing chain with more accumulated work, causing the node to roll back its current tip and call `rollback_block` via `IndexerSync::rollback`. The sync loop in `util/indexer-sync/src/lib.rs` calls `indexer.rollback()` whenever the parent hash of the next block does not match the indexer tip:

```rust
} else {
    indexer.rollback().expect("rollback block should be OK");
``` [3](#0-2) 

The bug is triggered on every rollback that involves outputs carrying a type script, which is extremely common (DAO deposits, UDT cells, NFTs, etc.). Any operator running the rich indexer with a PostgreSQL backend is affected by every reorg, however shallow.

### Recommendation

Replace `row_lock` with `row_type` in the final `match` block of `script_exists_in_output`:

```rust
// util/rich-indexer/src/indexer/remove.rs  line 252
// Before (buggy):
match row_lock.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}

// After (correct):
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Add a PostgreSQL-backed integration test for `rollback_block` that verifies scripts used exclusively as type scripts are **not** deleted when other outputs still reference them.

### Proof of Concept

**Setup (PostgreSQL backend):**

1. Append a block containing a transaction with one output that has:
   - `lock_script_id = 1` (script A)
   - `type_script_id = 2` (script B, used **only** as a type script — no output uses it as a lock)

2. Append a second block containing a transaction that spends the output from step 1 and creates a new output with:
   - `lock_script_id = 3` (script C)
   - `type_script_id = 2` (script B still referenced as type script)

3. Trigger a rollback of block 2 (simulating a reorg).

**Expected:** `script_exists_in_output(2, tx)` returns `true` because script B is still referenced as `type_script_id` in the block-1 output. Script B is **not** deleted.

**Actual (PostgreSQL):** `script_exists_in_output(2, tx)` executes the lock-script query (finds nothing → `false`), then executes the type-script query into `row_type` (finds a row → would be `true`), but then re-evaluates `row_lock.try_get::<bool, _>(0)` which returns `Ok(false)`. The function returns `false`. Script B is added to `script_id_list_to_remove` and deleted.

**Result:** The `script` table no longer contains script B. Any subsequent `get_cells` or `get_transactions` RPC call filtering by script B returns empty results, even though live outputs referencing it still exist in the `output` table. [4](#0-3) [5](#0-4)
