### Title
Stale Pre-Computed Accumulator Overwrites In-Place-Updated `total_tx_size`/`total_tx_cycles` After Ancestor Eviction — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's `total_tx_size` and `total_tx_cycles` accumulators are computed **before** ancestor-eviction side-effects mutate them in place, then the stale pre-eviction snapshot is written back unconditionally. Every time the ancestor-eviction path fires, the evicted entries' sizes and cycles are permanently added to the running totals instead of being subtracted, causing `total_tx_size` and `total_tx_cycles` to diverge upward from the actual pool contents without bound.

---

### Finding Description

`PoolMap::add_entry` executes the following sequence:

```
// Step 1 – snapshot totals BEFORE eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
//   = (self.total_tx_size + entry.size,
//      self.total_tx_cycles + entry.cycles)

// Step 2 – may call remove_entry_and_descendants → remove_entry
//           → update_stat_for_remove_tx, which DECREMENTS
//           self.total_tx_size / self.total_tx_cycles in place
evicts = self.check_and_record_ancestors(&mut entry)?;

// Step 3 – OVERWRITES the correctly-decremented live values
//           with the stale pre-eviction snapshot
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` before any eviction occurs. [2](#0-1) 

`check_and_record_ancestors` calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in place for every evicted entry. [3](#0-2) [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, `self.total_tx_size` already reflects the correct post-eviction value. Lines 218–219 then unconditionally overwrite it with the stale snapshot, re-adding the evicted entries' sizes and cycles. The net effect per eviction event is:

```
actual_total  = old_total − Σ(evicted.size) + entry.size
stored_total  = old_total                   + entry.size   ← stale
error         = +Σ(evicted.size)  (grows with every eviction)
```

The error accumulates permanently across repeated submissions.

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee-rate entry
}
``` [6](#0-5) 

An inflated `total_tx_size` causes `limit_size` to over-evict legitimate, high-fee-rate transactions that should remain in the pool. The same inflated values are surfaced verbatim through the `get_tx_pool_info` RPC: [7](#0-6) 

Consequences:
- **Legitimate transactions are silently evicted** from the mempool even when the pool is well below its actual byte limit.
- **Miners assemble smaller, less profitable blocks** because high-fee transactions are incorrectly expelled.
- **`get_tx_pool_info` reports false `total_tx_size`/`total_tx_cycles`**, misleading operators and fee-estimation clients.
- The divergence is **permanent** (until node restart or `clear_tx_pool`) and grows monotonically with each triggered eviction.

---

### Likelihood Explanation

The eviction branch inside `check_and_record_ancestors` fires when:

1. The incoming transaction's ancestor count exceeds `max_ancestors_count`, **and**
2. Removing `cell_ref_parents` (in-pool transactions that use an output of the new tx as a cell dep) would bring the count back under the limit. [8](#0-7) 

An unprivileged transaction sender can deliberately construct this scenario:

- Submit a chain of transactions up to `max_ancestors_count − 1` depth.
- Submit a transaction whose output is used as a cell dep by one of those in-pool transactions.
- Submit a new transaction spending that output; its ancestor count now exceeds the limit, triggering eviction of the cell-ref-parent.

This is reachable via the standard `send_transaction` JSON-RPC endpoint with no special privileges. The scenario can be repeated to inflate the counters without bound.

---

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` returns (i.e., after all in-place decrements have been applied), rather than before:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, Default::default()));
    }
    // Validate capacity headroom before mutating state
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    let evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Apply the addition AFTER evictions have already decremented the counters
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);

    Ok((true, evicts))
}
```

Alternatively, call `recompute_total_stat()` unconditionally at the end of `add_entry` whenever evictions occurred, mirroring the underflow-recovery logic already present in `update_stat_for_remove_tx`.

---

### Proof of Concept

**Setup:**

1. Configure a node with `max_ancestors_count = 25`.
2. Submit 24 transactions in a chain: `tx_root → tx1 → tx2 → … → tx23`.
3. Submit `tx_dep`, which uses `tx_root`'s output as a **cell dep** (not an input). Now `tx_dep` is a `cell_ref_parent` of `tx_root`.
4. Submit `tx_new` that **spends** `tx_root`'s output. Its ancestor count = 25 (tx_root + 23 descendants + itself), which equals `max_ancestors_count`. The `cell_ref_parents` set = {`tx_dep`}.

**Trigger:**

`check_and_record_ancestors` detects `ancestors_count (25) > max_ancestors_count (25)` is false here; adjust to `max_ancestors_count = 24` so the count is 25 > 24. It then evicts `tx_dep` (size = S_dep, cycles = C_dep) via `remove_entry_and_descendants`, which decrements `self.total_tx_size -= S_dep`. Lines 218–219 then overwrite `self.total_tx_size` with the pre-eviction snapshot, re-adding `S_dep`.

**Verification:**

```
before submission:  total_tx_size = T
after submission:   expected = T - S_dep + S_new
                    actual   = T          + S_new   ← inflated by S_dep
```

Call `get_tx_pool_info` to observe `total_tx_size` exceeds the sum of all entries' actual sizes. Repeat to grow the divergence unboundedly, eventually causing `limit_size` to evict all pool contents even when the pool is empty.

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

            ))
        })?;

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
