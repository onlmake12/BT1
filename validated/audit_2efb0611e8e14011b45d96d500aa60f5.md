Audit Report

## Title
Remote Peer Bypasses `max_tx_verify_cycles` via `declared_cycles` in `_process_tx` — (`tx-pool/src/process.rs`)

## Summary
`_process_tx` sets `max_cycles` directly from the peer-supplied `declared_cycles` value without ever consulting `TxPoolConfig.max_tx_verify_cycles`. Any connected P2P peer can force the node to execute up to `consensus.max_block_cycles()` cycles of CKB-VM script verification per transaction, completely bypassing the operator-configured limit. By streaming such transactions, an attacker can saturate verify workers and cause sustained CPU exhaustion and network congestion.

## Finding Description

**Root cause — `_process_tx` ignores `max_tx_verify_cycles`:**

At `tx-pool/src/process.rs:720`, `max_cycles` is set from the peer-supplied value:

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
```

`self.tx_pool_config.max_tx_verify_cycles` is never consulted. The value flows directly into `verify_rtx` (`tx-pool/src/util.rs:108`), which calls `verify_with_pause(max_tx_verify_cycles, command_rx)` — running the CKB-VM for up to `declared_cycles` cycles.

**The only P2P guard is insufficient:**

In `sync/src/relayer/transactions_process.rs:63-74`, the only check on `declared_cycles` is whether it exceeds `max_block_cycles` (banning the peer if so). Any value in the range `(max_tx_verify_cycles, max_block_cycles]` passes unchecked and is forwarded to `submit_remote_tx` → `_process_tx`.

**`max_tx_verify_cycles` is only used in the reorg path:**

`readd_detached_tx` at `tx-pool/src/process.rs:884` correctly caps verification at `max_tx_verify_cycles`. The remote relay path (`_process_tx`) does not.

**`VerifyQueue` uses `large_cycle_threshold` only for classification:**

At `tx-pool/src/component/verify_queue.rs:212-214`, `large_cycle_threshold` (derived from `max_tx_verify_cycles`) is used only to route transactions into "large" vs "small" worker buckets — not to reject or cap them.

**`DeclaredWrongCycles` check does not protect against this:**

The mismatch check at `tx-pool/src/process.rs:736-749` only fires if `declared != verified.cycles`. An attacker who crafts a transaction whose scripts consume exactly `N` cycles (where `N > max_tx_verify_cycles` but `N ≤ max_block_cycles`) will pass this check, and the transaction will be accepted into the pool after consuming the full `N` cycles.

**Call chain:**

`TransactionsProcess::execute()` → `submit_remote_tx(tx, declared_cycles, peer)` → `enqueue_verify_queue` → `VerifyMgr::Worker::process_inner` (`verify_mgr.rs:147-154`) → `_process_tx(entry.tx, entry.remote.map(|e| e.0), ...)`.

## Impact Explanation

This matches the **High** impact category: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

An attacker with a standard P2P connection can force each verify worker to spend up to `max_block_cycles` (~3.5 billion on mainnet) of CKB-VM execution per transaction, versus the operator-configured `max_tx_verify_cycles` (e.g., 70,000,000 — a 50× amplification). With multiple verify workers and a stream of such transactions, the node's CPU is saturated, block propagation degrades, and mining rewards may be missed. The cost to the attacker is negligible.

## Likelihood Explanation

The attack requires only a standard P2P peer connection and the normal relay flow: announce a tx hash via `RelayTransactionHashes`, wait for the node to send `GetRelayTransactions`, then respond with a high-cycle transaction. No special privileges, leaked keys, or victim mistakes are required. The existing integration test `SendLargeCyclesTxToRelay` (`test/src/specs/tx_pool/send_large_cycles_tx.rs:96-141`) sets `max_tx_verify_cycles = 5000` and explicitly asserts that a node accepts a large-cycles tx relayed from a peer, confirming the bypass is reachable in production. The attack is repeatable and can be sustained indefinitely.

## Recommendation

In `_process_tx`, cap `max_cycles` at `min(declared_cycles, self.tx_pool_config.max_tx_verify_cycles)` for remote transactions. If `declared_cycles > max_tx_verify_cycles`, reject the transaction immediately before verification begins (returning `Reject::ExceededMaximumCycles` or similar), ensuring the operator-configured limit is always the hard ceiling on per-transaction verification work. The check in `TransactionsProcess::execute()` should also be extended to ban peers that declare cycles exceeding `max_tx_verify_cycles`, not just `max_block_cycles`.

## Proof of Concept

**Preconditions:** Node configured with `max_tx_verify_cycles = 70_000_000`; `consensus.max_block_cycles() = 3_500_000_000`.

1. Connect as a peer to the target node.
2. Announce a tx hash via `RelayTransactionHashes`.
3. Wait for the node to send `GetRelayTransactions` for that hash.
4. Craft a transaction whose scripts consume exactly `N = 500_000_000` cycles (`N > max_tx_verify_cycles`, `N ≤ max_block_cycles`).
5. Send `RelayTransactions` with `declared_cycles = 500_000_000`.
6. The node passes the `declared_cycles > max_block_cycles` guard (`transactions_process.rs:66`), enqueues the tx, and `_process_tx` sets `max_cycles = 500_000_000`.
7. `verify_rtx` runs 500M cycles. Since `declared == verified.cycles`, `DeclaredWrongCycles` is not triggered.
8. The tx is accepted into the pool after consuming 500M cycles — 7× the operator's configured limit.
9. Repeat with concurrent transactions to exhaust all verify workers.

**Direct confirmation:** The existing `SendLargeCyclesTxToRelay` integration test (`test/src/specs/tx_pool/send_large_cycles_tx.rs:135-141`) sets `max_tx_verify_cycles = 5000` and asserts `Node0 should accept tx` for a large-cycles relayed transaction, proving the bypass is exercised and passes in the current codebase.
