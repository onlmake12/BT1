### Title
Unconditional `broadcast_exit_signals()` on Protocol Handler Panic Enables Remote Node Crash - (`network/src/network.rs`)

---

### Summary

When tentacle emits `ServiceError::ProtocolHandleError` (triggered by a panic in any `CKBProtocolHandler` implementation), the `handle_error` function in `network/src/network.rs` unconditionally calls `broadcast_exit_signals()`, which cancels the global `CancellationToken` and sends shutdown signals to every CKB service thread — crashing the entire node process. A single remote peer that can induce a panic in any protocol handler's `received()` path can exploit this to crash any reachable CKB node.

---

### Finding Description

In `network/src/network.rs`, the `ServiceError::ProtocolHandleError` arm of `handle_error` reads: [1](#0-0) 

```rust
ServiceError::ProtocolHandleError { proto_id, error } => {
    debug!("ProtocolHandleError: {:?}, proto_id: {}", error, proto_id);

    let ProtocolHandleErrorKind::AbnormallyClosed(opt_session_id) = error;  // line 691
    {
        if let Some(id) = opt_session_id {
            self.network_state.ban_session(...);  // optional ban
        }
        // ...sentry capture...
        broadcast_exit_signals();  // line 717 — UNCONDITIONAL
    }
}
```

Line 691 uses an **irrefutable `let` binding** — not `if let` — meaning `AbnormallyClosed` is the only variant of `ProtocolHandleErrorKind`. This compiles only because the enum has a single variant, so the destructure always succeeds. Execution then falls through unconditionally to `broadcast_exit_signals()` on line 717, regardless of whether the panic was isolated to a single session.

`broadcast_exit_signals()` in `util/stop-handler/src/stop_register.rs` performs a **global node shutdown**: [2](#0-1) 

It sets `RECEIVED_STOP_SIGNAL = true`, cancels the global `TOKIO_EXIT` `CancellationToken`, and drains all `CROSSBEAM_EXIT_SENDERS`. Every CKB service thread listening on these signals will terminate.

---

### Impact Explanation

Any panic in any `CKBProtocolHandler::received()` implementation — regardless of which session caused it — triggers a full node shutdown. The ban at line 693–700 is conditional on `opt_session_id` being `Some`, but `broadcast_exit_signals()` is not gated on anything. The invariant that a single-session protocol panic must be isolated is violated by design. [3](#0-2) 

---

### Likelihood Explanation

Protocol handlers (`identify`, `ping`, `discovery`, `disconnect_message`, `hole_punching`) contain `unwrap()` and `expect()` calls on peer-supplied data paths. [4](#0-3) 

An attacker who connects as an inbound peer and sends bytes that trigger any such `unwrap()` panic will cause tentacle to emit `ServiceError::ProtocolHandleError { error: AbnormallyClosed(Some(session_id)) }`, which routes directly to `broadcast_exit_signals()`. No PoW, no key, no privileged access is required — only a TCP connection to the P2P port.

---

### Recommendation

Replace the unconditional `broadcast_exit_signals()` call with session-scoped error handling:

1. Remove `broadcast_exit_signals()` from the `ProtocolHandleError` arm entirely.
2. Keep (and strengthen) the `ban_session` call for the offending peer.
3. Log the panic at `error!` level and emit a Sentry event, but do not propagate to a node-wide shutdown.
4. If a global shutdown is desired for *global* (non-session) handler panics (i.e., `opt_session_id == None`), gate it explicitly: only call `broadcast_exit_signals()` when `opt_session_id.is_none()`.

---

### Proof of Concept

1. Implement a mock `CKBProtocolHandler` whose `received()` panics on a specific byte pattern (e.g., `0xDE 0xAD`).
2. Register it as a protocol substream handler in a test node.
3. Connect a peer, open the protocol substream, send `[0xDE, 0xAD]`.
4. Observe tentacle catching the panic and emitting `ServiceError::ProtocolHandleError { error: AbnormallyClosed(Some(session_id)) }`.
5. Assert `broadcast_exit_signals()` fires and all service threads receive the cancellation signal.
6. Confirm the node process exits — triggered by a single malformed message from an unprivileged peer.

The root cause is entirely in production code at: [5](#0-4) [2](#0-1)

```rust
pub fn broadcast_exit_signals() {
    debug!("Received exit signal; broadcasting exit signal to all threads");
    RECEIVED_STOP_SIGNAL.store(true, std::sync::atomic::Ordering::SeqCst);
    TOKIO_EXIT.cancel();
    CROSSBEAM_EXIT_SENDERS
        .lock()
        .iter()
        .for_each(|tx| match tx.try_send(()) {
            Ok(_) => {}
            Err(TrySendError::Full(_)) => info!("Ckb process has received exit signal"),
            Err(TrySendError::Disconnected(_)) => {
                debug!("broadcast thread: channel is disconnected")
            }
        });
}
```
