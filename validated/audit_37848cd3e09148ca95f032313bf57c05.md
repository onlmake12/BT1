### Title
Zero-Price Feeds Silently Skip Deviation Check, Permanently Blocking `updateOnDeviation`-Only Subscriptions - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler._validateShouldUpdatePrices`, a division-by-zero guard unconditionally `continue`s the deviation loop whenever either the stored previous price or the incoming current price is zero. If **all** price feeds in a subscription are skipped this way, the loop exits without ever returning a valid timestamp, and the function falls through to `revert SchedulerErrors.UpdateConditionsNotMet()`. This permanently blocks any keeper from pushing price updates to subscriptions that use `updateOnDeviation` as their sole update criterion, whenever the relevant feeds carry a zero price.

---

### Finding Description

`_validateShouldUpdatePrices` evaluates whether a price update satisfies the subscription's trigger criteria. For deviation-based subscriptions it iterates over every feed and computes:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;          // ← silently skips the feed entirely
}

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);

if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
    return updateTimestamp;   // ← only valid exit for deviation path
}
```

After the loop, the function unconditionally reverts:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
```

Two realistic zero-price scenarios cause every feed to be skipped:

1. **Stored previous price is 0.** A Pyth `int64` price field can be 0. If a feed was stored with `price == 0` (e.g., a market that opened at zero, a misconfigured feed, or a feed whose first stored update carried price 0), every subsequent keeper call will `continue` for that feed and, if it is the only feed, always revert.

2. **Incoming current price is 0.** If a token collapses to zero (the most extreme deviation imaginable), the incoming `currentPrice` is 0, the guard fires, the deviation is never computed, and the update is rejected — precisely when it is most critical.

In both cases the subscription is stuck: the heartbeat path is absent (subscription uses deviation-only), the deviation path is silently bypassed, and the only reachable code path is the terminal revert.

This is structurally identical to M-8: a guard intended to prevent a divide-by-zero produces a zero tolerance threshold (`0 / 10000 == 0`) that can never be satisfied, so every legitimate operation reverts.

---

### Impact Explanation

- Any active subscription configured with `updateOnDeviation: true` and `updateOnHeartbeat: false` becomes permanently frozen if all its feeds hit the zero-price condition.
- Keepers calling `updatePriceFeeds` always revert with `UpdateConditionsNotMet`, wasting gas and leaving price data stale indefinitely.
- Consumers reading prices via `getPricesNoOlderThan` will also start reverting once the staleness window expires, making the subscription's data completely unavailable.
- The subscription balance continues to be locked (active subscriptions cannot withdraw below minimum balance), so the subscriber also loses access to their funds until they deactivate.

---

### Likelihood Explanation

- Pyth price feeds use `int64` for price; a value of 0 is within the valid type range and is not rejected anywhere in the parsing or storage path.
- A token that crashes to zero (e.g., a de-pegged stablecoin, an exploit draining a pool) will produce exactly this condition on the incoming `currentPrice`.
- A subscription with a single feed whose stored price is 0 (first update stored a zero price) is permanently bricked with no recovery path short of deactivating and recreating the subscription.
- The condition is reachable by any unprivileged keeper submitting a valid Pyth price update — no special access is required.

---

### Recommendation

Replace the unconditional `continue` with logic that treats a transition involving zero as an infinite deviation (always trigger):

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A zero price represents an extreme/undefined deviation;
    // treat it as exceeding any finite threshold and trigger the update.
    return updateTimestamp;
}
```

Alternatively, if the intent is to skip zero-priced feeds without blocking the subscription, track whether at least one non-zero feed was evaluated and only revert if a non-zero feed failed to meet the threshold.

---

### Proof of Concept

1. Create a subscription with `updateOnDeviation: true`, `updateOnHeartbeat: false`, one price feed, `deviationThresholdBps = 100`.
2. Push a first update where `priceFeeds[0].price.price == 0`. This passes (first update, `previousFeed.id == bytes32(0)` triggers early return). The zero price is stored.
3. Push a second update where `priceFeeds[0].price.price == 1e8` (a massive deviation from 0).
4. `_validateShouldUpdatePrices` reads `previousPrice = 0`, hits `continue`, exits the loop, and reverts with `UpdateConditionsNotMet`.
5. The subscription can never be updated again via the deviation path regardless of how large the price move is. [1](#0-0) [2](#0-1) [3](#0-2)
