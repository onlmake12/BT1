Audit Report

## Title
Unauthenticated Sender in `ConnectionRequestDeliveredProcess::execute()` Enables Peer-Store Poisoning via Forced Identify Session — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
`ConnectionRequestDeliveredProcess::execute()` does not verify that the peer sending a `ConnectionRequestDelivered` message is the peer identified by `content.to`. An attacker who is a connected peer can seed `inflight_requests` with an attacker-controlled `PeerId` via the Discovery protocol, then send a crafted `ConnectionRequestDelivered` message to force the victim to open an outbound TCP connection to an attacker-controlled endpoint and run the Identify protocol against it, allowing the attacker to inject arbitrary addresses into the victim's peer store.

## Finding Description

**Step 1 — Seeding `inflight_requests` via Discovery.**

`add_new_addrs` in `network/src/protocols/discovery/mod.rs` (L347–362) inserts attacker-supplied addresses and flags directly into the peer store via `peer_store.add_addr` with no sender authentication. [1](#0-0) 

`fetch_nat_addrs` in `network/src/peer_store/peer_store_impl.rs` (L243–266) selects addresses where `last_connected_at_ms == 0` and the PeerId is not in `connected_peers`. A freshly advertised attacker address satisfies both conditions. [2](#0-1) 

`notify()` in `network/src/protocols/hole_punching/mod.rs` (L179–242) calls `fetch_nat_addrs` and inserts the returned PeerIds into `inflight_requests`, so `FAKE_PEER_ID` enters the map on the next 5-minute tick. [3](#0-2) 

**Step 2 — `listen_addrs` validation does not bind to the actual sender.**

The only check on `listen_addrs` in `connection_request_delivered.rs` (L51–71) is that any embedded PeerId must equal `content.to`. Since `content.to` is `FAKE_PEER_ID` (attacker-generated), the attacker freely embeds their actual listener IP:port alongside `FAKE_PEER_ID`. [4](#0-3) 

**Step 3 — `execute()` reaches the NAT traversal branch without authenticating the sender.**

With `route` empty and `content.from` set to the victim's own public PeerId (obtained via Identify), the condition `self_peer_id != &content.from` is false. `inflight_requests.remove(&content.to)` succeeds for `FAKE_PEER_ID`, and `try_nat_traversal` is called. There is no check that the session delivering this message belongs to the peer identified by `content.to`. [5](#0-4) 

**Step 4 — `try_nat_traversal` connects to the attacker's listener.**

`try_nat_traversal` (L237–285) spawns a task that attempts TCP connections to the addresses in `listen_addrs`. On success, `raw_session` is called with `RawSessionInfo::outbound(Identify)`, opening a full Identify session to the attacker's endpoint. [6](#0-5) 

When `bind_addr` is `None` (always on non-Linux; on Linux unless `reuse_port_on_linux` is set), `create_socket` still creates a functional unbound socket that connects to the attacker's endpoint. [7](#0-6) 

**Step 5 — Identify response poisons the peer store.**

`add_remote_listen_addrs` in `network/src/protocols/identify/mod.rs` (L488–494) unconditionally calls `peer_store.add_addr` for every address in the attacker's Identify response. [8](#0-7) 

The only guard in `received_identify` (L389–397) is a network-name check (`verify`), which the attacker passes by using the correct public CKB network name. [9](#0-8) 

## Impact Explanation

The victim node establishes an outbound TCP connection to an attacker-controlled endpoint and runs the Identify protocol. The attacker controls the full Identify response, injecting arbitrary addresses into the victim's peer store. Repeated execution (re-seeding `inflight_requests` on each 5-minute `notify()` tick) allows progressive peer-store poisoning, enabling an eclipse attack that isolates the victim from the honest network. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as a successfully eclipsed node suppresses block and transaction propagation and degrades network health.

## Likelihood Explanation

The attack requires only a standard P2P connection and the ability to send Discovery and HolePunching protocol messages, both available to any connected peer. The victim's PeerId is public via Identify. The `inflight_requests` window is 5 minutes, giving ample time to deliver the crafted message after each `notify()` tick. No special privileges, leaked keys, or majority hashpower are required. The attack works on all platforms because `bind_addr = None` still allows outbound connections.

## Recommendation

1. **Authenticate the sender**: Verify that a `ConnectionRequestDelivered` message arrives from the session whose PeerId matches `content.to`. Bind each `inflight_requests` entry to the specific session the `ConnectionRequest` was sent toward, and reject `ConnectionRequestDelivered` messages arriving from any other session.
2. **Scope `inflight_requests` entries**: Store `(PeerId, expected_session_id)` pairs instead of bare `PeerId` timestamps, so a crafted message from a different session cannot consume the entry.
3. **Cross-check `listen_addrs`**: Validate that the IP:port in `listen_addrs` matches the observed remote address of the connected peer that sent the message, rather than accepting arbitrary values.

## Proof of Concept

1. Attacker node `A` connects to victim `V`.
2. `A` sends a Discovery message advertising `/ip4/<A_public_ip>/tcp/<A_port>/p2p/<FAKE_PEER_ID>` with flags `SYNC|DISCOVERY|RELAY`. `FAKE_PEER_ID` is a freshly generated key pair controlled by `A`.
3. Wait for `V`'s `notify()` to fire (≤5 minutes). `FAKE_PEER_ID` is now in `V.inflight_requests`.
4. `A` opens a TCP listener on `<A_public_ip>:<A_port>`.
5. `A` sends `HolePunchingMessage::ConnectionRequestDelivered` to `V` with:
   - `from` = `V`'s PeerId (obtained from Identify)
   - `to` = `FAKE_PEER_ID`
   - `route` = `[]`, `sync_route` = `[]`
   - `listen_addrs` = `[/ip4/<A_public_ip>/tcp/<A_port>/p2p/<FAKE_PEER_ID>]`
6. `V` processes the message: `route` is empty, `content.from == self_peer_id`, `inflight_requests.remove(FAKE_PEER_ID)` succeeds → `try_nat_traversal` is called.
7. `V` connects to `<A_public_ip>:<A_port>`; `A`'s listener accepts.
8. `V` calls `raw_session(stream, addr, RawSessionInfo::outbound(Identify))`.
9. `A` sends a crafted Identify response with the correct CKB network name and attacker-chosen `listen_addrs`.
10. **Assert**: `V`'s peer store now contains attacker-chosen addresses. Repeat from step 3 on each subsequent `notify()` tick to progressively fill the peer store.

    pub fn fetch_nat_addrs(&mut self, count: usize, required_flags: Flags) -> Vec<AddrInfo> {
        // Get info:
        // 1. Never connected
        // 2. Not already connected
        // 3. Ip4 / Ip6 address only

        let peers = &self.connected_peers;

        let filter = |peer_addr: &AddrInfo| {
            required_flags_filter(required_flags, Flags::from_bits_truncate(peer_addr.flags))
                && extract_peer_id(&peer_addr.addr)
                    .map(|peer_id| !peers.contains_key(&peer_id))
                    .unwrap_or_default()
                && peer_addr.addr.iter().any(|p| {
                    matches!(
                        p,
                        p2p::multiaddr::Protocol::Ip4(_) | p2p::multiaddr::Protocol::Ip6(_)
                    )
                })
                && peer_addr.last_connected_at_ms == 0
        };

        self.addr_manager.fetch_random(count, filter)
    }
