### Title
Post-Insertion Limit Check in `add_ask_for_txs` Allows Unprivileged Peer to Exhaust `unknown_tx_hashes` Queue and Trigger O(n) Mutex-Held Scan - (File: `sync/src/types/mod.rs`)

---

### Summary

`SyncState::add_ask_for_txs` inserts up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) hashes from a peer into the shared `unknown_tx_hashes` queue **before** checking whether the global soft-cap (`MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000) has been exceeded. When the cap is exceeded, the function performs an O(n) linear scan of the entire queue while holding the `unknown_tx_hashes` mutex. A peer that fills the queue but stays below the per-peer ban threshold receives `Status::ignored()` — no ban — and the inserted hashes remain in the queue indefinitely (no TTL, no cleanup on peer disconnect). This allows an unprivileged P2P peer to permanently saturate the queue, causing every subsequent peer that announces 32,767 hashes to be immediately banned, disrupting transaction propagation across the node.

---

### Finding Description

In `sync/src/types/mod.rs`, `SyncState::add_ask_for_txs` is called from `TransactionHashesProcess::execute` whenever a peer sends a `RelayTransactionHashes` P2P message. The function:

1. **Inserts first, checks later.** All hashes from the incoming message (up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767) are unconditionally inserted into the `unknown_tx_hashes` `KeyedPriorityQueue` while holding its `Mutex`. Only after insertion does the function check whether the queue length has exceeded `MAX_UNKNOWN_TX_HASHES_SIZE` (50,000).

2. **O(n) scan under mutex.** When the post-insertion check fires, the function iterates over every entry in the queue — potentially 50,000+ entries — counting how many belong to the current peer. This entire scan holds the `unknown_tx_hashes` mutex, blocking all concurrent callers (`pop_ask_for_txs`, `mark_as_known_txs`, etc.).

3. **Soft ban threshold, no cleanup.** The peer is banned only if its personal count reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`. If the global cap is exceeded but the per-peer count is below the threshold, the function returns `Status::ignored()` — no ban, no rollback of the just-inserted hashes. Entries in `unknown_tx_hashes` have no TTL and are not removed when a peer disconnects or is banned.

The `Relayer` applies a token-bucket rate limiter of 30 requests/second per `(PeerIndex, message_type)` pair, but this does not prevent the queue from being filled: two messages of 32,767 hashes each saturate the global cap in under one second. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

### Impact Explanation

**Phase 1 — Queue saturation (2 messages, ~1 second):**
- Peer A sends message 1: 32,767 hashes inserted → queue = 32,767 (< 50,000) → `Status::ok()`, peer not banned.
- Peer A sends message 2: 32,767 more hashes inserted → queue = 65,534 (≥ 50,000) → O(65,534-entry) scan → peer A has 65,534 entries ≥ 32,767 → peer A banned.

**Phase 2 — Collateral banning of legitimate peers:**
- Queue remains at 65,534 entries (no cleanup on ban).
- Legitimate peer B sends 32,767 hashes → queue = 98,301 → O(n) scan → peer B has 32,767 entries ≥ 32,767 → peer B banned.
- Every subsequent peer that announces a full batch is immediately banned.

**Phase 3 — Sustained attack:**
- Peer A reconnects after the ban expires (`BAD_MESSAGE_BAN_TIME` = 5 minutes) and repeats. The queue never drains because the malicious hashes are never resolved (the peer never responds to `GetRelayTransactions`).

**Consequences:**
- Transaction propagation is disrupted: the node cannot track unknown transactions from honest peers.
- Honest peers are banned, degrading the node's connectivity.
- The O(n) mutex-held scan causes CPU spikes and blocks `pop_ask_for_txs` and `mark_as_known_txs` on every attack message.
- Memory grows beyond the intended 50,000-entry cap before any enforcement action. [5](#0-4) [6](#0-5)

### Likelihood Explanation

- **Entry path is fully unprivileged.** Any peer that completes the P2P handshake can send `RelayTransactionHashes`. No stake, no key, no special role required.
- **Cost is trivial.** Two messages of 32,767 32-byte hashes each = ~2 MB of data, sent in under one second.
- **Rate limiter does not prevent saturation.** The 30 req/sec limiter allows 2 messages before the queue is full; the attack completes within the first second.
- **No automatic recovery.** Entries have no TTL; the queue stays saturated until legitimate transactions happen to match the garbage hashes (extremely unlikely with random hashes).
- **Reconnect cycle is cheap.** After a 5-minute ban, the attacker reconnects and re-saturates in 2 messages. [7](#0-6) [8](#0-7) 

---

### Recommendation

1. **Check before inserting.** Query the per-peer count in `unknown_tx_hashes` before inserting any hashes. If the peer is already at or near `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, reject the message immediately without touching the queue.

2. **Maintain a per-peer counter.** Store a `HashMap<PeerIndex, usize>` alongside `unknown_tx_hashes` so the per-peer count can be read in O(1) instead of O(n). Update it atomically with insertions and removals.

3. **Evict on peer disconnect/ban.** When a peer is banned or disconnects, remove its entries from `unknown_tx_hashes` to prevent the queue from staying saturated after the attacker is gone.

4. **Add a TTL to queue entries.** Entries that have been in the queue longer than a configurable timeout (e.g., 2× `RETRY_ASK_TX_TIMEOUT_INCREASE`) should be expired, preventing indefinite saturation by non-responsive peers. [9](#0-8) [10](#0-9)

### Proof of Concept

```
1. Connect an unprivileged peer to the CKB node (standard P2P handshake).

2. Send RelayTransactionHashes message #1:
   - tx_hashes: [H1, H2, ..., H32767]  (32,767 random 32-byte hashes)
   - Result: all 32,767 hashes inserted into unknown_tx_hashes.
   - Queue length: 32,767 < MAX_UNKNOWN_TX_HASHES_SIZE (50,000).
   - Status::ok() returned. Peer NOT banned.

3. Send RelayTransactionHashes message #2 (within the same second):
   - tx_hashes: [H32768, H32769, ..., H65534]  (32,767 more random hashes)
   - Result: all 32,767 hashes inserted → queue length: 65,534.
   - Post-insertion check fires: 65,534 >= 50,000.
   - O(65,534-entry) scan: peer has 65,534 entries >= 32,767 → peer banned.
   - Queue remains at 65,534 entries (no cleanup).

4. Connect a legitimate peer and send RelayTransactionHashes with 32,767 hashes:
   - All 32,767 hashes inserted → queue: 98,301.
   - O(n) scan: legitimate peer has 32,767 entries >= 32,767 → legitimate peer banned.

5. Repeat step 4 for every new peer: each is banned immediately upon announcing transactions.

6. After BAD_MESSAGE_BAN_TIME (5 minutes), reconnect the malicious peer and repeat
   from step 2 to keep the queue saturated indefinitely.
```

The attack requires no on-chain funds, no cryptographic material, and no privileged access — only the ability to open a P2P connection to the target node. [1](#0-0) [2](#0-1) [3](#0-2)

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }

        // Check `unknown_tx_hashes`'s length after inserting the arrival `tx_hashes`
        if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE
            || unknown_tx_hashes.len()
                >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
        {
            warn!(
                "unknown_tx_hashes is too long, len: {}",
                unknown_tx_hashes.len()
            );

            let mut peer_unknown_counter = 0;
            for (_hash, priority) in unknown_tx_hashes.iter() {
                for peer in priority.peers.iter() {
                    if *peer == peer_index {
                        peer_unknown_counter += 1;
                    }
                }
            }
            if peer_unknown_counter >= MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER {
                return StatusCode::TooManyUnknownTransactions.into();
            }

            return Status::ignored();
        }

        Status::ok()
    }

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "TxHashes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    relay_transaction_hashes.tx_hashes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
