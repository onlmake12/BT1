### Title
Entropy `revealWithCallback` Legacy Path Calls Consumer Callback Without Try/Catch, Permanently Blocking Request Fulfillment — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`'s `revealWithCallback` function, the legacy execution path (triggered when `req.gasLimit10k == 0`) invokes the consumer's `_entropyCallback` directly without any try/catch protection. If the consumer's callback reverts for any reason — including a paused token inside the callback — the entire `revealWithCallback` transaction reverts. Because the request is cleared before the callback call (checks-effects-interactions), the revert undoes the clearance, leaving the request permanently stuck in `CALLBACK_NOT_STARTED` state. The provider (Fortuna) can never fulfill the request.

---

### Finding Description

`revealWithCallback` has two distinct execution branches:

**New path** (`req.gasLimit10k != 0` and `callbackStatus == CALLBACK_NOT_STARTED`): Uses `excessivelySafeCall` to catch any revert from the consumer callback, emits `CallbackFailed`, and transitions the request to `CALLBACK_FAILED` state, allowing recovery.

**Legacy path** (the `else` branch, reached when `req.gasLimit10k == 0` OR when `callbackStatus == CALLBACK_FAILED`): Calls `_entropyCallback` directly with no error handling:

```solidity
// Lines 661–681
address callAddress = req.requester;
...
clearRequest(provider, sequenceNumber);   // ← cleared before callback
...
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(  // ← no try/catch
        sequenceNumber,
        provider,
        randomNumber
    );
}
```

`clearRequest` is called **before** the callback. If `_entropyCallback` reverts, the EVM unwinds the entire transaction, including the `clearRequest` side-effect. The request is restored to its pre-call state (`CALLBACK_NOT_STARTED`). The provider's `revealWithCallback` call reverts every time it is attempted, and there is no recovery path — the request is permanently stuck.

This path is reached for any provider whose `defaultGasLimit` is `0` (i.e., providers that have not opted into the new callback-failure-state flow). The condition is set at request time:

```solidity
// Lines 268–271
if (providerInfo.defaultGasLimit == 0) {
    // Provider doesn't support the new callback failure state flow
    // Set gasLimit10k to 0 to disable.
```

A consumer contract whose `echoCallback` / `entropyCallback` internally calls a **pausable ERC-20** (e.g., to distribute a reward token to the user) will revert whenever that token is paused, exactly mirroring the RED-token scenario in the external report.

---

### Impact Explanation

**Medium.** Any randomness request made through a provider with `defaultGasLimit == 0` whose consumer callback reverts (e.g., because a token used inside the callback is paused) becomes permanently unresolvable. The provider's keeper (Fortuna) cannot fulfill the request; the user never receives their random number. Because fees are credited to the provider at request time (not fulfillment time), the user also has no refund path — they must submit a new request and pay again, with no guarantee the new request will succeed if the underlying cause (paused token) persists.

---

### Likelihood Explanation

**Medium.** Consumer contracts that transfer or mint tokens inside their entropy callback are a common pattern (e.g., NFT minting, reward distribution). Many such tokens implement OpenZeppelin's `ERC20Pausable` or similar. A governance pause, emergency freeze, or regulatory action on any such token is sufficient to trigger the DoS. Providers that have not explicitly set `defaultGasLimit` (and thus use the legacy path) remain exposed.

---

### Recommendation

Apply the same `excessivelySafeCall` / try-catch pattern used in the new path to the legacy path as well. On callback revert, transition the request to `CALLBACK_FAILED` and emit `CallbackFailed`, giving the requester a visible failure state and allowing a future recovery attempt (with the revert reason visible). Alternatively, deprecate the legacy path entirely and require all providers to set a non-zero `defaultGasLimit`.

---

### Proof of Concept

1. Provider `P` has `defaultGasLimit == 0` (legacy path active).
2. Consumer contract `C` implements `entropyCallback` which calls `pausableToken.transfer(user, reward)`.
3. User calls `requestV2{value: fee}()` → request stored with `gasLimit10k == 0`, `callbackStatus = CALLBACK_NOT_STARTED`.
4. `pausableToken` admin pauses the token.
5. Fortuna (keeper) calls `revealWithCallback(P, seqNum, userContrib, providerContrib)`.
6. Execution enters the legacy `else` branch; `clearRequest` executes, then `_entropyCallback` is called.
7. Inside `_entropyCallback`, `pausableToken.transfer(...)` reverts with `"Pausable: paused"`.
8. The entire transaction reverts; `clearRequest` is undone; request remains `CALLBACK_NOT_STARTED`.
9. Fortuna retries → same revert every time.
10. Request is permanently stuck; user never receives randomness; no refund mechanism exists.
