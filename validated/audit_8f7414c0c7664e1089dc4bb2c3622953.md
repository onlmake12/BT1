Audit Report

## Title
Unverified Caller-Supplied `from` Peer ID in Hole-Punching `ConnectionRequest`/`ConnectionSync` Allows State Poisoning and Forced Outbound Connections - (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

The CKB hole-punching protocol accepts the `from` field in `ConnectionRequest` and `ConnectionSync` messages directly from the message payload without verifying it matches the actual peer ID of the delivering session. A single connected attacker can forge `ConnectionRequest` messages with arbitrary `from` peer IDs to poison `pending_delivered` with attacker-controlled listen addresses, then trigger outbound TCP connection attempts to those addresses via a forged `ConnectionSync`. Sustained at the session rate limit, this exhausts file descriptors and crashes the target node.

## Finding Description

**Root cause — unverified `from` in `ConnectionRequest`**

`RequestContent::try_from` parses `from` directly from the message bytes with no cross-check against the actual session: [1](#0-0) 

`execute()` passes this unverified `content.from` directly to `respond_delivered`: [2](#0-1) 

`respond_delivered` inserts the attacker-controlled listen addresses into `pending_delivered` keyed by the forged peer ID: [3](#0-2) 

**Root cause — unverified `from` in `ConnectionSync`**

`SyncContent::try_from` has the identical defect — `from` is parsed from the payload with no session identity check: [4](#0-3) 

`execute()` uses the attacker-supplied `content.from` to look up the poisoned `pending_delivered` entry: [5](#0-4) 

It then spawns `try_nat_traversal` tasks for every address in the poisoned entry, and on success calls `control.raw_session(...)`: [6](#0-5) 

Critically, `ConnectionSyncProcess` does not even receive a `peer: PeerIndex` parameter in its constructor, so the actual sender identity is structurally unavailable: [7](#0-6) 

**Why existing guards fail**

- **`HOLE_PUNCHING_INTERVAL` check**: Keyed on `from_peer_id` (i.e., `content.from`). Varying the forged `from` value on each message bypasses it entirely: [8](#0-7) 

- **`forward_rate_limiter`**: Configured at 1 req/s, keyed on `(content.from, content.to, msg_item_id)`. Since `content.from` is attacker-controlled, varying it per message bypasses the limiter: [9](#0-8) 

- **Session `rate_limiter`**: Keyed on `(session_id, msg.item_id())` at 30 req/s. This is the only real constraint, but 30 `ConnectionRequest` + 30 `ConnectionSync` messages per second is sufficient to exhaust file descriptors: [10](#0-9) 

- **`inflight_requests` guard**: Only applies in `ConnectionRequestDeliveredProcess` when the current node is the `from` originator. It does not protect the `to` target node from `pending_delivered` poisoning: [11](#0-10) 

- **`pending_delivered` cleanup**: Entries are only pruned every 5 minutes with a 5-minute TTL, so poisoned entries persist long enough to be exploited: [12](#0-11) 

## Impact Explanation

**High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

Each forged `ConnectionSync` causes the target node to spawn an async task that opens an outbound TCP socket to an attacker-chosen address. If the attacker's server does not complete the handshake, each socket remains open for the OS connection timeout (typically 20–120 seconds on Linux). At 30 `ConnectionSync` messages/second (session rate limit), the node accumulates 30 × 20 = 600+ concurrent open file descriptors within 20 seconds, exhausting the default per-process fd limit (~1024 on Linux) and crashing the node. Additionally, a successful NAT traversal to the attacker's endpoint results in `raw_session` being called, adding the attacker as a new inbound session and bypassing normal peer discovery and connection limits.

## Likelihood Explanation

**High.** The attacker only needs to be a connected peer — reachable by any unprivileged external party. No keys, credentials, or special privileges are required. The forged `from` field is a plain byte sequence; no cryptographic material is needed. The hole-punching protocol is active whenever the node has outbound connections and NAT traversal is needed. The attack is repeatable and automatable with a simple loop varying the forged `from` peer ID on each iteration.

## Recommendation

In `ConnectionRequestProcess::execute()`, after parsing `content`, resolve the actual peer ID of the delivering session from the peer registry using `self.peer` and reject the message if `content.from` does not match:

```rust
// In ConnectionRequestProcess::execute(), after parsing content:
let actual_peer_id = self.protocol.network_state
    .peer_registry
    .read()
    .get_peer(self.peer)
    .map(|p| p.peer_id.clone());

if actual_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId
        .with_context("from peer id does not match actual sender");
}
```

For `ConnectionSyncProcess`, the `peer: PeerIndex` parameter must be added to the constructor (mirroring `ConnectionRequestProcess`) so the same check can be applied in `execute()`. The call site in `mod.rs` at line 134–143 must be updated to pass `context.session.id`.

## Proof of Concept

```
Precondition: Attacker (peer A) is directly connected to target node C.

Step 1 — Forge ConnectionRequest:
  A → C:
    ConnectionRequest {
      from: <arbitrary_peer_B_id>,   // forged; not A's real peer ID
      to:   <peer_C_id>,
      listen_addrs: [<attacker_ip:port>],
      max_hops: 6,
      route: []
    }

Step 2 — C processes request (connection_request.rs:145-147):
  self_peer_id == content.to → respond_delivered(peer_B_id, C, [attacker_ip:port])
  C stores: pending_delivered[peer_B_id] = ([attacker_ip:port], now)
  C sends ConnectionRequestDelivered back to A (self.peer)

Step 3 — Forge ConnectionSync:
  A → C:
    ConnectionSync {
      from: <arbitrary_peer_B_id>,   // same forged identity
      to:   <peer_C_id>,
      route: [],
      sync_route: []
    }

Step 4 — C processes sync (connection_sync.rs:111-115):
  listens_info = pending_delivered[peer_B_id] = [attacker_ip:port]
  C calls try_nat_traversal(bind_addr, attacker_ip:port)
  → outbound TCP SYN to attacker_ip:port (fd held open until timeout)
  → on success: control.raw_session(...) establishes P2P session with attacker

Repeat Steps 1–4 with distinct peer_B_id values at up to 30/s (session
rate limiter) to accumulate 600+ open file descriptors within 20 seconds,
exhausting the process fd limit and crashing the node.
```
