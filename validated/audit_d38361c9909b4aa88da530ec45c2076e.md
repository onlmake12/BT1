### Title
Stale Price Replay via Zero-Price Bypass in `update_price_feeds_internal` — (File: `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`)

---

### Summary

The Stylus Pyth receiver's `update_price_feeds_internal` function contains a flawed update guard that allows any attacker to overwrite a **newer** stored price with an **older** one whenever the currently stored price value is exactly zero. This is a direct analog to the external report's "incorrect value in record" class: a special-case branch bypasses the normal freshness check and accepts unvalidated (stale) data.

---

### Finding Description

In `update_price_feeds_internal`, the condition controlling whether a new price update is written to storage is:

```rust
if recent_price_info.publish_time.get() < price_return.1
    || recent_price_info.price.get() == I64::ZERO
{
    // write all fields: publish_time, expo, price, conf, ema_price, ema_conf
}
``` [1](#0-0) 

The first clause (`publish_time < new_publish_time`) is the correct freshness guard. The second clause (`price == I64::ZERO`) is the bug: it unconditionally allows any incoming update to overwrite the stored record whenever the stored `price` field is zero, **regardless of whether the incoming update is older**.

The reference EVM implementation does not have this second clause — it only updates when the incoming publish time is strictly newer: [2](#0-1) 

The Stylus contract is the only receiver that introduces this extra `|| price == 0` branch.

---

### Impact Explanation

If a price feed legitimately stores `price = 0` (e.g., a binary option that expired worthless, or any asset whose Pyth-reported price rounds to zero at a given exponent), an attacker can replay any previously valid, Wormhole-signed accumulator update for that feed — even one with an older `publish_time` — and have it accepted. The stored record is then rolled back to the older, stale price. Any DeFi protocol on the Stylus chain reading `get_price_unsafe` or `get_price_no_older_than` from this contract will receive the stale price, potentially enabling liquidation manipulation, incorrect collateral valuation, or arbitrage against the stale value.

---

### Likelihood Explanation

- **Attacker-controlled entry path**: `update_price_feeds` is a public, permissionless function callable by any transaction sender. No privileged role is required.
- **Precondition**: The stored `price` field must be `I64::ZERO`. This is unusual for most assets but is a valid on-chain state (e.g., after a legitimate zero-price update, or for newly initialized feeds where `publish_time != 0` but `price == 0` due to a first update carrying a zero price).
- **Replay material**: All historical Wormhole-signed accumulator updates are publicly available from Hermes. The attacker does not need to forge any cryptographic material.
- **Likelihood**: Medium — the zero-price precondition is uncommon but reachable, and the exploit itself requires only a public function call with a replayed valid update.

---

### Recommendation

Remove the `|| recent_price_info.price.get() == I64::ZERO` clause. The update guard should be solely based on `publish_time` freshness, consistent with every other Pyth receiver implementation:

```rust
// Before (vulnerable):
if recent_price_info.publish_time.get() < price_return.1
    || recent_price_info.price.get() == I64::ZERO

// After (correct):
if recent_price_info.publish_time.get() < price_return.1
``` [3](#0-2) 

---

### Proof of Concept

1. A price feed `F` receives a legitimate update at time `T2` with `price = 0`, `publish_time = T2`. The Stylus contract stores this.
2. An attacker fetches an older, still-valid Wormhole-signed accumulator update for feed `F` from Hermes, with `publish_time = T1 < T2` and `price = P` (any non-zero value).
3. The attacker calls `update_price_feeds(older_update_data)` on the Stylus Pyth receiver, paying the required fee.
4. Inside `update_price_feeds_internal`, the guard evaluates:
   - `recent_price_info.publish_time.get() < price_return.1` → `T2 < T1` → **false** (stored is newer)
   - `recent_price_info.price.get() == I64::ZERO` → `0 == 0` → **true**
5. The condition passes; the stored record is overwritten with `publish_time = T1`, `price = P`.
6. Subsequent calls to `get_price_unsafe(F)` or `get_price_no_older_than(F, age)` return the stale price `P` with the older timestamp `T1`. [4](#0-3) [5](#0-4)
