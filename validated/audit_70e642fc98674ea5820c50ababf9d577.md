Audit Report

## Title
Missing Per-Peer Rate Limiter in Filter Protocol Enables Unbounded RocksDB Read Amplification — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filters_process.rs`)

## Summary
The `BlockFilter` protocol handler processes `GetBlockFilters`, `GetBlockFilterHashes`, and `GetBlockFilterCheckPoints` messages with no per-peer rate limiting. Each `GetBlockFilters` message triggers up to 2,000 RocksDB reads and each `GetBlockFilterHashes` message triggers up to ~4,001 reads. Any unprivileged inbound peer can send these small messages in a tight loop, and with up to ~117 inbound peers allowed by default, the I/O amplification scales linearly with peer count, capable of saturating disk I/O and starving the async executor.

## Finding Description
**Root cause — no rate limiter on `BlockFilter`:**
The `BlockFilter` struct contains only a `shared: Arc<SyncShared>` field with no `rate_limiter` member. [1](#0-0) 

`try_process` dispatches directly to each process handler with zero rate-limiting checks before or after dispatch: [2](#0-1) 

**Contrast — Relay protocol has an explicit governor-based rate limiter:**
`Relayer` holds a `rate_limiter: RateLimiter<(PeerIndex, u32)>` initialized at 30 req/s: [3](#0-2) [4](#0-3) 

And checks it before every dispatch in `try_process`: [5](#0-4) 

**Per-message DB read amplification — `GetBlockFilters`:**
`BATCH_SIZE = 1000`; each loop iteration calls `get_block_hash` then `get_block_filter`, yielding up to 2,000 RocksDB reads per message. The 1.8 MB size cap only terminates the loop early for large filters; for early mainnet blocks with small filters, all 1,000 iterations complete: [6](#0-5) [7](#0-6) 

**Per-message DB read amplification — `GetBlockFilterHashes`:**
`BATCH_SIZE = 2000`; each loop iteration calls `get_block_hash` chained with `get_block_filter_hash` (2 reads), plus 1–2 extra reads for the parent hash lookup before the loop, yielding up to ~4,001 reads per message. No size cap exists here: [8](#0-7) [9](#0-8) 

**Peer count amplification:**
Default config allows up to 125 total peers with 8 outbound, meaning up to ~117 inbound peers: [10](#0-9) 

**Existing checks are insufficient:**
The only existing guard is the 1.8 MB response size cap in `GetBlockFilters`, which limits response payload size but does not bound the request rate. Malformed messages result in a ban, but well-formed `GetBlockFilters` messages are processed unconditionally. The `disconnected` handler in `BlockFilter` performs no cleanup of any rate-limiting state (unlike `Relayer.disconnected` which calls `self.rate_limiter.retain_recent()`): [11](#0-10) [12](#0-11) 

## Impact Explanation
Sustained high-rate `GetBlockFilters`/`GetBlockFilterHashes` messages from multiple inbound peers cause RocksDB I/O saturation and async executor thread starvation, degrading or crashing the target CKB node. This maps to the allowed impact: **"Vulnerabilities which could easily crash a CKB node"** (High, 10001–15000 points). The attack requires no privileged access, no PoW, and no special resources — only the ability to establish inbound TCP connections and send a small valid message repeatedly.

## Likelihood Explanation
- The Filter protocol is enabled by default in production `ckb.toml`.
- The attack message is trivially constructable (9 bytes, valid deserialization, no validation beyond that).
- The contrast with the Relay protocol's explicit rate limiter confirms this is an unintentional omission.
- The prior fix (PR #4972) addressed response size but left request rate unbounded, showing the team is aware of amplification risks in this handler but has not addressed the rate dimension.
- Any attacker capable of establishing inbound connections to a CKB node can execute this attack.

## Recommendation
Add a governor-based rate limiter to `BlockFilter` mirroring the Relay protocol:

1. Add `rate_limiter: RateLimiter<(PeerIndex, u32)>` to the `BlockFilter` struct in `sync/src/filter/mod.rs`.
2. Initialize it in `BlockFilter::new` with a quota of 30 req/s or lower, given the higher per-message cost of filter reads vs. relay messages.
3. In `try_process`, check `self.rate_limiter.check_key(&(peer, message.item_id()))` before dispatching to any handler, returning `StatusCode::TooManyRequests` on failure.
4. Apply the same guard to `GetBlockFilterHashes` (BATCH_SIZE=2000, up to ~4,001 reads/msg) and `GetBlockFilterCheckPoints`.
5. Call `self.rate_limiter.retain_recent()` in the `disconnected` handler to prevent unbounded memory growth.

## Proof of Concept
```
1. Connect N peers (e.g., N=50) to a target CKB full node with block filter enabled.
2. Each peer sends in a tight loop:
     GetBlockFilters { start_number: 0 }
   (9-byte message, no PoW or auth required)
3. Observe on the target node:
   - RocksDB read IOPS spike to N × ~2000 reads/round
   - CPU usage on async executor threads increases
   - Sync latency for honest peers degrades
   - Node I/O wait increases proportionally with N
4. Repeat with GetBlockFilterHashes { start_number: 0 } for ~4001 reads/msg.
5. Compare against a patched node with a 30 req/s rate limiter:
   - IOPS capped at N × 30 × 2000 reads/s regardless of send rate.

The existing integration test harness at test/src/specs/sync/block_filter.rs provides
a template for peer connection and filter message construction to implement this PoC.
```
