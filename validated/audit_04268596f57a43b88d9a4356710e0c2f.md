Audit Report

## Title
Attacker-Controlled Snappy Varint Triggers Up-to-8MB Heap Allocation Before Decompression Verification — (`network/src/compress.rs`)

## Summary
In `LengthDelimitedCodecWithCompress::decode`, `BytesMut::zeroed(decompressed_bytes_len)` is allocated unconditionally based on the snappy stream's varint header — a value entirely under attacker control — before any verification that the compressed payload is valid. An unprivileged remote peer can trigger this allocation with an ~8-byte wire frame. With 117 simultaneous inbound connections (default configuration), this yields ~924MB peak RSS. No codec-layer ban or reconnect throttle exists, so the attacker reconnects immediately and repeats, causing continuous allocator pressure that can crash the node via OOM.

## Finding Description
**Root cause — `network/src/compress.rs` line 242:**

```rust
let mut buf = BytesMut::zeroed(decompressed_bytes_len);
match SnapDecoder::new().decompress(&data[1..], &mut buf) {
```

`decompress_len(&data[1..])` reads the snappy varint from the attacker-controlled stream header. If the returned value is ≤ `MAX_UNCOMPRESSED_LEN` (8MB, line 13), `BytesMut::zeroed(decompressed_bytes_len)` is allocated unconditionally. Only then is decompression attempted; if the payload is garbage, `decompress` returns `Err`, the allocation is freed, and the connection is dropped — but the peak RSS spike already occurred.

**Why the `max_frame_length` guard does not help — `network/src/protocols/support_protocols.rs` lines 122–137:**

The `length_delimited.decode(src)?` at line 226 of `compress.rs` enforces the per-protocol compressed wire frame limit (2MB for Sync, 4MB for RelayV3). A crafted frame of ~8 bytes — `COMPRESS_FLAG` byte + snappy varint encoding 7.9MB + a few garbage bytes — trivially passes this check because the check operates on the compressed wire size, not the claimed decompressed size.

**Why no ban is applied:**

`ban_peer` lives in `DefaultCKBProtocolContext` at the application layer (`network/src/protocols/mod.rs` line 684). When `decode` returns `Err`, tentacle closes the connection at the codec layer; the application-layer handler is never invoked, so `ban_peer` is never called. The attacker reconnects immediately.

**`enable_compress` does not gate the decoder:**

The `decode` function checks `(data[0] & COMPRESS_FLAG) != 0` regardless of `self.enable_compress`. The encoder respects the flag; the decoder does not. All protocols built via `CKBProtocol::new_with_support_protocol` hardcode `compress: true`, so this is moot in practice, but the decoder is universally vulnerable.

**Inbound peer limit — `network/src/peer_registry.rs` lines 113–121 and `util/app-config/src/configs/network.rs` line 356:**

`max_inbound_peers()` = `max_peers − max_outbound_peers`. With defaults of 125 and 8, this is 117 simultaneous inbound connections. Each sends one crafted frame: 117 × 7.9MB ≈ **924MB peak simultaneous allocation**.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

With 117 simultaneous connections each sending a single ~8-byte crafted frame, the target node allocates ~924MB in parallel. On nodes with ≤1GB RAM (common for validators and light nodes), this triggers OOM kill. Even on nodes with more RAM, the continuous reconnect-and-repeat cycle causes sustained allocator churn, degraded P2P throughput across all protocols (Sync, RelayV3, LightClient, Filter), and eventual crash under sustained attack.

## Likelihood Explanation
The attack requires only a TCP connection to port 8115 (the default public P2P port). No authentication, no PoW, no key material, no prior relationship with the target node is needed. The crafted frame is trivial to construct (8 bytes). The default `max_peers = 125` provides no meaningful barrier — the attacker opens 117 connections from multiple IPs or uses IP spoofing. After each disconnect, reconnection is immediate because no codec-layer ban is applied. The attack is fully repeatable and automatable.

## Recommendation
1. **Primary fix**: Move the allocation after lightweight structural validation of the compressed payload. Use `snap::raw::decompress_len` only to size-check, then validate the compressed stream header bytes before allocating. Alternatively, replace `BytesMut::zeroed(decompressed_bytes_len)` with a streaming/lazy decompressor that does not pre-allocate the full claimed size.
2. **Secondary mitigation**: Add a codec-layer peer ban (or exponential backoff with IP-level rate limiting) when `decode` returns `Err` for a compressed frame, to prevent immediate reconnect-and-repeat.
3. **Defense in depth**: Consider reducing `MAX_UNCOMPRESSED_LEN` to match the actual maximum legitimate message size for each protocol, rather than a single global 8MB cap.

## Proof of Concept

```python
import socket, struct, time

# Snappy varint encoding of 7,864,320 (7.9 MB)
n = 7_864_320
varint = b''
while n > 0x7F:
    varint += bytes([(n & 0x7F) | 0x80])
    n >>= 7
varint += bytes([n])

# Frame: COMPRESS_FLAG=0x80, snappy varint for 7.9MB, garbage payload
payload = bytes([0x80]) + varint + b'\xff\xff\xff'

# LengthDelimited framing: 4-byte big-endian length prefix
frame = struct.pack('>I', len(payload)) + payload

# After tentacle secio handshake, send frame on Sync protocol channel.
# Each send triggers BytesMut::zeroed(7,864,320) on the target node.
# Connection is dropped; reconnect immediately and repeat.
# With 117 simultaneous connections: ~924MB peak RSS on target.
for _ in range(117):
    s = socket.socket()
    s.connect(('TARGET_IP', 8115))
    # ... complete tentacle secio handshake ...
    s.sendall(frame)
    time.sleep(0.01)
```

**Minimal unit test to confirm the allocation path:**
```rust
#[test]
fn test_attacker_controlled_allocation() {
    use tokio_util::codec::Decoder;
    use p2p::bytes::BytesMut;
    use crate::compress::LengthDelimitedCodecWithCompress;
    use tokio_util::codec::LengthDelimitedCodec;

    let mut codec = LengthDelimitedCodecWithCompress::new(
        true, LengthDelimitedCodec::new(), 1.into()
    );
    // COMPRESS_FLAG + snappy varint for 7,864,320 + garbage
    let payload: Vec<u8> = vec![0x80, 0x80, 0x80, 0xe0, 0x03, 0xff, 0xff, 0xff];
    let mut frame = BytesMut::new();
    frame.extend_from_slice(&(payload.len() as u32).to_be_bytes());
    frame.extend_from_slice(&payload);
    // This triggers BytesMut::zeroed(7_864_320) before returning Err
    let result = codec.decode(&mut frame);
    assert!(result.is_err()); // allocation occurred, then freed
}
