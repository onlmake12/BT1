### Title
Missing Non-Empty Guard on `updates` in `pyth_price.withdraw` Allows Zero-Verification Invocation — (`lazer/contracts/cardano/validators/pyth_price.ak`)

### Summary
The `withdraw` validator iterates over the caller-supplied `updates` list to cryptographically verify each price message. Because Aiken's `list.for_each` on an empty list is a no-op that returns `Void`, passing `updates = []` satisfies every `expect` in the validator body and returns `True` with zero message verification performed.

### Finding Description
In `pyth_price.ak`, the `withdraw` handler performs three checks in sequence:

1. Locate the Pyth state UTxO via a reference input. [1](#0-0) 
2. Assert the calling script hash is the active withdraw script. [2](#0-1) 
3. Iterate `updates`, calling `message.parse_and_verify` on each element. [3](#0-2) 

There is no check that `updates` is non-empty before or after the loop. When `updates = []`, step 3 iterates zero times, the `expect _ = list.for_each(...)` succeeds trivially, and the validator returns `True`. [4](#0-3) 

### Impact Explanation
The `withdraw` validator is the on-chain trust anchor for Pyth Lazer on Cardano. Consumer contracts are expected to verify that this validator was invoked in the transaction (by inspecting `tx.withdrawals`) and then read the redeemer to obtain the set of cryptographically verified price updates. If the validator can be invoked with an empty redeemer and still succeed, the invariant "validator was called ⟹ at least one price was verified" is broken. A consumer contract that treats the presence of the withdrawal as proof of verification, without separately asserting the redeemer is non-empty, will accept a transaction that carries no verified price data at all. This is an on-chain program flaw in the Pyth contract itself, not solely a consumer-side issue.

### Likelihood Explanation
Any unprivileged relayer can construct a valid transaction: include the Pyth state UTxO as a reference input, set the withdrawal credential to the active `withdraw_script` hash, and supply `updates = []` as the redeemer. No key material, governance access, or privileged role is required. The preconditions are entirely public.

### Recommendation
Add an explicit non-empty guard at the top of the `withdraw` handler before the loop:

```aiken
expect list.length(updates) > 0
```

or equivalently pattern-match to ensure at least one element is present. This makes the invariant explicit and enforced at the contract level rather than relying on consumer contracts to re-check it.

### Proof of Concept
A unit test analogous to the existing `validate_single_price` test, but with `updates = []`, would pass today:

```aiken
test validate_empty_updates_bypasses_verification() {
  let ref_input =
    signer_ref_input(
      test_pyth_id,
      assets.from_asset(test_pyth_id, config.state_nft, 1),
      [Pair(test_signer_pubkey, interval.before(1000))],
    )
  // No `fail` annotation — this is expected to return True
  pyth_price.withdraw(
    test_pyth_id,
    [],                              // empty updates list
    Script(test_withdraw_script_hash),
    test_tx([ref_input]),
  )
}
```

The validator returns `True` and the redeemer contains zero verified messages, confirming the missing invariant. [5](#0-4)
