### Title
`updatePriceFeeds()` Does Not Refund Excess ETH to Caller - (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary
`Pyth.updatePriceFeeds()` uses a `msg.value < requiredFee` check (i.e., `>=` semantics) but never refunds the excess ETH to the caller. Any ETH above the required fee is silently retained by the contract, causing callers to lose funds unintentionally.

### Finding Description
In `Pyth.sol`, `updatePriceFeeds` computes the required fee from the number of price updates and checks only that `msg.value` is at least that amount:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
``` [1](#0-0) 

There is no subsequent refund of `msg.value - requiredFee`. The excess ETH accumulates in the contract balance and is only recoverable via a privileged governance `WithdrawFee` action — the original caller has no way to reclaim it.

This is structurally identical to the reported vulnerability: a `>=` fee guard with no excess-refund path.

### Impact Explanation
Any caller of `updatePriceFeeds` who sends more ETH than `getTotalFee(totalNumUpdates)` permanently loses the difference. This is a direct, unrecoverable financial loss for the user. The excess is not credited to any per-user balance; it simply inflates the contract's ETH balance until governance withdraws it.

**Impact: Medium** — direct ETH loss for callers; magnitude bounded by overpayment amount per transaction.

### Likelihood Explanation
`updatePriceFeeds` is the primary entry point for price-feed consumers across all Pyth-integrated protocols. Overpayment is realistic in several scenarios:
- Callers hardcode a fee estimate that becomes stale after a governance `SetTransactionFee` action lowers the fee.
- Callers pass a conservative `msg.value` buffer to avoid reverts (common pattern in DeFi integrations).
- The `getUpdateFee(uint updateDataSize)` overload (line 82–93) deliberately over-estimates by assuming 255 updates per batch, so callers using it will routinely overpay. [2](#0-1) 

**Likelihood: Medium** — the deprecated `getUpdateFee(uint)` overload structurally guarantees overpayment for any caller that uses it.

### Recommendation
Either refund the excess after the fee check:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
uint excess = msg.value - requiredFee;
if (excess > 0) {
    (bool ok, ) = payable(msg.sender).call{value: excess}("");
    require(ok, "refund failed");
}
```

Or enforce an exact-fee requirement:

```diff
- if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
+ if (msg.value != requiredFee) revert PythErrors.InsufficientFee();
```

### Proof of Concept
1. Governance sets `transactionFeeInWei = 1000` and `singleUpdateFeeInWei = 0`.
2. Caller invokes `updatePriceFeeds{value: 1 ether}(updateData)` where `updateData` is empty (0 updates), so `requiredFee = 1000`.
3. The call succeeds; `address(pyth).balance` increases by `1 ether`.
4. The caller has lost `1 ether - 1000 wei` with no recourse.

This is confirmed by the governance test, which shows `pyth.updatePriceFeeds{value: 1 ether}(updateData)` succeeding and the full `1 ether` remaining in the contract: [3](#0-2)
