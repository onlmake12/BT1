### Title
Request Cleared Before Callback Execution Enables Permanent DoS on Price Update Callbacks — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the `executeCallback` function clears the user's request and credits the provider's fees **before** invoking the consumer's `_echoCallback`. The callback is wrapped in a `try/catch` block, so failures are silently swallowed. If the callback reverts, the request is permanently gone with no retry path, causing a permanent denial-of-service on the user's price update.

### Finding Description

In `Echo.sol`'s `executeCallback`, the execution order is:

1. Provider fees are credited: `_state.providers[providerToCredit].accruedFeesInWei += ...`
2. The request is permanently deleted: `clearRequest(sequenceNumber)`
3. `firstUnfulfilledSeq` is advanced past the now-cleared slot
4. The consumer callback is invoked under `try/catch` [1](#0-0) 

The `try/catch` block silently absorbs any revert from `_echoCallback`, emitting only a `PriceUpdateCallbackFailed` event. Because `clearRequest` already ran, the sequence number slot is gone and there is no mechanism to re-queue or retry the request.

The developers themselves flagged this ordering risk in an inline TODO: [2](#0-1) 

This is structurally identical to the reported pattern: **state is mutated (request cleared, fees credited) before the action (callback) completes, and the action can fail silently**.

### Impact Explanation

Any caller who can invoke `executeCallback` (the function has no visible access-control gate beyond the provider address argument) can supply valid price data that passes Pyth parsing but triggers a revert inside the consumer's `_echoCallback`. The request is permanently erased. The user paid the full fee, receives no price update, and cannot retry. This is a permanent, irreversible denial-of-service on every affected price-update request.

### Likelihood Explanation

`executeCallback` is permissionlessly callable by any address. A malicious provider, or any third party who front-runs the legitimate provider, can call it at will. Consumer contracts that revert on unexpected price values (e.g., staleness guards, circuit breakers) are common and provide a natural trigger. The attack requires no privileged access, no leaked keys, and no off-chain coordination beyond submitting a single transaction.

### Recommendation

Apply the checks-effects-interactions pattern correctly:

1. Invoke `_echoCallback` **first** (inside the `try/catch`).
2. Only on success: credit provider fees, call `clearRequest`, and advance `firstUnfulfilledSeq`.
3. On failure: leave the request active so the provider (or a retry mechanism) can re-attempt delivery, consistent with how `Entropy.sol` handles `CALLBACK_FAILED` state.

Alternatively, adopt the same `CALLBACK_FAILED` state machine used in `Entropy.sol` (`revealWithCallback`) so that failed requests remain retryable. [3](#0-2) 

### Proof of Concept

```
1. User calls Echo.requestPriceUpdatesWithCallback{value: fee}(provider, publishTime, priceIds, gasLimit)
   → sequenceNumber N is stored; req.requester = UserConsumer

2. UserConsumer._echoCallback is implemented with a guard that reverts
   (e.g., require(prices[0].price > 0, "bad price"))

3. Attacker (or provider) calls Echo.executeCallback(provider, N, updateData, priceIds)
   → line 161: provider.accruedFeesInWei += fee   ← state mutated
   → line 164: clearRequest(N)                     ← request deleted
   → line 176: try IEchoConsumer(req.requester)._echoCallback(...)
               → reverts inside consumer
   → line 183: catch → emit PriceUpdateCallbackFailed  ← silent failure

4. Request N is gone. User cannot retry. Provider is paid. User receives nothing.
``` [1](#0-0)
