Audit Report

## Title
Unbounded `last_n_numbers` allocation via `difficulty_boundary=U256::zero()` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
An unprivileged light-client peer can send a crafted `GetLastStateProof` message with `difficulty_boundary=U256::zero()`, `last_n_blocks=499`, `start_number=0`, and a valid tip hash. The upfront limit check passes (998 ≤ 1000), but `difficulty_boundary=0` causes the binary-search function to immediately return block 0 as the boundary. The subsequent adjustment guard only widens the boundary when too few blocks follow it — not when too many do — so `last_n_numbers` spans the entire chain height N, and `complete_headers` performs N sequential DB lookups and MMR root computations, entirely bypassing `GET_LAST_STATE_PROOF_LIMIT`.

## Finding Description

**Limit check** at lines 201–205 only validates input fields: [1](#0-0) 

With `difficulties=[]` and `last_n_blocks=499`: `0 + 499×2 = 998 ≤ 1000` — passes.

**Validation of `difficulty_boundary=U256::zero()` with empty `difficulties`** at lines 259–266: [2](#0-1) 

`difficulties.last()` returns `None`, so `None.map(...).unwrap_or(false)` = `false` — passes.

**`get_first_block_total_difficulty_is_not_less_than(0, N, &U256::zero())`** at lines 30–33: [3](#0-2) 

Genesis block's total difficulty is always ≥ 0, so `start_total_difficulty >= U256::zero()` is unconditionally true. The function immediately returns `Some((0, genesis_diff))`, setting `difficulty_boundary_block_number = 0`.

**Adjustment guard** at lines 313–316 only fires when there are *too few* blocks after the boundary: [4](#0-3) 

With `difficulty_boundary_block_number=0` and a tall chain, `N - 0 ≥ 499` — guard does not fire.

**`last_n_numbers` allocation** at lines 318–319: [5](#0-4) 

Produces `(0..N).collect()` — a `Vec<u64>` of N entries, unbounded.

**Branch at line 321** — since `difficulty_boundary_block_number == 0`, the `if` is false and execution falls to the `else` at line 345–346, returning the unbounded vec directly: [6](#0-5) 

**`complete_headers` is then called with all N block numbers**, performing N `get_ancestor` + `get_block` + `chain_root_mmr` DB lookups: [7](#0-6) 

Note that `get_ancestor` itself traverses the chain from `last_hash` to the target block, making the total DB work O(N²) for N blocks.

The `GET_LAST_STATE_PROOF_LIMIT = 1000` constant is entirely bypassed: [8](#0-7) 

## Impact Explanation

For a chain of height N, a single crafted message causes:
- A `Vec<u64>` allocation of N entries (8N bytes).
- N calls to `complete_headers`, each performing `get_ancestor` (O(N) chain traversal), `get_block`, and `chain_root_mmr` — total O(N²) synchronous DB/MMR operations on the server's snapshot.
- A `SendLastStateProof` response containing N `VerifiableHeader` structs (potentially gigabytes of serialized data).

On a chain of 1M blocks, this blocks the server thread for the duration of all DB I/O and can exhaust memory. Repeated requests from multiple peers can cause OOM or extreme latency, crashing the node.

**Impact: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a valid tip hash (publicly observable on the P2P network) and a single crafted `GetLastStateProof` message. No proof-of-work, no key, no privileged role is needed. Any peer connected to the light-client protocol port can trigger it repeatedly and from multiple connections simultaneously.

## Recommendation

After computing `difficulty_boundary_block_number`, enforce that the derived `last_n_numbers` length is bounded:

```rust
if last_block_number - difficulty_boundary_block_number > constant::GET_LAST_STATE_PROOF_LIMIT as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples after difficulty boundary");
}
```

Additionally, explicitly reject `difficulty_boundary=U256::zero()` at input validation time, since every block satisfies it and it serves no legitimate client purpose.

## Proof of Concept

1. Run a local CKB devnet with the light-client protocol enabled and mine 100,000 blocks.
2. Obtain the tip hash `tip_hash` from the P2P network.
3. Send the following message as an unprivileged peer:
   ```
   GetLastStateProof {
       last_hash: tip_hash,
       start_hash: genesis_hash,
       start_number: 0,
       last_n_blocks: 499,
       difficulty_boundary: U256::zero(),
       difficulties: [],
   }
   ```
4. Observe that `last_n_numbers.len() == 100_000` (not ≤ 1000), the server performs 100,000+ DB lookups, and the response is hundreds of MB.
5. Add the assertion `assert!(last_n_numbers.len() <= 1000)` after line 319 — it fails, confirming the invariant violation.
