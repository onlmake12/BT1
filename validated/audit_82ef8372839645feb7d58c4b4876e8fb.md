### Title
Off-by-one in `MAX_VMS_COUNT` spawn guard allows 17 concurrent VMs instead of 16 — (`script/src/scheduler.rs`)

### Summary

`Scheduler::process_message_box` uses a strict `>` comparison instead of `>=` when checking whether the VM count limit has been reached before spawning a new child VM. A script author can exploit this to create one VM beyond the declared `MAX_VMS_COUNT = 16` limit.

### Finding Description

`MAX_VMS_COUNT` is defined as `16`: [1](#0-0) 

The spawn guard at line 532 reads: [2](#0-1) 

When exactly 16 VMs are alive across `self.suspended` and `self.instantiated`, the expression evaluates as `16 > 16 == false`, so the guard does **not** fire and `boot_vm` is called, creating a 17th VM. The correct operator is `>=`.

### Impact Explanation

- **Resource limit bypass:** A script group execution can hold 17 concurrent VMs instead of the intended 16, violating the invariant documented at line 33–34.
- **Consensus divergence:** Transaction validity in CKB is consensus-critical. A transaction whose script spawns exactly 17 VMs would be accepted by nodes running the buggy code and rejected by nodes running a corrected build (using `>=`). This produces a chain split between patched and unpatched nodes.
- **Memory pressure:** Each extra VM carries its own heap, stack, and register state. While one extra VM is not catastrophic alone, the invariant break opens the door to further resource accounting errors.

### Likelihood Explanation

The exploit requires only a crafted V2 script submitted as a normal transaction — no privileged access, no PoW, no key material. The path is: transaction submission → script group execution → `Scheduler::process_message_box` → `Message::Spawn` arm → off-by-one guard. It is concretely reproducible in a unit test.

### Recommendation

Change `>` to `>=` on line 532:

```rust
// Before (buggy)
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct)
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

### Proof of Concept

1. Write a V2 script that spawns child VMs in a loop.
2. Spawn 15 children from the root VM (total = 16 including root), keeping all alive via `wait` not yet called.
3. Issue one more `spawn` syscall from the root VM.
4. **Expected (correct):** return code `MAX_VMS_SPAWNED` (8).
5. **Actual (buggy):** return code `SUCCESS` (0) and a 17th VM is created, confirmed by inspecting `self.instantiated.len() + self.suspended.len() == 17` after the call. [3](#0-2)
