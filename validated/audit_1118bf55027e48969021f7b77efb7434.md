Audit Report

## Title
Non-Deterministic Orphan Expiry via Single-Entry HashMap Sampling in `need_clean` — (`chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` samples only the first entry from a `HashMap<packed::Byte32, LonelyBlockHash>` to decide whether all blocks sharing a `parent_hash` should be evicted. Because Rust's `HashMap` iteration order is non-deterministic, a malicious P2P peer can inject a crafted block sharing a `parent_hash` with a legitimate orphan but carrying an expired `epoch_number`, causing either premature eviction of the legitimate orphan (sync disruption) or indefinite on-disk retention of the crafted block (storage bloat). No PoW is required because epoch continuity is only checked in the contextual `HeaderVerifier`, not in the non-contextual path that gates orphan pool insertion.

## Finding Description

**Root cause — `need_clean` samples one arbitrary entry:**

`InnerPool::need_clean` at [1](#0-0)  calls `map.iter().next()` on the inner `HashMap<packed::Byte32, LonelyBlockHash>`, returning an arbitrary entry. The comment at line 112 acknowledges this is intentional ("get 1st block"), but the assumption that all sibling blocks share the same epoch is incorrect and exploitable.

The inner structure is `HashMap<ParentHash, HashMap<packed::Byte32, LonelyBlockHash>>` [2](#0-1) , which allows multiple blocks with different `epoch_number` values to share the same `parent_hash`. If the arbitrary first entry is expired, `remove_blocks_by_parent` evicts the entire sibling group; if it is not expired, the entire group is retained.

**`epoch_number` is self-reported with no chain-state validation:**

When a `LonelyBlock` is converted to `LonelyBlockHash`, the `epoch_number` is taken directly from `block.epoch().number()` [3](#0-2)  — the block's own field — with no validation against the chain state.

**`non_contextual_verify` does not check epoch continuity or PoW:**

`asynchronous_process_block` calls `non_contextual_verify` [4](#0-3) , which runs only `BlockVerifier` and `NonContextualBlockTxsVerifier`. `BlockVerifier::verify` checks only proposals limit, block bytes, cellbase structure, duplicates, and merkle roots [5](#0-4)  — no PoW, no epoch continuity. A block with an arbitrary `epoch_number` (e.g., epoch 1) and a valid cellbase/merkle root passes this check.

**Block is written to disk before orphan pool insertion:**

After passing `non_contextual_verify`, the block is written to disk via `insert_block` [6](#0-5)  before being handed to the orphan broker. The crafted block therefore persists on disk regardless of later eviction decisions.

**Cleanup is triggered every 60 seconds:**

The `clean_expired_orphan_timer` fires every 60 seconds [7](#0-6) , calling `clean_expired_orphans`, which calls `clean_expired_blocks` on the pool. This is the reliable trigger window.

**Full exploit path:**

1. Attacker observes a legitimate orphan block B with `parent_hash = P` and `epoch_number = N` (current epoch, not expired).
2. Attacker crafts block A with `parent_hash = P`, `epoch_number = 1` (expired when `tip_epoch >= 7`), valid merkle root, valid cellbase, no PoW needed.
3. Block A passes `non_contextual_verify`, is written to disk, and inserted into the orphan pool under `blocks[P]`.
4. `blocks[P]` now contains both A (epoch 1) and B (epoch N).
5. When `clean_expired_blocks` fires, `need_clean(P, tip_epoch)` calls `map.iter().next()`:
   - If A is returned: `need_clean` returns `true` → `remove_blocks_by_parent(P)` evicts both A and B. Block B's disk record is deleted, its header view and block status are stripped. Sync stalls until B is re-requested.
   - If B is returned: `need_clean` returns `false` → neither block is evicted. Block A (and its on-disk record) is retained indefinitely.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

In the premature-eviction case, each targeted node must re-request the evicted legitimate orphan block, generating additional P2P traffic. The attack is repeatable every 60 seconds and amplifiable across many `parent_hash` values simultaneously, causing measurable network congestion. In the indefinite-retention case, crafted blocks accumulate on disk and in the in-memory `block_status_map` with no cleanup path, causing per-node resource exhaustion. Both effects compound with scale.

## Likelihood Explanation

The attacker is an unprivileged P2P sync peer requiring no PoW and no key or privilege. Crafting a valid block structure with an arbitrary `epoch_number` requires only constructing a `BlockBuilder` payload with consistent merkle roots and a well-formed cellbase — all computable locally. Target `parent_hash` values are observable from P2P gossip. The 60-second cleanup timer provides a predictable, repeatable trigger window. The attack is indefinitely repeatable.

## Recommendation

Replace the single-entry sample with a check over all entries under the parent. Evict only when every sibling block is expired:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            !map.is_empty()
                && map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

Alternatively, track the minimum `epoch_number` per `parent_hash` group at insertion time in `InnerPool::insert` to avoid iterating all entries on every cleanup tick.

## Proof of Concept

Add the following test to `chain/src/tests/orphan_block_pool.rs`:

```rust
#[test]
fn test_need_clean_nondeterministic_with_mixed_epochs() {
    use crate::utils::orphan_block_pool::{EXPIRED_EPOCH, OrphanBlockPool};
    use crate::{LonelyBlock, LonelyBlockHash};
    use ckb_chain_spec::consensus::ConsensusBuilder;
    use ckb_systemtime::unix_time_as_millis;
    use ckb_types::core::{BlockBuilder, EpochNumberWithFraction};
    use std::sync::Arc;

    let consensus = ConsensusBuilder::default().build();
    let pool = OrphanBlockPool::with_capacity(10);
    let parent = consensus.genesis_block().header();
    let tip_epoch = 20_u64; // EXPIRED_EPOCH=6, so epoch<=13 is expired

    // Block A: epoch 1 (expired: 1 + 6 < 20)
    let block_a = BlockBuilder::default()
        .parent_hash(parent.hash())
        .timestamp(unix_time_as_millis())
        .number(parent.number() + 1)
        .epoch(EpochNumberWithFraction::new(1, 0, 1000))
        .nonce(1u128.pack())
        .build();

    // Block B: epoch 15 (NOT expired: 15 + 6 = 21, not < 20)
    let block_b = BlockBuilder::default()
        .parent_hash(parent.hash())
        .timestamp(unix_time_as_millis() + 1)
        .number(parent.number() + 1)
        .epoch(EpochNumberWithFraction::new(15, 0, 1000))
        .nonce(2u128.pack())
        .build();

    pool.insert(LonelyBlock { block: Arc::new(block_a), switch: None, verify_callback: None }.into());
    pool.insert(LonelyBlock { block: Arc::new(block_b.clone()), switch: None, verify_callback: None }.into());

    let evicted = pool.clean_expired_blocks(tip_epoch);

    // Correct behavior: block_b (epoch 15) must never be evicted.
    // Bug: non-deterministic HashMap iteration may return block_a first,
    // causing need_clean to return true and evict both blocks.
    let evicted_hashes: Vec<_> = evicted.iter().map(|b| b.hash()).collect();
    assert!(
        !evicted_hashes.contains(&block_b.hash()),
        "block_b (epoch 15) must not be evicted"
    );
}
```

Run with multiple HashMap seeds (e.g., `RUSTFLAGS="-C target-cpu=native"` across multiple runs or with a fuzzer varying the seed) to observe both outcomes. The existing `test_remove_expired_blocks` test at [8](#0-7)  only tests the case where all blocks share the same epoch, and does not cover the mixed-epoch sibling scenario.
