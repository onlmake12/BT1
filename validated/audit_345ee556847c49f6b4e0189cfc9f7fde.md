Audit Report

## Title
Attacker-Controlled SQL OFFSET in `get_tx_with_cell` Enables Unbounded O(n) Database Scan via Crafted Pagination Cursor — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
When `group_by_transaction` is `false` or absent, the `get_transactions` RPC decodes the last 4 bytes of the `after` cursor as an `i32` OFFSET and passes it verbatim to the SQL engine with no upper-bound validation. An unprivileged caller can craft a cursor whose offset field is `i32::MAX` (2,147,483,647), forcing the database to materialize and discard billions of rows before returning results, making the rich-indexer RPC effectively unavailable for the duration of the query.

## Finding Description
The cursor validation at [1](#0-0)  checks only that `after.len() == 12`; no range check is applied to the decoded offset value. `decode_i32` at [2](#0-1)  performs only a byte-length check and returns any `i32` value, including `i32::MAX`. The decoded offset is passed directly to `query_builder.offset(offset)` at [3](#0-2)  with no cap. The `request_limit` guard at [4](#0-3)  only bounds the `LIMIT` clause, not `OFFSET`. By contrast, the grouped path (`get_tx_with_cells`) uses keyset pagination at [5](#0-4)  (`WHERE tx_id > after`) and is unaffected.

## Impact Explanation
SQL `OFFSET N` requires the database engine to scan and discard N rows before returning results. With `offset = i32::MAX`, every crafted request forces a full sequential scan of the result set, blocking the rich-indexer RPC for the duration of the query. Repeated requests can sustain this condition indefinitely. This constitutes a **local RPC API crash/unavailability**, matching the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**. The rich-indexer is an optional component separate from the core CKB node consensus path and does not affect block propagation, validation, or the CKB economy.

## Likelihood Explanation
The rich-indexer RPC is unauthenticated by default. The cursor format (12 bytes, last 4 = LE `i32` offset) is directly observable from any legitimate `get_transactions` response. No special privileges, keys, or hashpower are required. The exploit is trivially repeatable by any network-accessible caller.

## Recommendation
After decoding the offset, reject any cursor whose offset exceeds `request_limit` (a legitimate server-generated cursor will always have `offset <= limit`):

```rust
if offset as u32 > self.request_limit as u32 {
    return Err(Error::Params("Invalid cursor: offset out of range".to_string()));
}
```

A more robust long-term fix is to replace OFFSET-based pagination with keyset pagination using `(last_tx_id, last_io_index)` as the cursor, matching the approach already used in the grouped path (`get_tx_with_cells`), which uses `WHERE tx_id > after` and is O(1) regardless of position.

## Proof of Concept
1. Enable rich-indexer. Index a chain with a lock script appearing in many transaction inputs/outputs.
2. Call `get_transactions` with `group_by_transaction=false`, `limit=1` to obtain a valid 12-byte `last_cursor`.
3. Craft a new `after` value: take any valid 8-byte `last_id` prefix (e.g., `\x01\x00\x00\x00\x00\x00\x00\x00`), append `\xff\xff\xff\x7f` (= `i32::MAX` in little-endian) as the last 4 bytes.
4. Submit `get_transactions` with this crafted `after` cursor.
5. Observe that query latency grows proportionally to the offset value. The 12-byte length check passes, `decode_i32` succeeds, and `query_builder.offset(2147483647)` is emitted into the SQL, forcing the DB to skip over two billion rows.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
