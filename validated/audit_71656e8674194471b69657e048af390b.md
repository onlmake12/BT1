### Title
`min_fee_rate` Bypass via High-Cycle Low-Size Transactions — (`File: tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission check in `check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size (`tx_size`). However, the actual fee rate used for block inclusion, pool eviction, and miner sorting uses `get_transaction_weight(tx_size, cycles)` — which is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. An unprivileged RPC caller or P2P peer can craft a transaction with a small serialized size but very high cycle consumption, causing the admission check to pass with a fee far below the intended `min_fee_rate` threshold when measured by weight.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee rate using only `tx_size`:

```rust

// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0)

    std::cmp::max(
        tx_size as u64,

``` [5](#0-4) 

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`: [6](#0-5) 

The discrepancy: `check_tx_fee` uses `tx_size` alone, but the true cost metric is `weight = max(tx_size, cycles * 0.000_170_571_4)`. When cycles dominate, `weight >> tx_size`, and the admission check is far too permissive.

---

### Impact Explanation

An attacker crafts a transaction with:
- **Small serialized size**: ~200 bytes (a minimal 1-input/1-output transaction)
- **High cycle consumption**: up to `max_tx_verify_cycles = 70,000,000` (default config)

**Admission check** (`check_tx_fee`, using `tx_size = 200`):
- `min_fee = 1000 shannons/KW * 200 bytes / 1000 = 200 shannons`
- Transaction passes with fee = 200 shannons ✓

**Actual weight** (used for mining/eviction):
- `weight = max(200, 70_000_000 * 0.000_170_571_4) = max(200, 11,940) = 11,940`
- Effective fee rate = `200 * 1000 / 11,940 ≈ 16.7 shannons/KW`

This is ~**60× below** the configured `min_fee_rate = 1000 shannons/KW`. [7](#0-6) 

Consequences:
1. Attackers can flood the mempool (up to 180 MB) with transactions whose effective fee rate is far below `min_fee_rate`, at a fraction of the intended cost.
2. These transactions are deprioritized by miners (sorted by weight-based fee rate), so they are unlikely to be mined quickly — but they still occupy mempool space.
3. Legitimate transactions may be evicted or delayed as the pool fills with artificially cheap-to-admit, high-cycle transactions.
4. The `min_fee_rate` invariant — intended to prevent mempool spam — is effectively bypassed.

---

### Likelihood Explanation

- **Entry path**: Any unprivileged `send_transaction` RPC caller or P2P relay peer can submit such a transaction. No special privileges are required.
- **Craft difficulty**: Low. The attacker needs a CKB script that runs a tight loop consuming many cycles (e.g., a simple VM loop). The script itself can be tiny (a few bytes of bytecode), keeping `tx_size` small while driving cycles near `max_tx_verify_cycles`.
- **Cost**: The attacker pays only `~200 shannons` per transaction instead of the intended `~11,940 shannons`, a ~60× discount on mempool admission cost.
- **Repeatability**: The attacker can submit many such transactions in parallel, limited only by the 180 MB pool cap and the `max_ancestors_count` limit.

---

### Recommendation

Replace the size-only minimum fee check in `check_tx_fee` with a weight-based check, consistent with how the actual fee rate is computed everywhere else:

```rust
// Use weight (max of size and cycle-equivalent bytes) for the min fee check
let weight = get_transaction_weight(tx_size, /* actual cycles needed here */);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

The challenge is that `check_tx_fee` is called in `pre_check` before script execution (cycles are not yet known). The fix options are:

1. **Use declared cycles** (for remote/relay transactions) as an upper bound for the weight in the pre-check.
2. **Re-run the fee check after verification** (when actual cycles are known) using `get_transaction_weight(tx_size, verified.cycles)` before inserting the entry into the pool.
3. **Use `max_tx_verify_cycles` as a conservative upper bound** in the pre-check, which would be the strictest safe approximation.

Option 2 is the most accurate analog to the external report's fix (compute the minimum using the actual execution metric, not the user-supplied/pre-execution value).

---

### Proof of Concept

1. Deploy a CKB script that runs a tight loop consuming ~70,000,000 cycles. The bytecode can be very small (e.g., a RISC-V loop of ~20 bytes).
2. Create a transaction referencing this script as a lock, with 1 input and 1 output. Serialized size ≈ 200 bytes.
3. Set the transaction fee to `200 shannons` (= `min_fee_rate * tx_size / 1000 = 1000 * 200 / 1000`).
4. Submit via `send_transaction` RPC.

**Expected (correct) behavior**: Rejected with `LowFeeRate` because effective fee rate ≈ 16.7 shannons/KW < 1000 shannons/KW.

**Actual behavior**: Accepted into the mempool because `check_tx_fee` only checks `fee >= min_fee_rate.fee(tx_size) = 200 shannons`, which passes.

The transaction sits in the mempool with an effective fee rate of ~16.7 shannons/KW, far below `min_fee_rate`. Repeating this with many transactions fills the pool at ~60× lower cost than intended. [8](#0-7) [9](#0-8) [4](#0-3)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
