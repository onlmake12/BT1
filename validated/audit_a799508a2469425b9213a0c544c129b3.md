Audit Report

## Title
Tx-Pool Admission and RBF Fee Check Uses Serialized Size Only, Allowing High-Cycle Transactions to Underpay Fees — (File: tx-pool/src/util.rs, tx-pool/src/pool.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the transaction's serialized byte size, while the pool's own priority and eviction logic uses `get_transaction_weight(size, cycles)`. For cycle-heavy, byte-small transactions the weight can be ~60× larger than the size, so an unprivileged submitter can enter the pool paying a fee ~60× below the intended minimum. The same size-only calculation in `calculate_min_replace_fee` allows RBF replacements of high-cycle pool entries to pay less than the actual resource cost of the displaced transaction.

## Finding Description

**Pool admission — `check_tx_fee`, `tx-pool/src/util.rs` L42–45:**

The function computes the minimum required fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

This is the only fee gate. In `tx-pool/src/process.rs`, `check_tx_fee` is called at L289 and L294 before script verification, when actual cycles are not yet known: [2](#0-1) 

There is no secondary fee check after `verify_rtx` completes and actual cycles are available. The code path from `_process_tx` through `submit_entry` never re-evaluates the fee against the weight-based minimum.

After admission, `TxEntry::fee_rate()` in `tx-pool/src/component/entry.rs` L115–117 correctly uses `get_transaction_weight`:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [3](#0-2) 

The transaction is admitted at a size-only fee, then scored for mining using the correct (much larger) weight — it has a very low effective fee rate and will be deprioritized or evicted, but only after consuming pool resources.

**RBF replacement fee — `calculate_min_replace_fee`, `tx-pool/src/pool.rs` L101–103:**

```rust
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
    let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [4](#0-3) 

The caller at L98 passes `tx.size` (the existing pool entry's serialized size): [5](#0-4) 

If the existing entry has high cycles but small serialized size, the extra RBF fee is computed on size alone, allowing a replacement to pay less than the actual resource cost of the displaced transaction.

**Weight function — `util/types/src/core/tx_pool.rs` L298–303:**

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [6](#0-5) 

With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` (confirmed at L279): [7](#0-6) 

A transaction with `size = 200` bytes and `cycles = 70,000,000` (the default `max_tx_verify_cycles`) has:
- Actual weight: `max(200, 11,940) = 11,940`
- Fee required by admission: `min_fee_rate × 200 / 1000`
- Fee that should be required: `min_fee_rate × 11,940 / 1000`
- **Discount factor: ~59.7×**

## Impact Explanation

**High — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

An attacker can continuously submit cycle-heavy/byte-small transactions that pass the size-only admission gate while paying ~60× less than the intended minimum fee rate. These transactions occupy pool memory and consume block cycle budget at a steep discount. Because their effective fee rate (weight-based) is far below `min_fee_rate`, they are deprioritized for mining and eventually evicted — but the attacker can continuously resubmit them, sustaining pool pressure at minimal cost. The RBF path additionally allows displacing legitimate pool transactions while paying less than the intended minimum replacement cost, undermining the economic guarantees of the RBF mechanism.

## Likelihood Explanation

**Medium-High.** The attack requires no privileged access, no key material, and no majority hashpower. Any CKB script author can deploy a lock script with a tight loop consuming near-`max_tx_verify_cycles` cycles. The `send_transaction` RPC and P2P relay are both open to unprivileged users. The maximum cycle limit (70,000,000) is publicly documented and reachable. The attack is repeatable and cheap to sustain.

## Recommendation

1. **Pool admission**: After script verification completes and actual cycles are known, perform a second fee check using `get_transaction_weight(tx_size, actual_cycles)` before inserting the entry into the pool. If a pre-verification gate is still desired, use `declared_cycles` (when provided by the submitter for P2P relay) or `max_tx_verify_cycles` as a conservative upper bound.

2. **RBF replacement fee**: In `calculate_min_replace_fee`, replace `self.config.min_rbf_rate.fee(size as u64)` with `self.config.min_rbf_rate.fee(get_transaction_weight(size, cycles))`, where `cycles` is the actual cycle count of the existing pool entry being replaced.

## Proof of Concept

1. Deploy a CKB lock script that consumes ~70,000,000 cycles (e.g., a tight loop).
2. Craft a transaction spending a cell locked by this script, with minimal outputs (serialized size ≈ 200 bytes).
3. Set the transaction fee to `min_fee_rate × 200 / 1000 = 200 shannons` (at default `min_fee_rate = 1000 shannons/KW`).
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons` and accepts the transaction.
6. After verification, `TxEntry::fee_rate()` computes `weight = max(200, 11,940) = 11,940` and the effective fee rate is `200 × 1000 / 11,940 ≈ 16 shannons/KW` — ~62× below `min_fee_rate`.
7. Repeat continuously to sustain pool pressure. Each submission costs ~200 shannons instead of the intended ~11,940 shannons, a ~59.7× discount.
8. For RBF: submit a replacement for an existing high-cycle/small-size pool transaction; the extra RBF fee is computed on size alone, paying less than the actual resource cost of the replacement.
