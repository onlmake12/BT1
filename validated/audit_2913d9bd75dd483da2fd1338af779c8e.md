Audit Report

## Title
Unbounded `NOT IN` Clause in `get_cells` / `get_cells_capacity` Scales O(N) with tx-pool Dead Cells — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
When `index_tx_pool = true` is configured, `get_cells` and `get_cells_capacity` collect every entry from `Pool::dead_cells` and emit one SQL placeholder per entry into a `NOT IN (…)` clause with no cap. Because `dead_cells` is an unbounded `HashSet<OutPoint>` that grows with every accepted pool transaction, an attacker who fills the tx-pool forces every subsequent `get_cells` / `get_cells_capacity` RPC call to construct, bind, and submit a proportionally large SQL query, degrading RPC performance linearly with pool size up to effective unresponsiveness.

## Finding Description
**Unbounded pool accumulation.**
`Pool` in `util/indexer-sync/src/pool.rs` defines `dead_cells` as a plain `HashSet<OutPoint>` with no size cap: [1](#0-0) 

Every accepted pool transaction inserts all its inputs unconditionally with no limit check: [2](#0-1) 

**Unbounded NOT IN construction in `get_cells`.**
All dead cells are collected and expanded into a single `NOT IN` clause with one `$N` placeholder per entry — no cap is applied before or during this loop: [3](#0-2) 

Each `tx_hash` is then individually bound to the query, one bind call per dead cell: [4](#0-3) 

**`request_limit` guard is irrelevant.**
The existing guard caps the number of *result rows* returned, not the size of the `NOT IN` clause constructed before the query executes: [5](#0-4) 

**Same pattern in `get_cells_capacity`.**
Identical unbounded dead-cell collection and `NOT IN` expansion exists in `get_cells_capacity`: [6](#0-5) 

**Upper bound from tx-pool size.**
The default `max_tx_pool_size` is 180 MB: [7](#0-6) 

At a minimum transaction size of ~200 bytes, this permits ~900,000 pool transactions and a corresponding number of dead cells, producing a `NOT IN` clause of ~13 MB and ~28 MB of bound parameters per RPC call.

## Impact Explanation
Each `get_cells` and `get_cells_capacity` RPC call incurs O(N) work in SQL string construction, parameter binding, and database query-plan compilation proportional to the number of dead cells in the indexer pool. At pool saturation this renders the rich-indexer RPC effectively unresponsive for all callers. This matches **Note (0–500 points) — Any local RPC API crash**, and also **Low (501–2000 points) — Any other important performance improvements for CKB**, with the higher bound applicable when the RPC becomes fully unresponsive.

## Likelihood Explanation
The attack requires `index_tx_pool = true` (opt-in but a documented production feature) and sufficient live UTXOs to spend. The fee cost to fill 180 MB at the default minimum fee rate of 1000 shannons/KB is approximately 1.8 CKB — economically trivial. UTXO pre-creation is a one-time cost. Even a partial fill (e.g., 10,000 transactions) produces measurably increased latency on every `get_cells` call. The attack is repeatable as long as the pool remains full.

## Recommendation
Cap the number of dead cells included in the `NOT IN` clause to a fixed maximum (e.g., 1,000). If the live pool exceeds that cap, either omit the filter (accepting a small risk of transiently returning a just-spent cell) or return an explicit error to the caller. Alternatively, store dead cells in a temporary database table and join against it rather than expanding them inline into the query string, which also avoids database parameter-count limits.

## Proof of Concept
1. Configure the node with `index_tx_pool = true` in the indexer section.
2. Pre-create N distinct live cells on-chain (e.g., N = 10,000 or 100,000).
3. Submit N transactions to the tx-pool, each spending one distinct live cell (ensuring each transaction is valid and accepted).
4. Call `get_cells` via RPC and measure: SQL string length (loggable via a debug patch), parameter count, and wall-clock latency.
5. Repeat for increasing values of N and assert that latency and SQL string length grow linearly with N.
6. At N ≈ 900,000 (pool saturation at default 180 MB), assert that the RPC becomes effectively unresponsive or returns a database error due to query size limits.
