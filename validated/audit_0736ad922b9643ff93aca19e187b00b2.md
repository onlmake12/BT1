### Title
Unbounded SQL Aggregation in `get_cells_capacity` Allows RPC DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`)

---

### Summary

`AsyncRichIndexerHandle::get_cells_capacity` performs an unbounded `SUM(output.capacity)` SQL aggregation with no row limit and no query timeout. Unlike `get_cells` and `get_transactions`, it never checks `self.request_limit`. Any caller with RPC access can trigger a full-table aggregation scan repeatedly, exhausting DB CPU/I/O and stalling the async RPC thread pool.

---

### Finding Description

`get_cells` enforces `self.request_limit` immediately on entry: [1](#0-0) 

`get_cells_capacity` has no such guard — it proceeds directly to SQL construction: [2](#0-1) 

The field `request_limit` exists on `AsyncRichIndexerHandle` and is passed in at construction time: [3](#0-2) 

But it is never read inside `get_cells_capacity`. The SQL issued is an unbounded `SUM` aggregation with no `LIMIT` clause, no `OFFSET`, and no timeout.

The regular (RocksDB) indexer mitigates this with `TimeoutIterator` and a configurable `timeout_limit` (default 10 s). The rich-indexer has **no equivalent timeout mechanism** — confirmed by the absence of any `timeout_limit` usage in the rich-indexer codebase. [4](#0-3) 

When `script_search_mode=prefix` and `args` is empty (`0x`), `build_query_script_id_sql` generates:

```sql
SELECT script.id FROM script
WHERE code_hash = $1 AND hash_type = $2
  AND args >= $3 AND args < $4
```

with `$3 = []` (empty bytes). Any byte string is `>= []` in lexicographic order, so this matches **every script row** sharing the given `code_hash`/`hash_type`. On mainnet, the secp256k1 lock (`0x9bd7e06f...`) has millions of associated live cells. The outer query then does `SUM(output.capacity)` across all of them with no bound. [5](#0-4) 

The default `request_limit` when unconfigured is `usize::MAX`: [6](#0-5) 

The config comment explicitly warns that omitting `request_limit` can make the machine unresponsive: [7](#0-6) 

---

### Impact Explanation

An attacker who can reach the RPC endpoint (localhost by default, but publicly exposed by many infrastructure operators) can issue repeated `get_cells_capacity` calls with a broad prefix (e.g., secp256k1 code_hash + empty args). Each call forces a full `SUM` aggregation over millions of rows in SQLite or PostgreSQL. Because the rich-indexer uses async sqlx and the RPC thread pool awaits the DB future, concurrent flood requests saturate the pool, causing all RPC responses (including unrelated ones) to stall — sustained denial-of-service of the node's RPC layer.

---

### Likelihood Explanation

- The RPC is reachable by any local process by default; many node operators expose it publicly for wallet/DeFi infrastructure.
- No authentication is required.
- The call requires a single JSON-RPC request with a known mainnet code_hash and empty args.
- The asymmetry with `get_cells` (which has the guard) makes this an obvious oversight, not a design choice.
- The config documentation itself acknowledges the risk of omitting `request_limit`, yet `get_cells_capacity` ignores it regardless of configuration.

---

### Recommendation

Add the same `request_limit` guard at the top of `get_cells_capacity` that exists in `get_cells`. Additionally, implement a query-level timeout (e.g., SQLite `busy_timeout`, PostgreSQL `statement_timeout`, or a `tokio::time::timeout` wrapper) consistent with the `timeout_limit` used by the regular indexer's `TimeoutIterator`.

---

### Proof of Concept

```json
// Repeated calls with empty args prefix against a large mainnet SQLite DB:
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [{
    "script": {
      "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
      "hash_type": "type",
      "args": "0x"
    },
    "script_type": "lock",
    "script_search_mode": "prefix"
  }]
}
```

This generates:
```sql
SELECT CAST(SUM(output.capacity) AS BIGINT) AS total_capacity
FROM output
JOIN (SELECT script.id FROM script
      WHERE code_hash = ? AND hash_type = ?
        AND args >= x'' AND args < x'<upper>') AS query_script
  ON output.lock_script_id = query_script.id
WHERE output.is_spent = 0
```

No `LIMIT`, no timeout. Benchmark against a DB with 10M+ live cells to observe CPU saturation and RPC latency degradation. Compare with `get_cells` (same prefix, `limit=1`) which returns immediately due to `LIMIT 1` in SQL. [8](#0-7)

```rust
fn build_query_script_id_sql(
    db_driver: DBDriver,
    script_search_mode: &Option<IndexerSearchMode>,
    param_index: &mut usize,
) -> Result<String, Error> {
    let mut query_builder = SqlBuilder::select_from("script");
    query_builder
        .field("script.id")
        .and_where_eq("code_hash", format!("${}", param_index));
    *param_index += 1;
    query_builder.and_where_eq("hash_type", format!("${}", param_index));
    *param_index += 1;
    match script_search_mode {
        Some(IndexerSearchMode::Prefix) | None => {
            query_builder.and_where_ge("args", format!("${}", param_index));
            *param_index += 1;
            query_builder.and_where_lt("args", format!("${}", param_index));
            *param_index += 1;
        }
        Some(IndexerSearchMode::Exact) => {
            query_builder.and_where_eq("args", format!("${}", param_index));
            *param_index += 1;
        }
        Some(IndexerSearchMode::Partial) => {
            match db_driver {
                DBDriver::Postgres => {
                    query_builder.and_where(format!("args LIKE ${}", param_index));
                }
                DBDriver::Sqlite => {
                    query_builder.and_where(format!("instr(args, ${}) > 0", param_index));
                }
            }
            *param_index += 1;
        }
    }
    let sql_sub_query = query_builder
        .subquery()
        .map_err(|err| Error::DB(err.to_string()))?;
    Ok(sql_sub_query)
}
```
