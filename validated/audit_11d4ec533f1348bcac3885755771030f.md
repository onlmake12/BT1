### Title
Minimum Fee Rate Admission Check Uses Raw Byte Size Instead of Transaction Weight, Allowing High-Cycle Transactions to Enter the Mempool Below the Intended Fee Floor — (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using the transaction's raw serialized byte size as the weight, but the protocol's actual weight metric is `max(size_bytes, cycles × DEFAULT_BYTES_PER_CYCLES)`. For a transaction whose cycle cost dominates, the real weight can be an order of magnitude larger than the byte size. The code explicitly acknowledges this discrepancy in a comment but proceeds with the incorrect unit anyway. As a result, any unprivileged transaction sender can submit computationally expensive transactions to the mempool while paying a fee that is far below the node's configured `min_fee_rate` when that rate is interpreted in its documented unit of shannons per kilo-weight.

---

### Finding Description

**Root cause — mixed unit assumption in the fee admission gate**

`FeeRate` is defined and documented throughout the codebase as *shannons per kilo-weight*, where weight is:

```
weight = max(tx_size_bytes, cycles × DEFAULT_BYTES_PER_CYCLES)
``` [1](#0-0) 

`DEFAULT_BYTES_PER_CYCLES` is `≈ 0.000_170_571_4`, derived from `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`.

The admission gate in `check_tx_fee` computes the minimum required fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [2](#0-1) 

`FeeRate::fee(weight)` is `self.0 * weight / 1000`. [3](#0-2) 

By passing `tx_size` where `weight` is expected, the function implicitly assumes `size == weight`. The code's own comment admits this is wrong:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [4](#0-3) 

No subsequent weight-based fee check is performed after `verify_rtx` returns the actual cycle count. The transaction is admitted to the pool based solely on the size-based pre-check.

**Exploit path**

An attacker crafts a transaction with:
- Small serialized size (e.g., 200 bytes — achievable with a compact script cell dep and minimal inputs/outputs)
- A script that consumes close to `MAX_BLOCK_CYCLES` (e.g., 3,500,000 cycles — the two-in-two-out reference)

Concrete numbers with `min_fee_rate = 1000 shannons/kW`:

| Metric | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 3,500,000 |
| `weight` = `max(200, 3,500,000 × 0.000_170_571_4)` | **597 bytes** |
| Fee required by `check_tx_fee` | `1000 × 200 / 1000 = 200 shannons` |
| Fee required by correct weight check | `1000 × 597 / 1000 = 597 shannons` |
| Discount factor | **~3×** |

For a script consuming the full `MAX_BLOCK_CYCLES = 3,500,000,000`:

| Metric | Value |
|---|---|
| `weight` | `max(200, 3,500,000,000 × 0.000_170_571_4)` ≈ **597,000 bytes** |
| Fee required by `check_tx_fee` | `200 shannons` |
| Fee required by correct weight check | `597,000 shannons` |
| Discount factor | **~2,985×** |

The attacker submits such transactions via the public `send_transaction` RPC or P2P relay. Each transaction passes `check_tx_fee`, then triggers full script execution in `verify_rtx`, consuming significant CPU. The mempool fills with entries whose actual fee rate (shannons/kilo-weight) is far below `min_fee_rate`. [5](#0-4) 

---

### Impact Explanation

- **Mempool resource exhaustion**: An attacker can flood the mempool with high-cycle transactions at a tiny fraction of the intended fee cost, bypassing the economic spam-prevention mechanism.
- **CPU exhaustion on full nodes**: Each admitted transaction triggers full CKB-VM script execution (`verify_rtx`). With a ~3000× fee discount, the attacker can cause ~3000× more CPU work per shannon spent than the protocol intends to allow.
- **Fee market distortion**: Legitimate high-cycle transactions compete with artificially cheap attacker transactions in the mempool, degrading fee estimation accuracy and block assembly quality.
- **Miner revenue loss**: Miners may assemble blocks containing low-fee-rate transactions that displace higher-fee-rate ones, because the pool's scoring (`AncestorsScoreSortKey`) uses the correct weight, but admission does not. [6](#0-5) 

---

### Likelihood Explanation

- **Attacker profile**: Any unprivileged user with RPC or P2P access. No keys, no special role, no majority hashpower required.
- **Trigger**: Submit a transaction with a compact serialization but a script that loops for many cycles. This is trivially constructable using the always-success or any custom lock script.
- **Preconditions**: None beyond a standard node connection.
- **Existing mitigations**: The mempool has a total size cap (`max_tx_pool_size`) and a per-transaction size limit (`TRANSACTION_SIZE_LIMIT = 512 KB`), but neither limits cycle count at admission. The cycle limit is only enforced during block assembly, not mempool admission. [7](#0-6) 

---

### Recommendation

Replace the size-based pre-check with a weight-based check. Since cycles are not yet known at the time `check_tx_fee` is called (they are determined by `verify_rtx`), one of the following approaches should be used:

1. **Perform a second fee rate check after `verify_rtx`** using the actual cycles returned, computing `weight = get_transaction_weight(tx_size, actual_cycles)` and rejecting if `fee < min_fee_rate.fee(weight)`.

2. **Use a conservative upper-bound cycle estimate** (e.g., based on script size or a configurable per-byte cycle budget) for the pre-check, so the size-based gate is not trivially bypassed by high-cycle scripts.

The correct weight function is already available: [8](#0-7)

### Proof of Concept

1. Construct a CKB transaction with:
   - One input cell locked by a script that executes a tight loop consuming `N` cycles (e.g., `N = 3,500,000,000`, near `MAX_BLOCK_CYCLES`).
   - One output cell returning most capacity to self.
   - Serialized size ≈ 200 bytes.
   - Fee = `min_fee_rate × 200 / 1000` shannons (e.g., 200 shannons at default rate).

2. Submit via `send_transaction` RPC.

3. Observe: `check_tx_fee` passes (200 shannons ≥ 200 shannons required by size check). `verify_rtx` executes the script, consuming `N` cycles. Transaction enters the mempool.

4. Correct weight = `max(200, 3,500,000,000 × 0.000_170_571_4)` ≈ 597,000. Correct min fee = 597,000 shannons. Actual fee paid = 200 shannons — a **~2,985× shortfall**.

5. Repeat with many such transactions to exhaust mempool capacity and node CPU at negligible fee cost. [1](#0-0) [5](#0-4)

/// Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec.
/// The precision is set so that the difference between MAX_BLOCK_CYCLES * DEFAULT_BYTES_PER_CYCLES
/// and MAX_BLOCK_BYTES is less than 1.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"
)]
pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
