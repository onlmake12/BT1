Audit Report

## Title
Unbounded `GetBlocks` Replay Allows Unauthenticated Peer to Exhaust Node Resources — (`sync/src/synchronizer/get_blocks_process.rs`)

## Summary
`GetBlocksProcess::execute()` allocates a fresh `HashSet` on every invocation, limiting deduplication to within a single message only. The `Synchronizer` dispatches `GetBlocks` with no rate-limiting wrapper, unlike `Relayer` which applies an explicit `RateLimiter<(PeerIndex, u32)>` before any message dispatch. An unprivileged peer can flood the node with repeated `GetBlocks` messages, each triggering up to 32 unconditional async `SendBlock` task spawns, exhausting the async task pool, CPU, and outbound bandwidth.

## Finding Description
In `sync/src/synchronizer/get_blocks_process.rs`, a fresh `HashSet` is allocated on every call to `execute()`: [1](#0-0) 

This dedup only guards against duplicates *within* one message. Sending N messages each containing 32 distinct valid hashes bypasses it entirely — `StatusCode::RequestDuplicate` is never triggered across messages.

For each valid block hash passing the `BLOCK_VALID` status check, an async task is unconditionally spawned with no semaphore, backpressure, or task count limit: [2](#0-1) 

`Synchronizer::try_process` dispatches `GetBlocks` with no rate-limiting check — the `Synchronizer` struct carries no `rate_limiter` field at all: [3](#0-2) 

By contrast, `Relayer` defines and holds an explicit `RateLimiter<(PeerIndex, u32)>`: [4](#0-3) [5](#0-4) 

And applies it before any message dispatch: [6](#0-5) 

No equivalent guard exists in `Synchronizer`. The per-message cap of `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) and the length check against `MAX_HEADERS_LEN` (2,000) only bound work *per message*, not across repeated messages: [7](#0-6) [8](#0-7) 

## Impact Explanation
Each spawned task serializes a full block from storage and sends it over the network. With no rate limit, N messages yield N×32 unbounded `spawn` calls. This can exhaust the async task pool, saturate outbound bandwidth (amplification: ~1 KB request → MB-scale responses), and consume CPU for repeated block serialization and DB reads. This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires only a valid P2P connection and knowledge of 32 valid block hashes, which are trivially obtainable from any public block explorer or by syncing headers first. No PoW, no key, and no privileged role is needed. The attacker maintains a single TCP connection and sends small, well-formed messages in a tight loop. The `Relayer` protocol already demonstrates developer awareness that rate limiting is necessary — its absence in `Synchronizer` for `GetBlocks` is a concrete, exploitable gap.

## Recommendation
Add a per-peer rate limiter to `Synchronizer` for `GetBlocks` messages, mirroring the `RateLimiter<(PeerIndex, u32)>` pattern already used in `Relayer`. Additionally, consider maintaining a per-peer LRU cache of recently served block hashes to reject re-requests within a time window, and/or banning peers that exceed a configurable request threshold within a sliding window.

## Proof of Concept
1. Connect a peer to the node's sync port.
2. Obtain 32 valid block hashes from the chain (e.g., blocks 1–32).
3. In a tight loop, send N `SyncMessage::GetBlocks` messages each containing those same 32 hashes (all distinct within each message, so `StatusCode::RequestDuplicate` is never triggered).
4. Observe: N×32 `SendBlock` async tasks are spawned; the peer is never banned; node CPU and outbound bandwidth spike proportionally to N.

State-transition unit test sketch:
```rust
// Send N identical GetBlocks messages with 32 unique valid hashes
for _ in 0..N {
    let process = GetBlocksProcess::new(msg.as_reader(), &synchronizer, peer, &nc);
    assert_eq!(process.execute(), Status::ok()); // never banned
}
// Assert nc.sent_messages.len() == N * 32
```
