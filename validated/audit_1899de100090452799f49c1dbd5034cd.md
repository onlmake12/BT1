### Title
No User Refund Mechanism When Entropy Callback Permanently Fails — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In Pyth Entropy, fees are permanently credited to the provider and Pyth treasury at **request time**, before any callback is attempted. In the legacy V1 flow (`gasLimit10k == 0`), if the requester's callback always reverts, the provider's `revealWithCallback` transaction always reverts, leaving the request permanently active and the user's fee irrecoverable. No user-facing cancel or refund function exists anywhere in the contract.

---

### Finding Description

**Fee accounting at request time (permanent)**

In `requestHelper`, fees are credited immediately and unconditionally when the user calls `requestWithCallback`: [1](#0-0) 

`providerInfo.accruedFeesInWei` and `_state.accruedPythFeesInWei` are incremented in the same transaction as the user's request. These credits are not rolled back if the callback later fails.

**Legacy V1 path — callback called without try/catch**

When a provider has `defaultGasLimit == 0`, all requests for that provider get `req.gasLimit10k = 0`: [2](#0-1) 

In `revealWithCallback`, when `req.gasLimit10k == 0`, the code takes the `else` branch (line 661). It clears the request first, then calls the callback **without any error catching**: [3](#0-2) 

If `_entropyCallback` reverts, the entire `revealWithCallback` transaction reverts — including the `clearRequest` call on line 666. The request is restored to active state. The provider can never successfully fulfill the request because every attempt reverts.

**No user refund function**

The only withdrawal functions in the contract are:
- `withdraw` / `withdrawAsFeeManager` — provider-only
- `withdrawFee` — Pyth admin-only [4](#0-3) [5](#0-4) 

There is no function allowing a user to cancel an in-flight request or recover their fee.

The interface documentation also explicitly states excess value is not refunded: [6](#0-5) 

---

### Impact Explanation

A user who calls `requestWithCallback` against a legacy provider (`defaultGasLimit == 0`) and whose callback contract always reverts (e.g., due to a logic bug, an `assert`, or an out-of-gas condition at the callback's gas budget) will:

1. Permanently lose their paid fee (credited to provider/Pyth at request time).
2. Never receive their random number.
3. Have no on-chain mechanism to recover funds.

The provider also cannot fulfill the request — every `revealWithCallback` attempt reverts — so the sequence slot is consumed and the request is permanently stuck.

---

### Likelihood Explanation

- Legacy providers with `defaultGasLimit == 0` are still reachable; the code explicitly handles this case.
- Callback reverts are a documented, common failure mode (the protocol even has a dedicated debug guide for them).
- Any user whose callback contract has a bug (division by zero, failed `require`, insufficient balance, etc.) hits this path.
- The user has no way to anticipate this at request time, since the callback is only invoked in a separate provider transaction.

---

### Recommendation

Add a user-callable cancel/refund function that allows the original requester to reclaim their fee if the request has been active for longer than a configurable timeout, or if the request is in a permanently unfulfillable state. This mirrors the `dropMessage` refund pattern discussed in the Scroll report. At minimum, the new V2 `CALLBACK_FAILED` state should expose a user-facing refund path so that users are not permanently penalized for callback failures beyond their control.

---

### Proof of Concept

1. Deploy a provider with `defaultGasLimit == 0` (legacy mode).
2. Deploy a consumer contract whose `_entropyCallback` always reverts (e.g., `revert("broken")`).
3. Consumer calls `requestWithCallback{value: fee}(provider, userRandomNumber)`.
   - Fee is immediately credited: `providerInfo.accruedFeesInWei += providerFee` and `_state.accruedPythFeesInWei += remainder`.
4. Provider calls `revealWithCallback(provider, seqNum, userContrib, providerContrib)`.
   - `clearRequest` executes (line 666).
   - `_entropyCallback` is called (line 676) and reverts.
   - Entire transaction reverts; request is restored to active.
5. Repeat step 4 — always reverts. Request is permanently stuck.
6. Call `withdraw` as the user — reverts (`Unauthorized`). No refund path exists.
7. Provider can call `withdraw` and collect `providerFee` even though the service was never delivered.
