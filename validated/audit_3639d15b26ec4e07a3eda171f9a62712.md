Audit Report

## Title
Stale Pre-Eviction Stat Overwrite in `add_entry` Permanently Inflates `total_tx_size` — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures a pre-eviction snapshot of `self.total_tx_size + entry.size` before `check_and_record_ancestors` may evict entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`. The eviction correctly decrements `self.total_tx_size` in place, but lines 218–219 unconditionally overwrite the field with the stale snapshot, discarding all eviction decrements. The invariant `total_tx_size == Σ entry.size` is broken, causing `limit_size` to evict legitimate transactions that would otherwise fit within pool capacity.

## Finding Description
The exact sequence in `add_entry` (lines 200–221): [1](#0-0) 

- **Lines 210–211**: `updated_stat_for_add_tx` is a pure read (lines 711–728) that returns `self.total_tx_size + entry.size` without mutating state. At this point `self.total_tx_size = T`, so the snapshot is `T + entry.size`. [2](#0-1) 

- **Line 213**: `check_and_record_ancestors` enters the eviction branch (line 603) when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618), which calls `remove_entry` for each evicted entry. [3](#0-2) 

- **Line 247** inside `remove_entry`: `update_stat_for_remove_tx` mutates `self.total_tx_size` in place. After evicting entries of total size `X`, `self.total_tx_size = T - X`. [4](#0-3) 

- **Lines 218–219**: The stale snapshot `T + entry.size` is written back unconditionally, overwriting the correct post-eviction value `T - X + entry.size`. The inflation is `X` bytes — the total size of all evicted entries. [5](#0-4) 

`update_stat_for_remove_tx`'s only corrective path (`recompute_total_stat`) is triggered solely on underflow (lines 742–749), not on inflation, so the corrupted counter persists indefinitely. [6](#0-5) 

## Impact Explanation
`limit_size` in `tx-pool/src/pool.rs` loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts lowest-fee-rate entries. [7](#0-6) 

With `total_tx_size` inflated by `X` bytes, `limit_size` evicts `X` bytes worth of legitimate pending/proposed transactions that would otherwise have remained. Each attacker-triggered eviction event adds to the inflation; the effect accumulates across repeated triggers. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, because legitimate transactions are silently dropped from the mempool, degrading transaction propagation and block inclusion across targeted nodes.

## Likelihood Explanation
Any unprivileged peer or RPC caller can submit transactions. The trigger requires a chain of ~1,001 transactions where at least one ancestor is a cell-ref-parent (a pooled transaction referencing as `cell_dep` an output the new transaction spends as an input). The attacker pays fees for ~1,001 transactions but recovers most CKB when the chain is committed or expires. The inflation persists after the chain is gone, so a single successful trigger permanently degrades pool accounting on the targeted node until restart. The attack is repeatable and the inflation accumulates.

## Recommendation
Move the stat computation to after all evictions have completed. The overflow guard (`updated_stat_for_add_tx`) can still be called at the top for early rejection, but its return value must not be used to overwrite the field after evictions have occurred. Replace lines 218–219 with a direct in-place increment applied after `check_and_record_ancestors` returns:

```rust
// Remove the pre-eviction snapshot assignment entirely.
// After evictions have correctly decremented self.total_tx_size:
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .expect("overflow already checked above");
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .expect("overflow already checked above");
```

## Proof of Concept
1. Fill the pool with a chain `T1 → T2 → … → T1000` where `T1` also references cell `C` as a `cell_dep`.
2. Submit `T1001` spending cell `C` as an input with `T1000` as parent. Ancestor count = 1,001 > 1,000; `T1` is a cell-ref-parent; the branch at line 603 fires and `T1` (plus descendants) are evicted via `remove_entry_and_descendants`.
3. `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` by `size(T1)` during eviction.
4. Lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, re-adding `size(T1)`.
5. Query `tx_pool_info` RPC: `total_tx_size` exceeds the true sum of entry sizes by `size(T1)`.
6. Submit any new transaction: `limit_size` fires and evicts a legitimate transaction that would otherwise have fit.
7. Repeat steps 1–6 to accumulate inflation and progressively drain the pool of legitimate transactions.

A unit test can assert the invariant `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum::<usize>()` immediately after `add_entry` returns in the eviction branch, which will fail against the current code.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
