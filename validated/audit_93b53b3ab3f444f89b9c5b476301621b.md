### Title
Unbounded DB Read Amplification and Bandwidth Exhaustion via Unauthenticated `GetTransactionsProof` — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

### Summary
The light client protocol server's `GetTransactionsProofProcess::execute()` accepts up to 1000 tx_hashes per message with no per-peer rate limiting. An unprivileged remote peer can flood the server with messages containing 1000 unknown hashes, each triggering 1000 DB reads and a ~32 KB response, with no throttle to bound the aggregate cost.

### Finding Description
`LightClientProtocol::try_process` dispatches `GetTransactionsProof` messages directly to `GetTransactionsProofProcess::execute()` without any rate-limiting check. Inside `execute()`, the only guards are an empty-check and an upper-bound check (`> 1000`). A message with exactly 1000 hashes passes both. The handler then calls `snapshot.get_transaction_info(tx_hash)` for every hash in a `partition` loop. If all hashes are unknown (trivially achieved with random 32-byte values), all 1000 DB reads return `None`, `found` is empty, and the server still constructs and sends a `SendTransactionsProofV1` response containing a 1000-entry `missing_tx_hashes` Byte32Vec (~32 KB). The `Relayer` protocol has an explicit `RateLimiter` keyed by `(peer, message_type)` capped at 30 req/s; the light client protocol has no equivalent.

### Impact Explanation
Each request costs the server 1000 synchronous DB reads plus a ~32 KB outbound response. A single attacker sending 100 such requests per second imposes 100,000 DB read IOPS and ~3.2 MB/s of outbound bandwidth per attacking peer. Multiple peers multiply this linearly. This can saturate disk I/O and network egress on a full node running the light client server, causing denial of service to legitimate light clients and degrading full-node operation.

### Likelihood Explanation
The attack requires only a valid main-chain block hash (publicly available) and the ability to open a P2P connection. No key, stake, or privileged role is needed. The protocol is enabled on production full nodes that serve light clients. The attack is trivially scriptable.

### Recommendation
Add a per-peer rate limiter to `LightClientProtocol`, mirroring the pattern already used in `Relayer`:
- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
- In `try_process`, check `rate_limiter.check_key(&(peer_index, message.item_id()))` before dispatching, returning `StatusCode::TooManyRequests` on failure.
- Additionally, consider a per-request cost cap (e.g., limit `tx_hashes` to a lower value such as 100) to reduce the per-request DB amplification factor.

### Proof of Concept
```
1. Obtain any valid tip block hash H from the CKB chain (public RPC).
2. Connect to a CKB full node with light client protocol enabled.
3. In a loop at maximum speed:
   a. Build GetTransactionsProof { last_hash: H, tx_hashes: [rand_32_bytes(); 1000] }
   b. Send over the P2P connection.
4. Observe: server performs 1000 get_transaction_info DB reads per message,
   responds with ~32 KB SendTransactionsProofV1 per message,
   no TooManyRequests rejection is ever returned.
5. Measure DB read IOPS and outbound bandwidth — both scale linearly with
   request rate, unbounded.
```
