### Title
`CapacityVerifier::valid_dao_withdraw_transaction` Incorrectly Skips Capacity Overflow Check for DAO Phase-1 (Prepare) Transactions, Enabling Capacity Inflation — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` for **any** transaction that has a DAO-typed input, including phase-1 (deposit→prepare) transactions. This causes `CapacityVerifier::verify()` to unconditionally skip the `OutputsSumOverflow` check for those transactions. Because the DAO type script only enforces capacity equality on the DAO cells themselves and not on co-mingled non-DAO cells, an attacker can include non-DAO inputs and outputs in the same phase-1 transaction and inflate the non-DAO output capacity beyond the non-DAO input capacity — creating CKB from nothing.

---

### Finding Description

`CapacityVerifier::verify()` contains the following guard:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
    // OutputsSumOverflow check
}
```

The comment reads:
> "DAO withdraw transaction is verified via the type script of DAO cells"

`valid_dao_withdraw_transaction()` is:

```rust
fn valid_dao_withdraw_transaction(&self) -> bool {
    self.resolved_transaction
        .resolved_inputs
        .iter()
        .any(|cell_meta| cell_uses_dao_type_script(&cell_meta.cell_output, &self.dao_type_hash))
}
```

This predicate fires for **any** input that carries the DAO type script — including a **phase-1 (prepare)** input, which is a deposit cell whose 8-byte data field is all zeros (`deposited_block_number == 0`). The distinction between phase-1 and phase-2 is made in `DaoCalculator::transaction_maximum_withdraw` via the `deposited_block_number > 0` branch, but `valid_dao_withdraw_transaction` makes no such distinction.

For a **phase-2 (withdraw)** transaction the skip is correct: the DAO type script enforces the maximum withdrawal amount (which legitimately exceeds input capacity due to accrued interest).

For a **phase-1 (prepare)** transaction the skip is **incorrect**: the DAO type script only verifies that each DAO output's capacity equals the corresponding DAO input's capacity (and that lock-script sizes match). It does **not** verify the capacity balance of non-DAO cells in the same transaction. With the `OutputsSumOverflow` check skipped, no other verifier enforces that constraint either.

**Attack scenario:**

1. Attacker holds a DAO deposit cell `D` (capacity = 1000 CKB) and a non-DAO cell `N` (capacity = 10 CKB).
2. Attacker constructs a phase-1 transaction:
   - Inputs: `D` (DAO, 1000 CKB) + `N` (non-DAO, 10 CKB)
   - Outputs: `D'` (DAO prepare cell, 1000 CKB — satisfies DAO type script) + `N'` (non-DAO, 10 + X CKB, X > 0)
3. `valid_dao_withdraw_transaction()` returns `true` because `D` carries the DAO type script → `OutputsSumOverflow` check is skipped.
4. The DAO type script runs and verifies `D'`.capacity == `D`.capacity ✓.
5. No verifier checks that `N'`.capacity ≤ `N`.capacity.
6. The transaction is accepted; the attacker has created X CKB from nothing.

---

### Impact Explanation

**Impact: High.**

A successful exploit inflates the total CKB supply. Every block that commits such a transaction permanently increases the live-cell capacity sum beyond the protocol-defined issuance schedule. This corrupts the DAO accumulation-rate (`AR`) field used to compute interest for all depositors, distorts miner reward calculations, and undermines the economic security model of the chain. The inflation is irreversible once committed.

---

### Likelihood Explanation

**Likelihood: Medium.**

Any unprivileged transaction sender who holds at least one DAO deposit cell (obtainable by anyone) can craft and submit such a transaction. No special privileges, keys, or majority hash power are required. The only prerequisite is having previously deposited into the DAO, which is a normal user action.

---

### Recommendation

`valid_dao_withdraw_transaction()` must distinguish phase-1 from phase-2 DAO inputs. Only phase-2 inputs (where `deposited_block_number > 0` in the cell data) justify skipping the `OutputsSumOverflow` check. The function should be updated to return `true` only when at least one input is a **phase-2** DAO cell (i.e., its 8-byte data encodes a non-zero block number), mirroring the logic already present in `DaoCalculator::transaction_maximum_withdraw`.

    pub fn verify(&self) -> Result<(), Error> {
        // skip OutputsSumOverflow verification for resolved cellbase and DAO
        // withdraw transactions.
        // cellbase's outputs are verified by RewardVerifier
        // DAO withdraw transaction is verified via the type script of DAO cells
        if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
            let inputs_sum = self.resolved_transaction.inputs_capacity()?;
            let outputs_sum = self.resolved_transaction.outputs_capacity()?;

            if inputs_sum < outputs_sum {
                return Err((TransactionError::OutputsSumOverflow {
                    inputs_sum,
                    outputs_sum,
                })
                .into());
            }
        }
