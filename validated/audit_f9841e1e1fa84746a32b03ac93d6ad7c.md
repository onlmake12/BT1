### Title
Unprotected `initialize` Function Allows Front-Running Takeover of Proxy Contracts - (File: `target_chains/ethereum/contracts/contracts/entropy/EntropyUpgradable.sol`)

---

### Summary

`EntropyUpgradable.initialize()` is a `public` function protected only by OpenZeppelin's `initializer` modifier, which prevents re-initialization but does not restrict *who* can call it first. If the proxy is deployed without initialization data in the same transaction, an attacker can front-run the deployer's `initialize` call and become the owner and admin of the contract.

---

### Finding Description

`EntropyUpgradable.initialize()` is declared `public` with only the `initializer` modifier:

```solidity
function initialize(
    address owner,
    address admin,
    uint128 pythFeeInWei,
    address defaultProvider,
    bool prefillRequestStorage
) public initializer {
    ...
    __Ownable_init();
    __UUPSUpgradeable_init();
    Entropy._initialize(admin, pythFeeInWei, defaultProvider, prefillRequestStorage);
    _transferOwnership(owner);
}
``` [1](#0-0) 

The `initializer` modifier only prevents the function from being called **more than once**. It does not restrict the caller. The proxy's storage slot for `_initialized` starts at `0` (uninitialized). Any address that calls `initialize` before the deployer does will succeed, setting themselves as `owner` and `admin`.

The constructor uses the older `constructor() initializer {}` pattern rather than `_disableInitializers()`:

```solidity
constructor() initializer {}
``` [2](#0-1) 

This sets `_initialized = 1` on the **implementation** contract only. The **proxy** contract's storage remains at `_initialized = 0` until `initialize()` is called through the proxy. `_disableInitializers()` (used correctly in `SchedulerUpgradeable`) sets `_initialized = type(uint8).max`, blocking all future `reinitializer` calls on the implementation as well. [3](#0-2) 

The same pattern (`constructor() initializer {}`) is present in `EchoUpgradeable`, `ExecutorUpgradable`, and `PythUpgradable`: [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Impact Explanation

For `EntropyUpgradable`, a successful front-run gives the attacker:

1. **Full ownership** (`onlyOwner`) — ability to call `_authorizeUpgrade` and upgrade the proxy to a malicious implementation.
2. **Admin role** — ability to set `pythFeeInWei`, `defaultProvider`, and other governance parameters.
3. **Ability to drain accrued fees** — via `withdrawFees` (admin-gated) or by upgrading to a draining implementation. [7](#0-6) 

The deployer's subsequent `initialize` call would revert with "already initialized," forcing a full redeployment. If the team does not notice the hijack and the attacker leaves the contract appearing functional, user funds (accrued fees) and protocol governance are permanently compromised.

---

### Likelihood Explanation

- **Requires non-atomic deployment**: The attack window exists only if the proxy is deployed in one transaction and `initialize` is called in a separate transaction. If the deployment script uses `ERC1967Proxy(impl, initData)` (atomic), the window does not exist.
- **Mempool monitoring**: An attacker must watch for the proxy deployment transaction and submit a front-running `initialize` call with higher gas.
- **Likelihood is low but non-zero**: This exact attack class has been executed against other protocols. The impact is critical enough to warrant mitigation regardless of likelihood.

---

### Recommendation

1. **Use `_disableInitializers()` in all implementation constructors** (already done correctly in `SchedulerUpgradeable` and `PythLazer`):

```solidity
/// @custom:oz-upgrades-unsafe-allow constructor
constructor() {
    _disableInitializers();
}
```

2. **Deploy proxies atomically**: Always pass initialization calldata to the proxy constructor so that `initialize` is called in the same transaction as proxy deployment, eliminating the front-running window entirely.

3. **Consistent pattern**: Align `EntropyUpgradable`, `EchoUpgradeable`, `ExecutorUpgradable`, and `PythUpgradable` with the `_disableInitializers()` pattern already used in `SchedulerUpgradeable`.

---

### Proof of Concept

```
1. Deployer broadcasts: deploy ERC1967Proxy(EntropyUpgradable_impl, "")
   → Proxy deployed, _initialized = 0 in proxy storage

2. Attacker sees tx in mempool, broadcasts with higher gas:
   EntropyUpgradable(proxy).initialize(
       attacker,   // owner
       attacker,   // admin
       0,          // pythFeeInWei
       attacker,   // defaultProvider
       false
   )
   → Succeeds. Attacker is now owner and admin.

3. Deployer's initialize() call reverts: "Initializable: contract is already initialized"

4. Attacker calls upgradeTo(maliciousImpl) via onlyOwner → drains all accrued fees.
``` [1](#0-0)
