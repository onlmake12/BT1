### Title
`MAX_DEPOSIT_LIMIT` Bypass for Permanent Subscriptions via Repeated `addFunds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` enforces the `MAX_DEPOSIT_LIMIT` cap for permanent subscriptions by checking only the **per-call** `msg.value`, not the **cumulative** `status.balanceInWei`. Because `addFunds` also has no `onlyManager` access control, any unprivileged caller can repeatedly invoke it with amounts just at or below `MAX_DEPOSIT_LIMIT`, accumulating a balance that far exceeds the intended cap. This is the direct analog of the prePO `userWithdrawLimitPerPeriod` bypass: a per-operation accounting check that fails to track the running total.

---

### Finding Description

`createSubscription` correctly gates the initial deposit:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [1](#0-0) 

`addFunds` repeats the same per-call check but never compares the **post-deposit cumulative balance** against the cap:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [2](#0-1) 

The function carries no `onlyManager` modifier, so **any address** may call it on any active subscription:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
``` [3](#0-2) 

Compare with `withdrawFunds`, which correctly gates on `onlyManager`:

```solidity
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount
) external override onlyManager(subscriptionId) {
``` [4](#0-3) 

Permanent subscriptions explicitly prohibit any withdrawal:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [5](#0-4) 

The combination means ETH deposited beyond `MAX_DEPOSIT_LIMIT` is **permanently and irrecoverably locked** in the contract.

---

### Impact Explanation

`MAX_DEPOSIT_LIMIT` exists to bound how much ETH can be irreversibly committed to a single permanent subscription. By bypassing it, an attacker (or the subscription owner) can:

1. Lock an **arbitrarily large** amount of ETH in a permanent subscription with no withdrawal path.
2. Inflate the contract's ETH holdings beyond the intended safety ceiling, increasing the value at risk from any future contract vulnerability.
3. Grief third parties: because `addFunds` is permissionless, an attacker can force-deposit ETH into **any** active permanent subscription owned by someone else, permanently locking the attacker's ETH in a subscription they do not control — a one-way griefing vector that also inflates the contract's balance.

---

### Likelihood Explanation

- No privileged role is required; any EOA can call `addFunds`.
- The bypass requires only repeated transactions, each with `msg.value ≤ MAX_DEPOSIT_LIMIT`.
- The test suite itself confirms the permissionless nature of `addFunds`:

```solidity
// Anyone can add funds (not just manager)
vm.prank(address(0x123));
scheduler.addFunds{value: extraFunds}(subscriptionId);
``` [6](#0-5) 

---

### Recommendation

In `addFunds`, replace the per-call check with a cumulative check:

```solidity
// Before (vulnerable):
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;

// After (fixed):
status.balanceInWei += msg.value;
if (params.isPermanent && status.balanceInWei > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

Additionally, consider adding `onlyManager` to `addFunds` for permanent subscriptions to prevent third-party griefing deposits.

---

### Proof of Concept

1. Deploy `Scheduler` with `MAX_DEPOSIT_LIMIT = X`.
2. Call `createSubscription{value: X}(params)` where `params.isPermanent = true`. Succeeds; `balanceInWei = X`.
3. Call `addFunds{value: X}(subscriptionId)` — `msg.value == X`, which is **not** `> MAX_DEPOSIT_LIMIT`, so the check passes. `balanceInWei = 2X`.
4. Repeat step 3 `N` times. Final `balanceInWei = (N+1) * X`, all permanently locked, with no revert at any step. [7](#0-6)

    /// BALANCE MANAGEMENT

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```
