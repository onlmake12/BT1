### Title
Provider Self-Request Enables Full Random Number Manipulation - (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` does not check that `msg.sender != provider`. A registered provider can call `requestWithCallback` / `requestV2` using their own address as the `provider` argument. Because the provider knows their entire hash chain in advance, they also know `providerContribution` for the assigned sequence number before committing to `userContribution`. This collapses the two-party security model into a single party, allowing the provider to choose any desired random number output.

---

### Finding Description

The Entropy protocol's security guarantee is stated explicitly in the contract header:

> "This protocol has the property that the result is random as long as either A or B are honest."

The two parties are the **provider** (commits to a hash chain up-front) and the **user/requester** (commits to `userContribution` at request time). The guarantee holds only when these are distinct, mutually distrusting parties.

`requestHelper` is the internal entry point for all request variants: [1](#0-0) 

The only provider-existence check is:

```solidity
if (_state.providers[provider].sequenceNumber == 0)
    revert EntropyErrors.NoSuchProvider();
```

There is **no check** that `msg.sender != provider`. The requester field is set unconditionally: [2](#0-1) 

The commitment stored is: [3](#0-2) 

When `msg.sender == provider`, the provider already knows `providerInfo.currentCommitment` (their own hash chain value) **before** choosing `userCommitment`. They can therefore iterate over candidate `userContribution` values off-chain, compute `combineRandomValues(userContribution, providerContribution, 0)` for each (blockhash is `false` in `requestV2`/`requestWithCallback`), and select the one that produces the desired outcome. They then commit to that `userContribution` on-chain. [4](#0-3) 

At reveal time, `revealHelper` validates both contributions against the stored commitment: [5](#0-4) 

This validation passes because the provider chose `userContribution` to be consistent with the commitment they submitted. The manipulation is cryptographically valid from the contract's perspective.

---

### Impact Explanation

Any consumer contract that uses a provider controlled by the attacker receives a fully attacker-chosen random number. Applications built on Pyth Entropy — lotteries, NFT trait generation, on-chain games, randomized liquidation ordering — are completely compromised. The attacker can guarantee any outcome they desire on every single request they make to themselves.

The provider fee accrues to `providerInfo.accruedFeesInWei` (the attacker's own balance), so the net cost to the attacker is only the Pyth protocol flat fee (`_state.pythFeeInWei`), making repeated exploitation cheap. [6](#0-5) 

---

### Likelihood Explanation

Provider registration is **permissionless** — anyone can call `register()` with any commitment and fee: [7](#0-6) 

The attacker needs no privileged access, no leaked keys, and no governance majority. The only prerequisite is registering as a provider and paying the Pyth flat fee per request. The attack is fully on-chain and requires no off-chain collusion. Likelihood is **high** for any application that allows users to choose their own provider.

---

### Recommendation

Add a self-request guard in `requestHelper`:

```solidity
if (msg.sender == provider) revert EntropyErrors.SelfRequest();
```

This mirrors the existing `AuctionSelfBid` pattern from the referenced report and directly enforces the two-party assumption the protocol's security model depends on. [8](#0-7) 

---

### Proof of Concept

1. Attacker deploys a contract `AttackerConsumer` implementing `IEntropyConsumer._entropyCallback`.
2. Attacker calls `entropy.register(fee, commitment, metadata, chainLength, uri)` — registers themselves as provider `P`. They hold the full hash chain locally.
3. For sequence number `N` (the next to be assigned), attacker reads `providerInfo.currentCommitment` = `x_{N-1}` from chain state.
4. Off-chain, attacker iterates `userContribution` values and computes `target = combineRandomValues(userContribution, x_N, 0)` where `x_N` is the known hash-chain value for slot `N`. They select `userContribution = U` such that `target` is the desired outcome (e.g., winning lottery ticket index).
5. Attacker calls `entropy.requestV2{value: fee}(P, U, 0)` from `AttackerConsumer`. Sequence number `N` is assigned. Commitment stored = `keccak256(keccak256(U), x_{N-1})`.
6. Attacker calls `entropy.revealWithCallback(P, N, U, x_N)`. `revealHelper` validates `keccak256(x_N) == x_{N-1}` ✓ and `keccak256(keccak256(U), keccak256(x_N)) == commitment` ✓.
7. `AttackerConsumer._entropyCallback` receives the pre-chosen `target` as the random number.
8. Attacker uses `target` to claim the lottery prize / favorable NFT trait / game outcome. [9](#0-8)

    function revealHelper(
        EntropyStructsV2.Request storage req,
        bytes32 userContribution,
        bytes32 providerContribution
    ) internal returns (bytes32 randomNumber, bytes32 blockHash) {
        bytes32 providerCommitment = constructProviderCommitment(
            req.numHashes,
            providerContribution
        );
        bytes32 userCommitment = constructUserCommitment(userContribution);
        if (
            keccak256(bytes.concat(userCommitment, providerCommitment)) !=
            req.commitment
        ) revert EntropyErrors.IncorrectRevelation();

        blockHash = bytes32(uint256(0));
        if (req.useBlockhash) {
            bytes32 _blockHash = blockhash(req.blockNumber);

            // The `blockhash` function will return zero if the req.blockNumber is equal to the current
            // block number, or if it is not within the 256 most recent blocks. This allows the user to
            // select between two random numbers by executing the reveal function in the same block as the
            // request, or after 256 blocks. This gives each user two chances to get a favorable result on
            // each request.
            // Revert this transaction for when the blockHash is 0;
            if (_blockHash == bytes32(uint256(0)))
                revert EntropyErrors.BlockhashUnavailable();

            blockHash = _blockHash;
        }

        randomNumber = combineRandomValues(
            userContribution,
            providerContribution,
            blockHash
        );

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            req.provider
        ];
        if (providerInfo.currentCommitmentSequenceNumber < req.sequenceNumber) {
            providerInfo.currentCommitmentSequenceNumber = req.sequenceNumber;
            providerInfo.currentCommitment = providerContribution;
        }
