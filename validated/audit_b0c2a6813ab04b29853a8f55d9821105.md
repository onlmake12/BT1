Audit Report

## Title
Unbounded JSON-RPC Batch Request Bypasses Batch-Size Guard by Default — (File: rpc/src/server.rs)

## Summary
`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only populated when `rpc_batch_limit` is explicitly set in `ckb.toml`. The default shipped config leaves the field commented out, so the `OnceLock` is never written and the `if let Some(batch_size)` guard in `handle_jsonrpc` is never entered. Any RPC caller can POST a single HTTP request containing an arbitrarily large JSON-RPC batch array and force the server to process every call with no cap, causing RPC unresponsiveness for the duration of the batch.

## Finding Description
`rpc_batch_limit` is declared as `Option<usize>` with no `#[serde(default)]` override; serde defaults absent `Option` fields to `None` (`util/app-config/src/configs/rpc.rs`, L44). In `RpcServer::new`, `JSONRPC_BATCH_LIMIT` is only initialized inside `if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit` (`rpc/src/server.rs`, L53–55); when the config field is `None`, the `OnceLock` is never written. In `handle_jsonrpc`, the guard at L275–282 is `if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get() && calls.len() > *batch_size`; when the `OnceLock` is unset, `get()` returns `None`, the guard is skipped entirely, and `stream::iter(calls).then(...)` dispatches every call in the batch unconditionally. The default `resource/ckb.toml` (L205–208) explicitly documents "By default, there is no limitation on the size of batch request size" and leaves `rpc_batch_limit` commented out. The only constraint is the 30-second `TimeoutLayer` and `max_request_body_size`; within those bounds, a caller can submit tens of thousands of calls per batch, saturating the RPC server for the full timeout window and making it unresponsive to concurrent legitimate requests.

## Impact Explanation
**Note (0–500 points) — Any local RPC API crash.** A single unauthenticated HTTP POST with a large batch array saturates the RPC handler for up to 30 seconds, causing concurrent legitimate RPC requests to time out or queue indefinitely. This constitutes a local RPC API crash/unresponsiveness matching the "Note" impact class. The submitted PoC uses only `ping` calls; the OOM-kill path via expensive calls (`get_block` verbosity=2) is described but not concretely demonstrated, so severity is not upgraded to "High."

## Likelihood Explanation
No authentication, no special privilege, and no prior knowledge beyond the public JSON-RPC API is required. Every operator using the default generated config without explicitly adding `rpc_batch_limit` is affected. The RPC port is localhost-bound by default, but production deployments routinely expose it to internal networks or proxies. The attack is trivially repeatable with a single HTTP client.

## Recommendation
Set a safe non-`None` default for `rpc_batch_limit` in `Config` (e.g., `2000`) so the guard is active out of the box without operator action. The `OnceLock` initialization in `RpcServer::new` should then always be populated regardless of whether the operator overrides the value.

## Proof of Concept
```python
import json, requests

ENDPOINT = "http://127.0.0.1:8114"

batch = [{"jsonrpc": "2.0", "method": "ping", "id": i} for i in range(50_000)]

# Single POST; server processes all calls with no batch-size limit
resp = requests.post(ENDPOINT, json=batch, timeout=35)
# Concurrent legitimate RPC requests time out during processing
```
Confirm by issuing a concurrent `get_tip_block_number` call during batch processing and observing it stall until the batch completes or the 30-second timeout fires.
