### Title
Hardcoded `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` Constant Permanently Freezes Scheduler Price Updates When Pyth Network Is Delayed — (File: `target_chains/ethereum/pulse_sdk/solidity/SchedulerConstants.sol`)

---

### Summary

`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is declared as a Solidity `constant` (1 hour) in `SchedulerConstants.sol` and is used in `Scheduler.sol`'s `_validateShouldUpdatePrices()` to gate every `updatePriceFeeds()` call. Because it is a compile-time constant with no admin setter, no privileged role can relax it. If the Pyth network or Hermes relay experiences a disruption lasting more than one hour, every keeper call to `updatePriceFeeds()` reverts with `TimestampTooOld`, permanently freezing all active Scheduler subscriptions until a full contract upgrade is executed.

---

### Finding Description

`SchedulerConstants.sol` declares:

```solidity
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
``` [1](#0-0) 

`Scheduler.sol` inherits `SchedulerConstants` and applies this constant inside `_validateShouldUpdatePrices()`:

```solidity
uint256 minAllowedTimestamp = (block.timestamp >
    PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;

if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(
        updateTimestamp,
        block.timestamp
    );
}
``` [2](#0-1) 

This check runs on every `updatePriceFeeds()` call before any price is stored. The `updateTimestamp` is the maximum `publishTime` across all price feeds in the submitted update data. [3](#0-2) 

`SchedulerGovernance.sol` exposes admin-settable parameters (`setSingleUpdateKeeperFeeInWei`, `setMinimumBalancePerFeed`) but provides **no setter for `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`**. [4](#0-3) 

Because the value is a Solidity `constant`, it is baked into the bytecode at compile time and cannot be changed by any on-chain governance action short of a full proxy upgrade.

---

### Impact Explanation

When the Pyth network or Hermes relay is disrupted for more than one hour (e.g., due to network congestion, a Pythnet outage, or a Wormhole guardian set issue), all price feed `publishTime` values in the update data will be older than `block.timestamp - 1 hours`. Every keeper call to `updatePriceFeeds()` will revert unconditionally. All active Scheduler subscriptions are frozen: no price updates can be stored, and any downstream protocol (DeFi lending, perpetuals, options) relying on Scheduler for push-based price delivery will operate on stale prices for the entire duration of the outage plus the time required to execute a contract upgrade. This can cause incorrect liquidations, mispriced trades, and loss of user funds in integrated protocols.

---

### Likelihood Explanation

Pyth network disruptions exceeding one hour are uncommon but realistic. Historical incidents on Wormhole and Pythnet have caused multi-hour outages. Additionally, a targeted network-layer disruption of Hermes (the off-chain relay) for more than one hour would produce the same effect without requiring any on-chain attack. The 1-hour window is tight relative to the time required to detect, diagnose, and respond to an outage. Unlike the analogous `MAX_SLIPPAGE` case, there is no emergency lever available to the admin to temporarily widen the window and unblock the subscriptions.

---

### Recommendation

Convert `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` from a compile-time `constant` to a mutable state variable with an admin-only setter, analogous to the existing `setSingleUpdateKeeperFeeInWei` and `setMinimumBalancePerFeed` governance functions. A reasonable default of 1 hour should be preserved, but the admin (or owner) should be able to temporarily widen the window to recover from a network disruption and then restore it afterward — exactly the pattern the referenced audit team adopted for `maxSlippage`.

```solidity
// In SchedulerState: replace constant with state variable
uint64 public pastTimestampMaxValidityPeriod = 1 hours;

// In SchedulerGovernance: add setter
function setPastTimestampMaxValidityPeriod(uint64 newPeriod) external {
    _authorizeAdminAction();
    pastTimestampMaxValidityPeriod = newPeriod;
}
```

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with `updateOnHeartbeat = true, heartbeatSeconds = 60`.
2. Advance `block.timestamp` by 61 minutes (`vm.warp(block.timestamp + 3660)`).
3. Obtain a valid Pyth update whose `publishTime` is `block.timestamp - 3660` (i.e., 61 minutes ago — the most recent available data).
4. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` as an unprivileged keeper.
5. The call reverts with `SchedulerErrors.TimestampTooOld` because `updateTimestamp (now - 3660) < minAllowedTimestamp (now - 3600)`.
6. No admin function exists to widen the window; the subscription remains permanently frozen until a proxy upgrade is executed. [5](#0-4)
