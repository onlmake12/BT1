### Title
Attacker-Controlled Ping Manipulation Enables Deterministic Eviction of Legitimate Inbound Peers — (`network/src/peer_registry.rs`)

---

### Summary

The eviction-protection logic in `try_evict_inbound_peer` claims to protect peers based on "characteristics that an attacker [is] hard to simulate or manipulate," but all three protection criteria — `ping_rtt`, `last_ping_protocol_message_received_at`, and `connected_time` — are trivially manipulable by an unprivileged attacker via standard P2P messages. A newly connected legitimate peer has `None` for both ping fields (mapping to `u64::MAX` in comparisons, the worst possible value) and the shortest `connected_time`, causing it to be deterministically placed into the eviction candidate pool and evicted.

---

### Finding Description

**`sort_then_drop` mechanics** — the function sorts ascending by the comparator, then truncates to keep the first `list.len() - n` elements, effectively *protecting* (dropping from candidates) the last `n` elements: [1](#0-0) 

**Step 1 — Ping RTT protection (protect 8 lowest RTT):** [2](#0-1) 

The comparator is `peer2_ping.cmp(&peer1_ping)` — descending order. After sorting, the last 8 (lowest RTT = best) are protected; the first elements (highest RTT = worst) remain as candidates. A newly connected legitimate peer has `ping_rtt = None`, which maps to `u64::MAX` — the worst possible value — placing it at the front of the sorted list and keeping it as a candidate.

**Step 2 — Last ping message protection (protect 8 most recent):** [3](#0-2) 

Same pattern: `peer2_last_message.cmp(&peer1_last_message)` is descending. A newly connected legitimate peer has `last_ping_protocol_message_received_at = None` → `u64::MAX` duration (oldest possible), keeping it as a candidate.

**Step 3 — Connected time protection (protect half with longest connection):** [4](#0-3) 

`peer2.connected_time.cmp(&peer1.connected_time)` is descending (newest first). The last `protect_peers` elements (oldest connections) are protected. The legitimate peer, having just connected, is the newest and remains as a candidate.

**How the attacker manipulates the fields:**

`ping_rtt` is set in `pong_received` as `now.saturating_duration_since(last_ping)`, where `last_ping` is the time the *node* sent the ping. The attacker responds to pings immediately, achieving sub-second RTT → `as_secs() = 0`: [5](#0-4) 

`last_ping_protocol_message_received_at` is set in `ping_received` to `Instant::now()` whenever the node receives a Ping message from a peer. The attacker sends Ping messages at will: [6](#0-5) 

A newly connected legitimate peer initializes both fields to `None`: [7](#0-6) 

**Concrete trace with `max_inbound = 10`, 9 attacker peers + 1 legitimate peer:**

| Step | Candidates before | Action | Candidates after |
|---|---|---|---|
| Start | 10 (9 attacker + 1 legit) | — | 10 |
| RTT sort (protect 8) | 10 | legit has `u64::MAX`, 8 attacker peers protected | 2 (legit + 1 attacker) |
| Last-ping sort (protect 8) | 2 | `2 ≤ 8`, nothing dropped | 2 |
| Connected-time sort (protect 1) | 2 | legit is newest → stays; attacker peer is oldest → protected | 1 (legit only) |
| Group eviction | 1 | only legit peer in pool | **legit evicted** |

The result is deterministic: the legitimate peer is always evicted.

---

### Impact Explanation

An attacker who fills `max_inbound - 1` slots before a legitimate peer connects can ensure that legitimate peer is evicted every time it reconnects. By repeating this cycle, the attacker can eclipse the victim node — isolating it from honest peers. Eclipse attacks on CKB nodes enable double-spend attacks against merchants and mining pool isolation.

---

### Likelihood Explanation

The attacker needs multiple IP addresses to fill inbound slots (or exploit any absence of per-IP connection limits). Once slots are filled, the attack is deterministic and requires only standard P2P Ping messages — no privileged access, no PoW, no key material. The attack is repeatable: each time the legitimate peer reconnects, it is evicted again.

---

### Recommendation

1. **Replace `ping_rtt.as_secs()` with finer granularity** (e.g., `as_millis()`) so sub-second RTT differences are meaningful.
2. **Do not use `last_ping_protocol_message_received_at` as a protection criterion** — it is directly writable by any peer sending a Ping message. Use a criterion the attacker cannot control, such as the node's own outbound ping timestamps.
3. **Protect newly connected peers** — add a grace period or a separate protection bucket for peers connected within the last N seconds, preventing them from being eviction candidates immediately after connecting.
4. **Consider Bitcoin Core's approach**: protect peers that have sent novel transactions or blocks, which requires real network participation rather than trivial protocol messages.

---

### Proof of Concept

```
1. Configure victim node with max_inbound = 10.
2. Attacker opens 9 inbound connections from distinct IPs.
3. For each attacker connection:
   a. Respond to all Ping messages immediately (minimizes ping_rtt to 0 as_secs()).
   b. Send Ping messages to the victim every 100ms (keeps last_ping_protocol_message_received_at fresh).
4. Wait for a legitimate peer to connect (now at max_inbound = 10).
5. Attacker opens a 10th inbound connection → triggers try_evict_inbound_peer.
6. Assert: legitimate peer is disconnected; all 9 original attacker peers survive.
7. Repeat from step 4 — legitimate peer can never maintain a connection.
```

The eviction is deterministic because the legitimate peer has `ping_rtt = None` (→ `u64::MAX`), `last_ping_protocol_message_received_at = None` (→ `u64::MAX`), and the shortest `connected_time` — failing all three protection criteria simultaneously.

fn sort_then_drop<T, F>(list: &mut Vec<T>, n: usize, compare: F)
where
    F: FnMut(&T, &T) -> std::cmp::Ordering,
{
    list.sort_by(compare);
    if list.len() > n {
        list.truncate(list.len() - n);
    }
}
