### Title
Entropy Provider Can Set `feeInWei = type(uint128).max` to Overflow `getFeeV2`, Permanently Disabling Randomness Requests - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol` imposes no upper-bound check on the `feeInWei` parameter in either `register()` or `setProviderFee()`. Any registered provider can set `feeInWei = type(uint128).max`. When a user subsequently calls any request function, the internal `getFeeV2(provider, gasLimit)` performs the unchecked addition `getProviderFee(...) + _state.pythFeeInWei`, which overflows `uint128` under Solidity 0.8+ checked arithmetic and reverts. This permanently blocks all randomness requests to that provider until the provider lowers their fee — an exact structural analog to M-04.

---

### Finding Description

**Root cause — no upper-bound check in `register()` and `setProviderFee()`:**

`register()` (line 129) directly assigns the caller-supplied `feeInWei` with no validation:

```solidity
// Entropy.sol line 111-129
function register(
    uint128 feeInWei,
    ...
) public override {
    ...
    provider.feeInWei = feeInWei;   // no upper-bound check
``` [1](#0-0) 

`setProviderFee()` (line 810-827) has the same omission:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;   // no upper-bound check
``` [2](#0-1) 

**Overflow in `getFeeV2`:**

Every request path calls `requestHelper`, which calls `getFeeV2(provider, callbackGasLimit)` at line 234:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
``` [3](#0-2) 

`getFeeV2` returns:

```solidity
return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
``` [4](#0-3) 

When `provider.feeInWei = type(uint128).max` and `defaultGasLimit == 0`, `getProviderFee` returns `type(uint128).max`. Adding any non-zero `pythFeeInWei` overflows `uint128` and reverts under Solidity 0.8+ checked arithmetic.

**Test confirmation:**

The existing test suite explicitly documents this behavior:

```solidity
// Entropy.t.sol line 612-622
// Check that overflowing the fee arithmetic causes the transaction to revert.
vm.prank(provider1);
random.register(MAX_UINT128, provider1Proofs[0], hex"0100", 100, provider1Uri);
vm.expectRevert();
random.getFee(provider1);
``` [5](#0-4) 

The test acknowledges the overflow but no mitigation was added to the production code.

---

### Impact Explanation

Any user calling `request()`, `requestV2()`, `requestWithCallback()`, or `requestV2WithCallback()` targeting the affected provider will receive an opaque arithmetic overflow revert. The provider's entire randomness service is effectively frozen. If the affected provider is the **default provider** (set by admin), then `getFeeV2()` (the zero-argument convenience overload) also reverts, breaking the default request flow for all users of the contract on that chain. Users have no way to distinguish this from a legitimate "insufficient fee" error.

---

### Likelihood Explanation

Any address can register as an Entropy provider — `register()` is `public` with no access control. A registered provider can call `setProviderFee(type(uint128).max)` at any time. The attack requires no special privilege, no leaked key, and no governance majority. A malicious provider could use this to extort users: freeze the service, then demand out-of-band payment before restoring a sane fee. The test suite already demonstrates the exact overflow path, confirming it is reachable with a single transaction.

---

### Recommendation

Add an upper-bound check in both `register()` and `setProviderFee()` (and `setProviderFeeAsFeeManager()`):

```solidity
uint128 constant MAX_FEE_IN_WEI = type(uint128).max / 2; // or a protocol-defined cap

function setProviderFee(uint128 newFeeInWei) external override {
    if (newFeeInWei > MAX_FEE_IN_WEI) revert EntropyErrors.FeeExceedsMaximum();
    ...
}
```

This mirrors the recommended mitigation in M-04: disallow setting the parameter to a value that would cause a downstream arithmetic failure.

---

### Proof of Concept

1. Deploy `Entropy` with `pythFeeInWei = 1 wei`.
2. Call `random.register(type(uint128).max, commitment, metadata, 100, uri)` from any EOA.
3. Call `random.getFee(provider)` — transaction reverts with arithmetic overflow.
4. Call `random.request{value: ...}(provider, userCommitment, false)` — reverts at `requestHelper` line 234 for the same reason.
5. The provider's randomness service is now completely inaccessible to all users.

This is confirmed by the existing test `testGetFee` in `Entropy.t.sol` lines 612–622. [6](#0-5)

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[

        provider.feeInWei = newFeeInWei;
