### Title
Unbounded `while` Loop in `executeCallback` Enables Gas Griefing DoS on Legitimate Callbacks — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo::executeCallback` contains an unbounded `while` loop that linearly scans cleared sequence-number slots to advance `_state.firstUnfulfilledSeq`. An unprivileged attacker can create a large number of requests, fulfill all of them except the lowest sequence number, and force the while loop to iterate over every cleared slot when the lowest sequence number is eventually fulfilled. This can push the gas cost of that single `executeCallback` call beyond the block gas limit, permanently preventing the callback from being executed.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs the following loop:

```solidity
// Echo.sol lines 169–174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

`findRequest` performs at least one cold storage read per iteration (first checking `_state.requests[shortKey]`, then falling back to `_state.requestsOverflow[key]`). Each iteration therefore costs roughly 2,100–7,300 gas (cold SLOAD + SSTORE to increment `firstUnfulfilledSeq`).

The loop only terminates when it encounters a sequence number whose slot is still active. If an attacker has pre-filled and then cleared N consecutive sequence numbers above `firstUnfulfilledSeq`, the loop must walk all N slots before stopping.

**Attack path:**

1. Attacker registers as a provider with zero fees (`baseFeeInWei = 0`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`).
2. Attacker calls `requestPriceUpdatesWithCallback` N times, paying only `pythFeeInWei` per request (the protocol-fixed portion). Sequence numbers assigned: `S`, `S+1`, …, `S+N`.
3. Attacker calls `executeCallback` for sequence numbers `S+1` through `S+N`, crediting themselves as `providerToCredit` and recovering the provider-fee portion. Each of these calls terminates the while loop immediately because `firstUnfulfilledSeq = S` is still active.
4. Now `firstUnfulfilledSeq = S` and slots `S+1` through `S+N` are all cleared.
5. When the legitimate provider (or anyone) calls `executeCallback` for sequence number `S`, the while loop must iterate N times through cleared slots before it can advance `firstUnfulfilledSeq` past `S+N`.

At ~2,100 gas per cold SLOAD, N ≈ 14,000 iterations exhausts Ethereum's ~30 M gas block limit. The net cost to the attacker is only `pythFeeInWei × N` (the non-recoverable protocol fee), which can be negligible if `pythFeeInWei` is set low.

The code itself acknowledges the problem with a TODO comment immediately above the loop:

> "TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests."

The gas benchmark test `testMultipleRequestsOutOfOrderFulfillment` also explicitly documents the worst-case linear scan.

---

### Impact Explanation

A malicious user can make `executeCallback` for the lowest outstanding sequence number exceed the block gas limit, making it impossible for any provider to fulfill that callback. The requester's funds are locked in the contract and the callback is never delivered. Because `firstUnfulfilledSeq` is a global counter shared across all users, a single attacker can block the entire queue.

**Impact:** High — permanent DoS of pending price-update callbacks; requester funds locked.

---

### Likelihood Explanation

`requestPriceUpdatesWithCallback` and `executeCallback` are both permissionless. The attacker only needs to pay `pythFeeInWei` per request (recoverable provider fees are returned to themselves). With a low `pythFeeInWei` setting the attack is cheap. No privileged access, leaked keys, or external oracle manipulation is required.

**Likelihood:** Medium — requires capital proportional to `pythFeeInWei × N`, but the attacker recovers the provider-fee portion, making the net cost low.

---

### Recommendation

Replace the linear scan with a data structure that does not require iterating over cleared slots:

- **Doubly-linked list of active requests** (as noted in the TODO comment): maintain `prev`/`next` pointers so that clearing a request is O(1) and advancing `firstUnfulfilledSeq` is O(1).
- **Alternatively**, remove `firstUnfulfilledSeq` entirely and have `getFirstActiveRequests` scan lazily off-chain; the on-chain `executeCallback` does not need to maintain this pointer at all if it is only used for off-chain discovery.
- If the pointer must be maintained on-chain, cap the number of iterations in the while loop to a small constant (e.g., 32, matching `NUM_REQUESTS`) to bound gas consumption.

### Proof of Concept

```solidity
// Attacker registers as provider with zero fees
echo.registerProvider(0, 0, 0);

// Step 1: create N+1 requests (attacker pays pythFeeInWei each)
uint64[] memory seqs = new uint64[](N + 1);
for (uint i = 0; i <= N; i++) {
    seqs[i] = echo.requestPriceUpdatesWithCallback{value: pythFeeInWei}(
        attackerAddress, block.timestamp, priceIds, callbackGasLimit
    );
}
// seqs[0] = S (lowest), seqs[1..N] = S+1..S+N

// Step 2: fulfill seqs[1] through seqs[N] — each call's while loop
// terminates immediately because seqs[0]=S is still active.
for (uint i = 1; i <= N; i++) {
    echo.executeCallback(attackerAddress, seqs[i], updateData, priceIds);
}
// Now firstUnfulfilledSeq == S, slots S+1..S+N are all cleared.

// Step 3: legitimate provider attempts to fulfill seqs[0]=S.
// The while loop now iterates N times through cleared slots → OOG.
echo.executeCallback(legitimateProvider, seqs[0], updateData, priceIds); // reverts OOG
```

    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];

            return req;
        } else {
