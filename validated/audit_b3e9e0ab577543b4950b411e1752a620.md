Audit Report

## Title
`consecutive_failed` Never Reset on Successful Package Inclusion Causes Premature Block-Template Loop Termination — (`tx-pool/src/component/tx_selector.rs`)

## Summary

In `TxSelector::txs_to_commit`, the `consecutive_failed` counter is initialized once and incremented on every size/cycles or ancestor-check failure, but is never reset to zero after a successful package inclusion. Because the counter accumulates across successes, the selection loop terminates after 4001 total failures rather than 4001 *consecutive* failures. An unprivileged attacker can exploit this by flooding the proposed pool with transactions whose `ancestors_size` is just below `size_limit` but exceeds `size_limit` once any prior transaction has been committed, causing the counter to exhaust and the loop to break early — leaving legitimate, individually-fitting transactions unselected and block templates severely under-filled.

## Finding Description

`consecutive_failed` is initialized once at the top of `txs_to_commit`: [1](#0-0) 

The initial filter passes attacker transactions whose `ancestors_size` is just below `size_limit` (e.g., 599,950 ≤ 600,000): [2](#0-1) 

After even one legitimate transaction is committed (e.g., `size = 100`), each attacker transaction fails the cumulative size check and increments the counter: [3](#0-2) 

The ancestor-not-proposed path also increments the counter: [4](#0-3) 

The success path (lines 207–220) adds entries to the block and calls `update_modified_entries`, but contains **no `consecutive_failed = 0` reset**: [5](#0-4) 

The constant and its comment explicitly state the intent is to stop "when the block is *close to full*" — a heuristic for *consecutive* failures, not total failures: [6](#0-5) 

The missing reset means that after 4001 attacker transactions each incrementing `consecutive_failed` by 1, the loop breaks unconditionally at line 158–159, skipping all remaining proposed transactions regardless of whether they individually fit the remaining block space.

The call chain from an unprivileged submitter: `send_transaction` (RPC) → pending pool → proposed pool → `package_txs`: [7](#0-6) 

This is invoked on every block template refresh: [8](#0-7) 

Existing guards are insufficient: the initial `ancestors_size <= size_limit` filter is bypassed because attacker transactions are crafted to pass it individually; the `failed_txs` set only protects against re-processing `modified_entries` entries, not regular proposed-pool entries; and a new `TxSelector` is constructed on each call, resetting `failed_txs` but not preventing the attack from repeating each block.

## Impact Explanation

Every miner calling `get_block_template` receives a systematically under-filled block template. In the PoC scenario, a 600,000-byte block is filled with only 100 bytes (0.02% utilization) while legitimate transactions remain unselected. Repeated each block, this constitutes a sustained, low-cost mechanism to reduce network throughput and cause transaction backlog — matching the allowed impact: **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

- Triggerable by any unprivileged user via the standard `send_transaction` RPC; no special access required.
- CKB's two-phase commit means attacker transactions move to the proposed pool after the proposal window (~2 blocks); miners naturally include cheap short-ID proposals.
- The attacker needs ~4001 transactions with `ancestors_size` in `(size_limit - S, size_limit]`, where `S` is observable from the mempool. This is straightforward to craft.
- The attacker's transactions are never committed (they fail the cumulative size check), so the attacker pays minimum fees but occupies no block space — cost is low relative to impact.
- The attack must be repeated each block (new `TxSelector` per call), but the cost remains constant and low.

## Recommendation

Add `consecutive_failed = 0;` immediately after a successful package inclusion, restoring the intended semantics that the counter tracks *consecutive* failures:

```rust
self.update_modified_entries(&ancestors);
consecutive_failed = 0; // reset: successful inclusion means block is not yet "full"
```

This should be placed at line 221, immediately after the `update_modified_entries` call at line 220 in `tx-pool/src/component/tx_selector.rs`. [9](#0-8) 

## Proof of Concept

```
Parameters:
  size_limit = 600_000 bytes
  S = 100 bytes  (size of first committed tx, fee_rate = 1000)
  attacker_ancestors_size = 599_950
    → passes initial filter: 599_950 ≤ 600_000
    → fails cumulative check after tx_A: 100 + 599_950 = 600_050 > 600_000
  attacker_fee_rate = 999  (sorted between tx_A and tx_B)
  attacker_tx_count = 4001

Proposed pool (sorted by fee rate, descending):
  [0]       tx_A (fee_rate=1000, ancestors_size=100)    → COMMITTED; size=100
  [1..4001] attacker_tx_i (fee_rate=999, ancestors_size=599_950)
                                                        → FAIL size check; consecutive_failed=1..4001
                                                        → consecutive_failed > 4000 → BREAK
  [4002]    tx_B (fee_rate=500, ancestors_size=100)     → NEVER REACHED

Expected: tx_A and tx_B both committed (200 bytes used).
Actual:   only tx_A committed; block is 599,900 bytes under-filled.

Verification: add consecutive_failed = 0 after update_modified_entries; re-run;
tx_B is now reached and committed.
```
