Audit Report

## Title
Unauthenticated Miner Notify HTTP Server Allows Coinbase Redirection and Mining Reward Theft — (File: `miner/src/client.rs`)

## Summary
When `listen` is configured, `listen_block_template_notify` binds a raw TCP listener with no authentication layer. Any network-reachable attacker can POST a crafted `BlockTemplate` containing an attacker-controlled coinbase lock script. The miner solves PoW for the injected template and submits it; the CKB node accepts the block and pays all block rewards to the attacker's address.

## Finding Description
`spawn_background` checks `self.config.listen` and spawns `listen_block_template_notify`, which binds a `TcpListener` and dispatches every connection to `handle` with no authentication: [1](#0-0) 

The `handle` function accepts any HTTP POST body, deserializes it as `BlockTemplate`, and immediately calls `update_block_template` — no token, no IP allowlist, no signature: [2](#0-1) 

`update_block_template` has a deduplication guard using `fetch_update` with the closure `|id| if id != work_id || id == 0 { Some(work_id) } else { None }`. When the injected template uses `work_id = 0`, the condition simplifies to `(id != 0) || (id == 0)` — a tautology — so `fetch_update` always succeeds and the attacker's template is always forwarded to workers via `new_work_tx`: [3](#0-2) 

`submit_block` in `rpc/src/module/miner.rs` only runs `HeaderVerifier` and checks that the parent hash exists, then calls `blocking_process_block`. There is no constraint on the coinbase lock script: [4](#0-3) 

`BlockAssembler::notify()` in `tx-pool/src/block_assembler/mod.rs` sends plain unauthenticated HTTP POSTs to configured notify URLs — confirming the protocol carries no shared secret the miner's listener could verify: [5](#0-4) 

## Impact Explanation
An attacker who can reach the miner's notify port can permanently redirect all block rewards to their own address. Every block mined while the attacker's template is active pays the coinbase to the attacker. This is direct, irreversible theft of mining rewards — a concrete financial attack matching the **Critical** impact class: "Vulnerabilities which could easily damage CKB economy."

## Likelihood Explanation
The `listen` feature is a supported, documented production feature. The default config shows `listen = "127.0.0.1:8888"` as a commented-out example: [6](#0-5) 

Operators deploying miner and node on separate machines — a common production topology — must bind `listen` to a non-localhost address, exposing the port to network-reachable attackers. Even with a localhost binding, any co-located process can exploit it. The exploit requires a single HTTP POST with no credentials, keys, or privileged access. It is repeatable and persistent until the miner is restarted or the attacker's template is displaced by a legitimate one.

## Recommendation
Add a shared-secret authentication mechanism. A configurable token should be added to `MinerClientConfig` and `BlockAssemblerConfig`. `BlockAssembler::notify()` should include the token as an HTTP header (e.g., `Authorization: Bearer <token>`), and the miner's `handle` function must reject requests missing or presenting an incorrect token before deserializing the body. Alternatively, enforce a strict IP allowlist in `listen_block_template_notify` so only the configured CKB node's address is accepted.
