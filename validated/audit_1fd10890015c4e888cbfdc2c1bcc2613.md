### Title
`PythAggregatorV3.latestRoundData()` Returns Stale Price Without Staleness Check — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, explicitly recommended in Pyth's own migration guide for protocols switching from Chainlink. Every price-returning function in the contract — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `decimals()` — calls `pyth.getPriceUnsafe(priceId)`, which carries no staleness guarantee and may return a price from arbitrarily far in the past. No `publishTime` recency check is performed before returning the price to callers.

---

### Finding Description

`PythAggregatorV3.latestRoundData()` is implemented as:

```solidity
function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
``` [1](#0-0) 

`getPriceUnsafe()` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency checks: [2](#0-1) 

Two compounding issues exist:

1. **No staleness check on `publishTime`**: `updatedAt` is set to `price.publishTime`, which may be hours or days old. No comparison against `block.timestamp` is performed.

2. **`answeredInRound` is always equal to `roundId`**: Both are set to `uint80(price.publishTime)`. The standard Chainlink staleness guard used by downstream protocols — `require(answeredInRound >= roundId)` — will always pass, giving a false sense of freshness even when the price is stale.

The same pattern applies to `latestAnswer()` and `getRoundData()`: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink drop-in replacement (as recommended in Pyth's official migration guide) and calls `latestRoundData()` will receive a price that may be arbitrarily stale, with no on-chain revert or signal. The standard Chainlink staleness guard (`answeredInRound >= roundId`) is structurally bypassed because both values are always identical. This enables an attacker to exploit stale prices for collateral manipulation, undercollateralized borrowing, or favorable trade execution — the same class of impact as the reference report. [5](#0-4) 

---

### Likelihood Explanation

Pyth's own documentation actively directs Chainlink-migrating protocols to deploy `PythAggregatorV3` as a direct replacement. Protocols following this guide will inherit the stale-price exposure. The likelihood is elevated because:

- The migration guide presents `PythAggregatorV3` as a transparent drop-in with no staleness caveat at the `latestRoundData()` level.
- Protocols that previously relied on Chainlink's heartbeat-based freshness may not add a secondary `publishTime` check.
- No update to the Pyth on-chain state is required by the attacker; they simply wait for the price to go stale and then interact with the vulnerable protocol.

---

### Recommendation

Replace `getPriceUnsafe` in all price-returning functions of `PythAggregatorV3` with `getPriceNoOlderThan`, using a configurable `maxAge` parameter set at construction time. Alternatively, add an explicit `require(block.timestamp - price.publishTime <= maxAge)` guard before returning. The `maxAge` should be configurable by the deployer to match the protocol's risk tolerance (e.g., 60 seconds for lending, 30 seconds for derivatives).

```solidity
uint256 public maxAge; // set in constructor

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a price feed ID.
2. Allow the on-chain Pyth price to go stale (no `updatePriceFeeds` call for > heartbeat period).
3. Call `latestRoundData()` on the adapter — it returns the old price with `updatedAt` = old `publishTime` and `answeredInRound == roundId`, passing all standard Chainlink staleness guards.
4. A downstream lending protocol using this adapter will accept the stale collateral price, allowing undercollateralized borrowing or inflated collateral valuation. [6](#0-5)
