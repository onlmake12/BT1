Audit Report

## Title
Unbounded Per-Peer Request Rate in `LightClientProtocol` Enables RocksDB I/O Exhaustion DoS — (`util/light-client-protocol-server/src/lib.rs`)

## Summary

`LightClientProtocol` accepts `GetBlocksProof` and `GetTransactionsProof` messages from any peer with no per-peer rate limiting. Each max-size request (1,000 hashes) unconditionally triggers `mmr.gen_proof(1000_positions)`, performing O(1,000 × log₂ N) RocksDB reads. Because the protocol is enabled by default and well-formed requests are never penalised, a single attacker can sustain a flood that saturates the RocksDB layer shared with block validation, degrading or stalling normal node operation.

## Finding Description

`LightClientProtocol` holds only `pub shared: Shared` — no `RateLimiter`, no per-peer counter, no token bucket. Every incoming message is parsed and forwarded to `try_process` with zero throttling; the only early exit is for malformed bytes, which triggers `BAD_MESSAGE_BAN_TIME`. A well-formed request is never penalised.

`GET_BLOCKS_PROOF_LIMIT` and `GET_TRANSACTIONS_PROOF_LIMIT` cap hashes per message at 1,000 but impose no constraint on message frequency. `GetBlocksProofProcess::execute` and `GetTransactionsProofProcess::execute` each only reject empty or oversized messages; a max-size valid request passes straight through to `reply_proof`.

`reply_proof` calls `snapshot.chain_root_mmr(last_block.number() - 1)` and then `mmr.gen_proof(items_positions)` for every accepted request. `chain_root_mmr` constructs an MMR whose store backend is the RocksDB-backed `Snapshot`, making every proof generation a direct RocksDB read amplification event.

A grep across the entire `util/light-client-protocol-server/` tree returns zero matches for `rate_limit`, `RateLimiter`, or `governor`, confirming the complete absence of any throttle. By contrast, `HolePunching` uses a `governor::RateLimiter` keyed by peer at 30 req/s.

## Impact Explanation

A single attacker sustaining a flood of max-size `GetBlocksProof` requests forces ~23,000 RocksDB reads per message (1,000 positions × ~23 levels at N = 10,000,000). With up to 125 inbound peers each doing the same, the aggregate I/O load saturates the RocksDB instance shared with block validation and chain sync, causing the node to become unresponsive or crash. This maps to **High (10,001–15,000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The attack requires only a valid P2P connection and 1,000 publicly known block hashes — no PoW, no key material, no privileged role. The LightClient protocol is on by default on every standard CKB full node. The per-request cost is deterministic and the attack is trivially repeatable in a tight loop from a single machine.

## Recommendation

Add a per-peer `governor::RateLimiter` to `LightClientProtocol`, analogous to the one in `HolePunching` (e.g., 10–30 requests/second per peer). Additionally, consider reducing `GET_BLOCKS_PROOF_LIMIT` / `GET_TRANSACTIONS_PROOF_LIMIT` or introducing a per-request work budget proportional to chain height to bound the worst-case RocksDB read amplification.

## Proof of Concept

1. Run a local CKB full node with default configuration (LightClient enabled).
2. Collect 1,000 valid block hashes from the chain (publicly available via RPC).
3. Open a P2P connection to the node using the LightClient protocol.
4. In a tight loop, send `GetBlocksProof { last_hash: <tip_hash>, block_hashes: [1000 valid hashes] }`.
5. Monitor RocksDB I/O metrics and block validation latency on the target node.
6. Observe I/O saturation and degraded or stalled block processing — reproducible locally without mainnet access by running a simulated chain of sufficient height.

        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
