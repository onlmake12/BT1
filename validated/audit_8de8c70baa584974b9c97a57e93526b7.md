Audit Report

## Title
Per-Transaction Full Cycle Budget Allows N×max_block_cycles Verification Work Before Block Rejection — (`verification/contextual/src/contextual_block_verifier.rs`)

## Summary
`BlockTxsVerifier::verify` passes the full `max_block_cycles()` (3,500,000,000) as the cycle cap to every transaction's `ContextualTransactionVerifier::verify` call, regardless of how many cycles prior transactions consumed. The aggregate cycle sum is only checked after all transactions complete. A malicious miner can craft a block with up to ~1,000 transactions each consuming nearly `max_block_cycles` cycles, forcing every verifying node to expend up to 1,000 × 3.5B = 3.5 trillion CPU cycles before the block is rejected.

## Finding Description
In `BlockTxsVerifier::verify`, all resolved transactions are iterated via Rayon's `par_iter()`. For each transaction not in the verification cache, `ContextualTransactionVerifier::verify` is called with the full `max_block_cycles()` as the cycle cap:

```rust
// verification/contextual/src/contextual_block_verifier.rs:432-434
.verify(
    self.context.consensus.max_block_cycles(),
    skip_script_verify,
)
```

`ContextualTransactionVerifier::verify` passes this cap directly to `self.script.verify(max_cycles)`, which is `TransactionScriptsVerifier::verify`. That verifier correctly decrements the budget across script groups within a single transaction, but the cap for the entire transaction is the full block budget. After all parallel transactions complete, the aggregate is checked:

```rust
// verification/contextual/src/contextual_block_verifier.rs:468-469
if sum > self.context.consensus.max_block_cycles() {
    Err(BlockErrorKind::ExceededMaximumCycles.into())
```

There is no progressive reduction of the remaining cycle budget across transactions. No `max_tx_cycles` or per-transaction limit exists in the codebase. The only guard is the post-hoc aggregate check, which fires only after all N transactions have already been fully executed.

Contrast with the correct pattern inside `TransactionScriptsVerifier::verify` (`script/src/verify.rs:203-204`), where each script group receives `max_cycles - cycles` (the remaining budget), not the full cap.

The tx-pool also uses `max_block_cycles()` per transaction (`tx-pool/src/util.rs:720`), but there the mempool enforces a single-transaction context. In block verification, the aggregate constraint makes the per-transaction full-budget grant an over-allocation that is only caught after all work is done.

## Impact Explanation
This is a **High** severity issue matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

`MAX_BLOCK_CYCLES = TWO_IN_TWO_OUT_CYCLES × TWO_IN_TWO_OUT_COUNT = 3,500,000 × 1,000 = 3,500,000,000` cycles. A block can contain up to ~1,000 transactions (bounded by `MAX_BLOCK_BYTES = 597,000` bytes). A malicious miner crafts each transaction with a RISC-V loop script consuming just under `max_block_cycles` cycles. Each transaction individually passes its per-transaction cycle check (it does not exceed `max_block_cycles`), but the aggregate sum far exceeds the block limit, so the block is ultimately rejected.

Before rejection, every honest verifying node has executed up to 1,000 × 3,500,000,000 = 3.5 × 10¹² total CPU cycles across Rayon's parallel workers. With 8 threads, wall-clock cost per malicious block is approximately 125× the cost of a legitimate max-cycle block — sustained CPU saturation lasting minutes per node. This directly degrades block propagation and verification throughput across the network.

## Likelihood Explanation
Mining in CKB is permissionless. An attacker with even a small fraction of hashpower will occasionally produce valid PoW blocks. A single valid block is sufficient to trigger the expensive verification on all connected peers. The RISC-V loop scripts needed to consume near-`max_block_cycles` cycles are trivially constructable. The attack is repeatable as long as the attacker can find valid PoW solutions, and the cost-to-damage ratio is highly favorable: one valid block causes sustained CPU saturation on every verifying node in the network.

## Recommendation
Pass the **remaining** cycle budget to each transaction verifier, decrementing it by the cycles consumed by each preceding transaction — mirroring the correct pattern in `TransactionScriptsVerifier::verify` (`script/src/verify.rs:203-204`) where `max_cycles - cycles` is passed to each script group.

Because transactions are currently verified via `par_iter()`, the parallel structure must be addressed. Two viable approaches:
1. Switch to sequential iteration for block verification, maintaining a running `remaining_cycles` counter passed to each `ContextualTransactionVerifier::verify` call.
2. Retain parallelism but pre-compute a per-transaction cycle cap (e.g., `max_block_cycles / tx_count`) before the parallel phase, so no single transaction can consume more than its fair share of the block budget.

## Proof of Concept
1. Construct a RISC-V script that loops for exactly `max_block_cycles - 1` cycles (trivially: a tight loop with a counter).
2. Create ~1,000 valid CKB transactions, each referencing a cell with this script as its lock script.
3. Mine a valid PoW block containing all ~1,000 transactions.
4. Broadcast the block to the network.
5. Observe: every receiving node executes all 1,000 transactions in parallel (each consuming ~3.5B cycles), totaling ~3.5T CPU cycles before `ExceededMaximumCycles` is returned at line 469.
6. The aggregate sum check at line 468 fires only after all script executions complete, confirming the full verification cost was incurred before rejection.
