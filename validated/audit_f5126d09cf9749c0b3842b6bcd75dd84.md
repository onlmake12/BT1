### Title
Off-by-One Epoch Height Mismatch in RFC0044 Activation Bypasses `DaoScriptSizeVerifier` and MMR Extension Enforcement at Epoch Boundary — (`tx-pool/src/block_assembler/mod.rs`, `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

The RFC0044 activation check in both the block assembler and the contextual block verifier uses the **parent block's** epoch number instead of the **current block's** epoch number. This creates a one-block window at the activation epoch boundary where two RFC0044 consensus rules are silently skipped: (1) the mandatory MMR chain-root block extension, and (2) the `DaoScriptSizeVerifier` check on DAO withdrawal transactions. The code itself contains an explicit comment acknowledging this as an off-by-one bug. The mismatch is structurally analogous to the reported `voteweighted::median` issue — data from the previous height drives a rule that should be governed by the current height.

---

### Finding Description

**Root cause — `build_extension` in the block assembler:** [1](#0-0) 

```rust

    // so be careful, it needs to be preserved for consistency reasons and not fixed directly.

```

The comment is explicit: the assembler feeds the **tip** (parent) epoch number into `rfc0044_active`, not the epoch of the block being assembled.

    let rfc0044_active_epoch = match self.id.as_str() {
        mainnet::CHAIN_SPEC_NAME => softfork::mainnet::RFC0044_ACTIVE_EPOCH,  // 8651
        testnet::CHAIN_SPEC_NAME => softfork::testnet::RFC0044_ACTIVE_EPOCH,  // 5711
        _ => 0,
    };

}
``` [5](#0-4) 

**The height mismatch in concrete terms:**

For the very first block of epoch `RFC0044_ACTIVE_EPOCH` (e.g., 8651 on mainnet):

| Data point | Value |
|---|---|
| New block's epoch number | 8651 |
| Parent block's epoch number | 8650 |
| `rfc0044_active(8651)` | `true` — RFC0044 **should** be enforced |
| `rfc0044_active(8650)` | `false` — RFC0044 is **not** enforced |

Because both the assembler and the verifier pass the **parent's** epoch number, the first block of the activation epoch is assembled without the MMR extension and verified without the `DaoScriptSizeVerifier` check.

**Secondary mismatch — tx-pool applies `DaoScriptSizeVerifier` unconditionally:** [6](#0-5) 

```rust
DaoScriptSizeVerifier::new(rtx, snapshot.cloned_consensus(), snapshot.as_data_loader())
    .verify()?;
```

The tx-pool's `verify_rtx` applies `DaoScriptSizeVerifier` with **no** `rfc0044_active` guard. The block verifier applies it only when `rfc0044_active(parent.epoch().number())` is true. At the first block of the activation epoch, the tx-pool would reject a DAO withdrawal with a mismatched lock script, but the block verifier would accept the same transaction if it arrived via `submit_block` RPC (bypassing the tx-pool entirely).

---

### Impact Explanation

Two RFC0044 consensus rules are unenforced for exactly one block — the first block of the activation epoch:

1. **MMR chain-root extension omission**: `BlockExtensionVerifier` does not require the block extension field to contain the MMR chain root. A miner can produce a valid (accepted) block without it. Light clients and any verifier relying on the MMR root for the activation-epoch boundary block will find it absent.

2. **`DaoScriptSizeVerifier` bypass**: A miner can include a DAO Phase-2 withdrawal transaction whose output lock script differs in serialized size from the corresponding deposit cell's lock script. The tx-pool would have rejected this transaction, but a block submitted directly via the `submit_block` RPC bypasses the tx-pool. The block verifier accepts it because `rfc0044_active(parent.epoch().number())` evaluates to `false`. The `DaoScriptSizeVerifier` was introduced specifically to close a known DAO script vulnerability where mismatched lock sizes can be exploited; bypassing it re-opens that window for one block. [7](#0-6) 

---

### Likelihood Explanation

- The window is exactly **one block** per chain (the first block of `RFC0044_ACTIVE_EPOCH`).
- On mainnet (epoch 8651) and testnet (epoch 5711) those blocks have already been produced; the window is closed for those networks.
- On any devnet, private chain, or future chain using the same codebase, the window opens at the configured activation epoch.
- Exploitation requires a miner to be the producer of that specific block and to craft the bypass intentionally. The attacker-controlled entry point is the `submit_block` RPC or direct block production, both reachable by an unprivileged miner.
- Likelihood is **low** for mainnet/testnet (window already passed) but **medium** for new deployments.

---

### Recommendation

Replace all three uses of `parent.epoch().number()` in the RFC0044 activation gate with the **current block's** epoch number. For the block assembler, derive the new block's epoch from `next_epoch_ext` before calling `rfc0044_active`. For the verifiers, use `self.header.epoch().number()` (the block being verified) rather than `self.parent.epoch().number()`. The existing comment in `build_extension` acknowledges the bug and warns that both sites must be changed atomically to avoid a consensus split between nodes running old and new code.

---

### Proof of Concept

1. Identify the first block number of `RFC0044_ACTIVE_EPOCH` on the target chain.
2. Craft a DAO Phase-2 withdrawal transaction where the output lock script has a different serialized size than the input deposit cell's lock script (deposit committed after `starting_block_limiting_dao_withdrawing_lock`).
3. Build a block at that exact block number containing the crafted transaction. Do **not** include the MMR chain-root block extension.
4. Submit via `submit_block` RPC, bypassing the tx-pool.
5. Observe: the block is accepted. `BlockExtensionVerifier` does not reject the missing extension because `rfc0044_active(parent.epoch().number()) = rfc0044_active(RFC0044_ACTIVE_EPOCH - 1) = false`. `TransactionsVerifier` does not invoke `DaoScriptSizeVerifier` for the same reason.
6. On the second block of the activation epoch, the same transaction would be rejected because `parent.epoch().number() = RFC0044_ACTIVE_EPOCH`, making `rfc0044_active` return `true`.

    /// Verifies that for all Nervos DAO transactions, withdrawing cells must use lock scripts
    /// of the same size as corresponding deposit cells
    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
