### Title
Decompressed Frame Size Not Bounded by Per-Protocol `max_frame_length` — (`network/src/compress.rs`)

### Summary

The `LengthDelimitedCodecWithCompress::decode` function enforces the per-protocol `max_frame_length` only on the **compressed** wire frame. After decompression, the output is only checked against the global `MAX_UNCOMPRESSED_LEN = 8 MB`. An unprivileged remote peer can send a compressed frame just below a protocol's `max_frame_length` that decompresses to up to 8 MB, bypassing the intended per-protocol size limit entirely.

### Finding Description

`MAX_UNCOMPRESSED_LEN` is defined as 8 MB: [1](#0-0) 

Per-protocol limits are much smaller: [2](#0-1) 

In `decode`, the `length_delimited` inner codec enforces `max_frame_length` on the compressed bytes. After that, the only decompression guard is: [3](#0-2) 

There is **no** subsequent check that `decompressed_bytes_len <= max_frame_length`. The function returns the full decompressed `BytesMut` to the caller.

Additionally, the `enable_compress` flag is only consulted in `encode` (line 161), **not** in `decode`. A remote peer can always send a COMPRESS_FLAG-tagged frame and the decoder will decompress it regardless of local configuration. [4](#0-3) 

`CKBProtocol::build` wires this codec for Sync, Relay, and other handlers: [5](#0-4) 

### Impact Explanation

| Protocol | `max_frame_length` | Max decompressed | Amplification |
|---|---|---|---|
| Ping | 1 KB | 8 MB | 8192× |
| Discovery | 512 KB | 8 MB | 16× |
| Sync | 2 MB | 8 MB | 4× |
| RelayV3 | 4 MB | 8 MB | 2× |

Every peer connection can deliver 8 MB to the upper-layer handler regardless of the protocol's intended limit. With many peers, this causes:
- **OOM**: each peer can force 8 MB allocations per message via `BytesMut::zeroed(decompressed_bytes_len)`
- **CPU exhaustion**: snappy decompression of near-maximum frames on every message
- **Handler misbehavior**: molecule deserialization of 8 MB inputs in handlers (e.g., `Synchronizer`) designed for ≤2 MB messages

### Likelihood Explanation

Any unauthenticated peer on the P2P network can trigger this. No PoW, no key, no privilege required — just a TCP connection and a crafted snappy frame. The attack is trivially scriptable and affects all nodes simultaneously.

### Recommendation

In `decode`, after decompression succeeds, add a check against the codec's own `max_frame_length`:

```rust
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::Error::new(io::ErrorKind::InvalidData, "decompressed data too large"));
}
```

This must be inserted between the `decompress_len` call and the `BytesMut::zeroed` allocation to also prevent the allocation itself. [3](#0-2) 

### Proof of Concept

1. Connect to a CKB node on the Sync protocol (protocol id 100, `max_frame_length = 2 MB`).
2. Construct a snappy-compressed payload: take 8 MB of zeros, compress with snappy (compresses to ~8 KB due to repetition — well under 2 MB).
3. Prepend `COMPRESS_FLAG` (0x80) and send as a length-delimited frame.
4. The `length_delimited` codec accepts it (compressed size ≪ 2 MB).
5. `decompress_len` returns 8 MB < `MAX_UNCOMPRESSED_LEN` → accepted.
6. `BytesMut::zeroed(8MB)` is allocated; decompression succeeds.
7. `CKBHandler::received` is called with an 8 MB slice; the Sync handler attempts molecule deserialization of an 8 MB `SyncMessage`.

Repeat from N peers simultaneously to exhaust memory.
