### Title
No Refund of Excess Fee Payment Combined with Dynamic Fee Adjustment Creates Permanent Fund Loss for Entropy Users - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`'s `requestHelper`, any `msg.value` exceeding the required fee is permanently credited to `_state.accruedPythFeesInWei` (Pyth's treasury) rather than refunded to the caller. Because the Fortuna keeper dynamically adjusts provider fees based on gas prices, and no `maxFee` slippage-protection parameter exists on any `requestV2` variant, users who send a buffer to avoid reverts permanently lose that buffer, and users who send exactly the queried fee can have their transaction revert when the fee increases between their `getFeeV2()` call and their `requestV2()` call.

### Finding Description

**Root cause — no refund of excess in `requestHelper`:**

In `Entropy.sol`, `requestHelper` checks that `msg.value >= requiredFee`, then credits the provider with exactly `providerFee` and credits Pyth with the entire remainder:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

Any `msg.value` above `requiredFee` is silently absorbed into `accruedPythFeesInWei` and is never returned. [1](#0-0) 

**Dynamic fee adjustment by the Fortuna keeper:**

The keeper's `adjust_fee_if_necessary` function periodically recalculates the target fee as a function of current gas price and updates the on-chain `feeInWei` for the provider:

```rust
let max_callback_cost: u128 = estimate_tx_cost(middleware, legacy_tx, gas_limit).await?;
let mut target_fee = std::cmp::max(
    (max_callback_cost * u128::from(target_profit_pct)) / 100,
    min_fee_wei,
);
// ...
let contract_call = contract.set_provider_fee_as_fee_manager(provider_address, target_fee);
send_and_confirm(contract_call).await?;
``` [2](#0-1) 

This means the on-chain fee can increase between a user's `getFeeV2()` read and their `requestV2()` submission.

**No `maxFee` slippage-protection parameter:**

None of the `requestV2` variants in `IEntropyV2` accept a `maxFee` parameter. Users have no on-chain mechanism to bound the fee they are willing to pay: [3](#0-2) 

The interface documentation acknowledges both facts but provides no mitigation:
> "Note that the fee can change over time… excess value is *not* refunded to the caller." [4](#0-3) 

### Impact Explanation

Users face two concrete financial harms:

1. **Transaction revert / gas waste**: A user who calls `getFeeV2()` and immediately submits `requestV2{value: fee}()` can have their transaction revert with `InsufficientFee` if the keeper raised the fee in the intervening block. The user loses the gas cost with no service rendered.

2. **Permanent loss of excess ETH**: A user who defensively sends `msg.value = fee + buffer` to avoid reverts permanently loses the buffer. It is credited to `accruedPythFeesInWei` (Pyth governance treasury) and is irrecoverable by the user. The test suite explicitly confirms this behavior — an overpayment of 10,000 wei goes entirely to `accruedPythFees`: [5](#0-4) 

### Likelihood Explanation

The Fortuna keeper runs `adjust_fee_wrapper` on a configurable poll interval and targets a fee proportional to `gas_price * gas_limit * profit_pct`. During periods of gas price volatility (common on EVM chains), the keeper can raise fees by 20–100% within a single block interval. Any user or integrating contract that caches the fee value (even for a single block) is exposed. Integrating contracts that call `getFeeV2()` and `requestV2()` in separate transactions are directly affected. [6](#0-5) 

### Recommendation

1. Add a `maxFee` parameter to `requestV2` variants and revert if `msg.value > maxFee`, giving callers slippage protection analogous to DEX `amountOutMin` guards.
2. Alternatively, refund `msg.value - requiredFee` to `msg.sender` at the end of `requestHelper` when `msg.value > requiredFee`.

### Proof of Concept

1. User calls `getFeeV2(provider, gasLimit)` → returns `F = 0.001 ETH`.
2. In the next block, the Fortuna keeper calls `setProviderFeeAsFeeManager(provider, F')` where `F' = 0.0012 ETH` due to a gas price spike.
3. **Path A**: User submits `requestV2{value: F}()` → reverts `InsufficientFee`. Gas is wasted.
4. **Path B**: User submits `requestV2{value: F + 0.0005 ETH}()` (buffer to avoid reverts). The call succeeds. `providerInfo.accruedFeesInWei += F'`. `accruedPythFeesInWei += (F + 0.0005 ETH) - F' = 0.0003 ETH`. The user permanently loses 0.0003 ETH to the Pyth treasury with no recourse. pub async fn adjust_fee_wrapper(
    contract: Arc<InstrumentedSignablePythContract>,
    chain_state: BlockchainState,
    provider_address: Address,
    poll_interval: Duration,
    legacy_tx: bool,
    min_profit_pct: u64,
    target_profit_pct: u64,
    max_profit_pct: u64,
    min_fee_wei: u128,
    max_fee_wei: Option<u128>,
    metrics: Arc<KeeperMetrics>,
) {
    // The maximum balance of accrued fees + provider wallet balance. None if we haven't observed a value yet.
    let mut high_water_pnl: Option<U256> = None;
    // The sequence number where the keeper last updated the on-chain fee. None if we haven't observed it yet.
    let mut sequence_number_of_last_fee_update: Option<u64> = None;
    loop {
        if let Err(e) = adjust_fee_if_necessary(
            contract.clone(),
            chain_state.id.clone(),
            provider_address,
            legacy_tx,
            min_profit_pct,
            target_profit_pct,
            max_profit_pct,
            min_fee_wei,
            max_fee_wei,
            &mut high_water_pnl,
            &mut sequence_number_of_last_fee_update,
            metrics.clone(),
        )
        .in_current_span()
        .await
        {
            tracing::error!("Fee adjustment failed: {:?}", e);
        }
        time::sleep(poll_interval).await;
    }
