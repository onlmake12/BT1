Audit Report

## Title
Off-by-One Epoch Gate Skips `DaoScriptSizeVerifier` for First Block of RFC0044 Activation Epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

## Summary

`BlockTxsVerifier::verify` gates `DaoScriptSizeVerifier` using `self.parent.epoch().number()` instead of the current block's epoch number. Because `RFC0044_ACTIVE_EPOCH = 8651`, the first block of epoch 8651 has a parent in epoch 8650, causing `rfc0044_active(8650)` to return `false` and the verifier to be skipped for exactly that one block. A miner who wins the PoW race for that block can include a DAO phase-1 withdrawal with a lock script size that differs from the deposit cell's lock script size, permanently committing a malformed DAO withdrawal to the canonical chain.

## Finding Description

`RFC0044_ACTIVE_EPOCH` is hardcoded to `8651`: [1](#0-0) 

`rfc0044_active` uses a `>=` comparison, so `rfc0044_active(8650)` returns `false`: [2](#0-1) 

The gate in `BlockTxsVerifier::verify` passes `self.parent.epoch().number()` — the parent's epoch — not the current block's epoch: [3](#0-2) 

`BlockTxsVerifier` holds both `self.header` (current block) and `self.parent` as separate fields, so the current block's epoch is directly accessible but is not used: [4](#0-3) 

For the first block of epoch 8651: `self.parent.epoch().number()` = 8650, `rfc0044_active(8650)` = `8650 >= 8651` = `false`, so `DaoScriptSizeVerifier` is skipped. For every subsequent block in epoch 8651+, the parent is already in epoch 8651, so the verifier runs correctly. The bypass window is exactly one block.

The tx-pool path calls `DaoScriptSizeVerifier` unconditionally (no epoch gate), so a transaction submitted through the mempool would be rejected. However, a miner can construct a block directly, bypassing the tx-pool entirely: [5](#0-4) 

## Impact Explanation

A miner can permanently commit a DAO phase-1 withdrawal transaction to the canonical chain where the withdrawing cell's lock script size differs from the deposit cell's lock script size, violating the RFC0044 invariant. All nodes replaying the chain use the same gate and accept the block. This constitutes an incorrect implementation of a system script (DAO) behavior — mapping to **High** impact (10001–15000 points): *Incorrect implementation or behavior of CKB-VM or system scripts*. Depending on whether the lock-size mismatch enables extraction of incorrect CKB amounts, it may also qualify as *Vulnerabilities which could easily damage CKB economy* (Critical).

## Likelihood Explanation

Requires valid PoW for the first block of epoch 8651 — a standard miner capability, not a privileged role. The window is one block, but any miner who mines that block can exploit it deterministically. The exploit is locally testable without any external dependencies or victim cooperation.

## Recommendation

Replace `self.parent.epoch().number()` with the current block's epoch number in the gate condition:

```rust
// Use current block's epoch, not parent's
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
``` [6](#0-5) 

## Proof of Concept

1. Build a test chain advancing to the last block of epoch 8650 (parent is in epoch 8650, current block is the first block of epoch 8651).
2. Construct a DAO deposit cell committed at a block number ≥ `starting_block_limiting_dao_withdrawing_lock()`.
3. Construct a phase-1 withdrawal transaction where the output cell uses the DAO type script but has a lock script of a **different size** than the deposit cell's lock script.
4. Mine the first block of epoch 8651 containing this transaction (bypassing the tx-pool).
5. Call `BlockTxsVerifier::verify` — it evaluates `rfc0044_active(8650)` = `false`, skips `DaoScriptSizeVerifier`, and returns `Ok`.
6. Assert the block is accepted by the chain and the malformed withdrawal is permanently committed.
