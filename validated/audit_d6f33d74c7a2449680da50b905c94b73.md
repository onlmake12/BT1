Audit Report

## Title
Peer Store `check_purge` Eviction Failure Allows Unprivileged Peer to DoS Peer Discovery — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
`check_purge` has two eviction strategies, both of which can be simultaneously defeated by filling the 16384-entry peer store with addresses spread across more than 4096 distinct `/16` network groups (≤4 per group), all with `last_connected_at_ms=0` and `attempts_count=0`. Strategy 1 finds no non-connectable addresses; strategy 2 finds no group exceeding 4 members. The result is `Err(PeerStoreError::EvictionFailed)`, which `add_new_addrs` silently discards, permanently blocking peer discovery on the victim node.

## Finding Description
`add_addr` calls `check_purge()?` before every insertion (L75). `check_purge` activates when `addr_manager.count() >= 16384` (L328). Strategy 1 collects addresses where `is_connectable()` returns `false` (L341–355). `AddrInfo::new` always initializes `last_connected_at_ms=0` and `attempts_count=0` (types.rs L65–76), so `is_connectable` returns `true` for every freshly added address (types.rs L89–105): the `attempts_count >= ADDR_MAX_RETRIES(3)` branch requires 3 failed attempts, and the 7-day/10-failure branch requires prior connection history. Strategy 1 therefore yields zero candidates. Strategy 2 groups addresses by `/16` subnet (`Group::IP4([bits[0], bits[1]])`), sorts groups by size, takes the top half, and evicts 2 random addresses only from groups with `addrs.len() > 4` (L378). If the attacker spreads ≤4 addresses across each of >4096 distinct `/16` subnets, no group exceeds 4 members, the `if addrs.len() > 4` branch is never taken, and `candidate_peers` is empty. L399–401 then returns `Err(PeerStoreError::EvictionFailed)`. `DiscoveryAddressManager::add_new_addrs` catches this error and emits only a `debug!` log (L355–360), raising no operator alert. The `DuplicateFirstNodes` guard (L181) only prevents a second non-announce `Nodes` message per session; the attacker uses 6 independent sessions. No per-session address ingestion rate limit exists.

## Impact Explanation
After saturation, every subsequent `add_addr` call returns `EvictionFailed` and the victim node can no longer learn about any new peers via the discovery protocol. Applied cheaply across many nodes, this degrades network-wide peer discovery infrastructure and can partition bootstrapping or recovering nodes from the network. This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
Any peer that can establish a TCP connection to the victim node can execute this attack. No authentication, on-chain funds, or special privileges are required. The attacker needs approximately 6 connections, each sending one `Nodes` message (1000 items × 3 addresses from distinct `/16` subnets = 3000 addresses per message, 18000 total). IPv4 provides 65536 possible `/16` subnets, making the address space trivially sufficient. The attack is cheap, repeatable, and the failure is invisible to operators.

## Recommendation
1. **Add a last-resort eviction strategy**: when both primary strategies yield no candidates, evict the `N` addresses with `last_connected_at_ms == 0` and the lowest `attempts_count` (never-tried addresses), regardless of group membership.
2. **Rate-limit address ingestion per session**: cap the number of new addresses accepted from a single session to a small fraction of `ADDR_COUNT_LIMIT`.
3. **Prefer evicting never-connected addresses first**: addresses with `last_connected_at_ms == 0` and `attempts_count == 0` have the lowest trust and should be the first eviction candidates when the store is full.

## Proof of Concept
```
1. Attacker opens 6 TCP connections to the victim CKB node.
2. On each connection, after the discovery handshake, the attacker sends:
     DiscoveryMessage::Nodes { announce: false, items: [1000 Node entries] }
   where each Node has 3 addresses from distinct /16 subnets
   (e.g., 1.0.0.1, 2.0.0.1, 3.0.0.1, ..., up to 18000 unique /16 groups).
3. After 6 messages, the peer store contains ≥16384 entries, all with
   last_connected_at_ms=0, attempts_count=0 (connectable), spread across
   >4096 /16 groups (≤4 per group).
4. Any subsequent call to peer_store.add_addr() triggers check_purge():
   - Strategy 1: all addresses are connectable → 0 candidates.
   - Strategy 2: no group has >4 members → 0 candidates.
   - Returns Err(PeerStoreError::EvictionFailed).
5. add_new_addrs() logs a debug message and discards the error.
6. The victim node can no longer learn about any new peers via discovery.

Unit test plan:
- Construct a PeerStore, fill it with 16384 AddrInfo entries
  (last_connected_at_ms=0, attempts_count=0) spread across >4096 /16 groups
  (≤4 per group).
- Call add_addr() with a new address.
- Assert the return value is Err(PeerStoreError::EvictionFailed).
```

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
