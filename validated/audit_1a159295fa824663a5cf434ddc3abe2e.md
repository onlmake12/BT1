### Title
`stake_caps_parameters::set_parameters` Doesn't Validate Input Parameters, Enabling Authority Takeover and Broken OIS Staking Caps — (File: `pythnet/stake_caps_parameters/programs/stake_caps_parameters/src/lib.rs`)

---

### Summary

The `set_parameters` function in the `stake_caps_parameters` Solana program accepts a `Parameters` struct and writes it directly to on-chain storage with no validation of the field values. Specifically, it does not check that `parameters.current_authority != Pubkey::default()`. Because the authority gate explicitly permits calls when `stored_parameters.current_authority == Pubkey::default()` (the zero-initialized state), any unprivileged user can call `set_parameters` before the account is initialized, or after the authority accidentally resets it to the default key, and write arbitrary `m`, `z`, and `current_authority` values — permanently taking over the authority and corrupting the OIS staking cap parameters.

---

### Finding Description

The sole on-chain logic in `set_parameters` is:

```rust
pub fn set_parameters(ctx: Context<SetParameters>, parameters: Parameters) -> Result<()> {
    let stored_parameters = &mut ctx.accounts.parameters;
    require!(
        ctx.accounts.signer.key() == stored_parameters.current_authority
            || stored_parameters.current_authority == Pubkey::default(),
        ErrorCode::WrongAuthority
    );
    **stored_parameters = parameters;
    Ok(())
}
```

The account is declared with `init_if_needed`, so before the Pyth team initializes it, `current_authority` is `Pubkey::default()`. The `require!` check passes for **any** signer in that state, and the entire struct is overwritten with the caller-supplied values. No field-level validation exists:

- **`parameters.current_authority`** — no check that it is not `Pubkey::default()`. Setting it to default re-opens the gate for the next caller, enabling a permanent authority-takeover chain.
- **`parameters.m`** — no check that `m > 0`. The pool-cap formula is `C_p = M · Σ 1/max(n_s, Z)`. With `m = 0` every publisher's cap collapses to zero.
- **`parameters.z`** — no check that `z > 0`. With `z = 0` the denominator becomes `max(n_s, 0) = n_s`; for any symbol with zero publishers `n_s = 0`, causing a division-by-zero in the Hermes off-chain computation that publishes caps on-chain.

The `Parameters` struct and the program's only instruction are:

```rust
pub struct Parameters {
    pub current_authority: Pubkey,
    pub m: u64,
    pub z: u64,
}
```

The `m` and `z` values are read by the staking front-end and by Hermes to compute per-publisher stake caps that are then posted to the `publisher_caps` program and consumed by the on-chain OIS staking logic.

---

### Impact Explanation

1. **Authority takeover**: An attacker who calls `set_parameters` while `current_authority == Pubkey::default()` (initial state, or after an accidental reset) becomes the sole authority over the OIS cap parameters.
2. **Staking cap destruction**: Setting `m = 0` makes every publisher's pool cap zero. The OIS staking program distributes rewards proportional to `min(stake, cap)`; with cap = 0 no rewards are distributed to any publisher or delegator, breaking the entire Oracle Integrity Staking incentive layer.
3. **Off-chain computation failure**: Setting `z = 0` causes Hermes to encounter a division-by-zero when computing caps for symbols with no publishers, potentially crashing the cap-publishing pipeline and stalling cap updates for all publishers.

---

### Likelihood Explanation

- **Initial-state window**: The `PARAMETERS_ADDRESS` PDA is fixed at `879ZVNagiWaAKsWDjGVf8pLq1wUBeBz7sREjUh3hrU36`. On any new deployment or network where the account has not yet been initialized, the window is open to any transaction sender.
- **Accidental reset**: The authority can call `set_parameters` with `current_authority = Pubkey::default()` (e.g., a tooling bug or copy-paste error in the CLI), re-opening the gate. The CLI at `pythnet/stake_caps_parameters/cli/src/main.rs` accepts `--authority` as a raw `Pubkey` argument with no client-side validation.
- **No on-chain guard**: There is no `require!(parameters.current_authority != Pubkey::default())` anywhere in the program.

---

### Recommendation

Add field-level validation inside `set_parameters` before writing to storage:

```rust
pub fn set_parameters(ctx: Context<SetParameters>, parameters: Parameters) -> Result<()> {
    let stored_parameters = &mut ctx.accounts.parameters;
    require!(
        ctx.accounts.signer.key() == stored_parameters.current_authority
            || stored_parameters.current_authority == Pubkey::default(),
        ErrorCode::WrongAuthority
    );
    // New: validate all fields
    require!(
        parameters.current_authority != Pubkey::default(),
        ErrorCode::InvalidAuthority
    );
    require!(parameters.m > 0, ErrorCode::InvalidM);
    require!(parameters.z > 0, ErrorCode::InvalidZ);
    **stored_parameters = parameters;
    Ok(())
}
```

This mirrors the fix applied to `BondBaseSDA.setDefaults` (adding relational constraints between parameters) and prevents both the authority-reset attack and the zero-value corruption of the cap formula.

### Proof of Concept

**Scenario A — Initial-state takeover (unprivileged):**

1. The `stake_caps_parameters` PDA has not yet been initialized on a new deployment.
2. Attacker calls `set_parameters` with `{ current_authority: attacker_pubkey, m: 0, z: 0 }`.
3. `stored_parameters.current_authority == Pubkey::default()` → `require!` passes.
4. Attacker is now the sole authority; `m = 0` and `z = 0` are written on-chain.
5. Hermes reads `m = 0`, computes all publisher caps as 0, posts them to `publisher_caps`.
6. OIS staking distributes zero rewards to all publishers and delegators.

**Scenario B — Authority reset (post-initialization):**

1. Current authority calls `set_parameters` with `{ current_authority: Pubkey::default(), m: 1, z: 1 }` (e.g., tooling bug).
