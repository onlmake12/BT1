Audit Report

## Title
O(n) Global Mutex Scan in `SyncState::add_ask_for_txs` Triggered by Unprivileged Relay Peers — (`sync/src/types/mod.rs`)

## Summary
`SyncState::add_ask_for_txs` acquires the `unknown_tx_hashes` mutex and, when the queue reaches `MAX_UNKNOWN_TX_HASHES_SIZE` (50,000), performs a full linear scan over all entries while still holding the lock. Two unprivileged P2P peers each sending one `RelayTransactionHashes` batch of 32,767 hashes is sufficient to saturate the queue, after which every subsequent relay message from any peer re-triggers the O(n) scan, blocking `pop_ask_for_txs` and `mark_as_known_txs` for the duration of each scan.

## Finding Description
In `sync/src/types/mod.rs`, `add_ask_for_txs` acquires `unknown_tx_hashes.lock()` at line 1484 and holds it for the entire function body. After inserting hashes, it checks the queue length at line 1507:

```rust
if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE   // 50,000
    || unknown_tx_hashes.len() >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
```

When true, it performs a full linear scan at lines 1517–1522 — still holding the mutex — to count per-peer entries:

```rust
for (_hash, priority) in unknown_tx_hashes.iter() {
    for peer in priority.peers.iter() {
        if *peer == peer_index {
            peer_unknown_counter += 1;
        }
    }
}
```

This is O(n × p) where n = 50,000 and p = peers per entry. The entry point is `TransactionHashesProcess::execute` in `sync/src/relayer/transaction_hashes_process.rs` line 49, called directly from the relay protocol handler `Relayer::received` for every incoming `RelayTransactionHashes` P2P message. The per-message cap `MAX_RELAY_TXS_NUM_PER_BATCH` = 32,767 means two peers each sending one full batch (2 × 32,767 = 65,534 > 50,000) saturates the queue. A rate limiter exists (30 req/s per peer keyed by `(PeerIndex, message_type)`) but does not prevent the attack — it only caps the O(n) scan frequency to 30/s per attacker peer, which is still sustained contention. The existing `take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` guard at line 1488 limits per-call insertion but does not affect the post-insertion scan. Both `pop_ask_for_txs` (line 1454) and `mark_as_known_txs` (line 1444) acquire the same mutex and are blocked for the full duration of each scan.

## Impact Explanation
This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**. Sustained O(n) mutex hold times on the relay thread's shared `unknown_tx_hashes` lock stall `pop_ask_for_txs` (called every 100ms via `ASK_FOR_TXS_TOKEN`) and `mark_as_known_txs`, degrading transaction propagation throughput. Applied simultaneously to multiple nodes by the same two attacker peers, this degrades transaction relay across the network. Block propagation via compact blocks uses separate code paths and is not directly blocked, so the impact is accurately scoped to transaction relay congestion.

## Likelihood Explanation
Requires only two unprivileged P2P peers with no PoW, no keys, and no privileged access. Each peer sends a single valid `RelayTransactionHashes` message with 32,767 novel hashes — a protocol-conforming message that passes all existing checks. The queue remains saturated as long as the attacker peers stay connected and the node does not drain the hashes. The rate limiter permits up to 30 re-triggers per second per attacker peer, sustaining the contention indefinitely. The attack is repeatable, low-cost, and applicable to any reachable CKB node.

## Recommendation
Maintain a per-peer counter in `PeerState` (e.g., `unknown_tx_hash_count: AtomicUsize`) incremented on insert and decremented on removal. Replace the O(n) scan at lines 1516–1523 with an O(1) lookup of this counter. The `take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` guard already limits per-call insertion; the post-insertion per-peer check must also be O(1) to eliminate the vulnerability.

## Proof of Concept
1. Connect peers A and B to the target node.
2. Peer A sends `RelayTransactionHashes` with 32,767 novel tx hashes (passes `MAX_RELAY_TXS_NUM_PER_BATCH` check).
3. Peer B sends `RelayTransactionHashes` with 32,767 different novel tx hashes.
4. `unknown_tx_hashes.len()` is now ≥ 50,000 (`MAX_UNKNOWN_TX_HASHES_SIZE`).
5. Peer A sends another `RelayTransactionHashes` message (any content).
6. `add_ask_for_txs` acquires the mutex, hits the condition at line 1507, and iterates all 50,000+ entries while holding the lock.
7. Instrument `unknown_tx_hashes.lock()` hold time: it scales linearly with queue size, not with the incoming batch size.
8. Repeat step 5 at up to 30/s (rate limiter cap) to sustain mutex contention and stall `pop_ask_for_txs` (fired every 100ms via `ASK_FOR_TXS_TOKEN` notify).
