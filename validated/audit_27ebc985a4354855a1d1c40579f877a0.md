### Title
Hardcoded `CKB2021_START_EPOCH = 0` in mainnet constants causes premature CKB2021 hardfork activation, leading to consensus divergence for nodes syncing from genesis — (`util/constant/src/hardfork/mainnet.rs`)

### Summary
`mainnet::CKB2021_START_EPOCH` is set to `0` (with the historically correct value `5414` commented out). `CKB2021::new_mirana()` uses this constant for RFC-0029, RFC-0030, RFC-0031, RFC-0036, and RFC-0038 activation epochs. Any node using `new_mirana()` for mainnet consensus will treat all five of these hardfork rules as active from genesis, rather than from epoch 5414 (~2022-05-10).

### Finding Description [3](#0-2) 

The commented-out line `// pub const CKB2021_START_EPOCH: u64 = 5414;` confirms the intended value. The live value `0` is a development/testing override that was not reverted before production use.

`new_mirana()` propagates this to five RFC fields: [4](#0-3) 

The `is_X_enabled(epoch)` methods generated by `define_methods!` compare the stored epoch against the current block epoch. With the stored value being `0`, every block at every epoch satisfies `epoch >= 0`, so all five rules are considered active from block 0.

### Impact Explanation
- **RFC-0030** (`check_length_in_epoch_since`): Adds a strict check that `index < length` in epoch-encoded `since` fields. Pre-fork mainnet blocks that contained transactions with `index >= length` in epoch-since were valid under pre-fork rules. A node with `CKB2021_START_EPOCH = 0` would reject those blocks, diverging from the canonical chain.
- **RFC-0038** (`disallow_over_max_dep_expansion_limit`): Adds a new dep-expansion cap. Any pre-fork transaction that exceeded this limit (valid pre-fork) would be rejected by the misconfigured node.
- **RFC-0031** (`reuse_uncles_hash_as_extra_hash`): Changes block header semantics. Applying this from epoch 0 alters how pre-fork block headers are validated.
- **RFC-0029** and **RFC-0036** are more permissive (allow previously-rejected patterns), so they are less likely to cause rejection of historical blocks, but they still alter the validation surface.

A node syncing from genesis on mainnet with this code cannot reliably reproduce the historical chain state, constituting a consensus divergence.

### Likelihood Explanation
This affects every operator running a mainnet node built from this codebase who syncs from genesis. The misconfiguration is in a hardcoded constant, not a runtime config, so it cannot be corrected without a code change. The `RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH` constant being correctly set to `5414` confirms this is an accidental regression, not intentional design.

### Recommendation
Restore the correct value:
```rust
// util/constant/src/hardfork/mainnet.rs
pub const CKB2021_START_EPOCH: u64 = 5414;
```
Add a compile-time or startup assertion that `CKB2021_START_EPOCH == RFC0028_RFC0032_RFC0033_RFC0034_START_EPOCH` for mainnet, since both groups activated at the same epoch.

### Proof of Concept
1. Build a CKB node from this codebase.
2. Configure it for mainnet (`new_mirana()` is used).
3. Sync from genesis.
4. Locate any mainnet block in epochs 0–5413 containing a transaction with an epoch-encoded `since` where `index >= length` (RFC-0030 violation under post-fork rules).
5. Observe the node rejects the block while the canonical mainnet accepted it — the node cannot complete IBD (Initial Block Download) and forks from mainnet at that block.
