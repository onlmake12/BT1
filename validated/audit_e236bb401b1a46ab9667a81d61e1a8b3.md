Audit Report

## Title
`DaoScriptSizeVerifier` Lock Script Size Check Bypassed via Index Mismatch Between DAO Deposit Input and Withdrawing Output — (`File: verification/src/transaction_verifier.rs`)

## Summary
`DaoScriptSizeVerifier::verify()` pairs resolved inputs with transaction outputs strictly by position using `zip`. If a DAO deposit input and its corresponding DAO withdrawing output appear at different indices in the same transaction, the size equality check is silently skipped. An attacker can exploit this to create a withdrawing cell with a smaller lock script than the deposit cell, causing `DaoCalculator::calculate_maximum_withdraw` to compute a larger `counted_capacity` and pay out more CKB than was deposited.

## Finding Description
The verifier at `verification/src/transaction_verifier.rs` lines 847–888 iterates over `(resolved_inputs[i], outputs[i])` pairs via `zip` and skips any pair where either side does not carry the DAO type script:

```rust
for (i, (input_meta, cell_output)) in self
    .resolved_transaction
    .resolved_inputs
    .iter()
    .zip(self.resolved_transaction.transaction.outputs())
    .enumerate()
{
    if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
        && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
    {
        continue;
    }
    ...
    if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
        return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
    }
}
```

An attacker constructs a Phase 1 (prepare) transaction with mismatched indices:

| Index | Input | Output |
|-------|-------|--------|
| 0 | fee cell (non-DAO) | withdrawing cell (DAO, **small** lock script) |
| 1 | deposit cell (DAO, **large** lock script, data = `0x0000000000000000`) | change cell (non-DAO) |

- Pair `(input[0]=non-DAO, output[0]=DAO)` → input fails `cell_uses_dao_type_script` → `continue`
- Pair `(input[1]=DAO deposit, output[1]=non-DAO)` → output fails `cell_uses_dao_type_script` → `continue`

The size mismatch is never detected. The transaction passes both tx-pool admission (`tx-pool/src/util.rs` lines 110–115) and block-level verification (`verification/contextual/src/contextual_block_verifier.rs` lines 28–31).

The verifier was introduced explicitly as a "temporary solution" because the DAO C script does not itself enforce lock script size equality (comment at line 817–818). The DAO C script in Phase 1 does not enforce same-index pairing between deposit inputs and withdrawing outputs, so no on-chain script catches the bypass.

In Phase 2 withdrawal, `DaoCalculator::calculate_maximum_withdraw` (`util/dao/src/lib.rs` lines 149–156) computes `occupied_capacity` from the **withdrawing** cell's (`output`) lock script size:

```rust
let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
let output_capacity: Capacity = output.capacity().into();
let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
```

A smaller lock script on the withdrawing cell → smaller `occupied_capacity` → larger `counted_capacity` → larger `withdraw_counted_capacity` → attacker receives more CKB than deposited.

The existing unit test `test_dao_disables_different_lock_script_size` (`verification/src/tests/transaction_verifier.rs` lines 927–983) only validates the same-index case (deposit at `resolved_inputs[1]`, withdrawing output at `outputs[1]`) and does not cover the mismatched-index bypass.

## Impact Explanation
This is a **Critical** vulnerability matching the impact class: *"Vulnerabilities which could easily damage CKB economy."* An attacker can steal CKB from the DAO contract proportional to the lock script size difference and the DAO interest multiplier. For a large deposit with a significant lock script size difference (e.g., 80 bytes), the attacker extracts funds belonging to other depositors. The attack is repeatable and scales with deposit size.

## Likelihood Explanation
The attack requires no special privilege, key leak, or majority hashpower. Any user can submit a Phase 1 transaction with inputs and outputs at mismatched indices via standard RPC. The transaction is accepted by the tx-pool and committed to a block because `DaoScriptSizeVerifier` silently passes. After the lock period, Phase 2 withdrawal yields more CKB than deposited.

## Recommendation
Replace the index-based `zip` pairing with an explicit cross-index search: for every input cell that is a DAO deposit (DAO type script, all-zero data, block number ≥ `starting_block_limiting_dao_withdrawing_lock`), scan **all** outputs for any DAO-typed output and verify its lock script size matches. Alternatively, enforce as a consensus rule that DAO deposit inputs and their corresponding withdrawing outputs must occupy the same index, and reject any transaction that violates this ordering.

## Proof of Concept
Construct a Phase 1 transaction:

```rust
// inputs[0] = fee cell (non-DAO lock, any data)
// inputs[1] = deposit cell (DAO type, data = [0u8;8], large lock script, e.g. 100-byte args)
// outputs[0] = withdrawing cell (DAO type, data = block_number_le, small lock script, e.g. 20-byte args)
// outputs[1] = change cell (non-DAO)

let tx = TransactionBuilder::default()
    .input(CellInput::new(fee_out_point, 0))
    .input(CellInput::new(deposit_out_point, 0))
    .output(
        CellOutput::new_builder()
            .capacity(deposit_capacity)
            .lock(small_lock_script)       // 20-byte args
            .type_(Some(dao_type_script))
            .build()
    )
    .output_data(block_number_le_bytes)
    .output(change_cell)
    .output_data(Bytes::default())
    .build();
```

`DaoScriptSizeVerifier` pairs `(fee_cell, withdrawing_cell)` → input not DAO → skip; pairs `(deposit_cell, change_cell)` → output not DAO → skip. Transaction is accepted and committed. In Phase 2, the DAO C script computes `occupied_capacity` using the 20-byte lock script, yielding a withdrawal larger than the correct amount computed from the 100-byte deposit lock script.

A unit test mirroring `test_dao_disables_different_lock_script_size` but with the DAO deposit at `resolved_inputs[1]` and the DAO withdrawing output at `outputs[0]` (non-DAO at `outputs[1]`) would confirm the verifier returns `Ok(())` instead of `Err(DaoLockSizeMismatch)`, proving the bypass.
