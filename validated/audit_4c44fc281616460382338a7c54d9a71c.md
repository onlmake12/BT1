Audit Report

## Title
Attacker-Controlled `declared_cycles` Used as `max_cycles` Causes Incorrect Peer Ban via `ExceededMaximumCycles` — (`tx-pool/src/process.rs`, `tx-pool/src/verify_mgr.rs`)

## Summary
In `_process_tx`, the `declared_cycles` value from a remote peer's `RelayTransactions` P2P message is used directly as `max_cycles` for script verification. When an attacker sends a valid transaction with `declared_cycles = actual_cycles - 1`, the verifier returns `ExceededMaximumCycles` — a `Script`-kind error — which `is_malformed_tx()` classifies as malformed, triggering `ban_malformed` on the sending peer. The `DeclaredWrongCycles` guard is never reached because `try_or_return_with_snapshot!` exits early on the verification error.

## Finding Description

The full code path is confirmed:

**Step 1:** `submit_remote_tx` passes `declared_cycles` with no validation into the pipeline. [1](#0-0) 

**Step 2:** The worker passes `entry.remote.map(|e| e.0)` (the attacker's `declared_cycles`) as the `declared_cycles` argument to `_process_tx`. [2](#0-1) 

**Step 3:** `_process_tx` sets `max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles())`. With `declared_cycles = Some(actual - 1)`, `max_cycles = actual - 1`. [3](#0-2) 

**Step 4:** `verify_rtx` runs with `max_cycles = actual - 1`. The script needs `actual` cycles, so it returns `ScriptError::ExceededMaximumCycles`, which converts to `ErrorKind::Script` via the `From<TransactionScriptError> for Error` impl. [4](#0-3) 

**Step 5:** `try_or_return_with_snapshot!(verified_ret, snapshot)` at line 734 exits early on this error. The `DeclaredWrongCycles` guard at lines 736–749 is **never reached**. [5](#0-4) 

**Step 6:** `after_process` evaluates `reject.is_malformed_tx()`. For `Reject::Verification(err)` with `ErrorKind::Script`, `is_malformed_from_verification` returns `!format!("{}", error).contains(ARGV_TOO_LONG_TEXT)`. The `ExceededMaximumCycles` display string (`"ExceededMaximumCycles: expect cycles <= {0}"`) does not contain `ARGV_TOO_LONG_TEXT` (`"@@@VM@@@UNEXPECTED@@@ARGV@@@TOOLONG@@@"`), so it returns `true`. [6](#0-5) [7](#0-6) [8](#0-7) 

**Step 7:** `ban_malformed(peer, ...)` is called on the sending peer. [9](#0-8) 

**No lower-bound check exists** on `declared_cycles` in `add_tx`, `resumeble_process_tx`, or `submit_remote_tx`. [10](#0-9) 

**Relay propagation of wrong `declared_cycles`:** The realistic attack vector for banning honest peers is that the attacker, acting as an intermediary relay node, sends a valid transaction to honest peer B with `declared_cycles = actual - 1`. If peer B's relay logic forwards `declared_cycles` as received, peer B will relay the same wrong value to the target node and be banned. The attacker need not connect to the target node directly.

## Impact Explanation

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. By systematically causing target nodes to ban honest relay peers, an attacker can deplete a node's peer slots, isolating it from the network. An isolated node cannot receive blocks or transactions, degrading its ability to participate in consensus. The attack requires no hashpower, no keys, and no privileged access — only a P2P connection and knowledge of a valid transaction's cycle count (trivially obtained by local execution).

## Likelihood Explanation

The `declared_cycles` field in `RelayTransactions` is fully attacker-controlled with no authentication or bounds checking anywhere in the pipeline. The attacker only needs to know the actual cycle count of any valid transaction. This is trivially obtained by running the script locally or observing a successful relay. The attack is repeatable with different transactions and different target peers, and requires no special resources.

## Recommendation

In `_process_tx`, do not use `declared_cycles` as `max_cycles`. Always verify with the consensus maximum as the upper bound, then perform the equality check afterward:

```rust
// Always use consensus max as the verification limit
let max_cycles = self.tx_pool_config.max_tx_verify_cycles;
let verified = try_or_return_with_snapshot!(
    verify_rtx(Arc::clone(&snapshot), Arc::clone(&rtx), tx_env, &verify_cache, max_cycles, command_rx).await,
    snapshot
);
// Then check declared matches actual
if let Some(declared) = declared_cycles {
    if declared != verified.cycles {
        return Some((Err(Reject::DeclaredWrongCycles(declared, verified.cycles)), snapshot));
    }
}
```

Note: `DeclaredWrongCycles` is currently `is_malformed_tx() = true` (line 92 of `tx_pool.rs`), which still triggers `ban_malformed`. To fully fix the issue, `DeclaredWrongCycles` should be changed to `is_malformed_tx() = false` so that peers relaying a transaction with wrong cycles (possibly received from a third party) are not banned — only the transaction is rejected. [11](#0-10) 

## Proof of Concept

1. Set up a CKB devnet node (target node T) and connect two peers: attacker node A and honest node B.
2. Craft a transaction using the `always_success` lock script. Determine its actual cycle count (e.g., 537 cycles) by running it locally.
3. From attacker node A, send a `RelayTransactions` P2P message to target node T with the valid transaction and `declared_cycles = 536`.
4. Observe in target node T's logs: `verify_rtx` runs with `max_cycles = 536`, returns `ExceededMaximumCycles`, `after_process` calls `ban_malformed(peer_A)`. Node A is banned for 3 days.
5. From honest node B, relay the same transaction with `declared_cycles = 537`.
6. Observe: transaction is accepted into the pool. Node A remains banned.
7. To demonstrate honest-peer banning: configure attacker node A as an intermediary relay for honest node B. Have A forward the transaction to B with `declared_cycles = 536`. If B's relay logic forwards `declared_cycles` as-is to T, T will ban B.

fn is_malformed_from_verification(error: &Error) -> bool {
    match error.kind() {
        ErrorKind::Transaction => error
            .downcast_ref::<TransactionError>()
            .expect("error kind checked")
            .is_malformed_tx(),
        ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
        ErrorKind::Internal => {
            error
                .downcast_ref::<InternalError>()
                .expect("error kind checked")
                .kind()
                == InternalErrorKind::CapacityOverflow
        }
        _ => false,
    }
}
