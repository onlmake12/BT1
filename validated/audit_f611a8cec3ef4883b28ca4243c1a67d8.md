Looking at the production code in `util/dao/src/lib.rs` and the test documentation in `util/dao/src/tests.rs`, I can identify a structural analog to the SushiToken delegate-accounting bug.

---

### Title
DAO Withdrawal Permanently Blocked by Header-Dep Index Width Mismatch Between Rust `DaoCalculator` and C VM — (`util/dao/src/lib.rs`)

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the full 8-byte (u64) witness `input_type` value as the `header_deps` array index. The on-chain C VM (dao.c) reads only the **lowest byte** of that same value. When a transaction sender encodes a witness index whose value exceeds 255 (e.g., `257 = 0x0000000000000101`), the two implementations resolve to **different header deps**: the C VM resolves to index `1` (the correct deposit block), while Rust resolves to index `257` (a different block). The Rust node then rejects the transaction via a block-number mismatch check, even though the C VM script would accept it. This permanently prevents the user from withdrawing their NervosDAO funds through the standard node.

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` decodes the witness `input_type` field as a full `u64` and uses it directly as the `header_deps` array index:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // full u64 used as index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The C VM (dao.c) reads only the **lowest byte** of the same 8-byte field. This is the exact same structural error as the SushiToken bug: the code uses the wrong representative entity (`full_u64_index` instead of `low_byte_index`) when performing the critical accounting lookup.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` explicitly documents this split:

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

With witness `input_type = 257` (`0x0000000000000101`):
- **C VM**: reads low byte `0x01 = 1` → `header_deps[1]` = correct deposit block → **accepts**
- **Rust**: reads full u64 `257` → `header_deps[257]` = wrong block → block-number check at line 105 fails → **rejects**

The block-number guard at line 105 is the safety net that catches the mismatch:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
```

This error propagates through `transaction_fee` (called during tx-pool admission) and through `dao_field_with_current_epoch` → `withdrawed_interests` (called during block assembly). In the block assembler, `calc_dao` is called with `?`, so a `DaoError` from any DAO withdrawal transaction causes `update_full` to return an error, stalling block template generation.

### Impact Explanation

A transaction sender who submits a DAO withdrawal with a witness index > 255 (e.g., 257, where `header_deps[1]` is the correct deposit block) will have their transaction:

1. **Accepted by the C VM** during script execution (correct deposit block found at low-byte index).
2. **Rejected by the Rust tx-pool** during fee calculation (`transaction_fee` → `transaction_maximum_withdraw` → wrong block resolved → `DaoError::InvalidOutPoint`).

The user's DAO funds are permanently locked: the transaction is valid on-chain (C VM accepts it) but the Rust node refuses to relay or include it. Additionally, if such a transaction somehow enters the pool (e.g., via a node that skips fee validation), the block assembler's `calc_dao` call fails with `?`, stalling block template updates for the miner.

### Likelihood Explanation

The attacker-controlled entry path is the standard `send_transaction` RPC. The trigger requires crafting a DAO withdrawal with more than 256 `header_deps` and encoding a witness index whose value exceeds 255 but whose low byte points to the correct deposit block. While unusual in normal usage, this is fully within the capability of any unprivileged transaction sender. A user with a large number of header deps (e.g., batching many DAO inputs) could hit this naturally.

### Recommendation

Align the Rust index resolution with the C VM by reading only the lowest byte of the witness `input_type` value:

```rust
// Before (reads full u64):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))

// After (reads only lowest byte, matching C VM behavior):
Ok(u64::from(header_deps_index_data.unwrap()[0]))
```

Alternatively, add an explicit validation that rejects witness indices > 255 at tx-pool admission time with a clear error message, so users are not silently blocked.

### Proof of Concept

The discrepancy is directly documented in the test suite. With `witness input_type = 257`:

- `header_deps[1]` = deposit block (number 100) — C VM resolves here via low byte
- `header_deps[257]` = withdraw block (number 200) — Rust resolves here via full u64
- Cell data encodes deposit block number = 100
- Rust check: `deposit_header.number() (200) != deposited_block_number (100)` → `DaoError::InvalidOutPoint`
- C VM check: `deposit_header.number() (100) == deposited_block_number (100)` → accepted

The test asserts `result.is_err()`, confirming the Rust rejection path is live. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
