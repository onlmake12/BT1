### Title
Griefing via Pre-Funding PDA Permanently Blocks Publisher Initialization — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize_publisher.rs`)

### Summary
An unprivileged attacker can permanently prevent any publisher from being initialized by sending 1 lamport to the deterministic `publisher_config` PDA before the authority calls `InitializePublisher`. The `system_instruction::create_account` call fails because the Solana system program rejects account creation when the destination already has a non-zero balance. No recovery path exists.

### Finding Description
`initialize_publisher` uses `system_instruction::create_account` to allocate the `publisher_config` PDA:

```rust
invoke_signed(
    &system_instruction::create_account(
        authority.key,
        publisher_config.key,
        lamports,
        publisher_config::SIZE.try_into()...,
        program_id,
    ),
    ...
)?;
``` [1](#0-0) 

The Solana system program's `CreateAccount` instruction requires the destination account to have exactly 0 lamports. If an attacker sends even 1 lamport to the PDA address (which is fully computable off-chain from public inputs), the `create_account` CPI will return `SystemError::AccountAlreadyInUse` and the entire transaction reverts. The program has no alternative initialization path and no instruction to reclaim lamports from an uninitialized PDA.

### Impact Explanation
The `publisher_config` PDA address is deterministic and immutable. Once pre-funded, the publisher can never be initialized, can never submit prices, and is permanently excluded from the oracle network. If the targeted publisher is critical to price aggregation, this causes protocol insolvency (stale/missing prices). The cost to the attacker is 1 lamport (~$0.000000001).

### Likelihood Explanation
Both inputs to `find_program_address` — `program_id` and the publisher's `pubkey` — are public on-chain. Any actor can compute the PDA and send 1 lamport before the authority acts. The attack is cheap, irreversible, and requires no privileged access.

### Recommendation
Replace `system_instruction::create_account` with the three-step pattern that handles pre-funded accounts:
1. `system_instruction::transfer` — top up to rent-exempt minimum (accounting for existing lamports)
2. `system_instruction::allocate` — set the data size
3. `system_instruction::assign` — transfer ownership to the program

This is the standard Solana mitigation for this class of griefing attack and is used by Anchor's `init_if_needed` and other production programs.

```rust
pub fn validate_publisher_config_for_init<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    bump: u8,
    publisher: &Pubkey,
    program_id: &Pubkey,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let publisher_config = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    // We use `find_program_address` to guarantee that only one publisher_config
    // is created per publisher.
    let (publisher_config_pda, expected_bump) = Pubkey::find_program_address(
        &[PUBLISHER_CONFIG_SEED.as_bytes(), &publisher.to_bytes()],
        program_id,
    );
    ensure!(ProgramError::InvalidInstructionData, bump == expected_bump);
    ensure!(ProgramError::InvalidArgument, publisher_config.is_writable);
    ensure!(
        ProgramError::MissingRequiredSignature,
        pubkey_eq(publisher_config.key, &publisher_config_pda)
    );
    Ok(publisher_config)
}
