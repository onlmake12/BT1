### Title
Unprivileged Caller Can Drain Subscription Balance via Inflated `tx.gasprice` in `updatePriceFeeds` - (File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol)

---

### Summary

The `updatePriceFeeds` function in the Scheduler contract is publicly callable by any address with no access control. The keeper fee paid from the subscription balance is calculated using `tx.gasprice`, which is fully controlled by the caller. An attacker can call `updatePriceFeeds` with an artificially inflated `tx.gasprice` to extract more funds from the subscription balance than the actual gas cost warrants, profiting from the `GAS_OVERHEAD` constant (30,000 gas units) scaled by the inflated price. This drains subscription balances far faster than subscription managers anticipated when funding their subscriptions.

---

### Finding Description

In `_processFeesAndPayKeeper`, the keeper fee is calculated as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
...
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
```

`GAS_OVERHEAD = 30000` is a fixed constant added to the actual gas consumed, and the entire sum is multiplied by `tx.gasprice`. The contract imposes **no cap on `tx.gasprice`**.

Since `updatePriceFeeds` has no access control modifier, any address can call it whenever update conditions are met. The attacker's economics per call:

| Item | Value |
|---|---|
| Attacker pays to network | `gasUsed × tx.gasprice` |
| Attacker receives from subscription | `(gasUsed + GAS_OVERHEAD) × tx.gasprice + keeperSpecificFee` |
| **Net profit** | **`GAS_OVERHEAD × tx.gasprice + keeperSpecificFee`** |

With `GAS_OVERHEAD = 30000` and `tx.gasprice = 1000 gwei`, the attacker profits `30,000 × 1000 gwei = 0.03 ETH + keeperSpecificFee` per call. The subscription balance is drained at a rate proportional to `tx.gasprice`, far exceeding what the subscription manager anticipated when funding the subscription.

The root cause is the combination of:
1. `updatePriceFeeds` being permissionless (by design for decentralization)
2. `_processFeesAndPayKeeper` using unbounded `tx.gasprice` in the fee formula
3. `GAS_OVERHEAD` acting as a profit multiplier for the attacker [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Subscription managers fund their subscriptions based on expected prevailing gas prices. An attacker calling `updatePriceFeeds` with an inflated `tx.gasprice` drains the subscription balance at a rate proportional to the gas price multiple used.

**Concrete example**: A subscription funded with 1 ETH at an expected gas price of 50 gwei. An attacker using 1000 gwei (20× normal) drains the balance ~20× faster. The subscription manager receives far fewer price updates than they paid for, and the attacker extracts the difference as profit. The attacker's profit is always positive regardless of `tx.gasprice` because `GAS_OVERHEAD × tx.gasprice + keeperSpecificFee > 0`.

This is a direct financial loss to subscription managers — their deposited ETH is consumed by an adversarial keeper rather than funding legitimate price updates. [4](#0-3) 

---

### Likelihood Explanation

The attack requires no privileged access, no leaked keys, and no governance manipulation. Any address can:

1. Enumerate active subscriptions via the public `getActiveSubscriptions` function
2. Fetch valid Pyth price update data from the public Hermes endpoint
3. Wait for the heartbeat period to elapse (e.g., 60 seconds for a typical subscription)
4. Submit `updatePriceFeeds` with an inflated `tx.gasprice`

The attacker is profitable on every successful call. The only constraint is that update conditions must be met, which happens on a predictable schedule. The attack is repeatable every heartbeat interval for the lifetime of the subscription. [5](#0-4) 

---

### Recommendation

Cap the effective gas price used in the keeper fee calculation to a protocol-configured maximum stored in `SchedulerState`:

```solidity
uint256 effectiveGasPrice = Math.min(tx.gasprice, _state.maxKeeperGasPrice);
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

The `maxKeeperGasPrice` parameter should be settable by the admin and calibrated to a reasonable upper bound for the target chain (e.g., 500 gwei on Ethereum mainnet). This preserves the keeper incentive model while preventing adversarial gas price inflation from draining subscription balances. [6](#0-5)

### Proof of Concept

```solidity
// Attacker contract
contract AttackerKeeper {
    IScheduler scheduler;
    
    function attack(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external {
        // Called with tx.gasprice = 1000 gwei (set via wallet/RPC)
        // GAS_OVERHEAD = 30000, so attacker profits:
        // 30000 * 1000 gwei = 0.03 ETH + keeperSpecificFee per call
        scheduler.updatePriceFeeds(subscriptionId, updateData);
        // msg.sender (this contract) receives totalKeeperFee
    }
}
```

**Steps**:
1. Create a subscription with `heartbeatSeconds = 60`, fund with 1 ETH.
2. Wait 60 seconds.
3. Fetch valid update data from `https://hermes.pyth.network/` for the subscribed price IDs.
4. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` with `tx.gasprice = 1000 gwei`.
5. Contract pays attacker `(gasUsed + 30000) × 1000 gwei + keeperSpecificFee`.
6. Attacker pays `gasUsed × 1000 gwei` to the network.
7. Net profit: `30000 × 1000 gwei = 0.03 ETH + keeperSpecificFee` per call.
8. Repeat every 60 seconds — subscription balance is drained ~20× faster than at normal 50 gwei gas prices. [7](#0-6) [8](#0-7)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
