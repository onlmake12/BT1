### Title
Requester-Controlled Callback Revert Causes Permanent Request DOS in Legacy Entropy Path - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

In `Entropy.sol`, when a provider has `defaultGasLimit == 0` (the legacy mode), `revealWithCallback` makes a direct, uncaught call to the requester's `_entropyCallback`. A malicious requester can deploy a contract whose `_entropyCallback` always reverts, permanently preventing the provider's keeper from fulfilling the request. Because `clearRequest` is called before the callback but the entire transaction reverts on callback failure, the request is never cleared from storage.

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`:

```
if (req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED) {
    // Safe path: excessivelySafeCall catches reverts, emits CallbackFailed
} else {
    // Legacy path: direct call, revert propagates
    clearRequest(provider, sequenceNumber);   // ← cleared first
    IEntropyConsumer(callAddress)._entropyCallback(...);  // ← revert here
}
``` [1](#0-0) 

When `req.gasLimit10k == 0` (set when `providerInfo.defaultGasLimit == 0`), the `else` branch is taken. [2](#0-1) 

Inside the `else` branch, `clearRequest` executes at line 666 **before** the external call at line 676. [3](#0-2) 

If `_entropyCallback` reverts, the EVM unwinds the entire transaction, including the `clearRequest` storage write. The request remains in storage with `callbackStatus == CALLBACK_NOT_STARTED`, and every subsequent keeper attempt to call `revealWithCallback` will hit the same revert. The request is permanently stuck.

The safe path (lines 578–660) uses `excessivelySafeCall` to catch reverts and transitions the request to `CALLBACK_FAILED`, allowing recovery. The legacy path has no such protection. [4](#0-3) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms this behavior: when a provider has no `defaultGasLimit` set and the consumer's callback reverts, `revealWithCallback` reverts and the request is never cleared. [5](#0-4) 

### Impact Explanation

- The provider's keeper service cannot fulfill the stuck request; every attempt wastes gas and reverts.
- The request permanently occupies a sequence number slot in the provider's chain.
- The requester's paid fee is non-refundable (by design), so the requester loses funds.
- At scale, an attacker can exhaust a legacy provider's sequence number range (`endSequenceNumber`) by flooding it with reverting-callback requests, fully DOSing the provider.

### Likelihood Explanation

- Any unprivileged user can call `requestWithCallback` or `requestV2` with a contract address as `msg.sender` whose `_entropyCallback` unconditionally reverts.
- Providers with `defaultGasLimit == 0` are still valid, registered providers; the production Fortuna keeper serves them.
- No special privilege, leaked key, or external oracle manipulation is required.
- The attack is cheap: the attacker only pays the request fee per stuck slot.

### Recommendation

1. **Immediate**: In the `else` branch of `revealWithCallback`, wrap the `_entropyCallback` call in a low-level `call` (or `excessivelySafeCall`) and handle the revert gracefully (e.g., emit a `CallbackFailed` event and leave the request in a recoverable state) rather than propagating the revert.
2. **Long-term**: Deprecate the `gasLimit10k == 0` legacy path entirely and require all providers to set a non-zero `defaultGasLimit` before accepting new callback requests.

### Proof of Concept

1. Deploy a malicious consumer contract:
```solidity
contract MaliciousConsumer is IEntropyConsumer {
    address entropy;
    constructor(address _entropy) { entropy = _entropy; }
    function getEntropy() internal view override returns (address) { return entropy; }
    function _entropyCallback(uint64, address, bytes32) external override {
        revert("DOS");
    }
    function request(address provider, bytes32 userRand) external payable returns (uint64) {
        return IEntropy(entropy).requestWithCallback{value: msg.value}(provider, userRand);
    }
}
```

2. Ensure the target provider has `defaultGasLimit == 0` (legacy mode).
3. Call `consumer.request{value: fee}(provider, userRand)` — this creates a request with `gasLimit10k == 0`.
4. Provider's keeper calls `revealWithCallback(provider, seqNum, userRand, providerProof)`.
5. Transaction reverts. The request is never cleared. Every subsequent keeper attempt also reverts.
6. Repeat step 3 to exhaust the provider's sequence number range. [6](#0-5) [7](#0-6)
