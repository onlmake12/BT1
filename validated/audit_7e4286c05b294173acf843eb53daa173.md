### Title
`n_sync_started` Counter Permanently Inflated Due to Missing Rollback When Peer Disappears Mid-Transition — (`sync/src/synchronizer/mod.rs`)

---

### Summary

In `start_sync_headers`, the global `n_sync_started` counter is atomically incremented **before** `peer_state.start_sync()` is called. If the peer is concurrently removed from `peers.state` between those two operations (a real race window because `peers.state` is a `DashMap` shared with the Relay protocol handler), `get_mut` returns `None`, `start_sync` is never invoked, and `n_sync_started` is never decremented. No recovery path exists: the `disconnected` handler only decrements the counter when `peer_state.sync_started() == true`, which is never set in this scenario. During IBD the IBD-guard (`if ibd && x != 0 { None }`) then permanently blocks every future call to `start_sync_headers`, freezing header synchronization.

---

### Finding Description

`start_sync_headers` in `sync/src/synchronizer/mod.rs` executes the following sequence for each candidate peer:

**Step 1 — commit the state transition (increment counter):**
```rust
if self.shared().state().n_sync_started()
    .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
        if ibd && x != 0 { None } else { Some(x + 1) }
    })
    .is_err()
{
    break;
}
``` [1](#0-0) 

**Step 2 — attempt to mark the peer as `Started`:**
```rust
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
}
``` [2](#0-1) 

If `get_mut` returns `None` (peer already removed), `start_sync` is never called, so `peer_state.chain_sync.headers_sync_state` remains `Initialized`/`SyncProtocolConnected` — never `Started`. [3](#0-2) 

The only places that decrement `n_sync_started` are `suspend_sync`, `tip_synced`, and `Peers::disconnected`. All three guard the decrement with `if peer_state.sync_started()`: [4](#0-3) [5](#0-4) 

Because `start_sync` was never called, `sync_started()` returns `false`, so none of these paths ever fire. `n_sync_started` is permanently stuck at the inflated value.

**Why the race window is real:** `peers.state` is a `DashMap` inside `SyncShared`, which is shared between the Sync protocol handler and the Relay protocol handler. The Relay protocol's `disconnected` callback calls `SyncState::disconnected`, which calls `Peers::disconnected`, which removes the peer from `peers.state`. The Relay handler runs on a separate async task from the Sync handler's `notify` callback, so the removal can interleave between the `fetch_update` (Step 1) and the `get_mut` (Step 2). [6](#0-5) 

---

### Impact Explanation

During IBD the guard `if ibd && x != 0 { None }` causes `fetch_update` to return `Err` whenever `n_sync_started >= 1`. With the counter stuck at 1 and no peer in the `Started` state, every subsequent call to `start_sync_headers` immediately breaks out of the loop without selecting any peer. The node can no longer download headers, which blocks block download and leaves the node permanently stuck in IBD. The only recovery is a process restart. [7](#0-6) 

---

### Likelihood Explanation

**Low.** The race window is narrow (between two consecutive lines in the same function). However, a malicious peer can repeatedly connect and disconnect at high frequency to increase the probability of hitting the window. Because the `SEND_GET_HEADERS_TOKEN` timer fires periodically, the attacker gets many attempts per minute. A single successful race permanently corrupts the counter for the lifetime of the process.

---

### Recommendation

1. **Rollback on miss:** If `get_mut` returns `None`, decrement `n_sync_started` before continuing:
   ```rust
   if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
       peer_state.start_sync(HeadersSyncController::from_header(&tip));
   } else {
       // peer disappeared; undo the counter increment
       self.shared().state().n_sync_started()
           .fetch_sub(1, Ordering::AcqRel);
       continue;
   }
   ```

2. **Atomic check-and-set:** Perform the counter increment and the `start_sync` call under the same `DashMap` shard lock so the peer cannot be removed between the two operations.

3. **Periodic sanity sweep:** Add a background check that recomputes `n_sync_started` from the actual count of peers with `sync_started() == true` and corrects any divergence, analogous to the background sweeper recommended in the original report.

---

### Proof of Concept

1. Node enters IBD (`is_initial_block_download() == true`).
2. Attacker peer connects; Sync protocol adds it to `peers.state` with state `SyncProtocolConnected`.
3. `SEND_GET_HEADERS_TOKEN` fires; `start_sync_headers` collects the peer in its candidate list.
4. `fetch_update` succeeds; `n_sync_started` becomes 1.
5. Concurrently, the Relay protocol's `disconnected` callback fires for the same peer (attacker disconnects), calling `SyncState::disconnected` → `Peers::disconnected` → `peers.state.remove(&peer)`. Because `sync_started()` is still `false` at this point, `n_sync_started` is **not** decremented.
6. Back in `start_sync_headers`, `peers.state.get_mut(&peer)` returns `None`; `start_sync` is skipped.
7. `n_sync_started` is now 1 with no peer in `Started` state.
8. All future `start_sync_headers` calls hit `if ibd && x != 0 { None }` → `break` immediately.
9. The node never downloads another header batch; IBD is permanently stalled until restart. [8](#0-7) [9](#0-8) [10](#0-9)

    fn start_sync_headers(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let now = unix_time_as_millis();
        let active_chain = self.shared.active_chain();
        let ibd = active_chain.is_initial_block_download();
        let peers: Vec<PeerIndex> = self
            .peers()
            .state
            .iter()
            .filter(|kv_pair| kv_pair.value().can_start_sync(now, ibd))
            .map(|kv_pair| *kv_pair.key())
            .collect();

        if peers.is_empty() {
            return;
        }

        let tip = self.better_tip_header();

        for peer in peers {
            // Only sync with 1 peer if we're in IBD
            if self
                .shared()
                .state()
                .n_sync_started()
                .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {

                .is_err()

            }

                }
            }

            debug!("Start sync peer={}", peer);
            active_chain.send_getheaders_to_peer(nc, peer, tip.number_and_hash());
        }
    }

    pub fn disconnected(&self, peer: PeerIndex) {
        if let Some(peer_state) = self.state.remove(&peer).map(|(_, peer_state)| peer_state) {
            if peer_state.sync_started() {
                // It shouldn't happen
                // fetch_sub wraps around on overflow, we still check manually
                // panic here to prevent some bug be hidden silently.
                assert_ne!(
                    self.n_sync_started.fetch_sub(1, Ordering::AcqRel),
                    0,
                    "n_sync_started overflow when disconnects"
                );
            }

            // Protection node disconnected
            if peer_state.peer_flags.is_protect {
                assert_ne!(
                    self.n_protected_outbound_peers
                        .fetch_sub(1, Ordering::AcqRel),
                    0,
                    "n_protected_outbound_peers overflow when disconnects"
                );
            }
        }
    }
