Audit Report

## Title
`RecentReject::put` Never Increments `self.total_keys_num`, Allowing Unbounded Disk Growth via Rejected Transaction Spam — (`tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put`, the incremented key count is computed into a local variable `total_keys_num` but is never written back to `self.total_keys_num`. Because `self.total_keys_num` is initialized from a RocksDB estimate (0 on a fresh DB) and never updated in `put()`, the `shrink()` guard is never triggered. An unprivileged attacker can spam low-fee-rate transactions indefinitely, causing the RocksDB TTL store to grow without bound and exhaust node disk space.

## Finding Description
In `RecentReject::put` (lines 62–69 of `tx-pool/src/component/recent_reject.rs`), the local binding `total_keys_num` holds `self.total_keys_num + 1`, but `self.total_keys_num` is never assigned:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // self.total_keys_num is never updated here
} else {
    self.shrink()?;
}
```

On a fresh DB, `self.total_keys_num` is initialized to 0 from the RocksDB estimate (lines 39–52). Every subsequent call to `put()` re-computes `0 + 1 = 1` as the local variable. Since `count_limit > 1` in any realistic configuration, the guard `total_keys_num > self.count_limit` is never true, and `shrink()` is never called. The only place `self.total_keys_num` is ever updated is inside `shrink()` itself (lines 110–111). Since `shrink()` is unreachable, `self.total_keys_num` stays frozen at 0 forever, and the underlying RocksDB store grows without bound.

The existing test at line 39 of `tx-pool/src/component/tests/recent_reject.rs` asserts `assert!(recent_reject.total_keys_num < 100)`, which passes trivially because `total_keys_num` is always 0 — it does not validate that `shrink()` actually ran.

## Impact Explanation
The `RecentReject` RocksDB TTL store accumulates entries indefinitely. Even with a finite TTL, RocksDB does not immediately reclaim space — compaction is not guaranteed to run promptly. A sustained spam attack can exhaust node disk space, causing the CKB node to crash or become unavailable. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack is fully unprivileged. Any P2P peer can submit transactions with a fee rate below the minimum threshold. `LowFeeRate` satisfies `should_recorded()` (it is not `Duplicated`) and does NOT satisfy `is_malformed_tx()`, so the attacker is never banned for submitting low-fee-rate transactions and can sustain the spam indefinitely without being disconnected.

## Recommendation
Add the missing assignment in `RecentReject::put` so that `self.total_keys_num` is incremented on every call:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // <-- missing line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```
