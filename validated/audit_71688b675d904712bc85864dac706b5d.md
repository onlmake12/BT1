Audit Report

## Title
Unsolicited `SendBlock` Full Deserialization Before Inflight Check Enables CPU Exhaustion — (File: `sync/src/synchronizer/block_process.rs`)

## Summary
`BlockProcess::execute()` unconditionally performs full block deserialization (byte copy + Blake2b hash computation) before checking whether the block was ever solicited. The `Synchronizer` has no rate limiter for `SendBlock` messages and does not wrap the handler in `tokio::task::block_in_place`, unlike all other sync message handlers. Any peer can flood the node with valid-molecule `SendBlock` messages for unsolicited blocks, consuming CPU per message with no throttle and blocking the async executor thread.

## Finding Description
In `sync/src/synchronizer/block_process.rs:35`, `BlockProcess::execute()` calls:
```rust
let block = Arc::new(self.message.block().to_entity().into_view());
```
`to_entity()` copies all raw bytes into owned types; `into_view()` computes Blake2b hashes for the block header and all transactions. Only after this work does line 43 call `shared.new_block_received(&block)`.

`new_block_received` (defined in `sync/src/types/mod.rs:1200-1207`) calls `write_inflight_blocks().remove_by_block(...)`. If the block hash was never inserted into `inflight_states` by the local node, `remove_by_block` returns `false` (via `.is_some()` on `None` at `types/mod.rs:791-818`), and `new_block_received` returns `false` at line 1206. The block is silently dropped, but all CPU work has already been done.

In `try_process` (`mod.rs:412-418`), the `SendBlock` arm is:
```rust
packed::SyncMessageUnionReader::SendBlock(reader) => {
    if reader.check_data() {
        BlockProcess::new(reader, self, peer, nc).execute()
    } else { ... }
}
```
Unlike `GetHeaders` (line 397-400), `SendHeaders` (402-405), and `GetBlocks` (407-410), which all wrap their handlers in `tokio::task::block_in_place`, `SendBlock` runs directly on the async executor thread. The `Synchronizer` struct (`mod.rs:357-362`) has no rate limiter field, in contrast to the `Relayer`.

## Impact Explanation
An attacker with a single P2P connection can send a continuous stream of valid-molecule `SendBlock` messages for blocks not in the local inflight set. Each message triggers full byte copy and Blake2b hash computation over the entire block payload (up to ~597 KB), with no throttle. Because processing runs on the async executor thread without `block_in_place`, it blocks the tokio runtime, degrading or making the node unresponsive to all other peers. This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires only a valid P2P handshake — no special privileges, no valid PoW. Valid molecule encoding is trivially constructable. The `check_data()` call at line 413 is a lightweight structural check, not a deserialization guard. The attack is repeatable indefinitely with a single connection and minimal bandwidth relative to the CPU cost imposed on the victim.

## Recommendation
1. **Move the inflight check before deserialization**: Use the zero-copy `SendBlockReader` to extract the block hash from raw molecule bytes and check `inflight_states` before calling `to_entity().into_view()`.
2. **Wrap `SendBlock` processing in `tokio::task::block_in_place`**, consistent with `GetHeaders`, `SendHeaders`, and `GetBlocks` handlers in `try_process`.
3. **Add a rate limiter to `Synchronizer`** for `SendBlock` messages, mirroring the existing `Relayer` rate limiter.

## Proof of Concept
Connect to a CKB node as a peer. Construct valid-molecule `SyncMessage::SendBlock` payloads containing a large block structure with any block hash not in the node's inflight set (no valid PoW required). Send these messages in a tight loop from a single connection. Observe CPU usage spike proportionally to block size per message with no throttling applied. The async executor thread will be blocked during each deserialization, measurably degrading response latency for all other connected peers and eventually rendering the node unresponsive.
