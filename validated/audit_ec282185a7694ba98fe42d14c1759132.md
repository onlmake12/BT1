### Title
ETH Refund via `.transfer()` Causes DoS for Contract Callers — (`lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate` refunds excess ETH to `msg.sender` using the low-level `.transfer()` primitive, which forwards only 2300 gas. Any contract caller whose `receive`/`fallback` function consumes more than 2300 gas will have the entire `verifyUpdate` call revert, permanently blocking that contract from using the Lazer verification service when it overpays the fee.

### Finding Description
In `PythLazer.verifyUpdate`, after checking that `msg.value >= verification_fee`, the contract refunds the surplus to the caller:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [1](#0-0) 

Solidity's `.transfer()` hard-caps the gas forwarded to the recipient at **2300**. This is sufficient for an EOA but not for a contract whose `receive` or `fallback` function performs any non-trivial work (e.g., emitting an event, updating a state variable, or delegating to a proxy). When such a contract sends `msg.value > verification_fee`, the `.transfer()` reverts, rolling back the entire `verifyUpdate` call.

The rest of the Pyth codebase (Entropy, Scheduler, Echo) correctly uses the `call{value: ...}("")` pattern with a return-value check:

```solidity
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "...");
``` [2](#0-1) 

`PythLazer.sol` is the only production contract in scope that still uses `.transfer()` for an ETH send to an arbitrary caller.

### Impact Explanation
Any on-chain contract integrating PythLazer (e.g., a DeFi protocol, a keeper bot contract, or a proxy-based wallet) that sends `msg.value > verification_fee` will have every `verifyUpdate` call revert if its `receive`/`fallback` uses more than 2300 gas. The contract is permanently unable to consume Lazer price updates via the normal fee path, effectively bricking its integration. The locked ETH is not at risk (the call reverts), but the service is unavailable to that caller class.

### Likelihood Explanation
Lazer is designed to be called by on-chain contracts that need verified price data. It is common for such contracts to send a small ETH buffer above the exact fee to avoid fee-change races. Modern proxy patterns (EIP-1967, EIP-1822) and smart-contract wallets routinely use `receive` functions that exceed 2300 gas. The trigger requires no privileged access — any unprivileged Lazer updater/consumer contract that overpays is affected.

### Recommendation
Replace `.transfer()` with the low-level `.call` pattern already used elsewhere in the codebase:

```solidity
if (msg.value > verification_fee) {
    (bool refunded, ) = payable(msg.sender).call{
        value: msg.value - verification_fee
    }("");
    require(refunded, "Refund failed");
}
```

### Proof of Concept
1. Deploy a contract `Consumer` whose `receive()` function emits an event (costs ~750 gas, but any proxy adds more).
2. `Consumer` calls `PythLazer.verifyUpdate{value: verification_fee + 1 wei}(update)`.
3. The `.transfer(1 wei)` call forwards 2300 gas to `Consumer.receive()`.
4. If `Consumer` is behind an EIP-1967 proxy (adds ~2600 gas to `receive`), the transfer reverts.
5. The entire `verifyUpdate` call reverts; `Consumer` cannot obtain a verified Lazer payload. [3](#0-2)
