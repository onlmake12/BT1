### Title
DAO Withdrawal Fee Validator Reads `header_dep_index` as Full `u64` While On-Chain C Script Reads Only Lowest Byte — (`File: util/dao/src/lib.rs`)

### Summary
`DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` reads the DAO `header_dep_index` from the witness as a full 8-byte little-endian `u64`, while the on-chain C DAO script (`dao.c`) reads only the lowest byte (treating the field as a `uint8_t`). For any index value ≥ 256, the two layers resolve to different `header_dep` entries. This causes the Rust tx-pool fee validator to reject valid DAO withdrawal transactions that the on-chain C VM would accept, and conversely to accept fee-check for transactions the C VM would reject — a concrete data-validation inconsistency between the pre-admission layer and the authoritative execution layer.

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit-header index from the witness `input_type` field:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and then uses it directly as a `usize` index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
``` [1](#0-0) 

The on-chain C DAO script (`dao.c`) reads the same field as a single byte (`uint8_t`), so for any encoded index value whose full `u64` representation differs from its lowest byte (i.e., any value ≥ 256), the two layers resolve to **different** `header_dep` entries.

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` explicitly documents this split:

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
``` [2](#0-1) 

The test asserts Rust returns an error because it resolves index 257 to the withdraw block (number 200), which does not match the cell data (deposit block number 100). The C VM, however, resolves the lowest byte (1) to the deposit block (number 100), which matches — so the C VM would accept the transaction.

The fee check is invoked in the tx-pool admission path via `check_tx_fee`:

```rust
let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
    .transaction_fee(rtx)
    .map_err(|err| Reject::Malformed(...))?;
``` [3](#0-2) 

### Impact Explanation

**Direction 1 — False rejection (primary impact):** A valid DAO withdrawal transaction (one the C VM would accept) with `header_dep_index ≥ 256` is rejected by `check_tx_fee` with `Reject::Malformed` before it ever reaches C VM script verification. The user cannot submit the transaction through the normal `send_transaction` RPC path. This is a denial-of-service against DAO withdrawals that use a large `header_deps` list.

**Direction 2 — False acceptance of fee check:** A transaction where Rust resolves to a valid deposit header (full u64 index) but the C VM resolves to a different header (lowest byte) passes the Rust fee check. It then proceeds to C VM script verification in the tx-pool, which correctly rejects it. No funds are at risk, but the fee-check layer provides no meaningful pre-filtering for this class of transaction.

### Likelihood Explanation

Normal DAO withdrawals carry 2 header_deps (deposit block + withdraw block), so the index is always 0 or 1 — well below 256. The discrepancy is only reachable when a transaction carries more than 256 header_deps, which is unusual but not impossible (e.g., a batch DAO withdrawal aggregating many deposits). The issue is already exercised by a test in the repository, confirming the code path is reachable and the behavior is reproducible. An unprivileged RPC caller submitting a `send_transaction` request is the attacker-controlled entry point.

### Recommendation

In `DaoCalculator::transaction_maximum_withdraw`, replace the full `u64` read with a single-byte read to match the C DAO script's behavior:

```rust
// Current (reads full u64):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))

// Should match dao.c (reads only lowest byte):
Ok(header_deps_index_data.unwrap()[0] as u64)
```

Additionally, add a bounds check that rejects any `input_type` value whose upper 7 bytes are non-zero, or document the intended maximum index and enforce it consistently in both the Rust validator and the C script.

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` directly demonstrates the discrepancy:

1. Build a DAO withdrawal transaction with 258 `header_deps`.
2. Place the deposit block at index 1 and the withdraw block at index 257.
3. Set `witness.input_type = 257u64` (little-endian 8 bytes).
4. C VM reads lowest byte = 1 → deposit block → block number matches cell data → **C VM accepts**.
5. Rust reads full u64 = 257 → withdraw block → block number (200) ≠ cell data (100) → **Rust rejects with `DaoError::InvalidOutPoint`**.
6. `check_tx_fee` propagates this as `Reject::Malformed`, blocking the transaction from the tx-pool. [4](#0-3) [5](#0-4) [6](#0-5)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
