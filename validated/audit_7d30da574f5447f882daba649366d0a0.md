### Title
Unprivileged Caller Can Steal `SignatureSet` Rent Lamports via `close_signature_set` — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

---

### Summary

The `CloseSignatureSet` instruction closes a `SignatureSet` account and transfers its rent lamports to `sol_destination`. However, `sol_destination` is only required to be a valid transaction signer — there is no check that it is the original creator/funder of the `signature_set` account. Any unprivileged attacker can call this instruction with their own keypair as `sol_destination`, providing any existing `PostedVaaV1` PDA that references the target `signature_set`, and steal the rent deposited by the original relayer.

---

### Finding Description

The `CloseSignatureSet` accounts struct is defined as:

```rust
#[derive(Accounts)]
pub struct CloseSignatureSet<'info> {
    #[account(mut)]
    sol_destination: Signer<'info>,          // ← any signer, no authority check

    #[account(
        seeds = [PostedVaaV1::SEED_PREFIX, posted_vaa.message_hash().as_ref()],
        bump,
        has_one = signature_set              // ← only checks field equality
    )]
    posted_vaa: Account<'info, LegacyAnchorized<PostedVaaV1>>,

    #[account(
        mut,
        close = sol_destination              // ← rent goes to whoever sol_destination is
    )]
    signature_set: Account<'info, AccountVariant<SignatureSet>>,
}
``` [1](#0-0) 

The constraints are:
- `sol_destination`: must be a `Signer` — **no authority or ownership check**.
- `posted_vaa`: must be a valid PDA and its `signature_set` field must equal the provided `signature_set` key (`has_one`).
- `signature_set`: closed with `close = sol_destination`, transferring all lamports to the attacker.

The `PostedVaaV1Info` struct stores the `signature_set` pubkey as a plain field set at `post_vaa` time: [2](#0-1) 

This field is set during the legacy `post_vaa` instruction to `signature_set.key()`: [3](#0-2) 

There is no stored record of who originally funded the `signature_set` account, and no check is performed to enforce that `sol_destination` matches any such creator. The `SignatureSet` account itself contains only `sig_verify_successes`, `message_hash`, and `guardian_set_index` — no `creator` or `payer` field: [4](#0-3) 

---

### Impact Explanation

An attacker can drain rent lamports from any `SignatureSet` account associated with a legacy-path `PostedVaaV1` (i.e., where `posted_vaa.signature_set != Pubkey::default()`). The `SignatureSet` account for a 19-guardian set is approximately 67 bytes, holding ~0.00089 SOL in rent. While small per account, this can be executed against every existing `SignatureSet` on-chain, stealing the full rent balance from the original relayer who funded each one. The relayer permanently loses their deposited rent — they can no longer call `close_signature_set` themselves because the account will already be closed.

---

### Likelihood Explanation

The attack requires no special privileges: any funded keypair can sign the transaction. All inputs (`posted_vaa` PDA, `signature_set` pubkey) are publicly discoverable on-chain. The attack is permissionless, repeatable, and requires no coordination. Any `PostedVaaV1` created via the legacy `post_vaa` path (which sets a non-default `signature_set`) is a valid target.

---

### Recommendation

Add a constraint that `sol_destination` must equal the account that originally funded the `signature_set`. Since `SignatureSet` has no stored `payer` field, the fix requires one of:

1. **Add a `payer` field to `SignatureSet`** at creation time (in `verify_signatures` / `post_vaa`) and enforce `sol_destination == signature_set.payer` in `CloseSignatureSet`.
2. **Require `sol_destination` to be the `PostedVaaV1` payer** if that is recorded, or restrict the instruction so only the account that paid for the `posted_vaa` can reclaim the `signature_set` rent.
3. As a minimal short-term fix, remove the `close = sol_destination` macro and instead transfer lamports only to a hardcoded authority derived from the `posted_vaa` payer, once that field is tracked.

---

### Proof of Concept

```rust
// Setup: relayer creates signature_set and posted_vaa via legacy path
// posted_vaa.signature_set == victim_sig_set.key()

// Attacker (different keypair) calls:
let ix = close_signature_set(
    sol_destination = attacker_keypair.pubkey(),  // attacker's own key
    posted_vaa     = victim_posted_vaa_pda,       // publicly known PDA
    signature_set  = victim_sig_set,              // publicly known from posted_vaa field
);
// Constraints pass:
//   - attacker_keypair signs → Signer check passes
//   - posted_vaa PDA seeds valid → seeds/bump check passes
//   - posted_vaa.signature_set == victim_sig_set.key() → has_one passes
//   - close = sol_destination → all lamports transferred to attacker

// Assert: attacker lamports increased by signature_set rent amount
// Assert: relayer lamports unchanged (they received nothing)
// Assert: signature_set account closed
```
