### Title
Inbound Peer GetNodes Triggers Unbounded Peer-Store Lookup Without Direction Guard — (`network/src/protocols/discovery/mod.rs`)

### Summary

The `DiscoveryProtocol::received` handler processes `GetNodes` messages from **any** session regardless of direction. An unprivileged remote inbound peer can send one `GetNodes` message per connection, causing the node to execute an O(N) Fisher-Yates shuffle over the entire peer store and serialize up to 1,000 addresses per connection. Because the per-session `received_get_nodes` guard only prevents duplicates within a single session, an attacker cycling through fresh inbound connections bypasses it entirely. The `misbehave` implementation carries a `FIXME` and imposes no IP-level ban, so there is no durable penalty.

---

### Finding Description

**Missing direction guard in the GetNodes handler**

`SessionState::new` sends `GetNodes` only for outbound sessions: [1](#0-0) 

For inbound sessions `received_get_nodes` is initialized to `false` and no message is sent. However, the `received` handler applies no symmetric restriction — it processes `GetNodes` from any session: [2](#0-1) 

The only guard is the per-session duplicate check at line 110, which is bypassed by opening a new connection for each `GetNodes`.

**Expensive peer-store operation triggered per connection**

`addr_mgr.get_random(2500, required_flags)` calls `peer_store.fetch_random_addrs(2500, flags)`: [3](#0-2) 

Which calls `addr_manager.fetch_random(2500, filter)` — a full Fisher-Yates shuffle over **all** stored addresses: [4](#0-3) 

The benchmark confirms the store is exercised at 10,000–20,000 entries: [5](#0-4) 

After the shuffle, up to `MAX_ADDR_TO_SEND = 1000` entries are serialized and sent: [6](#0-5) 

**No IP-level ban on misbehavior**

The `misbehave` implementation carries a `FIXME` and unconditionally returns `Disconnect` with no IP ban or penalty: [7](#0-6) 

**Inbound connection limit does not prevent cycling**

The default configuration allows up to `max_peers − max_outbound_peers = 125 − 8 = 117` inbound connections: [8](#0-7) 

When at capacity, `accept_peer` evicts an existing inbound peer to make room for the new one: [9](#0-8) 

This means the attacker's new connection is accepted, sends `GetNodes`, triggers the expensive lookup, and then disconnects — the eviction mechanism does not prevent the cycle.

---

### Impact Explanation

Each inbound connection that sends one `GetNodes` forces:
1. An O(N) Fisher-Yates shuffle over the entire `random_ids` vector (up to ~20,000 entries) under the `peer_store` mutex.
2. Serialization of up to 1,000 `Node` entries.
3. A network write of the `Nodes` response.

Because `with_peer_store_mut` serializes access, a flood of concurrent inbound connections queues these operations, stalling all other peer-store consumers (outbound dialing, feeler, sync). Sustained attack across many nodes degrades block and transaction processing.

---

### Likelihood Explanation

The attack requires only the ability to open TCP connections to a listening CKB node — no credentials, no PoW, no privileged role. The secio handshake adds per-connection overhead for the attacker but does not prevent the attack at scale. The missing direction check is a straightforward protocol-level omission, not a subtle race condition.

---

### Recommendation

1. **Reject `GetNodes` from inbound sessions**: In `received`, check `context.session.ty.is_inbound()` and return (or call `misbehave`) before executing the peer-store lookup.
2. **Implement IP-level rate limiting / banning** in `misbehave` (resolve the `FIXME`).
3. **Add a global rate limit** on `get_random` calls (e.g., one response per IP per N seconds).

---

### Proof of Concept

Call sequence:

```
attacker connects (inbound TCP + secio handshake)
  → EventHandler::handle_session_open → PeerRegistry::accept_peer (evicts one peer if full)
  → DiscoveryProtocol::connected → SessionState::new (inbound: received_get_nodes=false, no GetNodes sent)
  → attacker sends DiscoveryMessage::GetNodes { count: 1000, ... }
  → DiscoveryProtocol::received
      state.received_get_nodes == false  → no disconnect
      state.received_get_nodes = true
      addr_mgr.get_random(2500, flags)
        → peer_store.fetch_random_addrs(2500, flags)
          → addr_manager.fetch_random(2500, filter)   ← O(N) shuffle, mutex held
      serialize up to 1000 Node entries
      send Nodes response
  → attacker disconnects, reconnects with new peer_id, repeat
```

A benchmark test measuring CPU time for N rapid inbound connections each sending `GetNodes` would show linear growth in `fetch_random_addrs` time with no bound other than the inbound connection limit, which the eviction mechanism renders ineffective as a defense.

    fn get_random(&mut self, n: usize, flags: Flags) -> Vec<(Multiaddr, Flags)> {
        let fetch_random_addrs = self
            .network_state
            .with_peer_store_mut(|peer_store| peer_store.fetch_random_addrs(n, flags));
        let addrs = fetch_random_addrs
            .into_iter()
            .filter_map(|paddr| {
                if !self.is_valid_addr(&paddr.addr) {
                    return None;
                }
                let f = Flags::from_bits_truncate(paddr.flags);
                Some((paddr.addr, f))
            })
            .collect();
        trace!("Discovered random addrs: {:?}", addrs);
        addrs
    }

    pub fn fetch_random<F>(&mut self, count: usize, filter: F) -> Vec<AddrInfo>
    where
        F: Fn(&AddrInfo) -> bool,
    {
        let mut duplicate_ips = HashSet::new();
        let mut addr_infos = Vec::with_capacity(count);
        let mut rng = rand::thread_rng();
        let now_ms = ckb_systemtime::unix_time_as_millis();
        for i in 0..self.random_ids.len() {
            // reuse the for loop to shuffle random ids
            // https://en.wikipedia.org/wiki/Fisher%E2%80%93Yates_shuffle
            let j = rng.gen_range(i..self.random_ids.len());
            self.swap_random_id(j, i);
            let addr_info: AddrInfo = self.id_to_info[&self.random_ids[i]].to_owned();
            match multiaddr_to_socketaddr(&addr_info.addr) {
                Some(socket_addr) => {
                    let ip = socket_addr.ip();
                    let is_unique_ip = !duplicate_ips.contains(&ip);
                    // A trick to make our tests work
                    // TODO remove this after fix the network tests.
                    let is_test_ip = ip.is_unspecified() || ip.is_loopback();
                    if (is_test_ip || is_unique_ip)
                        && addr_info.is_connectable(now_ms)
                        && filter(&addr_info)
                    {
                        duplicate_ips.insert(ip);
                        addr_infos.push(addr_info);
                    }
                }
                None => {
                    if filter(&addr_info) {
                        if addr_info.is_connectable(now_ms)
                            || addr_info
                                .addr
                                .iter()
                                .any(|p| matches!(p, Protocol::Onion3(_)))
                        {
                            addr_infos.push(addr_info);
                        } else {
                            debug!(
                                "addr {:?} is not connectable and not an onion address",
                                addr_info.addr
                            );
                        }
                    }
                }
            }
            if addr_infos.len() == count {
                break;
            }
        }
        addr_infos
