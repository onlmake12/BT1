### Title
Duplicate Header Hashes Inserted into `unknown_header_list` Without Existence Check During IBD — (`sync/src/types/mod.rs`)

### Summary

During Initial Block Download (IBD), when a trusted outbound/whitelist/protect peer sends a `GetHeaders` P2P message containing duplicate block locator hashes, those duplicates are inserted verbatim into the per-peer `unknown_header_list` (`Vec<Byte32>`) without any deduplication check. This is a direct analog of the reported pattern: an item is added to a list without verifying it is not already present.

### Finding Description

`PeerState::unknown_header_list` is declared as a plain `Vec<Byte32>`: [1](#0-0) 

The function that appends to it is `Peers::insert_unknown_header_hash`: [2](#0-1) 

It calls `.push(hash)` unconditionally — no existence check is performed before insertion.

This function is called from `SyncShared::insert_peer_unknown_header_list`, which iterates over the raw locator hashes received from the peer's `GetHeaders` message: [3](#0-2) 

The `header_list` argument is built directly from the peer-supplied `block_locator_hashes` field of the `GetHeaders` message, with no deduplication: [4](#0-3) 

If the peer sends `[H1, H1, H2]` and neither H1 nor H2 is in the local header map, both copies of H1 are pushed into `unknown_header_list`, producing `[H1, H1, H2]`.

The block fetcher later consumes this list via `take_unknown_last` (pop from end) and, when a hash is still unknown, re-inserts it via `insert_unknown_header_hash`: [5](#0-4) 

This second call site also performs no duplicate check before pushing back.

### Impact Explanation

1. **Redundant sync work**: The block fetcher processes the same hash multiple times per fetch cycle, wasting CPU and repeated header-map lookups.
2. **Delayed IBD progress**: `get_best_known_less_than_tip_and_unknown_empty` excludes a peer from certain sync scheduling while its `unknown_header_list` is non-empty. Duplicates inflate the list length, keeping the peer excluded longer than necessary and slowing IBD convergence.
3. **Memory inflation**: The list grows proportionally to the number of duplicates injected. While bounded by `MAX_LOCATOR_SIZE` per message, repeated `GetHeaders` messages (each populating the list after it drains) can re-trigger the issue.

### Likelihood Explanation

- The attacker must be connected as an outbound, whitelist, or protect peer. Outbound peers are normal peers the local node discovers and connects to via the P2P discovery protocol — any peer reachable on the network qualifies.
- The attack requires only sending a `GetHeaders` message with repeated hashes in the locator during IBD, which is trivially constructable.
- No special privilege, key, or majority hashpower is required.

### Recommendation

In `insert_peer_unknown_header_list`, deduplicate the incoming `header_list` before iterating, or change `unknown_header_list` from `Vec<Byte32>` to a `LinkedHashSet<Byte32>` (already available in the codebase at `util/src/linked_hash_set.rs`) to preserve insertion order while preventing duplicates. At minimum, add a contains-check inside `insert_unknown_header_hash` before calling `.push`.

### Proof of Concept

1. A malicious peer connects to a victim CKB node as an outbound peer.
2. The victim node enters IBD.
3. The malicious peer sends a `GetHeaders` P2P message with `block_locator_hashes = [H_unknown, H_unknown, H_unknown, ...]` (all identical, unknown to the victim), up to `MAX_LOCATOR_SIZE` entries.
4. `GetHeadersProcess::execute` calls `shared.insert_peer_unknown_header_list(peer, block_locator_hashes)`.
5. Since `unknown_header_list_is_empty` is true, the loop runs and calls `insert_unknown_header_hash` for each copy of `H_unknown`, inserting N duplicates into the `Vec`.
6. The block fetcher now pops `H_unknown`, finds it still unknown, pushes it back, and breaks — repeating this on every fetch cycle until `H_unknown` is resolved, while the N−1 remaining duplicates sit in the list keeping `unknown_header_list_is_empty` false and blocking the peer from normal sync scheduling.     pub fn insert_peer_unknown_header_list(&self, pi: PeerIndex, header_list: Vec<Byte32>) {
        // update peer's unknown_header_list only once
        if self.state().peers.unknown_header_list_is_empty(pi) {
            // header list is an ordered list, sorted from highest to lowest,
            // so here you discard and exit early
            for hash in header_list {
                if let Some(header) = self.shared().header_map().get(&hash) {
                    self.state()
                        .peers
                        .may_set_best_known_header(pi, header.as_header_index());
                    break;
                } else {
                    self.state().peers.insert_unknown_header_hash(pi, hash)
                }
            }
        }
    }
