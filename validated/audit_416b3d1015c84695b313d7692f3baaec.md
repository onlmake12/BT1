### Title
`PythAggregatorV3.latestRoundData()` Returns Stale Price Without Any Staleness Validation — (`target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter contract, calls `getPriceUnsafe()` in every price-reading function (`latestRoundData`, `latestAnswer`, `getRoundData`). This bypasses all staleness and validity checks. During a market crash or network disruption, the adapter silently returns an arbitrarily old price without reverting, directly mirroring the Chainlink min/max circuit-breaker class of vulnerability.

---

### Finding Description

`PythAggregatorV3` is the production SDK contract Pyth provides for protocols migrating from Chainlink. It implements `AggregatorV3Interface` so existing Chainlink consumers can swap in Pyth with minimal code changes.

Every price-reading function in the contract calls `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
roundId = uint80(price.publishTime);
return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);

// latestAnswer — line 54
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
return int256(price.price);

// getRoundData — line 89
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
```

`getPriceUnsafe` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee. The safe alternative, `getPriceNoOlderThan(id, age)`, is available in the same `IPyth` interface but is never used here.

The adapter performs **zero** of the following checks:
- No staleness check (`price.publishTime` vs `block.timestamp`)
- No confidence interval check (`price.conf` / `price.price` ratio)
- No positivity check on `price.price`

The contract is explicitly recommended in Pyth's official migration guide as a drop-in Chainlink replacement.

---

### Impact Explanation

Chainlink-compatible lending/borrowing protocols that adopt `PythAggregatorV3` as their oracle will call `latestRoundData()`. If the Pyth price feed for a volatile asset becomes stale during a market crash (e.g., due to a network disruption, Hermes outage, or insufficient keeper activity), the adapter returns the last cached pre-crash price without reverting. An attacker can then:

1. Observe that the on-chain Pyth price is stale and inflated relative to the true market price.
2. Borrow the maximum amount against collateral valued at the stale (inflated) price.
3. Walk away with bad debt once the price is eventually corrected.

This is the exact scenario that caused the Venus/LUNA incident referenced in the original report — the oracle returned a wrong price without reverting, allowing over-borrowing.

---

### Likelihood Explanation

- `PythAggregatorV3` is a production SDK contract actively recommended for Chainlink migration.
- Pyth is a pull oracle; price feeds only update when someone submits an update transaction. During high-volatility events, keeper bots may lag or fail.
- Protocols that follow the Chainlink convention of checking `updatedAt` from `latestRoundData()` will see a stale timestamp and can protect themselves — but many protocols do not perform this check, and the adapter itself never reverts.
- The vulnerability is triggered by any scenario where the Pyth feed is not updated promptly, which is a realistic operational condition.

---

### Recommendation

Replace `getPriceUnsafe()` with `getPriceNoOlderThan()` and add a configurable `maxAge` parameter to the constructor:

```solidity
uint public maxAge;

constructor(address _pyth, bytes32 _priceId, uint _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // ...
}
```

This causes the adapter to revert (with `StalePrice`) when the price is too old, matching the safety behavior that Chainlink-compatible protocols expect.

---

### Proof of Concept

1. A lending protocol deploys `PythAggregatorV3` for a volatile asset (e.g., a small-cap token) following Pyth's migration guide.
2. The asset experiences a sudden 90% price crash. Keeper bots fail to submit a Pyth update in time (network congestion, outage, or simply no one calling `updateFeeds`).
3. An attacker calls `latestRoundData()` on the adapter — it returns the pre-crash price (e.g., $100) without reverting, because `getPriceUnsafe()` has no staleness guard.
4. The attacker deposits the crashed asset as collateral, which the protocol values at $100 (stale price).
5. The attacker borrows the maximum allowed amount against this inflated collateral value.
6. When the price is eventually updated on-chain to $10, the collateral is worth far less than the debt — the protocol accumulates bad debt with no recourse.

**Root cause**: `PythAggregatorV3.latestRoundData()` (and sibling functions) unconditionally call `getPriceUnsafe()`. [1](#0-0) 

**Safe alternative available but unused**: `getPriceNoOlderThan(id, age)` in `AbstractPyth.sol` reverts with `StalePrice` when the price is too old. [2](#0-1) 

**Deployment context**: Pyth's official migration guide explicitly recommends deploying `PythAggregatorV3` as a Chainlink drop-in for lending protocols. [3](#0-2)

This guide explains how to migrate an EVM application that uses Chainlink price feeds to Pyth price feeds.
Pyth provides a Chainlink-compatible interface for its price feeds to make this process simple.
There are two main steps to the migration:

1. Deploy the [`PythAggregatorV3`](https://github.com/pyth-network/pyth-crosschain/blob/main/target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol) contract to provide a Chainlink-compatible feed interface.
2. Schedule price updates for the feeds required by your app.

## Install Pyth SDKs

The `PythAggregatorV3` contract is provided in the [Pyth Price Feeds Solidity SDK](https://github.com/pyth-network/pyth-crosschain/tree/main/target_chains/ethereum/sdk/solidity).
Add this SDK to the dependencies of your EVM contract.

**Truffle/Hardhat**

If you are using Truffle or Hardhat, simply install the NPM package:

```bash copy
npm install @pythnetwork/pyth-sdk-solidity
```

**Foundry**

If you are using Foundry, you will need to create an NPM project if you don't already have one.
From the root directory of your project, run:

```bash copy
npm init -y
npm install @pythnetwork/pyth-sdk-solidity
```

Then add the following line to your `remappings.txt` file:

```text copy
@pythnetwork/pyth-sdk-solidity/=node_modules/@pythnetwork/pyth-sdk-solidity
```

## Deploy Adapter Contract

First, deploy the `PythAggregatorV3` contract from `@pythnetwork/pyth-sdk-solidity` as a replacement for your application's Chainlink price feeds.
`PythAggregatorV3` is an adapter contract that wraps the Pyth contract and implements Chainlink's `AggregatorV3Interface`.

One important difference between Pyth and Chainlink is that the Pyth contract holds data for all price feeds; in contrast, Chainlink has separate instances of `AggregatorV3Interface` for each feed.
The adapter contract resolves this discrepancy by wrapping a single Pyth price feed.
Users should deploy an instance of this adapter for every required price feed, then point their existing app to the addresses of the deployed adapter contracts.
