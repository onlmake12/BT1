### Title
`OrphanPool::remove_orphan_tx` Removes Parent Without Cascading to Dependent Children, Permanently Stranding Them — (`tx-pool/src/component/orphan.rs`)

### Summary

`OrphanPool::remove_orphan_tx` removes a single orphan transaction from the pool without cascading the removal to any orphan transactions that depend on its outputs. When a parent orphan is evicted (via random size-limit eviction, expiry, or the `remove_transaction` RPC), its orphan children remain in the pool indefinitely — they can never be promoted to pending because their parent is gone, and they occupy pool slots until they individually expire (~80 minutes). An unprivileged peer can exploit the random eviction path to deliberately strand orphan children and exhaust the orphan pool's 100-slot capacity.

### Finding Description

`OrphanPool` maintains two maps:

- `entries: HashMap<ProposalShortId, Entry>` — the orphan transactions themselves
- `by_out_point: HashMap<OutPoint, HashSet<ProposalShortId>>` — maps each input `OutPoint` of an orphan to the set of orphan IDs that consume it [1](#0-0) 

`remove_orphan_tx` removes the target entry and cleans up its own input references in `by_out_point`, but does **not** look up or remove orphan children that reference the removed transaction's **outputs**: [2](#0-1) 

After `remove_orphan_tx(tx_A)`:
- `tx_A` is gone from `entries`
- `by_out_point[tx_A_input]` is cleaned up
- But `by_out_point[tx_A_output_0]` → `{tx_B}` is **not** cleaned up
- `tx_B` remains in `entries` with no reachable parent — it will never be found by `find_by_previous` (since that requires the parent tx to be present to call `output_pts()` on it), and will never be promoted to pending [3](#0-2) 

This removal is triggered in three places:

**1. Random size-limit eviction** — when the pool exceeds 100 entries, a random orphan is evicted with no cascade: [4](#0-3) 

**2. Expiry eviction** — expired orphans are removed individually: [5](#0-4) 

**3. `remove_transaction` RPC** — the RPC handler calls `remove_orphan_tx` on a single ID and returns early, leaving children stranded: [6](#0-5) 

The `remove_transaction` RPC is documented as removing "a transaction and all transactions which depends on it" — but this guarantee only holds for the `TxPool` (pending/proposed), not for the `OrphanPool`: [7](#0-6) 

The `TxPool::remove_tx` correctly uses `remove_entry_and_descendants`, but the orphan branch does not: [8](#0-7) 

### Impact Explanation

Stranded orphan children permanently occupy slots in the 100-slot orphan pool until they individually expire (`ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL` ≈ 80 minutes). An attacker who can keep the pool filled with stranded children prevents legitimate orphan transactions from being accepted, degrading transaction relay for the node. Additionally, `by_out_point` entries for the removed parent's outputs remain as stale references, causing unbounded memory growth proportional to the number of stranded children. [9](#0-8) 

### Likelihood Explanation

Any unprivileged P2P peer can relay transactions. The attack requires sending at most 101 transactions (to fill the pool and trigger eviction). The random eviction makes the outcome probabilistic per attempt, but the attacker can repeat the pattern cheaply. The `remove_transaction` RPC path is deterministic and requires only local RPC access (a supported local CLI/RPC user).

### Recommendation

`remove_orphan_tx` should cascade: after removing the target entry, collect all orphan IDs in `by_out_point` that reference any of the removed transaction's output `OutPoint`s, and recursively remove them as well. Alternatively, introduce a `remove_orphan_tx_and_descendants` function (mirroring `remove_entry_and_descendants` in `PoolMap`) and use it in all eviction and RPC removal paths. [10](#0-9) 

### Proof of Concept

```
State: OrphanPool is at capacity (100 entries).

1. Attacker relays tx_parent (orphan — its own parent is unknown on-chain).
   → Pool now has 101 entries, limit_size() fires.
   → A random orphan is evicted. Suppose tx_parent is evicted.

2. Attacker relays tx_child (orphan — its input is tx_parent's output).
   → tx_child is added to entries.
   → by_out_point[tx_parent_output_0] = {tx_child}.

3. tx_parent is gone. tx_child is now stranded:
   - find_by_previous() is never called with tx_parent (it's not in the pool).
   - tx_child will never be promoted to pending.
   - tx_child occupies a pool slot for ~80 minutes.

4. Attacker repeats steps 1–3 to fill the pool with stranded children,
   blocking all legitimate orphan transactions from being accepted.
```

Alternatively, via the `remove_transaction` RPC:

```
1. tx_parent is in the orphan pool.
2. tx_child (orphan, child of tx_parent) is also in the orphan pool.
3. Caller invokes remove_transaction(tx_parent_hash).
4. remove_tx() finds tx_parent in the orphan pool, calls remove_orphan_tx(&id), returns true.
5. tx_child remains stranded in the orphan pool despite the RPC documentation
   promising removal of "all transactions which depends on it".
``` [11](#0-10) [12](#0-11)

    pub fn remove_orphan_tx(&mut self, id: &ProposalShortId) -> Option<Entry> {
        self.entries.remove(id).inspect(|entry| {
            debug!("remove orphan tx {}", entry.tx.hash());
            for out_point in entry.tx.input_pts_iter() {
                if let Some(ids_set) = self.by_out_point.get_mut(&out_point) {
                    ids_set.remove(id);

                    if ids_set.is_empty() {
                        self.by_out_point.remove(&out_point);
                    }
                }
            }
        })
    }

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

        {
            let mut queue = self.verify_queue.write().await;
            if queue.remove_tx(&id).is_some() {
                return true;
            }
        }
        {
            let mut orphan = self.orphan.write().await;
            if orphan.remove_orphan_tx(&id).is_some() {
                return true;
            }

        let mut tx_pool = self.tx_pool.write().await;
        tx_pool.remove_tx(&id)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
