### Title
Attacker-Controlled Snappy Varint Causes 2× Memory Amplification Before Decompression in `LengthDelimitedCodecWithCompress::decode` — (`network/src/compress.rs`)

### Summary
`LengthDelimitedCodecWithCompress::decode` allocates a zeroed buffer sized by the attacker-supplied snappy varint header (`decompressed_bytes_len`) before attempting decompression. The guard only checks `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (8 MB), but `MAX_UNCOMPRESSED_LEN` is 2× the RelayV3 `max_frame_length` (4 MB). An unprivileged peer can send a 4 MB wire frame that triggers an 8 MB heap allocation, then fail decompression, drop the connection, and reconnect — with no ban applied.

### Finding Description
In `decode()` at line 242:

```rust
let mut buf = BytesMut::zeroed(decompressed_bytes_len);
match SnapDecoder::new().decompress(&data[1..], &mut buf) {
    Ok(_) => Ok(Some(buf)),
    Err(e) => {
        debug!("snappy decompress error: {:?}", e);
        Err(io::ErrorKind::InvalidData.into())
    }
}
```

The allocation at line 242 is unconditional once `decompressed_bytes_len ≤ MAX_UNCOMPRESSED_LEN`. An attacker crafts a payload where:
- Byte 0: `0b1000_0000` (COMPRESS_FLAG)
- Bytes 1–5: snappy varint encoding `8388608` (8 MB)
- Remaining bytes: arbitrary garbage

`decompress_len` reads the varint and returns 8388608. The check at line 235 passes. `BytesMut::zeroed(8388608)` allocates 8 MB. `decompress` fails on the garbage body. The error propagates, the connection is dropped (no ban), and the 8 MB allocation is freed — but the peak RSS spike already occurred.

### Impact Explanation
With 117 simultaneous inbound connections (default config), each sending one such frame, the node experiences a transient peak allocation of 117 × 8 MB ≈ **936 MB** above baseline. Since no ban is applied on codec errors, the attacker can reconnect immediately and repeat at a rate limited only by TCP/noise handshake overhead. On nodes with constrained memory (e.g., 2–4 GB RAM running a full node with RocksDB cache), this can cause OOM termination or severe memory pressure degrading node performance.

### Likelihood Explanation
The attack requires only an inbound TCP connection — no authentication, no PoW, no valid CKB message. The snappy varint is trivially crafted (a few bytes). The absence of a ban on codec-level errors means the attacker can cycle connections indefinitely. This is concretely reachable on any public CKB node with default configuration.

### Recommendation
Before allocating the decompressed buffer, add a second bound: `decompressed_bytes_len` must not exceed a reasonable multiple of the actual wire frame size (e.g., the known snappy worst-case expansion ratio of ~1.004×, or simply cap at `max_frame_length`). For example:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN
    || decompressed_bytes_len > self.length_delimited.max_frame_length() * 2
{
    return Err(io::ErrorKind::InvalidData.into());
}
```

This ensures the allocation is bounded by the wire frame size, not by the attacker-controlled varint.
