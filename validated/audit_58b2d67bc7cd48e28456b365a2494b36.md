### Title
Silent u64 Truncation in DAO Withdrawal Capacity Calculation Produces Wrong Withdrawal Amount — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a `u128` intermediate value, then casts it to `u64` with a bare `as u64` truncating cast. If the intermediate value exceeds `u64::MAX`, the high bits are silently discarded, causing the function to return a withdrawal capacity far smaller than the depositor is entitled to — without any error. This is inconsistent with every other analogous u128→u64 narrowing in the same file, all of which use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently wraps when `withdraw_counted_capacity > u64::MAX`. The resulting `withdraw_capacity` is then a small, wrong value. The `safe_add` that follows may succeed (no overflow), so the function returns `Ok(wrong_capacity)` with no indication of error.

Every other u128→u64 narrowing in the same file uses the safe pattern:

```rust
// secondary_block_reward, line 204:
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch, line 258:
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

The defect is a missing upper-bound check on the computed withdrawal value — the direct analog of the reported `pool::get_fee` vulnerability where `final_fee` could exceed 100% of the transaction amount due to an absent cap.

---

### Impact Explanation

`calculate_maximum_withdraw` is called in two places:

1. **RPC `calculate_dao_maximum_withdraw`** (`rpc/src/module/experiment.rs` lines 259–266, 288–294) — the value returned to the user to determine how much to put in their withdrawal output.
2. **Consensus verification** via `transaction_maximum_withdraw` → `transaction_fee` (`util/dao/src/lib.rs` lines 30–36) — used to compute the transaction fee as `maximum_withdraw - outputs_capacity`. [4](#0-3) [5](#0-4) 

If truncation occurs, the RPC returns a wrong (too small) capacity. A user who builds a withdrawal transaction using this value sets `outputs_capacity = truncated_value`. During consensus verification, `transaction_fee = deposited_amount - truncated_value`, which is nearly the entire deposited principal. The transaction is accepted, but the user loses the difference as an unintended fee — analogous to a fee exceeding 100% of the transaction value.

---

### Likelihood Explanation

For `withdraw_counted_capacity` to overflow `u64::MAX`:

```
counted_capacity × withdrawing_ar / deposit_ar > 2^64 − 1
```

`deposit_ar` starts at `10^16` and grows at roughly 1.344% per year (secondary issuance rate). The total CKB supply is bounded at ~33.6 billion CKB = ~3.36 × 10^18 shannons. For overflow with the full supply deposited, `ar` must grow to ~5.5× its genesis value, requiring ~120+ years of chain operation. **On mainnet this is currently unreachable.** On a custom chain with a higher secondary issuance rate or a modified genesis `ar`, the threshold is lower and could be reached sooner. The code defect is real and inconsistent with the rest of the codebase; the likelihood on mainnet is very low.

---

### Recommendation

Replace the silent `as u64` cast with the same checked pattern used elsewhere in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This makes overflow explicit and consistent with `secondary_block_reward` and `dao_field_with_current_epoch`.

---

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` (lines 296–349) demonstrates that the function is expected to return an error on overflow — but it only catches the case where `safe_add` overflows after the truncation. A case where `withdraw_counted_capacity` overflows u64 but the truncated value + `occupied_capacity` fits in u64 would silently return a wrong result and pass the test. [6](#0-5) 

Concretely: set `deposit_ar = 10^16`, `withdrawing_ar = 6 × 10^16` (achievable on a chain with high secondary issuance), `counted_capacity = 4 × 10^18` shannons (40 billion CKB, plausible on a custom chain). Then:

```
withdraw_counted_capacity = 4×10^18 × 6×10^16 / 10^16
                          = 2.4×10^19  >  u64::MAX (≈1.844×10^19)
```

`as u64` truncates to `2.4×10^19 mod 2^64 ≈ 5.5×10^18`. The function returns `Ok(Capacity::shannons(5.5×10^18 + occupied_capacity))` — roughly 55% of the correct value — with no error. The user loses ~45% of their deposited CKB as an unintended transaction fee. [7](#0-6)
