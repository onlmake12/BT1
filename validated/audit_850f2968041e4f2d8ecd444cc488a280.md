### Title
Single Malicious Peer Can Monopolize the OrphanPool, Evicting All Legitimate Orphan Transactions — (`tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` in CKB's tx-pool enforces a global cap of 100 entries (`DEFAULT_MAX_ORPHAN_TRANSACTIONS`) but imposes **no per-peer limit**. When the pool is full, eviction is random. A single malicious P2P peer can cheaply flood the pool with 100+ orphan transactions (referencing non-existent parent outputs), causing all legitimate orphan transactions from honest peers to be randomly evicted. When the legitimate parent transactions later arrive, their children are no longer in the pool and will not be automatically promoted, breaking the normal transaction relay flow.

---

### Finding Description

**Root cause — `tx-pool/src/component/orphan.rs`**

The pool enforces only a global size bound:

```rust
pub(crate) const DEFAULT_MAX_ORPHAN_TRANSACTIONS: usize = 100;
```

When `add_orphan_tx` is called, it inserts the entry unconditionally and then calls `limit_size()`:

```rust
pub fn add_orphan_tx(&mut self, tx: TransactionView, peer: PeerIndex, declared_cycle: Cycle) -> Vec<Byte32> {
    if self.entries.contains_key(&tx.proposal_short_id()) {
        return vec![];
    }
    self.entries.insert(tx.proposal_short_id(), Entry::new(tx.clone(), peer, declared_cycle));
    for out_point in tx.input_pts_iter() {
        self.by_out_point.entry(out_point).or_default().insert(tx.proposal_short_id());
    }
    // DoS prevention: do not allow OrphanPool to grow unbounded
    self.limit_size()
}
```

`limit_size()` evicts by expiry first, then **randomly** until the pool is at or below 100:

```rust
while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
    // Evict a random orphan:
    let id = self.entries.keys().next().cloned().expect("bound checked");
    if let Some(entry) = self.remove_orphan_tx(&id) {
        evicted_txs.push(entry.tx.hash());
    }
}
```

There is **no per-peer accounting** anywhere in `OrphanPool`. The `peer` field stored in `Entry` is never used for admission control.

**Attack path:**

1. Attacker connects as a normal P2P peer (no privilege required).
2. Attacker crafts 100+ syntactically valid CKB transactions whose inputs reference outputs of non-existent (or not-yet-relayed) parent transactions. These are cheap to produce — no fees are paid because they never confirm.
3. Each transaction is relayed via the standard P2P relay path: `submit_remote_tx` → `_process_tx` → missing-input branch → `add_orphan`.
4. The pool fills to 100 entries, all owned by the attacker.
5. Any legitimate orphan transaction submitted by an honest peer triggers `limit_size()`, which randomly evicts one entry. Because the attacker holds all 100 slots, the honest transaction is immediately evicted.
6. The attacker continuously re-submits evicted entries to maintain full occupancy.

**Consequence for honest peers:**

When an honest peer's parent transaction finally arrives and `process_orphan_tx` is called, the child transaction is no longer in the pool. The child must be resubmitted independently, breaking the automatic orphan-promotion flow and potentially causing the transaction to be permanently dropped if the submitting peer has already moved on.

---

### Impact Explanation

- **Service degradation**: The orphan promotion mechanism (`process_orphan_tx`) silently fails for all honest peers while the attacker holds the pool. Transactions that depend on in-flight parents will not be automatically accepted when the parent confirms.
- **Transaction relay disruption**: Honest relayers that sent a child transaction before its parent will see the child silently disappear from the pool. Wallets and applications relying on automatic orphan resolution will experience stuck or dropped transactions.
- **Sustained attack**: The attacker's cost is negligible — orphan transactions are never confirmed and never pay fees. On low-fee networks (e.g., testnets, sidechains) the attack is essentially free.

---

### Likelihood Explanation

Any unauthenticated P2P peer can relay transactions. No keys, stake, or special role is required. The attacker needs only a single TCP connection and the ability to construct syntactically valid CKB transactions with unknown input out-points. The pool size is only 100, making it trivial to fill. The attack is sustainable indefinitely at near-zero cost.

---

### Recommendation

Add a per-peer cap inside `add_orphan_tx`. Before inserting, count how many entries in `self.entries` already belong to `peer` and reject (or evict the attacker's own oldest entry) if the per-peer quota is exceeded. A reasonable default is `DEFAULT_MAX_ORPHAN_TRANSACTIONS / MAX_PEERS_CONTRIBUTING` (e.g., 10 per peer). This mirrors the fix pattern used in Bitcoin Core's orphan pool and prevents any single peer from monopolizing the shared resource.

---

### Proof of Concept

```
1. Attacker peer connects to a CKB node.
2. Attacker generates 100 transactions T_1..T_100 where each T_i.inputs[0]
   references OutPoint { tx_hash: random_hash_i, index: 0 } (unknown parent).
3. Attacker relays T_1..T_100 via the Relay protocol.
   → Each hits: submit_remote_tx → _process_tx → missing-input → add_orphan_tx
   → OrphanPool fills to 100 entries, all from attacker.
4. Honest peer relays T_honest (child of a pending parent P).
   → add_orphan_tx is called; limit_size() evicts a random entry.
   → Because all 100 slots are attacker-owned, T_honest is immediately evicted.
5. Parent P arrives; process_orphan_tx(P) finds no children → T_honest is lost.
6. Attacker re-submits any evicted T_i to maintain full occupancy.
```

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }
