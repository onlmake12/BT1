### Title
Attacker Can Continuously DoS Any User's UTXO Spending via Witness-Spoofed Tx-Pool Slot Squatting — (`tx-pool/src/process.rs`, `tx-pool/src/util.rs`)

---

### Summary

CKB's tx-pool deduplicates transactions by `ProposalShortId` (the first 10 bytes of `tx_hash`), which is derived solely from the transaction body excluding witnesses. An unprivileged attacker who observes any live UTXO can construct a transaction referencing the same inputs and outputs as a legitimate pending transaction but with garbage witnesses, submit it first, and occupy the `proposal_short_id` slot in the verify queue. While the attacker's transaction is being verified (and will ultimately fail script execution), the legitimate user's transaction is rejected with `Reject::Duplicated`. Because failed-verification transactions are never penalized with a peer ban, the attacker can repeat this indefinitely at zero on-chain cost, permanently preventing the victim from spending their UTXO.

---

### Finding Description

**Root cause — `ProposalShortId`-based deduplication before script verification**

`ProposalShortId` is the first 10 bytes of `tx_hash`, which is computed over the `RawTransaction` (inputs, outputs, cell_deps, header_deps, output_data) but **not** over witnesses:

```
tx_hash  = blake2b(RawTransaction)   // witnesses excluded
proposal_short_id = tx_hash[0..10]
``` [1](#0-0) 

The tx-pool admission path in `resumeble_process_tx` checks the verify queue and orphan pool by `proposal_short_id` **before** any script execution:

```rust
if self.verify_queue_contains(&tx).await {
    return Err(Reject::Duplicated(tx.hash()));
}
``` [2](#0-1) 

`verify_queue_contains` and `orphan_contains` both key on `proposal_short_id`: [3](#0-2) 

The pool-map collision check (`check_txid_collision`) also uses `proposal_short_id`: [4](#0-3) 

**Attack path**

1. Attacker observes a live UTXO owned by victim (all UTXOs are public on-chain).
2. Attacker constructs `tx_attack`: same inputs (victim's outpoints), same or smaller outputs (to satisfy the fee check), but **garbage witnesses**.
3. Attacker submits `tx_attack` via `send_transaction` RPC.
4. `non_contextual_verify` passes — it checks structure and size, not witness validity.
5. `pre_check` passes — inputs resolve to live cells, fee check passes (outputs < inputs capacity).
6. `tx_attack` enters the verify queue under the victim's `proposal_short_id`.
7. Victim submits `tx_legit` (same inputs, valid witnesses) → `verify_queue_contains` returns `true` → `Reject::Duplicated`.
8. Verify worker runs `verify_rtx` on `tx_attack` → script execution fails (garbage witnesses) → `tx_attack` is evicted from the queue.
9. Attacker immediately resubmits `tx_attack` → step 4–8 repeats.

The attacker is **never banned**: `ban_malformed` is only triggered for `Reject::Malformed`, not `Reject::Verification`: [5](#0-4) 

The existing integration test `TransactionHashCollisionDifferentWitnessHashes` confirms that a second transaction with the same `tx_hash` but different witnesses is rejected with `PoolRejectedDuplicatedTransaction`: [6](#0-5) 

---

### Impact Explanation

An unprivileged attacker can prevent any CKB user from spending any of their UTXOs indefinitely. Each attack cycle costs the attacker only an RPC call; no on-chain fee is ever paid because the attacker's transaction never confirms. The victim's transaction is perpetually rejected with `PoolRejectedDuplicatedTransaction` as long as the attacker keeps resubmitting. This is a complete griefing DoS on UTXO spending — the functional equivalent of the Folks Finance `createAccount` front-running: a unique identifier (here, `proposal_short_id` / UTXO outpoint set) is squatted by an invalid submission, blocking the legitimate one.

---

### Likelihood Explanation

- All UTXOs are publicly visible on-chain; no privileged information is required.
- The attacker needs only to call `send_transaction` via the public JSON-RPC endpoint.
- No on-chain cost is incurred (transactions never confirm).
- No peer ban is applied for `Reject::Verification` failures.
- The attack is trivially scriptable and can target any UTXO at any time.

---

### Recommendation

1. **Do not use `proposal_short_id` as the sole deduplication key for the verify queue.** Use the full `tx_hash` (or `witness_hash`) so that a transaction with different witnesses does not occupy the slot of a legitimate transaction.
2. **Apply rate-limiting or temporary bans for peers that repeatedly submit transactions failing script verification.** Treat repeated `Reject::Verification` from the same peer similarly to `Reject::Malformed`.
3. **Consider admitting only one transaction per input outpoint set into the verify queue**, and prefer the one with the higher fee rate, similar to RBF logic.

---

### Proof of Concept

```rust
// Attacker observes victim's UTXO: outpoint = (victim_tx_hash, 0)
// Victim's planned tx: inputs=[outpoint], outputs=[99 CKB to victim], witnesses=[valid_sig]

// Attacker constructs tx_attack:
let tx_attack = TransactionBuilder::default()
    .input(CellInput::new(OutPoint::new(victim_tx_hash, 0), 0))
    .output(CellOutputBuilder::default().capacity(capacity_bytes!(99)).build())
    .output_data(Bytes::new())
    .witness(Bytes::from(b"garbage_witness".to_vec()))  // invalid signature
    .build();

// tx_attack has the same tx_hash (and proposal_short_id) as tx_legit
// because tx_hash excludes witnesses.
assert_eq!(tx_attack.proposal_short_id(), tx_legit.proposal_short_id());
assert_ne!(tx_attack.witness_hash(), tx_legit.witness_hash());

// Step 1: Attacker submits tx_attack → enters verify queue
node.rpc_client().send_transaction(tx_attack.data().into());

// Step 2: Victim submits tx_legit → rejected
let result = node.rpc_client().send_transaction_result(tx_legit.data().into());
assert!(result.err().unwrap().to_string().contains("PoolRejectedDuplicatedTransaction"));

// Step 3: tx_attack fails script verification → evicted from queue
// Step 4: Attacker resubmits tx_attack → repeat indefinitely at zero cost
```
