Audit Report

## Title
Unbounded DB Read Amplification via Unauthenticated `GetTransactionsProof` with No Per-Peer Rate Limiting — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary
`LightClientProtocol` has no rate limiter field and `try_process` dispatches `GetTransactionsProof` directly to `GetTransactionsProofProcess::execute()` without any per-peer throttle check. An unprivileged peer can send a stream of messages each containing 1000 tx_hashes, forcing 1000 synchronous DB reads and a full response per message, with no server-side rejection path. The `Relayer` protocol applies an explicit per-peer `RateLimiter` before any dispatch; the light client protocol has no equivalent.

## Finding Description
`LightClientProtocol` contains only `pub shared: Shared` with no rate limiter field. [1](#0-0) 

`try_process` dispatches `GetTransactionsProof` directly to `execute()` with no preceding rate check. [2](#0-1) 

Inside `execute()`, the only guards are an empty-check and an upper-bound check (`> GET_TRANSACTIONS_PROOF_LIMIT`). A message with exactly 1000 hashes passes both. [3](#0-2) 

`GET_TRANSACTIONS_PROOF_LIMIT` is 1000. [4](#0-3) 

The handler then calls `snapshot.get_transaction_info(tx_hash)` for every hash in a `partition` loop. With all-unknown hashes, all 1000 DB reads return `None`, `found` is empty, and the server still constructs and sends a `SendTransactionsProofV1` response with a 1000-entry `missing_tx_hashes` list. [5](#0-4) 

A grep for `rate_limit|rate_limiter|RateLimiter|TooManyRequests` across all of `util/light-client-protocol-server/**` returns zero matches, confirming no equivalent protection exists anywhere in the module.


By contrast, `Relayer::try_process` checks `rate_limiter.check_key(&(peer, message.item_id()))` before any dispatch and returns `TooManyRequests` on failure. [6](#0-5) 

The `last_hash` validity check at line 44 is trivially satisfied using any publicly known tip block hash from the public RPC. [7](#0-6) 

## Impact Explanation
Each request costs the server 1000 synchronous DB reads plus a full outbound response. A single attacker sending requests at maximum speed imposes unbounded DB read IOPS and outbound bandwidth; multiple peers multiply this linearly. This can saturate disk I/O and network egress on a full node running the light client server, causing it to become unresponsive or crash. This matches the allowed **High** impact: *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation
The attack requires only a valid main-chain block hash (obtainable from the public RPC) and the ability to open a P2P connection. No key, stake, or privileged role is needed. The attack is fully scriptable, repeatable, and scales linearly with the number of attacking peers.

## Recommendation
Add a per-peer rate limiter to `LightClientProtocol`, mirroring the pattern in `Relayer`:
- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`.
- In `try_process`, check `rate_limiter.check_key(&(peer_index, message.item_id()))` before dispatching any handler, returning `StatusCode::TooManyRequests` on failure.
- Additionally, consider lowering `GET_TRANSACTIONS_PROOF_LIMIT` (e.g., to 100) to reduce per-request DB amplification independent of rate limiting. [4](#0-3) 

## Proof of Concept
```
1. Obtain any valid tip block hash H from the CKB chain (public RPC).
2. Connect to a CKB full node with light client protocol enabled.
3. In a tight loop:
   a. Build GetTransactionsProof { last_hash: H, tx_hashes: [rand_32_bytes(); 1000] }
   b. Send over the P2P connection.
4. Observe: server performs 1000 get_transaction_info DB reads per message,
   responds with a full SendTransactionsProofV1 containing 1000 missing_tx_hashes,
   no TooManyRequests rejection is ever returned.
5. Measure DB read IOPS and outbound bandwidth — both scale linearly with
   request rate, unbounded by any server-side control.
```
