### Title
`addFunds()` Deposit Cap Check Uses `msg.value` Instead of Cumulative Balance, Allowing Permanent Subscriptions to Exceed `MAX_DEPOSIT_LIMIT` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `addFunds()` function enforces `MAX_DEPOSIT_LIMIT` (100 ETH) for permanent subscriptions by checking only the **incoming `msg.value`** against the cap, not the **resulting cumulative balance**. An unprivileged user can call `addFunds()` repeatedly with amounts â‰¤ `MAX_DEPOSIT_LIMIT` to accumulate a balance far exceeding the cap, permanently locking arbitrary ETH in the contract.

---

### Finding Description

`MAX_DEPOSIT_LIMIT` is defined as `100 ether` in `SchedulerConstants.sol`: [1](#0-0) 

In `addFunds()`, the guard is:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
``` [2](#0-1) 

The check `msg.value > MAX_DEPOSIT_LIMIT` validates only the **single deposit amount**, not `status.balanceInWei + msg.value`. After the check passes, the deposit is unconditionally added to the stored balance. The same flaw exists in `createSubscription()`, but there the balance starts at zero so `msg.value` equals the resulting balance â€” making it safe. The flaw is exclusively exploitable through repeated `addFunds()` calls. [3](#0-2) 

---

### Impact Explanation

Permanent subscriptions cannot be deactivated and funds cannot be withdrawn: [4](#0-3) 

This means any ETH deposited into a permanent subscription is **permanently locked**. The `MAX_DEPOSIT_LIMIT` exists precisely to cap how much ETH can be irrevocably locked per subscription. By bypassing it, an attacker can lock an unbounded amount of ETH in the contract. While the attacker loses their own ETH, this constitutes a griefing/DoS vector against the contract's ETH accounting and may violate protocol invariants relied upon by integrators or the keeper network.

---

### Likelihood Explanation

The entry point is fully permissionless â€” any address that owns a permanent subscription (created via `createSubscription`) can call `addFunds()` with no access restrictions. The exploit requires only repeated transactions with `msg.value â‰¤ MAX_DEPOSIT_LIMIT`. No privileged role, leaked key, or external oracle manipulation is needed. [5](#0-4) 

---

### Recommendation

Change the check in `addFunds()` to validate the **post-deposit cumulative balance** rather than the single incoming value:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && (status.balanceInWei + msg.value) > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

This mirrors the correct pattern: the cap should bound the total locked balance, not just any single top-up.

---

### Proof of Concept

1. Alice calls `createSubscription{value: 100 ether}(params)` with `params.isPermanent = true`.
   - Check: `100 ether > 100 ether` â†’ **false** â†’ passes. `balanceInWei = 100 ether`.

2. Alice calls `addFunds{value: 100 ether}(subscriptionId)`.
   - Check: `100 ether > 100 ether` â†’ **false** â†’ passes. `balanceInWei = 200 ether`.

3. Alice calls `addFunds{value: 100 ether}(subscriptionId)` again.
   - Check: `100 ether > 100 ether` â†’ **false** â†’ passes. `balanceInWei = 300 ether`.

4. Repeat N times â†’ `balanceInWei = N Ă— 100 ether`, all permanently locked.

The `MAX_DEPOSIT_LIMIT` is fully bypassed. The existing test `testPermanentSubscriptionDepositLimit` only tests that a single `addFunds` call with `> MAX_DEPOSIT_LIMIT` reverts (Test 3), but does **not** test the cumulative case, leaving this bypass undetected. [6](#0-5)
