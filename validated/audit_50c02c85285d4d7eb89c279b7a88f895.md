### Title
`InitLoadUnverified` thread panic leaves `is_verifying_unverified_blocks_on_startup` permanently `true`, causing `ckb import` to hang forever and the synchronizer to stop fetching blocks — (`chain/src/init_load_unverified.rs`)

### Summary
If the `init_load_unverified_blocks` thread panics before reaching the `store(false, ...)` call in `InitLoadUnverified::start`, the startup flag is never cleared. Every caller that spins on or gates against this flag is permanently affected: `ckb import` enters an infinite busy-wait loop, and the P2P synchronizer silently skips block fetching on every timer tick.

### Finding Description
`InitLoadUnverified::start` calls `find_and_verify_unverified_blocks` and only stores `false` into the `AtomicBool` after that call returns normally. Inside `find_and_verify_unverified_blocks`, the closure passed to `find_unverified_blocks` calls `.expect("unverified block must be in db")` on the result of `store().get_block(hash)`. If the database contains a `COLUMN_NUMBER_HASH` entry for a block whose body was never fully written (a realistic outcome of a mid-write crash), `get_block` returns `None` and the thread panics. Because there is no `Drop` implementation on `InitLoadUnverified` and no `catch_unwind` wrapper in the thread closure, the `Arc<AtomicBool>` is dropped with its value still `true`. The `chain_service_thread` joins the panicked thread with `let _ = init_load_unverified_thread.join()`, discarding the error.

### Impact Explanation
- **`ckb import` hangs forever**: `Import::read_from_json` spins in `while self.chain.is_verifying_unverified_blocks_on_startup() { sleep(10ms) }` with no timeout or escape condition.
- **Synchronizer permanently stalled**: `Synchronizer::find_blocks_to_fetch` returns immediately on every IBD/non-IBD timer tick, so the node never requests new blocks from peers after the restart.

### Likelihood Explanation
The trigger is a database state where `COLUMN_NUMBER_HASH` has an entry for a block whose body is absent — a realistic outcome of a node crash during block ingestion. Any operator who restarts a node after such a crash and then runs `ckb import` will hit the hang. The scenario is locally reproducible without any network interaction.

### Recommendation
Implement `Drop` for `InitLoadUnverified` (or use a RAII guard) that unconditionally stores `false` into the `AtomicBool`, ensuring the flag is cleared regardless of whether `start` returns normally or unwinds. Alternatively, wrap the body of the thread closure in `std::panic::catch_unwind` and store `false` in the `Err` branch. The `.expect()` in `find_and_verify_unverified_blocks` should also be converted to a logged error with graceful continuation rather than a panic.

### Proof of Concept
1. Start a CKB node, let it store a block partially (kill -9 after `COLUMN_NUMBER_HASH` is written but before the block body is committed).
2. Restart the node. The `init_load_unverified_blocks` thread will call `get_block(hash)` → `None` → `.expect()` → panic.
3. Run `ckb import <any-file>`. Observe that `read_from_json` never progresses past the startup-flag spin loop.
4. Alternatively, assert in a test that after injecting a panic into `find_and_verify_unverified_blocks`, `is_verifying_unverified_blocks_on_startup()` eventually returns `false` — the assertion will fail indefinitely.
