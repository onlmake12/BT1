### Title
Off-by-One in RFC0044 Activation Epoch Check Causes First Activation-Epoch Block to Omit MMR Chain Root — (`File: tx-pool/src/block_assembler/mod.rs`)

---

### Summary

The block assembler's `build_extension` function checks whether RFC0044 (MMR/light-client) is active by querying the **tip's** (parent's) epoch number rather than the **new block's** epoch number. This off-by-one causes the first block of the RFC0044 activation epoch to be assembled without the mandatory MMR chain-root extension. The block verifier (`BlockExtensionVerifier`) contains the identical off-by-one and therefore accepts the malformed block. The result is a permanent gap in the MMR chain at the activation boundary, breaking light-client verification for that block.

---

### Finding Description

`build_extension` in `tx-pool/src/block_assembler/mod.rs` is called by the miner's block-template pipeline to decide whether to embed the MMR chain root in the block extension field:

```rust
pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
    let tip_header = snapshot.tip_header();
    // The use of the epoch number of the tip here leads to an off-by-one bug,
    // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
    let mmr_activate = snapshot
        .consensus()
        .rfc0044_active(tip_header.epoch().number());   // ← uses TIP epoch, not new-block epoch
    ...
}
```

The new block being assembled is at height `tip + 1`, which may be the **first block of the RFC0044 activation epoch**. At that exact boundary, `tip_header.epoch().number()` is still the *previous* epoch number (one below the activation threshold), so `rfc0044_active` returns `false` and no MMR extension is produced.

`rfc0044_active` simply checks `target >= RFC0044_ACTIVE_EPOCH`:

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    let rfc0044_active_epoch = match self.id.as_str() {
        mainnet::CHAIN_SPEC_NAME => softfork::mainnet::RFC0044_ACTIVE_EPOCH,  // 8651
        testnet::CHAIN_SPEC_NAME => softfork::testnet::RFC0044_ACTIVE_EPOCH,  // 5711
        _ => 0,
    };
    target >= rfc0044_active_epoch
}
```

`BlockExtensionVerifier::verify` in `verification/contextual/src/contextual_block_verifier.rs` contains the **same** off-by-one:

```rust
let mmr_active = self
    .context
    .consensus
    .rfc0044_active(self.parent.epoch().number());  // ← same parent-epoch check
```

Because both sides share the same boundary error, the first block of the activation epoch is assembled without the MMR extension **and** accepted by all full nodes without error. The code comment in `build_extension` explicitly acknowledges this: *"The use of the epoch number of the tip here leads to an off-by-one bug, so be careful, it needs to be preserved for consistency reasons and not fixed directly."*

---

### Impact Explanation

RFC0044 (CKB light-client protocol) requires every block from the activation epoch onward to embed the MMR chain root in its extension field. Light clients use this chain root to verify block headers without downloading the full chain. Because the first block of the activation epoch is permanently committed to the chain without the MMR extension, light clients face an unverifiable gap: they cannot construct or validate a continuous MMR proof that spans the activation boundary. Any light-client that attempts to verify headers crossing this boundary will fail or must trust the gap, undermining the security guarantee RFC0044 is designed to provide.

---

### Likelihood Explanation

This is a deterministic, protocol-level defect that fires exactly once per network at the RFC0044 activation epoch. On mainnet the activation epoch is hardcoded to `8651`; on testnet to `5711`. Any miner who mines the first block of that epoch via the standard `get_block_template` RPC will produce the defective block. No special attacker capability is required — the ordinary miner/block-template workflow is the entry path.

---

### Recommendation

The correct epoch to test against is the **new block's** epoch, not the parent's. The assembler should derive the candidate epoch from `snapshot.epoch_ext()` (the epoch for the block being built) and pass that epoch number to `rfc0044_active`. The same correction must be applied symmetrically in `BlockExtensionVerifier::verify` (using the block's own epoch rather than `self.parent.epoch().number()`). Because both sites must change together to avoid a consensus split, the fix requires a coordinated update to both the assembler and the verifier.

---

### Proof of Concept

1. Configure a dev chain with `RFC0044_ACTIVE_EPOCH = N` for some small `N`.
2. Mine blocks until the tip is the last block of epoch `N-1`.
3. Call `get_block_template` — the returned template will have `extension: null` (no MMR root), even though the candidate block number is the first block of epoch `N`.
4. Submit the block; `BlockExtensionVerifier` accepts it because it also checks `parent.epoch().number() = N-1 < N`.
5. Mine the second block of epoch `N`; its template now correctly includes the MMR extension (parent epoch is now `N`).
6. A light client attempting to verify headers across the epoch-`N` boundary finds the first block of epoch `N` has no chain root, breaking the continuous MMR proof.
