### Title
Single Entropy Provider Offline or Censoring Permanently Blocks User Requests With No Refund or Cancel Mechanism — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a user submits a randomness request via `requestWithCallback()` or `requestV2()`, the provider fee is immediately credited to the provider's accrued balance before the provider has fulfilled anything. If the provider goes offline, stops operating, or selectively censors specific requests, the user's request is permanently stuck with no cancel, refund, or expiry mechanism anywhere in the contract.

---

### Finding Description

In `requestHelper()`, the provider fee is immediately credited to `providerInfo.accruedFeesInWei` at request time, before the provider has revealed anything:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The request is then stored and waits indefinitely for the provider to call `revealWithCallback()`, which requires the provider to supply their secret `providerContribution` from their private hash chain — a value only the provider possesses. [2](#0-1) 

The `isActive()` function marks a request as active solely based on `sequenceNumber != 0`, with no time-based expiry:

```solidity
function isActive(EntropyStructsV2.Request storage req) internal view returns (bool) {
    return req.sequenceNumber != 0;
}
``` [3](#0-2) 

The contract itself acknowledges this design gap in a comment inside `allocRequest()`:

> "There is a chance that some requests never get revealed and remain active forever." [4](#0-3) 

A search of the entire `Entropy.sol` (1090 lines) confirms there is **no** `cancel`, `refund`, `expire`, `deadline`, or `timeout` function. The `EntropyState.sol` storage layout also contains no deadline or expiry field per request: [5](#0-4) 

The `request()` function's own NatSpec explicitly states: *"Note that excess value is **not** refunded to the caller."* — and there is no path to reclaim the base fee either. [6](#0-5) 

---

### Impact Explanation

If the chosen provider:
- Goes permanently offline
- Selectively censors a specific user's sequence number
- Exhausts their hash chain (`endSequenceNumber` reached) and stops re-registering
- Has their private key compromised and stops operating

…then:
1. The user's paid ETH/native-token fee is permanently locked in `providerInfo.accruedFeesInWei` — the provider can withdraw it at any time via `withdraw()`, while the user has no recourse.
2. Any application depending on the entropy callback (NFT mint, on-chain game, lottery) is permanently blocked with no way to retry, cancel, or recover.
3. The `requestsOverflow` mapping will accumulate permanently stuck requests, increasing gas costs for future users who hash-collide into the same slot. [7](#0-6) 

---

### Likelihood Explanation

- The vast majority of users call `requestV2()` with no explicit provider argument, which routes to `_state.defaultProvider` — a single Pyth-operated address. This creates a single point of failure for the entire user base. [8](#0-7) 

- Provider downtime, key rotation failures, or selective censorship of specific contracts (e.g., a competitor's NFT project) are realistic operational scenarios.
- The provider has already collected the fee at request time, removing the economic incentive to fulfill the request.

---

### Recommendation

1. **Add a request expiry/deadline**: Store a `requestDeadline` (e.g., `block.number + N`) in each `Request` struct. After the deadline passes, allow the user to call a `cancelRequest()` function that clears the request and refunds the fee.
2. **Delay fee crediting**: Credit the provider fee only upon successful `revealWithCallback()`, not at request time, so unserved requests can be refunded.
3. **Allow provider substitution**: After a timeout, allow the user to re-route the request to a different provider.

---

### Proof of Concept

1. User calls `requestWithCallback{value: fee}(defaultProvider, userContribution)`.
2. `requestHelper()` immediately credits `providerInfo.accruedFeesInWei += providerFee` and stores the request with `sequenceNumber = N`.
3. The default provider goes offline (or censors this specific user).
4. No `cancel`, `refund`, or `expire` function exists. The user's ETH is permanently locked.
5. The provider calls `withdraw(providerFee)` and receives the user's fee despite never fulfilling the request.
6. The user's application callback (`_entropyCallback`) is never invoked. The request remains in `_state.requests` or `_state.requestsOverflow` forever, as `isActive()` returns `true` indefinitely. [9](#0-8) [10](#0-9)

            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;

    // Allocate storage space for a new in-flight request. This method returns a pointer to a storage slot
    // that the caller should overwrite with the new request. Note that the memory at this storage slot may
    // -- and will -- be filled with arbitrary values, so the caller *must* overwrite every field of the returned
    // struct.
    function allocRequest(
        address provider,
        uint64 sequenceNumber
    ) internal returns (EntropyStructsV2.Request storage req) {
        (, uint8 shortKey) = requestKey(provider, sequenceNumber);

        req = _state.requests[shortKey];
        if (isActive(req)) {
            // There's already a prior active request in the storage slot we want to use.
            // Overflow the prior request to the requestsOverflow mapping.
            // It is important that this code overflows the *prior* request to the mapping, and not the new request.
            // There is a chance that some requests never get revealed and remain active forever. We do not want such
            // requests to fill up all of the space in the array and cause all new requests to incur the higher gas cost
            // of the mapping.
            //
            // This operation is expensive, but should be rare. If overflow happens frequently, increase
            // the size of the requests array to support more concurrent active requests.
            (bytes32 reqKey, ) = requestKey(req.provider, req.sequenceNumber);
            _state.requestsOverflow[reqKey] = req;
        }
    }

contract EntropyInternalStructs {
    struct State {
        // Admin has the rights to update pyth configs
        address admin;
        // Fee charged by the pyth protocol in wei.
        uint128 pythFeeInWei;
        // Total quantity of fees (in wei) earned by the pyth protocol that are currently stored in the contract.
        // This quantity is incremented when fees are paid and decremented when fees are withdrawn.
        // Note that u128 can store up to ~10^36 wei, which is ~10^18 in native base tokens, which should be plenty.
        uint128 accruedPythFeesInWei;
        // The protocol sets a provider as default to simplify integration for developers.
        address defaultProvider;
        // Hash table for storing in-flight requests. Table keys are hash(provider, sequenceNumber), and the value is
        // the current request (if one is currently in-flight).
        //
        // Due to the vagaries of EVM opcode costs, it is inefficient to simply use a mapping here. Overwriting zero-valued
        // storage slots with non-zero values is expensive in EVM (21k gas). Using a mapping, each new request starts
        // from all-zero values, and thus incurs a substantial write cost. Deleting non-zero values does refund gas, but
        // unfortunately the refund is not substantial enough to matter.
        //
        // This data structure is a two-level hash table. It first tries to store new requests in the requests array at
        // an index determined by a few bits of the request's key. If that slot in the array is already occupied by a
        // prior request, the prior request is evicted into the requestsOverflow mapping. Requests in the array are
        // considered active if their sequenceNumber is > 0.
        //
        // WARNING: the number of requests must be kept in sync with the constants below
        EntropyStructsV2.Request[32] requests;
        mapping(bytes32 => EntropyStructsV2.Request) requestsOverflow;
        // Mapping from randomness providers to information about each them.
        mapping(address => EntropyStructsV2.ProviderInfo) providers;
        // proposedAdmin is the new admin's account address proposed by either the owner or the current admin.
        // If there is no pending transfer request, this value will hold `address(0)`.
        address proposedAdmin;
        // Seed for in-contract PRNG. This seed is used to generate user random numbers in some callback flows.
        bytes32 seed;
    }
}
