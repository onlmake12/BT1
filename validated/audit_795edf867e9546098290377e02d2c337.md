### Title
Legacy Entropy Callback Path Permanently Locks User Funds on Reverting Callback — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains two execution paths. The modern path (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` to catch callback reverts and transitions the request to a recoverable `CALLBACK_FAILED` state. The **legacy path** (when `req.gasLimit10k == 0`) calls `IEntropyConsumer(callAddress)._entropyCallback(...)` directly with no try/catch. If the callback reverts, the entire transaction reverts, the request remains permanently stuck in `CALLBACK_NOT_STARTED`, and the user has no on-chain mechanism to recover their paid fee.

---

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`: [1](#0-0) 

When `gasLimit10k != 0`, the callback is invoked via `excessivelySafeCall`, failures are caught, and the request moves to `CALLBACK_FAILED` — a recoverable state. [2](#0-1) 

When `gasLimit10k == 0` (the legacy path), the callback is invoked **directly** with no error handling: [3](#0-2) 

`clearRequest` is called first (CEI pattern), then `_entropyCallback` is called bare. If `_entropyCallback` reverts, the entire transaction reverts, `clearRequest` is rolled back, and the request is restored to `CALLBACK_NOT_STARTED`. Every subsequent keeper attempt will also revert. The request is permanently stuck.

`gasLimit10k` is set to `0` whenever the provider's `defaultGasLimit` is `0`: [4](#0-3) 

A provider's `defaultGasLimit` starts at `0` (zero-initialized storage) and can be explicitly reset to `0` via `setDefaultGasLimit(0)`: [5](#0-4) 

The comment in `requestHelper` describes this as merely "disabling" the new flow, with no warning that it permanently exposes users to fund loss: [6](#0-5) 

There is no user-facing refund function. The only withdrawal paths are `withdraw` and `withdrawAsFeeManager`, which are restricted to providers: [7](#0-6) 

---

### Impact Explanation

A user who makes a request through a provider with `defaultGasLimit == 0` and whose callback contract reverts (due to a bug, out-of-gas, or any revert) permanently loses their paid fee. The request can never be fulfilled and can never be refunded. This is a direct, irreversible loss of user funds. The impact scales with the number of users on providers that have not set a non-zero `defaultGasLimit`.

---

### Likelihood Explanation

Any provider that has not explicitly called `setDefaultGasLimit` retains the zero-initialized default, placing all their users in the legacy path. The `requestWithCallback` function passes `gasLimit = 0`, which routes through the same legacy path when the provider's default is also `0`: [8](#0-7) 

A user whose callback reverts for any reason — including a common out-of-gas scenario — triggers the permanent lock. The Entropy documentation explicitly warns that callbacks should never revert, but does not warn that doing so on a legacy-path provider causes permanent fund loss rather than a retryable failure.

---

### Recommendation

1. In the legacy path (`gasLimit10k == 0`), wrap the `_entropyCallback` invocation in a low-level call with error catching, analogous to the `excessivelySafeCall` used in the modern path.
2. Add a user-facing `requestRefund` function that allows a requester to reclaim their fee after a request has been stuck for a configurable number of blocks.
3. Update the comment on `gasLimit10k = 0` to explicitly warn that this disables the callback failure recovery flow and can result in permanent user fund loss.
4. Consider deprecating the legacy path entirely by requiring all providers to set a non-zero `defaultGasLimit` before accepting new requests.

---

### Proof of Concept

1. Provider registers with `defaultGasLimit == 0` (never calls `setDefaultGasLimit`).
2. User calls `requestWithCallback(provider, userContribution)` paying the required fee. `requestHelper` sets `req.gasLimit10k = 0`.
3. User's callback contract is deployed with logic that always reverts.
4. Keeper (or anyone) calls `revealWithCallback`. The condition `req.gasLimit10k != 0` is false, so the legacy else-branch executes.
5. `clearRequest` runs, then `IEntropyConsumer(callAddress)._entropyCallback(...)` reverts.
6. The entire transaction reverts. `clearRequest` is rolled back. Request remains in `CALLBACK_NOT_STARTED`.
7. Every subsequent call to `revealWithCallback` for this sequence number also reverts.
8. The user's fee is permanently locked in `providerInfo.accruedFeesInWei` with no on-chain recovery path. [3](#0-2)

    function setDefaultGasLimit(uint32 gasLimit) external override {
