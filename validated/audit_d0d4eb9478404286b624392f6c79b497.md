### Title
Orphan-parent invalidation incorrectly bans the innocent sender of a child block — (`chain/src/orphan_broker.rs`, `sync/src/synchronizer/block_process.rs`)

### Summary

When a block B is placed in the orphan pool (because its parent P is not yet stored), the `verify_callback` closure stored with B permanently captures the `PeerIndex` of whoever sent B. If a different peer later causes P to be marked `BLOCK_INVALID`, `OrphanBroker::process_invalid_block` fires B's callback with an `InternalErrorKind::Other` error. Because `is_internal_db_error` returns `false` for `Other`, the callback unconditionally calls `post_sync_process`, which bans the peer that sent B — even though B itself may be structurally valid.

### Finding Description

**Step 1 — Block B enters the orphan pool.**

In `BlockProcess::execute`, a `verify_callback` is constructed that closes over `peer_id` (the sender of B): [1](#0-0) 

This `RemoteBlock` is forwarded to `ChainService::asynchronous_process_block`, which calls `OrphanBroker::process_lonely_block`. Because P's status is neither `BLOCK_STORED` nor `BLOCK_INVALID`, B is inserted into the orphan pool: [2](#0-1) 

**Step 2 — Parent P is invalidated by a different peer.**

When any block is processed, `search_orphan_leaders()` is called at the end of `process_lonely_block`: [3](#0-2) 

`search_orphan_leader` checks if the leader (P) is `BLOCK_INVALID` and, if so, calls `process_invalid_block` on every orphan child: [4](#0-3) 

**Step 3 — `process_invalid_block` fires B's callback with `InternalErrorKind::Other`.** [5](#0-4) 

The error kind is `InternalErrorKind::Other`, not `Database` or `System`.

**Step 4 — `is_internal_db_error` returns `false` for `Other`.** [6](#0-5) 

So the early-return guard in the callback is not taken: [7](#0-6) 

**Step 5 — `post_sync_process` bans peer B.**

`StatusCode::BlockIsInvalid = 401` falls in the `400..500` range, so `should_ban()` returns `Some(BAD_MESSAGE_BAN_TIME)`: [8](#0-7) 

`post_sync_process` then calls `nc.ban_peer(peer_id_of_B, ban_time, ...)`: [9](#0-8) 

### Impact Explanation

An unprivileged attacker (peer A) can selectively ban any honest peer B from the victim node. The attacker only needs to:
1. Know that peer B has sent a block B whose parent P is not yet stored on the victim (i.e., B is in the orphan pool).
2. Send an invalid version of P to the victim.

The victim will then ban peer B with `BAD_MESSAGE_BAN_TIME`, severing the connection and preventing reconnection for the ban duration. This can be used to partition the victim's peer set or to isolate it from specific honest peers.

### Likelihood Explanation

The precondition — a block sitting in the orphan pool — is common during normal sync, especially during IBD or when blocks arrive out of order. The attacker needs only a single P2P connection to the victim and knowledge of one block hash in the victim's orphan pool (observable via timing or network monitoring). No hashpower, keys, or privileged access are required.

### Recommendation

In `process_invalid_block`, do not fire the `verify_callback` with a banning-eligible error. Options:
- Fire the callback with `Ok(false)` (already-verified sentinel) instead of `Err(...)`.
- Use `InternalErrorKind::System` (which `is_internal_db_error` returns `true` for) so the callback's guard suppresses the ban.
- Add a dedicated `InternalErrorKind` variant (e.g., `ParentInvalid`) and extend `is_internal_db_error` (or a new predicate) to suppress banning for it.

### Proof of Concept

```
Victim node V, Peer A (attacker), Peer B (honest)

1. Peer B sends block B (parent = P, P not stored on V).
   → V: orphan_pool.insert(B, callback=ban(peer_B))

2. Peer A sends block P with an invalid field (e.g., bad PoW).
   → V: non_contextual_verify(P) fails
   → V: insert_block_status(P, BLOCK_INVALID)
   → V: search_orphan_leaders() finds P as leader
   → V: process_invalid_block(B)
        fires callback(Err(InternalErrorKind::Other("parent P is invalid...")))
        is_internal_db_error → false
        post_sync_process(peer_B, BlockIsInvalid=401)
        should_ban() → Some(BAD_MESSAGE_BAN_TIME)
        nc.ban_peer(peer_B, BAD_MESSAGE_BAN_TIME)

3. Peer B is now banned from V despite having sent a valid block.
```

pub fn is_internal_db_error(error: &Error) -> bool {
    if error.kind() == ErrorKind::Internal {
        let error_kind = error
            .downcast_ref::<InternalError>()
            .expect("error kind checked")
            .kind();
        if error_kind == InternalErrorKind::DataCorrupted {
            panic!("{}", error)
        } else {
            return error_kind == InternalErrorKind::Database
                || error_kind == InternalErrorKind::System;
        }
    }
    false
