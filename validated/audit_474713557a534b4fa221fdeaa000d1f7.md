### Title
`Echo#withdrawAsFeeManager` Enforces Fee Manager Check Even When `msg.sender == provider`, Permanently Locking Provider Fees — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol` has no direct `withdraw()` function for providers. The only withdrawal path for provider-accrued fees is `withdrawAsFeeManager(address provider, uint128 amount)`, which unconditionally requires `msg.sender == feeManager`. Because `feeManager` defaults to `address(0)` on registration and is never set automatically, any provider who does not explicitly call `setFeeManager` has their accrued fees permanently locked. This is the direct analog of M-8: an authorization check that should be bypassed when the caller is the same as the fund owner is instead always enforced.

---

### Finding Description

When a provider registers in `Echo.sol` via `registerProvider`, the `feeManager` field in `ProviderInfo` is left at its zero-initialized default (`address(0)`):

```solidity
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    provider.feePerFeedInWei = feePerFeedInWei;
    provider.feePerGasInWei = feePerGasInWei;
    provider.isRegistered = true;
    // feeManager is NOT set — defaults to address(0)
}
``` [1](#0-0) 

The only withdrawal path for provider fees is `withdrawAsFeeManager`, which unconditionally checks `msg.sender == feeManager`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
}
``` [2](#0-1) 

Unlike `Entropy.sol`, which provides a separate `withdraw()` function that uses `msg.sender` directly as the provider key (allowing providers to always withdraw their own fees), `Echo.sol` exposes no such path: [3](#0-2) 

The `IEcho` interface confirms there is no provider-direct `withdraw()`: [4](#0-3) 

Because `msg.sender` can never equal `address(0)` in a normal transaction, the check `msg.sender == address(0)` is always false, making it impossible for a provider without an explicitly set fee manager to withdraw their fees.

---

### Impact Explanation

Provider fees accrued in `Echo.sol` are permanently locked for any provider that does not explicitly call `setFeeManager` before attempting withdrawal. This is a direct loss of funds for Echo providers. The default state after `registerProvider` is a locked fee balance with no recovery path other than calling `setFeeManager` first — a non-obvious prerequisite not enforced or documented at registration time.

---

### Likelihood Explanation

Any Echo provider who registers and accrues fees without first calling `setFeeManager` will be unable to withdraw. This is the default state. Providers migrating from Entropy (which has a direct `withdraw()`) would not expect this requirement. The likelihood is high for any new provider integration.

---

### Recommendation

Add a direct `withdraw(uint128 amount)` function for providers (mirroring `Entropy.sol`), or modify `withdrawAsFeeManager` to bypass the fee manager check when `msg.sender == provider`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    // Allow provider to withdraw their own fees directly (analogous to ERC20 spender == from)
    if (msg.sender != provider) {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
    }
    ...
}
```

---

### Proof of Concept

1. Provider calls `registerProvider(baseFee, feePerFeed, feePerGas)`. `feeManager` is `address(0)`.
2. Users call `requestPriceUpdatesWithCallback`, accruing fees to the provider's `accruedFeesInWei`.
3. Provider calls `withdrawAsFeeManager(providerAddress, amount)`.
4. Transaction reverts: `"Only fee manager"` because `msg.sender != address(0)`.
5. Provider has no alternative withdrawal path — fees are permanently locked.

The Entropy contract's `withdraw()` at line 150 demonstrates the correct pattern: providers should always be able to withdraw their own fees directly without requiring a fee manager intermediary. [5](#0-4)
