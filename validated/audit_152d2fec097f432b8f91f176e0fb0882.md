### Title
Orphan Transaction Pool Exhaustion via Zero-Cost Relay Flooding Causes Random Eviction of Legitimate Transactions - (File: `tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` in CKB's tx-pool accepts relay transactions with unresolved inputs without any fee verification, and enforces a hard cap of 100 entries (`DEFAULT_MAX_ORPHAN_TRANSACTIONS`). When the cap is exceeded, eviction is **random** with no fee-rate or priority ordering. An unprivileged P2P peer can flood the orphan pool with transactions referencing fabricated, non-existent inputs at zero on-chain cost, permanently saturating the pool and causing legitimate users' orphan transactions to be randomly displaced.

---

### Finding Description

`OrphanPool::add_orphan_tx` accepts any transaction whose inputs are missing from the chain or pool, stores it unconditionally, and then calls `limit_size()` to enforce the 100-entry cap:

```
// tx-pool/src/component/orphan.rs, line 16
pub(crate) const DEFAULT_MAX_ORPHAN_TRANSACTIONS: usize = 100;
``` [1](#0-0) 

No fee rate is checked before insertion because the inputs are unresolved — the fee cannot be computed. The eviction logic in `limit_size()` first removes expired entries, then evicts **randomly** using `HashMap::keys().next()` until the pool is at or below 100: [2](#0-1) 

The entry path from the P2P relay layer is in `after_process`: when a relayed transaction fails with `is_missing_input`, it is unconditionally forwarded to `add_orphan`: [3](#0-2) 

There is no per-peer quota on orphan pool slots. The relayer's per-peer rate limiter (30 req/s) only throttles message frequency, not orphan pool occupancy: [4](#0-3) 

An attacker connects as a P2P peer and sends 100 `RelayTransactions` messages, each containing a structurally valid transaction whose inputs reference fabricated, non-existent `OutPoint`s. Each transaction passes the relay message check, fails input resolution, and is placed in the orphan pool. Because the fabricated inputs will never appear on-chain, the attacker's entries persist for the full `ORPHAN_TX_EXPIRE_TIME` (100 × `MAX_BLOCK_INTERVAL`): [5](#0-4) 

Once the pool is saturated, every subsequent legitimate orphan transaction triggers a random eviction. The attacker can continuously re-submit to replace any slot freed by expiry, maintaining saturation indefinitely at zero on-chain cost.

---

### Impact Explanation

CKB's two-phase commit model (propose → commit) means that child transactions sent before their parent is confirmed are a normal, expected use case. These child transactions enter the orphan pool and are promoted once the parent is seen. An attacker who saturates the orphan pool causes legitimate child transactions to be randomly evicted before their parent arrives. The evicted transaction is silently dropped — the node sends a `TxVerificationResult::Reject` back to the relayer, marking the tx as `unknown` in the bloom filter, and the user must resubmit. Under sustained attack, resubmitted transactions are evicted again immediately, effectively preventing any orphan transaction from persisting long enough to be resolved. This disrupts normal transaction relay for all peers connected to the targeted node.

---

### Likelihood Explanation

The attack requires only a single P2P connection to the target node, which is open to any unprivileged peer on the network. The attacker needs no CKB tokens, no on-chain fees, and no special privileges. Filling 100 slots at 30 req/s takes approximately 3–4 seconds. Maintaining saturation requires only periodic re-submission to replace expired entries. The attack is cheap, repeatable, and requires no coordination.

---

### Recommendation

1. **Per-peer orphan slot quota**: Track orphan entries by originating `PeerIndex` and cap the number of orphan transactions accepted from any single peer (e.g., 10–15 slots per peer). This is the standard mitigation used by Bitcoin Core.
2. **Fee-rate-ordered eviction**: Instead of random eviction, evict the entry with the lowest declared cycle-per-byte ratio, making it costly to maintain high-priority slots.
3. **Increase the cap with per-peer accounting**: A larger global cap combined with per-peer limits prevents a single peer from monopolizing the pool.

---

### Proof of Concept

1. Attacker establishes a P2P connection to a CKB node using the `RelayV3` protocol.
2. Attacker constructs 100 transactions, each spending a unique fabricated `OutPoint` (random tx hash, index 0) that does not exist on-chain or in the pool.
3. Attacker sends each transaction as a `RelayTransactions` message with an arbitrary `declared_cycle` value.
4. Each transaction fails input resolution in `after_process` → `is_missing_input` → `add_orphan` → `OrphanPool::add_orphan_tx`.
5. After 100 submissions, `limit_size()` is called on every subsequent insertion and randomly evicts one entry.
6. A legitimate user sends a child transaction (e.g., spending the output of an unconfirmed parent). It enters the orphan pool but is immediately randomly evicted with ~100% probability while the attacker maintains 100 slots.
7. The attacker periodically re-sends expired entries to keep the pool saturated. The legitimate user's transaction is never retained long enough for the parent to arrive and resolve it.

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
