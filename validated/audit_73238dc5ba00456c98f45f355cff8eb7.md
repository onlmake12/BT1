### Title
Unbounded `setExclusivityPeriod()` Allows Admin to Permanently Block Callback Execution and Lock User Fees - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

The `Echo` contract's `setExclusivityPeriod()` function accepts any `uint32` value with no upper-bound cap. If the admin sets `exclusivityPeriodSeconds` to `type(uint32).max` (~136 years), only the originally assigned provider can ever call `executeCallback()` for any pending request. Since there is no user-facing cancel or refund function, user fees paid at request time are permanently locked in the contract if the assigned provider is unresponsive or malicious.

### Finding Description

`setExclusivityPeriod()` in `Echo.sol` imposes no maximum on the value it accepts:

```solidity
function setExclusivityPeriod(uint32 periodSeconds) external override {
    require(msg.sender == _state.admin, "Only admin can set exclusivity period");
    uint256 oldPeriod = _state.exclusivityPeriodSeconds;
    _state.exclusivityPeriodSeconds = periodSeconds;
    emit ExclusivityPeriodUpdated(oldPeriod, periodSeconds);
}
``` [1](#0-0) 

The exclusivity check in `executeCallback()` gates all non-assigned providers:

```solidity
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(providerToCredit == req.provider,
        "Only assigned provider during exclusivity period");
}
``` [2](#0-1) 

User fees are stored in `req.fee` at request time and are only credited to a provider upon successful `executeCallback()`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [3](#0-2) 

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [4](#0-3) 

There is no `cancelRequest()` or user-facing refund function anywhere in the contract. The only fee-exit paths are `withdrawFees()` (admin-only, for Pyth protocol fees) and `withdrawAsFeeManager()` (provider fee manager only). Neither returns funds to the original requester. [5](#0-4) 

The `EchoState` struct confirms `exclusivityPeriodSeconds` is a plain `uint32` with no enforced ceiling: [6](#0-5) 

### Impact Explanation

If the admin sets `exclusivityPeriodSeconds = type(uint32).max` (4,294,967,295 seconds ≈ 136 years):

1. Every pending and future request enters a permanent exclusivity window.
2. Only the request's originally assigned provider can call `executeCallback()`.
3. If that provider is offline, deregistered, or colluding with the admin, no callback is ever executed.
4. The user's ETH (stored in `req.fee`) is irrecoverably locked — there is no refund path.
5. The Pyth protocol fee portion (`_state.pythFeeInWei`) is already credited to `_state.accruedFeesInWei` at request time, so even the admin's `withdrawFees()` does not return funds to users.

This is a direct structural analog to the DaosLive `extendTime()` bug: a privileged role can manipulate a time parameter without bound, permanently denying users access to their funds.

### Likelihood Explanation

The admin is a single address set at initialization with no on-chain multisig requirement enforced by the `Echo` contract itself. The `setExclusivityPeriod()` function requires only `msg.sender == _state.admin` — a single transaction. A compromised or malicious admin key can execute this atomically. The `uint32` type allows values up to ~136 years, which is practically infinite. No timelock, governance vote, or cap prevents this. [7](#0-6) 

### Recommendation

1. **Impose a hard cap** on `exclusivityPeriodSeconds`, e.g., `require(periodSeconds <= MAX_EXCLUSIVITY_PERIOD, "Exceeds max exclusivity period")` where `MAX_EXCLUSIVITY_PERIOD` is a reasonable constant (e.g., 300 seconds).
2. **Add a user refund path**: implement a `cancelRequest(uint64 sequenceNumber)` function callable by the requester after the exclusivity period has elapsed without fulfillment, returning `req.fee` to `req.requester`.
3. **Apply the new cap retroactively** to the exclusivity check so that existing requests are not permanently locked if the period is later reduced.

### Proof of Concept

```solidity
// 1. Admin sets exclusivity period to max uint32 (~136 years)
vm.prank(admin);
echo.setExclusivityPeriod(type(uint32).max);

// 2. User submits a request and pays fee
vm.deal(user, totalFee);
vm.prank(user);
uint64 seq = echo.requestPriceUpdatesWithCallback{value: totalFee}(
    defaultProvider, block.timestamp, priceIds, callbackGasLimit
);

// 3. Any non-assigned provider (or user themselves) tries to execute callback
vm.warp(block.timestamp + 365 days * 10); // 10 years later
vm.prank(secondProvider);
vm.expectRevert("Only assigned provider during exclusivity period");
echo.executeCallback(secondProvider, seq, updateData, priceIds);

// 4. No refund function exists — user's fee is permanently locked
// echo.cancelRequest(seq); // Does not exist
```

The user's fee remains in `req.fee` indefinitely with no recovery mechanism, directly mirroring the DaosLive refund-blocking pattern.
