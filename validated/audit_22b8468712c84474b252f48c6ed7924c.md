### Title
Unchecked `i64`-to-`u64` Cast on DB-Read Capacity Produces Inflated Cell Capacity in Rich-Indexer RPC — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`, `get_cells.rs`, `insert.rs`)

---

### Summary

The rich-indexer stores CKB cell capacity (a `u64`) as a signed `i64` in the database, then reads it back and casts it to `u64` with an unchecked Rust `as` cast. If the stored `i64` is negative — which can occur when the SQL `SUM` of capacities overflows `BIGINT`, or if any individual capacity exceeds `i64::MAX` — the cast silently produces a massively inflated `u64` value. This is the direct analog of the `int256 → uint256` unchecked cast in `PluginPriceOracle`.

---

### Finding Description

**Write path** — `build_output_cell_rows` in `insert.rs`:

```rust
let cell_capacity: u64 = cell.capacity().into();
let cell_row = (
    output_index as i32,
    cell_capacity as i64,   // ← unchecked u64→i64; wraps negative if > i64::MAX
    ...
``` [1](#0-0) 

**Read path 1** — `get_cells_capacity` RPC:

```rust
let capacity = match capacity {
    Some(capacity) => capacity as u64,   // ← unchecked i64→u64; no negativity check
    None => return Ok(None),
};
``` [2](#0-1) 

The SQL query that feeds this value is:

```sql
CAST(SUM(output.capacity) AS BIGINT) AS total_capacity
``` [3](#0-2) 

If the `BIGINT` sum overflows (e.g., in SQLite where integer overflow wraps silently), the `i64` result is negative. The subsequent `capacity as u64` cast in Rust is a bitwise reinterpretation — a negative `i64` such as `-1` becomes `u64::MAX` (18446744073709551615).

**Read path 2** — `build_indexer_cell` in `get_cells.rs`:

```rust
let output = CellOutputBuilder::default()
    .capacity(row.get::<i64, _>("capacity") as u64)   // ← same unchecked cast
``` [4](#0-3) 

**Read path 3** — `build_cell_output` in `insert.rs`:

```rust
let cell_output = CellOutput::new_builder()
    .capacity(capacity as u64)   // ← same unchecked cast
``` [5](#0-4) 

Notably, the codebase already recognises the `u64`/`i64` boundary problem for *filter* values and explicitly clamps them in `convert_max_values_in_search_filter`, but applies no equivalent guard on the capacity values that are *read back* from the database. [6](#0-5) 

---

### Impact Explanation

Any RPC caller invoking `get_cells_capacity` or `get_cells` against a rich-indexer node receives the inflated capacity value. Downstream wallets, dApps, and tooling that rely on these endpoints to determine available CKB balance or to select cells for transaction construction would observe a wildly incorrect (inflated) capacity, leading to:

- Incorrect cell selection / fee calculation in transaction builders.
- False belief that an address holds far more CKB than it actually does.
- Potential inability to construct valid transactions (capacity mismatch at signing time).

---

### Likelihood Explanation

CKB's current total issuance (~33.6 billion CKB = ~3.36 × 10¹⁸ shannons) is below `i64::MAX` (~9.22 × 10¹⁸), so no single valid cell and no honest aggregate sum can currently overflow `i64`. However:

1. **SQLite silent integer overflow**: SQLite's integer arithmetic wraps silently on overflow. A sufficiently large `SUM` across many cells (e.g., querying all cells on a heavily used address) could overflow `BIGINT` in SQLite and return a negative value without any error.
2. **Future protocol changes**: Any increase to the total CKB supply cap or a change in the capacity unit would push values closer to or past `i64::MAX`.
3. **Database corruption / direct DB writes**: An operator or attacker with DB write access (e.g., via a separate bug) could insert a row with a capacity stored as a negative `i64`, which would then be silently reinterpreted as a huge `u64` on every read.

The entry path is fully unprivileged: any RPC caller can trigger `get_cells_capacity` or `get_cells` with no authentication.

---

### Recommendation

Add an explicit non-negativity check before every `i64 as u64` cast on values read from the database:

```rust
// get_cells_capacity.rs
let capacity = match capacity {
    Some(c) if c >= 0 => c as u64,
    Some(c) => return Err(Error::DB(format!("negative capacity in DB: {}", c))),
    None => return Ok(None),
};
```

Apply the same guard in `build_indexer_cell` (`get_cells.rs` line 271) and `build_cell_output` (`insert.rs` line 634). Optionally, add a guard on the write side in `build_output_cell_rows` to reject any `u64` capacity that exceeds `i64::MAX` before storing it.

---

### Proof of Concept

1. Start a CKB node with the rich-indexer enabled (SQLite backend).
2. Index enough cells so that `SUM(output.capacity)` for a given lock script approaches or exceeds `i64::MAX` (achievable on a test network by issuing cells with large capacities).
3. Call `indexer_get_cells_capacity` via RPC for that lock script.
4. Observe that the returned `capacity` field is a huge `u64` value (e.g., `18446744073709551615`) instead of the correct sum, because the SQL `BIGINT` sum wrapped to a negative `i64` and was then cast to `u64` without validation.

// This function is used to convert u64::max values to i64::max in an IndexerSearchKeyFilter instance.
// The primary reason for this conversion is the limitation of the relational database used by the rich-indexer.
// The database can only handle integers up to i64::max.
// Secondly, in our application, the range of i64 is sufficient for our needs, so converting u64::max to i64::max does not cause any loss of information.
// Therefore, before passing the filter to the rich-indexer, we need to convert u64::max values to i64::max.
pub(crate) fn convert_max_values_in_search_filter(
    filter: &Option<IndexerSearchKeyFilter>,
) -> Option<IndexerSearchKeyFilter> {
    filter.as_ref().map(|f| {
        let convert_range = |range: &Option<IndexerRange>| -> Option<IndexerRange> {
            range.as_ref().map(|r| {
                let start = if (i64::MAX as u64) < u64::from(r.start()) {
                    i64::MAX as u64
                } else {
                    r.start().into()
                };
                let end = if (i64::MAX as u64) < u64::from(r.end()) {
                    i64::MAX as u64
                } else {
                    r.end().into()
                };
                IndexerRange::new(start, end)
            })
        };

        IndexerSearchKeyFilter {
            script: f.script.clone(),
            script_len_range: convert_range(&f.script_len_range),
            output_data: f.output_data.clone(),
            output_data_filter_mode: f.output_data_filter_mode,
            output_data_len_range: convert_range(&f.output_data_len_range),
            output_capacity_range: convert_range(&f.output_capacity_range),
            block_range: convert_range(&f.block_range),
        }
    })
}
