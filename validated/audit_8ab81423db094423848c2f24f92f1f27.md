### Title
`TtlFilter::contains()` Ignores TTL Expiry in `SyncState::already_known_tx()`, Causing Stale Transaction Filter Entries to Suppress Re-Relay Requests — (`File: sync/src/types/mod.rs`)

---

### Summary

`TtlFilter::contains()` does not check whether a stored entry has exceeded its TTL. The `SyncState::already_known_tx()` method calls `contains()` directly without first calling `remove_expired()`, meaning expired ("stale") transaction hashes are still reported as "known." This is the direct CKB analog of the Chainlink stale-price bug: data is consumed from a time-bounded store without verifying that the stored value is still within its valid period.

---

### Finding Description

`TtlFilter<T>` in `sync/src/types/mod.rs` is designed to track recently-seen transaction hashes with a TTL. It stores each item alongside its insertion timestamp and provides `remove_expired()` to evict stale entries. However, `contains()` performs no TTL check whatsoever:

```rust
// sync/src/types/mod.rs  lines 345-347
pub fn contains(&self, item: &T) -> bool {
    self.inner.contains(item)   // ← no TTL check
}
```

The correct pattern — used in both `transaction_hashes_process.rs` and `transactions_process.rs` — is to call `remove_expired()` before `contains()`:

```rust
// sync/src/relayer/transaction_hashes_process.rs  lines 39-46
let mut tx_filter = state.tx_filter();
tx_filter.remove_expired();          // ← expiry enforced here
...
.filter(|tx_hash| !tx_filter.contains(tx_hash))
```

But `SyncState::already_known_tx()` skips this step entirely:

```rust
// sync/src/types/mod.rs  lines 1534-1536
pub fn already_known_tx(&self, hash: &Byte32) -> bool {
    self.tx_filter.lock().contains(hash)   // ← remove_expired() never called
}
```

`already_known_tx()` is called in `sync/src/relayer/block_proposal_process.rs` to decide whether to request a proposed transaction from a peer. Because expired entries are never evicted before the check, a transaction hash that was seen long ago (beyond the TTL) is still treated as "known," and the node silently skips requesting it.

---

### Impact Explanation

When a block proposal arrives containing a transaction whose hash is in the stale filter, `already_known_tx()` returns `true` and the node does not issue a fetch request for that transaction. If the transaction was evicted from the local tx-pool in the interim (e.g., due to pool expiry or a restart), the node will be unable to validate the committed block containing that transaction without an additional round-trip or re-sync. In a targeted scenario, an adversary who previously relayed a transaction hash to a victim node can exploit the stale filter window to cause the victim to silently skip re-fetching a transaction it no longer holds, delaying or disrupting its block-processing pipeline.

---

### Likelihood Explanation

Any unprivileged P2P peer can relay transaction hashes to a node (via `RelayTransactionHashes`), causing those hashes to be inserted into `tx_filter`. After the TTL elapses, the entries become stale but are never evicted by `already_known_tx()`. The window of staleness is bounded by the LRU capacity (`FILTER_SIZE`) and the TTL (`FILTER_TTL`), both of which are fixed constants. No special privilege is required; a normal relay peer is sufficient.

---

### Recommendation

Add a TTL check inside `contains()` itself, or call `remove_expired()` before every `contains()` call at every call-site. The cleanest fix is to make `contains()` TTL-aware:

```rust
pub fn contains(&self, item: &T) -> bool {
    if let Some(&inserted_at) = self.inner.peek(item) {
        let now = ckb_systemtime::unix_time().as_secs();
        inserted_at + self.ttl >= now
    } else {
        false
    }
}
```

Alternatively, mirror the pattern already used in `transaction_hashes_process.rs` and `transactions_process.rs` by calling `remove_expired()` inside `already_known_tx()` before delegating to `contains()`.

---

### Proof of Concept

1. Peer A sends a `RelayTransactionHashes` message containing hash `H` to the victim node. `H` is inserted into `tx_filter` with timestamp `T0`.
2. The TTL elapses (`now > T0 + FILTER_TTL`). The entry is stale but `remove_expired()` has not been called via `already_known_tx()`.
3. A miner commits a block containing transaction `H`. The victim node receives a `BlockProposal` message listing `H`.
4. `block_proposal_process.rs` calls `already_known_tx(&H)` → `tx_filter.lock().contains(&H)` → returns `true` (stale entry still present).
5. The node does not request transaction `H` from any peer. If `H` has since been evicted from the local pool, the node cannot validate the committed block without a separate sync mechanism. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    /// Removes expired items.
    pub fn remove_expired(&mut self) {
        let now = ckb_systemtime::unix_time().as_secs();
        let expired_keys: Vec<T> = self
            .inner
            .iter()
            .filter_map(|(key, time)| {
                if *time + self.ttl < now {
                    Some(key)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        for k in expired_keys {
            self.remove(&k);
        }
    }
