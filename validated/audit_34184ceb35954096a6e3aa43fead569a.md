Audit Report

## Title
Stale `evict_key` on Ancestor Entries After `remove_entry_and_descendants` — (File: tx-pool/src/component/pool_map.rs)

## Summary
`remove_entry_and_descendants` tears down all link records via `remove_entry_links` before calling `remove_entry` on each removed transaction. Because `update_ancestors_index_key` relies on `calc_ancestors` to find surviving ancestors, and `calc_ancestors` returns an empty set once the link record is deleted, surviving ancestors never receive `sub_descendant_weight` and their `evict_key` is never recomputed. This leaves ancestors with permanently inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, `descendants_count`, and `evict_key`, corrupting the pool's eviction ordering.

## Finding Description
`remove_entry_and_descendants` first collects the root and all descendant IDs, then calls `remove_entry_links` on every one of them before iterating to call `remove_entry`:

```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
for id in &removed_ids {
    self.remove_entry_links(id);
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
```

`remove_entry_links` removes the target from its parents' children sets and then calls `self.links.remove(id)`, which deletes the entry's own link record from `TxLinksMap.inner` entirely.

After that teardown, `remove_entry` is called for each removed ID. Inside `remove_entry`, `update_ancestors_index_key` is invoked at L242. `update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())` to discover which pool entries to update. `calc_ancestors` delegates to `calc_relative_ids`, which calls `self.inner.get(short_id)`. Since `remove_entry_links` already called `self.links.remove(id)`, the entry is absent from `inner` and `get` returns `None`, yielding an empty `HashSet`. The `for anc_id in &ancestors` loop body never executes. Surviving ancestors never receive `sub_descendant_weight`, and their `evict_key` is never recomputed.

The comment at L256 confirms the intent was only to suppress `update_descendants_index_key` for removed entries, but the side effect silently breaks ancestor accounting as well.

## Impact Explanation
When the pool reaches `max_tx_pool_size`, `limit_size` calls `next_evict_entry` in a loop to drop the lowest-priority transaction. `next_evict_entry` uses `iter_by_evict_key` as the sole ordering criterion. Because an ancestor's `evict_key` is stale and inflated, it is ranked higher than its true fee rate warrants and is skipped in favour of transactions with genuinely lower fee rates. The result is that a low-fee-rate ancestor survives eviction rounds it should not survive, while legitimate higher-fee-rate transactions submitted by honest users are rejected with `Reject::Full`. This constitutes a **High** impact: a bad design that can cause CKB network congestion with few costs, as the attack is cheap, repeatable, and requires no privilege.

## Likelihood Explanation
The trigger is fully unprivileged and requires only standard RPC access (`send_transaction`). The attacker submits a low-fee parent P, then a high-fee child C spending P's output (inflating P's `evict_key`), then a conflicting child C′ (triggering `resolve_conflict` → `remove_entry_and_descendants(C)`, which fails to deflate P's `evict_key`). Repeating steps 2–3 with fresh children keeps P's `evict_key` perpetually inflated. No majority hash power, no privileged role, and no social engineering is required. The cost per inflation cycle is only the fee for submitting two unconfirmed transactions.

## Recommendation
Snapshot the surviving ancestors of the root transaction **before** tearing down any link records. After all removals are complete, iterate over the removed entries and call `sub_descendant_weight` + recompute `evict_key` for each surviving ancestor:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Snapshot surviving ancestors BEFORE tearing down links.
    let surviving_ancestors: HashSet<ProposalShortId> = self
        .links
        .calc_ancestors(id)
        .into_iter()
        .filter(|a| !removed_ids.contains(a))
        .collect();

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Update surviving ancestors' descendant accounting.
    for removed_entry in &removed {
        for anc_id in &surviving_ancestors {
            self.entries.modify_by_id(anc_id, |e| {
                e.inner.sub_descendant_weight(removed_entry);
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    removed
}
```

## Proof of Concept
```
State: pool is near its size limit.

1. submit_transaction(P)   // low fee rate, e.g. 1 shannon/byte
   → P.descendants_fee = P.fee
   → P.evict_key reflects fee_rate(P)

2. submit_transaction(C)   // high fee rate, spends P output
   → record_entry_descendants → update_ancestors_index_key(C, Add) fires
   → P.descendants_fee += C.fee   (inflated)
   → P.evict_key = max(descendants_feerate, feerate)  ← now high

3. submit_transaction(C')  // conflicts with C (same input)
   → resolve_conflict → remove_entry_and_descendants(C)
   → remove_entry_links(C): self.links.remove(C) deletes C's link record
   → remove_entry(C): update_ancestors_index_key calls calc_ancestors(C)
   → calc_ancestors returns ∅ (C's record gone from links.inner)
   → P.descendants_fee is NOT decremented  (still inflated)
   → P.evict_key is NOT recomputed         (still high)

4. Pool fills up; limit_size calls next_evict_entry.
   → P is skipped because its evict_key shows a high fee rate.
   → A legitimate transaction Q with a genuinely higher fee rate
     than P is evicted instead.
   → Q's submitter receives Reject::Full.

5. Repeat steps 2–3 with fresh children to keep P's evict_key
   permanently inflated at zero marginal cost.
```
