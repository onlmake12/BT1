### Title
Incorrect TWAP Price for Negative `priceDiff` Due to Truncation-Toward-Zero Instead of Floor Division — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol` and `target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

Both the EVM (`Pyth.sol`) and Solana (`lib.rs`) TWAP implementations divide a signed `priceDiff`/`price_diff` by an unsigned `slotDiff`/`slot_diff` using native integer division, which truncates toward zero in both Solidity and Rust. When the cumulative price difference is negative (prices falling) and not exactly divisible by the slot count, the computed TWAP is 1 unit higher than the mathematically correct floor value. This is the same rounding-direction bug as the Uniswap `OracleLibrary` tick issue in the reference report.

---

### Finding Description

**Solidity — `calculateTwap` in `Pyth.sol`:**

```solidity
int128 priceDiff = twapPriceInfoEnd.cumulativePrice -
    twapPriceInfoStart.cumulativePrice;
// ...
int128 twapPrice = priceDiff / int128(uint128(slotDiff));  // truncates toward zero
``` [1](#0-0) 

**Rust — `calculate_twap` in `lib.rs`:**

```rust
let price_diff = end_msg.cumulative_price
    .checked_sub(start_msg.cumulative_price)...;
// ...
let price = i64::try_from(price_diff / i128::from(slot_diff))  // truncates toward zero
``` [2](#0-1) 

In both Solidity and Rust, signed integer division truncates toward zero (not toward negative infinity). When `priceDiff < 0` and `priceDiff % slotDiff != 0`, the result is one unit larger than the correct mathematical floor. For example:

- `priceDiff = -7`, `slotDiff = 3`
- Correct floor: `⌊-7/3⌋ = -3`
- Truncation toward zero: `-7 / 3 = -2` ← **returned value is wrong by +1**

Uniswap's `OracleLibrary` explicitly corrects for this with:
```solidity
if (tickCumulativesDelta < 0 && (tickCumulativesDelta % secondsAgo != 0)) arithmeticMeanTick--;
```

Pyth's TWAP code has no equivalent correction in either implementation.

---

### Impact Explanation

The TWAP price returned by `parseTwapPriceFeedUpdates` (EVM) or `post_twap_updates` (Solana) is systematically biased **upward by 1 unit** whenever:
1. The cumulative price has decreased over the TWAP window (`priceDiff < 0`), and
2. The price difference is not exactly divisible by the slot count.

This means the TWAP price is slightly inflated during falling-price periods. Downstream consumers (e.g., lending protocols, derivatives, liquidation engines) using the TWAP for collateral valuation or liquidation thresholds will receive a price that is 1 unit higher than correct. The magnitude of the error is 1 unit at the price's base denomination (e.g., for a feed with exponent `-8`, this is `10^-8` of the quote currency). While small per instance, the error is deterministic and systematic — it always rounds in the same direction, which can be exploited or can cause consistent mispricing in protocols that rely on the TWAP.

---

### Likelihood Explanation

This condition is triggered any time:
- A TWAP is computed over a window where prices have been net-falling, **and**
- The cumulative price difference is not exactly divisible by the slot count.

Both conditions are routine in normal market operation. Any unprivileged user can submit valid TWAP update messages (start + end) to trigger `calculateTwap`/`calculate_twap`. No special access is required.

---

### Recommendation

Apply the same floor-division correction used by Uniswap's `OracleLibrary`.

**Solidity (`Pyth.sol`):**
```solidity
int128 twapPrice = priceDiff / int128(uint128(slotDiff));
if (priceDiff < 0 && (priceDiff % int128(uint128(slotDiff)) != 0)) {
    twapPrice -= 1;
}
```

**Rust (`lib.rs`):**
```rust
let mut price_quotient = price_diff / i128::from(slot_diff);
if price_diff < 0 && price_diff % i128::from(slot_diff) != 0 {
    price_quotient -= 1;
}
let price = i64::try_from(price_quotient)
    .map_err(|_| ReceiverError::TwapCalculationOverflow)?;
```

### Proof of Concept

**Concrete numeric example (EVM):**

- `twapPriceInfoStart.cumulativePrice = 0`
- `twapPriceInfoEnd.cumulativePrice = -7`
- `slotDiff = 3`

Current code:
```
priceDiff = -7
twapPrice = -7 / 3 = -2  (Solidity truncates toward zero)
```

Correct result:
```
twapPrice = ⌊-7 / 3⌋ = -3
```

The returned TWAP price is `-2` instead of the correct `-3`. Any consumer reading this TWAP sees a price 1 unit higher than the true time-weighted average, consistently during falling-price windows where the division is inexact. [3](#0-2) [4](#0-3)
