### Title
`max_request_body_size` Config Never Enforced — Unbounded `BytesVisitor` Allocation via RPC — (`rpc/src/server.rs`, `util/jsonrpc-types/src/bytes.rs`)

### Summary

The `max_request_body_size` configuration field (default 10 MiB) is parsed and stored but **never applied** to the Axum HTTP server. No `RequestBodyLimitLayer` or equivalent middleware exists anywhere in the RPC server setup. As a result, `BytesVisitor::visit_str` can be driven to allocate `vec![0; bytes.len() >> 1]` from an arbitrarily large hex string with no cap, enabling a local caller to OOM-kill the node process.

### Finding Description

**Dead config guard.** `RpcConfig` carries `max_request_body_size` [1](#0-0)  and the default `ckb.toml` documents it as "Default is 10MiB" [2](#0-1) . However, `RpcServer::new` only consumes `config.rpc_batch_limit`, `config.listen_address`, `config.ws_listen_address`, and `config.tcp_listen_address` — `max_request_body_size` is never read or passed anywhere. [3](#0-2) 

The Axum router is built with `CorsLayer`, `TimeoutLayer`, and `Extension` layers only — no `tower_http::limit::RequestBodyLimitLayer` or `axum::extract::DefaultBodyLimit` is present. [4](#0-3) 

A grep for `RequestBodyLimit`, `body_limit`, `DefaultBodyLimit` across the entire repo returns zero matches.

**Unbounded allocation.** Once the oversized body reaches `handle_jsonrpc`, it is passed directly to `serde_json` for deserialization. `BytesVisitor::visit_str` then executes: [5](#0-4) 

```rust
let mut buffer = vec![0; bytes.len() >> 1]; // we checked length
hex_decode(bytes, &mut buffer)...
```

There is no size check before this allocation. A 200 MB hex string in a `witness` field causes a ~100 MB allocation per call; a 2 GB string causes a ~1 GB allocation.

**TCP transport is separately limited** to 2 MB via `LinesCodec::new_with_max_length(2 * 1024 * 1024)` [6](#0-5) , but the HTTP transport (the primary and default transport) has no such guard.

### Impact Explanation

A caller with access to the RPC port (localhost by default, but commonly exposed in practice) can send a single `send_transaction` request with one oversized `witness` field and cause the node process to be OOM-killed by the OS. This crashes the node without any PoW, consensus, or authentication requirement. Impact is scoped to **local RPC API crash**.

### Likelihood Explanation

The RPC binds to `127.0.0.1:8114` by default, limiting exposure to local processes. However: (a) many operators expose the RPC publicly, (b) any local unprivileged process on the same machine can reach it, and (c) the config option gives operators a false sense of protection. The exploit requires a single HTTP POST — no special tooling needed.

### Recommendation

Apply `tower_http::limit::RequestBodyLimitLayer` (or Axum's `DefaultBodyLimit`) to the HTTP router using the configured `max_request_body_size` value:

```rust
use tower_http::limit::RequestBodyLimitLayer;

let app = Router::new()
    ...
    .layer(RequestBodyLimitLayer::new(config.max_request_body_size));
```

This must be done in `start_server` in `rpc/src/server.rs`, and `max_request_body_size` must be threaded through as a parameter. The TCP path already has an equivalent guard via `LinesCodec::new_with_max_length`.

### Proof of Concept

```bash
# Generate a 200MB hex string (400MB of hex digits = 200MB decoded)
python3 -c "print('0x' + 'ab' * 100_000_000)" > big_hex.txt

curl -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d "{\"jsonrpc\":\"2.0\",\"method\":\"send_transaction\",\"params\":[{\"version\":\"0x0\",\"cell_deps\":[],\"header_deps\":[],\"inputs\":[],\"outputs\":[],\"outputs_data\":[],\"witnesses\":[\"$(cat big_hex.txt)\"]},\"passthrough\"],\"id\":1}"
```

Monitor RSS of the `ckb` process; it will spike by ~100 MB per request. Repeating the call (or using a single 2 GB string) triggers the OOM killer and kills the node.

        if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
            let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
        }

        let rpc = Arc::new(io_handler);

        let http_address = Self::start_server(
            &rpc,
            config.listen_address.to_owned(),
            handler.clone(),
            false,
        )
        .inspect(|&local_addr| {
            info!("Listen HTTP RPCServer on address: {}", local_addr);
        })
        .unwrap();

        let ws_address = if let Some(addr) = config.ws_listen_address {
            let local_addr =
                Self::start_server(&rpc, addr, handler.clone(), true).inspect(|&addr| {
                    info!("Listen WebSocket RPCServer on address: {}", addr);
                });
            local_addr.ok()
        } else {
            None
        };

        let tcp_address = if let Some(addr) = config.tcp_listen_address {
            let local_addr = handler.block_on(Self::start_tcp_server(rpc, addr, handler.clone()));
            if let Ok(addr) = &local_addr {
                info!("Listen TCP RPCServer on address: {}", addr);
            };
            local_addr.ok()
        } else {
            None
        };

        Self {
            http_address,
            tcp_address,
            ws_address,
        }
    }
