### Title
Unauthenticated `from` PeerId in `ConnectionSync` Enables Forced Outbound TCP Connections to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
Any connected P2P peer can cause a victim CKB node to make repeated outbound TCP connection attempts to arbitrary attacker-controlled IP:port combinations by exploiting two missing identity validations: (1) the `from` field in `ConnectionRequest` is never checked against the actual session peer identity, allowing `pending_delivered` to be seeded with attacker-controlled addresses under an arbitrary spoofed PeerId; (2) the `from` field in `ConnectionSync` is also never checked against the actual session peer identity, allowing the attacker to trigger NAT traversal using those stored addresses.

### Finding Description

**Phase 1 — Seed `pending_delivered` with attacker-controlled addresses**

`ConnectionRequestProcess::execute()` parses the `from` field purely as valid PeerId bytes: [1](#0-0) 

There is no check that `content.from` equals the PeerId of the actual session (`self.peer`). When `self_peer_id == &content.to`, the handler calls `respond_delivered(content.from, ...)`: [2](#0-1) 

`respond_delivered` filters `remote_listens` to TCP/IPv4/IPv6 only (non-TCP transports are dropped), then unconditionally stores them: [3](#0-2) 

Result: `pending_delivered[X] = ([attacker_ip:port], now)` for any arbitrary PeerId `X` chosen by the attacker.

**Phase 2 — Trigger NAT traversal to those addresses**

`ConnectionSyncProcess::new()` does not accept a `peer` parameter at all: [4](#0-3) 

`execute()` parses `from` with the same format-only check: [5](#0-4) 

When `self_peer_id == content.to`, it looks up `pending_delivered` using the attacker-supplied `content.from`: [6](#0-5) 

If found, it spawns a `try_nat_traversal` task for each stored address: [7](#0-6) 

**`try_nat_traversal` behavior**: it retries TCP `connect()` every ~200 ms for a 30-second window — approximately 150 TCP SYN packets per triggered address: [8](#0-7) 

**Why existing mitigations do not block this:**

- `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. The attacker can rotate the spoofed `from` PeerId to bypass it entirely.
- The `HOLE_PUNCHING_INTERVAL` guard in `respond_delivered` only prevents re-seeding the same `from` key within the interval; the attacker only needs one successful seed.
- The TCP address filter in `respond_delivered` (lines 196–215) only drops non-TCP transports; attacker-controlled TCP IPv4/IPv6 addresses pass through.

### Impact Explanation
A single connected peer can cause the victim node to make ~150 outbound TCP SYN packets to any IP:port over 30 seconds per triggered address. With `ADDRS_COUNT_LIMIT` addresses per request, the attacker can fan out to multiple targets per round. Concrete impacts:
- **Port scanning / internal network probing** via the victim's egress
- **SSRF** against internal services reachable from the victim host
- **Connection slot exhaustion** if many attackers coordinate or the attacker rotates `from` PeerIds rapidly

### Likelihood Explanation
The attacker only needs a single established P2P session with the victim. No special privileges, no PoW, no key material. The two-message sequence (`ConnectionRequest` then `ConnectionSync`) is trivially constructable. The victim's peer ID is publicly observable on the P2P network.

### Recommendation
In `ConnectionRequestProcess::execute()`, validate that `content.from` equals the PeerId of the actual session peer (`self.peer`) before calling `respond_delivered`. Similarly, pass the sending `peer`'s PeerId into `ConnectionSyncProcess` and validate `content.from` against it before performing the `pending_delivered` lookup. This enforces the invariant that NAT traversal is only initiated toward addresses authenticated to the actual originating peer.

### Proof of Concept
```
1. Attacker connects to victim as peer A (PeerId = A).
2. Attacker sends ConnectionRequest{from=X, to=<victim_peer_id>, listen_addrs=[attacker_ip:1234]}.
   → Victim stores pending_delivered[X] = ([attacker_ip:1234], now).
3. Attacker sends ConnectionSync{from=X, to=<victim_peer_id>, route=[]}.
   → Victim finds pending_delivered[X], spawns try_nat_traversal(attacker_ip:1234).
   → Victim makes ~150 TCP SYN packets to attacker_ip:1234 over 30 seconds.
4. Observe inbound TCP SYNs at attacker_ip:1234 originating from victim's IP.
```
