### Title
Silent u128-to-u64 Truncation in NervosDAO Withdrawal Capacity Calculation — (File: util/dao/src/lib.rs)

### Summary
In `DaoCalculator::calculate_maximum_withdraw`, the intermediate 128-bit product `withdraw_counted_capacity` is cast to `u64` using a silent Rust `as u64` truncating cast instead of a checked conversion. If the 128-bit result exceeds `u64::MAX`, the high bits are silently discarded, producing a drastically incorrect (too small) withdrawal capacity that is returned as a valid `Ok(...)` result rather than an error.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the maximum withdrawal amount for a NervosDAO cell:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The multiplication is correctly widened to `u128` to avoid overflow in the product. However, the final result is cast back to `u64` using `withdraw_counted_capacity as u64`, which is a **silent wrapping/truncating cast** in Rust. If `withdraw_counted_capacity` exceeds `u64::MAX` (≈1.84×10¹⁹), the upper bits are silently discarded and the function returns `Ok(wrong_small_value)` instead of `Err(DaoError::Overflow)`.

This is in direct contrast to every other analogous u128→u64 narrowing in the same file, all of which use the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern:

- `miner_issuance128`: [2](#0-1) 
- `ar_increase128`: [3](#0-2) 
- `reward128` in `secondary_block_reward`: [4](#0-3) 

The `withdraw_counted_capacity` path is the only one that uses the unchecked `as u64` cast.

### Impact Explanation

If `withdraw_counted_capacity` wraps around on truncation to a small value `v`, then `Capacity::shannons(v).safe_add(occupied_capacity)` may succeed (no overflow), returning `Ok(v + occupied_capacity)` — a value far smaller than the correct withdrawal amount. The depositor would receive far fewer CKB than they are entitled to, with no error signal. This affects:

1. **NervosDAO depositors**: Incorrect (too small) withdrawal capacity returned by `calculate_maximum_withdraw`, which is called during transaction fee verification (`transaction_fee` → `transaction_maximum_withdraw`).
2. **Block DAO field computation**: `withdrawed_interests` calls `transaction_maximum_withdraw`, so a silently wrong value propagates into the DAO field written into block headers, corrupting the global DAO accounting state.
3. **RPC callers**: `calculate_dao_maximum_withdraw` RPC directly calls `calculate_maximum_withdraw` and returns the silently wrong value to users. [5](#0-4) 

### Likelihood Explanation

For `withdraw_counted_capacity` to exceed `u64::MAX`, the product `counted_capacity * withdrawing_ar` must exceed `u64::MAX * deposit_ar`. Since `counted_capacity ≤ u64::MAX` and `withdrawing_ar ≥ deposit_ar`, this requires `withdrawing_ar / deposit_ar > 1 + ε` for a cell with `counted_capacity` near `u64::MAX`. Given the total CKB supply (~33.6 billion CKB = ~3.36×10¹⁸ shannons, roughly 18% of `u64::MAX`) and the slow growth of the accumulate rate `ar`, triggering this on mainnet is practically infeasible today. However, the code defect is real and structurally inconsistent with all other overflow checks in the same function. A future protocol change, a devnet/testnet scenario with large synthetic capacities, or a long-lived deposit over many decades could bring this into range.

### Recommendation

Replace the silent `as u64` cast with the same checked pattern used everywhere else in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This makes the overflow handling consistent with `miner_issuance128`, `ar_increase128`, and `reward128`.

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` (lines 296–350) is intended to verify overflow detection but only catches the case where `safe_add` overflows after the truncation. It does **not** cover the case where truncation produces a small value that passes `safe_add` silently. [6](#0-5) 

A concrete silent-truncation scenario (pseudocode):

```
deposit_ar       = 10_000_000_000_000_000   (10^16, genesis default)
withdrawing_ar   = 20_000_000_000_000_000   (2×, after extreme long-term accumulation)
counted_capacity = 18_446_744_073_709_551_615  (u64::MAX)

withdraw_counted_capacity (u128) = u64::MAX * 2 = 36_893_488_147_419_103_230
withdraw_counted_capacity as u64 = 36_893_488_147_419_103_230 % 2^64
                                 = 18_446_744_073_709_551_614  (u64::MAX - 1)
```

In this case the truncation happens to be close to the correct value, but for other ratios the truncated value can be arbitrarily small (e.g., if `withdraw_counted_capacity = u64::MAX + 1`, the cast yields `0`), causing the function to return `Ok(Capacity::shannons(0 + occupied_capacity))` — a near-zero withdrawal for a maximum-capacity deposit.

fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
