Audit Report

## Title
Unbounded NAT Traversal Task Spawning via Non-Consuming `.get()` in `ConnectionSyncProcess::execute` â€” (`network/src/protocols/hole_punching/component/connection_sync.rs`)

## Summary

`ConnectionSyncProcess::execute` reads `pending_delivered` with `.get()` at line 114 instead of `.remove()`, so the entry is never consumed. Every `ConnectionSync` message that passes the per-`(from, to)` rate limiter spawns a new `runtime::spawn` task holding up to `ADDRS_COUNT_LIMIT=24` TCP connection futures. An attacker with a single P2P session can pre-seed `pending_delivered` via `ConnectionRequest`, then drain file descriptors by sending one `ConnectionSync` per second per seeded `from` PeerID indefinitely, crashing the victim node.

## Finding Description

**Step 1 â€” Seeding `pending_delivered`**

`ConnectionRequestProcess::respond_delivered` inserts into `pending_delivered` when the victim is the `to` target: [1](#0-0) 

A re-insertion guard exists (`HOLE_PUNCHING_INTERVAL = 2 min`), but it only prevents re-insertion â€” the existing entry remains permanently accessible: [2](#0-1) 

The `from` field is parsed from attacker-controlled message content with no check that it matches the actual session peer ID, so the attacker can seed arbitrary `from` PeerIDs: [3](#0-2) 

**Step 2 â€” Non-consuming read in `ConnectionSyncProcess::execute`**

When the victim receives a `ConnectionSync` with `to=victim_peer_id`, it looks up `pending_delivered` with `.get()`, not `.remove()`: [4](#0-3) 

The entry is never removed. Every subsequent `ConnectionSync` for the same `from` finds the same entry and spawns a new `runtime::spawn` task: [5](#0-4) 

This is asymmetric with `ConnectionRequestDeliveredProcess`, which correctly uses `.remove()`: [6](#0-5) 

**Step 3 â€” Rate limiters do not prevent accumulation**

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)` at 1 req/sec: [7](#0-6) 

This limits to 1 task/sec per `(from, to)` pair. The outer per-session rate limiter allows 30 messages/sec: [8](#0-7) 

With 30 distinct `from` PeerIDs (each seeded via one `ConnectionRequest`), the attacker spawns 30 tasks/sec, each holding up to 24 TCP FDs.

**Step 4 â€” `pending_delivered` is only cleaned up every 5 minutes** [9](#0-8) [10](#0-9) 

Entries live for up to `TIMEOUT = 5 min`. Over that window: 30 tasks/sec Ă— 300 sec = **9,000 tasks**, each with up to 24 pending TCP connections = **216,000 concurrent FD-holding connection attempts**.

## Impact Explanation

Each spawned task holds up to 24 TCP sockets in `SYN_SENT` state until the OS connect timeout (~75 s on Linux). Sustained task spawning exhausts the node's file descriptor limit, causing failure to accept new P2P connections, failure to create RPC sockets, and potential panics on internal socket operations â€” a full local DoS. This matches the allowed CKB bounty impact: **High â€” Vulnerabilities which could easily crash a CKB node (10001â€“15000 points)**.

## Likelihood Explanation

- Attacker only needs one live P2P connection to the victim.
- Victim's peer ID is public (advertised in the P2P network).
- No PoW, no key material, no privileged role required.
- Seeding 30 `from` PeerIDs requires only 30 `ConnectionRequest` messages, all sendable within 1 second (bounded by the 30 req/sec outer limiter).
- The attack is fully automatable and sustainable for the lifetime of the P2P session.

## Recommendation

Replace `.get()` with `.remove()` in `ConnectionSyncProcess::execute`:

```rust
// connection_sync.rs, line 111-115
let listens_info = self
    .protocol
    .pending_delivered
    .remove(&content.from)   // was: .get(&content.from)
    .map(|info| info.0);
```

This enforces the invariant that each `pending_delivered` entry triggers at most one NAT traversal attempt, matching the behavior of `inflight_requests.remove()` in `ConnectionRequestDeliveredProcess`.

## Proof of Concept

1. Connect to victim node via P2P.
2. For each `i` in `0..30`: send `ConnectionRequest` with `from=PeerID_i`, `to=victim`, `listen_addrs=[24 Ă— 1.2.3.4:9999]` (unreachable). Victim inserts `pending_delivered[PeerID_i] = ([24 addrs], now)`.
3. Every second, send 30 `ConnectionSync` messages, one per `PeerID_i` with `to=victim`.
4. Each message passes the 1/sec `forward_rate_limiter` per `(PeerID_i, victim)` pair and spawns a new `select_ok` task with 24 TCP futures.
5. After 60 seconds: 1,800 tasks Ă— 24 sockets = **43,200 FDs consumed**.
6. After 300 seconds (one cleanup cycle): **216,000 FDs consumed**.
7. Observe node failure to accept new connections / RPC socket errors.
