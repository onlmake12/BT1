Audit Report

## Title
`ScriptHashTypeVerifier` Skips Type Script Hash-Type Validation, Enabling Cheap Contextual-Verification DoS — (File: `verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` validates the hash type of each output's lock script but never inspects the optional type script. A transaction whose output carries a type script with a disallowed hash type (e.g., `Data3` = 6) passes `NonContextualTransactionVerifier::verify()` and is forwarded into the contextual pipeline — cell-dep and input resolution from RocksDB, script-group construction — before being rejected by `select_version()`. Because the input cell is never consumed, an attacker with a single live cell can sustain this at zero ongoing fee cost.

## Finding Description

`ScriptHashTypeVerifier::verify()` (lines 796–814) iterates over outputs and checks only `output.lock().hash_type()`:

```rust
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { … }
    } else { … }
}
``` [1](#0-0) 

`output.type_().to_opt()` is never called. `ENABLED_SCRIPT_HASH_TYPE` is `{0, 1, 2, 4}`, so `Data3` (6) is excluded, but only for lock scripts. [2](#0-1) 

`NonContextualTransactionVerifier::verify()` calls `self.script_hash_type.verify()` as its final gate before the transaction enters the contextual pipeline: [3](#0-2) 

The downstream rejection in `select_version()` (`script/src/types.rs`) is reached only after cell-dep and input resolution (RocksDB lookups) and script-group construction — exactly the expensive work the non-contextual gate was designed to prevent. The VM is not instantiated, but the resolution and grouping overhead is real.

Existing tests cover only the lock-script path: [4](#0-3) 

No test exercises a disallowed hash type on the type script slot.

## Impact Explanation

This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The tx-pool verification worker is forced to perform repeated RocksDB cell lookups and script-group construction for every submitted transaction. Because each transaction is rejected before fee deduction and the input cell is never spent, a single small UTXO sustains an indefinite stream of expensive-to-verify, free-to-submit transactions, degrading throughput for legitimate users.

## Likelihood Explanation

Any CKB holder with one live cell can trigger this. Crafting a `Script` with `hash_type = 6` is trivial via the `send_transaction` JSON-RPC endpoint. No mining power, no privileged access, and no key material beyond the attacker's own wallet is required. The attack is repeatable with no cooldown and no fee drain.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to validate the type script when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto<ScriptHashType>` + `ENABLED_SCRIPT_HASH_TYPE` logic. Add unit tests for both `InvalidScriptHashType` and `ScriptHashTypeNotPermitted` error paths on the type script slot, mirroring `test_unknown_hash_type_output_lock` and `test_not_enabled_hash_type_output_lock`. [5](#0-4) 

## Proof of Concept

1. Obtain any live cell (any wallet holding CKB).
2. Build a transaction:
   - Input: the live cell.
   - Output: a `CellOutput` whose `type_` field is a `Script` with `hash_type = 6` (`Data3`) and any 32-byte `code_hash`.
3. Submit via `send_transaction` JSON-RPC.
4. Observe: the node does **not** return `ScriptHashTypeNotPermitted` at the non-contextual stage; it proceeds to cell-dep/input resolution and script-group construction before returning `InvalidScriptHashType`.
5. Because the transaction is rejected, the input cell remains unspent. Repeat step 3 in a tight loop.

A unit test analogous to `test_not_enabled_hash_type_output_lock` (lines 100–122) but setting `hash_type = Data3` on the output's type script (not lock script) will confirm that `ScriptHashTypeVerifier::verify()` incorrectly returns `Ok(())` instead of `Err(ScriptHashTypeNotPermitted)`. [1](#0-0)

#[test]
pub fn test_unknown_hash_type_output_lock() {
    let transaction = TransactionBuilder::default()
        .output(
            CellOutput::new_builder()
                .lock(Script::default().as_builder().hash_type(3).build())
                .build(),
        )
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);

    assert_error_eq!(
        verifier.verify().unwrap_err(),
        TransactionError::InvalidScriptHashType {
            hash_type: 3.into(),
        },
    );
}
