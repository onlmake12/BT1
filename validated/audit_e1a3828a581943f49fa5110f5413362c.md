### Title
Transaction Submission Accepted Before Tx-Pool Service Is Ready — (`rpc/src/module/pool.rs`, `sync/src/relayer/transactions_process.rs`)

---

### Summary

The `send_transaction` RPC handler and the relay-layer `TransactionsProcess::execute` both submit transactions into the tx-pool without first checking whether the tx-pool service has completed its initialization (`service_started()`). The node exposes a dedicated `tx_pool_ready` RPC precisely to signal this readiness state, but neither the local RPC submission path nor the remote relay submission path enforces it. This is the direct CKB analog of the missing `whenNotPaused` guard: an operation that should be gated on a readiness/state flag proceeds unconditionally.

---

### Finding Description

`TxPoolController` tracks initialization state via an `AtomicBool` named `started`: [1](#0-0) 

This flag is set to `true` only at the very end of `TxPoolServiceBuilder::start()`, **after** the background processing loops are spawned but **before** `load_persisted_data` is called to re-inject previously persisted transactions: [2](#0-1) 

The node exposes `tx_pool_ready` as an explicit readiness probe: [3](#0-2) 

However, `send_transaction` — the primary RPC entry point for any local RPC caller — performs no such check before forwarding the transaction to the pool: [4](#0-3) 

Identically, the P2P relay handler `TransactionsProcess::execute` — invoked for every `RelayTransactions` message received from a connected peer — submits directly to the pool without checking `service_started()`: [5](#0-4) 

---

### Impact Explanation

During the startup window — after the node begins accepting RPC and P2P connections but before `started` is set to `true` and `load_persisted_data` completes — the tx-pool's in-memory state is incomplete: previously persisted transactions have not yet been re-inserted. Any transaction submitted during this window is evaluated against an empty or partially populated pool.

Concretely:

1. **False rejection of valid transactions**: A transaction that spends an output from a persisted (not yet re-loaded) pending transaction will be rejected as spending a dead/unknown cell, even though it would be valid once the pool is fully initialized.
2. **Operational inconsistency**: The pool processes externally submitted transactions interleaved with the persisted-data reload, producing non-deterministic admission results depending on race timing.
3. **Relay path exposure**: A P2P peer can trigger the same condition by sending `RelayTransactions` messages immediately after connecting, with no authentication or rate-limit barrier specific to pool readiness.

---

### Likelihood Explanation

The vulnerability window is the node startup period. Any RPC caller (e.g., a wallet, a script, a monitoring tool) that calls `send_transaction` immediately after the node's RPC port becomes reachable — without first calling `tx_pool_ready` — will hit this window. The relay path is reachable by any connected P2P peer with no additional preconditions. The window is short but deterministically present on every node restart.

---

### Recommendation

Add a `service_started()` guard at the top of `send_transaction` in `rpc/src/module/pool.rs`, returning a clear "pool not ready" error if the check fails:

```rust
fn send_transaction(&self, tx: Transaction, outputs_validator: Option<OutputsValidator>) -> Result<H256> {
    let tx_pool = self.shared.tx_pool_controller();
    if !tx_pool.service_started() {
        return Err(RPCError::custom(RPCError::Invalid, "tx-pool service is not ready"));
    }
    // ... existing logic
}
```

Apply the same guard in `TransactionsProcess::execute` in `sync/src/relayer/transactions_process.rs` before dispatching to `submit_remote_tx`.

---

### Proof of Concept

1. Start a CKB node that has persisted tx-pool data on disk.
2. Immediately upon the RPC port becoming reachable (before `tx_pool_ready` returns `true`), call `send_transaction` with a transaction that spends an output from a persisted pending transaction.
3. Observe the transaction is rejected (dead-cell or unknown-input error) even though it would be valid once the pool finishes loading.
4. Confirm via `tx_pool_ready` that the pool was not yet initialized at the time of submission.

The relay variant: connect a peer immediately after the node starts and send a `RelayTransactions` message containing a transaction dependent on a persisted pool entry; observe the same incorrect rejection.

```rust
    fn send_transaction(
        &self,
        tx: Transaction,
        outputs_validator: Option<OutputsValidator>,
    ) -> Result<H256> {
        let tx: packed::Transaction = tx.into();
        let tx: core::TransactionView = tx.into_view();

        self.check_output_validator(outputs_validator, &tx)?;

        let tx_pool = self.shared.tx_pool_controller();
        let submit_tx = tx_pool.submit_local_tx(tx.clone());

        if let Err(e) = submit_tx {
            error!("Send submit_tx request error {}", e);
            return Err(RPCError::ckb_internal_error(e));
        }

        let tx_hash = tx.hash();
        match submit_tx.unwrap() {
            Ok(_) => Ok(tx_hash.into()),
            Err(reject) => Err(RPCError::from_submit_transaction_reject(&reject)),
        }
    }
```
