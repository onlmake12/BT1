Audit Report

## Title
Unbounded DB Read Amplification via `GetLastStateProof` with Maximum Difficulties Array — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The `GetLastStateProof` handler permits exactly 1000 `difficulties` entries when `last_n_blocks=0` due to a strict `>` comparison against `GET_LAST_STATE_PROOF_LIMIT` (1000). Each entry triggers an O(log H) binary search over the full block range with one `get_block_total_difficulty` DB read per step. With no per-peer rate limiter on `LightClientProtocol` and no ban issued on successful responses, an attacker can sustain a tight loop of requests each causing ~24,000 DB reads, degrading or halting the node.

## Finding Description

**Limit check off-by-one:** In `execute()`, the guard is:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // 1000
```

With `difficulties.len() = 1000` and `last_n_blocks = 0`, the expression evaluates to `1000 > 1000` → `false`, so the check does not fire and all 1000 entries proceed. [1](#0-0) [2](#0-1) 

**Binary search cost per difficulty:** `get_first_block_total_difficulty_is_not_less_than` binary-searches `[start_block_number, end_block_number)` issuing one `get_block_total_difficulty` DB read per iteration — O(log H) reads per call. [3](#0-2) 

**Search range stays O(H):** `get_block_numbers_via_difficulties` is called with `start_block_number` and `difficulty_boundary_block_number`. The `start_block_number` narrowing (`num - 1`) only shrinks the range by 1 per iteration; for 1000 difficulties targeting blocks 1–1000 on a chain of height H ≈ 12 M, each binary search still spans ≈ H blocks → O(log H) reads each → ~24,000 total DB reads per request. [4](#0-3) [5](#0-4) 

**No ban on success:** `should_ban()` only returns `Some` for 4xx status codes. A valid request returns `StatusCode::OK` (200), which does not trigger a ban. [6](#0-5) 

**No rate limiting:** `LightClientProtocol` holds only `shared: Shared` — no `governor::RateLimiter` field. `received()` calls `try_process()` directly on every message with no throttle, unlike `HolePunching` which gates every message through a keyed rate limiter before processing. [7](#0-6) [8](#0-7) [9](#0-8) 

## Impact Explanation

A single unprivileged peer can sustain a tight loop of `GetLastStateProof` messages, each causing ~24,000 synchronous RocksDB reads. This saturates the node's DB I/O, starves block-processing and other protocol threads of DB access, and can degrade or halt the targeted CKB node. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation

The attack requires only a valid P2P connection to a node with the light client server enabled. Total-difficulty values for any block are publicly readable from the chain. No PoW, no special privileges, no leaked keys are needed. The absence of any per-peer rate limit on this handler makes sustained exploitation straightforward and repeatable.

## Recommendation

1. **Add a per-peer rate limiter** to `LightClientProtocol::received` analogous to `HolePunching`'s `governor::RateLimiter<(PeerIndex, u32)>`, gating each message type per peer before calling `try_process`.
2. **Fix the off-by-one** in the limit check: change `>` to `>=` so that the maximum permitted count is `GET_LAST_STATE_PROOF_LIMIT - 1`, or reduce the constant to account for the O(log H) multiplier.
3. **Cap total DB reads per request**: track the cumulative number of `get_block_total_difficulty` calls across all difficulties in a single message and abort early if a per-request budget is exceeded.

## Proof of Concept

1. Connect to a CKB node with the light client server enabled; chain height H ≈ 12 M.
2. Query the chain to obtain `total_difficulty(i)` for blocks `i = 1 … 1001` (public data).
3. Craft and send:
   ```
   GetLastStateProof {
     last_hash:           <current tip hash>,
     start_hash:          <genesis hash>,
     start_number:        0,
     last_n_blocks:       0,
     difficulty_boundary: total_difficulty(tip),
     difficulties:        [total_difficulty(1), total_difficulty(2), …, total_difficulty(1000)]
   }
   ```
4. Observe: limit check passes (`1000 + 0 > 1000` → false); server performs 1000 binary searches each spanning `[k-1, H)`; server returns `SendLastStateProof` (200 OK); peer is not banned.
5. Repeat in a tight loop. Each iteration issues ≈ 24,000 DB reads with no throttle, progressively degrading node responsiveness.

        &mut self,

        peer: PeerIndex,

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
