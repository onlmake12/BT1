Audit Report

## Title
Unbounded `tx_hashes` Array in `get_tx_indices` Enables RPC DoS - (`rpc/src/module/chain.rs`)

## Summary
`get_tx_indices` in `rpc/src/module/chain.rs` checks only for an empty input before iterating over every supplied hash with a synchronous RocksDB read per entry. Both `get_transaction_proof` and `get_transaction_and_witness_proof` pass their `tx_hashes: Vec<H256>` arguments directly to this helper with no upper-bound validation, while the equivalent light-client P2P handler enforces a hard cap of 1,000 hashes. Repeated requests with large valid hash arrays can saturate the RPC thread pool and RocksDB read path, degrading or halting RPC service for all callers.

## Finding Description
`get_tx_indices` at `rpc/src/module/chain.rs:2286–2316` performs only an emptiness check, then unconditionally iterates over every hash calling `snapshot.get_transaction_info` (a synchronous RocksDB read) for each one: [1](#0-0) 

Both public entry points pass `tx_hashes: Vec<H256>` directly to this helper with no pre-validation: [2](#0-1) [3](#0-2) 

After `get_tx_indices` returns, `get_transaction_proof` additionally calls `CBMT::build_merkle_proof` over all block transactions: [4](#0-3) 

The `verify_transaction_proof` and `verify_transaction_and_witness_proof` methods also accept unbounded `proof.indices` and `proof.lemmas` arrays fed directly into `CBMT::retrieve_leaves` with no size guard: [5](#0-4) 

By contrast, the light-client P2P handler enforces a hard cap of 1,000: [6](#0-5) 

defined in: [7](#0-6) 

The only existing guard is the 10 MiB request body limit: [8](#0-7) 

The block size consensus limit (`MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT = 597 * 1000 ≈ 597 KB`) naturally caps the number of valid hashes per block at roughly 1,000, which is consistent with the light-client limit. However, this does not prevent the attack: an attacker can repeatedly submit requests with all valid hashes from a large block, forcing proportional RocksDB reads and Merkle proof construction on every request. [9](#0-8) 

## Impact Explanation
This matches **Note (0–500 points): Any local RPC API crash**. Flooding the endpoint with requests containing the maximum valid hash count forces proportional synchronous RocksDB reads in `get_tx_indices` plus `CBMT::build_merkle_proof` over all block transactions per request, exhausting the RPC server's capacity to serve legitimate callers. The node's consensus and P2P layers are unaffected; only the RPC service is impacted.

## Likelihood Explanation
The attack requires only: (1) access to the RPC port (default `127.0.0.1:8114`, but commonly exposed on internal networks or via reverse proxy), and (2) valid transaction hashes from a single block (freely available from any block explorer or via `get_block`). No authentication is required. The attack is trivially automatable and costs only network bandwidth.

## Recommendation
Add an upper-bound check at the start of `get_tx_indices`, consistent with the light-client protocol limit:

```rust
const MAX_TX_HASHES_PER_PROOF_REQUEST: usize = 1000;

fn get_tx_indices(&self, tx_hashes: Vec<H256>, ...) -> Result<...> {
    if tx_hashes.is_empty() {
        return Err(RPCError::invalid_params("Empty transaction hashes"));
    }
    if tx_hashes.len() > MAX_TX_HASHES_PER_PROOF_REQUEST {
        return Err(RPCError::invalid_params(format!(
            "Too many transaction hashes, limit is {MAX_TX_HASHES_PER_PROOF_REQUEST}"
        )));
    }
    ...
}
```

Apply the same limit to `proof.indices` and `proof.lemmas` in `verify_transaction_proof` and `verify_transaction_and_witness_proof` before constructing `MerkleProof` and calling `CBMT::retrieve_leaves`.

## Proof of Concept
1. Query `get_block` on a block with many transactions to collect all tx hashes.
2. Send the following in a tight loop:
```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_transaction_proof",
  "params": [["0x<tx_hash_1>", ..., "0x<tx_hash_N>"], null]
}
```
3. Each request forces N synchronous RocksDB reads in `get_tx_indices` plus `CBMT::build_merkle_proof` over all block transactions. Flooding the endpoint with such requests exhausts the RPC server's capacity to serve legitimate callers.
