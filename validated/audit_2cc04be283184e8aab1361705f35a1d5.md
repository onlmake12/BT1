### Title
Unbounded Per-Request Full-Block Deserialization and MMR Proof Generation in `GetTransactionsProofProcess::execute` Enables DoS — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

---

### Summary

Any unprivileged remote peer connected to the light-client sub-protocol can send a single `GetTransactionsProof` message containing up to 1,000 transaction hashes, each belonging to a distinct confirmed block. The server performs a full block deserialization, CBMT proof build, witnesses-root computation, uncles load, and extension load for every unique block, then calls `mmr.gen_proof(1000 positions)` — O(1000 × log N) MMR node reads — all within one synchronous request handler. There is no per-peer rate limiter anywhere in the light-client protocol server.

---

### Finding Description

**Constant (no rate limit):**
`GET_TRANSACTIONS_PROOF_LIMIT = 1000` is the only guard. [1](#0-0) 

The `LightClientProtocol::received` handler calls `try_process` directly with no rate-limiting wrapper. A grep across the entire `util/light-client-protocol-server/` tree returns zero hits for `RateLimiter`, `rate_limit`, `throttle`, `per_second`, or `quota`. [2](#0-1) 

**Per-block work inside the loop (lines 82–126):**

For each of up to 1,000 distinct blocks the handler calls:

1. `snapshot.get_block(&block_hash)` — full block deserialization from RocksDB (all transactions + witnesses). [3](#0-2) 

2. `CBMT::build_merkle_proof(block.transactions().iter()...)` — iterates every transaction in the block to build the proof. [4](#0-3) 

3. `block.calc_witnesses_root()` — hashes every witness in the block. [5](#0-4) 

4. `snapshot.get_block_uncles(&block_hash)` and `snapshot.get_block_extension(&block_hash)` — two additional DB reads per block. [6](#0-5) 

**Contrast with `GetBlocksProof`:** that handler calls `get_block_header` (header only, ~200 bytes), not `get_block` (full block, potentially megabytes). [7](#0-6) 

**MMR proof generation:**

After the loop, `reply_proof` calls `mmr.gen_proof(items_positions)` with up to 1,000 leaf positions on a chain of N blocks, requiring O(1,000 × log N) MMR node reads from the store. [8](#0-7) 

---

### Impact Explanation

A single request forces the node to:
- Deserialize up to 1,000 full blocks from RocksDB (each potentially several MB).
- Compute `calc_witnesses_root()` for each (CPU-bound hash iteration over all witnesses).
- Execute `mmr.gen_proof(1000 positions)` — O(1,000 × log N) additional DB reads, growing with chain length.

With no rate limiting, an attacker can pipeline many such requests from one or more peers, saturating both I/O bandwidth (RocksDB reads) and CPU (hashing). This can starve the node's async runtime of threads, delaying or blocking block propagation, sync, and RPC responses — a practical availability impact.

---

### Likelihood Explanation

- The attacker needs only valid, confirmed tx hashes — all publicly observable on-chain.
- No PoW, no key, no privileged role required.
- The light-client protocol is exposed to any peer that connects and negotiates the `LightClient` sub-protocol.
- The attack is repeatable and pipelinable.

---

### Recommendation

1. **Add a per-peer rate limiter** to `LightClientProtocol`, mirroring the `RateLimiter<(PeerIndex, u32)>` pattern used in the `Relayer`. [9](#0-8) 

2. **Cap the number of distinct blocks** per `GetTransactionsProof` request (e.g., 10–50), separate from the tx-hash count limit. The current limit of 1,000 tx hashes can map to 1,000 distinct blocks. [10](#0-9) 

3. **Consider loading only block headers** for the filtered-block response where possible, deferring full-block loads.

---

### Proof of Concept

```
1. Mine a chain with 1,000+ blocks, each containing one unique transaction.
2. Collect one tx_hash per block (1,000 hashes total) from the public chain state.
3. Connect to the target node as a light-client peer.
4. Send a single GetTransactionsProof message:
     tx_hashes = [tx_1, tx_2, ..., tx_1000]   // one per distinct block
     last_hash  = <current tip hash>
5. Observe: the node performs 1,000 × get_block() + calc_witnesses_root()
   + mmr.gen_proof(1000 positions).
6. Repeat in a tight loop (no rate limit enforced).
7. Measure: RocksDB read amplification and CPU utilization spike;
   block-relay and sync latency increase proportionally.
```
