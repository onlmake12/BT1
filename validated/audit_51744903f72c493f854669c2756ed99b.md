Audit Report

## Title
Stale Pre-Eviction Stat Snapshot Overwrites Correct Post-Eviction Totals in `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` is called before `check_and_record_ancestors` and captures `self.total_tx_size + entry.size` and `self.total_tx_cycles + entry.cycles` into local variables. When `check_and_record_ancestors` evicts entries via `remove_entry_and_descendants` â†’ `update_stat_for_remove_tx`, those decrements are applied directly to `self.total_tx_size`/`self.total_tx_cycles` in-place. The stale local snapshot is then unconditionally written back at lines 218â€“219, permanently re-inflating both counters by the aggregate size and cycles of all evicted entries. Repeated exploitation drives `total_tx_size` to the pool limit without the pool actually being full, causing all subsequent valid transactions to be rejected with `Reject::Full`.

## Finding Description

`updated_stat_for_add_tx` is a `&self` (non-mutating) method that returns `(self.total_tx_size + tx_size, self.total_tx_cycles + cycles)` without modifying `self`. [1](#0-0) 

In `add_entry`, this snapshot is captured at lines 210â€“211, before `check_and_record_ancestors` is called at line 213. [2](#0-1) 

`check_and_record_ancestors` enters the eviction branch at line 603 when `ancestors_count > max_ancestors_count` but the excess is covered by `cell_ref_parents`. It calls `remove_entry_and_descendants` for each candidate, which internally calls `update_stat_for_remove_tx`, directly decrementing `self.total_tx_size` and `self.total_tx_cycles` in-place. [3](#0-2) [4](#0-3) 

After `check_and_record_ancestors` returns, lines 218â€“219 unconditionally write the stale local variables back to `self`, erasing all correct in-place decrements performed during eviction. [5](#0-4) 

Concretely, if before `add_entry` `self.total_tx_size = X`, the snapshot captures `X + entry.size`. If eviction removes entries totaling `E` bytes, `update_stat_for_remove_tx` correctly sets `self.total_tx_size = X - E`. Line 218 then overwrites this with `X + entry.size`, inflating the counter by `E` bytes relative to the correct value of `X - E + entry.size`.

## Impact Explanation

`total_tx_size` is the primary guard for the `max_tx_pool_size` limit, checked via `updated_stat_for_add_tx` which returns `Reject::Full` when the limit is reached. Each eviction event permanently inflates `total_tx_size` by the size of evicted entries. An attacker who repeatedly triggers the eviction path compounds this inflation until `total_tx_size >= max_tx_pool_size`, at which point all subsequent `send_transaction` RPC calls are rejected with `Reject::Full` even though the pool holds far fewer bytes than the limit. `total_tx_cycles` is similarly inflated and exposed via the `tx_pool_info` RPC. Applied across multiple nodes, this constitutes **CKB network congestion with low cost**, matching the **High** impact class (10001â€“15000 points).

## Likelihood Explanation

The eviction path is reachable by any unprivileged user with RPC access. The attacker must construct a chain of â‰Ą125 transactions where each uses the prior transaction's output as a cell dep (not an input), making them `cell_ref_parents`. The condition at line 603 â€” `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` with `ancestors_count > max_ancestors_count` â€” is satisfiable with exactly this construction. The attack is repeatable and compounds: each iteration inflates the counters further. No special privileges, leaked keys, or victim mistakes are required.

## Recommendation

Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so the snapshot reflects the post-eviction state of `self.total_tx_size`/`self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals only after all evictions are done
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, eliminate the local-variable pattern entirely and apply the increment directly to `self.total_tx_size`/`self.total_tx_cycles` after all mutations complete, removing the possibility of a stale snapshot.

## Proof of Concept

1. Submit a chain of 125 transactions `T1 â†’ T2 â†’ â€¦ â†’ T125` where each `Ti` uses `T(i-1)`'s output as a **cell dep** (not an input), making all 125 appear as `cell_ref_parents` of a subsequent transaction.
2. Submit `T_new` that also references `T1`'s output as a cell dep. Its ancestor count is 126, exceeding `max_ancestors_count = 125`.
3. `check_and_record_ancestors` enters the eviction branch at line 603 (since `126 - 125 = 1 <= 125`), evicts `T125` (and its descendants) via `remove_entry_and_descendants`, calling `update_stat_for_remove_tx(T125.size, T125.cycles)`, which correctly decrements `self.total_tx_size`.
4. `add_entry` then writes back the stale `total_tx_size` (computed before eviction at line 211), re-inflating it by `T125.size`.
5. Repeat steps 1â€“4. `total_tx_size` grows by `T125.size` per iteration without bound relative to actual pool contents.
6. Eventually `total_tx_size >= max_tx_pool_size` and all subsequent `send_transaction` RPC calls return `Reject::Full`.
