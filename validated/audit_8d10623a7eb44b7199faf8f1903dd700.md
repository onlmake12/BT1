### Title
Peer Store `check_purge()` Network-Group Inflation Allows Targeted Eviction of Legitimate Peer Addresses — (`network/src/peer_store/peer_store_impl.rs`)

---

### Summary

The `check_purge()` function in `PeerStore` uses a network-group eviction strategy that is vulnerable to group inflation. An attacker who can advertise addresses via the Discovery protocol can flood addresses in the same /16 network as known legitimate peers, inflating that group to become the largest, and causing 2 randomly chosen addresses from the inflated group (which includes legitimate peers) to be evicted per purge cycle. Over repeated cycles, legitimate peer addresses are systematically displaced by attacker-controlled ones.

---

### Finding Description

`check_purge()` implements a two-phase eviction strategy:

**Phase 1** — evict non-connectable addresses: [1](#0-0) 

**Phase 2** — only if Phase 1 evicts nothing, group by network segment and randomly evict from the largest groups: [2](#0-1) 

The network group for IPv4 is computed as the first two octets (a /16 prefix): [3](#0-2) 

The eviction logic:
1. Sorts all groups by descending size
2. Takes the top `len / 2` groups
3. For any group with `> 4` peers, randomly selects 2 addresses to remove [4](#0-3) 

There is no protection against an attacker deliberately placing addresses into the same /16 group as legitimate peers. If an attacker floods addresses in `10.0.x.x` (same /16 as legitimate peers also in `10.0.y.y`), the combined group becomes the largest. Phase 2 then randomly evicts 2 addresses from that combined pool — with probability proportional to the ratio of legitimate addresses in the group.

The `ADDR_COUNT_LIMIT` is 16384: [5](#0-4) 

`add_addr()` calls `check_purge()` on every insertion when the store is full: [6](#0-5) 

---

### Impact Explanation

- Legitimate peer addresses are evicted from the peer store and replaced by attacker-controlled addresses.
- The node preferentially dials attacker-controlled peers for outbound connections, degrading peer diversity.
- After legitimate peers disconnect, the node cannot reconnect to them (their addresses are gone), increasing eclipse attack surface.
- Currently connected peers are not immediately affected, but the long-term outbound peer pool is corrupted.

---

### Likelihood Explanation

- The Discovery protocol allows any connected peer to advertise arbitrary addresses with no per-peer rate limit enforced inside `add_addr()`.
- The attacker only needs one inbound or outbound connection to the victim to begin flooding addresses.
- The /16 networks of well-known CKB peers are publicly observable.
- Filling 16384 slots is achievable over time; each `add_addr()` call that triggers Phase 2 evicts 2 legitimate addresses while the attacker can re-add their own.
- All freshly advertised addresses pass `is_connectable()` (they have `last_connected_at_ms = 0` and zero failed attempts), so Phase 1 never fires and Phase 2 is always reached when the store is full.

---

### Recommendation

1. **Cap per-/16 group size before eviction**: Reject or deprioritize new addresses from a /16 group that already holds more than a threshold (e.g., 256) entries, preventing group inflation.
2. **Protect recently-connected addresses from random eviction**: Addresses with a recent `last_connected_at_ms` should be shielded from Phase 2 random selection, similar to Bitcoin Core's eviction protection tiers.
3. **Evict only from the attacker's group, not the victim's**: When a new address arrives from group G and triggers eviction, prefer to evict from group G itself rather than from the globally largest group.

### Proof of Concept

```rust
// Fill peer store: 8000 legitimate peers in 10.0.0.x, 8000 attacker peers in 10.0.1.x
// Both map to Group::IP4([10, 0]) — same /16
let mut peer_store = PeerStore::default();
let now = ckb_systemtime::unix_time_as_millis();

// Add 8000 legitimate peers (10.0.0.x)
for i in 0u32..8000 {
    let addr: Multiaddr = format!("/ip4/10.0.0.{}/tcp/8114/p2p/{}", i % 256, PeerId::random().to_base58()).parse().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Add 8000 attacker peers (10.0.1.x) — same /16 group [10, 0]
for i in 0u32..8000 {
    let addr: Multiaddr = format!("/ip4/10.0.1.{}/tcp/8114/p2p/{}", i % 256, PeerId::random().to_base58()).parse().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Add 384 peers in a different /16 to ensure len >= 2 so take(len/2) >= 1
for i in 0u32..384 {
    let addr: Multiaddr = format!("/ip4/192.168.0.{}/tcp/8114/p2p/{}", i % 256, PeerId::random().to_base58()).parse().unwrap();
    peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
}
// Store is now at 16384. Next add_addr triggers check_purge Phase 2.
// Phase 2: group [10,0] has 16000 entries (largest), group [192,168] has 384.
// len=2, take(1) => only [10,0] group considered.
// 2 random addresses evicted from [10,0] — 50% chance each is a legitimate peer.
// Assert: after N eviction cycles, legitimate peer count in [10,0] decreases.
```
