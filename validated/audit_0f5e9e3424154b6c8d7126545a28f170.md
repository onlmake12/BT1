### Title
Out-of-Bounds Index Panic in `TxConfirmStat::remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

---

### Summary

`remove_unconfirmed_tx` guards the `block_unconfirmed_txs` array access with a `tx_age >= len()` check, but applies **no equivalent guard** before indexing `confirm_blocks_to_failed_txs[tx_age - 1]`. When a tracked transaction remains in the pool for more than `MAX_CONFIRM_BLOCKS` (1000) blocks and is then evicted with `count_failure = true`, `tx_age - 1 >= 1000` causes a Rust index-out-of-bounds **panic**, crashing the tx-pool service and the entire node.

---

### Finding Description

`confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` are both allocated with length `MAX_CONFIRM_BLOCKS = 1000`: [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 protects only the `block_unconfirmed_txs` write: [2](#0-1) 

When `tx_age >= 1001`, the `if tx_age >= self.block_unconfirmed_txs.len()` branch correctly routes to `old_unconfirmed_txs`, but execution falls through to line 215 unconditionally:

```rust
self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
```

`tx_age - 1 >= 1000` is out of bounds for a 1000-element `Vec`, causing a **panic**.

The full call chain that reaches this with `count_failure = true`:

- `fee_estimator.reject_tx(&tx_hash)` — registered as the reject callback in `shared_builder.rs` [3](#0-2) 
- → `Algorithm::reject_tx` → `drop_tx` (hardcodes `count_failure = true`) [4](#0-3) 
- → `drop_tx_inner` → `remove_unconfirmed_tx(tx_record.height, self.best_height, ..., true)` [5](#0-4) 

The reject callback is fired from both `remove_expired` and `limit_size`: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

A Rust `Vec` index-out-of-bounds panics unconditionally in release builds (no `unsafe`, no UB — just a guaranteed `panic!`). The tx-pool service runs as a Tokio task; an unhandled panic in that task terminates the process. The entire CKB node crashes.

---

### Likelihood Explanation

The default `expiry_hours = 12`: [8](#0-7) 

At ~10 s/block, 1001 blocks ≈ 2.78 hours — well within the 12-hour expiry window. A tx submitted at height H and never mined will still be tracked in `tracked_txs` at height H + 1001 when it is eventually evicted. No special privileges are required: any unprivileged peer can submit a valid, minimum-fee-rate transaction via the standard P2P relay path. The attacker does not need to control hashpower or perform a Sybil attack. They simply submit one valid tx and wait; natural chain growth does the rest. Pool size eviction (`limit_size`) can accelerate the trigger to any time after 1001 blocks by flooding the pool with higher-fee-rate transactions.

---

### Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` write, mirroring the existing guard:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This silently drops the failure sample for txs older than `MAX_CONFIRM_BLOCKS`, which is consistent with how `block_unconfirmed_txs` already handles the same case (routing to `old_unconfirmed_txs` and discarding the per-block slot).

---

### Proof of Concept

```rust
#[test]
fn test_remove_unconfirmed_tx_oob_panic() {
    let mut stat = TxConfirmStat::default(); // MAX_CONFIRM_BLOCKS = 1000

    // Simulate: tx tracked at height 0
    let bucket_index = stat.add_unconfirmed_tx(0, FeeRate::from_u64(1000)).unwrap();

    // Advance the window 1001 times so best_height = 1001
    for h in 1..=1001u64 {
        stat.move_track_window(h);
    }

    // tx_age = 1001 - 0 = 1001; tx_age - 1 = 1000 → OOB on confirm_blocks_to_failed_txs (len=1000)
    // This panics:
    stat.remove_unconfirmed_tx(0, 1001, bucket_index, true);
}
```
