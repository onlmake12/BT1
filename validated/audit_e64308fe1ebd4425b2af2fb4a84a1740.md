### Title
Single Global `validTimePeriodSeconds` Applied to All Assets Regardless of Feed Update Frequency — (`File: target_chains/ethereum/contracts/contracts/aave/PythPriceOracleGetter.sol`)

---

### Summary

`PythPriceOracleGetter` uses one contract-wide `validTimePeriodSeconds` for every asset's staleness check. Because Pyth price feeds have fundamentally different update cadences (crypto feeds update sub-second; equity feeds only update during market hours; FX feeds follow forex sessions), a single threshold cannot be correct for all assets simultaneously. When the threshold is widened to accommodate slow-updating feeds, stale crypto prices pass the freshness check and are returned to Aave as valid collateral valuations.

---

### Finding Description

`PythAssetRegistry` stores a single `validTimePeriodSeconds` field that applies to every registered asset:

```
// PythAssetRegistry.sol line 18
uint validTimePeriodSeconds;
```

`PythPriceOracleGetter.getAssetPrice()` calls `getPriceNoOlderThan` with this single value for every asset, regardless of that asset's actual feed update frequency:

```solidity
// PythPriceOracleGetter.sol lines 63-66
PythStructs.Price memory price = pyth().getPriceNoOlderThan(
    priceId,
    PythAssetRegistry.validTimePeriodSeconds()   // same value for ALL assets
);
```

The constructor accepts one `validTimePeriodSeconds` and stores it globally:

```solidity
// PythPriceOracleGetter.sol line 49
PythAssetRegistry.setValidTimePeriodSeconds(validTimePeriodSeconds);
```

There is no per-asset override mechanism anywhere in `PythAssetRegistry`.

Pyth feeds have heterogeneous update cadences:
- **Crypto** (BTC/USD, ETH/USD): updated every few seconds around the clock.
- **Equities** (AAPL/USD, TSLA/USD): updated only during NYSE/NASDAQ trading hours; the `publishTime` of an equity feed on a Saturday is from Friday's close — roughly 60+ hours in the past.
- **FX / Commodities**: follow their own session schedules.

A deployer who registers both crypto and equity assets in the same `PythPriceOracleGetter` instance faces an impossible choice:

| Threshold | Crypto feeds | Equity feeds (off-hours) |
|---|---|---|
| Tight (e.g. 60 s) | Correct | Always revert → DoS |
| Loose (e.g. 86 400 s) | Stale prices accepted | Correct |

The only operationally viable choice for a mixed-asset registry is a loose threshold, which directly enables stale crypto price consumption.

---

### Impact Explanation

When `validTimePeriodSeconds` is set loosely (e.g. 24 hours) to keep equity feeds functional:

1. A crypto asset's on-chain price may be many hours old (e.g. due to low keeper activity or a temporary network disruption).
2. `getAssetPrice()` returns this stale price without reverting because `age < validTimePeriodSeconds` still holds.
3. Aave uses the returned value for collateral valuation and liquidation threshold calculations.
4. An attacker can exploit the discrepancy between the stale on-chain price and the true market price to:
   - Borrow against overvalued collateral (if the stale price is higher than current).
   - Avoid a liquidation that should have been triggered (if the stale price keeps the health factor above 1).
   - Liquidate a position that should not yet be liquidatable (if the stale price is lower than current).

This results in direct financial loss to Aave liquidity providers and/or borrowers.

---

### Likelihood Explanation

- Pyth's pull model means on-chain prices are only as fresh as the last submitted update. During periods of low keeper activity, network congestion, or deliberate griefing (withholding updates), crypto prices can lag by hours while remaining within a loose threshold.
- Any unprivileged transaction sender can call `getAssetPrice()` and trigger downstream Aave logic (borrow, repay, liquidate) using the stale value — no special role is required.
- The structural impossibility of choosing a single correct threshold for a mixed-asset registry makes this a near-certain misconfiguration in any real deployment that includes both crypto and non-crypto assets.

---

### Recommendation

Replace the single `validTimePeriodSeconds` in `PythAssetRegistry` with a per-asset mapping:

```solidity
// PythAssetRegistry.sol — proposed change
mapping(address => uint) assetValidTimePeriodSeconds;
```

Extend `setAssetsSources` (or add a new setter) to accept per-asset thresholds alongside `priceIds`, and update `getAssetPrice` to look up the per-asset value:

```solidity
PythStructs.Price memory price = pyth().getPriceNoOlderThan(
    priceId,
    _registryState.assetValidTimePeriodSeconds[asset]
);
```

This mirrors the fix recommended for the analogous `ChainlinkPriceOracle` issue and allows crypto assets to use a tight threshold (e.g. 60 s) while equity assets use a threshold appropriate for their session schedule (e.g. 90 000 s to cover a weekend).

---

### Proof of Concept

**Setup:** Deploy `PythPriceOracleGetter` with two assets:
- `WBTC` → Pyth BTC/USD feed (crypto, updates every few seconds)
- `AAPL` → Pyth AAPL/USD feed (equity, updates only during market hours)

**Required `validTimePeriodSeconds`:** Must be ≥ 259 200 s (72 hours) to survive a weekend without reverting on `AAPL`.

**Attack (Saturday morning, BTC price has dropped 15% since Friday):**

1. The on-chain BTC/USD price was last updated Friday at market close — 18 hours ago. It is still within the 72-hour threshold.
2. Attacker calls `getAssetPrice(WBTC)` → receives Friday's (higher) BTC price.
3. Attacker opens a borrow position on Aave using WBTC as collateral, borrowing the maximum amount permitted by the stale (inflated) collateral value.
4. The true BTC price is 15% lower; the position is immediately undercollateralized at true market prices.
5. Aave LPs bear the shortfall when the position is eventually liquidated at a loss.

The root cause is entirely within `PythPriceOracleGetter.sol` (line 63–66) and `PythAssetRegistry.sol` (line 18, 72–74): no external oracle manipulation is required. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
