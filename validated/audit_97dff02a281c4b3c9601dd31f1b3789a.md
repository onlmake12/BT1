The bug in `FullSuspendedState::size()` is real and concrete. Here is the analysis:

### Title
`FullSuspendedState::size()` Undercounts `inherited_fd` State Size, Enabling Below-Cost Large-State Suspension — (`script/src/types.rs`)

### Summary

`FullSuspendedState::size()` miscalculates the contribution of `inherited_fd` to the total state size. The field is `Vec<(VmId, Vec<Fd>)>` — a vector of per-VM fd lists — but the size formula treats each outer entry as if it contains exactly one `Fd`, ignoring the inner `Vec<Fd>` lengths entirely. An unprivileged script author can spawn many child VMs each inheriting many file descriptors, causing the cycle charge for suspend/resume to be far below the actual memory cost of the state.

### Finding Description

In `script/src/types.rs`, `FullSuspendedState` declares:

```rust
pub inherited_fd: Vec<(VmId, Vec<Fd>)>,
``` [1](#0-0) 

The `size()` method at line 546 computes:

```rust
+ (self.inherited_fd.len() * (size_of::<Fd>())) as u64
``` [2](#0-1) 

This multiplies the **number of outer entries** (i.e., number of VMs that have inherited fds) by `size_of::<Fd>()`. It does **not** iterate into the inner `Vec<Fd>` to sum the actual fd counts per VM. It also omits the `VmId` component of each tuple.

The correct formula should be:
```rust
self.inherited_fd.iter().fold(0u64, |acc, (_, fds)| {
    acc + size_of::<VmId>() as u64 + (fds.len() * size_of::<Fd>()) as u64
})
```

With 16 VMs each inheriting 64 fds:
- **Reported size**: `16 * size_of::<Fd>()` = 16 × 8 = 128 bytes
- **Actual size**: `16 * 64 * size_of::<Fd>()` = 8192 bytes (plus VmId overhead)
- **Undercount factor**: 64× [3](#0-2) 

### Impact Explanation

`size()` is documented as the basis for cycles charged during suspend/resume operations. An attacker who crafts a script (ScriptVersion::V2) that spawns many child VMs each with large `inherited_fd` lists can accumulate a suspended state that is orders of magnitude larger than what the cycle meter accounts for. This is a resource exhaustion / economic attack: the attacker pays cycle costs proportional to VM count, not total fd count, while the node must store and serialize the full state.

### Likelihood Explanation

The spawn syscall with `inherited_fd` is a supported production feature in ScriptVersion::V2. No privileged access is required — any transaction submitted to the network can exercise this path. The parameters (16 VMs × 64 fds) are well within plausible script behavior and do not require any special conditions beyond ScriptVersion::V2 activation.

### Recommendation

Replace the flat `inherited_fd.len() * size_of::<Fd>()` term with a fold that sums the inner `Vec<Fd>` lengths per VM entry, and also accounts for the `VmId` per entry:

```rust
+ self.inherited_fd.iter().fold(0u64, |acc, (_, fds)| {
    acc + size_of::<VmId>() as u64 + (fds.len() * size_of::<Fd>()) as u64
})
``` [2](#0-1) 

### Proof of Concept

Construct a `FullSuspendedState` with `inherited_fd` containing 16 entries each holding 64 `Fd` values. Call `size()`. The returned value will be `16 * size_of::<Fd>()` (128 bytes on a 64-bit system), while the actual serialized contribution is `16 * 64 * size_of::<Fd>()` = 8192 bytes — a 64× discrepancy, confirming cycles charged are far below actual state cost.
