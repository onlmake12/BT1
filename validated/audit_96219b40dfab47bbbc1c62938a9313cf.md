### Title
`PythAggregatorV3.latestRoundData()` Returns Stale Price Without Validity Checks — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink-compatible adapter contract, designed as a drop-in replacement for `AggregatorV3Interface`. Its `latestRoundData()` (and related functions) unconditionally call `getPriceUnsafe()`, which performs **no staleness check** and **no price validity check**. Additionally, the returned `answeredInRound` is always set equal to `roundId`, which means the standard Chainlink staleness guard (`require(answeredInRound >= roundId)`) that downstream consumers apply will **always pass silently**, even when the price is arbitrarily stale.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3Interface` on top of Pyth price feeds. Every price-reading function in the contract delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestRoundData — line 110
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
roundId = uint80(price.publishTime);
return (
    roundId,
    int256(price.price),
    price.publishTime,
    price.publishTime,
    roundId          // answeredInRound == roundId always
);
``` [1](#0-0) 

`getPriceUnsafe()` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee: [2](#0-1) 

Three compounding problems exist:

1. **No staleness check**: `getPriceUnsafe()` is used instead of `getPriceNoOlderThan()`. If Pyth's on-chain price has not been updated (due to network congestion, Hermes downtime, or a Wormhole VAA delivery failure), the returned price can be hours or days old.

2. **`answeredInRound` is always equal to `roundId`**: Both are set to `uint80(price.publishTime)`. The canonical Chainlink staleness guard used by virtually every Chainlink-integrated protocol is `require(answeredInRound >= roundId, "Stale price")`. Because `answeredInRound == roundId` unconditionally, this check **always passes**, giving downstream consumers a false sense of freshness.

3. **No `price > 0` check**: `price.price` is `int64` in Pyth's data model. The adapter casts it directly to `int256` without asserting positivity. A zero or negative price value would propagate silently.

The same pattern applies to `latestAnswer()` and `getRoundData()`: [3](#0-2) [4](#0-3) 

---

### Impact Explanation

`PythAggregatorV3` is Pyth's officially published migration path for protocols moving from Chainlink. The [Chainlink migration guide](https://docs.pyth.network/price-feeds/core/migrate-an-app-to-pyth/chainlink) instructs integrators to deploy this contract and point their existing app at it. Any such protocol that applies the standard Chainlink staleness pattern (`require(answeredInRound >= roundId)`) will receive no protection against stale prices.

Concrete impact on integrating DeFi protocols:
- **Lending protocols**: Stale collateral prices allow undercollateralized borrowing or block valid liquidations.
- **Derivatives/perps**: Stale mark prices allow traders to open/close positions at outdated prices, extracting value from the protocol.
- **AMMs with oracle pricing**: Stale prices enable arbitrage against the protocol at the expense of LPs.

This matches the MEDIUM severity of the reference report: impact is realized only when the Pyth price feed is in a degraded state (stale), but the window can be extended by an attacker who deliberately avoids calling `updateFeeds()`.

---

### Likelihood Explanation

- `PythAggregatorV3` is a production SDK artifact published in `@pythnetwork/pyth-sdk-solidity` and actively recommended in Pyth's official migration documentation.
- Any unprivileged caller can invoke `latestRoundData()` — no special role is required.
- Pyth's pull-oracle model means the on-chain price is only updated when someone calls `updatePriceFeeds()`. If no keeper is running, the price goes stale naturally. An attacker can exploit this window without any privileged access.
- The `answeredInRound == roundId` tautology means protocols that believe they are protected by the standard Chainlink guard are silently unprotected.

---

### Recommendation

Replace `getPriceUnsafe()` with `getPriceNoOlderThan()` using a configurable `maxAge` parameter, and add explicit validity checks:

```solidity
uint256 public maxPriceAge; // set in constructor, e.g. 60 seconds

function latestRoundData() external view returns (
    uint80 roundId, int256 answer,
    uint256 startedAt, uint256 updatedAt, uint80 answeredInRound
) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxPriceAge);
    require(price.price > 0, "PythAggregatorV3: invalid price");
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

The same fix should be applied to `latestAnswer()`, `getRoundData()`, and `decimals()`.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a valid `priceId`.
2. Do **not** call `updateFeeds()` — allow the on-chain price to go stale (or simulate by forking at a block where the price is old).
3. Call `latestRoundData()`. Observe:
   - It returns without reverting.
   - `updatedAt` reflects the old `publishTime`.
   - `answeredInRound == roundId` (both equal `uint80(price.publishTime)`).
4. Apply the standard Chainlink guard: `require(answeredInRound >= roundId, "Stale price")` — it passes silently.
5. A lending protocol using this adapter would accept the stale price for collateral valuation, enabling undercollateralized borrowing. [5](#0-4) [6](#0-5)
