### Title
Incomplete Hash-Type Validation in `ScriptHashTypeVerifier` Checks Only Lock Script, Skips Type Script — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` field of each output's **lock script** against the set of consensus-permitted hash types (`ENABLED_SCRIPT_HASH_TYPE`). It never inspects the **type script** `hash_type` field. The struct-level comment claims it "Verify[s] that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules," but the actual check is incomplete. This is a direct analog to the reported pattern: a wrong/incomplete variable is checked, and the description misleads about what is actually validated.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, the `ScriptHashTypeVerifier` is defined as:

```rust
// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }
        Ok(())
    }
}
```

Every branch exclusively reads `output.lock().hash_type()`. The `output.type_()` field — which is an `Option<Script>` and may carry its own `hash_type` — is never examined. A transaction output with a valid lock script hash type but a not-yet-activated type script hash type (e.g., `Data3 = 6`, `Data4 = 8`, …) passes this verifier without error.

The `NonContextualTransactionVerifier` doc comment is more precise ("Check whether output lock hash type within enabled range"), but the `ScriptHashTypeVerifier` struct comment says "transaction outputs" (plural, implying both scripts), creating the same misleading-description pattern as the reported vulnerability.

`check_data` (which validates that hash_type bytes are valid enum variants) is called in the sync/relayer P2P path but **not** in the tx-pool admission path — confirmed by the absence of any `check_data` call in `tx-pool/**/*.rs`. So for RPC-submitted transactions, the only non-contextual hash-type range check is `ScriptHashTypeVerifier`, and it is incomplete.

---

### Impact Explanation

A transaction sender or RPC caller can craft a transaction whose output carries a type script with a not-yet-enabled `hash_type` value (e.g., `Data3`). Such a transaction:

1. Passes `check_data` — `Data3` is a valid enum variant.
2. Passes `ScriptHashTypeVerifier` — only the lock script is inspected.
3. Enters the tx pool and the verify queue.
4. Fails at contextual script execution with `ScriptError::InvalidScriptHashType`, which is classified as a malformed transaction (`is_malformed_from_verification` returns `true` for `ErrorKind::Script`).

The consequence is that the expensive contextual verification path (script VM execution) is reached for transactions that should have been rejected cheaply at the non-contextual stage. An attacker can use this to force the node to perform unnecessary work per submitted transaction, bypassing the fast-path rejection that `ScriptHashTypeVerifier` is designed to provide. The peer is eventually banned, but only after resources have been consumed.

---

### Likelihood Explanation

Exploitation requires only the ability to submit a transaction via RPC (`send_transaction`) or the P2P relay path. No special privileges, keys, or majority hash power are needed. Crafting a transaction with a type script whose `hash_type` byte is set to an unpermitted value (e.g., `0x06` for `Data3`) is trivial. The attacker can repeat this from many IP addresses before bans accumulate.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also inspect the type script hash type for each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Check lock script hash type
        Self::check_hash_type(output.lock().hash_type())?;
        // Check type script hash type (if present)
        if let Some(type_script) = output.type_().to_opt() {
            Self::check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

Also update the `NonContextualTransactionVerifier` doc comment at line 70 to reflect that both lock and type script hash types are checked, and correct the `ScriptHashTypeVerifier` struct comment to accurately describe the full scope of the validation.

### Proof of Concept

Submit via RPC:

```json
{
  "method": "send_transaction",
  "params": [{
    "version": "0x0",
    "cell_deps": [...],
    "header_deps": [],
    "inputs": [{ "previous_output": { "tx_hash": "0x...", "index": "0x0" }, "since": "0x0" }],
    "outputs": [{
      "capacity": "0x...",
      "lock": { "code_hash": "0x...", "hash_type": "type", "args": "0x" },
      "type": {
        "code_hash": "0xdeadbeef...",
        "hash_type": "0x06",   // Data3 — not yet enabled
        "args": "0x"
      }
    }],
    "outputs_data": ["0x"],
    "witnesses": ["0x"]
  }, "passthrough"]
}
```

The node accepts the transaction into the tx pool (non-contextual verification passes), then expends script-VM resources during contextual verification before finally rejecting it as malformed.

/// Context-independent verification checks for transaction
///
/// Basic checks that don't depend on any context
/// Contains:
/// - Check for version
/// - Check for size
/// - Check inputs and output empty
/// - Check for duplicate deps
/// - Check for whether outputs match data
/// - Check whether output lock hash type within enabled range
pub struct NonContextualTransactionVerifier<'a> {
    pub(crate) version: VersionVerifier<'a>,
    pub(crate) size: SizeVerifier<'a>,
    pub(crate) empty: EmptyVerifier<'a>,
    pub(crate) duplicate_deps: DuplicateDepsVerifier<'a>,
    pub(crate) outputs_data_verifier: OutputsDataVerifier<'a>,
    pub(crate) script_hash_type: ScriptHashTypeVerifier<'a>,
}

impl<'a> NonContextualTransactionVerifier<'a> {
    /// Creates a new NonContextualTransactionVerifier
    pub fn new(tx: &'a TransactionView, consensus: &'a Consensus) -> Self {
        NonContextualTransactionVerifier {
            version: VersionVerifier::new(tx, consensus.tx_version()),
            size: SizeVerifier::new(tx, consensus.max_block_bytes()),
            empty: EmptyVerifier::new(tx),
            duplicate_deps: DuplicateDepsVerifier::new(tx),
            outputs_data_verifier: OutputsDataVerifier::new(tx),
            script_hash_type: ScriptHashTypeVerifier::new(tx),
        }
    }

    /// Perform context-independent verification
    pub fn verify(&self) -> Result<(), Error> {
        self.version.verify()?;
        self.size.verify()?;
        self.empty.verify()?;
        self.duplicate_deps.verify()?;
        self.outputs_data_verifier.verify()?;
        self.script_hash_type.verify()?;
        Ok(())
    }

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
