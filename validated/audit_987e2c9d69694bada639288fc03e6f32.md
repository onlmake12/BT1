Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Unique `from` PeerID Flood — (`network/src/protocols/hole_punching/component/connection_request.rs`, `network/src/protocols/hole_punching/mod.rs`)

## Summary
An unprivileged remote peer can exhaust heap memory on a CKB node by repeatedly sending `ConnectionRequest` messages addressed to the target node's own PeerID, each with a distinct random `from` PeerID. The `forward_rate_limiter` is bypassed because each unique `from` value creates a fresh rate-limiter bucket. The `pending_delivered` map has no size cap and is only pruned every 5 minutes, allowing unbounded growth at 30 entries/sec per connection.

## Finding Description

**`pending_delivered` is an unbounded `HashMap`** (`mod.rs` line 44):
`pending_delivered: HashMap<PeerId, PendingDeliveredInfo>` — no capacity limit, no eviction policy. [1](#0-0) 

**Per-session rate limiter** (`mod.rs` lines 95–107):
Keyed on `(session_id, msg.item_id())`. For `ConnectionRequest`, `item_id()` is always `0`. Allows at most 30 messages/sec per session — the only real throttle, and it is per-connection. [2](#0-1) 

**`forward_rate_limiter` bypass** (`connection_request.rs` lines 132–143):
Keyed on `(content.from, content.to, msg_item_id)`. Each unique `from` PeerID creates a fresh 1/sec bucket. With a new random `from` per message, every message passes this check unconditionally. [3](#0-2) 

**`respond_delivered` deduplication is per-`from`** (`connection_request.rs` lines 161–167):
The guard checks `pending_delivered.get(&from_peer_id)`. With unique `from` values, this lookup always misses, so the deduplication never fires. [4](#0-3) 

**`remote_listens.is_empty()` guard is trivially satisfied** (`connection_request.rs` lines 196–219):
The filter accepts any TCP address with IPv4/IPv6. The attacker includes one address like `/ip4/1.2.3.4/tcp/1234/p2p/<from_peer_id>` to pass this check. [5](#0-4) 

**Unconditional insert after successful `send_message_to`** (`connection_request.rs` lines 234–237):
After the `ConnectionRequestDelivered` reply is sent back to the attacker's session (which succeeds as long as the connection is open), a new entry is unconditionally inserted into `pending_delivered`. [6](#0-5) 

**Cleanup only every 5 minutes** (`mod.rs` lines 169–175):
`pending_delivered.retain(...)` is only called in the `notify` callback, which fires every `CHECK_INTERVAL = 5 min`. Entries inserted at the start of a window survive the full 5 minutes. [7](#0-6) 

**`forward_rate_limiter` internal state also grows unboundedly**:
`retain_recent()` on `forward_rate_limiter` is only called on peer disconnect (`mod.rs` lines 66–69), not periodically. Each unique `(from, to, item_id)` triple permanently occupies a slot in the `HashMapStateStore` until disconnect. [8](#0-7) 

## Impact Explanation

**Per-connection memory growth (single attacker session):**
- Rate: 30 entries/sec × 300 sec (5-min window) = 9,000 entries
- Per entry: `PeerId` key (~39 bytes) + `Vec<Multiaddr>` (up to 24 addresses × ~30 bytes = ~720 bytes) + `u64` ≈ ~800 bytes
- Single connection: ~7.2 MB per 5-minute window

With N simultaneous attacker connections: N × 9,000 entries. At 100 connections: ~720 MB. At 1,000 connections: ~7.2 GB — sufficient for OOM on typical node hardware.

The `forward_rate_limiter` internal `HashMapStateStore` adds a compounding factor: 9,000 entries per connection per window, each holding a `(PeerId, PeerId, u32)` key (~82 bytes) plus governor rate state, never pruned until disconnect.

This matches: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

- The target node's PeerID is publicly discoverable via the identify/discovery protocols — no privileged access required.
- The attacker needs only standard P2P connections and the ability to craft `ConnectionRequest` messages with one valid TCP `listen_addr`.
- The `forward_rate_limiter` bypass is trivial: generate a new random `PeerId` per message.
- The per-session rate limit (30/sec) is the only real constraint, and it is per-connection — opening more connections linearly amplifies the attack.
- No PoW, no privileged role, no leaked keys required.

## Recommendation

1. **Cap `pending_delivered`**: Enforce a hard maximum size (e.g., 1,024 entries). On overflow, reject new insertions or evict the oldest entry (LRU).
2. **Periodic `forward_rate_limiter` pruning**: Call `forward_rate_limiter.retain_recent()` inside the `notify` callback (every 5 minutes), not only on disconnect.
3. **Bind `pending_delivered` inserts to authenticated sessions**: Only insert into `pending_delivered` when the `from` PeerID corresponds to a currently-connected peer, preventing phantom PeerID injection.
4. **Reduce `CHECK_INTERVAL`**: A 5-minute cleanup window is too long given the 30/sec insertion rate. Reduce to 30–60 seconds.

## Proof of Concept

```
1. Discover target node's PeerID via identify protocol.
2. Open K TCP connections to the target node (K = desired amplification factor).
3. On each connection, in a loop at 30 msg/sec:
   a. Generate a fresh random PeerId (e.g., random 39-byte multihash).
   b. Craft a ConnectionRequest with:
      - from = fresh_random_peer_id
      - to   = target_node_peer_id
      - listen_addrs = [/ip4/1.2.3.4/tcp/1234/p2p/<fresh_random_peer_id>]  (1 valid TCP addr)
      - max_hops = 1
      - route = []
   c. Send over the HolePunching protocol stream.
4. After 5 minutes, observe pending_delivered.len() ≈ K × 9,000.
5. Measure RSS growth: ~800 bytes × K × 9,000.
   At K=100: ~720 MB. At K=1000: ~7.2 GB → OOM.
```

The `forward_rate_limiter` check passes for every message (each `from` is unique, so each gets a fresh 1/sec bucket). The `respond_delivered` deduplication check (lines 161–167) is bypassed because no two messages share the same `from`. The `remote_listens.is_empty()` guard (line 217) is satisfied by the single valid TCP address. Each message that reaches line 237 inserts unconditionally into the unbounded map.
