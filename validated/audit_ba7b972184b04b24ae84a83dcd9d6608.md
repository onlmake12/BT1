### Title
Lazer Signer Verification Lacks Domain Separation — Cross-Chain Replay of Signed Price Updates Accepted as Valid - (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary

`PythLazer.verifyUpdate` computes the signed hash as `keccak256(payload)` with no chain ID, contract address, or any domain separator included. Because the same trusted signer keys are registered across all EVM deployments, a valid signed Lazer update captured on chain A is cryptographically indistinguishable from a freshly-signed update on chain B. An unprivileged Lazer relayer can replay any in-window signed update across chains, bypassing the intended per-chain authentication binding.

### Finding Description

In `PythLazer.verifyUpdate`, the message hash fed to `ECDSA.tryRecover` is constructed as:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` bytes contain price data and a timestamp but **no chain ID and no contract address**. The trusted signer set is maintained independently per chain via `updateTrustedSigner`, but the same signer keys are registered on every deployment. [2](#0-1) 

Because `isValidSigner` only checks expiry against `block.timestamp` and the signature check is purely over `keccak256(payload)`, any `(update_bytes, signature)` pair that was valid on chain A will pass `verifyUpdate` on chain B without modification. [3](#0-2) 

The contract returns `(payload, signer)` to the caller and performs no timestamp staleness check itself; that responsibility is delegated entirely to downstream consumers.

### Impact Explanation

An unprivileged Lazer relayer can:

1. Observe a signed update on chain A at time T where the price is favorable (e.g., BTC/USD = $50,000).
2. At time T+Δ (where Δ is within the downstream contract's staleness window), submit the identical `update` bytes to `verifyUpdate` on chain B.
3. `verifyUpdate` returns success — the signature is valid, the signer is trusted, no chain binding is checked.
4. The downstream DeFi contract on chain B receives the replayed price as if it were freshly attested, while the true current price may have moved significantly.

This enables selective-replay price manipulation: the attacker cherry-picks the most favorable in-window signed update across all chains and injects it into the target chain, enabling incorrect liquidations, over-collateralized borrows, or arbitrage against protocols that consume Lazer prices.

**Impact**: High — direct manipulation of price data consumed by DeFi protocols.

### Likelihood Explanation

**Likelihood**: Medium.

- No privileged access is required; any address can call `verifyUpdate` with a fee of 1 wei.
- The attacker only needs to monitor the public Lazer update stream (already broadcast to relayers) and submit a replay transaction within the downstream staleness window (typically 60–120 seconds).
- The attack is fully deterministic and requires no cryptographic forgery — only replaying an already-valid signed blob.

### Recommendation

Include a domain separator in the signed hash that binds the signature to the specific chain and contract:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    keccak256(payload)
));
```

Alternatively, adopt EIP-712 structured signing so that the Lazer signer infrastructure produces chain-specific signatures. The trusted signer's off-chain signing service must include `chainId` and the contract address in the signed message.

### Proof of Concept

1. Deploy `PythLazer` on chain A (chainId=1) and chain B (chainId=137) with the same trusted signer registered on both.
2. On chain A, call `verifyUpdate{value: 1 wei}(update_bytes)` — succeeds, returns `(payload, signer)`.
3. On chain B, call `verifyUpdate{value: 1 wei}(update_bytes)` with the **identical** `update_bytes` — also succeeds, returns the same `(payload, signer)`.
4. The downstream contract on chain B accepts the cross-chain replayed price as valid.

No signature forgery is needed. The root cause is that `keccak256(payload)` at line 93 contains no chain-binding context, making the verification equivalent to the biometric app's `onAuthenticationSucceeded` callback that carries no `CryptoObject` — the cryptographic check passes but is not bound to the intended execution context.
