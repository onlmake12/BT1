### Title
Entropy `requestWithCallback()` / `requestV2()` Does Not Validate Caller Implements `IEntropyConsumer` — Fee Permanently Lost With No Refund Path - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `requestWithCallback()` and `requestV2()` accept any `msg.sender` as the requester and immediately distribute the fee to the provider and Pyth without verifying that `msg.sender` implements the `IEntropyConsumer` interface. If a contract that does not implement `_entropyCallback` calls these functions, the fee is permanently lost: the provider can never successfully execute `revealWithCallback()` against that requester, and there is no on-chain refund or cancellation mechanism.

---

### Finding Description

`requestHelper()` stores `msg.sender` as `req.requester` and immediately credits fees:

```solidity
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
...
req.requester = msg.sender;
``` [1](#0-0) 

No check is made that `msg.sender` implements `IEntropyConsumer._entropyCallback`. The interface contract itself only enforces the caller-is-Entropy guard *inside* the callback, not at request time:

```solidity
function _entropyCallback(...) external {
    address entropy = getEntropy();
    require(entropy != address(0), "Entropy address not set");
    require(msg.sender == entropy, "Only Entropy can call this function");
    entropyCallback(sequence, provider, randomNumber);
}
``` [2](#0-1) 

**Legacy path (gasLimit == 0):** `revealWithCallback` clears the request *before* calling the callback. If the callback reverts (because `_entropyCallback` does not exist on the requester), the entire `revealWithCallback` transaction reverts. The request is un-cleared and stays active forever. The fee, however, was distributed in the *prior* `requestWithCallback` transaction and is not reverted. [3](#0-2) 

**New path (gasLimit != 0):** `excessivelySafeCall` catches the revert and moves the request to `CALLBACK_FAILED`. The request can be retried, but every retry will also fail. The fee is still gone. [4](#0-3) 

There is no `cancelRequest` or refund function anywhere in `Entropy.sol`. The comment in the interface only says "The address calling this function *should be* a contract that inherits from `IEntropyConsumer`" — this is advisory, not enforced. [5](#0-4) 

The same pattern exists in `Echo.sol`'s `requestPriceUpdatesWithCallback`, which stores `msg.sender` as `req.requester` without validating `IEchoConsumer` compliance, and where the provider fee is locked inside the request struct until `executeCallback` succeeds. [6](#0-5) 

---

### Impact Explanation

Any contract that calls `requestWithCallback()` or `requestV2()` without implementing `IEntropyConsumer` will:

1. Pay the full fee (provider fee + Pyth fee) — immediately and irrevocably distributed.
2. Receive a sequence number but never receive the random number callback.
3. Have no on-chain path to cancel the request or recover funds.

In the legacy path the provider's keeper bot will waste gas on perpetually-reverting `revealWithCallback` calls. In the new path the request is permanently stuck in `CALLBACK_FAILED`. Either way the user's fee is lost with no recourse.

---

### Likelihood Explanation

The entry path is fully unprivileged: any EOA or contract can call `requestWithCallback()`. Contracts that integrate Entropy without inheriting `IEntropyConsumer` (e.g., proxy patterns, factory-deployed consumers, or contracts that call Entropy on behalf of another address) will silently trigger this condition. The documentation warning is off-chain only; there is zero on-chain enforcement.

---

### Recommendation

1. Add an ERC-165 `supportsInterface(IEntropyConsumer.interfaceId)` check in `requestHelper` when `isRequestWithCallback == true`, reverting if the caller is a contract that does not implement the interface.
2. Alternatively, add a `cancelRequest` function that allows the original requester to cancel an unfulfillable request and receive a partial refund (at minimum the provider fee portion).
3. Clarify the `@param` / NatSpec comment to state explicitly that calling from a non-`IEntropyConsumer` contract will result in permanent fee loss.

---

### Proof of Concept

1. Deploy a contract `BadConsumer` that does **not** inherit `IEntropyConsumer` but calls `entropy.requestWithCallback{value: fee}(provider, userRandom)`.
2. Observe that the fee is deducted and `providerInfo.accruedFeesInWei` increases immediately.
3. Have the provider call `revealWithCallback(provider, seqNum, userRandom, providerRandom)`.
   - **Legacy path**: transaction reverts with a call-to-non-existent-function error; request stays active; fee already gone.
   - **New path**: `CallbackFailed` event emitted; request moves to `CALLBACK_FAILED`; fee already gone.
4. Confirm there is no function in `Entropy.sol` that allows `BadConsumer` to recover its fee. [7](#0-6) [8](#0-7)
