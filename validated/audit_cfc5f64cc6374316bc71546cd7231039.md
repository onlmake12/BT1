### Title
O(N) Linear Scan Under `unknown_tx_hashes` Mutex After Insert-Before-Check in `add_ask_for_txs` — (`sync/src/types/mod.rs`)

---

### Summary

`SyncState::add_ask_for_txs` inserts up to 32,767 hashes into the global `unknown_tx_hashes` queue **before** checking the global size limit. When the queue reaches `MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`, every subsequent call performs a full O(N) linear scan over all queue entries while holding the `unknown_tx_hashes` `Mutex`. Because the overflow response is `Status::ignored()` (no peer ban, no disconnect), an attacker with two or more cheap P2P connections can permanently keep the queue above the threshold and force every `add_ask_for_txs` invocation to pay the O(N) scan cost, blocking `pop_ask_for_txs` and `mark_as_known_txs` for the duration.

---

### Finding Description

**Entry point — `TransactionHashesProcess::execute`**

A remote peer sends a `RelayTransactionHashes` P2P message. The only guard is a per-message size check:

```
if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH { … }
```

There is no per-peer rate limit and no cooldown. After filtering already-known hashes through `tx_filter`, the function calls `state.add_ask_for_txs(self.peer, tx_hashes)` unconditionally. [1](#0-0) 

**Insert-before-check in `add_ask_for_txs`**

The mutex is acquired at line 1484 and held for the entire function. Lines 1486–1503 insert up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767` new entries unconditionally. Only after all insertions does the code check whether the global limit has been exceeded: [2](#0-1) 

**O(N) scan triggered on overflow**

When `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE (50,000)`, the code performs a nested linear scan over every entry and every peer-list inside each entry to count how many hashes belong to the calling peer: [3](#0-2) 

**No penalty on overflow**

The overflow branch returns `Status::ignored()`, not a ban or disconnect. The attacker is never penalized for triggering this path repeatedly. [4](#0-3) 

**Constants** [5](#0-4)

### Impact Explanation

The `unknown_tx_hashes` `Mutex` is shared by three callers:

- `add_ask_for_txs` — called on every inbound `RelayTransactionHashes` message
- `pop_ask_for_txs` — called by the `ASK_FOR_TXS_TOKEN` periodic timer
- `mark_as_known_txs` — called when a transaction is verified

While `add_ask_for_txs` holds the mutex and executes the O(N) scan, both `pop_ask_for_txs` and `mark_as_known_txs` block. If the attacker drives a high rate of inbound messages from multiple connections, the mutex becomes a serialization bottleneck: the timer that actually dispatches `GetRelayTransactions` requests stalls, and verified transactions cannot be removed from the queue, causing the queue to stay inflated and the O(N) scan to fire on every subsequent call. The result is degraded tx-relay throughput across the node.

---

### Likelihood Explanation

The attack requires only:
1. Two or more standard P2P connections (no PoW, no stake, no privilege).
2. Two batches of 32,767 unique fake 32-byte hashes each (total ~2 MB of data) to push the queue past 50,000.
3. Continued low-rate messages to keep the queue above the threshold (since `pop_ask_for_txs` drains it over time).

The `tx_filter` TTL filter does not prevent this because the attacker uses fresh hashes each time. The overflow response is silent (`Status::ignored()`), so the attacker receives no feedback and is never disconnected.

---

### Recommendation

1. **Check before insert**: Evaluate the per-peer and global limits before inserting any hashes, and reject the batch early if either limit would be exceeded.
2. **Replace the O(N) scan with O(1) accounting**: Maintain a per-peer counter (e.g., in `PeerState`) that is incremented/decremented alongside insertions and removals, eliminating the need to scan the entire queue.
3. **Enforce a hard per-peer cap at insertion time**: Once a peer's counter reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, reject further hashes from that peer without touching the global queue.
4. **Consider banning or rate-limiting peers that repeatedly trigger the overflow path** rather than silently ignoring the condition.

---

### Proof of Concept

```
1. Connect two attacker peers A and B to the target node.
2. Peer A sends RelayTransactionHashes with 32,767 unique fake hashes → queue size ~32,767.
3. Peer B sends RelayTransactionHashes with 32,767 different unique fake hashes → queue size ~50,000+, overflow branch fires.
4. Now send any RelayTransactionHashes message from any peer.
   → add_ask_for_txs acquires the mutex and iterates over all ~50,000 entries.
5. Measure time-to-return of add_ask_for_txs vs. baseline (queue empty).
6. Flood with concurrent messages from multiple connections; observe that
   pop_ask_for_txs and mark_as_known_txs are delayed proportionally.
```

The queue can be kept inflated indefinitely because the overflow response is `Status::ignored()` and the attacker is never disconnected.

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
