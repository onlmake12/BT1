### Title
Provider Cannot Withdraw Own Accrued Fees Due to Missing Direct Withdrawal Function — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` exposes only `withdrawAsFeeManager()` as the fee-withdrawal path. Unlike `Entropy.sol`, which provides a separate `withdraw()` function callable directly by the provider (`msg.sender`), `Echo.sol` has no equivalent. Any registered Echo provider that has not explicitly set a fee manager has their accrued fees permanently locked in the contract with no way to retrieve them.

### Finding Description

In `Entropy.sol`, two distinct withdrawal paths exist:

1. `withdraw(uint128 amount)` — callable by the provider itself (uses `msg.sender` as the provider key).
2. `withdrawAsFeeManager(address provider, uint128 amount)` — callable only by the designated fee manager. [1](#0-0) 

In `Echo.sol`, only the second path exists:

```solidity
function withdrawAsFeeManager(
    address provider,
    uint128 amount
) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

There is no `withdraw()` function in `Echo.sol` that allows the provider to withdraw their own fees directly. The `setProviderFee` function in the same contract correctly allows **both** the provider and the fee manager:

```solidity
require(
    msg.sender == provider ||
        msg.sender == _state.providers[provider].feeManager,
    "Only provider or fee manager can invoke this method"
);
``` [3](#0-2) 

This inconsistency — where fee-setting allows both parties but fee-withdrawal allows only the fee manager — is the direct analog of the reported bug: an intended caller (the provider) is silently excluded from a function they should be able to invoke.

### Impact Explanation

Any Echo provider that registers without calling `setFeeManager` has their accrued `accruedFeesInWei` balance permanently inaccessible. The funds are not stolen, but they are locked with no recovery path. Even if the provider later calls `setFeeManager(address(self))`, the withdrawal sends ETH to `msg.sender` (the fee manager address), which works only if the provider remembers this workaround. The impact is loss of provider funds / broken protocol functionality. [4](#0-3) 

### Likelihood Explanation

Setting a fee manager is optional and not required during `registerProvider`. Any provider who registers and accrues fees without having set a fee manager — a realistic default state — is immediately affected. The entry path is fully permissionless: any address can call `registerProvider` and begin accruing fees. [5](#0-4) 

### Recommendation

Add a `withdraw(uint128 amount)` function to `Echo.sol` that mirrors the one in `Entropy.sol`, allowing the provider (`msg.sender`) to withdraw their own accrued fees directly, without requiring a fee manager to be set:

```solidity
function withdraw(uint128 amount) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(provider.isRegistered, "Provider not registered");
    require(provider.accruedFeesInWei >= amount, "Insufficient balance");
    provider.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "withdrawal to msg.sender failed");
    emit FeesWithdrawn(msg.sender, amount);
}
```
