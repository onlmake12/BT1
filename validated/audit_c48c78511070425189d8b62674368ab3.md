### Title
Tx-Pool Panic via Stale Parent Reference After Cascading Ancestor Eviction — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `check_and_record_ancestors`, when a `cell_ref_parent` is evicted via `remove_entry_and_descendants`, its pool descendants are also silently removed. The eviction loop only calls `parents.remove(next_id)` for the directly evicted entry, leaving any descendants that were also direct parents of the incoming transaction in the `parents` set. The subsequent `calc_relation_ids(parents)` returns those stale IDs, and `_record_ancestors` calls `get_by_id_checked` on them, triggering an unconditional `expect("inconsistent pool")` panic that crashes the node.

---

### Finding Description

`check_and_record_ancestors` in `tx-pool/src/component/pool_map.rs` handles the case where a new transaction's ancestor count exceeds `max_ancestors_count`. When the excess is attributable to `cell_ref_parents` — pool transactions that reference a cell as a dep that the new transaction consumes as an input — the function attempts to evict them. [1](#0-0) 

The eviction loop calls `remove_entry_and_descendants(next_id)`, which removes the targeted cell_ref_parent **and all of its pool descendants**: [2](#0-1) 

However, the loop only calls `parents.remove(next_id)` for the directly targeted entry. If a descendant of the evicted cell_ref_parent was also a direct parent of the incoming transaction (e.g., the new tx spends that descendant's output), it remains in `parents` even though it has been removed from the pool and from `links`.

After the loop, the code recalculates ancestors from the now-stale `parents` set: [3](#0-2) 

`calc_relation_ids` in `links.rs` unconditionally inserts every ID from `stage` into `relation_ids`, even if the ID is absent from `self.inner` (i.e., already removed from the pool): [4](#0-3) 

The stale descendant ID therefore appears in the returned `ancestors` set. `_record_ancestors` then iterates over `ancestors` and calls `get_by_id_checked` on each: [5](#0-4) 

`get_by_id_checked` calls `.expect("inconsistent pool")` on the result: [6](#0-5) 

Because the stale descendant is no longer in the pool, `get_by_id` returns `None`, and the `expect` **panics**, crashing the node process.

---

### Impact Explanation

An unprivileged transaction sender can crash any CKB full node by submitting a crafted sequence of transactions that triggers the panic inside `add_entry` → `check_and_record_ancestors` → `_record_ancestors` → `get_by_id_checked`. The crash is a hard process abort (Rust panic), constituting a complete denial-of-service against the targeted node. The node must be restarted to recover, and the attack can be repeated indefinitely.

---

### Likelihood Explanation

The attacker only needs to submit transactions via the standard `send_transaction` RPC — no privileged access, no key material, and no majority hashpower is required. The required transaction graph (a cell_ref_parent with a descendant that is also a direct parent of the new transaction, with total ancestor count exactly one above `max_ancestors_count`) is straightforward to construct. The default `max_ancestors_count` is 25, which is easily reachable. [7](#0-6) 

---

### Recommendation

After calling `remove_entry_and_descendants(next_id)` in the eviction loop, also remove all descendants of `next_id` from `parents`. The simplest fix is to compute the set of removed IDs before eviction and subtract them from `parents`:

```rust
let removed = self.remove_entry_and_descendants(next_id);
for r in &removed {
    parents.remove(&r.proposal_short_id());
}
ancestors_count = ancestors_count.saturating_sub(1);
evicted.extend(removed);
```

Alternatively, after the eviction loop, filter `parents` to retain only IDs still present in the pool:

```rust
parents.retain(|id| self.entries.get_by_id(id).is_some());
```

The `assert!` at line 636 should also be converted to a proper `Result`-returning error rather than a panic, consistent with the FIXME comment already present at line 583. [8](#0-7)

### Proof of Concept

**Setup** (`max_ancestors_count` = 25, default):

1. Submit **tx_A**: spends any confirmed cell; includes `cell_X` as a `CellDep` (Code type).
2. Submit **tx_B**: spends `tx_A`'s output (making tx_B a pool descendant of tx_A). Does **not** reference `cell_X` as a dep.
3. Submit **tx_C1 → tx_C2 → … → tx_C23**: a linear chain of 23 transactions, each spending the previous one's output.
4. Submit **new_tx** with:
   - Input 0: `cell_X` (the live cell that tx_A references as a dep) — this makes tx_A a `cell_ref_parent`
   - Input 1: tx_B's output — this makes tx_B a direct parent via input
   - Input 2: tx_C23's output — this makes tx_C23 a direct parent via input

**State at submission of new_tx**:
- `cell_ref_parents` = {tx_A}
- `parents` = {tx_A, tx_B, tx_C23}
- `ancestors` = {tx_A, tx_B, tx_C1…tx_C23} → 25 entries
- `ancestors_count` = 26 > 25
- `ancestors_count − cell_ref_parents.len()` = 25 ≤ 25 → enters eviction branch

**Eviction loop**:
- Evicts tx_A → `remove_entry_and_descendants(tx_A)` removes **tx_A and tx_B** from pool and links
- `ancestors_count` decremented to 25; `parents.remove(tx_A)` → `parents` = {**tx_B**, tx_C23}
- Loop exits (25 ≤ 25)

**Post-eviction**:
- `calc_relation_ids({tx_B, tx_C23}, Parents)` → returns {tx_B, tx_C1…tx_C23} (tx_B included despite being removed)
- `_record_ancestors` calls `get_by_id_checked(tx_B)` → `None.expect("inconsistent pool")` → **PANIC / node crash** [9](#0-8) [10](#0-9) [5](#0-4)

    pub fn calc_relation_ids(
        &self,
        mut stage: HashSet<ProposalShortId>,
        relation: Relation,
    ) -> HashSet<ProposalShortId> {
        let mut relation_ids = HashSet::with_capacity(stage.len());

        while let Some(id) = stage.iter().next().cloned() {
            //recursively
            if let Some(tx_links) = self.inner.get(&id) {
                for direct_id in tx_links.get_direct_ids(relation) {
                    if !relation_ids.contains(direct_id) {
                        stage.insert(direct_id.clone());
                    }
                }
            }
            stage.remove(&id);
            relation_ids.insert(id);
        }
        relation_ids
    }
