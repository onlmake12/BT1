### Title
Provider Accrued Fees Permanently Locked When Fee Manager Set to Zero Address - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.sol` allows a registered provider to call `setFeeManager(address(0))` without any zero-address guard. Because `Echo.sol` provides **no direct provider `withdraw` function** (unlike `Entropy.sol`), setting the fee manager to `address(0)` permanently locks all of the provider's accrued fees inside the contract with no recovery path.

### Finding Description
`Echo.sol`'s `setFeeManager` accepts any address, including `address(0)`, without validation:

```solidity
function setFeeManager(address manager) external override {
    require(
        _state.providers[msg.sender].isRegistered,
        "Provider not registered"
    );
    address oldFeeManager = _state.providers[msg.sender].feeManager;
    _state.providers[msg.sender].feeManager = manager;   // no zero-address check
    emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
}
``` [1](#0-0) 

The only provider-fee withdrawal path in `Echo.sol` is `withdrawAsFeeManager`, which enforces `msg.sender == feeManager`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

If `feeManager == address(0)`, the `require` can never be satisfied (no transaction can originate from `address(0)`), so the provider's `accruedFeesInWei` balance is permanently inaccessible.

`Echo.sol` has no alternative `withdraw(uint128)` function for providers (unlike `Entropy.sol`, which does provide one at line 150). The only other withdrawal function is `withdrawFees`, which is admin-only and drains the Pyth protocol fee pool, not provider balances: [3](#0-2) 

The `IEntropy.sol` interface comment explicitly documents that passing `address(0)` to `setFeeManager` is the intended way to "disable the fee manager role": [4](#0-3) 

This creates a dangerous cross-contract assumption: a provider familiar with `Entropy.sol` (where disabling the fee manager is safe because `withdraw()` still works) may apply the same pattern to `Echo.sol`, permanently losing their fees.

### Impact Explanation
A registered Echo provider who calls `setFeeManager(address(0))` — whether intentionally to "disable" the role or by mistake — permanently loses all accrued fees stored in `_state.providers[provider].accruedFeesInWei`. There is no admin override, no recovery function, and no upgrade path that would unilaterally restore access. The ETH is locked in the contract forever.

### Likelihood Explanation
The `IEntropy.sol` interface (shared documentation for the Entropy/Echo ecosystem) explicitly documents `address(0)` as the canonical way to disable the fee manager role. Any provider who reads this documentation and applies it to `Echo.sol` will trigger the bug. Additionally, a provider who initially set a fee manager and later wants to revoke that role has no other documented mechanism. Likelihood is **medium**: it requires a provider to take a self-directed action, but the action is explicitly documented as valid in the shared interface.

### Recommendation
Add a zero-address guard in `Echo.sol`'s `setFeeManager`:

```solidity
function setFeeManager(address manager) external override {
    require(_state.providers[msg.sender].isRegistered, "Provider not registered");
    require(manager != address(0), "Fee manager cannot be zero address");
    ...
}
```

Alternatively, add a direct `withdraw(uint128 amount)` function for providers (mirroring `Entropy.sol`) so that providers can always recover their fees regardless of the fee manager state.

### Proof of Concept
1. Provider calls `Echo.registerProvider(baseFee, feedFee, gasRate)` — provider is now registered.
2. Fees accrue in `_state.providers[provider].accruedFeesInWei` as users call `requestPriceUpdatesWithCallback`.
3. Provider calls `Echo.setFeeManager(address(0))` — no revert, `feeManager` is now `address(0)`.
4. Provider (or anyone) calls `Echo.withdrawAsFeeManager(provider, amount)` — reverts with `"Only fee manager"` because `msg.sender != address(0)`.
5. No other function exists to withdraw provider fees. Funds are permanently locked.
