### Title
Excess `msg.value` Permanently Locked in `Pyth.sol` `updatePriceFeeds` — No Refund of Overpayment - (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`Pyth.sol`'s `updatePriceFeeds` (and `parsePriceFeedUpdates`) are `payable` functions that only enforce `msg.value >= requiredFee`. Any ETH sent above the required fee is never refunded and is permanently locked in the contract, recoverable only via a privileged governance `WithdrawFee` action. This is a direct analog to the Paraspace adapter vulnerability where excess `msg.value` is silently consumed.

---

### Finding Description

`updatePriceFeeds` computes `requiredFee` from the actual update data after processing it, then checks:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
```

There is no `msg.value == requiredFee` check and no refund of `msg.value - requiredFee`. The excess ETH is silently retained by the contract. The same pattern applies to `parsePriceFeedUpdates` and `updatePriceFeedsIfNecessary`, all of which are `payable` with only a lower-bound fee check.

The governance test at line 559 explicitly demonstrates this: sending `1 ether` when the fee is `1000 wei` results in the full `1 ether` being locked in the contract.

---

### Impact Explanation

Any unprivileged caller of `updatePriceFeeds` who sends more ETH than `requiredFee` permanently loses the excess. The ETH is locked in the Pyth contract and can only be recovered via a governance `WithdrawFee` VAA — a privileged, multi-step process that does not return funds to the original sender. The user has no self-service recovery path.

This is a direct loss of user funds, not merely a theoretical concern: fee estimation race conditions (fee changes between `getUpdateFee` call and `updatePriceFeeds` execution), integrator bugs, or simple overpayment all trigger the loss.

---

### Likelihood Explanation

`updatePriceFeeds` is the primary entry point for all price feed updates on EVM chains and is called by any unprivileged updater. Fee estimation is done off-chain via `getUpdateFee`, and the actual fee is computed on-chain from the parsed update data. A mismatch between estimated and actual fee (e.g., due to a fee governance change between estimation and submission, or an integrator passing a slightly higher value for safety) causes the excess to be silently consumed. This is a realistic, common scenario for any active integrator.

---

### Recommendation

Add a refund of excess `msg.value` at the end of `updatePriceFeeds` (and all other payable fee-collecting functions):

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
if (msg.value > requiredFee) {
    payable(msg.sender).transfer(msg.value - requiredFee);
}
```

Alternatively, enforce strict equality: `if (msg.value != requiredFee) revert PythErrors.InvalidFee();`

Apply the same fix to `parsePriceFeedUpdates`, `parsePriceFeedUpdatesUnique`, `parseTwapPriceFeedUpdates`, and `updatePriceFeedsIfNecessary`.

---

### Proof of Concept

1. Governance sets `singleUpdateFeeInWei = 1` and `transactionFeeInWei = 0`.
2. Caller fetches `getUpdateFee(updateData)` → returns `1 wei`.
3. Between the estimation call and the submission, governance raises `transactionFeeInWei` to `1000 wei`. Now `requiredFee = 1001 wei`.
4. Caller submits `updatePriceFeeds{value: 1 ether}(updateData)` (using a safe overpayment buffer).
5. The check `msg.value < requiredFee` passes (1 ether > 1001 wei).
6. `1 ether - 1001 wei` is permanently locked in the contract. The caller has no refund path.

This is confirmed by the existing governance test:

```solidity
// PythGovernance.t.sol line 559
pyth.updatePriceFeeds{value: 1 ether}(updateData);
assertEq(address(pyth).balance, 1 ether); // full 1 ether locked
