### Title
Missing `BLOCK_INVALID` Early-Exit in `HeaderAcceptor::accept()` Enables Repeated Eaglesong PoW Re-Verification — (`sync/src/synchronizer/headers_process.rs`)

---

### Summary

A remote, unprivileged P2P peer can cause a CKB node to re-run the full `HeaderVerifier::verify()` pipeline — including an Eaglesong hash computation — on every receipt of a `SendHeaders` message containing headers already marked `BLOCK_INVALID`, because `accept()` has no early-exit guard for that status. The FIXME comment in the code explicitly acknowledges this gap.

---

### Finding Description

`HeaderAcceptor::accept()` reads the block status at entry but only short-circuits on `HEADER_VALID`: [1](#0-0) 

The comment at line 301–302 explicitly flags the missing guard:

```
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
```

`BLOCK_INVALID` is bit 12 (`1 << 12`), completely disjoint from `HEADER_VALID` (bit 0): [2](#0-1) 

So a header carrying `BLOCK_INVALID` does **not** trigger the early return. Execution falls through to `prev_block_check` (which checks the *parent*, not the header itself) and then to `non_contextual_check`: [3](#0-2) 

`non_contextual_check` calls `self.verifier.verify(self.header)`, which is `HeaderVerifier::verify()`. That function runs **PoW first**: [4](#0-3) 

`PowVerifier` calls `EaglesongPowEngine::verify()`, which unconditionally runs the full Eaglesong hash: [5](#0-4) 

When PoW fails, the error is neither `BlockError(UnknownParent)` nor `HeaderError::is_too_new()`, so `non_contextual_check` returns `Err(true)` and `accept()` writes `BLOCK_INVALID` back to the store: [6](#0-5) 

On the **next** `SendHeaders` message containing the same header, the cycle repeats from the top — `BLOCK_INVALID` is read, the `HEADER_VALID` guard is not triggered, and Eaglesong runs again.

---

### Impact Explanation

Each `SendHeaders` message may carry up to `MAX_HEADERS_LEN = 2000` headers: [7](#0-6) 

Each header triggers one full Eaglesong hash computation. An attacker who pre-generates a batch of 2000 headers with arbitrary (invalid) nonces can replay that same batch repeatedly, forcing the victim node to re-run up to 2000 Eaglesong hashes per message with zero additional attacker cost. This is a classic asymmetric CPU-exhaustion DoS: the attacker pays only network bandwidth; the victim pays CPU on every receipt.

---

### Likelihood Explanation

- **No mining required.** Headers with invalid PoW nonces are sufficient; the attacker does not need any hashpower.
- **Reachable via standard P2P.** `SendHeaders` is a normal sync protocol message accepted from any peer.
- **No effective rate-limit or ban visible in this path.** The function returns `StatusCode::HeadersIsInvalid`, but even if the peer is banned, the attacker can reconnect or use multiple peers.
- **FIXME comment confirms developer awareness** that the guard is absent.

---

### Recommendation

At the top of `accept()`, after reading `status`, add an explicit early-return for `BLOCK_INVALID`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    state.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) { … }
```

This resolves the FIXME and ensures known-invalid headers are rejected in O(1) without touching the PoW engine.

---

### Proof of Concept

1. Craft a `HeaderView` with an arbitrary (invalid) nonce — no mining needed.
2. Call `HeaderAcceptor::accept()` once. Observe that `BLOCK_INVALID` is written to the store and `non_contextual_check` ran (Eaglesong executed).
3. Call `HeaderAcceptor::accept()` a second time with the **same header**.
4. Observe that `non_contextual_check` runs **again** (Eaglesong executes again), confirming no short-circuit on `BLOCK_INVALID`.
5. Repeat step 3–4 in a tight loop or via a network fuzzer sending `SendHeaders` with the same 2000-header batch to measure CPU impact.
