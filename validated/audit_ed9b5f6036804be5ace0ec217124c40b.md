### Title
Excess Fee Coin Permanently Confiscated in `update_price_feeds` — (`target_chains/aptos/contracts/sources/pyth.move`)

---

### Summary

`update_price_feeds` accepts a caller-supplied `Coin<AptosCoin>` and only asserts that it is *at least* the required fee, but then unconditionally deposits the **entire** coin to `@pyth`. Any amount above the required fee is permanently transferred to the Pyth contract with no refund path. This is the direct Aptos analog of the Seaport `OrderFulfilled` discrepancy: the amount actually taken by the protocol diverges from the amount that should have been charged.

---

### Finding Description

In `target_chains/aptos/contracts/sources/pyth.move`, `update_price_feeds` computes the required fee after processing all VAAs, checks sufficiency, then deposits the whole coin:

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>) {
    let total_updates = 0;
    while (!vector::is_empty(&vaas)) {
        total_updates = total_updates + update_price_feed_from_single_vaa(...);
    };
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    coin::deposit(@pyth, fee);   // ← entire coin, not just update_fee
}
``` [1](#0-0) 

The check on line 178 only enforces a lower bound; the deposit on line 179 takes `coin::value(&fee)` in full. The surplus `coin::value(&fee) − update_fee` is irrecoverably credited to `@pyth`.

Contrast with `update_price_feeds_with_funder`, the entry-function sibling, which correctly withdraws **exactly** `update_fee` from the signer:

```move
let fee = coin::withdraw<AptosCoin>(account, update_fee);
coin::deposit(@pyth, fee);
``` [2](#0-1) 

`update_price_feeds` is a `public fun` callable by any Move module. Its companion `update_price_feeds_if_fresh` (also a callable entry fun) forwards the caller-supplied coin directly into `update_price_feeds` without trimming it:

```move
update_price_feeds(vaas, fee);   // fee may exceed update_fee
``` [3](#0-2) 

`get_update_fee` parses the same accumulator messages to derive the same count, so the two values *can* agree — but the contract never enforces equality, only `≤`. [4](#0-3) 

---

### Impact Explanation

Any caller of `update_price_feeds` or `update_price_feeds_if_fresh` who supplies a coin whose value exceeds the exact required fee permanently loses the surplus to `@pyth`. There is no withdrawal function, no refund path, and no event that records the discrepancy. Integrating Move modules that round up, use a cached fee estimate, or pass a pre-minted coin of fixed denomination will silently overpay on every call. The lost funds accumulate in `@pyth` and are unrecoverable by the caller.

---

### Likelihood Explanation

`update_price_feeds` is the primary non-entry integration point documented for Aptos (the docs show `coin::withdraw(user, pyth::get_update_fee(...))` as the recommended pattern, but this is advisory, not enforced). Any integrating contract that:

- caches the fee from a prior `get_update_fee` call and the VAA content changes between the two calls,
- rounds up to a safe margin,
- or passes a coin minted to a fixed denomination,

will overpay. The `update_price_feeds_if_fresh` entry function accepts a raw `Coin<AptosCoin>` parameter from the caller with no trimming, making the surface directly reachable by unprivileged users.

---

### Recommendation

Extract only the required amount before depositing:

```move
let surplus = coin::extract(&mut fee, coin::value(&fee) - update_fee);
coin::deposit(@pyth, fee);
// return surplus to caller, e.g. coin::deposit(signer::address_of(caller), surplus)
```

Alternatively, enforce exact equality (`update_fee == coin::value(&fee)`) so callers are forced to supply the precise amount, consistent with how `update_price_feeds_with_funder` behaves.

---

### Proof of Concept

1. Governance sets `base_update_fee = 100`.
2. A VAA containing 3 price feeds is prepared; `get_update_fee` returns `300`.
3. Caller mints a coin of value `500` (e.g., a round-number budget) and calls `update_price_feeds(vaas, coin_500)`.
4. Line 178 passes: `300 ≤ 500`.
5. Line 179 deposits `500` to `@pyth`.
6. Caller loses `200` AptosCoin with no event, no refund, and no revert.
7. `update_price_feeds_with_funder` called with the same VAA would have withdrawn exactly `300` — confirming the asymmetry between the two entry points. [5](#0-4)

    public fun get_update_fee(update_data: &vector<vector<u8>>): u64 {
        let i = 0;
        let total_updates = 0;
        while (i < vector::length(update_data)) {
            let cur = cursor::init(*vector::borrow(update_data, i));
            let header: u64 = deserialize::deserialize_u32(&mut cur);
            if (header == PYTHNET_ACCUMULATOR_UPDATE_MAGIC) {
                //TODO: this may be expensive and can be optimized by not verifying the messages
                let updates = parse_and_verify_accumulator_message(&mut cur);
                total_updates = total_updates + vector::length(&updates);
            }
            else {
                total_updates = total_updates + 1;
            };
            cursor::rest(cur);
            i = i + 1;
        };
        state::get_base_update_fee() * total_updates
    }
