### Title
Tx-Pool Write Lock Held for O(N × max_cycle_time) During Reorg via `readd_detached_tx` + `block_in_place` — (`tx-pool/src/process.rs`, `tx-pool/src/util.rs`)

### Summary

During a chain reorganization, `update_tx_pool_for_reorg` acquires the tx-pool write lock and, while holding it, calls `readd_detached_tx`, which synchronously re-verifies every detached transaction using `block_in_place(ContextualTransactionVerifier::verify(...))`. With `max_tx_verify_cycles = 70_000_000` per transaction, the write lock is held for the full cumulative verification time of all detached transactions, blocking every concurrent tx submission, RPC query, and block assembly operation.

In `update_tx_pool_for_reorg`, the write lock is taken at line 836 and the entire block (lines 836–851) holds it, including the `.await` call to `readd_detached_tx`: [1](#0-0) 

**Step 2 — `readd_detached_tx` calls `verify_rtx` with `command_rx = None`:**

For every transaction in the detached set, `verify_rtx` is called with `None` as the `command_rx` argument (line 901), meaning no chunked/pausable verification path is used: [2](#0-1) 

**Step 3 — `verify_rtx` with `command_rx = None` falls into the `block_in_place` branch:**

In `util.rs`, when `command_rx` is `None`, the function skips the async `verify_with_pause` path and instead calls `tokio::task::block_in_place`, running `ContextualTransactionVerifier::verify(max_tx_verify_cycles, false)` synchronously on the current thread: [3](#0-2) 

`block_in_place` does **not** yield the async task — it runs the closure synchronously. The write lock guard (held in the outer async function's stack frame) remains locked for the entire duration of the closure.

**Step 4 — `max_tx_verify_cycles` is 70,000,000:** [4](#0-3) 

With N detached transactions each consuming ~70M cycles, the write lock hold time is O(N × single_tx_verify_time).

### Impact Explanation

The tx-pool write lock is a Tokio async `RwLock`. While it is held:
- All concurrent `send_transaction` RPC calls block waiting for the write lock.
- All `get_block_template` (block assembly) calls block.
- All tx-pool read operations (e.g., `get_raw_tx_pool`) block.

A reorg involving K detached blocks each containing M near-max-cycle transactions results in a write-lock hold time proportional to K × M × (time to verify 70M cycles). This can easily exceed RPC timeout thresholds, causing the node to appear unresponsive.

---

### Likelihood Explanation

The attacker model has one prerequisite beyond transaction submission: a reorg must occur with the attacker's high-cycle transactions in the detached blocks. This can happen via:
1. **Natural reorgs** — even 1–2 block reorgs occur on mainnet without any attacker. If high-cycle transactions were included in those blocks, the vulnerability triggers automatically.
2. **Attacker with minimal hashpower** — a small competing chain (not majority hashpower) is sufficient to detach 1–2 blocks.

The attacker does **not** need majority hashpower. Submitting near-max-cycle transactions is unprivileged and fee-gated only.

---

### Recommendation

Release the write lock before re-verifying detached transactions. The pattern should be:
1. Acquire write lock → update pool state (remove attached, evict detached) → release write lock.
2. Re-verify detached transactions outside the write lock (using the existing `verify_with_pause` / chunked path with `command_rx`).
3. Acquire write lock again → insert verified entries.

Alternatively, use the `command_rx`-based `verify_with_pause` path (lines 102–115 of `util.rs`) in `readd_detached_tx` so verification is chunked and yields between cycles, allowing the async runtime to interleave other work even if the lock must be held. [5](#0-4) 

---

### Proof of Concept

1. Construct K blocks each containing M transactions with scripts that consume ~70M cycles.
2. Submit these transactions; let honest miners include them.
3. Trigger a reorg of depth K (or wait for a natural one).
4. Observe that `update_tx_pool_for_reorg` → `readd_detached_tx` holds the write lock for approximately K × M × T_verify seconds.
5. During this window, all `send_transaction` and `get_block_template` RPC calls time out.

The call chain is: `update_tx_pool_for_reorg` (holds write lock) → `readd_detached_tx` → `verify_rtx(command_rx=None)` → `block_in_place(ContextualTransactionVerifier::verify(70_000_000, false))`. [6](#0-5)

    async fn readd_detached_tx(
        &self,
        tx_pool: &mut TxPool,
        txs: Vec<TransactionView>,
        fetched_cache: HashMap<Byte32, CacheEntry>,
    ) {
        let max_cycles = self.tx_pool_config.max_tx_verify_cycles;
        for tx in txs {
            let tx_size = tx.data().serialized_size_in_block();
            let tx_hash = tx.hash();
            if let Ok((rtx, status)) = resolve_tx(tx_pool, tx_pool.snapshot(), tx, false)
                && let Ok(fee) = check_tx_fee(tx_pool, tx_pool.snapshot(), &rtx, tx_size)
            {
                let verify_cache = fetched_cache.get(&tx_hash).cloned();
                let snapshot = tx_pool.cloned_snapshot();
                let tip_header = snapshot.tip_header();
                let tx_env = Arc::new(status.with_env(tip_header));
                if let Ok(verified) = verify_rtx(
                    snapshot,
                    Arc::clone(&rtx),
                    tx_env,
                    &verify_cache,
                    max_cycles,
                    None,
                )
                .await
                {
                    let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
                    if let Err(e) = _submit_entry(tx_pool, status, entry, &self.callbacks) {
                        error!("readd_detached_tx submit_entry {} error {}", tx_hash, e);
                    } else {
                        debug!("readd_detached_tx submit_entry {}", tx_hash);
                    }
                }
            }
        }
    }
