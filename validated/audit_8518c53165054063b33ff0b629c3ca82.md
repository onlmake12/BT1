### Title
Subscription Manager Hardcoded to `msg.sender` in `createSubscription` with No Transfer Mechanism — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.createSubscription()`, the subscription manager role is permanently assigned to `msg.sender` with no option to specify a different address and no mechanism to transfer the role afterward. This is the direct Pyth analog to the reported `tx.origin` pattern: an important privileged identity is derived from the transaction sender rather than being a configurable input, breaking compatibility with factory contracts, smart contract wallets, and other contract-based callers.

---

### Finding Description

`Scheduler.createSubscription()` unconditionally sets the subscription manager to `msg.sender`:

```solidity
// Map subscription ID to manager
_state.subscriptionManager[subscriptionId] = msg.sender;
``` [1](#0-0) 

The `onlyManager` modifier then gates all privileged subscription operations — `updateSubscription` and `withdrawFunds` — exclusively to this address:

```solidity
modifier onlyManager(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] != msg.sender) {
        revert SchedulerErrors.Unauthorized();
    }
    _;
}
``` [2](#0-1) 

`withdrawFunds` sends ETH back to `msg.sender` (not to a configurable recipient), compounding the issue:

```solidity
(bool sent, ) = msg.sender.call{value: amount}("");
``` [3](#0-2) 

A search of the entire `Scheduler.sol` confirms there is **no `transferManager`, `setManager`, or equivalent function** anywhere in the contract. Once set at creation time, the manager address is immutable.

---

### Impact Explanation

If a factory contract, a Gnosis Safe, or an Account Abstraction wallet calls `createSubscription`, that contract address becomes the permanent manager. If the factory contract:

- does not implement forwarding for `updateSubscription` / `withdrawFunds`, or
- is later upgraded to a new address (breaking the old address), or
- is self-destructed,

then the subscription's ETH balance is **permanently locked** in the `SchedulerUpgradeable` contract with no recovery path. The `isPermanent` flag makes this worse: permanent subscriptions explicitly prohibit withdrawals, but even non-permanent subscriptions are affected because the manager address cannot be changed.

The `withdrawFunds` function also sends funds to `msg.sender` rather than to a configurable recipient, meaning even if the manager contract can call `withdrawFunds`, the ETH goes to the calling contract rather than to the intended human beneficiary.

---

### Likelihood Explanation

Factory-pattern subscription creation is a standard DeFi pattern. Any protocol that wraps `createSubscription` in a factory or router (e.g., to batch-create subscriptions or abstract gas payment) will trigger this issue. Smart contract wallets (Gnosis Safe, ERC-4337 AA) are increasingly common and would set the Safe/AA contract as manager — which works as long as the Safe remains operational, but provides no fallback if the Safe is bricked or the threshold signers are lost.

---

### Recommendation

1. Add an optional `manager` parameter to `createSubscription` so callers can specify a different manager address (defaulting to `msg.sender` if not provided):

```solidity
function createSubscription(
    SchedulerStructs.SubscriptionParams memory subscriptionParams,
    address manager  // new parameter; use msg.sender if address(0)
) external payable override returns (uint256 subscriptionId) {
    address effectiveManager = (manager == address(0)) ? msg.sender : manager;
    _state.subscriptionManager[subscriptionId] = effectiveManager;
    ...
}
```

2. Add a `transferManager(uint256 subscriptionId, address newManager)` function restricted to `onlyManager`, following a two-step propose/accept pattern to prevent accidental lockout.

3. Add a `recipient` parameter to `withdrawFunds` so the manager can direct funds to an address other than `msg.sender`.

---

### Proof of Concept

1. Deploy a factory contract `SubscriptionFactory` that calls `scheduler.createSubscription{value: ...}(params)`.
2. The factory becomes `subscriptionManager[id]`.
3. If `SubscriptionFactory` has no `withdrawFunds` forwarding function, call `scheduler.withdrawFunds(id, balance)` directly — it reverts with `Unauthorized` because `msg.sender` is the EOA, not the factory.
4. The deposited ETH is permanently locked. No admin override exists in `SchedulerUpgradeable` to recover it. [4](#0-3) [5](#0-4)

        SchedulerStructs.SubscriptionParams memory subscriptionParams
    ) external payable override returns (uint256 subscriptionId) {
        _validateSubscriptionParams(subscriptionParams);

        // Calculate minimum balance required for this subscription
        uint256 minimumBalance = this.getMinimumBalance(
            uint8(subscriptionParams.priceIds.length)
        );

        // Ensure enough funds were provided
        if (msg.value < minimumBalance) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Check deposit limit for permanent subscriptions
        if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        // Set subscription to active
        subscriptionParams.isActive = true;

        subscriptionId = _state.subscriptionNumber++;

        // Store the subscription parameters
        _state.subscriptionParams[subscriptionId] = subscriptionParams;

        // Initialize subscription status

        status.priceLastUpdatedAt = 0;
        status.balanceInWei = msg.value;
        status.totalUpdates = 0;
        status.totalSpent = 0;

        _state.subscriptionManager[subscriptionId] = msg.sender;

        _addToActiveSubscriptions(subscriptionId);

        emit SubscriptionCreated(subscriptionId, msg.sender);

    }

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
