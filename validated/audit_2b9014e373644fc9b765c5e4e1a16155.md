### Title
Cancelled Network Alert Replay via Bounded `cancel_filter` LRU Eviction — (`util/network-alert/src/notifier.rs`)

---

### Summary

The `Notifier` in the CKB network-alert subsystem tracks cancelled alert IDs in a bounded `LruCache<u32, ()>` capped at 128 entries (`CANCEL_FILTER_SIZE`). Once 128 subsequent cancellation events fill the cache, the oldest cancelled alert ID is silently evicted. After eviction, `has_received()` returns `false` for that ID, and any unprivileged P2P peer holding the original signed alert bytes can replay it — causing the cancelled alert to be re-accepted, re-added to `noticed_alerts`, and re-broadcast across the entire network.

---

### Finding Description

**Root cause — bounded cancel tracking:**

`CANCEL_FILTER_SIZE` is set to 128: [1](#0-0) 

The `cancel_filter` field is an `LruCache<u32, ()>`: [2](#0-1) 

It is initialized with that fixed capacity: [3](#0-2) 

**How cancellation is recorded:**

When an alert with `cancel > 0` is received, `cancel()` is called, which inserts the cancelled ID into the LRU cache: [4](#0-3) 

**How replay is prevented (and how it fails):**

`has_received()` checks both `received_alerts` and `cancel_filter`: [5](#0-4) 

Once 128 subsequent `cancel_filter.put(...)` calls are made (from 128 new alerts each carrying a distinct `cancel` value), the LRU cache evicts the oldest entry. After eviction, `has_received(evicted_id)` returns `false`. The original cancelled alert can now pass the duplicate-check gate.

**The P2P receive path does not re-check cancellation after the gate:** [6](#0-5) 

The flow is:
1. `has_received(alert_id)` → `false` (evicted) → does not return early
2. `verify_signatures(&alert)` → `Ok` (original signatures are still cryptographically valid)
3. Alert is broadcast to all connected peers
4. `notifier.add(&alert)` → alert is re-inserted into `received_alerts` and `noticed_alerts`

**`clear_expired_alerts` does not protect `cancel_filter`:**

The expiry cleanup removes entries from `received_alerts` and `noticed_alerts` but never touches `cancel_filter`: [7](#0-6) 

This means the only protection against replay of a cancelled-but-not-expired alert is the LRU cache, which is exhaustible.

**`cancel_filter.contains` does not refresh LRU order:**

In the `lru` crate, `contains` is a peek — it does not update recency. So a cancelled alert ID that is frequently checked via `has_received` but was inserted long ago will still be evicted before a recently inserted entry that has never been checked.

---

### Impact Explanation

Once a cancelled alert ID is evicted from `cancel_filter`, any peer that retained the original signed alert bytes (which were broadcast over P2P and are publicly available) can replay the alert to any node. The replayed alert:

- Passes signature verification (the original 2-of-4 Nervos Foundation signatures remain valid)
- Is re-added to `noticed_alerts` and surfaced to node operators via `get_blockchain_info`
- Is re-broadcast to all connected peers, propagating the cancelled alert network-wide

The practical consequence is that node operators see a cancelled alert as active — for example, a "please upgrade, critical bug" warning that the Foundation already retracted. This can cause unnecessary operator actions, erode trust in the alert system, and undermine the Foundation's ability to retract alerts.

---

### Likelihood Explanation

The prerequisite is that 128 valid signed alerts carrying distinct `cancel` values are processed by a node since its last restart. The Nervos Foundation controls alert issuance (2-of-4 multisig), so the attacker cannot accelerate this condition. However:

- The attacker is entirely **passive** until the eviction occurs — they only need to retain the original alert bytes received over P2P.
- The `cancel_filter` is **in-memory only** and resets on every node restart. After a restart, the counter resets to zero, but so does `received_alerts`, meaning the node is also unaware of the cancellation until it receives the cancelling alert from peers. If the cancelling alert has already expired and been cleared from peers' `received_alerts`, the restarted node will never receive it from honest peers, and the attacker can replay the original alert immediately — with zero LRU eviction required.
- Over a long-running node lifecycle, the Nervos Foundation issuing 128 alerts (including cancellations) is plausible.

Likelihood: **Low** for the LRU-eviction path; **Moderate** for the post-restart path when the cancelling alert has expired.

---

### Recommendation

1. **Replace `LruCache` with an unbounded `HashSet<u32>`** for `cancel_filter`. Cancelled alert IDs are a small, bounded set in practice (alert IDs are `u32` but the Foundation issues very few alerts). An unbounded set eliminates the eviction window entirely.

2. **Persist `cancel_filter` to disk** (e.g., in RocksDB alongside other node state) so that node restarts do not lose cancellation history.

3. **In `has_received`, use `get` instead of `contains`** on the LRU cache so that active lookups refresh recency and prevent premature eviction of in-use entries.

4. **In `clear_expired_alerts`, also prune `cancel_filter`** entries whose corresponding cancelling alert has expired — but only after ensuring the cancelling alert's `notice_until` is at least as long as the original alert's `notice_until` (enforce this at alert construction time).

---

### Proof of Concept

```
1. Foundation broadcasts Alert A: id=1, notice_until=now+365days, cancel=0
   → All nodes accept it; cancel_filter is empty; received_alerts[1] = Alert A

2. Foundation broadcasts Alert B: id=2, notice_until=now+365days, cancel=1
   → All nodes call cancel(1): cancel_filter = {1: ()}, received_alerts removes key 1

3. Foundation (or natural protocol activity) broadcasts 128 more alerts,
   each with a distinct cancel value (e.g., cancel=1000, 1001, ..., 1127)
   → cancel_filter fills to capacity; LRU evicts id=1 (oldest entry)
   → cancel_filter no longer contains 1

4. Attacker (any P2P peer that stored Alert A bytes) sends Alert A to a target node:
   - alert_relayer::received() is called
   - has_received(1) → false  (not in received_alerts, not in cancel_filter)
   - verify_signatures(&alert_a) → Ok  (original sigs still valid)
   - Alert A is broadcast to all peers
   - notifier.add(&alert_a) → received_alerts[1] = Alert A; noticed_alerts includes Alert A

5. Target node and all downstream peers now display the cancelled Alert A as active.
```

    pub fn add(&mut self, alert: &Alert) {
        let alert_id = alert.raw().id().into();
        let alert_cancel = alert.raw().cancel().into();
        if self.has_received(alert_id) {
            return;
        }
        // checkout cancel_id
        if alert_cancel > 0 {
            self.cancel(alert_cancel);
        }
        // add to received alerts
        self.received_alerts.insert(alert_id, alert.clone());

        // check conditions, figure out do we need to notice this alert
        if !self.is_version_effective(alert) {
            debug!("Received a version ineffective alert {:?}", alert);
            return;
        }

        if self.noticed_alerts.contains(alert) {
            return;
        }
        self.notify_controller.notify_network_alert(alert.clone());
        self.noticed_alerts.push(alert.clone());
        // sort by priority
        self.noticed_alerts.sort_by_key(|a| {
            let priority: u32 = a.raw().priority().into();
            u32::MAX - priority
        });
    }

    /// Cancel alert id
    pub fn cancel(&mut self, cancel_id: u32) {
        self.cancel_filter.put(cancel_id, ());
        self.received_alerts.remove(&cancel_id);
        self.noticed_alerts.retain(|a| {
            let id: u32 = a.raw().id().into();
            id != cancel_id
        });
    }

    /// Clear all expired alerts
    pub fn clear_expired_alerts(&mut self, now: u64) {
        self.received_alerts.retain(|_id, alert| {
            let notice_until: u64 = alert.raw().notice_until().into();
            notice_until > now
        });
        self.noticed_alerts.retain(|a| {
            let notice_until: u64 = a.raw().notice_until().into();
            notice_until > now
        });
    }

    /// Whether id received
    pub fn has_received(&self, id: u32) -> bool {
        self.received_alerts.contains_key(&id) || self.cancel_filter.contains(&id)
    }
