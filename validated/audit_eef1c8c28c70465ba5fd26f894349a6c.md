Audit Report

## Title
`DaoScriptSizeVerifier` and `CapacityVerifier` Bypass via `hash_type = "data"` DAO Type Script — (`File: verification/src/transaction_verifier.rs`)

## Summary

`cell_uses_dao_type_script` hard-codes a check for `hash_type == ScriptHashType::Type`, so any DAO cell whose type script references the DAO binary via `hash_type = "data"` (using the publicly-known `CODE_HASH_DAO` data hash) is invisible to both `DaoScriptSizeVerifier` and `CapacityVerifier`. This allows a transaction sender to submit a DAO withdrawal whose output lock script differs in serialised size from the deposit cell's lock script, bypassing the sole node-level guard against the known DAO lock-script-size mismatch weakness and enabling extraction of capacity beyond what the DAO interest formula permits.

## Finding Description

`cell_uses_dao_type_script` at lines 525–534 of `verification/src/transaction_verifier.rs` requires both `hash_type == ScriptHashType::Type` AND `code_hash == dao_type_hash`:

```rust
fn cell_uses_dao_type_script(cell_output: &CellOutput, dao_type_hash: &Byte32) -> bool {
    cell_output
        .type_()
        .to_opt()
        .map(|t| {
            Into::<u8>::into(t.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
                && &t.code_hash() == dao_type_hash
        })
        .unwrap_or(false)
}
```

A cell whose type script uses `{ code_hash: CODE_HASH_DAO, hash_type: "data" }` resolves and executes the identical DAO binary via CKB's data-hash resolution path, but this function returns `false` for it.

**`DaoScriptSizeVerifier::verify` (lines 855–858):** uses this function as its sole gate; when it returns `false`, the lock-script-size equality check is skipped entirely.

**`CapacityVerifier::valid_dao_withdraw_transaction` (lines 517–522):** uses this function to decide whether to suppress `OutputsSumOverflow`; when it returns `false`, the suppression is not applied, but the attacker simply sets `outputs_capacity ≤ inputs_capacity` to pass the standard check.

**`ScriptHashTypeVerifier` (lines 796–814):** only iterates `output.lock().hash_type()`, never `output.type_().hash_type()`. Furthermore, `ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` explicitly includes `0u8` (Data), `2u8` (Data1), and `4u8` (Data2), so a type script with `hash_type = "data"` is not blocked at the non-contextual verification stage.

**Exploit flow:**
1. Attacker deposits into DAO using `{ code_hash: CODE_HASH_DAO, hash_type: 0 (data), args: 0x }` as the type script and a large lock script. The genesis DAO cell is included as a cell dep; the DAO binary is resolved by data hash and accepts the deposit.
2. Attacker constructs a withdrawal with the same `hash_type = "data"` DAO type script but a *smaller* lock script, reducing occupied capacity. `outputs_capacity ≤ inputs_capacity`.
3. `cell_uses_dao_type_script` returns `false` for both cells → `DaoScriptSizeVerifier` skips the pair → lock-script-size mismatch is not caught.
4. `valid_dao_withdraw_transaction` returns `false` → `OutputsSumOverflow` suppression not triggered → standard capacity check passes because `outputs ≤ inputs`.
5. DAO script executes via data-hash resolution, validates the withdrawal without independently checking lock script sizes (the verifier is explicitly documented as "a temporary solution till Nervos DAO script can be properly upgraded").
6. Transaction accepted; withdrawal cell has smaller occupied capacity than deposit cell, freeing shannons beyond the DAO interest entitlement.

## Impact Explanation

`DaoScriptSizeVerifier` is the **only** consensus-enforced guard against the known DAO lock-script-size mismatch vulnerability. Bypassing it allows capacity extraction beyond what the DAO interest formula should permit, constituting incorrect behavior of a system script protection mechanism. This maps to **High — Incorrect implementation or behavior of CKB-VM or system scripts (10001–15000 points)**.

## Likelihood Explanation

`CODE_HASH_DAO` is a public constant embedded in the chain spec and importable from `ckb_resource`. Any unprivileged transaction sender can construct the bypass without miner collusion, key material, or privileged access. The technique requires knowledge of CKB's multi-hash-type script resolution (documented in the RFC) but is straightforwardly constructable by a motivated attacker. It is repeatable and not dependent on race conditions or external state.

## Recommendation

Extend `cell_uses_dao_type_script` to also match cells whose type script uses `hash_type = "data"`, `"data1"`, or `"data2"` with `code_hash == CODE_HASH_DAO` (the data hash of the DAO binary). Pass the DAO data hash as an additional parameter and add a second branch:

```rust
fn cell_uses_dao_type_script(
    cell_output: &CellOutput,
    dao_type_hash: &Byte32,
    dao_data_hash: &Byte32,
) -> bool {
    cell_output
        .type_()
        .to_opt()
        .map(|t| {
            let ht: u8 = t.hash_type().into();
            (ht == ScriptHashType::Type as u8 && &t.code_hash() == dao_type_hash)
                || (matches!(ht, DATA | DATA1 | DATA2) && &t.code_hash() == dao_data_hash)
        })
        .unwrap_or(false)
}
```

Alternatively, resolve the script binary from the cell dep and compare the raw data hash directly, independent of the `hash_type` field.

## Proof of Concept

1. **Deposit transaction**: output with `type_script = { code_hash: CODE_HASH_DAO, hash_type: 0 (data), args: 0x }`, `data = 0x0000000000000000`, and a large lock script (e.g., 200 bytes). Include the genesis DAO cell as a cell dep by out-point. The DAO script executes via data-hash resolution and accepts the deposit.

2. **Withdrawal transaction (phase 1 → phase 2)**: input is the deposit cell above; output has the same `hash_type = "data"` DAO type script but a *smaller* lock script (e.g., 33 bytes). Set `outputs_capacity ≤ inputs_capacity`.

3. **Node verification trace**:
   - `ScriptHashTypeVerifier`: only checks lock script hash_type (0 = Data is in `ENABLED_SCRIPT_HASH_TYPE`) → passes.
   - `cell_uses_dao_type_script` → `false` for both cells → `DaoScriptSizeVerifier` skips → **no lock-script-size check**.
   - `valid_dao_withdraw_transaction` → `false` → `OutputsSumOverflow` suppression not applied → standard check passes because `outputs ≤ inputs`.
   - DAO script executes via data-hash resolution → validates withdrawal → passes.
   - Transaction accepted into a block.

4. **Result**: withdrawal cell has smaller occupied capacity than deposit cell; attacker recovers shannons that should have remained locked, beyond the DAO interest entitlement.
