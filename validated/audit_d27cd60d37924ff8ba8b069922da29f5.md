Audit Report

## Title
`RecentReject::put()` Never Persists Incremented `total_keys_num`, Permanently Disabling `shrink()` and Enabling Unbounded Reject-DB Growth — (`tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put()`, the incremented key count is stored only in a local variable and never written back to `self.total_keys_num`. Because the field never advances past its initialization value, the guard condition `total_keys_num > self.count_limit` is always false, `shrink()` is never invoked, and the reject RocksDB instance grows without bound. An unauthenticated remote peer can exploit this by continuously submitting transactions that fail validation, exhausting disk space and halting the node.

## Finding Description
`RecentReject::put()` (lines 62–69 of `tx-pool/src/component/recent_reject.rs`) computes `self.total_keys_num.checked_add(1)` into a local `total_keys_num` but never assigns the result back to `self.total_keys_num`. Consequently:

- At construction (`build()`, lines 39–52), `self.total_keys_num` is set from `estimate_num_keys_cf`, which returns 0 for a fresh DB.
- On every subsequent call to `put()`, the expression evaluates to `0 + 1 = 1`, the guard `1 > count_limit` (default 10,000,000) is false, and `shrink()` is skipped.
- `shrink()` (lines 104–113) is the only other write site for `self.total_keys_num`; since it is never called, the field remains 0 for the lifetime of the process.
- The underlying `DBWithTTL` (RocksDB with TTL) accumulates entries indefinitely. TTL-driven compaction reclaims space only lazily and cannot keep pace with a sustained write attack.

The reachable call path from an unauthenticated peer is: P2P relay / `send_transaction` RPC → `process_tx` → `after_process` (lines 522–524 of `tx-pool/src/process.rs`) → `put_recent_reject` (lines 428–438) → `RecentReject::put()`. Any transaction satisfying `reject.should_recorded()` writes a JSON-serialized `PoolTransactionReject` record to the DB.

## Impact Explanation
Unbounded growth of the reject DB leads to full disk exhaustion. When the disk is full, RocksDB cannot perform write-ahead log flushes or compactions for any DB instance on the node (chain DB, tx-pool DB, reject DB). This halts block processing and chain sync entirely — a complete node crash. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The exploit requires no privileges, no proof-of-work, no key material, and no victim interaction. Any peer on the P2P network or any caller of the `send_transaction` RPC can submit transactions that fail validation (double-spend, insufficient fee, malformed script, etc.). The bug is deterministic and reproducible on every fresh node. The attacker controls submission rate and can sustain the attack indefinitely. The only partial mitigation — RocksDB TTL compaction — is asynchronous and cannot bound disk usage under a sustained write load.

## Recommendation
Add the missing assignment in the non-shrink branch of `put()`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // ← add this line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

No change is needed in `shrink()` itself, as it already re-estimates and assigns `self.total_keys_num` after dropping a shard.

## Proof of Concept
```rust
let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
let count_limit = 10u64;
let mut rr = RecentReject::build(tmp_dir.path(), 2, count_limit, -1).unwrap();

for i in 0..100u64 {
    let key = Byte32::new(blake2b_256(i.to_le_bytes()));
    rr.put(&key, Reject::Malformed(i.to_string(), Default::default())).unwrap();
}

// total_keys_num was never updated; shrink() was never called
assert_eq!(rr.total_keys_num, 0);
// DB holds ~100 entries despite count_limit = 10
```

After 100 insertions with `count_limit = 10`, `total_keys_num` remains 0 and `shrink()` has never fired. Scaling to millions of rejected transactions exhausts disk space, crashing the node.
