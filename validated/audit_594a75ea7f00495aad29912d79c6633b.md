### Title
Ping-RTT Eviction Protection Defeated by `as_secs()` Truncation — (`network/src/peer_registry.rs`)

### Summary

`try_evict_inbound_peer` uses `Duration::as_secs()` to compare peer ping RTTs in the ping-based protection step. Because `as_secs()` truncates sub-second precision, every peer with a ping RTT between 0 ms and 999 ms maps to the same sort key of `0`. This makes the ping protection step completely ineffective for all typical internet peers, allowing an attacker to ensure its high-latency (but still sub-second) peers are treated identically to genuinely low-latency honest peers.

### Finding Description

In `try_evict_inbound_peer`, the first `sort_then_drop` call is intended to protect the `EVICTION_PROTECT_PEERS` (= 8) peers with the lowest ping from eviction: [1](#0-0) 

The comparator extracts the sort key as: [2](#0-1) 

`Duration::as_secs()` returns only the whole-second component, discarding all sub-second precision. Any `ping_rtt` value in `[0 ms, 999 ms]` maps to `0u64`. The sort therefore produces no meaningful ordering when all peers have sub-second pings — which is the normal case for internet-connected peers.

`sort_then_drop` uses `Vec::sort_by`, which is a stable sort. However, `candidate_peers` is built from `self.peers.values()` over a `HashMap`: [3](#0-2) 

`HashMap` iteration order is non-deterministic (randomized per-process by the hash seed). When all sort keys are equal, the stable sort preserves this arbitrary order, so the 8 "protected" peers are chosen randomly rather than by merit.

The existing test masks this bug by assigning `Duration::from_secs(0)` to "lowest ping" peers — a whole-second value that survives the truncation unchanged: [4](#0-3) 

### Impact Explanation

An attacker who controls inbound connections can deliberately delay their pong responses to any value in `[1 ms, 999 ms]`. Their peers will receive the same sort key (`0`) as honest peers with 1 ms pings. The ping protection step — explicitly designed to protect peers whose characteristics are "hard for an attacker to simulate or manipulate" — degrades to a random lottery. With 8 attacker peers and 8 honest peers all mapping to `0`, each has a 50 % chance of being in the protected set per eviction event. Over repeated eviction cycles (each new inbound connection when slots are full), the attacker can probabilistically displace honest peers.

### Likelihood Explanation

- Any unprivileged peer can connect inbound and control its pong timing.
- Sub-second RTTs are universal on the internet; the broken branch is always taken.
- The attacker needs only to fill the inbound slot table, which is a standard precondition for eclipse attacks.
- No PoW, key material, or privileged access is required.

### Recommendation

Replace `as_secs()` with `as_millis()` (or `as_nanos()`) in both the ping and the recent-message comparators inside `try_evict_inbound_peer`:

```rust
// ping protection
.map(|p| p.as_millis())
.unwrap_or(u128::MAX)

// recent-message protection
.map(|t| now.saturating_duration_since(t).as_millis())
.unwrap_or(u128::MAX)
``` [2](#0-1) [5](#0-4) 

Update the test to use millisecond-granularity values (e.g., `Duration::from_millis(1)`) to catch regressions.
