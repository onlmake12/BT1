### Title
Protocol Fees Permanently Locked in `PriceInfoObject` — No Extraction Path Exists — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary

Every call to `update_single_price_feed` deposits the user's SUI fee into a `PriceInfoObject` shared object via `deposit_fee_coins`. No function anywhere in the production Sui contract set can remove or transfer those coins out. All protocol fees are permanently locked.

### Finding Description

`update_single_price_feed` in `pyth.move` unconditionally calls `price_info::deposit_fee_coins`, which stores the fee `Coin<SUI>` as a dynamic object field under `FEE_STORAGE_KEY = b"fee_storage"` on the `PriceInfoObject`: [1](#0-0) 

`deposit_fee_coins` either adds a new dynamic object field or `coin::join`s into the existing one — there is no corresponding remove path: [2](#0-1) 

A `get_balance` reader exists, confirming the balance accumulates, but no `withdraw_fee_coins` or equivalent function is defined anywhere in `price_info.move`: [3](#0-2) 

A grep across all production Sui `.move` sources for `withdraw_fee`, `FEE_STORAGE_KEY`, and `dynamic_object_field::remove` returns **only** `price_info.move` itself — confirming zero extraction paths exist.

The governance module dispatches five actions (`set_governance_data_source`, `set_data_sources`, `set_update_fee`, `set_stale_price_threshold`, `set_fee_recipient`) — none of which touch the fee balance stored in `PriceInfoObject`: [4](#0-3) 

The `set_fee_recipient` module itself carries a comment that makes the situation explicit:

> "The previous version of the contract sent the fees to a recipient address but this state is not used anymore. This module is kept for backward compatibility." [5](#0-4) 

`State` still stores `fee_recipient_address` and exposes `get_fee_recipient`, but no code path ever uses that address to pull coins from any `PriceInfoObject`: [6](#0-5) [7](#0-6) 

### Impact Explanation

Every SUI fee paid by every user across every price feed update is permanently locked inside the corresponding `PriceInfoObject` shared object. The fee recipient can never retrieve them. This is a direct, permanent loss of all protocol fee revenue — matching the scope criterion of "locking of user/protocol funds."

### Likelihood Explanation

This triggers on every normal, unprivileged call to `update_single_price_feed`. No special attacker action is required; the design flaw activates automatically on every legitimate use of the protocol. Likelihood is **certain** given any real usage.

### Recommendation

1. Add a `public(friend) fun withdraw_fee_coins(price_info_object: &mut PriceInfoObject, ctx: &mut TxContext): Coin<SUI>` to `price_info.move` that calls `dynamic_object_field::remove` on `FEE_STORAGE_KEY`.
2. Add a governance action (e.g., `transfer_fee`) that calls this function and transfers the resulting coin to `state::get_fee_recipient(pyth_state)`.
3. Alternatively, redirect fees at deposit time to a central fee collector on `State` (analogous to Wormhole's `FeeCollector` pattern) rather than scattering them across per-feed shared objects.

### Proof of Concept

```
// 1. Call update_single_price_feed N times with valid fees
for i in 0..N {
    vec = update_single_price_feed(&pyth_state, vec, &mut price_info_object, fee_coin, &clock);
}
// 2. Assert fees accumulated (already confirmed by existing test at pyth.move:817)
assert!(price_info::get_balance(&price_info_object) == N * base_update_fee);
// 3. Attempt any extraction — no public or friend function exists that calls
//    dynamic_object_field::remove or coin::take on FEE_STORAGE_KEY.
//    The fee_recipient_address in State is never used to pull coins.
// => All N * base_update_fee SUI is permanently locked.
```

The existing test at `pyth.move` line 817 already asserts `get_balance == DEFAULT_BASE_UPDATE_FEE` after one update, inadvertently documenting the lock-in: [8](#0-7)

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
