### Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Enables Unbounded Work via Crafted `last_n_blocks` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The `GetLastStateProofProcess::execute` function contains an integer overflow in its rate-limiting guard. An unprivileged light-client peer can send a `GetLastStateProof` message with `last_n_blocks = 0x8000_0000_0000_0000` and `difficulties = []`, causing the guard expression `(last_n_blocks as usize) * 2` to wrap to `0` in release mode. This bypasses the `GET_LAST_STATE_PROOF_LIMIT = 1000` check entirely, after which the server allocates and processes a `Vec` of every block number from `start_block_number` to `last_block_number`, triggering OOM or extreme CPU/IO exhaustion proportional to the full chain height.

---

### Finding Description

**The overflow in the guard:** [1](#0-0) 

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

On a 64-bit system, `usize` is 64 bits. With `last_n_blocks = 0x8000_0000_0000_0000`:

- `last_n_blocks as usize = 0x8000_0000_0000_0000`
- `0x8000_0000_0000_0000 * 2 = 0x1_0000_0000_0000_0000` → wraps to `0` in release mode (Rust's default release profile does not enable `overflow-checks`)
- `0 + 0 = 0 < 1000` → guard passes

The `last_n_blocks` field is a `Uint64` in the molecule schema, so any 64-bit value is wire-valid: [2](#0-1) 

**The unbounded allocation after bypass:** [3](#0-2) 

```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    (sampled_numbers, last_n_numbers)
```

Since `last_n_blocks = 0x8000_0000_0000_0000` exceeds any realistic chain height, the condition is always true, and `last_n_numbers` collects **every block number** from `start_block_number` to `last_block_number`. For a chain with N blocks, this is an O(N) `Vec<u64>` allocation.

**The O(N) DB work:** [4](#0-3) 

`complete_headers` then iterates the entire `block_numbers` vec, performing `get_ancestor`, `get_block`, and `chain_root_mmr` calls for each entry — O(N) database lookups.

---

### Impact Explanation

A single malicious peer message causes the server to:
1. Allocate a `Vec<u64>` of size equal to the full chain height (e.g., 80 MB for 10M blocks)
2. Perform O(N) database reads and MMR root computations

This can exhaust memory (OOM kill) or saturate CPU/IO, crashing or hanging the node process. The light-client protocol is a supported production path.

---

### Likelihood Explanation

- No authentication or PoW required — any peer that connects via the light-client protocol can send this message
- The crafted value `0x8000_0000_0000_0000` is a single field in a molecule-encoded message, trivially constructable
- Rust release builds wrap on overflow by default; no special build configuration is needed by the attacker
- The chain only needs to be long enough to make the allocation/work significant; mainnet CKB has millions of blocks

---

### Recommendation

Replace the unchecked arithmetic with a saturating or checked operation:

```rust
// Option 1: use saturating_mul to prevent wraparound
let total = self.message.difficulties().len()
    .saturating_add((last_n_blocks as usize).saturating_mul(2));
if total > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Or cap `last_n_blocks` before the arithmetic:

```rust
let last_n_blocks = (last_n_blocks as usize)
    .min(constant::GET_LAST_STATE_PROOF_LIMIT);
```

Additionally, add a direct bound on `last_n_blocks` itself before any arithmetic:

```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

### Proof of Concept

```
1. Enable light-client protocol on a CKB node with a chain of N blocks (e.g., N = 1_000_000).
2. Connect as a light-client peer.
3. Send LightClientMessage::GetLastStateProof with:
     last_hash       = <valid main-chain tip hash>
     start_hash      = <any>
     start_number    = 0
     last_n_blocks   = 0x8000_0000_0000_0000
     difficulty_boundary = 1
     difficulties    = []
4. Observe: the guard at line 201 evaluates to 0 < 1000 (passes).
5. Observe: last_n_numbers = (0..N).collect() allocates O(N) memory.
6. Observe: complete_headers performs N DB lookups.
7. Node process crashes (OOM) or becomes unresponsive.
```
