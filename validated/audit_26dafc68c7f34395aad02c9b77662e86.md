### Title
Silent Callback Skip for EOA Requesters in `revealWithCallback` Emits False Success Event — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When an EOA calls `requestWithCallback` or `requestV2`, the Entropy contract accepts the fee and creates a callback-type request. During fulfillment via `revealWithCallback`, the callback is silently skipped (because EOAs have no code), yet the contract emits `EntropyEventsV2.Revealed` with `callbackFailed = false`, falsely reporting success. The EOA loses their fee, the random number is never delivered, and no error is thrown — a direct structural analog to the fCash receiver-mismatch bug.

---

### Finding Description

`requestWithCallback` and `requestV2` are `public payable` functions with no guard preventing EOA callers: [1](#0-0) 

Both delegate to `requestHelper` with `isRequestWithCallback = true`, which sets `req.callbackStatus = CALLBACK_NOT_STARTED` and, when the provider has a non-zero `defaultGasLimit`, sets `req.gasLimit10k` to a non-zero value: [2](#0-1) 

During `revealWithCallback`, the new-path branch fires when `req.gasLimit10k != 0 && callbackStatus == CALLBACK_NOT_STARTED`: [3](#0-2) 

`excessivelySafeCall` to an EOA address always returns `success = true` with empty return data (EVM CALL to a codeless address succeeds unconditionally). The contract then takes the success branch, emits `RevealedWithCallback` and `EntropyEventsV2.Revealed` with `callbackFailed = false`, and clears the request: [4](#0-3) 

The same silent skip occurs in the legacy path (`gasLimit10k == 0`) via an explicit `extcodesize` check that simply omits the callback for EOAs without reverting, again emitting `callbackFailed = false`: [5](#0-4) 

The fee accounting in `requestHelper` confirms that excess value (including the full fee paid by the EOA) is never refunded: [6](#0-5) 

This is explicitly noted in the interface NatDoc but not enforced on-chain: [7](#0-6) 

---

### Impact Explanation

An EOA that calls `requestWithCallback` or `requestV2`:

1. Pays the full fee (provider fee + Pyth protocol fee).
2. Has a request created with `callbackStatus = CALLBACK_NOT_STARTED`.
3. Receives no callback when the request is fulfilled — the random number is generated but immediately discarded.
4. Observes `EntropyEventsV2.Revealed` with `callbackFailed = false`, falsely indicating the callback succeeded.
5. Cannot recover the fee or the random number (request is cleared).

The financial loss is bounded by the per-request fee, but the false `callbackFailed = false` event also corrupts off-chain monitoring and keeper logic that relies on this field to detect failures.

---

### Likelihood Explanation

- `requestWithCallback` and `requestV2` are `public payable` with no EOA guard — any wallet can call them.
- The function name `requestWithCallback` strongly implies a callback will be delivered; there is no on-chain revert to signal otherwise.
- Developers integrating Entropy from an EOA context (e.g., scripts, multisigs, or proxy patterns where `extcodesize == 0` at construction time) will silently lose funds.
- The Fortuna keeper service calls `revealWithCallback` permissionlessly for any pending request, so fulfillment happens automatically without the EOA being able to intercept it. [8](#0-7) 

---

### Recommendation

Add an EOA guard at the entry point of `requestWithCallback` and `requestV2`:

```solidity
function requestWithCallback(...) public payable override returns (uint64) {
    uint256 len;
    assembly { len := extcodesize(caller()) }
    if (len == 0) revert EntropyErrors.RequesterMustBeContract();
    ...
}
```

Alternatively, in the legacy path, emit `callbackFailed = true` (not `false`) when the callback is skipped for an EOA, so that monitoring systems and keepers correctly identify the non-delivery.

---

### Proof of Concept

1. EOA (`user`) calls `requestWithCallback(provider, userContribution)` with `msg.value = getFee(provider)`. Provider has `defaultGasLimit > 0`.
2. `requestHelper` creates a request: `callbackStatus = CALLBACK_NOT_STARTED`, `gasLimit10k > 0`.
3. Fortuna keeper calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
4. New path fires (`gasLimit10k != 0 && callbackStatus == CALLBACK_NOT_STARTED`).
5. `excessivelySafeCall(user, ...)` returns `success = true` (EOA call always succeeds in EVM).
6. `EntropyEventsV2.Revealed` emitted: `callbackFailed = false`.
7. `clearRequest` removes the request from storage.
8. `user` never receives any callback. Fee is not refunded. Random number is lost. No revert or error is surfaced anywhere in the transaction.
