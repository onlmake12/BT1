### Title
Reward Parameters Updated Without Advancing Epoch Accounting in Integrity Pool — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

### Summary

The `update_y` and `update_delegation_fee` instructions in the Pyth Oracle Integrity Staking (OIS) integrity pool update reward-affecting parameters without first advancing/settling the current epoch's reward accounting. This is structurally identical to the SavingsVest bug: parameters that govern reward calculation are changed mid-epoch, causing the new parameters to be retroactively applied to stake that was active under the old parameters. An unprivileged staking user can sandwich a legitimate admin parameter-update transaction to claim inflated rewards or cause other delegators to receive reduced rewards.

---

### Finding Description

The integrity pool tracks per-epoch rewards using `pool_data.last_updated_epoch`, `pool_data.claimable_rewards`, and per-publisher `delegation_fees`. Two instructions update parameters that directly affect reward calculation:

**`update_y`** — updates the global reward rate `y` stored in `pool_config`:

```
accounts: [reward_program_authority (signer), pool_config (writable), system_program]
args:     [y: u64]
```

**`update_delegation_fee`** — updates the per-publisher delegation fee stored in `pool_data.delegation_fees[]`:

```
accounts: [reward_program_authority (signer), pool_data (writable), pool_config (writable), system_program]
args:     [delegation_fee: u64]
```

The critical observation from the IDL is that **`update_y` does not include `pool_data` in its account list at all**. Since `pool_data` holds `last_updated_epoch` and the event log (which snapshots `y` per epoch), `update_y` structurally cannot call `advance_delegation_record` or any equivalent epoch-settlement logic before overwriting `y` in `pool_config`.

The reward formula is `R_p = y · min(S_p, C_p)`. When `y` is changed mid-epoch without first settling the current epoch, the next call to `advance_delegation_record` will compute the entire epoch's reward using the new `y`, regardless of how long the old `y` was in effect.

Similarly, `update_delegation_fee` writes directly to `pool_data.delegation_fees[]` without evidence of first advancing the delegation record for the affected publisher. The delegation fee determines the split of `R_p` between the publisher and delegators: `delegator_reward = R_p · (1 - fee)`. Changing this mid-epoch without settling means the new fee is applied retroactively to the entire unsettled epoch.

---

### Impact Explanation

**Scenario A — `y` decreased (delegators harmed):**
A staking user who delegated at the start of an epoch earns rewards at rate `y_old` for the entire epoch. If `update_y` is called mid-epoch to set `y_new < y_old` without advancing, the next `advance_delegation_record` computes rewards at `y_new` for the full epoch. Delegators lose the difference `(y_old - y_new) · min(S_p, C_p) · (fraction of epoch elapsed)`.

**Scenario B — `y` increased (sandwich attack for profit):**
An attacker observes a pending `update_y` transaction that increases `y`. The attacker front-runs by delegating a large stake. The `update_y` executes without advancing. The attacker then calls `advance_delegation_record` and claims rewards at `y_new` for the full epoch, even though they only staked for a fraction of it. The attacker then undelegates. This is the exact sandwich attack Picodes described for SavingsVest.

**Scenario C — `delegation_fee` increased (delegators harmed):**
If `update_delegation_fee` is called to increase the fee without first settling, delegators lose rewards they earned at the lower fee rate for the portion of the epoch already elapsed.

---

### Likelihood Explanation

- The `reward_program_authority` is expected to update `y` and `delegation_fee` as part of normal protocol operation (e.g., when OP-PIP-103 is reversed and `y` is set to a non-zero value, or when fee parameters are tuned).
- On Solana, pending transactions are visible in the mempool/RPC before finalization, making sandwich attacks feasible.
- The attack requires no special access — any staking user can delegate, observe the pending admin transaction, and call `advance_delegation_record`.
- The `y = 0` current state (per OP-PIP-103) means the impact is latent but will activate the moment `y` is set to a non-zero value via `update_y`.

---

### Recommendation

Both `update_y` and `update_delegation_fee` should advance the pool's epoch accounting before modifying their respective parameters. Specifically:

1. `update_y` should include `pool_data` as a writable account and call the equivalent of `advance_pool_data(current_epoch)` before writing the new `y` to `pool_config`.
2. `update_delegation_fee` should call `advance_delegation_record` for all affected publishers before updating `pool_data.delegation_fees[]`.

This mirrors the mitigation applied to SavingsVest: call `accrue()` before updating sensible parameters.

The reward formula `R_p = y · min(S_p, C_p)` and delegation fee split are documented: [4](#0-3) [5](#0-4) 

**Attack sequence:**
1. Admin submits `update_y(y_new > y_old)` transaction (visible on-chain before finalization).
2. Attacker front-runs: calls `delegate(large_amount)` to publisher pool.
3. `update_y` executes: `pool_config.y = y_new`, no `advance_delegation_record` called.
4. Attacker calls `advance_delegation_record`: full epoch rewards computed at `y_new` for attacker's stake, despite only being delegated for a fraction of the epoch.
5. Attacker undelegates and claims inflated `claimable_rewards`.

> **Note:** The Rust source files for the integrity pool program were not available in the indexed codebase. The root cause is demonstrated structurally via the IDL account lists. If the Rust implementation contains an internal `advance` call not reflected in the IDL accounts, this finding would not apply. A full audit should verify the on-chain program source directly.
