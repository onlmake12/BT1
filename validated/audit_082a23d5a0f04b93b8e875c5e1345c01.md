Audit Report

## Title
VerifyQueue Enforces Hardcoded 256 MB Ceiling Independent of Operator-Configured `max_tx_pool_size`, Enabling Memory Exhaustion DoS — (`tx-pool/src/component/verify_queue.rs`)

## Summary
The CKB tx-pool two-stage admission pipeline stages transactions in a `VerifyQueue` before full script verification. The verify queue enforces a hardcoded 256 MB ceiling (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE`) that is entirely independent of the operator-configured `max_tx_pool_size`. The main pool's `limit_size()` eviction logic has no awareness of the verify queue, and the verify queue's byte consumption is never included in `total_tx_size`. Any unprivileged RPC caller or P2P peer can flood the verify queue up to 256 MB regardless of the operator's memory budget, causing actual node memory consumption to far exceed the configured limit and potentially triggering OOM on memory-constrained nodes.

## Finding Description
**Root cause — hardcoded ceiling never derived from config:**

`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` is a compile-time constant of 256 MB with no connection to `TxPoolConfig::max_tx_pool_size`: [1](#0-0) 

`is_full()` checks only against this constant: [2](#0-1) 

`add_tx()` enforces this check and rejects only when the queue exceeds 256 MB, with no reference to `max_tx_pool_size`: [3](#0-2) 

**`limit_size()` is blind to the verify queue:**

The eviction loop in `pool.rs` only iterates `pool_map` entries with statuses `Pending`, `Gap`, and `Proposed`. It never reads or drains the verify queue: [4](#0-3) 

**`TxPoolInfo` reports verify-queue size as a count, not bytes:**

`total_tx_size` is sourced exclusively from `pool_map.total_tx_size`. The verify queue contributes only `verify_queue_size` (a transaction count), so the byte consumption driving eviction decisions is invisible to operators and monitoring: [5](#0-4) 

**Exploit flow:**
1. Attacker submits up to ~512 transactions each near the 512 KB `TRANSACTION_SIZE_LIMIT`. Transactions need only pass `non_contextual_verify` (size check + not cellbase + basic structure) — inputs need not exist on-chain.
2. Each transaction enters `VerifyQueue::add_tx()`, passes `is_full()` until the 256 MB ceiling is reached, and is queued for async script verification.
3. While awaiting CPU-bound contextual verification, all transactions occupy memory in the verify queue.
4. `limit_size()` is never called on the verify queue; eviction only fires against `pool_map`.
5. Node RSS grows by up to 256 MB regardless of `max_tx_pool_size`. If `max_tx_pool_size = 10 MB`, the verify queue alone consumes 25× the intended budget. [6](#0-5) 

## Impact Explanation
This is a **High** severity issue: **Vulnerabilities which could easily crash a CKB node.**

On a memory-constrained node where an operator has set a low `max_tx_pool_size` (e.g., 10 MB or 50 MB) to bound memory usage, the verify queue alone can consume 256 MB — completely defeating the operator's memory budget. Combined with the main pool (`max_tx_pool_size`) and orphan pool (`DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries), the true memory ceiling is `max_tx_pool_size + 256 MB + orphan overhead`, not `max_tx_pool_size` as the operator intends. On nodes with limited RAM this causes OOM-kill, halting block production and P2P connectivity. Even on well-provisioned nodes, the verify queue acts as a free 256 MB staging buffer any peer can fill, degrading verification throughput for legitimate transactions. [7](#0-6) 

## Likelihood Explanation
The attack requires no privilege: any RPC caller (`send_transaction`) or P2P relay peer can submit transactions. The only gate is `non_contextual_verify`, which checks size ≤ 512 KB and basic structure — inputs need not reference live cells. An attacker can craft ~512 maximum-size transactions (large witness blobs, any lock script) and submit them in rapid succession. Script verification is CPU-bound and asynchronous, so transactions linger in the queue while memory is held. This is a low-cost, low-skill, repeatable attack requiring no keys, no hashpower, and no Sybil capability. [8](#0-7) 

## Recommendation
1. Replace the hardcoded `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` with a value derived from `TxPoolConfig::max_tx_pool_size` (e.g., pass `max_tx_pool_size` into `VerifyQueue::new()` and store it as an instance field), so the verify queue respects the operator's memory budget.
2. Include `verify_queue.total_tx_size` in the combined size check so that `limit_size()` can account for in-flight transactions when deciding whether to accept new ones.
3. Expose `verify_queue_total_tx_size` (bytes, not count) in `TxPoolInfo` so operators and monitoring systems can observe the true memory consumption of the staging area.
4. Alternatively, enforce a per-peer rate limit on verify-queue submissions to bound the rate at which any single source can fill the staging area.

## Proof of Concept
1. Configure a CKB node with `max_tx_pool_size = 10_000_000` (10 MB) in `ckb.toml`.
2. Generate ~512 syntactically valid transactions each close to 512 KB (e.g., large witness data; inputs may reference non-existent cells — they will fail contextual verification later but pass `non_contextual_verify`).
3. Submit all 512 transactions via `send_transaction` RPC in rapid succession.
4. Query `tx_pool_info`: observe `verify_queue_size` grows to ~512 while `total_tx_size` remains near 0 (transactions are still being verified).
5. Observe node RSS grows by ~256 MB — 25× the configured `max_tx_pool_size` — while the operator-visible `total_tx_size` metric shows no alarm signal.
6. For a unit-level test: use `VerifyQueue::set_total_tx_size_for_test()` to confirm `is_full()` only fires at 256 MB regardless of any pool config value. [9](#0-8)

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
