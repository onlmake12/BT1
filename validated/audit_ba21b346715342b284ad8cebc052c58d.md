### Title
`MAX_DEPOSIT_LIMIT` for Permanent Subscriptions Can Be Bypassed via Repeated `addFunds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The `addFunds` function in `Scheduler.sol` checks `msg.value > MAX_DEPOSIT_LIMIT` **before** adding the new deposit to `status.balanceInWei`. Because the check only validates the single incoming deposit rather than the resulting total balance, any subscription manager can bypass the cap by making multiple smaller deposits, each individually below the limit.

### Finding Description

In `addFunds`, the deposit-limit guard reads:

```solidity
// Check deposit limit for permanent subscriptions
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}

status.balanceInWei += msg.value;
``` [1](#0-0) 

The guard tests only the **incoming** `msg.value`, not the **resulting** `status.balanceInWei + msg.value`. A caller who sends exactly `MAX_DEPOSIT_LIMIT` wei on each call will pass the check every time, yet the stored balance grows without bound. The same off-by-one pattern exists in `createSubscription` but is harmless there because the balance starts at zero; the exploitable path is exclusively through `addFunds`. [2](#0-1) 

Permanent subscriptions are explicitly non-withdrawable:

```solidity
// Prevent withdrawals from permanent subscriptions
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [3](#0-2) 

So any ETH deposited beyond `MAX_DEPOSIT_LIMIT` is permanently locked in the contract with no recovery path.

### Impact Explanation

**Impact: Medium.**

A subscription manager can lock an arbitrarily large amount of ETH into a permanent subscription, far exceeding the `MAX_DEPOSIT_LIMIT` invariant. Because permanent subscriptions cannot be updated or withdrawn from, the excess ETH is irrecoverably locked. The protocol's intended cap on permanently locked funds is rendered meaningless. There is no direct theft of other users' funds, but the protocol's accounting invariant is broken and user funds can be permanently over-committed.

### Likelihood Explanation

**Likelihood: Medium.**

The entry point (`addFunds`) is permissionless for the subscription manager and requires no special privileges. Any manager of a permanent subscription can trigger this by sending multiple transactions each just at or below `MAX_DEPOSIT_LIMIT`. The steps are trivial and require no coordination or external conditions.

### Recommendation

Change the guard in `addFunds` to check the **post-deposit** balance:

```diff
- if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
+ if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
      revert SchedulerErrors.MaxDepositLimitExceeded();
  }
``` [1](#0-0) 

### Proof of Concept

1. Deploy `Scheduler` with `MAX_DEPOSIT_LIMIT = X` (e.g. 100 ether).
2. Create a permanent subscription with `msg.value = X` — passes the check in `createSubscription`.
3. Call `addFunds{value: X}(subscriptionId)` — `msg.value == MAX_DEPOSIT_LIMIT`, so `msg.value > MAX_DEPOSIT_LIMIT` is **false**; the check passes and `status.balanceInWei` becomes `2X`.
4. Repeat step 3 `N` times; `status.balanceInWei` reaches `(N+1) * X`, all permanently locked, with no revert ever triggered.
