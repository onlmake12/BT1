Audit Report

## Title
`version_check()` Never Consults rfc0048 Hardfork Predicate, Permanently Blacklists Valid Post-Hardfork Blocks — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::version_check()` unconditionally rejects any block header whose `version` field is non-zero. The hardfork predicate `is_remove_header_version_reservation_rule_enabled`, which is supposed to lift this restriction at epoch 9690 (testnet) / 12,293 (mainnet), is defined in `CKB2023` but is never called anywhere in the production codebase. After the hardfork activation epoch passes, a miner that legitimately sets `header.version = 1` will have every relayed block permanently written to `BLOCK_INVALID` status on all peers running this code, causing a consensus split.

## Finding Description
`version_check()` at `sync/src/synchronizer/headers_process.rs:286–293` contains a bare `header.version() != 0` predicate with no epoch or consensus context:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
``` [1](#0-0) 

It is called unconditionally inside `accept()` after PoW and structural checks pass, and on failure the block hash is written to `BLOCK_INVALID`: [2](#0-1) 

The hardfork predicate intended to gate this check is generated in `util/types/src/core/hardfork/ckb2023.rs`: [3](#0-2) 

A repo-wide grep for `is_remove_header_version_reservation_rule_enabled` returns exactly one hit — its own definition site. By contrast, the sibling rfc_0049 predicate `is_vm_version_2_and_syscalls_3_enabled` is called five times in `script/src/types.rs`, demonstrating the established pattern for consuming hardfork switches. The rfc_0048 call site in `version_check()` is simply absent.

`HeaderVerifier::verify()` — invoked by `non_contextual_check()` before `version_check()` — performs only PoW, number, epoch, and timestamp checks; it performs no version check of its own: [4](#0-3) 

The activation epochs are hardcoded: testnet epoch 9690 (≈2024-10-25) and mainnet epoch 12,293 (≈2025-07-01): [5](#0-4) [6](#0-5) 

Both activation windows have already opened. Any miner that sets `header.version = 1` post-activation will have their blocks permanently blacklisted by every peer running this code.

## Impact Explanation
This is a **Critical — consensus deviation** vulnerability. Nodes running this code will permanently mark as `BLOCK_INVALID` any block whose header version is non-zero, even after the hardfork makes such blocks protocol-valid. Because the rejection is written to persistent `BLOCK_INVALID` status, it cannot be recovered by re-relay on the same peer. Nodes that correctly implement the full hardfork path would accept the block; nodes running this code would not. The result is a hard chain split: the two populations follow different tips, miner rewards on the minority chain are lost, and the network fragments.

## Likelihood Explanation
Both activation epochs have already passed (testnet ~Oct 2024, mainnet ~Jul 2025). The bug is latent only until a single miner sets a non-zero version field — a trivial one-field change that is explicitly permitted by rfc0048. No special attacker capability is required; any block producer can trigger this. The hardfork infrastructure being fully wired at the constant and type layers (but not at the enforcement layer) signals clear developer intent that was never completed.

## Recommendation
`version_check()` must be made hardfork-aware. Pass the `Consensus` object into `HeaderAcceptor` (it is already available at the call sites in `execute()` and `accept_first()`), then replace the bare version predicate with:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    let epoch = self.header.epoch().number();
    if self.header.version() != 0
        && !self.consensus
            .hardfork_switch
            .ckb2023
            .is_remove_header_version_reservation_rule_enabled(epoch)
    {
        state.invalid(Some(ValidationError::Version));
        return Err(());
    }
    Ok(())
}
```

The same guard should be added to any other code path that enforces the version-zero rule.

## Proof of Concept
1. Construct a `HeaderView` with `version = 1` and an epoch number `>= 9690` (testnet) or `>= 12293` (mainnet).
2. Solve PoW so `non_contextual_check()` / `PowVerifier` passes.
3. Call `HeaderAcceptor::accept()`.
4. Observe `version_check()` returns `Err(())` and `shared.insert_block_status(hash, BlockStatus::BLOCK_INVALID)` is executed.
5. Independently call `consensus.hardfork_switch.ckb2023.is_remove_header_version_reservation_rule_enabled(9690)` and confirm it returns `true` — proving the predicate exists and evaluates correctly but is never consulted by `version_check()`.
