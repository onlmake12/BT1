### Title
Inbound Session Bypasses `accept_peer` Ban/Limit Checks via Feeler PeerId Collision — (`network/src/peer_registry.rs`, `network/src/network.rs`)

---

### Summary

`is_feeler` is keyed solely on `PeerId`. When the node initiates a feeler dial to peer X, any inbound connection arriving from the same `PeerId` X — before the feeler session disconnects — is silently treated as the feeler, causing `accept_peer` to be skipped entirely. The inbound session is never inserted into `peer_registry.peers`, bypassing ban checks, inbound-limit enforcement, and all subsequent peer-tracking.

---

### Finding Description

**`feeler_peers` is a `HashMap<PeerId, Flags>`, not keyed by session direction:** [1](#0-0) 

**`add_feeler` inserts by `PeerId` when a feeler dial is initiated:** [2](#0-1) 

**`is_feeler` checks only `PeerId` — no check on session direction (inbound vs. outbound):** [3](#0-2) 

**`SessionOpen` handler: if `is_feeler` returns true, `accept_peer` is unconditionally skipped:** [4](#0-3) 

The `session_context.ty` (inbound vs. outbound) is never consulted before the `is_feeler` branch. The entire `accept_peer` path — including `peer_store.is_addr_banned`, `max_inbound` enforcement, and `self.peers.insert(session_id, peer)` — is bypassed.

**Race window:** `add_feeler` is called immediately after `dial_inner` succeeds: [5](#0-4) 

`remove_feeler` is only called in `Feeler::disconnected`, which fires after the feeler protocol negotiates and the node sends a disconnect: [6](#0-5) 

This window spans the full TCP round-trip + protocol negotiation time — easily several seconds — during which an inbound connection from the same `PeerId` will be misclassified.

---

### Impact Explanation

1. **Ban bypass**: A peer banned via `peer_store.is_addr_banned` can establish an inbound session by timing it to coincide with a feeler dial. The ban check at `peer_registry.rs:109` is never reached.
2. **Ghost session**: The session is never inserted into `peer_registry.peers` (`peer_registry.rs:137`), so it is invisible to `ProtocolTypeCheckerService` (which iterates `reg.peers()`), eviction logic, and all session-tracking code. It persists indefinitely consuming transport resources.
3. **Inbound limit bypass**: `connection_status.non_whitelist_inbound >= self.max_inbound` check (`peer_registry.rs:116`) is never evaluated.

---

### Likelihood Explanation

The attacker needs only two things:
1. Their address in the victim's peer store — achievable passively via the discovery protocol.
2. Timing an inbound TCP connection to arrive after `add_feeler` is called but before `remove_feeler` fires. Since the attacker receives the feeler dial attempt directly, they know exactly when to connect back. The window is deterministic from the attacker's perspective.

No privileged access, no hashpower, no social engineering required.

---

### Recommendation

In the `SessionOpen` handler, gate the `is_feeler` check on the session being outbound:

```rust
// network/src/network.rs, SessionOpen branch
if session_context.ty.is_outbound()
    && self.network_state.with_peer_registry(|reg| reg.is_feeler(&session_context.address))
{
    // feeler path
} else {
    // accept_peer path
}
```

Alternatively, `is_feeler` itself can be changed to require an outbound session type, or `feeler_peers` can be keyed by `SessionId` instead of `PeerId` to prevent cross-session collisions.
