Audit Report

## Title
Rolled-Back `insert_failure_ext` Allows Repeated Full Re-Verification of Invalid Blocks — (`chain/src/verify.rs`)

## Summary

When a block fails contextual verification inside `reconcile_main_chain`, `insert_failure_ext` writes `verified=Some(false)` into a `StoreTransaction` that is never committed on the error path. The block's `BlockExt` in the persistent store retains `verified: None`, causing the fast-reject guard in `verify_block` to be silently skipped on every re-submission. The only compensating guard — `BlockStatus::BLOCK_INVALID` in the in-memory `block_status_map` — is not checked on the full-block relay path (`accept_block`) and is entirely lost across node restarts. An attacker holding a single valid-PoW block that fails contextual verification can force the node to execute the full `ContextualBlockVerifier` pipeline on every re-submission.

## Finding Description

**Root cause — `insert_failure_ext` writes to a transaction that is never committed on failure**

Inside `reconcile_main_chain`, when `ContextualBlockVerifier::verify` returns `Err`, `insert_failure_ext` is called with the shared `txn`: [1](#0-0) 

`insert_failure_ext` sets `ext.verified = Some(false)` and calls `txn.insert_block_ext`: [2](#0-1) 

`reconcile_main_chain` then returns `Err`: [3](#0-2) 

Back in `verify_block`, the call site uses `?`: [4](#0-3) 

`db_txn.commit()` is only reached on the success path: [5](#0-4) 

Because `?` propagates the error, `db_txn` is dropped without commit. The `verified=Some(false)` write is rolled back. The block's `BlockExt` in the persistent store retains `verified: None`.

**Fast-reject guard cannot fire**

The guard that would short-circuit re-verification reads from the committed store: [6](#0-5) 

Since `verified` is `None` (never committed), the `let Some(verified) = ext.verified` pattern fails and the guard is skipped entirely. The block proceeds to full re-verification every time.

**Compensating in-memory guard is incomplete — full-block relay path gap**

After failure, `consume_unverified_blocks` sets `BLOCK_INVALID` in the in-memory map: [7](#0-6) 

The compact-block relay path checks this flag in `contextual_check`: [8](#0-7) 

However, the full-block relay path (`accept_block`) only checks `BLOCK_STORED`, not `BLOCK_INVALID`: [9](#0-8) 

`BLOCK_INVALID` is defined as `1 << 12` while `BLOCK_STORED` is `0x7` — completely disjoint bits: [10](#0-9) 

So `contains_block_status(..., BLOCK_STORED)` returns `false` when the status is `BLOCK_INVALID`, and the block is re-queued for full verification.

**Guard is also lost across restarts**

`BlockStatus::BLOCK_INVALID` lives only in the in-memory `block_status_map`. After a restart, the map is empty. `get_block_status` falls back to `get_block_ext(...).verified`: [11](#0-10) 

Since `verified` was never committed (`None`), and `delete_unverified_block` also removes the block from the store: [12](#0-11) 

`get_block_ext` returns `None`, mapping to `BlockStatus::UNKNOWN`. The block is treated as unseen, and full re-verification is triggered again on re-submission.

## Impact Explanation

An attacker who mines a single valid-PoW block that passes non-contextual checks but fails contextual verification (e.g., wrong cellbase reward, invalid uncle reward, or a maximum-cycle script that deliberately fails) can force the target node to execute the full `ContextualBlockVerifier` pipeline — including CKB-VM script execution up to `max_block_cycles` — on every re-submission via the full-block relay path, and again after every node restart. This causes sustained CPU exhaustion and degrades block-processing throughput. **Impact: High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The PoW cost is paid once; all subsequent re-submissions are essentially free (network bandwidth only).

## Likelihood Explanation

The attacker must produce one valid-PoW block with a deliberately failing contextual check. This is a real but one-time cost. After that, re-submission via the full-block relay path is available to any standard P2P peer with no privileged access. The restart gap is also reachable without any special conditions. Both attack vectors are persistent and repeatable.

## Recommendation

1. **Commit the failure ext in a separate, immediately-committed transaction** before returning the error from `reconcile_main_chain`, so `verified=Some(false)` is durably persisted regardless of the outer transaction's fate.
2. **Check `BLOCK_INVALID` in `accept_block`** (full-block relay path) in addition to the compact-block path, mirroring the guard already present in `compact_block_process.rs`.
3. **Derive `BlockStatus::BLOCK_INVALID` solely from the committed `block_ext.verified`** (or persist it to the database) so it survives node restarts without relying on the in-memory `block_status_map`.

## Proof of Concept

1. Construct a block with valid PoW and valid non-contextual structure, but with an incorrect cellbase reward (fails `RewardVerifier` inside `ContextualBlockVerifier`).
2. Submit the block via the P2P full-block relay path (`accept_block`). Observe full contextual verification runs and fails.
3. Query `get_block_ext(block_hash).verified` — assert it is `None`, not `Some(false)` (confirming the rollback).
4. Re-submit the same block via the full-block relay path. Observe full verification runs again (not fast-rejected by the `ext.verified` guard, and not blocked by the `BLOCK_STORED` check in `accept_block`).
5. Repeat step 4 in a loop; measure CPU usage on the target node to confirm sustained exhaustion.
6. Restart the node; re-submit the block; observe full verification runs again with no memory of prior failure (in-memory `block_status_map` is empty, `block_ext` was deleted by `delete_unverified_block`).

    pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
        match self.block_status_map().get(block_hash) {
            Some(status_ref) => *status_ref.value(),
            None => {
                if self.header_map().contains_key(block_hash) {
                    BlockStatus::HEADER_VALID
                } else {
                    let verified = self
                        .snapshot()
                        .get_block_ext(block_hash)
                        .map(|block_ext| block_ext.verified);
                    match verified {
                        None => BlockStatus::UNKNOWN,
                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,
                    }
                }
            }
        }

pub(crate) fn delete_unverified_block(
    store: &ChainDB,
    block_hash: Byte32,
    block_number: BlockNumber,
    parent_hash: Byte32,
) {
    info!(
        "parent: {}, deleting this block {}-{}",
        parent_hash, block_number, block_hash,
    );

    let db_txn = store.begin_transaction();
    let block_op: Option<BlockView> = db_txn.get_block(&block_hash);
    match block_op {
        Some(block) => {
            if let Err(err) = db_txn.delete_block(&block) {
                error!(
                    "delete block {}-{} failed {:?}",
                    block_number, block_hash, err
                );
                return;
            }
            if let Err(err) = db_txn.commit() {
                error!(
                    "commit delete block {}-{} failed {:?}",
                    block_number, block_hash, err
                );
                return;
            }

            info!(
                "parent: {}, deleted this block {}-{}",
                parent_hash, block_number, block_hash,
            );
        }
        None => {
            error!(
                "want to delete block {}-{}, but it not found in db",
                block_number, block_hash
            );
        }
    }
}
