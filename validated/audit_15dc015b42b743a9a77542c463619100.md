### Title
`check_tx_fee` Uses Transaction Size as Weight Proxy, Allowing High-Cycle Transactions to Bypass True Fee-Rate Enforcement - (File: `tx-pool/src/util.rs`)

### Summary

The `check_tx_fee` function in the CKB transaction pool computes the minimum required fee using only the serialized transaction **size** as the weight denominator, while the actual resource cost of a transaction is determined by a combined **weight** of size and execution cycles. This mirrors the oracle stable-price vulnerability: a fixed/stale metric is used to price a resource, and when the real cost diverges from the checked metric (high-cycle transactions), users can obtain block resources at a significant discount.

### Finding Description

In `tx-pool/src/util.rs`, the admission check for minimum fee rate is:

```rust
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(...)
        .transaction_fee(rtx)...;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    if fee < min_fee {
        return Err(Reject::LowFeeRate(...));
    }
    Ok(fee)
}
``` [1](#0-0) 

The code comment explicitly acknowledges the approximation: `"Theoretically we cannot use size as weight directly to calculate fee_rate"`. The actual weight used for block-level fee-rate statistics and block assembly is computed via `get_transaction_weight(size, cycles)`, which incorporates both serialized size **and** execution cycles:

```rust
let weight = get_transaction_weight(*size as usize, cycles);
if weight > 0 {
    fee_rates.push(FeeRate::calculate(fee, weight).as_u64());
}
``` [2](#0-1) 

The `min_fee_rate` default is 1,000 shannons/KW:

```rust
const DEFAULT_MIN_FEE_RATE: FeeRate = FeeRate::from_u64(1000);
``` [3](#0-2) 

The per-transaction cycle cap is 70,000,000:

```rust
max_tx_verify_cycles = 70_000_000
``` [4](#0-3) 

A transaction with 200 bytes of serialized size but 70,000,000 cycles passes the admission check by paying only `200 * 1000 / 1000 = 200 shannons`, while its true weight (size + cycles/factor) is orders of magnitude larger. The "price" (minimum fee) is pegged to size alone, just as the oracle pegged a stablecoin price to $1 regardless of market reality.

### Impact Explanation

An unprivileged transaction sender can:

1. Deploy a script that performs heavy computation (e.g., a tight loop consuming ~70M cycles).
2. Create cells locked by this script and craft transactions spending them.
3. Submit via the `send_transaction` RPC. The size-only fee check passes with a trivially small fee.
4. The transaction enters the pool and triggers full script execution (`ContextualTransactionVerifier`), consuming significant CPU per verification attempt.
5. At scale, an attacker floods the pool with many such transactions, each paying near-zero fees while consuming maximum verification cycles.
6. Miners who include these transactions receive far less fee per unit of block cycle budget consumed, degrading economic incentives and potentially exhausting the per-block cycle budget (`max_block_cycles`) with low-revenue transactions.

This is directly analogous to the oracle vulnerability: the "price" (fee) is fixed to a stale metric (size), while the real cost (cycles) can be arbitrarily higher, allowing the attacker to obtain block resources at a discount.

### Likelihood Explanation

- **Entry path is fully unprivileged**: any RPC caller can submit transactions via `send_transaction`.
- **No special setup required**: deploying a high-cycle script is a standard CKB operation.
- **Economically rational**: the attacker pays minimal fees while consuming maximum verification and block resources.
- **No consensus change needed**: the vulnerability exists in the current tx-pool admission logic.

### Recommendation

Replace the size-only weight in `check_tx_fee` with the actual transaction weight that incorporates cycles. Since cycles are not known until after script execution, a two-phase approach is appropriate:

1. **Pre-execution**: use size as a lower-bound check (current behavior, acceptable as a first gate).
2. **Post-execution**: after `verify_rtx` returns the actual cycle count, re-check the fee rate against `get_transaction_weight(size, cycles)` and reject if the effective fee rate falls below `min_fee_rate`.

This mirrors how the fee-rate statistics collector already computes weight:

```rust
let weight = get_transaction_weight(*size as usize, cycles);
fee_rates.push(FeeRate::calculate(fee, weight).as_u64());
``` [5](#0-4) 

### Proof of Concept

1. Deploy a CKB script that loops for ~70M cycles (valid, terminates within `max_tx_verify_cycles`).
2. Create a cell locked by this script with minimal data (e.g., 100 bytes total serialized transaction size).
3. Submit the transaction via `send_transaction` RPC with fee = `ceil(100 * 1000 / 1000)` = 100 shannons.
4. Observe: `check_tx_fee` passes because `min_fee = min_fee_rate.fee(100) = 100 shannons ≤ fee`.
5. The transaction enters the pool. `verify_rtx` executes the script, consuming ~70M cycles of CPU.
6. The effective fee rate by weight is `FeeRate::calculate(100_shannons, get_transaction_weight(100, 70_000_000))` — far below `min_fee_rate`.
7. Repeat with many such transactions to exhaust pool verification resources and block cycle budgets at near-zero cost. [6](#0-5)
