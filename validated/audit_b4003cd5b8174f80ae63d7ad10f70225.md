Audit Report

## Title
`RecentReject::put` Never Updates `self.total_keys_num`, Disabling the Count-Limit Guard — (`tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put`, the result of `self.total_keys_num.checked_add(1)` is bound to a local variable `total_keys_num` that is never written back to `self.total_keys_num`. The count-limit guard therefore always evaluates the same stale startup estimate against `count_limit`, so `shrink()` is never triggered and the on-disk `DBWithTTL` grows without bound within the TTL window. An unprivileged peer can exhaust node disk by continuously submitting recordable-rejected transactions.

## Finding Description
In `put()` (lines 62–69):

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // self.total_keys_num is NEVER updated here
} else {
    self.shrink()?;
}
```

`self.total_keys_num` is initialized at startup from RocksDB's `estimate_num_keys_cf` (lines 39–51). For an empty DB this is 0. Every subsequent `put()` call computes `0 + 1 = 1`, checks `1 > count_limit` (false for the default limit of 10,000,000), and returns without shrinking or updating the counter. `self.total_keys_num` is only mutated inside `shrink()` (lines 110–111), which is never reached. The existing unit test (line 39) asserts `recent_reject.total_keys_num < 100`, which passes vacuously because the counter stays at 0 regardless of how many entries are inserted. [1](#0-0) [2](#0-1) [3](#0-2) 

## Impact Explanation
The `recent_reject` RocksDB (`DBWithTTL`) grows without bound across the node's lifetime, bounded only by TTL expiry (default 7 days). The intended invariant — that `keep_rejected_tx_hashes_count` caps disk usage — is completely bypassed. Sustained submission of recordable-rejected transactions exhausts node disk, causing a crash or service disruption. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node*.

## Likelihood Explanation
Any unprivileged peer or RPC caller can trigger this. `should_recorded()` returns `true` for every `Reject` variant except `Duplicated`. The simplest vector is submitting transactions with random non-existent outpoints (`Reject::Resolve(OutPointError::Unknown(...))`), requiring no UTXOs and no fees — only valid CKB transaction serialization. These flow through `after_process` → `put_recent_reject` → `recent_reject.put()`. The default `count_limit` of 10,000,000 can be exceeded within the 7-day TTL window at any sustained submission rate. [4](#0-3) 

## Recommendation
Add `self.total_keys_num = total_keys_num;` in the non-shrink branch of `put()`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // ← add this line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

Also update the unit test to assert that the actual RocksDB key count (via `estimate_total_keys_num`) stays within `count_limit` after many puts, rather than asserting the stale `total_keys_num` field.
