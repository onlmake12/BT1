Audit Report

## Title
Unauthenticated Peer Can Force Victim Node to TCP-Connect to Arbitrary Hosts via Hole-Punching Protocol — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
An attacker with any direct P2P connection to a victim node can inject arbitrary TCP addresses into the victim's `pending_delivered` map by sending `ConnectionRequest` messages with spoofed `from` PeerIds and attacker-chosen `listen_addrs`. The attacker then triggers unbounded spawning of long-lived async tasks that repeatedly attempt TCP connections to arbitrary third-party hosts by sending `ConnectionSync` messages with matching spoofed `from` fields. No authentication or privileged role is required, and the per-key dedup and rate-limit guards are trivially bypassed with fresh random PeerIds, enabling file descriptor exhaustion and memory pressure sufficient to crash the node.

## Finding Description

**Root cause — `pending_delivered` keyed on attacker-controlled content field, no session binding**

In `respond_delivered()`, the dedup guard is keyed solely on `from_peer_id` extracted from message content, not from the actual session identity: [1](#0-0) 

A fresh random `PeerId` per request trivially bypasses this check. The `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)`: [2](#0-1) 

This is equally bypassed with fresh PeerIds. The `forward_rate_limiter` is configured at 1 req/s per key: [3](#0-2) 

The only remaining gate is the per-session `rate_limiter` at 30 req/s: [4](#0-3) 

The `listen_addrs` are filtered only for transport type (TCP + IPv4/IPv6) with no validation that the IP belongs to the sender's observed address: [5](#0-4) 

Attacker-controlled addresses are stored verbatim: [6](#0-5) 

**Step 2 — `ConnectionSync` triggers NAT traversal with no session binding**

`ConnectionSyncProcess` does not accept or store the session ID of the sender: [7](#0-6) 

When `route` is empty and `content.to == self_peer_id`, the node looks up `pending_delivered` by `content.from` — a field the attacker fully controls — and immediately spawns a task running `select_ok` over all stored addresses: [8](#0-7) 

The spawned task runs all `try_nat_traversal` futures concurrently: [9](#0-8) 

There is no check that the session sending `ConnectionSync` is the same session that sent the `ConnectionRequest` with the matching `from` PeerId.

**Step 3 — `try_nat_traversal` creates TCP sockets in a 30-second loop**

Each `try_nat_traversal` future creates a new TCP socket and attempts `connect()` every ~200ms for 30 seconds: [10](#0-9) 

Up to 24 addresses per `ConnectionRequest`: [11](#0-10) 

**`pending_delivered` cleanup only runs every 5 minutes:** [12](#0-11) 

**Resource exhaustion math:** 30 `ConnectionSync`/s (per-session rate limit ceiling) × 30s task lifetime = 900 concurrent spawned tasks. Each task runs up to 24 concurrent `try_nat_traversal` futures, each creating a new TCP socket every ~200ms → ~21,600 concurrent TCP connection attempts, rapidly exhausting the victim's file descriptor limit and async task memory. `pending_delivered` accumulates up to 30 entries/s × 5-minute TIMEOUT = 9,000 entries (up to 216,000 stored addresses) before cleanup.

## Impact Explanation

The victim node becomes a TCP amplification vector sending thousands of SYNs per second to arbitrary third-party IP:port targets. More critically, the unbounded spawning of long-lived async tasks and TCP sockets causes file descriptor exhaustion and memory pressure that can crash the CKB node. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**, and secondarily **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

Any peer that can establish a direct P2P connection to the victim can trigger this. No special role, leaked key, or majority hashpower is needed. The attack requires only two crafted messages per iteration and is repeatable at the rate-limiter ceiling (30 req/s per session). Fresh PeerIds bypass both the 2-minute dedup guard and the `forward_rate_limiter`. The victim's peer ID is publicly discoverable in the P2P network. An attacker can open multiple sessions to multiply the rate further.

## Recommendation

1. **Bind `pending_delivered` to the actual session**: record the `PeerIndex` (session ID) alongside the entry at insert time in `respond_delivered()`, and in `ConnectionSyncProcess::execute()`, verify that the session sending `ConnectionSync` matches the recorded session for the given `from` PeerId.
2. **Validate `listen_addrs` against the sender's observed address**: in `respond_delivered()`, reject addresses whose IP component does not match the remote IP of the sending session (available via `context.session.address`).
3. **Rate-limit `pending_delivered` inserts globally**: cap the total size of `pending_delivered` (e.g., 100 entries) regardless of `from` PeerId, to prevent map flooding with fresh PeerIds that bypass per-key dedup.
4. **Bound the number of concurrently spawned NAT traversal tasks**: track in-flight `try_nat_traversal` task count and reject new `ConnectionSync` triggers when the limit is reached.

## Proof of Concept

```
1. Attacker connects to victim (session S, any valid P2P connection).

2. For N = 1, 2, ..., 30 per second:
   a. Generate FAKE_PEER_ID_N = random PeerId.
   b. Send ConnectionRequest:
        from         = FAKE_PEER_ID_N
        to           = VICTIM_PEER_ID
        listen_addrs = [/ip4/TARGET_IP/tcp/TARGET_PORT/p2p/FAKE_PEER_ID_N]
                       (up to 24 addresses per message)
        route        = []
        max_hops     = 6
   c. Victim calls respond_delivered():
        - pending_delivered.get(FAKE_PEER_ID_N) → None → dedup passes
        - forward_rate_limiter keyed on (FAKE_PEER_ID_N, VICTIM_PEER_ID, item_id) → passes
        - address passes TCP/IPv4 filter
        - pending_delivered.insert(FAKE_PEER_ID_N, ([TARGET_IP:PORT], now))

3. For each FAKE_PEER_ID_N, send ConnectionSync:
        from  = FAKE_PEER_ID_N
        to    = VICTIM_PEER_ID
        route = []
   Victim calls ConnectionSyncProcess::execute():
        - route is empty, self == to
        - listens_info = pending_delivered.get(FAKE_PEER_ID_N) → Some([TARGET_IP:PORT])
        - runtime::spawn(select_ok([try_nat_traversal(bind_addr, TARGET_IP:PORT), ...]))

4. Each spawned task sends ~150 TCP SYNs to TARGET_IP:TARGET_PORT over 30 seconds.
   After ~30 seconds: 900 concurrent tasks × 24 sockets each → file descriptor exhaustion
   → victim node crashes or becomes unresponsive.
```

    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }
