### Title
Rust `DaoCalculator` Reads Full u64 Witness Index While C VM Reads Only Lowest Byte, Causing Consensus Split on DAO Withdrawal — (`File: util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` resolves the deposit-header index from the witness `input_type` field using the **full 8-byte (u64) value**, while the on-chain C VM (`dao.c`) resolves it using only the **lowest byte**. An unprivileged DAO depositor can craft a phase-2 withdrawal transaction with ≥258 `header_deps` and a witness index > 255 (where `lowest_byte(index)` points to the deposit block hash but `full_u64(index)` points to a different hash) to produce a transaction the C VM accepts but the Rust node rejects. This causes a consensus split: the Rust node permanently rejects any block containing such a transaction, while C-VM-based nodes accept it, and the depositor's DAO assets become permanently unwithdrawable through Rust nodes.

---

### Finding Description

**Root cause — `util/dao/src/lib.rs`, lines 91–96:**

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// …
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // ← full u64 used as array index
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
``` [1](#0-0) 

The 8-byte `input_type` witness field is decoded as a little-endian u64 and used directly as the `header_deps` array index. The on-chain `dao.c` script, however, reads only the **lowest byte** of that same 8-byte field when indexing into `header_deps`.

This discrepancy is explicitly documented in the test suite:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
header_deps[1]   = deposit_block.hash();
header_deps[257] = withdraw_block.hash();
// witness input_type = 257  (lowest byte = 1)
``` [2](#0-1) 

After resolving the deposit header, the Rust code performs a block-number consistency check:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [3](#0-2) 

With index 257:
- **Rust** resolves `header_deps[257]` → withdraw block (number 200) → `200 ≠ 100` → **error / reject**
- **C VM** resolves `header_deps[1]` → deposit block (number 100) → `100 == 100` → **accept**

The `DaoCalculator` is consumed by `FeeCalculator` inside the contextual transaction verification pipeline: [4](#0-3) 

and by the tx-pool admission path: [5](#0-4)

### Impact Explanation

A DAO depositor who controls their own withdrawal transaction can:

1. Build a phase-2 withdrawal with ≥258 `header_deps`.
2. Place the deposit block hash at position `p = lowest_byte(index)` and any other hash at position `index` (e.g., the withdraw block hash at 257, deposit block hash at 1).
3. Set the witness `input_type` to `index` (e.g., 257).

The C VM (`dao.c`) executes the DAO type script, resolves the deposit header correctly via the lowest byte, and **accepts** the transaction. The Rust `DaoCalculator` resolves the wrong header, fails the block-number check, and **rejects** the transaction — causing the Rust node to reject any block that contains it.

Consequences:
- **Consensus split**: Rust nodes and C-VM nodes diverge on chain tip.
- **Assets permanently stuck**: The depositor's CKBytes locked in the DAO cell cannot be withdrawn through any Rust node, mirroring the "assets get stuck" impact of the original report.
- **No privileged access required**: Any DAO depositor can trigger this with a self-crafted transaction.

---

### Likelihood Explanation

Any CKB user who has deposited into NervosDAO can construct this transaction unilaterally. The only prerequisite is having ≥1 DAO cell in phase 1 (prepare) state. The crafted transaction is structurally valid (correct molecule encoding, valid `since`, correct output capacity) and will pass all non-contextual checks. The discrepancy only surfaces at the `DaoCalculator` layer. Likelihood is **medium**: it requires deliberate construction but no special privileges, leaked keys, or majority hashpower.

---

### Recommendation

Align the Rust index resolution with the C VM behavior. In `util/dao/src/lib.rs`, after decoding the u64 index, truncate it to its lowest byte before indexing into `header_deps`:

```rust
// Current (full u64):
let header_dep_index = LittleEndian::read_u64(&header_deps_index_data.unwrap());

// Fixed (match C VM — lowest byte only):
let header_dep_index = LittleEndian::read_u64(&header_deps_index_data.unwrap()) as u8 as usize;
``` [6](#0-5) 

Alternatively, add a consensus-level validation that rejects any DAO withdrawal transaction whose witness index exceeds 255, so both Rust and C VM agree on the rejection boundary.

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` already demonstrates the split: [7](#0-6) 

To confirm the consensus split end-to-end:

1. Deploy a DAO deposit on a local devnet and advance to phase 1 (prepare).
2. Construct a phase-2 withdrawal transaction with 258 `header_deps`: dummy hashes at all positions except `header_deps[1]` = deposit block hash and `header_deps[257]` = withdraw block hash.
3. Set `WitnessArgs.input_type` = `257u64.to_le_bytes()`.
4. Submit the transaction directly to a miner (bypassing the tx pool).
5. Observe: the C VM executes the DAO script successfully (deposit header resolved via byte 1); the Rust node's `DaoCalculator` returns `DaoError::InvalidOutPoint` and rejects the block.

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
```
