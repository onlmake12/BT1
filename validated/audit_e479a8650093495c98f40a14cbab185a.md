Looking at the actual code, I need to trace the full attack path through `add_addr` → `check_purge` → `is_connectable` and the phase-2 group logic.

**Tracing `add_addr`:**

`add_addr` hardcodes `last_connected_at_ms=0` for all discovered addresses and calls `check_purge()` before inserting. [1](#0-0) 

**Tracing `is_connectable`:**

For a fresh address with `last_connected_at_ms=0` and `attempts_count=0`, the only non-connectable condition is `attempts_count >= ADDR_MAX_RETRIES (3)`, which is false. So all freshly-flooded addresses return `is_connectable=true`. [2](#0-1) 

**Tracing `check_purge` Phase 1:**

Phase 1 collects addresses where `!is_connectable(now_ms)`. Since all flooded addresses have `attempts_count=0`, none are filtered — `candidate_peers` is empty, and phase 1 removes nothing. [3](#0-2) 

**Tracing `check_purge` Phase 2:**

Phase 2 groups addresses by network group, sorts by group size descending, takes the top `len/2` groups, and only evicts from groups with **more than 4** peers (randomly picks 2). [4](#0-3) 

The critical condition is `if addrs.len() > 4`. If an attacker fills the store with exactly **4 addresses per network group** across **4096 distinct groups** (4096 × 4 = 16384 = `ADDR_COUNT_LIMIT`):
- `len = 4096`, `take(len/2) = take(2048)` groups
- Every group has exactly 4 addresses → `addrs.len() > 4` is false for all
- `candidate_peers` remains empty → `EvictionFailed` is returned [5](#0-4)

**Is the attack path realistic?**

Discovery messages are unauthenticated — any connected peer can advertise arbitrary addresses. `MAX_ADDR_TO_SEND=1000` means ~17 discovery messages suffice to flood 16384 entries. There is no per-session cap on how many addresses a peer can inject into the store. [7](#0-6) 

**Is the impact permanent?**

Not strictly permanent — the node's feeler dialer will eventually exhaust `ADDR_MAX_RETRIES=3` on fake addresses, making them non-connectable and evictable by phase 1. However, the attacker can continuously re-flood via discovery to replenish the store faster than the node can exhaust retries, sustaining the condition. The `EvictionFailed` error propagates out of `add_addr`, causing legitimate peer addresses from DNS seeding, identify protocol, and other discovery peers to be silently dropped. [8](#0-7)

### Title
Peer Store Exhaustion via Unauthenticated Discovery Address Flooding Defeats Both Purge Phases — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
An unprivileged remote peer can permanently fill the CKB peer store with 16384 fake addresses arranged as 4096 network groups of 4, defeating both eviction phases in `check_purge` and causing `EvictionFailed` on every subsequent `add_addr` call, preventing the node from learning new peer addresses.

### Finding Description
`add_addr` inserts discovered addresses with `last_connected_at_ms=0` and `attempts_count=0`. `is_connectable` returns `true` for any address with `attempts_count < ADDR_MAX_RETRIES (3)`, so all freshly-flooded addresses survive phase-1 eviction. Phase-2 only evicts from groups with `> 4` members; an attacker who distributes exactly 4 addresses across 4096 distinct `/16` subnets ensures every group has exactly 4 members, so phase-2 also evicts nothing. `EvictionFailed` is returned, and `add_addr` propagates the error, silently dropping all subsequent legitimate peer address insertions.

### Impact Explanation
The node's peer store is saturated with unverified fake addresses. Legitimate addresses from DNS seeding, identify handshakes, and other discovery peers cannot be stored. Over time, as existing connections drop, the node cannot replenish its outbound peer pool, leading to progressive network isolation. The node remains reachable for inbound connections but loses the ability to discover and connect to new outbound peers.

### Likelihood Explanation
The attack requires only a single inbound or outbound connection to the victim. Discovery messages are unauthenticated and can carry arbitrary addresses. Filling 16384 slots requires ~17 discovery messages (given `MAX_ADDR_TO_SEND=1000`). No special privileges, hashpower, or Sybil capability is needed — a single attacker-controlled node suffices.

### Recommendation
Give unverified addresses (those with `last_connected_at_ms=0`) lower eviction priority than verified ones. One approach: in phase-1, also evict addresses that have `last_connected_at_ms=0` and `attempts_count=0` when the store is full (treating "never tried, never connected" as lowest priority). Alternatively, cap the number of unverified addresses per network group at insertion time, or add a per-session rate limit on how many addresses the discovery protocol will accept.

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
