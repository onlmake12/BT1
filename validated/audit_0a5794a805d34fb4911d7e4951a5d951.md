### Title
`DownloadScheduler::decrease()` reads `task_count` instead of `timeout_count`, causing overly aggressive peer penalization - (File: sync/src/types/mod.rs)

### Summary
In `sync/src/types/mod.rs`, the `DownloadScheduler::decrease()` function is supposed to accumulate slow-response events in `timeout_count` and only decrement `task_count` after more than 2 accumulated events. However, it reads from `self.task_count` instead of `self.timeout_count` when updating the accumulator. Because `task_count` starts at 32 (`INIT_BLOCKS_IN_TRANSIT_PER_PEER`) and is always greater than 2, the threshold check is always satisfied, causing `task_count` to be decremented on **every** slow block response rather than only after 3 or more consecutive slow responses.

### Finding Description

`DownloadScheduler` tracks per-peer block download concurrency. The `task_count` field controls how many blocks can be requested in parallel from a peer, and `timeout_count` is intended to be a counter that accumulates slow-response events before penalizing the peer:

```rust
// sync/src/types/mod.rs
#[derive(Debug, Clone)]
pub struct DownloadScheduler {
    task_count: usize,      // starts at INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32
    timeout_count: usize,   // starts at 0
    hashes: HashSet<BlockNumberAndHash>,
}
``` [1](#0-0) 

The `decrease()` function is called when a block response falls in the `NormalToUpper` or `UpperToMax` time quantile:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: reads task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [2](#0-1) 

The line `self.timeout_count = self.task_count.saturating_add(num)` should be `self.timeout_count = self.timeout_count.saturating_add(num)`. Because `task_count` is initialized to 32 and `num` is 1 or 2, the expression `task_count + num` is always ≥ 3, which is always `> 2`. The accumulation guard is therefore bypassed on every call, and `task_count` is decremented unconditionally on every slow response.

This is called from `remove_by_block()` in the hot path of block download completion:

```rust
TimeQuantile::NormalToUpper => {
    if should_punish { set.decrease(1) }
}
TimeQuantile::UpperToMax => {
    if should_punish { set.decrease(2) }
}
``` [3](#0-2) 

When `task_count` reaches 0, the peer is disconnected in `prune()`:

```rust
download_schedulers.retain(|k, v| {
    if v.task_count == 0 {
        disconnect_list.insert(*k);
        false
    } else { true }
});
``` [4](#0-3) 

### Impact Explanation

The intended design requires **more than 2 accumulated slow responses** before penalizing a peer's `task_count`. The bug causes `task_count` to be decremented on **every single** slow response. Starting from 32, a peer reaches `task_count = 0` (and is disconnected) after only 32 slow responses instead of the intended ~96. This causes:

1. Legitimate peers with variable network latency to be disconnected ~3× faster than the protocol intends.
2. Reduced block download parallelism and degraded sync throughput.
3. A remote sync peer can deliberately send slow responses to force its own disconnection or to exhaust the node's peer slots faster than intended, weakening network connectivity.

### Likelihood Explanation

High. The `remove_by_block()` path is exercised on every completed block download. Any sync peer experiencing normal network jitter (responses in the `NormalToUpper` or `UpperToMax` time range) will trigger `decrease()`. The bug is always active — there is no configuration or flag that bypasses it.

### Recommendation

Change the assignment in `decrease()` to accumulate into `timeout_count` rather than overwrite it with `task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // fix: was self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

### Proof of Concept

1. A sync peer connects and begins serving blocks.
2. The peer's responses consistently fall in the `NormalToUpper` time quantile (between `normal_time` and `low_time` ms).
3. Each completed block download calls `remove_by_block()` → `decrease(1)`.
4. Inside `decrease(1)`: `self.timeout_count = self.task_count.saturating_add(1)`. With `task_count = 32`, `timeout_count` becomes 33. Since 33 > 2, `task_count` is decremented to 31 and `timeout_count` reset to 0.
5. On the next slow response: `timeout_count = 31 + 1 = 32 > 2` → `task_count` decremented to 30.
6. After 32 such responses, `task_count = 0`. On the next `prune()` call, the peer is added to `disconnect_list` and disconnected.
7. The intended behavior would only decrement `task_count` after `timeout_count` accumulates past 2 (i.e., after 3 slow responses), making the peer survive ~96 slow responses before disconnection. [2](#0-1) [5](#0-4)
