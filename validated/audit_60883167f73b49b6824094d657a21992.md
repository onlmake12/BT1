## Title
Stale `evict_key` on Surviving Ancestors After Batch Removal — (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`remove_entry_and_descendants` tears down all link graph records via `remove_entry_links` before calling `remove_entry` on any entry. When `remove_entry` subsequently calls `update_ancestors_index_key`, `calc_ancestors` returns an empty set because the root's record has already been deleted from `self.links.inner`. Surviving in-pool parents of the removed root never receive the `sub_descendant_weight` correction, leaving their `evict_key` permanently inflated for the lifetime of those entries in the pool.

## Finding Description

`remove_entry_and_descendants` (L252–265) runs two sequential passes over `removed_ids`:

```rust
for id in &removed_ids {
    self.remove_entry_links(id);   // pass 1: tears down ALL link records
}
removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()  // pass 2
```

`remove_entry_links` (L418–430) does three things for each entry: removes it from each parent's children set, removes it from each child's parents set, then calls `self.links.remove(id)` — deleting the entry's entire record from `self.links.inner`.

In pass 2, `remove_entry` (L235–250) calls `update_ancestors_index_key` (L242) before doing anything else. `update_ancestors_index_key` (L432–445) calls `self.links.calc_ancestors(&child.proposal_short_id())`, which resolves to `calc_relative_ids` (links.rs L37–50):

```rust
let direct = self
    .inner
    .get(short_id)          // returns None — record already deleted in pass 1
    .map(|link| link.get_direct_ids(relation))
    .cloned()
    .unwrap_or_default();   // empty set
```

`calc_relation_ids` on an empty set returns an empty set. The `for anc_id in &ancestors` loop in `update_ancestors_index_key` (L435) never executes. Surviving parents of the root — which are **not** in `removed_ids` — never have `sub_descendant_weight` called and never have their `evict_key` recomputed via `e.evict_key = e.inner.as_evict_key()` (L442).

`EvictKey` (entry.rs L234–247) is computed as `max(descendants_feerate, own_feerate)` with `descendants_count`. After the batch removal, each surviving parent `P` retains stale inflated values for both fields. `next_evict_entry` (L380–385) iterates `iter_by_evict_key()` ascending; a stale high `evict_key` on `P` places it toward the high end of the eviction index, causing it to be skipped when the pool needs to shed low-value entries.

The comment at L256 acknowledges only half the intent: it prevents redundant descendant-index updates for entries being removed, but simultaneously suppresses the necessary ancestor-index update for entries that are **not** being removed.

## Impact Explanation

**Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism.** The eviction index is permanently corrupted for any surviving ancestor of a batch-removed subtree. Low-fee-rate transactions that should be first candidates for eviction are retained with inflated priority. When the pool is full, legitimate higher-fee transactions may be rejected because the pool appears occupied by entries that should have already been evicted. The corruption persists for the full lifetime of the affected entry in the pool.

## Likelihood Explanation

No attacker capability is required. The trigger is any call to `remove_entry_and_descendants` on a transaction that has at least one in-pool parent not included in the removal batch. The condition is repeatable on every block that confirms a transaction whose pool-descendants are evicted while its pool-ancestors survive.

## Recommendation

Before the batch `remove_entry_links` loop, collect the surviving ancestors of the root (those not in `removed_ids`) and perform the `evict_key` update for each removed entry. Concretely, call `calc_ancestors(root_id)` while the link graph is still intact, filter out entries in `removed_ids`, then call `sub_descendant_weight` and recompute `evict_key` for each surviving ancestor for every entry being removed. Alternatively, restructure `remove_entry_and_descendants` to call `update_ancestors_index_key` for the root entry before the batch `remove_entry_links` loop runs.

## Proof of Concept

Construct a unit test in `tx-pool/src/component/pool_map.rs`:

1. Insert three transactions: grandparent `G` (low fee rate), parent `P` (low fee rate, child of `G`), child `C` (high fee rate, child of `P`).
2. Record `G`'s `evict_key` — it should be inflated by `P`'s and `C`'s high fee rate contribution.
3. Call `pool_map.remove_entry_and_descendants(&P.id)` — removes `P` and `C`.
4. Assert that `G`'s `evict_key` has been updated to reflect only `G`'s own fee rate (i.e., `evict_key.fee_rate == G.own_feerate` and `evict_key.descendants_count == 1`).
5. Without the fix, the assertion fails: `G.evict_key` still contains the inflated values from the now-removed `P` and `C`.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }

                self.links.remove_parent(&child, id);

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
