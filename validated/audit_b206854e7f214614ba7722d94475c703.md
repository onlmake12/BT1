### Title
Unprotected External Callback in Legacy `revealWithCallback` Path Permanently Locks User Funds — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains a legacy code path (entered when `req.gasLimit10k == 0`) that calls the consumer contract's `_entropyCallback` without any error handling. If the consumer contract's callback reverts, the entire `revealWithCallback` transaction reverts. Because the request was already cleared before the callback, the revert rolls back the clearance, leaving the request permanently active but unfulfillable. The user's paid fee is permanently locked with no recovery mechanism.

---

### Finding Description

`revealWithCallback` in `Entropy.sol` branches on whether the request has an explicit gas limit:

```
if (req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED) {
    // NEW SAFE PATH: uses excessivelySafeCall, catches reverts, sets CALLBACK_FAILED state
} else {
    // LEGACY PATH: direct unprotected call
}
```

The **new path** (lines 574–660) uses `excessivelySafeCall` to catch reverts and transitions the request to a `CALLBACK_FAILED` state, enabling recovery.

The **legacy path** (lines 661–702) is entered when `req.gasLimit10k == 0`, which occurs when the provider has never called `setDefaultGasLimit` (i.e., `providerInfo.defaultGasLimit == 0`). In this path:

1. `clearRequest(provider, sequenceNumber)` is called at line 666 — **before** the callback.
2. The callback is invoked at lines 675–680 with no try-catch or safe call:

```solidity
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    );
}
```

If `_entropyCallback` reverts, the entire transaction reverts, rolling back `clearRequest`. The request remains active in storage. Every subsequent call to `revealWithCallback` will reach the same legacy path and revert again. There is no `CALLBACK_FAILED` state machine in this path, so there is no recovery route.

The only alternative, `reveal`, requires `req.requester == msg.sender` (line 513), meaning only the consumer contract itself can call it. If the consumer contract is broken, this path is also inaccessible. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

- **User fee permanently locked**: The user paid a fee at request time. If the callback always reverts, the request can never be cleared and the fee can never be recovered.
- **Request permanently stuck**: No admin function, no cancel function, and no alternative fulfillment path exists for legacy-path requests whose callbacks revert.
- **Contrast with new path**: The new path (`gasLimit10k != 0`) explicitly handles this with `excessivelySafeCall` and the `CALLBACK_FAILED` state, demonstrating that the protocol designers recognized the risk but did not retrofit the legacy path. [4](#0-3) 

---

### Likelihood Explanation

- Any provider that has not called `setDefaultGasLimit` retains `defaultGasLimit == 0`, causing all requests to that provider to use `gasLimit10k == 0` and enter the legacy path.
- Any consumer contract with a bug in `_entropyCallback` (e.g., a storage write that runs out of gas, a require that fails, or a logic error) will trigger this condition.
- The Fortuna keeper (off-chain fulfillment service) will repeatedly attempt `revealWithCallback` and fail, wasting gas and never fulfilling the request.
- This is reachable by any unprivileged user who deploys a consumer contract and uses a legacy provider. [5](#0-4) 

---

### Recommendation

Wrap the legacy callback in a try-catch block, mirroring the protection already present in the new path. If the callback reverts in the legacy path, emit a failure event and clear the request (do not roll back), so the user's request is resolved and the fee accounting is finalized:

```solidity
// Legacy path
clearRequest(provider, sequenceNumber);
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    ) {
        // success
    } catch {
        emit CallbackFailed(provider, callAddress, sequenceNumber, ...);
    }
}
```

Alternatively, migrate all providers to set `defaultGasLimit != 0` so all requests use the safe new path exclusively.

---

### Proof of Concept

1. Provider P registers with `defaultGasLimit == 0` (never calls `setDefaultGasLimit`).
2. User U deploys a consumer contract `C` whose `_entropyCallback` always reverts (e.g., `revert("broken")`).
3. U calls `requestWithCallback(P, userContribution)` paying the required fee. Request is stored with `gasLimit10k == 0`.
4. Fortuna keeper calls `revealWithCallback(P, seqNum, userContribution, providerContribution)`.
5. Execution enters the `else` (legacy) branch. `clearRequest` is called. Then `C._entropyCallback(...)` is called and reverts.
6. The entire transaction reverts. `clearRequest` is rolled back. Request remains active.
7. Every subsequent call to `revealWithCallback` repeats steps 5–6 and reverts.
8. U's fee is permanently locked. The request is permanently stuck. [6](#0-5) [7](#0-6)
