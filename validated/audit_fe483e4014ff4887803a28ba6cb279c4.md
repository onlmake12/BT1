### Title
Stale `_state.pythFeeInWei` vs. dynamic `pyth.getUpdateFee()` causes arithmetic underflow DoS in `Echo::executeCallback` - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo::executeCallback` computes the provider's credit as `(req.fee + msg.value) - pythFee`, where `pythFee` is the **live** fee returned by `pyth.getUpdateFee(updateData)` at execution time. However, `req.fee` was computed at request time as `msg.value_request - _state.pythFeeInWei`, where `_state.pythFeeInWei` is a **static** initialization parameter. When the live Pyth fee exceeds `req.fee + msg.value_execution`, Solidity 0.8's checked arithmetic causes an underflow revert, permanently blocking all pending requests.

### Finding Description

**At request time** (`requestPriceUpdatesWithCallback`):

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);   // line 84
_state.accruedFeesInWei += _state.pythFeeInWei;                  // line 99
```

The user pays `_state.pythFeeInWei + providerFee`. Only `providerFee` is stored in `req.fee`; the Pyth portion is immediately credited to `_state.accruedFeesInWei`.

**At execution time** (`executeCallback`):

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);                 // line 145 â€” live, dynamic
pyth.parsePriceFeedUpdates{value: pythFee}(...);                 // line 146-153
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);         // line 161-162
```

`pythFee` is the **actual** fee demanded by the Pyth oracle contract, which depends on the update data supplied by the executor and can change via Pyth governance. The subtraction on line 162 assumes `req.fee + msg.value â‰Ą pythFee`. This assumption breaks whenever the live Pyth fee exceeds the static `_state.pythFeeInWei` that was used to compute `req.fee`.

**Root cause:** `_state.pythFeeInWei` is a fixed value set at initialization and never updated to track the live Pyth fee. The `getFee` view function even documents this gap:

```solidity
// Note: The provider needs to set its fees to include the fee charged by the Pyth contract.
// Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the
// fee computation on IPyth assumes it has the full updated data.
```

The code itself acknowledges the risk on lines 155â€“156:
> "if executeCallback can revert, then funds can be permanently locked in the contract."

### Impact Explanation

When `pyth.getUpdateFee(updateData) > req.fee + msg.value_execution`:

1. The subtraction `(req.fee + msg.value) - pythFee` underflows â†’ `executeCallback` reverts.
2. Every pending request created while `_state.pythFeeInWei` was lower than the live Pyth fee becomes permanently unfulfillable.
3. User funds (`req.fee` = provider portion) are locked in the contract with no recovery path.
4. Consumer contracts never receive their price-update callbacks, breaking any downstream protocol logic that depends on Echo.

This is a **DoS + fund lock** impact on the Echo price-update delivery system.

### Likelihood Explanation

- The Pyth protocol fee is governance-controlled and has changed historically. Any upward adjustment after Echo requests are created triggers the bug for all in-flight requests.
- Even without a governance change, `_state.pythFeeInWei` is set once at deployment. If the deployer underestimates the Pyth fee (or the Pyth fee is per-update-data-item and the executor supplies more items than anticipated), the condition is met immediately.
- `executeCallback` is callable by any unprivileged address; no special role is needed to trigger the revert path.
- The `requestPriceUpdatesWithCallback` entry point is fully open to any user, so the pool of affected requests can be large.

### Recommendation

Replace the static `_state.pythFeeInWei` deduction with the actual live Pyth fee at execution time, and adjust `_state.accruedFeesInWei` accordingly:

```diff
- _state.accruedFeesInWei += _state.pythFeeInWei;   // request time: accrue static estimate
+ // do not pre-accrue; settle at execution time

// executeCallback:
  uint256 pythFee = pyth.getUpdateFee(updateData);
  pyth.parsePriceFeedUpdates{value: pythFee}(...);
+ _state.accruedFeesInWei += pythFee;               // accrue actual Pyth fee
  _state.providers[providerToCredit].accruedFeesInWei +=
-     SafeCast.toUint128((req.fee + msg.value) - pythFee);
+     SafeCast.toUint128(req.fee + msg.value);       // provider gets full stored fee + executor top-up
```

Alternatively, keep the pre-accrual but credit the pre-accrued amount back against the live fee:

```solidity
uint256 effectivePythCost = pythFee > _state.pythFeeInWei
    ? pythFee - _state.pythFeeInWei   // extra cost comes from req.fee
    : 0;                               // surplus stays in accruedFeesInWei
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - effectivePythCost);
```

### Proof of Concept

1. Deploy Echo with `pythFeeInWei = 1 wei`.
2. Register a provider with `baseFeeInWei = 5 wei`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`.
3. User calls `requestPriceUpdatesWithCallback{value: 6 wei}(...)`.
   - `req.fee = 6 - 1 = 5 wei`; `_state.accruedFeesInWei = 1 wei`.
4. Pyth governance raises its fee to `7 wei`.
5. Executor calls `executeCallback` with `msg.value = 0`.
   - `pythFee = pyth.getUpdateFee(updateData) = 7 wei`
   - `pyth.parsePriceFeedUpdates{value: 7}(...)` succeeds (contract holds 6 wei, but `_state.accruedFeesInWei = 1` covers the gap only if the contract balance is sufficient â€” in practice the contract may revert here too).
   - `(5 + 0) - 7` â†’ **arithmetic underflow â†’ revert**.
6. The request is permanently stuck; user funds are locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
