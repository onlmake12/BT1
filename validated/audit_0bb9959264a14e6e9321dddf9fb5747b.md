### Title
Provider Fee Permanently Locked in Echo Contract When `executeCallback` Is Never Called - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, when a user calls `requestPriceUpdatesWithCallback`, the provider fee portion of `msg.value` is stored in `req.fee` inside the request struct rather than being immediately credited to the provider. If `executeCallback` is never invoked for that request (e.g., the provider goes offline or abandons the request), the ETH stored in `req.fee` is permanently locked in the contract with no recovery path for either the user or the provider.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the fee accounting splits `msg.value` into two parts:

```solidity
// Echo.sol line 84
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
// Echo.sol line 99
_state.accruedFeesInWei += _state.pythFeeInWei;
```

The Pyth protocol fee is immediately credited to `_state.accruedFeesInWei`, but the provider fee (`req.fee`) is only held inside the request struct. It is only credited to the provider inside `executeCallback`:

```solidity
// Echo.sol lines 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

If `executeCallback` is never called, `req.fee` is never credited to any tracked balance. The contract's two withdrawal paths — `withdrawFees` (drains `_state.accruedFeesInWei`, Pyth's share) and `withdrawAsFeeManager` (drains `_state.providers[provider].accruedFeesInWei`, provider's share) — cannot reach the ETH sitting in unfulfilled `req.fee` fields. There is no `cancelRequest`, no user refund, and no admin sweep for orphaned request fees.

The contract itself acknowledges a related concern in a TODO comment at lines 155–157:

```
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
```

However, the actual lockup path is not a revert — it is the complete absence of a call to `executeCallback`.

---

### Impact Explanation

Every unfulfilled request permanently locks `req.fee` wei in the Echo contract. Over time, as providers rotate, go offline, or simply fail to fulfill requests, the cumulative locked ETH grows with no on-chain mechanism to recover it. Neither the user who paid, the provider who was assigned, nor the admin can retrieve these funds. This is a direct, irreversible loss of user-paid ETH.

---

### Likelihood Explanation

The scenario is realistic and reachable by any unprivileged user:
- A user calls `requestPriceUpdatesWithCallback` and pays the fee.
- The assigned provider fails to call `executeCallback` (network issues, provider shutdown, economic disincentive if gas cost exceeds `req.fee`, or deliberate abandonment).
- After the exclusivity period expires, any other provider may call `executeCallback`, but there is no on-chain incentive or enforcement mechanism compelling them to do so.
- The contract's `getFirstActiveRequests` view function confirms that unfulfilled requests can accumulate indefinitely.

---

### Recommendation

Add a `cancelRequest` function that allows the original requester (or anyone after a timeout) to cancel an unfulfilled request and refund `req.fee` to the requester. Alternatively, immediately credit the provider fee to `_state.providers[provider].accruedFeesInWei` at request time (matching the design of `Entropy.sol` line 237), accepting that fees are non-refundable by design and documenting this clearly.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `msg.value = 1 ether`, `_state.pythFeeInWei = 0.1 ether`.
   - `_state.accruedFeesInWei += 0.1 ether` ✓ (tracked)
   - `req.fee = 0.9 ether` (stored in request struct, untracked in any withdrawable balance)
2. Provider never calls `executeCallback`.
3. Admin calls `withdrawFees(0.1 ether)` — succeeds, recovers only Pyth's share.
4. `_state.providers[provider].accruedFeesInWei == 0` — provider has nothing to withdraw.
5. The `0.9 ether` in `req.fee` is permanently locked. No function in the contract can retrieve it. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
