Audit Report

## Title
`migrate_transaction_info` Traverses `COLUMN_UNCLES` Instead of `COLUMN_TRANSACTION_INFO`, Leaving All Pre-Migration Transaction Info Records Unconverted — (File: `util/migrate/src/migrations/table_to_struct.rs`)

## Summary
The `ChangeMoleculeTableToStruct` migration (version `20200703124523`) contains a copy-paste error in `migrate_transaction_info`: it calls `db.traverse(COLUMN_UNCLES, ...)` instead of `db.traverse(COLUMN_TRANSACTION_INFO, ...)`. As a result, every `TransactionInfo` record in the old 68-byte molecule Table encoding is never converted to the new 52-byte Struct encoding. Simultaneously, uncle-keyed data is spuriously written into `COLUMN_TRANSACTION_INFO`, corrupting the transaction existence index. Any node that ran this migration from pre-v0.35.0 data has permanently stale or corrupted transaction info for all historical transactions.

## Finding Description
`migrate_transaction_info` defines a closure that correctly targets `COLUMN_TRANSACTION_INFO` for writes and uses `TRANSACTION_INFO_SIZE = 52` as the guard. However, the `db.traverse` call at line 93 iterates over `COLUMN_UNCLES` instead of `COLUMN_TRANSACTION_INFO`.

Uncle header records stored in `COLUMN_UNCLES` are 240 bytes. Since `240 != 52`, the `value.len() != TRANSACTION_INFO_SIZE` guard is always true for every uncle entry, so the closure fires unconditionally on every uncle record. It writes `&value[16..]` (224 bytes of uncle header data) into `COLUMN_TRANSACTION_INFO` keyed by uncle hashes — not transaction hashes.

The actual `COLUMN_TRANSACTION_INFO` entries (keyed by tx hash, 68 bytes in old Table format) are never visited and remain unconverted.

After migration, `get_transaction_info` in `store/src/store.rs` deserializes these 68-byte records using `from_slice_should_be_ok`, which applies fixed Struct offsets without length validation. Field accessors (`block_number()`, `block_epoch()`, `key()`) read at wrong offsets, producing garbage values. `get_transaction_with_info` then uses the corrupted `tx_info.key()` to look up `COLUMN_BLOCK_BODY`, finds no match, and returns `None` for any pre-migration confirmed transaction. Additionally, `transaction_exists` returns `true` for uncle hashes because the migration wrote uncle-keyed entries into `COLUMN_TRANSACTION_INFO`.

## Impact Explanation
**Medium (2001–10000 points): Suboptimal/incorrect implementation of CKB state storage mechanism.** The `COLUMN_TRANSACTION_INFO` column is permanently left in the old Table encoding for all pre-migration transactions on any node that ran this migration. All reads via `get_transaction_info` and `get_transaction` return corrupted or missing data. The transaction existence index is also corrupted (uncle hashes appear as confirmed transactions). The impact is local to affected nodes and does not directly cause network-wide crashes or consensus deviation, placing it firmly in the Medium tier.

## Likelihood Explanation
Any node operator who upgraded from a CKB version prior to v0.35.0 (released July 2020) to v0.35.0 or later without re-syncing from genesis would have triggered this migration automatically on startup or via `ckb migrate`. No attacker action is required; the bug is triggered by the standard, documented upgrade procedure. The migration version `20200703124523` is stamped in the database after running, so affected nodes cannot re-run the migration without a new migration step. The bug is deterministic and affects every `COLUMN_TRANSACTION_INFO` entry on every affected node.

## Recommendation
Change line 93 of `util/migrate/src/migrations/table_to_struct.rs` from:
```rust
db.traverse(COLUMN_UNCLES, &mut transaction_info_migration, mode, LIMIT)?;
```
to:
```rust
db.traverse(COLUMN_TRANSACTION_INFO, &mut transaction_info_migration, mode, LIMIT)?;
```
Because the migration version `20200703124523` is already stamped in affected databases, a new migration with a new version string must also be added to: (1) re-scan `COLUMN_TRANSACTION_INFO` and rewrite any remaining 68-byte Table-format entries to the 52-byte Struct format, and (2) remove the spuriously inserted uncle-hash entries from `COLUMN_TRANSACTION_INFO`.

## Proof of Concept
1. Start a CKB node on any version prior to v0.35.0 and sync at least one block so `COLUMN_TRANSACTION_INFO` has entries in 68-byte Table format.
2. Upgrade the binary to v0.35.0+ and run `ckb migrate` (or start the node, which auto-migrates). The migration completes and stamps version `20200703124523`.
3. Query any pre-migration transaction via RPC: `get_transaction(<tx_hash>)`.
4. Observe `null` returned despite the transaction being confirmed on-chain, because `get_transaction_with_info` resolves a corrupted `key` from the mangled `TransactionInfo` and fails to find the block body entry.
5. Conversely, call `transaction_exists(<uncle_hash>)` — observe `true` returned for an uncle hash, because the migration wrote uncle-keyed data into `COLUMN_TRANSACTION_INFO`.
