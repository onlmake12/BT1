### Title
Excess `msg.value` Permanently Captured as Protocol Fees in Entropy `requestHelper` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` absorbs the entire `msg.value` surplus above the provider's share into `accruedPythFeesInWei` without refunding the caller. Any Entropy user or integrating contract that overpays — whether due to fee changes between `getFeeV2()` and execution, or by forwarding a larger `msg.value` — permanently loses the excess ETH to the Pyth protocol treasury with no recovery path.

---

### Finding Description

In `requestHelper`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The Pyth fee bucket receives `msg.value - providerFee`, not the fixed `pythFeeInWei`. Any overpayment above `requiredFee` is silently added to `accruedPythFeesInWei` rather than returned to `msg.sender`. The interface documentation explicitly acknowledges this:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) 

This is confirmed by the existing test suite, which asserts that a 10,000 wei overpayment is absorbed into Pyth's accrued fees:

```solidity
requestWithFee(user2, pythFeeInWei + provider2FeeInWei + 10000, provider2, 42, false);
...
assertEq(random.getAccruedPythFees(), pythFeeInWei * 5 + 10000);
``` [3](#0-2) 

The `getFeeV2` function confirms the intended fee is `providerFee + pythFeeInWei`, not `msg.value`: [4](#0-3) 

---

### Impact Explanation

Any ETH sent above `getFeeV2(provider, gasLimit)` is permanently captured by the Pyth protocol treasury (`accruedPythFeesInWei`). There is no refund mechanism and no way for the caller to recover the excess. This affects:

1. **Integrating contracts** that forward `msg.value` without precisely computing the fee (e.g., `entropy.requestV2{value: msg.value}()`).
2. **Users** who add a buffer to avoid `InsufficientFee` reverts due to fee volatility.
3. **Any caller** whose fee computation races against a provider calling `setProviderFee` between `getFeeV2()` and `requestV2()` execution — if the fee *decreases*, the user overpays; if it *increases*, the transaction reverts.

The excess is not dust: a contract forwarding 1 ETH when the fee is 0.001 ETH loses 0.999 ETH permanently to the protocol treasury.

---

### Likelihood Explanation

- The `requestV2()` family of functions is the primary entry point for all Entropy users.
- Integrating contracts commonly forward `msg.value` without exact fee pre-computation — this is a standard Solidity pattern.
- Provider fees can change at any time via `setProviderFee`, creating a race condition between fee query and request submission.
- The behavior is documented but not enforced as a hard invariant, meaning integrators can easily miss it. [5](#0-4) 

---

### Recommendation

Refund the excess `msg.value` to `msg.sender` at the end of `requestHelper`:

```solidity
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // fixed pythFee
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

Alternatively, enforce `msg.value == requiredFee` with a strict equality check and revert on overpayment, making the behavior explicit and preventing silent fund loss.

---

### Proof of Concept

1. Deploy `Entropy` with `pythFeeInWei = 100` and a provider with `feeInWei = 100` (total required = 200 wei).
2. Call `requestV2{value: 1 ether}()`.
3. Observe `getAccruedPythFees()` increases by `1 ether - 100 = 999...900 wei` instead of `100 wei`.
4. The caller's 1 ether minus 200 wei is permanently locked in the contract, withdrawable only by the Pyth admin via `withdrawFee`. [6](#0-5) [7](#0-6)

    function withdrawFee(address targetAddress, uint128 amount) external {
        require(targetAddress != address(0), "targetAddress is zero address");
        _authoriseAdminAction();

        if (amount > _state.accruedPythFeesInWei)
            revert EntropyErrors.InsufficientFee();

        _state.accruedPythFeesInWei -= amount;

        (bool success, ) = targetAddress.call{value: amount}("");
        require(success, "Failed to withdraw fees");

        emit FeeWithdrawn(targetAddress, amount);
    }
