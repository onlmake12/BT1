### Title
Stale Cached `pythFeeInWei` in Echo Causes Fee Desync with Live Pyth Oracle Fee — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo._initialize()` caches `pythFeeInWei` into `_state.pythFeeInWei` at deployment time. This value is used to charge users in `requestPriceUpdatesWithCallback()` and to credit Echo's `accruedFeesInWei`. However, `executeCallback()` always queries the **live** Pyth oracle fee via `pyth.getUpdateFee(updateData)`. When Pyth governance updates the oracle's `singleUpdateFeeInWei`, the two values diverge, causing incorrect fee accounting and potential provider insolvency or callback DoS.

### Finding Description

In `Echo._initialize()`, the Pyth protocol fee is stored as a fixed value:

```solidity
_state.pythFeeInWei = pythFeeInWei;
``` [1](#0-0) 

There is no admin function anywhere in `Echo.sol` or `EchoUpgradeable.sol` to update `_state.pythFeeInWei` after initialization. [2](#0-1) 

When a user calls `requestPriceUpdatesWithCallback`, the fee charged is based on the stale cached value:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
// ...
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
// ...
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [3](#0-2) 

`getFee()` uses `_state.pythFeeInWei` as the base fee:

```solidity
uint96 baseFee = _state.pythFeeInWei; // Fixed fee to Pyth
``` [4](#0-3) 

But `executeCallback()` always fetches the **live** Pyth fee:

```solidity
IPyth pyth = IPyth(_state.pyth);
uint256 pythFee = pyth.getUpdateFee(updateData);
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
    value: pythFee
}(...)
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [5](#0-4) 

The Pyth oracle's fee is updatable via governance (`setTransactionFee` in `PythGovernance.sol`): [6](#0-5) 

### Impact Explanation

**If Pyth governance increases `singleUpdateFeeInWei`:**

- `pyth.getUpdateFee(updateData)` returns a value larger than `_state.pythFeeInWei`
- `req.fee` (set at request time as `msg.value - _state.pythFeeInWei`) is smaller than the actual Pyth fee
- In `executeCallback`, `(req.fee + msg.value) - pythFee` underflows if the provider does not send extra ETH to compensate
- Since there is no `cancelRequest` function in Echo, user funds are permanently locked if no provider is willing to subsidize the shortfall
- Providers are forced to cover the fee gap out of pocket, making fulfillment economically irrational

**If Pyth governance decreases `singleUpdateFeeInWei`:**

- Users are overcharged: they pay `_state.pythFeeInWei` but Pyth only costs less
- Echo's `accruedFeesInWei` is overcollected relative to what is actually paid to Pyth
- Providers receive inflated credit, draining Echo's balance

### Likelihood Explanation

Pyth governance routinely adjusts `singleUpdateFeeInWei` across chains. Any such update immediately creates a desync for all pending and future Echo requests. The entry path requires no special attacker capability — any user calling `requestPriceUpdatesWithCallback` after a governance fee update is affected. The lack of any `setPythFee` admin function in Echo means the desync persists until a full contract upgrade.

### Recommendation

Echo should not cache `pythFeeInWei` as a static value. Instead, `getFee()` should query `IPyth(_state.pyth).getUpdateFee(...)` dynamically, or an admin function should be added to sync `_state.pythFeeInWei` with the live Pyth fee. At minimum, `executeCallback` should verify that the collected Pyth fee portion (`_state.pythFeeInWei`) is sufficient to cover `pyth.getUpdateFee(updateData)` before proceeding.

### Proof of Concept

1. Deploy `EchoUpgradeable` with `pythFeeInWei = 1 wei` (matching current Pyth fee).
2. User calls `requestPriceUpdatesWithCallback`, paying `getFee()` = `1 wei + providerFees`. `req.fee = providerFees`, `accruedFeesInWei += 1 wei`.
3. Pyth governance executes `setTransactionFee` raising `singleUpdateFeeInWei` to `100 wei`.
4. Provider calls `executeCallback` with `msg.value = 0`.
5. `pythFee = pyth.getUpdateFee(updateData) = 100 wei`.
6. `(req.fee + 0) - 100 wei` underflows → transaction reverts.
7. No cancel mechanism exists → user's ETH is permanently locked in Echo.

    function _initialize(
        address admin,
        uint96 pythFeeInWei,
        address pythAddress,
        address defaultProvider,
        bool prefillRequestStorage,
        uint32 exclusivityPeriodSeconds
    ) internal {
        require(admin != address(0), "admin is zero address");
        require(pythAddress != address(0), "pyth is zero address");
        require(
            defaultProvider != address(0),
            "defaultProvider is zero address"
        );

        _state.admin = admin;
        _state.accruedFeesInWei = 0;
        _state.pythFeeInWei = pythFeeInWei;
        _state.pyth = pythAddress;
        _state.currentSequenceNumber = 1;

        // Two-step initialization process:
        // 1. Set the default provider address here
        // 2. Provider must call registerProvider() in a separate transaction to set their fee
        // This ensures the provider maintains control over their own fee settings
        _state.defaultProvider = defaultProvider;
        _state.exclusivityPeriodSeconds = exclusivityPeriodSeconds;

        if (prefillRequestStorage) {
            for (uint8 i = 0; i < NUM_REQUESTS; i++) {
                Request storage req = _state.requests[i];
                req.sequenceNumber = 0;
                req.publishTime = 1;
                req.callbackGasLimit = 1;
                req.requester = address(1);
            }
        }
    }
