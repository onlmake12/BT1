### Title
`RecentReject::put` Never Increments `total_keys_num`, Disabling the Count-Cap and Allowing Unbounded Disk Growth - (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put` computes `self.total_keys_num + 1` into a **local** variable but never writes the result back to `self.total_keys_num`. The in-memory counter therefore stays frozen at its initialization value for the entire lifetime of the node. Because the `count_limit` guard compares against this stale counter, `shrink()` is never triggered by normal operation, and the on-disk `recent_reject` RocksDB grows without bound. Any unprivileged RPC caller or P2P peer that can cause transactions to be rejected can exploit this to exhaust the node's disk space.

---

### Finding Description

`RecentReject` is the tx-pool subsystem that records recently-rejected transaction hashes so that callers can query why a transaction was rejected. It is backed by a sharded RocksDB-with-TTL instance and is meant to be bounded by two config knobs:

- `keep_rejected_tx_hashes_days` (default **7 days**) — RocksDB TTL per entry
- `keep_rejected_tx_hashes_count` (default **10 000 000**) — in-memory count cap; when exceeded, `shrink()` drops a random shard

The struct tracks the current estimated key count in `total_keys_num`: [1](#0-0) 

`total_keys_num` is initialised from the DB estimate in `build()`: [2](#0-1) 

Every call to `put()` is supposed to increment this counter and trigger `shrink()` when the cap is exceeded. The actual code is: [3](#0-2) 

**The bug**: `self.total_keys_num.checked_add(1)` computes the new value into the local binding `total_keys_num`, but `self.total_keys_num` is **never assigned the result**. After every `put()` call `self.total_keys_num` remains unchanged. The only place `self.total_keys_num` is ever written after construction is inside `shrink()`: [4](#0-3) 

Because `put()` never increments the counter, the condition `total_keys_num > self.count_limit` is evaluated against a value that never grows, so `shrink()` is never called. The DB accumulates entries limited only by the TTL and available disk space.

`put_recent_reject` is called from two sites — the synchronous reject callback registered in `shared_builder.rs` and the async `after_process` path in `process.rs`: [5](#0-4) [6](#0-5) 

The default `keep_rejected_tx_hashes_count` is 10 million entries: [7](#0-6) 

With the count cap broken, an attacker is not constrained by this limit at all.

---

### Impact Explanation

An attacker who can cause transactions to be rejected (trivially achievable — submit transactions with insufficient fee, invalid witnesses, or double-spend inputs) can write an unbounded number of records into the `recent_reject` RocksDB. Each record persists for up to 7 days (default TTL). RocksDB does not reclaim TTL-expired space until compaction runs, so disk usage grows monotonically. Sustained flooding exhausts the node's disk, causing the node process to crash or become unable to write new chain data, resulting in a full denial of service.

---

### Likelihood Explanation

The entry path requires no privilege: any RPC caller (`send_transaction`) or P2P peer (relayed transaction) can submit transactions that will be rejected. The reject path is exercised on every invalid or low-fee transaction. The bug is present on every node that has `recent_reject` enabled (the default). The attacker needs only to submit a high volume of cheap-to-construct invalid transactions.

---

### Recommendation

In `put()`, write the incremented value back to `self.total_keys_num` before the cap check:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ... db.put ...
    match self.total_keys_num.checked_add(1) {
        Some(new_total) => {
            self.total_keys_num = new_total;   // ← missing assignment
            if new_total > self.count_limit {
                self.shrink()?;
            }
        }
        None => {
            self.shrink()?;
        }
    }
    Ok(())
}
```

---

### Proof of Concept

The existing unit test `test_basic` inadvertently demonstrates the bug. It inserts 160 entries (two rounds of 80) into a `RecentReject` with `limit = 100` and then asserts `total_keys_num < 100`: [8](#0-7) 

Because `total_keys_num` is never incremented, it stays at 0 throughout, `shrink()` is never called, and the assertion passes — but for the wrong reason. The DB actually contains up to 160 live entries (subject to shard collisions), far exceeding the intended limit of 100. A correct implementation would call `shrink()` after the 101st insertion and the test would need to verify that the DB was actually pruned.

    pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
        let hash_slice = hash.as_slice();
        let shard = self.get_shard(hash_slice).to_string();
        let reject: ckb_jsonrpc_types::PoolTransactionReject = reject.into();
        let json_string = serde_json::to_string(&reject)?;
        self.db.put(&shard, hash_slice, json_string)?;

        if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
            if total_keys_num > self.count_limit {
                self.shrink()?;
            }
        } else {
            // overflow occurred, try shrink
            self.shrink()?;
        }
        Ok(())
    }

    fn shrink(&mut self) -> Result<u64, AnyError> {

        self.db.drop_cf(&shard)?;

use ckb_types::{core::tx_pool::Reject, packed::Byte32};

#[test]
fn test_basic() {
    let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
    let shard_num = 2;
    let limit = 100;
    let ttl = -1;

    let mut recent_reject = RecentReject::build(tmp_dir.path(), shard_num, limit, ttl).unwrap();

    for i in 0..80u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        recent_reject
            .put(&key, Reject::Malformed(i.to_string(), Default::default()))
            .unwrap();
    }

    for i in 0..80u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        let reject: ckb_jsonrpc_types::PoolTransactionReject =
            Reject::Malformed(i.to_string(), Default::default()).into();
        assert_eq!(
            recent_reject.get(&key).unwrap().unwrap(),
            serde_json::to_string(&reject).unwrap()
        )
    }

    for i in 0..80u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        recent_reject
            .put(&key, Reject::Malformed(i.to_string(), Default::default()))
            .unwrap();
    }

    assert!(recent_reject.total_keys_num < 100);
