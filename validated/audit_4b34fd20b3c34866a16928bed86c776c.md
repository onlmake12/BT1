### Title
Arithmetic Mean of Fee-Rate Ratios in `FeeRateCollector::statistics()` Produces Manipulable, Statistically Incorrect Output — (`File: rpc/src/util/fee_rate.rs`)

---

### Summary

`FeeRateCollector::statistics()` computes the **arithmetic mean of per-transaction fee rates** (i.e., `Σ(fee_i / weight_i) / n`) rather than the correct **weighted mean** (`Σfee_i / Σweight_i`). Because fee rates are ratios, the arithmetic mean is dominated by tiny transactions with extreme fee rates. Any unprivileged transaction sender can get one dust-sized, high-fee-rate transaction confirmed and inflate the reported mean by orders of magnitude, causing wallets that consume the `mean` field of `get_fee_rate_statistics` to massively overpay fees.

---

### Finding Description

In `rpc/src/util/fee_rate.rs`, the `statistics()` method collects individual per-transaction fee rates and then calls `mean()`:

```rust
fee_rates.push(FeeRate::calculate(fee, weight).as_u64());
// ...
Some(FeeRateStatistics {
    mean: mean(&fee_rates).into(),
    median: median(&mut fee_rates).into(),
})
``` [1](#0-0) 

The `mean()` helper is a plain arithmetic mean over the collected `u64` fee-rate values:

```rust
fn mean(numbers: &[u64]) -> u64 {
    let sum: u128 = numbers.iter().map(|number| u128::from(*number)).sum();
    (sum / numbers.len() as u128) as u64
}
``` [2](#0-1) 

Each element pushed into `fee_rates` is `FeeRate::calculate(fee, weight)`, which is `fee * 1000 / weight` (shannons per kilo-weight): [3](#0-2) 

The weight itself is `max(tx_size_bytes, cycles * DEFAULT_BYTES_PER_CYCLES)`: [4](#0-3) 

**The mathematical error:** For a set of transactions, the correct aggregate fee rate is:

```
correct_mean = Σ fee_i / Σ weight_i   (weighted mean)
```

What the code computes is:

```
computed_mean = Σ (fee_i / weight_i) / n   (arithmetic mean of ratios)
```

These are equal only when all weights are identical. When weights differ — which is the normal case — the arithmetic mean is dominated by the smallest-weight transactions. A single 1-byte transaction paying 1,000,000 shannons produces a fee rate of 10^9 shannons/KW. Mixed with 100 normal transactions at 1,000 shannons/KW, the arithmetic mean becomes ≈ 10^7 shannons/KW, while the correct weighted mean stays near 1,000 shannons/KW — a 10,000× distortion.

The `BlockExt` struct documents the invariant that `txs_sizes` has one more element than `txs_fees` (cellbase is included in sizes but not fees), and the code correctly skips the cellbase with `.skip(1)`: [5](#0-4) [6](#0-5) 

The alignment is correct, but the aggregation formula is wrong.

---

### Impact Explanation

The `get_fee_rate_statistics` RPC (and its deprecated alias `get_fee_rate_statics`) is the primary on-chain signal wallets and integrators use to select a fee rate for new transactions: [7](#0-6) 

A wallet consuming the `mean` field will set fees proportional to the reported mean. If the mean is inflated by orders of magnitude, the wallet overpays by the same factor. If the mean is deflated (attacker floods blocks with large zero-fee transactions), the wallet underpays and transactions stall. Neither outcome is detectable by the wallet without independent verification.

---

### Likelihood Explanation

The attack requires only that the adversary submit one or more transactions with an extreme fee rate and have them confirmed. This is a standard, unprivileged transaction-sender capability. The cost to the attacker is bounded by the fee they choose to pay (e.g., 1,000,000 shannons ≈ 0.01 CKB for a 1-byte transaction at 10^9 shannons/KW). The distortion persists for the entire `target` window (up to 101 blocks). Any RPC caller can trigger the statistics read; no special role is required.

---

### Recommendation

Replace the arithmetic mean of fee-rate values with a weighted mean computed over the raw fee and weight accumulators:

```rust
// Accumulate totals instead of individual rates
let mut total_fee: u128 = 0;
let mut total_weight: u128 = 0;
for (fee, cycles, size) in itertools::izip!(...) {
    let weight = get_transaction_weight(*size as usize, cycles);
    if weight > 0 {
        total_fee += fee.as_u64() as u128;
        total_weight += weight as u128;
        fee_rates.push(FeeRate::calculate(fee, weight).as_u64()); // keep for median
    }
}
let mean_fee_rate = if total_weight > 0 {
    (total_fee * 1000 / total_weight) as u64
} else {
    0
};
```

The `median` computation is unaffected and is already robust to outliers.

---

### Proof of Concept

Using the existing test harness in `rpc/src/tests/fee_rate.rs`:

```
Block A: 1 tx, fee=1_000_000 shannons, weight=1 byte  → rate = 1_000_000_000 shannons/KW
Block B: 1 tx, fee=1_000 shannons,     weight=1000 bytes → rate = 1_000 shannons/KW

Arithmetic mean (current code) = (1_000_000_000 + 1_000) / 2 = 500_000_500 shannons/KW
Weighted mean (correct)        = (1_000_000 + 1_000) / (1 + 1000) * 1000 ≈ 1_000 shannons/KW
```

The attacker pays 1,000,000 shannons (≈ 0.01 CKB) to cause wallets to report a fee rate 500,000× above the true market rate for the two-block window. The `mean` field of `FeeRateStatistics` is the manipulated output; the `median` is unaffected. [8](#0-7) [3](#0-2) [4](#0-3)
