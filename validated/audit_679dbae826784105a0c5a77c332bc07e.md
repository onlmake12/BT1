Audit Report

## Title
Unbounded Dead-Cell Set Injected Into SQL `NOT IN` Clause Causes O(N) Indexer RPC Degradation — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
When `index_tx_pool = true`, every call to `get_cells` and `get_cells_capacity` collects the entire `Pool::dead_cells` `HashSet` without any cap and inlines one SQL placeholder per entry into a `NOT IN (…)` clause. Because the pool's dead-cell set is unbounded and grows with every pending transaction, a pool filled with many small transactions forces every subsequent indexer RPC call to allocate an O(N) SQL string, bind O(N) parameters, and submit a query whose planning and evaluation cost scales linearly with pool occupancy, degrading indexer RPC responsiveness for all users.

## Finding Description
In `get_cells`, the entire dead-cell iterator is collected unconditionally:

```rust
dead_cells = pool
    .dead_cells()
    .map(|out_point| { ... })
    .collect::<Vec<(_, u32)>>()   // no cap
```

One `($N, output_index)` placeholder is then emitted per entry and joined into the SQL string, and one `tx_hash` binding is appended per entry in a second loop. The identical pattern exists in `get_cells_capacity`. The `Pool` struct stores dead cells in a plain `HashSet<OutPoint>` with no eviction or size limit; `new_transaction` inserts all inputs unconditionally. The only pool bound is `max_tx_pool_size = 180_000_000` (bytes), which at ~200 bytes per minimal transaction allows on the order of 900,000 pending transactions and a corresponding number of dead cells. No guard, cap, or sampling exists between `pool.dead_cells()` and SQL construction.

## Impact Explanation
The impact is measurable O(N) latency growth per `get_cells` / `get_cells_capacity` RPC call proportional to pool occupancy: SQL string allocation, parameter binding, and DB query-plan compilation all scale linearly with the number of dead cells. This constitutes a performance degradation of the CKB indexer RPC, matching **Low (501–2000 points): Any other important performance improvements for CKB**. No crash, data loss, or consensus effect occurs.

## Likelihood Explanation
The vulnerability is only reachable when `index_tx_pool = true`, which defaults to `false`. Exploitation requires the operator to have opted in, the attacker to possess real UTXOs and pay fees to fill the pool, and either local access or an exposed RPC port. These preconditions impose a meaningful economic and access barrier, making opportunistic exploitation unlikely but targeted exploitation by a motivated actor feasible against infrastructure operators (exchanges, explorers, wallets) who commonly enable tx-pool indexing and expose the RPC.

## Recommendation
Cap the number of dead cells injected into the SQL clause at a configurable maximum (e.g., 10,000). Above the threshold, either skip the `NOT IN` filter entirely (accepting a small risk of returning pool-spent cells that the caller must re-check) or use a temporary table / JOIN instead of an inline list. Alternatively, enforce a maximum entry count in `Pool::new_transaction` and evict oldest entries when the cap is reached.

## Proof of Concept
1. Configure the node with `index_tx_pool = true` and the rich-indexer enabled.
2. Fund a wallet with 10,000+ distinct UTXOs.
3. Submit 10,000 transactions each spending a distinct UTXO via `send_transaction`.
4. Call `get_cells` repeatedly and record response latency.
5. Verify that the generated SQL string contains exactly 10,000 `($N, idx)` placeholders and that latency grows linearly compared to a baseline with an empty pool.
