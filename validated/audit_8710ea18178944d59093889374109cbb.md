### Title
`FeeRate::fee()` Integer Division Truncation Silently Returns Zero, Allowing Fee-Less Transactions to Bypass Minimum Fee Rate Enforcement — (`util/types/src/core/fee_rate.rs`)

---

### Summary

The `FeeRate::fee()` function computes the minimum required fee using integer division `(fee_rate * weight) / 1000`. When `fee_rate * weight < 1000`, the result truncates to zero. This zero is then used as `min_fee` in `check_tx_fee`, causing the fee gate to pass any transaction — including zero-fee transactions — without rejection. This is a direct structural analog to the Aave `wadMul` truncation: a fee-calculation helper silently returns 0 for non-zero inputs, allowing fee-less submissions.

---

### Finding Description

**Root cause — `FeeRate::fee()` in `util/types/src/core/fee_rate.rs`:**

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [1](#0-0) 

`KW` is the hard-coded divisor 1000. Whenever `self.0 * weight < 1000`, integer division floors the result to **0**, and `Capacity::shannons(0)` is returned — even though both `fee_rate` and `weight` are strictly positive.

**Propagation — `check_tx_fee` in `tx-pool/src/util.rs`:**

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
// reject txs which fee lower than min fee rate
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [2](#0-1) 

When `min_fee_rate * tx_size < 1000`, `min_fee` is 0. The comparison `fee < 0` is always false for a `u64`-backed `Capacity`, so **every transaction passes**, including those with `fee = 0`.

**Same truncation in RBF enforcement — `calculate_min_replace_fee` in `tx-pool/src/pool.rs`:**

```rust
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [3](#0-2) 

When `min_rbf_rate * size < 1000`, `extra_rbf_fee = 0`, so a replacement transaction is required to pay only the sum of replaced fees — no incremental fee at all — defeating the economic disincentive of RBF.

**Configured fee rates:**

The default `min_fee_rate` is 1 000 shannons/KW and `min_rbf_rate` is 1 500 shannons/KW. [4](#0-3) 

At these defaults, `min_fee = 1000 * tx_size / 1000 = tx_size`, which is always ≥ 1 for any real transaction. However, the `min_fee_rate` field is fully operator-configurable with no lower-bound validation: [5](#0-4) 

Any operator who sets `min_fee_rate` to a value in the range `[1, 999]` shannons/KW — a plausible choice for a low-fee relay node or a testnet node — will silently allow zero-fee transactions for all transactions whose serialized size is below `1000 / min_fee_rate` bytes. At `min_fee_rate = 1`, every transaction under 1 000 bytes (the vast majority of CKB transactions) passes with zero fee.

---

### Impact Explanation

A tx-pool submitter or RPC caller can submit a transaction with `outputs_capacity == inputs_capacity` (zero fee) to any node whose `min_fee_rate` is set below the truncation threshold. The node's `check_tx_fee` gate passes silently, the transaction enters the pool, and — if the node mines — it can be committed on-chain without paying any fee. For RBF, the same truncation means a replacement transaction need not pay the required incremental fee, undermining the spam-prevention guarantee of RBF.

---

### Likelihood Explanation

The default `min_fee_rate = 1000` shannons/KW is safe because `1000 * tx_size / 1000 = tx_size ≥ 1`. However:

- Operators running low-fee relay nodes, testnets, or private networks routinely lower `min_fee_rate`.
- The configuration field accepts any `u64` with no documented minimum or truncation warning.
- The existing test `TransactionRelayLowFeeRate` explicitly sets `min_fee_rate = 0` on one node, demonstrating that non-default values are a supported and tested use case. [6](#0-5) 

Any operator who sets `min_fee_rate` to a small but non-zero value (e.g., 1–999) will unknowingly accept zero-fee transactions, believing the fee gate is active.

---

### Recommendation

Replace floor division with ceiling division in `FeeRate::fee()`:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    // Ceiling division: (a + b - 1) / b
    let fee = self.0.saturating_mul(weight).saturating_add(KW - 1) / KW;
    Capacity::shannons(fee)
}
```

This ensures that any non-zero `fee_rate` applied to any non-zero `weight` always produces a minimum fee of at least 1 shannon, eliminating the silent truncation-to-zero. The same fix should be applied wherever `FeeRate::fee()` is used for enforcement (both `check_tx_fee` and `calculate_min_replace_fee`).

---

### Proof of Concept

**Setup:** Configure a node with `min_fee_rate = 1` (1 shannon/KW).

**Arithmetic:**
```
min_fee = FeeRate(1).fee(999)
        = (1 * 999) / 1000
        = 999 / 1000
        = 0   ← integer truncation
```

**Result:** `check_tx_fee` evaluates `fee < 0`, which is always `false` for `u64`. A transaction with `outputs_capacity == inputs_capacity` (fee = 0 shannons) passes the gate and enters the pool.

**Threshold formula:** For any configured `min_fee_rate = R`, all transactions with serialized size `< 1000 / R` bytes are admitted with zero fee. At `R = 1`, this covers transactions up to 999 bytes — the majority of CKB transactions. [1](#0-0) [7](#0-6)
