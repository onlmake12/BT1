### Title
Missing Type Script `hash_type` Validation in `ScriptHashTypeVerifier` Allows Bypass of Non-Contextual Admission Gate — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` field of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but performs no equivalent check on each output's **type** script. An unprivileged tx-pool submitter can craft a transaction whose output carries a type script with a syntactically valid but consensus-disabled `ScriptHashType` value (e.g., `Data3 = 6`). That transaction passes the cheap non-contextual gate, enters the verify queue, and is only rejected during the far more expensive contextual (script-execution) stage. Submitting many such transactions is a reachable, zero-privilege resource-exhaustion path against the verify queue and the CPU budget allocated to script execution.

---

### Finding Description

**Root cause — `ScriptHashTypeVerifier::verify()` is incomplete** [1](#0-0) 

The loop body reads:

```rust
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { … }
    } else { … }
}
```

`output.type_()` is never inspected. A type script with `hash_type = 6` (`Data3`) is a valid molecule encoding (molecule's `check_data` accepts any even value or `1`), is a valid `ScriptHashType` enum variant, but is **not** in `ENABLED_SCRIPT_HASH_TYPE`. The verifier silently passes it. [2](#0-1) 

**Call sites that rely on this gate**

`NonContextualTransactionVerifier` composes `ScriptHashTypeVerifier` as its final sub-check: [3](#0-2) 

This verifier is invoked at two critical admission points:

1. **Tx-pool admission** — `tx-pool/src/util.rs::non_contextual_verify` is the first filter applied to every submitted transaction: [4](#0-3) 

2. **Block transaction verification** — `NonContextualBlockTxsVerifier` applies the same verifier to every transaction in a received block: [5](#0-4) 

**What happens to the malformed transaction after it passes**

The transaction enters `verify_rtx`, which constructs a `ContextualTransactionVerifier` and calls `self.script.verify(max_cycles)`. The script verifier calls `select_version` on the type script, which rejects any `ScriptHashType` not yet activated: [6](#0-5) 

The transaction is eventually rejected, but only after consuming the verify-queue slot and the CPU cycles of script-group construction and version selection.

---

### Impact Explanation

An unprivileged RPC caller or relay peer can flood the node with transactions that each carry an output type script with `hash_type = Data3` (or any future not-yet-enabled even value). Each transaction:

- Passes the O(1) non-contextual gate instantly.
- Occupies a slot in the bounded verify queue.
- Forces the node to perform script-group resolution and `select_version` before the rejection is issued.

At scale this exhausts the verify-queue capacity and the CPU budget reserved for contextual verification, degrading or blocking admission of legitimate transactions. The attack requires no stake, no key material, and no privileged access — only the ability to submit transactions via RPC or the P2P relay path.

---

### Likelihood Explanation

The attack surface is fully reachable by any unprivileged tx-pool submitter or RPC caller. Crafting the malformed transaction requires only setting `hash_type = 6` in a type script field, which is trivially achievable with any CKB SDK. No special chain state is required.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_ht = output.lock().hash_type();
        if let Ok(ht) = TryInto::<ScriptHashType>::try_into(lock_ht) {
            let val: u8 = ht.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType { hash_type: lock_ht }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_ht = type_script.hash_type();
            if let Ok(ht) = TryInto::<ScriptHashType>::try_into(type_ht) {
                let val: u8 = ht.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType { hash_type: type_ht }.into());
            }
        }
    }
    Ok(())
}
```

This mirrors the pattern already applied to the cellbase witness lock script in `CellbaseVerifier`: [7](#0-6)

### Proof of Concept

1. Construct a transaction with one input (any live cell) and one output whose `type_` script has `hash_type = 6` (`Data3`) and any `code_hash`.
2. Submit via `send_transaction` RPC.
3. Observe: the node accepts the transaction past non-contextual verification (no `ScriptHashTypeNotPermitted` error at this stage).
4. Observe: the transaction is later rejected during contextual verification with `InvalidVmVersion(2)` or `InvalidScriptHashType`.
5. Repeat in a tight loop from multiple connections to saturate the verify queue and measure degraded admission latency for legitimate transactions.

The root cause is confirmed at: [1](#0-0) 

where `output.type_()` is never read, leaving the type-script `hash_type` field entirely unvalidated at the non-contextual admission gate.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
