### Title
Dual-Token Fee Fallback Allows Systematic Underpayment via Allowance Manipulation — (`target_chains/starknet/contracts/src/pyth.cairo`)

---

### Summary

The Starknet Pyth contract accepts two independent ERC20 tokens for fee payment (`fee_token_address1`/`single_update_fee1` and `fee_token_address2`/`single_update_fee2`). The fees are set in raw token units with no on-chain price normalization. Because the two supported tokens (ETH and STRK) have materially different USD values, and because the fallback logic in `update_price_feeds_internal` can be triggered by any caller simply by holding zero allowance for the primary token, any unprivileged updater can systematically route all fee payments through the economically cheaper token, underpaying relative to the primary token's intended cost.

---

### Finding Description

In `update_price_feeds_internal`, fee collection proceeds as follows:

```cairo
let fee1_transfered = transfer_fee(
    num_updates, caller, contract,
    self.fee_token_address1.read(),
    self.single_update_fee1.read(),
);
if !fee1_transfered {
    let fee2_transfered = transfer_fee(
        num_updates, caller, contract,
        self.fee_token_address2.read(),
        self.single_update_fee2.read(),
    );
    if !fee2_transfered {
        panic_with_felt252(UpdatePriceFeedsError::InsufficientFeeAllowance.into());
    }
}
``` [1](#0-0) 

The helper `transfer_fee` returns `false` (triggering the fallback) whenever the caller's allowance for token1 is below the required amount:

```cairo
if fee_contract.allowance(caller, contract) < total_fee {
    return false;
}
``` [2](#0-1) 

The two fees are stored as independent raw-unit values with no cross-token price oracle:

```cairo
single_update_fee1: u256,
single_update_fee2: u256,
``` [3](#0-2) 

Governance sets each fee independently via `SetFee` (always targets token1) or `SetFeeInToken` (targets either token by address): [4](#0-3) 

The `set_fee` function applies a decimal exponent but performs no cross-token value normalization: [5](#0-4) 

The Starknet documentation confirms both tokens are live payment options and that fees are currently set to their minimum values (1 wei ETH, 1 wei STRK). ETH trades at roughly $3,000 and STRK at roughly $0.30, making 1 wei ETH approximately 10× more expensive in USD terms than 1 wei STRK. Because the caller controls their own ERC20 allowances, they can trivially set their ETH allowance to zero and approve only STRK, forcing the contract to always collect the cheaper token.

---

### Impact Explanation

Any caller of `update_price_feeds`, `update_price_feeds_if_necessary`, `parse_price_feed_updates`, or `parse_unique_price_feed_updates` (all of which route through `update_price_feeds_internal`) can permanently pay with the cheaper token. The protocol collects less fee revenue than governance intended when it set the primary token's fee. At the current 10× price ratio between ETH and STRK, every price-feed update costs the updater 10× less than the primary fee implies. If the ratio widens further (e.g., ETH appreciates while STRK depreciates), the underpayment grows proportionally. This is a direct protocol revenue loss reachable by any unprivileged transaction sender with no special access.

---

### Likelihood Explanation

Exploitation requires only that the caller approve 0 of token1 and the required amount of token2 before calling any update function. This is a zero-cost, zero-privilege, always-available action. Any rational fee-minimizing updater will do this automatically. Likelihood is **high**.

---

### Recommendation

1. **Preferred**: Introduce an on-chain price oracle (e.g., a Pyth price feed for ETH/USD and STRK/USD) to normalize both fees to a common USD target before accepting payment, analogous to how the EVM deployment script already does this at deploy time (`deploy_evm_pricefeed_contracts.ts` lines 137–156).
2. **Alternative**: Expose an explicit `token` parameter on `update_price_feeds` so callers declare their payment token upfront; remove the silent fallback. This makes the choice transparent and auditable.
3. **Minimum**: Governance must actively monitor token price ratios and rebalance `single_update_fee2` whenever the ETH/STRK ratio shifts materially. Document this as a required operational procedure.

---

### Proof of Concept

**Setup** (matching test deploy defaults):
- `single_update_fee1 = 1` (1 wei ETH, token1 = ETH)
- `single_update_fee2 = 1` (1 wei STRK, token2 = STRK)
- ETH price = $3,000 → 1 wei ETH ≈ $3 × 10⁻¹⁵
- STRK price = $0.30 → 1 wei STRK ≈ $3 × 10⁻¹⁶ (10× cheaper)

**Steps**:
1. Attacker calls `IERC20(eth_token).approve(pyth_contract, 0)` — sets ETH allowance to zero.
2. Attacker calls `IERC20(strk_token).approve(pyth_contract, 1)` — approves exactly 1 STRK wei.
3. Attacker calls `pyth.update_price_feeds(valid_update_data)`.
4. Inside `update_price_feeds_internal`, `transfer_fee(..., eth_token, 1)` checks `allowance(caller, contract) = 0 < 1` → returns `false`.
5. Fallback: `transfer_fee(..., strk_token, 1)` checks `allowance = 1 >= 1`, balance sufficient → `transferFrom` succeeds → returns `true`.
6. Price feeds are updated. Attacker paid 1 STRK wei ($3 × 10⁻¹⁶) instead of 1 ETH wei ($3 × 10⁻¹⁵).

The test suite confirms this exact fallback path is reachable and functional: [6](#0-5) 

The test `test_accepts_secondary_fee_if_first_allowance_insufficient` explicitly demonstrates that approving only 500 of token1 (below the 1000 required) causes the contract to silently collect 2000 of token2 instead — confirming the attacker-controlled fallback path is live production behavior.

        fn set_fee(ref self: ContractState, value: u64, expo: u64, token: ContractAddress) {
            let new_fee = apply_decimal_expo(value, expo);
            let old_fee = if token == self.fee_token_address1.read() {
                let old_fee = self.single_update_fee1.read();
                self.single_update_fee1.write(new_fee);
                old_fee

                panic_with_felt252(GovernanceActionError::InvalidGovernanceMessage.into())
            };
            let event = FeeSet { old_fee, new_fee, token };
