Audit Report

## Title
Unbounded GetHeaders Response Amplification via Genesis-Hash Locator — (File: `sync/src/synchronizer/get_headers_process.rs`)

## Summary
Any unprivileged peer can send a `GetHeaders` message with a single-entry locator equal to the genesis hash. The `Synchronizer` has no per-peer rate limiting on incoming `GetHeaders` messages. Each 32-byte request reliably elicits a `SendHeaders` response of up to 2,000 headers (~384 KB), yielding a ~12,000× bandwidth amplification ratio that can be sustained in a tight loop with no ban triggered.

## Finding Description
`GetHeadersProcess::execute()` in `sync/src/synchronizer/get_headers_process.rs` (L47–66) validates only that `locator_size ≤ MAX_LOCATOR_SIZE` (101) and that the node is not in IBD. A single-entry locator `[genesis_hash]` passes both checks.

`locate_latest_common_block` in `sync/src/types/mod.rs` (L1857–1903) first verifies `locator.last() == genesis_hash` (L1867), which passes. The iterator at L1872–1877 then finds `(index=0, latest_common=Some(0))` because the genesis block is always present at height 0. The early-return condition `if index == 0 || latest_common == Some(0)` (L1879) is immediately satisfied, returning `Some(0)` — not `None`.

`get_locator_response(0, hash_stop)` (L1905–1921) then iterates from block 1 up to `min(tip, MAX_HEADERS_LEN=2000)`, performing up to 2,000 `get_block_hash` + `get_block_header` store lookups and returning all headers unconditionally.

The resulting `Status::ok()` return means `post_sync_process` never calls `should_ban()` with a 4xx code, so the attacker peer is never disconnected or banned. The `GetHeadersMissCommonAncestors` (414) ban path in `sync/src/status.rs` (L177) is only reached when `locate_latest_common_block` returns `None`, which does not happen with `[genesis_hash]`.

The `Relayer` protocol has a `governor::RateLimiter` at 30 req/sec keyed by `(PeerIndex, message_type)` (`sync/src/relayer/mod.rs` L81–92), but the `Synchronizer` struct has no equivalent field or guard. The `pending_get_headers` LRU cache (`sync/src/types/mod.rs` L1929–1951) is only consulted inside `send_getheaders_to_peer` (the node's own outgoing requests) and is never read inside `GetHeadersProcess::execute()`.

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** A single attacker connection generates ~384 KB of outbound traffic per message (2,000 headers × ~192 bytes, per the comment at `util/constant/src/sync.rs` L23–24). At network line speed, one peer can saturate the victim's uplink. Each request also triggers up to 2,000 sequential store lookups, amplifying CPU and I/O load. An attacker controlling multiple IPs multiplies the effect linearly across the inbound connection limit.

## Likelihood Explanation
The attack requires only: (1) a TCP connection to a non-IBD CKB node, (2) knowledge of the genesis hash (hardcoded in the chain spec and publicly available). No PoW, no stake, no special peer status, and no authentication is required. The attacker is never banned because the response path returns `Status::ok()`. The attack is trivially repeatable in a tight loop.

## Recommendation
1. Add a per-peer rate limiter to `Synchronizer::try_process()` for `GetHeaders` messages, analogous to the `governor::RateLimiter` already present in `Relayer::new()` (`sync/src/relayer/mod.rs` L91–92).
2. Alternatively, track the last `GetHeaders` response timestamp per peer in `SyncState` and enforce a minimum inter-response interval (e.g., 1 second).
3. Consider treating a locator consisting solely of the genesis hash as a protocol violation (`StatusCode::ProtocolMessageIsMalformed`) since it conveys no useful sync information from a well-behaved peer and would trigger the existing ban path.

## Proof of Concept
```rust
// Connect to a non-IBD CKB node, then loop:
let genesis_hash: Byte32 = consensus.genesis_hash();
let message = SyncMessage::new_builder()
    .set(
        GetHeaders::new_builder()
            .block_locator_hashes(vec![genesis_hash])
            .hash_stop(Byte32::zero())
            .build(),
    )
    .build()
    .as_bytes();

loop {
    net.send(node, SupportProtocols::Sync, message.clone());
    // 32 bytes sent → ~384 KB received (2000 headers × ~192 bytes)
    // Amplification: ~12,000×; no ban, no disconnect
}
```
Verify by measuring bytes-sent vs. bytes-received and asserting the ratio approaches `MAX_HEADERS_LEN × ~192 / 32`. Confirm the peer is never disconnected by checking that `post_sync_process` receives `Status::ok()` on every iteration.
