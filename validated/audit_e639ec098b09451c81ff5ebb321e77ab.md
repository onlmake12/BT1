### Title
Permanently Unfulfillable Requests via Missing `publishTime` Lower-Bound Validation Cause Fee Lock and Provider Queue Stall — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` accepts any `publishTime` value as long as it is not more than 60 seconds in the future. There is no lower-bound check. An unprivileged caller can submit requests with `publishTime = 0` (or any epoch-era timestamp for which no Pyth price data exists). When the provider later calls `executeCallback`, the inner `parsePriceFeedUpdates` call reverts because no price data exists at that timestamp, causing the entire transaction to revert. The request is never cleared, its fee is permanently locked in the contract, and `_state.firstUnfulfilledSeq` stalls at the poisoned sequence number, degrading the provider's off-chain fulfillment service.

---

### Finding Description

**Entry point — missing lower-bound on `publishTime`:** [1](#0-0) 

The only guard is an upper-bound check:

```solidity
require(publishTime <= block.timestamp + 60, "Too far in future");
```

`publishTime = 0` passes this check unconditionally.

**Fulfillment path — `publishTime` used as exact timestamp window:** [2](#0-1) 

```solidity
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
    value: pythFee
}(
    updateData,
    priceIds,
    SafeCast.toUint64(req.publishTime),   // minPublishTime = 0
    SafeCast.toUint64(req.publishTime)    // maxPublishTime = 0
);
```

`parsePriceFeedUpdates` requires at least one price update whose `publishTime` falls within `[minPublishTime, maxPublishTime]`. No Pyth price data has ever been published at Unix timestamp 0, so this call always reverts. Because `clearRequest` and the fee credit happen **after** this call, the entire transaction reverts and the request remains active forever. [3](#0-2) 

**`firstUnfulfilledSeq` stall:** [4](#0-3) 

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

This loop only advances past **inactive** (fulfilled) requests. A poisoned request with `sequenceNumber != 0` is still active, so `firstUnfulfilledSeq` stalls at the poisoned sequence number permanently. Every subsequent call to `getFirstActiveRequests` returns the unfulfillable request first. [5](#0-4) 

**Fixed-size primary array context:** [6](#0-5) 

The 32-slot primary array can be saturated with poisoned requests (each hashing to a distinct `shortKey`), forcing all subsequent legitimate requests into the unbounded overflow mapping at higher gas cost.

---

### Impact Explanation

1. **Permanent fee lock**: The attacker's paid fee (`req.fee`) is credited to no one and remains locked in the contract because `clearRequest` never executes.
2. **`firstUnfulfilledSeq` stall**: The pointer used by `getFirstActiveRequests` stalls at the first poisoned sequence number. The provider's off-chain fulfillment service (analogous to Fortuna) continuously retries the unfulfillable request, wasting gas on every poll cycle.
3. **Elevated gas for legitimate requests**: Once the 32 primary slots are occupied by poisoned requests, all new legitimate requests overflow to the mapping, increasing their storage cost.
4. **No cancellation path**: There is no `cancel` or `refund` function in Echo, so poisoned requests cannot be removed by anyone — not the requester, not the provider, not the admin.

---

### Likelihood Explanation

Any unprivileged address can call `requestPriceUpdatesWithCallback` with `publishTime = 0`. The only cost is the required fee (`getFee(provider, callbackGasLimit, priceIds)`). Unlike the SIZE analog where the attacker recovered their tokens via `refund()`, here the attacker forfeits the fee — but the fee is also lost to the protocol (locked, not credited). A griever willing to pay fees proportional to the number of poisoned requests can stall the provider queue indefinitely. The 32-slot primary array can be saturated with 32 such requests, each chosen to hash to a distinct `shortKey`.

---

### Recommendation

Add a minimum `publishTime` lower-bound in `requestPriceUpdatesWithCallback`, analogous to the existing upper-bound:

```solidity
uint256 constant MAX_PAST_VALIDITY = 1 hours;
require(
    publishTime >= block.timestamp - MAX_PAST_VALIDITY,
    "Too far in past"
);
```

This mirrors the approach used in the Pulse Scheduler, which enforces `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` to reject stale timestamps: [7](#0-6) 

Additionally, consider adding a `cancelRequest` function callable by the original requester (with fee forfeiture to the protocol) to allow removal of stuck requests.

---

### Proof of Concept

```solidity
// 1. Attacker submits a request with publishTime = 0
uint64 seqNum = echo.requestPriceUpdatesWithCallback{value: fee}(
    provider,
    0,          // publishTime = 0 — passes the only check (0 <= block.timestamp + 60)
    priceIds,
    callbackGasLimit
);

// 2. Provider attempts to fulfill — always reverts
// parsePriceFeedUpdates(updateData, priceIds, 0, 0) reverts:
// no Pyth price data has publishTime == 0
echo.executeCallback(provider, seqNum, updateData, priceIds); // REVERTS

// 3. Request remains active; firstUnfulfilledSeq stalls at seqNum
// getFirstActiveRequests(1) always returns this unfulfillable request
// Provider off-chain service loops forever retrying it
```

    function getFirstActiveRequests(
        uint256 count
    )
        external
        view
        override
        returns (Request[] memory requests, uint256 actualCount)
    {
        requests = new Request[](count);
        actualCount = 0;

        // Start from the first unfulfilled sequence and work forwards
        uint64 currentSeq = _state.firstUnfulfilledSeq;

        // Continue until we find enough active requests or reach current sequence
        while (
            actualCount < count && currentSeq < _state.currentSequenceNumber
        ) {
            Request memory req = findRequest(currentSeq);
            if (isActive(req)) {
                requests[actualCount] = req;
                actualCount++;
            }
            currentSeq++;
        }

        // If we found fewer requests than asked for, resize the array
        if (actualCount < count) {
            assembly {
                mstore(requests, actualCount)
            }
        }
    }
