### Title
Deprecated `getUpdateFee(uint updateDataSize)` Overestimates Required Fee by Up to 255x Due to Unit Mismatch — (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

The deprecated `getUpdateFee(uint updateDataSize)` function in `Pyth.sol` contains a unit mismatch analogous to the zkSync M-02 report: the parameter `updateDataSize` is in units of *update data items* (count of `bytes[]` array elements), but the function multiplies by the constant `255` (maximum messages per accumulator update item), treating each item as if it always contains 255 messages. This causes the function to return a fee estimate up to 255× higher than the actual fee enforced by `updatePriceFeeds`. Because `updatePriceFeeds` does not refund excess ETH, users who rely on this function to estimate fees lose the overpaid amount to the contract.

---

### Finding Description

The deprecated overload at lines 82–93 computes:

```solidity
/// This method is deprecated, please use the `getUpdateFee(bytes[])` instead.
function getUpdateFee(
    uint updateDataSize
) public view returns (uint feeAmount) {
    // In the accumulator update data a single update can contain
    // up to 255 messages and we charge a singleUpdateFee per each message
    return
        255 *
        singleUpdateFeeInWei() *
        updateDataSize +
        transactionFeeInWei();
}
``` [1](#0-0) 

`updateDataSize` is the **count of `bytes[]` array elements** (i.e., the number of accumulator update blobs). The function hard-codes `255` as the number of messages per blob, which is the *maximum possible*, not the actual count.

The actual fee enforcement in `updatePriceFeeds` correctly counts real messages:

```solidity
uint totalNumUpdates = 0;
for (uint i = 0; i < updateData.length; ) {
    totalNumUpdates += updatePriceInfosFromAccumulatorUpdate(updateData[i]);
    ...
}
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
``` [2](#0-1) 

`getTotalFee` is:

```solidity
function getTotalFee(uint totalNumUpdates) private view returns (uint requiredFee) {
    return (totalNumUpdates * singleUpdateFeeInWei()) + transactionFeeInWei();
}
``` [3](#0-2) 

The unit mismatch is:

| Location | Unit used |
|---|---|
| `getUpdateFee(uint)` denominator | `updateDataSize` × 255 (max-message-equivalent) |
| `updatePriceFeeds` enforcement | actual message count |

A typical accumulator update blob contains 1–10 messages, not 255. The deprecated function therefore overestimates the fee by a factor of `255 / actual_messages_per_blob`.

Because `updatePriceFeeds` only checks `msg.value >= requiredFee` and has no refund path, any ETH sent above the true required fee is permanently locked in the contract.

---

### Impact Explanation

A user who calls `getUpdateFee(uint updateDataSize)` to estimate the fee and then sends that amount to `updatePriceFeeds` will:

1. Receive a fee quote of `255 * singleUpdateFeeInWei * N + transactionFeeInWei`.
2. Pay that amount as `msg.value`.
3. Have the actual required fee deducted as `actualMessages * singleUpdateFeeInWei + transactionFeeInWei`.
4. Lose `(255 - actualMessages) * singleUpdateFeeInWei * N` ETH to the contract with no refund.

For a single update blob containing 1 price feed message, the user overpays by **254×** the per-message fee. This is a direct, quantifiable loss of user funds.

---

### Likelihood Explanation

The function is `public` and still deployed on-chain. Integrators using older SDK versions, third-party wrappers, or any code that calls `getUpdateFee(uint)` (the single-argument overload) are affected. The function signature is distinct from the correct `getUpdateFee(bytes[])` overload, so it can be called independently. Any transaction sender who uses this overload to pre-compute their `msg.value` will overpay.

---

### Recommendation

Remove the deprecated `getUpdateFee(uint updateDataSize)` function entirely, or replace its body with an unconditional revert pointing users to `getUpdateFee(bytes[])`. If backward compatibility must be preserved, fix the calculation to avoid the 255× multiplier — for example, return `singleUpdateFeeInWei() * updateDataSize + transactionFeeInWei()` as a conservative lower bound, or document clearly that the return value is a worst-case ceiling and that excess ETH will not be refunded.

---

### Proof of Concept

```
Scenario: 1 update blob containing 1 price feed message, singleUpdateFeeInWei = 1 wei

getUpdateFee(1)  →  255 * 1 * 1 + transactionFeeInWei  =  255 + transactionFeeInWei

updatePriceFeeds actual required fee  →  1 * 1 + transactionFeeInWei  =  1 + transactionFeeInWei

User overpays: 254 wei, permanently locked in contract.

At singleUpdateFeeInWei = 1e9 wei (1 gwei) and 10 blobs × 1 message each:
  Quoted fee  = 255 * 1e9 * 10 = 2,550,000,000,000 wei = 2550 gwei
  Actual fee  =   1 * 1e9 * 10 =    10,000,000,000 wei =   10 gwei
  Loss        = 2540 gwei per call
```
