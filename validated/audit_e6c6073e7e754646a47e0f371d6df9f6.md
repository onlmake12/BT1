### Title
Missing Owner Validation in `undelegate` Allows Anyone to Force-Undelegate Another User's Staked Positions — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `undelegate` instruction in the integrity pool program requires `owner` to be a transaction signer but imposes **no constraint validating that the signer is the actual owner stored in `stakeAccountMetadata`**. An attacker can supply their own wallet as `owner` (satisfying the signer check) while targeting any victim's `stakeAccountPositions`, force-closing the victim's delegated staking positions without consent.

---

### Finding Description

In the staking program, every instruction that modifies a user's stake account enforces ownership via Anchor's `relations` constraint:

```json
// staking program — closePosition (governance/pyth_staking_sdk/src/idl/staking.json)
{
  "name": "owner",
  "relations": ["stake_account_metadata"],   // ← enforces owner == stakeAccountMetadata.owner
  "signer": true,
  "writable": true
}
```

The integrity pool's `undelegate` instruction omits this constraint entirely:

```json
// integrity-pool — undelegate (governance/pyth_staking_sdk/src/idl/integrity-pool.json)
{
  "name": "owner",
  "writable": true,
  "signer": true
  // ← NO relations constraint; only signer check
}
```

The `stakeAccountMetadata` account in `undelegate` carries the comment `"CHECK : This AccountInfo is safe because it's a checked PDA"`, confirming the program only verifies PDA derivation — it does **not** verify `owner.key == stakeAccountMetadata.owner`. [1](#0-0) [2](#0-1) 

The `StakeAccountMetadataV2` struct stores an `owner: pubkey` field that identifies the legitimate owner of each stake account. Without a `relations` check, the integrity pool program never compares the transaction signer against this stored field. [3](#0-2) 

---

### Impact Explanation

An attacker can call `undelegate` with:
- Their own wallet as `owner` (passes the `signer: true` check)
- Any victim's `stakeAccountPositions` (a public account)

The program will proceed to close the victim's delegated position in the integrity pool. The victim loses their active delegation, forfeits accrued rewards for the current epoch, and has their staking position disrupted without consent. Because `stakeAccountPositions` accounts are publicly discoverable on-chain, any delegator is a potential target.

---

### Likelihood Explanation

The attack requires no special privileges — only the ability to submit a Solana transaction. The victim's `stakeAccountPositions` address is publicly visible on-chain. The attacker pays only transaction fees. This is a straightforward, low-cost griefing attack executable by any unprivileged user.

---

### Recommendation

Add a `relations` constraint on `owner` in the `undelegate` instruction, mirroring the pattern already used in the staking program's `closePosition` and `withdrawStake`:

```json
{
  "name": "owner",
  "relations": ["stake_account_metadata"],
  "signer": true,
  "writable": true
}
```

This causes Anchor to automatically enforce `owner.key == stakeAccountMetadata.owner` before executing the instruction body. [4](#0-3) [5](#0-4) 

---

### Proof of Concept

1. Attacker identifies victim's `stakeAccountPositions` address on-chain (public account).
2. Attacker constructs an `undelegate` transaction:
   - `owner` = attacker's wallet (signs the transaction)
   - `stakeAccountPositions` = victim's account
   - `stakeAccountMetadata` = PDA derived from victim's `stakeAccountPositions` (deterministic, publicly computable)
   - `publisher` = any publisher the victim is delegated to
3. The program validates:
   - `owner` is a signer ✓ (attacker's wallet signed)
   - `stakeAccountMetadata` is a valid PDA ✓ (correctly derived)
   - `owner.key == stakeAccountMetadata.owner` — **not checked** ✗
4. The victim's delegation position is closed without their consent. [6](#0-5) [7](#0-6) 

---

**Caveat:** The Rust source files for the integrity pool program are not available in the indexed codebase. It is possible that a manual `require!(owner.key == stake_account_metadata.owner)` check exists inside the instruction handler body, or that the CPI into the staking program's `closePosition` (which does carry a `relations` constraint) enforces ownership transitively. The IDL structure — specifically the absence of `relations` on `owner` and the `CHECK` comment on `stakeAccountMetadata` — is the basis for this finding. Full confirmation requires inspection of the on-chain program source.
