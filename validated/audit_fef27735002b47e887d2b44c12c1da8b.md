### Title
Incomplete `ScriptHashTypeVerifier` Omits Type Script Hash Type Validation — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` only validates the `hash_type` field of the **lock script** of each transaction output against `ENABLED_SCRIPT_HASH_TYPE`, but entirely omits the same check for the **type script** of each output. This means a transaction carrying an output with a non-permitted type script hash type (e.g., `Data3`, `Data4`, …) passes the cheap non-contextual filter, is admitted to the tx-pool, and is relayed to peers before being rejected during the expensive contextual/script-execution phase.

---

### Finding Description

`ScriptHashTypeVerifier` is documented as:

> *"Verify that the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules."*

Its `verify()` implementation iterates over outputs and checks only `output.lock().hash_type()`:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The `ENABLED_SCRIPT_HASH_TYPE` set permits only `Data(0)`, `Type(1)`, `Data1(2)`, and `Data2(4)`: [2](#0-1) 

The `ScriptHashType` enum, however, is defined for values up to `Data127` (i.e., `N << 1` for `N` in `3..=127`): [3](#0-2) 

A `CellOutput` has both a mandatory lock script and an optional type script. The `check_data` primitive validates that the raw byte is a parseable `ScriptHashType` enum value, but does **not** enforce the `ENABLED_SCRIPT_HASH_TYPE` constraint: [4](#0-3) 

So a transaction output with `type_script.hash_type = Data3 (6)` passes `check_data` (valid enum value) and passes `ScriptHashTypeVerifier` (only lock is checked), but would fail later in `TxInfo::extract_script_and_dep_index` with `ScriptError::InvalidScriptHashType` during contextual script execution: [5](#0-4) 

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier`, which is the first, cheapest gate applied to every incoming transaction: [6](#0-5) 

It is invoked in `non_contextual_verify` in the tx-pool before any expensive resolution or script execution: [7](#0-6) 

---

### Impact Explanation

A crafted transaction with an output whose type script uses a non-permitted `ScriptHashType` (e.g., `Data3`) bypasses the non-contextual filter and:

1. Is admitted to the tx-pool, consuming pool memory.
2. Is relayed to connected peers via the relay protocol, wasting network bandwidth across the entire P2P graph.
3. Forces every receiving node to run the full, expensive `ContextualTransactionVerifier` (including script group construction and `verify_rtx`) before the transaction is finally rejected.

This is a resource-exhaustion / DoS vector reachable by any unprivileged transaction submitter (RPC `send_transaction` or P2P relay). The attacker pays only the cost of crafting and submitting the transaction; every full node in the network pays the cost of contextual verification.

---

### Likelihood Explanation

The entry path requires no special privilege: any party can submit a transaction via the public RPC endpoint or by connecting as a P2P peer. Crafting a transaction with an output type script whose `hash_type` byte is set to `6` (`Data3`) is trivial. The missing check is in the cheapest, first-pass filter, so the amplification ratio (attacker cost vs. network cost) is high.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of the optional type script on each output, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }

        // NEW: type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

This aligns the non-contextual filter with the intent stated in its own doc-comment and with the `check_data` pattern already applied to `CellOutput` at the serialization layer.

### Proof of Concept

1. Construct a `TransactionView` with one output whose type script has `hash_type = 6` (`Data3`).
2. Instantiate `ScriptHashTypeVerifier::new(&tx)` and call `.verify()`.
3. Observe that it returns `Ok(())` — the invalid type script hash type is not caught.
4. Submit this transaction via `send_transaction` RPC; it passes `non_contextual_verify` and enters the tx-pool.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
