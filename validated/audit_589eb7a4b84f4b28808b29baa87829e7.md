### Title
`DaoScriptSizeVerifier` Checks Only Lock Script Byte Size, Not Content â€” Allows Lock Script Substitution During DAO Prepare Phase (`File: verification/src/transaction_verifier.rs`)

---

### Summary

The `DaoScriptSizeVerifier` enforces that the lock script of a DAO prepare-phase output cell has the same serialized byte size as the corresponding deposit cell's lock script. However, it only compares `total_size()` and does not verify that the lock script content (`code_hash`, `hash_type`, `args`) is identical. This is a direct analog to the JBSplitsStore incomplete sameness check: a "locked" property is validated by a partial field comparison, allowing the remaining fields to be silently overwritten.

---

### Finding Description

`DaoScriptSizeVerifier::verify()` is the sole consensus-level guard on lock script constraints during the DAO depositâ†’prepare (withdraw phase 1) transition. Its own doc comment states it is *"a temporary solution till Nervos DAO script can be properly upgraded"*, confirming that the on-chain DAO C script (`dao.c`) does **not** enforce lock script identity between deposit and prepare cells.

The critical check at line 885 is:

```rust
if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
    return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
}
``` [1](#0-0) 

This compares only the serialized byte length of the two lock scripts. It does not compare `code_hash`, `hash_type`, or `args`. Any two lock scripts that serialize to the same number of bytes will pass this check, regardless of their actual content.

The verifier is invoked in both the tx-pool admission path and the block-commit path: [2](#0-1) [3](#0-2) 

The `DaoScriptSizeVerifier` is the only place in the consensus pipeline that constrains the lock script of the prepare output relative to the deposit input. The DAO type script itself only enforces capacity equality, not lock script identity. [4](#0-3) 

---

### Impact Explanation

An attacker who can satisfy the deposit cell's lock script can craft a prepare transaction whose output cell carries a **different** lock script of the same byte length. For example:

- Deposit cell lock: `secp256k1_blake160` with `args = [0u8; 20]` â†’ `total_size = N`
- Prepare cell lock: `secp256k1_blake160` with `args = [1u8; 20]` â†’ `total_size = N`

The `DaoScriptSizeVerifier` passes. The prepare cell is now controlled by a different key. The original depositor's lock script no longer guards the DAO cell at the prepare stage. The attacker can then complete withdraw phase 2 using the substituted lock.

This allows **lock script substitution** during the DAO prepare phase: the "locked" property (lock script identity) is only partially checked (size only), so the content fields (`code_hash`, `hash_type`, `args`) can be silently overwritten â€” exactly the same class of bug as the JBSplitsStore `preferClaimed`/`preferAddToBalance` omission.

---

### Likelihood Explanation

The attacker must be able to satisfy the original deposit cell's lock script to spend it in the prepare transaction. This is a necessary prerequisite. However, this is directly reachable by any unprivileged transaction sender who controls the deposit cell (i.e., the depositor themselves, or anyone who has obtained the signing key). No special privilege, majority hashpower, or social engineering is required. The transaction is submitted via the standard RPC/tx-pool path. [5](#0-4) 

---

### Recommendation

Replace the `total_size()` comparison with a full equality check of the lock script. Since `packed::Script` implements `PartialEq` via byte-slice comparison (`as_slice()`), this is a single-line fix:

```rust
// Before (incomplete â€” size only):
if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
    return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
}

// After (complete â€” full content equality):
if input_meta.cell_output.lock() != cell_output.lock() {
    return Err((TransactionError::DaoLockScriptMismatch { index: i }).into());
}
``` [6](#0-5) 

---

### Proof of Concept

1. Construct a deposit transaction with a DAO type script and lock script `A` where `A.args = [0u8; 20]` (total serialized size = N bytes).
2. Construct a prepare transaction that:
   - Spends the deposit cell (satisfying lock script `A`)
   - Outputs a cell with DAO type script and lock script `B` where `B.args = [1u8; 20]` (total serialized size = N bytes, same as `A`)
   - Sets output data to the deposit block number (non-zero, marking it as a prepare cell)
3. Submit the prepare transaction. `DaoScriptSizeVerifier` passes because `A.total_size() == B.total_size()`.
4. The prepare cell is now guarded by lock script `B` (attacker's key), not `A` (original depositor's key).
5. The attacker completes withdraw phase 2 using lock script `B`, claiming the DAO interest. [7](#0-6) [8](#0-7)

    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
