### Title
Governance VAA Sequence Reordering Allows Any Relayer to Permanently Block Lower-Sequence VAAs — (`lazer/contracts/sui/sources/governance.move`)

### Summary
`governance::process_incoming` enforces only a "strictly greater than" sequence check (`seen_sequence < sequence`). Because both `upgrade` and `update_trusted_signer` are `public` functions callable by any address, any relayer holding two valid Wormhole-signed governance VAAs can submit the higher-sequence one first, permanently rendering the lower-sequence VAA unprocessable.

### Finding Description
`process_incoming` advances `seen_sequence` to the submitted VAA's sequence number and rejects any future VAA whose sequence is not strictly greater: [1](#0-0) 

The check is `seen_sequence < sequence`, not `seen_sequence + 1 == sequence`. This means processing seq=6 before seq=5 is permitted, but after seq=6 is committed, seq=5 fails permanently with `EOldSequenceNumber`.

Both governance entry points are `public fun` with no capability or signer guard beyond the VAA itself: [2](#0-1) [3](#0-2) 

Wormhole VAAs are public once signed by the guardian network. Any relayer who possesses both VAAs can submit them in any order.

### Impact Explanation
A malicious (or even accidentally misordered) relayer can permanently prevent a specific governance VAA from executing. If the blocked VAA is a contract upgrade patching a live vulnerability, the governance team must issue a brand-new VAA at a higher sequence number — introducing delay and operational disruption. The original VAA is irrecoverably skipped.

### Likelihood Explanation
Requires two valid governance VAAs to be in flight simultaneously, which occurs during routine governance batches (e.g., an upgrade followed immediately by a signer rotation). No privileged access is needed; the attacker only needs to relay a publicly available VAA before the intended one lands.

### Recommendation
Replace the non-strict ordering check with a strict sequential check:

```move
// Before (vulnerable):
assert!(self.seen_sequence < sequence, EOldSequenceNumber);

// After (strict sequential):
assert!(self.seen_sequence + 1 == sequence, EOldSequenceNumber);
```

If intentional skipping of sequence numbers is required by the governance design, use an explicit set of processed sequences rather than a single high-water mark.

### Proof of Concept
1. Initialize `Governance` with `seen_sequence = 0`.
2. Obtain two valid guardian-signed VAAs: seq=5 (upgrade) and seq=6 (signer update).
3. Call `update_trusted_signer(state, vaa_seq6)` → succeeds, `seen_sequence = 6`.
4. Call `upgrade(state, vaa_seq5)` → `assert!(6 < 5)` fails → aborts with `EOldSequenceNumber`.
5. VAA seq=5 can never be processed again. [4](#0-3)

public(package) fun process_incoming(
    self: &mut Governance,
    vaa: VAA
): vector<u8> {
    let sequence = vaa.sequence();
    let (chain_id, address, payload) = vaa.take_emitter_info_and_payload();
    assert!(self.chain_id == chain_id, EMismatchedEmitterChainID);
    assert!(self.address == address, EMismatchedAddress);
    assert!(self.seen_sequence < sequence, EOldSequenceNumber);
    self.seen_sequence = sequence;
    payload
}
