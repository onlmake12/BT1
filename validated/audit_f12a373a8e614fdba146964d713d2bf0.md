### Title
Overpayment of Native ETH Not Refunded to Requester in `requestPriceUpdatesWithCallback` - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.requestPriceUpdatesWithCallback` accepts `msg.value >= requiredFee` but never refunds the excess to the caller. Instead, the entire overpayment silently accrues to the provider's fee balance. There is no documentation warning users of this behavior (unlike `Entropy.sol`, which explicitly states "excess value is *not* refunded to the caller").

### Finding Description
In `Echo.requestPriceUpdatesWithCallback`, the fee check only enforces a minimum:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
```

The stored request fee is then set as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

This means any overpayment beyond `requiredFee` is silently folded into `req.fee`. When `executeCallback` is later called, the provider receives:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

The `req.fee` already contains the overpayment, so the provider is credited the full excess. The original requester receives no refund.

The `IEcho` interface comment even states `"The msg.value must be equal to getFee(callbackGasLimit)"`, implying exact payment is expected — yet the contract silently accepts and retains any surplus. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
Any user who sends more ETH than `getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess to the provider. The provider can immediately withdraw it via `withdrawFees`. The requester has no mechanism to recover the overpayment. This is a direct, irreversible loss of user funds. [4](#0-3) 

### Likelihood Explanation
Overpayment is realistic in several scenarios:

1. **Provider fee update race**: A provider can call `setProviderFee` between the user's off-chain `getFee` call and transaction inclusion, changing `baseFeeInWei`, `feePerFeedInWei`, or `feePerGasInWei`. If the fee decreases, the user overpays.
2. **Defensive overpayment**: Users or integrating contracts (e.g., wallets, aggregators) commonly send a small buffer above the computed fee to avoid reverts, as is standard practice in DeFi.
3. **Stale fee estimate**: Any latency between fee computation and transaction submission can result in overpayment if the provider lowers fees. [5](#0-4) 

### Recommendation
After computing `requiredFee`, refund any excess `msg.value` to `msg.sender`:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();

// Refund overpayment
uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool refunded, ) = msg.sender.call{value: excess}("");
    require(refunded, "Refund failed");
}

// Store only the exact required fee
req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);
```

Alternatively, enforce exact payment (`msg.value == requiredFee`) and document this clearly, consistent with the `IEcho` interface comment.

### Proof of Concept

1. Provider registers with `baseFeeInWei = 1000`, `feePerFeedInWei = 100`, `feePerGasInWei = 1`.
2. User calls `getFee(provider, 500_000, priceIds)` off-chain and gets `requiredFee = X`.
3. User submits `requestPriceUpdatesWithCallback{value: X + 500}(...)` with a 500 wei buffer.
4. Transaction is included. `req.fee = (X + 500) - pythFeeInWei` — 500 wei above the provider's expected fee.
5. Provider calls `executeCallback`. Provider's `accruedFeesInWei` increases by `req.fee - pythFee`, which includes the 500 wei overpayment.
6. Provider calls `withdrawFees(500)` and extracts the user's overpayment.
7. User has no recourse to recover the 500 wei. [6](#0-5) [7](#0-6)
