### Title
Missing Payload Timestamp Staleness Validation in `verifyUpdate` Allows Replay of Arbitrarily Old Lazer Price Data - (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` validates the ECDSA signer's key expiry but performs no validation of the payload's embedded timestamp against the current block time and has no replay-protection mechanism. An unprivileged Lazer relayer can submit a validly-signed but arbitrarily old price update — or replay the same stale update indefinitely — and the call will succeed, returning the stale payload to any consumer contract that relies on `verifyUpdate` as its sole security boundary.

---

### Finding Description

`verifyUpdate` performs exactly three checks:

1. EVM format magic bytes match
2. ECDSA signature recovers to a non-zero address
3. The recovered signer address is in the trusted-signer mapping and has not expired (`block.timestamp < trustedSignerToExpiresAtMapping[signer]`) [1](#0-0) 

The payload bytes returned to the caller contain a `uint64` timestamp (the price-data generation time), as parsed by `PythLazerLib.parsePayloadHeader`: [2](#0-1) 

That timestamp is part of the signed data and is therefore authentic — but `verifyUpdate` never compares it against `block.timestamp`. There is also no nonce, sequence number, or consumed-hash registry, so the identical byte string can be submitted repeatedly.

The analogy to the reported CRL issue is direct:

| CRL finding | Pyth Lazer analog |
|---|---|
| `getCrl` only checks that returned bytes are non-empty | `verifyUpdate` only checks that the signer key has not expired |
| CRL `nextUpdate` field is never validated | Payload timestamp is never validated against `block.timestamp` |
| Updating a certificate does not delete the old CRL | Rotating a signer key does not invalidate previously signed (old) payloads |

---

### Impact Explanation

Any consumer contract that calls `verifyUpdate` and then uses the returned payload without independently checking the embedded timestamp will accept stale or replayed price data. Because `verifyUpdate` is the documented verification entry point and the SDK documentation shows consumers parsing the timestamp only after the call succeeds, many integrators will treat a successful return as a freshness guarantee.

A relayer holding a validly-signed payload from hours or days ago can submit it at any time (e.g., during a price spike) to make a consumer contract see a favorable stale price, enabling profitable manipulation of any protocol that settles against the Lazer price.

---

### Likelihood Explanation

The entry path requires only the ability to call `verifyUpdate` with a fee — any address can do this. Old signed payloads are routinely available to anyone who subscribed to the Lazer WebSocket stream. No privileged access, key compromise, or governance majority is needed.

---

### Recommendation

- **Short term**: Inside `verifyUpdate`, after recovering the signer, parse the payload's leading timestamp field and revert if it is older than a configurable `maxPayloadAge` (e.g., 60 seconds):
  ```solidity
  uint64 payloadTimestamp = uint64(bytes8(payload[4:12])); // after 4-byte magic
  require(block.timestamp <= payloadTimestamp / 1e6 + maxPayloadAge, "stale payload");
  ```
- **Long term**: Maintain a mapping of consumed payload hashes to prevent replay of any previously accepted update, analogous to Wormhole's VAA claim accounts.

---

### Proof of Concept

1. At time `T`, a Lazer relayer captures a validly-signed update blob (e.g., BTC price = $50,000).
2. At time `T + 3600` (one hour later), BTC price is $60,000 on-chain.
3. The relayer calls `verifyUpdate{value: fee}(oldBlob)`.
4. `isValidSigner` passes because the signer key has not expired.
5. `verifyUpdate` returns the payload containing the stale $50,000 price.
6. A consumer DeFi contract (e.g., a lending protocol) uses this price to allow under-collateralized borrowing, causing a loss of funds.

No check inside `verifyUpdate` prevents steps 3–5. [3](#0-2)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
