Audit Report

## Title
Stale Pre-Eviction Stat Overwrite in `add_entry` Permanently Inflates `total_tx_size` — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the new aggregate statistics (`total_tx_size`, `total_tx_cycles`) are captured as local variables before `check_and_record_ancestors` may evict entries. Eviction correctly decrements `self.total_tx_size` via `update_stat_for_remove_tx`, but the stale pre-eviction snapshot is then unconditionally written back at lines 218–219, re-inflating the counter by the total size of all evicted entries. The invariant `total_tx_size == Σ entry.size` is broken, causing `limit_size` to evict legitimate transactions that would otherwise fit within the pool capacity.

## Finding Description
The exact sequence in `add_entry` (lines 200–221):

```
L210-211: let (total_tx_size, total_tx_cycles) =
              self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
          // Snapshot: total_tx_size = self.total_tx_size + entry.size
          // self.total_tx_size is still T (pre-eviction)

L213:     evicts = self.check_and_record_ancestors(&mut entry)?;
          // → remove_entry_and_descendants → remove_entry (L263)
          //   → update_stat_for_remove_tx(evicted.size, evicted.cycles) (L247)
          //   self.total_tx_size is now correctly T − X

L218-219: self.total_tx_size  = total_tx_size;   // ← writes T + E (stale)
          self.total_tx_cycles = total_tx_cycles; // ← writes stale value
          // Correct value would be T − X + E; actual is T + E
```

`updated_stat_for_add_tx` (lines 711–728) is a pure read that returns `self.total_tx_size + tx_size` without mutating state. `update_stat_for_remove_tx` (lines 733–758) mutates `self.total_tx_size` in place. Because the snapshot is taken before evictions and applied after, the eviction decrements are silently discarded.

The eviction branch is reached inside `check_and_record_ancestors` (lines 603–625) when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. In that branch, `remove_entry_and_descendants` is called for each cell-ref-parent candidate, each of which calls `remove_entry` → `update_stat_for_remove_tx`.

`recompute_total_stat` (the only corrective path) is only triggered on underflow (lines 742–749), which does not occur here because the counter is inflated, not underflowed. The inflation therefore persists indefinitely until node restart.

## Impact Explanation
`limit_size` (pool.rs lines 292–329) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts lowest-fee-rate entries. With `total_tx_size` inflated by `X` bytes, `limit_size` evicts `X` bytes worth of legitimate pending/proposed transactions that would otherwise have remained. Each attacker-triggered eviction event adds to the inflation; the effect accumulates across repeated triggers. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, because legitimate transactions are silently dropped from the mempool, degrading transaction propagation and block inclusion across targeted nodes.

## Likelihood Explanation
Any unprivileged peer or RPC caller can submit transactions. The trigger requires a chain of ~1 001 transactions where at least one ancestor is a cell-ref-parent (a pooled transaction referencing as `cell_dep` an output the new transaction spends as an input). The attacker pays fees for ~1 001 transactions but recovers most CKB when the chain is committed or expires. The inflation persists after the chain is gone, so a single successful trigger permanently degrades pool accounting on the targeted node until restart. The attack is repeatable and the inflation accumulates.

## Recommendation
Move the stat computation to after all evictions have completed, so the snapshot reflects the true post-eviction pool state. The simplest correct fix is to replace the pre-eviction snapshot with a direct in-place increment applied after `check_and_record_ancestors` returns:

```rust
// Remove the pre-eviction snapshot entirely.
// After evictions have correctly decremented self.total_tx_size:
self.total_tx_size = self.total_tx_size.checked_add(entry.size)
    .expect("overflow checked before entry");
self.total_tx_cycles = self.total_tx_cycles.checked_add(entry.cycles)
    .expect("overflow checked before entry");
```

The overflow guard (`updated_stat_for_add_tx`) can still be called at the top for early rejection, but its return value must not be used to overwrite the field after evictions have occurred.

## Proof of Concept
1. Fill the pool with a chain `T1 → T2 → … → T1000` where `T1` also references cell `C` as a `cell_dep`.
2. Submit `T1001` spending cell `C` as an input with `T1000` as parent. Ancestor count = 1 001 > 1 000; `T1` is a cell-ref-parent; the branch at line 603 fires and `T1` (plus descendants) are evicted via `remove_entry_and_descendants`.
3. `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` by `size(T1)` during eviction.
4. Lines 218–219 overwrite `self.total_tx_size` with the pre-eviction snapshot, re-adding `size(T1)`.
5. Query `tx_pool_info` RPC: `total_tx_size` exceeds the true sum of entry sizes by `size(T1)`.
6. Submit any new transaction: `limit_size` fires and evicts a legitimate transaction that would otherwise have fit.
7. Repeat steps 1–6 to accumulate inflation and progressively drain the pool of legitimate transactions.

A unit test can assert the invariant `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum::<usize>()` immediately after `add_entry` returns in the eviction branch, which will fail against the current code.
