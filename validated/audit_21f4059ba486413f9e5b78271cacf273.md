### Title
Unauthenticated RPC Setter Functions Allow Any Caller to Disable Networking or Manipulate the Peer Ban List - (File: rpc/src/module/net.rs)

---

### Summary

The `set_network_active`, `clear_banned_addresses`, and `set_ban` RPC methods in `rpc/src/module/net.rs` perform critical node-state mutations with no authentication or caller-identity check. Any process or remote host that can reach the RPC endpoint can invoke them freely, mirroring the original report's pattern of unprotected setter functions.

---

### Finding Description

`NetRpc` exposes three mutating endpoints with no access-control guard:

1. **`set_network_active(state: bool)`** — passes the caller-supplied boolean directly to `network_controller.set_active(state)`. A single call with `false` shuts down all P2P activity.
2. **`clear_banned_addresses()`** — calls `network_controller.clear_banned_addrs()` unconditionally, wiping the entire ban list in one request.
3. **`set_ban(address, command, …)`** — inserts or deletes arbitrary IP/subnet entries from the ban list with no identity check.

None of these functions inspect the caller's identity, require a token, or check any configuration flag before executing. The implementations are: [1](#0-0) [2](#0-1) [3](#0-2) 

The trait declarations confirm no guard is present at the interface level either: [4](#0-3) [5](#0-4) [6](#0-5) 

---

### Impact Explanation

| Call | Effect |
|---|---|
| `set_network_active(false)` | Completely disables all P2P networking; the node stops syncing, relaying, and propagating transactions/blocks. |
| `clear_banned_addresses()` | Removes every entry from the ban list; previously banned malicious or misbehaving peers can immediately reconnect. |
| `set_ban("x.x.x.x", "delete", …)` | Selectively unbans a specific attacker-controlled peer. |
| `set_ban("x.x.x.x/0", "insert", …)` | Bans the entire IPv4 space, effectively isolating the node from all peers. |

The first two are single-call, zero-argument attacks. The node operator has no in-protocol way to distinguish a legitimate administrative call from a malicious one.

---

### Likelihood Explanation

CKB's RPC server supports binding to non-localhost addresses (a documented, supported configuration used in hosted nodes, cloud deployments, and monitoring setups). When the RPC listen address is `0.0.0.0` or any non-loopback interface, the endpoint is reachable by any network peer. Even on a localhost-only binding, any co-resident process (another container, a compromised dependency, a web service on the same host) can reach it. No credential is required in either case. The attack requires only a single HTTP POST.

---

### Recommendation

**Short term:** Add a configurable API token or shared-secret check to the RPC server. Mutating endpoints (`set_network_active`, `clear_banned_addresses`, `set_ban`, `add_node`, `remove_node`) should be gated behind this check. Alternatively, expose them only on a separate, localhost-only administrative socket distinct from the public read-only RPC socket.

**Long term:** Introduce a capability model in the RPC layer that separates read-only methods from state-mutating ones, and require explicit operator opt-in (with a secret) to enable the mutating surface.

---

### Proof of Concept

```bash
# Disable all P2P networking on a node whose RPC is reachable
curl -s -X POST http://<node-ip>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}'
# -> {"jsonrpc":"2.0","result":null,"id":1}
# Node is now isolated from the entire P2P network.

# Wipe the entire ban list
curl -s -X POST http://<node-ip>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_banned_addresses","params":[],"id":2}'
# -> {"jsonrpc":"2.0","result":null,"id":2}
# All previously banned malicious peers may now reconnect.
```

No authentication, no token, no prior knowledge of node state is required. The calls succeed unconditionally. [7](#0-6)

    fn set_ban(
        &self,
        address: String,
        command: String,
        ban_time: Option<Timestamp>,
        absolute: Option<bool>,
        reason: Option<String>,
    ) -> Result<()> {
        let ip_network = address.parse().map_err(|_| {
            RPCError::invalid_params(format!(
                "Expected `params[0]` to be a valid IP address, got {address}"
            ))
        })?;

        match command.as_ref() {
            "insert" => {
                let ban_until = if absolute.unwrap_or(false) {
                    ban_time.unwrap_or_default().into()
                } else {
                    unix_time_as_millis()
                        + ban_time
                            .unwrap_or_else(|| DEFAULT_BAN_DURATION.into())
                            .value()
                };
                self.network_controller
                    .ban(ip_network, ban_until, reason.unwrap_or_default());
                Ok(())
            }
            "delete" => {
                self.network_controller.unban(&ip_network);
                Ok(())
            }
            _ => Err(RPCError::invalid_params(format!(
                "Expected `params[1]` to be in the list [insert, delete], got {address}"
            ))),
        }
    }
