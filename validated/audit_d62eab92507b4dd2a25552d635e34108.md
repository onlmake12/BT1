### Title
Unbounded Loop in `constructProviderCommitment` Enables DoS of Entropy Provider Reveals — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `constructProviderCommitment` function in `Entropy.sol` performs a `while` loop that iterates exactly `req.numHashes` times. `numHashes` is stored per-request as the distance between the request's assigned sequence number and the provider's `currentCommitmentSequenceNumber` at request time. When a provider leaves `maxNumHashes` at its default value of `0` (which disables the only guard), an unprivileged user can submit enough requests to push `numHashes` to a value that makes any subsequent `reveal` / `revealWithCallback` call exceed the block gas limit, permanently bricking the provider's ability to fulfill outstanding requests.

---

### Finding Description

**Root cause — the unbounded loop:** [1](#0-0) 

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

Every call to `reveal` or `revealWithCallback` reaches `revealHelper`, which calls `constructProviderCommitment(req.numHashes, ...)`.

**How `numHashes` is set at request time:** [2](#0-1) 

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
```

`assignedSequenceNumber` is the provider's monotonically-increasing `sequenceNumber` counter, incremented by every user request. `currentCommitmentSequenceNumber` only advances when the provider explicitly calls `advanceProviderCommitment`. The gap between the two — and therefore `numHashes` — grows by 1 for every request submitted to that provider.

**The disabled guard:** [3](#0-2) 

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

`maxNumHashes` is a provider-configured field that defaults to `0`. When it is `0` the condition `providerInfo.maxNumHashes != 0` is `false`, so the entire guard is skipped. Any provider that has not explicitly called `setMaxNumHashes` is unprotected.

---

### Impact Explanation

Each iteration of the `while` loop executes one `keccak256` over 32 bytes (~30–36 gas). At Ethereum's ~30 M gas block limit, roughly **800 000–1 000 000 iterations** saturate a block. Once `numHashes` for any outstanding request reaches that threshold, the corresponding `reveal` / `revealWithCallback` transaction will always revert with an out-of-gas error regardless of the gas limit supplied by the caller. Because `numHashes` is immutably stored in the request struct at submission time, the provider cannot retroactively reduce it; those requests are permanently un-fulfillable. Users who paid fees for those requests lose their randomness and their funds are locked in the provider's accrued-fee balance.

---

### Likelihood Explanation

- `maxNumHashes` defaults to `0`, so every provider that has not explicitly called `setMaxNumHashes` is vulnerable from the moment of registration.
- Any unprivileged address can call `requestV2` / `requestWithCallback` and pay the provider's fee. On chains where the fee is small (e.g., a few cents), an attacker can accumulate the required request count at low cost.
- The provider need not be negligent: even organic usage (many legitimate users requesting randomness while the provider is slow to call `advanceProviderCommitment`) can push `numHashes` into the danger zone.
- The attack is irreversible for already-submitted requests; the provider cannot "undo" stored `numHashes` values.

---

### Recommendation

1. **Change the default behaviour**: treat `maxNumHashes == 0` as "use a protocol-defined safe default" (e.g., 1 000) rather than "no limit". Alternatively, require providers to set a non-zero `maxNumHashes` during `register`.
2. **Document the risk prominently**: providers must understand that failing to call `setMaxNumHashes` and `advanceProviderCommitment` regularly exposes them to this DoS.
3. **Add a protocol-level cap**: enforce an absolute upper bound on `numHashes` at request time regardless of the provider's setting, so that no single reveal can ever consume more than a safe fraction of the block gas limit.

---

### Proof of Concept

1. Provider `P` calls `register(...)` without ever calling `setMaxNumHashes`. `P.maxNumHashes == 0`.
2. Attacker submits `N` calls to `requestV2(P, gasLimit)`, paying the required fee each time. After each call `P.sequenceNumber` increments by 1; `P.currentCommitmentSequenceNumber` remains at its initial value `S₀`.
3. The `N`-th request is stored with `req.numHashes = S₀ + N − S₀ = N`.
4. When `P` (or anyone) calls `revealWithCallback(P, seqN, ...)`, the EVM must execute `constructProviderCommitment(N, ...)`, looping `N` times.
5. For `N ≈ 900 000` on Ethereum mainnet (~27 M gas for the loop alone), the transaction reverts out-of-gas. The request is permanently stuck; the user's randomness is never delivered.
