### Title
Governance Sequence Non-Strict Ordering Allows Permanent Skip of Earlier VAAs — (`lazer/contracts/cardano/lib/pyth/governance.ak`)

### Summary

The `parse_and_verify_action` function enforces only `seen_sequence < new_sequence` (a "greater than" check), not strict sequential ordering (`seen_sequence + 1 == new_sequence`). Any unprivileged relayer who submits a higher-sequence governance VAA first permanently blocks all lower-sequence VAAs from ever being applied.

### Finding Description

In `parse_and_verify_action`: [1](#0-0) 

```aiken
let seen_sequence = u64.as_int(body.sequence)
expect governance.seen_sequence < seen_sequence   // only requires strictly greater
(
  parser.run(governance_action(), body.payload),
  Governance { ..governance, seen_sequence },
)
```

The check at line 37 accepts **any** sequence number strictly greater than the current `seen_sequence`. It does not require the next sequence to be exactly `seen_sequence + 1`.

The `GovernanceAction` redeemer path in the `pyth_state.update` validator is **open to any caller** — there is no `is_owner` guard on it (unlike `PurgeExpiredWithdrawScripts`): [2](#0-1) 

The VAA must be guardian-signed (verified at line 33 via `parse_and_verify_prepared`), so the attacker cannot forge VAAs. However, all guardian-signed VAAs are publicly available on the Wormhole network. If the Pyth governance emitter has published VAAs S+1 and S+2, a relayer can submit S+2 first. After that, `seen_sequence = S+2`, and any attempt to submit S+1 fails because `S+2 < S+1` is false — permanently.

### Impact Explanation

If VAA S+1 was a critical `UpgradeWithdrawScript` (e.g., reverting a bad upgrade) or `UpgradeSpendScript`, it is permanently unprocessable. The contract is stuck with the old (potentially broken) script. Staking rewards or price-feed withdrawals dependent on the withdraw script remain frozen with no recovery path. [3](#0-2) 

### Likelihood Explanation

- Governance VAAs are rare, but the attack only requires **two** pending VAAs simultaneously (not 1000 as the question frames it).
- The Wormhole network exposes all signed VAAs publicly; any observer can race to submit a higher-sequence VAA first.
- No privileged access is needed — the `GovernanceAction` redeemer path has no owner check.
- The attack is a one-shot, irreversible griefing: once `seen_sequence` advances past S+1, there is no on-chain recovery.

### Recommendation

Replace the non-strict ordering check with strict sequential enforcement:

```aiken
// Before (vulnerable):
expect governance.seen_sequence < seen_sequence

// After (fixed):
expect governance.seen_sequence + 1 == seen_sequence
```

This ensures governance VAAs are always applied in the exact order they were emitted, matching the invariant that no sequence is permanently skippable.

### Proof of Concept

1. Current `seen_sequence = S`.
2. Wormhole governance emits VAA S+1 (e.g., `UpgradeWithdrawScript`) and VAA S+2 (e.g., `UpdateTrustedSigner`). Both are publicly available on the Wormhole network.
3. Attacker submits VAA S+2 to `pyth_state.update` with redeemer `GovernanceAction(vaa_S2)`.
4. On-chain check: `S < S+2` → passes. `seen_sequence` is updated to `S+2`.
5. Governance team attempts to submit VAA S+1. On-chain check: `S+2 < S+1` → **fails**. Transaction is rejected.
6. VAA S+1 (`UpgradeWithdrawScript`) is permanently blocked. The withdraw script cannot be upgraded, and any staking rewards or withdrawals dependent on the new script remain frozen. [4](#0-3)

pub fn parse_and_verify_action(
  vaa: PreparedVAA,
  governance: Governance,
  guardians: Guardians,
) -> (GovernanceAction, Governance) {
  let VAA { body, .. } = vaa.parse_and_verify_prepared(vaa, guardians)
  expect u16.as_int(body.emitter_chain) == governance.emitter_chain
  expect body.emitter_address == governance.emitter_address
  let seen_sequence = u64.as_int(body.sequence)
  expect governance.seen_sequence < seen_sequence
  (
    parser.run(governance_action(), body.payload),
    Governance { ..governance, seen_sequence },
  )
}
