Audit Report

## Title
Non-Deterministic `CURRENT_CYCLES` Syscall Due to `iteration_cycles` Loss on Scheduler Resume — (`script/src/scheduler.rs`, `script/src/syscalls/current_cycles.rs`)

## Summary
`CurrentCycles::ecall` computes its return value as `total_cycles + machine.cycles()`, permanently excluding `iteration_cycles`. After `iterate_outer` clears `iteration_cycles` and calls `process_io()`, internal `resume_vm`/`suspend_vm` calls re-populate `iteration_cycles` with `SPAWN_EXTRA_CYCLES_BASE` charges. When `Scheduler::resume()` is later called, it unconditionally zeros `iteration_cycles` without first folding the saved value into `total_cycles`, permanently discarding those charges. Because `step_cycles` is a node-level tuning parameter, nodes that suspend the scheduler more frequently lose more cycles, causing `CURRENT_CYCLES` to return different values for the same script execution — a consensus deviation.

## Finding Description

**`CURRENT_CYCLES` reads only `total_cycles + machine.cycles()`:**
`CurrentCycles::ecall` loads `self.base` (which is `Arc::clone(&vm_context.base_cycles)`, the same `Arc<AtomicU64>` as `total_cycles`) and adds `machine.cycles()`. `iteration_cycles` is never consulted. [1](#0-0) 

**`iterate_outer` leaves `iteration_cycles` non-zero after `process_io()`:**
The sequence is: (1) run VM via `iterate_inner`, (2) `consume_cycles(self.iteration_cycles)` moves charges to `total_cycles`, (3) `self.iteration_cycles = 0`, (4) `self.process_io()`. Step 4 calls `ensure_vms_instantiated`, which calls `resume_vm` and/or `suspend_vm`. Each such call adds `SPAWN_EXTRA_CYCLES_BASE` directly to `iteration_cycles`, leaving it non-zero at the end of `iterate_outer`. [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

**`FullSuspendedState` captures the non-zero `iteration_cycles`:**
`Scheduler::suspend()` saves `iteration_cycles: self.iteration_cycles` into `FullSuspendedState`, preserving whatever non-zero value was left by `process_io()`. [6](#0-5) 

**`Scheduler::resume()` unconditionally zeros `iteration_cycles` without restoring the saved value:**
After calling `ensure_vms_instantiated` (which may itself add more `SPAWN_EXTRA_CYCLES_BASE` charges), line 242 sets `scheduler.iteration_cycles = 0`. The saved `full.iteration_cycles` is never added to `total_cycles` or restored. Those cycles are permanently discarded. [7](#0-6) 

**The comment's reasoning is partially incorrect:**
The comment at line 239 justifies zeroing `iteration_cycles` because "suspending/resuming a scheduler is not part of execution consensus." This is correct for the scheduler-level overhead itself, but `iteration_cycles` at that point also contains cycles already charged by `resume_vm`/`suspend_vm` during `process_io()` — cycles that are part of VM execution accounting, not scheduler bookkeeping. The code itself acknowledges the broader class of this bug at lines 88–94. [8](#0-7) 

**Divergence across nodes:**
A node with smaller `step_cycles` suspends the scheduler more frequently. Each suspension after a `process_io()` that triggered VM swapping discards one or more `SPAWN_EXTRA_CYCLES_BASE` amounts. A node with larger `step_cycles` (fewer suspensions) retains those cycles in `total_cycles` via the next `iterate_outer`'s `consume_cycles` call. The two nodes return different values from `CURRENT_CYCLES` for identical script execution.

## Impact Explanation
This is a **consensus deviation** — Critical (15001–25000 points). A crafted V2 script can call `CURRENT_CYCLES` (syscall 2042) and branch on whether the result exceeds a threshold straddling the per-suspension `SPAWN_EXTRA_CYCLES_BASE` discrepancy. Nodes with different `step_cycles` produce different exit codes for the same transaction, causing a network split.

## Likelihood Explanation
Moderate-to-high. Script deployment on CKB is fully permissionless. The `step_cycles` parameter legitimately differs across node operators as a performance tuning knob, not a consensus rule. The attacker needs only to publish a transaction using the crafted script — no victim mistakes, no leaked keys, no privileged access required. The bug is documented in source and explicitly preserved, reducing the chance of an emergency patch.

## Recommendation
**(a) Include `iteration_cycles` in `CURRENT_CYCLES`:** Pass `iteration_cycles` into `VmContext` so `CurrentCycles::ecall` computes `total_cycles + iteration_cycles + machine.cycles()`.

**(b) Preserve `iteration_cycles` through scheduler resume:** In `Scheduler::resume()`, instead of unconditionally zeroing `iteration_cycles` at line 242, first add the saved `full.iteration_cycles` into `total_cycles` before zeroing, so no charged cycles are silently dropped. The comment should distinguish between scheduler-level suspend/resume overhead (legitimately unchargeable) and IO-transfer/VM-swap cycles already in `iteration_cycles` (which must be preserved for determinism).

Either fix is consensus-breaking and should be scheduled for the next hardfork superseding Meepo.

## Proof of Concept
1. Write a V2 lock script (RISC-V binary) that: spawns a child VM, creates a pipe, performs a write from child and read from parent (forcing `process_io()` to call `ensure_vms_instantiated` with VM swapping, charging `SPAWN_EXTRA_CYCLES_BASE` into `iteration_cycles`), then calls `CURRENT_CYCLES` (syscall 2042), exits 0 if result ≥ `THRESHOLD`, exits 1 otherwise. Set `THRESHOLD` between the cycle count observed with zero scheduler suspensions and the count observed with one suspension.

2. In a Rust test, construct a `TransactionScriptsVerifier` and verify the same transaction twice:
   - Run A: `step_cycles = u64::MAX` (single pass, no scheduler suspension → `iteration_cycles` never zeroed mid-execution via resume).
   - Run B: `step_cycles = 1` (suspend after every cycle → `iteration_cycles` zeroed on each resume, losing `SPAWN_EXTRA_CYCLES_BASE` per suspension).

3. Assert both runs return the same exit code. The assertion will fail, demonstrating the consensus deviation.
