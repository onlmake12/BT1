### Title
NervosDAO Accumulation Rate (`ar`) Integer Division Truncation Causes Permanent, Irrecoverable Loss of Depositor Interest — (`util/dao/src/lib.rs`)

---

### Summary

In `dao_field_with_current_epoch` (`util/dao/src/lib.rs`), the per-block accumulation-rate increase `ar_increase` is computed with integer (floor) division. The truncated remainder — up to 1 shannon per block — is silently discarded and never redistributed. Because the withdrawal formula in `calculate_maximum_withdraw` multiplies depositor capacity by the same under-grown `ar`, every NervosDAO depositor permanently receives less interest than the protocol intends, with no on-chain mechanism to recover the shortfall.

---

### Finding Description

**Root cause — `ar_increase` truncation (lines 256–261):**

```rust
// util/dao/src/lib.rs  dao_field_with_current_epoch()
let ar_increase128 =
    u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
let current_ar = parent_ar
    .checked_add(ar_increase)
    .ok_or(DaoError::Overflow)?;
```

The true increment is `parent_ar × g2 / C`; the code stores `⌊parent_ar × g2 / C⌋`. The discarded remainder is `0 ≤ ε < 1` per block. Because `ar` is never corrected, the error accumulates monotonically over the life of the chain.

**Compounding truncation — `miner_issuance` (lines 242–246):**

```rust
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
let miner_issuance = Capacity::shannons(...);
let nervosdao_issuance = current_g2.safe_sub(miner_issuance)?;
```

`miner_issuance` is also floored, so `nervosdao_issuance` is slightly *larger* than the true DAO share. This excess is added to `current_s` (line 252–254) but can never be fully claimed by depositors because `ar` is simultaneously under-grown.

**Withdrawal shortfall — `calculate_maximum_withdraw` (lines 152–154):**

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
```

The withdrawal amount is `⌊counted_capacity × withdrawing_ar / deposit_ar⌋`. Because `withdrawing_ar` is already below its true value (from the accumulated `ar_increase` truncation), the depositor receives strictly less than the protocol-intended interest. The shortfall is not queued, not redistributed, and not recoverable.

**No redistribution path exists.** Unlike the primary block reward, which distributes its remainder via `remainder_reward` (one extra shannon to the first `remainder_reward` blocks per epoch — `util/types/src/core/extras.rs` lines 234–241), the `ar_increase` remainder has no analogous carry-forward mechanism.

---

### Impact Explanation

Every NervosDAO depositor loses a small but permanent fraction of their interest on every withdrawal. The loss is bounded per block by 1 shannon in `ar_increase`, but it compounds across all blocks between deposit and withdrawal. For a depositor holding `D` shannons over `N` blocks:

```
max_loss ≈ D × N / ar_0   (ar_0 = 10^16)
```

With `D = 10^17` shannons (1 billion CKB) and `N = 10^7` blocks (~10 years):

```
max_loss ≈ 10^17 × 10^7 / 10^16 = 10^8 shannons = 1 CKB
```

Aggregated across all depositors and the full chain lifetime, the total irrecoverable shortfall can reach tens of thousands of CKB. The excess accumulates in `current_s` (the NervosDAO secondary-issuance tracker) and is permanently unclaimable — the on-chain analog of tokens locked in the reward contract.

---

### Likelihood Explanation

This is a deterministic, protocol-level arithmetic property. It is triggered unconditionally on every block that processes the DAO field update (i.e., every block after genesis). Any user who deposits into NervosDAO and later withdraws is affected. No special conditions, privileged access, or adversarial behavior is required; the entry path is a standard NervosDAO deposit/withdraw transaction pair submitted by any unprivileged transaction sender.

---

### Recommendation

Carry the truncation remainder forward, analogous to how `remainder_reward` is handled for primary block rewards:

1. Store the per-block `ar_increase` remainder in the DAO field (or a separate accumulator).
2. Add the accumulated remainder to `ar_increase` once it reaches 1, ensuring no shannon is permanently discarded.

Alternatively, increase the fixed-point precision of `ar` (e.g., use a 128-bit accumulator internally) so that the per-block rounding error is negligible over the chain's lifetime.

---

### Proof of Concept

1. At block `N`, deposit `D` shannons into NervosDAO. Record `deposit_ar = AR_N`.
2. Let the chain advance `K` blocks. Each block computes:
   ```
   ar_increase = ⌊AR_i × g2_i / C_i⌋   (discards up to 1 shannon)
   ```
3. At block `N+K`, initiate withdrawal. The node computes:
   ```
   withdraw_counted = ⌊D_counted × AR_{N+K} / AR_N⌋
   ```
   Because `AR_{N+K}` is at most `K` shannons below its true value, the depositor receives up to `D_counted × K / AR_N` fewer shannons than the protocol intends.
4. The shortfall is not queued in `queuedRewards` or any equivalent structure; it is silently discarded with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
