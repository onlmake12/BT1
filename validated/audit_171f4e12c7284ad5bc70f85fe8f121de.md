### Title
Missing On-Chain Payload Timestamp Staleness Enforcement in `PythLazer.verifyUpdate()` Enables Stale Price Replay - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` verifies only the ECDSA signature and signer-key expiry of a Lazer price update. It never reads or validates the `uint64 timestamp` embedded in the payload against `block.timestamp`. Any unprivileged caller can replay an arbitrarily old but validly-signed Lazer update and receive a successful verification response, causing downstream protocols to consume stale prices as if they were fresh.

---

### Finding Description

`PythLazer.verifyUpdate()` performs the following checks:

1. Fee payment
2. EVM format magic (`0x2A2A2A1A`)
3. ECDSA signature recovery
4. Signer key not expired (`block.timestamp < trustedSignerToExpiresAtMapping[signer]`) [1](#0-0) 

After these checks, the raw `payload` bytes are returned to the caller. The payload begins with a `uint64 timestamp` field, as parsed by `PythLazerLib.parsePayloadHeader`: [2](#0-1) 

The `Update` struct confirms `timestamp` is a first-class field of every Lazer payload: [3](#0-2) 

`verifyUpdate()` never reads this field. There is no `require(payloadTimestamp + MAX_AGE >= block.timestamp)` or equivalent guard anywhere in the contract. The signer-key expiry check (`expiresAt`) governs only the *key's* validity window, not the *data's* freshness.

The official integration documentation instructs consumers to check `_timestamp > timestamp` (monotonicity) but does not instruct them to check `_timestamp` against `block.timestamp`: [4](#0-3) 

This means the contract's own documented usage pattern does not enforce staleness, and the contract itself provides no safety net.

---

### Impact Explanation

Any protocol that calls `verifyUpdate()` and uses the returned payload for financial decisions (lending, liquidation, derivatives settlement, collateral valuation) can be exploited with a stale price:

- **Lending/collateral protocols**: An attacker replays a Lazer update from when an asset was priced higher, borrows against inflated collateral, and defaults when the real price is lower — identical to the `Pawn.sol` scenario in the external report.
- **Liquidation avoidance**: A borrower whose position is currently undercollateralized replays an old update showing a higher collateral price to block liquidation.
- **Derivative settlement manipulation**: A trader replays a favorable historical price to settle a position at an advantageous rate.

The signer key's `expiresAt` can be months in the future (governance-controlled rotation schedule), so the replay window is effectively unbounded within that period.

---

### Likelihood Explanation

- **Attacker-controlled entry**: Any EOA can call `verifyUpdate()` with a historical update blob. No privileged role is required.
- **Low cost**: The only cost is the `verification_fee` (initialized to `1 wei`) plus gas.
- **Historical data availability**: Lazer updates are streamed over WebSocket and can be captured and stored by any observer. Replaying them on-chain requires no special access.
- **No replay prevention in the contract**: There is no nonce, no sequence number, and no timestamp bound enforced by `verifyUpdate()`.

---

### Recommendation

Add an on-chain staleness bound inside `verifyUpdate()`. After extracting the payload, parse the leading 12 bytes (4-byte magic + 8-byte timestamp) and enforce a configurable maximum age:

```solidity
// After extracting payload at line 92:
uint32 PAYLOAD_MAGIC = 2479346549;
require(uint32(bytes4(payload[0:4])) == PAYLOAD_MAGIC, "invalid payload magic");
uint64 payloadTimestamp = uint64(bytes8(payload[4:12]));
require(
    block.timestamp <= payloadTimestamp + maxPayloadAgeSeconds,
    "payload too old"
);
```

`maxPayloadAgeSeconds` should be a governance-settable parameter (e.g., defaulting to 60 seconds for RealTime channel, longer for fixed-rate channels). This mirrors the `getPriceNoOlderThan()` staleness pattern used throughout the rest of the Pyth ecosystem. [5](#0-4) 

---

### Proof of Concept

1. At `T=0`, the Lazer signer signs a payload with `timestamp=T` and `BTC/USD = $100,000`. An observer captures this update blob.
2. At `T=3600` (1 hour later), BTC/USD has dropped to `$80,000`. The signer key has not expired.
3. The attacker calls `pythLazer.verifyUpdate{value: 1 wei}(oldUpdateBlob)`.
4. `verifyUpdate` checks: fee ✓, magic ✓, signature ✓, `block.timestamp < expiresAt` ✓. Returns `payload` with `timestamp=T` and `price=$100,000`.
5. The integrating lending protocol receives the "verified" payload, parses `price=$100,000`, and allows the attacker to borrow against $100,000 collateral.
6. The attacker defaults; the protocol recovers only $80,000 worth of collateral.
7. Net loss: $20,000 per position, scalable across all open positions. [6](#0-5) [2](#0-1)

        // Require fee and refund excess

        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
