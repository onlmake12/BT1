### Title
Permissionless Free Provider Registration Enables Permanent `requestsOverflow` Slot Griefing, Increasing Gas Costs for All Entropy Users - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The `register()` function in `Entropy.sol` imposes no fee, no minimum stake, and no rate limit. Combined with the fixed 32-slot `requests[32]` array in `EntropyState.sol`, an attacker can register a fake provider at zero cost, make 32 targeted requests (paying only the protocol fee), never reveal them, and permanently force every subsequent legitimate request into the expensive `requestsOverflow` mapping — permanently increasing gas costs for all Entropy users.

### Finding Description

**Root cause 1 — Free, permissionless `register()`:**

The `register()` function accepts any `feeInWei` value including `0`, requires no deposit, and has no rate limiting:

```solidity
function register(
    uint128 feeInWei,       // attacker sets to 0
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,     // attacker sets to type(uint64).max
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    ...
    provider.feeInWei = feeInWei;
    ...
    provider.sequenceNumber += 1;
``` [1](#0-0) 

**Root cause 2 — Fixed 32-slot request array:**

`EntropyState.sol` defines only 32 slots for in-flight requests:

```solidity
uint8 public constant NUM_REQUESTS = 32;
bytes1 public constant NUM_REQUESTS_MASK = 0x1f;
...
EntropyStructsV2.Request[32] requests;
mapping(bytes32 => EntropyStructsV2.Request) requestsOverflow;
``` [2](#0-1) 

**Root cause 3 — `requestKey` maps to only 5 bits:**

```solidity
function requestKey(address provider, uint64 sequenceNumber)
    internal pure returns (bytes32 hash, uint8 shortHash) {
    hash = keccak256(abi.encodePacked(provider, sequenceNumber));
    shortHash = uint8(hash[0] & NUM_REQUESTS_MASK);  // 5 bits → 32 slots
}
``` [3](#0-2) 

**Root cause 4 — `allocRequest` pushes evicted requests to expensive overflow storage:**

```solidity
function allocRequest(address provider, uint64 sequenceNumber)
    internal returns (EntropyStructsV2.Request storage req) {
    (, uint8 shortKey) = requestKey(provider, sequenceNumber);
    req = _state.requests[shortKey];
    if (isActive(req)) {
        // Overflow the prior request to the requestsOverflow mapping.
        // This operation is expensive, but should be rare.
        (bytes32 reqKey, ) = requestKey(req.provider, req.sequenceNumber);
        _state.requestsOverflow[reqKey] = req;
    }
}
``` [4](#0-3) 

The comment explicitly acknowledges: *"This operation is expensive, but should be rare. If overflow happens frequently, increase the size of the requests array."*

### Impact Explanation

Once all 32 slots are permanently occupied by the attacker's unrevealed requests, every new `requestV2()` / `request()` call from any legitimate user triggers an SSTORE from zero-to-nonzero into `requestsOverflow` — costing an additional ~20,000 gas per request. On high-throughput chains this permanently degrades the economic viability of the Entropy protocol for all integrators. The attacker's requests can never be evicted because `reveal()` requires `req.requester == msg.sender`, so only the attacker can clear their own slots, and they simply never do. [5](#0-4) 

### Likelihood Explanation

The attack requires:
1. Calling `register()` once — costs only EVM gas, no protocol fee.
2. Making at most ~112 requests (birthday-problem bound to cover all 32 slots) against the attacker's own provider — each costs only `pythFeeInWei` (the protocol fee), since the attacker sets their own `providerFee = 0`.
3. Never calling `reveal()` — passive inaction.

On low-fee chains (Arbitrum, Base, BNB Chain) where `pythFeeInWei` is small, the total attack cost is negligible. The attack is permanent and irreversible without a contract upgrade.

### Recommendation

1. **Require a non-refundable registration deposit** in `register()` to raise the cost of creating fake providers.
2. **Increase `NUM_REQUESTS`** from 32 to a larger value (e.g., 256 or 1024) to make slot exhaustion economically prohibitive.
3. **Enforce a minimum `feeInWei`** for registered providers, or require that requests against a provider carry a minimum fee above `pythFeeInWei` alone.
4. **Add a per-address request cap** or require a bond that is slashed if requests are not revealed within a timeout window.

### Proof of Concept

```python
# Pseudocode — attacker script

# Step 1: Register a fake provider with zero provider fee and max chain length
entropy.register(
    feeInWei=0,
    commitment=keccak256(b"fake"),
    commitmentMetadata=b"",
    chainLength=2**64 - 1,
    uri=b"http://attacker.invalid"
)
# Cost: only EVM gas, no protocol fee

# Step 2: Precompute which sequence numbers map to each of the 32 slots
attacker_addr = <attacker_address>
covered_slots = set()
seq = 1
requests_to_make = []
while len(covered_slots) < 32:
    slot = keccak256(attacker_addr + seq.to_bytes(8,'big'))[0] & 0x1f
    if slot not in covered_slots:
        covered_slots.add(slot)
        requests_to_make.append(seq)
    seq += 1

# Step 3: Submit exactly those requests, paying only pythFeeInWei each
for _ in requests_to_make:
    entropy.requestV2{value: pythFeeInWei}(attacker_addr, random_bytes32(), 0)

# Step 4: Never call reveal() — slots remain occupied forever
# Result: every subsequent legitimate requestV2() call incurs +~20k gas overflow cost
```

The `requestKey` slot mapping is deterministic and publicly computable off-chain, so the attacker can identify the exact sequence numbers needed to cover all 32 slots before submitting any transaction. [3](#0-2) [6](#0-5)
