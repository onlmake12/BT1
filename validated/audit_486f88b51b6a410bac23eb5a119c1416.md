### Title
Integer Overflow in Limit Check Enables Unbounded Vec Allocation via Crafted `GetLastStateProof` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged remote peer can send a single `GetLastStateProof` P2P message with `last_n_blocks = 2^63` to trigger a two-stage exploit: (1) the size-limit guard overflows to zero and passes, and (2) the reorg branch allocates a `Vec` of length proportional to the chain height, causing an OOM crash of the full-node process.

---

### Finding Description

**Stage 1 — Limit check overflow (line 201)**

```rust
// get_last_state_proof.rs:199-205
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT          // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

On a 64-bit host `usize` is 64 bits. With `last_n_blocks = 2^63`:

- `last_n_blocks as usize` = `9_223_372_036_854_775_808`
- `* 2` = `2^64` → wraps to **`0`** in Rust release mode (no `checked_mul`/`saturating_mul` is used)
- `0 + 0 = 0 > 1000` → **false** → guard passes [1](#0-0) [2](#0-1) 

**Stage 2 — Unbounded reorg Vec allocation (lines 237-247)**

```rust
// get_last_state_proof.rs:237-247
let reorg_last_n_numbers = if start_block_number == 0
    || snapshot.get_ancestor(&last_block_hash, start_block_number)
        .map(|header| header.hash() == start_block_hash)
        .unwrap_or(false)
{
    Vec::new()
} else {
    let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
    (min_block_number..start_block_number).collect()   // ← unbounded
};
```

The reorg branch is entered when `start_block_hash` does not match the canonical ancestor at `start_block_number` — trivially achievable by the attacker by supplying any non-matching hash. With `last_n_blocks = 2^63` and `start_block_number = N` (any value ≤ chain height):

- `min(N, 2^63) = N` (since every real chain height is < 2^63)
- `min_block_number = N - N = 0`
- `(0..N).collect::<Vec<_>>()` → allocates a `Vec` of **N** `u64` entries [3](#0-2) 

**Stage 3 — Expensive follow-up work**

The resulting `reorg_last_n_numbers` is chained into `block_numbers` and passed to `complete_headers`, which calls `snapshot.get_ancestor` for every entry — amplifying the attack from a memory allocation into a sustained CPU/IO storm. [4](#0-3) 

---

### Impact Explanation

A single unauthenticated P2P message can cause the full-node process to attempt allocating a `Vec<u64>` of up to `chain_height` entries (e.g., 5000 entries × 8 bytes = 40 KB for the Vec alone, but the subsequent `complete_headers` loop performs one `get_ancestor` DB lookup per entry, making this a sustained resource exhaustion). On a chain of meaningful height the combined allocation + I/O load can exhaust memory or stall the node's async runtime, effectively crashing or hanging the process. No authentication, PoW, or privileged role is required.

---

### Likelihood Explanation

The exploit requires only:
1. A valid `last_hash` pointing to any tip block on the main chain (publicly observable).
2. `start_block_number` set to any value ≤ chain height (also publicly observable).
3. `start_hash` set to any value that does not match the canonical ancestor (e.g., all-zeros).
4. `last_n_blocks = 2^63` (a single field in the serialized message).

All four conditions are trivially satisfiable by any peer that can open a P2P connection.

---

### Recommendation

Replace the plain multiplication with a saturating or checked variant in the limit check:

```rust
// Before (vulnerable):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

Additionally, cap `last_n_blocks` before using it in the reorg branch:

```rust
let last_n_blocks = last_n_blocks.min(constant::GET_LAST_STATE_PROOF_LIMIT as u64);
```

This ensures `reorg_last_n_numbers` is bounded by `GET_LAST_STATE_PROOF_LIMIT` regardless of the peer-supplied value. [1](#0-0)

### Proof of Concept

```rust
// Pseudocode unit test
let last_n_blocks: u64 = 1u64 << 63;          // 2^63
let start_block_number: u64 = 5000;

// Simulate limit check (release mode wrapping):
let limit_val = (last_n_blocks as usize).wrapping_mul(2);  // → 0
assert_eq!(limit_val, 0);                                   // passes guard

// Simulate reorg branch:
let min_block_number = start_block_number
    - start_block_number.min(last_n_blocks);               // 5000 - 5000 = 0
let reorg_last_n_numbers: Vec<u64> = (min_block_number..start_block_number).collect();
assert_eq!(reorg_last_n_numbers.len(), 5000);              // unbounded, not ≤ 1000
```

With a real chain of height 500,000 and `start_block_number = 250,000`, the Vec would contain 250,000 entries, and `complete_headers` would issue 250,000 `get_ancestor` calls — all from a single P2P message.
