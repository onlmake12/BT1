Audit Report

## Title
Asymmetric Descendant-Weight Accounting in `remove_entry_and_descendants` Allows Inflation of Ancestor `EvictKey` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::remove_entry_and_descendants` strips all link records for the removed subtree before invoking `remove_entry` on each entry. Because `update_ancestors_index_key` resolves ancestors through `self.links`, and those links are already gone, surviving ancestors never receive the corresponding `sub_descendant_weight` call. An unprivileged attacker can exploit this asymmetry to repeatedly inflate a parent transaction's `descendants_fee`, making it appear high-value to the eviction comparator and rendering it permanently eviction-resistant at negligible cost.

## Finding Description

**Add path (correct):** `add_entry` calls `record_entry_descendants`, which ends with `update_ancestors_index_key(entry, EntryOp::Add)`. [1](#0-0) 

`update_ancestors_index_key` resolves ancestors via `self.links.calc_ancestors` and calls `add_descendant_weight(child)` on each one. [2](#0-1) 

**Remove path (broken):** `remove_entry_and_descendants` first calls `remove_entry_links` for every entry in the removed subtree, then calls `remove_entry` for each. [3](#0-2) 

`remove_entry_links` removes the entry's own record from `self.links.inner` and also removes it from its parents' children sets. [4](#0-3) 

After this, when `remove_entry` calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)`: [5](#0-4) 

`calc_ancestors` calls `calc_relative_ids`, which looks up `self.links.inner.get(short_id)`. Since the entry was already removed from `self.links`, this returns `None` and `unwrap_or_default()` yields an empty set. [6](#0-5) 

The loop over ancestors is therefore empty; `sub_descendant_weight` is never called on any surviving ancestor. The comment in `remove_entry_and_descendants` acknowledges the intent to suppress `update_descendants_index_key` (correct, since descendants are also being removed), but the same link teardown inadvertently suppresses `update_ancestors_index_key` as well (incorrect, since ancestors survive).

**Trigger path:** `resolve_conflict` is called on every `send_transaction` RPC call when the submitted transaction spends an input already claimed by a pool entry. It calls `remove_entry_and_descendants` on the conflicting entry, leaving its ancestors with stale, inflated descendant weights. [7](#0-6) 

## Impact Explanation

The stale `descendants_fee` / `descendants_size` / `descendants_cycles` / `descendants_count` fields feed directly into `EvictKey` computation: [8](#0-7) 

`EvictKey` ordering selects the entry with the lowest `fee_rate` for eviction first. [9](#0-8) 

An inflated `descendants_fee` raises `descendants_feerate`, which raises `EvictKey.fee_rate`, pushing the parent to the back of the eviction queue. A transaction with a real fee rate of 0.01 shannon/byte can be made to appear as if it has an arbitrarily high fee rate, making it immune to eviction. Repeated across many parent transactions, this consumes pool capacity and distorts fee estimation, matching the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation

The attack requires only the ability to call `send_transaction` RPC — available to any unprivileged user. The attacker needs:
- One UTXO to fund `tx_parent` (stays in pool)
- A supply of UTXOs (UTXO_B_1, UTXO_B_2, …) to serve as the conflict trigger

Each cycle: submit `tx_child_N` spending `tx_parent`'s output and `UTXO_B_N`; submit `tx_conflict_N` spending `UTXO_B_N`. The conflict removes `tx_child_N` without decrementing `tx_parent.descendants_fee`. The conflict transactions themselves can carry minimal fees and may never be confirmed, so the on-chain cost per cycle approaches zero. The cycle is repeatable in a tight loop.

## Recommendation

Before erasing any link records in `remove_entry_and_descendants`, update the surviving ancestors of the root entry while links are still intact. Concretely:

1. Before the `remove_entry_links` loop, call `update_ancestors_index_key(root_entry, EntryOp::Remove)` for the root (and for each removed entry whose direct parents are not also in the removed set) while `self.links` is still consistent.
2. Only then proceed to strip links and call `remove_entry`.

Alternatively, pass a pre-computed ancestor set into `remove_entry` so that the link teardown order does not affect ancestor accounting.

## Proof of Concept

**Setup:**
- `tx_parent` in pool: `fee = 1 shannon`, `size = 100`. Initial state: `descendants_fee = 1`, `descendants_count = 1`.
- Attacker holds UTXOs: `UTXO_B_1, UTXO_B_2, …`

**Cycle (repeat N times):**

1. Submit `tx_child_N` spending `tx_parent`'s output and `UTXO_B_N` with `fee = 1000 shannons`.
   - `add_entry` → `update_ancestors_index_key(tx_child_N, Add)` → `tx_parent.descendants_fee += 1000`.

2. Submit `tx_conflict_N` spending `UTXO_B_N` (same input as `tx_child_N`'s second input).
   - `resolve_conflict` → `remove_entry_and_descendants(tx_child_N)`:
     - `remove_entry_links(tx_child_N)` erases `tx_child_N` from `self.links`
     - `remove_entry(tx_child_N)` → `update_ancestors_index_key(tx_child_N, Remove)` → `calc_ancestors` returns `{}` → **`tx_parent.descendants_fee` is NOT decremented**

**After N cycles:**
```
tx_parent.descendants_fee  = 1 + N × 1000   (correct value: 1)
tx_parent.EvictKey.fee_rate ≈ N × 1000 / 100  (correct value: ~0.01 shannon/byte)
```

`tx_parent` is now at the back of the eviction queue despite having a real fee rate of `0.01 shannon/byte`. A unit test can assert this invariant by checking `pool_entry.evict_key.fee_rate` before and after the cycle and verifying it does not grow unboundedly. [3](#0-2) [10](#0-9)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

        let ancestors: HashSet<ProposalShortId> =

    /// Update ancestor state for add an entry
    pub fn add_descendant_weight(&mut self, entry: &TxEntry) {
        self.descendants_count = self.descendants_count.saturating_add(1);
        self.descendants_size = self.descendants_size.saturating_add(entry.size);
