### Title
Minimum Fee Rate Check Uses Only Serialized Size, Ignoring Cycles Weight — Allows High-Cycles Transactions to Bypass `min_fee_rate` Enforcement (`tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission check in `check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size as the weight denominator. However, the actual fee rate of a `TxEntry` — used for pool ordering, eviction, and fee statistics — is computed using `get_transaction_weight(size, cycles)`, which is `max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For transactions whose script execution consumes many cycles, the cycles-derived weight dominates and can be orders of magnitude larger than the serialized size. Because the fee check runs before script execution (cycles are unknown at that point), a transaction with a small serialized size but high cycle consumption passes the minimum fee rate gate while its actual effective fee rate is far below the configured threshold.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` enforces the minimum fee rate using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The actual weight used everywhere else — in `TxEntry::fee_rate()`, pool scoring, eviction, and fee statistics — is:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

The processing pipeline in `_process_tx` runs `pre_check` (which calls `check_tx_fee`) **before** `verify_rtx` (which produces the actual cycle count). After verification, the entry is created with the real cycles and no second fee check is performed:

```rust
let (tip_hash, rtx, status, fee, tx_size) = try_or_return_with_snapshot!(ret, snapshot);
// ... verify_rtx runs here, producing verified.cycles ...
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
``` [4](#0-3) 

The `TxEntry::fee_rate()` method then computes the real fee rate using the full weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

**Concrete example:**

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 |
| `cycles * DEFAULT_BYTES_PER_CYCLES` | ≈ 11,940 bytes |
| `get_transaction_weight(200, 70_000_000)` | 11,940 |
| `min_fee_rate` | 1,000 shannons/KW |
| Fee required by `check_tx_fee` | `1000 * 200 / 1000 = 200 shannons` |
| Fee required by actual weight | `1000 * 11940 / 1000 = 11,940 shannons` |
| **Effective fee rate of admitted entry** | `200 * 1000 / 11940 ≈ 16 shannons/KW` |

The transaction is admitted paying 200 shannons but its actual effective fee rate is ~16 shannons/KW — 60× below the 1,000 shannons/KW minimum.

---

### Impact Explanation

1. **Fee rate floor bypass**: Any tx-pool submitter (RPC `send_transaction` or P2P relay) can craft a transaction with a script that consumes many cycles but has a small serialized body. Such a transaction passes `check_tx_fee` while its real fee rate is far below `min_fee_rate`.

2. **Fee rate statistics corruption**: The `FeeRateCollector` and fee estimator algorithms (`weight_units_flow`, `confirmation_fraction`) observe the actual weight-based fee rates of confirmed transactions. Admitted below-minimum entries corrupt these statistics, causing the fee estimator to recommend artificially low fee rates to users. [6](#0-5) 

3. **Pool ordering distortion**: Pool scoring and eviction use the real weight-based fee rate. Admitted below-minimum entries occupy pool space at a lower effective priority than their nominal fee suggests, displacing legitimate transactions. [7](#0-6) 

4. **Miner revenue impact**: Block assembly selects transactions by weight-based fee rate. Below-minimum-fee-rate transactions that were admitted can be included in blocks, reducing miner revenue per unit of block weight consumed.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via the standard `send_transaction` RPC or the P2P relay protocol — both are fully unprivileged. The attacker needs to author a lock or type script that consumes many cycles (e.g., a tight computation loop) while keeping the script binary and transaction structure small. This is straightforward with CKB-VM. The `max_tx_verify_cycles` bound (default `TWO_IN_TWO_OUT_CYCLES * 20 ≈ 70M cycles`) limits the maximum weight inflation factor to roughly `70M * 0.000170571 / typical_tx_size`, which for a 200-byte transaction is ~60×. [8](#0-7) 

---

### Recommendation

Replace the size-only minimum fee check in `check_tx_fee` with a weight-based check. Since cycles are not yet known at pre-check time, one of two approaches should be used:

**Option A — Two-phase check**: Keep the cheap size-only pre-check as a lower bound, then add a second fee rate check after `verify_rtx` returns the actual cycles, using `get_transaction_weight(tx_size, verified.cycles)`.

**Option B — Require declared cycles at admission**: For all submissions (not just relayed ones), require a `declared_cycles` hint and use it in the fee check. The existing `DeclaredWrongCycles` rejection already enforces that declared cycles match actual cycles.

The second check would look like:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

---

### Proof of Concept

1. Author a CKB script that runs a tight loop consuming ~70,000,000 cycles but compiles to a small binary (~100 bytes).
2. Build a transaction with `tx_size ≈ 200` bytes using this script as the lock.
3. Set `outputs_capacity = inputs_capacity - 200` (fee = 200 shannons).
4. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000`.
5. Observe: `check_tx_fee` passes (`200 >= 1000 * 200 / 1000 = 200`).
6. After verification, `TxEntry::fee_rate()` returns `200 * 1000 / 11940 ≈ 16 shannons/KW`, far below the 1,000 shannons/KW minimum.
7. The transaction is admitted to the pool and will appear in fee rate statistics with a sub-minimum fee rate. [9](#0-8) [10](#0-9)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
