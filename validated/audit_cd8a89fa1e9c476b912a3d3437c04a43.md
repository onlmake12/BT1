### Title
Unchecked Return Values of `suspend_chunk_process` / `continue_chunk_process` Silently Leave Tx-Pool Chunk Verifier Permanently Suspended — (`chain/src/verify.rs`)

---

### Summary

In `ConsumeUnverifiedBlocks::start()`, the `Result<(), AnyError>` return values of `suspend_chunk_process()` and `continue_chunk_process()` are unconditionally discarded with `let _ =`. If `continue_chunk_process()` fails silently, the tx-pool's async chunk-verification worker is left permanently suspended, blocking all future large-cycle transaction verification without any error surfaced to the operator or the peer.

---

### Finding Description

`ConsumeUnverifiedBlocks::start()` wraps every block-verification call with a suspend/resume pair sent to the tx-pool's chunk-verification channel:

```rust
// chain/src/verify.rs  lines 81, 101
let _ = self.tx_pool_controller.suspend_chunk_process();
// ... block verification ...
let _ = self.tx_pool_controller.continue_chunk_process();
```

The same pattern appears for the truncate path at lines 110–112. [1](#0-0) 

Both functions are defined in `tx-pool/src/service.rs` and return `Result<(), AnyError>`:

```rust
pub fn suspend_chunk_process(&self) -> Result<(), AnyError> {
    self.chunk_tx.send(ChunkCommand::Suspend)
        .map_err(handle_send_cmd_error).map_err(Into::into)
}
pub fn continue_chunk_process(&self) -> Result<(), AnyError> {
    self.chunk_tx.send(ChunkCommand::Resume)
        .map_err(handle_send_cmd_error).map_err(Into::into)
}
``` [2](#0-1) 

`chunk_tx` is a `watch::Sender<ChunkCommand>`. `watch::Sender::send` returns `Err` when **all receivers have been dropped** — i.e., when the tx-pool service's async chunk-verification task has exited. If the tx-pool service exits or panics between the `suspend` and `continue` calls (or before the `continue` call), `continue_chunk_process()` returns `Err`, which is silently swallowed. The chunk-verification worker is never sent `ChunkCommand::Resume`, so it remains in `Suspend` state indefinitely.

---

### Impact Explanation

The chunk-verification path handles transactions whose cycle count exceeds `max_tx_verify_cycles`. These are queued in the verify queue and processed asynchronously via the `ChunkCommand` channel. If the worker is permanently suspended:

- All large-cycle transactions submitted via P2P relay or local RPC are silently stalled in the verify queue and never admitted to the pool.
- Miners cannot include those transactions in block templates.
- The node continues to accept and relay blocks normally, so the operator has no obvious signal that the tx-pool is degraded.

The secondary risk (if `suspend_chunk_process()` fails silently at line 81) is that block verification proceeds while the chunk worker is still running, creating a window where the chunk worker may be reading tx-pool state concurrently with the block-attachment logic that mutates it.

---

### Likelihood Explanation

The tx-pool service is a long-running Tokio task. Its chunk-verification receiver is dropped only if that task panics or is cancelled. A block relayer (unprivileged peer) that can trigger a panic or abnormal exit in the tx-pool service — for example by submitting a transaction that exercises a bug in the async verification path — would cause the next block's `continue_chunk_process()` call to fail silently, permanently suspending the chunk worker. Because the error is swallowed, the node continues operating in a degraded state with no automatic recovery.

---

### Recommendation

Replace the `let _ =` discards with explicit error handling:

```rust
if let Err(e) = self.tx_pool_controller.suspend_chunk_process() {
    error!("failed to suspend chunk process: {}", e);
}
// ... block verification ...
if let Err(e) = self.tx_pool_controller.continue_chunk_process() {
    error!("failed to resume chunk process: {}", e);
    // attempt recovery or trigger node shutdown
}
```

Apply the same fix to the truncate path (lines 110–112). Consider adding a watchdog that detects a permanently-suspended chunk worker and restarts it.

---

### Proof of Concept

1. Node starts normally; tx-pool service and chunk-verification worker are running.
2. An unprivileged peer relays a block. `ConsumeUnverifiedBlocks::start()` receives it and calls `suspend_chunk_process()` (line 81).
3. Between lines 81 and 101, the tx-pool service's chunk-verification Tokio task exits (e.g., due to a panic triggered by a concurrent large-cycle tx submission).
4. Line 101 executes `continue_chunk_process()`, which calls `chunk_tx.send(ChunkCommand::Resume)`. Because the receiver is dropped, this returns `Err(...)`.
5. `let _ =` discards the error. No log, no recovery.
6. All subsequent large-cycle transactions submitted via `submit_remote_tx` or local RPC are enqueued in the verify queue but never dequeued, because the chunk worker is permanently suspended.
7. The node continues to process blocks and relay compact blocks normally, giving no external indication of the degraded tx-pool state. [3](#0-2) [4](#0-3)
