### Title
Unprivileged Tx Sender Can Manipulate `WeightUnitsFlow` Fee Rate Estimator via Mempool Flooding — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

The `WeightUnitsFlow` fee rate estimator in CKB is susceptible to manipulation by any unprivileged transaction sender. By submitting transactions at targeted fee rate tiers, an attacker can inflate both the historical `flow_speed_buckets` and the live `current_weight_buckets` used in `do_estimate`, causing `estimate_fee_rate` to return an artificially elevated fee rate. This is the direct CKB analog to the TWAP/spread manipulation described in the external report.

---

### Finding Description

The `WeightUnitsFlow` algorithm in `util/fee-estimator/src/estimator/weight_units_flow.rs` estimates the recommended fee rate using two attacker-influenced inputs:

**1. `current_weight_buckets` (live mempool state)** [1](#0-0) 

The live mempool snapshot (`all_entry_info.pending` + `all_entry_info.proposed`) is consumed directly with no manipulation guard. Every transaction accepted into the pool contributes to this snapshot.

**2. `flow_speed_buckets` (historical time-windowed average)** [2](#0-1) 

`accept_tx` unconditionally records every accepted transaction into `self.txs` keyed by `current_tip`. The historical flow is then computed as: [3](#0-2) 

`flow_speed_buckets[i] = total_weight_in_bucket_i_over_historical_blocks / historical_blocks`

This is a direct time-windowed average — the CKB analog of `premium_twap`.

**3. The decision gate** [4](#0-3) 

The algorithm returns the **lowest** bucket index `i` where:
`current_weight[i] + flow_speed[i] * target_blocks <= (MAX_BLOCK_BYTES * 85/100) * target_blocks`

To inflate the estimate, an attacker makes low-fee buckets **fail** this condition by inflating `current_weight[i]` and/or `flow_speed[i]`.

**4. `historical_blocks` window** [5](#0-4) 

For `HighPriority` mode (`target_blocks = 5`), `historical_blocks = 10` blocks (~5 minutes). The attacker only needs to sustain the manipulation for 10 blocks.

---

### Impact Explanation

`estimate_fee_rate` is a public RPC endpoint: [6](#0-5) 

Wallets and applications call this to determine appropriate transaction fees. An attacker who inflates the estimate causes users to overpay fees. Conversely, an attacker who floods the pool with minimum-fee-rate transactions can make the estimator believe the low-fee backlog is enormous, pushing the recommended fee rate upward even when the true market rate is low.

The `removed_weight` threshold per 5 blocks is:
`(MAX_BLOCK_BYTES * 85/100) * 5 = 597,000 * 0.85 * 5 ≈ 2,537,250 bytes`

To make bucket 10 (fee_rate ≈ 10,000 shannons/KW) fail, the attacker needs ~2.5 MB of transactions at fee rate ≥ 10,000 shannons/KW in the pool. Cost: `10,000 * 2,500,000 / 1000 = 25,000,000 shannons ≈ 0.25 CKB`. This is economically trivial.

---

### Likelihood Explanation

- **Entry path**: Any node reachable via `send_transaction` RPC or P2P relay. No privileged role required.
- **Cost**: Submitting ~2.5 MB of transactions at the target fee rate tier costs fractions of a CKB.
- **Persistence**: For `HighPriority` mode, the manipulation window is only 10 blocks. For `NoPriority` mode (`historical_blocks = 256`), the attacker must sustain the attack for ~2 hours but the effect is longer-lasting.
- **No decay**: Unlike `ConfirmationFraction` (which applies a decay factor), `WeightUnitsFlow` has no decay on historical flow data — all transactions within the window are weighted equally. [7](#0-6) 

The only expiry is block-number-based, not volume-based.

---

### Recommendation

1. **Rate-limit per-sender contribution** to `self.txs` in `accept_tx` to prevent a single sender from dominating the historical flow window.
2. **Cap the maximum weight contribution** per block in `flow_speed_buckets` to a fraction of `MAX_BLOCK_BYTES`, analogous to the `clip_max_book_index_spread` safeguard in the report.
3. **Apply a decay or outlier-trimming** to the historical flow data (similar to `ConfirmationFraction`'s `decay_factor`) so that sudden spikes in transaction volume do not permanently skew the estimate.
4. **Document** that `estimate_fee_rate` is advisory and should not be the sole input for fee decisions in high-value applications.

---

### Proof of Concept

**Goal**: Inflate `estimate_fee_rate(HighPriority)` from ~1,000 to ~10,000 shannons/KW.

**Setup**: `target_blocks = 5`, `historical_blocks = 10`, `removed_weight ≈ 2,537,250 bytes`.

**Steps**:

1. Attacker submits ~2,600 transactions, each ~1,000 bytes, at fee rate = 10,000 shannons/KW (fee = 10 shannons per tx). Total fee cost ≈ 26,000 shannons ≈ 0.00026 CKB.

2. These transactions enter the mempool. `accept_tx` records each one into `self.txs[current_tip]`: [2](#0-1) 

3. After 10 blocks, `flow_speed_buckets[10] = 2,600,000 / 10 = 260,000 bytes/block`.

4. In `do_estimate`, for bucket 10:
   - `current_weight[10] ≈ 2,600,000` (attacker's txs still in pool)
   - `added_weight = 260,000 * 5 = 1,300,000`
   - `total = 3,900,000 > 2,537,250` → bucket 10 **fails**

5. Buckets 1–10 all fail. The algorithm returns the first passing bucket above 10, e.g., bucket 11 → fee_rate = 12,000 shannons/KW.

6. Any RPC caller invoking `estimate_fee_rate` with `HighPriority` receives an inflated estimate: [8](#0-7)

    pub(crate) async fn estimate_fee_rate(
        &self,
        estimate_mode: EstimateMode,
        enable_fallback: bool,
    ) -> Result<FeeRate, AnyError> {
        let all_entry_info = self.tx_pool.read().await.get_all_entry_info();
        match self
            .fee_estimator
            .estimate_fee_rate(estimate_mode, all_entry_info)
        {
            Ok(fee_rate) => Ok(fee_rate),
            Err(err) => {
                if enable_fallback {
                    let target_blocks =
                        FeeEstimator::target_blocks_for_estimate_mode(estimate_mode);
                    self.tx_pool
                        .read()
                        .await
                        .estimate_fee_rate(target_blocks)
                        .map_err(Into::into)
                } else {
                    Err(err.into())
                }
            }
        }
    }
