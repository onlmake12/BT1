## Verdict

### Title
Fee Charged Before Freshness Check Permanently Locks Funds on Stale Updates — (`target_chains/sui/contracts/sources/pyth.move`)

### Summary
`update_single_price_feed` deposits the `base_update_fee` into `PriceInfoObject` unconditionally before calling `update_cache`, which contains the freshness guard. If the submitted update is stale (its `publish_time` is not strictly greater than the cached value), the price state is unchanged but the fee is already irrecoverably locked inside the object.

### Finding Description
The ordering in `update_single_price_feed` is:

1. `assert!(base_update_fee <= coin::value(&fee))` — fee sufficiency check
2. `price_info::deposit_fee_coins(price_info_object, fee)` — **fee deposited**
3. `update_cache(…)` → `is_fresh_update(…)` — **freshness check**

If `is_fresh_update` returns `false`, execution falls through silently. No abort, no refund. The `price_info.move` module has no withdrawal function; the `Coin<SUI>` is stored as a dynamic object field and is inaccessible after deposit.

An attacker (or any user) holding a valid but older accumulator VAA (one that Wormhole has not yet consumed) can:
1. Let M_new be applied to the `PriceInfoObject` (by anyone).
2. Call `create_authenticated_price_infos_using_accumulator` with M_old's VAA (valid, unused, older timestamp).
3. Call `update_single_price_feed` with the resulting `HotPotatoVector` and pay the fee.
4. `is_fresh_update` returns `false`; price unchanged; fee permanently locked.

### Impact Explanation
Every stale update call results in `base_update_fee` SUI being permanently frozen inside `PriceInfoObject`. There is no governance, admin, or user-callable path to recover these funds. Over time, repeated stale submissions (accidental or deliberate) accumulate irrecoverable SUI in every affected `PriceInfoObject`. This matches **permanent freezing of unclaimed yield** — the fees are protocol revenue that can never be claimed.

### Likelihood Explanation
Pyth publishes VAAs continuously. Many valid VAAs are never submitted on-chain. Any party holding an unsubmitted older VAA can trigger this after a fresher update has been applied. This requires no privilege, no key leak, and no governance action — only a valid (unused) old Wormhole VAA from a legitimate Pyth data source.

### Recommendation
Move the fee deposit **after** the freshness check, or abort (refund) if `is_fresh_update` returns `false`:

```move
// Option A: charge only on fresh update
if (is_fresh_update(cur_price_info, price_info_object)) {
    price_info::deposit_fee_coins(price_info_object, fee);
    price_info::update_price_info_object(price_info_object, cur_price_info);
} else {
    // return fee to caller or abort
    transfer::public_transfer(fee, tx_context::sender(ctx));
}
```

Alternatively, add a `withdraw_fees` function gated by an admin capability so accumulated fees are not permanently frozen regardless.

### Proof of Concept

    public fun get_balance(price_info_object: &PriceInfoObject): u64 {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
