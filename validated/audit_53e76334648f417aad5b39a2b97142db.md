### Title
Permanent Subscription Deposit Limit Bypass via Repeated `addFunds()` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `addFunds()` function in `Scheduler.sol` checks only the per-transaction `msg.value` against `MAX_DEPOSIT_LIMIT`, not the cumulative `status.balanceInWei`. Since `addFunds()` has no access control and any address can call it, an attacker can bypass the 100 ETH deposit cap for permanent subscriptions by making multiple calls each below the limit, permanently locking an unbounded amount of ETH.

### Finding Description

The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT = 100 ether` cap on permanent subscriptions to prevent excessive ETH from being permanently locked (permanent subscriptions cannot withdraw funds). However, the check in `addFunds()` is performed only on the single-transaction `msg.value`, not on the resulting cumulative `status.balanceInWei`:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    // ...
    // ❌ Only checks per-transaction msg.value, not cumulative balance
    if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }

    // ❌ Cumulative balance grows unboundedly across calls
    status.balanceInWei += msg.value;
    // ...
}
``` [1](#0-0) 

The function has no access control — any address can call it for any subscription ID. The protocol's own test suite explicitly confirms this design: [2](#0-1) 

The `MAX_DEPOSIT_LIMIT` constant is 100 ETH: [3](#0-2) 

By contrast, `createSubscription()` correctly applies the same per-transaction check at creation time: [4](#0-3) 

But neither function ever checks whether `status.balanceInWei` after the deposit would exceed `MAX_DEPOSIT_LIMIT`. The check is on the wrong variable — `msg.value` (the incremental amount) instead of `status.balanceInWei` (the cumulative total).

Withdrawals from permanent subscriptions are permanently blocked: [5](#0-4) 

### Impact Explanation

The `MAX_DEPOSIT_LIMIT` protection for permanent subscriptions is completely nullified. An unprivileged caller can:

1. Create a permanent subscription with exactly `MAX_DEPOSIT_LIMIT` (100 ETH).
2. Call `addFunds(subscriptionId){value: MAX_DEPOSIT_LIMIT}` repeatedly — each call passes the per-tx check.
3. Push `status.balanceInWei` to an arbitrary multiple of 100 ETH (e.g., 10,000 ETH in 100 calls).
4. Since permanent subscriptions cannot withdraw, all deposited ETH is permanently locked in the contract.

The intended economic invariant — that no single permanent subscription can lock more than 100 ETH — is entirely defeated. A subscription manager can lock unlimited ETH in their own permanent subscription, or a third party can grief any permanent subscription by forcing it to accumulate far more ETH than intended.

### Likelihood Explanation

**High.** The entry path requires no privilege: `addFunds()` is a public payable function with no access control. The bypass requires only splitting a large deposit into multiple transactions each ≤ 100 ETH. The protocol's own test (`testAnyoneCanAddFunds`) confirms that any address can call `addFunds()` for any subscription. The bypass is trivially executable by any on-chain actor.

### Recommendation

Replace the per-transaction `msg.value` check with a cumulative balance check after the deposit:

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    // ...
    status.balanceInWei += msg.value;

    // ✅ Check cumulative balance, not per-tx msg.value
    if (params.isPermanent && status.balanceInWei > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }
    // ...
}
```

Apply the same fix to `createSubscription()` for consistency (though the single-call nature there makes it less critical).

### Proof of Concept

```solidity
// Attacker bypasses 100 ETH permanent subscription cap
function testDepositLimitBypass() public {
    // Create permanent subscription at exactly the limit
    SchedulerStructs.SubscriptionParams memory params = createDefaultSubscriptionParams(2, address(reader));
    params.isPermanent = true;
    uint256 maxLimit = scheduler.MAX_DEPOSIT_LIMIT(); // 100 ether

    vm.deal(address(this), maxLimit);
    uint256 subId = scheduler.createSubscription{value: maxLimit}(params);

    // Now bypass the limit by calling addFunds() 10 times with 100 ETH each
    for (uint i = 0; i < 10; i++) {
        vm.deal(address(this), maxLimit);
        // Each call passes the per-tx check (msg.value == MAX_DEPOSIT_LIMIT, not >)
        scheduler.addFunds{value: maxLimit}(subId);
    }

    (, SchedulerStructs.SubscriptionStatus memory status) = scheduler.getSubscription(subId);
    // Balance is now 1100 ETH — 11x the intended limit — permanently locked
    assertEq(status.balanceInWei, maxLimit * 11);
}
```

    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
