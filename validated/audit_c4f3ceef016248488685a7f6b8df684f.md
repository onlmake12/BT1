Audit Report

## Title
Integer Overflow in Limit Guard Bypasses Sample Cap, Enabling O(chain-height) Allocation and DB Reads — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The limit guard at line 201 of `GetLastStateProofProcess::execute` performs an unchecked `usize` multiplication that wraps to zero in Rust release builds when `last_n_blocks = 2^63`. This silently bypasses the 1000-sample cap, allowing any peer that completes the light-client handshake to force the server to allocate a `Vec<u64>` proportional to the entire chain height and perform one `get_ancestor` plus one MMR root computation per entry, causing an OOM crash on mainnet-scale chains.

## Finding Description

**Root cause — line 201:**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
``` [1](#0-0) 

In Rust release builds, integer overflow uses wrapping semantics. With `last_n_blocks = 2^63`:

```
(2^63_usize) * 2  =  2^64 mod 2^64  =  0
```

With an empty `difficulties` vector: `0 + 0 > 1000` → `false` → guard bypassed.

**Downstream unbounded allocation — lines 291–296:** [2](#0-1) 

Because `last_n_blocks = 2^63` exceeds any realistic chain height, `last_block_number - start_block_number <= last_n_blocks` is always true. The server executes:

```rust
let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
```

This allocates a `Vec<u64>` of `last_block_number - start_block_number` entries — O(chain height). At 10 million blocks that is ~80 MB for the index vector alone.

**O(N) DB amplification — `complete_headers`, lines 132–153:** [3](#0-2) 

Every entry in `block_numbers` triggers a `get_ancestor` DB walk and an MMR root computation via `self.snapshot.chain_root_mmr(*number - 1)`, turning the O(N) allocation into O(N) blocking DB reads on the async executor thread.

**Existing checks are insufficient:**
- The only size guard is the overflowing expression at line 201.
- The `difficulties` validation block (lines 252–288) is skipped entirely when `difficulties` is empty.
- There is no independent upper-bound check on the raw `last_n_blocks` value before arithmetic.

The constant is confirmed at: [4](#0-3) 

## Impact Explanation

A single malformed `GetLastStateProof` message causes the full-node light-client server to allocate memory and perform DB I/O proportional to the entire chain height. On mainnet with millions of blocks this exhausts heap memory and crashes the node process. This matches **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**.

## Likelihood Explanation

- The attacker needs only a valid `last_hash` on the main chain (trivially obtained from `SendLastState`) and any `start_hash` that does not match the canonical block at `start_number`.
- No PoW, stake, or privileged role is required — any peer completing the light-client handshake can send this message.
- The overflow value `2^63` is a single fixed constant; no brute-force or fuzzing is needed.
- The light-client protocol is opt-in but enabled in production deployments.
- The attack is repeatable: the node crashes and can be re-targeted after restart.

## Recommendation

Replace the unchecked multiplication with saturating arithmetic before the comparison:

```rust
// Before (vulnerable in release):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
let sample_count = (last_n_blocks as usize)
    .saturating_mul(2)
    .saturating_add(self.message.difficulties().len());
if sample_count > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, add an explicit early rejection on the raw value before any arithmetic:

```rust
if last_n_blocks > constant::GET_LAST_STATE_PROOF_LIMIT as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}
```

## Proof of Concept

```rust
// Craft a GetLastStateProof message against a node with light-client enabled
// and a chain of N blocks (N > 1).
let msg = GetLastStateProof {
    last_hash:           <any valid tip hash on main chain>,   // from SendLastState
    start_hash:          <any hash NOT equal to canonical block at height 1>,
    start_number:        1u64,
    last_n_blocks:       1u64 << 63,   // 2^63 — wraps limit check to 0 in release
    difficulty_boundary: U256::MAX,
    difficulties:        vec![],       // empty → skips difficulty validation block
};
// Release build: server allocates Vec of (N-1) u64s, then performs (N-1)
// get_ancestor + MMR root calls → OOM crash for large N.
// Fixed build: returns MalformedProtocolMessage immediately.
```

To reproduce deterministically: run a CKB node in release mode with light-client enabled, sync to any height > 1, then send the above message from a connected peer. Monitor RSS; the node process will be killed by the OOM killer.
