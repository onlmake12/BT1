### Title
Network Secret Key and Onion Private Key Stored Unencrypted on Filesystem with TOCTOU Permission Race — (File: `util/app-config/src/configs/network.rs`, `util/onion/src/onion_service.rs`)

---

### Summary

CKB writes its node P2P identity key (`secret_key`) and Tor onion hidden-service key (`onion_private_key`) to disk in plaintext with no encryption at rest. The network secret key write path additionally has a TOCTOU race: the file is created with default (umask-derived) permissions, the raw key bytes are flushed, and only then are permissions tightened to `0o400`. Any co-tenant process that wins the race reads the 32-byte secp256k1 private key in the clear. On non-Unix platforms the onion key file receives no permission restriction at all.

---

### Finding Description

**Root cause 1 — `write_secret_to_file` TOCTOU + plaintext storage**

`util/app-config/src/configs/network.rs` lines 265–285:

```rust
pub fn write_secret_to_file(secret: &[u8], path: PathBuf) -> Result<(), Error> {
    fs::OpenOptions::new()
        .create(true)
        .write(true)
        .truncate(true)
        .open(path)                    // ← file created with umask-derived perms (e.g. 0o644)
        .and_then(|mut file| {
            file.write_all(secret)?;   // ← raw key bytes written while file is still world-readable
            #[cfg(unix)]
            {
                use std::os::unix::fs::PermissionsExt;
                file.set_permissions(fs::Permissions::from_mode(0o400))  // ← restricted only after write
            }
            ...
        })
}
``` [1](#0-0) 

The file is opened without specifying creation-time permissions. On Linux the default umask is `0o022`, so the file is born `0o644` (world-readable). The 32-byte raw secp256k1 private key is written while the file still carries those open permissions. Only after `write_all` succeeds does the code call `set_permissions(0o400)`. Any process that opens the file between creation and the `set_permissions` call obtains the plaintext key.

This function is called unconditionally at node startup via `fetch_private_key` → `write_secret_key_to_file`: [2](#0-1) 

And also directly from the `ckb peer-id gen` CLI subcommand: [3](#0-2) 

Even outside the race window, the key is stored as raw binary bytes with zero encryption. Any filesystem backup, snapshot, or disk image taken after node startup captures the live private key.

**Root cause 2 — `create_tor_secret_key` plaintext + missing non-Unix permissions**

`util/onion/src/onion_service.rs` lines 163–197:

```rust
fn create_tor_secret_key(onion_private_key_path: String) -> Result<TorSecretKeyV3, Error> {
    let key = torut::onion::TorSecretKeyV3::generate();
    ...
    #[cfg(unix)]
    {
        use std::os::unix::fs::OpenOptionsExt;
        options = options.mode(0o600);   // ← Unix only; no equivalent for Windows
    }
    let mut file = options.open(&onion_private_key_path)...;
    file.write_all(
        &base64::engine::general_purpose::STANDARD
            .encode(key.as_bytes())      // ← base64 is encoding, not encryption
            .into_bytes(),
    )...;
}
``` [4](#0-3) 

Base64 is trivially reversible. On Windows there is no `#[cfg(not(unix))]` fallback to set the file read-only (contrast with `write_secret_to_file` which does have one). The 64-byte Tor v3 secret key is therefore written to a file with default Windows ACLs — readable by any user in the same group or by `SYSTEM`.

The onion key path is resolved and the file is created during `OnionService::new` at node startup: [5](#0-4) 

---

### Impact Explanation

**Network secret key (`secret_key`):** This 32-byte secp256k1 key is the node's permanent P2P identity. It is loaded by `fetch_private_key` and passed directly into the secio encrypted-transport layer as `local_private_key`: [6](#0-5) 

An attacker who obtains this key can:
- Derive the node's `PeerId` and impersonate it on the P2P network.
- Establish authenticated secio sessions that remote peers believe originate from the legitimate node.
- Facilitate eclipse attacks by presenting the stolen identity to peers that whitelist or trust the victim node.

**Onion private key (`onion_private_key`):** This 64-byte Ed25519 key controls the node's Tor v3 hidden-service address. Theft allows an attacker to stand up a rogue hidden service at the same `.onion` address, silently intercepting or dropping connections from Tor-only peers.

---

### Likelihood Explanation

- **Shared hosting / cloud VMs:** Multi-tenant environments are common for CKB node operators. A co-tenant process running as a different UID can win the TOCTOU race on the `secret_key` file during the millisecond window between `open` and `set_permissions`.
- **Filesystem backups / snapshots:** Any backup agent (e.g., rsync, LVM snapshot, cloud-provider disk snapshot) that runs while the node is live captures the plaintext key regardless of file permissions.
- **Windows deployments:** The onion private key file is created with default ACLs, making it readable by any local user or `SYSTEM` process with no race required.
- **No attacker network access required:** The entry path is purely local filesystem read access, which is within the "supported local CLI/RPC user" scope and mirrors the ZecWallet scenario exactly.

---

### Recommendation

1. **Eliminate the TOCTOU race:** On Unix, use `OpenOptionsExt::mode(0o400)` (or `0o600`) at file-creation time so the file is never born world-readable:
   ```rust
   #[cfg(unix)]
   {
       use std::os::unix::fs::OpenOptionsExt;
       options = options.mode(0o400);
   }
   options.create(true).write(true).truncate(true).open(path)
   ```
2. **Add non-Unix permission restriction for the onion key:** Mirror the `#[cfg(not(unix))]` readonly block already present in `write_secret_to_file` into `create_tor_secret_key`.
3. **Encrypt keys at rest:** Wrap the raw key bytes with a passphrase-derived key (e.g., Argon2 + AES-GCM) or delegate storage to the OS keychain/credential store, so a filesystem read alone is insufficient to recover the private key.

### Proof of Concept

```bash
# Terminal 1 — start CKB node (triggers write_secret_key_to_file on first run)
ckb run -C /tmp/ckb-node &

# Terminal 2 — race the TOCTOU window (or simply read after startup)
# The file is created world-readable until set_permissions fires
inotifywait -e create /tmp/ckb-node/data/network/ && \
  cat /tmp/ckb-node/data/network/secret_key | xxd
# Output: 32 raw bytes of the secp256k1 private key

# Derive peer ID from stolen key (using ckb CLI)
ckb peer-id from-secret --secret-path /tmp/stolen_key
# peer_id: <same as victim node>
```

The stolen key can then be loaded into a custom node binary to impersonate the victim's peer identity on the CKB P2P network.

fn create_tor_secret_key(onion_private_key_path: String) -> Result<TorSecretKeyV3, Error> {
    let key = torut::onion::TorSecretKeyV3::generate();
    info!(
        "Generated new onion service v3 key for address: {}",
        key.public().get_onion_address()
    );

    #[cfg_attr(not(unix), allow(unused_mut))]
    let mut file_options = OpenOptions::new();
    #[cfg_attr(not(unix), allow(unused_mut))]
    let mut options = file_options.create(true).truncate(true).write(true);

    #[cfg(unix)]
    {
        use std::os::unix::fs::OpenOptionsExt;
        options = options.mode(0o600);
    }

    let mut file = options.open(&onion_private_key_path).map_err(|err| {
        InternalErrorKind::Other.other(format!(
            "Failed to open onion private key for writing: {:?}",
            err
        ))
    })?;
    file.write_all(
        &base64::engine::general_purpose::STANDARD
            .encode(key.as_bytes())
            .into_bytes(),
    )
    .map_err(|err| {
        InternalErrorKind::Other.other(format!("Failed to write onion private key: {:?}", err))
    })?;

    Ok(key)
}
