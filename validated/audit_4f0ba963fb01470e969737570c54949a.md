### Title
Unbounded `runtime::spawn` on Victim via Unsynchronized `ConnectionSync` + `pending_delivered` — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
An unprivileged remote attacker can cause the victim CKB node to spawn an unbounded number of long-running async tasks (each running up to 30 seconds and opening up to 24 TCP sockets) by: (1) flooding the victim with `ConnectionRequest` messages using unique spoofed `from` peer IDs to populate `pending_delivered`, then (2) sending matching `ConnectionSync` messages that trigger `runtime::spawn` on the victim for each entry — exhausting file descriptors and the async runtime.

### Finding Description

The three-phase hole-punching protocol is:

```
ConnectionRequest  →  (victim stores in pending_delivered, sends back ConnectionRequestDelivered)
ConnectionRequestDelivered  →  (from-node sends ConnectionSync back toward victim)
ConnectionSync  →  (victim looks up pending_delivered, spawns NAT traversal task)
```

**Phase 1 — Populate `pending_delivered` on the victim.**

When the victim receives `ConnectionRequest(from=from_i, to=victim)`, `respond_delivered()` is called: [1](#0-0) 

The guard checks only whether the **same** `from_peer_id` was seen within `HOLE_PUNCHING_INTERVAL`. With unique `from_i` peer IDs, this check never fires. The entry is inserted unconditionally: [2](#0-1) 

The `forward_rate_limiter` is keyed by `(from, to, item_id)`. With unique `from_i`, every key is fresh — the limiter does not block. The per-session `rate_limiter` (30 req/s) is bypassable by using multiple relay sessions.

**Phase 2 — Send `ConnectionSync` directly to the victim.**

`ConnectionSync` processing in `connection_sync.rs` has **no requirement** that a `ConnectionRequestDelivered` was previously processed. It only looks up `pending_delivered`: [3](#0-2) 

Critically, this is `.get()`, **not** `.remove()`. The entry is never consumed, so the same `from_i` entry can trigger arbitrarily many spawns.

**Phase 3 — Victim spawns unbounded tasks.**

For each matching `ConnectionSync`, the victim calls `runtime::spawn`: [4](#0-3) 

Each spawned task runs `select_ok(tasks)` where `tasks` contains up to `ADDRS_COUNT_LIMIT=24` futures, each calling `try_nat_traversal()`: [5](#0-4) 

Each `try_nat_traversal` future loops for up to **30 seconds**, opening a new `TcpSocket` on every ~200 ms retry iteration: [6](#0-5) 

### Impact Explanation

- **File descriptor exhaustion**: N tasks × 24 addresses × ~150 sockets/task lifetime = O(N×3600) open FDs. At N=500 this exceeds typical OS limits (1024–65535).
- **Async runtime saturation**: Each `runtime::spawn` occupies a slot in the Tokio thread pool for up to 30 seconds. With enough concurrent tasks the runtime stalls, blocking all other protocol processing.
- **Node crash / unresponsiveness**: Either FD exhaustion causes `TcpSocket::new_*` to return `EMFILE`/`ENFILE` errors propagating into the p2p layer, or the runtime becomes unresponsive to block/tx relay, causing the node to be dropped from the network.

### Likelihood Explanation

- Attacker needs only standard P2P connections to relay nodes — no privileged access, no PoW, no keys.
- Unique `from_i` peer IDs are trivially generated (random bytes).
- The `pending_delivered` map is cleaned only every 5 minutes (`TIMEOUT=5min`, `CHECK_INTERVAL=5min`): [7](#0-6) 

- During that window, the attacker can accumulate thousands of entries and replay `ConnectionSync` for each.
- The `forward_rate_limiter` (1/s per `(from,to,item_id)` triple) is ineffective because each unique `from_i` is a distinct key: [8](#0-7) 

### Recommendation

1. **Change `.get()` to `.remove()`** in `connection_sync.rs` so each `pending_delivered` entry can only trigger one spawn.
2. **Cap the total number of concurrently live NAT traversal tasks** with a semaphore or bounded channel (e.g., max 4–8 concurrent tasks).
3. **Bound `pending_delivered` map size** — reject new `ConnectionRequest` entries once the map exceeds a threshold (e.g., 64 entries).
4. **Rate-limit `ConnectionSync` per victim** (keyed by `to` peer ID), not just per `(from, to, item_id)`.

### Proof of Concept

```
1. Attacker connects to relay_1, relay_2, ..., relay_K (K sessions, each allows 30 req/s).
2. For i in 1..N:
     Send ConnectionRequest(from=random_peer_id_i, to=victim, listen_addrs=[24 routable IPs])
     via relay_j (round-robin).
   → victim.pending_delivered now has N entries.
3. For i in 1..N:
     Send ConnectionSync(from=random_peer_id_i, to=victim) via any relay.
   → victim calls runtime::spawn() N times.
   → Each task opens up to 24 TcpSockets and loops for 30 seconds.
4. At N=500: ~12,000 concurrent sockets → EMFILE or runtime stall → node crash.
5. Repeat step 3 every 30 seconds (before tasks expire) to sustain the attack.
   pending_delivered entries persist for 5 minutes, so step 2 only needs to run once per 5 min.
```
