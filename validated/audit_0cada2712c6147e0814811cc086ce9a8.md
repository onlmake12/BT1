Now I have enough context to analyze the analog. Let me look at one more detail about the `_echoCallback` return type and the Solidity try-catch behavior with no-code addresses.

### Title
Silent Callback Success on No-Code Requester Address in `try`-Block — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.executeCallback` uses a Solidity `try`-`catch` block to invoke `IEchoConsumer(req.requester)._echoCallback`. Because `_echoCallback` returns `void`, a call to any address with no deployed code (an EOA, or a contract that self-destructed after requesting) causes the EVM to return `success = true` with empty return data. Solidity's `try` statement treats this as a successful call, executes the success branch, and emits `PriceUpdateExecuted` — even though no consumer logic ever ran.

---

### Finding Description

In `Echo.sol`, `executeCallback` performs the following pattern:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    // Callback succeeded
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(..., reason);
} catch {
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
``` [1](#0-0) 

The `_echoCallback` function in `IEchoConsumer` is declared with no return value:

```solidity
function _echoCallback(
    uint64 sequenceNumber,
    PythStructs.PriceFeed[] memory priceFeeds
) external { ... }
``` [2](#0-1) 

When the EVM executes a `CALL` to an address with no deployed bytecode, it returns `(success=true, returndata="")`. Because `_echoCallback` has no return type, Solidity's `try` machinery has nothing to ABI-decode from the empty return data and therefore considers the call successful. The `try` block executes, and `emitPriceUpdate` fires the `PriceUpdateExecuted` event:

```solidity
event PriceUpdateExecuted(
    uint64 indexed sequenceNumber,
    address indexed provider,
    bytes32[] priceIds,
    ...
);
``` [3](#0-2) 

`requestPriceUpdatesWithCallback` stores `msg.sender` as `req.requester` with no check that the caller is a contract:

```solidity
req.requester = msg.sender;
``` [4](#0-3) 

This means any EOA can register a request. When the provider later calls `executeCallback`, the try block silently succeeds, and `PriceUpdateExecuted` is emitted as if the consumer received the price data.

---

### Impact Explanation

- `PriceUpdateExecuted` is emitted for a callback that was never delivered to any consumer logic.
- Off-chain systems (indexers, dashboards, keeper bots) that monitor `PriceUpdateExecuted` to confirm price data delivery will incorrectly record the request as fulfilled.
- The `PriceUpdateCallbackFailed` event — the correct signal for a failed delivery — is never emitted, so no recovery or alerting path is triggered.
- A user who paid the full fee expecting their on-chain consumer logic to execute receives no callback, yet the protocol records the interaction as a success.

---

### Likelihood Explanation

- Any EOA can call `requestPriceUpdatesWithCallback` directly; no special role is required.
- A contract-based requester can also self-destruct between the request and the callback execution, producing the same no-code condition at callback time.
- The provider calling `executeCallback` has no incentive to check whether the requester is a contract — they are paid regardless (fee accounting occurs before the try block). [5](#0-4) 

---

### Recommendation

Add an `extcodesize` check on `req.requester` before the try block, or check it at request time in `requestPriceUpdatesWithCallback`:

```solidity
// At request time:
uint256 size;
address requester = msg.sender;
assembly { size := extcodesize(requester) }
require(size > 0, "Requester must be a contract");
```

Alternatively, mirror the approach used in `Executor.sol`, which already performs this check before making an external call:

```solidity
assembly { len := extcodesize(callAddress) }
if (len == 0) revert ExecutorErrors.InvalidContractTarget();
``` [6](#0-5) 

---

### Proof of Concept

1. EOA `alice` calls `requestPriceUpdatesWithCallback(provider, publishTime, priceIds, gasLimit)` with sufficient fee. `req.requester = alice` (an EOA, no code).
2. Provider calls `executeCallback(provider, sequenceNumber, updateData, priceIds)`.
3. Inside `executeCallback`, the try block calls `IEchoConsumer(alice)._echoCallback{gas: gasLimit}(...)`.
4. The EVM issues `CALL` to `alice` (no code) → returns `(true, "")`.
5. Solidity's `try` sees `success=true`, no return value to decode → executes the success branch.
6. `emitPriceUpdate` fires `PriceUpdateExecuted`.
7. No `PriceUpdateCallbackFailed` is ever emitted. Alice's callback logic never ran. Off-chain systems record the request as successfully fulfilled.
