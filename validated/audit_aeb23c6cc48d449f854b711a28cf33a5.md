### Title
Plaintext Database Password Stored in `ckb.toml` Without File-Permission Check - (File: `util/app-config/src/configs/rich_indexer.rs`)

### Summary
CKB's rich-indexer PostgreSQL backend stores the database password in plaintext inside `ckb.toml`. No permission check is performed on `ckb.toml` at startup, and the official documentation and example configuration both show the weak default password `"123456"` without any security warning. An attacker with read access to `ckb.toml` can steal the credential and gain full write access to the rich-indexer PostgreSQL database, corrupting the indexed data that CKB's RPC layer serves to clients.

### Finding Description

`RichIndexerConfig` in `util/app-config/src/configs/rich_indexer.rs` declares a `db_password` field that is deserialized directly from `ckb.toml`: [1](#0-0) 

The hardcoded default is the trivially weak string `"123456"`: [2](#0-1) 

This password is assembled verbatim into the PostgreSQL connection URL in `util/rich-indexer/src/store.rs`: [3](#0-2) 

The official example configuration `resource/ckb.toml` and the rich-indexer README both reproduce `db_password = "123456"` without any security caveat: [4](#0-3) [5](#0-4) 

**Contrast with the network secret key:** CKB already knows how to handle sensitive files correctly. `read_secret_key()` in `util/app-config/src/configs/network.rs` explicitly checks Unix file-mode bits and emits a `warn!` if the secret-key file is world-readable: [6](#0-5) 

`write_secret_to_file()` also enforces `0o400` on creation: [7](#0-6) 

No equivalent permission check exists for `ckb.toml`, which is the file that carries `db_password`. The systemd deployment guide even instructs operators to set `ckb.toml` to mode `644` (world-readable): [8](#0-7) 

A secondary plaintext credential is `tor_password` in `OnionConfig`, also read from `ckb.toml` with no permission check: [9](#0-8) 

### Impact Explanation

An attacker who reads `ckb.toml` (e.g., a co-tenant on the same host, a backup-storage reader, or any process running as a different user on a system where `ckb.toml` is `644`) obtains the PostgreSQL password. With that credential they can connect directly to the rich-indexer database and:

- Delete or corrupt indexed cell/transaction rows, causing `get_cells`, `get_transactions`, and `get_cells_capacity` RPC calls to return wrong or empty results.
- Insert fabricated rows, causing RPC clients to believe cells exist that do not, or that transactions have different scripts/capacities than they actually do.

Applications and wallets that rely on the rich-indexer RPC for UTXO discovery or balance queries would receive silently incorrect data.

### Likelihood Explanation

The systemd guide sets `ckb.toml` to `0644`, making it world-readable by default on multi-user systems. The documented default password is `"123456"`, so even without reading the file an attacker can try the default. No runtime warning is emitted when the file has broad permissions, so operators have no signal that the credential is exposed.

### Recommendation

1. At startup, check the Unix permissions of `ckb.toml` (and any file containing secrets) and emit a `warn!` if the mode allows group or world read, mirroring the existing logic in `read_secret_key()`.
2. Remove the hardcoded default `"123456"` from `default_db_password()`; require the operator to set an explicit value when `db_type = "postgres"`.
3. Support supplying `db_password` via an environment variable or standard input so it need not appear in the config file at all.
4. Update `resource/ckb.toml`, `util/rich-indexer/README.md`, and the systemd guide to document the risk of storing database credentials in a world-readable file.

### Proof of Concept

```bash
# On a system where ckb.toml is 644 (as the systemd guide recommends):
$ ls -la /var/lib/ckb/ckb.toml
-rw-r--r-- 1 ckb ckb 4096 Jun 19 10:00 /var/lib/ckb/ckb.toml

# Any local user can extract the credential:
$ grep db_password /var/lib/ckb/ckb.toml
db_password = "123456"

# Attacker connects directly to the rich-indexer PostgreSQL instance:
$ psql postgres://postgres:123456@127.0.0.1:5432/ckb-rich-indexer

# Corrupt indexed cell data:
ckb-rich-indexer=# DELETE FROM output WHERE id < 1000000;
DELETE 999999

# CKB RPC now returns empty results for those cells:
$ curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"get_cells","params":[...]}'
{"result":{"last_cursor":"","objects":[]}}
```

```rust
pub fn write_secret_to_file(secret: &[u8], path: PathBuf) -> Result<(), Error> {
    fs::OpenOptions::new()
        .create(true)
        .write(true)
        .truncate(true)
        .open(path)
        .and_then(|mut file| {
            file.write_all(secret)?;
            #[cfg(unix)]
            {
                use std::os::unix::fs::PermissionsExt;
                file.set_permissions(fs::Permissions::from_mode(0o400))
            }
            #[cfg(not(unix))]
            {
                let mut permissions = file.metadata()?.permissions();
                permissions.set_readonly(true);
                file.set_permissions(permissions)
            }
        })
}
```

```rust
pub fn read_secret_key(path: PathBuf) -> Result<Option<secio::SecioKeyPair>, Error> {
    let mut file = match fs::File::open(path.clone()) {
        Ok(file) => file,
        Err(_) => return Ok(None),
    };
    let warn = |m: bool, d: &str| {
        if m {
            ckb_logger::warn!(
                "Your network secret file's permission is not {}, path: {:?}. \
                Please fix it as soon as possible",
                d,
                path
            )
        }
    };
    #[cfg(unix)]
    {
        use std::os::unix::fs::PermissionsExt;
        warn(
            file.metadata()?.permissions().mode() & 0o177 != 0,
            "less than 0o600",
        );
    }
    #[cfg(not(unix))]
    {
        warn(!file.metadata()?.permissions().readonly(), "readonly");
    }
```
