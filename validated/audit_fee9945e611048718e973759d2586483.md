Audit Report

## Title
Missing `BLOCK_INVALID` Early-Exit in `HeaderAcceptor::accept()` Causes Redundant Re-Verification of Rejected Headers ‚Äî (File: sync/src/synchronizer/headers_process.rs)

## Summary
`HeaderAcceptor::accept()` lacks an early-exit for `BLOCK_INVALID` status, acknowledged by a `// FIXME` comment at line 301. Because `BLOCK_INVALID` (`1 << 12`) shares no bits with `HEADER_VALID` (`1`), the status guard at line 304 is bypassed for already-rejected headers, causing `HeaderVerifier::verify()` to re-run on each re-submission. The actual impact is a minor performance inefficiency ‚Äî one redundant verification per `SendHeaders` batch ‚Äî not a node-crashing DoS.

## Finding Description
In `HeaderAcceptor::accept()` (lines 295‚Äì358 of `sync/src/synchronizer/headers_process.rs`), the only status-based early-exit is:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) { ... return result; }
```

`BlockStatus::BLOCK_INVALID = 1 << 12 = 4096` and `BlockStatus::HEADER_VALID = 1` share no bits (`4096 & 1 == 0`), so a header stored as `BLOCK_INVALID` in `block_status_map` falls through to `prev_block_check()` (line 324), then `non_contextual_check()` (line 334), which calls `self.verifier.verify(self.header)`.

`HeaderVerifier::verify()` (verification/src/header_verifier.rs, lines 32‚Äì50) runs PoW verification first via `PowVerifier`. For a header with **invalid PoW**, this fails after one Eaglesong hash computation and returns immediately ‚Äî the median-time and epoch checks never execute. For a header with **valid PoW but `version != 0`**, the full verifier runs (PoW + parent lookup + number + epoch + median-time over up to `median_time_block_count` ancestors), but the attacker must mine a valid header to reach this path.

Critically, `HeadersProcess::execute()` (lines 132‚Äì152) returns `StatusCode::HeadersIsInvalid` immediately after `accept_first` returns `Invalid`. The loop over headers 2‚Äì2000 (lines 154‚Äì179) is never reached. This means at most **one** redundant `HeaderVerifier::verify()` call is forced per `SendHeaders` batch, not 2000.

The contrast with `compact_block_process.rs` (lines 259‚Äì261) is valid ‚Äî that path correctly guards with `status.contains(BlockStatus::BLOCK_INVALID)` ‚Äî but the performance gap between the two paths is small given the 1√ó amplification.

## Impact Explanation
**Low (501‚Äì2000 points) ‚Äî Performance improvement for CKB.** The missing guard causes one unnecessary `HeaderVerifier::verify()` call per `SendHeaders` message for an already-rejected header. For invalid-PoW headers this is one Eaglesong hash (nanoseconds). For valid-PoW + bad-version headers the cost is higher but requires the attacker to mine, making the attack economically unattractive. This does not constitute a node-crashing or network-congesting DoS; it is a suboptimal code path confirmed by the codebase's own `// FIXME` comment.

## Likelihood Explanation
Any peer can trigger the redundant re-verification by re-sending a previously rejected header. For invalid-PoW headers the cost to the victim is negligible. For valid-PoW headers the attacker bears significant mining cost. The peer is likely banned after sending invalid headers, further limiting sustained exploitation.

## Recommendation
Add an explicit `BLOCK_INVALID` early-exit at the top of `HeaderAcceptor::accept()`, resolving the existing `// FIXME`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidParent));
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
    return result;
}
```
