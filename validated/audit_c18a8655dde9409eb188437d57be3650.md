Audit Report

## Title
Unbounded `forward_rate_limiter` Memory Growth via Persistent HolePunching Connection — (`network/src/protocols/hole_punching/mod.rs`)

## Summary

`HolePunching` holds a `forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>` backed by `HashMapStateStore`, whose entries are only evicted via `retain_recent()` in `disconnected()`. The periodic `notify()` callback (every 5 minutes) never calls `retain_recent()` on either rate limiter. An attacker who maintains a persistent connection and sends `ConnectionRequest` or `ConnectionRequestDelivered` messages with unique attacker-controlled `(from, to)` PeerId pairs causes `forward_rate_limiter`'s internal `HashMap` to grow without bound, leading to OOM and node crash.

## Finding Description

`HolePunching` declares two rate limiters: [1](#0-0) 

`retain_recent()` is called on both only inside `disconnected()`: [2](#0-1) 

The `notify()` callback, fired every `CHECK_INTERVAL` (5 minutes), cleans `pending_delivered` and `inflight_requests` but **never** calls `retain_recent()` on either rate limiter: [3](#0-2) 

In `received()`, the `rate_limiter` is keyed by `(session_id, msg.item_id())`. Since `msg.item_id()` is the union discriminant (0, 1, or 2), this key space is bounded at 3 entries per session — not a concern: [4](#0-3) 

However, once that check passes, `forward_rate_limiter` is checked with **attacker-controlled** `from` and `to` PeerIds extracted from the message payload: [5](#0-4) [6](#0-5) 

`governor`'s `HashMapStateStore` inserts a new `HashMap` entry for every unique key on `check_key()`. The `forward_rate_limiter` quota is 1 per second per `(from, to, item_id)` key, so each unique pair creates a new entry. The outer `rate_limiter` allows up to 30 `ConnectionRequest` messages per second per session, meaning an attacker can inject 30 unique `(from, to, item_id)` entries per second. Without `retain_recent()` being called in `notify()`, expired entries are never evicted for the lifetime of the connection.

## Impact Explanation

This is a **High** severity vulnerability matching: *"Vulnerabilities which could easily crash a CKB node."*

At 30 entries/second, with each governor entry consuming ~100–200 bytes (two `PeerId` heap allocations + state), a single persistent peer accumulates ~2.6 MB/hour, ~62 MB/day, and ~1.8 GB/month. A single unprivileged peer can exhaust node memory and trigger an OOM crash with no operator-visible warning and no self-healing mechanism.

## Likelihood Explanation

Any unprivileged remote peer that can open a HolePunching protocol session can execute this. No proof-of-work, no key material, no Sybil attack is required. The attacker simply holds the connection open and sends valid-looking `ConnectionRequest` messages with rotating `from`/`to` PeerId bytes at ≤30 msg/s to stay under the `rate_limiter` cap. The `rate_limiter` guard limits throughput but does not prevent key diversity in `forward_rate_limiter`.

## Recommendation

Call `retain_recent()` on both rate limiters inside `notify()` so that stale entries are periodically evicted independent of connection lifecycle:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();           // add
    self.forward_rate_limiter.retain_recent();   // add
    let now = unix_time_as_millis();
    self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
    // ...
}
```

This ensures the `CHECK_INTERVAL` (5 minutes) timer drives reclamation regardless of whether any peer ever disconnects.

## Proof of Concept

1. Connect to a victim CKB node and open the HolePunching protocol session.
2. In a loop, send `ConnectionRequest` messages where `from` and `to` are freshly generated random PeerIds (valid 32-byte ed25519 public keys), at ≤30 msg/s to stay under the `rate_limiter` cap.
3. Monitor the victim node's RSS over time.
4. Assert: RSS grows monotonically; `retain_recent()` is never triggered because `disconnected()` is never called; the `forward_rate_limiter` map grows at ~30 entries/second.
5. After ~24 hours, RSS increase attributable to `forward_rate_limiter` exceeds several hundred MB from this single connection alone.
