### Title
Unauthenticated Miner Notify HTTP Endpoint Allows Injection of Crafted Block Templates, Redirecting Mining Rewards — (File: `miner/src/client.rs`)

---

### Summary

The `ckb-miner` binary supports a "notify mode" where it listens on a configurable HTTP address for block template push notifications from the CKB node. The HTTP handler that receives these notifications performs no authentication or source validation whatsoever. Any party who can reach the miner's notify port can POST a crafted `BlockTemplate` — including one with an attacker-controlled `cellbase_witness.lock` — causing the miner to mine valid blocks that redirect future block rewards to the attacker's lock script.

---

### Finding Description

**Root cause — `miner/src/client.rs`, `handle` function (lines 358–369):**

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }

    Ok(Response::new(Empty::new()))
}
``` [1](#0-0) 

The function accepts every incoming HTTP request unconditionally. There is no check on the request origin, no shared secret, no HMAC, and no comparison against the node's own RPC URL. Any valid JSON body that deserializes as a `BlockTemplate` is immediately accepted and forwarded to the mining workers via `update_block_template`.

**`update_block_template` performs no validation either (lines 293–312):**

```rust
fn update_block_template(&self, block_template: BlockTemplate) {
    let work_id = block_template.work_id.into();
    let updated = |id| {
        if id != work_id || id == 0 {
            Some(work_id)
        } else {
            None
        }
    };
    if self.current_work_id
        .fetch_update(Ordering::SeqCst, Ordering::SeqCst, updated)
        .is_ok()
    {
        let work: Work = block_template.into();
        if let Err(e) = self.new_work_tx.send(Works::New(work)) { ... }
    }
}
``` [2](#0-1) 

The only guard is a `work_id` deduplication check. An attacker bypasses it trivially by using `work_id = 0` (the condition `id == 0` forces an update) or any `work_id` that differs from the current one.

**The notify listener binds to an attacker-reachable address:**

The miner config supports binding the notify listener to any `SocketAddr`:

```rust
pub listen: Option<SocketAddr>,
``` [3](#0-2) 

The documented example in `resource/ckb-miner.toml` shows `listen = "127.0.0.1:8888"`, but operators running mining pools routinely bind this to a network-accessible address so the CKB node can push templates to a remote miner process. [4](#0-3) 

**How the cellbase witness controls future rewards:**

In `tx-pool/src/block_assembler/mod.rs`, `build_cellbase_witness` embeds the miner's lock script into the cellbase witness:

```rust
let cellbase_lock = Script::new_builder()
    .args(config.args.as_bytes())
    .code_hash(&config.code_hash)
    .hash_type(hash_type)
    .build();
``` [5](#0-4) 

And `build_cellbase` uses this witness to construct the cellbase transaction:

```rust
let cellbase_witness = Self::build_cellbase_witness(config, snapshot);
...
let witness = cellbase_witness.as_bytes();
``` [6](#0-5) 

The `cellbase_witness.lock` is the lock script that will receive the block reward for the **current** block when it is finalized in a future block. This field is not validated by the consensus layer — it is arbitrary data from the miner's perspective. The node only validates that the cellbase **output** lock matches the `target_lock` derived from the *previous* block's cellbase witness (a publicly readable on-chain value).

**End-to-end exploit path:**

1. Attacker identifies a miner running with `listen` enabled on a reachable address.
2. Attacker reads the current chain tip to compute the correct `target_lock` (the lock script that must appear in the cellbase output — derived from the previous block's cellbase witness, which is public).
3. Attacker crafts a `BlockTemplate` JSON with:
   - Correct `target_lock` in the `cellbase.data.outputs[0].lock` (so the block passes node verification).
   - Attacker's own lock script embedded in `cellbase.data.witnesses[0]` (the `CellbaseWitness.lock` field).
   - A `work_id` of `0` (or any value differing from the miner's current work ID).
4. Attacker HTTP-POSTs the crafted template to the miner's notify port.
5. The miner's `handle` function accepts it without any check and calls `update_block_template`.
6. Mining workers receive the new work and begin solving PoW on the attacker-controlled template.
7. When a solution is found, the miner calls `submit_block` on the CKB node. The node verifies the block (header PoW, cellbase output capacity, `target_lock` match) — all checks pass because the attacker used the correct `target_lock`.
8. The block is accepted and propagated. Its cellbase witness now contains the attacker's lock script.
9. After the finalization delay (`finalization_delay_length` blocks), the block reward for this block is paid to the attacker's lock script in a future block's cellbase output.

---

### Impact Explanation

**Direct theft of mining rewards.** Every block the miner produces while processing the injected template pays its future block reward to the attacker's lock script instead of the legitimate miner's. The attacker does not need to solve PoW themselves — they redirect the miner's own hashrate to produce blocks that enrich the attacker. The legitimate miner bears all electricity and hardware costs while receiving nothing.

The impact is proportional to the miner's hashrate and the duration before the injection is detected. On a large mining pool with a network-exposed notify endpoint, this could represent continuous theft of all block rewards.

---

### Likelihood Explanation

The `listen` notify mode is a documented, supported feature explicitly intended for mining pool deployments where the CKB node and miner run on separate machines. The config file ships with the option commented out but with a clear example. Any operator following the mining pool setup pattern who binds the listener to a non-loopback address is immediately vulnerable. An attacker needs only network reachability to the port — no credentials, no keys, no prior knowledge beyond the chain's public state.

---

### Recommendation

Add origin authentication to the miner's notify HTTP endpoint. The simplest approach is a shared secret (token) that the CKB node includes in the `Authorization` header when posting templates, and that the miner verifies before accepting any template. Alternatively, restrict the listener to only accept connections from the configured `rpc_url` host. At minimum, document clearly that the notify listener must never be exposed to untrusted networks without an authenticating reverse proxy.
