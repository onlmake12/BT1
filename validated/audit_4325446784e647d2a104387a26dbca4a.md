Audit Report

## Title
Onion3 (Tor) Peers Evade All Score-Based and Identify-Triggered Bans — (`network/src/peer_store/peer_store_impl.rs`, `network/src/peer_store/ban_list.rs`)

## Summary
`ban_addr` and `is_addr_banned` both delegate to `multiaddr_to_socketaddr`, which returns `None` for Onion3 multiaddrs. Consequently, calling `ban_addr` on a Tor-connected peer silently skips the `ban_list` insertion while still removing the peer from `addr_manager`. Because `is_addr_banned` also returns `false` for every Onion3 address, the peer is never blocked from reconnecting, completely nullifying ban enforcement for all Tor-connected peers.

## Finding Description
**Root cause — `ban_addr` skips `ban_list` for Onion3 addresses:**
In `network/src/peer_store/peer_store_impl.rs` lines 286–292, `ban_addr` only calls `ban_network` inside an `if let Some(addr) = multiaddr_to_socketaddr(addr)` branch. For `/onion3/…` multiaddrs, `multiaddr_to_socketaddr` returns `None`, so the branch is never entered and nothing is written to `ban_list`. The `addr_manager.remove(addr)` call on line 291 still executes unconditionally, removing the peer from the known-good address set, but the ban record is never created.

**`is_addr_banned` always returns `false` for Onion3 addresses:**
In `network/src/peer_store/ban_list.rs` lines 68–72, `is_addr_banned` chains `multiaddr_to_socketaddr(addr).map(…).unwrap_or_default()`. For Onion3, `multiaddr_to_socketaddr` returns `None`, `.map(…)` produces `None`, and `unwrap_or_default()` returns `false`. Every ban check for a Tor peer passes.

**`accept_peer` relies on `is_addr_banned` as the sole reconnection gate:**
`network/src/peer_registry.rs` lines 109–111 call `peer_store.is_addr_banned(&remote_addr)` and return `PeerError::Banned` only if it returns `true`. Because it never does for Onion3, the banned Tor peer passes this check on every reconnection attempt.

**`ban_session` (the production ban entry point) calls `ban_addr` with `peer.connected_addr`:**
`network/src/network.rs` lines 264–268 pass `&peer.connected_addr` directly to `ban_addr`. For a Tor-connected peer, `connected_addr` is an Onion3 multiaddr, making the entire ban a no-op in `ban_list`.

**Onion3 is a supported, production-enabled transport:**
The `listen_on_onion` config option is present in `resource/ckb.toml` and `util/app-config/src/configs/network.rs`, and `util/onion/src/onion_service.rs` lines 41–50 construct production Onion3 multiaddrs. Furthermore, `network/src/peer_store/addr_manager.rs` lines 74–82 explicitly special-case Onion3 to keep such peers connectable via `fetch_random`, confirming they are treated as first-class peers.

## Impact Explanation
A misbehaving Tor peer that triggers a ban (via the Identify protocol's wrong-network-name check or score-based misbehavior via `report`) will be disconnected from the current session but will be able to reconnect immediately as an inbound peer, passing the `is_addr_banned` check every time. This allows a Tor peer to cycle through reconnections indefinitely, exhausting inbound connection slots and bypassing all misbehavior enforcement on any node with `listen_on_onion = true`. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (10001–15000 points).

## Likelihood Explanation
- Onion3 support is an opt-in but documented production feature (`listen_on_onion = true`).
- The attacker requires only a Tor client and knowledge of the target node's onion address — no privileged access, no hashpower, no key material.
- The bug is triggered by the normal ban code path; no special message crafting is required beyond whatever behavior normally triggers a ban (e.g., sending an Identify message with a wrong network name).
- The attack is repeatable indefinitely with no cooldown, since no ban record is ever written.

## Recommendation
The ban system must be extended to handle non-IP transports. Two concrete options:

1. **Onion3-keyed ban list**: Store banned Onion3 host bytes (the 35-byte public key) in a separate set alongside the IP-keyed `HashMap<IpNetwork, BannedAddr>` in `BanList`. Update both `ban_addr` and `is_addr_banned` to check/insert this set when `multiaddr_to_socketaddr` returns `None` and the address contains an `Onion3` protocol component.

2. **PeerId-based banning**: Since every connected peer has a `PeerId`, maintain a `HashSet<PeerId>` of banned peer IDs and check it in `accept_peer` before the `is_addr_banned` call.

Either approach must be applied consistently in both the write path (`ban_addr`) and the read/enforcement path (`is_addr_banned` / `accept_peer`).

## Proof of Concept
```rust
use ckb_network::{
    Flags, peer_store::{PeerStore, Behaviour},
    multiaddr::Multiaddr,
};

let mut peer_store = PeerStore::default();
let onion_addr: Multiaddr =
    "/onion3/vww6ybal4bd7szmgncyruucpgfkqahzddi37ktceo3ah7ngmcopnpyyd:1234/p2p/QmXXX"
        .parse().unwrap();

// Simulate discovery/outbound add
peer_store.add_addr(onion_addr.clone(), Flags::all()).unwrap();

// Trigger a ban (as ban_session would do)
peer_store.ban_addr(&onion_addr, 3_600_000, "misbehavior".into());

// Both assertions fail, proving the ban is a no-op
assert!(peer_store.ban_list().count() > 0);      // FAILS: count == 0
assert!(peer_store.is_addr_banned(&onion_addr)); // FAILS: returns false
// The peer reconnects freely, passing accept_peer's is_addr_banned check every time.
```
