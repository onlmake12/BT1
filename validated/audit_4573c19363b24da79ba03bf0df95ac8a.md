### Title
Miner-Controlled Block Timestamp Biases `since`-Based Relative Time Locks — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

CKB miners have direct, protocol-permitted control over the `timestamp` field of every block they produce, within the window `(block_median_time_of_parent, now + 15s]`. After the CKB2021 hardfork, the `SinceVerifier` uses the **raw block timestamp** (not the median) as the `base_timestamp` for relative-timestamp `since` locks. A miner who mines the block in which an input cell is confirmed can therefore shift the starting point of any relative time lock on that cell by up to ~15 seconds in either direction, accelerating or delaying when the cell becomes spendable.

---

### Finding Description

The `BlockTemplate.current_time` field is explicitly documented as miner-adjustable:

> "Miners can increase it to the current time." [1](#0-0) 

The only consensus enforcement is in `TimestampVerifier::verify()`:

- **Lower bound**: `header.timestamp() > block_median_time(parent, 37 blocks)`
- **Upper bound**: `header.timestamp() <= now + ALLOWED_FUTURE_BLOCKTIME` (15 000 ms) [2](#0-1) [3](#0-2) 

This gives every miner a window of up to ~15 seconds in which to freely choose the block timestamp.

In `SinceVerifier::verify_relative_lock()`, when the CKB2021 hardfork flag `is_block_ts_as_relative_since_start_enabled` is active, the `base_timestamp` for a relative-timestamp `since` lock is taken directly from the **raw block timestamp** of the block that confirmed the input cell — not from the median:

```rust
let base_timestamp = if hardfork_switch
    .ckb2021
    .is_block_ts_as_relative_since_start_enabled(epoch_number)
{
    self.data_loader
        .get_header_fields(&info.block_hash)
        .expect("header exist")
        .timestamp          // ← raw miner-set timestamp
} else {
    self.parent_median_time(&info.block_hash)
};
let current_median_time = self.block_median_time(&parent_hash);
let required_timestamp = base_timestamp
    .checked_add(timestamp)
    .ok_or(TransactionError::InvalidSince { index })?;
if current_median_time < required_timestamp {
    return Err((TransactionError::Immature { index }).into());
}
``` [4](#0-3) 

Because `base_timestamp` is the raw block timestamp, the miner who mines block B (the block that confirms the input cell) can:

- Set `B.timestamp` to the **maximum** (`now + 15s`) → `required_timestamp` is pushed up by up to 15 s → the relative lock expires **later** than the user intended.
- Set `B.timestamp` to the **minimum** (just above `median_of_prev_37`) → `required_timestamp` is pulled down → the relative lock expires **earlier** than the user intended.

The `block_median_time` helper that computes the median over 37 blocks is used only for the **current** tip comparison, not for the base: [5](#0-4) 

---

### Impact Explanation

Any transaction that uses a **relative timestamp `since` lock** (flag bits `0xC000...`) and whose input cell was confirmed after the CKB2021 hardfork activation is affected. The miner who produced the confirming block can shift the effective lock expiry by up to `ALLOWED_FUTURE_BLOCKTIME` (15 seconds) in either direction. For time locks measured in seconds (e.g., short-lived payment channels, atomic swaps with tight deadlines), this is a meaningful manipulation window. For locks measured in hours or epochs, the impact is negligible in practice.

The manipulation is **unilateral** — it requires only that the attacker mines the single block that confirms the input cell, with no majority hashpower needed.

---

### Likelihood Explanation

Any miner (pool or solo) who happens to mine the block containing the target input cell can perform this manipulation at zero additional cost — it is a free parameter within the block template. The `get_block_template` RPC explicitly exposes `current_time` as a field miners may adjust: [1](#0-0) 

The `BlockAssembler` sets `current_time` to `max(unix_time_as_millis(), prev_tip_timestamp + 1)`, but miners are free to override this before submission: [6](#0-5) 

---

### Recommendation

For relative-timestamp `since` verification, use the **median time of the block** that confirmed the input cell (i.e., `parent_median_time(&info.block_hash)`) as `base_timestamp` even after CKB2021, rather than the raw block timestamp. This mirrors the pre-CKB2021 behavior and is resistant to single-miner timestamp manipulation because the median of 37 blocks cannot be shifted by more than one block's worth of influence. Alternatively, document the 15-second manipulation bound explicitly in the RFC so that application developers using short relative time locks are aware of the risk.

---

### Proof of Concept

1. Alice creates a cell in block B with a relative-timestamp `since` lock of `T` seconds (e.g., `T = 60`).
2. Mallory is the miner who produces block B. She sets `B.timestamp = now + 14_999 ms` (just under the 15-second cap).
3. `base_timestamp = B.timestamp = now + 14_999 ms`.
4. `required_timestamp = base_timestamp + T * 1000 = now + 14_999 + 60_000 ms`.
5. The lock now expires ~15 seconds **later** than if Mallory had set `B.timestamp` to the minimum.
6. Conversely, if Mallory sets `B.timestamp` to the minimum (just above the 37-block median), `required_timestamp` is ~15 seconds **earlier**, allowing the cell to be spent sooner than Alice intended.

The attacker-controlled entry path is the `submit_block` RPC / block relay, with the manipulated field being `header.timestamp` inside the block that confirms the input cell. [7](#0-6) [8](#0-7)

    pub fn verify(&self) -> Result<(), Error> {
        // skip genesis block
        if self.header.is_genesis() {
            return Ok(());
        }

        let min = self.data_loader.block_median_time(
            &self.header.data().raw().parent_hash(),
            self.median_block_count,
        );
        if self.header.timestamp() <= min {
            return Err(TimestampError::BlockTimeTooOld {
                min,
                actual: self.header.timestamp(),
            }
            .into());
        }
        let max = self.now + ALLOWED_FUTURE_BLOCKTIME;
        if self.header.timestamp() > max {
            return Err(TimestampError::BlockTimeTooNew {
                max,
                actual: self.header.timestamp(),
            }
            .into());
        }
        Ok(())
    }
```

    fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
        let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
        let mut block_hash = block_hash.clone();
        for _ in 0..median_block_count {
            let header_fields = self
                .get_header_fields(&block_hash)
                .expect("parent header exist");
            timestamps.push(header_fields.timestamp);
            block_hash = header_fields.parent_hash;

            if header_fields.number == 0 {
                break;
            }
        }

        // return greater one if count is even.
        timestamps.sort_unstable();
        timestamps[timestamps.len() >> 1]
    }
