Audit Report

## Title
Unauthenticated Peer Can Poison `observed_addrs` via Crafted `IdentifyMessage`, Propagating Attacker-Controlled Addresses Network-Wide — (`network/src/protocols/identify/mod.rs`)

## Summary

An unprivileged inbound peer can send an `IdentifyMessage` containing an arbitrary `observed_addr`. The victim node stores it unconditionally in `NetworkState::observed_addrs`, then includes it in the `listen_addrs` field of every subsequent identify handshake. Receiving peers add those addresses to their peer stores via `add_remote_listen_addrs`, enabling network-wide propagation of attacker-controlled addresses with a single TCP connection and no credentials.

## Finding Description

**Step 1 — `process_observed` accepts any address without validation.**

`process_observed` (L152–169) calls `self.callback.add_observed_addr(observed, info.session.id)` and unconditionally returns `MisbehaveResult::Continue`. No check is made against the session's actual remote address (`session.address`). The return value of `add_observed_addr` is discarded.

**Step 2 — `add_observed_addr` stores the address without any plausibility check.**

`add_observed_addr` (L497–507) only appends the local peer-id if absent, then calls `network_state.add_observed_addr(session_id, addr)`. No validation against the session's socket address, reachability, or IP plausibility is performed.

**Step 3 — `local_listen_addrs` fills shortfalls from `observed_addrs`.**

`local_listen_addrs` (L458–470): when `listens.len() < MAX_RETURN_LISTEN_ADDRS` (10), the shortfall is filled from `observed_addrs`, which includes the attacker-supplied address.

**Step 4 — The poisoned address is broadcast to every new peer.**

In `connected` (L211–229), `local_listen_addrs()` is called and filtered only by `is_reachable` (global IP check). An attacker supplying any globally routable IP (e.g., `1.2.3.4`) trivially passes this filter.

**Step 5 — Receiving peers unconditionally add the address to their peer store.**

`add_remote_listen_addrs` (L488–494) calls `peer_store.add_addr` for every address in the received `listen_addrs`, with no direction check and no verification that the address was actually observed.

**Existing guard is insufficient.** The comment at L415–419 in `received_identify` explicitly acknowledges that inbound session addresses should not be added to the peer store because they are unverified and "will be broadcast to the entire network." However, this guard applies only to `add_outbound_addr` for the session's own address — it does not protect against attacker-supplied addresses injected via `observed_addr` and re-broadcast as `listen_addrs`.

## Impact Explanation

**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single inbound TCP connection allows an attacker to inject an arbitrary globally-routable address into the victim's `observed_addrs`. This address is included in every subsequent identify handshake the victim sends, causing all connecting peers to add it to their peer stores. Because peer stores are shared further via the discovery protocol, the poisoned address propagates transitively across the network. If the address is unreachable, peer discovery degrades network-wide (connection attempts fail, slots are wasted). If the address points to an attacker-controlled node, it enables eclipse attack setup against downstream peers.

## Likelihood Explanation

- Requires only a standard inbound TCP connection — no credentials, stake, or hashpower.
- Most production CKB nodes have fewer than 10 public addresses, so the `observed_addrs` fallback in `local_listen_addrs` is always active.
- The `is_reachable` filter at send time is trivially bypassed by using any public IP.
- The attacker can hold the connection open indefinitely while the poisoned address propagates to many peers.
- The attack is repeatable: the attacker can reconnect and re-inject after disconnection.

## Recommendation

1. **Validate `observed_addr` against the session's actual remote address** in `process_observed`. The node already knows the peer's socket address (`session.address`); reject the observed address if its IP does not match.
2. **Apply `is_reachable` filtering inside `add_observed_addr`**, not only at send time.
3. **Do not include `observed_addrs` in `listen_addrs` sent to other peers**, or mark them as unverified and exclude them from peer-store propagation. The existing comment at L415–419 already documents this design intent for session addresses — extend it to observed addresses.
4. **Return `MisbehaveResult::Disconnect`** (or penalize via `misbehave`) when an implausible observed address is received, consistent with how `DuplicateReceived` and `InvalidData` are handled.

## Proof of Concept

```
1. Attacker opens inbound TCP connection to victim node (session_id = S1).
2. Attacker sends IdentifyMessage {
       listen_addrs: [],
       observed_addr: /ip4/1.2.3.4/tcp/8115,   // attacker-controlled or unreachable IP
       identify: <valid network identify bytes>
   }
3. Victim: process_observed → add_observed_addr →
       network_state.observed_addrs[S1] = /ip4/1.2.3.4/tcp/8115/p2p/<victim_peer_id>
4. Honest peer P2 connects to victim.
5. Victim: connected → local_listen_addrs():
       public_addrs returns < 10 entries
       → observed_addrs fills gap with /ip4/1.2.3.4/tcp/8115/p2p/<victim_peer_id>
       → is_reachable(1.2.3.4) == true → address passes filter
6. Victim sends IdentifyMessage to P2 with listen_addrs containing the poisoned address.
7. P2: process_listens → add_remote_listen_addrs → peer_store.add_addr(/ip4/1.2.3.4/tcp/8115/...)
8. P2 later dials 1.2.3.4:8115 (attacker) or wastes a connection slot on an unreachable address.
9. P2's discovery protocol re-shares the poisoned address to further peers.

    fn local_listen_addrs(&mut self) -> Vec<Multiaddr> {
        let mut listens = self.listen_addrs();

        if listens.len() < MAX_RETURN_LISTEN_ADDRS {
            let observe_addrs = self
                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
            listens.extend(observe_addrs);
            listens
        } else {
            listens
        }
    }
