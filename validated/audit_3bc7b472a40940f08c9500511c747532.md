### Title
Unvalidated External Callback Call in Legacy Provider Path Permanently Locks Entropy Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains two execution paths. The modern path (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` with full error handling. The legacy path (when `req.gasLimit10k == 0`, i.e., providers with `defaultGasLimit == 0`) makes a raw, unprotected external call to `IEntropyConsumer(callAddress)._entropyCallback(...)` with no `try/catch` or success-check. If the consumer's callback reverts, the entire `revealWithCallback` transaction reverts, permanently preventing the request from ever being fulfilled.

---

### Finding Description

In `requestHelper`, when a provider has `providerInfo.defaultGasLimit == 0`, the stored request gets `req.gasLimit10k = 0`: [1](#0-0) 

When `revealWithCallback` is later called for such a request, the condition `req.gasLimit10k != 0` is false, so execution falls into the `else` branch (the legacy path): [2](#0-1) 

In this branch, `clearRequest` is called **before** the external callback (checks-effects-interactions pattern), but the callback itself is invoked with no error handling: [3](#0-2) 

If `_entropyCallback` reverts, the entire transaction reverts — including the `clearRequest` call — so the request is never cleared. Because there is no cancel mechanism and no retry-with-error-capture path for legacy requests, the request is permanently stuck in an active state.

Contrast this with the modern path, which uses `excessivelySafeCall` and emits a `CallbackFailed` event, allowing the request to transition to `CALLBACK_FAILED` and be retried: [4](#0-3) 

---

### Impact Explanation

An Entropy request made against a legacy provider (one with `defaultGasLimit == 0`) whose requester is a contract that reverts in `_entropyCallback` can never be fulfilled. The request slot is permanently occupied, the provider wastes gas on every fulfillment attempt, and the user's paid fee is unrecoverable. This is a targeted, per-request Denial of Service affecting the **availability** of the Entropy fulfillment flow for legacy providers.

**Impact: 4 / 10** — Scoped to individual requests on legacy providers; does not affect other users or the global contract state.

---

### Likelihood Explanation

The attacker must:
1. Identify or use a provider with `defaultGasLimit == 0` (legacy providers that have not opted into the new callback failure flow).
2. Deploy a consumer contract whose `_entropyCallback` always reverts (trivial).
3. Call `requestWithCallback` or `requestV2` through that consumer.

No privileged access is required. The entry point (`requestWithCallback` / `requestV2`) is fully open to any unprivileged user.

**Likelihood: 3 / 10** — Requires a legacy provider to be active; modern providers with `defaultGasLimit != 0` are unaffected.

---

### Recommendation

Apply the same `try/catch` (or `excessivelySafeCall`) pattern used in the modern path to the legacy `else` branch. If the callback reverts, emit a failure event and clear the request rather than reverting the entire transaction:

```solidity
// Legacy path — wrap the callback to avoid permanent lock
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber,
        provider,
        randomNumber
    ) {} catch {
        emit CallbackFailed(provider, callAddress, sequenceNumber, ...);
    }
}
```

Alternatively, migrate all providers to set a non-zero `defaultGasLimit` and deprecate the legacy path entirely.

---

### Proof of Concept

1. Deploy a legacy provider with `defaultGasLimit == 0` (or use an existing one).
2. Deploy a malicious consumer:
```solidity
contract MaliciousConsumer is IEntropyConsumer {
    address entropy;
    constructor(address _entropy) { entropy = _entropy; }
    function getEntropy() internal view override returns (address) { return entropy; }
    function entropyCallback(uint64, address, bytes32) internal override {
        revert("always fails");
    }
}
```
3. Call `requestWithCallback(legacyProvider, userRandomNumber)` from `MaliciousConsumer` with the required fee.
4. Attempt `revealWithCallback(legacyProvider, sequenceNumber, userContribution, providerContribution)` — the transaction reverts every time.
5. The request is permanently active; `getRequestV2(legacyProvider, sequenceNumber).sequenceNumber` remains non-zero indefinitely. [5](#0-4)
