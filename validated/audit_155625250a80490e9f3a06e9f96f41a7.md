### Title
Silent u64 Overflow in `FeeRate::calculate` Produces Wrong Fee Rate — (`File: util/types/src/core/fee_rate.rs`)

---

### Summary

`FeeRate::calculate` multiplies a transaction fee (u64 shannons) by `KW = 1000` using `saturating_mul`, which silently clamps the intermediate product to `u64::MAX` instead of returning an error. This produces a silently incorrect fee rate whenever `fee_shannons × 1000 ≥ 2^64`. The wrong fee rate propagates into tx-pool ordering, fee-rate statistics RPC responses, and fee-estimation algorithms — all reachable by an unprivileged tx-pool submitter or block relayer.

---

### Finding Description

In `util/types/src/core/fee_rate.rs`, `FeeRate::calculate` computes:

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
```

where `KW = 1000` and `fee` is a `Capacity` (u64 shannons). [1](#0-0) 

The overflow threshold is:

```
fee > u64::MAX / 1000 ≈ 1.844 × 10^16 shannons ≈ 184,467,440 CKB
```

The total CKB issuance is ~33.6 billion CKB, so values above this threshold are representable on-chain. When the threshold is crossed, `saturating_mul` silently returns `u64::MAX` and the division by `weight` produces a wrong, clamped fee rate — no error is raised, no caller is notified.

The same pattern appears in `FeeRate::fee`:

```rust
let fee = self.0.saturating_mul(weight) / KW;
``` [2](#0-1) 

By contrast, every other financial calculation in the codebase that involves multiplication uses either `u128` widening or `checked_mul` with an explicit `Overflow` error: [3](#0-2) [4](#0-3) 

`FeeRate::calculate` is the only financial multiplication that uses `saturating_mul` and discards the overflow signal.

---

### Impact Explanation

`FeeRate::calculate` is called in three externally-observable paths:

1. **Tx-pool ordering / eviction** — `TxEntry::fee_rate()`, `AncestorsScoreSortKey`, and `EvictKey` all call `FeeRate::calculate`. A saturated fee rate causes the affected transaction to appear to have the maximum possible fee rate (`u64::MAX / weight`), corrupting the priority ordering of the mempool. [5](#0-4) 

2. **Fee-rate statistics RPC** — `FeeRateCollector::statistics` calls `FeeRate::calculate(fee, weight)` for every committed transaction and pushes the result into the statistics vector. A saturated value inflates the reported mean/median fee rate, misleading wallets and users. [6](#0-5) 

3. **Fee-estimation algorithms** — Both `confirmation_fraction` and `weight_units_flow` estimators call `FeeRate::calculate` when accepting a transaction. A wrong fee rate corrupts the estimator's internal state, causing future `estimate_fee_rate` RPC calls to return wrong values. [7](#0-6) 

---

### Likelihood Explanation

The overflow threshold (~184 million CKB as a single transaction fee) is high but not impossible:

- The total CKB supply (~33.6 billion CKB) is ~182× the threshold, so cells with sufficient capacity exist on-chain.
- A tx-pool submitter who controls cells totaling >184 million CKB and constructs a transaction that burns nearly all of it as fee crosses the threshold.
- A block relayer who relays a valid committed block containing such a transaction triggers the overflow in the fee-rate statistics path — no mempool submission is required.
- The `FeeRateCollector` path is triggered by any RPC caller invoking `get_fee_rate_statistics` after such a block is committed, with no further attacker action needed.

Likelihood is **low** for the tx-pool path (requires owning ~184M CKB) and **low-to-medium** for the statistics/estimation path (only requires a valid committed block to exist).

---

### Recommendation

Replace `saturating_mul` with a `u128`-widened intermediate, matching the pattern already used in `DaoCalculator::calculate_maximum_withdraw` and `dao_field_with_current_epoch`:

```rust
// FeeRate::calculate
pub fn calculate(fee: Capacity, weight: u64) -> Self {
    if weight == 0 {
        return FeeRate::zero();
    }
    let fee128 = u128::from(fee.as_u64()) * u128::from(KW);
    FeeRate::from_u64((fee128 / u128::from(weight)).min(u64::MAX as u128) as u64)
}

// FeeRate::fee
pub fn fee(self, weight: u64) -> Capacity {
    let fee128 = u128::from(self.0) * u128::from(weight);
    Capacity::shannons((fee128 / u128::from(KW)).min(u64::MAX as u128) as u64)
}
```

Alternatively, return a `Result` and propagate the overflow to callers, consistent with `safe_mul_ratio`. [8](#0-7) 

---

### Proof of Concept

```
fee = 18_446_744_073_709_552 shannons  (≈ 184,467,440 CKB, just above u64::MAX/1000)
weight = 597  (a typical two-in-two-out transaction)

Expected fee_rate = fee * 1000 / weight
                  = 18_446_744_073_709_552_000 / 597
                  ≈ 30_899_906_xxxxxx  (a large but finite value)

Actual (saturating) fee_rate = u64::MAX / 597
                              = 30_899_906_xxxxxx  ← same by coincidence here,
                                                     but for weight=1:
                              = u64::MAX / 1 = 18_446_744_073_709_551_615
                                (completely wrong)
```

A transaction submitter crafts a transaction consuming cells totalling `>184,467,440 CKB` with near-zero outputs. After the transaction is committed in a block, any RPC caller invoking `get_fee_rate_statistics` receives a silently inflated fee rate. Fee-estimation algorithms that ingested this transaction will subsequently return wrong estimates to all callers until their internal state is reset.
