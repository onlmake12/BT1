Audit Report

## Title
Systematic Eclipse via Largest-Group Eviction Bias in `try_evict_inbound_peer` — (`network/src/peer_registry.rs`)

## Summary
`try_evict_inbound_peer` applies three sequential protection passes (lowest ping, most-recent message, longest connection time) before selecting the largest `/16` group for eviction. An attacker who pre-fills inbound slots from distinct `/16` subnets with low-ping, active, early-connecting peers satisfies all three protection criteria, leaving legitimate peers sharing a `/16` as the sole eviction candidates. The largest-group step then deterministically evicts them, enabling systematic replacement of a node's entire inbound peer set.

## Finding Description
**`sort_then_drop` semantics** (`network/src/peer_registry.rs` L55–63): sorts the candidate list and calls `truncate(list.len() - n)`, which retains the first `len - n` elements (worst) and removes the last `n` (best = protected). All three protection passes use this helper. [1](#0-0) 

**Step 2 — ping protection** (L151–165): comparator `peer2_ping.cmp(&peer1_ping)` sorts descending (highest ping first). The 8 lowest-ping peers sit at the tail and are dropped from the candidate list (protected). Peers with `ping_rtt = None` map to `u64::MAX` and sort to the front, remaining as eviction candidates. An attacker maintaining `ping_rtt = Some(low_ms)` is protected; legitimate peers with no measured ping are not. [2](#0-1) 

**Step 3 — message recency** (L168–183): comparator `peer2_last_message.cmp(&peer1_last_message)` sorts descending (stalest first). Peers with `last_ping_protocol_message_received_at = None` map to `u64::MAX` and remain candidates. Attacker peers sending frequent pings are protected; passive legitimate peers are not. [3](#0-2) 

**Step 4 — connection time** (L185–188): `protect_peers = candidate_peers.len() >> 1`. `connected_time` is an `Instant` set at connection time (`peer.rs` L104). Comparator `peer2.connected_time.cmp(&peer1.connected_time)` sorts descending (most recently connected first). `truncate(len - protect_peers)` keeps the recently-connected half as candidates and protects the longest-connected half. Attacker peers that connected before legitimate peers have earlier `Instant` values (longer connection time) and are protected; legitimate peers that connected later remain candidates. [4](#0-3) [5](#0-4) 

**Step 5 — largest-group eviction** (L191–210): remaining candidates are grouped by `/16` subnet (`network_group.rs` L26–28: `Group::IP4([bits[0], bits[1]])`). `.max_by_key(|group| group.len())` selects the largest group. Attacker peers each occupy a distinct `/16` (singleton groups of size 1). Legitimate peers sharing a `/16` form a group of size ≥ 2, which is always the largest group and is always selected. [6](#0-5) [7](#0-6) 

**`_peer_store` unused** (L142): the parameter is prefixed with `_` and never consulted, so no reputation or historical scoring mitigates the bias. [8](#0-7) 

**Exploit flow:**
1. Attacker pre-fills `N - K` inbound slots from distinct `/16` subnets, connecting before legitimate peers, maintaining low ping and frequent messages.
2. `K` legitimate peers connect from the same `/16` (filling the remaining slots).
3. Attacker connects a new peer, triggering `try_evict_inbound_peer`.
4. All three protection passes eliminate attacker peers from the candidate set; legitimate peers remain.
5. Legitimate peers form the only multi-peer `/16` group and one is evicted.
6. The new attacker peer takes the freed slot.
7. Repeat until all legitimate peers are replaced.

**Existing guards are insufficient:** the whitelist check only protects explicitly configured peers; the ban check only blocks already-banned addresses; neither limits per-`/16` inbound count nor introduces randomness that would break the deterministic targeting. [9](#0-8) 

## Impact Explanation
Systematic inbound eclipse: an attacker can replace all non-whitelisted inbound connections with attacker-controlled peers. An eclipsed node receives only attacker-supplied block and transaction data on its inbound side, enabling chain-view manipulation, mempool poisoning, and — combined with outbound connection interference — full eclipse enabling double-spend attacks against merchants and exchanges relying on the node. Maps to **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points); the inbound-only scope prevents an automatic Critical rating, but the attack is low-cost, deterministic, and repeatable.

## Likelihood Explanation
With `max_inbound = 125` and `K = 2`, the attacker needs 123 distinct `/16` subnets. This is achievable via cloud providers (AWS, GCP, Azure each span thousands of `/16` prefixes), VPS networks, or residential proxy services at moderate cost. No cryptographic secrets, no hashpower, and no privileged access are required. The attack is fully deterministic: once the attacker holds enough IPs and connects first, every subsequent eviction round removes a legitimate peer with certainty.

## Recommendation
1. **Cap per-`/16` inbound connections:** reject or deprioritize new inbound connections from a `/16` that already has `max(1, max_inbound / 32)` connected peers, preventing attacker saturation.
2. **Randomized eviction fallback:** with probability ≥ 50%, evict a uniformly random candidate instead of always targeting the largest group, breaking deterministic targeting.
3. **Treat `ping_rtt = None` as neutral:** use a separate "unverified" bucket rather than mapping `None` to `u64::MAX`, so unverified peers are not automatically ranked as worst.
4. **Use `_peer_store` reputation:** the parameter is already threaded through `try_evict_inbound_peer`; prefer evicting peers with poor historical scores.
5. **Limit per-`/16` representation before largest-group step:** if any single group exceeds `max(1, candidates / 4)`, randomly evict from it regardless of whether it is the largest.

## Proof of Concept
A minimal unit test demonstrating the bias (correcting the PoC's invalid IP formatting and assertion):

```rust
#[test]
fn test_eclipse_inbound_via_eviction_bias() {
    // N=10, K=2: 8 attacker peers in distinct /16s (connect first, low ping),
    // 2 legitimate peers in 100.0.x.x (connect after, ping_rtt=None)
    let mut peer_store = PeerStore::default();
    let mut registry = PeerRegistry::new(10, 3, false, vec![], true);

    // Attacker peers: 1.0.0.1 through 8.0.0.1 (distinct /16s), connect first
    for i in 1u8..=8 {
        let addr = format!("/ip4/{}.0.0.1/tcp/1234/p2p/{}", i, PeerId::random().to_base58())
            .parse::<Multiaddr>().unwrap();
        registry.accept_peer(addr, i.into(), RawSessionType::Inbound, &mut peer_store).unwrap();
        registry.get_peer_mut(i.into()).unwrap().ping_rtt = Some(Duration::from_millis(10));
    }
    // Legitimate peers: both in 100.0.x.x (/16 = [100, 0]), connect after
    for i in 9u8..=10 {
        let addr = format!("/ip4/100.0.{}.1/tcp/1234/p2p/{}", i, PeerId::random().to_base58())
            .parse::<Multiaddr>().unwrap();
        registry.accept_peer(addr, i.into(), RawSessionType::Inbound, &mut peer_store).unwrap();
        // ping_rtt = None → u64::MAX, not protected in step 2
    }

    // Inbound full (10/10). Trigger eviction with a new attacker peer.
    let new_addr = format!("/ip4/50.0.0.1/tcp/1234/p2p/{}", PeerId::random().to_base58())
        .parse::<Multiaddr>().unwrap();
    let evicted = registry.accept_peer(new_addr, 11.into(), RawSessionType::Inbound, &mut peer_store)
        .unwrap().expect("should evict");
    // The evicted peer must be from the legitimate /16 group [100, 0]
    assert_eq!(evicted.network_group(), Group::IP4([100, 0]));
}
```

The invariant holds because: (a) attacker peers with `ping_rtt = Some(10ms)` are always protected in step 2; (b) attacker peers with earlier `connected_time` are protected in step 4; (c) the two legitimate peers in `Group::IP4([100, 0])` form the only multi-peer group and are always selected in step 5.
