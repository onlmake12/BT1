### Title
Unsorted `offsets` Vector Breaks `partition_point` Binary Search in `HashChainState::reveal` - (File: apps/fortuna/src/state.rs)

### Summary
`HashChainState` explicitly documents that its `offsets` vector "Must be stored in sorted order" because `reveal()` uses `partition_point` (a binary-search variant) to locate the correct hash chain. However, `HashChainState::new()` accepts an arbitrary `offsets` vector without validating or enforcing sorted order, creating the same unsorted-collection-breaks-binary-search class of bug as the reported finding.

### Finding Description

`HashChainState` maps on-chain sequence numbers to provider hash chains. The struct comment on line 133 states the invariant explicitly:

```rust
pub struct HashChainState {
    // The sequence number where the hash chain starts. Must be stored in sorted order.
    offsets: Vec<usize>,
    hash_chains: Vec<PebbleHashChain>,
}
``` [1](#0-0) 

`reveal()` relies on this invariant by calling `partition_point`, which is only correct on a sorted slice:

```rust
pub fn reveal(&self, sequence_number: u64) -> Result<[u8; 32]> {
    let sequence_number: usize = sequence_number.try_into()?;
    let chain_index = self
        .offsets
        .partition_point(|x| x <= &sequence_number)
        .checked_sub(1)
        ...
``` [2](#0-1) 

`HashChainState::new()`, however, performs **no sort and no order validation** — it only checks that the two vectors have equal length:

```rust
pub fn new(offsets: Vec<usize>, hash_chains: Vec<PebbleHashChain>) -> Result<HashChainState> {
    if offsets.len() != hash_chains.len() {
        return Err(anyhow::anyhow!(
            "Offsets and hash chains must have the same length."
        ));
    }
    Ok(HashChainState { offsets, hash_chains })
}
``` [3](#0-2) 

If a provider rotates their on-chain commitment to a sequence number lower than a previously registered one, the keeper reconstructs `HashChainState` with an unsorted `offsets` vector. `partition_point` on an unsorted slice produces an undefined index, causing `reveal()` to select the **wrong hash chain** for a given sequence number.

### Impact Explanation

`reveal()` is the function the Fortuna keeper calls to produce the provider's random contribution for an entropy request. Selecting the wrong hash chain returns a hash value from an unrelated chain segment. The combined random number delivered to the requesting contract is therefore incorrect — breaking the entropy guarantee for any user whose request is fulfilled during the window where the unsorted state is active. Depending on the sequence of offsets, `partition_point` may also return an out-of-bounds index (after `checked_sub(1)` fails), causing the keeper to error out and stop fulfilling requests entirely (denial of service for entropy users).

### Likelihood Explanation

Any registered Entropy provider (an unprivileged role — anyone can call `register`) can rotate their commitment on-chain to a sequence number lower than a prior commitment. The Fortuna keeper reads these on-chain commitments and reconstructs `HashChainState` without sorting. The trigger requires only a provider to perform an out-of-order commitment rotation, which is a normal operational action with no privileged access required.

### Recommendation

Enforce the sorted-order invariant inside `HashChainState::new()`. Either sort the offsets (and correspondingly reorder `hash_chains`) before storing them, or return an error if the caller supplies an unsorted slice:

```rust
pub fn new(offsets: Vec<usize>, hash_chains: Vec<PebbleHashChain>) -> Result<HashChainState> {
    if offsets.len() != hash_chains.len() {
        return Err(anyhow::anyhow!("Offsets and hash chains must have the same length."));
    }
    // Enforce the invariant required by partition_point in reveal()
    if offsets.windows(2).any(|w| w[0] > w[1]) {
        return Err(anyhow::anyhow!("Offsets must be in sorted order."));
    }
    Ok(HashChainState { offsets, hash_chains })
}
```

### Proof of Concept

```rust
let chain1 = PebbleHashChain::new([0u8; 32], 10, 1);
let chain2 = PebbleHashChain::new([1u8; 32], 10, 1);

// Offsets supplied in REVERSE order — new() accepts this without error
let state = HashChainState {
    offsets: vec![20, 5],          // unsorted: 20 before 5
    hash_chains: vec![chain1, chain2],
};

// partition_point on [20, 5] for sequence_number=8:
//   it finds the first index where offset > 8 → index 0 (since 20 > 8)
//   checked_sub(1) → chain_index = 0  (chain starting at offset 20)
// Correct answer would be chain_index = 1 (chain starting at offset 5)
// Wrong hash chain is selected → incorrect random value returned
let result = state.reveal(8);
// result comes from chain1 (offset 20) instead of chain2 (offset 5)
