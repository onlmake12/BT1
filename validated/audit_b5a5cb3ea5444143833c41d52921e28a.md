Audit Report

## Title
Integer Overflow in Guard Allows Unbounded DB Iteration via `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The guard intended to cap server-side work at `GET_LAST_STATE_PROOF_LIMIT = 1000` uses an unchecked multiplication `(last_n_blocks as usize) * 2` that wraps to zero in Rust release mode when `last_n_blocks = 2^63`. A remote, unprivileged peer can bypass this guard entirely, causing the server to iterate over the full chain history — performing two DB reads per block — in a single request handler invocation.

## Finding Description
`last_n_blocks` is decoded directly from the peer message as an attacker-controlled `u64`: [1](#0-0) 

The guard at lines 201–205 uses unchecked multiplication on `usize`: [2](#0-1) 

On a 64-bit target, `usize` is 64 bits. In Rust release mode (no `overflow-checks` by default), `(2^63_usize) * 2 = 0` via two's-complement wrap. With `difficulties.len() = 0`, the guard evaluates `0 + 0 > 1000` → **false** → bypassed.

After the guard, `reorg_last_n_numbers` is computed without any independent bound on `last_n_blocks`: [3](#0-2) 

With `last_n_blocks = 2^63` and `start_block_number = N`, `min(N, 2^63) = N`, so `min_block_number = 0` and `reorg_last_n_numbers = (0..N)` — N entries. The `else` branch is triggered by sending any `start_hash` that does not match the actual ancestor, which requires no privilege.

Separately, the `last_n_numbers` path at lines 291–297 is also unbounded: with `last_n_blocks = 2^63`, the condition `last_block_number - start_block_number <= last_n_blocks` is always true, so `last_n_numbers = (start_block_number..last_block_number)`. Setting `start_block_number = 0` yields ~14M entries with no reorg branch needed. [4](#0-3) 

Both paths feed into `complete_headers`, which performs `get_ancestor` + `chain_root_mmr(*number - 1).get_root()` per entry with no size check: [5](#0-4) 

The constant being defeated: [6](#0-5) 

## Impact Explanation
On a mainnet node with ~14 million blocks, a single crafted `GetLastStateProof` message triggers ~14 million sequential DB reads in one request handler. An attacker sending this repeatedly — or from multiple peers simultaneously — causes sustained CPU and I/O exhaustion, degrading block validation, peer sync, and all other node functions to the point of effective unavailability. This matches **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
The attack requires only a P2P connection to the light-client protocol port — no authentication, no PoW, no stake. The crafted message is a single valid molecule-encoded `GetLastStateProof` packet. The overflow value `2^63` is a fixed constant. Any peer that can connect to the node's light-client port can trigger this, and the attack is trivially repeatable.

## Recommendation
Replace the unchecked multiplication with saturating arithmetic, and add an explicit upper bound on `last_n_blocks` before any further use:

```rust
// Reject oversized last_n_blocks directly
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("last_n_blocks too large");
}

// Use saturating arithmetic in the existing guard
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, add a size check on `reorg_last_n_numbers` and `last_n_numbers` after they are computed and before `complete_headers` is called, as defense-in-depth.

## Proof of Concept

```rust
// Craft the malicious message (path 1: reorg branch)
let msg = packed::GetLastStateProof::new_builder()
    .last_hash(valid_main_chain_tip_hash)   // any valid main-chain block hash
    .start_hash(Byte32::zero())             // wrong hash → triggers else branch
    .start_number(13_999_999u64.pack())     // large start_number on a 14M-block chain
    .last_n_blocks(9223372036854775808u64.pack()) // 2^63: (as usize)*2 == 0 in release
    .difficulty_boundary(some_valid_boundary)
    // difficulties: empty → difficulties.len() == 0
    .build();
// Guard: 0 + 0 > 1000 → false → passes
// reorg_last_n_numbers = (0..13_999_999) → ~14M entries → ~28M DB reads

// Craft the malicious message (path 2: last_n_numbers branch, no reorg needed)
let msg = packed::GetLastStateProof::new_builder()
    .last_hash(valid_main_chain_tip_hash)
    .start_hash(Byte32::zero())
    .start_number(0u64.pack())              // start=0 → reorg_last_n_numbers=[]
    .last_n_blocks(9223372036854775808u64.pack()) // 2^63
    .difficulty_boundary(some_valid_boundary)
    .build();
// Guard: 0 + 0 > 1000 → false → passes
// last_block_number - 0 <= 2^63 → true → last_n_numbers = (0..14_000_000) → ~14M DB reads
```

Send this message repeatedly or from multiple peers to sustain exhaustion.
