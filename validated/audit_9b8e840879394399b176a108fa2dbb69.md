### Title
Echo `executeCallback` Permanently Locks User Funds via Dynamic Pyth Fee Underflow - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, the provider fee credit in `executeCallback` is computed as `(req.fee + msg.value) - pythFee`, where `pythFee` is the **dynamic** result of `pyth.getUpdateFee(updateData)` at callback time. However, `req.fee` was set at request time using the **static** `_state.pythFeeInWei`. If `pythFee` exceeds `req.fee + msg.value`, Solidity 0.8's checked arithmetic causes an underflow revert, permanently locking the user's escrowed funds in the contract.

---

### Finding Description

**At request time** (`requestPriceUpdatesWithCallback`):

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
```

`req.fee` stores the provider's portion as `msg.value − _state.pythFeeInWei`, where `_state.pythFeeInWei` is a **static** value set at contract initialization. [1](#0-0) 

**At callback time** (`executeCallback`):

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);   // dynamic
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(...);
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // ← underflow if pythFee > req.fee + msg.value
``` [2](#0-1) 

`pyth.getUpdateFee(updateData)` charges for **every price feed present in `updateData`**, not just the ones requested:

```solidity
totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(updateData[i], offset);
return getTotalFee(totalNumUpdates);
``` [3](#0-2) 

The `priceIds` array is verified against `req.priceIdPrefixes` (only the first 8 bytes of each ID), but `updateData` itself is **unconstrained** — it can contain arbitrarily many extra price feeds beyond those requested. This inflates `pythFee` without bound. [4](#0-3) 

The developers themselves flagged this risk with a TODO comment:

> "TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract." [5](#0-4) 

---

### Impact Explanation

When `executeCallback` reverts due to the underflow, the in-flight `Request` is never cleared. The user's escrowed ETH (paid at request time) remains permanently locked in the Echo contract with no recovery path. There is no cancellation or refund mechanism for unfulfilled requests.

**Severity: High** — direct, irreversible loss of user funds.

---

### Likelihood Explanation

Two realistic trigger paths exist for an unprivileged actor:

1. **Pyth oracle fee increase (passive):** If Pyth governance raises `singleUpdateFeeInWei` after a request is submitted, the actual `pythFee` at callback time exceeds `_state.pythFeeInWei` baked into `req.fee`. If the gap is large enough, the subtraction underflows.

2. **Attacker-inflated `updateData` (active):** After the exclusivity period expires, `executeCallback` is callable by **anyone**. An attacker submits `updateData` containing many extra price feeds. `pyth.getUpdateFee(updateData)` scales linearly with the number of feeds, so the attacker can make `pythFee` arbitrarily large, guaranteeing the underflow revert and permanently locking the victim's funds. [6](#0-5) 

---

### Recommendation

Replace the dynamic `pyth.getUpdateFee(updateData)` in the provider credit calculation with the static `_state.pythFeeInWei` that was already accounted for at request time:

```solidity
// Instead of:
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

// Use:
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - _state.pythFeeInWei);
```

This ensures the provider credit calculation uses the same fee basis as the request, eliminating the underflow vector. The actual Pyth oracle fee is still paid correctly via `{value: pythFee}` — only the accounting split changes.

Alternatively, add a hard cap: `pythFee = min(pythFee, req.fee + msg.value)` and absorb any shortfall from `_state.accruedFeesInWei`.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` paying exactly `getFee(provider, gasLimit, priceIds)` — the minimum required fee. `req.fee = providerFee`, `_state.accruedFeesInWei += _state.pythFeeInWei`.

2. Exclusivity period expires (e.g., 60 seconds).

3. Attacker constructs `updateData` containing the required price IDs **plus** 100 additional price feeds. `pyth.getUpdateFee(updateData)` returns `101 × singleUpdateFeeInWei`.

4. Attacker calls `executeCallback(providerToCredit, sequenceNumber, updateData, priceIds)` with `msg.value = 0`.

5. Inside `executeCallback`: `pythFee = 101 × singleUpdateFeeInWei >> req.fee`. The subtraction `(req.fee + 0) - pythFee` underflows → revert.

6. The request is never cleared. User's ETH is permanently locked. [7](#0-6) [8](#0-7)

    function getUpdateFee(
        bytes[] calldata updateData
    ) public view override returns (uint feeAmount) {
        uint totalNumUpdates = 0;
        for (uint i = 0; i < updateData.length; i++) {
            if (
                updateData[i].length > 4 &&
                UnsafeCalldataBytesLib.toUint32(updateData[i], 0) ==
                ACCUMULATOR_MAGIC
            ) {
                (
                    uint offset,
                    UpdateType updateType
                ) = extractUpdateTypeFromAccumulatorHeader(updateData[i]);
                if (updateType != UpdateType.WormholeMerkle) {
                    revert PythErrors.InvalidUpdateData();
                }
                totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(
                    updateData[i],
                    offset
                );
            } else {
                revert PythErrors.InvalidUpdateData();
            }
        }
        return getTotalFee(totalNumUpdates);
