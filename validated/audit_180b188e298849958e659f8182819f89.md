### Title
Fee Sufficiency Check Compares Raw Token Amounts Across Different Denominations Without Spot-Rate Conversion — (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

---

### Summary

The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract sums raw coin amounts across multiple token denominations (e.g., `uosmo`, `uion`, `ibc/FF30...`) and compares the raw total against the required fee amount — without converting any of the alternative tokens to the base denomination (`uosmo`) using a spot price. This is a direct unit-mismatch analog to M-23: a nominal amount in one token is treated as equivalent to the same nominal amount in a different token, even though their real values differ.

---

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the Osmosis feature-gated `is_fee_sufficient` function (lines 163–197) accepts coins in multiple denominations, sums their raw `u128` amounts unconverted, and checks whether that raw sum meets the required fee:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(PythContractError::InvalidFeeDenom { ... })?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}

let base_denom_fee = get_update_fee(deps, data)?;
// base_denom_fee is in uosmo; total_amount may be in uion or ibc/...
Ok(base_denom_fee.amount.u128() <= total_amount)
```

The code itself documents the correct fix and acknowledges the omission:

```

and:

// It's not important to convert the price right now. For now
// we are keeping the base fee amount same for each valid denom -> 1
// but this logic will be updated to use spot price for different valid tokens in future
```

The companion `get_update_fee_for_denom` function (lines 717–736) also returns the same nominal amount regardless of the requested denom, confirming no conversion is applied anywhere in the fee path. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Any caller of `update_price_feeds` on the Osmosis deployment can pay the required fee entirely in a cheaper alternative token (e.g., `uion` or the IBC-wrapped token) at the same nominal integer amount as the required `uosmo` fee. Because `uion` and `uosmo` have different market prices, the actual USD value paid can be substantially less than the protocol intends to charge. If governance raises the fee to a non-negligible amount (the current value is 1 uosmo per update), the underpayment gap scales proportionally. The fee mechanism — which exists to prevent spam and compensate the protocol — is rendered economically ineffective for any caller who uses an alternative denom.

---

### Likelihood Explanation

The entry path is fully unprivileged: any user submitting a price update on Osmosis can exploit this by simply sending `uion` instead of `uosmo`. The Osmosis `txfee` module already whitelists alternative denoms, so the denom-validity check passes. The only current mitigation is that the fee is set to 1 uosmo (negligible), but this is a governance parameter that can be changed at any time. The code itself marks the correct fix as a TODO, confirming the team is aware the current behavior is incorrect.

---

### Recommendation

Before comparing against `base_denom_fee.amount`, convert each accepted alternative-denom coin to its `uosmo` equivalent using the Osmosis `TxFeesQuerier` spot price (which the code already queries for denom validation). The corrected logic should:

1. For each coin in `info.funds`, query the spot price of that denom relative to `uosmo`.
2. Multiply `coin.amount` by the spot price to get the `uosmo`-equivalent value.
3. Sum the converted values.
4. Compare the converted total against `base_denom_fee.amount`.

This matches the approach described in the existing TODO comment and is consistent with how the non-Osmosis variant uses `has_coins` with a single denom.

---

### Proof of Concept

Assume the governance fee is set to `1000 uosmo` per update, and `1 uion = 0.01 uosmo` at current market rates.

1. Attacker calls `update_price_feeds` on the Osmosis contract, sending `funds: [{ denom: "uion", amount: 1000 }]`.
2. `is_allowed_tx_fees_denom` returns `true` for `uion` (it is whitelisted).
3. `total_amount` = 1000 (raw `uion` units).
4. `base_denom_fee.amount` = 1000 (raw `uosmo` units).
5. Check: `1000 <= 1000` → `true`. Fee accepted.
6. Actual USD value paid: `1000 uion × $0.01/uion = $10`.
7. Required USD value: `1000 uosmo × $1.00/uosmo = $1000`.
8. Attacker underpays by 99%, and the price feed update is accepted. [3](#0-2) [4](#0-3)

    // TxFeesQuerier uses stargate queries which we can't mock as of now.
