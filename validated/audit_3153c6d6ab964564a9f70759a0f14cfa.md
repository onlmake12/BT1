Audit Report

## Title
`DownloadScheduler::decrease` Assigns `task_count` Instead of `timeout_count` on RHS, Draining Peer Concurrency Budget ~3× Faster Than Intended — (File: sync/src/types/mod.rs)

## Summary
In `DownloadScheduler::decrease` (L521–527), the accumulator assignment reads `self.task_count.saturating_add(num)` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` initializes to 32 and `num` is 1 or 2, the result always exceeds 2, so the guard branch fires on every single call rather than only after 3 accumulated timeouts. This causes any peer delivering blocks in the upper latency quantiles to have its concurrency budget (`task_count`) drained approximately 3× faster than the protocol intends, degrading sync throughput for that peer slot.

## Finding Description
`DownloadScheduler` is defined at L482–532 of `sync/src/types/mod.rs`. The `decrease` method at L521–527 is intended to accumulate timeout events in `timeout_count` and only decrement `task_count` once `timeout_count` exceeds 2. The actual code at L522 assigns `self.task_count.saturating_add(num)` to `self.timeout_count`, overwriting it with a value equal to `task_count + num` on every call. Since `task_count` initializes to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) and `num` is 1 or 2, the expression is always ≥ 3 until `task_count` reaches 1, meaning the `if self.timeout_count > 2` branch fires on every call. `decrease` is called from `remove_by_block` (L799–810) whenever a block arrives in the `NormalToUpper` (calls `decrease(1)`) or `UpperToMax` (calls `decrease(2)`) latency quantile and `should_punish` is true (L786: `self.download_schedulers.len() > self.protect_num`). Any peer — including legitimate high-latency peers — that delivers blocks in these quantiles will have its `task_count` drained to 0 in ~32 slow blocks instead of ~96. When `task_count` reaches 0, `can_fetch()` (L504–506) returns 0 and the block fetcher skips that peer entirely.

## Impact Explanation
The bug causes the download scheduler to penalize high-latency peers ~3× more aggressively than the protocol intends. Legitimate peers with elevated but acceptable latency lose their concurrency budget prematurely, reducing the node's effective sync parallelism. This matches **Low (501–2000 points): Any other important performance improvements for CKB**. The node does not crash, no consensus deviation occurs, and no economic damage results.

## Likelihood Explanation
The conditions are met during normal IBD whenever the node has more connected peers than `protect_num` and any peer delivers blocks in the upper latency quantiles. A malicious peer reachable over the P2P network can deliberately delay responses to trigger this path with no special privileges. Legitimate high-latency peers trigger it passively. The effect is bounded to the affected peer's slot; other peers are unaffected.

## Recommendation
Fix the assignment in `DownloadScheduler::decrease` to accumulate into `self.timeout_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

## Proof of Concept
1. Construct a `DownloadScheduler` with default state (`task_count = 32`, `timeout_count = 0`).
2. Call `decrease(1)` once. With the bug: `timeout_count = 32 + 1 = 33 > 2` → `task_count` becomes 31, `timeout_count` resets to 0.
3. Call `decrease(1)` again: `timeout_count = 31 + 1 = 32 > 2` → `task_count` becomes 30.
4. After 31 such calls, `task_count = 1`; one more call yields `timeout_count = 1 + 1 = 2`, which is not `> 2`, so `task_count` stays at 1. The budget is effectively exhausted after ~32 calls.
5. With the fix, the first two calls accumulate `timeout_count` to 2 without decrementing `task_count`; only the third call triggers the decrement, confirming the ~3× difference.
