### Title
Unbounded SQL `NOT IN` Clause in `get_cells` Scales O(N) with tx-pool Dead Cells — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

### Summary

When `index_tx_pool = true` is configured (a supported production option), `get_cells` reads every entry from the indexer's `Pool::dead_cells` set and emits one SQL placeholder per entry into a `NOT IN (…)` clause. Because neither the indexer `Pool` nor the SQL-building path caps the number of dead cells, an attacker who fills the tx-pool with many transactions each spending a distinct UTXO can force every subsequent `get_cells` call to construct, bind, and submit a multi-megabyte SQL query to the database, degrading RPC performance proportionally to pool size.

### Finding Description

**Indexer pool — no size cap.**
`util/indexer-sync/src/pool.rs` defines:

```rust
pub struct Pool {
    dead_cells: HashSet<OutPoint>,   // line 21 — unbounded
}
```

Every accepted tx-pool transaction inserts all its inputs: [1](#0-0) 

No eviction or cap is applied to `dead_cells` independently of the tx-pool.

**`get_cells` — unbounded NOT IN construction.** [2](#0-1) 

For N dead cells this produces:

```sql
… AND (tx_hash, output_index) NOT IN (($1, 0), ($2, 0), … ($N, 0))
```

followed by N individual `.bind(tx_hash)` calls: [3](#0-2) 

**`request_limit` does not help.**
The existing guard (lines 26–34) caps the number of *result rows*, not the size of the NOT IN clause: [4](#0-3) 

**Indirect bound from tx-pool size.**
The only ceiling on dead cells is the tx-pool's `max_tx_pool_size` (default 180 MB): [5](#0-4) 

With a realistic minimum transaction size of ~200 bytes, this allows up to ~900,000 transactions and a corresponding number of dead cells. A NOT IN clause with 900,000 placeholders produces a SQL string of ~13 MB and requires binding ~28 MB of parameters per RPC call.

The same pattern is present in `get_cells_capacity`: [6](#0-5) 

### Impact Explanation

Each `get_cells` (and `get_cells_capacity`) RPC call incurs O(N) work in SQL string construction, parameter binding, and database query-plan compilation, where N is the current number of dead cells in the indexer pool. At pool saturation this degrades the rich-indexer RPC to the point of unresponsiveness for all callers, not just the attacker.

### Likelihood Explanation

The attack requires `index_tx_pool = true` (opt-in, but a documented production feature) and enough live UTXOs to fill the pool. The fee cost to fill 180 MB at the default minimum fee rate of 1000 shannons/KB is approximately 1.8 CKB — economically trivial. The UTXO creation cost is higher but one-time. Even a partial fill (e.g., 10,000 transactions) produces a measurably larger SQL string and increased latency on every `get_cells` call.

### Recommendation

Cap the number of dead cells included in the NOT IN clause to a fixed maximum (e.g., 1,000). If the live pool exceeds that cap, either omit the filter (accepting a small risk of returning a just-spent cell) or return an error. Alternatively, store dead cells in a temporary DB table and join against it rather than expanding them inline into the query string.

### Proof of Concept

1. Enable `index_tx_pool = true` in the node's indexer config.
2. Pre-create N distinct live cells on-chain (e.g., N = 10,000 / 100,000).
3. Submit N transactions to the tx-pool, each spending one distinct live cell.
4. Call `get_cells` via RPC and measure: SQL string length, parameter count, and wall-clock latency.
5. Assert that latency and SQL string length grow linearly with N, and that at N ≈ 900,000 (pool saturation) the RPC becomes effectively unresponsive.
