### Title
`total_tx_size` permanently inflated after cell_ref_parent eviction in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

### Summary
`add_entry` pre-computes the new pool size totals into local variables **before** calling `check_and_record_ancestors`, which may evict existing transactions (via `remove_entry` → `update_stat_for_remove_tx`, mutating `self.total_tx_size` in-place). The stale pre-eviction local value is then unconditionally written back to `self.total_tx_size`, permanently inflating it by the sum of evicted transaction sizes.

### Finding Description
In `PoolMap::add_entry` (`pool_map.rs:200–221`):

1. `updated_stat_for_add_tx` (line 210–211) computes `total_tx_size = self.total_tx_size + entry.size` into a **local variable**.
2. `check_and_record_ancestors` (line 213) may evict `cell_ref_parent` transactions by calling `remove_entry`, which calls `update_stat_for_remove_tx` and **decrements `self.total_tx_size` in-place**.
3. Line 218 unconditionally assigns `self.total_tx_size = total_tx_size` (the stale local), **overwriting the correctly decremented value**.

The invariant `total_tx_size == Σ entries[i].size` is broken after every such eviction. The error accumulates across repeated submissions.

### Impact Explanation
`total_tx_size` is used to determine whether the pool is full and to drive eviction of low-fee transactions. An inflated `total_tx_size` causes the pool to believe it is fuller than it actually is, triggering premature eviction of legitimate high-fee transactions and incorrect rejection of new submissions. This directly damages miner revenue and distorts the fee market. The inflation is permanent and cumulative — it is never corrected unless `recompute_total_stat` is triggered by an underflow (which requires a separate removal path).

### Likelihood Explanation
The trigger condition — a new transaction spending a cell that is currently used as a cell dep by a pool transaction — is a normal, valid scenario in CKB's UTXO/cell model. No special privilege, key, or hashpower is required. An attacker can deliberately construct such transactions to repeatedly inflate the counter.

### Recommendation
Move the `total_tx_size`/`total_tx_cycles` assignment to **after** `check_and_record_ancestors` returns, computing the final value from the already-updated `self.total_tx_size`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// self.total_tx_size has already been decremented for any evictions
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Now add only the new entry's contribution
self.total_tx_size = self.total_tx_size.checked_add(entry.size)...;
self.total_tx_cycles = self.total_tx_cycles.checked_add(entry.cycles)...;
```

Alternatively, subtract the evicted sizes from the pre-computed local before assigning.

### Proof of Concept
Invariant fuzz test: after any `add_entry` call that returns a non-empty `evicts` set, assert:

```rust
assert_eq!(
    pool_map.total_tx_size,
    pool_map.entries.iter().map(|(_, e)| e.inner.size).sum::<usize>()
);
```

This assertion will fail whenever `check_and_record_ancestors` evicts at least one transaction.
