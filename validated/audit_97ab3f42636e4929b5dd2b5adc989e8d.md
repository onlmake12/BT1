Audit Report

## Title
`check_tx_fee` Uses Serialized Size Instead of Canonical Weight for Min-Fee-Rate Enforcement, Allowing Sub-Threshold Transactions Into the Mempool — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` enforces `min_fee_rate` using only the transaction's serialized byte size, while the canonical weight used everywhere else in the codebase is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns actual cycles, no second fee-rate check is performed. An attacker can craft a transaction with small serialized size but near-maximum script execution cycles, pay only the size-based minimum fee, and have the transaction admitted to the pool with an effective fee rate far below `min_fee_rate`.

## Finding Description

**Root cause — size-only fee check:**

In `tx-pool/src/util.rs` at line 45, `check_tx_fee` computes:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code itself acknowledges the discrepancy at lines 42–44:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
``` [1](#0-0) 

**Canonical weight function (not used in fee check):**

`get_transaction_weight` in `util/types/src/core/tx_pool.rs` lines 298–303 computes `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. [2](#0-1) 

This same function is used in `TxEntry::fee_rate()` at `tx-pool/src/component/entry.rs` lines 115–118, in `AncestorsScoreSortKey` (lines 221–232), and in `EvictKey` (lines 234–247) — everywhere except the admission gate. [3](#0-2) 

**Execution order — no post-verification fee check:**

In `tx-pool/src/process.rs`, `_process_tx` calls `pre_check` (which calls `check_tx_fee`) at line 715, then calls `verify_rtx` at line 724. After `verify_rtx` returns actual cycles at line 734, the code only checks for declared-vs-actual cycle mismatch (lines 736–748) and then creates the `TxEntry` with actual cycles at line 751. There is no second fee-rate check against the true weight. [4](#0-3) 

**Concrete example:** A transaction with `tx_size = 100` bytes and `cycles = 70,000,000` passes `check_tx_fee` with a fee of 100 shannons (satisfying `min_fee_rate = 1000 shannons/KW` on size alone). Its actual weight is `max(100, 70_000_000 × DEFAULT_BYTES_PER_CYCLES) ≈ 11,940` bytes, giving an effective fee rate of `≈ 8.4 shannons/KW` — approximately 119× below the configured threshold.

## Impact Explanation

This matches the High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker can flood the mempool with transactions whose effective fee rates are far below `min_fee_rate`, degrading fee estimation accuracy for all users and displacing legitimate transactions at a fraction of the intended cost. While the eviction mechanism (`EvictKey`) correctly uses weight-based fee rate and would evict these transactions first when the pool is full, an attacker who continuously resubmits can maintain a persistent presence and sustain the degradation. [5](#0-4) 

## Likelihood Explanation

Any unprivileged transaction sender can exploit this via the `send_transaction` RPC or P2P relay. The attacker deploys a complex lock or type script (once, as a cell dep) consuming near-`max_tx_verify_cycles` cycles, then submits many transactions referencing it. Each transaction has small serialized size but high script execution cycles. The recurring cost is only the size-based minimum fee (e.g., 100 shannons per transaction) rather than the ~11,940 shannons that would be required if weight were used. The attack requires no special privileges and is repeatable.

## Recommendation

Replace the size-only proxy in `check_tx_fee` with the canonical weight function. Since actual cycles are unknown at pre-check time, use the declared cycles (available for remote transactions via the `declared_cycles` parameter in `_process_tx`) or a conservative upper bound (`max_tx_verify_cycles`) for local transactions:

```rust
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(max_tx_verify_cycles));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

Alternatively, perform a second fee-rate check after `verify_rtx` returns the actual cycles, rejecting transactions whose true effective fee rate falls below `min_fee_rate`.

## Proof of Concept

1. Deploy a lock script consuming ~70,000,000 cycles as a cell dep on a local node.
2. Construct a transaction spending a cell locked by that script. The transaction body is ~100 bytes serialized.
3. Set the transaction fee to exactly `ceil(min_fee_rate × tx_size / 1000) = 100 shannons` (for `min_fee_rate = 1000`, `tx_size = 100`).
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` passes: `fee (100) >= min_fee (100)` — size-only check at line 45 of `tx-pool/src/util.rs`.
6. `verify_rtx` executes the script, consuming ~70,000,000 cycles.
7. No second fee-rate check is performed (lines 736–751 of `tx-pool/src/process.rs`).
8. The transaction is admitted. Inspect via `get_transaction` RPC — the entry has `cycles ≈ 70,000,000`, `size ≈ 100`, `fee = 100 shannons`, giving effective fee rate ≈ 8.4 shannons/KW.
9. Repeat with many input cells to fill the mempool with sub-threshold-fee-rate transactions and observe fee estimation degradation via `estimate_fee_rate` RPC.
