### Title
Stale Accounting Overwrite in `add_entry` Inflates `total_tx_size`/`total_tx_cycles` After Cell-Ref Eviction — (`File: tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new aggregate pool statistics (`total_tx_size`, `total_tx_cycles`) are computed from the pool state **before** `check_and_record_ancestors` runs. That helper can evict existing entries (via `remove_entry_and_descendants` → `update_stat_for_remove_tx`), correctly decrementing the live counters. However, the pre-computed stale values are then unconditionally written back, silently overwriting the correct post-eviction counters. The result is that `total_tx_size` and `total_tx_cycles` are permanently inflated by the size/cycles of every evicted entry, causing `limit_size` to over-evict legitimate transactions from the pool.

---

### Finding Description

**Root cause — `pool_map.rs::add_entry` (lines 200–221):**

```rust

    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    ...
    // (1) Snapshot new totals BEFORE any eviction

    self.total_tx_cycles = total_tx_cycles;

| Before call | `S` | pool baseline |

Each evicted entry goes through `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles`: [4](#0-3) [5](#0-4) 

The inflated `total_tx_size` is then consumed by `limit_size`, which evicts entries until the counter drops below `max_tx_pool_size`: [6](#0-5) 

`limit_size` is called immediately after `_submit_entry` inside the write-lock in `submit_entry`: [7](#0-6) 

---

### Impact Explanation

Every time the cell-ref eviction path fires, `total_tx_size` is inflated by the combined serialized size of all evicted entries. `limit_size` then sees a pool that appears larger than it actually is and evicts additional legitimate pending/gap/proposed transactions to compensate. Those transactions receive a `Reject::Full` callback and are dropped from the pool. Miners lose fee revenue; users whose transactions are evicted must resubmit. Repeated triggering accumulates drift, making the pool appear perpetually over-capacity and causing continuous spurious eviction of honest transactions. This is a **tx-pool integrity / resource-accounting** failure reachable by any unprivileged transaction sender.

---

### Likelihood Explanation

The trigger condition is reachable without any privileged access:

1. Attacker submits a chain of ~25 transactions (filling `max_ancestors_count`).
2. Attacker submits a separate transaction that uses one of those chain transactions as a **cell dep** (making it a `cell_ref_parent`).
3. Attacker submits a new transaction that spends from the chain AND references the cell-dep transaction, pushing `ancestors_count` above the limit while `cell_ref_parents` can absorb the excess.
4. `check_and_record_ancestors` evicts the cell-dep transaction; `total_tx_size` is inflated.
5. `limit_size` over-evicts honest transactions.

The default `max_ancestors_count` is 25, which is easily reachable. No special keys, hashpower, or network position are required — only the ability to submit transactions via RPC or P2P relay.

---

### Recommendation

Move the `total_tx_size`/`total_tx_cycles` assignment to **after** all mutations complete, computing the final values from the live counters rather than from a pre-eviction snapshot. The simplest fix is to replace the stale-snapshot pattern with an incremental update applied only after `check_and_record_ancestors` returns:

```rust
// After all evictions and insertions are done:
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .ok_or_else(|| Reject::Full(...))?;
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(...))?;
```

This mirrors the checks-effects-interactions fix: perform all state-changing operations first, then update the accounting counters from the resulting live state.

---

### Proof of Concept

**Preconditions:** Node with default config (`max_ancestors_count = 25`, `max_tx_pool_size` near capacity).

1. Submit a cellbase-spending transaction `T0` with one output.
2. Build a chain `T1 → T2 → … → T25` each spending the previous output (25 ancestors).
3. Submit a transaction `Tdep` that uses `T0`'s output as a **cell dep** (not an input).
4. Submit a transaction `Tnew` that spends `T25`'s output AND lists `Tdep` as a cell dep. Now `Tnew`'s ancestor set = {T0…T25, Tdep} = 27 > 25, but `cell_ref_parents = {Tdep}` so `27 - 1 = 26 > 25`... adjust chain length so the condition `ancestors_count.saturating_sub(cell_ref_parents.len()) <= max_ancestors_count` is satisfied.
5. Observe via `get_pool_info` RPC that `total_tx_size` is larger than the sum of all currently-pooled transactions' sizes.
6. Submit one more transaction; observe that `limit_size` evicts a legitimate transaction even though the real pool size is within bounds.

**Expected (correct) behavior:** `total_tx_size` equals the sum of sizes of all entries currently in the pool.
**Actual behavior:** `total_tx_size` is inflated by the size of `Tdep` (and its descendants), causing spurious eviction.

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
