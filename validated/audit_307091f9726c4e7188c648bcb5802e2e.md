Audit Report

## Title
Off-by-One Epoch Boundary in `rfc0044_active` Gate Skips `DaoScriptSizeVerifier` for First Block of Activation Epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

## Summary
In `BlockTxsVerifier::verify`, the gate controlling `DaoScriptSizeVerifier` passes `self.parent.epoch().number()` to `rfc0044_active` instead of the current block's epoch number. For the single block that opens epoch A (the RFC0044 activation epoch), the parent is still in epoch A-1, so the gate evaluates to `false` and the verifier is never called. A miner who produces that block can include a DAO phase-1 withdrawal whose output lock script differs in byte-length from the deposit cell's lock script, permanently committing a transaction that violates the RFC0044 invariant.

## Finding Description
The faulty gate is at `verification/contextual/src/contextual_block_verifier.rs` line 445:

```rust
if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
    DaoScriptSizeVerifier::new(...).verify()?;
}
```

`rfc0044_active` in `spec/src/consensus.rs` lines 1005–1012 is:

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    let rfc0044_active_epoch = ...;   // 8651 mainnet, 5711 testnet
    target >= rfc0044_active_epoch
}
```

Let A = `RFC0044_ACTIVE_EPOCH`. For the first block of epoch A, `self.parent.epoch().number()` = A-1. The predicate becomes `A-1 >= A` = `false`, so `DaoScriptSizeVerifier` is not invoked. `self.header` (the current block's header) is a field of `BlockTxsVerifier` (line 334) and is already used to construct `TxVerifyEnv` at line 400, so the correct epoch is available but unused in the gate.

The tx-pool path (`tx-pool/src/util.rs` lines 110–128) calls `DaoScriptSizeVerifier` unconditionally, but a miner building a block template directly bypasses the tx-pool entirely, so that guard is irrelevant for block-level validation.

## Impact Explanation
This is an incorrect implementation of the RFC0044 system-script invariant: the `DaoScriptSizeVerifier` is the sole block-level enforcement of the rule that DAO withdrawal output lock scripts must match the deposit cell's lock script size. Skipping it for one block allows a permanently committed on-chain transaction that violates this invariant. This maps to the allowed impact: **Incorrect implementation or behavior of CKB-VM or system scripts — High (10001–15000 points)**.

## Likelihood Explanation
The exploitable window is exactly one block per chain lifetime (the first block of epoch 8651 on mainnet, epoch 5711 on testnet). Any miner who wins that block slot can exploit it with no special privilege beyond normal block production. The miner constructs the block template directly, bypassing the tx-pool's unconditional `DaoScriptSizeVerifier` call. No victim mistake or external dependency is required.

## Recommendation
Replace `self.parent.epoch().number()` with the current block's epoch number in the gate:

```rust
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
    DaoScriptSizeVerifier::new(...).verify()?;
}
```

`self.header` is already a field of `BlockTxsVerifier` and is used on line 400 of the same function, so no structural change is needed.

## Proof of Concept
1. Configure a devnet with `RFC0044_ACTIVE_EPOCH = N` (small value for testing).
2. Mine until the chain tip is the last block of epoch N-1.
3. Construct a block at epoch N containing a DAO phase-1 withdrawal where the output lock script byte-length differs from the deposit cell's lock script byte-length.
4. Submit the block via P2P relay or RPC (bypassing the tx-pool).
5. Observe: the node accepts the block — `rfc0044_active(N-1)` = false, verifier skipped.
6. Construct an identical block at epoch N+1 (parent epoch = N): the node rejects it — `rfc0044_active(N)` = true, verifier runs and returns `DaoLockSizeMismatch`.

This confirms the one-block bypass window at the epoch boundary.
