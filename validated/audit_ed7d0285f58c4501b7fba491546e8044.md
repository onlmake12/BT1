### Title
Unchecked Stale Price and Integer Truncation in `PythAggregatorV3` Chainlink Adapter ŌĆö (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3.sol` is Pyth's official Chainlink `AggregatorV3Interface`-compatible adapter, distributed as part of the `@pythnetwork/pyth-sdk-solidity` SDK. It contains two direct analogs of the reported vulnerability class:

1. Every price-reading function calls `getPriceUnsafe()` with no staleness check, silently returning prices from arbitrarily far in the past.
2. `decimals()` performs `uint8(-1 * int8(price.expo))` where `price.expo` is `int32`, causing silent truncation for any exponent outside `[-128, 127]`.

---

### Finding Description

**Root Cause 1 ŌĆö Unchecked stale price (timestamp not validated)**

All five price-reading functions in `PythAggregatorV3` call `pyth.getPriceUnsafe(priceId)`:

```solidity
// latestAnswer()
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

// latestRoundData()
function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    ...
}
```

`getPriceUnsafe()` is explicitly documented as returning a price "from arbitrarily far in the past" with no recency guarantee. No comparison of `price.publishTime` against `block.timestamp` is performed anywhere in the contract. A downstream Chainlink-compatible protocol calling `latestRoundData()` or `latestAnswer()` will silently receive a price that may be hours or days old.

**Root Cause 2 ŌĆö Integer truncation in `decimals()`**

```solidity
function decimals() public view virtual returns (uint8) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return uint8(-1 * int8(price.expo));
}
```

`price.expo` is typed `int32` (range: ŌłÆ2,147,483,648 to 2,147,483,647). The explicit narrowing cast `int8(price.expo)` silently truncates the high 24 bits at runtime in Solidity Ōēź0.8.x (explicit casts truncate; only arithmetic operators revert on overflow). For any `price.expo` outside `[-128, 127]`, the cast produces a wrong value. For example, `price.expo = -200` ŌåÆ `int8(-200)` silently becomes `int8(56)` ŌåÆ `-1 * 56 = -56` ŌåÆ `uint8(-56)` reverts, causing a DoS on `decimals()`. For values that do not produce a negative intermediate, the returned `uint8` is silently wrong, causing downstream protocols to scale prices by the wrong factor.

---

### Impact Explanation

- **Stale price**: Any protocol that deploys `PythAggregatorV3` as a drop-in Chainlink feed and calls `latestRoundData()` / `latestAnswer()` will receive a price that may be arbitrarily old. If the Pyth on-chain price has not been updated recently (e.g., due to network congestion or lack of a keeper), the protocol will operate on stale data, enabling price-manipulation exploits (e.g., opening/closing positions at outdated prices).
- **`decimals()` truncation**: Downstream protocols use `decimals()` to normalize the raw price integer. A silently wrong decimals value causes the protocol to treat the price as off by a power of 10, leading to massively incorrect collateral valuations, liquidation thresholds, or trade execution prices.

---

### Likelihood Explanation

- **Stale price**: Medium. Pyth is a pull oracle; on-chain prices are only updated when someone calls `updatePriceFeeds()`. If no keeper is running for a given feed, the price can become arbitrarily stale. Any protocol relying on `PythAggregatorV3` without its own update mechanism is exposed.
- **`decimals()` truncation**: Low for the revert path (Pyth's published exponents are typically ŌłÆ8 or ŌłÆ9, well within `int8`). However, the cast is structurally unsound: the `int32` type contract allows any value, and a future feed or misconfiguration could trigger it silently.

---

### Recommendation

1. Replace `getPriceUnsafe()` with `getPriceNoOlderThan(priceId, maxAge)` in all price-reading functions, where `maxAge` is a configurable staleness threshold set at construction time.
2. Replace `int8(price.expo)` with a bounds-checked cast (e.g., require `price.expo >= -128 && price.expo <= 0`) before narrowing, or use `int32` arithmetic throughout and only cast to `uint8` after validation.

---

### Proof of Concept

**Stale price:**
1. Deploy `PythAggregatorV3` pointing to any Pyth price feed.
2. Do not call `updateFeeds()` for 24 hours.
3. Call `latestRoundData()` ŌĆö it returns the 24-hour-old price with no revert or staleness flag.
4. A downstream lending protocol using this adapter will accept the stale price for collateral valuation.
