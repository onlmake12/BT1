### Title
`min_fee_rate` Admission Check Ignores Cycles Component of Block Weight — (`tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function enforces the `min_fee_rate` admission policy using only the transaction's serialized size (`tx_size`), but the actual block resource consumed by a transaction is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For transactions with high cycles but low serialized size, the actual block weight is significantly larger than `tx_size`, so the minimum fee required is understated. This allows any unprivileged transaction sender to submit transactions that enter the pool with effective fee rates well below `min_fee_rate`.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    let reject = Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
    return Err(reject);
}
``` [1](#0-0) 

This uses only `tx_size` (the serialized size in block). The code itself acknowledges the gap with the comment: *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."* [2](#0-1) 

However, the actual block weight used for transaction selection and block assembly is computed by `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This weight is used for eviction key and fee-rate scoring inside the pool: [4](#0-3) 

but is **never applied as an admission gate**. The `check_tx_fee` pre-verification check is the only fee-rate rejection point, and it uses only `tx_size`. After script execution completes and cycles are known, no second fee-rate check is performed against the actual weight.

The block assembler then selects transactions using `txs_to_commit`, which enforces both `size_limit` and `cycles_limit` independently: [5](#0-4) 

So the pool correctly limits block weight at selection time, but the admission gate (min_fee_rate) only accounts for the size dimension, not the cycles dimension.

---

### Impact Explanation

With `max_tx_verify_cycles = 70,000,000` and a minimal transaction serialized size of ~200 bytes:

- Actual block weight = `max(200, 70,000,000 × 0.000170571)` = `max(200, 11,940)` = **11,940 bytes**
- Fee check uses 200 bytes → `min_fee = 200 shannons` (at `min_fee_rate = 1,000 shannons/KB`)
- Correct minimum based on actual weight = **11,940 shannons**
- Discrepancy: **~60×**

An attacker can fill the pool with high-cycle, low-size transactions at roughly 1/60th of the intended minimum cost. The `max_tx_pool_size` limit (180 MB) is enforced against `pool_map.total_tx_size` which sums `entry.size` (serialized bytes), not weight: [6](#0-5) 

So the pool can be saturated with transactions that each consume far more block-cycle budget than their fee implies, degrading block quality and enabling cheap mempool spam.

---

### Likelihood Explanation

Any unprivileged transaction sender reachable via the `send_transaction` RPC can exploit this. Crafting a transaction with a high-cycle lock or type script (e.g., a script that performs many iterations) while keeping the serialized transaction body small is straightforward. No special privileges, keys, or majority hashpower are required.

---

### Recommendation

After script verification completes and cycles are known, add a post-verification fee-rate check using the actual block weight:

```rust
let weight = get_transaction_weight(tx_size, cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64()));
}
```

This mirrors the existing pre-check in `check_tx_fee` but uses the weight that the block assembler actually enforces.

---

### Proof of Concept

1. Craft a transaction whose lock script loops for ~70,000,000 cycles but whose serialized size is ~200 bytes.
2. Submit via `send_transaction` RPC.
3. `check_tx_fee` computes `min_fee = min_fee_rate.fee(200)` = 200 shannons and accepts the transaction.
4. Script execution completes; cycles = 70,000,000; actual weight = 11,940 bytes.
5. The transaction enters `pool_map` with `entry.size = 200`, `entry.cycles = 70,000,000`.
6. `pool_map.total_tx_size` increases by only 200, so the pool size limit is barely consumed.
7. Repeat: an attacker can submit ~900,000 such transactions before hitting the 180 MB `max_tx_pool_size` limit, each consuming 11,940 bytes of block weight, for a total block-weight footprint of ~10.7 GB — far exceeding what the pool size limit was intended to bound. [7](#0-6) [8](#0-7) [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
