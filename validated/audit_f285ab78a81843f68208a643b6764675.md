### Title
Peer Store Permanent Eviction Deadlock via Adversarial Address Distribution — (`network/src/peer_store/peer_store_impl.rs`)

### Summary

`check_purge` phase-2 contains two compounding logic flaws — `take(len / 2)` integer-division truncation and an `addrs.len() > 4` guard — that together produce zero eviction candidates when the store is filled with addresses uniformly distributed across ≥4096 network groups of exactly 4 entries each. Every subsequent `add_addr` call returns `Err(PeerStoreError::EvictionFailed)`, permanently blocking peer discovery until entries age out naturally.

### Finding Description

`check_purge` is called unconditionally at the top of `add_addr` before inserting any new address: [1](#0-0) 

Phase-1 collects non-connectable entries. A freshly `add_addr`-inserted entry has `last_connected_at_ms = 0` and `attempts_count = 0`, which makes `is_connectable` return `true` (neither the `ADDR_MAX_RETRIES` nor `ADDR_MAX_FAILURES` threshold is reached): [2](#0-1) 

So phase-1 yields nothing. Phase-2 then groups all 16384 entries by network segment, sorts groups by descending size, and applies two filters: [3](#0-2) 

**Flaw 1 — `take(len / 2)`:** With 4096 groups, `len / 2 = 2048`. The top 2048 groups are selected. All have exactly 4 entries.

**Flaw 2 — `addrs.len() > 4`:** The guard requires strictly more than 4 entries to produce any eviction candidates. With exactly 4 entries per group, `4 > 4` is `false` for every selected group. `flat_map` returns `None` for all of them.

Result: `candidate_peers` is empty, and the function returns: [4](#0-3) 

### Impact Explanation

The error propagates out of `add_addr` and is silently swallowed at `debug` level in the discovery protocol handler: [5](#0-4) 

The node does not crash, but its peer store is permanently full and rejects every new discovered address. The node can no longer learn about new peers, making it unable to recover lost connections or sync with new nodes. Recovery only occurs naturally after enough entries accumulate 3+ failed connection attempts (`ADDR_MAX_RETRIES = 3`) and become non-connectable — a process that can take days if the adversarial addresses are reachable or the feeler rate is low. [6](#0-5) 

### Likelihood Explanation

The attack entry point is the standard P2P discovery protocol. `DiscoveryAddressManager::add_new_addrs` is called whenever a `Nodes` message is received: [7](#0-6) 

A single connected peer can send up to `MAX_ADDR_TO_SEND = 1000` addresses per `Nodes` message: [8](#0-7) 

Filling 16384 slots requires ~17 such messages. The addresses only need to pass `is_valid_addr` (globally routable), not be actually reachable. An attacker with access to a modest number of routable IP prefixes (or knowledge of real public IPs across 4096 /24 subnets, which is trivially available from public BGP data) can craft this distribution. No PoW, no privileged role, and no Sybil majority is required — a single connected peer suffices.

### Recommendation

Fix both flaws together:

1. **Replace `take(len / 2)` with `take((len + 1) / 2)`** (ceiling division) to avoid dropping the last group when `len` is odd, and more importantly to ensure at least one group is always considered when `len >= 1`.
2. **Change `addrs.len() > 4` to `addrs.len() >= 4`** (or lower the threshold to `>= 2`) so that groups at the boundary are eligible for eviction.
3. **Add a fallback:** if phase-2 still produces zero candidates after the above fixes (e.g., all groups have exactly 1 entry), evict the single lowest-scored or oldest entry unconditionally rather than returning `EvictionFailed`.

fn test_eviction_deadlock() {
    use crate::{Flags, PeerId, peer_store::{ADDR_COUNT_LIMIT, PeerStore}};
    use p2p::multiaddr::Multiaddr;

    let mut peer_store = PeerStore::default();
    // Fill with 4096 groups × 4 entries = 16384 entries (ADDR_COUNT_LIMIT)
    // Each group is a distinct /24 (x.y.z.0/24), 4 ports per group
    let mut count = 0;
    'outer: for a in 1u8..=255 {
        for b in 0u8..=255 {
            for port in 1u16..=4 {
                if count >= ADDR_COUNT_LIMIT { break 'outer; }
                let addr: Multiaddr = format!(
                    "/ip4/{}.{}.1.1/tcp/{}/p2p/{}",
                    a, b, port, PeerId::random().to_base58()
                ).parse().unwrap();
                peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
                count += 1;
            }
        }
    }
    assert_eq!(count, ADDR_COUNT_LIMIT);

    // One more address from a new group — must succeed but will fail
    let extra: Multiaddr = format!(
        "/ip4/10.0.0.1/tcp/9999/p2p/{}", PeerId::random().to_base58()
    ).parse().unwrap();
    // This returns Err(EvictionFailed) — peer discovery is now permanently blocked
    assert!(peer_store.add_addr(extra, Flags::COMPATIBILITY).is_err());
}
