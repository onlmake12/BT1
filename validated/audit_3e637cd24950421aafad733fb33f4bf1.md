### Title
Unrestricted Caller with Uncapped `tx.gasprice` in `updatePriceFeeds` Enables Subscription Balance Drain - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary
`Scheduler.updatePriceFeeds()` has no caller restriction and pays `msg.sender` a keeper fee calculated as `(gasUsed + GAS_OVERHEAD) * tx.gasprice + singleUpdateKeeperFeeInWei * numPriceIds`. Because `tx.gasprice` is never capped, any unprivileged attacker can call the function with an artificially inflated gas price whenever update conditions are met, extracting a profit of `GAS_OVERHEAD * tx.gasprice` per call and draining the subscription balance far beyond what legitimate keeper costs would be.

---

### Finding Description

`updatePriceFeeds()` is intentionally permissionless — any address may call it and receive a keeper payment from the subscription's deposited ETH balance. The payment is computed in `_processFeesAndPayKeeper`:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
...
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
```

There is no upper bound on `tx.gasprice`. The only guard is that `totalKeeperFee` must not exceed `status.balanceInWei`. An attacker can therefore set `tx.gasprice` to the maximum value that keeps `totalKeeperFee ≤ balance`, extracting nearly the entire subscription balance in a single call while only paying the network for actual gas consumed.

**Attacker profit per call:**
- Attacker pays to network: `actualGasUsed × tx.gasprice`
- Attacker receives from subscription: `(actualGasUsed + GAS_OVERHEAD) × tx.gasprice + keeperSpecificFee`
- **Net profit: `GAS_OVERHEAD × tx.gasprice + keeperSpecificFee`**

By maximising `tx.gasprice` up to `(balance − keeperSpecificFee) / (actualGasUsed + GAS_OVERHEAD)`, the attacker extracts a fraction `GAS_OVERHEAD / (actualGasUsed + GAS_OVERHEAD)` of the entire subscription balance per eligible update window. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

- **Subscription balance drain**: A subscription owner who deposits ETH to fund legitimate price updates loses funds at a rate far exceeding actual keeper costs. An attacker can extract `GAS_OVERHEAD / (actualGasUsed + GAS_OVERHEAD)` of the balance per update cycle.
- **Subscription deactivation**: Once the balance falls below `minimumBalance`, the subscription becomes inactive and stops receiving price updates, breaking downstream consumers that depend on it.
- **Theft of deposited ETH**: The attacker profits in ETH at the direct expense of the subscription owner, with no requirement for any privileged role. [3](#0-2) 

---

### Likelihood Explanation

- **Entry path is fully permissionless**: `updatePriceFeeds()` has no `onlyManager` or whitelist guard. Any EOA or contract can call it.
- **Trigger windows are predictable**: Heartbeat-based subscriptions open an update window every `heartbeatSeconds` seconds. An attacker can monitor `getActiveSubscriptions()` (also unrestricted) to enumerate targets and their heartbeat intervals.
- **No economic barrier**: The attacker's only cost is the actual gas consumed at the inflated price, which is exactly offset by the subscription payment. The `GAS_OVERHEAD` constant provides guaranteed profit.
- **Realistic on all supported EVM chains**: The Scheduler is deployed on Ethereum and L2s. On L2s where base fees are negligible, the attacker can set `tx.gasprice` to an extreme value at near-zero real cost. [4](#0-3) [5](#0-4) 

---

### Recommendation

1. **Cap `tx.gasprice`**: Introduce a configurable `maxGasPriceInWei` parameter (settable by admin). In `_processFeesAndPayKeeper`, replace `tx.gasprice` with `Math.min(tx.gasprice, maxGasPriceInWei)`.
2. **Alternatively, cap total keeper fee**: Enforce a maximum keeper fee per update (e.g., a multiple of `singleUpdateKeeperFeeInWei * numPriceIds`) so that gas cost reimbursement cannot exceed a reasonable bound.
3. **Restrict callers to a keeper registry**: Require callers to be registered keepers, analogous to how the original infrastructure report recommended restricting open ports to only P2P ports. [2](#0-1) 

---

### Proof of Concept

```
Setup:
  - Subscription with balance = 1 ETH, heartbeatSeconds = 60, numPriceIds = 2
  - actualGasUsed ≈ 200,000 gas, GAS_OVERHEAD = X (constant in SchedulerConstants)
  - singleUpdateKeeperFeeInWei = 0.0001 ETH, keeperSpecificFee = 0.0002 ETH

Attack (after 60 seconds elapse):
  1. Attacker computes max_gasprice = (1 ETH - 0.0002 ETH) / (200,000 + GAS_OVERHEAD)
  2. Attacker calls scheduler.updatePriceFeeds(subscriptionId, validUpdateData)
     with tx.gasprice = max_gasprice
  3. _processFeesAndPayKeeper pays attacker ≈ 1 ETH from subscription balance
  4. Attacker's actual network cost = 200,000 × max_gasprice
  5. Net profit = GAS_OVERHEAD × max_gasprice > 0

Result:
  - Subscription balance drained from 1 ETH to ~0 in a single heartbeat window
  - Subscription deactivates; downstream consumers receive no further price updates
  - Attacker profits proportional to GAS_OVERHEAD / (actualGasUsed + GAS_OVERHEAD) × balance
``` [6](#0-5) [2](#0-1)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
