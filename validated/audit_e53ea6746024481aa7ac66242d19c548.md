Audit Report

## Title
Unauthenticated Block Template Injection via Miner Notify HTTP Server — (File: miner/src/client.rs)

## Summary
When the CKB miner is configured in notify mode, `listen_block_template_notify` binds a plain HTTP server that accepts `BlockTemplate` payloads from any TCP client with no authentication, IP filtering, or token verification. Any attacker who can reach the configured port can inject a `BlockTemplate` with an attacker-controlled `cellbase` lock script, redirecting all mining hashpower to pay block rewards to the attacker's address.

## Finding Description
`Client::spawn_background` (lines 204–232) starts the inbound HTTP server when `config.listen` is `Some`. Every accepted connection is dispatched to the `handle` free function: [1](#0-0) 

The peer address returned by `listener.accept()` is discarded with `_` and never inspected: [2](#0-1) 

`parse_authorization` is only invoked when building outbound RPC requests in `Rpc::new`; it is never consulted on the inbound server path: [3](#0-2) 

`update_block_template` fires unconditionally whenever the incoming `work_id` differs from the stored atomic value (or the stored value is `0`), immediately sending `Works::New(work)` to all mining workers: [4](#0-3) 

The `listen` field is an unconstrained `Option<SocketAddr>` with no loopback enforcement anywhere in the codebase: [5](#0-4) 

Because `work_id` is fully attacker-controlled and the initial stored value is `0`, the very first injected payload always passes the `id == 0` branch and triggers the update. Subsequent payloads with monotonically increasing `work_id` values ensure the attacker's template is never displaced by a legitimate node notification.

## Impact Explanation
**Mining reward theft** matches the allowed CKB bounty impact "Vulnerabilities which could easily damage CKB economy" (Critical, 15001–25000 points). An attacker injects a `BlockTemplate` whose `cellbase` output carries the attacker's lock script. The miner solves valid PoW on that template and submits it; the block reward is paid to the attacker's address. At pool scale this constitutes systematic theft of CKB block rewards. A secondary impact is continuous PoW DoS: injecting templates with an invalid `parent_hash` or stale block number causes all hashpower to be wasted on blocks the node will reject on `submit_block`.

## Likelihood Explanation
Notify mode is the recommended low-latency mining configuration. The default example uses `127.0.0.1:8888`, but pool operators and cloud mining setups routinely bind to `0.0.0.0` or a specific non-loopback interface to receive notifications from a remote CKB node. Any process or peer that can reach the TCP port — co-located processes, LAN peers, or remote attackers if the port is exposed — can exploit this with a single HTTP POST. No credentials, no prior knowledge of internal state, and no special tooling are required.

## Recommendation
1. In `listen_block_template_notify`, capture the peer address from `listener.accept()` instead of discarding it with `_`, pass it into `handle`, and reject connections whose source IP does not match the host portion of `config.rpc_url`.
2. Alternatively, require a shared-secret token in a request header (e.g., `X-CKB-Notify-Token`), configured in `ckb-miner.toml` and verified in `handle` before deserialization.
3. At minimum, enforce at startup that `config.listen` resolves to a loopback address and abort with a clear error if a non-loopback address is supplied.
