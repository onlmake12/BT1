### Title
Incorrect Variable Used in `DownloadScheduler::decrease` Accumulation, Causing Overly Aggressive Peer Penalty and Premature Disconnection — (`sync/src/types/mod.rs`)

---

### Summary

In `sync/src/types/mod.rs`, the `DownloadScheduler::decrease` function uses `self.task_count` instead of `self.timeout_count` when accumulating the penalty counter. This mirrors the H-08 class of bug: an incorrect variable reference in a counter/index check within a scheduling loop causes the wrong accounting behavior. The result is that the soft-penalty accumulation mechanism is bypassed entirely, and every slow block response immediately decrements the peer's download task quota, causing peers to be disconnected far faster than intended.

---

### Finding Description

`DownloadScheduler` is the per-peer download quota manager inside `InflightBlocks`. It tracks how many concurrent block downloads a peer is allowed (`task_count`) and uses a soft-penalty counter (`timeout_count`) to dampen the rate at which `task_count` is reduced. [1](#0-0) 

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // ← BUG
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

The intended logic is:

```rust
self.timeout_count = self.timeout_count.saturating_add(num); // accumulate penalty
```

Instead, `self.task_count` is used. Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (a value well above 2) and `num` is 1 or 2, the expression `self.task_count.saturating_add(num)` is **always greater than 2**. The `if` branch fires on every single call, immediately decrementing `task_count` and resetting `timeout_count` to 0. The accumulation guard is completely neutralized.

`decrease` is called from `remove_by_block` whenever a block response falls in the `NormalToUpper` or `UpperToMax` time zones: [2](#0-1) 

When `task_count` reaches 0, `prune` disconnects the peer: [3](#0-2) 

---

### Impact Explanation

The `NormalToUpper` zone is explicitly described as a **normal** operating zone (not a penalty zone): [4](#0-3) 

With the bug, every block response that falls in the `NormalToUpper` zone (normal latency) causes `task_count` to be decremented by 1 immediately. A peer that consistently responds at normal-but-not-fast latency will have its `task_count` driven to 0 and be disconnected. If all of a node's sync peers respond in the normal latency zone (common in cross-continental or loaded networks), the node will disconnect from all of them, halting block synchronization entirely.

---

### Likelihood Explanation

Any unprivileged sync peer can trigger this path simply by responding to block download requests with slightly elevated latency (within the `NormalToUpper` zone). No special privileges, keys, or majority hashpower are required. The bug is triggered on every block delivery that falls in the normal-to-slow latency range, which is routine network behavior. The entry path is: peer sends a block → `remove_by_block` is called → `time_analyzer.push_time(elapsed)` returns `NormalToUpper` or `UpperToMax` → `set.decrease(1)` or `set.decrease(2)` is called → bug fires. [5](#0-4) 

---

### Recommendation

Change line 522 from:

```rust
self.timeout_count = self.task_count.saturating_add(num);
```

to:

```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

This restores the intended accumulation behavior: `task_count` is only decremented after `num` has been accumulated into `timeout_count` enough times to exceed the threshold of 2.

---

### Proof of Concept

Assume `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 16` and a peer consistently responds in the `NormalToUpper` zone:

1. Initial state: `task_count = 16`, `timeout_count = 0`
2. Call `decrease(1)`:
   - `timeout_count = 16 + 1 = 17` (should be `0 + 1 = 1`)
   - `17 > 2` → `task_count = 15`, `timeout_count = 0`
3. Call `decrease(1)` again:
   - `timeout_count = 15 + 1 = 16`
   - `16 > 2` → `task_count = 14`, `timeout_count = 0`
4. After 16 such calls: `task_count = 0` → peer is disconnected by `prune`

With the correct code, `timeout_count` would need to accumulate to 3 before any decrement, meaning `task_count` would only decrease once every 3 slow responses — a 3× slower penalty rate, matching the documented design intent. [6](#0-5)

pub struct DownloadScheduler {
    task_count: usize,
    timeout_count: usize,

impl Default for DownloadScheduler {
    fn default() -> Self {
        Self {
            hashes: HashSet::default(),
            task_count: INIT_BLOCKS_IN_TRANSIT_PER_PEER,
            timeout_count: 0,
        }
    }
}

impl DownloadScheduler {
    fn inflight_count(&self) -> usize {
        self.hashes.len()
    }

    fn can_fetch(&self) -> usize {
        self.task_count.saturating_sub(self.hashes.len())
    }

    pub(crate) const fn task_count(&self) -> usize {
        self.task_count
    }

    fn increase(&mut self, num: usize) {
        if self.task_count < MAX_BLOCKS_IN_TRANSIT_PER_PEER {
            self.task_count = ::std::cmp::min(
                self.task_count.saturating_add(num),
                MAX_BLOCKS_IN_TRANSIT_PER_PEER,
            )
        }
    }

    fn decrease(&mut self, num: usize) {
        self.timeout_count = self.task_count.saturating_add(num);
        if self.timeout_count > 2 {
            self.task_count = self.task_count.saturating_sub(1);
            self.timeout_count = 0;
        }
    }

    fn punish(&mut self, exp: usize) {
        self.task_count >>= exp
    }
}
