### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Arbitrary Pending Transactions from the Pool - (`rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` (and `clear_tx_pool` / `clear_tx_verify_queue`) RPC methods in `rpc/src/module/pool.rs` perform no caller authentication or ownership verification. Any HTTP client that can reach the RPC endpoint — including a malicious web page exploiting the node's `CorsLayer::permissive()` CORS policy — can silently evict any pending transaction (and its entire descendant chain) from the pool without the original submitter's consent. This is the direct CKB analog of the MapleLoan `makePayment()` missing-access-control bug: a state-mutating function callable by anyone that harms a different party.

---

### Finding Description

**Root cause — no authentication on the RPC server**

`rpc/src/server.rs` builds the Axum HTTP router with a fully permissive CORS layer and no authentication middleware:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    ...
    .layer(CorsLayer::permissive())   // ← any origin allowed
    ...
``` [1](#0-0) 

There is no token check, no IP-allowlist enforcement, and no session validation anywhere in the request pipeline. Every registered RPC method is reachable by any HTTP client that can send a POST to the listening address.

**Root cause — `remove_transaction` has no ownership check**

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
``` [2](#0-1) 

The method accepts any `tx_hash` and immediately forwards it to `remove_local_tx`. There is no check that `msg.sender` (the RPC caller) is the original submitter of the transaction, no signature, no token, and no ownership field.

The underlying pool operation removes the target transaction **and all of its descendants**:

```rust
pub(crate) fn remove_tx(&mut self, id: &ProposalShortId) -> bool {
    let entries = self.pool_map.remove_entry_and_descendants(id);
    !entries.is_empty()
}
``` [3](#0-2) 

**`clear_tx_pool` and `clear_tx_verify_queue` are equally unguarded**

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [4](#0-3) 

```rust
fn clear_tx_verify_queue(&self) -> Result<()> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_verify_queue()
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
``` [5](#0-4)

### Impact Explanation

1. **Targeted transaction eviction**: An attacker who knows (or can enumerate via `get_raw_tx_pool`) a victim's pending `tx_hash` can call `remove_transaction` to silently drop it and every child transaction from the pool. The victim's transaction is gone without any on-chain record; the victim must resubmit and pay fees again.

2. **Cascading descendant removal**: Because `remove_entry_and_descendants` is called, a single RPC call can wipe an entire pre-built transaction chain (e.g., a multi-step DeFi sequence or a batch of pre-signed time-locked transfers).

3. **Time-sensitive transaction griefing**: Transactions that use `since` relative/absolute time locks have a validity window. Repeated eviction forces resubmission and can cause the window to expire, permanently preventing confirmation.

4. **Full pool wipe**: `clear_tx_pool` drops every pending and proposed transaction in one call, resetting the block assembler state. This is a complete denial-of-service against all pending transactions on the node.

---

### Likelihood Explanation

**Attack vector 1 — CSRF via browser (no network exposure needed)**

Because `CorsLayer::permissive()` is set, any web page loaded in a browser on the same machine as the node can issue cross-origin POST requests to `http://127.0.0.1:8114`. A malicious advertisement, phishing page, or compromised dApp frontend can silently call `remove_transaction` or `clear_tx_pool` against the local node without any user interaction beyond visiting the page.

**Attack vector 2 — publicly exposed RPC**

Many node operators, exchanges, and dApp backends expose the RPC port publicly (or on a LAN) for wallet and application connectivity. In that configuration any network-reachable client can call these methods directly.

**Attacker preconditions**: No keys, no privileged access, no hashpower. Only the ability to send an HTTP POST to the RPC port (directly or via CSRF).

---

### Recommendation

1. **Add an authentication token**: Require a secret bearer token (configured at node startup) in the `Authorization` header for all state-mutating RPC methods. Reject requests without a valid token with HTTP 401.

2. **Restrict CORS**: Replace `CorsLayer::permissive()` with an explicit allowlist (e.g., `null` origin only, or a configured set of trusted origins) to eliminate the CSRF vector.

3. **Ownership check for `remove_transaction`**: At minimum, record the source IP or a submitter token when a transaction is submitted via `send_transaction`, and require the same identity to call `remove_transaction` on it.

4. **Separate privileged methods**: Move `clear_tx_pool`, `clear_tx_verify_queue`, and `remove_transaction` into a separate admin module that is disabled by default and requires explicit opt-in with authentication.

---

### Proof of Concept

**Preconditions**: Alice runs a CKB node with the Pool RPC module enabled (default). She submits a pending transaction `TX_A` via `send_transaction`. Bob knows `TX_A`'s hash (observable via `get_raw_tx_pool` or public mempool explorers).

**Step 1** — Bob (or a malicious web page in Alice's browser) sends:

```http
POST http://127.0.0.1:8114/ HTTP/1.1
Content-Type: application/json

{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "remove_transaction",
  "params": ["<TX_A_hash>"]
}
```

**Step 2** — The server calls `remove_local_tx(TX_A_hash)` with no authentication check. `remove_entry_and_descendants` removes `TX_A` and all child transactions.

**Step 3** — Response: `{"result": true}`. `TX_A` is gone from the pool. Alice's transaction will never be confirmed unless she resubmits.

**Step 4** — Bob repeats for every resubmission, or calls `clear_tx_pool` once to wipe the entire pool.

**Expected outcome**: Alice's pending transaction is permanently suppressed at zero cost to the attacker. For time-locked transactions, this causes irreversible loss of the confirmation window.
