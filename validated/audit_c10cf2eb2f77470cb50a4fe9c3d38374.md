### Title
`BLOCK_INVALID` Status Ignored in `HeaderAcceptor::accept()` Allows Re-processing of Known-Invalid Headers — (`File: sync/src/synchronizer/headers_process.rs`)

### Summary
The `HeaderAcceptor::accept()` function in the CKB sync protocol checks for `BLOCK_INVALID` status before processing a received header but **does not act on it**. A developer-acknowledged `FIXME` comment explicitly notes the missing guard. Any unprivileged remote peer can send a `SendHeaders` P2P message containing headers already marked `BLOCK_INVALID`, causing the node to re-run all verification logic on them and, in edge cases where the parent's status has changed, to re-insert a previously-rejected header as valid — updating the node's `shared_best_header` and triggering unnecessary block downloads.

---

### Finding Description

In `sync/src/synchronizer/headers_process.rs`, the `HeaderAcceptor::accept()` function reads the block status and then only branches on `BLOCK_INVALID` via a comment, not code:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer best_known and return early
    return result;
}
// No check for BLOCK_INVALID here — falls through to full re-verification
``` [1](#0-0) 

After this, the function runs three sequential checks:

1. `prev_block_check` — rejects if the parent is `BLOCK_INVALID`
2. `non_contextual_check` — runs `HeaderVerifier::verify()` (PoW, timestamp, etc.)
3. `version_check` — rejects if version ≠ 0

If all three pass, `insert_valid_header` is called unconditionally:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
``` [2](#0-1) 

`insert_valid_header` inserts the header into the shared `header_map`, updates the sending peer's `best_known_header`, and may advance the global `shared_best_header`: [3](#0-2) 

The `BlockStatus` flags are defined as independent bit-fields; `BLOCK_INVALID` uses bit 12 and shares no bits with `HEADER_VALID`, so `status.contains(BlockStatus::HEADER_VALID)` is always `false` when the status is `BLOCK_INVALID`, and the early-return guard never fires for invalid headers: [4](#0-3) 

By contrast, the compact-block relay path (`compact_block_process.rs`) correctly checks and rejects `BLOCK_INVALID` before any further processing: [5](#0-4) 

A secondary instance of the same pattern exists in `BlockFetcher::fetch()`. The fetch loop only skips blocks with `BLOCK_STORED` or `BLOCK_RECEIVED` status; blocks with `BLOCK_INVALID` fall through to the `else` branch and are added to the inflight download list, causing the node to request blocks it already knows are invalid: [6](#0-5) 

---

### Impact Explanation

**Primary (`headers_process.rs`):**

- **CPU waste (always):** Every `SendHeaders` message containing a `BLOCK_INVALID` header causes the node to re-run `prev_block_check`, `non_contextual_check` (including `HeaderVerifier::verify()`), and `version_check`. A peer can spam this repeatedly.
- **State corruption (edge case):** A header H may be marked `BLOCK_INVALID` solely because its parent P was `BLOCK_INVALID` at the time. If P's entry is later removed from the in-memory `block_status_map` (e.g., via `remove_block_status` after a successful verification of a sibling chain, or after a node restart clears the map while P was never persisted to the store), `prev_block_check` will pass for H on re-submission. If H's own non-contextual checks pass, `insert_valid_header` is called, inserting H into the `header_map`, updating the peer's `best_known_header`, and potentially advancing `shared_best_header` to a block on a previously-rejected chain. This causes the block fetcher to request and download blocks from that chain, all of which will fail contextual verification, wasting bandwidth and CPU.

**Secondary (`block_fetcher.rs`):**

- Blocks with `BLOCK_INVALID` status are requested from peers, wasting bandwidth and inflight slots.

---

### Likelihood Explanation

The CPU-waste path is trivially reachable: any connected peer can send a `SendHeaders` message (P2P `SyncProtocol`) with any header hash. If the node has previously marked that hash `BLOCK_INVALID`, the missing guard means the node re-verifies it on every receipt. The state-corruption path requires the parent's in-memory status to have been cleared, which is less common but possible during normal operation (e.g., after a reorg clears entries, or across a restart where only some statuses were persisted). The FIXME comment in the production source confirms the developers are aware of the gap.

---

### Recommendation

Add an explicit `BLOCK_INVALID` guard immediately after reading the status in `HeaderAcceptor::accept()`, mirroring the pattern already used in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None); // or a dedicated ValidationError variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return
}
```

Similarly, add a `BLOCK_INVALID` skip in `BlockFetcher::fetch()`:

```rust
if status.contains(BlockStatus::BLOCK_STORED) {
    // ...
} else if status.contains(BlockStatus::BLOCK_RECEIVED) {
    // Do not download repeatedly
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    // Do not download known-invalid blocks
} else if ... {
    fetch.push(header)
}
```

### Proof of Concept

1. Connect a peer to a CKB node using the `SyncProtocol`.
2. Send any block header that the node has previously rejected and marked `BLOCK_INVALID` (e.g., a header with an invalid parent, or one that failed PoW).
3. Wrap it in a `SendHeaders` (`packed::SendHeaders`) P2P message and deliver it to the node.
4. Observe (via debug logging) that `HeaderAcceptor::accept()` is entered and all three sub-checks (`prev_block_check`, `non_contextual_check`, `version_check`) are executed, rather than the function returning immediately with an invalid status.
5. Repeat in a loop to demonstrate unbounded CPU consumption per peer connection.

For the state-corruption path: after step 2, manually remove the parent's entry from `block_status_map` (simulating a restart or eviction), then re-send the child header and observe `insert_valid_header` being called and `shared_best_header` being updated to the previously-invalid header.     pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

    pub fn get_block_status(&self, block_hash: &Byte32) -> BlockStatus {
        match self.block_status_map().get(block_hash) {
            Some(status_ref) => *status_ref.value(),
            None => {
                if self.header_map().contains_key(block_hash) {
                    BlockStatus::HEADER_VALID
                } else {
                    let verified = self
                        .snapshot()
                        .get_block_ext(block_hash)
                        .map(|block_ext| block_ext.verified);
                    match verified {
                        None => BlockStatus::UNKNOWN,
                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,
                    }
                }
            }
        }
    }

    pub fn contains_block_status<T: ChainStore>(
        &self,
        block_hash: &Byte32,
        status: BlockStatus,
    ) -> bool {
        self.get_block_status(block_hash).contains(status)
    }

    pub fn insert_block_status(&self, block_hash: Byte32, status: BlockStatus) {
        self.block_status_map.insert(block_hash, status);
