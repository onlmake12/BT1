Audit Report

## Title
Duplicate `tx_hashes` in `GetTransactionsProof` P2P Message Causes Panic via `.expect()` on `None` from `CBMT::build_merkle_proof` â€” (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` does not deduplicate `tx_hashes` before building the CBMT Merkle proof. Sending `tx_hashes = [h, h]` for a known committed transaction causes duplicate leaf indices to be passed to `CBMT::build_merkle_proof`, which returns `None`, triggering an unconditional `.expect()` panic in the light-client protocol handler. This is reachable by any unprivileged peer connected to the light-client P2P protocol.

## Finding Description

The `execute` method only checks for empty and over-limit `tx_hashes`; there is no deduplication guard: [1](#0-0) 

The `partition` call preserves duplicates â€” both copies of `h` land in `found` because the same transaction exists on-chain: [2](#0-1) 

Both copies are then pushed into `txs_in_blocks[block_hash]` as a `Vec`, producing `[(tx, I), (tx, I)]` with the same index `I`: [3](#0-2) 

`CBMT::build_merkle_proof` is then called with indices `[I, I]`. The `merkle_cbt` library rejects duplicate leaf indices and returns `None`. The unconditional `.expect()` panics: [4](#0-3) 

`CBMT` is a type alias for `ExCBMT<Byte32, MergeByte32>` from the `merkle_cbt` crate, which is known to return `None` for duplicate indices: [5](#0-4) 

By contrast, `GetBlocksProofProcess` explicitly guards against duplicates with a `HashSet` before any processing: [6](#0-5) 

The RPC path (`get_tx_indices`) also uses a `HashSet` and returns an error on duplicate indices: [7](#0-6) 

The `received` handler directly awaits `try_process` â†’ `execute` with no panic isolation: [8](#0-7) [9](#0-8) 

## Impact Explanation

A panic in an async handler propagates through the tentacle P2P framework's task for the `LightClientProtocol`. This crashes the light-client protocol service task, causing a denial-of-service against all light-client peers, or potentially the entire node process. `LightClientProtocol` is registered as a production protocol: [10](#0-9) 

This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10001â€“15000 points).

## Likelihood Explanation

Any unprivileged peer connected via the light-client P2P protocol can trigger this. The attacker only needs one committed transaction hash (trivially obtained from any block explorer or by observing the chain). Sending `tx_hashes = [h, h]` is a single crafted P2P message requiring no special privileges, no PoW, and no key material. The attack is repeatable and requires no victim interaction.

## Recommendation

Add a duplicate-hash check at the top of `execute`, mirroring the pattern in `GetBlocksProofProcess`:

```rust
let mut uniq = HashSet::new();
if !self.message.tx_hashes().to_entity().into_iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage.with_context("duplicate tx hash exists");
}
```

Alternatively, replace the `Vec` in `txs_in_blocks` with a `HashMap<tx_hash, (tx, index)>`, which naturally deduplicates by hash and prevents duplicate indices from ever reaching `CBMT::build_merkle_proof`.

## Proof of Concept

1. Start a CKB node with `LightClient` protocol enabled.
2. Mine a block containing a transaction with hash `h`.
3. Connect as a light-client peer and send:
   ```
   GetTransactionsProof {
     last_hash: <current tip hash>,
     tx_hashes: [h, h]
   }
   ```
4. Observe the node panics at `get_transactions_proof.rs:97` with message `"build proof with verified inputs should be OK"`.

Unit test equivalent: construct a `MockChain`, commit a transaction, send `GetTransactionsProof` with `tx_hashes = [tx.hash(), tx.hash()]`, and assert no panic occurs and a valid or error `Status` is returned (currently it panics before returning).
