### Title
Deviation Check in `_validateShouldUpdatePrices` Ignores Price Exponent, Causing Incorrect Trigger Decisions - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's deviation-based update trigger compares raw Pyth price mantissas without normalizing for the `expo` field. Because Pyth prices are fixed-point numbers (`actual_price = price × 10^expo`), comparing raw `int64` mantissas across updates where the exponent differs produces a wildly incorrect deviation percentage. This can cause false deviation triggers that drain subscription balances through keeper fee payments, or suppress legitimate triggers when the exponent shifts in the opposite direction.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation check reads:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;   // raw mantissa only
int64 previousPrice = previousFeed.price.price;     // raw mantissa only

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
``` [1](#0-0) 

Neither `priceFeeds[i].price.expo` nor `previousFeed.price.expo` is read or used. The `PythStructs.Price` struct carries both a `price` (mantissa) and an `expo` (scale factor), and the actual value is `price × 10^expo`. [2](#0-1) 

Pyth's own SDK (`PythUtils.convertToUint`) explicitly normalizes the exponent before any arithmetic: [3](#0-2) 

The Scheduler deviation check performs no such normalization.

---

### Impact Explanation

**Scenario A – False trigger / balance drain**

- Stored previous price: `price = 100_000_000, expo = -8` → actual = **1.00 USD**
- Incoming new price: `price = 1_000_000, expo = -6` → actual = **1.00 USD** (identical)
- Raw deviation: `|1_000_000 − 100_000_000| / 100_000_000 × 10_000 = 9_900 bps (99%)`
- With a typical `deviationThresholdBps = 100` (1%), the update is falsely triggered.
- The keeper collects `gasCost + singleUpdateKeeperFeeInWei × numPriceIds` from the subscription balance for a no-op update.

**Scenario B – Suppressed trigger**

- Stored previous price: `price = 1_000_000, expo = -6` → actual = **1.00 USD**
- Incoming new price: `price = 200_000_000, expo = -8` → actual = **2.00 USD** (100% real deviation)
- Raw deviation: `|200_000_000 − 1_000_000| / 1_000_000 × 10_000 = 1_990_000 bps`

In this direction the trigger fires (correctly by accident), but the computed bps value is astronomically wrong and could overflow `uint256` in edge cases, or interact incorrectly with any future logic that uses the computed deviation value.

The primary exploitable impact is **Scenario A**: a keeper can repeatedly submit valid Pyth updates where the exponent has changed (a legitimate Pyth protocol event), collecting keeper fees each time while the subscription balance is drained without any real price movement.

---

### Likelihood Explanation

Pyth price feeds do change their exponent. The exponent is part of the signed price attestation and is not fixed by the Scheduler contract. Any time Pyth adjusts the exponent for a subscribed feed (e.g., when a token's price range shifts by an order of magnitude), the next update submitted to the Scheduler will compute a wildly incorrect deviation. A keeper monitoring the mempool for such exponent-change updates can immediately call `updatePriceFeeds` to collect fees. No privileged access is required; `updatePriceFeeds` has no caller restriction. [4](#0-3) 

---

### Recommendation

Normalize both prices to a common exponent before computing the deviation. The simplest approach is to use `PythUtils.convertToUint` on both the current and previous price with the same `targetDecimals`, then compute the deviation on the normalized values:

```solidity
uint256 currentNorm  = PythUtils.convertToUint(
    priceFeeds[i].price.price, priceFeeds[i].price.expo, 18);
uint256 previousNorm = PythUtils.convertToUint(
    previousFeed.price.price, previousFeed.price.expo, 18);

uint256 numerator    = currentNorm > previousNorm
    ? currentNorm - previousNorm
    : previousNorm - currentNorm;
uint256 deviationBps = Math.mulDiv(numerator, 10_000, previousNorm);
```

This mirrors the normalization already used elsewhere in the Pyth SDK.

---

### Proof of Concept

1. Deploy `Scheduler` and create a subscription with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%).
2. Submit an initial update with `price = 100_000_000, expo = -8` (actual = 1.00 USD). This is stored as `previousFeed`.
3. Pyth legitimately changes the exponent for the same feed to `-6`. The new attestation carries `price = 1_000_000, expo = -6` (actual = 1.00 USD — no real price change).
4. Any keeper calls `updatePriceFeeds(subscriptionId, updateData)` with this new attestation.
5. Inside `_validateShouldUpdatePrices`:
   - `currentPrice = 1_000_000`
   - `previousPrice = 100_000_000`
   - `numerator = 99_000_000`
   - `deviationBps = 99_000_000 * 10_000 / 100_000_000 = 9_900`
   - `9_900 >= 100` → update is triggered
6. The keeper receives `gasCost + singleUpdateKeeperFeeInWei × numPriceIds` from the subscription balance, despite zero actual price movement.
7. This can be repeated every time the exponent oscillates, continuously draining the subscription. [5](#0-4) [6](#0-5)
