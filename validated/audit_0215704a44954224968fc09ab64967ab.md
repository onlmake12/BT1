### Title
Missing "Withdraw Maximum Available" Mechanism in `withdrawFunds` Causes Repeated Reverts Due to Keeper Race Conditions ‚Äî (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `withdrawFunds` function in `Scheduler.sol` requires the caller to supply an exact `amount`. For active subscriptions, the function enforces a minimum balance floor. Because the subscription balance is continuously and unpredictably drained by permissionless keeper calls to `updatePriceFeeds`, a subscription manager who computes the maximum withdrawable amount off-chain and submits a transaction can have that transaction revert if a keeper update lands in the same block. There is no "withdraw all available above minimum" sentinel (e.g., `type(uint256).max`), forcing the manager to retry with a smaller amount and pay extra gas each time.

---

### Finding Description

`withdrawFunds` in `Scheduler.sol` accepts an explicit `amount` and, when the subscription is active, enforces:

```solidity
if (status.balanceInWei - amount < minimumBalance) {
    revert SchedulerErrors.InsufficientBalance();
}
``` [1](#0-0) 

The minimum balance is computed as `numPriceFeeds √ó minimumBalancePerFeed`:

```solidity
return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
``` [2](#0-1) 

The subscription balance is drained on every successful keeper call to `updatePriceFeeds`, which deducts both the Pyth oracle fee and a dynamic keeper fee (gas cost + per-feed premium):

```solidity
status.balanceInWei -= pythFee;   // line 305
...
status.balanceInWei -= totalKeeperFee;  // line 856
``` [3](#0-2) [4](#0-3) 

Keepers are permissionless ‚Äî anyone can call `updatePriceFeeds` ‚Äî and are actively incentivized to do so. The keeper fee is gas-price-dependent and therefore not predictable off-chain at the time the manager constructs their withdrawal transaction.

The `IScheduler` interface documents the constraint but provides no "withdraw max" path for active subscriptions:

```solidity
/// @dev A minimum balance must be maintained for active subscriptions. To withdraw past
/// the minimum balance limit, deactivate the subscription first.
function withdrawFunds(uint256 subscriptionId, uint256 amount) external;
``` [5](#0-4) 

The workaround (deactivate first, then withdraw) requires an additional transaction and temporarily stops price feed updates for the subscription.

---

### Impact Explanation

A subscription manager trying to withdraw the maximum available balance above the minimum floor will have their transaction revert whenever a keeper update lands between their off-chain balance query and their transaction's inclusion. In high-activity periods (frequent heartbeat or deviation triggers), this can cause repeated reverts, wasting gas on each failed attempt. The manager is forced to either:

1. Withdraw a conservatively small amount (leaving funds stranded), or
2. Deactivate the subscription first (two transactions, service interruption), or
3. Retry multiple times, paying gas on each revert.

No funds are permanently lost, but the operational cost and friction are real and directly caused by the missing "withdraw max" mechanism.

---

### Likelihood Explanation

Keepers are permissionless and economically incentivized to call `updatePriceFeeds` as soon as trigger conditions are met. In production, multiple keepers compete to land updates. Any subscription with active heartbeat or deviation criteria will have its balance drained at unpredictable intervals. A manager attempting to withdraw the maximum available amount during normal operation will frequently encounter this race condition, especially on chains with short block times or high keeper activity.

---

### Recommendation

**Short term:** Support a sentinel value (e.g., `type(uint256).max`) in `withdrawFunds` that is interpreted as "withdraw all available above the minimum balance." The implementation would be:

```solidity
if (amount == type(uint256).max) {
    uint256 minimumBalance = params.isActive
        ? this.getMinimumBalance(uint8(params.priceIds.length))
        : 0;
    amount = status.balanceInWei > minimumBalance
        ? status.balanceInWei - minimumBalance
        : 0;
}
```

This is the same pattern used by ERC-4626 vaults and many DeFi protocols for "redeem all."

**Short term (alternative):** Expose a view function `getWithdrawableBalance(uint256 subscriptionId)` that returns the current maximum withdrawable amount, and document that this value can change between blocks due to keeper activity.

**Long term:** Consider adding a slippage/minimum-out parameter to `withdrawFunds` so the manager can specify "withdraw `amount` but accept any amount down to `minAmount`," allowing the transaction to succeed even if the balance has been partially drained.

---

### Proof of Concept

1. Alice creates a subscription with 2 price feeds. `minimumBalance = 2 √ó minimumBalancePerFeed`.
2. She adds extra funds so `balanceInWei = minimumBalance + 1 ETH`.
3. She calls `getSubscription` off-chain and computes `withdrawable = balanceInWei - minimumBalance = 1 ETH`.
4. She submits `withdrawFunds(subscriptionId, 1 ETH)`.
5. In the same block, a keeper calls `updatePriceFeeds`, deducting `0.01 ETH` in fees. Now `balanceInWei = minimumBalance + 0.99 ETH`.
6. Alice's transaction executes: `balanceInWei - amount = minimumBalance + 0.99 ETH - 1 ETH = minimumBalance - 0.01 ETH < minimumBalance` ‚Üí reverts with `InsufficientBalance`.
7. Alice must retry with `0.99 ETH` or less, paying gas again. If another keeper update lands before her retry, she must reduce the amount again.

The `_processFeesAndPayKeeper` function confirms the balance deduction is variable (gas-price-dependent) and cannot be predicted at transaction construction time:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
status.balanceInWei -= totalKeeperFee;
``` [6](#0-5)
