### Title
`RecentReject::put()` never increments `total_keys_num`, making `shrink()` permanently unreachable — (`tx-pool/src/component/recent_reject.rs`)

### Summary
`RecentReject::put()` computes a local `total_keys_num = self.total_keys_num + 1` but never writes it back. `self.total_keys_num` is frozen at its initial value (the RocksDB estimate at open time, often 0 on a fresh or restarted DB). The `shrink()` guard is therefore never triggered, and the `recent_reject` RocksDB store grows without bound.

### Finding Description
In `put()` (line 62), `self.total_keys_num.checked_add(1)` produces a temporary local value. The comparison `total_keys_num > self.count_limit` uses this local, but `self.total_keys_num` is never assigned the result. On every subsequent call, the same stale value is used. `shrink()` is only reachable if the initial RocksDB estimate already exceeds `count_limit` at open time — which does not happen on a fresh DB or after a restart where `estimate_num_keys_cf` returns 0 (a documented RocksDB behavior for unflushed data). [4](#0-3) 

### Impact Explanation
An unprivileged attacker who continuously submits transactions that are rejected (invalid fee, malformed, double-spend, etc.) via P2P relay or the `send_transaction` RPC causes unbounded writes to the `recent_reject` RocksDB store. Disk exhaustion crashes the node or causes write errors that halt tx-pool operation.

### Likelihood Explanation
- No special privilege required; rejected transactions are a normal operational event.
- The bug is present from the first `put()` call on any fresh or restarted node.
- The TTL mechanism only removes entries during compaction, which RocksDB may defer; it does not bound peak disk usage.
- The existing test passes only because it checks the broken counter, not actual DB size.

### Recommendation
In `put()`, assign the incremented value back before the comparison:

```rust
self.total_keys_num = self.total_keys_num.saturating_add(1);
if self.total_keys_num > self.count_limit {
    self.shrink()?;
}
```

`shrink()` already re-queries the real estimate and updates `self.total_keys_num`, so the counter self-corrects after each shrink.

### Proof of Concept
1. Open a `RecentReject` with `count_limit = 100`.
2. Call `put()` 200 times with distinct hashes.
3. Assert `db.estimate_num_keys_cf(cf)` across all shards sums to ~200 (far above limit).
4. Assert `recent_reject.total_keys_num == 0` (or whatever the initial estimate was) — confirming the counter was never incremented and `shrink()` was never called.
5. Repeat after a node restart to confirm the estimate-at-open is also 0, making the condition permanently false from the first write.

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        let total_keys_num = self.estimate_total_keys_num()?;
        self.total_keys_num = total_keys_num;
