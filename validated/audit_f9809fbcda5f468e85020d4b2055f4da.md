### Title
Entropy Provider Can Set Unbounded `feeInWei` Causing `getFeeV2()` Overflow DoS — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `register()` and `setProviderFee()` functions in `Entropy.sol` accept a `feeInWei` parameter with no upper-bound validation. Any permissionlessly registered provider can set `feeInWei = type(uint128).max`, causing `getFeeV2()` to revert due to uint128 overflow (when `pythFeeInWei > 0`), permanently DoS-ing all new user requests to that provider.

---

### Finding Description

`register()` performs only one parameter check — `chainLength != 0` — and sets `provider.feeInWei = feeInWei` unconditionally: [1](#0-0) 

`setProviderFee()` similarly applies no bounds check: [2](#0-1) 

`getFeeV2(address, uint32)` computes the total fee as:

```solidity
return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
``` [3](#0-2) 

`getProviderFee` returns `provider.feeInWei` directly when `defaultGasLimit == 0` or `roundedGasLimit <= defaultGasLimit`: [4](#0-3) 

If `feeInWei = type(uint128).max` and `pythFeeInWei > 0` (as in all production deployments), the addition `type(uint128).max + pythFeeInWei` overflows `uint128`. Solidity 0.8+ reverts on overflow, so `getFeeV2` becomes permanently uncallable for that provider. Even if `pythFeeInWei == 0`, the returned fee of `type(uint128).max` makes it physically impossible for any user to supply sufficient `msg.value`.

The `ProviderInfo` struct confirms `feeInWei` is `uint128` with no enforced ceiling: [5](#0-4) 

---

### Impact Explanation

Any protocol or user that calls `getFeeV2(provider, gasLimit)` — including the no-arg `requestV2()` path which uses `getDefaultProvider()` — will receive a revert. This permanently blocks new randomness requests to the affected provider. Protocols that have hardcoded a specific provider address (not the default) lose access to entropy entirely. Users with pending integrations cannot obtain fees or submit requests.

---

### Likelihood Explanation

Provider registration is fully permissionless — any address can call `register()`. A malicious actor can register, attract integrators, then call `setProviderFee(type(uint128).max)` to DoS all new requests. The `setProviderFeeAsFeeManager` path also allows a delegated fee manager to trigger this: [6](#0-5) 

No privileged role is required beyond being a registered provider, which is open to anyone.

---

### Recommendation

Add an upper-bound check in both `register()` and `setProviderFee()`:

```solidity
uint128 constant MAX_PROVIDER_FEE = type(uint128).max / 2; // or a protocol-defined cap
if (feeInWei > MAX_PROVIDER_FEE) revert EntropyErrors.AssertionFailure();
```

This prevents the overflow in `getFeeV2` and ensures fees remain practically payable.

---

### Proof of Concept

1. Call `register(type(uint128).max, commitment, metadata, 100, uri)` from any address — succeeds with no revert.
2. Call `getFeeV2(attackerProvider, 0)` — reverts with arithmetic overflow (since `type(uint128).max + pythFeeInWei` overflows).
3. Any user or protocol attempting `requestV2(attackerProvider, gasLimit)` cannot proceed because the fee query reverts before `msg.value` can be validated.
4. Alternatively, an existing provider calls `setProviderFee(type(uint128).max)` — same result. [7](#0-6) [8](#0-7)

    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);

            oldFeeInWei,
            newFeeInWei,
            bytes("")
