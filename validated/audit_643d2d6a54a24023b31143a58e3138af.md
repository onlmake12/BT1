### Title
Unbounded `pending_delivered` HashMap Growth and Uncapped `runtime::spawn` Task Flooding via Crafted Hole-Punching Messages — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

An unprivileged remote peer can exhaust memory and saturate the async runtime of a victim CKB node by sending crafted `ConnectionRequest` messages with distinct attacker-controlled `from` PeerIds (causing unbounded growth of the `pending_delivered` HashMap), then following up with `ConnectionSync` messages referencing those entries to trigger an uncapped number of `runtime::spawn` tasks, each of which opens up to 24 concurrent TCP connection attempts that run for up to 30 seconds each.

---

### Finding Description

**Phase 1 — Unbounded `pending_delivered` insertion**

`pending_delivered` is a plain `HashMap<PeerId, PendingDeliveredInfo>` with no size cap. [1](#0-0) 

Insertion happens in `ConnectionRequestProcess::respond_delivered()`: [2](#0-1) 

The only guards before insertion are:

1. **Per-session rate limiter** — 30 req/s per `(session_id, msg_item_id)`. This is the only real throttle. [3](#0-2) 

2. **`forward_rate_limiter`** — 1/s per `(from, to, msg_item_id)`. Because the attacker uses a **distinct `from` PeerId per message**, each message creates a new key in the rate limiter. This guard is completely bypassed. [4](#0-3) 

3. **`HOLE_PUNCHING_INTERVAL` guard** — only blocks re-insertion of the **same** `from_peer_id` within 2 minutes. With distinct `from` PeerIds this is also bypassed. [5](#0-4) 

The `from` field is attacker-controlled arbitrary bytes; the victim's peer ID (the `to` field) is public information exchanged during the P2P handshake. The `listen_addrs` need only be syntactically valid TCP addresses (e.g., pointing to a black hole).

Result: at 30 req/s over the 5-minute cleanup window, a single session inserts **30 × 300 = 9,000 entries** into `pending_delivered` before `notify()` runs. [6](#0-5) 

**Phase 2 — Uncapped `runtime::spawn` task flooding**

When a `ConnectionSync` message arrives with `to` = victim and `from` matching a `pending_delivered` entry, the code unconditionally spawns a new async task: [7](#0-6) 

There is no cap on how many tasks may be spawned. Each task calls `select_ok(tasks)` where `tasks` contains up to 24 `try_nat_traversal` futures. Each `try_nat_traversal` runs a retry loop for up to **30 seconds**, creating a new TCP socket and attempting a connection every ~200 ms: [8](#0-7) 

The `forward_rate_limiter` for `ConnectionSync` is also keyed by `(from, to, msg_item_id)`, so distinct `from` PeerIds bypass it identically: [9](#0-8) 

At 30 `ConnectionSync` messages/s, after 30 seconds there are **900 concurrent tasks**, each holding up to 24 TCP connection futures = **21,600 concurrent TCP connection attempts** from a single session.

**Compounding factor — `forward_rate_limiter` itself grows unboundedly**

The `forward_rate_limiter` is a `HashMapStateStore<(PeerId, PeerId, u32)>` with no size cap. `retain_recent()` is only called on session disconnect, not during normal operation. With 9,000 distinct `from` PeerIds, the rate limiter itself accumulates 9,000 entries. [10](#0-9) [11](#0-10) 

---

### Impact Explanation

- **Memory**: `pending_delivered` grows to ~9,000 entries per session (each holding up to 24 `Multiaddr` objects ≈ 2–3 KB/entry ≈ 27 MB), plus the `forward_rate_limiter` HashMap, plus heap allocations for each spawned task's futures.
- **Async runtime saturation**: Tens of thousands of concurrent TCP connection attempts (each lasting up to 30 s) starve the Tokio thread pool, blocking legitimate protocol processing across all sessions.
- **Scales with sessions**: Multiple attacker sessions multiply the effect linearly; the `forward_rate_limiter` and `pending_delivered` are shared state with no global cap.

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no privileges, no PoW, no leaked keys. The `from` field is arbitrary bytes; the victim's peer ID is public. The attacker needs to craft well-formed `ConnectionRequest` and `ConnectionSync` messages, which is straightforward given the published protobuf schema. The rate limit of 30/s per session is the only real throttle, and it is insufficient to prevent the attack over the 5-minute cleanup window.

---

### Recommendation

1. **Cap `pending_delivered` size**: Reject insertion when `pending_delivered.len()` exceeds a small constant (e.g., 256 or `max_outbound`).
2. **Cap spawned tasks**: Track the number of in-flight NAT traversal tasks and refuse to spawn new ones beyond a limit.
3. **Global rate limit on unique `from` PeerIds**: Add a counter or bloom filter to bound the number of distinct `from` PeerIds accepted per cleanup interval, independent of the per-`(from,to)` rate limiter.
4. **Periodic `retain_recent()` on `forward_rate_limiter`**: Call it in `notify()` rather than only on disconnect to prevent unbounded growth of the rate limiter itself.

### Proof of Concept

```
1. Attacker connects to victim (single P2P session).
2. For i in 1..9000:
     Send ConnectionRequest {
       from = PeerId::random(),   // distinct each time
       to   = victim_peer_id,     // public, from handshake
       listen_addrs = ["1.2.3.4:9999"],  // unreachable TCP addr
       route = [],
       max_hops = 6,
     }
     // Passes forward_rate_limiter (new key each time)
     // Passes HOLE_PUNCHING_INTERVAL check (new from each time)
     // Inserts into pending_delivered
     sleep(1/30 s)  // stay within per-session 30 req/s limit

3. Record all 9000 from PeerIds used above.

4. For each from_i in recorded_from_peerids:
     Send ConnectionSync {
       from  = from_i,
       to    = victim_peer_id,
       route = [],
     }
     // Passes forward_rate_limiter (distinct key)
     // Finds pending_delivered[from_i] -> spawns runtime::spawn task
     // Task runs try_nat_traversal for 30s, making TCP connects to 1.2.3.4:9999

5. Observe: victim heap grows; Tokio thread pool saturated;
   legitimate sync/block messages stop being processed.
```
