Let me check for any body size limits in the axum router configuration and the RpcConfig structure.

### Title
Unbounded JSON-RPC Batch Processing When `rpc_batch_limit` Is Unconfigured — (`rpc/src/server.rs`)

### Summary

When an operator does not set `rpc_batch_limit` in the CKB config, `JSONRPC_BATCH_LIMIT` remains an empty `OnceLock`. The batch-size guard in `handle_jsonrpc` is gated on `JSONRPC_BATCH_LIMIT.get()` returning `Some`; when it returns `None` the guard is skipped entirely and every call in the batch is executed with no upper bound. An unprivileged HTTP caller can submit a single POST containing thousands of expensive RPC calls and drive the node to CPU/memory exhaustion.

### Finding Description

`JSONRPC_BATCH_LIMIT` is a process-wide `OnceLock<usize>` declared at module level: [1](#0-0) 

It is populated only when the operator explicitly sets `rpc_batch_limit`: [2](#0-1) 

The corresponding config field is `Option<usize>` with no default: [3](#0-2) 

The batch-size enforcement in `handle_jsonrpc` is:

```rust
if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
    && calls.len() > *batch_size
{
    return make_error_response(...);
}
``` [4](#0-3) 

When `JSONRPC_BATCH_LIMIT` is empty (the default), `get()` returns `None`, the `if let` arm is never entered, and execution falls through to unbounded sequential processing: [5](#0-4) 

There is no `RequestBodyLimitLayer` or `DefaultBodyLimit` applied to the axum router, so body size is also unconstrained: [6](#0-5) 

The only mitigating layer is a 30-second `TimeoutLayer`. This does not prevent the attack because:
- Memory for all parsed calls is allocated before the timeout begins counting against processing.
- An attacker can open many concurrent connections, each consuming up to 30 seconds of CPU and holding a large allocation.
- The streaming response (`StreamBodyAs::json_array`) keeps the connection and its resources live for the full timeout window.

The test harness explicitly sets `rpc_batch_limit: Some(1000)`, confirming the developers are aware a limit is needed but left the production default unguarded: [7](#0-6) 

### Impact Explanation

An unprivileged caller with HTTP access to the RPC port (default 8114) can exhaust node CPU and memory by sending a single POST with a JSON array of thousands of expensive calls (`get_block_by_number`, `get_raw_tx_pool` with `verbose=true`, `estimate_cycles`). With concurrent connections the effect compounds. The node becomes unresponsive or OOM-killed, constituting a denial-of-service against the RPC server and indirectly against all node functions (block relay, tx-pool, miner).

### Likelihood Explanation

The default `ckb.toml` does not set `rpc_batch_limit`, so every operator who uses the generated config is exposed. The RPC port is sometimes exposed to semi-trusted or public networks (e.g., exchange infrastructure, public RPC endpoints). The attack requires only a standard HTTP client and knowledge of any valid RPC method name.

### Recommendation

1. Set a hard-coded safe default for `JSONRPC_BATCH_LIMIT` (e.g., 20) that is used when `rpc_batch_limit` is absent, rather than leaving the `OnceLock` empty.
2. Apply `axum::extract::DefaultBodyLimit::max(N)` as a router layer to cap request body size independently of the batch limit.
3. Document the `rpc_batch_limit` field as security-critical in the default `ckb.toml`.

### Proof of Concept

```python
import requests, json

url = "http://<node>:8114/"
batch = [
    {"jsonrpc": "2.0", "method": "get_block_by_number",
     "params": ["0x1"], "id": i}
    for i in range(50_000)
]
# JSONRPC_BATCH_LIMIT.get() == None → guard skipped → all 50k calls execute
r = requests.post(url, json=batch, stream=True, timeout=120)
# Node CPU pegs at 100%; memory grows; /ping stops responding
```
