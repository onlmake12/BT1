### Title
Unprivileged Front-Run of `Initialize` Allows Attacker to Seize Permanent Authority Over Publisher Registration — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary
The `initialize` instruction has no access control on who may call it or what `authority` value they supply. Any signer can race the deployer to call `Initialize` first, writing an attacker-controlled pubkey into the singleton config PDA. Because the config PDA is a one-time-write singleton, the deployer's subsequent `Initialize` will always fail with `AlreadyInitialized`, and the attacker permanently controls which publishers and buffers are registered.

### Finding Description

`validate_payer` only checks that `key[0]` is a signer and writable — it imposes no identity constraint: [1](#0-0) 

`InitializeArgs.authority` is a raw 32-byte field with no validation — any bytes are accepted: [2](#0-1) 

`initialize` writes whatever `args.authority` the caller supplied directly into the config PDA: [3](#0-2) 

`accounts::config::create` rejects a second call only after the first succeeds (`data.format != 0`), making the config PDA a permanent, one-time-write singleton: [4](#0-3) 

On Solana, program deployment and initialization are separate transactions. In the window between deployment and the deployer's `Initialize` call, any observer can submit their own `Initialize` with `attacker_pubkey` as `authority`. Whichever transaction lands first wins permanently.

### Impact Explanation

The `authority` stored in the config PDA is the sole gate for `InitializePublisher`: [5](#0-4) 

An attacker who seizes the authority can:
- Register arbitrary publisher/buffer pairs via `InitializePublisher`.
- Block the legitimate deployer from ever registering real publishers (since the deployer cannot call `Initialize` again and cannot update the authority — there is no `UpdateAuthority` instruction).

The program is permanently bricked for the legitimate operator unless it is redeployed.

### Likelihood Explanation

Solana's mempool is public. Any bot monitoring for new program deployments can detect the program ID and immediately submit a front-running `Initialize`. The attack requires no privileged access, no leaked keys, and no governance majority — only the ability to submit a transaction before the deployer.

### Recommendation

Restrict `Initialize` to a known, hard-coded privileged signer (e.g., the program upgrade authority), or pass the expected authority as an account key that must sign the transaction rather than as a free data field. For example:

- Require `key[0]` (payer) to equal a hard-coded deployer pubkey checked in `validate_payer`, **or**
- Require the account at `key[0]` to be the BPF upgradeable loader's upgrade authority for this program ID, **or**
- Accept `authority` only as a signer account (`key[3] authority [signer]`) so the authority must co-sign the `Initialize` transaction, making it impossible to impersonate.

pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)

pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);
    }
    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
    if data.format != 0 {
        return Err(ReadAccountError::AlreadyInitialized);
    }
    data.format = FORMAT;
    data.authority = authority;
    Ok(data)
