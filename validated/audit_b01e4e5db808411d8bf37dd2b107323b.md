### Title
Delegator Can Escape Slashing by Calling `undelegate` Before `slash` Is Applied to Their Account — (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

In Pyth's Oracle Integrity Staking (OIS) system, the `undelegate` instruction does not include `delegationRecord` in its account set and therefore cannot enforce that all pending slash events have been applied before a delegator removes their position. A delegator who monitors the chain for `create_slash_event` transactions can front-run the permissionless `slash` call by undelegating first, escaping the slash penalty entirely.

---

### Finding Description

The OIS integrity pool uses a two-step slashing flow:

1. **`create_slash_event`** — A privileged instruction (requires `reward_program_authority` signer) that records a slash event on-chain for a given publisher. Once executed, the event is publicly visible on-chain.

2. **`slash`** — A permissionless instruction that applies a recorded slash event to a specific delegator's `stakeAccountPositions`, using `delegationRecord.next_slash_event_index` to track which slashes have been applied.

The `DelegationRecord` account tracks two fields:
- `last_epoch` — last epoch for which rewards were advanced
- `next_slash_event_index` — the next slash event index that must be applied to this delegator

The `undelegate` instruction's account set (from the IDL) is:

```
owner, poolData, poolConfig, publisher, configAccount,
stakeAccountPositions, stakeAccountMetadata, stakeAccountCustody,
stakingProgram, systemProgram
```

`delegationRecord` is **absent** from `undelegate`'s accounts. This means the on-chain program cannot read or enforce `next_slash_event_index` during `undelegate`. There is no on-chain check that all pending slash events have been applied before a delegator removes their position.

By contrast, the `slash` instruction **does** include `delegationRecord` as a writable account and requires `stakeAccountPositions` to still hold the delegated position. Once a position is undelegated (moved to cooldown), the slash can no longer be applied to it.

The Pythian Council has up to one full epoch after the incident epoch to finalize a slash event. This creates a multi-epoch window during which a delegator can observe `create_slash_event` on-chain and call `undelegate` before `slash` is executed against their account. ### Impact Explanation

Slashing is the core security mechanism of OIS. It penalizes publishers (and their delegators) for data misprintings. If delegators can escape slashing by front-running the `slash` call with `undelegate`, the economic deterrent is broken:

- The delegator retains their full stake despite being in a pool that caused harm.
- The remaining delegators who did not escape absorb a larger proportional slash.
- The protocol's security guarantee — that bad oracle behavior is penalized — is undermined.

This is a direct analog to the external report: just as a staker could withdraw before `updateStrategyRewards()` was called to avoid losses, a Pyth OIS delegator can `undelegate` before `slash` is applied to avoid their share of the penalty. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

- `create_slash_event` is a public on-chain transaction, immediately visible to all participants.
- The Pythian Council has up to one epoch after the incident to finalize the slash event, giving delegators ample time to react.
- `undelegate` is a permissionless, self-service instruction callable by any delegator at any time.
- No special knowledge or privileged access is required — only the ability to monitor on-chain events and submit a transaction.
- The SDK's `unstakeFromPublisher` and `unstakeFromAllPublishers` methods make this trivially automatable. [7](#0-6) [8](#0-7) 

---

### Recommendation

The `undelegate` instruction should enforce that all pending slash events have been applied to the delegator's `delegationRecord` before allowing undelegation. Concretely:

1. Add `delegationRecord` to the `undelegate` account set.
2. Before processing the undelegation, verify that `delegation_record.next_slash_event_index == pool_data.num_slash_events[publisher_index]` (i.e., no unapplied slash events remain for this delegator).
3. Alternatively, atomically apply all pending slashes within the `undelegate` instruction itself before removing the position.

This mirrors the recommendation in the external report: require the state-update step (`advance_delegation_record` / `slash`) to be completed before the exit step (`undelegate`) is permitted. [9](#0-8) 

---

### Proof of Concept

1. Publisher P is in a pool. Delegator D has staked 1,000,000 PYTH to P's pool.
2. P publishes a bad price. The Pythian Council creates a slash event via `create_slash_event` (slash ratio = 500 bps). This transaction is visible on-chain.
3. D observes the `create_slash_event` transaction before any `slash` call is made against their account.
4. D immediately calls `undelegate(position_index, full_amount)` on their position. Because `undelegate` does not check `delegationRecord.next_slash_event_index`, this succeeds.
5. D's position is now in cooldown state. The `slash` instruction, which requires the position to be in the delegated state, can no longer be applied to D's account.
6. D waits out the cooldown period and withdraws their full 1,000,000 PYTH, having paid zero slashing penalty.
7. The remaining delegators in P's pool absorb D's share of the slash on top of their own. [10](#0-9) [11](#0-10) [12](#0-11)

## Slashing

Slashing is an important aspect of the OIS protocol to ensure the integrity of the system.

The slashed amount for each pool is calculated as follows:

$$
\large{\bold{SL_p} = w \cdot S_p = w \cdot (S^{p}_{p} + S^{d}_{p})}
$$

Where:

- $SL_p$ is the slashed amount for the publisher $p$ pool.
- $w$ is the slashing rate.
- $S_p$ is the stake assigned to the publisher $p$ pool, made of self-staked amount $S^{p}_{p}$ and delegated stake $S^{d}_{p}$, or $S_{p} = S^{p}_{p} + S^{d}_{p}$.

Here $SL_p$ is uniformly allocated to both the self-staking publisher and delegators in the pool, pro-rata to their respective stake.

  public async unstakeFromPublisher(
    stakeAccountPositions: PublicKey,
    publisher: PublicKey,
    positionState: PositionState.LOCKED | PositionState.LOCKING,
    amount: bigint,
  ) {
    const stakeAccountPositionsData = await this.getStakeAccountPositions(
      stakeAccountPositions,
    );
    const currentEpoch = await getCurrentEpoch(this.connection);

    let remainingAmount = amount;
    const instructionPromises: Promise<TransactionInstruction>[] = [];

    const eligiblePositions = stakeAccountPositionsData.data.positions
      .map((p, i) => ({ index: i, position: p }))
      .reverse()
      .filter(
        ({ position }) =>
          position.targetWithParameters.integrityPool?.publisher !==
            undefined &&
          position.targetWithParameters.integrityPool.publisher.equals(
            publisher,
          ) &&
          positionState === getPositionState(position, currentEpoch),
      );

    for (const { position, index } of eligiblePositions) {
      if (position.amount < remainingAmount) {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        remainingAmount -= position.amount;
      } else {
        instructionPromises.push(
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(remainingAmount))
            .accounts({
              publisher,
              stakeAccountPositions,
            })
            .instruction(),
        );
        break;
      }
    }

    const instructions = await Promise.all(instructionPromises);
    return sendTransaction(instructions, this.connection, this.wallet);
  }

  public async unstakeFromAllPublishers(
    stakeAccountPositions: PublicKey,
    positionStates: (PositionState.LOCKED | PositionState.LOCKING)[],
  ) {
    const [stakeAccountPositionsData, currentEpoch] = await Promise.all([
      this.getStakeAccountPositions(stakeAccountPositions),
      getCurrentEpoch(this.connection),
    ]);

    const instructions = await Promise.all(
      stakeAccountPositionsData.data.positions
        .map((position, index) => {
          const publisher =
            position.targetWithParameters.integrityPool?.publisher;
          return publisher === undefined
            ? undefined
            : { index, position, publisher };
        })
        // By separating this filter from the next, typescript can narrow the
        // type and automatically infer that there will be no `undefined` values
        // in the array after this line.  If we combine those filters,
        // typescript won't narrow properly.
        .filter((positionInfo) => positionInfo !== undefined)
        .filter(({ position }) =>
          (positionStates as PositionState[]).includes(
            getPositionState(position, currentEpoch),
          ),
        )
        .reverse()
        .map(({ position, index, publisher }) =>
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({ publisher, stakeAccountPositions })
            .instruction(),
        ),
    );

    return sendTransaction(instructions, this.connection, this.wallet);
  }
