### Title
Missing Upper Bound on `heartbeatSeconds` in Permanent Subscriptions Locks ETH Forever - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
`Scheduler._validateSubscriptionParams` enforces only a lower-bound (`!= 0`) on `heartbeatSeconds` and `deviationThresholdBps`, but no upper bound. When a permanent subscription is created with an astronomically large `heartbeatSeconds` (e.g., `type(uint32).max` ≈ 136 years), the heartbeat condition can never be satisfied in practice, and because the subscription is permanent, both `withdrawFunds` and `updateSubscription` are permanently blocked. The deposited ETH is locked in the contract forever.

### Finding Description

`_validateSubscriptionParams` in `Scheduler.sol` validates `heartbeatSeconds` only against zero:

```solidity
if (
    params.updateCriteria.updateOnHeartbeat &&
    params.updateCriteria.heartbeatSeconds == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [1](#0-0) 

There is no upper-bound check. A value of `type(uint32).max` (4,294,967,295 seconds ≈ 136 years) passes validation silently.

The heartbeat trigger condition in `_validateShouldUpdatePrices` is:

```solidity
if (
    lastUpdateTime == 0 ||
    updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds
) {
    return updateTimestamp;
}
``` [2](#0-1) 

With `heartbeatSeconds = type(uint32).max`, this condition (after the first update) will never be satisfied within any realistic timeframe.

For permanent subscriptions, both escape hatches are blocked:

- `withdrawFunds` reverts unconditionally for permanent subscriptions:

```solidity
if (params.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [3](#0-2) 

- `updateSubscription` also reverts for permanent subscriptions:

```solidity
if (currentParams.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

The same issue applies to `deviationThresholdBps`: setting it to `type(uint32).max` (~42,949,673%) means the deviation condition can never be met for any real asset. If `updateOnHeartbeat = false` and only `updateOnDeviation = true`, the subscription never updates and, for permanent subscriptions, the ETH is locked. [5](#0-4) 

### Impact Explanation

For a permanent subscription created with `heartbeatSeconds = type(uint32).max` (and `updateOnDeviation = false`), the deposited ETH balance is permanently locked in the `Scheduler` contract. There is no admin rescue path, no governance override, and no way for the subscription manager to recover the funds. The subscription remains active and visible to keepers but can never be updated (after the first free update at `lastUpdateTime == 0`), so the balance drains only on the first call and then sits idle forever. [6](#0-5) 

### Likelihood Explanation

Low. This requires the subscription manager to fat-finger or misconfigure `heartbeatSeconds` when creating a permanent subscription. Non-permanent subscriptions are not affected because the manager can call `updateSubscription` to correct the value. The scenario is analogous to the NFTLootbox `_duration` fat-finger described in the reference report. [7](#0-6) 

### Recommendation

Add an upper-bound check for `heartbeatSeconds` and `deviationThresholdBps` inside `_validateSubscriptionParams`. For example:

```solidity
uint32 constant MAX_HEARTBEAT_SECONDS = 365 days; // ~1 year
uint32 constant MAX_DEVIATION_BPS = 10_000;        // 100%

if (params.updateCriteria.updateOnHeartbeat &&
    params.updateCriteria.heartbeatSeconds > MAX_HEARTBEAT_SECONDS) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
if (params.updateCriteria.updateOnDeviation &&
    params.updateCriteria.deviationThresholdBps > MAX_DEVIATION_BPS) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [8](#0-7) 

### Proof of Concept

1. Attacker/user calls `createSubscription` with:
   - `isPermanent = true`
   - `updateCriteria.updateOnHeartbeat = true`
   - `updateCriteria.heartbeatSeconds = type(uint32).max` (4,294,967,295)
   - `updateCriteria.updateOnDeviation = false`
   - `msg.value = getMinimumBalance(numFeeds)` (e.g., 1 ETH)

2. `_validateSubscriptionParams` passes — only `heartbeatSeconds != 0` is checked.

3. Subscription is stored as permanent with the extreme heartbeat.

4. On the very first `updatePriceFeeds` call, `lastUpdateTime == 0` so the condition passes and the subscription updates once (spending some ETH on keeper fees).

5. All subsequent `updatePriceFeeds` calls revert with `UpdateConditionsNotMet` because `updateTimestamp < lastUpdateTime + type(uint32).max`.

6. `withdrawFunds(subscriptionId, amount)` → reverts `CannotUpdatePermanentSubscription`.

7. `updateSubscription(subscriptionId, fixedParams)` → reverts `CannotUpdatePermanentSubscription`.

8. Remaining ETH balance is permanently locked. [9](#0-8) [10](#0-9)

    /// @notice Validates subscription parameters.
    /// @param params The subscription parameters to validate.
    function _validateSubscriptionParams(
        SchedulerStructs.SubscriptionParams memory params
    ) internal pure {
        // No zero‐feed subscriptions
        if (params.priceIds.length == 0) {
            revert SchedulerErrors.EmptyPriceIds();
        }

        // Price ID limits and uniqueness
        if (
            params.priceIds.length >
            SchedulerConstants.MAX_PRICE_IDS_PER_SUBSCRIPTION
        ) {
            revert SchedulerErrors.TooManyPriceIds(
                params.priceIds.length,
                SchedulerConstants.MAX_PRICE_IDS_PER_SUBSCRIPTION
            );
        }
        for (uint i = 0; i < params.priceIds.length; i++) {
            for (uint j = i + 1; j < params.priceIds.length; j++) {
                if (params.priceIds[i] == params.priceIds[j]) {
                    revert SchedulerErrors.DuplicatePriceId(params.priceIds[i]);
                }
            }
        }

        // Whitelist size limit and uniqueness
        if (params.readerWhitelist.length > MAX_READER_WHITELIST_SIZE) {
            revert SchedulerErrors.TooManyWhitelistedReaders(
                params.readerWhitelist.length,
                MAX_READER_WHITELIST_SIZE
            );
        }
        for (uint i = 0; i < params.readerWhitelist.length; i++) {
            for (uint j = i + 1; j < params.readerWhitelist.length; j++) {
                if (params.readerWhitelist[i] == params.readerWhitelist[j]) {
                    revert SchedulerErrors.DuplicateWhitelistAddress(
                        params.readerWhitelist[i]
                    );
                }
            }
        }

        // Validate update criteria
        if (
            !params.updateCriteria.updateOnHeartbeat &&
            !params.updateCriteria.updateOnDeviation
        ) {
            revert SchedulerErrors.InvalidUpdateCriteria();
        }
        if (
            params.updateCriteria.updateOnHeartbeat &&
            params.updateCriteria.heartbeatSeconds == 0
        ) {
            revert SchedulerErrors.InvalidUpdateCriteria();
        }
        if (
            params.updateCriteria.updateOnDeviation &&
            params.updateCriteria.deviationThresholdBps == 0
        ) {
            revert SchedulerErrors.InvalidUpdateCriteria();
        }
    }

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
