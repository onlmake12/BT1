Audit Report

## Title
`PoolMap::add_entry` Overwrites `total_tx_size`/`total_tx_cycles` With Stale Pre-Computed Values After In-Flight Evictions — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `add_entry`, `updated_stat_for_add_tx` captures a snapshot of `total_tx_size + entry.size` before `check_and_record_ancestors` runs. When `check_and_record_ancestors` evicts pool entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size` in place. The unconditional assignment at L218–219 then overwrites the correctly-decremented live field with the stale pre-eviction snapshot, permanently overcounting `total_tx_size` by the sum of evicted entries' sizes. This causes `limit_size` to evict additional legitimate transactions that should remain in the pool.

## Finding Description
`updated_stat_for_add_tx` (L711–729) takes `&self` (immutable borrow) and returns candidate totals without modifying `self`:

```rust
// L716: snapshot only — self is not mutated
let total_tx_size = self.total_tx_size.checked_add(tx_size)...
``` [1](#0-0) 

In `add_entry`, the snapshot is captured at L210–211, then `check_and_record_ancestors` is called at L213: [2](#0-1) 

`check_and_record_ancestors` enters the eviction branch (L603–625) when `ancestors_count > max_ancestors_count` but `ancestors_count.saturating_sub(cell_ref_parents.len()) <= max_ancestors_count`, calling `remove_entry_and_descendants` at L618: [3](#0-2) 

`remove_entry_and_descendants` → `remove_entry` calls `update_stat_for_remove_tx` at L247, which decrements `self.total_tx_size` in place at L738–740: [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, `self.total_tx_size` correctly equals `original − evicted_sizes`. The assignment at L218 then sets it back to `original + entry.size`, discarding the eviction credits:

```
correct value  = original − evicted_sizes + entry.size
reported value = original + entry.size   (overcounted by evicted_sizes)
```

The `cell_ref_parents` eviction path is confirmed reachable: L531–533 populates `cell_ref_parents` whenever a new transaction's input's `previous_output` is found in `self.edges.deps` — i.e., when an existing pool transaction declared that same output as a `cell_dep`: [6](#0-5) 

## Impact Explanation
`limit_size` (pool.rs L298) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting entries until the condition is false: [7](#0-6) 

An overcounted `total_tx_size` causes it to evict `evicted_sizes` worth of additional legitimate pending/proposed transactions with `Reject::Full`, even though the pool is physically within limits. This matches the **High (10001–15000 points)** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs.*

## Likelihood Explanation
The trigger requires submitting a transaction whose input spends an output simultaneously referenced as a `cell_dep` by existing pool transactions, with the resulting ancestor count exceeding `max_ancestors_count` (default 25) while the non-cell-ref ancestor count does not. An unprivileged attacker can construct this deliberately: submit N transactions each declaring a known on-chain output `O` as a `cell_dep`, then submit a transaction spending `O` as an input via the standard `send_transaction` RPC. The damage (overcount) scales with N, making repeated triggering economically viable for disruption.

## Recommendation
Remove the pre-computation of `(total_tx_size, total_tx_cycles)` before `check_and_record_ancestors`. Perform only the overflow check, then after all mutations are complete, increment the live fields directly:

```rust
// Overflow check only — do not capture snapshot totals
self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Increment against the already-eviction-decremented live fields:
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .expect("overflow already checked above");
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .expect("overflow already checked above");
```

## Proof of Concept
1. Create a `PoolMap` with `max_ancestors_count = 25`.
2. Insert 26 transactions `T1…T26`, each declaring output `O` of an on-chain transaction `P` as a `cell_dep`. Each has no pool-internal parents (`ancestors_count = 1`). Confirm `total_tx_size = 26 * S`.
3. Insert `T_attack` whose input spends output `O` of `P`. Inside `add_entry`: `updated_stat_for_add_tx` snapshots `26*S + S_attack`; `check_and_record_ancestors` finds `cell_ref_parents = {T1…T26}`, `ancestors_count = 27 > 25`, evicts 2 entries via `remove_entry_and_descendants`, decrementing `self.total_tx_size` to `24*S`; the assignment at L218 sets `self.total_tx_size = 26*S + S_attack` (overcounted by `2*S`).
4. Assert `pool_map.total_tx_size == 26*S + S_attack` (overcounted) vs. correct `24*S + S_attack`.
5. Call `limit_size` with `max_tx_pool_size` just above `24*S + S_attack`; observe it evicts additional legitimate transactions due to the inflated counter despite the pool being physically within limits.
