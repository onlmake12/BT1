Audit Report

## Title
Unbounded `vec!` Allocation via Uncapped `max_bucket_index_by_fee_rate` Causes OOM Node Crash ‚Äî (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary

`Algorithm::max_bucket_index_by_fee_rate` applies a linear formula with no upper bound in its final match arm, allowing arbitrarily large bucket indices for high fee rates. `do_estimate` uses this index directly to size two `Vec<u64>` allocations without any cap or guard. A single high-fee-rate transaction in the mempool combined with a call to `estimate_fee_rate` causes the node to attempt a multi-GB heap allocation, crashing via OOM.

## Finding Description

**Root cause 1 ‚Äî unbounded `max_bucket_index_by_fee_rate`:**

The final match arm at `weight_units_flow.rs:357` is:
```rust
x => (x + t * 11_500) / (100 * t),   // t = 1000
```
There is no cap on the returned `usize`. For a fee of 10,000 CKB (10^12 shannons) on a 60-byte transaction, `FeeRate::calculate` yields `fee_rate = (10^12 * 1000) / 60 ‚âà 1.667 √ó 10^13`, producing `max_bucket_index ‚âà 1.67 √ó 10^8`.

**Root cause 2 ‚Äî unchecked `vec!` allocations in `do_estimate`:**

At lines 219 and 248:
```rust
let mut buckets = vec![0u64; max_bucket_index + 1];
```
No bounds check exists before either allocation. Two allocations of `167_000_001 √ó 8` bytes ‚âà 2.67 GB total.

**Root cause 3 ‚Äî `FeeRate::calculate` uses `saturating_mul`:**

At `fee_rate.rs:15`, `fee.as_u64().saturating_mul(KW) / weight`. For fees ‚â• ~1.844 √ó 10^16 shannons, `fee * 1000` saturates to `u64::MAX`, yielding `fee_rate ‚âà 3.07 √ó 10^17` and a bucket index in the trillions ‚Äî making the allocation astronomically larger.

**Exploit path:**
1. Attacker submits a transaction with a high fee (e.g., 10,000 CKB) and minimum weight (60 bytes) to the mempool.
2. Attacker (or any caller with RPC access) invokes `estimate_fee_rate`.
3. `process.rs:estimate_fee_rate` calls `get_all_entry_info` then `FeeEstimator::estimate_fee_rate` ‚Üí `Algorithm::estimate_fee_rate` ‚Üí `do_estimate`.
4. `do_estimate` computes `max_bucket_index ‚âà 1.67 √ó 10^8` and attempts two `vec![0u64; ...]` allocations totaling ~2.67 GB. The OS OOM killer terminates the process.

**No existing guards:** There is no fee-rate cap in tx-pool admission, no maximum bucket index constant, and no pre-allocation size check anywhere in this path.

## Impact Explanation

This matches **High: Vulnerabilities which could easily crash a CKB node**. For any node operator who has opted into `WeightUnitsFlow` (via `algorithm = "WeightUnitsFlow"` in `[fee_estimator]`) and exposed the RPC beyond localhost, the crash is deterministic and repeatable via a single transaction submission and one RPC call. The node process is terminated by the OS OOM killer with no recovery path short of restart.

## Likelihood Explanation

Two opt-in conditions are required: `WeightUnitsFlow` must be explicitly configured (it is commented out in the default `ckb.toml`), and the RPC must be reachable by the attacker (default binding is `127.0.0.1:8114`, but public nodes commonly expose it). For nodes meeting both conditions, the attack requires only a temporary capital outlay (~10,000 CKB) and a single RPC call. The attacker does not permanently lose funds if the node crashes before block confirmation. The attack is repeatable and requires no special privileges.

## Recommendation

Cap the return value of `max_bucket_index_by_fee_rate` to a safe constant:

```rust
const MAX_BUCKET_INDEX: usize = 1000;

fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
    let t = FEE_RATE_UNIT;
    let index = match fee_rate.as_u64() {
        x if x <= 10_000 => x / t,
        x if x <= 50_000 => (x + t * 10) / (2 * t),
        x if x <= 200_000 => (x + t * 100) / (5 * t),
        x if x <= 500_000 => (x + t * 400) / (10 * t),
        x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
        x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
        x => (x + t * 11_500) / (100 * t),
    };
    (index as usize).min(MAX_BUCKET_INDEX)
}
```

Alternatively, add a guard in `do_estimate` immediately after computing `max_bucket_index`:
```rust
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_BUCKET_INDEX);
```

This test fails on the current code (index ‚âà 167,000,000), demonstrating the unbounded growth. Running `do_estimate` with a mempool entry at this fee rate will trigger an OOM crash on any system without sufficient free memory.
