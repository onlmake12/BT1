Audit Report

## Title
Forward Rate Limiter Bypass via Attacker-Controlled `content.from`/`content.to` Enables Unbounded Forwarding and Memory Growth - (`network/src/protocols/hole_punching/`)

## Summary

The `HolePunching` protocol's `forward_rate_limiter` is keyed entirely by attacker-supplied `(content.from, content.to, msg_item_id)` values parsed from the message payload. An attacker can rotate `content.from` on every message to generate a fresh rate-limit bucket each time, bypassing the 1 req/sec per-pair cap and causing the node to forward up to 30 messages/sec (the outer cap) â€” a 30Ă— amplification over the intended limit. The `HashMapStateStore` backing the limiter is never cleaned up during a live connection, growing without bound at 30 entries/sec.

## Finding Description

`HolePunching` declares two rate limiters:

```rust
rate_limiter: RateLimiter<(PeerIndex, u32)>,
forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
``` [1](#0-0) 

The outer limiter is checked first in `received()`, keyed by the network-assigned `session_id` which the peer cannot forge:

```rust
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() { ... }
``` [2](#0-1) 

This correctly caps any single peer at 30 messages/sec. The `forward_rate_limiter` is then checked inside each message processor. In `ConnectionRequestProcess::execute()`:

```rust
if self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
    .is_err()
``` [3](#0-2) 

`content.from` and `content.to` are parsed directly from the attacker-controlled message bytes:

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec())...;
let to   = PeerId::from_bytes(value.to().raw_data().to_vec())...;
``` [4](#0-3) 

The identical pattern appears in `ConnectionRequestDeliveredProcess::execute()`: [5](#0-4) 

And in `ConnectionSyncProcess::execute()`: [6](#0-5) 

Because the key is entirely attacker-controlled, sending 30 messages/sec each with a distinct random `from` PeerId creates 30 independent buckets per second â€” none of which ever hits the 1 req/sec cap. All 30 messages pass the `forward_rate_limiter` and are forwarded to peers.

The `HashMapStateStore` backing `forward_rate_limiter` accumulates one entry per unique key. `retain_recent()` is called **only** in `disconnected()`:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
``` [7](#0-6) 

The periodic `notify()` handler (fired every 5 minutes) cleans up `pending_delivered` and `inflight_requests` but **not** `forward_rate_limiter`:

```rust
self.pending_delivered.retain(|_, (_, t)| (now - *t) < TIMEOUT);
self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
``` [8](#0-7) 

For a connection that never disconnects, the `forward_rate_limiter` map grows at 30 entries/sec indefinitely. After 1 hour: ~108,000 entries; after 24 hours: ~2.6 million entries, consuming hundreds of megabytes.

## Impact Explanation

**High â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The `forward_rate_limiter` is intended to ensure the same `(from, to)` forwarding path is processed at most once per second. The bypass raises the effective forwarding rate from 1 req/sec to 30 req/sec per attacker session. Each forwarded `ConnectionRequest` is gossip-broadcast to `sqrt(N)` peers (see `forward_message`), who in turn forward it further, compounding the amplification across the P2P network. Additionally, the unbounded `HashMapStateStore` growth constitutes a memory exhaustion vector on any node that maintains a long-lived connection with the attacker.

## Likelihood Explanation

**High.** Any peer that establishes a single HolePunching protocol session can exploit this immediately. No special privileges, keys, or majority hashpower are required. The attacker simply sends valid `ConnectionRequest` messages with a freshly generated random `from` PeerId in each one (`PeerId::random()`). The outer 30 req/sec cap is the only real constraint, making this trivially and continuously exploitable for the duration of the connection.

## Recommendation

Key the `forward_rate_limiter` by the actual network peer identity (`session_id` / `PeerIndex`) rather than by attacker-controlled message fields, matching the pattern already used by the outer `rate_limiter`:

```rust
// Replace:
self.forward_rate_limiter.check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))

// With:
self.forward_rate_limiter.check_key(&(self.peer, self.msg_item_id))
```

If the semantic intent is to deduplicate forwarding of the same logical `(from, to)` request across *different* peers, use a separate bounded LRU cache (e.g., a fixed-capacity `LruCache<(PeerId, PeerId, u32), ()>`) instead of an unbounded `HashMapStateStore` keyed on attacker-supplied data. Additionally, add `forward_rate_limiter.retain_recent()` to the `notify()` periodic handler to bound memory growth regardless of connection lifetime.

## Proof of Concept

1. Attacker connects to a CKB node and negotiates the `HolePunching` protocol.
2. Attacker sends 30 `ConnectionRequest` messages per second (outer cap), each with a freshly generated `PeerId::random()` as `from`, a fixed valid `to` PeerId, and a valid `listen_addrs` list.
3. For each message, `forward_rate_limiter.check_key(&(content.from, content.to, item_id))` sees a brand-new key and passes immediately â€” the 1 req/sec limit is never triggered for any key.
4. The node forwards all 30 messages/sec to its peers via gossip (`sqrt(N)` peers each), who forward them further, amplifying traffic across the network.
5. After N seconds, the `forward_rate_limiter`'s internal `HashMapStateStore` contains NĂ—30 entries. After 24 hours of a single connection: ~2.6 million entries consuming ~260â€“520 MB of memory, with no cleanup until disconnect.
6. A unit test can confirm the bypass: create a `HolePunching` instance, call `forward_rate_limiter.check_key` 30 times with distinct `(PeerId::random(), fixed_to, item_id)` tuples, and assert all 30 return `Ok(())` â€” demonstrating the limiter never fires.
