### Title
Banned Peer's Orphan Transactions Persist in OrphanPool After `ban_malformed` Revokes Network Access — (`tx-pool/src/component/orphan.rs`, `tx-pool/src/process.rs`)

---

### Summary

When `ban_malformed` is called in `tx-pool/src/process.rs`, it bans the peer at the network level and removes their entries from the `VerifyQueue`, but it does **not** remove the peer's pre-submitted transactions from the `OrphanPool`. Because `OrphanPool` has no `remove_by_peer` method, a banned peer's orphan transactions persist and are later processed — with the banned peer's declared cycles — when a parent transaction arrives from any source, allowing the banned peer to inject transactions into the tx pool despite their ban.

---

### Finding Description

**Two-level state, one-level revocation.**

The `ban_malformed` function in `tx-pool/src/process.rs` performs two cleanup actions on ban:

```rust
// tx-pool/src/process.rs lines 701-702
self.network.ban_peer(peer, DEFAULT_BAN_TIME, reason);
self.verify_queue.write().await.remove_txs_by_peer(&peer);
```

The `VerifyQueue` has an explicit `remove_txs_by_peer` method (`tx-pool/src/component/verify_queue.rs` lines 159–168) that filters entries by `PeerIndex` and removes them. However, the `OrphanPool` (`tx-pool/src/component/orphan.rs`) stores entries with a `peer: PeerIndex` field but exposes **no equivalent removal method by peer**. Its only removal paths are `remove_orphan_tx` (by ID), `remove_orphan_txs` (by IDs), and `limit_size` (by expiry/count). There is no `remove_by_peer` or equivalent.

**Attack flow:**

1. Malicious peer submits a child transaction whose parent is unknown → stored in `OrphanPool` as `Entry { tx, peer, cycle, expires_at }`.
2. The same peer submits a malformed transaction → `ban_malformed` is triggered:
   - Peer is banned at the network level.
   - Peer's `VerifyQueue` entries are purged.
   - **Orphan pool entries from this peer are untouched.**
3. A parent transaction arrives from any source (another peer, local RPC). `process_orphan_tx` is called.
4. `find_orphan_by_previous` returns the banned peer's orphan entry.
5. The orphan is processed via `_process_tx(orphan.tx.clone(), Some(orphan.cycle), None)` (line 626) using the banned peer's declared cycles.
6. If the orphan passes full verification, it is accepted into the tx pool and the relayer is notified with `original_peer: Some(orphan.peer)` (line 632) — the banned peer's `PeerIndex`.

The `OrphanPool` capacity is bounded at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries and entries expire after `ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL` seconds, so the window is real but bounded.

---

### Impact Explanation

A peer banned for malicious behavior (e.g., submitting a malformed transaction) can still have pre-submitted orphan transactions accepted into the tx pool. The ban is intended to revoke all influence of the peer on the node's state, but the orphan pool represents a second, independent authorization channel that is not cleared on ban. Concretely:

- **Transaction injection bypass**: The banned peer's transactions enter the pending pool, consuming pool resources and potentially displacing legitimate transactions.
- **Declared-cycles abuse**: The orphan is processed with `orphan.cycle` (the banned peer's declared value). If the declared cycles are incorrect, the tx is rejected with `DeclaredWrongCycles` but no additional penalty is applied (peer already banned). If correct, the tx is accepted.
- **Relay notification with banned peer's identity**: `TxVerificationResult::Ok { original_peer: Some(orphan.peer) }` is sent to the relayer for a banned peer, which may cause relay logic to attempt communication with the banned peer.

---

### Likelihood Explanation

This is reachable by any unprivileged external peer with no special access. The attacker only needs to:
1. Connect to the target node.
2. Submit one or more orphan transactions (child txs whose parents are not yet in the pool).
3. Trigger a ban (e.g., by sending a malformed transaction).

The orphan pool holds up to 100 entries, so an attacker can pre-load up to 100 orphan transactions before being banned. The orphan entries persist until expiry or until the pool is evicted by newer entries. This is a straightforward, low-cost attack requiring no privileged access.

---

### Recommendation

Add a `remove_by_peer` method to `OrphanPool` and call it inside `ban_malformed` alongside the existing `VerifyQueue` cleanup:

```rust
// In OrphanPool (tx-pool/src/component/orphan.rs)
pub fn remove_by_peer(&mut self, peer: &PeerIndex) {
    let ids: Vec<_> = self.entries
        .iter()
        .filter(|(_, entry)| &entry.peer == peer)
        .map(|(id, _)| id.clone())
        .collect();
    self.remove_orphan_txs(ids.into_iter());
}
```

```rust
// In ban_malformed (tx-pool/src/process.rs)
self.network.ban_peer(peer, DEFAULT_BAN_TIME, reason);
self.verify_queue.write().await.remove_txs_by_peer(&peer);
self.orphan.write().await.remove_by_peer(&peer);  // ADD THIS
```

This mirrors the existing `remove_txs_by_peer` pattern in `VerifyQueue` and ensures that banning a peer revokes all of their pending influence on the node's transaction state.

---

### Proof of Concept

1. Connect a malicious peer to a CKB node.
2. Construct a valid child transaction `child_tx` spending an output of a not-yet-submitted `parent_tx`.
3. Relay `child_tx` to the node with correct declared cycles → it enters `OrphanPool` with `peer = attacker_peer_index`.
4. Relay a malformed transaction (e.g., wrong declared cycles) → `ban_malformed` is triggered:
   - `self.network.ban_peer(attacker_peer_index, ...)` — peer is banned.
   - `self.verify_queue.write().await.remove_txs_by_peer(&attacker_peer_index)` — verify queue cleared.
   - `OrphanPool` still contains `child_tx` with `peer = attacker_peer_index`.
5. Submit `parent_tx` from a different, legitimate peer or via local RPC.
6. `process_orphan_tx` is triggered, finds `child_tx` in the orphan pool, and calls `_process_tx(child_tx, Some(orphan.cycle), None)`.
7. `child_tx` passes full verification and is accepted into the pending pool.
8. The banned peer has successfully injected `child_tx` into the tx pool despite being banned.

**Verification**: Confirm `OrphanPool` has no `remove_by_peer` method at `tx-pool/src/component/orphan.rs` lines 74–167, and confirm `ban_malformed` at `tx-pool/src/process.rs` lines 679–703 only calls `remove_txs_by_peer` on the `VerifyQueue` and not on the `OrphanPool`.     pub(crate) async fn process_orphan_tx(&self, tx: &TransactionView) {
        let mut orphan_queue: VecDeque<TransactionView> = VecDeque::new();
        orphan_queue.push_back(tx.clone());

        while let Some(previous) = orphan_queue.pop_front() {
            let orphans = self.find_orphan_by_previous(&previous).await;
            for orphan in orphans.into_iter() {
                if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles {
                    debug!(
                        "process_orphan {} added to verify queue; find previous from {}",
                        orphan.tx.hash(),
                        tx.hash(),
                    );
                    let orphan_id = orphan.tx.proposal_short_id();
                    match self
                        .enqueue_verify_queue(
                            orphan.tx.clone(),
                            false,
                            Some((orphan.cycle, orphan.peer)),
                        )
                        .await
                    {
                        Ok(_) => {
                            self.remove_orphan_tx(&orphan_id).await;
                        }
                        Err(reject) => {
                            warn!(
                                "process_orphan {} failed to enqueue verify queue: {}; keep orphan from {}",
                                orphan.tx.hash(),
                                reject,
                                tx.hash(),
                            );
                        }
                    }
                } else if let Some((ret, _snapshot)) = self
                    ._process_tx(orphan.tx.clone(), Some(orphan.cycle), None)
                    .await
                {
                    match ret {
                        Ok(_) => {
                            self.send_result_to_relayer(TxVerificationResult::Ok {
                                original_peer: Some(orphan.peer),
                                tx_hash: orphan.tx.hash(),
                            });
                            debug!(
                                "process_orphan {} success, find previous from {}",
                                orphan.tx.hash(),
                                tx.hash()
                            );
                            self.remove_orphan_tx(&orphan.tx.proposal_short_id()).await;
                            orphan_queue.push_back(orphan.tx);
                        }
                        Err(reject) => {
                            debug!(
                                "process_orphan {} reject {}, find previous from {}",
                                orphan.tx.hash(),
                                reject,
                                tx.hash(),
                            );

                            if !is_missing_input(&reject) {
                                self.remove_orphan_tx(&orphan.tx.proposal_short_id()).await;
                                if reject.is_malformed_tx() {
                                    self.ban_malformed(orphan.peer, format!("reject {reject}"))
                                        .await;
                                }
                                if reject.is_allowed_relay() {
                                    self.send_result_to_relayer(TxVerificationResult::Reject {
                                        tx_hash: orphan.tx.hash(),
                                    });
                                }
                                if reject.should_recorded() {
                                    self.put_recent_reject(&orphan.tx.hash(), &reject).await;
                                }
                            }
                        }
                    }
                }
            }
        }
    }

    pub fn remove_orphan_tx(&mut self, id: &ProposalShortId) -> Option<Entry> {
        self.entries.remove(id).inspect(|entry| {
            debug!("remove orphan tx {}", entry.tx.hash());
            for out_point in entry.tx.input_pts_iter() {
                if let Some(ids_set) = self.by_out_point.get_mut(&out_point) {
                    ids_set.remove(id);

                    if ids_set.is_empty() {
                        self.by_out_point.remove(&out_point);
                    }
                }
            }
        })
    }

    pub fn remove_orphan_txs(&mut self, ids: impl Iterator<Item = ProposalShortId>) {
        for id in ids {
            self.remove_orphan_tx(&id);
        }
        self.shrink_to_fit();
    }
