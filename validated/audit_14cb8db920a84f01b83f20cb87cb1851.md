### Title
Deviation-Only Pulse Subscription Permanently DoS's `getPricesNoOlderThan()` During Stable Market Conditions — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract allows subscriptions configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`. When the subscribed asset's price is stable (deviation below threshold), no keeper can ever successfully call `updatePriceFeeds()`, so `priceLastUpdatedAt` never advances. Any consumer calling `getPricesNoOlderThan()` or `getEmaPricesNoOlderThan()` will permanently revert with `StalePrice`, causing a complete DoS of the subscription's price-read functionality.

---

### Finding Description

The `Scheduler` contract supports two update trigger modes: heartbeat (time-based) and deviation (price-change-based). These can be combined or used independently. The validation logic in `_validateShouldUpdatePrices()` enforces that at least one condition is met before accepting an update.

When a subscription is created with **only** deviation criteria (`updateOnHeartbeat: false`, `updateOnDeviation: true`), the contract will revert with `UpdateConditionsNotMet` on every `updatePriceFeeds()` call unless the price has moved by at least `deviationThresholdBps` since the last stored update. [1](#0-0) 

For stable assets (stablecoins, pegged assets, low-volatility tokens), the price may remain within the deviation band for extended periods — hours or days. During this time, no keeper can advance `priceLastUpdatedAt`.

The freshness-checked read functions check `priceLastUpdatedAt` against `block.timestamp`: [2](#0-1) [3](#0-2) 

Once `block.timestamp - priceLastUpdatedAt > age_seconds`, both `getPricesNoOlderThan()` and `getEmaPricesNoOlderThan()` permanently revert with `PythErrors.StalePrice`. There is no mechanism to force a refresh without price deviation occurring.

The subscription parameter validation explicitly permits this configuration: [4](#0-3) 

The `UpdateCriteria` struct confirms both flags are independently settable: [5](#0-4) 

---

### Impact Explanation

Any protocol that:
1. Creates a Pulse subscription with `updateOnDeviation: true, updateOnHeartbeat: false`, and
2. Calls `getPricesNoOlderThan()` or `getEmaPricesNoOlderThan()` with a strict `age_seconds`

will have those calls permanently revert with `StalePrice` whenever the underlying asset price is stable for longer than `age_seconds`. This is a complete DoS of the subscription's freshness-checked price-read interface. Dependent DeFi protocols (lending, derivatives, liquidation engines) that rely on these calls will be unable to execute any price-dependent operations during stable market conditions — precisely when they are most likely to be in use.

---

### Likelihood Explanation

Deviation-only subscriptions are a natural gas-optimization choice: protocols that want to avoid paying for heartbeat updates on stable assets (e.g., USDC/USD, stETH/ETH) will configure deviation-only criteria. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is only 1 hour: [6](#0-5) 

Any stable asset that doesn't move by `deviationThresholdBps` within 1 hour will trigger the DoS. For a 1% threshold on a stablecoin, this is essentially guaranteed. The likelihood is **high** for stable/pegged assets and **medium** for low-volatility assets.

---

### Recommendation

1. **Require heartbeat when deviation-only is configured**: In `_validateSubscriptionParams()`, enforce that `updateOnHeartbeat: true` must always be set alongside `updateOnDeviation: true`. A heartbeat acts as a guaranteed upper bound on staleness.

2. **Alternatively, add a maximum-staleness bypass**: Allow `updatePriceFeeds()` to succeed if `block.timestamp - priceLastUpdatedAt > PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`, regardless of deviation, to prevent permanent DoS.

3. **Document the risk**: At minimum, add NatSpec warnings to `createSubscription()` and `_validateSubscriptionParams()` that deviation-only subscriptions will cause `getPricesNoOlderThan()` to revert during stable market conditions.

---

### Proof of Concept

```
1. Alice creates a subscription:
   - priceIds: [USDC/USD]
   - updateCriteria: { updateOnHeartbeat: false, heartbeatSeconds: 0,
                       updateOnDeviation: true, deviationThresholdBps: 100 }
   - Funds: sufficient balance

2. First update succeeds (priceLastUpdatedAt = T0, price = 1.000)

3. Time passes: T0 + 2 hours. USDC/USD price = 1.001 (0.1% deviation, below 1% threshold).

4. Keeper calls updatePriceFeeds(subscriptionId, updateData):
   → _validateShouldUpdatePrices() checks deviation: 0.1% < 1% threshold
   → Reverts: UpdateConditionsNotMet

5. Consumer calls getPricesNoOlderThan(subscriptionId, priceIds, 3600):
   → distance(block.timestamp, priceLastUpdatedAt) = 7200 > 3600
   → Reverts: StalePrice  ← permanent DoS

6. No keeper can unblock this without USDC deviating by ≥1%, which may never happen.
``` [7](#0-6) [8](#0-7)

                // Get the previous price feed for this price ID using subscriptionId

                    .priceUpdates[subscriptionId][priceFeeds[i].id];

                // If there's no previous price, this is the first update
                if (previousFeed.id == bytes32(0)) {
                    return updateTimestamp;
                }

                // Calculate the deviation percentage
                int64 currentPrice = priceFeeds[i].price.price;
                int64 previousPrice = previousFeed.price.price;

                // Skip if either price is zero to avoid division by zero
                if (previousPrice == 0 || currentPrice == 0) {
                    continue;
                }

                // Calculate absolute deviation basis points (scaled by 1e4)
                uint256 numerator = SignedMath.abs(
                    currentPrice - previousPrice

                uint256 denominator = SignedMath.abs(previousPrice);
                uint256 deviationBps = Math.mulDiv(
                    numerator,
                    10_000,
                    denominator

                if (

                ) {
                    return updateTimestamp;
                }
            }

            .subscriptionStatuses[subscriptionId];

        // may be slightly ahead of this chain.
