Audit Report

## Title
Missing `MAX_RELAY_TXS_NUM_PER_BATCH` Count Check in `TransactionsProcess::execute()` Enables CPU Exhaustion via Oversized Relay Batch ‚Äî (File: `sync/src/relayer/transactions_process.rs`)

## Summary

`TransactionsProcess::execute()` begins deserializing every transaction in a received `RelayTransactions` message before applying any count guard. Both sibling handlers (`TransactionHashesProcess` and `GetTransactionsProcess`) reject messages exceeding `MAX_RELAY_TXS_NUM_PER_BATCH` (32 767) at entry. An unprivileged relay peer can send a single `RelayTransactions` message containing arbitrarily many transactions, forcing the node to spend CPU deserializing the entire payload before the per-hash filter discards unsolicited entries.

## Finding Description

**Root cause.** `TransactionsProcess::execute()` at lines 37‚Äì57 of `sync/src/relayer/transactions_process.rs` contains no count guard. The `.map()` call at line 48 fully deserializes each transaction (`tx.transaction().to_entity().into_view()`) before the `.filter()` predicate at line 49 is evaluated. Because Rust iterator adapters are lazy and fused element-by-element, deserialization cost is O(N) in the total number of transactions in the message, regardless of how many survive the filter. [1](#0-0) 

**Confirmed present in sibling handlers.**
- `TransactionHashesProcess::execute()` lines 29‚Äì35 rejects if `tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH`. [2](#0-1) 
- `GetTransactionsProcess::execute()` lines 35‚Äì39 rejects if `message_len > MAX_RELAY_TXS_NUM_PER_BATCH`. [3](#0-2) 

**Why the `unknown_tx_hashes` filter does not mitigate.** The filter at lines 49‚Äì55 ensures only requested transactions reach the tx pool, but it does not prevent deserialization of every entry. An attacker pads the message with arbitrarily many fabricated transactions; all are deserialized before any are discarded. [4](#0-3) 

**Why the rate limiter does not mitigate.** The rate limiter at lines 116‚Äì123 of `sync/src/relayer/mod.rs` caps the number of `RelayTransactions` *messages* per second per peer, keyed on `(peer, message.item_id())`. It places no bound on the number of transactions *within* each message. [5](#0-4) 

**Why outbound truncation does not mitigate.** `ask_for_txs()` at line 614 of `sync/src/relayer/mod.rs` truncates outbound `GetRelayTransactions` to `MAX_RELAY_TXS_NUM_PER_BATCH`. This limits how many transactions the node *requests*, but the attacker controls the response and can include any number of additional transactions beyond what was requested. [6](#0-5) 

`MAX_RELAY_TXS_NUM_PER_BATCH` is defined as `32767` in `util/constant/src/sync.rs` line 68 and is already imported in both sibling handler files but not in `transactions_process.rs`. [7](#0-6) 

## Impact Explanation

The attack forces the victim node to perform unbounded deserialization work on every oversized `RelayTransactions` message received. Sustained at the rate-limiter ceiling, each message carrying transactions up to the network frame size limit, this constitutes a CPU-exhaustion denial-of-service reachable from any connected relay peer with no authentication. This matches **High ‚Äî Vulnerabilities which could easily crash a CKB node** and **High ‚Äî Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

The attacker requires only a standard relay peer connection ‚Äî no keys, no stake, no mining power. The fabricated padding transactions need only be well-formed molecule-encoded structures that pass `check_data()` at line 136 of `sync/src/relayer/mod.rs`; they do not need to be semantically valid. The attack is repeatable indefinitely (reconnect after a ban, or use multiple peers). The pre-staging step (sending `RelayTransactionHashes` to trigger requests) is optional; the deserialization cost is incurred regardless of how many entries survive the filter. [8](#0-7) 

## Recommendation

Add the same count guard at the top of `TransactionsProcess::execute()` that already exists in the two sibling handlers, and import `MAX_RELAY_TXS_NUM_PER_BATCH` from `crate::relayer`:

```rust
pub fn execute(self) -> Status {
+   let tx_count = self.message.transactions().len();
+   if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
+       return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
+           "RelayTransactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})",
+       ));
+   }
    let shared_state = self.relayer.shared().state();
    ...
```

The check must be placed before any iteration or deserialization begins.

## Proof of Concept

1. Connect to a CKB node as a relay peer using the `RelayV3` protocol.
2. *(Optional)* Send one or more `RelayTransactionHashes` messages within the 32 767-hash limit to cause the node to issue `GetRelayTransactions` requests back to the attacker.
3. Construct a `RelayTransactions` molecule message containing `K √ó MAX_RELAY_TXS_NUM_PER_BATCH` entries (K ‚â• 2), where each entry is a syntactically valid but semantically arbitrary `RelayTransaction` that passes `check_data()`.
4. Send the message. Observe that `TransactionsProcess::execute()` returns `Status::ok()` and that the node's CPU usage spikes proportionally to K.
5. Repeat at the rate-limiter ceiling to sustain the load.
6. A unit test in `sync/src/relayer/tests/` can construct a `RelayTransactionsReader` with `2 √ó MAX_RELAY_TXS_NUM_PER_BATCH` entries, call `TransactionsProcess::new(...).execute()`, and assert the result is `StatusCode::ProtocolMessageIsMalformed` ‚Äî which currently fails, confirming the missing guard.
