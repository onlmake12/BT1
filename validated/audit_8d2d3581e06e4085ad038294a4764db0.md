### Title
`RecentReject::put()` Never Updates `total_keys_num`, Bypassing the Count Limit and Enabling Unbounded Disk Growth — (File: `tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put()` computes `self.total_keys_num + 1` into a local variable to check against `count_limit`, but **never writes the incremented value back to `self.total_keys_num`**. As a result, the in-memory counter stays at its startup estimate forever, the `shrink()` guard is never triggered, and the on-disk rejected-transaction database grows without bound. Any unprivileged RPC caller can exploit this by submitting a stream of transactions that are rejected and recorded, exhausting node disk space and causing a crash.

---

### Finding Description

`RecentReject` is the tx-pool component that records recently rejected transactions so that callers can query why a transaction was rejected. It is backed by a RocksDB-with-TTL database sharded across `DEFAULT_SHARDS = 5` column families. [1](#0-0) 

At construction time, `total_keys_num` is seeded from RocksDB's `estimate_num_keys_cf`: [2](#0-1) 

Every time a rejected transaction is recorded, `put()` is called: [3](#0-2) 

**The bug is on lines 62–65.** `self.total_keys_num.checked_add(1)` produces a *local* binding named `total_keys_num`. The code checks whether that local value exceeds `count_limit`, but **never assigns it back to `self.total_keys_num`**. After `put()` returns, `self.total_keys_num` is identical to what it was before the call. On a freshly started node the estimate is typically `0`; after any number of `put()` calls it remains `0`. The condition `0 + 1 > count_limit` is false for any `count_limit ≥ 1`, so `shrink()` is never invoked.

`shrink()` is the only place that updates `self.total_keys_num` from the real DB state: [4](#0-3) 

Because `shrink()` is never reached, the database grows without bound regardless of the configured `keep_rejected_tx_hashes_count` limit (default **10,000,000**). [5](#0-4) 

The existing unit test inadvertently confirms the bug: `assert!(recent_reject.total_keys_num < 100)` passes trivially because `total_keys_num` is always `0`, not because the shrink logic worked. [6](#0-5) 

---

### Impact Explanation

The `RecentReject` database is written to disk under the node's data directory. With the shrink guard permanently disabled, every rejected transaction permanently consumes disk space (a JSON-serialized rejection reason plus a 32-byte key, typically 150–400 bytes per entry). An attacker who submits rejected transactions at a sustained rate will exhaust the node's disk, causing RocksDB writes to fail, which propagates as fatal errors through the tx-pool service and crashes the node. This is a **complete denial of service** against the targeted node.

---

### Likelihood Explanation

The attack requires only the ability to call the `send_transaction` JSON-RPC endpoint, which is open to any network-reachable client on a node that exposes its RPC port (the default). No keys, stake, or privileged access are needed. Rejected transactions are recorded via `put_recent_reject` for both local RPC submissions and relayed transactions from peers: [7](#0-6) [8](#0-7) 

The reject callback registered at node startup also calls `recent_reject.put()` directly for pool-eviction events: [9](#0-8) 

An attacker can craft transactions that are cheap to generate (e.g., transactions with a fee below `min_fee_rate`, or transactions spending already-dead cells) and submit them in a tight loop. Each submission costs the attacker nothing on-chain; each one writes a permanent record to disk.

---

### Recommendation

In `put()`, write the incremented value back to `self.total_keys_num` before the limit check:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ... serialize and write to db ...
    match self.total_keys_num.checked_add(1) {
        Some(new_total) => {
            self.total_keys_num = new_total;          // ← missing assignment
            if self.total_keys_num > self.count_limit {
                self.shrink()?;
            }
        }
        None => {
            self.shrink()?;
        }
    }
    Ok(())
}
``` [10](#0-9) 

---

### Proof of Concept

1. Start a CKB node with default configuration (RPC port exposed).
2. In a loop, submit transactions via `send_transaction` that will be rejected — for example, transactions whose fee rate is below `min_fee_rate`, or transactions that double-spend a confirmed cell. These are trivially constructable with no on-chain cost.
3. Each rejected transaction whose `Reject` variant satisfies `should_recorded()` is written to the `RecentReject` RocksDB database via `put()`.
4. Because `self.total_keys_num` is never incremented, the condition `total_keys_num > count_limit` evaluates to `0 + 1 > 10_000_000` → `false` on every call, so `shrink()` is never invoked.
5. After submitting millions of rejected transactions, the node's disk fills up. RocksDB write failures propagate as errors in the tx-pool service, crashing the node.

The default `keep_rejected_tx_hashes_count` of 10,000,000 and TTL of 7 days provide no protection because the enforcement mechanism (`shrink()`) is unreachable. [3](#0-2) [11](#0-10)

    pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
        let hash_slice = hash.as_slice();
        let shard = self.get_shard(hash_slice).to_string();
        let reject: ckb_jsonrpc_types::PoolTransactionReject = reject.into();
        let json_string = serde_json::to_string(&reject)?;
        self.db.put(&shard, hash_slice, json_string)?;

        if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
            if total_keys_num > self.count_limit {
                self.shrink()?;
            }
        } else {
            // overflow occurred, try shrink
            self.shrink()?;
        }
        Ok(())
    }

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        let total_keys_num = self.estimate_total_keys_num()?;
        self.total_keys_num = total_keys_num;
        Ok(total_keys_num)
