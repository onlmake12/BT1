### Title
Unclaimed Delegation Rewards Permanently Lost on Publisher Stake Account Reassignment — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`, `governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

---

### Summary

The `set_publisher_stake_account` instruction in the Integrity Pool program changes the publisher's registered stake account in `pool_data` without requiring the old `DelegationRecord` to be advanced to the current epoch first. Because `DelegationRecord` is a PDA keyed by `[publisher, stake_account_positions]`, the old record is abandoned after reassignment. Any rewards accumulated since `last_epoch` in the old delegation record are permanently unclaimable.

---

### Finding Description

The `DelegationRecord` account tracks two fields: `last_epoch` (the last epoch for which rewards were computed) and `next_slash_event_index`. Rewards are computed lazily via `advance_delegation_record`, which walks from `last_epoch` to the current epoch and transfers tokens to the delegator's custody.

When `set_publisher_stake_account` is called, it updates `pool_data.publisher_stake_accounts[publisher_index]` to point to the new stake account. The old `DelegationRecord` PDA — keyed to `[publisher, old_stake_account_positions]` — is not advanced before the reassignment. After the reassignment, the old delegation record is orphaned: the publisher's stake account pointer in `pool_data` now points to the new account, and any future `advance_delegation_record` calls will use the new account's delegation record (a fresh PDA with `last_epoch = 0`).

The rewards accumulated between `old_delegation_record.last_epoch` and the epoch of reassignment are permanently lost because:

1. The old delegation record PDA still exists on-chain but is no longer reachable through the normal claim flow.
2. The `advance_delegation_record` instruction computes self-delegation rewards using the publisher's current stake account from `pool_data`; after reassignment, the old account is no longer the publisher's registered account.
3. There is no enforcement in `set_publisher_stake_account` requiring `delegation_record.last_epoch == current_epoch` before proceeding.

This is structurally identical to the ERC20Rebasing analog: a state transition (mode change / stake account reassignment) silently discards accumulated but unclaimed value by only considering the "new" state and ignoring the pending component.

---

### Impact Explanation

A publisher who has accumulated unclaimed self-staking rewards over multiple epochs and then calls `set_publisher_stake_account` (to reassign or remove their stake account) permanently loses all rewards accumulated since their last `advance_delegation_record` call. The lost amount equals `y * min(self_stake, pool_cap) * (current_epoch - last_epoch)` for each missed epoch. When `y > 0`, this is a direct, irreversible loss of PYTH tokens from the publisher's claimable balance.

---

### Likelihood Explanation

- `set_publisher_stake_account` is callable by the publisher themselves (error code `PublisherOrRewardAuthorityNeedsToSign` confirms the publisher is a valid signer).
- Publishers routinely reassign stake accounts during operational changes.
- The SDK's `reassignPublisherStakeAccount` and `removePublisherStakeAccount` wrappers call `setPublisherStakeAccount` directly without first calling `advanceDelegationRecord`.
- Currently `y = 0` per OP-PIP-103, so no rewards are actively accruing. However, the code path is live and the vulnerability will be triggered as soon as `y > 0` is set.

---

### Recommendation

In `set_publisher_stake_account`, require that the old delegation record is up-to-date before allowing reassignment. Specifically, add a constraint: `delegation_record.last_epoch == current_epoch`. Alternatively, call `advance_delegation_record` as a CPI within `set_publisher_stake_account` before updating `pool_data.publisher_stake_accounts`. The SDK wrappers (`reassignPublisherStakeAccount`, `removePublisherStakeAccount`) should also be updated to prepend `advanceDelegationRecord` instructions before the reassignment instruction.

---

### Proof of Concept

1. Publisher P has stake account A registered in `pool_data`.
2. Over epochs E to E+N, P earns self-staking rewards but does not call `advance_delegation_record`. The `DelegationRecord` PDA `[P, A]` has `last_epoch = E`.
3. P undelegates their self-stake from account A (required by `CurrentStakeAccountShouldBeUndelegated`).
4. P calls `set_publisher_stake_account` with `current = A`, `new = B`. `pool_data.publisher_stake_accounts[P_index]` is updated to B.
5. The `DelegationRecord` PDA `[P, A]` still has `last_epoch = E` and holds N epochs of unclaimed rewards.
6. P calls `advance_delegation_record` with stake account B. A new `DelegationRecord` PDA `[P, B]` is initialized with `last_epoch = current_epoch`. The rewards in `[P, A]` are never transferred.
7. P's N epochs of accumulated rewards are permanently lost.

  async setPublisherStakeAccount(
    publisher: PublicKey,
    stakeAccountPositions: PublicKey,
    newStakeAccountPositions: PublicKey | undefined,
  ) {
    const instruction = await this.integrityPoolProgram.methods
      .setPublisherStakeAccount()
      .accounts({
        currentStakeAccountPositionsOption: stakeAccountPositions,
        // eslint-disable-next-line unicorn/no-null
        newStakeAccountPositionsOption: newStakeAccountPositions ?? null,
        publisher,
      })
      .instruction();

    return;
  }
