Audit Report

## Title
`FeeRateCollector::statistics()` Unconditional `.expect()` on `txs_sizes` Panics for Pre-`BlockExtV1` Canonical Blocks — (`File: rpc/src/util/fee_rate.rs`)

## Summary

`FeeRateCollector::statistics()` calls `.expect()` unconditionally on `BlockExt.txs_sizes` at line 93 of `rpc/src/util/fee_rate.rs`. Blocks stored before `BlockExtV1` (pre-v0.106) are deserialized with `txs_sizes: None` via the old 5-field schema path. Any such block appearing in the last `target` (up to 101) canonical-chain blocks causes a panic in the RPC handler, triggering a local RPC API crash reachable by any unprivileged caller of `get_fee_rate_statistics` or `get_fee_rate_statics`.

## Finding Description

**Root cause — unconditional `.expect()` at line 93:**

In `rpc/src/util/fee_rate.rs`, the `statistics()` closure destructures `BlockExt` and panics if `txs_sizes` is `None`: [1](#0-0) 

**How `txs_sizes: None` reaches the canonical chain:**

`get_block_ext` in `store/src/store.rs` branches on `count_extra_fields()`: when `0` (old pre-v0.106 5-field schema), it calls `reader.into()`, which maps to the `From<packed::BlockExtReader>` impl that explicitly sets `txs_sizes: None`: [2](#0-1) [3](#0-2) 

**`FeeRateProvider::collect` performs no `txs_sizes` guard:**

`collect` iterates all block numbers in range and folds over every returned `BlockExt` without filtering on `txs_sizes` being `Some`: [4](#0-3) 

**`insert_ok_ext` accepts `txs_sizes: None`:**

`insert_ok_ext` takes `txs_sizes: Option<Vec<u64>>` and assigns it directly to `ext.txs_sizes`, so a verified block can legitimately be stored with `txs_sizes: None`: [5](#0-4) 

**No test exercises the `txs_sizes: None` path:**

All unit tests in `rpc/src/tests/fee_rate.rs` always supply `txs_sizes: Some(...)`, confirming the absence of a guard: [6](#0-5) 

**The store's own test stores `verified: Some(true)` with `txs_sizes: None`**, confirming this is a valid stored state: [7](#0-6) 

## Impact Explanation

A panic inside the `statistics()` closure propagates through `Iterator::fold` and out of the RPC handler. This causes a local RPC API crash, matching the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation

The condition is triggered on any node whose canonical chain includes blocks stored in the pre-v0.106 5-field `BlockExt` format and whose chain tip is within 101 blocks of the schema transition point. A single unprivileged RPC call to `get_fee_rate_statistics` or `get_fee_rate_statics` is sufficient to trigger the panic. The condition is also reproducible in a unit test without any node setup by appending a `BlockExt` with `txs_sizes: None` to `DummyFeeRateProvider`.

## Recommendation

In the `statistics()` closure in `rpc/src/util/fee_rate.rs`, replace the unconditional `.expect()` at line 93 with a graceful skip:

```rust
let txs_sizes = match txs_sizes {
    Some(s) => s,
    None => return fee_rates, // skip blocks without size data
};
```

Optionally, also skip blocks where `block_ext.verified != Some(true)` to avoid consuming unverified data.

## Proof of Concept

1. In `rpc/src/tests/fee_rate.rs`, add a `BlockExt` with `txs_sizes: None` to `DummyFeeRateProvider` and call `FeeRateCollector::new(&provider).statistics(None)` — this panics immediately, reproducing the bug without any node setup.
2. Alternatively, run a CKB node that was operational before v0.106, ensure the chain tip is within 101 blocks of the schema transition point, and call `get_fee_rate_statistics` via JSON-RPC with no parameters. `FeeRateProvider::collect` fetches old `BlockExt` records with `txs_sizes: None`, and `statistics()` panics at the `.expect()` call.
