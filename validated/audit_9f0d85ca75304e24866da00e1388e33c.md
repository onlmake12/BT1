### Title
Missing `from == to` Validation Enables State Corruption and Forced Outbound TCP to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request.rs`)

### Summary

`ConnectionRequestProcess::execute()` does not check whether `content.from == content.to`. When an attacker sends a `ConnectionRequest` with both fields set to the victim's own peer ID, the victim's node treats itself as the destination, calls `respond_delivered` with `from_peer_id == self_peer_id`, and inserts its own peer ID into `pending_delivered` keyed to attacker-controlled addresses. A subsequent `ConnectionSync(from=victim_peer_id, to=victim_peer_id)` then causes the victim to spawn repeated outbound TCP connection attempts to those attacker-controlled addresses for up to 30 seconds.

### Finding Description

In `execute()`, the only guard that involves `self_peer_id` is the route-loop check and the destination check: [1](#0-0) 

Line 128 checks `content.route.contains(self_peer_id)` — but with an empty route this is always `false`. Line 145 checks `self_peer_id == &content.to` — when `to == victim_peer_id` this is `true`, so `respond_delivered(content.from, ...)` is called with `content.from` also equal to `victim_peer_id`. There is **no check that `content.from != content.to`** and **no check that `content.from != self_peer_id`**.

Inside `respond_delivered`, after filtering `remote_listens` to valid TCP/IP4/IP6 addresses, the victim unconditionally inserts: [2](#0-1) 

This stores `victim_peer_id → ([attacker_addr], now)` in `pending_delivered`.

The `listen_addrs` validation in `TryFrom` only requires that any embedded peer ID in the address matches `content.from`: [3](#0-2) 

Since `from == victim_peer_id` (known to the attacker via peer discovery), the attacker crafts `/ip4/<attacker_ip>/tcp/<port>/p2p/<victim_peer_id>` — this passes validation trivially.

When the attacker then sends `ConnectionSync(from=victim_peer_id, to=victim_peer_id, route=[])`, `ConnectionSyncProcess::execute()` finds `self_peer_id == content.to`, looks up `pending_delivered[content.from]` = `pending_delivered[victim_peer_id]` = `[attacker_addr]`, and spawns: [4](#0-3) 

`try_nat_traversal` then makes repeated outbound TCP `connect()` calls to the attacker's address for up to 30 seconds with ~200ms retry intervals: [5](#0-4) 

### Impact Explanation

1. **State corruption**: `pending_delivered` is poisoned — the victim's own peer ID is inserted as a key with attacker-controlled address data. This is semantically invalid and persists for up to 5 minutes (`TIMEOUT`).
2. **Forced outbound TCP (SSRF)**: The victim node makes repeated TCP connection attempts to any attacker-specified IPv4/IPv6 TCP address. This enables port scanning of internal network resources reachable from the victim node, and can be used to probe firewalled services.
3. **Resource exhaustion**: Each `ConnectionSync` triggers a new 30-second async task with ~150 TCP connection attempts. The rate limiter keys on `(from, to, item_id)` — since `from == to == victim_peer_id`, the attacker can rotate `item_id` or wait for the rate limit window to reset and repeat.

### Likelihood Explanation

Prerequisites are minimal:
- Attacker needs the victim's peer ID — this is public and discoverable via the P2P discovery protocol.
- Attacker needs a direct P2P connection to the victim — this is the normal operating condition for any peer.

No special privileges, no leaked keys, no majority hashpower. The attack is a single crafted P2P message.

### Recommendation

Add an explicit rejection of `from == to` at the top of `execute()`, and additionally reject `content.from == self_peer_id` (a node should never be the `from` in a request it receives):

```rust
// In execute(), after parsing content:
if content.from == content.to {
    return StatusCode::InvalidFromPeerId
        .with_context("from and to must be different peers");
}
if &content.from == self_peer_id {
    return StatusCode::InvalidFromPeerId
        .with_context("from must not be the local peer");
}
```

The same guard should be applied in `ConnectionSyncProcess::execute()` for the `ConnectionSync` message.

### Proof of Concept

```
1. Attacker learns victim_peer_id via peer discovery.
2. Attacker connects to victim over P2P.
3. Attacker sends:
     ConnectionRequest {
       from: victim_peer_id,
       to:   victim_peer_id,
       listen_addrs: [/ip4/192.168.1.1/tcp/8080/p2p/<victim_peer_id>],
       route: [],
       max_hops: 6,
     }
4. Victim executes:
     self_peer_id == content.to  →  true
     respond_delivered(victim_peer_id, victim_peer_id, [/ip4/192.168.1.1/tcp/8080/...])
     pending_delivered.insert(victim_peer_id, ([/ip4/192.168.1.1/tcp/8080/...], now))
5. Attacker sends:
     ConnectionSync { from: victim_peer_id, to: victim_peer_id, route: [] }
6. Victim executes:
     self_peer_id == content.to  →  true
     pending_delivered.get(victim_peer_id)  →  [/ip4/192.168.1.1/tcp/8080/...]
     spawns try_nat_traversal → repeated TCP connect() to 192.168.1.1:8080 for 30s
```

Unit test assertion: after step 3, assert `pending_delivered` does not contain `victim_peer_id`; after step 5, assert `try_nat_traversal` is never called.
