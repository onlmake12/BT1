Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Malicious `GetBlockProposal` P2P Messages — (File: `sync/src/types/mod.rs`)

## Summary
A malicious peer can send `GetBlockProposal` relay messages containing up to 3,000 fake `ProposalShortId` values per message. For each ID absent from the local tx-pool, the node unconditionally inserts it into the `pending_get_block_proposals` DashMap with no size cap. The map is only cleared by a periodic timer drain, and entries for a disconnected peer are never removed. With multiple Sybil peers, the map grows without bound between drain cycles, causing sustained heap growth that can crash the node.

## Finding Description

`SyncState` holds an unbounded DashMap: [1](#0-0) 

Initialized with no capacity limit: [2](#0-1) 

`GetBlockProposalProcess::execute()` enforces only a per-message count limit of `max_block_proposals_limit × max_uncles_num` (1,500 × 2 = 3,000): [3](#0-2) 

For every proposal ID not found in the tx-pool, the handler calls `insert_get_block_proposals` with no total-map size guard: [4](#0-3) 

`insert_get_block_proposals` inserts every ID unconditionally: [5](#0-4) 

The only relief valve is `drain_get_block_proposals`, which clones the entire map before clearing it — an O(n) allocation — and is triggered only by the relay timer: [6](#0-5) 

Critically, the `disconnected()` handler does **not** remove a peer's entries from `pending_get_block_proposals`. It only removes inflight blocks and calls `peers().disconnected()`: [7](#0-6) 

The per-peer rate limiter caps each peer at 30 messages/second per message type: [8](#0-7) 

This is a per-peer cap. With `k` colluding peers, the insertion rate is `k × 30 × 3,000 = 90,000k` entries/second. Each `ProposalShortId` key is 10 bytes; each `HashSet<PeerIndex>` value adds allocator overhead. At 100 peers the map accumulates ~9 million entries per second before the drain fires.

## Impact Explanation

This is a **High** severity vulnerability matching the allowed impact: *"Vulnerabilities which could easily crash a CKB node."* Sustained insertion of fake proposal IDs causes unbounded heap growth. When the drain fires, the O(n) clone of a large map causes a latency spike in the relay timer loop, delaying processing of legitimate compact blocks and transactions. Under sufficient peer count and drain interval, the node's RSS grows until OOM kill or severe swap pressure stalls the node entirely.

## Likelihood Explanation

Any unprivileged peer can send `GetBlockProposal` messages; no authentication, stake, or valid transaction is required. Generating 3,000 random 10-byte IDs per message costs negligible CPU. The per-peer rate limit (30 req/s) is easily saturated by a single connection and multiplies linearly with the number of Sybil peers an attacker controls. The attack is repeatable and stateless — the attacker only needs TCP connectivity to the victim. The CHANGELOG records prior fixes for the same class of bug (`#3093`, `#3094`, `#3110`), confirming this pattern is realistic and has been exploited before.

## Recommendation

1. **Cap `pending_get_block_proposals`**: Enforce a maximum entry count (e.g., `max_block_proposals_limit × max_peers`). Reject or silently drop insertions once the cap is reached.
2. **Per-peer sub-map limit**: Track how many pending IDs each peer has contributed and evict the oldest entries from that peer when its quota is exceeded.
3. **Clean up on disconnect**: In `SyncState::disconnected()`, remove all entries in `pending_get_block_proposals` where the `HashSet<PeerIndex>` contains the disconnected peer, and remove the key entirely if the set becomes empty.
4. **Avoid full clone in drain**: Replace the clone-then-clear pattern with `std::mem::take` or an atomic swap to avoid the O(n) allocation spike.
5. **Reduce drain interval or make it event-driven**: Drain more frequently so the window of accumulation is smaller.

## Proof of Concept

```
1. Establish N TCP connections to the victim node (N = number of Sybil peers).
2. For each peer, in a tight loop (≤30 msg/s to stay under rate limit):
   a. Generate 3,000 random 10-byte ProposalShortId values.
   b. Build a GetBlockProposal message with any known tip block_hash
      and the 3,000 fake proposal IDs.
   c. Send the message over the RelayV3 protocol.
3. None of the fake IDs exist in the victim's tx-pool, so all 3,000 are inserted
   into pending_get_block_proposals on every message.
4. Between relay timer ticks, the DashMap grows by N × 30 × 3,000 entries.
5. Monitor victim RSS; it grows linearly with N and time until OOM or drain fires.
6. When drain fires, the O(n) clone causes a measurable relay-timer latency spike.
7. Verify: add a log or metric on pending_get_block_proposals.len() before and
   after each drain to confirm unbounded growth between ticks.
```
