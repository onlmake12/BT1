### Title
Non-Deterministic HashMap Iteration in `need_clean` Enables Orphan Pool Cleanup Bypass via Heterogeneous Epoch Injection — (`chain/src/utils/orphan_block_pool.rs`)

---

### Summary

`InnerPool::need_clean` samples only the first entry returned by `HashMap::iter()` to decide whether to evict **all** blocks under a given `parent_hash`. Because Rust's `HashMap` uses a randomly-seeded `RandomState`, the "first" entry is non-deterministic across process restarts. An unprivileged P2P peer can exploit this by inserting two blocks under the same `parent_hash` with epoch numbers that straddle the expiry threshold: one expired, one not. Depending on which entry `iter().next()` returns, the node either permanently retains expired orphan blocks (memory/disk DoS) or prematurely evicts non-expired ones (sync disruption).

---

### Finding Description

**The flawed function** (`chain/src/utils/orphan_block_pool.rs`, lines 113–122):

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← arbitrary HashMap entry
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
``` [1](#0-0) 

The inner map is `HashMap<packed::Byte32, LonelyBlockHash>`. `iter().next()` returns an arbitrary entry; the result governs whether `clean_expired_blocks` calls `remove_blocks_by_parent`, which removes **every** block under that parent in one shot. [2](#0-1) 

**Why heterogeneous epoch numbers are reachable:**

`epoch_number` in `LonelyBlockHash` is taken verbatim from the block header field `block.epoch().number()` with no independent validation: [3](#0-2) 

The gate before orphan insertion is `non_contextual_verify`, which calls only `BlockVerifier`: [4](#0-3) 

`BlockVerifier::verify` checks proposals limit, block bytes, cellbase structure, duplicate txs, and merkle root — **no PoW, no epoch validation**: [5](#0-4) 

PoW and epoch consistency are only checked in `HeaderVerifier`, which is **contextual** (requires the parent header) and is never invoked for orphan blocks whose parent is unknown. Therefore, a peer can send blocks with arbitrary `epoch` fields in the header and they will pass `non_contextual_verify` and be stored to disk and inserted into the orphan pool. [6](#0-5)

### Impact Explanation

**Scenario A — Indefinite retention (resource exhaustion DoS):**
Attacker sends B1 (`epoch = tip_epoch − 1`, not expired) and B2 (`epoch = 1`, expired) under the same fabricated `parent_hash`. If `iter().next()` returns B1, `need_clean` returns `false` and B2 is never cleaned. The attacker repeats this for many distinct `parent_hash` values. Expired blocks accumulate in the in-memory `InnerPool` and on-disk (via `insert_block` at line 148 of `chain_service.rs`), growing without bound until the node is restarted. [7](#0-6) 

**Scenario B — Premature eviction (sync disruption):**
If `iter().next()` returns B2 (expired), `need_clean` returns `true` and `remove_blocks_by_parent` evicts **all** blocks under that parent, including B1. `clean_expired_orphans` then calls `delete_unverified_block` and `remove_block_status` for B1, destroying its on-disk record and status entry. A legitimate block that was waiting for its parent to arrive is silently discarded. [8](#0-7) 

---

### Likelihood Explanation

The attack requires only a P2P connection and the ability to send `SendBlock` messages — no hashpower, no keys, no privileged role. The attacker constructs two minimal block structs (valid merkle root, valid cellbase, arbitrary epoch field) and sends them via the sync protocol. The `EXPIRED_EPOCH` constant is 6, so on mainnet (currently well past epoch 6) any epoch value ≤ `tip_epoch − 7` triggers the expiry condition. The non-determinism is per-process-run (Rust's `RandomState` seeds once at startup), so the attacker can probe which scenario applies by observing whether orphan counts grow or shrink, then tune the attack accordingly. [9](#0-8) 

---

### Recommendation

Replace the single-sample check with a check over **all** entries under the parent. For correct "evict only when all are expired" semantics:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            !map.is_empty()
                && map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH < tip_epoch)
        })
        .unwrap_or_default()
}
```

This ensures the eviction decision is deterministic and consistent regardless of HashMap iteration order or attacker-controlled epoch values.
