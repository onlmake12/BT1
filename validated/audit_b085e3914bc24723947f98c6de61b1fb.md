Audit Report

## Title
Off-by-One Epoch Boundary in `rfc0044_active` Gate Skips `DaoScriptSizeVerifier` for First Block of Activation Epoch — (`verification/contextual/src/contextual_block_verifier.rs`)

## Summary
`BlockTxsVerifier` gates `DaoScriptSizeVerifier` using `self.parent.epoch().number()` instead of the current block's epoch number. For the single block that opens epoch `RFC0044_ACTIVE_EPOCH`, the parent is still in epoch `A-1`, so the gate evaluates to `false` and the verifier is never called. A miner who produces that block can include a DAO withdrawal whose output lock script byte-length differs from the deposit cell's lock script byte-length, permanently committing a transaction that violates the RFC0044 invariant.

## Finding Description
`BlockTxsVerifier` (lines 323–329) holds both `self.header` (the current block) and `self.parent` (the parent block). The gate at line 445 passes the **parent's** epoch number to `rfc0044_active`:

```rust
// contextual_block_verifier.rs L444-453
}.and_then(|result| {
    if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
        DaoScriptSizeVerifier::new(...).verify()?;
    }
    Ok(result)
})
```

`rfc0044_active` (consensus.rs L1005-1012) uses a simple `>=` comparison:

```rust
pub fn rfc0044_active(&self, target: EpochNumber) -> bool {
    let rfc0044_active_epoch = ...;
    target >= rfc0044_active_epoch
}
```

Let `A` = `RFC0044_ACTIVE_EPOCH` (mainnet: 8651, testnet: 5711). For the first block of epoch `A`, `self.parent.epoch().number()` = `A-1`. The check becomes `A-1 >= A` = `false`. `DaoScriptSizeVerifier` is not invoked. Every node running this binary accepts the block identically — there is no fork, but the RFC0044 lock-script-size invariant is unenforceable for that one block.

## Impact Explanation
This is an incorrect implementation of a CKB system script verification rule (RFC0044 / DAO withdrawal). A miner can permanently commit a DAO withdrawal transaction with a mismatched lock script size to the canonical chain. Because lock script capacity is part of the DAO withdrawal accounting, a larger output lock script can allow the miner to claim more CKB capacity than the deposit cell entitles them to, constituting direct economic damage to the CKB DAO. This maps to the **High** impact class: *Incorrect implementation or behavior of CKB-VM or system scripts* (10001–15000 points), with an argument for **Critical** under *Vulnerabilities which could easily damage CKB economy* depending on the magnitude of the capacity discrepancy achievable.

## Likelihood Explanation
The window is exactly one block per chain lifetime. No special privilege beyond normal mining is required. The miner constructs the block template directly, bypassing the tx-pool's own `DaoScriptSizeVerifier` call. On mainnet the window opens at epoch 8651; on testnet at epoch 5711. Any miner who wins that specific block slot can exploit it without coordination or victim interaction.

## Recommendation
Replace `self.parent.epoch().number()` with the current block's epoch number in the gate:

```rust
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
```

`self.header` is already a field of `BlockTxsVerifier` (line 325), so no structural change is needed. This ensures `DaoScriptSizeVerifier` is applied to every block whose own epoch is ≥ `RFC0044_ACTIVE_EPOCH`, including the first block of the activation epoch.

## Proof of Concept
1. Configure a devnet with `RFC0044_ACTIVE_EPOCH = N` (small value for fast reproduction).
2. Mine until the chain tip is the last block of epoch `N-1`.
3. Construct a block at epoch `N` containing a DAO withdrawal transaction where the output lock script byte-length is larger than the deposit cell's lock script byte-length.
4. Submit the block via P2P relay or RPC.
5. **Expected (buggy) result:** node accepts the block; gate evaluates `rfc0044_active(N-1)` = `false`, verifier skipped.
6. Repeat with a block at epoch `N+1` (parent epoch = `N`): node rejects the same transaction; gate evaluates `rfc0044_active(N)` = `true`, verifier runs and returns an error.
