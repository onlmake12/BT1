### Title
Integer Overflow in Guard Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Unbounded Vec Allocation — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The guard at line 201 intended to bound all work in `GetLastStateProofProcess::execute` contains an integer overflow. When `last_n_blocks = u64::MAX` (or any value where `(last_n_blocks as usize) * 2` wraps), the guard silently evaluates to a small number in release mode and passes. The reorg path then allocates a `Vec` of `start_block_number` entries, and the `last_n_numbers` path allocates a `Vec` of `last_block_number - start_block_number` entries — together proportional to the full chain height — with no effective bound.

---

### Finding Description

**Guard check (lines 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // = 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) [2](#0-1) 

In Rust **release mode**, integer overflow wraps silently (two's complement). With `last_n_blocks = u64::MAX`:

- `last_n_blocks as usize = usize::MAX = 18446744073709551615`
- `usize::MAX * 2` wraps to `18446744073709551614` (`usize::MAX - 1`)
- Adding `difficulties.len() = 2`: `18446744073709551614 + 2 = 0` (wraps again)
- `0 > 1000` → **false** → guard is bypassed

Any `difficulties.len() >= 2` causes the double-wrap to a value below 1000, bypassing the guard entirely.

**Reorg path (lines 237–247) — unbounded allocation:**

```rust
let reorg_last_n_numbers = if start_block_number == 0 || ... {
    Vec::new()
} else {
    let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
    (min_block_number..start_block_number).collect()   // allocates start_block_number entries
};
``` [3](#0-2) 

With `last_n_blocks = u64::MAX`: `min(start_block_number, u64::MAX) = start_block_number`, so `min_block_number = 0`, and the range `(0..start_block_number)` allocates a `Vec` of `start_block_number` entries.

**`last_n_numbers` path (lines 291–296) — second unbounded allocation:**

```rust
if last_block_number - start_block_number <= last_n_blocks {
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [4](#0-3) 

`last_block_number - start_block_number <= u64::MAX` is always true, so this branch always fires, allocating a `Vec` of `last_block_number - start_block_number` entries.

**Combined:** `block_numbers` at line 350–354 chains all three, totalling `last_block_number` entries — proportional to full chain height. [5](#0-4) 

Each entry then drives a `get_ancestor` + `get_block` + MMR root DB lookup in `complete_headers`. [6](#0-5) 

---

### Impact Explanation

An unprivileged light-client peer can send a single crafted `GetLastStateProof` message with `last_n_blocks = u64::MAX`, `difficulties.len() >= 2`, a valid `last_hash` on the main chain, and a `start_hash` that is **not** an ancestor of `last_hash` (triggering the reorg branch). The server allocates a `Vec<BlockNumber>` proportional to the chain height (CKB mainnet: ~14 million blocks ≈ 112 MB for the Vec alone), then performs that many sequential DB reads. This causes OOM or severe CPU/IO exhaustion on the full node, matching the stated scope.

---

### Likelihood Explanation

The light-client protocol is a standard P2P protocol path; any peer can send `GetLastStateProof`. No authentication, PoW, or privileged role is required. The overflow is deterministic and reproducible in release builds. A single message is sufficient.

---

### Recommendation

Replace the overflow-prone guard with checked/saturating arithmetic:

```rust
let sample_count = (last_n_blocks as usize)
    .saturating_mul(2)
    .saturating_add(self.message.difficulties().len());
if sample_count > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, the reorg path at line 246 should independently cap its range:

```rust
let capped_n = min(last_n_blocks, constant::GET_LAST_STATE_PROOF_LIMIT as u64);
let min_block_number = start_block_number - min(start_block_number, capped_n);
```

### Proof of Concept

```rust
// Pseudocode unit test
let last_n_blocks: u64 = u64::MAX;
let difficulties_len: usize = 2;

// Simulate the guard in release mode (wrapping arithmetic)
let guard_value = difficulties_len + (last_n_blocks as usize).wrapping_mul(2);
// guard_value = 2 + (usize::MAX - 1) wraps to 0
assert!(guard_value <= 1000, "guard bypassed: {}", guard_value);  // passes — guard is bypassed

// Reorg path
let start_block_number: u64 = 10_000;
let min_block_number = start_block_number - start_block_number.min(last_n_blocks); // = 0
let reorg_numbers: Vec<u64> = (min_block_number..start_block_number).collect();
assert_eq!(reorg_numbers.len(), 10_000); // NOT bounded by 1000

// last_n_numbers path (last_block_number = 14_000_000 on mainnet)
let last_block_number: u64 = 14_000_000;
let last_n_numbers: Vec<u64> = (start_block_number..last_block_number).collect();
assert_eq!(last_n_numbers.len(), 13_990_000); // ~112 MB allocation
```
