### Title
Uncapped `tx.gasprice` in Keeper Fee Calculation Allows Griefing / Balance Drain - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_processFeesAndPayKeeper` function in `Scheduler.sol` computes the keeper's gas reimbursement using `tx.gasprice` directly, with no upper bound. Because `updatePriceFeeds` is callable by any unprivileged address, a malicious keeper can inflate `tx.gasprice` to drain a subscription's entire ETH balance in a single transaction.

---

### Finding Description

`updatePriceFeeds` is a permissionless external function — any address may call it for any active subscription. [1](#0-0) 

At the end of a successful update, it calls `_processFeesAndPayKeeper`, which computes the keeper reimbursement as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [2](#0-1) 

`tx.gasprice` is entirely set by the transaction sender. There is no cap, no comparison against a reference price, and no sanity check. The computed `totalKeeperFee` is then deducted from `status.balanceInWei` and transferred to `msg.sender`: [3](#0-2) 

`GAS_OVERHEAD` is a fixed constant of 30,000 gas: [4](#0-3) 

---

### Impact Explanation

A malicious keeper sets `tx.gasprice` to an astronomically large value (e.g., `subscriptionBalance / (gasUsed + GAS_OVERHEAD)`). The contract computes a `totalKeeperFee` equal to (or just below) the subscription's entire `balanceInWei`, passes the `InsufficientBalance` check, deducts the full balance, and sends it to the attacker. A single call to `updatePriceFeeds` can drain a subscription's entire ETH balance. Subscriptions can hold up to `MAX_DEPOSIT_LIMIT = 100 ether` for permanent subscriptions. [5](#0-4) 

---

### Likelihood Explanation

The attack is permissionless — no special role is required. The attacker only needs to:
1. Submit a valid price update (publicly available Pyth update data).
2. Set `tx.gasprice` to a large value.

The attacker pays `gasUsed * tx.gasprice` in real gas costs, but receives `(gasUsed + GAS_OVERHEAD) * tx.gasprice` back from the subscription. The net gain is `GAS_OVERHEAD * tx.gasprice`. If the attacker is a validator/block proposer (or uses a private mempool / MEV relay), they recover the gas cost they paid, making the attack nearly free. Even without validator access, the attacker profits by the `GAS_OVERHEAD` component, and can target high-value subscriptions to make the attack economically worthwhile.

---

### Recommendation

Cap `tx.gasprice` to a reasonable maximum before using it in the fee calculation, analogous to the fix applied in the referenced ZkStack report. For example:

```solidity
uint256 effectiveGasPrice = tx.gasprice < MAX_GAS_PRICE ? tx.gasprice : MAX_GAS_PRICE;
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
```

`MAX_GAS_PRICE` could be a governance-configurable parameter or a hardcoded conservative ceiling (e.g., 500 gwei). Alternatively, the contract could read the block base fee (`block.basefee`) and add a small tip allowance, rejecting or capping any `tx.gasprice` that exceeds `block.basefee + MAX_PRIORITY_FEE`.

---

### Proof of Concept

1. A subscription exists with `balanceInWei = B` (e.g., 10 ETH).
2. Attacker estimates that `updatePriceFeeds` consumes approximately `G` gas (e.g., 200,000 gas).
3. Attacker sets `tx.gasprice = B / (G + 30000)` (just under the drain threshold).
4. Attacker calls `updatePriceFeeds(subscriptionId, validUpdateData)` with that gas price.
5. Inside `_processFeesAndPayKeeper`:
   - `gasCost = (G + 30000) * tx.gasprice ≈ B`
   - `totalKeeperFee ≈ B`
   - `status.balanceInWei < totalKeeperFee` check passes (barely).
   - `status.balanceInWei` is zeroed out.
   - `B` wei is transferred to `msg.sender`.
6. Attacker paid `G * tx.gasprice ≈ B * G / (G + 30000)` in gas, and received `B`. Net profit ≈ `B * 30000 / (G + 30000)`. As a validator, the attacker recovers the gas cost entirely, netting the full `B`. [1](#0-0) [6](#0-5)
