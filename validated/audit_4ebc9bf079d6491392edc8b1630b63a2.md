Audit Report

## Title
Incomplete Panic Recovery in `ConsumeUnverifiedBlocks::start()` Leaves `unverified_tip` Unreset and Block Status Uncleaned — (File: `chain/src/verify.rs`)

## Summary
When `consume_unverified_blocks` panics, the `catch_unwind` recovery block in `ConsumeUnverifiedBlocks::start()` only removes the block hash from `is_pending_verify` but omits the two critical cleanup steps performed by the normal error path: resetting `unverified_tip` to the actual verified chain tip and marking the block as `BLOCK_INVALID`. This leaves `unverified_tip` potentially frozen at the panicked block's number and the block's status stuck at `BLOCK_RECEIVED`, causing a persistent sync stall that requires a node restart to recover.

## Finding Description

**Panic handler — incomplete cleanup** (`chain/src/verify.rs` L86–96): [1](#0-0) 

The handler only calls `self.processor.is_pending_verify.remove(&block_hash)`. It does not call `set_unverified_tip` or `insert_block_status(..., BLOCK_INVALID)`.

**Normal error path — complete cleanup** (`chain/src/verify.rs` L153–190): [2](#0-1) 

The `Err` branch resets `unverified_tip` to the actual verified tip and marks the block `BLOCK_INVALID` (unless it is an internal DB error).

**`unverified_tip` is advanced when a block is dispatched** (`chain/src/orphan_broker.rs` L180–196): [3](#0-2) 

**`block_fetcher` skips blocks with `BLOCK_RECEIVED` status** (`sync/src/synchronizer/block_fetcher.rs` L269–270): [4](#0-3) 

This is the decisive stall mechanism: a block whose status is `BLOCK_RECEIVED` is silently skipped with "Do not download repeatedly" — it will never be re-fetched.

**`block_fetcher` also uses `unverified_tip` as a hard gate** (`sync/src/synchronizer/block_fetcher.rs` L111–129): [5](#0-4) 

**Exploit chain:**
1. Block B at height N is dispatched to the preload channel → `unverified_tip = N`.
2. `consume_unverified_blocks` panics while processing B.
3. `catch_unwind` fires: only `is_pending_verify.remove(&B.hash())` executes. `unverified_tip` stays at N; B's status stays `BLOCK_RECEIVED`.
4. In IBD mode, `block_fetcher` uses `get_unverified_tip().number() + 1` as the fetch start, so it attempts to fetch B+1, B+2, etc. These blocks fail verification because B has no committed `BlockExt` (the `get_block_ext(&parent_hash)` call at L255–260 returns `None`). Each failure resets `unverified_tip` to the verified tip M via the normal `Err` path — but only if subsequent blocks were already queued.
5. Even after `unverified_tip` is reset, B's status remains `BLOCK_RECEIVED`. The `block_fetcher` loop at L269 skips B unconditionally: `else if status.contains(BlockStatus::BLOCK_RECEIVED) { // Do not download repeatedly }`. B can never be re-fetched from any peer.
6. All descendants of B fail verification with "parent's block ext not found" (an `InternalErrorKind::Other` error, not an internal DB error), so they are marked `BLOCK_INVALID` and deleted. The chain cannot advance past height N−1.
7. The node is permanently stalled until restart, at which point `block_status_map` is rebuilt and B's `BLOCK_RECEIVED` status is cleared.

## Impact Explanation
The node's chain tip is permanently frozen at the last verified height before the panic. Block B occupies its chain slot with an unresolvable `BLOCK_RECEIVED` status that prevents re-download, and all descendants fail without a valid parent `BlockExt`. This constitutes a **persistent sync stall** requiring a node restart to recover, matching the allowed impact: **High — Vulnerabilities which could easily crash a CKB node** (chain advancement is equivalent to a crash from a sync perspective).

## Likelihood Explanation
The developers explicitly anticipated panics by wrapping `consume_unverified_blocks` in `catch_unwind` — this is production defensive code, not a theoretical construct. Panics in Rust arise from `unwrap()`/`expect()` on `None`, integer overflow in debug builds, out-of-bounds slice access, or explicit `panic!()` calls anywhere in the verification pipeline (script execution, capacity accounting, MMR operations, fork resolution). For a deliberate attack, the attacker must submit a block that passes proof-of-work, which requires significant computational resources. However, the same stall can be triggered by any naturally occurring panic from a latent `unwrap()` bug hit by a valid mainnet block, making this a reliability concern independent of adversarial intent.

## Recommendation
In the panic handler inside `ConsumeUnverifiedBlocks::start()`, perform the same `unverified_tip` reset and block status update that the normal error path performs:

```rust
if let Err(payload) = catch_unwind(AssertUnwindSafe(|| {
    self.processor.consume_unverified_blocks(unverified_task);
})) {
    error!(
        "consume unverified block {}-{} panicked: {}",
        block_number, block_hash,
        panic_payload_to_string(payload.as_ref())
    );
    self.processor.is_pending_verify.remove(&block_hash);

    // Reset unverified_tip to the actual verified tip
    if let Some(tip) = self.processor.shared.store().get_tip_header() {
        if let Some(tip_ext) = self.processor.shared.store().get_block_ext(&tip.hash()) {
            self.processor.shared.set_unverified_tip(
                ckb_shared::HeaderIndex::new(tip.number(), tip.hash(), tip_ext.total_difficulty)
            );
        }
    }
    // Mark block invalid so it cannot re-enter the pipeline
    self.processor.shared.insert_block_status(block_hash, BlockStatus::BLOCK_INVALID);
}
```

## Proof of Concept
1. Run a CKB node in IBD mode.
2. Inject a temporary `panic!("test")` at the start of `consume_unverified_blocks` in `chain/src/verify.rs`.
3. Allow the node to receive and dispatch one block to the preload channel; observe via logs that `unverified_tip` advances to that block's number.
4. The panic fires; observe via logs that only `is_pending_verify.remove` executes in the `catch_unwind` handler.
5. Inspect `get_unverified_tip().number()` — it remains at the panicked block's number (if it was the last dispatched block).
6. Inspect the block's status in `block_status_map` — it remains `BLOCK_RECEIVED`.
7. Observe that `block_fetcher` skips the block at the `status.contains(BlockStatus::BLOCK_RECEIVED)` branch (`sync/src/synchronizer/block_fetcher.rs` L269) and never re-requests it.
8. Observe that all descendant blocks fail with "parent's block ext not found" and are marked `BLOCK_INVALID`.
9. Confirm the node's chain tip does not advance and requires a restart to recover.
