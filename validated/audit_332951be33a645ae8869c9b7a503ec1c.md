### Title
Unvalidated `from` Field Bypasses `forward_rate_limiter`, Enabling Gossip Amplification — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

An unprivileged remote peer can send up to 30 `ConnectionRequest` messages per second (the outer rate limit), each carrying a distinct attacker-chosen `from` PeerID. Because the `forward_rate_limiter` is keyed on `(from, to, item_id)` and `from` is never validated against the actual session's peer ID, every message gets a fresh bucket and passes the inner limiter. When `to` is unknown, each message triggers a gossip `filter_broadcast` to `sqrt(N)` peers, yielding **30 × sqrt(N) downstream messages per second from a single connection**.

---

### Finding Description

**Outer rate limiter** — `HolePunching::received` in `mod.rs`:

```
rate_limiter keyed by (session_id, msg.item_id())   quota = 30/s
```

This correctly caps the attacker to 30 `ConnectionRequest` messages per second per session. [1](#0-0) 

**Inner forward rate limiter** — `ConnectionRequestProcess::execute`:

```
forward_rate_limiter keyed by (content.from, content.to, msg_item_id)   quota = 1/s
```

The intent is to allow at most one forward per `(from, to)` pair per second. [2](#0-1) 

**Root cause — `from` is parsed from the message body, never cross-checked against the session:**

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec())...
```

There is no assertion that `content.from == session.peer_id`. The attacker fully controls this field. [3](#0-2) 

**Gossip broadcast path — triggered when `to` is not in the peer registry:**

```rust
// Broadcast to a number of nodes equal to the square root of the total connection count using gossip.
let mut total = self.protocol.network_state.with_peer_registry(|p| p.peers().len()).isqrt();
self.p2p_control.filter_broadcast(TargetSession::Filter(...), proto_id, new_message).await
``` [4](#0-3) 

**Attack sequence:**

1. Attacker opens one TCP session to a CKB node.
2. Sends 30 `ConnectionRequest` messages/second. Each message has:
   - `from` = a freshly generated random PeerID (unique per message)
   - `to` = a random PeerID not present in the network
   - `listen_addrs` = one valid TCP multiaddr
   - `max_hops` = any value ≤ `MAX_HOPS` (6)
3. Outer limiter: all 30 pass (budget = 30/s per session).
4. Inner limiter: each message hits a brand-new `(from_i, to, item_id)` bucket → all 30 pass.
5. `to` is unknown → `forward_message` → `filter_broadcast` to `sqrt(N)` peers.
6. Each of those `sqrt(N)` peers receives the message and repeats the same logic (unknown `to` → another `sqrt(N)` broadcast), potentially cascading further hops up to `MAX_HOPS = 6`. [5](#0-4) 

---

### Impact Explanation

- **Direct (1 hop):** 30 × sqrt(N) downstream messages/second per attacker session.
- **Cascading (up to 6 hops):** Each intermediate node that also doesn't know `to` re-broadcasts to its own sqrt(N) peers. Worst-case amplification is 30 × sqrt(N)^6 before `max_hops` reaches 0, though in practice the route deduplication (`route.contains(self_peer_id)`) limits revisits to the same node. [6](#0-5) 
- **Memory pressure:** The `forward_rate_limiter` `HashMapStateStore` accumulates one entry per unique `(from, to, item_id)` tuple. With 30 unique `from` values/second and no eviction until `retain_recent()` is called on disconnect, a long-lived session inflates the map unboundedly. [7](#0-6) 

---

### Likelihood Explanation

The attack requires only a single standard P2P connection — no special privileges, no PoW, no keys. Generating random PeerIDs is trivial. The path is entirely within production code and is locally reproducible with a single modified CKB peer.

---

### Recommendation

1. **Validate `from` against the session's actual peer ID.** Before the `forward_rate_limiter` check, assert:
   ```rust
   let session_peer_id = self.protocol.network_state
       .with_peer_registry(|r| r.get_peer(self.peer).map(|p| p.peer_id.clone()));
   if session_peer_id.as_ref() != Some(&content.from) {
       return StatusCode::InvalidFromPeerId.with_context("from does not match session peer");
   }
   ```
   This collapses all messages from one session to a single `from` value, making the outer `rate_limiter` the sole effective cap. [8](#0-7) 

2. **Re-key `forward_rate_limiter` on `(session_id, to, item_id)`** instead of `(from, to, item_id)`, since `session_id` is not attacker-controlled. [9](#0-8) 

3. **Cap `forward_rate_limiter` map growth** by calling `retain_recent()` periodically (not only on disconnect). [7](#0-6) 

---

### Proof of Concept

```rust
// Pseudocode — run against a local CKB devnet with N peers
let session = connect_to_victim_node();
loop {
    for _ in 0..30 {
        let from = PeerId::random();          // unique per message
        let to   = PeerId::random();          // unknown to the network
        let msg  = build_connection_request(from, to, valid_listen_addr, max_hops=6);
        session.send(msg);
    }
    sleep(Duration::from_secs(1));
}
// Instrument: count HolePunching messages received by all other N peers.
// Assert: total received > 30 * sqrt(N) per second.
```

Each of the 30 messages passes both rate limiters and triggers one `filter_broadcast` to `sqrt(N)` peers, confirming the O(sqrt(N)) amplification per attacker message. [10](#0-9)

            Err(status) => return status,
        };
        if content.listen_addrs.len() > ADDRS_COUNT_LIMIT || content.listen_addrs.is_empty() {
            return StatusCode::InvalidListenAddrLen
                .with_context("the listen address count is too large or empty");
        }

        if content.max_hops > MAX_HOPS {
            return StatusCode::InvalidMaxTTL.into();
        }
        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }

        let self_peer_id = self.protocol.network_state.local_peer_id();
        if content.route.contains(self_peer_id) {
            return StatusCode::Ignore.with_context("the message is passed, ignore it");
        }

        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionRequest",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionRequest");
        }

        if self_peer_id == &content.to {
            self.respond_delivered(content.from, &content.to, content.listen_addrs)
                .await
        } else if content.max_hops == 0u8 {
            StatusCode::ReachedMaxHops.into()
        } else {
            self.forward_message(self_peer_id, &content.to).await
        }
    }
