Audit Report

## Title
Unbounded Full-Pool Iteration in `get_raw_tx_pool` Stalls Tx-Pool Service Message Loop — (File: rpc/src/module/pool.rs)

## Summary
`get_raw_tx_pool(verbose=true)` dispatches an unbounded full-pool scan through the tx-pool service's sequential async message loop with no pagination, rate limit, or server-side cap. While the handler holds the read lock and iterates every entry, all other messages — including `SubmitLocalTx`, `GetBlockTemplate`, and `SubmitRemoteTx` — are queued and cannot be dequeued. An attacker with TCP access to the RPC port can issue back-to-back calls to continuously occupy the service loop, delaying block template assembly on mining nodes and degrading the node's network participation.

## Finding Description
`get_raw_tx_pool` at `rpc/src/module/pool.rs:703-718` unconditionally calls `TxPoolController::get_all_entry_info()` (verbose path) or `get_all_ids()` (non-verbose path) with no limit parameter. [1](#0-0) 

The `Message::GetAllEntryInfo` handler at `tx-pool/src/service.rs:1000-1006` acquires the async read lock via `service.tx_pool.read().await`, calls `tx_pool.get_all_entry_info()` synchronously while holding it, and only releases after the full iteration completes. [2](#0-1) 

The service is a single sequential async message loop using a bounded `mpsc` channel (`DEFAULT_CHANNEL_SIZE = 512`). No message can be dequeued while the current handler is executing. [3](#0-2) 

`TxPool::get_all_entry_info()` at `tx-pool/src/pool.rs:464-487` calls `score_sorted_iter_by_statuses` and `sorted_proposed_iter`, both of which traverse the entire multi-index map with no limit, materializing all results into a `HashMap`/`Vec` before returning. [4](#0-3) 

`score_sorted_iter_by_statuses` at `tx-pool/src/component/pool_map.rs:408-416` iterates all entries via `iter_by_score().rev().filter_map(...)` with no bound. [5](#0-4) 

The pool is bounded by `max_tx_pool_size` in bytes, not by transaction count, meaning a large number of minimum-size transactions can fill it. [6](#0-5) 

No `rpc_rate_limit`, `RateLimit`, or `throttle` pattern exists anywhere in the RPC or tx-pool call chain — confirmed by grep returning only unrelated results in `network/` and `sync/`. [7](#0-6) 

Secondary instances exist: `get_max_update_time` at `tx-pool/src/component/pool_map.rs:130-136` iterates all entries on every `tx_pool_info` RPC call, and `get_tx_detail` at `tx-pool/src/pool.rs:682-711` calls `get_ids()` (full O(N) scan) to compute `rank_in_pending`, then performs a second linear scan plus unbounded `calc_descendants`/`calc_ancestors` traversals. [8](#0-7) [9](#0-8) 

## Impact Explanation
This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**. Mining nodes commonly expose the RPC port for mining pool integrations. An attacker targeting such nodes with back-to-back `get_raw_tx_pool(verbose=true)` calls continuously occupies the tx-pool service loop, stalling `GetBlockTemplate` message processing. Delayed block template assembly directly degrades the miner's ability to produce blocks, reducing effective hash rate participation and slowing network block production. The attacker's cost is zero beyond network access. [2](#0-1) 

## Likelihood Explanation
The endpoint is a standard documented RPC with no authentication requirement. Any caller with TCP access to port 8114 can trigger it. The pool can be grown by organic network activity without attacker involvement. The attack is trivially repeatable in a tight loop at zero marginal cost per call. No existing guard, rate limit, or pagination parameter exists anywhere in the call chain. [1](#0-0) 

## Recommendation
1. Add `limit` and `offset` (or cursor) parameters to `get_raw_tx_pool` and enforce a server-side maximum entries per call.
2. Move full-pool serialization off the service's main message loop: snapshot only the IDs under the read lock, release the lock, then serialize outside it.
3. For `get_tx_detail`, avoid calling `get_ids()` (full O(N) scan) to compute `rank_in_pending`; maintain a rank index or restrict this endpoint to debug/internal builds only.
4. Replace `get_max_update_time`'s full iteration with a maintained high-watermark timestamp updated incrementally on entry insertion/removal. [8](#0-7) 

## Proof of Concept
1. Connect to a CKB node's RPC port (default 8114).
2. Submit many low-fee transactions via `send_transaction` to grow the tx-pool toward `max_tx_pool_size`.
3. In a tight loop, call `get_raw_tx_pool` with `verbose: true`.
4. Concurrently call `get_block_template` and `send_transaction` and measure response latency.
5. Observe that concurrent calls experience significantly increased latency proportional to pool size and call frequency, because the tx-pool service message loop is occupied processing each unbounded `GetAllEntryInfo` message before it can dequeue the next message.

        let tx_pool = self.shared.tx_pool_controller();

        let raw = if verbose.unwrap_or(false) {
            let info = tx_pool

                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Verbose(info.into())

            let ids = tx_pool
                .get_all_ids()
                .map_err(|err| RPCError::custom(RPCError::CKBInternalError, err.to_string()))?;
            RawTxPool::Ids(ids.into())
        };
        Ok(raw)
    }

    pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
        let pending = self
            .pool_map
            .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let proposed = self
            .pool_map
            .sorted_proposed_iter()
            .map(|entry| (entry.transaction().hash(), entry.to_info()))
            .collect();

        let conflicted = self
            .conflicts_cache
            .iter()
            .map(|(_id, tx)| tx.hash())
            .collect();
        TxPoolEntryInfo {
            pending,
            proposed,
            conflicted,
        }
    }

    pub(crate) fn get_tx_detail(&self, id: &ProposalShortId) -> Option<PoolTxDetailInfo> {
        if let Some(entry) = self.pool_map.get_by_id(id) {
            let ids = self.get_ids();
            let rank_in_pending = if entry.status == Status::Proposed {
                0
            } else {
                let tx_hash = entry.inner.transaction().hash();
                ids.pending
                    .iter()
                    .enumerate()
                    .find(|(_, hash)| &tx_hash == *hash)
                    .map(|r| r.0)
                    .unwrap_or_default()
                    + 1
            };
            let res = PoolTxDetailInfo {
                timestamp: entry.inner.timestamp,
                entry_status: entry.status.to_string(),
                pending_count: self.pool_map.pending_size(),
                rank_in_pending,
                proposed_count: ids.proposed.len(),
                descendants_count: self.pool_map.calc_descendants(id).len(),
                ancestors_count: self.pool_map.calc_ancestors(id).len(),
                score_sortkey: entry.inner.as_score_key().into(),
            };
            Some(res)
        } else {
            None
        }
    }
