### Title
Hard-Coded Alert Signing Keys Have No Revocation Mechanism — Network-Wide Malicious Alert Injection - (File: `util/app-config/src/configs/alert_signature.toml`, `util/network-alert/src/verifier.rs`)

---

### Summary

The CKB network alert system hard-codes four secp256k1 public keys directly into the binary. There is no on-chain or off-chain revocation mechanism. If any two of the four corresponding private keys are compromised, an attacker can sign and broadcast arbitrary alert messages to every node on the network indefinitely — with no way to invalidate the compromised keys short of shipping and deploying a new binary to all nodes.

---

### Finding Description

The network alert system requires 2-of-4 multi-signatures from a fixed set of public keys to broadcast an alert to all CKB nodes. These keys are embedded at compile time:

`util/app-config/src/configs/alert_signature.toml`
```toml
signatures_threshold = 2
public_keys = [
  "0x03933a9b116c5017561742c37ae69acb0dca3329a52c479c85df5bb387c8ac8715",
  "0x038da23240e5a4234601902cf3db3cdfc1b3fdb2db2a54ba6204f6ca1d6ef6129a",
  "0x02adc94b64a9809019139fe70bd26aa0d787772a1ad645a4bcb1456fb3e1105f09",
  "0x0369eca725513fc94685cd0b8ccebc7be874afea38d23ccb090566aa1c50d696b1",
]
``` [1](#0-0) 

These bytes are compiled directly into the binary via `include_bytes!`: [2](#0-1) 

The `Verifier` struct loads this fixed key set at startup and never refreshes it: [3](#0-2) 

Signature verification in `verify_signatures` calls `verify_m_of_n` against this static `HashSet<Pubkey>`: [4](#0-3) 

Once an alert passes verification, `AlertRpcImpl::send_alert` stores it locally and broadcasts it to all connected peers: [5](#0-4) 

The `AlertRelayer::received` handler then re-broadcasts to all connected peers on the P2P layer: [6](#0-5) 

There is no mechanism anywhere in the codebase to mark a specific public key as revoked, to rotate the key set at runtime, or to signal to running nodes that a key has been compromised.

---

### Impact Explanation

An attacker who obtains two of the four private keys can:

1. Broadcast arbitrary alert messages to every CKB node on the network (e.g., "Critical: upgrade immediately to `<malicious binary URL>`").
2. Use the `cancel` field to suppress any legitimate emergency alert that the remaining honest key holders attempt to send.
3. Repeat this indefinitely — the compromised keys remain valid until a patched binary is deployed to every node, which requires coordinated action across the entire operator ecosystem.

The alert message is displayed to node operators and is explicitly designed to be trusted as an authoritative signal from the Nervos Foundation. A convincing malicious alert directing operators to install a backdoored binary or send funds to an attacker-controlled address is a realistic social-engineering vector with network-wide reach.

---

### Likelihood Explanation

The four keys belong to "early CKB developers" (per `rpc/src/module/alert.rs` line 19). The threshold is only 2-of-4. Key material held by individual developers is subject to the full range of endpoint compromise risks (phishing, malware, insider threat, hardware theft). The absence of any revocation path means the exposure window after a compromise is unbounded — it lasts until a new release is cut, reviewed, merged, built, and deployed by every node operator on the network, a process that can take weeks to months.

---

### Recommendation

**Short term:**
- Add a runtime-configurable key set (e.g., read from a local config file that operators can update without recompiling) so that a compromised key can be removed by operators immediately.
- Publish a signed key-rotation procedure and document it so operators know how to respond to a compromise.

**Long term:**
- Increase the threshold (e.g., 3-of-5 or 3-of-6) to raise the bar for a successful compromise.
- Implement periodic key-rotation exercises analogous to certificate revocation drills, so the response path is tested before it is needed.
- Consider an on-chain governance mechanism for key rotation so that the key set can be updated without requiring a binary release.

---

### Proof of Concept

The following demonstrates the complete attack path using the existing code:

1. Attacker obtains two private keys corresponding to any two of the four hard-coded public keys in `alert_signature.toml`.

2. Attacker constructs a malicious alert and signs it with both keys (mirroring the pattern in `util/network-alert/src/tests/generate_alert_signature.rs`):

```rust
let raw_alert = packed::RawAlert::new_builder()
    .notice_until(far_future_ms)
    .id(99999u32)
    .priority(100u32)
    .message("CRITICAL: upgrade to https://attacker.example/ckb-malicious")
    .build();
let hash = raw_alert.calc_alert_hash();
let sig1 = compromised_key_1.sign_recoverable(&(&hash).into()).unwrap();
let sig2 = compromised_key_2.sign_recoverable(&(&hash).into()).unwrap();
```

3. Attacker submits via the `send_alert` JSON-RPC endpoint (enabled on any node with `RpcModule::Alert`):

```json
{
  "jsonrpc": "2.0",
  "method": "send_alert",
  "params": [{
    "id": "0x1869f",
    "cancel": "0x0",
    "priority": "0x64",
    "message": "CRITICAL: upgrade to https://attacker.example/ckb-malicious",
    "notice_until": "0x99999999999",
    "signatures": ["<sig1_hex>", "<sig2_hex>"]
  }],
  "id": 1
}
```

4. `AlertRpcImpl::send_alert` calls `self.verifier.verify_signatures(&alert)` — this passes because the signatures are valid against the still-trusted hard-coded keys. [7](#0-6) 

5. The alert is stored and broadcast via `network_controller.broadcast_with_handle`, then relayed peer-to-peer by every node's `AlertRelayer` until it reaches the entire network. [8](#0-7) 

6. Because the `Verifier` holds a static `HashSet<Pubkey>` loaded at startup with no refresh path, the compromised keys remain valid on every running node until operators manually redeploy a patched binary. [9](#0-8)
