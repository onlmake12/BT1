### Title
Entropy Provider Can Frontrun User Requests to Extract Excess Fee Payments — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any registered Entropy provider can call `setProviderFee` with an arbitrarily large value at any time, with no cap and no timelock. Because excess `msg.value` is never refunded to the caller, a malicious provider can sandwich a user's `requestV2` transaction — raising the fee before the request lands and restoring it afterward — to silently drain the user's ETH buffer.

---

### Finding Description

`setProviderFee` in `Entropy.sol` imposes no upper bound on `newFeeInWei` and takes effect immediately:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;
    ...
}
``` [1](#0-0) 

Inside `requestHelper`, the fee is evaluated against the **current** on-chain provider fee at execution time, not at the time the user queried it:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [2](#0-1) 

The SDK documentation explicitly warns that fees can change and that **excess value is not refunded**:

> "Note that provider fees can change over time… Further note that excess value is *not* refunded to the caller." [3](#0-2) 

Because of this, any ETH the user sends above the minimum (a common defensive practice to avoid reverts) is silently absorbed — split between the provider and the Pyth fee pool — with no refund path.

### Impact Explanation

A malicious provider can execute a sandwich attack:

1. **Setup**: Provider registers with a low fee (e.g., 0.001 ETH) to attract users.
2. **User action**: A user's contract calls `getFeeV2()` → receives 0.001 ETH + pythFee. The contract sends 0.01 ETH as a buffer to avoid reverts.
3. **Frontrun**: Provider calls `setProviderFee(0.009 ETH)` in the same block, before the user's transaction.
4. **User's tx executes**: `requiredFee = 0.009 + pythFee < 0.01 ETH` → passes the check. Provider accrues 0.009 ETH instead of 0.001 ETH.
5. **Backrun**: Provider calls `setProviderFee(0.001 ETH)` to restore the advertised rate.

The user loses 0.008 ETH more than expected with no recourse. The attack scales with the size of the user's ETH buffer and can be repeated across every request. Integrating contracts that send a fixed large `msg.value` (e.g., `address(this).balance`) are fully drainable.

---

### Likelihood Explanation

- **Entry path is permissionless**: Anyone can call `register()` to become a provider. No privileged access is required.
- **Mempool visibility**: On EVM chains without private mempools, pending `requestV2` transactions are visible and frontrunnable.
- **No cap, no delay**: `setProviderFee` is a single-transaction, unbounded state change. The attack requires only two additional transactions (raise + restore).
- **Incentive**: The provider keeps all excess ETH above the Pyth flat fee. On high-traffic chains, this is profitable at scale.
- **Documented risk acknowledged but unmitigated**: The SDK warns fees can change but provides no on-chain protection.

---

### Recommendation

1. **Enforce a maximum fee cap**: Add a governance-controlled `maxProviderFeeInWei` and revert in `setProviderFee` if `newFeeInWei > maxProviderFeeInWei`.
2. **Add a timelock on fee increases**: Require that fee increases only take effect after a delay (e.g., 24 hours), so users can observe and react to changes before they are effective.
3. **Refund excess `msg.value`**: Return `msg.value - requiredFee` to the caller in `requestHelper`. This eliminates the economic incentive for the sandwich attack entirely, since there is no excess to capture.

---

### Proof of Concept

```solidity
// Attacker is a registered Entropy provider
// User's contract sends 10x fee buffer

// 1. Provider advertises low fee: 0.001 ETH
provider.setProviderFee(0.001 ether);

// 2. User's contract queries fee and sends buffer
uint128 fee = entropy.getFeeV2(providerAddr, gasLimit); // = 0.001 ETH + pythFee
// User sends 0.01 ETH (10x buffer)

// 3. Provider frontruns: raises fee to 0.009 ETH
provider.setProviderFee(0.009 ether); // <-- lands before user's tx

// 4. User's requestV2{value: 0.01 ether}(...) executes:
//    requiredFee = 0.009 + pythFee < 0.01 ETH → passes
//    providerInfo.accruedFeesInWei += 0.009 ETH  ← provider steals 9x
//    accruedPythFeesInWei += (0.01 - 0.009 - pythFee)

// 5. Provider backruns: restores fee
provider.setProviderFee(0.001 ether);

// Result: user paid 0.009 ETH to provider instead of 0.001 ETH.
// Net theft per request: 0.008 ETH
```

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
