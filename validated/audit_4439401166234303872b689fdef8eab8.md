### Title
Missing Contract Existence Check in `excessivelySafeCall` Callback Path Silently Clears Entropy Requests Without Delivery - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `revealWithCallback` function contains two distinct callback execution paths. The newer path (triggered when `req.gasLimit10k != 0`) invokes `req.requester.excessivelySafeCall(...)` without first verifying that `req.requester` has deployed code. Per EVM specification, a low-level `call` to a codeless address returns `(true, "")`. This causes the `if (success)` branch to execute, permanently clearing the request via `clearRequest` — without the consumer ever receiving the random number.

The older path in the same function explicitly guards against this with an `extcodesize` check before calling the consumer. The new path omits this guard entirely.

---

### Finding Description

In `Entropy.sol`, `revealWithCallback` has two execution branches:

**Old path (lines 662–701)** — explicitly checks contract existence before calling:

```solidity
uint len;
assembly {
    len := extcodesize(callAddress)
}
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(
        sequenceNumber, provider, randomNumber
    );
}
``` [1](#0-0) 

**New path (lines 574–660)** — uses `excessivelySafeCall` with no existence check:

```solidity
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND,
    256,
    abi.encodeWithSelector(
        IEntropyConsumer._entropyCallback.selector,
        sequenceNumber, provider, randomNumber
    )
);
``` [2](#0-1) 

If `req.requester` has no deployed code at callback time, `excessivelySafeCall` (which is a raw assembly `call` under the hood) returns `(true, "")`. The `if (success)` branch then fires:

```solidity
if (success) {
    emit RevealedWithCallback(...);
    emit EntropyEventsV2.Revealed(...);
    clearRequest(provider, sequenceNumber);
}
``` [3](#0-2) 

`clearRequest` permanently deletes the request from storage. The random number is irrecoverably lost.

The new path is activated when `req.gasLimit10k != 0`, which is set during `requestHelper` when the provider has a non-zero `defaultGasLimit` or the user explicitly passes a `gasLimit > 0` to `requestV2`: [4](#0-3) 

---

### Impact Explanation

When `req.requester` has no code at callback time:

1. `excessivelySafeCall` returns `(true, "")` — EVM behavior for calls to codeless addresses.
2. The `success == true` branch executes.
3. `clearRequest` permanently removes the in-flight request from storage.
4. The consumer contract never receives `_entropyCallback`.
5. Any application logic gated on the random number (e.g., lottery resolution, NFT minting, game outcome) never executes.
6. The request cannot be retried — it no longer exists in the contract.

The user paid the full entropy fee for a callback that was silently discarded.

---

### Likelihood Explanation

Two realistic trigger conditions exist:

1. **Self-destruct after request**: A consumer contract calls `requestV2(provider, gasLimit)` then self-destructs before `revealWithCallback` is called. On chains that have not yet adopted EIP-6780 (Cancun), `selfdestruct` works at any time. Pyth Entropy is deployed on many chains; not all have Cancun. This is low-frequency but non-zero.

2. **EOA caller with non-zero gas limit**: An EOA calls `requestV2(provider, gasLimit)` with `gasLimit > 0`. EOAs always have `extcodesize == 0`. The new path fires, `excessivelySafeCall` returns `(true, "")`, and the request is cleared without callback. The old path explicitly skips the callback for EOAs (via the `extcodesize` guard) but still clears the request correctly. The new path clears it silently as a "success." The user loses their fee with no recourse.

The inconsistency between the two paths — one with an existence guard, one without — confirms this is an unintentional omission rather than a design choice.

---

### Recommendation

Before invoking `excessivelySafeCall` in the `gasLimit10k != 0` branch, add the same `extcodesize` guard used in the old path:

```solidity
uint len;
assembly { len := extcodesize(req.requester) }
if (len == 0) {
    // Handle as failed callback or skip silently, consistent with old path
    req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;
    return;
}
(success, ret) = req.requester.excessivelySafeCall(...);
```

Alternatively, unify both paths to share the same existence check logic to prevent future divergence.

---

### Proof of Concept

1. Deploy a consumer contract `C` that calls `requestV2(provider, 300_000)` in its constructor, paying the required fee. `req.gasLimit10k` is set to a non-zero value.
2. `C` calls `selfdestruct(owner)` (on a pre-Cancun chain), removing its code.
3. The provider calls `revealWithCallback(provider, sequenceNumber, userContribution, providerContribution)`.
4. The `req.gasLimit10k != 0` branch is entered.
5. `req.requester.excessivelySafeCall(...)` is called on the now-codeless address of `C`.
6. EVM returns `(true, "")` — no code, call succeeds vacuously.
7. `clearRequest(provider, sequenceNumber)` executes — request permanently deleted.
8. `C`'s `_entropyCallback` was never invoked. The random number is lost. The fee is non-refundable. [5](#0-4)
