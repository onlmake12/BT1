Audit Report

## Title
Unbounded `NOT IN` Clause in `get_cells` / `get_cells_capacity` Scales O(N) with tx-pool Dead Cells — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells.rs`)

## Summary
When `index_tx_pool = true` is configured, `get_cells` and `get_cells_capacity` collect every entry from `Pool::dead_cells` and emit one SQL placeholder per entry into a `NOT IN (…)` clause with no cap. Because `dead_cells` is an unbounded `HashSet<OutPoint>` that grows with every accepted pool transaction, an attacker who fills the tx-pool forces every subsequent `get_cells` / `get_cells_capacity` RPC call to construct and submit a proportionally large SQL query. At sufficient pool size this causes the RPC to return a database error (crash) due to SQL parameter-count limits.

## Finding Description
`util/indexer-sync/src/pool.rs` defines `dead_cells` as a plain `HashSet<OutPoint>` with no size cap: [1](#0-0) 

Every accepted pool transaction inserts all its inputs unconditionally: [2](#0-1) 

In `get_cells`, all dead cells are collected and expanded into a single `NOT IN` clause with one `$N` placeholder per entry, with no cap: [3](#0-2) 

Each `tx_hash` is then individually bound to the query: [4](#0-3) 

The `request_limit` guard caps only the number of *result rows* returned, not the size of the `NOT IN` clause constructed before the query executes: [5](#0-4) 

The identical pattern exists in `get_cells_capacity`: [6](#0-5) 

The default `max_tx_pool_size` is 180 MB: [7](#0-6) 

SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is 999 (older builds) or 32,766 (newer builds); PostgreSQL's limit is 65,535. At a minimum transaction size of ~200 bytes, the 180 MB pool permits ~900,000 pool transactions and a corresponding number of dead cells, far exceeding both database parameter-count limits. When the limit is exceeded, the database driver returns an error, causing the RPC call to fail with a DB error — a concrete local RPC crash.

## Impact Explanation
Each `get_cells` and `get_cells_capacity` RPC call incurs O(N) work in SQL string construction, parameter binding, and database query-plan compilation proportional to the number of dead cells. Once the number of dead cells exceeds the database's parameter-count limit (999–65,535 depending on backend and version), the RPC returns a hard database error on every call. This matches the allowed impact: **Note (0–500 points) — Any local RPC API crash**.

## Likelihood Explanation
The attack requires `index_tx_pool = true`, which is an opt-in but documented production feature. The attacker must submit enough valid transactions to the pool to exceed the database parameter limit. At SQLite's lower limit of 999 parameters, only ~1,000 pool transactions spending distinct UTXOs are needed — a trivially small number. The fee cost at the default minimum fee rate of 1,000 shannons/KB is negligible. UTXO pre-creation is a one-time on-chain cost. The attack is repeatable as long as the pool remains populated.

## Recommendation
Cap the number of dead cells included in the `NOT IN` clause to a fixed maximum (e.g., 500, safely below SQLite's minimum parameter limit). If the live pool exceeds that cap, either omit the filter (accepting a small risk of transiently returning a just-spent cell) or return an explicit error. Alternatively, store dead cells in a temporary database table and join against it rather than expanding them inline, which eliminates the parameter-count problem entirely.

## Proof of Concept
1. Configure the node with `index_tx_pool = true` in the indexer section.
2. Pre-create at least 1,000 distinct live cells on-chain.
3. Submit 1,000 transactions to the tx-pool, each spending one distinct live cell (ensuring each is valid and accepted).
4. Call `get_cells` via RPC.
5. On SQLite with the default parameter limit, the RPC returns a database error because the generated query exceeds `SQLITE_MAX_VARIABLE_NUMBER`.
6. Confirm the error is reproducible on every subsequent `get_cells` / `get_cells_capacity` call while the pool remains populated.
