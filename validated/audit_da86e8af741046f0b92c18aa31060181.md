Audit Report

## Title
`RecentReject::put()` Never Increments `self.total_keys_num`, Permanently Disabling the Shrink Guard and Enabling Unbounded Disk Growth — (File: `tx-pool/src/component/recent_reject.rs`)

## Summary
`RecentReject::put()` computes `self.total_keys_num.checked_add(1)` into a local binding but never writes it back to `self.total_keys_num`. As a result, the in-memory counter remains at its startup estimate indefinitely, the `shrink()` guard is never triggered, and the on-disk rejected-transaction RocksDB database grows without bound. An unprivileged attacker can exploit this by submitting a stream of cheaply-crafted rejected transactions via the public `send_transaction` RPC, exhausting node disk space and crashing the node.

## Finding Description
`RecentReject` is backed by a RocksDB-with-TTL database sharded across `DEFAULT_SHARDS = 5` column families. At construction, `total_keys_num` is seeded from `estimate_num_keys_cf` (lines 39–52). On every `put()` call (lines 55–71), the code does:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

`total_keys_num` here is a **local variable** bound by the `if let Some(...)` pattern. It is never assigned back to `self.total_keys_num`. After `put()` returns, `self.total_keys_num` is identical to its pre-call value. On a fresh node this is `0`; the condition `0 + 1 > 10_000_000` is always false, so `shrink()` is never invoked. `shrink()` (lines 104–113) is the **only** place that updates `self.total_keys_num` from the real DB state — but it is unreachable. The existing unit test (`assert!(recent_reject.total_keys_num < 100)` at line 39 of the test file) passes trivially because the counter is always `0`, not because shrink logic functioned correctly. The RocksDB TTL (default 7 days) provides only eventual cleanup via compaction and does not enforce the count limit.

## Impact Explanation
The `RecentReject` database is written to disk under the node's data directory. With the shrink guard permanently disabled, every rejected transaction permanently consumes disk space until TTL-driven compaction occurs. An attacker sustaining a submission rate faster than compaction reclaims space will exhaust the node's disk. RocksDB write failures propagate as fatal errors through the tx-pool service, crashing the node. This is a **complete denial of service against a targeted CKB node**, matching the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points).

## Likelihood Explanation
The attack requires only the ability to call the `send_transaction` JSON-RPC endpoint, which is open to any network-reachable client on a node that exposes its RPC port (the default configuration). No keys, stake, or privileged access are needed. Rejected transactions are recorded via `put_recent_reject` for both local RPC submissions (`process.rs` lines 428–438, 522–524) and pool-eviction events registered at startup (`shared_builder.rs` lines 576–585). An attacker can craft transactions that are cheap to generate (e.g., below `min_fee_rate`, or spending already-spent cells) and submit them in a tight loop. Each submission costs nothing on-chain; each one writes a permanent record to disk until TTL compaction reclaims it.

## Recommendation
In `put()`, assign the incremented value back to `self.total_keys_num` before the limit check:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    let hash_slice = hash.as_slice();
    let shard = self.get_shard(hash_slice).to_string();
    let reject: ckb_jsonrpc_types::PoolTransactionReject = reject.into();
    let json_string = serde_json::to_string(&reject)?;
    self.db.put(&shard, hash_slice, json_string)?;

    match self.total_keys_num.checked_add(1) {
        Some(new_total) => {
            self.total_keys_num = new_total;  // ← missing assignment
            if self.total_keys_num > self.count_limit {
                self.shrink()?;
            }
        }
        None => {
            self.shrink()?;
        }
    }
    Ok(())
}
```

The unit test assertion should also be updated to verify that `shrink()` actually fired and reduced the count, rather than relying on the counter being `0`.

## Proof of Concept
1. Start a CKB node with default configuration (RPC port exposed, `keep_rejected_tx_hashes_count = 10_000_000`).
2. In a loop, submit transactions via `send_transaction` that will be rejected and satisfy `should_recorded()` — e.g., transactions with fee rate below `min_fee_rate` or double-spending confirmed cells. These are trivially constructable with no on-chain cost.
3. Each rejected transaction is written to the `RecentReject` RocksDB database via `put()`.
4. Because `self.total_keys_num` is never incremented, the condition `total_keys_num > count_limit` evaluates to `0 + 1 > 10_000_000` → `false` on every call, so `shrink()` is never invoked.
5. Confirm the bug in the existing unit test: run `test_basic` and observe that `recent_reject.total_keys_num` is `0` after 160 `put()` calls — the assertion `total_keys_num < 100` passes only because the counter was never updated, not because shrink worked.
6. After submitting millions of rejected transactions, the node's disk fills up. RocksDB write failures propagate as errors in the tx-pool service, crashing the node.
