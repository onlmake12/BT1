### Title
Unbounded `pending_get_block_proposals` Growth via `GetBlockProposal` Message Flood Causes Memory Exhaustion DoS — (File: `sync/src/types/mod.rs`)

---

### Summary

A malicious P2P peer can flood the `SyncState::pending_get_block_proposals` `DashMap` by repeatedly sending `GetBlockProposal` relay messages containing unique `ProposalShortId` values that do not exist in the local tx pool. Because `insert_get_block_proposals` applies no size bound to the map, the structure grows without limit. When the relayer's periodic `prune_tx_proposal_request` fires, it clones the entire unbounded map and iterates every entry, causing proportional memory and CPU exhaustion that can render the node unresponsive.

---

pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,

**Expensive drain — `drain_get_block_proposals`**
(`sync/src/types/mod.rs`, lines 1586–1592)

```rust
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();   // O(n) clone
    self.pending_get_block_proposals.clear();
    ret
}
```

`prune_tx_proposal_request` calls this on a timer, then iterates every entry to build per-peer responses. Both the clone and the subsequent iteration are O(n) in the number of accumulated entries. If the map has been flooded between two timer ticks, both operations become proportionally expensive.

**Attack path**

1. Attacker connects to the target node as an inbound relay peer (no privileged access required).
2. Attacker sends a continuous stream of `GetBlockProposal` messages, each carrying up to 3 000 unique `ProposalShortId` values that are absent from the node's tx pool.
3. No per-peer message-rate limit exists for this message type; the only rejection path is exceeding the per-message count ceiling or sending duplicates *within* a single message.
4. `pending_get_block_proposals` accumulates entries at up to 3 000 per message. At a modest 100 messages/second the map grows by ~300 000 entries/second.
5. On the next `prune_tx_proposal_request` tick the node must clone and iterate the entire map, consuming CPU and memory proportional to the flood volume.

---

### Impact Explanation

- **Memory exhaustion**: each entry costs ≈10 bytes (key) + `HashSet` overhead (~48 bytes) + one `PeerIndex` (~8 bytes) ≈ 66 bytes. One million entries ≈ 66 MB; ten million ≈ 660 MB. A sustained flood can exhaust available RAM and trigger OOM.
- **CPU exhaustion**: the O(n) clone and iteration in `drain_get_block_proposals` / `prune_tx_proposal_request` stall the relayer's async task, delaying or blocking block relay and transaction propagation for all peers.
- **Node DoS**: the combination of memory pressure and CPU stall can make the node unresponsive to legitimate peers, effectively partitioning it from the network.

---

### Likelihood Explanation

Any peer that can establish an inbound relay (RelayV3) connection can trigger this. No mining power, no keys, and no special protocol state are required. The attacker only needs to generate arbitrary 10-byte `ProposalShortId` values and send them in valid `GetBlockProposal` messages. The per-message limit (3 000) is large enough to make flooding practical even from a single connection.

---

### Recommendation

1. **Add a hard size cap** to `pending_get_block_proposals`. Reject or silently drop insertions once the map exceeds a configurable threshold (e.g., `MAX_PENDING_GET_BLOCK_PROPOSALS = 10_000`).
2. **Add a per-peer quota** inside `insert_get_block_proposals`: count how many entries in the map already belong to `pi` and refuse further insertions beyond a per-peer limit.
3. **Consider banning** peers whose cumulative unknown-proposal submissions exceed the per-peer quota, analogous to the `TooManyUnknownTransactions` ban already applied in `add_ask_for_txs`.

---

### Proof of Concept

```
1. Connect to the target CKB node as a relay peer (SupportProtocols::RelayV3).

2. In a tight loop, build GetBlockProposal messages each containing
   MAX_BLOCK_PROPOSALS_LIMIT * MAX_UNCLES_NUM - 1 (≤ 2 999) unique
   ProposalShortId values that are not present in the node's tx pool
   (e.g., random 10-byte values).

3. Send each message without triggering the per-message ban:
     message_len < limit  →  no ProtocolMessageIsMalformed
     no duplicates within a message  →  no RequestDuplicate

4. Observe that pending_get_block_proposals grows by ~3 000 entries
   per message with no eviction.

5. When prune_tx_proposal_request fires (relayer timer), the node
   performs an O(n) clone of the flooded DashMap followed by an O(n)
   iteration, causing measurable CPU spike and memory growth.

6. Sustain the flood to exhaust available memory or stall the relayer
   event loop, preventing legitimate compact-block relay.

    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        let message_len = self.message.proposals().len();
        {
            // The block proposal request is separate from uncles,
            // so here the limit is only used to calculate the maximum value of uncles
            let limit = shared.consensus().max_block_proposals_limit()
                * (shared.consensus().max_uncles_num() as u64);
            if message_len as u64 > limit {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "GetBlockProposal proposals count({message_len}) > consensus max_block_proposals_limit({limit})"
                ));
            }
        }

        let proposals: HashSet<packed::ProposalShortId> =
            self.message.proposals().to_entity().into_iter().collect();

        if proposals.len() != message_len {
            return StatusCode::RequestDuplicate.with_context("Request duplicate proposal");
        }

        let fetched_transactions = {
            let tx_pool = self.relayer.shared.shared().tx_pool_controller();
            let fetch_txs = tx_pool.fetch_txs(proposals.clone()).await;
            if let Err(e) = fetch_txs {
                debug_target!(
                    crate::LOG_TARGET_RELAY,
                    "relayer tx_pool_controller send fetch_txs error: {:?}",
                    e
                );
                return Status::ok();
            }
            fetch_txs.unwrap()
        };
        // Transactions that do not exist on this node
        let not_exist_proposals: Vec<packed::ProposalShortId> = proposals
            .into_iter()
            .filter(|short_id| !fetched_transactions.contains_key(short_id))
            .collect();

        // Cache request, try process on timer
        self.relayer
            .shared()
            .state()
            .insert_get_block_proposals(self.peer, not_exist_proposals);
```
