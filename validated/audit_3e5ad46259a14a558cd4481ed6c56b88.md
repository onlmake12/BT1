Audit Report

## Title
Peer-Controlled `declared_cycles` as VM Cap Enables Permanent Transaction Relay Suppression - (File: `tx-pool/src/process.rs`)

## Summary
A remote peer can supply `declared_cycles = 1` in a `RelayTransactions` message for any valid transaction. The victim node uses this attacker-controlled value as the VM cycle cap, causing `ExceededMaximumCycles` â€” a `Script` error classified as malformed. Because `is_allowed_relay()` returns `false` for malformed rejections, `TxVerificationResult::Reject` is never sent to the relayer, so the tx hash is never removed from the known-tx filter. Since the hash was inserted before verification, honest peers are permanently blocked from relaying that transaction to the victim for the 4-hour filter TTL, renewable indefinitely.

## Finding Description

**Step 1 â€” Hash inserted before verification** (`sync/src/relayer/transactions_process.rs`):

The only pre-filter bans `declared_cycles > max_block_cycles`. A value of `1` passes trivially. Immediately after, the hash is unconditionally marked as known:

```rust
shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));
``` [1](#0-0) 

**Step 2 â€” Peer-supplied value used verbatim as VM cap** (`tx-pool/src/process.rs`):

```rust
let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
``` [2](#0-1) 

With `declared_cycles = 1`, `verify_rtx` runs the VM with `max_cycles = 1`. Any script requiring more than 1 cycle (e.g., secp256k1 at 537 cycles) halts with `ExceededMaximumCycles`.

**Step 3 â€” `DeclaredWrongCycles` is NOT produced on VM failure** (`tx-pool/src/process.rs`):

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// DeclaredWrongCycles check at L736-748 is only reached when verify_rtx returns Ok
``` [3](#0-2) 

When `verify_rtx` returns `Err(ExceededMaximumCycles)`, the macro short-circuits and returns `Reject::Verification(Script error)` â€” the `DeclaredWrongCycles` branch is never evaluated.

**Step 4 â€” Script error is classified as malformed** (`util/types/src/core/tx_pool.rs`):

```rust
ErrorKind::Script => !format!("{}", error).contains(ARGV_TOO_LONG_TEXT),
``` [4](#0-3) 

The `ExceededMaximumCycles` display string is `"ExceededMaximumCycles: expect cycles <= {0}"` â€” it does not contain `ARGV_TOO_LONG_TEXT`, so `is_malformed_from_verification` returns `true`. [5](#0-4) 

**Step 5 â€” `is_allowed_relay()` returns `false`** (`util/types/src/core/tx_pool.rs`):

```rust
pub fn is_allowed_relay(&self) -> bool {
    matches!(self, Reject::DeclaredWrongCycles(..))
        || (!matches!(self, Reject::LowFeeRate(..)) && !self.is_malformed_tx())
}
``` [6](#0-5) 

The reject is `Reject::Verification(Script)` â€” malformed, not `DeclaredWrongCycles` â€” so `is_allowed_relay()` is `false`.

**Step 6 â€” `TxVerificationResult::Reject` is never sent** (`tx-pool/src/process.rs`):

```rust
if reject.is_malformed_tx() {
    self.ban_malformed(peer, format!("reject {reject}")).await;
}
if reject.is_allowed_relay() {
    self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash: tx_hash.clone() });
}
``` [7](#0-6) 

The peer is banned, but `TxVerificationResult::Reject` is never enqueued.

**Step 7 â€” Without `TxVerificationResult::Reject`, the hash is never removed** (`sync/src/relayer/mod.rs`):

```rust
TxVerificationResult::Reject { tx_hash } => {
    self.shared.state().remove_from_known_txs(&tx_hash);
}
``` [8](#0-7) 

This is the only removal path. It is never reached. The hash remains in the filter for the full TTL.

**Filter TTL**: `FILTER_TTL = 4 * 60 * 60` seconds (4 hours). [9](#0-8) 

**Contrast with over-declaration**: When `declared_cycles > actual`, `verify_rtx` returns `Ok`, the `DeclaredWrongCycles` branch fires, `is_allowed_relay()` returns `true`, `TxVerificationResult::Reject` is sent, and the hash IS removed â€” honest peers can re-relay. The under-declaration path is the unhandled case. [10](#0-9) 

## Impact Explanation

An unprivileged remote peer can suppress any specific valid transaction from a victim node's mempool for the 4-hour filter TTL, renewable indefinitely via IP rotation after the 3-day ban. Applied at scale across multiple nodes â€” particularly mining nodes â€” this constitutes targeted transaction censorship. Targeted suppression of high-value or time-sensitive transactions across multiple nodes matches **High â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **Critical â€” Vulnerabilities which could easily damage CKB economy** when applied systematically to block specific transactions from confirmation.

## Likelihood Explanation

The attack requires only a standard P2P connection and knowledge of the target transaction hash (observable from any node's mempool broadcast). No key material, hashpower, or privileged access is needed. The attacker sends one `RelayTransactionHashes` message, waits for `GetRelayTransactions`, then sends `RelayTransactions` with `cycles = 1`. The 3-day IP ban is bypassable via IP rotation or Tor. The attack is repeatable every 4 hours per victim node per transaction, making indefinite suppression feasible at low cost.

## Recommendation

1. **Do not use `declared_cycles` as the VM `max_cycles` cap.** Always verify with `consensus.max_block_cycles()` as the hard cap. Use `declared_cycles` only as a hint for fee/priority calculation after successful verification.
2. **Treat `ExceededMaximumCycles` caused by a peer-supplied cap as `DeclaredWrongCycles`**: when `verified_ret` fails with a cycles-exceeded error and `declared_cycles` is below the consensus cap, return `Reject::DeclaredWrongCycles` so `is_allowed_relay()` returns `true` and the hash is cleared from the filter.
3. **Alternatively, mark the tx hash as known only after successful verification**, or unconditionally remove it from the filter on any non-`Duplicated` failure path.

## Proof of Concept

1. Observe a valid pending transaction `T` with actual script cost of 537 cycles (standard secp256k1 lock) on the network.
2. Connect to victim node as a peer via `RelayV3`.
3. Send `RelayTransactionHashes` containing `T`'s hash; wait for `GetRelayTransactions` response.
4. Send `RelayTransactions` with `T` and `cycles = 1`.
5. **Victim node**: marks `T`'s hash as known (`transactions_process.rs:76`), runs VM with `max_cycles = 1`, gets `ExceededMaximumCycles` â†’ `Reject::Verification(Script)`, bans attacker peer, but does **not** send `TxVerificationResult::Reject` â†’ hash remains in filter.
6. Honest peer B now tries to relay `T` with correct `cycles = 537`. The victim's filter check `!tx_filter.contains(&T.hash())` returns `false` â†’ `T` is silently dropped at `transactions_process.rs:49-55`.
7. `T` never enters the victim's tx-pool.
8. Repeat from step 2 using a new IP before the 4-hour TTL expires to maintain suppression indefinitely.

The existing integration test `DeclaredWrongCyclesChunk` confirms the over-declaration path is handled correctly, but the under-declaration path (cycles below actual) produces a different error code that bypasses the re-relay mechanism entirely. [11](#0-10)
