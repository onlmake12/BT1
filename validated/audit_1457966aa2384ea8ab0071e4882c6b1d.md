### Title
No Cancel/Refund Mechanism for Pending Requests Causes User Fees to Be Permanently Locked - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo.sol` contract stores user-paid fees inside pending request structs (`req.fee`). There is no `cancelRequest` or refund function. If a provider never calls `executeCallback`, the fee portion stored in the request is permanently locked in the contract with no recovery path for the user.

---

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the total `msg.value` is split:

- `_state.pythFeeInWei` is credited to `_state.accruedFeesInWei` (withdrawable by admin via `withdrawFees`)
- The remainder (`msg.value - _state.pythFeeInWei`) is stored in `req.fee` inside the pending request struct [1](#0-0) 

The only code path that ever releases `req.fee` is `executeCallback`, which credits it to the provider's `accruedFeesInWei`: [2](#0-1) 

There is no `cancelRequest`, no timeout/expiry enforcement, and no refund path. The admin's `withdrawFees` only covers `_state.accruedFeesInWei` (the Pyth protocol portion), not the per-request `req.fee` values: [3](#0-2) 

The contract's own developer comments acknowledge this risk explicitly: [4](#0-3) 

The full interface exposes no cancel or refund entry point: [5](#0-4) 

---

### Impact Explanation

Any fee paid by a user into a pending request that is never fulfilled by the provider is permanently locked in the `Echo` contract. Neither the user, the admin, nor the fee manager has any mechanism to recover it. Over time, as providers go offline or selectively ignore requests, the contract accumulates irrecoverable ETH. This is a direct loss-of-funds for users.

---

### Likelihood Explanation

Any registered provider can stop fulfilling requests at any time — there is no on-chain penalty or forced fulfillment. A provider going offline (infrastructure failure), rotating keys without re-registering, or acting maliciously by ignoring specific requests all trigger this condition. The 60-second `publishTime` cap prevents far-future requests but does not prevent a provider from simply never calling `executeCallback` on a valid past-time request. The `exclusivityPeriodSeconds` window allows other providers to step in after expiry, but there is no guarantee any other provider will fulfill an arbitrary request either. [6](#0-5) 

---

### Recommendation

1. Add a `cancelRequest(uint64 sequenceNumber)` function callable by the original requester after a configurable timeout (e.g., after `publishTime + exclusivityPeriodSeconds + buffer`), which calls `clearRequest` and refunds `req.fee` to `req.requester`.
2. Alternatively, track all pending request fees in a separate aggregate and ensure the admin's `withdrawFees` cannot drain ETH that belongs to unfulfilled requests.
3. Consider a penalty mechanism against the assigned provider for unfulfilled requests, as noted in the existing TODO comment.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback{value: totalFee}(provider, publishTime, priceIds, gasLimit)`.
2. `req.fee = totalFee - pythFeeInWei` is stored in the request struct; `_state.accruedFeesInWei += pythFeeInWei`.
3. Provider never calls `executeCallback` (offline or malicious).
4. User has no function to call to recover `req.fee`.
5. Admin calls `withdrawFees(accruedFeesInWei)` — only the Pyth protocol portion is withdrawn; `req.fee` remains locked in the contract forever.
6. `getFirstActiveRequests` confirms the request is still active indefinitely with no expiry. [7](#0-6) [8](#0-7)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }

        require(msg.sender == _state.admin, "Only admin can withdraw fees");

    function getFirstActiveRequests(
        uint256 count
    )
        external
        view
        override
        returns (Request[] memory requests, uint256 actualCount)
    {
        requests = new Request[](count);
        actualCount = 0;

        // Start from the first unfulfilled sequence and work forwards
        uint64 currentSeq = _state.firstUnfulfilledSeq;

        // Continue until we find enough active requests or reach current sequence
        while (
            actualCount < count && currentSeq < _state.currentSequenceNumber
        ) {
            Request memory req = findRequest(currentSeq);
            if (isActive(req)) {
                requests[actualCount] = req;
                actualCount++;
            }
            currentSeq++;
        }

        // If we found fewer requests than asked for, resize the array
        if (actualCount < count) {
            assembly {
                mstore(requests, actualCount)
            }
        }
    }
