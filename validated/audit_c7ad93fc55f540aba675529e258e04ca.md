### Title
`check_tx_fee` Uses Raw Transaction Byte Size Instead of Weight for Minimum Fee Enforcement, Allowing High-Cycle Transactions to Bypass `min_fee_rate` — (File: tx-pool/src/util.rs)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the raw serialized byte size of a transaction, not the proper **weight** (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). For high-cycle, small-size transactions, the weight can be orders of magnitude larger than the byte size. This allows any unprivileged RPC caller to submit transactions whose effective fee rate (fee / weight) is far below the configured `min_fee_rate`, bypassing the pool's spam-prevention threshold.

---

### Finding Description

The minimum fee gate in `check_tx_fee` is:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The correct weight function is defined as:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4_f64` (≈ `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES`). [3](#0-2) 

The maximum allowed cycles per transaction is `max_tx_verify_cycles = 70,000,000` (default from `TWO_IN_TWO_OUT_CYCLES × 20`). [4](#0-3) 

For a transaction with 70 M cycles and a minimal byte size of ~100 bytes:

- **Weight** = max(100, 70,000,000 × 0.000_170_571_4) = max(100, **11,940**) = **11,940 bytes**
- **Fee check** (current): `min_fee_rate.fee(100)` → at `min_fee_rate = 1000 shannons/KW`, requires only **100 shannons**
- **Effective fee rate**: 100 × 1000 / 11,940 ≈ **8.4 shannons/KW** — 99% below the configured minimum

The pool's sorting and eviction logic correctly uses `get_transaction_weight` with both size and cycles: [5](#0-4) 

But the admission gate (`check_tx_fee`) does not, so the transaction is already inside the pool before the proper weight is ever applied.

---

### Impact Explanation

Any RPC caller can submit a transaction with a complex lock/type script (high cycles, small serialized body) that:

1. Passes the size-only fee gate and enters the tx-pool.
2. Has an effective fee rate (by weight) that is a small fraction of `min_fee_rate` — potentially 100× below the threshold.
3. Consumes pool capacity (`max_tx_pool_size`) and competes with legitimate transactions.
4. If included by a miner, yields far less fee per unit of block resource consumed than the node operator intended to enforce.

The `min_fee_rate` is the primary economic spam-prevention mechanism for the tx-pool. Bypassing it for the cycle-dominant class of transactions undermines that protection entirely for those transactions.

---

### Likelihood Explanation

- **Entry path**: Any unprivileged user calling `send_transaction` via the JSON-RPC API.
- **Craft requirement**: A transaction referencing an existing on-chain script cell (e.g., a complex lock script) with a small witness/output body. No privileged access, no key leakage, no majority hashpower required.
- **Detectability**: The bypass is silent — the node logs only the size-based fee check result, not the weight-based effective fee rate.

---

### Recommendation

Replace the size-only fee check with a weight-based check. Because cycles are not known until after script execution, the proper fix is to perform the weight-based minimum fee check **after** the cycle count is determined (i.e., post-verification), and reject the transaction at that point if the effective fee rate is below `min_fee_rate`:

```rust
// After cycles are known:
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

Alternatively, enforce a stricter pre-check using the worst-case weight (`max_tx_verify_cycles`) so that any transaction that could possibly be cycle-dominant is required to pay the correspondingly higher fee upfront.

---

### Proof of Concept

1. Deploy any script cell on-chain that consumes close to `max_tx_verify_cycles` (70 M cycles) when executed — e.g., a lock script that performs intensive hashing.
2. Construct a transaction spending a cell locked by that script, with a minimal witness and a single small output. Serialized size ≈ 150 bytes.
3. Set the transaction fee to `min_fee_rate × 150 / 1000 = 150 shannons` (at default `min_fee_rate = 1000`).
4. Submit via `send_transaction` RPC.
5. Observe: the transaction is accepted into the pool. Its actual weight ≈ 11,940 bytes; its effective fee rate ≈ 12.6 shannons/KW — roughly **79× below** the configured `min_fee_rate` of 1000 shannons/KW.
6. Repeat to fill the pool with such transactions at a fraction of the intended cost.
