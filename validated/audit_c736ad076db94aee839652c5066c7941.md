### Title
Double Wormhole Fee Charge in `post_message` Helper — (`target_chains/solana/crates/wormhole-solana/src/message.rs`)

### Summary
The `post_message` helper in `message.rs` explicitly transfers `config.params.fee` to the fee_collector before invoking the Wormhole CPI. The bundled core bridge's `PostMessage` handler also calls `handle_message_fee`, which performs a second identical transfer. Every call to `post_message` debits the payer `2 × config.params.fee`.

### Finding Description
`message.rs::post_message` (lines 117–123) issues a `system_instruction::transfer` of `config.params.fee` lamports from `payer` to `fee_collector`. It then CPIs into the Wormhole core bridge `PostMessage` instruction (lines 126–152). The core bridge's handler (`post_message/mod.rs` line 118) unconditionally calls `handle_message_fee`, which performs a second `system_program::transfer` of `config.fee_lamports` from the same payer to the same fee_collector (lines 353–362). The result is `payer.lamports -= 2 × fee` per emission.

The root cause is a stale assumption: `message.rs` was written for the legacy Wormhole bridge, which only *checked* that the fee had been pre-paid. The updated core bridge in this repo changed `handle_message_fee` to *perform* the transfer itself (as documented in its own comment), but `message.rs` was never updated to remove its pre-payment.

### Impact Explanation
- Every price-update relayer or Pyth instruction that calls `post_message` is overcharged by `config.params.fee` lamports per call.
- A payer holding exactly `rent_exempt_minimum + 1 × fee` lamports will have their transaction fail (insufficient lamports for the second transfer), rendering the program unable to post Wormhole messages until the account is topped up.
- Accumulated over many updates, this drains relayer/payer accounts at 2× the intended rate.

### Likelihood Explanation
This triggers on every single invocation of `post_message` when `config.params.fee > 0`. No special attacker setup is required — any relayer calling a Pyth price-update instruction that internally uses this helper is affected unconditionally.

### Recommendation
Remove the explicit pre-payment block in `message.rs` (lines 116–123), since the core bridge now handles the fee transfer internally:

```rust
// DELETE this block — the core bridge's handle_message_fee now does this:
// if config.params.fee > 0 {
//     invoke_signed(
//         &system_instruction::transfer(&payer, &fee_collector, config.params.fee),
//         accounts,
//         &[],
//     )?;
// }
```

Alternatively, if the intent is to keep `message.rs` compatible with the *old* bridge (balance-check model), the core bridge's `handle_message_fee` must be reverted to a balance-check rather than a transfer. Pick one approach and apply it consistently.

### Proof of Concept
1. Deploy the bundled core bridge with `fee_lamports = F > 0`.
2. Fund a payer with `rent_exempt_min + 2F` lamports.
3. Call any Pyth instruction that invokes `post_message`.
4. Observe `fee_collector.lamports` increased by `2F` and `payer.lamports` decreased by `2F`.
5. Fund a second payer with exactly `rent_exempt_min + F` lamports and repeat — the transaction fails with "insufficient lamports," confirming the program cannot operate. [1](#0-0) [5](#0-4)

fn handle_message_fee<'info>(
    config: &Account<'info, LegacyAnchorized<Config>>,
    payer: &AccountInfo<'info>,
    fee_collector: &Option<UncheckedAccount<'info>>,
    system_program: &Program<'info, System>,
) -> Result<()> {
    if config.fee_lamports > 0 {
        let fee_collector = fee_collector
            .as_ref()
            .ok_or(error!(ErrorCode::AccountNotEnoughKeys).with_account_name("fee_collector"))?;

        // In the old implementation, integrators were expected to pay the fee outside of this
        // instruction and this instruction handler had to check that the lamports on the fee
        // collector account were at least as much as the last lamports in the config plus the fee
        // amount.
        //
        // Now we just transfer the lamports from the payer to the fee collector for the exact fee
        // amount.
        system_program::transfer(
            CpiContext::new(
                system_program.key(),
                system_program::Transfer {
                    from: payer.to_account_info(),
                    to: fee_collector.to_account_info(),
                },
            ),
            config.fee_lamports,
        )
    } else {
        // Nothing to do.
        Ok(())
    }
}
