### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any Pending Transaction Without Ownership Verification — (`rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` JSON-RPC method in the `Pool` module removes any transaction from the tx pool given only its hash. The RPC server has no authentication layer and performs no check that the caller is the submitter of the transaction. Any process that can reach the RPC endpoint — which is enabled in the default module list — can silently evict any pending transaction submitted by any other user, enabling targeted transaction censorship and griefing.

---

### Finding Description

**Root cause — no submitter identity stored, no caller check at removal:**

`remove_transaction` is declared in the `PoolRpc` trait and implemented in `PoolRpcImpl`:

```rust
// rpc/src/module/pool.rs:662-669
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

The only input is `tx_hash`. No caller identity, no signature, no token. The call flows directly to `TxPoolController::remove_local_tx` → `TxPoolService::remove_tx`:

```rust
// tx-pool/src/process.rs:440-455
pub(crate) async fn remove_tx(&self, tx_hash: Byte32) -> bool {
    let id = ProposalShortId::from_tx_hash(&tx_hash);
    { let mut queue = self.verify_queue.write().await;
      if queue.remove_tx(&id).is_some() { return true; } }
    { let mut orphan = self.orphan.write().await;
      if orphan.remove_orphan_tx(&id).is_some() { return true; } }
    let mut tx_pool = self.tx_pool.write().await;
    tx_pool.remove_tx(&id)
}
```

This removes the transaction from the verify queue, orphan pool, pending pool, and proposed pool — with no identity check at any layer.

**No authentication on the RPC server:**

The HTTP server is built with `CorsLayer::permissive()` and dispatches every request via `io.handle_call(call, T::default())` — there is no authentication middleware, no token validation, and no per-method access control beyond module-level enable/disable:

```rust
// rpc/src/server.rs:119-129
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())
    ...
```

**`Pool` module is enabled by default:**

```toml
# resource/ckb.toml:190
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

`remove_transaction` is part of the `Pool` module and is therefore active on every default node deployment.

**Analog to the reported vulnerability:**

The Frakt `getLotTicketByStaking` bug allowed any caller to reference a stake account they did not own. Here, `remove_transaction` allows any RPC caller to reference and destroy a transaction they did not submit. The missing check is structurally identical: the operation mutates state tied to a resource (a pending transaction) without verifying the caller's relationship to that resource.

---

### Impact Explanation

An attacker who can reach the RPC endpoint can:

1. **Enumerate pending transactions** via `get_raw_tx_pool` (also in the `Pool` module, no auth).
2. **Continuously call `remove_transaction`** on any or all of them, evicting them before they are proposed or committed.
3. **Selectively censor specific transactions** — e.g., a competing DeFi transaction, a DAO vote, or a time-sensitive withdrawal — by polling the pool and removing the target transaction every time it is resubmitted.
4. **Remove dependent transaction chains**: the implementation removes the target and all descendants, so a single call can evict an entire CPFP chain.

The result is permanent denial of confirmation for targeted transactions as long as the attacker can reach the endpoint. No funds are directly stolen, but time-locked transactions (e.g., DAO withdrawals with `since` constraints) can be made to expire, and fee-bumping strategies can be neutralized.

---

### Likelihood Explanation

**Default configuration — localhost only:** `listen_address = "127.0.0.1:8114"`. The attacker must be co-located on the same host. This is a realistic scenario for:

- Shared hosting or VPS environments where multiple tenants run processes.
- Node operators who also run wallets, dApps, or indexers on the same machine.
- Server-side web applications that proxy user requests to the local node RPC.
- Any SSRF vulnerability in a co-located web service that can make outbound HTTP POST requests to `127.0.0.1:8114`.

**Misconfigured deployments:** The documentation warns against exposing the RPC but does not enforce it. Operators who bind to `0.0.0.0` (common in Docker/cloud deployments) expose the endpoint to the network, making the attacker profile entirely remote with no privileges.

**No rate limiting, no authentication, no per-caller tracking** — the attack is trivially scriptable and requires only knowledge of the target transaction hash, which is public once the transaction is broadcast.

---

### Recommendation

1. **Track submitter identity at submission time.** When a transaction enters the pool via `submit_local_tx`, record the source (e.g., a session token, Unix socket credential, or a caller-supplied secret). Require the same credential to be presented for `remove_transaction`.

2. **Alternatively, restrict `remove_transaction` to a separate privileged RPC module** (e.g., a new `Admin` module) that is disabled by default and requires explicit opt-in, similar to the `Debug` module.

3. **Add an authentication layer to the RPC server** (e.g., HTTP Basic Auth via a configurable token) so that only the node operator can invoke state-mutating pool operations.

4. **At minimum, document `remove_transaction` as a privileged operator-only method** and move it out of the default `Pool` module so it is not silently available on all nodes.

---

### Proof of Concept

**Preconditions:** Attacker has network access to the RPC endpoint (localhost or misconfigured remote). `Pool` module is enabled (default).

**Steps:**

1. Victim submits a transaction:
```json
POST http://127.0.0.1:8114/
{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<tx>, "passthrough"]}
// Response: {"result": "0xTARGET_TX_HASH"}
```

2. Attacker observes the pool (or learns the hash via any public channel):
```json
{"id":2,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}
// Response lists all pending tx hashes including 0xTARGET_TX_HASH
```

3. Attacker removes the victim's transaction without any credential:
```json
{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xTARGET_TX_HASH"]}
// Response: {"result": true}
```

4. The transaction is gone from the pool. If the victim resubmits, the attacker repeats step 3 in a loop. The transaction is never confirmed.
