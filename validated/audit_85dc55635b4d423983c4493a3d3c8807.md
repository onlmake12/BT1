### Title
RBF `replace_count` Scales Super-Linearly When Conflicts Share Descendants, Blocking Valid Replacements - (`tx-pool/src/pool.rs`)

### Summary

In `check_rbf`, the `replace_count` guard (Rule #5, max 100 replacement candidates) is computed by summing each conflict's descendant count independently. When multiple conflicting transactions share descendants — which is valid in CKB's UTXO model — the counter double-counts those shared descendants, causing it to grow quadratically with the number of conflicts. A valid RBF replacement whose actual unique-transaction count is within the 100-tx limit is incorrectly rejected.

### Finding Description

`check_rbf` in `tx-pool/src/pool.rs` enforces Rule #5 as follows:

```rust
let mut replace_count: usize = 0;
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    replace_count += descendants.len() + 1;   // ← per-conflict accumulation
    if replace_count > MAX_REPLACEMENT_CANDIDATES {
        return Err(Reject::RBFRejected(...));
    }
    ...
}
```

`calc_descendants` returns a `HashSet<ProposalShortId>` of all transitive children of a single conflict. [1](#0-0) 

When two conflicts `tx_A` and `tx_B` both have `tx_C` as a descendant (e.g., `tx_C` spends one output from `tx_A` and one from `tx_B`), `tx_C` is counted once for `tx_A` and again for `tx_B`. The actual unique set of transactions to be replaced is `{tx_A, tx_B, tx_C}` = 3, but `replace_count` becomes 4.

The accumulation is never deduplicated across conflicts. [2](#0-1) 

By contrast, `calculate_min_replace_fee` correctly deduplicates via a `HashMap` keyed on `id` before summing fees, so the fee check is unaffected. [3](#0-2) 

The pattern mirrors the Wagmi bug exactly: a per-item value (`descendants.len()`) is computed on the full set and then multiplied by the item count (number of conflicts), producing quadratic growth instead of linear growth.

### Impact Explanation

An unprivileged attacker who can submit transactions to the mempool can permanently block a victim from RBF-replacing attacker-controlled conflicting transactions. Concretely:

- Victim holds UTXOs `U1 … UN`.
- Attacker submits `tx_A` (spends `U1`), `tx_B` (spends `U2`), …, `tx_N` (spends `UN`).
- Attacker submits `M` descendant transactions each spending outputs from multiple of the above, creating `M` shared descendants.
- Victim's replacement transaction spends all `U1 … UN` and triggers `check_rbf`.
- `replace_count` = `N × (M + 1)` instead of `N + M`.
- For `N = 10`, `M = 5`: actual unique count = 15, but `replace_count` = 60. For `N = 10`, `M = 9`: actual = 19, `replace_count` = 100 — right at the boundary. For `N = 10`, `M = 10`: actual = 20, `replace_count` = 110 → rejected.

The victim cannot replace the attacker's transactions via RBF, and the attacker's transactions may be mined, permanently spending the victim's UTXOs.

### Likelihood Explanation

RBF is an opt-in feature enabled when `min_rbf_rate > min_fee_rate`. [4](#0-3)  Any node operator who enables RBF exposes users to this attack. The attacker only needs to submit a small number of cheap transactions (no special privilege, no key material, no majority hashpower). The attack is fully reachable via the standard `send_transaction` RPC endpoint.

### Recommendation

Replace the per-conflict accumulation with a single deduplicated union of all descendants across all conflicts, then count that union:

```rust
let mut all_descendants: HashSet<ProposalShortId> = HashSet::new();
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    all_descendants.extend(descendants);
}
let replace_count = all_descendants.len() + conflicts.len();
if replace_count > MAX_REPLACEMENT_CANDIDATES { ... }
```

This mirrors how `calculate_min_replace_fee` already deduplicates via a `HashMap` before summing fees. [5](#0-4) 

### Proof of Concept

Setup (all submitted via `send_transaction` RPC, no privileges required):

1. Victim holds two live cells at `OutPoint(H_prev, 0)` and `OutPoint(H_prev, 1)`.
2. Attacker submits `tx_A`: input = `OutPoint(H_prev, 0)`, output = `OutPoint(H_A, 0)`.
3. Attacker submits `tx_B`: input = `OutPoint(H_prev, 1)`, output = `OutPoint(H_B, 0)`.
4. Attacker submits `tx_C`: inputs = `[OutPoint(H_A, 0), OutPoint(H_B, 0)]` — a shared descendant of both `tx_A` and `tx_B`.

Now the pool contains: `tx_A`, `tx_B`, `tx_C`.

5. Victim submits replacement `tx_R`: inputs = `[OutPoint(H_prev, 0), OutPoint(H_prev, 1)]` with higher fee.

Inside `check_rbf`:
- `conflicts` = `[tx_A, tx_B]` (both share input with `tx_R`).
- Iteration 1 (`tx_A`): `descendants({tx_A})` = `{tx_C}`, `replace_count` += 2 → `replace_count` = 2.
- Iteration 2 (`tx_B`): `descendants({tx_B})` = `{tx_C}`, `replace_count` += 2 → `replace_count` = 4.
- Actual unique set = `{tx_A, tx_B, tx_C}` = 3.

Scale this to `N = 51` conflicts each with 1 shared descendant: `replace_count` = 102 > `MAX_REPLACEMENT_CANDIDATES` (100), but actual unique count = 52. The replacement is rejected with `"Tx conflict with too many txs"`. [6](#0-5)
