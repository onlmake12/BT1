### Title
Unbounded Transaction Size in `advanceDelegationRecord` Causes Reward Claim Failure for Users with Many Publisher Delegations — (File: `governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

---

### Summary

The `advanceDelegationRecord` function in the Pyth staking SDK bundles an unbounded number of on-chain instructions — one `advanceDelegationRecord` and one `mergeDelegationPositions` per publisher the user has delegated to — into a **single Solana transaction** with no batching. A user delegated to the maximum number of publishers (up to 20, the position cap) will generate a transaction with 40+ instructions referencing 90+ unique accounts, far exceeding Solana's 1232-byte transaction size limit. This permanently prevents affected users from claiming staking rewards.

---

### Finding Description

`advanceDelegationRecord` in `pyth-staking-client.ts` calls `getAdvanceDelegationRecordInstructions`, which iterates over every publisher the user has an active delegation position with and generates two instructions per publisher:

1. `advanceDelegationRecord` (integrity pool program)
2. `mergeDelegationPositions` (integrity pool program)

All of these instructions are then concatenated and submitted as a **single transaction**:

```
[
  ...instructions.advanceDelegationRecordInstructions,
  ...instructions.mergePositionsInstruction,
]
``` [1](#0-0) 

The `PoolData` account supports up to **1024 publishers** in the pool, and a user's `PositionData` account supports up to **20 positions** (the hard cap enforced by `TooManyPositions`). [2](#0-1) [3](#0-2) 

With 20 positions across 20 different publishers, the resulting transaction would reference:
- 20 `publisher` accounts
- 20 `delegationRecord` PDAs
- 20 `publisherStakeAccountPositions` (optional)
- 20 `publisherStakeAccountCustody` (optional)
- Shared accounts: `payer`, `stakeAccountPositions`, `poolData`, `poolConfig`, `poolRewardCustody`, `stakeAccountCustody`, `tokenProgram`, `systemProgram`, `stakingProgram`, `stakeAccountMetadata`, `configAccount`, `owner`

This yields **90+ unique account keys** × 32 bytes = **2880+ bytes** for the account key section alone, far exceeding Solana's hard 1232-byte transaction size limit. The transaction will be rejected by the runtime before execution.

The same pattern exists in `unstakeFromAllPublishers`, which also bundles all `undelegate` instructions into a single transaction: [4](#0-3) 

---

### Impact Explanation

A staking user who has delegated to many publishers (e.g., 10–20) will be **unable to claim their accrued staking rewards** because `advanceDelegationRecord` — the only reward-claiming path — will always produce a transaction that exceeds Solana's size limit. The user's rewards are not lost (they remain in the pool), but they are inaccessible until the SDK is fixed. Similarly, `unstakeFromAllPublishers` would fail, preventing full exit from all delegations in one call.

---

### Likelihood Explanation

Any unprivileged staking user can reach this state by creating positions with multiple different publishers. The `PositionData` account allows up to 20 positions, and a user can spread these across different publishers. The `stakeToGovernance` flow even calls `mergeTargetPositions` after each stake, which consolidates same-target positions, but cross-publisher positions are never merged. A user who has delegated to 10+ publishers over time will hit this limit. This is a realistic scenario for active participants in the integrity pool. [5](#0-4) 

---

### Recommendation

Implement batching in `advanceDelegationRecord` and `unstakeFromAllPublishers`. Split the generated instructions into chunks that fit within Solana's transaction size limit (e.g., 3–5 publishers per transaction) and submit them sequentially. The SDK already has the off-chain logic to enumerate publishers; it only needs to chunk the resulting instruction arrays before calling `sendTransaction`.

---

### Proof of Concept

1. Create a stake account and delegate to 15 different publishers (one position each, within the 20-position cap).
2. Wait one epoch so rewards accrue.
3. Call `client.advanceDelegationRecord(stakeAccountPositions)`.
4. `getAdvanceDelegationRecordInstructions` returns 15 `advanceDelegationRecord` + 15 `mergeDelegationPositions` = 30 instructions referencing ~70 unique accounts (~2240 bytes of account keys alone).
5. `sendTransaction` submits all 30 instructions in one transaction.
6. The Solana runtime rejects the transaction with a serialization/size error.
7. The user cannot claim rewards regardless of how many times they retry. [6](#0-5)

  public async unstakeFromAllPublishers(
    stakeAccountPositions: PublicKey,
    positionStates: (PositionState.LOCKED | PositionState.LOCKING)[],
  ) {
    const [stakeAccountPositionsData, currentEpoch] = await Promise.all([
      this.getStakeAccountPositions(stakeAccountPositions),
      getCurrentEpoch(this.connection),
    ]);

    const instructions = await Promise.all(
      stakeAccountPositionsData.data.positions
        .map((position, index) => {
          const publisher =
            position.targetWithParameters.integrityPool?.publisher;
          return publisher === undefined
            ? undefined
            : { index, position, publisher };
        })
        // By separating this filter from the next, typescript can narrow the
        // type and automatically infer that there will be no `undefined` values
        // in the array after this line.  If we combine those filters,
        // typescript won't narrow properly.
        .filter((positionInfo) => positionInfo !== undefined)
        .filter(({ position }) =>
          (positionStates as PositionState[]).includes(
            getPositionState(position, currentEpoch),
          ),
        )
        .reverse()
        .map(({ position, index, publisher }) =>
          this.integrityPoolProgram.methods
            .undelegate(index, convertBigIntToBN(position.amount))
            .accounts({ publisher, stakeAccountPositions })
            .instruction(),
        ),
    );

    return sendTransaction(instructions, this.connection, this.wallet);
  }
