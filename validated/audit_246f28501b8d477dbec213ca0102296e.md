### Title
Missing Gas Sufficiency Check in `Echo.executeCallback` Allows Silent Callback Failure While Provider Is Still Credited — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` invokes the user's callback with an explicit gas cap (`gas: req.callbackGasLimit`) but performs **no on-chain check** that the calling context actually has enough gas to forward that amount. Because the provider is credited and the request is cleared **before** the callback is attempted, a caller (including the provider itself) can supply insufficient gas, cause the callback to silently fail via the `catch` block, and still receive full payment — with no retry path available to the user.

---

### Finding Description

In `Echo.executeCallback`, the execution order is:

1. **Provider credited** — `_state.providers[providerToCredit].accruedFeesInWei += ...` (line 161–162)
2. **Request cleared** — `clearRequest(sequenceNumber)` (line 164)
3. **Callback attempted** — `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` (lines 176–179) [1](#0-0) [2](#0-1) 

Due to the EVM's 63/64 forwarding rule (EIP-150), if `executeCallback` is entered with total gas `G`, the callback can receive **at most** `G × 63/64` gas. If `G × 63/64 < callbackGasLimit`, the callback receives less gas than the user paid for, runs out of gas, and the bare `catch` block at line 192 silently absorbs the out-of-gas revert and emits `PriceUpdateCallbackFailed`. [3](#0-2) 

Because the request was already cleared and the provider already credited before the callback was attempted, the user has **no recourse**: the request cannot be retried, and the provider keeps the fee.

The `IEcho` NatSpec acknowledges the 1.5× overhead requirement in documentation only — it is **not enforced on-chain**: [4](#0-3) 

Compare this to `Entropy.sol`, which has an explicit gas sufficiency guard before treating a callback failure as genuine:

```solidity
} else if (
    (startingGas * 31) / 32 >
    uint256(req.gasLimit10k) * TEN_THOUSAND
) {
    // sufficient gas was available → genuine callback failure
``` [5](#0-4) 

`Echo.sol` has no equivalent guard.

---

### Impact Explanation

A malicious provider (or any third party) calls `executeCallback` with just enough gas to execute the function's own logic and the `catch` block, but not enough to forward `callbackGasLimit` to the callback. The result:

- The callback receives less gas than `callbackGasLimit` and reverts out-of-gas.
- The `catch` block emits `PriceUpdateCallbackFailed` — a non-reverting outcome.
- The provider is credited the full fee.
- The request is permanently cleared with no retry mechanism.
- The user's `echoCallback` never executes, despite having paid for `callbackGasLimit` gas.

This breaks the core service guarantee of Echo: that the callback will be executed with the gas the user paid for.

---

### Likelihood Explanation

`executeCallback` is a public, permissionless function with no access control. [6](#0-5) 

The provider has a direct financial incentive: by calling with insufficient gas they save on transaction costs while still receiving the full fee. Any registered provider can exploit this on every request they fulfill. The `callbackGasLimit` field is stored in the `Request` struct and is visible on-chain, making it trivial to compute the exact gas needed to trigger the failure. [7](#0-6) 

---

### Recommendation

1. **Add a gas sufficiency check** before the callback, mirroring `Entropy.sol`:

```solidity
uint256 startingGas = gasleft();
// ... callback try/catch ...
// After catch: if callback failed AND startingGas * 31/32 > callbackGasLimit → genuine failure
// Otherwise → revert with InsufficientGas()
if (!success && (startingGas * 31) / 32 <= uint256(req.callbackGasLimit)) {
    revert InsufficientGas();
}
```

2. **Move `clearRequest` and provider credit to after a successful callback**, or implement a retry mechanism (similar to `Entropy.sol`'s `CALLBACK_FAILED` state) so that a failed callback due to insufficient gas can be retried.

3. **Enforce the 1.5× gas requirement on-chain** rather than only in documentation, by reverting `executeCallback` if `gasleft() < (3 * uint256(req.callbackGasLimit)) / 2`.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 500_000`, paying the corresponding fee.
2. Malicious provider calls `executeCallback{gas: 400_000}(...)`.
3. Due to 63/64 rule, the callback receives at most `400_000 × 63/64 ≈ 393_750` gas — less than the 500,000 promised.
4. The callback runs out of gas; the bare `catch` block fires and emits `PriceUpdateCallbackFailed`.
5. The provider's `accruedFeesInWei` was already incremented at step (1) of the function body; the request was already cleared.
6. The user's `echoCallback` never ran. The provider keeps the fee. No retry is possible.
