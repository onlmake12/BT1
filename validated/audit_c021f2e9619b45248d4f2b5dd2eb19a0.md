### Title
Entropy Provider Can Set Fee to Arbitrary Value With No Upper Bound or Timelock, Enabling Frontrunning and Excess ETH Capture — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

`Entropy.setProviderFee` and `Entropy.setProviderFeeAsFeeManager` allow any registered Entropy provider (or their fee manager) to set `feeInWei` to any `uint128` value — including `type(uint128).max` — with no upper bound, no timelock, and immediate on-chain effect. Because the Entropy contract explicitly does **not** refund excess `msg.value` to callers, a provider can frontrun a user's pending `requestWithCallback` / `requestV2` transaction by raising their fee to match or exceed the user's `msg.value`, causing the excess ETH to be silently absorbed as Pyth protocol fees rather than returned to the user.

---

### Finding Description

`setProviderFee` in `Entropy.sol` (lines 810–827) and `setProviderFeeAsFeeManager` (lines 829–855) update `provider.feeInWei` to any caller-supplied `uint128` with no validation:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;          // ← no upper bound, immediate
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
    ...
}
``` [1](#0-0) 

The fee is consumed at request time inside `requestHelper`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [2](#0-1) 

The IEntropy interface explicitly documents that excess value is **not** refunded:

> "Note that excess value is *not* refunded to the caller." [3](#0-2) 

The total fee is `providerFee + pythFeeInWei`: [4](#0-3) 

Any `msg.value` above `providerFee` flows into `accruedPythFeesInWei` — it is not returned to the user.

---

### Impact Explanation

**Frontrunning / excess ETH capture:** A user's integrating contract calls `getFee(provider)` off-chain, then submits `requestWithCallback{value: fee}(...)`. A malicious provider observes the pending transaction in the mempool and frontruns it with `setProviderFee(fee)` (raising their fee to exactly the user's `msg.value`). The user's transaction now succeeds (fee check passes), but the entire `msg.value` is consumed — the provider receives their new higher fee and Pyth receives the remainder. The user pays significantly more than they agreed to.

**Griefing / DoS:** A provider can set `feeInWei = type(uint128).max`, making all future requests to that provider revert with `InsufficientFee`. If the provider is the contract's `defaultProvider`, this effectively DoS-es the entire Entropy contract for users relying on the default. [5](#0-4) 

**No timelock:** The fee change is atomic and takes effect in the same block, giving users no opportunity to react.

---

### Likelihood Explanation

Any address that has called `register()` on the Entropy contract becomes a provider and can immediately call `setProviderFee`. This is a permissionless role — no admin approval is required to register. A malicious actor can register, attract users (e.g., by being set as the default provider or advertising a low fee), and then execute the frontrun. The attack is straightforward to execute on any EVM chain where Entropy is deployed and mempool transactions are visible.

---

### Recommendation

1. **Enforce an upper bound** on `feeInWei` (e.g., a protocol-level `MAX_PROVIDER_FEE` constant).
2. **Introduce a timelock or two-step fee update**: providers announce a new fee, and it only takes effect after a delay (e.g., 24 hours), giving users time to react.
3. **Refund excess `msg.value`** to the caller in `requestHelper` rather than silently absorbing it into `accruedPythFeesInWei`.

---

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
