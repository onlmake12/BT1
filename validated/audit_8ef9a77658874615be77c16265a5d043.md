### Title
Scheduler Allows Individual Feed Prices to Be Overwritten with Older Timestamps via Max-Timestamp Ordering — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract enforces update ordering using the **maximum** `publishTime` across all feeds in a batch, not per-feed timestamps. Because `_storePriceUpdates` is called unconditionally after this single max-timestamp check passes, an attacker can submit a valid Wormhole-signed update where one feed carries a newer timestamp (to satisfy the ordering gate) while another feed carries an **older** timestamp than what is currently stored. This overwrites the stale feed's cached price with an even older value.

---

### Finding Description

In `_validateShouldUpdatePrices`, the contract computes `updateTimestamp` as the maximum `publishTime` across all feeds in the submitted update: [1](#0-0) 

It then rejects the update only if this maximum is not strictly greater than `status.priceLastUpdatedAt`: [2](#0-1) 

After validation passes, `_storePriceUpdates` is called with the full `priceFeeds` array: [3](#0-2) 

The design rationale is explicitly documented — closed equity markets will carry a `publishTime` from their last trading period, so the max-timestamp heuristic is used: [4](#0-3) 

However, this means that if a new update's maximum timestamp exceeds `priceLastUpdatedAt`, **every** feed in the batch is stored — including feeds whose individual `publishTime` is older than the value currently cached for that feed. There is no per-feed timestamp guard in the storage path.

This is structurally identical to the RedstoneCoreOracle bug: the cache-ordering check uses an aggregate/proxy timestamp (`block.timestamp` there; `max(publishTime)` here) rather than the individual price's own observation timestamp.

By contrast, the core EVM Pyth contract correctly enforces per-feed ordering: [5](#0-4) 

The Scheduler lacks this per-feed guard.

---

### Impact Explanation

A protocol consuming prices from the Scheduler (e.g., a lending protocol using ETH/USD and AAPL/USD) could have one feed's stored price silently rolled back to an older value. Depending on the direction of the price movement, this could enable:

- **Over-borrowing**: if the stale price is higher than the current price for a collateral asset.
- **Illegitimate liquidation**: if the stale price is lower than the current price for a collateral asset.

The Scheduler's stored prices are consumed by reader contracts via `getLatestPrice`; if those readers do not independently enforce a per-feed staleness check against `publishTime`, they will silently use the rolled-back price.

---

### Likelihood Explanation

The attacker requires a **valid, Wormhole-signed** Pythnet update where:
- At least one feed (e.g., a crypto feed) has a `publishTime` strictly greater than `status.priceLastUpdatedAt`.
- At least one other feed (e.g., an equity feed whose market just closed) has a `publishTime` older than the value currently stored for that feed.

This is a **realistic and recurring** condition. Equity markets close daily; during after-hours periods, every Pythnet slot will contain equity feeds with a `publishTime` from the last close. An attacker monitoring the mempool can selectively submit such an update at a moment that benefits their position, front-running the honest keeper.

The `updatePriceFeeds` entry point on the Scheduler has no caller restriction — any address can submit a valid update: [6](#0-5) 

---

### Recommendation

In `_storePriceUpdates` (or immediately before calling it), add a per-feed timestamp check analogous to `updateLatestPriceIfNecessary` in the core contract: only overwrite a stored feed if the incoming `publishTime` is strictly greater than the currently stored `publishTime` for that feed. This mirrors the correct pattern already used in `PythSetters.updateLatestPriceIfNecessary`. [7](#0-6) 

---

### Proof of Concept

**Setup**: A subscription tracks two feeds — `CRYPTO/USD` (always-open) and `EQUITY/USD` (market hours only).

**Step 1 — Initial update (slot S1, block time T1):**
- `CRYPTO/USD.publishTime = T1`
- `EQUITY/USD.publishTime = T1` (market open)
- `updateTimestamp = T1`, `priceLastUpdatedAt = T1`
- Both feeds stored with `publishTime = T1`.

**Step 2 — Equity market closes at T1+ε.**

**Step 3 — Attacker waits for slot S2 (block time T2 > T1):**
- A valid Pythnet update exists: `CRYPTO/USD.publishTime = T2`, `EQUITY/USD.publishTime = T0` (last close, `T0 < T1`).
- Both feeds share the same slot (slot check passes).
- `updateTimestamp = max(T2, T0) = T2 > T1 = priceLastUpdatedAt` → ordering check passes.
- `_storePriceUpdates` stores both feeds.
- **Result**: `EQUITY/USD` is now stored with `publishTime = T0 < T1`, a price older than what was previously cached.

**Step 4 — Attacker exploits the stale `EQUITY/USD` price** in a lending protocol that reads from the Scheduler without an independent per-feed staleness check.
