### Title
Unbounded `ids[]` Parameter in Hermes REST Endpoints Enables Resource-Consumption DOS — (`apps/hermes/server/src/api/rest/v2/latest_price_updates.rs`, `apps/hermes/server/src/api/rest/latest_price_feeds.rs`, `apps/hermes/server/src/api/rest/latest_vaas.rs`)

---

### Summary

Every Hermes REST price-update endpoint accepts an unbounded `ids[]` query-parameter array. No application-level count limit is enforced before the server allocates memory, iterates over the IDs, and builds a response. An unprivileged caller can send a single request containing thousands of duplicate valid price IDs, forcing large allocations and expensive processing before any rejection occurs. The per-IP rate limit (10 req / 10 s) is per-request, not per-ID, so it does not bound the per-request cost.

---

### Finding Description

`LatestPriceUpdatesQueryParams`, `LatestPriceFeedsQueryParams`, and `LatestVaasQueryParams` all declare the IDs field as an unbounded `Vec<PriceIdInput>`:

```rust
ids: Vec<PriceIdInput>,
``` [1](#0-0) [2](#0-1) [3](#0-2) 

The only validation performed before processing is `validate_price_ids`, which checks whether each ID exists in the known set — it does **not** check the count:

```rust
pub async fn validate_price_ids<S>(
    state: &ApiState<S>,
    price_ids: &[PriceIdentifier],
    remove_invalid: bool,
) -> Result<Vec<PriceIdentifier>, RestError>
{
    let available_ids = Aggregates::get_price_feed_ids(state).await;
    let (valid_ids, invalid_ids): (Vec<_>, Vec<_>) = price_ids
        .iter()
        .copied()
        .partition(|id| available_ids.contains(id));
    ...
    Ok(valid_ids)
}
``` [4](#0-3) 

After `validate_price_ids` returns, the handler immediately calls `get_price_feeds_with_update_data` with the full (potentially enormous) validated list, then serializes and encodes every entry into the response:

```rust
let price_feeds_with_update_data =
    Aggregates::get_price_feeds_with_update_data(state, &price_ids, RequestTime::Latest)
        .await ...;
let encoded_data: Vec<String> = price_update_data
    .into_iter()
    .map(|data| match params.encoding { ... })
    .collect();
``` [5](#0-4) 

An attacker can repeat a single known-valid price ID (e.g., BTC/USD) ten thousand times in one GET request. All ten thousand copies pass `validate_price_ids` (they are valid), causing:

1. A 10,000-element `Vec` allocation for the validated IDs.
2. A 10,000-entry cache lookup in `get_price_feeds_with_update_data`.
3. A 10,000-entry JSON/hex response serialization and transmission.

The public Hermes rate limit is 10 requests per 10 seconds per IP address. [6](#0-5) 

Because the limit is per-request and not per-ID, a single allowed request can carry an arbitrarily large payload. With even a modest number of source IPs the attacker can sustain continuous high-cost requests.

---

### Impact Explanation

Hermes is the sole off-chain relay that serves signed price-update proofs to every Pyth consumer. If Hermes becomes unavailable, no on-chain contract on any supported chain can receive fresh price data. This directly blocks:

- DeFi protocols that call `updatePriceFeeds` before every trade.
- The Scheduler keeper network (`updatePriceFeeds` in `Scheduler.sol`) that depends on Hermes for update data.
- Any application using the SSE streaming endpoint.

A sustained attack exhausting Hermes memory or CPU constitutes a high-impact availability failure for the entire Pyth pull-oracle ecosystem.

---

### Likelihood Explanation

The attack requires only valid HTTP GET requests to a public endpoint — no authentication, no on-chain funds, no privileged role. The attacker needs only a handful of source IPs to stay under the per-IP rate limit while sustaining a high aggregate request rate. The Pyth network currently publishes ~500 price feed IDs; an attacker can repeat any subset of them arbitrarily many times in a single query string. The attack is trivially scriptable.

---

### Recommendation

Add an application-level maximum on the number of IDs accepted per request, analogous to the `MAX_PRICE_IDS` guard in `Echo.sol` and `MAX_PRICE_IDS_PER_SUBSCRIPTION` in `SchedulerConstants.sol`:

```rust
const MAX_IDS_PER_REQUEST: usize = 500; // or a suitable operational limit

if price_id_inputs.len() > MAX_IDS_PER_REQUEST {
    return Err(RestError::TooManyPriceIds);
}
```

This check must be placed **before** `validate_price_ids` is called, so that no allocation or cache lookup occurs for oversized requests. The same guard should be applied to all endpoints that accept `ids[]`: `/v2/updates/price/latest`, `/v2/updates/price/stream`, `/api/latest_price_feeds`, and `/api/latest_vaas`. [7](#0-6) [8](#0-7) 

---

### Proof of Concept

```bash
# Construct a query string with 5000 copies of the BTC/USD price ID
BTCUSD="e62df6c8b4a85fe1a67db44dc12de5db330f7ac66b72dc658afedf0f4a415b43"
QUERY=$(python3 -c "print('&'.join(['ids[]=' + '$BTCUSD'] * 5000))")

# Single request — passes the rate limiter, forces 5000-entry processing
curl -s "https://hermes.pyth.network/v2/updates/price/latest?${QUERY}" > /dev/null
```

Each such request forces Hermes to:
1. Allocate a 5,000-element `Vec<PriceIdentifier>`.
2. Perform 5,000 HashSet lookups in `validate_price_ids`.
3. Execute `get_price_feeds_with_update_data` for 5,000 IDs.
4. Serialize and transmit a 5,000-entry JSON response.

Repeating at the allowed rate of 10 requests per 10 seconds from a single IP yields 50,000 ID-lookups per second sustained from one source. Scaling across a small number of IPs produces a resource-exhaustion condition on the Hermes process. [9](#0-8) [4](#0-3)

```rust
pub async fn validate_price_ids<S>(
    state: &ApiState<S>,
    price_ids: &[PriceIdentifier],
    remove_invalid: bool,
) -> Result<Vec<PriceIdentifier>, RestError>
where
    S: Aggregates,
{
    let state = &*state.state;
    let available_ids = Aggregates::get_price_feed_ids(state).await;

    // Partition into (valid_ids, invalid_ids)
    let (valid_ids, invalid_ids): (Vec<_>, Vec<_>) = price_ids
        .iter()
        .copied()
        .partition(|id| available_ids.contains(id));

    if invalid_ids.is_empty() || remove_invalid {
        // All IDs are valid
        Ok(valid_ids)
    } else {
        // Return error with list of missing IDs
        Err(RestError::PriceIdsNotFound {
            missing_ids: invalid_ids,
        })
    }
}
