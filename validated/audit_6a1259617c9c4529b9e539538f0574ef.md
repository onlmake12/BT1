### Title
Out-of-bounds index in `TxConfirmStat::remove_unconfirmed_tx` when `tx_age > MAX_CONFIRM_BLOCKS` causes node panic — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary

`remove_unconfirmed_tx` guards the `block_unconfirmed_txs` access with a bounds check but leaves the `confirm_blocks_to_failed_txs[tx_age - 1]` access completely unguarded. When a tracked transaction remains unconfirmed for more than `MAX_CONFIRM_BLOCKS` (1000) blocks and is then evicted from the tx-pool, `tx_age - 1 >= 1000` indexes past the end of a `Vec` of length 1000, causing an unconditional Rust index-out-of-bounds **panic** that terminates the node process.

### Finding Description

`confirm_blocks_to_failed_txs` is allocated with exactly `MAX_CONFIRM_BLOCKS = 1000` rows (valid indices `0..=999`): [1](#0-0) [2](#0-1) 

Inside `remove_unconfirmed_tx`, the code correctly detects `tx_age >= 1000` and redirects the `block_unconfirmed_txs` decrement to `old_unconfirmed_txs`: [3](#0-2) 

However, immediately after that branch, the `confirm_blocks_to_failed_txs` write uses the same unbounded `tx_age` with **no corresponding guard**: [4](#0-3) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is past the last valid index (999). Rust's `Vec` indexing panics on out-of-bounds access in release builds.

The call chain that sets `count_failure = true` is:

- `reject_tx` → `drop_tx` (line 429) → `drop_tx_inner(count_failure = true)` (line 416) → `remove_unconfirmed_tx(..., count_failure = true)` (line 418) [5](#0-4) [6](#0-5) 

### Impact Explanation

A node process panic terminates the CKB node entirely. Any peer that can submit a valid low-fee transaction and wait for it to age past 1000 blocks before the tx-pool evicts it can deterministically crash the target node. This is a **High** severity denial-of-service.

### Likelihood Explanation

The CKB tx-pool does not enforce a hard 1000-block eviction limit tied to `MAX_CONFIRM_BLOCKS`. A transaction with a fee rate below the pool's eviction threshold can remain tracked by the fee estimator for well over 1000 blocks. The attacker needs only to submit one valid (but low-fee) transaction and wait; no special privilege, key, or hashpower is required.

### Recommendation

Add a bounds check before the `confirm_blocks_to_failed_txs` write, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure {
    if let Some(row) = self.confirm_blocks_to_failed_txs.get_mut(tx_age - 1) {
        row[bucket_index] += 1f64;
    }
    // silently drop the sample when tx_age > MAX_CONFIRM_BLOCKS
}
```

This is consistent with the design intent: transactions older than `MAX_CONFIRM_BLOCKS` are already treated as "old" and their unconfirmed-count is moved to `old_unconfirmed_txs`; failure samples beyond the tracking window should likewise be discarded.

### Proof of Concept

```rust
#[test]
fn test_no_panic_on_old_failed_tx() {
    let mut algo = Algorithm::new();
    algo.update_ibd_state(false); // mark ready

    // Simulate current_tip = 0, accept a tx
    // (current_tip is set via commit_block; we need to advance it first)
    // Advance best_height to 0 by committing a dummy block at height 0 is not
    // possible directly, so we use the internal track_tx path:
    let tx_hash = ckb_types::packed::Byte32::default();
    let info = TxEntryInfo { fee: 1000.into(), size: 100, cycles: 1000 };
    algo.accept_tx(tx_hash.clone(), info); // tracked at current_tip=0

    // Advance best_height by 1001 blocks without confirming the tx
    for h in 1u64..=1001 {
        algo.commit_block(&make_empty_block(h));
    }

    // This must NOT panic; before the fix it panics with index out of bounds
    algo.reject_tx(&tx_hash);
}
```

The assertion is that `tx_age - 1` (= 1000) must be `< confirm_blocks_to_failed_txs.len()` (= 1000) — which it is not, proving the out-of-bounds access.
