### Title
Unbounded `while` Loop in `executeCallback` Enables Gas-Exhaustion DoS and Fund Lock — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that linearly scans storage for every sequence number between `_state.firstUnfulfilledSeq` and `_state.currentSequenceNumber`. An unprivileged user can force this scan to grow arbitrarily large, causing the fulfillment transaction to exceed the block gas limit and permanently locking the requester's funds.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` advances `_state.firstUnfulfilledSeq` past all inactive (already-fulfilled) requests:

```solidity
// Echo.sol lines 169-174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at least one `SLOAD` against the 32-slot `requests` array and, for overflow entries, a second cold `SLOAD` against `requestsOverflow`:

```solidity
// Echo.sol lines 310-321
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

The developers themselves flag this in a TODO comment immediately above the loop:

> "I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. a better solution would be a doubly-linked list of active requests." [3](#0-2) 

`_state.firstUnfulfilledSeq` is never explicitly initialised; it defaults to `0`. `_state.currentSequenceNumber` starts at `1` and increments with every new request: [4](#0-3) 

There is no cap on `currentSequenceNumber`, so the gap `currentSequenceNumber − firstUnfulfilledSeq` is unbounded.

---

### Impact Explanation

**Scenario — attacker-controlled gap growth:**

1. Attacker submits request with sequence number `1` (pays the minimum fee; provider may deprioritise it).
2. Over time (or via the attacker's own transactions), requests `2 … N` are created and fulfilled. Each fulfillment call runs the while loop but stops immediately at `firstUnfulfilledSeq = 1` because request `1` is still active — so no gas problem yet.
3. When request `1` is eventually fulfilled, `clearRequest(1)` marks it inactive, and the while loop must now scan every sequence number from `1` to `N`.

**Gas cost per iteration (warm main-array slot):** ~130 gas (`keccak256` + one warm `SLOAD`).  
**Gas cost per iteration (cold overflow slot):** ~2,230 gas.

At 30 M gas limit, the loop exhausts gas after roughly **13,000–230,000 iterations** depending on overflow ratio. Once the transaction reverts, the requester's fee is permanently locked because there is no refund path for a failed `executeCallback`. [5](#0-4) 

---

### Likelihood Explanation

- `executeCallback` has **no access control**; any address may call it.
- `requestPriceUpdatesWithCallback` is open to any address that pays the fee.
- The gap grows naturally in a live system whenever a low-priority request (e.g., one with a marginal fee) sits unfulfilled while higher-priority requests accumulate and are fulfilled.
- No privileged role, leaked key, or external oracle manipulation is required.
- The code's own TODO comment confirms the developers consider this a known gas-cost concern, increasing the likelihood it will be triggered in production. [6](#0-5) 

---

### Recommendation

Replace the linear scan with a data structure that allows O(1) advancement of `firstUnfulfilledSeq`:

- **Doubly-linked list** of active sequence numbers (as suggested in the TODO).
- **Bitmap** of active slots: a single `SLOAD` of a 256-bit word covers 256 sequence numbers.
- **Hard cap** on loop iterations as a short-term mitigation (e.g., `if (++iterations > MAX_SCAN) break`), accepting that `firstUnfulfilledSeq` may lag but preventing revert.

---
