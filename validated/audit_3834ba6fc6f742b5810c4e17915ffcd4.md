### Title
Unbounded Concurrent Decompression Allocation in `LengthDelimitedCodecWithCompress::decode` Enables Remote OOM — (`network/src/compress.rs`)

### Summary

An unprivileged remote peer can open many TCP connections to a CKB node and, on each, send a RelayV3 (protocol_id=101) frame whose snappy-compressed payload is ≤ 4 MB but whose declared decompressed length is exactly 8 MB (`MAX_UNCOMPRESSED_LEN`). The decoder allocates `BytesMut::zeroed(8 MB)` per frame with no global memory cap or rate limit on concurrent decompression. With enough simultaneous connections the aggregate heap allocation can exhaust node memory, crashing the node and causing consensus deviation.

---

### Finding Description

**Constants and limits (all confirmed in source):**

- `MAX_UNCOMPRESSED_LEN = 1 << 23` = 8 MB [1](#0-0) 

- `max_frame_length` for `RelayV3` = 4 MB (compressed wire limit) [2](#0-1) 

**Decode path — the vulnerable allocation:**

`LengthDelimitedCodecWithCompress::decode` first delegates to the inner `length_delimited` codec, which enforces the 4 MB compressed-frame limit. It then reads the snappy header to obtain `decompressed_bytes_len` and checks:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {   // strictly >
    return Err(io::ErrorKind::InvalidData.into());
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len);  // ← allocation
``` [3](#0-2) 

Because the guard is **strictly greater-than**, a frame whose declared decompressed size equals exactly 8 MB passes the check and causes an 8 MB heap allocation before any application-level logic runs. There is no global semaphore, no per-connection decompression budget, and no backpressure mechanism that would prevent N such allocations from occurring simultaneously across N connections.

**Compression is enabled for all production protocols** via `CKBProtocol::new_with_support_protocol`, which hard-codes `compress: true` and wires `LengthDelimitedCodecWithCompress` as the codec: [4](#0-3) [5](#0-4) 

**Amplification arithmetic:**

Snappy can compress 8 MB of highly-compressible data (e.g., repeated bytes) into well under 4 MB. The attacker therefore does not need to send 4 MB per connection; a few KB of wire data per connection is sufficient to trigger an 8 MB allocation. With `max_connection_number` connections (stated as 1024 in the question; confirmed to exist in `network/src/network.rs` but the exact value was not read in this analysis), the worst-case simultaneous allocation is:

```
1024 connections × 8 MB = 8 GB heap
``` [6](#0-5) 

---

### Impact Explanation

If the node's available memory is less than the aggregate simultaneous allocation, the OS OOM-killer terminates the process. A crashed full node stops participating in block propagation and consensus. If enough nodes are targeted simultaneously, block relay stalls and the network can experience consensus deviation. This matches the Critical (15001–25000) scope.

---

### Likelihood Explanation

The attack requires only:
1. The ability to open TCP connections to the victim's P2P port (publicly reachable on mainnet).
2. The ability to craft a valid snappy-compressed frame — the snappy format is public and trivial to implement.
3. No PoW, no key material, no privileged role.

The only practical friction is establishing a large number of simultaneous connections before the peer-management layer disconnects them, but this is a timing/rate issue, not a cryptographic barrier.

---

### Recommendation

1. **Change the guard to `>=`** so that a frame claiming exactly `MAX_UNCOMPRESSED_LEN` is also rejected:
   ```rust
   if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN {
   ``` [7](#0-6) 

2. **Add a global decompression semaphore** (e.g., `tokio::sync::Semaphore`) that caps the total bytes being decompressed concurrently across all sessions, independent of per-connection limits.

3. **Reduce `MAX_UNCOMPRESSED_LEN`** to a value consistent with the largest legitimate protocol message, or make it per-protocol rather than a single global constant.

4. **Apply per-connection message-rate limiting** at the codec layer so that a single peer cannot continuously trigger large allocations.

---

### Proof of Concept

```rust
// Craft a snappy frame that decompresses to exactly MAX_UNCOMPRESSED_LEN (8 MB)
let plaintext = vec![0u8; 8 * 1024 * 1024]; // 8 MB of zeros
let compressed = snap::raw::Encoder::new().compress_vec(&plaintext).unwrap();
// compressed.len() << 4 MB for zero-filled input

// Frame format: [4-byte length (compressed.len()+1)] [0x80 compress flag] [compressed bytes]
let mut frame = Vec::new();
frame.extend_from_slice(&((compressed.len() + 1) as u32).to_be_bytes());
frame.push(0x80); // COMPRESS_FLAG
frame.extend_from_slice(&compressed);

// Open N connections to victim's P2P port, send `frame` on each simultaneously.
// Each triggers BytesMut::zeroed(8 MB) in LengthDelimitedCodecWithCompress::decode.
// Assert victim RSS stays below safe threshold — it will not.
```

The frame's compressed size is well under 4 MB (snappy compresses 8 MB of zeros to ~32 KB), so it passes the `max_frame_length` check. The declared decompressed length equals `MAX_UNCOMPRESSED_LEN` exactly, so it passes the `>` guard. The 8 MB allocation proceeds unconditionally. [8](#0-7)
