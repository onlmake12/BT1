### Title
No `maxFee` Guard in `requestHelper` Allows Provider to Frontrun Fee Increase and Extract Excess `msg.value` from Entropy Users - (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract's `requestHelper` function — called by all `request`, `requestWithCallback`, and `requestV2` variants — enforces only a **minimum fee** check (`msg.value >= requiredFee`) with no upper-bound (`maxFee`) parameter. Because excess `msg.value` is explicitly **not refunded** and provider fees can be changed at any time via `setProviderFee`, a malicious or opportunistic provider can frontrun a user's pending request transaction by raising their fee, causing the user to overpay with no recourse.

---

### Finding Description

In `requestHelper`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The interface documentation explicitly states:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation. Further note that excess value is *not* refunded to the caller." [2](#0-1) 

Providers can change their fee at any time with no delay or timelock:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
``` [3](#0-2) 

The fee manager role (set via `setFeeManager`) can also change provider fees via `setProviderFeeAsFeeManager`: [4](#0-3) 

**Attack path:**

1. User calls `getFeeV2(provider, gasLimit)` off-chain — returns, say, 100 wei.
2. User submits `requestV2{value: 500 wei}(provider, gasLimit)` with a buffer to avoid reverts from fee fluctuations (a common and recommended practice).
3. Provider (or their fee manager) observes the pending transaction in the mempool and frontruns it with `setProviderFee(499 wei)` (just below the user's `msg.value`).
4. User's transaction executes: `requiredFee = 499 + pythFee`. Since `msg.value = 500 >= requiredFee`, the transaction succeeds.
5. `providerFee = 499 wei` is credited to the provider. The user paid 499 wei instead of the expected 100 wei — a 399 wei loss — with no refund.

The `accruedPythFeesInWei` receives `msg.value - providerFee = 1 wei` (the remainder), while the provider captures the inflated fee. The user has no way to specify a maximum acceptable fee.

---

### Impact Explanation

Users who send excess `msg.value` (a standard defensive practice explicitly encouraged by the documentation's warning that fees change over time) can be drained of their buffer by a malicious provider. The loss is bounded by the user's `msg.value` minus the minimum required fee. For contracts that wrap Entropy and pass user funds through (e.g., a gaming contract that collects fees from players and forwards them), the loss can be amplified across many users. There is no mechanism for the user to recover the overpaid amount.

---

### Likelihood Explanation

Any registered Entropy provider can execute this attack — registration is permissionless via `register()`. The provider controls their own fee via `setProviderFee` or delegates it to a fee manager via `setFeeManager`. Mempool monitoring is standard MEV infrastructure. The attack is most profitable on chains with public mempools and low block times. The documentation's own guidance to "explicitly compute `getFeeV2()` prior to each invocation" implicitly acknowledges fee volatility, making users likely to add buffers. [5](#0-4) 

---

### Recommendation

Add a `maxFee` parameter to `requestV2` (and its variants) that reverts if `msg.value > maxFee`:

```solidity
function requestV2(
    address provider,
    bytes32 userRandomNumber,
    uint32 gasLimit,
    uint128 maxFee          // NEW: revert if fee exceeds this
) public payable returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
    if (msg.value > maxFee) revert EntropyErrors.FeeTooHigh();
    ...
}
```

Alternatively, refund excess `msg.value` to the caller after deducting `requiredFee`. Either approach eliminates the frontrunning surface.
