### Title
Hardcoded Script Hashes with Empty-Args Prefix Allow Any Args to Pass `WellKnownScriptsOnly` Validator — (`File: rpc/src/module/pool.rs`)

---

### Summary

The `WellKnownScriptsOnly` output validator in the CKB RPC layer hardcodes the `anyone_can_pay` and `cheque` script code-hashes with empty `args` (`"0x"`). The matching function `is_well_known_script` uses a `starts_with` check on args. Because every byte slice starts with an empty slice, the args check is trivially bypassed: **any args value passes for these two scripts**. A transaction sender who explicitly requests `well_known_scripts_only` validation receives false safety assurance and can inadvertently (or be tricked into) sending funds to cells with empty, malformed, or attacker-chosen args.

---

### Finding Description

In `rpc/src/module/pool.rs`, `build_well_known_lock_scripts` registers the `anyone_can_pay` and `cheque` scripts with empty args:

```json
{ "code_hash": "0xd369597ff47f29fbc0d47d2e3775370d1250b85140c670e4718af712983a2354",
  "hash_type": "type", "args": "0x" }
{ "code_hash": "0xe4d4ecc6e5f9a059bf2f7a82cca292083aebc0c421566a52484fe2ec51a9fb0c",
  "hash_type": "type", "args": "0x" }
``` [1](#0-0) 

The matching predicate is:

```rust
fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script.args().as_slice().starts_with(well_known_script.args().as_slice())
}
``` [2](#0-1) 

Because `well_known_script.args()` is empty (`0x`), `starts_with(&[])` is unconditionally `true` in Rust. The args field is therefore **never validated** for these two scripts. The validator only checks `hash_type` and `code_hash`.

This is called from `validate_well_known_lock_scripts`, which is the fallback branch of `validate_lock_script` after the secp256k1 checks fail: [3](#0-2) 

The validator is invoked on every `send_transaction` or `test_tx_pool_accept` call when the caller passes `outputs_validator: "well_known_scripts_only"`: [4](#0-3) 

---

### Impact Explanation

The `anyone_can_pay` script (RFC-0026) requires a 20-byte (or 20+1+1 byte) lock arg identifying the cell owner. With empty args, the cell has **no owner** — any transaction can spend it, making the funds immediately unrecoverable by the sender. The `cheque` script similarly requires a 40-byte arg encoding sender and receiver lock hashes; empty or truncated args produce undefined script behavior.

A user or wallet that calls `send_transaction` with `"well_known_scripts_only"` trusts the node to reject unsafe outputs. Because the args check is a no-op, the node silently accepts outputs with:
- Empty args → anyone-spendable cell (permanent fund loss)
- Arbitrary attacker-supplied args → funds delivered to wrong recipient
- Malformed args → script execution failure at spend time

The validator's stated purpose — "restricts the lock script and type script usage" — is not fulfilled for these two scripts.

---

### Likelihood Explanation

Any RPC caller (wallet, dApp, exchange) that uses `well_known_scripts_only` and constructs an `anyone_can_pay` or `cheque` output with incorrect args will have those outputs accepted without warning. This is reachable by any unprivileged RPC user submitting a transaction. No special privileges, keys, or majority hashpower are required.

---

### Recommendation

Replace the empty `"args": "0x"` entries with the minimum valid args length for each script, and change the matching logic from `starts_with` to an exact-length check or a minimum-length check:

1. For `anyone_can_pay`: require `args.len() >= 20` (blake160 of pubkey).
2. For `cheque`: require `args.len() == 40` (two blake160 hashes).
3. Alternatively, change `is_well_known_script` to enforce a minimum args length when the well-known script's args are empty, rather than treating empty as a wildcard prefix.

---

### Proof of Concept

Submit via RPC:

```json
{
  "method": "send_transaction",
  "params": [
    {
      "outputs": [{
        "capacity": "0x2540be400",
        "lock": {
          "code_hash": "0xd369597ff47f29fbc0d47d2e3775370d1250b85140c670e4718af712983a2354",
          "hash_type": "type",
          "args": "0x"
        }
      }],
      ...
    },
    "well_known_scripts_only"
  ]
}
```

The validator accepts this output (empty args, `anyone_can_pay` code_hash) because `[].starts_with(&[])` is `true`. The resulting cell is spendable by anyone. The caller receives no error and believes the output was validated as safe. [2](#0-1) [5](#0-4)
