### Title
Expired Alert Re-Relay Bypass via Missing `cancel_filter` Update in `clear_expired_alerts` — (`util/network-alert/src/notifier.rs`)

### Summary
`clear_expired_alerts()` removes expired alerts from `received_alerts` and `noticed_alerts` but does not insert their IDs into `cancel_filter`. After expiry, `has_received()` returns `false` for those IDs, allowing any peer to re-relay the original signed alert bytes and have them accepted as new, re-triggering subscriber notifications and re-broadcast.

### Finding Description
`Notifier::clear_expired_alerts` uses `retain()` to drop entries from `received_alerts` and `noticed_alerts` whose `notice_until < now`. It never calls `self.cancel_filter.put(id, ())` for the removed IDs. `has_received()` returns `true` only if the ID is in `received_alerts` OR `cancel_filter`. Once expiry clears the former and the latter was never populated, `has_received(X)` returns `false`. An attacker who captured the original alert bytes (which were broadcast on the P2P network) can re-send them; the `received()` handler in `AlertRelayer` passes the `has_received` gate, re-verifies the still-valid signatures, calls `notifier.add()`, which fires `notify_controller.notify_network_alert()` again and re-broadcasts to all connected peers.

### Impact Explanation
- `notify_network_alert` fires again for an expired alert, causing subscribers (e.g., RPC clients, UI) to re-display or re-process a stale alert.
- The node re-broadcasts the expired alert to all connected peers; those peers have also cleared it from their `received_alerts`, so the re-relay cascades network-wide until each node re-adds it (at which point `has_received` returns `true` again — until the next `clear_expired_alerts` cycle).
- `noticed_alerts` is re-sorted on each re-add.
- `clear_expired_alerts` is triggered on every peer `connected()` event, so an attacker can repeatedly cycle: connect → trigger expiry cleanup → send expired alert → repeat.

### Likelihood Explanation
Any unprivileged P2P peer can execute this. The original alert bytes are public (broadcast to all peers). No key material or privileged access is needed. The only precondition is that a legitimately-signed alert has expired, which is a normal lifecycle event.

### Recommendation
In `clear_expired_alerts`, insert each removed ID into `cancel_filter` before dropping it:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ());
            false
        } else {
            true
        }
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

This mirrors the invariant already enforced by `cancel()`: once an alert ID is processed and removed, it is permanently recorded in `cancel_filter` so `has_received` continues to return `true`.

### Proof of Concept
```rust
// 1. Add alert with notice_until = T
let alert = build_alert(1, 0, None, None, T);
notifier.add(&alert);
// notify_network_alert called once

// 2. Advance time past T, clear expired
notifier.clear_expired_alerts(T + 1);
assert!(!notifier.has_received(1)); // gap confirmed

// 3. Re-relay same alert (same valid signatures)
notifier.add(&alert);
// notify_network_alert called a SECOND time — invariant broken
assert_eq!(notify_count, 1); // FAILS: count is 2
```
