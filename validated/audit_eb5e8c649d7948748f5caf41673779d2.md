Audit Report

## Title
Per-Session Rate Limiter Bypass via Reconnection — (`sync/src/relayer/mod.rs`, `network/src/protocols/hole_punching/mod.rs`)

## Summary
The `Relayer` and `HolePunching` protocol handlers key their rate limiters by `(PeerIndex, u32)`, where `PeerIndex` is a session-scoped integer reset on every new TCP connection. Because `TooManyRequests` (code 110, 1xx range) never triggers a ban, and `disconnected` only calls `retain_recent()` rather than removing the exhausted peer's bucket, an attacker can bypass the 30 req/sec cap indefinitely by cycling TCP connections to obtain a fresh rate-limit bucket each time.

## Finding Description
**Relayer — rate limiter keyed by session-scoped PeerIndex:** [1](#0-0) [2](#0-1) 

Rate is checked per message in `try_process` using `(peer, message.item_id())` as the key: [3](#0-2) 

On disconnect, only `retain_recent()` is called — the disconnecting peer's specific bucket is **not** removed: [4](#0-3) 

**HolePunching — identical pattern:** [5](#0-4) [6](#0-5) 

**Why existing checks fail — no ban on rate-limit hit:**
`TooManyRequests` is code `110`, which is in the 1xx informational range: [7](#0-6) 

`should_ban()` only bans 4xx codes, so `TooManyRequests` always returns `None`: [8](#0-7) 

The IP-level ban check at connection time is only triggered by explicit `ban_peer` calls, which never happen for `TooManyRequests`: [9](#0-8) 

**Exploit flow:** When a peer disconnects and reconnects, the tentacle P2P layer assigns a new `PeerIndex`. The old rate-limit entry for the previous `PeerIndex` is orphaned (eventually pruned by `retain_recent()`), and the new connection starts with a completely empty bucket. There is no IP-keyed rate-limit state that persists across reconnections.

## Impact Explanation
An attacker can sustain relay message rates far exceeding the intended 30 req/sec cap. With TCP + secio handshake overhead on the order of ~50ms, an attacker can send 30 messages, reconnect, send 30 more — achieving an effective throughput of ~600 messages/sec (20× the intended cap) against a single node. Each message triggers CPU work (deserialization, tx-pool lookups, proposal lookups) and memory allocation. This constitutes a **High** impact matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
Any unprivileged peer that can open inbound TCP connections to the target node can exploit this. The default CKB configuration accepts inbound peers. No special keys, privileges, or hashpower are required. The attacker needs only one IP address and the ability to open and close TCP connections rapidly. The reconnection cost is low (TCP + handshake), making the bypass practical and repeatable indefinitely.

## Recommendation
- Key the rate limiter by the peer's **IP address** (extracted from `connected_addr`/`remote_addr`) rather than by `PeerIndex`. This mirrors how the ban list works and survives reconnection.
- Alternatively, in the `disconnected` handler, explicitly remove the disconnecting peer's entries from the rate limiter hashmap using the known `PeerIndex`, and maintain a separate IP-keyed counter that persists across reconnections.
- Consider escalating repeated rate-limit violations from the same IP to a temporary IP-level ban via `ban_peer`, analogous to how malformed messages are handled.

## Proof of Concept
```
1. Attacker peer (IP: 1.2.3.4) connects → assigned PeerIndex=7
2. Sends 30 GetRelayTransactions/sec → rate_limiter[(7, item_id)] exhausted
   → StatusCode::TooManyRequests(110) returned → no ban_peer call
3. Attacker disconnects and immediately reconnects → assigned PeerIndex=8
4. rate_limiter has no entry for (8, item_id) → fresh bucket
5. Sends 30 more GetRelayTransactions/sec → all accepted ✅
6. Repeat indefinitely → sustained ~600 msg/sec flood vs. intended 30/sec cap
```

Minimal test plan: write an integration test that connects a mock peer, exhausts the rate limiter for `PeerIndex=N`, disconnects, reconnects (obtaining `PeerIndex=N+1`), and asserts that relay messages are accepted again — confirming the bucket reset.
