### Title
Entropy Provider Fee Front-Running Causes DoS and Unrefunded Overpayment — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper` function reads `providerInfo.feeInWei` from live mutable storage at the moment the user's transaction executes. A registered provider can call `setProviderFee()` (or `setProviderFeeAsFeeManager()`) in the same block to change the fee between when the user queries `getFeeV2()` off-chain and when the request transaction lands. There is no user-supplied maximum-fee guard. This creates two exploitable outcomes: (1) a provider-controlled DoS where a fee increase causes the user's transaction to revert with `InsufficientFee`, and (2) a silent fund loss where a fee decrease causes the user to overpay with the excess permanently captured by the protocol rather than refunded.

---

### Finding Description

`requestHelper` is the internal entry point for all `requestV2` / `requestWithCallback` variants:

```solidity
// Entropy.sol lines 233-239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

`getFeeV2` and `getProviderFee` both read `provider.feeInWei` live from storage:

```solidity
// Entropy.sol lines 760-793
function getFeeV2(address provider, uint32 gasLimit) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
function getProviderFee(address providerAddr, uint32 gasLimit) internal view returns (uint128 feeAmount) {
    EntropyStructsV2.ProviderInfo memory provider = _state.providers[providerAddr];
    ...
    return provider.feeInWei;   // live storage read
}
```

Any registered provider can change `feeInWei` at will with no delay or cap:

```solidity
// Entropy.sol lines 810-827
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    ...
}
```

The `IEntropyV2` interface explicitly warns callers that fees can change and that **excess value is not refunded**:

> "Note that the fee can change over time … excess value is *not* refunded to the caller."

No `maxFee` parameter exists in any `requestV2` variant, so users have no on-chain mechanism to bound what they will pay.

**Attack path A — DoS via fee increase:**
1. User calls `getFeeV2(provider, gasLimit)` → receives `F`.
2. User submits `requestV2{value: F}(provider, …)`.
3. Provider front-runs with `setProviderFee(F_new)` where `F_new > F`.
4. When the user's transaction executes, `requiredFee = F_new + pythFee > msg.value` → `revert InsufficientFee`.
5. User loses gas; request never created.

**Attack path B — Silent overpayment via fee decrease:**
1. User calls `getFeeV2(provider, gasLimit)` → receives `F_high`.
2. User submits `requestV2{value: F_high}(provider, …)`.
3. Provider front-runs with `setProviderFee(F_low)` where `F_low < F_high`.
4. `requiredFee = F_low + pythFee < msg.value` → check passes.
5. `providerFee = F_low`; `_state.accruedPythFeesInWei += (msg.value - F_low)`.
6. The delta `F_high − F_low` is silently absorbed into Pyth protocol fees; user is never refunded.

---

### Impact Explanation

**Attack A (DoS):** A malicious provider can grief any user who selects them by watching the mempool and front-running every `requestV2` call. The user's transaction reverts, they lose gas, and they cannot obtain randomness from that provider. Repeated execution permanently blocks the user from using that provider.

**Attack B (fund loss):** A provider who first advertises a high fee, waits for users to submit transactions with that fee, then drops the fee just before those transactions mine, causes users to silently overpay. The excess is captured by the protocol and is unrecoverable by the user. The magnitude equals `(advertised fee − new fee) × number of in-flight requests`.

---

### Likelihood Explanation

Provider registration is permissionless — any address can call `register()`. A malicious actor can register as a provider, attract users (e.g., by advertising a competitive fee), and then execute either attack. `setProviderFee` has no time-lock, no cap, and no cooldown. The front-run requires only a standard higher-gas-price transaction in the same block, which is trivially achievable on any EVM chain. The `IEntropyV2` documentation acknowledges fee mutability but provides no on-chain protection, making this a realistic and repeatable attack vector.

---

### Recommendation

Add a `maxFeeInWei` parameter to `requestV2` (and its variants). Inside `requestHelper`, revert if `requiredFee > maxFeeInWei`:

```solidity
function requestV2(
    address provider,
    bytes32 userRandomNumber,
    uint32 gasLimit,
    uint128 maxFeeInWei   // new parameter
) external payable returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
    if (requiredFee > maxFeeInWei) revert EntropyErrors.FeeTooHigh();
    ...
}
```

This mirrors the fix recommended in the Teller report: take the expected parameter as user input and revert if the live value differs.

Additionally, refund excess `msg.value` to the caller when `msg.value > requiredFee` to eliminate the silent overpayment vector.

### Proof of Concept

```solidity
// Foundry test sketch
function testProviderFeeDoS() public {
    // 1. Provider registers with fee = 1 wei
    vm.prank(provider);
    entropy.register(1, commitment, meta, 1000, uri);

    // 2. User queries fee off-chain
    uint128 fee = entropy.getFeeV2(provider, 0);  // = 1 + pythFee

    // 3. Provider front-runs: raise fee to 1 ether
    vm.prank(provider);
    entropy.setProviderFee(1 ether);

    // 4. User's transaction executes with old fee value
    vm.deal(user, fee);
    vm.prank(user);
    vm.expectRevert(EntropyErrors.InsufficientFee.selector);
    entropy.requestV2{value: fee}(provider, bytes32(0), 0);
    // User lost gas; no randomness obtained.
}

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
