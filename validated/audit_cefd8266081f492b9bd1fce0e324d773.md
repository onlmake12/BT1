### Title
Unbounded `peers` Vec Growth via Intra-Message Duplicate Hashes in `RelayTransactionHashes` — (`sync/src/relayer/transaction_hashes_process.rs`)

### Summary

An unprivileged remote peer can send a single `RelayTransactionHashes` P2P message containing up to `MAX_RELAY_TXS_NUM_PER_BATCH` (32767) identical `Byte32` hashes. Because neither `execute()` nor `add_ask_for_txs()` deduplicates within the input Vec, and because the post-insertion size guard checks only the number of *distinct* keys in `unknown_tx_hashes` (not the size of individual `peers` Vecs), the `UnknownTxHashPriority.peers` Vec for a single hash entry grows to 32767 identical `peer_index` entries per message. Repeated messages with the same hash grow it further without bound.

---

### Finding Description

**Step 1 — `execute()` passes all 32767 identical hashes through** [1](#0-0) 

The count guard rejects only if `len > 32767`, so exactly 32767 hashes pass. The `tx_filter` check is read-only during the iterator: [2](#0-1) 

`TtlFilter::contains()` only reads; it never inserts during this loop. So all 32767 identical hashes pass the filter and are collected into `tx_hashes`. [3](#0-2) 

**Step 2 — `add_ask_for_txs()` calls `push_peer()` 32766 extra times for the same key** [4](#0-3) 

`.take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)` limits the *iteration count* to 32767, not the number of *distinct* keys. With 32767 identical hashes:
- Iteration 1: `Vacant` → creates entry with `peers: vec![peer_index]`
- Iterations 2–32767: `Occupied` → calls `push_peer(peer_index)` 32766 more times [5](#0-4) 

Result: `peers` Vec for that single hash has 32767 identical `peer_index` entries.

**Step 3 — The post-insertion guard is bypassed** [6](#0-5) 

The guard checks `unknown_tx_hashes.len()` — the number of *distinct* keys. With 32767 identical hashes, only **1** distinct key was inserted, so `len() = 1 < MAX_UNKNOWN_TX_HASHES_SIZE (50000)`. The guard never fires.

**Step 4 — Repeated messages grow the Vec unboundedly**

The hash is never added to `tx_filter` until the transaction is actually received and verified via `mark_as_known_txs`. If the hash corresponds to a nonexistent transaction, it stays in `unknown_tx_hashes` indefinitely. Each subsequent message with the same 32767 identical hashes adds another 32767 entries to the same `peers` Vec. [7](#0-6) 

**Constants:** [8](#0-7) 

`MAX_RELAY_TXS_NUM_PER_BATCH = 32767`, `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767`, `MAX_UNKNOWN_TX_HASHES_SIZE = 50000`.

---

### Impact Explanation

1. **Memory exhaustion**: Each message with 32767 identical hashes grows one `peers` Vec by 32767 × `sizeof(PeerIndex)` ≈ 128–256 KB. Across repeated messages (same hash, never resolved), the Vec grows without bound.

2. **Secondary CPU exhaustion**: If the attacker also sends messages with many *distinct* hashes (each message containing 32767 identical copies of a unique hash), they can fill `unknown_tx_hashes` with up to 50000 distinct entries, each with a 32767-entry `peers` Vec. When the post-insertion guard eventually triggers, the `peer_unknown_counter` loop at lines 1517–1523 iterates over all peers in all priorities — potentially 50000 × 32767 ≈ 1.6 billion iterations — causing severe CPU stall under the `unknown_tx_hashes` mutex. [9](#0-8) 

3. **Functional degradation**: `next_request_peer()` uses `swap_remove(0)` to cycle through peers. With 32767 identical entries, the node sends `GetRelayTransactions` to the same peer 32767 times (once per 30-second retry cycle), wasting network and processing resources. [10](#0-9) 

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no privileges, no PoW, no keys. The message is structurally valid (32767 hashes ≤ `MAX_RELAY_TXS_NUM_PER_BATCH`). No rate limiter was observed applied to `RelayTransactionHashes` before `process()` is called. [11](#0-10) 

The attack is local-testable and requires no special network position.

---

### Recommendation

1. **Deduplicate hashes in `execute()`** before passing to `add_ask_for_txs()`:
   ```rust
   let tx_hashes: Vec<_> = { ... }.into_iter().collect::<HashSet<_>>().into_iter().collect();
   ```

2. **Bound `peers` Vec size in `push_peer()`**: cap the Vec at a reasonable limit (e.g., number of connected peers) to prevent unbounded growth regardless of caller behavior.

3. **Check `peers` Vec size in the post-insertion guard** in addition to `unknown_tx_hashes.len()`.

---

### Proof of Concept

```
1. Connect to a CKB node as a relay peer.
2. Construct a RelayTransactionHashes message with tx_hashes = [H; 32767]
   where H is any Byte32 not in the node's tx_filter.
3. Send the message.
4. Inspect unknown_tx_hashes: the entry for H has peers.len() == 32767.
5. Send the same message again (H is still not in tx_filter since no tx was received).
6. peers.len() == 65534. Repeat to exhaust memory.
``` [12](#0-11) [13](#0-12)

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "TxHashes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    relay_transaction_hashes.tx_hashes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }

        // Check `unknown_tx_hashes`'s length after inserting the arrival `tx_hashes`
        if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE
            || unknown_tx_hashes.len()
                >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
        {
            warn!(
                "unknown_tx_hashes is too long, len: {}",
                unknown_tx_hashes.len()
            );

            let mut peer_unknown_counter = 0;
            for (_hash, priority) in unknown_tx_hashes.iter() {
                for peer in priority.peers.iter() {
                    if *peer == peer_index {
                        peer_unknown_counter += 1;
                    }
                }
            }
            if peer_unknown_counter >= MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER {
                return StatusCode::TooManyUnknownTransactions.into();
            }

            return Status::ignored();
        }

        Status::ok()
    }
