Audit Report

## Title
Stale Counter Snapshot Overwrites Post-Eviction State in `PoolMap::add_entry`, Permanently Inflating `total_tx_size`/`total_tx_cycles` тАФ (`File: tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, the aggregate counters `total_tx_size` and `total_tx_cycles` are snapshotted via `updated_stat_for_add_tx` before `check_and_record_ancestors` runs. When that function evicts cell-ref-parent transactions via `remove_entry_and_descendants`, `update_stat_for_remove_tx` correctly decrements the live counters. However, `add_entry` then unconditionally overwrites those corrected values with the stale pre-eviction snapshot, permanently inflating both counters by the total size and cycles of every evicted entry. An unprivileged attacker can exploit this repeatedly to make the pool appear full, causing `limit_size` to evict legitimate transactions and eventually rendering the node's tx-pool unable to accept any new submissions.

## Finding Description
The exact execution sequence in `add_entry` (lines 200тАУ221):

```
L210-211: (total_tx_size, total_tx_cycles) = updated_stat_for_add_tx(entry.size, entry.cycles)
          // snapshot = self.total_tx_size + new_size, self.total_tx_cycles + new_cycles

L213:     evicts = check_and_record_ancestors(&mut entry)
          //  тЖ│ line 618: remove_entry_and_descendants(next_id)
          //    тЖ│ line 247: update_stat_for_remove_tx(size, cycles)
          //      тЖ│ lines 738-740: self.total_tx_size -= evicted_size  тЖР CORRECT

L218-219: self.total_tx_size  = total_tx_size   // OVERWRITES the corrected value
          self.total_tx_cycles = total_tx_cycles  // OVERWRITES the corrected value
```

`check_and_record_ancestors` (lines 603тАУ625) fires the eviction branch when:
- `ancestors_count > max_ancestors_count` (default 25), **and**
- `ancestors_count тИТ cell_ref_parents.len() тЙд max_ancestors_count`

Under this condition it calls `remove_entry_and_descendants` (line 618), which calls `remove_entry` (line 263), which calls `update_stat_for_remove_tx` (line 247) for every removed entry. Those decrements to `self.total_tx_size` and `self.total_tx_cycles` are then silently discarded when lines 218тАУ219 assign the stale snapshot.

**Net effect per triggering call:**

| Counter | Correct value | Actual value |
|---|---|---|
| `total_tx_size` | `old + new_size тИТ evicted_sizes` | `old + new_size` |
| `total_tx_cycles` | `old + new_cycles тИТ evicted_cycles` | `old + new_cycles` |

The inflation is permanent and cumulative.

## Impact Explanation
`total_tx_size` is the sole gate for pool-full eviction in `limit_size` (pool.rs, line 298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter causes this loop to evict legitimate, already-admitted transactions unnecessarily. Once the inflated counter exceeds `max_tx_pool_size` (180 MB by default) and the pool is drained empty, `next_evict_entry()` returns `None` on every iteration, the `if let Some(id)` branch never executes, and the while condition remains permanently true тАФ producing an **infinite loop that hangs the node process**. This matches the allowed impact: **High тАФ Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The eviction path is reachable by any unprivileged `send_transaction` RPC caller with no special privileges. The attacker constructs a 25-deep input chain (T0тЖТтАжтЖТT24), then submits T25 spending T24's output with T0's output as a `cell_dep`. This satisfies the eviction condition at line 603 with `ancestors_count = 26`, `cell_ref_parents.len() = 1`. After each iteration, T0тАУT24 are removed from the pool, returning the attacker's UTXOs to their pre-T0 state, so the same UTXO set can be reused immediately. Each iteration inflates the counter by approximately `25 * avg_tx_size`. At minimum fee rate (1,000 shannons/KB) and ~200-byte transactions, reaching 180 MB requires roughly 36,000 iterations costing on the order of a few CKB total. No majority hashpower, Sybil attack, or privileged access is required.

## Recommendation
Move the counter assignment to **after** `check_and_record_ancestors` returns, computing the new totals from the already-updated `self.total_tx_size`/`self.total_tx_cycles`:

```rust
// In add_entry, replace lines 210-211 and 218-219:
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, subtract the evicted entries' sizes/cycles from the pre-computed snapshot before assigning. Additionally, `limit_size` should add a `break` or early exit when `next_evict_entry()` returns `None` to prevent the infinite loop that results from an inflated counter with an empty pool.

## Proof of Concept
**Setup** (default `max_ancestors_count = 25`, `max_tx_pool_size = 180_000_000`):

1. Submit transactions T0тЖТT1тЖТтАжтЖТT24 (25-deep input chain). All admitted; `total_tx_size = 25 * S`.
2. Submit T25 whose input spends T24's output and whose `cell_deps` includes T0's output cell. `ancestors_count = 26 > 25`; `26 тИТ 1 = 25 тЙд 25` тЖТ eviction branch at line 603 fires.
3. `remove_entry_and_descendants(T0)` removes T0 and descendants T1тАУT24 (25 entries). `update_stat_for_remove_tx` correctly sets `self.total_tx_size = 0`.
4. Line 218 assigns `self.total_tx_size = 25*S + S_T25` (stale snapshot). Pool holds only T25 but counter reads `25*S + S_T25`.
5. Repeat from step 1 (T0тАУT24 UTXOs are back in the unspent set). Each iteration adds `~25*S` to the counter.
6. After ~`180_000_000 / (25 * S)` iterations, `total_tx_size тЙе max_tx_pool_size`. `limit_size` evicts all remaining entries. Pool is empty but counter remains inflated. `limit_size` enters an infinite loop, hanging the node.
