- `priceLastUpdatedAt` is set to the Pythnet `publishTime` (line 340)
- The **heartbeat trigger** (`_validateShouldUpdatePrices`, lines 401–409) measures elapsed time purely in Pythnet time: `updateTimestamp >= lastUpdateTime + heartbeatSeconds`

### Title
Inconsistent Time Bases Between Heartbeat Trigger and Staleness Check Causes Premature `StalePrice` Reverts — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
`Scheduler.sol` stores `priceLastUpdatedAt` as a Pythnet `publishTime` and measures the heartbeat interval in Pythnet time, but `getPricesNoOlderThan` / `getEmaPricesNoOlderThan` check freshness by comparing `block.timestamp` (EVM chain time) against `priceLastUpdatedAt` (Pythnet time). Because Pythnet time and EVM chain time can diverge, prices appear stale to readers before the keeper's next heartbeat update is actually due, causing unexpected `StalePrice` reverts in downstream protocols.

### Finding Description

`priceLastUpdatedAt` is set to the Pythnet `publishTime` of the most recent update batch:

```solidity
// Scheduler.sol line 340
status.priceLastUpdatedAt = latestPublishTime;
``` [1](#0-0) 

The heartbeat trigger in `_validateShouldUpdatePrices` measures elapsed time entirely in Pythnet time — both `updateTimestamp` and `lastUpdateTime` are Pythnet `publishTime` values:

```solidity
// Scheduler.sol lines 401-409
uint256 lastUpdateTime = status.priceLastUpdatedAt;
if (
    lastUpdateTime == 0 ||
    updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds
) {
    return updateTimestamp;
}
``` [2](#0-1) 

The staleness check in `getPricesNoOlderThan` (and `getEmaPricesNoOlderThan`) measures elapsed time as the distance between `block.timestamp` (EVM chain time) and `priceLastUpdatedAt` (Pythnet time):

```solidity
// Scheduler.sol lines 551-552
if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();
``` [3](#0-2) 

The same inconsistency exists in `getEmaPricesNoOlderThan`: [4](#0-3) 

**Concrete scenario** (let `delta` = divergence between EVM chain time and Pythnet time, EVM ahead):

| Event | Pythnet time | EVM `block.timestamp` |
|---|---|---|
| Last update stored | `T_p` | `T_e = T_p + delta` |
| `getPricesNoOlderThan(H)` starts reverting | — | `T_e + (H - delta)` |
| Next heartbeat update due | `T_p + H` | `T_e + H` (approx.) |

There is a window of `delta` seconds where `getPricesNoOlderThan(heartbeatSeconds)` reverts with `StalePrice` even though the keeper has not missed a heartbeat. A subscription manager who sets `age_seconds = heartbeatSeconds` — the natural and expected configuration — will experience this gap.

### Impact Explanation

Any whitelisted reader contract calling `getPricesNoOlderThan(subscriptionId, priceIds, heartbeatSeconds)` will receive unexpected `StalePrice` reverts during the `delta`-second window before each heartbeat update. Downstream DeFi protocols (e.g., lending protocols using Scheduler prices for collateral valuation or liquidation triggers) will fail to execute critical operations during this window. The subscription manager has no way to query the exact "safe" `age_seconds` value to avoid this, because the divergence between Pythnet time and EVM chain time is not exposed by the contract.

### Likelihood Explanation

Pythnet time and EVM chain time are generally close but not identical. Divergences of several seconds to a few minutes are realistic, especially during periods of network congestion or Pythnet slot timing differences. The impact scales with the divergence: for a 60-second heartbeat and a 10-second divergence, prices appear stale for 10 out of every 60 seconds — a 17% failure rate for time-sensitive operations. Any unprivileged whitelisted reader can trigger this path by calling `getPricesNoOlderThan`.

### Recommendation

Align the two time bases. Either:

1. **Store `priceLastUpdatedAt` as `block.timestamp`** (EVM chain time) instead of Pythnet `publishTime`, and adjust the heartbeat check to compare `block.timestamp` against `priceLastUpdatedAt + heartbeatSeconds`. This makes both the heartbeat trigger and the staleness check use EVM chain time.

2. **Alternatively**, expose a `getNextHeartbeatDue(subscriptionId)` view function that returns `priceLastUpdatedAt + heartbeatSeconds` in Pythnet time, and document that `age_seconds` in `getPricesNoOlderThan` must account for the Pythnet/EVM clock divergence.

Option 1 is simpler and eliminates the inconsistency entirely.

### Proof of Concept

1. Create a subscription with `heartbeatSeconds = 60`.
2. Keeper submits an update with Pythnet `publishTime = T_p`. `priceLastUpdatedAt` is set to `T_p`. Assume `block.timestamp = T_p + 30` (EVM is 30 seconds ahead of Pythnet).
3. 31 seconds later in EVM time: `block.timestamp = T_p + 61`. Call `getPricesNoOlderThan(subscriptionId, priceIds, 60)`.
4. `distance(T_p + 61, T_p) = 61 > 60` → reverts with `StalePrice`.
5. But the heartbeat is not due yet: the next update requires a new Pythnet `publishTime >= T_p + 60`. The keeper cannot submit such an update until Pythnet time reaches `T_p + 60`, which in EVM time is approximately `T_p + 60 + 30 = T_p + 90`.
6. Result: `getPricesNoOlderThan(60)` reverts for ~29 seconds before the keeper can satisfy the heartbeat condition.
