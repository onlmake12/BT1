### Title
Silent Callback Success on Non-Existent Requester in `revealWithCallback` New Gas-Limit Path — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `revealWithCallback` function contains two distinct callback execution paths. The **new path** (triggered when `req.gasLimit10k != 0`) dispatches the callback via `excessivelySafeCall` — a low-level assembly call — **without any prior contract-existence check**. Per EVM specification, a low-level `call` to a non-existent or self-destructed address returns `success = true` with empty return data. This causes the protocol to treat the callback as successfully delivered, permanently clear the request, and emit `RevealedWithCallback`, even though the requester never received the random number. The **old path** (lines 661–702) correctly guards against this with an explicit `extcodesize` check, making the omission in the new path a clear regression.

---

### Finding Description

`revealWithCallback` in `Entropy.sol` branches on whether `req.gasLimit10k != 0`:

**New path (lines 574–660):**
```solidity
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND,
    256,
    abi.encodeWithSelector(
        IEntropyConsumer._entropyCallback.selector,
        sequenceNumber,
        provider,
        randomNumber
    )
);
``` [1](#0-0) 

There is **no `extcodesize` check** before this call. `ExcessivelySafeCall` is a thin assembly wrapper around the EVM `CALL` opcode. The EVM specification states: *"The low-level functions call, delegatecall and staticcall return true as their first return value if the called account is non-existent."* Consequently, if `req.requester` is an EOA or a self-destructed contract, `success` is `true`, `ret` is empty, and the code proceeds to:

```solidity
if (success) {
    emit RevealedWithCallback(...);
    emit EntropyEventsV2.Revealed(..., false, ret, ...);
    clearRequest(provider, sequenceNumber);
}
``` [2](#0-1) 

The request is permanently cleared and both events are emitted with `callbackFailed = false`, signalling a successful delivery that never occurred.

**Old path (lines 661–702)** — correctly guarded:
```solidity
uint len;
assembly {
    len := extcodesize(callAddress)
}
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(...);
}
``` [3](#0-2) 

The old path only invokes the callback when the requester is a live contract. The new path has no equivalent guard, creating an inconsistency that silently swallows the callback for non-contract requesters.

---

### Impact Explanation

When `req.requester` has no deployed bytecode (EOA or self-destructed contract) and `req.gasLimit10k != 0`:

1. `excessivelySafeCall` returns `(true, "")`.
2. The protocol emits `RevealedWithCallback` and `EntropyEventsV2.Revealed` with `callbackFailed = false`.
3. `clearRequest` permanently deletes the in-flight request.
4. The random number is generated and consumed but **never delivered** to the requester.
5. The requester's paid fees are non-refundable; the provider accrues their fee as if the service was rendered.

Off-chain indexers, provider dashboards, and user-facing UIs that rely on the `RevealedWithCallback` event will report the request as successfully fulfilled, misleading all observers. There is no recovery path: the request is gone, the random number is gone, and the `CALLBACK_FAILED` retry flow is never entered.

---

### Likelihood Explanation

Two realistic triggering scenarios exist, both reachable by an unprivileged Entropy user:

1. **EOA caller with gas limit**: Any EOA can call `requestV2(provider, gasLimit)` with a non-zero `gasLimit`. This is a valid public entry point with no contract-type enforcement on `msg.sender`. The provider's subsequent `revealWithCallback` will silently succeed. [4](#0-3) 

2. **Self-destructed consumer contract**: A contract that called `requestV2` with a gas limit and was later self-destructed (e.g., an upgradeable proxy that was bricked, or a contract with an intentional `selfdestruct`) will have its pending request silently cleared on the next `revealWithCallback` call.

Scenario 1 requires no special privileges and is directly reachable by any transaction sender.

---

### Recommendation

Add an `extcodesize` guard in the new gas-limit path, mirroring the existing check in the old path, before invoking `excessivelySafeCall`:

```solidity
// Check if the requester is a contract account.
uint len;
assembly { len := extcodesize(req.requester) }
if (len == 0) {
    // Requester has no code; treat as a failed callback or revert.
    revert EntropyErrors.InvalidRevealCall();
}
(success, ret) = req.requester.excessivelySafeCall(...);
```

Alternatively, align both paths by moving the `extcodesize` check to a shared pre-condition at the top of `revealWithCallback`, before the branch.

---

### Proof of Concept

1. EOA `Alice` calls `requestV2(defaultProvider, 50_000)` paying the required fee. `req.gasLimit10k = 5` (non-zero), `req.requester = Alice` (EOA, `extcodesize == 0`). [5](#0-4) 

2. Provider calls `revealWithCallback(provider, seqNum, userContrib, providerContrib)`.

3. Branch condition `req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED` is true → new path is taken. [6](#0-5) 

4. `req.requester.excessivelySafeCall(...)` is called on Alice's EOA address. EVM returns `(true, "")` because the account has no code. [1](#0-0) 

5. `success == true` → `RevealedWithCallback` and `EntropyEventsV2.Revealed(... callbackFailed=false ...)` are emitted; `clearRequest` is called. [2](#0-1) 

6. Alice's random number is permanently lost. The protocol reports success. No retry is possible.
