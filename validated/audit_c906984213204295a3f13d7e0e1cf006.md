Audit Report

## Title
No Per-Peer Admission Limit in `OrphanPool` Enables Single-Peer Monopolization via Random Eviction - (`tx-pool/src/component/orphan.rs`)

## Summary
`OrphanPool::add_orphan_tx` imposes no per-peer quota before inserting an orphan transaction. A single remote peer can fill all 100 slots of the orphan pool with crafted transactions referencing non-existent inputs. Every subsequent orphan submission from any legitimate peer triggers random eviction via `limit_size`, which evicts the victim's entry with ~99% probability, permanently starving CPFP chains from the pool.

## Finding Description
`DEFAULT_MAX_ORPHAN_TRANSACTIONS` is 100 and `ORPHAN_TX_EXPIRE_TIME` is `100 * MAX_BLOCK_INTERVAL` — a very long window before natural expiry. [1](#0-0)  `add_orphan_tx` inserts any structurally valid orphan transaction without checking how many entries the submitting peer already holds. The `Entry` struct records `peer: PeerIndex` [2](#0-1)  but this field is never consulted for admission control — only stored. When the pool exceeds 100 entries, `limit_size` evicts by calling `self.entries.keys().next()` on a `HashMap`, whose iteration order is non-deterministic (Rust's `HashMap` uses a random seed via `RandomState`), making eviction effectively random with no per-peer fairness. [3](#0-2)  The full P2P entry path — `after_process` → `add_orphan` → `add_orphan_tx` — performs no per-peer quota check at any layer. [4](#0-3) [5](#0-4)  The orphan path is triggered whenever `is_missing_input(reject)` is true, which occurs for any transaction whose inputs are not yet in the UTXO set. An attacker crafts transactions with random `OutPoint`s as inputs: these pass `non_contextual_verify` (structural checks only, no input resolution) and fail resolution with a missing-input error, landing in the orphan pool. The only guard in `add_orphan_tx` is a duplicate-key check (`contains_key`), which does not limit per-peer count. [6](#0-5) 

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** An attacker connecting to multiple CKB nodes simultaneously (each requiring only a standard P2P connection and 100 minimal orphan transactions) can monopolize the orphan pool on each node. This permanently prevents CPFP chains from being promoted to the pending pool when their parent confirms, disrupting fee-bump transaction propagation across the network. The attack can be applied to all reachable nodes at negligible cost, constituting network-wide congestion of the orphan-resolution mechanism. [7](#0-6) 

## Likelihood Explanation
The attack requires only a standard P2P connection and the ability to craft structurally valid CKB transactions with fabricated input `OutPoint`s — no keys, no PoW, no privileged access. The cost is 100 minimal-size transactions per target node. The long `ORPHAN_TX_EXPIRE_TIME` (`100 * MAX_BLOCK_INTERVAL`) means the attacker does not need to continuously re-submit; slots remain occupied for an extended period. The attacker can re-fill any slot that is accidentally evicted. This is concretely reachable on mainnet by any node operator. [7](#0-6) 

## Recommendation
Add a per-peer cap inside `add_orphan_tx`. Before inserting, count how many entries in `self.entries` already belong to `peer` and reject (or evict the oldest from that peer) if the count exceeds `DEFAULT_MAX_ORPHAN_TRANSACTIONS / MAX_PEERS_PER_ORPHAN_SLOT` (e.g., 10–25). A more robust alternative is to change `limit_size` eviction to prefer evicting from the peer with the most entries rather than evicting at random: iterate `self.entries`, find the peer with the highest entry count, and evict one of their entries. This provides per-peer fairness without requiring a separate counter structure. [8](#0-7) 

## Proof of Concept
1. Connect peer A to a CKB node via standard P2P.
2. Peer A submits 100 orphan transactions: each is a structurally valid CKB transaction whose inputs reference non-existent `OutPoint`s (random 32-byte tx-hash + index 0). Each passes `non_contextual_verify` but fails input resolution with `is_missing_input`.
3. After 100 submissions, `OrphanPool::len()` == 100 (pool is full; `limit_size` does not evict since `len > 100` is false).
4. Connect peer B. Peer B submits 1 orphan transaction (same construction, different tx hash).
5. `add_orphan_tx` inserts it → `len()` == 101 → `limit_size` evicts one entry via `self.entries.keys().next()`. [3](#0-2) 
6. Assert: peer B's tx hash appears in the `evicted_txs` return value with ~99% probability (100/101); all 100 of peer A's txs remain. Verify by inspecting the `peer` field of remaining entries. [9](#0-8) 
7. Repeat steps 4–6 to confirm peer B is permanently starved.

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
