Audit Report

## Title
`check_tx_fee` Minimum Fee Rate Admission Uses Serialized Size Only, Allowing Cycle-Heavy Transactions to Bypass the Threshold - (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, while the actual transaction weight used for pool ranking and eviction is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, these two values diverge by up to ~120×, allowing transactions with actual fee rates far below the configured `min_fee_rate` to pass the admission gate. No second fee-rate check is performed after `verify_rtx` returns the actual cycle count.

## Finding Description
`check_tx_fee` in `tx-pool/src/util.rs` (lines 42–52) explicitly uses only `tx_size` as the weight denominator, with a comment acknowledging this is intentional as a "cheap check":

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The actual transaction weight is defined in `util/types/src/core/tx_pool.rs` (lines 298–303) as `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`, where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`: [2](#0-1) 

In `_process_tx` (`tx-pool/src/process.rs`, lines 705–777), the flow is:
1. `pre_check` → calls `check_tx_fee` with `tx_size` only (lines 715–717)
2. `verify_rtx` → returns actual cycles (lines 724–734)
3. Declared-vs-verified cycle check for remote txs (lines 736–749)
4. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with actual cycles (line 751)
5. **No second fee-rate check using actual weight is performed** [3](#0-2) 

Once inside the pool, `TxEntry::fee_rate()` and the sort/evict keys correctly use `get_transaction_weight(size, cycles)`: [4](#0-3) [5](#0-4) 

The eviction key (`EvictKey`) also uses actual weight, so cycle-heavy low-fee-rate transactions are evicted first when the pool fills up. However, this is a reactive mechanism — the transactions have already been admitted and occupy pool space until eviction occurs.

**Concrete arithmetic** (with `min_fee_rate = 1,000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):
- Size-based gate for a 200-byte tx: `1,000 × 200 / 1,000 = 200 shannons` → **passes**
- Actual weight: `max(200, 70,000,000 × 0.000_170_571_4) ≈ 11,940`
- Actual fee rate: `200 × 1,000 / 11,940 ≈ 16 shannons/KW` — ~62× below threshold

## Impact Explanation
An unprivileged attacker can submit cycle-heavy, byte-small transactions that pass the admission check while carrying actual fee rates far below the configured `min_fee_rate`. These transactions enter the pool legitimately, are ranked near the bottom of the fee-rate queue (deprioritized by miners), and occupy pool memory. While the eviction mechanism using actual weight does evict them first when the pool fills, the attacker can continuously resubmit, causing sustained pool churn and degrading pool quality. This matches the allowed impact: **High — bad design which could cause CKB network congestion with few costs**, since the per-transaction fee cost is extremely low (100–200 shannons).

## Likelihood Explanation
The attack requires only a valid RPC connection or peer relay path. The attacker needs UTXOs to spend (a real but modest cost) and a deployed high-cycle script (one-time setup). The per-transaction fee is ~62–125× below the configured threshold. A determined attacker with modest CKB holdings can sustain continuous pool churn. No privileged access, key material, or majority hashpower is required.

## Recommendation
After `verify_rtx` returns the actual cycle count, re-run the fee-rate check using the true weight before creating the `TxEntry`. Insert the following in `_process_tx` after line 734 (after `verify_rtx` succeeds) and before line 751 (`TxEntry::new`):

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Some((Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        actual_min_fee.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

## Proof of Concept
1. Deploy a type script on-chain that consumes close to `max_tx_verify_cycles` (70 M) cycles.
2. Craft a transaction with:
   - Serialized size ≈ 200 bytes (one input, one output, small lock script)
   - The deployed high-cycle type script referenced via cell dep (not serialized in tx body)
   - Fee = `1,000 × 200 / 1,000 = 200 shannons`
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes `min_fee = fee(200) = 200 shannons`. Fee ≥ min_fee → **accepted**.
5. After `verify_rtx`, actual cycles ≈ 70,000,000. Actual weight ≈ 11,940. Actual fee rate ≈ 16 shannons/KW — 62× below the 1,000 shannons/KW threshold.
6. Transaction sits in the pool ranked near the bottom. Repeat with additional UTXOs to sustain pool churn.
7. Verification: inspect pool entries via `get_pool_info` RPC; observe transactions with fee rates far below `min_fee_rate`.

pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

/// vbytes has been deprecated, renamed to weight to prevent ambiguity
#[deprecated(
    since = "0.107.0",
    note = "Please use the get_transaction_weight instead"
)]
pub fn get_transaction_virtual_bytes(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}

/// The miners select transactions to fill the limited block space which gives the highest fee.
/// Because there are two different limits, serialized size and consumed cycles,
/// the selection algorithm is a multi-dimensional knapsack problem.
/// Introducing the transaction weight converts the multi-dimensional knapsack to a typical knapsack problem,
/// which has a simple greedy algorithm.
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```
