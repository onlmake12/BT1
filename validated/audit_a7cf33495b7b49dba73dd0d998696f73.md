Audit Report

## Title
Non-Deterministic `CURRENT_CYCLES` Syscall Due to `iteration_cycles` Loss on Scheduler Resume — (`script/src/scheduler.rs`, `script/src/syscalls/current_cycles.rs`)

## Summary
`CURRENT_CYCLES` (syscall 2042) returns `total_cycles + machine.cycles()`, never including `iteration_cycles`. After each `iterate_outer` call, `process_io()` charges `SPAWN_EXTRA_CYCLES_BASE` into `iteration_cycles` via `resume_vm()`/`suspend_vm()`. When the outer scheduler is then suspended and resumed, `Scheduler::resume()` unconditionally zeros `iteration_cycles` at line 242, permanently discarding those charges. Nodes with different `step_cycles` values suspend at different iteration boundaries, losing different cumulative amounts, so `CURRENT_CYCLES` returns divergent values for the same script execution — a consensus deviation.

## Finding Description

**`CURRENT_CYCLES` omits `iteration_cycles`:**

`CurrentCycles::ecall` computes only `base_cycles.load() + machine.cycles()`: [1](#0-0) 

`base_cycles` is an `Arc` clone of `self.total_cycles` wired at VM creation: [2](#0-1) 

`iteration_cycles` is never included in this computation.

**How `iteration_cycles` becomes non-zero after `iterate_outer`:**

`iterate_outer` calls `consume_cycles(self.iteration_cycles)` (moving pending charges into `total_cycles`), resets `iteration_cycles = 0`, then calls `process_io()`: [3](#0-2) 

Inside `process_io()`, `ensure_vms_instantiated()` is called for read/write VM pairs: [4](#0-3) 

`resume_vm()` and `suspend_vm()` each add `SPAWN_EXTRA_CYCLES_BASE` directly to `iteration_cycles`: [5](#0-4) [6](#0-5) 

After `process_io()` returns, `iteration_cycles` is non-zero and has **not** yet been moved to `total_cycles`.

**`suspend()` preserves the non-zero `iteration_cycles`:** [7](#0-6) 

**`resume()` unconditionally zeros it:** [8](#0-7) 

The comment justifies this as not charging scheduler overhead, but it also discards the `SPAWN_EXTRA_CYCLES_BASE` charges accumulated by `process_io()` in the *previous* iteration — charges that are not scheduler overhead and that would have been moved to `total_cycles` at the next `consume_cycles()` call on a non-suspending node.

**The concrete divergence:**

- **Node A** (large `step_cycles`, no suspension between iteration N and N+1): `SPAWN_EXTRA_CYCLES_BASE` charges from `process_io()` at iteration N flow into `total_cycles` at iteration N+1's `consume_cycles(self.iteration_cycles)` call (line 423). `CURRENT_CYCLES` in iteration N+1 reflects them.
- **Node B** (small `step_cycles`, suspends after iteration N): those same charges are in `iteration_cycles` at suspension time, preserved in `FullSuspendedState`, then zeroed at `resume()` line 242. They are permanently absent from `total_cycles`. `CURRENT_CYCLES` in iteration N+1 returns a smaller value by exactly the lost `SPAWN_EXTRA_CYCLES_BASE` amount.

The codebase explicitly acknowledges this bug class: [9](#0-8) 

However, the comment describes a single consistent behavior, whereas the actual behavior varies with `step_cycles` — a per-node configuration parameter, not a consensus parameter — making the divergence real and exploitable.

## Impact Explanation

A crafted V2 script that spawns a child VM over a pipe, performs a pipe read/write (triggering `ensure_vms_instantiated` inside `process_io()`, charging `SPAWN_EXTRA_CYCLES_BASE` into `iteration_cycles`), then calls `CURRENT_CYCLES` and branches on the result will produce exit code 0 on nodes that did not suspend after the relevant iteration and exit code 1 on nodes that did. Two honest nodes producing different exit codes for the same transaction constitutes a **consensus deviation**, matching the Critical impact class: "Vulnerabilities which could easily cause consensus deviation."

## Likelihood Explanation

Moderate. Script deployment on CKB is permissionless — no special privilege is required. The attacker deploys a crafted lock or type script and submits a transaction using it. `step_cycles` legitimately differs between nodes as a node-level tuning parameter, so the precondition is always satisfied on a real network. The bug is intentionally preserved for the Meepo hardfork per the source comments, reducing the probability of an immediate patch but confirming developer awareness of the issue class.

## Recommendation

In `Scheduler::resume()`, instead of unconditionally zeroing `iteration_cycles`, first flush the saved `iteration_cycles` value into `total_cycles` before zeroing, so that IO-processing charges from the previous iteration are not lost:

```rust
// Flush IO-processing charges that were pending at suspension time
// into total_cycles before zeroing, so they are not lost.
self.consume_cycles(full.iteration_cycles).unwrap();
// Zero iteration_cycles so that the scheduler-resume overhead added
// by ensure_vms_instantiated below is still not charged.
scheduler.iteration_cycles = 0;
```

Alternatively, include `iteration_cycles` in the value returned by `CURRENT_CYCLES` (i.e., `total_cycles + iteration_cycles + machine.cycles()`). Either fix should be scheduled for the hardfork that supersedes Meepo, with the comment at line 242 updated to distinguish outer-scheduler overhead (legitimately uncharged) from IO-processing charges (which must be preserved for determinism).

## Proof of Concept

Differential unit test using the `Scheduler` API directly:

1. Deploy a V2 lock script that: (a) spawns a child VM sharing a pipe, (b) transfers N bytes over the pipe (causing `ensure_vms_instantiated` inside `process_io()` to call `resume_vm`/`suspend_vm`, adding `SPAWN_EXTRA_CYCLES_BASE` to `iteration_cycles`), (c) calls `CURRENT_CYCLES` (syscall 2042), (d) exits 0 if result > `THRESHOLD`, else exits 1. Set `THRESHOLD` between the two expected cycle counts (differing by `SPAWN_EXTRA_CYCLES_BASE` per suspend/resume pair).
2. Verify the same transaction twice using `TransactionScriptsVerifier`:
   - **Run A**: `step_cycles = u64::MAX` (no outer-scheduler suspension between iterations).
   - **Run B**: `step_cycles = 1` (suspend after every iteration).
3. Assert both runs return the same exit code — the assertion will fail, demonstrating the consensus deviation.
