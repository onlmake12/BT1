### Title
Unprivileged Front-Run of `initialize()` with Zero-Byte Authority Permanently Bricks Publisher Registration — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

---

### Summary

`initialize()` has no access control on the caller and no validation on the `authority` argument. Any account can call it first and store `[0u8;32]` (the system program pubkey) as the authority. Because the config PDA can only be initialized once and the system program can never sign a transaction, `initialize_publisher` is permanently bricked.

---

### Finding Description

`validate_payer` only checks that the payer is a signer and writable — it does **not** verify the payer is a specific privileged deployer: [1](#0-0) 

`InitializeArgs.authority` is a raw `[u8; 32]` passed directly into `accounts::config::create()` with no non-zero or validity check: [2](#0-1) 

`accounts::config::create()` writes whatever bytes are supplied, with no guard against the all-zeros pubkey: [3](#0-2) 

Once written, the config is permanently sealed — a second call to `create()` returns `AlreadyInitialized`: [4](#0-3) 

`validate_authority`, called by every `initialize_publisher` invocation, requires the signer's key to equal `config.authority`. If that value is `[0u8;32]`, the system program must sign — which is impossible: [5](#0-4) 

---

### Impact Explanation

- The config PDA is a singleton; once created it cannot be replaced or upgraded by any existing instruction.
- With `authority=[0u8;32]`, `validate_authority` will always return `MissingRequiredSignature` for every real signer.
- `initialize_publisher` is the only path to register a publisher; it is permanently blocked.
- No publisher can ever be registered → no prices can be published → **permanent protocol-level freeze of the price-store program**.

---

### Likelihood Explanation

- The program is open-source and the `Initialize` instruction is public with no privileged guard.
- An attacker only needs to submit one transaction after program deployment and before the legitimate deployer.
- On Solana, program deployment and initialization are separate transactions, creating a race window.
- The attacker pays only rent (~0.001 SOL) and a transaction fee.
- No special knowledge, keys, or permissions are required.

---

### Recommendation

1. **Validate `authority` is non-zero** inside `accounts::config::create()` or `initialize()`:
   ```rust
   ensure!(ProgramError::InvalidArgument, args.authority != [0u8; 32]);
   ```
2. **Restrict who can call `initialize`**: require the payer to be a known upgrade authority or embed the expected authority pubkey as a program constant, then verify `payer.key == EXPECTED_AUTHORITY`.
3. **Add an `update_authority` instruction** (gated on the current authority signing) so a misconfigured authority can be corrected without redeployment.

---

pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)

pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);
    }
    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
    if data.format != 0 {
        return Err(ReadAccountError::AlreadyInitialized);
    }
    data.format = FORMAT;
    data.authority = authority;
    Ok(data)
