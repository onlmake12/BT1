### Title
Unauthenticated `observed_addr` Injection via Identify Protocol Poisons Victim's Advertised Listen Addresses — (`network/src/protocols/identify/mod.rs`)

### Summary

An unprivileged remote peer can send an `IdentifyMessage` with an arbitrary `observed_addr`, which the victim stores without validation and subsequently re-broadcasts to all other peers as part of its own listen addresses.

### Finding Description

**Step 1 — Attacker-controlled entry point:**

In `process_observed`, the `observed_addr` field from the incoming `IdentifyMessage` is passed directly to `add_observed_addr` with no validation of whether the address matches the actual connection source: [1](#0-0) 

The function always returns `MisbehaveResult::Continue` — there is no rejection path for a bogus address.

**Step 2 — Unconditional storage:**

`IdentifyCallback::add_observed_addr` only appends the local peer ID if missing, then stores the attacker-supplied address directly into `NetworkState::observed_addrs`: [2](#0-1) 

No check is made that the address matches the actual TCP source of the session.

**Step 3 — Poisoned address included in outbound Identify messages:**

`local_listen_addrs` extends the victim's public address list with `observed_addrs` whenever fewer than `MAX_RETURN_LISTEN_ADDRS` (10) public addresses are configured: [3](#0-2) 

**Step 4 — Broadcast to third peers:**

When the victim connects to any new peer, it sends an `IdentifyMessage` whose `listen_addrs` is built from `local_listen_addrs()`, filtered only by `is_reachable` (which passes any globally routable IP like `1.2.3.4`): [4](#0-3) 

The attacker-injected address passes this filter and is included in the victim's outbound Identify message to every subsequent peer.

**Missing guard:** The `observed_addrs` field in `NetworkState` is a `HashMap<PeerIndex, Multiaddr>`: [5](#0-4) 

There is no cross-check against the session's actual remote address at any point in the pipeline.

### Impact Explanation

- Any node that connects to the victim will receive the attacker-injected address as one of the victim's listen addresses.
- Third-party nodes will waste connection attempts dialing the bogus address.
- An attacker can redirect traffic intended for the victim to an arbitrary IP:port (e.g., a competitor node or a sink), degrading the victim's effective reachability across the network.
- The attack is persistent for the duration of the attacker's session and affects every peer the victim connects to afterward.

### Likelihood Explanation

- Requires only a standard P2P connection — no privileges, no keys, no hashpower.
- The Identify protocol is the first protocol opened on every connection.
- The attack is trivially reproducible with a minimal custom P2P client.
- Most nodes will have fewer than 10 configured `public_addrs`, making the `observed_addrs` extension path active by default.

### Recommendation

In `process_observed` (or in `IdentifyCallback::add_observed_addr`), validate that the IP portion of the received `observed_addr` matches the actual remote address of the session (`info.session.address`). Only accept the observed address if the IP matches the connection source. Additionally, consider requiring a minimum number of independent peers to report the same observed address before accepting it (consensus-based NAT detection, as used by Bitcoin Core and libp2p).

### Proof of Concept

1. Connect to victim node and complete the Identify handshake.
2. Send `IdentifyMessage { observed_addr: /ip4/1.2.3.4/tcp/9999/p2p/<victim_peer_id>, listen_addrs: [...], identify: <valid_identify_bytes> }`.
3. Maintain the connection.
4. Have a third node connect to the victim.
5. Observe that the victim's Identify message to the third node includes `/ip4/1.2.3.4/tcp/9999/p2p/<victim_peer_id>` in its `listen_addrs`.

    fn local_listen_addrs(&mut self) -> Vec<Multiaddr> {
        let mut listens = self.listen_addrs();

        if listens.len() < MAX_RETURN_LISTEN_ADDRS {
            let observe_addrs = self
                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
            listens.extend(observe_addrs);
            listens
        } else {
            listens
        }
    }
