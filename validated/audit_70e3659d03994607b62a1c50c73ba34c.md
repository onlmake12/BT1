Audit Report

## Title
Unbounded `forward_rate_limiter` `HashMapStateStore` Growth via Persistent Connection — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
`HolePunching::forward_rate_limiter` is a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`. The `governor` crate only reclaims expired entries when `retain_recent()` is explicitly called, but this is wired exclusively to `disconnected()`. The periodic `notify()` callback (every 5 minutes) never calls `retain_recent()` on either rate limiter. An unprivileged peer that maintains a persistent connection and sends messages with unique attacker-controlled `(from, to)` PeerId pairs causes the `HashMapStateStore` to grow monotonically, bounded only by the outer rate limiter's 30 req/sec cap, leading to unbounded memory growth and potential node crash.

## Finding Description

**Root cause:** `retain_recent()` is called only in `disconnected()`: [1](#0-0) 

The `notify()` callback fires every `CHECK_INTERVAL` (5 minutes) and cleans `pending_delivered` and `inflight_requests`, but never calls `retain_recent()` on either rate limiter: [2](#0-1) 

A `grep_search` for `retain_recent` across the entire repo confirms only 2 occurrences, both in `disconnected()` — no other cleanup path exists.

**Attacker-controlled keys:** All three message processors key `forward_rate_limiter` on `(content.from, content.to, self.msg_item_id)`, where `from` and `to` are parsed directly from the incoming message payload with no validation against the actual session identity: [3](#0-2) [4](#0-3) [5](#0-4) 

**Exploit flow:**
1. Attacker establishes a single persistent TCP connection with HolePunching enabled.
2. The outer `rate_limiter` (keyed by `(PeerIndex, u32)`) caps throughput at 30 req/sec per session: [6](#0-5) 
3. Within that cap, attacker sends messages with a fresh unique `(from, to)` PeerId pair each time. Each unique pair inserts a new entry into `HashMapStateStore`.
4. Since `notify()` never calls `retain_recent()`, entries accumulate indefinitely as long as the connection persists.
5. `disconnected()` would reclaim entries, but the attacker never disconnects.

**Type declaration confirming `HashMapStateStore` backing:** [7](#0-6) 

## Impact Explanation

Memory grows at ~30 new entries/sec per persistent attacker session. Each entry stores two `PeerId` values (~39 bytes each) plus governor's internal per-key state (~80–120 bytes), totalling ~150–200 bytes per entry:

| Duration | Entries | Approx. memory |
|---|---|---|
| 10 minutes | 18,000 | ~3.6 MB |
| 1 hour | 108,000 | ~21 MB |
| 24 hours | 2,592,000 | ~500 MB |

With multiple concurrent attacker sessions the impact scales linearly. Sustained attack leads to OOM and node crash. This matches **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation

HolePunching is opt-in via `support_protocols` config: [8](#0-7) 

When enabled, any unprivileged peer that can establish a P2P connection can exploit this. No authentication, proof-of-work, or privileged role is required. The attacker only needs to maintain a TCP session and send valid-but-unique HolePunching messages at the permitted rate. The `from`/`to` PeerIds in the message body are not validated against the actual session identity, so arbitrary unique pairs can be freely generated. The attack is repeatable, deterministic, and requires no victim interaction.

## Recommendation

Add `retain_recent()` calls inside the `notify()` callback so expired entries are reclaimed periodically regardless of session lifetime:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();           // add
    self.forward_rate_limiter.retain_recent();   // add

    let status = self.network_state.connection_status();
    let now = unix_time_as_millis();
    self.pending_delivered
        .retain(|_, (_, t)| (now - *t) < TIMEOUT);
    self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);
    // ... rest of existing logic ...
}
```

This ensures that at most one `CHECK_INTERVAL` (5 minutes) worth of stale entries accumulates, bounding the map size to at most ~9,000 entries (~1.8 MB) regardless of session duration.

## Proof of Concept

1. Connect to a CKB node with `HolePunching` in `support_protocols`.
2. In a loop at 30 messages/sec (within the outer rate limit), send `ConnectionSync` messages where `from` and `to` are freshly generated random valid PeerIds (valid multihash bytes), never reusing a pair.
3. Never disconnect.
4. After 60 seconds: assert the internal `HashMapStateStore` of `forward_rate_limiter` contains ~1,800 entries with zero reclamation (verifiable via memory profiling or by instrumenting `governor`'s map size).
5. After 10 minutes: ~18,000 entries; after 1 hour: ~108,000 entries — all stale but never freed.
6. Scale to N concurrent sessions to multiply growth by N, accelerating OOM.
