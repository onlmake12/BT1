Audit Report

## Title
Missing Deduplication in `GetTransactionsProofProcess::execute` Enables Unbanned DB Lookup Amplification — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

## Summary

`GetTransactionsProofProcess::execute` accepts up to 1000 tx hashes with no deduplication guard. An attacker sending 1000 identical hashes in a single message triggers 2000 DB lookups (1000× `get_transaction_info` + 1000× `get_transaction_with_info`) for a single unique transaction. Because the handler returns `OK` (200), `should_ban` never fires and the peer is never banned, allowing indefinite repetition.

## Finding Description

The only input validation in `execute` is an empty check (L33–35) and a length check `> 1000` (L37–39), meaning exactly 1000 identical hashes pass through unchallenged.

The `partition` at L54–64 calls `snapshot.get_transaction_info(tx_hash)` for every element in the input, including all 1000 duplicates. The resulting `found` vec (1000 entries of the same hash) is then iterated at L67–75, calling `snapshot.get_transaction_with_info(&tx_hash)` for each. The `HashMap` at L66 deduplicates only at the block level for subsequent operations — the 2000 DB calls have already occurred.

By contrast, `GetBlocksProofProcess::execute` (L62–70) uses an explicit `HashSet` deduplication guard that returns `StatusCode::MalformedProtocolMessage` (400) for any duplicate hash. `GetTransactionsProofProcess` has no equivalent guard and returns `OK` (200) at the end via `reply_proof`, so `should_ban` (which only fires for codes `400..500`) returns `None` and no ban is issued.

## Impact Explanation

A single unprivileged light-client peer can repeatedly send `GetTransactionsProof` messages each containing 1000 copies of a known tx hash, forcing 2000 synchronous DB lookups per message (1000× amplification over the expected 2) with no rate-limiting or ban enforcement. This maps to **"Suboptimal implementation of CKB state storage mechanism"** (Medium, 2001–10000 points), as the storage layer is accessed with a 1000× amplification factor indefinitely. It may also qualify as **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (High, 10001–15000 points) if sustained I/O saturation of the storage layer degrades node participation in the broader network.

## Likelihood Explanation

The entrypoint is the P2P light-client protocol, reachable by any peer without any key, PoW, or privileged role. The attacker only needs one valid tx hash confirmed on the main chain. The asymmetry with `GetBlocksProofProcess` (which has the fix) strongly indicates an unintentional omission. No victim mistake or external dependency is required, and the attack is trivially repeatable in a loop.

## Recommendation

Add a `HashSet` deduplication check in `GetTransactionsProofProcess::execute` immediately after the length check, mirroring `GetBlocksProofProcess::execute`:

```rust
use std::collections::HashSet;

let tx_hashes_vec: Vec<_> = self.message.tx_hashes().to_entity().into_iter().collect();
let mut uniq = HashSet::new();
if !tx_hashes_vec.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This returns a 4xx status, which `should_ban` will convert into a `BAD_MESSAGE_BAN_TIME` (5-minute) peer ban.

## Proof of Concept

1. Connect a light-client peer to a CKB node with the light-client protocol server enabled.
2. Obtain any valid tx hash `H` confirmed on the main chain.
3. Send a `GetTransactionsProof` P2P message with `tx_hashes = [H; 1000]` and a valid `last_hash`.
4. Instrument `snapshot.get_transaction_info` and `snapshot.get_transaction_with_info` with call counters.
5. Assert both counters reach 1000 (vs. expected 1 each for a single unique hash).
6. Confirm the peer is not banned after the response (check `should_ban` returns `None` for the `OK` status).
7. Repeat step 3 in a loop; observe sustained DB load with no ban applied.
