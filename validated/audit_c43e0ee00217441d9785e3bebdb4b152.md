### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed Timestamp, Allowing Stale Closed-Market Prices to Pass Freshness Guard - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan` and `Scheduler.getEmaPricesNoOlderThan` validate freshness using `status.priceLastUpdatedAt`, which is the **maximum** `publishTime` across all feeds in the last update batch. Because the Scheduler explicitly allows closed-market feeds to carry timestamps from their last trading period (potentially days old), the subscription-level max timestamp can be fresh while individual stored feed prices are arbitrarily stale. Any whitelisted reader calling `getPricesNoOlderThan` with a strict `age_seconds` receives stale per-feed prices without a revert.

---

### Finding Description

In `Scheduler.updatePriceFeeds`, the contract deliberately passes `minPublishTime = 0` to `parsePriceFeedUpdatesWithConfig` to accommodate closed-market feeds: [1](#0-0) 

The `_validateShouldUpdatePrices` function then computes `updateTimestamp` as the **maximum** `publishTime` across all feeds in the batch: [2](#0-1) 

This maximum is stored as `status.priceLastUpdatedAt`: [3](#0-2) 

`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is only 1 hour: [4](#0-3) 

But the comment in `SchedulerConstants.sol` explicitly acknowledges that closed-market feeds carry timestamps from "a few days ago": [5](#0-4) 

When a reader calls `getPricesNoOlderThan`, the staleness check compares `block.timestamp` against `status.priceLastUpdatedAt` (the subscription-level max), not against each individual feed's `publishTime`: [6](#0-5) 

The function then returns individual prices via `getPricesUnsafe`, which carries no per-feed staleness guarantee: [7](#0-6) 

The same flaw exists in `getEmaPricesNoOlderThan`: [8](#0-7) 

**Concrete scenario:**
- Subscription has 2 feeds: Feed A (equities, closed market, `publishTime = T - 2 days`) and Feed B (crypto, `publishTime = T - 10 minutes`).
- Keeper calls `updatePriceFeeds`; `priceLastUpdatedAt` is set to `T - 10 minutes` (the max).
- Reader calls `getPricesNoOlderThan(subscriptionId, [feedA_id], 3600)` (1-hour age limit).
- Staleness check: `distance(block.timestamp, T - 10 min) = 10 min < 3600` → **passes**.
- Returned price for Feed A has `publishTime = T - 2 days` — **2 days stale**, violating the caller's `age_seconds` constraint.

The existing test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` in `PulseScheduler.t.sol` directly confirms this behavior is reachable: [9](#0-8) 

---

### Impact Explanation

Any protocol integrating the Scheduler and calling `getPricesNoOlderThan` as a "safe" staleness-checked API will silently receive stale prices for closed-market feeds. The function name and NatSpec documentation (`"Returns the price that is no older than age seconds"`) imply a per-feed freshness guarantee that the implementation does not provide. Downstream protocols relying on this guarantee for liquidations, collateral valuation, or derivative settlement can make incorrect financial decisions based on prices that are days old.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support mixed subscriptions containing both crypto (always-on) and traditional-market (closed-hours) feeds. The `minPublishTime = 0` design choice and the `SchedulerConstants` comment confirm this is an intended use case. Any subscription containing at least one closed-market feed alongside one active feed will exhibit this behavior on every update cycle during market-closed hours. Any whitelisted reader (or any reader if `whitelistEnabled = false`) can trigger the issue by calling `getPricesNoOlderThan` — no privileged access is required.

---

### Recommendation

Replace the subscription-level `priceLastUpdatedAt` check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed `publishTime` check. After retrieving each `PriceFeed` from `_getPricesInternal`, validate that `distance(block.timestamp, priceFeed.price.publishTime) <= age_seconds` before including it in the result, reverting (or skipping with a flag) if any requested feed fails the per-feed staleness check.

---

### Proof of Concept

1. Deploy `Scheduler` with a subscription containing two price IDs: one crypto feed and one equity feed.
2. During equity market-closed hours, call `updatePriceFeeds` with a batch where the equity feed has `publishTime = block.timestamp - 2 days` and the crypto feed has `publishTime = block.timestamp - 5 minutes`. The update succeeds because the max timestamp (5 min ago) is within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`.
3. `status.priceLastUpdatedAt` is set to `block.timestamp - 5 minutes`.
4. Call `getPricesNoOlderThan(subscriptionId, [equityFeedId], 3600)`.
5. The staleness check passes (`5 min < 3600 s`), and the function returns the equity price with `publishTime = block.timestamp - 2 days` — 2 days stale — without reverting. [10](#0-9) [2](#0-1) [11](#0-10)

        view

        returns (PythStructs.Price[] memory prices)
    {
        SchedulerStructs.SubscriptionStatus memory status = _state
            .subscriptionStatuses[subscriptionId];

        // Use distance (absolute difference) since pythnet timestamps
        // may be slightly ahead of this chain.
        if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
            revert PythErrors.StalePrice();

        prices = this.getPricesUnsafe(subscriptionId, priceIds);
