### Title
Stale Per-Feed Price Returned Without Revert in `getPricesNoOlderThan` / `getEmaPricesNoOlderThan` Due to Subscription-Level Timestamp Aggregation — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol` (Pyth Pulse) stores a single subscription-level `priceLastUpdatedAt` equal to the **maximum** `publishTime` across all feeds in a batch update. The "safe" read functions `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` check only this aggregate timestamp, not the individual feed's `publishTime`. As a result, a caller requesting a specific feed with a tight age window receives a stale price without any revert, despite the function's freshness guarantee.

---

### Finding Description

In `_validateShouldUpdatePrices`, the subscription-level update timestamp is computed as the **maximum** `publishTime` across all feeds in the batch: [1](#0-0) 

The comment explicitly acknowledges that closed-market feeds will carry a `publishTime` from their last trading period, which may be days old. This maximum timestamp is then stored as `status.priceLastUpdatedAt`: [2](#0-1) 

When a reader calls `getPricesNoOlderThan`, the staleness check compares `block.timestamp` against this subscription-level aggregate, not against the individual feed's `publishTime`: [3](#0-2) 

The same flaw exists in `getEmaPricesNoOlderThan`: [4](#0-3) 

**Concrete scenario:**
- A subscription contains Feed A (active equity, `publishTime = now`) and Feed B (closed equity, `publishTime = now - 3 days`).
- A keeper pushes a valid update; `priceLastUpdatedAt` is set to `now` (the max).
- A reader calls `getPricesNoOlderThan(subscriptionId, [feedB_id], 60)`.
- The check `distance(block.timestamp, now) > 60` is `false` → no revert.
- The returned price for Feed B has `publishTime = now - 3 days` — massively stale — but the function silently returns it as if it were fresh.

---

### Impact Explanation

Any DeFi protocol (lending, derivatives, liquidation engine) that integrates with Pyth Pulse and calls `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` expecting a per-feed freshness guarantee will silently receive arbitrarily stale prices for closed-market feeds. This can cause:

- Incorrect collateral valuations leading to missed or wrongful liquidations.
- Mispriced derivatives settled against stale data.
- Financial loss to protocol users proportional to the price divergence over the stale period.

The Pulse README explicitly recommends `getPricesNoOlderThan` as the safe read path, making this the primary integration surface for downstream protocols. [5](#0-4) 

---

### Likelihood Explanation

- Subscriptions mixing active and closed-market feeds (e.g., equities alongside crypto) are a realistic and documented use case.
- The Scheduler README and `_validateShouldUpdatePrices` comments explicitly describe this mixed-feed scenario as intended behavior.
- Any reader calling `getPricesNoOlderThan` with a tight age window (e.g., 60 seconds) on a subscription containing a closed-market feed will trigger this silently.
- No privileged access is required; any whitelisted reader (or permissionless reader if `whitelistEnabled = false`) can trigger this path.

---

### Recommendation

Replace the subscription-level aggregate staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each returned feed's individual `publishTime`. After fetching the feeds via `_getPricesInternal`, iterate over each result and verify `distance(block.timestamp, feed.price.publishTime) <= age_seconds` before returning. This ensures the freshness guarantee is per-feed, not per-subscription-cycle.

---

### Proof of Concept

1. Deploy `Scheduler` with a subscription containing two feeds: `AAPL/USD` (equity, closed market) and `BTC/USD` (crypto, active).
2. A keeper calls `updatePriceFeeds` with a valid batch where `BTC/USD.publishTime = T` (current) and `AAPL/USD.publishTime = T - 259200` (3 days ago, last trading session). Both share the same Pythnet slot, so `PriceSlotMismatch` is not triggered.
3. `priceLastUpdatedAt` is set to `T` (the max).
4. A reader calls `getPricesNoOlderThan(subscriptionId, [AAPL_USD_id], 60)`.
5. The check `distance(block.timestamp, T) > 60` evaluates to `false` → no revert.
6. The function returns `AAPL/USD` price with `publishTime = T - 259200` — 3 days stale — without any error. [6](#0-5) [7](#0-6)

        uint256 subscriptionId,

    )
        external
        view
        override
        onlyWhitelistedReader(subscriptionId)
        returns (PythStructs.Price[] memory prices)
    {
        SchedulerStructs.SubscriptionStatus memory status = _state
            .subscriptionStatuses[subscriptionId];

        // Use distance (absolute difference) since pythnet timestamps
        // may be slightly ahead of this chain.
        if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
            revert PythErrors.StalePrice();

        prices = this.getPricesUnsafe(subscriptionId, priceIds);
    }
