### Title
Unbounded `pending_get_block_proposals` accumulation enables relay-timer CPU/memory exhaustion - (`sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

---

### Summary

Any unprivileged peer can send repeated `GetBlockProposal` relay messages containing proposal IDs that do not exist in the local tx pool. Each such message causes entries to be inserted into the `pending_get_block_proposals` `DashMap` with **no total-size limit**. A periodic timer handler (`prune_tx_proposal_request`) drains and iterates the entire accumulated map, performing a tx-pool lookup and per-entry response dispatch for every entry. With many peers each sending messages at the per-peer rate limit, the map can grow to millions of entries between timer ticks, causing unbounded CPU and memory consumption in the relay subsystem.

---

### Finding Description

**Inflation path — `GetBlockProposalProcess::execute`**

`GetBlockProposalProcess::execute` in `sync/src/relayer/get_block_proposal_process.rs` accepts a `GetBlockProposal` message, checks that the proposal count does not exceed `max_block_proposals_limit * max_uncles_num` (line 38–44), queries the tx pool for those IDs, and then calls `insert_get_block_proposals` for every ID that is **not** found in the pool:

```rust

    .shared()

**Unbounded insertion — `insert_get_block_proposals`**

`insert_get_block_proposals` in `sync/src/types/mod.rs` inserts every supplied ID into the shared `pending_get_block_proposals: DashMap<ProposalShortId, HashSet<PeerIndex>>` with **no capacity check or eviction**:

```rust

pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}

**Unbounded iteration — `prune_tx_proposal_request`**

`prune_tx_proposal_request` in `sync/src/relayer/mod.rs` is called on the `TX_PROPOSAL_TOKEN` timer. It drains the **entire** accumulated map, issues a single `fetch_txs` call to the tx-pool controller with all accumulated IDs, then iterates over every entry to dispatch responses:

`drain_get_block_proposals` clones the entire map before clearing it, doubling the peak memory cost:

```rust

**Per-message limit does not bound total accumulation**

The per-message check (line 38–44 of `get_block_proposal_process.rs`) caps a single message at `max_block_proposals_limit * max_uncles_num` entries. With default consensus values of 1500 and 2, this is 3000 IDs per message. The Relayer's rate limiter allows 30 messages/second per `(PeerIndex, message_type)` pair:

```rust
// sync/src/relayer/mod.rs:89-92
// setup a rate limiter keyed by peer and message type that lets through 30 requests per second
let quota = governor::Quota::per_second(std::num::NonZeroU32::new(30).unwrap());
let rate_limiter = RateLimiter::hashmap(quota);
``` [6](#0-5) 

At 30 messages/s × 3000 IDs/message = **90,000 unique IDs/second per peer**. With `MAX_RELAY_PEERS = 128` peers, the map can accumulate up to **11.5 million entries per second** — all of which `prune_tx_proposal_request` must clone, look up in the tx pool, and iterate over on each timer tick. [7](#0-6) 

---

### Impact Explanation

- **Memory exhaustion**: Each `ProposalShortId` is 10 bytes; with millions of entries plus `HashSet<PeerIndex>` overhead, heap usage grows rapidly.
- **CPU exhaustion**: `drain_get_block_proposals` clones the entire map; `fetch_txs` sends all IDs to the tx-pool controller (a channel message with a large payload); the subsequent iteration dispatches network sends per peer per ID.
- **Relay subsystem stall**: The `TX_PROPOSAL_TOKEN` timer handler is blocked for the duration of the iteration, delaying all other relay timer work (`ASK_FOR_TXS_TOKEN`, `TX_HASHES_TOKEN`).
- **Cascading effect**: A stalled relay timer delays transaction propagation and compact-block proposal resolution for all honest peers.

---

### Likelihood Explanation

- **No special privilege required**: Any peer that completes the P2P handshake can send `GetBlockProposal` messages.
- **Trivially constructable payload**: The attacker generates arbitrary 10-byte `ProposalShortId` values that are guaranteed not to be in the victim's tx pool (e.g., random bytes). No valid transaction or block is needed.
- **Amplified by peer count**: The attack scales linearly with the number of attacker-controlled connections, up to `MAX_RELAY_PEERS = 128`.
- **No ban triggered**: `GetBlockProposal` with unknown IDs returns `Status::ok()` — the peer is never penalized or disconnected for sending proposals not in the pool. [8](#0-7) 

---

### Recommendation

1. **Add a total-size cap to `insert_get_block_proposals`**: Reject or silently drop insertions once `pending_get_block_proposals.len()` exceeds a configurable maximum (e.g., `max_block_proposals_limit * max_uncles_num * MAX_RELAY_PEERS`).
2. **Add a per-peer cap**: Track how many pending entries each peer has contributed and reject further insertions from that peer once the per-peer quota is exceeded.
3. **Avoid cloning the entire map in `drain_get_block_proposals`**: Use `std::mem::take` on a `Mutex`-wrapped map, or swap in an empty `DashMap`, to avoid doubling peak memory.
4. **Ban peers that repeatedly send unknown proposals**: Increment a per-peer counter for proposals not found in the pool; disconnect and ban after a threshold.

---

### Proof of Concept

1. Attacker establishes up to 128 relay connections to the victim node.
2. Each connection continuously sends `GetBlockProposal` messages at 30 msg/s, each containing 3000 randomly generated `ProposalShortId` values (10 random bytes each) that are not in the victim's tx pool.
3. `GetBlockProposalProcess::execute` passes the per-message count check (3000 ≤ 3000), finds none of the IDs in the tx pool, and calls `insert_get_block_proposals` for all 3000 IDs.
4. Between consecutive `TX_PROPOSAL_TOKEN` timer ticks (~100 ms), the 128 peers collectively insert up to 128 × 3 × 3000 = **1,152,000 entries** into `pending_get_block_proposals`.
5. `prune_tx_proposal_request` fires, clones the 1.15M-entry map, sends all IDs to the tx-pool controller, and iterates over every entry — consuming significant CPU and memory on every tick.
6. Sustained over seconds, heap usage grows into the hundreds of megabytes and the relay timer loop becomes unresponsive to honest peers.

    async fn prune_tx_proposal_request(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());
                }
            }
        }

        let mut relay_bytes = 0;
        let mut relay_proposals = Vec::new();
        for (peer_index, txs) in peer_txs {
            for tx in txs {
                let data = tx.data();
                let tx_size = data.total_size();
                if relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH {
                    send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals))
                        .await;
                    relay_bytes = tx_size;
                } else {
                    relay_bytes += tx_size;
                }
                relay_proposals.push(data);
            }
            if !relay_proposals.is_empty() {
                send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals)).await;
                relay_bytes = 0;
            }
        }
