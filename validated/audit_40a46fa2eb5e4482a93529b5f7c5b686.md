### Title
WeightUnitsFlow Fee Estimator Historical Flow Data Can Be Permanently Inflated by Unprivileged Transaction Submitters — (File: `util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

The `WeightUnitsFlow` fee estimator records every accepted transaction in a persistent historical flow table (`self.txs`) but **never removes entries when transactions are rejected or evicted from the pool**. An unprivileged attacker can submit many large low-fee-rate transactions via the public `send_transaction` RPC, immediately replace them via RBF (paying a trivial extra fee), and leave the inflated flow data in place for up to 256 blocks (~2 hours). During that window every call to `estimate_fee_rate` returns an artificially elevated fee rate, causing all users of the node who rely on this estimate to overpay.

---

### Finding Description

**Root cause — `accept_tx` writes, nothing ever deletes:**

`WeightUnitsFlow::accept_tx` unconditionally appends a `TxStatus` entry to `self.txs` keyed by the current tip block number: [1](#0-0) 

`FeeEstimator::reject_tx` explicitly does **nothing** for the `WeightUnitsFlow` variant: [2](#0-1) 

This is in contrast to `ConfirmationFraction`, which does call `algo.write().reject_tx(tx_hash)` on rejection.

**How the historical flow data is used in estimation:**

`do_estimate` builds `flow_speed_buckets` by calling `sorted_flowed`, which reads all entries in `self.txs` from the last `historical_blocks` blocks: [3](#0-2) 

The flow speed is then used to project future mempool weight:

```
added_weight = flow_speed_buckets[bucket_index] * target_blocks
passed = current_weight + added_weight <= removed_weight
``` [4](#0-3) 

The algorithm iterates from the lowest fee-rate bucket upward and returns the **first** bucket that passes. Inflating `flow_speed_buckets` for low-fee buckets makes those buckets fail the check, forcing the algorithm to return a higher fee rate.

**Data retention window:**

`expire()` only removes entries older than `historical_blocks(MAX_TARGET) = 256` blocks: [5](#0-4) 

So injected flow data persists for up to 256 blocks (~2 hours at the average 28-second block interval).

**Entry path — public RPC:**

Any caller of the public `send_transaction` RPC causes `register_pending` to fire, which calls `fee_estimator_clone.accept_tx(tx_hash, entry_info)`: [6](#0-5) 

No authentication or rate-limiting is applied to this path.

---

### Impact Explanation

On any CKB node configured to use the `WeightUnitsFlow` fee estimator, an attacker can persistently inflate the `flow_speed_buckets` for low-fee-rate buckets. This causes `estimate_fee_rate` to skip those buckets and return a higher fee rate than the true market rate. The inflated estimate persists for up to 256 blocks (~2 hours) after the attacker's transactions are removed. Wallets and services that call `estimate_fee_rate` and use the result to set transaction fees will systematically overpay during the attack window. The attack can be repeated continuously to maintain the inflation indefinitely.

---

### Likelihood Explanation

The attack is cheap and requires no special privilege:

- **Entry**: `send_transaction` is a public, unauthenticated RPC endpoint.
- **Cost**: The attacker pays `min_fee_rate` (1,000 shannons/KB) for the initial transactions and `min_rbf_rate` (1,500 shannons/KB) for the replacements — a difference of 500 shannons/KB per transaction, which is negligible.
- **Persistence**: Each batch of injected transactions inflates flow data for 256 blocks. The attacker can resubmit every ~2 hours to maintain the effect indefinitely.
- **No special knowledge required**: The attacker only needs to know the node's RPC endpoint and the `min_fee_rate` threshold. [7](#0-6) 

---

### Recommendation

When a transaction is rejected or evicted from the pool, its contribution to the historical flow data in `WeightUnitsFlow` should be removed. Concretely:

1. Track accepted transactions in `WeightUnitsFlow` by hash (similar to `ConfirmationFraction`'s `tracked_txs`).
2. Implement a `reject_tx` handler for `WeightUnitsFlow` that removes the corresponding `TxStatus` entry from `self.txs`.
3. Wire this handler into `FeeEstimator::reject_tx` (currently a no-op for `WeightUnitsFlow`). [2](#0-1) 

---

### Proof of Concept

```
1. Configure the CKB node with `[fee_estimator] algorithm = "WeightUnitsFlow"`.

2. Query baseline estimate:
   curl -X POST ... -d '{"method":"estimate_fee_rate","params":[]}'
   # Returns, e.g., 1000 shannons/KB

3. Submit N large transactions (e.g., 500 KB each) with fee rate just above
   min_fee_rate (1000 shannons/KB) via send_transaction RPC.
   Each accepted tx is recorded in WeightUnitsFlow::self.txs at current_tip.

4. Immediately replace each transaction via RBF (fee rate >= 1500 shannons/KB,
   spending the same inputs). The original transactions are rejected.
   WeightUnitsFlow::reject_tx is a no-op — the original entries remain in self.txs.

5. Query estimate_fee_rate again:
   # Returns an inflated value, e.g., 5000+ shannons/KB, because
   # flow_speed_buckets for low-fee buckets is now inflated by the
   # N large transactions, causing those buckets to fail the
   # current_weight + added_weight <= removed_weight check.

6. The inflated estimate persists for up to 256 blocks (~2 hours).
   Repeat step 3-4 every ~2 hours to maintain the effect indefinitely.
```

The attack cost per 2-hour cycle is: `N * tx_size * (min_rbf_rate - min_fee_rate)` = `N * 500KB * 500 shannons/KB`. For N=100 transactions, that is 25,000,000 shannons = 0.25 CKB per cycle — negligible compared to the economic harm caused to users who overpay based on the inflated estimate.
