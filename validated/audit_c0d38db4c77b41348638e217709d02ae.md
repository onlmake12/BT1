### Title
Stale Price Returned Without Staleness Check in `PythAggregatorV3` Chainlink Adapter — (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's own production Chainlink AggregatorV3-compatible adapter contract. Every price-reading function in the contract calls `pyth.getPriceUnsafe()` without checking `publishTime`, meaning any protocol that integrates this adapter as a Chainlink-compatible oracle will silently receive arbitrarily stale prices.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink AggregatorV3 interface as a Pyth-provided adapter. All five price-reading functions call `pyth.getPriceUnsafe(priceId)` with no subsequent `publishTime` validation:

- `decimals()` — line 41
- `latestAnswer()` — line 54
- `latestTimestamp()` — line 59
- `getRoundData()` — line 89
- `latestRoundData()` — line 110

```solidity
// PythAggregatorV3.sol

function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId); // no staleness check
    return int256(price.price);
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId); // no staleness check
    ...
}
```

The `IPyth` interface explicitly documents the danger:

> *"This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past. Users of this function should check the `publishTime` in the price to ensure that the returned price is sufficiently recent."*

`getPriceNoOlderThan(id, age)` exists precisely to enforce a staleness bound, but `PythAggregatorV3` never uses it. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Impact: High**

`PythAggregatorV3` is Pyth's own published adapter that downstream protocols deploy to integrate Pyth with any Chainlink-compatible system (lending protocols, derivatives, liquidation engines, etc.). When `latestRoundData()` or `latestAnswer()` is called, the returned `answer` may reflect a price from arbitrarily far in the past. Protocols relying on this for:

- **Collateral valuation** — may over-value collateral using a stale high price, allowing under-collateralized borrowing.
- **Liquidation thresholds** — may miss liquidations if the stale price does not reflect a real-world price drop.
- **Trade/settlement execution** — an attacker can exploit the gap between the stale on-chain price and the real market price to extract value. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Low**

Price staleness occurs when no one has called `updateFeeds()` recently — during network outages, market closures, or periods of low keeper activity. The `updateFeeds()` function is a separate, optional call with no enforcement mechanism inside the adapter itself. Because `PythAggregatorV3` is a widely-deployed Pyth SDK artifact used by many integrators, the probability that at least one deployed instance is operating with a stale price at any given time is non-trivial. [5](#0-4) 

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable parameter set at construction time (e.g., 60 seconds for DeFi use cases). Alternatively, add an explicit `publishTime` staleness check after each `getPriceUnsafe` call and revert if the price is too old.

```solidity
uint256 public maxPriceAge; // set in constructor

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxPriceAge);
    ...
}
``` [6](#0-5) 

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a price feed ID.
2. Do **not** call `updateFeeds()` for a period exceeding the Pyth `validTimePeriod` (e.g., 60 seconds).
3. Call `latestRoundData()`. The returned `answer` and `updatedAt` reflect the last stored price, which is now stale — no revert, no error.
4. A lending protocol using this adapter for collateral pricing will compute collateral value using the stale price, enabling an attacker to borrow against inflated (stale) collateral or avoid liquidation when the real price has dropped.

```solidity
// Attacker scenario (pseudocode):
// 1. Price of ETH is $3000 on-chain (stale), real price is $2000
// 2. Attacker deposits ETH as collateral
// 3. Protocol calls latestRoundData() -> returns $3000 (stale)
// 4. Attacker borrows at 66% LTV against $3000 = $2000 USDC
// 5. Attacker's position is actually under-collateralized at real price
``` [7](#0-6) [1](#0-0) [8](#0-7)
