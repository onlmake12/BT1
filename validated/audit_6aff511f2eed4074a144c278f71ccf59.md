### Title
Rate Limit Bypass via Reconnection — `PeerIndex`-Keyed Limiter Resets to Full Burst on Reconnect (`sync/src/relayer/mod.rs`, `network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `Relayer` and `HolePunching` protocol handlers enforce per-peer message rate limits using `governor::RateLimiter` keyed by `(PeerIndex, u32)`. `PeerIndex` is a type alias for `SessionId` — a per-connection ephemeral integer, not a stable cryptographic peer identity. When a peer exhausts its rate limit and then disconnects and reconnects, it receives a new `SessionId` and therefore a completely fresh, fully-replenished token bucket. The `disconnected` handler calls `retain_recent()` on the limiter, which only prunes globally stale entries and does not remove the old session's entry by identity — but this is irrelevant because the reconnecting peer's new `PeerIndex` has no entry at all, giving it a full burst allowance. This is the direct structural analog of M-05: a state-resetting event (reconnection instead of `_updateAddress`) silently restores the full buffer cap.

---

### Finding Description

**Root cause — `Relayer`:**

`Relayer.rate_limiter` is declared as `RateLimiter<(PeerIndex, u32)>` with a quota of 30 requests per second per `(peer, message_type)` pair. [1](#0-0) 

`PeerIndex` is defined as a plain alias for `SessionId`: [2](#0-1) 

On disconnect, the handler calls `retain_recent()` — a global housekeeping call that removes entries whose token buckets have fully recovered. It does **not** remove the specific disconnecting peer's entry by key, and it does **not** bind the rate-limit state to the peer's cryptographic identity (`PeerId`): [3](#0-2) 

When the peer reconnects, the tentacle P2P layer assigns a new `SessionId`. The rate limiter has no entry for this new `(PeerIndex, message_type)` key, so `check_key` succeeds immediately and the peer is granted a full fresh burst of 30 tokens: [4](#0-3) 

**Root cause — `HolePunching`:**

The same pattern exists in `HolePunching.rate_limiter`, also keyed by `(PeerIndex, u32)` at 30/sec, with `retain_recent()` called on every disconnect: [5](#0-4) [6](#0-5) 

Note: `HolePunching.forward_rate_limiter` is correctly keyed by `(PeerId, PeerId, u32)` and is **not** affected.

**Contrast with M-05:**

| | M-05 (Volt) | CKB analog |
|---|---|---|
| Trigger | Admin calls `_updateAddress` | Peer disconnects and reconnects |
| Reset mechanism | `bufferStored = _bufferCap` (explicit) | New `PeerIndex` → no entry in limiter (implicit) |
| Who controls trigger | Privileged admin | Unprivileged peer (self-triggered) |
| Result | Full buffer restored | Full burst restored |

---

### Impact Explanation

An unprivileged peer can continuously bypass the 30 msg/sec relay rate limit by cycling connections. Each reconnect costs one TCP + P2P handshake but yields a completely fresh 30-token burst. Over N reconnects within a second, the peer can deliver N × 30 relay messages (e.g., `RelayTransactions`, `GetRelayTransactions`, `GetBlockTransactions`, `GetBlockProposal`) to the victim node. This allows:

- **CPU exhaustion**: Each relay message triggers non-trivial processing (transaction validation, compact block reconstruction, proposal lookups).
- **Effective DoS**: The rate limit is the only per-peer admission control for relay messages; bypassing it removes the only throttle protecting the node's relay processing loop.

---

### Likelihood Explanation

The attack requires only a standard TCP connection to the node's P2P port — no special privileges, no keys, no majority hashpower. The reconnection overhead (TCP + tentacle handshake) is small relative to the processing cost of relay messages. A single attacker with a modest connection rate can sustain a multiplied message flood. The attack is fully self-contained and requires no coordination.

---

### Recommendation

Key the rate limiter by `PeerId` (the stable cryptographic identity extracted from the multiaddr) rather than `PeerIndex` (the ephemeral session integer). This ensures that exhausting the rate limit on one connection carries over to any subsequent reconnection from the same peer identity, exactly as the `forward_rate_limiter` in `HolePunching` already does correctly.

```rust
// Current (vulnerable): resets on every reconnect
rate_limiter: RateLimiter<(PeerIndex, u32)>

// Fixed: persists across reconnects
rate_limiter: RateLimiter<(PeerId, u32)>
```

The `PeerId` is available at connection time via `extract_peer_id(&session_context.address)` and is already stored in `Peer::connected_addr`. The `disconnected` handler should additionally call `rate_limiter.shrink_to_fit()` or remove the specific `PeerId` entry after a configurable ban/cooldown window to prevent unbounded memory growth.

---

### Proof of Concept

```
1. Attacker connects to victim CKB node (obtains PeerIndex = N).
2. Attacker sends 30 GetRelayTransactions messages in rapid succession
   → rate_limiter.check_key(&(N, item_id)) returns Ok for first 30,
     then Err (TooManyRequests) for subsequent ones.
3. Attacker immediately disconnects.
   → Relayer::disconnected fires: rate_limiter.retain_recent()
     (PeerIndex=N entry is retained since bucket is not yet recovered;
      regardless, it is irrelevant to the next step.)
4. Attacker reconnects (obtains PeerIndex = N+1).
   → rate_limiter has NO entry for (N+1, item_id).
5. Attacker sends another 30 GetRelayTransactions messages.
   → All 30 pass: check_key(&(N+1, item_id)) returns Ok.
6. Repeat from step 3 indefinitely.

Net effect: attacker delivers 30 × K relay messages per K reconnects,
with no effective per-identity cap.
```
