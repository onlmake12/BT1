### Title
Unbounded Outbound Bandwidth Amplification via `GetRelayTransactions` Batching Loop — (`sync/src/relayer/get_transactions_process.rs`)

### Summary

A single `GetRelayTransactions` P2P message containing up to 32,767 tx hashes can cause the responding node to emit an unbounded number of outbound `RelayTransactions` messages — one per ~1 MiB batch — with no cap on total outbound bytes per request.

### Finding Description

`MAX_RELAY_TXS_NUM_PER_BATCH` is 32,767 and `MAX_RELAY_TXS_BYTES_PER_BATCH` is 1 MiB. [1](#0-0) 

`execute()` enforces only a count guard on the inbound message: [2](#0-1) 

After fetching all matching transactions from the pool, the batching loop sends one `RelayTransactions` message every time the running byte total would exceed 1 MiB: [3](#0-2) 

There is **no cap on the number of batches**. If N transactions each have `total_size()` ≈ 1 MiB, the loop emits N separate outbound messages — one per transaction — for a single inbound request.

The rate limiter in `try_process` (30 req/s per `(PeerIndex, message_type)`) only throttles **incoming** messages: [4](#0-3) 

A single request that passes the rate limiter once can trigger an arbitrarily large burst of outbound traffic.

### Impact Explanation

The tx pool is bounded by `max_tx_pool_size` (default ~180 MB), so the realistic worst case per request is ~180 outbound messages × 1 MiB = ~180 MiB. With the rate limiter permitting 30 `GetRelayTransactions` messages per second from a single peer, a sustained attack yields ~5.4 GiB/s of outbound bandwidth from one peer connection — sufficient to saturate a typical node's uplink and degrade relay performance for all other peers. Impact is network performance degradation (DoS), matching the Low (501–2000) scope. [5](#0-4) 

### Likelihood Explanation

The precondition — a populated tx pool — is realistic. An attacker with modest CKB holdings can submit many large valid transactions to fill the pool, then immediately send a `GetRelayTransactions` message requesting all of them. Alternatively, the attacker can exploit a naturally full pool without submitting any transactions themselves, by learning tx hashes from `RelayTransactionHashes` broadcasts. No privileged access is required; any P2P peer can send `GetRelayTransactions`.

### Recommendation

Add a cap on the **total number of batches** (or total outbound bytes) emitted per single `GetRelayTransactions` request. For example, break out of the loop after sending a fixed maximum number of batches (e.g., 4–8), or track cumulative bytes sent and stop once a per-request outbound limit is reached. Additionally, consider applying per-peer outbound rate limiting at the send layer, not just on inbound message receipt.

### Proof of Concept

1. Connect to a CKB node as an unprivileged P2P peer.
2. Submit (or observe) N large transactions (each ~900 KB) into the target node's tx pool.
3. Send one `GetRelayTransactions` message containing all N tx hashes (N ≤ 32,767).
4. Observe: the node sends back N separate `RelayTransactions` messages, each ~900 KB.
5. Measure total bytes received: N × ~900 KB from a single ~few-KB inbound message.
6. Repeat at 30 req/s (rate limiter maximum) to sustain amplified outbound load.

The invariant violated: a single peer request must not cause unbounded outbound bandwidth consumption. The batching loop at lines 86–102 of `get_transactions_process.rs` has no guard on the number of batches sent. [3](#0-2)
