Audit Report

## Title
Rate-Limit Bypass: `TooManyRequests = 110` Outside `should_ban()` Guard Allows Persistent Relay Flooding — (`sync/src/relayer/mod.rs`, `sync/src/status.rs`)

## Summary
`StatusCode::TooManyRequests` is assigned value `110`, placing it in the 1xx informational range. `Status::should_ban()` gates exclusively on codes in `400..500`, so it always returns `None` for `TooManyRequests`. A peer that continuously exceeds the relay rate limiter is never disconnected — it receives only a `debug_target!` log entry and can retry immediately, indefinitely, occupying a peer slot and forcing the node to receive and partially parse every message.

## Finding Description
In `sync/src/relayer/mod.rs`, `try_process()` checks the rate limiter keyed by `(PeerIndex, message.item_id())` at 30 req/s. On failure it returns `StatusCode::TooManyRequests.with_context(message.item_name())` (lines 116–123). `process()` then calls `status.should_ban()` on the returned status (lines 195–204). `should_ban()` opens with `if !(400..500).contains(&(self.code as u16)) { return None; }` (lines 165–168 of `sync/src/status.rs`). Since `TooManyRequests = 110`, the condition `!(400..500).contains(&110)` is `true`, so `None` is returned immediately. `nc.ban_peer()` is never called. The status falls through to the `else if !status.is_ok()` branch (lines 213–221), emitting only a debug log. The peer remains connected and can repeat the flood indefinitely. By contrast, `HolePunching` (lines 95–107 of `network/src/protocols/hole_punching/mod.rs`) returns early at the rate-limit check site, never propagating `TooManyRequests` to any ban logic. The relayer does not follow this pattern. Every inbound message — even one rejected by the rate limiter — still requires network receive, molecule deserialization to reach `item_id()`, and a hashmap lookup in the rate limiter. With 7 non-`CompactBlock` relay message types, up to 7 × 30 = 210 req/s of actual processing can be forced per peer, with unlimited additional messages hitting the rejection path.

## Impact Explanation
This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** A single attacker with one TCP connection can permanently occupy a peer slot and sustain an unbounded message flood. The node is forced to receive and partially parse every message. With multiple attacker connections, peer slots are exhausted, crowding out legitimate peers and degrading network connectivity. The rate limiter does not prevent this because it only limits processing depth, not connection persistence.

## Likelihood Explanation
Any connected peer can trigger this. No special privilege, key, or hashpower is required. The attacker needs only a standard TCP connection and the ability to send relay protocol messages faster than 30/s — trivially achievable with a standard CKB client or raw socket. The attack is repeatable, persistent, and requires no victim interaction.

## Recommendation
The simplest fix is to handle the rate-limit rejection at the call site in `try_process()`, mirroring the `HolePunching` pattern: log the rejection and `return` (returning `Status::ok()` or `Status::ignored()`) rather than propagating `TooManyRequests` upward. Alternatively, move `TooManyRequests` into the 4xx range (e.g., `TooManyRequests = 429`) so `should_ban()` covers it automatically, or add an explicit arm in `should_ban()` for `StatusCode::TooManyRequests` returning a ban duration.

## Proof of Concept
1. Connect to a CKB node as a peer.
2. Send `GetRelayTransactions` (or any non-`CompactBlock` relay message) at >30 req/s continuously.
3. Observe: the node logs `debug … TooManyRequests` for each excess message but never calls `ban_peer`. The peer remains connected indefinitely.
4. Confirm `should_ban()` returns `None` for `TooManyRequests = 110` by inspection of the `!(400..500).contains(&110)` guard — `110` is not in `[400, 500)`, so the early return fires.
5. Compare with a patched version where the rate-limit check in `try_process()` returns early (as `HolePunching` does): the peer is never banned but the status is never propagated, or alternatively where `TooManyRequests` is in the 4xx range: the peer is disconnected after the first rate-limit violation.
