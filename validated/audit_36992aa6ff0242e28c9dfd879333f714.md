### Title
Unvalidated Ordering of Peer-Supplied `block_locator_hashes` Breaks Early-Exit Invariant in `insert_peer_unknown_header_list`, Disrupting IBD Block Fetching — (`sync/src/types/mod.rs`)

---

### Summary

During Initial Block Download (IBD), a CKB node stores a peer's `GetHeaders` locator hashes into `unknown_header_list` and later uses that list to advance the peer's `best_known_header`. Both the population loop (`insert_peer_unknown_header_list`) and the consumption loop (`BlockFetcher::fetch`) rely on the invariant that the list is ordered **highest-to-lowest** block number. Because the ordering comes directly from a peer-controlled P2P message with no validation, a malicious outbound peer can send an intentionally misordered `block_locator_hashes` list to violate the invariant, causing the early-exit `break` to fire at the wrong point. The result is that `best_known_header` for that peer is never correctly advanced, and the local node stops requesting blocks from that peer for the remainder of IBD.

---

### Finding Description

**Root cause — population side (`insert_peer_unknown_header_list`):**

`GetHeadersProcess::execute` receives a `GetHeaders` P2P message, extracts `block_locator_hashes` verbatim, and — when the local node is in IBD and the sender is an outbound/whitelist/protect peer — passes the raw, unvalidated slice directly to `insert_peer_unknown_header_list`: [1](#0-0) 

`insert_peer_unknown_header_list` then iterates the list with an early-exit `break` that is only correct if the list is sorted highest-to-lowest: [2](#0-1) 

The comment on line 1184 explicitly documents the ordering assumption ("header list is an ordered list, sorted from highest to lowest, so here you discard and exit early"), but nothing enforces it. Unknown hashes are appended to the tail of `unknown_header_list` via `push`: [3](#0-2) 

**Root cause — consumption side (`BlockFetcher::fetch`):**

The consumption loop pops from the **end** of the Vec (`take_unknown_last` → `Vec::pop`) and breaks as soon as it encounters a hash that is still unknown: [4](#0-3) 

`take_unknown_last` is: [5](#0-4) 

With the correct ordering (highest pushed first → lowest at the tail), `pop` yields the lowest unknown hash first, allowing `best_known_header` to advance incrementally as blocks arrive. With a misordered list (lowest pushed first → highest at the tail), `pop` yields the highest hash first. That hash is almost certainly still unknown (not yet downloaded), so the loop breaks on the very first iteration and `best_known_header` is never updated.

**`PeerState.unknown_header_list` field:** [6](#0-5) 

---

### Impact Explanation

`best_known_header` is the sole signal used by `BlockFetcher` to determine the upper bound of blocks it will request from a peer. If it is never advanced beyond its initial `None` / genesis value, `BlockFetcher::fetch` returns `None` for that peer at line 159–167 (no `best_known` → skip peer), and the local node never issues `GetBlocks` requests to that peer for the duration of IBD. An attacker controlling several outbound slots can therefore starve the victim node of block-download bandwidth during IBD, prolonging sync time proportionally to the fraction of outbound slots they occupy.

---

### Likelihood Explanation

The attack requires only that the adversary run a reachable CKB node and wait for victim nodes to open outbound connections to it — a realistic, low-cost condition. The malformed `GetHeaders` message is a single P2P packet. No privileged access, key material, or majority hash power is needed. The guard `flag.is_outbound || flag.is_whitelist || flag.is_protect` is satisfied by any normal outbound peer. [7](#0-6) 

---

### Recommendation

Validate the ordering of `block_locator_hashes` before trusting the early-exit invariant. In `insert_peer_unknown_header_list`, after collecting hashes into `unknown_header_list`, sort the list by block number (ascending, so that `pop` yields the lowest first) rather than relying on the peer to supply them in the correct order. Alternatively, look up each hash's block number from the header map or store and insert into a sorted structure (e.g., `BTreeMap` keyed by block number) so that the consumption order is always correct regardless of the peer-supplied ordering.

---

### Proof of Concept

1. Attacker runs a CKB node. Victim node opens an outbound connection to it during IBD.
2. Attacker sends a `SyncMessage::GetHeaders` with `block_locator_hashes` ordered **lowest-to-highest** (i.e., reversed from the expected highest-to-lowest).
3. `GetHeadersProcess::execute` calls `insert_peer_unknown_header_list(peer, reversed_hashes)`.
4. The loop pushes all unknown hashes in reversed order (lowest first → highest at tail). No known hash is encountered, so no `break` fires and `best_known_header` is not set.
5. `BlockFetcher::fetch` calls `take_unknown_last`, popping the **highest** hash. That hash is not yet in the header map (IBD is ongoing), so the `else` branch fires: the hash is pushed back and the loop `break`s immediately.
6. `best_known_header` for the attacker peer remains `None`. `BlockFetcher::fetch` returns `None` at the `peer_best_known_header()` check, and no `GetBlocks` is ever sent to this peer.
7. Repeating across multiple outbound slots degrades IBD throughput proportionally.

    pub fn insert_peer_unknown_header_list(&self, pi: PeerIndex, header_list: Vec<Byte32>) {
        // update peer's unknown_header_list only once
        if self.state().peers.unknown_header_list_is_empty(pi) {
            // header list is an ordered list, sorted from highest to lowest,
            // so here you discard and exit early
            for hash in header_list {
                if let Some(header) = self.shared().header_map().get(&hash) {
                    self.state()
                        .peers
                        .may_set_best_known_header(pi, header.as_header_index());
                    break;
                } else {
                    self.state().peers.insert_unknown_header_hash(pi, hash)
                }
            }
        }
    }
