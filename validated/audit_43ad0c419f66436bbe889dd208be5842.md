### Title
Aggregate `total_tx_size` Counter Overstated When Ancestor-Eviction Occurs During `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the aggregate `total_tx_size` counter is pre-computed before `check_and_record_ancestors` runs, but written back **after** it runs. When `check_and_record_ancestors` evicts transactions (via `remove_entry_and_descendants`), those removals correctly decrement `self.total_tx_size` through `update_stat_for_remove_tx`. However, the pre-computed value — which does not account for those removals — is then unconditionally written back, permanently overstating `total_tx_size` by the sum of the evicted transactions' sizes. This is the direct CKB analog of the Neptune Mutual conflated pool balance bug: an aggregate counter is read from a stale pre-computed value rather than from the correctly-updated per-operation record.

---

### Finding Description

In `tx-pool/src/component/pool_map.rs`, `add_entry` follows this sequence:

```
1. total_tx_size = self.total_tx_size + entry.size   // pre-computed, stored in local var
2. check_and_record_ancestors(&mut entry)             // may call remove_entry_and_descendants
      └─> remove_entry(id)
              └─> update_stat_for_remove_tx(size, cycles)
                      └─> self.total_tx_size -= size  // correctly decrements the field
3. self.total_tx_size = total_tx_size                 // OVERWRITES with stale pre-computed value
``` [1](#0-0) 

The eviction inside `check_and_record_ancestors` is triggered when a submitted transaction has more than `max_ancestors_count` ancestors, but the excess is attributable to cell-dep-referenced parents (`cell_ref_parents`): [2](#0-1) 

When that branch fires, `remove_entry_and_descendants` is called for each evicted entry: [3](#0-2) 

`remove_entry` calls `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size`: [4](#0-3) 

But the final two lines of `add_entry` overwrite `self.total_tx_size` with the pre-computed local variable that was captured **before** any evictions occurred: [5](#0-4) 

**Concrete arithmetic:**
- Before `add_entry`: `self.total_tx_size = S`
- Pre-computed local: `total_tx_size = S + entry.size`
- After eviction of a tx with size `R`: `self.total_tx_size = S − R`
- After the overwrite: `self.total_tx_size = S + entry.size` ← **wrong**
- Correct value: `S − R + entry.size`
- **Overstatement per trigger: `R`** (size of evicted tx(s))

---

### Impact Explanation

`total_tx_size` is the sole counter used by `limit_size` to enforce the pool size cap: [6](#0-5) 

With an overstated `total_tx_size`, `limit_size` evicts additional legitimate transactions even when the actual pool occupancy is within `max_tx_pool_size`. Each trigger of the eviction path inflates the counter by the size of the evicted entries. Repeated triggering accumulates the overstatement, causing the pool to appear perpetually over-full and evicting valid pending/proposed transactions. This mirrors the Neptune Mutual impact: an overstated aggregate balance prevents legitimate operations (there: reward claims; here: transaction retention).

The overstated value is also surfaced directly to RPC callers via `tx_pool_info`: [7](#0-6) 

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` requires:
1. A transaction whose total ancestor count exceeds `max_ancestors_count` (default 1,000).
2. The excess is attributable to cell-dep-referenced parents (`cell_ref_parents`), so that `ancestors_count − cell_ref_parents.len() ≤ max_ancestors_count`.

An unprivileged `send_transaction` RPC caller can craft this scenario by:
- Submitting a chain of transactions that approaches the ancestor limit.
- Submitting a final transaction with a cell dep referencing one of those chain transactions, pushing the ancestor count just over the limit.

The 1,000-ancestor threshold is a meaningful barrier, but it is reachable on a busy mempool or by a deliberate attacker willing to pay fees for a long chain. The bug is triggered by normal protocol-level transaction submission with no privileged access required.

---

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes, so that any evictions performed during ancestor resolution are already reflected in `self.total_tx_size` before the addition is applied:

```rust
// Remove the pre-computation before check_and_record_ancestors.
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Apply the new entry's contribution to the already-correct post-eviction counter.
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, move the `updated_stat_for_add_tx` call to after `check_and_record_ancestors` and remove the pre-computation entirely.

---

### Proof of Concept

1. Fill the pool with a chain of exactly `max_ancestors_count` (1,000) transactions `T1 → T2 → … → T1000`, each spending the previous output. Total pool size = `S`.
2. Submit transaction `T_new` with:
   - One input spending `T1000`'s output (making `T1000` a parent, `T1–T999` ancestors via inputs → `ancestors_count = 1001`).
   - One cell dep referencing `T500`'s output (`cell_ref_parents = {T500}`).
3. Inside `add_entry`:
   - `updated_stat_for_add_tx` captures `total_tx_size = S + size(T_new)`.
   - `check_and_record_ancestors` detects `1001 > 1000` but `1001 − 1 = 1000 ≤ 1000`, enters the eviction branch.
   - `remove_entry_and_descendants(T500)` removes `T500` (and its descendants `T501–T1000`), decrementing `self.total_tx_size` by `sum(size(T500..T1000))`.
   - `self.total_tx_size = total_tx_size` overwrites with `S + size(T_new)`, ignoring the evictions.
4. `self.total_tx_size` is now overstated by `sum(size(T500..T1000))`.
5. `limit_size` is called; it sees `total_tx_size > max_tx_pool_size` and evicts additional legitimate transactions from the pool even though the actual occupancy is well below the limit.
6. Repeat to accumulate unbounded overstatement.

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
