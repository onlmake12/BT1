Audit Report

## Title
Min-Fee-Rate Invariant Broken by Size-Only Weight in `check_tx_fee` — (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` computes the minimum required fee using only serialized byte size as the weight, not `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Cycles are determined by `verify_rtx` after the fee gate passes, but no weight-based fee rate re-check occurs before pool admission. An unprivileged caller can submit cycle-heavy, byte-small transactions that pass the fee gate at fee rates far below the configured `min_fee_rate`, forcing the node to execute up to the block cycle limit of script verification per transaction at near-zero cost.

## Finding Description
`check_tx_fee` in `tx-pool/src/util.rs` explicitly acknowledges the theoretical incorrectness in a comment and uses only `tx_size as u64` as the weight argument:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

This function is called in `pre_check` at `tx-pool/src/process.rs` L289 before any script execution: [2](#0-1) 

The admission flow in `_process_tx` is:
1. `pre_check` → `check_tx_fee(tx_size)` — fee gate passes on byte size alone
2. `verify_rtx` → actual cycles returned
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with actual cycles
4. `submit_entry` — admitted to pool

Between steps 2 and 4, there is no weight-based fee rate re-check: [3](#0-2) 

The weight-based `fee_rate()` in `TxEntry` and `EvictKey` are used exclusively for pool ordering and eviction, not admission: [4](#0-3) [5](#0-4) 

`get_transaction_weight` confirms the correct formula that is never applied at admission: [6](#0-5) 

With `min_fee_rate = 1000 shannons/KW` and a 100-byte transaction consuming 70M cycles:
- Actual weight: `max(100, 70,000,000 × 0.000_170_571_4) = 11,940 bytes`
- `check_tx_fee` requires: `1000 × 100 / 1000 = 100 shannons`
- Actual fee rate if 100 shannons paid: `100 × 1000 / 11,940 ≈ 8 shannons/KW` — 125× below minimum

The `DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES * 20` establishes the per-tx cycle cap: [7](#0-6) 

Note: the actual `max_cycles` used in `_process_tx` is `declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles())`, which for undeclared submissions defaults to the full block cycle limit — potentially larger than `DEFAULT_MAX_TX_VERIFY_CYCLES`, making the worst-case impact greater than described. [8](#0-7) 

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. An attacker can repeatedly submit cycle-heavy transactions via the `send_transaction` RPC, each forcing the node to execute up to the block cycle limit of CKB-VM script verification while paying only a fraction of the required fee. The `min_fee_rate` anti-spam invariant — the primary economic defense against pool flooding — is bypassed at admission. Sustained submission congests the node's verification pipeline at a cost 125× (or more) below the intended floor.

## Likelihood Explanation
The entry point is the `send_transaction` RPC, reachable by any unprivileged caller with no keys or special roles. The attacker writes a CKB-VM lock script consuming near-maximum cycles, stores it in a cell dep to keep the transaction body small (~100–200 bytes serialized), and submits transactions spending cells locked by this script. The transaction is consensus-valid; it simply pays less than the weight-based minimum. The attack is repeatable with no per-attempt barrier beyond the nominal fee (100 shannons ≈ negligible).

## Recommendation
After `verify_rtx` returns actual cycles, perform a weight-based fee rate check before `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

The existing size-only pre-check in `check_tx_fee` can remain as a cheap early rejection for obviously underpriced transactions; the post-verification check enforces the true invariant.

## Proof of Concept
1. Write a CKB-VM lock script that executes a tight loop consuming ~70,000,000 cycles. Deploy it in a cell.
2. Construct a transaction spending a cell locked by this script, with the script stored as a cell dep (keeping serialized tx size ~100 bytes). Set `fee = 100 shannons`.
3. Submit via `send_transaction` RPC to a node with default `min_fee_rate = 1000`.
4. `check_tx_fee` computes `min_fee = 1000 × 100 / 1000 = 100 shannons`. Fee equals min_fee → admitted past the gate.
5. `verify_rtx` executes the script, consuming 70M cycles. No post-verification fee rate check occurs.
6. Transaction enters the pool with actual fee rate ≈ 8 shannons/KW — 125× below `min_fee_rate`.
7. Repeat to continuously force 70M-cycle verifications at negligible cost, congesting the node's verification pipeline.
