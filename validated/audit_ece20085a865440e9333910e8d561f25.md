Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequest` Allows Spoofed `pending_delivered` Poisoning and Hole-Punching DoS — (`File: network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol's `respond_delivered` function inserts entries into `pending_delivered` keyed by `content.from`, a field taken directly from the message payload with no validation against the authenticated session peer ID. Any connected peer can spoof an arbitrary victim peer ID in the `from` field, causing the target node to block legitimate hole-punching requests from that victim for `HOLE_PUNCHING_INTERVAL` (2 minutes) and to store attacker-controlled addresses for subsequent NAT traversal attempts. The attack requires only a single authenticated P2P connection and one message per 2 minutes per victim peer ID.

## Finding Description

**Root cause:** In `execute()`, when `self_peer_id == &content.to`, the code calls `respond_delivered(content.from, ...)` directly:

```rust
// connection_request.rs L145-147
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs).await
```

Inside `respond_delivered` (L161-167), the function checks `pending_delivered` keyed by `from_peer_id`:

```rust
if let Some((_, t)) = self.protocol.pending_delivered.get(&from_peer_id) {
    let now = unix_time_as_millis();
    if now - t < HOLE_PUNCHING_INTERVAL {
        return StatusCode::Ignore ...;
    }
}
```

If absent or stale, it sends a response back to the actual session peer (`self.peer`) and then inserts:

```rust
// L234-237
let now = unix_time_as_millis();
self.protocol.pending_delivered.insert(from_peer_id, (remote_listens, now));
```

`from_peer_id` is `content.from` — a plain byte field in the packed message. There is no comparison of `content.from` against the authenticated session peer ID. The session is authenticated via `secio`, so the real peer ID is known and trusted, but it is never checked against the `from` field.

**Existing guards are insufficient:**
- `rate_limiter` (L95-107 of `mod.rs`) is keyed by `(session_id, msg.item_id())` — allows 30 req/sec per session, far more than the 1 message per 2 minutes needed per victim.
- `forward_rate_limiter` (L132-143 of `connection_request.rs`) is keyed by `(content.from, content.to, msg_item_id)` — the attacker controls `content.from` and can vary it freely across victim peer IDs, bypassing this limiter entirely.

**Downstream impact via `ConnectionSync`:** When a `ConnectionSync` arrives, `connection_sync.rs` (L111-115) looks up `pending_delivered.get(&content.from)` and uses the stored addresses for NAT traversal:

```rust
let listens_info = self.protocol.pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
```

If the attacker poisoned this entry with their own address, the victim's NAT traversal attempt connects to the attacker's address instead of the victim's real address.

**Attack sequence:**
1. Attacker A (peer_id = `A_id`) establishes a legitimate P2P connection to node N.
2. A sends `ConnectionRequest { from: victim_id, to: N_id, listen_addrs: [A_addr], route: [] }`.
3. N sees `self_peer_id == content.to`, calls `respond_delivered(victim_id, N_id, [A_addr])`.
4. No existing entry for `victim_id` → N sends a response to A and inserts `pending_delivered[victim_id] = ([A_addr], T)`.
5. For the next 2 minutes, any legitimate `ConnectionRequest` from `victim_id` to N returns `StatusCode::Ignore`.
6. Any subsequent `ConnectionSync` from `victim_id` causes N to attempt NAT traversal to `A_addr` instead of the victim's real address.
7. A repeats step 2 every 119 seconds to maintain the block indefinitely.

## Impact Explanation

An unprivileged peer with a single connection can permanently block NAT traversal for arbitrary peer IDs against the target node at negligible cost. Scaled across multiple nodes simultaneously, an attacker can prevent targeted peers behind NAT from establishing connections to the CKB P2P network entirely. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** — specifically, systematic degradation of NAT peer onboarding across the network with one message per 2 minutes per victim per node.

## Likelihood Explanation

- Requires only a single authenticated P2P connection — no special privileges.
- The `from` field has no cryptographic binding to the session; it is plain bytes in the packed message.
- Cost is minimal: one `ConnectionRequest` message per 2 minutes per victim peer ID.
- The attacker can target many victim peer IDs in a single session (rate limiter allows 30 req/sec).
- The `forward_rate_limiter` does not prevent this because the attacker controls `content.from`.

## Recommendation

In `ConnectionRequestProcess::execute`, before calling `respond_delivered`, validate that `content.from` matches the authenticated session peer ID. The session peer ID can be resolved from the peer registry using `self.peer` (a `PeerIndex`):

```rust
// Reject if the from field does not match the actual session peer
let actual_peer_id = self.protocol.network_state
    .with_peer_registry(|reg| {
        reg.get_peer(self.peer).and_then(|p| Some(p.peer_id.clone()))
    });
if actual_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId
        .with_context("from does not match authenticated session peer");
}
```

This ensures only the legitimate owner of a peer ID can create `pending_delivered` entries for that peer ID.

## Proof of Concept

```
1. Attacker A connects to node N (authenticated session, peer_id = A_id).
2. A sends: ConnectionRequest { from: victim_id, to: N_id, listen_addrs: [A_addr], route: [] }
3. N: content.to == self_peer_id → respond_delivered(victim_id, N_id, [A_addr])
   → pending_delivered[victim_id] = ([A_addr], T)
4. Victim V (peer_id = victim_id) sends: ConnectionRequest { from: victim_id, to: N_id, ... }
5. N: pending_delivered[victim_id] exists, now - T < HOLE_PUNCHING_INTERVAL → returns Ignore
   → V never receives a response; hole punching fails.
6. A repeats step 2 every 119 seconds → V is permanently blocked from hole-punching with N.
7. To verify address poisoning: after step 3, send ConnectionSync { from: victim_id, to: N_id, route: [] }
   → N attempts NAT traversal to A_addr instead of V's real address.
```
