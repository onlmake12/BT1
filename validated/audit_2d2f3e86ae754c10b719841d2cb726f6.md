Audit Report

## Title
Unauthenticated Outbound Raw Session to Attacker-Controlled Endpoint via Crafted `ConnectionRequestDelivered` — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

## Summary
`ConnectionRequestDeliveredProcess::execute()` trusts the `from`, `to`, and `listen_addrs` fields of an incoming message without verifying that the sender is the peer identified by `content.to`, or that `listen_addrs` belongs to that peer. An attacker who is a connected peer can arrange for an arbitrary `PeerId` to appear in the victim's `inflight_requests` (by advertising a fake address via Discovery), then send a crafted `ConnectionRequestDelivered` message causing the victim to establish an outbound TCP connection to an attacker-controlled endpoint and run the Identify protocol against it, enabling peer-store poisoning.

## Finding Description

**Step 1 — Seeding `inflight_requests` with an attacker-chosen PeerId.**

`notify()` calls `fetch_nat_addrs()` to populate `inflight_requests`. [1](#0-0) 

`fetch_nat_addrs` selects addresses from the peer store that (a) match required flags, (b) are not currently connected, and (c) have `last_connected_at_ms == 0`. [2](#0-1) 

The Discovery protocol's `add_new_addrs` inserts attacker-supplied addresses (with attacker-supplied flags) directly into the peer store via `peer_store.add_addr`, which sets `last_connected_at_ms = 0`. [3](#0-2) 

The attacker therefore advertises `/ip4/<public_ip>/tcp/<port>/p2p/<FAKE_PEER_ID>` with flags `SYNC|DISCOVERY|RELAY` via Discovery. Because `FAKE_PEER_ID` is not in `connected_peers` and `last_connected_at_ms == 0`, `fetch_nat_addrs` returns it, and `notify()` inserts `FAKE_PEER_ID` into `inflight_requests`. [4](#0-3) 

**Step 2 — `listen_addrs` validation is insufficient.**

The only check on `listen_addrs` is that any embedded PeerId must equal `content.to`. Since `content.to` is `FAKE_PEER_ID` (which the attacker generated), the attacker freely embeds it alongside their actual listener IP:port. [5](#0-4) 

**Step 3 — `execute()` reaches the NAT traversal branch.**

With `route` empty and `content.from` set to the victim's own public PeerId, the condition `self_peer_id != &content.from` is false. The code then calls `inflight_requests.remove(&content.to)`, which succeeds for `FAKE_PEER_ID`, and proceeds to `try_nat_traversal`. [6](#0-5) 

**Step 4 — `try_nat_traversal` connects to the attacker's listener unconditionally.**

`try_nat_traversal` retries TCP connections for up to 30 seconds. On success, `raw_session` is called with `RawSessionInfo::outbound(Identify)`. [7](#0-6) 

When `bind_addr` is `None` (always on non-Linux; on Linux unless `reuse_port_on_linux` is set), `create_socket` still creates a functional unbound socket that connects to the attacker's endpoint. [8](#0-7) 

**Step 5 — Identify response poisons the peer store.**

`add_remote_listen_addrs` unconditionally calls `peer_store.add_addr` for every address in the attacker's Identify response. [9](#0-8) 

The only guard in `received_identify` is a network-name check (`verify`), which the attacker passes by using the correct public CKB network name. [10](#0-9) 

## Impact Explanation

The victim node establishes an outbound TCP connection to an attacker-controlled endpoint and runs the Identify protocol. The attacker controls the full Identify response, injecting arbitrary addresses into the victim's peer store. Repeated execution (the attacker can re-seed `inflight_requests` on each 5-minute `notify()` tick) allows progressive peer-store poisoning, enabling an eclipse attack that isolates the victim from the honest network. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as a successfully eclipsed node can be used to suppress block/transaction propagation and degrade network health.

## Likelihood Explanation

The attack requires only a standard P2P connection and the ability to send Discovery and HolePunching protocol messages — both available to any connected peer. The victim's PeerId is public via Identify. The `inflight_requests` window is 5 minutes, giving ample time to deliver the crafted message after each `notify()` tick. No special privileges, leaked keys, or majority hashpower are required. The attack works on all platforms (not just Linux with `reuse_port_on_linux=true`) because `bind_addr = None` still allows outbound connections.

## Recommendation

1. **Authenticate the sender**: Verify that a `ConnectionRequestDelivered` message arrives from the session whose PeerId matches `content.to`. Bind each `inflight_requests` entry to the specific session the `ConnectionRequest` was sent toward, and reject `ConnectionRequestDelivered` messages arriving from any other session.
2. **Cross-check `listen_addrs`**: Validate that the IP:port in `listen_addrs` matches the observed remote address of the connected peer that sent the message, rather than accepting arbitrary values.
3. **Scope `inflight_requests` entries**: Store `(PeerId, expected_session_id)` pairs instead of bare `PeerId` timestamps, so a crafted message from a different session cannot consume the entry.

## Proof of Concept

1. Attacker node `A` connects to victim `V`.
2. `A` sends a Discovery message advertising `/ip4/<A_public_ip>/tcp/<A_port>/p2p/<FAKE_PEER_ID>` with flags `SYNC|DISCOVERY|RELAY`. `FAKE_PEER_ID` is a freshly generated key pair controlled by `A`.
3. Wait for `V`'s `notify()` to fire (≤5 minutes). `FAKE_PEER_ID` is now in `V.inflight_requests`.
4. `A` opens a TCP listener on `<A_public_ip>:<A_port>`.
5. `A` sends `HolePunchingMessage::ConnectionRequestDelivered` to `V` with:
   - `from` = `V`'s PeerId (obtained from Identify)
   - `to` = `FAKE_PEER_ID`
   - `route` = `[]`, `sync_route` = `[]`
   - `listen_addrs` = `[/ip4/<A_public_ip>/tcp/<A_port>/p2p/<FAKE_PEER_ID>]`
6. `V` processes the message: `route` is empty, `content.from == self_peer_id`, `inflight_requests.remove(FAKE_PEER_ID)` succeeds → `try_nat_traversal` is called.
7. `V` connects to `<A_public_ip>:<A_port>`; `A`'s listener accepts.
8. `V` calls `raw_session(stream, addr, RawSessionInfo::outbound(Identify))`.
9. `A` sends a crafted Identify response with the correct CKB network name and attacker-chosen `listen_addrs`.
10. **Assert**: `V`'s peer store now contains attacker-chosen addresses. Repeat from step 3 on each subsequent `notify()` tick to progressively fill the peer store.

    pub fn fetch_nat_addrs(&mut self, count: usize, required_flags: Flags) -> Vec<AddrInfo> {
        // Get info:
        // 1. Never connected
        // 2. Not already connected
        // 3. Ip4 / Ip6 address only

        let peers = &self.connected_peers;

        let filter = |peer_addr: &AddrInfo| {
            required_flags_filter(required_flags, Flags::from_bits_truncate(peer_addr.flags))
                && extract_peer_id(&peer_addr.addr)
                    .map(|peer_id| !peers.contains_key(&peer_id))
                    .unwrap_or_default()
                && peer_addr.addr.iter().any(|p| {
                    matches!(
                        p,
                        p2p::multiaddr::Protocol::Ip4(_) | p2p::multiaddr::Protocol::Ip6(_)
                    )
                })
                && peer_addr.last_connected_at_ms == 0
        };

        self.addr_manager.fetch_random(count, filter)
    }
