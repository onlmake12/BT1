### Title
Lazer `verifyUpdate` Never Validates Payload Price Timestamp Against `block.timestamp`, Enabling Stale-Price Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` authenticates the ECDSA signature and checks that the signing key has not expired, but it never validates the **price timestamp embedded inside the payload** against `block.timestamp`. Any previously broadcast, validly-signed Lazer payload can be replayed at a later time and will pass `verifyUpdate` unchanged, allowing an attacker to feed a consumer protocol a stale price that was signed in the past.

---

### Finding Description

`verifyUpdate` in `PythLazer.sol` performs two checks:

1. The ECDSA signature over the payload is valid and recovers to a non-zero address.
2. The recovered signer address has not expired: `block.timestamp < trustedSignerToExpiresAtMapping[signer]`.

```solidity
// lazer/contracts/evm/src/PythLazer.sol  lines 66-106
function isValidSigner(address signer) public view returns (bool) {
    return block.timestamp < trustedSignerToExpiresAtMapping[signer];
}

function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    ...
    payload = update[71:71 + payload_len];
    bytes32 hash = keccak256(payload);
    (signer, , ) = ECDSA.tryRecover(hash, ...);
    if (signer == address(0)) { revert("invalid signature"); }
    if (!isValidSigner(signer)) { revert("invalid signer"); }
    // ← no check: payload timestamp vs block.timestamp
}
```

The payload itself contains a price timestamp (the Lazer feed's publish time). That timestamp is returned opaquely inside `payload` to the caller, but `verifyUpdate` never parses it or compares it to `block.timestamp`. There is also no nonce, sequence number, or one-time-use mechanism in the contract.

Because Lazer payloads are broadcast publicly (they are meant to be submitted by any relayer), any observer can capture a signed payload at time T and re-submit it at time T+N. The contract will accept it as long as the signer key has not expired. Signer keys are configured with long validity windows (the test fixture uses `3000000000000000` as the expiry value).

The Sui implementation passes a `Clock` object into `verify_le_ecdsa_message` and checks `clock.timestamp_ms() < expires_at_ms` for the signer, but similarly never validates the payload's internal price timestamp against the clock. The Aptos implementation has the same pattern.

---

### Impact Explanation

A consumer protocol that calls `verifyUpdate` and then uses the price from the returned payload will receive a cryptographically "verified" but arbitrarily stale price. An attacker who:

1. Observes a Lazer payload at time T when the price is P (favorable for the attack),
2. Waits until the real price moves to P' (unfavorable for the target protocol),
3. Replays the old payload through `verifyUpdate`,

can cause the consumer to execute trades, liquidations, or settlements at the stale price P. This is directly analogous to the Buffer Protocol issue: the signed data's timestamp is never validated against the execution context, so the signed data can be used at a time the signer never intended.

Financial loss to consumer protocols and their users is the concrete impact.

---

### Likelihood Explanation

- Lazer payloads are publicly observable (broadcast to all subscribers/relayers).
- No on-chain replay protection exists in `PythLazer.sol`.
- Signer keys have long validity windows.
- The attacker needs only to be a standard EVM transaction sender — no privileged role is required.
- Consumer protocols that rely solely on `verifyUpdate` for freshness assurance (a reasonable assumption given the function's name and purpose) are fully exposed.

---

### Recommendation

`verifyUpdate` should parse the price timestamp from the payload and require it to be within an acceptable staleness window relative to `block.timestamp`, for example:

```solidity
uint64 payloadTimestampMs = parseTimestampFromPayload(payload);
require(
    block.timestamp * 1000 <= payloadTimestampMs + MAX_STALENESS_MS,
    "stale payload"
);
require(
    payloadTimestampMs <= block.timestamp * 1000 + MAX_FUTURE_MS,
    "future payload"
);
```

Alternatively, introduce a monotonically increasing sequence number or nonce into the payload and track consumed nonces on-chain to prevent replay entirely.

---

### Proof of Concept

1. At block time T, relayer submits a valid Lazer update. The payload contains price `P` and timestamp `T`.
2. Attacker captures the raw `update` bytes from the mempool or event logs.
3. Price moves to `P'` (e.g., 10% lower) at block time `T + 3600`.
4. Attacker calls `PythLazer.verifyUpdate{value: fee}(capturedUpdate)`.
5. `isValidSigner` passes because the signer key expiry (e.g., year 2100) is still in the future.
6. `verifyUpdate` returns the original `payload` containing price `P` and timestamp `T`.
7. The consumer contract receives a "verified" payload and executes logic (e.g., opens a position, avoids a liquidation) using the 1-hour-old price `P` instead of the current price `P'`.
