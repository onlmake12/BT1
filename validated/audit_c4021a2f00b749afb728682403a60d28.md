### Title
Hard-Coded `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` Is Too Loose for Tight-Heartbeat Scheduler Subscriptions — (File: `target_chains/ethereum/pulse_sdk/solidity/SchedulerConstants.sol`)

---

### Summary

The Pyth Pulse Scheduler enforces a single hard-coded `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` for **all** subscriptions, regardless of their configured heartbeat interval. Because `updatePriceFeeds()` is callable by any unprivileged address, an attacker can obtain a valid but stale Pyth price update (up to 59 minutes old) from the Hermes historical API and submit it to a subscription that expects 60-second updates. The Scheduler accepts the stale timestamp, stores the old price, and serves it to downstream readers — exactly mirroring the Deriverse pattern where a single hard-coded tolerance was too loose for a higher-risk context.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
``` [1](#0-0) 

This constant is consumed in `Scheduler._validateShouldUpdatePrices()`:

```solidity
uint256 minAllowedTimestamp = (block.timestamp >
    PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;

if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [2](#0-1) 

The `updatePriceFeeds()` entry point has **no access control** — any address may call it. Internally it calls `parsePriceFeedUpdatesWithConfig` with `minPublishTime = 0`, so the Pyth contract will parse and return the price from any submitted update whose timestamp falls in `[0, block.timestamp + 10s]`:

```solidity
(PythStructs.PriceFeed[] memory priceFeeds, uint64[] memory slots) =
    pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
        updateData,
        params.priceIds,
        0,   // minPublishTime — no lower bound enforced by Pyth
        curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
        false, true, false
    );
``` [3](#0-2) 

The only lower-bound check is the 1-hour window above. There is **no per-subscription tightening** of this window. A subscription configured with `heartbeatSeconds = 60` (1 minute) is subject to the same 1-hour acceptance window as a subscription with `heartbeatSeconds = 3600`.

`_validateSubscriptionParams()` validates `deviationThresholdBps != 0` and `heartbeatSeconds != 0`, but imposes **no upper bound** on either field and **no relationship** between `heartbeatSeconds` and the protocol-level timestamp window:

```solidity
if (
    params.updateCriteria.updateOnHeartbeat &&
    params.updateCriteria.heartbeatSeconds == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [4](#0-3)

### Impact Explanation

A subscription with `heartbeatSeconds = 60` is intended to serve prices no older than ~60 seconds. Because the Scheduler accepts any update whose `publishTime` is within the last hour, an attacker can:

1. Fetch a valid Pyth accumulator update from 55 minutes ago via the Hermes historical endpoint.
2. Call `updatePriceFeeds(subscriptionId, updateData)` — no privilege required.
3. The Scheduler stores the 55-minute-old price and sets `priceLastUpdatedAt` to that stale timestamp.
4. Downstream readers calling `getPricesUnsafe` or `getEmaPricesUnsafe` receive a price that is 55 minutes stale.

For volatile crypto assets, a 55-minute-old price can differ by 5–15 % from the current market price. DeFi protocols (lending, perps, structured products) that consume this subscription for liquidation thresholds or collateral valuation would make incorrect decisions — exactly the leveraged-loss scenario described in the reference report, but expressed as oracle staleness rather than order-fill slippage. [5](#0-4) 

---

### Likelihood Explanation

- `updatePriceFeeds()` is permissionless; no role or key is required.
- The Hermes REST API (`/v2/updates/price/{publish_time}`) publicly serves signed historical accumulator updates for any past timestamp, making the stale payload trivially obtainable.
- The only prerequisite is that the target subscription's `priceLastUpdatedAt` is older than the attacker's chosen stale timestamp (i.e., the subscription has not been refreshed more recently than the stale update). This is realistic whenever a keeper is slow, a chain is congested, or the subscription is newly created.
- No flash loan, governance key, or Sybil attack is required.

---

### Recommendation

1. **Per-subscription validity cap**: Add an optional `maxPastTimestampSeconds` field to `SubscriptionParams` and enforce `updateTimestamp >= block.timestamp - min(PAST_TIMESTAMP_MAX_VALIDITY_PERIOD, params.maxPastTimestampSeconds)` in `_validateShouldUpdatePrices`. This mirrors the "allow users to set a custom slippage limit, with a protocol-defined maximum" recommendation from the reference report.

2. **Heartbeat-aware floor**: As a simpler alternative, enforce that the accepted timestamp window is at most `max(heartbeatSeconds * N, MINIMUM_FLOOR)` for subscriptions with `updateOnHeartbeat = true`, so a 60-second heartbeat subscription cannot accept a 55-minute-old price.

3. **Tighten the protocol default**: For always-open crypto markets, 1 hour is far looser than necessary. A default of 5–10 minutes would still accommodate brief keeper outages while dramatically reducing the stale-price attack surface.

---

### Proof of Concept

```
1. Deploy / identify a Scheduler subscription with:
     heartbeatSeconds = 60, updateOnHeartbeat = true
     priceLastUpdatedAt = T-70min  (subscription not recently refreshed)

2. Query Hermes historical API:
     GET /v2/updates/price/{T-55min}?ids[]=<priceId>
   → returns a valid signed accumulator update with publishTime = T-55min

3. Call (from any EOA):
     scheduler.updatePriceFeeds(subscriptionId, [stalePythUpdateData])

4. _validateShouldUpdatePrices checks:
     updateTimestamp (T-55min) >= block.timestamp - 1 hour  ✓  (55 min < 60 min)
     updateTimestamp (T-55min) >  priceLastUpdatedAt (T-70min) ✓

5. Subscription stores price from T-55min; priceLastUpdatedAt = T-55min.

6. Reader calls getPricesUnsafe(subscriptionId, priceIds)
   → receives a price 55 minutes stale, indistinguishable from a fresh update
     unless the reader independently checks publishTime.
``` [6](#0-5) [7](#0-6)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

        // Verify all price feeds have the same Pythnet slot.
        // All feeds in a subscription must be updated at the same time.
        uint64 slot = slots[0];
        for (uint8 i = 1; i < slots.length; i++) {
            if (slots[i] != slot) {
                revert SchedulerErrors.PriceSlotMismatch();
            }
        }

        // Verify that update conditions are met, and that the timestamp
        // is more recent than latest stored update's. Reverts if not.
        uint256 latestPublishTime = _validateShouldUpdatePrices(
            subscriptionId,
            params,
            status,
            priceFeeds
        );

        // Update status and store the updates
        status.priceLastUpdatedAt = latestPublishTime;
        status.totalUpdates += priceFeeds.length;

        _storePriceUpdates(subscriptionId, priceFeeds);

        _processFeesAndPayKeeper(status, startGas, params.priceIds.length);

        emit PricesUpdated(subscriptionId, latestPublishTime);
    }
