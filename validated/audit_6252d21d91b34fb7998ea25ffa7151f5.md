### Title
`n_sync_started` Counter Permanently Leaked via TOCTOU Race in `start_sync_headers`, Blocking IBD — (`File: sync/src/synchronizer/mod.rs`)

---

### Summary

In `start_sync_headers`, the global `n_sync_started` atomic counter is incremented **before** the corresponding peer state is transitioned to `Started`. If a peer disconnects in the window between these two non-atomic operations, the counter is permanently incremented with no corresponding decrement and no recovery path. During Initial Block Download (IBD), this permanently prevents the node from selecting any new sync peer, stalling chain synchronization indefinitely.

---

### Finding Description

`start_sync_headers` in `sync/src/synchronizer/mod.rs` (lines 652–692) executes the following sequence for each eligible peer:

1. **Collect** eligible peers into a `Vec<PeerIndex>` snapshot (lines 656–662).
2. **Atomically increment** `n_sync_started` via `fetch_update` (lines 672–682).
3. **Separately** call `peer_state.start_sync()` via a `DashMap::get_mut` (lines 683–687).

Steps 2 and 3 are **not atomic with respect to each other**. The `n_sync_started` `AtomicUsize` and the `DashMap<PeerIndex, PeerState>` are independent data structures. In CKB's multi-threaded tokio runtime, the `disconnected` async handler can execute concurrently on a different thread between these two steps.

**Race sequence:**

| Thread A (`notify` → `start_sync_headers`) | Thread B (`disconnected`) |
|---|---|
| Collects peer P in eligible list | |
| `n_sync_started.fetch_update(…)` → increments to 1 | |
| | `state.remove(&P)` → removes P from DashMap |
| | `peer_state.sync_started()` → `false` (not yet `Started`) |
| | Does **not** decrement `n_sync_started` |
| `state.get_mut(&P)` → `None` (P already removed) | |
| `start_sync` **never called** | |

After this race, `n_sync_started == 1` but no peer is in the `Started` state. There is no recovery path: `suspend_sync`, `tip_synced`, and `disconnected` all guard their decrements with `if peer_state.sync_started()`, and the eviction loop only iterates over peers still present in `state`.

The `disconnected` handler in `sync/src/types/mod.rs` (lines 901–924) only decrements `n_sync_started` when `peer_state.sync_started()` is `true` at the moment of removal. Since `start_sync` was never called, this guard is `false` and the decrement is skipped.

---

### Impact Explanation

During IBD, `start_sync_headers` enforces single-peer sync via:

```rust
if ibd && x != 0 { None } else { Some(x + 1) }
```

If `n_sync_started` is stuck at 1 (or any non-zero value), the `fetch_update` returns `Err` for every subsequent call during IBD, causing `break` immediately. No new sync peer is ever selected. The node cannot download headers or blocks and IBD is permanently stalled until the process is restarted. This is a **denial of sync** against a node in IBD — the most critical phase of node operation.

---

### Likelihood Explanation

The race window is narrow (two consecutive non-awaited operations), but:

- `start_sync_headers` is called on every `SEND_GET_HEADERS_TOKEN` timer tick, giving repeated opportunities.
- A malicious peer controls exactly when it disconnects and can time the disconnect to coincide with the tick.
- The attack requires only an unprivileged inbound or outbound peer connection — no keys, no stake, no special privilege.
- The attack can be retried cheaply until it succeeds.

---

### Recommendation

Atomically combine the counter increment and the peer-state transition. One approach: move the `n_sync_started` increment **inside** the `DashMap::get_mut` critical section so that if the peer is absent (already disconnected), the counter is never incremented:

```rust
for peer in peers {
    let mut incremented = false;
    if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
        // Only increment if the peer is still present
        if self.shared().state().n_sync_started()
            .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
                if ibd && x != 0 { None } else { Some(x + 1) }
            }).is_err()
        {
            break;
        }
        peer_state.start_sync(HeadersSyncController::from_header(&tip));
        incremented = true;
    }
    if !incremented { continue; }
    // ...
}
```

Alternatively, add a periodic sanity check that recomputes `n_sync_started` from the actual peer states and corrects any drift.

**`started()` returns `true` only for `HeadersSyncState::Started` — which is never set in the race:** [3](#0-2) 

**IBD guard that permanently blocks on non-zero `n_sync_started`:** [4](#0-3) 

**`suspend_sync` and `tip_synced` also guard on `sync_started()` — no recovery path:** [5](#0-4) 

**Attack steps:**
1. Attacker peer connects to a victim node that is in IBD.
2. Victim's `start_sync_headers` is triggered by the periodic `SEND_GET_HEADERS_TOKEN` timer.
3. Attacker disconnects precisely after `n_sync_started.fetch_update` increments the counter but before `peer_state.start_sync()` is called.
4. `n_sync_started` is now 1; no peer is in `Started` state; no eviction or cleanup will ever decrement it.
5. All subsequent `start_sync_headers` calls during IBD immediately `break` because `ibd && x != 0` causes `fetch_update` to return `Err`.
6. The victim node's IBD is permanently stalled — it cannot download any further headers or blocks until restarted.
