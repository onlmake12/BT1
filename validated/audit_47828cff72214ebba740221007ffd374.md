Audit Report

## Title
`AddBlockFilterHash::migrate` Panics on Startup After Reorg When Main-Chain Filter Data at Fork Height Is Absent — (`util/migrate/src/migrations/add_block_filter_hash.rs`)

## Summary

The fork-chain walk in `AddBlockFilterHash::migrate` exits with `header` pointing to the first diverging fork block at height K and returns `header.number()` (= K) as `latest_built_filter_data_block_number`. The migration loop then unconditionally calls `.expect("filter data stored")` on the main-chain block at height K. If the node shut down after a reorg was committed to `COLUMN_INDEX` but before the filter service wrote the new main-chain filter at K, `get_block_filter` returns `None` and the process panics on every subsequent startup attempt.

## Finding Description

**Fork-chain walk returns K, not K-1.**

`util/migrate/src/migrations/add_block_filter_hash.rs` lines 29–37:
```rust
let mut header = chain_db.get_block_header(&block_hash).expect("header stored");
while !chain_db.is_main_chain(&header.parent_hash()) {
    header = chain_db.get_block_header(&header.parent_hash()).expect("parent header stored");
}
header.number()   // K: first fork block, NOT K-1 (last shared ancestor)
```
The loop exits when `header.parent_hash()` is on the main chain, so `header` is the first block that diverged (height K). Its parent at K-1 is the last shared ancestor. `header.number()` = K is one too high.

**`is_main_chain` is keyed on block hash in `COLUMN_INDEX`** (`store/src/store.rs` line 279–281). After a reorg, `COLUMN_INDEX[number=K]` maps to the new main-chain hash at K; the fork block hash at K has no entry, so `is_main_chain(fork_hash_K)` = false — confirming the walk correctly identifies the fork block but returns its height K.

**Migration loop reads main-chain filter data at height K** (lines 61–64):
```rust
let block_hash = chain_db.get_block_hash(block_number).expect("index stored");
let filter_data = chain_db.get_block_filter(&block_hash).expect("filter data stored");
```
`get_block_hash(K)` returns the new main-chain hash at K. `get_block_filter` reads `COLUMN_BLOCK_FILTER` keyed by that hash. If the node shut down after the reorg was committed to `COLUMN_INDEX` but before `insert_block_filter` wrote the new main-chain block at K (`store/src/transaction.rs` lines 392–414), `COLUMN_BLOCK_FILTER[main_chain_hash_K]` is absent, `get_block_filter` returns `None`, and `.expect("filter data stored")` panics unconditionally.

**The `build_filter_data_for_block` guard does not protect the migration.** The guard in `block-filter/src/filter.rs` lines 131–137 checks `COLUMN_BLOCK_FILTER_HASH` and only runs during the live filter service. The migration has no equivalent guard and panics instead of skipping or clamping.

## Impact Explanation

The node process terminates with a panic during the mandatory `AddBlockFilterHash` migration on every startup attempt. The node cannot start at all. Recovery requires manual RocksDB surgery or a full re-sync. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node** — the crash is permanent (not transient) and affects every node meeting the preconditions.

## Likelihood Explanation

All three preconditions are routine operational events on a PoW chain: block filter is commonly enabled on nodes serving light clients (v0.105+), reorgs occur naturally, and the window between a reorg being committed to `COLUMN_INDEX` and the filter service writing the first new main-chain block is non-zero — any SIGTERM, OOM kill, power loss, or graceful shutdown during that window produces the vulnerable DB state. No attacker action is required; the state arises from normal node operation. Every such node is permanently broken on upgrade to v0.108+.

## Recommendation

Change the fork-chain walk return value from `header.number()` (K) to `header.number().saturating_sub(1)` (K-1) so the migration loop only covers blocks 0 through K-1, all of which have guaranteed main-chain filter data:

```rust
// Before (buggy): returns K — the fork block height
header.number()

// After (correct): returns K-1 — the last common ancestor
header.number().saturating_sub(1)
```

The identical walk in `block-filter/src/filter.rs` lines 91–104 intentionally returns K as a *start* for rebuilding (it needs to rebuild from K on the new main chain). The migration's use of the same value as an *upper bound* for already-built data is the bug; only the migration needs the fix.

## Proof of Concept

1. Construct a RocksDB fixture with:
   - `COLUMN_META[META_LATEST_BUILT_FILTER_DATA_KEY]` = `fork_block_hash_K` (not in `COLUMN_INDEX`)
   - `COLUMN_BLOCK_HEADER[fork_block_hash_K]` = valid header at height K, parent = `main_chain_hash_{K-1}`
   - `COLUMN_INDEX[main_chain_hash_{K-1}]` = K-1 (parent is on main chain)
   - `COLUMN_INDEX[K]` = `main_chain_hash_K` (different from `fork_block_hash_K`)
   - `COLUMN_BLOCK_FILTER[fork_block_hash_K]` = valid filter bytes
   - `COLUMN_BLOCK_FILTER[main_chain_hash_K]` = **absent**
   - `COLUMN_BLOCK_FILTER` entries for blocks 0 through K-1 on the main chain = present
2. Run `AddBlockFilterHash::migrate` against this DB.
3. Observe panic: `thread 'main' panicked at 'filter data stored'` when `block_number == K`, because `get_block_hash(K)` returns `main_chain_hash_K` and `get_block_filter(main_chain_hash_K)` returns `None`.
