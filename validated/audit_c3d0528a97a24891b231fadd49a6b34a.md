### Title
No Refund Mechanism for Unfulfilled Requests Permanently Locks User-Paid Provider Fees in Echo Contract - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, when a user calls `requestPriceUpdatesWithCallback`, the provider fee portion of `msg.value` is stored in the request struct as `req.fee`. If `executeCallback` is never called for that request (provider goes offline, price data unavailable for the requested timestamp, etc.), the ETH stored in `req.fee` is permanently locked in the contract. There is no `cancelRequest`, `refundRequest`, or any other mechanism for the user to reclaim these funds, and neither the admin's `withdrawFees` nor the fee manager's `withdrawAsFeeManager` covers unfulfilled request fees.

---

### Finding Description

When `requestPriceUpdatesWithCallback` is called, the fee is split into two parts:

```solidity
// Echo.sol line 84
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
// Echo.sol line 99
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

- `_state.accruedFeesInWei` (Pyth protocol fee) → withdrawable by admin via `withdrawFees`
- `req.fee` (provider fee) → only credited to `providers[providerToCredit].accruedFeesInWei` inside `executeCallback`

```solidity
// Echo.sol lines 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
clearRequest(sequenceNumber);
``` [2](#0-1) 

If `executeCallback` is never called, `req.fee` ETH sits in the contract balance indefinitely. The two withdrawal paths available are:

1. `withdrawFees(uint128 amount)` — admin only, drains `_state.accruedFeesInWei` (Pyth protocol fees only)
2. `withdrawAsFeeManager(address provider, uint128 amount)` — fee manager only, drains `providers[provider].accruedFeesInWei` (provider accrued fees only) [3](#0-2) [4](#0-3) 

Neither path touches `req.fee` stored in unfulfilled requests. The contract's own developer comments acknowledge this gap:

```solidity

// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
// TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
``` [5](#0-4) 

The `EchoState.sol` confirms `req.fee` is a `uint96` field stored per-request with no separate accounting variable tracking total locked provider fees: [6](#0-5) 

---

### Impact Explanation

Any ETH paid as provider fee (`msg.value - pythFeeInWei`) for a request that is never fulfilled via `executeCallback` is permanently locked in the Echo contract. There is no on-chain path — for the user, the admin, or the provider — to recover these funds. Over time, as requests go unfulfilled (provider downtime, stale `publishTime` windows, etc.), the locked ETH accumulates with no recovery mechanism.

---

### Likelihood Explanation

The `publishTime` constraint (`publishTime <= block.timestamp + 60`) means requests are time-sensitive. If a provider is temporarily offline or slow, the price data for the exact `publishTime` may become unavailable from Hermes, making `executeCallback` impossible to call successfully. Additionally, the exclusivity period means only the assigned provider can fulfill the request during the initial window, increasing the chance of non-fulfillment if that specific provider is unresponsive. Any user interacting with Echo faces this risk on every request.

---

### Recommendation

Add a `cancelRequest` function that allows the original requester to reclaim `req.fee` after a timeout period (e.g., after the exclusivity period has elapsed and the request remains unfulfilled). The Pyth protocol fee (`pythFeeInWei`) already credited to `accruedFeesInWei` should be retained, but `req.fee` should be refunded to `req.requester`:

```solidity
function cancelRequest(uint64 sequenceNumber) external {
    Request storage req = findActiveRequest(sequenceNumber);
    require(msg.sender == req.requester, "Only requester can cancel");
    require(
        block.timestamp > req.publishTime + _state.exclusivityPeriodSeconds + CANCEL_TIMEOUT,
        "Too early to cancel"
    );
    uint96 refundAmount = req.fee;
    clearRequest(sequenceNumber);
    (bool sent, ) = req.requester.call{value: refundAmount}("");
    require(sent, "Refund failed");
}
```

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback{value: 1 ether}(provider, publishTime, priceIds, gasLimit)`.
2. Contract stores `req.fee = 1 ether - pythFeeInWei` and increments `accruedFeesInWei += pythFeeInWei`.
3. Provider goes offline; `executeCallback` is never called for this `sequenceNumber`.
4. Admin calls `withdrawFees(accruedFeesInWei)` — only recovers `pythFeeInWei`, not `req.fee`.
5. `req.fee` ETH (the bulk of the user's payment) remains locked in the contract with no callable function to retrieve it.
6. Confirmed by inspecting all external functions in `IEcho.sol` — none provide a refund path for the requester. [7](#0-6)
