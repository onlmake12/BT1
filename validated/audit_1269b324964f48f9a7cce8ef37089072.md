### Title
Entropy Provider Can Front-Run User Requests to Extract Inflated Fees Without Delay or Refund - (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.setProviderFee` allows any registered provider to update their `feeInWei` at any time with no delay, no timelock, and no upper bound. Because `requestHelper` reads the live fee from storage at execution time, and because excess `msg.value` is never refunded to the caller, a malicious provider can front-run a user's request transaction to extract more ETH than the user intended to pay.

---

### Finding Description

`setProviderFee` in `Entropy.sol` immediately overwrites `provider.feeInWei` with no constraints:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
    ...
}
``` [1](#0-0) 

When a user calls `requestWithCallback` or `requestV2`, `requestHelper` computes the required fee by reading the current on-chain `feeInWei` at that moment:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [2](#0-1) 

Critically, **excess `msg.value` is never refunded** — it is silently credited to the Pyth protocol fee pool, not returned to the user. The `IEntropy` interface comment confirms this: *"Note that excess value is not refunded to the caller."* [3](#0-2) 

The `getProviderFee` helper reads `provider.feeInWei` live from storage, so any fee change that lands before the user's request is immediately effective: [4](#0-3) 

**Attack scenario:**

1. Provider advertises `feeInWei = 0.001 ETH` to attract users.
2. User calls `getFee(provider)` off-chain → receives `0.001 ETH`.
3. User submits `requestWithCallback{value: 0.002 ETH}(provider, ...)` with a small buffer to avoid reverts.
4. Provider monitors the mempool, sees the pending request, and front-runs it with `setProviderFee(0.002 ETH - 1 wei)` — just below the user's `msg.value`.
5. User's transaction executes: the inflated fee is deducted, the provider accrues `0.002 ETH - 1 wei`, and the user receives no refund.
6. Provider calls `setProviderFee(0.001 ETH)` to reset and attract the next victim.

The same attack applies to `setProviderFeeAsFeeManager`, which the Fortuna keeper uses: [5](#0-4) 

---

### Impact Explanation

A malicious Entropy provider can systematically extract the full `msg.value` buffer from every user request, earning significantly more than the advertised fee. Users have no on-chain mechanism to specify a maximum acceptable fee (no slippage parameter), and excess ETH is permanently lost to them. Over time this drains user funds and erodes trust in the Entropy service.

---

### Likelihood Explanation

Registration as an Entropy provider is permissionless — anyone can call `register`. A malicious actor can register, advertise a competitive fee, and run an automated front-running bot. The attack requires only standard mempool visibility (available on all public EVM chains where Entropy is deployed) and two cheap transactions per victim. No privileged access, leaked key, or governance majority is needed. [6](#0-5) 

---

### Recommendation

1. **Introduce a fee-update delay**: Store a `pendingFee` and a `feeEffectiveAt` timestamp. Only apply the new fee after a minimum delay (e.g., 24 hours), giving users time to observe the change before submitting requests.
2. **Add a user-supplied `maxFee` parameter** to `requestWithCallback`/`requestV2` that causes the transaction to revert if the current fee exceeds the user's expectation.
3. **Refund excess `msg.value`** to the caller rather than crediting it to the Pyth fee pool, so that users who send a buffer are not penalized.

### Proof of Concept

```solidity
function test_provider_frontRuns_request_with_higher_fee() public {
    // Setup: provider registered with low fee
    uint128 advertisedFee = 0.001 ether;
    uint128 inflatedFee   = 0.002 ether - 1;

    vm.prank(provider1);
    random.setProviderFee(advertisedFee);

    // User queries fee off-chain and sends a small buffer
    uint128 userPayment = 0.002 ether;
    vm.deal(user1, userPayment);

    // Provider front-runs: bumps fee to just below user's msg.value
    vm.prank(provider1);
    random.setProviderFee(inflatedFee);

    // User's request executes at the inflated fee
    vm.prank(user1);
    random.requestWithCallback{value: userPayment}(
        provider1,
        bytes32(uint256(42))
    );

    // Provider accrued the inflated fee, not the advertised one
    EntropyStructsV2.ProviderInfo memory info = random.getProviderInfoV2(provider1);
    assertEq(info.accruedFeesInWei, inflatedFee); // provider extracted max

    // Provider resets fee to attract next user
    vm.prank(provider1);
    random.setProviderFee(advertisedFee);
}
```

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
