### Title
Missing `chainId` in Signed Payload Hash Enables Cross-Chain Replay of Lazer Price Updates — (`File: lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` computes the signature hash as `keccak256(payload)` with no chain ID or contract address bound into the signed data. Because Pyth Lazer is deployed on multiple EVM chains sharing the same trusted signer keys, a valid signed update from one chain is cryptographically indistinguishable from a valid update on any other chain. An unprivileged relayer can capture a price update from chain A and replay it on chain B.

---

### Finding Description

In `verifyUpdate()`, the only data committed to the signature is the raw payload bytes:

```solidity
// lazer/contracts/evm/src/PythLazer.sol  line 93
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The payload itself (parsed in `PythLazerLib.parsePayloadHeader`) contains only: a format magic, a timestamp, a channel byte, and feed data. None of these fields are chain-specific.

```solidity
// lazer/contracts/evm/src/PythLazerLib.sol  lines 122-135
uint32 FORMAT_MAGIC = 2479346549;
// ...
timestamp = _readBytes8(update, pos);   // microsecond timestamp
channel = PythLazerStructs.Channel(...); // RealTime / FixedRate50 / etc.
feedsLen = ...;
``` [2](#0-1) 

There is no `block.chainid`, no contract address, and no EIP-712 domain separator anywhere in the hash construction. The outer EVM wrapper magic (`EVM_FORMAT_MAGIC = 706910618`) is also chain-agnostic — it is a format discriminator, not a chain discriminator. [3](#0-2) 

The `PythLazerStructs.Update` struct confirms no chain-binding field exists in the payload:

```solidity
struct Update {
    uint64 timestamp;
    Channel channel;
    Feed[] feeds;
}
``` [4](#0-3) 

---

### Impact Explanation

Pyth Lazer is deployed on multiple EVM chains (Ethereum, Arbitrum, Base, Optimism, etc.) using the same set of trusted signer keys registered via `updateTrustedSigner`. Because the signature hash is chain-agnostic, a signed update accepted on chain A is equally valid on chain B.

An attacker can:
1. Observe a valid `verifyUpdate` call on chain A (e.g., Ethereum mainnet).
2. Extract the `update` bytes from the calldata.
3. Submit the identical bytes to `PythLazer.verifyUpdate()` on chain B (e.g., Arbitrum).
4. The call succeeds — `ECDSA.tryRecover` returns the same trusted signer address, `isValidSigner` passes.

The returned `payload` is then consumed by downstream DeFi protocols. If the price on chain A at the time of signing differs from the current price on chain B (e.e., due to latency, different liquidity, or deliberate timing), the attacker injects a stale or off-market price into chain B's oracle consumers. This can be used to manipulate liquidations, borrow limits, or derivative settlements on any protocol that calls `verifyUpdate` and trusts the returned payload. [5](#0-4) 

---

### Likelihood Explanation

- **Attacker entry point**: Fully unprivileged. Any address can call `verifyUpdate` with any `bytes calldata update`.
- **Operational cost**: Monitoring public mempool/RPC on one chain and submitting the same calldata on another is trivial.
- **Window of opportunity**: The payload timestamp is not validated inside `PythLazer` itself — timestamp freshness is left to the consuming contract. Many integrations check only `_timestamp > lastTimestamp`, which a replayed update from a few seconds ago on another chain easily satisfies.
- **Precondition**: Pyth Lazer must be live on at least two EVM chains simultaneously, which is already the case.

---

### Recommendation

Bind the signed hash to the specific deployment by including `block.chainid` and `address(this)` in the hash, following EIP-712:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

Or adopt a full EIP-712 domain separator so that the Lazer signing infrastructure can produce chain-specific signatures. The Lazer server must include the target chain ID in the data it signs.

---

### Proof of Concept

1. Deploy `PythLazer` on two local forks (chain ID 1 and chain ID 42161) with the same trusted signer key.
2. On chain 1, call `verifyUpdate{value: fee}(update)` with a valid signed update — it succeeds.
3. On chain 42161, call `verifyUpdate{value: fee}(update)` with the **identical** `update` bytes — it also succeeds, returning the same `payload` and `signer`.
4. Both chains accept the same signature without any modification, confirming the cross-chain replay.

The existing test in `lazer/contracts/evm/test/PythLazer.t.sol` already demonstrates that `verifyUpdate` accepts a hardcoded `update` blob with no chain context — the same blob would pass on any chain ID. [6](#0-5)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
