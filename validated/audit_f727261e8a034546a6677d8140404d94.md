### Title
Minimum Fee Rate Check Uses Only Serialized Size, Not Actual Transaction Weight, Enabling High-Cycle Transactions to Bypass Effective Fee Floor — (`tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` enforces the minimum fee rate using only the transaction's serialized byte size, while the actual transaction weight used everywhere else in the system is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. A transaction submitter can craft a small-serialized-size transaction with near-maximum cycle consumption, pay only `min_fee_rate × tx_size`, pass the admission check, and enter the tx-pool at an effective fee rate up to ~12× below the configured minimum.

---

### Finding Description

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code comment explicitly acknowledges the discrepancy:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [1](#0-0) 

However, the canonical transaction weight used for fee rate calculation everywhere else in the system is defined in `util/types/src/core/tx_pool.rs`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This same weight function is used in `TxEntry::fee_rate()`, `AncestorsScoreSortKey`, `EvictKey`, and the RPC fee-rate statistics collector. [3](#0-2) [4](#0-3) 

The `check_tx_fee` call happens in `pre_check`, before full script verification determines actual cycles. No second fee-rate check using actual weight is performed after cycles are known. [5](#0-4) 

---

### Impact Explanation

An attacker submits a transaction with:
- Small serialized size `S` (e.g., 1,000 bytes)
- Scripts consuming near-maximum cycles `C` (up to `max_tx_verify_cycles = 70,000,000`)
- Fee = `min_fee_rate × S / 1000` (just enough to pass the size-only check)

Concrete numbers with default config (`min_fee_rate = 1,000` shannons/KB):
- `tx_size = 1,000` bytes → `min_fee = 1,000` shannons
- `cycles = 70,000,000` → `cycles_equivalent = 70,000,000 × 0.000_170_571_4 ≈ 11,940` bytes
- Actual weight = `max(1,000, 11,940) = 11,940`
- Effective fee rate = `1,000 × 1,000 / 11,940 ≈ 83.8` shannons/KB

The transaction is admitted at an effective fee rate **~12× below** the configured `min_fee_rate`. This:
1. Allows pool admission bypass — the transaction enters the pool despite violating the effective fee floor
2. Consumes pool space disproportionate to fee paid, potentially displacing legitimate transactions
3. Propagates to peer nodes via relay, consuming network bandwidth at below-minimum cost
4. Distorts pool ordering: once admitted, `TxEntry::fee_rate()` correctly reflects the low weight-based rate, so miners are unlikely to include it — but the pool slot is occupied [6](#0-5) [7](#0-6) 

---

### Likelihood Explanation

Any unprivileged tx-pool submitter (via `send_transaction` RPC or P2P relay) can trigger this. No special access is required. The attacker only needs to:
1. Know the node's `min_fee_rate` (publicly visible via `get_tx_pool_info` RPC)
2. Craft a transaction with a script that loops to consume cycles (trivial with CKB-VM)
3. Submit it with fee = `min_fee_rate × tx_size / 1000`

The condition is always satisfiable as long as `cycles * DEFAULT_BYTES_PER_CYCLES > tx_size`, which holds for any transaction consuming more than ~5,865 cycles per byte of serialized size. [8](#0-7) 

---

### Recommendation

After full script verification completes and actual cycles are known, re-check the fee rate using the true transaction weight:

```rust
let weight = get_transaction_weight(tx_size, actual_cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This second check should be performed in the post-verification submission path (e.g., `submit_entry`) where `cycles` is available, replacing or supplementing the size-only pre-check. [9](#0-8) 

---

### Proof of Concept

1. Query `min_fee_rate` via RPC: `get_tx_pool_info` → e.g., `1000` shannons/KB.
2. Construct a transaction:
   - One input cell consuming a live cell
   - One output cell (same capacity minus fee)
   - Lock script: a tight loop consuming exactly `69,000,000` cycles
   - Serialized size: ~500 bytes
3. Set fee = `1000 × 500 / 1000 = 500` shannons.
4. Submit via `send_transaction`.
5. Observe: transaction is accepted into the pool.
6. Query `get_transaction` and inspect the entry: effective fee rate = `500 × 1000 / max(500, 69,000,000 × 0.000_170_571_4)` ≈ `500,000 / 11,769` ≈ **42 shannons/KB** — far below the 1,000 shannons/KB minimum.
7. The transaction occupies a pool slot and is relayed to peers, all at ~24× below the intended minimum fee rate. [1](#0-0) [10](#0-9)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
