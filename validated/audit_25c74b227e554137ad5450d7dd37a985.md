### Title
PostgreSQL-Only `script_exists_in_output` Copy-Paste Bug Causes Incorrect Script Deletion During Rollback — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

A copy-paste error in `script_exists_in_output` causes the function to always return the result of the `lock_script_id` existence check on PostgreSQL, completely ignoring the `type_script_id` check. During `rollback_block`, this causes scripts that are referenced only as `type_script_id` in surviving outputs to be incorrectly deleted, permanently corrupting the indexer's script table and making all cells bearing that type script invisible to RPC queries.

---

### Finding Description

In `script_exists_in_output`, the function performs two SQL `EXISTS` queries: one for `lock_script_id` and one for `type_script_id`. The final `match` block at line 252 is supposed to evaluate `row_type`, but instead re-evaluates `row_lock`: [1](#0-0) 

```rust
// pg type is BOOLEAN
match row_lock.try_get::<bool, _>(0) {   // ← BUG: should be row_type
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

**PostgreSQL path:** `row_lock.try_get::<bool, _>(0)` succeeds (PostgreSQL returns `BOOLEAN` for `EXISTS`), so the `Ok(r)` arm is taken and `r` — the result of the **lock** query — is returned. The `row_type` result is never consulted. The function effectively only checks `lock_script_id`.

**SQLite path:** `row_lock.try_get::<bool, _>(0)` fails (SQLite returns `BIGINT`), so the `Err(_)` arm is taken and `row_type.get::<i64, _>(0) == 1` is correctly evaluated. SQLite is unaffected.

This is called from `rollback_block` after the rolled-back block's outputs have already been deleted: [2](#0-1) 

The sequence is:
1. Line 25: `remove_batch_by_blobs("output", ...)` — outputs of the rolled-back block are deleted.
2. Lines 29–38: for each deleted output, `script_exists_in_output` is called to check whether the script is still referenced by any surviving output.
3. Line 39: scripts deemed unreferenced are deleted.

For a script used **only as `type_script_id`** (not as `lock_script_id`) in surviving outputs from earlier blocks, the PostgreSQL-broken function returns `false` (lock check fails, type check is skipped), so the script is added to `script_id_list_to_remove` and deleted — even though earlier-block outputs still hold a foreign-key reference to it via `type_script_id`. [3](#0-2) 

---

### Impact Explanation

After the incorrect deletion, the `output` table retains rows whose `type_script_id` column points to a now-deleted `script` row. Any RPC query that joins `output` with `script` on `type_script_id` (e.g., `get_cells`, `get_transactions`, `get_cells_capacity` filtered by type script) will return zero results for those cells. The corruption is permanent until the indexer is fully reset and resynced from genesis. Wallets and DApps relying on the PostgreSQL-backed rich indexer will silently lose visibility of all cells bearing the affected type script.

---

### Likelihood Explanation

The trigger is a **natural blockchain reorg** — a routine event on any live CKB node. No attacker action is required; short reorgs (1–2 blocks) occur organically whenever two miners find competing blocks near-simultaneously. Any PostgreSQL-backed rich indexer node that experiences even a single reorg where the rolled-back block contains outputs whose type script is not also used as a lock script in surviving outputs will hit this bug. The condition is common: type scripts (e.g., UDT/NFT scripts) are routinely distinct from lock scripts.

The bug is PostgreSQL-specific and SQLite-correct, meaning it is invisible in the default embedded-SQLite deployment and only manifests in the explicitly-configured PostgreSQL deployment documented in the README. [4](#0-3) 

---

### Recommendation

Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// pg type is BOOLEAN
match row_type.try_get::<bool, _>(0) {   // ← fix: was row_lock
    Ok(r) => Ok(r),
    // sqlite type is BIGINT
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
```

Additionally, deduplicate `script_id_list_to_remove` before calling `remove_batch_by_blobs` to avoid redundant work when multiple outputs in the same block share a script ID. [3](#0-2) 

---

### Proof of Concept

1. Start a CKB node with `db_type = "postgres"` in `[indexer_v2.rich_indexer]`.
2. Index block A (height N−1) containing output O₀ with `type_script_id = S` and `lock_script_id = L₁` (where `L₁ ≠ S`).
3. Index block B (height N) containing outputs O₁ and O₂, both with `type_script_id = S` and `lock_script_id = L₂` (where `L₂ ≠ S`).
4. Trigger a rollback of block B (simulate a reorg).
5. **Expected:** script `S` survives because O₀ in block A still references it as `type_script_id`.
6. **Actual (PostgreSQL):** `script_exists_in_output(S)` checks only `lock_script_id = S` (finds nothing), returns `false`, and `S` is deleted. O₀ now has a dangling `type_script_id`.
7. Query `get_cells` with `type_script = S` → returns 0 results instead of 1.
8. Repeat with SQLite → returns 1 result (correct), confirming the differential. [5](#0-4)

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}

## Enabling Rich Indexer with PostgreSQL

To enable PostgreSQL, you must first set up a functional PostgreSQL service on your own. Please refer to [Server Administration](https://www.postgresql.org/docs/16/admin.html) for guidance. It is recommended to install version 16 or above.

For hardware with 4 cores and 8 GB of RAM, it is recommended to make the following two configuration parameter adjustments in PostgreSQL to achieve optimal query performance.

postgresql.conf:

```conf
#------------------------------------------------------------------------------
# RESOURCE USAGE (except WAL)
#------------------------------------------------------------------------------

# - Memory -

shared_buffers = 2GB                    # min 128kB
```

```conf
#------------------------------------------------------------------------------
# QUERY TUNING
#------------------------------------------------------------------------------

# - Other Planner Options -

jit = off                               # allow JIT compilation
```

Next, configure the PostgreSQL connection parameters for the CKB node.

ckb.toml:

```toml
# CKB rich-indexer has its unique configuration.
[indexer_v2.rich_indexer]
# By default, it uses an embedded SQLite database.
# Alternatively, you can set up a PostgreSQL database service and provide the connection parameters.
db_type = "postgres"
db_name = "ckb-rich-indexer"
db_host = "127.0.0.1"
db_port = 5432
db_user = "postgres"
db_password = "123456"
```

Finally, start the CKB node:

```bash
ckb run -C <path> --rich-indexer
```
