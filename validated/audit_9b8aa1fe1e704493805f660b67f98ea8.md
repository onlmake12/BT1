### Title
Blocking `thread::sleep` in `try_broadcast` Starves Tokio Executor Thread — (`network/src/network.rs`)

### Summary
`NetworkController::try_broadcast` retries failed sends using `thread::sleep(100ms)` in a loop for up to 60 iterations (6 seconds total). When called from within a tokio async context (e.g., a protocol handler or RPC handler running on the tokio runtime), this blocks the executor thread, starving all other async tasks scheduled on that thread.

### Finding Description
`P2P_SEND_TIMEOUT` (6s) and `P2P_TRY_SEND_INTERVAL` (100ms) define the retry budget. [1](#0-0)  When the p2p send channel returns `WouldBlock` (channel full), `try_broadcast` sleeps the calling thread for 100ms and retries, up to 60 times. The sleep used is `std::thread::sleep` (blocking), not `tokio::time::sleep` (async-cooperative). [5](#0-4)  The `ServiceHandle` trait is implemented with `async fn` handlers that run on the tokio runtime. [4](#0-3)  If any of these handlers invoke `NetworkController::broadcast` or `send_message_to`, the tokio executor thread is blocked for up to 6 seconds.

### Impact Explanation
A blocked tokio executor thread cannot process other async tasks: block relay, transaction propagation, peer keepalives, and sync responses all stall. In a consensus network, a 6-second propagation delay is significant and could contribute to orphaned blocks or degraded sync performance across the network.

### Likelihood Explanation
An attacker needs only to: (1) connect as a peer, (2) stop consuming incoming data (trivially done by not reading the TCP socket), causing the node's outgoing send buffer to fill. The node will naturally attempt to broadcast blocks and transactions to all peers including the attacker's, triggering the retry loop. No special privileges or hashpower are required.

### Recommendation
Replace `thread::sleep` in `try_broadcast` with `tokio::time::sleep` and make the function `async`, or offload the retry loop to a dedicated blocking thread via `tokio::task::spawn_blocking`. Alternatively, on `WouldBlock`, drop the message and log a warning rather than spinning — the p2p layer already has its own backpressure mechanisms.

### Proof of Concept
```rust
// Fill the p2p send channel to a peer by connecting and not reading.
// Then from a tokio::spawn task, call NetworkController::broadcast.
// Measure that the spawned task does not yield for ~6 seconds,
// starving other tasks on the same executor thread.
tokio::spawn(async move {
    let start = Instant::now();
    controller.broadcast(proto_id, data); // blocks thread::sleep x60
    assert!(start.elapsed() >= Duration::from_secs(5));
});
// A second task on the same thread will not be polled during those 6s.
``` [1](#0-0)
