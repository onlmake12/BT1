### Title
Incorrect Multi-Denom Fee Validation Adds Raw Token Amounts Without Price Conversion — (`File: target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

---

### Summary

The Osmosis-specific `is_fee_sufficient` function in the Pyth CosmWasm contract sums raw token amounts across different denominations without converting them to a common price unit. A caller can satisfy the fee check by paying in an allowed alternative denom (e.g., `uion`) whose market value is lower than the required `uosmo` fee, effectively underpaying.

---

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the `#[cfg(feature = "osmosis")]` variant of `is_fee_sufficient` iterates over all coins sent by the caller and adds their raw amounts into a single `total_amount` variable:

```rust
let mut total_amount = 0u128;
for coin in &info.funds {
    if coin.denom != state.fee.denom && !is_allowed_tx_fees_denom(deps, &coin.denom) {
        return Err(...)?;
    }
    total_amount = total_amount.checked_add(coin.amount.u128())...;
}
let base_denom_fee = get_update_fee(deps, data)?;
Ok(base_denom_fee.amount.u128() <= total_amount)
```

The code itself documents the flaw in a comment immediately above the comparison:

> *"NOTE: the base fee denom right now is = denom: 'uosmo', amount: 1, which is almost negligible. It's not important to convert the price right now. For now we are keeping the base fee amount same for each valid denom -> 1 but this logic will be updated to use spot price for different valid tokens in future"*

The comment at the top of the function also states the intended (but unimplemented) logic:

> *"convert each of them to the base token that is 'uosmo', combine all the converted token, check with `has_coins`"*

This conversion is never performed. The raw amount of any allowed denom is treated as equivalent to the same raw amount of `uosmo`. [1](#0-0) 

---

### Impact Explanation

Any caller of `update_price_feeds` or `parse_price_feed_updates` on the Osmosis deployment can pay the required fee using an allowed alternative denom (e.g., `uion`) at a 1:1 raw amount ratio against `uosmo`. If that denom trades at a lower price than `uosmo`, the caller pays less economic value than required. As the price divergence between `uosmo` and allowed alternative denoms grows, the underpayment grows proportionally. This undermines the fee mechanism that is intended to compensate the protocol for price feed updates. [2](#0-1) 

---

### Likelihood Explanation

The entry path is fully permissionless: any user can call `update_price_feeds` with `info.funds` containing an allowed alternative denom. The Osmosis txfee module already whitelists certain IBC tokens (e.g., `uion`, IBC-routed assets) as valid fee denoms. Price divergence between these tokens and `uosmo` is a normal market condition, not a hypothetical. The code's own `TODO`/`FIXME` comments confirm the developers are aware this is incomplete. [3](#0-2) [4](#0-3) 

---

### Recommendation

Replace the raw-amount summation with a price-weighted conversion. For each coin in `info.funds`, query the Osmosis spot price of that denom against `uosmo` (using the pool ID returned by `TxfeesQuerier::denom_pool_id`) and convert the coin's amount to its `uosmo`-equivalent before adding it to `total_amount`. Only after all coins are converted to a common denomination should the total be compared against `base_denom_fee.amount`. [5](#0-4) 

---

### Proof of Concept

1. Deploy the Pyth CosmWasm contract on Osmosis with `fee = { denom: "uosmo", amount: 100 }`.
2. Suppose `uion` is an allowed denom (whitelisted in the Osmosis txfee module) and currently trades at 0.1 `uosmo` per `uion`.
3. Call `update_price_feeds` with `info.funds = [{ denom: "uion", amount: 100 }]`.
4. `is_fee_sufficient` computes `total_amount = 100` and `base_denom_fee.amount = 100`, so `100 <= 100` → `true`.
5. The fee check passes, but the caller paid only 10 `uosmo`-equivalent in economic value instead of the required 100 `uosmo`.
6. The underpayment factor scales directly with the price ratio between `uosmo` and the alternative denom. [6](#0-5)
