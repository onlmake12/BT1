### Title
Protocol Fees Permanently Locked in `PriceInfoObject` — No Withdrawal Path Exists — (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

---

### Summary

Every call to `update_single_price_feed` deposits the caller's fee coins into the target `PriceInfoObject` via `deposit_fee_coins`. No function in the contract ever withdraws or transfers those coins out. The `fee_recipient_address` stored in `State` is never used in any transfer operation. All collected protocol fees are permanently locked.

---

### Finding Description

`update_single_price_feed` enforces a minimum fee and then immediately deposits it into the `PriceInfoObject`: [1](#0-0) 

`deposit_fee_coins` is a **public** function that joins the incoming `Coin<SUI>` into a dynamic field keyed by `FEE_STORAGE_KEY` on the `PriceInfoObject`: [2](#0-1) 

There is no corresponding `withdraw_fee_coins`, `collect_fees`, or any function that moves coins out of that dynamic field. A full grep across all production Move sources for `withdraw`, `collect_fee`, and `transfer.*SUI` returns zero hits outside of `state.move`'s field definition.

The `fee_recipient_address` field in `State` is stored and settable via governance: [3](#0-2) [4](#0-3) 

But it is **never read in any transfer context** — `get_fee_recipient` is a pure getter with no downstream usage that moves coins.

The `set_fee_recipient` module itself confirms this is a known regression from the previous design: [5](#0-4) 

The old behavior (sending fees to a recipient address) was removed, but no replacement withdrawal mechanism was introduced.

---

### Impact Explanation

Every `update_single_price_feed` call irrecoverably deposits `base_update_fee` SUI into the `PriceInfoObject`. After N updates across all price feeds, the total locked amount is `N × base_update_fee`. Because `PriceInfoObject` objects are shared Sui objects with no owner-controlled withdrawal entry point, and because no governance action can extract the coins either, the funds are permanently inaccessible. Protocol revenue is entirely frozen on-chain.

---

### Likelihood Explanation

This is triggered by the normal, intended user flow — calling `update_single_price_feed` — which any unprivileged account can do. No special access, leaked key, or governance majority is required. Every price update that has ever been submitted since deployment has already contributed to the locked balance. The condition is already active in production.

---

### Recommendation

Add a privileged `withdraw_fees` entry function (gated on `LatestOnly` or a governance capability) that extracts the `Coin<SUI>` from the `FEE_STORAGE_KEY` dynamic field of a `PriceInfoObject` and transfers it to `state::get_fee_recipient(pyth_state)`. Alternatively, route fees directly to the recipient address at update time instead of storing them in the `PriceInfoObject`, restoring the behavior that `set_fee_recipient.move` describes as the previous design.

---

### Proof of Concept

Invariant test (pseudocode):

```
// Setup: base_update_fee = F, N price updates submitted
for i in 0..N:
    update_single_price_feed(pyth_state, price_updates, price_info_object, coin(F), clock)

// Assert fees are locked in PriceInfoObject
assert(price_info::get_balance(price_info_object) == N * F)

// Assert fee_recipient received nothing
let recipient = state::get_fee_recipient(pyth_state)
assert(sui::balance_of(recipient) == 0)  // unchanged from before updates
```

The existing test at `pyth.move` line 817 already confirms the first assertion holds: [6](#0-5) 

No corresponding assertion exists anywhere that the recipient address ever receives funds, because no such transfer ever occurs.

```text
    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
```
