Audit Report

## Title
Concurrent Destructive-Read Race in `HeaderMapKernel::get` Causes Node Panic — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary

`HeaderMapKernel::get` uses a destructive `backend.remove(hash)` to promote sled-resident entries back into memory. Because no lock is held between the memory miss and the backend remove, two threads racing on the same evicted key will both call `backend.remove`; the second receives `None`, propagates it up through `get_header_index_view`, and causes `insert_valid_header` to panic unconditionally via `.expect("parent should be verified")`, crashing the node process.

## Finding Description

In `shared/src/types/header_map/kernel_lru.rs` lines 109–142, `get` first calls `self.memory.get_refresh(hash)`, which acquires and immediately releases a `RwLock` write guard (`memory.rs` L88). After that guard is released, there is no synchronization protecting the subsequent backend operations:

```rust
if let Some(view) = self.backend.remove(hash) {   // line 132 — destructive, unguarded
    self.memory.insert(view.clone());
    Some(view)
} else {
    None   // Thread B lands here
}
```

`backend.remove` in `backend_sled.rs` (L90–100) calls `sled::Db::remove`, which is atomic per-key: exactly one caller receives `Some`, all others receive `None`. Two threads that both miss in memory and both pass the `is_empty()` guard will race; the loser returns `None` from `get`.

The caller chain is:

1. `insert_valid_header` (`sync/src/types/mod.rs` L1094) computes `store_first = tip_number >= header.number()`. During IBD, `tip_number < header.number()`, so `store_first = false`.
2. `get_header_index_view` with `store_first=false` (L1158–1165) calls `self.shared.header_map().get(hash)` first, then falls back to the chain store. A parent header H that has been evicted to sled but not yet committed to the chain store is absent from both after Thread B's `remove` wins.
3. `insert_valid_header` L1101–1103 unconditionally panics:

```rust
let parent_header_index = self
    .get_header_index_view(&parent_hash, store_first)
    .expect("parent should be verified");   // PANIC
```

Two concurrent callers of `insert_valid_header` exist with no external serialization:
- `HeaderAcceptor::accept` in `headers_process.rs` L356 (one per peer per `SendHeaders` message)
- `CompactBlockProcess::execute` in `compact_block_process.rs` L78

The `limit_memory` background task (`mod.rs` L57–76, `kernel_lru.rs` L168–182) runs every 5 seconds and evicts entries from memory to sled whenever the count exceeds `WARN_THRESHOLD` (100,000 × `size_of::<HeaderIndexView>()`), making sled-resident entries a routine occurrence during IBD.

## Impact Explanation

An unprivileged remote peer can crash a CKB node process. The panic is unrecoverable (`expect` on `None` calls `panic!`, terminating the thread and, because this runs in a Tokio task context, the node). This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10001–15000 points).

## Likelihood Explanation

During IBD the node connects to multiple peers simultaneously and processes their `SendHeaders` messages concurrently. The 5-second eviction cycle means sled-resident entries are continuously present. Any two peers sharing a common ancestor header H in the sled tier whose child headers arrive within the same scheduling quantum trigger the race. No special attacker capability is required beyond connecting as a peer and sending valid (or previously-seen) header chains; this is a normal operating condition, not an edge case.

## Recommendation

Replace the destructive `backend.remove` in `get` with a non-destructive `backend.get` (already implemented at `backend_sled.rs` L56–61). Both racing threads will read the same value and insert it into memory (idempotent). The backend entry can then be lazily cleaned up by `limit_memory` or a separate eviction pass. Alternatively, wrap the entire memory-miss → backend-lookup → memory-insert sequence in a per-key or coarse mutex so that only one thread performs the promotion at a time.

## Proof of Concept

```rust
// Minimal concurrent test (unit test style)
let map = Arc::new(HeaderMapKernel::<SledBackend>::new(None, 0, Arc::new(AtomicBool::new(true))));

// Insert H and force eviction to sled
map.insert(H.clone());
map.limit_memory(); // memory_limit=0 → all entries evicted to sled

let map_a = Arc::clone(&map);
let map_b = Arc::clone(&map);

let t_a = thread::spawn(move || map_a.get(&H.hash()));
let t_b = thread::spawn(move || map_b.get(&H.hash()));

let r_a = t_a.join().unwrap();
let r_b = t_b.join().unwrap();

// One of these will be None, demonstrating the bug:
assert!(r_a.is_some(), "Thread A lost the race");
assert!(r_b.is_some(), "Thread B lost the race"); // ← fails
```

The failing assertion maps directly to the production panic at `.expect("parent should be verified")` in `insert_valid_header` (`sync/src/types/mod.rs` L1103), reachable from any peer sending concurrent `SendHeaders` messages during IBD.
