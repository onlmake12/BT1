### Title
Unprivileged Rent Theft via Missing Authority Check in `close_signature_set` ŌĆö (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

---

### Summary

The `close_signature_set` instruction closes a `SignatureSet` account and transfers its rent lamports to `sol_destination`. There is no constraint requiring `sol_destination` to be the original payer who funded the `SignatureSet`. Any unprivileged signer can supply their own address as `sol_destination` and steal the rent from any existing `SignatureSet` that has a corresponding `PostedVaaV1` account.

---

### Finding Description

The `CloseSignatureSet` accounts struct is:

```rust
pub struct CloseSignatureSet<'info> {
    #[account(mut)]
    sol_destination: Signer<'info>,          // ŌåÉ any signer, no authority constraint

    #[account(
        seeds = [PostedVaaV1::SEED_PREFIX, posted_vaa.message_hash().as_ref()],
        bump,
        has_one = signature_set
    )]
    posted_vaa: Account<'info, LegacyAnchorized<PostedVaaV1>>,

    #[account(
        mut,
        close = sol_destination              // ŌåÉ rent goes to whoever signed
    )]
    signature_set: Account<'info, AccountVariant<SignatureSet>>,
}
``` [1](#0-0) 

The only validation performed is:
- `posted_vaa` is a valid PDA for the given message hash
- `posted_vaa.signature_set == signature_set.key()` (via `has_one`)

There is no check that `sol_destination` matches any stored authority. The `SignatureSet` state struct stores only `sig_verify_successes`, `message_hash`, and `guardian_set_index` ŌĆö it has no `write_authority` or `payer` field: [2](#0-1) 

The `SignatureSet` is created in `verify_signatures` by a `payer` signer, but that payer pubkey is never written into the account: [3](#0-2) 

The legacy `post_vaa` instruction stores the `signature_set` pubkey into `PostedVaaV1.info.signature_set`, which is what the `has_one` constraint in `close_signature_set` validates against: [4](#0-3) 

This creates a fully public, permissionless path: given any `PostedVaaV1` account whose `signature_set` field is non-zero (i.e., created via the legacy `post_vaa` flow), any attacker can call `close_signature_set` with themselves as `sol_destination` and receive the rent.

Note: VAAs posted via the newer `post_vaa_v1` instruction set `signature_set: Default::default()` (all-zero pubkey), so those are not affected. Only the legacy flow is vulnerable. [5](#0-4) 

---

### Impact Explanation

An attacker can:
1. Scan on-chain `PostedVaaV1` accounts for non-zero `signature_set` fields (all public state).
2. Call `close_signature_set(sol_destination=attacker, posted_vaa=X, signature_set=Y)` for each.
3. Receive the rent lamports from every such `SignatureSet`.

Alternatively, the attacker can front-run a victim's `close_signature_set` transaction by observing it and submitting their own version first (Solana's Gulf Stream propagation makes transactions observable before finalization).

The victim's transaction then fails with "account already closed," and they permanently lose their rent ŌĆö they cannot re-close an already-closed account.

**Scoped impact:** Theft of `SignatureSet` rent lamports from the original payer. Per-account value is small (~1,000ŌĆō2,000 lamports), but the attack is permissionless and can be applied to all existing legacy-flow `PostedVaaV1` accounts in bulk.

---

### Likelihood Explanation

- No privileged access required ŌĆö any keypair can execute this.
- All required account addresses are derivable from public on-chain state.
- The attack is not limited to front-running; it can be executed proactively against all existing accounts at any time.
- Likelihood is **high** for anyone motivated to sweep small rent amounts at scale.

---

### Recommendation

Add a `write_authority` (or `payer`) field to the `SignatureSet` struct, populated during `verify_signatures`. Then add a constraint in `CloseSignatureSet`:

```rust
#[account(mut, constraint = sol_destination.key() == signature_set.write_authority)]
sol_destination: Signer<'info>,
```

Alternatively, store the authority in `PostedVaaV1` and enforce it via a `has_one` constraint on `sol_destination`.

---

### Proof of Concept

```rust
// Attacker observes or scans for a PostedVaaV1 with non-zero signature_set field.
// posted_vaa_key: known PDA
// sig_set_key: stored in posted_vaa.info.signature_set

let ix = close_signature_set(
    sol_destination: attacker_keypair.pubkey(),  // attacker receives rent
    posted_vaa: posted_vaa_key,
    signature_set: sig_set_key,
);
// Sign with attacker_keypair and submit.
// Rent from sig_set_key is transferred to attacker.
// Victim's subsequent close_signature_set fails: account already closed.
```

No special privileges, no leaked keys, no governance majority required. The attack is locally testable against any legacy-flow `PostedVaaV1` account.
