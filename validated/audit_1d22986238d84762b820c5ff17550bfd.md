### Title
Unbounded SQL OFFSET via Crafted Pagination Cursor in `get_tx_with_cell` Enables Single-Request DB Full-Scan DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

---

### Summary

The `get_transactions` RPC endpoint (with `group_by_transaction=false`) decodes the `after` cursor's last 4 bytes as an uncapped `i32` offset and passes it verbatim to the SQL `OFFSET` clause. An unprivileged caller can supply a cursor whose offset bytes decode to `i32::MAX` (2,147,483,647), forcing the database to perform a full-scan skip of every row in the result set on every call, with no per-call cost bound.

---

### Finding Description

**Cursor parsing — length check only, no value validation:**

In `get_transactions`, the `after` cursor is validated only for byte length (must be 12): [1](#0-0) 

`decode_i32` performs no range check — it accepts any 4-byte little-endian value including `[0xFF, 0xFF, 0xFF, 0x7F]` = `i32::MAX`: [2](#0-1) 

**Uncapped offset injected into SQL:**

The decoded offset is passed directly to `query_builder.offset()` with no cap, guard, or maximum: [3](#0-2) 

**`limit` is capped; `offset` is not:**

The `limit` parameter is explicitly bounded by `request_limit`: [4](#0-3) 

No equivalent guard exists for the offset value anywhere in the call path.

**Resulting SQL:**

With `last=0` (bytes `[0,0,0,0,0,0,0,0]`) and `offset=i32::MAX` (bytes `[0xFF,0xFF,0xFF,0x7F]`), the emitted query becomes:

```sql
SELECT tx_id, block.block_number, ...
FROM (... UNION ...) AS res
JOIN ckb_transaction ON res.tx_id = ckb_transaction.id
JOIN block ON ckb_transaction.block_id = block.id
WHERE tx_id >= 0
ORDER BY tx_id ASC
LIMIT <limit>
OFFSET 2147483647
```

Both SQLite and PostgreSQL must materialize and iterate through every row satisfying `tx_id >= 0` (the entire indexed dataset) before returning the empty result.

---

### Impact Explanation

Each crafted request forces an O(N) full-scan of the entire transaction index, where N is the total number of indexed input/output rows. On a mainnet-scale node (tens of millions of rows), a single request can hold a DB connection thread for several seconds. Because the RPC server accepts concurrent requests and the DB thread pool is finite, a low-rate stream of such requests (no brute force required) can saturate all available DB threads, stalling all other indexer queries and causing RPC congestion for legitimate callers.

---

### Likelihood Explanation

The rich-indexer RPC is exposed on a configurable port. Operators who expose it beyond localhost (common for infrastructure nodes serving wallets or dApps) are directly reachable. The exploit requires only a single well-formed JSON-RPC call with a 12-byte hex cursor — no authentication, no special privileges, no prior knowledge of the DB state. The crafted cursor passes all existing validation (length check, byte decoding) without error.

---

### Recommendation

Cap the decoded offset before use. The offset represents the number of cells already seen within the current `tx_id` page; it should never exceed the maximum number of cells a single transaction can have (bounded by consensus). A simple guard suffices:

```rust
let offset = decode_i32(offset)?;
// Offset is a within-tx cell count; cap it to a safe maximum
if offset < 0 || offset > MAX_CELLS_PER_TX {
    return Err(Error::Params("Invalid 'after' cursor offset".to_string()));
}
```

Alternatively, validate that the offset matches the actual cell count of the referenced `tx_id` before issuing the query.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
