### Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Enables Unbounded Work — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

The guard expression on line 201 performs `(last_n_blocks as usize) * 2` without overflow protection. In release mode, Rust wraps on overflow. An attacker can set `last_n_blocks = 0x8000_0000_0000_0000u64` so the multiplication wraps to `0`, bypassing the `GET_LAST_STATE_PROOF_LIMIT = 1000` check entirely. Downstream, the same huge `last_n_blocks` value causes the server to collect and process every block number from `start_block_number` to `last_block_number`, proportional to full chain height.

---

### Finding Description

**Step 1 — Overflow in the guard** [1](#0-0) 

`last_n_blocks` is a `u64` read directly from the peer message. The guard casts it to `usize` and multiplies by 2:

```
last_n_blocks = 0x8000_0000_0000_0000u64
(last_n_blocks as usize) = 0x8000_0000_0000_0000usize   // valid on 64-bit
(last_n_blocks as usize) * 2 = 0x0000_0000_0000_0000    // wraps to 0 in release mode
```

With `difficulties = []`, the full expression is `0 + 0 > 1000` → **false**. The guard is bypassed.

**Step 2 — Unbounded `last_n_numbers` collection** [2](#0-1) 

Because `last_n_blocks = 0x8000_0000_0000_0000` is astronomically large, the condition `last_block_number - start_block_number <= last_n_blocks` is always true for any realistic chain. The code then executes:

```rust
let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
```

For a 10M-block chain with `start_block_number = 0`, this allocates a `Vec` of 10 million `u64` entries (~80 MB), then passes it to `complete_headers`.

**Step 3 — O(N) DB work per block** [3](#0-2) 

`complete_headers` calls `get_ancestor`, `get_block`, and `chain_root_mmr` for every entry in `last_n_numbers`. Each call hits the chain store. For 10M blocks this is tens of millions of DB reads, causing OOM or CPU exhaustion.

**The constant definition:** [4](#0-3) 

---

### Impact Explanation

A single malicious light-client peer can crash the full node process (OOM or CPU exhaustion) by sending one crafted `GetLastStateProof` message. No authentication or special privilege is required. The light-client protocol is a supported production path.

---

### Likelihood Explanation

- Light-client protocol is an opt-in but documented production feature.
- The attacker only needs to know a valid main-chain tip hash (publicly observable).
- `last_n_blocks` is a `Uint64` field in the molecule schema, so any 64-bit value is wire-valid.
- The overflow only triggers in release builds (the standard deployment mode); debug builds would panic, masking the issue during development.

---

### Recommendation

Replace the unchecked multiplication with a saturating or checked variant:

```rust
// Before (vulnerable):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

Additionally, add an independent upper-bound check on `last_n_blocks` itself before any arithmetic:

```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

### Proof of Concept

```rust
// Pseudocode: craft a message with overflowing last_n_blocks
let content = packed::GetLastStateProof::new_builder()
    .last_hash(tip_hash)           // valid main-chain tip
    .start_hash(genesis_hash)
    .start_number(0u64)
    .last_n_blocks(0x8000_0000_0000_0000u64)  // triggers overflow
    .difficulty_boundary(U256::one())
    // difficulties left empty
    .build();
```

In release mode: guard evaluates to `0 + 0 > 1000` → false. Server proceeds to `(0..N).collect()` and `complete_headers` over all N blocks, exhausting memory/CPU.
