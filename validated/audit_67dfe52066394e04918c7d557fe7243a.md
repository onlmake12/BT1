### Title
Deployer Retains Permanent Ownership of `PythLazer` Proxy After Deployment — (`lazer/contracts/evm/script/PythLazerDeploy.s.sol`)

---

### Summary

`PythLazerDeployScript.run()` deploys the `PythLazer` UUPS proxy and initializes it with the hardcoded `deployer` EOA as the owner, but never transfers ownership to the intended governance/admin address. A separate, manually-run script (`PythLazerChangeOwnership.s.sol`) is required to complete the transfer. If that step is skipped or delayed, the deployer EOA retains full owner privileges indefinitely.

---

### Finding Description

In `PythLazerDeploy.s.sol`, the `deployProxy` function explicitly sets `topAuthority = deployer` (the hardcoded constant EOA `0x78357316239040e19fC823372cC179ca75e64b81`) and passes it as the sole argument to `initialize(address)`:

```solidity
// Set the top authority to the deployer for the time being
address topAuthority = deployer;

vm.startBroadcast();
addr = createX.deployCreate3({
    salt: salt,
    initCode: abi.encodePacked(
        type(ERC1967Proxy).creationCode,
        abi.encode(
            impl,
            abi.encodeWithSignature("initialize(address)", topAuthority)
        )
    )
});
``` [1](#0-0) 

`PythLazer.initialize` calls `__Ownable_init(_topAuthority)`, making the deployer EOA the permanent owner:

```solidity
function initialize(address _topAuthority) public initializer {
    __Ownable_init(_topAuthority);
    ...
}
``` [2](#0-1) 

The `run()` function calls `deployProxy` and then only writes output — it never calls `transferOwnership` to hand control to the governance executor:

```solidity
function run() public {
    address impl = deployImplementation("lazer:impl");
    address proxy = deployProxy("lazer:proxy", impl);
    _writeOutput(impl, proxy, "./deployment-output.json");
}
``` [3](#0-2) 

The ownership transfer is deferred to a completely separate script, `PythLazerChangeOwnership.s.sol`, which must be run manually after deployment: [4](#0-3) 

The comment `// Set the top authority to the deployer for the time being` is an explicit acknowledgment that the deployer is a temporary placeholder, but the deploy script provides no mechanism to complete the handoff atomically.

---

### Impact Explanation

The `PythLazer` owner controls two critical privileged functions:

1. **`updateTrustedSigner(address, uint256)`** — adds or removes trusted signers whose ECDSA signatures are accepted as valid Lazer price updates. An owner can add a malicious signer or remove all legitimate signers, breaking price feed integrity for all consumers.
2. **`_authorizeUpgrade`** — allows the owner to upgrade the proxy to an arbitrary implementation, enabling complete contract takeover. [5](#0-4) 

If the deployer key is compromised during the window between deployment and ownership transfer, or if the transfer step is simply omitted, an attacker controlling the deployer key can manipulate trusted signers to inject arbitrary price data accepted by all on-chain Lazer consumers.

---

### Likelihood Explanation

The likelihood is **High**:

- The deploy script itself contains the comment `"for the time being"`, confirming the deployer ownership is intentionally temporary but the script provides no automated path to complete the transfer.
- The ownership transfer is a separate, manually-invoked Foundry script (`PythLazerChangeOwnership.s.sol`) with no enforcement that it is run after deployment.
- Multi-chain deployments (the script is designed for cross-chain use) multiply the opportunity for the transfer step to be missed on one or more chains.
- The deployer address is a hardcoded constant, meaning the same EOA key is reused across all chains, making it a high-value target.

---

### Recommendation

Transfer ownership to the governance executor address atomically within `deployProxy` or `run()`, immediately after the proxy is deployed, rather than relying on a separate manual script:

```solidity
function run() public {
    address impl = deployImplementation("lazer:impl");
    address proxy = deployProxy("lazer:proxy", impl);

    address governanceExecutor = vm.envAddress("GOVERNANCE_EXECUTOR");
    vm.startBroadcast();
    PythLazer(proxy).transferOwnership(governanceExecutor);
    vm.stopBroadcast();

    _writeOutput(impl, proxy, "./deployment-output.json");
}
```

Alternatively, pass the governance executor address directly to `initialize` instead of the deployer, eliminating the intermediate ownership state entirely.

---

### Proof of Concept

1. `PythLazerDeployScript.run()` is executed by the deployer EOA (`0x78357316239040e19fC823372cC179ca75e64b81`).
2. `deployProxy` initializes the proxy with `topAuthority = deployer`, making the deployer EOA the owner.
3. `run()` returns without calling `transferOwnership`.
4. The deployer EOA is now the sole owner of `PythLazer` on the deployed chain.
5. If `PythLazerChangeOwnership.s.sol` is not run (or is run late), the deployer key retains the ability to call `updateTrustedSigner` to add a malicious signer address.
6. Any subsequent call to `PythLazer.verifyUpdate` with a payload signed by the malicious signer will pass validation, injecting attacker-controlled price data into all on-chain Lazer consumers. [6](#0-5)
