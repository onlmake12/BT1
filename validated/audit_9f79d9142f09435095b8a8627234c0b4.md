### Title
Non-Atomic Upgrade Creates Intermediate Broken State Blocking All Price Feed Updates - (`target_chains/sui/contracts/sources/migrate.move`)

### Summary

The Pyth Sui contract upgrade is a two-step process split across separate transactions. Between `commit_upgrade` (step 1) and `migrate()` (step 2), the `State` object's version field still holds the old version while the new package is already deployed. Every version-gated function — including `update_single_price_feed` — calls `state::assert_latest_only`, which compares the state's stored version against `version_control::current_version()` from the **new** package. This comparison fails during the inter-transaction window, causing a complete DoS of all price feed updates on Sui. If `migrate()` is never successfully executed, the DoS is permanent.

### Finding Description

The Pyth Sui upgrade flow is explicitly documented as non-atomic:

> "WARNING: The migration does *not* proceed atomically with the upgrade (as they are done in separate transactions)." [1](#0-0) 

The two-step process is:

1. **Transaction 1**: `authorize_upgrade` → `commit_upgrade` — publishes the new package and updates `UpgradeCap`, but the `State`'s `CurrentVersion` dynamic field still holds the old version type.
2. **Transaction 2**: `migrate()` — calls `state::migrate_version()` to advance `CurrentVersion` from old to new. [2](#0-1) 

Between these two transactions, `update_single_price_feed` immediately calls `state::assert_latest_only`: [3](#0-2) 

`assert_latest_only` calls `package_utils::assert_version` with `version_control::current_version()` from the **new** package: [4](#0-3) 

Since the state's `CurrentVersion` dynamic field still holds the old version type, this assertion fails. The old package cannot be called either — the Sui VM enforces that only the latest published package can invoke its own functions. The result is a complete DoS of all version-gated operations.

The same pattern applies to `create_price_feeds_using_accumulator`: [5](#0-4) 

And all governance operations: [6](#0-5) 

### Impact Explanation

During the inter-transaction window, **no price feed on Sui can be updated**. Any DeFi protocol that calls `update_single_price_feed` will revert. If `migrate()` fails (e.g., due to a digest mismatch or VAA replay issue), the contract is permanently bricked: the old package is rejected by the Sui VM, and the new package fails at `assert_latest_only`. There is no recovery path without a further governance upgrade.

### Likelihood Explanation

This window occurs on every Pyth Sui contract upgrade. The Sui CLI explicitly performs the upgrade and migrate in separate transactions: [7](#0-6) 

While the window is typically short, block production on Sui is fast (~400ms), meaning multiple transactions from integrators can land in this gap. Additionally, any failure in `migrate()` makes the DoS permanent.

### Recommendation

1. Combine `commit_upgrade` and `migrate()` into a single programmable transaction block (PTB) where possible, or use a Sui-native atomic upgrade pattern.
2. If atomicity is not achievable, add an explicit version-bypass flag to `update_single_price_feed` that allows the old package to continue serving updates until `migrate()` completes.
3. Add an on-chain guard that detects the intermediate state (committed but not migrated) and emits an event or sets a flag, so integrators can detect and handle the window.

### Proof of Concept

1. Governance submits an upgrade VAA. `authorize_upgrade` + `commit_upgrade` execute in Transaction 1. New package is live; `State.CurrentVersion` = `V_old`.
2. Before Transaction 2 (`migrate()`) lands, a relayer calls `update_single_price_feed` from the new package address.
3. `state::assert_latest_only` calls `package_utils::assert_version(&self.id, version_control::current_version())`. The new package's `current_version()` returns `V_new`, but the state stores `V_old`. The assertion aborts with `E_INCORRECT_VERSION`.
4. All price feed updates on Sui are blocked until `migrate()` succeeds. If `migrate()` never succeeds, the contract is permanently non-functional. [8](#0-7)

    fun handle_migrate(
        pyth_state: &mut State,
        receipt: WormholeVAAVerificationReceipt,
    ) {
        // See `version_control` module for hard-coded configuration.
        state::migrate_version(pyth_state);

        // This capability ensures that the current build version is used.
        let latest_only = state::assert_latest_only(pyth_state);

        let digest = contract_upgrade::take_upgrade_digest(receipt);
        state::assert_authorized_digest(
            &latest_only,
            pyth_state,
            digest
        );

        // Finally emit an event reflecting a successful migrate.
        let package = state::current_package(&latest_only, pyth_state);
        sui::event::emit(MigrateComplete { package });
    }
