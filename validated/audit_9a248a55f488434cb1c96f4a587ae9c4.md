### Title
TYPE_ID Built-in Bypasses Lock Authorization: Any Cell Locked by TYPE_ID Script Spendable Without Proof — (`script/src/type_id.rs`, `script/src/verify.rs`)

---

### Summary

`verify_script_group` dispatches to `TypeIdSystemScript::verify` based solely on `code_hash == TYPE_ID_CODE_HASH` and `hash_type == Type`, without checking `group.group_type`. When a cell's **lock script** carries those two fields, the resulting lock `ScriptGroup` always has `input_indices.len() == 1` and `output_indices.len() == 0`. Inside `TypeIdSystemScript::verify`, the only cryptographic check is gated on `input_indices.is_empty()` (the creation branch). With one input and zero outputs that branch is never entered, so the function returns `Ok(TYPE_ID_CYCLES)` unconditionally — the lock is satisfied with no signature, no witness, and no hash verification.

---

### Finding Description

**Dispatch in `verify_script_group`** — `script/src/verify.rs` lines 427–444:

```rust
fn verify_script_group(&self, group: &ScriptGroup, max_cycles: Cycle) -> Result<Cycle, ScriptError> {
    if group.script.code_hash() == TYPE_ID_CODE_HASH.into()
        && Into::<u8>::into(group.script.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
    {
        // dispatched here for BOTH Lock and Type groups — group.group_type is never checked
        verifier.verify()
    } else {
        self.run(group, max_cycles)
    }
}
``` [1](#0-0) 

The condition tests only `code_hash` and `hash_type`. `group.group_type` (`Lock` vs `Type`) is never inspected.

**Authorization logic in `TypeIdSystemScript::verify`** — `script/src/type_id.rs` lines 29–77:

The only cryptographic check is:

```rust
if self.script_group.input_indices.is_empty() {
    // hash check: args must equal blake2b(first_input || output_index)
}
Ok(TYPE_ID_CYCLES)   // reached unconditionally when input_indices is non-empty
``` [2](#0-1) 

For a **lock group**, `input_indices` always contains the spending input index (≥ 1) and `output_indices` is always empty (lock scripts appear only on inputs). The three checks that do execute are:

| Check | Value for lock group | Result |
|---|---|---|
| `max_cycles < TYPE_ID_CYCLES` | depends on caller, trivially satisfied | pass |
| `args().len() != 32` | attacker supplies exactly 32 bytes | pass |
| `input_indices.len() > 1 \|\| output_indices.len() > 1` | 1 > 1 is false; 0 > 1 is false | pass |

The hash-verification branch (`input_indices.is_empty()`) is **never entered**, and `Ok(TYPE_ID_CYCLES)` is returned. [3](#0-2) 

---

### Impact Explanation

Any on-chain cell whose lock script is `Script { code_hash: TYPE_ID_CODE_HASH, hash_type: Type, args: <any 32 bytes> }` can be spent by an unprivileged attacker submitting an ordinary transaction. No witness, no signature, and no knowledge of any secret is required. The attacker simply constructs a transaction with that cell as an input and any desired output, and the node's script verifier accepts it.

---

### Likelihood Explanation

The attack is fully self-contained and requires no special privileges:

1. Identify (or create) a live cell whose lock script matches the pattern above.
2. Construct a `ResolvedTransaction` spending it with any output.
3. Submit via the standard P2P/RPC transaction submission path.
4. `TransactionScriptsVerifier::verify` → `verify_script_group` → `TypeIdSystemScript::verify` returns `Ok(TYPE_ID_CYCLES)`.

The same flaw exists in `verify_group_with_chunk` and `verify_group_with_signal`, so all verification code paths are affected. [4](#0-3) [5](#0-4) 

---

### Recommendation

In `verify_script_group` (and its siblings), add a guard that restricts the TYPE_ID built-in to **Type** groups only:

```rust
if group.script.code_hash() == TYPE_ID_CODE_HASH.into()
    && Into::<u8>::into(group.script.hash_type()) == Into::<u8>::into(ScriptHashType::Type)
    && group.group_type == ScriptGroupType::Type   // <-- add this
{
    ...TypeIdSystemScript...
} else {
    self.run(group, max_cycles)
}
```

Alternatively, add the same guard inside `TypeIdSystemScript::verify` itself and return an error for Lock groups. [6](#0-5)

    .code_hash(TYPE_ID_CODE_HASH.pack())
    .hash_type(ScriptHashType::Type.into())
    .args(Bytes::from(vec![0u8; 32]).pack())
    .build();

let input_cell = CellOutputBuilder::default()
    .capacity(capacity_bytes!(1000))
    .lock(lock)
    .build();

// Spend it — no witness, no signature, output can be anything

    .input(input.clone())
    .output(CellOutputBuilder::default().capacity(capacity_bytes!(999)).lock(attacker_lock).build())
    .build();

let rtx = ResolvedTransaction { transaction, resolved_inputs: vec![...input_cell...], ... };

let verifier = TransactionScriptsVerifier::new(...);
// Passes: lock group has input_indices=[0], output_indices=[]
// TypeIdSystemScript::verify skips hash check, returns Ok(TYPE_ID_CYCLES)
