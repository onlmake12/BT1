### Title
Missing Sender Peer Validation in `BlockProposalProcess` Allows Any Peer to Inject Proposal Transactions — (File: `sync/src/relayer/block_proposal_process.rs`)

---

### Summary

`BlockProposalProcess` is the only relayer message handler that does not receive the sending `peer` index. It processes `BlockProposal` messages and submits transactions to the tx pool without verifying that the responding peer was the one originally asked via `GetBlockProposal`. Any connected peer can send a `BlockProposal` with matching inflight proposal IDs and have its transactions accepted, consuming the inflight slot and silencing the legitimate peer's response.

---

### Finding Description

In `sync/src/relayer/mod.rs`, the `try_process` dispatch for `BlockProposal` omits the `peer` and `nc` arguments that every other handler receives:

```rust
packed::RelayMessageUnionReader::BlockProposal(reader) => {
    BlockProposalProcess::new(reader, self).execute().await   // no peer, no nc
}
``` [1](#0-0) 

Compare with `BlockTransactionsProcess` and `TransactionsProcess`, which both receive `peer`:

```rust
packed::RelayMessageUnionReader::BlockTransactions(reader) => {
    BlockTransactionsProcess::new(reader, self, nc, peer).execute().await
}
``` [2](#0-1) 

Because `BlockProposalProcess` has no `peer` field, its `execute()` function cannot perform any per-peer origin check. It only checks whether the proposal short IDs were previously inflight:

```rust
let removes = sync_state.remove_inflight_proposals(&proposals);
for (previously_in, tx) in removes.into_iter().zip(unknown_txs) {
    if previously_in {
        sync_state.mark_as_known_tx(tx.hash());
        asked_txs.push(tx);
    }
}
``` [3](#0-2) 

The `previously_in` guard only confirms the proposal was requested *from someone*; it does not confirm it was requested from the peer now responding. The inflight-proposal store (`insert_inflight_proposals`) records proposal IDs and block numbers but not the target peer, so there is no stored state to compare against even if `peer` were passed in. [4](#0-3) 

By contrast, `TransactionsProcess` explicitly gates processing on the requesting peer:

```rust
.filter(|(tx, _)| {
    !tx_filter.contains(&tx.hash())
        && unknown_tx_hashes
            .get_priority(&tx.hash())
            .map(|priority| priority.requesting_peer() == Some(self.peer))
            .unwrap_or_default()
})
``` [5](#0-4) 

And `BlockTransactionsProcess` gates on a per-peer map entry:

```rust
if let Entry::Occupied(mut value) = peers_map.entry(self.peer) {
``` [6](#0-5) 

`BlockProposalProcess` has neither guard.

---

### Impact Explanation

An attacker who is a connected peer and can observe (or infer) which proposal short IDs are currently inflight can:

1. Send a `BlockProposal` message containing transactions whose `ProposalShortId` values match the inflight set.
2. `remove_inflight_proposals` removes those IDs from the inflight set and returns `previously_in = true`.
3. The attacker's transactions are marked as known and submitted to the tx pool via `notify_txs_async`.
4. When the legitimate peer's `BlockProposal` arrives, `remove_inflight_proposals` returns `previously_in = false` for all IDs (already removed), so `asked_txs` is empty and the response is silently ignored (`Status::ignored()`).

Concrete consequences:
- **Block reconstruction stall**: if the attacker's transactions are valid but differ from what the miner included (e.g., a conflicting double-spend variant), the node marks them as known, never re-requests the correct ones, and cannot reconstruct the compact block.
- **Tx-pool pollution**: the attacker can force the node to evaluate and potentially accept attacker-chosen transactions under the guise of a proposal response.
- **Inflight slot exhaustion**: repeated fake responses drain inflight slots, preventing legitimate proposal fetching.

---

### Likelihood Explanation

- The attacker only needs to be a connected peer (unprivileged, no key required).
- Inflight proposal IDs are observable by any peer that monitors `GetBlockProposal` messages on the P2P network, or can be inferred from the public mempool.
- The rate limiter in `try_process` exempts `CompactBlock` but still applies to `BlockProposal`, providing only a 30 req/s cap per peer — not a meaningful barrier for a targeted attack on a specific inflight set.
- No special timing is required; the attacker simply needs to respond before the legitimate peer.

---

### Recommendation

1. **Track the target peer per inflight proposal**: extend `insert_inflight_proposals` to record which peer each proposal was requested from (analogous to `peers_map` in `pending_compact_blocks`).
2. **Pass `peer` and `nc` to `BlockProposalProcess`**: mirror the signature of every other handler.
3. **Validate the sender in `execute()`**: before calling `remove_inflight_proposals`, confirm that `self.peer` matches the peer recorded for each proposal ID. Reject or ignore responses from non-matching peers.

---

### Proof of Concept

```
Setup:
  - Node N has sent GetBlockProposal{proposals: [P1, P2]} to legitimate peer L.
  - Attacker A is also connected to N.

Attack:
  1. A observes the GetBlockProposal message (plaintext P2P).
  2. A sends BlockProposal{transactions: [TX1, TX2]} where
       ProposalShortId::from_tx_hash(TX1.hash()) == P1
       ProposalShortId::from_tx_hash(TX2.hash()) == P2
     but TX1/TX2 are attacker-chosen variants (e.g., double-spend of TX1_real).

Result in BlockProposalProcess::execute() on N:
  - unknown_txs = [TX1, TX2]  (not yet in tx_filter)
  - proposals   = [P1, P2]
  - remove_inflight_proposals([P1,P2]) -> [(true, TX1), (true, TX2)]
  - mark_as_known_tx(TX1.hash()), mark_as_known_tx(TX2.hash())
  - notify_txs_async([TX1, TX2])  -- attacker's txs enter tx pool pipeline

When L later sends BlockProposal{transactions: [TX1_real, TX2_real]}:
  - remove_inflight_proposals([P1,P2]) -> [(false,_),(false,_)]  -- already removed
  - asked_txs is empty -> Status::ignored()
  - N never processes the correct transactions; compact block reconstruction fails.
```
