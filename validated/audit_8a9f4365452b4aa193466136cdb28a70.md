### Title
Unbounded Tokio Task Accumulation Due to Missing HTTP Timeout in Miner RPC Client — (File: `miner/src/client.rs`)

---

### Summary

The `Rpc` HTTP client in the CKB miner is constructed without any timeout configuration, and each outbound RPC call spawns a new unbounded tokio task. If the CKB node's RPC endpoint is slow or unresponsive, tasks accumulate indefinitely, leading to goroutine/task leaks and memory exhaustion in the miner process.

---

### Finding Description

In `miner/src/client.rs`, the `Rpc::new` function builds a `hyper_util` legacy `HttpClient` with no timeout set:

```rust
let https = hyper_tls::HttpsConnector::new();
let client = HttpClient::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(https);
``` [1](#0-0) 

For every RPC request received on the internal channel, a new tokio task is spawned that calls `client.request(req).await` with no timeout wrapper:

```rust
loop_handle.spawn(async move {
    let request = match client
        .request(req)
        .await
        ...
``` [2](#0-1) 

The internal channel has a capacity of 65,535 messages:

```rust
let (sender, mut receiver) = mpsc::channel(65_535);
``` [3](#0-2) 

This means up to 65,535 tasks can be spawned simultaneously, each holding an open HTTP connection that never times out.

**Contrast with the block assembler's `poster` client**, which also lacks a timeout on the underlying `HttpClient` construction: [4](#0-3) 

However, the block assembler's `notify` function correctly wraps each request in `tokio::time::timeout(notify_timeout, client.request(req))`: [5](#0-4) 

No equivalent protection exists in the miner's `Rpc` client.

---

### Impact Explanation

If the CKB node's RPC endpoint becomes slow or unresponsive — whether due to overload, a network issue, or a deliberately slow server — the miner process spawns up to 65,535 tokio tasks that each hold an open TCP connection and await a response indefinitely. This causes:

1. **Task/goroutine leaks**: Tasks never complete and are never cleaned up.
2. **Memory exhaustion**: Each pending task retains its stack, request buffer, and connection state.
3. **Miner process crash or freeze**: The miner becomes unable to submit valid blocks, halting mining operations.

---

### Likelihood Explanation

The miner's `rpc_url` points to a CKB node RPC endpoint. In production, the CKB node's RPC can become slow under heavy load (e.g., many concurrent RPC callers, expensive chain queries). The miner polls `get_block_template` on every `poll_interval` tick and also submits blocks via `submit_block`. Both paths go through the same timeout-free `Rpc` client. Under sustained RPC slowness, task accumulation is automatic and requires no special attacker action beyond causing the node to be slow. [6](#0-5) 

---

### Recommendation

Add a timeout to the `HttpClient` builder in `Rpc::new`, and additionally wrap each `client.request(req).await` in a `tokio::time::timeout`, mirroring the pattern already used in the block assembler's `notify` function:

```rust
// Set connection-level timeout on the connector
let mut connector = hyper_tls::HttpsConnector::new();
// Use a timeout wrapper around each spawned request
let request_timeout = Duration::from_secs(30);
loop_handle.spawn(async move {
    let request = match tokio::time::timeout(request_timeout, client.request(req)).await {
        Ok(Ok(res)) => BodyExt::collect(res.into_body()).await.map_err(RpcError::Http).map(|t| t.to_bytes()),
        Ok(Err(err)) => Err(RpcError::Client(err)),
        Err(_elapsed) => Err(RpcError::Canceled),
    };
    if sender.send(request).is_err() {
        error!("rpc response send back error")
    }
});
```

---

### Proof of Concept

1. Configure the CKB miner with `rpc_url` pointing to a CKB node.
2. Make the CKB node's RPC endpoint artificially slow (e.g., by saturating it with expensive `get_block_template` calls from many concurrent RPC callers, or by inserting a delay at the TCP level using `tc netem`).
3. Observe the miner process: with `poll_interval` set to a small value, the miner continuously calls `fetch_block_template`, each call spawning a new tokio task via `loop_handle.spawn`. Since `client.request(req).await` never times out, tasks accumulate.
4. Monitor memory usage of the miner process — it grows unboundedly until the process is OOM-killed or becomes unresponsive, halting all block submissions. [7](#0-6)
