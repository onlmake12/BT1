Audit Report

## Title
Off-by-one in `MAX_VMS_COUNT` spawn guard allows 17th VM creation — (`script/src/scheduler.rs`)

## Summary
`MAX_VMS_COUNT` is defined as `16`, but the spawn guard at line 532 uses strict `>` instead of `>=`. When exactly 16 VMs are alive, the condition `16 > 16` evaluates to `false`, the guard does not fire, and `boot_vm` is called — creating a 17th VM in violation of the documented invariant. The fix is a one-character change from `>` to `>=`.

## Finding Description
`MAX_VMS_COUNT` is set to `16`: [1](#0-0) 

The `Message::Spawn` arm checks: [2](#0-1) 

When `suspended.len() + instantiated.len() == 16`, the expression `16 > 16` is `false`. The guard is skipped, execution falls through to `boot_vm` at line 539, and a 17th VM is instantiated. The existing fd-ownership check at lines 525–531 is unrelated and provides no protection against this path. No other guard exists between the check and `boot_vm`. [3](#0-2) 

## Impact Explanation
This is an incorrect implementation of CKB-VM's documented VM-count invariant, matching the allowed impact: **"Incorrect implementation or behavior of CKB-VM or system scripts" (High, 10001–15000 points)**. The extra VM allocation consumes memory beyond the intended cap. While the cycle limit bounds total execution and prevents unbounded exhaustion, the invariant violation is concrete and observable. A patched node (rejects at 16) and an unpatched node (allows 17) would disagree on script execution validity, creating a consensus divergence risk during any rolling upgrade — escalating the concern further.

## Likelihood Explanation
Trivially reachable by any unprivileged script author. No PoW, no key material, no privileged access required. A script that spawns 15 children (root + 15 = 16 total) and then issues one additional `spawn` syscall reliably triggers the bug. The condition is deterministic and repeatable on every node running the affected binary.

## Recommendation
Change `>` to `>=` on line 532:

```rust
// Before
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
// After
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

## Proof of Concept
Write a unit test in `script/src/verify/tests/ckb_latest/features_since_v2023.rs` (where `MAX_VMS_COUNT` and `MAX_VMS_SPAWNED` are already referenced):

1. Construct a V2 script that spawns itself recursively until 15 children exist (root + 15 = 16 total VMs).
2. Issue one additional `spawn` syscall from any live VM.
3. Assert the return value in register `A0` equals `MAX_VMS_SPAWNED` (8).

With the current `>` guard, the return code is `SUCCESS` (0), confirming the 17th VM was created. After changing to `>=`, the return code is `MAX_VMS_SPAWNED` (8), confirming the guard fires correctly at the boundary.
