### Title
`whitelistEnabled` Defaults to `false`, Allowing Any Address to Read Subscription Price Data Despite a Configured `readerWhitelist` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `onlyWhitelistedReader` modifier gates all price-read functions behind a per-subscription `whitelistEnabled` boolean. Because Solidity initializes `bool` fields to `false`, any subscription created without explicitly setting `whitelistEnabled = true` silently disables the whitelist — even when a non-empty `readerWhitelist` is supplied. `_validateSubscriptionParams` performs no consistency check between the two fields, so the whitelist array is stored but never enforced, and any unprivileged address can read the subscription's price data.

---

### Finding Description

**Root cause — `SubscriptionParams.whitelistEnabled` defaults to `false`:**

In `SchedulerStructs.sol`, the struct field is a plain `bool`:

```solidity
struct SubscriptionParams {
    bytes32[] priceIds;
    address[] readerWhitelist;
    bool whitelistEnabled;   // ← Solidity default: false
    bool isActive;
    bool isPermanent;
    UpdateCriteria updateCriteria;
}
``` [1](#0-0) 

**Guard logic — `onlyWhitelistedReader` short-circuits on `whitelistEnabled == false`:**

```solidity
modifier onlyWhitelistedReader(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] == msg.sender) { _; return; }

    // If whitelist is not used, allow any reader
    if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) {
        _;
        return;
    }
    // ... whitelist membership check only reached when whitelistEnabled == true
}
``` [2](#0-1) 

**Missing validation — `_validateSubscriptionParams` never checks consistency:**

The validation function enforces whitelist size and uniqueness but never requires `whitelistEnabled = true` when `readerWhitelist.length > 0`:

```solidity
// Whitelist size limit and uniqueness
if (params.readerWhitelist.length > MAX_READER_WHITELIST_SIZE) { revert ...; }
for (uint i = 0; i < params.readerWhitelist.length; i++) {
    for (uint j = i + 1; ...) {
        if (params.readerWhitelist[i] == params.readerWhitelist[j]) { revert ...; }
    }
}
// ← No check: if readerWhitelist.length > 0, require whitelistEnabled == true
``` [3](#0-2) 

**Affected read functions** — all four price-read entry points use `onlyWhitelistedReader`:

- `getPricesUnsafe` [4](#0-3) 
- `getPricesNoOlderThan` [5](#0-4) 
- `getEmaPricesUnsafe` [6](#0-5) 
- `getEmaPricesNoOlderThan` [7](#0-6) 

**Exploit flow:**

1. A subscription manager calls `createSubscription` and passes a `readerWhitelist` containing only their own protocol addresses, but omits `whitelistEnabled: true` (or relies on the struct zero-value).
2. `_validateSubscriptionParams` accepts the params without error.
3. The subscription is stored with `whitelistEnabled = false` and a non-empty `readerWhitelist` that is never consulted.
4. Any unprivileged EOA or contract calls `getPricesUnsafe(subscriptionId, [])` and receives all price feeds without restriction.

---

### Impact Explanation

A subscription manager who intends to gate proprietary price data behind a whitelist (e.g., to protect a trading strategy or enforce a paid-access model) silently receives no protection. Any on-chain address can read the subscription's price feeds at zero cost. This constitutes:

- **Unauthorized data access / confidentiality breach**: price data intended for restricted consumers is exposed to the entire network.
- **Free-riding on a paid service**: the manager funds keeper updates; competitors consume the resulting data without paying.
- **Indirect financial harm**: front-running or information-asymmetry attacks become possible for any actor who reads the restricted feed before the intended consumer acts on it.

---

### Likelihood Explanation

Solidity's zero-initialization of `bool` fields is a well-known footgun. A developer who constructs `SubscriptionParams` in memory (e.g., via a struct literal or ABI-decoded calldata) and populates `readerWhitelist` without explicitly writing `whitelistEnabled: true` will silently disable the guard. No on-chain warning, revert, or event signals the misconfiguration. The `_validateSubscriptionParams` function, which is the natural place to catch this, performs no cross-field consistency check. Likelihood is **medium**: the mistake is easy to make and invisible until an unauthorized reader is detected off-chain.

---

### Recommendation

**Short term:** Add a consistency check in `_validateSubscriptionParams`:

```solidity
if (params.readerWhitelist.length > 0 && !params.whitelistEnabled) {
    revert SchedulerErrors.WhitelistProvidedButNotEnabled();
}
```

**Long term:** Consider defaulting `whitelistEnabled` to `true` whenever a non-empty `readerWhitelist` is supplied, or restructure the API so that providing a whitelist implicitly enables enforcement, eliminating the two-field inconsistency entirely.

---

### Proof of Concept

```solidity
// Attacker (Eve) reads a subscription whose manager intended to restrict access

// 1. Manager creates subscription with a whitelist but forgets whitelistEnabled
SchedulerStructs.SubscriptionParams memory params;
params.priceIds        = priceIds;
params.readerWhitelist = [managerAddress];   // intended restriction
// params.whitelistEnabled is NOT set → defaults to false
params.isActive        = true;
params.updateCriteria  = UpdateCriteria({updateOnHeartbeat: true, heartbeatSeconds: 60, ...});

uint256 subId = scheduler.createSubscription{value: minBalance}(params);

// 2. Eve (unprivileged) reads the subscription
// onlyWhitelistedReader: whitelistEnabled == false → short-circuits, allows any caller
PythStructs.Price[] memory prices = scheduler.getPricesUnsafe(subId, new bytes32[](0));
// → succeeds; Eve receives all price feeds
```

    modifier onlyWhitelistedReader(uint256 subscriptionId) {
        // Manager is always allowed
        if (_state.subscriptionManager[subscriptionId] == msg.sender) {
            _;
            return;
        }

        // If whitelist is not used, allow any reader
        if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) {
            _;
            return;
        }

        // Check if caller is in whitelist
        address[] storage whitelist = _state
            .subscriptionParams[subscriptionId]
            .readerWhitelist;
        bool isWhitelisted = false;
        for (uint i = 0; i < whitelist.length; i++) {
            if (whitelist[i] == msg.sender) {
                isWhitelisted = true;
                break;
            }
        }

        if (!isWhitelisted) {
            revert SchedulerErrors.Unauthorized();
        }
        _;
    }
