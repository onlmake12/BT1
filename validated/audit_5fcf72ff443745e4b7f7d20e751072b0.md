### Title
Unbounded `get_cells_capacity` scan holds RPC threads for full timeout, enabling thread-pool exhaustion — (`util/indexer/src/service.rs`)

### Summary
`IndexerHandle::get_cells_capacity` performs a full prefix scan over RocksDB with no per-request entry limit. The only guard is a `TimeoutIterator` that terminates the scan after `timeout_limit` (default 10 s) and returns an error. Because the handler is synchronous, each call blocks a jsonrpc thread for up to 10 seconds. An unprivileged caller sending concurrent `get_cells_capacity` requests with empty `args` (matching all cells under a `code_hash`) can exhaust the RPC thread pool, causing sustained node-level RPC congestion.

### Finding Description
`get_cells_capacity` (lines 686–854, `util/indexer/src/service.rs`) differs from `get_cells` and `get_transactions` in one critical way: it accepts no `limit` parameter and never consults `self.request_limit`. The scan runs until either the prefix boundary is reached or `TimeoutIterator` fires. With a large number of indexed cells under one `code_hash` and empty `args`, the scan runs for the full `timeout_limit` (default 10 s) before returning an error. The synchronous RPC handler blocks its thread for that entire duration. Sending `T` concurrent requests (where `T` equals the jsonrpc thread pool size) saturates the pool, blocking all other RPC calls. [1](#0-0) [8](#0-7) 

### Impact Explanation
- RPC thread pool exhaustion: all RPC methods (including non-indexer ones) become unresponsive for the duration of the attack.
- Each attacker request costs only a TCP connection and a small JSON payload; the defender pays 10 CPU-seconds of RocksDB I/O per request.
- The attack is self-sustaining: as soon as a thread is freed (timeout fires), the attacker reissues the request.

### Likelihood Explanation
- The indexer RPC is publicly exposed by default when the indexer is enabled.
- No authentication is required.
- The precondition (many cells under one `code_hash`) is trivially met on mainnet for common lock scripts (e.g., the secp256k1 lock used by millions of cells).
- The attack requires only repeated JSON-RPC calls with a single well-known `code_hash` and empty `args`.

### Recommendation
Apply `self.request_limit` inside `get_cells_capacity` as an early-exit count, consistent with how `get_cells` uses its `limit` parameter. Additionally, consider running indexer RPC handlers on a dedicated, bounded async executor (as the rich-indexer already does) to isolate them from the main RPC thread pool.

### Proof of Concept
1. Enable the CKB indexer on a node with many cells indexed under `0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8` (secp256k1 lock).
2. Send `T` concurrent requests (where `T` = jsonrpc thread pool size, default 4):
```json
{
  "jsonrpc": "2.0", "method": "get_cells_capacity", "id": 1,
  "params": [{"script": {"code_hash": "0x9bd7e06f...", "hash_type": "type", "args": "0x"}, "script_type": "lock"}]
}
```
3. Observe that all threads are held for 10 s (timeout fires), all other RPC calls time out, and the node is unresponsive to RPC for the attack duration. [5](#0-4) [9](#0-8)

            .map(|mode| *mode == IndexerSearchMode::Partial)
            .unwrap_or(false)
        {
            return Err(Error::invalid_params(
                "the CKB indexer doesn't support search_key.script_search_mode partial search mode, \
                please use the CKB rich-indexer for such search",
            ));
        }

        let (prefix, from_key, direction, skip) = build_query_options(
            &search_key,
            KeyPrefix::CellLockScript,
            KeyPrefix::CellTypeScript,
            IndexerOrder::Asc,
            None,
        )?;
        let filter_script_type = match search_key.script_type {
            IndexerScriptType::Lock => IndexerScriptType::Type,
            IndexerScriptType::Type => IndexerScriptType::Lock,
        };

            search_key.script_search_mode,
            Some(IndexerSearchMode::Exact)
        );
        let filter_options: FilterOptions = search_key.try_into()?;
        let mode = IteratorMode::From(from_key.as_ref(), direction);
        let snapshot = self.store.inner().snapshot();
        let mut iter = TimeoutIterator::new(snapshot.iterator(mode).skip(skip), self.timeout_limit);
