### Title
Verify Queue Admission Without Fee Check Enables Low-Cost DoS of Transaction Pool — (`tx-pool/src/process.rs`)

### Summary

The CKB tx-pool enqueues incoming transactions into the verify queue **before** performing the fee-rate check. An unprivileged P2P peer can exploit the normal transaction relay protocol to flood the 256 MB verify queue with structurally valid but zero-fee transactions crafted with fake inputs, causing legitimate transactions to be rejected with `Reject::Full` at near-zero attacker cost.

---

### Finding Description

**Two-stage pipeline with fee check in the wrong stage:**

The relay submission path is:

```
submit_remote_tx
  → resumeble_process_tx_and_notify_full_reject
    → resumeble_process_tx
      → non_contextual_verify   ← structural check only, no fee check
      → enqueue_verify_queue    ← transaction admitted here, NO fee check
```

The fee check only happens later, when a worker dequeues the entry:

```
Worker::process_inner
  → _process_tx
    → pre_check
      → check_tx_fee            ← fee check happens HERE, after admission
```

`resumeble_process_tx` in `tx-pool/src/process.rs` calls `enqueue_verify_queue` without any fee gate: [1](#0-0) 

`check_tx_fee` in `tx-pool/src/util.rs` compares the fee against `min_fee_rate * tx_size`, but this runs only after the transaction already occupies verify queue memory: [2](#0-1) 

**Verify queue hard limit is 256 MB:** [3](#0-2) 

When the queue is full, `add_tx` returns `Reject::Full`, blocking all subsequent enqueue attempts: [4](#0-3) 

**`non_contextual_verify` does not check input existence or fees:**

`NonContextualTransactionVerifier` checks only structural properties (version, input/output counts, script sizes, capacity overflow). It does **not** verify that inputs exist on-chain or that the fee rate is sufficient. An attacker can craft transactions with random fake input `OutPoint`s that pass this check at zero cost.

**Reachable P2P entry path:**

The relay handler in `sync/src/relayer/transactions_process.rs` processes incoming `RelayTransactions` messages. The filter requires only that the node previously requested the transaction hash from this peer: [5](#0-4) 

An attacker can satisfy this by first sending `RelayTransactionHashes` (announcing fake transaction hashes), causing the node to issue `GetRelayTransactions` requests, then responding with the crafted zero-fee transactions. This is the standard relay protocol flow.

**Attack sequence:**

1. Attacker connects as a P2P peer.
2. Attacker sends `RelayTransactionHashes` with many new transaction hashes.
3. Node requests them (adds to `unknown_tx_hashes`).
4. Attacker sends `RelayTransactions` with structurally valid transactions containing fake inputs and zero fees (each up to `TRANSACTION_SIZE_LIMIT = 512 KB`). [6](#0-5) 
5. Transactions pass `non_contextual_verify` and are enqueued in the verify queue.
6. ~500 such transactions (512 KB each) saturate the 256 MB verify queue.
7. All subsequent legitimate transaction submissions are rejected with `Reject::Full`.
8. Workers eventually drain the queue (fee check fails, transactions rejected), but attacker repeats immediately.

---

### Impact Explanation

While the verify queue is saturated, every new transaction submitted via RPC or relayed from other peers is rejected with `Reject::Full`. Legitimate users cannot get transactions into the pool. The attacker can maintain this state continuously by re-flooding as the queue drains. This is analogous to M-04: the absence of an upfront economic barrier (fee check before admission) makes the disruption essentially free for the attacker.

---

### Likelihood Explanation

Any unprivileged P2P peer can execute this attack using the standard relay protocol. No special privileges, keys, or hashpower are required. The attacker's cost is only network bandwidth and the CPU to craft ~500 structurally valid transactions. The attack is repeatable and can be sustained indefinitely.

---

### Recommendation

Move the fee-rate check to **before** verify queue admission. In `resumeble_process_tx`, perform a lightweight fee pre-check (using the declared cycles and transaction size) before calling `enqueue_verify_queue`. Alternatively, add a minimum fee-rate gate in `enqueue_verify_queue` itself that rejects transactions whose declared fee rate (computable from the transaction's serialized size and a conservative fee estimate) falls below `min_fee_rate` without requiring full contextual resolution.

---

### Proof of Concept

```
1. Attacker peer connects to CKB node.
2. Attacker sends RelayTransactionHashes([H1, H2, ..., H500])
   where H_i = hash of crafted_tx_i.
3. Node responds with GetRelayTransactions([H1..H500]).
4. Attacker sends RelayTransactions([crafted_tx_1, ..., crafted_tx_500])
   where each crafted_tx_i:
     - inputs: [OutPoint { tx_hash: random_bytes, index: 0 }]  ← fake, non-existent
     - outputs: [CellOutput { capacity: 0xFFFFFFFF, lock: always_success }]
     - witnesses: [valid_witness_structure]
     - serialized_size ≈ 512 KB (TRANSACTION_SIZE_LIMIT)
     - fee: 0 shannons
5. Each tx passes non_contextual_verify (structural check only).
6. Each tx is enqueued: 500 × 512 KB ≈ 256 MB → verify queue full.
7. All legitimate send_transaction RPC calls now return Reject::Full.
8. Workers drain queue (resolve_tx fails for fake inputs, or check_tx_fee
   fails for zero fee), but attacker immediately re-floods.
```

        &self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, remote).await?;

        if self.orphan_contains(&tx).await {
            debug!("reject tx {} already in orphan pool", tx.hash());
            return Err(Reject::Duplicated(tx.hash()));
        }

        if self.verify_queue_contains(&tx).await {
            return Err(Reject::Duplicated(tx.hash()));
        }
        self.enqueue_verify_queue(tx, is_proposal_tx, remote).await
    }

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)

    pub fn execute(self) -> Status {
        let shared_state = self.relayer.shared().state();
        let txs: Vec<(TransactionView, Cycle)> = {
            // ignore the tx if it's already known or it has never been requested before
            let mut tx_filter = shared_state.tx_filter();
            tx_filter.remove_expired();
            let unknown_tx_hashes = shared_state.unknown_tx_hashes();

            self.message
                .transactions()
                .iter()
                .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
                .filter(|(tx, _)| {
                    !tx_filter.contains(&tx.hash())
                        && unknown_tx_hashes
                            .get_priority(&tx.hash())
                            .map(|priority| priority.requesting_peer() == Some(self.peer))
                            .unwrap_or_default()
                })
                .collect()
        };

        if txs.is_empty() {
            return Status::ok();
        }

        let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
        if txs
            .iter()
            .any(|(_, declared_cycles)| declared_cycles > &max_block_cycles)
        {
            self.nc.ban_peer(
                self.peer,
                DEFAULT_BAN_TIME,
                String::from("relay declared cycles greater than max_block_cycles"),
            );
            return Status::ok();
        }

        shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));

        let tx_pool = self.relayer.shared.shared().tx_pool_controller().clone();
        let peer = self.peer;
        self.relayer
            .shared
            .shared()
            .async_handle()
            .spawn(async move {
                for (tx, declared_cycles) in txs {
                    if let Err(e) = tx_pool
                        .submit_remote_tx(tx.clone(), declared_cycles, peer)
                        .await
                    {
                        error!("submit_tx error {}", e);
                    }
                }
            });

        Status::ok()
    }
