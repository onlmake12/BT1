### Title
Unauthenticated Block Template Injection via Miner Notify HTTP Listener Allows Mining Reward Theft - (File: `miner/src/client.rs`)

### Summary

The CKB miner's "notify mode" HTTP listener (`listen_block_template_notify`) accepts block template pushes from any connecting client with no authentication, IP restriction, or shared-secret check. This endpoint is designed to be called exclusively by the trusted CKB node's block assembler, but there is no access control to enforce this. An attacker who can reach the listener port can inject a crafted `BlockTemplate` containing their own lock script in the cellbase witness, causing the miner to mine a valid block whose future block reward is paid to the attacker's address.

### Finding Description

When the CKB miner is configured in notify mode (`[miner.client] listen = "<addr>"`), it starts an HTTP server via `listen_block_template_notify`. Every incoming HTTP POST is handled by the free function `handle`:

```rust
// miner/src/client.rs lines 358-369
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();

    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);   // ← no auth check
    }
    Ok(Response::new(Empty::new()))
}
``` [1](#0-0) 

There is no check on the source IP, no shared secret, no HMAC, and no token. Any process that can open a TCP connection to the listen address can call `update_block_template` with an arbitrary `BlockTemplate`.

`update_block_template` then forwards the template to the mining workers:

```rust
// miner/src/client.rs lines 293-312
fn update_block_template(&self, block_template: BlockTemplate) {
    let work_id = block_template.work_id.into();
    let updated = |id| {
        if id != work_id || id == 0 { Some(work_id) } else { None }
    };
    if self.current_work_id.fetch_update(..., updated).is_ok() {
        let work: Work = block_template.into();
        self.new_work_tx.send(Works::New(work)) ...
    }
}
``` [2](#0-1) 

The `work_id` comparison is a deduplication guard, not an authentication check. An attacker sets `work_id = 0` (the condition `id == 0` is always true on the first call) or any value differing from the current one to guarantee acceptance.

The miner then solves PoW on the injected template and submits the resulting block via `submit_block`:

```rust
// miner/src/miner.rs lines 179-181
self.client.submit_block(&work.work_id.to_string(), block.data())
``` [3](#0-2) 

**Reward theft path**: The `BlockTemplate` contains a `cellbase` field whose `CellbaseWitness` encodes the lock script that will receive the block reward when it is finalized (after `PROPOSAL_WINDOW.farthest + 1` blocks). The `CellbaseVerifier` only checks that the witness is a valid `CellbaseWitness` with an enabled hash type — it does **not** verify that the lock matches the node's configured `block_assembler` lock:

```rust
// verification/src/block_verifier.rs lines 106-124

The `RewardVerifier` checks the **output** lock of the current cellbase against the target lock derived from a **previous** block's cellbase witness — not the current block's witness. So a block with an attacker-controlled cellbase witness passes all consensus checks and, once finalized, pays the reward to the attacker's lock. [5](#0-4) 

### Impact Explanation

An attacker who can reach the miner's notify listener can:

1. **Steal block rewards**: Inject a template with their own lock script in the cellbase witness. The miner mines a fully valid block; after `PROPOSAL_WINDOW.farthest + 1` blocks the reward is paid to the attacker's address, not the legitimate miner's.
2. **Waste hashpower**: Inject templates building on a stale or attacker-chosen parent hash, causing the miner to produce orphan blocks and earn nothing.
3. **Transaction censorship / inclusion**: Inject templates with a chosen transaction set, overriding the node's fee-ordered selection.

Impact 1 is direct financial loss to the miner with no recovery path once the block is confirmed.

### Likelihood Explanation

The notify mode is an opt-in feature (`listen` is commented out by default in `resource/ckb-miner.toml`). [6](#0-5) 

However, when enabled, the default suggested address is `127.0.0.1:8888`. Any process running on the same host (a co-located malicious process, a compromised dependency, or a web-based SSRF) can exploit this without any credentials. If a miner binds to `0.0.0.0`, the attack is remotely reachable with no privilege. Mining operations commonly run multiple processes on the same host, making local exploitation realistic.

### Recommendation

Add origin verification to the notify listener. The simplest approach is a configurable shared secret (e.g., a bearer token or HMAC) that the CKB node includes in the `Authorization` header when posting templates, and that the miner verifies before calling `update_block_template`. Alternatively, restrict accepted connections to a configurable allowlist of source IPs (e.g., `127.0.0.1` only by default).

### Proof of Concept

**Precondition**: Miner is running with `listen = "127.0.0.1:8888"` (or any reachable address).

**Steps**:

1. Attacker calls `get_block_template` on the CKB node to obtain a valid current template.
2. Attacker replaces the `cellbase.data.witnesses[0]` field with a `CellbaseWitness` encoding their own lock script (any valid secp256k1 lock pointing to the attacker's pubkey hash).
3. Attacker POSTs the modified template as JSON to `http://127.0.0.1:8888/`:
   ```
   POST / HTTP/1.1
   Content-Type: application/json

   { "version": "0x0", "compact_target": "...", "cellbase": { "data": { "witnesses": ["<attacker_witness>"], ... } }, "work_id": "0x0", ... }
   ```
4. The `handle` function deserializes the template and calls `update_block_template` — no authentication check occurs.
5. The miner's workers receive `Works::New(work)` containing the attacker's template and begin solving PoW on it.
6. Upon finding a valid nonce, the miner calls `submit_block` on the CKB node. The block passes `CellbaseVerifier` (valid witness format) and `RewardVerifier` (output lock matches the *previous* target block's witness, not the current one).
7. After `PROPOSAL_WINDOW.farthest + 1` blocks, the block reward is finalized and paid to the attacker's lock script.

    pub fn verify(&self) -> Result<(), Error> {
        let cellbase = &self.resolved[0];
        let no_finalization_target =
            (self.parent.number() + 1) <= self.context.consensus.finalization_delay_length();

        let (target_lock, block_reward) = self.context.finalize_block_reward(self.parent)?;
        let output = CellOutput::new_builder()
            .capacity(block_reward.total)
            .lock(target_lock.clone())
            .build();
        let insufficient_reward_to_create_cell = output.is_lack_of_capacity(Capacity::zero())?;

        if no_finalization_target || insufficient_reward_to_create_cell {
            let ret = if cellbase.transaction.outputs().is_empty() {
                Ok(())
            } else {
                Err((CellbaseError::InvalidRewardTarget).into())
            };
            return ret;
        }

        if !insufficient_reward_to_create_cell {
            if cellbase.transaction.outputs_capacity()? != block_reward.total {
                return Err((CellbaseError::InvalidRewardAmount).into());
            }
            if cellbase
                .transaction
                .outputs()
                .get(0)
                .expect("cellbase should have output")
                .lock()
                != target_lock
            {
                return Err((CellbaseError::InvalidRewardTarget).into());
            }
        }

        Ok(())
    }
