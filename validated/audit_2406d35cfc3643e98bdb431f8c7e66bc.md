### Title
Staking Accounting Tracking Error: `del_state.total_delegation` Does Not Reflect Pending Slash Events, Enabling Slash Evasion via `undelegate` Front-Running — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The Pyth Oracle Integrity Staking (OIS) integrity pool program tracks per-publisher delegation amounts in `PoolData.del_state[i].total_delegation`, a static field updated only on `delegate`/`undelegate`. When `create_slash_event` is called, this field is **not updated**. Separately, the `undelegate` instruction does not include `delegationRecord` in its account list, meaning it cannot enforce that pending slash events have been applied before a user exits. A delegator who monitors the chain for `create_slash_event` transactions can front-run the slash by calling `undelegate` before `slash` is applied to their specific account, potentially escaping the penalty entirely.

---

### Finding Description

**Static tracking variable does not reflect pending slashes**

`PoolData` stores per-publisher delegation state as:

```
del_state[i]: { total_delegation: u64, delta_delegation: i64 }
self_del_state[i]: { total_delegation: u64, delta_delegation: i64 }
``` [1](#0-0) 

When `create_slash_event` is called by `reward_program_authority`, it creates a `SlashEvent` PDA with `epoch`, `slash_ratio`, and `slash_custody`, and increments `pool_data.num_slash_events[publisher_index]`. It does **not** reduce `del_state[i].total_delegation`. [2](#0-1) 

The slash is applied lazily — either via `advance_delegation_record` (permissionless, processes rewards and pending slashes for one user) or via the direct `slash` instruction (also permissionless, applies one specific `SlashEvent` to one user's `stake_account_positions` and transfers tokens from `stake_account_custody`). [3](#0-2) 

**`undelegate` does not include `delegationRecord` in its accounts**

The `undelegate` instruction's account list is:

```
owner (signer), poolData, poolConfig, publisher,
configAccount, stakeAccountPositions, stakeAccountMetadata,
stakeAccountCustody, stakingProgram, systemProgram
``` [4](#0-3) 

`delegationRecord` — which holds `next_slash_event_index` — is absent. The program therefore cannot check whether `next_slash_event_index < pool_data.num_slash_events[publisher_index]` (i.e., whether there are unapplied slashes) at the time of undelegation.

**`totalStaked` display uses the stale `del_state` directly**

```typescript
totalStaked:
  sumDelegations(poolData.delState) + sumDelegations(poolData.selfDelState),
``` [5](#0-4) 

```typescript
const sumDelegations = (values) =>
  values.reduce((acc, value) => acc + value.totalDelegation + value.deltaDelegation, 0n);
``` [6](#0-5) 

After `create_slash_event` but before all individual `slash` calls complete, `totalStaked` overstates the actual staked amount — a direct analog to `assetsAllocatedToStrategies` not reflecting slashing losses.

---

### Impact Explanation

1. **Slash evasion**: A delegator who observes `create_slash_event` on-chain can immediately call `undelegate`, moving their position from `LOCKED` to `PREUNLOCKING`. If the `slash` instruction enforces `UnexpectedPositionState` (error 6005) and only operates on `LOCKED` positions, the delegator's tokens are never slashed. The slash is absorbed by remaining stakers, diluting the penalty. [7](#0-6) 

2. **Overstated pool accounting**: `del_state[i].total_delegation` overstates actual staked amounts between `create_slash_event` and full slash application. When `y > 0`, rewards are computed on this inflated base, distributing excess rewards.

3. **Incorrect `totalStaked` display**: The staking dashboard shows an inflated total, misleading users about pool health post-slash.

**Impact: Medium** — Direct financial loss (slash evasion) for the protocol/DAO; overstated reward base when `y > 0`.

---

### Likelihood Explanation

- `create_slash_event` is an on-chain transaction visible to all participants before finality.
- `undelegate` is a permissionless user instruction callable in the same block or the next.
- Solana's mempool is not public, but the transaction is visible once included; a delegator monitoring via websocket can react within seconds.
- Slashing events are rare but high-value (up to 5% of pool stake per the rulebook). [8](#0-7) 

**Likelihood: Medium** — Requires active monitoring and fast reaction, but the attack is straightforward for a sophisticated delegator.

---

### Recommendation

1. **Enforce `OutdatedDelegatorAccounting` in `undelegate`**: Add `delegationRecord` as a required account in the `undelegate` instruction and verify `delegation_record.next_slash_event_index == pool_data.num_slash_events[publisher_index]` before allowing undelegation. The error code `OutdatedDelegatorAccounting` (6008) already exists in the IDL but is not enforced at this instruction boundary. [9](#0-8) 

2. **Apply slashes atomically on `create_slash_event`**: Rather than lazy per-user application, update `del_state[i].total_delegation` immediately when `create_slash_event` is called, so the pool-level accounting is always accurate.

3. **Allow `slash` on all position states**: Ensure the `slash` instruction can be applied to positions in `PREUNLOCKING` and `UNLOCKING` states, not only `LOCKED`, to close the front-running window.

---

### Proof of Concept

1. Delegator stakes 1,000,000 PYTH to publisher P. `del_state[P].total_delegation = 1_000_000`.
2. A data misprint occurs. `reward_program_authority` calls `create_slash_event(publisher=P, index=0, slash_ratio=500_bps)`. `pool_data.num_slash_events[P] = 1`. `del_state[P].total_delegation` remains `1_000_000` (not updated).
3. Delegator's bot detects the `create_slash_event` transaction on-chain.
4. Delegator immediately calls `undelegate(position_index, amount=1_000_000)`. Since `delegationRecord` is absent from `undelegate`'s accounts, no check on `next_slash_event_index` is performed. Position moves to `PREUNLOCKING`.
5. `slash` is called on the delegator's account. If `slash` enforces `UnexpectedPositionState` for non-`LOCKED` positions, the call fails. The delegator's 1,000,000 PYTH is never slashed.
6. After the cooldown period, the delegator withdraws the full 1,000,000 PYTH, having avoided the 50,000 PYTH slash. [10](#0-9) [11](#0-10) 

> **Note**: Full confirmation of step 5 (whether `slash` enforces position state) requires the Rust source of the integrity pool program, which is not present in this repository. The IDL-level evidence (absent `delegationRecord` in `undelegate`, existence of `OutdatedDelegatorAccounting` error that is not enforced there, and `UnexpectedPositionState` error in the program) is consistent with this attack path being valid.
