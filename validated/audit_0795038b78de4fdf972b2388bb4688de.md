Audit Report

## Title
Rate Limiter Keyed by Ephemeral `PeerIndex` (`SessionId`) Allows Rate Limit Bypass via Reconnect Cycling — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
The `HolePunching` protocol's `rate_limiter` is keyed by `(PeerIndex, u32)` where `PeerIndex` is a type alias for `SessionId`, an ephemeral integer that is reassigned fresh on every TCP connection. An attacker can bypass the 30 msg/sec cap by disconnecting and reconnecting, obtaining a new `SessionId` and thus a clean rate limit slate. Each bypassed `ConnectionRequest` can trigger a gossip broadcast to `sqrt(N)` peers, enabling network amplification with no meaningful cost barrier.

## Finding Description
The `HolePunching` struct declares its primary rate limiter keyed by the ephemeral session ID:

```rust
rate_limiter: RateLimiter<(PeerIndex, u32)>,
``` [1](#0-0) 

`PeerIndex` is confirmed to be a type alias for `SessionId`, which is a monotonically increasing integer assigned per TCP connection:

```rust
pub type PeerIndex = SessionId;
``` [2](#0-1) 

In `received()`, the rate check uses this ephemeral `session_id`:

```rust
if self.rate_limiter.check_key(&(session_id, msg.item_id())).is_err() {
    ...
    return;
}
``` [3](#0-2) 

The `disconnected()` handler only calls `retain_recent()` (a GC operation) and does **not** remove or invalidate the disconnected session's rate limit entry. Crucially, it also does **not** ban the peer:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    ...
}
``` [4](#0-3) 

When a peer reconnects, `accept_peer` checks for duplicate `PeerId` among **currently connected** peers only. After disconnect, the peer is removed from the registry, so the `PeerIdExists` guard does not prevent reconnection: [5](#0-4) 

The new connection receives a fresh `SessionId` (e.g., `N+1`), so the key `(N+1, item_id)` has no prior rate limit history and the 30 msg/sec window starts fresh. The rate limiter is never triggered on reconnect.

The `forward_rate_limiter` is correctly keyed by `(PeerId, PeerId, u32)` — stable identity-based keys — but the primary `rate_limiter` guarding all message dispatch is not: [6](#0-5) 

The critical amplification path is confirmed in `ConnectionRequestProcess::execute()` → `forward_message()`: when the target peer is not directly connected, the node performs a gossip broadcast to `sqrt(total_peers)` other nodes:

```rust
let mut total = self.protocol.network_state
    .with_peer_registry(|p| p.peers().len()).isqrt();
// filter_broadcast to sqrt(N) peers
``` [7](#0-6) 

## Impact Explanation
This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs.*

An attacker cycling connections at rate `R` reconnects/sec delivers `30 × R` `ConnectionRequest` messages per second to the victim node. Each message that reaches `forward_message()` (i.e., where the target peer is not directly connected, which is the common case) triggers a `filter_broadcast` to `sqrt(N)` peers. With 125 connected peers, each attacker message fans out to ~11 additional nodes, each of which may repeat the same processing. This constitutes a low-cost, amplified network congestion attack against the CKB P2P layer.

## Likelihood Explanation
The attack requires only a standard inbound P2P connection — no special privileges, no leaked keys, no majority hashpower. The attacker implements a simple loop: connect → send 30 `ConnectionRequest` messages → disconnect → reconnect. The secio handshake adds per-reconnect latency but does not prevent the attack at meaningful scale. The `forward_rate_limiter` (keyed by `(from_PeerId, to_PeerId, item_id)`) provides a secondary check, but an attacker can trivially vary the `from` and `to` fields in each message to bypass it as well, since those fields are attacker-controlled content in the message payload. [8](#0-7) 

## Recommendation
Key the `rate_limiter` by `(PeerId, u32)` instead of `(PeerIndex, u32)`. The `PeerId` is stable across reconnects and is available via `extract_peer_id(&context.session.address)` at the point of the `received()` call. This matches the design already used by `forward_rate_limiter`. Additionally, consider banning peers that repeatedly trigger the rate limiter rather than silently dropping messages. [9](#0-8) 

## Proof of Concept
```
Attacker controls a node with a valid keypair (stable PeerId P).

loop:
  1. Connect to victim node → assigned new SessionId N
  2. Send 30 ConnectionRequest messages with arbitrary from/to PeerIds
     → all pass rate_limiter check for key (N, item_id)
     → each triggers forward_message() → filter_broadcast to sqrt(peers) nodes
  3. Disconnect (SessionId N is retired; no ban issued)
  4. Reconnect → assigned new SessionId N+1
  5. Send 30 more ConnectionRequest messages
     → all pass rate_limiter check for key (N+1, item_id) (fresh window)
  ...

Total messages processed per second = 30 × reconnect_frequency.
Each message fans out to sqrt(N) additional peers via gossip broadcast.
Network amplification factor = sqrt(connected_peers).
```
