Audit Report

## Title
`WellKnownScriptsOnly` Args Validation Is a No-Op via `starts_with(&[])` — (`rpc/src/module/pool.rs`)

## Summary

`is_well_known_script` checks script args using Rust's `starts_with`, but every well-known lock and type script on both mainnet and testnet is hardcoded with `"args": "0x"` (empty bytes). Because `well_known_script.args().as_slice()` is always `&[]`, the call `script.args().as_slice().starts_with(&[])` is unconditionally `true` in Rust. The args field is never validated: any script sharing a well-known `code_hash`+`hash_type` with arbitrary args passes the `WellKnownScriptsOnly` validator, defeating its purpose and potentially causing wallets and exchanges to broadcast transactions that send funds to unspendable or attacker-controlled addresses.

## Finding Description

`is_well_known_script` is defined at `rpc/src/module/pool.rs` lines 912–919:

```rust
fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script
            .args()
            .as_slice()
            .starts_with(well_known_script.args().as_slice())
}
``` [1](#0-0) 

Every well-known lock script (`anyone_can_pay`, `cheque`) and type script (`Simple UDT`) for both mainnet and testnet is hardcoded with `"args": "0x"`: [2](#0-1) [3](#0-2) 

`well_known_script.args().as_slice()` therefore always returns `&[]`. In Rust, `<[u8]>::starts_with(&[])` returns `true` for every possible byte slice — the third condition is a tautology. The args field is never actually compared.

The exploit path is fully reachable by any unprivileged RPC caller:

1. Caller invokes `send_transaction` with `OutputsValidator::WellKnownScriptsOnly`.
2. `check_output_validator` dispatches to `WellKnownScriptsOnlyValidator::validate`. [4](#0-3) 
3. `validate_lock_script` falls through to `validate_well_known_lock_scripts` after the two secp256k1 checks fail on `code_hash` mismatch. [5](#0-4) 
4. `validate_well_known_lock_scripts` calls `is_well_known_script` and returns `Ok(())` on any match. [6](#0-5) 
5. Because the args check is a no-op, a script with the `anyone_can_pay` or `cheque` `code_hash`+`hash_type` and 100 bytes of arbitrary args passes without error.

## Impact Explanation

The `WellKnownScriptsOnly` validator exists specifically so wallets and exchanges can assert that every output in a transaction uses a script with known-safe semantics — including valid args. With the args check broken, an attacker can supply a payment address whose lock script carries a well-known `code_hash` but malformed or attacker-chosen args. The wallet's validator accepts it, the transaction is broadcast, and funds are sent to either an unspendable address (permanent destruction of CKB) or an address spendable only by the attacker. This constitutes concrete economic damage to CKB holders and undermines the safety guarantee that `WellKnownScriptsOnly` is documented to provide, matching the **"Vulnerabilities which could easily damage CKB economy"** impact class.

## Likelihood Explanation

`anyone_can_pay` and `cheque` scripts are widely deployed. Any party that can present a payment address to a wallet or exchange using `WellKnownScriptsOnly` — a standard integration pattern — can trigger the bypass. The attacker needs only to set the `code_hash` and `hash_type` to a well-known value and supply arbitrary args; no privileged access, no key material, and no victim mistake is required beyond the victim using the validator as documented.

## Recommendation

Replace the `starts_with` comparison with strict equality on args:

```rust
fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script.args() == well_known_script.args()
}
```

If prefix-matching is intentionally desired for future well-known scripts that carry non-empty args, add an explicit guard before the `starts_with` call:

```rust
&& (!well_known_script.args().as_slice().is_empty()
    && script.args().as_slice().starts_with(well_known_script.args().as_slice()))
```

## Proof of Concept

```rust
use ckb_types::{packed, prelude::*};

fn is_well_known_script(script: &packed::Script, well_known_script: &packed::Script) -> bool {
    script.hash_type() == well_known_script.hash_type()
        && script.code_hash() == well_known_script.code_hash()
        && script
            .args()
            .as_slice()
            .starts_with(well_known_script.args().as_slice())
}

#[test]
fn prove_args_bypass() {
    // Mirrors build_well_known_lock_scripts: "args": "0x" → empty
    let well_known = packed::Script::new_builder()
        .code_hash([0xd3u8; 32].pack())
        .hash_type(1u8.into())
        .args(packed::Bytes::default()) // &[]
        .build();

    // Attacker: same code_hash/hash_type, 100 arbitrary bytes as args
    let attacker_script = packed::Script::new_builder()
        .code_hash([0xd3u8; 32].pack())
        .hash_type(1u8.into())
        .args(vec![0xffu8; 100].pack())
        .build();

    // Should be false; is unconditionally true — bypass confirmed
    assert!(
        is_well_known_script(&attacker_script, &well_known),
        "BUG: starts_with(&[]) is always true; args are never validated"
    );
}
```

Running this test against the current implementation passes, confirming the invariant violation. The fix (replacing `starts_with` with `==`) causes the assertion to fail as expected, proving the guard is restored.
