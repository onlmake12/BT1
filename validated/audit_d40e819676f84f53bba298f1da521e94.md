### Title
Unbounded `last_n_numbers` via `difficulty_boundary=0` Bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged remote peer can send a `GetLastStateProof` message with `difficulty_boundary=U256::zero()` and `difficulties=[]` to force the server to build a `last_n_numbers` vector of length equal to the entire chain, then call `complete_headers` on all of them. The `GET_LAST_STATE_PROOF_LIMIT=1000` guard does not bound this path.

### Finding Description

**The guard (lines 201–205)** only checks:

```rust
self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
``` [1](#0-0) 

With `difficulties=[]` and `last_n_blocks=10` this evaluates to `0 + 20 = 20`, well under 1000. The guard passes.

**The vulnerable path** (lines 291–348) enters the `else` branch when `last_block_number - start_block_number > last_n_blocks`: [2](#0-1) 

It calls `get_first_block_total_difficulty_is_not_less_than(start_block_number, last_block_number, &difficulty_boundary)` with `difficulty_boundary=0`: [3](#0-2) 

Inside that function, the very first check at lines 30–32 short-circuits immediately because every block's `total_difficulty >= U256::zero()` is always true, returning `Some((start_block_number, ...))`: [4](#0-3) 

So `difficulty_boundary_block_number = start_block_number = 0`.

**The adjustment at lines 313–316** is then skipped because `last_block_number - 0 = N >= last_n_blocks`: [5](#0-4) 

This causes `last_n_numbers = (0..last_block_number).collect()` — a vector of **N entries** (the entire chain): [6](#0-5) 

Since `difficulty_boundary_block_number == 0`, the `if difficulty_boundary_block_number > 0` branch is skipped and `sampled_numbers = []`: [7](#0-6) 

**`complete_headers` is then called with N entries**, each performing:
- `snapshot.get_ancestor(last_hash, number)` (O(depth) traversal)
- `snapshot.get_block(...)` (disk I/O)
- `snapshot.chain_root_mmr(*number - 1).get_root()` (MMR computation) [8](#0-7) 

The constant `GET_LAST_STATE_PROOF_LIMIT = 1000` is defined but provides zero protection here: [9](#0-8) 

### Impact Explanation
A single malicious peer can send one crafted `GetLastStateProof` message and force the server to perform O(N) disk reads and MMR root computations, where N is the full chain length. On mainnet with hundreds of thousands of blocks this exhausts CPU and I/O, stalling the light-client protocol server for all peers.

### Likelihood Explanation
The attack requires no privileges, no PoW, no keys — only a valid P2P connection to the light-client protocol port. The crafted message is trivially constructable: `last_hash=tip`, `start_number=0`, `start_hash=genesis`, `last_n_blocks=10`, `difficulty_boundary=U256::zero()`, `difficulties=[]`. It passes every validation check before the unbounded loop.

### Recommendation
After computing `difficulty_boundary_block_number`, enforce that `last_n_numbers` cannot exceed `GET_LAST_STATE_PROOF_LIMIT`. Specifically, add a check immediately after line 319:

```rust
if last_n_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many last_n blocks");
}
```

Additionally, the upfront guard should account for the actual bounded range, not just the message-supplied `last_n_blocks` value.

### Proof of Concept
1. Spin up a node with a chain of N = 50,000 blocks.
2. Connect as a light-client peer and send:
   ```
   GetLastStateProof {
       last_hash:          <tip hash>,
       start_number:       0,
       start_hash:         <genesis hash>,
       last_n_blocks:      10,
       difficulty_boundary: U256::zero(),
       difficulties:       [],
   }
   ```
3. Observe that `complete_headers` is invoked with a 50,000-entry slice and that wall-clock time scales linearly with N, while a request with `last_n_blocks=10` and a valid `difficulty_boundary` completes in constant bounded time.
