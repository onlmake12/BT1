Looking at the actual code to validate each claim in the report.

Audit Report

## Title
Missing IBD State Check in `GetBlocksProcess::execute()` Allows Inbound Peers to Bypass IBD Restrictions — (File: `sync/src/synchronizer/get_blocks_process.rs`)

## Summary
The module-level documentation for the synchronizer explicitly states that a node in IBD mode must respond with `packed::InIBD` to both `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess::execute()` correctly enforces this guard, but `GetBlocksProcess::execute()` has no `is_initial_block_download()` check at all. Any inbound peer can send `GetBlocks` to a node in IBD and receive block data in response, with no `InIBD` signal ever sent back.

## Finding Description
`sync/src/synchronizer/mod.rs` lines 1–7 document the invariant:

> "When CKB node is in IBD mode, it will respond `packed::InIBD` to `GetHeaders` and `GetBlocks` requests"

`GetHeadersProcess::execute()` enforces this at lines 53–66 via `active_chain.is_initial_block_download()`, calling `self.send_in_ibd()` and returning `Status::ignored()`.

`GetBlocksProcess::execute()` (lines 33–97) performs only a message-size check (`block_hashes.len() > MAX_HEADERS_LEN`) and a `BlockStatus::BLOCK_VALID` guard per hash, then serves blocks unconditionally. There is no `is_initial_block_download()` call anywhere in the function.

The dispatch layer `Synchronizer::try_process()` (mod.rs lines 396–422) applies no top-level IBD guard before routing to `GetBlocksProcess`, confirming the check must live inside the handler itself — exactly as `GetHeadersProcess` does.

The actual per-request serving is bounded by `.take(INIT_BLOCKS_IN_TRANSIT_PER_PEER)` (line 45), which is `32` blocks (not 2000 as the report states — `MAX_HEADERS_LEN = 2000` is only the malformed-message rejection threshold). However, the attacker can repeat requests continuously with no rate limit on the sync protocol's `GetBlocks` path, and only blocks already carrying `BLOCK_VALID` status are served, meaning blocks the node has already fully verified and stored.

## Impact Explanation
During IBD, an unprivileged inbound peer can repeatedly send `GetBlocks` with 32 known block hashes per message. The node retrieves each block from RocksDB, serializes it, and spawns an async task to transmit `SendBlock` responses — all without ever sending `InIBD`. This consumes outbound bandwidth and I/O that the node needs for its own IBD download pipeline. The peer receives no signal to stop. This matches the allowed impact: **Low (501–2000 points) — any other important performance improvements for CKB**, as it degrades IBD completion speed for the targeted node. It does not crash the node or cause consensus deviation.

## Likelihood Explanation
Any peer that completes the Tentacle handshake and establishes an inbound connection can immediately send `SyncMessage::GetBlocks`. No special privilege, key, or stake is required. The attacker only needs to know valid block hashes (e.g., genesis through any early block), which are publicly available. The attack is repeatable with no cooldown enforced by the sync protocol.

## Recommendation
Add the IBD guard to `GetBlocksProcess::execute()` immediately after the size check and before `active_chain` is used, mirroring `GetHeadersProcess`:

```rust
let active_chain = self.synchronizer.shared.active_chain();

if active_chain.is_initial_block_download() {
    debug!(
        "Ignoring getblocks from peer={} because node is in IBD.",
        self.peer
    );
    // send InIBD response (reuse the same helper pattern as GetHeadersProcess)
    return Status::ignored();
}
```

## Proof of Concept
1. Start Node A with an empty or early-chain database so `is_initial_block_download()` returns `true`.
2. Attacker peer B establishes an inbound TCP connection and completes the Tentacle/Sync handshake.
3. B sends `SyncMessage::GetBlocks` with 32 hashes of blocks known to be stored on Node A (e.g., genesis+1 through genesis+32).
4. Node A's `GetBlocksProcess::execute()` passes the size check, skips any IBD check, finds each block via `active_chain.get_block()`, and sends 32 `SendBlock` responses. No `InIBD` message is sent.
5. B repeats step 3 in a tight loop. Node A's outbound bandwidth is continuously consumed serving B.
6. Contrast: sending `SyncMessage::GetHeaders` to the same node immediately returns `InIBD` and is ignored, as intended.
