### Title
Excess ETH Sent to `requestPriceUpdatesWithCallback` Is Not Refunded to the Caller but Silently Credited to the Provider — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.requestPriceUpdatesWithCallback`, the contract only enforces a minimum fee (`msg.value >= requiredFee`) but never refunds any excess. The entire `msg.value - pythFeeInWei` is stored as `req.fee` and later credited to the provider in `executeCallback`. Any overpayment by the caller is permanently lost to the provider rather than returned to the user.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` computes a required fee and checks only that the caller has paid at least that amount:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
```

It then stores the provider's portion of the fee as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

This stores the **entire** `msg.value` (minus the fixed Pyth protocol fee) as the provider fee — not just `requiredFee - pythFeeInWei`. If the caller sends any amount above `requiredFee`, the excess is silently absorbed into `req.fee`.

Later, in `executeCallback`, this stored fee is credited directly to the provider:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

There is no path by which the overpaid amount is returned to the original requester (`req.requester`).

---

### Impact Explanation

Any user or integrating contract that sends `msg.value > getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess ETH. The overpayment is credited to the provider's `accruedFeesInWei` balance and is withdrawable by the provider (or their fee manager). The original caller has no mechanism to recover it.

This is a direct fund-loss vulnerability for users: excess ETH is routed to the wrong recipient (provider) instead of being refunded to the caller.

---

### Likelihood Explanation

Medium-to-high. Users and integrating contracts routinely send slightly more than the exact required fee to guard against `InsufficientFee` reverts caused by fee changes between fee estimation and transaction execution. The `IEcho` interface comment states `"The msg.value must be equal to getFee(callbackGasLimit)"` but this is not enforced on-chain — the contract silently accepts and absorbs any overpayment. Frontend tooling that adds a small buffer (e.g., `fee * 1.1`) will consistently cause users to lose funds.

---

### Recommendation

After deducting `requiredFee` from `msg.value`, refund the excess to `msg.sender`:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();

// Refund excess
uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "Refund failed");
}

req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);
```

Alternatively, enforce strict equality (`msg.value == requiredFee`) to prevent silent overpayment.

---

### Proof of Concept

1. Provider registers with `baseFeeInWei = 1000`, `feePerFeedInWei = 100`, `feePerGasInWei = 1`. `pythFeeInWei = 500`.
2. User calls `getFee(provider, 100_000, [priceId])` → returns `500 + 1000 + 100 + 100_000 = 101_600 wei`.
3. User calls `requestPriceUpdatesWithCallback{value: 110_000}(...)` (overpays by 8,400 wei).
4. `req.fee = 110_000 - 500 = 109_500` (instead of the correct `101_100`).
5. `_state.accruedFeesInWei += 500` (Pyth gets its correct share).
6. Provider calls `executeCallback`; provider is credited `109_500 + 0 - pythFee_on_update ≈ 109_500`.
7. The 8,400 wei overpayment is permanently credited to the provider. The user has no recourse. [1](#0-0) [2](#0-1) [3](#0-2)
