### Title
Delegator Loses Unclaimed OIS Rewards After Fully Unstaking From a Publisher - (`governance/pyth_staking_sdk/src/pyth-staking-client.ts`)

### Summary
The `getAdvanceDelegationRecordInstructions` function in the Pyth staking SDK only generates `advance_delegation_record` claim instructions for publishers where the caller has **active positions** in their stake account. When a delegator fully closes all positions for a given publisher, that publisher is silently filtered out of the claim loop. Because OIS rewards are epoch-lagged (rewards for epoch N are only claimable after epoch N ends), any rewards accrued during the final active epoch are permanently inaccessible to the user.

### Finding Description

The `getAdvanceDelegationRecordInstructions` function builds the list of publishers to claim from by scanning the delegator's current positions array:

```typescript
const publishers = allPublishers
  .map((publisher) => {
    const positionsWithPublisher =
      stakeAccountPositionsData.data.positions.filter(
        ({ targetWithParameters }) =>
          targetWithParameters.integrityPool?.publisher.equals(
            publisher.pubkey,
          ),
      );

    let lowestEpoch;
    for (const position of positionsWithPublisher) {
      lowestEpoch = bigintMin(position.activationEpoch, lowestEpoch);
    }

    return { ...publisher, lowestEpoch };
  })
  .filter(({ lowestEpoch }) => lowestEpoch !== undefined);
``` [1](#0-0) 

If `positionsWithPublisher` is empty (all positions for that publisher have been closed), `lowestEpoch` remains `undefined` and the publisher is excluded from `filteredPublishers`. No `advance_delegation_record` instruction is generated for it. [2](#0-1) 

The public `advanceDelegationRecord` (which is what the staking UI calls as `claim`) then only submits instructions for the filtered set: [3](#0-2) 

The `DelegationRecord` PDA (keyed by `[publisher, stake_account_positions]`) still exists on-chain with `last_epoch < current_epoch` after positions are closed, meaning unclaimed rewards are present but the SDK never submits the instruction to collect them. [4](#0-3) 

The same filtering logic also affects `getClaimableRewards`, so the UI will not even display the pending balance to the user: [5](#0-4) 

### Impact Explanation

Any delegator who closes all their positions for a publisher (via `unstakeFromPublisher` / `unstakeAllIntegrityStaking`) without first calling `claim` will permanently lose the rewards accrued during the final active epoch. Because OIS rewards are epoch-lagged by design, there is always at least one epoch of unclaimed rewards at the moment of unstaking. The `DelegationRecord` account remains on-chain but becomes unreachable through the standard SDK/UI flow. [6](#0-5) 

### Likelihood Explanation

This is a normal user action: unstaking from a publisher is a routine operation exposed directly in the staking UI. No special conditions are required. Any delegator who unstakes without manually claiming first (which the UI does not prompt them to do) will trigger the loss. The `getClaimableRewards` function has the same blind spot, so the user receives no warning that rewards are pending. [7](#0-6) 

### Recommendation

`getAdvanceDelegationRecordInstructions` should not rely solely on active positions to determine which publishers to claim from. It should additionally query all `DelegationRecord` PDAs associated with the stake account (or accept an explicit list of publishers) and generate `advance_delegation_record` instructions for any record where `last_epoch < current_epoch`, regardless of whether active positions still exist. Alternatively, the `unstakeFromPublisher` flow should atomically call `advance_delegation_record` before closing positions, mirroring the fix applied to the veNFT merge case.

### Proof of Concept

1. Delegator stakes 1000 PYTH to publisher P at epoch 10. `DelegationRecord { last_epoch: 10 }` is created.
2. Epoch 11 ends. Rewards for epoch 11 are now claimable but `last_epoch` is still 10.
3. Delegator calls `unstakeFromPublisher(stakeAccount, P, LOCKED, 1000)`. All positions for P are removed from the positions array.
4. Delegator calls `claim(stakeAccount)` → `advanceDelegationRecord` → `getAdvanceDelegationRecordInstructions`.
5. Inside the function, `positionsWithPublisher` for P is empty → `lowestEpoch = undefined` → P is filtered out.
6. No `advance_delegation_record` instruction is submitted for P.
7. `DelegationRecord { last_epoch: 10 }` remains on-chain; epoch 11 rewards are never transferred to the delegator's custody account and are permanently stranded. [8](#0-7)

  public async getClaimableRewards(
    stakeAccountPositions: PublicKey,
    simulationPayer?: PublicKey,
  ) {
    const instructions = await this.getAdvanceDelegationRecordInstructions(
      stakeAccountPositions,
      simulationPayer,
    );

    let totalRewards = 0n;

    for (const instruction of instructions.advanceDelegationRecordInstructions) {
      const tx = new Transaction().add(instruction);
      tx.feePayer = simulationPayer ?? this.wallet.publicKey;
      // eslint-disable-next-line @typescript-eslint/no-deprecated
      const res = await this.connection.simulateTransaction(tx);
      const val = res.value.returnData?.data[0];
      if (val === undefined) {
        continue;
      }
      const buffer = Buffer.from(val, "base64").reverse();
      totalRewards += BigInt("0x" + buffer.toString("hex"));
    }

    return {
      expiry:
        instructions.lowestEpoch === undefined
          ? undefined
          : epochToDate(instructions.lowestEpoch + 53n),
      totalRewards,
    };
  }
