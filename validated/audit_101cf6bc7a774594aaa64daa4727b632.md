### Title
DAO Withdrawal `header_dep_index` Truncation Mismatch Between `DaoCalculator` and On-Chain Script Allows Invalid Transactions to Pass Fee Verification — (File: `util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator` reads the `header_dep_index` witness field as a full `u64`, while the on-chain DAO C script interprets only the **lowest byte** of that value. An attacker can exploit this discrepancy to craft a DAO withdrawal transaction that passes Rust-side fee verification (and is therefore admitted to the tx-pool and assembled into a block) but is rejected by the on-chain script, causing miners to produce invalid blocks.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` reads the `header_dep_index` from the witness as a full `u64` and uses it to index into `header_deps`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)   // full u64 cast to usize
    .and_then(|hash| header_deps.get(&hash))
    .ok_or(DaoError::InvalidOutPoint)
```

The on-chain DAO C script, however, reads only the **lowest byte** of this `u64` value when resolving the header dep index. This discrepancy is explicitly documented in the test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs`:

```rust
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
let mut header_deps = vec![dummy; 258];
header_deps[1] = deposit_block.hash();
header_deps[257] = withdraw_block.hash();
// ...
assert!(result.is_err(), "expected Err, got {result:?}");
```

The test documents one direction of the mismatch (Rust rejects, C VM accepts). The **reverse direction** — Rust accepts, C VM rejects — is equally constructible and is the dangerous path:

| `header_dep_index` | `header_deps[1]` | `header_deps[257]` | Rust result | C VM result |
|---|---|---|---|---|
| `257` (0x101) | wrong block (≠ deposited block number) | correct deposit block | **accepts** | **rejects** |

An attacker who controls the transaction witness can place the correct deposit block at index 257 and an arbitrary block at index 1. The `DaoCalculator` resolves index 257 → correct block number → fee check passes. The C VM resolves index 1 → wrong block number → script exits with error. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

1. Attacker owns a DAO cell deposited at block `N`.
2. Attacker crafts a withdrawal transaction with `header_dep_index = 257` (0x101), `header_deps[257]` = block `N` (correct), `header_deps[1]` = any canonical block with number ≠ `N`.
3. `DaoCalculator::transaction_fee` resolves index 257 → block `N` → block-number check passes → transaction admitted to tx-pool.
4. Miner assembles a block containing this transaction.
5. On-chain DAO C script resolves index 1 → wrong block → block-number mismatch → script fails → **block is invalid**.
6. The network rejects the block; the miner loses the block reward.

The attacker pays only the cost of submitting a transaction. The miner bears the full cost of a wasted block. Repeated submissions can persistently degrade mining profitability and cause chain-tip instability. [3](#0-2) 

---

### Likelihood Explanation

- The attacker only needs a live DAO cell (deposited at any block) and the ability to submit a transaction to the tx-pool — both are available to any unprivileged user.
- Constructing the malicious `header_deps` array requires no special privilege: `header_deps` is a free field in the transaction.
- The tx-pool performs no range check on `header_dep_index`; the only guard is the block-number equality check, which the attacker satisfies for the Rust path.
- No Sybil attack, hashpower, or key material is required.

---

### Recommendation

In `transaction_maximum_withdraw`, after reading `header_dep_index` as `u64`, verify it fits in a `u8` before using it as an array index, to match the on-chain script's truncation behavior:

```rust
let header_dep_index = LittleEndian::read_u64(&header_deps_index_data.unwrap());
if header_dep_index > u8::MAX as u64 {
    return Err(DaoError::InvalidDaoFormat);
}
```

Alternatively, read the field as `u8` directly, consistent with the DAO C script's interpretation. [4](#0-3)

fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
