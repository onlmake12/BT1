### Title
Unbounded Response via `difficulty_boundary=0` with Empty `difficulties` Bypasses Size Guard — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary
An unprivileged remote peer can send a `GetLastStateProof` message with `difficulties=[]` and `difficulty_boundary=U256::zero()`. The `.unwrap_or(false)` guard at line 262 silently skips boundary validation when `difficulties` is empty. `difficulty_boundary=0` then causes `get_first_block_total_difficulty_is_not_less_than` to resolve immediately to `start_block_number`, setting `difficulty_boundary_block_number=0` and making `last_n_numbers` span the entire chain. The existing size check at line 201 only limits `difficulties.len() + last_n_blocks * 2` and does not bound the actual number of headers returned.

### Finding Description
The check at lines 259–266 is intended to enforce that `difficulty_boundary` is strictly greater than all sampled difficulties. When `difficulties` is empty, `difficulties.last()` returns `None`, the `.map()` short-circuits to `None`, and `.unwrap_or(false)` returns `false` — the check is a no-op. `difficulty_boundary=0` then flows into `get_first_block_total_difficulty_is_not_less_than` where `start_total_difficulty >= U256::zero()` is trivially true for any block, returning `start_block_number` immediately. This collapses `difficulty_boundary_block_number` to `start_block_number`, and `last_n_numbers = (start_block_number..last_block_number)` covers the entire chain. The server then calls `complete_headers` for every block, computing MMR roots and fetching block data for each one.

### Impact Explanation
The server performs O(N) disk reads, O(N) MMR computations, and sends an O(N) response (N = chain length) for a single crafted P2P message. On a chain with 10,000 blocks, this returns all 10,000 verifiable headers. An attacker can repeatedly send such messages to exhaust CPU, memory, and bandwidth on the full node serving light clients.

### Likelihood Explanation
The attack requires only a valid `last_hash` on the main chain (trivially obtained via `GetLastState`) and the ability to send a `LightClientMessage` over P2P. No privileges, keys, or hashpower are required. The crafted message is structurally valid and passes all format checks.

### Recommendation
Add an explicit rejection when `difficulty_boundary == U256::zero()`, or bound the actual computed `last_n_numbers.len()` against `GET_LAST_STATE_PROOF_LIMIT`. The size check at line 201 must account for the actual response size, not just the input field sizes.
