### Title
Cross-Chain Replay of Signed Lazer Price Updates Due to Missing Chain ID in Signed Payload Hash - (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate` computes the signature hash as `keccak256(payload)` where the payload contains no chain identifier. Because the same trusted signers are registered on 20+ EVM chains at the same contract address, a signed Lazer price update intended for one chain can be replayed verbatim on any other chain, allowing an attacker to feed stale or cross-chain-mismatched prices to DeFi protocols.

---

### Finding Description

In `lazer/contracts/evm/src/PythLazer.sol`, the `verifyUpdate` function recovers the signer from:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` is parsed by `PythLazerLib.parsePayloadHeader` and contains: a 4-byte format magic, an 8-byte timestamp, a 1-byte channel, a 1-byte feed count, and then feed price data. [2](#0-1) 

**No chain ID, contract address, or any chain-specific identifier is included in the payload or in the hash.** The payload structure (`Update`) only carries `timestamp`, `channel`, and `feeds`. [3](#0-2) 

`contract_manager/src/store/contracts/EvmLazerContracts.json` confirms that the identical contract address `0xACeA761c27A909d4D3895128EBe6370FDE2dF481` is deployed on more than 20 EVM chains including `base`, `arbitrum`, `polygon`, `bsc`, `monad`, `berachain_mainnet`, `hyperevm`, `soneium`, `injective_evm`, and others. [4](#0-3) 

The same trusted signers are registered on all these chains via the deployment script `deploy_evm_lazer_contracts.ts`. [5](#0-4) 

---

### Impact Explanation

An attacker who captures a valid signed Lazer price update from chain A (e.g., Arbitrum) can submit the identical `update` bytes to `verifyUpdate` on chain B (e.g., Base). The call will succeed because:

1. `keccak256(payload)` produces the same hash on both chains.
2. `ECDSA.tryRecover` recovers the same signer address.
3. `isValidSigner(signer)` returns `true` because the same trusted signer is registered on both chains. [6](#0-5) 

A DeFi protocol on chain B that calls `verifyUpdate` and then uses the returned `payload` to update its price state will accept the replayed price. If the price on chain B has moved significantly since the captured update was signed, the attacker can use the stale price to exploit lending protocols, perpetual DEXes, or any other price-sensitive application that integrates Lazer.

---

### Likelihood Explanation

The preconditions are fully satisfied by the existing deployment:

- The same contract address is live on 20+ chains.
- The same trusted signer keys are registered on all chains (by design, since Pyth Lazer uses a single set of signers globally).
- Signed Lazer updates are publicly broadcast to all subscribers via the WebSocket API, so any subscriber can capture and replay them.

The attacker needs only a WebSocket subscription to the Lazer stream and the ability to submit a transaction on the target chain. No privileged access is required.

---

### Recommendation

Include the EVM chain ID (and optionally the contract address) in the data that is signed by the trusted signer. The simplest fix is to change the hash computation in `verifyUpdate` to:

```solidity
bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
```

This binds each signed update to a specific chain and contract instance, preventing cross-chain replay. The Lazer signing infrastructure would need to be updated to include `chainId` and `contractAddress` in the signed message accordingly.

---

### Proof of Concept

1. Subscribe to the Pyth Lazer WebSocket stream and capture a signed `evm`-format update `U` for any price feed.
2. Submit `U` to `PythLazer.verifyUpdate` on chain A (e.g., Arbitrum) — succeeds, returns `payload` and `signer`.
3. Submit the **identical** `U` to `PythLazer.verifyUpdate` on chain B (e.g., Base, `0xACeA761c27A909d4D3895128EBe6370FDE2dF481`) — also succeeds, because `keccak256(payload)` is identical, the same signer is trusted, and no chain-binding check exists.
4. A consumer contract on chain B that calls `verifyUpdate` and stores the returned price will now hold the price from chain A's update, which may be stale or divergent from chain B's true market price. [7](#0-6)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
