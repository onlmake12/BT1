Audit Report

## Title
Missing IBD State Check in `GetBlocksProcess::execute()` Allows Peers to Bypass IBD Restriction — (File: `sync/src/synchronizer/get_blocks_process.rs`)

## Summary
The module-level documentation in `sync/src/synchronizer/mod.rs` explicitly states that a node in IBD must respond with `packed::InIBD` to both `GetHeaders` and `GetBlocks` requests. `GetHeadersProcess::execute()` correctly enforces this guard via `active_chain.is_initial_block_download()`, but `GetBlocksProcess::execute()` contains no such check, allowing any inbound peer to elicit full block data from a node during IBD in violation of the documented protocol invariant.

## Finding Description
The documented invariant at `sync/src/synchronizer/mod.rs` lines 1–7 states: [1](#0-0) 

`GetHeadersProcess::execute()` enforces this at lines 53–66 by calling `active_chain.is_initial_block_download()`, invoking `self.send_in_ibd()`, and returning `Status::ignored()`: [2](#0-1) 

`GetBlocksProcess::execute()` performs only a size check, a genesis-hash check, a dedup check, and a `BLOCK_VALID` status check — there is no `is_initial_block_download()` call anywhere in the function. After those checks pass, it unconditionally retrieves and transmits blocks: [3](#0-2) 

The dispatch in `Synchronizer::try_process()` routes `GetBlocks` directly to `GetBlocksProcess` with no top-level IBD guard: [4](#0-3) 

The `contains_block_status(..., BlockStatus::BLOCK_VALID)` check at line 60 is not a substitute: during IBD the node has already validated all blocks from genesis up to its current sync point, so a large and growing range of blocks is servable. [5](#0-4) 

## Impact Explanation
Any inbound peer can repeatedly send `SyncMessage::GetBlocks` with up to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` known block hashes. The node will retrieve each block from RocksDB, serialize it, and transmit it — never sending `InIBD` as the protocol requires. This consumes outbound bandwidth and disk I/O that the node needs for its own IBD download and verification pipeline, slowing IBD completion. This maps to **Low (501–2000 points): Any other important performance improvements for CKB**, as the impact is resource consumption on a single node during IBD rather than a network-wide crash or consensus deviation.

## Likelihood Explanation
No special privilege is required. Any peer that completes the Tentacle handshake and establishes an inbound connection can immediately send `SyncMessage::GetBlocks`. Block hashes from genesis onward are publicly known. The attack path is: establish TCP connection → complete Tentacle handshake → send `SyncMessage::GetBlocks` with known block hashes → repeat. This is fully reachable by any unprivileged network participant, and the `INIT_BLOCKS_IN_TRANSIT_PER_PEER` cap does not prevent repeated requests.

## Recommendation
Add the IBD guard to `GetBlocksProcess::execute()` immediately after the size check, mirroring `GetHeadersProcess`. Acquire `active_chain`, call `active_chain.is_initial_block_download()`, invoke a `send_in_ibd` helper (analogous to the one in `GetHeadersProcess` at lines 101–115), and return `Status::ignored()`: [6](#0-5)

## Proof of Concept
1. Start a CKB node (Node A) from genesis so it enters IBD.
2. From a second host, establish an inbound TCP connection to Node A and complete the Tentacle/Sync protocol handshake.
3. Send a `SyncMessage::GetBlocks` message containing the hashes of blocks 1–16 (genesis+1 through genesis+16, all publicly known and validated early in IBD).
4. Observe that Node A responds with `SendBlock` messages for each requested block rather than `InIBD`.
5. Contrast: send `SyncMessage::GetHeaders` to Node A — it immediately returns `InIBD` and logs the ignore message, as intended.
6. Repeat step 3 in a loop from multiple connections; measure Node A's outbound bandwidth and RocksDB read I/O increasing while its own IBD download rate decreases.

                block_hashes.len(),

        let active_chain = self.synchronizer.shared.active_chain();
