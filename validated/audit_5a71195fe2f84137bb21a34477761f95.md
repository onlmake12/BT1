Audit Report

## Title
Unbounded Memory Allocation via Indexer RPC `get_cells`/`get_transactions` When `request_limit` Is Unset — (`util/indexer/src/service.rs`)

## Summary
`IndexerConfig::default()` sets `request_limit` to `None`, which `IndexerService::new` and `RichIndexerService::new` both convert to `usize::MAX`. The guard `limit > self.request_limit` is never true for any caller-supplied `u32` value on 64-bit platforms (`u32::MAX as usize = 4_294_967_295 < usize::MAX`), so the check is unconditionally bypassed. The only real bound is a 10-second `TimeoutIterator`, within which all matching RocksDB records are deserialized and heap-allocated into a single `Vec` before any timeout check occurs, enabling OOM-driven node crashes via repeated concurrent RPC calls.

## Finding Description

**Default config leaves `request_limit` as `None`:**
`IndexerConfig::default()` explicitly sets `request_limit: None` (line 65, `util/app-config/src/configs/indexer.rs`).

**Both service constructors convert `None` to `usize::MAX`:**
- `IndexerService::new`: `request_limit: config.request_limit.unwrap_or(usize::MAX)` (line 98, `util/indexer/src/service.rs`)
- `RichIndexerService::new`: identical pattern (line 51, `util/rich-indexer/src/service.rs`)

**The limit guard is trivially bypassed in `get_cells`:**
```rust
let limit = limit.value() as usize;  // limit.value() returns u32
if limit > self.request_limit {       // self.request_limit = usize::MAX
```
`u32::MAX as usize = 4_294_967_295`. The comparison `4_294_967_295 > 18_446_744_073_709_551_615` is always `false`. The guard never fires. The identical pattern exists in `get_transactions` (lines 388–397).

**All matching records are materialized into a `Vec` before the timeout check:**
```rust
let cells = iter
    ...
    .take(limit)
    .collect::<Vec<_>>();   // full heap allocation happens here
if iter.is_timed_out() {    // checked AFTER peak RSS is reached
    Err(...)
```
The `TimeoutIterator` stops yielding after 10 seconds, but by that point the `Vec` has already been incrementally heap-allocated. Returning `Err` drops the `Vec` only after peak RSS is already reached.

**Exploit flow:**
1. Attacker sends `get_cells` with `limit = 0xFFFFFFFF` against a populated indexer store.
2. Guard check passes (`u32::MAX as usize < usize::MAX`).
3. A RocksDB snapshot iterator scans all matching records for up to 10 seconds.
4. Each `IndexerCell` (output, output_data, scripts, capacity, block_number, tx_index) is deserialized and pushed into a heap `Vec`.
5. On a mainnet node with millions of UTXOs, this allocates hundreds of megabytes to multiple gigabytes per request.
6. Four concurrent such requests multiply RSS growth; the OS OOM-killer terminates the `ckb` process.

## Impact Explanation
**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

An OOM-killer invocation terminates the `ckb` process, constituting a node crash. The attack requires only a single JSON-RPC call per request and is repeatable with no cooldown. Multiple concurrent requests compound RSS growth. The node becomes unresponsive to legitimate traffic before the OOM kill, and restarts do not fix the vulnerability since the default configuration is unchanged.

## Likelihood Explanation
- The vulnerable default (`request_limit = None` → `usize::MAX`) requires no operator misconfiguration; it is the out-of-the-box state.
- Any caller with access to the RPC port (common for nodes serving dApps, wallets, or light clients) can trigger this.
- The exploit is a single JSON-RPC call with one field set to `0xffffffff`; no authentication, no special tooling, no chain state manipulation required.
- The `TimeoutIterator` mitigates a full-store scan but does not prevent large allocations on a populated store within the 10-second window.
- The same vulnerability exists in `RichIndexerService` and `AsyncRichIndexerHandle`, doubling the attack surface.

## Recommendation
1. Change the default for `request_limit` from `None`/`usize::MAX` to a safe bounded value (e.g., `500` or `1000`) in `IndexerConfig::default()`.
2. Enforce a hard cap independent of configuration in both `get_cells` and `get_transactions`:
   ```rust
   const MAX_INDEXER_LIMIT: usize = 1000;
   let effective_limit = limit.min(self.request_limit).min(MAX_INDEXER_LIMIT);
   ```
3. Apply the same fix to `RichIndexerService` and `AsyncRichIndexerHandle`, which share the same `unwrap_or(usize::MAX)` pattern.
