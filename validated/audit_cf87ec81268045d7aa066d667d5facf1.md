### Title
O(n²) Nested Loops with Storage Reads in `_clearRemovedPriceUpdates` and `_validateSubscriptionParams` Can Make `updateSubscription` Prohibitively Expensive or Unexecutable - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol` allows subscriptions with up to `MAX_PRICE_IDS_PER_SUBSCRIPTION = 255` price IDs and `MAX_READER_WHITELIST_SIZE = 255` whitelist addresses. Two internal functions called by `updateSubscription` — `_validateSubscriptionParams` and `_clearRemovedPriceUpdates` — contain O(n²) nested loops. The second nested loop in `_clearRemovedPriceUpdates` performs up to 65,025 warm storage reads, consuming ~6.5M gas alone. Combined with the O(n²) memory loops in `_validateSubscriptionParams`, a single `updateSubscription` call at maximum array sizes can cost ~7–8M gas, making it prohibitively expensive and potentially unexecutable on chains with lower block gas limits.

---

### Finding Description

`SchedulerConstants.sol` sets both caps at 255: [1](#0-0) 

`_validateSubscriptionParams` (called from both `createSubscription` and `updateSubscription`) contains two independent O(n²) nested loops over **memory** arrays — one for `priceIds` uniqueness and one for `readerWhitelist` uniqueness: [2](#0-1) 

`_clearRemovedPriceUpdates` (called from `updateSubscription`) contains two nested loops. The second nested loop iterates `newPriceIds` (memory) in the outer loop and `currentPriceIds` (storage) in the inner loop, performing up to 255 × 255 = 65,025 storage reads: [3](#0-2) 

`updateSubscription` calls both functions sequentially: [4](#0-3) 

---

### Impact Explanation

At 255 priceIds and 255 whitelist entries:

- `_validateSubscriptionParams` nested loops: ~32,385 + ~32,385 = ~64,770 memory iterations (~200K gas)
- `_clearRemovedPriceUpdates` first nested loop: 255 cold storage reads for `currentPriceIds[i]` (~535,500 gas) + 65,025 memory reads (~195K gas)
- `_clearRemovedPriceUpdates` second nested loop: 65,025 warm storage reads for `currentPriceIds[j]` at 100 gas each = **~6,502,500 gas**
- Total for `updateSubscription`: **~7–8M gas**

On chains with a 15M or lower block gas limit (e.g., some L2s or future Ethereum configurations), this transaction would revert entirely. Even on Ethereum mainnet (~30M limit), the cost is extreme. A subscription manager who creates a max-size subscription may find it **impossible to update, deactivate, or change parameters** of their own subscription — permanently locking their funds in an active subscription they cannot modify.

---

### Likelihood Explanation

Any user can call `createSubscription` with 255 priceIds and 255 whitelist addresses — both are explicitly permitted by the on-chain constants. The subscription manager then calls `updateSubscription` (e.g., to deactivate, change criteria, or modify the whitelist), triggering the expensive loops. No privileged access is required to reach this state. The entry path is: `createSubscription(255 priceIds, 255 whitelist)` → later `updateSubscription(...)` → gas exhaustion.

---

### Recommendation

1. Replace the O(n²) uniqueness checks in `_validateSubscriptionParams` with a temporary mapping or sorted-array approach (O(n log n) or O(n)).
2. Replace the O(n²) nested loops in `_clearRemovedPriceUpdates` with a mapping-based set-difference algorithm (O(n + m)).
3. Alternatively, significantly reduce `MAX_PRICE_IDS_PER_SUBSCRIPTION` and `MAX_READER_WHITELIST_SIZE` to values where the worst-case gas is well within safe limits (e.g., ≤ 50 each), and add a comment documenting the gas analysis.

---

### Proof of Concept

1. Deploy `Scheduler` on a chain with a 15M block gas limit.
2. Call `createSubscription` with `priceIds.length = 255` and `readerWhitelist.length = 255`, providing the minimum balance.
3. Call `updateSubscription` on the same subscription ID with any parameter change (e.g., toggle `updateOnHeartbeat`).
4. Observe the transaction reverts with out-of-gas, or on Ethereum mainnet, observe gas consumption of ~7–8M gas — making the call economically unviable and the subscription permanently unmodifiable. [5](#0-4) [1](#0-0)

    function _clearRemovedPriceUpdates(
        uint256 subscriptionId,
        bytes32[] storage currentPriceIds,
        bytes32[] memory newPriceIds
    ) internal returns (bool newPriceIdsAdded) {
        // Iterate through old price IDs
        for (uint i = 0; i < currentPriceIds.length; i++) {
            bytes32 oldPriceId = currentPriceIds[i];
            bool found = false;

            // Check if the old price ID exists in the new list
            for (uint j = 0; j < newPriceIds.length; j++) {
                if (newPriceIds[j] == oldPriceId) {
                    found = true;
                    break; // Found it, no need to check further
                }
            }

            // If not found in the new list, delete its stored update data
            if (!found) {
                delete _state.priceUpdates[subscriptionId][oldPriceId];
            }
        }

        // Check if any new price IDs were added
        for (uint i = 0; i < newPriceIds.length; i++) {
            bytes32 newPriceId = newPriceIds[i];
            bool found = false;

            // Check if the new price ID exists in the current list
            for (uint j = 0; j < currentPriceIds.length; j++) {
                if (currentPriceIds[j] == newPriceId) {
                    found = true;
                    break;
                }
            }

            // If a new price ID was added, mark as changed
            if (!found) {
                newPriceIdsAdded = true;
                break;
            }
        }

        return newPriceIdsAdded;
    }
