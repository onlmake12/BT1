### Title
Entropy User Fee Permanently Lost When Provider Fails to Fulfill Request - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Pyth Entropy contract credits provider and protocol fees to their respective balances at the moment a user submits a randomness request, before any service is rendered. If the provider never calls `revealWithCallback` or `reveal`, the user's fee is permanently lost with no on-chain refund or cancellation mechanism.

---

### Finding Description

In `requestHelper()`, the fee split occurs immediately upon request submission:

```solidity
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

This happens before any randomness is generated or delivered. The provider's balance is immediately spendable via `withdraw()`:

```solidity
function withdraw(uint128 amount) public override {
    ...
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [2](#0-1) 

A complete audit of the contract's public interface (`IEntropyV2`) confirms there is no `cancelRequest`, timeout-based refund, or any user-callable function to recover fees from an unfulfilled request. [3](#0-2) 

The interface documentation itself acknowledges the one-way nature of the fee: *"excess value is not refunded to the caller"* — but this extends further: even the base fee is non-refundable regardless of whether the service is ever rendered. [4](#0-3) 

---

### Impact Explanation

A user who calls `requestV2()` or `requestWithCallback()` and pays the required fee receives no randomness and no refund if the chosen provider:
- Goes offline permanently
- Runs out of committed random numbers (`OutOfRandomness`) after the fee is paid
- Deliberately withholds the reveal to grief a specific user

The provider can immediately call `withdraw()` to extract the accrued fees, leaving the user with a permanently pending request and zero recourse on-chain. The user's funds (fee paid) are permanently lost.

---

### Likelihood Explanation

Any address can register as a provider via `register()`. Users who select a non-default provider (i.e., not the Pyth-operated Fortuna service) are exposed to this risk. A malicious actor can register as a provider, attract requests by advertising a low fee, collect fees, and never fulfill. Even for the default provider, an operational failure (key loss, infrastructure outage) leaves all in-flight request fees unrecoverable by users.

---

### Recommendation

1. **Pull-payment refund with timeout**: Introduce a `cancelRequest(address provider, uint64 sequenceNumber)` function callable by the requester after a configurable timeout (e.g., N blocks). On cancellation, the fee should be refunded to the requester and deducted from the provider's `accruedFeesInWei`.
2. **Defer fee crediting**: Credit provider fees only upon successful `reveal`/`revealWithCallback`, not at request time. This mirrors the pull-payment pattern recommended in the referenced tBTC audit.
3. **Escrow model**: Hold fees in escrow per-request and release them to the provider only on fulfillment, with a timeout-based refund path for the user.

---

### Proof of Concept

1. Attacker registers as a provider: `entropy.register(1 ether, commitment, metadata, 100, uri)`
2. Victim calls `entropy.requestV2{value: 1 ether + pythFee}(attackerProvider, userRandom, 0)`
3. Inside `requestHelper`, `attackerProvider.accruedFeesInWei += 1 ether` immediately.
4. Attacker calls `entropy.withdraw(1 ether)` — funds extracted.
5. Attacker never calls `reveal` or `revealWithCallback`.
6. Victim's request remains in `CALLBACK_NOT_STARTED` state indefinitely.
7. No on-chain function exists for the victim to recover their fee. [5](#0-4) [6](#0-5)
