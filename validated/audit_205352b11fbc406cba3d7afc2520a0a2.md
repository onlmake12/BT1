### Title
Scheduler `getMinimumBalance` Omits Fixed Gas-Overhead Base, Allowing Subscriptions That Keepers Cannot Profitably Update — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getMinimumBalance` computes only a per-feed component (`numFeeds × minimumBalancePerFeed`) and explicitly omits a base component for the fixed `GAS_OVERHEAD` (30,000 gas) that `_processFeesAndPayKeeper` always charges. Any unprivileged user who creates a subscription with exactly the minimum balance will produce an active subscription whose balance is structurally insufficient to cover the actual keeper fee, so keepers will never be able to update it. The code itself acknowledges the gap with a `TODO` comment.

---

### Finding Description

`getMinimumBalance` returns:

```
minimumBalance = numPriceFeeds × minimumBalancePerFeed
``` [1](#0-0) 

The `TODO` comment on line 737 explicitly flags the missing base component:

```solidity
// TODO: Consider adding a base minimum balance independent of feed count
return uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
``` [2](#0-1) 

The actual keeper fee charged in `_processFeesAndPayKeeper` is:

```
totalKeeperFee = (startGas − gasleft() + GAS_OVERHEAD) × tx.gasprice
               + singleUpdateKeeperFeeInWei × numPriceIds
``` [3](#0-2) 

`GAS_OVERHEAD` is a fixed constant of **30,000 gas**, independent of the number of feeds: [4](#0-3) 

The structural gap is:

```
actualKeeperFee − minimumBalance
  = GAS_OVERHEAD × tx.gasprice + (singleUpdateKeeperFeeInWei − minimumBalancePerFeed) × numFeeds
  + actualGasUsed × tx.gasprice
```

When `minimumBalancePerFeed` is set equal to `singleUpdateKeeperFeeInWei` (the natural configuration), the per-feed terms cancel and the shortfall is always at least `GAS_OVERHEAD × tx.gasprice + actualGasUsed × tx.gasprice > 0` whenever `tx.gasprice > 0`.

`createSubscription` enforces only the minimum balance floor: [5](#0-4) 

There is no check that the deposited balance can cover even a single keeper update. A subscription created with exactly `getMinimumBalance()` wei will always revert at the `_processFeesAndPayKeeper` balance check: [6](#0-5) 

The test suite itself demonstrates this by having to mock `getMinimumBalance` to zero in order to test the keeper-fee revert path, confirming that the minimum balance and the keeper fee are independently configured and can be misaligned: [7](#0-6) 

---

### Impact Explanation

Any user who creates a subscription with exactly the minimum balance (the documented and expected usage pattern) produces an active subscription that keepers can never profitably update. The subscription occupies a slot in `activeSubscriptionIds`, is returned by `getActiveSubscriptions`, and causes keepers to waste gas attempting updates that always revert. Downstream consumers of the Scheduler that rely on fresh prices for those subscriptions will receive permanently stale data. The subscription manager's funds are locked until they manually deactivate and withdraw.

---

### Likelihood Explanation

The vulnerability is triggered by the normal, documented usage of `createSubscription{value: getMinimumBalance(...)}`. No special attacker capability is required — any unprivileged address can call `createSubscription`. On any chain where `tx.gasprice > 0` (all production EVM chains), the shortfall `GAS_OVERHEAD × tx.gasprice` is always positive. At 1 gwei gas price the shortfall is `30,000 × 1e9 = 3×10¹³ wei ≈ $0.075`; at 10 gwei it is `≈ $0.75`. This will occur organically whenever users follow the documented minimum-balance pattern.

---

### Recommendation

1. Add a separate admin-configurable `minimumBaseBalance` (in wei) that accounts for the fixed gas overhead, and include it in `getMinimumBalance`:

```solidity
function getMinimumBalance(uint8 numPriceFeeds)
    external view override returns (uint256)
{
    return _state.minimumBaseBalance
         + uint256(numPriceFeeds) * this.getMinimumBalancePerFeed();
}
```

2. Expose `setMinimumBaseBalance` alongside `setMinimumBalancePerFeed` in `SchedulerGovernance`.

3. Document that `minimumBaseBalance` should be set to at least `GAS_OVERHEAD × expectedMaxGasPrice + expectedUpdateGasUsed × expectedMaxGasPrice` to ensure keepers are always profitable.

---

### Proof of Concept

**Setup** (matching the test suite defaults):
- `minimumBalancePerFeed = 1e16 wei` (0.01 ETH)
- `singleUpdateKeeperFeeInWei = 1e14 wei` (0.0001 ETH)
- `tx.gasprice = 0.5 gwei` (as used in `testUpdatePriceFeedsRevertsInsufficientBalanceForKeeperFee`)
- 2 price feeds

**Steps**:
1. User calls `createSubscription{value: getMinimumBalance(2)}` → deposits `2 × 1e16 = 2e16 wei`.
2. Keeper calls `updatePriceFeeds(subscriptionId, updateData)`.
3. Inside `_processFeesAndPayKeeper`:
   - `gasCost = (actualGasUsed + 30_000) × 0.5e9`
   - At ~200k gas used: `gasCost ≈ (200_000 + 30_000) × 5e8 = 1.15e14 wei`
   - `keeperSpecificFee = 1e14 × 2 = 2e14 wei`
   - `totalKeeperFee ≈ 3.15e14 wei`
   - After Pyth fee deduction, `status.balanceInWei < totalKeeperFee` → **revert `InsufficientBalance`**
4. The subscription remains active but is permanently un-updatable at the minimum balance. [8](#0-7)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
