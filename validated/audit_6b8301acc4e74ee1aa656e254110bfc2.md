### Title
Read-After-Delete on Overflow Request Storage Reference Silences Callback Delivery - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, `executeCallback` holds a `storage` reference (`req`) to a request, calls `clearRequest` which may `delete` the entire overflow mapping entry, and then reads `req.requester` and `req.callbackGasLimit` from the now-zeroed storage slot. For any request stored in `_state.requestsOverflow`, the callback is silently dispatched to `address(0)` with 0 gas, the requester never receives their price update, and the provider is still credited the full fee.

---

### Finding Description

`executeCallback` obtains a `storage` reference on line 111:

```solidity
Request storage req = findActiveRequest(sequenceNumber);
```

`findRequest` (called by `findActiveRequest`) returns either a slot from the fixed `requests[32]` array or a slot from `requestsOverflow`:

```solidity
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];   // storage ref into overflow mapping
    }
}
```

A request lands in `requestsOverflow` whenever `allocRequest` finds the main slot occupied by another active request:

```solidity
function allocRequest(uint64 sequenceNumber) internal returns (Request storage req) {
    (, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (isActive(req)) {
        (bytes32 reqKey, ) = requestKey(req.sequenceNumber);
        _state.requestsOverflow[reqKey] = req;   // displaced to overflow
    }
}
```

With only 32 main slots (`NUM_REQUESTS = 32`, mask `0x1f`), overflow is routine under concurrent usage.

Back in `executeCallback`, after crediting the provider fee at line 161–162, the request is cleared at line 164:

```solidity
clearRequest(sequenceNumber);
```

`clearRequest` behaves differently depending on where the request lives:

```solidity
function clearRequest(uint64 sequenceNumber) internal {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    Request storage req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        req.sequenceNumber = 0;              // main slot: only sequenceNumber zeroed
    } else {
        delete _state.requestsOverflow[key]; // overflow: ENTIRE struct zeroed
    }
}
```

For overflow requests, `delete _state.requestsOverflow[key]` zeros every field of the struct. Because `req` in `executeCallback` is a `storage` reference pointing into that same mapping slot, all subsequent reads through `req` return zero.

Lines 177–179 then read the now-zeroed fields:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{   // req.requester == address(0)
        gas: req.callbackGasLimit                 // req.callbackGasLimit == 0
    }(sequenceNumber, priceFeeds)
```

The call goes to `address(0)` (no code, EVM call succeeds vacuously) with 0 gas. The `catch` branches at lines 183–200 also read `req.requester` post-deletion, so even the failure events emit `address(0)`.

The sibling contract `Entropy.sol` explicitly avoids this by copying needed fields to local/memory variables before clearing, and even carries the comment `// WARNING: DO NOT USE req BELOW HERE AS ITS CONTENTS HAS BEEN CLEARED`. `Echo.sol` has no such safeguard.

---

### Impact Explanation

For every request that was displaced into `requestsOverflow`:

- The requester's `_echoCallback` is never invoked; the price update is silently lost.
- The provider is credited `req.fee + msg.value - pythFee` (read before the delete, so correct), meaning the provider is paid for a callback they did not deliver.
- `clearRequest` permanently removes the request; there is no retry path.
- The requester's deposited fee is consumed with no service rendered.

---

### Likelihood Explanation

With `NUM_REQUESTS = 32` and a short-key mask of `0x1f`, any two concurrent requests whose sequence numbers hash to the same low-5-bit value cause one to be displaced to overflow. Under normal protocol usage with multiple simultaneous subscribers this is a routine occurrence, not an edge case. No special privilege is required to call `executeCallback`; any address may trigger it.

---

### Recommendation

Save `req.requester` and `req.callbackGasLimit` into local variables **before** calling `clearRequest`, mirroring the pattern already used in `Entropy.sol`:

```solidity
// Save fields before clearing storage
address requester       = req.requester;
uint32  callbackGasLimit = req.callbackGasLimit;

clearRequest(sequenceNumber);

// Use local copies below — do NOT read req after this point
try
    IEchoConsumer(requester)._echoCallback{gas: callbackGasLimit}(sequenceNumber, priceFeeds)
{ ... }
```

Apply the same fix to the `req.requester` reads inside the `catch` blocks (lines 189, 198).

---

### Proof of Concept

1. User A calls `requestPriceUpdatesWithCallback`; their request is assigned `sequenceNumber = S1` and lands in `requests[shortKey]`.
2. User B calls `requestPriceUpdatesWithCallback`; their request hashes to the same `shortKey`. `allocRequest` moves User A's request to `requestsOverflow[keccak256(provider, S1)]` and places User B's request in the main slot.
3. A provider (or any address) calls `executeCallback(provider, S1, updateData, priceIds)`.
4. `findActiveRequest(S1)` returns a `storage` reference into `requestsOverflow[keccak256(provider, S1)]`.
5. Provider fee is credited correctly (reads `req.fee` before the delete).
6. `clearRequest(S1)` executes `delete _state.requestsOverflow[keccak256(provider, S1)]` — the entire struct is zeroed.
7. `req.requester` is now `address(0)`; `req.callbackGasLimit` is now `0`.
8. `IEchoConsumer(address(0))._echoCallback{gas: 0}(S1, priceFeeds)` is called; the EVM call to `address(0)` succeeds vacuously (no code at address 0).
9. `emitPriceUpdate` fires, giving the appearance of success.
