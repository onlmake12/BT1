### Title
Keeper-Controlled `tx.gasprice` Allows Subscription Balance Drain in `_processFeesAndPayKeeper` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `_processFeesAndPayKeeper` function computes the keeper reimbursement using `tx.gasprice`, which is entirely set by the keeper (the transaction sender). Because `updatePriceFeeds` is permissionless, a malicious keeper can submit the transaction with an arbitrarily inflated gas price, draining a subscription's entire balance in a single update call.

---

### Finding Description

In `Scheduler.sol`, `updatePriceFeeds` is callable by any address. At the start of the call, `startGas = gasleft()` is recorded. After all work is done, `_processFeesAndPayKeeper` computes:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

`tx.gasprice` is the gas price of the transaction, which is set exclusively by the caller (the keeper). There is no cap or sanity check on `tx.gasprice` in the fee calculation. The computed `totalKeeperFee` is then deducted from `status.balanceInWei` and transferred to `msg.sender`:

```solidity
status.balanceInWei -= totalKeeperFee;
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
``` [2](#0-1) 

The subscription manager who deposited funds has no ability to constrain the gas price used when a keeper calls `updatePriceFeeds`. The only protection is `InsufficientBalance` revert if the computed fee exceeds the balance, but this means the keeper can set `tx.gasprice` to exactly drain the entire balance in one call.

The entry point is fully permissionless:

```solidity
function updatePriceFeeds(uint256 subscriptionId, bytes[] calldata updateData) external override {
    uint256 startGas = gasleft();
    ...
``` [3](#0-2) 

---

### Impact Explanation

A malicious keeper can:
1. Observe a subscription with a large balance.
2. Call `updatePriceFeeds` with a valid update (satisfying trigger conditions) but with an extremely high `tx.gasprice` (e.g., thousands of gwei).
3. The contract computes `gasCost = gasUsed * extremeGasPrice` and transfers the entire subscription balance to the keeper.

The subscription manager loses all deposited funds in a single transaction. The keeper profits far beyond their actual transaction cost. This is a direct, unprivileged theft of user funds.

---

### Likelihood Explanation

`updatePriceFeeds` is explicitly designed to be permissionless — any address can call it. The keeper only needs to supply valid update data satisfying the subscription's trigger criteria (heartbeat or deviation), which is publicly observable off-chain via Hermes. No special access, leaked keys, or governance majority is required. The attack is straightforward and profitable.

---

### Recommendation

Cap the effective gas price used in the fee calculation to a reasonable on-chain reference, such as `block.basefee + maxPriorityFeePerGas_cap`, or use `min(tx.gasprice, block.basefee * multiplier)`. Alternatively, introduce a `maxGasPriceInWei` parameter in `SubscriptionParams` that the subscription manager sets at creation time, and use `min(tx.gasprice, params.maxGasPriceInWei)` in `_processFeesAndPayKeeper`. This mirrors the fix pattern from the referenced report: errors (or costs) outside the payer's control should not be charged to them without bound.

---

### Proof of Concept

1. Subscription manager creates a subscription with 1 ETH balance, 2 price feeds, heartbeat = 60s.
2. After 60 seconds, a malicious keeper calls:
   ```solidity
   scheduler.updatePriceFeeds{gasPrice: 10000 gwei}(subscriptionId, validUpdateData);
   ```
3. Suppose `gasUsed ≈ 200,000` and `GAS_OVERHEAD = 30,000`:
   - `gasCost = (200,000 + 30,000) * 10,000 gwei = 2,300,000,000 gwei = 2.3 ETH`
4. Since `balanceInWei = 1 ETH < 2.3 ETH`, the call reverts with `InsufficientBalance`.
5. Keeper adjusts: sets `gasPrice` so that `gasCost ≈ 1 ETH - keeperSpecificFee`, draining the full balance.
6. The keeper receives nearly the entire subscription balance, far exceeding their real transaction cost of `~230,000 gas * normal_gasprice`. [4](#0-3) [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
