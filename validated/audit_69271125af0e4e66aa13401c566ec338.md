### Title
Aptos Pyth Governance Contract Missing `SET_FEE_RECIPIENT` (Action Code 5) Handler Causes Governance VAA to Always Revert - (`target_chains/aptos/contracts/sources/governance/governance.move`)

---

### Summary

The Aptos Pyth governance contract's `execute_governance_instruction` entry function does not implement a handler for the `SET_FEE_RECIPIENT` governance action (action code `5`). The Sui contract defines and dispatches this action, but the Aptos `governance_action.move` caps valid action codes at `SET_STALE_PRICE_THRESHOLD` (`4`). Any governance VAA carrying action code `5` submitted to the Aptos contract will always abort with `invalid_governance_action`, making the action permanently unexecutable on Aptos.

---

### Finding Description

**Sui** defines six governance actions in `target_chains/sui/contracts/sources/governance/governance_action.move`:

```
CONTRACT_UPGRADE        = 0
SET_GOVERNANCE_DATA_SOURCE = 1
SET_DATA_SOURCES        = 2
SET_UPDATE_FEE          = 3
SET_STALE_PRICE_THRESHOLD = 4
SET_FEE_RECIPIENT       = 5   ← present in Sui
```

The Sui `governance.move` dispatches all six, including:
```move
} else if (action == governance_action::new_set_fee_recipient()) {
    set_fee_recipient::execute(&latest_only, pyth_state, governance_instruction::destroy(instruction));
}
```

**Aptos** defines only five governance actions in `target_chains/aptos/contracts/sources/governance/governance_action.move`:

```
CONTRACT_UPGRADE        = 0
SET_GOVERNANCE_DATA_SOURCE = 1
SET_DATA_SOURCES        = 2
SET_UPDATE_FEE          = 3
SET_STALE_PRICE_THRESHOLD = 4
// SET_FEE_RECIPIENT (5) is absent
```

The `from_u8` validator enforces this hard ceiling:
```move
assert!(CONTRACT_UPGRADE <= value && value <= SET_STALE_PRICE_THRESHOLD, error::invalid_governance_action());
```

The Aptos `governance.move` dispatch has no branch for action code `5` and falls through to:
```move
} else {
    governance_instruction::destroy(instruction);
    assert!(false, error::invalid_governance_action());
}
```

**Execution flow on Aptos when action = 5:**
1. `parse_and_verify_governance_vaa` verifies the VAA and calls `state::set_last_executed_governance_sequence(sequence)`.
2. `governance_instruction::from_byte_vec` calls `governance_action::from_u8(5)`, which aborts because `5 > SET_STALE_PRICE_THRESHOLD`.
3. The Aptos Move VM rolls back all state changes (including the sequence number update) atomically.
4. The VAA is permanently unexecutable on Aptos — every submission attempt aborts at the same point.

---

### Impact Explanation

Any governance VAA with action code `5` (`SetFeeRecipient`) targeting Aptos will always fail. Because Aptos Move transactions are atomic, the governance sequence number is rolled back on abort, so the VAA cannot be "consumed" — it simply can never be executed. If Pyth governance ever needs to broadcast a `SetFeeRecipient` instruction to all chains simultaneously, Aptos will be the only chain that permanently rejects it, creating a cross-chain governance state inconsistency. The `set_fee_recipient` module in Sui is explicitly kept "for backward compatibility," meaning the protocol-level action code is still live and routable.

---

### Likelihood Explanation

The `SET_FEE_RECIPIENT` action is described in Sui as a legacy action ("not used anymore"), so the probability of governance issuing such a VAA is currently low. However, the action code is still part of the canonical Pyth governance protocol (code `5`), and the Sui contract still handles it. Any future governance operation that touches fee-recipient state on all chains would silently fail on Aptos. The entry path is permissionless: anyone can relay a properly-signed governance VAA to the Aptos contract.

---

### Recommendation

Add `SET_FEE_RECIPIENT` (value `5`) to `target_chains/aptos/contracts/sources/governance/governance_action.move` and add the corresponding dispatch branch in `target_chains/aptos/contracts/sources/governance/governance.move`, mirroring the Sui implementation. If fee-recipient state is intentionally unsupported on Aptos, the dispatch should at minimum silently accept and discard the instruction (rather than aborting) so that cross-chain governance broadcasts do not permanently fail on Aptos.
