### Title
Unauthenticated `from` Field in Hole-Punching Protocol Allows Forced NAT Traversal to Attacker-Controlled IPs — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

An unprivileged peer connected to a victim CKB node can send a spoofed `ConnectionRequest` (with `from=arbitrary_peer_id`, `listen_addrs=[attacker_ip]`) followed by a `ConnectionSync` (with `from=arbitrary_peer_id`) to force the victim to initiate a TCP connection to an attacker-controlled IP via `raw_session`. Because the `from` field is never authenticated against the actual sender's peer identity, the attacker can repeat this with many different spoofed peer IDs to fill the victim's outbound connection slots with attacker-controlled nodes, enabling an eclipse attack.

---

### Finding Description

**Root cause:** Neither `ConnectionRequestProcess` nor `ConnectionSyncProcess` verifies that the `from` field in the message corresponds to the actual peer sending the message. The `from` field is taken verbatim from the message payload.

**Step 1 — Poison `pending_delivered`:**

The attacker sends a `ConnectionRequest` with `from=fake_peer_id`, `to=victim`, `listen_addrs=[attacker_ip]`.

In `ConnectionRequestProcess::execute`, when `self_peer_id == &content.to`, `respond_delivered` is called: [1](#0-0) 

Inside `respond_delivered`, after passing the `HOLE_PUNCHING_INTERVAL` check (which is keyed only by `from_peer_id`, so any fresh spoofed ID passes), the attacker's IP is stored: [2](#0-1) 

The `listen_addrs` parsing in `RequestContent::try_from` appends `from`'s peer ID to any address that lacks one, so `attacker_ip` is stored with `fake_peer_id` appended — no check that `from` matches the actual TCP session peer: [3](#0-2) 

**Step 2 — Trigger NAT traversal to attacker IP:**

The attacker then sends a `ConnectionSync` with `from=fake_peer_id`, `to=victim`, `route=[]`.

In `ConnectionSyncProcess::execute`, when `self_peer_id == &content.to`, the code looks up `pending_delivered[content.from]` — which now contains `[attacker_ip]` — and spawns a NAT traversal task: [4](#0-3) [5](#0-4) 

The victim calls `raw_session` with the attacker's IP. After the TCP connection, the Identify protocol runs and the attacker presents their own peer ID — the victim now has a live connection to an attacker-controlled node.

**Rate limiter bypass:**

The `forward_rate_limiter` is keyed by `(from, to, item_id)`: [6](#0-5) 

Using a different spoofed `from` peer ID for each request trivially bypasses this. The per-session `rate_limiter` allows 30 req/sec: [7](#0-6) 

The attacker can send 30 `ConnectionRequest` + 30 `ConnectionSync` messages per second per session, each with a distinct spoofed `from`, triggering 30 outbound NAT traversal connections per second to attacker-controlled IPs.

---

### Impact Explanation

The victim is forced to initiate outbound TCP connections to arbitrary attacker-controlled IPs. By exhausting the victim's outbound connection slots (`max_outbound`) with attacker-controlled nodes, the attacker achieves an eclipse: the victim only receives blocks and transactions relayed by the attacker, enabling consensus message manipulation and deviation.

---

### Likelihood Explanation

Any single connected peer can execute this. No special privileges, no leaked keys, no majority hashpower. The attacker only needs one established P2P session. The rate limiter is trivially bypassed by varying the spoofed `from` peer ID. The attack is fully local-testable.

---

### Recommendation

In `respond_delivered`, verify that `from_peer_id` matches the actual peer ID of the session (`self.peer`). Specifically, look up the peer ID for `self.peer` from the peer registry and reject the message if `content.from != actual_sender_peer_id`. Apply the same check in `ConnectionSyncProcess::execute` before consuming `pending_delivered`. This ensures the `from` field cannot be spoofed by a connected peer.

---

### Proof of Concept

1. Attacker establishes a P2P session to victim.
2. Attacker sends `ConnectionRequest { from: fake_peer_id, to: victim_peer_id, listen_addrs: [attacker_ip], max_hops: 1, route: [] }`.
3. Victim calls `respond_delivered`, inserts `pending_delivered[fake_peer_id] = ([attacker_ip], now)`, sends `ConnectionRequestDelivered` back.
4. Attacker sends `ConnectionSync { from: fake_peer_id, to: victim_peer_id, route: [] }`.
5. Victim looks up `pending_delivered[fake_peer_id]`, spawns NAT traversal to `attacker_ip`, calls `raw_session(stream, attacker_ip, RawSessionInfo::inbound(...))`.
6. Attacker's node accepts the TCP connection; Identify protocol runs; victim now has a live connection to attacker.
7. Repeat steps 2–6 with different `fake_peer_id` values (up to 30/sec per session) to fill all outbound slots.
