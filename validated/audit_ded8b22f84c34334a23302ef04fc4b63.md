Audit Report

## Title
Peer Entry in `pending_compact_blocks` Not Removed on `BlockTransactions` Verification Failure — (`sync/src/relayer/block_transactions_process.rs`)

## Summary
In `BlockTransactionsProcess::execute()`, when `BlockTransactionsVerifier::verify()` or `BlockUnclesVerifier::verify()` fails via the `attempt!` macro, or when `ReconstructionResult::Error` is returned, the function exits without calling `value.remove()` on the peer's `OccupiedEntry` in `peers_map`. The stale entry persists in `pending_compact_blocks`, and `SyncState::disconnected()` never cleans it up. A subsequent compact block from the same peer for the same block hash is rejected with `CompactBlockIsAlreadyPending`, forcing fallback to full-block sync.

## Finding Description
In `block_transactions_process.rs`, the execution enters nested `Entry::Occupied` guards at lines 65–72. The `attempt!` macro (defined in `sync/src/status.rs` lines 22–29) performs a bare `return ret` on any non-OK status:

```rust
macro_rules! attempt {
    ($code:expr) => {{
        let ret = $code;
        if !ret.is_ok() {
            return ret;
        }
        ret
    }};
}
```

At lines 80–89, two `attempt!` calls can trigger this early return:

```rust
attempt!(BlockTransactionsVerifier::verify(
    compact_block, expected_transaction_indexes, &received_transactions,
));
attempt!(BlockUnclesVerifier::verify(
    compact_block, expected_uncle_indexes, &received_uncles,
));
```

Neither `value.remove()` nor `pending.remove()` is called before returning. The `OccupiedEntry` `value` is simply dropped, leaving the peer's `(expected_transaction_indexes, expected_uncle_indexes)` tuple intact in `peers_map`. The same absence of cleanup applies to `ReconstructionResult::Error` at lines 160–162, which also returns without touching the peer entry. The only path that removes the entry is `ReconstructionResult::Block` at line 122 (`pending.remove()`).

`contextual_check` in `compact_block_process.rs` at lines 284–291 then detects the stale entry and blocks re-initiation:

```rust
if pending_compact_blocks
    .get(&block_hash)
    .map(|(_, peers_map, _)| peers_map.contains_key(&peer))
    .unwrap_or(false)
{
    return StatusCode::CompactBlockIsAlreadyPending.with_context(block_hash);
}
```

`SyncState::disconnected()` at lines 1607–1616 of `sync/src/types/mod.rs` removes inflight blocks and peer state but never acquires the `pending_compact_blocks` lock, so stale entries survive peer disconnection. The epoch-based `retain` cleanup at lines 112–116 of `compact_block_process.rs` only fires inside the `ReconstructionResult::Block` arm and only removes entries from epochs older than the accepted block — it does not address stale per-peer entries within the current epoch.

The `attempt!` failure returns a 4xx status code (e.g., `BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock = 410`), which triggers `should_ban()` returning `Some(BAD_MESSAGE_BAN_TIME)` per `status.rs` lines 166–179. This means each stale entry costs the attacker a peer connection ban, requiring a new IP per entry. This increases cost relative to the claim but does not eliminate the attack for an adversary with multiple IPs (cloud providers, VPNs, botnets). The stale entries persist across reconnections and survive until epoch-based eviction fires on a successful block reconstruction.

The test `test_missing_requested_uncle_response_is_rejected` in `sync/src/relayer/tests/block_transactions_process.rs` (lines 20–73) confirms the verifier fires and returns the error status but contains no assertion that the peer entry is removed afterward, corroborating the missing cleanup.

## Impact Explanation
Stale per-peer entries accumulate in `pending_compact_blocks` across block hashes within an epoch. Affected peer-block hash pairs fall back to full-block sync, increasing bandwidth and latency for block propagation. With sufficient attacker-controlled connections (each banned after one invalid response), this degrades compact block relay for targeted block hashes across the network. This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, noting that the banning mechanism raises the per-entry cost but does not prevent a motivated attacker with multiple IPs from executing the attack at scale.

## Likelihood Explanation
Any inbound or outbound peer can trigger this without special privileges, keys, or hashpower. The attacker controls both the compact block message and the `BlockTransactions` response. The condition is reachable on mainnet from any connected peer. Each stale entry requires a new peer connection due to banning, but this is accessible via cloud IPs, VPNs, or botnets. The attack is repeatable across block hashes within an epoch before epoch-based eviction fires.

## Recommendation
In `BlockTransactionsProcess::execute()`, on every early-return path — both `attempt!` failures and `ReconstructionResult::Error` — call `value.remove()` to remove the peer's entry from `peers_map`. If `peers_map` becomes empty after removal, call `pending.remove()` to remove the entire block hash entry. Additionally, `SyncState::disconnected()` in `sync/src/types/mod.rs` should acquire the `pending_compact_blocks` lock and remove all per-peer entries for the disconnecting peer index, removing the block hash entry entirely if its `peers_map` becomes empty.

## Proof of Concept
1. Connect to a CKB node as peer P.
2. Observe or mine a block B with at least one non-prefilled transaction.
3. Send a valid `CompactBlock` message for B with one short ID → node inserts `(P, ([tx_index], []))` into `pending_compact_blocks` and sends `GetBlockTransactions`.
4. Respond with a `BlockTransactions` message where `transactions.len() != expected_transaction_indexes.len()` → `BlockTransactionsVerifier::verify` fails, `attempt!` returns `BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock` (410), peer P is banned, but the peer entry is NOT removed from `pending_compact_blocks`.
5. After ban expiry or from a new IP, reconnect and send the same `CompactBlock` again → node returns `CompactBlockIsAlreadyPending` (102), confirming the stale entry blocks re-initiation.
6. Repeat steps 3–5 for different block hashes within the same epoch (each requiring a new peer connection) to grow `pending_compact_blocks` without epoch-based eviction firing.
7. Unit test confirmation: extend `test_missing_requested_uncle_response_is_rejected` in `sync/src/relayer/tests/block_transactions_process.rs` to assert `pending_compact_blocks.get(&block_hash).unwrap().1.contains_key(&peer_index)` after the failed execute — this assertion will pass, proving the entry is not cleaned up. fn test_missing_requested_uncle_response_is_rejected() {
    let (_chain, relayer, _) = build_chain(5);
    let peer_index: PeerIndex = 100.into();
    let rt = tokio::runtime::Builder::new_current_thread()
        .enable_all()
        .build()
        .unwrap();

    let tx1 = TransactionBuilder::default().build();
    let uncle = BlockBuilder::default().build();

    let block = BlockBuilder::default()
        .transaction(tx1)
        .uncle(uncle.as_uncle())
        .build();

    let prefilled = vec![0usize].into_iter().collect();
    let compact_block = CompactBlock::build_from_block(&block, &prefilled);
    let block_hash = compact_block.header().calc_header_hash();

    {
        let mut pending_compact_blocks =
            rt.block_on(relayer.shared.state().pending_compact_blocks());
        pending_compact_blocks.insert(
            block_hash.clone(),
            (
                compact_block,
                HashMap::from_iter(vec![(peer_index, (vec![], vec![0]))]),
                ckb_systemtime::unix_time_as_millis(),
            ),
        );
    }

    let block_transactions: BlockTransactions = packed::BlockTransactions::new_builder()
        .block_hash(block_hash)
        .transactions(Vec::<packed::Transaction>::new())
        .uncles(Vec::<packed::UncleBlock>::new())
        .build();

    let mock_protocol_context = MockProtocolContext::new(SupportProtocols::RelayV3);
    let nc = Arc::new(mock_protocol_context);

    let process = BlockTransactionsProcess::new(
        block_transactions.as_reader(),
        &relayer,
        Arc::<MockProtocolContext>::clone(&nc),
        peer_index,
    );

    assert_eq!(
        rt.block_on(process.execute()),
        StatusCode::BlockUnclesLengthIsUnmatchedWithPendingCompactBlock.into(),
    );
}
