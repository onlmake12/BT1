### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed `publishTime`, Allowing Stale Individual Prices to Bypass the Age Constraint — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `getPricesNoOlderThan` (and `getEmaPricesNoOlderThan`) validates freshness against `status.priceLastUpdatedAt`, which is set to the **maximum** `publishTime` across all feeds in the subscription's last update batch. Individual feeds with much older `publishTime` values (e.g., closed-market assets) are returned without any per-feed age check, silently violating the caller's `age_seconds` constraint.

---

### Finding Description

In `_validateShouldUpdatePrices`, the subscription-level timestamp is computed as the maximum `publishTime` across all feeds: [1](#0-0) 

This maximum is then stored as `status.priceLastUpdatedAt`: [2](#0-1) 

`getPricesNoOlderThan` checks only this subscription-level max timestamp, then returns all prices unconditionally: [3](#0-2) 

The same flaw exists in `getEmaPricesNoOlderThan`: [4](#0-3) 

The design comment in `_validateShouldUpdatePrices` explicitly acknowledges that closed-market feeds will carry stale timestamps from their last trading period: [5](#0-4) 

**Concrete scenario:**

A subscription contains two feeds:
- Feed A (crypto, 24/7): `publishTime = now − 30s`
- Feed B (equity, market closed): `publishTime = now − 3 days`

`priceLastUpdatedAt` = `now − 30s` (the max).

A caller invokes `getPricesNoOlderThan(subscriptionId, [feedA, feedB], 60)`:
1. Check: `distance(now, now−30s) = 30 ≤ 60` → **passes**
2. Returns Feed A (30s old) **and** Feed B (3 days old)

The caller's `age_seconds = 60` constraint is silently violated for Feed B.

---

### Impact Explanation

Any protocol using `getPricesNoOlderThan` to enforce price freshness before executing financial logic (liquidations, collateral valuation, settlement) may receive individual prices that are arbitrarily stale. The function's name and interface imply a per-price guarantee that the implementation does not provide. This broken invariant can cause:

- Incorrect collateral valuations using stale equity/commodity prices
- Missed or erroneous liquidations
- Incorrect settlement prices in derivatives protocols

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` constant is 1 hour, meaning a closed-market feed's price can be up to 1 hour old at update time — but the feed's own `publishTime` can be days older (from the last trading session), and this is explicitly permitted by the design. [6](#0-5) 

---

### Likelihood Explanation

Multi-asset subscriptions mixing 24/7 crypto feeds with equity, commodity, or FX feeds (which have market hours) are a natural and documented use case for the Scheduler. Any such subscription triggers this condition whenever a closed-market feed is included. The `getPricesNoOlderThan` function is the recommended safe read path (vs. `getPricesUnsafe`), so integrators will rely on it for freshness guarantees.

---

### Recommendation

Replace the subscription-level timestamp check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed `publishTime` check. After fetching prices via `_getPricesInternal`, iterate over each returned `PythStructs.Price` and verify `distance(block.timestamp, price.publishTime) <= age_seconds` before returning. Revert with `StalePrice` if any individual feed fails the check.

---

### Proof of Concept

```solidity
// Subscription with one 24/7 crypto feed and one equity feed (market closed)
// Feed A publishTime = block.timestamp - 30 seconds
// Feed B publishTime = block.timestamp - 3 days (last equity close)
// priceLastUpdatedAt = block.timestamp - 30 seconds (max of the two)

// Caller expects all prices to be no older than 60 seconds:
PythStructs.Price[] memory prices =
    scheduler.getPricesNoOlderThan(subscriptionId, [feedA, feedB], 60);

// Staleness check: distance(now, now-30s) = 30 <= 60 → PASSES
// prices[0].publishTime = now - 30s  ✓ (within 60s)
// prices[1].publishTime = now - 3 days  ✗ (violates age_seconds=60, but no revert)
// Protocol proceeds with a 3-day-old equity price believing it is fresh
```
