Audit Report

## Title
Silent u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Accounting — (`util/dao/src/lib.rs`)

## Summary

`DaoCalculator::calculate_maximum_withdraw` computes a u128 intermediate value `withdraw_counted_capacity` and narrows it to u64 via a bare `as u64` cast at line 156, silently discarding the upper bits when the value exceeds `u64::MAX`. Every other u128→u64 narrowing in the same `impl` block uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When triggered, the truncated value propagates into `withdrawed_interests`, causing a `safe_sub` underflow that permanently rejects any block containing such a DAO withdrawal — a consensus deviation between the node's DAO accounting layer and the CKB-VM script layer.

## Finding Description

In `util/dao/src/lib.rs` at lines 152–156, `calculate_maximum_withdraw` performs:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

When `withdraw_counted_capacity` (u128) exceeds `u64::MAX`, the `as u64` cast wraps to the lower 64 bits — a small, incorrect value — and `safe_add(occupied_capacity)` succeeds, returning a silently wrong `Ok(Capacity)`.

Every other u128→u64 narrowing in the same file uses the checked form:

```rust
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

```rust
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
``` [3](#0-2) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [4](#0-3) 

The truncated value propagates through `transaction_maximum_withdraw` → `withdrawed_interests`, where:

```rust
maximum_withdraws
    .safe_sub(input_capacities)   // underflows: truncated < actual deposited amount
    .map_err(Into::into)
``` [5](#0-4) 

`withdrawed_interests` returns an error, which propagates through `dao_field_with_current_epoch` at line 222, causing the block to be rejected. [6](#0-5) 

The existing overflow test `check_withdraw_calculation_overflows` only exercises the `safe_add` overflow path (where `withdraw_counted_capacity as u64` + `occupied_capacity` > `u64::MAX`), not the silent truncation path where `withdraw_counted_capacity as u128` > `u64::MAX` but the truncated value + `occupied_capacity` fits in u64. [7](#0-6) 

Additionally, the public RPC `calculate_dao_maximum_withdraw` calls `calculate_maximum_withdraw` directly at lines 259 and 288, silently returning a grossly incorrect (truncated) capacity value to the caller instead of a `DaoError`. [8](#0-7) 

## Impact Explanation

When triggered, the `as u64` truncation produces a `withdraw_capacity` far smaller than the actual accrued interest. The `withdrawed_interests` subtraction underflows (`maximum_withdraws < input_capacities`), causing `dao_field_with_current_epoch` to return an error. Any block containing such a DAO withdrawal is permanently rejected by all nodes running this code, even though the on-chain DAO script (running in CKB-VM) accepts the transaction. This is a **consensus split**: the block is valid per the script layer but invalid per the node's DAO accounting layer. The affected depositor's funds become permanently unwithdrawable. This maps to the **Critical** impact class: "Vulnerabilities which could easily cause consensus deviation" and "Vulnerabilities which could easily damage CKB economy."

## Likelihood Explanation

For `withdraw_counted_capacity` (u128) to exceed `u64::MAX ≈ 1.844 × 10¹⁹`:

```
counted_capacity × withdrawing_ar / deposit_ar > 1.844 × 10¹⁹
```

With `counted_capacity` bounded by the total CKB supply (~3.36 × 10¹⁸ shannons) and initial AR = 10¹⁶, the AR ratio must exceed ~5.49×. Given `DEFAULT_SECONDARY_EPOCH_REWARD`, this requires approximately 42 years of continuous mainnet operation. Likelihood on mainnet is **very low in the near term**. However, the defect is real, is inconsistent with every other u128→u64 narrowing in the same file, and is exploitable on any chain with a higher secondary issuance rate or lower initial AR (e.g., a custom `dev.toml` chain spec). The defect requires no attacker privilege — any depositor whose DAO cell happens to be processed when the AR ratio is sufficiently large will trigger it.

## Recommendation

Replace the silent `as u64` cast with the same checked conversion used everywhere else in the file:

```rust
// Before (lines 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

Add a unit test that sets `withdrawing_ar` large enough that `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX` but `+ occupied_capacity` would not overflow, verifying that `DaoError::Overflow` is returned rather than a silently truncated `Ok` value.

## Proof of Concept

Construct a scenario with:
- `counted_capacity = 3_360_000_000_000_000_000` (~3.36 × 10¹⁸ shannons, near total CKB supply)
- `deposit_ar = 10_000_000_000_000_000` (10¹⁶, genesis AR)
- `withdrawing_ar = 60_000_000_000_000_000` (6 × 10¹⁶, AR after ~42 years)

```
withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 60_000_000_000_000_000
    / 10_000_000_000_000_000
  = 20_160_000_000_000_000_000   ← exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 20_160_000_000_000_000_000 mod 2^64
  = 1_713_255_926_290_448_384   ← small, incorrect value; safe_add succeeds
```

`withdrawed_interests` then computes:
```
maximum_withdraws = 1_713_255_926_290_448_384 + occupied_capacity  (small)
input_capacities  = 3_360_000_000_000_000_000 + occupied_capacity  (large)
safe_sub → underflow → DaoError
```

A minimal unit test can reproduce this by directly calling `calculate_maximum_withdraw` with the above parameters (encoded into deposit/withdrawing block headers' DAO fields) and asserting `DaoError::Overflow` is returned — the current code instead returns a silently truncated `Ok` value.

#[test]
fn check_withdraw_calculation_overflows() {
    let output = CellOutput::new_builder()
        .capacity(Capacity::shannons(18_446_744_073_709_550_000))
        .build();
    let tx = TransactionBuilder::default().output(output.clone()).build();
    let epoch = EpochNumberWithFraction::new(1, 100, 1000);
    let deposit_header = HeaderBuilder::default()
        .number(100)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_000_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();
    let db = RocksDB::open_in(&tmp_dir, COLUMNS);
    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();
    txn.commit().unwrap();

    let consensus = Consensus::default();
    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.calculate_maximum_withdraw(
        &output,
        Capacity::bytes(0).expect("should not overflow"),
        &deposit_block.hash(),
        &withdrawing_block.hash(),
    );
    assert!(result.is_err());
}
