### Title
Peer Store Exhaustion via Discovery Address Flooding Causes Permanent `EvictionFailed` — (`network/src/peer_store/peer_store_impl.rs`, `network/src/peer_store/types.rs`)

### Summary

An unprivileged remote peer can exhaust the peer store by injecting exactly `ADDR_COUNT_LIMIT` (16 384) globally routable addresses from ≥4 097 distinct `/16` network groups via the discovery protocol. Because every freshly-added address is unconditionally `is_connectable = true` and no network group exceeds 4 entries, `check_purge` finds zero eviction candidates and returns `Err(EvictionFailed)`. Every subsequent `add_addr` call — from discovery or identify — is rejected, permanently blocking the node from learning new peer addresses until the feeler service exhausts all fake entries.

### Finding Description

**`is_connectable` returns `true` for all freshly-added addresses.**

`AddrInfo::new` initialises `last_tried_at_ms = 0`, `last_connected_at_ms = 0`, `attempts_count = 0`. [1](#0-0) 

`is_connectable` short-circuits on `tried_in_last_minute` (false for `last_tried_at_ms = 0`), then checks `attempts_count >= ADDR_MAX_RETRIES` (false for 0), then checks `attempts_count >= ADDR_MAX_FAILURES` (false for 0), and falls through to `return true`. [2](#0-1) 

**`check_purge` finds no eviction candidates.**

Phase 1 collects addresses where `!is_connectable(now_ms)`. With all entries connectable, `candidate_peers` is empty. [3](#0-2) 

Phase 2 groups by network group (`Group::IP4([octet0, octet1])` — the first two octets of the IPv4 address) and evicts 2 from any group with `len > 4`. [4](#0-3) [5](#0-4) 

If the attacker distributes 16 384 addresses across ≥4 097 distinct `/16` groups (≤4 per group), no group exceeds the threshold. `candidate_peers` is again empty, and the function returns `Err(EvictionFailed)`. [6](#0-5) 

**The discovery protocol is the unauthenticated injection path.**

`DiscoveryAddressManager::add_new_addrs` iterates every received address and calls `peer_store.add_addr` with no per-session rate limit or total-injection cap. [7](#0-6) 

`verify_nodes_message` permits up to `MAX_ADDR_TO_SEND = 1000` items × `MAX_ADDRS = 3` addresses = 3 000 addresses per `Nodes` message. Six messages suffice to inject 16 384+ addresses. [8](#0-7) 

The `is_valid_addr` filter only rejects non-globally-routable IPs; the attacker uses fabricated but syntactically routable IPs. [9](#0-8) 

`ADDR_COUNT_LIMIT = 16384` and `check_purge` only fires when `count >= ADDR_COUNT_LIMIT`. [10](#0-9) [11](#0-10) 

### Impact Explanation

Once the store is full, every call to `add_addr` (from discovery or identify) returns `Err(EvictionFailed)`. The node cannot register any new peer address. The identify protocol logs an error and silently drops the address. [12](#0-11) 

Recovery requires the feeler service to attempt connections to all 16 384 fake addresses and accumulate `ADDR_MAX_RETRIES = 3` failures each (49 152 feeler dials at `FEELER_CONNECTION_COUNT = 10` per tick). This can take hours to days, during which the node is isolated from new peer discovery — a prerequisite for an eclipse attack. [13](#0-12) [14](#0-13) 

### Likelihood Explanation

- Requires only **one** inbound or outbound P2P connection; no privileged role, no PoW, no Sybil.
- IPv4 has 65 536 possible `/16` groups; distributing 16 384 addresses across 4 097 groups is trivial.
- The attack fits in ~6 discovery `Nodes` messages, deliverable in seconds.
- No existing per-session injection cap or total-address-per-source limit prevents it.

### Recommendation

1. **Track addresses per source session** and cap the number of new addresses accepted from a single peer (e.g., 500 total per session lifetime).
2. **Evict connectable-but-never-tried addresses** in `check_purge` phase 2 as a last resort, prioritising entries with `last_connected_at_ms == 0` and `attempts_count == 0` (i.e., never dialled).
3. **Enforce per-network-group admission limits** at insertion time in `add_addr`, rejecting new addresses from a group that already holds ≥4 entries when the store is near capacity.

### Proof of Concept

```rust
// State test (no network required)
let mut peer_store = PeerStore::default();
// Fill with 16384 addresses across 4097 /16 groups (≤4 per group)
// Groups: 1.0.x.x, 1.1.x.x, ..., 16.0.x.x  (4 addresses each)
let mut count = 0u32;
'outer: for a in 1u8..=255 {
    for b in 0u8..=255 {
        for port in 1u16..=4 {
            if count >= 16384 { break 'outer; }
            let addr: Multiaddr = format!(
                "/ip4/{}.{}.0.1/tcp/{}/p2p/{}",
                a, b, port, PeerId::random().to_base58()
            ).parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
            count += 1;
        }
    }
}
assert_eq!(peer_store.addr_manager().count(), 16384);
// Any new address now fails
let new_addr: Multiaddr = "/ip4/200.200.200.1/tcp/9999/p2p/..."
    .parse().unwrap();
let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
assert!(result.is_err()); // Err(EvictionFailed)
```

        // do not remove addr tried in last minute

            return true;
        }
        // we give up if never connect to this addr

        }
        true
    }
