### Title
Permanent Freezing of All Accumulated SUI Update Fees — No Withdrawal Path Exists from `PriceInfoObject` Fee Storage — (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`, `governance/set_fee_recipient.move`)

---

### Summary

Every SUI coin paid as a price-update fee is deposited into a `PriceInfoObject` dynamic field (`FEE_STORAGE_KEY`) and can never be withdrawn. The `fee_recipient_address` field in `State` is explicitly documented as unused, and no `withdraw` function exists anywhere in the Sui Pyth contract. All accumulated protocol fees are permanently frozen.

---

### Finding Description

`update_single_price_feed` in `pyth.move` collects a `Coin<SUI>` fee from the caller and deposits it into the `PriceInfoObject` via `price_info::deposit_fee_coins`: [1](#0-0) 

`deposit_fee_coins` stores the coin as a dynamic object field under `FEE_STORAGE_KEY`: [2](#0-1) 

`price_info.move` exposes `get_balance` to read the accumulated amount, but provides **no corresponding withdraw function**: [3](#0-2) 

The `fee_recipient_address` field in `State` is settable via the `SET_FEE_RECIPIENT` governance action: [4](#0-3) [5](#0-4) 

However, `set_fee_recipient.move` opens with an explicit admission that this field is dead: [6](#0-5) 

The complete governance action enum confirms there is no `WITHDRAW_FEE` action — only `CONTRACT_UPGRADE`, `SET_GOVERNANCE_DATA_SOURCE`, `SET_DATA_SOURCES`, `SET_UPDATE_FEE`, `SET_STALE_PRICE_THRESHOLD`, and `SET_FEE_RECIPIENT`: [7](#0-6) 

A full-text search for `withdraw`, `transfer.*fee`, and `fee.*transfer` across all production `.move` files under `target_chains/sui/contracts/sources/` returns **zero matches**.

---

### Impact Explanation

Every SUI coin paid by any caller of `update_single_price_feed` is irrecoverably locked inside the corresponding `PriceInfoObject`. There is no governance action, no public entry function, and no privileged path that can extract these coins. The `fee_recipient_address` governance mechanism is a vestigial field that was intentionally disconnected from fee distribution in a prior refactor but never replaced with a working withdrawal mechanism. The total locked amount grows monotonically with every price update across every price feed, permanently.

---

### Likelihood Explanation

This is not a probabilistic risk — it is a deterministic, already-active condition. Every mainnet price update since deployment has contributed SUI to the permanently locked pool. No special attacker action is required; the normal operation of the protocol causes the loss.

---

### Recommendation

1. Add a `withdraw_fee_coins` function to `price_info.move` (friend-gated to `pyth::pyth`) that extracts the `Coin<SUI>` from `FEE_STORAGE_KEY`.
2. Add a `WITHDRAW_FEE` governance action (or a direct privileged entry function) in `pyth.move` that calls `withdraw_fee_coins` and transfers the balance to `state::get_fee_recipient(pyth_state)`.
3. Alternatively, if fees are intentionally non-withdrawable (i.e., burned), remove `fee_recipient_address`, `set_fee_recipient`, and the `SET_FEE_RECIPIENT` governance action entirely to eliminate the misleading invariant.

---

### Proof of Concept

1. Deploy the Sui Pyth contract.
2. Call `update_single_price_feed` N times, each time paying `base_update_fee` SUI. Confirm `price_info::get_balance(price_info_object)` increases by `base_update_fee` per call.
3. Submit a `SET_FEE_RECIPIENT` governance VAA pointing to any address `R`.
4. Confirm `state::get_fee_recipient(pyth_state) == R`.
5. Exhaustively call every public entry function in the module as address `R`. Assert that `price_info::get_balance(price_info_object)` never decreases and that `R`'s SUI balance never increases from protocol fees.
6. Observe that the accumulated fees are permanently locked with no reachable code path to recover them.
