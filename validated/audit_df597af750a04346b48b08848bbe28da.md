Audit Report

## Title
Fee Rate Admission Check Uses Raw Size Instead of Transaction Weight, Enabling Cycle-Heavy Transactions to Underpay Fees - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` enforces the minimum fee rate using only the serialized byte size of a transaction, not its true weight (`max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`). Because actual cycle consumption is not known until after script verification, and no second fee check is performed post-verification, an unprivileged sender can craft a transaction with a small byte footprint but near-maximum cycle consumption, passing the admission gate while paying a fraction of the fee the transaction actually demands from block space.

## Finding Description
`check_tx_fee` (lines 28–54, `tx-pool/src/util.rs`) computes the minimum required fee as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The comment immediately above this line explicitly acknowledges the approximation:

```
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
```

The correct weight function used everywhere else in the codebase is `get_transaction_weight` (`util/types/src/core/tx_pool.rs`, lines 298–303):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

`TxEntry::fee_rate()` (`entry.rs`, lines 115–117) correctly uses this weight for post-admission sorting and eviction. The split is:

- **Admission** (`pre_check` → `check_tx_fee`): size-only denominator.
- **Post-admission** (`fee_rate()`, eviction, block assembly sorting): weight denominator.

In `_process_tx` (`process.rs`, lines 705–776), the flow is:
1. `pre_check` → `check_tx_fee` with size only (line 715/289).
2. `verify_rtx` → actual cycles now known (line 734).
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` created (line 751).
4. `submit_entry` called — **no second fee check against weight**.

The actual cycle count is available at line 751 but is never used to re-validate the fee against the true weight.

## Impact Explanation
With `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` and `max_block_cycles = 70,000,000`:

- Maximum weight from cycles alone: `70,000,000 × 0.000_170_571_4 ≈ 11,940` bytes.
- A 300-byte transaction with 70M cycles: size-based min fee = 300 shannons; weight-based min fee = 11,940 shannons.
- Underpayment ratio: ~40×.

An attacker can sustain a stream of cycle-heavy transactions in the mempool at ~40× below the intended cost, consuming the block cycle budget while paying far less than the fee rate they actually demand. Although the eviction policy correctly uses weight-based fee rate and will eventually evict these transactions when the pool fills, the attacker can continuously re-submit, forcing nodes to repeatedly perform expensive script verification at 40× below the intended admission cost. This matches the allowed impact: **"bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation
The entry path is fully open: any RPC caller can invoke `send_transaction`, and any P2P peer can relay a transaction. Writing a CKB script that loops to consume ~70M cycles is a standard capability for any script author. The `max_block_cycles` cap bounds the maximum exploitation ratio (~40×) but does not eliminate it. The discrepancy is acknowledged in a code comment, confirming the developers are aware it is an approximation. No special privilege, leaked key, or victim mistake is required.

## Recommendation
After `verify_rtx` completes and `verified.cycles` is known, perform a second fee check using the true weight before calling `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

This requires passing the pool config into `_process_tx` or reading it from `self`. Alternatively, use `declared_cycles` (when provided by the caller) as an upper bound in `check_tx_fee` itself, falling back to `max_block_cycles` when absent, so the pre-check is already conservative.

## Proof of Concept
1. Write a CKB lock script that executes a tight loop consuming ~70,000,000 cycles. Serialize the transaction; its byte size is ~300 bytes.
2. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1000 shannons/KW`.
3. `check_tx_fee` computes `min_fee = fee(300) = 300 shannons`. Set `fee = 300 shannons`. Check passes.
4. `verify_rtx` runs; `verified.cycles ≈ 70,000,000`. No second fee check is performed.
5. `TxEntry` is created with `cycles = 70,000,000`; true weight ≈ 11,940. `fee_rate()` returns ~25 shannons/KW — far below `min_fee_rate = 1000`.
6. The transaction enters the pool. Repeat with many such transactions to saturate the block cycle budget at ~40× below the intended minimum cost.
