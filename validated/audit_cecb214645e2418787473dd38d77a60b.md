### Title
`revealWithCallback` Missing Front-Run Protection Causes Provider Transaction Revert — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `revealWithCallback` function in `Entropy.sol` is callable by anyone. An attacker can front-run the Fortuna keeper's (provider's) pending `revealWithCallback` transaction by copying its parameters from the mempool and submitting the same call with higher gas. The attacker's transaction clears the request and executes the callback; the provider's original transaction then reverts with `NoSuchRequest`. This is a direct analog to the `Permit2` front-run: a one-time-use on-chain operation with no caller restriction can be consumed by an attacker, causing the legitimate submitter's transaction to revert.

---

### Finding Description

The `reveal` function explicitly guards against this attack class. Its comment at line 494–495 reads:

> "This function must be called by the same `msg.sender` that originally requested the random number. This check prevents denial-of-service attacks where another actor front-runs the requester's reveal transaction."

It enforces this with:

```solidity
if (req.requester != msg.sender) {
    revert EntropyErrors.Unauthorized();
}
``` [1](#0-0) 

`revealWithCallback`, by contrast, carries no such restriction. Its comment explicitly states:

> "Anyone can call this method to fulfill a request, but the callback will only be made to the original requester." [2](#0-1) 

The function immediately calls `findActiveRequest`, which **reverts** with `NoSuchRequest` if the request has already been cleared: [3](#0-2) 

Attack path:

1. User calls `requestV2(provider, userRandomNumber, gasLimit)`. The request is stored on-chain with `callbackStatus = CALLBACK_NOT_STARTED`.
2. The Fortuna keeper constructs and broadcasts `revealWithCallback(provider, seqNum, userContribution, providerContribution)`.
3. Attacker observes the pending transaction in the mempool, copies all four parameters, and rebroadcasts with a higher gas price.
4. Attacker's transaction lands first: `revealHelper` validates the commitment, the callback executes on `req.requester`, and `clearRequest` zeroes out the request slot.
5. The keeper's original transaction calls `findActiveRequest` — the slot is now zeroed (`sequenceNumber == 0`), so `isActive` returns `false`, and the function reverts with `NoSuchRequest`. [4](#0-3) 

In the legacy code path (`gasLimit10k == 0`), `clearRequest` is called **before** the callback: [5](#0-4) 

In the new code path (`gasLimit10k != 0`), `clearRequest` is called after a successful `excessivelySafeCall`: [6](#0-5) 

In both paths, once the attacker's transaction succeeds, the request slot is cleared and the keeper's transaction has no recoverable path — it reverts unconditionally.

---

### Impact Explanation

The Fortuna keeper loses gas on every front-run. A sustained attacker can grief the keeper by front-running every `revealWithCallback` submission, forcing repeated gas expenditure with zero reward. The user still receives their random number (the callback executes via the attacker's transaction), so there is no direct fund loss — matching the external report's medium severity classification of "transaction failure or confusion, does not necessarily result in direct fund loss."

---

### Likelihood Explanation

All EVM chains with a public mempool expose pending transactions. The attacker needs only to:
- Monitor the mempool for `revealWithCallback` calls to the Entropy contract address
- Copy the four calldata parameters
- Resubmit with a higher gas price

No special knowledge, privileged access, or off-chain secret is required. The `providerContribution` is visible in the pending transaction calldata. This is a straightforward, low-cost, repeatable attack.

---

### Recommendation

Wrap the `findActiveRequest` call (and the subsequent logic) in a guard that silently returns when the request is no longer active, instead of reverting. This mirrors the try-catch recommendation from the external report:

```solidity
function revealWithCallback(
    address provider,
    uint64 sequenceNumber,
    bytes32 userContribution,
    bytes32 providerContribution
) public override {
    EntropyStructsV2.Request storage req = findRequest(provider, sequenceNumber);

    // If the request is no longer active (already fulfilled by a front-runner),
    // silently return rather than reverting, analogous to a try-catch on Permit2.permit.
    if (!isActive(req) || req.provider != provider || req.sequenceNumber != sequenceNumber) {
        return;
    }
    // ... rest of existing logic
}
```

This prevents unintended reverts caused by a front-runner consuming the one-time-use request slot, without changing the happy-path behavior.

### Proof of Concept

```
1. User calls requestV2(provider, 0xUSER_RAND, 100_000) with sufficient fee.
   → sequenceNumber N is assigned; callbackStatus = CALLBACK_NOT_STARTED.

2. Fortuna keeper broadcasts:
   revealWithCallback(provider, N, 0xUSER_RAND, 0xPROVIDER_PROOF)

3. Attacker sees the pending tx, copies params, submits same call with +10% gas price.

4. Attacker's tx mines first:
   - findActiveRequest(provider, N) → succeeds
   - revealHelper validates commitment → OK
   - excessivelySafeCall → _entropyCallback fires on req.requester
   - clearRequest(provider, N) → slot zeroed

5. Keeper's tx mines next:
   - findActiveRequest(provider, N) → isActive returns false (sequenceNumber == 0)
   - reverts with EntropyErrors.NoSuchRequest
   - Keeper loses gas; no fee recovery.
``` [7](#0-6)
