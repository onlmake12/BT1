### Title
Snappy Varint-Driven Pre-Decompression Heap Amplification — (`network/src/compress.rs`)

### Summary

In `LengthDelimitedCodecWithCompress::decode`, the node unconditionally allocates a buffer sized by the attacker-controlled snappy varint **before** attempting decompression. An unprivileged remote peer can send a wire frame of ~200 bytes that forces an 8 MB heap allocation per message, enabling memory exhaustion across all 125 default inbound connections.

---

### Finding Description

The decode path reads the snappy uncompressed-length varint from the incoming frame and immediately allocates a zeroed buffer of that size: [1](#0-0) 

The guard at line 235 only rejects values **strictly greater than** `MAX_UNCOMPRESSED_LEN` (8 MB = `1 << 23`): [2](#0-1) 

So any value in `[1, 8_388_608]` passes the check and causes `BytesMut::zeroed(decompressed_bytes_len)` to execute at line 242 — allocating up to 8 MB — before the decompressor has validated a single byte of actual output. [3](#0-2) 

The wire-frame size is bounded by `max_frame_length` per protocol: [4](#0-3) 

Snappy achieves compression ratios of up to ~65 000:1 on repetitive data. A valid snappy stream encoding 8 MB of zero bytes compresses to roughly 200 bytes. The attacker therefore sends a **~200-byte wire frame** that forces an **8 MB allocation**, a ~40 000:1 amplification ratio. The allocation is real regardless of whether decompression ultimately succeeds or fails — the buffer is allocated first, then the decompressor is invoked.

The default peer limit is 125 inbound connections: [5](#0-4) 

---

### Impact Explanation

With 125 concurrent inbound connections each flooding messages at the maximum rate, the node sustains 125 × 8 MB = **1 GB of concurrent heap pressure** from decompression buffers alone, before any protocol-layer processing. On a node with 2–4 GB of RAM this causes OOM / node crash. The attack is amplified further because each connection can pipeline multiple frames before the previous decode completes.

---

### Likelihood Explanation

- No privileges required: any peer that completes the TCP + secio handshake can send compressed protocol messages.
- Compression is enabled by default on all `CKBProtocol` instances (`compress: true`). [6](#0-5) 
- The craft is trivial: `snap::raw::Encoder` on 8 MB of `\x00` bytes produces a valid ~200-byte payload; prepend `COMPRESS_FLAG` and a 4-byte length prefix.
- No PoW, no key material, no special timing required.

---

### Recommendation

Replace the pre-allocation pattern with a two-step approach:

1. **Validate the varint against the actual wire-frame size before allocating.** A legitimate snappy stream cannot decompress to more than roughly `max_frame_length × max_compression_ratio` bytes. Reject frames where `decompress_len > wire_payload_len * SNAPPY_MAX_RATIO` (e.g., ratio = 250).
2. **Allocate lazily / use `decompress_vec`** (which allocates only what is actually written) instead of `BytesMut::zeroed(decompressed_bytes_len)`.
3. Lower `MAX_UNCOMPRESSED_LEN` to match the largest legitimate `max_frame_length` (currently 4 MB for RelayV3), not a fixed 8 MB constant.

---

### Proof of Concept

```rust
// Craft a ~200-byte wire frame that forces an 8 MB allocation in decode()
use snap::raw::Encoder;
use tokio_util::codec::Decoder;

let payload = vec![0u8; 8_388_607];          // 8 MB - 1 of zeros
let compressed = Encoder::new().compress_vec(&payload).unwrap(); // ~200 bytes

// Build the on-wire frame: [4-byte length][COMPRESS_FLAG=0x80][compressed]
let frame_len = (compressed.len() + 1) as u32;
let mut wire = BytesMut::new();
wire.put_u32(frame_len);
wire.put_u8(0x80);                           // COMPRESS_FLAG
wire.extend_from_slice(&compressed);

// Feed into decoder — triggers BytesMut::zeroed(8_388_607) at line 242
let mut codec = LengthDelimitedCodecWithCompress::new(true, ...);
let _ = codec.decode(&mut wire);             // 8 MB allocated here

// Repeat in a loop across 125 connections → RSS grows by 8 MB per iteration
// independent of wire-byte count (~200 bytes per frame)
``` [7](#0-6)
