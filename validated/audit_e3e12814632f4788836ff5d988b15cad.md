### Title
Unconstrained `spill` Account in `upgrade_contract` Allows Any Relayer to Steal Buffer Lamports — (`target_chains/solana/programs/core-bridge/src/legacy/processor/governance/upgrade_contract.rs`)

---

### Summary

The `spill` account in the `UpgradeContract` instruction context carries only `#[account(mut)]` — no address constraint, no signer requirement, and no derivation from the VAA payload. Any unprivileged relayer who submits the `upgrade_contract` instruction can set `spill` to an attacker-controlled address and receive all excess lamports drained from the upgrade buffer by the BPF Loader Upgradeable program.

---

### Finding Description

The `UpgradeContract` Anchor accounts struct declares `spill` as a fully unconstrained writable account:

```rust
/// Spill account to collect excess lamports.
///
/// CHECK: This account receives any lamports after the result of the upgrade.
#[account(mut)]
spill: UncheckedAccount<'info>,
``` [1](#0-0) 

The `constraints()` function validates the governance VAA (correct emitter, guardian set index, chain, and buffer pubkey), but performs **no check whatsoever on `spill`**:

```rust
require_keys_eq!(
    Pubkey::from(decree.implementation()),
    ctx.accounts.buffer.key(),
    CoreBridgeError::ImplementationMismatch
);
// No check on spill
``` [2](#0-1) 

The governance VAA payload encodes only the `implementation` (buffer pubkey) — it does **not** encode a spill recipient. The `spill` key is then passed verbatim to the BPF Loader Upgradeable CPI:

```rust
invoke_signed(
    &bpf_loader_upgradeable::upgrade(
        &crate::ID,
        &ctx.accounts.buffer.key(),
        &ctx.accounts.upgrade_authority.key(),
        &ctx.accounts.spill.key(),   // ← fully attacker-controlled
    ),
    ...
)
``` [3](#0-2) 

The BPF Loader Upgradeable `Upgrade` instruction transfers all lamports from the buffer account that exceed the post-upgrade rent-exempt minimum directly to the `spill` pubkey. Because the buffer is funded by the governance/multisig authority to hold the new program bytecode, those lamports are protocol funds.

---

### Impact Explanation

An attacker who observes a valid, posted governance upgrade VAA on-chain can immediately submit `upgrade_contract` with `spill` set to their own wallet. The BPF Loader Upgradeable program will transfer `buffer_lamports − post_upgrade_buffer_rent` SOL to the attacker. For a large program (e.g., the Core Bridge itself), the buffer can hold tens of SOL in rent-exempt lamports. The claim account provides replay protection, so the first submitter wins — the legitimate relayer or governance operator is permanently deprived of those lamports.

---

### Likelihood Explanation

- Governance upgrade VAAs are public once posted on-chain; no privileged access is needed to observe them.
- The attacker only needs to be a normal transaction submitter (payer signer) — no leaked keys, no guardian compromise.
- Front-running a known pending upgrade transaction is a standard MEV technique on Solana.
- The precondition (a valid governance VAA exists) is a routine operational state during any upgrade cycle.

---

### Recommendation

Constrain `spill` to a known, trusted address. The simplest fix is to require `spill` to equal `payer` (the transaction signer), or to derive it from a governance-controlled PDA. For example:

```rust
#[account(mut, address = ctx.accounts.payer.key())]
spill: UncheckedAccount<'info>,
```

Alternatively, encode the intended spill recipient inside the governance VAA payload and enforce `require_keys_eq!(spill.key(), decree.spill(), ...)` in `constraints()`.

---

### Proof of Concept

1. Guardian set signs and posts a valid `upgrade_contract` governance VAA for the Core Bridge with a known buffer pubkey `B`.
2. Attacker monitors the chain, sees the VAA posted.
3. Attacker calls `upgrade_contract` with:
   - `vaa` = the posted VAA account
   - `buffer` = `B` (must match VAA, so this is fixed)
   - `spill` = attacker's own pubkey `A`
   - `payer` = attacker's keypair (signs the tx)
4. `constraints()` passes: VAA is valid, buffer matches, spill is unchecked.
5. `claim_vaa` creates the claim PDA (replay protection consumed).
6. `bpf_loader_upgradeable::upgrade` CPI executes; BPF Loader transfers `buffer_lamports − rent_minimum` to `A`.
7. Assert: attacker's balance increased by the buffer's excess lamports; legitimate operator receives nothing. [4](#0-3)

#[access_control(UpgradeContract::constraints(&ctx))]
fn upgrade_contract(ctx: Context<UpgradeContract>, _args: EmptyArgs) -> Result<()> {
    let vaa = VaaAccount::load(&ctx.accounts.vaa).unwrap();

    // Create the claim account to provide replay protection. Because this instruction creates this
    // account every time it is executed, this account cannot be created again with this emitter
    // address, chain and sequence combination.
    utils::vaa::claim_vaa(
        CpiContext::new(
            ctx.accounts.system_program.key(),
            utils::vaa::ClaimVaa {
                claim: ctx.accounts.claim.to_account_info(),
                payer: ctx.accounts.payer.to_account_info(),
            },
        ),
        &crate::ID,
        &vaa,
        None,
    )?;

    // Finally upgrade.
    invoke_signed(
        &bpf_loader_upgradeable::upgrade(
            &crate::ID,
            &ctx.accounts.buffer.key(),
            &ctx.accounts.upgrade_authority.key(),
            &ctx.accounts.spill.key(),
        ),
        &ctx.accounts.to_account_infos(),
        &[&[UPGRADE_SEED_PREFIX, &[ctx.bumps.upgrade_authority]]],
    )
    .map_err(Into::into)
}
