### Title
Uncapped Halving Counter in `primary_epoch_reward` Causes Shift Overflow — (File: `spec/src/consensus.rs`)

### Summary
`Consensus::primary_epoch_reward` computes a `halvings` counter by dividing `epoch_number` by `primary_epoch_reward_halving_interval` without capping it at 63 — the maximum valid right-shift amount for a `u64`. In Rust release builds, `u64 >> n` where `n >= 64` wraps the shift amount modulo 64 (hardware masking), causing the primary epoch reward to silently reset to `initial_primary_epoch_reward` instead of reaching zero. In debug builds, the node panics.

### Finding Description

`Consensus::primary_epoch_reward` at `spec/src/consensus.rs:670-673`:

```rust
pub fn primary_epoch_reward(&self, epoch_number: u64) -> Capacity {
    let halvings = epoch_number / self.primary_epoch_reward_halving_interval();
    Capacity::shannons(self.initial_primary_epoch_reward.as_u64() >> halvings)
}
```

`halvings` is a raw `u64` division result with no upper bound. When `halvings >= 64`, the expression `initial_primary_epoch_reward.as_u64() >> halvings` overflows the shift:

- **Debug builds**: Rust panics with `"attempt to shift right with overflow"`.
- **Release builds**: The hardware masks the shift amount to `halvings % 64`, so `>> 64` becomes `>> 0`, returning the full `initial_primary_epoch_reward` instead of zero.

This function is called from `primary_epoch_reward_of_next_epoch` at line 995:

```rust
fn primary_epoch_reward_of_next_epoch(&self, epoch: &EpochExt) -> Capacity {
    if !(epoch.number() + 1).is_multiple_of(self.primary_epoch_reward_halving_interval()) {
        epoch.primary_reward()
    } else {
        self.primary_epoch_reward(epoch.number() + 1)  // <-- overflow here
    }
}
```

Which is called from `next_epoch_ext` at lines 832–833 and 940–941 during every epoch-boundary block. The chain service invokes `next_epoch_ext` for every block it processes (`chain/src/verify.rs:310-316`), making this reachable via any block submission at an epoch boundary.

### Impact Explanation

**Release builds**: At epoch `64 × halving_interval`, the primary epoch reward silently resets to `initial_primary_epoch_reward` (the full genesis-era reward) instead of zero. This restarts the entire halving schedule, minting tokens that the protocol's fixed-supply model prohibits. The incorrect reward is embedded into the `EpochExt` stored on-chain and used to compute every subsequent block's cellbase output capacity, permanently corrupting the issuance schedule.

**Debug builds**: The node panics and crashes when processing the first epoch-boundary block at or after epoch `64 × halving_interval`, causing a consensus-layer DoS for all nodes running debug builds.

### Likelihood Explanation

On mainnet, `primary_epoch_reward_halving_interval = 8760` epochs. The overflow triggers at epoch `64 × 8760 = 560,640`, approximately 256 years from genesis. This makes the mainnet impact extremely remote. However:

- The `halving_interval` is a configurable chain-spec parameter. Any chain (dev, staging, or custom) with a small `halving_interval` (e.g., `halving_interval = 2` as used in `test/src/specs/consensus/insufficient_reward.rs`) reaches the overflow at epoch 128.
- The staging spec (`resource/specs/staging.toml:92`) sets `primary_epoch_reward_halving_interval = 8760`, same as mainnet.
- The code is structurally identical to the reported pattern: an uncapped counter used in an arithmetic operation that silently wraps.

### Recommendation

Cap `halvings` at `u64::BITS - 1` (i.e., 63) before performing the shift. Once `halvings >= 64`, the reward is mathematically zero:

```rust
pub fn primary_epoch_reward(&self, epoch_number: u64) -> Capacity {
    let halvings = epoch_number / self.primary_epoch_reward_halving_interval();
    if halvings >= 64 {
        return Capacity::zero();
    }
    Capacity::shannons(self.initial_primary_epoch_reward.as_u64() >> halvings)
}
```

### Proof of Concept

1. Configure a chain spec with `primary_epoch_reward_halving_interval = 1`.
2. Advance the chain to epoch 64 (the last block of epoch 63 triggers `next_epoch_ext` → `primary_epoch_reward_of_next_epoch` → `primary_epoch_reward(64)`).
3. In a **debug build**: the node panics with `"attempt to shift right with overflow"` at `spec/src/consensus.rs:672`.
4. In a **release build**: `primary_epoch_reward(64)` returns `initial_primary_epoch_reward` (e.g., `capacity_bytes!(100)`) instead of zero. The epoch-64 `EpochExt` is stored with `base_block_reward = initial_primary_epoch_reward / epoch_length`, and all miners from epoch 64 onward receive the full genesis-era reward, violating the fixed-supply invariant.
