### Title
Race Condition in `setFeeManager` Allows Outgoing Fee Manager to Drain Provider Fees Before Revocation — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `Entropy.sol` contract allows a provider to designate a fee manager via `setFeeManager`. When a provider calls `setFeeManager` to replace or revoke their current fee manager, the outgoing fee manager can observe the pending transaction in the Ethereum mempool and front-run it with a `withdrawAsFeeManager` call at a higher gas price, draining all accrued provider fees before their authorization is revoked.

---

### Finding Description

The `setFeeManager` function in `Entropy.sol` atomically replaces the fee manager for a provider:

```solidity
function setFeeManager(address manager) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) {
        revert EntropyErrors.NoSuchProvider();
    }
    address oldFeeManager = provider.feeManager;
    provider.feeManager = manager;
    ...
}
``` [1](#0-0) 

The fee manager is authorized to withdraw all accrued fees on behalf of the provider via `withdrawAsFeeManager`:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    ...
    if (providerInfo.feeManager != msg.sender) {
        revert EntropyErrors.Unauthorized();
    }
    require(providerInfo.accruedFeesInWei >= amount, "Insufficient balance");
    providerInfo.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    ...
}
``` [2](#0-1) 

There is no time-lock, two-step revocation, or atomic "revoke-and-drain" mechanism. The provider cannot atomically revoke the fee manager's access and protect the accrued fees in a single transaction.

The IEntropy interface documents this single-step replacement behavior:

> "Only one address can be the fee manager for a provider at a time — calling this function again with a new value will override the previous value." [3](#0-2) 

---

### Impact Explanation

A provider who wants to replace a malicious or compromised fee manager cannot do so without risk. The outgoing fee manager, upon seeing the `setFeeManager` transaction in the Ethereum mempool, can submit `withdrawAsFeeManager(provider, allAccruedFees)` with a higher gas price. If the withdrawal transaction is mined first, all accrued fees are drained to the outgoing fee manager's address before their authorization is revoked. The provider suffers a direct, irreversible financial loss of all accrued fees.

---

### Likelihood Explanation

This is realistic and requires no special infrastructure. Any fee manager can passively monitor the Ethereum mempool (e.g., via a standard Ethereum node or a service like Flashbots) for `setFeeManager` calls targeting their address. The attack requires only a single `withdrawAsFeeManager` transaction submitted with a higher gas price — a standard front-running technique on Ethereum. The scenario is especially likely when a provider discovers a fee manager is acting adversarially and attempts to revoke access.

---

### Recommendation

1. **Two-step revocation**: Introduce a pending fee manager change with a time-lock (e.g., 24 hours). The old fee manager's access is revoked only after the delay, giving the provider time to also withdraw fees.
2. **Atomic revoke-and-withdraw**: Add a `revokeAndWithdraw(address newManager, uint128 amount)` function that atomically changes the fee manager and withdraws fees to the provider in a single transaction.
3. **Withdrawal cap during transition**: Prevent the fee manager from withdrawing fees once a pending fee manager change has been submitted.

---

### Proof of Concept

1. Provider `P` has accrued `100 ETH` in fees inside `Entropy.sol`. Fee manager `FM_old` is currently authorized.
2. Provider `P` decides to replace `FM_old` with `FM_new` and submits `setFeeManager(FM_new)` to the mempool.
3. `FM_old` monitors the mempool, detects the pending `setFeeManager` call, and immediately submits `withdrawAsFeeManager(P, 100 ETH)` with a higher gas price.
4. `FM_old`'s withdrawal transaction is mined first. `FM_old` receives all 100 ETH.
5. Provider `P`'s `setFeeManager(FM_new)` transaction is mined next. `FM_new` is now the fee manager, but `accruedFeesInWei == 0`.
6. Provider `P` has lost all accrued fees. The revocation of `FM_old` was ineffective at protecting the funds. [1](#0-0) [2](#0-1)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer

        require(sent, "withdrawal to msg.sender failed");
