### Title
Missing Chain ID and Contract Address in `PythLazer.verifyUpdate()` Signed Digest Enables Cross-Chain Signature Replay — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` computes the signed hash as `keccak256(payload)` with no chain ID or contract address bound into the digest. Because Pyth Lazer deploys the same trusted signer set across multiple EVM chains, a valid price-update signature produced for one chain is cryptographically valid on every other chain where the same signer is registered.

---

### Finding Description

In `verifyUpdate()`, the digest used for ECDSA recovery is:

```solidity
payload = update[71:71 + payload_len];
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` encodes only price-feed data (timestamp, channel, feed IDs, prices). It contains no EVM `chainid`, no `address(this)`, and no other chain-binding context. There is no EIP-712 domain separator or equivalent. The trusted-signer registry is shared across all EVM deployments of `PythLazer` (Ethereum, Arbitrum, Optimism, etc.), so the same key that signs for chain A is registered as trusted on chain B. [2](#0-1) 

This is the direct analog of the SFA-06 finding: just as the COLLATERAL token was absent from the market-maker's digest (allowing a signature for token X to be used with token Y), the chain ID and contract address are absent from the Lazer signer's digest, allowing a signature for chain A to be used on chain B.

---

### Impact Explanation

An unprivileged Lazer relayer/updater can:

1. Capture a valid, unexpired price-update blob from chain A (e.g., Ethereum mainnet).
2. Submit the identical bytes to `verifyUpdate()` on chain B (e.g., Arbitrum) where the same trusted signer is registered.
3. The call succeeds: `keccak256(payload)` is identical on both chains, so `ECDSA.tryRecover` returns the same signer address, which passes `isValidSigner`.

Concrete harm scenarios:
- **Testnet → mainnet replay**: If a testnet deployment shares a trusted signer with mainnet (common during rollout), a testnet-generated update is accepted on mainnet.
- **Stale cross-chain injection**: An attacker monitors for a moment when chain A has a stale price (e.g., market closed, feed paused) and replays that update on chain B where the price is actively moving, causing consumer contracts on chain B to act on incorrect data.
- **Future upgrade risk**: If `PythLazer` is upgraded to a new implementation that changes payload semantics, old signatures remain valid against the new contract because the digest does not bind to `address(this)`.

---

### Likelihood Explanation

- `verifyUpdate()` is a public, payable function callable by any address — no privileged role required.
- The Pyth Lazer architecture explicitly uses the same trusted signer keys across all EVM chains (governance updates signers via the same Wormhole VAA mechanism on every chain).
- The payload format is chain-agnostic by design (timestamp + feed prices are global).
- The only natural mitigation is the timestamp in the payload: consumer contracts that enforce a tight staleness window reduce the replay window. However, `PythLazer.verifyUpdate()` itself performs **no timestamp check** — it returns the raw payload to the caller, leaving staleness enforcement entirely to downstream consumers, which may not all implement it correctly. [3](#0-2) 

---

### Recommendation

Bind the signed digest to the specific chain and contract by including `block.chainid` and `address(this)` in the hash:

```solidity
bytes32 hash = keccak256(abi.encodePacked(block.chainid, address(this), payload));
```

Alternatively, adopt EIP-712 with a proper domain separator. The off-chain Lazer signer must be updated to include these fields when producing signatures.

---

### Proof of Concept

1. On Ethereum mainnet (chain ID 1), call `verifyUpdate{value: fee}(update)` with a freshly signed Lazer blob — it succeeds and returns `(payload, signer)`.
2. Take the identical `update` bytes (unchanged).
3. On Arbitrum (chain ID 42161), call `verifyUpdate{value: fee}(update)` — the same `keccak256(payload)` hash is computed, `ECDSA.tryRecover` returns the same signer address, `isValidSigner` returns `true` (same signer is registered on Arbitrum), and the call succeeds.
4. The Arbitrum consumer contract receives price data that was signed for Ethereum, with no on-chain indication that it originated from a different chain.

Root cause: `keccak256(payload)` at line 93 of `lazer/contracts/evm/src/PythLazer.sol` — no chain ID, no contract address, no domain separator. [4](#0-3)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
