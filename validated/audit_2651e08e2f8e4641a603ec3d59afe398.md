### Title
First Governance VAA with Sequence 0 Permanently Rejected in Lazer Sui Contract - (File: lazer/contracts/sui/sources/governance.move)

### Summary

The `Governance` struct in the Pyth Lazer Sui contract initializes `seen_sequence` to `0`. The replay-protection check in `process_incoming` uses a strict less-than comparison (`self.seen_sequence < sequence`). Because Wormhole VAA sequences start at `0`, the very first governance VAA from a freshly-created emitter (sequence `0`) will always fail this check, permanently blocking that governance action.

### Finding Description

In `lazer/contracts/sui/sources/governance.move`, the `Governance` struct is initialized with `seen_sequence: 0`:

```move
public(package) fun new(chain_id: u16, address: ExternalAddress): Governance {
    Governance {
        chain_id,
        address,
        seen_sequence: 0,   // ← sentinel collides with first valid Wormhole sequence
    }
}
```

The replay-protection check in `process_incoming` is:

```move
assert!(self.seen_sequence < sequence, EOldSequenceNumber);
self.seen_sequence = sequence;
```

Wormhole emitter sequences start at `0`. When the first governance VAA arrives with `sequence = 0`, the check evaluates `0 < 0 = false` and aborts with `EOldSequenceNumber`. The VAA is consumed by Wormhole (it is verified and its payload extracted before the check), so it cannot be replayed. The governance action encoded in that VAA is permanently lost.

The full call chain is:

```
actions::update_trusted_signer(state, vaa)
  → state::unwrap_ptgm(self, &current_cap, vaa)
      → self.governance.process_incoming(vaa)   ← aborts here
```

The same path applies to `actions::upgrade`.

The test VAAs embedded in `actions.move` use sequences `2`, `3`, and `4` — deliberately avoiding sequence `0` — which masks this defect in the test suite.

### Impact Explanation

If the governance emitter is freshly created (sequence counter at `0`), the first governance VAA is permanently unprocessable. Concretely:

- The first `update_trusted_signer` call cannot succeed, leaving the Lazer contract with **zero trusted signers** and therefore unable to verify any price update.
- The first `upgrade` call cannot succeed, blocking contract upgrades.

Because the VAA is consumed by Wormhole before the abort, the sequence-0 slot is permanently skipped. The operator must send a second VAA (sequence `1`) to proceed, but the action encoded in sequence `0` is irrecoverably lost.

### Likelihood Explanation

The Pyth mainnet governance emitter has sent thousands of messages, so its sequence counter is far above `0` in production. However:

1. Any **new deployment** (new chain, testnet, staging environment) that provisions a fresh Wormhole emitter specifically for Lazer governance will hit this on the very first governance action.
2. The `init_lazer` entry point accepts an arbitrary `emitter_address`, so a fresh emitter is a realistic configuration.
3. No external preconditions are required — the bug fires automatically on the first VAA from a zero-sequence emitter.

### Recommendation

Replace the `u64` sentinel `0` with an `Option<u64>` to distinguish "no sequence seen yet" from "sequence 0 was seen":

```move
public struct Governance has copy, drop, store {
    chain_id: u16,
    address: ExternalAddress,
    seen_sequence: Option<u64>,   // None = no VAA processed yet
}

public(package) fun new(chain_id: u16, address: ExternalAddress): Governance {
    Governance { chain_id, address, seen_sequence: option::none() }
}

public(package) fun process_incoming(self: &mut Governance, vaa: VAA): vector<u8> {
    let sequence = vaa.sequence();
    let (chain_id, address, payload) = vaa.take_emitter_info_and_payload();
    assert!(self.chain_id == chain_id, EMismatchedEmitterChainID);
    assert!(self.address == address, EMismatchedAddress);
    if (self.seen_sequence.is_some()) {
        assert!(*self.seen_sequence.borrow() < sequence, EOldSequenceNumber);
    };
    self.seen_sequence = option::some(sequence);
    payload
}
```

### Proof of Concept

1. Deploy the Lazer Sui contract with a fresh Wormhole emitter (sequence counter = 0).
2. Construct a valid governance VAA from that emitter with `sequence = 0` encoding an `UpdateTrustedSigner` action.
3. Call `actions::update_trusted_signer(state, vaa)`.
4. The transaction aborts with `EOldSequenceNumber` because `process_incoming` evaluates `0 < 0 = false`.
5. The VAA is consumed; sequence `0` is permanently skipped.
6. The Lazer contract has no trusted signers and cannot verify any price update until a VAA with `sequence ≥ 1` is submitted.

public fun update_trusted_signer(state: &mut State, vaa: VAA) {
    let current_cap = state.current_cap();
    let (header, mut parser) = state.unwrap_ptgm(&current_cap, vaa);
    assert!(header.is_update_trusted_signer(), ENotUpdateTrustedSigner);

    let public_key = parser.take_bytes(secp256k1_compressed_pubkey_len());
    let expires_at = parser.take_u64_be();
    parser.destroy_empty();
    state.update_trusted_signer(&current_cap, public_key, expires_at);
}
