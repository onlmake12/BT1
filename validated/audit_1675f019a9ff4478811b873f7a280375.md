Audit Report

## Title
`check_tx_fee` Enforces Minimum Fee Rate Using Serialized Size Instead of Weight, Allowing Below-Minimum Fee Rate Transactions Into the Pool — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` gates pool admission using only the transaction's serialized byte size to compute the minimum required fee, while the rest of the system uses `get_transaction_weight(size, cycles) = max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns actual cycles in `_process_tx`, no weight-based fee-rate check is applied before `submit_entry`. An attacker can submit a transaction with small serialized size but high VM cycles that passes the size-only admission check while carrying an actual weight-based fee rate far below `min_fee_rate`, causing every receiving node to execute up to 70M cycles of script verification before relaying the transaction to all peers.

## Finding Description

**Root cause:** `check_tx_fee` at `tx-pool/src/util.rs:45` computes:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The code itself acknowledges the mismatch with an inline comment at lines 42–44:
> *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly."* [1](#0-0) 

The weight-based formula used everywhere else is defined at `util/types/src/core/tx_pool.rs:298–303`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` at line 279. [3](#0-2) 

**Exploit flow in `_process_tx` (`tx-pool/src/process.rs:705–777`):**

1. `pre_check` (line 715) calls `check_tx_fee` with `tx_size` only (line 289) — size-only gate passes.
2. `verify_rtx` (lines 724–734) executes scripts and returns actual `verified.cycles`.
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed at line 751.
4. `submit_entry` is called at line 753. [4](#0-3) 

There is **no post-execution fee-rate check** using `get_transaction_weight(tx_size, verified.cycles)` between steps 2 and 4. The entry enters the pool with cycles that make its true weight-based fee rate far below `min_fee_rate`.

`TxEntry::fee_rate` at `tx-pool/src/component/entry.rs:115–117` uses the weight formula for scoring and eviction, confirming the divergence from the admission check:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

## Impact Explanation

**Concrete bypass with defaults (`min_fee_rate = 1000 shannons/KW`, `max_tx_verify_cycles = 70,000,000`):**

| Parameter | Value |
|---|---|
| Serialized size | 200 bytes |
| Cycles | 70,000,000 |
| Weight | max(200, 70,000,000 × 0.000_170_571_4) = max(200, 11,940) = **11,940** |
| Admission min_fee (size-only) | 1000 × 200 / 1000 = **200 shannons** |
| Fee paid | 200 shannons (passes) |
| Actual fee rate | 200 × 1000 / 11,940 ≈ **~16 shannons/KW** (~60× below floor) |

Each such transaction causes every receiving node to execute 70M cycles of script verification, then relay the transaction to all peers, who repeat the same process. The attacker pays only 200 shannons per transaction while imposing O(peers × 70M cycles) CPU cost across the network.

**Impact: High (10001–15000 points) — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation

The exploit requires only an unprivileged `send_transaction` RPC call or a P2P relay message. The attacker must control a RISC-V lock script that executes a tight loop — straightforward with any compute-heavy script. No privileged access, key material, or majority hashpower is required. The attack is repeatable with many transactions, each costing only the size-based minimum fee. The comment in the source code explicitly acknowledges the theoretical mismatch, confirming the behavior is observable and not a misreading of the code.

## Recommendation

Add a post-execution fee-rate gate in `_process_tx` (`tx-pool/src/process.rs`) after `verify_rtx` returns actual cycles, before constructing `TxEntry` and calling `submit_entry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let actual_fee_rate = FeeRate::calculate(fee, weight);
if actual_fee_rate < tx_pool_config.min_fee_rate {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, ...)), snapshot));
}
```

The existing size-only check in `check_tx_fee` can remain as a cheap pre-filter but must not be the sole gate.

## Proof of Concept

1. Deploy a RISC-V lock script that executes a tight loop consuming ~70,000,000 cycles but compiles to a small binary (~200 serialized bytes in the transaction).
2. Construct a transaction spending a cell locked by that script, with `fee = 200 shannons` (matching the size-only minimum at default `min_fee_rate = 1000 shannons/KW`).
3. Submit via `send_transaction` RPC.
4. **Expected (broken):** `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200 shannons`; fee equals `min_fee`; transaction is admitted.
5. **Actual:** After script execution, `cycles ≈ 70,000,000`, `weight = 11,940`, `actual_fee_rate ≈ 16 shannons/KW` — ~60× below `min_fee_rate`.
6. The transaction is in the pool and relayed to all peers, each of which admits it via the same flawed check after executing 70M cycles.
7. Repeat with many transactions to amplify network-wide CPU cost at minimal attacker cost.
