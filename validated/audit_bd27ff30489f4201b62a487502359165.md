### Title
Unverified `from` Field in `ConnectionRequest` Enables Peer Identity Spoofing to Disrupt Hole Punching — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The hole punching protocol's `ConnectionRequest` handler trusts the attacker-supplied `from` peer ID from the message payload without verifying it against the actual sender's session. Any connected peer can forge the `from` field to be any arbitrary peer ID, poisoning the `pending_delivered` map on the target node. This blocks legitimate hole-punching attempts from the spoofed peer for up to 2 minutes per injection and causes the target to attempt NAT traversal toward the attacker's addresses.

---

### Finding Description

In `ConnectionRequestProcess::execute()`, the `from` field is deserialized directly from the wire message:

```rust
let from = PeerId::from_bytes(value.from().raw_data().to_vec()).map_err(|_| {
    StatusCode::InvalidFromPeerId.with_context("the from peer id is invalid")
})?;
```

There is no check that `content.from` matches the actual sender (`self.peer`). When the receiving node is the intended target (`self_peer_id == &content.to`), it calls `respond_delivered(content.from, ...)`:

```rust
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs).await
```

Inside `respond_delivered`, the node first checks whether a recent entry already exists for `from_peer_id`:

```rust
if let Some((_, t)) = self.protocol.pending_delivered.get(&from_peer_id) {
    let now = unix_time_as_millis();
    if now - t < HOLE_PUNCHING_INTERVAL {   // 2 minutes
        return StatusCode::Ignore ...;
    }
}
```

Then it inserts the requester's listen addresses under the spoofed peer ID:

```rust
self.protocol.pending_delivered.insert(from_peer_id, (remote_listens, now));
```

`HOLE_PUNCHING_INTERVAL` is 2 minutes and `TIMEOUT` is 5 minutes. An attacker who refreshes the poison every ~2 minutes can keep the entry alive indefinitely.

Later, in `ConnectionSyncProcess::execute()`, when the target receives a `ConnectionSync`, it looks up the stored listen addresses by `content.from`:

```rust
let listens_info = self.protocol.pending_delivered.get(&content.from)
    .map(|info| info.0.clone());
// ...
let tasks = listens.into_iter()
    .map(|listen_addr| Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
    .collect::<Vec<_>>();
```

The `ConnectionSync` message's `from` field is also unverified against the actual sender, so the attacker can trigger this path as well.

---

### Impact Explanation

An unprivileged peer (Mallory) connected to any relay or directly to the target (Bob) can:

1. Send `ConnectionRequest { from: Alice's PeerID, to: Bob's PeerID, listen_addrs: [Mallory's TCP addr] }`.
2. Bob writes `pending_delivered[Alice] = (Mallory's addr, now)`.
3. Any legitimate `ConnectionRequest` from Alice to Bob within the next 2 minutes is silently dropped (`StatusCode::Ignore`).
4. Mallory sends `ConnectionSync { from: Alice's PeerID, to: Bob's PeerID }` to Bob; Bob attempts TCP NAT traversal to Mallory's address, wasting connection resources.
5. By re-sending every ~2 minutes, Mallory permanently prevents Alice from hole-punching with Bob.

Peers behind NAT that rely on hole punching for connectivity can be selectively isolated from specific peers. The attack is cheap (one message per 2-minute window), targeted (any `(Alice, Bob)` pair), and requires no special privilege beyond being a connected P2P peer.

---

### Likelihood Explanation

Any peer that has established a P2P session with the target node can send a `ConnectionRequest` message. The `HolePunching` protocol is a standard service protocol enabled on all nodes. The attacker needs only to know the target peer ID of the victim (Alice), which is publicly observable from the P2P peer store or discovery protocol. The attack requires one message per 2-minute window and no cryptographic material.

---

### Recommendation

When the receiving node is the final target (`self_peer_id == &content.to`), verify that `content.from` matches the peer ID of the actual sender (`self.peer`). Since the message may arrive via relay nodes, the first hop's peer ID should be checked against the first entry in `content.route` (or the direct sender if `route` is empty). Alternatively, require the `ConnectionRequest` to carry a signature over `(from, to, listen_addrs, nonce)` using the `from` peer's private key, so the target can cryptographically verify the claimed identity before writing to `pending_delivered`.

---

### Proof of Concept

```
1. Mallory establishes a P2P session with Bob (any connected peer).

2. Mallory crafts:
     ConnectionRequest {
       from:         Alice's PeerID,
       to:           Bob's PeerID,
       listen_addrs: [Mallory's TCP address (no embedded peer ID)],
       max_hops:     6,
       route:        [],
     }
   and sends it to Bob over the HolePunching protocol channel.

3. Bob's ConnectionRequestProcess::execute() parses content.from = Alice's PeerID
   (no check against self.peer = Mallory's session).
   self_peer_id == content.to → calls respond_delivered(Alice's PeerID, ...).

4. respond_delivered() finds no existing entry for Alice → inserts:
     pending_delivered[Alice's PeerID] = ([Mallory's TCP addr], now)
   Bob sends ConnectionRequestDelivered back to Mallory.

5. Alice sends a legitimate ConnectionRequest { from: Alice, to: Bob }.
   Bob's respond_delivered() checks pending_delivered[Alice] → entry exists,
   now - t < HOLE_PUNCHING_INTERVAL (2 min) → returns Ignore.
   Alice's hole-punch attempt is silently dropped.

6. Mallory sends ConnectionSync { from: Alice's PeerID, to: Bob's PeerID } to Bob.
   Bob looks up pending_delivered[Alice] = Mallory's TCP addr,
   spawns try_nat_traversal() toward Mallory's address.

7. Mallory repeats step 2 every ~2 minutes to keep the entry fresh.
   Alice is permanently unable to hole-punch with Bob.
```

**Root cause**: `connection_request.rs` line 36 parses `from` from the payload; `connection_request.rs` line 146 passes it directly to `respond_delivered`; `connection_request.rs` line 237 writes it into `pending_delivered` — all without verifying `content.from == actual_sender_peer_id`.
