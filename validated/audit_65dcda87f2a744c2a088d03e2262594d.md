### Title
Content-Length-Based Size Check Bypassable via Chunked Transfer Encoding, Enabling Memory DoS - (File: `apps/mcp/src/http.ts`)

### Summary

The MCP HTTP server in `apps/mcp/src/http.ts` attempts to guard against oversized request bodies by checking the `content-length` header before processing. However, this check is ineffective because `content-length` is absent in chunked-transfer-encoded requests, causing `Number(undefined)` to evaluate to `NaN`, which is never greater than `MAX_BODY_BYTES`. The full request body is then buffered into memory by `transport.handleRequest()` with no actual size enforcement.

### Finding Description

In `apps/mcp/src/http.ts`, the `/mcp` POST handler performs the following check before delegating to the MCP transport:

```typescript
const contentLength = Number(req.headers["content-length"]);
if (contentLength > MAX_BODY_BYTES) {
    res.writeHead(413, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ error: "Payload Too Large" }));
    return;
}
// ...
await transport.handleRequest(req, res);
``` [1](#0-0) 

When an HTTP client sends a request using chunked transfer encoding (i.e., `Transfer-Encoding: chunked` without a `Content-Length` header), `req.headers["content-length"]` is `undefined`. `Number(undefined)` evaluates to `NaN`, and `NaN > MAX_BODY_BYTES` evaluates to `false` in JavaScript, so the guard is silently skipped.

The `StreamableHTTPServerTransport.handleRequest()` call then reads and buffers the entire request body into memory before processing it, with no incremental size check. [2](#0-1) 

The `MAX_BODY_BYTES` constant is defined as 1 MB, but this limit is never enforced for chunked requests.

### Impact Explanation

An unprivileged attacker can send arbitrarily large POST bodies to the `/mcp` endpoint using chunked transfer encoding. The server will buffer the entire payload into memory before any size check can reject it. Repeated or concurrent requests with multi-gigabyte bodies can exhaust the Node.js process heap, causing an out-of-memory crash or severe degradation of the MCP service. This is a memory-based Denial-of-Service condition reachable by any unauthenticated network client.

### Likelihood Explanation

The `/mcp` endpoint is publicly exposed on port 8080 with no authentication at the transport layer. Chunked transfer encoding is a standard HTTP/1.1 feature supported by all HTTP clients (`curl --data-binary`, Python `requests`, etc.). The bypass requires no special knowledge or privileges — any attacker who can reach the port can trigger it. The TODO comment in the code itself acknowledges that the K8s ingress body-size limit is not configured by default, meaning there is no compensating control at the edge. [3](#0-2) 

### Recommendation

Replace the `content-length` header check with an actual streaming body reader that accumulates bytes incrementally and aborts once the running total exceeds `MAX_BODY_BYTES`, before passing the request to `transport.handleRequest()`. For example:

```typescript
await new Promise<void>((resolve, reject) => {
  let received = 0;
  req.on("data", (chunk: Buffer) => {
    received += chunk.length;
    if (received > MAX_BODY_BYTES) {
      req.destroy();
      res.writeHead(413, { "Content-Type": "application/json" });
      res.end(JSON.stringify({ error: "Payload Too Large" }));
      reject(new Error("Payload too large"));
    }
  });
  req.on("end", resolve);
  req.on("error", reject);
});
```

Alternatively, configure `nginx.ingress.kubernetes.io/proxy-body-size` at the ingress level (as noted in the existing TODO) as a defense-in-depth measure, but this should not be the only control.

### Proof of Concept

```bash
# Send a 10 MB chunked body to the /mcp endpoint, bypassing the 1 MB content-length check
python3 -c "
import socket, time
payload = b'X' * (10 * 1024 * 1024)
chunk = hex(len(payload))[2:].encode() + b'\r\n' + payload + b'\r\n0\r\n\r\n'
s = socket.create_connection(('localhost', 8080))
s.sendall(
    b'POST /mcp HTTP/1.1\r\n'
    b'Host: localhost\r\n'
    b'Transfer-Encoding: chunked\r\n'
    b'Content-Type: application/json\r\n\r\n' + chunk
)
print(s.recv(4096))
"
```

Because no `Content-Length` header is sent, `Number(undefined)` = `NaN`, the guard at line 54 evaluates to `false`, and `transport.handleRequest()` buffers the full 10 MB body. Repeating this with many concurrent connections exhausts the process heap. [1](#0-0)
