### Title
Unbounded `unknown_tx_hashes` Queue Growth via Multi-Peer `RelayTransactionHashes` Flooding — (`sync/src/types/mod.rs`)

### Summary
`SyncState::add_ask_for_txs` inserts up to 32767 hashes per peer call **before** checking the soft limit. The post-insertion check bans only the offending peer but does not evict inserted entries. With N peers each sending 32767 unique hashes, the queue grows to N×32767 entries, far exceeding `MAX_UNKNOWN_TX_HASHES_SIZE` (50000), with no eviction path.

### Finding Description
In `sync/src/types/mod.rs`, `add_ask_for_txs` first inserts all hashes: [1](#0-0) 

Then checks the limit **after** insertion: [2](#0-1) 

The ban path (`TooManyUnknownTransactions`) fires only when the current peer's own count reaches `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`. For peer 1 (which fills the queue from 0 to 32767), the check never fires at all since 32767 < 50000. For peer 2 onward, the check fires and bans them, but their 32767 entries are already in the queue and are never removed. The `pop_ask_for_txs` function re-inserts entries after popping: [3](#0-2) 

Entries are only removed via `mark_as_known_txs`, which requires the node to actually receive the transaction. An attacker sending hashes for non-existent transactions ensures permanent retention.

The constants confirm the arithmetic: [4](#0-3) 

The entry point is `TransactionHashesProcess::execute`, which only checks the per-message count limit (≤ 32767) before calling `add_ask_for_txs`: [5](#0-4) 

### Impact Explanation
Queue size = N × 32767 entries. Each `KeyedPriorityQueue` entry holds a `Byte32` key (32 bytes) plus `UnknownTxHashPriority` (an `Instant`, a `Vec<PeerIndex>`, and a `bool`, ~50–80 bytes) plus internal HashMap/heap overhead (~100 bytes). At 125 peers (typical CKB max): ~4.1M entries × ~200 bytes ≈ **~820 MB** of heap growth. On resource-constrained nodes or VPS deployments this causes OOM. Even on well-provisioned nodes, the mutex contention on `unknown_tx_hashes` during `pop_ask_for_txs` and `add_ask_for_txs` causes severe latency degradation for all relay processing.

### Likelihood Explanation
The attacker needs multiple inbound connections with unique source IPs (or uses the inbound peer slot limit). No PoW, no keys, no privileged access required — only the ability to open relay connections and send `RelayTransactionHashes` messages with unique 32-byte hashes. The existing integration test (`test/src/specs/relay/too_many_unknown_transactions.rs`) only validates the single-peer ban path and does not assert the global queue bound across multiple peers. [6](#0-5) 

### Recommendation
Move the size check **before** insertion, or enforce a hard cap by refusing to insert when `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE`. Alternatively, evict the lowest-priority entries when the cap is reached (LRU/priority-based eviction). The per-peer counter check can remain as the ban trigger, but the global queue must be hard-bounded before any insertion occurs.

    pub fn execute(self) -> Status {
        let state = self.relayer.shared().state();
        {
            let relay_transaction_hashes = self.message;
            if relay_transaction_hashes.tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "TxHashes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    relay_transaction_hashes.tx_hashes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
        }

        let tx_hashes: Vec<_> = {
            let mut tx_filter = state.tx_filter();
            tx_filter.remove_expired();
            self.message
                .tx_hashes()
                .iter()
                .map(|x| x.to_entity())
                .filter(|tx_hash| !tx_filter.contains(tx_hash))
                .collect()
        };

        state.add_ask_for_txs(self.peer, tx_hashes)
    }
