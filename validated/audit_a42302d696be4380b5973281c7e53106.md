Audit Report

## Title
Unbounded Per-Peer MMR Proof Request Rate in `LightClientProtocol` Enables I/O Exhaustion DoS — (`util/light-client-protocol-server/src/lib.rs`)

## Summary

`LightClientProtocol` contains no per-peer rate limiter. Any peer can send an unlimited stream of `GetBlocksProof` messages, each triggering expensive O(positions × log₂ N) RocksDB reads via `mmr.gen_proof`. Because the RocksDB instance is shared with block validation and sync, a sustained flood from even a single attacker can saturate I/O and stall or crash the node.

## Finding Description

**No rate limiter in `LightClientProtocol`:** The struct holds only `pub shared: Shared` with no throttle field. [1](#0-0) 

`received()` parses the message and immediately dispatches to `try_process` with no rate check of any kind. [2](#0-1) 

**Per-message cap is not a rate limit:** `GET_BLOCKS_PROOF_LIMIT = 1000` and `GET_TRANSACTIONS_PROOF_LIMIT = 1000` bound only the number of hashes per message, not how frequently a peer may send such messages. [3](#0-2) 

`GetBlocksProofProcess::execute` rejects oversized messages but unconditionally proceeds to proof generation for any valid ≤1000-hash request. [4](#0-3) 

**Expensive work per request:** `reply_proof` calls `snapshot.chain_root_mmr(last_block.number() - 1)` and then `mmr.gen_proof(items_positions)` for every accepted request, performing O(positions × log₂ N) RocksDB reads. [5](#0-4) 

`chain_root_mmr` constructs an MMR backed by RocksDB reads proportional to chain height. [6](#0-5) 

**`BAD_MESSAGE_BAN_TIME` does not apply:** The ban is only triggered for malformed messages. A well-formed 1,000-hash `GetBlocksProof` with a valid `last_hash` returns `Status::ok()` and is never penalised. [7](#0-6) 

**LightClient is enabled by default:** `default_support_all_protocols()` includes `LightClient`, and the shipped `ckb.toml` lists it explicitly. [8](#0-7) 

**Contrast with `HolePunching`:** That protocol defines a `governor::RateLimiter` keyed by `(PeerIndex, message_type)` and checks it before any processing. No equivalent exists anywhere under `util/light-client-protocol-server/`. [9](#0-8) [10](#0-9) 

A grep for `rate_limit`, `RateLimiter`, or `throttle` across the entire `util/light-client-protocol-server/` directory returns zero matches, confirming the absence of any throttle.

## Impact Explanation

At chain height N = 10,000,000, each `GetBlocksProof` with 1,000 hashes forces ~23,000 RocksDB reads for MMR proof generation plus additional reads for block headers, uncles, and extensions. The RocksDB instance is shared with block validation and sync; saturating it stalls those subsystems. A single attacker in a tight loop, or up to 125 inbound peers simultaneously, can easily crash or stall a CKB full node. This matches the allowed **High (10001–15000 points)** impact: *"Vulnerabilities which could easily crash a CKB node."*

## Likelihood Explanation

The attack requires only a valid P2P connection and 1,000 publicly known block hashes — no PoW, no key material, no privileged role. LightClient is on by default on every standard CKB full node. The per-request cost is deterministic and the attack is trivially repeatable in a tight loop.

## Recommendation

Add a per-peer rate limiter to `LightClientProtocol` analogous to the one in `HolePunching`: a `governor::RateLimiter` keyed by `(PeerIndex, message_type)` checked at the top of `received()` before any dispatch. Additionally, consider reducing `GET_BLOCKS_PROOF_LIMIT` / `GET_TRANSACTIONS_PROOF_LIMIT` or introducing a per-request work budget tied to chain height.

## Proof of Concept

1. Connect to a standard CKB full node (LightClient enabled by default).
2. Collect 1,000 valid block hashes from the public chain (all publicly available).
3. In a tight loop, send `GetBlocksProof { last_hash: <tip_hash>, block_hashes: [1000 valid hashes] }`.
4. Observe RocksDB I/O saturation and increased latency / stall for block validation on the target node.

The per-request cost is deterministic and locally reproducible without mainnet access by running a local node with a simulated chain of sufficient height.

        &mut self,
        nc: Arc<dyn CKBProtocolContext + Sync>,
        peer: PeerIndex,
        data: Bytes,
    ) {
        trace!("LightClient.received peer={}", peer);

        let msg = match packed::LightClientMessageReader::from_slice(&data) {
            Ok(msg) => msg.to_enum(),
            _ => {
                warn!(
                    "LightClient.received a malformed message from Peer({})",
                    peer
                );
                nc.ban_peer(
                    peer,
                    constant::BAD_MESSAGE_BAN_TIME,
                    String::from("send us a malformed message"),
                );
                return;
            }
        };

        let item_name = msg.item_name();
        let status = self.try_process(&nc, peer, msg).await;
        if let Some(ban_time) = status.should_ban() {
            error!(
                "process {} from {}; ban {:?} since result is {}",
                item_name, peer, ban_time, status
            );
            nc.ban_peer(peer, ban_time, status.to_string());
        } else if status.should_warn() {
            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
        }
    }
