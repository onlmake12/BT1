### Title
Missing Deduplication in `GetTransactionsProofProcess::execute` Allows Per-Request DB Lookup Amplification — (`util/light-client-protocol-server/src/components/get_transactions_proof.rs`)

### Summary

`GetTransactionsProofProcess::execute` accepts up to `GET_TRANSACTIONS_PROOF_LIMIT` (1000) tx hashes with no deduplication check. Sending 1000 identical hashes causes 2000 redundant DB lookups (1000× `get_transaction_info` + 1000× `get_transaction_with_info`) for a single unique transaction. The analogous handler `GetBlocksProofProcess::execute` explicitly guards against this with a `HashSet` check and returns a ban-worthy `MalformedProtocolMessage`. `GetTransactionsProofProcess` has no such guard and returns `OK`, so no ban is issued.

### Finding Description

In `get_transactions_proof.rs`, the `execute` method:

1. Checks `tx_hashes.len() > 1000` → rejects. Exactly 1000 identical hashes pass this check. [1](#0-0) 

2. Iterates all 1000 hashes (including duplicates) in a `partition`, calling `snapshot.get_transaction_info(tx_hash)` for each: [2](#0-1) 

3. Iterates the `found` vec (1000 entries of the same hash) calling `snapshot.get_transaction_with_info(&tx_hash)` for each: [3](#0-2) 

By contrast, `GetBlocksProofProcess::execute` explicitly deduplicates using a `HashSet` and returns `MalformedProtocolMessage` (a ban-triggering 4xx code) on duplicates: [4](#0-3) 

`Status::should_ban` only bans on 4xx codes. Since `GetTransactionsProofProcess` returns `OK` (200) for a duplicate-filled request, no ban is issued: [5](#0-4) 

The limit constant confirms 1000 is the maximum allowed: [6](#0-5) 

### Impact Explanation

A single malicious light-client peer can send repeated `GetTransactionsProof` messages each containing 1000 copies of a known tx hash. Each message triggers 2000 DB lookups (vs. the expected 2 for 1 unique tx), a 1000× amplification factor. Because the response is `OK`, no ban is applied and the peer can repeat indefinitely. This is a CPU/IO amplification DoS against the light-client protocol server.

### Likelihood Explanation

The entrypoint is the P2P light-client protocol, reachable by any unprivileged peer. No PoW, key, or privileged role is required. The asymmetry with `GetBlocksProofProcess` (which has the fix) strongly suggests this is an unintentional omission.

### Recommendation

Add a `HashSet` deduplication check in `GetTransactionsProofProcess::execute` immediately after the length check, mirroring the pattern in `GetBlocksProofProcess::execute`:

```rust
let mut uniq = HashSet::new();
if !tx_hashes_vec.iter().all(|h| uniq.insert(h)) {
    return StatusCode::MalformedProtocolMessage
        .with_context("duplicate tx hash exists");
}
```

This returns a 4xx status, which `should_ban` will convert into a peer ban.

### Proof of Concept

Send a `GetTransactionsProof` P2P message with `tx_hashes = [known_tx_hash; 1000]` and a valid `last_hash`. Instrument `snapshot.get_transaction_info` and `snapshot.get_transaction_with_info` with call counters. Assert the counters reach 1000 each rather than 1 each. Confirm the peer is not banned after the response.
