### Title
Fortuna Entropy Provider Defaults `confirmed_block_status` to `BlockStatus::Latest`, Enabling Re-org-Based Randomness Manipulation - (File: apps/fortuna/src/chain/reader.rs)

### Summary
The Fortuna Entropy provider's `confirmed_block_status` configuration field defaults to `BlockStatus::Latest` via Rust's `#[default]` derive and `#[serde(default)]`. `Latest` is the least-safe block status — it is not re-org-resistant. Any operator who omits this field from their config (or relies on the default) will have the provider reveal random numbers based on unfinalized blocks, breaking the Entropy protocol's security guarantee that the provider must wait for "sufficient block confirmations" before revealing.

### Finding Description

The `BlockStatus` enum in `apps/fortuna/src/chain/reader.rs` marks `Latest` as the Rust default:

```rust
pub enum BlockStatus {
    #[default]
    Latest,       // ← unsafe default
    Finalized,
    Safe,
}
``` [1](#0-0) 

The `EthereumConfig` struct in `apps/fortuna/src/config.rs` uses `#[serde(default)]` for `confirmed_block_status`, meaning any config file that omits this key silently inherits `BlockStatus::Latest`:

```rust
#[serde(default)]
pub confirmed_block_status: BlockStatus,
``` [2](#0-1) 

This value is then propagated directly into `BlockchainState` at runtime:

```rust
confirmed_block_status: chain_config.confirmed_block_status,
``` [3](#0-2) 

The keeper's `get_latest_safe_block` function calls `get_block_number(chain_state.confirmed_block_status)` — but with the default, this fetches the **latest** (unfinalized) block, not a safe or finalized one:

```rust
chain_state.contract.get_block_number(chain_state.confirmed_block_status).await
``` [4](#0-3) 

The revelation endpoint enforces the delay check against this same unfinalized block number:

```rust
if current_block_number.saturating_sub(state.reveal_delay_blocks) < block_number {
    return Err(RestError::PendingConfirmation);
}
``` [5](#0-4) 

The same unsafe default exists in the `argus` app's `BlockStatus` and `EthereumConfig`: [6](#0-5) [7](#0-6) 

### Impact Explanation

The Entropy protocol's security guarantee explicitly requires the provider to wait for "sufficient block confirmations" before revealing `x_i`:

> "After sufficient block confirmations, the provider submits a transaction to the contract revealing their contribution." [8](#0-7) 

With `confirmed_block_status = Latest`:

1. A user submits a randomness request in block N (latest, unfinalized).
2. The provider's keeper immediately counts block N as "confirmed" and reveals `x_i` after only `reveal_delay_blocks` latest-block increments.
3. If block N is re-orged away, the user's request disappears from the canonical chain.
4. The user can re-submit with a **different** `x_U` (user contribution) in the re-orged chain, while the provider's `x_i` for that sequence slot is already known (it was revealed on-chain in the orphaned fork or observable in the mempool).
5. Since `r = hash(x_i, x_U)`, the user can now choose `x_U` to bias the output toward a favorable result.

This breaks the core security property: randomness is only guaranteed if the user cannot know `x_i` before committing `x_U`.

### Likelihood Explanation

- **Default is unsafe**: Any operator who does not explicitly set `confirmed_block_status: Finalized` or `confirmed_block_status: Safe` in their YAML config silently runs with `Latest`. There is no validation in `Config::load` that rejects or warns about this.
- **No minimum enforcement**: Unlike other fields (e.g., `replica_id`, `backup_delay_seconds`), there is no startup check that enforces a safe block status.
- **Chains with short finality windows** (e.g., Polygon, BSC, Avalanche) are particularly susceptible to shallow re-orgs, making exploitation realistic without a 51% attack.
- The Entropy user is an **unprivileged external actor** who can freely submit requests and observe the provider's reveal transactions.

### Recommendation

1. Change the `#[default]` for `BlockStatus` from `Latest` to `Finalized` in `apps/fortuna/src/chain/reader.rs` and `apps/argus/src/adapters/ethereum.rs`.
2. Add a startup validation in `Config::load` (`apps/fortuna/src/config.rs`) that warns or errors if `confirmed_block_status == BlockStatus::Latest` on chains known to have re-org risk.
3. Document the security implications of each `BlockStatus` variant clearly in the config schema.

### Proof of Concept

1. Deploy Fortuna with a config that omits `confirmed_block_status` (or relies on the default). The service starts with `confirmed_block_status = Latest`.
2. User A submits a randomness request on a chain with occasional re-orgs (e.g., Polygon). The request lands in block N.
3. After `reveal_delay_blocks` latest blocks, the keeper reveals `x_i` for sequence number `s`. The reveal transaction is broadcast and visible in the mempool.
4. User A observes `x_i` from the pending reveal transaction.
5. User A (or a colluding validator) triggers a shallow re-org that orphans block N.
6. User A re-submits their request with a crafted `x_U'` such that `hash(x_i, x_U')` produces a favorable outcome (e.g., wins a lottery).
7. The provider's keeper re-reveals `x_i` for the same sequence slot in the new canonical chain, and the user receives the biased random number. [9](#0-8) [10](#0-9)
