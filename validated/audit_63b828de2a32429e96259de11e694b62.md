### Title
Chain-Sync Eviction Timeout Indefinitely Reset by Incremental Header Submissions — (File: `sync/src/synchronizer/mod.rs`)

---

### Summary

The `eviction()` function in `sync/src/synchronizer/mod.rs` resets the `chain_sync.timeout` eviction deadline whenever a peer's `best_known_header` difficulty catches up to the *previously recorded* `total_difficulty` snapshot — not the current local tip. A malicious outbound peer can exploit this by repeatedly sending headers that incrementally improve to just meet the recorded threshold, causing the eviction deadline to reset to `now + CHAIN_SYNC_TIMEOUT` indefinitely, preventing the peer from ever being disconnected.

---

### Finding Description

The `eviction()` function implements a chain-sync timeout to disconnect outbound peers that persistently lag behind the local chain tip. The logic has three branches:

**Branch 1 (lines 590–600):** If the peer's `best_known_header.total_difficulty >= local_total_difficulty` (current tip), clear the timeout — the peer has caught up.

**Branch 2 (lines 601–614):** If `timeout == 0` (first detection) **OR** `best_known_header.total_difficulty >= state.chain_sync.total_difficulty` (peer caught up to the *previously recorded* difficulty snapshot), **reset** the timeout to `now + CHAIN_SYNC_TIMEOUT` and update `total_difficulty` to the current tip.

**Branch 3 (lines 615–641):** If the timeout has expired, send a final `GetHeaders` or evict. [1](#0-0) 

The reset condition in Branch 2 is the root cause. When the timeout is reset, `state.chain_sync.total_difficulty` is updated to the current local tip difficulty. This creates a repeating window:

1. Local tip difficulty is `D0` at time `T0`. Timeout set to `T0 + CHAIN_SYNC_TIMEOUT`, `recorded_difficulty = D0`.
2. The local chain advances; tip difficulty is now `D1 > D0`.
3. Before the timeout expires, the peer sends a header with difficulty `D_peer` where `D0 ≤ D_peer < D1`.
4. Branch 2 condition fires: `D_peer >= recorded_difficulty (D0)` is true, but `D_peer < local_total_difficulty (D1)` is also true, so Branch 1 does not fire.
5. Timeout resets to `now + CHAIN_SYNC_TIMEOUT`, `recorded_difficulty = D1`.
6. Attacker repeats: sends headers that just meet `D1` before the new timeout expires.
7. The peer is never evicted.

The `may_set_best_known_header` function only updates the peer's best known header if the new one is strictly better, so the attacker only needs to send one header per cycle that meets the threshold. [2](#0-1)

### Impact Explanation

An attacker controlling a peer that the local node has established an **outbound** connection to can hold that connection slot indefinitely without contributing useful blocks. CKB nodes have a limited number of outbound connection slots. By occupying one or more slots with non-contributing peers, the attacker degrades the node's ability to sync from honest peers, slowing or stalling block synchronization. In a targeted attack against a specific node (e.g., a miner or exchange node), this can contribute to eclipse-attack preconditions by exhausting outbound slots.

The eviction mechanism is specifically designed to protect against this class of slow/stalling peer — the reset bug directly nullifies that protection for outbound non-protected, non-whitelist peers.

---

### Likelihood Explanation

- The attacker only needs to control one peer that the victim node connects to outbound (e.g., by running a publicly reachable CKB node).
- The attacker must send headers at a rate that keeps up with the local chain's difficulty growth — one header per `CHAIN_SYNC_TIMEOUT` window is sufficient.
- No privileged access, leaked keys, or majority hashpower is required.
- The attack is sustainable as long as the attacker's node remains reachable and sends valid (but stale) headers. [4](#0-3) 

---

### Recommendation

The reset in Branch 2 should only fire when the peer has genuinely made progress toward the **current** local tip, not just the previously recorded snapshot. One approach: remove the `>= state.chain_sync.total_difficulty` reset condition entirely, so the timeout is only cleared (Branch 1) when the peer fully catches up, and only set once (Branch 2 with `timeout == 0`). Alternatively, cap the number of times the timeout can be reset per peer connection, or use a monotonically advancing deadline that cannot be pushed back by incremental header submissions.

### Proof of Concept

```
Setup:
  - Local node tip difficulty: D0 = 1000
  - CHAIN_SYNC_TIMEOUT = T (e.g., 20 minutes in ms)
  - Attacker controls outbound peer P

Step 1 (t=0):
  eviction() runs. P's best_known_header difficulty = 0 < D0.
  Branch 2 (timeout==0): timeout = 0 + T, recorded_difficulty = 1000.

Step 2 (t = T - 1ms, just before expiry):
  Local chain has advanced: local_total_difficulty = D1 = 1100.
  Attacker sends P a header with difficulty = 1000 (== recorded_difficulty).
  eviction() runs:
    - P.best_known = 1000 < D1 (1100) → Branch 1 does NOT fire.
    - P.best_known (1000) >= recorded_difficulty (1000) → Branch 2 fires.
    - timeout resets to (T-1ms) + T = 2T - 1ms.
    - recorded_difficulty = 1100.

Step 3 (t = 2T - 2ms):
  Local chain: D2 = 1200.
  Attacker sends header with difficulty = 1100 (== recorded_difficulty).
  Branch 2 fires again. timeout = 3T - 2ms. recorded_difficulty = 1200.

→ Peer P is never evicted. Connection slot held indefinitely.
```

The attacker-controlled entry path is the standard CKB sync protocol (`SupportProtocols::Sync`), specifically the `SendHeaders` / `Headers` message handler that calls `may_set_best_known_header`, which feeds into the `eviction()` check on the next timer tick. [5](#0-4)

    pub fn eviction(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let active_chain = self.shared.active_chain();
        let mut eviction = Vec::new();
        let better_tip_header = self.better_tip_header();
        for mut kv_pair in self.peers().state.iter_mut() {
            let (peer, state) = kv_pair.pair_mut();
            let now = unix_time_as_millis();

            if let Some(ref mut controller) = state.headers_sync_controller {
                let better_tip_ts = better_tip_header.timestamp();
                if let Some(is_timeout) = controller.is_timeout(better_tip_ts, now) {
                    if is_timeout {
                        eviction.push(*peer);
                        continue;
                    }
                } else {
                    active_chain.send_getheaders_to_peer(
                        nc,
                        *peer,
                        better_tip_header.number_and_hash(),
                    );
                }
            }

            // On ibd, node should only have one peer to sync headers, and it's state can control by
            // headers_sync_controller.
            //
            // The header sync of other nodes does not matter in the ibd phase, and parallel synchronization
            // can be enabled by unknown list, so there is no need to repeatedly download headers with
            // multiple nodes at the same time.
            if active_chain.is_initial_block_download() {
                continue;
            }
            if state.peer_flags.is_outbound {
                let best_known_header = state.best_known_header.as_ref();
                let (tip_header, local_total_difficulty) = {
                    (
                        active_chain.tip_header().to_owned(),
                        active_chain.total_difficulty().to_owned(),
                    )
                };
                if best_known_header
                    .map(|header_index| header_index.total_difficulty().clone())
                    .unwrap_or_default()
                    >= local_total_difficulty
                {
                    if state.chain_sync.timeout != 0 {
                        state.chain_sync.timeout = 0;
                        state.chain_sync.work_header = None;
                        state.chain_sync.total_difficulty = None;
                        state.chain_sync.sent_getheaders = false;
                    }
                } else if state.chain_sync.timeout == 0
                    || (best_known_header.is_some()
                        && best_known_header
                            .map(|header_index| header_index.total_difficulty().clone())
                            >= state.chain_sync.total_difficulty)
                {
                    // Our best block known by this peer is behind our tip, and we're either noticing
                    // that for the first time, OR this peer was able to catch up to some earlier point
                    // where we checked against our tip.
                    // Either way, set a new timeout based on current tip.
                    state.chain_sync.timeout = now + CHAIN_SYNC_TIMEOUT;
                    state.chain_sync.work_header = Some(tip_header);
                    state.chain_sync.total_difficulty = Some(local_total_difficulty);
                    state.chain_sync.sent_getheaders = false;
                } else if state.chain_sync.timeout > 0 && now > state.chain_sync.timeout {
                    // No evidence yet that our peer has synced to a chain with work equal to that
                    // of our tip, when we first detected it was behind. Send a single getheaders
                    // message to give the peer a chance to update us.
                    if state.chain_sync.sent_getheaders {
                        if state.peer_flags.is_protect || state.peer_flags.is_whitelist {
                            if state.sync_started() {
                                self.shared().state().suspend_sync(state);
                            }
                        } else {
                            eviction.push(*peer);
                        }
                    } else {
                        state.chain_sync.sent_getheaders = true;
                        state.chain_sync.timeout = now + EVICTION_HEADERS_RESPONSE_TIME;
                        active_chain.send_getheaders_to_peer(
                            nc,
                            *peer,
                            state
                                .chain_sync
                                .work_header
                                .as_ref()
                                .expect("work_header be assigned")
                                .into(),
                        );
                    }
                }
            }
        }
        for peer in eviction {
            info!("Timeout eviction peer={}", peer);
            if let Err(err) = nc.disconnect(peer, "sync timeout eviction") {
                debug!("synchronizer disconnect error: {:?}", err);
            }
        }
    }

// Only in effect for connections that are outbound, non-manual,
// non-protected and non-whitelist.

    pub fn may_set_best_known_header(&self, peer: PeerIndex, header_index: HeaderIndex) {
        if let Some(mut peer_state) = self.state.get_mut(&peer) {
            if let Some(ref known) = peer_state.best_known_header {
