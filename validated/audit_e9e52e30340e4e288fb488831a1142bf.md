Audit Report

## Title
Missing `BLOCK_INVALID` Guard Allows Repeated Re-verification of Known-Invalid Headers — (File: sync/src/synchronizer/headers_process.rs)

## Summary
In `HeaderAcceptor::accept()`, the fetched `BlockStatus` is only tested for `HEADER_VALID`; the `BLOCK_INVALID` flag is silently ignored. A developer `FIXME` comment at lines 301–302 explicitly acknowledges this gap. Any unauthenticated P2P peer can repeatedly send `SendHeaders` messages containing a pre-crafted invalid header, forcing the node to re-run `HeaderVerifier::verify()` (including Eaglesong PoW verification) on every delivery with no early-exit guard, causing sustained CPU exhaustion.

## Finding Description
In `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::accept()` (lines 295–358):

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer state and return early
    return result;
}
// No BLOCK_INVALID branch — falls through unconditionally
```

After the `HEADER_VALID` branch, the function unconditionally reaches:
- `prev_block_check` (line 324) — checks parent status, calls `insert_block_status(..., BLOCK_INVALID)` on failure
- `non_contextual_check` (line 334) — calls `self.verifier.verify(self.header)`, which includes Eaglesong PoW verification; calls `insert_block_status(..., BLOCK_INVALID)` on hard failure
- `version_check` (line 346) — calls `insert_block_status(..., BLOCK_INVALID)` on failure

On the first delivery, one of these checks marks the header `BLOCK_INVALID`. On every subsequent delivery, `get_block_status` returns `BLOCK_INVALID`, the `HEADER_VALID` branch is skipped, and all three checks re-execute from scratch. The stored `BLOCK_INVALID` status is fetched but never acted upon.

The contrast with `sync/src/relayer/compact_block_process.rs` lines 259–261 is direct — that path correctly guards:
```rust
} else if status.contains(BlockStatus::BLOCK_INVALID) {
    return StatusCode::BlockIsInvalid.with_context(block_hash);
}
```

The `HeadersProcess::execute()` loop (lines 154–179) iterates over all headers in a `SendHeaders` message (up to `MAX_HEADERS_LEN` = 2000) with no per-hash deduplication for the `BLOCK_INVALID` case, calling `acceptor.accept()` for each.

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker with a single unauthenticated P2P connection can force sustained, unbounded CPU consumption on any reachable CKB node by sending a tight loop of `SendHeaders` messages, each containing up to 2000 copies of the same known-invalid header. Each header re-triggers Eaglesong PoW verification and all other non-contextual checks with no useful work performed. Targeting multiple nodes simultaneously degrades network-wide sync capacity and constitutes network congestion achievable at minimal attacker cost.

## Likelihood Explanation
- Any unauthenticated P2P peer can send `SendHeaders` messages; no privilege, key, or special role is required.
- The attacker needs only one header that fails any of the three checks (invalid PoW nonce, invalid parent, wrong version). The first delivery marks it `BLOCK_INVALID`; every subsequent delivery re-triggers the full check path.
- The `FIXME` comment at lines 301–302 confirms this is a known, unresolved gap, not a speculative edge case.
- No per-hash deduplication guard exists for the `BLOCK_INVALID` case in the processing loop.
- The attack is stateless on the attacker side: a single pre-crafted header, sent in a loop, is sufficient.

## Recommendation
Add an explicit early-return for `BLOCK_INVALID` immediately after the `HEADER_VALID` check in `HeaderAcceptor::accept()`, mirroring the pattern already used in `compact_block_process.rs`:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing logic
    return result;
}
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
```

This resolves the acknowledged `FIXME` and closes the re-verification loop at zero additional cost per subsequent delivery.

## Proof of Concept
1. Connect to a CKB node as an unprivileged P2P peer using the sync protocol.
2. Craft a `SendHeaders` message containing a header with a valid structure but an invalid PoW nonce (does not satisfy the Eaglesong target).
3. Send the message once. The node runs `non_contextual_check` → `HeaderVerifier::verify()` → PoW fails → `insert_block_status(hash, BLOCK_INVALID)` (line 341).
4. In a tight loop, re-send the identical `SendHeaders` message (optionally with up to 2000 copies of the same invalid header per message, per `MAX_HEADERS_LEN`).
5. On each iteration, `accept()` fetches `BLOCK_INVALID` from the status map, skips the `HEADER_VALID` branch (line 304), and falls through to re-run `non_contextual_check` → `HeaderVerifier::verify()` again.
6. Observe sustained CPU consumption on the victim node with no corresponding useful work, confirming the missing guard. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
