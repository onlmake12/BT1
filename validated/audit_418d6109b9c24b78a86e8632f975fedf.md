**The bug is real.** Here is the exact code:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [1](#0-0) 

The intended design is that `timeout_count` accumulates penalty calls, and only after it exceeds 2 does `task_count` decrement. The bug assigns `task_count + num` to `timeout_count` instead of `timeout_count + num`.

**Tracing the arithmetic to find the stuck state:**

- `INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32`, so `task_count` starts at 32.
- Each `decrease(1)` call (NormalToUpper path): `timeout_count = task_count + 1`. Since `task_count >= 2`, this is always `> 2`, so `task_count` decrements by 1 and `timeout_count` resets to 0.
- When `task_count = 1`: `timeout_count = 1 + 1 = 2`. The condition `> 2` is **false**. `task_count` stays at 1 permanently.
- Disconnection in `prune` only fires when `task_count == 0`. [2](#0-1) 

So a peer consistently landing in `NormalToUpper` will have `task_count` reduced to 1 but **never reach 0**, and is never evicted.

**The `NormalToUpper` zone is reachable by a remote peer:**

`remove_by_block` is called when a block response arrives. It computes `elapsed` and calls `time_analyzer.push_time(elapsed)`. If `elapsed > normal_time && elapsed <= low_time`, the result is `NormalToUpper`, which calls `set.decrease(1)`. [3](#0-2) 

Default thresholds: `normal_time = 1250ms`, `low_time = 1500ms`. An attacker responding in ~1300–1499ms lands in this zone. [4](#0-3) 

**Does `prune`'s timeout path save us?**

`prune` calls `punish(2)` (right-shift by 2) only when a block request exceeds `BLOCK_DOWNLOAD_TIMEOUT = 30s`. [5](#0-4) 

If the attacker responds within 30 seconds (just slowly, at ~1300ms), the timeout path never fires. The peer is permanently stuck at `task_count = 1` and is never disconnected.

**`should_punish` guard:**

`decrease` is only called when `should_punish` is true, i.e., `download_schedulers.len() > protect_num` (default 4). [6](#0-5) 

This means the attack requires more than 4 connected peers, which is normal on mainnet.

**`TimeAnalyzer` adaptation concern:**

The `TimeAnalyzer` is shared across all peers and adapts thresholds via exponential moving average. If the attacker is the only slow peer, thresholds may drift upward, eventually reclassifying the attacker's responses as `FastToNormal`. However, the attacker can continuously adjust their response delay to track just above the adapting `normal_time`, sustaining the attack.

### Title
`DownloadScheduler::decrease` assigns `task_count + num` to `timeout_count` instead of accumulating it, preventing `task_count` from reaching 0 for NormalToUpper peers — (`sync/src/types/mod.rs`)

### Summary
A one-character bug (`self.task_count` instead of `self.timeout_count`) in `DownloadScheduler::decrease` causes `task_count` to get permanently stuck at 1 for any peer whose block responses consistently fall in the `NormalToUpper` time quantile. Since peer eviction requires `task_count == 0`, such peers are never disconnected.

### Finding Description
In `DownloadScheduler::decrease` at line 522:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

The intended design is a leaky-bucket: accumulate `timeout_count` across calls, and only decrement `task_count` once the bucket exceeds 2. The bug resets `timeout_count` to `task_count + num` on every call instead of incrementing it. This has two consequences:

1. While `task_count >= 2`, `task_count + num > 2` is always true, so `task_count` decrements on every call (faster than intended).
2. When `task_count = 1` and `num = 1` (NormalToUpper path): `timeout_count = 1 + 1 = 2`, which fails the `> 2` check. `task_count` is never decremented to 0. The peer is never evicted.

### Impact Explanation
A remote peer that deliberately responds to block requests in ~1300–1499ms (above `normal_time`, below `low_time`) will:
- Have `task_count` reduced from 32 to 1 over time (already penalized)
- Then be permanently stuck at `task_count = 1`, never reaching 0
- Never be disconnected by the `prune` eviction check (`task_count == 0`)
- Permanently occupy a download slot

With multiple such peers, honest nodes' download concurrency is degraded. The attacker avoids the 30-second hard timeout by always responding (just slowly).

### Likelihood Explanation
The attack requires only a standard P2P connection and the ability to delay block responses to ~1300–1499ms. No special privileges, keys, or hashpower are needed. The attacker must track the adapting `normal_time` threshold, which is observable indirectly, but this is a low-cost sustained operation.

### Recommendation
Fix line 522 to accumulate `timeout_count` correctly:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

### Proof of Concept
```rust
let mut sched = DownloadScheduler::default(); // task_count = 32, timeout_count = 0
// Drive task_count down to 1

    sched.decrease(1);
