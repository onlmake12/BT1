### Title
Unchecked ETH Refund Return Value Silently Locks Caller Funds - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

In `PythAggregatorV3.sol`, the `updateFeeds()` function performs a low-level `.call{}("")` to refund excess ETH to `msg.sender` after paying the Pyth update fee. The `bool success` return value is explicitly captured but never checked. If the refund call fails, the transaction succeeds anyway and the excess ETH is permanently locked inside the `PythAggregatorV3` contract.

---

### Finding Description

In `updateFeeds()`, after paying the exact update fee to the Pyth contract, the remaining contract balance is sent back to `msg.sender` via a low-level call:

```solidity
// refund remaining eth
// solhint-disable-next-line no-unused-vars
(bool success, ) = payable(msg.sender).call{
    value: address(this).balance
}("");
```

The `// solhint-disable-next-line no-unused-vars` comment explicitly acknowledges that `success` is unused. There is no `require(success, ...)` or any other check. If the call reverts (e.g., `msg.sender` is a contract without a `receive()` or `fallback()` function, or the callee reverts intentionally), the failure is silently swallowed and the excess ETH remains trapped in the `PythAggregatorV3` contract with no recovery mechanism. [1](#0-0) 

---

### Impact Explanation

Any caller (especially a smart contract) that sends more ETH than `getUpdateFee()` requires will permanently lose the excess. The `PythAggregatorV3` contract has no sweep or withdrawal function, so locked ETH is irrecoverable. This is a direct path to asset loss for any overpaying caller whose address cannot receive ETH.

---

### Likelihood Explanation

The entry path is fully unprivileged — any account calling `updateFeeds{value: X}()` where `X > fee` triggers the refund path. Contract callers (e.g., DeFi protocols integrating `PythAggregatorV3` as a Chainlink-compatible adapter) commonly lack `receive()` functions, making silent refund failure a realistic scenario. The `PythAggregatorV3` is explicitly documented as a production integration adapter for Chainlink-compatible apps, increasing the probability of contract callers. [2](#0-1) 

---

### Recommendation

Check the return value of the refund call and revert if it fails:

```solidity
(bool success, ) = payable(msg.sender).call{
    value: address(this).balance
}("");
require(success, "ETH refund failed");
```

Alternatively, use OpenZeppelin's `Address.sendValue()` which reverts on failure, or use `transfer()` (though `transfer()` has a fixed 2300 gas stipend which may be insufficient in some contexts). The safest fix is the explicit `require(success, ...)` check.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract.
2. Deploy an attacker contract with no `receive()` function that calls `updateFeeds{value: 1 ether}(priceUpdateData)` where the actual fee is, say, 1 wei.
3. Observe: the transaction succeeds, the price feed is updated, but `1 ether - 1 wei` is permanently locked in `PythAggregatorV3` with no way to recover it.
4. Confirm: `address(pythAggregatorV3).balance > 0` after the call, and no withdrawal function exists on the contract. [3](#0-2)

        // Update the prices to the latest available values and pay the required fee for it. The `priceUpdateData` data
