### Title
Governance Sequence Skipping via Permissionless Relayer Allows Permanent Suppression of Any Governance Action — (`target_chains/sui/contracts/sources/governance/governance.move`)

---

### Summary

The governance sequence guard uses a strictly-greater-than check (`sequence > last_executed`) rather than an exact-next check (`sequence == last_executed + 1`). Because governance VAA relay is fully permissionless, any unprivileged actor who holds a valid higher-sequence VAA can submit it first, permanently making any lower-sequence VAA unexecutable.

---

### Finding Description

In `execute_governance_instruction`, the only ordering guard is:

```move
assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
``` [1](#0-0) 

After passing, the state is immediately updated to the submitted sequence:

```move
state::set_last_executed_governance_sequence(&latest_only, pyth_state, sequence);
``` [2](#0-1) 

The identical pattern exists in `contract_upgrade::authorize_upgrade`:

```move
assert!(sequence > state::get_last_executed_governance_sequence(pyth_state),
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER);
state::set_last_executed_governance_sequence_unchecked(pyth_state, sequence);
``` [3](#0-2) 

The entry points `governance::verify_vaa` and `governance::execute_governance_instruction` are both `public` with no signer/role restriction — only the VAA's emitter chain/address is validated:

```move
assert!(
    pyth::state::is_valid_governance_data_source(pyth_state, vaa_data_source),
    E_INVALID_GOVERNANCE_DATA_SOURCE
);
``` [4](#0-3) 

Wormhole VAAs are public once signed by guardians. Any observer can relay any signed VAA in any order. There is no `consumed_vaas` check in the governance execution path — the sequence number is the sole replay/ordering protection. [5](#0-4) 

---

### Impact Explanation

Attack sequence (last_executed = N):

1. Attacker relays VAA(seq=N+2): `N+2 > N` passes → `last_executed` set to `N+2`.
2. Legitimate relay of VAA(seq=N+1): `N+1 > N+2` fails → aborts with `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER`.
3. VAA(seq=N+1) is permanently unexecutable. No recovery path exists in the contract.

If VAA(N+1) is a contract upgrade patching a live vulnerability, that upgrade is permanently suppressed. This directly matches the stated scope: *governance voting result manipulation that changes execution away from the voted outcome*.

---

### Likelihood Explanation

- Wormhole VAAs are broadcast publicly once guardian-signed; any network observer can relay them.
- During normal governance operations, multiple VAAs may be signed and pending simultaneously (e.g., a parameter change at N+1 followed by an upgrade at N+2).
- The attacker needs no special privilege — only the ability to submit a Sui transaction calling the public `verify_vaa` + `execute_governance_instruction` functions with a valid higher-sequence VAA.
- The window exists whenever two governance VAAs are signed but not yet executed on-chain.

---

### Recommendation

Replace the greater-than check with a strict equality-to-next check:

```move
assert!(
    sequence == state::get_last_executed_governance_sequence(pyth_state) + 1,
    E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER
);
```

Apply the same fix in `contract_upgrade::authorize_upgrade`. [6](#0-5) 

If intentional skipping is ever needed (e.g., emergency bypass), it should require an explicit privileged override, not be silently exploitable by any relayer.

---

### Proof of Concept

```
Initial state: last_executed_governance_sequence = 4

Step 1: Attacker calls verify_vaa(VAA_seq_6) → receipt_6
        Attacker calls execute_governance_instruction(receipt_6)
        Check: 6 > 4  ✓  → last_executed set to 6

Step 2: Legitimate relayer calls verify_vaa(VAA_seq_5) → receipt_5
        Legitimate relayer calls execute_governance_instruction(receipt_5)
        Check: 5 > 6  ✗  → ABORT E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER

Result: VAA_seq_5 (e.g., a critical contract upgrade) is permanently unexecutable.
```
