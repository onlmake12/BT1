### Title
Unbounded `runtime::spawn` in `ConnectionSyncProcess::execute` Enables FD Exhaustion via Fake `from` Keys — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

An unprivileged remote peer with a single P2P connection can exhaust the victim node's OS file descriptor table by: (1) flooding `pending_delivered` with entries keyed on distinct fake `from` peer IDs, then (2) sending matching `ConnectionSync` messages that each trigger an unbounded `runtime::spawn` of a task holding up to 24 concurrent `try_nat_traversal` futures, each of which opens a `TcpSocket` for up to 30 seconds per retry cycle.

---

### Finding Description

**Phase 1 — Populate `pending_delivered` with distinct `from` keys**

`ConnectionRequestProcess::respond_delivered` inserts attacker-controlled listen addresses into `pending_delivered` whenever the victim is the `to` target: [1](#0-0) 

The only per-key guard is a `HOLE_PUNCHING_INTERVAL` (2 min) cooldown on the *same* `from` key: [2](#0-1) 

Using K distinct fake `from` peer IDs trivially bypasses this. The `pending_delivered` `HashMap` has no size cap: [3](#0-2) 

The session-level `rate_limiter` (30 req/sec per `(session_id, msg_item_id)`) slows but does not prevent accumulation — 30 entries/sec × 300 sec (5-min `TIMEOUT` window) = 9,000 entries. [4](#0-3) 

**Phase 2 — Trigger unbounded task spawning via `ConnectionSync`**

`ConnectionSyncProcess::execute` looks up `content.from` in `pending_delivered` and, on a hit, unconditionally calls `runtime::spawn`: [5](#0-4) [6](#0-5) 

The `forward_rate_limiter` is keyed on `(from, to, msg_item_id)`: [7](#0-6) 

Each `ConnectionSync` with a distinct `from` is a fresh key — the 1/sec limit is fully bypassed. There is **no cap** on the number of spawned tasks.

**Phase 3 — FD exhaustion inside `try_nat_traversal`**

Each spawned task drives up to 24 concurrent `try_nat_traversal` futures via `select_ok`. Each future opens a new `TcpSocket` on every retry iteration and holds it for the 200 ms connect timeout: [8](#0-7) 

The retry loop runs for up to 30 seconds: [9](#0-8) 

`create_socket` allocates a real OS file descriptor on every call: [10](#0-9) 

---

### Impact Explanation

At 30 `ConnectionSync` messages/sec (session rate limit), after 30 seconds there are 900 concurrent spawned tasks. Each task holds up to 24 sockets simultaneously during the connect phase. Peak concurrent open sockets: **900 × 24 = 21,600**. This exceeds the Linux default soft FD limit (1,024) and approaches the hard limit (65,536), causing `EMFILE`/`ENFILE` errors that crash or severely degrade the CKB node and sever all P2P connections.

---

### Likelihood Explanation

- Requires only one unauthenticated P2P connection to the victim.
- The victim's peer ID is public (exchanged during P2P handshake).
- No PoW, no key material, no privileged role needed.
- The attack is fully automatable with a script that generates K random `PeerId` values.
- The `pending_delivered` entries survive for 5 minutes (`TIMEOUT`), giving ample time to accumulate entries before triggering the sync flood.

---

### Recommendation

1. **Cap `pending_delivered` size**: Enforce a hard maximum (e.g., 256 entries) and evict oldest entries on overflow.
2. **Bound concurrent NAT traversal tasks**: Maintain a global semaphore or counter; reject new spawns when the limit is reached.
3. **Rate-limit `ConnectionSync` by `to` (victim) rather than `(from, to)`**: The current `forward_rate_limiter` key allows unlimited distinct `from` keys to bypass the 1/sec guard.
4. **Validate `from` peer ID authenticity**: Only accept `ConnectionSync` messages whose `from` field matches a peer ID that was legitimately involved in a prior `ConnectionRequest` routed through a connected session, not an arbitrary attacker-supplied value.

---

### Proof of Concept

```
1. Connect to victim (single TCP session).
2. For i in 1..K:
     Send ConnectionRequest(from=random_peer_id_i, to=victim_peer_id,
                            listen_addrs=[24 routable TCP addrs], route=[])
     # victim inserts random_peer_id_i -> ([24 addrs], now) into pending_delivered
3. For i in 1..K:
     Send ConnectionSync(from=random_peer_id_i, to=victim_peer_id, route=[])
     # victim spawns task with select_ok over 24 try_nat_traversal futures
4. After ~30 seconds, assert open fd count on victim process exceeds ulimit -n.
   Node crashes or stops accepting connections with EMFILE.
```

With K=30 (one second of messages at the session rate limit), 720 sockets are open simultaneously. With K=900 (30 seconds), ~21,600 sockets are open, reliably exhausting the default Linux FD table.

fn create_socket(
    bind_addr: Option<SocketAddr>,
    target_addr: SocketAddr,
) -> Result<TcpSocket, std::io::Error> {
    let socket = match bind_addr {
        Some(listen_addr) => match (listen_addr.ip(), target_addr.ip()) {
            (IpAddr::V4(_), IpAddr::V4(_)) => {
                let socket = TcpSocket::new_v4()?;
                socket.set_reuseaddr(true)?;
                #[cfg(all(unix, not(target_os = "solaris"), not(target_os = "illumos")))]
                socket.set_reuseport(true)?;
                socket.bind(listen_addr)?;
                socket
            }
            (IpAddr::V6(_), IpAddr::V6(_)) => {
                let socket = TcpSocket::new_v6()?;
                socket.set_reuseaddr(true)?;
                #[cfg(all(unix, not(target_os = "solaris"), not(target_os = "illumos")))]
                socket.set_reuseport(true)?;
                socket.bind(listen_addr)?;
                socket
            }
            (IpAddr::V4(_), IpAddr::V6(_)) => TcpSocket::new_v6()?,
            (IpAddr::V6(_), IpAddr::V4(_)) => TcpSocket::new_v4()?,
        },
        None => match target_addr.ip() {
            IpAddr::V4(_) => TcpSocket::new_v4()?,
            IpAddr::V6(_) => TcpSocket::new_v6()?,
        },
    };
    Ok(socket)
}
