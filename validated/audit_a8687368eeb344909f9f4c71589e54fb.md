### Title
Missing Payload Freshness Check in `verifyUpdate` Enables Indefinite Replay of Stale Lazer Price Updates - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

The `verifyUpdate` function in `PythLazer.sol` enforces only the signer certificate's `expiresAt` but performs no check on the timestamp embedded inside the signed price payload. Any signed Lazer update message remains replayable for the entire lifetime of the signer's certificate — potentially months — with no per-message expiry or sequence-number protection. This is the direct on-chain analog of the reported "no idle timeout / extended auth lifetime" class: a credential that should be short-lived is instead valid for an unbounded window.

---

### Finding Description

`PythLazer.sol` stores trusted signers with an `expiresAt` field and exposes `verifyUpdate`:

```solidity
function isValidSigner(address signer) public view returns (bool) {
    return block.timestamp < trustedSignerToExpiresAtMapping[signer];
}

function verifyUpdate(bytes calldata update)
    external payable
    returns (bytes calldata payload, address signer)
{
    ...
    payload = update[71:71 + payload_len];
    bytes32 hash = keccak256(payload);
    (signer, , ) = ECDSA.tryRecover(hash, uint8(update[68]) + 27,
        bytes32(update[4:36]), bytes32(update[36:68]));
    if (signer == address(0)) revert("invalid signature");
    if (!isValidSigner(signer)) revert("invalid signer");   // ← only check
}
``` [1](#0-0) 

The function verifies:
1. Fee payment
2. EVM format magic
3. ECDSA signature validity
4. Signer certificate not expired (`block.timestamp < expiresAt`)

It does **not** verify that the timestamp embedded inside `payload` is recent. The Lazer payload carries a price-feed timestamp (the publish time of the price data), but `verifyUpdate` never reads or bounds-checks it. Because Lazer update messages are publicly broadcast, any observer can capture a signed message and replay it at any future time — as long as the signer's certificate has not yet expired.

Signer certificates are set via governance with an `expiresAt` that can be months or years in the future:

```solidity
function updateTrustedSigner(address trustedSigner, uint256 expiresAt)
    external onlyOwner { ... }
``` [2](#0-1) 

There is no nonce, sequence number, or per-message expiry window anywhere in the contract. Compare this with the Pyth governance contract, which explicitly prevents replay via a monotonically increasing sequence number:

```solidity
if (vm.sequence <= lastExecutedGovernanceSequence())
    revert PythErrors.OldGovernanceMessage();
``` [3](#0-2) 

The Lazer Sui and Aptos contracts do enforce signer expiry correctly, but neither adds per-message replay protection either. The EVM contract is the most exposed because it is the primary integration surface for DeFi protocols.

---

### Impact Explanation

An integrator contract that calls `verifyUpdate` and uses the returned `payload` without independently parsing and bounding the embedded publish timestamp will silently accept arbitrarily stale price data. In a DeFi context (lending, perpetuals, options), stale prices enable:

- **Liquidation manipulation**: replay a price update from a period of extreme volatility to trigger or block liquidations at incorrect prices.
- **Arbitrage against the protocol**: replay a price that diverges significantly from the current market price to extract value from AMMs or lending pools that rely on Lazer.

The attacker does not need any privileged access. Lazer update messages are publicly observable on-chain or via the Lazer feed. The only prerequisite is that the signer's certificate has not yet expired — a window that can span months.

---

### Likelihood Explanation

- Lazer update messages are publicly broadcast; capturing historical messages requires no special access.
- The EVM contract is the primary integration target for DeFi protocols on EVM chains.
- Integrators who read the function signature (`verifyUpdate` → returns `payload, signer`) may reasonably assume the contract has validated freshness, since the function name implies full verification.
- No documentation or NatSpec in the contract warns callers to independently check the payload timestamp.

Likelihood: **Medium** — requires a DeFi integrator that omits the payload timestamp check, combined with an attacker who captures and replays an old signed message.

---

### Recommendation

1. Parse the publish timestamp from the payload inside `verifyUpdate` and enforce a configurable maximum age (e.g., `block.timestamp - publishTime <= maxPayloadAge`).
2. Alternatively, add a per-message nonce or sequence number to the signed payload and track consumed nonces on-chain, preventing any replay regardless of age.
3. At minimum, add explicit NatSpec documentation warning integrators that `verifyUpdate` does **not** enforce payload freshness and that callers must check the embedded timestamp.

---

### Proof of Concept

1. Observe a Lazer EVM update message broadcast on-chain at time `T` with price `P_old`. Record the raw `update` bytes.
2. Wait until the market price moves significantly to `P_current ≠ P_old` (e.g., 24 hours later).
3. Call `verifyUpdate{value: 1 wei}(update_old)` on the deployed `PythLazer` contract.
4. The call succeeds: `isValidSigner` passes because the signer certificate has not expired; no timestamp check exists.
5. The contract returns `(payload_old, signer)` — the payload containing `P_old` — with no revert.
6. Any integrator contract that feeds the returned payload into its pricing logic without checking the embedded publish time will price assets at `P_old`, enabling the attacker to exploit the price discrepancy. [4](#0-3)
