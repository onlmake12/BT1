### Summary

A single unprivileged remote peer can cause unbounded heap growth in `SyncState::unknown_tx_hashes` by repeatedly sending `RelayTransactionHashes` messages containing the same set of K < 32767 unknown tx hashes. Each call to `add_ask_for_txs` unconditionally appends the sender's `PeerIndex` to the `peers` Vec of every matching `UnknownTxHashPriority` entry via `push_peer`, with no deduplication or cap. The post-insertion guard only checks the **count of distinct hash keys** in the map, not the **total size of all `peers` Vecs**, so it never fires when K < 32767 and only one peer is connected.

### Finding Description

**`push_peer` is unconditional:** [1](#0-0) 

Every call to `add_ask_for_txs` for an already-known hash hits the `Occupied` branch and calls `push_peer` before any guard is evaluated: [2](#0-1) 

**The guard checks the wrong dimension:**

The post-insertion check at line 1507 evaluates `unknown_tx_hashes.len()`, which is the number of **distinct hash keys** in the `KeyedPriorityQueue`, not the sum of all `peers.len()` values across entries: [3](#0-2) 

**The off-by-one bypass (single peer):**

Constants are:
- `MAX_RELAY_TXS_NUM_PER_BATCH = 32767`
- `MAX_UNKNOWN_TX_HASHES_SIZE = 50000`
- `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767` [4](#0-3) 

With one connected peer and K = **32766** hashes (one below the per-message cap):
- `unknown_tx_hashes.len() = 32766`
- Condition 1: `32766 >= 50000` â†’ **false**
- Condition 2: `32766 >= 1 Ã— 32767 = 32767` â†’ **false**

The guard **never fires**. The peer can send the same 32766 hashes in message after message; each call appends `peer_index` to every entry's `peers` Vec without limit.

**The multi-peer bypass (N â‰¥ 2):**

With N = 2 peers and K = 32767 hashes:
- Condition 2: `32767 >= 2 Ã— 32767 = 65534` â†’ **false**

As more peers connect, the threshold `peers.state.len() Ã— 32767` grows faster than `unknown_tx_hashes.len()` (which is capped at K â‰¤ 32767), so the guard is permanently bypassed for any N â‰¥ 2.

**No tx_filter protection:**

The `tx_filter` in `execute()` only removes hashes that have been **resolved** (tx received and processed). Unknown hashes are never added to `tx_filter`, so the same unknown hashes pass the filter on every repeated message: [5](#0-4) 

There is no rate-limit on `RelayTransactionHashes` messages and no mechanism to disconnect a peer that never delivers the requested transactions.

### Impact Explanation

Each `PeerIndex` is a small integer (â‰ˆ8 bytes). With K = 32766 hashes and M repeated messages from one peer:

The `unknown_tx_hashes` mutex is held during the entire insertion loop, so the growing allocation also causes lock contention that degrades all relay processing.

### Likelihood Explanation

The attacker needs only a single P2P connection to a CKB node. No PoW, no keys, no Sybil attack is required. The message format is standard (`RelayTransactionHashes`), the hashes need not correspond to real transactions, and the node has no rate-limit on this message type. The bypass (K = 32766) is trivially discovered by reading the constants.

### Recommendation

1. **Deduplicate or cap `peers` in `push_peer`**: before appending, check `if !self.peers.contains(&peer_index)` or enforce `if self.peers.len() < MAX_PEERS_PER_HASH`.
2. **Guard on total peer-list size, not key count**: replace the post-insertion check with a bound on `Î£ priority.peers.len()` across all entries, or track a running total.
3. **Rate-limit `RelayTransactionHashes` per peer**: apply a token-bucket or message-rate cap at the protocol handler level.
