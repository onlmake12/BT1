Audit Report

## Title
Missing Per-Peer Rate Limiter in `BlockFilter` Protocol Enables Unbounded RocksDB Read Amplification — (`sync/src/filter/mod.rs`)

## Summary
The `BlockFilter` protocol handler lacks a per-peer rate limiter, unlike `Relayer` which explicitly enforces a 30 req/s cap per `(PeerIndex, message_id)` pair. Any unprivileged peer that connects to a node with `block_filter` enabled can send `GetBlockFilterHashes` messages at maximum TCP rate, each triggering up to 4002 RocksDB reads (2000 loop iterations × 2 reads each, plus 2 parent reads). With up to 125 inbound connections (the default `max_peers`), this generates up to ~500,000 RocksDB reads per message cycle, saturating the shared I/O path and degrading block import latency and sync throughput on the targeted node.

## Finding Description
**Root cause — missing rate limiter field and guard:**

`BlockFilter` holds only `shared: Arc<SyncShared>` with no rate limiter: [1](#0-0) 

`BlockFilter::try_process` dispatches directly to handlers with no throttle check: [2](#0-1) 

By contrast, `Relayer` carries a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field: [3](#0-2) 

And checks it at the top of `try_process` before any dispatch: [4](#0-3) 

**Read amplification per message:**

`GetBlockFilterHashesProcess::execute` loops up to `BATCH_SIZE = 2000` times, calling `get_block_hash` and `get_block_filter_hash` (two direct RocksDB reads) per iteration, plus two additional reads for the parent when `start_number > 0`: [5](#0-4) [6](#0-5) 

`GetBlockFiltersProcess` adds up to 2000 reads (BATCH_SIZE=1000, 2 reads/block): [7](#0-6) 

`GetBlockFilterCheckPointsProcess` adds up to 4000 reads (BATCH_SIZE=2000, 2 reads/block): [8](#0-7) 

**No throttle in the `received` handler:**

`BlockFilter::received` calls `self.process(...)` with no rate check: [9](#0-8) 

**Connection limit context:**

The default config allows up to 125 peers (`max_peers = 125`), and the service layer allows up to 1024 raw connections: [10](#0-9) [11](#0-10) 

## Impact Explanation
The concrete impact is local node performance degradation: saturating RocksDB's shared I/O path with filter-hash reads starves concurrent block validation and sync reads, increasing block import latency and potentially causing the node to fall behind the chain tip (sync stall). The node does not crash. This matches **Low (501–2000 points): Any other important performance improvements for CKB**, as the impact is local to the targeted node and does not affect the broader network or cause a node crash.

## Likelihood Explanation
- `block_filter` is opt-in, limiting the exposed population to nodes that have explicitly enabled it.
- Any unprivileged peer can connect and send these messages; no authentication, stake, or PoW is required.
- The attack is trivially scriptable: open N connections (up to `max_peers`), loop sending a 9-byte `GetBlockFilterHashes { start_number: 0 }` message from each.
- The contrast with `Relayer`'s explicit 30 req/s cap confirms this is an oversight, not an intentional design choice.

## Recommendation
Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`'s implementation at `sync/src/relayer/mod.rs` lines 63–67 and 88–98) and check it at the top of `BlockFilter::try_process` before dispatching to any of the three handler processes. A cap of 30 req/s per `(peer, message_type)` pair, consistent with `Relayer`'s quota, would bound per-peer I/O cost to at most 120,000 RocksDB reads/s across all peers at the default `max_peers = 125`.

## Proof of Concept
1. Enable `block_filter` on a CKB node and allow it to build filter data for ≥2000 blocks.
2. Open N TCP connections (up to `max_peers = 125`) to the node's P2P port, each completing the Filter protocol handshake.
3. From each connection, send `GetBlockFilterHashes { start_number: 0 }` in a tight loop.
4. Monitor RocksDB read IOPS via `ckb_metrics` or `rocksdb.stats` and block import latency.
5. Assert: read IOPS scales linearly with N; block import latency degrades proportionally; the node falls behind the chain tip under sustained load.
