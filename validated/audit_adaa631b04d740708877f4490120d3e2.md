### Title
Permanent Fund Lock: Update Fees Deposited into `PriceInfoObject` Are Irrecoverable — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `pyth::pyth::update_single_price_feed` deposits the caller's SUI fee into a dynamic field of the target `PriceInfoObject` via `price_info::deposit_fee_coins`. No withdrawal, extraction, or transfer function exists anywhere in the contract suite to move these funds out. The `fee_recipient_address` stored in `State` is vestigial — the module that previously routed fees to it was explicitly retired and now only updates the address field without ever using it for disbursement.

### Finding Description
In `pyth.move`, `update_single_price_feed` (line 277) unconditionally calls `price_info::deposit_fee_coins(price_info_object, fee)`. [6](#0-5) 

`deposit_fee_coins` either creates a new `Coin<SUI>` dynamic object field keyed by `FEE_STORAGE_KEY`, or joins the incoming coin into the existing one: [1](#0-0) 

A full audit of all `.move` sources confirms there is no counterpart function — no `withdraw_fee_coins`, no `collect_fees`, no `transfer_to_recipient` — anywhere in the module. `get_balance` is read-only. [7](#0-6) 

The governance module `set_fee_recipient` can update `State.fee_recipient_address`, but its own comment acknowledges the fee-routing logic was removed: [8](#0-7) 

`State` stores `fee_recipient_address` but no code path ever reads it to disburse funds: [9](#0-8) 

### Impact Explanation
Every SUI token paid as an update fee is permanently locked inside the corresponding `PriceInfoObject`. Since `PriceInfoObject` is a shared object with no owner-controlled destruction path and no fee-extraction entry point, the locked balance is irrecoverable by any party — including the designated `fee_recipient_address`. The total locked amount grows linearly with the number of price updates across all feeds. On a high-throughput deployment this constitutes unbounded, permanent loss of protocol revenue.

### Likelihood Explanation
This triggers on every normal, legitimate call to `update_single_price_feed`. No special attacker action is required; the fund lock is the default behavior of the current code. Any user updating any price feed contributes to the locked pool.

### Recommendation
Add a privileged `withdraw_fees` entry function (gated on `LatestOnly` or a governance capability) to `price_info` that extracts the `Coin<SUI>` from the `FEE_STORAGE_KEY` dynamic field and transfers it to `State.fee_recipient_address`. Alternatively, route fees directly to `fee_recipient_address` inside `update_single_price_feed` using `sui::transfer::public_transfer` instead of depositing into the `PriceInfoObject`.

### Proof of Concept
1. Deploy the contract with `base_update_fee = 50`.
2. Call `update_single_price_feed` N times with valid VAAs and `fee = 50` SUI each.
3. Assert `price_info::get_balance(price_info_object) == 50 * N` — passes.
4. Search all modules for any function that calls `dynamic_object_field::remove` or `coin::take` on `FEE_STORAGE_KEY` — none found.
5. Attempt to recover funds via `fee_recipient_address` — no entry point exists; funds are permanently locked.

The test at `pyth.move:817` already confirms step 3 as expected behavior: [10](#0-9)

        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);

        };
