Audit Report

## Title
`DownloadScheduler::decrease()` reads `task_count` instead of `timeout_count`, bypassing slow-response accumulation guard - (File: sync/src/types/mod.rs)

## Summary
In `sync/src/types/mod.rs`, the `decrease()` method at L521–527 assigns `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` is initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (always >> 2), the guard `timeout_count > 2` is trivially satisfied on every call, causing `task_count` to be decremented on every slow block response rather than only after 3 or more accumulated slow responses. This causes peers to be disconnected approximately 3× faster than the protocol intends.

## Finding Description
`DownloadScheduler` tracks per-peer block download concurrency with `task_count` (initialized to `INIT_BLOCKS_IN_TRANSIT_PER_PEER`) and `timeout_count` (initialized to 0). [1](#0-0) 

The `decrease()` method contains the bug: [2](#0-1) 

Line 522 overwrites `timeout_count` with `task_count + num` instead of accumulating into `timeout_count`. Since `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (a value >> 2) and `num` is 1 or 2, the result is always > 2, so the guard fires unconditionally: `task_count` is decremented and `timeout_count` is reset to 0 on every single call. The intended accumulation (requiring 3+ slow responses before penalizing) never occurs.

`decrease()` is called from `remove_by_block()` on every completed block download whose elapsed time falls in the `NormalToUpper` or `UpperToMax` quantile: [3](#0-2) 

When `task_count` reaches 0, `prune()` adds the peer to `disconnect_list`: [4](#0-3) 

## Impact Explanation
This matches **Low (501–2000 points): Any other important performance improvements for CKB**. The bug degrades sync performance by disconnecting peers with variable latency ~3× faster than the protocol intends (after ~32 slow responses instead of ~96). This reduces block download parallelism and degrades sync throughput for any node experiencing normal network jitter. The impact does not rise to High because there is no node crash and no network-wide congestion — the effect is bounded to per-peer concurrency reduction and premature disconnection of individual peers.

## Likelihood Explanation
No attacker capability is required. The `remove_by_block()` path is exercised on every completed block download. Any peer whose responses fall in the `NormalToUpper` or `UpperToMax` time quantile — a normal occurrence under variable network conditions — will trigger `decrease()`. The bug is always active for any syncing peer experiencing latency jitter.

## Recommendation
Change line 522 to accumulate into `self.timeout_count` rather than overwrite it with `self.task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // fix: was self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

## Proof of Concept
1. A sync peer connects; its `DownloadScheduler` is initialized with `task_count = INIT_BLOCKS_IN_TRANSIT_PER_PEER` (e.g., 32), `timeout_count = 0`.
2. The peer's block responses consistently fall in the `NormalToUpper` quantile (normal under network jitter).
3. Each completed block download calls `remove_by_block()` → `decrease(1)`.
4. Inside `decrease(1)`: `self.timeout_count = 32 + 1 = 33`. Since 33 > 2, `task_count` is decremented to 31 and `timeout_count` reset to 0.
5. On the next slow response: `timeout_count = 31 + 1 = 32 > 2` → `task_count` decremented to 30.
6. After 32 such responses, `task_count = 0`. The next `prune()` call disconnects the peer.
7. A unit test can confirm this by constructing a `DownloadScheduler`, calling `decrease(1)` 32 times, and asserting `task_count == 0` (demonstrating the bug) versus the fixed behavior where `task_count` remains at 31 after 32 calls (only decremented once per 3 slow responses).
