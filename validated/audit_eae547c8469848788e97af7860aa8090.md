### Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Destroys DAO Depositor Interest — (`File: util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO depositor using a u128 intermediate value, then casts it to u64 with a bare `as u64` (silent truncation). Every other analogous u128→u64 conversion in the same file uses `u64::try_from(…).map_err(|_| DaoError::Overflow)?`. When the intermediate value overflows u64, the returned withdrawal capacity is silently wrong — the depositor's accrued interest is effectively destroyed — and the DAO state field written into block headers is also corrupted, affecting all future depositors.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits if `withdraw_counted_capacity` exceeds `u64::MAX`. The three other u128→u64 conversions in the same file all use the checked form:

```rust
// secondary_block_reward (line 204)
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;

// dao_field_with_current_epoch (line 245)
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);

// dao_field_with_current_epoch (line 258)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`deposit_ar` starts at `10^16` (genesis accumulate rate). With `counted_capacity` near the total CKB supply (~3.36 × 10^18 shannons) and `deposit_ar` at genesis, overflow occurs when `withdrawing_ar > ~5.36 × 10^16` — i.e., when the accumulate rate has grown by a factor of ~5.4 from genesis.

`calculate_maximum_withdraw` is called from three reachable paths:

1. **`transaction_maximum_withdraw` → `withdrawed_interests` → `dao_field_with_current_epoch`**: The DAO state field (`current_s`) written into every block header is computed using the truncated value. A too-small `withdrawed_interests` inflates `current_s`, corrupting the DAO interest ledger for all future depositors. [4](#0-3) [5](#0-4) 

2. **`transaction_fee`**: The fee charged for a DAO withdrawal transaction is computed from the truncated maximum-withdraw value, producing a wrong fee. [6](#0-5) 

3. **RPC `calculate_dao_maximum_withdraw`**: Any RPC caller (wallet, dApp) querying the maximum withdrawal amount receives a silently wrong (far too small) answer. [7](#0-6) 

Note also that `CapacityVerifier` explicitly **skips** the `OutputsSumOverflow` check for DAO withdrawal transactions, delegating correctness to the DAO type script. The node-side `calculate_maximum_withdraw` is therefore the sole host-side guard for the withdrawal amount, making the truncation undetected at the Rust layer. [8](#0-7) 

### Impact Explanation

When the truncation fires:

- The depositor's accrued DAO interest is silently destroyed — they receive a drastically wrong (bit-truncated) capacity instead of their rightful withdrawal.
- The DAO state field `S` (secondary issuance accumulated in the DAO) is inflated in every subsequent block header, causing all future depositors to be credited phantom interest that the DAO cannot actually pay — a slow-motion insolvency.
- The RPC `calculate_dao_maximum_withdraw` returns a wrong value, misleading wallets and users into constructing invalid or under-valued withdrawal transactions.

### Likelihood Explanation

The overflow requires `withdrawing_ar` to grow to ~5.4× the genesis value (`10^16`). The accumulate rate grows at roughly `g2 / C` per block, where `g2` is the secondary block issuance (~6.14 × 10^13 shannons/epoch) and `C` is total capacity (~3.36 × 10^18 shannons). At this rate, reaching 5.4× would take on the order of centuries under current parameters. Likelihood is therefore **very low** on mainnet within any near-term horizon, but the code is demonstrably inconsistent with every other analogous conversion in the same file, constituting a latent programmer mistake of the same class as the reference report.

### Recommendation

Replace the silent cast with the same checked conversion used everywhere else in the file:

```rust
// Before (silent truncation):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (consistent with the rest of the file):
let withdraw_counted_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_u64).safe_add(occupied_capacity)?
``` [9](#0-8) 

### Proof of Concept

```
deposit_ar  = 10_000_000_000_000_000   (genesis DEFAULT_GENESIS_ACCUMULATE_RATE)
withdrawing_ar = 54_000_000_000_000_000  (5.4× genesis — reachable after sufficient time)
counted_capacity = 3_360_000_000_000_000_000  (≈ total CKB supply in shannons)

withdraw_counted_capacity (u128)
  = 3_360_000_000_000_000_000 × 54_000_000_000_000_000
    / 10_000_000_000_000_000
  = 18_144_000_000_000_000_000   (> u64::MAX = 18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_144_000_000_000_000_000 & 0xFFFF_FFFF_FFFF_FFFF
  ≠ correct value

Result: depositor receives a bit-truncated, drastically wrong capacity.
```

The entry path is any unprivileged transaction sender submitting a NervosDAO phase-2 withdrawal transaction, or any RPC caller invoking `calculate_dao_maximum_withdraw`, once the accumulate rate has grown sufficiently.

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
            DaoWithdrawingCalculationKind::WithdrawingOutPoint(withdrawing_out_point) => {
                let (_tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                let withdrawing_out_point: packed::OutPoint = withdrawing_out_point.into();
                let (withdrawing_tx, withdrawing_header_hash) = snapshot
