Audit Report

## Title
Stale `EvictKey` on Surviving Ancestors After `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-clears all link entries for the entire removed subtree before calling `remove_entry` on each node. Because `remove_entry` calls `update_ancestors_index_key`, which resolves ancestors via the now-cleared links, it finds an empty ancestor set and performs no updates. Surviving ancestors of the removed root permanently retain inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`, and a correspondingly stale `evict_key`, allowing a low-fee transaction to evade eviction indefinitely.

## Finding Description
`remove_entry_and_descendants` (lines 252–265) first calls `remove_entry_links` for every node in the subtree, then calls `remove_entry` on each. [1](#0-0) 

`remove_entry_links` (lines 418–430) removes the node from `self.links.inner` entirely via `self.links.remove(id)`. [2](#0-1) 

`remove_entry` (line 242) then calls `update_ancestors_index_key(&entry.inner, EntryOp::Remove)`. [3](#0-2) 

`update_ancestors_index_key` (lines 432–445) calls `self.links.calc_ancestors(&child.proposal_short_id())`. [4](#0-3) 

`calc_ancestors` → `calc_relative_ids` (links.rs lines 42–47): `self.inner.get(short_id)` returns `None` because the link was already removed, so `direct` is an empty `HashSet` and `calc_relation_ids` returns an empty set. [5](#0-4) 

The loop in `update_ancestors_index_key` never executes. `sub_descendant_weight` is never called on any surviving ancestor, and `e.evict_key = e.inner.as_evict_key()` is never refreshed. [6](#0-5) 

`EvictKey` is computed from `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` — all four remain inflated. [7](#0-6) 

## Impact Explanation
`next_evict_entry` selects the eviction candidate by iterating `iter_by_evict_key()` in ascending order. [8](#0-7) 

`EvictKey` ordering is lowest `fee_rate` first; a higher `fee_rate` means the entry is less likely to be evicted. [9](#0-8) 

A surviving ancestor whose removed descendants had high fees retains an inflated `fee_rate` in its `evict_key`, making it appear more valuable than it is and permanently suppressing its eviction. An attacker can exploit this to park arbitrarily many low-fee transactions in the pool, preventing legitimate eviction and causing the pool to reject higher-fee transactions when full. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
The attack requires only standard P2P transaction relay — no privileged access, no PoW, no key material. The attacker submits two sets of ordinary valid transactions (the descendant chain and the conflict). The pool size limit bounds the descendant chain length but does not prevent the attack. The stale state persists for the entire remaining lifetime of the ancestor transaction in the pool. The attack is repeatable at low marginal cost per parked transaction.

## Recommendation
In `remove_entry_and_descendants`, update surviving ancestors **before** clearing links. Specifically, collect the ancestor set of the root `id` while links are still intact, then for each node being removed, call `sub_descendant_weight` and refresh `evict_key` on each surviving ancestor. Alternatively, restructure the function to call `update_ancestors_index_key` (with `EntryOp::Remove`) for the root node before any `remove_entry_links` call, ensuring the ancestor walk can still traverse the live link graph.

## Proof of Concept
Invariant test: after `remove_entry_and_descendants(tx_root)`, for every remaining entry `e` that was an ancestor of `tx_root`:

```rust
assert_eq!(
    e.descendants_count,
    1 + pool_map.calc_descendants(&e.proposal_short_id()).len()
);
```

This assertion will fail because `e.descendants_count` still includes the removed subtree nodes, while `calc_descendants` (operating on the now-correct links) returns the true current set. [10](#0-9) [11](#0-10)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
