### Title
Concurrent Preload/Consume Race Causes Uncaught Panic and Node Crash via Crafted 3-Block Chain — (`chain/src/preload_unverified_blocks_channel.rs`)

---

### Summary

An unprivileged P2P peer can crash a CKB node by relaying a 3-block chain A→B→C where block A passes non-contextual verification but fails contextual verification. A race between the `preload_unverified_block` thread and the `verify_blocks` thread causes the preload thread to call `.expect("parent header stored")` on a block header that the consume thread has already deleted, producing an uncaught panic that terminates the preload thread and cascades to a full node crash.

---

### Finding Description

The pipeline has three concurrent threads:

```
ChainService  ──sends──►  preload_unverified_block  ──sends──►  verify_blocks
(OrphanBroker)             (PreloadUnverifiedBlocksChannel)      (ConsumeUnverifiedBlocks)
```

**Step 1 — All three blocks pass non-contextual verification and enter the pipeline.**

`ChainService::asynchronous_process_block` runs non-contextual checks and calls `insert_block` for each of A, B, C. All three succeed because non-contextual checks do not validate DAO commitments or other contextual rules. [1](#0-0) 

**Step 2 — OrphanBroker sends A, B, C to the preload channel.**

`process_lonely_block` checks whether the parent is `BLOCK_STORED` or in `is_pending_verify`. After A is processed, A is inserted into `is_pending_verify`. When B arrives, its parent (A) is in `is_pending_verify`, so B is also dispatched via `process_descendant`. Same for C (parent B is in `is_pending_verify`). All three end up in `preload_unverified_tx`. [2](#0-1) [3](#0-2) 

**Step 3 — Preload thread sends UnverifiedBlock(A) and UnverifiedBlock(B) into the bounded channel (capacity 128).**

`load_full_unverified_block_by_hash` loads the block body and the parent header from the store for each task. After sending A and B, the preload thread dequeues C's task and begins loading C's parent header — B's header. [4](#0-3) 

**Step 4 — Consume thread processes A (fails) then B (fails, deletes B).**

`consume_unverified_blocks` calls `verify_block`. A fails contextual verification (e.g., invalid DAO). `delete_unverified_block(A)` removes A from the store; A's status becomes `BLOCK_INVALID`. The consume thread immediately processes B: `verify_block` detects A is `BLOCK_INVALID` and returns an error. `delete_unverified_block(B)` removes B — including its header — from the store. [5](#0-4) [6](#0-5) 

**Step 5 — Preload thread panics on the now-deleted B header.**

Concurrently, the preload thread is executing `load_full_unverified_block_by_hash` for C. It calls:

```rust
self.shared
    .store()
    .get_block_header(&parent_hash)   // parent_hash == B_hash
    .expect("parent header stored")   // B was deleted → None → PANIC
``` [7](#0-6) 

**Step 6 — The panic is not caught; the preload thread terminates.**

The `catch_unwind` guard exists only in the **consume** thread's loop, not in the preload thread's loop: [8](#0-7) [9](#0-8) 

When the preload thread panics, the `unverified_block_tx` sender is dropped. The consume thread's `unverified_block_rx.recv()` returns `Err`, causing it to `return` (exit). The chain service thread then joins both threads and exits. The node is dead. [10](#0-9) 

---

### Impact Explanation

A single crafted 3-block relay from any P2P peer crashes the target node's chain service entirely. Because the chain service thread, preload thread, and consume thread all terminate, the node stops processing blocks and becomes permanently unresponsive until manually restarted. If the same blocks are gossiped to multiple peers simultaneously, the crash is network-wide.

---

### Likelihood Explanation

The race window is wide. The `unverified_block_tx` channel holds up to 128 items; the preload thread can enqueue A and B before the consume thread processes either. Contextual failure on A (e.g., invalid DAO) is detected quickly, so the consume thread deletes B well before the preload thread finishes loading C's parent header from disk. No special timing is required — normal IBD or relay conditions are sufficient. The attacker needs only to construct structurally valid blocks (passing `BlockVerifier` + `NonContextualBlockTxsVerifier`) with an invalid DAO field in A, which requires no privileged access or hash power.

---

### Recommendation

1. **Replace `.expect` with graceful error handling** in `load_full_unverified_block_by_hash`. If `get_block_header` returns `None`, log an error and drop the task (or send a sentinel error to the consume thread) rather than panicking.
2. **Add `catch_unwind`** around `self.preload_unverified_channel(...)` in `PreloadUnverifiedBlocksChannel::start`, mirroring the pattern already used in `ConsumeUnverifiedBlocks::start`.
3. **Consider a tombstone check**: before loading C's parent header, check whether the parent's block status is `BLOCK_INVALID`; if so, skip the task and invoke the callback with an error.

---

### Proof of Concept

```
1. Construct block A at height N+1 with:
   - Valid PoW, valid structure (passes BlockVerifier + NonContextualBlockTxsVerifier)
   - Invalid DAO commitment in the header (fails ContextualBlockVerifier)
2. Construct block B at height N+2 with parent = hash(A), valid structure.
3. Construct block C at height N+3 with parent = hash(B), valid structure.
4. Send all three via P2P SendBlock messages to the target node in rapid succession.
5. Observe: the preload_unverified_block thread panics with
   "parent header stored" on C's task, the verify_blocks thread exits,
   and the node's chain service terminates.
```

pub(crate) fn delete_unverified_block(
    store: &ChainDB,
    block_hash: Byte32,
    block_number: BlockNumber,
    parent_hash: Byte32,
) {
    info!(
        "parent: {}, deleting this block {}-{}",
        parent_hash, block_number, block_hash,
    );

    let db_txn = store.begin_transaction();
    let block_op: Option<BlockView> = db_txn.get_block(&block_hash);
    match block_op {
        Some(block) => {
            if let Err(err) = db_txn.delete_block(&block) {
                error!(
                    "delete block {}-{} failed {:?}",
                    block_number, block_hash, err
                );
                return;
            }
            if let Err(err) = db_txn.commit() {
                error!(
                    "commit delete block {}-{} failed {:?}",
                    block_number, block_hash, err
                );
                return;
            }

            info!(
                "parent: {}, deleted this block {}-{}",
                parent_hash, block_number, block_hash,
            );
        }
        None => {
            error!(
                "want to delete block {}-{}, but it not found in db",
                block_number, block_hash
            );
        }
    }
}
