### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Validation, Allowing Disallowed Hash Types in Output Type Scripts — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that the `hash_type` field of every output's **lock script** is within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`), but performs **no equivalent check on the output's type script**. Any unprivileged transaction sender can craft a transaction whose output carries a type script with a disallowed or future `hash_type` value (e.g., `Data3 = 6`, `Data4 = 8`, …). Such a transaction passes the non-contextual verifier and is admitted to the tx pool, forcing every node to perform expensive contextual script resolution before the transaction is ultimately rejected.

---

### Finding Description

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier`, the cheap, context-free gate that runs before any database or VM work:

```
// verification/src/transaction_verifier.rs  lines 61-101
pub struct NonContextualTransactionVerifier<'a> {
    ...
    pub(crate) script_hash_type: ScriptHashTypeVerifier<'a>,
}

pub fn verify(&self) -> Result<(), Error> {
    ...
    self.script_hash_type.verify()?;
    Ok(())
}
```

Its `verify` method iterates only over the **lock** script of each output:

```rust
// lines 796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())   // ← lock only
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else { ... }
    }
    Ok(())   // ← type script hash_type is never inspected
}
```

`ScriptHashType` is defined with an open-ended set of future variants:

```rust
// util/gen-types/src/core.rs  lines 16-32
pub enum ScriptHashType {
    Type  = 1,
    Data  = 0,
    Data1 = 2,
    Data2 = 4,
    // DataN = N << 1  for N in 3..=127
}
```

`ENABLED_SCRIPT_HASH_TYPE` contains only the values permitted by the current consensus rules (currently `{0, 1, 2, 4}`). Values such as `Data3 = 6` are explicitly not permitted. The lock-script path correctly rejects them; the type-script path is never reached.

The existing test suite confirms the gap — only the lock-script case is exercised:

```rust
// verification/src/tests/transaction_verifier.rs  lines 101-122
#[test]
pub fn test_not_enabled_hash_type_output_lock() {
    // ... sets hash_type = Data3 on the LOCK script only
    assert_error_eq!(verifier.verify().unwrap_err(), ...ScriptHashTypeNotPermitted...);
}
// No corresponding test for the TYPE script
```

---

### Impact Explanation

1. **Tx-pool pollution / resource exhaustion (DoS)**: A transaction with a type script carrying `hash_type = Data3` passes `NonContextualTransactionVerifier` and is inserted into the tx pool. Every node must then run the full contextual pipeline — cell resolution, script grouping, and VM invocation — before the transaction is finally rejected. An attacker can flood the pool with such transactions at negligible cost.

2. **Consensus inconsistency risk**: If a miner assembles a block containing such a transaction (possible because the non-contextual check passes), different nodes may handle the subsequent script-execution failure differently depending on implementation details, creating a potential for divergent chain state.

3. **Hardfork boundary bypass**: The `ENABLED_SCRIPT_HASH_TYPE` gate is the enforcement point for which VM versions are active at a given block height. Bypassing it for type scripts undermines the hardfork activation mechanism for type-script-governed cells (e.g., UDT tokens, DAO cells).

---

### Likelihood Explanation

The entry path requires only the ability to submit a transaction via RPC (`send_transaction`) or P2P relay — no privileged access, no key material, no majority hashpower. The malformed field is a single byte in the Molecule-encoded script struct, trivially set by any wallet or script author. The bug is reachable on mainnet today.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        check_hash_type(output.lock().hash_type())?;

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

Add a corresponding test mirroring `test_not_enabled_hash_type_output_lock` for the type-script field.

### Proof of Concept

A transaction sender submits via RPC:

```json
{
  "outputs": [{
    "lock": { "code_hash": "0x...", "hash_type": "data", "args": "0x" },
    "type": { "code_hash": "0x...", "hash_type": "0x06", "args": "0x" },
    "capacity": "0x..."
  }]
}
```

`hash_type = 0x06` corresponds to `Data3`, which is not in `ENABLED_SCRIPT_HASH_TYPE`.

**Step 1**: `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`. The lock script has `hash_type = Data = 0`, which is permitted. The type script's `hash_type = 0x06` is **never checked**. The verifier returns `Ok(())`.

**Step 2**: The transaction is admitted to the tx pool.

**Step 3**: During contextual verification (script execution), the VM attempts to resolve the type script with `hash_type = Data3`. This fails because `Data3` is not a supported VM version, and the transaction is rejected — but only after consuming significant node resources.
