Audit Report

## Title
Tx-Pool Admission Uses Size-Only Fee Check, Allowing Cheap Cycle-Budget Exhaustion Griefing - (File: tx-pool/src/util.rs)

## Summary
`check_tx_fee` gates pool admission using only serialized byte size, while actual resource cost is measured by `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. An attacker can craft a ~200-byte transaction referencing a high-cycle cell dep, pay only the size-based minimum fee (~200 shannons), and force the node to execute up to 70M cycles of script verification before the transaction can be evicted. No weight-based fee check exists post-verification, and the pool's cycle accumulator (`total_tx_cycles`) has no eviction trigger.

## Finding Description

**Size-only admission gate confirmed:**

`check_tx_fee` in `tx-pool/src/util.rs` lines 42–45 explicitly uses `tx_size` only, with a comment acknowledging the gap:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

**No post-verification weight-based check confirmed:**

In `tx-pool/src/process.rs`, after `verify_rtx` returns `verified.cycles` at line 734, the code immediately constructs `TxEntry` at line 751 and calls `submit_entry` at line 753 with no additional fee gate:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// ... declared cycles mismatch check only ...
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
``` [2](#0-1) 

**Weight-based fee rate used only for eviction, not admission:**

`TxEntry::fee_rate()` in `entry.rs` lines 115–117 correctly computes weight-based fee rate using `get_transaction_weight(self.size, self.cycles)`, and `EvictKey` in `sort_key.rs` lines 80–103 evicts by this rate. However, this only applies *after* admission — a transaction already in the pool. [3](#0-2) [4](#0-3) 

**No cycle-based eviction trigger confirmed:**

`limit_size` in `pool.rs` line 298 only triggers on `total_tx_size > config.max_tx_pool_size`. The `total_tx_cycles` field in `component/pool_map.rs` lines 70–71 is tracked but never used as an eviction condition anywhere in `pool.rs`. [5](#0-4) [6](#0-5) 

**Exploit flow:**
1. Deploy a script consuming ~69,999,999 cycles in a live cell dep (keeping tx size small).
2. Craft a ~200-byte transaction referencing that cell dep. Fee = 200 shannons.
3. Submit via `send_transaction` RPC. `check_tx_fee` passes (size-based: `1000 × 200 / 1000 = 200 shannons`).
4. `verify_rtx` executes the script: ~70M cycles consumed on the node.
5. `TxEntry` is created with `size=200`, `cycles=70_000_000`, `fee=200`. Weight-based `fee_rate ≈ 16.7 shannons/KW` — ~60× below minimum — but the transaction is already admitted.
6. Repeat with different UTXOs. Each submission forces full script execution before eviction.

## Impact Explanation

**High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Each submitted transaction forces the node to spend CPU executing up to 70M cycles of script regardless of whether it is eventually evicted. The attacker's cost per unit of CPU disruption is ~60× lower than for legitimate transactions. Continuous resubmission of evicted transactions sustains CPU load and pool churn at low marginal cost. [7](#0-6) 

## Likelihood Explanation

- **Entry path**: Any unprivileged user via `send_transaction` RPC or P2P relay. No special role required.
- **Setup cost**: A high-cycle script deployed once in a cell dep; attacker needs only UTXOs to spend.
- **Persistence**: Evicted transactions can be continuously resubmitted, maintaining CPU pressure at low marginal cost.
- **Naturally occurring**: Complex lock scripts (ZK verifiers, multi-sig aggregation) submitted with minimum fees can trigger this without malicious intent. [8](#0-7) 

## Recommendation

After `verify_rtx` returns `verified.cycles` in `_process_tx`, apply a weight-based minimum fee check before constructing the `TxEntry`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

The existing size-only check in `check_tx_fee` can be retained as a fast pre-filter before script execution. The post-verification check closes the gap between admission cost and actual resource cost. Additionally, consider adding a `total_tx_cycles`-based eviction trigger in `limit_size` analogous to the existing size-based trigger. [9](#0-8) 

## Proof of Concept

1. Deploy a CKB script that loops for exactly 69,999,999 cycles. Store it in a live cell on-chain.
2. Craft a transaction: one input (spendable UTXO), one output (capacity minus fee), one cell dep referencing the high-cycle script. Serialized size ≈ 200 bytes. Fee = 200 shannons.
3. Submit via `send_transaction` RPC.
   - `check_tx_fee`: `min_fee = 1000 × 200 / 1000 = 200 shannons` ✓ passes.
   - `verify_rtx` executes the script: ~70M cycles consumed.
   - `TxEntry` created: `size=200`, `cycles=70_000_000`, `fee=200`.
   - `fee_rate() = FeeRate::calculate(200, max(200, 11940)) ≈ 16.7 shannons/KW` — 60× below minimum — but already admitted.
4. Repeat with different UTXOs. Observe via `tx_pool_info` that `total_tx_cycles` grows rapidly while node CPU is saturated with script verification.
5. Confirm: attacker transactions are deprioritized for block inclusion (correct) but each submission still forces full script execution on the node. [6](#0-5)
