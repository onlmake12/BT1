Audit Report

## Title
Peer Store `check_purge` Eviction Failure Allows Unprivileged Peer to DoS Peer Discovery — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
The `check_purge` function in `PeerStore` has two eviction strategies that can both be defeated simultaneously by an attacker who fills the 16384-entry store with addresses spread across more than 4096 distinct `/16` network groups (≤4 per group). Because freshly announced addresses always have `attempts_count = 0` and `last_connected_at_ms = 0`, they are always considered connectable, defeating strategy 1. Because no group exceeds 4 members, strategy 2 also yields no candidates. The result is `Err(PeerStoreError::EvictionFailed)`, which `add_new_addrs` silently swallows, permanently blocking peer discovery for the victim node.

## Finding Description
`add_addr` calls `check_purge` before inserting any address. [1](#0-0) 

`check_purge` activates only when `addr_manager.count() >= ADDR_COUNT_LIMIT` (16384). [2](#0-1) 

Strategy 1 collects addresses where `is_connectable()` returns `false`. A freshly added `AddrInfo` is created with `last_connected_at_ms = 0` and `attempts_count = 0`. [3](#0-2) [4](#0-3) 

`is_connectable` returns `false` only when `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES (3)`, or when the address has been unreachable for >7 days with ≥10 failures. A freshly added address satisfies neither condition, so strategy 1 always finds zero candidates. [5](#0-4) 

Strategy 2 groups addresses by `/16` network group (`Group::IP4([bits[0], bits[1]])`), takes the top half of groups by size, and evicts 2 random addresses only from groups with **more than 4** members. If the attacker spreads ≤4 addresses across each of >4096 distinct `/16` subnets, no group exceeds 4 members, so the `if addrs.len() > 4` branch is never taken and `candidate_peers` is empty. [6](#0-5) [7](#0-6) 

When both strategies yield no candidates, the function returns `Err(PeerStoreError::EvictionFailed)`. [8](#0-7) 

`DiscoveryAddressManager::add_new_addrs` silently discards this error with only a `debug!` log, so no operator alert is raised and the failure is invisible. [9](#0-8) 

The discovery protocol accepts up to `MAX_ADDR_TO_SEND = 1000` items per `Nodes` message, each with up to `MAX_ADDRS = 3` addresses (3000 addresses per message), so 6 connections suffice to fill the store. [10](#0-9) 

## Impact Explanation
After the store is saturated, every subsequent `add_addr` call returns `EvictionFailed`, and the node can no longer learn about any new peers via the discovery protocol. This maps to **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**: applied cheaply and repeatedly across many nodes, it degrades the network-wide peer discovery infrastructure, can partition bootstrapping or recovering nodes from the network, and requires no on-chain funds or special privileges.

## Likelihood Explanation
Any peer that can establish a TCP connection to the victim node can execute this attack. No authentication or funds are required. The attacker needs approximately 6 connections and one `Nodes` message per connection containing 1000 items × 3 addresses from distinct `/16` subnets. The addresses do not need to be controlled by the attacker. The attack is cheap, repeatable, and the error is invisible to operators.

## Recommendation
1. **Add a last-resort eviction strategy**: when both primary strategies yield no candidates, evict the `N` addresses with `last_connected_at_ms == 0` and the lowest `attempts_count` (i.e., never-tried addresses), regardless of group membership.
2. **Rate-limit address ingestion per session**: cap the number of new addresses accepted from a single session to a small fraction of `ADDR_COUNT_LIMIT`.
3. **Prefer evicting never-connected addresses first**: addresses with `last_connected_at_ms == 0` and `attempts_count == 0` have the lowest trust and should be the first eviction candidates when the store is full.

## Proof of Concept
```
1. Attacker opens 6 TCP connections to the victim CKB node.
2. On each connection, after the discovery handshake, the attacker sends:
     DiscoveryMessage::Nodes { announce: false, items: [1000 Node entries] }
   where each Node has 3 addresses from distinct /16 subnets
   (e.g., 1.0.0.1, 2.0.0.1, 3.0.0.1, ..., up to 18000 unique /16 groups).
3. After 6 messages, the peer store contains ≥16384 entries, all with
   last_connected_at_ms=0, attempts_count=0 (connectable), spread across
   >4096 /16 groups (≤4 per group).
4. Any subsequent call to peer_store.add_addr() triggers check_purge():
   - Strategy 1: all addresses are connectable → 0 candidates.
   - Strategy 2: no group has >4 members → 0 candidates.
   - Returns Err(PeerStoreError::EvictionFailed).
5. add_new_addrs() logs a debug message and discards the error.
6. The victim node can no longer learn about any new peers via discovery.

Unit test plan:
- Construct a PeerStore, fill it with 16384 AddrInfo entries
  (last_connected_at_ms=0, attempts_count=0) spread across >4096 /16 groups
  (≤4 per group).
- Call add_addr() with a new address.
- Assert the return value is Err(PeerStoreError::EvictionFailed).
```

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
