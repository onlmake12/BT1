### Title
Tx-Pool Minimum Fee Check Uses Serialized Size Instead of Weight, Allowing Cycle-Heavy Transactions to Bypass Fair Fee Accounting - (File: tx-pool/src/util.rs)

---

### Summary

The `check_tx_fee` function in `tx-pool/src/util.rs` computes the minimum required fee using only the transaction's serialized byte size (`tx_size`), while the actual block resource consumed by a transaction is determined by `weight = max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Because cycles are only known after script execution (which happens after the fee check), a transaction sender can submit a compact transaction whose script consumes nearly the entire block cycle budget while paying fees proportional only to its small byte footprint — a discount of up to ~2000×. No second fee check is performed after cycles are known.

---

### Finding Description

CKB blocks are constrained by two independent limits: serialized byte size (`MAX_BLOCK_BYTES = 597,000`) and execution cycles (`MAX_BLOCK_CYCLES = 3,500,000,000`). To unify these into a single fee-rate metric, the codebase defines:

```rust
// util/types/src/core/tx_pool.rs
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;

pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [1](#0-0) 

`TxEntry::fee_rate()` correctly uses this weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

However, the tx-pool admission gate `check_tx_fee` does **not** use weight — it uses raw byte size only:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [3](#0-2) 

The code comment itself acknowledges the discrepancy. The admission flow is:

1. `pre_check` → calls `check_tx_fee` (fee checked against `min_fee_rate × tx_size`) [4](#0-3) 
2. `verify_rtx` → executes scripts, actual cycles become known [5](#0-4) 
3. `TxEntry` is created with actual cycles — **no second fee check against weight is performed**

After admission, the block assembler and fee estimator correctly use `get_transaction_weight` to rank transactions by fee rate. [6](#0-5) 

The structural gap is: the gatekeeper check uses `tx_size`, but the resource actually consumed is `weight = max(tx_size, cycles × 0.000_170_571_4)`.

---

### Impact Explanation

**Concrete numbers:**

| Parameter | Value |
|---|---|
| `MAX_BLOCK_CYCLES` | 3,500,000,000 |
| `DEFAULT_BYTES_PER_CYCLES` | 0.000_170_571_4 |
| Cycle-equivalent bytes at max cycles | ~597,000 bytes (= `MAX_BLOCK_BYTES`) |
| Minimum tx serialized size | ~300 bytes |
| Fee check uses | 300 bytes |
| Actual weight consumed | 597,000 bytes |
| Fee discount factor | ~2,000× |

An attacker submits a transaction with a ~300-byte body containing a script that loops to consume ~3.5 billion cycles. The fee check passes because `min_fee = min_fee_rate × 300 / 1000`. The transaction is admitted and occupies the entire block cycle budget of a block, while paying fees as if it were a 300-byte transaction. Repeating this across multiple blocks allows the attacker to continuously crowd out legitimate transactions at a tiny fraction of the fair cost.

This is directly analogous to the stableswap finding: in that case, liquidity added at a skewed ratio does not pay fees for the skew it introduces; here, a transaction that consumes a skewed ratio of cycles-to-bytes does not pay fees for the cycle consumption it introduces.

---

### Likelihood Explanation

- **Entry path**: Any unprivileged RPC caller (`send_transaction`) or P2P relay peer can submit transactions. No special role is required.
- **Script availability**: A cycle-consuming script (e.g., a tight loop in CKB-VM) can be deployed by anyone as a cell and referenced as a cell dep.
- **Cost**: The attacker pays `min_fee_rate × tx_size` shannons per transaction — a negligible amount relative to the block cycle budget consumed.
- **Repeatability**: The attack can be sustained across many blocks with low ongoing cost.

---

### Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee check using the true weight:

```rust
// After cycles are known:
let weight = get_transaction_weight(tx_size, cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

Alternatively, require submitters to declare cycles upfront (as is already done for relayed transactions via the `declared_cycles` parameter) and use the declared value in the initial fee check, then verify the declaration matches actual execution.

---

### Proof of Concept

1. Deploy a CKB-VM script cell that executes a tight loop consuming `N` cycles (e.g., `N ≈ MAX_BLOCK_CYCLES`).
2. Construct a transaction:
   - One input cell (any live cell the attacker owns)
   - One output cell (change back to attacker)
   - Cell dep referencing the loop script
   - Lock script = the loop script
   - Serialized size ≈ 300 bytes
   - Fee = `min_fee_rate × 300 / 1000` shannons (e.g., 1 shannon at 1000 shannons/KW)
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` passes: `fee ≥ min_fee_rate × 300 / 1000`. [7](#0-6) 
5. `verify_rtx` executes the script; actual cycles ≈ `MAX_BLOCK_CYCLES`; actual weight ≈ 597,000. No second fee check occurs.
6. The transaction enters the pool. The block assembler packages it; it consumes the entire block cycle budget.
7. Repeat: the attacker can sustain this attack paying ~1 shannon per block while consuming the full cycle budget of every block, effectively pricing out all other cycle-heavy transactions.
