### Title
Unauthenticated `from` Field in `ConnectionRequestDelivered` Allows Any Connected Peer to Evict `inflight_requests` Entries and Suppress NAT Traversal — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

Any connected peer can send a crafted `ConnectionRequestDelivered` message with `from` set to the victim node's own `peer_id` and `route` set to empty. The handler unconditionally removes the matching entry from `inflight_requests` without verifying that the sender is the actual relay path. The legitimate response that arrives later finds no entry and is silently dropped, permanently suppressing NAT traversal for that peer pair until the next retry cycle.

---

### Finding Description

The `execute()` function routes a `ConnectionRequestDelivered` message through three branches based on `content.route.last()` and `content.from`: [1](#0-0) 

When `route` is empty and `content.from == local_peer_id`, the code treats the current node as the originator of the hole-punch request and immediately calls:

```rust
let request_start = self.protocol.inflight_requests.remove(&content.to);
``` [2](#0-1) 

There is **no verification** that the actual P2P session (`self.peer`) corresponds to the relay path that originally carried the `ConnectionRequest`. The `from` field is arbitrary bytes in the molecule-encoded message — it is never cryptographically bound to the sender's session identity.

The local node's `peer_id` is public (advertised during peer discovery). The contents of `inflight_requests` are also observable: the node broadcasts `ConnectionRequest` messages to `sqrt(total_peers)` connected peers via `filter_broadcast`, so any connected peer learns which `to` peer IDs are being targeted: [3](#0-2) 

**Attack steps:**

1. Attacker connects to the victim node as a normal P2P peer.
2. Attacker observes a `ConnectionRequest` broadcast, learning `to_peer_id` (the NAT traversal target).
3. Attacker sends a `ConnectionRequestDelivered` with `from = victim_peer_id`, `to = to_peer_id`, `route = []`, and any syntactically valid `listen_addrs`.
4. The handler removes `to_peer_id` from `inflight_requests` at line 160.
5. The legitimate `ConnectionRequestDelivered` arrives later, finds no entry, and returns `StatusCode::Ignore` at line 175 — `try_nat_traversal` is never called. [4](#0-3) 

The `forward_rate_limiter` (keyed on `(from, to, item_id)`) limits to 1 message/second per pair, but the attacker only needs **one** message per retry cycle. The node retries every `CHECK_INTERVAL` (5 minutes): [5](#0-4) 

So the attacker sustains the suppression with ≤1 spoofed message per 5-minute window per target pair — well within the rate limit.

---

### Impact Explanation

NAT traversal via hole-punching is permanently suppressed for any targeted peer pair as long as the attacker remains connected. Nodes behind NAT that rely on this mechanism to establish outbound connections will fail to connect to the targeted peers. The impact is scoped to the hole-punching subsystem and does not affect already-established connections or consensus. This matches **Low (501–2000)** — targeted, persistent disruption of a specific connectivity feature.

---

### Likelihood Explanation

The attack requires only a standard P2P connection (no special privileges, no key material). The victim's `peer_id` and the `to` peer IDs in `inflight_requests` are both observable by any connected peer. The spoofed message passes all existing validation (rate limiter, field format checks, route length checks). The attack is repeatable indefinitely at negligible cost.

---

### Recommendation

Bind the `from` field to the actual session identity. Before reaching the `inflight_requests.remove` branch, verify that `self.peer` (the actual sender's `PeerIndex`) corresponds to a peer whose `peer_id` is on the expected relay path for the `(from, to)` pair. Alternatively, record the relay session ID alongside the timestamp in `inflight_requests` and reject any `ConnectionRequestDelivered` that does not arrive on that session.

---

### Proof of Concept

```
1. Insert (target_peer_id → timestamp) into inflight_requests (simulating a legitimate outbound ConnectionRequest).
2. Construct a ConnectionRequestDelivered with:
     from = local_peer_id
     to   = target_peer_id
     route = []
     listen_addrs = [any valid TCP multiaddr with target_peer_id appended]
3. Deliver it from an attacker-controlled session.
4. Assert inflight_requests no longer contains target_peer_id.
5. Deliver the legitimate ConnectionRequestDelivered for the same to field.
6. Assert try_nat_traversal is never invoked (StatusCode::Ignore is returned).
```
