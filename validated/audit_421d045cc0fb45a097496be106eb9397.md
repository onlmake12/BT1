### Title
Integer Overflow in Limit Check Enables Unbounded Vec Allocation via Crafted `GetLastStateProof` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The guard at line 201 that is meant to cap the total number of samples in a `GetLastStateProof` request uses unchecked `usize` arithmetic. In a release build, sending `last_n_blocks = u64::MAX` with exactly 2 difficulty entries causes the expression to wrap to `0`, silently passing the `> 1000` check. The code then unconditionally takes the "not enough blocks" branch and calls `.collect::<Vec<_>>()` over the entire chain height, causing an OOM crash on the serving node.

---

### Finding Description

**Overflow in the limit check (line 201):**

```rust
// util/light-client-protocol-server/src/components/get_last_state_proof.rs
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

On a 64-bit target in release mode, Rust integer overflow wraps silently:

- `u64::MAX as usize` = `usize::MAX` = `2^64 − 1`
- `usize::MAX * 2` wraps to `2^64 − 2` (`usize::MAX − 1`)
- `2 + (usize::MAX − 1)` = `usize::MAX + 1` wraps to `0`
- `0 > 1000` → **false** → the guard is bypassed

The constant is defined as:

```rust
pub const GET_LAST_STATE_PROOF_LIMIT: usize = 1000;
``` [2](#0-1) 

**Unbounded allocation (lines 291–296):**

After the guard is bypassed, the condition:

```rust
if last_block_number - start_block_number <= last_n_blocks  // N-1 <= u64::MAX, always true
```

is always satisfied, so execution takes the "not enough blocks" path:

```rust
let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [3](#0-2) 

With `start_block_number = 1` and a chain of height `N`, this allocates a `Vec` of `N − 1` `u64` entries — 8 bytes each. At 10 million blocks that is ~80 MB per request; at 100 million blocks it is ~800 MB. Multiple concurrent requests trivially exhaust available memory.

**Reorg path (lines 237–247) is not the amplifier:**

With `start_block_number = 1` and a mismatched `start_hash`, `min(1, u64::MAX) = 1`, so `reorg_last_n_numbers = (0..1)` — only one element. The amplification comes entirely from `last_n_numbers`. [4](#0-3) 

**Crafting a message that passes all other checks:**

| Field | Value | Why it passes |
|---|---|---|
| `last_hash` | current tip hash | passes `is_main_chain` check (line 210) |
| `start_number` | 1 | ≤ `last_block_number` (line 231) |
| `start_hash` | any wrong hash | triggers reorg path, but only 1 entry |
| `last_n_blocks` | `u64::MAX` | causes overflow |
| `difficulties` | `[genesis_td+1, genesis_td+2]` | monotonically increasing, < `difficulty_boundary` |
| `difficulty_boundary` | `genesis_td+3` | > max difficulty |

The first-difficulty check at line 268 requires `difficulties[0] > total_difficulty(block 0)`. Genesis total difficulty is public; the attacker sets `difficulties[0] = genesis_td + 1`. This passes. The "not enough blocks" branch at line 291 never uses `difficulties` at all, so the sampled-numbers path (lines 299–347) is never reached. [5](#0-4) 

---

### Impact Explanation

Any peer with light-client protocol access (no authentication required) can crash the full node process with an OOM kill by sending a single malformed `GetLastStateProof` message on a chain with a large block height. The crash is deterministic and repeatable.

---

### Likelihood Explanation

The light-client protocol is enabled by configuration and exposed over P2P. The exploit requires no keys, no PoW, no privileged role — only the ability to open a P2P connection and send a single crafted message. The overflow is silent in release builds (the standard deployment mode).

---

### Recommendation

Replace the unchecked arithmetic with a saturating or checked variant:

```rust
// Option A: saturating_add / saturating_mul
if (last_n_blocks as usize)
    .saturating_mul(2)
    .saturating_add(self.message.difficulties().len())
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: reject immediately if last_n_blocks > limit
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, the "not enough blocks" branch at line 296 should independently cap `last_n_numbers` to `GET_LAST_STATE_PROOF_LIMIT` entries regardless of `last_n_blocks`.

### Proof of Concept

```rust
// Pseudocode unit test
let genesis_td: U256 = snapshot.get_block_total_difficulty(0).unwrap();
let msg = GetLastStateProof {
    last_hash:          snapshot.tip_hash(),
    start_number:       1,
    start_hash:         H256::zero(),          // wrong hash → reorg path
    last_n_blocks:      u64::MAX,
    difficulty_boundary: genesis_td + 3,
    difficulties:       vec![genesis_td + 1, genesis_td + 2],  // len=2 → overflow wraps to 0
};
// On a chain of 10_000 blocks, execute() allocates Vec of 9_999 u64s.
// On mainnet (~15M blocks), allocates ~120 MB per request.
// Sending ~10 concurrent requests → OOM / process kill.
let status = process.execute().await;
assert_ne!(status.code(), StatusCode::MalformedProtocolMessage); // guard was bypassed
```
