### Title
Inbound Peer Can Force Unverified Address Into Network-Wide Discovery Broadcast — (`network/src/protocols/discovery/mod.rs`, `state.rs`)

### Summary

An unprivileged inbound peer can send a single `GetNodes{version: 1, listen_port: Some(port)}` message to cause the victim node to promote its `RemoteAddress::Init` to `RemoteAddress::Listen`, making the inbound peer's address eligible for network-wide announcement via `notify()` — without any verification that the address is publicly reachable. The identify protocol's own code contains a comment explicitly acknowledging this risk for inbound addresses, yet the discovery protocol's `received()` handler does not enforce the same protection.

**The smoking-gun comment in the identify protocol**

The developers explicitly recognized this risk for inbound addresses in `identify/mod.rs`:

```rust
// why don't set inbound here?
// because inbound address can't feeler during staying connected
// and if set it to peer store, it will be broadcast to the entire network,
// but this is an unverified address
``` [8](#0-7) 

The identify protocol deliberately avoids adding inbound addresses to the peer store for exactly this reason. The discovery protocol's `received()` handler does not apply the same guard.

---

### Impact Explanation

An attacker with a publicly routable IP establishes an inbound connection, completes the identify handshake (normal protocol flow), and sends one `GetNodes{version: 1, listen_port: Some(port)}`. The victim node promotes the inbound peer's address (attacker's real IP, attacker-chosen port) to `RemoteAddress::Listen`. After `ANNOUNCE_INTERVAL` (24 hours), `notify()` broadcasts this address to up to 3 other peers per tick. Those peers store and may further propagate the address. The injected address is never verified as reachable.

**Scope of injection**: The IP is constrained to the attacker's actual TCP connection IP (cannot be spoofed). The port is fully attacker-controlled. The claim of "arbitrary addresses" in the question is slightly overstated — the IP is real, but the port is arbitrary and the address is unverified.

**Practical impact**: The attacker can ensure other nodes attempt connections to them on a specific port, and can pollute the peer store of multiple nodes with unverified entries. At scale (many inbound connections across many victim nodes), this degrades peer discovery quality and can contribute to eclipse-attack infrastructure.

---

### Likelihood Explanation

- Requires only a standard inbound TCP connection — no special privileges.
- The identify protocol completes automatically as part of normal connection setup.
- `GetNodes` is the first message sent by any outbound peer; an inbound peer sending it is unusual but not blocked (only `DuplicateGetNodes` is checked, not direction).
- The 24-hour `ANNOUNCE_INTERVAL` means the attacker must maintain the connection, but this is trivially achievable.

---

### Recommendation

In `received()`, gate the `update_port()` / `change_to_listen()` calls on the session being **outbound**:

```rust
// Only promote to Listen for outbound sessions (address is verified by connection)
if context.session.ty.is_outbound() {
    if let Some(port) = listen_port {
        state.remote_addr.update_port(port);
        ...
    }
    if version >= state::REUSE_PORT_VERSION {
        state.remote_addr.change_to_listen();
    }
}
```

This mirrors the existing guard in the identify protocol and enforces the invariant that only outbound-confirmed addresses are eligible for network-wide announcement.

### Proof of Concept

1. Attacker node (publicly routable IP `A.B.C.D`) establishes inbound connection to victim.
2. Identify protocol completes normally — `identify_info` is set on the victim for this session.
3. Attacker sends `DiscoveryMessage::GetNodes { version: 1, count: 1000, listen_port: Some(9999), required_flags: COMPATIBILITY }`.
4. Victim's `state.remote_addr` transitions from `Init(A.B.C.D:ephemeral)` to `Listen(A.B.C.D:9999)`.
5. Advance mock clock past `ANNOUNCE_INTERVAL` (24h).
6. Call `notify()` on the victim's `DiscoveryProtocol`.
7. Assert: `A.B.C.D:9999` appears in `announce_multiaddrs` of up to 3 other connected sessions.
8. Assert: `A.B.C.D:9999` was never verified as reachable by the victim.
