### Title
Excess ETH Not Refunded to Caller in Entropy `request`/`requestWithCallback`/`requestV2` Functions - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

In `Entropy.sol`, the internal `requestHelper` function accepts `msg.value` and only checks that it is not less than `requiredFee`. Any excess ETH above `requiredFee` is silently absorbed into `_state.accruedPythFeesInWei` and is never returned to the caller. This affects all three public entry points: `request`, `requestWithCallback`, and `requestV2`.

### Finding Description

`requestHelper` computes the required fee and validates only the lower bound:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `_state.accruedPythFeesInWei`. If `msg.value > requiredFee`, the surplus is permanently captured by the protocol rather than returned to `msg.sender`. There is no refund path anywhere in `request`, `requestWithCallback`, or `requestV2`. [2](#0-1) [3](#0-2) 

The same root cause exists in `Echo.sol`'s `requestPriceUpdatesWithCallback`: excess ETH above `requiredFee` is stored in `req.fee` and later credited entirely to the provider's `accruedFeesInWei`, meaning the provider profits from user overpayment. [4](#0-3) [5](#0-4) 

### Impact Explanation

Any user who sends `msg.value` greater than `getFeeV2(provider, gasLimit)` permanently loses the excess ETH. The surplus is absorbed into the Pyth protocol fee pool (Entropy) or credited to the fulfilling provider (Echo). The user has no mechanism to recover it. This is a direct, irreversible loss of user funds with no compensating benefit.

### Likelihood Explanation

Fee amounts are dynamic (provider fees can change between the time a user queries `getFeeV2` and the time the transaction is mined). Users and integrating contracts commonly add a small buffer to `msg.value` to avoid `InsufficientFee` reverts under fee volatility. Every such overpayment results in a permanent loss. The `IEntropy` interface itself warns "excess value is *not* refunded to the caller," confirming this is a known, reachable condition. [6](#0-5) [7](#0-6) 

### Recommendation

After crediting the exact `requiredFee` to the protocol and provider balances, refund any surplus to `msg.sender`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
// refund excess
if (msg.value > requiredFee) {
    (bool ok, ) = msg.sender.call{value: msg.value - requiredFee}("");
    require(ok, "refund failed");
}
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee);
```

Alternatively, enforce `msg.value == requiredFee` with an exact-match check. Apply the same fix to `Echo.sol`'s `requestPriceUpdatesWithCallback`.

### Proof of Concept

1. Provider registers with fee = 1000 wei. `getFeeV2(provider, 0)` returns 1000 wei.
2. Between the user's fee query and transaction inclusion, the provider raises their fee to 1001 wei. The user's transaction still carries `msg.value = 1000 wei` ‚Äî but suppose the user defensively sent `msg.value = 1500 wei` to avoid a revert.
3. `requestHelper` passes the `< requiredFee` check (1500 >= 1001).
4. `_state.accruedPythFeesInWei += (1500 - providerFee)` ‚Äî the 499 wei surplus is absorbed by the protocol.
5. The user receives no refund and has permanently lost 499 wei with no recourse.

The same scenario applies to `Echo.sol`: the 499 wei surplus is stored in `req.fee` and later paid out to the fulfilling provider, not the user.

    function requestV2(
        address provider,
        bytes32 userContribution,
        uint32 gasLimit
    ) public payable override returns (uint64) {
        EntropyStructsV2.Request storage req = requestHelper(
            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed
            // random number. Hence, useBlockHash is set to false.
            false,
            true,
            gasLimit
        );

        emit RequestedWithCallback(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );
        emit EntropyEventsV2.Requested(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }
