Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Overwrite After Cell-Ref Eviction Inflates Pool Accounting — (File: tx-pool/src/component/pool_map.rs)

## Summary
`PoolMap::add_entry` captures pre-eviction pool totals into locals via `updated_stat_for_add_tx`, then calls `check_and_record_ancestors`, which can evict cell-ref-parent transactions and correctly decrements `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`. Lines 218–219 then unconditionally overwrite those fields with the stale pre-eviction locals, permanently discarding every decrement made during eviction. Each triggering submission inflates the pool's accounting by the combined size of all evicted entries, eventually causing `limit_size` to evict legitimate transactions and `updated_stat_for_add_tx` to reject new submissions as `Reject::Full` despite real capacity being available.

## Finding Description
**Root cause — `add_entry` (lines 200–221):**

Step 1 (lines 210–211): `updated_stat_for_add_tx` takes `&self` (immutable borrow) and returns `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` as locals. It does not mutate `self`. [1](#0-0) 

Step 2 (line 213): `check_and_record_ancestors` enters the eviction branch when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= max_ancestors_count` but `ancestors_count > max_ancestors_count`. It calls `remove_entry_and_descendants` for each cell-ref parent to be evicted. [2](#0-1) 

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` and `self.total_tx_cycles` for every removed entry. [3](#0-2) [4](#0-3) 

Step 3 (lines 218–219): After `check_and_record_ancestors` returns with `self.total_tx_size` correctly at `S − E`, the assignment restores it to `S + new_size` (the stale pre-eviction snapshot). The net inflation is `E` bytes and `E_cycles` cycles per triggering submission, and it accumulates across repeated submissions. [5](#0-4) 

**Why existing checks are insufficient:** `updated_stat_for_add_tx` does check for overflow and returns `Reject::Full` if the pre-eviction sum would overflow, but it cannot account for evictions that have not yet occurred. There is no post-eviction reconciliation step anywhere in `add_entry`. [6](#0-5) 

## Impact Explanation
`limit_size` drives all pool-size eviction decisions solely from `self.pool_map.total_tx_size`: [7](#0-6) 

An inflated counter causes `limit_size` to evict fee-paying transactions that should remain, and causes `updated_stat_for_add_tx` to return `Reject::Full` for new submissions even when the pool is physically under-capacity. The pool's effective capacity shrinks monotonically with each triggering submission. This constitutes a sustained, remotely-triggerable tx-pool denial-of-service that degrades transaction relay and mempool throughput across targeted nodes.

**Severity: High** — matches "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation
The eviction path in `check_and_record_ancestors` is reachable by any caller of the public `send_transaction` RPC with no special privileges, key material, or hashpower. The attacker submits a batch of transactions sharing a cell dep (making them cell-ref parents), then submits one transaction that consumes that cell dep as an input. Because the cell-ref-parent transactions are evicted from the pool and never confirmed, the attacker does not pay their fees. The attack is repeatable with fresh cell deps, and each iteration inflates `total_tx_size` by the combined size of the evicted entries. The existing FIXME comment at line 583 confirms developer awareness that `check_and_record_ancestors` performs irreversible mutations before all checks pass, but the accounting overwrite on lines 218–219 is a separate, unacknowledged issue. [8](#0-7) 

## Recommendation
Move the stat update to **after** all mutations (including evictions) have completed, so it reflects the true post-eviction state:

```rust
// Replace lines 218-219 with:
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .expect("already checked above");
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .expect("already checked above");
```

Alternatively, subtract evicted totals from the pre-computed locals before assigning:

```rust
let evict_size: usize   = evicts.iter().map(|e| e.size).sum();
let evict_cycles: Cycle = evicts.iter().map(|e| e.cycles).sum();
self.total_tx_size   = total_tx_size   - evict_size;
self.total_tx_cycles = total_tx_cycles - evict_cycles;
```

Both approaches eliminate the stale-overwrite. The first is simpler and less error-prone.

## Proof of Concept
**Setup:** `max_tx_pool_size = 200_000 bytes`, `max_ancestors_count = 1000`.

1. Submit root transaction `T0` whose output cell is used as a cell dep.
2. Submit 1001 transactions `T1…T1001`, each referencing `T0`'s output as a cell dep. All are accepted; `self.total_tx_size ≈ S`.
3. Submit `T_consume` spending `T0`'s output as an **input**. Inside `add_entry`:
   - `updated_stat_for_add_tx` snapshots `total_tx_size_local = S + size(T_consume)`.
   - `check_and_record_ancestors` detects `ancestors_count = 1002 > 1000`; `cell_ref_parents = {T1…T1001}`. It evicts the 2 lowest-fee entries (total size `E`), calling `update_stat_for_remove_tx` twice → `self.total_tx_size = S − E`.
   - Line 218 assigns `self.total_tx_size = S + size(T_consume)`, discarding `−E`.
   - Net inflation: `E` bytes.
4. Repeat steps 1–3 with fresh cell deps. Each iteration inflates `total_tx_size` by `E`.
5. After enough iterations, `total_tx_size > max_tx_pool_size` even though the pool holds far fewer bytes. `limit_size` begins evicting legitimate transactions; `updated_stat_for_add_tx` rejects all new submissions with `Reject::Full`.

A unit test can confirm this by asserting `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum::<usize>()` after each `add_entry` call that triggers the eviction branch, and observing the invariant violation. [9](#0-8)

        self.total_tx_size = total_tx_size;

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }
