### Title
Unhandled Callback Revert in Legacy `revealWithCallback` Path Permanently Locks Entropy Requests — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
When an Entropy provider has `defaultGasLimit == 0` (the initial/legacy state), any `requestWithCallback` call stores `gasLimit10k = 0` on the request. During `revealWithCallback`, the contract takes a legacy code path that invokes the consumer's `_entropyCallback` directly — with no error catching. If the callback reverts, the entire `revealWithCallback` transaction reverts (including the `clearRequest` side-effect), leaving the request permanently active but unfulfillable. An unprivileged Entropy user can exploit this to permanently grief the provider's Fortuna keeper service.

---

### Finding Description

In `requestHelper`, when `providerInfo.defaultGasLimit == 0`, the request is stored with `req.gasLimit10k = 0`: [1](#0-0) 

In `revealWithCallback`, the branch condition `req.gasLimit10k != 0 && req.callbackStatus == CALLBACK_NOT_STARTED` selects the **safe new path** (using `excessivelySafeCall` with error catching). When `gasLimit10k == 0`, the **else/legacy path** is taken instead: [2](#0-1) 

In this legacy path:
1. `clearRequest` is called first (CEI pattern).
2. The callback is invoked **directly** with no `try/catch` or `excessivelySafeCall`.
3. If the callback reverts, the **entire transaction reverts**, undoing `clearRequest`.
4. The request remains active with `callbackStatus == CALLBACK_NOT_STARTED`.

The new safe path (lines 574–660) uses `excessivelySafeCall` and transitions the request to `CALLBACK_FAILED` on revert, enabling recovery. The legacy path has no such mechanism: [3](#0-2) 

The `IEntropyConsumer._entropyCallback` wrapper itself can revert if the inner `entropyCallback` reverts: [4](#0-3) 

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` confirms this behavior — `revealWithCallback` reverts when the callback reverts and `gasLimit10k == 0`: [5](#0-4) 

---

### Impact Explanation

- **Permanently stuck request**: The entropy request can never be cleared. The user's paid fee is unrecoverable.
- **Provider keeper griefing**: The Fortuna service (`apps/fortuna`) will repeatedly attempt to call `revealWithCallback` for the stuck request. Each attempt costs the provider gas and always reverts. An attacker paying a single request fee can force unbounded gas expenditure by the provider's keeper.
- **No recovery path**: Unlike the new `CALLBACK_FAILED` state flow, the legacy path provides no mechanism to recover or skip a permanently reverting request.

**Impact: High** — permanent DoS of individual entropy requests and griefing of the provider's operational costs.

---

### Likelihood Explanation

- Any provider whose `defaultGasLimit` is still `0` (the initial state after `register()`) is vulnerable. Providers must explicitly call `setDefaultGasLimit` to opt into the safe path.
- An attacker only needs to: (1) identify a provider with `defaultGasLimit == 0`, (2) deploy a consumer contract whose `entropyCallback` always reverts, and (3) call `requestWithCallback`. No privileged access is required.
- The `requestWithCallback` function is publicly callable by any unprivileged Entropy user. [6](#0-5) 

**Likelihood: Medium** — depends on whether the live Fortuna provider has set a non-zero `defaultGasLimit`. If it has not, likelihood is High.

---

### Recommendation

1. In the legacy path (`gasLimit10k == 0`), wrap the direct callback invocation in a low-level `call` (or `excessivelySafeCall`) and handle the revert gracefully — either by transitioning to a `CALLBACK_FAILED` state or by emitting a `CallbackFailed` event and clearing the request.
2. Alternatively, require all new providers to set a non-zero `defaultGasLimit` during `register()`, preventing new requests from ever landing in the legacy path.
3. The Fortuna keeper should detect permanently reverting requests and stop retrying them after a configurable number of failures.

### Proof of Concept

```solidity
// Attacker's consumer contract
contract MaliciousConsumer is IEntropyConsumer {
    IEntropy entropy;
    constructor(address _entropy) { entropy = IEntropy(_entropy); }

    function attack(address provider) external payable {
        uint128 fee = entropy.getFee(provider);
        // provider must have defaultGasLimit == 0
        entropy.requestWithCallback{value: fee}(provider, bytes32(uint256(1)));
    }

    function entropyCallback(uint64, address, bytes32) internal override {
        revert("always reverts"); // callback always reverts
    }

    function getEntropy() internal view override returns (address) {
        return address(entropy);
    }
}

// Steps:
// 1. Deploy MaliciousConsumer pointing at the Entropy contract
// 2. Call attack(provider) where provider.defaultGasLimit == 0
// 3. Provider's Fortuna calls revealWithCallback -> always reverts
// 4. Request is permanently stuck; provider wastes gas on every retry
```

The request remains in `CALLBACK_NOT_STARTED` state indefinitely. The provider's Fortuna keeper has no on-chain mechanism to skip or clear it. [7](#0-6)
