### Title
No-Staleness Check on Lazer Payload Timestamp Enables Front-Running of Price State Transitions — (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate()` validates only the ECDSA signature and signer expiry. It does **not** validate the timestamp embedded in the payload against `block.timestamp`. Any caller can replay an arbitrarily old (but still validly-signed) Lazer payload to a consumer contract, presenting a stale price as current. This is the Pyth-native analog of the `SafeguardPool` front-running issue: a state transition (signer rotation / expiry boundary) creates a window where an attacker can front-run with a payload whose embedded price diverges from the real market price.

---

### Finding Description

`PythLazer.verifyUpdate()` performs two checks:

1. Signature recovery via `ECDSA.tryRecover` over `keccak256(payload)`.
2. `isValidSigner(signer)` — checks `block.timestamp < trustedSignerToExpiresAtMapping[signer]`. [1](#0-0) 

The payload itself contains a `uint64 timestamp` field (parsed by `PythLazerLib.parsePayloadHeader`): [2](#0-1) 

`verifyUpdate()` returns the raw `payload` bytes to the caller without ever comparing the payload's embedded timestamp to `block.timestamp`. There is no maximum-age check, no minimum-age check, and no replay protection on the timestamp.

The official EVM integration guide explicitly delegates this responsibility to the consumer:

> "Make sure to check `_timestamp > timestamp`" (shown in the example snippet) [3](#0-2) 

The Cardano integration guide makes the omission explicit at the protocol level:

> "It does not enforce freshness of the update, nor does it disallow verifying the same update multiple times." [4](#0-3) 

The analogous `SafeguardPool` vulnerability class is: a binary state transition (peg flag flip / signer expiry boundary) creates a guaranteed-profit front-run window. Here the state transition is the **signer expiry event**: just before a signer's `expiresAt` is reached, an attacker can observe the pending expiry on-chain, then front-run the governance transaction that rotates the signer by replaying an old payload signed by the about-to-expire key. Because `isValidSigner` only checks `block.timestamp < expiresAt`, the old payload passes verification right up to the expiry second. [5](#0-4) 

---

### Impact Explanation

A consumer contract that stores a Lazer price and uses it for swaps, liquidations, or collateral valuation can be fed an arbitrarily old price that was validly signed at the time of signing. If the real market price has moved significantly since that old payload was produced, the attacker profits by:

1. Submitting the stale payload to the consumer (passes `verifyUpdate` because the signer is still valid).
2. Executing a trade/liquidation at the stale price.
3. Closing the position at the real current price.

The impact is direct loss of funds from any consumer contract that does not independently enforce a freshness window — which the `PythLazer.sol` contract itself does not require or enforce.

---

### Likelihood Explanation

- The entry path is fully permissionless: any EOA can call `verifyUpdate` with any bytes payload.
- Old signed payloads are publicly observable (WebSocket stream history, mempool, or any node that recorded them).
- Signer rotation events are predictable from on-chain `expiresAt` values, giving attackers advance notice.
- Consumer contracts that follow the official EVM integration example only check `_timestamp > timestamp` (a monotonicity guard against replay within the same contract session), not a wall-clock freshness bound — making the attack viable against a large fraction of real integrations. [6](#0-5) 

---

### Recommendation

Add an on-chain maximum-age check inside `verifyUpdate()` after the payload is parsed:

```solidity
uint64 payloadTimestamp = uint64(bytes8(payload[4:12])); // after FORMAT_MAGIC
require(
    block.timestamp <= payloadTimestamp / 1e6 + MAX_PAYLOAD_AGE_SECONDS,
    "payload too old"
);
```

Alternatively, expose a `verifyUpdateWithMaxAge(bytes calldata update, uint64 maxAgeSeconds)` variant so consumers can opt into on-chain freshness enforcement without changing the base interface. The Cardano documentation warning should be elevated to a protocol-level guarantee rather than a consumer responsibility.

---

### Proof of Concept

1. Attacker subscribes to the Pyth Lazer WebSocket and records a signed payload `P_old` for feed `ETH/USD` at time `T_old` when price was `$3000`.
2. At time `T_now` (hours later), real ETH price is `$2800`. The signer key used to sign `P_old` is still valid (`block.timestamp < expiresAt`).
3. Attacker calls `pythLazer.verifyUpdate{value: fee}(P_old)` — succeeds because:
   - Signature over `keccak256(payload)` is valid.
   - `isValidSigner(signer)` returns `true`.
   - No timestamp check exists.
4. Consumer contract receives `payload` with embedded timestamp `T_old` and price `$3000`.
5. Consumer's only guard is `if (feedId == X && _timestamp > storedTimestamp)` — if `T_old > storedTimestamp` (e.g., first update of the session), the stale price is accepted.
6. Attacker executes a trade at the inflated `$3000` price, then closes at real `$2800`, extracting `$200/ETH` profit from the consumer's liquidity. [1](#0-0) [2](#0-1)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
