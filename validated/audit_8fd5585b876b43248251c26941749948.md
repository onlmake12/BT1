Audit Report

## Title
Unbounded `pending_get_block_proposals` Cache Enables Memory/CPU DoS via Repeated `GetBlockProposal` Relay Messages — (File: sync/src/types/mod.rs)

## Summary
`SyncState::pending_get_block_proposals` is a plain, unbounded `DashMap` with no size cap, no per-peer quota, and no eviction policy. Any peer that completes the CKB handshake can flood the node with `GetBlockProposal` relay messages containing unique fake `ProposalShortId` values, causing the map to grow without bound. The periodic `prune_tx_proposal_request` timer then performs O(N) memory and CPU work over the entire accumulated map, enabling a sustained node-level DoS or OOM crash.

## Finding Description
**Unbounded field declaration** — `sync/src/types/mod.rs` line 1330: [1](#0-0) 

The field is a plain `DashMap` with no declared capacity limit, no LRU eviction, and no per-peer accounting. Contrast with the adjacent `pending_get_headers` field which uses a bounded `LruCache`. [2](#0-1) 

**Unconditional insertion** — `insert_get_block_proposals` at lines 1594–1601: [3](#0-2) 

Every ID in the supplied vector is inserted via `entry(...).or_default().insert(pi)` with no size guard, no per-peer quota check, and no deduplication limit.

**Attacker-controlled entry path** — `GetBlockProposalProcess::execute()` at lines 35–44: [4](#0-3) 

The only guard is a per-message count check against `max_block_proposals_limit × max_uncles_num` (default ~3,000). There is no rate limit on how many such messages a peer may send, and no cap on the total accumulated map size.

Proposals absent from the tx pool are unconditionally inserted at lines 68–77: [5](#0-4) 

Random 10-byte `ProposalShortId` values will never match any tx-pool entry, so every ID passes the filter and is inserted into the unbounded map.

**O(N) periodic drain** — `prune_tx_proposal_request` at lines 549–560: [6](#0-5) 

And the subsequent iteration at lines 572–580: [7](#0-6) 

`drain_get_block_proposals` clones the entire map (O(N) allocation), then `fetch_txs` is called with all N keys (large tx-pool channel message), and the full map is iterated. All three operations are O(N) where N is attacker-controlled.

**`drain_get_block_proposals` clone-then-clear pattern** — lines 1586–1592: [8](#0-7) 

The full accumulated map is materialized in memory on every timer tick, meaning memory pressure spikes periodically rather than being amortized.

No rate limiting, per-peer quota, or map size cap exists anywhere in the relay handler code path.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

- **Memory exhaustion**: At 3,000 entries per message and 10,000 messages, the map holds 30 million entries. Each `ProposalShortId` is 10 bytes plus a `HashSet<PeerIndex>` with overhead; at scale this consumes several gigabytes of heap, causing an OOM crash.
- **CPU exhaustion**: When `prune_tx_proposal_request` fires, it clones N entries, serializes all N keys into a `Vec` for the tx-pool actor, and iterates the entire map. With millions of entries this stalls the async relay task for seconds, blocking all relay processing for every connected peer.
- **Tx-pool saturation**: The `fetch_txs` call is an async channel message to the tx-pool service. A massive payload can saturate the tx-pool's message queue, degrading transaction admission globally.

## Likelihood Explanation
Any unprivileged peer that completes the standard CKB P2P handshake can trigger this. No authentication, stake, or special role is required. A single TCP connection suffices. Generating random 10-byte `ProposalShortId` values costs negligible CPU on the attacker side. The per-message cap of ~3,000 IDs is not a meaningful barrier — sending thousands of messages per second over a single connection is trivial with standard networking. The attack is repeatable, cheap, and requires no victim mistakes.

## Recommendation
1. **Cap `pending_get_block_proposals`**: Enforce a maximum total size (e.g., `max_block_proposals_limit × max_uncles_num × max_connected_peers`) and drop new insertions when the cap is reached.
2. **Per-peer rate limiting**: Track how many proposals each peer has contributed and reject further insertions once a per-peer quota is exceeded within a time window.
3. **Replace with a bounded LRU cache**: Use a bounded `LruCache<ProposalShortId, HashSet<PeerIndex>>` (consistent with how `pending_get_headers` is handled at line 1331) so the map self-limits without explicit eviction logic.
4. **Validate proposal IDs against known blocks**: Only cache proposals that correspond to a block hash the node has recently seen, reducing the attack surface for fake IDs.

## Proof of Concept
```
1. Establish a peer connection to the target CKB node (standard P2P handshake).

2. In a tight loop, send GetBlockProposal relay messages:
   - Each message contains 3,000 unique random ProposalShortId values
     (10 random bytes each, guaranteed not to exist in any tx pool).
   - The per-message limit check passes (3,000 ≤ max_block_proposals_limit × max_uncles_num).

3. GetBlockProposalProcess::execute() calls tx_pool.fetch_txs() for each message.
   All 3,000 IDs are absent → all 3,000 are passed to insert_get_block_proposals().

4. After N messages, pending_get_block_proposals holds N × 3,000 entries with no eviction.

5. When prune_tx_proposal_request fires (periodic timer):
   - drain_get_block_proposals() clones and clears the map (O(N×3000) allocation).
   - fetch_txs() is called with N×3000 keys (large tx-pool message).
   - The for loop iterates all N×3000 entries (O(N×3000) CPU).

6. At N = 10,000 messages: 30,000,000 map entries → multi-GB heap → OOM crash,
   or multi-second relay stall → effective DoS for all connected peers.

Unit test: spawn a SyncState, call insert_get_block_proposals() in a loop with
unique random IDs until the map exceeds a threshold, then call drain_get_block_proposals()
and measure allocation size and iteration time to confirm O(N) behavior.
```
