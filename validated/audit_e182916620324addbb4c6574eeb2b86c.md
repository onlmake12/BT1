### Title
`verification_fee` Cannot Be Updated After Initialization — (`File: lazer/contracts/evm/src/PythLazer.sol`)

### Summary
The `PythLazer` EVM contract sets `verification_fee` to `1 wei` during `initialize()` and provides no setter function to update it. The fee is permanently locked at `1 wei` for the lifetime of the proxy unless a full UUPS implementation upgrade is performed. This is a direct analog to the Gearbox `priceFeeds` immutability issue: a critical protocol configuration value is set once and cannot be administratively changed.

### Finding Description
In `lazer/contracts/evm/src/PythLazer.sol`, the `initialize()` function sets `verification_fee = 1 wei` and the contract exposes no `setVerificationFee()` or equivalent function:

```solidity
function initialize(address _topAuthority) public initializer {
    __Ownable_init(_topAuthority);
    __UUPSUpgradeable_init();
    verification_fee = 1 wei;   // hardcoded, no setter exists
}
```

The only functions defined in the contract are `updateTrustedSigner()`, `isValidSigner()`, `verifyUpdate()`, and `version()`. None of them allow the owner to update `verification_fee`.

The `verifyUpdate()` function enforces this fee on every caller:

```solidity
function verifyUpdate(bytes calldata update)
    external payable returns (bytes calldata payload, address signer)
{
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);
    }
    ...
}
```

Because `verification_fee` is `public` storage and `1 wei` is effectively zero cost, any caller can invoke `verifyUpdate()` at negligible cost indefinitely. The owner has no administrative path to raise the fee without deploying a new implementation contract.

### Impact Explanation
- The protocol cannot adjust `verification_fee` to respond to changing conditions (gas price spikes, spam, revenue requirements).
- `verifyUpdate()` is callable by any address for `1 wei` — essentially free — enabling high-volume spam or DoS of the verification endpoint at negligible cost.
- Accrued fees from `verifyUpdate()` are permanently stuck at `1 wei` per call, making the fee mechanism economically meaningless.
- Changing the fee requires a full UUPS implementation upgrade, which is a heavyweight governance operation not designed for routine parameter tuning.

### Likelihood Explanation
The condition is always active from the moment of deployment. Any unprivileged Lazer consumer or attacker can call `verifyUpdate()` at `1 wei` per call. There is no time window or precondition required; the missing setter is a permanent protocol limitation.

### Recommendation
Add an owner-restricted setter for `verification_fee`, analogous to the existing `updateTrustedSigner()` pattern:

```solidity
function setVerificationFee(uint256 newFee) external onlyOwner {
    uint256 oldFee = verification_fee;
    verification_fee = newFee;
    emit VerificationFeeUpdated(oldFee, newFee);
}
```

### Proof of Concept
1. Deploy `PythLazer` proxy with any `_topAuthority`.
2. Observe `verification_fee == 1 wei` via the public getter.
3. Attempt to call any setter for `verification_fee` as the owner — no such function exists; the call reverts.
4. Call `verifyUpdate{value: 1}(validUpdate)` from any EOA — succeeds at `1 wei` cost.
5. Repeat step 4 in a loop: the fee can never be raised to throttle the caller.

    function updateTrustedSigner(

        uint256 expiresAt

        if (expiresAt == 0) {
            for (uint8 i = 0; i < trustedSigners.length; i++) {
                if (trustedSigners[i].pubkey == trustedSigner) {
                    trustedSigners[i].pubkey = address(0);
                    trustedSigners[i].expiresAt = 0;
                    delete trustedSignerToExpiresAtMapping[trustedSigner];
                    return;
                }

            revert("no such pubkey");

                if (trustedSigners[i].pubkey == trustedSigner) {

                    trustedSignerToExpiresAtMapping[trustedSigner] = expiresAt;
