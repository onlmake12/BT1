Audit Report

## Title
Missing `from != to` Identity Check in `ConnectionRequestProcess::execute` Enables `pending_delivered` Poisoning and Forced NAT Traversal to Attacker-Controlled Addresses — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

`ConnectionRequestProcess::execute` contains no guard enforcing `content.from != content.to`, and `content.from` is never validated against the actual sender's session `PeerId`. An attacker with a single established P2P connection can send a `ConnectionRequest` with both `from` and `to` set to the victim's own `PeerId`, causing the victim to insert attacker-controlled TCP addresses into `pending_delivered` under its own key. A subsequent `ConnectionSync` with the same `from == to == victim_peer_id` causes the victim to spawn repeated 30-second NAT traversal retry loops to attacker-controlled endpoints and, on success, register a `raw_session` to the attacker bypassing normal peer selection.

## Finding Description

**Root cause — no `from != to` guard and no sender-identity binding:**

In `ConnectionRequestProcess::execute`, the only routing branch after all validation is:

```rust
if self_peer_id == &content.to {
    self.respond_delivered(content.from, &content.to, content.listen_addrs).await
``` [1](#0-0) 

There is no check that `content.from != content.to`, and `content.from` is never validated against the actual session peer ID. When the attacker sets both fields to the victim's own `PeerId`, the condition is satisfied and `respond_delivered` is called with `from_peer_id = victim_peer_id`.

**Step 1 — `pending_delivered` poisoned:**

Inside `respond_delivered`, the 2-minute cooldown check only blocks re-poisoning the same key within the window — it does not prevent the initial poisoning:

```rust
if let Some((_, t)) = self.protocol.pending_delivered.get(&from_peer_id) {
    let now = unix_time_as_millis();
    if now - t < HOLE_PUNCHING_INTERVAL {
        return StatusCode::Ignore ...
    }
}
``` [2](#0-1) 

After filtering `remote_listens` to TCP/IPv4/IPv6 (attacker-controlled TCP addresses pass this filter), the state is written unconditionally:

```rust
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
``` [3](#0-2) 

`pending_delivered` now maps `victim_peer_id → ([attacker_addr], timestamp)`. The entry persists for 5 minutes (`TIMEOUT = 5 * 60 * 1000`). [4](#0-3) 

**Step 2 — `ConnectionSync` triggers NAT traversal to attacker-controlled addresses:**

In `ConnectionSyncProcess::execute`, when `route` is empty and `self_peer_id == &content.to`, the handler reads back the poisoned entry:

```rust
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
``` [5](#0-4) 

With `content.from = victim_peer_id`, it retrieves the attacker-controlled addresses and spawns NAT traversal tasks:

```rust
.map(|listen_addr| Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
``` [6](#0-5) 

`try_nat_traversal` retries TCP connections for up to 30 seconds with jitter. [7](#0-6) 

The `forward_rate_limiter` for `ConnectionSync` is keyed by `(from, to, msg_item_id)` at 1 req/s. [8](#0-7) 

Since the `pending_delivered` entry is not consumed on read and persists for 5 minutes, the attacker can send 1 `ConnectionSync` per second for 5 minutes, spawning up to 300 concurrent NAT traversal tasks, each running for up to 30 seconds.

**Step 3 — `raw_session` registered to attacker on success:**

```rust
let _ignore = control
    .raw_session(
        stream,
        addr,
        RawSessionInfo::inbound(listen_addr),
    )
    .await;
``` [9](#0-8) 

This registers an inbound session to the attacker's address, bypassing normal peer scoring and selection.

**Existing checks are insufficient:**

- The per-session rate limiter (`rate_limiter`, 30 req/s keyed by `(session_id, msg_item_id)`) limits message throughput but does not prevent the attack. [10](#0-9) 
- The `forward_rate_limiter` (1 req/s keyed by `(from, to, msg_item_id)`) limits repetition but does not block the first attempt, and with `from == to == victim_peer_id` the key is fixed — the attacker simply sends one message per second. [11](#0-10) 
- The `pending_delivered` 2-minute cooldown only prevents re-poisoning the same key within the window; it does not prevent the initial poisoning, and the entry is not consumed on read. [12](#0-11) 

## Impact Explanation

The attacker forces the victim to: (1) corrupt `pending_delivered` with attacker-controlled addresses keyed under the victim's own `PeerId`; (2) spawn up to 300 concurrent NAT traversal tasks over a 5-minute window, each making TCP connection attempts every ~200ms for 30 seconds — causing significant CPU and network resource exhaustion; (3) register `raw_session` connections to the attacker on TCP handshake success, bypassing normal peer selection and potentially saturating peer slots. Peer slot saturation can prevent legitimate peers from connecting to the victim, contributing to network-level isolation. This maps to **High severity** — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — as the attacker requires only a single established P2P connection and can sustain the attack indefinitely across rate-limit windows.

## Likelihood Explanation

The attack requires only a single established P2P connection — no special privileges, no PoW, no key material. The two crafted messages (`ConnectionRequest` then `ConnectionSync`) are valid wire-format messages that pass all existing syntactic checks. The `from` field is never verified against the sender's actual session `PeerId`, making identity spoofing trivial. The `forward_rate_limiter` allows 1 `ConnectionSync` per second per `(from, to, msg_type)` tuple; the `pending_delivered` entry persists for 5 minutes without being consumed. Both constraints are easily managed by a persistent attacker. The attack is repeatable every 5 minutes (when the `pending_delivered` entry expires and can be re-poisoned).

## Recommendation

1. Add an explicit identity check immediately after parsing in `ConnectionRequestProcess::execute`:
```rust
if content.from == content.to {
    return StatusCode::InvalidFromPeerId
        .with_context("from and to must be distinct peers");
}
```

2. Apply the same guard in `ConnectionSyncProcess::execute`.

3. Validate that `content.from` matches the sender's actual session `PeerId` (available via `context.session`) to prevent all identity spoofing, not just the `from == to` case. The `peer: PeerIndex` field is already available in `ConnectionRequestProcess`; resolve it to a `PeerId` via the peer registry and compare against `content.from`. [13](#0-12) 

4. Consider consuming (removing) the `pending_delivered` entry after it is read in `ConnectionSyncProcess::execute` to prevent repeated NAT traversal triggering from a single poisoned entry.

## Proof of Concept

```
1. Attacker peer A establishes a P2P connection to victim V.

2. A sends ConnectionRequest {
       from: V.peer_id,
       to:   V.peer_id,
       listen_addrs: [/ip4/<attacker_ip>/tcp/<attacker_port>],
       max_hops: 1,
       route: [],
   }

3. V processes it:
   - self_peer_id == content.to (V.peer_id == V.peer_id) → respond_delivered() called
   - pending_delivered.get(V.peer_id) → None → cooldown check passes
   - remote_listens filtered: TCP/IPv4 passes
   - ConnectionRequestDelivered sent back to A's session (succeeds)
   - pending_delivered[V.peer_id] = ([/ip4/<attacker_ip>/tcp/<attacker_port>], now)

4. A sends ConnectionSync {
       from: V.peer_id,
       to:   V.peer_id,
       route: [],
   }
   (repeated once per second for up to 5 minutes)

5. V processes each ConnectionSync:
   - route empty, self_peer_id == content.to → NAT traversal branch
   - pending_delivered.get(V.peer_id) → [/ip4/<attacker_ip>/tcp/<attacker_port>]
   - try_nat_traversal(bind_addr, attacker_addr) spawned (30s retry loop, ~200ms interval)
   - On TCP accept at attacker_ip:attacker_port:
     raw_session(stream, attacker_addr, RawSessionInfo::inbound(listen_addr)) registered

Assert:
  - pending_delivered contains V.peer_id → attacker_addr (should be absent)
  - V has spawned up to 300 concurrent NAT traversal tasks to attacker_ip:attacker_port
  - V has one or more raw inbound sessions registered to the attacker
  - V's peer slots may be saturated, blocking legitimate connections
```
