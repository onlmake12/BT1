### Title
`ScriptHashTypeVerifier` Enforces `ENABLED_SCRIPT_HASH_TYPE` Only on Lock Scripts, Silently Skipping Type Scripts — (`File: verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` is documented as enforcing that "the ScriptHashType of transaction outputs is within the range permitted by the current consensus rules," but its implementation only inspects `output.lock().hash_type()`. The type script (`output.type_()`) of every output is never checked against `ENABLED_SCRIPT_HASH_TYPE`. This mirrors the ENS M-01 pattern: a permission check that is supposed to cover a full domain (all output scripts) is silently over-scoped to only one sub-domain (lock scripts), leaving the other sub-domain (type scripts) completely unguarded at the non-contextual stage.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` is the consensus-level allowlist of currently-activated hash types:

```
{0 = Data, 1 = Type, 2 = Data1, 4 = Data2}
```

Future hash types (`Data3` = 6, `Data4` = 8, …, `Data127` = 254) are defined in the `ScriptHashType` enum but are intentionally excluded from this set.

`ScriptHashTypeVerifier::verify()` iterates over every output and checks only the lock script:

```rust
// verification/src/transaction_verifier.rs  L796-L814
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { … }
    } else { … }
    // output.type_() is never examined here
}
```

The type script field is never read. A transaction whose output carries a type script with `hash_type = Data3` (value 6) passes this verifier without error.

The check is invoked as part of `NonContextualTransactionVerifier::verify()`, which is the first gate in both the tx-pool admission path (`tx-pool/src/util.rs` → `non_contextual_verify`) and the block verification path.

The script execution engine (`script/src/types.rs`, `select_version` and `extract_script_and_dep_index`) does catch the disallowed hash type at execution time and returns `ScriptError::InvalidScriptHashType`. However, this is a **contextual** check that runs only after the transaction has already been admitted to the pool and fully resolved.

The error taxonomy also diverges: `TransactionError::ScriptHashTypeNotPermitted` (produced for lock scripts) is classified as `is_malformed_tx() = true` and causes immediate rejection with no pool entry. The script-execution error path does not carry this classification, so the transaction is treated as a verification failure rather than a structural malformation.

---

### Impact Explanation

1. **Tx-pool resource consumption**: Any unprivileged sender can submit transactions with a disallowed type-script hash type. These pass non-contextual verification, enter the pool, and force full contextual script execution before being rejected. This is a CPU/memory amplification vector against pool nodes.

2. **Inconsistent error semantics**: The same structural property (disallowed hash type on an output script) produces `ScriptHashTypeNotPermitted` (malformed, immediate) for lock scripts and `ScriptError::InvalidScriptHashType` (verification failure, deferred) for type scripts. Clients and monitoring tools that distinguish these error classes will observe inconsistent behavior.

3. **Future consensus risk**: If a future hardfork adds a new hash type to `ENABLED_SCRIPT_HASH_TYPE` and the non-contextual check is relied upon as the sole gate (e.g., when `Switch::DISABLE_SCRIPT` is active in block verification), the missing type-script check means outputs with the previously-disallowed hash type would have been silently admitted to the pool and could be committed to the chain on nodes that skip script execution.

---

### Likelihood Explanation

No privileged access is required. Any RPC caller or P2P peer can submit a transaction with an output whose type script carries `hash_type = 6` (`Data3`). The transaction is syntactically valid (molecule `check_data` passes because `verify_value(6)` is true — 6 is even), so it clears all structural deserialization checks and reaches `ScriptHashTypeVerifier` cleanly.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type of each output, mirroring the existing lock-script check:

```rust
// After the existing lock-script check, add:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

Apply the same fix to `CellbaseVerifier` in `verification/src/block_verifier.rs` (lines 135–143), which has the identical gap for cellbase output type scripts.

---

### Proof of Concept

1. Construct a `TransactionView` with one output whose lock script uses `hash_type = Data` (0, enabled) and whose type script uses `hash_type = Data3` (6, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Call `ScriptHashTypeVerifier::new(&tx).verify()`.
3. Observe: the verifier returns `Ok(())` — the disallowed type-script hash type is not caught.
4. Submit the same transaction via `send_transaction` RPC. It passes `non_contextual_verify`, enters the pool, proceeds to `ContextualTransactionVerifier`, and is only rejected there with `ScriptError::InvalidScriptHashType` — a deferred, more expensive rejection path.
5. For contrast, replace the type-script hash type with a disallowed value on the **lock** script instead: `ScriptHashTypeVerifier` immediately returns `Err(ScriptHashTypeNotPermitted)` before the transaction ever reaches the pool.
