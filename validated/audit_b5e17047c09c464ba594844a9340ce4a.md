Audit Report

## Title
Decompressed Frame Size Bypasses Per-Protocol `max_frame_length` Limit — (`network/src/compress.rs`)

## Summary
`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` only on the compressed wire frame, but after passing that check it allocates `BytesMut::zeroed(decompressed_bytes_len)` bounded solely by `MAX_UNCOMPRESSED_LEN` (8 MB) with no comparison against `self.length_delimited.max_frame_length()`. Any unprivileged peer can trigger oversized heap allocations by sending crafted snappy frames within the wire limit, enabling memory exhaustion and node crash.

## Finding Description
In `LengthDelimitedCodecWithCompress::decode` at line 226, `self.length_delimited.decode(src)?` correctly rejects compressed frames exceeding `max_frame_length`. [1](#0-0) 

After passing the wire check, `decompress_len(&data[1..])` reads the snappy-declared uncompressed length. The only guard is `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (8 MB). There is no check that `decompressed_bytes_len <= self.length_delimited.max_frame_length()`. The allocation at line 242 occurs unconditionally before actual decompression. [2](#0-1) 

The `process()` method enforces `max_frame_length` on the outbound encode path only and provides no protection for inbound decode. [3](#0-2) 

Per-protocol limits are set in `SupportProtocols::max_frame_length()` and wired into the codec via `CKBProtocol::build()`: [4](#0-3) [5](#0-4) 

**Two exploit paths:**

1. **Large-limit protocols (Sync 2 MB, RelayV3 4 MB, LightClient 2 MB, Filter 2 MB, Discovery 512 KB, HolePunching 512 KB):** Attacker sends a legitimately snappy-compressed payload (e.g., ~384 KB of zeros compressed from 8 MB) within the wire limit. `decompress_len` returns 8,388,608; `BytesMut::zeroed(8_388_608)` allocates 8 MB; decompression succeeds; `Ok(Some(buf))` delivers an 8 MB buffer to the protocol handler — far exceeding the intended per-protocol bound.

2. **Small-limit protocols (Ping 1 KB, Feeler 1 KB, DisconnectMessage 1 KB, Time 1 KB, Identify 2 KB, Alert 128 KB):** Attacker crafts a malformed snappy stream with a varint header declaring 8 MB but a truncated/invalid body ≤ wire limit. `decompress_len` returns 8,388,608; `BytesMut::zeroed(8_388_608)` allocates 8 MB; decompression fails; connection is dropped and buffer freed. The allocation is transient but real, and the attacker can reconnect and repeat.

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

For large-limit protocols, each inbound compressed frame causes a persistent 8 MB heap allocation delivered to the handler. With N concurrent peers each sending frames in a tight loop, heap pressure scales as N × 8 MB per message round, which can exhaust available memory and crash the node. For small-limit protocols, repeated reconnect-and-send cycles cause transient 8 MB allocations that stress the allocator. The missing check is a single missing bounds comparison; the existing `MAX_UNCOMPRESSED_LEN` guard is necessary but not sufficient.

## Likelihood Explanation
The attacker requires only a standard P2P connection — no authentication, no PoW, no privileged role. For path 1, the attacker sends a valid snappy-compressed frame within the wire limit that legitimately decompresses to 8 MB; this is straightforward with highly repetitive data. For path 2, the attacker crafts a snappy varint header declaring 8 MB followed by any bytes ≤ wire limit; this is trivially constructable. Both paths are reachable on mainnet from any peer. Path 1 persists across the connection lifetime; path 2 requires reconnection per attempt but connection establishment is cheap.

## Recommendation
In `LengthDelimitedCodecWithCompress::decode`, after computing `decompressed_bytes_len` and before allocating, add:

```rust
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::ErrorKind::InvalidData.into());
}
```

This mirrors the existing `MAX_UNCOMPRESSED_LEN` guard at line 235 and closes the gap between the wire-level and application-level size invariants. [6](#0-5) 

## Proof of Concept

**Path 1 (persistent, large-limit protocols — e.g., Sync, max_frame_length = 2 MB):**
1. Connect to a CKB node on the Sync protocol stream.
2. Compress 8 MB of all-zero bytes with snappy raw encoder (~384 KB compressed output, well within the 2 MB wire limit).
3. Prepend the `COMPRESS_FLAG` byte (0x80) and a 4-byte length prefix.
4. Send over the P2P connection.
5. Observe: `decode` passes the wire-size check (384 KB ≤ 2 MB), reads `decompressed_bytes_len = 8,388,608`, passes the `MAX_UNCOMPRESSED_LEN` check, allocates `BytesMut::zeroed(8,388,608)`, decompresses successfully, and returns `Ok(Some(buf))` where `buf.len() == 8,388,608`. The Sync handler receives an 8 MB buffer.
6. Repeat across N connections to amplify heap pressure by N × 8 MB per message round.

**Path 2 (transient, small-limit protocols — e.g., Ping, max_frame_length = 1 KB):**
1. Connect to a CKB node on the Ping protocol stream.
2. Craft a byte sequence: snappy varint encoding of 8,388,608 followed by arbitrary bytes, total ≤ 1022 bytes.
3. Prepend `COMPRESS_FLAG` (0x80) and a 4-byte length prefix (total ≤ 1024 bytes).
4. Send over the P2P connection.
5. Observe: `decode` passes the wire-size check (≤ 1024), reads `decompressed_bytes_len = 8,388,608`, passes the `MAX_UNCOMPRESSED_LEN` check, allocates `BytesMut::zeroed(8,388,608)` (8 MB heap allocation), then `decompress` fails (data mismatch), returns `Err` (connection dropped, buffer freed).
6. Reconnect and repeat to sustain allocator pressure.
