Audit Report

## Title
`check_tx_fee` Uses Serialized Size Instead of Transaction Weight for Minimum Fee Enforcement, Allowing High-Cycle Transactions to Bypass Effective Minimum Fee Rate - (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` enforces the minimum fee rate using only the transaction's serialized byte size, not the true transaction weight (`max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because no second fee check is performed after `verify_rtx` reveals the actual cycle count, an attacker can submit transactions with near-maximum cycle consumption at a fraction of the intended economic cost, forcing expensive CKB-VM verification and polluting the mempool with economically unviable transactions.

## Finding Description

In `tx-pool/src/util.rs` at L42–45, `check_tx_fee` explicitly uses `tx_size` for the minimum fee calculation, with a comment acknowledging this is intentional as a "cheap check":

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The true transaction weight used for mining selection and pool eviction is computed by `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

In `_process_tx`, the flow is:
1. `pre_check` → calls `check_tx_fee` (size-based only)
2. `verify_rtx` → executes scripts, returns actual cycles
3. **No second fee check** after `verify_rtx`
4. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is created and submitted [4](#0-3) 

The default `max_tx_verify_cycles` is `TWO_IN_TWO_OUT_CYCLES * 20 = 3_500_000 * 20 = 70_000_000` cycles. [5](#0-4) [6](#0-5) 

For a transaction with ~200 bytes serialized size and 70M cycles:
- Size-based minimum fee at 1,000 shannons/KW: `1000 × 200 / 1000 = 200 shannons`
- True weight: `max(200, 70_000_000 × 0.000_170_571_4) ≈ 11,940`
- Weight-based minimum fee: `1000 × 11,940 / 1000 = 11,940 shannons`
- Discount factor: ~60×

The `check_tx_fee` call in `pre_check` is confirmed at L289: [7](#0-6) 

## Impact Explanation

This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker paying only 200 shannons per transaction (instead of 11,940) can:
1. Force the node to execute 70M CKB-VM cycles per transaction during `verify_rtx`
2. Flood the verify queue (256MB limit) with high-cycle transactions, causing legitimate transactions to receive `Reject::Full`
3. Admit transactions to the pool at ~1/60th of the intended economic cost

The verify queue size limit (`DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256_000_000`) and pool eviction (which correctly uses weight-based fee rate via `EvictKey`) provide partial backstops, but do not prevent the verification-cost DoS — the node must execute all 70M cycles before discovering the transaction's true weight. [8](#0-7) [9](#0-8) 

## Likelihood Explanation

Any unprivileged RPC caller (`send_transaction`) or P2P relay peer can submit such a transaction. Crafting a transaction with high cycle consumption but small serialized size is straightforward — a script that runs a tight computation loop in a small binary achieves this. The attacker's cost is only the size-based minimum fee (~200 shannons per transaction at default settings), making sustained attacks economically viable at ~1/60th of the intended cost.

## Recommendation

After `verify_rtx` returns the actual cycle count in `_process_tx`, perform a second fee check using the true transaction weight before calling `submit_entry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Second fee check using actual weight
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = {
    let tx_pool = self.tx_pool.read().await;
    tx_pool.config.min_fee_rate.fee(weight)
};
if fee < min_fee_by_weight {
    return Some((
        Err(Reject::LowFeeRate(min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())),
        snapshot,
    ));
}
```

This ensures high-cycle transactions pay fees commensurate with their actual resource consumption before pool admission.

## Proof of Concept

1. Craft a CKB transaction whose lock script executes a tight loop consuming ~70,000,000 cycles. The script binary can be small (a few hundred bytes), keeping the serialized transaction size at ~200 bytes.
2. Set the transaction fee to exactly `ceil(1000 × 200 / 1000) = 200 shannons`.
3. Submit via `send_transaction` RPC.
4. Observe: `check_tx_fee` passes (200 ≥ 200 shannons), the node executes 70M cycles during `verify_rtx`, and the transaction is admitted to the pool.
5. Query `get_transaction` — the transaction is in the pool with status `pending`.
6. Observe that miners never include it (effective fee rate ≈ 16 shannons/KW, far below the 1,000 shannons/KW minimum).
7. Repeat to continuously consume node verification resources and fill the verify queue, causing legitimate transactions to be rejected with `Reject::Full`.
