### Title
`HeaderAcceptor::version_check` Uses Hardcoded Literal `0` Instead of Consensus `block_version` Parameter - (File: sync/src/synchronizer/headers_process.rs)

### Summary
`HeaderAcceptor::version_check` in the P2P sync layer checks incoming block headers against a hardcoded literal `0` instead of the consensus-configured `block_version` field. The `Consensus` struct exposes `block_version` as a dedicated configuration parameter with its own accessor, but `version_check` never reads it. This is structurally identical to the Exchange.sol bug: a named configuration constant exists and is used elsewhere, but the validation function ignores it in favor of a magic number.

### Finding Description
In `sync/src/synchronizer/headers_process.rs`, `HeaderAcceptor::version_check` is:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    if self.header.version() != 0 {   // hardcoded literal
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
``` [1](#0-0) 

The CKB codebase defines the authoritative block version constant in `util/types/src/constants.rs`:

```rust
pub const BLOCK_VERSION: Version = 0;
``` [2](#0-1) 

This constant is loaded into the `Consensus` struct as the `block_version` field: [3](#0-2) 

And exposed via `consensus.block_version()`: [4](#0-3) 

The block assembler correctly reads from consensus: `let version = consensus.block_version();` [5](#0-4) 

The `ConsensusBuilder` initializes `block_version` from the `BLOCK_VERSION` constant: [6](#0-5) 

`HeaderAcceptor` holds `active_chain: ActiveChain`, which provides access to the shared state and consensus, but `version_check` never uses it — it hardcodes `0` directly. [7](#0-6) 

### Impact Explanation
If `BLOCK_VERSION` (or `Consensus::block_version`) is ever updated to a new value (e.g., `1`) as part of a protocol upgrade, `version_check` will continue to reject headers carrying the new version as `BLOCK_INVALID`, permanently blacklisting them in the block status store. Upgraded peers relaying valid headers with the new version would be treated as sending invalid data. Conversely, if the consensus were updated to stop accepting version `0`, `version_check` would still pass version-`0` headers. The consensus parameter is the intended single source of truth for this check, but it is silently bypassed.

### Likelihood Explanation
Currently `BLOCK_VERSION = 0` and the hardcoded literal `0` agree, so there is no immediate divergence. The risk materializes the moment a protocol upgrade changes `block_version` in consensus without updating this hardcoded check — a realistic scenario given that the block assembler already reads from consensus correctly, creating a false sense that all version checks are consensus-driven.

### Recommendation
Replace the hardcoded literal with the consensus-provided value. `HeaderAcceptor` already holds `active_chain`, which can supply the consensus:

```rust
pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
    let expected = self.active_chain.sync_shared().shared().consensus().block_version();
    if self.header.version() != expected {
        state.invalid(Some(ValidationError::Version));
        Err(())
    } else {
        Ok(())
    }
}
```

### Proof of Concept
1. Modify `Consensus::block_version` to `1` (simulating a future protocol upgrade).
2. Have a peer send a `SendHeaders` P2P message containing a header with `version = 1`.
3. `HeaderAcceptor::version_check` checks `1 != 0` → returns `Err(())`.
4. `accept()` calls `shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID)` — the valid header is permanently blacklisted.
5. The node refuses to sync from any peer advertising this header, stalling IBD or tip sync. [8](#0-7)
