### Title
Unbounded Replay of Signed Lazer Updates in `verifyUpdate` Enables Stale Price Injection and Fee Drain - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.verifyUpdate` performs no replay protection. Any valid signed Lazer update can be submitted an unlimited number of times. This is directly analogous to the `signatureFreeMint` replay in the external report: a signed artifact is accepted without any consumed-message tracking, nonce, or freshness enforcement, allowing an unprivileged attacker to replay old signed data indefinitely.

### Finding Description

`PythLazer.verifyUpdate` verifies an ECDSA-signed Lazer update and returns the payload and signer address. The function checks:

1. Fee is paid
2. EVM format magic matches
3. Signature recovers to a non-zero address
4. Recovered signer is in the trusted-signer mapping and not expired [1](#0-0) 

What it does **not** check:

- Whether this exact `update` bytes have been submitted before (no consumed-update set, no hash registry)
- Whether the timestamp embedded in the payload is recent (no freshness window enforced on-chain)

The same signed `update` bytes can be submitted to `verifyUpdate` an unlimited number of times. Each call succeeds, charges `verification_fee`, and returns the identical `payload` and `signer`. There is no nonce, sequence number, or consumed-message set analogous to the VAA replay protection used elsewhere in the Pyth ecosystem. [2](#0-1) 

Compare with the Wormhole/Pyth VAA pattern, which explicitly tracks consumed hashes to prevent replay: [3](#0-2) 

No equivalent protection exists in `PythLazer.sol`.

### Impact Explanation

**Direct impact on integrators (realistic):** The Pyth Lazer EVM integration guide shows integrators writing public functions that call `verifyUpdate` and forward `msg.value` from the contract's own balance: [4](#0-3) 

An attacker who observes a valid Lazer update on-chain can replay those exact bytes to any such integrator contract. Each replay:

1. **Drains the integrator's ETH** by `verification_fee` per call (the integrator contract pays the fee, not the attacker).
2. **Injects a stale price** into the integrator's state if the integrator does not independently enforce timestamp freshness.

**Stale price injection** is the primary security impact. A Lazer update signed hours or days ago remains cryptographically valid as long as the signer has not expired. An attacker can force an integrator's on-chain price to revert to an arbitrarily old value, enabling profitable manipulation of any protocol that consumes that price (e.g., lending, perpetuals, options).

**Fee drain** is a secondary financial impact. With `verification_fee = 1 wei` at deployment, the cost per replay is negligible for the attacker, while the integrator pays each time.

### Likelihood Explanation

- The attack requires no privileged access: any EOA can call `verifyUpdate` or any public integrator function that wraps it.
- Valid signed Lazer updates are publicly observable on-chain from prior transactions.
- The Pyth Lazer EVM integration pattern explicitly shows integrators accepting arbitrary `bytes calldata` from callers and forwarding fees from contract balance.
- The Cardano Lazer docs explicitly acknowledge this property ("does not disallow verifying the same update multiple times"), confirming the absence of replay protection is a known design characteristic, not an accidental omission — but the EVM contract provides no warning and no on-chain mitigation. [5](#0-4) 

### Recommendation

Add a consumed-update registry to `PythLazer.sol`. Before returning, compute `keccak256(update)` and revert if the hash has been seen before, or store it and reject duplicates. Alternatively, enforce a maximum age on the timestamp embedded in the payload (e.g., reject updates older than 60 seconds relative to `block.timestamp`). The latter is simpler and aligns with how Pyth's core price feed contracts handle staleness.

    VictimIntegrator victim;

    // bytes captured from a prior on-chain transaction
    bytes oldUpdate = hex"2a22999a..."; // valid signed update from hours ago

    function attack() external {
        uint256 fee = lazer.verification_fee();
        // Replay the same signed update N times.
        // Each call succeeds: signer is still valid, no consumed-update check.
        for (uint i = 0; i < 1000; i++) {
            // victim.updatePrice calls verifyUpdate internally and pays fee from its balance
            victim.updatePrice{value: fee}(oldUpdate);
        }
        // Result: victim's price is now the stale value from the old update,
        // and victim has paid 1000 * verification_fee in ETH.
    }
}

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
