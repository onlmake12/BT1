### Title
`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` Always Revert for Whitelisted Subscriptions Due to Redundant `onlyWhitelistedReader` Check via `this.` External Call - (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` in `Scheduler.sol` internally call `this.getPricesUnsafe(...)` and `this.getEmaPricesUnsafe(...)` respectively. Because `this.` triggers an external call, `msg.sender` inside the callee becomes the `Scheduler` contract itself. Both callee functions carry the `onlyWhitelistedReader` modifier, which checks whether `msg.sender` is the subscription manager or is in the reader whitelist. The Scheduler contract address is neither, so when `whitelistEnabled == true` the call always reverts with `Unauthorized`, making the freshness-checked read functions completely unusable for any caller on whitelisted subscriptions.

---

### Finding Description

`getPricesNoOlderThan` passes the `onlyWhitelistedReader` guard correctly for the original caller, then delegates to `getPricesUnsafe` via `this.`:

```solidity
// Scheduler.sol line 535-555
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
)
    external view override
    onlyWhitelistedReader(subscriptionId)   // ← passes for manager / whitelisted reader
    returns (PythStructs.Price[] memory prices)
{
    ...
    prices = this.getPricesUnsafe(subscriptionId, priceIds);  // ← external call; msg.sender = Scheduler
}
```

`getPricesUnsafe` also carries `onlyWhitelistedReader`:

```solidity
// Scheduler.sol line 514-533
function getPricesUnsafe(
    uint256 subscriptionId,
    bytes32[] calldata priceIds
)
    external view override
    onlyWhitelistedReader(subscriptionId)   // ← now msg.sender == Scheduler contract
    returns (PythStructs.Price[] memory prices)
{ ... }
```

The modifier logic:

```solidity
// Scheduler.sol line 750-779
modifier onlyWhitelistedReader(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] == msg.sender) { _; return; }
    if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) { _; return; }
    // check whitelist array for msg.sender ...
    if (!isWhitelisted) revert SchedulerErrors.Unauthorized();
    _;
}
```

When `whitelistEnabled == true`:
- `msg.sender` is the Scheduler contract address (not the manager).
- The Scheduler contract is not in the `readerWhitelist`.
- The modifier reverts with `Unauthorized`.

The identical pattern exists in `getEmaPricesNoOlderThan` → `this.getEmaPricesUnsafe(...)`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any subscription that sets `whitelistEnabled = true` (the privacy-sensitive use case) loses access to the freshness-checked read functions `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`. These are the functions recommended by the SDK and documentation for safe price consumption. Affected callers — including the subscription manager and every whitelisted reader — are silently forced to either:

1. Use `getPricesUnsafe` / `getEmaPricesUnsafe` without any staleness guarantee, or
2. Be unable to read prices at all.

This breaks the core safety invariant of the Scheduler contract for whitelisted subscriptions and can lead to stale price consumption in downstream protocols.

---

### Likelihood Explanation

The trigger condition is `whitelistEnabled == true` on a subscription. The whitelist feature is explicitly designed for subscriptions that restrict reader access (e.g., protocols that pay for exclusive price feeds). Any such subscription is affected. The bug is deterministic — it triggers on every call to `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` when the whitelist is enabled. No special attacker action is required; the revert happens for all callers including the manager.

---

### Recommendation

Replace the `this.` external calls with direct internal calls to avoid the `msg.sender` context switch. Extract the shared logic into an internal helper (which `_getPricesInternal` already is) and call it directly:

```diff
function getPricesNoOlderThan(...) external view override onlyWhitelistedReader(subscriptionId) {
    ...
-   prices = this.getPricesUnsafe(subscriptionId, priceIds);
+   PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
+   prices = new PythStructs.Price[](priceFeeds.length);
+   for (uint i = 0; i < priceFeeds.length; i++) { prices[i] = priceFeeds[i].price; }
}
```

Apply the same fix to `getEmaPricesNoOlderThan`.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable`.
2. Create a subscription with `whitelistEnabled = true` and a non-zero `readerWhitelist`.
3. Push a price update via `updatePriceFeeds`.
4. Call `getPricesNoOlderThan(subscriptionId, priceIds, 60)` from the whitelisted reader address.
5. Observe revert with `SchedulerErrors.Unauthorized` — the call fails despite the caller being whitelisted, because the internal `this.getPricesUnsafe(...)` re-enters the contract with `msg.sender == Scheduler`, which is not in the whitelist.
6. Confirm `getPricesUnsafe` called directly from the same whitelisted reader succeeds. [5](#0-4) [6](#0-5)

    modifier onlyWhitelistedReader(uint256 subscriptionId) {
        // Manager is always allowed
        if (_state.subscriptionManager[subscriptionId] == msg.sender) {
            _;
            return;
        }

        // If whitelist is not used, allow any reader
        if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) {
            _;
            return;
        }

        // Check if caller is in whitelist
        address[] storage whitelist = _state
            .subscriptionParams[subscriptionId]
            .readerWhitelist;
        bool isWhitelisted = false;
        for (uint i = 0; i < whitelist.length; i++) {
            if (whitelist[i] == msg.sender) {
                isWhitelisted = true;
                break;
            }
        }

        if (!isWhitelisted) {
            revert SchedulerErrors.Unauthorized();
        }
        _;
    }
