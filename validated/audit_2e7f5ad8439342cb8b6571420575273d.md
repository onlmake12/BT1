Looking at the actual code, this is a real and concrete vulnerability. Let me trace through the exact mechanism.

**Key structural facts:**

1. `SignatureSet` is a **keypair account** (not a PDA). In `verify_signatures.rs`, the `signature_set` field uses `init_if_needed` with **no `seeds` constraint** — any fresh keypair can be used. [1](#0-0) 

2. The blocklist in `post_vaa.rs` only checks the **pubkey** of the `SignatureSet` account, not the **message hash** it contains. [2](#0-1) 

3. The `posted_vaa` account is a PDA seeded on `message_hash`, meaning there is exactly one `posted_vaa` per VAA body — and it has not been created yet for the blocked VAA bodies (that's the entire point of the blocklist). [3](#0-2) 

4. There is **no blocklist check in `verify_signatures`** — nothing prevents verifying a blocked message hash into a fresh `SignatureSet` keypair. [4](#0-3) 

**The bypass is straightforward:**

- Generate a fresh keypair `new_sig_set` (not in `INVALID_SIGNATURE_SET_KEYS`)
- Call `verify_signatures(new_sig_set, blocked_vaa_body)` — succeeds, writes quorum + message hash to `new_sig_set`
- Call `post_vaa(new_sig_set, blocked_vaa_body)` — blocklist check passes (`new_sig_set.key()` is not in the 16-entry list), quorum check passes, message hash matches, `posted_vaa` PDA is created

The blocklist is a pubkey-level patch applied to a message-hash-level problem. It is structurally insufficient.

### Title
Hardcoded `INVALID_SIGNATURE_SET_KEYS` blocklist is bypassable via a fresh `SignatureSet` keypair for the same VAA body — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_vaa.rs`)

### Summary
The 16-entry `INVALID_SIGNATURE_SET_KEYS` blocklist in `PostVaa::constraints` blocks specific `SignatureSet` account pubkeys from being used in `post_vaa`. However, because `SignatureSet` is a free keypair account (not a PDA), an attacker can create a new `SignatureSet` with a fresh keypair, re-run `verify_signatures` for the same blocked VAA body, and call `post_vaa` with the new account — bypassing the blocklist entirely.

### Finding Description
`INVALID_SIGNATURE_SET_KEYS` is a hardcoded list of 16 `SignatureSet` pubkeys that are rejected in `PostVaa::constraints`:

```rust
require!(
    !INVALID_SIGNATURE_SET_KEYS.contains(&signature_set.key().to_string().as_str()),
    CoreBridgeError::InvalidSignatureSet
);
``` [5](#0-4) 

The check is on the **account pubkey**, not on the **message hash** stored inside the account. The `SignatureSet` account is initialized via `init_if_needed` with no PDA seeds — any keypair is valid: [1](#0-0) 

`verify_signatures` has no restriction on which message hashes may be verified. An attacker who possesses (or can reconstruct) valid guardian signatures for a blocked VAA body can:

1. Generate a new keypair `K` (not in the blocklist).
2. Submit `[secp256k1_ix(blocked_body_hash, guardian_sigs), verify_signatures(K)]` — this writes quorum + `blocked_body_hash` into `K`.
3. Submit `post_vaa(K, blocked_vaa_args)` — the blocklist check passes, quorum passes, hash matches, and the `posted_vaa` PDA is created. [6](#0-5) 

### Impact Explanation
A previously-blocked VAA body is posted on-chain as a `PostedVaaV1` account. If the blocked VAA encodes a price update or governance action, this constitutes oracle manipulation or unauthorized governance execution. The `posted_vaa` PDA is keyed on `message_hash`, so once posted it is treated as authoritative by all downstream consumers.

### Likelihood Explanation
The attack requires only:
- Knowledge of the blocked VAA body (the 16 blocked `SignatureSet` accounts are public on-chain; their `message_hash` fields are readable)
- Valid guardian signatures for that body (these were already verified into the blocked accounts, so they exist on-chain in the secp256k1 instruction history or can be extracted from the original transactions)
- A fresh keypair and SOL for rent

No privileged access is required. Any unprivileged relayer can execute this.

### Recommendation
Replace the pubkey-level blocklist with a **message-hash-level blocklist**. Maintain a hardcoded set of blocked `MessageHash` values and check `signature_set.message_hash` in `PostVaa::constraints` instead of (or in addition to) the account pubkey. Alternatively, add the same message-hash check to `verify_signatures` to prevent blocked VAA bodies from ever reaching quorum in any `SignatureSet`.

```rust
    pub fn constraints(ctx: &Context<Self>, args: &PostVaaArgs) -> Result<()> {
        let guardian_set = ctx.accounts.guardian_set.inner();

        // Check that the guardian set is still active.
        let timestamp = Clock::get().map(Into::into)?;
        require!(
            guardian_set.is_active(&timestamp),
            CoreBridgeError::GuardianSetExpired
        );

        let signature_set = &ctx.accounts.signature_set;
        require!(
            !INVALID_SIGNATURE_SET_KEYS.contains(&signature_set.key().to_string().as_str()),
            CoreBridgeError::InvalidSignatureSet
        );

        // Number of verified signatures in the signature set account must be at least quorum with
        // the guardian set.
        require!(
            signature_set.num_verified() >= utils::quorum(guardian_set.keys.len()),
            CoreBridgeError::NoQuorum
        );

        // Recompute the message hash and compare it to the one in the signature set account.
        let recomputed = solana_program::keccak::hashv(&[
            &args.timestamp.to_be_bytes(),
            &args.nonce.to_be_bytes(),
            &args.emitter_chain.to_be_bytes(),
            &args.emitter_address,
            &args.sequence.to_be_bytes(),
            &[args.consistency_level],
            &args.payload,
        ]);
        require_eq!(
            MessageHash::from(recomputed),
            signature_set.message_hash,
            CoreBridgeError::InvalidMessageHash
        );

        // Done.
        Ok(())
    }
