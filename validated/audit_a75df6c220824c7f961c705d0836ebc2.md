### Title
Integer Overflow in `GetLastStateProof` Limit Check Enables Unbounded Allocation DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unauthenticated light-client peer can send a single `GetLastStateProof` message with `last_n_blocks = 2^63` to trigger a wrapping integer overflow in the guard check, bypassing the `GET_LAST_STATE_PROOF_LIMIT` entirely. The reorg branch then allocates a `Vec` proportional to the full chain height, followed by a second unbounded `Vec` for `last_n_numbers`, and then calls `complete_headers` for every entry — causing OOM and crashing the full-node process.

---

### Finding Description

**Step 1 — Overflow in the limit guard (line 201)** [1](#0-0) 

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `last_n_blocks = 2^63` on a 64-bit target:
- `last_n_blocks as usize = 2^63`
- `(2^63_usize) * 2 = 2^64` — wraps to **0** in Rust release mode (two's-complement wrapping)
- Guard becomes `0 + 0 > 1000` → **false** → passes

The constant is defined as: [2](#0-1) 

**Step 2 — Unbounded `reorg_last_n_numbers` allocation (line 245–246)** [3](#0-2) 

When `start_hash` does not match the ancestor at `start_block_number` (trivially crafted), the reorg branch executes:

```rust
let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
(min_block_number..start_block_number).collect()
```

With `last_n_blocks = 2^63 > start_block_number`:
- `min(start_block_number, 2^63) = start_block_number`
- `min_block_number = 0`
- Allocates `Vec` of **`start_block_number`** entries — uncapped, no reference to `GET_LAST_STATE_PROOF_LIMIT`

**Step 3 — Unbounded `last_n_numbers` allocation (line 291–297)** [4](#0-3) 

```rust
if last_block_number - start_block_number <= last_n_blocks {
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
```

Since `last_n_blocks = 2^63` is always ≥ any realistic chain span, this branch is always taken, allocating another Vec of `last_block_number - start_block_number` entries.

**Step 4 — `complete_headers` amplifies the damage** [5](#0-4) 

All three Vecs are chained and passed to `complete_headers`, which performs a DB lookup, block fetch, and MMR root computation for **every** entry — proportional to the full chain height H.

---

### Impact Explanation

With chain height H ≈ 3.4M (CKB mainnet as of 2026), setting `start_block_number = H/2`:
- `reorg_last_n_numbers`: ~1.7M entries
- `last_n_numbers`: ~1.7M entries
- `complete_headers` fetches ~3.4M `packed::VerifiableHeader` objects (each containing header data, uncles hash, extension, MMR root)

This exhausts process memory and crashes the full node. A single message is sufficient; no repetition is needed.

---

### Likelihood Explanation

- Any peer connecting via the light-client P2P protocol can send `GetLastStateProof` with no authentication
- The only required attacker knowledge is a valid tip hash, which is publicly observable
- The crafted `start_hash` just needs to differ from the real ancestor — any random 32-byte value works
- The overflow is deterministic in release builds (standard CKB deployment) [6](#0-5) 

---

### Recommendation

1. **Fix the overflow**: Replace the unchecked multiplication with a saturating or checked operation:
   ```rust
   if self.message.difficulties().len()
       + (last_n_blocks as usize).saturating_mul(2)
       > constant::GET_LAST_STATE_PROOF_LIMIT
   ```

2. **Cap `reorg_last_n_numbers` independently**: After computing `min_block_number`, clamp the range to at most `GET_LAST_STATE_PROOF_LIMIT` entries before collecting.

3. **Cap `last_n_numbers` independently**: The `(start_block_number..last_block_number)` range must also be bounded by the limit constant, not just by `last_n_blocks`.

4. **Add a combined post-collection length check**: Assert `reorg_last_n_numbers.len() + last_n_numbers.len() <= GET_LAST_STATE_PROOF_LIMIT` before calling `complete_headers`.

---

### Proof of Concept

```rust
// Pseudocode: craft the malicious message
let msg = GetLastStateProof {
    last_hash:          /* valid current tip hash, publicly observable */,
    last_n_blocks:      1u64 << 63,          // 2^63 — overflows limit check to 0
    start_number:       5000u64,             // any value ≤ chain height
    start_hash:         [0u8; 32].into(),    // wrong hash → triggers reorg branch
    difficulty_boundary: U256::MAX,
    difficulties:       vec![],              // empty → difficulties.len() = 0
};
// Guard: 0 + (2^63 as usize)*2 = 0 > 1000 → false → passes
// reorg_last_n_numbers = (0..5000).collect() → 5000 entries
// last_n_numbers = (5000..chain_height).collect() → (H-5000) entries
// complete_headers called for H entries → OOM
```

On a chain of height 3.4M, this allocates and processes ~3.4M headers from a single unauthenticated P2P message.
