Audit Report

## Title
`remove_expired` Uses Bare `remove_entry` Instead of `remove_entry_and_descendants`, Leaving Descendant Transactions with Stale Edges — (File: `tx-pool/src/pool.rs`)

## Summary
`remove_expired` in `tx-pool/src/pool.rs` calls `pool_map.remove_entry` for each expired transaction rather than `pool_map.remove_entry_and_descendants`. When a parent transaction expires before its child, the child remains in `entries` while its `edges.inputs` record — mapping the parent's output to the child's ID — persists. This stale mapping causes `find_conflict_tx` to falsely reject any new transaction spending that output as a double-spend, enabling low-cost, scalable network congestion.

## Finding Description

`PoolMap` maintains three coupled structures: `entries`, `edges` (input/dep/output relationships), and `links` (parent/child relationships). [1](#0-0) 

`remove_expired` collects all entries whose `timestamp + expiry < now_ms` and removes each with bare `remove_entry`: [2](#0-1) 

`remove_entry` calls `remove_entry_edges` and `remove_entry_links` only for the transaction being removed — it does not touch the edges of any descendant: [3](#0-2) 

Consider `tx_A` (submitted at T, creates output `O1`) and `tx_B` (submitted at T+1min, spends `O1`). When `remove_expired` fires at T+expiry, only `tx_A` is in the expired set. `tx_A` is removed; its own edges are cleaned. But `edges.inputs[O1] = tx_B` — registered when `tx_B` was added — is never removed, because `remove_entry_edges` only removes the edges *belonging to* the transaction being removed, not those of its children. `tx_B` remains in `entries` with a stale input edge pointing to a non-existent output.

The function's own comment confirms the intended behavior: [4](#0-3) 

Every other removal path correctly uses `remove_entry_and_descendants`: `limit_size` (L307), `remove_by_detached_proposal` (L343), `remove_tx` (L359), `resolve_conflict` (L310), and `resolve_conflict_header_dep` (L285). [5](#0-4) 

`remove_entry_and_descendants` first collects all descendant IDs via `calc_descendants`, removes all links, then removes each entry — ensuring `edges.inputs` is fully cleaned: [6](#0-5) 

`find_conflict_tx` directly consults `edges.inputs` to detect double-spends: [7](#0-6) 

Because `edges.inputs[O1] = tx_B` persists after `tx_A` expires, any new transaction `tx_C` spending `O1` is rejected as a conflict. `edges.inputs` is a `HashMap<OutPoint, ProposalShortId>`: [8](#0-7) 

## Impact Explanation

**High: bad designs which could cause CKB network congestion with few costs.**

An attacker submits many parent-child transaction pairs. After parents expire, stale `edges.inputs` entries block legitimate transactions from spending those outputs for up to ~12 hours (until the children expire). Stale children in `Status::Proposed` also corrupt block templates — every template including `tx_B` fails chain validation because `O1` no longer exists, forcing repeated template regeneration. Both effects are scalable: each pair costs only two transaction fees, and the attacker can block many output points simultaneously.

## Likelihood Explanation

Triggerable by any unprivileged user via standard RPC or P2P relay. No special access, no victim mistakes, no majority hashpower required. The attack requires only two transactions and a ~12-hour wait. It is fully repeatable and scalable to many output points in parallel.

## Recommendation

Replace the bare `remove_entry` call in `remove_expired` with `remove_entry_and_descendants`, consistent with all other removal paths and with the function's documented intent:

```rust
// tx-pool/src/pool.rs — remove_expired
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
    let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    for e in evicted {
        let reject = Reject::Expiry(e.timestamp);
        callbacks.call_reject(self, &e, reject);
    }
}
```

## Proof of Concept

1. Submit `tx_A` (spends live cell `C1`, creates output `O1`) at time T.
2. Submit `tx_B` (spends `O1`) at time T+1 minute.
3. Advance mock clock to T+expiry (past `tx_A`'s expiry, before `tx_B`'s).
4. Call `remove_expired`.
5. Assert `pool_map.entries` does not contain `tx_A` — passes.
6. Assert `pool_map.edges.inputs` contains no entry for `O1` — **fails**: `edges.inputs[O1] = tx_B` persists.
7. Assert `pool_map.entries` does not contain `tx_B` — **fails**: `tx_B` remains.
8. Submit `tx_C` spending `O1`; observe it is rejected by `find_conflict_tx` as a false double-spend.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
