### Title
`DaoScriptSizeVerifier` Missing `rfc0044_active` Guard in Block Verifier Allows DAO Lock-Script-Size Bypass Before Hardfork Activation — (`verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

The `DaoScriptSizeVerifier` — the sole consensus-layer guard against a known Nervos DAO script bug — is applied **unconditionally** in the tx-pool (`tx-pool/src/util.rs`) but is gated behind `rfc0044_active` in the block verifier (`verification/contextual/src/contextual_block_verifier.rs`). Before rfc0044 activates, the block verifier silently skips this check entirely, while the tx-pool enforces it. A miner who assembles a block directly (bypassing the tx-pool) can include a DAO withdrawal transaction whose output lock script differs in size from the deposit cell's lock script, exploiting the acknowledged DAO script bug to redirect or inflate a withdrawal.

---

### Finding Description

**Root cause — the two divergent code paths:**

**Path A — block verifier (`BlockTxsVerifier::verify`):**

```rust
// verification/contextual/src/contextual_block_verifier.rs  lines 444-453
.and_then(|result| {
    if self.context.consensus.rfc0044_active(self.parent.epoch().number()) {
        DaoScriptSizeVerifier::new(
            Arc::clone(tx),
            Arc::clone(&self.context.consensus),
            self.context.store.as_data_loader(),
        ).verify()?;
    }
    Ok(result)
})
```

`DaoScriptSizeVerifier` is **only called when `rfc0044_active` returns `true`**. Before that epoch, the entire verifier is skipped.

**Path B — tx-pool (`verify_rtx`):**

```rust
// tx-pool/src/util.rs  lines 117-131
block_in_place(|| {
    ContextualTransactionVerifier::new(...)
        .verify(max_tx_verify_cycles, false)
        .and_then(|result| {
            DaoScriptSizeVerifier::new(
                rtx,
                snapshot.cloned_consensus(),
                snapshot.as_data_loader(),
            )
            .verify()?;          // ← NO rfc0044_active guard
            Ok(result)
        })
        .map_err(Reject::Verification)
})
```

`DaoScriptSizeVerifier` is called **unconditionally** — no `rfc0044_active` check.

The same asymmetry exists in the `verify_with_pause` branch (lines 101–115 of the same file).

**What `DaoScriptSizeVerifier` protects:**

The verifier's own doc comment states:

> "Verifies that deposit cell and withdrawing cell in Nervos DAO use same sized lock scripts. **It provides a temporary solution till Nervos DAO script can be properly upgraded.**"

The DAO on-chain script has a known defect: it does not enforce that the lock script of the withdrawing output matches the lock script of the deposit input. `DaoScriptSizeVerifier` is the **only** node-side enforcement that prevents a withdrawal from substituting a different-sized (and therefore different) lock script. Before rfc0044 activation, the block verifier provides **zero** enforcement of this invariant.

**Exploit flow:**

1. Attacker deposits CKB into Nervos DAO using lock script `A` (size `N`), committed after `starting_block_limiting_dao_withdrawing_lock` (the internal block-number gate inside `DaoScriptSizeVerifier`).
2. Attacker constructs a Phase-2 withdrawal transaction whose output cell carries lock script `B` (size `M ≠ N`, controlled by the attacker). The DAO on-chain script accepts this because of its acknowledged size-check defect.
3. Attacker submits the transaction **directly to a miner** (e.g., via the miner's own RPC or a private relay), bypassing the tx-pool entirely.
4. The miner includes the transaction in a block. `BlockTxsVerifier::verify` runs; because `rfc0044_active` is `false`, `DaoScriptSizeVerifier` is never invoked. The block passes all consensus checks.
5. The withdrawal is committed on-chain with lock script `B`. Funds are now spendable only by the holder of `B`'s key — the attacker.

---

### Impact Explanation

- **Asset theft**: DAO deposits whose lock script can be swapped during withdrawal are permanently redirected to the attacker's key. The DAO script's acknowledged bug means the on-chain script will not catch the substitution; the only protection is the missing node-side check.
- **Scope**: Any DAO deposit committed after `starting_block_limiting_dao_withdrawing_lock` and before rfc0044 activation is at risk.
- **Severity**: Critical if rfc0044 is not yet active on mainnet; the window is the entire pre-activation period.

---

### Likelihood Explanation

The attacker must be a miner (or collude with one) to bypass the tx-pool. This does **not** require majority hashpower — any miner who mines even a single block can include the crafted transaction. Mining is an open, permissionless role on CKB. The SECURITY.md only excludes "51% attack" scenarios; a single-block inclusion does not qualify. Likelihood is **medium-low** (requires mining capability) but the impact if triggered is severe.

---

### Recommendation

**Short term:** Mirror the `rfc0044_active` guard into `tx-pool/src/util.rs::verify_rtx` so both code paths enforce the same rule set at the same activation boundary:

```rust
// tx-pool/src/util.rs — both non-cached branches
.and_then(|result| {
    if snapshot.cloned_consensus().rfc0044_active(/* current epoch */) {
        DaoScriptSizeVerifier::new(...).verify()?;
    }
    Ok(result)
})
```

**Long term:** Upgrade the Nervos DAO on-chain script to enforce lock-script identity directly, eliminating the need for the node-side workaround entirely and removing the divergence risk.

---

### Proof of Concept

```
Pre-condition: rfc0044 not yet activated (epoch < rfc0044 activation epoch).
               Deposit committed at block > starting_block_limiting_dao_withdrawing_lock.

1. tx_deposit = DAO deposit cell, lock = Script { code_hash: H, args: A, hash_type: T }
   lock.total_size() = N

2. tx_withdraw_phase2 = DAO withdrawal, output lock = Script { code_hash: H', args: A', hash_type: T' }
   lock.total_size() = M  (M ≠ N)
   DAO script passes due to known size-check defect.

3. Submit tx_withdraw_phase2 directly to miner RPC (not through tx-pool).
   tx-pool would reject: DaoScriptSizeVerifier fires unconditionally → DaoLockSizeMismatch.

4. Miner mines block containing tx_withdraw_phase2.
   BlockTxsVerifier::verify:
     rfc0044_active(parent.epoch()) == false  →  DaoScriptSizeVerifier NOT called.
     ContextualTransactionVerifier passes (DAO script bug).
   Block accepted by all nodes.

5. Output cell now locked by attacker-controlled Script { H', A', T' }.
   Original depositor's funds are stolen.
```

```rust
    /// Verifies that for all Nervos DAO transactions, withdrawing cells must use lock scripts
    /// of the same size as corresponding deposit cells
    pub fn verify(&self) -> Result<(), Error> {
        let dao_type_hash = self.dao_type_hash();
        for (i, (input_meta, cell_output)) in self
            .resolved_transaction
            .resolved_inputs
            .iter()
            .zip(self.resolved_transaction.transaction.outputs())
            .enumerate()
        {
            // Both the input and output cell must use Nervos DAO as type script
            if !(cell_uses_dao_type_script(&input_meta.cell_output, &dao_type_hash)
                && cell_uses_dao_type_script(&cell_output, &dao_type_hash))
            {
                continue;
            }

            // A Nervos DAO deposit cell must have input data
            let input_data = match self.data_loader.load_cell_data(input_meta) {
                Some(data) => data,
                None => continue,
            };

            // Only input data with full zeros are counted as deposit cell
            if input_data.into_iter().any(|b| b != 0) {
                continue;
            }

            // Only cells committed after the pre-defined block number in consensus is
            // applied to this rule
            if let Some(info) = &input_meta.transaction_info
                && info.block_number
                    < self
                        .consensus
                        .starting_block_limiting_dao_withdrawing_lock()
            {
                continue;
            }

            // Now we have a pair of DAO deposit and withdrawing cells, it is expected
            // they have the lock scripts of the same size.
            if input_meta.cell_output.lock().total_size() != cell_output.lock().total_size() {
                return Err((TransactionError::DaoLockSizeMismatch { index: i }).into());
            }
        }
        Ok(())
    }
```
