### Title
Deviation Calculation Ignores Price Exponent, Enabling False Keeper Update Triggers — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `_validateShouldUpdatePrices` function in `Scheduler.sol` computes price deviation using raw mantissa values while completely ignoring the Pyth price exponent (`expo`). When the exponent for a price feed changes between two updates — a legitimate Pyth network operation — the mantissa changes proportionally to preserve the actual price, but the deviation formula treats this as a massive price movement. This causes false-positive deviation triggers, paying the keeper for an update that reflects no real price change and draining the subscription's ETH balance.

---

### Finding Description

Pyth prices are encoded as `actual_price = price × 10^expo`. The `price` field alone is meaningless without `expo`. In `_validateShouldUpdatePrices`, the deviation is computed as:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;   // raw mantissa only
int64 previousPrice = previousFeed.price.price;     // raw mantissa only

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
```

Neither `priceFeeds[i].price.expo` nor `previousFeed.price.expo` is read or used. The calculation implicitly assumes both prices share the same exponent, so the exponent cancels in the ratio. This assumption holds only when the exponent is constant across updates.

When the Pyth network changes the exponent for a feed (e.g., from `-8` to `-7`), the mantissa is rescaled by a factor of 10 to represent the same actual price:

| Update | `price` (mantissa) | `expo` | Actual price |
|---|---|---|---|
| Previous | `100_000_000` | `-8` | `$1.00` |
| Current | `10_000_000` | `-7` | `$1.00` |

The deviation formula computes:

```
|10_000_000 − 100_000_000| / 100_000_000 × 10_000 = 90_000 bps (900%)
```

A 900% deviation is reported even though the actual price is unchanged. Any `deviationThresholdBps` configured by the subscriber will be exceeded, and the update is accepted.

---

### Impact Explanation

Once the Pyth network publishes a feed with a changed exponent, **every keeper** who calls `updatePriceFeeds(subscriptionId, updateData)` with that update will:

1. Pass the deviation check unconditionally (false positive).
2. Receive the full keeper payment: `gasCost + singleUpdateKeeperFeeInWei × numPriceIds` (line 849).
3. Cause `status.balanceInWei` and `status.totalSpent` to be debited for a spurious update.

This repeats for every subsequent update until the stored `previousFeed` is overwritten with a record that shares the new exponent. Subscriptions with many price feeds or high `singleUpdateKeeperFeeInWei` suffer proportionally larger losses. Subscribers cannot prevent this because the exponent change originates from the Pyth network, not from the subscription configuration.

---

### Likelihood Explanation

Pyth price feed exponents do change, though infrequently. The Pyth documentation does not guarantee exponent stability. When a feed is reconfigured (e.g., precision is increased or decreased), the exponent changes. At that moment, the first update submitted by any keeper will trigger the false-positive deviation. The entry point (`updatePriceFeeds`) is permissionless — any address can call it — so no privileged access is required to exploit the window.

---

### Recommendation

Before computing the deviation ratio, normalize both mantissas to a common exponent:

```solidity
int32 curExpo  = priceFeeds[i].price.expo;
int32 prevExpo = previousFeed.price.expo;
int64 curMantissa  = priceFeeds[i].price.price;
int64 prevMantissa = previousFeed.price.price;

if (curExpo != prevExpo) {
    int32 diff = curExpo - prevExpo;
    if (diff > 0) {
        // previous has finer precision; scale it down
        prevMantissa /= int64(int32(10 ** uint32(diff)));
    } else {
        // current has finer precision; scale it down
        curMantissa /= int64(int32(10 ** uint32(-diff)));
    }
}
// now compute deviation using curMantissa and prevMantissa
```

Alternatively, use `PythUtils.convertToUint` (already present in the repo) to convert both prices to a fixed decimal representation before comparing.

---

### Proof of Concept

1. A subscription is created with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%).
2. Keeper A calls `updatePriceFeeds` with update data: `price = 100_000_000`, `expo = -8` → stored as `previousFeed`.
3. The Pyth network reconfigures the feed to `expo = -7` (legitimate operation; actual price unchanged at `$1.00`).
4. Keeper B calls `updatePriceFeeds` with the new update data: `price = 10_000_000`, `expo = -7`.
5. `_validateShouldUpdatePrices` computes deviation = `|10_000_000 − 100_000_000| / 100_000_000 × 10_000 = 90_000 bps`.
6. `90_000 >= 100` → update is accepted; Keeper B receives the full keeper fee.
7. The subscription's `balanceInWei` is debited for a spurious update.

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
