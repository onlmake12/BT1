### Title
Excess ETH Sent to `requestV2` / `requestHelper` Is Permanently Captured Into `accruedPythFeesInWei` Instead of Being Refunded — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In Pyth Entropy, every request variant (`request`, `requestWithCallback`, `requestV2`) routes through `requestHelper`. That function checks `msg.value >= requiredFee` but never refunds the surplus. Instead, the entire `msg.value - providerFee` is credited to `_state.accruedPythFeesInWei`. Any ETH sent above the exact required fee is permanently captured by the Pyth protocol fee pool and is only recoverable by the privileged admin via `withdrawFee`. The caller receives no refund.

---

### Finding Description

`requestHelper` computes the required fee and validates the payment floor, but then unconditionally sweeps the full `msg.value - providerFee` into the Pyth fee accumulator:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

`requiredFee = providerFee + pythFeeInWei`. When `msg.value > requiredFee`, the surplus is silently absorbed into `accruedPythFeesInWei` rather than returned to `msg.sender`. The NatDoc on every public request variant explicitly acknowledges this:

> "Further note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

The accrued Pyth fees are only withdrawable by the privileged admin:

```solidity
function withdrawFee(address targetAddress, uint128 amount) external {
    _authoriseAdminAction();
    _state.accruedPythFeesInWei -= amount;
    (bool success, ) = targetAddress.call{value: amount}("");
    ...
}
``` [4](#0-3) 

This is structurally identical to the GlueX finding: a user sends more value than required to an intermediary function; the excess is not returned; the user permanently loses the surplus.

---

### Impact Explanation

Any caller who sends `msg.value > getFeeV2(provider, gasLimit)` loses the excess ETH permanently. The most realistic loss scenarios are:

1. **Integrating contracts that forward `msg.value` directly** — a common pattern is `entropy.requestV2{value: msg.value}(...)`. If the end-user sends slightly more than the exact fee (e.g., to avoid race conditions on fee changes), the surplus is captured.
2. **Fee changes between `getFeeV2` call and transaction inclusion** — if the Pyth admin lowers `pythFeeInWei` between the off-chain fee query and on-chain execution, the caller overpays and loses the difference.
3. **Hardcoded or rounded-up fee values** — integrators who round up or hardcode a fee buffer lose that buffer on every call.

The lost ETH accrues to `accruedPythFeesInWei` and is only recoverable by the Pyth admin, not by the original caller.

---

### Likelihood Explanation

- The behavior is reachable by any unprivileged Entropy user with a single transaction.
- Integrating contracts that pass `msg.value` through are common and explicitly shown in Pyth's own developer documentation examples.
- Fee changes via `setPythFee` or `setProviderFee` are governance-controlled and can happen at any time, creating windows where callers who pre-computed the fee will overpay.
- The `requestV2` family of functions is the primary entry point for all Entropy users. [5](#0-4) 

---

### Recommendation

Refund the surplus to `msg.sender` at the end of `requestHelper`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (_state.pythFeeInWei); // only the protocol portion

// Refund excess
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

Alternatively, enforce an exact-payment check (`msg.value == requiredFee`) to make overpayment an explicit revert rather than a silent loss.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import {Entropy} from "../contracts/entropy/Entropy.sol";
import {IEntropyV2} from "../entropy_sdk/solidity/IEntropyV2.sol";

contract EntropyOverpayTest is Test {
    IEntropyV2 entropy;
    address provider;

    function setUp() public {
        // deploy / fork setup omitted for brevity
    }

    function testExcessFeeNotRefunded() public {
        address user = address(0xBEEF);
        vm.deal(user, 1 ether);

        uint128 exactFee = entropy.getFeeV2();
        uint128 overpayment = exactFee + 0.01 ether;

        uint256 balanceBefore = user.balance;

        vm.prank(user);
        entropy.requestV2{value: overpayment}();

        uint256 balanceAfter = user.balance;

        // User lost the full overpayment, not just exactFee
        assertEq(balanceBefore - balanceAfter, overpayment);
        // Pyth fee pool absorbed the surplus
        assertGt(entropy.getAccruedPythFees(), exactFee - entropy.getFeeV2(provider, 0) + exactFee);
    }
}
```

The test confirms that `user.balance` decreases by `overpayment` (not `exactFee`), and `getAccruedPythFees()` absorbs the surplus. The user has no mechanism to recover the excess. [6](#0-5) [7](#0-6)

    function withdrawFee(address targetAddress, uint128 amount) external {
        require(targetAddress != address(0), "targetAddress is zero address");
        _authoriseAdminAction();

            revert EntropyErrors.InsufficientFee();

        _state.accruedPythFeesInWei -= amount;

        (bool success, ) = targetAddress.call{value: amount}("");
