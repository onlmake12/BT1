Audit Report

## Title
Per-Session Rate Limiter Bypass via Reconnection — (`sync/src/relayer/mod.rs`, `network/src/protocols/hole_punching/mod.rs`)

## Summary
Both `Relayer` and `HolePunching` protocol handlers key their `governor`-based rate limiters on `(PeerIndex, u32)`, where `PeerIndex` is a per-session alias for `SessionId`. A reconnecting peer receives a new `SessionId` and therefore a fresh, full token bucket. Because `TooManyRequests` (code 110, a 1xx informational code) never triggers a ban, an attacker can cycle TCP connections to sustain relay message throughput far above the intended 30 req/sec cap with no IP-level consequence.

## Finding Description

**`PeerIndex` is session-scoped:**
`PeerIndex` is defined as `pub type PeerIndex = SessionId` in `network/src/protocols/mod.rs` L33. [1](#0-0) 

**Rate limiters are keyed on `(PeerIndex, u32)`:**
`Relayer` declares `rate_limiter: RateLimiter<(PeerIndex, u32)>` and initializes it with `RateLimiter::hashmap(quota)` at 30 req/sec. [2](#0-1) 
`HolePunching` uses the identical pattern. [3](#0-2) [4](#0-3) 

**Rate-limit failure returns a 1xx code, never a ban:**
In `Relayer::try_process()`, exceeding the limit returns `StatusCode::TooManyRequests.with_context(...)`. [5](#0-4) 
`TooManyRequests = 110` is explicitly a 1xx informational code. [6](#0-5) 
`should_ban()` only returns `Some` for codes in `400..500`; code 110 falls outside this range and returns `None`. [7](#0-6) 
The same `should_ban()` logic applies in `HolePunching`. [8](#0-7) 
In `HolePunching.received()`, a rate-limit hit simply returns with no ban or disconnect. [9](#0-8) 

**Disconnect does not remove the peer's bucket:**
`Relayer::disconnected()` calls only `self.rate_limiter.retain_recent()`, which prunes globally stale entries but does not remove the disconnecting peer's specific bucket. [10](#0-9) 
`HolePunching::disconnected()` does the same. [11](#0-10) 
The new session after reconnection starts with a completely fresh, full 30-token bucket because its `(PeerIndex, u32)` key has never been seen before.

**IP-level ban is never reached:**
The IP-level ban check in `peer_registry.rs` L109–111 (`peer_store.is_addr_banned(&remote_addr)`) is only triggered by explicit `ban_peer()`/`ban_session()` calls, which require a 4xx status. Rate-limit violations never reach this path. [12](#0-11) 

**Exploit flow:**
1. Attacker connects (new `SessionId` → new `(PeerIndex, u32)` key → fresh 30-token bucket).
2. Sends 30 relay messages — all accepted.
3. 31st message returns `TooManyRequests`; no ban issued.
4. Attacker disconnects; victim calls `retain_recent()` only.
5. Attacker reconnects with a new keypair (new `SessionId` → fresh bucket).
6. Repeat indefinitely. With ~50 ms TCP+secio handshake overhead, effective sustained rate ≈ 600 relay messages/sec vs. the intended 30/sec cap — a ~20× bypass.

## Impact Explanation

Sustained relay message flooding at 20× the intended rate forces the victim node to perform repeated tx-pool lookups, message deserialization, and proposal/transaction processing for every message. This constitutes a low-cost, high-throughput resource exhaustion attack matching the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (CPU saturation and memory pressure from queued work), and *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (a flooded node falls behind on block relay, degrading its participation in the network).

## Likelihood Explanation

Any unprivileged peer reachable over TCP can exploit this. No special keys, hashpower, or protocol knowledge beyond standard CKB P2P is required. The default CKB configuration (`max_peers = 125`, `listen_addresses = ["/ip4/0.0.0.0/tcp/8115"]`) accepts inbound peers from the public internet. The attacker only needs to generate a fresh keypair per connection (trivial) to avoid the `PeerIdExists` check in `accept_peer()`. The attack is fully repeatable and requires no victim interaction.

## Recommendation

- Re-key the rate limiter by the peer's **IP address** (extracted from `session_context.address`) rather than by `PeerIndex`/`SessionId`. This mirrors how the ban list works and survives reconnection.
- Alternatively, on `disconnected`, explicitly remove the disconnecting peer's entries from the rate limiter hashmap using the known `PeerIndex` before the new session can claim a fresh bucket, and maintain a separate IP-keyed counter that persists across reconnections.
- Consider escalating repeated rate-limit violations from the same IP to a temporary IP-level ban via `peer_store.ban_addr()`, analogous to how malformed messages are handled.

## Proof of Concept

```
1. Attacker generates keypair_1, connects to victim (SessionId=7, PeerIndex=7).
2. Sends 30 GetRelayTransactions in rapid succession → all accepted (fresh bucket).
3. Sends 31st → StatusCode::TooManyRequests returned, no ban issued.
4. Attacker disconnects. victim calls retain_recent() only.
5. Attacker generates keypair_2, reconnects (SessionId=8, PeerIndex=8).
6. rate_limiter has no entry for (PeerIndex=8, item_id) → fresh 30-token bucket.
7. Sends 30 more GetRelayTransactions → all accepted. ✅
8. Repeat from step 3 indefinitely.

Effective rate: ~600 relay messages/sec vs. intended 30/sec cap.

Unit test plan: instantiate a Relayer, call try_process 31 times with PeerIndex(1)
to exhaust the bucket, assert TooManyRequests, call disconnected with PeerIndex(1),
then call try_process with PeerIndex(2) and assert the first call succeeds —
demonstrating the fresh-bucket bypass.
```
