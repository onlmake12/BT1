Audit Report

## Title
Minimum Fee Rate Admission Check Uses `tx_size` Instead of Transaction Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the serialized byte size (`tx_size`), while `TxEntry::fee_rate()` computes the actual fee rate using `get_transaction_weight(size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For compute-heavy transactions where cycles dominate, the admission threshold is evaluated against a smaller denominator than the one used for all subsequent fee-rate accounting. An unprivileged submitter can craft a transaction that passes `check_tx_fee` while its true weight-based fee rate is materially below `min_fee_rate`, enabling low-cost pool saturation.

## Finding Description
In `tx-pool/src/util.rs` at lines 42–52, `check_tx_fee` explicitly acknowledges the discrepancy in a comment and computes the minimum fee using only `tx_size`:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

The actual fee rate of every admitted `TxEntry` is computed in `tx-pool/src/component/entry.rs` at lines 114–118 using the full weight:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
```

`get_transaction_weight` is defined in `util/types/src/core/tx_pool.rs` at lines 298–303:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

The submission pipeline in `tx-pool/src/process.rs` calls `check_tx_fee` (with only `tx_size`) during `pre_check` at line 289, before script verification runs and before actual cycles are known. After verification at lines 724–732, the `TxEntry` is constructed at line 751 with the real `verified.cycles`. From that point, all fee-rate comparisons — eviction via `EvictKey` and pool-full rejection via `limit_size` — use the weight-based `fee_rate()`. There is no second fee-rate gate after cycles are known. The gap is structural: `check_tx_fee` is the sole admission gate, and it uses `tx_size` (always ≤ weight) as the denominator. When `cycles * DEFAULT_BYTES_PER_CYCLES > tx_size`, the attacker pays `min_fee_rate * tx_size / 1000` shannons, which passes the check, but the entry's true fee rate is `fee * 1000 / weight < min_fee_rate`.

## Impact Explanation
This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. An attacker deploys a compute-heavy lock or type script (permissionless on CKB) and submits many transactions spending cells locked by it. Each transaction pays a fee just above `min_fee_rate * tx_size / 1000` shannons — far below the true weight-based minimum. The tx-pool (default 180 MB) fills with these below-minimum-fee-rate transactions. Legitimate transactions that pay the correct weight-based fee rate are displaced or evicted. The pool's economic spam-protection threshold is silently bypassed, causing sustained congestion at low cost to the attacker.

## Likelihood Explanation
The exploit path is the standard `send_transaction` RPC, reachable by any unprivileged peer or local caller. No special role, key, or majority hashpower is required. Deploying a compute-heavy script is permissionless on CKB. The bypass is deterministic: for any transaction where `cycles * DEFAULT_BYTES_PER_CYCLES > tx_size`, the gap exists. The attacker can scale cycles (up to `max_tx_verify_cycles = 70,000,000`) to maximize the ratio `weight / tx_size` and minimize the effective fee rate while still passing admission. The attack is repeatable across many transactions.

## Recommendation
After script verification completes and `verified.cycles` is known, perform a second fee-rate check using the actual weight before inserting the entry into the pool in `tx-pool/src/process.rs` after line 751:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

The existing `check_tx_fee` call can remain as a cheap pre-verification filter (using `tx_size`) to avoid running expensive script verification on obviously-underpaying transactions; the weight-based check should be added as a post-verification gate.

## Proof of Concept
**Parameters:**
- `min_fee_rate` = 1,000 shannons/KW (default)
- `DEFAULT_BYTES_PER_CYCLES` = 0.000_170_571_4
- Transaction serialized size = 100 bytes
- Transaction cycles = 1,000,000

**Weight:** `max(100, 1_000_000 * 0.000_170_571_4) = max(100, 170) = 170`

**Admission check (uses tx_size = 100):**
`min_fee = 1000 * 100 / 1000 = 100 shannons` → attacker pays exactly 100 shannons → **passes** `check_tx_fee`

**True fee rate (uses weight = 170):**
`actual_fee_rate = 100 * 1000 / 170 ≈ 588 shannons/KW` — **41% below** `min_fee_rate`

Scaling to `cycles = 10,000,000` → `weight = 1,705`:
`actual_fee_rate = 100 * 1000 / 1705 ≈ 59 shannons/KW` — **94% below** `min_fee_rate`, while still passing admission.

**Reproducible steps:**
1. Deploy a script that loops for ~1,000,000 cycles on CKB devnet.
2. Create cells locked by this script.
3. Submit transactions spending those cells via `send_transaction` RPC, paying fee = `min_fee_rate * tx_size / 1000` shannons.
4. Observe transactions enter the pool; verify via `get_pool_tx_detail_info` that the entry's fee rate (weight-based) is below `min_fee_rate`.
5. Repeat until pool is saturated with below-minimum-fee-rate transactions, displacing legitimate traffic.
