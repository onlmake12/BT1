Audit Report

## Title
`ScriptHashTypeVerifier` Omits `hash_type` Validation for `type_` Scripts in Transaction Outputs — (File: `verification/src/transaction_verifier.rs`)

## Summary
`ScriptHashTypeVerifier::verify()` at lines 796–814 of `verification/src/transaction_verifier.rs` applies the `ENABLED_SCRIPT_HASH_TYPE` guard only to `output.lock().hash_type()` and never inspects `output.type_().hash_type()`. A transaction whose output carries a `type_` script with an unsupported `hash_type` byte passes the non-contextual gate and is admitted to the tx-pool, where it occupies a slot until contextual verification raises `InvalidScriptHashType` and rejects it. The concrete impact is tx-pool pollution enabling low-cost network congestion.

## Finding Description
`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and performs two checks on the `lock` script — a `TryInto::<ScriptHashType>` conversion guard and an `ENABLED_SCRIPT_HASH_TYPE` membership check — but the loop body contains no corresponding inspection of `output.type_()`. The `type_` field is an optional `Script` with its own `hash_type` byte; when present with a value outside `{0, 1, 2, 4}` (e.g., raw byte `6` for `Data3`), neither branch of the existing conditional touches it and the function returns `Ok(())`.

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` lines 7–12 permits only `{0u8, 1u8, 2u8, 4u8}`. The non-contextual tx-pool admission path in `tx-pool/src/util.rs` lines 56–62 calls `NonContextualTransactionVerifier::new(tx, consensus).verify()`, which invokes `ScriptHashTypeVerifier`. No ban or penalty mechanism for `InvalidScriptHashType` failures exists in the tx-pool code. At contextual verification, `extract_script_and_dep_index` in `script/src/types.rs` lines 832–860 calls `ScriptHashType::try_from(script.hash_type())` and returns a hard `ScriptError::InvalidScriptHashType` for any unsupported value — execution is never silently skipped, confirming Scenario B (silent bypass) is not valid. The same omission propagates through `NonContextualBlockTxsVerifier::verify()` at `verification/src/block_verifier.rs` lines 280–285, which calls `NonContextualTransactionVerifier::new(tx, self.consensus).verify()` for every block transaction.

Exploit flow:
1. Attacker calls `send_transaction` RPC with a transaction whose output `type_` script carries `hash_type = 6` (`Data3`) or any byte not in `{0,1,2,4}`.
2. `non_contextual_verify` → `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` returns `Ok(())` without inspecting `output.type_().hash_type()`.
3. Transaction is admitted to the tx-pool.
4. Contextual verification calls `extract_script_and_dep_index`, hits the `hash_type => { return Err(ScriptError::InvalidScriptHashType(...)) }` arm, and rejects the transaction.
5. The tx-pool slot is freed only after the contextual verification work (input resolution, I/O) has been performed.

## Impact Explanation
The concrete impact is tx-pool pollution: an unprivileged attacker can repeatedly submit transactions whose `type_` script carries an unsupported `hash_type`, each passing the cheap non-contextual gate and forcing the node to perform input resolution and initial contextual verification before rejection. No VM cycles are consumed, keeping per-transaction attacker cost low (only the minimum relay fee). At scale this wastes memory and I/O on transactions guaranteed to be rejected, matching the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. The absence of any ban or rate-limit on submitters of `InvalidScriptHashType` transactions in the tx-pool means the attack is repeatable without escalating cost to the attacker.

## Likelihood Explanation
The entry path requires only an RPC call to `send_transaction` or a P2P relay submission — no privileged access, no key material, no majority hashpower. Crafting a transaction with an output whose `type_` script has an unsupported `hash_type` byte is trivial. The attacker must pay the minimum relay fee per transaction, so the attack is not entirely free, but the cost-per-rejection is low because contextual failure occurs before any VM cycles are consumed. The omission is present in the current production code path for both tx-pool admission and block-level non-contextual verification.

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to apply the same `ENABLED_SCRIPT_HASH_TYPE` guard to `output.type_()` when present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

Extract the repeated guard into a private helper to prevent future divergence. Add a companion unit test mirroring `test_not_enabled_hash_type_output_lock` but targeting the `type_` script field.

## Proof of Concept
The existing test `test_not_enabled_hash_type_output_lock` at lines 100–122 of `verification/src/tests/transaction_verifier.rs` confirms that a `lock` script with `hash_type = Data3` is correctly rejected. An analogous test for `type_` scripts does not exist. The following test currently **passes** (i.e., `verify()` returns `Ok(())`), confirming the omission:

pub fn test_not_enabled_hash_type_output_type_script() {

                        .build(),
                ).pack())
                .build(),
        )
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);
    // Currently returns Ok(()), demonstrating the omission:
    assert!(verifier.verify().is_ok());
}
