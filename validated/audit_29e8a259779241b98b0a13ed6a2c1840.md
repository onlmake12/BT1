### Title
Integer Overflow in `GetLastStateProofProcess::execute` Bypasses Block-Count Limit, Enabling Full-Chain Traversal DoS — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

### Summary

The limit guard at line 201 computes `(last_n_blocks as usize) * 2` without overflow protection. In a Rust release build, this multiplication wraps silently. An unprivileged remote peer can send a crafted `GetLastStateProof` P2P message with `last_n_blocks = 2^63` (or any value > `usize::MAX/2`), causing the product to wrap to `0`, the guard to pass, and the server to subsequently iterate over every block in the chain.

### Finding Description

**Guard (line 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // = 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

`last_n_blocks` is a `u64` cast to `usize` (both 64-bit on any supported target). When `last_n_blocks = 2^63 = 9223372036854775808`:

```
(2^63 as usize) * 2  →  0   (wraps, release mode)
```

`difficulties.len() + 0 > 1000` is `false` for any normal request, so the guard is silently bypassed.

**Downstream unbounded work — reorg path (lines 237–247):**

```rust
} else {
    let min_block_number = start_block_number - min(start_block_number, last_n_blocks);
    (min_block_number..start_block_number).collect()
};
``` [2](#0-1) 

With `last_n_blocks = 2^63`, `min(start_block_number, last_n_blocks) = start_block_number`, so `min_block_number = 0` and `reorg_last_n_numbers = (0..start_block_number)` — the entire chain prefix. The attacker triggers this branch by supplying a `start_hash` that does not match the canonical ancestor at `start_block_number` (trivially crafted).

**Downstream unbounded work — last_n path (lines 291–297):**

```rust
if last_block_number - start_block_number <= last_n_blocks {
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [3](#0-2) 

Since no real chain has 2^63 blocks, this branch is always taken, producing `last_n_numbers` spanning the entire chain from `start_block_number` to the tip.

**Combined `block_numbers` (lines 350–354):**

```rust
let block_numbers = reorg_last_n_numbers
    .into_iter()
    .chain(sampled_numbers)
    .chain(last_n_numbers)
    .collect::<Vec<_>>();
``` [4](#0-3) 

This vector covers `0..last_block_number` — every block in the chain.

**`complete_headers` iterates all of them (lines 132–179):** for each block it calls `snapshot.get_ancestor()`, `snapshot.get_block()`, and `snapshot.chain_root_mmr()` — all disk-backed operations. [5](#0-4) 

### Impact Explanation

A single malicious peer can force the light-client server to perform O(chain_height) expensive DB lookups and MMR root computations per message. On mainnet (millions of blocks), this exhausts CPU and memory, hanging or crashing the light-client server. The constant `GET_LAST_STATE_PROOF_LIMIT = 1000` is completely defeated. [6](#0-5) 

### Likelihood Explanation

- No authentication or privilege required — any P2P peer can send `GetLastStateProof`.
- The crafted value (`2^63`) is a single field in a packed message; trivial to set.
- Rust release-mode wrapping is deterministic and well-documented; no environment-specific conditions needed.
- Reproducible locally with a single crafted message.

### Recommendation

Replace the unchecked multiplication with a saturating or checked variant:

```rust
// Option A: saturating_mul — safe, no panic
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Or add an explicit pre-check:

```rust
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT / 2 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

### Proof of Concept

```rust
// Craft a GetLastStateProof message with:
//   last_n_blocks  = 1u64 << 63          // 2^63
//   difficulties   = []                  // len = 0
//   last_hash      = <any valid tip hash>
//   start_hash     = <any hash != canonical ancestor at start_block_number>
//   start_number   = <any block number, e.g. 1000>
//
// In release mode:
//   (1u64 << 63) as usize = 9223372036854775808
//   9223372036854775808usize * 2 = 0  (wraps)
//   0 + 0 > 1000  →  false  →  guard bypassed
//
// Server then builds block_numbers = 0..tip_height
// and calls complete_headers on every block → DoS
```
