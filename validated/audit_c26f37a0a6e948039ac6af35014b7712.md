### Title
Attacker-Controlled 8MB Heap Allocation Per Compressed P2P Frame via Off-by-One in Snappy Decompressed-Length Guard — (`network/src/compress.rs`)

### Summary
The decompressed-length guard uses a strict `>` comparison against `MAX_UNCOMPRESSED_LEN` (8 388 608). A frame whose snappy header encodes `decompressed_len == 8388608` passes the guard and causes an 8 MB heap allocation before any decompression work is done. Because the allocation is proportional to an attacker-supplied header value and not to the actual compressed payload size, a tiny frame (tens of bytes on the wire) triggers an 8 MB allocation.

### Finding Description

`MAX_UNCOMPRESSED_LEN` is defined as `1 << 23` (8 388 608 bytes). [1](#0-0) 

In `LengthDelimitedCodecWithCompress::decode`, the guard is:

```rust
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN { … reject … }
let mut buf = BytesMut::zeroed(decompressed_bytes_len);
``` [2](#0-1) 

Because the comparison is strictly `>`, a value of exactly `8388608` passes and `BytesMut::zeroed(8388608)` is called unconditionally. The same off-by-one exists in `Message::decompress` with `vec![0; decompressed_bytes_len]`. [3](#0-2) 

The attacker crafts a snappy-framed payload where:
- The snappy varint header encodes `decompressed_len = 8388608`
- The actual compressed body is a handful of bytes (a valid snappy literal block)
- The outer length-delimited frame is therefore tiny (~15 bytes on the wire)

The decoder reads `decompress_len`, sees `8388608 ≤ MAX_UNCOMPRESSED_LEN`, allocates 8 MB, attempts decompression (which either succeeds trivially or fails), then returns. The 8 MB buffer is freed after the call, but the **peak live allocation per in-flight frame is 8 MB**.

### Impact Explanation

Frames from different peer connections are processed in separate async tasks. With N simultaneous connections each sending one such frame, the peak live allocation is `N × 8 MB`. CKB's default max-peers is 125, giving a theoretical peak of ~1 GB from the compression path alone, on top of normal node memory. On nodes with ≤1 GB RAM (common for light deployments), this is sufficient for OOM. Even below OOM, sustained memory pressure degrades performance and can trigger the OS OOM killer.

The claim that a **single** connection sending frames at high rate causes OOM is not accurate for sequential per-connection processing: each 8 MB allocation is freed before the next frame is decoded. However, the multi-connection vector is fully realistic and requires no special privileges beyond establishing a peer connection.

### Likelihood Explanation

- Compression is enabled by default.
- Any connected peer (inbound or outbound) can send raw bytes on any protocol substream.
- Crafting a valid snappy header with an arbitrary `decompressed_len` is trivial; the snappy format encodes this as a varint at the start of the stream.
- No PoW, no authentication, no rate-limit on frame submission prevents this.
- The attacker only needs to open connections up to the node's peer limit, which is publicly reachable.

### Recommendation

1. Change both guards from `>` to `>=`:
   ```rust
   if decompressed_bytes_len >= MAX_UNCOMPRESSED_LEN { … }
   ``` [4](#0-3) [5](#0-4) 

2. More robustly: **do not allocate based on the header value at all**. Instead, allocate based on the actual compressed input size (snappy's worst-case expansion ratio is ~1.004×), or use `snap::raw::Decoder::decompress_vec` which handles allocation internally and respects the actual data.

3. Add a per-connection rate limit on frames that trigger decompression errors.

    # repeat to sustain pressure
