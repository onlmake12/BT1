### Title
Silent Input Skip via `break` on Unindexed Cell in `insert_transaction` — (`util/rich-indexer/src/indexer/mod.rs`)

---

### Summary

`AsyncRichIndexer::insert_transaction` breaks out of the input-processing loop the moment `spend_cell()` returns `false` for any input. When a `cell_filter` or `block_filter` is configured, outputs that do not match the filter are never stored in the `output` table. Any valid on-chain transaction whose first input (`input[0]`) references such a filtered-out output will cause the loop to `break` immediately, leaving every subsequent input unprocessed — their referenced outputs are never marked `is_spent=1`.

---

### Finding Description

In `insert_transaction`, the input loop is:

```rust
if tx_index != 0 {
    for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
        let out_point = input.previous_output();
        if !spend_cell(&out_point, tx).await? {
            break;          // ← aborts the entire loop
        }
        // ... build input rows
    }
}
``` [1](#0-0) 

`spend_cell` issues an `UPDATE output SET is_spent = 1 WHERE ...` and returns `Ok(updated_rows > 0)`. [2](#0-1) 

When the referenced output was never inserted into the `output` table (because it was excluded by a `cell_filter` or `block_filter`), `updated_rows` is 0 and `spend_cell` returns `false`. The `break` then terminates the loop, so every input after `input[0]` is silently skipped — their outputs are never marked spent.

The legacy KV-based indexer handles the identical situation correctly with an implicit `continue` (the `if let Some(...)` simply falls through to the next iteration): [3](#0-2) 

`CustomFilters` supports both a `block_filter` (exclude entire blocks) and a `cell_filter` (exclude individual outputs). Both are documented, production-supported operator configurations. [4](#0-3) 

---

### Impact Explanation

After the bug fires, any output that was indexed (i.e., matched the filter and has a row in the `output` table) but whose spending transaction had an unindexed cell at `input[0]` will permanently retain `is_spent=0`. The `get_cells` RPC will return it as a live cell indefinitely. Downstream applications that use `get_cells` to enumerate spendable UTXOs will receive stale, incorrect data — cells that are already spent on-chain will appear as available. This permanently corrupts the indexer's UTXO view for all RPC consumers of the affected node.

---

### Likelihood Explanation

The precondition is that the operator has configured a `cell_filter` or `block_filter` — a documented, common deployment pattern (e.g., indexing only tokens of a specific type script). Once that configuration is in place, any on-chain participant can trigger the bug by submitting a valid transaction that spends an unindexed cell as `input[0]` alongside an indexed cell as `input[1]`. No special privilege is required; the transaction is fully valid from the consensus perspective. The bug is permanent and self-reinforcing: once an output is stuck at `is_spent=0`, no subsequent block processing will correct it.

---

### Recommendation

Replace `break` with `continue` on line 232:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // output not indexed; skip this input, process the rest
}
```

This matches the behavior of the legacy indexer and ensures that all inputs are processed regardless of whether any individual input's referenced output happens to be absent from the DB.

### Proof of Concept

1. Start the rich indexer with a `cell_filter` that matches only cells with `type_script1`.
2. Append **Block 1** containing:
   - `tx_A` with two outputs: `cell_X` (no type script → filtered out, not indexed) and `cell_Y` (has `type_script1` → indexed, `is_spent=0`).
3. Append **Block 2** containing:
   - `tx_B` with inputs `[cell_X, cell_Y]` (valid on-chain spend of both).
4. After `append(Block 2)`:
   - `spend_cell(cell_X)` → 0 rows updated → returns `false` → `break`.
   - `spend_cell(cell_Y)` is **never called**.
5. Assert: `SELECT is_spent FROM output WHERE ...` for `cell_Y` → **0** (should be 1).
6. Call `get_cells` for `cell_Y`'s lock script → it is returned as a live cell despite being spent on-chain. [5](#0-4)

pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
