Audit Report

## Title
Stale-Snapshot Overwrite in `add_entry` Inflates `total_tx_size`/`total_tx_cycles`, Enabling Mempool DoS — (`tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::add_entry` snapshots projected pool totals before calling `check_and_record_ancestors`, which may evict existing entries and correctly decrement `self.total_tx_size`/`self.total_tx_cycles` via `update_stat_for_remove_tx`. Immediately after, lines 218–219 unconditionally overwrite those fields with the pre-eviction snapshot, re-inflating the totals by the size and cycles of every evicted transaction. An unprivileged RPC caller can craft a transaction graph that repeatedly triggers this path, permanently inflating pool accounting until the pool appears full and rejects all new transactions.

## Finding Description
**Root cause — `add_entry` lines 200–221:**

`updated_stat_for_add_tx` (lines 711–729) is a pure read: it computes `self.total_tx_size + tx_size` and `self.total_tx_cycles + cycles` and returns them without mutating any field. The returned values are stored in local variables `total_tx_size` and `total_tx_cycles`.

`check_and_record_ancestors` (lines 588–640) enters the eviction branch at line 603 when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618) for each `cell_ref_parent`, which chains to `remove_entry` (line 247) → `update_stat_for_remove_tx` (lines 733–758), **mutating** `self.total_tx_size` and `self.total_tx_cycles` in place by subtracting each evicted entry's size and cycles.

After `check_and_record_ancestors` returns, lines 218–219 write the stale pre-eviction snapshot back:

```
self.total_tx_size  = total_tx_size;   // line 218 — stale: S + N
self.total_tx_cycles = total_tx_cycles; // line 219 — stale: C + cycles(T)
```

| Step | `self.total_tx_size` |
|---|---|
| Before call | `S` |
| After `updated_stat_for_add_tx` (local var) | snapshot = `S + N` |
| After evictions in `check_and_record_ancestors` | `S - E` (correct) |
| After overwrite at line 218 | `S + N` (stale, **E lost**) |

Correct value: `S - E + N`. Actual value: `S + N`. Inflation per call: `E`.

The `recompute_total_stat` fallback in `update_stat_for_remove_tx` (line 743) is irrelevant here — it only fires on underflow during removal, not on the subsequent overwrite.

**Eviction trigger:** `cell_ref_parents` are pool transactions whose outputs are referenced as cell deps by the new transaction's inputs — a standard, protocol-permitted CKB pattern. No special privilege is required to construct such a graph.

## Impact Explanation
`limit_size` in `pool.rs` (lines 292–328) evicts legitimate pending transactions whenever `self.pool_map.total_tx_size > self.config.max_tx_pool_size` (default 180 MB). With `total_tx_size` artificially inflated by repeated exploitation:

- Legitimate transactions are evicted from the pool even though actual memory usage is within the limit.
- New transactions from honest users are rejected with `Reject::Full` despite ample pool space.
- `total_tx_cycles` inflation distorts fee-rate estimation returned to miners and RPC callers.
- Repeated exploitation drives `total_tx_size` to `max_tx_pool_size`, after which the pool permanently rejects all incoming transactions — a complete mempool denial-of-service requiring a node restart to recover.

**Impact class: High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs (10001–15000 points).**

## Likelihood Explanation
The trigger requires: (1) submitting a chain of ≥25 ancestor transactions (achievable with default `max_ancestors_count = 25`), and (2) a dep-provider transaction whose output is referenced as a cell dep by the new transaction, making it a `cell_ref_parent`. Both are valid, protocol-permitted patterns. No privileged access, no key material, and no majority hashpower is required. Any RPC-accessible user — including remote callers if the RPC port is exposed — can execute this. The attack is repeatable: each round inflates `total_tx_size` by `size(D)`, and fresh dep-provider transactions can be submitted each round.

## Recommendation
Move the stat update **after** all evictions have completed, so it reflects the post-eviction pool state. Replace the pre-eviction snapshot pattern with a direct increment applied after `check_and_record_ancestors` returns:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Overflow check only — do not capture the snapshot
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));
    // Apply increment AFTER evictions so evicted sizes are not re-added
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, remove the snapshot entirely and apply the increment directly to `self.total_tx_size`/`self.total_tx_cycles` after all mutations, mirroring the paired decrement pattern used in `update_stat_for_remove_tx`.

## Proof of Concept
**Setup (default config: `max_ancestors_count = 25`, `max_tx_pool_size = 180 MB`):**

1. Submit dep-provider transaction `D` (e.g., 100 KB).
2. Submit ancestor chain `A1 → A2 → … → A24` (each spending the previous output; `A1` also spends an output that `D:0` is a cell dep for).
3. Submit new transaction `T` spending `A24`'s output and referencing `D:0` as a cell dep.

**Execution trace:**
1. `add_entry(T)`: `updated_stat_for_add_tx` snapshots `total_tx_size = S + size(T)`.
2. `check_and_record_ancestors`: `ancestors_count = 26 > 25`; `cell_ref_parents = {D}`; `26 - 1 = 25 ≤ 25` → eviction branch fires.
3. `remove_entry_and_descendants(D)` → `update_stat_for_remove_tx(size(D), ...)` → `self.total_tx_size = S - size(D)`.
4. Back in `add_entry`: `self.total_tx_size = total_tx_size` → `self.total_tx_size = S + size(T)`.
5. Inflation per round: `size(D)` (100 KB in this example).

Repeating with fresh `D` transactions accumulates inflation. After `ceil(180 MB / 100 KB) = 1800` rounds, `total_tx_size` reaches `max_tx_pool_size`, `limit_size` evicts all pending transactions, and the pool rejects every subsequent submission with `Reject::Full`.

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles

                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
