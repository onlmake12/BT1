### Title
`RecentReject::put` Never Increments `self.total_keys_num`, Permanently Disabling `count_limit` Enforcement — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put` computes a local variable `total_keys_num = self.total_keys_num.checked_add(1)` but **never writes it back to `self.total_keys_num`** in the non-shrink path. As a result, `self.total_keys_num` is permanently stuck at its initialization value (0 for a fresh DB), `shrink()` is never triggered, and the `count_limit` guard is completely inoperative for the lifetime of the process. An unprivileged attacker submitting a stream of rejected transactions causes unbounded disk growth in the `recent_reject` RocksDB-with-TTL store.

---

### Finding Description

In `RecentReject::put`:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ...
    self.db.put(&shard, hash_slice, json_string)?;   // entry written to disk

    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
        if total_keys_num > self.count_limit {
            self.shrink()?;
        }
        // ← self.total_keys_num is NEVER updated here
    } else {
        self.shrink()?;
    }
    Ok(())
}
``` [1](#0-0) 

`total_keys_num` is a **local binding** produced by `checked_add(1)`. It is used only for the comparison, but `self.total_keys_num` is never assigned the incremented value. After every `put` that does not trigger `shrink`, `self.total_keys_num` remains unchanged.

On a fresh DB, `build()` initializes `total_keys_num` from `estimate_num_keys_cf`, which returns 0 for empty column families: [2](#0-1) 

So on every subsequent `put`:
- `total_keys_num` (local) = 0 + 1 = **1**
- `1 > count_limit` (default: 10,000,000) → **false**
- `self.total_keys_num` stays at **0**
- `shrink()` is **never called**

`shrink()` is the only place that updates `self.total_keys_num` back from a real DB estimate: [3](#0-2) 

The default `count_limit` is 10,000,000 entries with a 7-day TTL: [4](#0-3) 

---

### Impact Explanation

Every rejected transaction (except `Reject::Duplicated`) is recorded via `should_recorded()`: [5](#0-4) 

This covers `LowFeeRate`, `Full`, `Resolve`, `Verification`, `RBFRejected`, `Expiry`, `Malformed`, etc. The recording path is called from both the P2P relay path and the local RPC path: [6](#0-5) 

Because `count_limit` is never enforced, the `recent_reject` RocksDB store grows without bound until TTL compaction runs (up to 7 days later). Each entry is a 32-byte key + JSON-serialized reject reason. At scale, this leads to disk exhaustion and node crash.

---

### Likelihood Explanation

The attack requires only the ability to submit transactions that are rejected for non-`Duplicated` reasons. Submitting transactions with a fee rate below the minimum (`LowFeeRate`) or referencing non-existent outpoints (`Resolve::Unknown`) achieves this without peer banning (only `is_malformed_tx()` triggers `ban_malformed`): [7](#0-6) 

`LowFeeRate` and `Resolve(Unknown/Dead)` are not malformed, so the peer is not banned and can continue submitting indefinitely. There is no per-peer rate limit on transaction submission visible in the codebase.

---

### Recommendation

In `RecentReject::put`, assign the incremented value back to `self.total_keys_num` in the non-shrink branch:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;   // ← missing assignment
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

### Proof of Concept

```rust
// With shard_num=2, count_limit=10, ttl=-1:
let mut rr = RecentReject::build(tmp_dir.path(), 2, 10, -1).unwrap();
assert_eq!(rr.total_keys_num, 0);

for i in 0..20u64 {
    let key = Byte32::new(blake2b_256(i.to_le_bytes()));
    rr.put(&key, Reject::LowFeeRate(Default::default(), 100, 0)).unwrap();
    // After each put, self.total_keys_num is still 0 — shrink() never fires
}

// Actual DB has 20 entries; total_keys_num is still 0; count_limit=10 was never enforced
assert_eq!(rr.total_keys_num, 0);  // passes — demonstrates the bug
assert!(rr.estimate_total_keys_num_pub() > 10); // actual DB has > count_limit entries
```

The existing test `test_basic` in `tx-pool/src/component/tests/recent_reject.rs` asserts `total_keys_num < 100` after inserting 160 entries with `count_limit=100`, but this assertion passes vacuously because `total_keys_num` is always 0 — it does not actually verify that `shrink()` ran. [8](#0-7)
