### Title
Permissionless `update_guardian_set_ttl` Forcibly Resets Guardian Set TTL, Shortening Old Guardian Set Validity Window — (`target_chains/solana/programs/core-bridge/src/legacy/processor/update_guardian_set_ttl.rs`)

### Summary
Any unprivileged caller can invoke `update_guardian_set_ttl` at any time to unconditionally overwrite `Config.guardian_set_ttl` to 86400 seconds. If the program was initialized with a larger TTL (e.g., 2,592,000 seconds / 30 days), an attacker can call this instruction immediately before a `guardian_set_update` is relayed, causing the old guardian set's `expiration_time` to be set to `now + 86400` instead of `now + 30_days`.

### Finding Description
`update_guardian_set_ttl` has no access control whatsoever — no `Signer` constraint beyond the implicit fee payer, no authority PDA, no governance VAA. The `UpdateGuardianSetTtl` accounts struct only requires the `Config` PDA (mutable): [5](#0-4) 

The function body is a single unconditional write: [6](#0-5) 

When `guardian_set_update` subsequently executes, it reads the (now-manipulated) `config.guardian_set_ttl` to compute the old guardian set's expiration: [7](#0-6) 

### Impact Explanation
If the deployment TTL was > 86400 seconds, the old guardian set's validity window is drastically shortened. Relayers holding VAAs signed by the old guardian set lose most of their submission window. VAAs that would have been valid for 30 days become invalid after 24 hours, causing bridge message loss. This matches the scoped impact: **smart contract unable to operate / griefing without profit motive**.

### Likelihood Explanation
The attack requires only a single permissionless transaction with no special accounts. The attacker only needs to front-run or race the `guardian_set_update` relay transaction. On Solana, this is straightforward. The precondition (TTL initialized to > 86400) must hold, which limits applicability to deployments not already using 86400 as the initial TTL.

### Recommendation
Add an authority check to `update_guardian_set_ttl`. Either:
- Require a governance VAA (like other governance actions), or
- Restrict the caller to the bridge upgrade authority PDA, or
- Remove the instruction entirely if the TTL is meant to be immutable at 86400.

### Proof of Concept
State-transition test:
1. Initialize the bridge with `guardian_set_ttl_seconds = 2_592_000` (30 days).
2. Call `update_guardian_set_ttl` from an unprivileged keypair — succeeds with no error.
3. Assert `config.guardian_set_ttl == 86400`.
4. Relay a valid governance `guardian_set_update` VAA.
5. Assert `curr_guardian_set.expiration_time == now + 86400` (not `now + 2_592_000`).
