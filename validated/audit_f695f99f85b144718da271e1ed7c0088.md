### Title
Collected Update Fees Are Permanently Locked in `PriceInfoObject` With No Withdrawal Path ŌĆö (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

---

### Summary

Every call to `update_single_price_feed` deposits the caller's fee `Coin<SUI>` into the corresponding `PriceInfoObject` via `deposit_fee_coins`. No function exists anywhere in the contract to withdraw those coins. The `fee_recipient_address` stored in `State` is explicitly documented as "not used anymore" and is never referenced in any transfer or withdrawal logic. All collected fees are permanently locked.

---

### Finding Description

**Fee deposit path** (`pyth.move`, `update_single_price_feed`):

```
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
price_info::deposit_fee_coins(price_info_object, fee);   // ŌåÉ coins go in, never come out
``` [1](#0-0) 

**Storage mechanism** (`price_info.move`, `deposit_fee_coins`): the `Coin<SUI>` is stored as a dynamic object field keyed by `FEE_STORAGE_KEY` on the `PriceInfoObject`. There is a `get_balance` reader but no corresponding `withdraw` or `take` function. [2](#0-1) 

**`fee_recipient_address` is a dead field**: `State` stores `fee_recipient_address` and exposes `get_fee_recipient()`, but this address is never the target of any `transfer::public_transfer` or `coin::split`/`join` call anywhere in the production code. [3](#0-2) 

**The governance module itself confirms this**: the comment at the top of `set_fee_recipient.move` reads verbatim:

> "The previous version of the contract sent the fees to a recipient address but this state is not used anymore. This module is kept for backward compatibility." [4](#0-3) 

The `set_fee_recipient` governance action still updates `State.fee_recipient_address`, but since nothing reads that field to route funds, it has zero effect on fee flow. [5](#0-4) 

A complete grep of all production `.move` files under `target_chains/sui/contracts/sources/` confirms: `deposit_fee_coins`, `FEE_STORAGE_KEY`, and `get_balance` appear **only** in `price_info.move` and `pyth.move`. There is no `withdraw_fee_coins`, no `take_fee`, and no `transfer` of the stored `Coin<SUI>` to any address.

---

### Impact Explanation

- Every SUI paid as an update fee is permanently locked inside the respective `PriceInfoObject` with no on-chain recovery path.
- `fee_recipient_address` is a dead configuration parameter; governance can update it but it has no effect.
- The protocol can never reclaim fees to fund operations, making the fee mechanism economically inert.
- This is a permanent, irreversible loss of funds for every fee paid since deployment.

---

### Likelihood Explanation

This is not a probabilistic risk ŌĆö it is a structural certainty. Every `update_single_price_feed` call with `base_update_fee > 0` locks funds. The condition is already met on any live deployment with a non-zero fee. No attacker action is required; the design itself causes the loss.

---

### Recommendation

Add a privileged `withdraw_fees` entry function (gated by `LatestOnly` or an admin capability) that:
1. Extracts the `Coin<SUI>` dynamic object field from a `PriceInfoObject` using `dynamic_object_field::remove`.
2. Transfers the coin to `state::get_fee_recipient(pyth_state)`.

Alternatively, if the intent is to burn fees, remove `fee_recipient_address` and the `set_fee_recipient` governance action entirely to avoid misleading operators.

---

### Proof of Concept

**Invariant that is violated** (pseudocode, locally testable in Move test framework):

```move
// After N calls to update_single_price_feed with base_update_fee > 0:
let total_locked = price_info::get_balance(&price_info_object_1)
                 + price_info::get_balance(&price_info_object_2)
                 + ...;
assert!(total_locked > 0);   // trivially true after any update

// Now assert a withdrawal path exists to fee_recipient:
// ŌåÆ No such function exists in the module ŌåÆ assertion fails
// ŌåÆ Invariant: total_locked == 0 OR withdraw_path_exists is BROKEN
```

The existing test `test_create_and_update_price_feeds_with_batch_attestation_success` already asserts `price_info::get_balance(&price_info_object_1) == DEFAULT_BASE_UPDATE_FEE` after one update, confirming funds are locked. [6](#0-5) 

No further exploit construction is needed: the locked funds are observable on-chain via `get_balance`, and the absence of any withdrawal function is statically verifiable from the source.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
