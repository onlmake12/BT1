### Title
`MAX_DEPOSIT_LIMIT` Bypass for Permanent Subscriptions via `updateSubscription` — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
The `Scheduler` contract enforces a `MAX_DEPOSIT_LIMIT` (100 ETH) on deposits into permanent subscriptions in both `createSubscription()` and `addFunds()`, but this restriction is entirely absent in the `payable` `updateSubscription()` function. A subscription manager can bypass the cap by creating a non-permanent subscription and then calling `updateSubscription()` with `newParams.isPermanent = true` and `msg.value > MAX_DEPOSIT_LIMIT`, permanently locking an arbitrary amount of ETH beyond the intended ceiling.

### Finding Description
`SchedulerConstants.sol` defines:

```solidity
uint256 public constant MAX_DEPOSIT_LIMIT = 100 ether;
```

`createSubscription()` enforces it: [1](#0-0) 

`addFunds()` enforces it: [2](#0-1) 

`updateSubscription()` is also `payable` and unconditionally credits `msg.value` to the balance **before** any permanence check: [3](#0-2) 

The guard at line 90 only blocks calls where `currentParams.isPermanent` is already `true`. It does **not** block the transition from non-permanent → permanent, and it does **not** check whether `msg.value` exceeds `MAX_DEPOSIT_LIMIT`. The subscription is then written with the caller-supplied `newParams` (which may have `isPermanent = true`) at line 150: [4](#0-3) 

### Impact Explanation
Permanent subscriptions prohibit withdrawals (`withdrawFunds` reverts with `CannotUpdatePermanentSubscription`). The `MAX_DEPOSIT_LIMIT` is the only safeguard preventing an unbounded amount of ETH from being irrecoverably locked in a single permanent subscription. By bypassing it, an attacker can lock an arbitrary amount of ETH (e.g., 1000 ETH) permanently in the contract, circumventing the protocol's intended accounting invariant. While the ETH belongs to the caller, the protocol's design explicitly caps this at 100 ETH per permanent subscription; exceeding it violates that invariant and could be used as a griefing or economic manipulation vector.

### Likelihood Explanation
Any subscription manager — an unprivileged role obtainable by anyone who calls `createSubscription()` — can trigger this in two transactions with no special access. The path is direct and requires no external dependencies.

### Recommendation
Add the `MAX_DEPOSIT_LIMIT` check inside `updateSubscription()` when the incoming `newParams.isPermanent` is `true`:

```solidity
// In updateSubscription(), after adding msg.value to balance:
if (newParams.isPermanent && currentStatus.balanceInWei > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
```

This mirrors the pattern already used in `createSubscription()` and `addFunds()`. [5](#0-4) 

### Proof of Concept
1. Alice calls `createSubscription(params)` with `params.isPermanent = false` and `msg.value = minimumBalance`. She becomes the subscription manager.
2. Alice calls `updateSubscription(subscriptionId, newParams){value: 200 ether}` where `newParams.isPermanent = true`.
3. Inside `updateSubscription`: `currentStatus.balanceInWei += 200 ether` executes (line 87). The guard at line 90 passes because `currentParams.isPermanent == false`. No `MAX_DEPOSIT_LIMIT` check is performed. `_state.subscriptionParams[subscriptionId] = newParams` writes `isPermanent = true` (line 150).
4. The subscription is now permanent with 200 ETH locked — double the intended maximum — and `withdrawFunds` will revert for this subscription forever.
