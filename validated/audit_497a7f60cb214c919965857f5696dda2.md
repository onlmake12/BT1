### Title
Missing Sanity Check on `commitment` Parameter in `Entropy.register()` Allows Provider to Create Permanently Broken State, Causing User Fee Loss - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `Entropy.register()` function accepts `commitment = bytes32(0)` without any validation. A provider who registers (accidentally or intentionally) with a zero commitment creates a permanently broken provider state: the provider appears registered with available sequence numbers, but no request can ever be fulfilled. Users who pay fees to request randomness from this provider lose their funds permanently, as the required callback can never be delivered.

---

### Finding Description

`Entropy.register()` validates only that `chainLength != 0`, but performs no check on the `commitment` parameter:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,       // ← no zero-check
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    ...
    provider.currentCommitment = commitment;          // stored as bytes32(0)
    provider.endSequenceNumber = provider.sequenceNumber + chainLength;
    provider.sequenceNumber += 1;
```

When `commitment = bytes32(0)` and `chainLength > 1`:

1. The provider is registered with `sequenceNumber = 1`, `endSequenceNumber = chainLength`, `currentCommitment = bytes32(0)`.
2. Users can call `requestV2(provider, ...)` and pay fees. The fee is immediately credited to `provider.accruedFeesInWei`.
3. The stored request commitment is `req.commitment = keccak256(userCommitment ‖ bytes32(0))`.
4. When the provider (or anyone) attempts `revealWithCallback(...)`, the contract computes `constructProviderCommitment(numHashes=1, providerContribution) = keccak256(providerContribution)` and requires it to equal `bytes32(0)`.
5. Finding a preimage of `bytes32(0)` under keccak256 is computationally infeasible → every reveal reverts with `IncorrectRevelation`.
6. The provider can re-register with a valid commitment to fix future requests, but all in-flight requests remain permanently unfulfillable because their stored `req.commitment` was computed against `bytes32(0)`.

The only check that exists (`chainLength == 0`) does not protect against this broken state.

---

### Impact Explanation

Users who request randomness from a broken provider:
- Pay the full fee (`providerFee + pythFee`) at request time.
- Fees are immediately credited to `provider.accruedFeesInWei` and `_state.accruedPythFeesInWei`.
- The callback (`entropyCallback`) is never invoked.
- There is no refund mechanism for unfulfilled requests.
- The provider can withdraw accrued fees via `withdraw()` at any time.

Result: **permanent loss of user fee payments** for every request made against the broken provider, with no recovery path for those specific requests.

---

### Likelihood Explanation

Two realistic paths:

1. **Accidental**: A legitimate provider has a bug in their off-chain key-management or hash-chain generation code that produces `bytes32(0)` as the commitment (e.g., uninitialized buffer, serialization error). The contract silently accepts it. The provider's service is broken and users lose fees before the provider notices and re-registers.

2. **Intentional**: A malicious actor registers as a provider with `bytes32(0)`, advertises low fees, collects user fee payments, and withdraws them — all without ever being able to fulfill a single request. The contract provides no protection against this.

Both paths are permissionless and require no privileged access.

---

### Recommendation

Add a zero-commitment guard in `register()`:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    if (commitment == bytes32(0)) revert EntropyErrors.AssertionFailure(); // ADD THIS
    ...
}
```

A zero `bytes32` commitment is cryptographically meaningless as a hash-chain anchor and should never be a valid registration value.

---

### Proof of Concept

```solidity
// 1. Malicious/buggy provider registers with zero commitment
vm.prank(brokenProvider);
entropy.register(
    1 wei,           // feeInWei
    bytes32(0),      // commitment — zero, no check
    hex"",           // commitmentMetadata
    1000,            // chainLength — appears to have 999 slots
    bytes("https://broken.example.com")
);

// 2. User requests randomness, paying the fee
uint256 fee = entropy.getFee(brokenProvider);
vm.deal(user, fee);
vm.prank(user);
uint64 seqNum = entropy.requestV2{value: fee}(
    brokenProvider,
    bytes32(uint256(42)),  // userContribution
    0
);
// Fee is now credited to brokenProvider.accruedFeesInWei

// 3. Provider (or anyone) attempts to reveal — always reverts
// constructProviderCommitment(1, anyValue) = keccak256(anyValue) ≠ bytes32(0)
vm.expectRevert(EntropyErrors.IncorrectRevelation.selector);
entropy.revealWithCallback(
    brokenProvider,
    seqNum,
    bytes32(uint256(42)),
    bytes32(uint256(99))  // any providerRevelation
);

// 4. Provider withdraws stolen fees
vm.prank(brokenProvider);
entropy.withdraw(uint128(fee)); // succeeds — user's fee is gone
```
