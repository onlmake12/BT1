Audit Report

## Title
`Scheduler::resume()` Unconditionally Zeroes `iteration_cycles`, Permanently Discarding Pre-Suspension IO Cycles â€” (File: `script/src/scheduler.rs`)

## Summary

`Scheduler::resume()` sets `scheduler.iteration_cycles = 0` at line 242 after calling `ensure_vms_instantiated`, which discards the non-zero `iteration_cycles` value faithfully saved by `suspend()` into `FullSuspendedState`. Because `process_io()` charges cycles via `resume_vm` â†’ `SPAWN_EXTRA_CYCLES_BASE` into `iteration_cycles` after the per-iteration reset but before a `Pause` error propagates, those cycles are never committed to `total_cycles` and vanish permanently on resume, causing the scheduler to undercount consumed cycles across every suspend/resume boundary.

## Finding Description

**Three-accumulator invariant.** The scheduler's total consumed cycles equals `total_cycles + iteration_cycles + machine.cycles()`. `consumed_cycles()` returns only `total_cycles` (L156â€“158). `iterate_outer` is the sole site that commits `iteration_cycles` into `total_cycles` and resets it.

**How non-zero `iteration_cycles` arises at suspension.** Inside `iterate_outer` (L417â€“456):
1. `iterate_inner` runs the VM; cycles accumulate in `iteration_cycles`.
2. `consume_cycles(self.iteration_cycles)` commits them to `total_cycles` (L423).
3. `remaining_cycles = limit_cycles - self.iteration_cycles` is computed (L424â€“426).
4. `self.iteration_cycles = 0` â€” reset (L428).
5. `self.process_io()` is called (L453). `process_io` calls `ensure_get_instantiated` â†’ `ensure_vms_instantiated` â†’ `resume_vm`, which adds `SPAWN_EXTRA_CYCLES_BASE` directly to `self.iteration_cycles` (L954â€“957). After step 5, `iteration_cycles = SPAWN_EXTRA_CYCLES_BASE` (or more, for multiple VMs).
6. `iterate_return?` propagates `Err(Pause)` â€” the function exits with `iteration_cycles > 0` (L454).

**`suspend()` correctly preserves the value.** `FullSuspendedState` explicitly documents that `iteration_cycles` will not always be zero (types.rs L498â€“501), and `suspend()` saves it faithfully (L267).

**`resume()` discards it.** The scheduler is constructed with `iteration_cycles: full.iteration_cycles` (L216), then `ensure_vms_instantiated` is called (adding more cycles per VM via `resume_vm`), and finally `scheduler.iteration_cycles = 0` (L242) zeroes the entire field â€” including the pre-suspension value that was never committed to `total_cycles`. The comment at L239â€“241 states the intent is to not charge cycles for resume overhead, but the implementation also discards the pre-suspension cycles.

**Cycle loss is permanent.** After resume, `total_cycles` is correct for the committed portion, but the X cycles from `process_io()` before suspension are in neither `total_cycles` nor `iteration_cycles`. `consumed_cycles()` returns only `total_cycles`, so the delta reported by `chunk_run` (verify.rs L481â€“491) is understated by X per suspend/resume boundary.

**Cycle-limit enforcement is bypassed.** In `resume_from_state` (verify.rs L276â€“341), `current_used` accumulates these deltas; it is understated by NĂ—X across N suspend/resume boundaries. The `remain_cycles` check (`limit_cycles.checked_sub(current_used)`) therefore passes when it should fail, allowing subsequent script groups to consume more cycles than permitted.

## Impact Explanation

This is an **incorrect implementation of CKB-VM cycle accounting** (High, 10001â€“15000 points). The tx-pool's chunked verifier accepts transactions whose actual cycle consumption exceeds `max_block_cycles` because the reported cycle count is understated by NĂ—`SPAWN_EXTRA_CYCLES_BASE` per script group across N suspend/resume boundaries. When a miner includes such a transaction, the block validator (non-resumable, unaffected by this bug) correctly counts all cycles and rejects the block as exceeding the cycle limit, wasting miner resources and causing repeated chain-tip instability if the attack is sustained.

## Likelihood Explanation

Any unprivileged user can trigger this by submitting a transaction that uses `spawn` + `pipe` syscalls so that child VMs are in `WaitForRead`/`WaitForWrite` states when the tx-pool issues a `ChunkCommand::Suspend`. No key material, privileged access, or majority hashpower is required. The bypass magnitude per boundary is at least `SPAWN_EXTRA_CYCLES_BASE` per VM resumed by `process_io()`, and it compounds linearly with the number of suspend/resume cycles.

## Recommendation

In `resume()`, save the pre-suspension `iteration_cycles` before calling `ensure_vms_instantiated`, then restore it afterward so that only the cycles added by `ensure_vms_instantiated` itself (the resume overhead) are discarded:

```rust
let saved_iteration_cycles = full.iteration_cycles;
// ... construct scheduler with iteration_cycles: full.iteration_cycles ...
scheduler.ensure_vms_instantiated(&full.instantiated_ids).unwrap();
// Discard only the cycles charged by ensure_vms_instantiated (resume overhead),
// but restore the pre-suspension iteration_cycles not yet in total_cycles.
scheduler.iteration_cycles = saved_iteration_cycles;
```

Alternatively, `suspend()` should flush `iteration_cycles` into `total_cycles` before saving state, so `iteration_cycles` is always zero at suspension boundaries and the reset in `resume()` is safe.

## Proof of Concept

**Minimal trace:**

1. `iterate_outer` runs: VM executes 800 cycles â†’ `iteration_cycles = 800`.
2. `consume_cycles(800)` â†’ `total_cycles = 800`. `iteration_cycles = 0`.
3. `process_io()` calls `resume_vm` for a pending IO pair â†’ `iteration_cycles += SPAWN_EXTRA_CYCLES_BASE` (e.g., 50).
4. `iterate_return?` propagates `Err(Pause)`. `iteration_cycles = 50`.
5. `chunk_run` catches `Pause`, calls `scheduler.suspend()`.
6. `suspend()` saves: `total_cycles = 800`, `iteration_cycles = 50`.
7. `resume()` restores `total_cycles = 800`, calls `ensure_vms_instantiated` (adds Y cycles), then sets `iteration_cycles = 0`. The 50 cycles are permanently lost.
8. Next chunk runs with `limit_cycles = max_cycles`. Scheduler believes only 800 cycles consumed; actual is 850. Script gets 50 extra cycles of budget.

**Test plan:** Write a Rust integration test that creates a scheduler with a spawned child VM in `WaitForRead` state, runs until `Pause`, calls `suspend()` + `resume()`, and asserts that `consumed_cycles()` after resume equals `consumed_cycles()` before suspend plus the `iteration_cycles` saved in `FullSuspendedState`. The assertion will fail, demonstrating the cycle loss.
