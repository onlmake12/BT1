### Title
Irrecoverable User Fee Loss When Echo Callback Fails — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.executeCallback`, the provider's fee is credited and the request is permanently cleared **before** the consumer callback is invoked. The callback is wrapped in a `try/catch`, so any revert is silently swallowed. There is no retry mechanism and no refund path. When the callback fails, the user permanently loses their fee without receiving the price update service they paid for.

---

### Finding Description

`Echo.executeCallback` follows this sequence:

1. **Credit provider fee** (line 161–162):
```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

2. **Clear the request** (line 164):
```solidity
clearRequest(sequenceNumber);
```

3. **Execute callback with try/catch** (lines 176–201):
```solidity
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(...);
} catch {
    emit PriceUpdateCallbackFailed(...);
}
```

When the callback reverts (out-of-gas, logic error, or any other reason), the `catch` block emits `PriceUpdateCallbackFailed` and returns normally. At this point:

- The request has already been cleared — it cannot be retried.
- The provider has already been credited the full fee — it cannot be refunded.
- The user's `echoCallback` was never successfully executed — the service was never delivered.

The developer's own TODO comment at line 155–156 acknowledges the tension:

> `// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.`
> `// If executeCallback can revert, then funds can be permanently locked in the contract.`

The try/catch was added to prevent the whole function from reverting, but this creates the inverse problem: the fee is permanently transferred to the provider even when the callback fails, with no recovery path for the user.

This contrasts directly with the Entropy contract, which introduces a `CALLBACK_FAILED` state (Entropy.sol lines 651, 554–556) that keeps the request alive and allows the callback to be retried after the root cause is fixed. Echo has no equivalent mechanism.

---

### Impact Explanation

A user who calls `requestPriceUpdatesWithCallback` and whose `_echoCallback` reverts for any reason permanently loses the fee they paid. The request is gone, the provider keeps the fee, and there is no on-chain path to retry or recover. This is a direct loss of user funds with no admin recovery option.

Additionally, a provider can exploit this by calling `executeCallback` with just enough gas to pass the outer function but insufficient gas to forward `req.callbackGasLimit` to the callback (the IEcho.sol NatSpec at line 62–64 documents a 1.5× gas requirement but the contract does not enforce it). The callback fails OOG, the provider collects the full fee, and the user receives nothing.

---

### Likelihood Explanation

Callback gas estimation is inherently imprecise. Any consumer whose callback uses slightly more gas than estimated at request time will trigger this silently. This is a normal operational condition, not an exotic edge case. The lack of a minimum gas enforcement in `executeCallback` makes the provider-griefing variant trivially reachable by any registered provider.

---

### Recommendation

1. **Add a retry mechanism**: Introduce a `CALLBACK_FAILED` state analogous to Entropy's `EntropyStatusConstants.CALLBACK_FAILED`. Keep the request alive when the callback fails so it can be retried once the consumer fixes their callback.

2. **Enforce gas forwarding**: Before executing the callback, verify that `gasleft() >= req.callbackGasLimit * 64 / 63` (accounting for the 63/64 EVM gas-passing rule) and revert with `InsufficientGas` if not. This prevents providers from collecting fees while deliberately starving the callback of gas.

3. **Separate fee crediting from callback success**: Only credit the provider's fee after a successful callback, or hold the fee in escrow and release it on retry success.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 100_000`, paying the required fee.
2. User's `echoCallback` implementation requires 110,000 gas (slightly underestimated).
3. Provider calls `executeCallback`.
4. Lines 161–162: provider's `accruedFeesInWei` is incremented by the full fee.
5. Line 164: `clearRequest(sequenceNumber)` removes the request from storage.
6. Lines 176–179: `_echoCallback` is called with `gas: 100_000`; it runs out of gas and reverts.
7. Lines 192–200: The `catch` block emits `PriceUpdateCallbackFailed` and returns normally.
8. Result: provider has the fee, request is gone, user's callback was never executed. No on-chain recovery path exists.
