### Title
Duplicate Out-Points Within a Dep Group Inflate Unmetered Node Validation Work - (`File: util/types/src/core/cell.rs`)

---

### Summary

`resolve_transaction_dep` does not deduplicate sub-out-points parsed from a dep group cell's data. An unprivileged transaction sender can craft a dep group cell containing up to `MAX_DEP_EXPANSION_LIMIT` (2048) identical out-points, causing every validating node to perform up to 2048× more unmetered CPU work in `TxData::new` during script verification, compared to a semantically equivalent transaction with a single-entry dep group.

---

### Finding Description

When a transaction references a `DepGroup`-typed `CellDep`, CKB expands it by reading the dep group cell's data as an `OutPointVec` and resolving each sub-out-point into a `CellMeta`. This expansion is performed in `resolve_transaction_dep` (`util/types/src/core/cell.rs`, lines 807–841):

```rust
let sub_out_points =
    parse_dep_group_data(data).map_err(|_| OutPointError::InvalidDepGroup(outpoint))?;

*remaining_dep_slots = remaining_dep_slots
    .checked_sub(sub_out_points.len())
    .ok_or(OutPointError::OverMaxDepExpansionLimit)?;

for sub_out_point in sub_out_points.into_iter() {
    resolved_cell_deps.push(cell_resolver(&sub_out_point, eager_load)?);
}
```

No deduplication is performed on `sub_out_points`. The existing `DuplicateDepsVerifier` (`verification/src/transaction_verifier.rs`, lines 437–458) only checks for duplicate `CellDep` entries at the transaction level (i.e., duplicate dep group out-points in the transaction's `cell_deps` list); it does **not** inspect the contents of a dep group cell's data for internal duplicates.

This is confirmed by the integration test `test_duplicate_in_depgroup_type` (`test/src/specs/hardfork/v2021/cell_deps.rs`, lines 800–810), which explicitly expects `PASS` for a dep group containing duplicate out-points.

The only guard is `MAX_DEP_EXPANSION_LIMIT` (2048), which bounds the total number of expanded entries but still allows up to 2048 duplicate entries from a single dep group.

All 2048 expanded `CellMeta` entries (including duplicates) are pushed into `resolved_cell_deps` and subsequently iterated in `TxData::new` (`script/src/types.rs`, lines 699–714):

```rust
for (i, cell_meta) in resolved_cell_deps.iter().enumerate() {
    let data_hash = data_loader
        .load_cell_data_hash(cell_meta)
        .expect("cell data hash");
    let lazy = LazyData::from_cell_meta(cell_meta);
    binaries_by_data_hash.insert(data_hash.to_owned(), (i, lazy.to_owned()));

    if let Some(t) = &cell_meta.cell_output.type_().to_opt() {
        binaries_by_type_hash
            .entry(t.calc_script_hash())
            .and_modify(|bin| bin.merge(&data_hash))
            .or_insert_with(|| Binaries::new(data_hash.to_owned(), i, lazy.to_owned()));
    }
}
```

For each of the 2048 duplicate entries, the node:
- Calls `load_cell_data_hash` (may fall back to DB if not in memory cache)
- Allocates a new `LazyData` wrapper via `LazyData::from_cell_meta`
- Overwrites `binaries_by_data_hash` with a redundant insert
- Calls `Binaries::merge` on `binaries_by_type_hash` for each duplicate (if the cell has a type script)

This work is **not cycle-metered** — it is pure node CPU overhead paid by every validating node, not by the transaction submitter.

---

### Impact Explanation

Every full node that validates the malicious transaction must perform O(2048) work in `TxData::new` instead of O(1), for what is semantically a single-cell dep. This overhead is incurred during block validation (which must complete within the block interval) and during tx-pool admission. A sustained stream of such transactions can inflate per-transaction validation cost by up to ~2048×, degrading block propagation and potentially causing nodes to fall behind the chain tip. The `MAX_DEP_EXPANSION_LIMIT` bounds the worst case but does not eliminate the griefing surface.

---

### Likelihood Explanation

The attack requires no special privilege. Any RPC caller or P2P transaction relayer can:
1. Publish a live cell whose data is an `OutPointVec` with 2048 copies of the same valid out-point.
2. Submit transactions referencing that cell as a `DepGroup`-typed `CellDep`.

The dep group cell itself is structurally valid (it parses as a non-empty `OutPointVec`). The referencing transaction passes `DuplicateDepsVerifier`, size checks, and all other non-contextual verifiers. The attack is cheap to mount and can be repeated across many transactions.

---

### Recommendation

In `resolve_transaction_dep` (`util/types/src/core/cell.rs`), after parsing `sub_out_points`, deduplicate them before resolving and before decrementing `remaining_dep_slots`. A `HashSet` check is sufficient:

```rust
let mut seen = HashSet::with_capacity(sub_out_points.len());
for sub_out_point in sub_out_points.into_iter() {
    if !seen.insert(sub_out_point.clone()) {
        return Err(OutPointError::InvalidDepGroup(outpoint));
    }
    *remaining_dep_slots = remaining_dep_slots
        .checked_sub(1)
        .ok_or(OutPointError::OverMaxDepExpansionLimit)?;
    resolved_cell_deps.push(cell_resolver(&sub_out_point, eager_load)?);
}
```

Alternatively, enforce that sub-out-points within a dep group are strictly ordered (by tx-hash then index), which both prevents duplicates and enables binary search during lookup.

---

### Proof of Concept

1. Create a cell whose `outputs_data` is `OutPointVec::new_builder().set(vec![op; 2048]).build().as_bytes()` where `op` is any live out-point.
2. Submit a transaction with `cell_deps: [CellDep { out_point: <above cell>, dep_type: DepGroup }]`.
3. The transaction passes all validation checks and is accepted into the tx-pool.
4. Every node that validates this transaction executes the loop in `TxData::new` 2048 times instead of once, with no additional cycle cost to the attacker.

    pub fn verify(&self) -> Result<(), Error> {
        let transaction = self.transaction;
        let mut seen_cells = HashSet::with_capacity(self.transaction.cell_deps().len());
        let mut seen_headers = HashSet::with_capacity(self.transaction.header_deps().len());

        if let Some(dep) = transaction
            .cell_deps_iter()
            .find_map(|dep| seen_cells.replace(dep))
        {
            return Err(TransactionError::DuplicateCellDeps {
                out_point: dep.out_point(),
            }
            .into());
        }
        if let Some(hash) = transaction
            .header_deps_iter()
            .find_map(|hash| seen_headers.replace(hash))
        {
            return Err(TransactionError::DuplicateHeaderDeps { hash }.into());
        }
        Ok(())
    }
