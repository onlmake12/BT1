### Title
`executeCallback()` Reverts Due to Pyth Oracle Fee Exceeding Stored Provider Fee — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`, the fee stored in `req.fee` at request time is computed as `msg.value - _state.pythFeeInWei`. At execution time, `executeCallback` subtracts the actual Pyth oracle fee (`pyth.getUpdateFee(updateData)`) from `req.fee`. If the actual Pyth oracle fee exceeds the provider fee portion stored in `req.fee`, the subtraction underflows (Solidity 0.8+ reverts), permanently locking user funds with no refund path.

### Finding Description

In `requestPriceUpdatesWithCallback`, the fee stored for the request is:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
```

`_state.pythFeeInWei` is the Echo protocol fee. The remainder (`req.fee`) is the provider fee portion, which is supposed to cover the actual Pyth oracle update fee. [1](#0-0) 

In `executeCallback`, the provider's accrued fees are computed as:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The `pythFee` here is the **live** Pyth oracle fee at execution time, not the fee charged to the user at request time. The contract's own comment acknowledges this decoupling:

```
// Note: The provider needs to set its fees to include the fee charged by the Pyth contract.
// Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the
// fee computation on IPyth assumes it has the full updated data.
``` [3](#0-2) 

If `pyth.getUpdateFee(updateData) > req.fee + msg.value`, the subtraction underflows and the entire `executeCallback` call reverts. Because `executeCallback` is the only fulfillment path and there is no user-facing refund function, the user's ETH is permanently locked in the contract.

The `getFee` view function returns `pythFeeInWei + providerBaseFee + providerFeedFee + gasFee`, where `pythFeeInWei` is the Echo protocol fee — **not** the Pyth oracle update fee. The Pyth oracle fee is dynamic and can change independently via Pyth governance. [4](#0-3) 

### Impact Explanation

When `pyth.getUpdateFee(updateData) > req.fee + msg.value`:

1. `executeCallback` reverts at the fee accounting line before the callback is made.
2. The request is never cleared (`clearRequest` is never reached).
3. The user's ETH (paid at request time) is permanently locked — there is no `refund` or `cancelRequest` function.
4. The provider cannot fulfill the request regardless of how many times they retry, because the fee mismatch is structural.

This is a direct analog to the external report: a fee deduction causes an amount to fall below an expected threshold, causing a revert that permanently fails the operation.

### Likelihood Explanation

- The Pyth oracle fee (`pyth.getUpdateFee`) is governance-controlled and can increase at any time.
- Providers set their fees at registration time and may not update them promptly after a Pyth oracle fee increase.
- Any executor calling `executeCallback` with `msg.value = 0` (the standard pattern, as shown in tests) will trigger the revert if the Pyth oracle fee has risen above the provider's configured fee.
- The `executeCallback` function is permissionless and callable by any address, including the Fortuna keeper. [5](#0-4) 

### Recommendation

Replace the fee accounting in `executeCallback` to use the Echo protocol fee (`_state.pythFeeInWei`) as the Pyth oracle fee budget, rather than subtracting the live `pythFee` from `req.fee`:

```solidity
// Instead of:
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

// Use:
require(pythFee <= _state.pythFeeInWei + req.fee + msg.value, "Insufficient fee for Pyth oracle");
_state.accruedFeesInWei += SafeCast.toUint128(_state.pythFeeInWei);
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

Additionally, add a `cancelRequest` / refund path so users can reclaim funds if a request becomes permanently unfulfillable.

### Proof of Concept

1. Provider registers with `baseFeeInWei = 500 wei`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`.
2. Echo protocol fee `_state.pythFeeInWei = 100 wei`.
3. User calls `requestPriceUpdatesWithCallback{value: 600}(...)`.
   - `req.fee = 600 - 100 = 500 wei`
4. Pyth governance increases the oracle update fee. Now `pyth.getUpdateFee(updateData) = 600 wei`.
5. Executor calls `executeCallback(provider, seqNum, updateData, priceIds)` with `msg.value = 0`.
   - `(req.fee + msg.value) - pythFee = (500 + 0) - 600` → **underflow → revert**
6. User's 600 wei is permanently locked. No refund path exists.

    function getFee(
        address provider,
        uint32 callbackGasLimit,
        bytes32[] calldata priceIds
    ) public view override returns (uint96 feeAmount) {
        uint96 baseFee = _state.pythFeeInWei; // Fixed fee to Pyth
        // Note: The provider needs to set its fees to include the fee charged by the Pyth contract.
        // Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the
        // fee computation on IPyth assumes it has the full updated data.
        uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
        uint96 providerFeedFee = SafeCast.toUint96(
            priceIds.length * _state.providers[provider].feePerFeedInWei
        );
        uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei; // Provider's per-gas rate
        uint256 gasFee = callbackGasLimit * providerFeeInWei; // Total provider fee based on gas
        feeAmount =
            baseFee +
            providerBaseFee +
            providerFeedFee +
            SafeCast.toUint96(gasFee); // Total fee user needs to pay
    }
