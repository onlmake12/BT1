### Title
Missing Payload Timestamp Staleness Check in `verifyUpdate` Allows Arbitrarily Old Lazer Price Updates to Pass Verification — (`lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

The `verifyUpdate` function in `PythLazer.sol` verifies the ECDSA signature and signer key expiry but performs **no check** that the price payload's embedded timestamp is recent relative to `block.timestamp`. A Lazer price update transaction can sit in the mempool for an extended period and, when eventually executed, will pass verification with a price that is arbitrarily stale. This is the direct EVM analog of the missing deadline check described in the external report.

---

### Finding Description

`PythLazer.verifyUpdate` is the sole on-chain verification entry point for Pyth Lazer (Pro) price updates on EVM chains. It performs the following checks:

1. Fee payment (`msg.value >= verification_fee`)
2. EVM format magic (`EVM_FORMAT_MAGIC`)
3. ECDSA signature recovery
4. Signer key expiry via `isValidSigner` (`block.timestamp < trustedSignerToExpiresAtMapping[signer]`) [1](#0-0) 

What it does **not** check is the timestamp embedded inside the payload itself. The payload bytes contain a `uint64` timestamp (in microseconds) as the first field after the magic, which is parsed by `PythLazerLib.parsePayloadHeader`: [2](#0-1) 

This timestamp is never compared to `block.timestamp` anywhere inside `verifyUpdate`. The contract only checks that the *signing key* has not expired — not that the *price data* is fresh.

The `isValidSigner` check: [3](#0-2) 

guards against a rotated/revoked key, but a valid key can sign a payload with a timestamp from hours or days ago, and the contract will accept it without complaint.

Unlike Pyth Core, which provides `getPriceNoOlderThan` as a built-in staleness guard at the read layer, Lazer has no equivalent on-chain primitive. Consumers receive the raw payload and must manually parse the timestamp and compare it to `block.timestamp`. The official integration guide shows consumers checking `_timestamp > timestamp` (monotonicity only), not `_timestamp ≈ block.timestamp` (freshness): [4](#0-3) 

This means the contract provides no on-chain freshness guarantee, which directly contradicts Lazer's core value proposition of ultra-low-latency price feeds.

---

### Impact Explanation

A user or protocol submits a transaction that includes a Lazer price update (e.g., to open a leveraged position). Due to mempool congestion or deliberate gas-price manipulation by an adversary, the transaction is delayed. When it executes minutes or hours later, `verifyUpdate` accepts the update because the signer key is still valid. The consumer contract receives a cryptographically "verified" price that is significantly stale. If the consumer does not independently enforce a tight staleness window (which many will not, given the absence of a built-in helper), trades execute at sub-optimal or manipulable prices — exactly the harm described in the external report.

Pyth's own best-practices documentation acknowledges this class of risk for pull-oracle integrations: [5](#0-4) 

---

### Likelihood Explanation

Mempool congestion is routine on all major EVM chains. An adversary who can observe a pending Lazer-update transaction can also manipulate its execution timing by front-running with higher-gas transactions to fill blocks, causing the victim's transaction to execute with a stale price. The entry path is fully permissionless — any address can call `verifyUpdate`.

---

### Recommendation

Add a configurable `maxAgeSeconds` parameter (or a contract-level constant) to `verifyUpdate`. After recovering the payload, parse the timestamp and enforce:

```solidity
uint64 payloadTimestampUs = uint64(bytes8(payload[4:12])); // after magic
uint256 payloadTimestampSec = payloadTimestampUs / 1_000_000;
require(
    block.timestamp <= payloadTimestampSec + maxAgeSeconds,
    "Lazer payload too old"
);
```

This mirrors the `ensure(deadline)` modifier pattern from Uniswap V2 and the `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` check already present in the Pulse Scheduler: [6](#0-5) 

---

### Proof of Concept

1. Obtain a valid Lazer signed update at block time `T` (e.g., BTC/USD = $100,000).
2. Submit a transaction calling a consumer contract that calls `PythLazer.verifyUpdate(update)` and uses the returned price.
3. Do not broadcast with competitive gas; let the transaction sit in the mempool.
4. At block time `T + 3600` (1 hour later, BTC/USD = $95,000), the transaction is mined.
5. `verifyUpdate` passes: the signer key is still valid (`block.timestamp < expiresAt`), the signature is valid, the magic is correct.
6. The consumer receives a verified price of $100,000 — 5% above the current market — and executes a trade at that stale price.
7. No revert occurs; no staleness error is emitted. [1](#0-0)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
