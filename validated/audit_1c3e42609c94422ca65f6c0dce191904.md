### Title
Targeted Tx-Pool Eviction via RBF Descendant-Boost Manipulation — (`tx-pool/src/component/entry.rs`, `tx-pool/src/component/pool_map.rs`, `tx-pool/src/pool.rs`, `tx-pool/src/process.rs`)

---

### Summary

An unprivileged transaction sender can cause a specific victim transaction to be evicted from the CKB tx-pool by exploiting the interaction between the `EvictKey` descendant-fee-rate boost and RBF's asymmetric input-dropping rule. The attacker first artificially elevates the victim's eviction priority (protecting it), fills the pool to near capacity, then uses RBF to replace their descendant with a non-descendant transaction — collapsing the victim's evict key — and triggering `limit_size` eviction of the victim's transaction.

---

### Finding Description

**`EvictKey` and descendant boost**

Each pool entry carries an `EvictKey` computed as:

```rust
// tx-pool/src/component/entry.rs  lines 234-247
let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
let feerate = FeeRate::calculate(entry.fee, weight);
EvictKey {
    fee_rate: descendants_feerate.max(feerate),
    ...
}
```

A transaction with a high-fee descendant therefore receives a high `EvictKey` and is protected from `limit_size` eviction. When that descendant is removed, `update_ancestors_index_key` immediately recalculates the ancestor's `evict_key`:

```rust
// tx-pool/src/component/pool_map.rs  lines 432-444
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors = self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();   // key is live-updated
        });
    }
}
```

**RBF Rule #2 allows dropping inputs**

`check_rbf` Rule #2 only rejects a replacement that *adds* new unconfirmed inputs; it does not require the replacement to *keep* all inputs of the replaced transaction:

```rust
// tx-pool/src/pool.rs  lines 596-609
if tx_inputs
    .iter()
    .any(|pt| !inputs.contains(pt) && !snapshot.transaction_exists(&pt.tx_hash()))
{
    return Err(Reject::RBFRejected("new Tx contains unconfirmed inputs"...));
}
```

This means a replacement transaction can silently drop the victim's output from its input set, severing the descendant relationship.

**`limit_size` runs immediately after RBF removal**

In `submit_entry`, the sequence is:

```rust
// tx-pool/src/process.rs  lines 136-152
let may_recovered_txs = self.process_rbf(tx_pool, &entry, &conflicts); // removes old tx, updates evict keys
let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?; // adds new tx
...
tx_pool.limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
    .map_or(Ok(()), Err)?;  // evicts lowest-evict-key tx if pool is full
```

`limit_size` evicts the entry with the smallest `EvictKey` (lowest fee rate, then fewest descendants, then latest timestamp):

```rust
// tx-pool/src/pool.rs  lines 298-326
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    if let Some(id) = next_evict_entry() {
        let removed = self.pool_map.remove_entry_and_descendants(&id);
        ...
    }
}
```

**Complete attack sequence**

1. Victim submits `Tx_V` (low fee rate `f_v`) to the pool.
2. Attacker submits `Tx_A` with inputs `[Tx_V_output, confirmed_output_X]` and high fee rate `f_a >> f_v`. `Tx_A` is a descendant of `Tx_V`; `Tx_V`'s `EvictKey.fee_rate` is raised to `f_a` (protected).
3. Attacker fills the pool to just below `max_tx_pool_size` with their own transactions, all with fee rates between `f_v` and `f_a`.
4. Attacker submits `Tx_B` with inputs `[confirmed_output_X]` only (dropping `Tx_V_output`), fee `> Tx_A_fee + min_rbf_fee`. `Tx_B` conflicts with `Tx_A` on `confirmed_output_X` → RBF triggers.
5. `process_rbf` removes `Tx_A`. `update_ancestors_index_key` fires, calling `sub_descendant_weight` on `Tx_V`, dropping its `EvictKey.fee_rate` back to `f_v`.
6. `Tx_B` is added (not a descendant of `Tx_V`). Pool is now over `max_tx_pool_size`.
7. `limit_size` runs. `Tx_V` is now the lowest `EvictKey` entry and is evicted.

---

### Impact Explanation

A targeted victim transaction is silently removed from the tx-pool without any consensus violation. The victim's transaction is denied confirmation. For time-sensitive transactions (e.g., a transaction that must be confirmed before a lock expires, or a transaction racing against another party), this constitutes a targeted denial-of-confirmation attack with direct financial impact on the victim. The attacker pays only gas (RBF fees) and the cost of temporarily holding capital in pool-filling transactions.

---

### Likelihood Explanation

RBF must be enabled (`min_rbf_rate > min_fee_rate`), which is the default configuration. The attacker needs enough capital to fill the pool (~180 MB default) and pay RBF fees. The attack requires no privileged access, no Sybil capability, and no majority hashpower. It is executable by any unprivileged RPC caller or transaction sender. Likelihood is **medium**: capital-intensive but straightforward for a well-resourced attacker targeting a high-value victim transaction.

---

### Recommendation

1. **Bound the descendant-boost contribution to `EvictKey`**: The `EvictKey` should not be fully determined by a single high-fee descendant that the attacker controls. Consider using the transaction's own fee rate as the primary eviction key, with descendant fee rate as a secondary tiebreaker only.

2. **Symmetric RBF input rule**: Rule #2 should also require that the replacement transaction retains all unconfirmed inputs of the replaced transaction (not just prohibit adding new ones). This prevents the attacker from severing the descendant relationship via RBF.

3. **Decouple `limit_size` from RBF submission**: After an RBF replacement, re-evaluate eviction candidates against the pool state *before* the replacement, not after, to prevent the replacement from being used as a trigger for targeted eviction.

---

### Proof of Concept

```
Setup:
  min_rbf_rate > min_fee_rate  (RBF enabled, default config)
  max_tx_pool_size = 180_000_000 bytes

Step 1: Victim submits Tx_V (fee_rate = 1000 shannons/KW, size = S_v)
        Tx_V.EvictKey.fee_rate = 1000

Step 2: Attacker submits Tx_A:
          inputs  = [Tx_V:output_0, confirmed_UTXO_X]
          fee_rate = 100_000 shannons/KW
        → Tx_V.EvictKey.fee_rate updated to 100_000 (protected)

Step 3: Attacker submits N filler transactions (fee_rate ~= 2000 shannons/KW)
        until pool_size = max_tx_pool_size - size(Tx_B)

Step 4: Attacker submits Tx_B:
          inputs  = [confirmed_UTXO_X]          ← Tx_V_output DROPPED
          fee     = Tx_A.fee + min_rbf_fee + 1
        RBF Rule #2 passes: Tx_B adds no new unconfirmed inputs.

Step 5: process_rbf removes Tx_A.
        update_ancestors_index_key fires on Tx_V:
          sub_descendant_weight(Tx_A)
          Tx_V.EvictKey.fee_rate = 1000  ← collapsed

Step 6: Tx_B added. pool_size > max_tx_pool_size.
        limit_size() runs.
        next_evict_entry() returns Tx_V (lowest EvictKey).
        Tx_V is evicted. Victim's transaction is gone.
```

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
