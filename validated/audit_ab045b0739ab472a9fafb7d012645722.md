Audit Report

## Title
Peer Store Permanently Blocked by `Group::None` Flooding via Onion3 Discovery Addresses — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
The `check_purge` second-pass eviction logic in `peer_store_impl.rs` uses `take(len / 2)` where `len` is the number of distinct network groups. When all 16384 stored addresses map to `Group::None` (as Onion3 addresses do, since `multiaddr_to_socketaddr` returns `None` for them), integer division yields `take(0)`, evicting nothing. Combined with the first pass being a no-op for fresh entries, `check_purge` returns `Err(PeerStoreError::EvictionFailed)` permanently, blocking all future `add_addr` calls.

## Finding Description

**Root cause — integer division in second-pass eviction:**

In `check_purge`, when the first pass evicts nothing (all entries are connectable), the second pass groups addresses by network group and calls `take(len / 2)`: [1](#0-0) 

With all 16384 entries mapping to `Group::None`, `peers_by_network_group.len() == 1`, so `take(1 / 2)` = `take(0)` due to Rust integer division, iterating over zero groups and producing an empty `candidate_peers`.

**`Group::None` for Onion3 addresses:**

`Group::from(&Multiaddr)` calls `multiaddr_to_socketaddr`. Onion3 addresses have no socket address representation, so they return `None` and fall through to `Group::None`: [2](#0-1) 

**First pass is a no-op for fresh entries:**

`AddrInfo::new` initializes `attempts_count = 0` and `last_connected_at_ms = 0`: [3](#0-2) 

In `is_connectable`, the only path returning `false` for `last_connected_at_ms == 0` requires `attempts_count >= ADDR_MAX_RETRIES` (i.e., `0 >= 3`), which is false. All 16384 freshly-added Onion3 entries are considered connectable, so the first pass evicts nothing: [4](#0-3) 

**Permanent `EvictionFailed`:**

With `candidate_peers` empty, `check_purge` returns the error: [5](#0-4) 

Every subsequent `add_addr` call hits the same path since the store remains at 16384 entries and the same logic repeats indefinitely: [6](#0-5) 

**Reachability via discovery — no rate limiting:**

`DiscoveryAddressManager::add_new_addrs` calls `peer_store.add_addr` for each received address with no per-source quota: [7](#0-6) 

Critically, `is_valid_addr` explicitly passes Onion3 addresses (returns `true` for `None` from `multiaddr_to_socketaddr`): [8](#0-7) 

## Impact Explanation

Once the peer store is saturated with `Group::None` entries, no legitimate IPv4/IPv6 peer address can ever be added. The node permanently loses the ability to discover new peers, reconnect after disconnects, or bootstrap. This matches **Medium (2001–10000): Suboptimal implementation of CKB state storage mechanism** — the peer store is the node's peer address storage mechanism, and this flaw allows an attacker to render it permanently inoperable for new entries.

## Likelihood Explanation

An attacker needs only P2P connections to a victim node. Each non-announce `Nodes` response can carry up to `MAX_ADDR_TO_SEND = 1000` items × `MAX_ADDRS = 3` addresses = 3000 Onion3 addresses per session. Across ~6 sessions (or via repeated announce messages over time), the attacker saturates all 16384 slots. The attack is cheap, deterministic, requires no special privileges beyond P2P connectivity, and is not rate-limited at the `add_addr` level. The Onion3 35-byte host field provides a vast unique address space. [9](#0-8) 

## Recommendation

1. Replace `take(len / 2)` with `take(len.saturating_add(1) / 2)` (ceiling division) so a single group is still considered for eviction.
2. Alternatively, use `take(std::cmp::max(1, len / 2))`.
3. Additionally, consider assigning each `Group::None` address its own unique group key (e.g., a hash of the full multiaddr) to prevent all non-IP addresses from collapsing into one eviction-resistant bucket. [10](#0-9) 

## Proof of Concept

```rust
#[test]
fn test_group_none_eviction_failure() {
    use crate::peer_store::{PeerStore, ADDR_COUNT_LIMIT};
    use crate::Flags;
    use p2p::multiaddr::Multiaddr;

    let mut store = PeerStore::default();

    // Fill store with 16384 unique Onion3 addresses (all map to Group::None)
    for i in 0u64..ADDR_COUNT_LIMIT as u64 {
        let mut host = [0u8; 35];
        host[..8].copy_from_slice(&i.to_le_bytes());
        let onion_addr: Multiaddr = format!(
            "/onion3/{}:1234",
            base32::encode(base32::Alphabet::RFC4648 { padding: false }, &host)
        ).parse().unwrap();
        let _ = store.add_addr(onion_addr, Flags::COMPATIBILITY);
    }

    // Attempt to add a legitimate IPv4 peer — returns Err(EvictionFailed)
    let ipv4_addr: Multiaddr = "/ip4/1.2.3.4/tcp/8115".parse().unwrap();
    let result = store.add_addr(ipv4_addr, Flags::COMPATIBILITY);
    assert!(result.is_ok(), "Legitimate peer should be addable after eviction");
}
```

The test will fail at the assertion, confirming the permanent `EvictionFailed` state. The fix can be verified by changing line 376 of `peer_store_impl.rs` from `.take(len / 2)` to `.take(std::cmp::max(1, len / 2))` and observing the test pass.

        if let Some(socket_addr) = multiaddr_to_socketaddr(multiaddr) {
            let ip_addr = socket_addr.ip();
            if ip_addr.is_loopback() {
                return Group::LocalNetwork;

            // IPv4 NetworkGroup

                let bits = ipv4.octets();
                return Group::IP4([bits[0], bits[1]]);
            }
            // IPv6 NetworkGroup
            if let IpAddr::V6(ipv6) = ip_addr {
                if let Some(ipv4) = ipv6.to_ipv4() {

                    return Group::IP4([bits[0], bits[1]]);
                }
                let bits = ipv6.octets();
                return Group::IP6([bits[0], bits[1], bits[2], bits[3]]);
            }
        }

        Group::None
    }

        // do not remove addr tried in last minute

        }
        // we give up if never connect to this addr
