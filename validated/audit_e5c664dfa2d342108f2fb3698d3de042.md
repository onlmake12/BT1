### Title
Inconsistent Fee-Rate Metric Between Pool Admission Check and Actual Entry Fee-Rate Calculation Allows Compute-Heavy Transactions to Bypass `min_fee_rate` — (`tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission check (`check_tx_fee`) computes the minimum required fee using only the raw serialized byte size of the transaction. However, the actual fee rate stored in `TxEntry` and used for pool sorting, eviction, and block assembly uses `get_transaction_weight(size, cycles) = max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For compute-heavy transactions where `cycles * DEFAULT_BYTES_PER_CYCLES > size`, the weight is larger than the size, meaning the admission check is systematically less strict than the actual fee-rate metric. An unprivileged transaction sender can craft a compute-heavy transaction that passes the `min_fee_rate` admission gate but whose actual effective fee rate is below `min_fee_rate`, enabling mempool spam and resource exhaustion.

---

### Finding Description

**Two divergent fee-rate calculations for the same transaction:**

**Path 1 — Pool admission (`check_tx_fee`):**

```rust
// tx-pool/src/util.rs, line 45
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The minimum fee is computed as `min_fee_rate * tx_size / 1000`, using only the raw serialized size.

The code itself acknowledges the inconsistency with a comment:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
``` [1](#0-0) 

**Path 2 — Actual fee rate in `TxEntry`:**

```rust
// tx-pool/src/component/entry.rs, line 115-118
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

The weight function is:

```rust
// util/types/src/core/tx_pool.rs, line 298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [4](#0-3) 

**The divergence:** For a transaction with `size = S` bytes and `cycles = C`, the admission check requires `fee >= min_fee_rate * S / 1000`. But the actual fee rate stored in the pool entry is `fee * 1000 / max(S, C * 0.000170571)`. When `C * 0.000170571 > S` (compute-heavy transaction), the actual fee rate is strictly less than `fee * 1000 / S`, so a transaction can satisfy the admission check while having an actual fee rate below `min_fee_rate`.

**Concrete example:**
- `min_fee_rate = 1000 shannons/KW` (default)
- Transaction: `size = 200 bytes`, `cycles = 2,000,000`
- Weight = `max(200, 2,000,000 * 0.000170571)` = `max(200, 341)` = `341`
- Admission requires: `fee >= 1000 * 200 / 1000 = 200 shannons`
- Actual fee rate with `fee = 200`: `200 * 1000 / 341 = 586 shannons/KW` — **below `min_fee_rate` of 1000**

The `pre_check` function in `tx-pool/src/process.rs` calls `check_tx_fee` with `tx_size = tx.data().serialized_size_in_block()`, confirming the size-only path is used at admission time. [5](#0-4) 

The same inconsistency propagates to pool sorting (`AncestorsScoreSortKey`) and eviction (`EvictKey`), both of which use `get_transaction_weight`: [6](#0-5)

### Impact Explanation

An unprivileged transaction sender can submit compute-heavy transactions that:

1. Pass the `min_fee_rate` admission gate (because `fee >= min_fee_rate * size / 1000`)
2. Have an actual effective fee rate below `min_fee_rate` (because `fee / weight < min_fee_rate`)
3. Occupy mempool slots, consuming memory and CPU verification resources
4. Get relayed to peers (who apply the same flawed admission check), propagating the spam network-wide

The pool's eviction logic will eventually evict these low-weight-fee-rate transactions when the pool fills, but until then they consume resources and displace legitimate transactions. The `min_fee_rate` policy — intended to prevent mempool spam — is effectively bypassed for the class of compute-heavy transactions.

---

### Likelihood Explanation

The attack is straightforward: any RPC caller with access to `send_transaction` can submit transactions with high cycle counts and fees calibrated to pass only the size-based check. CKB scripts (lock/type scripts) naturally produce high cycle counts. The attacker does not need any special privilege, key, or majority hash power. The default `min_fee_rate` of 1000 shannons/KW and `DEFAULT_BYTES_PER_CYCLES = 0.000170571` create a predictable gap that can be calculated precisely. [7](#0-6) 

---

### Recommendation

Replace the size-only weight in `check_tx_fee` with `get_transaction_weight(tx_size, cycles)`. Since cycles are not yet known at the pre-check stage (before script execution), the check should be deferred or repeated after verification using the verified cycle count. Alternatively, use the declared cycles (when provided by the relayer) as an upper bound for the weight in the admission check, consistent with how `max_cycles` is set from `declared_cycles` in `_process_tx`. [8](#0-7) 

---

### Proof of Concept

A transaction sender submits via `send_transaction` RPC:

- Transaction serialized size: `200 bytes`
- Actual execution cycles: `2,000,000` (e.g., a script with moderate computation)
- Fee paid: `200 shannons`
- Node `min_fee_rate`: `1000 shannons/KW` (default)

**Admission check** (`check_tx_fee`):
```
min_fee = 1000 * 200 / 1000 = 200 shannons
fee (200) >= min_fee (200) → ACCEPTED
```

**Actual fee rate** (stored in `TxEntry`):
```
weight = max(200, 2_000_000 * 0.000170571) = max(200, 341) = 341
fee_rate = 200 * 1000 / 341 = 586 shannons/KW  < min_fee_rate (1000)
```

The transaction is admitted to the pool with an effective fee rate of 586 shannons/KW, 41% below the configured minimum. Repeating this with many transactions fills the pool with sub-minimum-fee-rate entries, exhausting mempool resources and degrading node performance.
