### Title
Upstream RPC Error Messages Reflected Verbatim in HTTP 500 Responses — (`apps/hermes/server/src/api/rest/v2/price_feeds_metadata.rs`)

---

### Summary

The Hermes price service formats raw upstream `anyhow::Error` strings directly into `RestError::RpcConnectionError { message }`, which is then returned verbatim as the HTTP 500 response body to any API consumer. This mirrors the ZecWallet pattern exactly: an external error message is passed through without sanitization into a user-facing output. Any caller of `GET /v2/price_feeds` who triggers a backend failure receives the raw internal error string, leaking infrastructure details. If the upstream metadata backend is network-compromised, an attacker can inject arbitrary text into the HTTP response seen by API consumers.

---

### Finding Description

In `apps/hermes/server/src/api/rest/v2/price_feeds_metadata.rs`, the handler for `/v2/price_feeds` calls `state.get_price_feeds_metadata(...)` and maps any error directly into a `RestError::RpcConnectionError` carrying the full upstream error string:

```rust
.map_err(|e| {
    tracing::warn!("RPC connection error: {}", e);
    RestError::RpcConnectionError {
        message: format!("RPC connection error: {e}"),
    }
})?;
``` [1](#0-0) 

In `apps/hermes/server/src/api/rest.rs`, the `IntoResponse` implementation for `RestError::RpcConnectionError` passes the `message` field directly as the HTTP response body with no filtering:

```rust
RestError::RpcConnectionError { message } => {
    (StatusCode::INTERNAL_SERVER_ERROR, message).into_response()
}
``` [2](#0-1) 

The `e` value is an `anyhow::Error` chain from the upstream metadata RPC backend. Its `Display` output can include internal hostnames, service URLs, connection strings, TLS errors with peer addresses, or any other detail the upstream service embeds in its error chain. All of this is forwarded verbatim to the HTTP caller.

The other `RestError` variants all use static string literals, making `RpcConnectionError` the sole variant that carries dynamic, externally-sourced content into the response body. [3](#0-2) 

---

### Impact Explanation

**Information disclosure**: Any API consumer who queries `GET /v2/price_feeds` when the upstream metadata backend is unavailable or returning errors receives the raw internal error string. This can expose internal service hostnames, IP addresses, port numbers, TLS certificate details, or internal error codes that aid further reconnaissance.

**Message injection (network attacker)**: If an attacker can MITM or impersonate the upstream metadata RPC backend that Hermes connects to (analogous to the ZecWallet/zcashd scenario), they can craft arbitrary error message strings. These strings are returned verbatim in the HTTP 500 body to every API consumer querying `/v2/price_feeds` at that moment. Downstream integrators or monitoring systems that parse and display Hermes error responses could be fed misleading content.

The impact is bounded to the Hermes API layer — no on-chain funds are at risk — but the information disclosure is unconditional and requires no special attacker position.

---

### Likelihood Explanation

**Information disclosure path**: Likelihood is **high**. Any transient upstream failure (network blip, backend restart, misconfiguration) causes the raw error to be emitted. No attacker action is required; normal operational failures trigger it.

**Message injection path**: Likelihood is **low-medium**. Requires a network-level attacker on the path between Hermes and its upstream metadata backend. This is a realistic threat model for a hosted service (BGP hijack, compromised internal network segment, DNS poisoning of the backend endpoint).

---

### Recommendation

Replace the dynamic error format string with a static message in the HTTP response body. Log the full error internally (already done via `tracing::warn!`) but never forward it to the caller:

```rust
.map_err(|e| {
    tracing::warn!("RPC connection error: {}", e);
    RestError::RpcConnectionError {
        message: "An internal error occurred. Please try again later.".to_string(),
    }
})?;
```

Alternatively, remove the `message` field from `RpcConnectionError` entirely and use a static string in `IntoResponse`, consistent with every other `RestError` variant.

---

### Proof of Concept

1. Deploy Hermes with a metadata backend that is unreachable or returns an error containing a crafted string (e.g., `"RPC connection error: connect tcp 10.0.0.5:8080: connection refused"`).
2. Send `GET /v2/price_feeds` to the Hermes REST API.
3. Observe the HTTP 500 response body contains the raw upstream error string verbatim, including the internal IP address `10.0.0.5:8080`.

For the injection scenario: configure a mock backend that returns an HTTP error whose body or connection error message contains attacker-controlled text. Hermes will embed that text in the 500 response returned to all `/v2/price_feeds` callers. [4](#0-3)

impl IntoResponse for RestError {
    fn into_response(self) -> Response {
        match self {
            RestError::BenchmarkPriceNotUnique => {
                (StatusCode::NOT_FOUND, "Benchmark price is not unique").into_response()
            }
            RestError::UpdateDataNotFound => {
                (StatusCode::NOT_FOUND, "Update data not found").into_response()
            }
            RestError::CcipUpdateDataNotFound => {
                // Return "Bad Gateway" error because CCIP expects a 5xx error if it needs to retry
                // or try other endpoints. "Bad Gateway" seems the best choice here as this is not
                // an internal error and could happen on two scenarios:
                //
                // 1. DB Api is not responding well (Bad Gateway is appropriate here)
                // 2. Publish time is a few seconds before current time and a VAA Will be available
                //    in a few seconds. So we want the client to retry.
                (StatusCode::BAD_GATEWAY, "CCIP update data not found").into_response()
            }
            RestError::InvalidCCIPInput => {
                (StatusCode::BAD_REQUEST, "Invalid CCIP input").into_response()
            }
            RestError::PriceIdsNotFound { missing_ids } => {
                let missing_ids = missing_ids
                    .into_iter()
                    .map(|id| id.to_string())
                    .collect::<Vec<_>>()
                    .join(", ");

                (
                    StatusCode::NOT_FOUND,
                    format!("Price ids not found: {missing_ids}"),
                )
                    .into_response()
            }
            RestError::RpcConnectionError { message } => {
                (StatusCode::INTERNAL_SERVER_ERROR, message).into_response()
            }
            RestError::EndpointDeprecated { message } => {
                (StatusCode::BAD_REQUEST, message).into_response()
            }
        }
    }
}
