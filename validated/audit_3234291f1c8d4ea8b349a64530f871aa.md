### Title
Orphan Transaction Pool Has No Periodic Expiry Cleanup; High-Cycle Orphans Permanently Occupy Pool Slots When Verify Queue Is Full — (`tx-pool/src/component/orphan.rs`, `tx-pool/src/process.rs`)

---

### Summary

The `OrphanPool` in CKB's tx-pool has no periodic cleanup mechanism. Its only expiry/size-limiting function, `limit_size()`, is called exclusively from `add_orphan_tx()` — never on a timer or on each new block. Additionally, when a high-cycle orphan's parent is successfully processed but the verify queue is full, `process_orphan_tx()` explicitly keeps the orphan in the pool with no retry path. This creates a direct analog to the "bad debt never handled" vulnerability class: orphan pool slots accumulate unresolvable entries that are never cleared unless a new orphan happens to arrive, and there is no graceful mechanism to drain them.

---

### Finding Description

**Root cause 1 — No periodic orphan pool cleanup.**

`OrphanPool::limit_size()` is the only function that evicts expired entries:

```rust
fn limit_size(&mut self) -> Vec<Byte32> {
    let now = ckb_systemtime::unix_time().as_secs();
    let expires: Vec<_> = self.entries.iter()
        .filter_map(|(id, entry)| {
            if entry.expires_at <= now { Some(id) } else { None }
        })
        .cloned().collect();
    // ... evict expired, then evict random if still > DEFAULT_MAX_ORPHAN_TRANSACTIONS
}
```

It is called **only** from `add_orphan_tx()`:

```rust
pub fn add_orphan_tx(&mut self, tx: TransactionView, peer: PeerIndex, declared_cycle: Cycle) -> Vec<Byte32> {
    // ...
    self.limit_size()   // ← sole call site
}
```

By contrast, the main `TxPool` calls `remove_expired()` on every block via `_update_tx_pool_for_reorg()`. The orphan pool has no equivalent. If no new orphan arrives, expired entries accumulate indefinitely.

**Root cause 2 — High-cycle orphans are kept in the pool with no retry when the verify queue is full.**

In `process_orphan_tx()`, when a high-cycle orphan's parent is processed:

```rust
if orphan.cycle > self.tx_pool_config.max_tx_verify_cycles {
    let orphan_id = orphan.tx.proposal_short_id();
    match self.enqueue_verify_queue(orphan.tx.clone(), false, Some((orphan.cycle, orphan.peer))).await {
        Ok(_) => { self.remove_orphan_tx(&orphan_id).await; }
        Err(reject) => {
            warn!(
                "process_orphan {} failed to enqueue verify queue: {}; keep orphan from {}",
                orphan.tx.hash(), reject, tx.hash(),
            );
            // ← orphan stays; no retry scheduled
        }
    }
}
```

The orphan stays in the pool. `process_orphan_tx()` is only called when a new transaction is successfully processed. Since the parent is already processed, it will not be processed again in normal operation (no reorg). The orphan has no path back to the verify queue until either a reorg occurs, the orphan expires, or a new orphan triggers `limit_size()`. This behavior is explicitly confirmed by the existing test:

```rust
#[tokio::test]
async fn process_orphan_tx_keeps_high_cycle_orphan_when_verify_queue_is_full() {
    // ...
    assert!(service.orphan.read().await.contains_key(&orphan_id));
    assert!(!service.verify_queue.read().await.contains_key(&orphan_id));
}
```

**Root cause 3 — `remove_orphan_txs_by_attach` does not clean up stuck orphans.**

During block attachment, `remove_orphan_txs_by_attach` calls `process_orphan_tx(committed_tx)` (which retries the stuck orphan, but fails again if the queue is still full) and then removes only orphans that ARE the committed transactions — not their stuck children:

```rust
async fn remove_orphan_txs_by_attach<'a>(&self, txs: &LinkedHashSet<TransactionView>) {
    for tx in txs.iter() {
        self.process_orphan_tx(tx).await;  // retries, but fails if queue still full
    }
    let mut orphan = self.orphan.write().await;
    orphan.remove_orphan_txs(txs.iter().map(|tx| tx.proposal_short_id())); // removes committed txs only
}
```

The stuck high-cycle orphan child remains in the pool.

---

### Impact Explanation

The orphan pool is capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries. An attacker who fills the verify queue (by submitting many large-cycle transactions) and then sends orphan transactions whose parents arrive while the queue is full can cause up to 100 orphan pool slots to be occupied by stuck entries. Because `limit_size()` is never called periodically, these entries remain until a new orphan triggers cleanup. During this window (up to `ORPHAN_TX_EXPIRE_TIME = 100 × MAX_BLOCK_INTERVAL`), legitimate orphan transactions submitted by honest peers are randomly evicted. This degrades the node's ability to process valid transaction chains relayed out-of-order, which is a normal and expected relay pattern in CKB's P2P network.

---

### Likelihood Explanation

The verify queue can be filled by any unprivileged peer submitting transactions with declared cycles above `max_tx_verify_cycles`. This is a standard relay path (`submit_remote_tx` / `RelayV3`). No privileged access, key material, or majority hashpower is required. The attacker only needs to sustain verify-queue pressure while sending orphan transactions, which is achievable with modest resources.

---

### Recommendation

1. **Add a periodic cleanup timer for the orphan pool**, analogous to the `clean_expired_orphan_timer` already used for the block-level orphan pool in `ChainService::start_process_block()`. Call `limit_size()` (or a dedicated expiry sweep) on a fixed interval (e.g., every 60 seconds).

2. **Schedule a retry for stuck high-cycle orphans** rather than silently keeping them. When `enqueue_verify_queue` returns `Reject::Full`, record the orphan's ID in a retry set and attempt re-enqueue when the verify queue drains below its limit.

3. **Call `remove_expired` on the orphan pool during `_update_tx_pool_for_reorg`**, mirroring the treatment of the main pool.

---

### Proof of Concept

**Step 1**: Fill the verify queue by relaying transactions with declared cycles just above `max_tx_verify_cycles` until `verify_queue.total_tx_size` approaches the 256 MB limit.

**Step 2**: Relay up to 100 child transactions whose parents are not yet in the pool (orphans with `cycle > max_tx_verify_cycles`). Each is added to the orphan pool via `add_orphan_tx`.

**Step 3**: Relay the parent transactions. Each parent is accepted into the verify queue (or main pool). `process_orphan_tx(parent)` is called. For each high-cycle orphan child, `enqueue_verify_queue` returns `Reject::Full`. The orphan stays in the pool (confirmed by the existing unit test `process_orphan_tx_keeps_high_cycle_orphan_when_verify_queue_is_full`).

**Step 4**: Stop sending new orphans. Because `limit_size()` is never called without a new `add_orphan_tx`, the 100 stuck entries remain. Any legitimate orphan submitted by an honest peer triggers `add_orphan_tx`, which calls `limit_size()` — but since the stuck entries are not yet expired, the new legitimate orphan is randomly evicted instead.

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }
