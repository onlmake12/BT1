### Title
Unbounded Iteration in `get_raw_tx_pool` RPC Causes Memory and CPU Exhaustion — (`rpc/src/module/pool.rs`)

---

### Summary

The `get_raw_tx_pool` RPC method iterates over every entry in the transaction pool and serializes the full result into a single JSON response with no pagination, no limit, and no size cap on the output. Because the tx-pool is bounded only by total byte size of raw transaction data (default 180 MB), an attacker who fills the pool with minimum-size transactions can force the node to allocate and serialize a response orders of magnitude larger than the pool itself, exhausting memory and CPU on any RPC caller's request.

---

### Finding Description

`get_raw_tx_pool` in `rpc/src/module/pool.rs` dispatches to either `get_all_ids()` or `get_all_entry_info()` in `tx-pool/src/pool.rs`, both of which unconditionally collect every entry across all pool states into a single `Vec`/`HashMap` and return it:

```rust
// rpc/src/module/pool.rs  lines 703-718
fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
    let tx_pool = self.shared.tx_pool_controller();
    let raw = if verbose.unwrap_or(false) {
        let info = tx_pool.get_all_entry_info()...;
        RawTxPool::Verbose(info.into())
    } else {
        let ids = tx_pool.get_all_ids()...;
        RawTxPool::Ids(ids.into())
    };
    Ok(raw)
}
``` [1](#0-0) 

`get_all_entry_info()` iterates over pending, gap, proposed, and conflicted entries with no bound:

```rust
// tx-pool/src/pool.rs  lines 464-487
pub(crate) fn get_all_entry_info(&self) -> TxPoolEntryInfo {
    let pending = self.pool_map
        .score_sorted_iter_by_statuses(vec![Status::Pending, Status::Gap])
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();
    let proposed = self.pool_map.sorted_proposed_iter()
        .map(|entry| (entry.transaction().hash(), entry.to_info()))
        .collect();
    let conflicted = self.conflicts_cache.iter()
        .map(|(_id, tx)| tx.hash())
        .collect();
    TxPoolEntryInfo { pending, proposed, conflicted }
}
``` [2](#0-1) 

The pool is bounded by raw transaction bytes (`max_tx_pool_size`, default 180 MB): [3](#0-2) 

However, the RPC documentation shows a minimum transaction size of `0x112` (274 bytes). At that size, the pool can hold roughly **656,000 transactions**. In verbose mode, each entry is serialized with 7 metadata fields (`cycles`, `size`, `fee`, `ancestors_size`, `ancestors_cycles`, `ancestors_count`, `timestamp`). The resulting JSON response can be many times larger than the raw pool data, and the entire collection is built in memory before being sent.

There is no `limit`, `after`, or pagination parameter on `get_raw_tx_pool`: [4](#0-3) 

---

### Impact Explanation

An attacker who fills the pool with small, valid transactions (or who simply waits for a busy mempool) can call `get_raw_tx_pool(verbose=true)` repeatedly. Each call forces the node to:

1. Acquire a read lock on the pool and iterate all entries.
2. Allocate a `HashMap<H256, TxEntryInfo>` with hundreds of thousands of entries.
3. JSON-serialize the entire structure into a single response buffer.

This can exhaust the node's heap memory and saturate a CPU core for the duration of serialization. Repeated concurrent calls amplify the effect. The RPC server shares the process with the chain service, tx-pool, and sync subsystems, so memory pressure or CPU saturation directly degrades block processing and peer synchronization.

---

### Likelihood Explanation

The RPC endpoint is publicly documented and requires no authentication beyond network access to the RPC port. The default configuration binds to `127.0.0.1:8114`, but many operators expose this port to internal tooling or monitoring systems. Any process with RPC access — including a malicious script submitted as a monitoring hook, or a compromised tool — can trigger this. The tx-pool is routinely near capacity on mainnet during fee spikes, making the precondition (large pool) naturally satisfied without attacker effort.

---

### Recommendation

Add `limit` and `after` (cursor) pagination parameters to `get_raw_tx_pool`, mirroring the pattern already used by `get_cells` and `get_transactions` in the indexer RPC. The internal `get_ids` and `get_all_entry_info` methods in `tx-pool/src/pool.rs` should accept an offset and count, and the service layer in `tx-pool/src/service.rs` should thread these parameters through. [5](#0-4) 

---

### Proof of Concept

1. Fill the tx-pool with small transactions until `total_tx_size` approaches `max_tx_pool_size` (180 MB). At ~274 bytes per tx this is ~656,000 transactions.
2. Call `get_raw_tx_pool` with `verbose=true` from any process with RPC access:
   ```json
   {"id":1,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[true]}
   ```
3. The node allocates a `HashMap` with ~656,000 entries, serializes it to JSON (producing a response well over 1 GB), and holds the pool read lock for the entire duration.
4. Repeated concurrent calls exhaust heap memory and degrade or crash the node process.
