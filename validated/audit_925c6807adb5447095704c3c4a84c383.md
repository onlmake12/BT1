### Title
Unauthenticated `from` Field in `ConnectionRequest` Enables Peer Spoofing and Hole-Punching DoS — (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The hole-punching protocol's `ConnectionRequest` handler parses a `from` field from the attacker-controlled message payload but never validates that it matches the actual sending peer's identity. Any connected peer can forge `from` to be any victim peer ID. The target node then stores a rate-limit entry keyed on the spoofed peer ID, blocking the real victim from completing hole-punching through that node for the duration of `HOLE_PUNCHING_INTERVAL`.

---

### Finding Description

**Root cause — no `from == actual_sender` check:**

`TryFrom<&packed::ConnectionRequestReader<'_>> for RequestContent` parses `from` from the raw message bytes and validates only that embedded peer IDs inside `listen_addrs` are consistent with `from`. It never checks that `from` equals the peer ID of the session that delivered the message. [1](#0-0) 

There is no guard of the form:

```rust
if content.from != actual_sender_peer_id {
    return StatusCode::InvalidFromPeerId.with_context("from does not match sender");
}
```

**Vulnerable state mutation — `pending_delivered` keyed on the forged `from`:**

When the receiving node is the intended target (`to == self`), `respond_delivered` is called. It inserts a timestamped entry into `pending_delivered` using the attacker-supplied `from_peer_id` as the key: [2](#0-1) 

**Rate-limit gate that the forged entry trips:**

On any subsequent `ConnectionRequest` carrying the same `from`, the handler checks: [3](#0-2) 

If the stored timestamp is recent, the function returns `StatusCode::Ignore` immediately, silently discarding the legitimate request.

**`ConnectionSync` also trusts the forged `from`:**

The `ConnectionSync` handler looks up `pending_delivered` using `content.from` from the message, so the same spoofed entry poisons the NAT-traversal step as well: [4](#0-3) 

---

### Impact Explanation

An unprivileged attacker who has a single open connection to any node B can prevent an arbitrary victim peer A from completing hole-punching through B:

1. Attacker sends `ConnectionRequest { from: A, to: B, listen_addrs: [attacker_addr] }`.
2. B stores `pending_delivered[A] = (attacker_addrs, now)` and replies to the attacker's session.
3. When A sends its legitimate `ConnectionRequest` within `HOLE_PUNCHING_INTERVAL`, B returns `Ignore`.
4. A's NAT traversal through B silently fails.

By repeating the spoofed message just before `HOLE_PUNCHING_INTERVAL` expires, the attacker can sustain the DoS indefinitely against any victim peer, through any relay node the attacker is connected to. The victim receives no error and has no way to distinguish this from a normal relay failure.

---

### Likelihood Explanation

The attacker needs only one ordinary P2P connection to a CKB node that participates in hole-punching. No privileged keys, no majority hash-power, and no Sybil capability are required. The `HolePunching` protocol is a standard supported protocol exposed to all peers. [5](#0-4) 

---

### Recommendation

When the receiving node is the direct target (`to == self_peer_id`) and the message arrives on the **first hop** (i.e., the route list is empty, meaning the sender is the originator), validate:

```rust
let actual_sender_peer_id = /* extract from session context */;
if content.from != actual_sender_peer_id {
    return StatusCode::InvalidFromPeerId
        .with_context("from field does not match actual sender");
}
```

For forwarded messages (non-empty route), the `from` field legitimately differs from the immediate sender, so the check should be scoped to the first hop only. Additionally, consider authenticating the `from` field cryptographically (e.g., a signature over the request) to prevent spoofing across relay hops.

---

### Proof of Concept

```
Attacker (peer C) ──connected──► Node B (target, to == B)

1. C sends:
   ConnectionRequest {
       from: A,          // victim's peer ID — attacker-controlled, not validated
       to:   B,
       listen_addrs: [C's address],
       route: [],
       max_hops: 3,
   }

2. B: TryFrom parses content.from = A  (no check that A == C)
   B: respond_delivered(from_peer_id = A, ...)
   B: pending_delivered.insert(A, (C's addrs, now))   ← poisoned entry
   B: sends ConnectionRequestDelivered back to C's session

3. Victim A sends legitimate ConnectionRequest { from: A, to: B, ... }
   B: pending_delivered.get(&A) → Some((_, t))
   B: now - t < HOLE_PUNCHING_INTERVAL  → returns Ignore
   A's hole-punching through B silently fails.

4. C repeats step 1 every HOLE_PUNCHING_INTERVAL to sustain the DoS.
``` [6](#0-5)
