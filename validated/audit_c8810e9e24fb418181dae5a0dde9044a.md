### Title
Missing Domain Separation in Lazer Signature Verification Enables Cross-Chain Replay - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` recovers a signer from `keccak256(payload)` with no chain ID, contract address, or any domain-binding context included in the signed digest. A valid signed Lazer update accepted on one chain can be replayed verbatim on any other chain where the same trusted signer is registered, as long as the payload timestamp is still fresh.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate` function computes the hash to verify as:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The signed digest is purely `keccak256(payload)`. The payload contains a format magic number, a timestamp, a channel byte, and feed data — none of which are chain-specific or contract-specific. [2](#0-1) 

The `EVM_FORMAT_MAGIC` constant (`706910618`) is hardcoded and identical across all EVM deployments. Trusted signers are registered per-contract by the owner, and the same signer key is expected to be trusted on multiple chains simultaneously. [3](#0-2) 

There is no EIP-712 domain separator, no `block.chainid` binding, and no contract address binding in the signed message. This is the direct analog of the reported missing `DOMAIN_SEPARATOR()` vulnerability class.

### Impact Explanation
An unprivileged relayer who observes a valid `verifyUpdate` call on chain A (e.g., Ethereum mainnet) can extract the raw `update` bytes and immediately submit them to `verifyUpdate` on chain B (e.g., Arbitrum, Base, etc.) where the same trusted signer is registered. The signature will pass verification on chain B because the digest is identical. Any consumer contract on chain B that calls `verifyUpdate` and trusts the returned `(payload, signer)` will accept the cross-chain replayed price data as legitimate. Within the payload's timestamp window, this allows injection of price data signed for a different chain into any PythLazer-consuming contract on chain B.

### Likelihood Explanation
PythLazer is deployed on multiple EVM chains with the same trusted signer set. The `verifyUpdate` function is a public payable entry point callable by any address. A relayer or MEV bot monitoring on-chain transactions can trivially extract and replay valid update bytes across chains within the same timestamp window. No privileged access is required.

### Recommendation
Bind the signed digest to the chain and contract by incorporating `block.chainid` and `address(this)` into the hash, following EIP-712 domain separation:

```solidity
bytes32 domainSeparator = keccak256(abi.encode(
    keccak256("EIP712Domain(string name,uint256 chainId,address verifyingContract)"),
    keccak256("PythLazer"),
    block.chainid,
    address(this)
));
bytes32 hash = keccak256(abi.encodePacked("\x19\x01", domainSeparator, keccak256(payload)));
```

This ensures a signature produced for one chain and contract cannot be accepted on another.

### Proof of Concept
1. Deploy `PythLazer` on chain A (chainId=1) and chain B (chainId=42161) with the same trusted signer.
2. On chain A, call `verifyUpdate{value: fee}(update)` with a valid signed update. Transaction succeeds; `signer` is the trusted signer.
3. Extract the identical `update` bytes from the chain A transaction calldata.
4. On chain B, call `verifyUpdate{value: fee}(update)` with the same bytes within the timestamp window.
5. The call succeeds — `keccak256(payload)` is identical, `ECDSA.tryRecover` returns the same signer, `isValidSigner(signer)` returns `true` — and the cross-chain replayed price data is accepted as valid. [4](#0-3)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
