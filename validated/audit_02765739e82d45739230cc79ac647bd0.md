### Title
Entropy `requestHelper` Silently Absorbs All Excess ETH Into Protocol Fee Pool Without Refunding Caller - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The `Entropy.requestHelper()` internal function, called by every public request entry point (`request()`, `requestWithCallback()`, `requestV2()`), permanently captures the entire `msg.value` minus the provider fee into `_state.accruedPythFeesInWei`. Any ETH sent above the required fee is silently absorbed by the protocol and never returned to the caller.

### Finding Description
In `requestHelper`, fee accounting is performed as follows:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The expression `SafeCast.toUint128(msg.value) - providerFee` assigns the **entire** remainder of `msg.value` to Pyth's fee pool — not just the protocol's share of `requiredFee`. If `msg.value > requiredFee`, the surplus is silently captured. There is no refund path. The public interface explicitly acknowledges this:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

This affects all four public entry points that route through `requestHelper`:
- `request()` [4](#0-3) 
- `requestWithCallback()` [5](#0-4) 
- `requestV2(address, bytes32, uint32)` [6](#0-5) 
- `requestV2(address, uint32)` and `requestV2(uint32)` overloads [7](#0-6) 

The captured excess is only withdrawable by the Pyth admin via `EntropyGovernance.withdrawFee()`: [8](#0-7) 

The existing test suite even confirms this behavior — an overpayment of 10,000 wei is silently added to `accruedPythFees`: [9](#0-8) 

### Impact Explanation
Any Entropy user who sends `msg.value` exceeding the exact required fee permanently loses the surplus ETH to the protocol fee pool. The user has no mechanism to recover it. Concretely:

- A user who calls `getFeeV2(provider, gasLimit)` off-chain and then submits a transaction with a small buffer (a common defensive pattern to guard against fee increases) loses that buffer permanently.
- If the provider raises their fee via `setProviderFee()` between the user's fee query and their transaction landing, the user's transaction succeeds but the entire `msg.value` is consumed — the portion above the new required fee goes to `accruedPythFeesInWei`, not back to the user.
- Smart contract integrators that hardcode a fee ceiling or add a safety margin will silently overpay on every call.

### Likelihood Explanation
The scenario is realistic and reachable by any unprivileged Entropy user:

1. Provider fees can change at any time via `setProviderFee()` or `setProviderFeeAsFeeManager()`. [10](#0-9) 
2. The Pyth protocol fee can change via `EntropyGovernance.setPythFee()`. [11](#0-10) 
3. Integrators commonly add a small ETH buffer to avoid `InsufficientFee` reverts — this buffer is silently consumed.
4. The `requestV2()` no-argument overload uses `getDefaultProvider()` and `random()` internally, making exact fee pre-computation harder for callers. [12](#0-11) 

### Recommendation
Refund any ETH above `requiredFee` to `msg.sender` at the end of `requestHelper`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // only the protocol's share

// Refund excess
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "refund failed");
}
```

This mirrors the fix applied to `PythLazer.verifyUpdate()`, which already correctly refunds excess: [13](#0-12) 

### Proof of Concept
1. Deploy Entropy with provider fee = 100 wei, Pyth fee = 50 wei → `requiredFee = 150 wei`.
2. User calls `requestV2(provider, gasLimit)` with `msg.value = 200 wei` (50 wei buffer).
3. `requestHelper` executes: `providerInfo.accruedFeesInWei += 100`, `accruedPythFeesInWei += 100` (200 − 100).
4. User's 50 wei buffer is silently absorbed into `accruedPythFeesInWei`.
5. User receives no refund. Only the Pyth admin can recover those 50 wei via `withdrawFee()`.
6. Confirmed by the existing test at line 699: `assertEq(random.getAccruedPythFees(), pythFeeInWei * 5 + 10000)` — the 10,000 wei overpayment is fully captured. [9](#0-8)

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }

    function withdrawFee(address targetAddress, uint128 amount) external {
        require(targetAddress != address(0), "targetAddress is zero address");
        _authoriseAdminAction();

        if (amount > _state.accruedPythFeesInWei)
            revert EntropyErrors.InsufficientFee();

        _state.accruedPythFeesInWei -= amount;

        (bool success, ) = targetAddress.call{value: amount}("");
        require(success, "Failed to withdraw fees");

        emit FeeWithdrawn(targetAddress, amount);
    }
