Audit Report

## Title
Truncated 4-Byte Genesis Hash in `identify_name()` Allows Rogue Nodes to Pass Network Identity Check — (File: `spec/src/consensus.rs`)

## Summary
`identify_name()` in `spec/src/consensus.rs` at line 967 uses only the first 8 hex characters (4 bytes) of the 32-byte genesis hash to construct the P2P network identifier. `Identify::verify()` in `network/src/protocols/identify/mod.rs` performs no other chain-membership check beyond string equality of this truncated name, allowing an attacker to craft a rogue node that passes the identify handshake and occupies connection slots on mainnet nodes.

## Finding Description
`identify_name()` at `spec/src/consensus.rs:967` produces:
```rust
format!("/{}/{}", self.id, &genesis_hash[..8])
```
28 of 32 bytes of the genesis hash are discarded. The resulting string (e.g., `/ckb/92b197aa`) is the sole chain-membership token embedded in the identify message.

`Identify::verify()` at `network/src/protocols/identify/mod.rs:541–561` performs exactly one check:
```rust
if self.name != name {
    return None;
}
```
No genesis hash, no block header, no cryptographic proof is checked. The `name` field in the identify message is a plain string sent by the remote peer — there is no mechanism binding it to the peer's actual chain state.

An attacker has two paths, both trivial:
1. **No brute-force needed**: Modify the CKB node source to hardcode or freely set the `name` field to `/ckb/92b197aa`. The receiver has no way to verify the name matches the sender's actual genesis block.
2. **Brute-force path (as described)**: Vary any genesis field until `blake2b_256(genesis_block)` shares the same leading 4 bytes — a 2^32 search, achievable in seconds on a GPU.

Either way, `received_identify()` at lines 384–454 accepts the peer unconditionally when `verify()` returns `Some`, registers it in `PeerRegistry`, and opens sync/relay/block-filter protocols via `open_protocols`. The rogue node occupies an inbound or outbound slot bounded by `max_inbound`/`max_outbound`, preventing legitimate peers from connecting.

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker running many rogue nodes (all sending the mainnet identify name) can saturate connection slots on targeted mainnet nodes. Sync and relay protocols are opened against rogue peers, wasting CPU and memory before consensus-invalid messages are rejected downstream. Connection-slot exhaustion directly causes network congestion and can partition individual nodes from the honest network.

## Likelihood Explanation
The attacker requires no privileged access, no key material, and no majority hashpower. The simplest exploit requires only modifying the CKB node source to send the correct name string — no computation at all. The brute-force path requires only ~2^32 blake2b operations (seconds on commodity GPU hardware). The crafted node or spec is reusable indefinitely. The attack is fully automatable and repeatable.

## Recommendation
Use the full 64-character genesis hash in `identify_name()`:
```rust
pub fn identify_name(&self) -> String {
    let genesis_hash = format!("{:x}", Into::<H256>::into(&self.genesis_hash));
    format!("/{}/{}", self.id, &genesis_hash)  // full 32-byte hash
}
```
This raises the preimage search to 2^256 and makes the network identifier cryptographically unique per chain. Additionally, consider including the full genesis hash as a separate verified field in the identify message so the receiver can confirm it matches the sender's actual chain.

## Proof of Concept
1. Read the mainnet identify name from a running mainnet node or derive it: format the genesis hash from `docs/hashes.toml`, take the first 8 hex chars, prepend `/ckb/`.
2. Patch a CKB node to return this hardcoded string from `identify_name()` regardless of the actual genesis block loaded.
3. Start the patched node with any genesis spec (e.g., the default testnet spec) and connect it to a mainnet node.
4. Observe: `Identify::verify()` at line 545 passes (`self.name == name`), `received_identify()` returns `MisbehaveResult::Continue`, and sync/relay protocols are opened against the rogue peer.
5. Repeat with many rogue nodes to exhaust `max_inbound`/`max_outbound` slots on victim nodes, preventing legitimate peers from connecting.
