### Title
Expired Alert Replay via P2P — Missing `notice_until` Validation Allows Re-triggering of `network_alert_notify_script` — (File: `util/network-alert/src/alert_relayer.rs`, `util/network-alert/src/notifier.rs`)

---

### Summary

The P2P alert relay handler (`alert_relayer.rs`) accepts any alert that passes signature verification and is not already tracked in `has_received`. The `clear_expired_alerts` function removes expired alerts from `received_alerts` but does **not** add their IDs to `cancel_filter`. This creates a window where any unprivileged P2P peer who stored an old alert can replay it after expiry, causing the node to re-accept, re-broadcast, and re-notify subscribers — including executing the operator-configured `network_alert_notify_script`.

---

### Finding Description

**Root cause — two cooperating gaps:**

**Gap 1 — No `notice_until` check in the P2P relay path.**

The RPC entry point `send_alert` explicitly rejects expired alerts: [1](#0-0) 

But the P2P relay handler in `alert_relayer.rs` performs no such check. It only validates UTF-8 encoding, checks `has_received`, and verifies signatures: [2](#0-1) 

**Gap 2 — `clear_expired_alerts` does not populate `cancel_filter`.**

`has_received` returns `true` only if the alert ID is in `received_alerts` OR in `cancel_filter`: [3](#0-2) 

When `clear_expired_alerts` runs (triggered on every new peer connection), it removes expired alerts from `received_alerts` and `noticed_alerts`, but it never inserts the evicted IDs into `cancel_filter`: [4](#0-3) 

After this cleanup, `has_received(expired_id)` returns `false`. The `cancel_filter` (LRU, size 128) is only populated by the explicit `cancel()` call triggered by the `cancel` field of a new alert: [5](#0-4) 

**Exploit flow:**

1. Key holders broadcast Alert A (`id=1`, `notice_until=T`). All nodes receive and store it.
2. Time advances past `T`. The next peer connection triggers `clear_expired_alerts`, which removes Alert A from `received_alerts` and `noticed_alerts`. `cancel_filter` is untouched — `has_received(1)` now returns `false`.
3. An unprivileged P2P peer who stored the raw bytes of Alert A relays them to a node.
4. The relay handler checks `has_received(1)` → `false`; verifies signatures → valid (the original signatures are still cryptographically correct); broadcasts to all connected peers; calls `notifier.add(&alert)`.
5. `add` re-inserts Alert A into `received_alerts` and `noticed_alerts`, and calls `notify_controller.notify_network_alert(alert.clone())`: [6](#0-5) 

6. The notify service dispatches the alert to all subscribers and, if configured, spawns `network_alert_notify_script` with the alert message as an argument: [7](#0-6) 

7. The attacker can repeat step 3 indefinitely: each time `clear_expired_alerts` evicts the alert, the attacker re-relays it.

---

### Impact Explanation

**Impact: Medium**

- The expired alert re-appears in `noticed_alerts` and is returned by `get_blockchain_info`, misleading node operators into believing a critical condition is still active.
- More concretely, `network_alert_notify_script` — an operator-configured external process — is re-executed with the stale alert message on every replay cycle. Depending on the script (e.g., automated node restart, pager/SMS notification, emergency upgrade trigger), this can cause repeated disruptive actions with no operator-initiated trigger.
- The replayed alert is also re-broadcast to all connected peers, propagating the false state across the network.

The alert system is explicitly described as a mechanism for "critical problems" and "dangerous status" notifications. Persistent replay of a stale critical alert undermines operator trust and can cause automated response systems to act incorrectly.

---

### Likelihood Explanation

**Likelihood: Low**

- The attacker must be a connected P2P peer (no privileged role required — standard peer connection suffices).
- The attacker must have stored the raw bytes of a previously broadcast alert (trivially achievable by any node that was online when the alert was first sent).
- The attack window opens automatically after `notice_until` passes and `clear_expired_alerts` runs — no special timing or coordination is needed beyond waiting.
- No key material is required; the original valid signatures remain intact.

The main constraint is that the alert system is used rarely, so the pool of replayable alerts is small. However, every alert ever broadcast is permanently replayable by any peer who stored it.

---

### Recommendation

1. **In `alert_relayer.rs` `received` handler**: add a `notice_until` check before accepting the alert from a peer, mirroring the check in `send_alert`:
   ```rust
   let notice_until: u64 = alert.as_reader().raw().notice_until().into();
   if notice_until < ckb_systemtime::unix_time_as_millis() {
       // optionally ban peer or silently drop
       return;
   }
   ```
2. **In `Notifier::clear_expired_alerts`**: insert each evicted alert's ID into `cancel_filter` so that `has_received` permanently returns `true` for expired IDs, preventing re-acceptance even if the P2P check is bypassed.

---

### Proof of Concept

```
1. Node A receives Alert { id=1, notice_until=<past timestamp>, signatures=[valid 2-of-4] }
   → stored in received_alerts[1]

2. Time passes; next peer connection fires clear_expired_alerts()
   → received_alerts.remove(1)   // Alert gone
   → cancel_filter unchanged      // has_received(1) == false

3. Attacker peer sends raw bytes of Alert { id=1 } to Node A via P2P

4. alert_relayer::received():
   - has_received(1) → false      ✓ passes gate
   - verify_signatures() → Ok     ✓ original sigs still valid
   - quick_filter_broadcast(...)  → re-propagates to all peers
   - notifier.add(&alert)
     → received_alerts.insert(1, alert)
     → notify_controller.notify_network_alert(alert)
       → network_alert_notify_script executed with alert message

5. Repeat from step 3 indefinitely.
```
