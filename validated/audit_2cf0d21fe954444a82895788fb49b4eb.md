### Title
`descendants_*` Fields and `evict_key` of Ancestor Entries Not Updated After `remove_entry_and_descendants` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all parent-child links are torn down for every entry being removed **before** `remove_entry` is called on each of them. Because `update_ancestors_index_key` relies on those links to locate the surviving ancestors, it finds nothing and silently skips the `sub_descendant_weight` update. Ancestor entries that remain in the pool are left with permanently inflated `descendants_count`, `descendants_fee`, `descendants_size`, and `descendants_cycles`, and therefore a stale `evict_key`. This corrupts the eviction ordering used by `limit_size` every time a transaction with in-pool parents is removed via conflict resolution, RBF, size-limit eviction, or the RPC `remove_transaction` path.

---

### Finding Description

`remove_entry_and_descendants` first collects the target entry and all its descendants, then calls `remove_entry_links` for **all** of them in a loop, and only afterwards calls `remove_entry` for each:

```rust
// tx-pool/src/component/pool_map.rs  lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← tears down ALL links first
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // ← then removes each entry
        .collect()
}
```

Inside `remove_entry`, the first two operations are:

```rust
// lines 242-243
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
```

`update_ancestors_index_key` works by calling `self.links.calc_ancestors(&child.proposal_short_id())` to find every surviving ancestor, then calling `sub_descendant_weight(child)` on each and recomputing its `evict_key`:

```rust
// lines 432-445
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                ...
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because `remove_entry_links` already removed the entry from `self.links` (line 429: `self.links.remove(id)`), `calc_ancestors` returns an empty set. No ancestor is ever visited. The surviving ancestors' `descendants_count`, `descendants_fee`, `descendants_size`, `descendants_cycles` are never decremented, and their `evict_key` is never recomputed.

The comment in `remove_entry_and_descendants` acknowledges the intent to suppress `update_descendants_index_key` (since the descendants are being removed anyway), but the same early link-removal also silently suppresses `update_ancestors_index_key` for the **surviving** ancestors — which is the bug.

`remove_entry_and_descendants` is called from every major removal path:

| Caller | Trigger |
|---|---|
| `resolve_conflict` | conflicting input consumed by new tx |
| `resolve_conflict_header_dep` | invalid header dep |
| `limit_size` | pool over `max_tx_pool_size` |
| `remove_by_detached_proposal` | proposal window expired |
| `check_and_record_ancestors` | ancestor-count eviction |
| `remove_tx` (RPC) | `remove_transaction` RPC call |

The `EvictKey` is the sole ordering key used by `next_evict_entry` → `limit_size`. An inflated `descendants_feerate` or `descendants_count` makes an ancestor appear more valuable than it is, so it is skipped during eviction in favour of other entries that should have been kept.

---

### Impact Explanation

Every surviving ancestor of a removed transaction retains stale, over-counted `descendants_*` fields. Its `evict_key` is computed as `descendants_feerate.max(own_feerate)` plus `descendants_count` as a tiebreaker:

```rust
// entry.rs lines 234-247
EvictKey {
    fee_rate: descendants_feerate.max(feerate),
    timestamp: entry.timestamp,
    descendants_count: entry.descendants_count,
}
```

An inflated `descendants_feerate` or `descendants_count` causes the ancestor to sort later in the eviction order (i.e., it is treated as harder to evict). When the pool is full, `limit_size` will evict other, legitimately higher-value transactions instead of the stale ancestor. The stale state accumulates with every subsequent removal cycle, progressively degrading eviction accuracy. A low-fee transaction can be kept in the pool indefinitely by repeatedly attaching and removing high-fee children.

---

### Likelihood Explanation

The bug fires on every call to `remove_entry_and_descendants` where the removed entry has at least one surviving in-pool ancestor. This is a routine event: conflict resolution on every new transaction submission, RBF replacement, pool-full eviction, and proposal-window expiry all use this path. No special privilege is required — any unprivileged peer or RPC caller that submits transactions can trigger it.

**Concrete attacker-controlled path:**
1. Attacker submits parent tx **P** with a low fee rate.
2. Attacker submits child tx **C** spending an output of **P**, with a high fee rate. `add_entry` calls `record_entry_descendants` → `update_ancestors_index_key(C, Add)` → P's `descendants_fee`, `descendants_cycles`, `descendants_size`, `descendants_count` are incremented and P's `evict_key` is updated to reflect the high-fee descendant.
3. Attacker submits a conflicting tx **C′** that spends the same input as **C**. `resolve_conflict` calls `remove_entry_and_descendants(C)`. Links are torn down before `remove_entry(C)` runs, so `update_ancestors_index_key(C, Remove)` finds no ancestors. P's `descendants_*` fields remain at the inflated values from step 2.
4. P now has a permanently inflated `evict_key`. When the pool fills, `limit_size` skips P and evicts other transactions instead.
5. Steps 1–4 can be repeated to keep P in the pool indefinitely regardless of its actual fee rate.

---

### Recommendation

Before tearing down links, collect and update the surviving ancestors. The simplest correct fix is to call `update_ancestors_index_key` for each entry **before** `remove_entry_links` is called for it, or to save the ancestor sets before the link-removal loop and apply `sub_descendant_weight` afterwards.

Concretely, in `remove_entry_and_descendants`, replace the current two-phase approach with one that updates ancestor accounting before destroying the link information:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update surviving ancestors' descendant accounting BEFORE links are removed.
    for rid in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(rid).map(|e| e.inner.clone()) {
            self.update_ancestors_index_key(&entry, EntryOp::Remove);
        }
    }

    // Now remove links (suppresses update_descendants_index_key inside remove_entry,
    // which is intentional since all descendants are being removed anyway).
    for rid in &removed_ids {
        self.remove_entry_links(rid);
    }

    removed_ids
        .iter()
        .filter_map(|rid| self.remove_entry(rid))
        .collect()
}
```

### Proof of Concept

Given the existing test infrastructure in `tx-pool/src/component/tests/pending.rs`, the following scenario demonstrates the stale state:

1. Build a parent tx **P** and child tx **C** where C spends an output of P.
2. Add both to the pool. After `add_entry(C)`, P's `descendants_count` == 2, `descendants_fee` == P.fee + C.fee.
3. Call `pool.remove_entry_and_descendants(&C.proposal_short_id())`.
4. Assert P's `descendants_count` == 1 and `descendants_fee` == P.fee. **This assertion fails** — P still reports `descendants_count` == 2 and the combined fee, because `update_ancestors_index_key` was a no-op due to the pre-removed links.
5. Call `pool.next_evict_entry(Status::Pending)`. With the inflated `evict_key`, P is ordered as if it still has a high-fee descendant, causing it to be skipped in favour of other entries that should have been retained. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)
