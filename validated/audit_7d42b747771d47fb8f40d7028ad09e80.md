Audit Report

## Title
Global `unknown_tx_hashes` Soft Limit Bypassed by Post-Insertion Check in `add_ask_for_txs` - (File: sync/src/types/mod.rs)

## Summary
`SyncState::add_ask_for_txs` unconditionally inserts up to 32,767 entries per peer into the shared `unknown_tx_hashes` queue before checking the 50,000-entry global soft limit. Two peers each sending 32,767 disjoint novel tx hashes grow the map to 65,534 entries with no effective ceiling. Once oversized, every subsequent call triggers an O(map_size) full scan under the mutex, blocking `pop_ask_for_txs` (the 100ms relay timer) and `mark_as_known_txs`, stalling transaction propagation on the targeted node.

## Finding Description
**Code path:** `TransactionHashesProcess::execute` (`sync/src/relayer/transaction_hashes_process.rs:25–49`) → `SyncState::add_ask_for_txs` (`sync/src/types/mod.rs:1483–1532`).

**Root cause:** The insertion loop at lines 1486–1504 runs unconditionally for up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) hashes. The global-limit check at lines 1507–1529 is explicitly post-insertion (the comment at line 1506 reads: *"Check `unknown_tx_hashes`'s length after inserting the arrival `tx_hashes`"*). Constants in `util/constant/src/sync.rs:68–72` confirm the structural mismatch: `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = `MAX_RELAY_TXS_NUM_PER_BATCH` = 32,767, while `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000. Two peers × 32,767 = 65,534 > 50,000.

When the post-insertion threshold fires, the code performs an O(map_size) scan (lines 1517–1523) over all entries while holding `self.unknown_tx_hashes.lock()` to count entries belonging to the current peer. If that peer's count is < 32,767, `Status::ignored()` is returned but all inserted entries remain in the map. `pop_ask_for_txs` (lines 1453–1481) and `mark_as_known_txs` (lines 1443–1451) both acquire the same mutex and are directly blocked during this scan.

**Why existing guards fail:**
- The rate limiter in `Relayer::try_process` limits per-peer message frequency but does not bound cumulative map size across peers.
- `TransactionHashesProcess::execute` filters only hashes already in `tx_filter` (line 45); novel fake hashes pass through freely.
- The second condition `peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` grows with peer count, providing no tighter bound than 50,000 for small N.

## Impact Explanation
**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

With 2 peers the map exceeds its intended cap (65,534 vs 50,000). With N peers it grows to N × 32,767 with no effective ceiling. Once oversized, every `RelayTransactionHashes` message from any peer triggers the O(map_size) mutex-held scan, blocking the `ASK_FOR_TXS_TOKEN` relay timer (100ms interval) that dispatches `GetRelayTransactions`. This stalls transaction propagation on the targeted node and, if applied to multiple nodes, contributes to network-wide propagation delays.

## Likelihood Explanation
**Medium.** Any peer completing the P2P handshake can send `RelayTransactionHashes`. Hashes need not correspond to real transactions — only hashes already in `tx_filter` are rejected. An attacker needs only 2 connections to exceed the 50,000 global limit. The per-peer rate limiter slows map refill after `pop_ask_for_txs` drains it, but with 2+ peers operating concurrently the map stays oversized. No special privilege, key, or hashpower is required.

## Recommendation
Check the global limit **before** the insertion loop. If `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE`, perform the per-peer count check first and reject or skip insertion accordingly. Replace the O(map_size) per-peer scan with a separate `HashMap<PeerIndex, usize>` tracking per-peer entry counts, updated incrementally on insert/remove, to make every overflow check O(1).

## Proof of Concept
1. Connect two peers (Peer A, Peer B) to a CKB node after handshake.
2. Peer A sends one `RelayTransactionHashes` message with 32,767 unique, novel tx hashes (not in `tx_filter`). All are inserted; map size = 32,767 < 50,000 → post-insertion check does not fire.
3. Peer B sends one `RelayTransactionHashes` message with 32,767 **different** unique tx hashes. All are inserted; map size = 65,534. Post-insertion check fires (65,534 ≥ 50,000). Peer B's per-peer count = 32,767 ≥ `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` → `TooManyUnknownTransactions` returned, but all 32,767 entries are already in the map.
4. Observe `unknown_tx_hashes.len()` = 65,534, 31% above the intended 50,000 cap.
5. Send any subsequent `RelayTransactionHashes` from any peer: the post-insertion check fires immediately, triggering the O(65,534) full-map scan under the mutex. Instrument mutex hold time and observe `pop_ask_for_txs` (100ms timer) being blocked.
6. Add a third peer (Peer C) with 32,767 more hashes: map grows to 98,301 before the check fires; Peer C has < 32,767 entries → `Status::ignored()`, entries remain. Repeat to grow the map further with no bound.
