### Title
Entropy `revealWithCallback` Permits Selective Abort After Random Number Is Exposed in `CALLBACK_FAILED` State — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy::revealWithCallback` accepts requests in both `CALLBACK_NOT_STARTED` and `CALLBACK_FAILED` states. When a callback fails, the computed `randomNumber` is emitted in the public `CallbackFailed` event. A malicious requester can intentionally cause their callback to fail on the first invocation, observe the random number from the event log, and then selectively retry (if the outcome is favorable) or permanently abandon the request (if unfavorable). This breaks the fundamental security guarantee of the commit-reveal scheme.

---

### Finding Description

`revealWithCallback` performs the following state check:

```solidity
if (
    !(req.callbackStatus ==
        EntropyStatusConstants.CALLBACK_NOT_STARTED ||
        req.callbackStatus == EntropyStatusConstants.CALLBACK_FAILED)
) {
    revert EntropyErrors.InvalidRevealCall();
}
``` [1](#0-0) 

This allows the function to execute in both states. When the callback fails (transitions to `CALLBACK_FAILED`), the contract emits the fully computed `randomNumber` in the `CallbackFailed` event before the request is cleared:

```solidity
emit CallbackFailed(
    provider,
    req.requester,
    sequenceNumber,
    userContribution,
    providerContribution,
    randomNumber,   // <-- random number publicly revealed here
    ret
);
req.callbackStatus = EntropyStatusConstants.CALLBACK_FAILED;
``` [2](#0-1) 

The `revealHelper` computes the random number deterministically from `userContribution`, `providerContribution`, and `blockHash`:

```solidity
randomNumber = combineRandomValues(
    userContribution,
    providerContribution,
    blockHash
);
``` [3](#0-2) 

Once the first `revealWithCallback` call is mined, the `randomNumber` is permanently on-chain (in calldata and in the `CallbackFailed` event). The request remains live in `CALLBACK_FAILED` state, and `revealWithCallback` can be called again.

**Attack path:**

1. Attacker deploys a callback contract with an owner-controlled revert flag (or an upgradeable proxy).
2. Attacker calls `requestV2` with a gas limit, committing to `userContribution`.
3. Provider (or anyone) calls `revealWithCallback` — the callback intentionally reverts, `CallbackFailed` is emitted with `randomNumber`.
4. Attacker reads `randomNumber` from the event log.
5. **If favorable:** attacker flips the flag (or upgrades the proxy), calls `revealWithCallback` again — callback succeeds, outcome is accepted.
6. **If unfavorable:** attacker never retries — the request stays in `CALLBACK_FAILED` state indefinitely, effectively discarding the unfavorable outcome.

The `CALLBACK_FAILED` recovery path goes to the `else` branch, which calls the callback directly without gas-limit enforcement:

```solidity
} else {
    address callAddress = req.requester;
    ...
    clearRequest(provider, sequenceNumber);
    if (len != 0) {
        IEntropyConsumer(callAddress)._entropyCallback(
            sequenceNumber, provider, randomNumber
        );
    }
``` [4](#0-3)

### Impact Explanation

The Entropy protocol's security guarantee is that neither the provider nor the requester can predict or manipulate the random number. The `CALLBACK_FAILED` recovery mechanism breaks this guarantee for the requester: after the first `revealWithCallback` call, the random number is fully determined and publicly visible. The requester can then decide whether to accept or reject the outcome. Any Entropy consumer used for high-stakes outcomes — lotteries, NFT trait assignment, on-chain gaming — is vulnerable to this selective abort attack. The attacker pays only the gas cost of the failed first call; there is no additional fee to retry.

---

### Likelihood Explanation

The attack requires the requester to deploy a callback contract with controllable revert behavior (e.g., an owner-controlled flag, an upgradeable proxy, or a dependency on external mutable state). This is a realistic pattern for any sophisticated on-chain application. The `requestV2` entry point is permissionless and unprivileged. The `CallbackFailed` event is emitted on every failed callback, making the random number trivially observable. Likelihood is **medium**: not every consumer is vulnerable, but any consumer using upgradeable or flag-gated callbacks is.

---

### Recommendation

The `CALLBACK_FAILED` recovery path should not re-expose the random number to a requester who can selectively accept or reject it. Consider one or more of the following mitigations:

1. **Remove the `CALLBACK_FAILED` state entirely** and instead require the requester to submit a new request if their callback fails. This eliminates the information leak.

2. **Do not emit `randomNumber` in `CallbackFailed`** — emit only the revert reason. The random number is already in the calldata of the first `revealWithCallback` transaction, but removing it from the event raises the bar for casual exploitation.

3. **Restrict who can call `revealWithCallback` in `CALLBACK_FAILED` state** — e.g., only the original provider, not the requester. This prevents the requester from timing the retry.

4. **Apply a time-lock on retries** — require a minimum delay between the first failure and any retry, making the attack economically unattractive.

### Proof of Concept

```solidity
// Malicious callback contract
contract SelectiveAbortConsumer is IEntropyConsumer {
    address public entropy;
    bool public acceptOutcome;
    bytes32 public lastRandomNumber;

    constructor(address _entropy) { entropy = _entropy; }

    function requestEntropy(bytes32 userContribution) external payable returns (uint64) {
        address provider = IEntropy(entropy).getDefaultProvider();
        return IEntropy(entropy).requestV2{value: msg.value}(provider, userContribution, 100000);
    }

    function _entropyCallback(uint64, address, bytes32 randomNumber) external override {
        require(acceptOutcome, "Not accepting yet");
        lastRandomNumber = randomNumber;
        // use randomNumber favorably here
    }

    function setAcceptOutcome(bool _accept) external { acceptOutcome = _accept; }
}

// Attack sequence:
// 1. Deploy SelectiveAbortConsumer with acceptOutcome = false
// 2. Call requestEntropy{value: fee}(userContribution)
// 3. Provider calls revealWithCallback -> callback reverts -> CallbackFailed emitted with randomNumber
// 4. Read randomNumber from CallbackFailed event
// 5a. If favorable: call setAcceptOutcome(true), then call revealWithCallback again -> callback succeeds
// 5b. If unfavorable: do nothing, request stays in CALLBACK_FAILED state
```
