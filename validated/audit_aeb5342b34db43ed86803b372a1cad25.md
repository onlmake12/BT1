### Title
IPv4-Mapped IPv6 Address Group Collision Enables Inbound Peer Eviction Manipulation — (`network/src/network_group.rs`)

---

### Summary

`Group::from` in `network_group.rs` converts IPv4-mapped IPv6 addresses (e.g., `::ffff:1.2.3.4`) to the same `Group::IP4([1, 2])` bucket as native IPv4 peers from `1.2.x.x`. An attacker who connects with many such addresses inflates that /16 group, guaranteeing it is always selected as the largest group in `try_evict_inbound_peer`, and causing random eviction to fall on legitimate peers from that subnet.

---

### Finding Description

In `network_group.rs` lines 31–34, when the socket address is an `IpAddr::V6`, Rust's `to_ipv4()` is called unconditionally:

```rust
if let IpAddr::V6(ipv6) = ip_addr {
    if let Some(ipv4) = ipv6.to_ipv4() {   // matches ::ffff:a.b.c.d AND ::a.b.c.d
        let bits = ipv4.octets();
        return Group::IP4([bits[0], bits[1]]);
    }
``` [1](#0-0) 

`Ipv6Addr::to_ipv4()` returns `Some` for both IPv4-mapped (`::ffff:a.b.c.d`) and IPv4-compatible (`::a.b.c.d`) addresses. The result is that `::ffff:1.2.3.4` produces `Group::IP4([1, 2])` — identical to a native `1.2.3.4` peer.

In `try_evict_inbound_peer`, after the three protection passes (lowest ping, most-recent message, longest connection time), remaining candidates are bucketed by `peer.network_group()` and the **largest bucket is always chosen for eviction**:

```rust
.max_by_key(|group| group.len())
``` [2](#0-1) 

One peer is then chosen uniformly at random from that bucket:

```rust
evict_group.choose(&mut rng).map(|peer| { ... peer.session_id })
``` [3](#0-2) 

`peer.network_group()` delegates directly to `Group::from(&self.connected_addr)`: [4](#0-3)

### Impact Explanation

An attacker who controls N IPv4-mapped IPv6 addresses in `::ffff:1.2.x.x` joins the same `Group::IP4([1, 2])` as M legitimate native IPv4 peers. The combined group of size N+M is reliably the largest, so every eviction event targets it. Each eviction picks uniformly at random, giving a probability of M/(N+M) of hitting a legitimate peer. By keeping N large and continuously reconnecting evicted attacker peers, the attacker sustains pressure on the legitimate /16 cohort, gradually draining it. This degrades the victim node's diversity of honest inbound connections and can contribute to a network-partition / eclipse scenario.

---

### Likelihood Explanation

- IPv6 is widely supported; connecting with `::ffff:a.b.c.d` addresses is trivially achievable from any dual-stack host.
- No privileged access, PoW, or key material is required — only the ability to open inbound TCP connections to the target node.
- The inbound slot limit (`max_inbound`) is typically small (125 in Bitcoin-derived defaults), so the attacker needs only a modest number of connections to dominate a group.
- The protection passes (ping, recent message, connection time) do not prevent group inflation; they only protect individual peers within the already-inflated group.

---

### Recommendation

Replace the unconditional `to_ipv4()` call with `to_ipv4_mapped()` (stable since Rust 1.63), which returns `Some` **only** for the `::ffff:0:0/96` prefix, not for IPv4-compatible addresses. More importantly, IPv4-mapped IPv6 addresses should be assigned their own distinct group (e.g., keep them as `Group::IP6`) rather than collapsing them into `Group::IP4`. This prevents an attacker from artificially inflating a native IPv4 /16 group using IPv6 transport.

```rust
// Proposed fix in network_group.rs lines 31-35
if let IpAddr::V6(ipv6) = ip_addr {
    // Do NOT collapse mapped addresses into IP4 group
    let bits = ipv6.octets();
    return Group::IP6([bits[0], bits[1], bits[2], bits[3]]);
}
```

### Proof of Concept

A unit test mixing native IPv4 and IPv4-mapped IPv6 peers in the same /16 would confirm they hash to the same `Group` and that `try_evict_inbound_peer` always selects that combined group:

```rust
use std::net::{Ipv4Addr, Ipv6Addr};
let native   = Ipv4Addr::new(1, 2, 3, 4);
let mapped   = native.to_ipv6_mapped(); // ::ffff:1.2.3.4
// Both produce Group::IP4([1, 2]) under current code
assert_eq!(group_from_v4(native), group_from_v6_mapped(mapped));
```

With `max_inbound` slots filled by a mix of attacker (`::ffff:1.2.x.x`) and legitimate (`1.2.x.x`) peers, repeated calls to `try_evict_inbound_peer` will always target the inflated `IP4([1,2])` group, and over enough iterations, all legitimate peers in that /16 will be evicted.
