Audit Report

## Title
Unbounded O(N) Pool Iteration Holds tx-pool Write Lock on Every Block - (`tx-pool/src/process.rs`)

## Summary
`_update_tx_pool_for_reorg` in `tx-pool/src/process.rs` performs three unbounded O(N) iterations over all pool entries while holding the tx-pool write lock on every block. The pool is bounded only by serialized byte size (`max_tx_pool_size`), not by transaction count, allowing an unprivileged submitter to maximize N by filling the pool with many small transactions. This stalls all concurrent tx-pool operations (block template generation, transaction relay, submission) for the duration of each block's reorg processing.

## Finding Description

**Loop 1 & 2 — Gap and Pending full scans:** [1](#0-0) 

Both `get_by_status(&Status::Gap)` and `get_by_status(&Status::Pending)` iterate over every entry in the respective status bucket with no count cap.

**Loop 3 — `remove_expired` full pool scan:** [2](#0-1) 

`self.pool_map.iter()` visits every entry in the pool unconditionally. It is called at line 1110 of `_update_tx_pool_for_reorg` on every block. [3](#0-2) 

**`remove_entry` triggers ancestor/descendant BFS per removal:**
Each expired entry removed inside `remove_expired` calls `remove_entry`, which calls `update_ancestors_index_key` and `update_descendants_index_key`. [4](#0-3) 

Each of these calls `calc_ancestors`/`calc_descendants`, which is a BFS over the dependency graph. [5](#0-4) 

**No transaction count limit exists:**
`TxPoolConfig` has `max_tx_pool_size` (bytes) and `max_ancestors_count` (per-chain depth), but no total entry count cap. [6](#0-5) 

`PoolMap` enforces only `max_ancestors_count` and `total_tx_size`, not a total entry count. [7](#0-6) 

`limit_size` evicts only when `total_tx_size > max_tx_pool_size`, confirming the byte-only bound. [8](#0-7) 

With a default 180 MB pool and minimum CKB transactions of ~100–200 bytes, the pool can hold on the order of 900 K–1.8 M independent entries. All three loops run at O(N) per block under the write lock.

## Impact Explanation

All tx-pool operations — `send_transaction`, `get_block_template`, transaction relay — acquire the same write lock. While `_update_tx_pool_for_reorg` holds it iterating over a maximally-filled pool, every concurrent caller blocks. This directly delays block template generation for miners (missed block opportunities) and degrades transaction relay throughput. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points), as a single attacker node can degrade miner and relay performance across the network by keeping the pool filled.

## Likelihood Explanation

An unprivileged RPC caller or P2P peer can submit independent transactions each paying slightly above the minimum fee rate. Because `limit_size` evicts lowest-fee-rate entries first, the attacker only needs to outbid the floor. On testnet or a low-fee mainnet environment, filling a 180 MB pool with ~1 M minimum-size transactions is economically feasible. The attack is repeatable on every block since the pool refills between blocks. No special privileges, key material, or victim mistakes are required.

## Recommendation

1. **Add a `max_tx_pool_count` limit** to `PoolMap` enforced in `add_entry`, analogous to how `max_ancestors_count` is enforced in `check_and_record_ancestors`. Reject or evict when the count exceeds the cap. [9](#0-8) 

2. **Bound `remove_expired` iterations** by maintaining a timestamp-ordered index so only entries whose `expiry + timestamp < now_ms` are visited, eliminating the full-pool scan. [10](#0-9) 

3. **Bound the Gap/Pending scans in `_update_tx_pool_for_reorg`** by maintaining proposal-indexed sets so only entries matching current proposals are visited, rather than scanning all entries. [1](#0-0) 

## Proof of Concept

1. Start a CKB node with default config (`max_tx_pool_size = 180 MB`).
2. Via RPC, submit ~900 K independent minimal transactions (each spending a distinct UTXO, paying minimum fee rate + 1 shannons/byte) until `get_tx_pool_info` reports pool size near `max_tx_pool_size`.
3. In a separate thread, issue `get_block_template` or `send_transaction` RPC calls continuously.
4. On each new block arrival, observe that `get_block_template` latency spikes proportionally to pool entry count (measurable via RPC response time), confirming the write lock is held for the full O(N) iteration duration.
5. Compare baseline latency (empty pool) vs. filled pool to quantify the degradation.

    pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
        let now_ms = ckb_systemtime::unix_time_as_millis();

        let removed: Vec<_> = self
            .pool_map
            .iter()
            .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
            .map(|entry| entry.inner.clone())
            .collect();

        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            self.pool_map.remove_entry(&entry.proposal_short_id());
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }

    pub(crate) fn add_entry(
        &mut self,
        mut entry: TxEntry,
        status: Status,
    ) -> Result<(bool, HashSet<TxEntry>), Reject> {
        let tx_short_id = entry.proposal_short_id();
        let mut evicts = Default::default();
        if self.entries.get_by_id(&tx_short_id).is_some() {
            return Ok((false, evicts));
        }
        let (total_tx_size, total_tx_cycles) =
            self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
        trace!("pool_map.add_{:?} {}", status, entry.transaction().hash());
        evicts = self.check_and_record_ancestors(&mut entry)?;
        self.record_entry_edges(&entry)?;
        self.insert_entry(&entry, status);
        self.record_entry_descendants(&entry);
        self.track_entry_statics(None, Some(status));
        self.total_tx_size = total_tx_size;
        self.total_tx_cycles = total_tx_cycles;
        Ok((true, evicts))
    }
