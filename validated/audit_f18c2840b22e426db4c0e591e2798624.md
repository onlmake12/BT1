### Title
Softfork Activation Threshold Computed with Floor Division Instead of Ceiling, Allowing Lock-In Below Configured Minimum Ratio - (File: spec/src/versionbits/mod.rs)

### Summary
The `threshold_number` function in `spec/src/versionbits/mod.rs` computes the minimum number of signaling blocks required to lock in a softfork using integer floor division. Because the threshold is defined as a *minimum ratio*, the correct rounding direction is ceiling (round up). Using floor division silently lowers the effective activation threshold below the configured value, allowing a softfork to lock in with fewer signaling blocks than the protocol intends.

### Finding Description
The function `threshold_number` at line 475–479 of `spec/src/versionbits/mod.rs` computes:

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| ret.checked_div(threshold.denom()))
}
```

`checked_div` performs integer floor division (truncation toward zero). The result is used directly in the lock-in check at line 342–344:

```rust
let threshold_number = threshold_number(total, self.threshold())?;
if count >= threshold_number {
    next_state = ThresholdState::LockedIn;
```

The `threshold` field of a `Deployment` is documented as "the minimum ratio of block per period which indicate the locked_in of the softfork." To correctly enforce a *minimum* ratio `numer/denom`, the required count must satisfy:

```
count / total >= numer / denom
⟺ count >= total * numer / denom
```

Because `count` is an integer, the right-hand side must be rounded **up** (ceiling). The current code rounds **down** (floor), so the enforced threshold is:

```
floor(total * numer / denom) / total  <  numer / denom
```

whenever `total * numer` is not exactly divisible by `denom`.

**Concrete example** — mainnet Light Client softfork uses threshold 80% = 4/5. If a signaling period spans `total = 1001` blocks:
- Correct minimum: `ceil(1001 × 4 / 5) = ceil(800.8) = 801` blocks → 80.02%
- Actual minimum enforced: `floor(1001 × 4 / 5) = 800` blocks → 79.92%

The softfork locks in with only 79.92% miner support instead of the configured 80%.

### Impact Explanation
The effective activation threshold is silently lower than the configured `Deployment.threshold` value whenever `total * numer % denom != 0`. A softfork can transition from `Started` to `LockedIn` with a miner signaling fraction strictly below the configured minimum ratio. This is a consensus accounting error: the protocol's stated invariant ("at least `threshold` fraction of blocks must signal") is violated. All nodes running this code agree on the incorrect threshold, so there is no chain split, but the protocol guarantee is weakened — softforks activate with less miner support than intended.

### Likelihood Explanation
This is triggered automatically whenever a signaling period's total block count is not an exact multiple of the threshold denominator. For the mainnet Light Client deployment (threshold 4/5, period 2 epochs), epoch lengths vary due to the difficulty adjustment algorithm, making non-divisible totals the common case. Any miner signaling for a softfork reaches this code path without any privileged access.

### Recommendation
Replace floor division with ceiling division in `threshold_number`:

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .map(|ret| ret.div_ceil(threshold.denom()))
}
```

`u64::div_ceil` (stable since Rust 1.73) computes `ceil(a / b)` without overflow risk. This ensures the enforced count satisfies `count / total >= numer / denom` for all integer inputs.

### Proof of Concept

**Setup**: Deploy a softfork with `threshold = Ratio::new(4, 5)` (80%). Construct a signaling period where `total = 1001` blocks.

**Current behavior**:
- `threshold_number(1001, Ratio::new(4, 5))` = `floor(1001 * 4 / 5)` = `floor(800.8)` = `800`
- A miner coalition controlling 800/1001 = **79.92%** of blocks causes `count >= threshold_number` → `LockedIn`

**Expected behavior**:
- `ceil(1001 * 4 / 5)` = `ceil(800.8)` = `801`
- Lock-in requires 801/1001 = **80.02%** of blocks, correctly enforcing the ≥80% invariant

The attacker-controlled entry path is: a miner (or coordinated miners) submits blocks with the softfork version bit set. The `condition()` check at line 330 counts these blocks.
