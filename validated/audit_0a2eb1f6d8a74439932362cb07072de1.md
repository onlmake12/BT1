Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Permanently Inflated by Stale Pre-Eviction Locals in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, the new totals are computed into local variables before `check_and_record_ancestors` runs. When that function evicts existing pool entries via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, it correctly decrements `self.total_tx_size` and `self.total_tx_cycles`. However, `add_entry` then unconditionally overwrites those fields with the stale pre-eviction locals, erasing every decrement. Each successful trigger permanently inflates the counters by the total size/cycles of all evicted transactions, eventually causing `limit_size` to evict legitimate transactions and reject all new submissions with `Reject::Full`.

## Finding Description

**Step 1 — stale locals computed before any eviction:** [1](#0-0) 

`updated_stat_for_add_tx` takes `&self` and returns `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` without touching `self`. [2](#0-1) 

**Step 2 — eviction branch inside `check_and_record_ancestors` modifies `self`:**

The branch fires when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. [3](#0-2) 

Each call to `remove_entry_and_descendants` calls `remove_entry` for every removed id: [4](#0-3) 

`remove_entry` calls `update_stat_for_remove_tx`, which decrements `self.total_tx_size` and `self.total_tx_cycles`: [5](#0-4) [6](#0-5) 

**Step 3 — stale locals overwrite the post-eviction state:** [7](#0-6) 

Net result after one trigger:
```
self.total_tx_size  = old_total + new_entry_size          (written)
correct value       = old_total − Σ(evicted_sizes) + new_entry_size
inflation           = Σ(evicted_sizes)
```

The inflation is cumulative and permanent; no code path resets these counters except `recompute_total_stat`, which is only called on underflow.

## Impact Explanation

`limit_size` uses `total_tx_size` as its sole guard: [8](#0-7) 

An inflated counter causes `limit_size` to evict legitimate transactions that should remain in the pool, and causes `updated_stat_for_add_tx` to return `Reject::Full` for new submissions even when the pool has ample real capacity. By repeatedly triggering the eviction path, an attacker can drive `total_tx_size` arbitrarily high, making the tx-pool permanently reject all new transactions on the targeted node. This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, since a targeted node stops relaying and accepting transactions, degrading network propagation.

## Likelihood Explanation

Any unprivileged actor with access to `send_transaction` RPC or the P2P relay protocol can trigger this. The required setup is a chain of ~25 transactions where at least one ancestor uses a shared cell as a `cell_dep` (making it a "cell-ref parent"), followed by a new transaction whose ancestor count exceeds `max_ancestors_count` (default 25) but whose cell-ref parents account for the excess. No privileged keys, no majority hashpower, and no social engineering are required. The attack is repeatable and each iteration adds more inflation.

## Recommendation

Move the stat update to **after** `check_and_record_ancestors` completes, so evictions are already reflected before the new entry's contribution is added:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, Default::default()));
    }
    // Validate overflow potential before evictions, but do NOT store the result.
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    let evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Apply new entry's contribution on top of the post-eviction totals.
    self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
    self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
    Ok((true, evicts))
}
```

## Proof of Concept

1. Pre-fill the pool with a chain of 24 transactions `T1…T24` where `T_k` spends the output of `T_{k-1}`, and `T1` also uses a live cell `C` as a `cell_dep`.
2. Submit transaction `T25` that spends `T24`'s output and also uses cell `C` as a `cell_dep`. Now `T1` is a "cell-ref parent" of `T25`.
3. Submit transaction `T26` that spends `T25`'s output. Its ancestor count is 26 > 25 (`max_ancestors_count`), but `cell_ref_parents = {T1}`, so `26 − 1 = 25 ≤ 25`. The eviction branch fires: `T1` (and its descendants) is removed via `remove_entry_and_descendants`, decrementing `self.total_tx_size` by `Σ sizes(T1…descendants)`. Then lines 218–219 overwrite `self.total_tx_size` with the pre-eviction local, inflating it by `Σ sizes(T1…descendants)`.
4. Repeat steps 1–3 with fresh UTXOs. Each iteration adds more inflation.
5. Observe that `limit_size` begins evicting valid transactions and `updated_stat_for_add_tx` starts returning `Reject::Full` for new submissions despite the pool being well under its real capacity limit.
