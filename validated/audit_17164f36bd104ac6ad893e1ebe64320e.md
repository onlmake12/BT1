### Title
Excess `msg.value` Permanently Lost to Pyth Fee Pool in `Entropy.requestHelper` — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
In `Entropy.sol`, the `requestHelper` function consumes the **entire** `msg.value` by splitting it between the provider's accrued fees and Pyth's accrued fees. Any amount sent above the required fee is silently absorbed into `_state.accruedPythFeesInWei` and is irrecoverable by the caller. This is the direct analog of the PoolFactory ETH-locking bug: a user sends ETH, the contract logic routes it away from the user with no refund path.

### Finding Description
In `requestHelper` (called by `request`, `requestWithCallback`, `requestV2`), the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is consumed. The excess beyond `requiredFee` is added to `_state.accruedPythFeesInWei` — the Pyth protocol's fee pool — not returned to the caller. There is no refund path for the user.

This is explicitly documented in the interface:

> "excess value is *not* refunded to the caller" [2](#0-1) 

By contrast, `PythLazer.sol`'s `verifyUpdate` **does** refund excess:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [3](#0-2) 

The same no-refund pattern exists in `Pyth.sol`'s `updatePriceFeeds`:

```solidity
uint requiredFee = getTotalFee(totalNumUpdates);
if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
``` [4](#0-3) 

And in `Echo.sol`'s `requestPriceUpdatesWithCallback`, where `req.fee = msg.value - pythFeeInWei` stores the full overpayment as the provider's fee rather than refunding it: [5](#0-4) 

### Impact Explanation
Any user who sends more ETH than `getFeeV2()` requires — whether due to a race condition where the provider raises fees between estimation and execution, a hardcoded fee value, or a simple mistake — permanently loses the excess ETH. The excess is credited to Pyth's fee pool (`accruedPythFeesInWei`), which is only withdrawable by Pyth governance, not the original sender. The user has no recourse.

### Likelihood Explanation
Provider fees can change at any time via `setProviderFee`. A user who calls `getFeeV2()` off-chain and then submits a transaction with a small buffer (e.g., `fee * 1.1`) to avoid `InsufficientFee` reverts will silently lose the buffer. This is a common defensive pattern in EVM development. Integrators building on top of Entropy (e.g., smart contracts that hardcode a fee or add a safety margin) are particularly exposed.

### Recommendation
Refund excess `msg.value` to the caller in `requestHelper`, consistent with the behavior already implemented in `PythLazer.verifyUpdate`:

```solidity
if (msg.value > requiredFee) {
    (bool sent, ) = msg.sender.call{value: msg.value - requiredFee}("");
    require(sent, "refund failed");
}
```

Alternatively, enforce `msg.value == requiredFee` exactly (revert on overpayment), so users are never silently overcharged.

### Proof of Concept
1. Provider `P` has `feeInWei = 1000 wei`, Pyth fee = `100 wei`. `getFeeV2(P, 0)` returns `1100 wei`.
2. User calls `getFeeV2` off-chain, gets `1100 wei`, adds a 10% buffer, and sends `1210 wei` with `requestV2(P, randomNum, 0)`.
3. Inside `requestHelper`: `providerFee = 1000`, `providerInfo.accruedFeesInWei += 1000`, `_state.accruedPythFeesInWei += (1210 - 1000) = 210`.
4. The user paid `1210 wei` but only `1100 wei` was required. The extra `110 wei` is now in Pyth's fee pool.
5. The user cannot recover the `110 wei`. Only Pyth governance (via `withdrawFee`) can access it. [6](#0-5)
