### Title
`prune()` skips Header/TxHash deletion when no ConsumedOutPoint entries exist, causing unbounded storage growth — (`util/indexer/src/indexer.rs`)

### Summary
`Indexer::prune()` uses `min_block_number = BlockNumber::MAX` as the seek key for Header pruning. This value is only updated if at least one `ConsumedOutPoint` entry falls below `prune_to_block`. On a chain with no cell-spending transactions (coinbase-only blocks), no `ConsumedOutPoint` entries exist, so `min_block_number` stays at `u64::MAX`, the Header iterator seeks past all real entries, and neither `Header` nor `TxHash` records are ever deleted.

### Finding Description
In `Indexer::prune()`:

```rust
let mut min_block_number = BlockNumber::MAX;   // line 765
for (block_number, key) in iter
    ...
    .take_while(|(block_number, _key)| prune_to_block.gt(block_number))
{
    if min_block_number == BlockNumber::MAX {
        min_block_number = block_number;        // only set if loop body runs
    }
    batch.delete(key)?;
}

// Header pruning seek key built from min_block_number
let mut key_prefix_header = vec![KeyPrefix::Header as u8];
key_prefix_header.extend_from_slice(&min_block_number.to_be_bytes()); // line 785
``` [1](#0-0) 

When the `ConsumedOutPoint` iterator yields nothing (no cell-spending txs), `min_block_number` remains `u64::MAX`. The resulting seek key `[0xE0, 0xFF×8]` is lexicographically beyond every real Header key, so the forward iterator returns nothing and the `take_while` exits immediately. [2](#0-1) 

Meanwhile, `append()` writes a `Key::Header` entry for every block unconditionally: [3](#0-2) 

And writes a `Key::TxHash` entry for every matched transaction (including coinbase): [4](#0-3) 

These entries accumulate indefinitely because `prune()` never removes them.

### Impact Explanation
Every block appended to the indexer writes at least one `Header` entry and one `TxHash` entry. Without pruning, these grow linearly with chain height. Over time this exhausts disk space, causing RocksDB write failures and stalling the node. The indexer is a production component enabled by default in many deployments.

### Likelihood Explanation
The condition is triggered whenever the indexed chain has no non-coinbase transactions spending previously-indexed cells for a window longer than `keep_num` blocks. This is realistic on:
- Any newly-started or low-activity CKB chain/testnet.
- A chain segment where all spending transactions are filtered out by `custom_filters`.

No majority hashpower is required; the bug fires under ordinary low-activity conditions. An adversary who can relay a sequence of valid coinbase-only blocks (e.g., on a private or test network) can trigger it deterministically.

### Recommendation
Replace the sentinel with `0` so Header pruning always starts from the beginning of the Header key space:

```rust
// Before
let mut min_block_number = BlockNumber::MAX;

// After
let mut min_block_number = 0u64;
```

Alternatively, decouple the Header pruning start from the `ConsumedOutPoint` loop entirely and always seek from `[KeyPrefix::Header as u8]` with a `block_number <= prune_to_block` bound.

### Proof of Concept
1. Create an indexer with `keep_num = 10`, `prune_interval = 1`.
2. Append 25 coinbase-only blocks (no non-coinbase transactions).
3. Call `prune()` (triggered automatically at each `prune_interval` multiple).
4. Count `Key::Header` entries in the store — expect ≤ 11, observe 25 (none pruned).
5. Count `Key::TxHash` entries — same result.

The existing test `prune_should_not_delete_live_cells` (lines 1687–1769) uses exactly this coinbase-only pattern but does not assert that Header/TxHash counts decrease, so the bug is untested. [5](#0-4)
