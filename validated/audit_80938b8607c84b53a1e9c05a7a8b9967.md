Audit Report

## Title
Missing `BLOCK_INVALID` Guard in `HeaderAcceptor::accept()` Allows Re-elevation of Invalid Headers and Repeated Block Download Cycles — (`sync/src/synchronizer/headers_process.rs`)

## Summary
`HeaderAcceptor::accept()` contains an acknowledged `// FIXME` comment noting the missing `BLOCK_INVALID` guard. Because `BLOCK_INVALID = 1 << 12` shares no bits with `HEADER_VALID = 1`, a header already marked invalid in `block_status_map` bypasses the early-return, passes all three non-contextual sub-checks, and causes `insert_valid_header()` to be called — inserting the header into `header_map` and updating the peer's `best_known_header`. The block fetcher then issues repeated `GetBlocks` requests for a block the node already knows is invalid, creating a persistent download-and-re-verify cycle.

## Finding Description

**Bit-flag gap — confirmed:**
`BLOCK_INVALID = 1 << 12 = 4096` is an isolated bit with zero overlap with the `HEADER_VALID` chain (`1, 3, 7, 15`). [1](#0-0) 

**Missing guard in `accept()` — confirmed:**
The FIXME comment is present. The only early-return fires on `HEADER_VALID`; `BLOCK_INVALID` falls through to full re-validation. [2](#0-1) 

**Sub-checks pass for contextually-invalid headers:**
`prev_block_check` only rejects headers whose *parent* is `BLOCK_INVALID`. `non_contextual_check` runs `HeaderVerifier` (PoW + non-contextual rules). `version_check` requires `version == 0`. A header with valid PoW that failed a contextual rule (e.g., median-time-past, which requires knowing prior timestamps) passes all three. [3](#0-2) 

**`insert_valid_header()` inserts into `header_map` only — confirmed:**
It calls `header_map().insert(...)` and `may_set_best_known_header(...)` but never touches `block_status_map`. [4](#0-3) 

**`get_block_status` checks `block_status_map` first — confirmed:**
If the hash is present in `block_status_map`, that value (including `BLOCK_INVALID`) is returned immediately, regardless of `header_map`. [5](#0-4) 

**Block fetcher has no `BLOCK_INVALID` guard — confirmed:**
Only `BLOCK_STORED` and `BLOCK_RECEIVED` are checked. A header with status `BLOCK_INVALID` falls through to the `else if` branch and is inserted into the inflight list. [6](#0-5) 

**Chain verifier sets `BLOCK_INVALID` in `block_status_map`:**
`chain/src/chain_service.rs` (2 call sites) and `chain/src/verify.rs` (1 call site) call `insert_block_status(..., BLOCK_INVALID)` when contextual block verification fails — confirming that a header with valid PoW but a failing contextual rule will land in `block_status_map` as `BLOCK_INVALID`. [7](#0-6) 

**Every other status-check site guards `BLOCK_INVALID` explicitly:**
`compact_block_process.rs`, `orphan_broker.rs`, and `verify.rs` all check for `BLOCK_INVALID` before proceeding. [8](#0-7) [9](#0-8) 

**Exploit flow:**
1. Attacker submits a full block (or compact block with all transactions resolvable) whose header H has valid PoW but violates median-time-past. The chain verifier runs, fails contextual verification, and calls `insert_block_status(H, BLOCK_INVALID)`.
2. Attacker sends H via `SendHeaders`. `accept()` reads `status = BLOCK_INVALID`; the `HEADER_VALID` guard does not fire; `prev_block_check`, `non_contextual_check`, and `version_check` all pass; `insert_valid_header(H)` is called.
3. H is now in `header_map`; peer's `best_known_header` is set to H.
4. Block fetcher traverses from `best_known_header = H`. `get_block_status(H)` returns `BLOCK_INVALID` (from `block_status_map`). This does not match `BLOCK_STORED` or `BLOCK_RECEIVED`, so H is added to inflight and `GetBlocks` is issued.
5. Full block arrives, fails contextual verification again, is re-marked `BLOCK_INVALID`. H remains in `header_map`; `best_known_header` is not downgraded.
6. On the next block-fetch timer tick, step 4 repeats indefinitely without further attacker action.

## Impact Explanation

The node enters a persistent cycle of issuing `GetBlocks` for a known-invalid block, downloading the full block, re-running contextual verification, and re-marking it invalid on every block-fetch timer tick. This wastes CPU and bandwidth continuously. It does not crash the node, cause consensus deviation, or damage the economy. Impact: **Low (501–2000 points): Any other important performance improvements for CKB** — specifically eliminating avoidable repeated block downloads and re-verification work caused by a missing status guard.

## Likelihood Explanation

Any connected P2P peer can send `SendHeaders` messages freely. The precondition — obtaining a header H that is `BLOCK_INVALID` on the target node and passes non-contextual checks — requires mining a block with valid PoW that fails a contextual rule (e.g., median-time-past). This is a one-time real-hashpower cost. Once H is in `header_map`, the repeated download cycle runs automatically on every block-fetch timer tick without further attacker action. The attack is not repeatable across restarts because `block_status_map` is an in-memory `DashMap` that is repopulated only from on-disk `block_ext` data; a block that failed contextual verification and was not stored would not be repopulated, requiring the attacker to re-trigger the chain-verification failure after each restart.

## Recommendation

Add an explicit `BLOCK_INVALID` guard at the top of `accept()`, before the `HEADER_VALID` check, resolving the existing `// FIXME` comment:

```rust
pub fn accept(&self) -> ValidationResult {
    let mut result = ValidationResult::default();
    let sync_shared = self.active_chain.sync_shared();
    let state = self.active_chain.state();
    let shared = sync_shared.shared();

    let status = self.active_chain.get_block_status(&self.header.hash());

    // Resolves the FIXME: guard against already-invalid headers
    if status.contains(BlockStatus::BLOCK_INVALID) {
        result.invalid(None);
        return result;
    }

    if status.contains(BlockStatus::HEADER_VALID) {
        // ... existing path
    }
    // ... rest of validation
}
```

This prevents `insert_valid_header` from being called for headers already known to be invalid, consistent with every other status-check site in the codebase. [10](#0-9) 

## Proof of Concept

1. Connect to a CKB node as a P2P peer using the Sync + Relay protocols.
2. Mine a block H with valid PoW whose `timestamp` is below the median-time-past of the current tip (violating the MTP rule, a contextual check).
3. Send H as a full block via the Relay protocol. Observe via debug logging that the chain verifier calls `insert_block_status(H, BLOCK_INVALID)`.
4. Send a `SendHeaders` message containing H via the Sync protocol.
5. Observe the `"inserted valid header"` log line (emitted at multiples of 10000 blocks) or confirm via `block_status_map` inspection that `insert_valid_header` was called despite `BLOCK_INVALID` status; confirm the peer's `best_known_header` is now H.
6. Observe that on the next block-fetch timer tick (`IBD_BLOCK_FETCH_INTERVAL` or `NOT_IBD_BLOCK_FETCH_INTERVAL`), the node issues another `GetBlocks` for H's hash without any further attacker action, confirming the persistent download cycle.

    pub fn non_contextual_check(&self, state: &mut ValidationResult) -> Result<(), bool> {
        self.verifier.verify(self.header).map_err(|error| {
            debug!(
                "HeadersProcess accepted {:?} error {:?}",
                self.header.number(),
                error
            );
            // UnknownParentError surfaces as BlockError(UnknownParent), not
            // HeaderError.  Missing parent is a recoverable ordering/context
            // issue, not proof that this header is invalid.
            if error
                .downcast_ref::<BlockError>()
                .is_some_and(|e| e.kind() == BlockErrorKind::UnknownParent)
            {
                state.temporary_invalid(Some(ValidationError::Verify(error)));
                false
            } else if let Some(header_error) = error.downcast_ref::<HeaderError>() {
                if header_error.is_too_new() {
                    state.temporary_invalid(Some(ValidationError::Verify(error)));
                    false
                } else {
                    state.invalid(Some(ValidationError::Verify(error)));
                    true
                }
            } else {
                state.invalid(Some(ValidationError::Verify(error)));
                true
            }
        })
    }

    pub fn version_check(&self, state: &mut ValidationResult) -> Result<(), ()> {
        if self.header.version() != 0 {
            state.invalid(Some(ValidationError::Version));
            Err(())
        } else {
            Ok(())
        }
    }
