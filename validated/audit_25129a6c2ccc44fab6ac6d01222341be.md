### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp, Allowing Stale Per-Feed Prices to Pass - (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

The Pyth Pulse `Scheduler` contract's `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions check staleness against a single subscription-level timestamp (`priceLastUpdatedAt`) that is set to the **maximum** `publishTime` across all feeds in a batch. Individual feeds for closed markets can have `publishTime` values far older than `age_seconds`, yet they are returned to callers without triggering a revert, violating the function's own API contract.

### Finding Description

When `updatePriceFeeds` is called on the Scheduler, `_validateShouldUpdatePrices` computes `updateTimestamp` as the **maximum** `publishTime` across all feeds in the batch: [1](#0-0) 

This maximum is stored as the subscription's `priceLastUpdatedAt`: [2](#0-1) 

The design rationale is that closed markets have stale `publishTime` values from their last trading period, and the same-slot check ensures synchronization: [3](#0-2) 

However, when a consumer calls `getPricesNoOlderThan`, the staleness gate only checks `priceLastUpdatedAt` (the max across all feeds) against `age_seconds`: [4](#0-3) 

The same pattern applies to `getEmaPricesNoOlderThan`: [5](#0-4) 

The function then returns prices via `getPricesUnsafe`, which returns individual feed `publishTime` values that can be arbitrarily older than `age_seconds`. The test `testUpdatePriceFeedsSucceedsWithStaleFeedIfLatestIsValid` explicitly confirms that a feed with `stalePublishTime` (outside `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`, i.e., >1 hour old) is stored alongside a valid feed, and `priceLastUpdatedAt` is set to the valid (recent) timestamp: [6](#0-5) 

A consumer calling `getPricesNoOlderThan(subscriptionId, [closedMarketFeedId], 60)` expecting prices no older than 60 seconds will receive a price whose `publishTime` is hours old, with no revert.

The `SubscriptionStatus` struct documents `priceLastUpdatedAt` as "Timestamp of the last update. All feeds in the subscription are updated together," which is misleading when individual feed timestamps diverge significantly: [7](#0-6) 

### Impact Explanation

Any whitelisted reader (a DeFi protocol, AMM, lending protocol, or derivative platform) that calls `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` with a tight `age_seconds` window to ensure price freshness will silently receive stale prices for closed-market feeds. The function's API contract — that all returned prices are no older than `age_seconds` — is violated. This can lead to incorrect swap outputs, incorrect collateral valuations, or incorrect LP token calculations in downstream protocols, directly mirroring the reported vulnerability class.

### Likelihood Explanation

The Scheduler is designed to support subscriptions with mixed feeds (e.g., equities alongside crypto). Closed-market equity feeds are a documented and expected case. Any subscription containing at least one closed-market feed alongside an active feed will exhibit this behavior on every update. Whitelisted readers are the intended consumers of the Scheduler and will routinely call `getPricesNoOlderThan` expecting the staleness guarantee to hold per-feed.

### Recommendation

`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` should check staleness against each individual feed's `publishTime`, not the subscription-level `priceLastUpdatedAt`. The check should iterate over the returned feeds and revert if any individual feed's `publishTime` violates the `age_seconds` constraint:

```solidity
for (uint i = 0; i < prices.length; i++) {
    if (distance(block.timestamp, prices[i].publishTime) > age_seconds)
        revert PythErrors.StalePrice();
}
```

Alternatively, the documentation and interface should be updated to clearly state that `getPricesNoOlderThan` only guarantees that the **most recent** feed in the subscription is no older than `age_seconds`, and consumers must check individual `publishTime` fields for per-feed freshness.

### Proof of Concept

1. Create a Scheduler subscription with two price feeds: `feedA` (active crypto market) and `feedB` (closed equity market).
2. Call `updatePriceFeeds` with `feedA.publishTime = block.timestamp` and `feedB.publishTime = block.timestamp - 7200` (2 hours ago), both in the same Pythnet slot. This succeeds because `updateTimestamp = max(feedA.publishTime, feedB.publishTime) = block.timestamp`, which passes `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`.
3. `priceLastUpdatedAt` is set to `block.timestamp`.
4. Call `getPricesNoOlderThan(subscriptionId, [feedB_id], 60)`. The staleness check: `distance(block.timestamp, priceLastUpdatedAt) = 0 <= 60` — passes.
5. The returned price for `feedB` has `publishTime = block.timestamp - 7200`, which is 2 hours old — far exceeding the requested 60-second freshness guarantee — yet no revert occurs. [8](#0-7) [9](#0-8)

    function getPricesNoOlderThan(
        uint256 subscriptionId,
        bytes32[] calldata priceIds,
        uint256 age_seconds
    )
        external
        view
        override
        onlyWhitelistedReader(subscriptionId)
        returns (PythStructs.Price[] memory prices)
    {
        SchedulerStructs.SubscriptionStatus memory status = _state
            .subscriptionStatuses[subscriptionId];

        // Use distance (absolute difference) since pythnet timestamps
        // may be slightly ahead of this chain.
        if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
            revert PythErrors.StalePrice();

        prices = this.getPricesUnsafe(subscriptionId, priceIds);
    }
