### Title
Silent `u128→u64` Truncation in `calculate_maximum_withdraw` Produces Wrong Withdrawal Capacity — (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes a `u128` intermediate value and casts it to `u64` with `as u64`, silently truncating on overflow. Every other analogous u128→u64 conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. The missing check means that for a sufficiently large DAO deposit, the returned withdrawal capacity is silently wrong (too small), causing downstream `transaction_fee` to underflow and reject the withdrawal transaction from the tx-pool as malformed.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the withdrawable capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

`withdraw_counted_capacity` is a `u128`. If it exceeds `u64::MAX`, the `as u64` cast silently wraps to a small value. No error is returned.

Contrast this with every other u128→u64 narrowing in the same file, which all use the checked form:

```rust
// dao_field_with_current_epoch – miner_issuance
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
// dao_field_with_current_epoch – ar_increase
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
// secondary_block_reward
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) 

The inconsistency is the root cause. The `DaoError::ZeroC` guard in `genesis_dao_data_with_satoshi_gift` confirms the authors are aware that division-by-zero and arithmetic anomalies in DAO calculations must be guarded: [4](#0-3) 

The existing test `check_withdraw_calculation_overflows` confirms the overflow scenario is known and expected to return an error — but the test only passes because the truncated value happens to cause `safe_add` to overflow in that specific test case. For other large-but-not-maximum deposit values, the truncation produces a silently wrong small number that does **not** trigger `safe_add` overflow, so `Ok(wrong_capacity)` is returned. [5](#0-4) 

---

### Impact Explanation

`calculate_maximum_withdraw` is called from two paths:

**Path 1 — tx-pool admission (`send_transaction` RPC):**

```
send_transaction RPC
  → check_tx_fee (tx-pool/src/util.rs:28)
    → DaoCalculator::transaction_fee
      → transaction_maximum_withdraw
        → calculate_maximum_withdraw   ← truncation here
``` [6](#0-5) 

`transaction_fee` computes `maximum_withdraw - outputs_capacity`. If `maximum_withdraw` is silently truncated to a value smaller than `outputs_capacity`, `safe_sub` returns a capacity underflow error, which is mapped to `Reject::Malformed`. The withdrawal transaction is permanently rejected from the tx-pool. The depositor cannot reclaim their DAO funds through the normal submission path.

**Path 2 — RPC `calculate_dao_maximum_withdraw`:**

The RPC returns the silently truncated (wrong, too-small) capacity to the caller, misleading wallets and tooling about the actual withdrawable amount. [7](#0-6) 

---

### Likelihood Explanation

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`deposit_ar` starts at `10_000_000_000_000_000` (10^16) at genesis. `withdrawing_ar` grows slowly. For the ratio to exceed 1 + ε, only a small amount of time needs to pass. The binding constraint is `counted_capacity` being close to `u64::MAX` (≈ 1.84 × 10^19 shannons ≈ 184 billion CKB). While this is a large amount, institutional or protocol-level DAO depositors (e.g., foundation reserves, large custodians) could reach this threshold. The condition is not theoretical — the existing test explicitly constructs it with `18_446_744_073_709_550_000` shannons. [8](#0-7) 

---

### Recommendation

Replace the silent `as u64` cast with the same checked pattern used everywhere else in the file:

```rust
// Before (line 155-156):
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

// After:
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    ).safe_add(occupied_capacity)?;
``` [9](#0-8) 

---

### Proof of Concept

1. Create a DAO deposit cell with `capacity = u64::MAX - 1` shannons and minimal occupied capacity, so `counted_capacity ≈ u64::MAX`.
2. Mine enough blocks so that `withdrawing_ar > deposit_ar` (any non-zero interest accrual suffices).
3. Construct a DAO withdrawal transaction (phase 2) spending that cell, with `outputs_capacity` set to the correct interest-bearing amount.
4. Submit via `send_transaction` RPC.
5. **Observed:** The tx-pool calls `calculate_maximum_withdraw`, which computes `withdraw_counted_capacity > u64::MAX`, truncates it via `as u64` to a small value, then `transaction_fee` calls `maximum_withdraw.safe_sub(outputs_capacity)` which underflows, and the transaction is rejected with `Reject::Malformed("...", "expect (outputs capacity) <= (inputs capacity)")`.
6. **Expected:** `DaoError::Overflow` should be returned, or the correct capacity should be computed without truncation.

The existing unit test at `util/dao/src/tests.rs:295` already demonstrates the overflow scenario and asserts `result.is_err()`, confirming the overflow path is reachable — but the test does not verify that the error is `DaoError::Overflow` rather than a coincidental `safe_add` overflow on the truncated value. [10](#0-9)

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }

                    .get_transaction(&out_point.tx_hash())
