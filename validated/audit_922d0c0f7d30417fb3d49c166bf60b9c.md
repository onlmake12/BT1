### Title
Tx-Pool `total_tx_size`/`total_tx_cycles` Inflated by Evicted Transactions Due to Stale Snapshot Overwrite - (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool-wide accounting fields `total_tx_size` and `total_tx_cycles` are computed as temporary snapshot variables **before** a potential mid-function eviction path runs. When that eviction path fires, it correctly decrements the live fields via `update_stat_for_remove_tx`. However, the pre-computed snapshots are then unconditionally written back, silently overwriting the decrements. The result is that `total_tx_size` and `total_tx_cycles` are permanently inflated by the sizes and cycles of every evicted transaction, causing the pool to believe it is larger than it actually is and triggering spurious eviction of legitimate transactions.

---

### Finding Description

`add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```rust
// Step 1 – snapshot: captures self.total_tx_size + entry.size
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // lines 210-211

// Step 2 – may evict: calls remove_entry_and_descendants → update_stat_for_remove_tx
//          which DECREMENTS self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// Step 3 – overwrite: stale snapshot clobbers the decremented live values
self.total_tx_size  = total_tx_size;                            // line 218
self.total_tx_cycles = total_tx_cycles;                         // line 219
```

`updated_stat_for_add_tx` reads `self.total_tx_size` and `self.total_tx_cycles` at the moment of the call and adds the new entry's metrics:

```rust
let total_tx_size = self.total_tx_size.checked_add(tx_size)...;   // line 716
let total_tx_cycles = self.total_tx_cycles.checked_add(cycles)...; // line 722
```

`check_and_record_ancestors` contains an eviction branch that fires when the new transaction's ancestor count exceeds `max_ancestors_count` but can be reduced by removing `cell_ref_parents`:

```rust
while ancestors_count > self.max_ancestors_count {
    if let Some(next_id) = iter.next() {
        let removed = self.remove_entry_and_descendants(next_id); // line 618
        ...
        evicted.extend(removed);
    }
}
```

`remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx` correctly decrements `self.total_tx_size` and `self.total_tx_cycles` for every evicted transaction. But those decrements are immediately overwritten at lines 218-219 by the stale snapshot that was computed before any eviction occurred.

**Concrete arithmetic example:**

| Step | Event | `self.total_tx_size` |
|------|-------|----------------------|
| 0 | Initial pool | 1000 |
| 1 | `updated_stat_for_add_tx(size=50)` → snapshot = 1050 | 1000 (unchanged) |
| 2 | Evict tx of size 200 via `update_stat_for_remove_tx` | 800 |
| 3 | `self.total_tx_size = total_tx_size` (snapshot) | **1050** ← wrong |

Correct value should be `1000 − 200 + 50 = 850`. The pool is now over-counted by 200 (the evicted tx's size).

---

### Impact Explanation

`limit_size` in `tx-pool/src/pool.rs` uses `self.pool_map.total_tx_size` directly as the eviction trigger:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee transactions
}
```

An inflated `total_tx_size` causes `limit_size` to believe the pool is over capacity when it is not, triggering cascading eviction of legitimate, fee-paying transactions. An attacker who can repeatedly trigger the eviction path in `check_and_record_ancestors` can inflate `total_tx_size` without bound, eventually causing every newly submitted transaction to be immediately evicted by `limit_size` (called in `submit_entry` right after `add_entry`). This constitutes a **tx-pool denial-of-service**: the pool appears full to the size-limit check while actually holding far fewer bytes than the configured maximum.

`total_tx_cycles` is inflated by the same mechanism, affecting the cycles-based dimension of pool accounting and fee estimation (`estimate_fee_rate`).

---

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is reachable by any unprivileged `send_transaction` RPC caller. The attacker needs to:

1. Submit a set of transactions where some act as cell-dep providers (`cell_ref_parents`).
2. Submit a new transaction whose ancestor count exceeds `max_ancestors_count` (default 25) but whose `cell_ref_parents` count brings it back within the limit — the exact condition at line 603.
3. Repeat. Each iteration inflates `total_tx_size` by the size of the evicted cell-dep transaction.

No privileged access, no key material, no majority hash power, and no Sybil capability is required. The attack is fully constructible from the public RPC interface.

---

### Recommendation

Move the stat snapshot **after** all evictions have completed, or compute the final values incrementally. The simplest correct fix is to defer the stat update to after `check_and_record_ancestors` returns:

```rust
// After check_and_record_ancestors (which may have evicted txs and
// already decremented self.total_tx_size / self.total_tx_cycles):
self.total_tx_size = self.total_tx_size
    .checked_add(entry.size)
    .ok_or_else(|| Reject::Full(...))?;
self.total_tx_cycles = self.total_tx_cycles
    .checked_add(entry.cycles)
    .ok_or_else(|| Reject::Full(...))?;
```

This mirrors the pattern already used by `update_stat_for_remove_tx`, which mutates the live fields in-place rather than using a pre-computed snapshot.
