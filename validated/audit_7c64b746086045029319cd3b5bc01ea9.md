### Title
Unrestricted `initialize` Allows Any Signer to Hijack Config Authority — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary
The `initialize` instruction has no access control on who may call it. Any signer can invoke it before the legitimate deployer, supply an attacker-controlled `authority` pubkey, and permanently own the publisher registry.

### Finding Description

`validate_payer` only checks `is_signer` and `is_writable` — it does **not** verify the payer is the program upgrade authority or any other privileged key. [1](#0-0) 

`InitializeArgs.authority` is a free 32-byte field supplied entirely by the caller — no constraint ties it to the payer or any known deployer key. [2](#0-1) 

`initialize` calls `config::create` with whatever `args.authority` the caller provided, then exits successfully. [3](#0-2) 

`config::create` only rejects a call if `data.format != 0` (i.e., the account is already initialized). The first caller wins unconditionally. [4](#0-3) 

Once the config PDA exists, every subsequent `initialize` call fails because `system_instruction::create_account` will error on an already-existing account. The legitimate deployer's transaction is permanently blocked. [5](#0-4) 

### Impact Explanation

With attacker-controlled authority stored in the config PDA, `validate_authority` will accept only the attacker's signature for all `InitializePublisher` calls. [6](#0-5) 

The attacker can then:
- Register publishers with zero-length or attacker-owned buffers, freezing legitimate price submissions.
- Register arbitrary publishers pointing to attacker-controlled buffer accounts, enabling injection of fabricated prices into the oracle.

### Likelihood Explanation

This does **not** require traditional Ethereum-style mempool monitoring. The attack surface is simply the window between program deployment (public, observable on-chain) and the deployer's `initialize` transaction. An attacker who watches for new program deployments can race to call `initialize` immediately. On Solana, priority fees can further tilt ordering in the attacker's favor. The attack requires no privileged access — only a funded keypair.

### Recommendation

Bind `initialize` to a specific privileged signer. The canonical Solana pattern is to require the caller to be the **program upgrade authority** (stored in the program's `ProgramData` account). Concretely:

1. Add the `ProgramData` account to the `Initialize` account list.
2. In `validate_payer` (or a new `validate_deployer`), read the upgrade authority from `ProgramData` and assert `payer.key == upgrade_authority`.
3. Alternatively, hardcode the expected authority pubkey at compile time and assert it in `initialize`.

### Proof of Concept

Call sequence (no special privileges needed):

```
// Step 1 — attacker calls initialize before deployer
attacker::initialize(
    payer        = attacker_keypair,   // any funded signer
    config       = PDA("CONFIG"),
    system       = system_program,
    args.authority = attacker_pubkey,  // free field, no check
)
// → config PDA created, authority = attacker_pubkey

// Step 2 — attacker registers a publisher with a poisoned buffer
attacker::initialize_publisher(
    authority        = attacker_keypair,  // matches config.authority
    config           = PDA("CONFIG"),
    publisher_config = PDA("PUBLISHER_CONFIG", victim_publisher),
    buffer           = attacker_buffer,   // attacker-owned account
    system           = system_program,
)

// Step 3 — legitimate initialize now fails
deployer::initialize(...) // → system_instruction::create_account fails,
                          //   config PDA already exists
```

An integration test using `solana-program-test` can confirm this by submitting the attacker's `initialize` first, then asserting `config.authority == attacker_pubkey` and that the deployer's subsequent `initialize` returns an error.

```rust
pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)
```

```rust
pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {

    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
```
