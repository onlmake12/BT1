Audit Report

## Title
Silent u128→u64 Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Capacity — (File: `util/dao/src/lib.rs`)

## Summary
`DaoCalculator::calculate_maximum_withdraw` computes `withdraw_counted_capacity` as a `u128` and narrows it to `u64` via a silent `as u64` cast at line 156, silently truncating the upper 64 bits on overflow. Every other `u128`→`u64` narrowing in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. When overflow occurs, the function returns `Ok(truncated_capacity)` instead of `Err(DaoError::Overflow)`, causing valid DAO withdrawal transactions to be rejected and permanently freezing the depositor's funds.

## Finding Description
In `util/dao/src/lib.rs` lines 152–156:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently wraps to `withdraw_counted_capacity % 2^64` when the value exceeds `u64::MAX`. No error is returned; the function returns `Ok(truncated_capacity)`.

Three other narrowing conversions in the same file are safe:
- Line 204: `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?` [2](#0-1) 
- Line 245: `u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?` [3](#0-2) 
- Line 258: `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [4](#0-3) 

**Exploit path:**
1. A depositor creates a DAO cell with `counted_capacity` near `u64::MAX`.
2. The chain's `ar` ratio grows (any positive growth suffices given sufficient capacity).
3. The depositor submits a DAO withdrawal transaction. `transaction_maximum_withdraw` → `calculate_maximum_withdraw` returns a truncated (far too small) capacity.
4. `transaction_fee` computes `maximum_withdraw.safe_sub(outputs_capacity)` where `outputs_capacity` (the correct withdrawal amount) exceeds the truncated `maximum_withdraw`, causing `safe_sub` to return an error. [5](#0-4) 
5. The withdrawal transaction is permanently rejected. No alternative withdrawal path exists.

The RPC path at `rpc/src/module/experiment.rs` line 259 also calls `calculate_maximum_withdraw` directly and returns the silently wrong value to callers. [6](#0-5) 

## Impact Explanation
The concrete impact is permanent freezing of a DAO depositor's funds: a valid withdrawal transaction is rejected at the consensus layer because `transaction_fee` returns an error when `maximum_withdraw` is artificially deflated by truncation. This constitutes damage to the CKB economy — a depositor's funds become unrecoverable through the normal protocol path. The RPC additionally returns a silently incorrect withdrawal capacity to any caller, corrupting off-chain accounting. This matches the Critical allowed impact: **"Vulnerabilities which could easily damage CKB economy."**

## Likelihood Explanation
The overflow requires `counted_capacity * withdrawing_ar / deposit_ar > u64::MAX`. Since `ar` starts at `10^16` and grows monotonically, any positive `ar` growth combined with a `counted_capacity` sufficiently close to `u64::MAX` triggers the condition. The required cell capacity (~18.4 billion CKB in a single cell) is very large relative to typical usage, but the CKB protocol imposes no per-cell cap, and the threshold decreases as `ar` grows over the chain's lifetime. No privileged access is required — any DAO depositor who creates a sufficiently large cell is affected.

## Recommendation
Replace the silent cast with the checked conversion already used consistently elsewhere in the file:

```rust
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This matches the pattern at lines 204, 245, and 258 and surfaces overflow as `DaoError::Overflow` rather than silently producing a wrong capacity.

## Proof of Concept
**Parameters:**
- `deposit_ar` = 10,000,000,000,000,000 (initial ar, 10^16)
- `withdrawing_ar` = 10,000,000,001,000,000 (modest growth)
- `counted_capacity` = 18,446,744,073,000,000,000 (near `u64::MAX` = 18,446,744,073,709,551,615)

**Computation:**
```
withdraw_counted_capacity
  = 18_446_744_073_000_000_000 * 10_000_000_001_000_000
    / 10_000_000_000_000_000
  ≈ 18_446_744_074_844_674_407   // > u64::MAX
```

`as u64` wraps this to `18_446_744_074_844_674_407 % 2^64`, producing a value far below the correct withdrawal amount.

**Verification steps:**
1. Write a unit test in `util/dao/src/tests` constructing a mock header with the above `ar` values and a cell output with `counted_capacity` as above.
2. Call `calculate_maximum_withdraw` and assert the returned capacity equals the correct value.
3. Observe the test fails (returns a truncated value) with the current `as u64` cast.
4. Replace with `u64::try_from(...).map_err(|_| DaoError::Overflow)?` and observe the function returns `Err(DaoError::Overflow)` instead of a silently wrong `Ok(...)`.
