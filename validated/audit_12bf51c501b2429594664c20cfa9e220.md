### Title
Silent `u128 → u64` Truncation in `calculate_maximum_withdraw` Returns Wrong Withdrawal Capacity — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes an intermediate `u128` result and then casts it to `u64` with a bare `as u64`, silently discarding the high bits if the value overflows `u64::MAX`. The same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?` for every other analogous `u128 → u64` narrowing, making this a clear inconsistency. The function is reachable both from the `calculate_dao_maximum_withdraw` RPC and from the consensus-critical `transaction_fee` path used during block verification.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The multiplication is widened to `u128` precisely to avoid overflow during the intermediate step. However, the final narrowing back to `u64` uses `as u64`, which in Rust silently wraps/truncates — it never returns an error. If `withdraw_counted_capacity` exceeds `u64::MAX`, the returned `Capacity` is a drastically wrong (truncated) value, not an error.

The same file performs two identical `u128 → u64` narrowings in `dao_field_with_current_epoch` using the safe pattern:

```rust
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The inconsistency is direct: the same developer-authored file applies overflow-safe conversion everywhere except in `calculate_maximum_withdraw`.

---

### Impact Explanation

`calculate_maximum_withdraw` is called from two reachable paths:

**1. RPC path** — `ExperimentRpc::calculate_dao_maximum_withdraw` calls `calculator.calculate_maximum_withdraw(...)` directly and returns the result to the caller. [4](#0-3) 

An RPC caller querying the maximum DAO withdrawal amount would receive a silently wrong (truncated) capacity value, with no error signal.

**2. Consensus/verification path** — `transaction_maximum_withdraw` (the private method) calls `calculate_maximum_withdraw` at line 108, and `transaction_fee` calls `transaction_maximum_withdraw`. `DaoCalculator` and `transaction_fee` are imported and used in `verification/src/transaction_verifier.rs`. [5](#0-4) [6](#0-5) 

If the truncation occurs in the verification path, the computed fee for a DAO withdrawal transaction would be silently wrong, potentially causing a valid transaction to be rejected (computed fee appears negative) or an invalid one to be accepted (computed fee appears positive due to wrap-around).

---

### Likelihood Explanation

`withdraw_counted_capacity = counted_capacity * withdrawing_ar / deposit_ar`

- `counted_capacity` is bounded by total CKB supply (~3.36 × 10¹⁸ shannons, well below `u64::MAX` ≈ 1.84 × 10¹⁹).
- `withdrawing_ar / deposit_ar` is the DAO interest multiplier; it starts at 1.0 and grows slowly (secondary issuance is ~1.344 billion CKB/year against a ~33.6 billion CKB base, ~4% annually).
- For overflow: `counted_capacity * (withdrawing_ar / deposit_ar) > u64::MAX` requires the multiplier to exceed ~5.5×, which at ~4%/year would take well over a century.

**Likelihood on mainnet today: very low.** However:
- The code is structurally incorrect and inconsistent with the safe pattern used in the same file.
- The bug is latent and would silently activate without any error if the condition is ever met.
- On a testnet or custom chain with aggressive secondary issuance parameters, the condition could be reached much sooner.

---

### Recommendation

Replace the bare `as u64` cast with the same checked conversion used elsewhere in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [7](#0-6) 

---

### Proof of Concept

The inconsistency is directly visible within the same source file. `dao_field_with_current_epoch` uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?` for both `miner_issuance128` (line 245) and `ar_increase128` (line 258), while `calculate_maximum_withdraw` uses `withdraw_counted_capacity as u64` (line 156) for a structurally identical narrowing. The `as u64` cast in Rust is defined to truncate silently — no panic, no error — making this a latent silent-corruption bug rather than a detectable overflow. [1](#0-0) [8](#0-7) [9](#0-8)
