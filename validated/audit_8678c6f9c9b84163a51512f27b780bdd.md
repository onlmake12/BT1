### Title
DAO Withdrawal `header_deps` Index Truncation Discrepancy Enables Consensus Split - (`util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator` reads the `header_deps` index from the DAO withdrawal witness as a full `u64`, while the on-chain DAO type script (C VM) reads only the **lowest byte** of that same `u64`. A crafted transaction can exploit this discrepancy to produce a block that the C VM accepts but the Rust `DaoHeaderVerifier` rejects, causing a consensus split.

---

### Finding Description

In `transaction_maximum_withdraw`, the witness `input_type` field is decoded as a `u64` and used directly as a `usize` index into `header_deps`: [1](#0-0) 

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // ← full u64 used as index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The on-chain DAO type script (C VM) reads the same field but treats it as a 1-byte value, using only the **lowest byte** as the index. This is explicitly documented in the test added to capture the discrepancy: [2](#0-1) 

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

An attacker crafts a DAO withdrawal transaction with ≥258 `header_deps`:
- `header_deps[1]` = correct deposit block hash (C VM resolves here via `257 & 0xFF = 1`)
- `header_deps[257]` = withdraw block hash (Rust resolves here via full `257`)
- Witness `input_type` = `257u64` (little-endian 8 bytes)
- Cell data = deposit block number (e.g., `100`) [3](#0-2) 

**C VM path**: uses index 1 → deposit block (number 100) → matches cell data → accepts the transaction.

**Rust path**: uses index 257 → withdraw block (number 200) → `deposit_header.number() (200) != deposited_block_number (100)` → returns `DaoError::InvalidOutPoint`. [4](#0-3) 

The `DaoHeaderVerifier` in the contextual block verifier calls `DaoCalculator::dao_field()` and compares the result to the block header's DAO field. If `DaoCalculator` returns an error, the block is rejected with `BlockErrorKind::InvalidDAO`: [5](#0-4) 

This verifier is invoked unconditionally (unless `disable_daoheader` switch is set) during contextual block verification: [6](#0-5) 

---

### Impact Explanation

A miner who includes the crafted transaction produces a block that:
- The C VM (DAO type script) considers **valid** — the withdrawal is correctly authorized
- The Rust `DaoHeaderVerifier` considers **invalid** — `DaoCalculator` fails with `InvalidOutPoint`, causing `BlockErrorKind::InvalidDAO`

This is a **consensus split**: any alternative CKB implementation that faithfully follows the C VM's 1-byte index behavior would accept the block, while all Rust nodes reject it. Even within the Rust node, the C VM script execution succeeds but the DAO field verification fails — an internal inconsistency. A miner who includes such a transaction loses their block reward (DoS on miners). At scale, an attacker can repeatedly submit such transactions to grief miners or partition the network.

---

### Likelihood Explanation

The attack requires:
1. Crafting a DAO withdrawal transaction with ≥258 `header_deps` — straightforward, no privileged access needed
2. A miner to include the transaction — miners include valid-looking transactions from the mempool; the transaction passes script execution (C VM accepts it) and would not be filtered by the tx-pool's script verifier

The tx-pool does not run `DaoCalculator` on incoming transactions; it only runs script verification. The DAO field mismatch is only detected at block assembly/verification time. A miner running `calc_dao` during block assembly would encounter the error and drop the transaction: [7](#0-6) 

However, a malicious or buggy miner could still include it, causing their block to be rejected by all peers.

---

### Recommendation

Truncate `header_dep_index` to its lowest byte before using it as an index, to match the C VM DAO type script's behavior:

```rust
// In transaction_maximum_withdraw, after reading the u64:
let header_dep_index = header_dep_index & 0xFF;  // match C VM's 1-byte truncation
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
```

Alternatively, add an explicit validation that rejects any witness `input_type` value ≥ 256 with `DaoError::InvalidDaoFormat`, preventing the discrepancy from being reachable. [1](#0-0) 

---

### Proof of Concept

The existing test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` directly demonstrates the discrepancy: [8](#0-7) 

- `header_deps[1]` = deposit block (number 100) — what C VM resolves
- `header_deps[257]` = withdraw block (number 200) — what Rust resolves
- Witness index = 257 (`0x0000000000000101`)
- Cell data = `100u64` (deposit block number)

The test asserts `result.is_err()` — confirming that the Rust `DaoCalculator` rejects a transaction the C VM would accept. A miner including this transaction would produce a block rejected by all Rust nodes via `DaoHeaderVerifier` → `BlockErrorKind::InvalidDAO`.

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}

    pub fn verify(&self) -> Result<(), Error> {
        let dao = DaoCalculator::new(
            &self.context.consensus,
            &self.context.store.borrow_as_data_loader(),
        )
        .dao_field(self.resolved.iter().map(AsRef::as_ref), self.parent)
        .map_err(|e| {
            error_target!(
                crate::LOG_TARGET,
                "Error generating dao data for block {}: {:?}",
                self.header.hash(),
                e
            );
            e
        })?;

        if dao != self.header.dao() {
            return Err((BlockErrorKind::InvalidDAO).into());
        }
        Ok(())
    }
