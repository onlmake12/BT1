### Title
Unbounded `while` Loop in `executeCallback` Enables Gas DoS and Permanent Fund Locking - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that performs a cold storage lookup (`SLOAD`) for every sequence number between `_state.firstUnfulfilledSeq` and `_state.currentSequenceNumber`. An unprivileged attacker can flood the contract with pending requests to make this loop exceed the block gas limit, permanently preventing any provider from fulfilling legitimate requests and locking user funds in the contract.

---

### Finding Description

In `Echo.executeCallback`, after clearing a fulfilled request, the contract attempts to advance `_state.firstUnfulfilledSeq` by scanning forward through all sequence numbers:

```solidity
// Echo.sol lines 169–174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

Each iteration calls `findRequest`, which performs a storage lookup (SLOAD, 2100 gas cold). The loop has **no upper bound** on iterations — it will scan through every sequence number between `firstUnfulfilledSeq` and `currentSequenceNumber`. The code itself acknowledges this danger with an inline TODO:

> *"TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."*

The attack path is:
1. Attacker calls `requestPriceUpdatesWithCallback` N times, creating pending requests with sequence numbers 1 through N.
2. A provider (or the attacker) calls `executeCallback` for any request whose sequence number is greater than `firstUnfulfilledSeq` (e.g., sequence number N).
3. The `while` loop scans from `firstUnfulfilledSeq` (= 1) all the way to N, performing N−1 cold SLOADs.
4. With N large enough (~14,300 iterations at 2100 gas each to exhaust a 30M gas block), the transaction reverts.
5. `clearRequest` is called before the `while` loop, but the revert undoes it — the request is not cleared, the provider is not paid, and the requester's funds remain locked.

The `requestPriceUpdatesWithCallback` function is callable by any unprivileged user who pays the required fee. There is no cap on the total number of outstanding requests — `currentSequenceNumber` is a `uint64` and grows without bound. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

The contract's own comment at lines 155–156 explicitly states the consequence:

> *"If executeCallback can revert, then funds can be permanently locked in the contract."*

Once the `while` loop causes a revert, the affected request can never be fulfilled: the sequence number is still active, but every subsequent call to `executeCallback` for any request with a higher sequence number will also revert (the loop still starts from the same `firstUnfulfilledSeq`). This creates a permanent DoS on all future fulfillments and locks all requester fees in the contract with no recovery path. [4](#0-3) 

---

### Likelihood Explanation

`requestPriceUpdatesWithCallback` is callable by any address that pays the required fee. The fee is composed of a Pyth base fee plus a provider-set fee. If the provider's `baseFeeInWei`, `feePerFeedInWei`, and `feePerGasInWei` are set to low values (which is realistic for a competitive provider), the cost per request is minimal. At ~2100 gas per cold SLOAD and a 30M gas block limit, only ~14,300 unfulfilled requests are needed to make the loop exceed the block gas limit. An attacker willing to pay the fee for 14,300 requests can permanently brick the contract for all users. [5](#0-4) [6](#0-5) 

---

### Recommendation

1. **Cap the while loop iterations**: Add a maximum iteration count (e.g., `NUM_REQUESTS = 32`) to bound the gas cost of the scan.
2. **Use a doubly-linked list**: As the TODO comment itself suggests, replace the linear scan with a linked list so `firstUnfulfilledSeq` can be updated in O(1).
3. **Decouple the scan from `executeCallback`**: Move the `firstUnfulfilledSeq` advancement to a separate, gas-bounded maintenance function that is not on the critical path of request fulfillment. [7](#0-6) 

---

### Proof of Concept

```
1. Attacker calls requestPriceUpdatesWithCallback() N times
   (N ≈ 14,300 to exceed 30M gas block limit at 2100 gas/SLOAD)
   → _state.currentSequenceNumber = N+1
   → _state.firstUnfulfilledSeq = 1 (none fulfilled yet)

2. Provider calls executeCallback(providerToCredit, N, updateData, priceIds)
   → clearRequest(N) executes (line 164)
   → while loop begins at firstUnfulfilledSeq = 1
   → loop calls findRequest(1), findRequest(2), ..., findRequest(N-1)
   → each findRequest() does 1–2 SLOADs (2100 gas cold)
   → after ~14,300 iterations: gas exhausted → REVERT

3. clearRequest(N) is rolled back by the revert
   → request N is still active
   → provider receives no payment
   → requester's fee is permanently locked in the contract

4. All future executeCallback() calls for seq > 1 will also revert
   → entire Echo contract is bricked for all users
``` [8](#0-7) [9](#0-8)

        bytes[] calldata updateData,
        bytes32[] calldata priceIds

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
