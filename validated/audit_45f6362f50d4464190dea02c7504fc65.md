### Title
No Maximum Fee Bound in Entropy Request Functions Causes Unrefunded Overpayment — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `requestHelper` function in `Entropy.sol` accepts any `msg.value >= requiredFee` and permanently credits all excess beyond `providerFee` to `_state.accruedPythFeesInWei` (Pyth's treasury) with no refund. Because neither `request`, `requestWithCallback`, nor `requestV2` accept a `maxFee` parameter, users have no on-chain protection against overpaying when the fee changes between off-chain estimation and on-chain execution.

### Finding Description

In `requestHelper`, the fee accounting is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // line 234
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee(); // line 235
uint128 providerFee = getProviderFee(provider, callbackGasLimit);    // line 236
providerInfo.accruedFeesInWei += providerFee;                        // line 237
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value)        // line 238
    - providerFee);                                                   // line 239
``` [1](#0-0) 

Every wei of `msg.value` above `providerFee` is swept into `accruedPythFeesInWei` — not refunded. The public interfaces `request`, `requestWithCallback`, and `requestV2` expose no `maxFee` guard: [2](#0-1) [3](#0-2) [4](#0-3) 

The fee has two independently mutable components. The provider can call `setProviderFee` at any time: [5](#0-4) 

Governance can call `setPythFee` at any time: [6](#0-5) 

The Fortuna keeper service also actively adjusts provider fees in response to gas price changes: [7](#0-6) 

The `getFeeV2` view function combines both components: [8](#0-7) 

### Impact Explanation

A user or integrating contract that:
1. Reads `getFee(provider)` = 100 wei off-chain,
2. Sends 100 wei as `msg.value`,
3. Has the transaction delayed in the mempool while the provider lowers `feeInWei` (e.g., Fortuna reduces it as gas prices drop),

will have `requiredFee` drop to, say, 60 wei. The transaction succeeds, but `accruedPythFeesInWei` absorbs `100 − providerFee` instead of the expected `pythFeeInWei`. The user permanently loses the 40 wei difference to Pyth's treasury with no recourse.

Conversely, a user who defensively sends `2 × getFee(provider)` to avoid a revert from a fee increase loses the entire excess. Integrating contracts that cache the fee value are especially exposed because the cached value can become stale.

### Likelihood Explanation

The Fortuna keeper (`adjust_fee_if_necessary`) runs continuously and updates `feeInWei` whenever gas prices move outside a tolerance band. Fee changes are therefore routine, not exceptional. Any user whose transaction sits in the mempool during a fee adjustment is affected. The attack path requires no privileged access — any unprivileged user calling `request`, `requestWithCallback`, or `requestV2` is exposed. [9](#0-8) 

### Recommendation

Add a `maxFee` parameter to `request`, `requestWithCallback`, and `requestV2`, and revert if `requiredFee > maxFee`:

```solidity
function requestV2(
    address provider,
    bytes32 userContribution,
    uint32 gasLimit,
+   uint128 maxFee          // caller-specified upper bound
) public payable returns (uint64) {
    uint128 requiredFee = getFeeV2(provider, gasLimit);
+   if (requiredFee > maxFee) revert EntropyErrors.FeeTooHigh();
    ...
}
```

Alternatively, refund `msg.value − requiredFee` to `msg.sender` after the fee check, eliminating the need for a `maxFee` parameter entirely.

### Proof of Concept

1. Fortuna keeper observes falling gas prices and calls `setProviderFee(provider, newLowerFee)`.
2. Simultaneously, a user's `requestWithCallback{value: oldFee}(provider, userRandomNumber)` transaction is pending in the mempool.
3. The keeper's fee-update transaction is mined first (higher priority fee).
4. The user's transaction executes: `requiredFee = newLowerFee + pythFeeInWei < oldFee`.
5. `msg.value >= requiredFee` → transaction succeeds.
6. `accruedPythFeesInWei += (oldFee − providerFee)` — the full excess is swept to Pyth's treasury.
7. User receives their sequence number but permanently loses `oldFee − requiredFee` wei with no refund path. [1](#0-0) [10](#0-9)
