### Title
Permanently Unupdatable `single_update_fee` Locks Lazer Aptos Protocol Revenue at Nominal Zero — (`File: lazer/contracts/aptos/sources/pyth_lazer.move`)

---

### Summary

The Pyth Lazer Aptos contract initializes `single_update_fee` to a hardcoded nominal value of `1` (1 octa ≈ $0.000000001) and provides **no admin function to update it**. The admin role can only manage trusted signers. As a result, the protocol is permanently unable to collect meaningful fees from any Lazer message verification on Aptos.

---

### Finding Description

In `lazer/contracts/aptos/sources/pyth_lazer.move`, the `Storage` struct holds `single_update_fee`: [1](#0-0) 

During `initialize`, it is hardcoded to `1`: [2](#0-1) 

The comment `// Nominal fee` confirms this is a placeholder value. However, the entire contract exposes **no entry function to update `single_update_fee`**. The only admin-gated mutation function is `update_trusted_signer`: [3](#0-2) 

There is a `get_update_fee()` getter: [4](#0-3) 

But no corresponding setter exists anywhere in the module. The fee is enforced at call time: [5](#0-4) 

So the check passes — but only for 1 octa, permanently.

---

### Impact Explanation

Every call to `verify_message_with_funder` or `verify_message` on Aptos pays exactly 1 octa (10⁻⁸ APT) to the treasury, regardless of market conditions or intended fee policy. The Pyth Lazer Aptos contract can **never** collect meaningful protocol revenue. All Lazer price-feed verification on Aptos is effectively free for users. This is a permanent, irrecoverable state without a contract upgrade.

---

### Likelihood Explanation

Certainty is 100% — the fee is hardcoded at deployment and no code path exists to change it. Every single invocation of the verification entry point is affected. The condition requires no special attacker setup; any ordinary Lazer updater or integrator triggers it on every call.

---

### Recommendation

Add an admin-gated entry function to update `single_update_fee`, analogous to how `update_trusted_signer` gates signer management behind `AdminCapability`:

```move
public entry fun set_update_fee(
    account: &signer,
    new_fee: u64
) acquires Storage {
    assert!(
        exists<AdminCapability>(signer::address_of(account)),
        ENO_PERMISSIONS
    );
    let storage = borrow_global_mut<Storage>(@pyth_lazer);
    storage.single_update_fee = new_fee;
}
```

Also consider setting a non-trivial initial fee at deployment rather than the nominal `1`.

---

### Proof of Concept

1. Deploy the contract via `initialize(account, admin, treasury)`.
2. `get_update_fee()` returns `1`.
3. Call `verify_message_with_funder(account, message, signature, trusted_signer)` with any valid signed message — only 1 octa is deducted.
4. Attempt to call any function as admin to raise the fee — no such function exists; the transaction aborts with `ENO_PERMISSIONS` or simply does not exist in the ABI.
5. `get_update_fee()` still returns `1` indefinitely.

The treasury receives 1 octa per verification call for the lifetime of the contract, with no mechanism for correction short of a full contract upgrade.
