### Title
Overly Strict Access Control on `withdrawFees` Locks Accrued Pyth Protocol Fees Permanently - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
`Echo.withdrawFees()` is gated exclusively to `_state.admin` with no fallback, no admin-transfer mechanism, and no owner override. If the admin key is lost or compromised, all accrued Pyth protocol fees (`_state.accruedFeesInWei`) are permanently locked in the contract.

### Finding Description
In `Echo.sol`, the `withdrawFees` function enforces a single-address access control check:

```solidity
function withdrawFees(uint128 amount) external override {
    require(msg.sender == _state.admin, "Only admin can withdraw fees");
    require(_state.accruedFeesInWei >= amount, "Insufficient balance");

    _state.accruedFeesInWei -= amount;

    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "Failed to send fees");

    emit FeesWithdrawn(msg.sender, amount);
}
``` [1](#0-0) 

Every time a user calls `requestPriceUpdatesWithCallback`, the Pyth protocol fee (`_state.pythFeeInWei`) is added to `_state.accruedFeesInWei`: [2](#0-1) 

This balance can only be recovered by calling `withdrawFees`, which is restricted exclusively to `_state.admin`. Unlike the Entropy contract — which uses `_authoriseAdminAction()` allowing **both** admin and owner to withdraw fees — Echo has no such dual-role fallback: [3](#0-2) 

Furthermore, `EchoState.State` contains no `proposedAdmin` field and there is no `proposeAdmin`/`acceptAdmin` mechanism in Echo at all (unlike `EntropyGovernance`): [4](#0-3) 

While `EchoUpgradeable` inherits `Ownable2StepUpgradeable`, the owner role is only authorized to upgrade the implementation — it cannot call `withdrawFees`: [5](#0-4) 

### Impact Explanation
If the admin key is lost, rotated to an inaccessible address, or the admin wallet is compromised and drained of ETH (preventing gas for transactions), all accumulated Pyth protocol fees in `_state.accruedFeesInWei` are permanently locked. There is no recovery path: no admin transfer function, no owner override for fee withdrawal, and no permissionless trigger. The owner can upgrade the contract, but that is a complex and non-standard recovery path that may not be available in all scenarios.

### Likelihood Explanation
The Echo contract is a production contract with a single admin address as a single point of failure. Admin key loss or rotation errors are realistic operational risks. The Pyth protocol fees accumulate continuously with every `requestPriceUpdatesWithCallback` call, so the locked amount grows over time. The likelihood is medium: it requires an operational failure of the admin key, which is a known class of real-world incidents.

### Recommendation
Mirror the pattern used in `EntropyGovernance.withdrawFee()`: allow both the admin and the contract owner to call `withdrawFees`. Additionally, add a `proposeAdmin`/`acceptAdmin` two-step admin transfer mechanism (as exists in `EntropyGovernance`) so the admin role can be safely rotated. Alternatively, allow anyone to call `withdrawFees` with the funds sent to a fixed, immutable treasury address, eliminating the access control entirely.

### Proof of Concept
1. Deploy `EchoUpgradeable` with `admin = 0xADMIN` and `owner = 0xOWNER`.
2. Users call `requestPriceUpdatesWithCallback{value: fee}(...)` repeatedly; `_state.accruedFeesInWei` grows.
3. `0xADMIN` key is lost (e.g., hardware wallet destroyed, key rotation error).
4. `0xOWNER` attempts `echo.withdrawFees(amount)` → reverts with `"Only admin can withdraw fees"`.
5. Any other address attempts `echo.withdrawFees(amount)` → same revert.
6. `_state.accruedFeesInWei` is permanently inaccessible. The owner can only recover by upgrading the implementation to a version that bypasses the check — a non-trivial governance action with its own risks.
