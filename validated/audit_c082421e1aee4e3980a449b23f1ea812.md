Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Write-Back in `add_entry` Overwrites In-Place Decrements from Ancestor-Eviction — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`PoolMap::add_entry` snapshots `total_tx_size`/`total_tx_cycles` via `updated_stat_for_add_tx` before calling `check_and_record_ancestors`, which may evict pool entries and correctly decrement those fields in-place. The stale pre-eviction snapshot is then unconditionally written back at lines 218–219, permanently inflating both counters by the aggregate size/cycles of every entry evicted during ancestor-limit enforcement. Because `limit_size` drives its eviction loop off `total_tx_size`, the phantom inflation causes it to expel additional legitimate pending/proposed transactions equal in aggregate size to the inflation.

## Finding Description
In `add_entry` (lines 200–221 of `tx-pool/src/component/pool_map.rs`): [1](#0-0) 

`updated_stat_for_add_tx` (lines 711–729) takes `&self` and returns `(self.total_tx_size + entry.size, self.total_tx_cycles + entry.cycles)` without mutating any field — it is a pure read that produces a snapshot. [2](#0-1) 

`check_and_record_ancestors` (lines 588–640) enters the eviction branch when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618) for each evicted candidate. [3](#0-2) 

That call chain reaches `update_stat_for_remove_tx` (lines 733–758), which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` in-place: [4](#0-3) 

After `check_and_record_ancestors` returns, the write-back at lines 218–219 overwrites those decrements with the stale snapshot, leaving `total_tx_size` inflated by exactly `Σ size(evicted_i)` and `total_tx_cycles` inflated by `Σ cycles(evicted_i)`.

`limit_size` (lines 292–329 of `tx-pool/src/pool.rs`) loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts lowest-fee-rate entries until the condition is false. With an inflated counter, it evicts an additional cohort of legitimate transactions whose aggregate size equals the inflation, even though the pool is actually within its real limit. [5](#0-4) 

## Impact Explanation
An unprivileged submitter can repeatedly trigger the eviction branch, compounding the inflation with each submission. Over multiple iterations the effective pool capacity shrinks toward zero: legitimate pending/proposed transactions are silently expelled without any protocol error, preventing them from being mined. This constitutes a low-cost, repeatable denial-of-service against the CKB tx-pool that degrades miner revenue and blocks user transactions from reaching the chain — matching the **High** impact class: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation
The trigger requires no privileged access, no key material, and no majority hash-power. Any submitter can craft a transaction chain of length `max_ancestors_count - 1` where each transaction shares a cell dependency (`cell_dep_A`) with existing pool entries, then submit a new transaction that spends the chain tip and also references `cell_dep_A`. This makes `cell_ref_parents` non-empty and `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, entering the eviction branch. The attack is repeatable: each iteration inflates `total_tx_size` further, and the cost per iteration is only the fee required to get the crafted transaction accepted.

## Recommendation
Remove the pre-computed snapshot and instead apply the new entry's contribution directly to the already-correct post-eviction `self.total_tx_size`/`self.total_tx_cycles`:

```rust
// Validate overflow only — do not snapshot
self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Evictions happen here, correctly decrementing self.total_tx_size/cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Apply new entry's contribution to the post-eviction totals
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

## Proof of Concept
1. Configure a node with `max_ancestors_count = N` and a pool near `max_tx_pool_size`.
2. Submit a chain `tx0 → tx1 → … → tx_{N-1}`, each using `cell_dep_A`. Pool now has `N` entries.
3. Submit `tx_N` spending `tx_{N-1}` and also referencing `cell_dep_A`. Now `ancestors_count = N+1 > max_ancestors_count`, but `cell_ref_parents = {tx0,…,tx_{N-1}}`, so `ancestors_count - cell_ref_parents.len() = 1 <= max_ancestors_count` — eviction branch is taken.
4. `check_and_record_ancestors` evicts `tx_{N-1}` (lowest fee). `update_stat_for_remove_tx` decrements `self.total_tx_size` by `size(tx_{N-1})`.
5. `add_entry` writes back `total_tx_size = old_total + size(tx_N)`, ignoring the decrement. Actual pool size is `old_total + size(tx_N) - size(tx_{N-1})`; reported size is `old_total + size(tx_N)` — inflated by `size(tx_{N-1})`.
6. `limit_size` sees the inflated counter exceed `max_tx_pool_size` and evicts an additional legitimate transaction of size ≈ `size(tx_{N-1})`.
7. Repeat from step 3 with a fresh `tx_N`; each iteration compounds the inflation and expels more legitimate transactions.

A unit test can assert `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.inner.size).sum()` after each `add_entry` call that triggers the eviction branch, which will fail on the current code and pass after the fix.

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
