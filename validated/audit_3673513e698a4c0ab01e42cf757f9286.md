### Title
Unbounded `while` Loop Over Growing Sequence Counter in `Echo.executeCallback` Enables Permanent DoS of Request Fulfillment — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that linearly scans every sequence number from `_state.firstUnfulfilledSeq` to `_state.currentSequenceNumber` doing per-slot storage reads. Because `currentSequenceNumber` grows without bound as users create requests, an attacker can inflate the gap between `firstUnfulfilledSeq` and the target sequence number until the loop's gas cost exceeds the block gas limit, permanently preventing fulfillment of a specific request.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` advances `_state.firstUnfulfilledSeq` with the following loop:

```solidity
// Echo.sol lines 169-174
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs two storage reads: one into the fixed-size `_state.requests[shortKey]` array and, when the slot does not match, a second cold SLOAD into `_state.requestsOverflow[key]` (a unique key per sequence number):

```solidity
// Echo.sol lines 310-321
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

`_state.currentSequenceNumber` is incremented unconditionally for every call to `requestPriceUpdatesWithCallback`:

```solidity
// Echo.sol line 73
requestSequenceNumber = _state.currentSequenceNumber++;
``` [3](#0-2) 

The developers themselves flagged this in a TODO comment immediately above the loop:

> *"I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. A better solution would be a doubly-linked list of active requests."* [4](#0-3) 

Because the `while` loop executes **before** the `try/catch` callback and **after** `clearRequest`, a revert inside the loop (out-of-gas) rolls back `clearRequest` as well, leaving the request permanently active but unfulfillable.

---

### Impact Explanation

If the loop's cumulative gas cost exceeds the block gas limit, `executeCallback` can never succeed for the targeted sequence number. The request remains active forever, the user's callback is never executed, and the fee paid by the user is locked in the contract. This is a permanent, irreversible DoS of a specific request.

---

### Likelihood Explanation

Any unprivileged user can call `requestPriceUpdatesWithCallback` to inflate `currentSequenceNumber`. After the exclusivity period, anyone can call `executeCallback` to fulfill the intermediate requests. The attacker must pay provider fees for each intermediate request, making the attack economically bounded but not infeasible.

**Gas estimate per loop iteration** (fulfilled requests in overflow path):
- 1 warm SLOAD (`firstUnfulfilledSeq`): ~100 gas
- 1 warm SLOAD (`requests[shortKey]`): ~100 gas (only 32 slots, all warm after first pass)
- 1 cold SLOAD (`requestsOverflow[key]`): ~2,100 gas (unique key per sequence)
- 1 warm SSTORE (`firstUnfulfilledSeq++`): ~100 gas

≈ **2,400 gas/iteration**. At a 30 M gas block limit: **~12,500 intermediate requests** are sufficient to permanently block a target request. At low provider fee settings this is economically achievable.

---

### Recommendation

Replace the linear scan with a data structure that allows O(1) advancement of `firstUnfulfilledSeq`, such as a doubly-linked list of active requests (as the TODO comment already suggests), or a bitmap of active sequence numbers. Alternatively, cap the number of iterations per call and allow the loop to resume across multiple transactions.

---

### Proof of Concept

1. Victim submits `requestPriceUpdatesWithCallback(...)` → assigned sequence number **S** (e.g., S = 1).
2. Attacker calls `requestPriceUpdatesWithCallback(...)` **N** times → assigned sequence numbers **S+1, S+2, …, S+N**.
3. Attacker (or anyone after exclusivity) calls `executeCallback` for each of **S+1 … S+N**. For each of these calls, `firstUnfulfilledSeq` is still **S** (request S is still active), so the `while` loop runs **0 iterations** — these calls are cheap.
4. Provider attempts `executeCallback` for sequence **S**:
   - `clearRequest(S)` executes.
   - The `while` loop now starts at `firstUnfulfilledSeq = S`. Request S is cleared, so `isActive` returns false. The loop advances through S+1, S+2, …, S+N (all cleared), performing **N cold SLOADs** into `requestsOverflow`.
   - At N ≈ 12,500 the loop consumes > 30 M gas → transaction reverts.
   - `clearRequest(S)` is rolled back; request S remains active.
5. Every subsequent attempt to fulfill request S repeats step 4 and always reverts. Request S is permanently unfulfillable; the victim's callback is never executed and their fee is locked.
