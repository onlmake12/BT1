Audit Report

## Title
Duplicate-Index Amplification in `GetBlockTransactionsProcess::execute` Enables Remote OOM Node Crash — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
An unprivileged remote P2P peer can send a `GetBlockTransactions` message containing up to 32,767 duplicate transaction indexes. The handler performs no deduplication, clones the same `TransactionView` up to 32,767 times, and passes them into the molecule builder, which calls `Vec::with_capacity(expected_length())` with a `usize` that can reach ~19.56 GB, triggering an OOM abort that crashes the node process.

## Finding Description

`MAX_RELAY_TXS_NUM_PER_BATCH` is 32,767: [1](#0-0) 

The guard in `execute` checks only the count against this limit with no deduplication: [2](#0-1) 

The transaction lookup loop uses `filter_map` with no deduplication — if the same index appears 32,767 times, the same `TransactionView` is cloned 32,767 times: [3](#0-2) 

These clones are passed directly into `packed::BlockTransactions::new_builder().transactions(...).build()`: [4](#0-3) 

Inside `TransactionVecBuilder`, `expected_length()` sums the serialized sizes of all items as a `usize` with no cap: [5](#0-4) 

`build()` then calls `Vec::with_capacity(self.expected_length())` with that full `usize` value: [6](#0-5) 

`MAX_BLOCK_BYTES = TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT = 597 × 1,000 = 597,000 bytes`: [7](#0-6) 

`SizeVerifier` allows a single transaction up to `block_bytes_limit` (i.e., 597,000 bytes): [8](#0-7) 

**Math**: 32,767 × 597,000 ≈ **19.56 GB**. `Vec::with_capacity(19_561_899_000)` calls the global allocator for ~19.56 GB. When the allocation fails, Rust's default allocator calls `handle_alloc_error`, which **aborts the process**.

A secondary issue: `write()` casts `total_size as molecule::Number` (u32), which silently truncates if `total_size > u32::MAX`, producing a malformed length field — but the OOM abort occurs first: [9](#0-8) 

**Network-layer frame limit does not mitigate this**: The RelayV3 protocol has a `max_frame_length` of 4 MB, which limits the *incoming* `GetBlockTransactions` message. The crafted request (32,767 × 4-byte indexes ≈ 131 KB) is well within this limit. The OOM occurs on the *outgoing* response path, which is not subject to the incoming frame size check: [10](#0-9) 

## Impact Explanation
The node process aborts due to OOM, taking down the RPC API and all sync/relay functionality. This matches the allowed CKB bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**. Any connected peer can trigger this repeatedly, preventing the node from staying online.

## Likelihood Explanation
The attacker only needs to:
1. Connect as a P2P peer (no privilege required — the relay protocol is open to all peers).
2. Identify a stored block containing a transaction of meaningful size. Even a 10 KB transaction yields ~327 MB allocation; a 100 KB transaction yields ~3.27 GB (likely OOM on most nodes). Transactions up to 597 KB are consensus-valid.
3. Send a single crafted `GetBlockTransactions` message with 32,767 copies of the same index.

No PoW, no key, no special role is required. The attack is repeatable after each node restart.

## Recommendation
- **Deduplicate indexes** before the lookup loop (e.g., sort and dedup the index list, or collect into a `HashSet<u32>`).
- After deduplication, re-check that the resulting count is within a safe bound.
- Alternatively, cap the total serialized size of the response before building the molecule buffer, rejecting or truncating if it would exceed a safe threshold (e.g., `MAX_BLOCK_BYTES`).

## Proof of Concept
1. Mine or locate a stored block containing a transaction of size ≥ ~10 KB (larger = more reliable OOM; up to 597 KB is consensus-valid).
2. Connect to the target node as a P2P peer on the RelayV3 protocol (`/ckb/relay3`).
3. Send a `GetBlockTransactions` message with `block_hash` set to that block's hash and `indexes` set to `[0u32; 32767]` (32,767 copies of index 0).
4. Observe the node process abort with an OOM error; the RPC API becomes unreachable.
5. For a unit test: construct a mock block with a large transaction, call `GetBlockTransactionsProcess::execute` with a message containing 32,767 duplicate indexes, and assert that `expected_length()` returns a value exceeding available memory or that the process aborts.
