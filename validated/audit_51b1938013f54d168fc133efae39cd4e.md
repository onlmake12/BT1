### Title
Collected Update Fees Are Permanently Locked in Shared `PriceInfoObject` with No Withdrawal Path — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `update_single_price_feed` deposits the caller's fee `Coin<SUI>` into the target `PriceInfoObject` via `deposit_fee_coins`. No function in the entire module ever removes or transfers those coins out. The `fee_recipient_address` field in `State` is explicitly documented as unused. All collected protocol fees are permanently frozen.

### Finding Description
`price_info::deposit_fee_coins` (public, `price_info.move:105`) stores fee coins as a `dynamic_object_field` under `FEE_STORAGE_KEY = b"fee_storage"` on each `PriceInfoObject`. The only operations on this field are `dynamic_object_field::add` (first deposit) and `coin::join` (subsequent deposits). There is no corresponding `dynamic_object_field::remove`, no `coin::split`, and no `withdraw_fee_coins` function anywhere in the codebase. `state::get_fee_recipient` returns the stored address but it is never passed to any `transfer::public_transfer` or `coin::split` call. The governance module `set_fee_recipient.move` itself documents that fee-recipient logic was removed.

### Impact Explanation
Every SUI coin paid as an update fee by any user is permanently locked inside the shared `PriceInfoObject` for that price feed. The coins cannot be claimed by the protocol, the fee recipient, or anyone else. `get_balance` will monotonically increase and never decrease. This constitutes permanent freezing of all protocol fee revenue.

### Likelihood Explanation
This is triggered by the normal, intended usage of the protocol. Every call to `update_single_price_feed` — the primary public entry point for price updates — deposits fees. No special attacker action is required; the locking is a structural property of the current code.

### Recommendation
Add a privileged `withdraw_fee_coins` function (gated by `LatestOnly` or an admin capability) that calls `dynamic_object_field::remove<vector<u8>, Coin<SUI>>(&mut price_info_object.id, FEE_STORAGE_KEY)` and transfers the resulting coin to `state::get_fee_recipient(pyth_state)`. Alternatively, route fees directly to the recipient at deposit time instead of accumulating them in the object.

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
