### Title
Echo `executeCallback` Permanently Bricks Requests When Callback OOGs Due to Insufficient Gas — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.executeCallback` clears the request from storage **before** invoking the consumer callback. Because the callback is wrapped in a `try/catch` that silently absorbs out-of-gas errors, an attacker can call `executeCallback` with just enough gas to clear the request and execute the catch block, but not enough for the callback to succeed. The request is permanently deleted with no replay path.

### Finding Description

In `Echo.executeCallback`, the execution order is:

1. Validate the request
2. Credit the provider fees
3. **`clearRequest(sequenceNumber)`** — permanently deletes the request from storage
4. Update `firstUnfulfilledSeq`
5. `try IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)` — callback with a hard gas cap
6. `catch { emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)"); }` — silently absorbs OOG [1](#0-0) 

The request is cleared at line 164 before the callback at line 176. There is no `require(gasleft() >= callbackGasLimit * X)` guard anywhere in the function. [2](#0-1) 

The EVM's 63/64 rule means that if the outer context has fewer than `callbackGasLimit * 64/63` gas when the `try` block executes, the callback receives less than `callbackGasLimit` gas regardless of the Solidity-level `gas:` annotation. [3](#0-2) 

**Attack path:**

Let `G` = gas available to the outer context just before the `try` block.

- Callback receives `min(callbackGasLimit, G * 63/64)` gas.
- If the attacker supplies total gas such that `G * 63/64 < callbackGasLimit`, the callback is starved.
- The callback OOGs; the outer context retains `≈ G/64` gas.
- If `G/64` ≥ cost of the `catch` block (a single event emit, ~2 000 gas), the transaction **succeeds**.
- The request is already cleared; the consumer callback never executed.

For a typical `callbackGasLimit` of 1 000 000, the attacker supplies roughly `1 015 873` total gas (overhead + `callbackGasLimit * 64/63 - 1`). The callback gets ~984 127 gas, OOGs, the catch block runs with ~31 746 gas remaining, and the transaction finalises with the request permanently gone.

The developers themselves flagged this exact concern in a TODO comment: [4](#0-3) 

### Impact Explanation

Any pending Echo price-update request can be permanently bricked by an unprivileged attacker. The consumer contract's `echoCallback` never executes, the request is deleted, and there is no replay mechanism. The user paid fees for a callback that will never arrive. This is a permanent, irreversible loss of service for every affected request.

### Likelihood Explanation

`executeCallback` is permissionlessly callable by anyone after the exclusivity period expires. The required gas value is trivially computable from on-chain data (`req.callbackGasLimit` is stored in the request struct). No special privileges, leaked keys, or oracle manipulation are required. A single attacker transaction can brick one request; a loop can brick many.

### Recommendation

Apply the same pattern used in `Entropy.revealWithCallback`: check that sufficient gas is available **before** clearing the request, and revert (rather than silently catching) when the outer context cannot guarantee the callback receives its full gas allocation.

```solidity
// Before clearRequest, enforce gas sufficiency (analogous to Entropy.sol line 622)
require(
    (gasleft() * 31) / 32 >= req.callbackGasLimit,
    "Insufficient gas for callback"
);
```

Alternatively, move `clearRequest` **after** the callback (inside the success branch), and add a `CALLBACK_FAILED` replay state analogous to `EntropyStatusConstants.CALLBACK_FAILED`. [5](#0-4) 

### Proof of Concept

```
Setup:
  - Consumer requests a price update with callbackGasLimit = 1_000_000
  - Consumer's echoCallback uses ~950_000 gas

Attack:
  1. Attacker observes the request on-chain (callbackGasLimit = 1_000_000)
  2. Attacker calls executeCallback with gas = ~1_050_000
     (enough for pre-callback overhead + clearRequest + firstUnfulfilledSeq loop
      + try-call setup + catch block, but NOT enough for the callback to complete)
  3. Inside executeCallback:
     a. clearRequest(sequenceNumber) succeeds — request deleted
     b. try _echoCallback{gas: 1_000_000}(...) — callback receives ~984_000 gas
        (63/64 of remaining gas < 1_000_000), OOGs
     c. catch { emit PriceUpdateCallbackFailed(...) } — succeeds with ~30_000 gas
  4. Transaction succeeds; request is gone; consumer callback never ran
  5. No replay is possible — findActiveRequest will revert with NoSuchRequest
``` [6](#0-5)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.
        // After successful callback, update firstUnfulfilledSeq if needed
        while (
            _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
