### Title
Incomplete Capability Flag Advertisement: `Flags::DISCOVERY` Never Removed When Discovery Protocol Is Disabled — (`util/launcher/src/lib.rs`)

### Summary
When a CKB node is configured with `SupportProtocol::Discovery` absent from `support_protocols`, the launcher still advertises `Flags::DISCOVERY` in its P2P identify message to all peers. This is because `flags` is initialized to `Flags::all()` and the code only removes `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT` flags based on config — but never removes `DISCOVERY`. Peers that receive this identify message store the node's address with the `DISCOVERY` capability flag set, recommend it to other peers requesting Discovery-capable nodes, and attempt to open the Discovery protocol with it — all of which fail or produce incorrect network state.

### Finding Description

In `util/launcher/src/lib.rs`, the node's capability flags are constructed as follows:

```rust
let mut flags = Flags::all();

if support_protocols.contains(&SupportProtocol::Relay) { ... }
else { flags.remove(Flags::RELAY); }

if support_protocols.contains(&SupportProtocol::Filter) { ... }
else { flags.remove(Flags::BLOCK_FILTER); }

// Time: no flag defined, no removal needed

if support_protocols.contains(&SupportProtocol::LightClient) { ... }
else { flags.remove(Flags::LIGHT_CLIENT); }
``` [1](#0-0) 

`Flags::all()` includes `DISCOVERY = 0b10`, `SYNC = 0b100`, `RELAY = 0b1000`, `LIGHT_CLIENT = 0b10000`, and `BLOCK_FILTER = 0b100000`. [2](#0-1) 

There is no `flags.remove(Flags::DISCOVERY)` branch when `SupportProtocol::Discovery` is absent from `support_protocols`. Yet the Discovery protocol handler is conditionally registered in `NetworkService::new()`: [3](#0-2) 

The config documentation explicitly states Discovery is optional: [4](#0-3) 

The resulting `flags` value (with `DISCOVERY` still set) is encoded into the identify message sent to every connecting peer: [5](#0-4) 

### Impact Explanation

Peers that receive the identify message from a Discovery-disabled node will:

1. Store the node's address in their peer store with `Flags::DISCOVERY` set, via `add_outbound_addr`: [6](#0-5) 

2. Return this node's address in response to `GetNodes` requests that include `required_flags` containing `DISCOVERY`, via `fetch_random_addrs` and `fetch_addrs_to_attempt`: [7](#0-6) 

3. Attempt to open the Discovery protocol with this node (since `required_flags_filter` passes), which will fail at the protocol negotiation layer because the Discovery handler is not registered on the target node.

The `required_flags_filter` function has a special-case bypass only for the exact combination `RELAY | DISCOVERY | SYNC` (accepting `COMPATIBILITY` as a legacy alias), but not for other flag combinations involving `DISCOVERY`: [8](#0-7) 

This means the peer store across the network becomes polluted with incorrect capability data for any node that disables Discovery, degrading topology quality and causing wasted outbound connection attempts and protocol negotiation failures.

### Likelihood Explanation

The `support_protocols` list is a supported, documented configuration option. Any operator who removes `Discovery` from their `support_protocols` (a supported and documented action) will silently trigger this bug. The node will continue to advertise `Flags::DISCOVERY` to all peers it connects to, propagating incorrect capability data through the discovery network. No special attacker capability is required — any peer connecting to such a node receives the incorrect identify message.

### Recommendation

Add the missing flag removal in `util/launcher/src/lib.rs` alongside the existing removals:

```rust
if !support_protocols.contains(&SupportProtocol::Discovery) {
    flags.remove(Flags::DISCOVERY);
}
```

This mirrors the existing pattern used for `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT`. [9](#0-8) 

### Proof of Concept

1. Configure a CKB node with `support_protocols` that excludes `Discovery` (e.g., `["Ping", "Identify", "Sync", "Relay", "Time", "Alert"]`).
2. Start the node and connect a peer to it.
3. Observe the identify message sent by the node: `flags` will have bit `0b10` (`DISCOVERY`) set despite Discovery not being registered.
4. The connecting peer will store this node's address with `DISCOVERY` flag in its peer store.
5. When the connecting peer receives a `GetNodes` request with `required_flags` including `DISCOVERY`, it will include the Discovery-disabled node's address in its response.
6. Any node that subsequently tries to open the Discovery protocol with the Discovery-disabled node will receive a protocol negotiation failure.

pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    if required == Flags::RELAY | Flags::DISCOVERY | Flags::SYNC {
        t.contains(required) || t.contains(Flags::COMPATIBILITY)
    } else {
        t.contains(required)
    }
}
