### Title
Provider Accrued Fees Permanently Locked When No Fee Manager Is Set — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, provider fees accumulate in `_state.providers[provider].accruedFeesInWei` via `executeCallback()`, but there is **no direct `withdraw()` function for providers**. The sole withdrawal path is `withdrawAsFeeManager()`, which requires a fee manager to be explicitly set. Because `feeManager` defaults to `address(0)` at registration, any provider that never calls `setFeeManager()` has their accrued fees permanently locked in the contract.

---

### Finding Description

**Fee accrual path:**

In `requestPriceUpdatesWithCallback()`, the user pays `msg.value >= requiredFee`. The contract stores the provider's portion:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

In `executeCallback()`, the provider is credited:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

**The only withdrawal path for provider fees:**

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(
        msg.sender == _state.providers[provider].feeManager,
        "Only fee manager"
    );
    ...
}
``` [3](#0-2) 

**Default state at registration — `feeManager` is never initialized:**

```solidity
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.baseFeeInWei = baseFeeInWei;
    ...
    provider.isRegistered = true;
``` [4](#0-3) 

The `feeManager` field in `ProviderInfo` is never set during `registerProvider()`, so it defaults to `address(0)`. [5](#0-4) 

Since `msg.sender` can never equal `address(0)` in a live transaction, `withdrawAsFeeManager()` is permanently uncallable for any provider that has not explicitly invoked `setFeeManager()`.

**Contrast with Entropy**, which provides a direct `withdraw(uint128 amount)` callable by the provider itself:

```solidity
providerInfo.accruedFeesInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
``` [6](#0-5) 

`IEcho` exposes no equivalent direct-provider withdrawal function. [7](#0-6) 

---

### Impact Explanation

Any provider that registers and accrues fees without calling `setFeeManager()` cannot recover those fees. For smart-contract providers that do not implement a `setFeeManager()` call path, the lock is **permanent**. The ETH is held in the contract with no reachable code path to release it to the provider. This is a direct analog to the BandoRouterV1 pattern: fees are collected from users and credited to a balance, but the balance has no accessible exit for the intended recipient.

---

### Likelihood Explanation

Any provider (unprivileged role) who registers via `registerProvider()` and begins receiving fee credits through `executeCallback()` is affected if they omit the `setFeeManager()` step. The `registerProvider()` function gives no indication that a separate `setFeeManager()` call is required before fees can be withdrawn. Smart-contract providers that do not include this call are permanently locked out.

---

### Recommendation

Add a direct `withdraw(uint128 amount)` function callable by the provider themselves, mirroring the Entropy contract pattern:

```solidity
function withdraw(uint128 amount) external {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(provider.isRegistered, "Provider not registered");
    require(provider.accruedFeesInWei >= amount, "Insufficient balance");
    provider.accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "Withdrawal failed");
    emit FeesWithdrawn(msg.sender, amount);
}
```

---

### Proof of Concept

1. Provider calls `registerProvider(baseFee, feedFee, gasFee)` — `feeManager` defaults to `address(0)`.
2. Users call `requestPriceUpdatesWithCallback{value: getFee(...)}(provider, ...)` — fees are collected.
3. Keeper calls `executeCallback(provider, sequenceNumber, updateData, priceIds)` — `_state.providers[provider].accruedFeesInWei` is incremented.
4. Provider attempts to recover fees: no `withdraw()` function exists in `IEcho`.
5. Provider attempts `withdrawAsFeeManager(provider, amount)` — reverts with `"Only fee manager"` because `_state.providers[provider].feeManager == address(0)` and `msg.sender != address(0)`.
6. Provider fees remain locked in the contract indefinitely.
