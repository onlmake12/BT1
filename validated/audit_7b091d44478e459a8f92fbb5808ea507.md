### Title
HolePunching Protocol Accepts Arbitrary `listen_addrs` Without Private-IP Validation, Enabling SSRF-Like LAN Scanning - (File: `network/src/protocols/hole_punching/component/connection_request.rs`, `network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

---

### Summary

The CKB HolePunching protocol processes attacker-controlled `listen_addrs` from `ConnectionRequest` and `ConnectionRequestDelivered` P2P messages and passes them directly to `try_nat_traversal`, which opens real outbound TCP connections. Unlike the Discovery and Identify protocols, which call `is_reachable()` to reject private/loopback IPs, the HolePunching code only filters by transport type (TCP + Ip4/Ip6). Any connected peer can therefore force a victim CKB node to make TCP connection attempts to arbitrary IP addresses, including RFC-1918 private ranges and loopback, enabling LAN reconnaissance, resource exhaustion, and port scanning of the node operator's internal network.

---

### Finding Description

The HolePunching protocol is enabled by default when `SupportProtocol::HolePunching` is in the config. [1](#0-0) 

**Attack path 1 — attacker as `from` peer:**

An attacker connects to a victim CKB node and sends a `ConnectionRequest` message whose `listen_addrs` field contains private or loopback IP addresses (e.g., `/ip4/192.168.1.1/tcp/22`). The victim node (acting as the `to` peer) receives the message, parses the `listen_addrs`, and stores them in `pending_delivered` after applying only a transport-type filter:

```rust
// connection_request.rs lines 196-215
let remote_listens: Vec<Multiaddr> = remote_listens
    .into_iter()
    .filter_map(|addr| match find_type(&addr) {
        TransportType::Memory | TransportType::Onion | ... => None,
        TransportType::Tcp => {
            if addr.iter().any(|p| matches!(p, Protocol::Ip4(_) | Protocol::Ip6(_))) {
                Some(addr)   // ← private IPs pass through
            } else { None }
        }
    })
    .collect();
``` [2](#0-1) 

These addresses are stored in `pending_delivered` and later consumed by `ConnectionSyncProcess`, which calls `try_nat_traversal` with the attacker-supplied addresses. [3](#0-2) 

**Attack path 2 — attacker as `to` peer:**

When the victim node is the `from` peer and receives a `ConnectionRequestDelivered` message, it calls `try_nat_traversal` directly with `content.listen_addrs` from the message. The same transport-type-only filter applies:

```rust
// connection_request_delivered.rs lines 237-257
let tasks = remote_addrs
    .into_iter()
    .filter_map(|listen_addr| match find_type(&listen_addr) {
        TransportType::Tcp => {
            if listen_addr.iter().any(|p| matches!(p, Protocol::Ip4(_) | Protocol::Ip6(_))) {
                Some(Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
            } else { None }
        }
        _ => None,
    })
    .collect::<Vec<_>>();
``` [4](#0-3) 

**The actual TCP connection attempt:**

`try_nat_traversal` in `component/mod.rs` converts the `Multiaddr` to a `SocketAddr` and then retries a real TCP `connect()` for up to 30 seconds with no IP-class check:

```rust
// component/mod.rs lines 53-84
let net_addr = match multiaddr_to_socketaddr(&addr) { ... };
// ...
while start_time.elapsed() < timeout_duration {
    let socket = create_socket(bind_addr, net_addr)?;
    match runtime::timeout(..., socket.connect(net_addr)).await { ... }
}
``` [5](#0-4) 

**Contrast with other protocols that do validate:**

The Discovery protocol explicitly calls `is_reachable()` before storing or propagating any address: [6](#0-5) 

The Identify protocol similarly filters with `is_reachable()` in `process_listens`: [7](#0-6) 

The HolePunching protocol has no equivalent guard.

---

### Impact Explanation

- **LAN scanning / SSRF**: An attacker can enumerate services on the node operator's internal network by injecting private-range addresses (10.x.x.x, 172.16.x.x, 192.168.x.x, 127.x.x.x). Each injected address causes the victim node to attempt a TCP connection, revealing open ports through timing differences.
- **Resource exhaustion**: Each `try_nat_traversal` call spawns an async task that retries for up to 30 seconds. An attacker can inject up to `ADDRS_COUNT_LIMIT = 24` addresses per message, and the rate limiter only restricts to 30 messages/second per session, allowing sustained resource consumption. [8](#0-7) 
- **De-anonymization**: Connections to attacker-controlled external IPs reveal the node's egress IP, undermining operator privacy.

---

### Likelihood Explanation

Any peer that successfully establishes a P2P session with a CKB node can send `ConnectionRequest` or `ConnectionRequestDelivered` messages. No special privilege is required. The HolePunching protocol is enabled in the default configuration. The attacker only needs to be a connected peer, which is the normal state for any node on the network.

---

### Recommendation

Apply `is_reachable()` to every address extracted from `listen_addrs` in both `respond_delivered` (in `connection_request.rs`) and `try_nat_traversal` (in `connection_request_delivered.rs`) before storing or dialing them, mirroring the guard already present in the Discovery and Identify protocols:

```rust
use p2p::utils::{is_reachable, multiaddr_to_socketaddr};

// Before storing/dialing:
.filter(|addr| match multiaddr_to_socketaddr(addr) {
    Some(socket_addr) => is_reachable(socket_addr.ip()),
    None => false,  // reject non-IP addresses in hole-punching context
})
```

Additionally, consider rate-limiting the number of concurrent `try_nat_traversal` tasks globally, not just per-session.

---

### Proof of Concept

1. Connect to a target CKB node as a normal peer.
2. Send a `HolePunchingMessage::ConnectionRequest` with:
   - `from` = attacker's peer ID
   - `to` = any known peer ID on the network
   - `listen_addrs` = `[/ip4/192.168.1.1/tcp/22, /ip4/10.0.0.1/tcp/3306, /ip4/127.0.0.1/tcp/8545]`
   - `max_hops` = 1 (so the target node itself is the `to` peer, or route it through one hop)
3. The target node's `respond_delivered` stores the three addresses in `pending_delivered` after passing the TCP+Ip4 filter (no `is_reachable` check). [9](#0-8) 
4. When the `ConnectionSync` message arrives, the target node spawns `try_nat_traversal` tasks for each of the three addresses, making outbound TCP SYN packets to `192.168.1.1:22`, `10.0.0.1:3306`, and `127.0.0.1:8545` from the victim node's network interface. [5](#0-4) 
5. By observing which connections succeed or time out (via side-channel or by controlling the target IPs), the attacker maps open services on the victim's LAN.

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
