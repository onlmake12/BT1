### Title
`ScriptHashTypeVerifier` Skips Type Script `hash_type` Validation, Allowing Non-Enabled Values to Reach Expensive Contextual Verification — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify` iterates all transaction outputs but only validates the **lock** script's `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. The **type** script's `hash_type` is never checked at this non-contextual stage. A transaction submitter can craft outputs with a valid lock `hash_type` and a structurally valid but non-enabled type script `hash_type` (e.g., `Data3 = 6`), causing the transaction to pass all non-contextual checks and proceed into the more expensive contextual verification pipeline before being rejected.

---

### Finding Description

`ScriptHashTypeVerifier::verify` loops over outputs and checks only `output.lock().hash_type()`: [1](#0-0) 

`output.type_()` is never inspected here. The `ENABLED_SCRIPT_HASH_TYPE` set contains `{0, 1, 2, 4}` — `Data3 = 6` is absent: [2](#0-1) 

The earlier deserialization guard `check_data` does check both lock and type scripts, but only for **structural validity** (valid bit pattern via `verify_value`), not for **enablement**: [3](#0-2) 

`Data3 = 6` is `3 << 1 = 6`, an even number, so it passes `verify_value` and `check_data`. It is only rejected later in `select_version` during contextual script execution: [4](#0-3) 

The call chain is:

```
submit_transaction
  → NonContextualTransactionVerifier::verify
      → ScriptHashTypeVerifier::verify   ← passes (only checks lock)
  → contextual verification
      → select_version(type_script)      ← fails here (Data3 not enabled)
``` [5](#0-4) 

---

### Impact Explanation

Contextual verification requires resolving cell deps (database lookups), loading scripts, and invoking `select_version` for each type script. All of this work is performed and then discarded for every output carrying a non-enabled type script `hash_type`. With up to ~1500 outputs within the `max_block_bytes` limit, an attacker can force O(N_outputs) rounds of cell-dep resolution and script-loading overhead per submitted transaction, all of which are avoidable with a two-line non-contextual check. The impact is bounded but real: repeated submission of such transactions amplifies per-transaction verification cost proportional to output count.

---

### Likelihood Explanation

The attack path is fully unprivileged and reachable via the standard transaction submission RPC or P2P relay. `Data3 = 6` is a valid molecule-encoded byte, passes deserialization, passes `check_data`, and passes `ScriptHashTypeVerifier`. No special access, keys, or hashpower is required. The only cost to the attacker is the transaction fee (which can be minimized).

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify` to also validate `output.type_()` when present, mirroring the existing lock check:

```rust
// after the lock check, inside the same loop:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

This keeps the rejection cheap and consistent with the existing lock-script check. [1](#0-0) 

---

### Proof of Concept

```rust
// Construct a transaction with 100 outputs:
//   lock.hash_type = Data(0)   → valid, enabled
//   type.hash_type = Data3(6)  → valid bit pattern, NOT in ENABLED_SCRIPT_HASH_TYPE
let type_script = Script::default()
    .as_builder()
    .hash_type(ScriptHashType::Data3)  // 6u8
    .build();
let output = CellOutput::new_builder()
    .lock(Script::default())           // hash_type = Data(0)
    .type_(Some(type_script).pack())
    .build();
let tx = TransactionBuilder::default()
    .outputs(vec![output; 100])
    .build();

let verifier = ScriptHashTypeVerifier::new(&tx);
assert!(verifier.verify().is_ok());   // passes — type hash_type never checked

// Full contextual verification rejects it at select_version(type_script)
// after performing cell-dep resolution for all 100 outputs.
```
