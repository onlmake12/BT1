### Title
`FeeRateCollector::statistics` Panics on `txs_sizes: None`, Silently Denying Fee-Rate Queries to RPC Callers — (File: `rpc/src/util/fee_rate.rs`)

---

### Summary

`FeeRateCollector::statistics`, the implementation backing both the `get_fee_rate_statistics` and `get_fee_rate_statics` RPC methods, calls `.expect()` unconditionally on `BlockExt::txs_sizes`, which is typed `Option<Vec<u64>>`. Any block stored with `txs_sizes: None` — a valid, documented state — causes the RPC handler to panic. The result is that an unprivileged RPC caller can trigger a panic in the node process, and no fee-rate information is returned to any caller while the affected block range is in scope.

---

### Finding Description

`FeeRateCollector::statistics` iterates over the last `target` (default 21, max 101) blocks and, for each `BlockExt`, unconditionally unwraps `txs_sizes`:

```rust
// rpc/src/util/fee_rate.rs:93
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
```

`BlockExt::txs_sizes` is declared as `Option<Vec<u64>>`:

```rust
// util/types/src/core/extras.rs:40
pub txs_sizes: Option<Vec<u64>>,
```

The field is `None` in multiple production-relevant paths:
- Blocks written by older node versions before `txs_sizes` was introduced (the migration `set_2019_block_cycle_zero` confirms legacy blocks exist without this field).
- The `chain/src/verify.rs` block-ext construction path sets `txs_sizes: None` for unverified blocks.
- The `BlockExt` doc comment itself acknowledges the dependency: *"the `get_fee_rate_statistics` RPC function requires accurate `txs_sizes` and `txs_fees` data from `BlockExt`"* — implying the data is not always present.

The two RPC entry points that invoke this path are:

```rust
// rpc/src/module/chain.rs:2124-2131
fn get_fee_rate_statics(&self, target: Option<Uint64>) -> Result<Option<FeeRateStatistics>> {
    Ok(FeeRateCollector::new(self.shared.snapshot().as_ref())
        .statistics(target.map(Into::into)))
}

fn get_fee_rate_statistics(&self, target: Option<Uint64>) -> Result<Option<FeeRateStatistics>> {
    Ok(FeeRateCollector::new(self.shared.snapshot().as_ref())
        .statistics(target.map(Into::into)))
}
```

Both are synchronous, unauthenticated RPC methods callable by any client.

---

### Impact Explanation

When any block in the queried window has `txs_sizes: None`, the `.expect()` panics. In Rust, a panic in a synchronous RPC handler propagates to the jsonrpc dispatch layer. Depending on whether the framework uses `catch_unwind`, this either:

1. **Crashes the node process** — the most severe outcome, a full DoS.
2. **Aborts the RPC handler thread/task** — the caller receives an internal error and no fee-rate data.

In either case, the fee-rate query API is non-functional for the affected block range. This is the direct CKB analog to the Balancer `getAmountsOut` returning 0: the "query before action" function fails to return useful data, forcing users to submit transactions blind — without knowing the appropriate fee rate — or to not submit at all.

---

### Likelihood Explanation

Any node that:
- Was upgraded from a version predating `txs_sizes` storage, **or**
- Has recently processed blocks where `txs_sizes` was not populated (e.g., during IBD or fork resolution via `chain/src/verify.rs`)

will have `None`-valued `txs_sizes` in its recent block window. The default query window is 21 blocks, so a single such block in the last 21 is sufficient to trigger the panic. The RPC endpoint is unauthenticated and reachable by any peer or local client.

---

### Recommendation

Replace the `.expect()` with a graceful skip. If `txs_sizes` is `None` for a block, treat that block as contributing no fee-rate samples (consistent with the existing `if txs_sizes.len() > 1` guard that already handles the empty-vec case):

```rust
// rpc/src/util/fee_rate.rs:93
// Before:
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");

// After:
let Some(txs_sizes) = txs_sizes else { return fee_rates; };
```

This matches the intent of the outer `if txs_sizes.len() > 1 && !txs_fees.is_empty()` guard and eliminates the panic path entirely.

---

### Proof of Concept

1. Start a CKB node that was migrated from a pre-`txs_sizes` database, or manually insert a `BlockExt` with `txs_sizes: None` into the tip window (as done in `util/reward-calculator/src/tests.rs:83` and `chain/src/tests/find_fork.rs`).
2. Call the RPC:
   ```json
   {"id":1,"jsonrpc":"2.0","method":"get_fee_rate_statistics","params":[]}
   ```
3. The node panics at `rpc/src/util/fee_rate.rs:93` with `"expect txs_size's length >= 1"` because `txs_sizes` is `None` for a block in the default 21-block window.
