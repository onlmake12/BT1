### Title
Insufficient Fee Rate Enforcement for High-Cycle Transactions Allows Pool Admission Below `min_fee_rate` - (File: `tx-pool/src/util.rs`)

### Summary
The `check_tx_fee` function computes the minimum required fee using the transaction's serialized byte size (`tx_size`) instead of its weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). An unprivileged transaction sender can craft a small-serialized-size but high-cycle transaction that passes the fee admission check while carrying an effective fee rate far below the configured `min_fee_rate`, forcing the node to spend significant verification CPU at minimal cost.

### Finding Description

In `check_tx_fee`, the minimum fee is computed against raw serialized size:

```rust
// tx-pool/src/util.rs line 45
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [1](#0-0) 

The actual fee rate used everywhere else in the system (ordering, eviction, estimation) is computed via `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. When cycles dominate, `weight >> size`, so the size-based admission check is far weaker than the weight-based fee rate the rest of the system enforces.

The code comment at line 42–44 explicitly acknowledges the approximation — *"Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check"* — but there is **no subsequent post-verification check** using the actual weight once cycles are known. The `check_tx_fee` call is the sole admission gate. [3](#0-2) 

The `FeeRate::fee` helper that drives the check:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;  // KW = 1000
    Capacity::shannons(fee)
}
``` [4](#0-3) 

### Impact Explanation

With the default `min_fee_rate = 1000 shannons/KW`, an attacker submits a transaction with:

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 (max per `max_tx_verify_cycles`) |
| `weight` | `max(200, 70,000,000 × 0.000170571)` = **11,940** |
| `min_fee` (admission check) | `1000 × 200 / 1000` = **200 shannons** |
| Actual fee rate | `200 × 1000 / 11,940` ≈ **17 shannons/KW** |

The transaction passes the admission check (fee ≥ 200) but carries an effective fee rate ~59× below `min_fee_rate`. The node is forced to execute 70 million VM cycles of script verification at a cost of 200 shannons to the attacker. Repeated submissions pollute the pool with transactions that will never be mined (miners also use weight-based ordering) and exhaust node CPU. [5](#0-4) 

### Likelihood Explanation

The attack path is fully open to any unprivileged actor:

1. Craft a transaction with a compact serialized form but a script that consumes near-`max_tx_verify_cycles` cycles (e.g., a tight loop in a RISC-V lock script).
2. Set the fee to exactly `min_fee_rate × tx_size / 1000` shannons.
3. Submit via the public `send_transaction` RPC.

No special role, key, or majority hash power is required. The `send_transaction` endpoint is the standard public entry point. [6](#0-5) 

The default `max_tx_verify_cycles = 70,000,000` and pool size limit bound the per-transaction cost but do not prevent sustained low-cost flooding. [7](#0-6) 

### Recommendation

After script verification completes and actual cycles are known, perform a second fee-rate check using the real weight before inserting the entry into the pool:

```rust
let weight = get_transaction_weight(tx_size, actual_cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    ));
}
```

This ensures that every admitted transaction pays fees proportional to its actual resource consumption (both bytes and cycles), closing the gap between the cheap pre-verification check and the weight-based ordering used everywhere else.

### Proof of Concept

1. Author a CKB lock script that spins for ~70 M cycles (e.g., a tight counter loop compiled to RISC-V).
2. Build a transaction spending a live cell, locking the output with that script, serialized size ≤ 300 bytes.
3. Set output capacity so that `inputs_capacity − outputs_capacity = 300 shannons` (fee = 300, just above `min_fee_rate × 300 / 1000 = 300`).
4. Call `send_transaction` on a node with default config.
5. Observe: the node accepts the transaction (fee check passes), spends ~70 M cycles verifying it, and places it in the pool with an effective fee rate of `300 × 1000 / 11,940 ≈ 25 shannons/KW` — well below the 1000 shannons/KW floor.
6. Repeat in a loop; each iteration costs ~300 shannons and forces ~70 M cycles of node CPU. [8](#0-7) [9](#0-8)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
```
