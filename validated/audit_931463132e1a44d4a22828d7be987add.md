### Title
Unbounded `delay_for(ttl/2)` in NAT Traversal Task Allows Remote Resource Exhaustion — (`network/src/protocols/hole_punching/component/connection_request_delivered.rs`)

### Summary

`ConnectionRequestDeliveredProcess::try_nat_traversal` computes `ttl = now - start` from a timestamp stored in `inflight_requests` and passes it uncapped to `runtime::delay_for(ttl/2)` inside a spawned async task. A malicious relay peer that has observed a victim's `ConnectionRequest` broadcast can delay sending a crafted `ConnectionRequestDelivered` for up to `TIMEOUT - ε` (~5 minutes), causing the victim to spawn a task that sleeps for up to 150 seconds before initiating up to 24 concurrent TCP connection attempts.

### Finding Description

**Entrypoint:** Any directly-connected P2P peer acting as a relay node.

**Step 1 — Victim broadcasts `ConnectionRequest`.**
In `notify`, the victim broadcasts `ConnectionRequest` messages (containing `from_peer_id` and `to_peer_id`) to relay nodes and records the start time: [1](#0-0) 

**Step 2 — Attacker (relay node) observes the broadcast.**
The `ConnectionRequest` is sent via `filter_broadcast`, so any connected relay peer sees `content.from` and `content.to` in plaintext.

**Step 3 — Attacker waits ~5 minutes, then sends a crafted `ConnectionRequestDelivered`.**
The attacker crafts a message with `content.from = victim_peer_id`, `content.to = target_peer_id`, `content.route = []` (empty, so no further forwarding), and `content.listen_addrs` containing up to 24 valid TCP addresses.

**Step 4 — Victim computes uncapped `ttl` and spawns a long-lived task.** [2](#0-1) 

`ttl = now - start ≈ 300000ms`. This is passed directly to `try_nat_traversal`: [3](#0-2) 

There is **no cap** on `ttl` anywhere before this call. `TIMEOUT` is defined as: [4](#0-3) 

**Step 5 — Spawned task holds 24 futures for 150 seconds, then opens 24 TCP sockets.**
After the delay, `select_ok(tasks)` polls all 24 `try_nat_traversal` futures concurrently. Each future has its own 30-second retry loop: [5](#0-4) 

Total task lifetime per attack: `ttl/2 + 30s ≈ 180 seconds`.

### Impact Explanation

- **Memory:** Each spawned task holds 24 pinned futures for up to 150 seconds before they are polled.
- **File descriptors:** After the delay, up to 24 TCP sockets are opened per task and held for up to 30 seconds.
- **Multiplier:** The victim may have up to `max_outbound` entries in `inflight_requests` simultaneously (one per NAT traversal target). An attacker who has observed all broadcast `ConnectionRequest` messages can trigger one task per entry, multiplying the resource consumption by `max_outbound * 24`.
- **Repeatability:** The 5-minute `notify` cycle repopulates `inflight_requests`, allowing the attack to repeat every cycle.

### Likelihood Explanation

- The attacker only needs to be a directly-connected P2P peer (unprivileged, no key material required).
- `ConnectionRequest` broadcasts expose `from_peer_id` and `to_peer_id` to all relay peers.
- The `forward_rate_limiter` (1 req/sec per `(from, to, item_id)`) does not prevent the attack since only one message per pair is needed.
- The `inflight_requests` cleanup in `notify` runs every `CHECK_INTERVAL = 5 minutes`, giving the attacker a precise exploitation window. [6](#0-5) 

### Recommendation

Cap `ttl` before passing it to `try_nat_traversal`. For example:

```rust
let ttl = (now - start).min(TIMEOUT / 2);
self.try_nat_traversal(ttl, content.listen_addrs);
```

This bounds `delay_for` to at most `TIMEOUT/4 = 75 seconds` regardless of message timing.

### Proof of Concept

Unit-test `try_nat_traversal` with a `start` time 299 seconds in the past:

```rust
let start = unix_time_as_millis() - 299_000;
let ttl = unix_time_as_millis() - start; // ≈ 299000ms
// delay_for(ttl / 2) = delay_for(149500ms) ≈ 149 seconds
// assert this exceeds any safe maximum (e.g., 30 seconds)
assert!(ttl / 2 > 30_000, "task delay is unbounded: {}ms", ttl / 2);
```
