Audit Report

## Title
Stale `descendants_*` Accounting in `remove_entry_and_descendants` Leaves Ancestor Entries with Inflated Eviction Priority — (File: `tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` tears down all link entries for the entire subtree before calling `remove_entry` on each node. Because `update_ancestors_index_key` relies on those same links to find still-pooled ancestors, it silently finds no ancestors and skips `sub_descendant_weight`. Every still-pooled ancestor of the removed subtree root permanently retains inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`, raising its `EvictKey.fee_rate` and shielding it from eviction.

## Finding Description

**Root cause — ordering in `remove_entry_and_descendants`:**

`remove_entry_and_descendants` (lines 252–265) first calls `remove_entry_links` for every node in the subtree, then calls `remove_entry` on each:

```rust
// pool_map.rs lines 252-265
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);   // ALL links torn down here, including root's parent pointers
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))   // update_ancestors_index_key called here
        .collect()
}
```

`remove_entry_links` (lines 418–430) removes the node from its parents' children sets, removes the node from its children's parents sets, and calls `self.links.remove(id)` — erasing the node's own entry from `TxLinksMap`.

`remove_entry` then calls `update_ancestors_index_key` at line 242, which calls `self.links.calc_ancestors` at line 434. `calc_ancestors` delegates to `calc_relative_ids` (links.rs lines 37–50), which does `self.inner.get(short_id).map(...).unwrap_or_default()`. Since `remove_entry_links` already removed the root's entry from `self.links.inner`, `get` returns `None`, `unwrap_or_default()` returns an empty set, and the `for anc_id in &ancestors` loop body never executes. `sub_descendant_weight` is never called on any still-pooled ancestor.

**Contrast with single-entry path:** When `remove_entry` is called directly (not via `remove_entry_and_descendants`), links are still intact at the time `update_ancestors_index_key` runs at line 242 — links are removed at line 245, *after* the ancestor update. The bug is exclusive to the batch-removal path.

**Confirmed call sites:**
- `process_rbf` — `process.rs` line 205: `tx_pool.pool_map.remove_entry_and_descendants(id)` — attacker-controlled via public RPC
- `resolve_conflict` — `pool_map.rs` line 310 — fires on every committed block with conflicting pool txs
- `resolve_conflict_header_dep` — `pool_map.rs` line 285

## Impact Explanation

`descendants_*` fields feed directly into `EvictKey` (entry.rs lines 234–247):

```rust
let descendants_feerate = FeeRate::calculate(entry.descendants_fee, descendants_weight);
let feerate = FeeRate::calculate(entry.fee, weight);
EvictKey {
    fee_rate: descendants_feerate.max(feerate),  // inflated by stale descendants_fee
    ...
}
```

`next_evict_entry` picks the entry with the lowest `EvictKey` to evict. Inflated `descendants_feerate` raises `fee_rate`, making the parent tx less likely to be evicted than it should be.

An attacker submits a low-fee anchor tx (tx0) and a child tx (tx1), then RBF-replaces tx1 with tx2. Each replacement round adds a ghost `txN.fee` increment to tx0's `descendants_fee` without a corresponding decrement. After N rounds, tx0's apparent `descendants_feerate` is N× inflated, shielding it from eviction even when the pool is full. Legitimate high-fee txs are evicted in preference to the attacker's artificially-boosted anchor tx, degrading mining revenue and tx throughput for honest users.

This matches: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

- **Attacker entry path**: Any unprivileged tx-pool submitter via the public `send_transaction` RPC.
- **RBF availability**: Requires `min_rbf_rate > min_fee_rate` in node config. Nodes that enable RBF are fully exposed.
- **Cost**: Each RBF round costs only the fee delta (`min_rbf_rate × size`). The ghost increment per round equals the replaced tx's fee. With a minimal fee delta, the attacker accumulates large ghost `descendants_fee` cheaply.
- **Normal-operation trigger**: `resolve_conflict` fires on all nodes during normal block processing, causing the bug without any deliberate attacker. Any pool tx that is an ancestor of a tx evicted by a committed block will have stale `descendants_*` thereafter.
- **Repeatability**: Unlimited — each RBF round compounds the inflation.

## Recommendation

Before tearing down links in `remove_entry_and_descendants`, explicitly update the still-pooled ancestors of the subtree root while links are still intact:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Update still-pooled ancestors of the root BEFORE links are removed,
    // so calc_ancestors can still traverse them.
    if let Some(root_entry) = self.get(id) {
        let root_entry = root_entry.clone();
        self.update_ancestors_index_key(&root_entry, EntryOp::Remove);
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

## Proof of Concept

**Setup**: RBF enabled (`min_rbf_rate > min_fee_rate`), two confirmed UTXOs available.

1. Submit **tx0** spending a confirmed UTXO. tx0 enters pool: `descendants_fee = tx0.fee`, `descendants_count = 1`.
2. Submit **tx1** spending tx0's output. tx0's `descendants_fee` becomes `tx0.fee + tx1.fee`, `descendants_count = 2`.
3. Submit **tx2** (RBF replacement of tx1): same input as tx1, fee > tx1.fee + `min_rbf_rate × size`. `process_rbf` calls `remove_entry_and_descendants(&tx1_id)`. Due to the bug, tx0's `descendants_fee` is **not decremented** — stays at `tx0.fee + tx1.fee`.
4. tx2 is inserted as new child of tx0. tx0's `descendants_fee` incremented by tx2.fee → now `tx0.fee + tx1.fee + tx2.fee` (correct value: `tx0.fee + tx2.fee`).
5. Repeat steps 2–4 with tx3, tx4, … Each round adds another ghost `txN.fee` to tx0's `descendants_fee`.
6. Fill the pool. Observe that tx0's `EvictKey.fee_rate` is computed from the inflated `descendants_fee`, making tx0 appear more valuable than it is. Legitimate high-fee txs are evicted while tx0 remains.

**Verification**: Add an assertion after step 3 checking that `pool_map.get(&tx0_id).unwrap().descendants_fee == tx0.fee` — it will fail, showing `descendants_fee == tx0.fee + tx1.fee` instead.
