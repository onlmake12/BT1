Audit Report

## Title
`BLOCK_INVALID` Status Not Checked Before `HEADER_VALID` in `HeaderAcceptor::accept()`, Enabling Repeated Invalid Block Re-download â€” (File: `sync/src/synchronizer/headers_process.rs`)

## Summary

`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` that omits an early-return guard for `BlockStatus::BLOCK_INVALID`. Because `BLOCK_INVALID` (bit 12, value 4096) is disjoint from `HEADER_VALID` (bit 0, value 1), a block previously marked invalid bypasses the only status guard, passes all three header-level checks, and causes `insert_valid_header` to overwrite the `BLOCK_INVALID` cache entry with `HEADER_VALID`. Any connected peer can exploit this by repeatedly sending the header of a known-rejected block, causing the node to re-download and re-verify the invalid block body in a loop, wasting bandwidth and CPU.

## Finding Description

**Root cause â€” disjoint bit flags:**

`BlockStatus` is defined in `shared/src/block_status.rs` (L8â€“17):
- `HEADER_VALID = 1` (bit 0)
- `BLOCK_INVALID = 1 << 12 = 4096` (bit 12)

`4096 & 1 == 0`, so `status.contains(BlockStatus::HEADER_VALID)` is `false` when `status` is `BLOCK_INVALID`.

**The unguarded path in `accept()`:**

In `sync/src/synchronizer/headers_process.rs` (L301â€“322), the developer left an explicit `FIXME`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... early return for already-valid headers
    return result;
}
```

There is no corresponding guard for `BLOCK_INVALID`. When `status == BLOCK_INVALID`, the `contains(HEADER_VALID)` check evaluates to `false` and execution falls through to:

1. `prev_block_check` (L324â€“332) â€” checks whether the **parent** is invalid, not the block itself. Passes if the parent is on the valid chain.
2. `non_contextual_check` (L334â€“344) â€” runs `HeaderVerifier` on the header (PoW, structure). Passes for any block rejected for body/script reasons, since the header itself is structurally valid.
3. `version_check` (L346â€“354) â€” checks `header.version() == 0`. Passes for any standard block.

If all three pass, `sync_shared.insert_valid_header(self.peer, self.header)` is called at L356, which writes `HEADER_VALID` into `block_status_map`, overwriting the `BLOCK_INVALID` entry.

The `ValidationError` enum (L371â€“375) has only `Verify`, `Version`, and `InvalidParent` variants â€” no `InvalidBlock` variant â€” confirming the FIXME's open question about which error type to return remains unresolved.

**Re-download loop:**

`block_fetcher.rs` (L247â€“284) uses `get_block_status()` to decide whether to fetch a block body. With status reset to `HEADER_VALID` (not `BLOCK_STORED`, not `BLOCK_RECEIVED`), the fetcher adds the block to the inflight queue and requests its body from the peer. When the body arrives, verification fails again, `BLOCK_INVALID` is re-set â€” and the attacker can immediately send the header again to restart the cycle.

## Impact Explanation

**Allowed impact: High â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Each loop iteration causes a full block body download (potentially hundreds of KB per block) and a full block verification pass (CPU-intensive script execution, cellbase checks, etc.). The attacker cost is a single `SendHeaders` P2P message per iteration. Because `accept()` returns `ValidationState::Valid` (the default) when all three checks pass, `HeadersProcess::execute()` does not disconnect the peer. With multiple attacker peers each targeting different previously-rejected blocks, the node's download bandwidth and verification CPU can be saturated indefinitely.

## Likelihood Explanation

Any unprivileged peer connected over P2P can send `SendHeaders`. The attacker only needs the hash of a block the target node has previously rejected â€” observable from network behavior (node requested the block, then stopped). The header for such a block is freely available on the network. No special privilege, key, or majority hashpower is required. The attack is repeatable at negligible cost per iteration.

## Recommendation

Add an explicit early-return guard for `BLOCK_INVALID` at the top of `HeaderAcceptor::accept()`, resolving the `FIXME` in `sync/src/synchronizer/headers_process.rs` (L301â€“303):

```rust
let status = self.active_chain.get_block_status(&self.header.hash());

if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(Some(ValidationError::InvalidBlock));
    return result;
}

if status.contains(BlockStatus::HEADER_VALID) {
    // ... existing early-return path
}
```

A new `ValidationError::InvalidBlock` variant should be added to the `ValidationError` enum (L371â€“375) to resolve the FIXME's open question about which error type to return.

## Proof of Concept

1. Connect a malicious peer to a CKB node.
2. Relay a complete block whose **body** is invalid (e.g., a transaction with `OutputsSumOverflow`, or an invalid script). The node processes it, fails body verification, and calls `insert_block_status(hash, BlockStatus::BLOCK_INVALID)`.
3. The malicious peer sends a `SendHeaders` message containing only the **header** of that same block.
4. `HeadersProcess::execute()` â†’ `HeaderAcceptor::accept()` is called.
5. `get_block_status()` returns `BLOCK_INVALID` (4096). `status.contains(HEADER_VALID)` â†’ `4096 & 1 == 0` â†’ `false`. No early return.
6. `prev_block_check` passes (parent is on the valid chain). `non_contextual_check` passes (header PoW and structure are valid). `version_check` passes (version == 0).
7. `insert_valid_header` is called, overwriting `block_status_map[hash]` from `BLOCK_INVALID` to `HEADER_VALID`.
8. `BlockFetcher` sees `HEADER_VALID` for the block, adds it to inflight, and requests the body.
9. Body arrives, verification fails, `BLOCK_INVALID` is re-set.
10. Attacker sends `SendHeaders` again â€” repeat from step 4 indefinitely.

To reproduce in a unit test: pre-populate `block_status_map` with `BLOCK_INVALID` for a header hash, call `accept()` with a structurally valid header for that hash, and assert that the returned `ValidationResult` is `Invalid` and that `block_status_map` still contains `BLOCK_INVALID` â€” the current code fails both assertions.
