### Title
Incomplete Update Validation in `verifyUpdate()` — Missing Replay Protection Allows Stale Price Injection - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.verifyUpdate()` validates only the signer's identity and expiry. It performs no replay protection and no timestamp-freshness enforcement. Any previously valid Lazer update — signed by a still-trusted signer — can be replayed by an unprivileged caller to any consumer that accepts externally supplied update bytes, injecting stale prices into downstream DeFi protocols.

---

### Finding Description

`verifyUpdate()` performs three checks:

1. EVM format magic (`0x2A22999A`)
2. ECDSA signature recovery
3. `isValidSigner(signer)` — checks only `trustedSignerToExpiresAtMapping[signer]` [1](#0-0) 

`isValidSigner()` itself only reads the mapping, not the canonical `trustedSigners[100]` array: [2](#0-1) 

What is **not** checked:

- Whether the `keccak256(payload)` hash has been consumed before (no nonce/hash registry)
- Whether the timestamp embedded in the payload is fresh

The function returns `(payload, signer)` to the caller with no record that this exact update was already processed. [3](#0-2) 

The dual-storage design (`trustedSigners[100]` array + `trustedSignerToExpiresAtMapping`) is kept in sync only by `updateTrustedSigner()` (owner-only), so the mapping is always authoritative for currently registered signers. The incomplete validation is therefore not about the two structures diverging — it is about the absence of any replay guard at the protocol level. [4](#0-3) 

---

### Impact Explanation

A DeFi consumer that exposes a public `updatePrice(bytes calldata update)` entry point (the standard pattern for push-oracle consumers) will call `verifyUpdate()` with whatever bytes the caller supplies. An attacker who observed a valid update when, say, ETH/USD was $3,000 can replay that update after the price has moved to $2,000. `verifyUpdate()` accepts it — the signer is still trusted and not expired — and the consumer stores the stale $3,000 price. This enables:

- Incorrect liquidations (positions that should be liquidated are not, or vice versa)
- Incorrect collateral valuations
- Arbitrage against protocols using the stale price

---

### Likelihood Explanation

- The attacker needs only to observe any on-chain or off-chain Lazer update (publicly available).
- The signer key rotation period is long (governance-controlled), so a replayed update from hours or days ago will still pass `isValidSigner()`.
- Any consumer that accepts update bytes from an arbitrary `msg.sender` is directly exploitable — this is the standard oracle consumer pattern.
- No privileged access, leaked key, or Sybil attack is required.

---

### Recommendation

Track consumed update hashes in a mapping inside `verifyUpdate()`:

```solidity
mapping(bytes32 => bool) public usedUpdateHashes;

function verifyUpdate(...) external payable returns (...) {
    // ... existing checks ...
    bytes32 updateHash = keccak256(update);
    require(!usedUpdateHashes[updateHash], "update already used");
    usedUpdateHashes[updateHash] = true;
    // ...
}
```

Alternatively, enforce a maximum age on the timestamp embedded in the payload (e.g., reject updates older than 60 seconds).

---

### Proof of Concept

1. Observe a valid Lazer update `update_old` accepted on-chain when `ETH/USD = $3,000`.
2. Wait for the price to drop to `$2,000` and for the consumer's stored price to reflect `$2,000`.
3. Call the consumer's public `updatePrice(update_old)` function.
4. The consumer calls `verifyUpdate{value: fee}(update_old)`.
5. `verifyUpdate()` recovers the signer, confirms `isValidSigner(signer) == true` (signer not yet expired), and returns the old payload.
6. The consumer stores `$3,000` again — a $1,000 price discrepancy — enabling profitable exploitation of any protocol using this price. [1](#0-0)

    function verifyUpdate(
        bytes calldata update
    ) external payable returns (bytes calldata payload, address signer) {
        // Require fee and refund excess
        require(msg.value >= verification_fee, "Insufficient fee provided");
        if (msg.value > verification_fee) {
            payable(msg.sender).transfer(msg.value - verification_fee);
        }

        if (update.length < 71) {
            revert("input too short");
        }
        uint32 EVM_FORMAT_MAGIC = 706910618;

        uint32 evm_magic = uint32(bytes4(update[0:4]));
        if (evm_magic != EVM_FORMAT_MAGIC) {
            revert("invalid evm magic");
        }
        uint16 payload_len = uint16(bytes2(update[69:71]));
        if (update.length < 71 + payload_len) {
            revert("input too short");
        }
        payload = update[71:71 + payload_len];
        bytes32 hash = keccak256(payload);
        (signer, , ) = ECDSA.tryRecover(
            hash,
            uint8(update[68]) + 27,
            bytes32(update[4:36]),
            bytes32(update[36:68])
        );
        if (signer == address(0)) {
            revert("invalid signature");
        }
        if (!isValidSigner(signer)) {
            revert("invalid signer");
        }
    }
