### Title
Asymmetric `total_tx_size`/`total_tx_cycles` Accounting in `PoolMap::add_entry` Causes Inflated Pool Statistics Leading to Excessive Eviction - (`tx-pool/src/component/pool_map.rs`)

### Summary

`PoolMap::add_entry` computes the new `total_tx_size` and `total_tx_cycles` values **before** calling `check_and_record_ancestors`, which can internally evict transactions via `remove_entry_and_descendants`. Those evictions correctly decrement `self.total_tx_size` through `update_stat_for_remove_tx`, but `add_entry` then **overwrites** `self.total_tx_size` with the stale pre-eviction value. The result is that `total_tx_size` is permanently inflated by the aggregate size of every evicted transaction, mirroring the Maia `pbHermes` dilution pattern exactly: a resource counter is incremented on entry but the corresponding decrement (from evictions) is silently discarded.

### Finding Description

In `PoolMap::add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221):

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    // ...
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // ← snapshot BEFORE evictions
    // ...
    evicts = self.check_and_record_ancestors(&mut entry)?;          // ← may call remove_entry_and_descendants
    //                                                                    which calls remove_entry
    //                                                                    which calls update_stat_for_remove_tx
    //                                                                    → self.total_tx_size -= evicted_size
    // ...
    self.total_tx_size = total_tx_size;    // ← OVERWRITES the decremented value
    self.total_tx_cycles = total_tx_cycles; // ← OVERWRITES the decremented value
    Ok((true, evicts))
}
``` [1](#0-0) 

`updated_stat_for_add_tx` captures `self.total_tx_size + entry.size` at line 210–211 **before** any evictions occur. [2](#0-1) 

`check_and_record_ancestors` (lines 588–640) can evict transactions when the ancestor count exceeds `max_ancestors_count` and the excess is attributable to `cell_ref_parents`. Each eviction calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size`: [3](#0-2) 

But the final two lines of `add_entry` unconditionally overwrite `self.total_tx_size` and `self.total_tx_cycles` with the stale snapshot, discarding all decrements performed during eviction. [4](#0-3) 

The eviction path in `check_and_record_ancestors` is reached when:

```
ancestors_count > max_ancestors_count
AND
ancestors_count - cell_ref_parents.len() <= max_ancestors_count
``` [5](#0-4) 

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict transactions from the pool:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict lowest-fee transactions
}
``` [6](#0-5) 

Each time the eviction path in `check_and_record_ancestors` is triggered, `total_tx_size` is inflated by the aggregate serialized size of all evicted transactions. After repeated triggers, `total_tx_size` far exceeds the actual bytes in the pool. `limit_size` then evicts legitimate pending/proposed transactions to try to satisfy the (now impossible) condition `total_tx_size <= max_tx_pool_size`, causing a progressive DoS of the mempool for all other users. The pool can be drained of all valid transactions while `total_tx_size` remains inflated.

### Likelihood Explanation

An unprivileged tx-pool submitter (RPC `send_transaction` caller or P2P relay peer) can trigger this by:

1. Submitting a chain of ≥ 126 transactions (each spending the previous output), filling the ancestor limit.
2. Having some of those transactions declare a specific live cell as a `cell_dep`.
3. Submitting a new transaction that **spends** that same cell as an input. This makes those chain members `cell_ref_parents`, satisfying the eviction condition in `check_and_record_ancestors`.

Each such submission inflates `total_tx_size` by the sizes of the evicted chain members. The attack requires paying transaction fees for the chain, but the cost is bounded and the inflation is permanent until the node restarts.

### Recommendation

Compute `total_tx_size` and `total_tx_cycles` **after** `check_and_record_ancestors` completes, so that any evictions performed during ancestor resolution are already reflected in `self.total_tx_size` before the final assignment. Concretely, replace the pre-computation snapshot with a post-eviction incremental update:

```rust
// Remove the pre-computation:
// let (total_tx_size, total_tx_cycles) = self.updated_stat_for_add_tx(...)?;

// Validate capacity first (without mutating):
self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Increment AFTER evictions have already decremented:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

### Proof of Concept

```
Initial state: pool has 126 txs in a chain (tx_0 → tx_1 → … → tx_125).
tx_50 and tx_60 both declare output X as a cell_dep.
total_tx_size = S (actual pool size).

Step 1: Attacker submits tx_new:
  - input: spends output X
  - This makes tx_50 and tx_60 cell_ref_parents of tx_new.
  - ancestors_count = 127 (> 125), cell_ref_parents = {tx_50, tx_60}
  - 127 - 2 = 125 <= 125 → eviction path triggered.

Step 2: Inside add_entry:
  - updated_stat_for_add_tx: total_tx_size_snapshot = S + size(tx_new)
  - check_and_record_ancestors evicts tx_50 and its descendants:
      update_stat_for_remove_tx called → self.total_tx_size = S - size(evicted)
  - self.total_tx_size = total_tx_size_snapshot = S + size(tx_new)  ← WRONG
  - Actual pool size ≈ S - size(evicted) + size(tx_new)
  - Inflation = size(evicted)

Step 3: limit_size is called. It sees total_tx_size >> actual size.
  It evicts legitimate transactions until total_tx_size ≤ max_tx_pool_size,
  removing far more transactions than necessary.

Repeat Step 1–3 to progressively drain the pool.
```

    /// Calculate size and cycles statistics for adding a tx.
    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
