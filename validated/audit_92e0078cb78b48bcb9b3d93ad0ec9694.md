Audit Report

## Title
Out-of-Bounds Vec Index Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary

`TxConfirmStat::remove_unconfirmed_tx` correctly guards the `block_unconfirmed_txs` access when `tx_age >= 1000`, but unconditionally indexes `confirm_blocks_to_failed_txs[tx_age - 1]` without an equivalent bounds check. When a tracked transaction is evicted after more than 1000 blocks, `tx_age - 1 >= 1000` exceeds the Vec's valid range (0..999), causing a Rust index-out-of-bounds panic that crashes the tx-pool service task and renders the node non-functional.

## Finding Description

`MAX_CONFIRM_BLOCKS` is 1000 and all three tracking Vecs are allocated with that length: [1](#0-0) [2](#0-1) 

In `remove_unconfirmed_tx`, `tx_age` is computed as `tip_height.saturating_sub(entry_height) as usize`. The guard at line 208 correctly routes `block_unconfirmed_txs` access when `tx_age >= 1000`: [3](#0-2) 

However, the `count_failure` branch immediately after is **outside** that guard and uses the same unbounded `tx_age`: [4](#0-3) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds for a Vec of length 1000. Rust's `[]` indexing panics unconditionally.

The panic path is fully reachable: `shared_builder.rs` line 600 calls `fee_estimator.reject_tx(&tx_hash)` from the tx-pool reject callback: [5](#0-4) 

`reject_tx` calls `drop_tx`, which calls `drop_tx_inner(..., count_failure=true)`, which calls `remove_unconfirmed_tx` with `count_failure=true`: [6](#0-5) 

## Impact Explanation

A panic inside the tx-pool reject callback crashes the tx-pool service task, rendering the node unable to process transactions. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

All preconditions are reachable by an unprivileged user: (1) node configured with `algorithm = ConfirmationFraction` (a documented production option); (2) attacker submits a valid transaction via `send_transaction` RPC with fee above `min_fee_rate`; (3) the transaction is never confirmed and remains in the pool for >1000 blocks (~2.2 hours at 8 s/block) — the default `expiry_hours = 12` allows a tx to survive well past 1000 blocks before time-based eviction fires; (4) eviction triggers the panic path. No hashpower, no privileged access, and no victim mistakes are required. The attack is repeatable.

## Recommendation

Add a bounds check before the `count_failure` branch, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

## Proof of Concept

```rust
#[test]
fn test_remove_unconfirmed_tx_oob_panic() {
    let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
    let max_confirm_blocks = 1000;
    let decay = 0.993f64;
    let mut stat = TxConfirmStat::new(buckets, max_confirm_blocks, decay);

    // tx tracked at height 0, evicted at height 1001 → tx_age = 1001 > 1000
    let entry_height: u64 = 0;
    let tip_height: u64 = 1001;
    let bucket_index: usize = 0;

    // Prevent underflow on the block_unconfirmed_txs branch
    stat.bucket_stats[bucket_index].old_unconfirmed_txs = 1;

    // Panics: confirm_blocks_to_failed_txs[1000] is out of bounds (len=1000)
    stat.remove_unconfirmed_tx(entry_height, tip_height, bucket_index, true);
}
```
