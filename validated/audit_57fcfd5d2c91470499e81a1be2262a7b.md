### Title
`Executor.sol` Has No Wormhole Address Update Mechanism — Circular Dependency Permanently Bricks Governance If Wormhole Interface Changes - (File: `target_chains/ethereum/contracts/contracts/executor/Executor.sol`)

---

### Summary

`Executor.sol` stores a fixed `IWormhole private wormhole` reference set only at initialization. Unlike `PythGovernance.sol`, which has a dedicated `setWormholeAddress` governance action, the `Executor` contract exposes **no mechanism to update its Wormhole address or interface binding**. Because the only way to upgrade the Executor is through a Wormhole-verified VAA (via `execute(vaa)`), a breaking change to the Wormhole contract creates an unrecoverable circular dependency: the upgrade path requires Wormhole, but Wormhole is the thing that broke.

---

### Finding Description

`Executor.sol` initializes its Wormhole dependency once:

```solidity
IWormhole private wormhole;
...
wormhole = IWormhole(_wormhole);
```

Every governance action flows through `verifyGovernanceVM`, which calls:

```solidity
(IWormhole.VM memory vm, bool valid, ) = wormhole.parseAndVerifyVM(encodedVM);
```

The only defined `ExecutorAction` is `Execute`. There is no `SetWormholeAddress` action, no emergency owner bypass, and no direct `upgradeTo` path that skips VAA verification. The upgrade path itself is VAA-gated:

```solidity
// test confirms upgrade is done via execute(vaa) → executor.upgradeTo(newImpl)
abi.encodeWithSelector(ExecutorUpgradable.upgradeTo.selector, newImplementation)
```

The code even contains an unresolved TODO acknowledging the gap:

```solidity
enum ExecutorAction {
    // TODO: add an instruction to change the governance data source.
    Execute // 0
}
```

By contrast, `PythGovernance.sol` explicitly handles this case with `setWormholeAddress`, and the governance payload library ships `EvmSetWormholeAddress` for the main Pyth price contract. The Executor has no equivalent.

---

### Impact Explanation

If the Wormhole contract (an ERC1967 proxy) is upgraded to a new implementation that changes the `parseAndVerifyVM` signature, or if Wormhole governance migrates to a new contract address, the Executor's `execute()` function will revert on every call. Since `execute()` is the only entry point for all governance actions — including `upgradeTo` — the Executor becomes permanently bricked with no recovery path. All Entropy contracts that use the Executor as their owner/governance relay are also frozen: fee changes, provider updates, and contract upgrades all become impossible.

---

### Likelihood Explanation

Wormhole has historically upgraded its contracts and guardian sets. The Wormhole contract Pyth deploys is itself an ERC1967 proxy, meaning its implementation can be swapped by Wormhole governance. A breaking interface change is low-probability but non-zero, and the consequence is permanent rather than temporary. The asymmetry with `PythGovernance.sol` (which already has `setWormholeAddress`) shows the team is aware of this risk class but did not apply the same mitigation to the Executor.

---

### Recommendation

Add a `SetWormholeAddress` governance action to `Executor.sol`, mirroring the pattern already present in `PythGovernance.sol`. Alternatively, add an emergency `upgradeTo` path protected by a multisig owner that does not require Wormhole VAA verification, so that a Wormhole failure does not permanently block the upgrade escape hatch.

---

### Proof of Concept

**Step 1 — Wormhole upgrades its implementation**, changing `parseAndVerifyVM` to a new signature (e.g., adding a parameter or renaming the function).

**Step 2 — Any call to `Executor.execute(vaa)` reverts** at:

```solidity
// Executor.sol line 120
(IWormhole.VM memory vm, bool valid, ) = wormhole.parseAndVerifyVM(encodedVM);
```

because the ABI selector no longer matches the new implementation.

**Step 3 — The upgrade VAA** (which encodes `executor.upgradeTo(newImpl)`) also goes through `execute()`, so it also reverts. There is no alternative path.

**Step 4 — The Executor is permanently frozen.** All Entropy governance (fee changes, provider changes, contract upgrades) is blocked. The Executor's `lastExecutedSequence` can never advance, and no new implementation can be installed. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function verifyGovernanceVM(
        bytes memory encodedVM
    ) internal returns (IWormhole.VM memory parsedVM) {
        (IWormhole.VM memory vm, bool valid, ) = wormhole.parseAndVerifyVM(
            encodedVM
        );

        if (!valid) revert ExecutorErrors.InvalidWormholeVaa();

        if (
            vm.emitterChainId != ownerEmitterChainId ||
            vm.emitterAddress != ownerEmitterAddress
        ) revert ExecutorErrors.UnauthorizedEmitter();

        if (vm.sequence <= lastExecutedSequence)
            revert ExecutorErrors.MessageOutOfOrder();

        lastExecutedSequence = vm.sequence;

        return vm;
    }

    function setWormholeAddress(
        SetWormholeAddressPayload memory payload,
        bytes memory encodedVM
    ) internal {
        address oldWormholeAddress = address(wormhole());
        setWormhole(payload.newWormholeAddress);

        // We want to verify that the new wormhole address is valid, so we make sure that it can
        // parse and verify the same governance VAA that is used to set it.
        (IWormhole.VM memory vm, bool valid, ) = wormhole().parseAndVerifyVM(
            encodedVM
        );

        if (!valid) revert PythErrors.InvalidGovernanceMessage();

        if (!isValidGovernanceDataSource(vm.emitterChainId, vm.emitterAddress))
            revert PythErrors.InvalidGovernanceMessage();

        if (vm.sequence != lastExecutedGovernanceSequence())
            revert PythErrors.InvalidWormholeAddressToSet();

        GovernanceInstruction memory gi = parseGovernanceInstruction(
            vm.payload
        );

        if (gi.action != GovernanceAction.SetWormholeAddress)
            revert PythErrors.InvalidWormholeAddressToSet();

        // Purposefully, we don't check whether the chainId is the same as the current chainId because
        // we might want to change the chain id of the wormhole contract.

        // contract parses the payload correctly.

            revert PythErrors.InvalidWormholeAddressToSet();

        emit WormholeAddressSet(oldWormholeAddress, address(wormhole()));
    }
