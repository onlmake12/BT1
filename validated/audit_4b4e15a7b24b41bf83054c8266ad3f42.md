Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Unique-`from` `ConnectionRequest` Flood — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
An unprivileged remote peer can exhaust heap memory on a victim CKB node by flooding it with `ConnectionRequest` messages at 30 msg/sec using unique `from` PeerIds targeting the victim's own `local_peer_id`. Every message bypasses the `forward_rate_limiter` and the `pending_delivered` deduplication check, inserting a new unbounded entry into the map. Eviction only occurs every 5 minutes in `notify()`, allowing up to 9,000 entries per session per window, compounding linearly with additional sessions.

## Finding Description
`HolePunching::pending_delivered` is declared as `HashMap<PeerId, PendingDeliveredInfo>` with no capacity bound. [1](#0-0) 

**Guard 1 — session rate limiter** (`received()`, L95–107): keyed by `(session_id, msg.item_id())`, allowing 30 `ConnectionRequest` messages/sec per session. This passes for an attacker sending at exactly 30 msg/sec. [2](#0-1) 

**Guard 2 — `forward_rate_limiter`** (`execute()`, L132–143): keyed by `(content.from, content.to, msg_item_id)`. With a unique `from` PeerId per message, every message gets a fresh bucket — the limiter is fully bypassed. [3](#0-2) 

**Guard 3 — deduplication in `respond_delivered()`** (L161–167): skips only if the **same** `from_peer_id` was seen within `HOLE_PUNCHING_INTERVAL`. With unique `from` PeerIds, this check never fires. [4](#0-3) 

**Unbounded insert** (L234–237): after a successful `send_message_to`, every unique `from_peer_id` is inserted with no size cap. [5](#0-4) 

**Eviction** only occurs in `notify()` via `retain(...)`, which fires every `CHECK_INTERVAL = 300s`. [6](#0-5) 

The full insertion path is:

```
received()
  → rate_limiter.check_key((session_id, item_id))          // 30/sec per session — passes
  → execute()
      → forward_rate_limiter.check_key((from, to, item_id)) // bypassed: unique `from` each time
      → self_peer_id == content.to                          // victim is the target
      → respond_delivered(from_peer_id, ...)
          → pending_delivered.get(&from_peer_id)            // miss: new unique key
          → remote_listens non-empty check                  // passes: attacker supplies valid TCP addr
          → send_message_to(peer, ...)                      // succeeds: attacker is connected
          → pending_delivered.insert(from_peer_id, ...)     // unbounded growth
```

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.**

Per session per 5-minute window: 30 msg/sec × 300 s = 9,000 entries. Each entry stores up to `ADDRS_COUNT_LIMIT = 24` `Multiaddr` objects (~50 bytes each), yielding ~1.2 KB per entry, or ~10.8 MB per session per window. [7](#0-6) 

With multiple concurrent attacker sessions the rate multiplies linearly. The victim also sends a `ConnectionRequestDelivered` reply for every accepted message, amplifying outbound traffic. Sustained attack leads to OOM and node crash.

## Likelihood Explanation
The attack requires only a standard P2P connection to the victim. The victim's `PeerId` is publicly advertised via the P2P discovery layer. No special privileges, keys, or hashpower are needed. The attacker constructs valid `ConnectionRequest` messages with random `from` PeerIds and at least one valid TCP listen address (e.g., `/ip4/1.2.3.4/tcp/1234`) — both trivially achievable. The attack is repeatable and automatable.

## Recommendation
1. **Cap `pending_delivered`**: Enforce a maximum size (e.g., 1,000 entries). On overflow, reject new insertions or evict the oldest entry.
2. **Rate-limit insertions per session**: Track how many `pending_delivered` entries originated from each session and cap that count.
3. **Shorten or decouple the eviction interval**: Run `pending_delivered` eviction more frequently than `CHECK_INTERVAL`, or evict eagerly on insertion when the map exceeds a threshold.
4. **Validate `from` PeerId against the sending session**: Reject `ConnectionRequest` messages where `from` does not match the session's authenticated peer identity, eliminating spoofed `from` PeerIds entirely.

## Proof of Concept
```
1. Connect to victim node V (obtain its PeerId P from the P2P discovery layer).
2. From a single session, send 30 ConnectionRequest messages per second:
     from = PeerId::random()          // unique each time
     to   = P                         // victim's local_peer_id
     listen_addrs = [/ip4/1.2.3.4/tcp/1234]  // one valid TCP addr
     max_hops = 6
3. After 300 seconds (before the next notify() fires):
     assert pending_delivered.len() == 9000
     assert no eviction has occurred
4. Observe victim heap growth (~10 MB per session) and degraded HashMap performance.
5. Open additional sessions to multiply the effect until OOM.
```
