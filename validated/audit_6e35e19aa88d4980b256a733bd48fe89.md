Audit Report

## Title
Snappy Varint Pre-Allocation Before Decompression Enables Heap Amplification DoS — (`network/src/compress.rs`)

## Summary
In `LengthDelimitedCodecWithCompress::decode`, a zeroed buffer sized to the attacker-supplied snappy varint is allocated unconditionally before decompression is attempted. Any unprivileged inbound peer can craft a wire frame of ~10 bytes whose snappy varint claims up to 8,388,608 bytes of decompressed output, triggering an ~8 MB heap allocation per frame before any decompression validation occurs. The same pattern exists in `Message::decompress`. Under sustained reconnection flooding with up to 117 simultaneous inbound connections, this causes continuous RSS growth and eventual OOM on memory-constrained nodes.

## Finding Description
`MAX_UNCOMPRESSED_LEN` is defined as `1 << 23 = 8,388,608` at line 13. [1](#0-0) 

In `LengthDelimitedCodecWithCompress::decode`, the exact sequence is:

1. `self.length_delimited.decode(src)?` at line 226 enforces `max_frame_length` on the wire frame (e.g., 1 KB for Ping), limiting only the compressed wire bytes. [2](#0-1) 

2. The flag byte check at line 232 reads `data[0] & COMPRESS_FLAG` from the received data — not `self.enable_compress` — meaning any peer can trigger the compressed decode path regardless of the protocol's compression setting. [3](#0-2) 

3. `decompress_len(&data[1..])` at line 233 reads the attacker-controlled LEB128 varint from the compressed payload without validating the rest of the stream. [4](#0-3) 

4. The guard at line 235 uses strict `>`, so `decompressed_bytes_len = MAX_UNCOMPRESSED_LEN` (8,388,608) itself passes the check. Any value from 1 to 8,388,608 is accepted. [5](#0-4) 

5. `BytesMut::zeroed(decompressed_bytes_len)` at line 242 allocates up to ~8 MB **before** any decompression occurs. [6](#0-5) 

6. `SnapDecoder::new().decompress(...)` then fails because the payload does not actually decompress to the declared size, returning `Err(InvalidData)`. The connection is closed and the buffer is freed — but the allocation already occurred. [7](#0-6) 

The same pre-allocation pattern exists in `Message::decompress` at line 81: `vec![0; decompressed_bytes_len]` is allocated before decompression. [8](#0-7) 

All protocols built via `CKBProtocol::new_with_support_protocol` hardcode `compress: true`, making every protocol reachable. [9](#0-8) 

The codec is instantiated per-protocol in `build()` with the per-protocol `max_frame_length`. [10](#0-9) 

`max_frame_length` values confirm the amplification ratios (e.g., Ping: 1 KB wire → 8 MB heap = ~8,000× amplification). [11](#0-10) 

## Impact Explanation
**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

With `max_peers = 125` and `max_outbound_peers = 8`, up to ~117 inbound attacker-controlled connections are permitted by default. Each connection can simultaneously trigger a decode, yielding `117 × 8 MB ≈ 936 MB` of peak transient heap pressure per attack wave. Rust's allocator (jemalloc by default in CKB) does not immediately return freed memory to the OS; repeated allocation/free cycles cause heap fragmentation and RSS growth disproportionate to live allocations. Under sustained reconnection flooding, the node's RSS grows continuously, eventually triggering OOM on memory-constrained deployments.

## Likelihood Explanation
- No authentication, proof-of-work, or privileged role is required — any peer can connect as an inbound peer.
- A single IP can open up to 117 inbound connections.
- Wire cost is ~10 bytes per ~8 MB allocation cycle (~800,000× amplification for Ping).
- Each cycle closes the connection, but the attacker can immediately reconnect.
- The attack is locally reproducible with a trivially crafted payload.

## Recommendation
Move `BytesMut::zeroed(decompressed_bytes_len)` to **after** decompression validation, or use a streaming/bounded decompressor that does not require pre-allocating the full declared output size. At minimum, add a secondary guard that rejects frames where the declared decompressed size is disproportionate to the compressed payload size (e.g., reject if `decompressed_bytes_len > compressed_payload_len * MAX_COMPRESSION_RATIO`). The same fix must be applied to `Message::decompress` at line 81 (`vec![0; decompressed_bytes_len]`). Additionally, change the guard from `>` to `>=` so that `MAX_UNCOMPRESSED_LEN` itself is also rejected. [12](#0-11) [13](#0-12) 

## Proof of Concept
```rust
// Craft a snappy raw stream: LEB128(8_388_608) = [0x80, 0x80, 0x80, 0x04]
let mut compressed_payload: Vec<u8> = vec![0x80, 0x80, 0x80, 0x04];
compressed_payload.push(0x00); // minimal snappy literal (will fail decompression)

// Build wire frame: 4-byte length prefix + COMPRESS_FLAG (0x80) + compressed_payload
let frame_len = (1 + compressed_payload.len()) as u32;
let mut wire_frame = BytesMut::new();
wire_frame.put_u32(frame_len);
wire_frame.put_u8(0x80); // COMPRESS_FLAG
wire_frame.extend_from_slice(&compressed_payload);

let mut codec = LengthDelimitedCodecWithCompress::new(
    true,
    length_delimited::Builder::new().max_frame_length(1024).new_codec(),
    0.into(),
);
for _ in 0..1000 {
    let mut buf = wire_frame.clone();
    let _ = codec.decode(&mut buf);
    // Each iteration: BytesMut::zeroed(8_388_608) allocated then freed at line 242
    // RSS grows due to heap fragmentation / allocator retention
}
```

Each iteration allocates `BytesMut::zeroed(decompressed_bytes_len)` at line 242, then immediately frees it on decompression failure. RSS grows proportionally to iteration count, not wire-byte count. [14](#0-13)
