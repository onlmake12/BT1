### Title
Stale `evict_key` on ancestor entries after `remove_entry_and_descendants` skips ancestor index update — (`tx-pool/src/component/pool_map.rs`)

### Summary
`PoolMap::remove_entry_and_descendants` pre-removes all link entries before calling `remove_entry` on each tx. This causes `update_ancestors_index_key` to find an empty ancestor set (links already gone), so surviving ancestor A's `descendants_*` fields and `evict_key` are never decremented. A retains an inflated `EvictKey::fee_rate` and `descendants_count`, making it immune to correct eviction ordering.

### Finding Description

`remove_entry_and_descendants` first strips all link entries for the entire removal set, then calls `remove_entry` on each: [1](#0-0) 

`remove_entry` calls `update_ancestors_index_key`, which resolves ancestors via `self.links.calc_ancestors`: [2](#0-1) 

`calc_ancestors` calls `calc_relative_ids`, which looks up the entry in `self.links.inner`. Since `remove_entry_links` already called `self.links.remove(id)`, the entry is absent and the function returns an empty set: [3](#0-2) 

The loop over ancestors is therefore a no-op. A's `sub_descendant_weight` is never called, and `e.evict_key = e.inner.as_evict_key()` is never re-computed.

`EvictKey` is built from the stale `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`: [4](#0-3) 

`EvictKey::fee_rate` is `max(descendants_feerate, feerate)`. With stale descendants data, this value is inflated. The eviction comparator sorts ascending on `fee_rate` first, then `descendants_count`: [5](#0-4) 

`next_evict_entry` picks the smallest `evict_key` first: [6](#0-5) 

A's inflated `evict_key` pushes it toward the back of the eviction queue, protecting it from being selected by `limit_size`: [7](#0-6) 

The existing test `test_remove_entry_and_descendants` only checks that B and C are absent and that `calc_descendants` returns the correct set — it does **not** assert A's `descendants_count` or `evict_key`: [8](#0-7) 

### Impact Explanation

After the attack, A holds a stale `EvictKey` with an inflated `fee_rate` (derived from B's and C's fees). When the pool is full and `limit_size` iterates `iter_by_evict_key()`, A is ranked behind genuinely high-fee transactions and is not evicted. Legitimate higher-fee transactions submitted by other users are rejected with `Reject::Full` instead. The stale state persists until A is committed or the pool is cleared.

### Likelihood Explanation

The trigger is straightforward: submit A (low fee) → B (high fee, child of A) → C (high fee, child of B), then submit D double-spending B's input. `resolve_conflict` calls `remove_entry_and_descendants(B)`, leaving A stale. No privileged access, no PoW, no Sybil attack is required — only standard P2P transaction submission. The attacker pays fees for B, C, D but retains A in the pool indefinitely at an artificially high eviction rank.

### Recommendation

In `remove_entry_and_descendants`, before stripping links, collect and update ancestors of the removal set. One approach: for each tx in `removed_ids`, call `update_ancestors_index_key(tx, Remove)` **before** calling `remove_entry_links`, so that `calc_ancestors` can still traverse the live link graph. Alternatively, compute the ancestor set upfront and pass it explicitly to `remove_entry` to avoid the dependency on live links.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
