Audit Report

## Title
Unbounded `peers` Vec Growth in `UnknownTxHashPriority` via Repeated `RelayTransactionHashes` Below Guard Threshold — (`sync/src/types/mod.rs`)

## Summary
`push_peer()` appends to the `peers` Vec with no length cap. The size guard in `add_ask_for_txs` checks only `unknown_tx_hashes.len()` (count of distinct hash keys), which never increases when the same hashes are re-announced. A single peer sending K = 32766 hashes repeatedly bypasses the guard entirely, growing every entry's `peers` Vec without bound until the node OOM-crashes.

## Finding Description

`push_peer()` appends unconditionally: [1](#0-0) 

In `add_ask_for_txs`, the `Occupied` branch calls `push_peer` with no inspection of the current `peers` Vec length: [2](#0-1) 

The size guard fires only when `unknown_tx_hashes.len()` (distinct keys) exceeds the threshold — it never checks the length of individual `peers` Vecs: [3](#0-2) 

With one connected peer, the threshold is `1 × 32767 = 32767`. Sending K = 32766 hashes inserts 32766 distinct keys on the first message (`unknown_tx_hashes.len() = 32766 < 32767`), so the guard never fires. Every subsequent identical message hits the `Occupied` branch for all 32766 entries, calling `push_peer` 32766 times and growing each entry's `peers` Vec by 1, while `unknown_tx_hashes.len()` stays fixed at 32766. [4](#0-3) 

The `tx_filter` does not block this path. Hashes enter `tx_filter` only via `mark_as_known_txs`, called only after a transaction is resolved: [5](#0-4) 

Hashes sitting in `unknown_tx_hashes` are invisible to `tx_filter`, so the filter check in `execute()` passes on every re-announcement: [6](#0-5) 

The existing integration test only tests the K = 32767 boundary (exactly `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`), which triggers `TooManyUnknownTransactions` and a ban on the very first call — it does not cover the K = 32766 repeated-message path: [7](#0-6) 

## Impact Explanation

Memory grows as `K × M × sizeof(PeerIndex)` where K = 32766 and M = number of repeated messages. At M = 10,000: `32766 × 10,001 × 8 bytes ≈ 2.6 GB`. The node runs out of memory and crashes or becomes unresponsive. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The attack requires only a standard P2P connection. No PoW, no stake, no cryptographic material, and no coordination with other peers is needed. The hashes do not need to correspond to real transactions. The attacker simply sends the same `RelayTransactionHashes` message with 32766 fabricated hashes in a tight loop. The existing guard is trivially bypassed by staying one hash below the per-peer limit. The attack is repeatable and deterministic.

## Recommendation

Cap the `peers` Vec length inside `push_peer` or before calling it in the `Occupied` branch of `add_ask_for_txs`. A reasonable bound is the total number of connected peers (`self.peers.state.len()`). Additionally, deduplicate peer entries before appending — skip the push if the peer is already present in the Vec — to prevent a single peer from inflating the Vec at all. The `tx_filter` should also mark hashes as seen when they first enter `unknown_tx_hashes` to prevent the same peer from re-announcing the same hash repeatedly.

## Proof of Concept

```
1. Connect one peer to the target node (peers.state.len() = 1).
2. Compute threshold = 1 × 32767 = 32767. Choose K = 32766.
3. Send RelayTransactionHashes with 32766 novel fabricated hashes.
   → unknown_tx_hashes.len() = 32766 (Vacant path, all inserted).
   → Guard: 32766 >= 50000? NO. 32766 >= 32767? NO. Guard does not fire.
   → Returns Status::ok(). No ban.
4. Send the identical RelayTransactionHashes message repeatedly in a loop.
   → Each call: 32766 Occupied entries, push_peer called 32766 times.
   → unknown_tx_hashes.len() stays at 32766. Guard never fires.
   → Each entry's peers Vec grows by 1 per message.
5. After M = 10,000 messages:
   → Each of 32766 entries has a peers Vec of length 10,001.
   → Total PeerIndex allocations: 32766 × 10,001 × 8 bytes ≈ 2.6 GB.
6. Node OOM-crashes or becomes unresponsive.
```
