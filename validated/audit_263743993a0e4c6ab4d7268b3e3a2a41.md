### Title
Unprivileged Lamport Theft from Any `SignatureSet` Account via Missing Authority Check in `close_signature_set` — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/close_signature_set.rs`)

### Summary
The `close_signature_set` instruction closes a `SignatureSet` account and transfers its rent-exempt lamports to `sol_destination`. The only constraint on `sol_destination` is that it is a transaction signer. There is no check that `sol_destination` is the original payer who funded the `SignatureSet`. Any unprivileged attacker can call this instruction for any `PostedVaaV1` that has a non-zero `signature_set` field, redirecting the rent deposit to themselves.

### Finding Description
In `CloseSignatureSet`, the `sol_destination` field is declared as:

```rust
#[account(mut)]
sol_destination: Signer<'info>,
```

and the `signature_set` field is declared as:

```rust
#[account(
    mut,
    close = sol_destination
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,
``` [1](#0-0) 

The Anchor `close = sol_destination` macro transfers all lamports from `signature_set` to `sol_destination` and zeroes the account. The only structural constraint is `has_one = signature_set` on `posted_vaa`, which merely verifies that the `PostedVaaV1`'s stored `signature_set` pubkey matches the account passed in. [2](#0-1) 

The `PostedVaaV1` state stores the `signature_set` pubkey in its `info` field, which is publicly readable on-chain. [3](#0-2) 

An attacker can enumerate all `PostedVaaV1` PDAs (seeds: `b"PostedVAA" ++ message_hash`), read the `signature_set` pubkey from each, and call `close_signature_set` with themselves as `sol_destination`. The instruction succeeds for any valid `(posted_vaa, signature_set)` pair regardless of who originally funded the `signature_set`.

### Impact Explanation
The `SignatureSet` account is funded by the relayer payer during the `verify_signatures` legacy flow. [4](#0-3) 

The payer is recorded nowhere in the `SignatureSet` state — the account only stores `sig_verify_successes`, `message_hash`, and `guardian_set_index`. [5](#0-4) 

An attacker can systematically drain all `SignatureSet` accounts associated with posted governance VAAs. Governance execution via `execute_posted_vaa` does not require the `SignatureSet` to exist (it only needs `posted_vaa` and `claim_record`), so governance is not blocked — but the relayer's rent deposit is permanently stolen. [6](#0-5) 

The economic impact: every governance VAA relayed via the legacy `verify_signatures` → `post_vaa` path loses its `SignatureSet` rent deposit (~0.002 SOL per account for a 19-guardian set) to the attacker. If governance operations are frequent or the attacker runs a bot, this constitutes systematic theft from the relayer infrastructure.

### Likelihood Explanation
The attack requires no privileges, no leaked keys, and no special conditions. Any account that can sign a Solana transaction can execute it. All inputs (`PostedVaaV1` PDAs and their `signature_set` pubkeys) are publicly enumerable from on-chain state. The call sequence is a single instruction per target account.

### Recommendation
Add an authority constraint to `sol_destination`. The original payer's pubkey must be stored in the `SignatureSet` (or `PostedVaaV1`) at creation time, and `close_signature_set` must verify `sol_destination.key() == stored_payer`. For example:

```rust
// In SignatureSet state, add:
pub payer: Pubkey,

// In CloseSignatureSet constraints:
#[account(
    mut,
    close = sol_destination,
    constraint = signature_set.payer == sol_destination.key() @ CoreBridgeError::InvalidFeeRecipient
)]
signature_set: Account<'info, AccountVariant<SignatureSet>>,
```

Alternatively, store the payer in `PostedVaaV1` and enforce `has_one = sol_destination` on the posted VAA.

### Proof of Concept
```rust
// Attacker keypair: attacker_kp (any funded Solana wallet)
// 1. Enumerate all PostedVaaV1 accounts (getProgramAccounts with discriminator filter b"vaa\x01")
// 2. For each account, read signature_set pubkey at bytes [9..41] of account data
// 3. For each (posted_vaa_pda, sig_set_pubkey) pair where sig_set_pubkey != Pubkey::default():
let ix = close_signature_set(
    attacker_kp.pubkey(),  // sol_destination — receives all lamports
    posted_vaa_pda,
    sig_set_pubkey,
);
// 4. Send transaction signed by attacker_kp
// Result: sig_set_pubkey account is closed, lamports credited to attacker
```

The invariant "only the original payer can reclaim `SignatureSet` lamports" is violated because the program never records or checks the original payer identity. [7](#0-6)
