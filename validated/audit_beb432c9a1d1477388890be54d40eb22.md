### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Arbitrary Pending Transactions — (File: `rpc/src/module/pool.rs`)

---

### Summary

The `remove_transaction` RPC method in CKB's Pool module accepts an arbitrary transaction hash and removes that transaction (and all its descendants) from the tx-pool with no caller authentication or ownership check. Any RPC caller who can reach the endpoint can silently evict any other user's pending transaction, forcing the victim to resubmit — a direct analog to the flash-loan griefing pattern where an attacker acts on behalf of a victim using a pre-existing authorization.

---

### Finding Description

`remove_transaction` is declared in the `PoolRpc` trait and implemented in `PoolRpcImpl` with no identity or authorization check:

```rust
// rpc/src/module/pool.rs  line 662-669
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

The method is registered as a standard Pool-module RPC in `service_builder.rs` via `add_pool_rpc_methods`, which is enabled whenever `pool_enable()` returns true — the default for any production node. No per-caller identity, IP restriction, or ownership proof is enforced at any layer between the JSON-RPC server and the tx-pool write path.

The underlying `remove_tx` call in `tx-pool/src/process.rs` (line 440-455) removes the transaction from the verify queue, the orphan pool, and the main pool unconditionally:

```rust
pub(crate) async fn remove_tx(&self, tx_hash: Byte32) -> bool {
    let id = ProposalShortId::from_tx_hash(&tx_hash);
    { let mut queue = self.verify_queue.write().await;
      if queue.remove_tx(&id).is_some() { return true; } }
    { let mut orphan = self.orphan.write().await;
      if orphan.remove_orphan_tx(&id).is_some() { return true; } }
    let mut tx_pool = self.tx_pool.write().await;
    tx_pool.remove_tx(&id)
}
```

The attacker's only precondition is knowing the target transaction hash — which is public information, observable from `get_raw_tx_pool` or from P2P relay.

---

### Impact Explanation

1. **Persistent transaction eviction**: An attacker can continuously call `remove_transaction(victimTxHash)` every time the victim resubmits, preventing the victim's transaction from ever reaching the proposed or committed state.
2. **Forced fee escalation**: In a congested pool, each forced resubmission may require a higher fee rate to re-enter the pool, directly costing the victim CKB capacity.
3. **Time-sensitive transaction failure**: Transactions with `since` time-locks, DAO withdrawal deadlines, or HTLC expiry windows can be permanently invalidated if the attacker evicts them until the time window closes.
4. **Descendant cascade**: The removal also evicts all descendant transactions, amplifying the impact across an entire transaction chain belonging to the victim.

---

### Likelihood Explanation

Many CKB node operators expose their RPC endpoint publicly to serve wallets, dApps, or light clients. The Pool module (including `remove_transaction`) is enabled by default. Transaction hashes are publicly observable via `get_raw_tx_pool` or P2P relay. No privileged access, leaked key, or Sybil capability is required. The attack is cheap, repeatable, and scriptable.

---

### Recommendation

1. **Restrict `remove_transaction` to localhost or an authenticated channel** (e.g., a separate admin port not exposed in the public RPC config).
2. **Add submitter binding**: Record the source IP or a caller token at submission time and reject `remove_transaction` calls that do not match the original submitter.
3. **Document the risk explicitly**: If the method is intentionally public, the documentation should warn operators that exposing the Pool module publicly grants any caller the ability to evict any pending transaction.
