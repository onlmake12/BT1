### Title
Deviation-Only Subscriptions Permanently Blocked When All Stored Prices Are Zero — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler._validateShouldUpdatePrices()`, the deviation check loop silently `continue`s (skips) any feed whose stored or incoming price is zero. If **all** feeds in a subscription have a stored price of zero, the entire deviation loop completes without ever returning `updateTimestamp`, and the function falls through to `revert SchedulerErrors.UpdateConditionsNotMet()`. For subscriptions configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`, this permanently prevents any keeper from successfully calling `updatePriceFeeds()` after the first update stores zero prices — an exact structural analog to the CollateralizationOracleWrapper bug where the deviation check always returned false.

---

### Finding Description

`_validateShouldUpdatePrices()` contains the following deviation loop:

```solidity
if (params.updateCriteria.updateOnDeviation) {
    for (uint8 i = 0; i < priceFeeds.length; i++) {
        PythStructs.PriceFeed storage previousFeed = _state
            .priceUpdates[subscriptionId][priceFeeds[i].id];

        if (previousFeed.id == bytes32(0)) {
            return updateTimestamp;   // first-ever update always passes
        }

        int64 currentPrice  = priceFeeds[i].price.price;
        int64 previousPrice = previousFeed.price.price;

        // Skip if either price is zero to avoid division by zero
        if (previousPrice == 0 || currentPrice == 0) {
            continue;                 // ← silently skips the feed
        }
        ...
        if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
            return updateTimestamp;
        }
    }
}
revert SchedulerErrors.UpdateConditionsNotMet();
```

The first update for a subscription always succeeds because `previousFeed.id == bytes32(0)` triggers an early return. That first update then stores whatever prices were in the update data via `_storePriceUpdates()`. If those stored prices are zero (`previousFeed.price.price == 0`), every subsequent call to `_validateShouldUpdatePrices()` will hit `previousPrice == 0 → continue` for every feed, exhaust the loop, and unconditionally revert with `UpdateConditionsNotMet`. The deviation trigger is permanently dead for that subscription.

The same outcome occurs if the **incoming** update data contains all-zero prices (`currentPrice == 0`), because the `||` condition skips the feed regardless of which side is zero. [1](#0-0) 

The storage write that "locks in" the zero prices happens unconditionally after `_validateShouldUpdatePrices` returns: [2](#0-1) 

---

### Impact Explanation

A subscription configured with `updateOnDeviation: true` and `updateOnHeartbeat: false` (deviation-only mode) becomes permanently un-updatable by any keeper once zero prices are stored. Concretely:

- `updatePriceFeeds()` always reverts with `UpdateConditionsNotMet` for that subscription.
- The subscription's stored price data becomes permanently stale.
- Any whitelisted consumer calling `getPricesNoOlderThan()` will eventually receive a `StalePrice` revert, or — if using `getPricesUnsafe()` — will silently consume arbitrarily old prices.
- The subscription's ETH balance is not drained (calls revert), but the subscription is effectively bricked: no keeper incentive exists and no keeper can succeed.

---

### Likelihood Explanation

The Pyth oracle can legitimately publish `price = 0` for feeds that are newly listed, temporarily halted, or in an error state. The `parsePriceFeedUpdatesWithConfig` call in `updatePriceFeeds()` imposes no lower bound on the price value — it only enforces timestamp bounds. A single successful first-update call with zero-price data (which always passes because `previousFeed.id == bytes32(0)`) is sufficient to permanently brick a deviation-only subscription. An unprivileged keeper or the subscription manager themselves can trigger this state by submitting valid Pyth update data that happens to contain zero prices. [3](#0-2) 

---

### Recommendation

Replace the silent `continue` with logic that treats a zero-to-nonzero (or nonzero-to-zero) price transition as an infinite deviation — i.e., always trigger an update when either price is zero rather than skipping the feed:

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A price crossing zero is always a significant deviation
    if (previousPrice != currentPrice) {
        return updateTimestamp;
    }
    continue; // both are zero, no change
}
```

Alternatively, add a post-loop check: if all feeds were skipped due to zero prices and `updateOnDeviation` is the only active criterion, treat the update as valid (or revert with a dedicated error that signals the subscription needs reconfiguration).

---

### Proof of Concept

1. Create a subscription: `updateOnDeviation: true`, `updateOnHeartbeat: false`, `deviationThresholdBps: 100`.
2. Call `updatePriceFeeds()` with update data where all `price.price == 0`. This succeeds because `previousFeed.id == bytes32(0)` → early return. Zero prices are stored via `_storePriceUpdates()`.
3. Call `updatePriceFeeds()` again with update data where all `price.price == 1_000_000` (massive deviation from zero).
4. Inside `_validateShouldUpdatePrices()`: `previousPrice == 0` → `continue` for every feed → loop exhausts → `revert UpdateConditionsNotMet()`.
5. The subscription can never be updated by any keeper regardless of how large the price deviation is. [4](#0-3)
