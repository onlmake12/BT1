Audit Report

## Title
`forward_rate_limiter` Bypassed via Attacker-Controlled `from`/`to` Fields — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `HolePunching` protocol's `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)`, where both fields are deserialized directly from the attacker-supplied message body with no verification against the actual sending session. An attacker can bypass this inner rate limiter entirely by cycling through distinct fabricated `(from, to)` peer ID pairs, causing the node to forward up to 30 messages per second to `sqrt(N)` peers each — all passing the inner limiter unchecked. The `forward_rate_limiter` hashmap also grows without bound during a session, as `retain_recent()` is never called in `notify()`.

## Finding Description
Two rate limiters are initialized in `HolePunching::new()`:

- **Outer limiter** (`rate_limiter`): keyed by `(session_id, msg.item_id())` at 30 req/s — uses the real, non-spoofable session ID. [1](#0-0) 

- **Inner limiter** (`forward_rate_limiter`): keyed by `(content.from, content.to, msg_item_id)` at 1 req/s — uses attacker-supplied fields from the message body. [2](#0-1) 

The outer limiter is checked first using the real session ID: [3](#0-2) 

In `ConnectionRequestProcess::execute()`, the inner limiter is checked using `content.from` and `content.to` — both deserialized from the message payload, with no assertion that `content.from` matches the peer ID of the actual sending session: [4](#0-3) 

The same pattern is present in `ConnectionRequestDeliveredProcess::execute()`: [5](#0-4) 

And in `ConnectionSyncProcess::execute()`: [6](#0-5) 

When the target peer is not directly connected, `forward_message()` broadcasts to `sqrt(N)` peers: [7](#0-6) 

`retain_recent()` is only called on disconnect, never in the periodic `notify()` handler: [8](#0-7) [9](#0-8) 

The routing-loop claim in the submission is incorrect — the code does check `content.route.contains(self_peer_id)` and ignores already-routed messages, preventing loops: [10](#0-9) 

**Exploit path:**
1. Attacker connects as a normal P2P peer.
2. Attacker sends up to 30 `ConnectionRequest` messages/second (outer limiter allows this).
3. Each message carries a distinct fabricated `(from, to)` pair.
4. Each unique pair creates a fresh bucket in `forward_rate_limiter`, so all 30 pass the inner check.
5. The node forwards each message to `sqrt(total_peers)` other peers — the inner limiter provides zero protection.
6. The `forward_rate_limiter` hashmap grows by 30 entries/second per attacker connection; `retain_recent()` is only called on disconnect, not periodically.

## Impact Explanation
This matches **High (10001–15000 points): "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

Each forwarded `ConnectionRequest` is broadcast to `sqrt(N)` peers. At 30 req/s per attacker connection with N=100 peers, the node issues ~300 outbound forwarding sends/second. With multiple attacker connections, this multiplies linearly. Additionally, the unbounded hashmap growth (30 new entries/second/connection, never pruned mid-session) constitutes a slow memory exhaustion vector — at 30 req/s for 24 hours, a single connection produces ~2.6M entries consuming hundreds of MB.

## Likelihood Explanation
Only a standard P2P connection is required — no keys, stake, or privileged role. Crafting molecule-encoded messages with varying `from`/`to` byte sequences is straightforward with any P2P client. The inner rate limiter is never triggered, so no warning is logged and no ban is applied. The attack is repeatable and sustainable indefinitely.

## Recommendation
1. **Verify `content.from` against the actual session**: After parsing, assert that `content.from` equals the peer ID of the sending session. Reject (and ban) messages where they do not match.
2. **Re-key `forward_rate_limiter` by session**: Use `(peer_index, msg_item_id)` — the same key as the outer limiter — so the forwarding budget is tied to the real sender.
3. **Periodically prune the hashmap**: Call `forward_rate_limiter.retain_recent()` on the `CHECK_INTERVAL` timer in `notify()`, not only on disconnect.

## Proof of Concept
```
1. Connect to a CKB node on SupportProtocols::HolePunching.
2. In a loop at ≤30 iterations/second:
   a. Generate a fresh random PeerId for `from` (or increment a counter).
   b. Build a valid ConnectionRequest molecule message: from=<fresh_id>, to=<any_id>,
      max_hops=6, listen_addrs=[valid multiaddr with from's peer ID], route=[].
   c. Send to the node.
3. Observe: the node forwards each message to sqrt(N) peers (no TooManyRequests returned).
4. After T seconds: forward_rate_limiter hashmap contains T*30 entries;
   T*30*sqrt(N) forwarding sends have been issued to other peers.
```

    async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
        let status = self.network_state.connection_status();

        let now = unix_time_as_millis();
        self.pending_delivered
            .retain(|_, (_, t)| (now - *t) < TIMEOUT);
        self.inflight_requests.retain(|_, t| (now - *t) < TIMEOUT);

        if status.non_whitelist_outbound < status.max_outbound && status.total > 0 {
            let target = &self.network_state.required_flags;
            let addrs = self.network_state.with_peer_store_mut(|p| {
                p.fetch_nat_addrs(
                    (status.max_outbound - status.non_whitelist_outbound) as usize,
                    *target,
                )
            });

            let from_peer_id = self.network_state.local_peer_id();
            let listen_addrs = {
                let public_addr = self.network_state.public_addrs(ADDRS_COUNT_LIMIT);
                if public_addr.len() < ADDRS_COUNT_LIMIT {
                    let observed_addrs = self
                        .network_state
                        .observed_addrs(ADDRS_COUNT_LIMIT - public_addr.len());
                    let iter = public_addr
                        .iter()
                        .chain(observed_addrs.iter())
                        .map(Multiaddr::to_vec)
                        .map(|v| packed::Address::new_builder().bytes(v).build());
                    packed::AddressVec::new_builder().extend(iter).build()
                } else {
                    let iter = public_addr
                        .iter()
                        .map(Multiaddr::to_vec)
                        .map(|v| packed::Address::new_builder().bytes(v).build());
                    packed::AddressVec::new_builder().extend(iter).build()
                }
            };

            let mut inflight = Vec::new();
            for i in addrs {
                if let Some(to_peer_id) = extract_peer_id(&i.addr) {
                    let conn_req = {
                        let content = component::init_request(
                            from_peer_id,
                            &to_peer_id,
                            listen_addrs.clone(),
                        );
                        packed::HolePunchingMessage::new_builder()
                            .set(content)
                            .build()
                    };
                    let proto_id = SupportProtocols::HolePunching.protocol_id();

                    // Broadcast to a number of nodes equal to the square root of the total connection count using gossip.
                    let mut total = status.total.isqrt();
                    let _ignore = context
                        .filter_broadcast(
                            TargetSession::Filter(Box::new(move |_| {
                                total = total.saturating_sub(1);
                                total != 0
                            })),
                            proto_id,
                            conn_req.as_bytes(),
                        )
                        .await;
                    inflight.push(to_peer_id);
                }
            }

            let now = unix_time_as_millis();
            for peer_id in inflight {
                self.inflight_requests.insert(peer_id, now);
            }
        }
    }
