### Title
Permissionless Keeper Can Submit Stale Price Updates to Drain Subscription Funds and Manipulate Stored Prices - (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler.updatePriceFeeds()` function is permissionless and passes `minAllowedPublishTime = 0` to `parsePriceFeedUpdatesWithConfig`. This allows any keeper to cherry-pick any historically valid Pyth price update (up to `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` old) rather than the latest one. A malicious keeper can select an older price that artificially triggers deviation-based update criteria, draining subscription funds and storing a stale/incorrect price for downstream readers.

---

### Finding Description

`Scheduler.updatePriceFeeds()` has no access control on the caller (any address can call it) and delegates price validation entirely to `parsePriceFeedUpdatesWithConfig` with `minAllowedPublishTime = 0`:

```solidity
// Scheduler.sol line 311-319
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    0,  // <-- no lower bound on publish time
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    false,
    true,
    false
);
```

The only timestamp guard is in `_validateShouldUpdatePrices`, which rejects updates older than `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (based on the *highest* timestamp in the submitted data) and older than the last stored update. Within that window, any price is accepted.

A malicious keeper can:
1. Observe the current on-chain stored price in the Scheduler.
2. Browse Hermes history for a Pyth price update from within the validity window (e.g., 30 minutes ago) whose price value shows a large deviation from the stored price, even if the *current* price has not deviated.
3. Submit that older update to `updatePriceFeeds()`, satisfying the deviation trigger.
4. Collect the keeper payment from the subscription balance.
5. Leave readers with a stale, incorrect price.

For heartbeat-based subscriptions, the keeper can similarly submit any price from within the validity window that is newer than the last stored update, choosing whichever price is most favorable to them or to a protocol they are colluding with.

---

### Impact Explanation

- **Subscription fund drain**: Deviation-based subscriptions can be triggered spuriously by submitting old prices that show artificial deviation, paying the keeper from the subscription balance for each such update.
- **Stale/incorrect prices for readers**: Protocols consuming prices via `getPricesUnsafe` or `getPricesNoOlderThan` with a loose age parameter receive a price that does not reflect the current market, potentially causing incorrect liquidations, mispriced trades, or other financial harm.
- **Heartbeat manipulation**: For heartbeat subscriptions, the keeper can select which price within the validity window to store, biasing the stored price in a direction favorable to themselves or a colluding party.

---

### Likelihood Explanation

- `updatePriceFeeds` is explicitly permissionless by design (README states "Anyone can run a Keeper node; no registration is required").
- Pyth's Hermes API provides historical price updates, making it trivial to retrieve older signed price data.
- The validity window (`PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`) is on the order of one hour, providing a large selection of historical prices to choose from.
- Economic incentive exists: keepers are paid per successful update, incentivizing spurious triggering of deviation criteria.

---

### Recommendation

Pass a meaningful `minAllowedPublishTime` to `parsePriceFeedUpdatesWithConfig` instead of `0`. For example, require the submitted price to be no older than a tight bound (e.g., `block.timestamp - MAX_PRICE_AGE_SECONDS`) rather than relying solely on the post-parse `_validateShouldUpdatePrices` check. This ensures keepers cannot select arbitrarily old prices from within the validity window.

```solidity
uint64 minPublishTime = curTime > MAX_PRICE_AGE_SECONDS
    ? curTime - MAX_PRICE_AGE_SECONDS
    : 0;

pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,
    params.priceIds,
    minPublishTime,  // enforce freshness at parse time
    curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
    false,
    true,
    false
);
```

### Proof of Concept

1. A subscription is created with `updateOnDeviation = true`, `deviationThresholdBps = 100` (1%), and current stored price = $1000.
2. The current Pyth price is $1005 (0.5% deviation — below threshold, no legitimate update needed).
3. A malicious keeper queries Hermes for historical price data and finds a signed update from 45 minutes ago where the price was $990 (1% deviation from stored $1000).
4. The keeper calls `Scheduler.updatePriceFeeds(subscriptionId, [oldUpdateData])`.
5. `parsePriceFeedUpdatesWithConfig` accepts the update (timestamp is within `[0, now+10s]`).
6. `_validateShouldUpdatePrices` accepts it (timestamp > `priceLastUpdatedAt`, within `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`, deviation ≥ threshold).
7. The keeper is paid from the subscription balance; the stored price is now $990 (stale).
8. Readers calling `getPricesUnsafe` receive $990 instead of the actual ~$1005. [1](#0-0) [2](#0-1) [3](#0-2)

    function updatePriceFeeds(
        uint256 subscriptionId,
        bytes[] calldata updateData
    ) external override {
        uint256 startGas = gasleft();

        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Get the Pyth contract and parse price updates
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);

        // If we don't have enough balance, revert
        if (status.balanceInWei < pythFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // Parse the price feed updates with an acceptable timestamp range of [0, now+10s].
        // Note: We don't want to reject update data if it contains a price
        // from a market that closed a few days ago, since it will contain a timestamp
        // from the last trading period. Thus, we use a minimum timestamp of zero while parsing,
        // and we enforce the past max validity ourselves in _validateShouldUpdatePrices using
        // the highest timestamp in the update data.
        status.balanceInWei -= pythFee;
        status.totalSpent += pythFee;
        uint64 curTime = SafeCast.toUint64(block.timestamp);
        (
            PythStructs.PriceFeed[] memory priceFeeds,
            uint64[] memory slots
        ) = pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
                updateData,
                params.priceIds,
                0, // We enforce the past max validity ourselves in _validateShouldUpdatePrices
                curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,
                false,
                true,
                false
            );

    function _validateShouldUpdatePrices(
        uint256 subscriptionId,
        SchedulerStructs.SubscriptionParams storage params,
        SchedulerStructs.SubscriptionStatus storage status,
        PythStructs.PriceFeed[] memory priceFeeds
    ) internal view returns (uint256) {
        // Use the most recent timestamp, as some asset markets may be closed.
        // Closed markets will have a publishTime from their last trading period.
        // Since we verify all updates share the same Pythnet slot, we still ensure
        // that all price feeds are synchronized from the same update cycle.
        uint256 updateTimestamp = 0;
        for (uint8 i = 0; i < priceFeeds.length; i++) {
            if (priceFeeds[i].price.publishTime > updateTimestamp) {
                updateTimestamp = priceFeeds[i].price.publishTime;
            }
        }

        // Calculate the minimum acceptable timestamp (clamped at 0)
        // The maximum acceptable timestamp is enforced by the parsePriceFeedUpdatesWithSlots call
        uint256 minAllowedTimestamp = (block.timestamp >
            PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
            : 0;

        // Validate that the update timestamp is not too old
        if (updateTimestamp < minAllowedTimestamp) {
            revert SchedulerErrors.TimestampTooOld(
                updateTimestamp,
                block.timestamp
            );
        }

        // Reject updates if they're older than the latest stored ones
        if (
            status.priceLastUpdatedAt > 0 &&
            updateTimestamp <= status.priceLastUpdatedAt
        ) {
            revert SchedulerErrors.TimestampOlderThanLastUpdate(
                updateTimestamp,
                status.priceLastUpdatedAt
            );

### Keeper Network & Incentives

- Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`. The main goal of making this component a permissionless network rather a set of permissioned nodes is to enhance reliability for the feeds -- if one provider fails, others should be available to service the subscriptions. We can improve this reliability by sourcing independent providers, and by making it profitable to push updates, paid out by the users of the feeds.

- Keepers are paid directly by the subscription's funds held in this contract for each successful update they perform. The payment covers gas costs plus a premium, and payment is sent directly to `msg.sender` (the keeper) at the end of `updatePriceFeeds`. The first transaction included in a block that passes checks will succeed and receive the payment. Subsequent attempts for the same update interval will revert since we verify the update criteria on-chain. By only allowing updates when they are needed, we keep costs predictable for the users.
