### Title
Unrestricted `revealWithCallback()` Allows Front-Running of Provider Fulfillment Transactions — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.revealWithCallback()` carries no caller restriction — it is explicitly documented as callable by anyone. An unprivileged attacker can observe the Fortuna keeper's pending fulfillment transaction in the mempool and front-run it with a higher gas price. The attacker's transaction delivers the callback to the original requester and clears the request; the provider's transaction then fails because the request no longer exists. This is a direct structural analog to the pickle-finance `harvest()` displacement front-run.

---

### Finding Description

`revealWithCallback()` is declared `public` with no access-control modifier and no `msg.sender` check:

```solidity
// Anyone can call this method to fulfill a request, but the callback will only be made to the original requester.
function revealWithCallback(
    address provider,
    uint64 sequenceNumber,
    bytes32 userContribution,
    bytes32 providerContribution
) public override {
``` [1](#0-0) 

All four arguments needed to call the function are publicly observable:
- `provider` and `sequenceNumber` — from the `RequestedWithCallback` event emitted at request time.
- `userContribution` — also emitted in `RequestedWithCallback`.
- `providerContribution` — served by the provider's public API once the reveal delay has passed. [2](#0-1) 

**Legacy path (`gasLimit10k == 0`):** The request is cleared *before* the callback is invoked. If an attacker front-runs the Fortuna keeper's transaction with sufficient gas, the callback is delivered and the request is cleared. The keeper's transaction then hits `findActiveRequest` and reverts because the slot is empty — the keeper wastes gas. [3](#0-2) 

**New flow (`gasLimit10k != 0`):** The same displacement occurs. The attacker's transaction succeeds (callback delivered, request cleared); the keeper's transaction fails on `findActiveRequest`. [4](#0-3) 

The interface specification reinforces that this is an open, unrestricted surface: [5](#0-4) 

---

### Impact Explanation

- **Fortuna keeper gas drain**: Every front-run causes the keeper's `revealWithCallback` transaction to revert after consuming gas for signature verification, storage reads, and the `revealHelper` hash-chain walk. At scale (many concurrent requests), a sustained attacker can impose significant gas costs on the provider.
- **Operational disruption**: Fortuna's retry logic must handle unexpected reverts on already-cleared requests, adding latency and complexity to the fulfillment pipeline.
- **No randomness manipulation**: The random number is cryptographically committed; the attacker cannot alter it. The callback is always delivered to the original requester.
- **No direct financial gain for the attacker**: The attacker pays gas to front-run but receives nothing. This is a pure griefing / displacement attack.

---

### Likelihood Explanation

- All required call arguments are publicly observable on-chain (events) and via the provider's public API.
- Standard mempool monitoring (e.g., `eth_subscribe("pendingTransactions")`) is sufficient to detect and front-run keeper transactions.
- The attacker only needs to outbid the keeper's gas price — a trivial operation on any EVM chain.
- The Fortuna keeper is a known, predictable actor that submits `revealWithCallback` transactions regularly, making it a reliable target.

---

### Recommendation

Restrict `revealWithCallback()` to a set of authorized callers — at minimum, the registered provider for the given request, and optionally the original requester. A modifier analogous to the `onlyBenevolent` fix in the pickle-finance case would suffice:

```solidity
modifier onlyAuthorizedFulfiller(address provider, uint64 sequenceNumber) {
    EntropyStructsV2.Request storage req = findRequest(provider, sequenceNumber);
    require(
        msg.sender == req.provider ||
        msg.sender == req.requester,
        "Unauthorized fulfiller"
    );
    _;
}
```

Alternatively, document the griefing risk explicitly and ensure Fortuna implements gas-price escalation and retry logic to recover from displaced transactions.

---

### Proof of Concept

```
1. Alice (consumer contract) calls requestV2{value: fee}() → emits RequestedWithCallback(
       provider=P, requester=Alice, sequenceNumber=N, userRandomNumber=U
   )

2. Provider P's Fortuna keeper fetches providerContribution=C from its hash chain
   and submits:
       revealWithCallback(P, N, U, C)   [gas price = G]

3. Attacker Bob monitors the mempool, sees the pending transaction, and submits:
       revealWithCallback(P, N, U, C)   [gas price = G + 1 wei]

4. Bob's transaction is mined first:
   - revealHelper validates U and C ✓
   - Callback delivered to Alice ✓
   - Request cleared ✓

5. Fortuna's transaction is mined next:
   - findActiveRequest(P, N) → request slot is empty → REVERT
   - Fortuna loses gas, receives no fee credit for the fulfillment work.
```

The `userContribution` U is recoverable from the `RequestedWithCallback` event log; `providerContribution` C is available from the provider's public revelation API. No privileged access is required. [6](#0-5)
