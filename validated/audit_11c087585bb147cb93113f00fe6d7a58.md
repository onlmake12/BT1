### Title
Subscription Owner Cannot Cap Gas Price, Allowing Any Keeper to Drain Subscription Balance via Inflated `tx.gasprice` — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the keeper fee paid per `updatePriceFeeds` call is computed as `gasCost = (gasUsed + GAS_OVERHEAD) * tx.gasprice`. Because `tx.gasprice` is set freely by the keeper (the transaction submitter) and the `SubscriptionParams` struct contains no `maxGasPrice` or `maxKeeperFeePerUpdate` field, any unprivileged caller can submit an `updatePriceFeeds` transaction with an arbitrarily inflated gas price and drain a subscription's entire balance in a single call.

---

### Finding Description

`updatePriceFeeds` is a permissionless function — any address can call it as a keeper. [1](#0-0) 

Inside `_processFeesAndPayKeeper`, the keeper's payment is calculated using the raw `tx.gasprice` with no upper bound: [2](#0-1) 

The `SubscriptionParams` struct — the only configuration a subscription owner controls — contains no field to express a maximum acceptable gas price or maximum keeper fee per update: [3](#0-2) 

A keeper can therefore submit `updatePriceFeeds` with `tx.gasprice = 10_000 gwei` (or any value) on a chain where the real market gas price is `1 gwei`. The contract will compute `gasCost = gasUsed * 10_000_gwei` and transfer that amount from the subscription balance to `msg.sender`, with no check against any owner-specified limit.

---

### Impact Explanation

A subscription owner deposits ETH expecting it to last for many updates at prevailing gas prices. A malicious keeper can submit a single `updatePriceFeeds` transaction with an inflated `tx.gasprice` and extract the entire subscription balance in one call. The subscription owner has no on-chain mechanism to prevent this — there is no `maxGasPrice`, `maxKeeperFee`, or similar parameter in `SubscriptionParams`.

**Impact:** Complete loss of subscription funds (up to the full `balanceInWei`) per malicious keeper call. The subscription is effectively deactivated (balance drained to zero) without the owner's consent.

---

### Likelihood Explanation

`updatePriceFeeds` is callable by any unprivileged address with no whitelist or access control. A keeper is economically incentivized to maximize `tx.gasprice` to extract the maximum payment. On EVM chains, a transaction sender can set any `tx.gasprice` they wish — miners/validators accept high-fee transactions eagerly. This attack requires no special privilege, no leaked key, and no collusion: any EOA can execute it.

---

### Recommendation

Add a `maxGasPriceInWei` (or `maxKeeperFeePerUpdateInWei`) field to `SubscriptionParams` that the subscription owner sets at creation/update time. In `_processFeesAndPayKeeper`, enforce:

```solidity
require(tx.gasprice <= params.maxGasPriceInWei, "Gas price exceeds owner limit");
```

This mirrors how Uniswap/Sushiswap allow users to specify maximum slippage — the owner specifies the worst-case fee rate they accept, blocking keeper over-extraction.

---

### Proof of Concept

1. Alice creates a subscription with `balanceInWei = 1 ether`, heartbeat = 60 seconds.
2. Bob (attacker) waits for the heartbeat interval to elapse.
3. Bob calls `scheduler.updatePriceFeeds(aliceSubscriptionId, validUpdateData)` with `tx.gasprice = 100_000 gwei`.
4. Inside `_processFeesAndPayKeeper`:
   - `gasCost = (e.g. 500_000 gas + GAS_OVERHEAD) * 100_000 gwei ≈ several ETH`
   - If `gasCost > status.balanceInWei`, the call reverts with `InsufficientBalance` — but Bob can tune `tx.gasprice` to exactly drain the balance without reverting.
5. Bob receives Alice's entire subscription balance as keeper payment for a single legitimate update.
6. Alice's subscription balance is zero; she must re-fund or her subscription stops being served. [4](#0-3)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
