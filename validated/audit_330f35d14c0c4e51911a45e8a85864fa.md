### Title
Minimum Fee Rate Check Uses Transaction Size Instead of Weight, Allowing High-Cycle Transactions to Bypass Effective Fee Rate Enforcement — (`tx-pool/src/util.rs`)

---

### Summary

The `check_tx_fee` function enforces the minimum fee rate using only the transaction's serialized byte size, not the actual transaction weight (which accounts for both size and execution cycles). Any unprivileged transaction sender can craft a transaction with small serialized size but high cycle consumption, paying only the size-based minimum fee while having an effective fee rate far below the configured `min_fee_rate`. This is the direct CKB analog of the external report's pattern: a caller-controlled economic parameter (here, the transaction's size/cycle ratio) bypasses the intended economic floor, allowing pool pollution and sustained DoS against the tx-pool.

---

### Finding Description

**Root cause — `tx-pool/src/util.rs`, `check_tx_fee`, lines 44–52:** [1](#0-0) 

The minimum fee is computed as:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

`tx_size` is the raw serialized byte count. However, the actual transaction weight used everywhere else in the pool for ordering and fee-rate calculation is: [2](#0-1) 

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

The `TxEntry::fee_rate()` method — used for pool ordering and eviction — uses this weight: [4](#0-3) 

The code comment in `check_tx_fee` explicitly acknowledges the discrepancy:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [5](#0-4) 

**Concrete numbers:**

The default `max_tx_verify_cycles` is `TWO_IN_TWO_OUT_CYCLES * 20 = 70,000,000`. [6](#0-5) 

For a transaction with 200 serialized bytes and 70 M cycles:

| Metric | Value |
|---|---|
| Size-based min fee (at 1000 shannons/KB) | 200 shannons |
| Actual weight: `max(200, 70M × 0.000170571)` | **11,940 bytes** |
| Effective fee rate: `200 × 1000 / 11940` | **≈ 16.7 shannons/KW** |
| Configured `min_fee_rate` | 1000 shannons/KW |
| **Bypass factor** | **~60×** |

The transaction passes `check_tx_fee` (200 ≥ 200 shannons) but enters the pool with an effective fee rate 60× below the configured floor.

**Attack path:**

1. Attacker deploys a lock script that runs a tight loop consuming ~70 M cycles but has a small code footprint (~100–200 bytes on-chain).
2. Attacker creates UTXOs locked by this script.
3. Attacker crafts spending transactions: serialized size ≈ 200 bytes, fee = 200 shannons (passes size-based check), actual weight = 11,940 bytes.
4. Transactions are submitted via `send_transaction` RPC or relayed via P2P (`TransactionsProcess`). [7](#0-6) [8](#0-7) 

5. Each transaction passes `pre_check` → `check_tx_fee` (size-based) → enters the pool. [9](#0-8) 

6. Pool fills with transactions whose effective fee rate is ~16.7 shannons/KW.
7. Legitimate transactions are rejected with `PoolIsFull` once the 180 MB limit is reached. [10](#0-9) 

---

### Impact Explanation

The `min_fee_rate` configuration is the primary economic gate preventing pool spam. By exploiting the size-vs-weight discrepancy, an attacker can submit transactions that nominally satisfy the fee check but carry an effective fee rate up to ~60× below the floor. The pool fills with these transactions; legitimate users receive `PoolIsFull` rejections. Miners deprioritize the attacker's transactions (pool ordering uses actual weight), but the attacker can continuously re-submit to maintain saturation. At 200 shannons per transaction and 200 bytes per transaction, filling the 180 MB pool costs approximately **1.8 CKB in fees** — an extremely low-cost sustained DoS against the tx-pool admission layer.

---

### Likelihood Explanation

The entry path is fully unprivileged: any RPC caller or P2P peer can submit transactions. The attacker needs UTXOs locked by a high-cycle script, which requires an initial on-chain setup transaction but no privileged access. The cost per attack transaction is negligible. The `max_tx_verify_cycles` default of 70 M cycles is reachable by any script author. Likelihood is **medium-high** for a motivated attacker targeting network disruption.

---

### Recommendation

Replace the size-only weight in `check_tx_fee` with the actual transaction weight. Since cycles are unknown before verification, use the peer-declared cycles (available for remote transactions) as a conservative upper bound for the pre-verification fee check:

```rust
// For remote transactions, use declared_cycles to compute weight
let weight = get_transaction_weight(tx_size, declared_cycles.unwrap_or(0));
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

After verification, perform a second fee-rate check using the actual verified cycles to catch any remaining discrepancy. This mirrors the two-phase validation pattern already used for `DeclaredWrongCycles`.

---

### Proof of Concept

```
1. Deploy a RISC-V lock script that executes a tight loop for ~70M cycles.
   Script binary size: ~150 bytes.

2. Create a cell locked by this script with capacity = 61 CKB (minimum).

3. Craft a spending transaction:
   - 1 input (the cell above)
   - 1 output (same lock, capacity = input - 200 shannons)
   - Serialized size ≈ 200 bytes
   - Fee = 200 shannons

4. Submit via RPC: send_transaction(tx, "passthrough")

5. Observe: transaction accepted into pool.
   check_tx_fee: min_fee = 1000 * 200 / 1000 = 200 shannons ✓
   Actual weight: max(200, 70M * 0.000170571) = 11,940 bytes
   Effective fee rate: 200 * 1000 / 11940 ≈ 16.7 shannons/KW << 1000 shannons/KW

6. Repeat with 900,000 UTXOs (or use RBF-exempt chains up to max_ancestors_count=25)
   to fill the 180 MB pool at a total cost of ~1.8 CKB.

7. Legitimate transactions now receive PoolIsFull (-1106) errors.
```

    pub fn execute(self) -> Status {
        let shared_state = self.relayer.shared().state();
        let txs: Vec<(TransactionView, Cycle)> = {
            // ignore the tx if it's already known or it has never been requested before
            let mut tx_filter = shared_state.tx_filter();
            tx_filter.remove_expired();
            let unknown_tx_hashes = shared_state.unknown_tx_hashes();

            self.message
                .transactions()
                .iter()
                .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
                .filter(|(tx, _)| {
                    !tx_filter.contains(&tx.hash())
                        && unknown_tx_hashes
                            .get_priority(&tx.hash())
                            .map(|priority| priority.requesting_peer() == Some(self.peer))
                            .unwrap_or_default()
                })
                .collect()
        };

        if txs.is_empty() {
            return Status::ok();
        }

        let max_block_cycles = self.relayer.shared().consensus().max_block_cycles();
        if txs
            .iter()
            .any(|(_, declared_cycles)| declared_cycles > &max_block_cycles)
        {
            self.nc.ban_peer(
                self.peer,
                DEFAULT_BAN_TIME,
                String::from("relay declared cycles greater than max_block_cycles"),
            );
            return Status::ok();
        }

        shared_state.mark_as_known_txs(txs.iter().map(|(tx, _)| tx.hash()));

        let tx_pool = self.relayer.shared.shared().tx_pool_controller().clone();
        let peer = self.peer;
        self.relayer
            .shared
            .shared()
            .async_handle()
            .spawn(async move {
                for (tx, declared_cycles) in txs {
                    if let Err(e) = tx_pool
                        .submit_remote_tx(tx.clone(), declared_cycles, peer)
                        .await
                    {
                        error!("submit_tx error {}", e);
                    }
                }
            });

        Status::ok()
    }

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
