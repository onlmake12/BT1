Audit Report

## Title
Duplicate-Index Amplification in `GetBlockTransactionsProcess::execute` Enables Memory Exhaustion and Node Crash — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary
`GetBlockTransactionsProcess::execute` accepts up to 32,767 transaction indexes from a remote peer but performs no deduplication and applies no byte-size cap before serializing and transmitting the response. An unprivileged peer can send 32,767 copies of the same valid index, forcing the node to clone, allocate, and serialize the same transaction 32,767 times into a single unbounded `BlockTransactions` message, causing memory exhaustion and potential node crash.

## Finding Description
The handler enforces only a count check against `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767): [1](#0-0) 

After passing this check, each index is mapped directly to a `.cloned()` transaction with no deduplication: [2](#0-1) 

All collected transactions — including 32,767 copies of the same one — are packed into a single `BlockTransactions` message and sent with no byte cap: [3](#0-2) 

The file imports only `MAX_RELAY_TXS_NUM_PER_BATCH`; `MAX_RELAY_TXS_BYTES_PER_BATCH` is never imported or applied: [4](#0-3) 

By contrast, `GetTransactionsProcess` deduplicates via a `HashSet` and rejects duplicates with `StatusCode::RequestDuplicate`: [5](#0-4) 

It also enforces `MAX_RELAY_TXS_BYTES_PER_BATCH` before sending: [6](#0-5) 

`GetBlockProposalProcess` applies the same two guards: [7](#0-6) [8](#0-7) 

`MAX_RELAY_TXS_NUM_PER_BATCH` is 32,767: [9](#0-8) 

## Impact Explanation
A single 500 KB transaction referenced 32,767 times yields ~16 GB of allocation and serialization per request. Even a modest 10 KB transaction produces ~320 MB per request. The memory exhaustion occurs during the `collect::<Vec<_>>()` call and subsequent `build()` serialization — before any network-layer send limit could intervene. Multiple peers issuing this simultaneously can exhaust node memory and crash the process. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**. If coordinated across many peers, it escalates to **Critical — crash the whole CKB network**.

## Likelihood Explanation
The attacker requires only a standard P2P connection and knowledge of any block hash in the target node's store — both trivially obtained from the public chain. No privilege, key, or proof-of-work is required. The `GetBlockTransactions` message is an unauthenticated relay protocol message any connected peer may send. The attack is repeatable, parallelizable, and has no authentication barrier.

## Recommendation
Apply both fixes already present in sibling handlers:
1. **Deduplicate indexes** before lookup — collect into a `HashSet<u32>` and return `StatusCode::RequestDuplicate` if `set.len() != indexes.len()`.
2. **Enforce `MAX_RELAY_TXS_BYTES_PER_BATCH`** — accumulate serialized size and split or truncate the response when the cap is exceeded, mirroring `GetTransactionsProcess` lines 87–101.

## Proof of Concept
```rust
// Unit test sketch
let block = build_block_with_one_tx(500_000); // 500 KB transaction
store.insert_block(&block);

// Send 32,767 copies of index 0 — passes the count check (32767 <= 32767)
let indexes: Vec<u32> = vec![0u32; 32767];
let msg = build_get_block_transactions(block.hash(), indexes);
let response = execute_handler(msg).await;

// Response contains 32767 × 500 KB ≈ 16 GB of serialized data
// Node OOMs or crashes during collection/serialization
assert!(response.serialized_size() > MAX_RELAY_TXS_BYTES_PER_BATCH);
```
