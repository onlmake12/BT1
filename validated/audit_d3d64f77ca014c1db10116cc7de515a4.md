Audit Report

## Title
`ScriptHashTypeVerifier` Omits Type-Script Hash-Type Check on Transaction Outputs — (`File: verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates only the lock script's `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`. The type script on each output is never inspected. A transaction carrying a structurally valid but consensus-forbidden `hash_type` (e.g., `Data3 = 0x06`) on an output type script passes all non-contextual checks and is only rejected deep inside `ContextualTransactionVerifier`, after the input UTXO has been resolved and lock-script execution has begun. Because the transaction is never committed, the input UTXO is never consumed, enabling unlimited free resubmission.

## Finding Description

`ScriptHashTypeVerifier::verify()` loops over outputs and calls `output.lock().hash_type()` exclusively; there is no branch for `output.type_().to_opt()`. [1](#0-0) 

`ENABLED_SCRIPT_HASH_TYPE` permits only `{0, 1, 2, 4}`. [2](#0-1) 

The low-level structural gate `ScriptHashType::verify_value` accepts any even byte or `1` (`v.is_multiple_of(2) || v == 1`), so `Data3 = 6` passes it. [3](#0-2) 

This is confirmed by the existing unit test: `ScriptHashType::try_from(6)` returns `Ok(ScriptHashType::Data3)`. [4](#0-3) 

`CellOutputReader::check_data()` does check both lock and type scripts, but only via `verify_value`, not against `ENABLED_SCRIPT_HASH_TYPE`, so `Data3` passes it too. [5](#0-4) 

The transaction therefore clears `NonContextualTransactionVerifier::verify()` entirely. [6](#0-5) 

The struct comment explicitly documents only "output lock hash type within enabled range," confirming the type-script path was never added. [7](#0-6) 

Only when `ContextualTransactionVerifier` later calls `select_version()` does the catch-all arm fire and return `InvalidScriptHashType`. [8](#0-7) 

`SgData::new()` calls `select_version()` at script-group setup time, meaning the input UTXO has already been resolved and lock-script execution may have already begun before the type-script error surfaces. [9](#0-8) 

The existing test `test_not_enabled_hash_type_output_lock` covers only the lock-script case, leaving the type-script gap untested. [10](#0-9) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Every node that receives the crafted transaction runs `non_contextual_verify` (passes), resolves the input UTXO, and enters `verify_rtx` / `ContextualTransactionVerifier` before the rejection fires. [11](#0-10) [12](#0-11) 

Because the transaction is rejected and never committed, the input UTXO is never spent. The attacker can resubmit the same UTXO in a new transaction (with trivially varied output data to change the tx hash) indefinitely, forcing repeated contextual-verification CPU expenditure across the entire network at zero net cost.

## Likelihood Explanation

No privilege, key material, or miner cooperation is required. The only precondition is owning one live UTXO. Setting `hash_type = 0x06` on a `Script` struct is a single-byte change; `verify_value` accepts it; the non-contextual verifier passes it. The attack is fully repeatable since the UTXO is never consumed.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script of each output when present, mirroring the existing lock-script logic:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto::<ScriptHashType>` + `ENABLED_SCRIPT_HASH_TYPE` logic. [1](#0-0) 

Update the `NonContextualTransactionVerifier` struct comment to reflect that both lock and type script hash types are checked. [7](#0-6) 

Add a corresponding test analogous to `test_not_enabled_hash_type_output_lock` that places `Data3` on the type script and expects `ScriptHashTypeNotPermitted`. [10](#0-9) 

## Proof of Concept

1. Build a `TransactionView` with one input (any live UTXO) and one output: lock script uses `ScriptHashType::Data` (permitted); type script uses `hash_type = ScriptHashType::Data3` (value `0x06`).
2. Call `ScriptHashTypeVerifier::new(&tx).verify()` — returns `Ok(())` because only the lock script is inspected.
3. Call `NonContextualTransactionVerifier::new(&tx, consensus).verify()` — also returns `Ok(())`.
4. Submit via `send_transaction` RPC; the transaction enters `verify_rtx` and `ContextualTransactionVerifier`.
5. `select_version()` hits the catch-all arm for `Data3` and returns `Err(InvalidScriptHashType)`.
6. The input UTXO is not spent. Repeat from step 1 with a new output value (to vary the tx hash) — the attack loops indefinitely at zero net cost.

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
