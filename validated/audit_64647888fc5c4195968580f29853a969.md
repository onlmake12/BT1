### Title
Missing Event Emission in `addFunds` and `withdrawFunds` — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract contains two sensitive financial functions — `addFunds` and `withdrawFunds` — that transfer ETH into and out of subscription balances without emitting any events. Off-chain clients, monitoring systems, and indexers have no on-chain signal to track these fund movements.

---

### Finding Description

In `Scheduler.sol`, the `addFunds` function (lines 602–628) accepts ETH from any caller and credits it to a subscription's `balanceInWei`, while `withdrawFunds` (lines 630–662) deducts from `balanceInWei` and sends ETH to the subscription manager. Neither function emits an event. [1](#0-0) [2](#0-1) 

By contrast, every other state-changing function in the Scheduler ecosystem emits an event: `createSubscription` emits `SubscriptionCreated`, `updateSubscription` emits `SubscriptionUpdated`/`SubscriptionActivated`/`SubscriptionDeactivated`, `updatePriceFeeds` emits `PricesUpdated`, and all governance setters in `SchedulerGovernance.sol` emit their respective events. [3](#0-2) [4](#0-3) 

The `addFunds` function has **no access control** — any address may call it for any subscription ID. The `withdrawFunds` function is restricted to the subscription manager via the `onlyManager` modifier, but still emits nothing on success. [5](#0-4) 

---

### Impact Explanation

- Off-chain monitoring infrastructure (explorers, dashboards, keeper bots) cannot reconstruct the balance history of a subscription by listening to contract events alone.
- Subscription managers and protocol operators cannot receive event-driven alerts when funds are deposited or withdrawn.
- Any anomalous or unexpected withdrawal by a manager goes undetected by event-based watchers, delaying incident response.
- The `addFunds` path is particularly notable: because it is permissionless, third parties can silently top up any subscription's balance, and this is entirely invisible to event listeners.

---

### Likelihood Explanation

Both functions are externally callable on every deployed `SchedulerUpgradeable` instance. `addFunds` requires no privilege at all; `withdrawFunds` requires only that the caller is the subscription manager, a role held by the original creator of each subscription. Any subscription manager — an unprivileged user — can trigger the missing-event path on every withdrawal. The likelihood is **high** that these paths are exercised in normal protocol operation, since fund management is a routine user action.

---

### Recommendation

Add dedicated events and emit them in both functions. For example:

```solidity
event FundsAdded(uint256 indexed subscriptionId, address indexed sender, uint256 amount);
event FundsWithdrawn(uint256 indexed subscriptionId, address indexed manager, uint256 amount);
```

In `addFunds`:
```solidity
status.balanceInWei += msg.value;
emit FundsAdded(subscriptionId, msg.sender, msg.value);
```

In `withdrawFunds`:
```solidity
status.balanceInWei -= amount;
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "Failed to send funds");
emit FundsWithdrawn(subscriptionId, msg.sender, amount);
```

This mirrors the pattern already established by `FeesWithdrawn` in the `Echo` contract. [6](#0-5) 

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` (or use a live instance).
2. Create a subscription via `createSubscription{value: minBalance}(params)` — this emits `SubscriptionCreated`.
3. Call `addFunds{value: 1 ether}(subscriptionId)` — **no event is emitted**; the balance silently increases.
4. As the subscription manager, call `withdrawFunds(subscriptionId, 1 ether)` — **no event is emitted**; ETH is transferred out silently.
5. Query the event log for the contract: the deposit and withdrawal are completely absent, while the initial creation is visible. An off-chain indexer tracking subscription balances purely from events will show an incorrect balance.

```text
    function addFunds(uint256 subscriptionId) external payable override {
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];

        if (!params.isActive) {
            revert SchedulerErrors.InactiveSubscription();
        }

        // Check deposit limit for permanent subscriptions
        if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
            revert SchedulerErrors.MaxDepositLimitExceeded();
        }

        status.balanceInWei += msg.value;

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }
    }
```

```text
    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
```
