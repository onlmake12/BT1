### Title
Discarded `Status` Return Value from Mid-Loop `send_relay_transactions` Call Silently Swallows Network Errors - (File: `sync/src/relayer/get_transactions_process.rs`)

### Summary

In `GetTransactionsProcess::execute`, when responding to a peer's `GetRelayTransactions` message with a batch of transactions that exceeds `MAX_RELAY_TXS_BYTES_PER_BATCH`, the function splits the response into multiple sends. The intermediate batch send at line 91–92 discards the returned `Status` entirely, while the final batch send at line 100 correctly uses the `attempt!` macro to propagate errors. This is a direct analog to the reported pattern: a callee returns a meaningful status/data value that the caller silently drops.

---

### Finding Description

`GetTransactionsProcess::execute` in `sync/src/relayer/get_transactions_process.rs` handles a peer's request for relay transactions. When the total size of transactions to send exceeds `MAX_RELAY_TXS_BYTES_PER_BATCH`, the function splits the response into multiple calls to `send_relay_transactions`. The relevant loop is:

```rust
for tx in transactions {
    if relay_bytes + tx.total_size() > MAX_RELAY_TXS_BYTES_PER_BATCH {
        self.send_relay_transactions(std::mem::take(&mut relay_txs))
            .await;                          // ← Status return value DISCARDED
        relay_bytes = tx.total_size();
    } else {
        relay_bytes += tx.total_size();
    }
    relay_txs.push(tx);
}
if !relay_txs.is_empty() {
    attempt!(self.send_relay_transactions(relay_txs).await);  // ← Status IS handled
}
```

`send_relay_transactions` returns a `Status` value produced by `async_send_message_to`, which in turn calls `async_send_message`. That function returns `StatusCode::Network.with_context(...)` on failure. The `attempt!` macro (defined in `sync/src/status.rs`) causes early return with the error status when the call fails. However, the mid-loop call at lines 91–92 drops the returned `Status` entirely — no `attempt!`, no `if !status.is_ok()` check, no logging at the call site. [1](#0-0)

`async_send_message_to` delegates to `async_send_message`, which returns a non-ok `Status` on network failure: [3](#0-2) 

---

### Impact Explanation

When a requesting peer asks for enough transactions to trigger batch splitting (total size > `MAX_RELAY_TXS_BYTES_PER_BATCH`), any network-level failure on an intermediate batch is silently swallowed. The function proceeds to attempt subsequent batches and ultimately returns `Status::ok()` to the relay protocol handler regardless of whether intermediate sends succeeded. The relay handler therefore cannot take corrective action (e.g., disconnecting the misbehaving or unreachable peer, retrying, or logging a protocol-level error). Transaction relay to the requesting peer is partially or fully lost without any protocol-level indication of failure.

---

### Likelihood Explanation

Any unprivileged peer can send a `GetRelayTransactions` message requesting enough transaction hashes to cause the response to exceed `MAX_RELAY_TXS_BYTES_PER_BATCH`. This is a normal protocol operation. Network errors (peer disconnect, buffer full, OS-level send failure) are realistic and occur in production. No special privileges, keys, or majority hashpower are required.

---

### Recommendation

Apply the `attempt!` macro consistently to the mid-loop call, matching the treatment of the final batch:

```rust
for tx in transactions {
    if relay_bytes + tx.total_size() > MAX_RELAY_TXS_BYTES_PER_BATCH {
        attempt!(self.send_relay_transactions(std::mem::take(&mut relay_txs)).await);
        relay_bytes = tx.total_size();
    } else {
        relay_bytes += tx.total_size();
    }
    relay_txs.push(tx);
}
if !relay_txs.is_empty() {
    attempt!(self.send_relay_transactions(relay_txs).await);
}
```

This ensures that a network failure on any batch causes the handler to return a non-ok `Status`, allowing the relay protocol layer to respond appropriately.

---

### Proof of Concept

1. Connect an unprivileged peer to a CKB node.
2. Populate the node's tx-pool with enough transactions such that their combined relay size exceeds `MAX_RELAY_TXS_BYTES_PER_BATCH`.
3. Send a `GetRelayTransactions` message requesting those transaction hashes.
4. Simulate or induce a network send failure on the first batch (e.g., by closing the connection mid-flight or exhausting the send buffer).
5. Observe that `execute` returns `Status::ok()` despite the failure, and that the relay protocol handler takes no corrective action. The mid-loop `send_relay_transactions` call at line 91–92 produces a `StatusCode::Network` error that is never surfaced. [4](#0-3)

pub(crate) async fn async_send_message<Message: Entity>(
    protocol_id: ProtocolId,
    nc: &Arc<dyn CKBProtocolContext + Sync>,
    peer_index: PeerIndex,
    message: &Message,
) -> Status {
    // ignore Error return, only happens on shutdown case
    if let Err(err) = nc
        .async_send_message(protocol_id, peer_index, message.as_bytes())
        .await
    {
        let name = message_name(protocol_id, message);
        let error_message = format!("nc.send_message {name}, error: {err:?}");
        ckb_logger::error!("{}", error_message);
        return StatusCode::Network.with_context(error_message);
    }

    let bytes = message.as_bytes().len() as u64;
    let item_name = item_name(protocol_id, message);
    let protocol_name = protocol_name(protocol_id);
    metric_ckb_message_bytes(
        MetricDirection::Out,
        &protocol_name,
        &item_name,
        None,
        bytes,
    );

    Status::ok()
}
