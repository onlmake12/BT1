### Title
`RecentReject::put` Never Increments `total_keys_num`, Allowing Unbounded DB Growth via RBF-Driven `Reject::Invalidated` Spam — (`tx-pool/src/component/recent_reject.rs`)

---

### Summary

`RecentReject::put` computes `self.total_keys_num + 1` to check against `count_limit`, but **never writes the result back to `self.total_keys_num`**. The field stays at its construction-time estimate forever. Because `Reject::Invalidated` passes `should_recorded()`, an attacker who repeatedly submits RBF replacement pairs can fill the `recent_reject` RocksDB store without bound, violating the `count_limit` invariant and causing unbounded disk usage and I/O load.

---

### Finding Description

**Root cause — `put()` never updates `self.total_keys_num`:** [1](#0-0) 

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ...
    self.db.put(&shard, hash_slice, json_string)?;   // key written to DB

    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
        // `total_keys_num` is a NEW LOCAL BINDING — self.total_keys_num is NEVER updated
        if total_keys_num > self.count_limit {
            self.shrink()?;
        }
    } else {
        self.shrink()?;
    }
    Ok(())
}
```

The `if let Some(total_keys_num) = self.total_keys_num.checked_add(1)` pattern creates a **local shadow variable**. `self.total_keys_num` is never assigned the incremented value. Starting from a fresh DB where `self.total_keys_num == 0`, every call evaluates `0 + 1 > count_limit`, which is `false` for any `count_limit ≥ 1`. `shrink()` is never triggered. The DB grows without bound.

`shrink()` does correctly update `self.total_keys_num` from the DB estimate — but it is never reached: [2](#0-1) 

**`Reject::Invalidated` passes `should_recorded()`:** [3](#0-2) 

`should_recorded()` returns `true` for every variant except `Reject::Duplicated`. `Reject::Invalidated` is not excluded.

**RBF eviction path generates `Reject::Invalidated` and records it:**

When a new transaction evicts a conflicting one via RBF, `submit_entry` fires `Reject::Invalidated` through the registered reject callback: [4](#0-3) 

The registered callback unconditionally calls `recent_reject.put()` for any reject that passes `should_recorded()`: [5](#0-4) 

---

### Impact Explanation

Every RBF replacement inserts a new unique key into the `recent_reject` RocksDB store. Because `self.total_keys_num` is never incremented, the `count_limit` guard is never triggered. The store grows without bound, consuming unbounded disk space and generating increasing I/O load on the node. This matches the **High (10001–15000 pt)** scope: network congestion / node resource exhaustion at low attacker cost.

---

### Likelihood Explanation

RBF is an enabled production feature. An attacker needs only one valid UTXO. They submit tx A, then submit tx B spending the same input with a marginally higher fee rate (the minimum RBF fee bump). A is evicted as `Reject::Invalidated` and recorded. The attacker then submits tx C replacing B, and so on. Since none of these transactions are ever committed to a block, **no fees are ever actually paid on-chain**. The only cost is bandwidth. With multiple UTXOs the attack scales linearly.

---

### Recommendation

In `put()`, assign the incremented value back to `self.total_keys_num` before the limit check:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    // ...
    self.db.put(&shard, hash_slice, json_string)?;

    match self.total_keys_num.checked_add(1) {
        Some(new_total) => {
            self.total_keys_num = new_total;   // ← missing assignment
            if new_total > self.count_limit {
                self.shrink()?;
            }
        }
        None => {
            self.shrink()?;
        }
    }
    Ok(())
}
```

### Proof of Concept

```rust
// Pseudocode — run against a local CKB node with RBF enabled
let count_limit = node.config.keep_rejected_tx_hashes_count; // e.g. 50_000
let utxo = get_live_cell();

for i in 0..(count_limit * 10) {
    // Each iteration: submit tx with fee = base + i, spending `utxo`
    // The previous tx is evicted as Reject::Invalidated and recorded
    node.rpc.send_transaction(build_tx(utxo, fee = BASE_FEE + i));
}

// Assert: recent_reject DB key count >> count_limit
let actual = node.rpc.get_total_recent_reject_num();
assert!(actual > count_limit * 5,
    "DB not bounded: {} entries vs limit {}", actual, count_limit);
```

Because `self.total_keys_num` is never incremented, `shrink()` is never called, and `actual` will far exceed `count_limit`.
