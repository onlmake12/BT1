### Title
Peer Store Monopolization via Single-/16 Flood: `check_purge` `take(len/2)` Integer Division Yields Zero Evictions — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
`check_purge`'s second eviction phase uses `take(len / 2)` where `len` is the number of distinct network groups. When all stored addresses belong to a single /16, `len = 1` and `1 / 2 = 0` in integer division, so no candidates are selected and `EvictionFailed` is returned. An unprivileged peer can trigger this by sending crafted discovery `Nodes` messages that fill the 16384-entry peer store with addresses from a single /16.

### Finding Description
In `check_purge` (`peer_store_impl.rs` lines 327–404):

- **Phase 1** collects addresses where `!is_connectable(now_ms)`. Freshly-added addresses have `attempts_count = 0 < ADDR_MAX_RETRIES (3)` and `last_connected_at_ms = 0`, so `is_connectable` returns `true` for all of them. Phase 1 yields nothing.
- **Phase 2** groups addresses by `Group` (first two IPv4 octets), then calls `.take(len / 2)`. With `len = 1`, this is `.take(0)`, producing an empty iterator. `candidate_peers` is empty, and the function returns `Err(PeerStoreError::EvictionFailed)`.

The attacker entry point is the discovery protocol: `DiscoveryAddressManager::add_new_addrs` calls `peer_store.add_addr` for each address in a received `Nodes` message with no per-subnet admission limit. A single connected peer can send ~6 non-announce `Nodes` messages (each carrying up to 3000 addresses) to fill the 16384-slot store entirely with addresses from one /16.

### Impact Explanation
Once the store is full and `check_purge` permanently returns `EvictionFailed`, every subsequent `add_addr` call fails. The victim node cannot add any new peer addresses discovered via the P2P network. Its feeler and outbound connection pools will only draw from the attacker-supplied addresses (which may be unresponsive or attacker-controlled), effectively isolating the node from honest peers it has not already connected to. This degrades peer diversity and can contribute to eclipse-style isolation over time.

### Likelihood Explanation
The attacker only needs one accepted P2P connection to the victim (inbound or outbound). No PoW, no privileged role, no key material. The fill requires ~6 messages. The /16 can be any globally routable prefix (e.g., `1.2.0.0/16`), bypassing the `is_reachable` filter. The bug is deterministic and locally reproducible.

### Recommendation
Replace `take(len / 2)` with logic that always selects at least one group when the store is full, for example `take(len / 2).max(1)`, or restructure the eviction to guarantee at least one address is removed regardless of group count. Additionally, enforce a per-/16 admission cap in `add_addr` to prevent any single subnet from monopolizing the store.

        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr

        }
