### Title
Predictable On-Chain PRNG User Contribution Allows Entropy Outcome Manipulation — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `random()` internal function in `Entropy.sol`, used as the user contribution for three `requestV2` overloads, derives its output exclusively from publicly readable or attacker-controlled values: `block.timestamp`, `block.prevrandao`, `msg.sender`, and `_state.seed`. All four inputs are known to the caller before the transaction is mined, making the user contribution — and therefore the final random number — fully predictable. Any unprivileged Entropy user calling the PRNG-based `requestV2` variants can simulate the outcome off-chain and selectively revert unfavorable results.

---

### Finding Description

Three `requestV2` overloads in `Entropy.sol` generate the user contribution by calling the internal `random()` function instead of accepting a caller-supplied secret:

```solidity
function requestV2() external payable override returns (uint64 assignedSequenceNumber) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), 0);
}

function requestV2(uint32 gasLimit) external payable override returns (uint64 assignedSequenceNumber) {
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), gasLimit);
}

function requestV2(address provider, uint32 gasLimit) external payable override returns (uint64 assignedSequenceNumber) {
    assignedSequenceNumber = requestV2(provider, random(), gasLimit);
}
``` [1](#0-0) 

The `random()` function is:

```solidity
function random() internal returns (bytes32) {
    _state.seed = keccak256(
        abi.encodePacked(
            block.timestamp,
            block.prevrandao,
            msg.sender,
            _state.seed
        )
    );
    return _state.seed;
}
``` [2](#0-1) 

Every input is observable before the transaction executes:

| Input | Why it is known |
|---|---|
| `block.timestamp` | Set by the block proposer; known when building a transaction for a specific block |
| `block.prevrandao` | The RANDAO value of the *previous* block; fully public before the current block |
| `msg.sender` | The attacker's own address |
| `_state.seed` | A plain storage variable readable via `eth_getStorageAt` at any time | [3](#0-2) 

The final random number is `r = keccak256(userContribution, providerContribution, 0)`. [4](#0-3) 

The provider's `providerContribution` is the next value in their public hash chain, which is deterministically derivable from the on-chain `currentCommitment` and `currentCommitmentSequenceNumber`. [5](#0-4) 

Because both halves of the final hash are known before the request is submitted, the attacker can compute `r` off-chain, decide whether it is favorable, and only submit (or let land) the transaction when the result is desirable.

The `IEntropyV2` NatSpec acknowledges a weaker version of this risk (validator + provider collusion), but the actual exposure is stronger: the *caller alone* can predict the outcome with no trusted-party cooperation required. [6](#0-5) 

---

### Impact Explanation

Any on-chain application that calls `requestV2()`, `requestV2(uint32)`, or `requestV2(address, uint32)` — the three PRNG-based overloads — is vulnerable. The attacker can:

1. Read `_state.seed` from storage.
2. Compute `userContribution = keccak256(block.timestamp, block.prevrandao, attacker_address, seed)`.
3. Derive `providerContribution` from the provider's public hash chain.
4. Compute `r = keccak256(userContribution, providerContribution, 0)`.
5. If `r` is unfavorable, revert (via MEV infrastructure such as Flashbots) at zero on-chain cost.
6. Repeat until a favorable block produces the desired outcome.

This completely breaks the security guarantee of the commit-reveal protocol for these overloads. Games, lotteries, NFT mints, and DeFi protocols relying on these variants receive no meaningful randomness.

---

### Likelihood Explanation

- The three vulnerable overloads are the *recommended* entry points in the official documentation and the `CoinFlip` quickstart example, making them the most commonly used path.
- All required inputs are publicly readable with standard RPC calls (`eth_getStorageAt`, `eth_getBlockByNumber`).
- MEV infrastructure makes selective reversion free for the attacker.
- No privileged access, leaked keys, or third-party cooperation is required.

---

### Recommendation

Remove the PRNG-based `requestV2` overloads, or replace `random()` with a source that is not predictable by the caller before the transaction is included. Concretely:

- Require callers to supply their own `userRandomNumber` (as the full-control `requestV2(address, bytes32, uint32)` overload already does), generated off-chain with a CSPRNG and kept secret until reveal.
- If a no-argument convenience overload is desired, generate the user contribution from a value that is not known to the caller at request time, such as the hash of the *current* block's RANDAO (available only after the block is sealed), combined with a per-request nonce.

### Proof of Concept

```solidity
// Attacker contract
contract EntropyManipulator {
    IEntropyV2 entropy;
    bytes32 targetOutcome; // e.g., winning lottery number

    constructor(address _entropy, bytes32 _target) {
        entropy = IEntropyV2(_entropy);
        targetOutcome = _target;
    }

    function attack(address provider) external payable {
        // 1. Read current seed from storage slot (known layout)
        bytes32 currentSeed = /* eth_getStorageAt(_entropy, seedSlot) */;

        // 2. Compute the user contribution that random() will produce
        bytes32 userContribution = keccak256(
            abi.encodePacked(block.timestamp, block.prevrandao, address(this), currentSeed)
        );

        // 3. Derive provider's next revelation from their public hash chain
        bytes32 providerContribution = deriveProviderContribution(provider);

        // 4. Compute final random number
        bytes32 r = keccak256(abi.encodePacked(userContribution, providerContribution, bytes32(0)));

        // 5. Only proceed if outcome is favorable; otherwise revert (free via Flashbots)
        require(r == targetOutcome, "unfavorable outcome");

        uint256 fee = entropy.getFeeV2();
        entropy.requestV2{value: fee}(); // commits the predictable userContribution
    }
}
```

The `userContribution` emitted in the `Requested` event will match the pre-computed value, confirming the prediction. The attacker repeats across blocks until the desired `r` is produced.
