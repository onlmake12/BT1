Audit Report

## Title
Unbounded Resource Exhaustion via Crafted `GetBlocksProof`/`GetTransactionsProof` Without Peer Ban — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`, `util/light-client-protocol-server/src/lib.rs`)

## Summary

`GetBlocksProofProcess::execute` partitions requested block hashes by `is_main_chain` only, with no check that found blocks have `block_number ≤ last_block.number()`. When `reply_proof` builds an MMR scoped to `last_block.number() - 1`, positions for later blocks are out of range, causing `mmr.gen_proof` to return `Err` and the handler to return `StatusCode::InternalError` (500). Because `should_ban()` only covers `400..500`, the peer is never banned and can repeat this indefinitely, forcing up to ~4000 DB reads and a failed MMR computation per request with no bound.

## Finding Description

**Step 1 — Missing ancestor check.**
`GetBlocksProofProcess::execute` partitions `block_hashes` using only `is_main_chain`: [1](#0-0) 

There is no check that `header.number() ≤ last_block.number()`. Positions are then computed directly from each found block's actual number: [2](#0-1) 

**Step 2 — MMR scoped to `last_block.number() - 1`.**
`reply_proof` builds the MMR using only the range up to `last_block.number() - 1`: [3](#0-2) 

If `last_hash` is block 1 (MMR size = 1), positions for blocks 500–1499 are entirely out of range. `mmr.gen_proof(items_positions)` returns `Err`, and `StatusCode::InternalError` (500) is returned.

**Step 3 — `should_ban()` does not cover 5xx.**
The ban check uses `(400..500)`, which excludes 500: [4](#0-3) 

The peer receives only a `warn!` log and is never banned: [5](#0-4) 

**Step 4 — Identical flaw in `GetTransactionsProof`.**
The same partition-without-ancestor-check pattern exists: [6](#0-5) 

With positions pushed from the actual block number: [7](#0-6) 

**Step 5 — No rate limiting.**
There is no per-peer rate limiting anywhere in `received` or `try_process`: [8](#0-7) 

The per-request limit of 1000 hashes is enforced, but this is the attacker's lever, not a mitigation: [9](#0-8) 

## Impact Explanation

Each crafted request forces the server to perform up to 1000 `is_main_chain` DB lookups, up to 1000 `get_block_header` reads, up to 1000 `get_block_uncles` reads, up to 1000 `get_block_extension` reads, one MMR root computation, and one failed `gen_proof` call — all discarded. Because the peer is never banned and there is no rate limit, a single connection can sustain this at maximum throughput, causing persistent CPU and I/O exhaustion on the full node. This maps to **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation

The attack requires no privileges, no PoW, and no key material. Block hashes are publicly available from any chain explorer or peer. The `GetBlocksProof` message format is part of the documented light client protocol. Any peer that can connect to a node running the light client protocol server can trigger this immediately and repeat it indefinitely.

## Recommendation

1. **Add an ancestor check** in both `GetBlocksProofProcess::execute` and `GetTransactionsProofProcess::execute`: after partitioning, verify every block in `found` satisfies `block_number ≤ last_block.number()`. Blocks violating this should be moved to `missing` or cause the handler to return `StatusCode::InvalidRequest` (410), which triggers a ban.
2. **Extend `should_ban()`** to cover `InternalError` (500) when it is caused by client-supplied out-of-range data, or introduce a distinct 4xx code for "client data caused server-side failure."
3. **Add per-peer rate limiting** in the light client protocol handler as defense-in-depth.

## Proof of Concept

```
1. Attacker connects as a light client peer to a node with light client protocol enabled.
2. Attacker obtains block_1_hash (block number 1) and block_500_hash..block_1499_hash
   (all on the main chain, numbers 500–1499) from any public source.
3. Attacker sends:
     GetBlocksProof {
       last_hash:    block_1_hash,
       block_hashes: [block_500_hash, ..., block_1499_hash]  // 1000 hashes
     }
4. Server: all 1000 hashes pass is_main_chain → found = [500..1499]
5. positions = [leaf_index_to_pos(500), ..., leaf_index_to_pos(1499)]
6. reply_proof builds MMR for block 0 (size = 1, since last_block.number()-1 = 0)
7. mmr.gen_proof(positions) → Err → StatusCode::InternalError (500)
8. should_ban() → None (500 ∉ 400..500) → peer NOT banned
9. Repeat from step 3 indefinitely; ~4000 DB reads + failed MMR per iteration
```

            warn!("process {} from {}; result is {}", item_name, peer, status);
