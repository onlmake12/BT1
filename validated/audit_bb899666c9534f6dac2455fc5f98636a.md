### Title
Ancestor Descendant-Stats Permanently Inflated After `remove_entry_and_descendants` Due to Pre-Cleared Links — (`tx-pool/src/component/pool_map.rs`)

### Summary

`remove_entry_and_descendants` pre-removes all link entries for the entire subtree **before** calling `remove_entry` on each member. Because `update_ancestors_index_key` (called inside `remove_entry`) derives the ancestor set from the live links map, it finds an empty set for every removed transaction and never calls `sub_descendant_weight` on any ancestor that survives outside the removed subtree. Those survivors' `descendants_fee`, `descendants_cycles`, `descendants_count`, and `descendants_size` fields — and therefore their stored `EvictKey` — remain permanently inflated.

### Finding Description

**Step 1 — Pre-clear all links.**

`remove_entry_and_descendants` first iterates over the root and every descendant and calls `remove_entry_links` on each: [1](#0-0) 

`remove_entry_links` removes the entry from its parents' children sets, removes it from its children's parents sets, and then deletes the entry from `self.links` entirely: [2](#0-1) 

After this loop, **every node in the removed subtree is gone from `self.links`**, including the root A. Crucially, A's parent P (which is *not* being removed) has already had A removed from its children set, but P's own `TxEntry` fields (`descendants_fee`, etc.) have not been touched.

**Step 2 — `remove_entry` calls `update_ancestors_index_key` on already-cleared entries.**

For each removed id, `remove_entry` is called: [3](#0-2) 

Inside `remove_entry`, `update_ancestors_index_key` is called first (line 242). It queries `self.links.calc_ancestors(&child.proposal_short_id())`: [4](#0-3) 

Because A (and B, C) were already removed from `self.links` in Step 1, `calc_ancestors` returns an **empty `HashSet`** for every one of them. The `for anc_id in &ancestors` loop body never executes. `sub_descendant_weight` is never called on P.

**Step 3 — P's stored `EvictKey` is stale.**

`EvictKey` is computed from `descendants_fee`, `descendants_cycles`, and `descendants_count`: [5](#0-4) 

Because `modify_by_id` (which updates `e.evict_key`) is never reached for P, P's eviction priority in the multi-index map reflects a `descendants_feerate` that includes A, B, and C — transactions that no longer exist.

### Impact Explanation

An attacker who controls P (low fee) and A→B→C (high fees, children of P) can:

1. Submit P, then A→B→C. P's `descendants_count` becomes 4; `descendants_fee` = sum of all four fees.
2. Submit A′ that double-spends A's input. `resolve_conflict` calls `remove_entry_and_descendants(A)`, removing A, B, C.
3. P remains in the pool with `descendants_count = 4` and `descendants_fee` still including A+B+C.
4. P's `EvictKey.fee_rate = max(descendants_feerate, feerate)` is artificially high.
5. When the pool is full and the eviction loop runs (`next_evict_entry` → `iter_by_evict_key`), P is ordered as if it were a high-value transaction and is not evicted.
6. Legitimate transactions with genuinely higher fee rates may be rejected instead.

The attack is repeatable at low cost: the attacker pays fees for A, B, C once, then reclaims the UTXO via A′. P stays in the pool indefinitely.

### Likelihood Explanation

The attack requires only standard transaction submission over P2P or RPC — no privileged access, no PoW, no key leakage. The attacker controls all inputs and can craft the chain deterministically. The bug is triggered on every call to `remove_entry_and_descendants` where the removed root has a surviving parent, which also occurs during `resolve_conflict_header_dep` and the ancestor-count eviction path in `check_and_record_ancestors`.

### Recommendation

Compute and save the ancestor set of the subtree root **before** clearing any links, then apply `sub_descendant_weight` to those ancestors directly. Concretely, in `remove_entry_and_descendants`, before the `remove_entry_links` loop, collect `calc_ancestors(id)` for the root and call `update_ancestors_index_key` (or an equivalent bulk update) for each transaction being removed while links are still intact. Alternatively, restructure `remove_entry` so that `update_ancestors_index_key` is called before `remove_entry_links` even in the single-entry path, and skip it in the bulk path only after the ancestor stats have been correctly decremented.

### Proof of Concept

Build a chain P → A → B → C in a `PoolMap`, then call `remove_entry_and_descendants(A)` and assert:

```rust
// After removal of A, B, C:
let p = pool_map.get(&p_id).unwrap();
assert_eq!(p.descendants_count, 1);          // should be 1 (only P itself)
assert_eq!(p.descendants_fee, p_fee);        // should equal only P's own fee
assert_eq!(p.descendants_cycles, p_cycles);  // should equal only P's own cycles
```

Without the fix, all three assertions fail: `descendants_count` remains 4 and `descendants_fee`/`descendants_cycles` include A+B+C's contributions, confirming the stale `EvictKey` that prevents P's eviction. [6](#0-5)

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
