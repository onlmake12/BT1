Audit Report

## Title
Attacker-Controlled Snappy Varint Causes Pre-Decompression 8 MB Heap Allocation in P2P Decoder — (File: network/src/compress.rs)

## Summary
In `LengthDelimitedCodecWithCompress::decode`, the snappy uncompressed-length varint is read from the incoming frame and used to allocate a zeroed buffer (`BytesMut::zeroed(decompressed_bytes_len)`) before any decompression is attempted or validated. The only guard rejects values strictly greater than 8 MB, so an attacker can send a ~10-byte compressed frame that claims exactly 8 MB, forcing an 8 MB heap allocation per message. Repeated across concurrent connections this exhausts node memory and crashes the process.

## Finding Description
`LengthDelimitedCodecWithCompress::decode` (lines 219–262 of `network/src/compress.rs`) follows this path:

1. `self.length_delimited.decode(src)?` (line 226) buffers the compressed frame and enforces the per-protocol `max_frame_length` cap on the *compressed* payload size.
2. If `COMPRESS_FLAG` is set (line 232), `decompress_len(&data[1..])` (line 233) reads the snappy varint-encoded uncompressed length from the raw stream header — a value entirely under attacker control.
3. The only guard is `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (line 235), where `MAX_UNCOMPRESSED_LEN = 1 << 23 = 8 MB` (line 13). Any value ≤ 8 MB passes.
4. Line 242 immediately allocates: `let mut buf = BytesMut::zeroed(decompressed_bytes_len);`
5. `SnapDecoder::new().decompress(...)` then fails because the actual payload does not match the claimed size, the connection is dropped, but the 8 MB allocation has already occurred.

The outer `LengthDelimitedCodec` cap is irrelevant to this attack: it only limits the compressed frame size. A 5-byte compressed frame body (1 flag byte + 4-byte varint claiming 8 MB) is well within every protocol's `max_frame_length`, yet triggers the full 8 MB allocation. The two constraints are completely independent.

The same pre-allocation pattern also exists in `Message::decompress` at line 81 (`vec![0; decompressed_bytes_len]`), though the primary attack surface is the codec path.

## Impact Explanation
Each peer connection can force one 8 MB allocation before being disconnected. With CKB's default inbound peer limit, an attacker holding O(peers) concurrent connections accumulates O(peers × 8 MB) of live allocations. Because the attacker can reconnect immediately after disconnection (using multiple IPs or rotating through peer slots), allocations can be triggered in a tight loop. On a node with limited RAM this causes the OS OOM killer to terminate the `ckb` process, halting block validation and transaction relay.

This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* (10001–15000 points).

## Likelihood Explanation
The entry point is the public P2P TCP port, reachable by any unprivileged peer with no authentication, no prior state, and no special capability. The crafted frame is 9 bytes total (4-byte length prefix + 1 flag byte + 4-byte varint) and trivially constructed. The only friction — the per-protocol `max_frame_length` check — is satisfied automatically because the malicious frame is tiny. The attack is straightforward to automate and repeatable at high frequency.

## Recommendation
1. **Remove the pre-allocation based on the snappy header.** Use `snap::raw::Decoder::decompress_vec` (which allocates only as much as actually decompressed) or decompress into a size-capped buffer grown incrementally, so no allocation is made before decompression succeeds.
2. **Alternatively**, cap `decompressed_bytes_len` to `self.length_delimited.max_frame_length()` rather than the global `MAX_UNCOMPRESSED_LEN`. A legitimately compressed message cannot decompress to more than the protocol's own maximum uncompressed message size, so this eliminates the gap between the compressed-frame cap and the decompressed-allocation cap.
3. **Rate-limit or ban peers** that repeatedly trigger `InvalidData` errors on the compressed path to reduce reconnect-loop amplification.

## Proof of Concept
```rust
// Connect to any CKB P2P port and send one crafted frame.
// Wire format: [4-byte big-endian frame length][compress_flag][snappy varint for 8MB]

use std::io::Write;
use std::net::TcpStream;

fn main() {
    let mut stream = TcpStream::connect("TARGET:8115").unwrap();

    // snappy raw varint encoding of 8_388_608 (8 MB)
    // 8388608 = 0x800000; varint: 0x80 0x80 0x80 0x04
    let varint_8mb: &[u8] = &[0x80, 0x80, 0x80, 0x04];

    // frame body: compress_flag (0x80) + varint = 5 bytes
    let mut frame_body = vec![0x80u8]; // COMPRESS_FLAG
    frame_body.extend_from_slice(varint_8mb);

    // 4-byte big-endian length prefix
    let len = frame_body.len() as u32;
    stream.write_all(&len.to_be_bytes()).unwrap();
    stream.write_all(&frame_body).unwrap();

    // CKB node allocates BytesMut::zeroed(8_388_608) before failing decompression.
    // Repeat across many connections to exhaust node memory.
    println!("Sent {} bytes, forced 8 MB allocation on target.", 4 + frame_body.len());
}
```

Run this in parallel across the node's inbound peer slots (or from multiple IPs) to accumulate allocations until OOM. The allocation at line 242 of `network/src/compress.rs` occurs unconditionally before the `decompress` call at line 243 returns an error.
