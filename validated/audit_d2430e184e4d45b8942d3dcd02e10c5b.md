### Title
Missing Event Emissions for `addFunds` and `withdrawFunds` Balance State Changes — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `addFunds` and `withdrawFunds` functions modify critical subscription balance state (`status.balanceInWei`) without emitting any events. Every other state-mutating function in the contract emits a corresponding event, making these two functions a clear and consistent omission.

---

### Finding Description

In `Scheduler.sol`, the `addFunds` function (lines 602–628) accepts ETH and increments `status.balanceInWei` for a subscription, while `withdrawFunds` (lines 630–662) decrements `status.balanceInWei` and transfers ETH to `msg.sender`. Neither function emits an event. [1](#0-0) [2](#0-1) 

By contrast, every other state-mutating function in the same contract emits a corresponding event:

- `createSubscription` → `SubscriptionCreated` [3](#0-2) 
- `updateSubscription` → `SubscriptionUpdated`, `SubscriptionActivated`, `SubscriptionDeactivated` [4](#0-3) 
- `updatePriceFeeds` → `PricesUpdated` [5](#0-4) 

The `SchedulerGovernance` contract similarly emits events for all its state changes (`NewAdminProposed`, `NewAdminAccepted`, `SingleUpdateKeeperFeeSet`, `MinimumBalancePerFeedSet`). [6](#0-5) 

The subscription balance stored in `SchedulerState` is the sole mechanism controlling whether a subscription remains funded and whether keepers can be paid. [7](#0-6) 

---

### Impact Explanation

Keepers discover and monitor active subscriptions via `getActiveSubscriptions` and decide whether to call `updatePriceFeeds` based on subscription balance. Without events for `addFunds` and `withdrawFunds`, off-chain keeper infrastructure and monitoring systems cannot efficiently track balance changes. Specifically:

1. A subscription manager can silently drain a subscription's balance via `withdrawFunds` with no on-chain event emitted, leaving keepers unaware that the subscription is no longer sufficiently funded until they attempt an update and revert.
2. Depositors cannot confirm their `addFunds` transaction was processed by listening for an event; they must re-query state.
3. Historical analysis of subscription funding lifecycles is impossible without replaying all transactions to the contract.

The `_processFeesAndPayKeeper` function inside `updatePriceFeeds` will revert with `InsufficientBalance` if the balance is too low after a silent withdrawal, causing keepers to waste gas on failed transactions. [8](#0-7) 

---

### Likelihood Explanation

Both `addFunds` and `withdrawFunds` are callable by any subscription manager at any time with no special privilege. The `withdrawFunds` path is particularly likely to be exercised regularly as managers reclaim unused funds. The omission affects every invocation of these two functions from contract deployment onward.

---

### Recommendation

Add events to `SchedulerGovernance.sol` or `Scheduler.sol` and emit them in both functions:

```solidity
event FundsAdded(uint256 indexed subscriptionId, address indexed sender, uint256 amount, uint256 newBalance);
event FundsWithdrawn(uint256 indexed subscriptionId, address indexed manager, uint256 amount, uint256 newBalance);
```

In `addFunds`, emit after updating the balance:
```solidity
emit FundsAdded(subscriptionId, msg.sender, msg.value, status.balanceInWei);
```

In `withdrawFunds`, emit after the transfer succeeds:
```solidity
emit FundsWithdrawn(subscriptionId, msg.sender, amount, status.balanceInWei);
```

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` and create a subscription with `createSubscription{value: 1 ether}(...)`. Observe `SubscriptionCreated` is emitted.
2. Call `addFunds{value: 0.5 ether}(subscriptionId)`. Observe **no event is emitted**, yet `getSubscription(subscriptionId).status.balanceInWei` increases by `0.5 ether`.
3. Call `withdrawFunds(subscriptionId, 0.5 ether)`. Observe **no event is emitted**, yet the balance decreases and ETH is transferred to `msg.sender`.
4. An off-chain keeper monitoring `SubscriptionCreated` / `SubscriptionUpdated` / `PricesUpdated` events has no way to learn the current balance without polling `getSubscription` on every block.

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");
    }
