### Title
Unbounded O(N) Work in `GetLastStateProofProcess::execute` via `last_n_blocks=0` + `difficulty_boundary=0` Bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

A remote, unprivileged light-client peer can send a single crafted `GetLastStateProof` message with `last_n_blocks=0`, `difficulty_boundary=U256::zero()`, and an empty `difficulties` list. The guard that is supposed to bound server work (`GET_LAST_STATE_PROOF_LIMIT=1000`) is completely bypassed, and the server is forced to iterate over every block in the main chain — performing N MMR root DB reads and O(N log N) MMR proof generation — where N is the full chain height.

---

### Finding Description

**Guard check — lines 201–205:**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

With `difficulties=[]` and `last_n_blocks=0`, this evaluates to `0 + 0 = 0`, which is not `> 1000`. The check passes unconditionally.

**Branch selection — lines 291–297:**

```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number <= last_n_blocks {
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    ...
} else { ... }
``` [2](#0-1) 

With `last_n_blocks=0` and `start_block_number=0`, the condition `N <= 0` is false for any real chain, so execution falls into the `else` branch.

**`difficulty_boundary=0` collapses `difficulty_boundary_block_number` to 0 — lines 299–311:**

`get_first_block_total_difficulty_is_not_less_than(0, N, &U256::zero())` immediately returns `Some((0, ...))` at line 31 because any total difficulty satisfies `>= 0`. [3](#0-2) 

So `difficulty_boundary_block_number = 0`.

**`last_n_numbers` spans the entire chain — lines 318–319:**

```rust
let last_n_numbers = (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
// = (0..N).collect() → [0, 1, 2, ..., N-1]
``` [4](#0-3) 

**`difficulty_boundary_block_number > 0` guard skips sampled_numbers — line 321:** [5](#0-4) 

Since `difficulty_boundary_block_number = 0`, this block is skipped entirely, and the function returns `(Vec::new(), last_n_numbers)` where `last_n_numbers` has N entries.

**`complete_headers` performs N DB reads — lines 150–163:**

For each of the N block numbers, `complete_headers` calls `snapshot.chain_root_mmr(*number - 1).get_root()`, a synchronous DB read per block. [6](#0-5) 

**`reply_proof` then generates an MMR proof over N positions** — O(N log N) computation. [7](#0-6)

### Impact Explanation

A single peer, with no credentials or PoW, can force the full node to:
- Collect a `Vec` of N block numbers (N ≈ 1,000,000+ on mainnet)
- Perform N synchronous RocksDB reads (`chain_root_mmr(n).get_root()`)
- Generate an MMR proof over N leaf positions (O(N log N) CPU)
- Serialize and attempt to send a response of size O(N)

This can be repeated in a tight loop from multiple peers, exhausting CPU, I/O, and memory, and crashing or severely degrading the full node. The `GET_LAST_STATE_PROOF_LIMIT` invariant is completely defeated.

---

### Likelihood Explanation

The exploit requires only:
1. A valid `last_hash` pointing to the current chain tip (publicly observable)
2. Setting `last_n_blocks=0`, `difficulty_boundary=0`, `difficulties=[]`, `start_number=0`

No hashpower, no key, no privileged role. Any peer that can speak the light-client protocol can trigger this. The message is tiny; the server response work is O(N).

---

### Recommendation

The limit check must bound the **actual number of blocks that will be processed**, not just the input parameters. After computing `last_n_numbers` and `sampled_numbers`, assert:

```rust
if reorg_last_n_numbers.len() + sampled_numbers.len() + last_n_numbers.len()
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Additionally, `difficulty_boundary=0` should be rejected early, since it trivially collapses the sampled region to the entire chain.

---

### Proof of Concept

```
GetLastStateProof {
    last_n_blocks:        0,
    difficulty_boundary:  U256::zero(),
    difficulties:         [],
    start_number:         0,
    start_hash:           genesis_hash,
    last_hash:            current_tip_hash,   // publicly known
}
```

Trace:
1. Limit check: `0 + 0*2 = 0 ≤ 1000` → passes
2. `last_block_number - 0 <= 0` → false → else branch
3. `get_first_block_total_difficulty_is_not_less_than(0, N, 0)` → returns `(0, ...)`
4. `last_n_numbers = (0..N)` → N entries
5. `complete_headers` → N MMR root reads
6. `reply_proof` → O(N log N) MMR proof generation

On a chain of 1,000,000 blocks, a single message triggers ~1,000,000 DB reads and ~20,000,000 MMR node accesses. Sending this in a loop from multiple peers constitutes a practical DoS.
