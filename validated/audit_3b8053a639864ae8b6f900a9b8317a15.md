### Title
Stale Descendant-Weight Accounting in `remove_entry_and_descendants` Causes Incorrect Tx-Pool Eviction — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::remove_entry_and_descendants` pre-removes all link records for every entry in the subtree before calling `remove_entry` on each one. The stated intent is to prevent `update_descendants_index_key` from running (correct — descendants are all being removed). However, the same pre-removal also silences `update_ancestors_index_key` for the **root entry's ancestors**, which are **not** being removed. Those ancestors are left with permanently inflated `descendants_count / descendants_size / descendants_cycles / descendants_fee` fields and a stale `evict_key`, mirroring the external report's pattern: accounting state is mutated but the corresponding corrective action is never executed.

---

### Finding Description

**Root cause — `pool_map.rs` `remove_entry_and_descendants`:** [1](#0-0) 

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // update links state for remove, so that we won't update_descendants_index_key in remove_entry
    for id in &removed_ids {
        self.remove_entry_links(id);          // ← strips ALL link records for the subtree
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect()
}
```

`remove_entry` then calls `update_ancestors_index_key`: [2](#0-1) 

```rust
pub(crate) fn remove_entry(&mut self, id: &ProposalShortId) -> Option<TxEntry> {
    self.entries.remove_by_id(id).map(|entry| {
        self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);  // ← relies on links
        self.update_descendants_index_key(&entry.inner, EntryOp::Remove);
        self.remove_entry_edges(&entry.inner);
        self.remove_entry_links(id);
        self.track_entry_statics(Some(entry.status), None);
        self.update_stat_for_remove_tx(entry.inner.size, entry.inner.cycles);
        entry.inner
    })
}
```

`update_ancestors_index_key` resolves ancestors through `self.links.calc_ancestors`: [3](#0-2) 

```rust
fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
    let ancestors: HashSet<ProposalShortId> =
        self.links.calc_ancestors(&child.proposal_short_id());   // ← returns ∅ after pre-removal
    for anc_id in &ancestors {
        self.entries.modify_by_id(anc_id, |e| {
            match op {
                EntryOp::Remove => e.inner.sub_descendant_weight(child),
                EntryOp::Add    => e.inner.add_descendant_weight(child),
            };
            e.evict_key = e.inner.as_evict_key();
        });
    }
}
```

Because `remove_entry_links` was already called for every entry in the subtree, `calc_ancestors` returns an empty set for the root entry. The ancestors of the root entry that **remain in the pool** never receive the `sub_descendant_weight` call and their `evict_key` is never recomputed.

The fields that go stale on surviving ancestors: [4](#0-3) 

```rust
pub fn sub_descendant_weight(&mut self, entry: &TxEntry) {
    self.descendants_count   = self.descendants_count.saturating_sub(1);
    self.descendants_size    = self.descendants_size.saturating_sub(entry.size);
    self.descendants_cycles  = self.descendants_cycles.saturating_sub(entry.cycles);
    self.descendants_fee     = Capacity::shannons(
        self.descendants_fee.as_u64().saturating_sub(entry.fee.as_u64()),
    );
}
```

These four fields feed directly into `as_evict_key()`, which drives the pool's eviction ordering.

---

### Impact Explanation

After an RBF replacement removes a child transaction (and its descendants), the surviving parent transaction retains inflated `descendants_*` metrics and a stale `evict_key`. When the pool reaches its size/cycle limit and `limit_size` triggers eviction: [5](#0-4) 

the parent is ranked as if it still has the removed descendants. Depending on the eviction comparator, this can cause the parent to be evicted prematurely (if high descendant count increases eviction priority) or to block eviction of a genuinely low-value transaction. Either outcome is incorrect: a valid, fee-paying transaction may be silently dropped from the pool without the submitter being notified, requiring manual resubmission.

The `score` field (`AncestorsScoreSortKey`) used for block-assembly ordering is **not** updated by `update_ancestors_index_key`, so block-template construction is unaffected. The harm is confined to eviction decisions and the `descendants_*` counters exposed via `get_pool_tx_detail_info` RPC.

---

### Likelihood Explanation

`remove_entry_and_descendants` is called from `process_rbf` every time an RBF replacement succeeds: [6](#0-5) 

```rust
let all_removed: Vec<_> = conflicts
    .iter()
    .flat_map(|id| tx_pool.pool_map.remove_entry_and_descendants(id))
    .collect();
```

Any unprivileged user who submits a transaction via the `send_transaction` RPC that replaces a child transaction in an existing chain triggers this path. No special privilege is required; RBF is enabled whenever `min_rbf_rate > min_fee_rate` in the node configuration, which is the recommended production setting.

---

### Recommendation

Before stripping links in `remove_entry_and_descendants`, capture the ancestors of the root entry and decrement their descendant weights explicitly:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Fix: update surviving ancestors' descendant weights BEFORE removing links
    let root_entry = self.entries.get_by_id(id).map(|e| e.inner.clone());
    if let Some(root) = root_entry {
        self.update_ancestors_index_key(&root, EntryOp::Remove);
        // also subtract each descendant's weight from surviving ancestors
        for desc_id in self.calc_descendants(id) {
            if let Some(desc) = self.entries.get_by_id(&desc_id).map(|e| e.inner.clone()) {
                self.update_ancestors_index_key(&desc, EntryOp::Remove);
            }
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }
    removed_ids.iter().filter_map(|id| self.remove_entry(id)).collect()
}
```

Alternatively, refactor `remove_entry` to accept a flag that suppresses `update_descendants_index_key` without also suppressing `update_ancestors_index_key`.

---

### Proof of Concept

1. Submit a transaction chain: `tx_parent → tx_child → tx_grandchild` (each spending the previous output).
2. Submit `tx_replacement` that spends the same input as `tx_child` with a higher fee (RBF).
3. `process_rbf` calls `remove_entry_and_descendants(&tx_child.proposal_short_id())`.
4. `tx_child` and `tx_grandchild` are removed; `tx_parent` remains.
5. Inspect `tx_parent` via `get_pool_tx_detail_info`: `descendants_count` still reports 2 (should be 0), `evict_key` is stale.
6. Fill the pool to its size limit with low-fee transactions; observe that `tx_parent` may be evicted despite having a competitive fee rate, because its inflated `evict_key` ranks it as a high-eviction-priority entry.

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );

                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles

            }
        }
