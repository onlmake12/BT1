Audit Report

## Title
TOCTOU Race in `_process_tx` Enables Wasted Concurrent Script Verification of Conflicting Transactions — (File: `tx-pool/src/process.rs`)

## Summary

The `_process_tx` pipeline in `tx-pool/src/process.rs` performs a cell-liveness check under a read lock (`pre_check`), releases the lock, runs full script verification with no lock held (`verify_rtx`), and only acquires a write lock at `submit_entry` to detect conflicts. Because the read lock permits concurrent readers, N transactions all spending the same live cell can simultaneously pass `pre_check`, all undergo full `ContextualTransactionVerifier` script execution, and only one ultimately succeeds at `submit_entry`. The N-1 wasted verifications are exploitable by any P2P peer to saturate verify workers with minimal cost.

## Finding Description

**Step 1 — `pre_check` under read lock (lines 269–316):**

`pre_check` calls `with_tx_pool_read_lock`, which acquires `self.tx_pool.read()`. Multiple concurrent callers all hold the read lock simultaneously. Inside the closure, `resolve_tx(tx_pool, &snapshot, tx.clone(), false)` checks whether the input cells are live. If cell `C` has not yet been consumed by any pool entry, all N concurrent callers see it as live and return `Ok(...)`.

**Step 2 — `verify_rtx` with no lock (lines 724–732):**

After `pre_check` returns and the read lock is dropped, `_process_tx` calls `verify_rtx` with no lock held. This invokes `ContextualTransactionVerifier::verify_with_pause(max_tx_verify_cycles, command_rx)` — the most expensive operation in the pipeline. All N callers execute this concurrently.

**Step 3 — `submit_entry` under write lock (lines 96–170):**

Only here does the conflict check fire. With RBF disabled, `tx_pool.pool_map.find_conflict_outpoint(entry.transaction())` detects the double-spend and returns `Err(Reject::Resolve(OutPointError::Dead(outpoint)))` for transactions T2…TN. T1 succeeds; N-1 full script verifications were entirely wasted.

**Multiple concurrent workers confirmed in `verify_mgr.rs` (lines 179–180, 130–145):**

`VerifyMgr` spawns `max_tx_verify_workers` workers. Each worker independently pops an entry from the shared `VerifyQueue` under a write lock, then calls `_process_tx` — meaning multiple workers process different queue entries concurrently with no coordination between them.

**Verify queue deduplication does not prevent this (lines 349–351):**

`resumeble_process_tx` checks `verify_queue_contains` using `proposal_short_id` (first 10 bytes of tx hash). N distinct transactions spending the same cell have N distinct hashes and thus N distinct `proposal_short_id`s — all pass the deduplication check and are all enqueued.

**The design is intentional for RBF but creates the attack surface:**

The comment at line 305–307 explicitly acknowledges that conflicting transactions are allowed to proceed through verification before `submit_entry`. This design is correct for the RBF path (where a replacement tx conflicts with an existing pool entry), but it equally applies to N fresh transactions all spending the same live cell — none of which is yet in the pool — so all take the `Ok` branch in `pre_check` and proceed to full verification.

## Impact Explanation

An attacker controlling one live cell can construct N transactions spending that cell with a high-cycle lock script (up to `max_tx_verify_cycles` per transaction). All N enter the verify queue (bounded at `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE = 256 MB`), are dequeued by `max_tx_verify_workers` workers concurrently, each runs full `ContextualTransactionVerifier` script execution, and N-1 are rejected only at `submit_entry`. The attacker can continuously refill the queue after each batch drains, sustaining near-100% CPU utilization on all verify workers indefinitely. This delays block template generation and legitimate transaction processing on the targeted node.

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The cost to the attacker is one live cell and P2P network access — both available to any ordinary CKB user.

## Likelihood Explanation

The attack requires no privileged access. Any P2P peer can relay transactions. Constructing N valid transactions spending the same cell requires only ownership of that cell and a valid (or `always_success`-style) lock script. The verify queue size limit and per-transaction cycle cap bound the damage per batch, but the attacker can continuously replenish the queue. The attack is repeatable, cheap, and requires no majority hashpower or special protocol knowledge.

## Recommendation

Apply a "pending inputs" reservation before releasing the read lock or at the start of `submit_entry`. Specifically:

1. **Preferred:** Introduce a `pending_inputs: HashSet<OutPoint>` set in `TxPool`. In `pre_check` (still under the read lock), check that none of the transaction's input outpoints appear in `pending_inputs`. Upgrade `pre_check` to use a write lock (or add a separate write-lock step) to atomically insert the input outpoints into `pending_inputs` before releasing. Remove them in `submit_entry` (on success or failure).

2. **Alternative:** Upgrade `pre_check` to use `with_tx_pool_write_lock` entirely, accepting the throughput trade-off of serialized pre-checks.

3. **For the secondary TOCTOU in `resumeble_process_tx`:** Combine the orphan check, verify-queue check, and enqueue into a single write-lock section to eliminate the window between check and enqueue.

## Proof of Concept

1. Attacker owns live cell `C` on the CKB network with a high-cycle lock script (looping near `max_tx_verify_cycles`).
2. Attacker constructs transactions T1…TN, each spending cell `C` with different outputs (distinct tx hashes, distinct `proposal_short_id`s).
3. Attacker's peers P1…PN simultaneously relay T1…TN via the transaction relay protocol, triggering `submit_remote_tx` → `resumeble_process_tx` → `enqueue_verify_queue` for each.
4. All N transactions enter the verify queue (each has a unique `proposal_short_id`, queue size limit not yet exceeded).
5. `max_tx_verify_workers` workers each pop an entry and call `_process_tx` concurrently.
6. All N calls invoke `pre_check` under the read lock — cell `C` appears live to all of them (none has reached `submit_entry` yet). All N return `Ok`.
7. All N calls enter `verify_rtx` concurrently with no lock held, each running `ContextualTransactionVerifier::verify_with_pause(max_tx_verify_cycles, ...)`.
8. All N calls reach `submit_entry` (write lock). T1 is accepted; T2…TN are rejected with `Reject::Resolve(OutPointError::Dead(C))`.
9. N-1 full script verifications were wasted. Attacker repeats by relaying a new batch of conflicting transactions, continuously saturating all verify workers.
