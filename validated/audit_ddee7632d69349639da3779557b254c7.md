### Title
Peer Store Eviction Deadlock via Crafted Discovery Nodes Messages Silently Drops All New Peer Addresses — (`network/src/peer_store/peer_store_impl.rs`, `network/src/protocols/discovery/mod.rs`)

---

### Summary

`check_purge()` has a logical dead zone: when the peer store is at capacity with all addresses "connectable" and no /16 group exceeds 4 entries, both eviction passes produce zero candidates and the function returns `Err(PeerStoreError::EvictionFailed)`. `add_new_addrs()` silently discards this error via a `debug!()` log. An unprivileged remote peer can deliberately craft this state by injecting exactly `ADDR_COUNT_LIMIT=16384` addresses spread across 4096 distinct /16 subnets (4 per subnet) via P2P discovery `Nodes` messages, after which the victim node can no longer add any newly-discovered peer address.

---

### Finding Description

**Eviction pass 1 — non-connectable filter:**

`is_connectable()` returns `true` for any freshly-discovered address because `add_addr()` always stores it with `last_connected_at_ms=0` and `attempts_count=0`: [1](#0-0) 

With those values, none of the three `false` branches in `is_connectable()` fire: [2](#0-1) 

So pass 1 collects zero candidates and falls through to pass 2.

**Eviction pass 2 — network-group filter:**

The /16 grouping key is the first two octets of the IPv4 address: [3](#0-2) 

Pass 2 takes the top `len/2` groups by size and only evicts from groups where `addrs.len() > 4`: [4](#0-3) 

**The dead zone:** `ADDR_COUNT_LIMIT = 16384`. If the store holds exactly 4096 groups × 4 addresses = 16384 entries, every group has `len == 4`, so `addrs.len() > 4` is false for all groups. Pass 2 also produces zero candidates, and the function returns: [5](#0-4) 

**Silent discard in `add_new_addrs()`:** [6](#0-5) 

The `Err(EvictionFailed)` is caught, logged at `debug!` level, and discarded. No disconnect, no backpressure, no retry.

---

### Impact Explanation

Once the peer store is locked in this state, every subsequent call to `add_addr()` (the only path for discovery-sourced addresses) hits `check_purge()`, gets `Err(EvictionFailed)`, and is silently dropped. The victim node's peer store is permanently frozen at 16384 attacker-controlled entries. New honest peer addresses learned via the discovery protocol are permanently discarded. The victim's ability to organically expand its peer set is eliminated for the lifetime of the process (or until a restart clears the store).

---

### Likelihood Explanation

- The attacker needs only a single P2P connection to the victim; no privileged role is required.
- Discovery `Nodes` messages are rate-limited per message but not per session over time; the attacker can drip-feed addresses across many messages.
- The required address space (4096 distinct /16 prefixes × 4 addresses each) is trivially constructable from the public IPv4 space.
- `is_valid_addr()` filters loopback/private addresses, but the attacker can use globally-routable IPs (real or spoofed — the peer store does not verify reachability before storing).
- The attack is persistent: once the store is full in this configuration, it self-perpetuates because no eviction can occur.

---

### Recommendation

1. **Fix the eviction threshold**: Change `addrs.len() > 4` to `addrs.len() >= 4` (or `>= 2`) in pass 2 so that groups at the per-subnet cap are also eligible for eviction.
2. **Fallback eviction**: If both passes yield zero candidates, randomly evict a configurable number of entries rather than returning an error.
3. **Propagate or handle the error**: `add_new_addrs()` should not silently discard `Err(EvictionFailed)`; at minimum it should log at `warn!` level and consider disconnecting the peer that triggered the condition.
4. **Per-peer address rate limiting**: Bound how many addresses a single session can contribute to the peer store.

---

### Proof of Concept

```
1. Connect to victim node via P2P.
2. Send repeated Nodes (announce=true) messages containing addresses of the form:
     /ip4/<A>.<B>.x.y/tcp/<port>/p2p/<random_peer_id>
   where (A, B) ranges over 4096 distinct pairs and each pair contributes
   exactly 4 distinct (x, y, port, peer_id) tuples.
   Total: 4096 × 4 = 16384 addresses.
3. All addresses are stored via add_addr() with last_connected_at_ms=0,
   attempts_count=0 → all pass is_connectable().
4. Now send one more Nodes message with a new honest address.
5. add_addr() calls check_purge():
   - Pass 1: 0 non-connectable peers → empty candidate list.
   - Pass 2: 4096 groups, each with len==4, none > 4 → empty candidate list.
   - Returns Err(EvictionFailed).
6. add_new_addrs() logs at debug! and discards the error.
7. Assert: the new honest address is NOT in the peer store.
   Assert: the peer store count remains 16384.
   Assert: all subsequent add_addr() calls for honest peers also fail silently.
```

Key constants confirmed in source:
- `ADDR_COUNT_LIMIT = 16384` [7](#0-6) 
- `ADDR_MAX_RETRIES = 3`, `ADDR_MAX_FAILURES = 10` [8](#0-7) 
- Group key = first two IPv4 octets [9](#0-8) 
- Eviction threshold `> 4` [10](#0-9)

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }

    fn add_new_addrs(&mut self, _session_id: SessionId, addrs: Vec<(Multiaddr, Flags)>) {

        for (addr, flags) in addrs.into_iter().filter(|addr| self.is_valid_addr(&addr.0)) {

            self.network_state.with_peer_store_mut(|peer_store| {

                        err, addr
                    );
                }
            });
        }
    }
