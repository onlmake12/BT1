### Title
Silent `u128 → u64` Truncation in `DaoCalculator::calculate_maximum_withdraw` Produces Wrong Withdrawal Capacity — (`File: util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` uses a bare `as u64` cast to narrow a `u128` intermediate result. When the intermediate value exceeds `u64::MAX` but the truncated remainder plus `occupied_capacity` does not, the function silently returns a drastically understated withdrawal capacity instead of propagating `DaoError::Overflow`. Every other analogous multiplication in the same file uses `u64::try_from(…).map_err(|_| DaoError::Overflow)?`, making this a clear inconsistency with a concrete accounting impact.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the interest-adjusted withdrawal capacity as:

```rust
// util/dao/src/lib.rs  lines 152-156
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a **wrapping truncation**: if `withdraw_counted_capacity` is, say, `u64::MAX + K`, the cast yields `K - 1`. If `K - 1 + occupied_capacity ≤ u64::MAX`, `safe_add` succeeds and the function returns a capacity value that is `u64::MAX + 1` shannons smaller than the correct answer — with no error raised.

Every other `u128 → u64` narrowing in the same crate uses the safe form:

```rust
// dao_field_with_current_epoch — miner_issuance (lines 242-245)
let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
    / u128::from(parent_c.as_u64());
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
// dao_field_with_current_epoch — ar_increase (lines 256-258)
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The existing overflow test (`check_withdraw_calculation_overflows`) only exercises the path where `safe_add` itself overflows; it does **not** cover the silent-truncation path where the truncated value plus `occupied_capacity` fits in `u64`. [4](#0-3) 

---

### Impact Explanation

`calculate_maximum_withdraw` is called from three places:

1. **`transaction_maximum_withdraw`** → **`transaction_fee`** — used by `check_tx_fee` in the tx-pool to accept/reject transactions and compute the fee stored in the pool entry. [5](#0-4) [6](#0-5) 

2. **`withdrawed_interests`** → **`dao_field_with_current_epoch`** — the DAO field `s` (secondary issuance pool) written into every block header is updated as `current_s = parent_s + nervosdao_issuance - withdrawed_interests`. A silently wrong `withdrawed_interests` permanently corrupts the on-chain DAO accounting state. [7](#0-6) 

3. **`calculate_dao_maximum_withdraw` RPC** — returns the wrong capacity to callers, causing them to construct withdrawal transactions that will be rejected (or, in the silent-truncation case, accepted with a wrong output capacity). [8](#0-7) 

---

### Likelihood Explanation

The overflow condition requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`counted_capacity = output_capacity − occupied_capacity`. A cell with capacity near `u64::MAX` shannons (≈ 184 billion CKB) and any positive growth in `ar` since deposit satisfies this. The `ar` ratio needs to exceed `1 + ~2.3 × 10⁻¹⁰`, which is achievable after a modest number of blocks given CKB's secondary issuance schedule. The silent (non-error) path is triggered when the truncated remainder plus `occupied_capacity` does not itself overflow `u64`, which is the common case for small remainders.

Any DAO depositor who controls a cell with near-maximum capacity can reach this path without any privileged access, simply by submitting a standard DAO withdrawal transaction (phase 2) or calling the `calculate_dao_maximum_withdraw` RPC.

---

### Recommendation

Replace the bare `as u64` cast with the same checked conversion used elsewhere in the file:

```rust
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

Add a unit test that constructs a scenario where `withdraw_counted_capacity` overflows `u64` but the truncated value plus `occupied_capacity` does not, and asserts that the function returns `Err(DaoError::Overflow)`.

---

### Proof of Concept

```
deposit_ar      = 10_000_000_000_000_000
withdrawing_ar  = 10_000_000_002_000_000   (ar grew by 2e6 since deposit)
output_capacity = 18_446_744_073_709_551_000  (near u64::MAX)
occupied_capacity ≈ 4_200_000_000            (minimal lock, no data)

counted_capacity = 18_446_744_073_709_551_000 − 4_200_000_000
               = 18_446_744_069_509_551_000

withdraw_counted_capacity (u128)
  = 18_446_744_069_509_551_000 × 10_000_000_002_000_000
    / 10_000_000_000_000_000
  = 18_446_744_069_509_551_000 × 1.0000000002
  ≈ 18_446_744_073_194_660_200   -- still < u64::MAX, safe_add overflows

Increase ar growth slightly:
withdrawing_ar  = 10_000_000_010_000_000   (ar grew by 1e7)

withdraw_counted_capacity (u128)
  ≈ 18_446_744_069_509_551_000 × 1.000000001
  ≈ 18_446_744_087_956_295_000   -- exceeds u64::MAX (18_446_744_073_709_551_615)

as u64 truncation:
  18_446_744_087_956_295_000 mod 2^64
  = 18_446_744_087_956_295_000 − 18_446_744_073_709_551_616
  = 14_246_743_384

safe_add(4_200_000_000) = 18_446_743_384  -- succeeds, returns wrong small value
```

The function returns `Ok(Capacity::shannons(18_446_743_384))` — roughly 18 shannons — instead of `Err(DaoError::Overflow)`, silently corrupting the withdrawal amount by approximately `u64::MAX` shannons (≈ 184 billion CKB).

    fn calculate_dao_maximum_withdraw(
        &self,
        out_point: OutPoint,
        kind: DaoWithdrawingCalculationKind,
    ) -> Result<Capacity> {
        let snapshot: &Snapshot = &self.shared.snapshot();
        let consensus = snapshot.consensus();
        let out_point: packed::OutPoint = out_point.into();
        let data_loader = snapshot.borrow_as_data_loader();
        let calculator = DaoCalculator::new(consensus, &data_loader);
        match kind {
            DaoWithdrawingCalculationKind::WithdrawingHeaderHash(withdrawing_header_hash) => {
                let (tx, deposit_header_hash) = snapshot
                    .get_transaction(&out_point.tx_hash())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output = tx
                    .outputs()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;
                let output_data = tx
                    .outputs_data()
                    .get(out_point.index().into())
                    .ok_or_else(|| RPCError::invalid_params("invalid out_point"))?;

                match calculator.calculate_maximum_withdraw(
                    &output,
                    core::Capacity::bytes(output_data.len()).expect("should not overflow"),
                    &deposit_header_hash,
                    &withdrawing_header_hash.into(),
                ) {
                    Ok(capacity) => Ok(capacity.into()),
                    Err(err) => Err(RPCError::custom_with_error(RPCError::DaoError, err)),
                }
            }
