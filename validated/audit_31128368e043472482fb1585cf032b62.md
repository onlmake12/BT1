### Title
`ScriptHashTypeVerifier` Omits Type Script `hash_type` Validation, Allowing Non-Permitted Hash Types to Bypass Non-Contextual Checks - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` in `verification/src/transaction_verifier.rs` only validates the `hash_type` field of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but never validates the `hash_type` of each output's **type** script. An unprivileged transaction sender can craft a transaction whose type script carries a non-permitted `hash_type` value (e.g., `Data3 = 6`), bypass the non-contextual verifier, have the transaction admitted to the tx pool, and have it relayed across the P2P network — all before the transaction ultimately fails at script execution time.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` is defined as the set `{0, 1, 2, 4}` (Data, Type, Data1, Data2): [1](#0-0) 

`ScriptHashTypeVerifier::verify()` iterates over every transaction output and checks only `output.lock().hash_type()`: [2](#0-1) 

The type script field — `output.type_()` — is never inspected. The comment on `NonContextualTransactionVerifier` even documents this gap explicitly: *"Check whether output **lock** hash type within enabled range"*: [3](#0-2) 

The lower-level `check_data` function does check both lock and type scripts, but it uses `ScriptHashType::verify_value`, which accepts **all** structurally valid hash type bytes (any even number or `1`): [4](#0-3) [5](#0-4) 

`verify_value` accepts `Data3 = 6`, `Data4 = 8`, etc. — values that are **not** in `ENABLED_SCRIPT_HASH_TYPE`. So `check_data` does not compensate for the missing check in `ScriptHashTypeVerifier`.

The `ScriptHashTypeVerifier` is invoked as part of `NonContextualTransactionVerifier`, which runs on every transaction submitted to the tx pool and on every relayed transaction: [6](#0-5) 

---

### Impact Explanation

A transaction with `type_script.hash_type = Data3` (byte value `6`, not in `ENABLED_SCRIPT_HASH_TYPE`):

1. Passes `check_data` (6 is even → `verify_value(6) == true`).
2. Passes `ScriptHashTypeVerifier` (only the lock script is checked).
3. Is admitted to the tx pool.
4. Is relayed to all connected peers via the relay protocol.
5. Peers run the same `ScriptHashTypeVerifier` and also pass it.
6. The transaction propagates network-wide before ultimately failing at script execution in `select_version`: [7](#0-6) 

The concrete impact is **tx pool spam and network-wide relay of invalid transactions**. An attacker can flood the network with transactions that consume tx pool slots and relay bandwidth on every node, without any of those transactions ever being committable to a block.

---

### Likelihood Explanation

The entry path requires no privilege: any user with RPC access (the `send_transaction` endpoint) or any peer that can send a `RelayTransaction` message can trigger this. Crafting the transaction is trivial — set the type script's `hash_type` byte to `6` (Data3). No key material, no special role, and no majority hashpower is needed.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's type script, mirroring the existing lock script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }

        // ADD: type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }).into());
            }
        }
    }
    Ok(())
}
```

Also update the doc comment on `NonContextualTransactionVerifier` to reflect that both lock and type script hash types are checked.

### Proof of Concept

```rust
// Craft a transaction with a type script using Data3 (hash_type = 6),
// which is NOT in ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}.
let forbidden_type_script = Script::default()
    .as_builder()
    .hash_type(ScriptHashType::Data3) // byte value 6
    .build();

let transaction = TransactionBuilder::default()
    .output(
        CellOutput::new_builder()
            .lock(Script::default()) // lock hash_type = 0 (Data), valid
            .type_(Some(forbidden_type_script)) // type hash_type = 6, NOT in ENABLED set
            .build(),
    )
    .output_data(Default::default())
    .build();

let verifier = ScriptHashTypeVerifier::new(&transaction);
// This returns Ok(()) — the invalid type script hash_type is never checked.
assert!(verifier.verify().is_ok());
```

The transaction passes `NonContextualTransactionVerifier`, enters the tx pool, and is relayed to peers. It only fails later during contextual script execution in `select_version` at `script/src/types.rs:930-934`, after network-wide propagation has already occurred. [2](#0-1) [1](#0-0)
