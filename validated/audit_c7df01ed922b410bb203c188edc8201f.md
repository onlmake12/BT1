### Title
`total_tx_size`/`total_tx_cycles` Inflated After Ancestor-Eviction in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new entry's size and cycles are added to a local snapshot of the aggregate counters **before** `check_and_record_ancestors` runs. That inner call can evict existing pool entries (correctly decrementing `total_tx_size`/`total_tx_cycles` via `update_stat_for_remove_tx`), but the stale pre-eviction snapshot is then unconditionally written back, silently re-inflating both counters by the size of every evicted transaction.

---

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` computes the new aggregate totals before any evictions occur:

```rust
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // snapshot: self.total_tx_size + entry.size
``` [1](#0-0) 

It then calls `check_and_record_ancestors`, which — when the ancestor count exceeds `max_ancestors_count` but can be reduced by evicting `cell_ref_parents` — calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, correctly decrementing `self.total_tx_size` for each evicted entry: [2](#0-1) [3](#0-2) 

After `check_and_record_ancestors` returns, the stale local snapshot is written back unconditionally:

```rust
self.total_tx_size = total_tx_size;   // overwrites the correctly-decremented value
self.total_tx_cycles = total_tx_cycles;
``` [4](#0-3) 

**Concrete arithmetic:**

| Step | `self.total_tx_size` |
|---|---|
| Before `add_entry` | `X` |
| After `updated_stat_for_add_tx` (local var) | local = `X + entry.size` |
| After evictions inside `check_and_record_ancestors` | `self` = `X − E` (E = evicted sizes) |
| After line 218 overwrites | `self` = `X + entry.size` ← **E bytes never subtracted** |

Correct value: `X − E + entry.size`. Actual value: `X + entry.size`. Inflation per triggering call: `E`.

---

### Impact Explanation

1. **Over-eviction / pool starvation.** `limit_size` loops while `total_tx_size > max_tx_pool_size`. An inflated counter causes it to evict additional valid transactions that should remain in the pool, degrading throughput for all users. [5](#0-4) 

2. **False `tx_pool_info` RPC data.** `total_tx_size` and `total_tx_cycles` are returned directly to RPC callers via `tx_pool_info()`. Inflated values mislead fee-estimation tools, wallets, and monitoring infrastructure. [6](#0-5) 

3. **Incorrect pool-full rejection.** Because `total_tx_size` is used to gate admission into the verify queue and the pool itself, inflation can cause legitimate transactions to be rejected with `Reject::Full` even when actual pool occupancy is below the limit.

---

### Likelihood Explanation

The eviction branch in `check_and_record_ancestors` is triggered when:
- A new transaction's input consumes an output that is also referenced as a **cell dep** by one or more existing pool transactions (`cell_ref_parents` is non-empty), **and**
- The total ancestor count exceeds `max_ancestors_count` but falls within limit after removing those `cell_ref_parents`. [7](#0-6) 

This is reachable by any unprivileged actor via `send_transaction` RPC or the P2P relay protocol (RelayV3). A crafted transaction chain where one transaction's output is used both as a cell dep and as a spendable input by a later transaction is a realistic construction. The inflation accumulates across multiple such submissions, making the pool's reported size progressively diverge from reality.

---

### Recommendation

Move the `updated_stat_for_add_tx` call to **after** `check_and_record_ancestors` completes (so it reads the already-decremented `self.total_tx_size`), or eliminate the local snapshot entirely and apply the addition directly after all evictions:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
// Now self.total_tx_size already reflects evictions; add the new entry's contribution.
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

---

### Proof of Concept

1. Fill the pool with a chain: `tx_A` (output used as cell dep by `tx_B`), `tx_B` (cell dep = `tx_A` output), repeated until ancestor count is near `max_ancestors_count`.
2. Submit `tx_C` whose input spends `tx_A`'s output. This makes `tx_B` a `cell_ref_parent`. `check_and_record_ancestors` evicts `tx_B` (and its descendants) to satisfy the ancestor limit.
3. After `add_entry` returns, query `tx_pool_info`. Observe `total_tx_size` is larger than the sum of sizes of all remaining pool entries — inflated by the size of the evicted `tx_B` chain.
4. Repeat: each such submission inflates `total_tx_size` further, eventually causing `limit_size` to evict unrelated valid transactions or causing `send_transaction` to return `Reject::Full` for transactions that fit within the actual pool capacity. [8](#0-7) [9](#0-8)

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
