### Title
Missing Receive-Path Byte-Size Guard in `TransactionsProcess::execute` Allows Oversized `RelayTransactions` Messages — (`sync/src/relayer/transactions_process.rs`)

### Summary

`TransactionsProcess::execute` deserializes every transaction in a `RelayTransactions` message before applying any filter, and imposes no byte-size cap on the incoming message. The send path (`GetTransactionsProcess::execute`) enforces `MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB), but the receive path has no symmetric guard. A peer that has been designated as the `requesting_peer` for N tx hashes can send a single `RelayTransactions` message whose total serialized size far exceeds 1 MiB, forcing the receiving node to deserialize all transactions before any filtering occurs.

### Finding Description

**Send path guard (present):**
`GetTransactionsProcess::execute` enforces the byte budget before sending: [1](#0-0) 

**Receive path guard (absent):**
`TransactionsProcess::execute` maps (deserializes) every transaction before filtering: [2](#0-1) 

The `.map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))` call at line 48 runs `to_entity()` (copies raw bytes into an owned `packed::Transaction`) and `into_view()` (builds a `TransactionView`) for every transaction in the message. The `.filter(...)` at lines 49–55 runs only after deserialization of each element. Because Rust iterators are lazy, deserialization is interleaved with filtering — one transaction at a time — but there is no early exit or total-byte check before the loop begins. [3](#0-2) 

**Constants:** [4](#0-3) 

**Rate limiter** (30 req/s per `(peer, message_type)`) is the only upstream guard, but it limits message frequency, not message size: [5](#0-4) 

**Entry point** — the message reaches `TransactionsProcess` after only a structural `check_data()` validation: [6](#0-5) 

### Impact Explanation

- **I/O saturation**: At 30 `RelayTransactions` messages/second (rate-limiter ceiling), each carrying e.g. 32 transactions × ~100 KiB = ~3.2 MiB, the receiving node must read ~96 MiB/s of attacker-controlled data through the relay protocol handler.
- **CPU/allocator pressure**: Each transaction triggers `to_entity()` (byte copy) + `into_view()` (struct construction) before the filter check. Even though the lazy iterator means only one transaction is live at a time, the allocator is exercised for every transaction in every message.
- **No ban triggered**: The code path returns `Status::ok()` when all transactions are filtered out (e.g., because the attacker crafted hashes that are not in `unknown_tx_hashes`), so the peer is never penalized. [7](#0-6) 

### Likelihood Explanation

The precondition — being the `requesting_peer` for N tx hashes — is reachable through normal protocol interaction:
1. Attacker announces N tx hashes via `RelayTransactionHashes`.
2. Victim node adds them to `unknown_tx_hashes` and sends `GetRelayTransactions` back to the attacker, making the attacker the `requesting_peer`.
3. Attacker responds with a `RelayTransactions` message containing those N hashes mapped to oversized transactions (large witness fields).

No privileged access, no PoW, no key material is required. Any connected peer can execute this.

### Recommendation

Add a byte-size guard at the top of `TransactionsProcess::execute`, symmetric with the send-path guard in `GetTransactionsProcess::execute`:

```rust
pub fn execute(self) -> Status {
    // Guard: reject messages whose total serialized size exceeds the batch budget.
    let total_bytes: usize = self.message.transactions().iter()
        .map(|tx| tx.as_slice().len())
        .sum();
    if total_bytes > MAX_RELAY_TXS_BYTES_PER_BATCH {
        // optionally ban the peer
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("RelayTransactions exceeds MAX_RELAY_TXS_BYTES_PER_BATCH");
    }
    // ... existing logic
}
```

This check operates on the raw reader bytes (no deserialization) and mirrors the invariant already enforced on the send path. [8](#0-7) 

### Proof of Concept

1. Connect to a target node as a peer.
2. Send `RelayTransactionHashes` with 32 synthetic tx hashes. The target will issue `GetRelayTransactions` for them, registering the attacker as `requesting_peer`.
3. Craft a `RelayTransactions` message with those 32 hashes, each transaction body padded to ~100 KiB via witness data. Total message size: ~3.2 MiB.
4. Send the message. Observe: `TransactionsProcess::execute` deserializes all 32 transactions (calling `to_entity().into_view()` on each) before the filter runs. No ban is issued. `Status::ok()` is returned.
5. Repeat at the rate-limiter ceiling (30/s). The node sustains ~96 MiB/s of forced deserialization work with no penalty to the attacker.

    pub fn execute(self) -> Status {
        let shared_state = self.relayer.shared().state();
        let txs: Vec<(TransactionView, Cycle)> = {
            // ignore the tx if it's already known or it has never been requested before
            let mut tx_filter = shared_state.tx_filter();
            tx_filter.remove_expired();
            let unknown_tx_hashes = shared_state.unknown_tx_hashes();

            self.message
                .transactions()
                .iter()
                .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
                .filter(|(tx, _)| {
                    !tx_filter.contains(&tx.hash())
                        && unknown_tx_hashes
                            .get_priority(&tx.hash())
                            .map(|priority| priority.requesting_peer() == Some(self.peer))
                            .unwrap_or_default()
                })
                .collect()
        };
