Audit Report

## Title
Unbounded `pending_get_block_proposals` DashMap Growth via Peer-Supplied `GetBlockProposal` Messages — (File: `sync/src/types/mod.rs`)

## Summary
`insert_get_block_proposals` inserts peer-supplied `ProposalShortId` values into `pending_get_block_proposals: DashMap` with no size cap. Any unauthenticated P2P peer can flood this map with fabricated proposal IDs absent from the tx pool. The periodic drain in `prune_tx_proposal_request` performs a full `.clone()` of the map before clearing it, transiently doubling peak memory. With enough attacker peers, this causes unbounded memory growth and can OOM-kill the node.

## Finding Description
`GetBlockProposalProcess::execute()` in `sync/src/relayer/get_block_proposal_process.rs` validates only that the per-message proposal count does not exceed `max_block_proposals_limit * max_uncles_num` (≤ 3,000 on mainnet). [1](#0-0) 

Proposals absent from the tx pool are filtered and forwarded unconditionally to `insert_get_block_proposals`: [2](#0-1) 

`insert_get_block_proposals` performs no size check before inserting into the shared `DashMap`: [3](#0-2) 

The map is initialized as an unbounded `DashMap::new()` with no capacity argument or eviction policy: [4](#0-3) 

The only relief valve is `drain_get_block_proposals`, which performs a full O(n) `.clone()` before clearing — transiently doubling peak memory at every drain tick: [5](#0-4) 

The drain is triggered by `TX_PROPOSAL_TOKEN` every 100 ms: [6](#0-5) [7](#0-6) 

The rate limiter is 30 req/s per `(PeerIndex, message_id)` pair: [8](#0-7) [9](#0-8) 

**Attack arithmetic per 100 ms drain window:**
- 30 req/s × 3,000 proposals/req × 0.1 s = 9,000 entries per peer
- With N attacker peers: N × 9,000 entries accumulated before each drain
- Each entry: `ProposalShortId` (10 B) + `HashSet<PeerIndex>` + DashMap shard overhead (~100–200 B realistic)
- At 128 peers (a plausible connection count): ~1,152,000 entries × ~150 B ≈ **~173 MB** per drain cycle; the `.clone()` doubles this to **~346 MB** transiently before the clear

If `fetch_txs` inside `prune_tx_proposal_request` is slow (tx pool under load), the 100 ms notify fires again before the previous drain completes. Because `drain_get_block_proposals` clears the map at the start of the drain, a slow drain means the map re-fills during the await, and the next drain clone is again large — creating a positive feedback loop. [10](#0-9) 

## Impact Explanation
This matches the allowed CKB bounty impact: **"Vulnerabilities which could easily crash a CKB node" (High, 10001–15000 points)**. The victim node process is OOM-killed, taking it fully offline and preventing it from relaying or confirming transactions. The transient memory spike from the full-map clone can trigger OOM even before steady-state is reached.

## Likelihood Explanation
Triggering this requires only a standard unauthenticated P2P connection — no stake, no miner role, no authentication. Fabricating `ProposalShortId` values absent from the tx pool is trivial (random 10-byte values). The rate limiter (30 req/s per peer) slows but does not prevent the attack; multiple Sybil connections multiply the effect linearly. No hard cap or eviction policy guards `pending_get_block_proposals`. The attack is repeatable and requires no victim mistakes.

## Recommendation
Add a hard cap inside `insert_get_block_proposals`:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    const MAX_PENDING_PROPOSALS: usize = 10_000;
    if self.pending_get_block_proposals.len() >= MAX_PENDING_PROPOSALS {
        return;
    }
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
```

Additionally, replace the full-clone drain with `DashMap::drain()` or an atomic swap (e.g., `std::mem::replace` on a `Mutex<HashMap>`) to eliminate the transient memory doubling in `drain_get_block_proposals`.

## Proof of Concept
1. Connect to a victim CKB node as a standard P2P peer (no special privileges required).
2. Repeatedly send `GetBlockProposal` relay messages containing 3,000 random `ProposalShortId` values (none present in the victim's tx pool) at 30 messages/second per connection.
3. Open as many such connections as the node accepts (up to its peer limit).
4. Monitor the victim process RSS — it climbs by tens to hundreds of MB per 100 ms drain interval depending on peer count.
5. With sustained flooding from enough peers, the transient clone spike during `drain_get_block_proposals` triggers OOM and the victim process is killed.
