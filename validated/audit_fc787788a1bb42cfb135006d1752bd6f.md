### Title
Missing Upper Bound on `deviationThresholdBps` Allows Permanently Unsatisfiable Update Condition, Freezing Subscription Price Updates - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

`_validateSubscriptionParams` in `Scheduler.sol` only checks that `deviationThresholdBps != 0` but imposes no upper bound. A subscription manager can set `deviationThresholdBps` to any value up to `type(uint32).max` (4,294,967,295 bps ≈ 42,949,672%). Because the on-chain deviation calculation is bounded by realistic price movements (typically ≤ 10,000 bps for a 100% move), an impossibly high threshold makes the deviation condition permanently unsatisfiable. When combined with `updateOnHeartbeat = false`, every call to `updatePriceFeeds` reverts with `UpdateConditionsNotMet`, permanently freezing price updates for that subscription. For permanent subscriptions (which cannot be modified after creation), this also permanently locks up to 100 ETH of subscription funds.

### Finding Description

`_validateSubscriptionParams` validates `deviationThresholdBps` only against zero:

```solidity
if (
    params.updateCriteria.updateOnDeviation &&
    params.updateCriteria.deviationThresholdBps == 0
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [1](#0-0) 

There is no check that `deviationThresholdBps <= 10_000` (100%) or any other meaningful upper bound. The field is `uint32`, accepting values up to 4,294,967,295.

The deviation check in `_validateShouldUpdatePrices` computes:

```solidity
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
    return updateTimestamp;
}
``` [2](#0-1) 

`deviationBps` is `|currentPrice - previousPrice| * 10_000 / |previousPrice|`. For a 100% price move, this equals 10,000. For any realistic asset, this value is bounded well below `type(uint32).max`. If `deviationThresholdBps` is set to, say, `type(uint32).max`, the condition `deviationBps >= deviationThresholdBps` is never satisfied.

When `updateOnHeartbeat = false` and `updateOnDeviation = true` with an impossibly high threshold, the function always falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [3](#0-2) 

This is structurally identical to the Olympus M-20 bug: a missing boundary check on a parameter setter allows an extreme value that makes a downstream condition permanently unsatisfiable, freezing updates.

For **permanent subscriptions** (`isPermanent = true`), the subscription cannot be modified after creation:

```solidity
if (currentParams.isPermanent) {
    revert SchedulerErrors.CannotUpdatePermanentSubscription();
}
``` [4](#0-3) 

The deposit limit for permanent subscriptions is 100 ETH:

```solidity
if (subscriptionParams.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
``` [5](#0-4) [6](#0-5) 

A permanent subscription created with `deviationThresholdBps = type(uint32).max` and `updateOnHeartbeat = false` can never have its prices updated and its balance (up to 100 ETH) is permanently locked with no recovery path.

### Impact Explanation

1. **Frozen price feeds**: Any subscription with `updateOnDeviation = true`, `updateOnHeartbeat = false`, and `deviationThresholdBps > any achievable deviation` will have `updatePriceFeeds` permanently revert. Downstream protocols reading prices via `getPricesNoOlderThan` will receive `StalePrice` errors, breaking integrations.

2. **Permanent fund lock**: For permanent subscriptions, up to 100 ETH is locked with no withdrawal path, since `updatePriceFeeds` always reverts (fees are never deducted, keeper is never paid, and there is no separate `withdrawFunds` function for permanent subscriptions).

3. **Stale price data served**: `getPricesUnsafe` continues to return the last stored (stale) price without reverting, meaning protocols that use `getPricesUnsafe` instead of `getPricesNoOlderThan` silently consume stale data.

### Likelihood Explanation

- Any user can call `createSubscription` or `updateSubscription` with an arbitrary `deviationThresholdBps` value — no privileged role is required.
- A protocol integrator could accidentally set `deviationThresholdBps` to a large value (e.g., `100_000` intending "1000%") without realizing the deviation calculation is capped at ~10,000 bps for normal price moves.
- A malicious actor could deliberately create a permanent subscription with a locked threshold to grief a protocol that relies on a specific subscription ID.
- The validation gap is subtle: the only existing check is `!= 0`, which passes for any large value.

### Recommendation

Add an upper bound check in `_validateSubscriptionParams`:

```solidity
if (
    params.updateCriteria.updateOnDeviation &&
    (params.updateCriteria.deviationThresholdBps == 0 ||
     params.updateCriteria.deviationThresholdBps > 10_000)
) {
    revert SchedulerErrors.InvalidUpdateCriteria();
}
``` [1](#0-0) 

This mirrors the Olympus fix: change `> 10000` to `>= 10000` (or equivalently, add `> 10_000` as an upper bound). The value 10,000 bps (100%) is the natural semantic ceiling for a deviation threshold expressed in basis points of the previous price.

### Proof of Concept

```solidity
// 1. Create a subscription with deviation-only update criteria and impossibly high threshold
SchedulerStructs.UpdateCriteria memory criteria = SchedulerStructs.UpdateCriteria({
    updateOnHeartbeat: false,
    heartbeatSeconds: 0,
    updateOnDeviation: true,
    deviationThresholdBps: type(uint32).max  // 4,294,967,295 bps — passes validation (only checks != 0)
});

SchedulerStructs.SubscriptionParams memory params = SchedulerStructs.SubscriptionParams({
    priceIds: priceIds,
    readerWhitelist: new address[](0),
    whitelistEnabled: false,
    isActive: true,
    isPermanent: true,  // Cannot be updated after creation
    updateCriteria: criteria
});

uint256 subId = scheduler.createSubscription{value: 100 ether}(params);

// 2. Attempt to update prices — always reverts with UpdateConditionsNotMet
// because deviationBps (max ~10,000 for 100% move) < type(uint32).max
scheduler.updatePriceFeeds(subId, updateData);
// → reverts: UpdateConditionsNotMet

// 3. 100 ETH is permanently locked; prices are permanently stale
// getPricesNoOlderThan will revert with StalePrice for all callers
```

The `_validateSubscriptionParams` function accepts `deviationThresholdBps = type(uint32).max` because the only check is `!= 0`. [1](#0-0)  The deviation calculation in `_validateShouldUpdatePrices` can never produce a value near `type(uint32).max` for any real asset price, so the condition is permanently unsatisfiable.
