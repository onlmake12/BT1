### Title
Stale Ancestor `evict_key` After Batch Removal Due to Pre-cleared Links — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::remove_entry_and_descendants`, all link entries for the removed transactions are erased in a first pass before `remove_entry` is called for each id. Inside `remove_entry`, `update_ancestors_index_key` queries those same links to find which pool-resident ancestors need their `evict_key` updated. Because the links are already gone, the query returns an empty set and no ancestor `evict_key` is ever corrected. Ancestors that remain in the pool permanently carry inflated descendant-weight metadata, corrupting eviction ordering for the lifetime of those entries.

---

### Finding Description

`remove_entry_and_descendants` operates in two sequential loops:

**Loop 1 — pre-clear all links:** [1](#0-0) 

```rust
// update links state for remove, so that we won't update_descendants_index_key in remove_entry
for id in &removed_ids {
    self.remove_entry_links(id);
}
```

`remove_entry_links` calls `self.links.remove(id)`, which deletes the id's own entry from `TxLinksMap::inner`. [2](#0-1) 

**Loop 2 — remove each entry:** [3](#0-2) 

Inside `remove_entry`, the first call is: [4](#0-3) 

```rust
self.update_ancestors_index_key(&entry.inner, EntryOp::Remove);
```

`update_ancestors_index_key` calls `self.links.calc_ancestors(&child.proposal_short_id())`: [5](#0-4) 

`calc_ancestors` starts from the id's own link entry in `TxLinksMap::inner`. Because Loop 1 already called `self.links.remove(id)` for every removed id, the entry is gone. `calc_ancestors` returns an empty `HashSet`, the `for anc_id in &ancestors` body never executes, and **no ancestor's `evict_key` is updated**.

The analog to the `killGaugeTotally` bug is exact:

| Step | Solidity original | CKB analog |
|---|---|---|
| 1 | `delete poolForGauge[_gauge]` | `self.links.remove(id)` for all removed ids |
| 2 | `pool = poolForGauge[_gauge]` → returns `address(0)` | `calc_ancestors(id)` → returns `∅` |
| 3 | `gauges[pool]` never cleared | ancestor `evict_key` never updated |

---

### Impact Explanation

Every ancestor transaction that remains in the pool after a `remove_entry_and_descendants` call retains an inflated `evict_key` (it still accounts for the descendant weight of the removed transactions). The `evict_key` is the sole ordering criterion used by `next_evict_entry` to select which transaction to evict when the pool is full: [6](#0-5) 

Consequences:
- **Wrong eviction order**: an ancestor with inflated descendant weight appears "heavier" than it truly is, causing it to be evicted earlier or later than correct ordering demands.
- **Persistent corruption**: the stale `evict_key` is never corrected; it persists until the ancestor itself is removed.
- **Manipulable pool state**: an attacker can deliberately leave ancestors with inflated keys to protect low-fee transactions from eviction or to force eviction of high-fee competitors.

---

### Likelihood Explanation

The trigger path is fully reachable by any unprivileged tx-pool submitter:

1. Attacker submits a parent transaction P and a child transaction C (P → C) to the pool.
2. Attacker submits a new transaction that spends the same input as C, triggering `resolve_conflict`: [7](#0-6) 
3. `resolve_conflict` calls `remove_entry_and_descendants(&id_of_C)`.
4. P remains in the pool with a stale `evict_key` that still includes C's descendant weight.

No special privilege, key, or majority hashpower is required. The attack requires only two ordinary transactions and is repeatable.

---

### Recommendation

Capture the ancestor set **before** clearing links, or restructure the removal so that `update_ancestors_index_key` is called while links are still intact. One correct approach:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Collect ancestor sets BEFORE clearing any links
    for removed_id in &removed_ids {
        if let Some(entry) = self.entries.get_by_id(removed_id) {
            self.update_ancestors_index_key(&entry.inner.clone(), EntryOp::Remove);
        }
    }

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    removed_ids
        .iter()
        .filter_map(|id| self.remove_entry_skip_ancestor_update(id))
        .collect()
}
```

Alternatively, pass the pre-computed ancestor sets into `remove_entry` so it does not need to re-query the already-cleared `links`.

### Proof of Concept

**State before attack:**
- Pool contains: `P` (parent, fee=100), `C` (child of P, fee=10)
- `P.evict_key` includes C's descendant weight

**Attack:**
1. Submit `C2` spending the same input as `C` → `resolve_conflict` → `remove_entry_and_descendants(&C.id)`
2. Loop 1: `remove_entry_links(C.id)` → `links.inner.remove(C.id)` executed
3. Loop 2: `remove_entry(C.id)` → `update_ancestors_index_key(C, Remove)` → `calc_ancestors(C.id)` → `links.inner.get(C.id)` returns `None` → empty ancestor set → **P's `evict_key` not updated**

**State after attack:**
- Pool contains: `P` with `evict_key` still reflecting C's (now-removed) descendant weight
- `next_evict_entry` uses this stale key, producing incorrect eviction ordering
- Attacker can repeat to accumulate multiple ancestors with inflated keys, systematically distorting pool eviction for the duration of those entries' residence [8](#0-7) [9](#0-8) [5](#0-4)

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }
