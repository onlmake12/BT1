### Title
Node Process Crash via Stale `parents` Entry After Cascading Eviction in `check_and_record_ancestors` — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

A real crash exists in `check_and_record_ancestors`. The eviction loop removes a `cell_ref_parent` and all its descendants in one call, but only decrements `ancestors_count` by 1 and only removes the directly-evicted ID from `parents`. When a descendant of the evicted tx is also a `cell_ref_parent`, it remains in `parents` after the loop exits. The subsequent `calc_relation_ids` call returns that stale ID as an ancestor, and `_record_ancestors` then calls `get_by_id_checked` on it — which panics with `"inconsistent pool"` because the entry no longer exists.

The question's specific claim that `assert!(ancestors.len() < self.max_ancestors_count)` at line 636 fires is **incorrect** — that assert passes. The actual crash is one step later in `_record_ancestors`.

---

### Finding Description

**Root cause — three independent bugs that compose:**

**Bug 1:** `ancestors_count` is decremented by exactly 1 per loop iteration regardless of how many entries `remove_entry_and_descendants` actually removes. [1](#0-0) 

**Bug 2:** Only the directly-iterated candidate is removed from `parents`; its descendants that were also removed are not. [2](#0-1) 

**Bug 3:** `calc_relation_ids` silently includes IDs that are no longer in `self.links` — it adds them to `relation_ids` without finding any parents, so they appear as valid ancestors. [3](#0-2) 

**Crash site:** `_record_ancestors` calls `get_by_id_checked` on every ID in the recomputed `ancestors` set. If any of those IDs was removed from the pool, the `.expect("inconsistent pool")` panics. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

An unprivileged attacker can crash any CKB full node by submitting a crafted sequence of transactions via the standard `send_transaction` RPC. The node process panics and terminates. This is a remote denial-of-service with no authentication or special role required.

---

### Likelihood Explanation

The attack requires only valid CKB transactions submitted through the public RPC. No PoW, no key material, no privileged access. The setup (a chain of pool transactions where two share a cell dep on the same outpoint) is straightforward to construct. The default `max_ancestors_count` is 25, so the attacker needs a chain of 25 pool transactions with the first two sharing a cell dep — well within normal transaction construction.

---

### Recommendation

Fix all three bugs together:

1. After `remove_entry_and_descendants(next_id)`, decrement `ancestors_count` by `removed.len()` (the actual number of entries removed), not by 1.
2. After removal, also remove every returned entry's ID from `parents`, not just `next_id`.
3. After the loop, filter `parents` to only IDs still present in `self.links` before calling `calc_relation_ids`.

---

### Proof of Concept

**Setup** (`max_ancestors_count = N`, default 25):

1. Submit **tx_1**: spends confirmed input X; has `cell_dep` on confirmed output P; produces output O.
2. Submit **tx_2**: spends output O of tx_1 (making tx_2 a descendant of tx_1); also has `cell_dep` on P.
3. Submit **tx_3 … tx_N**: a chain, each spending the previous tx's output (no cell dep on P needed). This pushes `ancestors_count` for the next tx to N+1.
4. Submit **tx_C**: spends output P as an input.

**Execution trace in `check_and_record_ancestors` for tx_C:**

- `get_tx_ancenstors` returns `cell_ref_parents = {tx_1, tx_2}`, `ancestors_count = N+1`. [6](#0-5) 
- Guard: `(N+1) - 2 = N-1 ≤ N` → enters eviction branch. [7](#0-6) 
- Loop iteration 1 (`next_id = tx_1`):
  - `remove_entry_and_descendants(tx_1)` removes **tx_1, tx_2, tx_3 … tx_N** (entire chain). [8](#0-7) 
  - `ancestors_count = N+1 - 1 = N`. Loop exits (`N > N` is false).
  - `parents.remove(tx_1)` → `parents = {tx_2}`. **tx_2 is still in `parents` despite being removed from the pool.**
- Recompute: `calc_relation_ids({tx_2}, Parents)` → tx_2 not in `self.links` → returns `{tx_2}`. [3](#0-2) 
- Assert `1 < N` → passes (no crash here). [9](#0-8) 
- `_record_ancestors(entry, {tx_2}, {tx_2})` → `get_by_id_checked(tx_2)` → `.expect("inconsistent pool")` → **PANIC, node crashes**. [10](#0-9)
