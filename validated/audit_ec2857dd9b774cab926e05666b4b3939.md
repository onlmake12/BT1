### Title
Truncated Price ID Prefix Stored in Echo Requests Allows Price Feed Substitution — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `requestPriceUpdatesWithCallback` stores only the first 8 bytes (`bytes8`) of each requested 32-byte price feed ID. The `executeCallback` function then verifies only those 8 bytes when a provider fulfills the request. Because the full 32-byte price ID is never stored or fully verified, any price feed whose first 8 bytes collide with the originally requested feed can be substituted by the fulfilling provider, causing the consumer's callback to receive price data for the wrong asset.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the contract truncates each 32-byte `priceId` to its first 8 bytes and stores only that prefix:

```solidity
// Echo.sol lines ~87-98
req.priceIdPrefixes = new bytes8[](priceIds.length);
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

In `executeCallback`, the verification loop only checks these 8-byte prefixes against the caller-supplied `priceIds`:

```solidity
// Echo.sol lines ~128-141
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly { prefix := priceId }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
``` [2](#0-1) 

The caller-supplied `priceIds` (only prefix-verified) are then passed directly to `parsePriceFeedUpdates`:

```solidity
// Echo.sol lines ~146-153
PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{value: pythFee}(
    updateData,
    priceIds,   // <-- only 8 of 32 bytes were verified
    SafeCast.toUint64(req.publishTime),
    SafeCast.toUint64(req.publishTime)
);
``` [3](#0-2) 

The consumer's `_echoCallback` then receives the `priceFeeds` array returned for the substituted price ID, not the originally requested one. [4](#0-3) 

The `EchoState.Request` struct confirms that only `bytes8[] priceIdPrefixes` is stored — the full 32-byte IDs are never persisted: [5](#0-4) 

The error type itself acknowledges the asymmetry — it stores a `bytes8` on one side and a `bytes32` on the other: [6](#0-5) 

---

### Impact Explanation

Any provider (or, after the exclusivity period elapses, any unprivileged caller) can call `executeCallback` with a `priceId` that shares the same first 8 bytes as the originally requested feed but refers to a completely different asset. The 8-byte prefix check passes, `parsePriceFeedUpdates` returns data for the substituted feed, and the consumer's `_echoCallback` is invoked with incorrect price data. A consumer contract making financial decisions (e.g., liquidations, collateral valuations) based on the callback price would operate on wrong data, potentially leading to incorrect liquidations, mispriced collateral, or loss of funds.

---

### Likelihood Explanation

Pyth price feed IDs are 32-byte Keccak-derived identifiers. With hundreds of active feeds, the probability of a natural 8-byte prefix collision is low but non-negligible (birthday bound over ~500 feeds against a 64-bit space). More critically, the design is structurally unsound: the stored identifier is provably incomplete. After the configurable `exclusivityPeriodSeconds` elapses, **any** unprivileged address can call `executeCallback`, making the attacker-controlled entry path fully permissionless. The attacker only needs to find (or wait for) any two feeds sharing an 8-byte prefix, or exploit the fact that the Pyth network may add new feeds in the future that collide with existing ones.

---

### Recommendation

Store the full 32-byte price IDs in the `Request` struct instead of the truncated 8-byte prefixes. Replace `bytes8[] priceIdPrefixes` with `bytes32[] priceIds` and perform a full equality check in `executeCallback`. The gas savings from truncation do not justify the security trade-off.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `priceIds = [FEED_A]` where `FEED_A = 0xAABBCCDDEEFF00112233...` (32 bytes). The contract stores `priceIdPrefixes[0] = 0xAABBCCDDEEFF0011` (first 8 bytes only).

2. Attacker identifies `FEED_B = 0xAABBCCDDEEFF0011DEADBEEF...` — a different Pyth price feed whose first 8 bytes match `FEED_A`.

3. After `exclusivityPeriodSeconds` elapses, attacker calls:
   ```solidity
   echo.executeCallback(
       attacker,          // providerToCredit
       sequenceNumber,
       updateDataForFeedB,
       [FEED_B]           // passes 8-byte prefix check
   );
   ```

4. The prefix check `0xAABBCCDDEEFF0011 == 0xAABBCCDDEEFF0011` passes. `parsePriceFeedUpdates` is called with `FEED_B` and returns `FEED_B`'s price. The consumer's `_echoCallback` receives the price of `FEED_B` instead of `FEED_A`, with no revert or warning.
