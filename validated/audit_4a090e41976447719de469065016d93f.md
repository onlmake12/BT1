### Title
Malicious Provider Can Register With Invalid Commitment, Collect User Fees, and Never Fulfill Requests — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `Entropy` contract allows anyone to permissionlessly register as a randomness provider by posting an arbitrary `commitment` bytes32 value. There is no on-chain validation that the commitment is the root of a valid hash chain. A malicious provider can register with a fake commitment, attract users who pay fees for random number requests, immediately withdraw those fees, and then be permanently unable to fulfill any request — leaving all in-flight requests permanently stuck with no refund path for users.

---

### Finding Description

The `register()` function accepts any `bytes32 commitment` without verifying it is the root of a valid hash chain:

```solidity
function register(
    uint128 feeInWei,
    bytes32 commitment,
    bytes calldata commitmentMetadata,
    uint64 chainLength,
    bytes calldata uri
) public override {
    if (chainLength == 0) revert EntropyErrors.AssertionFailure();
    // ...
    provider.currentCommitment = commitment;  // no validity check
    provider.endSequenceNumber = provider.sequenceNumber + chainLength;
    // ...
}
``` [1](#0-0) 

When a user calls `requestV2()` / `requestWithCallback()`, the fee is **immediately** credited to `providerInfo.accruedFeesInWei` before any fulfillment occurs:

```solidity
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
``` [2](#0-1) 

The provider can then immediately call `withdraw()` to drain those fees: [3](#0-2) 

When anyone later attempts to call `revealWithCallback()`, the `revealHelper()` function will always revert with `IncorrectRevelation` because the attacker cannot produce a valid hash chain preimage for a fake commitment:

```solidity
bytes32 providerCommitment = constructProviderCommitment(req.numHashes, providerContribution);
if (keccak256(bytes.concat(userCommitment, providerCommitment)) != req.commitment)
    revert EntropyErrors.IncorrectRevelation();
``` [4](#0-3) 

There is no timeout, no refund mechanism, and no way for users to recover their paid fees once a request is in-flight. The `IEntropy` interface explicitly documents that excess value is not refunded, and there is no cancellation path at all: [5](#0-4) 

A second variant of this attack uses **re-registration**. A provider can operate legitimately to build trust, then call `register()` again with a fake commitment. The `sequenceNumber` is not reset on re-registration — it continues incrementing — so all new requests (assigned sequence numbers after the rotation) will be permanently unfulfillable, while the provider collects and withdraws fees from those new requests: [6](#0-5) 

---

### Impact Explanation

- Users who request random numbers from a malicious provider permanently lose their paid fees (provider fee + Pyth protocol fee).
- Downstream consumer contracts (e.g., NFT mints, on-chain lotteries, games) that depend on the `entropyCallback` are permanently stuck in a pending state with no recovery path.
- The malicious provider profits by withdrawing the accrued fees immediately after requests are submitted.
- The re-registration variant allows a previously trusted provider to rug users at scale after building reputation.

---

### Likelihood Explanation

Provider registration is fully permissionless — any address can call `register()`. The attack requires no privileged access, no leaked keys, and no external oracle manipulation. A malicious actor only needs to:
1. Deploy or use an EOA as a provider address.
2. Call `register()` with a fake `commitment` and a large `chainLength`.
3. Advertise a competitive fee to attract users.
4. Wait for requests to accumulate, then call `withdraw()`.

The re-registration variant is even more realistic: a legitimate provider who decides to exit can rotate to a fake commitment, collect one final round of fees, and disappear.

---

### Recommendation

1. **Require fee escrow until fulfillment**: Do not credit `providerInfo.accruedFeesInWei` at request time. Instead, hold the provider fee in escrow and only release it upon successful `reveal`/`revealWithCallback`. Refund the user if the request is not fulfilled within a timeout window.

2. **Add a request timeout and refund path**: Introduce a deadline (e.g., based on block number) after which an unfulfilled request can be cancelled by the user, returning their fee.

3. **Validate commitment on registration**: While it is impossible to fully validate a hash chain root on-chain without the full chain, requiring providers to post a bond (slashable on non-fulfillment) would create an economic disincentive for fake registrations.

---

### Proof of Concept

```
1. Attacker calls register(feeInWei=1 ether, commitment=bytes32(0xdeadbeef...), chainLength=1_000_000, uri="https://evil.example")
   → Attacker is now a registered provider with a fake commitment.

2. User U calls requestV2{value: 1 ether + pythFee}(attackerAddress, userRandomNumber, 0)
   → providerInfo.accruedFeesInWei += 1 ether  (immediately)
   → Request stored on-chain with callbackStatus = CALLBACK_NOT_STARTED

3. Attacker calls withdraw(1 ether)
   → Attacker receives 1 ether. Fees are gone.

4. Anyone calls revealWithCallback(attackerAddress, sequenceNumber, userContribution, anyBytes32)
   → constructProviderCommitment(numHashes, anyBytes32) != 0xdeadbeef...
   → Reverts with IncorrectRevelation every time.

5. User U's callback is never delivered. Their fee is permanently lost.
   Their downstream contract is permanently stuck.
```

        uint128 feeInWei,

        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.

        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0

        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;

        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));

        require(
            providerInfo.accruedFeesInWei >= amount,
