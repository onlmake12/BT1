### Title
Hardcoded Default Cycle Limit in Tx-Pool Admission Creates Consensus/Pool Discrepancy — (`util/app-config/src/legacy/tx_pool.rs`)

### Summary
CKB's tx-pool uses a hardcoded default `max_tx_verify_cycles` of `TWO_IN_TWO_OUT_CYCLES * 20 = 70,000,000` cycles. The consensus-level `MAX_BLOCK_CYCLES` is `TWO_IN_TWO_OUT_CYCLES * 1,000 = 3,500,000,000` cycles — a 50× gap. Any transaction whose scripts consume between 70 M and 3.5 B cycles is **valid by consensus** but **rejected by the tx-pool** on every node running the default configuration. An unprivileged RPC caller or tx-pool submitter who submits such a transaction will receive a permanent rejection with no indication that the transaction is actually consensus-valid.

### Finding Description
`TWO_IN_TWO_OUT_CYCLES` is defined as a hardcoded constant: [1](#0-0) 

`MAX_BLOCK_CYCLES` (the consensus block-level ceiling) is derived from it: [2](#0-1) 

The tx-pool's default admission limit is also derived from the same constant, but at only 2% of the block ceiling: [3](#0-2) 

The `TxPoolConfig` field is documented as a hard rejection gate — not a soft hint: [4](#0-3) 

The default is baked into the legacy deserialization path and becomes the operative value for any node that does not explicitly override `max_tx_verify_cycles` in `ckb.toml`: [5](#0-4) 

### Impact Explanation
A transaction author who writes a lock or type script that legitimately consumes, say, 100 M cycles (well within `MAX_BLOCK_CYCLES = 3.5 B`) will find their transaction permanently rejected by every default-configured full node's tx-pool. The rejection error (`ExceededMaximumCycles`) gives no indication that the transaction is consensus-valid. The transaction cannot propagate through the P2P relay layer and cannot be mined unless a miner directly injects it into their own block assembler — a path unavailable to ordinary users. This is the direct CKB analog of the in3-server hardcoded gas limit: a static resource ceiling causes a valid operation to fail at the admission layer.

### Likelihood Explanation
Any script author deploying a moderately complex lock or type script (e.g., a multi-party computation, a ZK verifier, or a recursive script) can easily exceed 70 M cycles while remaining far below 3.5 B. The 50× gap between the default pool limit and the consensus limit is large enough that real-world scripts will hit it. The entry path is fully unprivileged: any RPC caller invoking `send_transaction` or any peer relaying a transaction triggers this path.

### Recommendation
Replace the hardcoded `DEFAULT_MAX_TX_VERIFY_CYCLES = TWO_IN_TWO_OUT_CYCLES * 20` with a value derived from `MAX_BLOCK_CYCLES` (or set it equal to `MAX_BLOCK_CYCLES`), so that the tx-pool admission limit is consistent with the consensus validity rule. If a lower limit is desired for DoS protection, it should be clearly documented as a policy cap and the rejection error should distinguish "rejected by local policy" from "invalid by consensus," preventing user confusion and silent transaction loss.

### Proof of Concept
1. Construct a transaction whose lock script loops for ~100 M cycles (valid under `MAX_BLOCK_CYCLES = 3,500,000,000`).
2. Submit via `send_transaction` RPC to a node running default `ckb.toml` (`max_tx_verify_cycles = 70_000_000`).
3. The tx-pool rejects the transaction with an `ExceededMaximumCycles` error.
4. The same transaction, if directly assembled into a block by a miner, passes consensus verification — confirming the discrepancy between the hardcoded pool limit and the consensus limit. [6](#0-5) [7](#0-6) [8](#0-7)
