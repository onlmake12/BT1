Audit Report

## Title
Dep Group Expansion Performs Up to 2048 Uncharged DB Reads Per Transaction — (File: `util/types/src/core/cell.rs`)

## Summary

CKB's dep group resolution loop in `resolve_transaction_dep` performs up to 2048 `cell_provider.cell()` (RocksDB) reads per transaction for user-deployed dep groups, none of which are charged in cycles or fees. The fee model charges only on serialized transaction size, creating a ~2048× work-to-fee amplification that affects every full node performing block verification.

## Finding Description

`MAX_DEP_EXPANSION_LIMIT` is set to 2048 at line 33 of `util/types/src/core/cell.rs`. [1](#0-0) 

In `resolve_transaction_dep` (lines 807–841), when `dep_type == DepGroup`, the function calls `cell_resolver` once eagerly for the dep group cell itself (line 817), then iterates over all sub-out-points (line 829–831), calling `cell_resolver` for each — up to `remaining_dep_slots` times, which starts at `MAX_DEP_EXPANSION_LIMIT = 2048`. [2](#0-1) 

The `resolved_cells` cache at line 696 is a local `HashMap` scoped to a single `resolve_transaction` call. It deduplicates reads within one transaction but provides no cross-transaction caching. [3](#0-2) 

The `SYSTEM_CELL` static cache (line 31) is populated exclusively from the genesis block's five hardcoded system cell deps in `setup_system_cell_cache` (lines 856–925). User-deployed dep groups are never inserted into this cache and always fall through to the `resolve_transaction_dep` path at lines 781–801. [4](#0-3) [5](#0-4) 

The fee check in `tx-pool/src/util.rs` at line 45 computes the minimum fee as `min_fee_rate.fee(tx_size as u64)`, where `tx_size` is the serialized transaction byte size — not the number of dep group expansions or DB reads performed. [6](#0-5) 

`transferred_byte_cycles` in `script/src/cost_model.rs` applies only to data transferred via VM syscalls, not to the resolution phase. [7](#0-6) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The fee/work ratio is approximately 2048× per transaction. An attacker pays fees proportional to ~200 bytes of serialized transaction data while causing 2049 RocksDB reads (1 for the dep group cell + 2048 for sub-cells) during resolution. Because `resolve_transaction` is called in both the tx-pool admission path and the mandatory block verification path, every full node on the network performs these reads for every such transaction in every block, not just the targeted node's mempool.

## Likelihood Explanation

The upfront cost is minimal: one dep group cell encoding 2048 out-points (73,728 bytes of data) plus 2048 minimal live cells (~0.126 CKB total). Once deployed, the attacker submits transactions indefinitely via the standard `send_transaction` RPC with no special privileges. The attack is fully repeatable without additional on-chain setup. The test suite at line 419 of `test/src/specs/hardfork/v2021/cell_deps.rs` explicitly constructs a 2048-entry dep group (`&[&code_ay0_tx; 2048]`), confirming the maximum expansion is reachable by design and accepted by the protocol. [8](#0-7) 

## Recommendation

Charge cycles proportional to the number of dep group sub-cell lookups performed during `resolve_transaction_dep`. A fixed per-lookup cycle cost (e.g., a flat constant or `transferred_byte_cycles` applied to the sub-cell's `data_bytes`) should be added inside the loop at line 829 and accumulated into the transaction's total cycle consumption so it is reflected in the fee requirement. Alternatively, extend `check_tx_fee` to include a per-dep-expansion surcharge in the minimum fee calculation, making the fee model account for resolution work in addition to serialized transaction size.

## Proof of Concept

1. Deploy a dep group cell whose data is an `OutPointVec` of 2048 valid out-points (data size: 2048 × 36 = 73,728 bytes).
2. Deploy 2048 minimal live cells to be referenced by the dep group.
3. Submit N transactions via `send_transaction` RPC, each containing a single `CellDep` pointing to the dep group cell with `dep_type = DepGroup`.
4. Each transaction triggers `resolve_transaction_dep` → 1 eager DB read for the dep group cell + 2048 DB reads for sub-cells = 2049 DB reads total, in both tx-pool admission and block verification.
5. Fee paid per transaction: ~200 bytes × `min_fee_rate` (negligible).
6. Total DB reads across N transactions: N × 2049, with fees proportional only to N × 200 bytes.
7. The existing test at `test/src/specs/hardfork/v2021/cell_deps.rs` line 419 (`create_depgroup_celldep` with `&[&code_ay0_tx; 2048]`) provides a direct template for the PoC setup and confirms 2048-entry dep groups are accepted.

        .out_point(OutPoint::new(system_cell_transaction.hash(), 2))
        .dep_type(DepType::Code)
        .build();

    let secp_data_dep = CellDep::new_builder()
        .out_point(OutPoint::new(system_cell_transaction.hash(), 3))
        .dep_type(DepType::Code)
        .build();

        .out_point(OutPoint::new(secp_cell_transaction.hash(), 0))
        .dep_type(DepType::DepGroup)

        .dep_type(DepType::DepGroup)

        build_cell_meta_from_out_point(cell_provider, &secp_code_dep.out_point())

    let dao_dep_cell = build_cell_meta_from_out_point(cell_provider, &dao_dep.out_point())
        .expect("resolve dao_dep_cell");

    );
