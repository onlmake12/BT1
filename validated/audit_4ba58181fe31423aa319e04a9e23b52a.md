### Title
Missing Payload Timestamp Freshness Check in `verifyUpdate()` Allows Stale Price Acceptance - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary

`PythLazer.sol`'s `verifyUpdate()` function verifies the ECDSA signature and checks that the signer is trusted (not expired), but performs **no validation that the `timestamp` embedded in the signed payload is recent relative to `block.timestamp`**. A validator can hold a `verifyUpdate` transaction in the mempool and include it in a block arbitrarily later; the stale price payload will still pass all checks and be returned to the consumer contract as valid.

### Finding Description

`verifyUpdate()` extracts the raw payload bytes and verifies the ECDSA signature over them, then checks `isValidSigner(signer)`: [1](#0-0) 

The payload contains a `uint64 timestamp` field at bytes 4–12, as parsed by `PythLazerLib.parsePayloadHeader`: [2](#0-1) 

`verifyUpdate()` never calls `parsePayloadHeader` and never compares the payload timestamp against `block.timestamp`. It returns the raw `payload` bytes to the caller with no staleness guarantee: [3](#0-2) 

By contrast, the Pyth Scheduler contract enforces `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` on every price update it processes: [4](#0-3) 

### Impact Explanation

Any consumer contract that calls `verifyUpdate()` and does not independently validate the returned payload's timestamp will silently accept arbitrarily stale price data. DeFi protocols (lending, derivatives, AMMs) built on Pyth Lazer that rely on `verifyUpdate()` as their sole freshness gate can be exploited: an attacker submits a transaction with a fresh signed payload, a validator delays inclusion, and the consumer contract executes financial logic (e.g., collateral valuation, liquidation, settlement) against a price that may be minutes or hours old.

### Likelihood Explanation

On EVM chains with active MEV infrastructure, validators routinely delay or reorder transactions. A signed Lazer payload is valid as long as the signer's `expiresAt` has not passed (which can be months away). Any window between payload signing and block inclusion is exploitable. Consumer contracts that follow the documentation example without adding their own staleness guard are directly affected.

### Recommendation

Add a `maxAge` parameter (or a hardcoded constant) to `verifyUpdate()` that parses the payload timestamp and reverts if it is too old:

```solidity
function verifyUpdate(
    bytes calldata update,
    uint64 maxAge          // e.g. 60 seconds
) external payable returns (bytes calldata payload, address signer) {
    // ... existing magic / length / signature checks ...

    // Parse timestamp from payload (bytes 4–12 after the 4-byte payload magic)
    uint64 payloadTimestamp = uint64(bytes8(payload[4:12]));
    require(
        block.timestamp <= payloadTimestamp + maxAge,
        "price update too old"
    );

    if (!isValidSigner(signer)) revert("invalid signer");
}
```

Alternatively, expose a view constant `MAX_PRICE_AGE` and enforce it internally, mirroring the pattern used in `SchedulerConstants.PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`.

### Proof of Concept

1. Attacker (or any user) calls `consumerContract.updatePrice(freshUpdate)` where `freshUpdate` is a legitimately signed Lazer payload with `timestamp = T`.
2. A validator observes the pending transaction and withholds it from the block for `Δt` seconds (e.g., 5 minutes during high volatility).
3. The validator includes the transaction at time `T + Δt`.
4. `verifyUpdate()` checks: magic ✓, signature ✓, `isValidSigner` ✓ — no timestamp check.
5. The consumer contract receives `payload` with `timestamp = T` and acts on a price that is `Δt` seconds stale.
6. If the asset moved 2% in those 5 minutes, the consumer's collateral calculation, liquidation threshold, or settlement price is materially wrong — enabling the validator (or a colluding party) to profit from the discrepancy. [5](#0-4) [6](#0-5)
