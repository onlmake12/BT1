### Title
`getPricesNoOlderThan` Staleness Check Uses Subscription-Level Max Timestamp Instead of Per-Feed `publishTime` — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions check freshness against a single subscription-level timestamp (`priceLastUpdatedAt`), which is the **maximum** `publishTime` across all feeds in the subscription. When a subscription contains feeds with heterogeneous trading hours (e.g., a 24/7 crypto feed alongside an equity or commodity feed), the staleness check passes based on the crypto feed's fresh timestamp, while the returned equity/commodity price can be arbitrarily stale. A consumer contract relying on `getPricesNoOlderThan` as the safe API receives a false freshness guarantee.

---

### Finding Description

In `_validateShouldUpdatePrices`, the subscription's `priceLastUpdatedAt` is set to the **maximum** `publishTime` across all price feeds in the batch: [1](#0-0) 

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This value is then stored as `status.priceLastUpdatedAt`. [2](#0-1) 

In `getPricesNoOlderThan`, the staleness check is: [3](#0-2) 

```solidity
SchedulerStructs.SubscriptionStatus memory status = _state
    .subscriptionStatuses[subscriptionId];

if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
```

The check compares `block.timestamp` against the **subscription-level max timestamp**, not against the `publishTime` of the specific `priceIds` being requested. After passing this check, `getPricesUnsafe` is called, which returns the raw stored price for each requested feed — with no per-feed freshness validation. [4](#0-3) 

The same flaw exists in `getEmaPricesNoOlderThan`: [5](#0-4) 

The `IScheduler` interface documents `getPricesNoOlderThan` as the safe alternative to `getPricesUnsafe`: [6](#0-5) 

The Pulse README also explicitly recommends `get(Ema)PricesNoOlderThan` as the safe read path: [7](#0-6) 

---

### Impact Explanation

A subscription containing both a 24/7 crypto feed (BTC/USD, `publishTime = now - 30s`) and a closed-market equity feed (AAPL/USD, `publishTime = now - 8h`) will have `priceLastUpdatedAt = now - 30s`.

A consumer calling `getPricesNoOlderThan(subscriptionId, [AAPL_ID], 60)` will:
1. Pass the staleness check: `distance(now, now-30s) = 30 < 60` ✓
2. Receive the AAPL price with `publishTime = now - 8h`

The consumer contract believes the AAPL price is no older than 60 seconds. It is actually 8 hours old. Any DeFi protocol (lending, derivatives, liquidation engine) using this function to gate critical operations will make decisions based on a stale price while believing it is fresh — the exact analog of the Fei `TribeReserveStabilizer` bug.

---

### Likelihood Explanation

The Scheduler is explicitly designed to support subscriptions with mixed-market feeds (the comment in `_validateShouldUpdatePrices` acknowledges closed-market assets). Any subscription combining crypto feeds with equity, commodity, or FX feeds — a common real-world use case — creates the conditions for this bug. The `getPricesNoOlderThan` function is the recommended safe API, so consumer contracts will naturally rely on it without adding their own per-feed `publishTime` checks.

---

### Recommendation

Replace the subscription-level staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with a per-feed check against each individual price feed's `publishTime`:

```solidity
function getPricesNoOlderThan(...) external view ... {
    PythStructs.PriceFeed[] memory priceFeeds = _getPricesInternal(subscriptionId, priceIds);
    prices = new PythStructs.Price[](priceFeeds.length);
    for (uint i = 0; i < priceFeeds.length; i++) {
        if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
            revert PythErrors.StalePrice();
        prices[i] = priceFeeds[i].price;
    }
}
```

This ensures the freshness guarantee is applied to each returned price individually, matching the semantics implied by the function name and documented in the interface.

---

### Proof of Concept

1. Deploy a `Scheduler` subscription with two price feeds: `CRYPTO_ID` (24/7) and `EQUITY_ID` (market hours).
2. A keeper calls `updatePriceFeeds` with an update where `CRYPTO_ID.publishTime = T` and `EQUITY_ID.publishTime = T - 8h` (market closed). Both share the same Pythnet slot, so the update is accepted. `priceLastUpdatedAt = T`.
3. 30 seconds later (`block.timestamp = T + 30`), a whitelisted consumer calls `getPricesNoOlderThan(subscriptionId, [EQUITY_ID], 60)`.
4. The check: `distance(T+30, T) = 30 < 60` → **passes**.
5. The returned `EQUITY_ID` price has `publishTime = T - 8h` — 8 hours stale.
6. The consumer contract proceeds to use this price for a liquidation or collateral valuation, making an incorrect decision based on a price it believes is at most 60 seconds old.
