### Title
Multi-Denom Fee Sufficiency Check Ignores Exchange Rate Between Accepted Tokens - (`target_chains/cosmwasm/contracts/pyth/src/contract.rs`)

### Summary

The Osmosis-specific `is_fee_sufficient` function in the CosmWasm Pyth contract accepts fees in multiple token denominations but sums their raw amounts without any exchange-rate conversion, treating all accepted tokens as 1:1 with the base denom (`uosmo`). This is the same vulnerability class as the VaderPool report: a value denominated in token A is compared against a threshold denominated in token B assuming parity.

---

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/contract.rs`, the `is_fee_sufficient` function (compiled only under `#[cfg(feature = "osmosis")]`) iterates over all coins sent by the caller and accumulates their raw `u128` amounts into a single `total_amount`, regardless of denomination: [1](#0-0) 

The accumulated `total_amount` is then compared directly against `base_denom_fee.amount`, which is expressed in `uosmo`: [2](#0-1) 

The companion function `get_update_fee_for_denom` returns the same raw amount for any valid denom — it does not convert the required fee into the target denom's equivalent value: [3](#0-2) 

The code itself acknowledges the missing conversion with an explicit comment:

> "It's not important to convert the price right now. For now we are keeping the base fee amount same for each valid denom -> 1 but this logic will be updated to use spot price for different valid tokens in future"

The allowed non-base denoms include `uion` and an IBC-wrapped token (`ibc/FF3065...`), as confirmed by the test fixture: [4](#0-3) 

---

### Impact Explanation

An unprivileged caller submitting a price update on Osmosis can pay the required fee entirely in `uion` (or the IBC token) instead of `uosmo`. Because the check compares raw amounts with no exchange-rate adjustment, the caller satisfies the fee requirement by paying `N uion` when the protocol intended to collect `N uosmo`. If `uion` trades at a fraction of `uosmo`'s value, the protocol receives less economic value than intended. Conversely, a caller paying a mix of denoms (e.g., half `uosmo` + half `uion`) has their amounts summed as if they were the same unit, which is incorrect in either direction.

The current fee is set to 1 unit (negligible absolute value), so the immediate financial loss per transaction is minimal. However, if governance raises the fee, the denomination mismatch becomes directly exploitable to underpay by a factor proportional to the `uion`/`uosmo` exchange rate.

---

### Likelihood Explanation

Any user submitting a price update on Osmosis can trigger this path by sending `uion` or the IBC token as `msg.funds`. No privileged access is required. The entry point is the standard `execute` handler for `UpdatePriceFeeds`. The likelihood is **high** in the sense that the code path is reachable by any transaction sender, though the current financial impact is low due to the fee being set to 1 unit.

---

### Recommendation

Replace the raw-amount summation with a conversion step that normalizes all accepted denoms to the base denom (`uosmo`) using a tamper-resistant on-chain price source (e.g., Osmosis's TWAP or spot price oracle) before comparing against `base_denom_fee.amount`. The code already contains a TODO acknowledging this: "this logic will be updated to use spot price for different valid tokens in future." Until that conversion is implemented, the contract should restrict fee payment to the base denom only, removing the multi-denom acceptance path.

---

### Proof of Concept

1. Governance sets the update fee to `1000 uosmo`.
2. `uion` trades at 0.01 `uosmo` (i.e., 1 `uion` = 0.01 `uosmo`).
3. Caller submits `UpdatePriceFeeds` with `msg.funds = [{ denom: "uion", amount: "1000" }]`.
4. `is_fee_sufficient` computes `total_amount = 1000` and `base_denom_fee.amount = 1000`; the check `1000 <= 1000` passes.
5. The protocol accepted 1000 `uion` ≈ 10 `uosmo` in place of the required 1000 `uosmo` — a 100× underpayment.

The root cause is the missing exchange-rate conversion at: [5](#0-4)

/// Osmosis can support multiple tokens for transaction fees
/// This will return update fee for the given denom only if that denom is allowed in Osmosis's txFee module
/// Else it will throw error
pub fn get_update_fee_for_denom(deps: &Deps, vaas: &[Binary], denom: String) -> StdResult<Coin> {
    let config = config_read(deps.storage).load()?;

    // if the denom is not a base denom it should be an allowed one
    if denom != config.fee.denom && !is_allowed_tx_fees_denom(deps, &denom) {
        return Err(PythContractError::InvalidFeeDenom { denom })?;
    }

    // the base fee is set to -> denom = base denom of a chain, amount = 1
    // which is very minimal
    // for other valid denoms too we are using the base amount as 1
    // base amount is multiplied to number of vaas to get the total amount

    // this will be change later on to add custom logic using spot price for valid tokens
    Ok(coin(get_update_fee_amount(deps, vaas)?, denom))
}
