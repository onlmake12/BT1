### Title
Unbounded Full-Table UPDATE in Rich-Indexer Migration Blocks Indexer Writes Without Timeout â€” (`File: util/rich-indexer/resources/migrations/20240603_add_is_spent_to_output.sql`)

---

### Summary

The rich-indexer's only SQL migration (`20240603_add_is_spent_to_output.sql`) issues an unbounded `UPDATE output SET is_spent = 1 FROM input WHERE input.output_id = output.id` against the `output` table with no `SET statement_timeout`, no `SET lock_timeout`, and no row-batching. This migration runs automatically at node startup via `migrator.run(&pool).await?`. On a fully synced CKB mainnet PostgreSQL instance, the `output` table holds one row per cell output ever indexed â€” potentially hundreds of millions of rows. The UPDATE holds `ROW EXCLUSIVE` on `output` for the entire duration, blocking the indexer's concurrent `append()` writes and stalling rich-indexer RPC service for all callers until the migration completes.

---

### Finding Description

**Migration file** (`util/rich-indexer/resources/migrations/20240603_add_is_spent_to_output.sql`):

```sql
-- 20240630_add_is_spent_to_output.sql

ALTER TABLE output
ADD COLUMN is_spent INTEGER DEFAULT 0;

UPDATE output
SET is_spent = 1
FROM input
WHERE input.output_id = output.id;
``` [1](#0-0) 

Two problems compound each other:

**Problem 1 â€” `ALTER TABLE output ADD COLUMN`:** Even in PostgreSQL 11+ (where a constant-default column add avoids a heap rewrite), `ALTER TABLE` acquires `ACCESS EXCLUSIVE` on `output`. This blocks every concurrent read and write on the table for the duration of the DDL. No `SET lock_timeout` is present, so if any long-running query holds a conflicting lock, the `ALTER TABLE` queues indefinitely â€” and every subsequent query on `output` queues behind it.

**Problem 2 â€” Unbounded full-table UPDATE:** The `UPDATE output SET is_spent = 1 FROM input WHERE input.output_id = output.id` is a single, un-batched statement that scans every row in `output` and joins against `input`. It holds `ROW EXCLUSIVE` on `output` for the entire duration, blocking all concurrent `INSERT`/`UPDATE`/`DELETE` on that table. There is no `SET statement_timeout`, no `SET lock_timeout`, and no row-limit batching. On a fully synced mainnet node the `output` table has hundreds of millions of rows; this statement runs for hours.

**Automatic execution path** (`util/rich-indexer/src/store.rs`):

```rust
let migrator = Migrator::new(temp_dir.path()).await?;
migrator.run(&pool).await?;
``` [2](#0-1) 

`SQLXPool::connect()` is called unconditionally at node startup whenever `--rich-indexer` is active. `sqlx::migrate::Migrator::run` applies all pending migrations synchronously before the node proceeds. There is no operator gate, no warning, and no timeout guard.

The `output` table schema confirms it is the central hot table for all indexed cell outputs:

```sql

    id BIGSERIAL PRIMARY KEY,
    tx_id BIGINT NOT NULL,
    output_index INTEGER NOT NULL,
    capacity BIGINT NOT NULL,
    lock_script_id BIGINT,
    type_script_id BIGINT,
    data BYTEA

``` [3](#0-2) 

The indexer's `append()` path writes to `output` for every new block. While the migration UPDATE holds `ROW EXCLUSIVE`, `append()` blocks, the indexer falls behind chain tip, and all rich-indexer RPC responses return stale data.

---

### Impact Explanation

- **Indexer write stall**: The `append()` function in `util/rich-indexer/src/indexer/mod.rs` calls `self.async_runtime.block_on(future)` synchronously. While the migration UPDATE holds `ROW EXCLUSIVE` on `output`, every `append()` call blocks waiting for the lock. The indexer cannot advance its tip. [4](#0-3) 

- **RPC degradation for all callers**: Any external RPC caller using rich-indexer endpoints (cell queries, transaction queries, script queries) receives stale or incorrect data for the entire migration window â€” potentially hours on a production mainnet node.

- **No abort-on-timeout**: Because no `SET statement_timeout` is set, the UPDATE never self-terminates. If the operator kills the process mid-migration, the migration is rolled back by sqlx and must restart from scratch on the next boot, repeating the full-table scan.

- **Severity scales with data volume**: The `output` table grows monotonically with chain history. The longer a node has been synced, the worse the migration window. There is no documented maintenance window requirement, no operator warning in the startup log beyond `"Running migrations..."`, and no `--skip-migration` escape hatch.

---

### Likelihood Explanation

Any operator running a PostgreSQL-backed rich-indexer who upgrades from a CKB version that predates migration `20240603` to a version that includes it will trigger this automatically on the next `ckb run --rich-indexer`. The migration is embedded in the binary via `include_dir!` and runs unconditionally. No special configuration or attacker action is required â€” the trigger is a routine node upgrade. [5](#0-4) 

---

### Recommendation

**For the UPDATE â€” batch with a timeout guard:**

```sql
-- Set a statement timeout so the migration aborts cleanly rather than stalling forever
SET statement_timeout = '30min';
SET lock_timeout = '5s';

ALTER TABLE output ADD COLUMN IF NOT EXISTS is_spent INTEGER DEFAULT 0;

-- Batch the UPDATE to avoid holding ROW EXCLUSIVE for the full table duration
DO $$
DECLARE
  batch_size INT := 100000;
  last_id    BIGINT := 0;
  max_id     BIGINT;
BEGIN
  SELECT MAX(id) INTO max_id FROM output;
  WHILE last_id < max_id LOOP
    UPDATE output
    SET is_spent = 1
    FROM input
    WHERE input.output_id = output.id
      AND output.id > last_id
      AND output.id <= last_id + batch_size;
    last_id := last_id + batch_size;
    PERFORM pg_sleep(0.01); -- yield between batches
  END LOOP;
END $$;
```

**For the ALTER TABLE â€” add a `lock_timeout`** so that if a long-running query holds a conflicting lock, the DDL fails fast rather than queuing and blocking all subsequent queries behind it.

**Project convention**: Prepend every migration file with `SET lock_timeout = '5s'; SET statement_timeout = '60min';` as a baseline guard. Add a CI lint step that rejects migration files containing bare `UPDATE <table>` without a `WHERE id BETWEEN` or `LIMIT`-based batching pattern.

### Proof of Concept

1. Deploy a PostgreSQL-backed CKB rich-indexer and sync it to mainnet tip (hundreds of millions of `output` rows).
2. Upgrade the CKB binary to a version containing migration `20240603_add_is_spent_to_output.sql`.
3. Run `ckb run --rich-indexer`.
4. Observe in `pg_stat_activity` that the UPDATE statement is running and holding `ROW EXCLUSIVE` on `output`.
5. Attempt to call any rich-indexer RPC that writes to `output` (e.g., indexing a new block via `append()`): it blocks for the entire migration duration.
6. Observe that rich-indexer RPC responses return stale data (tip does not advance) until the migration completes.
