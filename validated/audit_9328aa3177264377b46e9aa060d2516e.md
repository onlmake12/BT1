### Title
Off-by-One in `MAX_VMS_COUNT` Guard Allows Scripts to Exceed the Concurrent VM Limit — (`script/src/scheduler.rs`)

---

### Summary

The `Scheduler` in `script/src/scheduler.rs` enforces a maximum of `MAX_VMS_COUNT = 16` concurrent VMs per script execution. The guard check uses a strict `>` comparison instead of `>=`, creating an off-by-one that allows a script author to spawn 17 concurrent VMs instead of the intended 16, bypassing the resource safety limit.

---

### Finding Description

`MAX_VMS_COUNT` is defined as 16 and is documented as "The maximum number of VMs that can be created at the same time." [1](#0-0) 

When processing a `Message::Spawn` in `process_message_box`, the guard check is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
    // return MAX_VMS_SPAWNED error
    continue;
}
let spawned_vm_id = self.boot_vm(...)?;
``` [2](#0-1) 

The condition `> 16` rejects only when the current count is **17 or more**, meaning it permits a spawn when the count is exactly 16. After `boot_vm` succeeds, the total becomes **17**. The correct guard should be `>= MAX_VMS_COUNT as usize` (i.e., `>= 16`), which would reject when the count is already at the limit.

The analogous pattern to the NFT report: just as `tokensPerMint` was checked but the state was not yet updated before the callback fired, here the limit is checked against the current count but the check boundary is wrong — the "act" (boot_vm) is permitted one step past the intended limit.

---

### Impact Explanation

A script author can craft a CKB lock/type script that issues 16 `ckb_spawn` calls (root VM + 16 children = 17 total concurrent VMs). This exceeds the intended `MAX_VMS_COUNT = 16` limit by one. Each extra VM consumes additional memory (snapshot state, ELF program, machine registers) and cycle budget beyond what the protocol intends to allow. Because all nodes share the same buggy code, the transaction is accepted by the network — there is no consensus split, but the resource accounting invariant is violated.

---

### Likelihood Explanation

Any unprivileged script author can trigger this. No special permissions, keys, or network conditions are required. The attacker simply writes a CKB script that calls `ckb_spawn` 16 times in sequence. The `spawn` syscall is a standard, documented CKB-VM feature available to all scripts since the Meepo hardfork (ScriptVersion::V2). [3](#0-2) [4](#0-3) 

---

### Recommendation

Change the guard from strict-greater-than to greater-than-or-equal:

```rust
// Before (off-by-one):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
``` [5](#0-4) 

This ensures the total number of concurrent VMs (suspended + instantiated) never exceeds `MAX_VMS_COUNT = 16` after a spawn.

---

### Proof of Concept

1. A script author writes a CKB lock script (ScriptVersion::V2) that calls `ckb_spawn` in a loop 16 times, each time spawning a child that simply sleeps (waits on a pipe).
2. At iteration 15 (count = 16 before the spawn), the check `16 > 16` evaluates to `false`, so the spawn is permitted.
3. `boot_vm` is called, inserting the 17th VM into `self.instantiated`.
4. The total concurrent VM count is now 17, exceeding `MAX_VMS_COUNT = 16`.
5. The 17th spawn at iteration 16 (count = 17) is the first one rejected with `MAX_VMS_SPAWNED`.

The existing test `check_spawn_max_vms_count` passes because it only verifies that *some* limit is enforced, not that the limit is exactly 16. [6](#0-5) [5](#0-4)

    fn boot_vm(&mut self, location: &DataLocation, args: VmArgs) -> Result<VmId, Error> {
        let id = self.next_vm_id;
        self.next_vm_id += 1;
        let (context, mut machine) = self.create_dummy_vm(&id)?;
        let (program, _) = {
            let sc = context.snapshot2_context.lock().expect("lock");
            sc.load_data(&location.data_piece_id, location.offset, location.length)?
        };
        self.load_vm_program(&context, &mut machine, location, program, args)?;
        // Newly booted VM will be instantiated by default
        while self.instantiated.len() >= MAX_INSTANTIATED_VMS {
            // Instantiated is a BTreeMap, first_entry will maintain key order
            let id = *self
                .instantiated
                .first_entry()
                .ok_or_else(|| Error::Unexpected("Map should not be empty".to_string()))?
                .key();
            self.suspend_vm(&id)?;
        }

        self.instantiated.insert(id, (context, machine));
        self.states.insert(id, VmState::Runnable);

        Ok(id)
    }
