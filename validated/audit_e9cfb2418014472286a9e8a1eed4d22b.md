### Title
Pre-Decompress 8MB Heap Allocation via Attacker-Controlled Snappy Header Varint — (`network/src/compress.rs`)

### Summary
The `LengthDelimitedCodecWithCompress::decode` function allocates a zeroed buffer sized by the attacker-controlled `decompress_len()` varint before performing any actual decompression. An unprivileged remote peer can craft a snappy frame with a header claiming 8MB uncompressed size but a tiny (e.g., 10-byte) compressed payload, causing the node to allocate `BytesMut::zeroed(8388608)` per such frame.

### Finding Description
In `network/src/compress.rs`, the `decode` function calls `decompress_len(&data[1..])` to read the claimed uncompressed length from the snappy stream header, then immediately allocates `BytesMut::zeroed(decompressed_bytes_len)` if the value is ≤ `MAX_UNCOMPRESSED_LEN`. [1](#0-0) 

The guard at line 235 only rejects values **strictly greater than** 8MB, so a claimed size of exactly 8,388,608 bytes passes and triggers the full allocation. [2](#0-1) 

The snappy stream header varint is entirely attacker-controlled. The `max_frame_length` per protocol (e.g., 4MB for RelayV3, 2MB for Sync) limits the compressed wire frame, not the claimed decompressed size. [3](#0-2) 

The same pattern exists in `Message::decompress` at line 81, which uses `vec![0; decompressed_bytes_len]`. [4](#0-3) 

### Impact Explanation
Each malicious frame causes an 8MB heap allocation before the error is detected. After `decompress` fails (snappy validates output length matches the header), the buffer is dropped and the connection is closed. However:

- The attacker can continuously re-establish connections (up to the node's `max_inbound` peer limit)
- In tokio's async runtime, multiple connections are processed concurrently across worker threads, meaning multiple 8MB allocations can be in-flight simultaneously
- The compression ratio amplification is ~800,000x (10 bytes on wire → 8MB allocation)

This can cause sustained memory pressure and heap fragmentation. A full OOM crash requires many concurrent connections with limited node memory, but significant performance degradation is achievable with fewer connections.

### Likelihood Explanation
The attack requires only a TCP connection to the node's P2P port — no authentication, no PoW, no privileged role. The malicious frame is trivial to craft: set the snappy header varint to 8,388,608 and append any valid or invalid snappy block body. The attack is repeatable as fast as TCP connections can be established.

### Recommendation
Allocate the decompression buffer based on the **actual compressed payload size** or a conservative heuristic (e.g., `min(decompressed_bytes_len, compressed_len * MAX_COMPRESSION_RATIO)`), not the attacker-supplied header value. Alternatively, decompress into a streaming/chunked buffer rather than pre-allocating the full claimed size upfront.
