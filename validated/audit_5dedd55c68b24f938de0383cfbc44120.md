### Summary

An unprivileged remote peer can force unbounded secp256k1 ECDSA recovery operations on every CKB node by flooding the P2P Alert protocol with crafted messages that each carry a distinct `alert_id`. The `has_received` dedup check is bypassable because it only tracks successfully verified alerts. The 5-minute ban is bypassable by reconnecting from a different IP. There is no rate limiting before the expensive cryptographic operations are performed.

### Impact Explanation

Each crafted alert message forces up to `pks.len()` (currently 4) secp256k1 ECDSA recovery operations (~50–200 µs each on modern hardware) with zero prior authentication or rate limiting. With N simultaneous connections from distinct IPs, the attacker sustains N × 4 recovery operations per message cycle. Because the alert protocol is broadcast to all peers, a single attacker can amplify the load across the entire network: every node that receives the crafted alert performs the same 4 recoveries before banning the sender. Sustained at scale, this degrades CPU availability for consensus-critical tasks (block validation, transaction relay).

---

### Likelihood Explanation

The attack requires no special privileges, no leaked keys, and no majority hashpower. It requires only:
- The ability to open P2P connections to CKB nodes (standard network access)
- Crafting 65-byte signatures that pass the cheap `is_valid()` range check (trivial)
- Cycling `alert_id` values (trivial, 2^32 space)
- Multiple IPs or reconnection after 5-minute ban (trivial with any botnet or VPN)

The attack is fully local-testable and requires no mainnet access.

---

### Recommendation

1. **Rate-limit alert messages per peer** before calling `verify_signatures` — e.g., one alert per peer per N seconds.
2. **Add a global rate limit** on alert verification (e.g., token bucket) so that even with many peers, the total ECDSA recovery rate is bounded.
3. **Extend the dedup to cover failed alerts**: record recently-seen `alert_id` values (in a bounded LRU) regardless of verification outcome, so the same `alert_id` cannot be replayed from a reconnected peer.
4. **Consider a proof-of-work or connection-age gate** before accepting alert messages from newly connected peers.
