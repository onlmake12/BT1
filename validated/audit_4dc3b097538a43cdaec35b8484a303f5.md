### Title
Missing `ScriptHashType` Validation for Output Type Scripts Bypasses Consensus Hash-Type Restriction â€” (`verification/src/transaction_verifier.rs`)

### Summary
`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the `hash_type` field only on each output's **lock script**, completely skipping the **type script**. An unprivileged transaction sender can craft an output whose type script carries a `ScriptHashType` value that is not in the consensus-permitted set `ENABLED_SCRIPT_HASH_TYPE`, bypassing the guard that exists to gate new hash-type features behind a hardfork activation.

### Finding Description

In `verification/src/transaction_verifier.rs`, `ScriptHashTypeVerifier::verify()` loops over outputs and calls `output.lock().hash_type()` exclusively: [1](#0-0) 

The method never calls `output.type_()` to inspect the type script's `hash_type`. The `NonContextualTransactionVerifier` â€” which is the first gate applied both in the tx-pool (`tx-pool/src/util.rs`) and during block verification â€” composes `ScriptHashTypeVerifier` as one of its sub-verifiers: [2](#0-1) 

Because `NonContextualTransactionVerifier::verify()` calls `self.script_hash_type.verify()` and that call only covers lock scripts, a transaction output whose **type script** carries a disallowed or not-yet-activated `ScriptHashType` byte passes this check silently.

The analog to the Algorand report is direct: just as the Algorand approval program checked `OptIn` and `CloseOut` but left `UpdateApplication` / `DeleteApplication` unchecked, `ScriptHashTypeVerifier` checks lock-script hash types but leaves type-script hash types unchecked.

### Impact Explanation
`ENABLED_SCRIPT_HASH_TYPE` is the consensus mechanism for gating new `ScriptHashType` variants (e.g., `Data2`, future variants) behind a hardfork. If a variant is implemented in the VM but not yet activated by consensus, an attacker can submit a transaction whose output type script uses that variant. The transaction passes `NonContextualTransactionVerifier` and enters the tx-pool or a mined block. When the `ContextualTransactionVerifier` later runs `TransactionScriptsVerifier`, the VM executes the type script under the new (not-yet-activated) hash-type semantics, effectively allowing pre-activation use of a feature that consensus intended to gate. This is a consensus-rule bypass reachable by any unprivileged transaction sender via `send_transaction` RPC. [3](#0-2) 

### Likelihood Explanation
The entry path is the public `send_transaction` RPC endpoint â€” no privilege required. The attacker only needs to know which `ScriptHashType` byte value is implemented in the VM but not yet in `ENABLED_SCRIPT_HASH_TYPE`. This information is visible in the open-source codebase. Likelihood is medium: it requires a window where a new hash type is VM-supported but consensus-gated, which is exactly the scenario `ENABLED_SCRIPT_HASH_TYPE` is designed to manage.

### Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate `output.type_()` when a type script is present. The check should mirror the existing lock-script check: attempt `TryInto::<ScriptHashType>` on the type script's `hash_type` byte and confirm the resulting value is in `ENABLED_SCRIPT_HASH_TYPE`. This closes the gap symmetrically for both script slots on every output.

### Proof of Concept
1. Identify a `ScriptHashType` byte value `N` that is handled by the CKB-VM but whose value is absent from `ENABLED_SCRIPT_HASH_TYPE`.
2. Construct a `TransactionView` with one output whose **lock script** uses a valid, permitted hash type (passes the existing check) and whose **type script** uses hash-type byte `N`.
3. Submit via `send_transaction` RPC (with `outputs_validator = "passthrough"` to skip the output validator).
4. Observe that `NonContextualTransactionVerifier::verify()` returns `Ok(())` â€” the type-script hash type is never inspected.
5. The transaction enters the pool and, if mined, the VM executes the type script under the ungated hash-type semantics. [4](#0-3)

// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
