### Title
Alert Signature Replay Across CKB Networks Due to Missing Chain-Specific Domain Separation - (File: `util/network-alert/src/verifier.rs`)

### Summary

The CKB network alert system signs and verifies alerts using a hash of the `RawAlert` struct content (`calc_alert_hash`). This hash contains no chain-specific identifier (no genesis hash, no chain name, no network ID). Because the same hardcoded developer public keys are used across all CKB networks (mainnet, testnet, preview), a valid alert signed for one network is cryptographically valid on any other CKB network. An unprivileged peer who receives a valid alert on testnet can replay it verbatim to mainnet nodes, which will accept, store, and re-broadcast it to all connected peers.

### Finding Description

**Root cause — no chain binding in the signed message:**

`calc_alert_hash` is defined in `util/gen-types/src/extension/calc_hash.rs`:

```rust
impl<'r> packed::RawAlertReader<'r> {
    pub fn calc_alert_hash(&self) -> packed::Byte32 {
        self.calc_hash()   // blake2b of raw RawAlert bytes only
    }
}
``` [1](#0-0) 

The `RawAlert` molecule schema contains only application-level fields — no chain ID, genesis hash, or network name:

```

**Verifier uses this chain-agnostic hash directly:**

    let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;

**Same keys used across all networks by default:**

The `NetworkAlertConfig::default()` loads a single bundled `alert_signature.toml` regardless of which chain is running: [4](#0-3) 

There is no per-chain key set. Mainnet, testnet, and preview all share the same hardcoded Nervos Foundation developer keys.

**Relay handler accepts any peer-supplied alert that passes signature check:**

In `AlertRelayer::received`, after parsing, the only gate is `verify_signatures`. If that passes, the alert is stored and re-broadcast to all connected peers: [5](#0-4) 

### Impact Explanation

An unprivileged attacker who runs a testnet node and receives a legitimately signed testnet alert (e.g., "CKB v0.105.* have bugs, please upgrade") can relay that exact byte-for-byte alert to mainnet nodes via the P2P Alert protocol. Mainnet nodes will:
1. Verify the signatures — they pass, because the same developer keys are used and the hash contains no chain discriminator.
2. Store the alert in their notifier.
3. Re-broadcast it to all connected mainnet peers.

All mainnet nodes will then display the alert to operators. If the testnet alert message is misleading in a mainnet context (e.g., references a version range that is safe on mainnet, or instructs operators to take a specific action), this can cause widespread confusion, unnecessary node shutdowns, or social-engineering-assisted attacks. The alert system is explicitly described as a mechanism for "critical problems" — injecting false critical alerts undermines the trust model of the entire alert infrastructure.

### Likelihood Explanation

The attack requires no privileged access. Any party who:
1. Runs a CKB testnet node (free, public network), and
2. Connects to mainnet nodes via the P2P Alert protocol

can replay any alert that has ever been broadcast on testnet. Testnet alerts are observable by any participant. The P2P Alert protocol (`SupportProtocols::Alert`) is open to all peers. The `identify_name` check only prevents cross-network P2P sessions at the identify layer, but a node can connect to both networks independently and manually forward the raw alert bytes. The `send_alert` RPC endpoint also accepts raw alert bytes from any local caller, providing another injection point. [6](#0-5) 

### Recommendation

Include a chain-specific domain separator in the alert signing message. The simplest fix is to mix the genesis hash (or `identify_name`) into the hash before signing:

```rust
pub fn calc_alert_hash(&self) -> packed::Byte32 {
    let mut hasher = new_blake2b();
    hasher.update(GENESIS_HASH_BYTES); // chain-specific constant
    hasher.update(self.as_slice());
    let mut result = [0u8; 32];
    hasher.finalize(&mut result);
    packed::Byte32::new(result)
}
```

Alternatively, add a `chain_id` or `network_name` field to `RawAlert` and include it in the hash. The `Consensus::identify_name()` value (`/{chain_id}/{genesis_hash[:8]}`) is already available and would serve as a suitable domain separator.

### Proof of Concept

1. Run a CKB testnet node and wait for (or trigger via `send_alert` RPC with valid testnet developer keys) a valid alert to be broadcast on testnet.
2. Capture the raw alert bytes from the P2P message or from `get_blockchain_info().alerts`.
3. Connect to a CKB mainnet node.
4. Submit the captured alert bytes via the mainnet node's `send_alert` RPC or directly via the P2P Alert protocol (`SupportProtocols::Alert`).
5. Observe that `verify_signatures` in `util/network-alert/src/verifier.rs` returns `Ok(())` — the signatures are valid because the hash is chain-agnostic and the same keys are used.
6. Observe the alert appearing in `get_blockchain_info().alerts` on all connected mainnet peers.

The existing test `test_alert_20230001` in `util/network-alert/src/tests/generate_alert_signature.rs` demonstrates that a fixed alert with fixed signatures verifies correctly — the same bytes would verify on any network, confirming the absence of chain binding. [7](#0-6)

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }
