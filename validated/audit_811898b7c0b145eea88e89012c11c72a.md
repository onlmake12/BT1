Audit Report

## Title
Unbounded Full-Pool Iteration in `get_raw_tx_pool` RPC with No Rate Limiting Enables Node Resource Exhaustion — (`rpc/src/module/pool.rs`)

## Summary

The `get_raw_tx_pool` RPC handler unconditionally iterates every entry in the transaction pool with no pagination, count cap, or rate limit. When `verbose=true`, it materializes a `TxEntryInfo` struct for every pool entry and serializes the entire result to JSON per call. The pool is bounded only by byte size (`max_tx_pool_size`, default 180 MB), not by entry count, so a pool filled with small-but-valid transactions can contain hundreds of thousands of entries. An attacker who fills the pool (requiring real CKB tokens) and then repeatedly calls this unauthenticated endpoint can sustain high CPU load, large transient heap allocations, and degraded node responsiveness.

## Finding Description

**Root cause — no bound on entries processed per call:**

`get_raw_tx_pool` in `rpc/src/module/pool.rs` (L703–718) dispatches to either `get_all_entry_info()` or `get_all_ids()` with no limit parameter:

```rust
fn get_raw_tx_pool(&self, verbose: Option<bool>) -> Result<RawTxPool> {
    let tx_pool = self.shared.tx_pool_controller();
    let raw = if verbose.unwrap_or(false) {
        let info = tx_pool.get_all_entry_info()...;
        RawTxPool::Verbose(info.into())
    } else {
        let ids = tx_pool.get_all_ids()...;
        RawTxPool::Ids(ids.into())
    };
    Ok(raw)
}
```

`get_all_entry_info()` in `tx-pool/src/pool.rs` (L464–487) runs three separate full-pool iterators — `score_sorted_iter_by_statuses` over pending+gap, `sorted_proposed_iter` over proposed, and a full scan of `conflicts_cache` — collecting a `TxEntryInfo` per entry. `get_ids()` (L448–462) performs the same two full-pool scans collecting every hash.

The service layer in `tx-pool/src/service.rs` (L1000–1013) acquires a pool read-lock for the entire duration of both operations:

```rust
Message::GetAllEntryInfo(Request { responder, .. }) => {
    let tx_pool = service.tx_pool.read().await;
    let info = tx_pool.get_all_entry_info();
    ...
}
```

**No authentication or rate limiting exists** on this endpoint — grep across `rpc/src/**` finds no auth, token, whitelist, or rate-limit logic applied to `get_raw_tx_pool`.

**Pool size bound is bytes-only:** `TxPoolConfig::max_tx_pool_size` (in `util/app-config/src/configs/tx_pool.rs`) is a byte cap with no corresponding entry-count cap. With the default 180 MB cap and realistic small CKB transactions (~200–500 bytes), the pool can hold tens to hundreds of thousands of entries.

**Exploit path:**
1. Attacker submits many small valid transactions (requires CKB tokens) until the pool approaches `max_tx_pool_size`.
2. Attacker (or any caller) repeatedly issues `POST /rpc {"method":"get_raw_tx_pool","params":[true]}` — this is unauthenticated and free.
3. Each call forces: full pool read-lock acquisition → three full-pool iterator passes → heap allocation of one `TxEntryInfo` per entry → full JSON serialization of the result.
4. Sustained calls keep CPU pegged on serialization and memory allocator under pressure from large transient allocations.

**Existing guards are insufficient:** The only pool-side guard is the byte-size cap, which bounds total pool memory but not the per-RPC-call work. There is no server-side limit on how many entries `get_raw_tx_pool` processes or how frequently it may be called.

## Impact Explanation

This matches **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** A degraded node cannot efficiently relay transactions or assemble blocks, contributing to network-level congestion. The CPU and memory pressure from repeated unbounded scans can make the node unresponsive to legitimate transaction submissions and block assembly. The RPC call itself is free and unauthenticated; only the pool-filling step requires CKB tokens, and that cost is bounded by the attacker's willingness to spend (and can be partially recovered via RBF or by waiting for transactions to expire).

## Likelihood Explanation

- `get_raw_tx_pool` is enabled by default and many operators expose the RPC port on non-loopback interfaces.
- No authentication or rate limiting is applied (confirmed by code search).
- The repeated-call phase (step 2) is zero-cost to the attacker.
- The pool-filling phase requires valid CKB tokens, which is a real but not prohibitive cost barrier — it is a one-time setup cost, and the repeated-call phase is then free indefinitely.
- The pool can also fill organically during high-activity periods, at which point any external caller can trigger the expensive scan for free.

## Recommendation

1. **Add pagination to `get_raw_tx_pool`**: Accept optional `limit` and `after` cursor parameters; return at most `N` entries per call (consistent with how indexer RPCs handle large result sets).
2. **Enforce a hard server-side cap** inside `get_all_entry_info` / `get_ids` regardless of caller input.
3. **Apply per-IP or global rate limiting** to `get_raw_tx_pool` and `estimate_fee_rate` at the RPC dispatch layer.
4. **Add an entry-count cap** to `TxPoolConfig` alongside the byte-size cap so worst-case iteration length is predictable.

## Proof of Concept

```bash
# Step 1: fill the pool with small valid transactions (requires CKB tokens)
for i in $(seq 1 50000); do
    ckb-cli tx send --tx <min_size_tx_$i>
done

# Step 2: repeatedly call get_raw_tx_pool verbose=true (free, unauthenticated)
while true; do
    curl -s -X POST http://<node>:8114 \
      -H 'Content-Type: application/json' \
      -d '{"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[true],"id":1}'
done
```

Each iteration of step 2 forces the node to acquire the pool read-lock, run three full-pool iterator passes, allocate a `TxEntryInfo` per entry, and serialize the entire result to JSON — with no server-side bound on entries processed or call frequency. Node CPU and memory usage can be monitored to confirm sustained elevation; block assembly latency and transaction relay throughput will degrade under sustained load.
