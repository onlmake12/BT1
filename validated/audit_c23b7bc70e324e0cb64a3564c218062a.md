### Title
Unbounded SQL OFFSET via Crafted `after` Cursor in `get_tx_with_cell` Causes Indexer DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

### Summary

An unprivileged RPC caller can supply a crafted `after` cursor to `get_transactions` (ungrouped mode) whose last 4 bytes decode to `i32::MAX` (`0x7FFFFFFF`). This value is passed without any upper-bound check directly as the SQL `OFFSET`, forcing the database to skip 2,147,483,647 virtual rows before returning results, causing severe CPU/IO exhaustion that hangs or crashes the CKB node's rich-indexer service.

### Finding Description

In `get_transactions`, when `group_by_transaction` is `Some(false)` or `None`, the `after` cursor is validated only for length (must be exactly 12 bytes): [1](#0-0) 

The last 4 bytes are decoded as a raw `i32` via `decode_i32`, which performs no bounds checking — it simply calls `i32::from_le_bytes(...)`: [2](#0-1) 

The resulting `(i64, i32)` tuple is passed to `get_tx_with_cell`, where the `i32` offset is applied directly to the SQL query builder with no upper-bound guard: [3](#0-2) 

This produces a SQL query of the form:
```sql
SELECT ... FROM (...) AS res
JOIN ckb_transaction ON ...
JOIN block ON ...
WHERE tx_id >= ?
ORDER BY tx_id ASC
LIMIT 100
OFFSET 2147483647
```

Both SQLite and PostgreSQL must internally enumerate and discard 2,147,483,647 rows before returning the (empty) result set, consuming unbounded CPU and I/O.

### Impact Explanation

A single RPC call with a 12-byte crafted cursor can pin the database engine for seconds to minutes depending on table size and hardware. Repeated calls (no authentication required) cause sustained resource exhaustion, effectively hanging or crashing the indexer service and making the node unresponsive to all subsequent indexer RPC requests.

### Likelihood Explanation

The rich-indexer RPC endpoint is unauthenticated by default. The cursor format (12 bytes, last 4 = little-endian i32) is trivially constructable. No PoW, key material, or privileged access is required. The only precondition is that the node operator has enabled `rich-indexer`.

### Recommendation

Add an upper-bound check on the decoded offset before using it. For example, reject or clamp any offset exceeding a safe maximum (e.g., the configured `request_limit`):

```rust
let offset = decode_i32(offset)?;
if offset < 0 || offset as u32 > self.request_limit {
    return Err(Error::invalid_params("cursor offset out of bounds"));
}
last_cursor = Some((last, offset));
```

Alternatively, redesign the cursor to use a keyset-pagination approach (store only the last `tx_id` and `io_index`) rather than an OFFSET, which eliminates the problem entirely and is more efficient.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
