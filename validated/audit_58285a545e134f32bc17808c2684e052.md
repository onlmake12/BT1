### Title
DAO Withdrawal Consensus Split via Witness Header-Dep Index Interpretation Mismatch — (File: `util/dao/src/lib.rs`)

---

### Summary

The Rust node's `DaoCalculator::transaction_maximum_withdraw` reads the header-dep index from the DAO withdrawal witness as a full `u64` little-endian integer, while the on-chain DAO C script (`dao.c`) reads only the **lowest byte** of that same 8-byte field. For any witness-encoded index value whose full `u64` differs from its lowest byte (i.e., any index ≥ 256), the two implementations resolve different entries in `header_deps`. The Rust node then fails a block-number cross-check and rejects the transaction, even though the C VM would accept it. This creates a reachable consensus split: a miner can include a crafted but C-VM-valid DAO withdrawal in a block, and every Rust node will reject that block.

---

### Finding Description

In `transaction_maximum_withdraw`, the deposit header is located by reading an 8-byte index from the witness `input_type` field and using it to index into `header_deps`:

```rust
// util/dao/src/lib.rs  lines 91-98
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)   // full u64 used as slice index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
```

The on-chain DAO C script reads the same 8-byte field but interprets only its **first (lowest) byte** as the index. For a witness that encodes the value `257` (`[0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]` in little-endian):

| Layer | Reads | Resolves |
|---|---|---|
| C VM (dao.c) | byte[0] = 1 | `header_deps[1]` |
| Rust node | full u64 = 257 | `header_deps[257]` |

A transaction author can therefore arrange `header_deps` so that:
- `header_deps[1]` = deposit block hash (C VM finds the correct deposit header → accepts)
- `header_deps[257]` = some other block hash (Rust finds a wrong header → block-number check at line 105 fails → rejects)

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this split, constructing exactly this layout and asserting `result.is_err()` from the Rust side while noting that the C VM resolves via the lowest byte.

---

### Impact Explanation

**Consensus split / chain halt for Rust nodes.** A miner who includes a crafted DAO withdrawal (C-VM-valid, Rust-rejected) in a block causes every Rust full node to reject that block as invalid. The miner's chain diverges from the Rust-node view of the canonical chain. Depending on hashpower, this can permanently fork the network or stall Rust nodes.

**DoS on DAO withdrawals.** Even without a miner, a user who submits such a transaction to the tx-pool receives a `DaoError::InvalidOutPoint` rejection, making their DAO withdrawal unprocessable through the normal RPC path.

---

### Likelihood Explanation

The attack requires:
1. Constructing a transaction with ≥ 258 `header_deps` entries (no protocol limit prevents this).
2. Encoding a witness index whose lowest byte points to the correct deposit header while the full `u64` points elsewhere.
3. Convincing a miner to include the transaction (or being a miner oneself).

Steps 1 and 2 are fully within the control of any transaction sender. Step 3 is the only friction. The likelihood is **medium**: the construction is straightforward, and a malicious miner can execute it unilaterally.

---

### Recommendation

Replace the full-`u64` index read with a single-byte read to match the C VM's behavior:

```rust
// Instead of:
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))

// Use:
Ok(u64::from(header_deps_index_data.unwrap()[0]))
```

Alternatively, add an explicit validation that rejects any witness index whose value exceeds `u8::MAX` (255), since the C VM cannot address beyond that range, making such transactions provably invalid under both interpreters.
