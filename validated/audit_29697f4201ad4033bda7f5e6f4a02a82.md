### Title
Unbounded Memory Allocation via `BytesVisitor::visit_str` in `submit_block` Extension Field Before Consensus Size Guard — (`util/jsonrpc-types/src/bytes.rs`)

### Summary

`BytesVisitor::visit_str` allocates a heap buffer proportional to the raw hex string length with no prior size check. The HTTP RPC server does not enforce the `max_request_body_size` config field. An attacker with access to the `submit_block` RPC endpoint can send an arbitrarily large `extension` hex string, causing the node to allocate tens or hundreds of megabytes before the 96-byte consensus guard in `BlockExtensionVerifier::verify` ever fires.

---

### Finding Description

**Step 1 — Allocation site with no bound check**

`BytesVisitor::visit_str` in `util/jsonrpc-types/src/bytes.rs` allocates a `Vec<u8>` of size `bytes.len() >> 1` (i.e., half the hex string length) with no prior size validation: [1](#0-0) 

A hex string `"0x" + "ff" * 20_000_000` (40 MB of ASCII) causes a 20 MB heap allocation here, before any consensus check.

**Step 2 — HTTP server does not enforce `max_request_body_size`**

The config struct defines `max_request_body_size: usize`: [2](#0-1) 

But `rpc/src/server.rs` builds the axum router with no `DefaultBodyLimit` or equivalent layer — the field is never read or applied to the HTTP server: [3](#0-2) 

The TCP server does apply a 2 MB limit via `LinesCodec::new_with_max_length(2 * 1024 * 1024)`, but the HTTP path has no such guard. [4](#0-3) 

**Step 3 — Deserialization happens before conversion and before consensus checks**

In `submit_block`, the `block: Block` parameter is fully deserialized (including the `extension` `JsonBytes`) by serde before the function body executes. The `From<Block> for packed::Block` conversion then runs: [5](#0-4) 

Only after this does `blocking_process_block` eventually reach `BlockExtensionVerifier::verify`, which checks `extension.len() > 96`: [6](#0-5) 

The 96-byte guard is a consensus check that fires **after** the multi-megabyte allocation has already occurred.

**Step 4 — `From<Block> for packed::Block` performs a second allocation**

The `extension.into()` call in the conversion copies the already-allocated `Bytes` into a `packed::Bytes`, doubling the memory footprint for the extension field: [7](#0-6) 

---

### Impact Explanation

An attacker with access to the HTTP RPC endpoint can send repeated `submit_block` requests with a giant `extension` field. Each request causes an allocation proportional to the extension size before any rejection. With concurrent requests and a large enough extension (e.g., 100 MB hex string → 50 MB allocation per request), the node's heap can be exhausted, causing an OOM crash or severe memory pressure that degrades all node operations.

---

### Likelihood Explanation

The `submit_block` RPC is a miner endpoint. In practice, mining pools and solo miners expose this endpoint to pool workers or automation scripts, sometimes on a LAN or VPN rather than strictly localhost. Any party with network access to the RPC port can exploit this. The attack requires no PoW, no valid block structure, and no cryptographic material — just a large JSON string. The `max_request_body_size` config field gives operators a false sense of security since it is never applied to the HTTP server.

---

### Recommendation

1. **Enforce `max_request_body_size` in the HTTP server** by adding a `tower_http::limit::RequestBodyLimitLayer` (or axum's `DefaultBodyLimit`) using the configured value before the route handlers.
2. **Add a size pre-check in `BytesVisitor::visit_str`** before allocating: reject strings whose decoded length exceeds a reasonable bound (e.g., 1 MB or a configurable cap).
3. **Add an early extension size check** in the `submit_block` handler or in `From<Block> for packed::Block` before the full conversion, so oversized extensions are rejected at the JSON layer rather than after allocation.

---

### Proof of Concept

```python
import requests, json

# 20 MB extension: 10 MB allocation per request
extension = "0x" + "ff" * 10_000_000

payload = {
    "jsonrpc": "2.0",
    "method": "submit_block",
    "params": [
        "work_id_0",
        {
            "header": { ... },  # any syntactically valid header fields
            "proposals": [],
            "transactions": [],
            "uncles": [],
            "extension": extension
        }
    ],
    "id": 1
}

# Send many concurrent requests to exhaust memory
for _ in range(50):
    requests.post("http://127.0.0.1:8114", json=payload)
```

Each request causes `BytesVisitor::visit_str` to allocate 10 MB before the node rejects the block at `BlockExtensionVerifier`. The node will reject with `ExceededMaximumBlockExtensionBytes`, but only after the allocation. Fifty concurrent requests = ~500 MB of transient heap pressure, with no rate limiting or body size enforcement on the HTTP path.
