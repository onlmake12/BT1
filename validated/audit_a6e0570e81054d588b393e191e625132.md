### Title
`CapacityVerifier` Skips Entire `OutputsSumOverflow` Check When Any Single Input Is a DAO Cell ŌĆö (`File: verification/src/transaction_verifier.rs`)

### Summary

`CapacityVerifier::valid_dao_withdraw_transaction()` uses `.any()` to detect whether a transaction is a DAO withdrawal. If even one input cell carries the DAO type script, the entire `OutputsSumOverflow` guard is bypassed for the whole transaction. The DAO type script itself only validates DAO-specific cells, so non-DAO inputs and outputs in the same transaction are never checked for capacity balance. An attacker who controls a DAO cell can craft a mixed transaction that creates CKB capacity out of thin air for the non-DAO portion.

### Finding Description

**Root cause ŌĆö partial predicate used to skip a global invariant:**

`CapacityVerifier::verify()` skips the `OutputsSumOverflow` check when `valid_dao_withdraw_transaction()` returns `true`: [1](#0-0) 

`valid_dao_withdraw_transaction()` returns `true` if **any** resolved input uses the DAO type script: [2](#0-1) 

The code comment justifies the skip with:

> "DAO withdraw transaction is verified via the type script of DAO cells" [3](#0-2) 

**Why the justification is incomplete:**

The DAO type script only validates the DAO cells ŌĆö it computes the maximum withdraw amount for each DAO input and checks that the corresponding output matches. It has no visibility into, and makes no assertion about, non-DAO inputs or outputs in the same transaction.

**Exploit flow:**

1. Attacker deposits 100 CKB into the DAO (normal deposit).
2. After the lock period, attacker constructs a withdrawal transaction:
   - **Input A**: DAO cell (100 CKB deposited, worth 110 CKB with interest) ŌĆö triggers `.any()` ŌåÆ `true`
   - **Input B**: Regular cell (50 CKB)
   - **Output 1**: Regular cell (110 CKB) ŌĆö DAO type script verifies this is correct
   - **Output 2**: Regular cell (100 CKB) ŌĆö no verifier checks this
3. `valid_dao_withdraw_transaction()` returns `true` ŌåÆ `OutputsSumOverflow` check is skipped entirely.
4. DAO type script runs, validates Output 1 (110 CKB) against Input A ŌĆö passes.
5. Output 2 (100 CKB) vs Input B (50 CKB): **50 CKB created from nothing**, with no verifier catching it.

**Analog to the external report:**

| External Report | CKB Analog |
|---|---|
| `hasSignedDocs()` returns `false` (not all docs signed) | `valid_dao_withdraw_transaction()` returns `true` (any DAO input) |
| `verifySignature()` signs only one doc, execution continues | `OutputsSumOverflow` check skipped for entire transaction |
| Remaining unsigned docs never enforced | Non-DAO capacity balance never enforced |
| Buyer acquires tokens without signing all documents | Attacker acquires CKB without providing matching inputs |

### Impact Explanation

An attacker with a valid DAO cell (a normal, permissionless action) can inflate their CKB balance by an arbitrary amount per withdrawal transaction. This is a direct violation of the fundamental CKB invariant that `sum(outputs.capacity) Ōēż sum(inputs.capacity)` for non-cellbase transactions. Repeated exploitation inflates the total CKB supply, breaking consensus and economic security across all nodes.

### Likelihood Explanation

Any CKB holder who has ever deposited into the DAO can exploit this. No privileged access, leaked keys, or majority hashpower is required. The attacker only needs to construct a valid DAO withdrawal transaction and add extra outputs ŌĆö a straightforward RPC call (`send_transaction`). The precondition (owning a DAO cell) is a normal, widely performed user action.

### Recommendation

Replace the global skip with a per-cell capacity accounting approach. The `OutputsSumOverflow` check should be applied to the non-DAO portion of the transaction independently:

```rust
pub fn verify(&self) -> Result<(), Error> {
    if !self.resolved_transaction.is_cellbase() {
        // Compute capacity balance only for non-DAO inputs/outputs.
        // DAO cells are verified by the DAO type script.
        let non_dao_inputs_sum = self.resolved_transaction
            .resolved_inputs
            .iter()
            .filter(|cm| !cell_uses_dao_type_script(&cm.cell_output, &self.dao_type_hash))
            .try_fold(Capacity::zero(), |acc, cm| {
                acc.safe_add(cm.cell_output.capacity().into())
            })?;
        let outputs_sum = self.resolved_transaction.outputs_capacity()?;
        let dao_outputs_sum = /* sum of DAO outputs */;
        let non_dao_outputs_sum = outputs_sum.safe_sub(dao_outputs_sum)?;

        if non_dao_inputs_sum < non_dao_outputs_sum {
            return Err(TransactionError::OutputsSumOverflow { ... }.into());
        }
    }
    // ... per-output occupied capacity checks unchanged
}
```

Alternatively, remove the skip entirely and let the DAO type script enforce its own invariant, while the `CapacityVerifier` enforces the global invariant.

### Proof of Concept

The existing test `test_skip_dao_capacity_check` already demonstrates the bypass ŌĆö it constructs a transaction with **zero inputs** and a 500 CKB DAO output and asserts `verifier.verify().is_ok()`, confirming the `OutputsSumOverflow` check is not enforced when the DAO type script is involved: [4](#0-3) 

A concrete exploit transaction would be:

```
inputs:
  [0] DAO cell (100 CKB, deposited at block N)   ŌåÉ triggers valid_dao_withdraw_transaction()
  [1] regular cell (50 CKB)

outputs:
  [0] regular cell (110 CKB)   ŌåÉ DAO type script validates this
  [1] regular cell (100 CKB)   ŌåÉ no verifier checks this; 50 CKB created from nothing

header_deps: [deposit_block_hash, withdraw_block_hash]
witnesses:   [DAO witness for input[0], empty]
```

`CapacityVerifier::valid_dao_withdraw_transaction()` returns `true` (input[0] is a DAO cell), the `OutputsSumOverflow` guard is skipped, the DAO type script validates output[0], and output[1]'s 50 CKB excess is accepted by every node.
