### Title
`getPricesNoOlderThan` Uses Subscription-Level Max Timestamp Instead of Per-Feed `publishTime`, Returning Stale Prices for Closed-Market Feeds — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` functions in the Pyth Pulse `Scheduler` contract check staleness against a single subscription-level field `status.priceLastUpdatedAt`. This field is set to the **maximum** `publishTime` across all feeds in the batch update. However, individual feeds — particularly closed-market feeds (e.g., equities outside trading hours) — may carry `publishTime` values that are days old. A reader protocol calling `getPricesNoOlderThan` with a tight `age_seconds` threshold will pass the staleness check (because the max timestamp is recent) but receive individual feed prices that are arbitrarily stale, with no indication that the freshness guarantee has been violated.

---

### Finding Description

**Root cause in `_validateShouldUpdatePrices`:**

When a keeper calls `updatePriceFeeds`, `_validateShouldUpdatePrices` computes `updateTimestamp` as the **maximum** `publishTime` across all feeds in the batch:

```solidity
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

This `updateTimestamp` is then stored as `status.priceLastUpdatedAt`:

```solidity
status.priceLastUpdatedAt = latestPublishTime;
```

The code comment explicitly acknowledges this design: *"Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period."*

**Broken guarantee in `getPricesNoOlderThan`:**

```solidity
function getPricesNoOlderThan(
    uint256 subscriptionId,
    bytes32[] calldata priceIds,
    uint256 age_seconds
) external view override onlyWhitelistedReader(subscriptionId)
  returns (PythStructs.Price[] memory prices)
{
    SchedulerStructs.SubscriptionStatus memory status = _state
        .subscriptionStatuses[subscriptionId];

    if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
        revert PythErrors.StalePrice();

    prices = this.getPricesUnsafe(subscriptionId, priceIds);
}
```

The staleness check compares `block.timestamp` against `priceLastUpdatedAt` (the max timestamp). If the subscription contains one active-market feed (e.g., ETH/USD, updated 30 seconds ago) and one closed-market feed (e.g., AAPL/USD, last updated 3 days ago during the prior trading session), then:

- `priceLastUpdatedAt` = ETH/USD's recent timestamp → staleness check passes for `age_seconds = 60`
- `getPricesUnsafe` returns both prices, including AAPL/USD with `publishTime` 3 days ago
- The caller receives a stale AAPL/USD price with no revert and no indication of staleness

The same flaw exists in `getEmaPricesNoOlderThan`.

---

### Impact Explanation

A reader protocol that calls `getPricesNoOlderThan(subscriptionId, [ETH_USD, AAPL_USD], 60)` to enforce a 60-second freshness window will:

1. Pass the staleness check (because `priceLastUpdatedAt` reflects ETH/USD's recent update)
2. Receive AAPL/USD with a `publishTime` from days ago
3. Use this stale price for financial decisions (collateral valuation, liquidation thresholds, trade pricing)

An adversary who knows the AAPL/USD price is stale can exploit the discrepancy between the stale on-chain price and the true current market price — for example, by opening a position against a lending protocol that uses the stale price as collateral value, or by avoiding a liquidation that should have been triggered.

---

### Likelihood Explanation

The Scheduler/Pulse contract is explicitly designed to support mixed-market subscriptions (the README and code comments both acknowledge closed-market feeds). Any subscription containing both 24/7 crypto feeds and session-based equity/commodity feeds will exhibit this behavior. The `getPricesNoOlderThan` function is the recommended safe API for readers (per the `IScheduler.sol` NatSpec and the Pulse README), making it likely that integrating protocols will rely on it for freshness guarantees.

---

### Recommendation

Replace the subscription-level `priceLastUpdatedAt` staleness check in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` with per-feed `publishTime` checks. For each requested price feed, verify that `distance(block.timestamp, feed.price.publishTime) <= age_seconds` before including it in the result. Alternatively, document clearly that `getPricesNoOlderThan` only guarantees that the **subscription was updated** within `age_seconds`, not that every individual feed price is that fresh, and provide a separate function that enforces per-feed freshness.

---

### Proof of Concept

**Scenario:**
1. A subscription contains two feeds: `ETH/USD` (24/7) and `AAPL/USD` (equity, market closed).
2. A keeper pushes an update at time `T`. ETH/USD has `publishTime = T`, AAPL/USD has `publishTime = T - 3 days` (last trading session).
3. `priceLastUpdatedAt` is set to `T` (the max).
4. At time `T + 30s`, a reader protocol calls `getPricesNoOlderThan(subscriptionId, [ETH_USD, AAPL_USD], 60)`.
5. `distance(T+30, T) = 30 < 60` → staleness check passes, no revert.
6. `getPricesUnsafe` returns AAPL/USD with `publishTime = T - 3 days`.
7. The reader protocol uses the 3-day-old AAPL/USD price believing it is at most 60 seconds old.
