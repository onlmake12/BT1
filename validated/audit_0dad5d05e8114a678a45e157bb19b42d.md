### Title
Legacy-path `revealWithCallback` propagates uncaught callback revert, permanently locking provider request slots — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When a request is made via `requestWithCallback` to a provider whose `defaultGasLimit == 0` (a "legacy" provider), the resulting request has `req.gasLimit10k = 0`. In `revealWithCallback`, this causes execution to take the legacy `else` branch, which calls `IEntropyConsumer(callAddress)._entropyCallback(...)` **without any error handling**. If the requester contract's callback always reverts, the entire `revealWithCallback` transaction reverts — including the preceding `clearRequest` call — leaving the request permanently stuck in storage. The provider can never clear that sequence number.

**Step 3 — The uncaught revert rolls back `clearRequest`**

`clearRequest` is called before the callback (checks-effects-interactions pattern), but there is no `try/catch` or `excessivelySafeCall` around `_entropyCallback`. If the callback reverts, the EVM rolls back the entire transaction, including the `clearRequest`. The request's `sequenceNumber` is restored to its non-zero value, keeping it permanently active. [3](#0-2) 

**Step 4 — Contrast with the new path**

The new path (gasLimit10k != 0) uses `excessivelySafeCall`, which catches reverts and transitions the request to `CALLBACK_FAILED` state, allowing the provider to eventually clear it. The legacy path has no such protection. [4](#0-3) 

**Step 5 — `isActive` check confirms the slot remains occupied**

A request is active as long as `sequenceNumber != 0`. Since the revert rolls back `clearRequest` (which sets `sequenceNumber = 0`), the slot remains active indefinitely. [5](#0-4)

### Impact Explanation

- The provider can never successfully call `revealWithCallback` for the stuck sequence number — every attempt reverts.
- The request slot in the 32-slot primary array (`_state.requests[32]`) is permanently occupied.
- An attacker can repeat this for all 32 primary slots (each mapping to a different `shortKey`), forcing all future requests into the overflow mapping (`requestsOverflow`), which is significantly more expensive in gas.
- The provider's sequence number for each stuck request is permanently wasted. [6](#0-5) 

---

### Likelihood Explanation

- Any unprivileged user can deploy a contract that always reverts in `_entropyCallback` and call `requestWithCallback` — no special access required.
- Legacy providers (those that registered before `defaultGasLimit` was introduced and never called `setDefaultGasLimit`) have `defaultGasLimit == 0` by default.
- The attacker only pays the provider fee per request; the griefing is permanent and unbounded in duration.
- The attack is local-testable and requires no external oracle or privileged role.

---

### Recommendation

Wrap the legacy-path callback in a `try/catch` (or use `excessivelySafeCall`) so that a reverting callback does not roll back `clearRequest`. For example:

```solidity
clearRequest(provider, sequenceNumber);
if (len != 0) {
    try IEntropyConsumer(callAddress)._entropyCallback(sequenceNumber, provider, randomNumber) {
        // success
    } catch {
        emit CallbackFailed(...);
    }
}
```

Alternatively, migrate all providers to the new gas-limit flow by requiring `defaultGasLimit > 0` for all active providers, or by treating `gasLimit10k == 0` requests as if they had a minimum safe gas limit.
