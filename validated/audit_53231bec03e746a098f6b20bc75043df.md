Audit Report

## Title
Unbounded NAT Traversal Task Spawning via Repeated `ConnectionSync` Messages — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

## Summary
An unprivileged connected peer can cause the victim node to spawn an unbounded number of concurrent async tasks by sending repeated `ConnectionSync` P2P messages. The `forward_rate_limiter` intended to throttle this path is bypassable by varying the attacker-controlled `content.from` field. The only effective cap is the per-session `rate_limiter` at 30 req/sec, which still allows ~21,600 concurrent futures from a single session over the 30-second task lifetime, exhausting file descriptors and saturating the Tokio runtime.

## Finding Description

**Two rate limiters exist in `HolePunching`:**

1. `rate_limiter` — keyed by `(session_id, msg.item_id())`, 30 req/sec per session per message type. Checked in `received()` before dispatch. [1](#0-0) 

2. `forward_rate_limiter` — keyed by `(content.from, content.to, msg_item_id)`, 1 req/sec per tuple. Checked inside `execute()`. [2](#0-1) 

**The `forward_rate_limiter` is bypassable.** Its key includes `content.from`, which is attacker-controlled message content. By using a different `from` peer ID in each `ConnectionSync` message, the attacker creates a fresh rate-limiter bucket for each message, bypassing the 1/sec cap entirely. [2](#0-1) 

**The `pending_delivered` precondition is satisfiable.** The map is populated when the victim processes a `ConnectionRequest` with `to == victim_peer_id`. The attacker sends `ConnectionRequest` messages with different fake `from` peer IDs and fake TCP listen addresses. Each inserts a new entry. The map has no size limit and entries expire only after `TIMEOUT = 5 minutes`. [3](#0-2) [4](#0-3) 

The `HOLE_PUNCHING_INTERVAL` check (2 min) at line 161–167 of `connection_request.rs` only deduplicates the same `from_peer_id`; since the attacker uses unique fake `from` IDs each time, this check is irrelevant. [5](#0-4) 

**The spawn path.** Once a `ConnectionSync` message passes both rate limiters and finds a `pending_delivered` entry for `content.from`, `execute()` unconditionally calls `runtime::spawn(select_ok(tasks))` with up to `ADDRS_COUNT_LIMIT = 24` concurrent `try_nat_traversal` futures, with no cap on total outstanding spawns. [6](#0-5) [7](#0-6) 

**Each `try_nat_traversal` task** runs a retry loop for up to 30 seconds, creating a new `TcpSocket` each iteration (~200ms per attempt). [8](#0-7) 

## Impact Explanation

- **Steady-state concurrent futures**: 30 `ConnectionSync`/sec × 30-second task lifetime × 24 futures/task = **21,600 concurrent async futures** from a single session.
- **Open file descriptors**: Each `try_nat_traversal` future holds one `TcpSocket` per iteration (~200ms), yielding ~10,800 simultaneously open sockets — sufficient to exhaust the default Linux per-process fd limit (1,024–65,536).
- **Async runtime pressure**: 21,600 live futures each with active `delay_for` timers saturates the Tokio timer wheel and task scheduler, degrading or halting all other node operations (block sync, transaction relay, RPC).

This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10,001–15,000 points).

## Likelihood Explanation

Any single connected peer can execute this attack. No special privilege, leaked key, or majority hashpower is required. The attacker only needs one P2P session and the ability to craft `ConnectionRequest` and `ConnectionSync` messages with arbitrary `from` peer IDs — both are standard P2P messages with no cryptographic authentication on the `from` field. The `pending_delivered` entries can be pre-populated in under a minute (30/sec × 60 sec = 1,800 entries), and the attack sustains itself as long as the session remains open.

## Recommendation

1. **Cap total outstanding NAT traversal tasks**: Maintain an `Arc<AtomicUsize>` counter or `Arc<Semaphore>` and reject the spawn if the limit is exceeded (e.g., max 50–100 concurrent tasks).
2. **Key the `forward_rate_limiter` on the sending session**, not on attacker-controlled `content.from`/`content.to`, or add a per-session cap on how many task-spawning `ConnectionSync` messages are processed.
3. **Bound the `pending_delivered` map size** (e.g., evict oldest entries when size exceeds a threshold) to prevent unbounded pre-population.
4. **Authenticate the `from` field**: Require that `content.from` matches the peer ID of the sending session, so the attacker cannot use arbitrary fake peer IDs to bypass per-(from,to) rate limits.

## Proof of Concept

```
1. Connect to victim node via P2P (one session).

2. Send 1,800 ConnectionRequest messages (over ~60 seconds), each with:
   - A unique random fake `from` peer ID (e.g., random 32-byte keys)
   - `to` = victim's peer ID
   - 24 fake TCP listen addresses (e.g., 192.0.2.1:1–192.0.2.24:1) with the
     fake `from` peer ID embedded in each multiaddr
   Each passes rate_limiter (30/sec) and forward_rate_limiter (different `from`
   each time). Each inserts one entry into pending_delivered.

3. Send 30 ConnectionSync messages/sec, each with:
   - `from` = one of the pre-populated pending_delivered keys
   - `to` = victim's peer ID
   - empty `route`
   Each passes rate_limiter (30/sec) and forward_rate_limiter (different `from`
   each time), finds a pending_delivered entry, and triggers runtime::spawn with
   24 try_nat_traversal futures.

4. Observe after 30 seconds:
   - `lsof -p <ckb_pid> | wc -l` shows ~10,000+ open sockets
   - Tokio task count metric shows ~900 live spawned tasks, each with 24 active
     futures (~21,600 total)
   - Node becomes unresponsive to block sync and RPC requests
```
