### Title
Missing Peer Ban Enforcement for Unsolicited Response Messages in `BlockFilter::try_process()` Allows Unbounded Message Flooding — (File: `sync/src/filter/mod.rs`)

---

### Summary

The `BlockFilter::try_process()` function in `sync/src/filter/mod.rs` recognizes three unsolicited response-type messages (`BlockFilters`, `BlockFilterHashes`, `BlockFilterCheckPoints`) that a server should never receive from a client, but explicitly fails to ban the offending peer. A developer-left `// TODO: ban remote peer` comment confirms the enforcement is intentionally incomplete. Because `Status::ignored()` is returned instead of a ban-triggering status, any unprivileged peer can send these messages indefinitely without penalty, bypassing the intended misbehavior detection entirely.

---

### Finding Description

In `sync/src/filter/mod.rs`, `BlockFilter::try_process()` dispatches incoming `BlockFilterMessageUnionReader` variants: [1](#0-0) 

The three response-type variants — `BlockFilters`, `BlockFilterHashes`, and `BlockFilterCheckPoints` — are server-to-client messages. A well-behaved peer should never send them unsolicited to a full node acting as a filter server. The handler recognizes this:

```rust
packed::BlockFilterMessageUnionReader::BlockFilters(_)
| packed::BlockFilterMessageUnionReader::BlockFilterHashes(_)
| packed::BlockFilterMessageUnionReader::BlockFilterCheckPoints(_) => {
    // remote peer should not send block filter to us without asking
    // TODO: ban remote peer
    warn_target!(...);
    Status::ignored()
}
```

The `// TODO: ban remote peer` comment is the root cause: the intended enforcement action is absent. The `process()` caller only bans a peer when `status.should_ban()` returns `Some(ban_time)`: [2](#0-1) 

`Status::ignored()` does not satisfy that condition, so no ban is issued. The `BlockFilter` struct also carries no rate limiter field — unlike `HolePunching`, which has both `rate_limiter` and `forward_rate_limiter`: [3](#0-2) 

There is therefore no secondary defense against flooding.

The analog to the external report is direct: the original vulnerability had a missing opcode in a filter guard that allowed an unintended code path to execute without the intended action. Here, the filter handler has a missing enforcement action (ban) for a recognized-but-mishandled message class, allowing unintended peer behavior (repeated unsolicited messages) without penalty.

---

### Impact Explanation

An unprivileged peer connected to the CKB filter protocol can:

1. Send unlimited unsolicited `BlockFilters`, `BlockFilterHashes`, or `BlockFilterCheckPoints` messages.
2. Never be disconnected or banned — the node only logs a warning.
3. Consume CPU (message parsing and dispatch through `try_process`) and trigger warning-level log spam on every message.
4. Effectively bypass the misbehavior detection system for the entire filter protocol, since the peer is never penalized regardless of how many invalid messages it sends. [1](#0-0) 

---

### Likelihood Explanation

Exploitation requires only a standard P2P connection to the filter protocol — no keys, no privileges, no special state. Any peer that negotiates the filter protocol can immediately begin sending unsolicited response messages. The attack is trivially scriptable and requires no coordination. [4](#0-3) 

---

### Recommendation

Resolve the `// TODO: ban remote peer` comment by returning a ban-triggering status for the three unsolicited response variants. Replace:

```rust
Status::ignored()
```

with:

```rust
StatusCode::UnexpectedProtocolMessage.into()
```

or an equivalent status that causes `process()` to call `nc.ban_peer(...)`. This mirrors the handling already used in `LightClientProtocol::try_process()` for its wildcard unexpected-message arm: [5](#0-4) 

Additionally, consider adding a rate limiter to `BlockFilter` consistent with the pattern used in `HolePunching`.

---

### Proof of Concept

1. Connect to a CKB full node and negotiate the filter protocol (`SupportProtocols::Filter`).
2. Craft and send a `BlockFilterMessage` containing a `BlockFilters` payload (a response-type message).
3. Observe: the node emits a `warn_target!` log entry but does **not** disconnect or ban the peer.
4. Repeat step 2 in a tight loop — the peer is never penalized regardless of message volume.

The entry path is fully attacker-controlled: any peer that speaks the filter protocol can reach this code path with zero privileges. [6](#0-5)
