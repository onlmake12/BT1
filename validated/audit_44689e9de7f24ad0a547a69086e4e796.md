### Title
Unbounded While Loop in `executeCallback` Enables Permanent DoS on Victim Request Fulfillment — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol`'s `executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past cleared requests. An unprivileged attacker can create a large number of requests immediately after a victim's request, fulfill all of them (leaving the victim's request as the first unfulfilled), and cause the while loop to exceed the block gas limit when the victim's request is eventually fulfilled — permanently locking the victim's fee in the contract.

---

### Finding Description

In `Echo.sol`, after `clearRequest(sequenceNumber)` is called, the following loop runs unconditionally:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Its purpose is to advance `firstUnfulfilledSeq` past all cleared (fulfilled) slots. There is **no upper bound** on how many iterations this loop can perform in a single transaction — it is bounded only by the gap between `firstUnfulfilledSeq` and the next active request.

Any address can call `requestPriceUpdatesWithCallback` to create a request (paying a fee), and any address can call `executeCallback` to fulfill a request after the exclusivity period. There is no cap on the number of requests that can be created. [2](#0-1) [3](#0-2) 

**Attack sequence:**

1. Victim creates request at sequence number `K`. `firstUnfulfilledSeq` advances to `K`.
2. Attacker creates `N` requests at sequence numbers `K+1` through `K+N`, paying fees for each.
3. Attacker calls `executeCallback` for requests `K+1` through `K+N` (providing valid Pyth price data, which is publicly available from Hermes). Each call runs the while loop but stops immediately because request `K` is still active — `firstUnfulfilledSeq` remains at `K`.
4. All slots `K+1` through `K+N` are now cleared. `firstUnfulfilledSeq` is still `K`.
5. Provider (or anyone) attempts to fulfill request `K` via `executeCallback(K, ...)`.
6. `clearRequest(K)` executes, marking slot `K` as inactive.
7. The while loop now scans from `K` through `K+N` — **N iterations** — each calling `findRequest`, which performs 1–2 cold storage reads (~2100–4200 gas each).
8. For sufficiently large `N` (≈7,000 at 30M block gas limit), the loop exceeds the block gas limit and the entire transaction reverts.
9. Because the revert undoes `clearRequest(K)`, request `K` is restored as active. The loop will OOG on every subsequent fulfillment attempt.

There is **no expiry or refund mechanism** for requests in `Echo.sol`. The victim's fee is permanently locked. [4](#0-3) 

---

### Impact Explanation

- **Permanent DoS** on the fulfillment of a targeted request: `executeCallback` will always revert with out-of-gas for that sequence number.
- **Permanent fund loss** for the victim: the fee paid in `requestPriceUpdatesWithCallback` is locked in the contract with no refund path.
- The victim's `_echoCallback` is never invoked, breaking any downstream protocol logic that depends on the price update.

Impact: **4/5** — direct, irreversible fund loss and protocol DoS for the targeted user.

---

### Likelihood Explanation

- The attacker must pay fees for `N` requests. If `pythFeeInWei` and provider fees are low, the cost is minimal.
- Valid Pyth price update data is freely available from the public Hermes API, so fulfilling the attacker's own requests requires no privileged access.
- The attack requires no special role, no leaked key, and no governance majority.
- The attacker must time the attack to occur after the victim's request and before it is fulfilled — a realistic window given that fulfillment is not instantaneous.

Likelihood: **2/5** — requires ETH cost proportional to `N` and correct timing, but is otherwise fully permissionless.

---

### Recommendation

1. **Cap the number of requests that can be created** (analogous to `CANDIDATE_COUNT_LIMIT` in the CoreDAO fix), or impose a minimum fee high enough to make the attack economically infeasible.
2. **Bound the while loop** by a maximum number of iterations per call (e.g., advance `firstUnfulfilledSeq` by at most `MAX_SCAN` per `executeCallback` invocation), and allow the loop to resume in subsequent calls.
3. **Add a request expiry and refund mechanism** so that unfulfilled requests can be cancelled and fees returned, eliminating the permanent fund-lock consequence.
