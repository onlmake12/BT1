### Title
RBF `replace_count` Overcounts Shared Descendants Across Multiple Conflicts, Causing False Rejection of Valid Replacements - (File: `tx-pool/src/pool.rs`)

### Summary

In `check_rbf`, the `replace_count` guard against `MAX_REPLACEMENT_CANDIDATES` (100) accumulates descendant counts per-conflict in a loop without deduplicating shared descendants across conflicts. When two or more conflicting transactions share common descendants, those descendants are counted multiple times, inflating `replace_count` beyond the actual number of unique transactions to be evicted. A legitimate RBF replacement whose true unique eviction set is ≤ 100 is incorrectly rejected with `"Tx conflict with too many txs"`.

### Finding Description

In `tx-pool/src/pool.rs`, `check_rbf` iterates over each direct conflict and accumulates `replace_count` by adding `descendants.len() + 1` for each conflict:

```rust
// tx-pool/src/pool.rs  lines 613-624
let mut replace_count: usize = 0;
let mut all_conflicted = conflicts.clone();
let ancestors = self.pool_map.calc_ancestors(&short_id);
for conflict in conflicts.iter() {
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    replace_count += descendants.len() + 1;          // ← no dedup across conflicts
    if replace_count > MAX_REPLACEMENT_CANDIDATES {
        return Err(Reject::RBFRejected(format!(
            "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
            replace_count, MAX_REPLACEMENT_CANDIDATES,
        )));
    }
    ...
    all_conflicted.extend(entries);
}
```

`calc_descendants` returns a `HashSet<ProposalShortId>` (no intra-conflict duplicates), but the same descendant ID can appear in the `HashSet` returned for **conflict A** and again in the `HashSet` returned for **conflict B** when a downstream transaction spends outputs from both A and B. Each such shared descendant is counted once per conflict that owns it, inflating `replace_count`.

**Concrete scenario:**

1. Attacker (or any user) submits tx-A (spends input X) and tx-B (spends input Y) to the pool.
2. A chain of transactions D1…D50 is built spending outputs of both A and B (diamond-shaped dependency graph). Each Di is a descendant of both A and B.
3. User submits new-tx spending both X and Y (conflicting with A and B) with a sufficiently high fee.
4. `check_rbf` computes:
   - For conflict A: `descendants.len()` = 50, `replace_count` += 51
   - For conflict B: `descendants.len()` = 50, `replace_count` += 51 → total = 102
5. `replace_count (102) > MAX_REPLACEMENT_CANDIDATES (100)` → **rejected**.
6. Actual unique transactions to evict = {A, B, D1…D50} = 52, well within the limit.

The fee calculation in `calculate_min_replace_fee` already handles this correctly by deduplicating via a `HashMap` keyed on `id`:

```rust
// tx-pool/src/pool.rs  lines 104-108
// don't account for duplicate txs
let replaced_fees: HashMap<_, _> = conflicts
    .iter()
    .map(|c| (c.id.clone(), c.inner.fee))
    .collect();
```

The `replace_count` check lacks the equivalent deduplication step.

### Impact Explanation

Any user who submits a valid RBF replacement transaction whose true eviction set is ≤ 100 unique transactions, but whose conflicts share descendants, is permanently denied the replacement for that input set. The user cannot replace stuck or low-fee transactions and loses gas fees on each failed attempt. This is a targeted, reproducible DoS on the RBF mechanism reachable by any unprivileged tx-pool submitter.

### Likelihood Explanation

Diamond-shaped transaction graphs (a tx spending outputs of two in-pool parents) are a natural consequence of multi-input transactions and are not unusual in practice. An attacker can deliberately construct such a graph to grief a specific user's RBF attempt. No privileged access, key material, or majority hashpower is required — only the ability to submit transactions to the pool, which is open to all.

### Recommendation

Collect all descendants across all conflicts into a single `HashSet` before counting, mirroring the deduplication already applied in `calculate_min_replace_fee`:

```rust
// Collect unique transactions to be replaced
let mut all_to_replace: HashSet<ProposalShortId> = HashSet::new();
for conflict in conflicts.iter() {
    all_to_replace.insert(conflict.id.clone());
    let descendants = self.pool_map.calc_descendants(&conflict.id);
    all_to_replace.extend(descendants);
}
let replace_count = all_to_replace.len();
if replace_count > MAX_REPLACEMENT_CANDIDATES {
    return Err(Reject::RBFRejected(format!(
        "Tx conflict with too many txs, conflict txs count: {}, expect <= {}",
        replace_count, MAX_REPLACEMENT_CANDIDATES,
    )));
}
```

### Proof of Concept

1. Enable RBF (`min_rbf_rate > min_fee_rate`).
2. Submit tx-A spending `outpoint_X`; submit tx-B spending `outpoint_Y`.
3. Submit 50 transactions D1…D50 where each Di spends an output of both A and B (or chains through them such that each is a descendant of both A and B).
4. Submit new-tx spending both `outpoint_X` and `outpoint_Y` with fee ≥ `min_replace_fee`.
5. Observe `check_rbf` returns `RBFRejected("Tx conflict with too many txs, conflict txs count: 102, expect <= 100")` even though only 52 unique transactions would be evicted. [1](#0-0) [2](#0-1) [3](#0-2)
