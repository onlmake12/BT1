### Title
PDA Pre-Funding Grief Permanently Blocks `Initialize` Instruction — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

---

### Summary

An unprivileged attacker can permanently block the `Initialize` instruction of the `pyth-price-store` program by transferring as little as 1 lamport to the config PDA address before the deployer calls `Initialize`. Because the instruction unconditionally uses `system_instruction::create_account`, which fails with `AccountAlreadyInUse` if the target account has any non-zero lamports, and because no recovery path exists in the program, the publisher registration system is rendered permanently inoperable without a program upgrade.

---

### Finding Description

The config PDA address is fully deterministic once the program is deployed: [1](#0-0) 

```
Pubkey::find_program_address(&[b"CONFIG"], &program_id)
```

The `initialize` processor unconditionally calls `system_instruction::create_account` via `invoke_signed`: [2](#0-1) 

Solana's system program rejects `create_account` with `AccountAlreadyInUse` if the target account already has **any** non-zero lamports — even 1 lamport — regardless of whether the account has data or an assigned owner. There is no fallback to `allocate` + `assign`, no lamport-balance check before the CPI, and no alternative initialization instruction.

The `validate_config` helper only verifies the PDA derivation and writability; it does not check for pre-existing lamports: [3](#0-2) 

The `config::create` guard (`data.format != 0`) is never reached because the CPI fails first: [4](#0-3) 

---

### Impact Explanation

- `Initialize` is the sole entry point for setting up the global config account.
- `InitializePublisher` requires a valid, initialized config account to proceed.
- `SubmitPrices` requires a valid publisher config, which requires `InitializePublisher`.
- Blocking `Initialize` therefore blocks the entire publisher registration and price submission pipeline.
- No instruction in the program can drain or reassign the pre-funded PDA, so there is no in-program recovery path. [5](#0-4) 

---

### Likelihood Explanation

- Cost: 1 lamport + one transaction fee (~5000 lamports total, negligible).
- The program ID is public once deployed; the config PDA is trivially computable off-chain.
- The attacker only needs to act in the window between program deployment and the first `Initialize` call — a window that is always non-zero in practice.
- No privileged access, leaked keys, or social engineering required.

---

### Recommendation

Replace the unconditional `create_account` CPI with a pattern that handles pre-funded accounts:

1. Check `config.lamports()` before calling `create_account`.
2. If lamports are already present, use `invoke_signed` with `system_instruction::allocate` and `system_instruction::assign` separately (after topping up rent if needed), instead of `create_account`.
3. Alternatively, use `system_instruction::transfer` to top up, then `allocate` + `assign` — the standard Solana pattern for PDA initialization that is robust to pre-funding. [6](#0-5) 

---

### Proof of Concept

```rust
// 1. Deploy pyth-price-store; note program_id.
// 2. Compute config PDA off-chain:
let (config_pda, _bump) =
    Pubkey::find_program_address(&[b"CONFIG"], &program_id);

// 3. Attacker sends 1 lamport to config_pda via a simple SOL transfer.
//    config_pda is now a system-owned account with 1 lamport and zero data.

// 4. Deployer calls Initialize — invoke_signed fires create_account CPI.
//    System program returns AccountAlreadyInUse → transaction fails.

// 5. No recovery instruction exists; program is permanently uninitialized.
assert!(initialize_result.is_err());
```

pub fn validate_config<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    bump: u8,
    program_id: &Pubkey,
    require_writable: bool,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let config = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    let (config_pda, expected_bump) =
        Pubkey::find_program_address(&[CONFIG_SEED.as_bytes()], program_id);
    ensure!(ProgramError::InvalidInstructionData, bump == expected_bump);
    ensure!(
        ProgramError::InvalidArgument,
        pubkey_eq(config.key, &config_pda)
    );
    if require_writable {
        ensure!(ProgramError::InvalidArgument, config.is_writable);
    }
    Ok(config)
