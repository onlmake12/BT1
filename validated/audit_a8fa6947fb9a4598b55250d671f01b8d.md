Audit Report

## Title
Missing `MAX_RELAY_TXS_NUM_PER_BATCH` Count Check in `TransactionsProcess::execute()` Enables CPU Exhaustion via Oversized Relay Batch â€” (`sync/src/relayer/transactions_process.rs`)

## Summary

`TransactionsProcess::execute()` iterates and deserializes every transaction in a received `RelayTransactions` message before applying any count guard, while both sibling handlers (`TransactionHashesProcess` and `GetTransactionsProcess`) reject messages exceeding `MAX_RELAY_TXS_NUM_PER_BATCH` (32 767) immediately. An unprivileged relay peer can send a single `RelayTransactions` message containing far more transactions than the protocol limit, forcing the node to spend CPU deserializing the entire payload before the per-hash filter discards the unsolicited entries.

## Finding Description

**Confirmed missing check.** `TransactionsProcess::execute()` at lines 37â€“57 of `sync/src/relayer/transactions_process.rs` begins processing immediately:

```rust
pub fn execute(self) -> Status {
    let shared_state = self.relayer.shared().state();
    let txs: Vec<(TransactionView, Cycle)> = {
        ...
        self.message
            .transactions()
            .iter()
            .map(|tx| (tx.transaction().to_entity().into_view(), tx.cycles().into()))
            .filter(|(tx, _)| {
                !tx_filter.contains(&tx.hash())
                    && unknown_tx_hashes
                        .get_priority(&tx.hash())
                        .map(|priority| priority.requesting_peer() == Some(self.peer))
                        .unwrap_or_default()
            })
            .collect()
    };
```

The `.map()` call (`to_entity().into_view()`) fully deserializes each transaction *before* the `.filter()` predicate is evaluated. Because Rust's iterator adapters are lazy and fused element-by-element, the deserialization cost is O(N) in the total number of transactions in the message, regardless of how many survive the filter.

**Confirmed present in sibling handlers.**
- `TransactionHashesProcess::execute()` lines 29â€“35: rejects if `tx_hashes().len() > MAX_RELAY_TXS_NUM_PER_BATCH`.
- `GetTransactionsProcess::execute()` lines 35â€“39: rejects if `message_len > MAX_RELAY_TXS_NUM_PER_BATCH`.

**Why the filter does not fully mitigate the issue.** The `unknown_tx_hashes` filter ensures that only transactions the local node actually requested from this peer are forwarded to the tx pool. However, it does not prevent the deserialization of every entry in the message. An attacker can pad the message with arbitrarily many fabricated transactions; all are deserialized before any are discarded.

**Why the rate limiter does not fully mitigate the issue.** The rate limiter in `Relayer::try_process()` (lines 116â€“123 of `sync/src/relayer/mod.rs`) caps the number of `RelayTransactions` *messages* at 30 per second per peer. It does not bound the number of transactions *within* each message. At 30 messages/second, each carrying transactions up to the network frame limit, the per-second deserialization work can be orders of magnitude larger than `MAX_RELAY_TXS_NUM_PER_BATCH`.

**Why the outbound truncation does not fully mitigate the issue.** `ask_for_txs()` (line 614 of `sync/src/relayer/mod.rs`) truncates outbound `GetRelayTransactions` to `MAX_RELAY_TXS_NUM_PER_BATCH`. This limits how many transactions the node *requests*, but the protocol places no constraint on how many the remote peer *delivers*. The attacker controls the response and can include any number of additional transactions beyond what was requested.

## Impact Explanation

The attack forces the victim node to perform unbounded deserialization work on the network I/O thread (or its async equivalent) for every oversized `RelayTransactions` message received. Sustained at 30 messages/second, each carrying transactions up to the network frame size limit, this constitutes a CPU-exhaustion denial-of-service reachable from any connected relay peer with no authentication. This matches the allowed impact: **High â€” Vulnerabilities which could easily crash a CKB node** and/or **High â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

The attacker needs only a standard relay peer connection â€” no keys, no stake, no mining power. The attack is repeatable indefinitely (reconnect after a ban, or use multiple peers). The fabricated transactions in the padding do not need to be valid; they only need to be well-formed molecule-encoded structures that pass `check_data()`. The pre-staging step (sending `RelayTransactionHashes` to cause the node to issue requests) is optional; the attacker can send an oversized `RelayTransactions` message at any time and the deserialization cost is incurred regardless of how many entries survive the filter.

## Recommendation

Add the same count guard at the top of `TransactionsProcess::execute()` that already exists in the two sibling handlers:

```rust
pub fn execute(self) -> Status {
+   let tx_count = self.message.transactions().len();
+   if tx_count > MAX_RELAY_TXS_NUM_PER_BATCH {
+       return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
+           "RelayTransactions count({tx_count}) > MAX_RELAY_TXS_NUM_PER_BATCH({MAX_RELAY_TXS_NUM_PER_BATCH})",
+       ));
+   }
    let shared_state = self.relayer.shared().state();
    ...
```

`MAX_RELAY_TXS_NUM_PER_BATCH` must be imported from `crate::relayer` (it is already imported in both sibling files). The check should be placed before any iteration or deserialization begins.

1. Connect to a CKB node as a relay peer using the `RelayV3` protocol.
2. *(Optional pre-staging)* Send one or more `RelayTransactionHashes` messages, each within the 32 767-hash limit, to cause the node to issue `GetRelayTransactions` requests back to the attacker.

4. Send the message. Observe that `TransactionsProcess::execute()` returns without error and that the node's CPU usage spikes proportionally to K.
5. Repeat at the rate-limiter ceiling (30 messages/second) to sustain the load.
