Audit Report

## Title
Off-by-One in Spawn VM Count Guard Allows Creating 17 VMs When Limit Is 16 - (File: script/src/scheduler.rs)

## Summary
The `Scheduler::process_message_box` function in `script/src/scheduler.rs` uses a strict `>` comparison instead of `>=` when checking whether the number of alive VMs has reached `MAX_VMS_COUNT` (16) before permitting a `spawn` syscall. When exactly 16 VMs are alive, the condition `16 > 16` evaluates to `false`, so `boot_vm` is called and a 17th VM is created. This is an incorrect implementation of the CKB-VM spawn syscall resource limit.

## Finding Description
`MAX_VMS_COUNT` is defined as `16` at `script/src/scheduler.rs:34`. In `process_message_box`, the `Message::Spawn` handler at line 532 guards with:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
    // return MAX_VMS_SPAWNED
}
```

When `suspended.len() + instantiated.len() == 16`, the condition `16 > 16` is `false`, so execution falls through to `boot_vm` at line 539, which inserts a new entry into `self.instantiated` (line 1035) and `self.states` (line 1036), creating a 17th concurrent VM. The guard is never triggered at the boundary.

The analogous `MAX_FDS` check at line 595 correctly uses `>=`:

```rust
if self.fds.len() as u64 >= MAX_FDS {
    // return MAX_FDS_CREATED
}
```

The inconsistency is direct evidence that the `MAX_VMS_COUNT` guard is erroneous. No other guard in the spawn path compensates for this; `boot_vm` performs no count check of its own.

## Impact Explanation
This is an incorrect implementation of the CKB-VM spawn syscall, which maps to the allowed bounty impact: **"Incorrect implementation or behavior of CKB-VM or system scripts" — High (10001–15000 points)**. The protocol-defined invariant `alive_vms ≤ MAX_VMS_COUNT` is silently violated. Because all nodes run the same buggy code, every node will accept a transaction whose script creates 17 concurrent VMs, meaning the protocol resource limit is universally unenforced at the boundary. The extra VM consumes scheduler state and memory beyond the protocol-intended maximum.

## Likelihood Explanation
Any unprivileged transaction sender can trigger this. A script that spawns 15 children (root + 15 = 16 alive VMs) and then issues one additional `ckb_spawn` call will receive `SUCCESS` instead of `MAX_VMS_SPAWNED`. No special privileges, leaked keys, or victim interaction are required. The entry path is a standard lock or type script submission.

## Recommendation
Change the comparison operator from `>` to `>=` at `script/src/scheduler.rs:532`:

```diff
- if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
+ if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

This makes the `MAX_VMS_COUNT` guard consistent with the `MAX_FDS` guard and correctly rejects a spawn when the alive VM count is already at the protocol limit.

## Proof of Concept
1. Write a CKB script (lock or type) that uses the `ckb_spawn` syscall recursively or in a chain until `suspended.len() + instantiated.len() == 16` (root VM + 15 spawned children, all still alive/waiting).
2. From any alive VM, issue one additional `ckb_spawn` call.
3. **Current behavior (`>`):** `16 > 16` is `false` → spawn proceeds → `boot_vm` inserts VM 17 → syscall returns `SUCCESS`.
4. **Expected behavior (`>=`):** `16 >= 16` is `true` → spawn is rejected → syscall returns `MAX_VMS_SPAWNED`.

The existing test `check_spawn_max_vms_count` at `script/src/verify/tests/ckb_latest/features_since_v2023.rs:102–106` invokes test case `[10]` against `testdata/spawn_cases` but does not specifically probe the exact boundary of 16 simultaneously alive VMs, leaving the off-by-one undetected. A targeted unit test that holds 16 VMs alive concurrently and asserts the next spawn returns `MAX_VMS_SPAWNED` would reproduce the failure under the current code.
