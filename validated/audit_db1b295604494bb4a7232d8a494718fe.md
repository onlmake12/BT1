### Title
`ScriptHashTypeVerifier` Checks Lock Script Hash Type But Not Type Script Hash Type — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces that output **lock** script `hash_type` values are within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE = {0,1,2,4}`), but performs **no equivalent check on output type scripts**. A transaction sender can craft an output whose type script carries a structurally valid but consensus-disabled `hash_type` (e.g., `Data3 = 6`). That transaction passes the non-contextual verifier, enters the tx-pool, and only fails later during full script execution — wasting CPU and pool resources. The pattern is directly analogous to the reported issue: one code path in a sibling set of checks enforces a type/variant constraint while the others do not.

---

### Finding Description

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier`, the earliest rejection gate for submitted transactions. [1](#0-0) 

The verifier iterates over every output and calls `output.lock().hash_type()`, checking it against `ENABLED_SCRIPT_HASH_TYPE`: [2](#0-1) 

The enabled set is defined as: [3](#0-2) 

The verifier never inspects `output.type_().to_opt()`. A type script with `hash_type = 6` (`Data3`) is a structurally valid molecule value — `ScriptHashType::verify_value(6)` returns `true` because `6.is_multiple_of(2)` — so it passes the P2P-layer `check_data()` gate: [4](#0-3) 

It also passes `ScriptHashTypeVerifier` because that verifier only reads the lock script. The type script hash type is only caught much later inside `TxInfo::select_version()`: [5](#0-4) 

The error type itself is named and documented as a **lock-script-only** concern: [6](#0-5) 

The asymmetry is clear: lock script hash type → rejected at non-contextual stage; type script hash type → accepted at non-contextual stage, rejected only at script execution.

---

### Impact Explanation

An unprivileged transaction sender (RPC caller or P2P relay peer) can submit transactions whose outputs carry a type script with a consensus-disabled `hash_type`. These transactions:

1. Pass `NonContextualTransactionVerifier` (the cheap early gate).
2. Enter the tx-pool, consuming pool slots.
3. Trigger full script execution verification, consuming CPU cycles, before being rejected.

Repeated submission of such transactions is a low-cost, amplified resource-exhaustion vector against the tx-pool and the script execution engine. Because the check is missing at the non-contextual layer, the node cannot cheaply discard these transactions without running the expensive script verifier.

---

### Likelihood Explanation

The entry path is trivially reachable: any RPC caller can invoke `send_transaction` with a crafted output. No special privilege, key, or network position is required. The `ScriptHashType` enum exposes named variants (`Data3`, `Data4`, …) that are structurally valid but not consensus-enabled, making construction of such a transaction straightforward.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also check the `hash_type` of each output's optional type script, mirroring the existing lock-script check:

```rust
// After the existing lock-script check, add:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

This makes the non-contextual gate symmetric for both script roles, consistent with how `check_data()` already validates both lock and type scripts.

---

### Proof of Concept

1. Construct a transaction output with a valid lock script (`hash_type = 0`) and a type script with `hash_type = 6` (`Data3`, structurally valid, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC.
3. Observe: `ScriptHashTypeVerifier::verify()` returns `Ok(())` — the transaction is not rejected at the non-contextual stage.
4. The transaction enters the tx-pool and only fails during script execution with `InvalidScriptHashType`.
5. Repeat at high rate to exhaust tx-pool slots and script execution CPU.

The existing test `test_not_enabled_hash_type_output_lock` in `verification/src/tests/transaction_verifier.rs` demonstrates the lock-script path is covered: [7](#0-6) 

No equivalent test exists for the type-script path, confirming the gap.
