### Title
`FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD` Hardcoded at 10 Seconds Causes Near-Constant Scheduler Blocking on Chains with Clock Lag — (File: `target_chains/ethereum/pulse_sdk/solidity/SchedulerConstants.sol`)

---

### Summary

`FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD = 10 seconds` is a hardcoded constant used as the upper bound for acceptable price-update timestamps in `Scheduler.updatePriceFeeds`. On any EVM chain whose `block.timestamp` consistently lags Pythnet's wall-clock by more than 10 seconds, every keeper call to `updatePriceFeeds` reverts inside `parsePriceFeedUpdatesWithConfig`, rendering the Scheduler permanently non-functional without any code change or governance action.

---

### Finding Description

In `SchedulerConstants.sol`:

```solidity
uint64 public constant FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD = 10 seconds;
``` [1](#0-0) 

In `Scheduler.updatePriceFeeds`, this constant is passed directly as the `maxPublishTime` argument to the Pyth oracle:

```solidity
uint64 curTime = SafeCast.toUint64(block.timestamp);
(PythStructs.PriceFeed[] memory priceFeeds, uint64[] memory slots) =
    pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
        updateData,
        params.priceIds,
        0,
        curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD,   // ← block.timestamp + 10 s
        false, true, false
    );
``` [2](#0-1) 

Pythnet timestamps are derived from Pythnet's own Solana-based clock, which tracks real wall-clock time. On EVM chains where `block.timestamp` is set by validators and can lag real time by more than 10 seconds (e.g., chains that inherit L1 Ethereum block timestamps on L2s, or chains with slow/irregular block production), every freshly-generated Pythnet price update will carry a `publishTime` that exceeds `block.timestamp + 10`. The Pyth contract enforces the `maxPublishTime` bound and reverts the call. Because the fee deduction at line 305 (`status.balanceInWei -= pythFee`) is inside the same transaction, the revert rolls it back — but the keeper's gas is wasted and no update is ever stored. [3](#0-2) 

The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` bound is enforced separately in `_validateShouldUpdatePrices` and is not the issue here; the blocking occurs earlier, inside the Pyth oracle call itself. [4](#0-3) 

---

### Impact Explanation

All active subscriptions on the affected chain stop receiving price updates. Protocols that depend on the Scheduler for heartbeat or deviation-triggered feeds will operate on indefinitely stale prices. Downstream consequences include incorrect on-chain valuations, missed liquidations, and incorrect settlement — identical in character to the original report's "vault effectively nonfunctional" outcome. The constant is immutable in the deployed contract; no keeper action or subscription parameter change can work around it.

---

### Likelihood Explanation

Several production EVM chains exhibit `block.timestamp` lag exceeding 10 seconds relative to real wall-clock time:

- **Arbitrum One / Nova**: `block.timestamp` is the L1 Ethereum block timestamp, updated only when an L1 block is produced (~12 s intervals). During the interval between L1 blocks, `block.timestamp` is frozen while Pythnet continues advancing. A Pythnet update produced 11 s after the last L1 block will have `publishTime > block.timestamp + 10`.
- **Chains with slow or irregular block production**: Any chain where validators set `block.timestamp` conservatively behind real time will exhibit the same pattern.

Because Pythnet produces updates continuously at ~400 ms intervals, the window in which a valid update exists (i.e., `publishTime ≤ block.timestamp + 10`) is narrow and may never be satisfied if the lag is structural. This makes the blocking near-constant, not occasional.

---

### Recommendation

Increase `FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD` to a value that accommodates realistic clock skew across all supported chains — at minimum 60 seconds, consistent with the 1-minute future-timestamp limit used in the Echo contract. Alternatively, make the value a constructor/initializer parameter so it can be set per deployment to match the target chain's block-time characteristics.

---

### Proof of Concept

1. Deploy the Scheduler on Arbitrum One. The last L1 Ethereum block was produced 11 seconds ago, so `block.timestamp` is 11 seconds behind real time.
2. Pythnet produces a price update at real time `T`; its `publishTime = T`.
3. Keeper calls `updatePriceFeeds`. Inside the call, `curTime = block.timestamp = T - 11`.
4. `parsePriceFeedUpdatesWithConfig` is called with `maxPublishTime = T - 11 + 10 = T - 1`.
5. The price update has `publishTime = T > T - 1`; the Pyth contract reverts.
6. The next Pythnet update (400 ms later) has `publishTime = T + 0.4`, still greater than `T - 1`. Every subsequent call reverts identically.
7. All subscriptions on this Scheduler deployment are permanently frozen with no keeper-accessible remedy.
