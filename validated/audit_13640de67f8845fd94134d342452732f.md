### Title
Unbounded Linear Scan in `executeCallback` Can Exhaust Gas and Permanently DoS Request Fulfillment — (`Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that linearly scans forward from `_state.firstUnfulfilledSeq` to advance the pointer past fulfilled requests. Each iteration performs a cold storage read (`SLOAD`, ~2100 gas). When many requests have been fulfilled out-of-order, a single `executeCallback` call must scan through all previously-fulfilled sequence numbers, potentially exhausting the block gas limit and causing the transaction to revert. This permanently prevents the target request from being fulfilled and locks user funds.

---

### Finding Description

In `Echo.sol`, after clearing a request and before invoking the consumer callback, the following loop runs:

```solidity
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

The code's own comment acknowledges the problem: *"TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."* [2](#0-1) 

`findRequest` performs a storage lookup for each sequence number. With cold SLOADs at ~2100 gas each, scanning 1,000 fulfilled-but-not-yet-scanned sequence numbers costs ~2.1M gas — well within range of exhausting a transaction's gas budget when combined with the rest of `executeCallback`'s logic (Pyth fee parsing, callback invocation, etc.).

The `getFirstActiveRequests` view function has the same linear scan pattern starting from `firstUnfulfilledSeq`:

```solidity
uint64 currentSeq = _state.firstUnfulfilledSeq;
while (actualCount < count && currentSeq < _state.currentSequenceNumber) {
    Request memory req = findRequest(currentSeq);
    ...
    currentSeq++;
}
``` [3](#0-2) 

The gas benchmark test explicitly documents the worst-case scenario — fulfilling requests in reverse order — and notes that the last fulfillment is the most expensive because it must scan through all previously-fulfilled sequence numbers: [4](#0-3) 

---

### Impact Explanation

If the while loop exhausts gas, the entire `executeCallback` transaction reverts. This means:

1. The request is **not cleared** — it remains active and locked.
2. The consumer callback is **never executed** — the user never receives their price update.
3. The provider is **not credited** — fees are locked.
4. The request **cannot be retried** with less gas because the loop length is determined by the global state (`firstUnfulfilledSeq` gap), not by the caller.

User funds (fees paid at request time) are permanently locked in the contract with no recovery path.

---

### Likelihood Explanation

The attack is reachable by any unprivileged user interacting with the Echo contract:

1. An attacker creates N requests (paying the required fee for each).
2. Requests 2 through N are fulfilled by the provider (or by anyone after the exclusivity period expires).
3. `firstUnfulfilledSeq` remains at 1 because request 1 is still active.
4. When request 1 is finally fulfilled, the while loop must scan through N−1 already-fulfilled sequence numbers.
5. At ~2100 gas per cold SLOAD, N ≈ 1,000 requests costs ~2.1M gas in the loop alone, which combined with the rest of `executeCallback` (Pyth fee parsing, callback gas, etc.) can exceed the block gas limit or the gas provided by the caller.

This does not require any privileged access, leaked keys, or external oracle manipulation. The attacker only needs to pay fees for N requests, which is a bounded, realistic cost.

Additionally, this can occur **organically** without a deliberate attacker: in a high-throughput environment where many requests are fulfilled out-of-order, the gap between `firstUnfulfilledSeq` and the current sequence can grow large enough to cause gas exhaustion on legitimate fulfillments.

---

### Recommendation

Replace the linear scan with a doubly-linked list of active requests (as the code's own TODO suggests), or use a skip-forward approach where `firstUnfulfilledSeq` is only advanced lazily and bounded per call (e.g., advance at most K steps per `executeCallback` invocation). A cap on the number of iterations per call would prevent gas exhaustion while still making progress:

```solidity
uint256 MAX_SCAN = 100;
uint256 scanned = 0;
while (
    scanned < MAX_SCAN &&
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
    scanned++;
}
```

---

### Proof of Concept

The existing benchmark test in `EchoGasBenchmark.t.sol` already demonstrates the worst-case scenario with 10 requests fulfilled in reverse order: [5](#0-4) 

Scaling this to 1,000+ requests (each paying the required fee) would cause the final `executeCallback` to revert due to gas exhaustion. The attacker's entry path is:

1. Call `requestPriceUpdatesWithCallback` N times (paying fees).
2. Allow or trigger fulfillment of requests 2..N (anyone can call `executeCallback` after the exclusivity period).
3. Attempt to fulfill request 1 — the while loop scans N−1 fulfilled slots, exhausting gas and reverting.
4. Request 1 is permanently stuck; user funds are locked.

    // in order to update _state.lastUnfulfilledReq

    // NOTE: Run test with `forge test --gas-report --match-test testMultipleRequestsOutOfOrderFulfillment`
    // and observe the `max` value for `executeCallback` to see the cost of the most expensive request.
    function testMultipleRequestsOutOfOrderFulfillment() public {
        uint64 timestamp = SafeCast.toUint64(block.timestamp);
        bytes32[] memory priceIds = createPriceIds(2);
        uint32 callbackGasLimit = 100000;
        uint128 totalFee = echo.getFee(
            defaultProvider,
            callbackGasLimit,
            priceIds
        );

        // Create 10 requests
        uint64[] memory sequenceNumbers = new uint64[](10);
        vm.deal(address(consumer), 10 ether);

        for (uint i = 0; i < 10; i++) {
            vm.prank(address(consumer));
            sequenceNumbers[i] = echo.requestPriceUpdatesWithCallback{
                value: totalFee
            }(
                defaultProvider,
                timestamp + uint64(i),
                priceIds,
                callbackGasLimit
            );
        }

        PythStructs.PriceFeed[] memory priceFeeds = createMockPriceFeeds(
            timestamp
        );
        mockParsePriceFeedUpdates(pyth, priceFeeds);
        bytes[] memory updateData = createMockUpdateData(priceFeeds);

        // Execute callbacks in reverse
        uint startGas = gasleft();
        for (uint i = 9; i > 0; i--) {
            echo.executeCallback(
                defaultProvider,
                sequenceNumbers[i],
                updateData,
                priceIds
            );
        }
        uint midGas = gasleft();

        // Execute the first request last - this would be the most expensive
        // in the original implementation as it would need to loop through
        // all sequence numbers
        echo.executeCallback(
            defaultProvider,
            sequenceNumbers[0],
            updateData,
            priceIds
        );
        uint endGas = gasleft();

        // Log gas usage for the last callback which would be the most expensive
        // in the original implementation (need to run test with -vv)
        console.log(
            "Gas used for last callback (seq 1): %s",
            vm.toString(midGas - endGas)
        );
        console.log(
            "Gas used for all other callbacks: %s",
            vm.toString(startGas - midGas)
        );
    }
