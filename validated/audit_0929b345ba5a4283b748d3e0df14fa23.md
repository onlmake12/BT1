### Title
Hole Punching `forward_rate_limiter` Bypass via `to` PeerId Rotation Enables Gossip Amplification — (`network/src/protocols/hole_punching/component/connection_request.rs`, `network/src/protocols/hole_punching/mod.rs`)

---

### Summary

An unprivileged remote peer can bypass the `forward_rate_limiter` in `ConnectionRequestProcess::execute` by rotating the `to` PeerId across successive `ConnectionRequest` messages. Because the limiter key is `(from, to, item_id)` and `item_id` is a fixed enum discriminant (`0` for all `ConnectionRequest` messages), each new `to` value creates a fresh, unthrottled key. Every such message that reaches `forward_message` — when the `to` peer is not directly connected — triggers a `filter_broadcast` to `sqrt(total_peers)` nodes. The outer `rate_limiter` caps the attacker at 30 messages/second from a single session, yielding a sustained amplification of **30 × sqrt(N)** outgoing messages per second per attacker connection.

---

### Finding Description

**Two rate limiters govern the path:**

**Outer `rate_limiter`** — keyed by `(session_id, msg.item_id())`: [1](#0-0) 

`item_id()` returns the enum discriminant — always `0` for `ConnectionRequest`. So the effective key is `(session_id, 0)`, allowing **30 `ConnectionRequest` messages per second** from a single peer session.

**Inner `forward_rate_limiter`** — keyed by `(from_peer_id, to_peer_id, msg_item_id)`: [2](#0-1) 

`msg_item_id` is also `msg.item_id()` = `0` for all `ConnectionRequest` messages. The limiter is configured at 1 request/second per key: [3](#0-2) 

**The bypass:** each unique `to` PeerId produces a distinct key `(from, to_i, 0)`. An attacker sending 30 messages/second with 30 different `to` values creates 30 fresh keys, all passing the limiter simultaneously.

**The amplification:** when `to` is not in the peer registry, `forward_message` calls `filter_broadcast` to `sqrt(total_peers)` nodes: [4](#0-3) 

**Secondary issue:** the `forward_rate_limiter` uses a `HashMapStateStore` that grows unboundedly as new `(from, to, 0)` keys are inserted. `retain_recent()` is only called on peer disconnect: [5](#0-4) 

A long-lived attacker connection continuously adds entries, causing unbounded memory growth.

---

### Impact Explanation

- **Amplification factor:** sqrt(N) per message; with the 30 req/sec outer cap, a single attacker connection generates **30 × sqrt(N)** outgoing `HolePunching` messages per second on the relay node.
- For N=125 peers (typical CKB node): ~330 outgoing messages/second per attacker.
- Multiple simultaneous attacker connections multiply the effect linearly.
- The `HolePunching` protocol shares the same P2P message queue as sync/relay traffic; sustained amplification degrades throughput for block and transaction propagation.
- Unbounded `forward_rate_limiter` map growth adds memory pressure proportional to attacker session duration × 30 keys/second.

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no authentication, no stake, no PoW. The attacker constructs valid `ConnectionRequest` messages with syntactically valid but unreachable `to` PeerIds (random 32-byte public keys). The bypass is mechanical: rotate `to` on each message. The path is fully local-testable with a single node and one attacker peer.

---

### Recommendation

1. **Fix the `forward_rate_limiter` key:** include a per-message nonce or timestamp from the message content, or key solely on `(from_peer_id, session_id)` with a global budget, so rotating `to` does not create fresh quota.
2. **Cap the `forward_rate_limiter` map size:** evict or reject new keys once the map exceeds a bound (e.g., 1,000 entries), or call `retain_recent()` periodically (not only on disconnect).
3. **Add a per-session broadcast budget:** track total `filter_broadcast` calls per session and enforce a hard cap independent of the `(from, to)` key space.

### Proof of Concept

```
1. Attacker connects to relay node R (N peers connected, none is the target).
2. Each second, attacker sends 30 ConnectionRequest messages:
     from = attacker_peer_id
     to   = PeerId::random()   // new value each message
     listen_addrs = [valid TCP addr]
     max_hops = 6
3. Outer rate_limiter key = (session_id, 0): allows all 30.
4. forward_rate_limiter key = (attacker_peer_id, to_i, 0): each is new, all pass.
5. forward_message: to_i not in peer_registry → filter_broadcast to sqrt(N) peers.
6. Outgoing messages from R per second = 30 × sqrt(N).
   For N=125: 330 outgoing HolePunching messages/second from one attacker connection.
7. Assert amplification_factor = outgoing / incoming = sqrt(N) >> 1. ✓
```
