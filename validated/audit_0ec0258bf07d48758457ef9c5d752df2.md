Audit Report

## Title
`break` on unindexed input silently skips `spend_cell()` for all subsequent inputs, leaving indexed outputs permanently marked `is_spent=0` — (`util/rich-indexer/src/indexer/mod.rs`)

## Summary
In `AsyncRichIndexer::insert_transaction`, the input loop calls `spend_cell()` for each input. When `spend_cell()` returns `false` (the referenced output is absent from the DB — the expected case for outputs filtered by `cell_filter`), the code executes `break`, terminating the entire loop. All subsequent inputs are silently skipped: their corresponding indexed outputs are never updated to `is_spent=1`, leaving them permanently marked as live cells in the indexer DB despite being consumed on-chain.

## Finding Description
`spend_cell()` (`util/rich-indexer/src/indexer/insert.rs`, lines 410–426) executes `UPDATE output SET is_spent = 1 WHERE tx_id = ... AND output_index = ...` and returns `Ok(rows_affected > 0)`. It returns `Ok(false)` whenever the referenced output is not present in the `output` table — which is the normal, expected result for any output excluded by `cell_filter`.

In `insert_transaction` (`util/rich-indexer/src/indexer/mod.rs`, lines 228–248), the caller checks this return value:

```rust
if !spend_cell(&out_point, tx).await? {
    break;   // <-- terminates the entire input loop
}
```

When `cell_filter` is active and the first input's previous output is not indexed, `spend_cell()` returns `false` and `break` exits the loop. Every subsequent input — including those whose previous outputs **are** indexed — never has `spend_cell()` called. Those outputs remain at `is_spent=0` in the DB indefinitely.

The correct behavior is `continue`: skip the current unindexed input and proceed to the next one.

## Impact Explanation
This is an incorrect implementation of the CKB state storage mechanism (rich-indexer). Indexed outputs that are actually spent on-chain will permanently appear as live/unspent cells in the indexer DB. RPC endpoints such as `get_cells` and balance queries will return these phantom unspent cells, causing wallets and applications relying on the indexer to compute incorrect UTXO sets. This matches the **Medium (2001–10000 points)** allowed impact: suboptimal/incorrect implementation of CKB state storage mechanism.

## Likelihood Explanation
The trigger requires: (1) `cell_filter` is enabled (a documented, supported operator configuration); (2) a valid on-chain transaction has at least two inputs; (3) the first input's previous output is excluded by the filter; (4) at least one subsequent input's previous output is indexed. Any unprivileged user can craft exactly such a transaction — it is a standard valid CKB transaction requiring no special privileges, no majority hashpower, and no leaked keys. The bug fires automatically during every `append` call that processes such a transaction.

## Recommendation
Replace `break` with `continue` at line 232 of `util/rich-indexer/src/indexer/mod.rs`:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // output not in DB (filtered out); skip this input, do not abort the loop
}
```

This preserves the existing intent (do not attempt to mark unindexed outputs as spent) while ensuring every subsequent input in the transaction is still processed.

## Proof of Concept
1. Start a CKB node with the rich-indexer and a `cell_filter` that excludes outputs with lock script `LOCK_A` but includes outputs with lock script `LOCK_B`.
2. Mine a block containing `tx_setup`: a cellbase plus a transaction that creates `output_A` (lock=`LOCK_A`, unindexed by filter) and `output_B` (lock=`LOCK_B`, indexed).
3. Mine a second block containing `tx_spend` with inputs `[output_A, output_B]` (unindexed output is input 0).
4. After the indexer processes the second block, query the DB:
   ```sql
   SELECT is_spent FROM output WHERE -- output_B's row
   ```
5. **Expected (correct):** `is_spent = 1`.
   **Actual (buggy):** `is_spent = 0` — `output_B` appears as a live cell despite being consumed on-chain, because the `break` at line 232 prevented `spend_cell()` from ever being called for input index 1.
