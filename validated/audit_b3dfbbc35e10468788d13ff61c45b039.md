### Title
`getPricesNoOlderThan` and `getEmaPricesNoOlderThan` Check Subscription-Level Max Timestamp Instead of Per-Feed Timestamps — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.getPricesNoOlderThan` and `Scheduler.getEmaPricesNoOlderThan` validate freshness using `status.priceLastUpdatedAt`, which is the **maximum** `publishTime` across all feeds in the subscription. Individual feeds can have `publishTime` values far older than the requested `age_seconds` (e.g., closed-market feeds), yet the staleness check passes and stale prices are returned without revert.

---

### Finding Description

In `_validateShouldUpdatePrices`, `status.priceLastUpdatedAt` is set to the **maximum** `publishTime` across all feeds in the batch:

```solidity
// Scheduler.sol lines 366–371
uint256 updateTimestamp = 0;
for (uint8 i = 0; i < priceFeeds.length; i++) {
    if (priceFeeds[i].price.publishTime > updateTimestamp) {
        updateTimestamp = priceFeeds[i].price.publishTime;
    }
}
```

The code comment explicitly acknowledges this design: *"Use the most recent timestamp, as some asset markets may be closed. Closed markets will have a publishTime from their last trading period."* [1](#0-0) 

Then `status.priceLastUpdatedAt` is written with this max value:

```solidity
// Scheduler.sol line 340
status.priceLastUpdatedAt = latestPublishTime;
``` [2](#0-1) 

However, `getPricesNoOlderThan` and `getEmaPricesNoOlderThan` both validate freshness against this **shared subscription-level max timestamp** rather than each individual feed's `publishTime`:

```solidity
// Scheduler.sol lines 546–554
SchedulerStructs.SubscriptionStatus memory status = _state
    .subscriptionStatuses[subscriptionId];

if (distance(block.timestamp, status.priceLastUpdatedAt) > age_seconds)
    revert PythErrors.StalePrice();

prices = this.getPricesUnsafe(subscriptionId, priceIds);
``` [3](#0-2) 

The same pattern is repeated in `getEmaPricesNoOlderThan`: [4](#0-3) 

**Concrete scenario:**

1. A subscription contains two feeds: `BTC/USD` (active, `publishTime = now`) and `GOLD/USD` (closed market, `publishTime = 3 days ago`).
2. A keeper calls `updatePriceFeeds`. `priceLastUpdatedAt` is set to `now` (the max).
3. A consumer calls `getPricesNoOlderThan(subscriptionId, [BTC, GOLD], 3600)` (1-hour freshness).
4. The check `distance(block.timestamp, priceLastUpdatedAt) > 3600` evaluates to `false` → **no revert**.
5. The consumer receives `GOLD/USD` with a `publishTime` 3 days old, believing it is fresh.

The analog to the external report is exact: the report checked `lpTokenETH.lastInteractedTimestamp` (a shared global token's state) instead of `_token.lastInteractedTimestamp` (the specific item's state). Here, `status.priceLastUpdatedAt` (a shared subscription-level max) is checked instead of each individual `priceFeed.price.publishTime`.

---

### Impact Explanation

Consumers relying on `getPricesNoOlderThan` or `getEmaPricesNoOlderThan` to enforce a freshness SLA receive prices that silently violate that SLA for any closed-market feed in the subscription. Downstream protocols (e.g., lending, derivatives) that use these functions as a freshness gate may execute trades or liquidations based on stale prices, leading to incorrect financial outcomes. The function's name and interface create a false guarantee of per-feed freshness.

---

### Likelihood Explanation

Mixed subscriptions containing both active and closed-market feeds (e.g., equities alongside crypto) are a natural and documented use case — the code comment explicitly anticipates closed-market feeds. Any consumer that subscribes to such a mix and calls `getPricesNoOlderThan` is affected. The entry path requires no special privileges: any whitelisted reader (or any reader if `whitelistEnabled = false`) can trigger the issue. [5](#0-4) 

---

### Recommendation

Replace the single subscription-level timestamp check with a per-feed check inside `_getPricesInternal`, or add a secondary per-feed staleness validation in `getPricesNoOlderThan` and `getEmaPricesNoOlderThan`:

```solidity
// In getPricesNoOlderThan, after retrieving priceFeeds:
for (uint i = 0; i < priceFeeds.length; i++) {
    if (distance(block.timestamp, priceFeeds[i].price.publishTime) > age_seconds)
        revert PythErrors.StalePrice();
}
```

This ensures the freshness guarantee is enforced per-feed rather than against the subscription-level maximum.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` with a Pyth mock.
2. Create a subscription with two price IDs: `feedA` and `feedB`.
3. Call `updatePriceFeeds` with `feedA.publishTime = block.timestamp` and `feedB.publishTime = block.timestamp - 3 days`. Both share the same Pythnet slot (passes slot check). `priceLastUpdatedAt` is set to `block.timestamp`.
4. Call `getPricesNoOlderThan(subscriptionId, [feedA, feedB], 3600)`.
5. Observe: no revert. `feedB` is returned with a `publishTime` 3 days old, violating the 1-hour freshness requirement. [6](#0-5)

    modifier onlyWhitelistedReader(uint256 subscriptionId) {
        // Manager is always allowed
        if (_state.subscriptionManager[subscriptionId] == msg.sender) {
            _;
            return;
        }

        // If whitelist is not used, allow any reader
        if (!_state.subscriptionParams[subscriptionId].whitelistEnabled) {
            _;
            return;
        }

        // Check if caller is in whitelist
        address[] storage whitelist = _state
            .subscriptionParams[subscriptionId]
            .readerWhitelist;
        bool isWhitelisted = false;
        for (uint i = 0; i < whitelist.length; i++) {
            if (whitelist[i] == msg.sender) {
                isWhitelisted = true;
                break;
            }
        }

        if (!isWhitelisted) {
            revert SchedulerErrors.Unauthorized();
        }
        _;
    }
