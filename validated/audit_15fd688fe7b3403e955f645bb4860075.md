### Title
Failed Callbacks in `Echo#executeCallback` Permanently Forfeit User Fees With No Retry тАФ (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

In `Echo.sol`'s `executeCallback`, the provider's fee is credited and the request is cleared **before** the user callback is attempted. When the callback fails, the failure is caught and an event is emitted, but the user's fee is permanently transferred to the provider with no refund and no retry path. This is a direct structural analog to the VUSD failed-withdrawal loss pattern.

### Finding Description

In `Echo#executeCallback`, the execution order is:

1. Provider fees are credited (line 161тАУ162)
2. The request is cleared from storage (line 164)
3. The callback is attempted inside a `try/catch` (lines 176тАУ201)
4. On failure, only a `PriceUpdateCallbackFailed` event is emitted тАФ no refund, no re-queue

```solidity
// Echo.sol lines 161тАУ164
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);

clearRequest(sequenceNumber);

// Echo.sol lines 176тАУ201
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(
        sequenceNumber, providerToCredit, priceIds, req.requester, reason
    );
} catch {
    emit PriceUpdateCallbackFailed(
        sequenceNumber, providerToCredit, priceIds, req.requester,
        "low-level error (possibly out of gas)"
    );
}
```

Because `clearRequest` is called before the callback, `findActiveRequest` will revert for that sequence number afterward тАФ there is no retry path. The user's fee (`req.fee`, stored at request time as `msg.value - pythFeeInWei`) is already credited to the provider and cannot be recovered.

The code itself acknowledges the risk in a TODO comment at lines 155тАУ158:

```solidity
// TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
// If executeCallback can revert, then funds can be permanently locked in the contract.
```

However, the comment only addresses the outer-function revert case. The callback-failure case (caught by `try/catch`) is not addressed, and in that path the user's fee is silently forfeited.

### Impact Explanation

A user who calls `requestPriceUpdatesWithCallback` and pays the required fee will permanently lose that fee if their `_echoCallback` implementation reverts for any reason (logic error, out-of-gas, missing `receive()`, etc.). The request is cleared and cannot be retried. The fee is credited to the provider even though the service was not delivered. There is no refund function and no re-queue mechanism anywhere in the contract.

### Likelihood Explanation

Callback failures are a normal operational occurrence: gas limit misconfiguration, consumer contract bugs, or a consumer contract upgrade that introduces a revert are all realistic. The `callbackGasLimit` is set by the user at request time and is not validated against actual execution cost. Any mismatch causes a silent, permanent fee loss. The entry path requires no privilege тАФ any unprivileged user calling `requestPriceUpdatesWithCallback` is exposed.

### Recommendation

Move `clearRequest` and fee crediting **after** a successful callback, or implement one of the following:

- **Retry mechanism**: On callback failure, keep the request active (do not call `clearRequest`) so the callback can be re-executed, mirroring the `CALLBACK_FAILED` state machine in `Entropy.sol`.
- **Refund on failure**: In the `catch` blocks, refund `req.fee` to `req.requester` instead of crediting it to the provider.
- **Separate fee settlement**: Credit the provider only for the gas/Pyth cost actually consumed, and refund the remainder to the user on failure.

### Proof of Concept

1. User deploys a consumer contract whose `_echoCallback` always reverts (or runs out of gas at the configured `callbackGasLimit`).
2. User calls `requestPriceUpdatesWithCallback` with `callbackGasLimit` and pays the required fee. Fee is stored as `req.fee = msg.value - pythFeeInWei`.
3. Provider (or anyone) calls `executeCallback` with valid `updateData`.
4. Inside `executeCallback`:
   - `_state.providers[providerToCredit].accruedFeesInWei` is incremented by `req.fee + msg.value - pythFee`. [1](#0-0) 
   - `clearRequest(sequenceNumber)` removes the request from storage. [2](#0-1) 
   - `_echoCallback` reverts; the `catch` block emits `PriceUpdateCallbackFailed` and returns. [3](#0-2) 
5. The user's fee is now in the provider's `accruedFeesInWei`. The request no longer exists. There is no function in `Echo.sol` to refund the user or re-queue the request. The user's ETH is permanently lost to them. [4](#0-3) [5](#0-4)
