### Title
Wrong Zero-Check Condition in Deviation Calculation Silently Skips 100% Price Crashes — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function checks `currentPrice == 0 || previousPrice == 0` before computing deviation. Only `previousPrice == 0` is needed to prevent division by zero (it is the denominator). The extra `currentPrice == 0` guard causes the deviation check to be silently skipped when a price crashes to zero, which is precisely the scenario where an update is most critical.

### Finding Description

The deviation calculation in `_validateShouldUpdatePrices` is:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}

uint256 numerator = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
```

The denominator is `SignedMath.abs(previousPrice)`. Division by zero can only occur when `previousPrice == 0`. The condition `currentPrice == 0` is not needed for arithmetic safety — it is the wrong variable to guard against.

When `currentPrice == 0` and `previousPrice != 0`, the true deviation is:

```
|0 - previousPrice| / |previousPrice| * 10_000 = 10_000 bps (100%)
```

This would always exceed any configured `deviationThresholdBps` and should unconditionally trigger an update. Instead, the `continue` statement skips this feed entirely. If all feeds in the batch have `currentPrice == 0`, the loop exhausts without returning, and the function reverts with `UpdateConditionsNotMet`. [1](#0-0) 

### Impact Explanation

A subscription configured with `updateOnDeviation = true` and `updateOnHeartbeat = false` will be unable to receive a price update when the reported price is zero. The keeper's `updatePriceFeeds` call reverts with `UpdateConditionsNotMet`, leaving stale (non-zero) prices stored on-chain. Any downstream consumer reading prices from the Scheduler will see an incorrect, stale price rather than the true zero value. This is a direct price-feed integrity failure for the Scheduler product. [2](#0-1) 

### Likelihood Explanation

A keeper (unprivileged relayer) is the entry point — no privileged access is required. The scenario is realistic: Pyth price feeds can legitimately publish a price of `0` for halted markets, delisted assets, or edge-case aggregation results. Any subscription that relies solely on deviation-based updates and whose tracked asset reports a zero price will be permanently blocked from updating until the price becomes non-zero again. [3](#0-2) 

### Recommendation

Remove `currentPrice == 0` from the skip condition. Only the denominator (`previousPrice`) needs to be checked:

```solidity
// Skip only if previousPrice is zero to avoid division by zero
if (previousPrice == 0) {
    continue;
}
```

When `currentPrice == 0` and `previousPrice != 0`, the deviation is 100% and should trigger an update normally. [4](#0-3) 

### Proof of Concept

1. Create a subscription with `updateOnDeviation = true`, `updateOnHeartbeat = false`, `deviationThresholdBps = 100` (1%).
2. Submit a first `updatePriceFeeds` call with `price = 1000` for all feeds. This succeeds and stores the baseline.
3. Submit a second `updatePriceFeeds` call with `price = 0` for all feeds (a valid Pyth-attested zero price).
4. `_validateShouldUpdatePrices` enters the deviation loop. For each feed: `previousPrice = 1000`, `currentPrice = 0`. The condition `currentPrice == 0` is true, so `continue` is executed.
5. The loop completes without returning. The function hits `revert SchedulerErrors.UpdateConditionsNotMet()`.
6. The keeper's transaction reverts. The on-chain price remains `1000` even though the true price is `0`. [5](#0-4)
