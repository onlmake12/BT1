The code confirms this is a real vulnerability. Here is the full analysis:

### Title
Fee Coins Permanently Frozen in `PriceInfoObject` on Stale Price Update — (`target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

`update_single_price_feed` unconditionally deposits the caller's fee into `PriceInfoObject` **before** the freshness check inside `update_cache`. When the submitted price timestamp is not newer than the cached one, the fee is consumed but the price state is unchanged. Because no withdrawal function exists anywhere in the contract, those coins are permanently frozen.

---

### Finding Description

**Step 1 — Fee deposited unconditionally**

In `pyth.move`, `update_single_price_feed` calls `price_info::deposit_fee_coins` at line 277, before any freshness evaluation: [1](#0-0) 

```
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
// store fee coins within price info object
price_info::deposit_fee_coins(price_info_object, fee);   // ← fee gone here
```

**Step 2 — Freshness check is inside `update_cache`, called after deposit**

`update_cache` only writes the new price when `is_fresh_update` returns `true`: [2](#0-1) 

`is_fresh_update` requires `update_timestamp > cached_timestamp` (strict): [3](#0-2) 

When the condition is false the function returns silently — no price update, no fee refund.

**Step 3 — No withdrawal path exists**

`price_info.move` exposes `deposit_fee_coins` (public) and `get_balance` (public), but **no withdraw function**: [4](#0-3) 

The `fee_recipient_address` field in `State` is stored but never used to route fees anywhere: [5](#0-4) 

The `set_fee_recipient` module itself documents that the fee-recipient mechanism is **deprecated and no longer used**: [6](#0-5) 

The governance dispatcher has no "withdraw fees" action: [7](#0-6) 

---

### Impact Explanation

Every SUI coin deposited via `deposit_fee_coins` into a `PriceInfoObject` is permanently locked inside that object's dynamic field (`FEE_STORAGE_KEY`). There is no function — privileged or otherwise — that can move those coins out. Fees paid for stale updates are therefore irrecoverably frozen.

---

### Likelihood Explanation

This is reachable by any unprivileged caller in normal operation:

- **Race condition**: two users prepare the same valid accumulator update concurrently; the second transaction arrives after the first has already advanced the cached timestamp. The second caller pays the fee and receives nothing.
- **Deliberate replay**: a caller submits the same `HotPotatoVector` twice (two separate transactions with the same accumulator message and a freshly verified VAA each time). The second call deposits the fee but `is_fresh_update` returns false.

No privileged role, leaked key, or governance majority is required.

---

### Recommendation

1. **Move the fee deposit after the freshness check**, or refund the fee if `is_fresh_update` returns false.
2. **Implement a fee-withdrawal path** (e.g., a governance action that sweeps accumulated fees from `PriceInfoObject` to the `fee_recipient_address`), or route fees directly to the recipient at deposit time instead of accumulating them inside the price object.

---

### Proof of Concept

```
// 1. Create price feed with timestamp T.
// 2. Call update_single_price_feed with update at timestamp T (same, not newer).
//    → deposit_fee_coins runs, fee is gone.
//    → update_cache runs, is_fresh_update returns false (T > T is false).
//    → price_info_object timestamp unchanged.
// 3. Assert price_info_object.price_info.timestamp == T  (unchanged).
// 4. Assert price_info::get_balance(price_info_object) == base_update_fee (fee locked).
// 5. Attempt to call any withdraw function → compile error: no such function exists.
```

The existing test at `pyth.move` line 817 already asserts that `get_balance` equals `DEFAULT_BASE_UPDATE_FEE` after one update call, confirming fees accumulate inside the object with no egress path. [8](#0-7)

    fun is_fresh_update(update: &PriceInfo, price_info_object: &PriceInfoObject): bool {
        // Get the timestamp of the update's current price
        let price_feed = price_info::get_price_feed(update);
        let update_timestamp = price::get_timestamp(&price_feed::get_price(price_feed));

        // Get the timestamp of the cached data for the price identifier
        let cached_price_info = price_info::get_price_info_from_price_info_object(price_info_object);
        let cached_price_feed =  price_info::get_price_feed(&cached_price_info);
        let cached_timestamp = price::get_timestamp(&price_feed::get_price(cached_price_feed));

        update_timestamp > cached_timestamp
    }

    public fun get_balance(price_info_object: &PriceInfoObject): u64 {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);
        coin::value(fee)
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
