Audit Report

## Title
Integer Overflow in `GetLastStateProofProcess::execute` Bypasses `GET_LAST_STATE_PROOF_LIMIT`, Enabling Remote DoS — (File: `util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The guard at line 201 of `get_last_state_proof.rs` is intended to cap total work per `GetLastStateProof` request to `GET_LAST_STATE_PROOF_LIMIT = 1000`. In Rust release builds, the expression `(last_n_blocks as usize) * 2` wraps to `0` when `last_n_blocks = u64::MAX/2 + 1`, causing the guard to evaluate `false` and be bypassed entirely. The server then collects all block numbers from `start_block_number` to `last_block_number` into a `Vec` and performs one DB read and one MMR root computation per entry — O(N) work per request from a single unauthenticated peer.

## Finding Description
**Root cause — overflow at the guard (lines 199–205):**

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `last_n_blocks = u64::MAX/2 + 1 = 9_223_372_036_854_775_808`:
- `last_n_blocks as usize = 9_223_372_036_854_775_808` (fits in 64-bit `usize`)
- `9_223_372_036_854_775_808usize * 2 = 2^64` → wraps to `0` in release mode (Rust wraps on overflow without `overflow-checks = true`)
- `0 + 0 > 1000` → `false` → guard bypassed

**Unbounded allocation after bypass (lines 291–297):**

```rust
let (sampled_numbers, last_n_numbers) = if last_block_number - start_block_number
    <= last_n_blocks
{
    let sampled_numbers = Vec::new();
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
    (sampled_numbers, last_n_numbers)
```

With `last_n_blocks = u64::MAX/2 + 1` and `start_block_number = 0`, the condition `last_block_number <= u64::MAX/2 + 1` is trivially true for any realistic chain height. The server collects every block number from `0` to `last_block_number` into a `Vec<u64>`.

**O(N) work in `complete_headers` (lines 132–163):**

For each of the N collected block numbers, `complete_headers` calls:
- `snapshot.get_ancestor(last_hash, *number)` — DB read
- `snapshot.get_block(&ancestor_header.hash())` — DB read
- `snapshot.chain_root_mmr(*number - 1).get_root()` — MMR computation

**Prerequisite checks do not block the attack:**
- `last_hash` must be a valid main-chain block hash (line 210) — publicly observable
- `difficulties = []` passes all difficulty validation checks (lines 254–288) since they all short-circuit on an empty slice
- `start_block_number = 0` avoids the `reorg_last_n_numbers` branch (line 237)
- `GET_LAST_STATE_PROOF_LIMIT = 1000` (constant.rs line 6) is the only intended bound and is bypassed

## Impact Explanation
A single unauthenticated P2P peer can force the full node to allocate a `Vec<u64>` of up to N entries (e.g., 10M blocks × 8 bytes = 80 MB) and perform N sequential DB reads and N MMR root computations per request. Repeated requests from one or more peers can exhaust memory and saturate I/O, crashing or severely degrading the node. This matches the allowed impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**.

## Likelihood Explanation
The attack requires only:
1. Knowledge of any valid main-chain block hash (publicly observable from any block explorer or peer)
2. Crafting a single P2P `GetLastStateProof` message with `last_n_blocks = u64::MAX/2 + 1`, `difficulties = []`, `start_number = 0`

No proof-of-work, no keys, no privileged access. The overflow is deterministic in all release builds on 64-bit systems. The light client protocol server is a production feature. The attack is trivially repeatable.

## Recommendation
Replace the plain multiplication with a saturating variant:

```rust
// Before (vulnerable):
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT

// After (safe):
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
```

`saturating_mul` returns `usize::MAX` on overflow, which is always `> 1000`, so the guard correctly rejects the malicious request. Alternatively, reject any `last_n_blocks > GET_LAST_STATE_PROOF_LIMIT / 2` before the cast.

## Proof of Concept
```rust
// Attacker constructs:
let msg = GetLastStateProof::new_builder()
    .last_hash(any_valid_main_chain_block_hash)   // publicly observable
    .start_hash(Byte32::default())
    .start_number(0u64.pack())
    .last_n_blocks((u64::MAX / 2 + 1).pack())     // triggers overflow
    .difficulty_boundary(U256::one().pack())
    .difficulties(Uint256Vec::default())           // empty, passes all checks
    .build();

// On the server (release mode, 64-bit):
// (u64::MAX/2 + 1) as usize = 9_223_372_036_854_775_808
// 9_223_372_036_854_775_808usize * 2 = 0  (wraps)
// 0 + 0 > 1000 → false → guard bypassed
// last_n_numbers = (0..N).collect() → N allocations + N DB reads + N MMR ops
```

A unit test can confirm the overflow: `assert_eq!((9_223_372_036_854_775_808usize).wrapping_mul(2), 0)`. A fuzz test targeting `execute` with `last_n_blocks` near `u64::MAX/2` will reproduce the unbounded allocation path.
