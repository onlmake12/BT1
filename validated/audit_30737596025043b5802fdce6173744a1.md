Audit Report

## Title
Fee Rate Minimum Check Uses Transaction Size Instead of Weight, Bypassing Cycle-Weighted Admission Control - (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using raw serialized byte size rather than the canonical transaction weight (`max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because this check runs before script execution — when cycles are unknown — and no weight-based fee check is performed after execution when cycles are known, an attacker can submit transactions with maximum cycles and minimal byte size that pass the size-based gate at a fee far below what the weight-based `min_fee_rate` would require. This enables low-cost flooding of the tx-pool with CPU-expensive transactions, exhausting node verification resources.

## Finding Description

`get_transaction_weight` defines the canonical weight used for fee rate calculations: [1](#0-0) 

`check_tx_fee` is the sole admission-time fee gate, but it uses raw `tx_size` instead of weight. The code comment explicitly acknowledges the theoretical incorrectness of this approach: [2](#0-1) 

The call sequence in `_process_tx` confirms the ordering problem: `pre_check` (which calls `check_tx_fee`) runs first at line 715, then `verify_rtx` (script execution, which yields actual cycles) runs at line 724, and then `TxEntry::new` is constructed at line 751 — with no second weight-based fee check anywhere in between: [3](#0-2) 

`pre_check` calls `check_tx_fee` at line 289 before cycles are known: [4](#0-3) 

`TxEntry::fee_rate()` correctly uses weight for sorting and eviction, but this is not a rejection gate — it only affects ordering after admission: [5](#0-4) 

The pool size limit is enforced against raw byte size (`total_tx_size`), not weight, meaning cycle-heavy but byte-small transactions consume minimal pool space: [6](#0-5) [7](#0-6) 

The same unit mismatch exists in the RBF path — `calculate_min_replace_fee` uses raw `size` instead of weight: [8](#0-7) 

## Impact Explanation

With `min_fee_rate = 1,000` shannons/KW (default) and a transaction of 200 bytes consuming 70,000,000 cycles (the default `max_tx_verify_cycles`):

- Actual weight: `max(200, 70,000,000 × 0.000170571)` ≈ **11,940**
- `min_fee` via size check: `1,000 × 200 / 1,000` = **200 shannons** (passes gate)
- `min_fee` via weight: `1,000 × 11,940 / 1,000` = **11,940 shannons** (correct threshold)

The effective fee rate of such a transaction is `200 × 1,000 / 11,940 ≈ 16 shannons/KW` — roughly **60× below** the configured threshold. Because the pool size limit is measured in raw bytes (`total_tx_size`), a 200-byte transaction consumes only 200 bytes of pool space. An attacker can pack ~100,000 such transactions into a 20 MB pool, each requiring 70M cycles of script execution to verify. The primary impact is **node resource exhaustion and network congestion at negligible cost**, matching the allowed High impact: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

## Likelihood Explanation

The attack requires only an unprivileged `send_transaction` RPC call. The attacker controls both transaction byte size (keep minimal: single input/output, compact witness) and script complexity (maximize cycles up to `max_tx_verify_cycles`, default 70,000,000). No special role, key, or majority hashpower is needed. The discrepancy between size-based gate and weight-based reality is largest precisely at the maximum-cycles boundary, which is trivially targetable. The attacker needs valid UTXOs to spend, but these can be prepared in advance with small CKB amounts. The attack is repeatable as long as the attacker has UTXOs.

## Recommendation

After script execution, when cycles are known, perform a second fee rate check using the actual weight before inserting the entry. In `_process_tx` in `tx-pool/src/process.rs`, after `verify_rtx` returns `verified.cycles`:

```rust
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

Apply the same correction to `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, substituting `get_transaction_weight(size, entry.cycles)` for the raw `size` argument passed to `min_rbf_rate.fee(...)`.

## Proof of Concept

1. Prepare a CKB UTXO with sufficient capacity.
2. Craft a transaction whose lock/type script runs a tight loop consuming ≈70,000,000 cycles, with serialized size ≈200 bytes (single input/output, compact witness).
3. Set the fee to 201 shannons (just above the size-based threshold of 200 shannons).
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` computes `min_fee = 1,000 × 200 / 1,000 = 200 shannons`; the transaction passes the gate.
6. `verify_rtx` executes the script, consuming 70,000,000 cycles; the entry is inserted with weight ≈11,940.
7. The effective fee rate stored in the pool is `201 × 1,000 / 11,940 ≈ 16 shannons/KW`, far below the 1,000 shannons/KW threshold.
8. Repeat with many pre-prepared UTXOs to fill the pool with cycle-expensive, fee-cheap transactions, exhausting node CPU via repeated script execution and crowding out legitimate higher-fee-rate transactions.
