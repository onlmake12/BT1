### Title
Pre-allocation of Up to 8 MB Per Connection via Attacker-Controlled Snappy Header Claim with Off-by-One Guard Bypass — (`network/src/compress.rs`)

### Summary

`LengthDelimitedCodecWithCompress::decode` allocates a heap buffer sized by the attacker-controlled `decompress_len` value **before** any decompression occurs. The guard uses strict `>` instead of `>=`, allowing a frame claiming exactly `MAX_UNCOMPRESSED_LEN` (8,388,608 bytes) to pass and trigger a full 8 MB allocation. With the default connection limit of 125 peers, an attacker can cause ~1 GB of simultaneous heap allocation with near-zero bandwidth.

### Finding Description

In `LengthDelimitedCodecWithCompress::decode`:

```
if decompressed_bytes_len > MAX_UNCOMPRESSED_LEN {   // line 235: strict >
    return Err(...);
}
let mut buf = BytesMut::zeroed(decompressed_bytes_len);  // line 242: allocates BEFORE decompress
match SnapDecoder::new().decompress(&data[1..], &mut buf) {
``` [1](#0-0) 

`MAX_UNCOMPRESSED_LEN = 1 << 23 = 8,388,608`. [2](#0-1) 

The `snap::raw::decompress_len` function reads the **varint-encoded uncompressed length from the snappy stream header** without performing any decompression. An attacker crafts a snappy stream whose header varint claims exactly 8,388,608 bytes, backed by a tiny (< 20 byte) actual payload. The guard evaluates `8388608 > 8388608 = false`, so it passes. `BytesMut::zeroed(8388608)` allocates 8 MB on the heap. `SnapDecoder::decompress` then fails (input exhausted before output buffer is filled), the error propagates, and the connection is dropped — but the 8 MB was already live on the heap during that window. [3](#0-2) 

### Impact Explanation

The default configuration sets `max_peers = 125` and `max_outbound_peers = 8`, leaving up to **117 inbound slots**. [4](#0-3) 

Each inbound connection is handled independently. If 117 connections simultaneously deliver a malicious frame, the node allocates `117 × 8 MB ≈ 936 MB` of heap simultaneously before any error propagates. On nodes with ≤ 1 GB RAM (common for validators and light nodes), this causes OOM / process crash.

The `max_frame_length` per protocol (e.g., 4 MB for RelayV3, 2 MB for Sync) limits the **compressed** frame size, not the claimed decompressed size. [5](#0-4) 

The attacker's compressed frame is ~20 bytes total (4-byte varint header + a few snappy literal bytes), well within every protocol's frame limit.

### Likelihood Explanation

- The P2P port is open to the internet by default.
- No authentication is required to open a TCP connection and send a compressed frame.
- The attack requires only ~117 concurrent TCP connections and ~2 KB of total bandwidth.
- The `PeerRegistry` eviction logic does not prevent 117 simultaneous inbound connections from distinct IPs. [6](#0-5) 

### Recommendation

1. **Fix the off-by-one**: Change `>` to `>=` on line 235 so that a frame claiming exactly `MAX_UNCOMPRESSED_LEN` is rejected.
2. **Fix the root cause**: Do not pre-allocate based on the attacker-supplied header value. Either use a streaming decompressor, or allocate a fixed-size scratch buffer and decompress into it, rejecting output that exceeds the limit incrementally.
3. **Consider a per-connection or global decompression memory budget** enforced before allocation.

### Proof of Concept

```
1. Craft a snappy stream: varint(8388608) || <minimal valid snappy literal block>
   Total size: ~20 bytes.
2. Wrap in a length-delimited frame with COMPRESS_FLAG set (byte 0 = 0x80).
3. Open 117 TCP connections to the CKB P2P port.
4. On each connection, complete the tentacle handshake and send the malicious frame
   on any compress-enabled protocol (e.g., Sync or RelayV3).
5. Observe: each connection triggers BytesMut::zeroed(8388608) before decompress fails.
6. Peak RSS spike ≈ 117 × 8 MB = 936 MB before any error propagates.
7. On a node with ≤ 1 GB RAM, the allocator OOMs and the process crashes.
``` [7](#0-6) [8](#0-7)
