### Title
Empty-address `Nodes(announce=false)` message sets `received_nodes=true` without delivering any addresses, permanently blocking legitimate peer discovery for that session — (`network/src/protocols/discovery/mod.rs`)

### Summary
`verify_nodes_message` only rejects items whose `addresses.len() > MAX_ADDRS`. Items with zero addresses pass. A single `Nodes{announce:false, items:[Node{addresses:[], ...}; 1000]}` message passes all validation, sets `received_nodes = true`, and adds zero addresses to the peer store. Any subsequent legitimate `Nodes{announce:false}` response (e.g., the real reply to `GetNodes`) is rejected as `DuplicateFirstNodes` and the session is disconnected.

### Finding Description
In `received()` → `DiscoveryMessage::Nodes` branch:

1. `verify_nodes_message` checks `item.addresses.len() > MAX_ADDRS` (strictly greater), so 0-address items pass. [4](#0-3) 

2. The `addrs` vec is built by `flat_map` over `node.addresses` — empty for 0-address items, yielding an empty `Vec`. [5](#0-4) 

3. `state.received_nodes = true` is set with no guard on `addrs.is_empty()`. [6](#0-5) 

4. `add_new_addrs` is called with an empty vec and returns immediately. [7](#0-6) 

5. The next `Nodes{announce:false}` hits the duplicate guard and disconnects. [3](#0-2) 

### Impact Explanation
For any session where the victim node sent `GetNodes` (outbound connections), the attacker can respond with this crafted message before the legitimate response arrives. The victim's `received_nodes` flag is poisoned: it believes it already received the initial peer list, but received zero addresses. The session's peer discovery slot is permanently consumed with no benefit. If the attacker is the victim's only or first bootstrap peer, the victim cannot populate its peer store from that session.

### Likelihood Explanation
Any peer that can establish a P2P connection to the victim can send this message. No authentication, no PoW, no special role required. The attacker just needs to respond to the victim's `GetNodes` before the legitimate response is processed — trivially achievable if the attacker controls the responding peer.

### Recommendation
In `verify_nodes_message`, add a check that non-announce Nodes messages must contain at least one item with at least one address:
```rust
// For announce=false, require at least one valid address
if !nodes.announce {
    let total_addrs: usize = nodes.items.iter().map(|n| n.addresses.len()).sum();
    if total_addrs == 0 {
        return Some(Misbehavior::InvalidData);
    }
}
```
Alternatively, guard `state.received_nodes = true` behind `!addrs.is_empty()`. [8](#0-7)

    let mut misbehavior = None;
    if nodes.announce {
        if nodes.items.len() > ANNOUNCE_THRESHOLD {
            warn!(
                "Number of nodes exceeds announce threshold {}",
                ANNOUNCE_THRESHOLD
            );
            misbehavior = Some(Misbehavior::TooManyItems {
                announce: nodes.announce,
                length: nodes.items.len(),
            });
        }
    } else if nodes.items.len() > MAX_ADDR_TO_SEND {
        warn!(

            nodes.items.len()

            announce: nodes.announce,
