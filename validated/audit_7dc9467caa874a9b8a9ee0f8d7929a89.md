### Title
Unbounded SQL OFFSET via Attacker-Controlled Pagination Cursor Enables Connection-Pool Exhaustion DoS — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

### Summary
The `get_transactions` RPC handler (with `group_by_transaction=false`) decodes the `after` cursor's last 4 bytes as an `i32` offset and passes it directly to the SQL query builder with no upper-bound validation. An unauthenticated caller can supply `i32::MAX` (0x7fffffff) as the offset, forcing the database to scan every matching row before returning an empty result, holding a connection for up to the 60-second `acquire_timeout`. With a pool capped at 10 connections, 10 concurrent crafted requests exhaust the pool.

### Finding Description

**Step 1 — Cursor parsing, no range check.** [1](#0-0) 

The only guard is a length check (`after.len() != 12`). `decode_i32` simply calls `i32::from_le_bytes` with no range validation: [2](#0-1) 

Any 4-byte value — including `[0xff, 0xff, 0xff, 0x7f]` = `i32::MAX` = 2,147,483,647 — is accepted.

**Step 2 — Offset injected directly into SQL.** [3](#0-2) 

The decoded offset is forwarded to `query_builder.offset(offset)` without any cap, producing:
```sql
SELECT ... FROM (... UNION ALL ...) AS res
JOIN ckb_transaction ... JOIN block ...
WHERE tx_id >= <last>
ORDER BY tx_id ASC
LIMIT <limit>
OFFSET 2147483647
```

**Step 3 — Connection held for full acquire_timeout.** [4](#0-3) 

The pool is configured with `acquire_timeout(60s)` and `max_connections(10)`. Both SQLite and PostgreSQL implement `OFFSET` by scanning and discarding rows sequentially. The scan is bounded by actual data (not literally 2 billion rows), but on a synced mainnet node with tens or hundreds of millions of cells, the query will scan all rows matching `tx_id >= last` before returning nothing, holding the connection for the full timeout.

### Impact Explanation
Ten concurrent crafted RPC calls exhaust the 10-connection pool. Subsequent legitimate RPC calls (including `get_cells`, `get_transactions`, and `get_cells_capacity`) block waiting for a connection and time out after 60 seconds. This is a low-cost, unauthenticated, repeatable denial-of-service against the rich-indexer subsystem.

### Likelihood Explanation
The RPC endpoint is publicly accessible by default. No authentication, no PoW, no privileged role is required. The attacker only needs to know the 12-byte cursor format (documented in the RPC README) and send a single JSON-RPC call. The attack is trivially scriptable.

### Recommendation
Add an upper-bound check on the decoded offset immediately after `decode_i32`. A reasonable maximum is the configured `request_limit` (which already caps `limit`). For example:

```rust
let offset = decode_i32(offset)?;
if offset < 0 || offset as usize > self.request_limit {
    return Err(Error::Params("Invalid 'after' cursor offset.".to_string()));
}
```

Alternatively, validate that the cursor was produced by a prior legitimate response by comparing the offset against the actual cell count for the referenced `tx_id`.
