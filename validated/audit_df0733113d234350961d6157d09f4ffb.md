### Title
Excess `msg.value` Permanently Locked in `Pyth.sol::updatePriceFeeds()` Due to Missing Refund - (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`Pyth.sol::updatePriceFeeds()` is a `payable` function that only enforces a minimum fee (`msg.value >= requiredFee`) but never refunds the excess. Any ETH sent above the required fee is permanently locked in the contract, recoverable only via a privileged governance `WithdrawFee` action. This is the direct analog of the NestedFactory M-03 pattern: a payable entry point silently absorbs native ETH that the protocol has no legitimate claim to.

---

### Finding Description

`updatePriceFeeds()` computes `requiredFee` from the number of price updates and checks only that the caller paid at least that amount:

```solidity
// Pyth.sol lines 64-79
function updatePriceFeeds(
    bytes[] calldata updateData
) public payable override {
    uint totalNumUpdates = 0;
    for (uint i = 0; i < updateData.length; ) {
        totalNumUpdates += updatePriceInfosFromAccumulatorUpdate(updateData[i]);
        unchecked { i++; }
    }
    uint requiredFee = getTotalFee(totalNumUpdates);
    if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
    // ← no refund of (msg.value - requiredFee)
}
``` [1](#0-0) 

The excess `msg.value - requiredFee` is never credited to any accounting variable and is never returned to the caller. It accumulates silently in the contract's raw ETH balance. The only withdrawal path is the privileged governance `WithdrawFee` instruction, which sends funds to a governance-chosen recipient, not the original overpayer.

Contrast this with `PythLazer.sol::verifyUpdate()`, which correctly refunds the excess:

```solidity
// PythLazer.sol lines 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [2](#0-1) 

`Entropy.sol::requestHelper()` has a related but distinct issue: it absorbs the entire `msg.value` into `accruedPythFeesInWei` (line 238-239), so excess is silently donated to the Pyth treasury rather than refunded. The interface documentation explicitly warns "excess value is *not* refunded to the caller," but this is a design choice that still results in user fund loss. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

A caller who sends `msg.value` greater than `getTotalFee(totalNumUpdates)` permanently loses the excess ETH. The contract has no user-accessible refund path. The only recovery mechanism is a governance `WithdrawFee` VAA, which sends funds to an arbitrary governance-chosen address — not the original sender. From the user's perspective the ETH is irrecoverably lost.

Realistic loss scenarios:
- A user sends a round number (e.g., `0.01 ETH`) when the actual fee is `1 wei`.
- A fee decrease occurs between the time the user calls `getUpdateFee()` and the time their transaction is mined; the user paid the old (higher) fee but only the new (lower) fee is consumed.
- A front-end or SDK hardcodes a fee estimate that is higher than the on-chain value.

---

### Likelihood Explanation

The `singleUpdateFeeInWei` is a small value (typically 1 wei on most chains), making it easy for users to accidentally overpay by orders of magnitude. Fee changes via governance are infrequent but do occur. Front-end integrations commonly add a buffer to fee estimates. The function is the primary entry point for all price-feed consumers, so call volume is high.

---

### Recommendation

Add a refund of excess ETH at the end of `updatePriceFeeds()`, mirroring the pattern already used in `PythLazer.sol::verifyUpdate()`:

```solidity
function updatePriceFeeds(bytes[] calldata updateData) public payable override {
    uint totalNumUpdates = 0;
    for (uint i = 0; i < updateData.length; ) {
        totalNumUpdates += updatePriceInfosFromAccumulatorUpdate(updateData[i]);
        unchecked { i++; }
    }
    uint requiredFee = getTotalFee(totalNumUpdates);
    if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
    // Refund excess
    if (msg.value > requiredFee) {
        (bool ok, ) = payable(msg.sender).call{value: msg.value - requiredFee}("");
        require(ok, "Pyth: ETH refund failed");
    }
}
```

Apply the same fix to `Entropy.sol::requestHelper()`: refund `msg.value - requiredFee` instead of silently crediting it to `accruedPythFeesInWei`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache-2.0
pragma solidity ^0.8.0;

import "forge-std/Test.sol";
import "../contracts/pyth/Pyth.sol";

contract ExcessEthLockedPoC is Test {
    Pyth pyth;

    function setUp() public {
        // deploy Pyth with singleUpdateFeeInWei = 1 wei
        // (standard mainnet configuration)
        pyth = deployPythWithFee(1);
    }

    function testExcessEthLocked() public {
        address user = makeAddr("user");
        vm.deal(user, 1 ether);

        bytes[] memory updateData = buildValidUpdateData(1); // 1 price feed
        // required fee = 1 wei, but user sends 1 ether
        vm.prank(user);
        pyth.updatePriceFeeds{value: 1 ether}(updateData);

        // user lost 1 ether - 1 wei permanently
        assertEq(user.balance, 0);
        // contract holds the excess with no user-accessible refund path
        assertEq(address(pyth).balance, 1 ether);
    }
}
