### Title
Hardcoded `DEFAULT_BYTES_PER_CYCLES` Ratio Diverges from Actual Consensus Parameters, Causing Incorrect Transaction Weight and Fee-Rate Calculations — (File: `util/types/src/core/tx_pool.rs`)

### Summary

`get_transaction_weight` uses a compile-time float constant `DEFAULT_BYTES_PER_CYCLES` to convert VM cycles into a byte-equivalent weight. This constant is hardcoded to the ratio `MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES` for the *default* chain spec, but both `max_block_bytes` and `max_block_cycles` are runtime-configurable consensus parameters. On any chain whose spec overrides either limit, the weight function silently produces wrong values, causing incorrect fee-rate admission decisions and incorrect fee estimates returned to RPC callers.

### Finding Description

`DEFAULT_BYTES_PER_CYCLES` is defined as a compile-time `f64` constant:

```
/// Equal to MAX_BLOCK_BYTES / MAX_BLOCK_CYCLES, see ckb-chain-spec.
pub const DEFAULT_BYTES_PER_CYCLES: f64 = 0.000_170_571_4_f64;
``` [1](#0-0) 

`get_transaction_weight` — the single function used everywhere to convert `(tx_size, cycles)` into a unified weight — unconditionally multiplies cycles by this constant:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

The constant is derived from the *default* spec values:

```rust
pub const MAX_BLOCK_BYTES: u64 = TWO_IN_TWO_OUT_BYTES * TWO_IN_TWO_OUT_COUNT;  // 597_000
pub(crate) const MAX_BLOCK_CYCLES: u64 = TWO_IN_TWO_OUT_CYCLES * TWO_IN_TWO_OUT_COUNT; // 3_500_000_000
``` [3](#0-2) 

However, both values are runtime-configurable fields of `Consensus`:

```rust
pub max_block_cycles: Cycle,
pub max_block_bytes: u64,
``` [4](#0-3) 

and can be overridden via `ConsensusBuilder`:

```rust
pub fn max_block_cycles(mut self, max_block_cycles: Cycle) -> Self { ... }
pub fn max_block_bytes(mut self, max_block_bytes: u64) -> Self { ... }
``` [5](#0-4) 

`get_transaction_weight` is called in the tx-pool entry path: [6](#0-5) 

and in the fee estimator, which also hardcodes `MAX_BLOCK_BYTES` as a constant rather than reading from the live consensus:

```rust
let removed_weight = (MAX_BLOCK_BYTES * 85 / 100) * target_blocks;
``` [7](#0-6) 

### Impact Explanation

When a chain spec sets `max_block_cycles` higher than the default (making the true `bytes_per_cycle` ratio larger), `get_transaction_weight` underestimates the weight of cycle-heavy transactions. Consequently:

1. **Tx-pool admission**: A transaction whose true weight exceeds the minimum-fee-rate threshold is admitted because the computed weight is too low, effectively bypassing the fee floor.
2. **Fee estimation**: `FeeRateCollector` and the `weight_units_flow` estimator return fee-rate recommendations that are too low, misleading all RPC callers (`get_fee_rate_statistics`, `estimate_fee_rate`).
3. **Block assembly**: Miners using the weight-based knapsack selector may pack suboptimal blocks.

The inverse holds when `max_block_cycles` is lower than the default: legitimate transactions are incorrectly rejected or over-priced.

### Likelihood Explanation

The dev chain spec (`resource/specs/dev.toml`) and staging spec (`resource/specs/staging.toml`) both expose configurable `max_block_cycles` and `max_block_bytes`. Any operator running a non-mainnet node, or any future mainnet hard fork that adjusts block limits, triggers the mismatch. A tx-pool submitter (unprivileged RPC caller) can exploit the underestimation path on such a node by submitting high-cycle, low-byte transactions with fees below the intended minimum.

### Recommendation

Replace the compile-time constant with a runtime computation derived from the live `Consensus` object. Pass `consensus.max_block_bytes()` and `consensus.max_block_cycles()` into `get_transaction_weight` (or compute the ratio once at node startup and store it alongside the consensus), so the weight function always reflects the actual chain parameters rather than the default approximation.

### Proof of Concept

On a chain configured with `max_block_cycles = 7_000_000_000` (2× default) and `max_block_bytes = 597_000` (unchanged), the true `bytes_per_cycle` ratio is `≈ 0.0000853`. The hardcoded constant is `0.000_170_571_4`, which is 2× too large — meaning cycle-heavy transactions are assigned 2× the correct weight. Conversely, if `max_block_cycles = 1_750_000_000` (½ default), the constant is 2× too small, and a transaction with 1 billion cycles and 1 byte of serialized size would be assigned weight `max(1, 1e9 × 0.000_170_571_4) = 170_571` instead of the correct `max(1, 1e9 / (1_750_000_000 / 597_000)) = 341_142`, halving its apparent fee rate and allowing it to pass a fee-rate check it should fail.
