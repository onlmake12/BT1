### Title
Verify Queue Can Be Saturated With Zero-Fee Transactions Before Fee Check, Enabling P2P Transaction Relay DOS — (`tx-pool/src/process.rs`, `tx-pool/src/component/verify_queue.rs`)

---

### Summary

The `VerifyQueue` is a bounded resource (256 MB) that gates all P2P-relayed transaction processing. The fee-rate check (`check_tx_fee`) is deferred until a worker dequeues and processes a transaction, but admission to the queue only requires passing `non_contextual_verify`, which cannot detect zero-fee or below-minimum-fee-rate transactions because it has no access to chain state. An unprivileged P2P peer can continuously relay structurally valid, zero-fee transactions to saturate the queue, causing all subsequent legitimate P2P transactions to be rejected with `Reject::Full`.

---

### Finding Description

**Two-stage processing split creates a fee-free admission window:**

The P2P relay path calls `submit_remote_tx` → `resumeble_process_tx_and_notify_full_reject` → `resumeble_process_tx` → `enqueue_verify_queue`. The only gate before queuing is `non_contextual_verify`: [1](#0-0) 

`non_contextual_verify` checks only structure-level properties (version, size ≤ 512 KB, non-empty inputs/outputs, duplicate deps, outputs data, script hash type): [2](#0-1) 

It performs **no fee check**. The fee check lives in `check_tx_fee`, which is called only from `pre_check`, which is called only by verify workers after a transaction is already dequeued: [3](#0-2) [4](#0-3) 

`check_tx_fee` requires a resolved transaction (`rtx`) to compute actual fee from input capacities — information that is unavailable without chain state and therefore cannot be checked at the `non_contextual_verify` stage.

**The VerifyQueue is bounded at 256 MB with no per-peer limit:** [5](#0-4) [6](#0-5) 

`add_tx` enforces only the total-size cap and a duplicate check — no fee gate, no per-peer accounting: [7](#0-6) 

When the queue is full, any new transaction (including legitimate ones) is rejected: [8](#0-7) 

---

### Impact Explanation

An attacker who saturates the 256 MB verify queue causes every subsequent P2P-relayed transaction to be immediately rejected with `Reject::Full`. The relayer is notified and marks the transaction as unknown in its filter: [9](#0-8) 

This blocks the entire P2P transaction relay pipeline for all peers connected to the victim node. Legitimate users cannot get their transactions into the pool for verification or eventual block inclusion. The attack is repeatable: workers drain the queue quickly (zero-fee transactions fail fast at `pre_check`), but the attacker can immediately refill it. The node's transaction processing is effectively held hostage for as long as the attacker continues sending.

---

### Likelihood Explanation

Any unprivileged P2P peer can trigger this. The attacker needs only to:

1. Connect to the victim node as a standard P2P peer.
2. Craft transactions that pass `non_contextual_verify`: valid version, at least one input and one output, no duplicate deps, outputs data matching outputs, size ≤ 512 KB (`TRANSACTION_SIZE_LIMIT`).
3. Set outputs capacity equal to inputs capacity (zero fee) — this is undetectable without chain state.
4. Use arbitrary (non-existent) input out-points — these also pass non-contextual checks.

With transactions near the 512 KB size limit, only ~500 transactions are needed to fill the 256 MB queue. The cost is bandwidth only; no CKB tokens are spent. The default `min_fee_rate` of 1,000 shannons/KB is entirely bypassed at the admission stage: [10](#0-9) 

---

### Recommendation

Perform a lightweight, stateless fee-rate pre-screen before admitting a transaction to the verify queue. While the exact fee cannot be computed without resolving inputs, a lower-bound check is possible: if the transaction's total output capacity is provably ≥ total declared input capacity (from the transaction's own fields), or if a declared fee field is present, reject below-minimum-rate transactions early. Alternatively, add per-peer admission rate limiting in `enqueue_verify_queue` so a single peer cannot monopolize the queue. A per-peer slot cap (analogous to the orphan pool's `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` per-pool cap) would bound the damage any single peer can cause: [11](#0-10) 

---

### Proof of Concept

1. Connect to a CKB node as a P2P peer.
2. Construct ~500 transactions, each ~512 KB, with:
   - Valid structure (non-empty inputs/outputs, correct version, no duplicate deps).
   - Arbitrary non-existent input out-points.
   - Output capacity equal to a plausible input capacity (zero fee).
3. Relay all transactions via the P2P relay protocol, triggering `submit_remote_tx` on the victim node.
4. Each transaction passes `non_contextual_verify` and is admitted to the `VerifyQueue` until `total_tx_size` reaches 256 MB.
5. Observe that any subsequent `submit_remote_tx` call from any peer returns `Reject::Full`.
6. Workers drain the queue (each transaction fails at `pre_check` due to unresolvable inputs or zero fee); attacker immediately re-sends to refill.
7. Legitimate transactions from honest peers are continuously rejected for the duration of the attack.

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }
