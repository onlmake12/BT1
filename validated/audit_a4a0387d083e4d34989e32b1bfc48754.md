Audit Report

## Title
Missing Per-Peer Rate Limit on `LightClientProtocol` Enables Unbounded MMR RocksDB Reads — (`util/light-client-protocol-server/src/lib.rs`)

## Summary

`LightClientProtocol` carries no rate-limiting field and performs no rate check before dispatching incoming messages. Every `GetLastState` message unconditionally triggers `get_verifiable_tip_header()`, which performs a live O(log N) RocksDB MMR root read. Any unprivileged peer can drive this path at maximum socket speed with no throttling, contending with block-processing I/O on the shared store. The `Relayer` and `HolePunching` protocols both apply a `governor`-based 30 req/s cap per `(peer, message_type)` that `LightClientProtocol` entirely lacks.

## Finding Description

`LightClientProtocol` is defined with only a `shared: Shared` field — no rate-limiter state exists: [1](#0-0) 

`try_process` dispatches directly to handlers with zero rate-limiting logic before any handler is invoked: [2](#0-1) 

`GetLastStateProcess::execute` unconditionally calls `get_verifiable_tip_header()` regardless of the `subscribe` flag: [3](#0-2) 

`get_verifiable_tip_header()` always takes a snapshot and, for any non-genesis tip, calls `chain_root_mmr(tip_number - 1).get_root()` — a live RocksDB read: [4](#0-3) 

`chain_root_mmr` constructs a `ChainRootMMR` sized to `leaf_index_to_mmr_size(block_number)`, and `get_root()` reads O(log N) MMR nodes from the shared RocksDB snapshot: [5](#0-4) 

By contrast, `Relayer::try_process` checks a `governor`-based `RateLimiter<(PeerIndex, u32)>` at 30 req/s before dispatching any non-PoW message: [6](#0-5) 

`HolePunching` applies the identical pattern: [7](#0-6) 

`constant.rs` defines no rate-limit constant for the light-client protocol: [8](#0-7) 

A grep for `rate_limiter`, `RateLimiter`, or `governor` in the entire `util/light-client-protocol-server/` tree returns zero matches, confirming the guard is entirely absent.

## Impact Explanation

The concrete impact is sustained, unbounded RocksDB read pressure on the shared store from a single unprivileged peer, contending with block-verification I/O. This degrades block-processing throughput proportionally to message rate. With up to 125 default inbound peers each able to do this simultaneously, the aggregate I/O load can cause a node to fall significantly behind chain tip or become unresponsive to legitimate peers. This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs** (10001–15000 points): the attacker cost is a single TCP connection and a tight loop of one-boolean messages; the victim cost is shared-store saturation affecting both block processing and P2P responsiveness.

## Likelihood Explanation

The attack requires only a standard inbound P2P connection — no privileges, no PoW, no key material. `GetLastState` is a minimal fixed-size message (one boolean field), so a single TCP connection can sustain thousands of messages per second. The node's `max_inbound` cap (default 125) bounds the number of peers but not per-peer message rate. The attack is trivially repeatable and requires no victim interaction.

## Recommendation

Add a `governor`-based `RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, initialize it in `new()` at 30 req/s per `(peer, message_type)` (matching the `Relayer`), and check it at the top of `try_process` before dispatching to any handler — mirroring the pattern in `Relayer::try_process` and `HolePunching`.

## Proof of Concept

1. Start a CKB full node with `LightClient` in `support_protocols`.
2. Connect a peer and send `GetLastState { subscribe: false }` in a tight loop at maximum socket speed.
3. Observe RocksDB read IOPS spike via `rocksdb.stats` and block-processing latency increase via `ckb_chain_process_block_duration` metrics.
4. Repeat with `GetRelayTransactions` at the same rate — the `Relayer`'s 30 req/s cap throttles it; the light-client handler does not throttle.
5. Scale to multiple simultaneous peers (up to `max_inbound`) to demonstrate aggregate store saturation.
