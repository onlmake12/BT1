Audit Report

## Title
Wrong row variable in `script_exists_in_output` causes premature script deletion on PostgreSQL during rollback — (`util/rich-indexer/src/indexer/remove.rs`)

## Summary
In `script_exists_in_output`, the final `match` at line 252 reads `row_lock` (the lock-script EXISTS result) instead of `row_type` (the type-script EXISTS result). On PostgreSQL, this always returns `Ok(false)` for the type-script check, causing `rollback_block` to delete script rows from the `script` table even when surviving outputs still reference them as type scripts. The indexer database is silently corrupted; full re-sync is the only recovery.

## Finding Description
`rollback_block` (lines 7–52) removes the tip block's outputs, then calls `script_exists_in_output` for each script ID to decide whether the script row is still needed. [1](#0-0) 

Inside `script_exists_in_output`, after the lock-script EXISTS query returns `false` and the early-return at lines 223–235 is not taken, the type-script EXISTS query is executed and stored in `row_type` (lines 237–249). The final `match` at line 252 then reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. [2](#0-1) 

On **PostgreSQL**, `row_lock` holds a `BOOLEAN` value already confirmed to be `false` (otherwise the early-return at line 225–227 would have fired). `row_lock.try_get::<bool, _>(0)` succeeds and returns `Ok(false)` unconditionally — `row_type` is never read. On **SQLite**, `row_lock.try_get::<bool, _>(0)` fails (SQLite returns `BIGINT`), so the `Err(_)` arm correctly reads `row_type.get::<i64, _>(0) == 1`. SQLite is unaffected. [3](#0-2) 

## Impact Explanation
Any script used exclusively as a `type_script_id` in surviving outputs is incorrectly added to `script_id_list_to_remove` and deleted by `remove_batch_by_blobs("script", ...)`. [4](#0-3)  Surviving `output` rows retain a now-dangling `type_script_id` foreign key. Subsequent RPC calls (`get_cells`, `get_transactions`) filtered by that type script return empty or incorrect results. This matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism** — the rich-indexer is CKB's cell/transaction state storage layer, and this bug causes incorrect state after any rollback on a PostgreSQL backend.

## Likelihood Explanation
Triggered by any chain reorganization on a node running the rich-indexer with a PostgreSQL backend. Reorgs are normal, unprivileged network events requiring no attacker capability beyond the ability to produce competing blocks (or simply waiting for a natural reorg). The minimal scenario requires only two blocks sharing a type script. SQLite deployments are unaffected.

## Recommendation
Change line 252 from `row_lock.try_get::<bool, _>(0)` to `row_type.try_get::<bool, _>(0)`:

```rust
// correct fix
match row_type.try_get::<bool, _>(0) {
    Ok(r) => Ok(r),
    Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
}
``` [5](#0-4) 

## Proof of Concept
1. Start CKB rich-indexer with PostgreSQL backend.
2. Append block A with a transaction output `{lock: L1, type: S}`.
3. Append block B with a transaction output `{lock: L2, type: S}` (same type script `S`, different lock).
4. Trigger rollback of block B (simulate a reorg).
5. Query `SELECT * FROM script WHERE id = <S_id>` — the row is absent even though block A's output still references it.
6. Call `get_cells` filtered by type script `S` — returns 0 results instead of 1.
