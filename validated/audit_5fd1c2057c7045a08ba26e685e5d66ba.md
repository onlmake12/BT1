### Title
Unauthenticated `remove_transaction` RPC Allows Any Caller to Evict Any Pending Transaction — (`rpc/src/module/pool.rs`)

### Summary

The `remove_transaction` JSON-RPC method accepts only a transaction hash and performs no ownership or authorization check before permanently evicting the target transaction (and all its descendants) from the tx-pool. Any caller with RPC access — including any local CLI/RPC user on a shared node or any client if the operator has exposed the RPC port — can silently drop another user's pending transaction. This is a direct structural analog to the Centrifuge M-04 finding: a state-mutating function that acts on another user's pending state without verifying the caller is authorized to do so.

### Finding Description

**Root cause — `rpc/src/module/pool.rs`, lines 254–255 and 662–669**

The RPC trait declares:

```rust
#[rpc(name = "remove_transaction")]
fn remove_transaction(&self, tx_hash: H256) -> Result<bool>;
```

The implementation is:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

No caller identity, session token, or ownership proof is checked. The only parameter is the hash of the target transaction.

**Execution path**

`remove_transaction` RPC → `TxPoolController::remove_local_tx` (`tx-pool/src/service.rs:273`) → `Message::RemoveLocalTx` handler (`tx-pool/src/service.rs:826–833`) → `TxPoolService::remove_tx` (`tx-pool/src/process.rs:440–455`), which removes the entry from the verify queue, orphan pool, and the main pool map, including all descendant transactions.

**No authentication layer**

The RPC server (`rpc/src/server.rs:52–95`) is a plain HTTP/WebSocket/TCP server with `CorsLayer::permissive()` and no token, session, or IP-based authentication middleware. The `RpcConfig` struct (`util/app-config/src/configs/rpc.rs`) has no authentication field. The default listen address is `127.0.0.1:8114`, but the attack surface is any process or user that can reach that socket.

### Impact Explanation

Any caller with RPC access can:

1. Observe the pool via `get_raw_tx_pool` to enumerate pending transaction hashes.
2. Call `remove_transaction(<victim_tx_hash>)` to immediately and permanently evict the victim's transaction and all its children from the pool.
3. Repeat in a loop to prevent any specific transaction from ever being confirmed.

The victim's transaction is silently dropped. The victim must resubmit, paying the fee again and waiting for re-propagation. An attacker can sustain this indefinitely at zero cost (no fee, no token required). In a shared-node or hosted-node scenario (e.g., a dApp backend, an exchange hot wallet node, or a multi-tenant RPC service), this enables targeted DoS of any user's pending transactions.

### Likelihood Explanation

The RPC is enabled by default with the `Pool` module (`resource/ckb.toml:190`). The default binding is localhost, but the prompt explicitly includes "supported local CLI/RPC user" as a valid attacker profile. In practice, many deployments expose the RPC to a local network or behind a reverse proxy without per-method access control. The attack requires only knowledge of the victim's transaction hash, which is publicly observable via `get_raw_tx_pool` or the P2P relay network.

### Recommendation

Add caller-identity verification to `remove_transaction`. Options include:

- Require the caller to provide a valid signature over the transaction hash using the lock key of one of the transaction's input cells, proving ownership of the cells being spent.
- Restrict `remove_transaction` to a separate privileged RPC module (e.g., `Debug` or a new `Admin` module) that is disabled by default and requires explicit operator opt-in.
- At minimum, document that `remove_transaction` must never be exposed to untrusted callers and gate it behind a configurable allowlist.

### Proof of Concept

```
# Step 1: Victim submits a transaction
curl -X POST http://127.0.0.1:8114 -d '{"id":1,"jsonrpc":"2.0","method":"send_transaction","params":[<victim_tx>, "passthrough"]}'
# Response: {"result": "0xVICTIM_TX_HASH"}

# Step 2: Attacker enumerates the pool (or learns the hash from P2P relay)
curl -X POST http://127.0.0.1:8114 -d '{"id":2,"jsonrpc":"2.0","method":"get_raw_tx_pool","params":[false]}'
# Response includes 0xVICTIM_TX_HASH

# Step 3: Attacker evicts the victim's transaction — no credentials required
curl -X POST http://127.0.0.1:8114 -d '{"id":3,"jsonrpc":"2.0","method":"remove_transaction","params":["0xVICTIM_TX_HASH"]}'
# Response: {"result": true}

# Victim's transaction is now gone from the pool. Repeat to sustain DoS.
```
