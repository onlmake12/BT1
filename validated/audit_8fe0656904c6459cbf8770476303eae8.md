### Title
`PythAggregatorV3.decimals()` Returns Wildly Wrong Value for Price Feeds with Non-Negative Exponents Due to Unsafe `int32`→`int8`→`uint8` Cast Chain - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3.decimals()` computes the Chainlink-compatible decimal count via `uint8(-1 * int8(price.expo))`. Because `price.expo` is `int32` and Pyth price feeds can legitimately carry non-negative exponents, the narrowing cast to `int8` followed by negation and cast to `uint8` produces a wildly incorrect decimal value (e.g., 251 for `expo = 5`). Any downstream Chainlink-compatible protocol that reads `decimals()` to scale the raw `latestAnswer()` / `latestRoundData()` price will misinterpret the price by a factor of up to 10^246.

---

### Finding Description

`PythAggregatorV3` is the official Pyth Chainlink-compatibility adapter, distributed as part of `@pythnetwork/pyth-sdk-solidity` and actively recommended in the Pyth migration guide. [1](#0-0) 

```solidity
function decimals() public view virtual returns (uint8) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return uint8(-1 * int8(price.expo));
}
```

`price.expo` is typed `int32` in `PythStructs.Price`: [2](#0-1) 

The cast chain `int8(price.expo)` silently truncates the `int32` value to 8 bits. For a **positive** exponent (e.g., `expo = 5`):

1. `int8(5)` → `5` (fine, fits in int8)
2. `-1 * int8(5)` → `int8(-5)` (fine, fits in int8)
3. `uint8(int8(-5))` → **`251`** (Solidity 0.8+ explicit cast wraps, does not revert)

The Pyth protocol explicitly supports positive exponents. The Aave integration test suite confirms this: [3](#0-2) 

```solidity
function testGetAssetPriceWorksWithPositiveExponent() public {
    // "display" price is 5_293_000
    testConversion(5_293, 3, 529_300_000_000_000, BASE_CURRENCY_UNIT);
```

The `PythUtils` library also explicitly tests positive exponents: [4](#0-3) 

The `latestAnswer()` / `latestRoundData()` functions return the raw `price.price` integer without any exponent adjustment: [5](#0-4) 

Callers are expected to use `decimals()` to interpret the scale. When `decimals()` returns 251 instead of the correct value, the price is misinterpreted by a factor of 10^(251 − correct_decimals).

A secondary overflow exists for `expo = -128`: `-1 * int8(-128)` = 128, which overflows `int8` and reverts in Solidity 0.8+, causing a DoS for any feed with that exponent.

---

### Impact Explanation

Any Chainlink-compatible protocol (lending, derivatives, AMM) that integrates `PythAggregatorV3` and reads `decimals()` to normalize the price will receive a catastrophically wrong scale factor for any Pyth price feed whose exponent is ≥ 0. This enables:

- **Collateral mispricing**: A lending protocol would compute collateral value as `price * 10^(251-8)` = 10^243× the real value, allowing unlimited borrowing against minimal collateral.
- **Liquidation bypass**: Positions that should be liquidated appear massively overcollateralized.
- **Price manipulation via feed selection**: Any actor who can influence which price feed ID is registered in a downstream protocol can trigger this by pointing to a feed with a positive exponent.

---

### Likelihood Explanation

- `PythAggregatorV3` is the official, documented migration path from Chainlink to Pyth, actively promoted in the developer hub.
- The Pyth protocol's own test suite and `PythUtils` library explicitly handle positive exponents as valid inputs.
- The `PythPriceOracleGetter` (Aave integration) correctly handles positive exponents using full `int32` arithmetic, confirming the Pyth team is aware positive exponents are valid — but this fix was not applied to `PythAggregatorV3`.
- Any protocol deploying `PythAggregatorV3` for a feed that currently or in the future carries a non-negative exponent is silently vulnerable.

---

### Recommendation

Replace the unsafe cast chain with a proper `int32`-aware computation:

```solidity
function decimals() public view virtual returns (uint8) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    require(price.expo <= 0, "PythAggregatorV3: positive exponent not supported");
    require(price.expo >= -255, "PythAggregatorV3: exponent out of range");
    return uint8(uint32(-price.expo));
}
```

Or, if positive exponents must be supported, document that `latestAnswer()` must be scaled by `10^expo` by the caller and remove the misleading `decimals()` function, or return 0 for positive exponents with a clear comment.

### Proof of Concept

```solidity
// SPDX-License-Identifier: Apache 2
pragma solidity ^0.8.0;

import {PythAggregatorV3} from "@pythnetwork/pyth-sdk-solidity/PythAggregatorV3.sol";
import {MockPyth} from "@pythnetwork/pyth-sdk-solidity/MockPyth.sol";

contract PoC {
    function demonstrate() external {
        MockPyth mockPyth = new MockPyth(60, 1);
        bytes32 feedId = bytes32(uint256(1));

        // Create a price update with expo = 5 (positive, valid Pyth exponent)
        bytes[] memory updateData = new bytes[](1);
        updateData[0] = mockPyth.createPriceFeedUpdateData(
            feedId,
            int64(5_293),   // price mantissa
            uint64(10),     // conf
            int32(5),       // expo = +5  <-- positive exponent
            int64(5_293),
            uint64(10),
            uint64(block.timestamp)
        );
        mockPyth.updatePriceFeeds{value: 1}(updateData);

        PythAggregatorV3 adapter = new PythAggregatorV3(address(mockPyth), feedId);

        uint8 d = adapter.decimals();
        // Expected: some meaningful value (e.g., 0 or revert)
        // Actual:   uint8(-1 * int8(5)) = uint8(-5) = 251
        assert(d == 251); // PASSES — wildly wrong decimal value
    }
}
```

The raw `latestAnswer()` returns `5293`. A downstream protocol computing `price / 10^decimals` = `5293 / 10^251` ≈ 0, or `price * 10^decimals` = `5293 * 10^251` (astronomical), depending on the direction of scaling — both are catastrophically wrong. [1](#0-0) [6](#0-5)
