### Title
Expired Network Alert Replay via Missing Dedup Persistence in `clear_expired_alerts` — (File: `util/network-alert/src/notifier.rs`)

---

### Summary

`Notifier::clear_expired_alerts` removes expired alerts from `received_alerts` but does not record their IDs in `cancel_filter`. After this purge, `has_received` returns `false` for those IDs. Any unprivileged P2P peer who retained the original signed alert bytes can relay them again; the node re-accepts, re-broadcasts, and re-notifies on the replayed alert. The Foundation's multi-sig is still cryptographically valid on the old bytes, so signature verification passes unconditionally.

---

### Finding Description

**Root cause — `notifier.rs` lines 135–144:**

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|_id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        notice_until > now
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

Expired alerts are dropped from `received_alerts` and `noticed_alerts`. They are **never** inserted into `cancel_filter`. [1](#0-0) 

**Dedup gate — `notifier.rs` lines 147–149:**

```rust
pub fn has_received(&self, id: u32) -> bool {
    self.received_alerts.contains_key(&id) || self.cancel_filter.contains(&id)
}
```

After `clear_expired_alerts` runs, neither map contains the expired ID, so `has_received` returns `false`. [2](#0-1) 

**P2P receive path — `alert_relayer.rs` lines 147–178:**

The `received` handler checks `has_received` first (line 147), then verifies signatures (line 151), then broadcasts (lines 166–176), then calls `notifier.add` (line 178). There is **no expiry check** anywhere in this path. [3](#0-2) 

**Contrast with RPC path — `alert.rs` lines 104–110:**

The `send_alert` RPC explicitly rejects alerts whose `notice_until` is in the past. The P2P relay path has no equivalent guard. [4](#0-3) 

**Trigger — `alert_relayer.rs` line 87:**

`clear_expired_alerts` is called every time any peer connects, so the replay window opens regularly and automatically. [5](#0-4) 

**Re-notification — `notifier.rs` lines 103–116:**

`add` inserts the replayed alert back into `received_alerts` and calls `notify_controller.notify_network_alert` without checking `notice_until`. [6](#0-5) 

**Analog mapping to QuailFinance:**

| QuailFinance | CKB |
|---|---|
| Signature authorizes joining a round | Foundation multi-sig authorizes an alert |
| Nonce passed as argument, not tracked server-side | Alert ID cleared from dedup state on expiry |
| Same signature replayed to join multiple times | Same signed alert bytes replayed via P2P after expiry |
| Blocks other participants, guarantees win | Re-triggers node notifications, re-broadcasts network-wide |

---

### Impact Explanation

1. **Re-notification**: `notify_controller.notify_network_alert` fires again for an expired alert. Node operators who configured external notification scripts (e.g., emergency shutdown hooks) will have those scripts re-triggered by a stale, attacker-chosen alert.
2. **Network-wide re-broadcast**: The replayed alert is forwarded to all connected peers (lines 166–176 of `alert_relayer.rs`), propagating the replay across the entire P2P network.
3. **Alert state corruption**: The expired alert re-appears in `received_alerts` and `noticed_alerts`, surfacing in `get_blockchain_info().alerts` responses and misleading operators and monitoring tools.
4. **Cancellation bypass (secondary)**: `cancel_filter` is an LRU cache of size 128 (`CANCEL_FILTER_SIZE`). If the Foundation has issued more than 128 cancel alerts, evicted cancel IDs lose their dedup protection entirely, allowing replay of explicitly cancelled alerts by any peer — with no expiry precondition required. [7](#0-6) 

---

### Likelihood Explanation

- **Attacker precondition**: Zero privilege. Any peer that was connected to the network while a legitimate alert was active retains the raw signed bytes. No keys, no special access.
- **Trigger frequency**: `clear_expired_alerts` is called on every peer connection event. On a live node with normal peer churn, the replay window opens continuously.
- **Effort**: Trivially low. The attacker replays verbatim bytes they already received; no cryptographic work is required.

---

### Recommendation

1. In `clear_expired_alerts`, record each expired alert's ID into `cancel_filter` before removing it from `received_alerts`:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ()); // prevent replay
            false
        } else {
            true
        }
    });
    // noticed_alerts retain unchanged
}
```

2. Add an expiry check in `Notifier::add` (and optionally in the P2P `received` handler) to reject alerts whose `notice_until` is already in the past, mirroring the guard already present in the RPC path (`alert.rs` lines 104–110). [8](#0-7) 

---

### Proof of Concept

1. A legitimate alert (ID = `0x1`, `notice_until = T`, valid Foundation 2-of-4 signatures) is broadcast. Attacker peer saves the raw bytes.
2. Time advances past `T`. Alert expires.
3. Any new peer connects to the victim node → `clear_expired_alerts` fires → alert ID `0x1` is removed from `received_alerts`; it was never in `cancel_filter`.
4. Attacker peer sends the saved raw bytes to the victim node via the Alert P2P protocol.
5. `has_received(0x1)` → `false` (both maps empty for this ID).
6. `verifier.verify_signatures(&alert)` → `Ok(())` (Foundation signatures are still valid over the original bytes).
7. `notifier.add(&alert)` → inserts into `received_alerts`, calls `notify_controller.notify_network_alert`.
8. Alert is broadcast to all connected peers; they repeat steps 5–8.
9. Network-wide re-propagation of the expired alert completes. Any operator-configured alert hook fires on every affected node.
