### Title
Duplicate Transaction Inputs Not Rejected, Enabling Capacity Double-Counting - (`verification/src/transaction_verifier.rs`, `util/types/src/core/cell.rs`)

---

### Summary

`NonContextualTransactionVerifier` checks for duplicate `cell_deps` and `header_deps` but contains **no analogous check for duplicate inputs**. The cell-resolution `check` function in `util/types/src/core/cell.rs` silently accepts a repeated input `OutPoint` via an early-return path intended for dep deduplication. As a result, `CapacityVerifier::verify` double-counts the capacity of the repeated cell, allowing a transaction sender to create outputs whose total capacity exceeds the capacity of the cells they actually own.

---

### Finding Description

**Vulnerability class (analog):** Duplicate entries in an array parameter that are not deduplicated before processing, leading to double-counting of a resource — here, CKB cell capacity instead of ERC-20 token balances.

**Step 1 — No duplicate-input check in `NonContextualTransactionVerifier`.**

`NonContextualTransactionVerifier` runs six sub-verifiers:

```
version → size → empty → duplicate_deps → outputs_data → script_hash_type
```

`DuplicateDepsVerifier` explicitly deduplicates `cell_deps` and `header_deps` using a `HashSet`, but there is no corresponding `DuplicateInputsVerifier`. A transaction whose `inputs` list contains the same `OutPoint` twice passes `NonContextualTransactionVerifier::verify` without error.

**Step 2 — `check` in `cell.rs` silently accepts the second occurrence.**

`ResolvedTransaction::check` uses a single `check_cell` closure for both inputs and cell deps:

```rust
let mut checked_cells: HashSet<OutPoint> = HashSet::new();
let mut check_cell = |out_point: &OutPoint| -> Result<(), OutPointError> {
    if seen_inputs.contains(out_point) {          // cross-tx double-spend guard
        return Err(OutPointError::Dead(...));
    }
    if checked_cells.contains(out_point) {        // ← silently accepts duplicate input
        return Ok(());
    }
    match cell_checker.is_live(out_point) { ... }
};

for cell_meta in &self.resolved_inputs {
    check_cell(&cell_meta.out_point)?;            // second occurrence returns Ok(())
}
```

`checked_cells` was designed to avoid re-checking the same dep cell referenced by multiple dep groups. Because the same closure is reused for inputs, the second occurrence of a duplicate input `OutPoint` hits the `checked_cells.contains` branch and returns `Ok(())` without error. `seen_inputs` (the cross-transaction guard) does not help here because it is only populated with the current transaction's inputs *after* all checks complete (line 382).

**Step 3 — `CapacityVerifier` double-counts capacity.**

`CapacityVerifier::verify` calls `self.resolved_transaction.inputs_capacity()`, which iterates over `resolved_inputs` — a `Vec<CellMeta>`. Because `resolved_inputs` is built directly from the transaction's input list (one `CellMeta` per input entry), a transaction with the same `OutPoint` twice produces two `CellMeta` entries for the same cell. `inputs_capacity` sums all entries, counting that cell's capacity twice.

The capacity check then passes:

```
inputs_sum (2 × cell_capacity) >= outputs_sum (up to 2 × cell_capacity)
```

allowing the attacker to create outputs totalling up to twice the capacity of the single cell they actually control.

---

### Impact Explanation

An unprivileged transaction sender can craft a transaction that lists the same live cell `OutPoint` N times in `inputs`. The `CapacityVerifier` will compute `inputs_sum = N × cell_capacity`, permitting `outputs_sum` up to the same inflated value. The attacker effectively creates CKB capacity out of thin air — up to `(N-1) × cell_capacity` per transaction — which is a direct theft of native token value from the network's total supply accounting. Any such transaction that passes verification and is committed to a block permanently inflates the circulating supply.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via the standard RPC (`send_transaction`) or P2P relay. No privileged access, key compromise, or majority hashpower is needed. The attacker needs only one live cell they control. The crafted transaction passes all existing non-contextual and contextual checks as described above. Likelihood is **high** for any actor motivated by financial gain.

---

### Recommendation

Add a `DuplicateInputsVerifier` (mirroring `DuplicateDepsVerifier`) inside `NonContextualTransactionVerifier`:

```rust
// In NonContextualTransactionVerifier::verify():
let mut seen = HashSet::with_capacity(tx.inputs().len());
if let Some(input) = tx.inputs_iter().find_map(|i| seen.replace(i.previous_output())) {
    return Err(TransactionError::DuplicateInputs { out_point: input }.into());
}
```

Alternatively, add the same guard inside `ResolvedTransaction::check` for the inputs loop, separate from the `checked_cells` set used for deps. The `checked_cells` early-return path should not apply to inputs.

---

### Proof of Concept

1. Attacker owns cell `C` with capacity 100 CKB at `OutPoint { tx_hash: H, index: 0 }`.
2. Attacker builds a transaction:
   - `inputs`: `[OutPoint(H,0), OutPoint(H,0)]` — same cell listed twice
   - `outputs`: one cell with capacity 200 CKB (attacker's lock)
3. `NonContextualTransactionVerifier::verify()` passes — no duplicate-input check exists.
4. `ResolvedTransaction::check()` passes — second input hits `checked_cells` early return.
5. `CapacityVerifier::verify()` computes `inputs_sum = 200 CKB`, `outputs_sum = 200 CKB` → passes.
6. Lock script executes for both input groups; since both reference the same cell and the attacker holds the key, both pass.
7. Transaction is committed; attacker now holds a 200 CKB cell having spent only 100 CKB.
