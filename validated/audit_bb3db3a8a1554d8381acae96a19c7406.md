### Title
Fallback Fee Rate Estimator Double-Counts Boundary Transaction, Producing Inconsistent Per-Block Rate — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

The fallback `estimate_fee_rate` function in `tx-pool/src/component/pool_map.rs` double-counts the transaction that triggers a block-capacity boundary. The same transaction is simultaneously treated as the last entry of the current simulated block **and** the first entry of the next simulated block. This makes the returned fee rate inconsistently defined — it cannot be reliably interpreted as the minimum fee rate to be included within the requested number of blocks — and causes the estimate to be wrong in a way that is not bounded or predictable.

---

### Finding Description

The fallback `estimate_fee_rate` in `pool_map.rs` is the algorithm used when the primary fee estimator (ConfirmationFraction or WeightUnitsFlow) returns `Error::NotReady` or `Error::LackData` — which occurs during IBD and immediately after node startup. [1](#0-0) 

The algorithm iterates transactions from highest to lowest fee rate and simulates filling blocks greedily. The bug is in the block-boundary reset at lines 353–354:

```rust
for entry in iter {
    current_block_bytes += entry.inner.size;       // (1) entry added to current block
    current_block_cycles += entry.inner.cycles;
    if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
        target_blocks -= 1;
        if target_blocks == 0 {
            return entry.inner.fee_rate();          // (2) returned as last of current block
        }
        current_block_bytes = entry.inner.size;    // (3) ALSO counted as first of next block
        current_block_cycles = entry.inner.cycles;
    }
}
```

The boundary entry is counted in **both** the current block (step 1) and the next block (step 3). This is the same structural defect as the external report: a rate-returning function whose formula is inconsistent because it does not correctly normalize by the right denominator (here: the correct per-block weight budget).

**Concrete counterexample** (`target_blocks = 2`, `max_block_bytes = 100`):

| Tx | size | fee_rate |
|----|------|----------|
| T1 | 60   | 10       |
| T2 | 90   | 8        |
| T3 | 20   | 6        |
| T4 | 60   | 4        |

*Current (buggy) execution:*
1. T1: `bytes = 60`, no overflow
2. T2: `bytes = 150 ≥ 100` → `target_blocks = 1`, reset to `bytes = 90` (T2 double-counted)
3. T3: `bytes = 110 ≥ 100` → `target_blocks = 0` → **returns fee_rate = 6**

*Correct execution:*
- Block 1: T1 (60) + T2 (90) = 150 → overflow, block 1 done
- Block 2: T3 (20) + T4 (60) = 80 → no overflow, pool exhausted
- **Should return `min_fee_rate`**

The function returns `6` instead of `min_fee_rate`. The estimate is wrong by an unbounded factor depending on pool composition.

The fallback is called from `pool.rs`: [2](#0-1) 

Which is invoked from `process.rs` when the primary estimator fails: [3](#0-2) 

And exposed via the `estimate_fee_rate` RPC in the `Experiment` module: [4](#0-3)

### Impact Explanation

Any caller of the `estimate_fee_rate` RPC receives a fee rate that is inconsistently calculated and cannot be reliably used to guarantee transaction inclusion within the requested number of blocks. In the worst case (as shown above), the function returns a non-zero fee rate when the correct answer is `min_fee_rate`, causing users to significantly overpay. Because the magnitude of the error depends on the size distribution of transactions in the pool, the returned value cannot be consistently interpreted as a per-block rate at all — directly analogous to the external report's finding that the rate "cannot even be consistently interpreted as the reward rate on the entire deposit amount."

The impact is financial: users who rely on this estimate to set transaction fees will overpay, and cannot guarantee a specific confirmation target.

---

### Likelihood Explanation

The `Experiment` module is **enabled by default** in the standard node configuration: [5](#0-4) 

The fallback path is triggered whenever the primary fee estimator lacks sufficient historical data — specifically during IBD and immediately after node startup, which are common operational states. Any local RPC caller (a supported, unprivileged role) can invoke `estimate_fee_rate` and receive the incorrect estimate. No special privileges, keys, or network access are required.

---

### Recommendation

Reset `current_block_bytes` and `current_block_cycles` to **zero** (not to the boundary entry's own size/cycles) when a block boundary is detected. The boundary entry should be counted in exactly one block:

```rust
// Option A: boundary entry is the LAST of the current block (reset next block to 0)
if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
    target_blocks -= 1;
    if target_blocks == 0 {
        return entry.inner.fee_rate();
    }
    current_block_bytes = 0;   // was: entry.inner.size
    current_block_cycles = 0;  // was: entry.inner.cycles
}
```

Or, alternatively, treat the boundary entry as the **first** of the next block (do not add it to the current block before checking):

```rust
// Option B: check overflow before adding
if current_block_bytes + entry.inner.size >= max_block_bytes
    || current_block_cycles + entry.inner.cycles >= max_block_cycles
{
    target_blocks -= 1;
    if target_blocks == 0 {
        return entry.inner.fee_rate();
    }
    current_block_bytes = entry.inner.size;
    current_block_cycles = entry.inner.cycles;
} else {
    current_block_bytes += entry.inner.size;
    current_block_cycles += entry.inner.cycles;
}
```

Either fix eliminates the double-counting and makes the returned fee rate a consistent, well-defined measure of the minimum rate needed for inclusion within `target_blocks` blocks.

    pub(crate) fn estimate_fee_rate(
        &self,
        mut target_blocks: usize,
        max_block_bytes: usize,
        max_block_cycles: Cycle,
        min_fee_rate: FeeRate,
    ) -> FeeRate {
        debug_assert!(target_blocks > 0);
        let iter = self.entries.iter_by_score().rev();
        let mut current_block_bytes = 0;
        let mut current_block_cycles = 0;
        for entry in iter {
            current_block_bytes += entry.inner.size;
            current_block_cycles += entry.inner.cycles;

                if target_blocks == 0 {
                    return entry.inner.fee_rate();
                }
                current_block_bytes = entry.inner.size;
                current_block_cycles = entry.inner.cycles;
            }
        }

        min_fee_rate
    }
