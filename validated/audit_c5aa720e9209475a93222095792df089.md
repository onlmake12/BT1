### Title
Silent Error Swallowing in `MigrationWorker::start()` Allows Subsequent Background Migrations to Execute After a Failed Migration — (`db-migration/src/lib.rs`)

---

### Summary

`MigrationWorker::start()` silently discards errors from individual background migration tasks and unconditionally continues to the next migration in the queue. This is the direct analog of the reported Fractional `join()`/`leave()` lifecycle bug: an operation that should halt after an unsuccessful state transition instead proceeds, leaving the system in an inconsistent state.

---

### Finding Description

In `db-migration/src/lib.rs`, the `MigrationWorker::start()` function drives background database migrations in a loop:

```rust
// db-migration/src/lib.rs lines 65–91
while let Some((name, task)) = self.tasks.lock().unwrap().pop_front() {
    select! {
        recv(self.inbox) -> msg => { ... break; }
        default => {
            if let Ok(db) = task.migrate(self.db.clone(), Arc::new(pb)) {
                db.put_default(MIGRATION_VERSION_KEY, task.version())
                    ...
                    .unwrap();
            }
            // ← Err(_) branch: silently falls through; loop continues
        }
    }
}
``` [1](#0-0) 

When `task.migrate()` returns `Err`, the `if let Ok(db)` arm is not taken. There is no `break`, no `return`, and no error propagation. The `while` loop pops the next task and runs it on the same underlying `self.db` — a database that was only partially (or incorrectly) transformed by the failed migration.

**Contrast with the synchronous path** in `run_migrate()`:

```rust
// db-migration/src/lib.rs lines 236–238
db = m.migrate(db, Arc::new(pb))?;   // ← ? propagates Err, halts chain
db.put_default(MIGRATION_VERSION_KEY, m.version())...?;
``` [2](#0-1) 

The synchronous path correctly aborts on the first failure. The asynchronous path does not.

**The failure path is explicitly designed into the framework.** The `BlockExt2019ToZero` background migration returns `Err` when the stop signal fires:

```rust
// util/migrate/src/migrations/set_2019_block_cycle_zero.rs line 84
if self.stop_background() {
    return Err(InternalErrorKind::Database.other("intrupted").into());
}
``` [3](#0-2) 

The `run_migrate_async` function collects all pending background migrations into a `VecDeque` and hands them to a single `MigrationWorker`: [4](#0-3) 

If a second background migration is ever added after `BlockExt2019ToZero` (the framework is explicitly built to support this), and the first migration fails mid-run, the second migration will execute against a partially transformed database without any guard.

---

### Impact Explanation

When Migration A fails and Migration B subsequently succeeds:

1. The `MIGRATION_VERSION_KEY` is advanced to B's version.
2. On the next node restart, `run_migrate_async` filters tasks with `mv.as_str() > v` — Migration A's version is now less than B's version, so **Migration A is permanently skipped**.
3. The database is left in a state where A's schema transformation was never completed, but the version key claims it was (because B ran and wrote its version).

For the existing `BlockExt2019ToZero` migration, this means `BlockExt.cycles` fields for pre-CKB2021 blocks may be left in a mixed state (some zeroed, some not), which affects block reward accounting and any consensus logic that reads `BlockExt`. [5](#0-4) 

---

### Likelihood Explanation

The trigger is a normal operator action: starting the node with pending background migrations and then stopping it (e.g., SIGTERM, restart, disk-full I/O error). The `BlockExt2019ToZero` migration explicitly handles this by returning `Err`. The bug is latent today because only one background migration exists, but the `MigrationWorker` loop is written to handle multiple, and the defect will activate the moment a second background migration is registered in `util/migrate/src/migrate.rs`. [6](#0-5) 

The entry path is the local CLI/RPC user (node operator) running `ckb run` or `ckb migrate`, which is explicitly in scope.

---

### Recommendation

Mirror the synchronous `run_migrate` error-handling pattern in `MigrationWorker::start()`. On migration failure, log the error and `break` out of the loop immediately — do not advance to the next task:

```rust
match task.migrate(self.db.clone(), Arc::new(pb)) {
    Ok(db) => {
        db.put_default(MIGRATION_VERSION_KEY, task.version())
            .map_err(|err| internal_error(format!("failed to migrate the database: {err}")))
            .unwrap();
    }
    Err(e) => {
        eprintln!("background migration '{}' failed: {:?}; halting migration chain", name, e);
        break;
    }
}
``` [7](#0-6) 

---

### Proof of Concept

1. Register two background migrations in `Migrations`, where the first one returns `Err` unconditionally.
2. Call `migrations.migrate(db, true)` (background mode).
3. Observe that the second migration's `migrate()` is called and its version is written to `MIGRATION_VERSION_KEY`.
4. On the next call to `migrations.migrate()`, the first migration is skipped because its version is now less than the stored version — the database is permanently in a partially migrated state.

The existing test `test_background_migration` in `db-migration/src/tests.rs` only tests the success path and the stop-signal path; it does not test the case where a migration returns `Err` mid-chain. [8](#0-7)

    fn run_migrate_async(&self, db: RocksDB, v: &str) {
        let migrations: VecDeque<(String, Arc<dyn Migration>)> = self
            .migrations
            .iter()
            .filter(|(mv, _)| mv.as_str() > v)
            .map(|(mv, m)| (mv.to_string(), Arc::clone(m)))
            .collect::<VecDeque<_>>();

        let all_can_resume = migrations.iter().all(|(_, m)| m.can_resume());
        let tasks = Arc::new(Mutex::new(migrations));
        let (tx, rx) = unbounded();
        let worker = MigrationWorker::new(tasks, db, rx);

        let exit_signal = ckb_stop_handler::new_crossbeam_exit_rx();
        let clone = v.to_string();
        let tx_clone = tx.clone();
        let notifier = thread::spawn(move || {
            let _ = exit_signal.recv();
            let res = SHUTDOWN_BACKGROUND_MIGRATION.set(true);
            let _ = tx_clone.send(Command::Stop);
            info!("set shutdown flag to true: {:?} version: {}", res, clone);
        });
        register_thread("migration-notifier", notifier);

        let handler = worker.start();
        if all_can_resume {
            info!("register thread: migration ....");
            register_thread("migration", handler);
        }
        tx.send(Command::Start).expect("send start command");
    }

    /// If `run_in_background` is true, long-running migrations will be executed asynchronously.
    pub fn migrate(&self, db: RocksDB, run_in_background: bool) -> Result<RocksDB, Error> {
        let db_version = self.get_migration_version(&db)?;
        match db_version {
            Some(ref v) => {
                info!("Current database version {}", v);
                self.check_migration_downgrade(v)?;
                let db = if !run_in_background {
                    self.run_migrate(db, v.as_str())?
                } else {
                    self.run_migrate_async(db.clone(), v.as_str());
                    db
                };
                Ok(db)
            }
            None => {
                // if version is none, but db is not empty
                // patch 220464f
                if self.is_non_empty_db(&db) {
                    return self.patch_220464f(db);
                }
                Ok(db)
            }
        }
    }
