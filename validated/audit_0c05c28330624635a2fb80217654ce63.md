### Title
Unbounded `get_cells_capacity` RPC Scan Enables DoS on Indexer-Enabled Nodes — (File: `rpc/src/module/indexer.rs`, `util/indexer/src/service.rs`, `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`)

---

### Summary

The `get_cells_capacity` RPC endpoint in both the basic Indexer and Rich Indexer modules accepts a `search_key` with **no `limit` parameter** and must scan **all matching live cells** to compute the total capacity. Unlike the sibling endpoints `get_cells` and `get_transactions`, which enforce a caller-supplied `limit` and validate it against a server-side `request_limit`, `get_cells_capacity` has no such bound. As the chain grows and more cells accumulate, the cost of each call grows proportionally. Operators who enable the Indexer or RichIndexer module for wallet/dapp support cannot disable this specific endpoint, creating a sustained DoS vector.

---

### Finding Description

**`get_cells` and `get_transactions` enforce a limit; `get_cells_capacity` does not.**

In the rich indexer, `get_transactions` explicitly validates the caller-supplied limit:

```rust
// util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs:23-32
let limit = limit.value();
if limit == 0 {
    return Err(Error::invalid_params("limit should be greater than 0"));
}
if limit as usize > self.request_limit {
    return Err(Error::invalid_params(format!(
        "limit must be less than {}",
        self.request_limit,
    )));
}
```

The `get_cells_capacity` function in the same codebase has **no equivalent check**:

```rust
// util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs:13-16
pub async fn get_cells_capacity(
    &self,
    search_key: IndexerSearchKey,
) -> Result<Option<IndexerCellsCapacity>, Error> {
```

It issues a raw SQL `SUM` aggregation over the entire `output` table filtered only by the caller-controlled `search_key`, with no `LIMIT` clause and no server-side row cap:

```rust
// util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs:26-27
let mut query_builder = SqlBuilder::select_from("output");
query_builder.field("CAST(SUM(output.capacity) AS BIGINT) AS total_capacity");
```

In the basic (RocksDB-backed) indexer, `get_cells_capacity` uses a `TimeoutIterator`, but the `timeout_limit` field is set at service construction time and may be very large; there is no per-request cap on the number of cells iterated:

```rust
// util/indexer/src/service.rs:720
let mut iter = TimeoutIterator::new(snapshot.iterator(mode).skip(skip), self.timeout_limit);
```

The RPC trait definition confirms the absence of a `limit` parameter:

```rust
// rpc/src/module/indexer.rs:879-883
#[rpc(name = "get_cells_capacity")]
fn get_cells_capacity(
    &self,
    search_key: IndexerSearchKey,
) -> Result<Option<IndexerCellsCapacity>>;
```

An attacker can craft a `search_key` with a short `code_hash` prefix (prefix search mode) that matches a large fraction of all live cells, forcing the node to aggregate capacity across millions of rows per request.

---

### Impact Explanation

A single RPC call to `get_cells_capacity` with a broad prefix search key (e.g., `script_search_mode: "prefix"` with a 1-byte `args` prefix) forces the indexer to scan and aggregate every matching live cell in the database. On a mature mainnet node with tens of millions of live cells, this can saturate I/O and CPU for seconds per request. Concurrent or rapid repeated calls can render the RPC server unresponsive, denying service to legitimate users. The rich indexer's SQL `SUM` query holds a database transaction open for the full duration, blocking concurrent writes.

---

### Likelihood Explanation

Any unprivileged RPC caller can trigger this. No authentication, no fee, no proof-of-work is required. Operators who expose the Indexer or RichIndexer module (common for public nodes serving wallets and dapps) cannot selectively disable `get_cells_capacity` without disabling the entire module, which would also remove `get_cells` and `get_transactions`. The attack cost is a single HTTP POST request per invocation.

---

### Recommendation

1. Add a server-side `request_limit` check to `get_cells_capacity` in both the basic indexer (`util/indexer/src/service.rs`) and the rich indexer (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`), analogous to the check already present in `get_transactions`.
2. Enforce a mandatory `block_range` filter or a maximum scan-row cap to bound the work per call.
3. Consider moving `get_cells_capacity` to a separate, non-default-enabled namespace (e.g., `indexer_admin_`) so operators can expose `get_cells` and `get_transactions` without exposing the unbounded aggregation endpoint.

---

### Proof of Concept

```json
{
  "id": 1,
  "jsonrpc": "2.0",
  "method": "get_cells_capacity",
  "params": [
    {
      "script": {
        "code_hash": "0x00",
        "hash_type": "type",
        "args": "0x"
      },
      "script_type": "lock",
      "script_search_mode": "prefix"
    }
  ]
}
```

Sending this request repeatedly to a node with the `Indexer` or `Rich_indexer` module enabled forces a full scan of all live cells on every call. The `args: "0x"` (empty prefix) matches every cell indexed under any lock script with the given `code_hash` prefix, maximizing the scan width.
