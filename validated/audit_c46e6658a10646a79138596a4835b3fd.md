Audit Report

## Title
Unbounded `OrphanBlockPool` Growth via Epoch-Bypass Flooding — (`chain/src/utils/orphan_block_pool.rs`)

## Summary

`InnerPool::insert` enforces no hard capacity limit; `HashMap::with_capacity` is a Rust pre-allocation hint only. The sole eviction path (`clean_expired_blocks`) checks `epoch_number + EXPIRED_EPOCH < tip_epoch`, which an attacker permanently defeats by setting crafted block epoch numbers ≥ `tip_epoch`. Any peer able to produce PoW-valid blocks with distinct unknown parent hashes can grow `blocks`, `parents`, and `leaders` without bound, eventually OOM-killing the node.

## Finding Description

**No capacity cap.** `InnerPool::with_capacity` pre-allocates but does not cap: [1](#0-0) 

`InnerPool::insert` performs zero size checks before inserting into all three structures: [2](#0-1) 

The production pool is initialized with `ORPHAN_BLOCK_SIZE = BLOCK_DOWNLOAD_WINDOW as usize = 8192` — again, only a hint: [3](#0-2) 

**Epoch-bypass defeats the only eviction path.** `clean_expired_blocks` iterates leaders and calls `need_clean`, which checks: [4](#0-3) 

Setting `epoch_number >= tip_epoch` makes `epoch_number + EXPIRED_EPOCH < tip_epoch` permanently false. The 60-second timer fires but removes nothing: [5](#0-4) 

**Reachable attack path.** A remote peer sends a block → `asynchronous_process_block` → `non_contextual_verify` (PoW checked) → `insert_block` (DB write) → `process_lonely_block`. When the parent hash is unknown and not invalid, the block falls unconditionally into the orphan pool: [6](#0-5) 

Each block with a unique parent hash adds one entry to `blocks`, one to `parents`, and one to `leaders`. The `process_block_rx` channel bounded at 24 slots throttles rate but does not prevent the attack: [7](#0-6) 

## Impact Explanation

Continuous insertion of orphan blocks with distinct parent hashes and `epoch_number >= tip_epoch` causes linear, unbounded memory growth across all three `InnerPool` structures with zero eviction. Each block is also persisted to the DB via `insert_block`, adding disk pressure. The node process is eventually OOM-killed. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The critical constraint is valid PoW, verified non-contextually via `BlockVerifier` in `non_contextual_verify`: [8](#0-7) 

On testnet/devnet (near-zero difficulty), this is trivially exploitable by any participant. On mainnet, it requires sustained hashpower investment but does not require majority hashpower — only a steady stream of valid blocks. The epoch-expiry bypass is unconditional and requires no special capability beyond valid PoW. The attack is repeatable and requires no victim interaction.

## Recommendation

1. **Enforce a hard capacity cap in `InnerPool::insert`**: store `capacity` as a field on `InnerPool`, check `self.parents.len() >= self.capacity` before inserting, and drop the incoming block (or evict a random leader and its descendants) when the cap is reached.
2. **Add per-peer orphan block accounting**: track how many orphans each peer has contributed and ban peers that exceed a threshold.
3. **Tighten epoch range checks** in non-contextual verification to reject blocks whose epoch number is implausibly far ahead of the current tip (requires passing tip epoch into the verifier).

## Proof of Concept

```rust
// Integration test on devnet (PoW difficulty near zero):
let pool = OrphanBlockPool::with_capacity(BLOCK_DOWNLOAD_WINDOW as usize);
let tip_epoch = shared.snapshot().tip_header().epoch().number();
for i in 0..100_000u64 {
    let block = BlockBuilder::default()
        .parent_hash(random_byte32())   // unique unknown parent each iteration
        .epoch(EpochNumberWithFraction::new(tip_epoch, i % 1000, 1000))
        .number(tip_epoch * 1000 + i)
        // valid PoW (trivial on devnet), valid cellbase
        .build();
    pool.insert(block.into());
}
assert_eq!(pool.len(), 100_000); // no eviction occurred

// Trigger eviction — confirms epoch bypass:
pool.clean_expired_blocks(tip_epoch);
assert_eq!(pool.len(), 100_000); // still 100_000, zero removed

// Monitor RSS: grows linearly with no bound enforced.
```

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
    }

    pub(crate) fn process_lonely_block(&self, lonely_block: LonelyBlockHash) {
        let block_hash = lonely_block.block_number_and_hash.hash();
        let block_number = lonely_block.block_number_and_hash.number();
        let parent_hash = lonely_block.parent_hash();
        let parent_is_pending_verify = self.is_pending_verify.contains(&parent_hash);
        let parent_status = self.shared.get_block_status(&parent_hash);
        if parent_is_pending_verify || parent_status.contains(BlockStatus::BLOCK_STORED) {
            debug!(
                "parent {} has stored: {:?} or is_pending_verify: {}, processing descendant directly {}-{}",
                parent_hash, parent_status, parent_is_pending_verify, block_number, block_hash,
            );
            self.process_descendant(lonely_block);
        } else if parent_status.eq(&BlockStatus::BLOCK_INVALID) {
            self.process_invalid_block(lonely_block);
        } else {
            self.orphan_blocks_broker.insert(lonely_block);
        }

        self.search_orphan_leaders();

        if let Some(metrics) = ckb_metrics::handle() {
            metrics
                .ckb_chain_orphan_count
                .set(self.orphan_blocks_broker.len() as i64)
