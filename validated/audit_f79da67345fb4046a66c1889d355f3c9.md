Audit Report

## Title
Full CKB-VM Execution Before Cycle Mismatch Check Enables Asymmetric CPU DoS via Relay Protocol — (`tx-pool/src/process.rs`)

## Summary
In `_process_tx`, full `ContextualTransactionVerifier` (CKB-VM script execution) runs unconditionally before the `declared_cycles != verified.cycles` check. Because the verify cache is only populated on the success path, every resubmission of the same transaction with `declared_cycles = actual + 1` forces a full VM execution. Combined with `remove_from_known_txs` removing the tx hash on rejection, an attacker with one valid UTXO and multiple IPs can sustain repeated full-VM occupations of the victim's verify queue at negligible cost.

## Finding Description
The execution order in `_process_tx` is confirmed at [1](#0-0) :

1. `pre_check` (line 715) — cheap input resolution and fee check
2. `fetch_tx_verify_cache` (line 719) — cache lookup by `witness_hash`
3. `max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles())` (line 720)
4. **`verify_rtx(...)` with `max_cycles`** (lines 724–732) — full CKB-VM execution
5. **Only then**: `if declared != verified.cycles` → `Reject::DeclaredWrongCycles` (lines 736–748)

With `declared_cycles = actual + 1`, `max_cycles` is set to `actual + 1`, which is sufficient for the VM to complete. The VM returns `verified.cycles = actual`, the mismatch is detected, and `DeclaredWrongCycles` is returned. Using `declared = actual - 1` would instead cause `ExceededMaximumCycles` before the mismatch check — the integration test at line 26 correctly uses `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1`, confirming the `actual + 1` path. [2](#0-1) 

The cache update is placed **after** the early return for `DeclaredWrongCycles` and only on the success path: [3](#0-2) 

So the cache is never populated on a `DeclaredWrongCycles` rejection. Every resubmission of the same witness triggers full VM execution. The cache fast-path (`TimeRelativeTransactionVerifier`) is only taken when `cache_entry` is `Some`: [4](#0-3) 

On rejection, `TxVerificationResult::Reject` causes `remove_from_known_txs` to remove the tx_hash from the `tx_filter`: [5](#0-4) 

This means the same tx_hash can be re-announced after reconnection from a new IP, since the filter no longer blocks it. The `DeclaredWrongCyclesAndRelayAgain` integration test explicitly demonstrates this behavior — after clearing the ban and waiting for the filter removal, the same tx can be relayed again. [6](#0-5) 

The only pre-VM guard in `TransactionsProcess::execute` is `declared_cycles > max_block_cycles` (immediate ban, no VM). A declared value of `actual + 1` passes this guard trivially: [7](#0-6) 

## Impact Explanation
Each relay submission with `declared_cycles = actual + 1` forces the victim node to run full `ContextualTransactionVerifier` including secp256k1 CKB-VM execution, occupying the verify queue for the full duration. The attacker spends O(1) CPU (send a P2P message); the victim spends O(secp256k1_cycles) CPU. With N IPs, the attacker can sustain N parallel verify-queue occupations, delaying legitimate transaction admission. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
**Preconditions:**
- Attacker must own at least one valid UTXO to pass `pre_check` — a low bar for any CKB participant
- Attacker needs multiple IPs to bypass the 3-day ban per IP — feasible via VPNs, proxies, or cloud instances

**Attack loop:**
1. Connect as relay peer
2. Send `RelayTransactionHashes([T.hash])`
3. Receive `GetRelayTransactions([T.hash])`
4. Send `RelayTransactions([{cycles: actual+1, transaction: T}])`
5. Victim runs full CKB-VM → detects mismatch → bans peer IP → removes T.hash from tx_filter
6. Reconnect from new IP → repeat (UTXO is never consumed; T.hash was removed from filter)

The UTXO is never consumed (tx is always rejected), so one UTXO suffices indefinitely. Cost scales with number of IPs, not CKB holdings.

## Recommendation
Populate `txs_verify_cache` on a `DeclaredWrongCycles` rejection as well as on success. Subsequent resubmissions of the same witness would then take the cheap `TimeRelativeTransactionVerifier` path, eliminating the repeated full-VM cost. Alternatively, move a plausibility check on `declared_cycles` before `verify_rtx` (e.g., reject if `declared_cycles` differs from a prior cached result), or add rate-limiting per UTXO input on the verify queue.

## Proof of Concept
```
1. Attacker owns UTXO U, creates valid secp256k1-signed tx T (actual cycles = C)
2. Connect to victim node as relay peer
3. Send RelayTransactionHashes([T.hash])
4. Receive GetRelayTransactions([T.hash])
5. Send RelayTransactions([{cycles: C+1, transaction: T}])
   → victim sets max_cycles = C+1, runs full secp256k1 VM (C cycles of CPU)
   → verified.cycles = C, declared = C+1 → DeclaredWrongCycles
   → victim bans attacker IP, removes T.hash from tx_filter
   → cache NOT populated
6. Reconnect from new IP → goto step 3
   (UTXO U is still unspent; T.hash is no longer filtered)
```

Confirmed by integration test `DeclaredWrongCycles` which uses `ALWAYS_SUCCESS_SCRIPT_CYCLE + 1` (i.e., `actual + 1`) and asserts the tx is rejected after full verification, and by `DeclaredWrongCyclesAndRelayAgain` which confirms the tx_hash is removed from the filter and the tx can be re-relayed after reconnection. [8](#0-7)

    crate::setup!(num_nodes: 3);

    fn run(&self, nodes: &mut Vec<Node>) {
        let node0 = &nodes[0];
        let node1 = &nodes[1];
        let node2 = &nodes[2];
        node0.mine_until_out_bootstrap_period();
        out_ibd_mode(nodes);

        let mut net = Net::new(
            self.name(),
            node0.consensus(),
            vec![SupportProtocols::RelayV3],
        );

        let tx = node0.new_transaction_spend_tip_cellbase();
        // relay tx to node0 with wrong cycles
        net.connect(node0);
        relay_tx(&net, node0, tx.clone(), ALWAYS_SUCCESS_SCRIPT_CYCLE + 1);
        let ret = wait_until(10, || node0.rpc_client().get_peers().is_empty());
        assert!(
            ret,
            "The address of net should be removed from node0's peers",
        );
        // connect node0 and node2, make sure node0's relay tx hash processing is working
        node0.rpc_client().clear_banned_addresses();
        node0.connect(node2);
        // removing invalid tx hash from node0's known tx filer is async, wait 5 seconds to make sure it's removed
        sleep(5);

        // connect node0 with node1, tx will be relayed from node1 to node0
        node0.connect(node1);

        // relay tx to node1 with correct cycles
        net.connect(node1);
        relay_tx(&net, node1, tx, ALWAYS_SUCCESS_SCRIPT_CYCLE);

        let result = wait_until(5, || {
            let tx_pool_info = node0.get_tip_tx_pool_info();
            tx_pool_info.orphan.value() == 0 && tx_pool_info.pending.value() == 1
        });
        assert!(
            result,
            "Tx with wrong cycles should be relayed again with correct cycle"
        );
    }
