### Title
Entropy Provider Can Front-Run User Requests via Immediate Fee Update — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`setProviderFee` and `setProviderFeeAsFeeManager` in `Entropy.sol` take effect immediately with no timelock or delay. A malicious Entropy provider (or their fee manager) can front-run a pending user `requestV2` / `requestWithCallback` transaction by raising `feeInWei` to an arbitrarily high value, causing the user's transaction to revert with `InsufficientFee`. In the `Echo.sol` variant, the impact escalates to direct fund extraction: because `req.fee` stores the full `msg.value − pythFeeInWei` (not just the required fee), a provider who raises their fee to match a user's `msg.value` buffer captures the entire overpayment.

---

### Finding Description

**Root cause — `Entropy.sol`**

`setProviderFee` writes the new fee directly to storage with no delay:

```solidity
// Entropy.sol line 810-827
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;          // ← immediate, unbounded
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei, bytes(""));
}
``` [1](#0-0) 

The fee manager variant (`setProviderFeeAsFeeManager`) has the same property: [2](#0-1) 

When a user calls `requestV2`, `requestHelper` reads the **current** fee at execution time:

```solidity
// Entropy.sol line 234-235
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [3](#0-2) 

There is no upper bound on `newFeeInWei` and no timelock. A provider can set `feeInWei = type(uint128).max` in the same block as a user's pending request.

**Escalated impact — `Echo.sol`**

`Echo.requestPriceUpdatesWithCallback` stores the **entire** user payment (minus the fixed Pyth protocol fee) in `req.fee`:

```solidity
// Echo.sol line 84
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [4](#0-3) 

When `executeCallback` is called, the provider is credited the full stored amount:

```solidity
// Echo.sol line 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [5](#0-4) 

`setProviderFee` in Echo also takes effect immediately: [6](#0-5)

### Impact Explanation

**Entropy.sol — DoS:**
A malicious provider (or compromised fee manager) front-runs any pending `requestV2` / `requestWithCallback` by calling `setProviderFee(type(uint128).max)`. Every in-flight user request reverts with `InsufficientFee`. The provider can sustain this indefinitely, permanently denying service to users of that provider. Users lose gas on failed transactions and cannot obtain randomness.

**Echo.sol — Fund extraction:**
If a user sends `msg.value = advertised_fee + buffer` (a common defensive pattern given the documented warning that fees can change), the provider front-runs with `setProviderFee` to raise their fee to exactly `msg.value − pythFeeInWei`. The user's transaction succeeds (fee check passes), but `req.fee` captures the full buffer. The provider collects the entire overpayment when executing the callback. The user has no recourse and no refund path.

---

### Likelihood Explanation

Low-to-medium. Any registered Entropy/Echo provider can execute this attack — provider registration is permissionless. The fee manager role (delegated via `setFeeManager`) adds a second attack surface: a compromised or malicious fee manager key is sufficient. On chains with a public mempool (all current EVM deployments), front-running is straightforward. The attack is detectable on-chain but not preventable by the user once the provider is malicious.

---

### Recommendation

1. **Introduce a fee-update delay / timelock**: require that a new fee is announced N blocks (or seconds) before it takes effect, giving users time to observe the change before submitting requests.
2. **Cap the maximum fee**: enforce an upper bound on `feeInWei` to prevent griefing via astronomically high fees.
3. **Refund excess value**: in `requestHelper`, refund `msg.value − requiredFee` to the caller so that users who send a buffer are not exposed to extraction.
4. **Echo-specific**: store only the `requiredFee` in `req.fee` (not the full `msg.value − pythFeeInWei`), and refund the difference to the requester immediately.

---

### Proof of Concept

```
// Entropy DoS scenario (Foundry pseudocode)
1. provider1 registers with feeInWei = 100 wei
2. user1 calls getFeeV2(provider1, 0) → 100 wei
3. user1 submits requestV2{value: 100 wei}(provider1, userContribution, 0)
   [tx is in mempool]
4. provider1 (or fee manager) front-runs:
       random.setProviderFee(type(uint128).max)
5. user1's requestV2 executes:
       requiredFee = getFeeV2(provider1, 0) = type(uint128).max
       msg.value (100 wei) < requiredFee → revert InsufficientFee
6. user1 loses gas; randomness request fails.

// Echo fund-extraction scenario
1. provider registers with baseFeeInWei = 100 wei, feePerGasInWei = 1 wei
2. user calls getFee(provider, gasLimit, priceIds) → 500 wei
3. user submits requestPriceUpdatesWithCallback{value: 1000 wei}(...)
   [user sent 2× as buffer]
4. provider front-runs:
       echo.setProviderFee(provider, 900 wei, ...) // raises to match msg.value
5. user's tx executes:
       requiredFee = getFee(...) = 1000 wei  (new fee)
       msg.value (1000 wei) >= requiredFee → tx succeeds
       req.fee = 1000 - pythFeeInWei  // full user payment stored
6. provider calls executeCallback → credited req.fee
   Provider extracted 500 wei extra from user with no recourse.
```

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();

        uint128 oldFeeInWei = providerInfo.feeInWei;
