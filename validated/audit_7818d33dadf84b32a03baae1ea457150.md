### Title
Non-Updatable Governance Data Source in `Executor` Permanently Bricks Contract and Locks ETH - (File: `target_chains/ethereum/contracts/contracts/executor/Executor.sol`)

---

### Summary

The `Executor` contract sets `ownerEmitterChainId` and `ownerEmitterAddress` at initialization with no mechanism to update them. The contract's own source code contains a `TODO` acknowledging this gap. Because the upgrade path in `ExecutorUpgradable` is itself gated by a valid VAA from the same frozen emitter, a governance data source transfer permanently bricks the Executor and locks any ETH it holds.

---

### Finding Description

In `Executor._initialize()`, two private state variables are written once and never again:

```solidity
// Executor.sol lines 44-45
uint16 private ownerEmitterChainId;
bytes32 private ownerEmitterAddress;
``` [1](#0-0) 

Every call to `execute()` routes through `verifyGovernanceVM()`, which hard-checks these two values:

```solidity
if (
    vm.emitterChainId != ownerEmitterChainId ||
    vm.emitterAddress != ownerEmitterAddress
) revert ExecutorErrors.UnauthorizedEmitter();
``` [2](#0-1) 

The `ExecutorAction` enum has only one member (`Execute`) and the code itself acknowledges the missing setter with a `TODO`:

```solidity
enum ExecutorAction {
    // TODO: add an instruction to change the governance data source.
    Execute // 0
}
``` [3](#0-2) 

The upgrade path in `ExecutorUpgradable.upgradeTo` is also gated by a valid governance VAA from the same frozen emitter:

```solidity
function upgradeTo(address newImplementation) external override onlyProxy {
    address oldImplementation = _getImplementation();
    _authorizeUpgrade(newImplementation);
    _upgradeToAndCallUUPS(newImplementation, new bytes(0), false);
    ...
}
``` [4](#0-3) 

`_authorizeUpgrade` calls `onlyOwner`, and the owner is the contract itself (`_transferOwnership(address(this))`), meaning the only way to authorize an upgrade is through `execute()` — which requires a valid VAA from the frozen emitter. This creates an unbreakable circular dependency.

The contract also has an unconditional ETH receiver:

```solidity
receive() external payable {}
``` [5](#0-4) 

---

### Impact Explanation

If the Pyth governance emitter is ever transferred to a new chain/address (a supported protocol operation via `AuthorizeGovernanceDataSourceTransfer` in `PythGovernance`), the Executor becomes permanently non-functional:

1. All calls to `execute()` revert with `UnauthorizedEmitter`.
2. All calls to `upgradeTo` / `upgradeToAndCall` also revert, because they require a valid `execute()` call.
3. Any ETH held in the Executor (sent via `receive()` or as `value` in governance calls) is permanently locked with no recovery path. [6](#0-5) 

---

### Likelihood Explanation

The Pyth EVM governance contract explicitly supports governance data source transfers (`AuthorizeGovernanceDataSourceTransfer`). [7](#0-6)  If such a transfer is executed for the emitter that the Executor trusts, the Executor is immediately and irrecoverably bricked. This is a realistic operational scenario, not a theoretical one.

---

### Recommendation

Add a new `ExecutorAction` variant (e.g., `UpdateGovernanceDataSource`) that allows updating `ownerEmitterChainId` and `ownerEmitterAddress` via a VAA signed by the **current** emitter before the transfer completes — mirroring the pattern used in `PythGovernance.setWormholeAddress`, which verifies the new address can parse the same VAA used to set it. [8](#0-7) 

---

### Proof of Concept

1. `ExecutorUpgradable` is deployed; `ownerEmitterChainId = 1`, `ownerEmitterAddress = 0xABCD...`.
2. Pyth governance issues an `AuthorizeGovernanceDataSourceTransfer` VAA, moving the governance emitter to chain `2`, address `0x1234...`.
3. Any subsequent VAA from `(2, 0x1234...)` fails `verifyGovernanceVM` → `UnauthorizedEmitter`.
4. Attempting `upgradeTo(newImpl)` also fails: it calls `execute()` internally, which calls `verifyGovernanceVM`, which reverts.
5. ETH previously sent to the Executor via `receive()` or accumulated during governance calls is permanently locked — no withdrawal function exists. [9](#0-8)

    function verifyGovernanceVM(
        bytes memory encodedVM
    ) internal returns (IWormhole.VM memory parsedVM) {
        (IWormhole.VM memory vm, bool valid, ) = wormhole.parseAndVerifyVM(
            encodedVM
        );

        if (!valid) revert ExecutorErrors.InvalidWormholeVaa();

        if (
            vm.emitterChainId != ownerEmitterChainId ||
            vm.emitterAddress != ownerEmitterAddress
        ) revert ExecutorErrors.UnauthorizedEmitter();

        if (vm.sequence <= lastExecutedSequence)
            revert ExecutorErrors.MessageOutOfOrder();

        lastExecutedSequence = vm.sequence;

        return vm;
    }
