### Title
Unauthenticated Observed Address Injection via IdentifyProtocol Enables Network-Wide Address Poisoning — (`network/src/protocols/identify/mod.rs`, `network/src/network.rs`)

### Summary

An unprivileged remote peer can send a crafted `IdentifyMessage` containing an arbitrary `observed_addr` (including any globally-routable IP). The victim node stores this address unconditionally in `NetworkState::observed_addrs` and subsequently re-broadcasts it to all newly connected peers as one of its own listen addresses, with no reachability verification or authenticity check.

### Finding Description

**Step 1 — No validation in `process_observed`:**

`process_observed` passes the attacker-supplied `observed_addr` directly to `add_observed_addr` with no content checks, and unconditionally returns `MisbehaveResult::Continue` regardless of what `add_observed_addr` returns (the return value is discarded): [1](#0-0) 

**Step 2 — Unconditional storage in `IdentifyCallback::add_observed_addr`:**

The callback only optionally appends the local peer ID if absent, then stores the address directly into `observed_addrs` with no IP-range, reachability, or consistency check: [2](#0-1) 

**Step 3 — Observed addresses are included in `local_listen_addrs`:**

When the victim node connects to any new peer, it calls `local_listen_addrs()`, which supplements its own public addresses with entries from `observed_addrs` whenever fewer than `MAX_RETURN_LISTEN_ADDRS` (10) addresses are available: [3](#0-2) 

**Step 4 — The `is_reachable` filter in `connected` does NOT block crafted global IPs:**

When broadcasting listen addresses to a new peer, the `connected` handler filters addresses through `is_reachable`. This only rejects private/loopback IPs. A crafted global IP (e.g., pointing to the attacker's own node) passes this filter unconditionally: [4](#0-3) 

**Step 5 — `observed_addrs` is not cleaned up on disconnect:**

The `unregister` callback (called on `disconnected`) does not remove the session's entry from `observed_addrs`, so the poisoned address persists after the attacker disconnects: [5](#0-4) 

**Step 6 — `observed_addrs` field in `NetworkState`:**

The field is a `HashMap<PeerIndex, Multiaddr>` with no eviction or validation logic at the storage layer: [6](#0-5) 

### Impact Explanation

The victim node advertises the attacker-controlled address as its own listen address to every peer it subsequently connects to. Those peers store it in their peer stores and may further propagate it via the discovery protocol. This enables:

- **Address poisoning**: The victim's identity in the network is partially hijacked.
- **Eclipse attack facilitation**: Peers attempting to connect to the victim may instead reach the attacker's node.
- **Topology manipulation**: The attacker can redirect traffic and degrade the victim's connectivity, affecting block/transaction relay and ultimately CKB economic security.

### Likelihood Explanation

- Requires only a single open P2P connection to the victim (inbound or outbound).
- No special privileges, keys, or majority hashpower needed.
- The crafted address persists after disconnect.
- Effective when the victim has fewer than 10 configured public addresses (common for nodes without explicit `public_addresses` config).

### Recommendation

1. **Validate `observed_addr` against the actual session's remote address**: Only accept an `observed_addr` if it matches the IP of the session that reported it (i.e., the peer can only report what it actually sees as the source IP of the connection).
2. **Apply `is_reachable` at insertion time** in `add_observed_addr`, not only at broadcast time.
3. **Remove `observed_addrs` entries on session disconnect** in the `unregister`/`disconnected` handler.
4. **Require quorum**: Only promote an observed address to `local_listen_addrs` if multiple independent peers report the same address.

### Proof of Concept

1. Attacker node connects to victim (or accepts a connection from victim).
2. Attacker sends an `IdentifyMessage` with `observed_addr` set to `<attacker_global_ip>/tcp/<port>`.
3. Victim calls `process_observed` → `IdentifyCallback::add_observed_addr` → `NetworkState::add_observed_addr`, storing the crafted address.
4. Victim subsequently connects to peer C and calls `local_listen_addrs()`, which includes the crafted address (since it passes `is_reachable`).
5. Peer C receives the crafted address as the victim's listen address and stores/propagates it.
6. Verify: monitor peer C's peer store or its outgoing identify messages to confirm the crafted address appears.

    fn unregister(&self, context: &ProtocolContextMutRef) {
        if context.session.ty.is_outbound() {
            // Due to the filtering strategy of the peer store, if the node is
            // disconnected after a long connection is maintained for more than seven days,
            // it is possible that the node will be accidentally evicted, so it is necessary
            // to reset the last_connected_time of the node when disconnected.
            self.network_state.with_peer_store_mut(|peer_store| {
                peer_store.update_outbound_addr_last_connected_ms(context.session.address.clone());
            });
        }
    }

        let mut listens = self.listen_addrs();

        if listens.len() < MAX_RETURN_LISTEN_ADDRS {
            let observe_addrs = self
                .network_state
                .observed_addrs(MAX_RETURN_LISTEN_ADDRS - listens.len());
            listens.extend(observe_addrs);
            listens
        } else {
            listens
        }
