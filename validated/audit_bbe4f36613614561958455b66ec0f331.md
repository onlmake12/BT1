Audit Report

## Title
Non-Atomic RBF Allows Permanent Mempool Eviction via Cell-Dep Ancestor Overflow Without Rollback — (`tx-pool/src/process.rs`, `tx-pool/src/component/pool_map.rs`)

## Summary
`submit_entry` in `process.rs` calls `process_rbf` to unconditionally remove conflicting transactions before calling `_submit_entry`. If `_submit_entry` subsequently fails with `ExceededMaximumAncestorsCount` — triggered by the incoming tx referencing a deep in-pool chain via a `cell_dep` — no rollback occurs. The FIXME comment at `pool_map.rs:582–587` incorrectly claims this path is unreachable because it only considers input-based ancestry, while `get_tx_ancenstors` also counts `cell_dep` parents. An unprivileged attacker can exploit this to permanently evict targeted transactions from the mempool without inserting a replacement.

## Finding Description

**Non-atomic ordering in `submit_entry`:**

`process.rs:136–137` calls `process_rbf` before `_submit_entry`. If `_submit_entry` returns `Err(...)`, the `?` operator propagates the error but the conflicts removed by `process_rbf` are never re-inserted:

```rust
let may_recovered_txs = self.process_rbf(tx_pool, &entry, &conflicts);
let evicted = _submit_entry(tx_pool, status, entry.clone(), &self.callbacks)?;
```

`process_rbf` at `process.rs:203–206` unconditionally removes all conflicting transactions and their descendants via `remove_entry_and_descendants`.

**The FIXME's flawed reasoning:**

`pool_map.rs:582–587` claims the failure path is unreachable because RBF blocks unknown inputs, so the new tx "can not be in a long transaction chain." This reasoning is incomplete: `get_tx_ancenstors` at `pool_map.rs:541–547` adds any in-pool transaction referenced via `cell_dep` to `parents`, not just input-based parents:

```rust
for cell_dep in entry.cell_deps() {
    let dep_pt = cell_dep.out_point();
    let id = ProposalShortId::from_tx_hash(&dep_pt.tx_hash());
    if self.links.inner.contains_key(&id) {
        parents.insert(id);
    }
}
```

`calc_relation_ids` then recursively traverses all ancestors of those parents, so a single `cell_dep` pointing to the tail of a 26-tx chain yields `ancestors_count = 27`.

**The RBF cell_dep check is insufficient:**

`pool.rs:648–660` only rejects cell_deps pointing to `all_conflicted` (the conflicting tx and its descendants). A cell_dep pointing to an unrelated in-pool chain (e.g., `tx_26`) is not blocked.

**The `calc_ancestors` disjointness check is a no-op for the new tx:**

`pool.rs:615` calls `self.pool_map.calc_ancestors(&short_id)` for the incoming tx. Since the tx is not yet in the pool, `links.inner.get(short_id)` returns `None`, `unwrap_or_default()` yields an empty set, and the disjointness check at `pool.rs:626` never fires for cell_dep-based ancestry.

**The error path:**

`pool_map.rs:626–628` returns `Err(Reject::ExceededMaximumAncestorsCount)` when `ancestors_count - cell_ref_parents.len() > max_ancestors_count`, with no rollback of already-removed conflicts. In the attack scenario, `cell_ref_parents` is empty (no in-pool tx has a `cell_dep` on the confirmed input O), so the eviction path at `pool_map.rs:603–625` is not taken.

**Attack construction:**

1. Build chain `tx_1 → tx_2 → ... → tx_26` in the pool (each spending the previous output). `tx_26` has exactly 25 in-pool ancestors.
2. Submit `conflict_tx` spending confirmed output `O`.
3. Submit RBF tx: `input = O` (confirmed, satisfies "no unknown inputs"), `cell_dep = output of tx_26` (not in `all_conflicted`, passes the cell_dep check), fee sufficient to pass RBF fee rules.
4. `check_rbf` passes all rules.
5. `process_rbf` removes `conflict_tx` and its descendants.
6. `_submit_entry` → `check_and_record_ancestors`: `get_tx_ancenstors` finds `tx_26` as a cell_dep parent; `ancestors = {tx_1, …, tx_26}`; `ancestors_count = 27`; `cell_ref_parents = {}`; `27 - 0 = 27 > 25` → `Err(ExceededMaximumAncestorsCount)`.
7. `conflict_tx` is permanently gone. The RBF tx is not inserted. No rollback occurs.

The `tx_1`–`tx_26` chain remains intact and can be reused for subsequent evictions.

## Impact Explanation

An unprivileged attacker can permanently remove targeted transactions from the mempool without inserting a replacement. This silently drops transactions (including high-fee or CPFP packages), directly damaging miner revenue and disrupting transaction finality. The attack is repeatable at low marginal cost after the initial chain setup. This matches **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, and potentially **Critical — Vulnerabilities which could easily damage CKB economy**, given that targeted, repeatable mempool eviction undermines miner incentives and transaction reliability.

## Likelihood Explanation

No special privileges are required. The attack is executable via the standard `send_transaction` RPC or P2P relay. The setup (a chain of 26 transactions) is inexpensive and straightforward. The attack is deterministic and repeatable, with the `tx_1`–`tx_26` chain reusable across multiple eviction attempts. The only cost to the attacker is the fee for the RBF tx itself, which is never inserted into the pool.

## Recommendation

Move the ancestor-count pre-check **before** `process_rbf`, as the FIXME comment itself suggests. In `submit_entry` (`process.rs`), call `get_tx_ancenstors` on the new tx and validate that `ancestors_count ≤ max_ancestors_count` before invoking `process_rbf`. This ensures the ancestor limit is enforced atomically before any pool state is mutated. Alternatively, implement a full rollback of removed transactions if `_submit_entry` fails, re-inserting them into the pool with their original metadata.

## Proof of Concept

1. Configure a node with default `max_ancestors_count = 25`.
2. Submit `tx_1 → tx_2 → ... → tx_26` (chain of 26 txs, each spending the previous output). Confirm all are in the pool.
3. Submit `conflict_tx` spending confirmed output `O`. Confirm it is in the pool.
4. Submit RBF tx: `input = O`, `cell_dep = output of tx_26`, fee > `min_replace_fee` for `conflict_tx`.
5. Assert: `conflict_tx` is absent from the pool; the RBF tx is absent from the pool; `tx_1`–`tx_26` remain in the pool.
6. Repeat steps 3–5 with a new `conflict_tx` to demonstrate repeatability at low cost.
