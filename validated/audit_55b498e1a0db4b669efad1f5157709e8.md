### Title
Keeper Financial Loss via Simulation-Execution Inconsistency in Scheduler `updatePriceFeeds` - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

### Summary
In `Scheduler.sol::updatePriceFeeds()`, a keeper (price pusher) can simulate a transaction successfully but have it revert on-chain due to timestamp-based validity checks evaluated at execution time. The keeper loses gas costs with no compensation, because the revert undoes all state changes including the keeper fee payment.

### Finding Description

`updatePriceFeeds()` performs several timestamp-dependent checks that are evaluated at `block.timestamp` during actual execution, not at simulation time:

1. **`parsePriceFeedUpdatesWithConfig`** uses `curTime + FUTURE_TIMESTAMP_MAX_VALIDITY_PERIOD` (10 seconds) as the max allowed publish time, where `curTime = SafeCast.toUint64(block.timestamp)`.

2. **`_validateShouldUpdatePrices`** enforces `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hour` — if the highest publish timestamp in the update data is older than `block.timestamp - 1 hour`, it reverts with `TimestampTooOld`.

3. **`_validateShouldUpdatePrices`** also reverts with `TimestampOlderThanLastUpdate` if `updateTimestamp <= status.priceLastUpdatedAt` — i.e., if another keeper already submitted a newer update between simulation and inclusion.

4. **`_validateShouldUpdatePrices`** reverts with `UpdateConditionsNotMet` if neither heartbeat nor deviation criteria are satisfied at execution time.

The execution flow is:

```
updatePriceFeeds()
  ├── status.balanceInWei -= pythFee  ← deducted early (line 305)
  ├── parsePriceFeedUpdatesWithConfig{value: pythFee}(...)  ← uses block.timestamp
  ├── _validateShouldUpdatePrices(...)  ← can revert on timing
  └── _processFeesAndPayKeeper(...)  ← keeper only paid if no revert
```

If any of the timestamp checks fail, the entire transaction reverts. The keeper receives zero compensation but has already spent gas executing the full function body (including the expensive `parsePriceFeedUpdatesWithConfig` call to the Pyth contract).

A deliberate attack path: an unprivileged actor monitors the mempool for a keeper's `updatePriceFeeds` transaction, front-runs it with their own `updatePriceFeeds` call (using the same or a newer price update blob, with higher gas price). The attacker's transaction succeeds and collects the keeper fee. The original keeper's transaction then reverts with `TimestampOlderThanLastUpdate` because `updateTimestamp <= status.priceLastUpdatedAt`, and the keeper loses gas.

### Impact Explanation

The keeper (price pusher) loses the gas cost of a failed `updatePriceFeeds` transaction. The gas cost is non-trivial: it includes the overhead of `parsePriceFeedUpdatesWithConfig` (Merkle proof verification, VAA parsing) plus all the validation logic. At high gas prices or with many price feeds, this can be a meaningful loss per failed transaction. Repeated front-running attacks can drain a keeper's operating budget.

**Impact: Medium** — direct financial loss to keepers; no protocol funds are at risk, but keeper sustainability is undermined.

### Likelihood Explanation

The `TimestampOlderThanLastUpdate` path is reachable whenever two keepers race to submit the same update. In competitive keeper networks this is a normal occurrence. The `TimestampTooOld` path is reachable when a keeper fetches price data near the 1-hour boundary and the transaction is delayed in the mempool. Both paths require no privileged access.

**Likelihood: Medium** — race conditions between keepers are common; deliberate front-running requires only mempool visibility.

### Recommendation

1. **Pre-flight staleness check**: Before executing the expensive `parsePriceFeedUpdatesWithConfig` call, add a view-level check that the price data's publish timestamp (extractable from `updateData` headers) is within the validity window. This allows keepers to detect stale data cheaply before committing gas.

2. **Refund partial gas on benign reverts**: Consider separating the Pyth fee deduction from the keeper fee payment so that if `_validateShouldUpdatePrices` reverts, the keeper is at least partially compensated for the gas spent on `parsePriceFeedUpdatesWithConfig`.

3. **Document the race condition risk**: Explicitly warn keepers in NatSpec that `updatePriceFeeds` may revert if another keeper submits a concurrent update, and recommend using `eth_call` with a recent block to re-validate conditions immediately before submission.

### Proof of Concept

1. Keeper A fetches fresh price data (publish time = `T`, where `T` is within the last hour).
2. Keeper A simulates `updatePriceFeeds(subscriptionId, updateData)` — simulation succeeds because `T > block.timestamp - 1 hour` and heartbeat condition is met.
3. Keeper B (attacker) observes Keeper A's pending transaction in the mempool.
4. Keeper B submits `updatePriceFeeds(subscriptionId, updateData)` with higher `gasPrice`, using the same or a newer price blob.
5. Keeper B's transaction is included first; `status.priceLastUpdatedAt` is set to `T` (or `T' > T`).
6. Keeper A's transaction is included next. `_validateShouldUpdatePrices` reaches:

```solidity
if (
    status.priceLastUpdatedAt > 0 &&
    updateTimestamp <= status.priceLastUpdatedAt  // T <= T (or T <= T')
) {
    revert SchedulerErrors.TimestampOlderThanLastUpdate(...);
}
```

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
