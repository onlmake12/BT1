### Title
Permissionless `Initialize` Accepts Zero Authority, Permanently Bricking `InitializePublisher` — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`, `accounts/config.rs`, `validate.rs`)

---

### Summary

The `Initialize` instruction is callable by any unprivileged signer and accepts an arbitrary `authority` value — including `[0u8; 32]` — with no validation. Because the config PDA can only be initialized once, an attacker who calls `Initialize` first with a zero authority permanently prevents any future `InitializePublisher` call from succeeding, since `validate_authority` requires a signer whose key matches `config.authority`, and the all-zeros pubkey has no known private key.

---

### Finding Description

**`config::create` performs no validation on the authority value:** [1](#0-0) 

Any `[u8; 32]` — including all-zeros — is accepted and written directly into the config account.

**`Initialize` is permissionless — only a generic signer/writable check on the payer:** [2](#0-1) 

There is no check that the caller is the program upgrade authority or any other privileged account.

**`config::create` enforces a one-time-only invariant via `AlreadyInitialized`:** [3](#0-2) 

Once the config PDA is created, it can never be re-initialized.

**`validate_authority` requires a signer whose key bytes exactly match `config.authority`:** [4](#0-3) 

If `config.authority == [0u8; 32]`, the `is_signer` check on line 71 and the key-equality check on line 77 can never both pass simultaneously — no account can sign as the all-zeros pubkey.

**`InitializePublisher` calls `validate_authority` unconditionally:** [5](#0-4) 

---

### Impact Explanation

With `config.authority = [0u8; 32]`, every call to `InitializePublisher` returns `MissingRequiredSignature` permanently. No publisher can ever be registered, and the price-store program is rendered inoperable for its core function. The config PDA cannot be re-created (it already exists and `AlreadyInitialized` is returned), and there is no admin-override or re-initialization path in the program.

---

### Likelihood Explanation

The `Initialize` instruction has no access control beyond requiring a generic signer. Any unprivileged actor who submits the transaction before the legitimate operator does — a race condition at deployment time — can set the zero authority. On Solana this requires either being a validator or monitoring the transaction pipeline, but the window exists between program deployment and the first `Initialize` call. Additionally, an operator mistake (accidentally passing a zero pubkey) produces the same permanent outcome with no recovery path.

---

### Recommendation

1. **Reject zero authority in `config::create`**: add `if authority == [0u8; 32] { return Err(...) }` before writing.
2. **Add access control to `Initialize`**: require the payer to be the program's upgrade authority (verifiable via the `BpfUpgradeableLoader` program data account), so only the deployer can call it.

---

pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);
    }
    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
    if data.format != 0 {
        return Err(ReadAccountError::AlreadyInitialized);
    }
    data.format = FORMAT;
    data.authority = authority;
    Ok(data)
