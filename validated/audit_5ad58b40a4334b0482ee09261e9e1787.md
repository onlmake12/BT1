Audit Report

## Title
Memory Amplification via High-Input-Count Orphan Transactions in `OrphanPool` — (`tx-pool/src/component/orphan.rs`)

## Summary
`OrphanPool::limit_size()` caps `entries` at `DEFAULT_MAX_ORPHAN_TRANSACTIONS` (100) but the corresponding `by_out_point` index is bounded only by `entries.len() × inputs_per_tx`. An attacker can relay 100 maximally-sized orphan transactions (each with ~10,600 inputs referencing distinct unknown out-points), keeping `entries.len() == 100` while `by_out_point.len()` reaches ~1,060,000 entries — a ~10,600× amplification. Combined with the raw transaction data (~51MB), total orphan pool memory reaches ~160MB from a single unauthenticated peer.

## Finding Description
`OrphanPool` maintains two maps: `entries` (one entry per transaction) and `by_out_point` (one entry per input across all orphan transactions). `len()` returns only `self.entries.len()`, and `limit_size()` evicts based solely on this count. `add_orphan_tx` inserts one `by_out_point` entry per input with no per-transaction input count limit. While `remove_orphan_tx` does clean up `by_out_point` on eviction, the steady-state bound is `DEFAULT_MAX_ORPHAN_TRANSACTIONS × max_inputs_per_tx`, not `DEFAULT_MAX_ORPHAN_TRANSACTIONS`. The exploit path is: remote peer sends `SendTransaction` → `after_process` → `is_missing_input` returns true (tx fails with `OutPointError::Unknown`) → `add_orphan` → `add_orphan_tx`. The only pre-admission check is `non_contextual_verify`, which rejects only if `tx_size > TRANSACTION_SIZE_LIMIT` (512KB) or if the transaction is a cellbase. A transaction with ~10,600 inputs referencing distinct unknown out-points fits within 512KB and passes this check trivially.

## Impact Explanation
100 orphan transactions × ~10,600 inputs = ~1,060,000 `by_out_point` entries (~108MB) plus ~51MB for raw transaction data ≈ **~160MB** from one attacker. Multiple coordinated peers multiply this linearly. Under sustained attack this causes severe memory pressure or OOM, crashing the node. This matches the **High** impact class: "Vulnerabilities which could easily crash a CKB node."

## Likelihood Explanation
The attack requires no privileges, no PoW, no keys, and no on-chain state. Any P2P relay peer can send `SendTransaction` messages. Constructing transactions with thousands of inputs referencing unknown out-points is trivial. The attacker needs no persistent state — each batch of 100 transactions can be sent in a tight loop, continuously replacing evicted orphans and sustaining memory pressure indefinitely.

## Recommendation
Enforce a per-transaction input count limit before admission to the orphan pool (e.g., reject if `tx.inputs().len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS` or some proportional threshold). Alternatively, track `by_out_point.len()` in `limit_size()` and evict entries when it exceeds a threshold proportional to `DEFAULT_MAX_ORPHAN_TRANSACTIONS`. A byte-based orphan pool size cap (as used by Bitcoin Core) would also address this class of amplification.

## Proof of Concept
```rust
let mut orphan = OrphanPool::new();
for i in 0..100 {
    // Build a tx with ~10_600 inputs, each referencing a distinct unknown OutPoint
    let tx = build_tx_with_n_distinct_inputs(10_600, i as u64);
    orphan.add_orphan_tx(tx, 0.into(), 0);
}
assert_eq!(orphan.entries.len(), 100);           // capped as expected
assert!(orphan.by_out_point.len() > 1_000_000);  // ~10,600× amplification
```
Each `add_orphan_tx` call passes `non_contextual_verify` (size < 512KB, not cellbase), fails resolution with `OutPointError::Unknown`, and is admitted to the orphan pool. `limit_size()` keeps `entries` at 100 but `by_out_point` reaches ~1,060,000 entries, confirming the amplification.

    pub fn remove_orphan_tx(&mut self, id: &ProposalShortId) -> Option<Entry> {
        self.entries.remove(id).inspect(|entry| {
            debug!("remove orphan tx {}", entry.tx.hash());
            for out_point in entry.tx.input_pts_iter() {
                if let Some(ids_set) = self.by_out_point.get_mut(&out_point) {
                    ids_set.remove(id);

                    if ids_set.is_empty() {
                        self.by_out_point.remove(&out_point);
                    }
                }
            }
        })
