Audit Report

## Title
Unvalidated `i32::MAX` OFFSET in `get_tx_with_cell` Enables DB Exhaustion via Crafted Pagination Cursor — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC endpoint (with `group_by_transaction=false` or omitted) decodes the `after` cursor's last 4 bytes as an `i32` offset with no upper-bound validation. An unprivileged caller can supply a cursor encoding `i32::MAX` (2,147,483,647), which is passed directly as a SQL `OFFSET`, forcing the database to sequentially scan and discard up to 2.1 billion rows per request. With no per-query timeout on the async rich-indexer path, repeated requests exhaust DB resources and render the node's RPC unresponsive.

## Finding Description
**Cursor parsing** (`get_transactions.rs` L44–53): the only validation is that `after` is exactly 12 bytes. The last 4 bytes are decoded via `decode_i32`, which only checks length — no range check is applied:

```rust
let offset = decode_i32(offset)?;   // accepts any i32, including i32::MAX
last_cursor = Some((last, offset));
``` [1](#0-0) 

`decode_i32` in `mod.rs` L286–294 confirms: only a length check, then `i32::from_le_bytes` — no bounds enforcement. [2](#0-1) 

**Offset injection** (`get_transactions.rs` L180–182): the decoded offset is passed directly to the SQL query builder:

```rust
if let Some((_, offset)) = last_cursor {
    query_builder.offset(offset);
}
``` [3](#0-2) 

This produces: `SELECT ... LIMIT 100 OFFSET 2147483647`, forcing a full sequential scan.

**`request_limit` guard** (`get_transactions.rs` L23–32) only caps the `limit` parameter, not the offset: [4](#0-3) 

**No timeout field** on `AsyncRichIndexerHandle` (`mod.rs` L22–27) — the struct carries only `store`, `pool`, and `request_limit`: [5](#0-4) 

**`async_handle()` construction** (`service.rs` L97–99) passes only `request_limit`, confirming no timeout is wired in: [6](#0-5) 

## Impact Explanation
Each crafted request forces the DB engine (SQLite by default, or PostgreSQL) to scan and discard up to 2,147,483,647 rows before returning. On a mainnet-populated rich-indexer DB, this consumes seconds to minutes of CPU and I/O per request. SQLite is single-writer; a long-running read query blocks the indexer sync loop and all concurrent RPC queries. Multiple concurrent crafted requests saturate the DB connection pool, causing all RPC calls to time out or fail — effectively crashing the node's RPC service. This matches: **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The rich-indexer RPC is unauthenticated and publicly accessible by default. The 12-byte cursor format is documented. Constructing the malicious cursor requires no key material, PoW, or privileged access — only knowledge of the little-endian encoding. The attack is stateless and infinitely repeatable from a single IP.

## Recommendation
Add an upper-bound check on the decoded offset immediately after `decode_i32` in `get_transactions.rs`, before constructing `last_cursor`:

```rust
if offset < 0 || offset as usize > self.request_limit {
    return Err(Error::Params("cursor offset out of range".to_string()));
}
```

Additionally, introduce a `timeout_limit` field on `AsyncRichIndexerHandle` (analogous to the non-async `IndexerHandle` path in `util/indexer/src/service.rs`) and wrap DB queries with a timeout future to bound worst-case query duration.

## Proof of Concept
Craft a 12-byte `after` cursor: 8 zero bytes (tx_id = 0) followed by `[0xff, 0xff, 0xff, 0x7f]` (offset = `i32::MAX` in little-endian):

```python
import json, requests, time

after_cursor = "0x" + "00" * 8 + "ffffff7f"
payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "get_transactions",
    "params": [
        {
            "script": {
                "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
                "hash_type": "type",
                "args": "0x"
            },
            "script_type": "lock",
            "group_by_transaction": False
        },
        "asc", "0x1", after_cursor
    ]
}
t0 = time.time()
r = requests.post("http://localhost:8114", json=payload)
print(f"elapsed: {time.time()-t0:.2f}s")
```

Compare elapsed time against a benign cursor (`"0x" + "00" * 12`). The crafted cursor will produce query time proportional to the full table size. Sending this request in a tight loop from multiple threads will exhaust the DB connection pool and block all subsequent RPC calls.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
