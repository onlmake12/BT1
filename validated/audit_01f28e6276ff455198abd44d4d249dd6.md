### Title
Tx-Pool `total_tx_size`/`total_tx_cycles` Accounting Corrupted When Ancestor-Eviction Occurs During `add_entry` - (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new accounting totals (`total_tx_size`, `total_tx_cycles`) are computed **before** a potential in-place eviction of ancestor transactions, then unconditionally written **after** those evictions have already modified the same fields. The evictions' subtractions are silently overwritten, leaving `total_tx_size` and `total_tx_cycles` permanently inflated by the sizes and cycles of the evicted entries.

---

### Finding Description

`PoolMap::add_entry` follows this sequence:

1. Compute candidate new totals: `(total_tx_size, total_tx_cycles) = updated_stat_for_add_tx(entry.size, entry.cycles)` — this is simply `self.total_tx_size + entry.size` and `self.total_tx_cycles + entry.cycles`.
2. Call `check_and_record_ancestors(&mut entry)`. When the incoming transaction's ancestor count exceeds `max_ancestors_count` but can be reduced by evicting low-fee "cell-ref parent" transactions, this function calls `remove_entry_and_descendants` one or more times. Each call chains into `remove_entry`, which calls `update_stat_for_remove_tx` — correctly **decrementing** `self.total_tx_size` and `self.total_tx_cycles` for every evicted entry.
3. After `check_and_record_ancestors` returns, `add_entry` unconditionally assigns:
   ```rust
   self.total_tx_size = total_tx_size;   // line 218
   self.total_tx_cycles = total_tx_cycles; // line 219
   ```
   These are the stale pre-eviction values from step 1. Every decrement applied in step 2 is discarded.

The net result after one such `add_entry` call:

```
actual pool size  = old_total - Σ(evicted_sizes) + new_entry_size
recorded total    = old_total                    + new_entry_size   ← inflated
```

The inflation equals the sum of all evicted entries' sizes (and cycles). Each subsequent eviction-triggering `add_entry` call compounds the error further. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict further transactions:

```rust
while self.pool_map.total_tx_size > self.config.max_tx_pool_size { … }
``` [5](#0-4) 

An inflated `total_tx_size` causes `limit_size` to believe the pool is over its size budget when it is not, triggering a cascade of unnecessary evictions of otherwise-valid pending or proposed transactions. Each such spurious eviction fires a `Reject` callback, permanently banning those transactions from the pool until resubmitted. Miners lose fee revenue; users experience unexpected transaction drops. Additionally, `updated_stat_for_add_tx` uses the inflated counter to gate admission of new transactions, causing legitimate submissions to be rejected with `Reject::Full` even when real pool capacity is available. [6](#0-5) 

---

### Likelihood Explanation

The eviction path inside `check_and_record_ancestors` is triggered when:
- A new transaction's ancestor count exceeds `max_ancestors_count` (default 25), **and**
- The excess is entirely attributable to "cell-ref parents" (pool transactions that reference the same cell as a dep, not as a consumed input).

An unprivileged RPC caller (`send_transaction`) can deliberately craft a chain of transactions that saturates the ancestor limit and then submit a transaction whose cell-dep overlaps with those ancestors. This is a realistic, low-cost attack requiring no special privilege, no majority hashpower, and no key material.

---

### Recommendation

Move the stat assignment to **after** `check_and_record_ancestors` completes, reading the already-updated `self.total_tx_size`/`self.total_tx_cycles` and adding only the new entry's contribution:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals AFTER evictions have already decremented the fields:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

Alternatively, accumulate the evicted sizes inside `check_and_record_ancestors` and subtract them from the pre-computed `total_tx_size`/`total_tx_cycles` before the final assignment.

---

### Proof of Concept

**Setup** (`max_ancestors_count = 25`, `max_tx_pool_size = 10 MB`):

| Step | Action | Actual pool bytes | `total_tx_size` |
|------|--------|-------------------|-----------------|
| 1 | Submit 25 txs forming a chain; each 1 000 B; last tx also used as cell-dep by a side tx S (500 B) | 25 500 B | 25 500 |
| 2 | Submit new tx T (200 B) whose ancestor chain = those 25 txs → count = 26 > 25; cell-ref parent = S → evict S (500 B + descendants) | 25 200 B | **25 700** (inflated by 500) |
| 3 | Repeat step 2 pattern N times | actual ≪ limit | `total_tx_size` grows unboundedly |
| 4 | `limit_size` fires; evicts legitimate pending txs | pool shrinks unnecessarily | — |

After step 2, `total_tx_size = 25 700` while the real pool holds only `25 200` bytes. The 500-byte eviction of S was applied to `self.total_tx_size` by `update_stat_for_remove_tx`, then immediately overwritten by the stale pre-computed value at lines 218–219. [7](#0-6) [8](#0-7)

    fn check_and_record_ancestors(
        &mut self,
        entry: &mut TxEntry,
    ) -> Result<HashSet<TxEntry>, Reject> {
        let tx = entry.transaction();
        let (ancestors, mut parents, cell_ref_parents) = self.get_tx_ancenstors(tx);

        let mut ancestors_count = ancestors.len() + 1;
        let mut evicted = Default::default();

        if ancestors_count <= self.max_ancestors_count {
            self._record_ancestors(entry, ancestors, parents);
            return Ok(evicted);
        }

        if ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count {
            // if ancestors count exceed limitation,
            // try to evict some conflicted transactions due to ref cells

            // sort them to find out the transactions with lowest fees
            let evict_candidates: Vec<ProposalShortId> = self
                .entries
                .iter_by_evict_key()
                .filter(move |entry| cell_ref_parents.contains(&entry.id))
                .map(|x| x.id.clone())
                .collect();

            let mut iter = evict_candidates.iter();
            while ancestors_count > self.max_ancestors_count {
                if let Some(next_id) = iter.next() {
                    let removed = self.remove_entry_and_descendants(next_id);
                    ancestors_count = ancestors_count.saturating_sub(1);
                    parents.remove(next_id);
                    evicted.extend(removed);
                } else {
                    break;
                }
            }
        } else {
            return Err(Reject::ExceededMaximumAncestorsCount);
        }

        // some txs in `parents` are removed, now `ancestors` need to re-caculate,
        let ancestors = self
            .links
            .calc_relation_ids(parents.clone(), Relation::Parents);

        // we can assume the number now is less than `max_ancestors_count`
        assert!(ancestors.len() < self.max_ancestors_count);

        self._record_ancestors(entry, ancestors, parents);
        Ok(evicted)
    }

    fn updated_stat_for_add_tx(
        &self,

                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    /// Update size and cycles statistics for remove tx
    /// cycles overflow is possible, currently obtaining cycles is not accurate
    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
