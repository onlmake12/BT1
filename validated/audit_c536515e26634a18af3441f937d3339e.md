### Title
Missing Duplicate Index Check in `GetBlockTransactionsProcess` Enables Bandwidth Amplification — (File: `sync/src/relayer/get_block_transactions_process.rs`)

---

### Summary

`GetBlockTransactionsProcess::execute()` validates the *count* of requested transaction indexes but never checks for *duplicate* indexes within the list. Any unprivileged P2P peer can craft a `GetBlockTransactions` message whose `indexes` field contains the same index repeated up to `MAX_RELAY_TXS_NUM_PER_BATCH` times, causing the responding node to serialize and transmit the same transaction data that many times in a single `BlockTransactions` reply.

---

### Finding Description

When a peer sends a `GetBlockTransactions` message, `GetBlockTransactionsProcess::execute()` performs two length-only guards: [1](#0-0) 

```rust
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH { … }
if get_block_transactions.uncle_indexes().len() > shared.consensus().max_uncles_num() { … }
```

After those guards pass, the handler iterates the raw `indexes` list without any deduplication: [2](#0-1) 

```rust
let transactions = self
    .message
    .indexes()
    .iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();

let uncles = self
    .message
    .uncle_indexes()
    .iter()
    .filter_map(|i| block.uncles().get(Into::<u32>::into(i) as usize))
    .collect::<Vec<_>>();
```

Every element of `indexes` — including duplicates — produces one entry in the response `Vec`. The resulting `BlockTransactions` message is then sent back to the peer: [3](#0-2) 

This is structurally identical to the NFTX `_addReceiver()` pattern: a list is populated by iterating an external input with no uniqueness check, so the same element can appear multiple times and be processed (here: serialized and transmitted) multiple times.

By contrast, the sibling handler `GetTransactionsProcess` for `GetRelayTransactions` **does** reject duplicate hashes: [4](#0-3) 

```rust
fn test_duplicate() {
    let content = packed::GetRelayTransactions::new_builder()
        .tx_hashes(vec![tx_hash.clone(), tx_hash])
        .build();
    …
    assert_eq!(
        rt.block_on(process.execute()),
        StatusCode::RequestDuplicate.with_context("Request duplicate transaction")
    );
}
```

The protection exists for one relay message type but is absent for the other.

---

### Impact Explanation

A peer that sends a `GetBlockTransactions` message with `MAX_RELAY_TXS_NUM_PER_BATCH` copies of the same index forces the node to:

1. Look up the same `TransactionView` that many times.
2. Serialize it that many times into the outgoing `BlockTransactions` message.
3. Transmit the inflated message over the network.

The amplification ratio equals `MAX_RELAY_TXS_NUM_PER_BATCH` × (size of the targeted transaction). For a large transaction this can be a significant outbound bandwidth spike per crafted request. Because the check is purely on list length, the attacker pays only the cost of sending a small request while the node pays the cost of sending a large, redundant response. Repeated from multiple connections this constitutes a targeted bandwidth-exhaustion vector against a specific node.

---

### Likelihood Explanation

The entry path requires only a standard P2P connection. Any peer that has completed the handshake can send a `GetBlockTransactions` message. No special privilege, key, or majority hash-power is needed. The crafted message is syntactically valid and passes all existing guards (length ≤ `MAX_RELAY_TXS_NUM_PER_BATCH`). The attack is trivially reproducible with a single crafted packet.

---

### Recommendation

Add a duplicate-index check immediately after the length guards, mirroring the pattern already used for `GetRelayTransactions`:

```rust
// After the existing length checks:
let mut seen = HashSet::new();
for idx in get_block_transactions.indexes().iter() {
    if !seen.insert(idx) {
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("Duplicate index in GetBlockTransactions");
    }
}
let mut seen_uncle = HashSet::new();
for idx in get_block_transactions.uncle_indexes().iter() {
    if !seen_uncle.insert(idx) {
        return StatusCode::ProtocolMessageIsMalformed
            .with_context("Duplicate uncle index in GetBlockTransactions");
    }
}
```

This brings `GetBlockTransactionsProcess` in line with the duplicate-rejection behavior already present in `GetTransactionsProcess`.

---

### Proof of Concept

1. Connect to a CKB node as a peer (complete the identify/sync handshake).
2. Obtain any block hash `H` that the node has stored.
3. Craft a `GetBlockTransactions` message:
   - `block_hash` = `H`
   - `indexes` = `[0, 0, 0, … 0]` repeated `MAX_RELAY_TXS_NUM_PER_BATCH` times (all pointing to the coinbase transaction, which is always present)
4. Send the message over the relay protocol.
5. Observe that the node replies with a `BlockTransactions` message containing `MAX_RELAY_TXS_NUM_PER_BATCH` copies of the coinbase transaction, each serialized in full.

The node performs no deduplication before building or sending the response. [5](#0-4)

```rust
    pub async fn execute(self) -> Status {
        let shared = self.relayer.shared();
        {
            let get_block_transactions = self.message;
            if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "Indexes count({}) > MAX_RELAY_TXS_NUM_PER_BATCH({})",
                    get_block_transactions.indexes().len(),
                    MAX_RELAY_TXS_NUM_PER_BATCH,
                ));
            }
            if get_block_transactions.uncle_indexes().len() > shared.consensus().max_uncles_num() {
                return StatusCode::ProtocolMessageIsMalformed.with_context(format!(
                    "UncleIndexes count({}) > consensus max_uncles_num({})",
                    get_block_transactions.uncle_indexes().len(),
                    shared.consensus().max_uncles_num(),
                ));
            }
        }

        let block_hash = self.message.block_hash().to_entity();
        debug_target!(
            crate::LOG_TARGET_RELAY,
            "get_block_transactions {}",
            block_hash
        );

        if let Some(block) = shared.store().get_block(&block_hash) {
            let transactions = self
                .message
                .indexes()
                .iter()
                .filter_map(|i| {
                    block
                        .transactions()
                        .get(Into::<u32>::into(i) as usize)
                        .cloned()
                })
                .collect::<Vec<_>>();

            let uncles = self
                .message
                .uncle_indexes()
                .iter()
                .filter_map(|i| block.uncles().get(Into::<u32>::into(i) as usize))
                .collect::<Vec<_>>();

            let content = packed::BlockTransactions::new_builder()
                .block_hash(block_hash)
                .transactions(
                    transactions
                        .into_iter()
                        .map(|tx| tx.data())
                        .collect::<Vec<_>>(),
                )
                .uncles(
                    uncles
                        .into_iter()
                        .map(|uncle| uncle.data())
                        .collect::<Vec<_>>(),
                )
                .build();
            let message = packed::RelayMessage::new_builder().set(content).build();

            return async_send_message_to(&self.nc, self.peer, &message).await;
        }

        Status::ok()
    }
```
