### Title
Unbounded `pending_get_block_proposals` Map Allows Peer-Flooding DoS Against `prune_tx_proposal_request` and the Tx-Pool Controller — (`sync/src/types/mod.rs`, `sync/src/relayer/mod.rs`)

---

### Summary

An unprivileged peer can flood a CKB node with `GetBlockProposal` relay messages containing fake/non-existent proposal short IDs. Because `insert_get_block_proposals` imposes no size cap on the `pending_get_block_proposals` `DashMap`, the map grows without bound between timer ticks. When `prune_tx_proposal_request` fires, it clones the entire map, issues a single `fetch_txs` call to the tx-pool controller with every accumulated ID, and iterates over all entries — causing O(n) memory allocation and O(n) sequential tx-pool lookups that block the tx-pool service from processing legitimate transactions.

---

### Finding Description

**Entry path — `GetBlockProposalProcess::execute`**

`sync/src/relayer/get_block_proposal_process.rs` handles incoming `GetBlockProposal` relay messages. It enforces a per-message size cap:

```
let limit = shared.consensus().max_block_proposals_limit()
    * (shared.consensus().max_uncles_num() as u64);
if message_len as u64 > limit { ... reject ... }
``` [1](#0-0) 

After the per-message check, proposals not found in the tx-pool are inserted into `pending_get_block_proposals` with no further guard:

```rust
self.relayer
    .shared()
    .state()
    .insert_get_block_proposals(self.peer, not_exist_proposals);
``` [2](#0-1) 

**Unbounded accumulation — `insert_get_block_proposals`**

`insert_get_block_proposals` appends every supplied ID into the `DashMap` with no size limit:

```rust
pub fn insert_get_block_proposals(&self, pi: PeerIndex, ids: Vec<packed::ProposalShortId>) {
    for id in ids.into_iter() {
        self.pending_get_block_proposals
            .entry(id)
            .or_default()
            .insert(pi);
    }
}
``` [3](#0-2) 

The backing field is a plain `DashMap` with no capacity bound:

```rust
pending_get_block_proposals: DashMap<packed::ProposalShortId, HashSet<PeerIndex>>,
``` [4](#0-3) 

**O(n) drain — `prune_tx_proposal_request`**

On each timer tick the relayer calls `prune_tx_proposal_request`, which:

1. **Clones** the entire map (O(n) heap allocation):
   ```rust
   let get_block_proposals = self.shared().state().drain_get_block_proposals();
   ```
2. **Issues a single `fetch_txs` call** to the tx-pool controller with every accumulated ID:
   ```rust
   let fetch_txs = tx_pool
       .fetch_txs(
           get_block_proposals
               .iter()
               .map(|kv_pair| kv_pair.key().clone())
               .collect(),
       )
       .await;
   ```
3. **Iterates over all entries** to build per-peer response batches. [5](#0-4) 

The `drain_get_block_proposals` clone-and-clear is itself O(n):

```rust
pub fn drain_get_block_proposals(
    &self,
) -> DashMap<packed::ProposalShortId, HashSet<PeerIndex>> {
    let ret = self.pending_get_block_proposals.clone();
    self.pending_get_block_proposals.clear();
    ret
}
``` [6](#0-5) 

The tx-pool controller is a single-threaded async service. A `fetch_txs` call with millions of fake IDs occupies it for the entire lookup duration, starving all other tx-pool operations (submission, reorg processing, block assembly).

---

### Impact Explanation

- **Tx-pool starvation**: The tx-pool controller processes requests sequentially. A single oversized `fetch_txs` call blocks all pending tx submissions, RBF checks, and block-template assembly for the duration of the lookup.
- **Memory exhaustion**: Each `ProposalShortId` is 10 bytes; with `max_block_proposals_limit * max_uncles_num ≈ 3000` IDs per message and no per-peer or total cap, an attacker sending messages at line rate can accumulate tens of millions of entries before the next timer tick.
- **Repeated attack**: Because the map is cleared on each drain, the attacker simply continues sending messages to re-fill it on every timer interval.

---

### Likelihood Explanation

- Any unauthenticated peer on the relay protocol can send `GetBlockProposal` messages.
- No rate limiting, connection-level quota, or ban is applied for sending many valid-sized messages.
- The per-message limit (`max_block_proposals_limit * max_uncles_num`) is a consensus parameter (mainnet: 1500 × 2 = 3000), giving each message a large payload budget.
- A single attacker with a modest connection can sustain the flood indefinitely.

---

### Recommendation

1. **Cap `pending_get_block_proposals` globally**: Reject or drop entries once the map exceeds a configurable maximum (e.g., `max_block_proposals_limit * max_connected_peers`).
2. **Cap per-peer contribution**: Track how many pending entries each peer has contributed and stop accepting new ones once the per-peer quota is reached.
3. **Chunk `fetch_txs` calls**: In `prune_tx_proposal_request`, split the accumulated IDs into fixed-size batches rather than issuing one unbounded call.
4. **Rate-limit `GetBlockProposal` messages per peer**: Apply a token-bucket or sliding-window counter and disconnect/ban peers that exceed it.

---

### Proof of Concept

```
1. Attacker connects to a CKB node as a relay peer.
2. Attacker generates N unique 10-byte ProposalShortIds that do not exist
   in the target node's tx-pool (trivially: random bytes).
3. Attacker sends repeated GetBlockProposal messages, each containing
   up to 3000 fake IDs (the per-message limit).
4. GetBlockProposalProcess::execute passes the per-message check,
   calls fetch_txs (which returns empty), then calls
   insert_get_block_proposals with all 3000 IDs.
5. Between timer ticks, pending_get_block_proposals accumulates
   millions of entries.
6. When prune_tx_proposal_request fires:
   a. drain_get_block_proposals() clones the entire map — O(n) allocation.
   b. fetch_txs() is called with all n IDs — the tx-pool controller
      is blocked for the entire lookup.
   c. Legitimate tx submissions, RBF checks, and get_block_template
      calls queue behind the blocked controller.
7. Repeat indefinitely; the node's tx-pool becomes unresponsive.
```

    async fn prune_tx_proposal_request(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());
                }
            }
        }

        let mut relay_bytes = 0;
        let mut relay_proposals = Vec::new();
        for (peer_index, txs) in peer_txs {
            for tx in txs {
                let data = tx.data();
                let tx_size = data.total_size();
                if relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH {
                    send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals))
                        .await;
                    relay_bytes = tx_size;
                } else {
                    relay_bytes += tx_size;
                }
                relay_proposals.push(data);
            }
            if !relay_proposals.is_empty() {
                send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals)).await;
                relay_bytes = 0;
            }
        }
