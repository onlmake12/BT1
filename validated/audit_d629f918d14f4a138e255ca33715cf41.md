Audit Report

## Title
Attacker-Controlled Snappy Varint Triggers Up-to-8MB Pre-Decompression Heap Allocation — (`network/src/compress.rs`)

## Summary
`LengthDelimitedCodecWithCompress::decode` allocates `BytesMut::zeroed(decompressed_bytes_len)` (up to 8MB) based solely on the attacker-supplied snappy varint header, before verifying that the compressed payload is valid. The `max_frame_length` guard operates on the compressed wire frame and does not bound the claimed decompressed size. No codec-layer ban is applied on decode failure, so an attacker can reconnect immediately and repeat the allocation cycle across all available inbound slots.

## Finding Description
In `network/src/compress.rs` line 233, `decompress_len(&data[1..])` reads the uncompressed length from the snappy stream varint — a value entirely under attacker control. If the returned value is ≤ `MAX_UNCOMPRESSED_LEN` (8MB, line 13), `BytesMut::zeroed(decompressed_bytes_len)` is unconditionally allocated at line 242. The actual decompression at line 243 may then fail (returning `Err` and closing the connection), but the allocation has already occurred and the peak RSS spike is real.

The `length_delimited.decode(src)?` call at line 226 enforces `max_frame_length` (2MB for Sync, 4MB for RelayV3 per `support_protocols.rs` lines 129–130) on the **compressed** wire frame. A crafted frame of ~11 bytes — `COMPRESS_FLAG` byte + snappy varint encoding 7.9MB + garbage — trivially passes this check. The `ban_peer` mechanism (implemented in `protocols/mod.rs` line 684) lives at the application layer and is never reached for codec-level errors; tentacle simply closes the connection, leaving the attacker free to reconnect immediately.

The same pattern exists in `Message::decompress` (line 81, using `vec![0; decompressed_bytes_len]`), but the primary attack surface is the codec path which is exercised for every inbound frame.

## Impact Explanation
With default configuration (`max_peers=125`, `max_outbound_peers=8`), up to 117 simultaneous inbound connections are accepted. Each connection sending one crafted frame triggers a ~8MB allocation: 117 × 8MB ≈ **936MB peak simultaneous heap allocation**. After disconnect, the attacker reconnects and repeats, causing continuous allocator churn and RSS spikes. On memory-constrained nodes this causes OOM termination; on well-provisioned nodes it causes sustained allocator pressure that degrades P2P message processing throughput across all protocols. This matches the allowed impact: **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attack requires only a TCP connection to port 8115 (the default public P2P port). The attacker must complete the tentacle/secio handshake, which is cryptographically trivial and requires no authentication, no PoW, and no key material beyond generating an ephemeral keypair. The crafted frame is ~11 bytes. The default `max_peers=125` provides no meaningful barrier. The attack is fully repeatable with no cooldown because no codec-layer ban is applied.

## Recommendation
Move the `BytesMut::zeroed` allocation **after** a lightweight structural validation of the compressed payload (e.g., verify the snappy stream identifier block before allocating), or replace the pre-allocated buffer with a lazy/streaming decompressor that does not pre-allocate the full claimed size. At minimum, add a codec-layer connection-rate limit or exponential backoff per source IP when `decode` returns `Err` for a compressed frame, to prevent immediate reconnect-and-repeat amplification.

## Proof of Concept
```python
import socket, struct, time

# Snappy varint encoding of 7,864,320 (~7.9 MB)
size = 7_864_320
varint = b""
while size > 0x7F:
    varint += bytes([(size & 0x7F) | 0x80])
    size >>= 7
varint += bytes([size])

# Frame: COMPRESS_FLAG=0x80, snappy varint for 7.9MB, garbage payload
payload = bytes([0x80]) + varint + b'\xff\xff\xff'

# LengthDelimited framing: 4-byte big-endian length prefix
frame = struct.pack('>I', len(payload)) + payload

# After completing tentacle secio handshake on port 8115,
# open the Sync or RelayV3 sub-protocol and send this frame.
# Each send triggers BytesMut::zeroed(7_864_320) on the target node.
# Connection is dropped; reconnect immediately and repeat.
