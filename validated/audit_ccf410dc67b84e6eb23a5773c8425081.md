### Title
Echo.sol `executeCallback` Unbounded `firstUnfulfilledSeq` Scan Loop Enables Permanent Fund Lock via Request Griefing — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded while loop that advances `_state.firstUnfulfilledSeq` past all consecutively cleared requests. An unprivileged attacker can create a large number of requests (with themselves as provider), execute their own callbacks to clear those requests while recovering their fees, and then cause any subsequent `executeCallback` call for an earlier victim request to exhaust the block gas limit. Because no `cancelRequest` or refund path exists, the victim's funds are permanently locked.

---

### Finding Description

After clearing a request, `executeCallback` runs:

```solidity
clearRequest(sequenceNumber);

while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each loop iteration calls `findRequest`, which performs one or two storage reads:

```solidity
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];
    }
}
``` [2](#0-1) 

The loop iterates up to `currentSequenceNumber − firstUnfulfilledSeq` times. `currentSequenceNumber` is unbounded — it increments with every new request:

```solidity
requestSequenceNumber = _state.currentSequenceNumber++;
``` [3](#0-2) 

**Attack path:**

1. Victim creates request with sequence number `S`, paying fee `F`.
2. Attacker creates `N` requests with sequence numbers `S+1 … S+N`, naming themselves as `provider`.
3. Attacker calls `executeCallback` for each of their own requests. Because `firstUnfulfilledSeq = S` and request `S` is still active, the while loop body never executes during these calls — `firstUnfulfilledSeq` stays at `S`. The attacker recovers their fees (net cost: `N × pythFeeInWei` + gas).
4. Now `firstUnfulfilledSeq = S`, requests `S+1 … S+N` are all cleared, `currentSequenceNumber = S+N+1`.
5. Any caller attempting `executeCallback` for request `S` triggers the while loop, which iterates `N` times. Each iteration performs a cold `SLOAD` of `requestsOverflow[key]` (~2,100 gas). At `N ≈ 14,000`, the loop alone consumes ~29.4 M gas, exceeding the EVM block gas limit (~30 M gas). The transaction reverts.
6. Because `clearRequest` is also reverted, request `S` remains active but can never be successfully executed. There is no `cancelRequest` or refund function, so the victim's fee `F` is permanently locked.

The developers acknowledge the gas problem in a TODO comment:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
``` [4](#0-3) 

The same unbounded scan also appears in `getFirstActiveRequests`, which iterates from `firstUnfulfilledSeq` to `currentSequenceNumber` and can be used to DoS RPC nodes:

```solidity
while (
    actualCount < count && currentSeq < _state.currentSequenceNumber
) {
    Request memory req = findRequest(currentSeq);
    ...
    currentSeq++;
}
``` [5](#0-4) 

---

### Impact Explanation

Permanent fund lock for Echo users. A victim's request fee is irrecoverable: `executeCallback` always reverts due to gas exhaustion, and no alternative refund path exists. The `EchoConsumer` contract never receives its price-feed callback, breaking any downstream protocol logic that depends on it.

---

### Likelihood Explanation

Any unprivileged address can create requests and execute their own callbacks. The attacker recovers their request fees (net cost is only `N × pythFeeInWei` + gas). If `pythFeeInWei` is small (as is typical for a fixed protocol fee), the attack is economically viable against any victim request whose locked value exceeds the attacker's gas cost. The attack is deterministic and requires no timing luck — the attacker simply needs to submit their requests after the victim's.

---

### Recommendation

1. **Cap loop iterations per call.** Advance `firstUnfulfilledSeq` by at most a fixed constant (e.g., 64) per `executeCallback` invocation, deferring further advancement to future calls.
2. **Add a `cancelRequest` function.** Allow the original requester to reclaim their fee after a timeout, eliminating the permanent-lock risk.
3. **Replace the linear scan with a doubly-linked list** of active requests (as noted in the existing TODO), so `firstUnfulfilledSeq` advancement is O(1).

---

### Proof of Concept

```solidity
// 1. Victim creates request 1 with fee F
echo.requestPriceUpdatesWithCallback{value: F}(attacker, publishTime, priceIds, gasLimit);
// seq = 1, firstUnfulfilledSeq = 1

// 2. Attacker creates 14,000 requests (seq 2..14001), provider = attacker
for (uint i = 0; i < 14000; i++) {
    echo.requestPriceUpdatesWithCallback{value: minFee}(attacker, publishTime, priceIds, gasLimit);
}
// currentSequenceNumber = 14002

// 3. Attacker executes own callbacks (seq 2..14001), recovering fees
//    Each call: while loop checks seq 1 (still active) → 0 iterations → no gas issue
for (uint64 seq = 2; seq <= 14001; seq++) {
    echo.executeCallback{value: pythFee}(attacker, seq, updateData, priceIds);
}
// firstUnfulfilledSeq still = 1; requests 2..14001 cleared

// 4. Anyone tries to execute callback for seq 1:
//    while loop iterates 14000 times → ~29.4M gas → OUT OF GAS → revert
//    Victim's fee F is permanently locked.
echo.executeCallback{value: pythFee, gas: 30_000_000}(provider, 1, updateData, priceIds);
// Reverts: out of gas
```
