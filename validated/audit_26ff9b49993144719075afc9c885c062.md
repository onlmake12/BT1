Audit Report

## Title
Bypassed `request_limit` Guard via `usize::MAX` Default Enables RPC-Triggered Node OOM Crash — (File: util/rich-indexer/src/service.rs)

## Summary
`RichIndexerService::new()` defaults `request_limit` to `usize::MAX` when unconfigured. The guard in `get_cells` and `get_transactions` compares a `Uint32` limit (max `4,294,967,295`) against `usize::MAX` (`18,446,744,073,709,551,615`), making the check permanently false. Any unauthenticated RPC caller can submit `limit = 0xFFFFFFFF` to force unbounded database row fetching and JSON serialization, exhausting node memory and causing an OOM crash.

## Finding Description
`RichIndexerService::new()` sets `request_limit: config.request_limit.unwrap_or(usize::MAX)` at [1](#0-0)  and the identical pattern exists in the built-in indexer at [2](#0-1) .

The guard in `get_cells` is:
```rust
if limit as usize > self.request_limit { ... }
``` [3](#0-2) 

With `limit = 0xFFFFFFFF` (`4,294,967,295` as `usize`) and `self.request_limit = usize::MAX` (`18,446,744,073,709,551,615`), the condition is always `false`. The guard is a no-op. The SQL query is issued with `LIMIT 4294967295`, and `fetch_all()` attempts to collect all matching rows into a `Vec<IndexerCell>` in heap memory, followed by serde JSON serialization.

For `get_cells_capacity`, there is no `limit` parameter at all — it executes an unbounded `SUM(output.capacity)` aggregation. However, this function uses `fetch_optional` returning a single aggregated row, so the OOM vector there is a slow full-table-scan query rather than heap exhaustion from row collection. [4](#0-3) 

The codebase itself documents the memory risk explicitly: [5](#0-4) 

## Impact Explanation
On a fully-synced mainnet node with millions of indexed cells, a single RPC call with `limit = 0xFFFFFFFF` causes `fetch_all()` to collect all indexed cells into heap memory, followed by serde JSON serialization at ~10× raw data size (as documented in `ckb.toml`). This exhausts available RAM and triggers an OOM kill of the node process, taking down all services including consensus-critical ones. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The RPC endpoint is reachable by any local or network-accessible caller with no authentication required. The vulnerable default (`usize::MAX`) is the out-of-the-box behavior — operators must explicitly opt in to the safe value. A single request is sufficient. No chain state knowledge is needed; an empty `args` with `prefix` search mode maximizes the result set. The attack is trivially repeatable.

## Recommendation
1. Replace `usize::MAX` with a safe hard default in both indexers:
   ```rust
   // util/rich-indexer/src/service.rs and util/indexer/src/service.rs
   request_limit: config.request_limit.unwrap_or(400),
   ```
2. Add a row-count guard or configurable query timeout to `get_cells_capacity` in `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_cells_capacity.rs`.
3. Consider making `request_limit` a required config field with no implicit default, forcing operators to make an explicit choice.
