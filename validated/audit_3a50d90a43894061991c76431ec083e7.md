### Title
Division-by-Zero in `calculateTwap` Due to Missing `publishSlot` Equality Check in `validateTwapPriceInfo` — (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

The `validateTwapPriceInfo` function in the EVM `Pyth.sol` contract uses a non-strict comparison (`>`) for `publishSlot` ordering, allowing equal slot values to pass validation. When `calculateTwap` is subsequently called with equal `publishSlot` values, `slotDiff = 0`, and three subsequent division operations panic-revert. Any unprivileged caller can trigger this by supplying the same valid Wormhole-signed accumulator update for both the start and end TWAP data arguments.

---

### Finding Description

In `validateTwapPriceInfo`, the slot ordering guard is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This only rejects `start > end`; it silently allows `start == end`. [1](#0-0) 

When `calculateTwap` is called with equal slots, `slotDiff` is zero:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // == 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // Panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // Panic: division by zero
...
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // Panic: division by zero
``` [2](#0-1) 

Solidity 0.8+ emits `Panic(0x12)` on integer division by zero, reverting the entire transaction.

The Solana implementation correctly uses a strict greater-than check, preventing this case entirely:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The EVM `parseTwapPriceFeedUpdates` entry point calls `validateTwapPriceInfo` and then `calculateTwap` without any additional slot-equality guard: [4](#0-3) 

---

### Impact Explanation

Any unprivileged caller can invoke `parseTwapPriceFeedUpdates` with the same valid Wormhole-signed accumulator update supplied for both `updateData[0]` (start) and `updateData[1]` (end). Both parsed `TwapPriceInfo` structs will carry identical `publishSlot` values. The validation in `validateTwapPriceInfo` passes (equal slots are not rejected), and `calculateTwap` immediately panics. The transaction reverts with an opaque `Panic(0x12)` instead of a clean `InvalidTwapUpdateDataSet` error.

For any protocol that wraps `parseTwapPriceFeedUpdates` in a critical path (e.g., a settlement or liquidation flow), an attacker can repeatedly trigger this revert to DoS that path. The fee is refunded on revert, so the attacker's cost is only gas.

**Impact: Medium** — DoS on TWAP price feed parsing; no funds are permanently lost, but dependent protocol flows can be blocked.

---

### Likelihood Explanation

**Likelihood: Low** — Requires deliberately supplying same-slot data. In normal operation, Pythnet produces start/end TWAP messages at different slots. However, the attack requires no privileged access, no key material, and no off-chain infrastructure beyond a single valid accumulator update already available on-chain or from Hermes.

---

### Recommendation

Change the slot ordering check in `validateTwapPriceInfo` from strict-greater to greater-or-equal, matching the Solana implementation:

```solidity
// Before (allows equal slots → division by zero)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {

// After (rejects equal slots → clean revert)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [5](#0-4) 

---

### Proof of Concept

1. Obtain any valid Wormhole-signed TWAP accumulator update `validUpdate` from Hermes (or reuse one already submitted on-chain).
2. Call:
   ```solidity
   pyth.parseTwapPriceFeedUpdates{value: fee}(
       [validUpdate, validUpdate],  // same update for start AND end
       [priceId]
   );
   ```
3. Both `startTwapPriceInfos[0].publishSlot` and `endTwapPriceInfos[0].publishSlot` are identical.
4. `validateTwapPriceInfo` passes — the check `start > end` is false when they are equal.
5. `calculateTwap` computes `slotDiff = 0` and immediately panics on `priceDiff / int128(uint128(0))`.
6. Transaction reverts with `Panic(0x12)` (division by zero) instead of `InvalidTwapUpdateDataSet`.

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
    }
