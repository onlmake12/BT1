### Title
Permanent Locking of All Protocol Fees in `PriceInfoObject` with No Withdrawal Path ŌĆö (`target_chains/sui/contracts/sources/price_info.move`, `pyth.move`)

---

### Summary

Every SUI fee paid via `update_single_price_feed` is deposited into the `PriceInfoObject`'s dynamic object field and **permanently locked there**. No withdrawal function exists anywhere in the contract ŌĆö not for users, not for governance, not for the `fee_recipient_address`. The `set_fee_recipient` governance module is explicitly deprecated and only changes a stored address; it never moves coins. Additionally, the fee is deposited **before** the freshness check, so users who submit a non-fresh (stale-timestamp) update lose their fee with zero benefit.

---

### Finding Description

**Step 1 ŌĆö Fee deposited unconditionally before freshness check.**

In `update_single_price_feed`, the fee is deposited at line 277, before the loop that checks whether the update is actually fresh: [1](#0-0) 

`update_cache` only writes the new price when `is_fresh_update` returns `true` (i.e., `update_timestamp > cached_timestamp`): [2](#0-1) 

If the update is not fresh, the fee is already deposited and the transaction does **not** abort ŌĆö it returns normally. The user's SUI is gone with no price update.

**Step 2 ŌĆö `deposit_fee_coins` accumulates without bound, no per-user accounting.** [3](#0-2) 

Coins are `coin::join`-ed into a single `Coin<SUI>` stored under `FEE_STORAGE_KEY`. There is no cap, no per-user record, and no refund path.

**Step 3 ŌĆö No withdrawal function exists anywhere.**

A full grep for `withdraw_fee`, `collect_fee`, `transfer.*fee_recipient`, and `fee_recipient.*transfer` across all `.move` sources returns **zero matches**. The only functions touching the fee field are `get_balance` (read-only) and `deposit_fee_coins` (write-only accumulate). [4](#0-3) 

**Step 4 ŌĆö The `set_fee_recipient` governance action is explicitly deprecated and non-functional.** [5](#0-4) 

The comment states: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore."* The governance action only updates `fee_recipient_address` in `State`; it never touches the coins locked in any `PriceInfoObject`. [6](#0-5) 

The governance dispatch loop in `governance.move` confirms there is no "withdraw fees" action: [7](#0-6) 

---

### Impact Explanation

Two concrete impacts:

1. **Protocol insolvency / permanent fee lock**: Every SUI fee ever paid to the protocol is permanently locked inside shared `PriceInfoObject`s. The protocol can never collect its own revenue. This satisfies the "protocol insolvency" criterion in scope.

2. **User funds lost on stale updates**: Any user who submits a valid (non-replayed) VAA whose price timestamp is Ōēż the cached timestamp pays `base_update_fee` SUI and receives nothing ŌĆö the price is not updated and the fee is not refunded. This is a direct, irreversible loss of user funds on every such call.

---

### Likelihood Explanation

- **Stale-update fee loss** is reachable whenever a publisher temporarily stops advancing timestamps, or when two different callers race to submit the same price window. No privilege is required; any address can call `update_single_price_feed`.
- **Permanent fee lock** is not conditional ŌĆö it applies to 100% of all fee payments from the moment of deployment. It is already in effect on mainnet.

---

### Recommendation

1. **Immediate transfer**: Replace `deposit_fee_coins` with an immediate `transfer::public_transfer(fee, state::get_fee_recipient(pyth_state))` so fees never accumulate in the shared object.
2. **Freshness-gated fee**: Move the fee deposit inside the `if (is_fresh_update(...))` branch, or abort the transaction when the update is not fresh, so users are not charged for no-op updates.
3. **Add a governance withdrawal**: If accumulation is intentional, add a governance action that calls `dynamic_object_field::remove` on `FEE_STORAGE_KEY` and transfers the coin to `fee_recipient_address`.

    public fun get_balance(price_info_object: &PriceInfoObject): u64 {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);
        coin::value(fee)
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
