### Title
Deviation Baseline Resets on Heartbeat-Triggered Updates, Allowing Unbounded Cumulative Price Drift to Bypass the Deviation Trigger - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices()` function measures price deviation against the last **stored** price (`previousFeed.price.price`), which is overwritten on every successful `updatePriceFeeds()` call — including those triggered by the heartbeat condition. When a subscription has both `updateOnHeartbeat` and `updateOnDeviation` enabled, each heartbeat-triggered update silently resets the deviation baseline to the current (already-drifted) price. This allows a price to drift arbitrarily far from its original value while the deviation trigger never fires, because each incremental step is measured only against the most recent stored snapshot rather than a stable reference.

---

### Finding Description

`_validateShouldUpdatePrices()` checks deviation by loading `previousFeed` from `_state.priceUpdates[subscriptionId][priceFeeds[i].id]`: [1](#0-0) 

The deviation is computed as `|currentPrice - previousPrice| / |previousPrice|` in basis points: [2](#0-1) 

After any successful `updatePriceFeeds()` call — regardless of whether it was triggered by heartbeat or deviation — `_storePriceUpdates()` is called unconditionally, overwriting `_state.priceUpdates[subscriptionId][...]` with the new price: [3](#0-2) 

This means the deviation baseline is always the price at the time of the **last update**, not a fixed reference. When the heartbeat fires and triggers an update, the stored price is refreshed to whatever the current price is — even if that price has already drifted significantly from the original.

---

### Impact Explanation

Protocols subscribing to the Scheduler with both heartbeat and deviation criteria enabled (a natural and expected configuration) will not receive timely deviation-triggered updates when prices drift gradually. The stored price in the Scheduler — returned via `getPricesUnsafe()` and `getPricesNoOlderThan()` — will lag far behind the actual market price: [4](#0-3) 

Downstream protocols (lending, derivatives, perps) reading from the Scheduler will receive prices that are stale by multiples of the deviation threshold, potentially enabling incorrect liquidations, undercollateralized positions, or exploitable mispricings. The deviation trigger — the primary mechanism for ensuring timely updates during volatile markets — is rendered ineffective.

---

### Likelihood Explanation

**High.** The combined heartbeat + deviation configuration is the standard recommended setup for Scheduler subscriptions. The vulnerability is structural and deterministic: it triggers automatically whenever the heartbeat fires while the price has drifted below the threshold. No adversarial action is required — normal market price movement combined with routine heartbeat updates is sufficient. Any unprivileged keeper calling `updatePriceFeeds()` on a heartbeat schedule causes the baseline reset. [5](#0-4) 

---

### Recommendation

Decouple the deviation baseline from the heartbeat-updated stored price. Two approaches:

1. **Store a separate deviation reference price** that is only updated when a deviation-triggered update occurs (not on heartbeat-triggered updates). The deviation check always compares against this stable reference.

2. **Track the price at the time of the last deviation-check reset** separately from `_state.priceUpdates`, and only reset it when the deviation threshold is actually crossed and an update is stored for that reason.

This mirrors the fix recommended in the original ConicPool report: the baseline for the deviation check must be a stable reference, not a value that silently resets on every update regardless of trigger reason.

---

### Proof of Concept

**Setup:** Subscription with `updateOnHeartbeat=true` (heartbeat=60s), `updateOnDeviation=true` (threshold=100 bps / 1%).

1. **T=0:** Price = 10000. First `updatePriceFeeds()` call stores `previousFeed.price.price = 10000`.

2. **T=59s:** Price drifts to 10090 (+0.9%). Deviation = 90 bps < 100 bps threshold → deviation trigger does not fire.

3. **T=60s:** Heartbeat fires. Keeper calls `updatePriceFeeds()`. `_validateShouldUpdatePrices()` returns via the heartbeat path (line 408) without checking deviation. `_storePriceUpdates()` overwrites `previousFeed.price.price = 10090`. [6](#0-5) 

4. **T=119s:** Price drifts to 10181 (+0.9% from 10090, but +1.81% from original 10000). Deviation check computes `|10181 - 10090| / 10090 = 90 bps` < 100 bps → deviation trigger does not fire.

5. **T=120s:** Heartbeat fires again. Stores `previousFeed.price.price = 10181`. Baseline resets again.

6. **Repeat:** After N heartbeat cycles, price = `10000 × (1.009)^N`. For N=8 (8 minutes), price ≈ 10755 (+7.55% from original), yet the deviation trigger has never fired. The Scheduler's stored price is always ~0.9% behind the actual price, never crossing the 1% threshold.

Protocols reading `getPricesUnsafe()` receive the heartbeat-updated price (e.g., 10090 at T=60s) while the actual market price is 10181 — a discrepancy that grows unboundedly with continued drift.
