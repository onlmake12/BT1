### Title
`addFunds` Deposit Limit Check Validates Only `msg.value` Instead of Cumulative Balance, Allowing Permanent Subscriptions to Exceed `MAX_DEPOSIT_LIMIT` — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `addFunds` function in `Scheduler.sol` checks whether the individual deposit (`msg.value`) exceeds `MAX_DEPOSIT_LIMIT`, but never checks whether the **resulting cumulative balance** (`status.balanceInWei + msg.value`) would exceed the limit. Because `addFunds` has no access control and permanent subscriptions cannot withdraw funds, any caller can repeatedly deposit up to `MAX_DEPOSIT_LIMIT` per call, permanently locking an unbounded amount of ETH in a permanent subscription.

---

### Finding Description

`MAX_DEPOSIT_LIMIT` is defined as `100 ether` in `SchedulerConstants.sol` and is intended to cap how much ETH can be deposited into a permanent subscription (which cannot be withdrawn). [1](#0-0) 

In `addFunds`, the guard is:

```solidity
if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
``` [2](#0-1) 

The check validates only the **single-call deposit amount** (`msg.value`), not the **post-deposit cumulative balance** (`status.balanceInWei + msg.value`). The same flaw exists in `createSubscription`: [3](#0-2) 

Because `addFunds` has no access control (any address may call it for any active subscription), an attacker or even the subscription manager can call `addFunds` with `msg.value = MAX_DEPOSIT_LIMIT` (exactly 100 ETH, which passes the `>` check) an arbitrary number of times. Each call adds 100 ETH to `status.balanceInWei`, and since permanent subscriptions block all withdrawals: [4](#0-3) 

…the ETH is permanently locked with no recovery path.

The root cause is structurally identical to the external report: the validation check uses the **wrong reference value** (the individual deposit amount rather than the resulting cumulative balance), so the intended cap is never actually enforced on the aggregate state.

---

### Impact Explanation

- A permanent subscription's balance can grow to an arbitrary multiple of `MAX_DEPOSIT_LIMIT` (e.g., 1000 ETH, 10 000 ETH, …).
- All deposited ETH is permanently locked — `withdrawFunds` unconditionally reverts for permanent subscriptions.
- Any third party (not just the subscription manager) can trigger this, since `addFunds` is permissionless.
- The `MAX_DEPOSIT_LIMIT` safety cap is completely ineffective.

---

### Likelihood Explanation

- `addFunds` is callable by any address with no authentication.
- The bypass requires only repeated calls with `msg.value ≤ MAX_DEPOSIT_LIMIT`; no special privileges, keys, or governance access are needed.
- A subscription manager who simply wants to top up their own permanent subscription will inadvertently bypass the limit on the very first top-up if they send exactly `MAX_DEPOSIT_LIMIT`.

---

### Recommendation

Replace the per-call check with a cumulative-balance check:

```solidity
// Before adding funds, verify the resulting balance does not exceed the limit
if (params.isPermanent && status.balanceInWei + msg.value > MAX_DEPOSIT_LIMIT) {
    revert SchedulerErrors.MaxDepositLimitExceeded();
}
status.balanceInWei += msg.value;
```

Apply the same fix to `createSubscription` for consistency (though the initial deposit is the first call, so the current check there is equivalent — the cumulative form is still clearer and future-proof).

---

### Proof of Concept

```solidity
// Assume MAX_DEPOSIT_LIMIT = 100 ether
// Step 1: Create a permanent subscription at the limit
uint256 subId = scheduler.createSubscription{value: 100 ether}(permanentParams);

// Step 2: addFunds with exactly MAX_DEPOSIT_LIMIT — passes the `>` check
scheduler.addFunds{value: 100 ether}(subId);  // balance = 200 ETH ✓ (no revert)

// Step 3: Repeat
scheduler.addFunds{value: 100 ether}(subId);  // balance = 300 ETH ✓ (no revert)

// Step 4: Attempt withdrawal — always reverts for permanent subscriptions
scheduler.withdrawFunds(subId, 1 ether);  // CannotUpdatePermanentSubscription ✗

// Result: 300 ETH permanently locked, limit bypassed 3×
```
