Audit Report

## Title
Stale `EvictKey` on Parent After Child Removed via `remove_entry_and_descendants` Allows Low-Fee Tx to Evade Pool Eviction — (`tx-pool/src/component/pool_map.rs`)

## Summary
`remove_entry_and_descendants` pre-strips all link entries before invoking `remove_entry`, which causes `update_ancestors_index_key` to resolve an empty ancestor set for every removed transaction. As a result, any parent P of a removed child C never has its `descendants_fee`/`descendants_size`/`descendants_cycles` decremented and its `evict_key` recalculated. Because `EvictKey.fee_rate = descendants_feerate.max(feerate)`, P permanently retains the inflated fee rate contributed by C, making P appear more valuable than it truly is in the eviction ordering.

## Finding Description
`remove_entry_and_descendants` (L252–265) first collects all descendants of the target ID, then calls `remove_entry_links` on every ID in the removal set (L257–259). `remove_entry_links` (L418–430) removes the ID from its parents' children sets, removes its parents from its own parents set, and finally calls `self.links.remove(id)` (L429), deleting the link entry entirely.

Subsequently, `remove_entry` is called for each removed ID (L263). Inside `remove_entry`, `update_ancestors_index_key(&entry.inner, EntryOp::Remove)` is invoked at L242. That function calls `self.links.calc_ancestors(&child.proposal_short_id())` (L433–434), which delegates to `calc_relative_ids` (links.rs L37–50). There, `self.inner.get(short_id)` returns `None` because the link entry was already deleted; `direct` becomes an empty set via `.unwrap_or_default()` (L47), and `calc_relation_ids` returns an empty `HashSet`. The `for anc_id in &ancestors` loop body (L435–444) never executes: `sub_descendant_weight` is never called on P, and `e.evict_key = e.inner.as_evict_key()` (L442) is never re-run for P.

The `EvictKey` is computed as `fee_rate: descendants_feerate.max(feerate)` (entry.rs L243). When C was added, P's `descendants_fee`/`descendants_size`/`descendants_cycles` were incremented and P's `evict_key.fee_rate` was elevated to reflect C's high fee rate. After C is removed via `remove_entry_and_descendants`, those fields and the indexed key are never corrected.

The comment at L256 ("update links state for remove, so that we won't update_descendants_index_key in remove_entry") confirms the pre-deletion is intentional for the *descendant* direction (avoiding redundant updates on entries being removed), but it is an unintended side-effect for the *ancestor* direction: ancestors of the removed subtree are not being removed, yet their descendant-accounting state is left permanently stale.

## Impact Explanation
An unprivileged attacker can permanently inflate P's `evict_key.fee_rate` in the mempool. When the pool reaches capacity and `next_evict_entry` (L380–385) selects the lowest-ranked entry by `evict_key`, P is skipped because its stale key shows a high fee rate. Any legitimate transaction M whose true fee rate falls between P's actual fee rate and P's inflated stale fee rate will be evicted before P. The attacker can repeat the inflation cycle at negligible cost (two low-fee transactions and one conflicting submission), keeping P in the pool indefinitely while displacing legitimate transactions. This constitutes a low-cost, deterministic, and repeatable attack that corrupts the pool's eviction ordering across all affected nodes, fitting **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The attack requires no special privileges, no hashpower, and no social engineering. The `resolve_conflict` path (L305–332) is always enabled and is triggered by any standard double-spend submission. The conflicting transaction C' does not need to spend P's output — it only needs to conflict with C on one input. The cost is two valid unconfirmed transactions plus one conflicting submission. The bug is deterministic and reproducible in a local test environment. The attacker can repeat the cycle indefinitely to maintain the inflation.

## Recommendation
Collect each removed entry's ancestor set *before* calling `remove_entry_links`, then apply `sub_descendant_weight` and `evict_key` recalculation to those ancestors after the removal. Concretely, in `remove_entry_and_descendants`, before the link-stripping loop, compute `ancestors_of_removed = calc_ancestors(id)` for the root ID (ancestors outside the removal set), then after all removals iterate over those ancestors and call `sub_descendant_weight` + `as_evict_key()` for each removed entry. Alternatively, add an optional pre-computed ancestor set parameter to `remove_entry` that bypasses the `calc_ancestors` call when links have already been torn down.

## Proof of Concept

// In tx-pool/src/component/tests/pending.rs (illustrative)
#[test]
fn test_evict_key_stale_after_child_removed_via_conflict() {
    let mut pool = PoolMap::new(1000);

    // P: low fee (1 shannon, size 100)
    let tx_p = build_tx(vec![(&Byte32::zero(), 0)], 2);
    // C: high-fee child of P, also spends external UTXO_C
    let tx_c = build_tx(vec![(&tx_p.hash(), 0), (&h256!("0x2").into(), 0)], 1);
    // C': conflicts with C on UTXO_C, does NOT spend P's output
    let tx_c_prime = build_tx(vec![(&h256!("0x2").into(), 0)], 1);
    // M: medium fee, independent
    let tx_m = build_tx(vec![(&h256!("0x3").into(), 0)], 1);

    let entry_p = TxEntry::dummy_resolve(tx_p.clone(), 2, Capacity::shannons(1), 100);
    let entry_c = TxEntry::dummy_resolve(tx_c.clone(), 2, Capacity::shannons(1000), 100);
    let entry_m = TxEntry::dummy_resolve(tx_m.clone(), 2, Capacity::shannons(50), 100);

    pool.add_entry(entry_p, Status::Pending).unwrap();
    pool.add_entry(entry_c, Status::Pending).unwrap();

    // Simulate conflict: remove C via resolve_conflict path
    pool.resolve_conflict(&tx_c_prime);

    pool.add_entry(entry_m, Status::Pending).unwrap();

    // With the bug: P's evict_key.fee_rate is still ~1000, so M is evicted first
    let first_evict = pool.next_evict_entry(Status::Pending).unwrap();
    // Expected (correct): first_evict == tx_p (fee_rate 1 < 50)
    // Actual (buggy):     first_evict == tx_m (P's stale evict_key shows ~1000)
    assert_eq!(
        first_evict,
        tx_p.proposal_short_id(),
        "P should be evicted first (lowest true fee rate), not M"
    );
}
