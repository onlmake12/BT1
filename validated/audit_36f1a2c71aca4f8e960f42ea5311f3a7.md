### Title
Two-Phase Commit Bypass via `ProposalShortId` Truncation Collision — (`File: verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

CKB's two-phase commit verifier enforces that every committed transaction was previously proposed, but it performs this check using `ProposalShortId` — a **10-byte (80-bit) truncation** of the full 32-byte transaction hash. A miner who finds two distinct transactions sharing the same `ProposalShortId` (a birthday attack requiring ~2^40 operations) can propose one transaction and commit a completely different one, bypassing the proposal requirement entirely. The proposer-reward calculator in `util/reward-calculator/src/lib.rs` is also keyed on `ProposalShortId`, enabling incorrect reward attribution as a secondary impact.

---

### Finding Description

**Root cause — truncated identifier in consensus check**

`ProposalShortId` is defined as the first 10 bytes of a transaction's Blake2b hash:

```rust
// util/gen-types/src/extension/shortcut.rs
pub fn from_tx_hash(h: &packed::Byte32) -> Self {
    let mut inner = [0u8; 10];
    inner.copy_from_slice(&h.as_slice()[..10]);
    inner.into()
}
``` [1](#0-0) 

The `TwoPhaseCommitVerifier::verify()` collects the short IDs of all committed (non-cellbase) transactions and checks them against the set of short IDs that appeared in the proposal window:

```rust
// verification/contextual/src/contextual_block_verifier.rs
let committed_ids: HashSet<_> = self
    .block
    .transactions()
    .iter()
    .skip(1)
    .map(TransactionView::proposal_short_id)   // ← 10-byte truncation
    .collect();

if committed_ids.difference(&proposal_txs_ids).next().is_some() {
    return Err((CommitError::Invalid).into());
}
``` [2](#0-1) 

The `proposal_txs_ids` set is also populated from on-chain proposal fields, which store `ProposalShortId` values (10 bytes), not full hashes. The check is therefore: *"does the committed transaction's 10-byte prefix appear in the proposal set?"* — not *"was this exact transaction proposed?"*

**The collision path**

Because `ProposalShortId` is only 80 bits, a birthday attack requires approximately 2^40 hash evaluations to find two distinct valid transactions `Tx_A` and `Tx_B` such that:

```
ProposalShortId(Tx_A) == ProposalShortId(Tx_B)
```

CKB transactions have many degrees of freedom (witnesses, output data, capacity values, etc.), making it straightforward to generate large numbers of syntactically valid candidate transactions for the birthday search. At ~10^9 hashes/second on commodity GPU hardware, 2^40 ≈ 10^12 operations is achievable in hours to days.

**Attack sequence**

1. Attacker (a miner or mining pool) performs an offline birthday search and finds `Tx_A`, `Tx_B` with `ProposalShortId(Tx_A) == ProposalShortId(Tx_B)` and `hash(Tx_A) ≠ hash(Tx_B)`.
2. Attacker includes `ProposalShortId(Tx_A)` in a block's proposal set at height `H_p` (proposing `Tx_A`).
3. At height `H_c` (within the commit window `[H_p + w_close, H_p + w_far]`), the attacker mines a block that commits `Tx_B` — a transaction that was **never actually proposed**.
4. `TwoPhaseCommitVerifier` computes `ProposalShortId(Tx_B)`, finds it in `proposal_txs_ids` (because `Tx_A` was proposed with the same short ID), and returns `Ok(())`.
5. All honest nodes accept the block, because they apply the same truncated check.

The codebase itself demonstrates that such collisions are structurally possible — the test suite explicitly constructs `fake_tx` objects with identical `ProposalShortId` but different hashes to test compact-block collision handling:

```rust
// sync/src/relayer/tests/compact_block_process.rs
let fake_tx = missing_tx.clone().fake_hash(fake_hash);
assert_eq!(missing_tx.proposal_short_id(), fake_tx.proposal_short_id());
assert_ne!(missing_tx.hash(), fake_tx.hash());
``` [3](#0-2) 

**Secondary impact — proposer reward miscalculation**

The reward calculator in `util/reward-calculator/src/lib.rs` also maps committed transactions to proposed ones using `ProposalShortId`:

```rust
let committed_idx_proc = |hash: &Byte32| -> Vec<ProposalShortId> {
    store
        .get_block_txs_hashes(hash)
        .into_iter()
        .skip(1)
        .map(|tx_hash| ProposalShortId::from_tx_hash(&tx_hash))
        .collect()
};
// ...
if target_proposals.remove(&id) {
    reward = reward.safe_add(tx_fee.safe_mul_ratio(proposer_ratio)?)?;
}
``` [4](#0-3) 

A collision causes the proposer reward for `Tx_B`'s fee to be attributed to whoever proposed `Tx_A`, not to the actual proposer of `Tx_B`. An attacker who proposes `Tx_A` and commits `Tx_B` collects both the miner reward (for committing `Tx_B`) and the proposer reward (because `Tx_A`'s short ID matches `Tx_B`'s short ID in the reward lookup).

---

### Impact Explanation

- **Consensus rule bypass**: The two-phase commitment requirement — a core CKB protocol invariant — can be violated by any miner willing to invest ~2^40 hash operations. Committed transactions need not have been proposed, undermining the pre-validation and front-running-prevention guarantees of the proposal window.
- **Proposer reward theft**: The attacker double-collects: miner reward for committing `Tx_B` plus proposer reward that should belong to the legitimate proposer of `Tx_B`.
- **Network-wide acceptance**: Because all nodes apply the same truncated check, the malicious block is accepted by the entire network — there is no fork or rejection.

---

### Likelihood Explanation

- The attacker is a miner or mining pool (an unprivileged but realistic role).
- The birthday search requires ~2^40 Blake2b evaluations. This is within reach of a mining pool's existing hardware over hours to days.
- No privileged keys, social engineering, or majority hash power is required.
- The attack is fully offline; the attacker submits a single valid block once the collision is found.

---

### Recommendation

Replace the `ProposalShortId`-based equality check in `TwoPhaseCommitVerifier` with a full 32-byte transaction hash check. Proposals should store (or be verifiable against) the full transaction hash, so that the commit check is:

```
full_tx_hash(committed_tx) ∈ proposal_full_hashes
```

The same fix must be applied to the proposer reward calculator in `util/reward-calculator/src/lib.rs`. If the protocol requires short IDs for bandwidth efficiency in the proposal phase, a separate full-hash binding must be enforced at commit time.

---

### Proof of Concept

```
1. Offline: run birthday search over CKB transaction space.
   For each candidate Tx, vary witness bytes and compute
   ProposalShortId = blake2b(Tx)[0..10].
   Stop when two candidates Tx_A, Tx_B share the same 10-byte prefix.
   Expected work: ~2^40 hashes.

2. At block H_p: mine a block whose proposals field contains
   ProposalShortId(Tx_A).  (Tx_A itself need not be broadcast.)

3. At block H_c in [H_p+2, H_p+10] (default proposal window):
   mine a block whose transaction list includes Tx_B.
   Tx_B was never proposed; its short ID collides with Tx_A's.

4. TwoPhaseCommitVerifier on every node computes:
     committed_ids = { ProposalShortId(Tx_B) }
     proposal_txs_ids ∋ ProposalShortId(Tx_A) == ProposalShortId(Tx_B)
     committed_ids.difference(&proposal_txs_ids) == ∅  → Ok(())

5. Block H_c is accepted by the entire network.
   Tx_B is committed without ever having been proposed.
   Attacker collects miner reward for Tx_B's fee plus
   proposer reward (via reward calculator short-ID match).
```
