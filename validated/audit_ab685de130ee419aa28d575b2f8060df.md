### Title
`break` on `spend_cell` returning `Ok(false)` silently truncates input processing and drops transactions from the rich indexer — (`util/rich-indexer/src/indexer/mod.rs`)

### Summary

In `AsyncRichIndexer::insert_transaction`, when `spend_cell` returns `Ok(false)` (zero rows affected because the referenced output was never indexed — filtered out by `cell_filter`), the code executes `break`, terminating the entire input loop. All subsequent inputs in the same transaction are never processed. This produces two distinct failure modes: (1) indexed outputs after the unindexed input are never marked spent, and (2) if no outputs matched either, the whole transaction is silently dropped from the indexer.

### Finding Description

In `util/rich-indexer/src/indexer/mod.rs` lines 228–248:

```rust
if tx_index != 0 {
    for (input_index, input) in tx_view.inputs().into_iter().enumerate() {
        let out_point = input.previous_output();
        if !spend_cell(&out_point, tx).await? {
            break;   // ← should be `continue`
        }
        ...
    }
}
``` [1](#0-0) 

`spend_cell` in `util/rich-indexer/src/indexer/insert.rs` issues an `UPDATE output SET is_spent = 1 WHERE ...` and returns `Ok(rows_affected > 0)`. [2](#0-1) 

When the referenced output was never inserted into the indexer (because its producing transaction was filtered out by `cell_filter`), zero rows are updated and `Ok(false)` is returned. The `break` then exits the loop, skipping every remaining input in the transaction.

The `is_tx_matched` flag is set to `true` only when an output passes `cell_filter` (lines 217–226) or when an input's referenced output is found in the indexer (lines 234–247). [3](#0-2) 

**Failure mode A — transaction has a matching output:** `is_tx_matched` is already `true`, so the transaction row is written to `ckb_transaction`. However, all inputs after the first unindexed one are silently skipped: their referenced outputs are never marked `is_spent = 1`, and no rows are written to the `input` table for them. Those cells permanently appear unspent in the indexer.

**Failure mode B — transaction has no matching output but has a matching input after an unindexed input:** The output loop leaves `is_tx_matched = false`. The input loop breaks before reaching the matching input. `is_tx_matched` stays `false`, the guard at line 255 fires, and the entire transaction is silently omitted from `ckb_transaction`. [4](#0-3) 

### Impact Explanation

- **Cells appear unspent when they are actually spent.** Any wallet or application querying `get_cells` or `get_transactions` via the rich indexer will see stale live-cell sets and incomplete transaction histories.
- **Transactions are silently omitted.** `get_transactions` will not return transactions that should be present, breaking balance accounting for any address whose cells are involved.
- The indexer database is permanently inconsistent after the affected block is committed; there is no self-healing mechanism.

### Likelihood Explanation

The trigger condition is a transaction whose input list contains at least one input referencing an output that was filtered out by `cell_filter`, followed by at least one input referencing an indexed output (or the transaction has no matching outputs at all). This is a normal, valid transaction structure that any block producer can include. No special privileges, hash power, or key material are required. The bug fires deterministically on every such transaction.

### Recommendation

Replace `break` with `continue` at line 232:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;   // output not indexed; skip this input, process the rest
}
```

This preserves the intended semantics: an unindexed input is simply ignored, but the loop continues so that all remaining inputs are processed and all indexed outputs are correctly marked spent.

### Proof of Concept

1. Configure the rich indexer with a `cell_filter` that matches only script `S`.
2. Produce block N containing transaction T1 with one output using script `S` (indexed) and one output using script `X` (not indexed).
3. Produce block N+1 containing transaction T2 with two inputs: `input[0]` = T1's output-1 (script `X`, not indexed), `input[1]` = T1's output-0 (script `S`, indexed). Give T2 one output using script `S` (matching).
4. Append both blocks to the indexer.
5. **Expected:** T2 appears in `ckb_transaction`; T1's output-0 has `is_spent = 1`; the `input` table has a row for `input[1]`.
6. **Actual:** `spend_cell` for `input[0]` returns `Ok(false)` → `break` → `input[1]` is never processed → T1's output-0 remains `is_spent = 0` → no row in `input` for `input[1]`. In the no-matching-output variant, T2 is absent from `ckb_transaction` entirely.

pub(crate) async fn spend_cell(
    out_point: &OutPoint,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let output_tx_hash = out_point.tx_hash().raw_data().to_vec();
    let output_index: u32 = out_point.index().into();

    let updated_rows = sqlx::query(
        r#"
            UPDATE output
            SET is_spent = 1
            WHERE
                tx_id = (SELECT ckb_transaction.id FROM ckb_transaction WHERE tx_hash = $1)
                AND output_index = $2
        "#,
    )
    .bind(output_tx_hash)
    .bind(output_index as i32)
    .execute(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?
    .rows_affected();

    Ok(updated_rows > 0)
}
