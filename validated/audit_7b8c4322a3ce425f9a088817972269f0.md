### Title
Unbounded `vec!` Allocation in `WeightUnitsFlow::do_estimate` via Attacker-Controlled Fee Rate — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

When the `WeightUnitsFlow` fee estimator is active, `do_estimate` derives a bucket index directly from the highest fee rate in the current mempool with no upper-bound cap, then allocates `vec![0u64; max_bucket_index + 1]` twice. An attacker who submits a transaction with a sufficiently high fee-to-weight ratio can make this index reach tens of millions to hundreds of billions, triggering an OOM allocation that crashes any node that subsequently calls the `estimate_fee_rate` RPC.

---

### Finding Description

`FeeRate::calculate` computes:

```rust
FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
``` [1](#0-0) 

`saturating_mul(1000)` can produce `u64::MAX` for any fee above `u64::MAX / 1000` shannons. Divided by a small `weight` (minimum realistic tx weight is ~100–200 bytes), the resulting `FeeRate` value can be in the range of `10^12` to `u64::MAX`.

`max_bucket_index_by_fee_rate` maps any fee rate above 2,000,000 shannons/KW via:

```rust
x => (x + t * 11_500) / (100 * t),
``` [2](#0-1) 

There is **no upper-bound clamp** on the returned `usize`. For a fee rate of `5 × 10^13` (achievable with ~100,000 CKB fee on a 200-byte tx), the index is ~500 million, requiring a 4 GB allocation. For fee rates near `u64::MAX / weight`, the index reaches ~1.84 × 10^14, requiring petabytes.

`do_estimate` then allocates **two** vectors of this size unconditionally:

```rust
let mut buckets = vec![0u64; max_bucket_index + 1];  // line 219
// ...
let mut buckets = vec![0u64; max_bucket_index + 1];  // line 248
``` [3](#0-2) [4](#0-3) 

The `max_fee_rate` is taken directly from the highest-fee-rate entry in the current mempool (`all_entry_info.pending` / `all_entry_info.proposed`), which is attacker-controlled:

```rust
let max_fee_rate = if let Some(fee_rate) = sorted_current_txs.first().map(|tx| tx.fee_rate) {
    fee_rate
} else {
    return Ok(constants::LOWEST_FEE_RATE);
};
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate);
``` [5](#0-4) 

The `TxStatus` for each pool entry is computed from `TxEntryInfo` via `TxStatus::new_from_entry_info`, which calls `FeeRate::calculate(info.fee, weight)` with no cap: [6](#0-5) 

---

### Impact Explanation

Any node with `WeightUnitsFlow` configured that receives a call to `estimate_fee_rate` (RPC Pool module, enabled by default) after the attacker's high-fee tx is in the mempool will attempt a multi-GB to multi-TB allocation and crash with OOM or a panic. Because the attacker's tx propagates via P2P relay, **every node** with this estimator active is affected simultaneously. The crash is repeatable as long as the tx remains in the pool.

---

### Likelihood Explanation

- `WeightUnitsFlow` is not the default estimator (default is `Dummy`), but it is a documented, production-supported option.
- The attacker must own and burn real CKB. A fee of ~100,000 CKB on a small transaction produces a ~4 GB allocation (reliable OOM). A fee of ~1,000,000 CKB produces ~40 GB.
- No privileged access is required. Transaction submission is open to all P2P participants.
- The `check_tx_fee` guard only enforces a **minimum** fee rate, not a maximum. [7](#0-6) 

- `TRANSACTION_SIZE_LIMIT = 512,000` bytes caps the maximum tx size, but this only sets a floor on `weight`, not a ceiling on `fee_rate`. [8](#0-7) 

---

### Recommendation

1. **Cap `max_bucket_index`** in `do_estimate` to a safe constant (e.g., the index corresponding to the highest bucket defined in `lowest_fee_rate_by_bucket_index`, which is ~`x > 135` → a few hundred at most).
2. **Clamp `FeeRate` values** stored in the fee estimator to a maximum sentinel (e.g., `FeeRate::from_u64(10_000_000)` — 10,000 shannons/byte).
3. **Add a size assertion** before `vec![0u64; max_bucket_index + 1]` and return `Err(Error::NoProperFeeRate)` if the index exceeds a safe threshold.

---

### Proof of Concept

```rust
// Pseudocode unit test
let mut algo = weight_units_flow::Algorithm::new();
algo.update_ibd_state(false);
// Simulate enough blocks for is_ready + historical data
for i in 1..=200 {
    algo.commit_block(&make_block(i));
}
// Accept a tx with fee = 100_000 CKB = 10^13 shannons, size = 200 bytes, cycles = 1
algo.accept_tx(TxEntryInfo { fee: Capacity::shannons(10_000_000_000_000), size: 200, cycles: 1, .. });

// Now simulate estimate_fee_rate with this tx as the only pool entry
// fee_rate = 10^13 * 1000 / 200 = 5 * 10^13
// max_bucket_index = (5*10^13 + 11_500_000) / 100_000 = 500_000_115
// vec![0u64; 500_000_116] = ~4 GB allocation → OOM
let result = algo.estimate_fee_rate(6, pool_with_attacker_tx);
// Node crashes before returning
```

The `max_bucket_index_by_fee_rate` function and the uncapped `vec!` allocation are the root cause. No consensus rule, PoW check, or tx-pool admission guard prevents a high-fee transaction from reaching the estimator with an unbounded fee rate. [2](#0-1) [9](#0-8)

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
