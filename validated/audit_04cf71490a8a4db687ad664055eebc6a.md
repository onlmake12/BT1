### Title
`ScriptHashTypeVerifier` Omits Type Script Hash-Type Validation, Allowing Non-Permitted Values Through — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only on each output's **lock script**, completely skipping the same check on the output's **type script**. A transaction sender can craft outputs whose type script carries any structurally-valid but consensus-disallowed `ScriptHashType` byte (e.g., `6`, `8`, `10`, …) and the verifier will accept it without error.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the only four permitted values:

```
{0 = Data, 1 = Type, 2 = Data1, 4 = Data2}
``` [1](#0-0) 

The lower-level structural check `ScriptHashType::verify_value()` is far more permissive — it accepts **any even byte or the value 1** (i.e., 0, 1, 2, 4, 6, 8, 10, …, 254): [2](#0-1) 

`check_data` on `CellOutputReader` calls `verify_value` for both lock and type scripts, so a type script with `hash_type = 6` passes the molecule-level structural check: [3](#0-2) 

`ScriptHashTypeVerifier::verify()` is the dedicated consensus-level gate that is supposed to enforce `ENABLED_SCRIPT_HASH_TYPE`. It iterates over every output and checks the **lock** script's hash type — but never touches the **type** script:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...);
            }
        } else {
            return Err(...);
        }
        // ← output.type_().to_opt() is never examined
    }
    Ok(())
}
``` [4](#0-3) 

The type script path is entirely absent. A transaction output with `type_script.hash_type = 6` (or any even value ≥ 6) will pass `ScriptHashTypeVerifier` without error.

---

### Impact Explanation

1. **Tx-pool pollution / resource waste**: Transactions carrying a disallowed type-script hash type pass non-contextual verification and are admitted to the tx-pool. They consume memory and CPU in the pool until eviction.

2. **Miner block-assembly waste**: A miner's block template can include such a transaction. When the block is validated by the script engine, `select_version()` returns `ScriptError::InvalidScriptHashType` or `ScriptError::InvalidVmVersion`, causing the entire block to be rejected and the miner's work to be discarded. [5](#0-4) 

3. **Inconsistency window**: The gap between what `ScriptHashTypeVerifier` accepts (lock-only) and what the script engine rejects (both lock and type) creates a class of transactions that are "valid" at the pool/relay layer but "invalid" at execution — a semantic inconsistency that can be exploited for targeted miner griefing.

---

### Likelihood Explanation

Any unprivileged RPC caller submitting via `send_raw_transaction`, or any peer relaying a transaction, can trigger this path. Constructing a transaction with a type script whose `hash_type` byte is `6` requires no special privilege and is trivially achievable with any CKB transaction builder. The attack is low-cost and repeatable.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's hash type for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock-script check
        let lock_hash_type = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?;
        let val: u8 = lock_hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }

        // NEW: type-script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_hash_type = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?;
            let val: u8 = type_hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        }
    }
    Ok(())
}
```

### Proof of Concept

1. Build a transaction with one output whose type script has `hash_type = 0x06` (structurally valid per `verify_value`, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_raw_transaction` RPC.
3. Observe: `ScriptHashTypeVerifier::verify()` returns `Ok(())` — the transaction is accepted into the pool.
4. Observe: when the script engine later calls `select_version()` on the type script, it returns `ScriptError::InvalidScriptHashType("The ScriptHashType/Data3 has not been activated…")`.
5. The transaction (or any block containing it) is rejected at execution time, but the pool admission check never caught it. [4](#0-3) [1](#0-0) [5](#0-4)
