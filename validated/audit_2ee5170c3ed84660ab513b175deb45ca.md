Based on my full code review, this is a real, demonstrable vulnerability. Here is the analysis:

### Title
Fees Deposited via `deposit_fee_coins` Are Permanently Locked in `PriceInfoObject` with No Recovery Path — (`target_chains/sui/contracts/sources/price_info.move`)

### Summary

Every call to `update_single_price_feed` deposits the caller's fee `Coin<SUI>` into the target `PriceInfoObject` via `deposit_fee_coins`. No function in the entire contract suite — including all governance modules — can withdraw or transfer those coins. The `fee_recipient_address` field in `State` is a dead field: governance can update it, but it is never read during fee collection or disbursement. All collected fees are permanently locked.

### Finding Description

`update_single_price_feed` unconditionally calls `price_info::deposit_fee_coins`, which stores the fee coin as a dynamic object field (`FEE_STORAGE_KEY`) on the `PriceInfoObject`: [1](#0-0) 

The first call uses `dynamic_object_field::add`; subsequent calls use `coin::join` on the existing coin. Neither path emits an event, returns the amount, or transfers funds anywhere. [2](#0-1) 

`get_balance` can read the accumulated balance, but there is no corresponding withdrawal function anywhere in the module: [3](#0-2) 

The `State` object stores a `fee_recipient_address` and governance can update it via `set_fee_recipient`, but this address is **never read** during fee deposit or at any other point in the fee lifecycle: [4](#0-3) [5](#0-4) 

The comment in `set_fee_recipient.move` is explicit: *"The previous version of the contract sent the fees to a recipient address but this state is not used anymore."* No replacement disbursement mechanism was added. [6](#0-5) 

A search across all governance modules (`governance.move`, `governance_action.move`, `set_fee_recipient.move`, `set_update_fee.move`, `set_data_sources.move`, `set_stale_price_threshold.move`, `contract_upgrade.move`) confirms that **none of them access `FEE_STORAGE_KEY`** or call any function that moves coins out of a `PriceInfoObject`: [7](#0-6) 

### Impact Explanation

Every price update permanently locks `base_update_fee` MIST of SUI inside the corresponding `PriceInfoObject`. With N price feeds each updated M times, the total locked amount is `N × M × base_update_fee`. These funds are irrecoverable: no privileged or unprivileged path exists to extract them. Updating `fee_recipient_address` via governance has zero effect on already-locked or future fees.

### Likelihood Explanation

This is triggered by every normal, unprivileged call to `update_single_price_feed` — the primary user-facing entry point for price updates. It requires no special conditions and is already happening on every deployment.

### Recommendation

Add a privileged `withdraw_fees(price_info_object: &mut PriceInfoObject, ctx: &mut TxContext)` function (gated on `LatestOnly` or a governance capability) that removes the `FEE_STORAGE_KEY` dynamic field and transfers the coin to `fee_recipient_address`. Alternatively, transfer the fee coin directly to `fee_recipient_address` inside `deposit_fee_coins` rather than storing it, which is consistent with how the previous contract version behaved per the comment in `set_fee_recipient.move`.

### Proof of Concept

1. Deploy the contract with `base_update_fee = 50`.
2. Call `update_single_price_feed` once → `get_balance(price_info_object) == 50`.
3. Call `update_single_price_feed` again → `get_balance(price_info_object) == 100`.
4. Search all `.move` files for any function that calls `dynamic_object_field::remove` or `coin::take` on `FEE_STORAGE_KEY` → zero results.
5. Call `execute_governance_instruction` with a `SET_FEE_RECIPIENT` payload → `fee_recipient_address` in `State` changes, but `get_balance(price_info_object)` is unchanged and no transfer occurs.

The existing test at `pyth.move:817` already asserts `get_balance == DEFAULT_BASE_UPDATE_FEE` after one update, confirming the accumulation behavior, but no test or production code ever drains that balance. [8](#0-7)

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
