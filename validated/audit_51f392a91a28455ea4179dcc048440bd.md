### Title
Division by Zero in `calculateTwap` When `publishSlot` Values Are Equal - (`target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

The EVM `Pyth.sol` contract's `validateTwapPriceInfo` function fails to reject TWAP message pairs where `start.publishSlot == end.publishSlot`. When this passes validation, `calculateTwap` computes `slotDiff = 0` and immediately divides by it, causing an unhandled Solidity panic (division by zero). The Solana receiver correctly enforces `end_msg.publish_slot > start_msg.publish_slot` (strictly greater), but the EVM contract only checks `start > end`, silently allowing the equal-slot case through.

### Finding Description

In `validateTwapPriceInfo`, the slot ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

This only rejects the case where start slot is strictly greater than end slot. When `start.publishSlot == end.publishSlot`, validation passes. Control then flows to `calculateTwap`, where:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot;
// slotDiff == 0 when slots are equal
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // panic: division by zero
uint128 twapConf  = confDiff  / uint128(slotDiff);           // panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // panic: division by zero
``` [2](#0-1) 

By contrast, the Solana receiver enforces strict inequality at the validation boundary:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The Solana path therefore returns a clean, typed error (`InvalidTwapSlots`) rather than panicking. The EVM path has no equivalent guard.

### Impact Explanation

Any call to `parseTwapPriceFeedUpdates` that supplies a start/end TWAP pair with identical `publishSlot` values will revert with a Solidity panic (error code `0x12`) rather than a descriptive custom error. Integrating contracts that catch specific custom errors (e.g., `InvalidTwapUpdateDataSet`) will not catch this panic, breaking their error-handling logic. If Hermes ever emits two TWAP snapshots at the same slot (a degenerate but not impossible edge case), the TWAP feature becomes completely unusable for that data pair, and any downstream protocol relying on it is blocked. [4](#0-3) 

### Likelihood Explanation

The entry point is `parseTwapPriceFeedUpdates`, callable by any unprivileged transaction sender. The data must be Wormhole-guardian-signed, so an attacker cannot forge arbitrary TWAP messages. However, the Hermes service could legitimately produce a same-slot pair (e.g., if the start and end snapshots are captured within the same Pythnet slot due to timing), and any user submitting that valid VAA pair would trigger the panic. The risk is low-to-moderate but the consequence is a hard revert with no graceful fallback. [5](#0-4) 

### Recommendation

Add an explicit equal-slot check in `validateTwapPriceInfo`, mirroring the Solana receiver's strict inequality:

```solidity
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This ensures `slotDiff` is always at least 1 before any division occurs, eliminating the panic path and aligning EVM behavior with the Solana implementation. [1](#0-0) 

### Proof of Concept

1. Obtain (or wait for Hermes to produce) two valid Wormhole-signed TWAP messages for the same feed where both have `publishSlot = S` for some slot `S`, with `prevPublishTime < publishTime` satisfied for each.
2. Call `parseTwapPriceFeedUpdates{value: fee}(updateData, priceIds)` on the EVM Pyth contract.
3. `validateTwapPriceInfo` passes because `S > S` is false.
4. `calculateTwap` computes `slotDiff = S - S = 0`.
5. `priceDiff / int128(uint128(0))` triggers a Solidity panic with code `0x12` (division by zero), reverting the transaction with an untyped panic rather than a descriptive error. [6](#0-5)

    function validateTwapPriceInfo(
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure {
        // First validate each individual price's uniqueness
        if (
            twapPriceInfoStart.prevPublishTime >= twapPriceInfoStart.publishTime
        ) {
            revert PythErrors.InvalidTwapUpdateData();
        }
        if (twapPriceInfoEnd.prevPublishTime >= twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateData();
        }

        // Then validate the relationship between the two data points
        if (twapPriceInfoStart.expo != twapPriceInfoEnd.expo) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
        if (twapPriceInfoStart.publishTime > twapPriceInfoEnd.publishTime) {
            revert PythErrors.InvalidTwapUpdateDataSet();
        }
    }

```text
    function calculateTwap(
        bytes32 priceId,
        PythStructs.TwapPriceInfo memory twapPriceInfoStart,
        PythStructs.TwapPriceInfo memory twapPriceInfoEnd
    ) private pure returns (PythStructs.TwapPriceFeed memory twapPriceFeed) {
        twapPriceFeed.id = priceId;
        twapPriceFeed.startTime = twapPriceInfoStart.publishTime;
        twapPriceFeed.endTime = twapPriceInfoEnd.publishTime;

        // Calculate differences between start and end points for slots and cumulative values
        uint64 slotDiff = twapPriceInfoEnd.publishSlot -
            twapPriceInfoStart.publishSlot;
        int128 priceDiff = twapPriceInfoEnd.cumulativePrice -
            twapPriceInfoStart.cumulativePrice;
        uint128 confDiff = twapPriceInfoEnd.cumulativeConf -
            twapPriceInfoStart.cumulativeConf;

        // Calculate time-weighted average price (TWAP) and confidence by dividing
        // the difference in cumulative values by the number of slots between data points
        int128 twapPrice = priceDiff / int128(uint128(slotDiff));
        uint128 twapConf = confDiff / uint128(slotDiff);

        // The conversion from int128 to int64 is safe because:
        // 1. Individual prices fit within int64 by protocol design
        // 2. TWAP is essentially an average price over time (cumulativePrice₂-cumulativePrice₁)/slotDiff
        // 3. This average must be within the range of individual prices that went into the calculation
        // We use int128 only as an intermediate type to safely handle cumulative sums
        twapPriceFeed.twap.price = int64(twapPrice);
        twapPriceFeed.twap.conf = uint64(twapConf);
        twapPriceFeed.twap.expo = twapPriceInfoStart.expo;
        twapPriceFeed.twap.publishTime = twapPriceInfoEnd.publishTime;

        // Calculate downSlotsRatio as a value between 0 and 1,000,000
        // 0 means no slots were missed, 1,000,000 means all slots were missed
        uint64 totalDownSlots = twapPriceInfoEnd.numDownSlots -
            twapPriceInfoStart.numDownSlots;
        uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff;

        // Safely downcast to uint32 (sufficient for value range 0-1,000,000)
        twapPriceFeed.downSlotsRatio = uint32(downSlotsRatio);

        return twapPriceFeed;
```
