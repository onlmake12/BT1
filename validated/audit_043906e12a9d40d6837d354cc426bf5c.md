### Title
Minimum Fee Check Ignores Cycle Cost, Allowing Under-Priced High-Cycle Transactions Into the Pool — (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the transaction's serialized byte size, ignoring the cycle cost. Because cycles are only known after script verification, and there is no post-verification fee-rate re-check using the full weight, any unprivileged `send_transaction` caller can submit a small-size but high-cycle transaction that passes the admission gate with a fee far below what the weight-based minimum fee rate requires.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee as:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [1](#0-0) 

The actual block capacity consumed by a transaction is defined in `util/types/src/core/tx_pool.rs` as:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [2](#0-1) 

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. [3](#0-2) 

The processing pipeline in `_process_tx` (`tx-pool/src/process.rs`) is:

1. `pre_check` → calls `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` — size-only gate, cycles unknown
2. `verify_rtx` → actual cycles become known
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry created with real cycles
4. `submit_entry` — **no secondary fee-rate check using `get_transaction_weight(tx_size, verified.cycles)`** [4](#0-3) 

`TxEntry::fee_rate()` does use the full weight for sorting:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [5](#0-4) 

But this is only used for ordering within the pool — there is no enforcement that the actual fee rate meets `min_fee_rate` after cycles are known.

---

### Impact Explanation

Consider a transaction with `tx_size = 100` bytes and `cycles = 10,000,000`:

- **Weight** = `max(100, 10,000,000 × 0.000_170_571_4)` = `max(100, 1705)` = **1705 bytes**
- **Minimum fee required by weight**: `min_fee_rate × 1705 / 1000`
- **Minimum fee enforced by `check_tx_fee`**: `min_fee_rate × 100 / 1000` — **~17× less**

Such a transaction is admitted to the pool. It:
- Consumes full CPU cost of script verification
- Occupies pool memory
- Consumes 1705 weight-units of block capacity while paying for only 100
- Crowds out legitimate transactions that paid the correct fee rate
- Causes miners who include it to receive less fee per unit of cycle budget consumed than the configured minimum

The gap scales linearly with cycles: a transaction running at the block cycle limit (`10,000,000,000` cycles) with a 100-byte body would require a fee ~170,000× below the weight-based minimum to pass admission.

---

### Likelihood Explanation

Any unprivileged RPC caller (`send_transaction`) or P2P transaction relayer can trigger this. Crafting a small script that runs a tight loop for many cycles is trivial in CKB-VM (RISC-V). No special privileges, keys, or majority hashpower are required. The attack is repeatable on every submission.

---

### Recommendation

After `verify_rtx` returns the actual cycle count, re-check the fee rate using the full weight before calling `submit_entry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Post-verification fee-rate re-check using actual weight
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight as u64);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}

let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

This mirrors the fix pattern from the ELFI M-17 report: account for the previously-untracked resource cost (cycles, analogous to `processExecutionFee()` gas) in the compensation/admission check.

---

### Proof of Concept

1. Craft a CKB transaction whose lock script runs a tight RISC-V loop consuming ~10,000,000 cycles; keep the script binary small (~100 bytes serialized).
2. Set `fee = min_fee_rate × 100 / 1000` shannons (just enough to pass the size-based gate).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` passes: `fee >= min_fee_rate.fee(100)`. ✓
5. `verify_rtx` completes; `verified.cycles = 10_000_000`.
6. `TxEntry` is created and admitted to the pool with no further fee-rate check.
7. The entry's actual `fee_rate()` = `fee / 1705 × 1000` ≈ `min_fee_rate / 17` — far below the configured minimum.
8. Repeat to fill the pool with under-priced high-cycle transactions, displacing legitimate traffic.
