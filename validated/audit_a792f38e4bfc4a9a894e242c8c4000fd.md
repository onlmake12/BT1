### Title
Zero Price Treated as Sentinel Skips Deviation Check, Suppressing Critical Price Updates — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function skips the deviation check for any price feed where either the current or previous price is zero, using `continue` to avoid a division-by-zero. This overloads zero as a sentinel value, identical to the GMX M-21 pattern. When a Pyth price feed's value drops to zero (or was previously zero), a deviation-only subscription will never trigger an update — the loop exhausts all feeds without returning, and the function reverts with `UpdateConditionsNotMet`. Downstream consumers of the subscription then read the stale, non-zero stored price instead of the actual zero price.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation check loop is:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;
}
``` [1](#0-0) 

When `currentPrice == 0` (price has crashed to zero) or `previousPrice == 0` (the stored price was zero), the entire deviation comparison for that feed is skipped. If all feeds in the subscription hit this condition, the loop completes without ever returning `updateTimestamp`, and execution falls through to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

This means `updatePriceFeeds()` reverts, the new price is not stored, and `_state.priceUpdates[subscriptionId][priceId]` continues to hold the old non-zero price. [3](#0-2) 

The root cause is identical to GMX M-21: zero is overloaded as both a legitimate price value and a sentinel meaning "skip this comparison."

---

### Impact Explanation

Subscribers using `updateOnDeviation: true` (with or without heartbeat) will not receive an on-chain price update when a feed's price drops to zero. Any protocol reading prices from the Scheduler via `getPricesUnsafe` or `getPricesNoOlderThan` will receive the stale, pre-crash price. This is directly analogous to M-21: users of the subscription read an incorrect (older, non-zero) price at the exact moment the true price is zero, which can cause mispriced liquidations, incorrect collateral valuations, or other financial errors in downstream DeFi protocols.

---

### Likelihood Explanation

- Pyth price feeds are `int64`, so a price of zero is a valid on-chain value (e.g., a newly listed or delisted asset, or an asset in extreme distress).
- Any unprivileged keeper calling `updatePriceFeeds()` with a valid Pyth VAA containing a zero price will trigger this path.
- Subscriptions configured with `updateOnDeviation: true` and `updateOnHeartbeat: false` are fully blocked from updating when this occurs.
- Subscriptions with both criteria enabled are partially protected (heartbeat will eventually trigger), but the deviation-triggered update is still suppressed, meaning the zero price is not stored until the heartbeat fires.

---

### Recommendation

Replace the `continue` sentinel pattern with an explicit trigger: if either price is zero, treat it as an infinite deviation (i.e., always trigger the update), rather than silently skipping the feed.

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A price of zero is a valid extreme value; treat as maximum deviation
    return updateTimestamp;
}
```

This mirrors the recommendation in GMX M-21: use an actual sentinel flag (or explicit logic) rather than overloading the meaning of a zero price.

---

### Proof of Concept

1. Create a Scheduler subscription with `updateOnDeviation: true`, `updateOnHeartbeat: false`, `deviationThresholdBps: 100`.
2. Submit a first `updatePriceFeeds` call with `price = 1000` for all feeds. This sets `previousPrice = 1000` in storage.
3. Submit a second `updatePriceFeeds` call with a valid Pyth VAA where `price = 0` for all feeds (a price crash to zero).
4. Observe: `_validateShouldUpdatePrices` enters the deviation loop, hits `previousPrice == 0 || currentPrice == 0` (since `currentPrice == 0`), executes `continue` for every feed, exits the loop, and reverts with `UpdateConditionsNotMet`.
5. `getPricesUnsafe` on the subscription still returns `price = 1000` — the stale pre-crash value — even though the true Pyth price is zero. [4](#0-3)
