The bug at line 252 is real and concrete. Let me confirm the exact mechanics before rendering a verdict.

### Title
Incorrect variable reference in `script_exists_in_output` causes type scripts to be deleted during rollback on PostgreSQL — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`script_exists_in_output` in `remove.rs` reads from the wrong row variable (`row_lock` instead of `row_type`) when evaluating the second EXISTS check on PostgreSQL. During any `rollback_block`, type scripts that are still referenced by surviving outputs are incorrectly deleted from the `script` table, permanently corrupting the rich-indexer's view of live cell ownership.

---

### Finding Description

`script_exists_in_output` performs two SQL EXISTS queries:

1. **`row_lock`** — checks whether `script_id` appears as a `lock_script_id` in any remaining `output` row. [1](#0-0) 

2. If `row_lock` is true, the function returns `Ok(true)` early. [2](#0-1) 

3. **`row_type`** — checks whether `script_id` appears as a `type_script_id` in any remaining `output` row. [3](#0-2) 

4. **The bug**: the final match at line 252 reads `row_lock.try_get::<bool, _>(0)` instead of `row_type.try_get::<bool, _>(0)`. [4](#0-3) 

Because execution only reaches line 252 when `row_lock` already returned false (otherwise the early return at line 226 would have fired), on **PostgreSQL** `row_lock.try_get::<bool, _>(0)` always yields `Ok(false)` here — the `row_type` result is never consulted. The function unconditionally returns `Ok(false)` for any script that is used exclusively as a type script, regardless of whether surviving outputs still reference it.

On **SQLite** the bug is accidentally masked: `try_get::<bool, _>(0)` fails with `Err(_)` because SQLite stores the EXISTS result as `BIGINT`, so the error branch at line 255 falls through to `row_type.get::<i64, _>(0) == 1`, which is correct. [5](#0-4) 

The caller in `rollback_block` collects every script for which `script_exists_in_output` returns false and bulk-deletes them from the `script` table. [6](#0-5) 

---

### Impact Explanation

On PostgreSQL, any reorg that rolls back a block whose outputs share a type script with outputs in surviving blocks will silently delete that type script row from the `script` table. All subsequent `get_cells` and `get_cells_capacity` queries that filter by that type script will return empty or incorrect results, permanently corrupting the indexer's view of live cell ownership until the database is rebuilt from scratch.

---

### Likelihood Explanation

Reorgs are a normal, unprivileged event on any live CKB network. A block relayer needs only to propagate a valid competing chain of equal or greater work. No special privileges, leaked keys, or majority hashpower are required. The precondition — two blocks sharing a type script — is trivially satisfied by any token/UDT deployment where multiple cells carry the same type script across different blocks.

---

### Recommendation

Change line 252 from:

```rust
match row_lock.try_get::<bool, _>(0) {
```

to:

```rust
match row_type.try_get::<bool, _>(0) {
```

This makes the second branch correctly evaluate the `type_script_id` EXISTS result instead of re-reading the already-evaluated `lock_script_id` result.

---

### Proof of Concept

State-transition test (PostgreSQL backend):

1. Index **Block A** containing `output_A` with `type_script = S` and `lock_script = L_A`.
2. Index **Block B** (child of A) containing `output_B` with `type_script = S` and `lock_script = L_B`.
3. Call `rollback()` to remove Block B.
4. Query `SELECT id FROM script WHERE id = <id_of_S>`.

**Expected**: row exists (Block A's `output_A` still references `S`).  
**Actual (PostgreSQL)**: row is absent — `script_exists_in_output` returned `false` for `S` because `row_lock` (lock check) was false and the type check result in `row_type` was never read.
