### Title
Cellbase Witness Lock Not Validated Against Node's Block Assembler Config in `submit_block` — (File: `rpc/src/module/miner.rs`)

---

### Summary

The `submit_block` RPC accepts a block whose cellbase witness lock (the script that will receive the current block's mining reward) is never validated against the node's configured `block_assembler.args`. Any caller with access to the Miner RPC — including a malicious or compromised miner client — can submit a block with an arbitrary cellbase witness lock, silently redirecting the node operator's future mining reward to an attacker-controlled address.

---

### Finding Description

CKB uses a Client-Server (CS) mining architecture. The node operator configures their reward address via `block_assembler.args` in `ckb.toml`. When `get_block_template` is called, `build_cellbase_witness` constructs the cellbase witness from this config:

```rust
// tx-pool/src/block_assembler/mod.rs:495-498
let cellbase_lock = Script::new_builder()
    .args(config.args.as_bytes())
    .code_hash(&config.code_hash)
    .hash_type(hash_type)
    .build();
```

The cellbase witness lock embedded in the template is the node operator's intended reward address. When the miner client solves the PoW and calls `submit_block`, the node's implementation does the following:

```rust
// rpc/src/module/miner.rs:260-298
fn submit_block(&self, work_id: String, block: Block) -> Result<H256> {
    // ...
    HeaderVerifier::new(snapshot, consensus).verify(&header)...;
    // parent check...
    self.chain.blocking_process_block(Arc::clone(&block))...;
    // No check of cellbase witness against block_assembler config
}
```

The `CellbaseVerifier` invoked during block processing only validates structural constraints on the cellbase witness — that it is a valid `CellbaseWitness` molecule encoding and that the lock's `hash_type` is an enabled value:

```rust
// verification/src/block_verifier.rs:106-124
if cellbase_transaction.witnesses().get(0)
    .and_then(|witness| {
        CellbaseWitness::from_slice(&witness.raw_data()).ok()
            .and_then(|cellbase_witness| {
                ScriptHashType::try_from(cellbase_witness.lock().hash_type()).ok()
                    .and_then(|hash_type| {
                        let val: u8 = hash_type.into();
                        ENABLED_SCRIPT_HASH_TYPE.contains(&val).then_some(())
                    })
            })
    }).is_none()
{
    return Err((CellbaseError::InvalidWitness).into());
}
```

The `RewardVerifier` only checks that the cellbase **output** lock of the current block matches the **target** block's previously committed cellbase witness lock — it enforces payment of past rewards, not the identity of the current block's reward recipient:

```rust
// verification/contextual/src/contextual_block_verifier.rs:262-271
if cellbase.transaction.outputs().get(0)
    .expect("cellbase should have output")
    .lock() != target_lock
{
    return Err((CellbaseError::InvalidRewardTarget).into());
}
```

No code path in the entire block submission pipeline compares the submitted block's cellbase witness lock against the node's `BlockAssemblerConfig.args`. The `work_id` parameter is similarly never validated — it is only used in log messages — but its impact is limited to observability.

---

### Impact Explanation

The cellbase witness lock in block N determines who receives block N's mining reward, finalized at block N + `PROPOSAL_WINDOW.farthest + 1`. A malicious miner client that modifies the cellbase witness before submitting a solved block causes the node operator to permanently lose that block's entire reward (primary issuance + secondary issuance + transaction fees + proposal rewards). The node accepts and propagates the block without any indication that the reward destination was altered. The loss is irreversible once the reward is finalized on-chain.

---

### Likelihood Explanation

The Miner RPC is typically bound to localhost, but the attack surface is realistic:

1. **Malicious miner client software**: A user who installs a third-party miner binary that silently replaces the cellbase witness lock with the attacker's address before calling `submit_block`. The node has no defense.
2. **Compromised miner client**: A supply-chain attack on the miner binary.
3. **Exposed RPC**: Node operators who expose the Miner RPC port to a LAN or the internet (common in pool setups) allow any reachable caller to submit a modified block.

No privileged access, leaked keys, or majority hashpower is required. The attacker only needs to reach the Miner RPC and supply a valid PoW solution — which the miner client already computes.

---

### Recommendation

In `submit_block` (`rpc/src/module/miner.rs`), after deserializing the block, extract the cellbase witness lock and compare it against the node's active `BlockAssemblerConfig`. If the node has a block assembler configured and the submitted block's cellbase witness lock does not match `config.code_hash`, `config.hash_type`, and `config.args`, reject the submission with a clear error. If the node has no block assembler configured (mining disabled), the check can be skipped or the submission rejected outright.

---

### Proof of Concept

1. Start a CKB node with `block_assembler.args = "0xLEGITIMATE_OPERATOR_PUBKEY_HASH"`.
2. Call `get_block_template` — the returned `cellbase.data.witnesses[0]` encodes a `CellbaseWitness` whose `lock.args` is `0xLEGITIMATE_OPERATOR_PUBKEY_HASH`.
3. Decode the cellbase witness, replace `lock.args` with `0xATTACKER_PUBKEY_HASH`, re-encode, and substitute it into the block.
4. Solve the PoW over the modified block (the transactions root changes because the cellbase witness changed).
5. Call `submit_block` with the modified block and any arbitrary `work_id` string.
6. The node accepts the block: `HeaderVerifier` passes (PoW is valid), `CellbaseVerifier` passes (witness is structurally valid with an enabled hash type), `RewardVerifier` passes (the cellbase output lock correctly pays the *previous* target block's reward).
7. After `PROPOSAL_WINDOW.farthest + 1` blocks, the reward for the submitted block is finalized to `0xATTACKER_PUBKEY_HASH`. The node operator receives nothing.

**Root cause**: `submit_block` in `rpc/src/module/miner.rs` never reads `self.shared`'s block assembler config to compare against the submitted block's cellbase witness lock. [1](#0-0)  The `CellbaseVerifier` only enforces structural validity of the witness, not its identity. [2](#0-1)  The `build_cellbase_witness` function shows the intended binding between `BlockAssemblerConfig.args` and the cellbase witness lock, but this binding is only enforced at template-generation time, not at submission time. [3](#0-2)  The `RewardVerifier` enforces payment of past rewards but does not constrain the current block's witness lock. [4](#0-3)

    pub(crate) fn build_cellbase_witness(
        config: &BlockAssemblerConfig,
        snapshot: &Snapshot,
    ) -> CellbaseWitness {
        let hash_type: ScriptHashType = config.hash_type.into();
        let cellbase_lock = Script::new_builder()
            .args(config.args.as_bytes())
            .code_hash(&config.code_hash)
            .hash_type(hash_type)
            .build();
        let tip = snapshot.tip_header();

        let mut message = vec![];
        if let Some(version) = snapshot.compute_versionbits(tip) {
            message.extend_from_slice(&version.to_le_bytes());
            message.extend_from_slice(b" ");
        }
        if config.use_binary_version_as_message_prefix {
            message.extend_from_slice(config.binary_version.as_bytes());
        }
        if !config.message.is_empty() {
            message.extend_from_slice(b" ");
            message.extend_from_slice(config.message.as_bytes());
        }

        CellbaseWitness::new_builder()
            .lock(cellbase_lock)
            .message(message)
            .build()
    }

    pub fn verify(&self) -> Result<(), Error> {
        let cellbase = &self.resolved[0];
        let no_finalization_target =
            (self.parent.number() + 1) <= self.context.consensus.finalization_delay_length();

        let (target_lock, block_reward) = self.context.finalize_block_reward(self.parent)?;
        let output = CellOutput::new_builder()
            .capacity(block_reward.total)
            .lock(target_lock.clone())
            .build();
        let insufficient_reward_to_create_cell = output.is_lack_of_capacity(Capacity::zero())?;

        if no_finalization_target || insufficient_reward_to_create_cell {
            let ret = if cellbase.transaction.outputs().is_empty() {
                Ok(())
            } else {
                Err((CellbaseError::InvalidRewardTarget).into())
            };
            return ret;
        }

        if !insufficient_reward_to_create_cell {
            if cellbase.transaction.outputs_capacity()? != block_reward.total {
                return Err((CellbaseError::InvalidRewardAmount).into());
            }
            if cellbase
                .transaction
                .outputs()
                .get(0)
                .expect("cellbase should have output")
                .lock()
                != target_lock
            {
                return Err((CellbaseError::InvalidRewardTarget).into());
            }
        }

        Ok(())
    }
