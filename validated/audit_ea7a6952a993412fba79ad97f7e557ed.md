### Title
Unbounded `feeInWei` in `setProviderFee()` Allows Provider to Permanently Grief Users via Arithmetic Overflow in `getFeeV2()` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `setProviderFee()` function in the Entropy contract accepts any `uint128` value for `newFeeInWei` with no upper-bound validation. A registered provider can set `feeInWei = type(uint128).max`, causing `getFeeV2()` to either revert with arithmetic overflow (when `pythFeeInWei > 0`) or return an impossibly large value that no user can ever satisfy, permanently blocking all new randomness requests to that provider.

---

### Finding Description

`setProviderFee()` writes the caller-supplied value directly to storage with no range check:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) {
        revert EntropyErrors.NoSuchProvider();
    }
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;   // ← no upper-bound check
    ...
}
``` [1](#0-0) 

Every new request flows through `requestHelper`, which calls `getFeeV2`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [2](#0-1) 

`getFeeV2` is:

```solidity
function getFeeV2(address provider, uint32 gasLimit) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [3](#0-2) 

`getProviderFee` with `gasLimit = 0` (or any `gasLimit ≤ defaultGasLimit`) returns `provider.feeInWei` directly:

```solidity
} else {
    return provider.feeInWei;
}
``` [4](#0-3) 

Two failure modes result:

- **Case A (`pythFeeInWei > 0`):** `type(uint128).max + pythFeeInWei` overflows `uint128` in Solidity 0.8+, causing `getFeeV2` to revert with a panic. Every call to `requestHelper` (and therefore every `requestV2` / `requestWithCallback` variant) reverts unconditionally.
- **Case B (`pythFeeInWei == 0`):** `getFeeV2` returns `type(uint128).max` (~3.4 × 10³⁸ wei, far exceeding total ETH supply). The `msg.value < requiredFee` check always holds, so every request reverts with `InsufficientFee`.

By contrast, `setDefaultGasLimit` **does** validate its input against `MAX_GAS_LIMIT`:

```solidity
// Check that we can round the gas limit into the 10k gas. This reverts
// if the provided value exceeds the max.
roundTo10kGas(gasLimit);
``` [5](#0-4) 

No equivalent guard exists for `feeInWei`.

---

### Impact Explanation

Any smart contract that hardcodes a provider address (a common pattern) is permanently unable to request new randomness once the provider sets `feeInWei = type(uint128).max`. Lottery, gaming, or NFT-mint contracts that hold user funds and depend on a specific provider for resolution become permanently stuck: users cannot trigger new draws, and funds locked inside those contracts are unrecoverable. The provider can execute this with a single transaction and zero cost beyond gas.

---

### Likelihood Explanation

Providers register permissionlessly. A malicious actor can register, attract integrators by advertising a low fee, and then call `setProviderFee(type(uint128).max)` to grief all downstream users. The attack requires no special privilege beyond being a registered provider, which is open to anyone. The `setProviderFeeAsFeeManager` path is equally unvalidated, so a compromised fee-manager account is a second entry point. [6](#0-5) 

---

### Recommendation

Add an upper-bound check in both `setProviderFee` and `setProviderFeeAsFeeManager`, mirroring the pattern already used in `setDefaultGasLimit`:

```solidity
uint128 constant MAX_FEE_IN_WEI = type(uint128).max / 2; // or a protocol-defined cap

function setProviderFee(uint128 newFeeInWei) external override {
    if (newFeeInWei > MAX_FEE_IN_WEI) revert EntropyErrors.FeeTooHigh();
    ...
}
```

This prevents the overflow in `getFeeV2` and ensures the fee can never exceed a value that is practically payable.

---

### Proof of Concept

```solidity
// 1. Malicious provider sets fee to type(uint128).max
vm.prank(maliciousProvider);
entropy.setProviderFee(type(uint128).max);

// 2. Any user attempting a new request reverts
uint256 fee = entropy.getFeeV2(maliciousProvider, 0);
// ↑ reverts with arithmetic overflow if pythFeeInWei > 0,
//   or returns type(uint128).max which no msg.value can satisfy.

vm.deal(user, type(uint128).max); // impossible in practice
vm.prank(user);
vm.expectRevert(EntropyErrors.InsufficientFee.selector);
entropy.requestV2{value: 0}(maliciousProvider, bytes32(uint(42)), 0);
// All new requests to maliciousProvider are permanently blocked.
```
