**Call site — every `SessionOpen`:** [2](#0-1) 

**Analysis of the eviction logic:**

The `index & 0x1 != 0` filter evicts every peer at an **odd index** from the full collected list — approximately **50% of all current non-whitelist inbound peers** — on a single new connection event. This is not evicting 1 peer to make room; it is a mass eviction.

**Precondition check:**

- `bootnode_mode = true`: This is a legitimate, documented CKB deployment mode for actual network bootnodes. It is not an "unsupported local configuration" — it is the intended production mode for CKB's bootstrap infrastructure.
- `max_inbound <= non_whitelist_inbound + 10`: A busy bootnode near capacity (within 10 slots of its limit) satisfies this naturally.
- Attacker entry: A standard unauthenticated TCP P2P connection to the bootnode's listen port.

**Concrete attack path:**

1. Attacker connects to a bootnode running in `bootnode_mode` that is near capacity (e.g., `max_inbound=125`, `non_whitelist_inbound=115`).
2. `SessionOpen` fires → `inbound_eviction()` is called **before** `accept_peer`, so the attacker's session is not yet in the registry.
3. Condition `125 <= 115 + 10` is true → collects all ~115 inbound peers → evicts ~57 of them (odd indices).
4. After eviction, `non_whitelist_inbound` drops to ~58. Condition is no longer met.
5. Attacker (and other peers) reconnect, bootnode fills back up to near capacity.
6. Attacker connects again → another ~50% mass eviction.

The `HashMap::values()` iteration order is non-deterministic, so the "odd index" selection is effectively random across the entire peer set — long-lived legitimate peers have no protection from this.

**Comparison to `try_evict_inbound_peer`:**

The normal eviction path in `peer_registry.rs` uses a careful multi-stage protection scheme (lowest ping, most recent messages, longest connection time, network group diversity) and evicts **exactly 1 peer**. [3](#0-2) 

The `inbound_eviction` bootnode path bypasses all of this and evicts ~50% of all peers with no protection criteria whatsoever.

**Why this is not "ordinary peer disconnect":**

A single attacker connection causes O(N/2) legitimate peer disconnections where N is the current inbound count. This is a churn amplification factor proportional to the bootnode's peer count, not a 1:1 disconnect.

### Title
Mass Inbound Peer Eviction via Single Connection in Bootnode Mode — (`network/src/network.rs`)

### Summary
The `inbound_eviction` function in `EventHandler`, triggered on every `SessionOpen` when `bootnode_mode` is enabled and the node is near inbound capacity, evicts approximately 50% of all current non-whitelist inbound peers per new connection. An unprivileged attacker making a single TCP connection to a near-capacity bootnode causes mass disconnection of legitimate peers with no protection criteria applied.

### Finding Description
In `network/src/network.rs`, `EventHandler::inbound_eviction` collects all non-whitelist inbound peers into a `Vec` and then applies `index & 0x1 != 0` to select every peer at an odd index for disconnection. Since `HashMap::values()` has non-deterministic order, this is a random ~50% cull of the entire inbound peer set. The function is called unconditionally on every `SessionOpen` event before `accept_peer`, meaning the triggering connection is not yet registered and is not subject to eviction itself.

The trigger condition `status.max_inbound <= status.non_whitelist_inbound.saturating_add(10)` is met whenever the bootnode is within 10 connections of its inbound limit — a normal operating state for a busy bootnode. [4](#0-3) 

### Impact Explanation
- Each attacker connection evicts ~N/2 legitimate peers (where N = current inbound count).
- Legitimate long-lived peers receive no protection (no ping-time, recency, or connection-duration criteria).
- The bootnode cannot maintain a stable peer set; peer discovery quality degrades proportionally to attacker connection rate.
- After each mass eviction the bootnode refills, enabling repeated triggering.

### Likelihood Explanation
CKB bootnodes are publicly advertised, always-on, and designed to accept many inbound connections. A bootnode near capacity is the normal steady state. The attack requires only the ability to open a TCP connection to the bootnode's P2P port — no authentication, no stake, no PoW.

### Recommendation
Replace the `index & 0x1 != 0` mass eviction with a single-peer eviction using the same protection criteria as `try_evict_inbound_peer` (ping RTT, recency, connection duration, network group). The bootnode path should evict at most 1 peer per new connection, not half the peer set. If periodic churn is desired for bootnode peer rotation, it should be implemented as a scheduled service with a bounded rate, not triggered per-connection.

### Proof of Concept
Simulate a `NetworkState` with `bootnode_mode = true`, `max_inbound = 20`, and 15 registered non-whitelist inbound peers (satisfying `20 <= 15 + 10`). Fire a `SessionOpen` event. Assert that `inbound_eviction()` returns ~7–8 session IDs for disconnection. Fire a second `SessionOpen` after refilling to 15 peers. Assert the same mass eviction recurs. Confirm that no peer's connection duration, ping RTT, or message recency is consulted during selection.

    fn try_evict_inbound_peer(&self, _peer_store: &PeerStore) -> Option<SessionId> {
        let mut candidate_peers = {
            self.peers
                .values()
                .filter(|peer| peer.is_inbound() && !peer.is_whitelist)
                .collect::<Vec<_>>()
        };
        // Protect peers based on characteristics that an attacker hard to simulate or manipulate
        // Protect peers which has the lowest ping
        sort_then_drop(
            &mut candidate_peers,
            EVICTION_PROTECT_PEERS,
            |peer1, peer2| {
                let peer1_ping = peer1
                    .ping_rtt
                    .map(|p| p.as_secs())
                    .unwrap_or_else(|| u64::MAX);
                let peer2_ping = peer2
                    .ping_rtt
                    .map(|p| p.as_secs())
                    .unwrap_or_else(|| u64::MAX);
                peer2_ping.cmp(&peer1_ping)
            },
        );

        // Protect peers which most recently sent messages
        sort_then_drop(
            &mut candidate_peers,
            EVICTION_PROTECT_PEERS,
            |peer1, peer2| {
                let now = Instant::now();
                let peer1_last_message = peer1
                    .last_ping_protocol_message_received_at
                    .map(|t| now.saturating_duration_since(t).as_secs())
                    .unwrap_or_else(|| u64::MAX);
                let peer2_last_message = peer2
                    .last_ping_protocol_message_received_at
                    .map(|t| now.saturating_duration_since(t).as_secs())
                    .unwrap_or_else(|| u64::MAX);
                peer2_last_message.cmp(&peer1_last_message)
            },
        );
        // Protect half peers which have the longest connection time
        let protect_peers = candidate_peers.len() >> 1;
        sort_then_drop(&mut candidate_peers, protect_peers, |peer1, peer2| {
            peer2.connected_time.cmp(&peer1.connected_time)
        });

        // Group peers by network group
        let evict_group = candidate_peers
            .into_iter()
            .fold(
                HashMap::new(),
                |mut groups: HashMap<Group, Vec<&Peer>>, peer| {
                    groups.entry(peer.network_group()).or_default().push(peer);
                    groups
                },
            )
            .values()
            .max_by_key(|group| group.len())
            .cloned()
            .unwrap_or_default();

        // randomly evict a peer
        let mut rng = thread_rng();
        evict_group.choose(&mut rng).map(|peer| {
            debug!("Disconnect inbound peer {:?}", peer.connected_addr);
            peer.session_id
        })
    }
