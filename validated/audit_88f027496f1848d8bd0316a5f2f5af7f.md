### Title
`total_tx_size` Accounting Omits Transactions in the Intermediate `verify_queue` State, Allowing Memory Limit Bypass — (`tx-pool/src/pool.rs`, `tx-pool/src/service.rs`)

---

### Summary

The `limit_size()` enforcement and the `total_tx_size` field reported via `tx_pool_info` RPC both read exclusively from `pool_map.total_tx_size`, which only tracks transactions that have completed verification and been admitted to the pool. Transactions currently sitting in the `verify_queue` — an intermediate state between submission and pool admission — are entirely excluded from this accounting. Because the two size budgets are independent and additive, an unprivileged submitter can cause total in-process memory to exceed the configured `max_tx_pool_size` by up to the hardcoded `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` (256 MB).

---

### Finding Description

CKB's tx-pool has a two-stage lifecycle for incoming transactions:

1. **`verify_queue`** — transactions are queued here immediately upon submission (via RPC `send_transaction` or P2P relay) and await async script verification by worker threads.
2. **`pool_map`** — after verification succeeds, transactions are moved here and tracked as `Pending`, `Gap`, or `Proposed`.

Each stage maintains its own independent size counter and its own size limit:

| Stage | Size counter | Limit |
|---|---|---|
| `verify_queue` | `verify_queue.total_tx_size` | `DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` = 256 MB (hardcoded) |
| `pool_map` | `pool_map.total_tx_size` | `config.max_tx_pool_size` (default 180 MB, configurable) |

The `limit_size()` function, which enforces the pool's memory budget, only inspects `pool_map.total_tx_size`:

```rust
// tx-pool/src/pool.rs
while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
    // evict from pool_map only
}
``` [1](#0-0) 

The `info()` function that populates the `tx_pool_info` RPC response similarly reads only from `pool_map`:

```rust
// tx-pool/src/service.rs
total_tx_size: tx_pool.pool_map.total_tx_size,
``` [2](#0-1) 

The `verify_queue_size` field in `TxPoolInfo` reports only the **count** of queued transactions, not their byte size:

```rust
verify_queue_size: verify_queue.len(),
``` [3](#0-2) 

The `verify_queue` enforces its own independent limit:

```rust
// tx-pool/src/component/verify_queue.rs
const DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE: usize = 256_000_000;

pub fn is_full(&self, add_tx_size: usize) -> bool {
    add_tx_size >= DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE - self.total_tx_size
}
``` [4](#0-3) [5](#0-4) 

Because the two limits are enforced independently and never summed, the maximum total in-process memory attributable to the tx-pool is `max_tx_pool_size + DEFAULT_MAX_VERIFY_QUEUE_TX_SIZE` = 180 MB + 256 MB = **436 MB**, not the 180 MB the operator configured.

This is the direct analog to the Casimir `getTotalStake()` bug: just as pending validators (step 2 of 3) were excluded from the total-stake accounting, transactions in the `verify_queue` (step 1 of 2) are excluded from the total-size accounting and from the eviction guard.

---

### Impact Explanation

An unprivileged submitter can fill the `verify_queue` to its 256 MB ceiling by submitting many transactions that declare a high cycle count (up to `max_tx_verify_cycles`), causing them to remain in the queue for an extended period while workers process them. Simultaneously, the `pool_map` can be at its 180 MB limit. The combined memory pressure (up to ~436 MB) is invisible to the `limit_size()` eviction loop and is under-reported by the `total_tx_size` RPC field. On memory-constrained nodes this can cause OOM conditions or severe performance degradation. Operators relying on `total_tx_size` to monitor pool health receive a systematically low reading that does not reflect actual memory consumption.

---

### Likelihood Explanation

The attack requires only the ability to submit transactions via the public `send_transaction` RPC or via P2P relay — both are available to any unprivileged peer. Transactions with high declared cycles are accepted into the verify_queue before verification completes, so no special script-execution knowledge is needed. The 256 MB verify_queue ceiling is hardcoded and cannot be reduced by node operators, making the gap between the configured limit and the actual limit permanent.

---

### Recommendation

The `limit_size()` eviction loop should incorporate the current `verify_queue.total_tx_size` when deciding whether the combined pool is over budget:

```rust
while self.pool_map.total_tx_size + verify_queue_size > self.config.max_tx_pool_size {
    // evict
}
```

Alternatively, the `verify_queue`'s hardcoded ceiling should be derived from `max_tx_pool_size` rather than being a fixed constant, so that the two limits are not independently additive. The `total_tx_size` field in `TxPoolInfo` should also include `verify_queue.total_tx_size` to give operators an accurate picture of actual memory usage.

---

### Proof of Concept

1. Configure a node with `max_tx_pool_size = 180_000_000` (default).
2. Submit enough transactions with high declared cycle counts (≤ `max_tx_verify_cycles`) via `send_transaction` RPC to fill the `verify_queue` toward 256 MB. Workers will be busy verifying them.
3. Simultaneously, submit enough fast-verifying transactions to fill `pool_map` to 180 MB.
4. Observe via `tx_pool_info` that `total_tx_size` reports ≤ 180 MB (only `pool_map`), while actual RSS of the `ckb` process reflects ~436 MB of tx-pool data.
5. `limit_size()` never fires against the verify_queue entries because it only checks `pool_map.total_tx_size > config.max_tx_pool_size`. [4](#0-3) [6](#0-5) [7](#0-6) [8](#0-7)

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
