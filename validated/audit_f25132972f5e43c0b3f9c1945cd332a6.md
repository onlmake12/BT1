### Title
`ScriptHashTypeVerifier` Checks Only Lock Script Hash Types, Silently Skips Type Script Hash Types — (`File: verification/src/transaction_verifier.rs`)

### Summary

`ScriptHashTypeVerifier::verify()` iterates over every transaction output and validates the `hash_type` of the **lock script** against `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the **type script** hash type. A transaction output carrying a type script with a future/unenabled hash type (e.g., `Data3` = 6) passes `NonContextualTransactionVerifier` and enters the tx-pool verify queue, only to be rejected later during expensive contextual script execution. This is the direct CKB analog of the Kakarot M-02 finding: a guard that should apply uniformly to both script slots is silently applied to only one.

### Finding Description

`ScriptHashTypeVerifier::verify()` in `verification/src/transaction_verifier.rs` (lines 796–814):

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else {
            return Err(...InvalidScriptHashType...);
        }
        // ← output.type_().to_opt() is never inspected here
    }
    Ok(())
}
``` [1](#0-0) 

`ENABLED_SCRIPT_HASH_TYPE` is the static set `{0, 1, 2, 4}` (Data, Type, Data1, Data2). Any value ≥ 6 (Data3 … Data127) is outside this set. [2](#0-1) 

The verifier is wired into `NonContextualTransactionVerifier` as the sole hash-type gate: [3](#0-2) 

`NonContextualBlockTxsVerifier` runs this same verifier for every transaction in a relayed block: [4](#0-3) 

The tx-pool calls `non_contextual_verify` first, then enqueues the transaction for expensive contextual verification: [5](#0-4) 

The actual rejection of a future hash type only happens inside `select_version` during script execution, deep inside `ContextualTransactionVerifier`: [6](#0-5) 

By contrast, the low-level structural check `check_data()` validates only that the bit pattern is well-formed (odd bit not set), not that the value is in the enabled set — so `Data3` (6) passes structural validation and reaches `ScriptHashTypeVerifier` unchallenged: [7](#0-6) 

**Root cause:** `ScriptHashTypeVerifier` calls `output.lock().hash_type()` but never calls `output.type_().to_opt().map(|t| t.hash_type())`. The check is applied to one script slot (lock) and silently skipped for the other (type), exactly mirroring the Kakarot pattern of applying a guard to contracts but not EOAs.

### Impact Explanation

1. **Resource exhaustion / DoS on tx-pool and block relay:** An unprivileged attacker crafts transactions whose outputs carry a type script with `hash_type = 6` (Data3). These pass `NonContextualTransactionVerifier` and are enqueued for full contextual verification. Each such transaction consumes a verification worker slot and CPU cycles before being rejected by `select_version`. Because the non-contextual gate is cheap and the contextual gate is expensive (script execution), the attacker can amplify resource consumption with minimal cost.

2. **Inconsistent validation semantics:** Lock scripts with future hash types are rejected immediately and cheaply (non-contextual). Type scripts with the same future hash types are accepted through the non-contextual layer and only rejected later. This asymmetry means nodes may propagate and partially process transactions that should be uniformly rejected at the earliest gate, creating divergent behavior between nodes that cache non-contextual results and those that do not.

3. **Block relay amplification:** A block containing transactions with invalid type script hash types passes `NonContextualBlockTxsVerifier` and forces every receiving node to perform contextual verification before the block is ultimately rejected.

### Likelihood Explanation

The attack requires no privileged access. Any RPC caller (`send_transaction`) or P2P relay peer can submit a transaction with an output whose type script has `hash_type = 6`. The construction is trivial: set `hash_type` to any even value ≥ 6 in the type script field of any output. The non-contextual check will pass, and the transaction will be enqueued. This is reachable from the default unprivileged attacker profile.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script hash type for each output, mirroring the existing lock script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        let lock_hash_type = TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
            .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: output.lock().hash_type() })?;
        let val: u8 = lock_hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
        }

        // ADD: type script check
        if let Some(type_script) = output.type_().to_opt() {
            let type_hash_type = TryInto::<ScriptHashType>::try_into(type_script.hash_type())
                .map_err(|_| TransactionError::InvalidScriptHashType { hash_type: type_script.hash_type() })?;
            let val: u8 = type_hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        }
    }
    Ok(())
}
```

Update the struct comment and `NonContextualTransactionVerifier` doc to reflect that both lock and type script hash types are checked.

### Proof of Concept

1. Construct a transaction with one output whose type script has `hash_type = 6` (Data3) and any lock script with a valid hash type (e.g., Data = 0).
2. Submit via `send_transaction` RPC or P2P relay.
3. Observe: `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` returns `Ok(())` — the transaction is enqueued.
4. Observe: `ContextualTransactionVerifier` → `ScriptVerifier` → `select_version` returns `Err(ScriptError::InvalidVmVersion(3))` — the transaction is rejected only at this late, expensive stage.
5. Repeat at high rate to exhaust the verification worker pool. [8](#0-7) [9](#0-8)

            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
