### Title
Unsafe Native ETH `.transfer()` in `verifyUpdate()` Refund Path Causes Permanent DoS for Smart Contract Callers - (File: lazer/contracts/evm/src/PythLazer.sol)

### Summary
`PythLazer.verifyUpdate()` uses the low-level `payable(msg.sender).transfer()` to refund excess ETH to callers who overpay the `verification_fee`. Because `.transfer()` forwards only 2300 gas, any smart contract caller whose `receive()` or `fallback()` function requires more than 2300 gas will cause the refund to revert, which reverts the entire `verifyUpdate()` call. This is the direct EVM analog of the reported StargateAdapter `safeTransfer` issue: a bare transfer primitive that silently fails for a class of valid callers.

### Finding Description
In `PythLazer.sol`, the `verifyUpdate()` function accepts a `msg.value` payment and refunds any excess over `verification_fee`:

```solidity
// PythLazer.sol lines 74-77
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
```

`address.transfer()` is hardcoded to forward exactly 2300 gas. Any smart contract caller (e.g., a DeFi protocol, a keeper bot contract, or any contract with a non-trivial `receive()`) that sends `msg.value > verification_fee` will trigger this path. If the caller's fallback/receive logic consumes more than 2300 gas (e.g., it emits an event, writes to storage, or calls another contract), the `.transfer()` reverts, which propagates and reverts the entire `verifyUpdate()` call.

The result is that smart contract integrators of Pyth Lazer **cannot use `verifyUpdate()`** unless they send exactly `verification_fee` — a value they must read from state before every call. Any contract that defensively overpays (a common and reasonable pattern) is permanently blocked.

### Impact Explanation
Smart contract callers that send `msg.value > verification_fee` to `verifyUpdate()` will always receive a revert. Since `verifyUpdate()` is the sole on-chain verification entry point for Pyth Lazer price data, this constitutes a functional DoS for the class of smart contract integrators that use standard overpayment patterns. No ETH is permanently lost (the revert returns funds), but the service is inaccessible to affected callers without a code change on their side.

### Likelihood Explanation
Smart contract integrators routinely send a small buffer above the required fee to guard against fee increases between the time they read the fee and the time their transaction is mined. Any such contract with a non-trivial `receive()` function (e.g., one that emits an event or updates state) will hit this revert on every call. The `verification_fee` is currently `1 wei`, making exact-payment easy today, but any future fee increase makes the overpayment pattern more common. The entry path is fully unprivileged — any external caller triggers it.

### Recommendation
Replace the bare `.transfer()` with a `.call{value: ...}("")` pattern, which forwards all available gas and checks the return value:

```solidity
if (msg.value > verification_fee) {
    (bool success, ) = payable(msg.sender).call{value: msg.value - verification_fee}("");
    require(success, "Refund failed");
}
```

This matches the pattern already used correctly elsewhere in the Pyth codebase (e.g., `Entropy.sol` lines 163–164, `Scheduler.sol` lines 660–661).

### Proof of Concept
1. Deploy a contract `Caller` with a `receive()` function that emits an event (costs >2300 gas).
2. From `Caller`, call `PythLazer.verifyUpdate{value: verification_fee + 1}(validUpdate)`.
3. The signature verification passes, but `payable(msg.sender).transfer(1)` is called with `msg.sender = address(Caller)`.
4. `Caller.receive()` exceeds 2300 gas → `.transfer()` reverts → entire `verifyUpdate()` reverts.
5. `Caller` can never successfully call `verifyUpdate()` with any `msg.value > verification_fee`.
