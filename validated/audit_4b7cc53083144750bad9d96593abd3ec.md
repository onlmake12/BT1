Audit Report

## Title
Unprivileged Child VM Can Steal Sibling's Exit Code via `ckb_wait` Without Parent Ownership Check — (`script/src/scheduler.rs`)

## Summary

`Wait::ecall` pushes any caller-supplied `target_id` into the shared message box with no parent-child relationship check. The `Message::Wait` handler in `process_message_box` delivers the exit code to any VM that requests it and then permanently removes the entry from `terminated_vms`, regardless of whether the requesting VM is the legitimate parent. A malicious child VM can therefore drain a sibling's exit-code entry, causing the true parent's subsequent `ckb_wait` to return `WAIT_FAILURE` and corrupting transaction-validation logic.

## Finding Description

**`Wait::ecall` — no ownership check:**
`target_id` is read from register A0 and pushed into the message box without any verification that the caller is the parent of the target VM. [1](#0-0) 

**`Message::Wait` handler — no parent check, entry consumed:**
The handler checks only whether `target_id` is in `terminated_vms` or `states`. There is no check that `vm_id` (the caller) is the parent of `args.target_id`. On a successful match, the entry is permanently removed from `terminated_vms` via `retain`. [2](#0-1) 

**No parent tracking in the scheduler struct:**
The `Scheduler` struct tracks `inherited_fd` for file-descriptor ownership but has no `parent` map recording spawn parentage. [3](#0-2) 

**Sequential, predictable PIDs:**
`boot_vm` assigns IDs by simple increment, making sibling PIDs trivially predictable. [4](#0-3) 

**Scheduling order amplifies the attack:**
The scheduler always runs the runnable VM with the largest ID first. A malicious child spawned after the root (higher ID) therefore runs before the root, giving it a reliable window to call `ckb_wait` on a sibling before the root does. [5](#0-4) 

**`joining_vms` fast-path does not mitigate this:**
When a VM terminates, `iterate_process_results` inserts the exit code into `terminated_vms` and delivers it to any VM already in `VmState::Wait`, but does **not** remove the entry from `terminated_vms` at that point. The entry remains in `terminated_vms` and is fully exposed to theft by any other running VM until `process_message_box` removes it at line 575. [6](#0-5) 

**Concrete attack flow:**
1. Root (ID=0) spawns child B (ID=1), then spawns malicious child A (ID=2).
2. B terminates; its exit code is inserted into `terminated_vms[1]`. B is removed from `states`.
3. A runs first (highest ID). A calls `ckb_wait(1)`. The handler finds `terminated_vms[1]`, delivers the exit code to A, marks A runnable, and removes `terminated_vms[1]`.
4. Root calls `ckb_wait(1)`. The handler finds neither `terminated_vms[1]` (stolen) nor `states[1]` (removed at termination). Root receives `WAIT_FAILURE`.

## Impact Explanation

This is an incorrect implementation of the `ckb_wait` syscall behavior, matching the allowed High impact: *"Incorrect implementation or behavior of CKB-VM or system scripts"* (10001–15000 points). A root script that spawns a user-supplied "plugin" child alongside a trusted child, then uses the trusted child's exit code to decide transaction validity, can be forced to receive `WAIT_FAILURE` instead of the real exit code. This allows a malicious plugin author to push the root script into an unintended code path — either accepting an invalid transaction or rejecting a valid one — without any privileged access.

## Likelihood Explanation

- Fully reachable by any unprivileged script author whose code is spawned as a child VM (e.g., sub-lock or plugin patterns).
- No leaked keys, majority hashpower, or victim mistakes required.
- PID guessing is trivial (sequential integers from `FIRST_VM_ID`); alternatively, the root can pass the sibling's PID via a pipe, making the attack deterministic.
- The scheduling policy (highest ID runs first) gives the malicious child a reliable execution window before the root.
- Locally testable and reproducible with a minimal two-child script.

## Recommendation

Track the parent VM ID at spawn time by adding a `parent: BTreeMap<VmId, VmId>` field to the `Scheduler` struct. In the `Message::Wait` handler inside `process_message_box`, before delivering the exit code, verify that `vm_id == parent[args.target_id]`; if not, set register A0 to `WAIT_FAILURE` and `continue`. This mirrors POSIX `waitpid` semantics where only the direct parent may reap a child. The `parent` map entry should be inserted in the `Message::Spawn` handler alongside the existing `inherited_fd` insertion. [7](#0-6) 

## Proof of Concept

Write a scheduler integration test using the `iterate` API:

1. Spawn child B (exits immediately with code 42).
2. Spawn malicious child A (calls `ckb_wait` on B's PID = `ROOT_VM_ID + 1`).
3. Run the scheduler until A completes; assert A received `SUCCESS` and exit code 42.
4. Have root call `ckb_wait` on B's PID.
5. Assert root receives `WAIT_FAILURE` (not `SUCCESS`/42).

This directly demonstrates that A drained B's `terminated_vms` entry, leaving root unable to reap its own child.
