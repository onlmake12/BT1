### Title
Unbounded HTTP Body Buffering in Miner Notify Endpoint Allows Local OOM Crash — (`miner/src/client.rs`)

### Summary

The miner's HTTP notify server unconditionally buffers the entire request body into memory with no size limit, allowing any process that can reach the TCP port to crash the miner via OOM by sending an arbitrarily large body.

### Finding Description

The `handle` function is the sole HTTP handler for the miner's block-template notify endpoint: [1](#0-0) 

Line 362 calls `BodyExt::collect(req).await?.aggregate()`, which streams and accumulates every byte of the incoming request body into a contiguous in-memory buffer before `serde_json::from_reader` is ever called. There is no `http_body_util::Limited` wrapper, no `Content-Length` check, and no per-connection byte cap anywhere in the miner codebase. [2](#0-1) 

The server is stood up as a plain `TcpListener` with no authentication and no middleware: [3](#0-2) 

A grep for `Limited`, `body_limit`, `max_body`, or `BodyLimit` in the entire `miner/` tree returns zero results, confirming no guard exists.

### Impact Explanation

An attacker who can reach the notify TCP port (local or network, depending on the configured `SocketAddr`) sends a single HTTP POST with a multi-gigabyte body. The Tokio runtime faithfully reads and buffers every byte into the heap. The miner process exhausts available memory and is killed by the OS OOM killer, halting all mining activity.

Scope: **miner process OOM crash (local RPC API crash)** — matches the stated scope exactly.

### Likelihood Explanation

- The notify endpoint is optional but explicitly documented and commonly used in production notify-mode deployments.
- If bound to `127.0.0.1`, any unprivileged local process can exploit it. If bound to `0.0.0.0`, any network peer can.
- No authentication, no rate limiting, no body cap — a single connection and a single request suffice.
- The exploit requires zero privileges and zero cryptographic material.

### Recommendation

Wrap the incoming body with `http_body_util::Limited` before collecting:

```rust
use http_body_util::Limited;

const MAX_BODY: u64 = 4 * 1024 * 1024; // 4 MiB is more than enough for a BlockTemplate

async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let limited = Limited::new(req, MAX_BODY);
    let body = BodyExt::collect(limited).await?.aggregate();
    // ...
}
```

This causes `collect` to return an error (not a panic) once the limit is exceeded, and the connection is cleanly dropped.

### Proof of Concept

```bash
# Start miner with notify mode enabled (listen = "127.0.0.1:18114")
# Then from any local shell:
python3 -c "
import socket, time
s = socket.create_connection(('127.0.0.1', 18114))
header = (
    b'POST / HTTP/1.1\r\n'
    b'Host: 127.0.0.1\r\n'
    b'Content-Type: application/json\r\n'
    b'Content-Length: 2147483648\r\n'   # 2 GiB
    b'\r\n'
)
s.sendall(header)
chunk = b'A' * (1024 * 1024)  # 1 MiB chunks
while True:
    s.sendall(chunk)
"
# Observe miner RSS grow unboundedly until OOM kill.
```
