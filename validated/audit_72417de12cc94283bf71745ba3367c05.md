Audit Report

## Title
Minimum Fee Rate Admission Gate Uses `tx_size` Instead of Weight, Allowing Cycle-Heavy Transactions Below `min_fee_rate` Into the Pool - (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum fee threshold using only the raw serialized byte size (`tx_size`), not the correct weight (`max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`). Because cycles are unknown at pre-check time and no second fee-rate check is performed post-verification, a transaction with a small serialized size but extremely high actual cycles passes the admission gate while its true fee rate is orders of magnitude below `min_fee_rate`. The code's own comment acknowledges this as a deliberate "cheap check," but no compensating post-verification check exists anywhere in the pipeline.

## Finding Description

`FeeRate` is defined as shannons per kilo-weight, and weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)` per `get_transaction_weight`:

```rust
// util/types/src/core/tx_pool.rs L298-303
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

`TxEntry::fee_rate()`, `EvictKey`, `AncestorsScoreSortKey`, and the fee-rate statistics RPC all use `get_transaction_weight` correctly. However, the admission gate does not:

```rust
// tx-pool/src/util.rs L42-45
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

The full `_process_tx` pipeline is:

1. `pre_check` → `check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)` — fee gate passes using only `tx_size`.
2. `verify_rtx(...)` — actual cycles are determined here.
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry is created with real cycles.
4. `submit_entry(...)` — entry is inserted into the pool.

There is **no second fee-rate check** after step 2 using `verified.cycles`. The `limit_size` function evicts by byte-size (`total_tx_size > max_tx_pool_size`), not by cycle budget, so cycle-heavy, byte-small transactions do not trigger eviction and can accumulate freely. The `EvictKey`-based eviction only fires when the byte-size limit is reached, at which point the attacker's transactions (with correct weight-based fee rate near zero) would be evicted last only if higher-fee-rate transactions are also present — but the attacker can keep resubmitting.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker submitting transactions with small serialized size (e.g., 200 bytes) and near-`MAX_BLOCK_CYCLES` actual cycles (e.g., 3,500,000,000) pays only `min_fee_rate × 200 / 1000 = 200 shannons` to pass the gate, while the correct threshold is `min_fee_rate × 596,999 / 1000 ≈ 597,000 shannons` — roughly 3,000× less. Each such transaction:

- Forces the node to execute up to `max_block_cycles` worth of script computation during `verify_rtx`, degrading the verification pipeline.
- Occupies pool entries with near-zero weight-based fee rate, consuming block cycle budget and crowding out legitimate high-fee transactions from block templates.
- Persists in the pool until expiry (hours), since byte-size-based `limit_size` eviction is not triggered by small-byte transactions.

Repeated submissions at negligible cost can saturate the node's verification queue and degrade block assembly throughput across the network.

## Likelihood Explanation

The attack path is the standard `send_transaction` RPC, reachable by any unprivileged local or remote caller. No special privilege, key, or majority hashpower is required. The attacker only needs to know `min_fee_rate` (publicly visible via `tx_pool_info` RPC) and craft a transaction whose lock/type script performs heavy computation but serializes to a small byte footprint — straightforward with any CKB script using loops or recursive operations. The attack is repeatable and cheap: each submission costs only `min_fee_rate × tx_size / 1000` shannons.

## Recommendation

After `verify_rtx` returns `verified.cycles`, perform a second fee-rate check using the actual weight before calling `submit_entry`:

```rust
// In _process_tx, after verify_rtx succeeds:
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

The pre-check in `check_tx_fee` can remain as a cheap lower-bound guard (rejecting obviously underpaying transactions before expensive verification), but it must be followed by a definitive post-verification check using the correct weight. The `check_tx_fee` signature should remain unchanged for the pre-check; a new `check_tx_fee_post_verify` function taking `cycles: Cycle` should be added and called in `_process_tx` after `verify_rtx`.

## Proof of Concept

1. Configure node with `min_fee_rate = 1000` shannons/KW.
2. Craft a CKB transaction: serialized size ≈ 200 bytes, lock script performs ~3,500,000,000 cycles of computation.
3. Set fee = 201 shannons (just above `min_fee_rate.fee(200) = 200`).
4. Submit via `send_transaction` RPC.
5. `check_tx_fee` computes `min_fee = 1000 × 200 / 1000 = 200`; fee 201 > 200 → **accepted into pre-check**.
6. `verify_rtx` executes the script, consuming ~3.5B cycles; `verified.cycles ≈ 3,500,000,000`.
7. No post-verification fee-rate check exists; `TxEntry::new(rtx, 3_500_000_000, 201_shannons, 200)` is created and inserted.
8. `entry.fee_rate()` = `FeeRate::calculate(201, get_transaction_weight(200, 3_500_000_000))` ≈ `201 × 1000 / 596_999` ≈ **0.34 shannons/KW** — ~3,000× below `min_fee_rate`.
9. `limit_size` is not triggered (total byte size barely increases); the entry persists.
10. Repeat to fill the pool with cycle-saturating, near-zero-fee-rate transactions, degrading block assembly and verification throughput.
