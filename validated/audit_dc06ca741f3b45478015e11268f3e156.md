### Title
Inflated `total_tx_size`/`total_tx_cycles` Accounting in `PoolMap::add_entry` When Ancestor-Eviction Occurs — (File: `tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the new `total_tx_size` and `total_tx_cycles` are computed from the **pre-eviction** pool state. When `check_and_record_ancestors` subsequently evicts transactions (via `remove_entry_and_descendants`), those evictions correctly decrement `self.total_tx_size` and `self.total_tx_cycles`. However, the code then **overwrites** these correctly-decremented values with the stale pre-eviction snapshot, permanently inflating the pool's size/cycle accounting by the sum of all evicted transactions' sizes and cycles.

---

### Finding Description

In `PoolMap::add_entry` (`tx-pool/src/component/pool_map.rs`, lines 200–221):

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    ...
    // Step 1: snapshot pre-eviction totals + new entry
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // line 210-211
    ...
    // Step 2: may evict pool entries, decrementing self.total_tx_size/cycles
    evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213
    ...
    // Step 3: OVERWRITES the post-eviction self.total_tx_size with stale snapshot
    self.total_tx_size = total_tx_size;                             // line 218
    self.total_tx_cycles = total_tx_cycles;                         // line 219
    Ok((true, evicts))
}
```

**Step 1** captures `total_tx_size = self.total_tx_size + entry.size` before any evictions.

**Step 2** — `check_and_record_ancestors` — triggers the eviction path when the new transaction's ancestor count exceeds `max_ancestors_count` but can be reduced to within the limit by removing "cell_ref_parents" (pool transactions that reference the same cell output as a dep that the new transaction's input consumes). It calls `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` and `self.total_tx_cycles` for each evicted entry. [1](#0-0) 

**Step 3** then unconditionally overwrites `self.total_tx_size` with the stale pre-eviction snapshot, erasing the decrements from Step 2. [2](#0-1) 

The eviction path in `check_and_record_ancestors` is confirmed at lines 615–625: [3](#0-2) 

`remove_entry` at line 247 is where `update_stat_for_remove_tx` is called during eviction: [4](#0-3) 

The net accounting error per trigger is: `inflation += sum(evicted_tx.size)` and `inflation += sum(evicted_tx.cycles)`.

---

### Impact Explanation

`total_tx_size` is the authoritative pool-size counter used in two critical enforcement paths:

1. **`limit_size`** (`tx-pool/src/pool.rs`, line 298): evicts transactions while `total_tx_size > max_tx_pool_size`. Inflated accounting causes legitimate transactions to be evicted unnecessarily. [5](#0-4) 

2. **`updated_stat_for_add_tx`** (`pool_map.rs`, lines 711–729): rejects new submissions with `Reject::Full` when `total_tx_size + new_size` overflows or exceeds limits. Inflated accounting causes legitimate transactions to be rejected even when the pool has physical capacity. [6](#0-5) 

3. **`TxPoolInfo` RPC** (`tx-pool/src/service.rs`, line 1089): exposes `total_tx_size` to callers, giving incorrect pool state. [7](#0-6) 

An attacker who repeatedly triggers the eviction path accumulates inflation monotonically. Once `total_tx_size` exceeds `max_tx_pool_size`, the pool rejects all new transactions — a complete tx-pool DoS — without the attacker needing to fill the pool with real data.

---

### Likelihood Explanation

The eviction path requires:
- A pool transaction **P** with unspent output **O**
- A pool transaction **D** that uses **O** as a `cell_dep` (not consuming it)
- A new transaction **N** that spends **O** as an input (making **D** a `cell_ref_parent` of **N**), while also having ≥ `max_ancestors_count` (default: 25) ancestors through the normal input chain

This is a valid and deliberate CKB transaction pattern. An unprivileged tx-pool submitter can craft this sequence with no special privileges:

1. Submit a chain T₁ → T₂ → … → T₂₅ (each spending the previous output)
2. Submit **D** using T₂₅'s output as a `cell_dep`
3. Submit **N** spending T₂₅'s output (N's ancestors = T₁…T₂₅ = 25, plus D as cell_ref_parent = 26 > 25; after evicting D: 25 ≤ 25 → eviction triggered)

Each iteration inflates `total_tx_size` by D's serialized size. The attacker pays fees for ~27 transactions per iteration but achieves permanent, cumulative pool accounting corruption.

---

### Recommendation

Move the stat computation to **after** `check_and_record_ancestors` completes, so it uses the post-eviction `self.total_tx_size` as the base:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Pre-check for overflow only (do not capture the value yet)
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Compute AFTER evictions have already decremented self.total_tx_size
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, replace lines 218–219 with incremental updates on the post-eviction state:
```rust
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

### Proof of Concept

**Setup:** Node with default `max_ancestors_count = 25`.

**Step 1:** Submit a 25-transaction chain spending a genesis UTXO:
```
T₁ (spends genesis) → T₂ (spends T₁.out[0]) → … → T₂₅ (spends T₂₄.out[0])
```
Pool state: 25 txs, `total_tx_size = S₂₅`.

**Step 2:** Submit transaction **D** with `cell_dep = T₂₅.out[0]` (no inputs from pool).
Pool state: 26 txs, `total_tx_size = S₂₅ + size(D)`.

**Step 3:** Submit transaction **N** with `input = T₂₅.out[0]` (spending it).
- `get_tx_ancenstors(N)`: ancestors = {T₁…T₂₅} (25), cell_ref_parents = {D} (1), total = 26 > 25
- `26 - 1 = 25 ≤ 25` → eviction path triggered
- D is evicted: `self.total_tx_size -= size(D)` → `self.total_tx_size = S₂₅`
- Lines 218–219 overwrite: `self.total_tx_size = (S₂₅ + size(D)) + size(N)`
- **Correct value** should be: `S₂₅ + size(N)`
- **Inflation** = `size(D)`

**Step 4:** Repeat Steps 1–3 with fresh UTXOs. Each iteration adds `size(D)` to the inflation.

**Step 5:** After enough iterations, `total_tx_size > max_tx_pool_size` even though the actual pool is nearly empty. All subsequent `send_transaction` RPC calls return `Reject::Full`, achieving a complete tx-pool DoS against all users of the node.

    fn updated_stat_for_add_tx(
        &self,
        tx_size: usize,
        cycles: Cycle,
    ) -> Result<(usize, Cycle), Reject> {
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_size {} overflows by add {}",
                self.total_tx_size, tx_size
            ))
        })?;
        let total_tx_cycles = self.total_tx_cycles.checked_add(cycles).ok_or_else(|| {
            Reject::Full(format!(
                "tx-pool total_tx_cycles {} overflows by add {}",
                self.total_tx_cycles, cycles
            ))
        })?;
        Ok((total_tx_size, total_tx_cycles))
    }

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
