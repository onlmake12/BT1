### Title
Hardcoded `chain_id` in CosmWasm Pyth Contract Diverges from Wormhole's Chain ID, Enabling Cross-Chain Governance Replay — (File: `target_chains/cosmwasm/contracts/pyth/src/state.rs`)

---

### Summary

The CosmWasm Pyth contract stores `chain_id` as a local state variable in `ConfigInfo` that must agree with the Wormhole contract's chain ID. The code itself acknowledges this with an explicit warning comment. Because `chain_id` is set once at instantiation with no post-deployment update mechanism and no on-chain cross-verification against the Wormhole contract, a misconfigured or stale value allows governance VAAs targeting a different Wormhole chain to be accepted and executed on this contract.

---

### Finding Description

In `target_chains/cosmwasm/contracts/pyth/src/state.rs`, the `ConfigInfo` struct carries:

```rust
// Warning: This id needs to agree with the wormhole chain id.
// We should read this directly from wormhole, but their contract doesn't expose it.
pub chain_id: u16,
``` [1](#0-0) 

This field is written exactly once — at instantiation — and is never touched by any governance action:

```rust
let state = ConfigInfo {
    ...
    chain_id: msg.chain_id,
    ...
};
``` [2](#0-1) 

It is then used as the sole chain-identity guard inside `execute_governance_instruction`:

```rust
if instruction.target_chain_id != state.chain_id && instruction.target_chain_id != 0 {
    Err(PythContractError::InvalidGovernancePayload)?;
}
``` [3](#0-2) 

The same guard is repeated inside `transfer_governance`:

```rust
if claim_vaa_instruction.target_chain_id != current_config.chain_id
    && claim_vaa_instruction.target_chain_id != 0
{
    Err(PythContractError::InvalidGovernancePayload)?
}
``` [4](#0-3) 

Because the Wormhole CosmWasm contract does not expose a `chain_id` getter (as the comment admits), there is no way to verify at runtime that `state.chain_id` matches the actual Wormhole chain ID. The two values are permanently decoupled after deployment.

---

### Impact Explanation

If `chain_id` is set to the Wormhole chain ID of a **different** chain (e.g., chain B instead of chain A):

1. Any legitimate governance VAA that Pyth governance emits with `target_chain_id = B` (intended for chain B) will pass the chain-identity check on chain A's contract.
2. Governance VAAs are public Wormhole messages — any observer can replay them to any contract.
3. An attacker replaying a chain-B governance VAA on chain A's contract can execute `SetDataSources`, `SetFee`, `UpgradeContract`, or `AuthorizeGovernanceDataSourceTransfer` on chain A — actions that were never authorized for chain A.
4. The `governance_sequence_number` replay guard does **not** prevent this: the sequence counter on chain A is independent of chain B's counter, so a fresh chain-B VAA will always have a higher sequence number than chain A's stored value.

The worst-case outcome is an unauthorized `UpgradeContract` pointing to attacker-controlled code, or a `SetDataSources` that redirects price feeds to a malicious emitter.

---

### Likelihood Explanation

**Low-to-medium.** The precondition is that the deployer sets `chain_id` to the wrong value. Given that:

- The Wormhole contract provides no getter to verify the value programmatically,
- The comment in the source code explicitly warns this is error-prone,
- CosmWasm deployments across many chains (Injective, Osmosis, Neutron, Terra, etc.) each require a separate instantiation with a manually supplied `chain_id`,

a misconfiguration during any one of these deployments is a realistic operational risk. Once misconfigured, the error is silent and undetectable without off-chain monitoring.

---

### Recommendation

1. Add a `chain_id()` query to the Wormhole CosmWasm contract and have the Pyth contract call it at instantiation (and optionally on every governance execution) to assert agreement.
2. Until the Wormhole contract exposes this, add a governance action `SetChainId` so the value can be corrected post-deployment without a full migration.
3. Emit the `chain_id` in an instantiation event so off-chain monitors can alert on mismatches.

---

### Proof of Concept

1. Deploy the Pyth CosmWasm contract on chain A (Wormhole chain ID = 19) but supply `chain_id = 18` (Injective's Wormhole chain ID) in the `InstantiateMsg`.
2. Pyth governance later emits a legitimate VAA with `target_chain_id = 18` to, e.g., change Injective's data sources. This VAA is publicly observable on the Wormhole guardian network.
3. An attacker submits this VAA to the chain-A contract via `ExecuteMsg::ExecuteGovernanceInstruction`.
4. The check at line 270 evaluates: `18 != 18` → **false** → the guard does not fire.
5. The sequence-number check passes because chain A's `governance_sequence_number` is independent of chain B's.
6. The `SetDataSources` (or any other) action executes on chain A, replacing its authorized price-feed emitters with those intended for Injective — or, in the `UpgradeContract` case, migrating the contract to an arbitrary code ID.
