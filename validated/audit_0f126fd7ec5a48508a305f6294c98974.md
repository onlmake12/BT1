The premise of the question is **factually correct**. Reading the actual code confirms the bug:

In `put()` at line 62, `total_keys_num` is a **local variable** computed from `self.total_keys_num.checked_add(1)`. The result is used only to decide whether to call `shrink()`, but **`self.total_keys_num` is never written back** in the non-shrink branch. [1](#0-0) 

`self.total_keys_num` is only updated inside `shrink()`: [2](#0-1) 

But `shrink()` is only called when `total_keys_num > self.count_limit`. Since `total_keys_num` is always `self.total_keys_num + 1` (a local, never persisted), and `self.total_keys_num` starts at the startup estimate (0 for a fresh node), the condition `0 + 1 > count_limit` is false for any `count_limit > 1`. So `shrink()` never fires, and the DB grows without bound.

### Title
`RecentReject::put` never writes back `total_keys_num`, allowing unbounded RocksDB growth — (`tx-pool/src/component/recent_reject.rs`)

### Summary
`RecentReject::put` computes a local `total_keys_num = self.total_keys_num.checked_add(1)` but never assigns it back to `self.total_keys_num`. The in-memory counter is therefore permanently stuck at its startup-estimated value, so the `count_limit` guard never triggers, and the `recent_reject` RocksDB column families grow without bound.

### Finding Description
In `RecentReject::put`:

```rust
// line 62-69
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // BUG: self.total_keys_num is never updated here
} else {
    self.shrink()?;
}
```

`self.total_keys_num` is initialized at startup from `estimate_num_keys_cf` (an approximate RocksDB stat). [3](#0-2) 

For a fresh node this is `0`. Every subsequent call to `put()` evaluates `0 + 1 = 1`, which is never `> count_limit` (default configured as `keep_rejected_tx_hashes_count`), so `shrink()` is never called and `self.total_keys_num` stays at `0` forever. [4](#0-3) 

The only place `self.total_keys_num` is ever written is inside `shrink()`: [5](#0-4) 

### Impact Explanation
An unprivileged remote peer can submit an unlimited number of unique transactions that are rejected for any recordable reason (`LowFeeRate`, `Full`, `Verification`, etc.). Each rejection calls `put_recent_reject` → `RecentReject::put`, which writes a new key to RocksDB but never enforces the `count_limit`. The `recent_reject` database grows without bound, consuming unbounded disk space and I/O, eventually exhausting node storage and causing a crash or severe degradation — matching the "CKB network congestion with few costs" scope.

The call path is fully reachable via P2P transaction relay: [6](#0-5) [7](#0-6) 

### Likelihood Explanation
The attack requires only the ability to submit transactions over P2P — no keys, no hashpower, no privileged access. Generating unique transactions that fail fee-rate checks is trivial. The bug is deterministic and reproducible on a fresh node with default configuration.

### Recommendation
Add `self.total_keys_num = total_keys_num;` in the non-shrink branch of `put()`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num; // FIX: persist the increment
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

### Proof of Concept
The existing test at `tx-pool/src/component/tests/recent_reject.rs` already demonstrates the invariant `recent_reject.total_keys_num < 100` after 160 inserts with `limit=100`, but it only passes because `shrink()` happens to fire when the startup estimate is non-zero after the second batch. A fresh-node scenario (startup estimate = 0, `count_limit = 10`, insert 10 000 entries) would show `total_keys_num` stuck at `0` while the DB contains 10 000 keys. [8](#0-7)

    pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
        let hash_slice = hash.as_slice();

        let reject: ckb_jsonrpc_types::PoolTransactionReject = reject.into();

                self.shrink()?;
            }
        } else {
            // overflow occurred, try shrink
            self.shrink()?;
        }
        Ok(())
    }

    fn shrink(&mut self) -> Result<u64, AnyError> {
        let mut rng = thread_rng();
        let shard = rng.sample(Uniform::new(0, self.shard_num)).to_string();
        self.db.drop_cf(&shard)?;
        self.db.create_cf_with_ttl(&shard, self.ttl)?;

        let total_keys_num = self.estimate_total_keys_num()?;
        self.total_keys_num = total_keys_num;
        Ok(total_keys_num)
