### Title
No Cancellation/Refund Mechanism for Unfulfilled Echo Price Update Requests - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
The `Echo` contract has no function allowing users to cancel or reclaim funds from unfulfilled price update requests. If a provider never calls `executeCallback`, the user's fee (`req.fee`) is permanently locked in the contract with no recovery path.

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, they pay a fee split into two parts:

1. `_state.pythFeeInWei` — immediately credited to `_state.accruedFeesInWei` (Pyth protocol treasury) at request time.
2. `req.fee = msg.value - _state.pythFeeInWei` — stored in the `Request` struct, only credited to a provider upon `executeCallback`. [1](#0-0) 

The `req.fee` portion is only released when `executeCallback` is successfully called: [2](#0-1) 

There is no `cancelRequest`, `refundRequest`, or any other function in `IEcho` or `Echo` that allows a user to reclaim their locked `req.fee` if the request is never fulfilled. [3](#0-2) 

The contract's own developer comments acknowledge this risk explicitly: [4](#0-3) 

The `EchoState.Request` struct stores the fee per-request, and `clearRequest` is only called inside `executeCallback` — there is no other code path that frees this storage or returns funds. [5](#0-4) 

### Impact Explanation

If a provider goes offline, deregisters, or simply ignores a request, the user's `req.fee` (the provider fee portion of `msg.value`) is permanently locked in the `Echo` contract. The user has no on-chain mechanism to recover it. The Pyth protocol fee (`_state.pythFeeInWei`) is also non-refundable since it is credited immediately at request time. This results in a permanent, irrecoverable loss of user funds proportional to the provider fee for each unfulfilled request.

### Likelihood Explanation

Medium. The `exclusivityPeriodSeconds` window means only the assigned provider can fulfill during that period. After the window, any caller may fulfill, but there is no economic incentive enforced on-chain to guarantee fulfillment. A provider that goes offline, loses their keeper, or faces a bug in their off-chain infrastructure will leave requests permanently unfulfilled. The `firstUnfulfilledSeq` tracker only helps enumerate stuck requests — it does not resolve them. [6](#0-5) 

### Recommendation

Add a `cancelRequest(uint64 sequenceNumber)` function that:
- Is callable only by `req.requester`
- Enforces a minimum waiting period (e.g., after the exclusivity period has elapsed plus a grace period) to prevent griefing providers
- Refunds `req.fee` to the requester
- Calls `clearRequest(sequenceNumber)`

The Pyth protocol fee (`_state.pythFeeInWei`) may be treated as non-refundable (service fee for the attempt), but the provider fee (`req.fee`) should be returned to the user when no provider fulfills the request.

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback{value: totalFee}(provider, publishTime, priceIds, gasLimit)`.
2. `req.fee = totalFee - pythFeeInWei` is stored in the `Request` struct; `pythFeeInWei` is credited to `accruedFeesInWei`.
3. The assigned provider's keeper goes offline permanently.
4. The exclusivity period elapses; no other party has economic incentive to call `executeCallback`.
5. User attempts to recover funds — no `cancelRequest` function exists in `IEcho` or `Echo`.
6. `req.fee` remains locked in the contract indefinitely with no recovery path. [7](#0-6)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
