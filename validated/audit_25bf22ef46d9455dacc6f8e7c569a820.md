### Title
Entropy Provider Can Arbitrarily Spike `feeInWei` Without Restriction, Causing In-Flight User Transactions to Revert — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s `setProviderFee()` and `setProviderFeeAsFeeManager()` allow a registered provider (or its designated fee manager) to change `feeInWei` to any arbitrary value at any time, with no magnitude cap, no timelock, and no minimum notice period. This is the direct Pyth analog of the rage-quit finding: just as a party host could silently reduce the rage-quit timestamp to deny members their exit right, an Entropy provider can silently spike the fee to deny users their ability to request randomness at the price they observed, causing their transactions to revert with `InsufficientFee` and permanently blocking integrating contracts that have a fixed ETH budget.

---

### Finding Description

`requestHelper` enforces that `msg.value >= getFeeV2(provider, callbackGasLimit)` at the moment the request transaction executes:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [1](#0-0) 

`getFeeV2` reads `provider.feeInWei` live from storage at execution time:

```solidity
function getFeeV2(address provider, uint32 gasLimit) public view override returns (uint128 feeAmount) {
    return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
}
``` [2](#0-1) 

`setProviderFee` imposes **zero** restrictions on the new value — no cap, no rate limit, no timelock:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
``` [3](#0-2) 

The same unrestricted write is available through `setProviderFeeAsFeeManager`, callable by the provider-designated fee manager:

```solidity
function setProviderFeeAsFeeManager(address provider, uint128 newFeeInWei) external override {
    ...
    providerInfo.feeInWei = newFeeInWei;
``` [4](#0-3) 

The interface exposes both entry points publicly: [5](#0-4)

### Impact Explanation

**Scenario A — front-run / sandwich attack on individual users:**
1. User calls `getFeeV2()` off-chain, observes fee = F.
2. User submits `requestV2{value: F}(...)`.
3. Provider (or fee manager) front-runs with `setProviderFee(type(uint128).max)`.
4. User's transaction executes after the fee update → `InsufficientFee` revert → user loses gas, receives no randomness.

**Scenario B — breaking integrating DeFi contracts:**
Many integrating contracts (e.g., lottery, NFT mint, gaming) call `getFeeV2()` and forward the result in the same external call. If the provider raises the fee between the contract's deployment/configuration and the user's invocation, every user call to that contract reverts. The contract is effectively bricked for randomness until it is redeployed or upgraded.

**Scenario C — fee set to zero, then spiked:**
A provider can set `feeInWei = 0` to attract integrators, then spike to an extreme value once integrators have deployed immutable contracts that forward exactly `getFeeV2()` wei. Those contracts permanently fail.

The impact is **denial of service** against Entropy users and integrating protocols, with no fund-recovery path for users whose transactions revert.

---

### Likelihood Explanation

The entry path requires only that the provider (a permissionless, self-registering role) or its fee manager call a single unrestricted setter. No governance approval, no leaked key, and no external oracle is needed. The Fortuna keeper already calls `setProviderFeeAsFeeManager` automatically in response to gas-price changes: [6](#0-5) 

This confirms the setter is exercised in production. A malicious provider needs only to call it with an extreme value. Front-running is straightforward on any EVM chain with a public mempool.

---

### Recommendation

1. **Timelock fee increases**: require that a fee increase be announced N blocks/seconds in advance before it takes effect. Decreases can remain immediate (they benefit users).
2. **Cap the per-update increase**: e.g., the new fee may not exceed `oldFee * MAX_MULTIPLIER` in a single call.
3. **Rate-limit updates**: allow at most one fee change per epoch (e.g., per 24 hours).

        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();

        uint128 oldFeeInWei = providerInfo.feeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
