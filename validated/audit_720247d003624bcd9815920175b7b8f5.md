The code path is concrete and traceable. Let me map it precisely.

**Inbound session initialization** (`state.rs` line 80-81): an inbound session sets `remote_addr = RemoteAddress::Init(context.session.address.clone())` — the ephemeral connection address. [1](#0-0) 

**`received()` GetNodes branch** (`mod.rs` lines 124–139): when `listen_port` is `None`, `update_port()` is skipped entirely. Then, unconditionally, if `version >= REUSE_PORT_VERSION`, `change_to_listen()` is called. [2](#0-1) 

**`change_to_listen()`** (`state.rs` lines 147–151): if `remote_addr` is still `Init(addr)` (which it is, since `update_port` was skipped), it is promoted to `Listen(addr)` — with the original ephemeral port intact. [3](#0-2) 

**`notify()` announce path** (`mod.rs` lines 231–237): `check_timer()` returns `Some(addr)` only for `RemoteAddress::Listen` variants. After the promotion, the ephemeral address passes `is_valid_addr()` (which only checks IP reachability, not port validity) and is added to `announce_list`. [4](#0-3) 

**Broadcast to 3 peers** (`mod.rs` lines 243–258): the ephemeral address is pushed into `announce_multiaddrs` of up to 3 random sessions per tick. [5](#0-4) 

**Partial mitigation check**: `node_flags()` returns `None` if `identify_info` is absent, which would block the announce. However, the identify protocol is a standard handshake that any connecting peer completes, so this is not a meaningful barrier for an attacker. [6](#0-5) 

The root cause is that `change_to_listen()` is called unconditionally on the `version >= REUSE_PORT_VERSION` branch, regardless of whether `listen_port` was provided. The design intent of REUSE_PORT is that on Linux, the outbound connection port equals the listen port — but this only applies to **outbound** sessions. For **inbound** sessions, the peer's connection port is their ephemeral outbound port, not their listen port. Outbound sessions are already initialized as `RemoteAddress::Listen` (line 79), so `change_to_listen()` is a no-op for them. The bug only manifests for inbound sessions. [7](#0-6)

### Title
Inbound Peer Can Promote Ephemeral Port to Listen Address via GetNodes version≥1 with listen_port=None — (`network/src/protocols/discovery/mod.rs`)

### Summary
An unprivileged inbound peer that sends `GetNodes { version: 1, listen_port: None }` causes the victim node to promote the peer's ephemeral TCP connection port to a `RemoteAddress::Listen` entry, which is then announced to up to 3 other peers per notify tick, polluting the network-wide peer store with unreachable addresses.

### Finding Description
In `DiscoveryProtocol::received()`, the `GetNodes` handler has two independent branches:

1. `if let Some(port) = listen_port` — calls `update_port(port)`, which replaces the ephemeral port with the declared listen port and transitions `Init → Listen`.
2. `if version >= REUSE_PORT_VERSION` — calls `change_to_listen()`, which transitions `Init → Listen` **without modifying the port**.

When an inbound peer sends `listen_port = None` with `version = 1`, branch 1 is skipped, leaving `remote_addr` as `Init(ephemeral_addr)`. Branch 2 then fires and calls `change_to_listen()`, promoting `Init(ephemeral_addr)` to `Listen(ephemeral_addr)`. The ephemeral port is now treated as a verified listen address.

In `notify()`, `check_timer()` returns this address (since it is now `Listen`), `is_valid_addr()` passes for any public IP, and the address is queued for announcement to up to 3 peers. Those peers call `add_new_addrs()` on receipt, inserting the unreachable ephemeral address into their peer stores.

### Impact Explanation
- The victim node announces the attacker's ephemeral port as a listen address to up to 3 peers per `ANNOUNCE_INTERVAL` (24 hours).
- Receiving peers add the address to their peer stores via `add_new_addrs()` → `peer_store.add_addr()`.
- Connections attempted to this address will fail (ephemeral port is closed), degrading peer discovery quality across the network.
- A single attacker with one public IP can continuously reconnect with fresh ephemeral ports, injecting a stream of bad addresses.

### Likelihood Explanation
The attack requires only: (1) establishing an inbound TCP connection to a CKB node, and (2) sending a single well-formed `GetNodes` message with `version=1` and `listen_port=None`. No authentication, no PoW, no privileged access. The attacker must also complete the identify protocol handshake to pass the `node_flags()` check in `notify()`, which is standard P2P behavior.

### Recommendation
The `change_to_listen()` call should be moved inside the `if let Some(port) = listen_port` block, or guarded with a check that `listen_port` was provided. The REUSE_PORT logic only makes sense when the peer has explicitly declared its listen port:

```rust
if let Some(port) = listen_port {
    state.remote_addr.update_port(port);
    // ...add to addr_mgr...
    // change_to_listen is already handled by update_port
} else if version >= state::REUSE_PORT_VERSION {
    // Only promote if listen_port was provided; remove this branch or guard it
}
```

Alternatively, `change_to_listen()` should only be called when `remote_addr` has already been updated by `update_port()` (i.e., is already `Listen`), making it a no-op in the `None` case.

### Proof of Concept
1. Connect inbound to a CKB node from a public IP (e.g., `1.2.3.4:54321`).
2. Complete the identify protocol handshake so `node_flags()` returns `Some(flags)`.
3. Send `GetNodes { version: 1, listen_port: None, count: 100, required_flags: 0 }`.
4. Wait for the next `notify()` tick (up to 60 seconds by default).
5. Observe that `announce_list` contains `1.2.3.4:54321` (the ephemeral port).
6. Observe that up to 3 connected peers receive a `Nodes { announce: true }` message containing `1.2.3.4:54321`.
7. Differential: repeat with `version: 0` — `change_to_listen()` is not called, `remote_addr` stays `Init`, `check_timer()` returns `None`, no announcement occurs.
