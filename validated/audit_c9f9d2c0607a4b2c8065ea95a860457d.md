### Title
Missing On-Chain Invariant Between Stored Provider Fee and Actual Pyth Oracle Update Fee Causes Permanent Fund Lock — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary
The Echo contract splits fees at request time using a fixed `_state.pythFeeInWei` value, but at callback execution time the actual Pyth oracle update fee (`pyth.getUpdateFee(updateData)`) is deducted from the stored provider fee (`req.fee`). There is no on-chain invariant ensuring `req.fee >= pyth.getUpdateFee(updateData)`. If this invariant is violated — by a provider setting fees too low or by a Pyth oracle fee increase — `executeCallback` always reverts, and user ETH is indefinitely locked with no refund path.

---

### Finding Description

**At request time** (`requestPriceUpdatesWithCallback`):

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
```

The Echo protocol fee (`_state.pythFeeInWei`) is immediately accrued. The remainder — the provider's portion — is stored in `req.fee`. [1](#0-0) 

**At callback time** (`executeCallback`):

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);
clearRequest(sequenceNumber);
```

The actual Pyth oracle update fee is deducted from `req.fee + msg.value`. If `req.fee + msg.value < pythFee`, Solidity 0.8 arithmetic causes an underflow revert. Because `clearRequest` is called **after** the credit line, the request is never cleared and the user's ETH remains locked. [2](#0-1) 

The code itself acknowledges the missing invariant with a comment:

> *"Note: The provider needs to set its fees to include the fee charged by the Pyth contract. Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the fee computation on IPyth assumes it has the full updated data."* [3](#0-2) 

There is no `cancelRequest`, timeout, or refund function anywhere in the contract. The only way to clear a request is a successful `executeCallback`. [4](#0-3) 

---

### Impact Explanation

If `req.fee < pyth.getUpdateFee(updateData)`:

1. Every call to `executeCallback` reverts on the underflow.
2. `clearRequest` is never reached; the request stays active forever.
3. The user's ETH (minus the already-accrued `_state.pythFeeInWei`) is permanently locked in the contract.
4. There is no refund, cancel, or timeout mechanism.

A third party *could* rescue the funds by calling `executeCallback` with a `msg.value` large enough to cover the shortfall, but there is no economic incentive to do so (the rescuer pays ETH; the provider receives the credit).

---

### Likelihood Explanation

The attack is reachable by an unprivileged provider (anyone can call `registerProvider`):

1. A malicious or misconfigured provider registers with fees set to zero or below the Pyth oracle update fee.
2. Users make requests to this provider (attracted by low fees or by it being the `defaultProvider`).
3. `req.fee = msg.value - _state.pythFeeInWei` is zero or near-zero.
4. `pyth.getUpdateFee(updateData) > req.fee`, so every `executeCallback` reverts.
5. User ETH is locked indefinitely.

Additionally, even a well-intentioned provider is exposed: if the Pyth oracle's `singleUpdateFeeInFeed` is raised by governance after requests are in flight, all pending requests with the old fee become unexecutable. [5](#0-4) 

---

### Recommendation

1. **Add an on-chain invariant check** in `requestPriceUpdatesWithCallback` or `executeCallback` ensuring `req.fee >= pyth.getUpdateFee(updateData)`. Since `updateData` is not available at request time, the check should be placed at the start of `executeCallback` with a clear revert message rather than an arithmetic underflow.

2. **Add a refund / cancellation mechanism** so users can reclaim their ETH if a request is not fulfilled within a timeout window.

3. **Clarify the fee split documentation**: the `_state.pythFeeInWei` in Echo is *not* the Pyth oracle update fee — it is Echo's own protocol fee. The provider's portion (`req.fee`) must cover the actual Pyth oracle update fee. This distinction should be enforced on-chain, not assumed off-chain.

---

### Proof of Concept

```
1. Deploy Echo with pythFeeInWei = 100 wei.
2. Provider registers with baseFeeInWei = 0, feePerFeedInWei = 0, feePerGasInWei = 0.
3. User calls requestPriceUpdatesWithCallback{value: 100}(..., priceIds=[X], callbackGasLimit=0).
   → req.fee = 100 - 100 = 0
   → _state.accruedFeesInWei += 100
4. Pyth oracle charges getUpdateFee(updateData) = 1 wei (minimum).
5. executeCallback is called:
   → (req.fee + msg.value) - pythFee = (0 + 0) - 1 → underflow → revert
6. Request is never cleared. User's 100 wei is locked forever.
   No cancelRequest() exists. No timeout exists.
```
