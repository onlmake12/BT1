### Title
Orphan Pool Denial-of-Service via Random Eviction — Attacker Can Permanently Occupy All Orphan Slots (`tx-pool/src/component/orphan.rs`)

---

### Summary

The CKB `OrphanPool` is capped at `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries. When the cap is exceeded, the eviction strategy in `limit_size()` removes entries **randomly** — using `HashMap::keys().next()` — with no consideration of fee rate, declared cycles, or transaction age. An unprivileged P2P peer can flood all 100 slots with transactions that reference non-existent parent outputs (inputs that will never be resolved), causing legitimate orphan transactions from honest users to be randomly evicted with ~99% probability. This is a direct analog to Issue #2 in the external report: just as nonce-gapped transactions permanently occupy tracked-pool slots in Monad, unresolvable orphan transactions permanently occupy orphan-pool slots in CKB.

---

### Finding Description

**Root cause — `OrphanPool::limit_size()` uses random eviction:** [1](#0-0) 

When `self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS` (100), the code evicts via:

```rust
let id = self.entries.keys().next().cloned().expect("bound checked");
```

`HashMap::keys().next()` is non-deterministic and has no relationship to fee rate, declared cycles, or any priority metric. There is no fee-based comparison against the incoming transaction.

**The cap:** [2](#0-1) 

**Attacker entry path — `after_process` in `process.rs`:**

When a remote peer relays a transaction whose inputs are not in the chain or pool, `is_missing_input` is true and `add_orphan` is called unconditionally: [3](#0-2) 

`add_orphan` calls `add_orphan_tx`, which inserts the transaction and then calls `limit_size()`: [4](#0-3) 

**Orphan expiry — attacker must refresh periodically:** [5](#0-4) 

`ORPHAN_TX_EXPIRE_TIME = 100 * MAX_BLOCK_INTERVAL`. The attacker must re-submit 100 fresh orphan transactions within this window to maintain the attack. The cost is low: 100 minimum-fee transactions every ~16–50 minutes (depending on `MAX_BLOCK_INTERVAL`).

**No fee-based eviction exists in the orphan pool.** Unlike the main `PoolMap`, which uses `EvictKey` sorted by fee rate: [6](#0-5) 

…the `OrphanPool` has no such structure. It stores entries in a plain `HashMap` with no ordering by fee or priority. [7](#0-6) 

---

### Impact Explanation

An attacker controlling 100 orphan-transaction slots prevents any legitimate orphan transaction from persisting in the pool. When a legitimate child transaction arrives before its parent (a normal use case in CKB's two-phase proposal/commit model), it is placed in the orphan pool. If the attacker already occupies all 100 slots, the new legitimate orphan is inserted and then immediately evicted with probability 100/101 ≈ 99%. The evicted transaction is sent a `TxVerificationResult::Reject` signal: [8](#0-7) 

This causes the relay layer to mark the transaction as unknown/rejected in its filter, preventing re-relay. The honest user's transaction is silently dropped and must be resubmitted. Under a sustained attack, legitimate orphan transactions can never persist long enough for their parent to arrive and resolve them.

---

### Likelihood Explanation

- **Attacker-controlled entry path**: Any P2P peer can relay transactions via the standard relay protocol. No privileged role is required.
- **Cost**: 100 transactions with minimum fee rate, refreshed every `ORPHAN_TX_EXPIRE_TIME`. The attacker's orphan transactions reference fabricated `OutPoint`s (non-existent outputs), so they cost only the minimum fee and are never committed.
- **No rate limiting per peer** on orphan insertion is visible in the `add_orphan_tx` path.
- **Persistence**: The attack is self-sustaining as long as the attacker refreshes before expiry.

---

### Recommendation

1. **Fee-based eviction**: When the orphan pool is full, evict the orphan with the lowest declared fee rate (or declared cycles per byte) rather than a random entry. This requires storing declared cycles alongside each entry (already present in `Entry.cycle`) and sorting by `cycle / tx_size` as a proxy for fee rate.

2. **Per-peer orphan limit**: Cap the number of orphan transactions accepted from a single peer (e.g., 10 per peer), preventing a single attacker from occupying all 100 slots.

3. **Incoming transaction priority**: Before evicting, compare the incoming transaction's fee rate against the lowest-fee-rate orphan currently in the pool. Only admit the new transaction if its fee rate is higher, mirroring the main pool's `limit_size` behavior.

---

### Proof of Concept

1. Attacker generates 100 transactions, each spending a fabricated `OutPoint` (e.g., `tx_hash = [0xAA; 32], index = 0`) that does not exist on-chain or in the pool.
2. Attacker relays all 100 transactions to the target CKB node via the P2P relay protocol. Each is rejected with `is_missing_input = true` and placed in the orphan pool via `add_orphan`.
3. The orphan pool is now at capacity (100/100).
4. An honest user sends a child transaction (spending the output of a pending parent) via P2P relay. It enters `add_orphan_tx`, is inserted, and `limit_size()` is called. One of the 101 entries is randomly evicted — the honest transaction is evicted with probability 100/101.
5. The attacker refreshes their 100 orphan transactions before `ORPHAN_TX_EXPIRE_TIME` elapses, maintaining the attack indefinitely.
6. The honest user's child transaction is never retained long enough for its parent to arrive and resolve it. [9](#0-8) [10](#0-9)

    fn limit_size(&mut self) -> Vec<Byte32> {
        let now = ckb_systemtime::unix_time().as_secs();
        let expires: Vec<_> = self
            .entries
            .iter()
            .filter_map(|(id, entry)| {
                if entry.expires_at <= now {
                    Some(id)
                } else {
                    None
                }
            })
            .cloned()
            .collect();

        let mut evicted_txs = vec![];

        for id in expires {
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        while self.len() > DEFAULT_MAX_ORPHAN_TRANSACTIONS {
            // Evict a random orphan:
            let id = self.entries.keys().next().cloned().expect("bound checked");
            if let Some(entry) = self.remove_orphan_tx(&id) {
                evicted_txs.push(entry.tx.hash());
            }
        }

        if !evicted_txs.is_empty() {
            trace!("OrphanTxPool full, evicted {} tx", evicted_txs.len());
            self.shrink_to_fit();
        }
        evicted_txs
    }

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }

impl Ord for EvictKey {
    fn cmp(&self, other: &Self) -> Ordering {
        if self.fee_rate == other.fee_rate {

                self.timestamp.cmp(&other.timestamp)
