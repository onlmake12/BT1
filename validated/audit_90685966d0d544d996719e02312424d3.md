Audit Report

## Title
Full-Node Peers Can Subscribe to Light Client State Broadcasts Without Peer-Type Verification — (`util/light-client-protocol-server/src/components/get_last_state.rs`)

## Summary
`GetLastStateProcess::execute()` unconditionally sets `peer.if_lightclient_subscribed = true` for any peer that sends `GetLastState(subscribe=true)`, with no check that the peer is a light-client-only peer. Because `ProtocolTypeCheckerService::opened_protocol_type()` uses an `.all()` superset check (not an exact match), a full-node peer that opens `Sync` plus `LightClient` passes validation and can subscribe. On every new block, the node sends `SendLastState` to every subscribed peer — including unintended full-node peers — wasting bandwidth proportional to the number of malicious subscribers. The claim's assertion of "N extra MMR root computations" is factually incorrect; the MMR root is computed once per block unconditionally, not once per subscriber.

## Finding Description

`GetLastStateProcess::execute()` sets the subscription flag with no peer-type guard:

```rust
// util/light-client-protocol-server/src/components/get_last_state.rs L29-38
pub(crate) async fn execute(self) -> Status {
    let subscribe: bool = self.message.subscribe().into();
    if subscribe {
        self.nc.with_peer_mut(self.peer, Box::new(|peer| {
            peer.if_lightclient_subscribed = true;  // no type check
        }));
    }
    ...
}
``` [1](#0-0) 

The `ProtocolTypeCheckerService` only requires that `Sync` is open (the sole entry in `required_protocol_ids`): [2](#0-1) 

Its `opened_protocol_type()` uses `.all()` — a superset check — so a peer with `Sync + LightClient` is classified `FullyOpen` and never disconnected: [3](#0-2) 

The `LightClient` protocol is registered as an open handler with no inbound peer restriction: [4](#0-3) 

The identify protocol's outbound path opens ALL non-Feeler protocols when `required_flags_filter` passes, meaning a standard full-node peer connecting to a LightClient-enabled node automatically opens the LightClient protocol: [5](#0-4) 

On every block relay, `build_and_broadcast_compact_block` computes the MMR root **once** (unconditionally, not per subscriber), then sends `SendLastState` to all peers with `if_lightclient_subscribed = true`: [6](#0-5) 

The same pattern applies in the miner RPC `submit_block` path: [7](#0-6) 

**Correction to the submitted claim**: The MMR root is computed exactly once per block regardless of subscriber count — not "N times." The actual overhead introduced by malicious subscribers is N extra `SendLastState` network sends per block, bounded by `max_inbound + max_outbound`.

## Impact Explanation

Every malicious subscriber forces one extra `SendLastState` network send per block. With connection limits (e.g., 125 inbound + 8 outbound), an attacker controlling the maximum number of connections causes bounded but unnecessary bandwidth consumption and message-construction overhead on every block. This is a performance degradation affecting the targeted node, matching **Low (501–2000 points): Any other important performance improvements for CKB**. The impact does not reach "crash a node" or "network-wide congestion" because the overhead is bounded by connection limits and the `SendLastState` payload is small.

## Likelihood Explanation

Any unprivileged peer reachable over P2P can trigger this. No special keys or privileges are required. The attacker connects, opens `Sync` (satisfying `ProtocolTypeCheckerService`) and `LightClient`, then sends one `GetLastState(subscribe=true)` message. This is repeatable up to the connection limit and requires no victim mistakes.

## Recommendation

In `GetLastStateProcess::execute()`, before setting `if_lightclient_subscribed = true`, verify that the peer has not opened full-node protocols (e.g., `Sync`/`RelayV3`). Concretely, check `peer.protocols` for the absence of `SupportProtocols::Sync.protocol_id()` before allowing subscription. Alternatively, extend `ProtocolTypeCheckerService` to recognize a `LightClientOnly` type and disconnect peers that open both full-node and light-client protocols simultaneously.

## Proof of Concept

1. Attacker connects to a CKB full node with LightClient enabled.
2. Attacker opens `Sync` (satisfying `ProtocolTypeCheckerService`'s `FullyOpen` check) and `LightClient` protocols.
3. Attacker sends `LightClientMessage::GetLastState { subscribe: true }` on the LightClient channel.
4. `GetLastStateProcess::execute()` sets `peer.if_lightclient_subscribed = true` with no type check.
5. On every subsequent block (via relay or miner RPC), the node sends `SendLastState` to the attacker's peer.
6. Repeating with `max_inbound` connections maximizes bandwidth drain per block.
