### Title
Division-by-Zero Panic in `calculateTwap` When Start and End Slots Are Equal — (`File: target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`Pyth.sol`'s `validateTwapPriceInfo` uses a non-strict inequality (`>`) when comparing `publishSlot` values, allowing the start and end slots to be equal. When equal, `slotDiff = 0`, and the subsequent divisions in `calculateTwap` trigger a Solidity 0.8+ `Panic(0x12)` revert instead of a meaningful protocol error.

### Finding Description

In `validateTwapPriceInfo`, the slot ordering check is:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

This only rejects the case where start slot is strictly greater than end slot. Equal slots (`start.publishSlot == end.publishSlot`) pass validation.

`calculateTwap` then computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot; // = 0
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // Panic: division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // Panic: division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // Panic: division by zero
```

All three divisions by `slotDiff` will trigger a `Panic(0x12)` revert in Solidity ≥0.8.0.

The Solana receiver (`pyth-solana-receiver/src/lib.rs`) correctly enforces strict inequality:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
```

The EVM contract is inconsistent with the Solana implementation and lacks this guard.

### Impact Explanation

Any call to `parseTwapPriceFeedUpdates` with two VAA-signed TWAP messages sharing the same `publishSlot` will revert with an opaque `Panic(0x12)` instead of a descriptive protocol error. The transaction reverts entirely (fee is returned), but the TWAP feature is silently broken for that input. Integrators relying on TWAP data for DeFi decisions (e.g., liquidations, pricing) would receive no result and no actionable error.

### Likelihood Explanation

The `publishSlot` field is embedded in Wormhole-signed accumulator messages; an unprivileged user cannot forge it. However, Pythnet can legitimately produce two TWAP snapshot messages within the same slot (e.g., at epoch boundaries or during rapid re-publishing). If a relayer submits such a pair, the EVM contract panics rather than returning `InvalidTwapSlots`. The Solana receiver already handles this case correctly, confirming the scenario is considered realistic by the protocol designers.

### Recommendation

Change the slot comparison in `validateTwapPriceInfo` from `>` to `>=` to match the Solana receiver's behavior:

```solidity
// Before (allows equal slots → slotDiff = 0 → division by zero)
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}

// After (rejects equal slots, consistent with Solana receiver)
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
```

### Proof of Concept

1. Obtain two valid Wormhole-signed TWAP accumulator messages for the same feed where both have `publishSlot = S` (e.g., two snapshots taken in the same Pythnet slot).
2. Call `parseTwapPriceFeedUpdates(updateData, priceIds)` with these as `updateData[0]` and `updateData[1]`.
3. `validateTwapPriceInfo` passes because `S > S` is false.
4. `calculateTwap` computes `slotDiff = S - S = 0`.
5. `priceDiff / int128(uint128(0))` triggers `Panic(0x12)` — division by zero.
6. Transaction reverts with an opaque panic instead of `InvalidTwapUpdateDataSet`.
