Audit Report

## Title
`PoolMap::add_entry` Overwrites Post-Eviction `total_tx_size`/`total_tx_cycles` With Stale Pre-Computed Values - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `total_tx_size` and `total_tx_cycles` are captured from the pre-eviction baseline at lines 210–211. When `check_and_record_ancestors` (line 213) subsequently evicts `cell_ref_parent` transactions — each of which correctly decrements the live fields via `update_stat_for_remove_tx` — those decrements are immediately overwritten by the stale local values at lines 218–219. The result is a permanent, cumulative inflation of both counters equal to the sum of all evicted entries' sizes and cycles per trigger, enabling a low-cost DoS that renders the tx pool non-functional.

## Finding Description

**Root cause:** `add_entry` (lines 200–221) pre-computes the new totals before any evictions, then unconditionally writes those stale values back after evictions have already modified the live fields.

Step-by-step code path:

1. **Pre-compute** (lines 210–211): `total_tx_size = self.total_tx_size + entry.size` is captured in a local variable while `self.total_tx_size = S`.

2. **Evict** (line 213 → `check_and_record_ancestors` lines 616–625): When `ancestors_count > max_ancestors_count` but `ancestors_count − cell_ref_parents.len() ≤ max_ancestors_count`, the loop calls `remove_entry_and_descendants(next_id)` → `remove_entry` (line 247) → `update_stat_for_remove_tx`, which directly sets `self.total_tx_size = S − e_s` (lines 738–740).

3. **Overwrite** (lines 218–219): `self.total_tx_size = total_tx_size` (the stale `S + entry.size`) discards the decrement, leaving `self.total_tx_size = S + entry.size` instead of the correct `S − e_s + entry.size`.

**Concrete arithmetic per trigger:**
- Inflation = `e_s` bytes / `e_c` cycles (the evicted entry's contribution)
- Inflation is additive across repeated triggers; the pool never self-corrects on this path (`recompute_total_stat` is only called on underflow in `update_stat_for_remove_tx`)

**Why existing guards fail:** `updated_stat_for_add_tx` only checks for integer overflow, not for logical correctness of the baseline. `update_stat_for_remove_tx` correctly decrements the live fields, but those fields are immediately clobbered by the stale snapshot.

## Impact Explanation

`total_tx_size` is the sole admission guard in `limit_size` (`pool.rs` line 298: `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`). Inflating it past `max_tx_pool_size` causes:

1. **Spurious `Reject::Full`** for all new submissions.
2. **Forced eviction of honest transactions** via the `limit_size` loop, which removes lowest-fee-rate entries until the (inflated) counter drops below the limit — but since the counter is wrong, it may evict far more than necessary or loop indefinitely if the pool empties before the counter corrects.

This constitutes a low-cost, unprivileged DoS that renders the tx pool non-functional for all participants on the targeted node, matching the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — High (10001–15000 points)**.

## Likelihood Explanation

The trigger requires only:
1. A chain of `max_ancestors_count − 1` in-pool transactions (T1→T2→…→T24).
2. One additional in-pool transaction X that uses an output of any Ti as a **cell dep** (making X a `cell_ref_parent`).
3. A new transaction Z that spends that same output as an **input**, pushing `ancestors_count` above `max_ancestors_count` and triggering the eviction branch.

All three steps are achievable via the public `send_transaction` RPC or P2P relay with no privileged access, no leaked keys, and a modest CKB balance. The attacker can repeat the sequence to accumulate unbounded inflation. The setup cost is O(max_ancestors_count) transactions per trigger cycle.

## Recommendation

Move the stat pre-computation to **after** all evictions have completed, so it captures the post-eviction baseline:

```rust
pub(crate) fn add_entry(
    &mut self,
    mut entry: TxEntry,
    status: Status,
) -> Result<(bool, HashSet<TxEntry>), Reject> {
    let tx_short_id = entry.proposal_short_id();
    let mut evicts = Default::default();
    if self.entries.get_by_id(&tx_short_id).is_some() {
        return Ok((false, evicts));
    }
    // Overflow guard only — do not capture the result yet
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

    evicts = self.check_and_record_ancestors(&mut entry)?;
    self.record_entry_edges(&entry)?;
    self.insert_entry(&entry, status);
    self.record_entry_descendants(&entry);
    self.track_entry_statics(None, Some(status));

    // Re-compute AFTER evictions so evicted sizes are already subtracted
    let (total_tx_size, total_tx_cycles) =
        self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
    self.total_tx_size = total_tx_size;
    self.total_tx_cycles = total_tx_cycles;
    Ok((true, evicts))
}
```

Alternatively, replace the snapshot pattern with a direct increment (`self.total_tx_size += entry.size`) applied only after all evictions, so intermediate decrements are preserved.

## Proof of Concept

1. Submit transactions T1→T2→…→T24 as a spend chain (24 transactions, each spending the previous output). Pool now has 24 entries; `total_tx_size = S`.
2. Submit transaction X that uses the output of T1 as a **cell dep**. X is now a `cell_ref_parent` of T1. Pool has 25 entries; `total_tx_size = S + X.size`.
3. Submit transaction Z that spends the same output of T1 as an **input** and has T1 as an ancestor.
   - `ancestors_count > max_ancestors_count` → eviction branch entered.
   - `cell_ref_parents = {X}`; eviction condition satisfied → eviction proceeds.
   - `remove_entry_and_descendants(X)` → `update_stat_for_remove_tx(X.size, X.cycles)` → `self.total_tx_size = S + X.size − X.size = S`.
   - Z is inserted; lines 218–219 write back `total_tx_size = S + X.size + Z.size` (stale pre-computed value).
   - Correct value should be `S + Z.size`; actual value is `S + X.size + Z.size`.
   - **Inflation = `X.size` bytes, `X.cycles` cycles.**
4. Call `get_pool_info` RPC: `total_tx_size` exceeds the sum of actual entry sizes by `X.size`.
5. Repeat steps 1–3 (reusing or replacing the chain) to accumulate inflation until `total_tx_size > max_tx_pool_size`, at which point all new submissions receive `Reject::Full` and `limit_size` begins evicting honest transactions.
