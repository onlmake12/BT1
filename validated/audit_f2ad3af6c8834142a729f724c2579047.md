Audit Report

## Title
Mempool Fee Admission Uses Raw Byte Size Instead of Transaction Weight, Enabling High-Cycle Transactions to Bypass the Fee Floor — (`tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using the transaction's raw serialized byte size, while `FeeRate` is defined as shannons per kilo-weight where weight is `max(size_bytes, cycles × DEFAULT_BYTES_PER_CYCLES)`. No weight-based fee check is performed after `verify_rtx` returns the actual cycle count. An unprivileged attacker can submit compact transactions with high-cycle scripts that pass the size-based admission gate, triggering full CKB-VM execution at a fraction of the intended fee cost, enabling CPU exhaustion and mempool flooding.

## Finding Description

**Root cause — size used where weight is required, with no post-verification correction**

`FeeRate` is documented as shannons per kilo-weight: [1](#0-0) 

The correct weight function is: [2](#0-1) 

The admission gate in `check_tx_fee` passes `tx_size` where weight is expected: [3](#0-2) 

The code's own comment acknowledges this is wrong but proceeds anyway. `FeeRate::fee(weight)` computes `self.0 * weight / 1000`: [4](#0-3) 

**No post-verification weight check exists.** In `_process_tx`, the flow is:

1. `pre_check` → `check_tx_fee` with `tx_size` (line 715)
2. `verify_rtx` → returns `verified.cycles` (lines 724–732)
3. `TxEntry::new(rtx, verified.cycles, fee, tx_size)` — entry admitted with actual cycles but fee approved only against size (line 751) [5](#0-4) 

After `verify_rtx` returns the actual cycle count, there is no second call to check whether `fee >= min_fee_rate.fee(get_transaction_weight(tx_size, verified.cycles))`. The size-based pre-check is the only admission gate.

**Concrete numbers** with `min_fee_rate = 1000 shannons/kW`, `tx_size = 200 bytes`, `cycles = 3,500,000,000` (near `MAX_BLOCK_CYCLES`):

| Metric | Value |
|---|---|
| `weight = max(200, 3.5B × 0.000_170_571_4)` | ≈ 597,000 bytes |
| Fee required by `check_tx_fee` | `1000 × 200 / 1000 = 200 shannons` |
| Fee required by correct weight check | `1000 × 597,000 / 1000 = 597,000 shannons` |
| Discount factor | **~2,985×** |

`DEFAULT_BYTES_PER_CYCLES`: [6](#0-5) 

**Existing guards are insufficient.** `TRANSACTION_SIZE_LIMIT = 512 KB` limits serialized size but not cycle count at admission: [7](#0-6) 

The eviction mechanism (`EvictKey`, `AncestorsScoreSortKey`) correctly uses weight-based fee rate, but this only applies after admission — it does not prevent the CPU cost of `verify_rtx` from being incurred: [8](#0-7) 

## Impact Explanation

This matches the allowed CKB bounty impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" — HIGH (10001–15000 points)**.

- **CPU exhaustion**: Each admitted transaction triggers full CKB-VM script execution in `verify_rtx`. With a ~3000× fee discount, an attacker causes ~3000× more CPU work per shannon spent than the protocol intends to allow.
- **Mempool flooding**: The attacker fills the mempool with high-cycle, low-fee-rate entries at negligible cost. The `max_tx_pool_size` cap is reached faster, degrading service for legitimate users.
- **Fee market distortion**: Pool scoring uses correct weight-based fee rate, but admission does not, creating an inconsistency that degrades block assembly quality and miner revenue.

## Likelihood Explanation

- **Attacker profile**: Any unprivileged user with RPC (`send_transaction`) or P2P relay access. No keys, no special role, no majority hashpower required.
- **Preconditions**: None beyond a standard node connection.
- **Trigger**: Submit a transaction with compact serialization (≈200 bytes) and a script that loops for near-`MAX_BLOCK_CYCLES` cycles. Constructable with any custom lock script (e.g., a tight loop in RISC-V bytecode).
- **Repeatability**: The attack is repeatable until the mempool is full; at that point, eviction uses weight-based fee rate, but the CPU damage from `verify_rtx` has already been incurred for each submitted transaction.

## Recommendation

Add a weight-based fee check immediately after `verify_rtx` returns the actual cycle count in `_process_tx`, before creating the `TxEntry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);

// Post-verification weight-based fee check
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

This requires passing `min_fee_rate` into `_process_tx` or reading it from the pool config. The size-based pre-check in `check_tx_fee` can remain as a cheap early rejection for obviously underpaying transactions, but must be followed by the weight-based check once cycles are known.

## Proof of Concept

1. Construct a CKB transaction:
   - One input cell locked by a script executing a tight loop consuming `N ≈ 3,500,000,000` cycles.
   - One output returning most capacity to self.
   - Serialized size ≈ 200 bytes.
   - Fee = `min_fee_rate × 200 / 1000` shannons (e.g., 200 shannons at default rate of 1000 shannons/kW).

2. Submit via `send_transaction` RPC or P2P relay.

3. Observe: `check_tx_fee` passes (`200 ≥ 200` shannons required by size check). `verify_rtx` executes the script, consuming `N` cycles and significant CPU time. Transaction enters the mempool.

4. Correct weight = `max(200, 3,500,000,000 × 0.000_170_571_4)` ≈ 597,000. Correct min fee = 597,000 shannons. Actual fee paid = 200 shannons — a **~2,985× shortfall**.

5. Repeat with many such transactions (each with a distinct input UTXO) to exhaust mempool capacity and node CPU at negligible fee cost. Each iteration incurs full `verify_rtx` execution on the node before any eviction logic runs.
