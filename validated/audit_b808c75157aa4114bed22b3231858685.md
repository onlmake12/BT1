### Title
Unbounded `forward_rate_limiter` HashMapStateStore Growth via Attacker-Controlled `from` PeerId — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The two-layer rate limiting design in the HolePunching protocol contains a structural gap: the outer `rate_limiter` caps message throughput per session, but the inner `forward_rate_limiter` is keyed on attacker-controlled `(from, to, item_id)` tuples. Because `retain_recent()` is never called during the periodic `notify` callback — only on `disconnected` — a peer that stays connected can grow the `forward_rate_limiter`'s `HashMapStateStore` without bound by cycling through distinct `from` PeerIds.

---

### Finding Description

**Layer 1 — outer `rate_limiter`:**
Keyed by `(session_id, msg.item_id())`, quota = 30 req/sec. [1](#0-0) 

**Layer 2 — `forward_rate_limiter`:**
Keyed by `(content.from, content.to, msg_item_id)`, quota = 1 req/sec. The key is fully attacker-controlled because `from` is a field inside the message payload, not derived from the session. [2](#0-1) 

**`retain_recent()` is only called on disconnect:** [3](#0-2) 

**The periodic `notify` (every `CHECK_INTERVAL` = 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter:** [4](#0-3) 

**`forward_rate_limiter` uses `HashMapStateStore`:** [5](#0-4) 

**Attack sequence:**
1. Connect to the victim node as an unprivileged peer (standard P2P connection).
2. Send 30 `ConnectionRequest` messages/sec, each with a freshly generated `from` PeerId.
3. Each message passes the outer limiter (30/sec budget is exactly consumed).
4. Each message calls `forward_rate_limiter.check_key(&(new_from, any_to, item_id))`, inserting a new entry into the `HashMapStateStore`.
5. Maintain the connection indefinitely — `retain_recent()` is never triggered.

**Growth rate:** 30 new entries/sec × 3600 sec/hr = 108,000 entries/hr per session. With N concurrent attacker sessions, growth is N × 108,000 entries/hr.

Each entry holds a `(PeerId, PeerId, u32)` key (~82 bytes) plus governor internal state and `HashMap` overhead (~100 bytes), totalling ~180 bytes/entry. At 125 concurrent sessions over one hour: ~125 × 108,000 × 180 bytes ≈ **2.4 GB**.

---

### Impact Explanation

Sustained memory growth leads to OOM conditions on the victim node, causing it to crash or be killed by the OS. This takes the node offline, halting its ability to validate blocks, relay transactions, and participate in consensus. The impact is a targeted DoS against individual full nodes.

---

### Likelihood Explanation

The attack requires only a standard P2P connection — no special privileges, no PoW, no keys. `PeerId` values are trivially generated (they are just public keys). The attacker controls the `from` field entirely via message payload. The missing `retain_recent()` call in `notify` is a straightforward omission that makes the attack persistent across the 5-minute check interval.

---

### Recommendation

Add `retain_recent()` calls for both rate limiters inside the `notify` callback, alongside the existing cleanup of `pending_delivered` and `inflight_requests`:

```rust
// in notify(), after the existing retain calls:
self.rate_limiter.retain_recent();
self.forward_rate_limiter.retain_recent();
```

This mirrors the pattern already used in `disconnected` and bounds the HashMap size to entries active within the current rate-limit window.

---

### Proof of Concept

**Invariant test sketch:**

```rust
// After sustained attack: send 30 ConnectionRequests/sec for CHECK_INTERVAL seconds
// each with a unique from PeerId, from a single session.
// Assert the forward_rate_limiter has not grown beyond outer_rate_limit entries:
assert!(
    protocol.forward_rate_limiter.len() <= 30,
    "forward_rate_limiter grew to {} entries — invariant violated",
    protocol.forward_rate_limiter.len()
);
```

The test would fail because `retain_recent()` is never called during `notify`, so all 30 × 300 = 9,000 entries (over one `CHECK_INTERVAL`) remain live in the `HashMapStateStore`.
