### Title
`Echo.setFeeManager` Accepts Zero-Address, Permanently Locking Provider Accrued Fees - (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.setFeeManager(address manager)` accepts `address(0)` without validation. Unlike `Entropy.sol`, which provides a direct `withdraw()` path for providers, `Echo.sol` has **no direct provider withdrawal function**. Once a provider sets their `feeManager` to `address(0)`, their accrued fees become permanently inaccessible.

### Finding Description

`Echo.setFeeManager` stores the `manager` argument directly with no zero-address guard:

```solidity
function setFeeManager(address manager) external override {
    require(
        _state.providers[msg.sender].isRegistered,
        "Provider not registered"
    );
    address oldFeeManager = _state.providers[msg.sender].feeManager;
    _state.providers[msg.sender].feeManager = manager;
    emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
}
```

The only withdrawal path for provider fees in `Echo.sol` is `withdrawAsFeeManager`, which enforces:

```solidity
require(
    msg.sender == _state.providers[provider].feeManager,
    "Only fee manager"
);
...
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "Failed to send fees");
```

If `feeManager == address(0)`, the `require` check can never pass (no transaction can originate from `address(0)`), so `withdrawAsFeeManager` is permanently blocked for that provider. There is no fallback `withdraw()` function for providers in `Echo.sol` — unlike `Entropy.sol`, which has:

```solidity
function withdraw(uint128 amount) public override {
    ...
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "withdrawal to msg.sender failed");
}
```

The `IEntropy.sol` interface explicitly documents that passing `address(0)` to `setFeeManager` is a valid way to "disable the fee manager role." A provider following this documented pattern in `Echo.sol` will permanently lose access to their accrued fees, since `Echo.sol` provides no alternative withdrawal path.

### Impact Explanation

Provider accrued fees (`accruedFeesInWei`) become permanently locked in the contract. There is no admin recovery path — `withdrawFees` in `Echo.sol` only withdraws the Pyth protocol fees (`_state.accruedFeesInWei`), not provider fees. The provider's funds are irrecoverable.

### Likelihood Explanation

Any registered provider can trigger this by calling `setFeeManager(address(0))`. The `IEntropy.sol` interface (which `Echo.sol` mirrors) explicitly documents zero-address as the mechanism to disable the fee manager role, making it a realistic mistake for a provider who wants to revoke a previously set fee manager. The likelihood is low-to-medium but the impact is permanent fund loss.

### Recommendation

Add a zero-address guard in `Echo.setFeeManager`, or add a direct `withdraw()` function for providers (as exists in `Entropy.sol`):

```solidity
function setFeeManager(address manager) external override {
    require(
        _state.providers[msg.sender].isRegistered,
        "Provider not registered"
    );
+   require(manager != address(0), "feeManager cannot be zero address");
    address oldFeeManager = _state.providers[msg.sender].feeManager;
    _state.providers[msg.sender].feeManager = manager;
    emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
}
```

Alternatively, add a provider-direct `withdraw()` path analogous to `Entropy.sol`.

### Proof of Concept

1. Provider registers: `echo.registerProvider(baseFee, feedFee, gasFee)`
2. Fees accrue via `requestPriceUpdatesWithCallback` calls
3. Provider calls `echo.setFeeManager(address(0))` (documented as valid in `IEntropy.sol`)
4. Provider attempts `echo.withdrawAsFeeManager(providerAddr, amount)` — reverts with `"Only fee manager"` because `address(0) != msg.sender`
5. No other withdrawal path exists for provider fees in `Echo.sol`
