### Title
Fuel `execute_governance_instruction` Cannot Handle `SetWormholeAddress` and Three Other Governance Actions Added After Initial Deployment — (`target_chains/fuel/contracts/pyth-interface/src/data_structures/governance_instruction.sw`)

---

### Summary

The Fuel Pyth contract's `execute_governance_instruction` dispatch only recognizes governance action bytes `0`–`5`. The canonical `TargetAction` spec defines ten actions (`0`–`9`). Actions `6` (`SetWormholeAddress`), `7` (`SetFeeInToken`), `8` (`SetTransactionFee`), and `9` (`WithdrawFee`) were added to the protocol after the Fuel contract was written and were never backported. Any governance VAA carrying one of these actions submitted to the Fuel contract is permanently unexecutable: the parser maps the action byte to `GovernanceAction::Invalid` and reverts before any state change occurs. The most critical consequence is that governance can never update the Wormhole contract address on Fuel.

---

### Finding Description

The canonical governance action set is defined in `governance/xc_admin/packages/xc_admin_common/src/governance_payload/PythGovernanceAction.ts`:

```
TargetAction = {
  UpgradeContract: 0,
  AuthorizeGovernanceDataSourceTransfer: 1,
  SetDataSources: 2,
  SetFee: 3,
  SetValidPeriod: 4,
  RequestGovernanceDataSourceTransfer: 5,
  SetWormholeAddress: 6,   // ← missing in Fuel
  SetFeeInToken: 7,        // ← missing in Fuel
  SetTransactionFee: 8,    // ← missing in Fuel
  WithdrawFee: 9,          // ← missing in Fuel
}
``` [1](#0-0) 

The EVM contract (`PythGovernanceInstructions.sol`) defines all ten actions and `PythGovernance.sol` dispatches all of them: [2](#0-1) [3](#0-2) 

The Fuel contract's `GovernanceAction` enum, however, stops at action `5`:

```sway
pub enum GovernanceAction {
    UpgradeContract: (), // 0
    AuthorizeGovernanceDataSourceTransfer: (), // 1
    SetDataSources: (), // 2
    SetFee: (), // 3
    SetValidPeriod: (), // 4
    RequestGovernanceDataSourceTransfer: (), // 5
    Invalid: (),
}
``` [4](#0-3) 

The parser maps any action byte `>= 6` to `GovernanceAction::Invalid` and immediately reverts:

```sway
let governance_action = match action_number {
    0 => GovernanceAction::UpgradeContract,
    1 => GovernanceAction::AuthorizeGovernanceDataSourceTransfer,
    2 => GovernanceAction::SetDataSources,
    3 => GovernanceAction::SetFee,
    4 => GovernanceAction::SetValidPeriod,
    5 => GovernanceAction::RequestGovernanceDataSourceTransfer,
    _ => GovernanceAction::Invalid,   // ← actions 6, 7, 8, 9 land here
};
require(match governance_action {
    GovernanceAction::Invalid => false,
    _ => true,
}, PythError::InvalidGovernanceAction);
``` [5](#0-4) 

Because the revert happens inside `parse_governance_instruction`, which is called **after** `verify_governance_vm` has already incremented and stored the governance sequence number, the governance VAA's sequence slot is consumed even though the action is never applied. The governance VAA cannot be replayed (sequence guard prevents it), so the action is permanently lost on Fuel.

By contrast, the Stylus contract (`target_chains/stylus/contracts/pyth-receiver/src/governance_structs.rs`) correctly defines and handles `SetWormholeAddress` (6), `SetTransactionFee` (8), and `WithdrawFee` (9): [6](#0-5) [7](#0-6) 

The Fuel contract was never updated when these actions were added to the protocol.

---

### Impact Explanation

**`SetWormholeAddress` (action 6) — Critical.** The Fuel contract stores and uses a Wormhole guardian set for all governance VAA verification. If the Wormhole contract on Fuel must be replaced (e.g., due to a Wormhole security incident or a chain-level upgrade), governance is permanently unable to update the address. The Fuel Pyth contract would be frozen: it could neither receive future governance instructions nor be upgraded, because all governance paths flow through Wormhole VAA verification.

**`SetTransactionFee` (action 8) and `WithdrawFee` (action 9).** Governance cannot adjust the transaction fee or recover accumulated fees from the Fuel contract. Fees are permanently locked.

**`SetFeeInToken` (action 7).** Governance cannot configure token-denominated fees on Fuel.

In all cases the governance VAA sequence number is consumed on the failed attempt, so the action cannot be retried with the same VAA.

---

### Likelihood Explanation

Wormhole has undergone contract upgrades in the past and will do so again. The `SetWormholeAddress` governance action exists precisely to handle this scenario. Because the Fuel contract is already deployed and the missing cases cannot be added without a contract upgrade (which itself requires governance), the contract is in a self-referential deadlock if the Wormhole address ever needs to change. The likelihood of needing `SetWormholeAddress` at some point in the contract's lifetime is **high**; the likelihood of needing `SetTransactionFee`/`WithdrawFee` is **medium**.

---

### Recommendation

Add `SetWormholeAddress` (6), `SetFeeInToken` (7), `SetTransactionFee` (8), and `WithdrawFee` (9) to the Fuel `GovernanceAction` enum and to the `parse_governance_instruction` match arm in `governance_instruction.sw`, mirroring the Stylus implementation. Add corresponding handler logic in `main.sw`'s `execute_governance_instruction` match, at minimum implementing `SetWormholeAddress` to update the stored Wormhole guardian-set source.

---

### Proof of Concept

1. Governance signs and broadcasts a VAA with:
   - Module = `Target` (1)
   - Action = `SetWormholeAddress` (6)
   - `targetChainId` = Fuel's Wormhole chain ID
   - Payload = new Wormhole contract address

2. The VAA is submitted to the Fuel Pyth contract via `execute_governance_instruction`.

3. `verify_governance_vm` succeeds; the governance sequence number is incremented and stored.

4. `parse_governance_instruction` reads action byte `6`, matches `_ => GovernanceAction::Invalid`, and `require(false, PythError::InvalidGovernanceAction)` reverts the transaction.

5. Because Fuel/Sway reverts roll back storage writes, the sequence number increment is also rolled back — meaning the VAA **can** be resubmitted. However, the action will always revert because the dispatch will never handle byte `6`. The Wormhole address is permanently unupdatable. [5](#0-4) [8](#0-7)
