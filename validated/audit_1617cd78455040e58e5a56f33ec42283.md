### Title
Cross-Chain Replay of Lazer Price Update Signatures — (`lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` hashes only the raw payload (`keccak256(payload)`) with no chain ID or contract address binding. Because the same contract is intentionally deployed at the same address on 40+ EVM chains with the same trusted signers, any valid Lazer price update accepted on one chain is cryptographically indistinguishable from a valid update on every other chain. An unprivileged relayer can replay a price update across chains.

### Finding Description
In `PythLazer.verifyUpdate()`, the message digest used for ECDSA recovery is:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The hash covers only the payload bytes. It does not include `block.chainid`, `address(this)`, or any other chain- or instance-specific domain separator. The Lazer payload itself contains only a timestamp, channel, and feed data — no chain ID field: [2](#0-1) 

The deployment script uses CreateX with `uint8(0)` in the salt, which the comment explicitly labels **"No crosschain replay protection"**, and the same proxy address (`0xACeA761c27A909d4D3895128EBe6370FDE2dF481`) is registered across every production EVM chain: [3](#0-2) [4](#0-3) 

The same trusted signers are registered on all chains via `updateTrustedSigner()`. Therefore, a signature that satisfies `isValidSigner(signer)` on Ethereum also satisfies it on Arbitrum, Base, BSC, Polygon, and every other deployment. [5](#0-4) 

### Impact Explanation
A relayer or any unprivileged caller who obtains a valid `update` blob accepted on Chain A can submit the identical bytes to `verifyUpdate()` on Chain B. The call succeeds, the fee is collected, and the returned `(payload, signer)` tuple is indistinguishable from a legitimately sourced update for Chain B. Consumer contracts that rely on `verifyUpdate()` as their sole authenticity gate cannot detect that the update originated from a different chain. If the Lazer server ever signs chain-differentiated payloads (e.g., different channel configurations, different feed subsets, or chain-specific metadata), an attacker can silently substitute a cross-chain replay for a genuine update, feeding incorrect price data to DeFi protocols that use Lazer for liquidations, swaps, or collateral valuation.

Even with today's globally-uniform payloads, the absence of domain separation violates the security model: the signature is supposed to attest "this price is valid for this chain at this time," but it actually attests only "this price is valid at this time."

### Likelihood Explanation
The attack requires no privileged access. Any party who can observe mempool or finalized transactions on one EVM chain (trivially possible) can extract the `update` calldata and resubmit it on any other chain. The contract is live on 40+ chains simultaneously. The deployment script's own comment acknowledges the absence of cross-chain replay protection, confirming this is a known gap rather than an oversight in analysis.

### Recommendation
Include `block.chainid` and `address(this)` in the signed digest, analogous to EIP-712 domain separation:

```solidity
bytes32 hash = keccak256(abi.encodePacked(
    block.chainid,
    address(this),
    payload
));
```

The Lazer server must correspondingly include these fields when constructing the message to sign. Long-term, adopt EIP-712 structured signing with a proper domain separator so that signatures are unambiguously scoped to a single contract instance on a single chain.

### Proof of Concept
1. On Ethereum mainnet, call `PythLazer(0xACeA761c27A909d4D3895128EBe6370FDE2dF481).verifyUpdate{value: fee}(updateBytes)` — succeeds, returns `(payload, signer)`.
2. Take the identical `updateBytes` calldata.
3. On Arbitrum mainnet, call `PythLazer(0xACeA761c27A909d4D3895128EBe6370FDE2dF481).verifyUpdate{value: fee}(updateBytes)` — succeeds with the same `(payload, signer)` output.
4. The Arbitrum consumer contract receives and accepts the Ethereum-originated price update with no indication of cross-chain origin.
