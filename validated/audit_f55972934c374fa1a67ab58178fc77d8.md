### Title
Silent `u128`Ôćĺ`u64` Truncation in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Capacity ÔÇö (File: `util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO cell using a `u128` intermediate value, then casts it to `u64` with a bare `as u64` truncation. Every other analogous `u128`Ôćĺ`u64` conversion in the same file uses `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. If the intermediate `u128` result exceeds `u64::MAX`, the silent truncation produces a drastically smaller capacity value with no error, causing the DAO accounting field `S` (secondary issuance accumulator) to be computed incorrectly and potentially causing block assembly to fail via an underflow in `current_s.safe_sub(withdrawed_interests)`.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `withdraw_counted_capacity as u64` cast at line 156 is a **silent truncating cast**. If `withdraw_counted_capacity > u64::MAX`, Rust silently discards the high bits, producing a value in `[0, u64::MAX]` that bears no meaningful relationship to the actual maximum withdrawal.

Every other `u128`Ôćĺ`u64` narrowing in the same file uses a checked conversion:

```rust
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

```rust
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [3](#0-2) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [4](#0-3) 

The inconsistency is structural: the `as u64` path is the only one that does not propagate an error on overflow.

**Downstream consequence ÔÇö block assembly underflow:**

`calculate_maximum_withdraw` is called by `transaction_maximum_withdraw`, which feeds `withdrawed_interests`:

```rust
maximum_withdraws.safe_sub(input_capacities).map_err(Into::into)
``` [5](#0-4) 

If truncation makes `maximum_withdraw < input_capacity` for a DAO cell, `withdrawed_interests` underflows here. Then in `dao_field_with_current_epoch`:

```rust
let current_s = parent_s
    .safe_add(nervosdao_issuance)
    .and_then(|s| s.safe_sub(withdrawed_interests))?;
``` [6](#0-5) 

A `DaoError::Overflow` propagates out of `dao_field_with_current_epoch`, causing block assembly to fail for any block containing a DAO withdrawal transaction. This is the direct CKB analog to `_maxRedemptionRequest()` reverting when the set is insolvent: the internal accounting function that computes the maximum redeemable amount fails, blocking all redemptions.

**Entry paths:**

1. `tx-pool/src/util.rs` `check_tx_fee` Ôćĺ `DaoCalculator::transaction_fee` Ôćĺ `transaction_maximum_withdraw` Ôćĺ `calculate_maximum_withdraw` (tx-pool admission for any DAO withdrawal tx) [7](#0-6) 

2. Block assembler `dao_field_with_current_epoch` Ôćĺ `withdrawed_interests` Ôćĺ `transaction_maximum_withdraw` Ôćĺ `calculate_maximum_withdraw` (block template generation) [8](#0-7) 

3. RPC `calculate_dao_maximum_withdraw` Ôćĺ `calculate_maximum_withdraw` directly (returns silently wrong capacity to caller) [9](#0-8) 

### Impact Explanation

If `withdraw_counted_capacity` overflows `u64`, the truncated value is silently used. Two outcomes are possible depending on the truncated value:

- **Silent loss of funds**: The truncated capacity is smaller than the actual maximum withdrawal. The user's DAO withdrawal transaction is accepted with a lower output capacity than they are entitled to, and the difference is permanently lost.
- **Block assembly failure**: If the truncated `maximum_withdraw` falls below the raw `input_capacity` of the DAO cell, `withdrawed_interests` underflows, `dao_field_with_current_epoch` returns `DaoError::Overflow`, and no block containing a DAO withdrawal can be assembled until the condition resolves ÔÇö a liveness failure for DAO withdrawals.

The `CapacityVerifier` does not protect against this because DAO withdrawal transactions are explicitly exempted from the `OutputsSumOverflow` check:

```rust
if !(self.resolved_transaction.is_cellbase() || self.valid_dao_withdraw_transaction()) {
``` [10](#0-9) 

### Likelihood Explanation

**Low in the near term; latent structural defect.** The condition `withdraw_counted_capacity > u64::MAX` requires:

```
counted_capacity ├Ś withdrawing_ar / deposit_ar > u64::MAX
```

Since `counted_capacity ÔëĄ u64::MAX` and `ar` starts at `10_000_000_000_000_000` (10┬╣ÔüÂ) and grows by a tiny fraction per block, the `ar` ratio would need to approximately double before a maximum-capacity cell triggers the truncation. This would take an extremely long time under normal chain operation. However, the defect is structural: the code is inconsistent with every other `u128`Ôćĺ`u64` conversion in the same file, and the existing test `check_withdraw_calculation_overflows` only catches the `safe_add` overflow path, not the silent truncation path. [11](#0-10) 

### Recommendation

Replace the bare `as u64` cast with a checked conversion consistent with the rest of the file:

```rust
let withdraw_counted_capacity_u64 = u64::try_from(withdraw_counted_capacity)
    .map_err(|_| DaoError::Overflow)?;
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?;
```

This aligns with the pattern used for `reward128`, `miner_issuance128`, and `ar_increase128` in the same `DaoCalculator` implementation. [1](#0-0) 

### Proof of Concept

Construct a DAO cell with `output_capacity` near `u64::MAX` and a `withdrawing_ar` sufficiently larger than `deposit_ar` such that:

```
(output_capacity - occupied_capacity) ├Ś withdrawing_ar / deposit_ar > u64::MAX
```

For example (pseudocode):
```
deposit_ar    = 10_000_000_000_000_000
withdrawing_ar = 20_000_000_000_000_001   // ar doubled
counted_capacity = u64::MAX - occupied_capacity

withdraw_counted_capacity (u128) = (u64::MAX - occ) * 20_000_000_000_000_001
                                   / 10_000_000_000_000_000
                                 Ôëł 2 * u64::MAX   // exceeds u64::MAX

withdraw_counted_capacity as u64 Ôëł u64::MAX - 1   // silent truncation, ~u64::MAX shannons lost
```

The `safe_add(occupied_capacity)` then succeeds (no error), returning a capacity approximately `u64::MAX` shannons less than the correct value. The caller (tx-pool fee checker or block assembler) receives a silently wrong result with no indication of error. [12](#0-11)
