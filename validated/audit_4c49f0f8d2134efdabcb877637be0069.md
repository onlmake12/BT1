Audit Report

## Title
Unbounded `last_n_numbers` allocation via `difficulty_boundary=U256::zero()` bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
An unprivileged light-client peer can send a crafted `GetLastStateProof` message with `difficulty_boundary=U256::zero()`, `last_n_blocks=499`, `start_number=0`, and a valid tip hash. The upfront limit check passes (998 ≤ 1000), but `difficulty_boundary=0` causes `get_first_block_total_difficulty_is_not_less_than` to immediately return block 0 as the boundary. The adjustment guard only widens the boundary when too few blocks follow it — not when too many do — so `last_n_numbers` spans the entire chain height N, and `complete_headers` performs O(N²) sequential DB lookups, entirely bypassing `GET_LAST_STATE_PROOF_LIMIT`.

## Finding Description

**Step 1 — Limit check passes.** [1](#0-0) 
With `difficulties=[]` and `last_n_blocks=499`: `0 + 499×2 = 998 ≤ 1000` — passes.

**Step 2 — Tall-chain branch is taken.** [2](#0-1) 
With `start_number=0` and chain height N >> 499, `N - 0 > 499`, so the `else` branch is entered.

**Step 3 — `difficulty_boundary=U256::zero()` validation passes.** [3](#0-2) 
`difficulties.last()` returns `None`, so `None.map(...).unwrap_or(false)` = `false` — no rejection.

**Step 4 — Binary search immediately returns block 0.** [4](#0-3) 
Genesis block's total difficulty is always `≥ U256::zero()`, so the function returns `Some((0, genesis_diff))` on the very first check, setting `difficulty_boundary_block_number = 0`.

**Step 5 — Adjustment guard does not fire.** [5](#0-4) 
With `difficulty_boundary_block_number=0` and N >> 499: `N - 0 ≥ 499` — the guard only widens the boundary when too few blocks follow it, never when too many do.

**Step 6 — Unbounded `last_n_numbers` is allocated.** [6](#0-5) 
Produces `(0..N).collect()` — a `Vec<u64>` of N entries, unbounded by any limit.

**Step 7 — The `else` branch returns the unbounded vec directly.** [7](#0-6) 
Since `difficulty_boundary_block_number == 0`, the `if` is false and execution falls to the `else`, returning `(Vec::new(), last_n_numbers)` with the full N-entry vec.

**Step 8 — `complete_headers` is called with all N block numbers.** [8](#0-7) 
For each of the N block numbers, `complete_headers` calls `get_ancestor(last_hash, number)`, which traverses the chain from the tip down to the target block — O(N) per call — plus `get_block` and `chain_root_mmr`. Total work is O(N²).

**The constant is entirely bypassed:** [9](#0-8) 

## Impact Explanation

For a chain of height N, a single crafted message causes: a `Vec<u64>` allocation of N entries (8N bytes); N calls to `complete_headers`, each performing an O(N) `get_ancestor` traversal, a `get_block` lookup, and a `chain_root_mmr` computation — total O(N²) synchronous DB/MMR operations on the server's snapshot; and a `SendLastStateProof` response containing N `VerifiableHeader` structs. On a chain of 1M blocks this blocks the server thread for the duration of all DB I/O and can exhaust memory. This matches **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation

The attack requires only a valid tip hash (publicly observable on the P2P network) and a single crafted `GetLastStateProof` message. No proof-of-work, no key, and no privileged role is needed. Any peer connected to the light-client protocol port can trigger it repeatedly and from multiple simultaneous connections, making it trivially repeatable.

## Recommendation

After computing `difficulty_boundary_block_number`, enforce that the derived `last_n_numbers` length is bounded:

```rust
if last_block_number - difficulty_boundary_block_number > constant::GET_LAST_STATE_PROOF_LIMIT as u64 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples after difficulty boundary");
}
```

Additionally, explicitly reject `difficulty_boundary=U256::zero()` at input validation time, since every block satisfies it and it serves no legitimate client purpose.

## Proof of Concept

1. Run a local CKB devnet with the light-client protocol enabled and mine 100,000 blocks.
2. Obtain the tip hash `tip_hash` from the P2P network.
3. Send the following message as an unprivileged peer:
   ```
   GetLastStateProof {
       last_hash: tip_hash,
       start_hash: genesis_hash,
       start_number: 0,
       last_n_blocks: 499,
       difficulty_boundary: U256::zero(),
       difficulties: [],
   }
   ```
4. Observe that `last_n_numbers.len() == 100_000` (not ≤ 1000), the server performs 100,000+ DB lookups, and the response is hundreds of MB.
5. Add the assertion `assert!(last_n_numbers.len() <= 1000)` after line 319 — it fails, confirming the invariant violation.
