Audit Report

## Title
Incorrect Cycle Accounting in `resume_from_state` Inflates Reported Cycles, Causing Valid Transactions to Be Rejected — (File: `script/src/verify.rs`)

## Summary
In `TransactionScriptsVerifier::resume_from_state`, the loop over subsequent script groups (after the resumed group) discards `used_cycles` and uses `consumed_cycles` for both the limit-tracking accumulator and the returned total cycle count. Because `consumed_cycles` includes VM suspension/resumption overhead charged by the Meepo hardfork scheduler, the returned cycle count is systematically inflated. This causes the tx-pool to reject valid transactions whose actual execution cycles are within `max_tx_verify_cycles` but whose inflated reported cycles exceed it.

## Finding Description

In `resume_from_state`, the loop over subsequent groups at lines 322–325 reads:

```rust
Ok(ChunkState::Completed(_, consumed_cycles)) => {
    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, consumed_cycles, group)?;
}
``` [1](#0-0) 

`used_cycles` is silently discarded (`_`) and `consumed_cycles` is used for both accumulators. Compare with the analogous loop in `resumable_verify` at lines 242–245, which correctly separates the two:

```rust
Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
    current_consumed_cycles =
        wrapping_cycles_add(current_consumed_cycles, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
}
``` [2](#0-1) 

The distinction is intentional and documented in the scheduler. The comment in `scheduler.rs` at lines 88–94 explicitly states that cycles charged by suspending/resuming VMs during IO processing are included in `consumed_cycles` but not reflected in `used_cycles`, and this behavior is preserved for the Meepo hardfork: [3](#0-2) 

The overhead constants that create the gap between `consumed_cycles` and `used_cycles` are: [4](#0-3) 

`SPAWN_EXTRA_CYCLES_BASE = 100_000` per spawn call means even a modest number of spawn calls creates a large gap. The inflated `cycles` value returned by `resume_from_state` is then compared against `max_tx_verify_cycles`: [5](#0-4) 

A secondary bug exists in `complete()` at line 395, where `current_group` is used instead of `group` in the `wrapping_cycles_add` call inside the subsequent-groups loop, causing incorrect error-source attribution on `CyclesOverflow`: [6](#0-5) 

## Impact Explanation

This is an **incorrect implementation or behavior of CKB-VM or system scripts** (High, 10001–15000 points). Valid transactions that use `spawn`/`exec`/pipe-IO syscalls and are verified via the chunked/resumable path can be permanently rejected from the tx-pool with `ExceededMaximumCycles`, even though their actual execution cycles are within the limit. The transaction cannot be resubmitted successfully via the same path, effectively censoring it from the pool.

## Likelihood Explanation

Any unprivileged transaction sender can trigger this by submitting a transaction that:
1. Uses `spawn`/`exec`/pipe-IO syscalls (standard CKB-VM V2 features, no special privilege required).
2. Contains multiple script groups (e.g., multiple inputs with different lock scripts).
3. Has total actual execution cycles close to `max_tx_verify_cycles` (70,000,000 by default).

The chunked verification path is activated automatically by the tx-pool when a script group's cycles exceed the per-step budget. With `SPAWN_EXTRA_CYCLES_BASE = 100_000`, only ~700 spawn calls are needed to push the inflated count over the 70M threshold. This is a realistic and repeatable condition.

## Recommendation

In `resume_from_state`, restore the correct separation between `used_cycles` (for the returned total) and `consumed_cycles` (for limit tracking), matching the pattern in `resumable_verify`:

```rust
// Fix for script/src/verify.rs lines 322–325
Ok(ChunkState::Completed(used_cycles, consumed_cycles)) => {
    current_used = wrapping_cycles_add(current_used, consumed_cycles, group)?;
    cycles = wrapping_cycles_add(cycles, used_cycles, group)?;
}
```

Additionally, fix the error-source attribution bug in `complete()` at line 395 by replacing `current_group` with `group` in the subsequent-groups loop body.

## Proof of Concept

1. Construct a transaction with two lock-script groups, each using `ckb_spawn` to launch a child VM that performs pipe IO (charging `SPAWN_EXTRA_CYCLES_BASE + SPAWN_YIELD_CYCLES_BASE` overhead per call).
2. Tune the scripts so that actual `used_cycles` ≈ `max_tx_verify_cycles - 1` but `consumed_cycles` (with overhead) > `max_tx_verify_cycles`.
3. Submit the transaction via `send_transaction` RPC.
4. The tx-pool invokes chunked verification; `resume_from_state` is called for the second script group.
5. The inflated `cycles` value causes `ExceededMaximumCycles`, and the transaction is rejected despite being valid.
6. Confirm by running the same transaction through `verify()` (non-chunked path), which accepts it successfully, demonstrating the divergence between the two paths.
