### Title
Silent Discard of RBF-Recovered Transaction Re-Enqueue Failures in tx-pool — (`File: tx-pool/src/process.rs`)

### Summary
Inside `TxPoolService::submit_entry`, after a Replace-By-Fee (RBF) operation removes conflicting transactions, the code attempts to re-enqueue "recovered" (non-directly-conflicting but displaced) transactions back into the verify queue. The return value of `queue.add_tx()` is unconditionally discarded with `let _ = ...`, meaning any failure — including a full queue — is silently swallowed with no error log and no retry. This is a direct analog to the ERC20 unchecked-transfer-return-value pattern: a critical state-mutating operation's failure is invisible to the system.

### Finding Description

In `tx-pool/src/process.rs`, the `submit_entry` method calls `process_rbf` to remove conflicting transactions and collect "may_recovered_txs" — transactions that were displaced but are not themselves in conflict with the new RBF transaction. These are supposed to be re-verified and re-submitted. The re-enqueue is done in a spawned task: [1](#0-0) 

```rust
if !may_recovered_txs.is_empty() {
    let self_clone = self.clone();
    tokio::spawn(async move {
        let mut queue = self_clone.verify_queue.write().await;
        for tx in may_recovered_txs {
            debug!("recover back: {:?}", tx.proposal_short_id());
            let _ = queue.add_tx(tx, false, None);  // ← return value silently discarded
        }
    });
}
```

`queue.add_tx()` returns `Result<bool, Reject>`. The `Reject` variants include `Reject::Full(...)` when the verify queue is at capacity. By writing `let _ = queue.add_tx(...)`, the code:

1. Does not log an error when the add fails.
2. Does not retry or fall back.
3. Does not propagate the failure to any caller.
4. Permanently loses the displaced transaction from the pool with no observable signal. [2](#0-1) 

### Impact Explanation

When `add_tx` fails (e.g., `Reject::Full`), the displaced transaction is permanently evicted from the tx-pool with no record in `recent_reject`, no relay notification, and no operator-visible log at `error!` or `warn!` level. The only trace is a `debug!` line printed *before* the failed add. From the pool's perspective, the transaction simply vanishes. For a user whose transaction was displaced by an RBF replacement, their transaction is gone — not re-queued, not rejected with a reason, not relayed — without any indication of failure. This constitutes silent, unrecoverable transaction loss from the pool.

### Likelihood Explanation

An unprivileged tx-pool submitter can trigger this:

1. Flood the verify queue with low-fee transactions to bring it to capacity (`NOTIFY_CHANNEL_SIZE = 128` for the notify layer; the verify queue has its own limit).
2. Submit an RBF transaction that conflicts with one or more existing pool entries, causing `process_rbf` to remove them and populate `may_recovered_txs`.
3. The spawned task runs while the queue is still full; `add_tx` returns `Reject::Full`; the result is discarded.
4. The displaced transactions are permanently lost.

This requires no privileged access — only the ability to submit transactions via RPC or P2P relay, which is available to any unprivileged peer or RPC caller. [3](#0-2) 

### Recommendation

Replace the silent discard with explicit error handling:

```rust
for tx in may_recovered_txs {
    debug!("recover back: {:?}", tx.proposal_short_id());
    if let Err(e) = queue.add_tx(tx.clone(), false, None) {
        warn!(
            "failed to re-enqueue RBF-recovered tx {}: {}",
            tx.proposal_short_id(),
            e
        );
        // Optionally: record in recent_reject or attempt alternative handling
    }
}
```

Additionally, consider whether a full verify queue should block or delay RBF processing rather than silently dropping recovered transactions.

### Proof of Concept

1. Connect to a CKB node's RPC endpoint as an unprivileged caller.
2. Submit enough transactions to saturate the verify queue.
3. Submit an RBF transaction (higher fee, same input) that conflicts with a pool transaction whose removal would produce `may_recovered_txs`.
4. Observe that the displaced transactions do not reappear in the pool (`get_raw_tx_pool` RPC), are not present in `recent_reject`, and produce no `error!`/`warn!` log — only a `debug!` line for the attempted re-enqueue, with no indication of failure.
