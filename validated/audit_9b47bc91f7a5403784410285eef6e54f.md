### Title
`addFunds` Enforces Post-Deposit Minimum Balance Check, Blocking Incremental Top-Ups - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s `addFunds` function reverts if the subscription's balance after the deposit is still below `getMinimumBalance()`. This mirrors the Isomorph M-5 pattern exactly: a user who wants to top up a drained subscription incrementally is blocked, even though every wei they add is strictly beneficial to the protocol.

---

### Finding Description

In `addFunds` (lines 602–628 of `Scheduler.sol`):

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    ...
    status.balanceInWei += msg.value;          // funds credited first

    if (params.isActive) {
        uint256 minimumBalance = this.getMinimumBalance(
            uint8(params.priceIds.length)
        );
        if (status.balanceInWei < minimumBalance) {   // ← unnecessary gate
            revert SchedulerErrors.InsufficientBalance();
        }
    }
}
``` [1](#0-0) 

After `updatePriceFeeds` drains a subscription's balance below `minimumBalance`, any call to `addFunds` with an amount smaller than the full deficit reverts. The user is forced to supply the entire shortfall in one transaction. There is no protocol-safety reason for this: the funds are already credited to `status.balanceInWei` before the check fires, so a partial top-up leaves the contract in a strictly better state than before.

The test suite itself documents this behaviour as intentional:

```solidity
// Try to add funds that would still leave balance below minimum
// Expect a revert with InsufficientBalance
vm.expectRevert(abi.encodeWithSelector(SchedulerErrors.InsufficientBalance.selector));
scheduler.addFunds{value: insufficientFunds}(subscriptionId);
``` [2](#0-1) 

---

### Impact Explanation

When a subscription's balance is drained below `minimumBalance` by repeated `updatePriceFeeds` calls, the subscription owner (or any third-party funder) cannot restore it incrementally. They must supply the entire deficit in a single transaction. If they cannot (e.g., limited ETH available, multi-sig batching constraints, or gas-limit concerns), the subscription remains underfunded indefinitely. Keepers will stop updating it, and all downstream consumers of that subscription will receive stale prices. This is a direct denial-of-service on the subscription's price-feed service, triggered by the owner's own inability to top up in small steps.

---

### Likelihood Explanation

Subscriptions are designed to be long-running and are funded at creation with only a minimum balance. Over time, `updatePriceFeeds` drains the balance. Any subscription that has been running long enough to fall below `minimumBalance` hits this restriction on the very next `addFunds` call that does not cover the full shortfall. This is a routine operational scenario, not an edge case. [3](#0-2) 

---

### Recommendation

Remove the post-deposit minimum-balance check from `addFunds`. Any deposit increases the balance and is unconditionally beneficial. The minimum-balance gate is appropriate only when *activating* a subscription (i.e., in `createSubscription` and `updateSubscription` when `willBeActive == true`), not when adding funds to an already-active one.

```solidity
function addFunds(uint256 subscriptionId) external payable override {
    SchedulerStructs.SubscriptionParams storage params = _state.subscriptionParams[subscriptionId];
    SchedulerStructs.SubscriptionStatus storage status = _state.subscriptionStatuses[subscriptionId];

    if (!params.isActive) {
        revert SchedulerErrors.InactiveSubscription();
    }
    if (params.isPermanent && msg.value > MAX_DEPOSIT_LIMIT) {
        revert SchedulerErrors.MaxDepositLimitExceeded();
    }

    status.balanceInWei += msg.value;
    // ← remove the post-deposit minimum-balance check entirely
}
```

---

### Proof of Concept

1. Create a subscription with 2 price feeds, funded to exactly `getMinimumBalance(2)`.
2. Call `updatePriceFeeds` five times (advancing time by 60 s each time) to drain the balance below `minimumBalance`.
3. Compute `deficit = minimumBalance - status.balanceInWei`.
4. Call `addFunds{value: deficit - 1}(subscriptionId)` — **reverts** with `InsufficientBalance`, even though the balance is now `minimumBalance - 1` (one wei short).
5. The subscription remains underfunded; keepers cannot profitably update it; downstream consumers receive stale prices.

This exact sequence is reproduced in the existing test `testAddFundsEnforcesMinimumBalance`: [4](#0-3)
