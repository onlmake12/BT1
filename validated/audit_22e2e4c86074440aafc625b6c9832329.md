Audit Report

## Title
Missing `BLOCK_INVALID` Status Guard in `HeaderAcceptor::accept()` Enables Repeated PoW Re-verification and Sync State Corruption - (`File: sync/src/synchronizer/headers_process.rs`)

## Summary

`HeaderAcceptor::accept()` guards against re-processing headers already marked `HEADER_VALID`, but contains a developer-acknowledged `FIXME` with no guard for `BLOCK_INVALID`. Because `BLOCK_INVALID` (`1 << 12`) is a disjoint bit from `HEADER_VALID` (`1`), a header whose block failed full contextual verification falls through all three sub-checks and reaches `insert_valid_header()`, updating the peer's `best_known_header` to point to a known-bad block and bypassing the cached-invalid-status fast-path on every repeated send. Any unprivileged peer can exploit this to force repeated PoW re-verification and corrupt the sync state machine.

## Finding Description

In `sync/src/synchronizer/headers_process.rs` at lines 295–357, `accept()` reads the block status and immediately checks only for `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer best_known_header and return early
    return result;
}
``` [1](#0-0) 

`BlockStatus::BLOCK_INVALID = 1 << 12` and `BlockStatus::HEADER_VALID = 1` are completely disjoint bits: [2](#0-1) 

A header whose block was marked `BLOCK_INVALID` (e.g., after script execution failure in `chain/src/verify.rs`) does **not** satisfy `status.contains(BlockStatus::HEADER_VALID)`, so execution falls through to `prev_block_check` (line 324), `non_contextual_check` (line 334), and `version_check` (line 346). A block that failed full contextual verification (invalid scripts, DAO accounting) will have a structurally valid header that passes all three lighter checks. Execution then reaches:

```rust
sync_shared.insert_valid_header(self.peer, self.header);
``` [3](#0-2) 

This inserts the header into `header_map` and updates the peer's `best_known_header` to the known-bad block's header. The `BLOCK_INVALID` status set in `chain/src/verify.rs` provides no protection against this path: [4](#0-3) 

By contrast, `compact_block_process.rs` correctly short-circuits on `BLOCK_INVALID` before any further processing: [5](#0-4) 

The `SendHeaders` path has no equivalent guard.

## Impact Explanation

**High — "Vulnerabilities or bad designs which could cause CKB network congestion with few costs"**

1. **CPU/DoS exhaustion**: `non_contextual_check` invokes `HeaderVerifier::verify()`, which includes non-trivial PoW verification. With no `BLOCK_INVALID` fast-path, any peer can send repeated `SendHeaders` messages containing a known-invalid header and force the node to re-run PoW verification on every message, with zero cost to the attacker (a single valid-header/invalid-block is sufficient). This is a sustained, low-cost, high-impact DoS against any CKB node.

2. **Sync state corruption**: `best_known_header` for the attacking peer is updated to point to an invalid block's header. The block fetcher (`block_fetcher.rs`) uses `best_known_header` to select sync targets and request blocks, meaning it may repeatedly request and attempt to fully verify a block that will always fail, stalling or misdirecting IBD.

## Likelihood Explanation

The exploit requires only a standard P2P connection — no privileges, no special network position. The attacker needs one block whose header is structurally valid but whose full block fails contextual verification (e.g., invalid scripts, invalid DAO fields). Such blocks are trivially constructable on a test network or observable on mainnet after any rejected block. The `FIXME` comment at line 301 is the developers' own acknowledgment that the gap exists. The attack is repeatable indefinitely.

## Recommendation

Add an early-return guard for `BLOCK_INVALID` immediately after reading the status, mirroring the pattern in `compact_block_process.rs`:

```rust
pub fn accept(&self) -> ValidationResult {
    let mut result = ValidationResult::default();
    let status = self.active_chain.get_block_status(&self.header.hash());

+   if status.contains(BlockStatus::BLOCK_INVALID) {
+       result.invalid(Some(ValidationError::InvalidParent));
+       return result;
+   }

    if status.contains(BlockStatus::HEADER_VALID) {
        // ... existing early return
    }
    // ...
}
```

A dedicated `ValidationError::BlockInvalid` variant would be cleaner than reusing `InvalidParent`, resolving the `FIXME`'s open question about which error type to return.

## Proof of Concept

1. Construct block B with a valid header (valid PoW, valid parent, version 0) but invalid body (e.g., a script that always fails).
2. Submit B to node N via the normal block submission path. N fully verifies B, script execution fails, and `BLOCK_INVALID` is written for B's hash in `block_status_map` (`chain/src/verify.rs` line 177).
3. Connect attacker peer P to N. P sends a `SendHeaders` message containing only B's header.
4. `HeadersProcess::execute()` calls `HeaderAcceptor::accept()` for B's header. `get_block_status` returns `BLOCK_INVALID`. The `HEADER_VALID` check at line 304 fails (disjoint bit). No `BLOCK_INVALID` guard exists.
5. `prev_block_check` passes (B's parent is valid and stored). `non_contextual_check` passes (PoW and structure are valid — this is the expensive step). `version_check` passes.
6. `insert_valid_header(peer, &B_header)` is called at line 356. P's `best_known_header` is now B.
7. Repeat step 3 in a tight loop. Each iteration forces a full `HeaderVerifier::verify()` (PoW check) with no caching protection from the `BLOCK_INVALID` status.
