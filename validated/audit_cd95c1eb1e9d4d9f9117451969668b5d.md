### Title
Missing `script_result_changed_at` for CKB2023 Causes Stale Tx-Pool Cache at VM v2 Hardfork Boundary — (`util/types/src/core/hardfork/ckb2021.rs`, `util/types/src/core/hardfork/ckb2023.rs`)

---

### Summary

`CKB2021` exposes `script_result_changed_at()` to signal the epoch at which VM version changes require tx-pool cache invalidation. `CKB2023` has no equivalent method. When RFC-0049 (VM v2 / Syscalls 3) activates at epoch 12,293 on mainnet, the tx-pool cache invalidation mechanism built for the CKB2021 VM v1 transition is never triggered for the VM v2 transition. Transactions cached under VM v1 semantics remain in the pool and may be re-evaluated with stale cycle counts or incorrect pass/fail results under VM v2.

---

### Finding Description

`CKB2021` defines `script_result_changed_at()`, which returns the epoch number at which VM v1 activates (RFC-0032). This is the hook used by the tx-pool to know when to flush verification caches because script execution results may change across the hardfork boundary. [1](#0-0) 

`CKB2023` introduces RFC-0049 (VM v2 + Syscalls 3), which similarly changes how `ScriptHashType::Type` scripts are executed — they now run on VM v2 instead of VM v1. However, `CKB2023` has no `script_result_changed_at()` or equivalent method: [2](#0-1) 

The VM version selection in `select_version()` is epoch-dependent: before RFC-0049, `ScriptHashType::Type` resolves to VM v1; after RFC-0049, it resolves to VM v2. This selection uses `epoch_number_without_proposal_window()` at verification time: [3](#0-2) 

The tx-pool verifies and caches transaction results keyed by witness hash. When the CKB2023 hardfork epoch arrives, the cache is not invalidated (no `script_result_changed_at` for RFC-0049), so previously cached VM v1 results are served for transactions that should now be re-evaluated under VM v2.

The CHANGELOG confirms this exact class of bug was present and fixed for CKB2021:

> `#3090: Tx-pool refreshes caches with the incorrect VM version after hardfork`
> `#3129: Tx-pool clear statistical data before re-run all transactions` [4](#0-3) 

The `CKB2021` struct's `script_result_changed_at` is the mechanism that was added to address those bugs. `CKB2023` does not implement the equivalent, leaving the same class of bug unaddressed for the VM v2 transition.

---

### Impact Explanation

**High** — A transaction sender submits a transaction to the tx-pool before epoch 12,293. The transaction is verified and cached under VM v1 semantics. When epoch 12,293 arrives and RFC-0049 activates, the cached result is served without re-execution. If the script behaves differently under VM v2 (different ISA extensions: `ISA_B`, `ISA_MOP` are the same, but new Syscalls 3 are available), the cached cycle count or pass/fail result is stale. This can cause:

1. A transaction that would fail under VM v2 to be included in a block template (miner assembles an invalid block).
2. A transaction that would succeed under VM v2 to be incorrectly rejected from the pool.

The `update_tx_pool_for_reorg` path does re-process transactions on reorg, but the epoch-boundary crossing without a reorg is not handled. [5](#0-4) 

---

### Likelihood Explanation

**Medium** — RFC-0049 activates at a fixed, known epoch (12,293 on mainnet, ~2025-07-01). Any transaction submitted to the tx-pool in the blocks immediately before that epoch boundary and not yet committed will be affected. The two-step proposal/commit window (up to 10 blocks) means transactions proposed just before the fork epoch are committed just after it, directly in the affected window. Any tx-pool submitter (RPC caller, relay peer) can trigger this by submitting a transaction with a `ScriptHashType::Type` lock script shortly before the hardfork epoch. [6](#0-5) 

---

### Recommendation

Add `script_result_changed_at()` to `CKB2023` returning `[rfc_0049]` (mirroring the pattern in `CKB2021`), and wire it into the tx-pool cache invalidation logic so that when epoch 12,293 is reached, all cached verification results are flushed and transactions are re-verified under VM v2 semantics. [1](#0-0) [7](#0-6) 

---

### Proof of Concept

1. At epoch 12,292 (one epoch before CKB2023 mainnet activation), submit a transaction via RPC (`send_transaction`) whose lock script uses `ScriptHashType::Type` and whose behavior differs between VM v1 and VM v2 (e.g., uses a new Syscalls 3 opcode that returns a different value under VM v1).
2. The tx-pool verifies and caches the result under VM v1 (pass or fail, with a cycle count).
3. The chain advances to epoch 12,293. RFC-0049 activates. `select_version()` now returns `ScriptVersion::V2` for `ScriptHashType::Type` scripts.
4. Because `CKB2023` has no `script_result_changed_at()`, the cache invalidation hook is never called.
5. The block assembler calls `get_block_template`, which reads the cached (stale) result and includes the transaction in the block template.
6. A miner mines the block. Full node verification re-executes the script under VM v2 and gets a different result, causing the block to be rejected — or, if the cached result was a false negative, the transaction is never included despite being valid. [1](#0-0) [8](#0-7)

    /// Returns the version of the machine based on the script and the consensus rules.
    pub fn select_version(&self, script: &Script) -> Result<ScriptVersion, ScriptError> {
        let is_vm_version_2_and_syscalls_3_enabled = self.is_vm_version_2_and_syscalls_3_enabled();
        let is_vm_version_1_and_syscalls_2_enabled = self.is_vm_version_1_and_syscalls_2_enabled();
        let script_hash_type = ScriptHashType::try_from(script.hash_type())
            .map_err(|err| ScriptError::InvalidScriptHashType(err.to_string()))?;
        match script_hash_type {
            ScriptHashType::Data => Ok(ScriptVersion::V0),
            ScriptHashType::Data1 => {
                if is_vm_version_1_and_syscalls_2_enabled {
                    Ok(ScriptVersion::V1)
                } else {
                    Err(ScriptError::InvalidVmVersion(1))
                }
            }
            ScriptHashType::Data2 => {
                if is_vm_version_2_and_syscalls_3_enabled {
                    Ok(ScriptVersion::V2)
                } else {
                    Err(ScriptError::InvalidVmVersion(2))
                }
            }
            ScriptHashType::Type => {
                if is_vm_version_2_and_syscalls_3_enabled {
                    Ok(ScriptVersion::V2)
                } else if is_vm_version_1_and_syscalls_2_enabled {
                    Ok(ScriptVersion::V1)
                } else {
                    Ok(ScriptVersion::V0)
                }
            }
            hash_type => {
                return Err(ScriptError::InvalidScriptHashType(format!(
                    "The ScriptHashType/{:?} has not been activated, and is not permitted for use.",
                    hash_type
                )));
            }
        }
    }
