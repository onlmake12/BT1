### Title
ETH Sent When Calling `Executor.execute` Is Not Forwarded to the Target Contract and Becomes Locked in the Executor - (File: target_chains/ethereum/contracts/contracts/executor/Executor.sol)

### Summary
The `execute` function in `Executor.sol` is declared `public payable`, but the ETH forwarded to the target contract is `gi.value` (decoded from the governance VAA payload), not `msg.value`. Any ETH a relayer sends alongside the `execute` call is silently retained in the Executor contract and never refunded, while the target contract receives ETH only from the Executor's pre-existing balance as specified by `gi.value`.

### Finding Description
`Executor.execute` is `public payable`, accepting native ETH from any caller:

```solidity
function execute(
    bytes memory encodedVm
) public payable returns (bytes memory response) {
    ...
    (success, response) = address(callAddress).call{value: gi.value}(
        gi.callData
    );
    ...
}
```

The value forwarded to `callAddress` is `gi.value`, which is parsed from the governance instruction embedded in the Wormhole VAA:

```solidity
gi.value = encodedInstruction.toUint256(index);
```

`gi.value` is entirely determined by the governance emitter and is independent of `msg.value`. Any ETH sent by the relayer via `msg.value` is added to the Executor's balance but is **never forwarded to the target** and **never refunded**. The Executor has a `receive()` fallback, so it silently absorbs the ETH:

```solidity
receive() external payable {}
```

A relayer who sends `msg.value = X` alongside a VAA that specifies `gi.value = 0` will lose `X` ETH: the target receives nothing, and the Executor's balance grows by `X`.

### Impact Explanation
Any unprivileged relayer who calls `execute` with `msg.value > 0` loses that ETH. The ETH is retained in the Executor contract and is only recoverable via a future governance instruction that explicitly specifies a non-zero `gi.value` pointing back to the relayer — an outcome entirely outside the relayer's control. Because `execute` is `public payable`, a relayer has a reasonable expectation that ETH sent with the call will be forwarded to the governance target, mirroring the pattern used in the external report. The mismatch between the `payable` interface and the actual forwarding logic is the root cause.

### Likelihood Explanation
`execute` is `public` — any address can call it with a valid VAA. A relayer who observes that a governance instruction requires `gi.value > 0` ETH to be forwarded may attempt to fund the call by attaching `msg.value` equal to `gi.value`, not realising the Executor must be pre-funded separately. This is a realistic operational mistake, especially for new integrators. The function's `payable` modifier actively invites this misuse.

### Recommendation
Two options:

1. **Remove `payable` from `execute`** if the intent is that the Executor must be pre-funded via `receive()` before governance calls. This makes the interface unambiguous.

2. **Refund excess `msg.value`** after the call, or forward `msg.value` to the target in addition to `gi.value`:

```solidity
(success, response) = address(callAddress).call{value: gi.value + msg.value}(
    gi.callData
);
```

Option 1 is safer and matches the existing test suite, which pre-funds the Executor with `vm.deal` rather than sending ETH through `execute`.

### Proof of Concept
1. Executor has 0 ETH balance.
2. Governance emitter publishes a VAA with `gi.value = 0` targeting `callAddress`.
3. Relayer calls `executor.execute{value: 1 ether}(vaa)`.
4. `address(callAddress).call{value: 0}(gi.callData)` executes — target receives **0 ETH**.
5. `address(executor).balance == 1 ether` — relayer's ETH is locked.
6. No refund path exists for the relayer; recovery requires a new governance VAA. [1](#0-0) [2](#0-1) [3](#0-2)

        bytes memory encodedVm
    ) public payable returns (bytes memory response) {
        IWormhole.VM memory vm = verifyGovernanceVM(encodedVm);

            vm.payload
        );

        if (gi.targetChainId != chainId && gi.targetChainId != 0)
            revert ExecutorErrors.InvalidGovernanceTarget();

        if (
            gi.action != ExecutorAction.Execute ||
            gi.executorAddress != address(this)
        ) revert ExecutorErrors.DeserializationError();

        // Check if the gi.callAddress is a contract account.
        uint len;
        address callAddress = address(gi.callAddress);
        assembly {
            len := extcodesize(callAddress)
        }
        if (len == 0) revert ExecutorErrors.InvalidContractTarget();

        bool success;
        (success, response) = address(callAddress).call{value: gi.value}(
            gi.callData
        );
