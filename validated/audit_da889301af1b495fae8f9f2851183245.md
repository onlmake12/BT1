### Title
Missing Peer Ban for Unsolicited BlockFilters/BlockFilterHashes/BlockFilterCheckPoints on Filter Protocol — (`sync/src/filter/mod.rs`)

### Summary
The `BlockFilter` protocol handler returns `Status::ignored()` for unsolicited response-type messages (`BlockFilters`, `BlockFilterHashes`, `BlockFilterCheckPoints`), which never triggers a peer ban. There is also no rate limiter on the Filter protocol handler. An unprivileged remote peer can flood a full node with large, well-formed messages indefinitely without being disconnected or banned.

### Finding Description

In `try_process`, the three response-type message arms explicitly return `Status::ignored()` with a developer-acknowledged `// TODO: ban remote peer` comment: [1](#0-0) 

`Status::ignored()` maps to `StatusCode::Ignored = 101`: [2](#0-1) 

The `should_ban()` function only returns a ban duration for codes in the 400–499 range, so code 101 produces no ban: [3](#0-2) 

The `process()` caller therefore never calls `nc.ban_peer()` for these messages: [4](#0-3) 

A grep of `sync/src/filter/mod.rs` for `rate_limiter` returns zero matches. By contrast, the `Relayer` handler has a `governor`-based rate limiter capped at 30 req/sec per `(PeerIndex, message_type)`: [5](#0-4) 

And `HolePunching` similarly has a per-peer rate limiter: [6](#0-5) 

The Filter protocol has neither guard.

`BlockFilters` messages are capped at 1.8 MB per message at the sender side: [7](#0-6) 

An attacker sending well-formed `BlockFilters` messages (which pass `from_compatible_slice`) at high rate will never be banned and will never be rate-limited at the application layer.

### Impact Explanation

- **Bandwidth exhaustion**: Up to 1.8 MB per message, unlimited rate, zero application-level throttling.
- **No automatic remediation**: The node never bans or disconnects the peer, so the attack is self-sustaining for the lifetime of the TCP connection.
- **Async handler load**: Every message is parsed, matched, and logged — cheap per message, but with no upper bound on message rate.
- **Severity correction vs. the question**: The claim of "consensus message processing starvation" is overstated. The per-message processing cost is trivial (parse + log + return). The realistic impact is **sustained bandwidth DoS and log spam**, not async runtime saturation. Severity is Medium–High, not Critical.

### Likelihood Explanation

Any peer that speaks the Filter protocol can trigger this with a single TCP connection. No special privileges, no PoW, no key material required. The `// TODO: ban remote peer` comment confirms the developers already identified this gap.

### Recommendation

Replace `Status::ignored()` in the three unsolicited-response arms with a ban-triggering status code (4xx range), e.g.:

```rust
StatusCode::ProtocolMessageIsMalformed
    .with_context("unsolicited BlockFilters/BlockFilterHashes/BlockFilterCheckPoints")
```

Additionally, add a `governor`-based rate limiter to `BlockFilter` (keyed by `(PeerIndex, message_type)`) mirroring the pattern already used in `Relayer` and `HolePunching`.

### Proof of Concept

1. Connect to a CKB full node on the Filter protocol (`SupportProtocols::Filter`).
2. Construct a valid `BlockFilters` molecule message (any `start_number`, empty `block_hashes`/`filters` vectors — passes `from_compatible_slice`).
3. Send it in a tight loop.
4. Observe: the node logs `"Received unexpected message from peer"` for every message, never calls `ban_peer`, and the connection remains open indefinitely.
5. Expected (fixed) behavior: peer is banned after the first such message with `BAD_MESSAGE_BAN_TIME` (5 minutes). [8](#0-7)
