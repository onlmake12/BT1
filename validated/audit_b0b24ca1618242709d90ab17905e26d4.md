Audit Report

## Title
Out-of-Bounds Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

## Summary
`TxConfirmStat::remove_unconfirmed_tx` guards the `block_unconfirmed_txs` circular-buffer write with a `tx_age >= self.block_unconfirmed_txs.len()` check, but the immediately following `confirm_blocks_to_failed_txs[tx_age - 1]` write is outside that guard with no bounds check. When a tracked transaction is evicted after more than 1000 blocks with `count_failure = true`, `tx_age - 1 >= 1000` indexes past the end of a `Vec` of length 1000, producing an unconditional Rust index-out-of-bounds panic that crashes the node's fee estimator.

## Finding Description
`MAX_CONFIRM_BLOCKS = 1000` is defined at line 22, and both `confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` are initialized with exactly 1000 rows: [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 only protects the `block_unconfirmed_txs` write. The `count_failure` branch at lines 214–215 is unconditional — it executes regardless of `tx_age`: [2](#0-1) 

When `tx_age > 1000`, `tx_age - 1 >= 1000` is an out-of-bounds index into a `Vec` of length 1000. Rust panics here with no recovery path.

The `count_failure = true` path is set by `drop_tx`: [3](#0-2) 

which is called from `reject_tx`: [4](#0-3) 

`drop_tx_inner` passes `self.best_height` as `tip_height`, meaning `tx_age = best_height - entry_height` can grow arbitrarily large: [5](#0-4) 

Full exploit path: unprivileged user submits a low-fee-rate tx → tx is tracked at `entry_height` → 1001+ blocks pass → tx-pool evicts the tx → `reject_tx` → `drop_tx(count_failure=true)` → `remove_unconfirmed_tx` with `tx_age > 1000` → `confirm_blocks_to_failed_txs[tx_age - 1]` panics. Confirmed txs use `count_failure = false` (via `process_block_tx`) and are safe.

## Impact Explanation
A Rust index-out-of-bounds panic is unrecoverable and terminates the thread. The fee estimator runs in the tx-pool service thread; a panic there halts transaction acceptance and block template generation on the affected node. This matches **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node**. The crash is conditional on the node operator having selected the `ConfirmationFraction` algorithm: [6](#0-5) 

## Likelihood Explanation
CKB blocks are produced every ~8–10 seconds; 1001 blocks ≈ 2.2–2.8 hours. The CKB tx-pool default expiry is on the order of days, so a low-fee-rate transaction submitted by any unprivileged user can trivially remain unconfirmed for 1001+ blocks before being evicted. No special privilege, key, or hashpower is required — only a valid transaction submission followed by waiting. The condition is repeatable and deterministic.

## Recommendation
Add a bounds check before the `confirm_blocks_to_failed_txs` write, mirroring the existing guard for `block_unconfirmed_txs`:

```rust
if count_failure && tx_age >= 1 && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

This clamps failure recording to the tracked window and silently drops samples for txs older than `MAX_CONFIRM_BLOCKS`, which is already the intended behavior for the unconfirmed-tx tracking side.

## Proof of Concept

```rust
#[test]
fn test_no_panic_on_long_lived_tx() {
    let mut stat = TxConfirmStat::default(); // MAX_CONFIRM_BLOCKS = 1000
    let entry_height: u64 = 0;
    let fee_rate = FeeRate::from_u64(1000);
    let bucket_index = stat.add_unconfirmed_tx(entry_height, fee_rate).unwrap();

    // Simulate 1001 blocks passing
    let tip_height: u64 = 1001;
    // tx_age = 1001, tx_age - 1 = 1000, confirm_blocks_to_failed_txs.len() = 1000
    // → index 1000 on a Vec of length 1000 → panic!
    stat.remove_unconfirmed_tx(entry_height, tip_height, bucket_index, true);
}
```

This test panics at line 215 on the current code: [7](#0-6)
