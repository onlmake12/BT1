Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Post-Eviction `total_tx_size`/`total_tx_cycles` in `add_entry` — (File: tx-pool/src/component/pool_map.rs)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures pre-eviction totals into locals before `check_and_record_ancestors` runs. That inner call can evict `cell_ref_parent` transactions via `remove_entry_and_descendants`, which correctly decrements `self.total_tx_size` through `update_stat_for_remove_tx`. The stale locals are then unconditionally written back at lines 218–219, permanently inflating `total_tx_size` by the size of every evicted transaction. This causes `limit_size` to spuriously evict valid transactions and causes future submissions to be prematurely rejected with `Reject::Full`.

## Finding Description
The exact sequence in `add_entry` (lines 210–219 of `tx-pool/src/component/pool_map.rs`):

```rust
// Step 1: locals = self.total_tx_size + entry.size  (no mutation)
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;   // lines 210–211

// Step 2: may evict cell_ref_parent txs; each eviction calls
//         update_stat_for_remove_tx → self.total_tx_size -= evicted_size
evicts = self.check_and_record_ancestors(&mut entry)?;          // line 213

// Step 3: stale pre-eviction value overwrites the decremented self.total_tx_size
self.total_tx_size  = total_tx_size;                            // line 218  ← BUG
self.total_tx_cycles = total_tx_cycles;                         // line 219
```

`updated_stat_for_add_tx` (lines 711–728) is a pure read: it reads `self.total_tx_size`, adds `tx_size`, and returns the result — it does not mutate any field. `check_and_record_ancestors` (lines 588–639) enters the eviction branch at line 603 when `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, and calls `remove_entry_and_descendants` (line 618) in a loop. `remove_entry_and_descendants` calls `remove_entry` for each removed transaction; `remove_entry` calls `update_stat_for_remove_tx` at line 247, which correctly decrements `self.total_tx_size`. Step 3 then blindly overwrites that decremented value with the stale local, inflating the counter by exactly the sum of evicted transaction sizes. The correct post-operation value is `S + new_size - evicted_size`; the actual stored value is `S + new_size`.

The only existing guard in `updated_stat_for_add_tx` is a `checked_add` overflow check (lines 716–727), which does not address the ordering problem. There is no re-read of `self.total_tx_size` after evictions complete.

## Impact Explanation
`total_tx_size` is the sole counter driving `limit_size` in `pool.rs` (line 298): `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter produces two concrete effects: (1) **Spurious evictions** — `limit_size` is called immediately after `add_entry` returns, sees an inflated total, and evicts additional valid fee-paying transactions that should remain in the pool; (2) **Premature rejection** — subsequent calls to `updated_stat_for_add_tx` start from an already-inflated baseline, causing valid transactions to be rejected with `Reject::Full` even when the pool has real capacity. An attacker who can repeatedly trigger the eviction path can continuously starve the mempool of legitimate transactions at low cost. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs*.

## Likelihood Explanation
The eviction path in `check_and_record_ancestors` is triggered when a new transaction has `cell_ref_parents` (in-pool transactions that use one of its cell deps) and the total ancestor count including those parents exceeds `max_ancestors_count` (default 25). This condition is reachable by any unprivileged `send_transaction` RPC caller or relay peer. An attacker can observe mempool state via `get_raw_tx_pool`, construct a chain of 24 transactions sharing a cell dep, submit a cell-dep-referencing transaction `R`, then submit a new transaction that pushes the ancestor count over 25, reliably triggering the bug. The attack is repeatable and requires no special privileges.

## Recommendation
Move `updated_stat_for_add_tx` to after all mutations (including `check_and_record_ancestors`) complete, so eviction-driven decrements are already reflected in `self.total_tx_size` before the new entry's contribution is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Re-read post-eviction totals, then add only the new entry's contribution
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, remove the local variables entirely and apply the increment in-place after all mutations complete.

## Proof of Concept
1. Fill the pool with a chain of 24 transactions `T1 → T2 → … → T24` where each spends the previous output.
2. Submit transaction `R` that takes `T1`'s output as a **cell dep** (making `R` a `cell_ref_parent`).
3. Submit `T_new` that spends `T1`'s output. `T_new` now has 24 ancestors + `R` as a cell-ref parent, exceeding `max_ancestors_count = 25`.
4. `check_and_record_ancestors` evicts `R` (size `S_R`) via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by `S_R`.
5. Step 3 of `add_entry` overwrites `self.total_tx_size` with the pre-eviction value, inflating it by `S_R`.
6. Verify via `get_tip_tx_pool_info` that `total_tx_size` exceeds the sum of actual entry sizes, and observe that `limit_size` subsequently evicts a valid transaction that fits within `max_tx_pool_size`.
7. A unit test can assert `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum()` after the above sequence to confirm the invariant is violated.
