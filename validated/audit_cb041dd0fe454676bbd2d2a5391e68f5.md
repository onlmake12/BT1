### Title
Global `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` Constant Allows Stale Price Data Injection Into Short-Heartbeat Subscriptions — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The Pyth Pulse `Scheduler` contract enforces a single hardcoded constant `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours` as the maximum acceptable age of submitted price data for **all** subscriptions, regardless of each subscription's configured `heartbeatSeconds`. Any unprivileged caller of `updatePriceFeeds()` can submit a valid but old Wormhole-attested Pyth price update (up to 59 minutes old) to a subscription whose heartbeat is, e.g., 10 seconds. The stale data passes all on-chain checks, is stored as the canonical price, and is served to downstream readers via `getPricesUnsafe()`.

---

### Finding Description

`SchedulerConstants.sol` defines:

```solidity
uint64 public constant PAST_TIMESTAMP_MAX_VALIDITY_PERIOD = 1 hours;
``` [1](#0-0) 

Inside `_validateShouldUpdatePrices()`, this constant is used to compute the minimum acceptable price timestamp for every subscription uniformly:

```solidity
uint256 minAllowedTimestamp = (block.timestamp >
    PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    ? (block.timestamp - PAST_TIMESTAMP_MAX_VALIDITY_PERIOD)
    : 0;

if (updateTimestamp < minAllowedTimestamp) {
    revert SchedulerErrors.TimestampTooOld(...);
}
``` [2](#0-1) 

Immediately after, the heartbeat check fires:

```solidity
if (params.updateCriteria.updateOnHeartbeat) {
    if (
        lastUpdateTime == 0 ||
        updateTimestamp >= lastUpdateTime + params.updateCriteria.heartbeatSeconds
    ) {
        return updateTimestamp;
    }
}
``` [3](#0-2) 

There is **no relationship** between `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` and `heartbeatSeconds`. For a subscription with `heartbeatSeconds = 10`, the staleness window accepted by the contract is 3,600× larger than the intended update cadence.

`updatePriceFeeds()` carries no access control — any address may call it: [4](#0-3) 

The Pyth price data submitted must be Wormhole-guardian-signed, but **historical signed updates are publicly available from Hermes** for any past timestamp. An attacker does not need to forge anything.

After a successful call, `status.priceLastUpdatedAt` is set to the stale `latestPublishTime`: [5](#0-4) 

Readers calling `getPricesUnsafe()` then receive the stale price with no revert: [6](#0-5) 

---

### Impact Explanation

Downstream DeFi protocols (lending, perps, AMMs) that read prices via `getPricesUnsafe()` — the natural choice when the subscription's own heartbeat is the freshness guarantee — receive prices that may be up to 59 minutes stale. This can cause:

- **Incorrect liquidations** (collateral valued at a stale price that has since moved significantly).
- **Mispriced derivative settlement** (perp funding rates or option payouts computed on stale data).
- **Arbitrage extraction** against any protocol that trusts the Scheduler's stored price as fresh.

The subscription's `priceLastUpdatedAt` is set to the stale publish timestamp, so `getPricesNoOlderThan()` callers with a tight threshold will revert (denial of service), while `getPricesUnsafe()` callers silently consume the stale value.

---

### Likelihood Explanation

- `updatePriceFeeds()` is permissionless — no privileged role is required.
- Valid historical Pyth price attestations for any past timestamp are freely retrievable from the public Hermes API.
- The attacker pays only the Pyth update fee (a few wei) and the keeper fee (reimbursed from the subscription balance), making the attack essentially free to the attacker while draining the victim subscription's balance.
- Any subscription with `heartbeatSeconds` significantly shorter than 3,600 seconds is affected, which covers all crypto-asset subscriptions.

---

### Recommendation

Bound the minimum allowed timestamp by the subscription's own heartbeat when heartbeat mode is active:

```solidity
uint64 effectiveMaxAge = PAST_TIMESTAMP_MAX_VALIDITY_PERIOD;
if (params.updateCriteria.updateOnHeartbeat) {
    effectiveMaxAge = uint64(params.updateCriteria.heartbeatSeconds);
}
uint256 minAllowedTimestamp = (block.timestamp > effectiveMaxAge)
    ? (block.timestamp - effectiveMaxAge)
    : 0;
```

This mirrors the per-oracle heartbeat approach recommended in the referenced Prisma Finance report and ensures that a subscription configured for 10-second updates cannot be satisfied by 59-minute-old data.

---

### Proof of Concept

1. Deploy Scheduler; create a subscription with `heartbeatSeconds = 10`, one crypto price feed (e.g. ETH/USD), and sufficient balance.
2. A legitimate keeper updates the subscription at block time `T` with a fresh price `P_fresh` (timestamp ≈ `T`). `priceLastUpdatedAt = T`.
3. At block time `T + 3500` (≈ 58 min later), the attacker fetches from Hermes a valid Wormhole-attested price update for the same feed with `publishTime = T + 11` (11 seconds after the last stored update — older than `T + 3500` by ~57 min, but newer than `priceLastUpdatedAt`).
4. Attacker calls `updatePriceFeeds(subscriptionId, staleUpdateData)`.
   - `updateTimestamp = T + 11`
   - `minAllowedTimestamp = (T+3500) - 3600 = T - 100` → passes (`T+11 > T-100`).
   - `updateTimestamp > priceLastUpdatedAt` → passes (`T+11 > T`).
   - Heartbeat: `T+11 >= T + 10` → passes.
5. `priceLastUpdatedAt` is now `T + 11`; the stored price is `P_stale` (from 57 minutes ago).
6. Any reader calling `getPricesUnsafe(subscriptionId, priceIds)` receives `P_stale` with no staleness revert, while the true market price may have moved substantially.
