### Title
Protocol Update Fees Permanently Locked in `PriceInfoObject` ŌĆö No Withdrawal Path Exists - (`target_chains/sui/contracts/sources/price_info.move`, `target_chains/sui/contracts/sources/pyth.move`)

---

### Summary

Every call to `update_single_price_feed` deposits the caller's `Coin<SUI>` fee into a `FEE_STORAGE_KEY` dynamic field on the shared `PriceInfoObject`. No function anywhere in the production codebase can extract those coins. The `fee_recipient_address` stored in `State` is vestigial and is explicitly documented as no longer used. All protocol fee revenue is permanently frozen.

---

### Finding Description

`update_single_price_feed` in `pyth.move` enforces a minimum fee check and then unconditionally deposits the **entire** supplied coin into the `PriceInfoObject`:

```
assert!(state::get_base_update_fee(pyth_state) <= coin::value(&fee), E_INSUFFICIENT_FEE);
price_info::deposit_fee_coins(price_info_object, fee);
``` [1](#0-0) 

`deposit_fee_coins` merges the coin into a `FEE_STORAGE_KEY` dynamic object field on the `PriceInfoObject`:

```move
public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
    if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(...)) {
        dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
    } else {
        coin::join(current_fee, fee_coins);
    };
}
``` [2](#0-1) 

`PriceInfoObject` is created as a **shared object** (`transfer::public_share_object`), so no address owns it and no one can unilaterally extract its dynamic fields. [3](#0-2) 

A full grep across all production `.move` files for `withdraw`, `sweep`, `transfer.*fee`, and `FEE_STORAGE_KEY` returns **zero matches outside `price_info.move` itself** ŌĆö confirming no extraction path exists.

The `fee_recipient_address` field in `State` and the `SET_FEE_RECIPIENT` governance action exist but only update an address field; they never move coins:

```move
/// The previous version of the contract sent the fees to a recipient address
/// but this state is not used anymore
/// This module is kept for backward compatibility
``` [4](#0-3) [5](#0-4) 

The `get_balance` read accessor confirms the balance accumulates but provides no withdrawal: [6](#0-5) 

---

### Impact Explanation

Every price update permanently destroys protocol fee revenue. Because `PriceInfoObject` is a shared object with no owner and no withdrawal entry point, the `Coin<SUI>` stored under `FEE_STORAGE_KEY` is irrecoverable under the current package. The loss compounds with every update across every price feed. This matches the "permanent freezing of protocol funds" scope criterion.

---

### Likelihood Explanation

This is not a conditional edge case ŌĆö it fires on **every** successful call to `update_single_price_feed`. The path is fully unprivileged: any caller supplying a valid VAA and sufficient fee triggers it. The codebase itself acknowledges the fee-recipient mechanism "is not used anymore," confirming this is a known architectural gap that was never closed.

---

### Recommendation

Add a privileged `withdraw_fees` entry function (gated by `LatestOnly` or a governance capability) to `price_info.move` that removes the `FEE_STORAGE_KEY` dynamic field and transfers the `Coin<SUI>` to `state::get_fee_recipient()`. Until then, all collected fees are unrecoverable.

---

### Proof of Concept

1. Deploy the contract with `base_update_fee = 50`.
2. Call `update_single_price_feed` with a `Coin<SUI>` of value `500` (10├Ś).
3. Assert `price_info::get_balance(&price_info_object) == 500` ŌĆö confirmed by the existing test pattern at line 817 of `pyth.move`.
4. Enumerate every `public` and `public entry` function across all `.move` sources; assert none accepts a `&mut PriceInfoObject` and returns or transfers a `Coin<SUI>`.
5. Assertion passes ŌĆö the 500 MIST is permanently locked. [7](#0-6)
