### Title
Missing Zero-Price Check in `deriveCrossRate` Causes Division-by-Zero Revert (DoS) - (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

---

### Summary

`PythUtils.deriveCrossRate` validates that `price2` is non-negative but does not check that it is non-zero. When `price2 == 0`, it is passed directly as the denominator to `Math.mulDiv`, which reverts with a division-by-zero panic. Any downstream contract that calls `deriveCrossRate` with a Pyth feed whose price is zero will be permanently bricked for that update cycle.

---

### Finding Description

`deriveCrossRate` in `PythUtils.sol` computes the cross-rate `price1 / price2`. The function guards against negative inputs:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
```

However, `price2 == 0` passes this check. The value is then cast to `uint64` and used as the denominator in two separate `Math.mulDiv` calls:

**Branch 1 (`deltaExponent > 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    10 ** uint64(deltaExponent),
    uint64(price2)          // denominator = 0 → revert
);
```

**Branch 2 (`deltaExponent <= 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // 0 * anything = 0 → revert
);
```

`Math.mulDiv` explicitly reverts on a zero denominator (via Solidity's built-in division check in the `high == 0` fast path, and via `panic(DIVISION_BY_ZERO)` in the 512-bit path).

---

### Impact Explanation

Any smart contract that integrates `deriveCrossRate` from the official Pyth Solidity SDK to compute synthetic cross-rates (e.g., ETH/EUR from ETH/USD and EUR/USD) will revert whenever the quote-side feed (`price2`) returns `0`. This causes a Denial of Service for all operations that depend on the cross-rate computation — including swaps, liquidations, collateral valuations, and any other price-gated logic. The revert is unrecoverable until the feed price becomes non-zero again.

---

### Likelihood Explanation

Pyth price feeds can legitimately publish a price of `0`. The protocol does not enforce a non-zero invariant at the oracle contract level for stored prices. A feed can return `price == 0` during market close, feed initialization, or a publisher outage. The `deriveCrossRate` function is the officially documented and recommended utility for cross-rate derivation, making it a high-value integration target. Any integrator following the official documentation example is exposed.

---

### Recommendation

Add an explicit zero-price check for `price2` before the division, analogous to the existing negative-price check:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
if (price2 == 0) {
    revert PythErrors.InvalidArgument(); // or a new ZeroDenominatorPrice error
}
```

A dedicated `ZeroDenominatorPrice` error in `PythErrors.sol` would be cleaner and more informative for callers.
