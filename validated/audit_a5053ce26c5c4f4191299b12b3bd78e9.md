### Title
Malicious Miner Can Bloat Block Storage Beyond `MAX_BLOCK_BYTES` via Uncle Proposals ‚Äî (File: `verification/src/block_verifier.rs`)

---

### Summary
The `BlockBytesVerifier` explicitly excludes uncle proposal bytes from the block size check using `serialized_size_without_uncle_proposals()`. A malicious miner can stuff uncle blocks with the maximum allowed proposals, causing all validating nodes to store and process data that exceeds the stated `MAX_BLOCK_BYTES` consensus limit, with no additional PoW cost.

---

### Finding Description

`BlockBytesVerifier::verify()` measures block size as:

```rust
let block_bytes = block.data().serialized_size_without_uncle_proposals() as u64;
``` [1](#0-0) 

The underlying function explicitly subtracts uncle proposal bytes from the total serialized block size:

```rust
pub fn serialized_size_without_uncle_proposals(&self) -> usize {
    let block_size = self.as_slice().len();
    let uncles_proposals_size = self
        .uncles()
        .iter()
        .map(|x| x.proposals().as_slice().len() - molecule::NUMBER_SIZE)
        .sum::<usize>();
    block_size - uncles_proposals_size
}
``` [2](#0-1) 

This means the effective size check is:

```
(total_block_size ‚àí uncle_proposals_size) ‚â§ MAX_BLOCK_BYTES
```

The consensus parameters that bound the maximum extra data are:

- `MAX_UNCLE_NUM = 2` uncles per block
- `MAX_BLOCK_PROPOSALS_LIMIT = 1_500` proposals per uncle
- Each `ProposalShortId` = 10 bytes [3](#0-2) [4](#0-3) 

Maximum extra data per block: **2 √ó 1,500 √ó 10 = 30,000 bytes**, on top of the `MAX_BLOCK_BYTES = 597,000` bytes limit. [5](#0-4) 

The `UnclesVerifier` does enforce a per-uncle proposals limit:

```rust
if uncle.data().proposals().len()
    > self.provider.consensus().max_block_proposals_limit() as usize
{
    return Err((UnclesError::ExceededMaximumProposalsLimit).into());
}
``` [6](#0-5) 

But there is **no check on the total uncle proposals across all uncles**, and uncle proposals are **never counted toward `MAX_BLOCK_BYTES`**. All nodes must store the full block including uncle proposals.

Additionally, `TwoPhaseCommitVerifier::verify()` iterates over uncle proposals from every block in the proposal window (9 blocks: `farthest=10`, `closest=2`) on every block verification:

```rust
if let Some(uncles) = self.context.store.get_block_uncles(&block_hash) {
    uncles
        .data()
        .into_iter()
        .for_each(|uncle| proposal_txs_ids.extend(uncle.proposals()));
}
``` [7](#0-6) 

This means each block verification loads and inserts up to `9 √ó 2 √ó 1,500 = 27,000` extra proposal IDs into a `HashSet`, in addition to the main-chain proposals.

---

### Impact Explanation

**Medium** ‚Äî A malicious miner can produce blocks that are up to 30,000 bytes (~5%) larger than the stated `MAX_BLOCK_BYTES` limit by filling uncle blocks with maximum proposals. All validating nodes must:
1. Store the full block including uncle proposals on disk.
2. Relay the full block over the network (compact block relay includes uncle blocks).
3. During `TwoPhaseCommitVerifier`, load and process uncle proposals from disk for every block in the 9-block proposal window, causing extra disk I/O and `HashSet` allocations on every block verification.

At one block per ~8 seconds, a miner controlling all blocks could force ~324 MB/day of extra storage on all nodes beyond what the `MAX_BLOCK_BYTES` limit implies.

---

### Likelihood Explanation

**Medium** ‚Äî Any miner can exploit this at no additional PoW cost. Uncle blocks are a natural byproduct of mining (blocks mined but not on the main chain). A miner simply includes many proposals in their mining attempts; if a block becomes an uncle, it carries those proposals into the next main-chain block. Uncle inclusion also earns uncle rewards, providing a positive economic incentive to include uncles regardless of this attack.

---

### Recommendation

1. **Enforce a total uncle proposals cap**: Add a check in `UnclesVerifier` or `BlockVerifier` that the sum of proposals across all uncles does not exceed `max_block_proposals_limit` (rather than per-uncle only).
2. **Alternatively, include uncle proposals in the block bytes limit**: Change `BlockBytesVerifier` to use the full serialized block size (`block.data().as_slice().len()`) rather than `serialized_size_without_uncle_proposals()`, and adjust `MAX_BLOCK_BYTES` to account for the expected uncle proposal overhead.

---

### Proof of Concept

1. A miner mines uncle blocks, each containing exactly `MAX_BLOCK_PROPOSALS_LIMIT` (1,500) proposals for arbitrary short IDs.
2. The miner includes 2 such uncle blocks in their next main-chain block.
3. The main-chain block passes `BlockBytesVerifier` because `serialized_size_without_uncle_proposals()` subtracts the 30,000 bytes of uncle proposals.
4. The main-chain block passes `UnclesVerifier` because each uncle individually satisfies `proposals.len() ‚â§ max_block_proposals_limit`.
5. All nodes accept, store, and relay the block ‚Äî including the 30,000 bytes of uncle proposals that exceed the effective `MAX_BLOCK_BYTES` limit.
6. For every subsequent block in the 9-block proposal window, `TwoPhaseCommitVerifier` loads and processes these uncle proposals from disk, causing extra I/O and memory allocation on all nodes.
