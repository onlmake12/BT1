### Title
Entropy Revelation Gated by Block Count Instead of Finality When `confirmed_block_status` Defaults to `Latest` — (File: `apps/fortuna/src/api/revelation.rs`)

---

### Summary

The `revelation` endpoint in Fortuna gates provider seed disclosure using a block-count delay (`reveal_delay_blocks`) measured against the block returned by `get_block_number(state.confirmed_block_status)`. Because `confirmed_block_status` is deserialized with `#[serde(default)]` and `BlockStatus` defaults to `BlockStatus::Latest`, any chain whose YAML config omits this field uses the latest (non-finalized) block as the reference. On Ethereum PoS or PoS-based L2s, an attacker who can stall finalization can wait for `reveal_delay_blocks` latest blocks to pass, obtain the provider seed via the public API, brute-force a favorable user commitment, and then reorg the chain to before the original request — biasing the entropy result.

---

### Finding Description

**Root cause — `confirmed_block_status` defaults to `Latest`:**

`EthereumConfig.confirmed_block_status` is annotated `#[serde(default)]`: [1](#0-0) 

`BlockStatus` derives `Default` with `Latest` as the default variant: [2](#0-1) 

Any chain config that omits `confirmed_block_status` therefore silently uses `BlockStatus::Latest`.

**How the revelation check works:**

In the `None`-branch of `revelation()` (no caller-supplied block number), the code fetches the current block at `confirmed_block_status` and compares it against the request's stored `block_number`: [3](#0-2) [4](#0-3) 

When `confirmed_block_status` is `Latest`, `current_block_number` is the tip of the chain. The guard `current_block_number.saturating_sub(reveal_delay_blocks) >= r.block_number` is satisfied as soon as `reveal_delay_blocks` new blocks are appended — regardless of whether any of those blocks are finalized.

The same block-count logic applies to the `Some(block_number)` branch where the caller supplies a block number: [5](#0-4) 

**Secondary issue — `get_request_v2` reads from latest regardless of `confirmed_block_status`:**

Even when an operator sets `confirmed_block_status: Finalized`, the on-chain request lookup in `ethereum.rs` calls `.call().await?` with no block tag, so it always reads from the latest (potentially unfinalized) state: [6](#0-5) 

This means the request existence check is not anchored to the finalized chain state, only the block-number arithmetic is.

---

### Impact Explanation

An attacker who obtains the provider's entropy seed before the user's request transaction is finalized can:

1. Brute-force a user commitment whose hash, combined with the known provider seed, produces an arbitrarily favorable random number.
2. Reorg the chain to before the original request and submit the new commitment.
3. Receive a biased entropy result from the higher-level protocol (e.g., an NFT mint, lottery, or DeFi mechanism).

This directly causes **loss of funds** for other participants in any protocol that relies on Fortuna entropy for fairness.

---

### Likelihood Explanation

- The `confirmed_block_status` field defaults to `Latest` via `#[serde(default)]`, so any chain config that does not explicitly set it to `Finalized` or `Safe` is vulnerable.
- The test fixtures in `api.rs` both use `BlockStatus::Latest`, suggesting this is the common deployment posture.
- Ethereum PoS finality stalls have occurred on mainnet (e.g., May 2023), demonstrating the attack precondition is realistic.
- The revelation endpoint is publicly reachable with no authentication; the attacker only needs to wait for `reveal_delay_blocks` latest blocks.

---

### Recommendation

**Short term:**
1. Change the `get_request_v2` implementation to anchor the state read to the finalized block, mirroring the report's recommendation:

```rust
self.get_request_v2(provider_address, sequence_number)
    .block(ethers::core::types::BlockNumber::Finalized)
    .call()
    .await?
```

2. Change the `#[serde(default)]` for `confirmed_block_status` to default to `BlockStatus::Finalized` (or at minimum `BlockStatus::Safe`) rather than `BlockStatus::Latest` for any Ethereum PoS deployment.

**Long term:**
Audit each supported chain's finality model and enforce that `confirmed_block_status` is set to the appropriate finality tag per chain, with a startup validation error if `Latest` is used on a known PoS chain.

---

### Proof of Concept

1. Deploy Fortuna against an Ethereum PoS chain with `confirmed_block_status` omitted (defaults to `Latest`) and `reveal_delay_blocks: 5`.
2. Attacker submits a `requestWithCallback` transaction at block B; the request is stored on-chain.
3. Attacker begins a finality-stall attack (e.g., exploiting a consensus client bug or controlling >1/3 of validators), preventing epoch finalization while blocks continue to be produced.
4. After 5 new latest blocks (B+5), the check at `revelation.rs:103–104` passes: `(B+5) - 5 >= B`.
5. Attacker calls `GET /v1/chains/{chain_id}/revelations/{sequence}` — the endpoint returns the provider seed.
6. Attacker brute-forces a `user_random_number` such that `keccak256(provider_seed || user_random_number)` is favorable.
7. Attacker reorgs the chain to before block B and submits a new `requestWithCallback` with the crafted commitment.
8. The resulting entropy is biased in the attacker's favor, causing loss of funds in the consuming protocol. [7](#0-6) [2](#0-1) [8](#0-7) [9](#0-8)

        &self,
