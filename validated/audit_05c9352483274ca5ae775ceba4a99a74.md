Audit Report

## Title
Unbounded Gossip Fan-Out in `forward_message` Enables O(N^(3/2)) Message Amplification — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

Any unprivileged peer connected to a CKB node can send a single `ConnectionRequest` whose `to` PeerID is unknown to the network. Every intermediate node that cannot resolve `to` in its local peer registry calls `filter_broadcast` to `√(local_peer_count)` neighbors. The `forward_rate_limiter` is keyed per-node on `(from, to, msg_item_id)` where `msg_item_id` is a constant union-variant index, so the attacker can rotate `(from, to)` pairs to reset every node's rate limiter for each new wave. The result is O(N^(3/2)) total messages across the network from a single attacker packet, sustained at 30 waves/second from one cheap connection.

## Finding Description

**Fan-out to √N when target is unknown.**
In `forward_message`, the `None` branch of the peer registry lookup calls `filter_broadcast` with a closure that counts down from `peers().len().isqrt()`, sending the message to approximately √N neighbors. [1](#0-0) 

**`forward_rate_limiter` key includes a constant `msg_item_id`.**
The rate limiter is keyed by `(content.from, content.to, self.msg_item_id)`. `msg_item_id` is passed in from `mod.rs` as `msg.item_id()`, which is the union variant index — a compile-time constant `0` for all `ConnectionRequest` messages. The effective key is therefore `(from, to)`. An attacker who rotates `(from, to)` pairs gets a fresh rate-limiter slot on every node in the network for each new wave. [2](#0-1) [3](#0-2) 

**Route check is path-specific, not global.**
`content.route.contains(self_peer_id)` only drops the message if the receiving node's own ID appears in the specific path that delivered this copy. `forward_request` appends the forwarding node's ID to the route before re-broadcasting, so the route is per-path. A node that receives the same `(from, to)` via two different paths will pass the route check on both arrivals; only the `forward_rate_limiter` stops the second forward — and that limiter is bypassed by rotating pairs. [4](#0-3) [5](#0-4) 

**`MAX_HOPS = 6` allows deep propagation.** [6](#0-5) 

**Amplification math.**
Each of the N nodes in the network forwards at most once per `(from, to)` pair (rate limiter). Each forward sends to √N peers. Total messages per attacker packet = N × √N = **O(N^(3/2))**. The outer rate limiter allows 30 `ConnectionRequest` packets/second per session, each with a fresh `(from, to)` pair, sustaining the amplification indefinitely. [7](#0-6) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

For a network of N = 10,000 nodes (each with ~100 connections):
- 1 attacker packet → ~1,000,000 forwarded messages across the network.
- 30 packets/second (outer rate limiter) with rotating `(from, to)` pairs → ~30,000,000 messages/second sustained from a single low-cost connection.

This constitutes network-wide message amplification and congestion achievable with no privilege, no PoW, and no key material.

## Likelihood Explanation

- Requires only a standard P2P connection to any honest CKB node.
- The `HolePunching` protocol is enabled by default on production nodes.
- The `to` PeerID only needs to be syntactically valid bytes; it does not need to correspond to any real peer.
- The `from` PeerID is also arbitrary; no signature or proof of ownership is required.
- The attack is repeatable at 30 req/s with rotating `(from, to)` pairs, sustaining the amplification indefinitely.
- No victim mistake or external dependency is required.

## Recommendation

1. **Add a global seen-message cache per node** keyed on a hash of `(from, to, nonce)` where `nonce` is a random value set by the originator and preserved through forwarding. Drop any message whose key is already in the cache. This collapses all duplicate deliveries to a single forward regardless of path.
2. **Alternatively, use a node-global forwarded-set** keyed on `(from, to)` alone (without the constant `msg_item_id`). Once a node has forwarded a `(from, to)` pair, it must not forward it again from any path.
3. **Do not use the union variant index as a rate-limiter key component** — it is a constant and provides no per-message discrimination. Replace it with a per-message nonce field in the protocol buffer.
4. **Cap `max_hops` at a lower value** (e.g., 3) to reduce the reachable amplification depth.

## Proof of Concept

```
1. Attacker connects to node A (standard P2P handshake).
2. Attacker sends:
     ConnectionRequest {
       from = <random_valid_peer_id_1>,
       to   = <random_unknown_peer_id_1>,
       max_hops = 6,
       listen_addrs = [<valid_tcp_addr>],
       route = []
     }
3. Node A: route check passes (A not in []), forward_rate_limiter passes
   (first (from1, to1, 0) on A), to1 not found →
   filter_broadcast to √N neighbors with route=[A].
4. Each of those √N nodes: route check passes (they are not in [A]),
   forward_rate_limiter passes (first time on each node),
   to1 not found → filter_broadcast to √N more neighbors.
5. This repeats for up to 6 hops. Each node forwards at most once
   (rate limiter), but there are up to N nodes, each sending to √N peers.
   Total messages ≈ N · √N >> O(N).
6. Immediately send a second packet with (from2, to2) — each node's
   rate limiter resets for the new key, producing another N·√N wave.
7. Repeat at 30 req/s. Instrument message counters across all nodes to
   confirm total received messages ≈ 30 · N · √N per second.
```

pub(crate) fn forward_request(
    request: packed::ConnectionRequestReader<'_>,
    current_id: &PeerId,
) -> packed::ConnectionRequest {
    let max_hops: u8 = request.max_hops().into();
    let message = request.to_entity();
    let new_route = message
        .route()
        .as_builder()
        .push(current_id.as_bytes())
        .build();
    message
        .as_builder()
        .max_hops(max_hops.saturating_sub(1))
        .route(new_route)
        .build()
}
