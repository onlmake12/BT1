### Title
Unbounded SQL OFFSET via Crafted Pagination Cursor in Ungrouped `get_transactions` — (`util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

---

### Summary

An unprivileged RPC caller can supply a hand-crafted 12-byte `after` cursor whose last 4 bytes decode to `i32::MAX` (2,147,483,647). The value is accepted without any upper-bound check and is emitted verbatim as `OFFSET 2147483647` in the SQL query, forcing SQLite or PostgreSQL to scan and discard up to ~2 billion rows per request. Repeated calls saturate the DB thread pool and make the indexer (and any RPC that shares the same DB connection pool) permanently unresponsive.

---

### Finding Description

**Cursor parsing — no bounds check on the offset field**

The ungrouped path (`group_by_transaction = false | None`) parses the `after` cursor at: [1](#0-0) 

The length check only enforces exactly 12 bytes. `decode_i32` at: [2](#0-1) 

…performs no range validation — it accepts any value in `[i32::MIN, i32::MAX]`.

**Offset injected directly into SQL**

The decoded value is forwarded without any cap to `query_builder.offset()`: [3](#0-2) 

This is the **only** call site of `query_builder.offset` in the entire codebase — the grouped path (`get_tx_with_cells`) uses a keyset `WHERE tx_id > after` and has no OFFSET at all.

**`limit` is validated; `offset` is not** [4](#0-3) 

The `limit` parameter is capped at `self.request_limit`, but there is no analogous guard for the cursor-derived offset.

---

### Impact Explanation

Both SQLite and PostgreSQL implement `OFFSET N` by executing the full query and discarding the first N rows. With `N = 2,147,483,647`, each request causes the DB engine to perform a full sequential scan of the result set (bounded only by the actual table size) before returning zero rows. Because the DB connection pool is shared across all indexer RPC calls, a small number of concurrent crafted requests is sufficient to exhaust the pool and block all legitimate queries indefinitely.

---

### Likelihood Explanation

- The RPC endpoint is reachable by any caller who can connect to the node's RPC port (commonly exposed on mainnet/testnet nodes).
- No authentication, PoW, or privileged role is required.
- The exploit requires only a single JSON-RPC call with a 12-byte hex-encoded `after` value.
- The cursor format is documented and the offset field position is trivially derivable from the source.

---

### Recommendation

Add an explicit upper-bound check on the decoded offset immediately after `decode_i32`, for example:

```rust
let offset = decode_i32(offset)?;
if offset < 0 {
    return Err(Error::Params("'after' offset must be non-negative".into()));
}
// Optionally cap at a sane maximum, e.g. self.request_limit as i32
```

Alternatively, replace the OFFSET-based pagination entirely with a keyset approach (as already used in the grouped path and in `get_cells`), which is both safe and O(log N).

import json, requests, struct

# 8-byte tx_id = 1 (any existing tx), 4-byte offset = i32::MAX

after  = "0x" + cursor.hex()   # "0x0100000000000000ffffff7f"

payload = {
    "id": 1, "jsonrpc": "2.0", "method": "get_transactions",
    "params": [{
        "script": {"code_hash": "0x" + "00"*32, "hash_type": "data", "args": "0x"},
        "script_type": "lock"
    }, "asc", "0x1", after]
}
# Send to http://localhost:8116 (or wherever the indexer RPC is exposed)
r = requests.post("http://localhost:8116", json=payload, timeout=300)
# DB CPU will spike; node becomes unresponsive to subsequent RPC calls
print(r.elapsed.total_seconds())  # expect many seconds / timeout
