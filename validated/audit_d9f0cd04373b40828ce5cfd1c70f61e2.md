### Title
Unbounded `last_n_numbers` in `GetLastStateProofProcess::execute` Bypasses `GET_LAST_STATE_PROOF_LIMIT` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The size guard in `GetLastStateProofProcess::execute` uses `last_n_blocks * 2` as a proxy for actual work, but when `difficulty_boundary` is set to the minimum value (1), `difficulty_boundary_block_number` resolves to block 0, causing `last_n_numbers = (0..H)` — **H elements**, not `last_n_blocks` elements. `complete_headers` then performs O(H × log H) DB reads with no further bound.

---

### Finding Description

**The guard (line 201–205):**

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT   // = 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

With `last_n_blocks=499`, `difficulties=[]`: `0 + 499×2 = 998 ≤ 1000` → **passes**. [1](#0-0) [2](#0-1) 

**The broken path (lines 299–346):**

With `start_block_number=0`, `last_block_number=H`, `difficulty_boundary=1`:

1. `get_first_block_total_difficulty_is_not_less_than(0, H, 1)` returns block 0 (genesis `total_difficulty ≥ 1`), so `difficulty_boundary_block_number = 0`.
2. Check `H - 0 < 499` is false for large H → no adjustment.
3. `last_n_numbers = (0..H).collect()` → **H elements**, not 499.
4. Since `difficulty_boundary_block_number == 0`, the `else` branch fires: `sampled_numbers = []`. [3](#0-2) 

**`complete_headers` cost per element (lines 132–163):**

For each of the H block numbers:
- `snapshot.get_ancestor(last_hash, number)` — O(log H) skip-list traversal
- `snapshot.chain_root_mmr(number - 1).get_root()` — O(log H) DB reads (MMR reads O(log N) nodes)

Total per request: **O(H × log H) DB reads**. [4](#0-3) 

The `chain_root_mmr` call creates a `ChainRootMMR` backed by the RocksDB snapshot, so every `get_root()` call issues real DB reads proportional to the MMR height. [5](#0-4) 

**No rate limiting exists** at the `received` handler level for this message type. [6](#0-5) 

---

### Impact Explanation

A single crafted `GetLastStateProof` message on a chain of height H causes O(H × log H) synchronous DB reads inside `execute`. On a chain of height 1,000,000, this is ~20,000,000 DB reads per request. An attacker sending a stream of such messages from multiple peers can saturate the node's I/O, causing CKB network congestion at negligible cost (no PoW, no stake, no fee).

---

### Likelihood Explanation

The light client protocol is a production P2P sub-protocol. Any peer that connects and speaks the `LightClient` protocol can send this message. No authentication, no PoW, no privileged role is required. The crafted parameters (`last_n_blocks=499`, `difficulties=[]`, `start_number=0`, `difficulty_boundary=1`) are trivially constructable.

---

### Recommendation

Replace the proxy guard with a direct bound on the actual computed `last_n_numbers` length. After computing `difficulty_boundary_block_number`, assert:

```rust
assert!(last_block_number - difficulty_boundary_block_number <= last_n_blocks);
```

Or, more robustly, cap `difficulty_boundary_block_number` to `max(start_block_number, last_block_number.saturating_sub(last_n_blocks))` before computing `last_n_numbers`, ensuring `last_n_numbers.len() ≤ last_n_blocks` always holds. Additionally, add per-peer rate limiting for `GetLastStateProof` messages.

---

### Proof of Concept

1. Connect to a CKB node with the light client protocol enabled on a chain of height H = 1,000,000.
2. Obtain the tip hash `tip_hash`.
3. Send:
   ```
   GetLastStateProof {
     last_hash:           tip_hash,
     start_hash:          genesis_hash,
     start_number:        0,
     last_n_blocks:       499,
     difficulty_boundary: U256::from(1u64),
     difficulties:        [],
   }
   ```
4. The guard computes `0 + 499×2 = 998 ≤ 1000` and passes.
5. `difficulty_boundary_block_number` resolves to 0; `last_n_numbers = (0..1_000_000)`.
6. `complete_headers` issues ~20,000,000 DB reads synchronously.
7. Repeat from multiple peers; measure node I/O saturation and block propagation latency degradation.

                item_name, peer, ban_time, status

            warn!("process {} from {}; result is {}", item_name, peer, status);
