### Title
Preload Thread Panic via Race Between Verify-Thread Block Deletion and Preload-Thread Parent Header Lookup — (`chain/src/preload_unverified_blocks_channel.rs`)

### Summary
A remote peer can crash the `preload_unverified_block` thread by sending two chained blocks P→C where P passes non-contextual (PoW) verification but fails contextual verification. The verify thread deletes P's header from `COLUMN_BLOCK_HEADER` while C is still queued in the preload channel. When the preload thread processes C, it calls `get_block_header(&P.hash()).expect("parent header stored")`, which panics because P's header no longer exists. The preload thread has no `catch_unwind` guard, so the panic terminates the thread and permanently halts all block verification on the node.

### Finding Description

**Thread architecture** (`chain/src/init.rs` lines 49–91):
- `preload_unverified_block` thread: reads `LonelyBlockHash` from a bounded channel (`BLOCK_DOWNLOAD_WINDOW * 10 = 20480`), loads the full block + parent header from DB, forwards `UnverifiedBlock` to the verify channel.
- `verify_blocks` thread: receives `UnverifiedBlock`, runs contextual verification, on failure calls `delete_unverified_block` which calls `StoreTransaction::delete_block`, which explicitly deletes `COLUMN_BLOCK_HEADER` (store/src/transaction.rs line 216).

**Race condition** (exact sequence):

1. Attacker sends P (valid PoW, passes `non_contextual_verify`, inserted into DB).
2. `process_lonely_block` sees P's parent is `BLOCK_STORED` → calls `process_descendant(P)` → inserts P into `is_pending_verify`, enqueues P in preload channel.
3. Attacker sends C (child of P, valid PoW, inserted into DB).
4. `process_lonely_block` sees P is in `is_pending_verify` → calls `process_descendant(C)` → inserts C into `is_pending_verify`, enqueues C in preload channel.
5. Preload thread dequeues P → calls `get_block_header(&P.parent_hash)` (succeeds) → sends `UnverifiedBlock{P}` to verify channel.
6. Verify thread dequeues P → contextual verification fails → calls `self.delete_unverified_block(&block)` → `StoreTransaction::delete_block` removes P's header from `COLUMN_BLOCK_HEADER`.
7. Preload thread dequeues C → calls `get_block_header(&P.hash()).expect("parent header stored")` → returns `None` → **panic**.
8. Panic propagates up through `preload_unverified_channel` → `start()` loop → thread terminates.
9. The `unverified_block_tx` sender is dropped; the verify thread's `unverified_block_rx` returns `Err` → verify thread also exits. [7](#0-6) [8](#0-7) [1](#0-0) 

### Impact Explanation
Crash of the `preload_unverified_block` thread permanently halts all block verification on the targeted node. The node can no longer advance its chain tip, effectively taking it offline from a consensus perspective. The impact is node-level denial of service requiring a restart.

### Likelihood Explanation
The attacker needs to mine two valid-PoW blocks where P fails contextual verification (e.g., invalid transaction referencing a non-existent cell, or invalid coinbase). This is not majority hashpower — any entity with mining hardware can mine two blocks. The race window is wide: the preload channel holds up to 20480 items and the verify channel holds 128; the attacker can widen the window by flooding the preload channel with legitimate blocks before sending P and C, ensuring the verify thread processes P and deletes it well before the preload thread reaches C.

### Recommendation
1. **Immediate fix**: Wrap `load_full_unverified_block_by_hash` in a `catch_unwind`, or replace `.expect("parent header stored")` with graceful error handling that logs and skips the block rather than panicking.
2. **Correct fix**: Do not delete P's block header from the DB until all children of P that are already in the preload channel have been processed. Alternatively, pass the parent header by value through the `LonelyBlockHash` struct at enqueue time (when the header is guaranteed to exist), eliminating the DB lookup in the preload thread entirely.
3. **Defense in depth**: Add a `catch_unwind` guard in `PreloadUnverifiedBlocksChannel::start()` mirroring the one in `ConsumeUnverifiedBlocks::start()`.
