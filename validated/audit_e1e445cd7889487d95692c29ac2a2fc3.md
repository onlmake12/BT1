### Title
Global `inflight_proposals` Map Not Keyed Per-Peer Enables Proposal Relay Denial of Service — (File: `sync/src/types/mod.rs`)

---

### Summary

`SyncState::inflight_proposals` is a `DashMap<ProposalShortId, BlockNumber>` — a global registry keyed only by proposal ID, with no per-peer dimension. A single malicious P2P peer that relays a compact block first can "claim" every proposal short ID in that block, permanently preventing the victim node from requesting those proposals from any legitimate peer. This is a direct structural analog to the reported nonce-not-per-address bug: a shared registry (not per-peer) causes one peer's action to silently block all other peers from performing the same action.

---

### Finding Description

**Root cause — `inflight_proposals` is global, not per-peer**

`SyncState` holds:

```rust
inflight_proposals: DashMap<packed::ProposalShortId, BlockNumber>,
``` [1](#0-0) 

`insert_inflight_proposals` inserts a `(ProposalShortId → BlockNumber)` entry and returns `true` only on first insertion (or if the new block number is strictly higher):

```rust
Entry::Occupied(mut occupied) => {
    if *occupied.get() < block_number {
        occupied.insert(block_number);
        true
    } else {
        false   // ← silently skipped for every other peer
    }
}
Entry::Vacant(entry) => {
    vacant.insert(block_number);
    true
}
``` [2](#0-1) 

**How proposals are requested — `request_proposal_txs`**

When a compact block arrives, `request_proposal_txs` is called. It calls `insert_inflight_proposals` and only sends `GetBlockProposal` to the peer for proposals where the return value is `true`:

```rust
let to_ask_proposals: Vec<ProposalShortId> = shared
    .state()
    .insert_inflight_proposals(fresh_proposals.clone(), block_hash_and_number.number)
    .into_iter()
    .zip(fresh_proposals)
    .filter_map(|(firstly_in, id)| if firstly_in { Some(id) } else { None })
    .collect();
``` [3](#0-2) 

If a second peer relays the same compact block at the same height, `insert_inflight_proposals` returns `false` for every proposal already in the map, so **no `GetBlockProposal` is ever sent to that peer**.

**`clear_expired_inflight_proposals` is only called on receipt of `BlockProposal`**

The only cleanup path is inside `BlockProposalProcess::execute`:

```rust
sync_state.clear_expired_inflight_proposals(
    shared.active_chain().tip_number()
        .saturating_sub(shared.consensus().tx_proposal_window().farthest()),
);
``` [4](#0-3) 

If the attacker never sends a `BlockProposal` response, this cleanup is never triggered for the attacker's proposals. Proposals remain stuck in `inflight_proposals` until the chain advances by `farthest` blocks **and** some other `BlockProposal` message happens to arrive and trigger the cleanup.

**`BlockProposal` responses are gated on `inflight_proposals`**

Even if a legitimate peer proactively sends a `BlockProposal`, it is silently dropped unless the proposal ID is already in `inflight_proposals` **and** `remove_inflight_proposals` succeeds:

```rust
let removes = sync_state.remove_inflight_proposals(&proposals);
for (previously_in, tx) in removes.into_iter().zip(unknown_txs) {
    if previously_in {
        sync_state.mark_as_known_tx(tx.hash());
        asked_txs.push(tx);
    }
}
``` [5](#0-4) 

The attacker's entry is in `inflight_proposals`, but the node never asked the legitimate peer, so the legitimate peer never sends `BlockProposal`. The entry is consumed only when the attacker responds — which they never do.

**Entry point — compact block relay**

`request_proposal_txs` is called from `CompactBlockProcess::execute` after header verification passes:

```rust
let proposals: Vec<_> = compact_block.proposals().into_iter().collect();
self.relayer.request_proposal_txs(
    &self.nc,
    self.peer,
    (header.number(), block_hash.clone()).into(),
    proposals,
);
``` [6](#0-5) 

The compact block header must pass PoW verification, but the attacker does not need to mine it — they only need to relay a legitimately mined compact block to the victim **before** other peers do.

---

### Impact Explanation

The victim node never receives the proposal transactions for the targeted proposals. Because CKB's two-phase commit requires a transaction to be proposed in block N and committed in block N+[2,10], a victim node whose `inflight_proposals` is poisoned for a given proposal ID will:

- Never add the corresponding transaction to its tx pool via the proposal relay path.
- Never include that transaction in a block it mines, losing fee revenue.
- Cause transaction confirmation delays for any transaction whose only relay path goes through the proposal mechanism.

If the attacker targets all proposals in every compact block they relay first, the victim node's miner is effectively cut off from the proposal-relay transaction pipeline for the duration of the attack.

---

### Likelihood Explanation

The attacker is an unprivileged P2P peer — no mining is required. The attacker only needs to:

1. Connect to the victim node as a standard P2P peer.
2. Receive a legitimately mined compact block from the network.
3. Relay it to the victim before other peers (a network-position advantage, not a cryptographic one).
4. Silently drop all `GetBlockProposal` requests from the victim.

This is a realistic network-level race condition. An attacker with low-latency connectivity to both miners and the victim can reliably win the relay race. The attack is persistent: the attacker repeats it for every new compact block, keeping `inflight_proposals` continuously poisoned.

---

### Recommendation

Track `inflight_proposals` per-peer. Change the map type from:

```rust
inflight_proposals: DashMap<packed::ProposalShortId, BlockNumber>
```

to:

```rust
inflight_proposals: DashMap<packed::ProposalShortId, (BlockNumber, PeerIndex)>
``` [7](#0-6) 

When a proposal is already in-flight for peer A, still allow sending `GetBlockProposal` to peer B if peer B's block number is equal or higher. Additionally, add a periodic timer-based cleanup of `inflight_proposals` (independent of receiving `BlockProposal` messages) so stale entries are evicted even when a peer withholds responses.

---

### Proof of Concept

1. Attacker peer connects to victim node.
2. A miner broadcasts compact block B at height H containing proposal short ID `P`.
3. Attacker receives B and immediately relays it to the victim.
4. Victim calls `request_proposal_txs(attacker_peer, H, [P])` → `insert_inflight_proposals([P], H)` returns `true` → victim sends `GetBlockProposal{P}` to attacker.
5. Attacker drops the `GetBlockProposal` message, never sending `BlockProposal`.
6. Legitimate peer also relays compact block B to victim.
7. Victim calls `request_proposal_txs(legit_peer, H, [P])` → `insert_inflight_proposals([P], H)` returns `false` (entry occupied at same height) → **no `GetBlockProposal` sent to legitimate peer**.
8. `clear_expired_inflight_proposals` is never triggered (no `BlockProposal` received).
9. Proposal `P` remains in `inflight_proposals` indefinitely (until chain advances by `farthest` blocks **and** some unrelated `BlockProposal` arrives).
10. The transaction corresponding to `P` is never added to the victim's tx pool via the proposal relay path; the victim's miner cannot commit it. [8](#0-7) [9](#0-8) [10](#0-9)

    pub fn insert_inflight_proposals(
        &self,
        ids: Vec<packed::ProposalShortId>,
        block_number: BlockNumber,
    ) -> Vec<bool> {
        ids.into_iter()
            .map(|id| match self.inflight_proposals.entry(id) {
                dashmap::mapref::entry::Entry::Occupied(mut occupied) => {

                        true
                    } else {
                        false
                    }
                }
                dashmap::mapref::entry::Entry::Vacant(vacant) => {
                    vacant.insert(block_number);
                    true
                }
            })
            .collect()
    }
