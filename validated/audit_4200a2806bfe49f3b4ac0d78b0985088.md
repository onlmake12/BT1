### Title
DAO Withdrawal `header_dep_index` Interpretation Mismatch Between Rust `DaoCalculator` and On-Chain C VM — (`util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the `header_dep_index` from the witness as a full `u64`, while the on-chain DAO script (`dao.c`) running inside CKB-VM reads only the **lowest byte** (treating it as `u8`). This cross-context interpretation mismatch means that a DAO withdrawal transaction with `header_dep_index > 255` is permanently rejected by every Rust node — even though the C VM would accept it — causing transaction censorship and a potential consensus split.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit-block pointer from the witness:

```rust
// dao contract stores header deps index as u64 in the input_type field of WitnessArgs
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and immediately uses the full 64-bit value as an array index:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
```

The on-chain `dao.c` script, however, reads only the **lowest byte** of the same 8-byte little-endian field. The unit test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this divergence:

```
// Pad header_deps to 258 entries so index 257 is valid.
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
```

For a witness value of `257` (`0x0101` in little-endian):

| Layer | Reads index as | Resolves to |
|---|---|---|
| C VM (`dao.c`) | `u8` → `1` | `header_deps[1]` = deposit block ✓ |
| Rust `DaoCalculator` | `u64` → `257` | `header_deps[257]` = withdraw block ✗ |

Rust then compares the resolved header's block number against the cell-data deposit number and finds a mismatch, returning `DaoError::InvalidOutPoint`. The C VM finds no mismatch and would accept the script. [1](#0-0) 

The same `DaoCalculator::transaction_fee` path is invoked in two production locations:

1. **Tx-pool admission** — `check_tx_fee` in `tx-pool/src/util.rs` (line 34–35): every submitted DAO withdrawal is checked here before entering the pool.
2. **Block verification** — `FeeCalculator::transaction_fee` in `verification/src/transaction_verifier.rs` (line 265–272): called during contextual block verification for every resolved transaction. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Transaction censorship (primary impact):** A user who constructs a DAO withdrawal transaction with `header_dep_index > 255` — placing the deposit block hash at the position the C VM resolves (lowest byte) while a different hash sits at the position Rust resolves (full u64) — will have their transaction permanently rejected by every Rust node at the tx-pool level. The user's DAO funds become unwithdrawable through any standard Rust node, even though the on-chain script would accept the transaction.

**Consensus split (secondary impact):** If a miner bypasses the tx-pool and directly includes such a transaction in a block, Rust nodes reject the block during `FeeCalculator::transaction_fee` verification (fee appears negative or invalid), while the C VM would pass script execution. This creates a split between Rust nodes and any hypothetical non-Rust implementation that defers entirely to the C VM. [4](#0-3) 

---

### Likelihood Explanation

A DAO withdrawal transaction normally carries only 2 header_deps (deposit block + withdraw block), so the index is 0 or 1 — well within u8 range. The discrepancy only activates when `header_dep_index ≥ 256`, which requires at least 257 entries in `header_deps`. A user consolidating many DAO cells in a single withdrawal transaction could reach this threshold. The attack is fully reachable by an unprivileged RPC caller with no special keys or roles.

---

### Recommendation

Align the index interpretation between Rust and the C VM. The simplest fix is to truncate the index to its lowest byte in `transaction_maximum_withdraw` before using it as an array index, matching the C VM's actual behavior:

```rust
// Match dao.c: read only the lowest byte of the u64 index
let header_dep_index = LittleEndian::read_u64(&header_deps_index_data.unwrap()) as u8 as usize;
```

Alternatively, if the protocol specification intends the full u64 to be used, the on-chain `dao.c` script must be updated to read the full 8-byte value. Either way, the two layers must agree on the same interpretation before any transaction is accepted or rejected.

---

### Proof of Concept

The existing unit test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` demonstrates the divergence directly:

- 258 `header_deps` are constructed; position `1` holds the deposit block hash, position `257` holds the withdraw block hash.
- Witness `input_type` = `257u64` in little-endian.
- C VM reads lowest byte → index `1` → deposit block (number 100) → matches cell data → **would accept**.
- Rust reads full u64 → index `257` → withdraw block (number 200) → `200 ≠ 100` → **rejects with `DaoError::InvalidOutPoint`**.

The test asserts `result.is_err()`, confirming Rust rejects a transaction the C VM would pass. [5](#0-4)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
