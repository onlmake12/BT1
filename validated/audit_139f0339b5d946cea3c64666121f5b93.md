### Title
Excess Verification Fee Refund Misdirected to Consumer Contract Instead of Original User - (File: `lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` refunds excess ETH to `msg.sender`. When called from an intermediate consumer contract (the canonical integration pattern), `msg.sender` is the consumer contract, not the end user who funded the transaction. Excess fees are permanently trapped in the consumer contract.

### Finding Description

In `PythLazer.verifyUpdate()`, the excess-fee refund is hardcoded to `msg.sender`:

```solidity
function verifyUpdate(
    bytes calldata update
) external payable returns (bytes calldata payload, address signer) {
    // Require fee and refund excess
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
        payable(msg.sender).transfer(msg.value - verification_fee);  // <-- refund to msg.sender
    }
    ...
}
``` [1](#0-0) 

The official Pyth Lazer EVM integration guide explicitly shows consumer contracts calling `verifyUpdate` from within their own `payable` functions:

```solidity
function updatePrice(bytes calldata priceUpdate) public payable {
  uint256 verification_fee = pythLazer.verification_fee();
  (bytes calldata payload, ) = verifyUpdate{ value: verification_fee }(update);
  ...
}
``` [2](#0-1) 

In this call chain:
- End user → `ConsumerContract.updatePrice{value: X}(...)` → `PythLazer.verifyUpdate{value: X}(...)`
- Inside `verifyUpdate`, `msg.sender` is `ConsumerContract`, **not** the end user
- Any excess ETH (`X - verification_fee`) is refunded to `ConsumerContract`, not the end user

This is especially problematic because `verification_fee` is a mutable state variable that can be changed by the owner at any time. A user who queries `verification_fee()` off-chain and sends exactly that amount may find the fee has decreased by the time their transaction executes, creating an excess that is silently absorbed by the consumer contract. Additionally, many real-world consumer contracts forward `msg.value` directly to `verifyUpdate` rather than querying the exact fee first. [3](#0-2) 

### Impact Explanation

Any excess ETH sent to `verifyUpdate` by a consumer contract is refunded to the consumer contract address, not the original transaction sender. Unless the consumer contract explicitly implements logic to forward this refund back to the user (which the official example does not), the ETH is permanently trapped. This constitutes a direct, unrecoverable financial loss to end users of any Pyth Lazer consumer contract.

**Impact: Medium** — direct ETH loss to users, but bounded by the excess over `verification_fee` (currently 1 wei default, but configurable).

### Likelihood Explanation

The integration pattern is explicitly documented and widely used. Consumer contracts that forward `msg.value` to `verifyUpdate` (rather than querying the exact fee) are common. Additionally, any fee decrease between query time and execution time creates an excess. The `verification_fee` is owner-controlled and can change.

**Likelihood: Medium** — requires a consumer contract that passes more than the exact fee, which is a natural and common pattern.

### Recommendation

Add an optional `refundAddress` parameter to `verifyUpdate`, or refactor to accept a recipient for the excess refund:

```diff
function verifyUpdate(
-   bytes calldata update
+   bytes calldata update,
+   address refundAddress
) external payable returns (bytes calldata payload, address signer) {
    require(msg.value >= verification_fee, "Insufficient fee provided");
    if (msg.value > verification_fee) {
-       payable(msg.sender).transfer(msg.value - verification_fee);
+       payable(refundAddress).transfer(msg.value - verification_fee);
    }
    ...
}
```

Alternatively, document clearly that consumer contracts **must** pass exactly `verification_fee` and must never forward `msg.value` directly.

### Proof of Concept

1. Deploy a consumer contract that forwards `msg.value` to `verifyUpdate`:
   ```solidity
   function updatePrice(bytes calldata update) external payable {
       pythLazer.verifyUpdate{value: msg.value}(update);
       // excess refund goes to address(this), not msg.sender
   }
   ```
2. User calls `updatePrice{value: 1 ether}(...)` where `verification_fee = 1 wei`
3. `verifyUpdate` refunds `1 ether - 1 wei` to the consumer contract
4. User receives nothing; ETH is trapped in the consumer contract [4](#0-3) [5](#0-4)
