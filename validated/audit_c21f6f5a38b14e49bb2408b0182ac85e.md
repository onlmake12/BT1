Audit Report

## Title
Snappy Decompression Allocation Bypasses `max_frame_length` Bound in `LengthDelimitedCodecWithCompress::decode()` — (File: network/src/compress.rs)

## Summary
`LengthDelimitedCodecWithCompress::decode()` enforces `max_frame_length` on the compressed wire frame but allocates a decompression buffer sized by the attacker-controlled snappy stream header value, bounded only by `MAX_UNCOMPRESSED_LEN` (8 MB). This allows any unprivileged peer to trigger up to 4× memory amplification per frame on the Sync protocol, enabling memory exhaustion across many concurrent connections.

## Finding Description
`MAX_UNCOMPRESSED_LEN` is hardcoded to `1 << 23` (8 MB) at `compress.rs:13` with no relationship to `max_frame_length`.

In `decode()` at `compress.rs:226`, `self.length_delimited.decode(src)` enforces `max_frame_length` on the wire frame (2 MB for Sync, 4 MB for RelayV3 per `support_protocols.rs:129-130`). This check passes for any compressed payload under the wire limit.

When the compress flag is set (`compress.rs:232`), `decompress_len(&data[1..])` reads the snappy stream header's declared output size — a value fully controlled by the attacker (`compress.rs:233`). The only guard is `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` (`compress.rs:235`), which permits values up to 8 MB.

`BytesMut::zeroed(decompressed_bytes_len)` is then unconditionally allocated at `compress.rs:242` before any actual decompression and before any check against `max_frame_length`. An attacker crafts a snappy stream whose compressed payload is just under `max_frame_length` (passes the wire check) and whose header declares `MAX_UNCOMPRESSED_LEN - 1` bytes of output (passes the `> MAX_UNCOMPRESSED_LEN` guard). The ~8 MB allocation executes regardless of the wire limit.

`CKBProtocol::build()` at `protocols/mod.rs:280-288` confirms `LengthDelimitedCodecWithCompress` is the live codec for all protocols, with `compress: true` set by default at `protocols/mod.rs:219` and `protocols/mod.rs:243`. The same allocation pattern exists in `Message::decompress()` at `compress.rs:81`.

## Impact Explanation
With N simultaneous peers each sending crafted compressed frames on Sync, the node allocates N × 8 MB instead of the intended N × 2 MB — a 4× amplification. For 100 inbound peers, this is 800 MB of peak allocation versus the expected 200 MB. On nodes with 1–2 GB RAM, sustained adversarial traffic can exhaust available memory and crash the node. This matches **High — Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
Any unprivileged peer that can establish a P2P connection can trigger this. No proof-of-work, key, or special role is required. The Sync and RelayV3 protocols are open to all peers by default. The attack is repeatable: a peer can continuously stream crafted frames, each triggering an 8 MB allocation. An attacker coordinating 50–100 peers can sustain the memory pressure needed to crash a target node.

## Recommendation
In `LengthDelimitedCodecWithCompress::decode()` at `compress.rs:235`, after reading `decompressed_bytes_len` from the snappy header, add a second guard:

```rust
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::ErrorKind::InvalidData.into());
}
```

This ensures the decompressed allocation is also bounded by `max_frame_length`, restoring the intended invariant. The same fix should be applied to `Message::decompress()` at `compress.rs:74`, where the identical pattern exists using the global `MAX_UNCOMPRESSED_LEN` constant without a per-protocol frame length bound.

## Proof of Concept
1. Construct a snappy stream using the `snap` crate with a header declaring `MAX_UNCOMPRESSED_LEN - 1` (7,999,999) bytes of uncompressed output, with a minimal compressed payload (a few hundred bytes — well under the 2 MB Sync wire limit).
2. Wrap it in a length-delimited frame and feed it to `LengthDelimitedCodecWithCompress::decode()` configured with `max_frame_length = 2 * 1024 * 1024` (Sync).
3. Observe that `BytesMut::zeroed(7_999_999)` is called — allocating ~8 MB — despite the wire frame being only a few hundred bytes.
4. Assert `allocated_bytes > max_frame_length` — the invariant is broken.
5. Repeat across N concurrent connections to observe N × 8 MB RSS growth, confirming the amplification.
