### Title
Non-Deterministic `min_replace_fee` Based on Mutable Pool Descendant State Allows Griefing of RBF Replacement — (`tx-pool/src/pool.rs`)

### Summary

The `min_replace_fee` value advertised to users via the `get_transaction` RPC is computed from the live, mutable tx-pool state (specifically the fees of all current descendant transactions). Because the pool state can change between when a user queries this value and when they submit their replacement transaction, the advertised threshold is non-deterministic. An unprivileged attacker can front-run a replacement attempt by appending high-fee child transactions to the target transaction, raising the actual required replacement fee above what was advertised, causing the replacement to be rejected.

### Finding Description

The `get_transaction` RPC (Chain module) returns a `min_replace_fee` field for pending transactions when RBF is enabled. This value is computed at query time inside `GetTransactionWithStatus` message handling:

```rust
// tx-pool/src/service.rs line 919
(TxStatus::Pending, tx_pool.min_replace_fee(entry))
```

`min_replace_fee` is computed by `TxPool::min_replace_fee()`:

```rust
// tx-pool/src/pool.rs lines 86-99
pub fn min_replace_fee(&self, tx: &TxEntry) -> Option<Capacity> {
    let mut conflicts = vec![self.get_pool_entry(&tx.proposal_short_id()).unwrap()];
    let descendants = self.pool_map.calc_descendants(&tx.proposal_short_id());
    let descendants = descendants.iter()
        .filter_map(|id| self.get_pool_entry(id))
        .collect::<Vec<_>>();
    conflicts.extend(descendants);
    self.calculate_min_replace_fee(&conflicts, tx.size)
}
```

`calculate_min_replace_fee` sums the fees of the target transaction **plus all its current descendants** and adds the `extra_rbf_fee`:


At submission time, `check_rbf` recomputes this value from the **current** pool state:

```rust
// tx-pool/src/pool.rs lines 662-676
let fee = entry.fee;

    if fee < min_replace_fee {
        return Err(Reject::RBFRejected(format!(
            "Tx's current fee is {}, expect it to >= {} to replace old txs",
            fee, min_replace_fee,
        )));
    }
}
```

The gap between query and submission is the attack window. Any unprivileged user can submit a valid child transaction spending an output of the target transaction. Each such child increases the `min_replace_fee` by its own fee amount. If an attacker adds one or more high-fee descendants between the victim's query and submission, the victim's replacement transaction will be rejected with a higher-than-expected threshold.

### Impact Explanation

A user who queries `get_transaction` to learn the `min_replace_fee`, constructs a replacement transaction with exactly that fee, and then submits it may receive a rejection: `"Tx's current fee is X, expect it to >= Y"` where Y > X. The user's replacement transaction is rejected and placed in the conflicts cache. The user must re-query, reconstruct, re-sign, and resubmit a new replacement transaction with a higher fee. In a sustained griefing scenario, an attacker can keep appending new descendants to the target transaction, making replacement economically infeasible or forcing the victim to pay an unbounded fee premium. This directly harms users relying on the `min_replace_fee` hint for RBF operations.

### Likelihood Explanation

The attack requires the attacker to:
1. Know the target transaction is in the pool (publicly observable via `get_raw_tx_pool`)
2. Construct a valid child transaction spending one of the target transaction's outputs (requires knowing the output structure, which is public)
3. Submit the child transaction before the victim's replacement arrives

All three steps are permissionless and require no privileged access. The attack is cheap: the attacker only needs to pay the `min_fee_rate` for the child transaction. The attack window exists for any RBF replacement attempt on any pending transaction with spendable outputs.

### Recommendation

1. **Document the advisory nature**: Clearly document in the RPC spec that `min_replace_fee` is a point-in-time estimate and may be stale by the time the replacement is submitted.
2. **Add a tolerance parameter**: Allow callers to pass a `fee_bump` margin to `send_transaction` so the node accepts a replacement if the fee exceeds `min_replace_fee_at_query_time` by a configurable buffer.
3. **Bound descendant growth**: Enforce a maximum number of descendants per transaction (already partially done with `MAX_REPLACEMENT_CANDIDATES = 100`), but also consider limiting how many new descendants can be added to a transaction that is already a replacement candidate.
4. **Atomic query-and-replace**: Expose an RPC that atomically checks the current `min_replace_fee` and submits the replacement in a single write-locked operation, eliminating the TOCTOU window.

### Proof of Concept

1. Alice has a pending transaction `tx_A` in the pool with fee = 1000 shannons and one unspent output.
2. Bob queries `get_transaction(tx_A.hash)` and receives `min_replace_fee = 1363` (fee + extra_rbf_fee).
3. Bob constructs replacement `tx_B` with fee = 1363 shannons and signs it.
4. Mallory observes `tx_A` in the pool and submits child `tx_C` spending `tx_A`'s output with fee = 5000 shannons.
5. Bob submits `tx_B`. `check_rbf` now computes `min_replace_fee = 1000 (tx_A) + 5000 (tx_C) + 363 (extra) = 6363`.
6. Bob's replacement is rejected: `"Tx's current fee is 1363, expect it to >= 6363 to replace old txs"`.
7. Bob must re-query, reconstruct, and resubmit with fee ≥ 6363. Mallory can repeat step 4 indefinitely.

```rust
    /// min_replace_fee = sum(replaced_txs.fee) + extra_rbf_fee
    fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize) -> Option<Capacity> {
        let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
        // don't account for duplicate txs
        let replaced_fees: HashMap<_, _> = conflicts
            .iter()
            .map(|c| (c.id.clone(), c.inner.fee))
            .collect();
        let replaced_sum_fee = replaced_fees
            .values()
            .try_fold(Capacity::zero(), |acc, x| acc.safe_add(*x));
        let res = replaced_sum_fee.map_or(Err(CapacityError::Overflow), |sum| {
            sum.safe_add(extra_rbf_fee)
        });
        if let Ok(res) = res {
            Some(res)
        } else {
            let fees = conflicts.iter().map(|c| c.inner.fee).collect::<Vec<_>>();
            error!(
                "conflicts: {:?} replaced_sum_fee {:?} overflow by add {}",
                conflicts.iter().map(|e| e.id.clone()).collect::<Vec<_>>(),
                fees,
                extra_rbf_fee
            );
            None
        }
    }
```
