### Title
Unprivileged Relayer Can Grief Contract Upgrade by Front-Running with Higher-Sequence Governance VAA — (`target_chains/sui/contracts/sources/governance/governance.move`, `contract_upgrade.move`)

---

### Summary

Both `execute_governance_instruction` and `authorize_upgrade` enforce a strictly-greater-than check against a single shared `last_executed_governance_sequence` counter. Because `execute_governance_instruction` has no caller access control, any unprivileged relayer holding a valid higher-sequence governance VAA can advance that counter past a pending upgrade VAA's sequence, causing every subsequent `authorize_upgrade` call for that VAA to abort.

---

### Finding Description

`State` holds one shared counter for all governance actions: [1](#0-0) 

`execute_governance_instruction` checks and updates it with no caller restriction: [2](#0-1) 

`authorize_upgrade` performs the identical check against the same counter: [3](#0-2) 

`verify_vaa`, the only gate before either function, validates only the governance emitter chain/address — it imposes no restriction on who may call it: [4](#0-3) 

**Attack path:**

1. Pyth governance signs and broadcasts upgrade VAA at sequence **N** (publicly available on Wormhole).
2. Pyth governance also signs a subsequent governance VAA (e.g., `set_update_fee`) at sequence **N+k** (k ≥ 1) — also publicly available.
3. Attacker calls `verify_vaa` + `execute_governance_instruction` with the N+k VAA. `last_executed_governance_sequence` is now N+k.
4. Legitimate relayer calls `verify_vaa` + `authorize_upgrade` with the upgrade VAA (sequence N). The assert `N > N+k` is false → aborts with `E_CANNOT_EXECUTE_GOVERNANCE_ACTION_WITH_OBSOLETE_SEQUENCE_NUMBER`. [5](#0-4) 

The attacker cannot forge VAAs — they need a legitimately guardian-signed VAA with a higher sequence. However, Wormhole VAAs are public once signed, and governance routinely emits multiple sequential VAAs (e.g., upgrade + parameter change). Any such pair is sufficient.

---

### Impact Explanation

The upgrade VAA for sequence N is permanently unexecutable until governance re-emits the upgrade at a sequence > N+k. If the attacker holds a supply of higher-sequence VAAs (e.g., a batch of governance actions emitted alongside the upgrade), they can keep advancing the counter, delaying the upgrade indefinitely. In an emergency scenario — a critical price-manipulation bug requiring an urgent patch — this griefing directly prevents the fix from being deployed, leaving the price oracle vulnerable for the duration of the delay.

---

### Likelihood Explanation

- **No privilege required.** Any address can relay a Wormhole VAA.
- **VAAs are public.** Once the guardian network signs a governance VAA it is observable by anyone monitoring Wormhole.
- **Realistic precondition.** Governance frequently emits multiple VAAs in sequence (upgrade + config change). The attacker only needs one higher-sequence VAA to execute the attack.
- **Trivially repeatable.** If governance re-emits the upgrade at N+k+1, the attacker relays the next available higher-sequence VAA.

---

### Recommendation

Decouple the sequence counter used for contract upgrades from the one used for routine governance instructions, **or** enforce that `authorize_upgrade` is not blocked by non-upgrade governance actions executed out of band. A minimal fix is to maintain a separate `last_executed_upgrade_sequence` in `State` that is only updated by `authorize_upgrade`, so that non-upgrade governance actions cannot advance it.

Alternatively, require that governance VAAs be relayed in strict ascending order by enforcing the sequence check at the Wormhole VAA verification layer before the receipt is issued, preventing any caller from skipping sequences.

    public fun verify_vaa(
        pyth_state: &State,
        verified_vaa: VAA,
    ): WormholeVAAVerificationReceipt {
        state::assert_latest_only(pyth_state);

        let vaa_data_source = pyth::data_source::new((vaa::emitter_chain(&verified_vaa) as u64), vaa::emitter_address(&verified_vaa));

        // The emitter chain and address must correspond to the Pyth governance emitter chain and contract.
        assert!(
            pyth::state::is_valid_governance_data_source(pyth_state, vaa_data_source),
            E_INVALID_GOVERNANCE_DATA_SOURCE
        );

        let digest = vaa::digest(&verified_vaa);

        let sequence = vaa::sequence(&verified_vaa);

        let payload = vaa::take_payload(verified_vaa);

        WormholeVAAVerificationReceipt { payload, digest, sequence }
    }
