### Title
Unauthenticated `clear_tx_pool` RPC Exposes Privileged Pool-Wipe Operation to Any Local Caller — (File: `rpc/src/module/pool.rs`)

---

### Summary

The `clear_tx_pool` RPC method in `rpc/src/module/pool.rs` is registered as a standard public endpoint in the `Pool` module with no per-caller authentication or authorization check. Any unprivileged process that can reach the RPC socket — by default any local user on the node host — can invoke it to atomically remove every pending transaction from the pool. The rationale for exposing this destructive, operator-level operation to any caller is never explained in the code or documentation, directly mirroring the external report's class of unclear public function visibility.

---

### Finding Description

The `PoolRpc` trait declares `clear_tx_pool` as a plain public RPC method:

```rust
// rpc/src/module/pool.rs
#[rpc(name = "clear_tx_pool")]
fn clear_tx_pool(&self) -> Result<()>;
```

The `PoolRpcImpl` implementation performs no caller identity check, no token/credential validation, and no role guard before executing the pool wipe. The CKB RPC framework has no per-method access control layer; the only runtime gate is whether the `Pool` module is listed in `rpc.modules` in `ckb.toml`.

The default production configuration enables the `Pool` module and binds the RPC listener to `127.0.0.1:8114`:

```toml
# resource/ckb.toml
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
listen_address = "127.0.0.1:8114"
```

Any process running under any local user account on the node host can therefore call `clear_tx_pool` with a single unauthenticated HTTP request. If an operator follows common deployment patterns (reverse proxy, Docker port mapping, or misconfigured `listen_address`) the method becomes reachable from the network.

The same pattern applies to `remove_transaction` (targeted single-tx removal) and `clear_tx_verify_queue` (clears the pending verification pipeline), both of which are also public Pool-module methods with no authorization guard.

---

### Impact Explanation

An unprivileged local caller (or remote caller if the RPC is exposed) can:

1. **Atomically empty the entire mempool** — all pending user transactions are silently dropped; users must detect the loss and resubmit.
2. **Repeatedly wipe the pool** in a tight loop, preventing any transaction from ever reaching the proposed/committed stage, effectively halting transaction throughput on the node.
3. **Selectively evict transactions** via `remove_transaction` to censor specific senders or fee-paying transactions without leaving any on-chain trace.
4. **Stall the verification pipeline** via `clear_tx_verify_queue`, causing in-flight script-verification work to be discarded.

Impact category: **service unavailability / severe degradation** of the transaction-processing subsystem, reachable by an unprivileged local RPC caller — an attacker profile explicitly listed as in-scope.

---

### Likelihood Explanation

The `Pool` module is enabled by default. Any co-located process (another user account, a compromised service, a malicious script run by the node operator) can issue the call with no credentials. The attack requires only a single HTTP POST to localhost and is trivially scriptable. If the node is deployed behind a reverse proxy that forwards all RPC paths, the surface extends to any network-reachable client.

---

### Recommendation

Introduce a per-method privilege tier for state-mutating Pool operations (`clear_tx_pool`, `remove_transaction`, `clear_tx_verify_queue`). Options include:

- A configurable `rpc.admin_token` header that must be present for destructive methods.
- A separate `admin_listen_address` binding (e.g., a Unix socket) for operator-only methods, distinct from the general RPC port.
- Explicit documentation stating the intended caller and the security assumption, so operators can make an informed decision about exposure.

---

### Proof of Concept

```bash
# From any process on the node host — no credentials required
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"clear_tx_pool","params":[]}'
# Returns: {"jsonrpc":"2.0","result":null,"id":1}
# All pending transactions are now gone from the pool.

# Looped to prevent any transaction from ever accumulating:
while true; do
  curl -s -X POST http://127.0.0.1:8114 \
    -H 'Content-Type: application/json' \
    -d '{"jsonrpc":"2.0","id":1,"method":"clear_tx_pool","params":[]}' \
    > /dev/null
done
```
