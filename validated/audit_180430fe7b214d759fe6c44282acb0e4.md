### Title
Network Secret Key Briefly World-Readable Due to Post-Write Permission Setting Race in `write_secret_to_file` - (File: util/app-config/src/configs/network.rs)

### Summary
The `write_secret_to_file` function creates the node's secp256k1 P2P identity secret key file with default filesystem permissions (world-readable under typical umask), writes the raw 32-byte private key, and only then restricts permissions to `0o400`. Any local user on the same host can read the key during this window. This is a TOCTOU race in key storage, directly analogous to the reported "insecure key handling" class.

### Finding Description
In `util/app-config/src/configs/network.rs`, `write_secret_to_file` opens the file using `fs::OpenOptions::new().create(true).write(true).truncate(true).open(path)` with no mode specified at creation time. [1](#0-0) 

On Unix, `create(true)` creates the file with mode `0o666` minus the process umask. With the common umask of `0o022`, the file is created as `0o644` (world-readable). The raw 32-byte secp256k1 private key is then written to this world-readable file. Only after `file.write_all(secret)?` completes does the code call `file.set_permissions(fs::Permissions::from_mode(0o400))`. [2](#0-1) 

The key is stored as raw unencrypted bytes — no encryption at rest.

Contrast this with `create_tor_secret_key` in `util/onion/src/onion_service.rs`, which correctly uses `OpenOptionsExt::mode(0o600)` at file creation time, atomically preventing world-readability: [3](#0-2) 

The network secret key is used by `fetch_private_key` to establish the node's P2P identity (secio/libp2p secp256k1 keypair): [4](#0-3) 

### Impact Explanation
A local user on the same host who reads the secret key file during the race window obtains the node's raw secp256k1 P2P private key. With this key, the attacker can:
- Impersonate the node's P2P identity on the network
- Perform man-in-the-middle attacks on the node's P2P sessions (secio handshake uses this key for authenticated encryption)
- Disrupt the node's network connectivity by connecting to peers using the stolen identity, causing the legitimate node to be rejected as a duplicate peer ID

### Likelihood Explanation
The race window exists every time the node is started for the first time (or after `ckb reset-data --network-secret-key`). On a multi-user server, any local user can `inotifywait` on the network data directory and immediately read the file when it appears. The window is small (microseconds to milliseconds) but reliably exploitable with filesystem event monitoring. The attacker profile is a "supported local CLI/RPC user" — explicitly in scope.

### Recommendation
Set the file mode atomically at creation time using `OpenOptionsExt::mode(0o400)` (or `0o600` then `0o400` after write), exactly as `create_tor_secret_key` does:

```rust
#[cfg(unix)]
{
    use std::os::unix::fs::OpenOptionsExt;
    options = options.mode(0o400);
}
```

This eliminates the race window entirely. Additionally, consider encrypting the key at rest rather than storing raw bytes.

### Proof of Concept
1. On a multi-user Linux host, register an inotify watch on the CKB network data directory:
   ```bash
   inotifywait -m -e create /path/to/ckb/data/network/ &
   ```
2. When the node starts for the first time, `write_secret_to_file` creates `secret_key` with mode `0o644`.
3. Immediately read the file before `set_permissions` executes:
   ```bash
   # triggered by inotify CREATE event:
   cat /path/to/ckb/data/network/secret_key | xxd
   ```
4. The 32-byte raw secp256k1 private key is captured. The attacker can now construct a `SecioKeyPair` from it and impersonate the node on the P2P network.

```rust
pub fn write_secret_to_file(secret: &[u8], path: PathBuf) -> Result<(), Error> {
    fs::OpenOptions::new()
        .create(true)
        .write(true)
        .truncate(true)
        .open(path)
        .and_then(|mut file| {
            file.write_all(secret)?;
            #[cfg(unix)]
            {
                use std::os::unix::fs::PermissionsExt;
                file.set_permissions(fs::Permissions::from_mode(0o400))
            }
            #[cfg(not(unix))]
            {
                let mut permissions = file.metadata()?.permissions();
                permissions.set_readonly(true);
                file.set_permissions(permissions)
            }
        })
}
```
