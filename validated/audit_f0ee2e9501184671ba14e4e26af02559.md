### Title
Stale `inflight_states` Entries After Peer Eviction via `prune()` Causes Sync Stall — (`File: sync/src/types/mod.rs`)

---

### Summary

When `InflightBlocks::prune()` evicts a peer by removing it from `download_schedulers` (because `task_count == 0`), it only cleans `inflight_states` for blocks within `tip + 20`. Blocks beyond that window remain in `inflight_states` with a stale peer reference. The subsequent `remove_by_peer()` call during disconnect finds no entry in `download_schedulers` and returns without cleaning those orphaned entries. Because `InflightBlocks::insert()` rejects any block already present in `inflight_states`, no other peer can be assigned to download those blocks, causing a sync stall proportional to the number of stuck entries.

---

### Finding Description

`InflightBlocks` maintains two parallel data structures:

- `download_schedulers: HashMap<PeerIndex, DownloadScheduler>` — per-peer download quota and hash set
- `inflight_states: BTreeMap<BlockNumberAndHash, InflightState>` — per-block in-flight record keyed by `(number, hash)`

**Step 1 — `prune()` removes the peer from `download_schedulers` but leaves orphaned `inflight_states` entries.**

`prune()` iterates only over blocks within `tip + 20`:

```rust
let end = tip + 20;
for (key, value) in states.iter() {
    if key.number > end { break; }
    if value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now {
        ...
        remove_key.push(key.clone());
    }
}
```

After removing those timed-out blocks and punishing the peer, it evicts the peer entirely when `task_count` reaches zero:

```rust
download_schedulers.retain(|k, v| {
    if v.task_count == 0 {
        disconnect_list.insert(*k);
        false          // peer removed from download_schedulers
    } else { true }
});
```

Blocks the peer held beyond `tip + 20` are **not** removed from `inflight_states` at this point.

**Step 2 — `remove_by_peer()` silently no-ops because the peer is already gone.**

`SyncState::disconnected()` calls `remove_by_peer()`:

```rust
pub fn remove_by_peer(&mut self, peer: PeerIndex) -> usize {
    self.download_schedulers
        .remove(&peer)          // returns None — peer already evicted by prune()
        .map(|blocks| {
            for block in blocks.hashes {
                state.remove(&block);   // never reached
                ...
            }
            blocks_count
        })
        .unwrap_or_default()    // returns 0
}
```

The orphaned `inflight_states` entries survive with a stale `peer` field pointing to the now-disconnected peer.

**Step 3 — `insert()` blocks any other peer from requesting those blocks.**

```rust
pub fn insert(&mut self, peer: PeerIndex, block: BlockNumberAndHash) -> bool {
    let state = self.inflight_states.entry(block.clone());
    match state {
        Entry::Occupied(_entry) => return false,   // block already "in-flight"
        Entry::Vacant(entry) => entry.insert(InflightState::new(peer)),
    };
    ...
}
```

`block_fetcher` skips any block for which `insert()` returns `false`:

```rust
} else if ... && state.write_inflight_blocks().insert(self.peer, (header.number(), hash).into()) {
    fetch.push(header)
}
```

Those blocks are therefore never fetched from any other peer until `unverified_tip` advances past `block.number - 20`, allowing a future `prune()` call to finally clean them.

---

### Impact Explanation

An unprivileged sync peer can cause the victim node to stall block download for up to `BLOCK_DOWNLOAD_WINDOW = 8192` blocks. Because `unverified_tip` cannot advance past the stuck range (those blocks cannot be requested from honest peers), the stall is self-reinforcing: `prune()` never reaches the stuck entries, and the node is unable to make sync progress for the affected range. This degrades or halts IBD and post-IBD catch-up for the duration of the attack.

---

### Likelihood Explanation

Any peer that can establish a sync connection can trigger this. The attacker:
1. Connects to the victim.
2. Announces a `best_known_header` at `victim_unverified_tip + BLOCK_DOWNLOAD_WINDOW` (valid headers obtainable from the public chain).
3. Ignores all `GetBlocks` requests.
4. After three `BLOCK_DOWNLOAD_TIMEOUT` (30 s) cycles (~90 s), `task_count` reaches 0, the peer is evicted, and up to 8 172 blocks (`8192 − 20`) are orphaned in `inflight_states`.
5. The attacker may reconnect and repeat.

No special privilege, key material, or majority hash power is required.

---

### Recommendation

`remove_by_peer()` must not rely solely on `download_schedulers` to discover which blocks to clean. Two complementary fixes:

1. **Scan `inflight_states` by peer on disconnect.** When `download_schedulers.remove(&peer)` returns `None`, fall back to a linear scan of `inflight_states` to remove all entries whose `InflightState::peer == peer`.

2. **Clean all peer blocks in `prune()` before eviction.** Before calling `download_schedulers.retain(...)`, iterate `inflight_states` and remove every entry belonging to a peer whose `task_count` is about to reach zero, regardless of block number.

---

### Proof of Concept

4. Attacker sends no Block responses.
5. After 30 s: prune(T) fires.

    pub fn prune(&mut self, tip: BlockNumber) -> HashSet<PeerIndex> {
        let now = unix_time_as_millis();
        let mut disconnect_list = HashSet::new();
        // Since statistics are currently disturbed by the processing block time, when the number
        // of transactions increases, the node will be accidentally evicted.
        //
        // Especially on machines with poor CPU performance, the node connection will be frequently
        // disconnected due to statistics.
        //
        // In order to protect the decentralization of the network and ensure the survival of low-performance
        // nodes, the penalty mechanism will be closed when the number of download nodes is less than the number of protected nodes
        let should_punish = self.download_schedulers.len() > self.protect_num;
        let adjustment = self.adjustment;

        let trace = &mut self.trace_number;
        let download_schedulers = &mut self.download_schedulers;
        let states = &mut self.inflight_states;

        let mut remove_key = Vec::new();
        // Since this is a btreemap, with the data already sorted,
        // we don't have to worry about missing points, and we don't need to
        // iterate through all the data each time, just check within tip + 20,
        // with the checkpoint marking possible blocking points, it's enough
        let end = tip + 20;
        for (key, value) in states.iter() {
            if key.number > end {
                break;
            }
            if value.timestamp + BLOCK_DOWNLOAD_TIMEOUT < now {
                if let Some(set) = download_schedulers.get_mut(&value.peer) {
                    set.hashes.remove(key);
                    if should_punish && adjustment {
                        set.punish(2);
                    }
                };
                if !trace.is_empty() {
                    trace.remove(key);
                }
                remove_key.push(key.clone());
                debug!(
                    "prune: remove InflightState: remove {}-{} from {}",
                    key.number, key.hash, value.peer
                );

                if let Some(metrics) = ckb_metrics::handle() {
                    metrics.ckb_inflight_timeout_count.inc();
                }
            }
        }

        for key in remove_key {
            states.remove(&key);
        }

        download_schedulers.retain(|k, v| {
            // task number zero means this peer's response is very slow
            if v.task_count == 0 {
                disconnect_list.insert(*k);
                false
            } else {
                true
            }
        });
        shrink_to_fit!(download_schedulers, SHRINK_THRESHOLD);

    pub fn insert(&mut self, peer: PeerIndex, block: BlockNumberAndHash) -> bool {
        let state = self.inflight_states.entry(block.clone());
        match state {
            Entry::Occupied(_entry) => return false,
            Entry::Vacant(entry) => entry.insert(InflightState::new(peer)),
        };

        if self.restart_number >= block.number {
            // All new requests smaller than restart_number mean that they are cleaned up and
            // cannot be immediately marked as cleaned up again.
            self.trace_number
                .insert(block.clone(), unix_time_as_millis());
        }

        let download_scheduler = self.download_schedulers.entry(peer).or_default();
        download_scheduler.hashes.insert(block)
    }

    pub fn remove_by_peer(&mut self, peer: PeerIndex) -> usize {
