### Title
Missing Receiver Interface Validation at Request Time Causes Permanently Stuck Entropy Requests - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `requestWithCallback` / `requestV2` functions in `Entropy.sol` accept any `msg.sender` as the requester without verifying at request time that the caller implements `IEntropyConsumer`. This is the direct analog of using `_mint` instead of `_safeMint`: the protocol defers the compatibility check to fulfillment time, where a revert in the legacy path (provider with `gasLimit10k == 0`) permanently locks the request and the paid fee, because `clearRequest` is called before the unguarded callback and the revert rolls back the storage deletion.

### Finding Description

`requestWithCallback` and `requestV2` store `msg.sender` as `req.requester` with no interface check:

```solidity
// Entropy.sol requestHelper (line 260)
req.requester = msg.sender;
```

At fulfillment time, `revealWithCallback` takes two distinct code paths.

**Legacy path** (`req.gasLimit10k == 0`):

```solidity
address callAddress = req.requester;
clearRequest(provider, sequenceNumber);   // storage cleared first
// ...
uint len;
assembly { len := extcodesize(callAddress) }
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(   // no try/catch
        sequenceNumber, provider, randomNumber
    );
}
```

`clearRequest` is called **before** the callback. If the requester contract does not implement `IEntropyConsumer._entropyCallback`, the bare external call reverts, the EVM unwinds the entire transaction (including the `clearRequest` storage write), and the request is restored to active state. Every subsequent `revealWithCallback` call will identically revert. The request is permanently unresolvable.

**New path** (`req.gasLimit10k != 0`):

```solidity
(success, ret) = req.requester.excessivelySafeCall(
    uint256(req.gasLimit10k) * TEN_THOUSAND,
    256,
    abi.encodeWithSelector(IEntropyConsumer._entropyCallback.selector, ...)
);
```

The revert is caught, the request transitions to `CALLBACK_FAILED`, and every retry also fails. The request is permanently stuck in `CALLBACK_FAILED`.

In both paths the fee (split between provider and Pyth) is locked in the contract and the provider's sequence-number slot is consumed.

`IEntropyConsumer._entropyCallback` itself does validate the caller:

```solidity
// IEntropyConsumer.sol line 14-15
require(entropy != address(0), "Entropy address not set");
require(msg.sender == entropy, "Only Entropy can call this function");
```

But this guard is inside the consumer contract that is never reached when the contract does not implement the interface at all.

### Impact Explanation

A contract that calls `requestWithCallback` or `requestV2` without inheriting `IEntropyConsumer` will have its request permanently stuck. The fee paid is irrecoverable. In the legacy path the provider's sequence-number slot is permanently consumed. If many such requests accumulate they degrade provider capacity. The impact is directly analogous to the Arcade `_mint` bug: assets (fees + randomness delivery) become irretrievable because the recipient contract cannot handle the callback.

### Likelihood Explanation

The `IEntropyV2` NatSpec explicitly states "The address calling this function should be a contract that inherits from the IEntropyConsumer interface," but this is advisory only — no on-chain enforcement exists. A developer integrating Entropy who forgets to inherit `IEntropyConsumer`, or who calls `requestV2` from a helper contract that proxies the call (making `msg.sender` the helper rather than the consumer), will silently create a permanently stuck request. This is a realistic integration mistake.

### Recommendation

At request time, verify that `msg.sender` can receive the callback, mirroring what `_safeMint` does for ERC-721. Two options:

1. **Interface check via ERC-165**: require `msg.sender` to declare support for `IEntropyConsumer` (add `supportsInterface` to the interface).
2. **Dry-run call**: perform a low-level `staticcall` to `_entropyCallback.selector` on `msg.sender` and revert if the selector is absent, before storing the request.

Additionally, expose a `cancelRequest` function so that stuck requests in `CALLBACK_FAILED` state can be cleared and fees partially refunded.

### Proof of Concept

1. Deploy a contract **without** inheriting `IEntropyConsumer`:
   ```solidity
   contract NoCallback {
       function request(address entropy, address provider) external payable {
           IEntropy(entropy).requestWithCallback{value: msg.value}(
               provider, bytes32(uint256(42))
           );
       }
   }
   ```
2. Call `NoCallback.request{value: fee}(entropyAddr, providerAddr)`.
3. Provider calls `revealWithCallback(provider, sequenceNumber, userContrib, providerContrib)`.
4. **Legacy path**: transaction reverts; request remains active. Every subsequent `revealWithCallback` call also reverts. Fee is locked.
5. **New path** (provider has `defaultGasLimit > 0`): `excessivelySafeCall` catches the revert; request moves to `CALLBACK_FAILED`. Every retry also fails. Fee is locked. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
