### Title
Hardcoded `DEFAULT_BYTES_PER_CYCLES` Diverges from Actual Consensus `max_block_cycles`, Causing Incorrect Transaction Weight and Fee Rate Calculations - (File: `util/types/src/core/tx_pool.rs`)

### Summary
`get_transaction_weight` uses a hardcoded constant `DEFAULT_BYTES_PER_CYCLES` derived from the compile-time default `MAX_BLOCK_CYCLES = 3_500_000_000`, but chain specs (dev, staging, and likely mainnet) configure `max_block_cycles = 10_000_000_000` at runtime. This ~2.86× discrepancy causes cycle-heavy transactions to have their weight systematically overestimated, making their fee rate appear ~2.86× lower than it actually is. As a result, legitimate transactions can be incorrectly rejected from the tx-pool, and RBF replacement decisions are computed on wrong data.

### Finding Description

`DEFAULT_BYTES_PER_CYCLES` is defined as a compile-time constant: [1](#0-0) 

Its value `0.000_170_571_4` is derived from the compile-time constants: [2](#0-1) 

- `MAX_BLOCK_BYTES = 597 * 1_000 = 597_000`
- `MAX_BLOCK_CYCLES = 3_500_000 * 1_000 = 3_500_000_000`
- Ratio = `597_000 / 3_500_000_000 ≈ 0.000_170_571_4`

However, the `Consensus` struct has a runtime-configurable `max_block_cycles` field: [3](#0-2) 

Chain spec files override this to `10_000_000_000`: [4](#0-3) [5](#0-4) 

The `build_consensus` function correctly wires the spec value into the `Consensus` struct: [6](#0-5) 

The block assembler and consensus verifier correctly use `consensus.max_block_cycles()` at runtime: [7](#0-6) 

But `get_transaction_weight` — used for all tx-pool fee rate calculations — ignores the runtime consensus value and always uses the hardcoded constant: [8](#0-7) 

This function is called in three critical places:

1. **Tx-pool entry fee rate and ordering** (`tx-pool/src/component/entry.rs`): [9](#0-8) 

2. **Fee rate statistics RPC** (`rpc/src/util/fee_rate.rs`): [10](#0-9) 

3. **Fee estimator** (`util/fee-estimator/src/estimator/weight_units_flow.rs`): [11](#0-10) 

### Impact Explanation

With `max_block_cycles = 10_000_000_000` (actual spec), the correct bytes-per-cycle ratio is:
`597_000 / 10_000_000_000 = 0.0000597`

But `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` is **2.857× larger**.

For a cycle-heavy transaction (e.g., cycles = 1,000,000, size = 100 bytes):
- **Correct weight**: `max(100, 1_000_000 × 0.0000597)` = `max(100, 59.7)` = **100 bytes**
- **Computed weight**: `max(100, 1_000_000 × 0.000_170_571_4)` = `max(100, 170.57)` = **170 bytes**

The computed weight is 70% larger than the actual size. If the transaction pays a fee of 100 shannons:
- **Actual fee rate**: `100 × 1000 / 100` = 1000 shannons/KB (meets `min_fee_rate`)
- **Computed fee rate**: `100 × 1000 / 170` ≈ 588 shannons/KB (below `min_fee_rate = 1000`)

The transaction is **incorrectly rejected** even though it pays a sufficient fee. Additionally, RBF replacement checks use the same incorrect weight, so a cycle-heavy replacement transaction may be incorrectly rejected as not paying a sufficient fee bump.

### Likelihood Explanation

Any CKB node running with a chain spec that sets `max_block_cycles` to a value other than the compile-time default `3_500_000_000` is affected. Both the dev and staging specs set `max_block_cycles = 10_000_000_000`. Any transaction sender submitting a cycle-heavy transaction to such a node triggers the incorrect calculation. No special privileges are required — this is reachable via the standard `send_transaction` RPC.

### Recommendation

`get_transaction_weight` should accept the actual `max_block_bytes` and `max_block_cycles` from the runtime `Consensus` object rather than using the hardcoded `DEFAULT_BYTES_PER_CYCLES`. The ratio should be computed as `consensus.max_block_bytes() as f64 / consensus.max_block_cycles() as f64` and passed into the function, or the function signature should be extended to accept these parameters. All call sites in the tx-pool, fee rate collector, and fee estimator should be updated to supply the runtime consensus values.

### Proof of Concept

1. Start a CKB node with `resource/specs/dev.toml` (sets `max_block_cycles = 10_000_000_000`).
2. Construct a transaction with a script that consumes ~1,000,000 cycles and has a serialized size of ~100 bytes.
3. Set the transaction fee to exactly `min_fee_rate × tx_size = 1000 × 100 / 1000 = 100 shannons` (sufficient by actual size).
4. Submit via `send_transaction` RPC.
5. The tx-pool computes weight = `max(100, 1_000_000 × 0.000_170_571_4)` = 170 bytes, yielding an apparent fee rate of ~588 shannons/KB.
6. The tx-pool rejects the transaction with `LowFeeRate` even though the fee is correct relative to the actual block resource consumption.

The root cause is the mismatch between the hardcoded `DEFAULT_BYTES_PER_CYCLES` at `util/types/src/core/tx_pool.rs:279` and the runtime `max_block_cycles` configured in the chain spec at `resource/specs/dev.toml:89`.
