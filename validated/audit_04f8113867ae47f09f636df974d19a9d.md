Audit Report

## Title
Insert-Before-Check O(n) Scan Under Global Mutex Enables Monotonically Growing Transaction Relay Pipeline Stall — (`sync/src/types/mod.rs`)

## Summary

`add_ask_for_txs` in `sync/src/types/mod.rs` unconditionally inserts incoming transaction hashes into the global `unknown_tx_hashes` queue before evaluating the size limit. Once the queue exceeds `MAX_UNKNOWN_TX_HASHES_SIZE` (50,000), every subsequent call performs an O(total queue depth) linear scan under the held mutex. Because the non-offending path returns `Status::ignored()` without evicting the just-inserted hashes, the queue grows monotonically, compounding the scan cost with each call. While the mutex is held, `pop_ask_for_txs` — which dispatches `GetTransactions` to all peers — is blocked, stalling the entire relay transaction-fetching pipeline.

## Finding Description

**Entry point:** `TransactionHashesProcess::execute` (`sync/src/relayer/transaction_hashes_process.rs`, L25–49) calls `state.add_ask_for_txs(self.peer, tx_hashes)` after filtering already-known hashes via `tx_filter`.

**Mutex acquisition and hold:** `add_ask_for_txs` (`sync/src/types/mod.rs`, L1483–1484) immediately acquires `self.unknown_tx_hashes.lock()` and holds it for the entire function body, including the scan.

**Insert-before-check:** Lines 1486–1504 unconditionally insert up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) hashes into the queue via `unknown_tx_hashes.entry(tx_hash)`. The size guard at lines 1507–1509 fires only **after** insertion, meaning hashes are always committed to the queue regardless of the outcome.

**O(n) scan:** When `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE` (50,000), lines 1516–1523 iterate over every `(_hash, priority)` pair in the queue to count entries belonging to `peer_index`. This is O(total queue depth) under the held mutex.

**No eviction on non-offending path:** If `peer_unknown_counter < MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`, line 1528 returns `Status::ignored()` without removing the just-inserted hashes. On the offending path (line 1525, `TooManyUnknownTransactions`), the hashes are also already inserted and not removed. The queue therefore grows monotonically with every call from any peer once the threshold is crossed.

**`pop_ask_for_txs` blocked:** `pop_ask_for_txs` (`sync/src/types/mod.rs`, L1453–1454) acquires the same `unknown_tx_hashes` mutex. While `add_ask_for_txs` holds the lock performing the O(n) scan, `pop_ask_for_txs` is blocked, stalling all outbound `GetTransactions` dispatches for all peers simultaneously.

**Existing guards insufficient:** The `MAX_RELAY_TXS_NUM_PER_BATCH` check in `execute` (L29–35) limits a single message to 32,767 hashes but does not prevent queue saturation across multiple peers or calls. The size guard in `add_ask_for_txs` fires post-insertion and does not evict, making it ineffective as a bound.

## Impact Explanation

Stalling `pop_ask_for_txs` under a global mutex blocks the relay transaction-fetching pipeline for all connected peers simultaneously. Because the queue grows monotonically after saturation (inserted hashes are never evicted on either return path), each subsequent call from any peer increases the scan cost, compounding the stall duration. An attacker sustaining the attack causes progressively longer mutex holds, degrading transaction relay throughput across the entire node. This matches the **High** impact class (10,001–15,000 points): "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation

The attack requires only unprivileged P2P peers capable of sending `RelayTransactionHashes` messages with unique transaction hashes. No proof-of-work, no privileged role, no leaked key, and no victim mistake is required. Two peers suffice to saturate the queue past the 50,000-entry threshold (2 × 32,767 = 65,534 unique hashes). Thereafter, any additional peer sending unique hashes triggers the O(n) scan and leaves its hashes in the queue, growing the scan cost indefinitely. The attack is repeatable, low-cost, and requires no coordination beyond controlling a small number of P2P connections.

## Recommendation

1. **Check before inserting:** Move the per-peer and global size guards to the top of `add_ask_for_txs`, before the insertion loop, so hashes from an over-quota peer are never added to the queue.
2. **Eliminate the O(n) scan:** Maintain a `HashMap<PeerIndex, usize>` per-peer counter updated incrementally during insertion, making the per-peer check O(1) and eliminating the need for the full queue iteration.
3. **Evict on overflow:** When the global limit is reached, evict the lowest-priority entries (e.g., oldest `request_time`) rather than silently accepting new hashes and returning `ignored`.
