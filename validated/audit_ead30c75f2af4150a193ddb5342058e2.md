### Title
`RecentReject::put()` Never Updates `total_keys_num`, Breaking the Shrink Bound — (`File: tx-pool/src/component/recent_reject.rs`)

---

### Summary

In `RecentReject::put()`, the in-memory counter `self.total_keys_num` is **never incremented**. The `checked_add(1)` call produces a local shadow variable used only for comparison; the result is never written back to `self`. As a result, the shrink mechanism that enforces `keep_rejected_tx_hashes_count` is permanently disabled after startup, allowing the `recent_reject` RocksDB to grow without bound. An unprivileged RPC caller can exploit this by submitting many distinct rejected transactions.

---

### Finding Description

`RecentReject` maintains a sharded RocksDB (`DBWithTTL`) that records recently-rejected transaction hashes. The field `total_keys_num` is intended to track the approximate number of stored keys so that `shrink()` (which drops a random shard) is triggered when the count exceeds `count_limit` (`keep_rejected_tx_hashes_count`).

In `put()`:

```rust
pub fn put(&mut self, hash: &Byte32, reject: Reject) -> Result<(), AnyError> {
    ...
    self.db.put(&shard, hash_slice, json_string)?;   // overwrites or inserts

    if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {  // local shadow!
        if total_keys_num > self.count_limit {
            self.shrink()?;
        }
    } else {
        self.shrink()?;
    }
    Ok(())
}
```

`self.total_keys_num.checked_add(1)` returns `Some(self.total_keys_num + 1)` into the **local** binding `total_keys_num`. `self.total_keys_num` is **never assigned** the new value. It is only mutated inside `shrink()`, which re-reads the DB estimate. Because the comparison always uses the same stale startup value, `shrink()` is never triggered (when the node starts with an empty or under-limit DB), and the DB grows without bound.

A secondary dimension of the same class: even if the counter were correctly incremented, `self.db.put()` silently **overwrites** an existing key without checking whether it is new. The counter should only be incremented for genuinely new keys, not for overwrites of existing rejection records — exactly the analog of the borgCore `paramOffsets` bug where a tracking array was appended to even when updating an existing entry.

The existing unit test `test_basic` passes trivially for the wrong reason: `total_keys_num` stays at `0` throughout (never incremented), so `assert!(recent_reject.total_keys_num < 100)` always holds, masking the broken shrink invariant.

---

### Impact Explanation

- The `recent_reject` DB is not bounded by `keep_rejected_tx_hashes_count`. Over time (bounded only by the TTL), it accumulates all rejected transaction records without eviction.
- Disk space is consumed unboundedly relative to the configured limit. On a node with a small disk or a high rejection rate, this can exhaust available storage, causing RocksDB write failures and node unavailability.
- `get_estimate_total_keys_num()` (exposed via the terminal RPC `total_recent_reject_num`) always returns the stale startup value, giving operators a permanently incorrect view of the rejection store size.
- If `total_keys_num` starts above `count_limit` at startup (e.g., after a crash with a large DB), every single `put()` call triggers `shrink()`, causing excessive random shard drops and loss of valid rejection records on every rejected transaction.

---

### Likelihood Explanation

The entry path is fully attacker-controlled and requires no privilege beyond RPC access (the default local RPC endpoint). An attacker submits many distinct transactions that are rejected (e.g., transactions with fees below `min_fee_rate`, or malformed transactions). Each rejection calls `put_recent_reject()` → `recent_reject.put()`. The counter never advances, so the shrink guard never fires. The attack is low-cost: crafting fee-insufficient transactions requires no CKB balance.

---

### Recommendation

1. **Persist the incremented value**: After `self.db.put()`, assign the updated count back to `self.total_keys_num`:
   ```rust
   self.total_keys_num = self.total_keys_num.saturating_add(1);
   if self.total_keys_num > self.count_limit {
       self.shrink()?;
   }
   ```
2. **Only increment for new keys**: Check whether the key already exists before incrementing, so that overwrites of existing rejection records do not inflate the counter (the direct analog of the borgCore fix):
   ```rust
   let is_new = self.db.get_pinned(&shard, hash_slice)?.is_none();
   self.db.put(&shard, hash_slice, json_string)?;
   if is_new {
       self.total_keys_num = self.total_keys_num.saturating_add(1);
   }
   if self.total_keys_num > self.count_limit {
       self.shrink()?;
   }
   ```
3. **Fix the unit test** to assert that shrink was actually triggered (e.g., verify that some keys were evicted after inserting beyond `count_limit` unique hashes).

### Proof of Concept

```
1. Configure node with keep_rejected_tx_hashes_count = 1000, keep_rejected_tx_hashes_days = 7.
2. Start with an empty recent_reject DB → total_keys_num = 0 at startup.
3. Submit 10,000 distinct transactions via send_transaction RPC, each with fee = 0
   (rejected as LowFeeRate, which satisfies should_recorded()).
4. Observe: total_keys_num reported by terminal RPC remains 0 throughout.
5. Observe: recent_reject DB on disk grows to hold all 10,000 records,
   far exceeding the configured limit of 1,000.
6. shrink() is never called; no shard is ever dropped.
```
