### Title
Missing Maximum Fee Guard in Entropy `requestV2` / `requestWithCallback` Enables Provider Front-Running Fee Extraction — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract's `requestHelper` function does not refund excess `msg.value` and provides no `maxFeeAccepted` parameter. Because any registered provider can call `setProviderFee()` at any time, a malicious provider can front-run a user's `requestV2()` / `requestWithCallback()` transaction to increase their fee, causing the user to either overpay (with the excess permanently captured by the protocol) or have their transaction reverted.

---

### Finding Description

In `Entropy.sol`, `requestHelper` enforces a minimum fee but silently absorbs all excess ETH:

```solidity
// Entropy.sol lines 234–239
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

The NatSpec on every `requestV2` variant explicitly states:

> "Note that excess value is *not* refunded to the caller."

Any registered provider can change their fee at any time via `setProviderFee()`:

```solidity
// Entropy.sol lines 810–820
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(...);
}
```

There is no `maxFeeAccepted` parameter on any `requestV2` / `requestWithCallback` variant. The attack path is:

1. User (or user's contract) calls `getFeeV2(provider)` off-chain → observes fee `F`.
2. User submits `requestV2{value: F + buffer}(...)` (sending a buffer is a common defensive pattern to avoid reverts from minor fee changes).
3. Malicious provider front-runs with `setProviderFee(F + buffer - pythFee)` to consume the entire buffer.
4. User's transaction executes: `requiredFee` now equals `msg.value`, so it passes. The provider captures the full buffer as `providerFee`; the user receives no refund.
5. Provider back-runs with `setProviderFee(F)` to restore the original fee.

Even without a buffer, if the user sends exactly `F` and the provider front-runs to `F+1`, the user's transaction reverts — a pure DoS/griefing with no cost to the provider beyond gas.

---

### Impact Explanation

**Financial loss:** When a user sends excess ETH (a common and recommended defensive pattern), a malicious provider can extract that excess as additional fee revenue. The ETH is permanently lost to the user — it is split between `providerInfo.accruedFeesInWei` and `_state.accruedPythFeesInWei` with no refund path.

**DoS:** When a user sends the exact fee, a provider can front-run to increase the fee by 1 wei, causing the user's transaction to revert. The provider can repeat this indefinitely to block a specific user from obtaining randomness.

**Impact: Medium** — direct ETH loss for users who send buffers; service denial for users who send exact fees.

---

### Likelihood Explanation

**Likelihood: Medium.**

- Any address can register as an Entropy provider (permissionless).
- `setProviderFee()` has no timelock, no delay, and no cap.
- Mempool monitoring to detect pending `requestV2` calls is trivial.
- The NatSpec itself warns that "the fee can change over time," implicitly acknowledging this race condition, but provides no on-chain protection.
- User contracts that defensively send `getFeeV2() * 2` (or any buffer) are directly exploitable.

---

### Recommendation

1. **Add a `maxFeeAccepted` parameter** to `requestV2` / `requestWithCallback`. Revert if `requiredFee > maxFeeAccepted`. This is the direct analog of `minAmountOut` in vault deposits.

2. **Alternatively, refund excess ETH** to `msg.sender` after deducting `requiredFee`. This eliminates the economic incentive for fee front-running entirely.

3. **Consider a fee-change timelock** on `setProviderFee()` so users can rely on a stable fee for at least one block.

---

### Proof of Concept

```
Setup:
  - pythFeeInWei = 100 wei
  - provider.feeInWei = 900 wei  →  getFeeV2() = 1000 wei

Step 1: User submits tx:
  requestV2{value: 1500}(provider, userRandom, 0)
  (500 wei buffer for safety)

Step 2: Provider front-runs (same block, higher gas):
  setProviderFee(1400)  →  getFeeV2() = 1500 wei

Step 3: User tx executes:
  requiredFee = 1500 == msg.value  →  no revert
  providerFee = 1400  →  providerInfo.accruedFeesInWei += 1400
  accruedPythFeesInWei += 100
  User receives: 0 refund (lost 500 wei buffer)

Step 4: Provider back-runs:
  setProviderFee(900)  →  fee restored

Net result: Provider extracted 500 extra wei from user with no risk.

        address provider,
        bytes32 userContribution,
        uint32 gasLimit

            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed

            true,
            gasLimit
        );

            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );

            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
