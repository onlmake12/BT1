### Title
Signed Integer Overflow in Deviation Check Permanently Blocks Price Updates — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function computes the price deviation numerator as `int64 - int64` arithmetic. In Solidity 0.8+, this reverts on overflow. When a stored price and an incoming price span the `int64` sign boundary (e.g., one is near `type(int64).max` and the other is near `type(int64).min`), the subtraction overflows and the entire `updatePriceFeeds` call reverts. Any subscription relying solely on `updateOnDeviation` is permanently unable to receive updates.

---

### Finding Description

Inside `_validateShouldUpdatePrices`, the deviation check reads:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int64 previousPrice = previousFeed.price.price;

if (previousPrice == 0 || currentPrice == 0) { continue; }

uint256 numerator = SignedMath.abs(
    currentPrice - previousPrice   // ← int64 subtraction, reverts on overflow
);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);

if (deviationBps >= params.updateCriteria.deviationThresholdBps) {
    return updateTimestamp;
}
```

`SignedMath.abs` accepts `int256`, so the implicit widening cast happens **after** the subtraction. The subtraction itself is evaluated as `int64` arithmetic. Under Solidity 0.8 checked arithmetic, any result outside `[type(int64).min, type(int64).max]` reverts unconditionally.

Concrete overflow trigger:
- `previousPrice = type(int64).max` (9 223 372 036 854 775 807)
- `currentPrice  = -1`
- `currentPrice - previousPrice = -1 - 9 223 372 036 854 775 807` → underflows `int64` → **revert**

Or symmetrically:
- `previousPrice = type(int64).min` (-9 223 372 036 854 775 808)
- `currentPrice  = 1`
- `currentPrice - previousPrice = 1 + 9 223 372 036 854 775 808` → overflows `int64` → **revert**

The zero-guard (`if (previousPrice == 0 || currentPrice == 0) continue`) does not protect against this case. [1](#0-0) 

---

### Impact Explanation

A subscription configured with `updateOnDeviation: true` and `updateOnHeartbeat: false` (deviation-only mode) becomes permanently unserviceable once the stored price and the incoming price span the `int64` sign boundary. Every call to `updatePriceFeeds` for that subscription reverts, regardless of how much time has passed or how large the real price move is. The subscription's balance drains toward zero as keepers repeatedly attempt and fail, and consumers of the subscription receive stale prices indefinitely.

Even when `updateOnHeartbeat` is also enabled, the heartbeat path returns early only when the heartbeat interval has elapsed. Between heartbeat windows, the deviation path is the only trigger, and it is broken for the affected price range.

---

### Likelihood Explanation

Pyth price feeds are `int64` with a per-feed exponent (typically −8). The maximum representable price at exponent −8 is ≈ $92 billion. A transition from a large positive price to any negative price (or vice versa) — possible for synthetic, inverse, or derivative feeds — can produce a difference exceeding `type(int64).max`. The Pyth protocol does not restrict `price` to positive values; the `PythStructs.Price.price` field is declared `int64` and negative prices are explicitly supported for certain product types. A keeper or subscription owner cannot prevent this revert once the stored price and the live price straddle the sign boundary. [2](#0-1) 

---

### Recommendation

Widen the subtraction to `int256` before computing the absolute value, eliminating the overflow:

```solidity
uint256 numerator = SignedMath.abs(
    int256(currentPrice) - int256(previousPrice)   // safe: int256 cannot overflow here
);
```

`int256` can represent any difference of two `int64` values without overflow (the maximum difference is `2 × 2^63 = 2^64`, well within `int256` range).

### Proof of Concept

```solidity
// Subscription: updateOnDeviation = true, updateOnHeartbeat = false
// Stored previousPrice = type(int64).max  (from a prior updatePriceFeeds call)
// Incoming currentPrice = -1              (valid Pyth price for a synthetic feed)

int64 currentPrice  = -1;
int64 previousPrice = type(int64).max;   // 9_223_372_036_854_775_807

// Solidity 0.8 checked arithmetic:
// currentPrice - previousPrice
//   = -1 - 9_223_372_036_854_775_807
//   = -9_223_372_036_854_775_808 - 1   ← underflows int64
// → transaction reverts with "Arithmetic over/underflow"
// → updatePriceFeeds reverts
// → subscription is permanently stuck
``` [3](#0-2)
