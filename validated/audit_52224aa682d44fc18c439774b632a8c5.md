### Title
Permanently Frozen Update Fees — No On-Chain Withdrawal Path from `PriceInfoObject` - (`target_chains/sui/contracts/sources/price_info.move`)

### Summary
Every call to `update_single_price_feed` deposits `Coin<SUI>` into a `PriceInfoObject` via `deposit_fee_coins`. No function anywhere in the contract surface removes those coins. The `fee_recipient_address` stored in `State` is never used to transfer funds. All accumulated update fees are permanently frozen.

### Finding Description
`pyth::pyth::update_single_price_feed` collects a `Coin<SUI>` fee from the caller and stores it inside the target `PriceInfoObject` as a dynamic object field keyed by `FEE_STORAGE_KEY`. The module `pyth::price_info` exposes `deposit_fee_coins` (public) and `get_balance` (public read), but provides no counterpart to extract coins. Neither `pyth::pyth`, `pyth::state`, nor any governance module contains a function that borrows-mut the `FEE_STORAGE_KEY` field to move coins out. The governance module `set_fee_recipient` itself carries a comment acknowledging the fee-recipient mechanism "is not used anymore." The governance action enum has no `WITHDRAW_FEE` variant. There is no upgrade path that retroactively adds a withdrawal function without a new package deployment.

### Impact Explanation
100% of all SUI paid as update fees since contract deployment is permanently locked inside shared `PriceInfoObject` objects. The designated `fee_recipient_address` can never receive those funds on-chain. Protocol revenue is entirely unrecoverable under the current package.

### Likelihood Explanation
This is not probabilistic — it is a structural certainty. Every price update call deposits fees; no call can withdraw them. The comment in `set_fee_recipient.move` confirms the withdrawal path was intentionally removed without a replacement.

### Recommendation
Add a privileged (governance-gated) `withdraw_fees` function to `pyth::price_info` that removes the `FEE_STORAGE_KEY` dynamic field and transfers the `Coin<SUI>` to `fee_recipient_address`. Gate it behind `LatestOnly` and a governance VAA, mirroring the Ethereum `WithdrawFee` governance action. Alternatively, redirect fees at deposit time via `sui::transfer::public_transfer` directly to `fee_recipient_address` instead of storing them in the object.

        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            return 0
        };
        let fee = dynamic_object_field::borrow<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY);
        coin::value(fee)
    }

    public fun deposit_fee_coins(price_info_object: &mut PriceInfoObject, fee_coins: Coin<SUI>) {
        if (!dynamic_object_field::exists_with_type<vector<u8>, Coin<SUI>>(&price_info_object.id, FEE_STORAGE_KEY)) {
            dynamic_object_field::add(&mut price_info_object.id, FEE_STORAGE_KEY, fee_coins);
        }
        else {
            let current_fee = dynamic_object_field::borrow_mut<vector<u8>, Coin<SUI>>(
                &mut price_info_object.id,
                FEE_STORAGE_KEY
            );
            coin::join(current_fee, fee_coins);
        };
    }
