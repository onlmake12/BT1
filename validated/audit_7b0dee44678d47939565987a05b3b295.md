Audit Report

## Title
Unbounded JSON-RPC Batch Request Processing Causes Application-Level DoS — (`rpc/src/server.rs`)

## Summary
CKB's JSON-RPC server processes batch requests with no enforced size limit by default. The `JSONRPC_BATCH_LIMIT` static is a `OnceLock<usize>` that is only initialized when the operator explicitly sets `rpc_batch_limit` in config; absent that setting, the batch guard is never triggered and all calls in any batch are processed unconditionally. Within the 10 MiB body size cap, an attacker can pack tens of thousands of RPC calls into a single request, causing unbounded CPU and memory consumption that can crash or severely degrade the node.

## Finding Description
In `rpc/src/server.rs`, the global limit is declared as:

```rust
static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new();
``` [1](#0-0) 

It is only initialized when `config.rpc_batch_limit` is `Some(...)`:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
``` [2](#0-1) 

The batch handler guard is:

```rust
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    {
        return make_error_response(...);
    }
    // processes ALL calls unconditionally
``` [3](#0-2) 

When `rpc_batch_limit` is absent (the default), `JSONRPC_BATCH_LIMIT.get()` returns `None`, the `if let Some(...)` arm is never taken, and every call in the batch is processed. The `rpc_batch_limit` field is typed `Option<usize>` with no default value: [4](#0-3) 

The test suite hard-codes `rpc_batch_limit: Some(1000)` to enable the limit, confirming it is not active in production unless explicitly configured: [5](#0-4) 

The only existing guard is `max_request_body_size` (10 MiB), which is insufficient: a 10 MiB batch of `get_tip_block_number` calls contains approximately 125,000 calls, each requiring a DB snapshot read and JSON serialization. Calls like `get_block_template` (tx-pool traversal and block assembly) amplify CPU cost per call by orders of magnitude.

## Impact Explanation
The RPC server runs in the same process as the CKB node. Submitting large batches of expensive calls (e.g., `get_block_template`) causes unbounded memory allocation for all responses before streaming begins, and saturates the async runtime. This can exhaust process memory and crash the node, or halt block template generation — directly impacting mining. This matches the allowed bounty impact: **High (10001–15000 points) — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The RPC defaults to `127.0.0.1:8114`, so remote exploitation requires the operator to have exposed the port — a common practice for wallet and dApp connectivity. Local exploitation (malicious process, compromised dependency, or script on the same host) requires no privilege and no network exposure. The attack is trivially repeatable in a tight loop. Likelihood is **Medium**: not universally exploitable remotely, but realistic given common deployment patterns and trivially exploitable locally.

## Recommendation
1. Set a safe non-`None` default for `rpc_batch_limit` in `RpcConfig` (e.g., 200–2000) so the limit is enforced unconditionally rather than opt-in.
2. Alternatively, enforce a hard-coded maximum in `handle_jsonrpc` even when `rpc_batch_limit` is not configured, and allow the config to lower it further.
3. Require operators to explicitly opt out of the limit rather than opt in, and document the risk prominently in the default `ckb.toml`.

## Proof of Concept
```bash
# Build a batch of 10,000 get_tip_block_number calls (~800 KB, well under 10 MiB)
python3 -c "
import json, sys
batch = [{'id': i, 'jsonrpc': '2.0', 'method': 'get_tip_block_number', 'params': []} for i in range(10000)]
sys.stdout.write(json.dumps(batch))
" > batch.json

# Send to a default CKB node (no rpc_batch_limit configured)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data-binary @batch.json | wc -c
```

With `rpc_batch_limit` absent from `ckb.toml`, the server processes all 10,000 calls and returns a 10,000-element JSON array. Repeating in a tight loop from a local process saturates the RPC runtime. Substituting `get_block_template` amplifies CPU cost per call by orders of magnitude, accelerating memory exhaustion and potential process crash.
