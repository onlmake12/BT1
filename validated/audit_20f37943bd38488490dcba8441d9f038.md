### Title
Unvalidated `txs_sizes: None` in `BlockExt` Causes Panic-DoS on `get_fee_rate_statistics` RPC — (`File: rpc/src/util/fee_rate.rs`)

### Summary

`FeeRateCollector::statistics()` calls `.expect()` unconditionally on the `txs_sizes` field of every `BlockExt` it processes. Blocks stored in the legacy `BlockExt` molecule format (before `BlockExtV1` was introduced) deserialize with `txs_sizes = None`. Any unprivileged RPC caller who triggers `get_fee_rate_statistics` or `get_fee_rate_statics` while the node's recent-block window contains at least one such legacy-format block will cause the node process to panic and crash.

### Finding Description

`FeeRateCollector::statistics()` iterates over up to 101 recent blocks and, for each one, unconditionally unwraps `txs_sizes`:

```rust
let txs_sizes = txs_sizes.expect("expect txs_size's length >= 1");
``` [1](#0-0) 

The `BlockExt` struct's `txs_sizes` field is typed `Option<Vec<u64>>` and is explicitly documented as potentially absent: [2](#0-1) 

The legacy molecule schema `BlockExt` (as opposed to `BlockExtV1`) has no `cycles` or `txs_sizes` fields: [3](#0-2) 

When the store deserializes a block stored in the old `BlockExt` format, it hard-codes both fields to `None`: [4](#0-3) 

The `FeeRateProvider` implementation for `Snapshot` fetches block exts directly from the store without any format guard: [5](#0-4) 

Both public RPC handlers delegate to `FeeRateCollector::statistics()` without any pre-check: [6](#0-5) 

**Attack path:**
1. Node has any block in the last 101-block window stored in the legacy `BlockExt` format (e.g., a node migrated from a pre-`BlockExtV1` version, a dev/test chain, or a node whose tip is within the first 101 blocks during initial sync).
2. Unprivileged RPC caller sends `get_fee_rate_statistics` (or the deprecated alias `get_fee_rate_statics`) with any `target` value that causes the collector to reach a block with `txs_sizes = None`.
3. The `.expect()` at line 93 panics, terminating the node process.

The `Chain` RPC module is enabled by default in production config: [7](#0-6) 

### Impact Explanation

A single crafted RPC call crashes the node process. The node must be restarted manually. If the condition persists (e.g., the tip stays within the affected block range), repeated calls keep the node down, constituting a sustained denial of service. No privileged access, key material, or network position is required — only the ability to reach the RPC port.

### Likelihood Explanation

On a long-running mainnet node the last 101 blocks are all recent and stored as `BlockExtV1`, so the window is not normally affected. The risk is elevated for: (a) nodes migrated from pre-`BlockExtV1` versions where the tip is near the migration boundary, (b) dev/private chains where the chain is short, and (c) nodes during initial sync whose tip is within the first 101 blocks. The CHANGELOG entry for v0.119.0 (`#4654`) confirms this code area was already known to have length-invariant bugs, indicating the `None` case was not fully audited at the same time. [8](#0-7) 

### Recommendation

Replace the hard `.expect()` with a graceful skip:

```rust
let txs_sizes = match txs_sizes {
    Some(v) => v,
    None => return fee_rates, // skip blocks without size data
};
```

Additionally, add an invariant check that `txs_sizes.len() == txs_fees.len() + 1` before the inner loop, returning `fee_rates` unchanged if violated, to prevent silent data corruption from malformed stored data.

### Proof of Concept

1. Run a CKB node on a dev chain or migrate a node from a pre-`BlockExtV1` version so that at least one block in the last 101 has `txs_sizes = None` in its stored `BlockExt`.
2. Call the RPC:
   ```json
   {"id":1,"jsonrpc":"2.0","method":"get_fee_rate_statistics","params":[null]}
   ```
3. The node process panics with `"expect txs_size's length >= 1"` and exits.
4. The node is unavailable until manually restarted; repeated calls keep it down if the condition persists.
