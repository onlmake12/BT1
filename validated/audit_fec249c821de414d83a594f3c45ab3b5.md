Audit Report

## Title
Remote peer triggers node panic via `get_block_uncles().expect()` on freezer-pruned uncle data in `GetBlocksProofProcess` — (`util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

When the CKB freezer is active, `delete_block_body` removes `COLUMN_BLOCK_UNCLE` for frozen blocks while leaving `COLUMN_BLOCK_HEADER` and `COLUMN_INDEX` intact. `GetBlocksProofProcess::execute` calls `get_block_uncles(&block_hash).expect("block uncles must be stored")` with no freezer-aware fallback, after confirming `is_main_chain()` via `COLUMN_INDEX`. Any remote peer can crash the node by sending a `GetBlocksProof` message containing a frozen block hash.

## Finding Description

The panic site is confirmed at lines 88–90 of `get_blocks_proof.rs`:

```rust
let uncles = snapshot
    .get_block_uncles(&block_hash)
    .expect("block uncles must be stored");
``` [1](#0-0) 

`get_block_uncles` reads exclusively from `COLUMN_BLOCK_UNCLE` with no freezer branch: [2](#0-1) 

`delete_block_body` (called by `wipe_out_frozen_data` with the comment `// remain header`) explicitly deletes `COLUMN_BLOCK_UNCLE` at line 97: [3](#0-2) 

`wipe_out_frozen_data` calls `batch.delete_block_body()` for every frozen block: [4](#0-3) 

`is_main_chain` reads from `COLUMN_INDEX`, which `delete_block_body` never touches, so it returns `true` for frozen blocks: [5](#0-4) 

By contrast, `get_block` has an explicit freezer-aware branch (lines 44–54) that retrieves the full block from flat files for frozen blocks, bypassing `get_block_uncles`. `get_blocks_proof.rs` calls `get_block_uncles` directly, skipping this path entirely: [6](#0-5) 

The handler is reachable from any connected peer via the light-client protocol message dispatch: [7](#0-6) 

Exploit chain: `is_main_chain(frozen_hash)` → `true` (COLUMN_INDEX intact); `get_block_header(frozen_hash)` → `Some(...)` (COLUMN_BLOCK_HEADER intact); `get_block_uncles(frozen_hash)` → `None` (COLUMN_BLOCK_UNCLE deleted); `.expect(...)` → **panic** → process crash.

## Impact Explanation

Any remote peer connected to a light-client-protocol-enabled node with the freezer active can crash the node process unconditionally by sending a single `GetBlocksProof` message. The panic unwinds the async task and terminates the process with no recovery path short of a manual restart. This matches the allowed CKB bounty impact: **"Vulnerabilities which could easily crash a CKB node" — High (10001–15000 points)**.

## Likelihood Explanation

Both the freezer and the light-client protocol server are non-default but documented, supported production features. Once both are enabled, every frozen block hash — all blocks older than the freeze threshold, publicly enumerable from any block explorer — is a valid trigger. The attacker requires no credentials, no proof-of-work, and no special knowledge beyond a network connection to the light-client protocol port and any old block hash. The attack is repeatable and requires no state.

## Recommendation

Replace the direct `get_block_uncles().expect()` call in `get_blocks_proof.rs` with a freezer-aware path mirroring the logic in `get_block()` (`store/src/store.rs` lines 44–54): check if the block number falls below `freezer.number()`, and if so, retrieve the full block from the freezer flat file and extract uncles from it. Alternatively, return `StatusCode::InternalError` instead of panicking when uncle data is absent for a main-chain block, consistent with how `reply_proof` handles MMR errors at lines 200–215 of `lib.rs`.

## Proof of Concept

1. Enable the freezer and the light-client protocol server in the node config; let the chain advance past the freeze threshold so at least one block is frozen (uncle data deleted from `COLUMN_BLOCK_UNCLE`, index and header retained).
2. Connect a peer to the light-client protocol port.
3. Send a `GetBlocksProof` message where `block_hashes` contains any frozen block hash and `last_hash` is the current tip (passes `is_main_chain()`).
4. Execution path: `is_main_chain(frozen_hash)` → `true`; `get_block_header(frozen_hash)` → `Some(...)`; `get_block_uncles(frozen_hash)` → `None`; `.expect("block uncles must be stored")` → **panic** → node process crashes.
