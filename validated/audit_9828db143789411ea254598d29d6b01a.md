Audit Report

## Title
Weak Single-Child Epoch Check in `need_clean` Enables Orphan Pool Memory Exhaustion — (File: `chain/src/utils/orphan_block_pool.rs`)

## Summary
`InnerPool::need_clean` samples only the first child returned by a non-deterministic `HashMap` iterator to decide whether to evict an entire orphan subtree. An attacker can insert two sibling orphan blocks under the same parent — one with a recent epoch, one with an expired epoch — so that the recent-epoch child is iterated first, permanently blocking eviction of the expired child. Because `OrphanBlockPool::insert` has no size cap and epoch numbers are not validated before orphan pool insertion, this allows unbounded memory growth that can crash a CKB node.

## Finding Description

**Root cause — `need_clean` samples one arbitrary child:**

`InnerPool::need_clean` at lines 112–122 of `chain/src/utils/orphan_block_pool.rs`:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .and_then(|map| {
            map.iter().next().map(|(_, lonely_block)| {   // ← arbitrary first child
                lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
            })
        })
        .unwrap_or_default()
}
```

`self.blocks` maps each parent hash to a `HashMap<packed::Byte32, LonelyBlockHash>` of its children. [1](#0-0)  `map.iter().next()` returns an **arbitrary** child because Rust `HashMap` iteration order is unspecified. The eviction decision for the entire subtree — including all siblings — is based on whichever child happens to be iterated first. [2](#0-1) 

`clean_expired_blocks` calls `need_clean` per leader and removes the whole subtree only if it returns `true`: [3](#0-2) 

**No size cap on insertion:**

`OrphanBlockPool::insert` has no size guard. The comment at line 125 explicitly forbids `LruCache` to prevent implicit pruning, meaning there is no fallback eviction path: [4](#0-3) 

**Epoch numbers are not validated before orphan pool insertion:**

`asynchronous_process_block` in `chain_service.rs` calls `non_contextual_verify` before routing to the orphan broker. [5](#0-4)  `BlockVerifier::verify` (the non-contextual verifier) only checks proposals limit, block bytes, cellbase structure, duplicates, and merkle root — **no PoW check, no epoch number check**: [6](#0-5)  PoW and epoch validation live in `HeaderVerifier`, which is context-dependent (requires the parent header) and is therefore not reachable for orphan blocks whose parent is unknown. [7](#0-6) 

**Exploit flow:**

1. Attacker connects to a CKB node via P2P.
2. Attacker crafts block **A**: `parent_hash = P` (unknown to node), `epoch_number = tip_epoch − 1` (recent, not expired). Passes non-contextual verification.
3. Attacker crafts block **B**: `parent_hash = P`, `epoch_number = tip_epoch − 10` (expired, should be cleaned). Passes non-contextual verification.
4. Both are inserted under `blocks[P]` in the orphan pool.
5. When `clean_expired_blocks(tip_epoch)` runs (every 60 seconds per the timer in `chain_service.rs`), `need_clean(P, tip_epoch)` samples one child. [8](#0-7) 
6. If A is sampled: `(tip_epoch − 1) + 6 < tip_epoch` → false → neither A nor B is evicted.
7. Attacker repeats with new parent hashes P1, P2, … Pn. Pool grows by 2 entries per round with no upper bound.

**Secondary failure mode:** If B is sampled first, `remove_blocks_by_parent` removes **all** children of P, including the valid recent block A, causing unnecessary re-requests and sync stalls. [9](#0-8) 

**Off-by-one:** The condition `epoch_number() + EXPIRED_EPOCH < tip_epoch` (strict `<`) retains blocks at exactly `tip_epoch − EXPIRED_EPOCH` for one extra epoch beyond the documented intent. [10](#0-9) 

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Repeated insertion of sibling orphan blocks with mixed epoch numbers causes the orphan pool to grow without bound. Because the pool is protected by an `RwLock` and consulted on every block arrival and every 60-second cleanup tick, unbounded growth leads to memory exhaustion and eventual node crash or severe unresponsiveness. The attack targets a single node, not the whole network, placing it in the High severity band.

## Likelihood Explanation

Any peer on the CKB P2P network can relay block messages. Non-contextual verification does not check PoW or epoch numbers, so crafting the required blocks requires no mining capability — only the ability to construct well-formed block structures (valid cellbase, correct merkle root, valid byte size). The attack is cheap, stateless, and indefinitely repeatable. The cleanup timer fires only once per minute, giving the attacker ample time to grow the pool between cleanup cycles.

## Recommendation

1. **Fix the weak condition** — replace the single-child sample with a check over all children; retain the subtree only if at least one child is still within the expiry window:

```rust
fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
    self.blocks
        .get(parent_hash)
        .map(|map| {
            map.values().all(|b| b.epoch_number() + EXPIRED_EPOCH <= tip_epoch)
        })
        .unwrap_or_default()
}
```

2. **Fix the off-by-one** — change `<` to `<=` so blocks at exactly `tip_epoch − EXPIRED_EPOCH` are cleaned as documented.

3. **Enforce a pool size cap** — add a maximum entry count to `OrphanBlockPool::insert` and evict the oldest/lowest-epoch entries when the cap is reached, independent of the epoch-based cleanup path.

## Proof of Concept

```
Constants: EXPIRED_EPOCH = 6, tip_epoch = 20, cleanup_interval = 60s

Setup:
  Craft Block A: parent_hash = P, epoch_number = 19 (recent)
  Craft Block B: parent_hash = P, epoch_number = 9  (expired)
  Both pass BlockVerifier (no PoW/epoch check in non-contextual path)

Attack loop (repeat with P1, P2, … Pn):
  Send A → inserted as blocks[P][hash_A]
  Send B → inserted as blocks[P][hash_B]

  clean_expired_blocks(20):
    need_clean(P, 20):
      map.iter().next() → (hash_A, A)   [non-deterministic, but A likely first]
      19 + 6 = 25 < 20 → false
      → subtree not evicted

Result: 2 blocks permanently retained per parent P.
        After n rounds: 2n blocks in pool, growing without bound.
        Node OOM crash after sufficient iterations.
```

Minimal unit test plan: construct an `InnerPool` directly, insert two `LonelyBlockHash` entries with the same parent and epoch numbers 19 and 9 respectively, call `clean_expired_blocks(20)`, and assert that `parents.len() == 2` (neither was evicted) — demonstrating the bug without any network setup.

    pub fn remove_blocks_by_parent(&mut self, parent_hash: &ParentHash) -> Vec<LonelyBlockHash> {
        // try remove leaders first
        if !self.leaders.remove(parent_hash) {
            return Vec::new();
        }

        let mut queue: VecDeque<packed::Byte32> = VecDeque::new();
        queue.push_back(parent_hash.to_owned());

        let mut removed: Vec<LonelyBlockHash> = Vec::new();
        while let Some(parent_hash) = queue.pop_front() {
            if let Some(orphaned) = self.blocks.remove(&parent_hash) {
                let (hashes, blocks): (Vec<_>, Vec<_>) = orphaned.into_iter().unzip();
                for hash in hashes.iter() {
                    self.parents.remove(hash);
                }
                queue.extend(hashes);
                removed.extend(blocks);
            }
        }

        debug!("orphan pool pop chain len: {}", removed.len());
        debug_assert_ne!(
            removed.len(),
            0,
            "orphan pool removed list must not be zero"
        );

        shrink_to_fit!(self.blocks, SHRINK_THRESHOLD);
        shrink_to_fit!(self.parents, SHRINK_THRESHOLD);
        shrink_to_fit!(self.leaders, SHRINK_THRESHOLD);
        removed
    }
