### Title
Incomplete Peer State Cleanup on Disconnect in `HolePunching` Protocol — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

The `HolePunching` P2P protocol handler fails to remove per-peer entries from its `inflight_requests` and `pending_delivered` state maps when a peer disconnects. This is a direct analog to the EIP-7579 finding where `onUninstall()` omits `delete walletSessionKeys[msg.sender]`, leaving stale per-user state behind. In CKB, any unprivileged remote peer can populate these maps and then disconnect, causing unbounded memory growth and stale state that persists across reconnections.

---

### Finding Description

`HolePunching` maintains two per-`PeerId` maps as part of its protocol state:

```rust
pub(crate) struct HolePunching {
    network_state: Arc<NetworkState>,
    bind_addr: Option<SocketAddr>,
    inflight_requests: HashMap<PeerId, u64>,          // populated on received messages
    pending_delivered: HashMap<PeerId, PendingDeliveredInfo>, // populated on received messages
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
    forward_rate_limiter: RateLimiter<(PeerId, PeerId, u32)>,
}
``` [1](#0-0) 

When a peer disconnects, the `disconnected` handler only calls `retain_recent()` on the two rate limiters and logs a debug message. It does **not** remove the disconnected peer's entries from `inflight_requests` or `pending_delivered`:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    debug!("HolePunching.disconnected session={}", context.session.id);
}
``` [2](#0-1) 

By contrast, the `connected` handler correctly registers the protocol version in the peer registry: [3](#0-2) 

The `inflight_requests` and `pending_delivered` maps are keyed by `PeerId` (not `SessionId`), so entries survive across reconnections of the same peer identity. The `received` handler populates these maps during normal protocol operation. There is no immediate cleanup path on disconnect.

---

### Impact Explanation

1. **Memory exhaustion (DoS)**: An attacker can repeatedly connect, send `ConnectionRequest` or `ConnectionRequestDelivered` messages to populate `inflight_requests` and `pending_delivered`, then disconnect. Each cycle leaves orphaned entries. Because `PeerId` is attacker-controlled (any peer can generate a fresh identity), the maps grow without bound, eventually exhausting node memory.

2. **Stale state / logic corruption**: When the same `PeerId` reconnects, the old `inflight_requests` timestamp entry is still present. The protocol may incorrectly treat the new session as a continuation of the old one, bypassing freshness checks or causing incorrect timeout accounting. Similarly, stale `pending_delivered` entries can cause the node to re-process or misroute hole-punching coordination messages for a new session.

---

### Likelihood Explanation

The `HolePunching` protocol is registered as a standard P2P protocol on the CKB network. Any unprivileged peer that can establish a TCP connection to a CKB node can open this protocol sub-stream and send valid `ConnectionRequest` or `ConnectionRequestDelivered` messages. No authentication, stake, or privileged role is required. The attack is fully reachable from the public internet.

---

### Recommendation

In the `disconnected` handler, extract the `PeerId` from the session address and remove the corresponding entries from both maps:

```rust
async fn disconnected(&mut self, context: ProtocolContextMutRef<'_>) {
    if let Some(peer_id) = extract_peer_id(&context.session.address) {
        self.inflight_requests.remove(&peer_id);
        self.pending_delivered.remove(&peer_id);
    }
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    debug!("HolePunching.disconnected session={}", context.session.id);
}
```

This mirrors the fix applied to the EIP-7579 report: explicitly deleting the per-user mapping entry in the uninstall/disconnect path.

---

### Proof of Concept

1. Attacker opens a TCP connection to a CKB full node and negotiates the `HolePunching` sub-protocol.
2. Attacker sends a valid `ConnectionRequest` message. The `received` handler processes it and inserts an entry into `inflight_requests` keyed by the attacker's `PeerId`.
3. Attacker closes the connection. `HolePunching::disconnected` fires, calls `retain_recent()` on rate limiters, but leaves the `inflight_requests` entry intact.
4. Attacker repeats steps 1–3 with fresh `PeerId` values (trivially generated). Each cycle adds a new orphaned entry.
5. After sufficient iterations, the node's heap is exhausted, causing an OOM crash or severe degradation.

Alternatively, the attacker reconnects with the same `PeerId`: the stale `inflight_requests` timestamp from the previous session is still present, causing the protocol to compute incorrect elapsed times and potentially misclassify the new request as a duplicate or timed-out entry. [4](#0-3)
