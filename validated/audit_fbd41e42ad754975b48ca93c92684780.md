Audit Report

## Title
Rate Limiter Counts Messages Not Hashes, Enabling ~983K Tx-Pool Lookups/Second Per Peer — (`sync/src/relayer/get_transactions_process.rs`, `sync/src/relayer/mod.rs`)

## Summary
The relay rate limiter consumes exactly one token per `GetRelayTransactions` message regardless of payload size. Because `MAX_RELAY_TXS_NUM_PER_BATCH` is 32 767 and the boundary check uses strict `>` (allowing exactly 32 767 hashes through), a single peer at the 30 msg/s cap can drive approximately 983 K `fetch_txs_with_cycles` hash-lookups per second into the serialized tx-pool controller, degrading legitimate tx-pool operations for all callers on the same node.

## Finding Description
**Rate limiter setup:** `Relayer::new` creates a `governor::RateLimiter` keyed by `(PeerIndex, u32)` with a quota of 30 tokens/second per `(peer, message_type)` pair (`sync/src/relayer/mod.rs` L89–92).

**Per-message token consumption:** `try_process` calls `check_key(&(peer, message.item_id()))`, consuming exactly one token per message with no awareness of how many hashes the message carries (`sync/src/relayer/mod.rs` L116–123).

**Batch size constant and boundary check:** `MAX_RELAY_TXS_NUM_PER_BATCH` is 32 767. The guard in `GetTransactionsProcess::execute` uses `message_len > MAX_RELAY_TXS_NUM_PER_BATCH`, so a message carrying exactly 32 767 hashes passes validation (`sync/src/relayer/get_transactions_process.rs` L35–39).

**Deduplication scope:** The `HashSet` dedup at L59–61 only catches repeated hashes *within* a single message; it provides no cross-message protection.

**Tx-pool call:** All hashes from a passing message are forwarded in one `fetch_txs_with_cycles` call to the serialized tx-pool controller actor (`sync/src/relayer/get_transactions_process.rs` L63).

**Exploit flow:** Attacker completes standard P2P handshake → sends 30 `GetRelayTransactions` messages/second, each containing 32 767 distinct (non-existent) hashes → rate limiter passes every message (1 token each) → tx-pool controller receives ~983 K hash-lookup requests/second → its serialized message queue saturates → mining template generation, tx admission, and proposal processing are delayed or stalled.

## Impact Explanation
The tx-pool controller is a single-threaded async actor; all tx-pool operations serialize through its channel. Sustained flooding at ~983 K lookups/second from a single peer can fill that channel, causing measurable latency or stalling of mining template generation and transaction propagation. This matches the allowed CKB bounty impact: **Low (501–2000) — Any other important performance improvements for CKB.**

## Likelihood Explanation
No privilege, proof-of-work, or key material is required. Any peer completing the standard P2P handshake can immediately begin the attack. Construction is trivial: generate 32 767 random 32-byte values per message, send 30 messages/second. The rate limiter passes every message by design. The attack is repeatable and sustained indefinitely from a single connection.

## Recommendation
Replace per-message token consumption with per-hash token consumption. Before calling `fetch_txs_with_cycles`, consume `message_len` tokens from the rate limiter using `check_key_n` or an equivalent n-token API rather than a fixed 1 token. Alternatively, maintain a separate per-peer hash-per-second budget (e.g., 1 000 hashes/second) enforced independently of the message count. Either approach closes the amplification gap between the rate-limit unit (messages) and the actual work unit (hash lookups).

## Proof of Concept
```python
import time, random

def random_byte32():
    return bytes(random.getrandbits(8) for _ in range(32))

while True:
    for _ in range(30):
        hashes = [random_byte32() for _ in range(32767)]  # all unique, all absent from pool
        msg = build_get_relay_transactions(tx_hashes=hashes)
        p2p_send(msg)          # rate limiter passes: 1 token consumed
        time.sleep(1/30)
    # Expected: tx-pool controller receives ~983K fetch_txs_with_cycles lookups/second
    # from this single peer, measurably increasing latency for all other tx-pool callers.
```
