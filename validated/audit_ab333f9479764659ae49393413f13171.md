### Title
Unbounded Repeated `InternalError` (5xx) Without Peer Ban in Light Client Protocol — (`util/light-client-protocol-server/src/lib.rs`, `components/get_blocks_proof.rs`)

---

### Summary

The light client protocol server's `should_ban()` function only bans peers for 4xx status codes. An unprivileged remote peer can craft a `GetBlocksProof` (or `GetTransactionsProof`) message that reliably triggers `StatusCode::InternalError` (500) by supplying a `last_hash` pointing to an early block while listing hashes of later main-chain blocks. This causes the server to perform up to 1000 DB reads and an MMR proof generation attempt per request — all of which fail — without ever banning the peer, allowing indefinite repetition.

---

### Finding Description

**Entry point:** `LightClientProtocol::received` dispatches to `try_process`, which calls `GetBlocksProofProcess::execute`.

**Missing ancestor check in `GetBlocksProofProcess::execute`:**

The handler partitions `block_hashes` into `found`/`missing` based solely on `is_main_chain`: [1](#0-0) 

There is no check that the found blocks have `block_number ≤ last_block.number()`. Positions are then computed directly from each block's actual number: [2](#0-1) 

**`reply_proof` creates an MMR scoped to `last_block.number() - 1`:** [3](#0-2) 

If `last_hash` is block 1 (number=1), the MMR covers only block 0 (size=1). Positions for blocks 500–1499 are far out of range, so `mmr.gen_proof(items_positions)` returns `Err`, and `StatusCode::InternalError` (500) is returned.

**`should_ban()` excludes all 5xx codes:** [4](#0-3) 

`(400..500).contains(&500)` is `false`. The peer receives only a `warn!` log and is never banned: [5](#0-4) 

**The same structural flaw exists in `GetTransactionsProof`:** transactions confirmed in blocks after `last_hash` are on the main chain, pass the partition check, and their block positions exceed the MMR size for `last_block`, hitting the same `InternalError` path in `reply_proof`. [6](#0-5) [7](#0-6) 

There is no rate limiting anywhere in the light client protocol handler. [8](#0-7) 

---

### Impact Explanation

Each crafted request (up to 1000 hashes, the maximum allowed) forces the server to perform:
- Up to 1000 `is_main_chain` DB lookups
- Up to 1000 `get_block_header` reads
- Up to 1000 `get_block_uncles` reads
- Up to 1000 `get_block_extension` reads
- One MMR root computation
- One failed `gen_proof` call

All of this work is discarded on every request. Because the peer is never banned and there is no rate limit, a single connection can sustain this at maximum throughput, causing persistent CPU and I/O congestion on the full node serving light clients.

---

### Likelihood Explanation

The attack requires no privileges, no PoW, and no key material. Any peer that knows a few block hashes (trivially obtained from the public chain) can construct the malicious message. The `GetBlocksProof` message format is part of the documented light client protocol, so crafting it is straightforward.

---

### Recommendation

1. **Add an ancestor check** in `GetBlocksProofProcess::execute` and `GetTransactionsProofProcess::execute`: after partitioning, verify that every block in `found` has `block_number ≤ last_block.number()`. Blocks violating this should be moved to `missing` (or the request should return `StatusCode::InvalidRequest`, a 4xx that triggers a ban).

2. **Extend `should_ban()` to cover 5xx codes** that are caused by client-supplied data (e.g., `InternalError` triggered by out-of-range positions), or introduce a distinct 4xx code for "client-supplied data caused server-side failure."

3. **Add per-peer rate limiting** in the light client protocol handler as a defense-in-depth measure.

---

### Proof of Concept

```
1. Attacker connects as a light client peer.
2. Attacker obtains block_1_hash (block number 1) and block_500_hash..block_1499_hash
   (all on the main chain, numbers 500–1499) from any public source.
3. Attacker sends:
     GetBlocksProof {
       last_hash:    block_1_hash,
       block_hashes: [block_500_hash, block_501_hash, ..., block_1499_hash]  // 1000 hashes
     }
4. Server executes GetBlocksProofProcess::execute:
   - All 1000 hashes pass is_main_chain → found = [500..1499]
   - positions = [leaf_index_to_pos(500), ..., leaf_index_to_pos(1499)]
5. reply_proof creates MMR for block 0 (size = 1).
6. mmr.gen_proof(positions) → Err → StatusCode::InternalError (500).
7. should_ban() → None (500 ∉ 400..500).
8. Peer is NOT banned. Repeat from step 3 indefinitely.
```

The server performs ~4000 DB reads and one failed MMR computation per iteration, with no bound on repetition rate from a single peer.
