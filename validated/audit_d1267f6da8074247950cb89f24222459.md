### Title
Fee Accounting Mismatch Between Static Configured Fee and Dynamic Actual Pyth Fee in Echo Contract — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary
In `Echo.sol`, `requestPriceUpdatesWithCallback` stores the provider's fee as `msg.value - _state.pythFeeInWei` (Echo's static configured Pyth fee). However, `executeCallback` deducts the **actual** dynamic Pyth fee (`pyth.getUpdateFee(updateData)`) from the provider's credit. These two values can diverge, causing the executor to be underpaid and potentially locking user funds permanently.

### Finding Description
In `requestPriceUpdatesWithCallback`, the stored provider fee is computed using Echo's static `_state.pythFeeInWei`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
``` [1](#0-0) 

At execution time in `executeCallback`, the actual Pyth fee is computed dynamically from the update data and deducted from the provider's credit:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);
...
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [2](#0-1) 

The executor's net profit from calling `executeCallback` is `req.fee - actual_pythFee`. If `pyth.getUpdateFee(updateData) > req.fee`, the executor loses money on every execution attempt and has no rational incentive to call the function.

`_state.pythFeeInWei` is Echo's own static configured fee (set at initialization), while `pyth.getUpdateFee(updateData)` is the actual fee charged by the underlying Pyth price feed contract, which depends on the number of price updates in the data and the current `singleUpdateFeeInWei` set by Pyth governance. [3](#0-2) 

### Impact Explanation
When `pyth.getUpdateFee(updateData) > req.fee` (i.e., the actual Pyth fee exceeds the provider fee portion stored in the request), no rational executor will call `executeCallback`. The user's ETH is permanently locked in the Echo contract with no mechanism to cancel or refund the request. The comment in the code acknowledges this risk: *"If executeCallback can revert, then funds can be permanently locked in the contract."* [4](#0-3) 

Additionally, Echo's `accruedFeesInWei` accumulates `_state.pythFeeInWei` per request, but the actual Pyth fee paid at execution time comes from the executor's `msg.value`. If the actual Pyth fee is higher than `_state.pythFeeInWei`, Echo over-collects from users (the excess sits in `accruedFeesInWei` and is withdrawable by admin), while providers must subsidize the difference.

### Likelihood Explanation
The Pyth contract's `singleUpdateFeeInWei` is adjustable via governance (`SetTransactionFee` / `SetSingleUpdateFee` governance actions). Any increase in the Pyth fee after a request is submitted but before it is executed causes the mismatch. Given that Echo requests can remain pending for an extended period (no expiry mechanism is enforced), fee changes during that window are realistic. The entry path is fully unprivileged: any user calling `requestPriceUpdatesWithCallback` creates a request that is vulnerable to this condition. [5](#0-4) 

### Recommendation
At request time, query the actual Pyth fee using `pyth.getUpdateFee` with the expected update data size, or store the actual Pyth fee paid at execution time and reconcile it against what was collected. Alternatively, implement a request cancellation/refund mechanism so users can recover funds if a request becomes unexecutable due to fee changes.

### Proof of Concept
1. Pyth governance sets `singleUpdateFeeInWei = 1 wei`. Provider sets fees such that `getFee(...)` returns `10 wei` (pythFeeInWei=1, providerFees=9).
2. User calls `requestPriceUpdatesWithCallback{value: 10}(...)`. `req.fee = 10 - 1 = 9`.
3. Pyth governance increases `singleUpdateFeeInWei` to `100 wei`. Now `pyth.getUpdateFee(updateData) = 100`.
4. Provider calls `executeCallback{value: 0}(...)`. Computation: `(9 + 0) - 100` → underflow revert in Solidity 0.8+.
5. Provider calls `executeCallback{value: 91}(...)` to avoid revert. Provider credit = `9 + 91 - 100 = 0`. Provider receives nothing for their work.
6. No rational executor will call `executeCallback`. The user's 10 wei is permanently locked. [6](#0-5)

        // Check provider exclusivity using configurable period

        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
