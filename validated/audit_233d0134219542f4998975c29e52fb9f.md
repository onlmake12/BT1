### Title
Unbounded Synchronous Script Execution via `test_tx_pool_accept` Bypasses Resumable Verify Queue, Enabling CPU Monopolization DoS â€” (`tx-pool/src/util.rs`, `tx-pool/src/process.rs`)

---

### Summary

`test_tx_pool_accept` runs full script verification synchronously via `block_in_place` with no `ChunkCommand` pause support (`command_rx = None`), while `send_transaction` enqueues transactions into a resumable verify queue that supports pausing. An unprivileged RPC caller can submit transactions with near-`max_block_cycles` scripts and call `test_tx_pool_accept` concurrently to saturate CPU and stall the tx-pool service actor.

---

### Finding Description

**Entrypoint:** `PoolRpc::test_tx_pool_accept` in `rpc/src/module/pool.rs` [1](#0-0) 

This calls `tx_pool_controller.test_accept_tx(tx)`, which dispatches to `TxPoolService::test_accept_tx`: [2](#0-1) 

Which calls `_test_accept_tx`: [3](#0-2) 

**Critical observation:** `verify_rtx` is called with `command_rx = None` (line 797) and `max_cycles = self.consensus.max_block_cycles()` (line 787).

Inside `verify_rtx` in `tx-pool/src/util.rs`, the `command_rx = None` branch falls through to `block_in_place`: [4](#0-3) 

`block_in_place` runs `ContextualTransactionVerifier::verify` synchronously on the current thread with no pause mechanism. The CKB-VM executes the script to completion (up to `max_block_cycles`) without any `ChunkCommand` interruption.

**Contrast with `send_transaction`:** `send_transaction` calls `submit_local_tx`, which routes through `resumeble_process_tx` â†’ `enqueue_verify_queue`. The verify queue worker calls `_process_tx` with a live `command_rx`: [5](#0-4) 

When `command_rx = Some(...)`, `verify_rtx` takes the `verify_with_pause` path: [6](#0-5) 

This allows the tx-pool service to send `ChunkCommand::Pause` to interrupt long-running verifications. `test_tx_pool_accept` has no such mechanism.

---

### Impact Explanation

Each concurrent `test_tx_pool_accept` call with a max-cycle script:
1. Calls `block_in_place`, consuming a thread for the full script execution duration (potentially seconds for `max_block_cycles`-level scripts)
2. Tokio spawns a replacement worker thread, but CPU is saturated by N concurrent CKB-VM instances
3. The tx-pool service actor is blocked processing the request, queuing all other tx-pool messages (including legitimate `send_transaction` calls)
4. Other RPC endpoints become unresponsive due to CPU saturation

The invariant violation is concrete: `test_tx_pool_accept` uses `max_block_cycles` as the cycle limit (line 787) with no pause support, while `send_transaction` uses the resumable path with `verify_with_pause`. [7](#0-6) 

---

### Likelihood Explanation

- `test_tx_pool_accept` is a standard public RPC method, no authentication required
- Many CKB node operators expose RPC publicly for dApp/wallet connectivity
- The attacker only needs to craft a valid transaction referencing a deployed script that loops near `max_block_cycles`; the same UTXO can be reused across calls since the tx is never committed
- No PoW, no privileged key, no Sybil attack required â€” pure RPC abuse

---

### Recommendation

1. Route `_test_accept_tx` through the same resumable verify queue as `send_transaction`, passing a live `command_rx` so verification can be paused/cancelled
2. Apply per-IP or global concurrency limits on `test_tx_pool_accept` at the RPC layer
3. Cap `max_cycles` for `test_tx_pool_accept` to a fraction of `max_block_cycles` (e.g., `max_tx_verify_cycles` used elsewhere)
4. Consider making `test_tx_pool_accept` a restricted/authenticated endpoint

---

### Proof of Concept

```
1. Deploy a CKB script that loops consuming exactly max_block_cycles - 1 cycles
2. Create a valid transaction spending a live UTXO with that script as lock
3. Send 100 concurrent JSON-RPC calls to test_tx_pool_accept with that transaction
4. Observe: CPU pegged at 100% across all cores, tx_pool_info RPC stops responding,
   send_transaction calls queue indefinitely
5. Confirm: same transaction via send_transaction completes quickly because
   verify_with_pause allows the scheduler to interleave and bound execution
```
