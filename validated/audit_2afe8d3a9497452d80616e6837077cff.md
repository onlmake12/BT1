## Analysis

Let me trace the exact code path for this claim.

### Step 1 — Entrypoint: Discovery Protocol

An attacker connects via P2P and sends `DiscoveryMessage::Nodes`. The handler calls `add_new_addrs`, which calls `peer_store.add_addr` for each address. Errors are silently swallowed with only a `debug!` log: [1](#0-0) 

### Step 2 — `add_addr` calls `check_purge`

`add_addr` is explicitly for untrusted/unverified addresses. It calls `check_purge()` before inserting: [2](#0-1) 

### Step 3 — `check_purge` eviction logic

`ADDR_COUNT_LIMIT = 16384`: [3](#0-2) 

When the store is full, `check_purge` first tries to evict non-connectable addresses. For fresh addresses (`last_connected_at_ms = 0`, `attempts_count = 0`), `is_connectable` returns `true`: [4](#0-3) 

So the first pass finds nothing. The second pass groups by network segment and only evicts from groups with **strictly more than 4** peers: [5](#0-4) 

The critical condition is `addrs.len() > 4` (line 378) — **strictly greater than 4**.

### Step 4 — Network Group is /16

The `Group` for IPv4 uses only the first two octets: [6](#0-5) 

### Step 5 — The Exploit Path

- Attacker sends Discovery `Nodes` messages (up to 1000 nodes × 3 addrs = 3000 addrs per message, so ~6 messages suffice)
- Advertises 16384 fake addresses: exactly 4 per /16 subnet across 4096 distinct /16 subnets
- All addresses are fresh → all `is_connectable = true` → first pass evicts nothing
- Second pass: 4096 groups, each with exactly 4 entries → `4 > 4` is `false` → no candidates
- `candidate_peers.is_empty()` → `return Err(PeerStoreError::EvictionFailed)`
- `add_addr` fails for every subsequent honest peer address
- The error is silently dropped in `add_new_addrs` (debug log only)

The attacker needs **only one P2P connection** — the addresses themselves are never verified before storage.

### Title
Peer Store Exhaustion via Crafted Discovery Addresses Causes Permanent `EvictionFailed`, Enabling Eclipse Attack — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
An unprivileged remote peer can permanently exhaust the CKB peer store by advertising 16384 fake addresses distributed as exactly 4 per /16 subnet via the Discovery protocol. Once full, `check_purge` cannot evict any entry because the eviction condition requires `> 4` peers per group (strict), and all fresh addresses are considered connectable. Every subsequent `add_addr` call returns `PeerStoreError::EvictionFailed`, silently preventing the victim from learning about any honest peers.

### Finding Description
`PeerStore::check_purge` has two eviction strategies:
1. Remove non-connectable addresses. Fresh addresses always pass `is_connectable` (zero attempts, zero connection time), so this yields nothing.
2. Group by `/16` network segment, take the top `len/2` groups by size, and evict 2 random peers from groups with `> 4` members (strict inequality).

If an attacker fills all 16384 slots with exactly 4 addresses per /16 subnet (4096 subnets × 4 = 16384), neither strategy produces any eviction candidates, and `Err(PeerStoreError::EvictionFailed)` is returned. The Discovery protocol's `add_new_addrs` silently ignores this error, so the victim node permanently stops accepting new peer addresses. [7](#0-6) 

### Impact Explanation
The victim node's peer store is permanently poisoned. It cannot learn about honest peers via Discovery, DNS seeding, or identify. This is a prerequisite for an eclipse attack: the attacker can then control all of the victim's outbound connections, enabling consensus deviation, double-spend relay manipulation, or transaction censorship.

### Likelihood Explanation
The attacker needs only **one P2P connection** to the victim. The Discovery protocol accepts up to 1000 nodes per message (3 addresses each), so ~6 messages suffice to fill 16384 slots. The addresses are never verified before storage — `add_addr` is explicitly for untrusted addresses. No PoW, no privileged role, no Sybil majority is required. [8](#0-7) 

### Recommendation
Change the eviction threshold from strict `> 4` to `>= 4` (or `> 3`), so groups of exactly 4 are also eligible for eviction. Additionally, consider:
- Capping the number of addresses accepted per session/connection
- Requiring a minimum number of successful connection attempts before an address occupies a permanent slot
- Treating `EvictionFailed` as a misbehavior signal against the advertising peer rather than silently discarding it

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
        }
        Ok(())
    }
