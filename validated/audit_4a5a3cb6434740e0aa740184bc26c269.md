### Title
`FeeRate::fee()` Integer Truncation Allows Zero-Fee Transactions to Bypass Minimum Fee Rate Check — (`util/types/src/core/fee_rate.rs`)

---

### Summary

`FeeRate::fee()` computes the minimum required fee using integer division. When `fee_rate * tx_size < 1000`, the result truncates to zero. `check_tx_fee` then computes `min_fee = 0`, and since the actual fee is an unsigned integer, the comparison `fee < min_fee` is never true — allowing zero-fee transactions to bypass the minimum fee rate gate entirely.

---

### Finding Description

`FeeRate::fee()` in `util/types/src/core/fee_rate.rs` is:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000
    Capacity::shannons(fee)
}
``` [1](#0-0) 

When `self.0 * weight < 1000`, integer division truncates the result to `0`. This computed value is used directly as the admission threshold in `check_tx_fee` in `tx-pool/src/util.rs`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
``` [2](#0-1) 

If `min_fee` truncates to `0`, the condition `fee < 0` is never satisfied for any unsigned `fee`, so every transaction — including zero-fee ones — passes the check. The same truncation affects `calculate_min_replace_fee` in `tx-pool/src/pool.rs`, where `extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64)` can also become zero: [3](#0-2) 

The analog to the WooFi `gamma = 0` pattern is exact:

| WooFi | CKB |
|---|---|
| `gamma = baseAmount * price * coeff / priceDec / baseDec` | `min_fee = fee_rate * tx_size / 1000` |
| Rounds to `0` with small `baseAmount` | Rounds to `0` with small `fee_rate` or `tx_size` |
| Price update skipped; swap still succeeds | Fee check skipped; zero-fee tx still admitted |
| Attacker loops to trade without price impact | Attacker loops to flood pool without paying fees |

---

### Impact Explanation

When `min_fee` truncates to zero, a tx-pool submitter (RPC caller via `send_transaction`) can:

1. Submit zero-fee transactions that pass `check_tx_fee` despite a non-zero `min_fee_rate` being configured.
2. Flood the tx-pool (default 180 MB) with zero-cost transactions in a loop, causing legitimate transactions to be delayed or evicted.
3. Bypass the RBF extra-fee requirement: `extra_rbf_fee = 0` means a replacement transaction only needs to match the replaced transaction's fee, not exceed it by the configured `min_rbf_rate` increment.

---

### Likelihood Explanation

The truncation triggers when `min_fee_rate * tx_size < 1000`. A typical CKB transaction is ~242 bytes (confirmed by the integration test log: `"requiring a transaction fee of at least 242 shannons"` with `min_fee_rate = 1000`). [4](#0-3) 

For a 242-byte transaction:
- `min_fee_rate = 1000` (default): `min_fee = 1000 * 242 / 1000 = 242` — safe.
- `min_fee_rate = 4`: `min_fee = 4 * 242 / 1000 = 0` — **bypassed**.
- `min_fee_rate = 1`: `min_fee = 1 * 999 / 1000 = 0` for any tx under 1000 bytes — **bypassed**.

Non-default `min_fee_rate` values are explicitly supported and used in production. The integration test template sets `min_fee_rate = 0`, and integration tests use values of `100`, `1500`, etc. [5](#0-4) [6](#0-5) 

The default `min_fee_rate = 1000` is safe, but any operator who lowers it below `1000 / tx_size` (e.g., to `1`–`4` for typical transactions) silently loses fee enforcement entirely.

---

### Recommendation

Replace floor division with ceiling division in `FeeRate::fee()` so that any non-zero `fee_rate` applied to any non-zero `weight` always yields at least 1 shannon:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    // Ceiling division: ensures min_fee >= 1 whenever fee_rate > 0 and weight > 0
    let fee = self.0.saturating_mul(weight).saturating_add(KW - 1) / KW;
    Capacity::shannons(fee)
}
``` [1](#0-0) 

---

### Proof of Concept

1. Configure a node with `min_fee_rate = 4` (4 shannons/KW — a plausible low-fee setting).
2. Construct a transaction of ~242 bytes with zero fee (outputs capacity == inputs capacity).
3. Submit via `send_transaction` RPC.
4. `check_tx_fee` computes: `min_fee = 4 * 242 / 1000 = 968 / 1000 = 0`.
5. The check `if fee < min_fee` becomes `if 0 < 0` → false → transaction is **admitted**.
6. Repeat in a tight loop to flood the tx-pool with zero-fee transactions at negligible cost.

The same loop pattern from the WooFi PoC applies directly: repeated submissions each pass the truncated-to-zero check, accumulating zero-fee transactions in the pool without any fee payment. [7](#0-6) [8](#0-7)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
