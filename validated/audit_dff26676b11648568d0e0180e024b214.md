Audit Report

## Title
IPv4-Mapped IPv6 Address Aliasing Bypasses Peer Ban List - (File: network/src/peer_store/ban_list.rs)

## Summary
`BanList::is_ip_banned_until` does not normalize IPv4-mapped IPv6 addresses (e.g., `::ffff:1.2.3.4`) to their IPv4 equivalents before performing ban lookups. A peer banned under its IPv4 address can immediately reconnect using the IPv4-mapped IPv6 form of the same address and bypass the ban entirely. The inverse bypass (banned as IPv6, reconnect as IPv4) is equally possible.

## Finding Description
When a peer is banned, `ban_addr` calls `ip_to_network(addr.ip())` which maps `IpAddr::V4(1.2.3.4)` → `IpNetwork::V4(1.2.3.4/32)` and stores it in the `HashMap<IpNetwork, BannedAddr>` keyed by that V4 network.

When the same peer reconnects presenting `/ip6/::ffff:1.2.3.4/tcp/<port>/p2p/<peer_id>`, `accept_peer` calls `peer_store.is_addr_banned(&remote_addr)`, which reaches `is_ip_banned_until`. There, `ip_to_network(IpAddr::V6(::ffff:1.2.3.4))` produces `IpNetwork::V6(::ffff:1.2.3.4/128)`. The HashMap lookup finds no entry (key family mismatch). The fallback `iter().any(|net| net.contains(ip))` evaluates `IpNetwork::V4(1.2.3.4/32).contains(IpAddr::V6(::ffff:1.2.3.4))`, which the `ipnetwork` crate returns `false` for cross-family comparisons. The function returns `false` and the peer is accepted.

The normalization fix already exists in `network_group.rs` (lines 31–35), where `ipv6.to_ipv4()` is used to map IPv4-mapped IPv6 addresses to `Group::IP4` for peer diversity accounting. This same normalization was never applied to the ban list. No other guard in the connection acceptance path performs this normalization.

## Impact Explanation
A banned peer can bypass the ban indefinitely by reconnecting with the IPv4-mapped IPv6 form of its address. This nullifies the node's primary runtime defense against actively misbehaving peers. An attacker who has been banned for sending malformed messages, triggering score-based bans, or causing resource exhaustion can immediately reconnect and repeat the attack. If the attack vector can exhaust node resources or disrupt sync, the ban bypass enables sustained attacks against a single CKB node that would otherwise be stopped — mapping to **High: Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
The attacker is an unprivileged peer with no special access. IPv4-mapped IPv6 is supported on all major operating systems and requires no new IP address, no cryptographic material, and no privileged access. A peer that has been banned has a direct incentive to reconnect. The bypass requires only changing the address family in the multiaddr used to connect. The attack is immediately repeatable after each ban.

## Recommendation
Normalize the incoming `IpAddr` at the top of `is_ip_banned_until` before any lookup. Apply the same normalization in `ip_to_network` or via a dedicated helper mirroring the logic in `network_group.rs`:

```rust
fn normalize_ip(ip: IpAddr) -> IpAddr {
    if let IpAddr::V6(v6) = ip {
        if let Some(v4) = v6.to_ipv4() {
            return IpAddr::V4(v4);
        }
    }
    ip
}
```

Call `normalize_ip(ip)` at the top of `is_ip_banned_until` before `ip_to_network`. Optionally apply the same normalization inside `ban_addr` so that bans are always stored in normalized form, preventing the inverse bypass direction as well.

## Proof of Concept
1. Node bans peer B at IPv4 `1.2.3.4` (e.g., via score-based ban or explicit `set_ban`). `ban_addr` stores `IpNetwork::V4(1.2.3.4/32)` in the ban list.
2. Peer B immediately reconnects using multiaddr `/ip6/::ffff:1.2.3.4/tcp/<port>/p2p/<peer_id>`.
3. `accept_peer` calls `peer_store.is_addr_banned`. `multiaddr_to_socketaddr` returns `SocketAddr::V6([::ffff:1.2.3.4]:port)`. `socket_addr.ip()` returns `IpAddr::V6(::ffff:1.2.3.4)`.
4. `ip_to_network` produces `IpNetwork::V6(::ffff:1.2.3.4/128)`. HashMap lookup: no entry. `iter().any` check: `IpNetwork::V4(1.2.3.4/32).contains(IpAddr::V6(::ffff:1.2.3.4))` → `false`.
5. `is_addr_banned` returns `false`. Peer B is accepted despite being banned.

A unit test can reproduce this directly: insert a `BannedAddr` with `address: IpNetwork::V4("1.2.3.4/32".parse().unwrap())` into a `BanList`, then call `is_ip_banned` with `IpAddr::V6("::ffff:1.2.3.4".parse().unwrap())` and assert the result is `true` — it will currently return `false`, confirming the bypass.
