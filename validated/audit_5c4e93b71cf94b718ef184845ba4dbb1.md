### Title
Blocklist Bypass via New `SignatureSet` Keypair with Replayed Guardian Signatures — (`target_chains/solana/programs/core-bridge/src/legacy/processor/post_vaa.rs`)

---

### Summary

The `INVALID_SIGNATURE_SET_KEYS` blocklist in `PostVaa::constraints` checks only the `SignatureSet` account's **keypair address**, not the **message hash** it contains. Because `SignatureSet` accounts are created with arbitrary keypairs (not PDAs), an attacker can generate a fresh keypair, replay the original guardian signatures (which are public on-chain data) through a new secp256k1 instruction, populate the new `SignatureSet` with the same `message_hash` as a blocklisted set, and call `post_vaa` — bypassing the blocklist entirely.

---

### Finding Description

**Step 1 — Blocklist check is keypair-address-only.**

In `PostVaa::constraints`, the only guard against blocked VAAs is:

```rust
require!(
    !INVALID_SIGNATURE_SET_KEYS.contains(&signature_set.key().to_string().as_str()),
    CoreBridgeError::InvalidSignatureSet
);
``` [1](#0-0) 

This checks the `SignatureSet` account's **public key**, not its `message_hash` field. There is no check that the `message_hash` stored in the `SignatureSet` is itself blocked.

**Step 2 — `SignatureSet` is a free-keypair account, not a PDA.**

The `signature_set` account in `VerifySignatures` is declared with `init_if_needed` and no `seeds` constraint:

```rust
#[account(
    init_if_needed,
    payer = payer,
    space = SignatureSet::compute_size(guardian_set.inner().keys.len())
)]
signature_set: Account<'info, SignatureSet>,
``` [2](#0-1) 

Any caller can generate a fresh Ed25519 keypair and use it as the `signature_set` account. There is no binding between the keypair address and the `message_hash` it will store.

**Step 3 — `verify_signatures` has no replay protection for guardian signatures.**

The `verify_signatures` instruction reads the preceding secp256k1 instruction from the instruction sysvar, extracts the recovered Ethereum pubkeys and the 32-byte message, and writes `true` into `sig_verify_successes` for each matching guardian index:

```rust
for (i, &signer_index) in guardian_indices.iter().enumerate() {
    require!(
        signers.get(i) == guardians.get(signer_index),
        CoreBridgeError::InvalidGuardianKeyRecovery
    );
    signature_set.sig_verify_successes[signer_index] = true;
}
``` [3](#0-2) 

The secp256k1 native program performs cryptographic verification but has **no replay protection**. Guardian signatures embedded in the original VAA bytes are public on-chain data. An attacker can copy them verbatim into a new secp256k1 instruction targeting the same 32-byte message hash.

**Step 4 — `PostedVaaV1` PDA is keyed on `message_hash`, not on `SignatureSet` keypair.**

The `posted_vaa` PDA is derived from:

```rust
seeds = [
    PostedVaaV1::SEED_PREFIX,
    signature_set.message_hash.as_ref()
],
``` [4](#0-3) 

Because the blocklisted signature sets were blocked **before** the `PostedVaaV1` PDA was ever created (that is the purpose of the blocklist), the PDA for the blocked `message_hash` does not exist on-chain. The `init` constraint will succeed when called with a new `SignatureSet` carrying the same hash.

**Step 5 — `upgrade_contract` consumes the `PostedVaaV1` via a `claim` PDA.**

The `upgrade_contract` governance instruction accepts any `PostedVaaV1` account owned by the Core Bridge program and creates a `claim` PDA derived from `[emitter_address, emitter_chain, sequence]` as its replay guard:

```rust
utils::vaa::claim_vaa(
    CpiContext::new(...),
    &crate::ID,
    &vaa,
    None,
)?;
``` [5](#0-4) 

If the `claim` PDA for the blocked VAA's `(emitter_address, emitter_chain, sequence)` does not yet exist (it was never created because `post_vaa` was always blocked), the governance instruction will execute successfully.

---

### Impact Explanation

A governance VAA that was blocked by the 16-entry keypair blocklist (e.g., a malicious contract upgrade payload) can be re-posted via a fresh `SignatureSet` keypair. Once the `PostedVaaV1` PDA exists, `upgrade_contract` can be called, invoking the BPF Loader Upgradeable program to replace the Core Bridge executable with an attacker-controlled buffer. This constitutes full unauthorized control of the Core Bridge program on Solana.

---

### Likelihood Explanation

**Prerequisites:**
1. The guardian set that signed the original blocked VAA must still be active (not expired). Guardian set rotations are infrequent; the blocklist was added as a patch, making it likely the same guardian set is still active.
2. The original VAA bytes (including guardian signatures) must be available. They are public on-chain data from the original relay attempt.
3. The `PostedVaaV1` PDA for the blocked `message_hash` must not already exist. By definition, the blocklist prevents this PDA from being created, so this condition holds.

All three prerequisites are plausible for the blocklisted VAAs. The attack requires no privileged access, no leaked keys, and no off-chain coordination beyond reading public on-chain data.

---

### Recommendation

Replace the keypair-address blocklist with a **message-hash blocklist**. The check should be:

```rust
const INVALID_MESSAGE_HASHES: [MessageHash; N] = [ /* hashes of blocked VAA bodies */ ];
require!(
    !INVALID_MESSAGE_HASHES.contains(&signature_set.message_hash),
    CoreBridgeError::InvalidSignatureSet
);
```

Alternatively, derive `SignatureSet` as a PDA from the `message_hash` so that only one `SignatureSet` can ever exist per message hash, making the blocklist address-stable. Either fix closes the bypass because the invariant becomes: "this message hash is blocked, regardless of which keypair carries it."

---

### Proof of Concept

**State test (local validator):**

1. Obtain the original blocked VAA bytes for any of the 16 blocklisted signature sets (e.g., `18eK1799CaNMGCUnnCt1Kq2uwKkax6T2WmtrDsZuVFQ`). Extract the guardian signatures and the 32-byte message hash.
2. Generate a fresh Ed25519 keypair `new_sig_set`.
3. Construct a transaction:
   - Instruction 0: secp256k1 program with the original guardian signatures and the same 32-byte message hash.
   - Instruction 1: `verify_signatures` pointing to `new_sig_set`, with `signer_indices` matching the original guardian indices.
4. Submit. Assert `new_sig_set` now has `num_verified() >= quorum`.
5. Call `post_vaa` with `new_sig_set` and the original VAA body fields. Assert the `PostedVaaV1` PDA is created (previously impossible due to the blocklist).
6. Call `upgrade_contract` with the newly created `PostedVaaV1` and an attacker-controlled buffer. Assert the `claim` PDA is created and the BPF Loader Upgradeable `upgrade` CPI is invoked.
