Audit Report

## Title
Type Script `hash_type` Not Validated in `ScriptHashTypeVerifier::verify()`, Allowing Bypass of Non-Contextual Consensus Gate — (File: verification/src/transaction_verifier.rs)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and checks only the **lock script** `hash_type` against `ENABLED_SCRIPT_HASH_TYPE`, never inspecting the **type script** `hash_type`. A transaction output whose lock uses a permitted `hash_type` (e.g., `0x01`) but whose type script carries a valid-but-excluded `hash_type` (e.g., `ScriptHashType::Data3 = 6`) passes the non-contextual gate on every node. During a hard-fork activation window this produces divergent acceptance decisions between upgraded and non-upgraded nodes, constituting a consensus split.

## Finding Description

`ScriptHashTypeVerifier::verify()` at [1](#0-0)  loops over outputs and calls `output.lock().hash_type()` exclusively — `output.type_()` is never consulted. The guarded constant is confirmed at [2](#0-1)  as `{0, 1, 2, 4}` (Data, Type, Data1, Data2).

`ScriptHashType::Data3` is a valid enum variant generated by the `seq!` macro at [3](#0-2)  with value `3 << 1 = 6`, which is **not** in `ENABLED_SCRIPT_HASH_TYPE`. A transaction with:

- lock `hash_type = 0x01` (Type — permitted)
- type script `hash_type = 0x06` (Data3 — excluded)

passes `ScriptHashTypeVerifier::verify()` with `Ok(())` and is forwarded to contextual verification.

The struct comment at [4](#0-3)  reads *"Check whether output lock hash type within enabled range"* — confirming the type script was never considered by design.

Existing tests at [5](#0-4)  (`test_unknown_hash_type_output_lock`, `test_not_enabled_hash_type_output_lock`) cover only the lock-script path; no test exercises the type-script field.

## Impact Explanation

**Critical (15001–25000 points) — Consensus deviation.**

During a CKB hard-fork activation window (as occurred for v2021 and v2023):

1. Upgraded nodes add the new hash type to `ENABLED_SCRIPT_HASH_TYPE` and handle it in contextual script execution.
2. A crafted transaction with a valid lock `hash_type` and a type script carrying the newly-activated `hash_type` passes `ScriptHashTypeVerifier` on **all** nodes (upgraded and non-upgraded alike) because type scripts are never checked.
3. Upgraded nodes execute the type script successfully and accept the transaction; non-upgraded nodes encounter an unrecognised hash type at contextual verification and reject it.
4. The two populations reach different chain states — a consensus split.

Even outside a fork window, the bug forces every node to run the full contextual verification pipeline (script group construction, VM setup) for transactions that should be rejected at the cheap non-contextual gate, enabling low-cost resource exhaustion matching **High — network congestion with few costs**.

## Likelihood Explanation

No special key material, miner cooperation, or privileged access is required. Any unprivileged caller of the `send_transaction` RPC can submit a transaction with an arbitrary `hash_type` byte in a type script field. The attack path is fully reachable on every node that processes externally submitted transactions. During any hard-fork transition — a recurring event in CKB's history — the consensus-split variant becomes immediately exploitable by any observer with no preconditions beyond network access.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to apply the identical `ENABLED_SCRIPT_HASH_TYPE` check to the type script when present, inside the same output loop at [1](#0-0) :

```rust
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        }).into());
    }
}
```

Update the struct comment at [4](#0-3)  to reflect both lock and type script coverage. Add unit tests mirroring `test_unknown_hash_type_output_lock` and `test_not_enabled_hash_type_output_lock` at [5](#0-4)  but targeting the type-script field.

## Proof of Concept

Directly reproducible as a unit test in `verification/src/tests/transaction_verifier.rs`:

```rust
#[test]
pub fn test_not_enabled_hash_type_output_type_script() {
    let transaction = TransactionBuilder::default()
        .output(
            CellOutput::new_builder()
                .lock(Script::default().as_builder().hash_type(1u8).build()) // Type — permitted
                .type_(
                    Some(
                        Script::default()
                            .as_builder()
                            .hash_type(ScriptHashType::Data3) // 0x06 — NOT in ENABLED_SCRIPT_HASH_TYPE
                            .build(),
                    )
                    .pack(),
                )
                .build(),
        )
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);

    // Currently returns Ok(()); should return ScriptHashTypeNotPermitted
    assert_error_eq!(
        verifier.verify().unwrap_err(),
        TransactionError::ScriptHashTypeNotPermitted {
            hash_type: ScriptHashType::Data3.into(),
        },
    );
}
```

Step 3 of the PoC — `verifier.verify()` returning `Ok(())` — is directly confirmed by the code at [1](#0-0) , which contains no branch that reads `output.type_()`.

    /// when the low 1 bit is 0, it indicates the data,
