### Title
Subscription Balance Permanently Stuck if Manager Is a Smart Contract Without ETH Receive Capability — (File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol)

---

### Summary

`Scheduler.sol::withdrawFunds()` always transfers ETH to `msg.sender` (the subscription manager). If the manager is a smart contract without a `receive()` or `fallback()` function, every withdrawal attempt reverts permanently, locking the subscription balance with no recovery path.

---

### Finding Description

The `withdrawFunds()` function is gated by the `onlyManager` modifier and unconditionally sends ETH to `msg.sender`:

```solidity
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount
) external override onlyManager(subscriptionId) {
    ...
    status.balanceInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
    require(sent, "Failed to send funds");
}
``` [1](#0-0) 

The subscription manager is permanently fixed to `msg.sender` at creation time:

```solidity
_state.subscriptionManager[subscriptionId] = msg.sender;
``` [2](#0-1) 

The `onlyManager` modifier enforces that only this exact address can call `withdrawFunds()`:

```solidity
modifier onlyManager(uint256 subscriptionId) {
    if (_state.subscriptionManager[subscriptionId] != msg.sender) {
        revert SchedulerErrors.Unauthorized();
    }
    _;
}
``` [3](#0-2) 

If the manager is a smart contract (e.g., a DAO, multisig, or DeFi protocol) that does not implement `receive()` or `fallback()`, the low-level `.call{value: amount}("")` will return `sent = false`, causing the `require` to revert. Since there is no alternative withdrawal path — no "fee manager" escape hatch, no `recipient` parameter, no admin override — the subscription balance is permanently locked.

This contrasts with `Entropy.sol`, which provides `withdrawAsFeeManager(address provider, uint128 amount)` as an escape hatch that sends ETH to a separately-configured fee manager address rather than the provider itself: [4](#0-3) 

`Scheduler.sol` has no equivalent mechanism.

---

### Impact Explanation

User-deposited ETH held in `status.balanceInWei` is permanently unrecoverable. The manager cannot withdraw their own funds, and no other address (including the Scheduler admin) can withdraw on their behalf. The balance accumulates as dead weight in the contract.

---

### Likelihood Explanation

Smart contracts routinely create Scheduler subscriptions — DeFi protocols, DAOs, multisigs, and keeper bots are all natural subscription managers. Many such contracts omit `receive()` functions by design. This is a realistic, non-exotic scenario that requires no privileged access or external oracle manipulation to trigger: any unprivileged smart contract that calls `createSubscription()` and lacks ETH receive capability is affected.

---

### Recommendation

Add an explicit `recipient` parameter to `withdrawFunds()` so the manager can direct ETH to any address capable of receiving it:

```solidity
function withdrawFunds(
    uint256 subscriptionId,
    uint256 amount,
    address payable recipient
) external override onlyManager(subscriptionId) {
    ...
    status.balanceInWei -= amount;
    (bool sent, ) = recipient.call{value: amount}("");
    require(sent, "Failed to send funds");
}
```

Alternatively, mirror the `Entropy.sol` pattern and allow the manager to register a separate withdrawal address.

---

### Proof of Concept

1. Deploy `ManagerContract` — a smart contract with no `receive()` or `fallback()` function.
2. From `ManagerContract`, call `createSubscription{value: minimumBalance}(params)`. The subscription is created; `_state.subscriptionManager[id] = address(ManagerContract)`.
3. From `ManagerContract`, call `withdrawFunds(id, amount)`.
4. Inside `withdrawFunds`, `(bool sent, ) = msg.sender.call{value: amount}("")` executes. Because `ManagerContract` has no `receive()`, the call returns `sent = false`.
5. `require(sent, "Failed to send funds")` reverts.
6. The subscription balance remains locked forever; no other address can call `withdrawFunds` because `onlyManager` blocks them. [5](#0-4)

    function withdrawFunds(
        uint256 subscriptionId,
        uint256 amount
    ) external override onlyManager(subscriptionId) {
        SchedulerStructs.SubscriptionStatus storage status = _state
            .subscriptionStatuses[subscriptionId];
        SchedulerStructs.SubscriptionParams storage params = _state
            .subscriptionParams[subscriptionId];

        // Prevent withdrawals from permanent subscriptions
        if (params.isPermanent) {
            revert SchedulerErrors.CannotUpdatePermanentSubscription();
        }

        if (status.balanceInWei < amount) {
            revert SchedulerErrors.InsufficientBalance();
        }

        // If subscription is active, ensure minimum balance is maintained
        if (params.isActive) {
            uint256 minimumBalance = this.getMinimumBalance(
                uint8(params.priceIds.length)
            );
            if (status.balanceInWei - amount < minimumBalance) {
                revert SchedulerErrors.InsufficientBalance();
            }
        }

        status.balanceInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send funds");

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer

        require(sent, "withdrawal to msg.sender failed");
