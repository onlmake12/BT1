### Title
Unbounded `firstUnfulfilledSeq` Scan Loop in `executeCallback` Enables Gas-Exhaustion DoS — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.sol::executeCallback` contains an unbounded `while` loop that scans forward through all cleared sequence numbers to advance `_state.firstUnfulfilledSeq`. An unprivileged attacker can create a large number of requests, fulfill all of them except the lowest-numbered one, and then cause the provider's `executeCallback` call for that lowest request to permanently revert out-of-gas. This permanently prevents the provider from fulfilling that request, locks the provider's fee for it, and breaks the `firstUnfulfilledSeq` pointer for all future `getFirstActiveRequests` calls.

---

### Finding Description

In `Echo.sol`, after clearing a fulfilled request, `executeCallback` runs this loop:

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest(currentSeq)`, which performs two storage reads:

1. A read of `_state.requests[shortKey]` (warm after the first 32 unique slots, 100 gas each).
2. If the main-array slot does not match the sequence number, a read of `_state.requestsOverflow[key]` — a unique `bytes32` key per sequence number, always a **cold SLOAD at 2,100 gas**. [2](#0-1) 

The fixed-size main array has only `NUM_REQUESTS = 32` slots. [3](#0-2) 

For any sequence number whose cleared slot in the main array was previously occupied by a different request (overflowed), `findRequest` will also read the overflow mapping — a cold SLOAD. With N cleared sequence numbers between `firstUnfulfilledSeq` and `currentSequenceNumber`, the loop costs approximately **N × 2,200 gas** (100 warm + 2,100 cold per iteration). At N ≈ 13,000 this exceeds Ethereum's 30 M block gas limit when combined with the rest of the transaction.

The developers acknowledged this in a TODO comment directly above the loop:

> "I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number. A better solution would be a doubly-linked list of active requests." [4](#0-3) 

---

### Impact Explanation

- The provider's `executeCallback` for the lowest-sequence-number request permanently reverts out-of-gas; the entire transaction (including fee credit and `clearRequest`) rolls back.
- The provider can **never** collect the fee locked in that request.
- `_state.firstUnfulfilledSeq` is permanently stuck, so every future call to `getFirstActiveRequests` starts scanning from the stuck pointer, degrading the off-chain keeper's ability to discover pending requests. [5](#0-4) 

The `getFirstActiveRequests` function is the primary mechanism used by the off-chain executor (Fortuna keeper) to discover requests to fulfill. [6](#0-5) 

---

### Likelihood Explanation

- Any unprivileged address can call `requestPriceUpdatesWithCallback` and `executeCallback` (after the exclusivity period, anyone may call `executeCallback` and credit any provider).
- The attacker pays fees for N requests but can recoup most of them by calling `executeCallback` for requests seq 2 through N themselves after the exclusivity window, crediting their own address as `providerToCredit`.
- The exclusivity period check only requires `providerToCredit == req.provider` during the window; after it expires, any caller may fulfill. [7](#0-6) 

The net cost to the attacker is the fee for a single request (seq 1) plus gas for N transactions, while the impact is permanent DoS of the provider's ability to fulfill seq 1 and degraded keeper operation.

---

### Recommendation

1. **Remove the unbounded scan loop.** Replace `firstUnfulfilledSeq` advancement with a doubly-linked list of active requests (as the TODO comment already suggests), or simply remove the pointer entirely and rely on event-log scanning in the off-chain keeper.
2. **Cap the loop.** If the pointer is kept, add a hard cap on the number of iterations per call (e.g., `MAX_SCAN = 256`) so the gas cost is bounded.
3. **Move `firstUnfulfilledSeq` advancement off-chain.** The keeper already scans events; the on-chain pointer is an optimization that introduces this vulnerability.

---

### Proof of Concept

```
Setup:
  - Echo contract deployed with defaultProvider P
  - Attacker A registers as a provider (or uses P)
  - Block gas limit: 30,000,000

Step 1: Attacker creates 14,000 requests (seq 1 – 14,000)
  for i in 1..14000:
    echo.requestPriceUpdatesWithCallback{value: fee}(P, now, priceIds, gasLimit)

Step 2: After exclusivity period expires, attacker calls executeCallback
  for seq in 2..14000:
    echo.executeCallback(attackerAddress, seq, updateData, priceIds)
  // Each call: while loop stops immediately (seq 1 still active), 0 iterations

Step 3: Provider P calls executeCallback for seq 1:
  echo.executeCallback(P, 1, updateData, priceIds)
  // clearRequest(1) marks seq 1 inactive
  // while loop: iterates from firstUnfulfilledSeq=1 to 14001
  //   ~14,000 iterations × 2,200 gas ≈ 30,800,000 gas → OUT OF GAS → REVERT
  // Transaction reverts; fee for seq 1 remains locked; firstUnfulfilledSeq stays at 1

Result:
  - Provider P can never fulfill seq 1 (always OOG)
  - getFirstActiveRequests always starts scanning from seq 1
  - Off-chain keeper's request discovery is permanently degraded
```

    function getFirstActiveRequests(
        uint256 count
    )
        external
        view
        override
        returns (Request[] memory requests, uint256 actualCount)
    {
        requests = new Request[](count);
        actualCount = 0;

        // Start from the first unfulfilled sequence and work forwards
        uint64 currentSeq = _state.firstUnfulfilledSeq;

        // Continue until we find enough active requests or reach current sequence
        while (
            actualCount < count && currentSeq < _state.currentSequenceNumber
        ) {
            Request memory req = findRequest(currentSeq);
            if (isActive(req)) {
                requests[actualCount] = req;
                actualCount++;
            }
            currentSeq++;
        }

        // If we found fewer requests than asked for, resize the array
        if (actualCount < count) {
            assembly {
                mstore(requests, actualCount)
            }
        }
    }
```
