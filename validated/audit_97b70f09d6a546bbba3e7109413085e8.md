### Title
Entropy Provider Address Is Permanently Bound With No Migration Mechanism, Causing Permanent Fee Lock If Provider Key Becomes Inaccessible — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The Entropy contract permanently binds a provider's entire registration — including all accrued fees — to the `msg.sender` address used at registration time. There is no function to transfer or migrate a provider identity to a new address. If a provider's key is lost or their address becomes non-functional and no `feeManager` was proactively set, all `accruedFeesInWei` stored in the contract are permanently locked with no recovery path.

---

### Finding Description

In `Entropy.sol`, `register()` stores the provider's `ProviderInfo` struct at `_state.providers[msg.sender]`:

```solidity
EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
```

Every subsequent provider management function — `withdraw`, `setProviderFee`, `setProviderUri`, `setFeeManager`, `setMaxNumHashes`, `setDefaultGasLimit` — also resolves the provider exclusively via `msg.sender`. There is no function in the contract that allows a provider to migrate their registration (including `accruedFeesInWei`, `sequenceNumber`, `currentCommitment`, etc.) to a new address.

The `feeManager` mechanism (`setFeeManager`) partially mitigates the fee-withdrawal concern by allowing a separate address to call `withdrawAsFeeManager`. However:

1. `feeManager` is **optional** — it defaults to `address(0)` and must be set proactively.
2. `withdrawAsFeeManager` sends ETH to `msg.sender` (the fee manager), not to the provider address, so it only helps if the fee manager was set **before** the provider key became inaccessible.
3. The fee manager cannot rotate commitments, fulfill requests, or perform any other provider-identity operations.

The `withdraw` function sends ETH directly to `msg.sender` (the provider address):

```solidity
(bool sent, ) = msg.sender.call{value: amount}("");
require(sent, "withdrawal to msg.sender failed");
```

If the provider address is a smart contract that becomes non-functional (e.g., a multisig that loses quorum, a proxy whose implementation is bricked), or if the EOA private key is lost, and no `feeManager` was set, the `accruedFeesInWei` balance is permanently locked.

---

### Impact Explanation

- **Permanent loss of accrued provider fees**: `accruedFeesInWei` in `ProviderInfo` is irrecoverable if the provider address is inaccessible and no `feeManager` was set.
- **Inability to fulfill pending requests**: In-flight requests assigned to the provider's sequence numbers cannot be revealed from a new address, breaking the randomness service for those users.
- **No commitment rotation from a new address**: The provider cannot advance `currentCommitment` or extend `endSequenceNumber` from a new address, causing `OutOfRandomness` reverts for all future requests to that provider.

---

### Likelihood Explanation

Providers are not required to be EOAs. Any address — including a smart contract wallet (multisig, proxy, or DAO-controlled contract) — can call `register()`. Real-world scenarios where the provider address becomes inaccessible include:

- A multisig provider loses quorum (signers leave, keys are lost).
- A proxy-based provider contract is upgraded to a broken implementation.
- An EOA provider loses their private key.

The `feeManager` field is optional and defaults to `address(0)`. Providers who do not proactively configure it (which is the default state) are fully exposed. The Fortuna keeper service itself uses a private key (`provider_config.private_key`), making key loss a realistic scenario.

---

### Recommendation

1. **Add a provider address migration function** that allows the current provider address to designate a new provider address, transferring the full `ProviderInfo` (including `accruedFeesInWei`) to the new key.
2. **Alternatively**, require providers to set a non-zero `feeManager` before they can accrue fees, ensuring a recovery path always exists.
3. **At minimum**, document that providers MUST set a `feeManager` to protect against key loss, and consider emitting a warning event when fees accrue on a provider with `feeManager == address(0)`.

---

### Proof of Concept

1. Provider registers at address `A`:
   ```solidity
   // msg.sender = A
   entropy.register(fee, commitment, metadata, chainLength, uri);
   ```
2. Users request randomness from provider `A`; `accruedFeesInWei` grows to a significant value.
3. Provider does **not** call `setFeeManager(...)` — `feeManager` remains `address(0)`.
4. Provider's key for address `A` is lost (or the smart contract at `A` becomes non-functional).
5. `withdraw(amount)` requires `msg.sender == A` → permanently inaccessible.
6. `withdrawAsFeeManager(A, amount)` checks `providerInfo.feeManager != msg.sender` → `address(0) != msg.sender` → reverts with `Unauthorized`.
7. All `accruedFeesInWei` for provider `A` are permanently locked in the contract with no admin recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

    function register(
        uint128 feeInWei,
        bytes32 commitment,
        bytes calldata commitmentMetadata,
        uint64 chainLength,
        bytes calldata uri
    ) public override {
        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.
        // Specifically, accruedFeesInWei is intentionally not set. On initial registration, it will be zero,
        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0
        // on initial registration.

        provider.feeInWei = feeInWei;

        provider.originalCommitment = commitment;
        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
        provider.currentCommitment = commitment;
        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

        emit EntropyEvents.Registered(
            EntropyStructConverter.toV1ProviderInfo(provider)
        );
        emit EntropyEventsV2.Registered(msg.sender, bytes(""));
    }

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            provider,
            msg.sender,
            amount,
            bytes("")
        );
    }

    function setFeeManager(address manager) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        address oldFeeManager = provider.feeManager;
        provider.feeManager = manager;
        emit ProviderFeeManagerUpdated(msg.sender, oldFeeManager, manager);
        emit EntropyEventsV2.ProviderFeeManagerUpdated(
            msg.sender,
            oldFeeManager,
            manager,
            bytes("")
        );
    }

        uint128 feeInWei;
        uint128 accruedFeesInWei;
        // The commitment that the provider posted to the blockchain, and the sequence number

        // and instead is stored to help providers track where they are in the hash chain.
        bytes32 originalCommitment;
        uint64 originalCommitmentSequenceNumber;
        // Metadata for the current commitment. Providers may optionally use this field to help
        // manage rotations (i.e., to pick the sequence number from the correct hash chain).
        bytes commitmentMetadata;
        // Optional URI where clients can retrieve revelations for the provider.
        // Client SDKs can use this field to automatically determine how to retrieve random values for each provider.
        // TODO: specify the API that must be implemented at this URI
        bytes uri;
        // The first sequence number that is *not* included in the current commitment (i.e., an exclusive end index).
        // The contract maintains the invariant that sequenceNumber <= endSequenceNumber.
        // If sequenceNumber == endSequenceNumber, the provider must rotate their commitment to add additional random values.
        uint64 endSequenceNumber;
        // The sequence number that will be assigned to the next inbound user request.
        uint64 sequenceNumber;
        // The current commitment represents an index/value in the provider's hash chain.
        // These values are used to verify requests for future sequence numbers. Note that
        // currentCommitmentSequenceNumber < sequenceNumber.

        // The currentCommitment advances forward through the provider's hash chain as values
        // are revealed on-chain.
        bytes32 currentCommitment;
        uint64 currentCommitmentSequenceNumber;
        // An address that is authorized to set / withdraw fees on behalf of this provider.
        address feeManager;
        // Maximum number of hashes to record in a request. This should be set according to the maximum gas limit
        // the provider supports for callbacks.
        uint32 maxNumHashes;
        // Default gas limit to use for callbacks.
        uint32 defaultGasLimit;
    }
