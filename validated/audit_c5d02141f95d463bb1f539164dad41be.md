### Title
Division-by-Zero in `calculateTwap` When Start and End Share the Same `publishSlot` - (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

---

### Summary

`Pyth::calculateTwap` divides by `slotDiff` (the difference between end and start `publishSlot`) in three places. The upstream validation function `validateTwapPriceInfo` only rejects the case where `start.publishSlot > end.publishSlot`, but **does not reject the equal case** (`start.publishSlot == end.publishSlot`). When both slots are equal, `slotDiff = 0` and all three divisions panic, causing `parseTwapPriceFeedUpdates` to revert for any caller who submits such a pair.

---

### Finding Description

`validateTwapPriceInfo` guards the slot ordering with a non-strict inequality:

```solidity
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

This allows `start.publishSlot == end.publishSlot` to pass validation. `calculateTwap` then computes:

```solidity
uint64 slotDiff = twapPriceInfoEnd.publishSlot - twapPriceInfoStart.publishSlot;
// slotDiff == 0 when slots are equal
int128 twapPrice = priceDiff / int128(uint128(slotDiff));   // division by zero
uint128 twapConf  = confDiff / uint128(slotDiff);           // division by zero
uint64 downSlotsRatio = (totalDownSlots * 1_000_000) / slotDiff; // division by zero
``` [2](#0-1) 

All three divisions will trigger a Solidity arithmetic panic (division by zero), reverting the entire `parseTwapPriceFeedUpdates` call.

By contrast, the Solana receiver correctly uses a **strict** greater-than check:

```rust
require!(
    end_msg.publish_slot > start_msg.publish_slot,
    ReceiverError::InvalidTwapSlots
);
``` [3](#0-2) 

The EVM implementation is inconsistent with the Solana implementation and is missing this guard.

---

### Impact Explanation

Any call to `parseTwapPriceFeedUpdates` where the start and end `TwapPriceInfo` share the same `publishSlot` will revert with a division-by-zero panic. This is a denial-of-service on the TWAP price feed calculation for EVM chains. Protocols relying on `parseTwapPriceFeedUpdates` for on-chain TWAP pricing (e.g., for liquidations, AMM pricing, or risk management) will be unable to obtain a TWAP result for that slot pair. [4](#0-3) 

---

### Likelihood Explanation

The entry path is fully unprivileged: any transaction sender can call `parseTwapPriceFeedUpdates` with two Wormhole-attested TWAP update blobs. The attested data is sourced from Hermes, which constructs TWAP messages from on-chain Pythnet slots. If a caller (or a relayer) submits two valid, Wormhole-verified TWAP messages that happen to carry the same `publishSlot` value — which is a valid on-chain state — the contract will revert. No key compromise, governance manipulation, or oracle lying is required; the attacker only needs to find or craft two valid TWAP messages with identical `publishSlot` fields. [5](#0-4) 

---

### Recommendation

Change the slot ordering check in `validateTwapPriceInfo` from a strict greater-than to a greater-than-or-equal check, mirroring the Solana implementation:

```solidity
// Before (allows equal slots through):
if (twapPriceInfoStart.publishSlot > twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}

// After (rejects equal slots):
if (twapPriceInfoStart.publishSlot >= twapPriceInfoEnd.publishSlot) {
    revert PythErrors.InvalidTwapUpdateDataSet();
}
``` [1](#0-0) 

---

### Proof of Concept

1. Obtain two valid Wormhole-attested TWAP update blobs for the same price feed where both blobs encode `publishSlot = S` for any slot `S`.
2. Call `parseTwapPriceFeedUpdates{value: fee}([startBlob, endBlob], [priceId])`.
3. `validateTwapPriceInfo` passes because `S > S` is `false`.
4. `calculateTwap` computes `slotDiff = S - S = 0`.
5. The line `int128 twapPrice = priceDiff / int128(uint128(0))` triggers a Solidity arithmetic panic, reverting the transaction. [6](#0-5)
