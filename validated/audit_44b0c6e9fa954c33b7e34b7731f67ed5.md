### Title
Lazer Signed Payload Lacks Domain Separator, Enabling Cross-Chain Replay — (`lazer/contracts/evm/src/PythLazer.sol`)

### Summary
`PythLazer.verifyUpdate()` signs and verifies only `keccak256(payload)` — the raw Lazer payload bytes — with no chain ID, contract address, or any other domain context folded in. Because the same trusted signer key and the same contract address are deployed across multiple EVM chains, any valid signed update from chain A can be replayed verbatim on chain B.

### Finding Description

In `PythLazer.verifyUpdate()`, the digest used for ECDSA recovery is constructed as:

```solidity
bytes32 hash = keccak256(payload);
(signer, , ) = ECDSA.tryRecover(
    hash,
    uint8(update[68]) + 27,
    bytes32(update[4:36]),
    bytes32(update[36:68])
);
``` [1](#0-0) 

The `payload` is the raw Lazer price feed bytes (magic, timestamp, channel, feed data). No chain ID, no `address(this)`, and no other domain-binding field is included in the hash. The verification then checks only that the recovered signer is in the trusted-signer registry:

```solidity
if (!isValidSigner(signer)) {
    revert("invalid signer");
}
``` [2](#0-1) 

The deployment script explicitly uses `uint8(0)` for the crosschain replay protection byte in the CREATE2 salt, meaning the contract is intentionally deployed at the **same address** on every chain:

```solidity
salt = bytes32(
    abi.encodePacked(
        deployer,
        uint8(0), // No crosschain replay protection
        seed
    )
);
``` [3](#0-2) 

The same trusted signer address is registered on all chains via `updateTrustedSigner`. Because `isValidSigner` only checks the signer's expiry timestamp:

```solidity
function isValidSigner(address signer) public view returns (bool) {
    return block.timestamp < trustedSignerToExpiresAtMapping[signer];
}
``` [4](#0-3) 

…it will pass on any chain where the signer is registered, for any payload that was legitimately signed anywhere.

### Impact Explanation

Any unprivileged attacker who observes a valid Lazer price update on chain A (e.g., Ethereum mainnet) can submit the identical `update` bytes to `verifyUpdate` on chain B (e.g., Arbitrum, Base, BNB Chain). The function will return `(payload, signer)` as if the update were freshly signed for chain B. Consumer contracts that call `verifyUpdate` and act on the returned payload will accept the replayed data. Concretely:

- **Stale price injection**: An attacker replays an old signed update (e.g., from a period of high volatility) on a chain where the current price has moved, causing downstream protocols (lending, derivatives, perpetuals) to use an incorrect price.
- **Same-chain replay**: Because `verifyUpdate` keeps no nonce or sequence-number state, the same update can also be submitted multiple times on the same chain, repeatedly triggering consumer callbacks with stale data.

### Likelihood Explanation

- The `update` bytes are broadcast on-chain and trivially observable by any party.
- The contract is live on multiple EVM chains with the same trusted signer.
- No special access or key material is required; the attacker only needs to copy calldata from one chain's transaction and submit it on another.
- The Lazer payload format includes a `timestamp` field parsed by `parsePayloadHeader`, but `verifyUpdate` itself does not enforce any freshness bound — that is left entirely to the consumer, which may not implement it correctly. [5](#0-4) 

### Recommendation

Introduce an EIP-712 domain separator into the signed digest, committing at minimum to `chainid` and `verifyingContract`:

```solidity
bytes32 domainSeparator = keccak256(abi.encode(
    keccak256("EIP712Domain(string name,uint256 chainId,address verifyingContract)"),
    keccak256("PythLazer"),
    block.chainid,
    address(this)
));
bytes32 hash = keccak256(abi.encodePacked("\x19\x01", domainSeparator, keccak256(payload)));
```

This ensures a signature produced for chain A cannot be accepted on chain B, and a signature produced for one contract instance cannot be accepted by another.

### Proof of Concept

1. On Ethereum mainnet, a Pyth Lazer updater submits a valid `update` blob to `PythLazer.verifyUpdate{value: fee}(update)`. The transaction is public.
2. Attacker copies the `update` calldata argument verbatim.
3. Attacker calls `PythLazer.verifyUpdate{value: fee}(update)` on Arbitrum (same contract address, same trusted signer registered).
4. `keccak256(payload)` produces the identical hash; `ECDSA.tryRecover` returns the same signer address; `isValidSigner` returns `true`.
5. `verifyUpdate` returns `(payload, signer)` — the Arbitrum consumer contract accepts the Ethereum-originated price data as if it were a fresh Arbitrum update.
6. Repeat with any historical update to inject arbitrarily stale prices.
