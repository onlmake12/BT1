### Title
Unbounded Peer Store Poisoning via Repeated `announce=true` Discovery Nodes Messages — (`network/src/protocols/discovery/mod.rs`)

### Summary

There is no per-session rate limit on inbound `announce=true` `Nodes` messages. An unprivileged remote peer can send an unlimited number of such messages, each carrying exactly `ANNOUNCE_THRESHOLD` (10) items with `MAX_ADDRS` (3) addresses, injecting 30 addresses per message into the peer store. Because `announce=true` never sets `received_nodes`, the one-per-session guard does not apply. A single attacker connection can fill the 16,384-entry peer store with attacker-controlled addresses, evicting legitimate peers and potentially causing all subsequent `add_addr` calls to return `EvictionFailed`.

---

### Finding Description

**`verify_nodes_message` allows exactly-at-threshold messages indefinitely**

`verify_nodes_message` uses strict greater-than comparisons: [1](#0-0) [2](#0-1) 

A message with exactly 10 items and 3 addresses each returns `None` (no misbehavior), so `misbehave` is never called and no disconnect occurs.

**`received_nodes` guard is bypassed by `announce=true`**

The one-per-session guard only applies to `announce=false`: [3](#0-2) 

`announce=true` messages skip both the check and the flag assignment entirely. There is no counter, timestamp, or token-bucket anywhere in the session state for announce messages.

**All addresses are forwarded to the peer store unconditionally**

After passing `verify_nodes_message`, addresses are passed directly to `add_new_addrs`, which calls `peer_store.add_addr` for every address that passes the `is_valid_addr` reachability filter: [4](#0-3) 

Errors from `add_addr` are only logged at `debug` level — no disconnect, no misbehavior score, no backpressure.

**Peer store eviction can fail, leaving the store permanently poisoned**

The peer store is capped at `ADDR_COUNT_LIMIT = 16384`: [5](#0-4) 

`check_purge` first tries to evict non-connectable entries, then falls back to evicting 2 peers from network groups with >4 members: [6](#0-5) 

If the attacker uses addresses spread across many distinct /16 subnets (each group ≤ 4 entries), neither eviction path succeeds and `check_purge` returns `Err(PeerStoreError::EvictionFailed)`. This error propagates to `add_addr`, which is silently swallowed by the discovery layer. The store remains full of attacker-controlled addresses.

---

### Impact Explanation

- **Peer store poisoning**: The 16,384-slot peer store is filled with attacker-controlled addresses, displacing legitimate peers.
- **Peer discovery degradation**: The victim node's future `GetNodes` responses and feeler-dial candidates are drawn from the poisoned store, steering it toward attacker-controlled endpoints.
- **Eviction failure**: With diverse-subnet addresses, `EvictionFailed` is returned on every subsequent `add_addr` call, permanently blocking new legitimate peer discovery for the lifetime of the store.

---

### Likelihood Explanation

The attack requires only a single inbound TCP connection and the ability to send well-formed protobuf messages. No PoW, no key material, no privileged role. Crafting valid-looking public IP multiaddrs is trivial. ~547 messages (16384 ÷ 30) suffice to fill the store. The `addr_known` per-session deduplication set only prevents the same address from being re-announced to others; it does not filter addresses before they reach `add_new_addrs`.

---

### Recommendation

1. **Add a per-session announce rate limit**: track the number of `announce=true` messages received per session and disconnect (or score-penalize) after a small threshold (e.g., 1 per minute).
2. **Count total addresses, not just items**: the real injection rate is `items × addresses_per_item`; rate-limit on total address count per session per time window.
3. **Apply misbehavior scoring on repeated announces**: rather than only disconnecting on structural violations, increment a score counter for each announce message and trigger `misbehave` after N messages.
4. **Harden eviction**: when `EvictionFailed` is returned, log at `warn`/`error` and consider disconnecting the session that triggered the overflow.

---

### Proof of Concept

```
1. Connect to a CKB node on the discovery protocol.
2. In a loop (1000 iterations):
   a. Construct a Nodes message: announce=true, 10 items,
      each item containing 3 unique public IP multiaddrs
      (e.g., /ip4/1.2.3.X/tcp/8115/p2p/<random_peer_id>).
   b. Send the encoded message.
3. After ~547 iterations, assert peer_store.addr_manager.count() == 16384.
4. Observe that no disconnect occurred and all subsequent add_addr
   calls return EvictionFailed (logged at debug level only).
5. Verify that fetch_random_addrs now returns only attacker-controlled addresses.
```

    fn check_purge(&mut self) -> Result<()> {
        if self.addr_manager.count() < ADDR_COUNT_LIMIT {
            return Ok(());
        }

        // Evicting invalid data in the peer store is a relatively rare operation
        // There are certain cleanup strategies here:
        // 1. First evict the nodes that have reached the eviction condition
        // 2. If the first step is unsuccessful, enter the network segment grouping mode
        //  2.1. Group current data according to network segment
        //  2.2. Sort according to the amount of data in the same network segment
        //  2.3. In the network segment with more than 4 peer, randomly evict 2 peer

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let candidate_peers: Vec<_> = self
            .addr_manager
            .addrs_iter()
            .filter_map(|addr| {
                if !addr.is_connectable(now_ms) {
                    Some(addr.addr.clone())
                } else {
                    None
                }
            })
            .collect();

        for key in candidate_peers.iter() {
            self.addr_manager.remove(key);
        }

        if candidate_peers.is_empty() {
            let candidate_peers: Vec<_> = {
                let mut peers_by_network_group: HashMap<Group, Vec<_>> = HashMap::default();
                for addr in self.addr_manager.addrs_iter() {
                    peers_by_network_group
                        .entry((&addr.addr).into())
                        .or_default()
                        .push(addr);
                }
                let len = peers_by_network_group.len();
                let mut peers = peers_by_network_group
                    .drain()
                    .map(|(_, v)| v)
                    .collect::<Vec<Vec<_>>>();

                peers.sort_unstable_by_key(|k| std::cmp::Reverse(k.len()));

                peers
                    .into_iter()
                    .take(len / 2)
                    .flat_map(move |addrs| {
                        if addrs.len() > 4 {
                            Some(
                                addrs
                                    .iter()
                                    .choose_multiple(&mut rand::thread_rng(), 2)
                                    .into_iter()
                                    .map(|addr| addr.addr.clone())
                                    .collect::<Vec<Multiaddr>>(),
                            )
                        } else {
                            None
                        }
                    })
                    .flatten()
                    .collect()
            };

            for key in candidate_peers.iter() {
                self.addr_manager.remove(key);
            }

            if candidate_peers.is_empty() {
                return Err(PeerStoreError::EvictionFailed.into());
            }
