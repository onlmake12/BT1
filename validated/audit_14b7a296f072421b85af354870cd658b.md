### Title
Permanent Subscription Funds Permanently Locked When Subscribed Price Feed Becomes Stale or Deprecated — (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's `isPermanent` subscription flag creates an irrecoverable locked-funds condition. When any price feed subscribed to a permanent subscription stops being published by Pythnet (deprecated, delisted, or experiencing a prolonged outage), `updatePriceFeeds` hard-reverts with `TimestampTooOld`, while `withdrawFunds` and `updateSubscription` (including deactivation) also unconditionally revert with `CannotUpdatePermanentSubscription`. There is no escape hatch. The subscription balance — up to `MAX_DEPOSIT_LIMIT = 100 ETH` — is permanently locked in the contract.

---

### Finding Description

`Scheduler.sol` implements a `isPermanent` flag for subscriptions. Once set, the flag is irreversible and blocks every state-changing operation on the subscription:

**`updateSubscription` reverts for permanent subscriptions:** [1](#0-0) 

This blocks deactivation (`isActive = false`), price ID changes, and any parameter modification.

**`withdrawFunds` reverts for permanent subscriptions:** [2](#0-1) 

**`updatePriceFeeds` enforces a 1-hour staleness window** via `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`: [3](#0-2) 

If the latest publish timestamp across all subscribed feeds is older than 1 hour, `_validateShouldUpdatePrices` hard-reverts: [4](#0-3) 

Additionally, `parsePriceFeedUpdatesWithConfig` is called with `strict = true`, meaning every subscribed price ID must be present in the update data: [5](#0-4) 

If Pyth stops publishing any one of the subscribed feeds (deprecated feed, delisted asset, prolonged network outage), the keeper cannot construct a valid update batch, and `updatePriceFeeds` will always revert. Because the subscription is permanent, the manager cannot deactivate it or withdraw the balance. The funds are permanently locked.

---

### Impact Explanation

Any ETH deposited into a permanent subscription is permanently locked if any subscribed price feed becomes unavailable. There is no admin override, no emergency withdrawal, and no governance mechanism to recover funds. The `MAX_DEPOSIT_LIMIT` of 100 ETH per subscription bounds the per-subscription loss, but there is no limit on the number of permanent subscriptions, so aggregate losses can be unbounded. This is a direct loss of user funds with no recovery path.

---

### Likelihood Explanation

Pyth has historically deprecated price feeds. The `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` is only 1 hour, meaning even a brief Pythnet outage (network congestion, validator downtime, feed migration) is sufficient to trigger the `TimestampTooOld` revert. Any user who creates a permanent subscription — a feature explicitly supported and tested by the protocol — is exposed to this risk for the lifetime of the subscription. The trigger condition (a feed going stale) is realistic and has precedent.

---

### Recommendation

1. **Add an emergency withdrawal path for permanent subscriptions** that is gated on the subscription being provably un-updatable (e.g., if `updatePriceFeeds` has not succeeded for longer than `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD`), or allow an admin/governance role to force-deactivate a permanent subscription and return funds.
2. **Alternatively**, do not block `withdrawFunds` and deactivation for permanent subscriptions — only block parameter changes (price IDs, update criteria, whitelist). The "permanent" guarantee can be scoped to the subscription's configuration, not its lifecycle.
3. **Short-term**: add a `recoverFunds` function callable by the subscription manager after a configurable staleness timeout, analogous to the Balancer report's recommendation to not hard-revert but fall back gracefully.

---

### Proof of Concept

1. Alice calls `createSubscription{value: 50 ether}(params)` with `isPermanent = true`, subscribing to price feed `0xABC`.
2. Alice calls `updateSubscription(id, params_with_isPermanent_true)` — this succeeds and locks the permanent flag.
3. Pyth deprecates feed `0xABC`. Pythnet stops publishing it.
4. Any keeper attempting `updatePriceFeeds(id, data)` will revert: either `parsePriceFeedUpdatesWithConfig` reverts because `0xABC` is missing from the update data (strict mode), or `_validateShouldUpdatePrices` reverts with `TimestampTooOld` because the latest timestamp in the batch is now > 1 hour old.
5. Alice calls `withdrawFunds(id, 50 ether)` → reverts: `CannotUpdatePermanentSubscription`.
6. Alice calls `updateSubscription(id, params_with_isActive_false)` → reverts: `CannotUpdatePermanentSubscription`.
