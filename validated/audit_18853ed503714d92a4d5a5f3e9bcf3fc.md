### Title
`DaoScriptSizeVerifier` Invocation Skipped on Cache-Hit Path in Tx-Pool Admission — (File: `tx-pool/src/util.rs`)

---

### Summary

In `tx-pool/src/util.rs`, the function `verify_rtx` has three branches. Both cache-miss branches call `DaoScriptSizeVerifier` after the main contextual verifier. The cache-hit branch calls only `TimeRelativeTransactionVerifier` and silently omits `DaoScriptSizeVerifier`. A transaction sender whose transaction witness hash is already present in the verification cache can therefore have a DAO-script-size-violating transaction admitted to the tx-pool without the RFC-0044 size constraint being enforced.

---

The cache-hit branch is structurally parallel to the two cache-miss branches but is missing the `DaoScriptSizeVerifier` call that both cache-miss branches include. This is a skipped function invocation in the tx-pool admission pipeline.

By contrast, the block-level verifier in `verification/contextual/src/contextual_block_verifier.rs` applies `DaoScriptSizeVerifier` unconditionally to **both** cache-hit and cache-miss paths via a shared `.and_then` combinator (lines 444–453), gated only on `rfc0044_active`:

```rust

        DaoScriptSizeVerifier::new(Arc::clone(tx), ...).verify()?;
    }
    Ok(result)
})
.skip(1) // skip cellbase tx
``` [4](#0-3) 

The tx-pool `verify_rtx` has no equivalent guard on the cache-hit branch.

---

### Impact Explanation

A transaction sender can submit a transaction whose witness hash is already present in the tx-pool's `TxVerificationCache`. On re-verification (e.g., after a reorg causes the tx-pool to re-admit pending transactions, or after a snapshot refresh), the cache-hit branch is taken and `DaoScriptSizeVerifier` is never called. A transaction that violates the RFC-0044 DAO script size limit is admitted to the tx-pool without the constraint being enforced.

If such a transaction is selected by the block assembler and included in a block template, the miner who mines that template will produce a block that fails `ContextualBlockVerifier::verify` (which correctly enforces `DaoScriptSizeVerifier` for all transactions). The miner loses the block reward for that round. Additionally, the tx-pool is persistently polluted with an unmineable transaction that will never be cleared by normal admission logic, since re-verification always takes the cache-hit path and always passes.

---

### Likelihood Explanation

The `TxVerificationCache` is an in-memory LRU cache keyed by `witness_hash`. The cache is populated by the cache-miss paths, which do call `DaoScriptSizeVerifier`. However, the cache is populated before RFC-0044 activation (when the block verifier does not yet enforce the constraint), meaning transactions cached in that window carry no `DaoScriptSizeVerifier` result. After RFC-0044 activates, those cached entries are re-used on the cache-hit path, and the now-required `DaoScriptSizeVerifier` check is silently skipped. Any transaction sender who submitted a DAO-output transaction before the hard fork epoch and whose transaction remains in the cache crosses this window. The cache is also shared across reorg-driven re-admissions, making the window wider than a single submission event.

---

### Recommendation

Add `DaoScriptSizeVerifier` to the cache-hit branch of `verify_rtx`, consistent with the two cache-miss branches. The check should be gated on the same `rfc0044_active` epoch condition used in the block verifier, or applied unconditionally (matching the current cache-miss behavior in `verify_rtx`):

```rust
if let Some(completed) = cache_entry {
    TimeRelativeTransactionVerifier::new(Arc::clone(&rtx), consensus.clone(), data_loader.clone(), Arc::clone(&tx_env))
        .verify()
        .map_err(Reject::Verification)?;
    DaoScriptSizeVerifier::new(rtx, consensus, data_loader)
        .verify()
        .map_err(Reject::Verification)?;
    Ok(*completed)
```

---

### Proof of Concept

1. Before RFC-0044 epoch activation, a transaction sender submits a transaction whose DAO-type output script data exceeds the RFC-0044 size limit. The tx-pool takes the cache-miss path, calls `DaoScriptSizeVerifier`, and — because `rfc0044_active` is false at the block level — the block verifier does not yet enforce the limit. The transaction is verified, cached under its `witness_hash`, and admitted to the tx-pool.

2. RFC-0044 activates. The node does not restart; the in-memory `TxVerificationCache` retains the entry.

3. A chain reorg occurs. The tx-pool re-admits pending transactions. `verify_rtx` is called again for the same transaction. The `witness_hash` is found in the cache. The cache-hit branch is taken. `DaoScriptSizeVerifier` is not called. The transaction is re-admitted to the tx-pool.

4. The block assembler selects the transaction for a block template. The miner mines the block. `ContextualBlockVerifier::verify` calls `BlockTxsVerifier::verify`, which applies `DaoScriptSizeVerifier` to all non-cellbase transactions (lines 444–453 of `contextual_block_verifier.rs`). The block fails verification. The miner's block reward is lost.

5. The transaction remains in the tx-pool indefinitely, as every subsequent re-verification takes the cache-hit path and passes. [5](#0-4) [6](#0-5)
