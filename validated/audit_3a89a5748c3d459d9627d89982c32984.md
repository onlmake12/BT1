### Title
EIP-7702 EOA Delegation Triggers Unbounded Gas Callback in Legacy `revealWithCallback` Path — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol::revealWithCallback` contains two execution paths. The modern path (when `req.gasLimit10k != 0`) uses `excessivelySafeCall` with an explicit gas cap. The **legacy path** (when `req.gasLimit10k == 0`, i.e., the provider has never set `defaultGasLimit`) uses a raw `extcodesize` check to decide whether to invoke `_entropyCallback`, and then calls it with **no gas limit**. With EIP-7702 active, an EOA requester can set a delegation designator, causing `extcodesize` to return non-zero and triggering the unbounded callback. The caller of `revealWithCallback` (typically the provider) pays for all gas consumed.

---

### Finding Description

`Entropy.sol` stores the gas limit for a request in `req.gasLimit10k`. When a provider has never called `setDefaultGasLimit` (i.e., `providerInfo.defaultGasLimit == 0`), the request is stored with `req.gasLimit10k = 0`: [1](#0-0) 

In `revealWithCallback`, the branch on `req.gasLimit10k != 0` selects the modern safe path (using `excessivelySafeCall` with an explicit gas cap). When `req.gasLimit10k == 0`, the legacy `else` branch executes: [2](#0-1) 

The legacy path:
1. Checks `extcodesize(callAddress)` to determine if the requester is a contract.
2. If non-zero, calls `IEntropyConsumer(callAddress)._entropyCallback(...)` with **no gas stipend** — all remaining gas (minus the 1/64 retained by EIP-150) is forwarded.

Historically, EOA requesters had `extcodesize == 0`, so the callback was skipped. Under EIP-7702 (live since the Pectra upgrade), an EOA can install a delegation designator in its code field, making `extcodesize > 0` and triggering the full unbounded callback.

The requester address is set to `msg.sender` at request time: [3](#0-2) 

`revealWithCallback` is permissionlessly callable by anyone: [4](#0-3) 

The modern path correctly caps gas via `excessivelySafeCall`: [5](#0-4) 

The legacy path has no equivalent protection.

---

### Impact Explanation

An EIP-7702-enabled EOA requester can:

1. **Siphon gas**: The callback receives ~`(63/64)` of remaining gas. The attacker's delegated contract can perform profitable on-chain operations (e.g., arbitrage swaps) at the provider's/caller's expense, since the gas cost is borne by the `revealWithCallback` caller.
2. **Cause OOG revert**: The callback can burn enough gas to cause `revealWithCallback` to revert out-of-gas, permanently blocking fulfillment of the request (the request remains in `CALLBACK_NOT_STARTED` state since `clearRequest` was already called before the callback, meaning the request is cleared but the callback reverts the whole transaction — the provider cannot fulfill the request without the attacker cooperating).

The impact is: **loss of gas funds for the provider/fulfiller** and **denial of service on request fulfillment** for legacy-path requests.

---

### Likelihood Explanation

- EIP-7702 is live on Ethereum mainnet (Pectra, May 2025) and several L2s.
- Any provider that has not called `setDefaultGasLimit` (i.e., `defaultGasLimit == 0`) is vulnerable. Legacy providers registered before the `gasLimit10k` feature was introduced fall into this category.
- The attacker only needs to: (a) identify a legacy provider, (b) set an EIP-7702 delegation on their EOA, (c) submit a valid entropy request to that provider. No privileged access is required.
- `revealWithCallback` is public and callable by anyone, so the attacker can also self-trigger the reveal.

---

### Recommendation

In the legacy `else` branch of `revealWithCallback`, replace the unbounded direct call with a gas-capped call. Either:

1. Apply a hard-coded maximum gas cap (e.g., the provider's `defaultGasLimit` or a protocol-defined constant) using `excessivelySafeCall`, mirroring the modern path.
2. Alternatively, deprecate the legacy path entirely by requiring all providers to set a non-zero `defaultGasLimit` before new requests are accepted.

The modern path already demonstrates the correct pattern: [5](#0-4) 

Apply the same `excessivelySafeCall` with an explicit gas limit to the legacy path at lines 675–681.

---

### Proof of Concept

1. Attacker EOA (`0xATTACKER`) signs an EIP-7702 authorization delegating to a contract `GasBurner` whose `_entropyCallback` performs expensive storage writes and returns normally.
2. Attacker calls `requestV2(provider, userRandom, 0)` on a legacy provider (one with `defaultGasLimit == 0`). The request is stored with `req.gasLimit10k = 0` and `req.requester = 0xATTACKER`.
3. Provider (or attacker) calls `revealWithCallback(provider, seqNum, userContrib, providerContrib)`.
4. Since `req.gasLimit10k == 0`, the `else` branch executes. `extcodesize(0xATTACKER)` returns 23 (EIP-7702 delegation designator length), so `len != 0`.
5. `IEntropyConsumer(0xATTACKER)._entropyCallback(...)` is called with all remaining gas (~`63/64` of the transaction gas at that point).
6. `GasBurner._entropyCallback` burns gas writing to attacker-controlled storage slots, then returns. The provider's transaction consumed far more gas than expected, with the excess benefiting the attacker's on-chain state. [6](#0-5)
