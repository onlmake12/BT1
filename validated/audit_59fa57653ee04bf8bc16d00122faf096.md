### Title
Unbounded `while` Loop in `Echo.executeCallback` Enables Permanent Fund Lock via Gas Exhaustion — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all cleared requests by performing a storage lookup (`findRequest`) on every iteration. Because `requestPriceUpdatesWithCallback` imposes no cap on the total number of outstanding requests (the `requestsOverflow` mapping is unbounded), an unprivileged attacker can create enough requests and arrange their fulfillment order so that the loop exhausts the block gas limit when the oldest unfulfilled request is finally executed. The entire transaction then reverts — including the preceding `clearRequest` call — permanently trapping the requester's funds.

---

### Finding Description

**Root cause — the unbounded loop:**

```solidity
// Echo.sol lines 164-174
clearRequest(sequenceNumber);

// TODO: I'm pretty sure this is going to use a lot of gas because it's doing
// a storage lookup for each sequence number.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

After clearing the just-fulfilled request, the loop scans forward from `firstUnfulfilledSeq` until it finds the next active request. Each iteration calls `findRequest`, which performs:

1. A `keccak256` hash to derive `shortKey` and `key`.
2. An SLOAD of `_state.requests[shortKey]` (the 32-slot fixed array).
3. If the slot does not match, an SLOAD of `_state.requestsOverflow[key]` — a deleted mapping slot, costing **2,100 gas** (cold read of a zero slot). [2](#0-1) 

**Why the loop is unbounded:**

`requestPriceUpdatesWithCallback` enforces only a 60-second future limit on `publishTime` and a per-request `MAX_PRICE_IDS = 10` cap. There is no limit on the total number of requests that can be created. Requests beyond the 32-slot fixed array spill into `requestsOverflow`, an unbounded `mapping(bytes32 => Request)`. [3](#0-2) [4](#0-3) 

**How `firstUnfulfilledSeq` stalls:**

`firstUnfulfilledSeq` only advances inside `executeCallback`. While request `S` (the oldest) remains active, every `executeCallback` call for a later request finds `isActive(findRequest(firstUnfulfilledSeq))` = `true` and exits the loop immediately — `firstUnfulfilledSeq` stays pinned at `S`. When `S` is finally cleared, the loop must scan through every sequence number from `S` to `currentSequenceNumber`, one SLOAD per step. [5](#0-4) 

**Exploit flow:**

1. Attacker calls `requestPriceUpdatesWithCallback` with `publishTime = block.timestamp + 59` → gets sequence number `S` (the "anchor" request).
2. Attacker calls `requestPriceUpdatesWithCallback` N more times with `publishTime = block.timestamp` → gets sequence numbers `S+1 … S+N`.
3. Provider fulfills `S+1 … S+N` in any order. During each call, `firstUnfulfilledSeq` stays at `S` (still active). All N overflow-mapping slots are deleted by `clearRequest`.
4. Provider attempts to fulfill `S`. `clearRequest(S)` executes, then the `while` loop iterates N times, each costing ~4,200 gas (two cold SLOADs). At N ≈ 7,000 the loop alone consumes ~29.4 M gas, exceeding the 30 M block gas limit.
5. The transaction reverts. `clearRequest(S)` is rolled back. Request `S` is permanently active but can never be fulfilled — the loop will always OOG.

The developer's own TODO comment on line 166 acknowledges the problem: *"I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number."* [6](#0-5) 

---

### Impact Explanation

- **Permanent fund lock**: The requester's ETH (paid as `req.fee`) is trapped in the contract forever. `executeCallback` for sequence `S` can never succeed once the loop length exceeds the block gas limit.
- **Provider DoS**: The provider cannot earn the fee for request `S` and cannot clear it from the active-request set.
- **Severity**: High — direct, irreversible loss of user funds with no recovery path.

---

### Likelihood Explanation

- **No privileged access required**: Any address can call `requestPriceUpdatesWithCallback` by paying the protocol fee.
- **Cost scales with N**: On Ethereum mainnet, creating ~7,000 requests is expensive. On L2s (Optimism, Base, Arbitrum) — the primary deployment targets for Echo — gas costs are 10–100× lower, making the attack economically viable.
- **Ordering is attacker-controlled**: The 60-second `publishTime` window lets the attacker ensure the anchor request `S` is always the last one ready for fulfillment.
- **Likelihood**: Medium on L2 deployments.

---

### Recommendation

**Short term**: Cap the maximum number of simultaneously active requests per provider (e.g., reject `requestPriceUpdatesWithCallback` if `currentSequenceNumber - firstUnfulfilledSeq > MAX_ACTIVE_REQUESTS`). This bounds the loop length.

**Long term**: Replace the linear scan with a doubly-linked list of active sequence numbers (as the TODO comment already suggests), so `firstUnfulfilledSeq` can be advanced in O(1) without iterating over cleared slots. [7](#0-6)

                uint64(block.timestamp),
                priceIds,
                100_000
            );
        }

    // After 59s, provider attempts to fulfill anchor seq S.
    // The while loop in executeCallback iterates N times → OOG → revert.
    // Anchor request S is permanently locked.
