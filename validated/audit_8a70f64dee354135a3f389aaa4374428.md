Audit Report

## Title
Fee-Rate Unit Mismatch in `calculate_min_replace_fee` and `check_tx_fee` Allows RBF Policy Bypass and Low-Fee Pool Admission for High-Cycles Transactions — (`tx-pool/src/pool.rs`, `tx-pool/src/util.rs`)

## Summary
`FeeRate` is defined as shannons per kilo-weight, where weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. Both `calculate_min_replace_fee` and `check_tx_fee` pass raw `tx_size` (bytes) as the weight argument to `FeeRate::fee()`, ignoring cycles entirely. For a transaction consuming near-maximum cycles (~70 M), the cycle-derived weight (~11,940) exceeds a minimal transaction's byte size (~200) by ~60×, causing the computed minimum fee to be underestimated by the same factor. An unprivileged `send_transaction` caller can exploit this to replace pool transactions while paying ~1/60th of the required RBF fee increment, and to admit transactions with an effective fee rate ~60× below `min_fee_rate`.

## Finding Description

**`FeeRate` unit definition:** `FeeRate` is "shannons per kilo-weight" and `fee()` computes `rate × weight / 1000`. [1](#0-0) [2](#0-1) 

**`get_transaction_weight` definition:** Weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. At `max_tx_verify_cycles = 70,000,000`, the cycle-derived weight is `70,000,000 × 0.000_170_571_4 ≈ 11,940`, versus a minimal tx size of ~200 bytes — a ~60× discrepancy. [3](#0-2) 

**Root cause 1 — `calculate_min_replace_fee` (RBF path):** The function signature accepts only `size: usize` and passes it directly to `self.config.min_rbf_rate.fee(size as u64)`. At the call site in `check_rbf`, the replacing `TxEntry` already carries `entry.cycles`, but only `entry.size` is forwarded. [4](#0-3) [5](#0-4) 

**Root cause 2 — `check_tx_fee` (admission path):** The function uses `tx_size as u64` as the weight argument. The comment explicitly acknowledges the mismatch but calls it a "cheap check". Examining `_process_tx`, `check_tx_fee` is called inside `pre_check` (before `verify_rtx`), and there is **no second fee-rate check** after cycles are determined — the `TxEntry` is constructed with the verified cycles and submitted directly. [6](#0-5) [7](#0-6) 

**`TxEntry::fee_rate()` uses correct weight** (confirming the discrepancy is only in the gate checks, not in the stored entry): [8](#0-7) 

**Default config enables RBF** (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`) and sets `max_tx_verify_cycles = 70_000_000`: [9](#0-8) 

## Impact Explanation

**RBF bypass (primary):** An attacker crafts a replacing transaction with small serialized size (~200 bytes) but a RISC-V lock script consuming ~70 M cycles. `calculate_min_replace_fee` computes the required extra fee as `1500 × 200 / 1000 = 300 shannons` instead of the correct `1500 × 11,940 / 1000 = 17,910 shannons` — a 60× underestimate. The attacker pays 301 shannons, evicts the victim transaction, and occupies the pool with an effective fee rate of ~25 shannons/KW (far below `min_fee_rate = 1000`). Miners sort by fee rate and will not include the replacing transaction, indefinitely delaying the original. This maps to **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**, as the attack can be repeated at minimal cost to grief arbitrary transactions.

**Pool admission bypass (secondary):** Any caller can submit high-cycles, small-size transactions with fee just above `min_fee_rate.fee(size)`. Each such transaction passes the size-only gate, consumes significant CPU during script verification (70 M cycles), and enters the pool with an effective fee rate far below `min_fee_rate`. Repeated submissions waste pool slots and impose CPU cost on all nodes.

## Likelihood Explanation

- RBF is enabled by default in the shipped `ckb.toml` (`min_rbf_rate = 1_500 > min_fee_rate = 1_000`).
- The attack requires only an unprivileged `send_transaction` RPC call — no special role, key, or majority hashpower.
- Crafting a RISC-V loop consuming near-maximum cycles is straightforward; CKB-VM supports arbitrary RISC-V programs.
- `max_tx_verify_cycles = 70,000,000` provides the upper bound, yielding a ~60× weight discrepancy for a minimal-size transaction.
- The attack is repeatable: after each replacement, the attacker can submit a new replacing transaction.

## Recommendation

**For `calculate_min_replace_fee`:** The `TxEntry` already carries `cycles` at the call site. Change the function signature to accept `cycles: u64` alongside `size: usize`, and replace `self.config.min_rbf_rate.fee(size as u64)` with `self.config.min_rbf_rate.fee(get_transaction_weight(size, cycles))`.

```rust
fn calculate_min_replace_fee(&self, conflicts: &[&PoolEntry], size: usize, cycles: u64) -> Option<Capacity> {
    let weight = get_transaction_weight(size, cycles);
    let extra_rbf_fee = self.config.min_rbf_rate.fee(weight);
    // ...
}
// call site:
self.calculate_min_replace_fee(&all_conflicted, entry.size, entry.cycles)
```

**For `check_tx_fee`:** Since cycles are not yet known at call time, the size-only check can remain as a lower-bound pre-filter, but a second fee-rate check using the correct weight must be added after `verify_rtx` returns the actual cycles (in `_process_tx`, between lines 734 and 751). Alternatively, pass the cycles from the verify cache entry if available.

## Proof of Concept

1. **Setup:** Node with default config (`min_fee_rate = 1000`, `min_rbf_rate = 1500`).

2. **Submit victim tx1:** Serialized size ≈ 200 bytes, cycles ≈ 1,000, fee = 200 shannons. Effective fee rate = 1000 shannons/KW. Accepted.

3. **Craft attacker tx2:** Same inputs as tx1 (conflict). Lock script is a RISC-V loop consuming 70,000,000 cycles. Serialized size ≈ 200 bytes.
   - `check_tx_fee`: `min_fee = 1000 × 200 / 1000 = 200 shannons`. Pay 501 shannons → passes.
   - `calculate_min_replace_fee`: `extra_rbf_fee = 1500 × 200 / 1000 = 300 shannons`. `min_replace_fee = 200 + 300 = 500 shannons`. Pay 501 shannons → passes RBF check.

4. **Actual weight of tx2:** `max(200, 70,000,000 × 0.000_170_571_4) = 11,940`. Effective fee rate = `501 × 1000 / 11,940 ≈ 41 shannons/KW` — 24× below `min_fee_rate`, 36× below `min_rbf_rate`.

5. **Result:** tx1 is evicted. tx2 occupies the pool at ~41 shannons/KW. Miners will not prioritize it, indefinitely delaying the original transaction. The correct extra RBF fee should have been `1500 × 11,940 / 1000 = 17,910 shannons`; the attacker paid 300 shannons — a 60× shortfall.
