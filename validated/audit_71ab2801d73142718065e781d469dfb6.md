Audit Report

## Title
Concurrent Destructive-Read Race in `HeaderMapKernel::get` Causes Node Panic — (`shared/src/types/header_map/kernel_lru.rs`)

## Summary
`HeaderMapKernel::get` calls `backend.remove(hash)` to promote a sled-resident entry back into memory, but this destructive operation is performed with no lock held after the memory `RwLock` is released. Two threads racing on the same sled-resident key will both miss in memory, both call `backend.remove`, and the second caller receives `None`. The caller chain unconditionally panics on `None` via `.expect("parent should be verified")` in `insert_valid_header`, crashing the node process.

## Finding Description

In `kernel_lru.rs` at L109–142, `HeaderMapKernel::get` first calls `self.memory.get_refresh(hash)`, which acquires and immediately releases a `RwLock` write guard (`memory.rs` L88). After that lock is released, there is no synchronization protecting the subsequent check-then-remove sequence against `backend`. Two threads can both observe a memory miss, both pass the `backend.is_empty()` guard at L125, and both call `self.backend.remove(hash)` at L132.

In `backend_sled.rs` L90–100, `remove` calls `self.db.remove(key.as_slice())`, which is atomic per-key in sled: the first caller receives `Some(view)`, the second receives `None`. Thread B therefore returns `None` from `get()`.

In `sync/src/types/mod.rs` L1143–1166, `get_header_index_view` with `store_first=false` (the IBD path, where `tip_number < header.number()`) calls `self.shared.header_map().get(hash)` first, then falls back to the chain store. Because header H has not yet been committed to the chain store (it exists only in the header map / sled backend), both lookups return `None`.

`insert_valid_header` at L1101–1103 then panics:
```rust
let parent_header_index = self
    .get_header_index_view(&parent_hash, store_first)
    .expect("parent should be verified");  // ← PANIC
```

`insert_valid_header` is called from two concurrent, unserialized P2P message-processing paths: `HeaderAcceptor::accept` in `headers_process.rs` L356 and `CompactBlockProcess::execute` in `compact_block_process.rs` L78.

## Impact Explanation

An unprivileged remote peer can trigger a node process crash. Because the panic is unrecoverable, the entire CKB node terminates. This matches the **High** impact class: *Vulnerabilities which could easily crash a CKB node* (10001–15000 points).

## Likelihood Explanation

During IBD, the node connects to multiple peers simultaneously and processes their `SendHeaders` messages concurrently. The memory eviction threshold (`WARN_THRESHOLD = size_of::<HeaderIndexView>() * 100_000`, `mod.rs` L29–31) is regularly hit, and `limit_memory()` runs every 5 seconds (`INTERVAL = 5000ms`, `mod.rs` L29), causing regular evictions to sled. Any two peers sharing a common ancestor header H that has been evicted to sled, whose child headers arrive within the same scheduling quantum, can trigger the race. This is a normal IBD operating condition. No special attacker capability is required beyond connecting as a peer and sending valid headers.

## Recommendation

Replace the destructive `backend.remove` in `get` with a non-destructive `backend.get`. Both racing threads will read the same value and insert it into memory (idempotent). The backend entry can then be lazily cleaned up (e.g., during `limit_memory` or via a separate GC pass), or removed under a coordinating lock only after confirming the entry is present in memory. Alternatively, wrap the entire memory-miss → backend-lookup → memory-insert sequence in a per-key or coarse mutex to serialize promotions.

## Proof of Concept

```rust
let map = Arc::new(HeaderMapKernel::new(None, MEMORY_LIMIT, ibd_finished));
// Insert H and evict it to sled backend
map.insert(H.clone());
map.limit_memory(); // forces eviction: memory.front_n → backend.insert_batch → memory.remove_batch

let map_a = Arc::clone(&map);
let map_b = Arc::clone(&map);

let t_a = thread::spawn(move || map_a.get(&H.hash()));
let t_b = thread::spawn(move || map_b.get(&H.hash()));

let r_a = t_a.join().unwrap();
let r_b = t_b.join().unwrap();

// One of r_a / r_b will be None due to the destructive remove race
assert!(r_a.is_some());
assert!(r_b.is_some()); // ← fails; maps directly to the production panic
```

The failing assertion corresponds exactly to the production panic at `.expect("parent should be verified")` in `insert_valid_header` (`sync/src/types/mod.rs` L1103), reachable from concurrent `SendHeaders` or compact block processing by any remote peer.
