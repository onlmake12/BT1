Audit Report

## Title
Peer Store Permanent DoS via Crafted /16-Subnet-Diverse Addresses — (`network/src/peer_store/peer_store_impl.rs`)

## Summary
The `check_purge` Phase 2 eviction in `PeerStore` uses a strict `> 4` threshold to select eviction candidates from network-segment groups. An attacker who fills `addr_manager` with exactly 4 addresses per /16 subnet (16384 total, matching `ADDR_COUNT_LIMIT`) causes every subsequent `add_addr` call to return `PeerStoreError::EvictionFailed`, permanently preventing honest peer addresses from being stored. The discovery protocol silently discards these errors at `debug` log level, making the condition invisible to operators.

## Finding Description
`check_purge` is called at the start of every `add_addr` invocation once `addr_manager.count() >= ADDR_COUNT_LIMIT` (16384).

**Phase 1** collects addresses where `is_connectable` returns `false`. Attacker-supplied addresses are added with `last_connected_at_ms = 0` and `attempts_count = 0` (the defaults in `AddrInfo::new`). `is_connectable` returns `true` for these because neither the `attempts_count >= ADDR_MAX_RETRIES` (3) nor the `attempts_count >= ADDR_MAX_FAILURES` (10) condition is satisfied. Phase 1 produces zero candidates and falls through.

**Phase 2** groups all addresses by `/16` network segment, sorts groups by descending size, takes the top `len / 2` groups, and evicts only from groups where `addrs.len() > 4`. With 4096 groups of exactly 4 addresses each:
- `len = 4096`, `take(2048)` selects the top half — all still have size 4
- `4 > 4` evaluates to `false` for every group
- `candidate_peers` is empty
- `Err(PeerStoreError::EvictionFailed)` is returned

`add_addr` propagates this error via `?`, and `add_new_addrs` in the discovery protocol catches it with a `debug!` log and discards it silently. The store remains permanently saturated.

## Impact Explanation
Once saturated, `add_addr` fails for all honest peer addresses relayed via discovery. `fetch_addrs_to_attempt` requires `last_connected_at_ms` within 3 days, so attacker addresses (with `last_connected_at_ms = 0`) are not returned there. However, `fetch_addrs_to_feeler` returns them (never-connected addresses), causing feeler connection slots to be wasted on attacker-controlled IPs that refuse or ignore connections. As existing outbound connections drop naturally, the node cannot replace them with honest peers, leading to outbound connection starvation and sync isolation. This matches the **High** impact class: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (10001–15000 points), as a targeted node loses the ability to discover and connect to honest peers.

## Likelihood Explanation
The attacker requires only one or a few connected discovery peers to relay crafted `Nodes` messages. `is_valid_addr` checks only `is_reachable` (globally routable IP), not actual connectivity. With `MAX_ADDR_TO_SEND = 1000` per message, filling 16384 slots requires approximately 17 messages. No privileged access, proof-of-work, or key material is required. The attack is repeatable and the saturated state is permanent until the node restarts and the peer store is rebuilt.

## Recommendation
Change the eviction threshold in `check_purge` Phase 2 from strictly greater-than to greater-than-or-equal:

```rust
// network/src/peer_store/peer_store_impl.rs, line 378
if addrs.len() >= 4 {   // was: > 4
```

This ensures groups at exactly the per-subnet limit are eligible for eviction, closing the gap that allows an attacker to pin every group at the threshold. Additionally, consider enforcing a per-source-/16-subnet ingestion cap inside `add_new_addrs` to limit how many addresses a single discovery session can contribute from any one subnet.

## Proof of Concept

```rust
#[test]
fn test_peer_store_eviction_dos() {
    let mut peer_store = PeerStore::default();
    // Fill with 4096 /16 subnets × 4 addresses each = 16384 = ADDR_COUNT_LIMIT
    for subnet in 0u16..4096 {
        for host in 1u8..=4 {
            let ip = format!("{}.{}.1.{}", subnet >> 8, subnet & 0xff, host);
            let addr: Multiaddr = format!("/ip4/{}/tcp/8115", ip).parse().unwrap();
            peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
        }
    }
    assert_eq!(peer_store.addr_manager().count(), 16384);

    // Any subsequent add_addr now permanently fails
    let new_addr: Multiaddr = "/ip4/200.200.200.1/tcp/8115".parse().unwrap();
    let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
    assert!(matches!(result, Err(e) if e.to_string().contains("EvictionFailed")));
    assert_eq!(peer_store.addr_manager().count(), 16384); // unchanged
}
```
