### Title
Fortuna Entropy Keeper Loads Secrets via `dotenv` Crate in Production Run Command — (File: `apps/fortuna/src/command/run.rs`)

---

### Summary

The production Fortuna Entropy keeper service explicitly calls `dotenv::dotenv()` at startup to load environment variables from a `.env` file. Because the call uses the `?` operator, the service **fails to start** if no `.env` file is present, effectively mandating that operators create one. No `.gitignore` rule excluding `.env` files was found in the repository. If a `.env` file containing keeper or provider private keys is accidentally committed to version control, an attacker can extract those keys and compromise the Entropy protocol.

---

### Finding Description

In `apps/fortuna/src/command/run.rs`, the `run()` function begins with:

```rust
let _ = dotenv::dotenv().map_err(|e| anyhow!("Failed to load .env file: {}", e))?;
``` [1](#0-0) 

The `?` operator propagates the error returned by `dotenv::dotenv()` when no `.env` file is found, causing the entire `run()` function to return an error. This means the Fortuna service **cannot start** without a `.env` file on disk, forcing operators to create one.

The Fortuna config (`apps/fortuna/src/config.rs`) defines a `SecretString` type used for all sensitive values:

```rust
#[derive(Clone, Debug, serde::Serialize, serde::Deserialize)]
pub struct SecretString {
    pub value: Option<String>,
    pub file: Option<String>,
}
``` [2](#0-1) 

This type holds the keeper private key, provider private key, and provider secret: [3](#0-2) [4](#0-3) 

The `config.sample.yaml` explicitly documents storing private keys as literal `value:` fields in the YAML config, and the same pattern applies to `.env` files: [5](#0-4) 

A search for `.env` entries in `.gitignore` files across the repository returned **no matches**, meaning there is no repository-level protection against accidentally committing a `.env` file containing secrets.

Additionally, `SecretString` derives both `Debug` and `serde::Serialize`. The full `Config` struct (including all `SecretString` fields) is stored inside `ApiState`: [6](#0-5) 

If the config is ever passed to a `tracing` macro or serialized through an API handler, the raw private key values would be emitted in plaintext.

---

### Impact Explanation

The Fortuna keeper private key is used to sign and submit on-chain entropy callback transactions. If an attacker obtains this key (via a committed `.env` file), they can:

1. **Drain the keeper wallet** — submit high-gas transactions using the keeper key to exhaust its ETH balance, halting all entropy callbacks.
2. **Submit fraudulent entropy callbacks** — call `revealWithCallback` on the Entropy contract with attacker-controlled random values, breaking the unpredictability guarantee for any user whose request is fulfilled by this keeper.

The provider `secret` is the 32-byte seed used to generate the entire hash chain. Exposure of this value allows an attacker to reconstruct every past and future random value the provider will ever reveal, completely breaking the Entropy protocol's security model for that provider.

**Impact: High** — direct financial loss (keeper wallet drain) and protocol integrity compromise (predictable randomness).

---

### Likelihood Explanation

- The `?` operator on `dotenv::dotenv()` **requires** a `.env` file to be present; operators cannot run the service without creating one.
- No `.gitignore` rule excluding `.env` files was found in the repository.
- The `config.sample.yaml` normalizes storing secrets as inline `value:` fields, making it likely that operators copy this pattern into `.env` or YAML files that may be committed.
- Accidental secret commits are a well-documented, common operational mistake.

**Likelihood: Medium** — realistic operational error with no mitigating repository-level controls.

---

### Recommendation

1. Replace the mandatory `dotenv::dotenv()?` call with a graceful fallback (e.g., `let _ = dotenv::dotenv();`) so the service does not require a `.env` file.
2. Remove the `dotenv` dependency from the production `run` command entirely; use only the `SecretString.file` path for secret injection.
3. Add `.env` to the root `.gitignore`.
4. Remove `Debug` and `serde::Serialize` from `SecretString`, or implement a custom `Debug`/`Serialize` that redacts the `value` field (e.g., always emits `"[REDACTED]"`).
5. Prefer the `SecretString.file` path (pointing to a secrets-manager-mounted file) over inline `value:` fields in all production deployments.

---

### Proof of Concept

1. Operator creates a `.env` file (required by the `?` on `dotenv::dotenv()`) containing:
   ```
   # .env
   KEEPER_PRIVATE_KEY=0xdeadbeef...
   ```
   or copies `config.sample.yaml` with `private_key.value: 0xdeadbeef...` and commits it.
2. No `.gitignore` rule blocks the commit.
3. Attacker reads the committed file from the public repository history.
4. Attacker uses the keeper private key to call `revealWithCallback` on the Entropy contract with attacker-chosen random values, or drains the keeper wallet via high-gas transactions.
