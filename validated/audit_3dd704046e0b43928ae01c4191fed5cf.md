### Title
Unbounded Hash Loop in `constructProviderCommitment` When `maxNumHashes == 0` Enables OOG-Freeze of User Funds — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

When a provider's `maxNumHashes` is `0` (the Solidity default for uninitialized storage), the guard that caps `req.numHashes` at request time is entirely skipped. `revealHelper` then calls `constructProviderCommitment(req.numHashes, ...)`, which iterates `numHashes` times. With enough concurrent unrevealed requests, this loop exhausts the block gas limit, permanently freezing the request and the user's paid fee.

---

### Finding Description

**Guard is disabled by default.**
In `requestHelper`, the cap check reads:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [1](#0-0) 

Because `maxNumHashes` is a `uint32` field in `ProviderInfo` and Solidity zero-initializes storage, every newly registered provider starts with `maxNumHashes == 0`, meaning the check is **never executed** unless the provider explicitly calls `setMaxNumHashes` with a non-zero value.

**`numHashes` grows with every unrevealed request.**
`req.numHashes` is set to `assignedSequenceNumber - providerInfo.currentCommitmentSequenceNumber`: [2](#0-1) 

`currentCommitmentSequenceNumber` only advances when a reveal is processed. So if N requests are made without any reveal, the Nth request stores `numHashes = N`.

**`revealHelper` passes `numHashes` directly into the loop.**
`revealHelper` calls `constructProviderCommitment(req.numHashes, providerContribution)` unconditionally: [3](#0-2) 

The contract's own documentation confirms the loop cost: *"Verification therefore may require computing multiple hashes (~ the number of concurrent requests)"*: [4](#0-3) 

**`revealWithCallback` is callable by anyone.**
The function has no `msg.sender` restriction — any relayer can trigger it: [5](#0-4) 

---

### Impact Explanation

If `numHashes` is large enough that `constructProviderCommitment` exhausts the block gas limit (~30 M gas on Ethereum mainnet), every call to `reveal` or `revealWithCallback` for that request will OOG-revert. The request slot is never cleared, and the user's paid fee is permanently locked in the contract with no recovery path.

---

### Likelihood Explanation

**Moderate.** The precondition (`maxNumHashes == 0`) is the **default state** for every provider that does not explicitly opt in to the guard. The Pyth default provider and any provider that omits `setMaxNumHashes` are permanently exposed. Accumulating enough concurrent requests to hit the gas limit requires capital (each request pays a fee), but on low-fee chains (BNB, Polygon, Arbitrum) the cost is negligible. A griefing attacker can make thousands of requests cheaply, then attempt to reveal the highest-`numHashes` request, causing it to OOG-freeze.

---

### Recommendation

1. **Change the default semantics of `maxNumHashes == 0`**: instead of "disabled", treat `0` as "use a safe protocol default" (e.g., 100 or 1000 hashes).
2. **Enforce the cap unconditionally** in `requestHelper`, removing the `!= 0` bypass:
   ```solidity
   uint32 effectiveMax = providerInfo.maxNumHashes == 0
       ? DEFAULT_MAX_NUM_HASHES
       : providerInfo.maxNumHashes;
   if (req.numHashes > effectiveMax) revert EntropyErrors.LastRevealedTooOld();
   ```
3. **Add a gas-cost assertion** in `constructProviderCommitment` or document the maximum safe `numHashes` per chain.

---

### Proof of Concept

```solidity
// 1. Register provider without calling setMaxNumHashes → maxNumHashes == 0
// 2. Make N requests without revealing (N = 100_000 on a cheap chain)
// 3. Call revealWithCallback for the Nth request
// 4. constructProviderCommitment loops N times → OOG revert
// 5. Request is permanently frozen; user fee is lost
```

The test file already confirms `maxNumHashes == 0` disables the check: [6](#0-5)
