### Title
Unauthenticated RPC Exposes All Privileged Node-Control Operations to Any Caller — (`File: rpc/src/server.rs`)

---

### Summary

The CKB JSON-RPC server implements zero authentication and zero per-method authorization. Every privileged, state-mutating operation — including disabling all P2P networking, clearing the entire transaction pool, banning peers, and removing specific transactions — is callable by any HTTP client that can reach the RPC port. The server additionally applies `CorsLayer::permissive()`, accepting requests from any origin. This is a direct structural analog of the MosaicVault/MosaicHolding "excessive privileges / no role separation" class: a single flat access surface with no credential gate controls all node-critical operations.

---

### Finding Description

**Root cause — no authentication in `handle_jsonrpc`**

`rpc/src/server.rs` builds the Axum router and dispatches every POST request through `handle_jsonrpc`:

```rust
async fn handle_jsonrpc<T: Default + Metadata>(
    Extension(io): Extension<Arc<MetaIoHandler<T>>>,
    req_body: Bytes,
) -> Response {
``` [1](#0-0) 

The function parses the JSON body and immediately calls `io.handle_call(call, T::default())`. There is no middleware layer, no token check, no API-key validation, and no per-method role guard anywhere in the RPC stack. A `grep` for `authorization`, `authenticate`, `auth_token`, `api_key`, `bearer`, or `basic_auth` across all of `rpc/src/**/*.rs` returns **zero matches**.

**Permissive CORS applied globally**

The router is assembled with:

```rust
.layer(CorsLayer::permissive())
``` [2](#0-1) 

This instructs the server to respond with `Access-Control-Allow-Origin: *` and accept pre-flight from any origin, removing the browser's same-origin barrier for any operator who exposes the port beyond localhost.

**Privileged operations reachable with no credential**

The Pool module exposes, with no auth gate:

- `clear_tx_pool` — atomically removes every pending transaction from the mempool:

``` [3](#0-2) 

- `remove_transaction` — removes a specific transaction and all its dependents: [4](#0-3) 

- `clear_tx_verify_queue` — flushes the pending verification queue: [5](#0-4) 

The Net module exposes, with no auth gate:

- `set_network_active(false)` — disables **all** P2P activity on the node: [6](#0-5) 

- `set_ban` — inserts or removes IP/CIDR bans on peers: [7](#0-6) 

- `add_node` / `remove_node` — forces peer connection or disconnection: [8](#0-7) 

All of these are registered in the same flat `IoHandler` with no per-method role check: [9](#0-8) 

**No authentication on the server side — miner Basic Auth is client-only**

The miner client optionally sends a `Basic` `Authorization` header when connecting to the node RPC. The server never validates it. There is no `Authorization` header extraction or verification anywhere in `rpc/src/`. [10](#0-9) 

---

### Impact Explanation

Any caller who can reach the RPC port — including any web page loaded in a browser on the same machine due to `CorsLayer::permissive()` — can:

1. **Call `set_network_active(false)`** — the node immediately stops all P2P activity: no new blocks received, no transactions relayed, no sync. The node is silently partitioned from the network until an operator manually re-enables it.

2. **Call `clear_tx_pool`** — every pending transaction is evicted. Users whose transactions were waiting for inclusion lose their queue position; fee-paying transactions must be resubmitted. An attacker can loop this call to permanently suppress mempool growth on a targeted node.

3. **Call `set_ban` with a broad CIDR** — legitimate peers are banned, degrading or eliminating the node's connectivity.

4. **Call `remove_transaction`** — a specific high-value pending transaction (e.g., a time-sensitive DeFi settlement) can be surgically evicted, enabling front-running or griefing.

Severity: **High**. The operations are irreversible without operator intervention and directly affect the node's ability to participate in consensus and relay user transactions.

---

### Likelihood Explanation

- The default `listen_address` is `127.0.0.1:8114`, but operators routinely expose the RPC publicly to serve dApps, wallets, and block explorers. The documentation warns against this but provides no enforcement mechanism.
- Even with localhost binding, `CorsLayer::permissive()` means a malicious web page visited by the node operator can issue cross-origin POST requests to `http://127.0.0.1:8114/` and invoke any method. Modern browsers restrict Private Network Access for cross-origin requests to localhost (Chrome 94+), but this protection is not universal and depends on browser version and configuration.
- No credential is required. The attack is a single HTTP POST with a JSON body — trivially scriptable.
- The attacker profile is explicitly in scope: "RPC caller" and "supported local CLI/RPC user."

---

### Recommendation

**Short term:**
- Implement a mandatory API token (e.g., a random 256-bit secret written to disk at node startup, required as a `Bearer` token or custom header on every state-mutating RPC call).
- Separate read-only methods (e.g., `get_block`, `tx_pool_info`) from write/control methods (e.g., `clear_tx_pool`, `set_network_active`, `set_ban`) into distinct authorization tiers enforced in the `handle_jsonrpc` middleware layer.
- Replace `CorsLayer::permissive()` with a restrictive CORS policy (e.g., same-origin only, or an explicit allowlist).

**Long term:**
- Introduce a role model in `RpcConfig`: read-only callers, pool-management callers, and network-management callers, each requiring a distinct credential.
- Document the security boundary clearly: the RPC port is an administrative interface, not a public API.

---

### Proof of Concept

With a CKB node running at default `127.0.0.1:8114` (or any exposed address), an attacker executes:

```bash
# Step 1: Disable all P2P networking on the target node
curl -s -X POST http://<target>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_network_active","params":[false],"id":1}'
# Response: {"jsonrpc":"2.0","result":null,"id":1}
# Node is now isolated from the P2P network.

# Step 2: Clear the entire transaction pool
curl -s -X POST http://<target>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":2}'
# Response: {"jsonrpc":"2.0","result":null,"id":2}
# All pending transactions are evicted.

# Step 3: Ban a legitimate peer by IP
curl -s -X POST http://<target>:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"set_ban","params":["0.0.0.0/0","insert",null,false,"attacker"],"id":3}'
# Response: {"jsonrpc":"2.0","result":null,"id":3}
# All peers are banned.
```

No credentials, no prior access, no privileged role required. The node is operationally disabled until an operator with local shell access manually reverses each action.

fn parse_authorization(url: &Uri) -> Option<HeaderValue> {
    let a: Vec<&str> = url.authority()?.as_str().split('@').collect();
    if a.len() >= 2 {
        if a[0].is_empty() {
            return None;
        }
        let mut encoded = "Basic ".to_string();
        base64::prelude::BASE64_STANDARD.encode_string(a[0], &mut encoded);
        let mut header = HeaderValue::from_str(&encoded).unwrap();
        header.set_sensitive(true);
        Some(header)
    } else {
        None
    }
}
