### Title
Unbounded `while` Loop in `executeCallback` Enables Permanent Fund Lock - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `executeCallback` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` past all consecutive inactive (fulfilled) requests. When requests are fulfilled out of order — a reachable condition for any unprivileged caller after the exclusivity period — the loop can iterate over an arbitrarily large range, causing the transaction to exceed the block gas limit and permanently preventing a victim's request from ever being fulfilled.

### Finding Description

After clearing the current request and crediting the provider, `executeCallback` runs: [1](#0-0) 

```solidity
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
```

The loop advances `firstUnfulfilledSeq` past every consecutive inactive request. Its iteration count equals the number of fulfilled-but-not-yet-scanned requests starting at `firstUnfulfilledSeq`. There is no on-chain cap on this value.

The critical ordering dependency: `firstUnfulfilledSeq` only advances when the request it currently points to is inactive. If request #1 remains active while requests #2 … #N are all fulfilled, `firstUnfulfilledSeq` stays at 1 for every one of those N−1 `executeCallback` calls. When request #1 is finally fulfilled, the loop must scan all N slots in a single transaction. [2](#0-1) 

The exclusivity-period check allows any address to call `executeCallback` once the period expires: [3](#0-2) 

This means an unprivileged attacker can deliberately trigger the out-of-order fulfillment pattern without being the assigned provider.

### Impact Explanation

If the while loop exceeds the block gas limit, the entire `executeCallback` call reverts — including the `clearRequest` and provider-credit steps that precede it. Because `firstUnfulfilledSeq` cannot advance past request #1 without successfully executing `executeCallback` for it, and because that call always reverts once N is large enough, the victim's request is permanently unfulfillable. The ETH deposited as the request fee is locked in the contract with no admin escape hatch or cancellation mechanism. [4](#0-3) 

### Likelihood Explanation

After the configurable `exclusivityPeriodSeconds` elapses, `executeCallback` is open to any caller: [3](#0-2) 

An attacker registers as a provider (permissionless via `registerProvider`), creates N requests pointing to themselves, fulfills requests #2 … #N (recouping provider fees), then attempts to fulfill request #1. Alternatively, a third party waits for the exclusivity period on other users' requests and fulfills them out of order. Pyth price data for any past `publishTime` is publicly available from Hermes, so providing valid `updateData` for each call is straightforward. The `MAX_PRICE_IDS` guard on `requestPriceUpdatesWithCallback` limits per-request cost but does not bound the total number of requests: [5](#0-4) 

Each while-loop iteration performs a storage read (`findRequest` → `_state.requests[shortKey]`), costing ~2 100 gas cold. At a 30 M gas limit, roughly 14 000 cold-slot iterations suffice to brick the call — achievable with a modest number of requests.

### Recommendation

Replace the unbounded `while` loop with a bounded scan (e.g., cap iterations at a constant `MAX_SCAN_STEPS` per call and let `firstUnfulfilledSeq` advance lazily across multiple transactions), or remove the `firstUnfulfilledSeq` maintenance from `executeCallback` entirely and compute it lazily in the view function `getFirstActiveRequests` only. A hard cap on the total number of live requests per provider would also limit the worst-case loop depth.

### Proof of Concept

1. Deploy Echo with `exclusivityPeriodSeconds = 60`.
2. Attacker calls `registerProvider(0, 0, 0)`.
3. Attacker calls `requestPriceUpdatesWithCallback(attacker, t, priceIds, gas)` N = 15 000 times, obtaining sequence numbers 1 … 15 000.
4. After 60 seconds, attacker calls `executeCallback(attacker, seq, updateData, priceIds)` for sequences 2 … 15 000 with valid Hermes price data. Each call succeeds; `firstUnfulfilledSeq` stays at 1 because request #1 is still active.
5. Attacker (or the assigned provider) calls `executeCallback` for sequence #1. The while loop attempts 14 999 storage reads, exhausts the 30 M gas limit, and reverts.
6. Request #1 is permanently unfulfillable; the requester's deposited ETH is locked with no recovery path.
