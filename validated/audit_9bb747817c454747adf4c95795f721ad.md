Audit Report

## Title
`DownloadScheduler::decrease` Overwrites `timeout_count` with `task_count`, Bypassing Soft-Penalty Accumulation — (`sync/src/types/mod.rs`)

## Summary
In `sync/src/types/mod.rs` line 522, `decrease` assigns `self.timeout_count = self.task_count.saturating_add(num)` instead of `self.timeout_count.saturating_add(num)`. Because `task_count` is initialized to 32, the result is always ≥ 33, which is always `> 2`, so the penalty branch fires on every single call. The intended two-strike accumulation buffer is completely bypassed, causing `task_count` to be decremented on every slow block response and peers to be disconnected prematurely.

## Finding Description
`DownloadScheduler` is initialized with `task_count = INIT_BLOCKS_IN_TRANSIT_PER_PEER = 32` and `timeout_count = 0`. The `decrease` function is meant to accumulate slow-response events in `timeout_count` and only decrement `task_count` after more than 2 have accumulated:

```rust
// sync/src/types/mod.rs L521-527
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num);  // BUG: task_count, not timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

On the very first call with `num = 1`: `self.task_count.saturating_add(1) = 33`. Since `33 > 2` is always true, `task_count` is decremented and `timeout_count` is reset to 0 immediately. On every subsequent call, `task_count` has decreased by 1 but is still ≥ 1 until it hits 0, so the condition remains always true. `timeout_count` never accumulates anything — it is overwritten with a value derived from `task_count` and immediately reset.

`decrease` is called from `remove_by_block` when `time_analyzer.push_time(elapsed)` returns `NormalToUpper` (calls `decrease(1)`) or `UpperToMax` (calls `decrease(2)`), subject to `should_punish = download_schedulers.len() > protect_num`. `protect_num` defaults to `MAX_OUTBOUND_PEERS_TO_PROTECT_FROM_DISCONNECT = 4`, a threshold easily exceeded during normal sync with multiple peers.

When `task_count` reaches 0, `prune` disconnects the peer:
```rust
// L692-700
download_schedulers.retain(|k, v| {
    if v.task_count == 0 {
        disconnect_list.insert(*k);
        false
    } else { true }
});
```

## Impact Explanation
This matches **Low (501–2000 points): Any other important performance improvements for CKB**. The bug miscalibrates the block download scheduler for every sync peer: `task_count` is decremented on every slow block response instead of every third, halving the number of concurrent block requests allowed per peer (`can_fetch()` returns `task_count - hashes.len()`). A peer consistently responding in the `NormalToUpper` range reaches `task_count = 0` after 32 slow responses instead of the intended ~96+, triggering disconnection. This degrades sync throughput and connectivity during IBD and normal sync. It does not crash the node, cause consensus deviation, or damage the economy.

## Likelihood Explanation
Any unprivileged sync peer can trigger this path by responding to `GetBlocks` with blocks arriving in the `NormalToUpper` or `UpperToMax` time quantile — a routine condition on congested or geographically distant links. No special privileges are required. The only guard (`should_punish`) is satisfied whenever more than 4 peers are connected, which is the normal operating state. The path is exercised continuously during sync.

## Recommendation
Fix line 522 to accumulate into `timeout_count` rather than overwrite it with `task_count`:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.timeout_count.saturating_add(num); // was: self.task_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
```

## Proof of Concept
Unit test against the actual struct:

```rust
#[test]
fn test_decrease_accumulates_timeout_count() {
    let mut sched = DownloadScheduler::default(); // task_count=32, timeout_count=0
    sched.decrease(1); // BUG: timeout_count = 32+1=33 > 2 → task_count becomes 31
    assert_eq!(sched.task_count(), 31); // fires immediately — should still be 32
    sched.decrease(1); // timeout_count = 31+1=32 > 2 → task_count becomes 30
    assert_eq!(sched.task_count(), 30); // fires again — accumulation never happens
}
```

With the fix applied, the first two `decrease(1)` calls would leave `task_count` at 32 (timeout_count accumulates to 1, then 2, neither exceeds 2), and only the third call would decrement it.
