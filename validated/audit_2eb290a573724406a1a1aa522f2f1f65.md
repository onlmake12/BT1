### Title
Missing `BLOCK_INVALID` Early-Return in `HeaderAcceptor::accept()` Returns Default-Valid Result for Known-Invalid Headers - (File: sync/src/synchronizer/headers_process.rs)

### Summary
`HeaderAcceptor::accept()` contains a developer-acknowledged `FIXME` noting that when a header's cached status is `BLOCK_INVALID`, the function should return early but does not. Because `ValidationResult` defaults to `ValidationState::Valid`, a header previously rejected by full block verification can be re-submitted by any peer, pass all three header-level sub-checks, and be re-inserted into the header map as valid — corrupting the sync peer state.

### Finding Description

`HeaderAcceptor::accept()` in `sync/src/synchronizer/headers_process.rs` reads the cached block status and branches only on `HEADER_VALID`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer best-known and return
    return result;
}
```

`BlockStatus` is a bitflag set:

```
HEADER_VALID  = 1
BLOCK_INVALID = 1 << 12   // = 4096
```

`BLOCK_INVALID` does **not** contain the `HEADER_VALID` bit, so `status.contains(BlockStatus::HEADER_VALID)` is `false` for an invalid header. The function falls through to `prev_block_check`, `non_contextual_check`, and `version_check`. If the header was marked `BLOCK_INVALID` by the full block verification pipeline (e.g., invalid transactions, capacity overflow, script failure) rather than by a header-level check, all three header-level sub-checks can pass. The function then calls `sync_shared.insert_valid_header(self.peer, self.header)` and returns the default `ValidationResult { state: ValidationState::Valid }` — the wrong answer for a known-invalid header.

The `ValidationResult` default is `Valid`:

```rust
#[derive(Default, Debug, Copy, Clone, PartialEq, Eq)]
pub enum ValidationState {
    #[default]
    Valid,
    ...
}
```

So the missing branch causes the function to silently succeed instead of signalling invalidity, directly analogous to the external report's `isTokenPresent: true` returned when the token is absent.

### Impact Explanation

When `insert_valid_header` is called for a `BLOCK_INVALID` header, the header is inserted into the in-memory header map and the peer's best-known header is updated to point at an invalid chain tip. The synchronizer's block fetcher uses the peer's best-known header to decide which blocks to download. A peer can therefore steer the local node's sync state machine toward a chain that has already been fully verified and rejected, causing repeated futile block download attempts and potentially displacing legitimate peers from the best-known-header slot.

### Likelihood Explanation

Any unprivileged network peer can trigger this by re-sending a `SendHeaders` message containing a header whose hash is already in the `block_status_map` with `BLOCK_INVALID`. The peer needs no special privileges, no key material, and no majority hash power. The `FIXME` comment confirms the developers are aware the guard is absent.

### Recommendation

Add an explicit early-return guard for `BLOCK_INVALID` at the top of `accept()`, before the `HEADER_VALID` branch:

```rust
if status.contains(BlockStatus::BLOCK_INVALID) {
    result.invalid(None);
    return result;
}
```

This mirrors the pattern already used in `compact_block_process.rs` (`contextual_check`), which correctly short-circuits on `BLOCK_INVALID` before any further processing.

### Proof of Concept

1. Node A fully verifies block B and marks its header hash as `BLOCK_INVALID` (e.g., script verification failure).
2. Attacker peer sends a `SendHeaders` message containing that header.
3. `HeadersProcess::execute()` calls `acceptor.accept()`.
4. `get_block_status` returns `BLOCK_INVALID` (4096); `status.contains(HEADER_VALID)` is `false` → no early return.
5. `prev_block_check`: parent is not marked invalid → `Ok(())`.
6. `non_contextual_check`: header-level PoW and timestamp checks pass (they passed during original sync) → `Ok(())`.
7. `version_check`: version is 0 → `Ok(())`.
8. `sync_shared.insert_valid_header(peer, header)` executes; peer's best-known header is set to the invalid header.
9. `accept()` returns `ValidationResult { state: Valid, error: None }`.
