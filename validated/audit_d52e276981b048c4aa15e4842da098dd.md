### Title
Unprotected `initialize()` Function Allows Front-Running to Set Malicious Wormhole Address and Data Sources â€” (File: `target_chains/stylus/contracts/pyth-receiver/src/lib.rs`)

### Summary

The `PythReceiver` Stylus contract's `initialize()` function has no caller access control. Any unprivileged address can call it before the deployer does, setting a malicious `wormhole` address, arbitrary `governance_emitter_address`, and arbitrary `data_source_emitter_chain_ids`/`data_source_emitter_addresses`. Once initialized by an attacker, the deployer's own `initialize` call reverts with `AlreadyInitialized`, leaving the contract permanently misconfigured.

### Finding Description

`initialize()` in `target_chains/stylus/contracts/pyth-receiver/src/lib.rs` is a `#[public]` function with no `msg.sender` or owner check. Its only guard is a single boolean flag:

```rust
if self.initialized.get() {
    return Err(PythReceiverError::AlreadyInitialized.into());
}
``` [1](#0-0) 

After that check, the function unconditionally writes all critical security parameters supplied by the caller:

```rust
self.wormhole.set(wormhole);
self.governance_data_source_chain_id.set(...);
self.governance_data_source_emitter_address.set(...);
// ... data sources ...
self.initialized.set(true);
``` [2](#0-1) 

There is no constraint that the caller must be the contract deployer, a multisig, or any privileged address. The function signature is fully public:

```rust
#[public]
impl PythReceiver {
    pub fn initialize(
        &mut self,
        wormhole: Address,
        ...
        governance_emitter_chain_id: u16,
        governance_emitter_address: [u8; 32],
        ...
    ) -> Result<(), PythReceiverError> {
``` [3](#0-2) 

Compare this to the Solana receiver, which is protected by Anchor's PDA `init` constraint (the config account is a PDA that can only be created once, and the payer must sign). The Stylus version has no equivalent protection. [4](#0-3) 

### Impact Explanation

An attacker who front-runs the deployer's `initialize` call can supply:

1. **A malicious `wormhole` address** â€” a contract they control that returns `valid = true` for any VAA. All subsequent `update_price_feeds` calls verify the VAA against this fake wormhole, so the attacker can post arbitrary price data for any feed.
2. **A malicious `governance_emitter_address`** â€” the attacker becomes the governance authority and can call `execute_governance_instruction` to change fees, data sources, or drain the contract's fee balance via `WithdrawFee`.
3. **Malicious `data_source_emitter_chain_ids`/`data_source_emitter_addresses`** â€” the attacker's own emitter is the only valid data source, so only their fabricated price updates are accepted.

Downstream protocols reading prices from this contract receive attacker-controlled values, enabling liquidation manipulation, oracle price manipulation, and theft of protocol funds.

### Likelihood Explanation

On Arbitrum Stylus, contract deployment and initialization are separate transactions. Any actor who monitors the mempool (or the Arbitrum sequencer itself) can observe the deployment transaction and submit an `initialize` call with higher priority before the deployer's initialization lands. The attack requires no special privilege, no leaked key, and no governance majority â€” only the ability to submit a transaction before the deployer.

### Recommendation

Add a deployer check to `initialize()`. The simplest approach is to record the deployer address at construction time (e.g., via a constructor or an immutable storage slot set during contract creation) and require `msg::sender() == deployer` inside `initialize`. Alternatively, combine deployment and initialization into a single atomic transaction using a factory/proxy pattern, eliminating the front-running window entirely.

### Proof of Concept

1. Attacker monitors Arbitrum mempool for a `PythReceiver` deployment transaction.
2. Attacker submits `initialize(attacker_wormhole, ..., attacker_governance_chain, attacker_governance_emitter, ...)` with higher gas priority before the deployer's `initialize` lands.
3. Attacker's `initialize` executes first; `self.initialized` is set to `true`.
4. Deployer's `initialize` reverts with `AlreadyInitialized`.
5. Attacker calls `update_price_feeds` with a VAA signed by their fake wormhole, posting BTC/USD = $1.
6. Any protocol reading this contract's price is manipulated.
