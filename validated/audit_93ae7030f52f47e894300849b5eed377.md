Audit Report

## Title
Unauthenticated `from` Field in Hole-Punching Protocol Enables Unbounded TCP Socket Exhaustion and Node Crash — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
An unprivileged remote peer can cause the local CKB node to spawn unbounded async tasks making repeated TCP connections to arbitrary attacker-specified IP:port targets. The `from` field in both `ConnectionRequest` and `ConnectionSync` messages is never verified against the actual sending session. By using distinct synthetic peer IDs, an attacker bypasses all per-`from` rate limits and exhausts the node's file descriptors and async task memory, crashing the node.

## Finding Description

**Root cause — `from` is never bound to the actual session**

`ConnectionRequestProcess` holds a `peer: PeerIndex` field but never uses it to verify that `content.from` matches the actual session's peer ID. [1](#0-0) 

When `self_peer_id == &content.to`, `respond_delivered` is called with the wire-supplied `content.from` and `content.listen_addrs` directly: [2](#0-1) 

Inside `respond_delivered`, the `HOLE_PUNCHING_INTERVAL` deduplication check is keyed on the attacker-controlled `from_peer_id`: [3](#0-2) 

On success, the attacker-supplied addresses are unconditionally inserted into `pending_delivered` under the attacker-supplied key: [4](#0-3) 

**`ConnectionSyncProcess` has no session field at all** [5](#0-4) 

When `self_peer_id == &content.to` and `content.route` is empty, `pending_delivered` is read using the attacker-controlled `content.from`: [6](#0-5) 

The retrieved addresses are collected into futures and passed to `select_ok` inside a single `runtime::spawn`. This spawns **one async task per `ConnectionSync`**, with up to 24 concurrent `try_nat_traversal` futures running inside it: [7](#0-6) 

**`try_nat_traversal` holds open TCP sockets for 30 seconds**

Each `try_nat_traversal` future loops for up to 30 seconds at ~200ms intervals, creating a new `TcpSocket` on every iteration: [8](#0-7) 

**Why existing guards are insufficient**

The `forward_rate_limiter` is keyed by `(content.from, content.to, msg_item_id)` in both handlers: [9](#0-8) [10](#0-9) 

Since `from` is entirely attacker-controlled, using a fresh synthetic peer ID per request changes the key and bypasses the 1 req/sec forward limit entirely. The `HOLE_PUNCHING_INTERVAL` check is equally bypassable for the same reason. The only effective throttle is the per-session `rate_limiter` at 30 req/sec: [11](#0-10) 

**Resource exhaustion math:** At 30 `ConnectionSync`/sec, each spawning 1 task holding up to 24 concurrent TCP sockets (`ADDRS_COUNT_LIMIT = 24`), after 30 seconds the node sustains ~900 concurrent tasks each holding up to 24 open sockets = ~21,600 concurrent TCP sockets. This exhausts the process's file-descriptor limit and async runtime memory, crashing the node. [12](#0-11) 

## Impact Explanation

**High (10,001–15,000 points) — Vulnerabilities which could easily crash a CKB node.**

A single attacker-controlled peer connection can exhaust the victim node's file descriptors within seconds by flooding it with `ConnectionRequest`/`ConnectionSync` pairs using distinct synthetic peer IDs. The node process crashes or becomes unresponsive, severing it from the network. The secondary impact is SSRF-equivalent arbitrary outbound TCP connections to attacker-specified hosts.

## Likelihood Explanation

Any peer that can establish a standard P2P connection to the local node can trigger this. No special privileges, no PoW, no key material are required. The two required messages (`ConnectionRequest` then `ConnectionSync`) can be sent back-to-back on the same session. The per-session rate limiter (30/sec) does not prevent the attack — it only sets the pace. The attack is fully repeatable and automatable.

## Recommendation

Bind the `from` field to the actual sending session. In `ConnectionRequestProcess::execute`, resolve the session's peer ID from the peer registry using `self.peer` and verify it equals `content.from` before calling `respond_delivered`. In `ConnectionSyncProcess`, pass the session ID and perform the same verification before reading `pending_delivered`. Additionally, cap the total number of concurrent `try_nat_traversal` tasks via a semaphore (e.g., `tokio::sync::Semaphore`) to bound resource consumption regardless of message rate.

## Proof of Concept

```
1. Attacker peer A establishes a standard P2P connection to local node L.
2. For N in 1..∞:
   a. A generates a fresh synthetic PeerId fake_id_N.
   b. A sends ConnectionRequest {
          from: fake_id_N,
          to:   L.peer_id,
          listen_addrs: [/ip4/192.168.1.1/tcp/22, ... (24 entries)],
          route: [],
          max_hops: 1
      }
      → L.respond_delivered() inserts (fake_id_N → ([24 addrs], now))
        into pending_delivered (HOLE_PUNCHING_INTERVAL check bypassed by fresh ID).
   c. A sends ConnectionSync {
          from: fake_id_N,
          to:   L.peer_id,
          route: []
      }
      → L.ConnectionSyncProcess reads pending_delivered[fake_id_N],
        spawns 1 runtime::spawn task running select_ok over 24 try_nat_traversal
        futures, each holding an open TCP socket for up to 30 seconds.
3. After 30 seconds at 30 req/sec: ~900 concurrent tasks × 24 sockets each
   = ~21,600 open TCP sockets → file-descriptor exhaustion → node crash.
```
