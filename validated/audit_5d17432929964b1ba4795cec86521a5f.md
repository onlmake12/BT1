### Title
No Rate Limiting on RPC Endpoints Enables CPU Exhaustion via Unbounded Batch `test_tx_pool_accept` Requests — (File: `rpc/src/server.rs`, `tx-pool/src/process.rs`)

---

### Summary
The CKB JSON-RPC server imposes no per-request or per-connection rate limiting on any endpoint. The batch request size limit (`rpc_batch_limit`) is explicitly disabled by default. An unprivileged RPC caller can send a single JSON-RPC batch request containing an unbounded number of `test_tx_pool_accept` calls, each of which synchronously executes full CKB-VM script verification (up to `max_block_cycles` per transaction). This exhausts CPU threads and can render the node unresponsive.

---

### Finding Description

**1. No rate limiting middleware on the RPC server**

`rpc/src/server.rs` builds the Axum router with only `CorsLayer::permissive()` and a 30-second `TimeoutLayer`. There is no connection limit, no per-IP throttle, and no request-rate middleware of any kind. [1](#0-0) 

**2. Batch request size limit is disabled by default**

`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only populated when `config.rpc_batch_limit` is explicitly set. The default configuration ships with this option commented out, and the comment itself acknowledges the risk. [2](#0-1) [3](#0-2) 

When `rpc_batch_limit` is not configured, the batch guard is never entered: [4](#0-3) 

**3. `test_tx_pool_accept` runs full CKB-VM script verification synchronously**

`test_tx_pool_accept` is designed to verify a transaction without inserting it into the pool. This means the pool-full (`PoolIsFull`) rejection that would stop `send_transaction` flooding does not apply here. Each call goes through `_test_accept_tx`, which calls `verify_rtx` with `max_block_cycles` (consensus default: ~3.5 billion cycles). [5](#0-4) [6](#0-5) 

`verify_rtx` runs `ContextualTransactionVerifier::verify` via `block_in_place`, consuming a full OS thread for the duration of each verification: [7](#0-6) 

---

### Impact Explanation

An attacker sends one HTTP POST to the RPC port containing a JSON-RPC batch array of N `test_tx_pool_accept` calls (N is unbounded by default). Each call triggers a synchronous `block_in_place` CKB-VM execution consuming up to `max_block_cycles` CPU cycles. With N in the thousands, all Tokio blocking threads are saturated, the tx-pool service queue backs up, and the node stops responding to legitimate RPC calls, P2P sync messages, and block submissions — a complete Denial-of-Service.

---

### Likelihood Explanation

The RPC server defaults to `127.0.0.1:8114`, so exploitation requires local access or a misconfigured node with the RPC port exposed. However, many node operators expose the RPC port to internal networks or behind reverse proxies without authentication. The `rpc_batch_limit` guard is off by default and the config comment explicitly warns about this. The attack requires only a single HTTP request with a crafted JSON body — no special privileges, no tokens, no keys.

---

### Recommendation

1. **Enable `rpc_batch_limit` by default** in `resource/ckb.toml` (e.g., `rpc_batch_limit = 200`) rather than leaving it commented out.
2. **Add a Tower rate-limiting middleware layer** (e.g., `tower::limit::ConcurrencyLimitLayer` or `tower_governor`) to the Axum router in `rpc/src/server.rs` `start_server`, keyed by source IP.
3. **Add a connection limit** to the TCP RPC server's accept loop in `start_tcp_server` (`rpc/src/server.rs` lines 176–194), which currently spawns an unbounded number of tasks.
4. Consider adding a per-method concurrency cap specifically for `test_tx_pool_accept` given its synchronous CKB-VM cost.

---

### Proof of Concept

```bash
# Build a batch of 5000 test_tx_pool_accept calls (rpc_batch_limit is None by default)
python3 -c "
import json, sys
call = {
  'jsonrpc': '2.0', 'method': 'test_tx_pool_accept', 'id': 1,
  'params': [{'version':'0x0','cell_deps':[],'header_deps':[],
              'inputs':[{'previous_output':{'tx_hash':'0x'+'a'*64,'index':'0x0'},'since':'0x0'}],
              'outputs':[],'outputs_data':[],'witnesses':[]}, 'passthrough']
}
print(json.dumps([call]*5000))
" > batch.json

# Send to a node with Pool module enabled
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d @batch.json
# Each of the 5000 calls enters _test_accept_tx -> verify_rtx -> block_in_place(ContextualTransactionVerifier::verify)
# saturating all Tokio blocking threads and stalling the node.
```
