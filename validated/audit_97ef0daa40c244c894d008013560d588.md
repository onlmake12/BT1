Audit Report

## Title
Write Lock Held Across Sequential `block_in_place` Script Execution in `readd_detached_tx` During Reorg — (`tx-pool/src/process.rs`, `tx-pool/src/util.rs`)

## Summary

In `update_tx_pool_for_reorg`, the tokio async write lock on `tx_pool` is acquired and held for the entire duration of `readd_detached_tx`, which sequentially calls `verify_rtx` (falling into `block_in_place` for CPU-bound CKB-VM script execution) for every detached transaction without releasing the lock between verifications. An attacker who can trigger a reorg with detached blocks containing high-cycle transactions can freeze all concurrent tx-pool operations for the full duration of sequential script verification.

## Finding Description

In `tx-pool/src/process.rs`, `update_tx_pool_for_reorg` acquires the write lock at line 836 and holds it until the closing brace at line 851, which encompasses the entire `readd_detached_tx` call:

```rust
let mut tx_pool = self.tx_pool.write().await;   // write lock acquired
_update_tx_pool_for_reorg(&mut tx_pool, ...);
self.readd_detached_tx(&mut tx_pool, retain, fetched_cache).await;
// lock dropped here
``` [1](#0-0) 

Inside `readd_detached_tx`, every detached transaction is verified sequentially in a loop, calling `verify_rtx` with `command_rx: None`: [2](#0-1) 

In `verify_rtx`, when `command_rx` is `None` and there is no cache hit, execution falls into the `block_in_place` branch, which runs synchronous CPU-bound CKB-VM script execution on the current thread: [3](#0-2) 

`block_in_place` does not release the async write lock guard — the guard lives in the outer task's state and remains held throughout. The result is that the write lock is held for the sum of all `block_in_place` durations across every detached transaction, sequentially, with no yield or lock release between them.

The entry point is the standard P2P block relay path: a reorg triggers `update_tx_pool_for_reorg` via `tx_pool_controller`: [4](#0-3) 

## Impact Explanation

While the write lock is held, every other task attempting to acquire the `tx_pool` write or read lock is suspended. This includes `submit_entry` (called for every incoming transaction), `get_block_template`, `get_tx_pool_info`, and all other tx-pool operations. The freeze duration is proportional to N_blocks × M_txs_per_block × max_tx_verify_cycles / VM_throughput. The dev config sets `max_tx_verify_cycles = 70_000_000`; mainnet/testnet values may differ. Even at 70M cycles per tx, a reorg of 3 blocks × 10 txs = 30 transactions produces a measurable freeze of all tx-pool admission and query operations. This matches the allowed impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs" (High, 10001–15000 points)**. [5](#0-4) 

## Likelihood Explanation

The attacker needs valid PoW for detached blocks. A 1–3 block reorg is achievable with a minority of hashpower through probabilistic private mining. The detached transactions are valid (they were previously committed), so no consensus rule is violated. The attack is repeatable: after the freeze ends, the attacker can trigger another reorg. The entry point is the standard P2P block relay path, reachable by any network peer.

## Recommendation

Release the write lock before script execution. The correct pattern — already used in `_process_tx` — is to perform `verify_rtx` outside the write lock and only re-acquire it for the brief `_submit_entry` call:

1. In `update_tx_pool_for_reorg`, drop the write lock after `_update_tx_pool_for_reorg` completes.
2. In `readd_detached_tx`, perform `resolve_tx` and `check_tx_fee` under a short-lived read lock (or snapshot), call `verify_rtx` without any lock held, then re-acquire the write lock only for `_submit_entry`.

Alternatively, pass a `command_rx` to `verify_rtx` in `readd_detached_tx` so it uses the async `verify_with_pause` path instead of `block_in_place`, and restructure lock acquisition to wrap only the pool mutation steps.

## Proof of Concept

1. Construct a private chain forking from block H with N blocks, each containing M transactions whose lock scripts loop until consuming `max_tx_verify_cycles` cycles.
2. Mine valid PoW for all N blocks privately.
3. Broadcast the private chain to a target node via P2P block relay.
4. The node accepts the reorg, calls `update_tx_pool_for_reorg`, acquires the write lock, and enters `readd_detached_tx`.
5. Concurrently submit transactions to the node's RPC (`send_transaction`) and measure response latency.
6. Assert: all concurrent submissions are blocked for approximately N × M × (max_tx_verify_cycles / VM_throughput) seconds, confirming the write lock is held throughout all `block_in_place` calls.
