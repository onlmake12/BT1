### Title
Entropy Provider Can Frontrun User Requests via Instant Fee Increase — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary
The Pyth Entropy contract allows any registered provider to instantly change their `feeInWei` via `setProviderFee` with no timelock or delay. Because `requestHelper` reads the live fee at execution time and excess `msg.value` is never refunded, a malicious provider can frontrun a user's `requestWithCallback`/`requestV2` transaction to either force a revert (DoS) or silently drain the user's excess ETH buffer into the provider's accrued balance.

### Finding Description
`setProviderFee` takes effect immediately with no delay: [1](#0-0) 

Inside `requestHelper`, the required fee is re-read from live state at execution time: [2](#0-1) 

The interface documentation explicitly warns that excess value is **not** refunded: [3](#0-2) 

The fee manager role (`setProviderFeeAsFeeManager`) has the same instant-change capability: [4](#0-3) 

The same pattern exists in the `Echo` contract's `setProviderFee`: [5](#0-4) 

### Impact Explanation
**Attack path 1 — DoS revert:**
1. User calls `getFee(provider)` off-chain → receives `X` wei.
2. User submits `requestWithCallback{value: X}(provider, ...)`.
3. Provider frontruns with `setProviderFee(X + 1)`.
4. `requestHelper` checks `msg.value < requiredFee` → reverts with `InsufficientFee`.
5. User wastes gas; the request is never registered.

**Attack path 2 — Silent excess capture:**
1. User, aware fees can change, sends `msg.value = X + buffer` to avoid reverts.
2. Provider frontruns with `setProviderFee(X + buffer - 1)` (just below `msg.value`).
3. Transaction succeeds; `providerFee = X + buffer - 1`; `accruedPythFeesInWei += 1`.
4. Provider captures the user's entire buffer — ETH the user never intended to pay the provider.

Because `msg.value` is never refunded, the user has no recourse once the transaction lands.

### Likelihood Explanation
Any permissionlessly registered provider can execute this. The Fortuna keeper already runs an automated `adjust_fee_wrapper` loop that calls `setProviderFeeAsFeeManager` on a schedule: [6](#0-5) 

A malicious provider can replicate this infrastructure to monitor the mempool and frontrun high-value requests. On chains with a public mempool (Ethereum mainnet, most EVM chains), this is straightforward. The `Echo` contract is equally exposed via its own `setProviderFee`. [7](#0-6) 

### Recommendation
Introduce a two-step fee update with a mandatory delay (timelock). The new fee should be committed in one transaction and only take effect after a fixed delay (e.g., 24 hours). This mirrors the standard recommendation from the referenced report and prevents same-block frontrunning. Alternatively, enforce a maximum fee increase percentage per update to bound the damage.

    function requestHelper(
        address provider,
        bytes32 userCommitment,
        bool useBlockhash,
        bool isRequestWithCallback,
        uint32 callbackGasLimit
    ) internal returns (EntropyStructsV2.Request storage req) {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (_state.providers[provider].sequenceNumber == 0)
            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
        if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.OutOfRandomness();
        providerInfo.sequenceNumber += 1;

        // Check that fees were paid and increment the pyth / provider balances.
        uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
        if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
        uint128 providerFee = getProviderFee(provider, callbackGasLimit);
        providerInfo.accruedFeesInWei += providerFee;
        _state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
            providerFee);

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
