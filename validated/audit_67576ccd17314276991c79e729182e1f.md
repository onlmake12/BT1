## Code Trace

### Entry Point: Discovery → `add_new_addrs` → `add_addr` [1](#0-0) 

`add_new_addrs` iterates over attacker-supplied addresses, filters by `is_valid_addr` (public/reachable IPs only), then calls `peer_store.add_addr`. The error from `add_addr` is **silently dropped** at `debug!` level — it is never propagated to the caller or used to disconnect the peer.

### `add_addr` calls `check_purge` unconditionally [2](#0-1) 

`add_addr` calls `self.check_purge()?` before inserting. If `check_purge` returns `Err`, the new address is never inserted. Critically, `add_addr` passes `last_connected_at_ms = 0` and `attempts_count = 0` to `AddrInfo::new`.

### `is_connectable` always returns `true` for freshly-added addresses [3](#0-2) 

A freshly-added address has `last_connected_at_ms = 0`, `attempts_count = 0`, `last_tried_at_ms = 0`. None of the three early-return conditions fire:
- `tried_in_last_minute` → false (last_tried_at_ms = 0)
- `last_connected_at_ms == 0 && attempts_count >= ADDR_MAX_RETRIES(3)` → false (count = 0)
- The timeout+failures condition → false

So all attacker-inserted addresses are permanently connectable until the node actually dials them.

### `check_purge` Step 1 finds nothing to evict [4](#0-3) 

Since all attacker addresses are connectable, `candidate_peers` is empty and no eviction occurs in step 1.

### `check_purge` Step 2: network-group eviction has a fatal gap [5](#0-4) 

The logic:
1. Groups addresses by `/16` network group
2. Sorts groups by descending size
3. Takes only the **top `len/2` groups**
4. Within those, only evicts 2 peers from groups with **strictly more than 4 members**

**The gap:** If the attacker fills `ADDR_COUNT_LIMIT = 16384` slots with addresses spread across ≥ 4096 distinct `/16` subnets (≤ 4 addresses per subnet), then:
- Every group has ≤ 4 members → the `addrs.len() > 4` condition is never true
- `candidate_peers` remains empty
- `return Err(PeerStoreError::EvictionFailed.into())` is reached [6](#0-5) 

### Constants confirm the math [7](#0-6) 

`ADDR_COUNT_LIMIT = 16384`. Filling it with 4096 distinct `/16` groups × 4 addresses each = exactly 16384 entries, all connectable, all immune to eviction.

## Verdict

### Title
Peer Store Denial-of-Discovery via Connectable-Address Flood Across Diverse /16 Subnets — (`network/src/peer_store/peer_store_impl.rs`)

### Summary
An unprivileged remote peer can flood the peer store via the discovery protocol with 16 384 connectable addresses spread across ≥ 4 096 distinct `/16` subnets (≤ 4 per subnet). Once full, `check_purge` can never evict any entry, causing every subsequent `add_addr` call to return `Err(EvictionFailed)`. The error is silently dropped by `add_new_addrs`, so the victim node permanently stops learning new peer addresses from discovery.

### Finding Description

`check_purge` has two eviction stages:

**Stage 1** — evict non-connectable addresses. Freshly-added discovery addresses always pass `is_connectable` (score = 100, `last_connected_at_ms = 0`, `attempts_count = 0`), so this stage produces nothing.

**Stage 2** — network-group eviction. It groups by `/16`, sorts by group size descending, takes the top `len/2` groups, and only evicts from groups with **strictly more than 4 members**. If the attacker ensures every `/16` group has ≤ 4 entries, no eviction occurs and `Err(EvictionFailed)` is returned.

The attack requires:
- 4 096 distinct public `/16` subnets (IPv4 has 65 536 possible `/16`s; this is trivially achievable)
- 4 addresses per subnet = 16 384 total
- Delivered via discovery `Nodes` messages (up to 1 000 addresses per message, so ~17 messages suffice; multiple sessions can be used)

The error path in `add_new_addrs` is:
```rust
if let Err(err) = peer_store.add_addr(addr.clone(), flags) {
    debug!("Failed to add discovered address to peer_store {:?} {:?}", err, addr);
}
```
The error is logged at `debug!` and discarded — no disconnect, no backpressure, no operator alert.

### Impact Explanation
Once the store is saturated, the victim node cannot add any new peer address from discovery or identify protocols. `fetch_addrs_to_feeler` and `fetch_addrs_to_attempt` will only return the attacker's fake addresses, which will fail to connect. The node's ability to discover honest peers is permanently degraded until the fake addresses accumulate enough failed dial attempts (`ADDR_MAX_RETRIES = 3` with `last_connected_at_ms == 0`) to become non-connectable — but the attacker can re-flood before that happens.

### Likelihood Explanation
- Requires only a single P2P connection (or a few) to the victim
- No PoW, no keys, no privileged access needed
- Discovery protocol imposes no per-session address-submission rate limit beyond `MAX_ADDR_TO_SEND = 1000` per message
- Public IPv4 space provides far more than 4 096 distinct `/16` subnets
- Attack is cheap, repeatable, and persistent

### Recommendation
1. **Fix the `> 4` threshold**: change to `>= 1` or remove the threshold entirely so any over-represented group is a candidate for eviction.
2. **Fix the `take(len/2)` truncation**: consider all groups, not just the top half.
3. **Rate-limit per-session address submissions** in the discovery protocol.
4. **Propagate or meter `EvictionFailed`**: at minimum, log at `warn!` level so operators can detect the condition.
5. **Cap addresses per source session** to prevent a single peer from dominating the store.

### Proof of Concept

// Fill with 4096 /16 groups × 4 addresses each = 16384 entries
for subnet in 0u32..4096 {
    let a = (subnet >> 8) as u8;
    let b = (subnet & 0xff) as u8;
    for host in 1u8..=4 {
        let addr: Multiaddr = format!(
            "/ip4/{}.{}.0.{}/tcp/8115/p2p/{}",
            a + 1, b, host, PeerId::random().to_base58()
        ).parse().unwrap();
        peer_store.add_addr(addr, Flags::COMPATIBILITY).unwrap();
    }
}
assert_eq!(peer_store.addr_manager().count(), 16384);

// Now any new address must fail
let new_addr: Multiaddr = format!(
    "/ip4/200.200.200.1/tcp/8115/p2p/{}", PeerId::random().to_base58()
).parse().unwrap();
let result = peer_store.add_addr(new_addr, Flags::COMPATIBILITY);
assert!(result.is_err(), "Expected EvictionFailed, got Ok — peer store is permanently full");

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr
        if self.last_connected_at_ms == 0 && self.attempts_count >= ADDR_MAX_RETRIES {
            return false;
        }
        // consider addr is not connectable if failed too many times
        if now_ms.saturating_sub(self.last_connected_at_ms) > ADDR_TIMEOUT_MS
            && (self.attempts_count >= ADDR_MAX_FAILURES)
        {
            return false;
        }
        true
    }
