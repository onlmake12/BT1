### Title
Missing per-peer rate limit on `GetBlockFilters` handler allows disproportionate RocksDB read amplification — (`sync/src/filter/get_block_filters_process.rs`)

### Summary
The `BlockFilter` P2P protocol handler processes `GetBlockFilters` messages with no per-peer rate limit. Each message triggers up to 2000 sequential RocksDB reads (`COLUMN_INDEX` + `COLUMN_BLOCK_FILTER`). Any unprivileged peer can send these messages in a tight loop, and with up to 125 simultaneous inbound peers doing the same, the serving node experiences sustained I/O and CPU pressure with no built-in throttle.

### Finding Description
`GetBlockFiltersProcess::execute` iterates up to `BATCH_SIZE = 1000` times, calling `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` on each iteration. Both calls resolve to RocksDB column reads (`COLUMN_INDEX` and `COLUMN_BLOCK_FILTER` respectively). The only early-exit is a 1.8 MB response-size cap, which for small per-block filters (typical on mainnet) is not reached within 1000 iterations.

The `BlockFilter` struct carries no `rate_limiter` field, and `BlockFilter::received()` dispatches directly to `self.process()` without any frequency check. This is in direct contrast to:
- `Relayer`, which gates every non-`CompactBlock` message at 30 req/s per `(PeerIndex, message_type)`.
- `HolePunching`, which gates every message at 30 req/s per `(SessionId, item_id)`.

### Impact Explanation
A single peer sending `GetBlockFilters(start_number=0)` in a tight loop forces the node to perform up to 2000 RocksDB reads per message. With `start_number=0` the same 1000 blocks are read repeatedly; RocksDB's block cache (default 256 MB) absorbs disk I/O after the first pass, but CPU cost for deserialization, molecule encoding, and response framing (up to 1.8 MB per reply) remains unbounded per peer. With the default `max_peers = 125`, 125 concurrent peers each saturating the handler can cause measurable throughput degradation on the serving node's sync and relay pipelines, which share the same async runtime.

### Likelihood Explanation
The filter protocol is enabled by default in the production `support_protocols` list. Any peer that negotiates the `/ckb/filter` protocol (protocol ID 121) can immediately send `GetBlockFilters` messages. No authentication, PoW, or prior sync state is required. The attack is trivially scriptable with any CKB P2P client library.

### Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` to `BlockFilter` (mirroring `Relayer`) and check it in `BlockFilter::received()` before dispatching to `try_process`. A quota of 10–30 `GetBlockFilters` messages per second per peer is consistent with legitimate light-client sync behavior and eliminates the amplification vector.

### Proof of Concept
1. Open N TCP connections to the target node, each completing the CKB handshake and negotiating `SupportProtocols::Filter`.
2. From each connection, send `GetBlockFilters { start_number: 0 }` in a tight loop.
3. Monitor the serving node's RocksDB read IOPS (via `/proc/<pid>/io` or `rocksdb.block.cache.miss` metrics) and CPU utilization.
4. Assert that both metrics scale linearly with N and that the node's block/header relay latency increases proportionally — demonstrating that a single peer class can cause disproportionate I/O load with no server-side throttle.
