Audit Report

## Title
Decompressed-Size Check Bypasses Per-Protocol `max_frame_length`, Enabling Up-to-8MB Heap Allocation per Frame — (`network/src/compress.rs`)

## Summary

`LengthDelimitedCodecWithCompress::decode` enforces `max_frame_length` only on the compressed wire frame (Gate 1), but allocates a decompression buffer sized by the attacker-controlled snappy varint header, bounded only by `MAX_UNCOMPRESSED_LEN = 1 << 23` (8,388,608 bytes). Because the guard uses strict `>`, a claimed decompressed size of exactly 8,388,608 passes the check and triggers `BytesMut::zeroed(8388608)` before any protocol-level size check runs. Any peer completing the P2P handshake can exploit this on the Ping protocol (1 KB `max_frame_length`) to force 8 MB heap allocations per frame, enabling OOM-based node crash under concurrent peer load.

## Finding Description

**Gate 1 — compressed frame size** (`compress.rs` line 226):
`self.length_delimited.decode(src)?` rejects frames whose wire length exceeds `max_frame_length`. For Ping this is 1,024 bytes. [1](#0-0) 

**Gate 2 — decompressed size** (`compress.rs` lines 233–242):
`decompress_len(&data[1..])` reads the snappy varint header (fully attacker-controlled). The only rejection threshold is `MAX_UNCOMPRESSED_LEN = 1 << 23`. The check `decompressed_bytes_len > MAX_UNCOMPRESSED_LEN` uses strict `>`, so a claimed length of exactly 8,388,608 evaluates to `false`, and `BytesMut::zeroed(8388608)` executes unconditionally — before any check against `max_frame_length`. [2](#0-1) 

There is no guard of the form `if decompressed_bytes_len > self.length_delimited.max_frame_length()` anywhere in the decode path. The `max_frame_length` field is consulted only in `process` (the encoder path): [3](#0-2) 

**Additional amplifier:** The decode path checks the flag byte in the received data (`data[0] & COMPRESS_FLAG`), not `self.enable_compress`. A remote peer can trigger the decompression path regardless of the local compression setting. [4](#0-3) 

**Why a tiny compressed frame can claim 8 MB:** Snappy LZ77 back-references compress 8 MB of a single repeated byte to ~130–200 bytes. The snappy varint at the start correctly encodes `8388608`. The wire frame is ~205 bytes — well within Ping's 1,024-byte `max_frame_length`.

**Per-protocol `max_frame_length` values** (all well below 8 MB): [5](#0-4) 

**Codec wired with `compress: true` by default** in both `CKBProtocol::new_with_support_protocol` and `CKBProtocol::new`: [6](#0-5) [7](#0-6) 

**Codec construction** passes `max_frame_length` to the inner `LengthDelimitedCodec` but this value is never consulted during decode for the decompressed size: [8](#0-7) 

## Impact Explanation

**High (10,001–15,000 points): Vulnerabilities which could easily crash a CKB node.**

A node with N inbound peers, each sending one crafted frame per second on the Ping protocol, sustains N × 8 MB of concurrent heap pressure from the codec layer alone. The allocation at `compress.rs` line 242 is not amortized — each frame triggers a fresh `BytesMut::zeroed(8388608)`. Under sufficient peer concurrency, this exhausts available heap and causes OOM, crashing the node process.

| Protocol | `max_frame_length` | Max allocation per frame | Amplification |
|---|---|---|---|
| Ping | 1 KB | 8 MB | 8,192× |
| Feeler | 1 KB | 8 MB | 8,192× |
| DisconnectMessage | 1 KB | 8 MB | 8,192× |
| Time | 1 KB | 8 MB | 8,192× |
| Identify | 2 KB | 8 MB | 4,096× |

## Likelihood Explanation

- No authentication or privilege required — any peer completing the P2P handshake can send protocol messages.
- The crafted frame is valid snappy; it passes all existing checks.
- The attack is trivially reproducible with any snappy library in a few lines of code.
- The Ping protocol is always open to all connected peers, making it the lowest-friction attack surface.
- The attack is repeatable at the rate the network allows frame delivery, with no per-frame cost to the attacker beyond bandwidth.
- The decode path does not gate on `self.enable_compress`, so the attack works even if the local node has compression disabled.

## Recommendation

Add a decompressed-size check against `max_frame_length` immediately after `decompress_len` succeeds, before the allocation in `LengthDelimitedCodecWithCompress::decode` (`compress.rs`, between lines 235 and 242):

```rust
if decompressed_bytes_len > self.length_delimited.max_frame_length() {
    return Err(io::ErrorKind::InvalidData.into());
}
```

This mirrors the existing guard in `process` (the encoder side, `compress.rs` line 144) and closes the asymmetry between encode and decode. [3](#0-2) 

## Proof of Concept

```rust
// Craft ~200-byte snappy stream that decompresses to 8 MB of zeros
let raw = vec![0u8; 8 * 1024 * 1024]; // 8 MB of zeros
let compressed = snap::raw::Encoder::new().compress_vec(&raw).unwrap();
// compressed.len() ≈ 130–200 bytes

// Build a wire frame: [4-byte length prefix][COMPRESS_FLAG=0x80][compressed payload]
// Total wire size ≈ 205 bytes — well within Ping's 1,024-byte max_frame_length

// Send on the Ping protocol connection.
// Node calls BytesMut::zeroed(8388608) before any protocol-level check.
// Assert: node heap grows by 8 MB per such frame received.
```

`decompress_len` on the crafted stream returns exactly `8388608`; the check `8388608 > 8388608` is `false`; the 8 MB allocation at `compress.rs` line 242 proceeds. Sending this frame from N concurrent peers causes N × 8 MB of simultaneous heap pressure, leading to OOM and node crash. [9](#0-8)
