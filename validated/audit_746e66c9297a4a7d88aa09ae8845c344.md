Audit Report

## Title
Fee Gate Uses Serialized Size Only, Ignoring Cycle Consumption — Cycle-Heavy Transactions Admitted Below True Cost - (File: `tx-pool/src/util.rs`)

## Summary
The tx-pool admission check `check_tx_fee` computes the minimum required fee using only the transaction's serialized byte size, explicitly ignoring cycle consumption. CKB's true transaction weight is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`, but no second fee check is performed after `verify_rtx` returns actual cycles. An unprivileged attacker can submit cycle-heavy transactions at the minimum size-based fee, triggering expensive VM execution at a fraction of the correct cost and saturating the verification worker pool.

## Finding Description
`check_tx_fee` in `tx-pool/src/util.rs` (L42–45) explicitly uses only `tx_size` for the minimum fee calculation, with a comment acknowledging this is a "cheap check":

```rust
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
```

This check runs inside `pre_check` (called at `process.rs` L715), before script execution. After `verify_rtx` returns actual cycles at `process.rs` L724–732, the code at L751 creates a `TxEntry` with the real cycles:

```rust
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

No second fee adequacy check against the cycle-adjusted weight is performed anywhere in `_process_tx`. The correct weight formula (`get_transaction_weight` in `util/types/src/core/tx_pool.rs` L298–303) is only used in `TxEntry::fee_rate()` (`entry.rs` L114–118) for eviction ordering and fee estimation — never for admission gating. A transaction with a tiny serialized size but maximum cycle consumption passes the fee gate at a fraction of the fee its actual resource cost warrants, and the full VM execution cost is incurred before any eviction can occur.

## Impact Explanation
This matches the **High** impact class: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

A transaction with ~200 bytes serialized size and 70,000,000 cycles (the default `max_tx_verify_cycles`) passes the fee gate at 200 shannons (size-only check at 1000 shannons/KW), while the correct weight-based minimum fee is `max(200, 70_000_000 × 0.000_170_571_4) = 11,940` shannons — approximately 60× underpriced. An attacker flooding the verify queue with such transactions saturates the verification worker pool, causing legitimate transactions to queue indefinitely and degrading or halting transaction processing across the network.

## Likelihood Explanation
The attack requires only: (1) a valid CKB transaction with a lock/type script executing a tight computation loop consuming near-maximum cycles, (2) sufficient CKB capacity to pay the underpriced fee, and (3) access to the standard `send_transaction` RPC endpoint or a peer relay connection. No privileged access, key leakage, or majority hashpower is required. The attack is repeatable with many UTXOs in parallel, each triggering a full VM execution at ~1.7% of the correct cost.

## Recommendation
After `verify_rtx` returns the actual cycle count in `_process_tx`, perform a second fee adequacy check using the full weight before creating the `TxEntry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(
        tx_pool_config.min_fee_rate,
        min_fee_by_weight.as_u64(),
        fee.as_u64(),
    )), snapshot));
}
```

The existing `check_tx_fee` can remain as the cheap pre-check to reject obviously underpaying transactions early, but a weight-accurate check must follow once cycles are known.

## Proof of Concept
1. Write a CKB script that executes a tight loop consuming ~70,000,000 cycles (near `max_tx_verify_cycles`).
2. Deploy it as a lock script on a cell with minimal capacity.
3. Craft a transaction spending that cell: serialized size ≈ 200 bytes.
4. Submit via `send_transaction` RPC with fee = 200 shannons (satisfies size-only check at 1000 shannons/KW, confirmed at `tx-pool/src/util.rs` L45).
5. Observe the transaction is accepted into the pool despite consuming 70M cycles of verification work — no second fee check exists between `verify_rtx` (`process.rs` L724) and `TxEntry::new` (`process.rs` L751).
6. Repeat with many UTXOs in parallel; each submission triggers a full 70M-cycle VM execution at a fee 60× below the weight-correct minimum.
7. The verification worker pool saturates; legitimate transactions queue indefinitely.
