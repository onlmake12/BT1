### Title
Legacy `revealWithCallback` Path Permanently Locks Entropy Requests via Reverting Callback — (`target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the legacy `revealWithCallback` code path (active when `req.gasLimit10k == 0`, i.e., the provider has never called `setDefaultGasLimit`) invokes `_entropyCallback` on the requester contract **directly, without any try-catch protection**. If the callback reverts, the entire `revealWithCallback` transaction reverts — including the preceding `clearRequest` call — leaving the request permanently active and unfulfillable. Because the provider fee is credited at **request time** (not reveal time), the provider retains the fee while the user's request is stuck forever.

---

### Finding Description

`revealWithCallback` has two distinct execution branches:

**New path** (`req.gasLimit10k != 0`): uses `excessivelySafeCall` to catch callback reverts gracefully, emitting `CallbackFailed` and moving the request to `CALLBACK_FAILED` state for recovery. [1](#0-0) 

**Legacy path** (`req.gasLimit10k == 0`): calls `_entropyCallback` directly with no error handling. If the callback reverts, the whole transaction reverts. [2](#0-1) 

The legacy path is active for **every provider whose `defaultGasLimit` is still 0** (the default). This is set at request time: [3](#0-2) 

The critical ordering in the legacy path is:
1. `clearRequest(provider, sequenceNumber)` — line 666
2. `IEntropyConsumer(callAddress)._entropyCallback(...)` — line 676 (no try-catch)

If step 2 reverts, step 1 is also rolled back. The request remains active. Since there is no alternative path to clear a `requestWithCallback` request (the `reveal` function only handles `CALLBACK_NOT_NECESSARY` requests), the slot is permanently occupied.

The provider fee is credited during `requestHelper` at request time — a separate, already-committed transaction: [4](#0-3) 

This means the provider keeps the fee while the request can never be cleared.

The existing test `testRequestWithCallbackAndRevealWithCallbackFailing` explicitly confirms this revert behavior for the legacy path: [5](#0-4) 

---

### Impact Explanation

- **User funds locked**: The user paid a fee at request time and will never receive their random number. The request cannot be cleared by any party.
- **Provider sequence number exhaustion**: An attacker can make many `requestWithCallback` calls with a reverting callback against any provider with `defaultGasLimit == 0`. Each request permanently consumes one sequence number slot. When `endSequenceNumber` is reached, the provider is forced to re-register.
- **No recovery path**: Unlike the new path (which has a `CALLBACK_FAILED` recovery state), the legacy path has no mechanism to clear a stuck request.

---

### Likelihood Explanation

- Any Entropy user can deploy a contract with a reverting `_entropyCallback` and call `requestWithCallback`.
- All providers that have not explicitly called `setDefaultGasLimit` are affected (legacy path is the default).
- The attacker pays the provider fee per stuck request, making this a paid but realistic DoS. A single stuck request permanently locks user funds.
- Accidental triggering is also possible: a legitimate user's callback contract that has a bug causing it to always revert will produce the same outcome.

---

### Recommendation

**Short term**: Apply the same `excessivelySafeCall` / try-catch pattern used in the new path to the legacy path as well. The legacy path should never propagate a callback revert to the outer transaction.

**Long term**: Require all providers to set a non-zero `defaultGasLimit` before accepting `requestWithCallback` calls, eliminating the legacy path entirely. Alternatively, add an admin-callable `forceExpireRequest` function that can clear permanently stuck requests.

---

// Attacker contract
contract MaliciousConsumer {
    IEntropy entropy;
    constructor(address _entropy) { entropy = IEntropy(_entropy); }

    // Always reverts on callback
    function _entropyCallback(uint64, address, bytes32) external pure {
        revert("blocked");

        // Fee is credited to provider here, in this transaction
        entropy.requestWithCallback{value: msg.value}(provider, userRandom);
