### Title
Equal-Fee-Rate Tx-Pool Eviction Displaces Older Transactions Before Newer Ones — (`tx-pool/src/component/sort_key.rs`)

---

### Summary

When the CKB tx-pool is full and a new transaction arrives with the same fee rate as existing transactions, the pool's eviction logic incorrectly evicts the **oldest** transaction first instead of the **newest** one. This is directly caused by a comparison direction bug in `EvictKey::cmp`: the tiebreaker uses `self.timestamp.cmp(&other.timestamp)` (ascending = oldest first), but the documented intent — stated in the code comment immediately above the struct — is to evict the **latest** timestamp. An unprivileged tx-pool submitter can exploit this to displace an earlier submitter's transaction by sending a transaction with the same fee rate when the pool is at capacity.

---

### Finding Description

`EvictKey` is the sort key used to select which transaction to evict when the pool exceeds `max_tx_pool_size`. It is defined in `tx-pool/src/component/sort_key.rs`:

```rust
/// First compare fee_rate, select the smallest fee_rate,
/// and then select the latest timestamp, for eviction,
/// the latest timestamp which also means that the fewer descendants may exist.
pub struct EvictKey {
    pub fee_rate: FeeRate,
    pub timestamp: u64,
    pub descendants_count: usize,
}
``` [1](#0-0) 

The `Ord` implementation for `EvictKey` is:

```rust
impl Ord for EvictKey {
    fn cmp(&self, other: &Self) -> Ordering {
        if self.fee_rate == other.fee_rate {
            if self.descendants_count == other.descendants_count {
                self.timestamp.cmp(&other.timestamp)   // <-- BUG
            } else {
                self.descendants_count.cmp(&other.descendants_count)
            }
        } else {
            self.fee_rate.cmp(&other.fee_rate)
        }
    }
}
``` [2](#0-1) 

`next_evict_entry` iterates in **ascending** order via `iter_by_evict_key()` and picks the **first** (smallest) entry: [3](#0-2) 

Because `self.timestamp.cmp(&other.timestamp)` is ascending, the entry with the **smallest** timestamp (i.e., the **oldest** transaction) sorts first and is therefore evicted first. The comment explicitly states the intent is the opposite: evict the **latest** timestamp.

`limit_size` in `tx-pool/src/pool.rs` calls `next_evict_entry` in a loop until the pool is within bounds: [4](#0-3) 

The bug is confirmed by the existing test `test_min_timestamp_evict`: three entries with identical fee rates and timestamps 30, 31, 32 sort to `[30, 31, 32]` — the oldest (30) is first and thus evicted first: [5](#0-4) 

And `test_pool_evict` confirms the same at the pool level: entry1 (oldest, same fee as entry2/entry3) is evicted first: [6](#0-5) 

---

### Impact Explanation

When the pool is at capacity, any unprivileged submitter can displace a previously-submitted transaction by sending a new transaction with the **same fee rate**. The victim's transaction — which arrived earlier and has a smaller timestamp — is selected for eviction first, even though it has equal economic priority. The victim's transaction is silently dropped from the pool and must be resubmitted, potentially at a higher fee rate. This breaks the fairness guarantee implied by the code comment and creates a griefing vector: an attacker who monitors the mempool can continuously re-submit same-fee-rate transactions to keep a target transaction evicted.

---

### Likelihood Explanation

The pool fills up regularly under normal network load. Any tx-pool submitter (RPC `send_transaction`) can trigger this path without any special privilege. The attacker only needs to know the victim's fee rate (observable from the mempool via `get_raw_tx_pool`) and submit a transaction with the same fee rate when the pool is near capacity. No key material, hashpower, or Sybil capability is required.

---

### Recommendation

Reverse the timestamp comparison in `EvictKey::cmp` so that the **newest** transaction (largest timestamp) is evicted first when fee rates and descendant counts are equal, matching the documented intent:

```rust
// Before (evicts oldest first — wrong):
self.timestamp.cmp(&other.timestamp)

// After (evicts newest first — correct):
other.timestamp.cmp(&self.timestamp)
``` [7](#0-6) 

The test `test_min_timestamp_evict` and `test_pool_evict` would need to be updated to assert the reversed eviction order after this fix.

---

### Proof of Concept

1. Pool is at `max_tx_pool_size` with victim's transaction `tx_v` (fee rate = R, timestamp = T1).
2. Attacker submits `tx_a` with fee rate = R (same), timestamp = T2 > T1.
3. `submit_entry` → `limit_size` is called.
4. `next_evict_entry(Status::Pending)` iterates `iter_by_evict_key()` ascending.
5. `tx_v.evict_key < tx_a.evict_key` because `T1 < T2` and fee rates are equal.
6. `tx_v` is selected and removed via `remove_entry_and_descendants`.
7. `tx_a` remains in the pool; `tx_v` is gone.

The victim's earlier transaction is displaced by the attacker's equal-fee-rate transaction, directly mirroring the M-08 pattern where earlier bidders at the same price are cut out by later bidders.

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
