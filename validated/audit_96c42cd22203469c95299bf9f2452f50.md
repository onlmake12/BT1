Audit Report

## Title
Stale `header_map` Entries on Block Verification Failure — (`chain/src/verify.rs`, `chain/src/orphan_broker.rs`)

## Summary
The failure branch of `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks` and `OrphanBroker::process_invalid_block` both omit `remove_header_view`, leaving `header_map` and `block_status_map` in an inconsistent state for every block that passes header validation but fails contextual verification. `insert_peer_unknown_header_list` queries `header_map` directly, bypassing `block_status_map`, so stale entries for invalid blocks can corrupt peer best-known-header tracking and degrade sync scheduling.

## Finding Description
**`chain/src/verify.rs` L141–190 — confirmed asymmetry:**

The success branch (L141–152) calls both `remove_block_status` (L143) and `remove_header_view` (L145). The failure branch (L153–190) calls `delete_unverified_block` (L173) and either `insert_block_status(BLOCK_INVALID)` (L177) or `remove_block_status` (L180), but never `remove_header_view`. The omission is confirmed; `remove_header_view` appears exactly once in this file (success path only).

**`chain/src/orphan_broker.rs` L88–105 — confirmed asymmetry:**

`process_invalid_block` calls `delete_block` (L93) and `insert_block_status(BLOCK_INVALID)` (L95–96) but omits `remove_header_view`. `clean_expired_orphans` (L134–156) in the same file correctly calls all three: `delete_block` (L147), `remove_header_view` (L148), and `remove_block_status` (L149).

**`shared/src/shared.rs` L425–445 — masking confirmed:**

`get_block_status` checks `block_status_map` first (L426–427); if `BLOCK_INVALID` is present, the stale `header_map` entry is masked for callers that go through this function.

**`sync/src/types/mod.rs` L1181–1196 — bypass confirmed:**

`insert_peer_unknown_header_list` iterates a peer-supplied header list and calls `self.shared().header_map().get(&hash)` directly (L1187), bypassing `block_status_map`. On the first hash found in `header_map` it calls `may_set_best_known_header` (L1190) and breaks. A stale entry for an invalid block therefore causes the node to record that peer's best-known header as the invalid block's `HeaderIndexView`, skewing download scheduling.

**Grep confirmation:** `remove_header_view` is called in exactly four places in the repo: `shared/src/shared.rs` (definition), `chain/src/verify.rs` (success path only), `chain/src/orphan_broker.rs` (`clean_expired_orphans` only), and `sync/src/relayer/mod.rs`. The failure paths in both `verify.rs` and `orphan_broker.rs` are confirmed omissions.

## Impact Explanation
The two in-memory maps `header_map` and `block_status_map` are left in an inconsistent state: `block_status_map` holds `BLOCK_INVALID` while `header_map` retains the corresponding `HeaderIndexView`. Entries accumulate without bound for every block that passes non-contextual header validation but fails contextual verification. The concrete secondary effect is that `insert_peer_unknown_header_list` can act on stale `header_map` data to set a peer's best-known header to an invalid block, degrading sync scheduling. This matches the allowed bounty impact: **Suboptimal implementation of CKB state storage mechanism (Medium, 2001–10000 points)**.

## Likelihood Explanation
Any unprivileged peer can trigger this by sending a block with valid PoW and structure (passing non-contextual header validation) but with invalid transactions (failing contextual verification). No special privileges are required. Each such block leaves one permanent stale `header_map` entry. Realistic triggers include: a miner producing a valid-PoW block with a double-spend; a compact block reconstructed with mismatched transactions; or a block from a fork with different contextual rules. The condition is reachable repeatedly at negligible cost.

## Recommendation
In the failure branch of `consume_unverified_blocks` (`chain/src/verify.rs` L173), add `self.shared.remove_header_view(&block_hash)` immediately after `self.delete_unverified_block(&block)`, mirroring the success branch. In `process_invalid_block` (`chain/src/orphan_broker.rs` L93), add `self.shared.remove_header_view(block_hash)` after `self.delete_block(&lonely_block)`, mirroring what `clean_expired_orphans` already does correctly.

## Proof of Concept
1. Connect a peer to a CKB node and send headers that pass non-contextual validation (valid PoW, valid parent chain). Each header is inserted into `header_map` via `insert_valid_header`.
2. Send the corresponding full blocks. Each passes `non_contextual_verify` and is stored to DB.
3. During `consume_unverified_blocks`, contextual verification fails (e.g., a transaction spends a non-existent cell).
4. The failure path executes: `delete_unverified_block` removes the block from DB, `insert_block_status(BLOCK_INVALID)` marks it invalid — `remove_header_view` is never called.
5. Assert `shared.header_map().contains_key(&block_hash)` returns `true` while `shared.block_status_map().get(&block_hash)` returns `BLOCK_INVALID`. The maps are inconsistent.
6. Send a `SendHeaders` message from a second peer whose header list includes the invalid block's hash. Observe that `insert_peer_unknown_header_list` hits the stale `header_map` entry and calls `may_set_best_known_header` with the invalid block's `HeaderIndexView`, incorrectly setting that peer's best-known header.
7. Repeat steps 1–4 with distinct valid-PoW headers; observe `header_map` growing monotonically with no corresponding cleanup.
