### Title
Unbounded Linear Scan in `executeCallback` Can Run Out of Gas - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` contains an unbounded `while` loop that linearly scans through every sequence number from `_state.firstUnfulfilledSeq` to `_state.currentSequenceNumber`, performing a storage lookup per iteration. When a large number of requests have been fulfilled out of order, this loop can consume enough gas to exceed the block gas limit, making the final `executeCallback` call impossible to execute.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` advances `_state.firstUnfulfilledSeq` past all already-cleared sequence numbers:

```solidity
// Echo.sol lines 166–174
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
// After successful callback, update firstUnfulfilledSeq if needed
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs two storage reads:

1. `_state.requests[shortKey]` — a fixed array of 32 slots (warm after first 32 accesses, ~100 gas each)
2. `_state.requestsOverflow[key]` — a mapping keyed by `keccak256(sequenceNumber)`, **cold for every unique sequence number** (~2,100 gas each) [2](#0-1) 

The `requestsOverflow` mapping is unbounded. Any sequence number whose `shortKey` (lower 5 bits of `keccak256`) collides with an already-occupied fixed-array slot is stored in the overflow mapping. Over time, with many requests, the majority of cleared entries live in the overflow mapping, each costing a cold SLOAD per loop iteration.

The `EchoState` contract confirms the fixed array is only 32 slots deep, while `requestsOverflow` is an unbounded `mapping(bytes32 => Request)`: [3](#0-2) [4](#0-3) 

The developer's own TODO comment at line 166 acknowledges the gas problem and suggests a doubly-linked list as a fix.

---

### Impact Explanation

**Severity: High**

- An `executeCallback` call that triggers a large scan will run OOG, reverting the entire transaction.
- The request being fulfilled remains active (its `clearRequest` already ran, but the fee credit and callback still need to complete — the TODO at line 155 even notes that a revert here can permanently lock funds).
- Providers cannot collect their accrued fees for the affected request.
- The `firstUnfulfilledSeq` pointer never advances past the stuck sequence, so every subsequent `executeCallback` that tries to advance it will also pay the full scan cost. [5](#0-4) 

---

### Likelihood Explanation

**Likelihood: Medium**

The attack path requires no privileged access. Any unprivileged caller can:

1. Call `requestPriceUpdatesWithCallback` N times (paying the required fee each time), creating sequence numbers 1 … N. The overflow mapping absorbs requests beyond the 32-slot fixed array.
2. Call `executeCallback` for sequence numbers N, N-1, …, 2 — fulfilling all requests except the oldest (seq 1). Each of these calls advances `firstUnfulfilledSeq` only by 0 or 1 because seq 1 is still active.
3. Call `executeCallback` for sequence number 1. The `while` loop must now scan from seq 1 to seq N, performing ~N cold SLOADs into `requestsOverflow`.

At ~2,200 gas per iteration (one warm fixed-array SLOAD + one cold overflow SLOAD), the Base block gas limit of 240,000,000 is reached at approximately **N ≈ 109,000 iterations**. Even at N = 10,000 the gas cost is ~22,000,000 gas — a substantial and unexpected cost imposed on the fulfilling relayer.

This can also occur organically without malicious intent if a provider's relayer fulfills requests out of order at scale.

---

### Recommendation

Replace the linear scan with a data structure that supports O(1) advancement of `firstUnfulfilledSeq`:

- **Doubly-linked list of active requests** (as suggested in the TODO): maintain `prev`/`next` pointers so that clearing a node and finding the new minimum is O(1).
- **Lazy pointer**: remove the `while` loop entirely and let `firstUnfulfilledSeq` be advanced lazily only up to a bounded number of steps per call (e.g., cap at 64 iterations per `executeCallback`), accepting that the pointer may lag.
- **Skip the pointer update**: if `firstUnfulfilledSeq` is only used by the off-chain `getFirstActiveRequests` view function, move the scan entirely off-chain and remove the on-chain loop. [6](#0-5)

### Proof of Concept

```
Setup:
- Deploy Echo with a registered provider.
- Attacker calls requestPriceUpdatesWithCallback() 110,000 times,
  paying the required fee each time.
  → _state.currentSequenceNumber = 110,001
  → _state.firstUnfulfilledSeq  = 1

Attack:
- Attacker (or provider relayer) calls executeCallback() for
  sequence numbers 110,000, 109,999, ..., 2.
  Each call: the while loop advances firstUnfulfilledSeq by 0
  (seq 1 is still active), so no scan occurs yet.

Final call:
- executeCallback(sequenceNumber = 1) is called.
- clearRequest(1) runs successfully.
- The while loop begins at firstUnfulfilledSeq = 1 and must
  scan to currentSequenceNumber = 110,001.
- Each iteration: findRequest(seq) → _state.requests[shortKey]
  (warm, ~100 gas) + _state.requestsOverflow[keccak256(seq)]
  (cold, ~2,100 gas) ≈ 2,200 gas/iter.
- Total loop gas: 110,000 × 2,200 ≈ 242,000,000 gas > Base limit.
- Transaction reverts with OOG.
- Provider cannot collect fees for seq 1; firstUnfulfilledSeq
  remains stuck at 1.
```

    function findRequest(
        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
        }
    }

    function getFirstActiveRequests(
        uint256 count
    )
        external
        view
        override
        returns (Request[] memory requests, uint256 actualCount)
    {
        requests = new Request[](count);
        actualCount = 0;

        // Start from the first unfulfilled sequence and work forwards
        uint64 currentSeq = _state.firstUnfulfilledSeq;

        // Continue until we find enough active requests or reach current sequence
        while (
            actualCount < count && currentSeq < _state.currentSequenceNumber
        ) {
            Request memory req = findRequest(currentSeq);
            if (isActive(req)) {
                requests[actualCount] = req;
                actualCount++;
            }
            currentSeq++;
        }

        // If we found fewer requests than asked for, resize the array
        if (actualCount < count) {
            assembly {
                mstore(requests, actualCount)
            }
        }
    }
