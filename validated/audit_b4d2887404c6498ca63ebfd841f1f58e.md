### Title
Malicious Admin Can Prevent Their Own Replacement via `proposeAdmin`/`acceptAdmin` Race Condition - (`EntropyGovernance.sol` and `SchedulerGovernance.sol`)

### Summary
Both `EntropyGovernance` and `SchedulerGovernance` implement a two-step admin transfer pattern (`proposeAdmin` → `acceptAdmin`). Because `proposeAdmin` is callable by **either the owner or the current admin**, a malicious/compromised admin can front-run the owner's replacement attempt by overwriting `_state.proposedAdmin` with an address they control, permanently blocking the owner from replacing them without atomic execution.

### Finding Description
In `EntropyGovernance.sol`, `proposeAdmin` is gated by `_authoriseAdminAction()`:

```solidity
function proposeAdmin(address newAdmin) public virtual {
    require(newAdmin != address(0), "newAdmin is zero address");
    _authoriseAdminAction();          // passes for admin OR owner
    _state.proposedAdmin = newAdmin;
    emit NewAdminProposed(_state.admin, newAdmin);
}
``` [1](#0-0) 

`_authoriseAdminAction()` in `EntropyUpgradable` allows both `owner()` and `_state.admin`:

```solidity
function _authoriseAdminAction() internal view override {
    if (msg.sender != owner() && msg.sender != _state.admin)
        revert EntropyErrors.Unauthorized();
}
``` [2](#0-1) 

`acceptAdmin()` only allows `_state.proposedAdmin` to finalize the transfer:

```solidity
function acceptAdmin() external {
    if (msg.sender != _state.proposedAdmin)
        revert EntropyErrors.Unauthorized();
    ...
}
``` [3](#0-2) 

The identical pattern exists in `SchedulerGovernance.sol`: [4](#0-3) 

with `_authorizeAdminAction()` in `SchedulerUpgradeable` also permitting both owner and admin: [5](#0-4) 

### Impact Explanation
A malicious or compromised admin can permanently block the owner from replacing them:

1. Owner calls `proposeAdmin(newAdmin)` → `_state.proposedAdmin = newAdmin`
2. Malicious admin front-runs by calling `proposeAdmin(attackerControlledAddress)` → `_state.proposedAdmin` is overwritten
3. `newAdmin` calls `acceptAdmin()` → reverts (`msg.sender != _state.proposedAdmin`)
4. Attacker-controlled address calls `acceptAdmin()` → malicious admin retains the role

The admin role controls critical protocol parameters:
- **Entropy**: `setPythFee`, `setDefaultProvider`, `withdrawFee` (direct fund extraction from accrued fees)
- **Scheduler/Pulse**: `setSingleUpdateKeeperFeeInWei`, `setMinimumBalancePerFeed`

A locked-in malicious admin can drain accrued Pyth fees via `withdrawFee`, set an attacker-controlled default provider for Entropy randomness, or manipulate keeper fees to grief Scheduler subscribers. [6](#0-5) 

### Likelihood Explanation
The owner (Pyth governance multisig) cannot atomically call `proposeAdmin` + `acceptAdmin` in a single transaction without a dedicated wrapper contract. Any block between the two transactions is a window for the malicious admin to front-run. On chains with public mempools (all EVM chains where these contracts are deployed), the malicious admin can observe the pending `proposeAdmin` transaction and submit a competing one with higher gas. Likelihood is **medium**: requires a compromised admin key, but once compromised, the attacker has strong incentive and a reliable mechanism to maintain control.

### Recommendation
Replace the two-step `proposeAdmin`/`acceptAdmin` pattern with a single owner-only function that directly sets the admin, analogous to the Compound fix:

```solidity
function setAdmin(address newAdmin) external onlyOwner {
    require(newAdmin != address(0), "newAdmin is zero address");
    address oldAdmin = _state.admin;
    _state.admin = newAdmin;
    emit NewAdminAccepted(oldAdmin, newAdmin);
}
```

If the two-step pattern is retained for safety (to avoid accidentally setting a wrong address), restrict `proposeAdmin` to **owner only**, removing the current admin's ability to call it.

### Proof of Concept

```solidity
// Foundry PoC
function testMaliciousAdminBlocksReplacement() public {
    address maliciousAdmin = admin;       // current admin, now compromised
    address newAdmin       = address(99); // owner's intended replacement
    address attackerAddr   = address(100);// address controlled by malicious admin

    // Step 1: Owner initiates replacement
    vm.prank(owner);
    random.proposeAdmin(newAdmin);
    assertEq(random.proposedAdmin(), newAdmin);

    // Step 2: Malicious admin front-runs before newAdmin calls acceptAdmin
    vm.prank(maliciousAdmin);
    random.proposeAdmin(attackerAddr);    // overwrites proposedAdmin
    assertEq(random.proposedAdmin(), attackerAddr);

    // Step 3: newAdmin's acceptAdmin now reverts
    vm.prank(newAdmin);
    vm.expectRevert(EntropyErrors.Unauthorized.selector);
    random.acceptAdmin();

    // Step 4: Attacker accepts, malicious admin retains control
    vm.prank(attackerAddr);
    random.acceptAdmin();
    assertEq(random.getAdmin(), attackerAddr); // malicious admin wins
}
```
