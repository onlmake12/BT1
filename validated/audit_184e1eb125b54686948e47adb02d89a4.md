### Title
Asymmetric Transaction Size Limits: Consensus-Valid Transactions Cannot Be Enqueued in the Tx-Pool — (File: `tx-pool/src/util.rs`, `util/types/src/core/tx_pool.rs`)

---

### Summary

CKB enforces two different, inconsistent upper bounds on transaction size: the consensus-level `SizeVerifier` (inside `NonContextualTransactionVerifier`) permits a single transaction up to `max_block_bytes` = **597,000 bytes**, while the tx-pool's `non_contextual_verify` additionally enforces a hardcoded `TRANSACTION_SIZE_LIMIT` = **512,000 bytes**. Any transaction in the range (512,001 – ~596,500 bytes) is **consensus-valid** and can be included in a block by a miner who constructs the block manually, but **cannot be submitted to the tx-pool** by any user via RPC or P2P relay. This is the direct CKB analog of the Optimism H01 finding.

---

### Finding Description

**Constant definitions:**

`MAX_BLOCK_BYTES` = `TWO_IN_TWO_OUT_BYTES × TWO_IN_TWO_OUT_COUNT` = 597 × 1,000 = **597,000 bytes** — the consensus total-block-size ceiling. [1](#0-0) 

`TRANSACTION_SIZE_LIMIT` = 512 × 1,000 = **512,000 bytes** — the tx-pool's per-transaction admission ceiling. [2](#0-1) 

**Consensus-level per-transaction check (`SizeVerifier`):**

`NonContextualTransactionVerifier` constructs a `SizeVerifier` with `block_bytes_limit = consensus.max_block_bytes()` (597,000): [3](#0-2) 

`SizeVerifier::verify` passes any transaction whose serialized size ≤ 597,000: [4](#0-3) 

This same verifier is applied to every transaction inside a block during block validation: [5](#0-4) 

**Tx-pool admission path (`non_contextual_verify`):**

The tx-pool first calls `NonContextualTransactionVerifier` (which checks ≤ 597,000), then **additionally** enforces `TRANSACTION_SIZE_LIMIT` = 512,000: [6](#0-5) 

This second check is the binding constraint for all tx-pool submissions — both local RPC (`send_transaction`) and remote P2P relay (`submit_remote_tx`): [7](#0-6) [8](#0-7) 

**Block-level total-size check:**

`BlockBytesVerifier` enforces that the entire serialized block (header + cellbase + transactions + proposals) is ≤ 597,000 bytes: [9](#0-8) 

A block containing a single transaction of ~550,000 bytes has a total size of roughly 550,000 + 208 (header) + ~400 (cellbase) ≈ 550,608 bytes — well within the 597,000-byte block limit. Such a block passes both `SizeVerifier` and `BlockBytesVerifier`.

**The gap:**

| Path | Effective per-tx limit |
|---|---|
| Tx-pool (`non_contextual_verify`) | 512,000 bytes |
| Block consensus (`SizeVerifier` + `BlockBytesVerifier`) | ~596,400 bytes |

Transactions in the range **512,001 – ~596,400 bytes** are consensus-valid but are permanently barred from the tx-pool.

---

### Impact Explanation

A miner who constructs a block manually (bypassing the block assembler, which reads only from the tx-pool) can include a transaction of, say, 550,000 bytes. Every other full node will accept the block — it passes `NonContextualBlockTxsVerifier` (each tx ≤ 597,000) and `BlockBytesVerifier` (total ≤ 597,000). However:

1. **Users cannot submit such transactions** through any standard path (RPC `send_transaction`, P2P relay via `TransactionsProcess`). They receive `ExceededTransactionSizeLimit`.
2. **After a chain reorg**, any such transaction that was in the orphaned chain cannot be re-submitted to the tx-pool and is permanently lost from the mempool.
3. **Asymmetry between miners and users**: miners can include transactions that users have no way to submit, creating a privileged inclusion channel outside normal fee-market competition.

---

### Likelihood Explanation

The attacker is a miner (block producer). Constructing a block manually without the block assembler is non-trivial but well within the capability of any mining pool operator. No majority hashpower, no Sybil attack, and no privileged key is required — only the ability to produce a valid proof-of-work on a manually assembled block. The entry path is fully reachable by any block-producing node.

---

### Recommendation

Align the two limits. Either:

1. **Lower the consensus `SizeVerifier` limit** to match `TRANSACTION_SIZE_LIMIT` (512,000 bytes), so that no transaction larger than 512,000 bytes can appear in any valid block; or
2. **Raise `TRANSACTION_SIZE_LIMIT`** to the actual maximum single-transaction size that can fit in a block (i.e., `max_block_bytes` minus the minimum block overhead for header + cellbase), and enforce this same value in both the tx-pool and the consensus `SizeVerifier`.

Option 1 is the safer fix and directly mirrors the Optimism PR #361 approach: enforce the same upper bound on both the "direct inclusion" path (block) and the "submission" path (tx-pool).

---

### Proof of Concept

```
1. Craft a transaction T with serialized_size_in_block() == 550_000 bytes
   (e.g., a single output with a 549,800-byte data field).

2. Submit via RPC:
     send_transaction(T)
   → Rejected: "Transaction size 550000 exceeded maximum limit 512000"
   (tx-pool/src/util.rs:68-72, TRANSACTION_SIZE_LIMIT = 512_000)

3. Manually assemble a block B:
     B.transactions = [cellbase, T]
     total serialized size ≈ 550_000 + 208 + 400 ≈ 550_608 bytes < 597_000

4. Mine a valid PoW on B and broadcast it.

5. All peers accept B:
   - NonContextualBlockTxsVerifier: 550_000 ≤ 597_000 ✓  (SizeVerifier)
   - BlockBytesVerifier: 550_608 ≤ 597_000 ✓

6. Trigger a reorg that orphans B.
   T cannot be re-submitted to any node's tx-pool → permanently lost.
```

```rust
pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
```

```rust
    pub(crate) async fn process_tx(
        &self,
        tx: TransactionView,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<Completed, Reject> {
        // non contextual verify first
        self.non_contextual_verify(&tx, remote).await?;

        if self.verify_queue_contains(&tx).await || self.orphan_contains(&tx).await {
            return Err(Reject::Duplicated(tx.hash()));
        }

        if let Some((ret, snapshot)) = self
            ._process_tx(tx.clone(), remote.map(|r| r.0), None)
            .await
        {
            self.after_process(tx, remote, &snapshot, &ret).await;
            ret
        } else {
            // currently, the returned cycles is not been used, mock 0 if delay
            Ok(Completed {
                cycles: 0,
                fee: Capacity::zero(),
            })
        }
    }
```
