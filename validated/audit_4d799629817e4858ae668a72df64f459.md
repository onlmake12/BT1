Audit Report

## Title
`Flags::DISCOVERY` Never Cleared When Discovery Protocol Is Disabled — (`util/launcher/src/lib.rs`)

## Summary
In `util/launcher/src/lib.rs`, `flags` is initialized to `Flags::all()` at line 429 and individual bits are removed for absent protocols, but no branch exists to remove `Flags::DISCOVERY` when Discovery is absent from `support_protocols`. Meanwhile, the Discovery handler in `network/src/network.rs` is conditionally registered only when Discovery is present. This mismatch causes any Discovery-disabled node to falsely advertise `Flags::DISCOVERY` in every Identify exchange, polluting the peer store of every peer it connects to.

## Finding Description
In `util/launcher/src/lib.rs`, `flags = Flags::all()` is set at line 429. The code removes `Flags::RELAY` (line 440), `Flags::BLOCK_FILTER` (line 455), and `Flags::LIGHT_CLIENT` (line 475) when those protocols are absent. A grep for `SupportProtocol::Discovery` in `util/launcher/src/lib.rs` returns no matches — there is no corresponding branch for Discovery.

In `network/src/network.rs` lines 897–914, the Discovery handler is only registered when `config.support_protocols.contains(&SupportProtocol::Discovery)`. The two code paths are inconsistent.

When an outbound connection completes the Identify handshake, the remote node's flags are stored directly into the peer store via `peer_store.add_outbound_addr(context.session.address.clone(), flags)` at `network/src/protocols/identify/mod.rs` line 422. Those stored flags are then evaluated by `required_flags_filter` in `fetch_addrs_to_attempt` (line 208) and `fetch_random_addrs` (line 277) of `network/src/peer_store/peer_store_impl.rs`. A Discovery-disabled node stored with `Flags::DISCOVERY` set will pass these filters and be returned to callers seeking Discovery-capable peers, even though the Discovery protocol will never open on that node.

## Impact Explanation
This is a concrete suboptimal implementation of CKB's peer state storage mechanism. The peer store — a core state storage component — is systematically populated with incorrect capability flags for any Discovery-disabled node. Nodes seeking Discovery peers will repeatedly dial addresses that pass the `Flags::DISCOVERY` filter but never open the Discovery protocol, wasting outbound connection slots and degrading routing table quality. The false advertisement propagates transitively via Discovery `Nodes` responses, polluting peer stores across the network. This maps to **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
The default `support_protocols` list in `resource/ckb.toml` line 112 includes Discovery, so the bug is dormant in default deployments. It is triggered by any operator who removes Discovery from `support_protocols`, which the configuration at line 111 explicitly documents as optional ("only 'Sync' and 'Identify' are mandatory, others are optional"). Light-client-only nodes or relay-only nodes following the documented customization path will silently trigger the mismatch. The false flag is set at startup and never corrected for the lifetime of the process.

## Recommendation
Add the missing `else` branch for `Flags::DISCOVERY` in `util/launcher/src/lib.rs`, immediately after the existing pattern for `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT`:

```rust
if support_protocols.contains(&SupportProtocol::Discovery) {
    // Discovery is registered inside NetworkService::new()
} else {
    flags.remove(Flags::DISCOVERY);
}
```

## Proof of Concept
1. Configure a CKB node with `support_protocols` omitting `"Discovery"` (valid per `resource/ckb.toml` line 111).
2. Start the node. Confirm `NetworkService::new()` skips the Discovery handler registration — the `if config.support_protocols.contains(&SupportProtocol::Discovery)` branch at `network/src/network.rs` line 897 is not taken.
3. Connect a second node. Capture the Identify message received by the second node; verify bit `0b10` (`Flags::DISCOVERY`) is set in the advertised flags, because `flags = Flags::all()` at `util/launcher/src/lib.rs` line 429 and no `flags.remove(Flags::DISCOVERY)` is ever executed.
4. Inspect the second node's peer store; confirm the first node's address is stored with `Flags::DISCOVERY` set via `peer_store.add_outbound_addr` at `network/src/protocols/identify/mod.rs` line 422.
5. Issue a `GetNodes` request with `required_flags = Flags::DISCOVERY` from a third node to the second node; confirm the first node's address is returned, passing the `required_flags_filter` check at `network/src/peer_store/peer_store_impl.rs` line 277.
6. Dial the first node from the third node; confirm Sync opens but Discovery never opens, proving the false advertisement.
