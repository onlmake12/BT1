Audit Report

## Title
Unbounded Memory Growth in `pending_get_block_proposals` via Unauthenticated Peer Messages — (`sync/src/types/mod.rs`, `sync/src/relayer/get_block_proposal_process.rs`)

## Summary
`SyncState::pending_get_block_proposals` is an uncapped `DashMap` that any peer completing the standard CKB handshake can grow without bound by sending `GetBlockProposal` relay messages containing random `ProposalShortId` values absent from the local tx-pool. `insert_get_block_proposals` inserts every supplied ID unconditionally, and `drain_get_block_proposals` clones the entire map before clearing it, creating an O(n) heap allocation proportional to attacker-controlled map size, enabling OOM crash or async-task stall.

## Finding Description
`SyncState` initializes `pending_get_block_proposals` as a bare `DashMap::new()` with no bound enforced anywhere in its lifecycle. [1](#0-0) 

`insert_get_block_proposals` iterates every supplied ID and inserts it unconditionally — no size check, no per-peer quota: [2](#0-1) 

`GetBlockProposalProcess::execute` applies only a single per-message length check against `max_block_proposals_limit * max_uncles_num` (≈3,000 IDs). All IDs absent from the tx-pool are forwarded directly to `insert_get_block_proposals` with no further guard: [3](#0-2) 

The periodic drain clones the entire `DashMap` before clearing it — an O(n) heap allocation proportional to the attacker-controlled map size: [4](#0-3) 

By contrast, the `unknown_tx_hashes` path has explicit soft limits (`MAX_UNKNOWN_TX_HASHES_SIZE = 50,000`, `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32,767`), demonstrating the design intent to cap peer-influenced state — an intent absent here: [5](#0-4) 

A `RateLimiter` type is defined in the relayer but is not applied to the `GetBlockProposal` handler: [6](#0-5) 

A grep for `MAX_PENDING_GET_BLOCK_PROPOSALS` or any size guard on `pending_get_block_proposals` returns no matches anywhere in the codebase, confirming the absence of any cap.

## Impact Explanation
A sustained attacker can exhaust node heap memory, causing an OOM crash, or make `drain_get_block_proposals`'s O(n) clone stall the async timer task, blocking proposal-relay processing for all peers. This maps to **High (10,001–15,000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
No privilege is required beyond completing the standard peer handshake. The attacker generates random 10-byte `ProposalShortId` values — no PoW, no valid transaction, no fee. A single peer sending one message per second at 3,000 IDs/message inserts 180,000 entries/minute. Multiple peers multiply this linearly. There is no per-peer frequency limit, no global map cap, and no banning triggered by this code path.

## Recommendation
1. **Cap the map**: Enforce a hard upper bound (e.g., `MAX_PENDING_GET_BLOCK_PROPOSALS`) inside `insert_get_block_proposals`, rejecting or evicting entries when the cap is reached.
2. **Per-peer quota**: Track per-peer contribution counts analogous to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER`; disconnect or ban peers that exceed the quota.
3. **Eliminate the full clone on drain**: Replace the clone-then-clear pattern in `drain_get_block_proposals` with `std::mem::take` or an atomic swap to avoid the O(n) allocation.

## Proof of Concept
```
# Attacker peer loop (one peer, one message/second):
loop:
    ids = [random_bytes(10) for _ in range(3000)]  # all absent from victim tx-pool
    send GetBlockProposal { proposals: ids } to victim node

# Victim node per message:
GetBlockProposalProcess::execute():
    message_len = 3000  # passes limit check (~3000 <= max_block_proposals_limit * max_uncles_num)
    fetch_txs(ids) → all missing
    not_exist_proposals = ids  # all 3000 forwarded
    insert_get_block_proposals(peer, not_exist_proposals)
    # pending_get_block_proposals grows by up to 3000 entries, no cap

# After N messages (N minutes at 1 msg/s):
pending_get_block_proposals.len() == N * 3000
drain_get_block_proposals():
    ret = self.pending_get_block_proposals.clone()  # O(N*3000) heap allocation → OOM or stall
```
