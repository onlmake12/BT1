### Title
Permanent Locking of User Funds in Echo Contract Due to Missing Cancel/Refund Mechanism - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary
The `Echo` contract accepts user fee payments in `requestPriceUpdatesWithCallback` and stores the provider-fee portion in `req.fee`. If the assigned provider never calls `executeCallback`, those funds are permanently locked in the contract with no cancel, timeout, or refund path available to the requester or admin.

---

### Finding Description
In `Echo.requestPriceUpdatesWithCallback`, the user pays `msg.value`. The Pyth protocol fee is immediately credited to `_state.accruedFeesInWei`, while the remainder is stored in `req.fee`:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
```

The provider fee (`req.fee`) is only credited to a provider inside `executeCallback`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

If `executeCallback` is never called for a given request, `req.fee` remains locked in the contract indefinitely. There is no `cancelRequest`, no timeout-based refund, and no admin rescue path for individual `req.fee` balances. The `withdrawFees` admin function only drains `_state.accruedFeesInWei` (Pyth fees), and `withdrawAsFeeManager` only drains `_state.providers[provider].accruedFeesInWei` (credited provider fees). Neither can recover locked `req.fee` amounts.

The developers themselves acknowledge this in inline TODOs:

> "if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."
> "there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback." [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

---

### Impact Explanation
User funds (the provider-fee portion of every request) are permanently locked in the `Echo` contract whenever a provider fails to call `executeCallback`. No withdrawal path exists for the requester, and neither the admin `withdrawFees` nor the fee-manager `withdrawAsFeeManager` can recover these amounts because they operate on separate accounting variables (`_state.accruedFeesInWei` and `_state.providers[p].accruedFeesInWei`) that are never credited from `req.fee` unless fulfillment occurs. [5](#0-4) 

---

### Likelihood Explanation
A malicious provider can register permissionlessly via `registerProvider`, accept requests from users, and simply never call `executeCallback`. The exclusivity period (`exclusivityPeriodSeconds`) only gates which provider may fulfill during a window; it does not force fulfillment or trigger a refund after expiry. Any provider — malicious or merely offline — causes permanent fund loss for every requester who targeted them. [6](#0-5) [7](#0-6) 

---

### Recommendation
Implement a cancel/refund flow:
1. Add a `cancelRequest(uint64 sequenceNumber)` function callable by the requester after a configurable timeout (e.g., `publishTime + timeout`).
2. On cancellation, refund `req.fee` to `req.requester` and clear the request.
3. Optionally slash the assigned provider's `accruedFeesInWei` as a penalty for non-fulfillment, consistent with the existing TODO comment. [8](#0-7) 

---

### Proof of Concept

1. Attacker deploys a contract and calls `Echo.registerProvider(baseFee, feePerFeed, feePerGas)`.
2. Victim calls `Echo.requestPriceUpdatesWithCallback{value: fee}(attackerProvider, publishTime, priceIds, gasLimit)`.
   - `req.fee = msg.value - pythFeeInWei` is stored in the request.
   - `_state.accruedFeesInWei += pythFeeInWei` is credited to Pyth.
3. Attacker never calls `executeCallback`.
4. Victim's `req.fee` is permanently locked. No function in the contract allows the victim to reclaim it:
   - `withdrawFees` → only drains `_state.accruedFeesInWei` (Pyth fees, already credited).
   - `withdrawAsFeeManager` → only drains `_state.providers[attacker].accruedFeesInWei` (zero, since executeCallback was never called).
   - No `cancelRequest` function exists.
5. Funds are irrecoverably locked. [9](#0-8) [10](#0-9)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
