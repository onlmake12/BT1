### Title
Indexer Pool `dead_cells` Set Lacks Reference Counting, Causing Stale Liveness State After RBF — (`util/indexer-sync/src/pool.rs`)

---

### Summary

The `Pool` struct in `util/indexer-sync/src/pool.rs` tracks which cells are consumed by pending transactions using a plain `HashSet<OutPoint>` (`dead_cells`). Because the set has no reference counting or reverse mapping from outpoint to consuming transactions, a Replace-By-Fee (RBF) event can cause the indexer to incorrectly report a cell as live even though a pending transaction is still consuming it. This is a direct structural analog to the reported Canto Identity bug: a forward-only tracking structure with no enforcement of uniqueness, where removing one "owner" silently removes the shared entry for all owners.

---

### Finding Description

`Pool` in `util/indexer-sync/src/pool.rs` is an async overlay that tracks which cells are consumed by pending pool transactions, used by the indexer to filter live cells:

```rust
#[derive(Default)]
pub struct Pool {
    dead_cells: HashSet<OutPoint>,
}
``` [1](#0-0) 

When a new transaction enters the pool, its inputs are inserted:

```rust
pub fn new_transaction(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        self.dead_cells.insert(input.previous_output());
    }
}
``` [2](#0-1) 

When a transaction is rejected, its inputs are unconditionally removed:

```rust
pub fn transaction_rejected(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        self.dead_cells.remove(&input.previous_output());
    }
}
``` [3](#0-2) 

There is no reference count and no reverse mapping from `OutPoint` to the set of transactions consuming it. A `HashSet` insert is idempotent: if two transactions consume the same outpoint, the set still contains only one entry. When either transaction is removed, the single entry is deleted — even if the other transaction is still live in the pool.

The indexer pool is driven by an async `tokio::select!` loop that reads from two independent channels: `new_transaction_receiver` and `reject_transaction_receiver`:

```rust
tokio::select! {
    Some(tx_entry) = new_transaction_receiver.recv() => { ... }
    Some((tx_entry, _reject)) = reject_transaction_receiver.recv() => { ... }
    ...
}
``` [4](#0-3) 

`tokio::select!` picks whichever branch is ready first, non-deterministically. During an RBF replacement, the main pool sends notifications in this order:
1. `reject_transaction(Tx_old)` — old tx evicted
2. `new_transaction(Tx_new)` — replacement tx accepted

Both `Tx_old` and `Tx_new` spend the same cell `C`. Because the two notifications go to separate channels, the indexer's `select!` loop can process them in this order:

| Step | Event processed | `dead_cells` state |
|------|----------------|-------------------|
| 1 | `new_transaction(Tx_old)` (from earlier) | `{C}` |
| 2 | `new_transaction(Tx_new)` | `{C}` (idempotent insert) |
| 3 | `reject_transaction(Tx_old)` | `{}` ← **C incorrectly removed** |

After step 3, `dead_cells` is empty. `Tx_new` is still pending in the main pool consuming cell `C`, but `is_consumed_by_pool_tx(C)` now returns `false`:

```rust
pub fn is_consumed_by_pool_tx(&self, out_point: &OutPoint) -> bool {
    self.dead_cells.contains(out_point)
}
``` [5](#0-4) 

The indexer then serves `C` as a live, spendable cell to any caller of `get_cells` or similar RPC methods, even though it is already consumed by a pending transaction.

---

### Impact Explanation

Any application or user querying the indexer's live-cell set via RPC (e.g., `get_cells`, `get_cells_capacity`) will receive incorrect data: a cell that is already consumed by a pending pool transaction is reported as live and available. This can cause:

- Double-spend attempts by wallets or dApps that trust the indexer's liveness data, leading to transaction failures and wasted fees.
- Incorrect balance or UTXO accounting in downstream applications.
- Incorrect reward or eligibility decisions in protocols that rely on indexer-reported live cells.

The impact is a persistent, silent data integrity violation in the indexer's pool overlay — not a transient read — because the `dead_cells` set is never corrected until the next `new_transaction` or `transaction_committed` event for the same outpoint.

---

### Likelihood Explanation

RBF is a supported and documented feature of the CKB tx-pool. Any unprivileged RPC caller can trigger it by submitting two transactions spending the same cell with a sufficiently higher fee rate. The race condition depends on the async scheduling of `tokio::select!` across two independent channels, which is non-deterministic under load. Under normal node operation with concurrent RPC submissions, the probability of the problematic ordering is non-negligible and increases with pool activity.

---

### Recommendation

Replace `HashSet<OutPoint>` with a reference-counted structure, such as `HashMap<OutPoint, HashSet<Byte32>>` (mapping each outpoint to the set of transaction hashes consuming it). On `new_transaction`, insert the tx hash into the set for each input. On `transaction_rejected` or `transaction_committed`, remove only the tx hash from the set; delete the outpoint entry only when the set becomes empty.

```rust
pub struct Pool {
    // outpoint -> set of tx hashes consuming it
    dead_cells: HashMap<OutPoint, HashSet<Byte32>>,
}

pub fn new_transaction(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        self.dead_cells
            .entry(input.previous_output())
            .or_default()
            .insert(tx.hash());
    }
}

pub fn transaction_rejected(&mut self, tx: &TransactionView) {
    for input in tx.inputs() {
        if let Entry::Occupied(mut e) = self.dead_cells.entry(input.previous_output()) {
            e.get_mut().remove(&tx.hash());
            if e.get().is_empty() {
                e.remove();
            }
        }
    }
}
```

This mirrors the mitigation in the external report: add a reverse mapping so that removing one "owner" does not silently remove the shared entry for all owners.

---

### Proof of Concept

1. Node has RBF enabled (`min_rbf_rate` configured).
2. Cell `C` is live on-chain.
3. Submit `Tx1` spending `C` via `send_transaction` RPC → indexer pool records `dead_cells = {C}`.
4. Submit `Tx2` spending `C` with higher fee via `send_transaction` RPC → main pool evicts `Tx1`, accepts `Tx2`.
5. Due to `tokio::select!` non-determinism, the indexer processes `new_transaction(Tx2)` before `reject_transaction(Tx1)`.
6. After `reject_transaction(Tx1)` is processed, `dead_cells = {}`.
7. Call `get_cells` RPC for the lock script owning `C` → indexer returns `C` as a live cell.
8. `Tx2` is still pending in the pool consuming `C`; the indexer's answer is incorrect.

The `test_orphan_allows_double_spends_of_unknown_input` unit test already demonstrates that the orphan pool intentionally allows two transactions to reference the same outpoint simultaneously. [6](#0-5)  The indexer pool has no analogous guard and no reference counting, making the stale-removal scenario reachable through the RBF path.
