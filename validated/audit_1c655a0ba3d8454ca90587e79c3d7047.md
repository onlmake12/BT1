### Title
Fee Rate Minimum Check Uses Raw Transaction Size Instead of Weight, Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` - (`tx-pool/src/util.rs`)

---

### Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using raw serialized transaction size as the weight argument to `FeeRate::fee()`. However, the canonical transaction weight is `max(size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions the actual weight can be ~60× larger than the size, so the fee check silently accepts transactions whose true fee rate is far below `min_fee_rate`. Any unprivileged tx-pool submitter can exploit this to force expensive script execution at a fraction of the intended cost.

---

### Finding Description

`FeeRate` is defined as shannons per kilo-weight (KW = 1000), where weight is computed by `get_transaction_weight`:

```rust
// util/types/src/core/tx_pool.rs
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,  // 0.000_170_571_4
    )
}
``` [1](#0-0) 

The admission gate `check_tx_fee` ignores cycles entirely and passes only `tx_size` as the weight:

```rust
// tx-pool/src/util.rs
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
``` [2](#0-1) 

`FeeRate::fee` divides by KW = 1000:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    let fee = self.0.saturating_mul(weight) / KW;
    Capacity::shannons(fee)
}
``` [3](#0-2) 

There is no second fee-rate check after script execution completes and actual cycles are known. The transaction is admitted to the pool with whatever cycles the scripts consumed.

**Concrete numbers** (mainnet defaults):

| Parameter | Value |
|---|---|
| `min_fee_rate` | 1,000 shannons/KW |
| `max_tx_verify_cycles` | 70,000,000 |
| `DEFAULT_BYTES_PER_CYCLES` | 0.000_170_571_4 |
| Max weight from cycles | 70,000,000 × 0.000_170_571_4 ≈ **11,940** |

For a 200-byte transaction consuming 70 M cycles:

- **Actual weight**: max(200, 11,940) = **11,940**
- **Fee `check_tx_fee` requires**: 1,000 × 200 / 1,000 = **200 shannons**
- **Fee the weight-correct check would require**: 1,000 × 11,940 / 1,000 = **11,940 shannons**
- **Discount factor**: ~60×

The code comment acknowledges the approximation but treats it as benign:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly" [4](#0-3) 

The assumption that size ≈ weight is the same class of unit-scale assumption error as the Chainlink oracle assuming `tokenFeedDecimals == 8`.

---

### Impact Explanation

An attacker can submit transactions that:

1. **Bypass `min_fee_rate`** by up to ~60× — paying 200 shannons while the correct threshold is 11,940 shannons.
2. **Force expensive script execution** on every receiving node at minimal cost. Each submission triggers up to 70 M cycles of CKB-VM execution before the transaction is admitted (or evicted). The attacker pays ~200 shannons per 70 M cycles of forced computation.
3. **Sustain pool churn** — even if the pool evicts these low-fee-rate entries when full, the attacker can continuously resubmit, keeping nodes busy with script verification at a fraction of the intended fee.

This is a resource-exhaustion DoS against the tx-pool verification pipeline, reachable by any unprivileged RPC caller.

---

### Likelihood Explanation

- Entry point is the standard `send_transaction` RPC, accessible to any peer or RPC caller with no special privileges.
- Crafting a high-cycle script is straightforward (e.g., a loop in CKB-VM).
- The attacker needs only enough CKB to pay ~200 shannons per attack transaction (≈ 0.000002 CKB), making sustained attacks economically trivial.
- No coordination, no privileged keys, no Sybil attack required.

---

### Recommendation

Replace the size-only cheap check with the actual weight calculation:

```rust
// tx-pool/src/util.rs  check_tx_fee
use ckb_types::core::tx_pool::get_transaction_weight;

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
    cycles: Cycle,          // pass actual cycles after script execution
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(...).transaction_fee(rtx)?;
    let weight = get_transaction_weight(tx_size, cycles);
    let min_fee = tx_pool.config.min_fee_rate.fee(weight);
    if fee < min_fee {
        return Err(Reject::LowFeeRate(...));
    }
    Ok(fee)
}
```

If a pre-execution cheap check is still desired for early rejection, it should be clearly separated from the post-execution authoritative check, and the post-execution check must use the real weight.

---

### Proof of Concept

1. Write a CKB lock script that loops for ~70 M cycles (e.g., a tight arithmetic loop in RISC-V).
2. Deploy it on-chain.
3. Create a transaction spending a cell locked by that script. Transaction serialized size ≈ 200 bytes.
4. Set the output capacity so that `fee = min_fee_rate × 200 / 1000 = 200 shannons` (just above the cheap-check threshold).
5. Submit via `send_transaction` RPC.

**Expected (correct) behavior**: rejected with `PoolRejectedTransactionByMinFeeRate` because actual weight ≈ 11,940 and required fee ≈ 11,940 shannons.

**Actual behavior**: accepted into the pool. The node executes 70 M cycles of script for 200 shannons — a ~60× fee discount. Repeating this in a loop exhausts node CPU at negligible cost to the attacker. [5](#0-4) [6](#0-5) [7](#0-6)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}

impl FeeRate {
    /// Calculates the fee rate from a total fee and weight.
    pub fn calculate(fee: Capacity, weight: u64) -> Self {
        if weight == 0 {
            return FeeRate::zero();
        }
        FeeRate::from_u64(fee.as_u64().saturating_mul(KW) / weight)
    }

    /// Creates a fee rate from shannons per kilo-weight.
    pub const fn from_u64(fee_per_kw: u64) -> Self {
        FeeRate(fee_per_kw)
    }

    /// Returns the fee rate as shannons per kilo-weight.
    pub const fn as_u64(self) -> u64 {
        self.0
    }

    /// Creates a zero fee rate.
    pub const fn zero() -> Self {
        Self::from_u64(0)
    }

    /// Calculates the fee for a given weight.
    pub fn fee(self, weight: u64) -> Capacity {
        let fee = self.0.saturating_mul(weight) / KW;
        Capacity::shannons(fee)
    }
