Audit Report

## Title
Stale `total_tx_size`/`total_tx_cycles` Snapshot Overwrite After In-Flight Eviction in `add_entry` Inflates Pool Accounting — (File: tx-pool/src/component/pool_map.rs)

## Summary
`PoolMap::add_entry` snapshots `self.total_tx_size + entry.size` into a local variable before calling `check_and_record_ancestors`, which can internally evict transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, correctly decrementing `self.total_tx_size` in place. After eviction, `add_entry` unconditionally overwrites `self.total_tx_size` and `self.total_tx_cycles` with the stale pre-eviction snapshot, permanently inflating them by the aggregate size/cycles of every evicted transaction. The inflated counters cause `limit_size` to over-evict legitimate transactions with `Reject::Full` even when the pool has physical capacity.

## Finding Description
The exact code path in `add_entry` (lines 200–221):

```
Line 210-211: let (total_tx_size, total_tx_cycles) =
                  self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
Line 213:      evicts = self.check_and_record_ancestors(&mut entry)?;
...
Line 218-219: self.total_tx_size = total_tx_size;
              self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) is a `&self` pure read — it returns `self.total_tx_size.checked_add(tx_size)` without mutating state. [1](#0-0) 

`check_and_record_ancestors` (lines 588–640), when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count` (line 603), calls `remove_entry_and_descendants` in a loop (line 618). [2](#0-1) 

`remove_entry_and_descendants` (lines 252–265) calls `remove_entry` for each removed id (line 263). `remove_entry` (lines 235–250) calls `update_stat_for_remove_tx` at line 247, which directly decrements `self.total_tx_size` and `self.total_tx_cycles` in place. [3](#0-2) 

`update_stat_for_remove_tx` (lines 733–758) writes the decremented values directly to `self.total_tx_size` / `self.total_tx_cycles`. [4](#0-3) 

After eviction, `add_entry` lines 218–219 unconditionally overwrite those correctly-decremented fields with the stale snapshot. [5](#0-4) 

Let `X = self.total_tx_size` before `add_entry`:
- Snapshot: `total_tx_size = X + entry.size`
- After eviction of size `E`: `self.total_tx_size = X - E`
- After line 218: `self.total_tx_size = X + entry.size` (inflation of `E`)
- Correct value should be: `X - E + entry.size`

Each invocation of the eviction branch inflates `self.total_tx_size` by `E` permanently (until node restart).

## Impact Explanation
`limit_size` loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size` and evicts the lowest-fee transactions. With an inflated counter, it evicts additional legitimate pending/proposed transactions that would otherwise remain. Repeated triggering progressively shrinks the effective pool capacity toward zero, preventing legitimate transactions from entering the mempool and degrading transaction throughput across the network. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The eviction branch is reachable by any unprivileged user via the `send_transaction` RPC or P2P relay. It requires submitting a transaction whose ancestor count exceeds `max_ancestors_count` (default 1,000) but whose excess ancestors are all cell-dep-referencing parents. Building such a chain has a fee cost, but because each trigger permanently inflates the counter, the attacker's cost is amortized across repeated triggers. The bug also fires in non-adversarial conditions whenever organic chains organically hit this path.

## Recommendation
Remove the pre-computed snapshot and instead apply the addition directly to `self.total_tx_size`/`self.total_tx_cycles` after `check_and_record_ancestors` completes:

```rust
// Remove lines 210-211 (the pre-snapshot).
// Retain updated_stat_for_add_tx as a pre-flight overflow check only,
// discarding its returned values:
self.updated_stat_for_add_tx(entry.size, entry.cycles)?; // overflow guard only

evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Apply increment to post-eviction state:
self.total_tx_size = self.total_tx_size.saturating_add(entry.size);
self.total_tx_cycles = self.total_tx_cycles.saturating_add(entry.cycles);
```

## Proof of Concept
1. Build a pool with `N` legitimate transactions totaling size just below `max_tx_pool_size`.
2. Construct a chain of 1,000 transactions where all are cell-dep parents of `T_target`, so `ancestors_count = 1,001 > 1,000` and `1,001 - 1,000 = 1 ≤ 1,000`.
3. Submit `T_target`. `check_and_record_ancestors` evicts one cell-dep parent of size `S` via `remove_entry_and_descendants`, correctly setting `self.total_tx_size -= S`. Lines 218–219 then overwrite with the stale snapshot, inflating `self.total_tx_size` by `S`.
4. `limit_size` is called immediately after; it observes `total_tx_size > max_tx_pool_size` and evicts one additional legitimate transaction.
5. Repeat steps 2–4: each iteration inflates `total_tx_size` by `S` and causes one additional legitimate transaction to be evicted, until the pool is effectively unusable.
6. A unit test can assert `pool_map.total_tx_size == expected_post_eviction_size` after a single `add_entry` call that triggers the eviction branch, and observe the inflation directly.

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
