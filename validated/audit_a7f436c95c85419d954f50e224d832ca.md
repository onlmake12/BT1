### Title
VAA Consumed Without Price Feed Creation via `update_size=0` in `parse_and_verify_accumulator_updates` — (`target_chains/sui/contracts/sources/pyth_accumulator.move`)

### Summary
`parse_and_verify_accumulator_updates` reads `update_size` as a single u8 with no lower-bound assertion. If an attacker crafts an accumulator message with `update_size=0`, the while loop is never entered, an empty `vector<PriceInfo>` is returned, and `create_price_feeds_using_accumulator` completes successfully — consuming the Wormhole VAA — without creating any price feed objects.

### Finding Description
In `parse_and_verify_accumulator_updates`, the number of price updates is read as a u8 with no assertion that it is greater than zero: [1](#0-0) 

The caller `create_price_feeds_using_accumulator` passes the returned (possibly empty) vector directly to `create_and_share_price_feeds_using_verified_price_infos` with no non-empty check: [2](#0-1) 

`create_and_share_price_feeds_using_verified_price_infos` itself silently no-ops on an empty vector: [3](#0-2) 

The VAA is consumed unconditionally via `vaa::take_payload(vaa)` before the empty vector is discovered: [4](#0-3) 

The embedded VAA bytes inside the accumulator message body are explicitly skipped and not validated against the outer `vaa` parameter: [5](#0-4) 

An attacker can therefore: (1) observe any valid Wormhole VAA from the public Wormhole network, (2) call `wormhole::vaa::parse_and_verify()` to obtain a `VAA` Move object, (3) craft an accumulator message with `update_size=0x00` immediately after the VAA section, and (4) call `create_price_feeds_using_accumulator` — the transaction succeeds, the VAA is consumed, and zero price feeds are created.

### Impact Explanation
The attacker can front-run any legitimate relayer attempting to initialize price feeds via `create_price_feeds_using_accumulator`. Each valid VAA can only be used once (Wormhole replay protection). By consuming VAAs with `update_size=0`, the attacker prevents price feed `PriceInfoObject`s from ever being created for new price identifiers. Downstream consumers that depend on those objects (e.g., DeFi protocols calling `get_price`) will be permanently blocked until a new VAA is published — which the attacker can immediately consume again. This is a sustained liveness DoS on price feed initialization.

### Likelihood Explanation
VAAs are publicly observable on the Wormhole Guardian network before any relayer submits them on-chain. The attacker only needs to front-run with a higher gas price. The crafted message is trivial to construct (change one byte). The cost is gas only. No privileged access, leaked keys, or governance majority is required.

### Recommendation
Add an assertion in `parse_and_verify_accumulator_updates` that `update_size > 0`:

```move
let update_size = deserialize::deserialize_u8(cursor);
assert!(update_size > 0, E_INVALID_UPDATE_DATA);
```

Alternatively, assert in `create_price_feeds_using_accumulator` that the returned `price_infos` vector is non-empty before consuming the VAA.

### Proof of Concept
1. Observe a valid accumulator VAA from the Wormhole network (e.g., `TEST_ACCUMULATOR_SINGLE_FEED` from the test suite).
2. Call `wormhole::vaa::parse_and_verify(worm_state, vaa_bytes, clock)` to obtain a `VAA` object.
3. Craft `malicious_accumulator_message`: identical header/major/minor/trailing/proof_type/vaa_size/vaa fields as the legitimate message, but replace the `update_size` byte (first byte after the VAA section) with `0x00`, and truncate everything after it.
4. Call `pyth::create_price_feeds_using_accumulator(pyth_state, malicious_accumulator_message, verified_vaa, clock, ctx)`.
5. Assert: transaction succeeds, zero `PriceInfoObject`s are created (shared_ids is empty), and the VAA is consumed (legitimate relayer's subsequent call with the same VAA fails). [6](#0-5)
