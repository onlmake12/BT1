### Title
Provider Can Front-Run User Requests via Unrestricted `setProviderFee`, Causing DoS or Unrefunded Overpayment - (File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol)

### Summary
The Pyth Entropy contract allows any registered provider (or their designated fee manager) to change their `feeInWei` at any time with no timelock or delay via `setProviderFee`. Because the fee is evaluated at request execution time and excess `msg.value` is explicitly **not refunded**, a provider can front-run a user's pending `request`/`requestV2` transaction to either (a) raise the fee and cause the user's transaction to revert with `InsufficientFee`, or (b) lower the fee so the user overpays and the surplus is permanently credited to Pyth protocol fees rather than returned to the user.

### Finding Description

`setProviderFee` in `Entropy.sol` immediately and atomically updates `provider.feeInWei` with no delay:

```solidity
// Entropy.sol line 810
function setProviderFee(uint128 newFeeInWei) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) revert EntropyErrors.NoSuchProvider();
    uint128 oldFeeInWei = provider.feeInWei;
    provider.feeInWei = newFeeInWei;
    ...
}
``` [1](#0-0) 

Inside `requestHelper`, the required fee is computed from the **current** on-chain state at execution time, and the check `msg.value < requiredFee` reverts if the user sent too little:

```solidity
// Entropy.sol line 234-235
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
``` [2](#0-1) 

When the user overpays (fee was lowered), the surplus is credited to `_state.accruedPythFeesInWei` — not returned to the caller:

```solidity
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [3](#0-2) 

The interface explicitly documents this non-refund behavior:

> "Note that provider fees can change over time... excess value is *not* refunded to the caller." [4](#0-3) 

The same unrestricted fee-change capability is available to the fee manager via `setProviderFeeAsFeeManager`: [5](#0-4) 

### Impact Explanation

**Attack path A — fee raised (DoS):**
1. User (or consumer contract) calls `getFeeV2(provider, gasLimit)` off-chain and submits `requestV2{value: fee}(provider, gasLimit)`.
2. Provider (or fee manager) observes the pending transaction in the mempool and front-runs it with `setProviderFee(newHigherFee)` at a higher gas price.
3. User's transaction executes with `msg.value < requiredFee` → reverts with `InsufficientFee`.
4. User loses gas; the randomness request fails. Consumer contracts that depend on the callback (e.g., on-chain games, lotteries, DeFi protocols) have their critical operation silently aborted.

**Attack path B — fee lowered (unrefunded overpayment):**
1. User queries `getFeeV2` and sends `msg.value = oldFee`.
2. Provider front-runs with `setProviderFee(lowerFee)`.
3. User's transaction succeeds but `(oldFee - lowerFee)` is permanently credited to Pyth protocol fees — the user loses the difference with no recourse.

### Likelihood Explanation

The attack requires a registered provider to act against their own users. Providers are external, permissionless entities (any address can call `register`). A malicious or compromised provider key can execute this at any time. Consumer contracts that cache the fee across transactions (a common pattern shown in the official docs) are particularly exposed. The mempool-based front-running is straightforward on any non-private-mempool chain where Pyth Entropy is deployed. [6](#0-5) 

### Recommendation

1. Introduce a **fee-change timelock**: when `setProviderFee` is called, record the new fee and a future activation timestamp (e.g., `block.timestamp + MIN_FEE_CHANGE_DELAY`). The `requestHelper` fee check should use the currently active fee, not the pending one.
2. Alternatively, **refund excess `msg.value`** to the caller in `requestHelper` so that fee decreases between query and execution do not result in user fund loss.
3. Emit a `ProviderFeeChangeScheduled` event so off-chain monitors and consumer contracts can react before the change takes effect.

### Proof of Concept

```solidity
// 1. Provider registers with fee = 1 ether
vm.prank(provider);
entropy.register(1 ether, commitment, metadata, 100, uri);

// 2. User queries fee and prepares transaction
uint128 fee = entropy.getFeeV2(provider, 0); // returns 1 ether + pythFee

// 3. Provider front-runs: raises fee to 2 ether
vm.prank(provider);
entropy.setProviderFee(2 ether);

// 4. User's transaction executes with old fee amount → reverts
vm.deal(user, fee);
vm.prank(user);
vm.expectRevert(EntropyErrors.InsufficientFee.selector);
entropy.requestV2{value: fee}(provider, bytes32(0), 0);
// User lost gas; request never created.
```

    // Set provider fee. It will revert if provider is not registered.
    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);

            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
