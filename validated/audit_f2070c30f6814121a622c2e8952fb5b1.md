Audit Report

## Title
Tx-Pool Admission Check Uses Raw `tx_size` Instead of Actual Weight, Allowing Cycle-Heavy Transactions to Bypass `min_fee_rate` — (File: `tx-pool/src/util.rs`)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` computes the minimum required fee using only the serialized byte size (`tx_size`) as the weight. The canonical weight used everywhere else in the pool is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. After `verify_rtx` returns the actual cycle count, no second fee-rate check is performed, so cycle-heavy transactions enter the pool and are relayed to peers at a fraction of the intended minimum fee rate.

## Finding Description

**Root cause — `check_tx_fee` (`tx-pool/src/util.rs` L42-52):**

The function explicitly acknowledges the limitation in a comment and uses `tx_size` alone:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { … }
``` [1](#0-0) 

`FeeRate::fee(weight)` computes `fee_rate * weight / KW` (`util/types/src/core/fee_rate.rs` L34-36), so `min_fee = min_fee_rate × tx_size / 1000`. [2](#0-1) 

**Canonical weight definition (`util/types/src/core/tx_pool.rs` L298-303):**

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
``` [3](#0-2) 

with `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4` at L279. [4](#0-3) 

**No second fee-rate check after verification — primary path (`_process_tx`, `tx-pool/src/process.rs` L715-751):**

`pre_check` (which calls `check_tx_fee`) runs at L715, `verify_rtx` runs at L724-732, and then `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is created directly at L751 with no post-verification fee-rate gate. [5](#0-4) 

**No second fee-rate check after verification — re-add path (`readd_detached_tx`, L878-905):**

`check_tx_fee` is called at L889, `verify_rtx` at L895-903, and `TxEntry::new` at L905 — again with no post-verification fee-rate check. [6](#0-5) 

**Divergence at runtime:**

`TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs` L115-118) correctly calls `get_transaction_weight(self.size, self.cycles)`, so the entry's recorded fee rate is far below `min_fee_rate` for cycle-heavy transactions — but the transaction is already admitted. [7](#0-6) 

**Concrete numeric example:**

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 (max) |
| `actual_weight` | `max(200, 70M × 0.000_170_571_4)` = **11,940** |
| `min_fee_rate` | 1,000 shannons/KW (default) |
| `min_fee` from `check_tx_fee` | `1,000 × 200 / 1,000` = **200 shannons** |
| Actual fee rate with fee = 200 | `200 × 1,000 / 11,940` ≈ **16.7 shannons/KW** |

The transaction passes the admission gate paying ~60× less than the minimum required fee rate.

## Impact Explanation

This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

An attacker can flood the tx-pool and P2P relay network with cycle-heavy transactions at ~1/60th of the intended minimum cost. These transactions pass `check_tx_fee` legitimately, are relayed to all peers (which accept them via the same lenient gate), occupy pool space and consume relay bandwidth at a fraction of the intended cost floor, and displace legitimate transactions during eviction (since eviction uses the correct weight-based fee rate via `EvictKey`). The `min_fee_rate` anti-spam mechanism is effectively nullified for the entire cycle-heavy transaction class. [8](#0-7) 

## Likelihood Explanation

No privilege is required. Any node's `send_transaction` RPC endpoint or P2P relay path accepts the transaction. Crafting a small-serialized, high-cycle transaction (a script with a tight loop) is straightforward. The default `max_tx_verify_cycles = 70,000,000` gives a worst-case weight ratio of ~60×, making the bypass significant, reliable, and repeatable by any external actor.

## Recommendation

After `verify_rtx` returns the actual cycle count, perform a second fee-rate check using the real weight in both `_process_tx` and `readd_detached_tx`:

```rust
// After verify_rtx succeeds (in both _process_tx and readd_detached_tx):
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let actual_min_fee = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < actual_min_fee {
    return Err(Reject::LowFeeRate(
        tx_pool.config.min_fee_rate,
        actual_min_fee.as_u64(),
        fee.as_u64(),
    ));
}
```

This mirrors the pattern already used correctly in `TxEntry::fee_rate()` (`tx-pool/src/component/entry.rs` L116) and `FeeRateCollector::statistics()` (`rpc/src/util/fee_rate.rs` L103). [7](#0-6) [9](#0-8) 

## Proof of Concept

1. Write a CKB lock script that executes a tight loop consuming ~70M cycles.
2. Build a transaction spending a cell locked by that script; serialized size ~200 bytes.
3. Set output capacity so that `inputs_capacity − outputs_capacity = 200 shannons` (fee).
4. Submit via `send_transaction` RPC to a node with default `min_fee_rate = 1,000 shannons/KW`.
5. `check_tx_fee` computes `min_fee = 1,000 × 200 / 1,000 = 200 shannons` → **passes**.
6. `verify_rtx` confirms actual cycles ≈ 70M; `TxEntry` is created with `actual_weight ≈ 11,940`.
7. `entry.fee_rate()` = `200 × 1,000 / 11,940 ≈ 16.7 shannons/KW` — far below the 1,000 shannons/KW minimum.
8. The transaction is in the pool and relayed to peers, all of which accept it through the same lenient gate.
9. Repeat with many such transactions to exhaust pool capacity and relay bandwidth at ~1/60th of the intended cost floor. [10](#0-9) [11](#0-10)

pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
) -> Result<Capacity, Reject> {
    let fee = DaoCalculator::new(snapshot.consensus(), &snapshot.borrow_as_data_loader())
        .transaction_fee(rtx)
        .map_err(|err| {
            Reject::Malformed(
                format!("{err}"),
                "expect (outputs capacity) <= (inputs capacity)".to_owned(),
            )
        })?;
    // Theoretically we cannot use size as weight directly to calculate fee_rate,
    // here min fee rate is used as a cheap check,
    // so we will use size to calculate fee_rate directly
    let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
    // reject txs which fee lower than min fee rate
    if fee < min_fee {
        let reject =
            Reject::LowFeeRate(tx_pool.config.min_fee_rate, min_fee.as_u64(), fee.as_u64());
        ckb_logger::debug!("Reject tx {}", reject);
        return Err(reject);
    }
    Ok(fee)
}
