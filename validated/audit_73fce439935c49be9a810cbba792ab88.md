### Title
No Mechanism to Change Admin Address in Echo Contract — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The Echo contract stores an `admin` address that gates all critical governance functions. Unlike the sibling Entropy and Scheduler contracts, which both implement a two-step `proposeAdmin`/`acceptAdmin` transfer mechanism, the Echo contract has no function to change the admin address. If the admin key is lost or compromised, accrued Pyth fees are permanently locked and all governance actions become permanently inaccessible.

---

### Finding Description

In `EchoState.sol`, the `State` struct contains an `admin` field set once during `_initialize()`: [1](#0-0) 

Three critical governance functions in `Echo.sol` are gated exclusively behind a direct `msg.sender == _state.admin` check — not the owner, only the admin: [2](#0-1) [3](#0-2) [4](#0-3) 

There is no `setAdmin`, `proposeAdmin`, or any equivalent function anywhere in the Echo contract hierarchy. The `ProviderInfo` struct also has no admin-change path: [5](#0-4) 

By direct contrast, both sibling contracts implement a proper two-step admin transfer:

- `EntropyGovernance.sol` has `proposeAdmin()`/`acceptAdmin()`: [6](#0-5) 

- `SchedulerGovernance.sol` has the identical pattern: [7](#0-6) 

Furthermore, both Entropy and Scheduler authorize admin actions for **either** the owner or the admin: [8](#0-7) [9](#0-8) 

Echo's checks are hardcoded to `_state.admin` only, so even the contract owner cannot perform admin actions or rotate the admin key.

---

### Impact Explanation

If the Echo admin key is lost or compromised:

1. **Accrued Pyth fees permanently locked** — `_state.accruedFeesInWei` can never be withdrawn because `withdrawFees()` requires `msg.sender == _state.admin`.
2. **Default provider frozen** — `setDefaultProvider()` becomes permanently inaccessible, preventing the protocol from switching away from a misbehaving or defunct provider.
3. **Exclusivity period frozen** — `setExclusivityPeriod()` cannot be adjusted, removing an important DoS-mitigation lever.

The only escape hatch would be a contract upgrade, but that requires the owner (a separate role), not the admin — and upgrades cannot retroactively recover already-accrued fees that were locked under the old admin.

---

### Likelihood Explanation

Admin key loss or compromise is a realistic operational risk for any long-lived on-chain contract. The absence of a rotation mechanism means a single key-management failure creates a permanent, unrecoverable loss of governance capability and potentially locked funds. The risk is low in probability but high in irreversibility.

---

### Recommendation

Implement a two-step admin transfer mechanism in Echo, mirroring the existing pattern in `EntropyGovernance.sol`:

```solidity
function proposeAdmin(address newAdmin) external {
    require(msg.sender == owner() || msg.sender == _state.admin, "Unauthorized");
    require(newAdmin != address(0), "newAdmin is zero address");
    _state.proposedAdmin = newAdmin;
}

function acceptAdmin() external {
    require(msg.sender == _state.proposedAdmin, "Not proposed admin");
    _state.admin = msg.sender;
    _state.proposedAdmin = address(0);
}
```

Also update the admin-gated function checks to allow the owner as a fallback (matching the Entropy/Scheduler pattern), so the owner can act as a recovery path if the admin key is lost.

---

### Proof of Concept

1. Echo is deployed; `_state.admin` is set to `adminAddress` at initialization.
2. `adminAddress` private key is lost or compromised.
3. Attacker or no one can call `withdrawFees()`, `setDefaultProvider()`, or `setExclusivityPeriod()` — all revert with `"Only admin can ..."`.
4. All accrued Pyth fees in `_state.accruedFeesInWei` are permanently locked.
5. No function exists in the Echo contract to rotate the admin to a new address.
6. The contract owner cannot perform these actions either, as the checks are `msg.sender == _state.admin` (not `owner()`), unlike Entropy and Scheduler.
