Audit Report

## Title
Expired Alert ID Not Recorded in `cancel_filter` Enables Replay Re-Propagation — (`util/network-alert/src/notifier.rs`)

## Summary
`clear_expired_alerts` removes expired entries from `received_alerts` and `noticed_alerts` but never writes their IDs into `cancel_filter`. After expiry, `has_received(id)` returns `false` for those IDs. An attacker holding the raw signed bytes of any previously valid alert can reconnect, replay those bytes, pass signature verification (which covers only `calc_alert_hash()`, not expiry), and cause the victim node to re-broadcast the alert to all connected peers — repeatable indefinitely.

## Finding Description
`Notifier` maintains two deduplication structures: `received_alerts: HashMap<u32, Alert>` and `cancel_filter: LruCache<u32, ()>`.

`has_received` at `notifier.rs` L147–149 gates every incoming alert: [1](#0-0) 

`clear_expired_alerts` at `notifier.rs` L135–144 silently drops expired entries from both maps without touching `cancel_filter`: [2](#0-1) 

By contrast, `cancel` at L125–127 correctly writes to `cancel_filter` before removing: [3](#0-2) 

The exploit path in `alert_relayer.rs`:

1. On every new peer connection, `connected()` at L87 calls `self.clear_expired_alerts()`, resetting deduplication state for all expired IDs: [4](#0-3) 

2. The `received` handler at L147 checks `has_received` — returns `false` for the cleared ID: [5](#0-4) 

3. `verify_signatures` at `verifier.rs` L35 signs only over `calc_alert_hash()` — no expiry or timestamp field is included in the message — so the original signatures remain valid: [6](#0-5) 

4. The handler broadcasts to all connected peers and calls `notifier.add(&alert)`, re-inserting the expired alert: [7](#0-6) 

`notifier.add` performs no expiry check before inserting into `received_alerts` and calling `notify_network_alert`: [8](#0-7) 

The existing test at `test_notifier.rs` L101–104 explicitly asserts `!notifier.has_received(1)` after expiry, confirming the deduplication gap is present: [9](#0-8) 

## Impact Explanation
An attacker with stored bytes of any previously valid signed alert can connect to arbitrary nodes, replay the alert, and cause network-wide re-propagation. Each victim node broadcasts to all its connected peers, producing unbounded fan-out across the P2P graph. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — the attacker requires no key material, no hashpower, and no special privilege; only previously observed network traffic and the ability to make P2P connections.

## Likelihood Explanation
Every node online during the original alert broadcast received and could have stored the raw signed bytes. The trigger (`clear_expired_alerts` on peer connect) fires automatically on every inbound connection with no timing precision required. The attack is repeatable indefinitely: connect → replay → disconnect → reconnect. A single attacker node suffices to initiate the cascade; no colluding nodes are needed.

## Recommendation
In `clear_expired_alerts`, record each removed ID into `cancel_filter` before dropping it, mirroring the existing `cancel` path:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ());
            false
        } else {
            true
        }
    });
    self.noticed_alerts.retain(|a| {
        let notice_until: u64 = a.raw().notice_until().into();
        notice_until > now
    });
}
```

This permanently suppresses expired IDs in `cancel_filter`, consistent with how explicitly cancelled alerts are handled in `cancel`.

## Proof of Concept
1. Run a victim node. Observe a valid alert with `id=42`, `notice_until=T` broadcast on the network; store the raw signed bytes.
2. Advance time past `T`.
3. Connect a peer to the victim. `connected()` fires `clear_expired_alerts(now > T)`, removing `id=42` from `received_alerts`. `cancel_filter` is unchanged.
4. Send the stored alert bytes over the P2P alert protocol.
5. Victim: `has_received(42)` → `false` (not in `received_alerts`, not in `cancel_filter`).
6. Victim: `verify_signatures` → `Ok(())` (signatures still valid over original `calc_alert_hash()`).
7. Victim: broadcasts to all connected peers; calls `notifier.add(&alert)` → re-inserts `id=42`, calls `notify_network_alert`.
8. Each receiving peer repeats steps 5–7, propagating the stale alert network-wide.
9. Repeat from step 3 to re-trigger indefinitely.

Unit test to confirm the gap: after `notifier.add(&alert)` and `notifier.clear_expired_alerts(after_expired_time)`, assert `notifier.has_received(id)` is `true` — this assertion currently fails, directly reproducing the bug, as demonstrated by the existing test at `test_notifier.rs` L101–104 which asserts the opposite.
