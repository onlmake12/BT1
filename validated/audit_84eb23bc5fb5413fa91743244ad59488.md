Audit Report

## Title
Synchronous Blocking VM Execution in `estimate_cycles` Starves RPC Tokio Runtime â€” (`rpc/src/module/chain.rs`)

## Summary

`CyclesEstimator::run()` calls `ScriptVerifier::verify(max_block_cycles)` synchronously on the RPC tokio runtime with no `spawn_blocking` or cycle cap. Concurrent requests exhaust the RPC runtime's worker threads, causing full RPC unavailability. The impact is limited to the RPC layer: contrary to the report's claim, the RPC server runs on a **dedicated** tokio runtime separate from P2P and block sync.

## Finding Description

`CyclesEstimator::run()` at `rpc/src/module/chain.rs` L2375â€“2405 executes the full VM synchronously using `consensus.max_block_cycles` as the limit with no cap or offloading:

```rust

    .verify(max_cycles)
```

The `handle_jsonrpc` async handler in `rpc/src/server.rs` calls `io.handle_call(call, T::default()).await` inline at L258 with no offloading. A search for `spawn_blocking`, `block_in_place`, `rayon`, or `tokio::task` in `rpc/src/**` returns zero results. A `TimeoutLayer` of 30 seconds at `server.rs` L125â€“128 caps individual requests but does not prevent sustained saturation via continuous replacement.

**Correction to the submitted report:** The `threads` config field is not unused. `ckb-bin/src/subcommand/run.rs` L92â€“96 reads `args.config.rpc.threads` and passes it to `new_global_runtime(Some(rpc_threads_num))` at L27, creating a **dedicated RPC tokio runtime** (`rpc_handle`). The RPC server is started with this dedicated handle at `util/launcher/src/lib.rs` L571. P2P networking uses `shared.async_handle()` (the main runtime) at L508, which is entirely separate. Blocking the RPC runtime does **not** stall P2P or block sync.

## Impact Explanation

Saturating the dedicated RPC runtime's worker threads causes full RPC unavailability, including `get_block_template` for miners and all other RPC methods. This matches **Note (0â€“500 points): Any local RPC API crash**. The submitted High-level impact claim (node crash via shared runtime starvation of P2P/sync) is not supported: the runtimes are isolated.

## Likelihood Explanation

Default listen address is `127.0.0.1`, restricting remote exploitation. Nodes that expose RPC publicly (wallets, dApps, public infrastructure) are directly vulnerable remotely. The attacker needs only HTTP access to the RPC port and a live cell referencing a max-cycle script. No authentication or privilege is required. The attack is repeatable and cheap to sustain by continuously replacing expiring requests before the 30-second timeout.

## Recommendation

- Offload `ScriptVerifier::verify()` to `tokio::task::spawn_blocking` or a bounded rayon pool inside `CyclesEstimator::run()`
- Add a configurable `max_cycles_for_estimate` in `RpcConfig` defaulting to a fraction of `max_block_cycles`
- Add a global or per-IP concurrency semaphore for compute-heavy RPC methods (`estimate_cycles`, `dry_run_transaction`)

## Proof of Concept

1. Compile a CKB script that executes a tight loop consuming exactly `consensus.max_block_cycles` cycles
2. Deploy it as a live cell on a node with public RPC
3. Craft an `estimate_cycles` JSON-RPC payload referencing that cell
4. Send `(rpc_thread_count + 2)` concurrent HTTP POST requests to the RPC endpoint
5. Simultaneously poll any other RPC method (e.g., `get_tip_block_number`) â€” observe request timeouts for the duration of the attack
6. Confirm recovery only after all `estimate_cycles` requests expire (30s timeout) and no new ones are sent
