Audit Report

## Title
Unvalidated `listen_addrs` in Hole-Punching Protocol Enables Unbounded Resource Exhaustion — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

## Summary
An unprivileged connected peer can supply up to 24 attacker-controlled TCP addresses (including loopback and RFC-1918 ranges) in a `ConnectionRequest` message. The victim stores them without IP-range validation, then on receipt of a `ConnectionSync` spawns 24 concurrent 30-second TCP retry loops against those addresses. By cycling fresh peer IDs the attacker bypasses all rate-limiting and interval guards, accumulating unbounded async tasks and socket churn that can exhaust the victim node's file descriptors and async runtime, crashing the node.

## Finding Description

**Root cause — no IP-range filter in `respond_delivered`:**

`ConnectionRequestProcess::execute` calls `respond_delivered` when `self_peer_id == &content.to`. [1](#0-0) 

Inside `respond_delivered`, the only filter applied to the attacker-supplied addresses is `TransportType::Tcp` with an `Ip4` or `Ip6` component. There is no check for loopback (`127.x.x.x`), unspecified (`0.0.0.0`/`::`), link-local, or RFC-1918 private ranges. [2](#0-1) 

Up to `ADDRS_COUNT_LIMIT = 24` such addresses are stored verbatim in `pending_delivered`: [3](#0-2) [4](#0-3) 

**Trigger — `ConnectionSync` spawns unbounded tasks:**

When the attacker sends a `ConnectionSync` with `from=attacker_peer_id, to=victim_peer_id`, `ConnectionSyncProcess::execute` retrieves the poisoned addresses and spawns a `runtime::spawn` task containing 24 concurrent `try_nat_traversal` futures via `select_ok`, with no additional address validation. [5](#0-4) [6](#0-5) 

**`try_nat_traversal` — 30-second socket-churning retry loop:**

Each future converts the multiaddr to a `SocketAddr` and enters a loop with a ~200 ms interval for a total of 30 seconds, creating and connecting a new `TcpSocket` on every iteration (~150 iterations × 24 addresses = ~3,600 socket operations per request). [7](#0-6) 

If a connection succeeds (e.g., a local service on `127.0.0.1:8114`), the stream is registered as an inbound P2P session: [8](#0-7) 

**Rate-limiting is ineffective against fresh peer IDs:**

The per-session `rate_limiter` (30 req/s) is keyed on `(session_id, msg.item_id())` — a new TCP connection yields a fresh session ID and a fresh allowance. [9](#0-8) 

The `forward_rate_limiter` (1 req/s) is keyed on `(from_peer_id, to_peer_id, msg_item_id)` — a new `from_peer_id` bypasses it entirely. [10](#0-9) 

The `HOLE_PUNCHING_INTERVAL` guard (2 minutes) is also keyed on `from_peer_id` via `pending_delivered` — same bypass. [11](#0-10) 

`pending_delivered` entries are only evicted after 5 minutes (`TIMEOUT`), so poisoned entries accumulate across many fresh-peer-ID requests within that window. [12](#0-11) 

## Impact Explanation

**High (10001–15000 points) — "Vulnerabilities which could easily crash a CKB node."**

Each attacker request (one `ConnectionRequest` + one `ConnectionSync` from a fresh peer ID) spawns 24 concurrent async tasks that each create a new OS socket every ~200 ms for 30 seconds. With N simultaneous fresh-peer-ID connections, N × 24 tasks run concurrently, each cycling through ~150 socket create/connect/drop operations. This rapidly exhausts the OS file-descriptor limit and saturates the async runtime's task pool, causing the victim node to fail to accept new P2P connections and ultimately become unresponsive or crash. The secondary impact — TCP connections to loopback/private addresses including the default RPC port 8114 — is real but falls outside the allowed bounty impact classes.

## Likelihood Explanation

- Requires only a standard P2P connection; no key, PoW, or privileged role.
- The victim's peer ID is public (exchanged during the identify handshake).
- The two-message sequence (`ConnectionRequest` → `ConnectionSync`) is the normal protocol flow; no protocol violation is needed.
- Generating fresh peer IDs to bypass all rate-limiting and interval guards is trivial (a new TCP connection suffices).
- The attack is repeatable and automatable.

## Recommendation

In `respond_delivered`, before inserting into `pending_delivered`, reject any address whose IP component is loopback, unspecified, link-local, or RFC-1918 private:

```rust
fn is_globally_routable(ip: IpAddr) -> bool {
    !ip.is_loopback() && !ip.is_unspecified()
        && !ip.is_multicast() && !is_private(ip)
}
```

Apply this predicate when building `remote_listens` at lines 196–215 of `connection_request.rs`. Apply the same guard in `try_nat_traversal` (`mod.rs`) before the retry loop as a defence-in-depth measure. Additionally, consider bounding the total number of concurrently live `try_nat_traversal` tasks across all peers to prevent resource exhaustion even from globally-routable addresses.

## Proof of Concept

```
1. Connect to victim as peer A (any valid peer ID, fresh TCP connection).

2. Send ConnectionRequest {
       from: peer_A_id,
       to:   victim_peer_id,
       listen_addrs: [
           /ip4/127.0.0.1/tcp/1,  /ip4/127.0.0.1/tcp/2,
           ... (24 entries total, all closed ports)
       ],
       max_hops: 0,
       route: []
   }
   → victim stores (peer_A_id → [24 addrs]) in pending_delivered.

3. Send ConnectionSync { from: peer_A_id, to: victim_peer_id, route: [] }
   → victim spawns runtime::spawn with select_ok over 24 try_nat_traversal futures.
   → each future loops for 30 s, creating a TcpSocket every ~200 ms.

4. Disconnect and reconnect as peer B (new session_id, new from_peer_id).
   Repeat steps 2–3.

5. Repeat with peers C, D, … in rapid succession.
   → N × 24 concurrent 30-second socket-churning tasks accumulate on the victim.
   → File-descriptor exhaustion and async-runtime saturation cause the node to
      stop accepting connections and eventually crash.

Variant: replace port 1 with 8114 (default RPC) to confirm open/closed status
via timing side-channel or connection acceptance.
```
