### Title
Provider Fee Permanently Locked in Unfulfilled Echo Requests with No Refund Mechanism - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`, when a user calls `requestPriceUpdatesWithCallback`, the provider fee portion of `msg.value` is stored in `req.fee` inside the request struct. This fee is only credited to a provider when `executeCallback` is later called. If `executeCallback` is never called (provider offline, fee too low to incentivize fulfillment, etc.), the provider fee is permanently locked in the contract. There is no `cancelRequest`, timeout, or refund function available to the requester.

---

### Finding Description

In `requestPriceUpdatesWithCallback`, the incoming `msg.value` is split into two parts:

1. The Pyth protocol fee is immediately credited to `_state.accruedFeesInWei`: [1](#0-0) 

2. The remainder — the provider fee — is stored inside the request struct as `req.fee`: [2](#0-1) 

The provider fee is only ever credited to a provider's withdrawable balance inside `executeCallback`: [3](#0-2) 

If `executeCallback` is never called for a given sequence number, `req.fee` remains trapped in the request struct indefinitely. The `clearRequest` helper only zeroes the sequence number or deletes the overflow entry — it does not transfer the fee anywhere: [4](#0-3) 

The admin's `withdrawFees` function only drains `_state.accruedFeesInWei` (the Pyth protocol portion), not the per-request `req.fee` fields: [5](#0-4) 

`withdrawAsFeeManager` similarly only touches `_state.providers[provider].accruedFeesInWei`, which is never populated for unfulfilled requests: [6](#0-5) 

The `IEcho` interface exposes no `cancelRequest`, `refundRequest`, or equivalent function: [7](#0-6) 

The developers themselves acknowledge the gap with a TODO comment inside `executeCallback`:

```
// TODO: there also needs to be some penalty mechanism in case the expected provider
// doesn't execute the callback.
``` [8](#0-7) 

This is structurally identical to the RAACNFT bug: funds flow into the contract (`req.fee`) with no code path that returns them to the payer when the expected counterparty action never occurs.

---

### Impact Explanation

Any user who calls `requestPriceUpdatesWithCallback` and whose request is never fulfilled loses the provider fee portion of their payment permanently. The funds are held in the request struct, unreachable by the requester, the provider, the fee manager, or the admin. The Pyth fee portion is safe (already credited), but the provider fee — which is the larger component for high-gas-limit callbacks — is irrecoverable.

**Severity: High** — direct, permanent loss of user funds during normal protocol operation.

---

### Likelihood Explanation

The exclusivity period (`_state.exclusivityPeriodSeconds`) means that after it expires, any provider can fulfill the request. However, fulfillment is economically voluntary. If the fee set at request time is insufficient to cover gas at fulfillment time, or if the assigned provider is offline and no other provider is willing to step in, the request will never be fulfilled. There is no on-chain expiry, no automatic refund, and no keeper obligation. The likelihood is **medium-high** given that fee levels are fixed at request time while gas prices fluctuate.

---

### Recommendation

Add a `cancelRequest` function that allows the original requester to reclaim `req.fee` after a configurable timeout has elapsed since `publishTime`:

```solidity
function cancelRequest(uint64 sequenceNumber) external {
    Request storage req = findActiveRequest(sequenceNumber);
    require(msg.sender == req.requester, "Not requester");
    require(
        block.timestamp > req.publishTime + CANCEL_TIMEOUT,
        "Too early to cancel"
    );
    uint96 refund = req.fee;
    clearRequest(sequenceNumber);
    (bool sent, ) = msg.sender.call{value: refund}("");
    require(sent, "Refund failed");
    emit RequestCancelled(sequenceNumber, msg.sender, refund);
}
```

Alternatively, mirror the Entropy contract's design where fees are credited to the provider immediately at request time (not deferred to fulfillment), so no funds are ever stranded in a request struct.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `msg.value = getFee(provider, gasLimit, priceIds)`. The provider fee is stored in `req.fee`; the Pyth fee is credited to `_state.accruedFeesInWei`.
2. The assigned provider goes offline. The exclusivity period expires. No other provider fulfills the request because the fee is below current gas cost.
3. The request sits in `_state.requests[shortKey]` or `_state.requestsOverflow` indefinitely.
4. The user calls every function in `IEcho` — none allows them to recover `req.fee`.
5. The admin calls `withdrawFees` — it only drains `_state.accruedFeesInWei`, leaving `req.fee` untouched.
6. `req.fee` is permanently locked.

     * @param publishTime The minimum publish time for price updates, it should be less than or equal to block.timestamp + 60
     * @param priceIds The price feed IDs to update. Maximum 10 price feeds per request.
     *        Requests requiring more feeds should be split into multiple calls.
     * @param callbackGasLimit The amount of gas allocated for the callback execution
     * @return sequenceNumber The sequence number assigned to this request
     * @dev Security note: The 60-second future limit on publishTime prevents a DoS vector where
     *      attackers could submit many low-fee requests for far-future updates when gas prices
     *      are low, forcing executors to fulfill them later when gas prices might be much higher.
     *      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
     *      the fee estimation unreliable.

     * @param providerToCredit The provider to credit for fulfilling the request. This may not be the provider that submitted the request (if the exclusivity period has elapsed).
