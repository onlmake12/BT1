### Title
Zero-Price `continue` Skip Silently Blocks All Deviation-Triggered Updates — (`File: target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

In `Scheduler.sol`, the `_validateShouldUpdatePrices` function iterates over price feeds to decide whether a deviation-based update should fire. A `continue` guard at line 429–431 skips any feed whose stored previous price **or** incoming current price is zero. When every feed in a subscription is skipped, the loop exits without ever returning `updateTimestamp`, and the function falls through to an unconditional `revert UpdateConditionsNotMet()`. For subscriptions configured with `updateOnDeviation: true` and `updateOnHeartbeat: false`, this permanently freezes the subscription: no keeper can push a new update, and downstream consumers are left with stale prices indefinitely.

---

### Finding Description

`_validateShouldUpdatePrices` evaluates two independent triggers in sequence: heartbeat (lines 400–410) and deviation (lines 413–451). The deviation branch loops over all price feeds:

```solidity
// Skip if either price is zero to avoid division by zero
if (previousPrice == 0 || currentPrice == 0) {
    continue;                          // ← line 429-431
}
``` [1](#0-0) 

If every feed in the subscription satisfies the skip condition, the `for` loop completes without executing `return updateTimestamp`, and control falls to:

```solidity
revert SchedulerErrors.UpdateConditionsNotMet();
``` [2](#0-1) 

Two concrete trigger paths exist:

**Path A — stored previous price is zero.** A prior update stored a zero price (Pyth legitimately reports zero for a delisted or halted asset). On every subsequent call, `previousPrice == 0` fires `continue` for that feed. If all feeds share this state, the subscription is permanently frozen.

**Path B — incoming current price is zero.** Pyth reports a zero price in the new update data. `currentPrice == 0` fires `continue`. Ironically, a price collapsing to zero represents a 100 % deviation — the most extreme possible move — yet the guard silently discards it and blocks the update.

The heartbeat branch (lines 400–410) is evaluated **before** the deviation branch, so it can rescue a subscription only if `updateOnHeartbeat: true` and the heartbeat interval has elapsed. Subscriptions configured with deviation-only triggers have no fallback. [3](#0-2) 

The `SubscriptionParams` struct explicitly supports deviation-only configuration (`updateOnHeartbeat: false`, `updateOnDeviation: true`), and the validation logic enforces that at least one criterion is set — meaning deviation-only is a valid, expected configuration. [4](#0-3) 

---

### Impact Explanation

Any active subscription using `updateOnDeviation`-only is permanently unable to receive price updates once any subscribed feed reports a zero price. The `_state.priceUpdates` mapping retains the stale data. Readers calling `getPricesNoOlderThan` will eventually receive a `StalePrice` revert; readers calling `getPricesUnsafe` will silently receive the frozen stale value. DeFi protocols consuming these prices may execute trades, liquidations, or collateral valuations against incorrect data, leading to direct financial loss. [5](#0-4) 

---

### Likelihood Explanation

Pyth price feeds can legitimately publish a zero `price` field for halted markets, delisted tokens, or during oracle outages. A keeper (unprivileged updater) submits valid, Pyth-signed `updateData` through the public `updatePriceFeeds` entry point — no special role is required. Once a zero price is stored, the subscription is frozen without any on-chain signal or event indicating the failure mode. The subscription manager has no in-protocol recovery path short of deactivating and recreating the subscription. [6](#0-5) 

---

### Recommendation

Replace the `continue` with a branch that explicitly handles the zero-price case as a maximum-deviation trigger (since any transition to or from zero is an infinite or 100 % deviation):

```solidity
if (previousPrice == 0 || currentPrice == 0) {
    // A price at zero represents maximum deviation; always trigger.
    return updateTimestamp;
}
```

If the intent is strictly to avoid division by zero while skipping the feed, the loop must track whether **any** non-skipped feed was evaluated, and treat an all-skipped result as a trigger (or at minimum as a non-revert pass-through) rather than silently falling to `UpdateConditionsNotMet`.

---

### Proof of Concept

1. Deploy `Scheduler` and create a subscription:
   - `updateOnHeartbeat: false`
   - `updateOnDeviation: true`, `deviationThresholdBps: 100`
   - One price feed `feedId`

2. Submit a first valid Pyth update where `price.price = 1000` for `feedId`. This succeeds (no previous price → `previousFeed.id == bytes32(0)` → returns immediately).

3. Submit a second valid Pyth update where `price.price = 0` for `feedId`.
   - `previousPrice = 1000`, `currentPrice = 0`
   - Line 429: `currentPrice == 0` → `continue`
   - Loop exits; falls to `revert UpdateConditionsNotMet()`
   - **Transaction reverts. Price is not updated.**

4. Submit any further update (even with `price.price = 9999`, a 900 % deviation):
   - `previousPrice = 1000` (still stored from step 2), `currentPrice = 9999`
   - Deviation = 899 % >> 1 % threshold → should trigger
   - But step 3 never stored the zero price, so this update **does** succeed.

   Alternatively, if step 3 had succeeded (e.g., via a different code path), `previousPrice` would be `0`, and step 4 would fire `continue` → permanent freeze. [7](#0-6)
