Audit Report

## Title
Unconditional `.expect()` on `txs_sizes: None` in `FeeRateCollector::statistics` Panics on Legacy `BlockExt` Records - (File: `rpc/src/util/fee_rate.rs`)

## Summary
`FeeRateCollector::statistics` calls `.expect(...)` on `txs_sizes` at line 93 without guarding against `None`. The legacy deserialization path for pre-v0.106 5-field `BlockExt` records unconditionally sets `txs_sizes: None`, making the panic deterministic for any node whose 101-block statistics window includes such records. The panic aborts the Tokio RPC task and drops the client connection; the node process itself continues running.

## Finding Description
`BlockExt` was extended in v0.106 from a 5-field molecule table to a 7-field `BlockExtV1` table. In `util/types/src/conversion/storage.rs` lines 139–151, the `Unpack<core::BlockExt> for packed::BlockExtReader` implementation hardcodes `txs_sizes: None` and `cycles: None` for legacy records. In `store/src/store.rs` lines 247–263, `get_block_ext` dispatches on `count_extra_fields()`: a value of `0` routes to the old reader (returning `txs_sizes: None`), while `2` routes to `BlockExtV1Reader` (which properly unpacks both fields). In `rpc/src/util/fee_rate.rs` line 93, `FeeRateCollector::statistics` calls `txs_sizes.expect("expect txs_size's length >= 1")` with no `None` guard. Any block in the 1–101 block window stored in the old format triggers the panic. Both RPC entry points `get_fee_rate_statics` and `get_fee_rate_statistics` in `rpc/src/module/chain.rs` lines 2124–2132 call `FeeRateCollector::new(...).statistics(...)` directly with no intervening error handling. The server at `rpc/src/server.rs` line 133 is spawned as a Tokio task; Tokio catches the unwind at the task boundary, so the node process survives.

## Impact Explanation
**Note (0–500 points): Any local RPC API crash.** The panic terminates the Tokio task handling the RPC connection, dropping the client connection, but the node process itself continues running. No `panic = "abort"` profile is set in the workspace. The claimed "High" severity (full node crash / denial of service) is not supported by the code behavior.

## Likelihood Explanation
Requires a node upgraded from a version prior to v0.106 with fewer than 101 blocks mined after the upgrade, so that the statistics window includes legacy 5-field `BlockExt` records. On mainnet this is practically impossible. On testnets, private networks, or nodes that were offline for a long time and then upgraded, the condition is realistic. Any RPC caller (including unauthenticated ones if the RPC port is exposed) can trigger the panic with a single call, and it is repeatable on every call while legacy blocks remain in the statistics window.

## Recommendation
Replace the unconditional `.expect(...)` at `rpc/src/util/fee_rate.rs` line 93 with a guard that skips blocks where `txs_sizes` is `None`:
```rust
let txs_sizes = match txs_sizes {
    Some(sizes) if !sizes.is_empty() => sizes,
    _ => return fee_rates, // skip legacy blocks lacking txs_sizes
};
```
This makes statistics collection gracefully skip pre-v0.106 blocks rather than panicking.

## Proof of Concept
1. Run a CKB node upgraded from pre-v0.106 on a private or test network with fewer than 101 blocks mined after the upgrade.
2. Send:
   ```json
   {"id": 1, "jsonrpc": "2.0", "method": "get_fee_rate_statistics", "params": []}
   ```
3. The Tokio task handling the request panics at `rpc/src/util/fee_rate.rs:93` with `expect("expect txs_size's length >= 1")`. The connection is dropped; the node process continues. Repeatable on every call while legacy blocks remain in the statistics window.
