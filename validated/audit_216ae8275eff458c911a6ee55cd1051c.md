### Title
`ScriptHashTypeVerifier` Only Validates Lock Script Hash Types, Silently Skipping Type Script Hash Types — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and checks only `output.lock().hash_type()`, completely omitting the type script (`output.type_()`). This mirrors the `getUsdPrice()` pattern: the function only handles one "path" (lock scripts) and silently passes for the other (type scripts), allowing transactions carrying non-permitted type-script hash types to bypass the cheap non-contextual gate and enter the tx pool.

---

### Finding Description

`ScriptHashTypeVerifier` is composed into `NonContextualTransactionVerifier` and is documented as verifying that the `ScriptHashType` of **transaction outputs** is within the consensus-permitted range. [1](#0-0) 

The permitted set is: [2](#0-1) 

`{0=Data, 1=Type, 2=Data1, 4=Data2}`. The `ScriptHashType` enum, however, is generated for all even values 0–254 plus 1: [3](#0-2) 

Values such as `Data3=6`, `Data4=8`, … are valid enum variants but are **not** in `ENABLED_SCRIPT_HASH_TYPE`.

The implementation of `verify()` is:

```rust
for output in self.transaction.outputs() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) { … }
    } …
}
``` [4](#0-3) 

`output.type_()` is never consulted. A transaction output carrying `hash_type = 6` (`Data3`) in its **type script** passes this verifier without error.

The downstream `select_version()` in the script engine does catch it: [5](#0-4) 

But that check runs only inside the expensive `ContextualTransactionVerifier` / script verifier, which executes **asynchronously after** the transaction has already been admitted to the tx pool. [6](#0-5) 

---

### Impact Explanation

An unprivileged tx-pool submitter can craft transactions that:

1. Pass `NonContextualTransactionVerifier` (including `ScriptHashTypeVerifier`) because the lock script is valid.
2. Are inserted into the pending pool, consuming pool slots and memory.
3. Trigger the asynchronous `ContextualTransactionVerifier`, wasting CPU on script-group construction and `select_version()` dispatch before rejection.

By flooding the node with such transactions the attacker can saturate the tx pool's bounded capacity (`max_tx_pool_size`) and starve legitimate transactions from being admitted, constituting a resource-exhaustion / DoS against the tx-pool admission path. The attack is cheap for the sender (no valid script execution required) and expensive for the node (full contextual verification pipeline per transaction).

---

### Likelihood Explanation

Any peer reachable via the `send_transaction` JSON-RPC endpoint can trigger this. No privileged role, key material, or majority hash power is required. Crafting a transaction with a syntactically valid structure but `hash_type = 6` in the type script is trivial using any CKB SDK. The gap between the cheap gate and the expensive gate is the exploitable window.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also inspect the type script of each output:

```rust
for output in self.transaction.outputs() {
    // existing lock-script check …

    // add: type-script check
    if let Some(type_script) = output.type_().to_opt() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: type_script.hash_type(),
            }.into());
        }
    }
}
```

This keeps the rejection at the cheap, synchronous, non-contextual stage — consistent with the stated purpose of `ScriptHashTypeVerifier` — and prevents invalid type-script hash types from ever reaching the tx pool.

---

### Proof of Concept

1. Build a `TransactionView` with:
   - One input spending any live cell.
   - One output whose **lock** script uses `hash_type = 0` (Data, permitted) and whose **type** script uses `hash_type = 6` (Data3, not in `ENABLED_SCRIPT_HASH_TYPE`).
   - Sufficient capacity so `check_tx_fee` passes.
2. Submit via `send_transaction` RPC.
3. `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()` iterates outputs, checks only `output.lock().hash_type() = 0` → **passes**.
4. Transaction enters the pending pool.
5. Asynchronous `ContextualTransactionVerifier` calls `select_version()` on the type script → `ScriptHashType::Data3` hits the catch-all arm → `Err(InvalidScriptHashType)` → transaction evicted.
6. Repeat at high rate to keep the pool saturated with eviction-pending entries, blocking admission of legitimate transactions. [1](#0-0) [2](#0-1) [5](#0-4)

// Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }
}
