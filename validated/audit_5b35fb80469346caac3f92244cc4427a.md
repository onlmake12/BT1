### Title
`ScriptHashTypeVerifier` Fails to Validate Output Type Script Hash Types, Allowing Bypass of Early Malformed-Transaction Detection - (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` is documented to enforce that all script hash types in transaction outputs are within the consensus-permitted set (`ENABLED_SCRIPT_HASH_TYPE`). However, it only inspects the **lock script** of each output and completely omits the **type script**. A transaction sender can craft an output whose type script carries a non-permitted hash type (e.g., `Data3 = 6`) and the verifier will silently pass it. The transaction is eventually rejected at script-execution time, but under a different error code that is **not** classified as a malformed transaction, so the submitting peer escapes the peer-score penalty that would otherwise be applied.

---

### Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the four consensus-permitted hash-type values:

```
0u8  // Data
1u8  // Type
2u8  // Data1
4u8  // Data2
```

Any other even value (e.g., `6 = Data3`, `8 = Data4`, …) is explicitly not permitted.

`ScriptHashTypeVerifier::verify()` in `verification/src/transaction_verifier.rs` is supposed to enforce this for every output in a transaction. Its implementation is:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
    }
    Ok(())
}
```

The loop body calls `output.lock().hash_type()` — the **lock** script — and never touches `output.type_()`. An output whose type script carries `Data3` (or any other non-permitted hash type) passes this verifier without error.

The mismatch is exact: the verifier lists four permitted hash types and claims to enforce them for "transaction outputs", but only enforces them for one of the two scripts that an output can carry.

---

### Impact Explanation

**Peer-score bypass / resource drain.** When a lock script carries a non-permitted hash type, `ScriptHashTypeVerifier` returns `TransactionError::ScriptHashTypeNotPermitted`, which `is_malformed_tx()` classifies as `true`. The submitting peer is penalised and eventually banned.

When a **type** script carries a non-permitted hash type, `ScriptHashTypeVerifier` returns `Ok(())`. The transaction proceeds to script execution, where `TxInfo::select_version` (or `extract_script_and_dep_index`) returns `ScriptError::InvalidScriptHashType`. That error surfaces as a script-execution failure, **not** as `TransactionError::ScriptHashTypeNotPermitted`, so `is_malformed_tx()` returns `false` and the peer is **not** penalised.

An unprivileged transaction sender can therefore submit an unbounded stream of structurally invalid transactions (invalid type-script hash type) without triggering the peer-ban mechanism, forcing the node to run the full non-contextual and contextual verification pipeline — including script-group construction — on each one before finally rejecting it.

**Likelihood:** High. The entry path is the standard `send_transaction` RPC or P2P relay. No privilege is required. The crafted transaction is trivially constructable: any valid transaction with one output whose type script's `hash_type` byte is set to `6` (Data3) suffices.

---

### Likelihood Explanation

Any unprivileged RPC caller or P2P relay peer can submit transactions. Constructing a transaction with an output type script whose `hash_type` byte is `6` requires no special knowledge or access. The node will accept it past `ScriptHashTypeVerifier`, spend resources on further verification, and only reject it deep in the script-execution path — without penalising the sender.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script of each output when one is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // Check lock script hash type (existing)
        check_hash_type(output.lock().hash_type())?;

        // Check type script hash type (missing today)
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

where `check_hash_type` encapsulates the existing `TryInto` + `ENABLED_SCRIPT_HASH_TYPE` logic. This ensures that any output with a non-permitted type-script hash type is rejected immediately with `ScriptHashTypeNotPermitted`, triggering the correct peer-score penalty.
