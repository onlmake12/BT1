### Title
Unauthenticated `from` PeerId in `ConnectionSync` Enables Attacker-Directed Outbound TCP Connections via Spoofed `pending_delivered` Lookup — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

---

### Summary

A single unprivileged peer connected to a victim CKB node can cause the victim to make outbound TCP connections to arbitrary attacker-controlled IP addresses by exploiting two missing identity validations in the hole-punching protocol: (1) the `from` field of a `ConnectionRequest` is never checked against the actual session peer, allowing `pending_delivered` to be seeded with attacker-controlled listen addresses under an arbitrary spoofed PeerId X; (2) the `from` field of a subsequent `ConnectionSync` is also never checked against the session peer, so the victim blindly looks up `pending_delivered[X]` and initiates NAT traversal toward those addresses.

---

### Finding Description

**Phase 1 — Seeding `pending_delivered` with attacker-controlled addresses**

In `ConnectionRequestProcess::execute`, when `self_peer_id == &content.to` (i.e., the victim is the intended target), the code calls `respond_delivered(content.from, ...)` where `content.from` is taken verbatim from the wire message: [1](#0-0) 

There is no check that `content.from` matches the actual session peer identity. The `remote_listens` stored are the listen addresses from the same message, filtered only to TCP+IP: [2](#0-1) 

An attacker connected to the victim sends `ConnectionRequest { from=X, to=<victim_peer_id>, listen_addrs=[attacker_IPs] }`. The victim stores `pending_delivered[X] = ([attacker_IPs], now)` and sends a `ConnectionRequestDelivered` back to the attacker's actual session — confirming success.

**Phase 2 — Triggering NAT traversal via spoofed `ConnectionSync`**

In `ConnectionSyncProcess::execute`, when `self_peer_id == &content.to`, the code looks up `pending_delivered` keyed on `content.from` — again taken verbatim from the wire message with no session-identity check: [3](#0-2) 

It then spawns an async task that calls `try_nat_traversal` for every address in the retrieved list, making real outbound TCP connections: [4](#0-3) [5](#0-4) 

The attacker sends `ConnectionSync { from=X, to=<victim_peer_id> }`. The victim finds `pending_delivered[X]` (populated in Phase 1), and initiates outbound TCP SYN packets to the attacker-controlled IPs.

**Why rate limiting does not prevent this**

The `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)`: [6](#0-5) 

Since the attacker controls `content.from`, they can rotate through distinct spoofed PeerIds to bypass the rate limit entirely. The per-session `rate_limiter` (keyed on `(session_id, item_id)`) limits message frequency per session but does not prevent the two-message attack sequence.

**Why the `HOLE_PUNCHING_INTERVAL` guard does not prevent this**

The 2-minute deduplication check in `respond_delivered` is keyed on `from_peer_id`: [7](#0-6) 

The attacker can use a fresh spoofed PeerId X for each attempt, bypassing this guard.

---

### Impact Explanation

The victim node makes outbound TCP connections to arbitrary IP:port pairs chosen by the attacker. Concrete impacts:

- **Port scanning / network reconnaissance**: the victim's IP is used to probe internal or external hosts
- **SSRF**: the victim connects to internal services (e.g., metadata endpoints, admin interfaces) reachable from the victim's network but not the attacker's
- **Connection slot exhaustion**: by rotating many spoofed PeerIds, an attacker can saturate the victim's outbound connection pool, degrading legitimate P2P connectivity across the network

---

### Likelihood Explanation

The attack requires only a single standard P2P connection to the victim — no special privileges, no PoW, no key material. The victim's PeerId is publicly discoverable. The two-message sequence (`ConnectionRequest` then `ConnectionSync`) is straightforward to craft. The `pending_delivered` entry persists for up to 5 minutes (`TIMEOUT`), giving ample time to send the follow-up `ConnectionSync`. [8](#0-7) 

---

### Recommendation

1. **Validate `from` against the actual session peer in `ConnectionRequest`**: when `self_peer_id == &content.to`, assert that `content.from` matches the PeerId of the actual connected session (`self.peer`). Reject messages where `content.from` does not match.

2. **Validate `from` against the actual session peer in `ConnectionSync`**: before looking up `pending_delivered[content.from]`, verify that `content.from` corresponds to the PeerId of the session that delivered the message.

3. **Key `pending_delivered` on the authenticated session PeerId**, not the wire-supplied `from` field, so that the stored listen addresses are always bound to a verified peer identity.

---

### Proof of Concept

```
Attacker (peer A, connected to Victim V):

Step 1 — Seed pending_delivered:
  Send ConnectionRequest {
    from  = <fabricated PeerId X>,
    to    = <victim's real PeerId>,
    listen_addrs = [/ip4/192.168.1.100/tcp/9999]   // attacker-controlled
  }
  → Victim stores: pending_delivered[X] = ([/ip4/192.168.1.100/tcp/9999], now)
  → Victim sends ConnectionRequestDelivered back to session A (confirms success)

Step 2 — Trigger NAT traversal:
  Send ConnectionSync {
    from  = <same fabricated PeerId X>,
    to    = <victim's real PeerId>,
    route = []
  }
  → Victim: self_peer_id == content.to → look up pending_delivered[X]
  → Victim: spawns try_nat_traversal([/ip4/192.168.1.100/tcp/9999])
  → Victim: sends outbound TCP SYN to 192.168.1.100:9999

Repeat with fresh X values to bypass rate limiting and HOLE_PUNCHING_INTERVAL.
```
