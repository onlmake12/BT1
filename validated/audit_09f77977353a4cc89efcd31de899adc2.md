## Impact Explanation
An attacker sending a sustained stream of maximal `GetBlocksProof` messages forces the target node to perform thousands of synchronous RocksDB reads and CPU-intensive MMR proof computations per message with no throttle. This can saturate the node's storage I/O and CPU, starving Sync and Relay protocol handlers and halting the node's participation in block propagation. This matches the allowed impact: **Vulnerabilities which could easily crash a CKB node (High, 10001â€“15000 points)** and **Vulnerabilities or bad designs which could cause CKB network congestion with few costs (High, 10001â€“15000 points)**.

## Likelihood Explanation
The attacker requires only a standard P2P connection to a default-configured CKB node and knowledge of 1000 main-chain block hashes, which are trivially obtained by syncing block headers. No proof-of-work, key material, or privilege is needed. The attack is fully repeatable and can be parallelized across multiple connections up to the default `max_peers` limit. The LightClient protocol is on by default, so no special node configuration is required on the victim side.

## Recommendation
- Introduce a per-peer sliding-window rate limit on `GetBlocksProof` (and similarly `GetTransactionsProof`) messages, e.g., a token-bucket allowing N requests per second per peer.
- Apply a short ban or exponential backoff when a peer exceeds the rate limit.
- Consider reducing `GET_BLOCKS_PROOF_LIMIT` or requiring the requester to demonstrate a valid light-client sync state before serving large proofs.
- Evaluate whether `mmr.gen_proof` for large position sets should be bounded or deferred to a background task with a concurrency cap.
