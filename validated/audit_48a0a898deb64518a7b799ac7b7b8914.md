Audit Report

## Title
`ScriptHashTypeVerifier` Fails to Validate Output Type Script `hash_type` Against Enabled Allowlist — (File: `verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and enforces that each output's **lock** script `hash_type` is within `ENABLED_SCRIPT_HASH_TYPE`, but never inspects the output's **type** script. Any unprivileged user can submit a transaction whose output carries a type script with a structurally valid but not-yet-activated `hash_type` (e.g., `6` = `Data3`). The transaction passes all non-contextual gates, is accepted into the tx-pool, and can be mined. The resulting live cell is permanently unspendable, inflating the live-cell set with provably dead UTXOs.

## Finding Description

`ENABLED_SCRIPT_HASH_TYPE` in `util/constant/src/consensus.rs` defines the four activated values `{0, 1, 2, 4}`: [1](#0-0) 

`ScriptHashTypeVerifier::verify()` loops over outputs and checks only `output.lock().hash_type()`: [2](#0-1) 

`output.type_()` is never read. A type script with `hash_type = 6` passes `verify_value` (even number → structurally valid) and is never inspected by `ScriptHashTypeVerifier`. The `NonContextualTransactionVerifier` documentation itself acknowledges only the lock script is checked: [3](#0-2) 

When any subsequent transaction attempts to consume the resulting cell, `select_version()` in `script/src/types.rs` hits the catch-all arm and returns `Err(ScriptError::InvalidScriptHashType(...))`: [4](#0-3) 

This makes the cell permanently unspendable. The exploit path is: craft output with lock `hash_type=1` (valid) and type `hash_type=6` (not activated) → submit via `send_raw_transaction` → passes `ScriptHashTypeVerifier` → accepted into pool → mined → live cell is forever frozen.

## Impact Explanation

The impact matches **Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism**. Every such transaction permanently adds dead cells to the live-cell set. The live-cell set is CKB's primary state storage structure; cells that can never be consumed are never pruned, inflating it indefinitely. While the attacker burns their own CKB, the state bloat is irreversible and cumulative — any number of such cells can be created at the cost of the minimum cell capacity, permanently degrading the storage invariant that live cells are spendable.

## Likelihood Explanation

No privileges, special tooling, or coordination are required. Any user who can call `send_raw_transaction` (a public RPC endpoint) and construct a raw transaction can trigger this. The structural validity check (`verify_value`) passes for `hash_type = 6` (even number), and the only guard that should catch it (`ScriptHashTypeVerifier`) does not inspect type scripts. The attack is repeatable at will.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock script check
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err(TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }.into());
        }
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
                }
            } else {
                return Err(TransactionError::InvalidScriptHashType {
                    hash_type: type_script.hash_type(),
                }.into());
            }
        }
    }
    Ok(())
}
```

## Proof of Concept

1. Build a raw transaction with one output: lock script `hash_type = 1` (Type, valid), type script `hash_type = 6` (Data3, not activated).
2. Submit via `send_raw_transaction` RPC.
3. Observe no `ScriptHashTypeNotPermitted` rejection — the transaction enters the pool.
4. Mine the transaction into a block.
5. Construct a spending transaction consuming the output.
6. Observe the node rejects it with `InvalidScriptHashType` / `InvalidVmVersion`, confirming the cell is permanently unspendable.

The root cause is confirmed at `ScriptHashTypeVerifier::verify()` [2](#0-1) , the allowlist enforced only for lock scripts [1](#0-0) , and the runtime rejection that makes the cell unspendable [4](#0-3) .
