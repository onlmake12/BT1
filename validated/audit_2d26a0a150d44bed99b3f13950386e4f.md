Audit Report

## Title
O(N²) Descendant/Ancestor Traversal in `remove_expired` via Sequential `remove_entry` Calls — (`tx-pool/src/component/pool_map.rs`, `tx-pool/src/pool.rs`)

## Summary

`remove_expired` removes all expired transactions by calling `remove_entry` sequentially. Each `remove_entry` call invokes `update_descendants_index_key` and `update_ancestors_index_key` before `remove_entry_links`, meaning the full link graph is intact during each traversal. For a chain of N transactions, sequential removal produces O(N²) total traversal work. This occurs inside `_update_tx_pool_for_reorg` while holding the tx-pool write lock, blocking all concurrent tx-pool operations on every affected node simultaneously.

## Finding Description

**Root cause — ordering in `remove_entry`:**

`remove_entry` calls `update_descendants_index_key` (full BFS over all descendants) and `update_ancestors_index_key` (full BFS over all ancestors) *before* `remove_entry_links`, so the link graph is fully intact during both traversals:

```
remove_entry (pool_map.rs:235-250)
  → update_ancestors_index_key   ← full ancestor BFS
  → update_descendants_index_key ← full descendant BFS
  → remove_entry_edges
  → remove_entry_links           ← links removed LAST
```

`update_descendants_index_key` (pool_map.rs:447-460) calls `self.links.calc_descendants`, which is an unbounded BFS in `calc_relation_ids` (links.rs:52-72) with no iteration cap.

**Contrast with the existing optimization in `remove_entry_and_descendants`:**

`remove_entry_and_descendants` (pool_map.rs:252-265) pre-removes all links *before* calling `remove_entry`, so `update_descendants_index_key` finds an empty descendant set. `remove_expired` does not use this path — it calls `remove_entry` directly.

**Quadratic accumulation in `remove_expired`:**

`remove_expired` (pool.rs:271-288) collects all expired entries and removes them one by one:

```rust
for entry in removed {
    self.pool_map.remove_entry(&entry.proposal_short_id());
    ...
}
```

For a chain tx₀ → tx₁ → … → tx_{N-1}, regardless of removal order, the total traversal work is O(N²/2): removing from the root first causes (N-1)+(N-2)+…+1 descendant traversals; removing from the leaf first causes the same count via ancestor traversals.

**Trigger path:**

`_update_tx_pool_for_reorg` (process.rs:1039-1114) calls `tx_pool.remove_expired(callbacks)` at line 1110 while holding the tx-pool write lock. This function is invoked on every new block on every node.

**Default configuration enabling worst case:**

- `DEFAULT_MAX_ANCESTORS_COUNT = 1_000` (legacy/tx_pool.rs:16) — chains up to 1,000 deep are accepted
- `DEFAULT_EXPIRY_HOURS = 12` (legacy/tx_pool.rs:18) — 12-hour window for setup
- `DEFAULT_MAX_TX_POOL_SIZE = 180_000_000` (legacy/tx_pool.rs:20) — ~900 independent chains of 1,000 txs fit

## Impact Explanation

When the attack triggers, `remove_expired` performs approximately 900 × 499,500 ≈ 450,000,000 BFS traversal steps under the tx-pool write lock in a single block-processing event. During this window, all nodes simultaneously cannot process new transactions, generate block templates, or respond to write-path RPCs. Because all nodes process the same block at the same time, the liveness degradation is network-wide. This matches: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.** The fee cost to fill 180 MB at the minimum fee rate of 1,000 shannons/KB is approximately 1.8 CKB, which is low relative to the network-wide impact.

## Likelihood Explanation

**Low-to-Medium.** The attacker must submit enough chained transactions to fill the pool (paying fees, consuming bandwidth), then wait 12 hours for expiry. No miner cooperation, privileged access, or leaked key is required — any peer reachable via the standard `send_transaction` RPC can execute this. The attack is self-limiting (pool fills at 180 MB) but repeatable after each expiry cycle. The 12-hour setup window and fee cost are the primary barriers.

## Recommendation

1. **Use `remove_entry_and_descendants` pattern in `remove_expired`**: Pre-remove links for all expired entries before calling `remove_entry`, so `update_descendants_index_key` finds an empty descendant set — identical to the existing optimization in `remove_entry_and_descendants` (pool_map.rs:256-259).
2. **Batch `remove_expired`**: Remove at most a bounded number of expired transactions per block (e.g., 256) to cap per-block lock-hold time.
3. **Cap traversal in `update_descendants_index_key`**: Skip the update if the descendant count exceeds a threshold, relying on lazy recomputation.

## Proof of Concept

1. Configure a CKB node with default settings (`max_ancestors_count=1000`, `expiry_hours=12`, `max_tx_pool_size=180_000_000`).
2. Submit a root transaction spending a confirmed UTXO, then 999 chained children (each spending the previous output). Repeat for ~900 independent chains until the pool is full.
3. Wait 12 hours.
4. When the next block arrives, `_update_tx_pool_for_reorg` → `remove_expired` is triggered.
5. `remove_expired` iterates over ~900,000 expired entries, calling `remove_entry` for each. For each chain of 1,000 txs, the cumulative BFS work is 999+998+…+1 = 499,500 steps. Total: ~450,000,000 operations under the write lock.
6. **Observed**: `send_transaction`, `get_block_template`, and all write-path RPCs time out or queue indefinitely for the duration of the lock hold. All network nodes are affected simultaneously since they all process the same block.
