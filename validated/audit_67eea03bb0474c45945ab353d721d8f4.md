Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Distinct `from` PeerIds in `ConnectionRequest` — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary
The `pending_delivered` HashMap in `HolePunching` has no size cap and is keyed by `from` PeerId. An attacker with a single connected session can insert an unbounded number of entries by sending `ConnectionRequest` messages with distinct random `from` peer IDs targeting the victim node as `to`. All per-entry deduplication guards are bypassed by key diversity, and cleanup runs only every 5 minutes, enabling sustained memory exhaustion that can crash the node.

## Finding Description

**Root cause: no size bound on `pending_delivered`.**

`pending_delivered` is declared as a plain `HashMap<PeerId, PendingDeliveredInfo>` with no capacity limit: [1](#0-0) 

Insertion at the end of `respond_delivered` is unconditional — no size check precedes it: [2](#0-1) 

**All guards are bypassable with distinct `from` IDs.**

The `HOLE_PUNCHING_INTERVAL` deduplication check only fires when the *same* `from_peer_id` already exists in the map. A fresh random `from` always bypasses it: [3](#0-2) 

The `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`. Each new `from` creates a fresh bucket; the first request per bucket always passes: [4](#0-3) 

**The only effective throttle is the outer per-session rate limiter at 30 req/sec:** [5](#0-4) 

**Cleanup runs only every 5 minutes** (`CHECK_INTERVAL = 5 * 60` seconds), via `retain` in `notify()`: [6](#0-5) [7](#0-6) 

**`forward_rate_limiter` internal `HashMapStateStore` also grows unboundedly.** `retain_recent()` is only called on session disconnect, not periodically: [8](#0-7) 

**Exploit path:**
1. Attacker connects one session to the victim node.
2. Attacker sends `ConnectionRequest` with `from=PeerId::random()`, `to=victim_local_peer_id`, `listen_addrs=[1–24 valid TCP/IP multiaddrs]`, `max_hops≤6`, `route=[]`.
3. Each message passes: outer rate limiter (respected at 30/sec), `forward_rate_limiter` (new bucket per `from`), `HOLE_PUNCHING_INTERVAL` check (new key), `remote_listens.is_empty()` check (valid addrs provided).
4. `respond_delivered` inserts a new entry into `pending_delivered` for each message.
5. Over a 5-minute window: 9,000 entries × up to 24 `Multiaddr` objects (~60 bytes each) ≈ **~13 MB per session per cycle**, with no hard cap.

## Impact Explanation

Sustained insertion causes unbounded heap growth in `pending_delivered` and the `forward_rate_limiter`'s `HashMapStateStore`. With N concurrent sessions the growth is linear (10 sessions → ~130 MB, 50 sessions → ~650 MB). Entries inserted just after a cleanup survive until the next one (~10 minutes worst-case), doubling the figure. On memory-constrained validators this causes OOM, crashing the node.

**Impact: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attacker needs only one connected P2P session and the ability to generate arbitrary `PeerId` values (`PeerId::random()` — trivial, no key material or privilege required). The victim's peer ID is public P2P network information. The `ConnectionRequest` message is accepted from any peer. The attack is fully automatable, repeatable across cleanup cycles, and requires no victim interaction or mistakes.

## Recommendation

1. **Cap `pending_delivered` at a fixed maximum size** (e.g., 1,024 entries). Reject new insertions when the cap is reached, or evict the oldest entry (LRU). Add the check immediately before the `insert` call in `respond_delivered`.
2. **Call `forward_rate_limiter.retain_recent()` periodically** (e.g., inside `notify()`) rather than only on disconnect, to bound the `HashMapStateStore` growth.
3. **Add a global insertion rate limit** on `respond_delivered` independent of `from` peer ID, so the total insertion rate into `pending_delivered` is bounded regardless of key diversity.

## Proof of Concept

```rust
// Pseudocode: attacker loop on one connected session
// victim_peer_id is obtained from normal P2P handshake
for i in 0..9000 {
    let fake_from = PeerId::random();
    send_connection_request(
        from         = fake_from,
        to           = victim_local_peer_id,
        listen_addrs = [/* 24 valid TCP/IP multiaddrs */],
        max_hops     = 6,
        route        = [],
    );
    sleep(Duration::from_millis(34)); // ~30 req/sec, respects outer limiter
}
// After 5 minutes (before cleanup fires):
assert!(protocol.pending_delivered.len() == 9000); // no bound exists — passes
// Each entry holds up to 24 Multiaddr objects (~60 bytes each)
// Total: ~13 MB per session per 5-minute window, unbounded across cycles
```

Each iteration satisfies every guard: `ADDRS_COUNT_LIMIT` (≤24 addrs), `MAX_HOPS` (≤6), `route` check (empty), `forward_rate_limiter` (new bucket), `HOLE_PUNCHING_INTERVAL` (new key), `remote_listens.is_empty()` (valid TCP/IP addrs). The only throttle — the 30/sec outer limiter — is respected. A unit test asserting `pending_delivered.len() <= MAX_BOUND` after 9,000 such messages would fail against the current code.
