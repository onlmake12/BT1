Audit Report

## Title
Missing Per-Peer Rate Limiter in `BlockFilter` Enables Head-of-Line Blocking via `GetBlockFilterCheckPoints` Flood — (`sync/src/filter/mod.rs`, `sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
The `BlockFilter` protocol handler processes all incoming messages from all peers sequentially in a single async context with no per-peer rate limiting. A single unprivileged peer can flood `GetBlockFilterCheckPoints` messages, each triggering up to 4,000 synchronous RocksDB reads (2,000 loop iterations × `get_block_hash` + `get_block_filter_hash`), monopolizing the handler and starving all other light-client peers of `GetBlockFilterHashes` and `GetBlockFilters` responses. This is a concrete, verifiable design gap directly contrasted by the rate limiters already present in `Relayer` and `HolePunching`.

## Finding Description

**Sequential dispatch:** `CKBHandler::received` in `network/src/protocols/mod.rs` fully awaits `self.handler.received(...).await` before returning, meaning no other peer's message can begin processing until the current one completes.

**No rate limiting in `BlockFilter`:** The `BlockFilter` struct contains only `shared: Arc<SyncShared>` — no `rate_limiter` field — and `try_process` dispatches directly to `GetBlockFilterCheckPointsProcess::execute` with no quota check. This is in direct contrast to `Relayer`, which has `rate_limiter: RateLimiter<(PeerIndex, u32)>` enforced at the top of `try_process`, and `HolePunching`, which checks its rate limiter before any processing.

**Per-message cost:** `GetBlockFilterCheckPointsProcess::execute` runs a loop of up to `BATCH_SIZE = 2000` iterations. Each iteration calls `active_chain.get_block_hash(block_number)` (a synchronous RocksDB read from `COLUMN_INDEX`) and `active_chain.get_block_filter_hash(&block_hash)` (a synchronous RocksDB read from `COLUMN_BLOCK_FILTER_HASH`). Neither call has an `.await` point; the entire loop runs without yielding to the async executor. On a fully-synced mainnet node, all 2,000 iterations complete, totalling 4,000 synchronous DB reads per message.

**Exploit path:**
1. Attacker opens one TCP connection on `SupportProtocols::Filter`.
2. Attacker sends `GetBlockFilterCheckPoints` messages in a tight loop with `start_number = 0`.
3. Each message occupies the handler for the full duration of 4,000 synchronous RocksDB reads.
4. Legitimate light-client peers sending `GetBlockFilterHashes` or `GetBlockFilters` are queued behind the flood and receive no responses until the flood stops.

## Impact Explanation
A single attacker with one TCP connection and negligible bandwidth can completely deny the Filter protocol service to all light clients on the targeted node. The attack requires no privileges, keys, or hashpower. The Filter protocol is the sole mechanism by which light clients obtain compact block filter data for transaction monitoring. Complete starvation of this handler maps to **High: bad designs which could cause CKB network congestion with few costs**, since the attacker's cost is one connection and small messages while the victim node's Filter handler is fully monopolized.

## Likelihood Explanation
The attack is trivially reproducible: any peer connecting on the Filter protocol can send arbitrary `GetBlockFilterCheckPoints` messages. No authentication or special state is required. The absence of a rate limiter is a concrete, verifiable code gap. The attack is locally reproducible with two mock peers and standard CKB test infrastructure (as already used in `test/src/specs/sync/block_filter.rs`).

## Recommendation
Add a per-peer, per-message-type `RateLimiter` to `BlockFilter`, mirroring the pattern in `Relayer`:

```rust
pub struct BlockFilter {
    shared: Arc<SyncShared>,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Check the limiter at the top of `try_process` before dispatching to any `*Process::execute`. A quota of 5–10 `GetBlockFilterCheckPoints` requests per second per peer is sufficient for legitimate light-client use while preventing flooding.

## Proof of Concept
1. Start a CKB node with the Filter protocol enabled and a synced chain of ≥4,000,000 blocks.
2. Connect mock peer A and mock peer B using `SupportProtocols::Filter` (as done in `test/src/specs/sync/block_filter.rs`).
3. Have peer A send `GetBlockFilterCheckPoints { start_number: 0 }` in a tight loop.
4. Have peer B send `GetBlockFilterHashes { start_number: 0 }` at a steady rate of 1 req/sec.
5. Measure peer B's response latency with and without peer A's flood.
6. Assert that peer B's latency increases proportionally to peer A's flood rate, confirming head-of-line blocking caused by the missing rate limiter in `BlockFilter::try_process`.
