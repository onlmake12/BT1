### Title
Excess `msg.value` Permanently Locked as Pyth Fees Instead of Refunded to Caller - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the internal `requestHelper` function absorbs the entire `msg.value` (minus the provider fee) into `_state.accruedPythFeesInWei`. When a user overpays — sending more than the required fee — the excess ETH is permanently credited to Pyth's treasury instead of being refunded to the caller. This is the direct analog of the OmniXMultisender bug where excess bridging fees were locked in the contract rather than returned to the user.

---

### Finding Description

In `requestHelper`, fee accounting is performed as follows:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The contract checks that `msg.value >= requiredFee`, but then credits **all** of `msg.value - providerFee` to `accruedPythFeesInWei`. There is no branch to refund `msg.value - requiredFee` back to `msg.sender` when the user overpays. The excess is silently absorbed into Pyth's fee balance.

This behavior is explicitly documented in the interface:

> "excess value is *not* refunded to the caller" [2](#0-1) 

The test suite confirms this accounting: when a user overpays by 10,000 wei, that amount is added to `accruedPythFees` rather than returned: [3](#0-2) 

This affects all public entry points that route through `requestHelper`: `request()`, `requestWithCallback()`, and all `requestV2()` overloads. [4](#0-3) 

---

### Impact Explanation

Any ETH sent above `getFeeV2(provider, gasLimit)` is permanently transferred to Pyth's `accruedPythFeesInWei` balance. The user has no mechanism to recover it. This is a direct, irreversible financial loss for the caller. The funds are not locked in a recoverable escrow — they are credited to Pyth's withdrawable fee pool, effectively confiscating the user's overpayment.

---

### Likelihood Explanation

Overpayment is a realistic and common scenario:

1. **Fee changes between estimation and execution**: The interface explicitly warns that "the fee can change over time" and instructs callers to re-query `getFeeV2()` before each call. Any race condition between a fee increase and a user's pre-computed value causes overpayment in the opposite direction (fee decrease after estimation).
2. **Integrating contracts forwarding excess ETH**: A consumer contract that forwards `msg.value` directly to `requestV2` without computing the exact fee will overpay whenever the caller sends more than needed.
3. **Gas limit rounding**: `gasLimit` is rounded up to the nearest 10k internally (`roundTo10kGas`), meaning a user who queries `getFeeV2(provider, 19000)` and then calls with `gasLimit=19000` pays for 20,000 gas — but if they query with `gasLimit=20000` and the provider's `defaultGasLimit` is lower, the fee is the same, creating confusion about exact amounts. [5](#0-4) 

---

### Recommendation

After computing `requiredFee`, refund any excess `msg.value` to `msg.sender` before crediting `accruedPythFeesInWei`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();

// Refund excess to caller
if (msg.value > requiredFee) {
    (bool ok, ) = payable(msg.sender).call{value: msg.value - requiredFee}("");
    require(ok, "refund failed");
}

uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee); // use requiredFee, not msg.value
```

Alternatively, enforce exact payment (`msg.value == requiredFee`) to eliminate ambiguity entirely.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1000` and `defaultGasLimit = 100000`. Pyth fee is 500 wei. `getFeeV2(provider, 0)` returns 1500 wei.
2. User calls `requestV2{value: 2000}(provider, userRandom, 0)` — overpaying by 500 wei.
3. Inside `requestHelper`:
   - `requiredFee = 1500`
   - `msg.value (2000) >= requiredFee (1500)` — passes
   - `providerInfo.accruedFeesInWei += 1000`
   - `_state.accruedPythFeesInWei += (2000 - 1000) = 1000` ← 500 wei excess absorbed
4. User's 500 wei overpayment is permanently credited to Pyth's treasury. The user has no recourse.

This is confirmed by the existing test at line 699 of `Entropy.t.sol`:
```solidity
assertEq(random.getAccruedPythFees(), pythFeeInWei * 5 + 10000);
```
The `+10000` is the overpayment from the user, now owned by Pyth. [6](#0-5)
