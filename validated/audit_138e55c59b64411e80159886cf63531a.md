### Title
No Minimum Provider Fee Enforcement Allows Entropy Callback Requests to Remain Permanently Unfulfilled — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `register()` and `setProviderFee()` functions in `Entropy.sol` accept `feeInWei = 0` with no minimum value check. A provider can register or update their fee to zero, making it economically irrational for any keeper to call `revealWithCallback`. Callback requests to such a provider are stored on-chain but never fulfilled, permanently locking user-paid fees and breaking application callbacks with no user recourse.

---

### Finding Description

The `register()` function unconditionally assigns the caller-supplied `feeInWei` to `provider.feeInWei` with no lower-bound check:

```solidity
provider.feeInWei = feeInWei;   // line 129 — no minimum enforced
```

Similarly, `setProviderFee()` and `setProviderFeeAsFeeManager()` accept any value including zero:

```solidity
provider.feeInWei = newFeeInWei;   // line 819 — no minimum enforced
```

When a user calls `requestV2(provider, gasLimit)` against a zero-fee provider, `requestHelper` computes:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);   // = 0 + pythFeeInWei
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);   // = 0
providerInfo.accruedFeesInWei += providerFee;   // += 0
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);   // all goes to Pyth
```

The keeper (Fortuna) only receives `providerFee = 0`. Since calling `revealWithCallback` costs gas, the keeper has zero economic incentive to fulfill the request. The request is stored with `callbackStatus = CALLBACK_NOT_STARTED` and remains there indefinitely. After `NUM_REQUESTS` subsequent requests to the same provider, the circular buffer slot is overwritten by `allocRequest`, permanently destroying the pending request. The user's `entropyCallback` is never invoked and the paid `pythFeeInWei` is locked in the contract with no refund path.

The `reveal()` path cannot rescue callback requests: it reverts with `InvalidRevealCall` when `callbackStatus != CALLBACK_NOT_NECESSARY`, so users cannot self-serve the reveal.

The off-chain Fortuna keeper does implement `adjust_fee_if_necessary`, but this only applies to the provider that Fortuna is configured to serve. A third-party provider with `feeInWei = 0` who does not run Fortuna has no mechanism to raise the fee, and no keeper will fulfill their requests.

---

### Impact Explanation

- User's `entropyCallback` is permanently never called, breaking any application logic that depends on it (e.g., NFT mints, game outcomes, lottery draws).
- User-paid fees (`pythFeeInWei`) are locked in `_state.accruedPythFeesInWei` with no refund mechanism.
- After `NUM_REQUESTS` new requests to the same provider, the unfulfilled request slot is silently overwritten, making recovery impossible even if the provider later starts a keeper.
- Any integrator that chose the zero-fee provider based on cost is permanently affected for all in-flight requests.

---

### Likelihood Explanation

Any Ethereum address can call `register()` without access control — the provider role is fully unpermissioned. A malicious actor can register with `feeInWei = 0`, advertise a "free" provider to attract cost-sensitive integrators, and never run a keeper. Integrators who select providers based on fee cost (lower = better for their users) are the primary target. The attack requires no special privileges, no leaked keys, and no governance majority.

---

### Recommendation

Enforce a protocol-level minimum provider fee in both `register()` and `setProviderFee()`:

```solidity
uint128 public constant MIN_PROVIDER_FEE = /* governance-set minimum */;

function register(..., uint128 feeInWei, ...) public override {
    if (feeInWei < MIN_PROVIDER_FEE) revert EntropyErrors.FeeTooLow();
    ...
}

function setProviderFee(uint128 newFeeInWei) external override {
    if (newFeeInWei < MIN_PROVIDER_FEE) revert EntropyErrors.FeeTooLow();
    ...
}
```

Alternatively, add a user-facing refund path for callback requests that have been pending longer than a configurable timeout, so users can recover their funds if a provider goes dark.

---

### Proof of Concept

1. Attacker calls `register(0, commitment, metadata, chainLength, uri)` — registers as a provider with `feeInWei = 0`. No revert occurs.
2. Attacker advertises their provider address as "zero-fee Entropy provider."
3. Integrator calls `requestV2(attackerProvider, gasLimit)` paying only `pythFeeInWei` (e.g., 1 wei). Request is stored with `callbackStatus = CALLBACK_NOT_STARTED`.
4. Attacker does not run a keeper. `providerFee = 0`, so no keeper has economic incentive to call `revealWithCallback`.
5. After `NUM_REQUESTS` new requests to the attacker's provider, `allocRequest` overwrites the slot. The request is gone.
6. Integrator's `entropyCallback` is never called. The 1 wei paid is locked in `_state.accruedPythFeesInWei`. No refund is possible.

        bytes calldata commitmentMetadata,

        if (chainLength == 0) revert EntropyErrors.AssertionFailure();

        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        // NOTE: this method implementation depends on the fact that ProviderInfo will be initialized to all-zero.

        // then on future registrations, it will be unchanged. Similarly, provider.sequenceNumber defaults to 0

        provider.originalCommitmentSequenceNumber = provider.sequenceNumber;

        provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
        provider.commitmentMetadata = commitmentMetadata;
        provider.endSequenceNumber = provider.sequenceNumber + chainLength;
        provider.uri = uri;

        provider.sequenceNumber += 1;

            EntropyStructConverter.toV1ProviderInfo(provider)
        );

    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }

    function setProviderFee(uint128 newFeeInWei) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];

        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }
        uint128 oldFeeInWei = provider.feeInWei;
        provider.feeInWei = newFeeInWei;
        emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            msg.sender,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
