### Title
Unguarded `Initialize` Allows Front-Run to Seize Permanent Program Authority — (`target_chains/solana/programs/pyth-price-store/src/processor/initialize.rs`)

### Summary

The `Initialize` instruction accepts any signer as payer and any 32-byte value as authority, with no check that the caller is the program's upgrade authority. Because the config PDA can only be initialized once, an attacker who calls `Initialize` before the legitimate deployer permanently seizes sole authority over all publisher initialization.

---

### Finding Description

The `initialize` function in `processor/initialize.rs` performs three account validations before writing the config PDA:

1. **`validate_payer`** (`validate.rs:32-39`): only checks `is_signer` and `is_writable` — any funded keypair qualifies.
2. **`validate_config`** (`validate.rs:46-64`): only checks that the account matches the deterministic PDA derived from `CONFIG_SEED` and the correct bump — no ownership or caller identity check.
3. **`accounts::config::create`** (`accounts/config.rs:38-49`): only guards against re-initialization via `data.format != 0`.

There is no check anywhere that the payer is the program's BPF upgrade authority or any other privileged key. The `authority` field in `InitializeArgs` is a raw `[u8; 32]` with no validation. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

On Solana, program deployment and the first instruction call are separate transactions. An attacker monitoring the network can observe the program being deployed and immediately submit an `Initialize` transaction with their own key as `authority` before the legitimate deployer does. Once the config PDA is created with `format = FORMAT` (non-zero), every subsequent `Initialize` call returns `AlreadyInitialized` — the state is permanent and irreversible because no instruction exists to update or reset the authority.

---

### Impact Explanation

All downstream privileged operations flow through `validate_authority`, which enforces that the signer's key matches `config.authority`: [5](#0-4) 

With the attacker's key stored as `config.authority`:
- The attacker is the sole account authorized to call `InitializePublisher`.
- The attacker can register arbitrary malicious publishers, injecting corrupt price data into the oracle.
- The legitimate deployer's `InitializePublisher` calls will always fail with `MissingRequiredSignature`.
- The legitimate deployer cannot re-initialize (blocked by `AlreadyInitialized`).
- There is no authority-update or recovery instruction in the program.

This constitutes permanent, irreversible corruption of the oracle's publisher registry and a complete lockout of the legitimate operator.

---

### Likelihood Explanation

- Solana's mempool is observable; an attacker can watch for program deployment and race the `Initialize` call.
- Even without mempool monitoring, an attacker can simply poll for the program ID becoming active and call `Initialize` speculatively.
- The attack requires no privileged access, no leaked keys, and no social engineering — only a funded keypair and knowledge of the program ID.
- The cost is a single transaction (~5000 lamports in fees).

---

### Recommendation

Add a check inside `initialize` that the payer is the program's BPF upgrade authority. On Solana, this is retrievable from the program's upgrade authority stored in the `ProgramData` account (BPF Upgradeable Loader). The fix should:

1. Require the `ProgramData` account as an additional input to `Initialize`.
2. Deserialize the upgrade authority from the `ProgramData` account.
3. Assert `payer.key == upgrade_authority`.

Alternatively, if the authority is known at compile time, it can be hardcoded as a program constant and checked against `args.authority` before writing.

---

pub fn initialize(
    program_id: &Pubkey,
    accounts: &[AccountInfo],
    args: &InitializeArgs,
) -> ProgramResult {
    let mut accounts = accounts.iter();
    let payer = validate_payer(accounts.next())?;
    let config = validate_config(accounts.next(), args.config_bump, program_id, true)?;
    let system = validate_system(accounts.next())?;

    let lamports = (Rent::get()?).minimum_balance(accounts::config::SIZE);

    invoke_signed(
        &system_instruction::create_account(
            payer.key,
            config.key,
            lamports,
            accounts::config::SIZE
                .try_into()
                .expect("unexpected overflow"),
            program_id,
        ),
        &[payer.clone(), config.clone(), system.clone()],
        &[&[CONFIG_SEED.as_bytes(), &[args.config_bump]]],
    )?;

    accounts::config::create(*config.data.borrow_mut(), args.authority)?;

    Ok(())

pub fn validate_payer<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let payer = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, payer.is_signer);
    ensure!(ProgramError::InvalidArgument, payer.is_writable);
    Ok(payer)
}

pub fn validate_authority<'a, 'b>(
    account: Option<&'b AccountInfo<'a>>,
    config: &AccountInfo<'a>,
) -> Result<&'b AccountInfo<'a>, ProgramError> {
    let authority = account.ok_or(ProgramError::NotEnoughAccountKeys)?;
    ensure!(ProgramError::MissingRequiredSignature, authority.is_signer);
    ensure!(ProgramError::InvalidArgument, authority.is_writable);
    let config_data = config.data.borrow();
    let config = accounts::config::read(*config_data)?;
    ensure!(
        ProgramError::MissingRequiredSignature,
        authority.key.to_bytes() == config.authority
    );
    Ok(authority)

pub fn create(data: &mut [u8], authority: [u8; 32]) -> Result<&mut Config, ReadAccountError> {
    if data.len() < size_of::<Config>() {
        return Err(ReadAccountError::DataTooShort);
    }
    let data: &mut Config = from_bytes_mut(&mut data[..size_of::<Config>()]);
    if data.format != 0 {
        return Err(ReadAccountError::AlreadyInitialized);
    }
    data.format = FORMAT;
    data.authority = authority;
    Ok(data)
