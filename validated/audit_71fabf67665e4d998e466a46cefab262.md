### Title
Missing Reject Notification After `remove_transaction` RPC Removes Transactions from TxPool - (File: `tx-pool/src/pool.rs`)

---

### Summary

The `remove_transaction` RPC endpoint removes a transaction and all its descendants from the tx-pool via `TxPool::remove_tx`, but never invokes the registered reject callbacks (`Callbacks::call_reject`). Every other removal path in the pool (expiry, size-limit eviction, committed-tx conflict, RBF replacement) calls `call_reject`, which triggers `notify_reject_transaction` to all subscribers. The silent removal path leaves RPC subscribers, the indexer pool view, the fee estimator, and the recent-reject record all in an inconsistent state.

---

### Finding Description

The `remove_transaction` RPC is a publicly accessible endpoint callable by any local RPC user. Its implementation chain is:

`rpc/src/module/pool.rs` → `PoolRpcImpl::remove_transaction` → `tx_pool_controller.remove_local_tx` → `TxPool::remove_tx` [1](#0-0) 

`TxPool::remove_tx` removes the target entry and all descendants via `pool_map.remove_entry_and_descendants`, but returns immediately without calling any callback: [2](#0-1) 

Compare this to every other removal path, all of which call `callbacks.call_reject(self, &entry, reject)` for each removed entry:

- **Size-limit eviction** (`limit_size`): calls `callbacks.call_reject` per evicted entry. [3](#0-2) 

- **Expiry removal** (`remove_expired`): calls `callbacks.call_reject` per expired entry. [4](#0-3) 

- **Committed-tx conflict** (`remove_committed_tx`): calls `callbacks.call_reject` for conflicting entries. [5](#0-4) 

- **RBF replacement** (`process_rbf`): calls `callbacks.call_reject` for replaced entries. [6](#0-5) 

The `call_reject` callback, registered in `shared/src/shared_builder.rs`, performs three critical actions: [7](#0-6) 

1. Records the rejection in `recent_reject` (queryable via RPC).
2. Notifies the relayer via `tx_relay_sender`.
3. Calls `notify_reject_transaction` on the `NotifyController`, which dispatches the event to all registered subscribers.

None of these happen when `remove_tx` is used.

The `NotifyController` subscriber model is used by the indexer's `PoolService`, which listens on `subscribe_reject_transaction` to remove transactions from its pool view: [8](#0-7) 

When `remove_transaction` is called, the indexer never receives the removal event, so its pool view permanently retains the removed transaction as pending.

---

### Impact Explanation

Any RPC caller (local CLI user, local application, or any process with RPC access) can call `remove_transaction` to silently drop a transaction and all its descendants from the tx-pool. The consequences are:

1. **Stale indexer pool view**: The indexer's in-memory pool continues to report the removed transaction as pending, causing incorrect results for pool-related indexer queries until the node restarts.
2. **Missing RPC subscription events**: Clients subscribed to `reject_transaction` events (e.g., wallets, monitoring tools) receive no notification that the transaction was removed, causing them to believe the transaction is still pending.
3. **Fee estimator inconsistency**: The fee estimator is not informed of the removal, potentially skewing fee rate estimates.
4. **No `recent_reject` record**: The removal is not recorded in the recent-reject store, so `get_transaction` will not reflect a rejection reason.

---

### Likelihood Explanation

The `remove_transaction` RPC is a documented, intentional endpoint reachable by any local RPC caller without authentication. It is exercised in integration tests. Any operator, local script, or application using this RPC will silently trigger the inconsistency. The likelihood is **high** for any deployment that uses the indexer or RPC subscriptions alongside `remove_transaction`.

---

### Recommendation

In `TxPool::remove_tx`, after collecting the removed entries via `pool_map.remove_entry_and_descendants`, iterate over them and call `callbacks.call_reject(self, &entry, reject)` with an appropriate `Reject` variant (e.g., a new `Reject::Removed` variant or reuse `Reject::Expiry`/`Reject::Full` with a descriptive message). This mirrors the pattern used in `limit_size`, `remove_expired`, and `remove_committed_tx`.

```rust
pub(crate) fn remove_tx(&mut self, id: &ProposalShortId, callbacks: &Callbacks) -> bool {
    let entries = self.pool_map.remove_entry_and_descendants(id);
    for entry in &entries {
        let reject = Reject::Removed("removed via RPC".to_string());
        callbacks.call_reject(self, entry, reject);
    }
    !entries.is_empty()
}
```

The `callbacks` parameter must be threaded through from the service layer, consistent with how other removal functions receive it.

---

### Proof of Concept

1. Start a CKB node with the indexer and an RPC subscription client subscribed to `reject_transaction`.
2. Submit a valid transaction via `send_transaction` — the subscriber receives a `new_transaction` event; the indexer records it as pending.
3. Call `remove_transaction` with the same tx hash via RPC.
4. **Observe**: The subscriber receives **no** `reject_transaction` event. The indexer still reports the transaction as pending. `get_transaction` shows no rejection reason. The fee estimator is not updated.
5. Compare: call `send_transaction` with a conflicting transaction to force a `Reject::Resolve` path — the subscriber **does** receive a `reject_transaction` event immediately.

The root cause is exclusively in `tx-pool/src/pool.rs` at `TxPool::remove_tx` (line 358–361), which is the only removal path that bypasses `callbacks.call_reject`. [2](#0-1)
