### Title
`Echo.executeCallback` Permanently Destroys Requests on Out-of-Gas Callback Without Sufficient Gas Check - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.executeCallback` clears the request and credits the provider **before** invoking the consumer callback. It catches all callback failures â€” including out-of-gas â€” with a bare `catch {}` and emits `PriceUpdateCallbackFailed`, treating the request as permanently fulfilled. Unlike `Entropy.revealWithCallback`, which reverts with `InsufficientGas` when the calling context did not forward enough gas to the callback, `Echo.executeCallback` has no such guard. An unprivileged caller can supply just enough gas to pass all pre-callback checks but not enough to properly forward `callbackGasLimit` to the consumer, permanently destroying the request with no retry path.

---

### Finding Description

`executeCallback` is a public, permissionless function (callable by anyone after the exclusivity period).

**Execution order in `executeCallback`:**

1. **Line 161â€“162**: Provider fees are credited to `providerToCredit.accruedFeesInWei`.
2. **Line 164**: `clearRequest(sequenceNumber)` permanently removes the request from storage.
3. **Lines 176â€“201**: The consumer callback is attempted inside a `try/catch`. [1](#0-0) 

The callback is forwarded exactly `req.callbackGasLimit` gas:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
```

Due to EVM's 63/64 rule, the actual gas forwarded to the callback is at most `(gasleft_before_call * 63) / 64`. If an attacker calls `executeCallback` with gas `G` such that after pre-callback operations the remaining gas `R` satisfies `(R * 63) / 64 < callbackGasLimit`, the callback receives less gas than required and reverts with OOG. The bare `catch {}` block catches this silently:

```solidity
} catch {
    // Out of gas or other low-level errors
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
``` [2](#0-1) 

The transaction succeeds. The request is already cleared and cannot be retried. The provider is already paid.

**Contrast with `Entropy.revealWithCallback`**, which explicitly measures `startingGas` before the call and reverts if the calling context did not provide sufficient gas:

```solidity
} else {
    // Callback reverted by (potentially) running out of gas, but the calling context
    // did not have enough gas to run the callback.
    revert EntropyErrors.InsufficientGas();
}
``` [3](#0-2) 

The guard condition used in Entropy is `(startingGas * 31) / 32 > gasLimit` â€” if this is false, the transaction reverts, preserving the request for a retry with sufficient gas. [4](#0-3) 

Echo has no equivalent check.

**Secondary amplifier â€” stale storage pointer for overflow requests:**

`clearRequest` for overflow-mapped requests calls `delete _state.requestsOverflow[key]`, zeroing the entire struct. The `req` storage pointer obtained before `clearRequest` now reads `address(0)` for `req.requester` and `0` for `req.callbackGasLimit`. The callback is then attempted as `IEchoConsumer(address(0))._echoCallback{gas: 0}(...)`, which silently succeeds (call to address(0) does not revert), causing `emitPriceUpdate` to fire as if the callback succeeded â€” while the actual consumer contract received nothing. [5](#0-4) 

---

### Impact Explanation

- **Consumer's callback is permanently lost.** The request is cleared before the callback; there is no retry mechanism analogous to Entropy's `CALLBACK_FAILED` state.
- **Provider is paid for an undelivered callback.** `accruedFeesInWei` is incremented before the callback attempt.
- **For overflow requests**, the callback fires against `address(0)` with 0 gas, and the contract incorrectly emits `PriceUpdateExecuted` as if the consumer was notified.
- Any on-chain logic in the consumer that depends on receiving the price update (e.g., settling a trade, updating a TWAP, releasing collateral) is permanently blocked.

---

### Likelihood Explanation

- `executeCallback` is **public with no access control** after the exclusivity period expires.
- The attacker needs only to submit a transaction with a crafted gas limit â€” no special privileges, no leaked keys, no governance majority.
- The exclusivity period (`exclusivityPeriodSeconds`) provides a window of protection, but once it expires any address can call `executeCallback`.
- Gas manipulation attacks of this type are well-understood and straightforward to execute on EVM chains.

---

### Recommendation

Add a pre-callback gas sufficiency check mirroring `Entropy.revealWithCallback`:

```solidity
uint256 startingGas = gasleft();
// ... callback attempt ...
// In the catch {} block:
if ((startingGas * 63) / 64 < uint256(req.callbackGasLimit)) {
    revert InsufficientGas();
}
```

This ensures that if the calling context did not forward enough gas to satisfy `callbackGasLimit`, the entire transaction reverts, preserving the request for a retry with sufficient gas.

Additionally, cache `req.requester` and `req.callbackGasLimit` into memory variables **before** calling `clearRequest`, to avoid reading zeroed storage for overflow requests.

---

### Proof of Concept

1. Alice deploys an `IEchoConsumer` contract with `callbackGasLimit = 1_000_000` and calls `requestPriceUpdatesWithCallback`, paying the required fee.
2. The exclusivity period expires.
3. Attacker calls:
   ```solidity
   echo.executeCallback{gas: 400_000}(
       providerToCredit, sequenceNumber, updateData, priceIds
   );
   ```
4. Pre-callback operations (storage reads, Pyth fee call, `clearRequest`, `firstUnfulfilledSeq` loop) consume ~250k gas. Remaining gas â‰ˆ 150k. Due to 63/64 rule, callback receives at most ~148k gas â€” far below the 1M limit.
5. `_echoCallback` reverts with OOG. The bare `catch {}` fires, emitting `PriceUpdateCallbackFailed`.
6. Transaction succeeds. `getRequest(sequenceNumber)` returns an empty struct â€” the request is gone. Alice's consumer never received the price update and has no recourse. [6](#0-5)

        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.

        while (

            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(

                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
    }
