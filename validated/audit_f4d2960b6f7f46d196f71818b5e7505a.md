### Title
Unbounded `pending_delivered` HashMap Growth via Unique-`from` ConnectionRequest Flooding — (`network/src/protocols/hole_punching/mod.rs`, `component/connection_request.rs`)

---

### Summary

An unprivileged remote peer with a single P2P connection can cause the `HolePunching::pending_delivered` HashMap to grow without bound between 5-minute cleanup cycles by sending `ConnectionRequest` messages that each carry a distinct random `from` PeerId targeting the victim's `local_peer_id`. No per-`from` cap exists; the only binding constraint is the per-session rate limiter (30 msg/sec), yielding up to ~9,000 new entries per session per cleanup window.

---

### Finding Description

**Entry point** — `HolePunching::received()` dispatches every inbound `ConnectionRequest` to `ConnectionRequestProcess::execute()`.

**Rate-limiter keying** — The outer rate limiter is keyed by `(session_id, msg.item_id())`: [1](#0-0) 

This limits the *total message rate* from one session to 30/sec, but places no cap on the number of distinct `from` PeerIds that can be processed.

The inner `forward_rate_limiter` is keyed by `(from, to, msg_item_id)`: [2](#0-1) 

With N unique `from` values, each gets its own independent bucket (1/sec quota), so all N can pass within the same second.

**Insertion path** — When `self_peer_id == content.to`, `respond_delivered()` is called. The only deduplication guard checks whether the *same* `from` was seen within the last 2 minutes: [3](#0-2) 

With unique `from` PeerIds, this guard is never triggered. On success, the entry is unconditionally inserted: [4](#0-3) 

**Cleanup** — `pending_delivered` is only pruned in `notify()`, which fires every 5 minutes (`CHECK_INTERVAL`): [5](#0-4) [6](#0-5) 

There is no cleanup on disconnect, no size cap, and no eviction policy.

**Secondary unbounded growth** — `forward_rate_limiter` (a `HashMapStateStore`) also accumulates one bucket per unique `(from, to)` pair. `retain_recent()` is only called on `disconnected()`: [7](#0-6) 

While the attacker remains connected, this map also grows without bound.

---

### Impact Explanation

- **Per session, per 5-minute window**: 30 msg/sec × 300 sec = **9,000 entries** maximum.
- **Per entry size**: `PeerId` (~39 B) + `Vec<Multiaddr>` (up to 24 entries × ~50–100 B each) + `u64` ≈ **~2,500 B**.
- **Single-session heap growth**: ~9,000 × 2,500 B ≈ **~22 MB per window**.
- With K simultaneous attacker sessions the growth scales linearly to **K × 22 MB per window**.
- Sustained over multiple windows (entries are retained for the full 5-minute TIMEOUT before pruning), heap pressure accumulates until OOM or severe degradation, crashing or stalling the node.

---

### Likelihood Explanation

- The victim's `local_peer_id` is publicly advertised on the P2P network (used in every identify/discovery handshake).
- No authentication, PoW, or stake is required to open a P2P connection and send `ConnectionRequest` messages.
- Generating random `PeerId` bytes is trivial; each message only needs a syntactically valid `from` field.
- The attack is fully local-testable with a single TCP connection.

---

### Recommendation

1. **Cap `pending_delivered` size**: Reject insertion (return `Ignore`) when `pending_delivered.len()` exceeds a hard limit (e.g., 256 or 1,024 entries).
2. **Per-session insertion limit**: Track how many `pending_delivered` entries originated from each session and evict or reject once a per-session cap is reached.
3. **Periodic `forward_rate_limiter` cleanup**: Call `retain_recent()` inside `notify()`, not only on `disconnected()`, to prevent the rate-limiter map from growing while an attacker stays connected.
4. **Tighten the outer rate limiter**: Lower the 30/sec quota or add a per-`from` PeerId rate limit.

---

### Proof of Concept

```
1. Connect to victim node V (whose local_peer_id = TARGET_ID is known from discovery).
2. In a loop, for i in 0..N:
     from_i = PeerId::random()
     msg = ConnectionRequest {
         from: from_i,
         to: TARGET_ID,
         listen_addrs: [valid_tcp_addr_1, ..., valid_tcp_addr_24],  // 24 entries
         route: [],
         max_hops: 6,
     }
     send msg over the HolePunching protocol stream
3. After each batch of 30 (1 second), assert pending_delivered.len() == i+1
   (observable via metrics or by measuring heap growth with /proc/<pid>/status VmRSS).
4. After 300 seconds (one cleanup window), pending_delivered holds ~9,000 entries
   consuming ~22 MB; repeat to accumulate across windows.
```
