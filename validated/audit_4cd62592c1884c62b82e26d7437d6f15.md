Audit Report

## Title
Missing Bounds Validation in `ExecV2` Syscall Silently Clamps Data Instead of Returning `SLICE_OUT_OF_BOUND` — (File: `script/src/syscalls/exec_v2.rs`)

## Summary
The `ExecV2` syscall handler omits the bounds validation that both `Spawn` and the legacy `Exec` syscall perform before processing `offset`/`length` parameters. When a script supplies out-of-bounds values, `ExecV2` enqueues the message without error, the scheduler's `load_data` call silently clamps the slice, and the VM either terminates with an internal ELF parse error or executes a truncated program — neither of which surfaces `SLICE_OUT_OF_BOUND` (code 3) to the calling script as the documented contract requires.

## Finding Description
In `exec_v2.rs`, the `ecall` handler extracts `offset` and `length` from the `bounds` register and immediately enqueues `Message::ExecV2` with no validation: [1](#0-0) 

The file does not import `SLICE_OUT_OF_BOUND` at all, confirming no such check exists.

By contrast, `spawn.rs` acquires the `snapshot2_context` lock, calls `sc.load_data(&data_piece_id, 0, 0)` to obtain `full_length`, and explicitly validates both conditions before enqueuing: [2](#0-1) 

And `exec.rs` performs the same validation inline after fetching the data: [3](#0-2) 

When the scheduler later processes `Message::ExecV2`, it calls `sc.load_data(...)` with the unvalidated parameters and propagates any error with `?`: [4](#0-3) 

The underlying `SgData::load_data` does not return an error for out-of-bounds inputs — it silently clamps: [5](#0-4) 

Two concrete failure modes result:

1. **`offset >= data_size`**: `offset` is clamped to `data.len()`, producing an empty `Bytes`. The subsequent ELF parse fails with an internal `VMError`, propagated as a scheduler `Error` — the calling script never receives `SLICE_OUT_OF_BOUND`.

2. **`offset + length > data_size`**: `length` is clamped to `full_length - offset`, producing a truncated prefix. If that prefix is a structurally valid ELF, the scheduler loads and executes it without any error signal to the calling script.

## Impact Explanation
This is **Incorrect implementation or behavior of CKB-VM or system scripts** (High, 10001–15000 points). The `ExecV2` syscall (number 2043) is a first-class CKB-VM system call in the Meepo multi-VM scheduler path. Its documented contract — consistent with `Spawn` and `Exec` — is to return `SLICE_OUT_OF_BOUND` when bounds parameters are invalid. Violating this contract means any on-chain script that issues `ExecV2` with out-of-bounds `bounds` and checks the return value for code 3 to implement fallback logic will instead observe an unrecoverable internal VM termination, breaking the script's control flow. In the truncation case, a different code path than intended may be silently executed.

## Likelihood Explanation
Any unprivileged script author who deploys a lock or type script using syscall 2043 can trigger this by supplying an arbitrary `bounds` register value. No special privilege, key material, or network position is required. The attacker controls the `bounds` value entirely through their own script code. The `ExecV2` syscall is available to all scripts running under the multi-VM scheduler.

## Recommendation
Before pushing `Message::ExecV2`, add the same pre-enqueue validation used in `spawn.rs` lines 107–132: acquire the `snapshot2_context` lock, call `sc.load_data(&data_piece_id, 0, 0)` to obtain `full_length`, then check:

```rust
if offset >= full_length {
    machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
    return Ok(true);
}
if length > 0 {
    let end = offset.checked_add(length).ok_or(VMError::MemOutOfBound)?;
    if end > full_length {
        machine.set_register(A0, Mac::REG::from_u8(SLICE_OUT_OF_BOUND));
        return Ok(true);
    }
}
```

This requires adding `snapshot2_context` to the `ExecV2` struct (mirroring `Spawn`) and importing `SLICE_OUT_OF_BOUND`.

## Proof of Concept
A script running under the multi-VM scheduler issues:

```c
// Case 1: offset far exceeds any real cell data size
uint64_t bounds = ((uint64_t)0xFFFFFFFF << 32) | 1;
uint64_t ret = syscall(2043, index, source, place, bounds, argc, argv);
// Expected: ret == 3 (SLICE_OUT_OF_BOUND), script handles fallback
// Actual:   scheduler terminates with internal ELF parse error; ret never set
```

```c
// Case 2: offset=0, length exceeds data size by 1
uint64_t data_size = /* actual cell data size */;
uint64_t bounds = (0ULL << 32) | (data_size + 1);
uint64_t ret = syscall(2043, index, source, place, bounds, argc, argv);
// Expected: ret == 3 (SLICE_OUT_OF_BOUND)
// Actual:   scheduler loads clamped data[0..data_size], executes it without error
```

A unit test can be written by constructing a `Scheduler` with a cell whose data is N bytes, invoking `ExecV2` with `offset = N + 1`, and asserting the script receives `SLICE_OUT_OF_BOUND` — the test will fail, confirming the bug.
