### Title
RPC-Driven Connection Pool Exhaustion Stalls Rich-Indexer Sync — (`util/rich-indexer/src/store.rs`)

### Summary

The rich-indexer's `SQLXPool` is configured with `max_connections(10)` and `acquire_timeout(60s)`. Both the RPC query path (`get_transactions`) and the indexer write path (`append`/`rollback`) draw connections from the **same shared pool** with no priority separation. An unprivileged caller who issues 10 concurrent `get_transactions` calls can exhaust all pool connections, causing every subsequent `store.transaction()` call in the sync loop to block for up to 60 seconds per cycle.

### Finding Description

**Pool configuration** — `util/rich-indexer/src/store.rs` lines 46–51: [1](#0-0) 

The SQLite (non-memory) backend gets `max_connections(10)` and `acquire_timeout(60s)`. There is no separate pool for write operations.

**RPC read path acquires a pool connection** — `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs` lines 35–39: [2](#0-1) 

`get_transactions` calls `self.store.transaction().await` (i.e., `pool.begin()`) and holds the connection for the entire duration of the SQL query — until the function returns and `tx` is dropped.

**Indexer write path draws from the same pool** — `util/rich-indexer/src/indexer/mod.rs` lines 156–161 and 180–185: [3](#0-2) [4](#0-3) 

Both `append` and `rollback` call `self.store.transaction().await` against the identical `SQLXPool` instance. There is no dedicated write connection, no pool priority, and no reservation.

**Sync loop is a tight blocking loop** — `util/indexer-sync/src/lib.rs` lines 160 and 170: [5](#0-4) [6](#0-5) 

`try_loop_sync` calls `indexer.append()` / `indexer.rollback()` in a continuous loop. `RichIndexer::append` uses `async_runtime.block_on(future)`, so the sync thread is blocked until the pool connection is acquired.

**Attack chain**:
1. Attacker opens 10 concurrent `get_transactions` RPC calls (e.g., with a broad script filter and large `limit`).
2. Each call acquires one connection via `pool.begin()` and holds it while the SQL query executes.
3. All 10 pool slots are occupied.
4. The sync loop calls `append` → `store.transaction()` → `pool.begin()` → waits up to 60 seconds for a slot.
5. Attacker keeps the 10 calls alive (e.g., by requesting large result sets or using slow network reads) and re-issues them as they complete, keeping the pool perpetually saturated.
6. The indexer tip lags indefinitely behind the chain tip.

### Impact Explanation

The indexer sync stalls for up to 60 seconds per blocked `append`/`rollback` call. An attacker who continuously re-issues 10 concurrent `get_transactions` calls can keep the pool exhausted indefinitely, causing the rich-indexer tip to fall arbitrarily far behind the chain tip. Applications relying on the rich-indexer (wallets, explorers, dApps) will observe stale or missing transaction/cell data.

### Likelihood Explanation

The RPC endpoint is publicly accessible with no authentication. The `request_limit` guard only caps the `limit` parameter per call — it does not limit concurrent connections or concurrent in-flight RPC calls. No rate-limiting or connection reservation exists for the write path. The attack requires only 10 simultaneous HTTP requests, which is trivially achievable.

### Recommendation

- Allocate a **separate `AnyPool`** (or reserve at least 1–2 connections) exclusively for write operations (`append`/`rollback`), so RPC reads can never starve the sync path.
- Alternatively, enforce a **per-client or global concurrency limit** on indexer RPC handlers (e.g., a semaphore capping concurrent in-flight queries below `max_connections - 1`).
- Consider reducing `acquire_timeout` for the write path so sync failures are detected and retried quickly rather than blocking for 60 seconds.

### Proof of Concept

```python
import asyncio, aiohttp, json

RPC = "http://127.0.0.1:8116"

async def long_get_transactions(session):
    payload = {
        "jsonrpc": "2.0", "id": 1, "method": "get_transactions",
        "params": [
            {"script": {"code_hash": "0x" + "00"*32, "hash_type": "data", "args": "0x"},
             "script_type": "lock"},
            "asc", "0x7fffffff", None   # max limit to maximize query duration
        ]
    }
    async with session.post(RPC, json=payload) as r:
        return await r.json()

async def main():
    async with aiohttp.ClientSession() as s:
        # Saturate all 10 pool connections
        tasks = [long_get_transactions(s) for _ in range(10)]
        # While these are in flight, trigger a block append via a side channel
        # and measure how long store.transaction() blocks in the sync loop.
        results = await asyncio.gather(*tasks)

asyncio.run(main())
```

Simultaneously monitor the indexer tip via `get_indexer_tip` before and after the 10 concurrent calls. The tip will not advance during the saturation window, confirming the sync stall.
