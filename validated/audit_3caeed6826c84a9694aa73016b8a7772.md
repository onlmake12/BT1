### Title
Capacity Conservation Bypass for Mixed-Input DAO Withdraw Transactions ‚Äî (`verification/src/transaction_verifier.rs`)

---

### Summary

`CapacityVerifier` unconditionally skips the `OutputsSumOverflow` check for **any** transaction that contains at least one DAO-type input, delegating enforcement entirely to the DAO type script. The DAO type script only enforces DAO-cell-specific output capacity (maximum withdraw amount); it does not enforce total transaction capacity conservation. A transaction sender who mixes one DAO input with ordinary non-DAO inputs can therefore produce outputs whose total capacity exceeds total inputs plus DAO interest, inflating the CKB supply.

---

### Finding Description

In `verification/src/transaction_verifier.rs`, `CapacityVerifier::verify()` guards the `OutputsSumOverflow` check with:

```rust

    let inputs_sum  = self.resolved_transaction.inputs_capacity()?;
    let outputs_sum = self.resolved_transaction.outputs_capacity()?;
    if inputs_sum < outputs_sum {
        return Err(TransactionError::OutputsSumOverflow { ‚Ä¶ }.into());
    }
}

The predicate `valid_dao_withdraw_transaction()` is:

        .resolved_inputs
        .iter()

``` [2](#0-1) 

`any(‚Ä¶)` returns `true` the moment **one** input carries the DAO type script. The entire `OutputsSumOverflow` check is then skipped for the whole transaction, including all non-DAO inputs and outputs.

The inline comment acknowledges the delegation:

```
// DAO withdraw transaction is verified via the type script of DAO cells
``` [3](#0-2) 

The DAO type script (RFC-0023) enforces only that the output cell for the withdrawing DAO cell carries at most the maximum-withdraw capacity (`calculate_maximum_withdraw`). It does not enforce that `sum(outputs) ‚â§ sum(inputs) + DAO_interest` across the whole transaction. The non-DAO inputs' capacity is therefore unguarded.

`DaoCalculator::calculate_maximum_withdraw` computes the per-cell ceiling:

```rust
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
Ok(withdraw_capacity)
``` [4](#0-3) 

No cross-cell or cross-transaction capacity balance is checked there.

---

### Impact Explanation

An unprivileged transaction sender who holds any DAO deposit can craft a withdraw transaction that also consumes ordinary cells and sets non-DAO outputs to an arbitrarily large capacity. Because `CapacityVerifier` skips the sum check and the DAO type script only validates the DAO output cell, the surplus capacity passes all node-level checks and is committed on-chain, permanently inflating the CKB supply. The magnitude of inflation per transaction is bounded only by the attacker's willingness to pay fees (which are themselves drawn from the inflated outputs).

---

### Likelihood Explanation

Any holder of a NervosDAO deposit ‚Äî a common, permissionless action ‚Äî can trigger this. No privileged role, leaked key, or majority hash power is required. The attacker constructs a standard DAO phase-2 withdraw transaction, adds one or more ordinary inputs, and sets the total output capacity above the legitimate sum. The transaction is valid from the node's perspective and will be relayed and mined normally.

---

### Recommendation

Replace the blanket skip with a bounded check: for DAO withdraw transactions, verify that `sum(outputs) ‚â§ sum(inputs) + withdrawed_DAO_interest`, where `withdrawed_DAO_interest` is computed via `DaoCalculator::withdrawed_interests`. The existing `DaoCalculator` already exposes this value:

```rust
fn withdrawed_interests(‚Ä¶) -> Result<Capacity, DaoError> { ‚Ä¶ }
``` [5](#0-4) 

`CapacityVerifier` should call this and enforce `outputs_sum ‚â§ inputs_sum + withdrawed_interests` instead of skipping the check entirely.

---

### Proof of Concept

1. Attacker holds a DAO deposit cell: 100 CKB principal, maximum withdraw = 110 CKB.
2. Attacker also holds a regular cell: 50 CKB.
3. Attacker constructs a DAO phase-2 withdraw transaction:
   - **Inputs**: DAO withdraw cell (100 CKB) + regular cell (50 CKB) ‚Üí legitimate total = 150 CKB + 10 CKB interest = 160 CKB.
   - **Outputs**: DAO output cell (110 CKB, satisfies DAO type script) + extra output cell (200 CKB).
   - Total outputs = 310 CKB >> 160 CKB.
4. `valid_dao_withdraw_transaction()` returns `true` (one DAO input present); `OutputsSumOverflow` check is skipped entirely.
5. DAO type script runs for the DAO input cell, verifies the 110 CKB output is ‚â§ maximum withdraw ‚Äî passes.
6. No other node-level check catches the 150 CKB surplus. The transaction is accepted, minting 150 CKB from nothing.
