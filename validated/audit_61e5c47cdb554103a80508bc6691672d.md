## Title
Minimum Fee Rate Admission Check Uses Serialized Size Instead of True Transaction Weight — (`tx-pool/src/util.rs`)

## Summary
`check_tx_fee` in `tx-pool/src/util.rs` gates pool admission using only the transaction's serialized byte size to compute the minimum required fee. After script execution, the pool uses `get_transaction_weight(tx_size, cycles) = max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)` for all ordering, eviction, and fee-rate accounting — but no second fee-rate check against this true weight is performed. An attacker can craft a compact transaction executing a high-cycle script, pass the size-only admission gate with a minimal fee, and occupy pool weight far exceeding what the fee would normally permit.

## Finding Description
In `tx-pool/src/util.rs` L42–52, `check_tx_fee` computes the minimum fee using only `tx_size`, with a code comment explicitly acknowledging the gap:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

In `tx-pool/src/process.rs`, `_process_tx` calls `pre_check` (which invokes `check_tx_fee`) at L715, then calls `verify_rtx` at L724–732 to obtain actual cycles. At L751, it constructs `TxEntry::new(rtx, verified.cycles, fee, tx_size)` with no post-verification fee-rate check against the true weight. The gap is confirmed: there is no second check anywhere in `_process_tx`, `submit_entry`, or `readd_detached_tx`.

The true weight used everywhere else is defined in `util/types/src/core/tx_pool.rs` L298–303:
```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)
}
```
where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. This weight drives pool ordering (`AncestorsScoreSortKey` via `entry.rs` L221–231), eviction priority (`EvictKey` via `entry.rs` L234–247), and fee rate display (`fee_rate()` at `entry.rs` L114–118).

The `limit_size` eviction in `pool.rs` L292–328 triggers only when `self.pool_map.total_tx_size > self.config.max_tx_pool_size`, where `total_tx_size` tracks serialized byte size — not weight. Compact high-cycle transactions contribute minimally to `total_tx_size`, so they do not trigger size-based eviction despite consuming disproportionate block weight capacity.

Exploit path:
1. Attacker deploys a compact RISC-V loop script (~200–300 bytes serialized).
2. Constructs a transaction with `tx_size ≈ 300`, `cycles ≈ 69,000,000`.
3. Sets `fee = min_fee_rate × tx_size / 1000 = 1000 × 300 / 1000 = 300 shannons`.
4. `check_tx_fee` passes: `fee (300) >= min_fee (300)`.
5. After VM execution, `weight = max(300, 69_000_000 × 0.000_170_571_4) ≈ 11,769`.
6. Effective fee rate ≈ `300 × 1000 / 11,769 ≈ 25 shannons/KB` — ~40× below the 1,000 shannons/KB minimum.
7. Transaction is admitted with no further rejection.

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."** With `max_tx_pool_size = 180 MB` and a ~40× fee discount, an attacker can occupy pool capacity that would normally require 40× more fee expenditure. Legitimate transactions paying the true minimum fee rate may be evicted or delayed, degrading mempool quality and miner revenue at a fraction of the intended cost. The pool's byte-size-based eviction boundary does not protect against this because the attacker's transactions are small in serialized size.

## Likelihood Explanation
The attack requires only: (1) a one-time deployment of a compact loop script on-chain (trivial cost); (2) repeated `send_transaction` RPC or P2P relay calls with no privileged access. The code comment at the root cause site (`// Theoretically we cannot use size as weight directly`) confirms this gap is known and unmitigated. The attack is fully repeatable and automatable by any unprivileged user.

## Recommendation
After `verify_rtx` returns `verified.cycles` in `_process_tx` (at `tx-pool/src/process.rs` L734), perform a second fee-rate check using the true weight before constructing the `TxEntry`:

```rust
let verified = try_or_return_with_snapshot!(verified_ret, snapshot);
// Post-verification: check fee rate against true weight
let weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(tx_pool_config.min_fee_rate, min_fee_by_weight.as_u64(), fee.as_u64())), snapshot));
}
let entry = TxEntry::new(rtx, verified.cycles, fee, tx_size);
```

The same post-verification check should be applied in `readd_detached_tx`. Alternatively, the `limit_size` eviction boundary could be changed to track total pool weight rather than total serialized byte size.

## Proof of Concept
1. Write a minimal RISC-V loop script executing ~69,000,000 cycles; compiled binary is ~150–300 bytes.
2. Deploy the script cell on-chain (one-time cost).
3. Construct a transaction with `tx_size ≈ 300` bytes using this script as a lock.
4. Set `fee = 300 shannons` (exactly `min_fee_rate × tx_size / 1000`).
5. Submit via `send_transaction` RPC.
6. Observe: `check_tx_fee` passes (`300 >= 300`); after VM execution `verified.cycles ≈ 69,000,000`; `weight = max(300, 11,769) = 11,769`; effective fee rate ≈ 25 shannons/KB (40× below minimum). Transaction is admitted.
7. Repeat with many UTXOs to fill the pool at ~1/40th the intended cost, displacing legitimate transactions without triggering the byte-size eviction boundary.
