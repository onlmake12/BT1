### Title
Attacker-Controlled Address Injection via `HOLE_PUNCHING_INTERVAL` Bypass in `respond_delivered()` - (`network/src/protocols/hole_punching/component/connection_request.rs`)

---

### Summary

The `respond_delivered()` function in the hole-punching protocol stores attacker-supplied `listen_addrs` in `pending_delivered` keyed only by `from` PeerId. The deduplication guard checks only the elapsed timestamp against `HOLE_PUNCHING_INTERVAL` (2 minutes) and does not validate that the `listen_addrs` are unchanged. After the interval elapses, an attacker can overwrite the stored entry with new attacker-controlled addresses. When a subsequent `ConnectionSync` arrives, the node blindly reads those addresses from `pending_delivered` and initiates outbound TCP connections to them.

---

### Finding Description

**Entry point — `ConnectionRequestProcess::execute()`**

The `execute()` method accepts a `ConnectionRequest` from any connected peer. When `self_peer_id == &content.to`, it calls `respond_delivered(content.from, ...)` passing the attacker-supplied `listen_addrs` directly. [1](#0-0) 

Critically, `content.from` is taken verbatim from the message with no check that it matches the actual sender's session peer ID. An attacker can place any `PeerId` in the `from` field. [2](#0-1) 

**Deduplication guard — timestamp-only, no address consistency check**

`respond_delivered()` checks whether an existing `pending_delivered` entry for `from_peer_id` is younger than `HOLE_PUNCHING_INTERVAL`. If the entry is older (or absent), execution falls through unconditionally and the entry is overwritten with the new attacker-supplied addresses. [3](#0-2) 

The overwrite at the end of the function stores the new attacker-controlled `remote_listens` under `from_peer_id`: [4](#0-3) 

**Consumption — `ConnectionSyncProcess::execute()` uses stored addresses without re-validation**

When a `ConnectionSync` message arrives with `from = X` and `to = victim`, the handler reads `pending_delivered.get(&content.from)` and passes the stored addresses directly to `try_nat_traversal`, spawning outbound TCP connection tasks to each address: [5](#0-4) 

There is no re-validation of the addresses at this point. The `ConnectionSync` message itself also does not carry `listen_addrs`; the addresses come entirely from the previously poisoned `pending_delivered` map.

**Rate limiter does not prevent the attack**

The `forward_rate_limiter` allows 1 request per second per `(from, to, msg_item_id)` tuple. [6](#0-5) 

After 2 minutes the rate limiter has long since reset, so the second `ConnectionRequest` passes without issue.

**`HOLE_PUNCHING_INTERVAL` constant** [7](#0-6) 

---

### Impact Explanation

The victim node makes outbound TCP connections to arbitrary attacker-controlled IP:port endpoints. The connection is registered as an inbound raw session via `RawSessionInfo::inbound(listen_addr)`, meaning the victim's P2P layer treats the attacker's server as a legitimate inbound peer. This enables:

- **Connection hijacking**: the victim's hole-punching attempt is redirected to an attacker-controlled server instead of the intended peer.
- **SSRF-equivalent**: the victim can be made to probe arbitrary TCP endpoints on behalf of the attacker.
- **Peer impersonation setup**: combined with the missing `from`-sender validation, the attacker can poison the entry for *any* `PeerId` (not just their own), causing the victim to connect to the attacker's server whenever a `ConnectionSync` for that peer ID arrives.

---

### Likelihood Explanation

The attacker only needs a standard P2P connection to the victim — no special privileges, no PoW, no key material. The two-step attack (first `ConnectionRequest` → wait 2 minutes → second `ConnectionRequest` with different addresses → `ConnectionSync`) is trivially scriptable. The `from` field is not authenticated against the session, so the attacker does not even need to wait if they target a `PeerId` that has no existing `pending_delivered` entry.

---

### Recommendation

1. **Validate `from` against the actual sender**: in `execute()`, verify that `content.from` matches the peer ID of the session that sent the message. Reject messages where they differ.
2. **Address consistency check in `respond_delivered()`**: when an existing `pending_delivered` entry exists and the interval has elapsed, compare the new `listen_addrs` against the stored ones. Reject or log if they differ.
3. **Bind `pending_delivered` entries to the session**: store the session ID alongside the addresses and require the `ConnectionSync` to arrive on the same session (or from a peer whose session matches the stored `from` peer ID).
