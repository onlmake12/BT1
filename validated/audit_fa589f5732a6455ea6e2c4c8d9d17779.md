### Title
Silent u128-to-u64 Truncation in NervosDAO Withdrawal Interest Calculation Causes Incorrect DAO Field and Depositor Loss - (File: util/dao/src/lib.rs)

### Summary

`DaoCalculator::calculate_maximum_withdraw` silently truncates a u128 intermediate result to u64 using an unchecked `as u64` cast. This is inconsistent with every other u128-to-u64 conversion in the same codebase, which all use `u64::try_from(...).map_err(|_| DaoError::Overflow)?`. For sufficiently large deposits held over long periods, the truncation silently produces a wrong (smaller) withdrawal maximum, corrupting both the `withdrawed_interests` value fed into the consensus-critical DAO field and the amount returned to the depositor.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the withdrawal amount as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast on line 156 is a **silent truncating cast** in Rust. If `withdraw_counted_capacity` exceeds `u64::MAX` (~1.84 × 10¹⁹), the upper bits are silently discarded and the function returns a drastically smaller value with no error.

This is directly inconsistent with the identical u128 intermediate pattern used in `dao_field_with_current_epoch` in the same file, which correctly uses checked conversion:

```rust
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
``` [2](#0-1) 

```rust
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [3](#0-2) 

The `calculate_maximum_withdraw` function is called from `transaction_maximum_withdraw`, which is called from `withdrawed_interests`, which feeds directly into `dao_field_with_current_epoch`:

```rust
let withdrawed_interests = self.withdrawed_interests(rtxs)?;
...
let current_s = parent_s
    .safe_add(nervosdao_issuance)
    .and_then(|s| s.safe_sub(withdrawed_interests))?;
``` [4](#0-3) 

`current_s` is the NervosDAO secondary issuance pool stored in the block header's DAO field, which is a consensus-critical value. A truncated `withdrawed_interests` causes `current_s` to be overstated, corrupting the DAO field for every subsequent block.

The same function is also exposed directly via the `calculate_dao_maximum_withdraw` RPC: [5](#0-4) 

The accumulate rate `ar` starts at `10^16` and grows at approximately the secondary issuance rate (~4% per year). After ~100 years, `ar` could reach ~5 × 10¹⁷. For `withdraw_counted_capacity` to exceed `u64::MAX`:

```
counted_capacity × withdrawing_ar / deposit_ar > 1.84 × 10¹⁹
counted_capacity × 50 > 1.84 × 10¹⁹
counted_capacity > 3.68 × 10¹⁷ shannons ≈ 3.68 billion CKB
```

This is ~11% of total CKB supply — plausible for a large institutional depositor in a long-lived chain.

The `secondary_block_issuance` function in `EpochExt` also divides by `self.length()` without a zero-check:

```rust
let mut g2 = Capacity::shannons(secondary_epoch_issuance.as_u64() / self.length());
``` [6](#0-5) 

However, epoch length is validated by `EpochVerifier` before reaching this path, so that is a separate, lower-severity concern.

### Impact Explanation

**Impact: Medium-High**

1. **Consensus-critical DAO field corruption**: A block containing a large DAO withdrawal transaction would have an incorrect `current_s` value in its DAO field. Nodes that independently verify the DAO field would reject the block, causing a chain split or block rejection.

2. **Depositor fund loss**: The `calculate_dao_maximum_withdraw` RPC returns a silently truncated (wrong) maximum withdrawal amount. A depositor constructing a withdrawal transaction based on this value would receive far less than their entitled interest, with no error or warning.

3. **Irreversibility**: Once a withdrawal transaction is committed with the truncated amount, the depositor cannot recover the lost interest.

### Likelihood Explanation

**Likelihood: Low**

The condition requires a single DAO cell holding approximately 3.68 billion CKB or more, held for approximately 100+ years. While this is an extreme scenario on today's chain, CKB is designed as a long-lived store-of-value chain, and the vulnerability is latent and grows worse over time as `ar` increases. No privileged access is required — any transaction sender can trigger this path by submitting a valid DAO withdrawal transaction.

### Recommendation

Replace the silent `as u64` cast with the same checked conversion pattern used elsewhere in the same file:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(
        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
    )
    .safe_add(occupied_capacity)?;
```

This makes the overflow explicit and consistent with `miner_issuance128` and `ar_increase128` handling in `dao_field_with_current_epoch`.

### Proof of Concept

The inconsistency is directly visible by comparing the two patterns in the same file:

**Vulnerable (line 156):**
```rust
Capacity::shannons(withdraw_counted_capacity as u64)  // silent truncation
``` [7](#0-6) 

**Safe pattern used elsewhere (lines 244–245, 258):**
```rust
u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?
u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?
``` [2](#0-1) [3](#0-2) 

A concrete numeric demonstration:
- `counted_capacity` = 4 × 10¹⁷ shannons (4 billion CKB)
- `withdrawing_ar` = 5 × 10¹⁷ (after ~100 years of growth)
- `deposit_ar` = 10¹⁶ (genesis value)
- `withdraw_counted_capacity` = 4×10¹⁷ × 5×10¹⁷ / 10¹⁶ = 2×10¹⁹
- `u64::MAX` = 1.844×10¹⁹
- `2×10¹⁹ as u64` = `2×10¹⁹ - 2^64` ≈ 1.56×10¹⁸ (silently wrong, ~12.8× too small)

The depositor would receive approximately 1.56 billion CKB instead of the correct ~2 billion CKB, losing ~440 million CKB in interest with no error raised. The `withdrawed_interests` fed into the DAO field would be equally wrong, corrupting `current_s` in the block header.
