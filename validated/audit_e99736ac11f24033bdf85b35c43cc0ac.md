### Title
Expired Network Alerts Can Be Replayed via P2P Because `AlertRelayer::received()` Skips `notice_until` Validation and `clear_expired_alerts()` Reopens Acceptance - (`File: util/network-alert/src/alert_relayer.rs`)

---

### Summary

The P2P alert relay handler in `util/network-alert/src/alert_relayer.rs` accepts incoming `Alert` messages without checking whether `notice_until` is in the future. Separately, `Notifier::clear_expired_alerts()` in `util/network-alert/src/notifier.rs` removes expired alerts from `received_alerts` without recording their IDs in `cancel_filter`, so `has_received(id)` returns `false` after expiry. Any unprivileged peer that captured a legitimately-signed alert while it was live can replay its raw bytes after it expires, causing the alert to be re-accepted, re-broadcast to all connected peers, and re-displayed to node operators indefinitely.

---

### Finding Description

**Root cause 1 — missing `notice_until` check in the P2P relay path**

`AlertRpcImpl::send_alert` (the RPC entry point) explicitly rejects alerts whose `notice_until` is in the past:

```rust
// rpc/src/module/alert.rs:104-110
let now_ms = ckb_systemtime::unix_time_as_millis();
let notice_until: u64 = alert.raw().notice_until().into();
if notice_until < now_ms {
    return Err(RPCError::invalid_params(...));
}
``` [1](#0-0) 

The P2P relay handler `AlertRelayer::received()` performs no equivalent check. After parsing and a UTF-8 validation, it only tests `has_received(alert_id)` and then calls `verify_signatures`. There is no guard against `notice_until < now`:

```rust
// util/network-alert/src/alert_relayer.rs:144-162
let alert_id = alert.as_reader().raw().id().into();
// ignore alert
if self.notifier.lock().has_received(alert_id) {
    return;
}
// verify
if let Err(err) = self.verifier.verify_signatures(&alert) { ... }
// ... broadcast and add
``` [2](#0-1) 

**Root cause 2 — `clear_expired_alerts` reopens the acceptance window**

`Notifier::clear_expired_alerts` removes expired entries from `received_alerts` (a `HashMap`) and from `noticed_alerts` (a `Vec`), but it does **not** insert the expired IDs into `cancel_filter`:

```rust
// util/network-alert/src/notifier.rs:135-144
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|_id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        notice_until > now
    });
    self.noticed_alerts.retain(|a| { ... });
}
``` [3](#0-2) 

`has_received` checks only `received_alerts` and `cancel_filter`:

```rust
pub fn has_received(&self, id: u32) -> bool {
    self.received_alerts.contains_key(&id) || self.cancel_filter.contains(&id)
}
``` [4](#0-3) 

After `clear_expired_alerts` runs, `has_received(id)` returns `false` for any previously-seen alert whose `notice_until` has passed. Combined with root cause 1, the relay handler will fully re-process and re-broadcast the replayed alert.

`clear_expired_alerts` is triggered every time a new peer connects:

```rust
// util/network-alert/src/alert_relayer.rs:87
self.clear_expired_alerts();
``` [5](#0-4) 

**The `RawAlert` schema confirms the signed payload includes `notice_until`** — the field is present and is part of the hash that is signed — but the relay path never validates it:

```
// util/gen-types/schemas/extensions.mol:445-453
table RawAlert {
    notice_until:   Uint64,
    id:             Uint32,
    ...
}
``` [6](#0-5)

### Impact Explanation

An unprivileged peer that was connected to the network while a legitimate developer alert was live has a copy of the fully-signed alert bytes. After the alert expires and `clear_expired_alerts` is triggered (by any new peer connection), the attacker can send those bytes to any CKB node over the Alert P2P protocol. The node will:

1. Pass the `has_received` check (returns `false` after expiry).
2. Pass signature verification (original developer signatures are still valid).
3. Re-add the alert to `received_alerts` and `noticed_alerts`.
4. Re-broadcast it to all connected peers.

Every node in the network will re-display the outdated alert (e.g., "upgrade immediately — critical bug in v0.105.*") to operators. This can cause false alarms, unnecessary emergency responses, and confusion about the actual state of the network. The attack can be repeated indefinitely because each replay cycle re-populates `received_alerts`, which will again be cleared on the next peer connection, reopening the window.

---

### Likelihood Explanation

The attack requires only:
- A copy of any previously-broadcast alert (trivially obtained by any peer that was connected during the original broadcast, or by monitoring the P2P network).
- The ability to connect to CKB nodes (any unprivileged peer).
- No cryptographic capability — the original developer signatures are reused verbatim.

The `clear_expired_alerts` trigger fires on every new peer connection, so the acceptance window reopens frequently in a live network.

---

### Recommendation

1. **Add a `notice_until` check in `AlertRelayer::received()`**, mirroring the check already present in `AlertRpcImpl::send_alert`:

```rust
let now_ms = ckb_systemtime::unix_time_as_millis();
let notice_until: u64 = alert.as_reader().raw().notice_until().into();
if notice_until < now_ms {
    nc.ban_peer(..., "send us an expired alert");
    return;
}
```

2. **In `clear_expired_alerts`, insert expired IDs into `cancel_filter`** before removing them from `received_alerts`, so `has_received` continues to return `true` for those IDs:

```rust
pub fn clear_expired_alerts(&mut self, now: u64) {
    self.received_alerts.retain(|id, alert| {
        let notice_until: u64 = alert.raw().notice_until().into();
        if notice_until <= now {
            self.cancel_filter.put(*id, ());
            false
        } else {
            true
        }
    });
    ...
}
```

Either fix alone is sufficient to block the replay; both together provide defense in depth.

---

### Proof of Concept

1. Connect a monitoring peer to the CKB network and capture the raw bytes of any broadcast `Alert` message (e.g., the real alert `20230001` whose signatures are publicly known from `util/network-alert/src/tests/generate_alert_signature.rs`). [7](#0-6) 

2. Wait for the alert's `notice_until` timestamp to pass.

3. Trigger a new peer connection to the target node (causing `clear_expired_alerts` to run and remove the alert from `received_alerts`).

4. Send the captured raw alert bytes to the target node on the Alert protocol channel.

5. Observe that `AlertRelayer::received()` accepts the message (no `notice_until` check, `has_received` returns `false`), re-adds it to `noticed_alerts`, and re-broadcasts it to all peers — confirmed by querying `get_blockchain_info` and seeing the expired alert re-appear in the `alerts` field.
