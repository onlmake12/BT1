Audit Report

## Title
Unbounded JSON-RPC Batch Request Processing Causes RPC Service Exhaustion — (`rpc/src/server.rs`)

## Summary
The `JSONRPC_BATCH_LIMIT` guard in `handle_jsonrpc` is only enforced when the operator explicitly sets `rpc_batch_limit` in config. By default the `OnceLock` is never initialized, so `JSONRPC_BATCH_LIMIT.get()` returns `None` and the `if let Some(...)` branch is never taken, allowing any caller to submit an arbitrarily large JSON-RPC batch within the 10 MiB body limit. This causes unbounded sequential RPC call processing, making the RPC API unresponsive to legitimate callers.

## Finding Description
In `rpc/src/server.rs` line 34, `JSONRPC_BATCH_LIMIT` is declared as `static JSONRPC_BATCH_LIMIT: OnceLock<usize> = OnceLock::new()`. It is only initialized at lines 53–55 when `config.rpc_batch_limit` is `Some(...)`. In `util/app-config/src/configs/rpc.rs` line 44, `rpc_batch_limit` is typed `Option<usize>` with no default value, so it is `None` unless explicitly configured. In `resource/ckb.toml` lines 205–208, the field is commented out with the note "By default, there is no limitation on the size of batch request size."

At `server.rs` lines 274–282, the guard is:
```rust
if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
    && calls.len() > *batch_size
```
When `JSONRPC_BATCH_LIMIT` is uninitialized, `.get()` returns `None`, the condition is never true, and execution falls through to `stream::iter(calls).then(...)` at lines 284–289, which processes every call in the batch sequentially. The only existing constraint is `max_request_body_size = 10485760` (10 MiB). Within 10 MiB, a caller can pack tens of thousands of small calls (e.g., `get_tip_block_number` at ~80 bytes each ≈ 125,000 calls). The `TimeoutLayer` at 30 seconds limits individual request duration but does not prevent repeated or concurrent batch submissions that continuously saturate the async worker pool. The test suite at `rpc/src/tests/setup.rs` line 184 explicitly sets `rpc_batch_limit: Some(1000)`, confirming the limit is inactive in production by default.

## Impact Explanation
An attacker submits a single HTTP POST with a JSON array of thousands of RPC calls. The server processes all calls sequentially, consuming CPU and memory proportional to batch size, and blocking the async executor from handling other requests. This renders the RPC API unresponsive to legitimate callers (miners, wallets, monitoring). This matches the allowed CKB bounty impact: **Note (0–500 points) — Any local RPC API crash**, as the RPC service becomes non-functional under this attack. The node's core consensus and p2p layers are not directly affected, so "crash a CKB node" (High) is not met. The default localhost binding (`127.0.0.1:8114`) further constrains the attack surface to local processes unless the operator has exposed the port.

## Likelihood Explanation
The default binding to `127.0.0.1` means remote exploitation requires the operator to have exposed the RPC port, which is common for wallet/dApp connectivity but not universal. Local exploitation (malicious process on the same host, compromised dependency) requires no privilege and no network exposure. The attack is trivially repeatable with a single `curl` command and requires no authentication. The absence of a default limit is explicitly documented in the default config, meaning most production deployments are affected unless the operator has read and acted on the comment.

## Recommendation
1. Change `rpc_batch_limit` in `RpcConfig` (`util/app-config/src/configs/rpc.rs`) from `Option<usize>` to `usize` with a safe default (e.g., 200 or 2000), enforced unconditionally.
2. Alternatively, add a hard-coded fallback maximum in `handle_jsonrpc` (`rpc/src/server.rs` lines 275–282) that applies when `JSONRPC_BATCH_LIMIT` is uninitialized, allowing the config to lower but not raise the limit.
3. Update `resource/ckb.toml` to enable `rpc_batch_limit` by default rather than leaving it commented out.

## Proof of Concept
```bash
# Build a batch of 10,000 get_tip_block_number calls (~800 KB, well under 10 MiB)
python3 -c "
import json, sys
batch = [{'id': i, 'jsonrpc': '2.0', 'method': 'get_tip_block_number', 'params': []} for i in range(10000)]
sys.stdout.write(json.dumps(batch))
" > batch.json

# Send to a default CKB node (no rpc_batch_limit configured)
curl -s -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  --data-binary @batch.json | wc -c
```
With `rpc_batch_limit` absent from `ckb.toml`, the server processes all 10,000 calls and returns a 10,000-element JSON array. Repeating this in a tight loop from a local process saturates the RPC runtime and blocks all concurrent RPC requests. Substituting `get_block_template` amplifies CPU cost per call significantly.
