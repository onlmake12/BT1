Looking at the vulnerability class — **missing authorization check for a privileged state-changing operation** — I searched the CKB codebase for functions that perform privileged pool/state mutations without verifying the caller holds any required role.

The most direct analog is in `rpc/src/module/pool.rs`.

---

### Title
Any RPC Caller Can Remove Transactions or Wipe the Entire Tx-Pool Without Authorization — (`rpc/src/module/pool.rs`)

### Summary
`PoolRpcImpl::remove_transaction` and `PoolRpcImpl::clear_tx_pool` perform privileged, irreversible state mutations on the transaction pool with zero authorization check. Any caller who can reach the JSON-RPC endpoint — explicitly an in-scope attacker profile — can silently drop individual transactions or flush the entire pending pool, bypassing any operator-level safeguard.

### Finding Description

`remove_transaction` (lines 662–669) calls `tx_pool.remove_local_tx(tx_hash.into())` directly:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| {
        error!("Send remove_tx request error {}", e);
        RPCError::ckb_internal_error(e)
    })
}
```

`clear_tx_pool` (lines 684–692) calls `tx_pool.clear_pool(snapshot)` directly:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool
        .clear_pool(snapshot)
        .map_err(|err| RPCError::custom(RPCError::Invalid, err.to_string()))?;
    Ok(())
}
```

Neither method carries a `#[cfg(feature = "internal")]` gate, an IP-allowlist check, a token/credential check, or any role assertion. They are unconditionally registered and served in production builds.

The analog to the Fei Protocol report is structural: just as `_burnFeiHeld` performs a privileged action (burning tokens) without checking the Burner role — meaning the safeguard of revoking the role is ineffective — these RPC methods perform privileged actions (removing/clearing transactions) without checking any operator role. If a node operator intends to protect the pool by restricting access, there is no in-code enforcement; the protection depends entirely on network-level configuration outside the application.

The downstream call chain is:

```
RPC caller → PoolRpcImpl::remove_transaction / clear_tx_pool
           → TxPoolController::remove_local_tx / clear_pool   (service.rs:371-378)
           → TxPool::clear / pool_map.clear                   (pool.rs:516-522, pool_map.rs:387-396)
```

No authorization gate exists at any layer of this chain.

### Impact Explanation

- **`remove_transaction`**: An attacker can selectively evict any specific pending transaction by hash. This enables targeted transaction censorship — e.g., preventing a user's time-sensitive transaction (e.g., a `since`-locked UTXO spend) from being included in a block. The victim must detect the drop and resubmit, potentially missing a time window.
- **`clear_tx_pool`**: An attacker can atomically drop every pending and proposed transaction from the pool. All in-flight transactions across all users of that node are silently discarded. This is a complete pool-state wipe with no notification to submitters.

Both operations are state-destructive and have no undo path.

### Likelihood Explanation

The RPC binds to `127.0.0.1` by default, but:
1. The bounty scope explicitly lists **"RPC caller"** and **"supported local CLI/RPC user"** as valid unprivileged attacker profiles.
2. Nodes serving exchanges, dApps, or light clients routinely expose the RPC on `0.0.0.0` or behind a reverse proxy with no per-method authentication.
3. No code change or special configuration is required by the attacker — a single valid JSON-RPC call suffices.
4. Neither method requires any parameter that is difficult to obtain (`tx_hash` is public; `clear_tx_pool` takes no parameters).

### Recommendation

Gate `remove_transaction` and `clear_tx_pool` (and `clear_tx_verify_queue`) behind an explicit operator-role check or a configurable allowlist of caller IPs/tokens enforced at the RPC handler layer — not solely at the network perimeter. A minimal fix is to add a configurable `rpc_admin_token` that must be supplied as a header or parameter for these destructive methods, analogous to how Bitcoin Core gates `stop`, `pruneblockchain`, etc.

### Proof of Concept

```bash
# 1. Start CKB node (RPC on default or exposed port)
# 2. Submit a transaction
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"send_transaction","params":[<tx_json>, "passthrough"],"id":1}'
# → returns tx_hash

# 3. Any caller removes it — no credential required
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"remove_transaction","params":["<tx_hash>"],"id":2}'
# → returns true; transaction silently gone from pool

# 4. Or wipe the entire pool
curl -X POST http://127.0.0.1:8114 \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","method":"clear_tx_pool","params":[],"id":3}'
# → returns null; all pending transactions dropped
