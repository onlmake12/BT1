### Title
Unbounded `pending_get_block_proposals` DashMap Growth via Unauthenticated `GetBlockProposal` Messages — (`sync/src/relayer/get_block_proposal_process.rs`, `sync/src/types/mod.rs`)

---

### Summary

An unprivileged relay peer can send `GetBlockProposal` messages containing proposal short IDs absent from the local tx-pool. These IDs are inserted into `pending_get_block_proposals` — an unbounded `DashMap` — with no size cap. A rate limiter exists but only throttles per-peer rate; with many peers sending distinct IDs, the map grows unboundedly between timer drains, enabling memory exhaustion DoS.

---

### Finding Description

`GetBlockProposalProcess::execute` enforces two checks before inserting into the map:

1. **Per-message count limit**: proposals must not exceed `max_block_proposals_limit * max_uncles_num` (mainnet: 1500 × 2 = 3000).
2. **Duplicate check**: all IDs in a single message must be unique. [1](#0-0) 

After `fetch_txs` returns, proposals absent from the tx-pool are passed directly to `insert_get_block_proposals` with no further bound: [2](#0-1) 

`insert_get_block_proposals` iterates over all IDs and inserts them into the DashMap with `or_default().insert(pi)` — no capacity check, no eviction: [3](#0-2) 

The DashMap itself is declared with no capacity limit: [4](#0-3) 

The only drain mechanism is `prune_tx_proposal_request`, called on a periodic timer (`TX_PROPOSAL_TOKEN`): [5](#0-4) 

`drain_get_block_proposals` clones the entire map before clearing it, temporarily doubling peak memory: [6](#0-5) 

---

### Impact Explanation

A `ProposalShortId` is 10 bytes. Each DashMap entry also carries a `HashSet<PeerIndex>` and DashMap shard overhead (~40–64 bytes per entry). With the rate limiter set at 30 req/s per `(peer, message_type)`: [7](#0-6) 

and a default peer cap of ~125 inbound + 8 outbound = ~133 peers: [8](#0-7) 

the maximum insertion rate is `133 peers × 30 req/s × 3000 IDs/req ≈ 12M entries/second`. At ~50 bytes/entry, that is ~600 MB/s of map growth. If the drain timer fires every few seconds, the map can accumulate several GB before being cleared. During the drain, `clone()` doubles peak RSS. The result is OOM / node crash.

---

### Likelihood Explanation

- **Attacker capability**: Any unprivileged peer can send `GetBlockProposal`. No PoW, no key, no privileged role required.
- **Rate limiter**: Limits each peer to 30 req/s but does **not** ban the peer on excess — the message is silently dropped and the attacker retries at the limit indefinitely.
- **Peer count**: Bounded but not small; an attacker operating a Sybil cluster of inbound connections (up to `max_inbound`, default 125) can sustain the attack.
- **ID space**: `ProposalShortId` is 10 bytes (80 bits); the attacker can generate effectively unlimited distinct IDs, preventing DashMap key deduplication from helping.
- **Local testability**: Straightforward to simulate with M goroutines/threads each opening a P2P session and sending max-limit `GetBlockProposal` messages with distinct random IDs, measuring DashMap size before and after the drain timer fires.

---

### Recommendation

1. **Add an explicit size cap** to `insert_get_block_proposals`: reject or drop insertions once `pending_get_block_proposals.len()` exceeds a configurable maximum (e.g., `max_peers × max_block_proposals_limit`).
2. **Per-peer entry accounting**: track how many entries each peer has contributed and evict the oldest or cap per-peer contribution.
3. **Reduce timer interval** or trigger `prune_tx_proposal_request` more aggressively under memory pressure.
4. **Avoid cloning** the full map in `drain_get_block_proposals`; use `std::mem::take` on a `Mutex`-wrapped map instead to eliminate the peak-doubling effect.

---

### Proof of Concept

```
1. Attacker opens N inbound connections to the victim node (N ≤ max_inbound, e.g. 125).
2. Each connection sends GetBlockProposal messages at 30 req/s (rate limit),
   each containing 3000 distinct random ProposalShortIds not present in the victim's tx-pool.
3. fetch_txs returns empty for all IDs → all 3000 IDs per message are passed to
   insert_get_block_proposals → inserted into pending_get_block_proposals with no cap.
4. Insertion rate: 125 × 30 × 3000 = 11.25M entries/second.
5. Measure pending_get_block_proposals.len() before and after the drain timer fires.
6. If timer interval T seconds: map holds ~11.25M × T entries = ~562 MB × T before drain.
7. During drain, clone() doubles peak RSS.
8. With T ≥ 5s: >2.8 GB peak RSS → OOM on constrained nodes.
``` [9](#0-8) [3](#0-2)
