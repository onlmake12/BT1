### Title
WeightUnitsFlow Fee Estimator Flow Window Can Be Gamed via Minimum-Fee Transaction Flooding - (File: `util/fee-estimator/src/estimator/weight_units_flow.rs`)

### Summary

The `WeightUnitsFlow` fee estimator records every accepted transaction in a sliding block-number window and computes a `flow_speed` by dividing total historical weight by `historical_blocks`. An unprivileged tx-pool submitter can flood the pool with minimum-fee-rate transactions to inflate the historical flow speed for the lowest fee bucket, causing `estimate_fee_rate` to either recommend an artificially elevated fee rate or return `NoProperFeeRate`, misleading wallets and services that rely on this RPC.

### Finding Description

`accept_tx` stores every accepted transaction keyed by `current_tip`: [1](#0-0) 

`do_estimate` then computes `flow_speed_buckets` by summing all historical weight in the window and dividing by `historical_blocks`: [2](#0-1) 

The decision for each bucket is: [3](#0-2) 

`historical_blocks` is `target_blocks * 2`, so for the default `MAX_TARGET = 128` blocks the window spans the last **256 blocks** (~2 hours): [4](#0-3) [5](#0-4) 

The `expire()` function only evicts entries older than `historical_blocks(MAX_TARGET)` blocks, so transactions that were already confirmed or time-expired from the pool **remain in the flow history** until 256 blocks have passed: [6](#0-5) 

There is no rate-limiting, authentication, or cap on how many transactions a single submitter can inject into the historical window. The minimum fee rate is a static config value: [7](#0-6) 

### Impact Explanation

An attacker who submits a large volume of minimum-fee-rate transactions over the 256-block historical window inflates `flow_speed_buckets[1]` (the minimum bucket). When `estimate_fee_rate` is subsequently called:

- `added_weight = flow_speed * target_blocks` becomes large for the minimum bucket.
- `current_weight + added_weight > removed_weight` causes the minimum bucket to fail.
- The estimator either recommends a higher fee tier than the network actually requires, or returns `Err(NoProperFeeRate)`.

Wallets, exchanges, and services that call `estimate_fee_rate` are misled into overpaying fees or failing to obtain an estimate. The attacker's cost is the fees paid for the spam transactions (which go to miners, not the attacker), making this a low-cost manipulation relative to the disruption caused.

### Likelihood Explanation

- The `WeightUnitsFlow` estimator must be explicitly enabled via config (`fee_estimator_config`), so only nodes that opt in are affected. [8](#0-7) 

- Any unprivileged RPC caller can submit transactions via `send_transaction` as long as they meet `min_fee_rate`.
- The attack window is 256 blocks (~2 hours), which is a realistic sustained effort.
- No mining power, key material, or privileged access is required.

### Recommendation

1. **Cap per-block contribution**: Limit the total weight recorded per block in `accept_tx` to prevent a single submitter from dominating the historical window.
2. **Weight by confirmation**: Only count transactions that were actually confirmed in a block toward the flow speed, rather than all transactions that entered the pool (analogous to the original report's suggestion to use volume over a timespan).
3. **Outlier dampening**: Apply a dampening filter (similar to `bounding_hash_rate` in the difficulty adjustment) to prevent the flow speed from changing too rapidly between estimation windows. [9](#0-8) 

### Proof of Concept

1. Enable `WeightUnitsFlow` in node config.
2. Over 256 blocks, repeatedly call `send_transaction` with minimum-fee-rate transactions (fee = `min_fee_rate * tx_size`). Each accepted transaction is stored via `accept_tx` at the current block height.
3. Allow those transactions to be confirmed or expire from the pool (they remain in `self.txs` until 256 blocks pass).
4. Submit one legitimate minimum-fee transaction to the current mempool.
5. Call `estimate_fee_rate`. The estimator computes `flow_speed_buckets[1]` as `(spam_weight_total) / 256`. With sufficient spam, `current_weight + flow_speed * target_blocks > removed_weight` for bucket 1, causing the estimator to skip it and return either a higher fee tier or `NoProperFeeRate`. [10](#0-9)
