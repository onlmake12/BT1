Audit Report

## Title
Timeout-Free Blocking `recv()` in `blocking_process_block_internal` Enables RPC Thread Exhaustion via Concurrent `submit_block` Calls — (`chain/src/chain_controller.rs`)

## Summary

`blocking_process_block_internal` in `chain/src/chain_controller.rs` enqueues a block for chain processing and then blocks the calling thread indefinitely on `verify_result_rx.recv()` with no timeout. The `submit_block` RPC handler calls this synchronously from within the axum/tokio async RPC server, blocking tokio worker threads for the full duration of block verification. Any RPC caller can send concurrent `submit_block` requests to exhaust the tokio thread pool and render the RPC interface unresponsive.

## Finding Description

**Root cause — timeout-free blocking recv:**

`blocking_process_block_internal` at `chain/src/chain_controller.rs:79–109` creates a oneshot channel, attaches the sender as a `verify_callback` on the `LonelyBlock`, enqueues the block via `asynchronous_process_lonely_block`, and then blocks unconditionally:

```rust
self.asynchronous_process_lonely_block(lonely_block);
verify_result_rx.recv().unwrap_or_else(|err| { ... })
```

`verify_result_rx.recv()` is `std::sync::mpsc::Receiver::recv()` — a blocking, timeout-free call. It returns only when the chain service fires the `verify_callback` with the verification result. There is no `recv_timeout`, no `select!` with a deadline, and no cancellation path.

**Call path from RPC:**

`submit_block` at `rpc/src/module/miner.rs:260–298` calls `self.chain.blocking_process_block(Arc::clone(&block))` at line 295–298, which calls `blocking_process_block_internal` directly. The `submit_block` method is a synchronous `fn` (not `async fn`) registered with the axum/tokio-based RPC server. When tokio dispatches this synchronous handler, the blocking `recv()` stalls a tokio worker thread for the entire duration of block verification.

**Partial mitigation that does not prevent exhaustion:**

The `process_block_sender` channel is bounded at 24 (`channel::bounded(24)` at `chain/src/init.rs:93`). `call_without_response` uses `sender.send()` (crossbeam blocking send), so callers beyond the 24-slot queue block on the send itself rather than the recv. This provides backpressure but does not prevent thread exhaustion — all concurrent callers still hold a tokio worker thread blocked (either on send or recv) until the chain service drains the queue. The claim's assertion that there is "no guard" is slightly overstated, but the bounded channel of 24 does not bound the number of blocked threads.

**No rate limiting or concurrency cap on `submit_block`:**

The RPC server (`rpc/src/server.rs`) applies only a 30-second HTTP timeout at the transport layer (`TimeoutLayer`) and an optional batch size limit. There is no per-method concurrency cap, no semaphore, and no `submit_block`-specific rate limit. The 30-second transport timeout does not cancel the in-flight `recv()` — the tokio task is still blocked even if the HTTP connection closes.

## Impact Explanation

Each concurrent `submit_block` call holds one tokio worker thread blocked in `verify_result_rx.recv()`. The axum RPC server runs on a shared tokio runtime. Exhausting the tokio thread pool (default: number of CPU cores for worker threads, up to 512 for blocking threads depending on dispatch) causes the RPC interface to become unresponsive — no further RPC calls (`get_tip_block_number`, `send_transaction`, `get_block_template`, etc.) can be served. This maps to **Note (0–500 points): Any local RPC API crash**, with potential escalation to **High: Vulnerabilities which could easily crash a CKB node** if the tokio runtime is shared with P2P and sync services (which cannot be fully confirmed from the reviewed code alone).

## Likelihood Explanation

The `submit_block` RPC endpoint is reachable by any configured RPC caller — miners, internal tooling, or any party with HTTP access to the RPC port. No cryptographic material, special privileges, or majority hashpower is required. The attacker only needs to send concurrent HTTP POST requests. Even syntactically valid but already-known blocks will queue in the chain service and hold all calling threads until processed. The bounded channel of 24 means 24 concurrent requests suffice to begin saturating the queue; additional concurrent requests block on the channel send itself, still holding threads.

## Recommendation

1. Replace `verify_result_rx.recv()` with `verify_result_rx.recv_timeout(Duration::from_secs(N))` in `blocking_process_block_internal`, returning an error if verification does not complete within the deadline.
2. Wrap the synchronous `submit_block` handler body in `tokio::task::spawn_blocking` or use `block_in_place` (as the tx-pool path does) to avoid blocking tokio worker threads.
3. Apply a concurrency semaphore on `submit_block` to bound the number of simultaneous in-flight block submissions.
4. Consider adopting the asynchronous path (`asynchronous_process_remote_block`) for `submit_block`, returning the block hash immediately and allowing callers to poll for confirmation.

## Proof of Concept

```bash
# Send 30 concurrent submit_block RPC calls (exceeds bounded channel of 24)
# Each call blocks a tokio worker thread in verify_result_rx.recv() with no timeout
for i in $(seq 1 30); do
  curl -s -X POST http://127.0.0.1:8114/ \
    -H 'Content-Type: application/json' \
    -d '{"id":1,"jsonrpc":"2.0","method":"submit_block","params":["0", <VALID_BLOCK_JSON>]}' &
done

# Attempt any other RPC call — will hang or time out
curl -s -X POST http://127.0.0.1:8114/ \
  -H 'Content-Type: application/json' \
  -d '{"id":2,"jsonrpc":"2.0","method":"get_tip_block_number","params":[]}'
```
