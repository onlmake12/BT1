### Title
MEV: Keeper Can Inflate `tx.gasprice` to Drain Subscription Balance Beyond Market Rate - (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._processFeesAndPayKeeper` reimburses the keeper using the raw `tx.gasprice` opcode with no upper bound or oracle check. A malicious keeper can submit `updatePriceFeeds` with an artificially inflated gas price — either by setting a very high `maxPriorityFeePerGas` on EIP-1559 chains, or by using a flashbots coinbase-bribe bundle on legacy chains — and extract the inflated amount directly from the subscription's deposited balance. The keeper pays the inflated gas cost to the miner but is made whole by the subscription, effectively transferring subscription funds to the miner (or to the keeper themselves if they are the block proposer).

---

### Finding Description

In `_processFeesAndPayKeeper` (Scheduler.sol line 846):

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
// ...
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
```

`tx.gasprice` on EIP-1559 chains equals `baseFeePerGas + priorityFeePerGas`. The keeper fully controls `maxPriorityFeePerGas` in their transaction. There is no cap, no TWAP oracle, and no comparison against a reference gas price. The keeper can set `maxPriorityFeePerGas` to an arbitrarily large value; the miner (or validator) will happily include the transaction because they receive the large tip. The keeper then recovers the entire inflated `tx.gasprice * gasUsed` from the subscription balance.

On legacy (non-EIP-1559) chains the same effect is achievable via a flashbots bundle: the keeper submits `updatePriceFeeds` with a high `gasPrice` and a separate coinbase-transfer bribe transaction. The miner accepts the bundle for its total reward; the keeper's `updatePriceFeeds` transaction lands with the inflated `tx.gasprice`, and the subscription pays accordingly.

The `updatePriceFeeds` entry point is permissionless — any address can call it — so no privileged role is required.

---

### Impact Explanation

A malicious keeper can drain a subscription's ETH balance at a rate orders of magnitude above the true market gas cost. For example, if the market base fee is 10 gwei and the keeper sets `maxPriorityFeePerGas = 990 gwei`, `tx.gasprice` becomes 1000 gwei and the subscription is charged 100× the real cost per update. The subscription runs out of funds far sooner than the owner anticipated, causing price feed updates to halt. DeFi protocols that depend on those feeds (e.g., lending protocols using Pyth Pulse prices) would then operate on stale data, potentially enabling liquidation manipulation or other downstream exploits. The subscription owner suffers direct loss of deposited ETH with no recourse.

---

### Likelihood Explanation

The attack requires only a standard EVM transaction with a high `maxPriorityFeePerGas`; no special tooling is needed on EIP-1559 chains. The keeper network is explicitly permissionless (anyone can call `updatePriceFeeds`). The economic incentive exists whenever the keeper is also the block proposer (common in MEV-boost environments) or has a private arrangement with one. The attack is cheap to execute and repeatable on every update interval.

---

### Recommendation

Replace the raw `tx.gasprice` read with a capped or oracle-sourced gas price:

1. **Hard cap**: Store an admin-configurable `maxGasPriceInWei` and use `min(tx.gasprice, maxGasPriceInWei)` in the fee calculation.
2. **EIP-1559 base-fee only**: Use `block.basefee` instead of `tx.gasprice`. The base fee is protocol-enforced and cannot be inflated by the keeper; the keeper's tip is excluded from the reimbursement.
3. **Gas price oracle**: Integrate a TWAP-based gas price oracle (e.g., a rolling average of recent `block.basefee` values) and cap reimbursement to that value.

Option 2 is the simplest and most robust for EIP-1559 chains:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * block.basefee;
```

For chains that do not support EIP-1559, a hard cap set by governance is the appropriate fallback.

---

### Proof of Concept

1. Attacker deploys a keeper bot.
2. A subscription exists with `balanceInWei = 1 ether`, market gas price = 10 gwei.
3. Attacker calls `updatePriceFeeds(subscriptionId, updateData)` with `maxPriorityFeePerGas = 990 gwei` (so `tx.gasprice = 1000 gwei`).
4. Inside `_processFeesAndPayKeeper`:
   - `gasUsed ≈ 200,000` (typical for the function)
   - `gasCost = 200,000 * 1000 gwei = 0.2 ETH`
   - `totalKeeperFee = 0.2 ETH + keeperSpecificFee`
5. The subscription balance is reduced by ~0.2 ETH in a single update instead of the expected ~0.002 ETH.
6. After ~5 updates the subscription is fully drained; price feeds stop updating.
7. The attacker (if also the block proposer) nets the 990 gwei/gas tip back as the validator reward, making the attack nearly free.

```text
    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
