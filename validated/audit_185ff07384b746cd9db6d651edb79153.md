### Title
`n_sync_started` Counter Permanently Leaked on Peer Disconnect Race in `start_sync_headers` — (File: `sync/src/synchronizer/mod.rs`)

---

### Summary

In `start_sync_headers`, the `n_sync_started` atomic counter is incremented **before** the corresponding peer state is transitioned to `Started`. If the peer disconnects in the narrow window between those two operations, the counter is permanently incremented with no path to decrement it. During IBD, `n_sync_started != 0` is the sole gate that prevents any new header-sync from starting, so a single successful race permanently freezes the node in IBD, unable to sync any further blocks.

---

### Finding Description

`start_sync_headers` in `sync/src/synchronizer/mod.rs` runs on a periodic timer (`SEND_GET_HEADERS_TOKEN`). Its logic is:

1. **Collect** eligible peers into a `Vec<PeerIndex>` by iterating `self.peers().state` (lines 656–662).
2. For each peer, **atomically increment** `n_sync_started` (lines 672–682).
3. **Separately** call `peer_state.start_sync(…)` via a second DashMap lookup (lines 683–687). [1](#0-0) 

Step 3 is a non-atomic, independent operation. If the peer disconnects between steps 1 and 3, `Peers::disconnected` removes the peer from `self.state` and checks `peer_state.sync_started()` before decrementing `n_sync_started`: [2](#0-1) 

Because `start_sync` was never called, `sync_started()` returns `false`, so `disconnected` does **not** decrement the counter. The counter is now `1` with no peer in `Started` state and no future code path that will ever decrement it.

`n_sync_started` is only decremented in `suspend_sync`, `tip_synced`, and `disconnected`, all of which guard on `peer_state.sync_started()`: [3](#0-2) 

The IBD gate in `start_sync_headers` is:

```rust
if ibd && x != 0 { None } else { Some(x + 1) }
``` [4](#0-3) 

Once `n_sync_started` is stuck at `1`, this condition permanently prevents any new sync from starting during IBD.

---

### Impact Explanation

A node stuck in IBD with `n_sync_started == 1` and no peer in `Started` state will never call `start_sync_headers` successfully again. `find_blocks_to_fetch` is also gated on `is_verifying_unverified_blocks_on_startup` and IBD state: [5](#0-4) 

The node cannot advance its chain tip, cannot leave IBD, and cannot process new blocks from the network. This is a **complete, permanent DoS of the IBD sync subsystem** until the node is restarted.

---

### Likelihood Explanation

The race window is small (two sequential DashMap operations), but:

- The attacker fully controls the timing of their own disconnect.
- The attack can be retried cheaply: connect, wait for the `SEND_GET_HEADERS_TOKEN` timer to fire (a fixed periodic interval), disconnect at the right moment.
- No special privileges are required — any unprivileged peer can connect and disconnect freely.
- A single successful race is sufficient to permanently freeze IBD sync.

---

### Recommendation

Increment `n_sync_started` only **after** confirming the peer state was successfully transitioned to `Started`. Restructure the critical section so both operations are performed under the same DashMap write lock, or roll back the counter increment if `get_mut` returns `None`:

```rust
// Acquire peer state first
if let Some(mut peer_state) = self.peers().state.get_mut(&peer) {
    // Only then increment the counter
    if self.shared().state().n_sync_started()
        .fetch_update(Ordering::AcqRel, Ordering::Acquire, |x| {
            if ibd && x != 0 { None } else { Some(x + 1) }
        })
        .is_err()
    {
        break;
    }
    peer_state.start_sync(HeadersSyncController::from_header(&tip));
} else {
    // Peer already gone — do not increment counter
    continue;
}
```

This eliminates the window where the counter is incremented but the peer state is never set to `Started`.

---

### Proof of Concept

1. Victim node starts in IBD with no peers.
2. Attacker peer connects; `sync_connected` sets its state to `SyncProtocolConnected`.
3. `SEND_GET_HEADERS_TOKEN` fires; `start_sync_headers` collects the attacker peer into `peers`.
4. `n_sync_started.fetch_update` succeeds, counter becomes `1`.
5. Attacker peer disconnects; `Peers::disconnected` removes the peer from state. Because `sync_started()` is `false` (step 4 completed but step 5 of `start_sync` has not yet run), `n_sync_started` is **not** decremented.
6. `self.peers().state.get_mut(&peer)` returns `None`; `start_sync` is never called.
7. `n_sync_started` is permanently `1`. Every subsequent call to `start_sync_headers` in IBD mode hits `if ibd && x != 0 { None }` and breaks immediately.
8. The victim node is frozen in IBD indefinitely. [6](#0-5) [7](#0-6)
