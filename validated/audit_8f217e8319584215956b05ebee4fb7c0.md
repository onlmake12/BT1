### Title
Provider Can Request Randomness From Itself, Collapsing 2-Party Security Guarantee — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` contains no check preventing `msg.sender == provider`. Because a registered provider knows its entire pre-committed hash chain, it can act as both parties in the commit-reveal protocol, compute the final random number before submitting the request, and choose a user contribution that produces any desired output.

---

### Finding Description

The Entropy protocol's security rests on a 2-party commit-reveal scheme. The contract itself documents this guarantee:

> "This protocol has the same security properties as the 2-party randomness protocol above: as long as either the provider or user is honest, the number r is random." [1](#0-0) 

The `requestHelper` function, which is the shared entry point for `request()`, `requestWithCallback()`, and all `requestV2()` variants, performs the following checks:

- Provider exists (`sequenceNumber != 0`)
- Sequence number is within bounds
- Sufficient fee is paid

It then unconditionally sets `req.requester = msg.sender`. [2](#0-1) 

There is **no check** of the form `require(msg.sender != provider)`. Provider registration is permissionless: [3](#0-2) 

A provider who calls `register()` stores their own hash chain locally. When they subsequently call `request(provider, hash(x_U), false)` with `provider == msg.sender`, they already know `x_i` (their contribution for the assigned sequence number). They can iterate over candidate `x_U` values off-chain, compute `r = hash(x_i, x_U)` for each, and submit only the `x_U` that produces the most favorable `r`. They then call `reveal()` (which enforces `req.requester == msg.sender`, a check they trivially satisfy) to finalize the result. [4](#0-3) 

For the callback path (`requestV2` / `requestWithCallback`), `revealWithCallback` is callable by anyone, so the provider can also fulfill their own callback requests. [5](#0-4) 

---

### Impact Explanation

The 2-party security guarantee collapses entirely when both parties are the same address. The provider can deterministically choose the output of any randomness request they make to themselves. Any on-chain application (e.g., a DeFi lottery, NFT mint, or game) that both operates its own Entropy provider and consumes randomness from that same provider is fully exploitable: the operator can guarantee any desired outcome for every request.

---

### Likelihood Explanation

Provider registration is permissionless. A DeFi protocol or individual actor can register as a provider (paying only the fee per request) and immediately exploit this for their own consumer contracts. The scenario of a protocol running its own provider for cost control or latency reasons is realistic and documented in the Entropy design. No privileged access, leaked key, or external oracle manipulation is required — only the ability to call `register()` followed by `request()`.

---

### Recommendation

Add a self-request exclusion check inside `requestHelper`:

```solidity
if (msg.sender == provider) revert EntropyErrors.SelfRequest();
```

This mirrors the Clearpool pattern of excluding the pool manager from the auction. Alternatively, document explicitly that providers must never use their own provider address as a consumer, and enforce this at the SDK/integration layer — though an on-chain check is strongly preferred since off-chain documentation cannot be enforced. [6](#0-5)

### Proof of Concept

```solidity
// Attacker registers as a provider
entropy.register{value: ...}(feeInWei, commitment, metadata, chainLength, uri);

// Attacker knows x_i for the next sequence number.
// Iterate off-chain to find x_U such that hash(x_i, x_U) == desiredOutcome.
bytes32 x_U = findFavorableContribution(x_i, desiredOutcome);

// Attacker requests from themselves
uint64 seq = entropy.request{value: fee}(
    attackerAddress,          // provider == msg.sender
    keccak256(abi.encode(x_U)),
    false
);

// Attacker reveals, obtaining the pre-chosen random number
bytes32 r = entropy.reveal(attackerAddress, seq, x_U, x_i);
// r == desiredOutcome
```

The `requestHelper` at line 260 sets `req.requester = msg.sender` with no guard against `msg.sender == provider`, and `reveal` at line 513 only checks `req.requester == msg.sender` — both conditions are trivially satisfied by the attacker. [7](#0-6) [8](#0-7)

    function requestHelper(
        address provider,
        bytes32 userCommitment,
        bool useBlockhash,
        bool isRequestWithCallback,
        uint32 callbackGasLimit
    ) internal returns (EntropyStructsV2.Request storage req) {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];
        if (_state.providers[provider].sequenceNumber == 0)
            revert EntropyErrors.NoSuchProvider();

        // Assign a sequence number to the request
        uint64 assignedSequenceNumber = providerInfo.sequenceNumber;
        if (assignedSequenceNumber >= providerInfo.endSequenceNumber)
            revert EntropyErrors.OutOfRandomness();
        providerInfo.sequenceNumber += 1;

        // Check that fees were paid and increment the pyth / provider balances.
        uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
        if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
        uint128 providerFee = getProviderFee(provider, callbackGasLimit);
        providerInfo.accruedFeesInWei += providerFee;
        _state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) -
            providerFee);

        // Store the user's commitment so that we can fulfill the request later.
        // Warning: this code needs to overwrite *every* field in the request, because the returned request can be
        // filled with arbitrary data.
        req = allocRequest(provider, assignedSequenceNumber);
        req.provider = provider;
        req.sequenceNumber = assignedSequenceNumber;
        req.numHashes = SafeCast.toUint32(
            assignedSequenceNumber -
                providerInfo.currentCommitmentSequenceNumber
        );
        if (
            providerInfo.maxNumHashes != 0 &&
            req.numHashes > providerInfo.maxNumHashes
        ) {
            revert EntropyErrors.LastRevealedTooOld();
        }
        req.commitment = keccak256(
            bytes.concat(userCommitment, providerInfo.currentCommitment)
        );
        req.requester = msg.sender;

    function reveal(
        address provider,
        uint64 sequenceNumber,
        bytes32 userContribution,
        bytes32 providerContribution
    ) public override returns (bytes32 randomNumber) {
        EntropyStructsV2.Request storage req = findActiveRequest(
            provider,
            sequenceNumber
        );

        if (
            req.callbackStatus != EntropyStatusConstants.CALLBACK_NOT_NECESSARY
        ) {
            revert EntropyErrors.InvalidRevealCall();
        }

        if (req.requester != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }
        bytes32 blockHash;
        (randomNumber, blockHash) = revealHelper(
            req,
            userContribution,
            providerContribution
        );
        emit Revealed(
            EntropyStructConverter.toV1Request(req),
            userContribution,
            providerContribution,
            blockHash,
            randomNumber
        );
        clearRequest(provider, sequenceNumber);
    }
