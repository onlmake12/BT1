### Title
`Echo.executeCallback` Validates Only 8-Byte Price ID Prefix, Allowing Provider to Substitute Arbitrary Price Feeds - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` stores only the first 8 bytes of each requested price ID at request time, then validates only those 8 bytes at callback execution time. Any price ID sharing the same 8-byte prefix as the originally requested one passes the check, allowing a provider (or any caller after the exclusivity period) to fulfill a request with data from a completely different price feed.

### Finding Description

In `requestPriceUpdatesWithCallback`, the contract deliberately truncates each 32-byte price ID to its first 8 bytes for storage:

```solidity
// Copy only the first 8 bytes of each price ID to storage
for (uint8 i = 0; i < priceIds.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly {
        prefix := priceId
    }
    req.priceIdPrefixes[i] = prefix;
}
``` [1](#0-0) 

In `executeCallback`, the validation only compares these 8-byte prefixes against the caller-supplied price IDs:

```solidity
for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
    bytes32 priceId = priceIds[i];
    bytes8 prefix;
    assembly {
        prefix := priceId
    }
    if (prefix != req.priceIdPrefixes[i]) {
        revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
    }
}
``` [2](#0-1) 

The `Request` struct confirms only `bytes8[] priceIdPrefixes` is stored — the full 32-byte price IDs are never persisted: [3](#0-2) 

After the prefix check passes, the caller-supplied `priceIds` (which may differ in bytes 9–32) are forwarded directly to `parsePriceFeedUpdates`: [4](#0-3) 

The resulting `priceFeeds` are then delivered to the consumer's `_echoCallback` without any further ID verification: [5](#0-4) 

### Impact Explanation

A provider can call `executeCallback` with a price ID `0xAABBCCDDEEFF0011_FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF` when the user originally requested `0xAABBCCDDEEFF0011_REAL_FEED_ID_BYTES`. The 8-byte prefix check passes, `parsePriceFeedUpdates` is called with the attacker-chosen full ID, and the consumer's `echoCallback` receives price data for a completely different asset. Consumer contracts making financial decisions (e.g., liquidations, option settlements, collateral checks) based on the delivered price would act on incorrect data. This constitutes **incorrect price data delivery** to on-chain consumers. [6](#0-5) 

### Likelihood Explanation

The attack is reachable by any registered provider at any time during the exclusivity period, and by **any unprivileged caller** after the exclusivity period elapses. No special privileges, leaked keys, or governance majority are required. The only prerequisite is constructing a valid Pyth `updateData` payload for a price feed whose ID shares the first 8 bytes with the target feed — a feasible offline computation. Pyth price IDs are public and enumerable, making prefix collisions discoverable. [7](#0-6) 

### Recommendation

Store and compare the full 32-byte price IDs. The gas-saving motivation for truncation is understandable, but the security tradeoff is unacceptable. Replace `bytes8[] priceIdPrefixes` with `bytes32[] priceIds` in the `Request` struct and compare full IDs in `executeCallback`. If storage cost is a concern, use a `keccak256` hash of the price ID array instead of individual prefixes. [8](#0-7) 

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback` with `priceIds = [0xAABBCCDDEEFF001122334455667788990011223344556677889900112233445566]`. The contract stores prefix `0xAABBCCDDEEFF0011`.

2. A provider (or any caller after exclusivity) constructs a substitute price ID: `0xAABBCCDDEEFF0011_FFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFF` — a different feed that shares the same 8-byte prefix.

3. The provider prepares valid Pyth `updateData` containing a price update for the substitute feed ID.

4. The provider calls `executeCallback(provider, sequenceNumber, updateData, [substituteId])`.

5. The prefix check at line 137 passes: `bytes8(substituteId) == bytes8(originalId)` → `0xAABBCCDDEEFF0011 == 0xAABBCCDDEEFF0011` ✓

6. `parsePriceFeedUpdates` is called with `substituteId` and returns price data for the substitute feed.

7. The consumer's `echoCallback` receives price data for the wrong asset and acts on it. [9](#0-8)
