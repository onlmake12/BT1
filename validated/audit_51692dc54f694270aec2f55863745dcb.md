### Title
Unauthenticated `from` Field in HolePunching P2P Messages Enables `pending_delivered` Cache Poisoning and Forced Outbound TCP Connections — (`network/src/protocols/hole_punching/component/connection_request.rs`, `connection_sync.rs`)

---

### Summary

The HolePunching protocol in `ckb-network` trusts the `from` field carried inside peer-controlled message payloads (`ConnectionRequest`, `ConnectionSync`) without verifying it against the actual session's authenticated peer identity. This is the direct structural analog of the martian-wallet channel-key override: in both cases an attacker-supplied field is placed at the same trust level as a node-authoritative field, allowing the attacker to substitute an arbitrary identity. In CKB the consequence is that any connected peer can poison the `pending_delivered` cache under an arbitrary peer ID and then trigger the victim node to initiate outbound TCP connections to attacker-controlled addresses.

---

### Finding Description

**Vulnerability class:** Unauthenticated identity field used as a trusted routing/cache key — direct analog of the "attacker-controlled spread overrides fixed channel key" pattern.

**Root cause — `ConnectionRequest` handler:**

In `ConnectionRequestProcess::execute()`, the `from` field is deserialized directly from the wire message:

```rust
// connection_request.rs line 36
let from = PeerId::from_bytes(value.from().raw_data().to_vec())...
``` [1](#0-0) 

This peer-controlled `from` value is then stored as the key in `pending_delivered`:

```rust
// connection_request.rs line 234-237
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
``` [2](#0-1) 

There is no check that `from_peer_id` equals the peer ID of the actual authenticated session (`context.session`). The `peer` field stored in `ConnectionRequestProcess` is only the `PeerIndex` (session handle), never compared to `content.from`.

**Root cause — `ConnectionSync` handler:**

`ConnectionSyncProcess::execute()` reads `content.from` from the wire message and uses it to look up `pending_delivered`:

```rust
// connection_sync.rs lines 111-115
let listens_info = self
    .protocol
    .pending_delivered
    .get(&content.from)
    .map(|info| info.0.clone());
``` [3](#0-2) 

If a matching entry is found, the node immediately spawns an async task that calls `try_nat_traversal` for every address stored in that entry:

```rust
// connection_sync.rs lines 119-162
let tasks = listens.into_iter()
    .map(|listen_addr| Box::pin(try_nat_traversal(self.bind_addr, listen_addr)))
    .collect::<Vec<_>>();
// ...
runtime::spawn(async move { ... select_ok(tasks).await ... });
``` [4](#0-3) 

`try_nat_traversal` opens a real TCP socket and repeatedly attempts to connect to the supplied address for up to 30 seconds:

```rust
// component/mod.rs lines 68-110
while start_time.elapsed() < timeout_duration {
    let socket = create_socket(bind_addr, net_addr)?;
    match runtime::timeout(..., socket.connect(net_addr)).await { ... }
    runtime::delay_for(actual_interval).await;
}
``` [5](#0-4) 

**Secondary root cause — `ConnectionRequestDelivered` handler:**

`inflight_requests.remove(&content.to)` uses the peer-controlled `to` field:

```rust
// connection_request_delivered.rs line 160
let request_start = self.protocol.inflight_requests.remove(&content.to);
``` [6](#0-5) 

An attacker who knows the target node's own peer ID can set `from = target_node_peer_id` and `to = victim_peer_id` to remove the victim's in-flight hole-punching request, silently canceling it.

**`forward_rate_limiter` bypass:**

All three handlers key the `forward_rate_limiter` on `(content.from, content.to, item_id)`:

```rust
// connection_request.rs lines 132-143
self.protocol.forward_rate_limiter
    .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
``` [7](#0-6) 

Because `from` is attacker-controlled, an attacker can rotate fake `from` values to generate a fresh rate-limit bucket for every message, bypassing the per-(from,to) throttle entirely.

---

### Impact Explanation

**Forced outbound TCP connections (SSRF-like):** By sending a `ConnectionRequest` with `from = fake_id` and `listen_addrs = [attacker_ip:port]`, then following up with a `ConnectionSync` with `from = fake_id`, the attacker causes the victim CKB node to spawn a 30-second TCP connection loop to any IPv4/IPv6 address and port of the attacker's choice. This can be used to:
- Probe internal network services from behind the CKB node's firewall
- Exhaust the node's socket/thread resources by triggering many concurrent 30-second traversal tasks
- Cause the node to send SYN packets to third-party hosts (TCP amplification / abuse vector)

**`pending_delivered` cache poisoning:** The attacker can pre-populate `pending_delivered[victim_peer_id]` with attacker-controlled addresses. The `HOLE_PUNCHING_INTERVAL` (2 minutes) deduplication check then blocks the legitimate `ConnectionRequest` from victim for 2 minutes, silently denying hole-punching service to that peer. [8](#0-7) 

**`inflight_requests` removal:** An attacker can cancel any peer's in-flight hole-punching request by sending a crafted `ConnectionRequestDelivered`.

---

### Likelihood Explanation

Any unprivileged peer that establishes a standard Tentacle P2P connection to a CKB node can send HolePunching protocol messages. No special privileges, keys, or majority hash power are required. The HolePunching protocol is enabled by default in the network stack. The attack requires only two sequential messages (`ConnectionRequest` + `ConnectionSync`) from a single connected session, making it trivially automatable. [9](#0-8) 

---

### Recommendation

Bind the `from` field to the authenticated session identity. In `ConnectionRequestProcess::execute()`, replace the peer-supplied `content.from` with the actual peer ID resolved from the session:

```rust
// Resolve the actual peer ID from the authenticated session
let actual_from = self.protocol.network_state
    .peer_registry
    .read()
    .get_peer(self.peer)
    .and_then(|p| p.connected_addr.peer_id().cloned());

// Reject if the message's `from` does not match the session's peer ID
if Some(&content.from) != actual_from.as_ref() {
    return StatusCode::InvalidFromPeerId
        .with_context("from field does not match session peer id");
}
```

Apply the same check in `ConnectionSyncProcess` and `ConnectionRequestDeliveredProcess`. The `forward_rate_limiter` key should use the verified session peer ID, not the message-supplied `from`.

### Proof of Concept

```
# Attacker connects to target CKB node (standard P2P)
# Step 1: Send ConnectionRequest with spoofed `from` and attacker-controlled listen_addrs
ConnectionRequest {
    from:         <arbitrary_fake_peer_id>,
    to:           <target_node_own_peer_id>,
    max_hops:     6,
    route:        [],
    listen_addrs: [/ip4/ATTACKER_IP/tcp/ATTACKER_PORT]
}
# → target stores pending_delivered[fake_peer_id] = ([ATTACKER_IP:PORT], now)

# Step 2: Send ConnectionSync with same fake `from`, empty route
ConnectionSync {
    from:  <arbitrary_fake_peer_id>,
    to:    <target_node_own_peer_id>,
    route: []
}
# → target looks up pending_delivered[fake_peer_id], finds ATTACKER_IP:PORT
# → target spawns try_nat_traversal task: TCP SYN loop to ATTACKER_IP:PORT for 30 seconds
# → repeat with different fake_peer_ids to spawn unlimited concurrent traversal tasks
``` [10](#0-9) [11](#0-10)

            StatusCode::ReachedMaxHops.into()

        let content = match SyncContent::try_from(&self.message) {
            Ok(content) => content,
            Err(status) => return status,
        };

        if content.route.len() > MAX_HOPS as usize {
            return StatusCode::InvalidRoute.with_context("the route length is too long");
        }
        if self
            .protocol
            .forward_rate_limiter
            .check_key(&(content.from.clone(), content.to.clone(), self.msg_item_id))
            .is_err()
        {
            debug!(
                "from: {}, to {}, item_name: {}, rate limit is reached",
                content.from, content.to, "ConnectionSync",
            );
            return StatusCode::TooManyRequests.with_context("ConnectionSync");
        }

        match content.route.last() {
            Some(next_peer_id) => self.forward_sync(next_peer_id).await,
            None => {
                let self_peer_id = self.protocol.network_state.local_peer_id();
                if self_peer_id != &content.to {
                    // forward the message to the `to` peer
                    self.forward_sync(&content.to).await
                } else {
                    // Current node should be the `to` target.
                    if let Some(metrics) = ckb_metrics::handle() {
                        metrics.ckb_hole_punching_passive_count.inc();
                    }

                    let listens_info = self
                        .protocol
                        .pending_delivered
                        .get(&content.from)
                        .map(|info| info.0.clone());

                    match listens_info {
                        Some(listens) => {
                            let tasks = listens
                                .into_iter()
                                .map(|listen_addr| {
                                    Box::pin(try_nat_traversal(self.bind_addr, listen_addr))
                                })
                                .collect::<Vec<_>>();

                            if tasks.is_empty() {
                                return StatusCode::Ignore.with_context("no valid listen address");
                            }

                            debug!(
                                "current peer is the target peer {}, start NAT traversal",
                                content.to
                            );

                            match self
                                .protocol
                                .network_state
                                .config
                                .listen_addresses
                                .first()
                                .cloned()

                                Some(listen_addr) => {

                                    runtime::spawn(async move {
                                        if let Ok(((stream, addr), _)) = select_ok(tasks).await {
                                            debug!("NAT traversal success, addr: {:?}", addr);
                                            if let Some(metrics) = ckb_metrics::handle() {
                                                metrics
                                                    .ckb_hole_punching_passive_success_count
                                                    .inc();
                                            }

                                            let _ignore = control

                                                    stream,
                                                    addr,
                                                    RawSessionInfo::inbound(listen_addr),
                                                )
                                                .await;
                                        }
                                    });
                                    Status::ok()
                                }
                                None => {
                                    StatusCode::Ignore.with_context("no listen address configured")
                                }
                            }
                        }
                        None => StatusCode::Ignore
                            .with_context("the from peer id is not in the pending list"),
                    }
