### Title
Unbounded `GetBlocks` Replay Allows Unauthenticated Peer to Exhaust Node Resources — (`sync/src/synchronizer/get_blocks_process.rs`)

### Summary
`GetBlocksProcess::execute()` contains a per-invocation `dedup` HashSet that only prevents duplicates *within* a single message. There is no cross-message rate limiting, no per-peer served-block tracking, and no ban triggered for repeated identical requests. An unprivileged peer can flood the node with repeated `GetBlocks` messages each containing 32 distinct valid block hashes, causing the node to spawn up to 32 async `SendBlock` tasks per message indefinitely.

### Finding Description
In `sync/src/synchronizer/get_blocks_process.rs`, `execute()` allocates a fresh `HashSet` on every call: [1](#0-0) 

This dedup only guards against duplicates *within* one message. Sending N messages each with 32 distinct hashes bypasses it entirely. The `Synchronizer::try_process` dispatches `GetBlocks` with no rate-limiting wrapper: [2](#0-1) 

Compare this to the `Relayer`, which has an explicit `RateLimiter<(PeerIndex, u32)>` keyed by peer and message type: [3](#0-2) 

No equivalent guard exists in `Synchronizer`. Each valid block hash causes an async task to be spawned: [4](#0-3) 

`INIT_BLOCKS_IN_TRANSIT_PER_PEER` is 32 and `MAX_HEADERS_LEN` is 2,000: [5](#0-4) 

A message with up to 2,000 hashes (all unique) passes the length check and the `take(32)` cap limits processing to 32 per message — but with no rate limit, N messages yield N×32 spawned tasks.

### Impact Explanation
Each spawned task serializes a full block from storage and sends it over the network. An attacker with a single TCP connection can:
- Exhaust the async task pool (unbounded `spawn` calls)
- Saturate outbound bandwidth with full block data (amplification: ~1 KB request → up to MB-scale responses per message)
- Consume CPU for repeated block serialization and DB reads

This constitutes a denial-of-service against the node's sync protocol with no cost to the attacker beyond maintaining a connection and sending small messages.

### Likelihood Explanation
The attack requires only a valid P2P connection and knowledge of 32 valid block hashes (trivially obtained from any public block explorer or by first syncing headers). No PoW, no key, no privileged role is needed. The `Relayer` protocol already demonstrates the developers are aware rate limiting is necessary — its absence in `Synchronizer` for `GetBlocks` is a concrete gap.

### Recommendation
Add a per-peer rate limiter to `Synchronizer` for `GetBlocks` messages, mirroring the `RateLimiter<(PeerIndex, u32)>` pattern already used in `Relayer`. Additionally, consider maintaining a per-peer LRU cache of recently served block hashes to reject re-requests within a time window, and/or ban peers that exceed a configurable request threshold.

### Proof of Concept
1. Connect a peer to the node's sync port.
2. Obtain 32 valid block hashes from the chain (e.g., blocks 1–32).
3. In a tight loop, send N `SyncMessage::GetBlocks` messages each containing those same 32 hashes (all distinct within each message, so `StatusCode::RequestDuplicate` is never triggered).
4. Observe: N×32 `SendBlock` async tasks are spawned; the peer is never banned; node CPU and outbound bandwidth spike proportionally to N.

State-transition unit test sketch:
```rust

for _ in 0..N {
    let process = GetBlocksProcess::new(msg.as_reader(), &synchronizer, peer, &nc);
    assert_eq!(process.execute(), Status::ok()); // never banned
}
// Assert nc.sent_messages.len() == N * 32
```
