The gate passes `self.parent.epoch().number()` — the **parent block's epoch number** — to `rfc0044_active`. [1](#0-0) 

This creates a concrete off-by-one at the activation boundary:

| Block | Parent epoch | `rfc0044_active(parent_epoch)` | `DaoScriptSizeVerifier` |
|---|---|---|---|
| First block of epoch `RFC0044_ACTIVE_EPOCH` | `RFC0044_ACTIVE_EPOCH - 1` | **false** | **SKIPPED** |
| Second block of epoch `RFC0044_ACTIVE_EPOCH` | `RFC0044_ACTIVE_EPOCH` | true | enforced |

Every full node applies the same check identically, so a block mined at this boundary with a DAO withdrawal containing a mismatched lock size would be accepted by the entire network — it is not a local/single-node issue.

### Title
One-block DAO lock-size invariant bypass at RFC0044 activation epoch boundary — (`verification/contextual/src/contextual_block_verifier.rs`)

### Summary
`BlockTxsVerifier::verify` gates `DaoScriptSizeVerifier` on `rfc0044_active(self.parent.epoch().number())`. For the very first block of the activation epoch, the parent's epoch number is `RFC0044_ACTIVE_EPOCH - 1`, so the gate evaluates to `false` and the size check is entirely skipped, even though the block itself is in the activation epoch.

### Finding Description
The guard at line 445 uses the **parent** block's epoch number rather than the current block's epoch number to decide whether to enforce `DaoScriptSizeVerifier`. Because CKB epoch numbers increment at block boundaries, the first block of epoch `RFC0044_ACTIVE_EPOCH` always has a parent whose epoch number is `RFC0044_ACTIVE_EPOCH - 1`. `rfc0044_active(RFC0044_ACTIVE_EPOCH - 1)` returns `false`, so `DaoScriptSizeVerifier::verify()` is never called for any transaction in that block. All subsequent blocks in the same epoch have a parent with epoch number `RFC0044_ACTIVE_EPOCH`, so the check is enforced from the second block onward. [1](#0-0) 

### Impact Explanation
A miner who wins the right to produce the first block of the activation epoch can include DAO withdrawal transactions whose lock scripts violate the size invariant that RFC0044 is meant to enforce. Because every honest node applies the same (buggy) gate, the block is accepted network-wide and becomes part of the canonical chain. The DAO lock-size invariant is permanently violated for those outputs, potentially enabling downstream script-execution bypasses or fund-locking issues that the RFC0044 check was designed to prevent.

### Likelihood Explanation
Any miner — including one with a single percent of hashpower — has a non-negligible probability of mining the first block of the activation epoch. The activation epoch is a known, publicly scheduled event, so a motivated attacker can prepare the malformed DAO withdrawal transaction in advance and submit it the moment they win that block. No privileged access, leaked keys, or majority hashpower is required.

### Recommendation
Replace `self.parent.epoch().number()` with the current block's epoch number:

```rust
// current block's epoch is encoded in self.header
if self.context.consensus.rfc0044_active(self.header.epoch().number()) {
    DaoScriptSizeVerifier::new(...).verify()?;
}
```

This ensures the check is enforced for every block whose own epoch is ≥ `RFC0044_ACTIVE_EPOCH`, including the very first one.

### Proof of Concept
Differential test:
1. Build a DAO withdrawal transaction with a lock script whose size violates the RFC0044 invariant.
2. Construct **Block A** whose parent is the last block of epoch `RFC0044_ACTIVE_EPOCH - 1` (parent epoch number = `RFC0044_ACTIVE_EPOCH - 1`). Include the malformed DAO withdrawal. Assert `BlockTxsVerifier::verify` returns `Ok` — the `DaoScriptSizeVerifier` branch is never entered.
3. Construct **Block B** whose parent is the first block of epoch `RFC0044_ACTIVE_EPOCH` (parent epoch number = `RFC0044_ACTIVE_EPOCH`). Include the same malformed DAO withdrawal. Assert `BlockTxsVerifier::verify` returns `Err` — the `DaoScriptSizeVerifier` branch fires and rejects it.

The asymmetry between Block A (accepted) and Block B (rejected) for identical DAO withdrawal content confirms the one-block enforcement gap.
