### Title
Unbound `write_authority` in `init_encoded_vaa` Enables Rent Theft via Front-Running — (`target_chains/solana/programs/core-bridge/src/processor/parse_and_verify_vaa/init_encoded_vaa.rs`)

---

### Summary

`init_encoded_vaa` accepts any signer as `write_authority` without verifying any relationship to the account that funded the `encoded_vaa` account. When `create_account` and `init_encoded_vaa` are executed in separate transactions — a pattern explicitly demonstrated in the protocol's own test suite — an attacker who observes the freshly-created, all-zero account on-chain can front-run the relayer's `init_encoded_vaa` call, install their own key as `write_authority`, and immediately drain the rent lamports via `close_encoded_vaa`.

---

### Finding Description

**Root cause — `init_encoded_vaa`:**

The `InitEncodedVaa` account struct validates only three things: [1](#0-0) 

1. `write_authority` is a `Signer` (any signer, no relationship to the account creator).
2. `encoded_vaa` is owned by `crate::ID`.
3. The header bytes of `encoded_vaa` are all zeros.

There is no constraint binding `write_authority` to the keypair that paid for the account or to the `encoded_vaa` keypair itself. The function then unconditionally writes the caller-supplied `write_authority` into the account header: [2](#0-1) 

**Downstream — `close_encoded_vaa`:**

`close_encoded_vaa` transfers all lamports to whoever is stored as `write_authority` in the header, with no check that this matches the original payer: [3](#0-2) 

**The vulnerable split-transaction pattern is explicitly demonstrated in the test suite:**

TX1 (confirmed): `system_instruction::create_account` — account is now on-chain, owned by `crate::ID`, all-zero bytes. [4](#0-3) 

TX2 (pending): `init_encoded_vaa` — the window where an attacker can front-run. [5](#0-4) 

Between TX1 confirmation and TX2 landing, the preconditions are fully met: account owned by `crate::ID`, data length `> VAA_START`, header all zeros.

---

### Impact Explanation

An attacker who front-runs `init_encoded_vaa` with `write_authority = attacker_key` becomes the sole authority over the account. They can immediately call `close_encoded_vaa`, which transfers all lamports (the relayer's rent deposit) to `attacker_key`. The relayer's `init_encoded_vaa` transaction will then fail with `AccountNotZeroed` (header is no longer zero), and the relayer loses their rent with no recourse.

The `close_encoded_vaa` comment even acknowledges the intent: *"This authority is the same signer that originally created the VAA accounts, so he is the one that will receive the lamports back"* — but this invariant is never enforced in code. [6](#0-5) 

---

### Likelihood Explanation

**Mitigating factor:** The production CLI bundles `create_account` + `init_encoded_vaa` + `write_encoded_vaa` in a single atomic transaction, eliminating the window: [7](#0-6) 

The JS SDK similarly packs create + init into the same TX1: [8](#0-7) 

**Aggravating factors:**
- The split-transaction pattern is a first-class, tested usage of the protocol (TX1/TX2 in the test suite).
- Any custom relayer, retry handler, or integrator that follows the test pattern or splits due to transaction size constraints is directly vulnerable.
- The JS SDK's `VAA_SPLIT_INDEX = 721` was chosen specifically to keep create+init+write in one tx; VAAs larger than this split point could force a different packing strategy.
- The attack is zero-cost, requires no privileged access, and is fully local-testable. [9](#0-8) 

---

### Recommendation

Add a constraint in `InitEncodedVaa` that binds `write_authority` to the `encoded_vaa` keypair. The simplest fix is to require `encoded_vaa` to also be a `Signer` (i.e., the keypair must co-sign `init_encoded_vaa`, proving the caller controls it). Since the relayer already holds the `encoded_vaa` keypair to sign `create_account`, this adds no operational burden:

```rust
#[derive(Accounts)]
pub struct InitEncodedVaa<'info> {
    write_authority: Signer<'info>,
    #[account(
        mut,
        owner = crate::ID,
        signer  // <-- require encoded_vaa keypair to co-sign
    )]
    encoded_vaa: UncheckedAccount<'info>,
}
```

This ensures only the holder of the `encoded_vaa` private key (i.e., the account creator) can call `init_encoded_vaa`, closing the front-running window regardless of transaction boundaries.

---

### Proof of Concept

```
Block N   : Relayer broadcasts TX1 = [create_account(payer=relayer, new_account=vaa_kp, owner=crate::ID)]
            → confirmed; vaa_kp.pubkey() is now on-chain, all-zero, owned by crate::ID

Block N+1 : Attacker observes vaa_kp.pubkey() from TX1 logs.
            Attacker broadcasts TX_attack = [init_encoded_vaa(write_authority=attacker, encoded_vaa=vaa_kp.pubkey())]
            Relayer broadcasts TX2         = [init_encoded_vaa(write_authority=relayer,  encoded_vaa=vaa_kp.pubkey())]

            If TX_attack lands first:
              → header.write_authority = attacker
              → Relayer's TX2 fails: AccountNotZeroed
              → Attacker calls close_encoded_vaa(write_authority=attacker, encoded_vaa=vaa_kp.pubkey())
              → All rent lamports transferred to attacker
```

The test infrastructure in `test_post_updates_with_wormhole.rs` already provides the exact scaffolding to write this as an integration test: replace TX2's `write_authority` with an attacker keypair, assert it succeeds, then assert `close_encoded_vaa` drains the lamports to the attacker.
