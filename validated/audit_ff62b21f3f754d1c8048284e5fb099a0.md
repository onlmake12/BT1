### Title
`PythAggregatorV3.latestRoundData()` Returns Stale Price Without Staleness Check - (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3`, the official Pyth Chainlink-compatibility adapter published in `@pythnetwork/pyth-sdk-solidity`, calls `pyth.getPriceUnsafe(priceId)` in every price-reading function — including `latestRoundData()`, `latestAnswer()`, `getRoundData()`, and `latestTimestamp()`. `getPriceUnsafe` explicitly makes no staleness check and can return a price from arbitrarily far in the past. Any downstream protocol that uses `PythAggregatorV3` as a drop-in Chainlink replacement (as recommended by Pyth's own migration guide) will silently consume stale prices with no revert or warning.

---

### Finding Description

`PythAggregatorV3` is Pyth's official adapter contract that implements Chainlink's `AggregatorV3Interface` on top of Pyth price feeds. It is part of the published `@pythnetwork/pyth-sdk-solidity` npm package and is the recommended migration path for Chainlink integrators per Pyth's documentation.

Every price-reading function in the contract delegates to `pyth.getPriceUnsafe(priceId)`:

```solidity
// Line 110 — latestRoundData()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// Line 54 — latestAnswer()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// Line 89 — getRoundData()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// Line 41 — decimals()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);

// Line 59 — latestTimestamp()
PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
``` [1](#0-0) 

`getPriceUnsafe` is documented as returning a price "from arbitrarily far in the past" with no recency guarantee: [2](#0-1) 

The safe alternative, `getPriceNoOlderThan(id, age)`, enforces a staleness bound and reverts with `StalePrice` if the price is too old: [3](#0-2) 

`PythAggregatorV3` never calls `getPriceNoOlderThan`. There is no configurable `maxAge` parameter, no staleness revert, and no on-chain signal to callers that the returned price may be hours or days old.

---

### Impact Explanation

Any protocol that integrates `PythAggregatorV3` as its price oracle — lending protocols, AMMs, perpetuals, insurance funds — will receive stale prices silently. This is the exact vulnerability class described in the reference report (Hubble Exchange M-2): stale oracle prices used for collateral valuation, liquidation thresholds, and trade execution. Consequences include:

- Borrowers exploiting a stale low price to borrow excess collateral
- Liquidations blocked because a stale high price makes an insolvent position appear solvent
- Traders arbitraging a stale price against the true market price

The impact is amplified because `PythAggregatorV3` is the **officially recommended** migration path from Chainlink, meaning many protocols will deploy it trusting it is safe.

---

### Likelihood Explanation

Pyth is a pull oracle. If a protocol relies on scheduled push updates (as the migration guide acknowledges is common for Chainlink-migrating apps) and those updates are delayed — due to network congestion, keeper failure, or a deliberate griefing attack that drains the keeper — the on-chain price becomes stale. An attacker monitoring the mempool can detect when the price is stale and exploit it. The entry path requires no privileged access: any unprivileged transaction sender can call the downstream protocol's functions (borrow, trade, liquidate) while the stale price is active. [4](#0-3) 

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`. Add a configurable `maxAge` parameter to the constructor (or as a settable state variable) so deployers can specify the acceptable staleness window appropriate for their use case:

```solidity
uint256 public maxAge;

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // ...
}
```

This mirrors the fix recommended in the reference report: enforce a tolerance time equal to the maximum acceptable update interval for the price feed.

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to the Pyth contract and a price feed (e.g., ETH/USD).
2. Allow the on-chain Pyth price to go stale (no `updatePriceFeeds` call for > N seconds).
3. Call `PythAggregatorV3.latestRoundData()`.
4. Observe: the call succeeds and returns the stale price with `updatedAt = price.publishTime` (the old timestamp). No revert occurs.
5. A downstream lending protocol calling `latestRoundData()` will use this stale price for collateral valuation, enabling the attacker to borrow against an outdated price. [5](#0-4) [6](#0-5)
