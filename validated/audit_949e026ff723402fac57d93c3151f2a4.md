### Title
Peer Entry in `pending_compact_blocks` Not Removed on `BlockTransactions` Verification Failure — (`sync/src/relayer/block_transactions_process.rs`)

### Summary

When a peer sends a `BlockTransactions` message and the structural verification fails (via the `attempt!` macro), the function returns early without removing the peer's entry from `pending_compact_blocks`. This leaves the peer permanently stuck in the "pending" state for that block hash. The peer cannot re-initiate compact block reconstruction (blocked by `CompactBlockIsAlreadyPending`), and the stale expected-index state cannot be updated, causing block propagation delay and unbounded memory growth in `pending_compact_blocks`.

### Finding Description

In `BlockTransactionsProcess::execute()`, when `BlockTransactionsVerifier::verify()` or `BlockUnclesVerifier::verify()` fails via the `attempt!` macro, the function returns early:

```rust
attempt!(BlockTransactionsVerifier::verify(
    compact_block,
    expected_transaction_indexes,
    &received_transactions,
));
attempt!(BlockUnclesVerifier::verify(
    compact_block,
    expected_uncle_indexes,
    &received_uncles,
));
``` [1](#0-0) 

On this early return, the peer's entry in `pending_compact_blocks` (keyed by `block_hash → peers_map → peer`) is **not removed**. The peer remains in `peers_map` with stale `expected_transaction_indexes` and `expected_uncle_indexes`. [2](#0-1) 

Subsequently, when the same peer sends the same compact block again, `contextual_check` in `CompactBlockProcess::execute()` detects the peer is already in `peers_map` and returns `CompactBlockIsAlreadyPending`:

```rust
if pending_compact_blocks
    .get(&block_hash)
    .map(|(_, peers_map, _)| peers_map.contains_key(&peer))
    .unwrap_or(false)
{
    return StatusCode::CompactBlockIsAlreadyPending.with_context(block_hash);
}
``` [3](#0-2) 

The peer is now locked out from re-initiating compact block reconstruction for that block hash. The only path remaining is to retry `BlockTransactions` with the same stale expected indexes — which will fail again if the peer's data is malformed.

The `pending_compact_blocks` map type is:

```rust
pub(crate) type PendingCompactBlockMap = HashMap<
    Byte32,
    (packed::CompactBlock, HashMap<PeerIndex, (Vec<u32>, Vec<u32>)>, u64),
>;
``` [4](#0-3) 

There is no cleanup of `pending_compact_blocks` entries on peer disconnect in `SyncState::disconnected()`:

```rust
pub fn disconnected(&self, pi: PeerIndex) {
    let removed_inflight_blocks_count = self.write_inflight_blocks().remove_by_peer(pi);
    ...
    self.peers().disconnected(pi);
}
``` [5](#0-4) 

The `disconnected` path removes inflight blocks and peer state, but **never touches `pending_compact_blocks`**, so stale peer entries accumulate.

### Impact Explanation

An unprivileged remote peer can:

1. Send a syntactically valid compact block → node adds `(peer, expected_indexes)` to `pending_compact_blocks` and requests missing transactions.
2. Respond with a structurally invalid `BlockTransactions` message (e.g., wrong transaction count, mismatched short IDs) → `attempt!` fires, function returns early, peer entry is **not removed**.
3. The peer is now permanently stuck in `pending_compact_blocks` for that block hash.
4. Any subsequent compact block relay from the same peer for the same block is rejected with `CompactBlockIsAlreadyPending`.
5. Repeating across many block hashes causes unbounded growth of `pending_compact_blocks`, consuming node memory.
6. Block propagation for the targeted block is delayed until another peer relays it or the sync protocol downloads the full block.

The `compare_with_pending_compact` 2-second grace period partially mitigates the propagation delay but does not address the memory growth or the peer lock-out. [6](#0-5) 

### Likelihood Explanation

Any unprivileged outbound or inbound peer can trigger this by sending a valid compact block followed by a malformed `BlockTransactions` response. No special privileges, keys, or majority hashpower are required. The attacker controls both messages. The condition is reachable on mainnet from any connected peer.

### Recommendation

In `BlockTransactionsProcess::execute()`, on any early return path (both `attempt!` failures and `ReconstructionResult::Error`), remove the peer's entry from `peers_map` in `pending_compact_blocks`. If `peers_map` becomes empty after removal, remove the entire block hash entry from `pending_compact_blocks`. Additionally, `SyncState::disconnected()` should iterate `pending_compact_blocks` and remove all entries for the disconnecting peer.

### Proof of Concept

1. Connect to a CKB node as a peer.
2. Mine or observe a block `B` with at least one non-prefilled transaction.
3. Send a valid `CompactBlock` message for `B` with one short ID → node adds `(self_peer, [tx_index])` to `pending_compact_blocks` and sends `GetBlockTransactions`.
4. Respond with a `BlockTransactions` message where `transactions.len() != expected_transaction_indexes.len()` → `BlockTransactionsVerifier::verify` fails, `attempt!` returns `BlockTransactionsLengthIsUnmatchedWithPendingCompactBlock`, peer entry is NOT removed.
5. Send the same `CompactBlock` again → node returns `CompactBlockIsAlreadyPending` (102).
6. Repeat steps 3–5 for different block hashes to grow `pending_compact_blocks` without bound. [7](#0-6) [8](#0-7)
