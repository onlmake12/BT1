### Title
Relayer Can Supply Insufficient Gas to Silently Fail Callback Sub-Calls While Provider Collects Fees — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In the legacy callback path of `revealWithCallback` (active when `req.gasLimit10k == 0`), the Entropy contract makes a bare, uncapped direct call to the user's `_entropyCallback` with no minimum-gas enforcement. A malicious relayer/provider can submit the transaction with just enough gas to let the top-level callback return successfully while any internal sub-calls inside the callback silently fail. Because provider fees are accrued at request time and the request is cleared before the callback executes, the provider retains the full fee even though the user's callback did not complete its intended logic.

---

### Finding Description

`revealWithCallback` branches on `req.gasLimit10k`:

**New flow** (`req.gasLimit10k != 0`, lines 574–660): uses `excessivelySafeCall` with an explicit gas cap and enforces `(startingGas * 31) / 32 > gasLimit10k * 10_000` to guarantee the callback received the user-specified gas budget before attributing a revert to a genuine failure.

**Legacy flow** (`req.gasLimit10k == 0`, lines 661–702): triggered whenever `providerInfo.defaultGasLimit == 0`. The request is cleared first (line 666), then the callback is invoked with a plain, uncapped call:

```solidity
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(   // no gas limit
        sequenceNumber,
        provider,
        randomNumber
    );
}
```

There is no check that the relayer supplied enough gas for the callback's internal operations. A relayer can:

1. Simulate the transaction to find the minimum gas at which the top-level `_entropyCallback` frame returns without reverting (e.g., because the callback does not check the return value of an internal call).
2. Submit `revealWithCallback` with exactly that gas, causing any sub-call inside the callback to silently fail (OOG at the sub-call level) while the outer frame succeeds.
3. The request is already cleared (line 666) and the provider's `accruedFeesInWei` was incremented at request time (line 237), so the provider retains the fee and the request cannot be retried.

The provider fee accrual at request time is the critical enabler:

```solidity
providerInfo.accruedFeesInWei += providerFee;   // line 237 – happens before any callback
```

The `defaultGasLimit == 0` path that sets `req.gasLimit10k = 0` is explicitly supported and documented:

```solidity
if (providerInfo.defaultGasLimit == 0) {
    // Provider doesn't support the new callback failure state flow
    req.gasLimit10k = 0;
}
```

---

### Impact Explanation

Any user whose callback contract makes an external sub-call without checking the return value (e.g., calling a DEX, minting an NFT, transferring tokens) can have that sub-call silently fail. The Entropy contract marks the request as fulfilled, the provider keeps the fee, and the user has no recourse: the request is cleared and cannot be re-submitted with the same sequence number. The user suffers both the fee loss and the failure of the intended on-chain action.

---

### Likelihood Explanation

- The legacy flow is reachable today for any provider that has not called `setDefaultGasLimit` (i.e., `defaultGasLimit == 0`).
- The provider itself typically runs the relayer (Fortuna keeper), so the same entity that benefits from the fee also controls the gas supplied to the callback.
- Identifying callbacks with unchecked sub-calls is straightforward via on-chain bytecode inspection or event monitoring.
- No special privilege beyond being the relayer (open to anyone per line 541) is required.

---

### Recommendation

In the legacy flow, add a minimum-gas guard analogous to the new flow before the callback is executed:

```solidity
uint256 startingGas = gasleft();
if (len != 0) {
    IEntropyConsumer(callAddress)._entropyCallback(...);
}
// Revert if the outer context did not supply enough gas for the callback
require(
    (startingGas * 31) / 32 > MINIMUM_LEGACY_GAS,
    "InsufficientGas"
);
```

Alternatively, deprecate the `gasLimit10k == 0` path entirely and require all providers to set a non-zero `defaultGasLimit` before accepting new requests.

---

### Proof of Concept

1. Provider registers with `defaultGasLimit = 0` (legacy mode). `req.gasLimit10k` will be `0` for all requests to this provider.
2. User deploys a callback contract that calls an external DEX router without checking the return value:
   ```solidity
   function entropyCallback(uint64, address, bytes32 rand) internal override {
       dexRouter.swap{value: 1 ether}(rand);  // return value not checked
   }
   ```
3. User calls `requestWithCallback`, paying the provider fee. Fee is immediately credited to `providerInfo.accruedFeesInWei` (line 237).
4. Malicious relayer simulates `revealWithCallback` off-chain, finds the minimum gas `G` at which `_entropyCallback` returns without reverting (the outer frame succeeds; the `dexRouter.swap` sub-call runs out of gas and fails silently).
5. Relayer submits `revealWithCallback` with gas = `G`. The outer callback succeeds, `clearRequest` was already called at line 666, and the provider keeps the fee.
6. The DEX swap never executed. The user's ETH is locked in the callback contract with no way to retry the Entropy request. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
