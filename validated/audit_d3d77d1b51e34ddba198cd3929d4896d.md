### Title
`GAS_OVERHEAD` Constant in Scheduler Cannot Be Adjusted Without Contract Upgrade - (File: `target_chains/ethereum/pulse_sdk/solidity/SchedulerConstants.sol`)

### Summary

The `GAS_OVERHEAD` constant used in keeper fee calculation inside `Scheduler._processFeesAndPayKeeper` is hardcoded as a Solidity `constant` and cannot be adjusted by the admin or owner without a full UUPS contract upgrade. By contrast, the two other fee-related parameters — `singleUpdateKeeperFeeInWei` and `minimumBalancePerFeed` — both have dedicated admin setter functions in `SchedulerGovernance`. This asymmetry means that if the actual transaction overhead diverges from the hardcoded 30,000 gas estimate (e.g., on L2 chains with different gas models, or after EVM opcode repricing), the keeper fee calculation is permanently miscalibrated until an upgrade is executed.

### Finding Description

`GAS_OVERHEAD = 30000` is declared as a `public constant` in `SchedulerConstants.sol`:

```solidity
/// Fixed gas overhead component used in keeper fee calculation.
/// This is a rough estimate of the tx overhead for a keeper to call updatePriceFeeds.
uint256 public constant GAS_OVERHEAD = 30000;
```

It is consumed directly in `Scheduler._processFeesAndPayKeeper`:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
```

The two other configurable fee parameters have admin-callable setters in `SchedulerGovernance`:

```solidity
function setSingleUpdateKeeperFeeInWei(uint128 newFee) external { ... }
function setMinimumBalancePerFeed(uint128 newMinimumBalance) external { ... }
```

No equivalent setter exists for `GAS_OVERHEAD`. Adjusting it requires deploying a new implementation and calling `upgradeToAndOnlyProxy`, which requires the `owner` role (not just `admin`), is a heavier operation, and introduces upgrade risk.

### Impact Explanation

If `GAS_OVERHEAD` is lower than the actual transaction overhead on a given chain (e.g., on an L2 where calldata costs dominate, or after an EVM hard fork reprices opcodes), keepers are systematically underpaid for the overhead portion of each `updatePriceFeeds` call. At high gas prices this shortfall can make keeper updates unprofitable, causing keepers to stop submitting updates. Active subscriptions then receive stale price data, which downstream protocols relying on Pyth Pulse prices will consume as if fresh. Conversely, if `GAS_OVERHEAD` is too high, subscription balances drain faster than expected, causing subscriptions to hit `InsufficientBalance` reverts prematurely and deactivate.

### Likelihood Explanation

The Scheduler is deployed across multiple EVM-compatible chains with heterogeneous gas models. The hardcoded value of 30,000 gas is described in the code itself as "a rough estimate." On L2 networks (Arbitrum, Optimism, Base, etc.) the actual transaction overhead can differ substantially from Ethereum mainnet. Additionally, EIP-4844 and future EVM upgrades can reprice base transaction costs. Because the contract is already live with active subscriptions, any miscalibration persists until the owner executes a full upgrade cycle.

### Recommendation

Move `GAS_OVERHEAD` from a `constant` in `SchedulerConstants.sol` into `SchedulerState.State` as a mutable `uint256 gasOverhead` field, and add a corresponding setter in `SchedulerGovernance`, mirroring the existing pattern for `singleUpdateKeeperFeeInWei`:

```solidity
function setGasOverhead(uint256 newGasOverhead) external {
    _authorizeAdminAction();
    uint256 oldGasOverhead = _state.gasOverhead;
    _state.gasOverhead = newGasOverhead;
    emit GasOverheadSet(oldGasOverhead, newGasOverhead);
}
```

This allows the admin to dynamically recalibrate keeper fee accounting per chain without requiring a contract upgrade.
