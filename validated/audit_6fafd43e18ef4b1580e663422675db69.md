Audit Report

## Title
Unauthenticated Inbound Peer Can Poison Peer Store via Identify Listen Addresses — (`network/src/protocols/identify/mod.rs`)

## Summary
`process_listens` in the identify protocol unconditionally stores up to 10 attacker-supplied listen addresses per inbound connection into the peer store, despite the adjacent `add_outbound_addr` path explicitly rejecting inbound addresses as unverified. By cycling ~1639 cheap inbound connections, each injecting 10 globally-routable addresses from distinct `/16` subnets, an attacker can fill the entire peer store (`ADDR_COUNT_LIMIT = 16384`) with fabricated entries. Both eviction strategies in `check_purge` fail against this pattern, permanently blocking legitimate address insertion and redirecting the feeler mechanism exclusively to attacker-controlled dead-end addresses.

## Finding Description

**Root cause — missing session-type guard in `process_listens`.**

`received_identify` explicitly guards `add_outbound_addr` behind `context.session.ty.is_outbound()`, with a comment acknowledging that inbound addresses are unverified and should not be stored: [1](#0-0) 

However, `process_listens` — called unconditionally for both inbound and outbound sessions at line 281 — has no equivalent guard: [2](#0-1) 

`add_remote_listen_addrs` then calls `peer_store.add_addr()` for every address, logging errors but taking no corrective action: [3](#0-2) 

`add_addr` inserts each address with `last_connected_at_ms = 0` (never verified): [4](#0-3) 

**Eviction bypass — both `check_purge` strategies fail against the attack.**

Step 1 evicts addresses where `is_connectable()` returns false. For freshly injected attacker addresses (`attempts_count = 0`, `last_connected_at_ms = 0`), neither non-connectable condition is met — `attempts_count >= ADDR_MAX_RETRIES (3)` is false, and `attempts_count >= ADDR_MAX_FAILURES (10)` is false — so `is_connectable` returns `true` and Step 1 evicts nothing: [5](#0-4) 

Step 2 groups addresses by network segment and only evicts from groups with `> 4` peers. If the attacker uses one address per distinct `/16` subnet (trivially achievable across the public IPv4 space), every group has exactly 1 entry and the `> 4` threshold is never met, so Step 2 also evicts nothing: [6](#0-5) 

Both steps fail → `Err(PeerStoreError::EvictionFailed)` is returned, and no new address can ever be inserted again.

**Feeler mechanism hijack.** `fetch_addrs_to_feeler` selects addresses where `last_connected_at_ms` is not within the last 3 days. Attacker addresses have `last_connected_at_ms = 0`, so `connected(|t| t > addr_expired_ms)` evaluates to `false`, and `!false = true` — they pass the filter unconditionally: [7](#0-6) 

**Attack parameters.** `MAX_ADDRS = 10` is the disconnect threshold; sending exactly 10 addresses per connection maximizes injection rate while staying below the limit: [8](#0-7) 

`ADDR_COUNT_LIMIT = 16384`, so `ceil(16384 / 10) = 1639` connections suffice: [9](#0-8) 

The attacker does not need 1639 simultaneous connections. The default `max_inbound` is not a barrier because the attacker cycles: connect → send identify → disconnect → repeat: [10](#0-9) 

## Impact Explanation

Once the peer store is saturated, `fetch_addrs_to_feeler` returns only attacker-controlled addresses, all subsequent `add_addr` calls fail silently with `EvictionFailed`, and the victim node cannot discover new honest peers. `fetch_addrs_to_attempt` and `fetch_random_addrs` require `last_connected_at_ms > 0`, so existing outbound connections are not immediately severed, but the node cannot replenish its peer pool after natural churn. The effect is persistent until a node restart with manual peer store purge. Applied at scale across many nodes simultaneously — which is feasible given the negligible per-node cost — this degrades the CKB P2P network's peer discovery fabric. This matches **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

No privilege is required — any peer that can open a TCP connection to the victim's P2P port qualifies. The cost is approximately 1639 TCP connections each carrying a small identify message, achievable from a single machine. No proof-of-work, stake, or key material is needed. The effect is persistent and survives connection churn without any further attacker action. `ADDR_MAX_RETRIES = 3` and `ADDR_MAX_FAILURES = 10` are confirmed constants that ensure freshly injected addresses are never evicted by Step 1: [11](#0-10) 

## Recommendation

1. **Reject inbound listen-address injection**: add an `if context.session.ty.is_outbound()` guard to `process_listens` (mirroring the existing guard on `add_outbound_addr`). The comment at line 416 already documents the rationale.
2. **Source-IP diversity cap**: limit the number of addresses insertable per source `/16` network group regardless of session type.
3. **Feeler confirmation before persistence**: only persist an address after a successful feeler connection verifies reachability.
4. **Unconditional eviction fallback**: ensure `check_purge` can always make progress (e.g., evict the lowest-scored or oldest-inserted address when both strategies fail).

    pub fn is_connectable(&self, now_ms: u64) -> bool {
        // do not remove addr tried in last minute
        if self.tried_in_last_minute(now_ms) {
            return true;
        }
        // we give up if never connect to this addr

        }

    }
