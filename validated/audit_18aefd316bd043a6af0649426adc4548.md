Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Correct Accounting in `add_entry`, Inflating `total_tx_size`/`total_tx_cycles` - (File: `tx-pool/src/component/pool_map.rs`)

## Summary

In `PoolMap::add_entry`, `updated_stat_for_add_tx` computes `self.total_tx_size + entry.size` into a local variable before any eviction occurs. When `check_and_record_ancestors` subsequently evicts entries via `remove_entry_and_descendants` → `update_stat_for_remove_tx`, those correct in-place decrements to `self.total_tx_size`/`self.total_tx_cycles` are immediately overwritten by the stale pre-eviction local snapshot. This permanently inflates pool accounting by the size of every evicted entry, eventually causing the pool to reject all new transactions with `Reject::Full`.

## Finding Description

In `add_entry` (lines 200–221):

```rust
// Line 210-211: snapshot BEFORE eviction — pure read, self not modified
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
// total_tx_size  = self.total_tx_size + entry.size   (stale local)
// total_tx_cycles = self.total_tx_cycles + entry.cycles (stale local)

// Line 213: may evict N entries; each remove_entry call invokes
//           update_stat_for_remove_tx, writing self.total_tx_size -= evict_size
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218-219: OVERWRITES the correctly-decremented self.total_tx_size
//                with the stale pre-eviction snapshot
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` takes `&self` and returns computed values without touching `self` (lines 711–729). `update_stat_for_remove_tx` writes directly to `self.total_tx_size`/`self.total_tx_cycles` (lines 738–740). `remove_entry` calls `update_stat_for_remove_tx` at line 247. The eviction loop in `check_and_record_ancestors` (lines 615–625) calls `remove_entry_and_descendants` → `remove_entry` for each evicted entry.

Concrete arithmetic: if before insertion `self.total_tx_size = S` and eviction removes entries totalling `evict_size`, the correct final value is `S - evict_size + entry.size`. Instead, `S + entry.size` is written — inflated by `evict_size` per eviction event. The error is cumulative and unbounded across repeated insertions.

The eviction branch is entered at line 603 when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`. No existing guard prevents the overwrite at lines 218–219.

## Impact Explanation

`total_tx_size` is the primary guard for the `max_tx_pool_size` limit. Permanent inflation causes the pool to report itself as fuller than it actually is, causing all subsequent `send_transaction` RPC calls to return `Reject::Full` even though real capacity exists. The node's mempool is effectively disabled until restart. This maps to **High (10001–15000 points): Vulnerabilities or bad designs which could cause CKB network congestion with few costs** — an attacker can disable the mempool of any reachable node with a small, repeatable transaction sequence requiring no special privileges.

## Likelihood Explanation

The eviction branch is reachable by any unprivileged tx-pool submitter. The attacker only needs to construct transactions where some act as cell-dep parents, pushing `ancestors_count` above `max_ancestors_count` (default 125) while keeping non-cell-ref ancestors within the limit. No leaked keys, victim mistakes, or special access are required — only the ability to submit transactions to the node's RPC. The attack is repeatable: each qualifying submission permanently inflates the counters by the size of the evicted entry.

## Recommendation

Move `updated_stat_for_add_tx` to **after** `check_and_record_ancestors` completes, so it reads the post-eviction `self.total_tx_size`/`self.total_tx_cycles`:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, eliminate the local-variable pattern entirely: apply the size/cycles increment directly to `self.total_tx_size`/`self.total_tx_cycles` after all mutations complete, so eviction decrements and the new-entry increment are never in conflict.

## Proof of Concept

1. Submit a chain of 125 transactions `T1 → T2 → … → T125` where each `Ti` references the output of `T(i-1)` as a **cell dep** (making them all `cell_ref_parents`).
2. Submit `T_new` referencing `T1`'s output as a cell dep. Its `ancestors_count` = 126 > 125.
3. `check_and_record_ancestors` enters the eviction branch (line 603): `126 - 125 = 1 ≤ 125`, so it evicts `T125` (lowest fee), calling `update_stat_for_remove_tx(T125.size, T125.cycles)` — `self.total_tx_size` is correctly decremented.
4. `add_entry` then writes back the stale `total_tx_size` (computed before eviction), re-inflating it by `T125.size`.
5. Repeat steps 1–4. Each iteration inflates `total_tx_size` by one evicted entry's size.
6. After enough iterations, `total_tx_size >= max_tx_pool_size` even though the pool holds far fewer bytes; all subsequent `send_transaction` RPC calls return `Reject::Full`.

A unit test can assert the invariant `pool_map.total_tx_size == pool_map.entries.iter().map(|e| e.size).sum()` before and after a sequence of insertions that trigger the eviction path, and observe it fails after step 4.
