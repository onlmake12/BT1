### Title
Synchronous Full-VM Execution in `estimate_cycles` RPC Saturates RPC Thread Pool — (`rpc/src/module/chain.rs`)

### Summary

`estimate_cycles` (Chain module, enabled by default) calls `ScriptVerifier::verify(max_block_cycles)` **synchronously** on the RPC handler thread with no rate limiting, no per-connection limit, and no cycle cap below `consensus.max_block_cycles`. An unprivileged caller can send N concurrent requests with a max-cycle looping script to saturate all RPC worker threads, causing RPC unavailability including for `get_block_template`.

### Finding Description

In `CyclesEstimator::run()`: [1](#0-0) 

The method:
1. Sets `max_cycles = consensus.max_block_cycles` with no lower cap
2. Calls `ScriptVerifier::new(...).verify(max_cycles)` — a fully synchronous, blocking VM execution
3. Returns only after the VM halts or exhausts cycles

The `estimate_cycles` RPC method dispatches directly to this: [2](#0-1) 

The RPC config has a finite `threads` pool and **no per-connection or per-method rate limit**: [3](#0-2) 

A search for `spawn_blocking`, `block_in_place`, `rayon`, or `tokio::task` in `rpc/src/**` returns **no results** — the VM execution is never offloaded to a blocking-safe context.

### Impact Explanation

Sending N concurrent `estimate_cycles` requests (where N ≥ `threads`) with a script that loops for `max_block_cycles` (~3.5 billion cycles on mainnet, `0xd09dc300`) saturates the entire RPC thread pool. All other RPC methods — including `get_block_template` used by miners — become unresponsive for the duration of the attack. This causes node-level RPC DoS.

### Likelihood Explanation

The attack requires only HTTP access to the RPC port and knowledge of a live cell to reference. No authentication, no PoW, no privileged role. The Chain module is enabled by default. The only practical barrier is that CKB's default RPC listen address is `127.0.0.1`, but nodes that expose RPC publicly (e.g., for wallet/dApp use) are directly vulnerable.

**One uncertainty**: I was unable to read `rpc/src/server.rs` to confirm whether the RPC uses tokio worker threads or a dedicated thread pool. If it uses a dedicated pool (as `jsonrpc-http-server` typically does), the tokio async runtime is not directly blocked — but the RPC thread pool is still fully saturated, producing the same observable DoS on all RPC methods.

### Recommendation

- Add a configurable `max_cycles_for_estimate` cap in `RpcConfig`, defaulting to a fraction of `max_block_cycles`
- Offload `ScriptVerifier::verify()` to `tokio::task::spawn_blocking` or a rayon pool with a bounded semaphore
- Add per-IP or global concurrency limits for compute-heavy RPC methods

### Proof of Concept

1. Deploy a CKB script that executes a tight loop consuming exactly `max_block_cycles` cycles
2. Publish it as a live cell on-chain
3. Send `threads + 1` concurrent `estimate_cycles` JSON-RPC calls referencing that cell
4. Observe that `get_block_template` calls time out for the duration of VM execution [4](#0-3) [5](#0-4)
