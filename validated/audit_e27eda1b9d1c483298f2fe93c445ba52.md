### Title
Node Advertises `Flags::DISCOVERY` Capability in Identify Protocol Even When Discovery Protocol Is Disabled — (`util/launcher/src/lib.rs`)

---

### Summary

In `util/launcher/src/lib.rs`, the function `start_network_and_rpc()` builds the node's capability flags starting from `Flags::all()` and selectively removes flags for disabled protocols. However, `Flags::DISCOVERY` is **never removed** even when `SupportProtocol::Discovery` is absent from `support_protocols`. This means a node that has Discovery disabled still broadcasts `Flags::DISCOVERY` in its Identify protocol handshake, falsely advertising a capability it does not serve. Connecting peers will attempt to open the Discovery protocol with this node and fail, and the node will be propagated through the peer store as a discovery-capable peer, degrading the quality of the entire peer discovery network.

---

### Finding Description

In `start_network_and_rpc()`, the identify flags are initialized to `Flags::all()` and then selectively trimmed: [1](#0-0) 

The pattern is consistent for `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT`:

```rust
let mut flags = Flags::all();

if support_protocols.contains(&SupportProtocol::Relay) { ... }
else { flags.remove(Flags::RELAY); }

if support_protocols.contains(&SupportProtocol::Filter) { ... }
else { flags.remove(Flags::BLOCK_FILTER); }

if support_protocols.contains(&SupportProtocol::LightClient) { ... }
else { flags.remove(Flags::LIGHT_CLIENT); }
```

**`Flags::DISCOVERY` is never removed.** There is no corresponding block:

```rust
// MISSING:
if !support_protocols.contains(&SupportProtocol::Discovery) {
    flags.remove(Flags::DISCOVERY);
}
```

Meanwhile, in `NetworkService::new()`, the Discovery protocol handler is only registered when `config.support_protocols.contains(&SupportProtocol::Discovery)`: [2](#0-1) 

The `flags` value is passed directly into `IdentifyCallback::new()` and encoded into the Identify message sent to every connecting peer: [3](#0-2) 

The `Flags` bitfield definition confirms `DISCOVERY = 0b10` is part of `Flags::all()`: [4](#0-3) 

---

### Impact Explanation

When a peer receives the Identify message and sees `Flags::DISCOVERY` set, it proceeds to open all non-Feeler protocols with the node: [5](#0-4) 

This includes attempting to open the Discovery protocol. Since the Discovery handler is not registered on the misconfigured node, the tentacle framework rejects the protocol open. Additionally, the peer store records this node's address with the `DISCOVERY` flag: [6](#0-5) 

The `fetch_random_addrs` and `fetch_nat_addrs` functions filter peers by `required_flags`, so this node will be returned to other nodes requesting discovery-capable peers: [7](#0-6) 

The `required_flags_filter` function will pass this node through for any query that includes `Flags::DISCOVERY`: [8](#0-7) 

Concrete consequences:
1. Peers waste connection slots attempting to use Discovery on a node that cannot serve it.
2. The node is propagated network-wide in discovery responses as a discovery-capable peer.
3. Nodes that need discovery peers will repeatedly receive and attempt to connect to this node, degrading peer discovery quality across the network.

---

### Likelihood Explanation

The default `support_protocols` list in `ckb.toml` includes all protocols: [9](#0-8) 

However, the config comment explicitly states "only 'Sync' and 'Identify' are mandatory, others are optional." Node operators running lightweight, relay-only, or specialized nodes are a realistic population who would remove `Discovery` from `support_protocols`. The code path is reachable by any peer that connects to such a node — no privileged access is required.

---

### Recommendation

Add the missing flag removal in `start_network_and_rpc()` in `util/launcher/src/lib.rs`, following the same pattern used for `RELAY`, `BLOCK_FILTER`, and `LIGHT_CLIENT`:

```rust
if support_protocols.contains(&SupportProtocol::Discovery) {
    // Discovery handler is registered inside NetworkService::new()
} else {
    flags.remove(Flags::DISCOVERY);
}
```

This mirrors the existing pattern and ensures the advertised `Flags` in the Identify message accurately reflect the protocols the node actually serves.

---

### Proof of Concept

1. Configure a CKB node with `support_protocols` that excludes `Discovery`:
   ```toml
   support_protocols = ["Ping", "Identify", "Feeler", "DisconnectMessage", "Sync", "Relay", "Time", "Alert"]
   ```
2. Start the node. In `start_network_and_rpc()`, `flags` is initialized to `Flags::all()` (line 429). `Flags::RELAY` is kept (Relay is present), but `Flags::DISCOVERY` is never removed.
3. Connect a peer. The peer receives an Identify message with `Flags::DISCOVERY` set (bit `0b10` is present in the flag field).
4. The peer calls `required_flags_filter(required_flags, flags)` — it passes, so the peer attempts `open_protocols` with `TargetProtocol::Filter(exclude Feeler)`, which includes the Discovery protocol ID.
5. The tentacle framework on the misconfigured node rejects the Discovery protocol open because no handler was registered for it.
6. The peer stores this node's address with `flags` containing `DISCOVERY`, and will return it to future `GetNodes` requests from other peers that request discovery-capable nodes.

    #[derive(Clone, Copy, PartialEq, Eq, Hash, Debug)]

        const LIGHT_CLIENT = 0b10000;

pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    if required == Flags::RELAY | Flags::DISCOVERY | Flags::SYNC {
        t.contains(required) || t.contains(Flags::COMPATIBILITY)
    } else {
        t.contains(required)
    }
}
