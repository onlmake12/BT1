### Title
Acknowledged Off-by-One Epoch Boundary in RFC0044 MMR Extension Activation Causes Missing Chain Root for First Block of Activation Epoch â€” (`tx-pool/src/block_assembler/mod.rs`, `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

Both `BlockAssembler::build_extension` and `BlockExtensionVerifier::verify` use the **parent block's epoch number** â€” not the new block's epoch number â€” to decide whether RFC0044 (CKB Light Client Protocol, MMR chain root) is active. This off-by-one at the epoch boundary causes the first block of the RFC0044 activation epoch to be assembled and accepted by all full nodes **without** the required MMR chain root extension. Light clients relying on RFC0044 cannot verify the chain root for that specific block, creating a one-block gap in the MMR chain that a malicious full node can exploit to serve unverifiable data to light clients. The bug is explicitly acknowledged in the source code with a comment.

---

### Finding Description

In `BlockAssembler::build_extension` (`tx-pool/src/block_assembler/mod.rs`), the activation check for RFC0044 reads:

```rust
pub(crate) fn build_extension(snapshot: &Snapshot) -> Result<Option<packed::Bytes>, AnyError> {
    let tip_header = snapshot.tip_header();
    // The use of the epoch number of the tip here leads to an off-by-one bug,
    // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
    let mmr_activate = snapshot
        .consensus()
        .rfc0044_active(tip_header.epoch().number());
``` [1](#0-0) 

The function is building the extension for the **next** block (tip + 1), but it queries `rfc0044_active` using the **tip's** (parent's) epoch number. When the tip is the last block of epoch Nâˆ’1 and the new block is the first block of epoch N (the activation epoch), `rfc0044_active(Nâˆ’1)` returns `false`, so `build_extension` returns `None` â€” no MMR extension is included.

Because both the assembler and verifier use the parent's epoch number, they are mutually consistent: the first block of the activation epoch is assembled without the MMR extension and the verifier accepts it without the MMR extension. All full nodes agree, so there is no consensus split â€” but the MMR chain has a one-block gap at the activation boundary.

---

### Impact Explanation

The first block of the RFC0044 activation epoch (epoch 8651 on mainnet, epoch 5711 on testnet) is committed to the canonical chain **without** the MMR chain root extension. RFC0044 light clients cannot obtain or verify an MMR proof for that specific block. A malicious full node serving light clients can provide incorrect header data for that block, and the light client has no cryptographic means to detect the fraud â€” the MMR chain root that would normally cover that block is absent. Every subsequent block's MMR root is computed over a chain that skips the extension for block N, so the gap is permanent and cannot be retroactively corrected without a chain reorganization.

---

### Likelihood Explanation

This is a deterministic, one-time event at a fixed epoch boundary. Any miner who wins the first block slot of the activation epoch will produce a block without the MMR extension (the block template from `get_block_template` RPC will not contain it). The verifier accepts it. No special attacker capability is required beyond operating a full node that serves RFC0044 light clients. The bug has already manifested on both mainnet and testnet if those epoch numbers have been reached.

---

### Recommendation

Both `build_extension` and `BlockExtensionVerifier::verify` must be updated **together** (to preserve consensus) to use the **new block's** epoch number rather than the parent's:

1. In `build_extension`, derive the new block's epoch from `current_epoch` (already computed by the caller via `next_epoch_ext`) and pass it in, or compute `next_epoch_ext` inside the function.
2. In `BlockExtensionVerifier::verify`, use the block's own epoch number (derivable from `epoch_ext` already computed in `ContextualBlockVerifier::verify` at line 645â€“655) instead of `self.parent.epoch().number()`.

Since the activation epochs are already hardcoded and the first block of each activation epoch is already on-chain, a migration note for RFC0044 light client implementations should also document this one-block gap.

### Proof of Concept

Let N = `RFC0044_ACTIVE_EPOCH` (8651 mainnet / 5711 testnet).

1. Tip = last block of epoch Nâˆ’1. `tip_header.epoch().number()` = Nâˆ’1.
2. Miner calls `get_block_template` â†’ `build_extension` is called â†’ `rfc0044_active(Nâˆ’1)` = `false` â†’ returns `None` â†’ block template has no MMR extension.
3. Miner submits block B (first block of epoch N) without MMR extension.
4. `BlockExtensionVerifier::verify` checks `rfc0044_active(parent.epoch().number())` = `rfc0044_active(Nâˆ’1)` = `false` â†’ `mmr_active = false` â†’ extension absence is accepted â†’ block B is committed.
5. Block B+1 onward: parent is in epoch N â†’ `rfc0044_active(N)` = `true` â†’ MMR extension required and present.
6. RFC0044 light client requests proof for block B â†’ no chain root extension exists â†’ light client cannot verify block B â†’ malicious full node can serve arbitrary header data for block B undetected.

        let tip_header = snapshot.tip_header();
        // The use of the epoch number of the tip here leads to an off-by-one bug,
        // so be careful, it needs to be preserved for consistency reasons and not fixed directly.
        let mmr_activate = snapshot
            .consensus()

            Ok(Some(bytes))
        } else {
            Ok(None)
        }
