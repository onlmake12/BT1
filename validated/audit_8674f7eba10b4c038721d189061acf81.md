Audit Report

## Title
Stale Pre-Eviction Totals Overwrite Post-Eviction Decrements in `add_entry`, Permanently Inflating Pool Size Accounting — (`tx-pool/src/component/pool_map.rs`)

## Summary
In `PoolMap::add_entry`, `updated_stat_for_add_tx` captures `self.total_tx_size + new_size` into local variables before `check_and_record_ancestors` runs. That inner call can evict transactions via `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which correctly decrements `self.total_tx_size` in-place. The stale pre-eviction locals are then unconditionally written back to `self.total_tx_size` and `self.total_tx_cycles`, erasing those decrements. The pool permanently believes it holds more data than it actually does, causing spurious evictions and premature rejection of valid transactions.

## Finding Description
In `add_entry` (lines 210–219 of `tx-pool/src/component/pool_map.rs`):

```rust
// Lines 210-211: captures S + new_size into locals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Line 213: may evict; each eviction calls update_stat_for_remove_tx
//           → self.total_tx_size -= evicted_size  (in-place, correct)
evicts = self.check_and_record_ancestors(&mut entry)?;

// Lines 218-219: overwrites with stale pre-eviction value
self.total_tx_size  = total_tx_size;   // S + new_size  ← BUG
self.total_tx_cycles = total_tx_cycles;
```

`updated_stat_for_add_tx` (lines 711–729) takes `&self` — it is a pure read that computes `self.total_tx_size + tx_size` and returns it as a local without mutating state.

`check_and_record_ancestors` (lines 588–640) enters the eviction branch at line 603 when `ancestors_count.saturating_sub(cell_ref_parents.len()) <= self.max_ancestors_count` but `ancestors_count > self.max_ancestors_count`. It calls `remove_entry_and_descendants` (line 618), which chains to `remove_entry` (line 247), which calls `update_stat_for_remove_tx` (line 247) — directly decrementing `self.total_tx_size` in-place.

After eviction: `self.total_tx_size = S − evicted_size`. The unconditional write-back at line 218 then sets it to `S + new_size`, inflating it by exactly `evicted_size`. The correct post-operation value should be `S − evicted_size + new_size`. No guard exists between lines 213 and 218 to detect or compensate for in-place mutations made during eviction.

## Impact Explanation
`total_tx_size` is the sole counter driving `limit_size` in `pool.rs` (lines 298–328), which loops `while self.pool_map.total_tx_size > self.config.max_tx_pool_size`. An inflated counter produces two concrete, compounding effects:

1. **Spurious further evictions**: `limit_size` is called after `add_entry` returns and sees an inflated `total_tx_size`, evicting additional valid fee-paying transactions that fit within the real pool capacity.
2. **Premature rejection of future submissions**: subsequent calls to `updated_stat_for_add_tx` start from an already-inflated baseline, causing valid transactions to be rejected with `Reject::Full` even when real capacity exists.

The inflation is cumulative and permanent across repeated eviction-triggering submissions. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (10001–15000 points).

## Likelihood Explanation
The eviction branch in `check_and_record_ancestors` is triggered when a new transaction has `cell_ref_parents` in the pool and total `ancestors_count > max_ancestors_count` (default 25), but the non-cell-ref ancestor count alone does not exceed the limit. Both conditions are reachable by any unprivileged caller via the `send_transaction` RPC or relay peer. An attacker who observes mempool state via `get_raw_tx_pool` can deliberately engineer the required chain structure. The inflation is cumulative across multiple such submissions, making the attack repeatable at low cost with no special privileges required.

## Recommendation
Move `updated_stat_for_add_tx` to after `check_and_record_ancestors` completes, so eviction-driven decrements are already reflected in `self.total_tx_size` before the new entry's contribution is added:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size  = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, drop the locals entirely and apply the increment in-place after all mutations complete: `self.total_tx_size = self.total_tx_size.checked_add(entry.size).ok_or(...)?`.

## Proof of Concept
1. Fill the pool with a chain of 24 transactions `T1 → T2 → … → T24` where each spends the previous output.
2. Submit a separate transaction `R` that takes `T1`'s output as a **cell dep** (`R` becomes a `cell_ref_parent` of the chain).
3. Submit a new transaction `T_new` that spends `T1`'s output. `T_new` now has 24 ancestors + `R` as a cell-ref parent, exceeding `max_ancestors_count = 25`.
4. `check_and_record_ancestors` enters the eviction branch (line 603 condition satisfied), evicts `R` (size `S_R`) via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by `S_R`.
5. The write-back at line 218 overwrites `self.total_tx_size` with the pre-eviction value, inflating it by `S_R`.
6. Verify via `get_tip_tx_pool_info` that `total_tx_size` exceeds the sum of actual entry sizes, and observe that `limit_size` subsequently evicts a valid transaction that fits within `max_tx_pool_size`.
7. Repeat steps 1–6 to confirm the inflation is cumulative across multiple eviction events.
