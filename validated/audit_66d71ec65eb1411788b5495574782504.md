### Title
Unbounded O(N) Full-Pool Iteration on Every `tx_pool_info` and `estimate_fee_rate` RPC Call Enables CPU Exhaustion — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

`PoolMap::get_max_update_time()` performs a full linear scan over every entry in the transaction pool on every `tx_pool_info` RPC call. The pool is bounded only by total byte size (`max_tx_pool_size = 180 MB`), not by entry count. With a documented minimum transaction size of 72 bytes, the pool can hold up to ~2.5 million entries. A second, heavier path exists in `estimate_fee_rate`: it collects all pool entries into a `Vec` and sorts them O(N log N) on every call. Both RPC endpoints are enabled by default, have no rate limiting, and are reachable by any local or network RPC caller.

---

### Finding Description

**Root cause — `get_max_update_time()` iterates all entries unconditionally:**

`PoolMap::get_max_update_time()` in `tx-pool/src/component/pool_map.rs` performs a full scan of every pool entry to find the maximum timestamp:

```rust
pub(crate) fn get_max_update_time(&self) -> u64 {
    self.entries
        .iter()
        .map(|(_, entry)| entry.inner.timestamp)
        .max()
        .unwrap_or(0)
}
``` [1](#0-0) 

This function is called unconditionally on every `tx_pool_info` RPC invocation inside `TxPoolService::info()`:

```rust
last_txs_updated_at: tx_pool.pool_map.get_max_update_time(),
``` [2](#0-1) 

The `tx_pool_info` RPC is in the `Pool` module, which is enabled by default. [3](#0-2) 

**Root cause — `estimate_fee_rate` is O(N log N) per call:**

`TxPoolService::estimate_fee_rate()` first calls `get_all_entry_info()`, which iterates all pending, gap, and proposed entries and materialises them into `HashMap`s: [4](#0-3) 

The result is then passed to `weight_units_flow::Algorithm::estimate_fee_rate()`, which collects all entries into a `Vec` and sorts them:

```rust
let mut current_txs: Vec<_> = all_entry_info
    .pending.into_values()
    .chain(all_entry_info.proposed.into_values())
    .map(TxStatus::new_from_entry_info)
    .collect();
current_txs.sort_unstable_by(|a, b| b.cmp(a));
``` [5](#0-4) 

When the primary estimator is not ready, the fallback path calls `pool_map.estimate_fee_rate()`, which also iterates all entries: [6](#0-5) 

The `estimate_fee_rate` RPC is in the `Experiment` module, enabled by default. [7](#0-6) 

**Pool size is bounded by bytes, not by entry count:**

The default `max_tx_pool_size` is 180 MB: [8](#0-7) 

The codebase documents the minimum serialised transaction size as 72 bytes: [9](#0-8) 

This means the pool can hold up to `180,000,000 / 72 ≈ 2,500,000` entries. There is no separate cap on the number of entries, and no rate limiting on RPC calls.

**Analog mapping to the Compound report:**

| Compound | CKB |
|---|---|
| `allMarkets` list grows with each new market | `pool_map.entries` grows with each submitted transaction |
| `claimComp()` iterates all markets | `get_max_update_time()` iterates all entries on every `tx_pool_info` call |
| `maxAssets` defined but not enforced as entry count | `max_tx_pool_size` limits bytes but not entry count |

---

### Impact Explanation

An attacker who fills the pool with ~2.5 million minimum-size transactions and then issues a sustained flood of `tx_pool_info` or `estimate_fee_rate` RPC calls forces the node to perform millions of iterations per request. Because `info()` holds the `tx_pool` read lock during the full scan, concurrent legitimate operations that need the write lock (transaction submission, block processing callbacks) are delayed. The `estimate_fee_rate` path additionally allocates and sorts a Vec of all entries, amplifying memory pressure. The result is severe CPU and memory exhaustion, degrading or halting RPC responsiveness and block-assembly throughput.

---

### Likelihood Explanation

The `Pool` and `Experiment` RPC modules are enabled by default. No authentication or rate limiting is applied to individual RPC methods. Filling the pool with small transactions is cheap: at `min_fee_rate = 1000 shannons/KB`, a 72-byte transaction costs 72 shannons (~0.00000072 CKB). A local RPC caller (the explicitly in-scope attacker profile) can trigger this without any privileged access.

---

### Recommendation

1. **Cache `last_txs_updated_at`**: Maintain a running maximum timestamp that is updated incrementally on every insert/remove, eliminating the O(N) scan in `get_max_update_time()`.
2. **Cap entry count independently of byte size**: Introduce a `max_tx_pool_count` limit so the pool cannot hold an unbounded number of entries regardless of their individual sizes.
3. **Rate-limit expensive RPC endpoints**: Apply per-IP or per-connection rate limiting to `tx_pool_info`, `get_raw_tx_pool`, and `estimate_fee_rate`.
4. **Bound `estimate_fee_rate` work**: Sample or cap the number of entries processed during fee estimation rather than sorting the entire pool on every call.

---

### Proof of Concept

```
# Step 1: Fill the pool with ~2.5M minimum-size transactions
# (each paying 72 shannons at min_fee_rate=1000 shannons/KB)
for i in $(seq 1 2500000); do
    ckb-cli tx send --tx <min_size_tx_$i>
done

# Step 2: Flood tx_pool_info — each call triggers a full O(N) scan
while true; do
    curl -s -X POST http://127.0.0.1:8114 \
      -H 'Content-Type: application/json' \
      -d '{"jsonrpc":"2.0","method":"tx_pool_info","params":[],"id":1}' &
done

# Step 3: Alternatively, flood estimate_fee_rate for O(N log N) per call
while true; do
    curl -s -X POST http://127.0.0.1:8114 \
      -H 'Content-Type: application/json' \
      -d '{"jsonrpc":"2.0","method":"estimate_fee_rate","params":[],"id":1}' &
done
```

Each `tx_pool_info` call iterates up to 2.5 million entries; each `estimate_fee_rate` call additionally allocates and sorts them. Sustained concurrent calls exhaust CPU and stall the node's RPC and block-assembly pipeline.

    pub(crate) fn estimate_fee_rate(
        &self,
        mut target_blocks: usize,
        max_block_bytes: usize,
        max_block_cycles: Cycle,
        min_fee_rate: FeeRate,
    ) -> FeeRate {
        debug_assert!(target_blocks > 0);
        let iter = self.entries.iter_by_score().rev();
        let mut current_block_bytes = 0;
        let mut current_block_cycles = 0;
        for entry in iter {
            current_block_bytes += entry.inner.size;
            current_block_cycles += entry.inner.cycles;
            if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
                target_blocks -= 1;
                if target_blocks == 0 {
                    return entry.inner.fee_rate();
                }
                current_block_bytes = entry.inner.size;
                current_block_cycles = entry.inner.cycles;
            }
        }

        min_fee_rate
    }

    pub fn estimate_fee_rate(
        &self,
        target_blocks: BlockNumber,
        all_entry_info: TxPoolEntryInfo,
    ) -> Result<FeeRate, Error> {
        if !self.is_ready {
            return Err(Error::NotReady);
        }

        let sorted_current_txs = {
            let mut current_txs: Vec<_> = all_entry_info
                .pending
                .into_values()
                .chain(all_entry_info.proposed.into_values())
                .map(TxStatus::new_from_entry_info)
                .collect();
            current_txs.sort_unstable_by(|a, b| b.cmp(a));
            current_txs
        };

        self.do_estimate(target_blocks, &sorted_current_txs)
    }
