### Title
`ScriptHashTypeVerifier` Omits Type Script Hash Type Validation, Allowing Transactions with Disallowed Type Script Hash Types to Bypass Non-Contextual Checks — (`verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` iterates over transaction outputs and validates the hash type of each output's **lock script** against `ENABLED_SCRIPT_HASH_TYPE`, but entirely skips the optional **type script** on the same output. A transaction sender can craft an output whose type script carries a disallowed hash type (e.g., `Data3` = 6) and the transaction will pass every non-contextual gate, be admitted to the tx-pool, and only be rejected much later during script execution — after consuming validation resources. This is a direct structural analog to the Fintroller "inconsistent configuration" class: one field of a two-field structure is validated while the other is silently ignored.

---

### Finding Description

**Root cause — `ScriptHashTypeVerifier::verify()`**

`verification/src/transaction_verifier.rs` lines 796–814:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

The loop body calls `output.lock().hash_type()` exclusively. `output.type_()` — which returns an `Option<Script>` — is never inspected. Every output can carry a type script, and that type script has its own `hash_type` byte.

**The allowed set**

`util/constant/src/consensus.rs` defines:

```rust
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // ScriptHashType::Data
    1u8, // ScriptHashType::Type
    2u8, // ScriptHashType::Data1
    4u8, // ScriptHashType::Data2
};
``` [2](#0-1) 

Values 6 (`Data3`), 8 (`Data4`), … 254 (`Data127`) are all structurally valid molecule bytes (even numbers pass `ScriptHashType::verify_value`) but are **not** in `ENABLED_SCRIPT_HASH_TYPE`. [3](#0-2) 

**Where the check is used**

`NonContextualTransactionVerifier` is the first gate applied to every submitted transaction:

```rust
pub fn verify(&self) -> Result<(), Error> {
    self.version.verify()?;
    self.size.verify()?;
    self.empty.verify()?;
    self.duplicate_deps.verify()?;
    self.outputs_data_verifier.verify()?;
    self.script_hash_type.verify()?;   // ← only checks lock
    Ok(())
}
``` [4](#0-3) 

A transaction whose output has `lock.hash_type = Data` (0, allowed) and `type.hash_type = Data3` (6, disallowed) passes every line above. The disallowed type script hash type is only caught later inside `TransactionScriptsVerifier` when `select_version()` is called per script group:

```rust
hash_type => {
    return Err(ScriptError::InvalidScriptHashType(format!(
        "The ScriptHashType/{:?} has not been activated, and is not permitted for use.",
        hash_type
    )));
}
``` [5](#0-4) 

**The inconsistency**

The non-contextual verifier is supposed to be a complete, cheap, context-free gate. It correctly rejects a lock script with `Data3`, but silently accepts the identical byte in a type script on the same output. The two fields of the same `CellOutput` are treated asymmetrically with no documented rationale.

---

### Impact Explanation

1. **Tx-pool pollution**: An unprivileged sender submits transactions whose outputs carry disallowed type script hash types. Each transaction passes `NonContextualTransactionVerifier` and enters the tx-pool. It is only evicted after the more expensive contextual/script-execution path runs and returns an error. Until eviction, the slot is occupied and displaces legitimate transactions.

2. **Wasted validation work**: The tx-pool runs script verification on transactions that a complete non-contextual check would have rejected for free. Under sustained submission this wastes CPU cycles proportional to the number of crafted transactions.

3. **Inconsistent node state across implementations**: Any alternative CKB implementation that trusts the non-contextual result as a definitive "structurally valid" signal will diverge from the reference node's contextual result, creating a potential consensus split surface.

---

### Likelihood Explanation

The attack requires only the ability to submit a transaction via the public `send_transaction` RPC — no keys, no stake, no privileged access. Crafting an output with a type script whose `hash_type` byte is 6 is trivial with any molecule serialization library. The attacker pays only the minimum fee required for tx-pool admission (or zero fee if the node accepts zero-fee transactions). The attack is repeatable at low cost.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the hash type of the type script on each output:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        check_hash_type(output.lock().hash_type())?;
        // add type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

This mirrors the existing lock-script logic and ensures both fields of every `CellOutput` are validated at the non-contextual stage, eliminating the asymmetry.

---

### Proof of Concept

1. Build a `RawTransaction` with one output where:
   - `lock.hash_type = 0x00` (Data — allowed)
   - `type.hash_type = 0x06` (Data3 — disallowed, but structurally valid molecule)
2. Submit via `send_transaction` RPC.
3. **Expected (current behaviour)**: the call succeeds; the transaction enters the tx-pool because `ScriptHashTypeVerifier` only inspects the lock script.
4. **Expected (after fix)**: the call returns `TransactionError::ScriptHashTypeNotPermitted { hash_type: 6 }` immediately, before any script execution.

The inconsistency is confirmed by the existing test `test_not_enabled_hash_type_output_lock` (which passes `Data3` in the **lock** and expects rejection) — no equivalent test exists for the **type** script position. [6](#0-5)
