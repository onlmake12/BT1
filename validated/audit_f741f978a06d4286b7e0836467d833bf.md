### Title
Excess Fee Not Refunded in `update_price_feeds` — (`target_chains/aptos/contracts/sources/pyth.move`)

### Summary

The Aptos `update_price_feeds` function accepts a `Coin<AptosCoin>` fee parameter, checks only that the coin's value is **at least** the required fee, but then deposits the **entire** coin to `@pyth` — permanently consuming any excess above the required fee. This is the Aptos analog of the M-24 "fee/refund accounting" vulnerability class: tokens paid in excess of what is required are silently absorbed rather than returned to the caller.

### Finding Description

In `target_chains/aptos/contracts/sources/pyth.move`, the `update_price_feeds` function computes the actual required fee after processing all VAAs, asserts the provided coin is sufficient, and then unconditionally deposits the full coin:

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>) {
    let total_updates = 0;
    while (!vector::is_empty(&vaas)) {
        total_updates = total_updates + update_price_feed_from_single_vaa(vector::pop_back(&mut vaas));
    };
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    coin::deposit(@pyth, fee);   // <-- entire coin deposited, not just update_fee
}
``` [1](#0-0) 

The check `assert!(update_fee <= coin::value(&fee), ...)` explicitly permits overpayment, but `coin::deposit(@pyth, fee)` deposits the full `Coin` object. Aptos's `Coin` type is a resource — it cannot be partially consumed without splitting it first. The function never calls `coin::extract` or `coin::split` to separate the required amount from the excess, so the entire balance is transferred to the Pyth module address with no refund path.

By contrast, the entry-function wrapper `update_price_feeds_with_funder` correctly withdraws **exactly** the required fee:

```move
let fee = coin::withdraw<AptosCoin>(account, update_fee);
coin::deposit(@pyth, fee);
``` [2](#0-1) 

The public `update_price_feeds` function lacks this precision.

### Impact Explanation

Any caller who passes a `Coin<AptosCoin>` with value greater than `update_fee` permanently loses the excess to the Pyth contract. The excess is unrecoverable — there is no withdrawal mechanism for overpaid fees. Integrators who compute the fee conservatively (e.g., rounding up, or using a cached fee estimate that is higher than the actual fee at execution time) will silently lose funds on every call.

### Likelihood Explanation

The public `update_price_feeds` function is the primary non-entry API for integrators. The documentation example shows:

```move
let coins = coin::withdraw(user, pyth::get_update_fee(&pyth_price_update));
pyth::update_price_feeds(pyth_price_update, coins);
``` [3](#0-2) 

Any integrator who does not call `get_update_fee` immediately before calling `update_price_feeds` in the same transaction — or who adds a safety buffer — will overpay. The fee is governance-adjustable, so a fee decrease between the time a coin is prepared and the time the transaction executes also produces overpayment.

### Recommendation

Split the coin before depositing, returning the remainder to the caller:

```move
public fun update_price_feeds(vaas: vector<vector<u8>>, fee: Coin<AptosCoin>) {
    // ... compute total_updates ...
    let update_fee = state::get_base_update_fee() * total_updates;
    assert!(update_fee <= coin::value(&fee), error::insufficient_fee());
    let exact_fee = coin::extract(&mut fee, update_fee);
    coin::deposit(@pyth, exact_fee);
    coin::deposit(sender_address, fee);  // refund excess
}
```

Alternatively, enforce that the caller passes exactly the required amount (as `update_price_feeds_with_funder` effectively does).

### Proof of Concept

1. Governance sets `base_update_fee = 50`.
2. Integrator calls `get_update_fee` and gets `50` for a single-VAA update.
3. Integrator withdraws `100` coins as a safety buffer.
4. Integrator calls `update_price_feeds(vaas, coin_of_100)`.
5. `update_fee = 50`, assertion passes (`50 <= 100`).
6. `coin::deposit(@pyth, coin_of_100)` — all 100 coins go to `@pyth`.
7. Integrator loses 50 coins with no recourse. [4](#0-3)

    public fun get_update_fee(update_data: &vector<vector<u8>>): u64 {
        let i = 0;
        let total_updates = 0;
        while (i < vector::length(update_data)) {
            let cur = cursor::init(*vector::borrow(update_data, i));
            let header: u64 = deserialize::deserialize_u32(&mut cur);
            if (header == PYTHNET_ACCUMULATOR_UPDATE_MAGIC) {
                //TODO: this may be expensive and can be optimized by not verifying the messages
                let updates = parse_and_verify_accumulator_message(&mut cur);
                total_updates = total_updates + vector::length(&updates);
            }
            else {
                total_updates = total_updates + 1;
            };
            cursor::rest(cur);
            i = i + 1;
        };
