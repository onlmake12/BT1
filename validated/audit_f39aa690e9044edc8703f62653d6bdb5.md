### Title
`Scheduler.updateSubscription` Skips `_clearRemovedPriceUpdates` When Subscription Remains Inactive, Leaving Stale Price Data Permanently Accessible — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.updateSubscription` contains an early-return path for the case where a subscription is already inactive and will remain inactive (`!wasActive && !willBeActive`). This path stores the new params but **skips the `_clearRemovedPriceUpdates` call**, leaving stale price data for removed price IDs permanently in `_state.priceUpdates[subscriptionId]`. When the subscription is later reactivated, `_clearRemovedPriceUpdates` is called but compares the already-updated (inactive) price IDs against the new ones — finding no difference — so the orphaned data is never cleared. `_getPricesInternal` does not validate that a requested price ID is still in `params.priceIds`, so any whitelisted reader (or any caller when `whitelistEnabled = false`) can retrieve stale price data for price IDs that were supposed to be removed.

---

### Finding Description

**Root cause — early return skips cleanup:** [1](#0-0) 

When both `wasActive` and `willBeActive` are `false`, the function stores `newParams` (which may contain a completely different `priceIds` array) and returns immediately, bypassing `_clearRemovedPriceUpdates`.

**The cleanup that is skipped:** [2](#0-1) 

**Why reactivation does not fix it:**

When the subscription is later reactivated (`wasActive=false`, `willBeActive=true`), `_clearRemovedPriceUpdates` is called with `currentParams.priceIds` — but `currentParams` was already overwritten in the inactive-update step. Both arguments are now the *new* price IDs, so the diff is empty and nothing is deleted.

**Stale data is readable without an `isActive` check:** [3](#0-2) 

`_getPricesInternal` reads directly from `_state.priceUpdates[subscriptionId][priceIds[i]]` for any caller-supplied price ID. It only checks that the stored entry is non-zero — it does **not** verify that the price ID is still in `params.priceIds`, and it does **not** check `params.isActive`.

**Read functions that expose this data:** [4](#0-3) 

`getPricesUnsafe`, `getPricesNoOlderThan`, `getEmaPricesUnsafe`, and `getEmaPricesNoOlderThan` all call `_getPricesInternal` and are gated only by `onlyWhitelistedReader`, not by `isActive`.

---

### Impact Explanation

**Concrete attack path:**

1. Subscription is active with `priceIds = [A, B, C]`, `whitelistEnabled = false` (open to all readers).
2. Manager deactivates the subscription (`isActive = false`). `_clearRemovedPriceUpdates([A,B,C], [A,B,C])` is called — no change.
3. Manager calls `updateSubscription` with `isActive = false` and `priceIds = [D, E]`. Early-return path fires; `_clearRemovedPriceUpdates` is **not** called. Price data for A, B, C remains in `_state.priceUpdates`.
4. Manager reactivates with `priceIds = [D, E]`. `_clearRemovedPriceUpdates([D,E], [D,E])` — no diff, no deletion.
5. Any unprivileged user calls `getPricesUnsafe(subscriptionId, [A, B, C])` and receives the stale price data for the removed feeds.

The impact is that price data the subscription manager intended to retire remains permanently queryable. Downstream contracts or users relying on the Scheduler as a trusted price source may consume stale, orphaned price data for feeds that are no longer actively maintained.

---

### Likelihood Explanation

The trigger requires the subscription manager to (a) deactivate, (b) change price IDs while inactive, and (c) reactivate — a plausible operational sequence when rotating feed sets. The subscription must have `whitelistEnabled = false` (or the attacker must be a whitelisted reader) for an unprivileged party to exploit it. Likelihood is **low-medium**.

---

### Recommendation

Move `_clearRemovedPriceUpdates` into the `!wasActive && !willBeActive` branch before storing the new params:

```solidity
if (!wasActive && !willBeActive) {
    _clearRemovedPriceUpdates(
        subscriptionId,
        currentParams.priceIds,
        newParams.priceIds
    );
    _state.subscriptionParams[subscriptionId] = newParams;
    emit SubscriptionUpdated(subscriptionId);
    return;
}
```

Additionally, consider adding an `isActive` check inside `_getPricesInternal` or the read functions, and validating that each requested price ID is present in `params.priceIds`.

### Proof of Concept

```solidity
// 1. Create subscription with feeds [A, B, C], whitelistEnabled=false
uint256 subId = scheduler.createSubscription{value: 1 ether}(paramsABC);

// 2. Push a price update so data exists for A, B, C
scheduler.updatePriceFeeds(subId, updateDataABC);

// 3. Deactivate
paramsABC.isActive = false;
scheduler.updateSubscription(subId, paramsABC);

// 4. Change price IDs while inactive — _clearRemovedPriceUpdates is SKIPPED
SchedulerStructs.SubscriptionParams memory paramsDE = paramsABC;
paramsDE.priceIds = [D, E];
scheduler.updateSubscription(subId, paramsDE); // early-return path

// 5. Reactivate with [D, E] — _clearRemovedPriceUpdates([D,E],[D,E]) = no-op
paramsDE.isActive = true;
scheduler.addFunds{value: 1 ether}(subId); // need funds first
scheduler.updateSubscription(subId, paramsDE);

// 6. Any unprivileged user reads stale data for A, B, C
bytes32[] memory oldIds = [A, B, C];
PythStructs.Price[] memory stalePrices = scheduler.getPricesUnsafe(subId, oldIds);
// stalePrices contains the old data — never cleared
```
