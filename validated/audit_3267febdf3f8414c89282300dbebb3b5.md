### Summary

`PythAggregatorV3` is Pyth's production Chainlink-compatible adapter contract. Every price-returning function in the contract calls `getPriceUnsafe()`, which explicitly makes no recency guarantees and can return a price from arbitrarily far in the past. No staleness check is performed before returning data to callers. Any downstream protocol that integrates this adapter as a drop-in Chainlink feed will silently consume stale prices.

---

### Finding Description

`PythAggregatorV3.sol` implements the Chainlink `AggregatorV3Interface` by wrapping Pyth price feeds. All five price-returning functions delegate to `pyth.getPriceUnsafe(priceId)`:

- `decimals()` â€” line 41
- `latestAnswer()` â€” line 54
- `latestTimestamp()` â€” line 59
- `getRoundData()` â€” line 89
- `latestRoundData()` â€” line 110

`IPyth.sol` documents `getPriceUnsafe` explicitly:

> *"This function returns the most recent price update in this contract without any recency checks. This function is unsafe as the returned price update may be arbitrarily far in the past."*

The safe alternative, `getPriceNoOlderThan(id, age)`, exists in `AbstractPyth.sol` (lines 50â€“60) and reverts with `StalePrice` when the price is too old. `PythAggregatorV3` never calls it.

Protocols migrating from Chainlink to Pyth via this adapter expect the same liveness guarantees they had with Chainlink. Chainlink's `latestRoundData()` is expected to revert or return a stale flag when data is not fresh. `PythAggregatorV3.latestRoundData()` silently returns whatever the last stored price is, regardless of age.

### Impact Explanation

Any protocol that:
1. Deploys `PythAggregatorV3` as a Chainlink-compatible price feed, and
2. Calls `latestRoundData()` or `latestAnswer()` without independently checking `updatedAt`

â€¦will consume arbitrarily stale prices. This enables:

- **Borrowing against stale collateral valuations** â€” an attacker can borrow more than the current collateral is worth if the price is stale-high.
- **Avoiding liquidation** â€” a position that should be liquidated at the current price is safe at the stale price.
- **Exploiting stale-low prices** â€” an attacker can liquidate healthy positions if the stale price is below the current market price.

These are direct financial losses to protocol users and LPs. The impact is high on any lending, derivatives, or margin protocol using this adapter.

---

### Likelihood Explanation

Pyth explicitly markets `PythAggregatorV3` as a migration path for Chainlink integrators (see `apps/developer-hub/content/docs/price-feeds/core/migrate-an-app-to-pyth/chainlink.mdx`). The contract is published in the `@pythnetwork/pyth-sdk-solidity` npm package. Protocols that deploy it and rely on the Chainlink interface convention (where `latestRoundData` is assumed to be fresh) will not add a secondary staleness check.

Price staleness occurs naturally during:
- L2 sequencer downtime (the exact scenario in the original report)
- Network-level disruptions preventing Hermes from posting updates
- Market closures for equity/FX feeds
- Any gap in the push-update schedule

An attacker only needs to wait for a staleness window and then interact with the vulnerable protocol. No privileged access is required.

---

### Recommendation

Replace all `getPriceUnsafe(priceId)` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable constructor parameter (e.g., defaulting to `3600` seconds). This mirrors the Chainlink best practice of checking `updatedAt` against a heartbeat threshold.

```solidity
uint256 public maxAge;

constructor(address _pyth, bytes32 _priceId, uint256 _maxAge) {
    priceId = _priceId;
    pyth = IPyth(_pyth);
    maxAge = _maxAge;
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceNoOlderThan(priceId, maxAge);
    // ...
}
```
