Audit Report

## Title
Unbounded `last_n_numbers` bypasses `GET_LAST_STATE_PROOF_LIMIT` in `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

## Summary
The upfront limit check at lines 201–205 bounds total samples using `last_n_blocks * 2`, implicitly assuming `last_n_numbers.len() ≤ last_n_blocks`. This assumption is violated when `difficulty_boundary` is set to a value of `1`, causing `difficulty_boundary_block_number` to resolve to `start_block_number` and `last_n_numbers` to span the entire chain. The resulting `block_numbers` vector can contain millions of entries, which are all processed by `complete_headers` with O(n) expensive database and MMR operations per entry.

## Finding Description
**Upfront check (lines 201–205):** [1](#0-0) 

The check computes `difficulties.len() + last_n_blocks * 2`. With `difficulties=[]` and `last_n_blocks=499`, this yields `998 ≤ 1000` — passes. The factor of `2` accounts for both `reorg_last_n_numbers` and `last_n_numbers`, each assumed to be at most `last_n_blocks` entries.

**Root cause — `get_first_block_total_difficulty_is_not_less_than` (lines 30–32):** [2](#0-1) 

When `difficulty_boundary = U256::from(1)`, the total difficulty at `start_block_number` (e.g., block 500) is always ≥ 1 on any real chain, so this function immediately returns `Some((start_block_number, ...))`. Thus `difficulty_boundary_block_number = start_block_number = 500`.

**Adjustment guard (lines 313–316) does not fire:** [3](#0-2) 

With `last_block_number = 1,000,000`, `difficulty_boundary_block_number = 500`, `last_n_blocks = 499`: `1,000,000 − 500 = 999,500 ≥ 499` → condition is false → no clamping occurs.

**`last_n_numbers` spans the entire chain (lines 318–319):** [4](#0-3) 

`(500..1_000_000)` produces 999,500 entries — far exceeding the intended limit of 1,000.

**Validation checks for `difficulties=[]` are all skipped (lines 259–288):** [5](#0-4) [6](#0-5) 

The check at 259–266 uses `.unwrap_or(false)` (empty vec → skipped). The check at 268–288 is gated on `difficulties.first()` being `Some` (empty vec → skipped). So `difficulty_boundary = 1` is never validated against chain state.

**`complete_headers` performs O(n) expensive work per entry (lines 132–177):** [7](#0-6) 

Each entry triggers `get_ancestor`, `get_block`, and `chain_root_mmr(...).get_root()` — all database/MMR operations. With ~999,500 entries, this causes severe CPU and memory exhaustion.

**Limit constant for reference:** [8](#0-7) 

## Impact Explanation
An unprivileged light-client peer can force the full-node server to allocate a `Vec` of millions of block numbers and then perform millions of database reads and MMR root computations inside `complete_headers`. On a mainnet node with millions of blocks, this causes severe CPU and memory exhaustion, leading to OOM kill or an indefinite hang. This matches **High: Vulnerabilities which could easily crash a CKB node** (10001–15000 points).

## Likelihood Explanation
The attack requires only a valid `last_hash` on the main chain (trivially obtained from `SendLastState`) and a `difficulty_boundary` of `1`. No proof-of-work, no key, no privileged role is needed. A single malicious peer can trigger this repeatedly at negligible cost (one crafted message per attack). The attack is fully deterministic and requires no timing or race conditions.

## Recommendation
After computing `block_numbers` at lines 350–354, add a post-condition guard:

```rust
if block_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Alternatively, fix the upfront check to bound `last_n_numbers` independently of `last_n_blocks` by clamping `difficulty_boundary_block_number` so that `last_block_number - difficulty_boundary_block_number ≤ last_n_blocks` is always enforced before constructing `last_n_numbers`.

## Proof of Concept
Craft a `GetLastStateProof` message with:
- `last_hash` = current tip hash (from `SendLastState`)
- `start_hash` = any hash not equal to the ancestor of `last_hash` at `start_number`
- `start_number` = 500
- `last_n_blocks` = 499
- `difficulty_boundary` = `U256::from(1u64)`
- `difficulties` = `[]`

On a node with chain height ≥ 1,000,000:
1. Limit check: `0 + 998 = 998 ≤ 1000` → passes.
2. `reorg_last_n_numbers` = up to 499 entries.
3. `difficulty_boundary_block_number` = 500 (total difficulty at block 500 ≥ 1).
4. `1,000,000 − 500 = 999,500 ≥ 499` → adjustment guard does not fire.
5. `last_n_numbers` = 999,500 entries.
6. `block_numbers.len()` ≈ 999,999 >> 1000.
7. `complete_headers` performs ~999,999 `get_ancestor` + `get_block` + `chain_root_mmr` calls → OOM or hang.
