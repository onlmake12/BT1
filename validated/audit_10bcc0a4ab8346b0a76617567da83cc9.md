### Title
Authorized Upgrade Hash Not Cleared After Execution Allows Unprivileged Replay of Contract Upgrade — (`target_chains/near/receiver/src/governance.rs`)

---

### Summary

In the NEAR Pyth receiver contract, `self.codehash` is never reset to zero after a successful `upgrade()` execution. Because `update_contract()` is callable by **anyone**, any unprivileged actor can re-submit the same code bytes after a legitimate upgrade has already completed, causing the contract to be re-deployed and `migrate()` to be re-executed — all without a new governance VAA.

---

### Finding Description

The two-step upgrade flow on NEAR is:

1. **Governance VAA** → `verify_gov_vaa_callback` → `set_upgrade_hash(codehash)` stores the authorized hash in `self.codehash`.
2. **Anyone** calls `update_contract()` → `upgrade()` checks `sha256(new_code) == self.codehash`, deploys the code, and chains `migrate()`.

The critical gap is in `upgrade()`:

```rust
fn upgrade(&mut self, new_code: Vec<u8>) -> Result<Promise, Error> {
    let signature = TryInto::<[u8; 32]>::try_into(env::sha256(&new_code)).unwrap();
    let default = <[u8; 32] as Default>::default();
    ensure!(signature != default, UnauthorizedUpgrade);
    ensure!(signature == self.codehash, UnauthorizedUpgrade);   // ← checked but never cleared

    Ok(Promise::new(env::current_account_id())
        .deploy_contract(new_code)
        .then(Self::ext(env::current_account_id()).migrate())   // ← re-runs on replay
        ...
    )
}
```

`self.codehash` retains the hash of the just-deployed code after the upgrade completes. [1](#0-0) 

Contrast this with the TON implementation, which **explicitly resets** `upgrade_code_hash = 0` after executing the upgrade: [2](#0-1) 

And the Aptos implementation, which uses Move's `move_from<ContractUpgradeAuthorized>(@pyth)` — a destructive read that removes the resource from global storage, making a second call abort immediately: [3](#0-2) 

The NEAR contract has no equivalent clearing step.

---

### Impact Explanation

After a legitimate upgrade from v1 → v2 completes:

- `self.codehash` still equals `sha256(v2_code)`.
- Any unprivileged caller can invoke `update_contract()` with the same `v2_code` bytes.
- The contract re-deploys v2 and re-executes `migrate()` — without any governance authorization.

If `migrate()` resets or modifies contract state (e.g., re-initializes storage fields, resets counters, or changes configuration), repeated invocations can corrupt live contract state. Even if `migrate()` is currently idempotent, the invariant that "each governance-authorized upgrade executes exactly once" is broken, and future `migrate()` implementations may not be safe to re-run.

Additionally, if governance later rolls back from v2 to v1 (new VAA, `codehash = sha256(v1_code)`), the rollback executes once, but then `self.codehash = sha256(v1_code)` persists — allowing anyone to re-run the rollback's `migrate()` indefinitely.

---

### Likelihood Explanation

- `update_contract()` is explicitly documented as **open to call by anyone**: [4](#0-3) 
- The attacker only needs to know the code bytes of the previously deployed version (publicly available on-chain).
- No privileged access, key material, or social engineering is required.
- The attack is executable immediately after any legitimate upgrade.

---

### Recommendation

Reset `self.codehash` to the zero array at the start of `upgrade()` before dispatching the async promise, so that even if the promise chain is replayed or re-entered, the authorization is consumed:

```rust
fn upgrade(&mut self, new_code: Vec<u8>) -> Result<Promise, Error> {
    let signature = TryInto::<[u8; 32]>::try_into(env::sha256(&new_code)).unwrap();
    let default = <[u8; 32] as Default>::default();
    ensure!(signature != default, UnauthorizedUpgrade);
    ensure!(signature == self.codehash, UnauthorizedUpgrade);

    // Consume the authorization — prevent replay
    self.codehash = default;

    Ok(Promise::new(env::current_account_id())
        .deploy_contract(new_code)
        .then(Self::ext(env::current_account_id()).migrate())
        ...)
}
```

This mirrors the TON fix (`upgrade_code_hash = 0`) and the Aptos design (`move_from` consuming the resource).

---

### Proof of Concept

1. Governance issues VAA authorizing upgrade to `v2_code`; `self.codehash = sha256(v2_code)`.
2. Attacker (or anyone) calls `update_contract()` with `v2_code` → upgrade succeeds, `migrate()` runs. `self.codehash` is **not cleared**.
3. Attacker immediately calls `update_contract()` again with the same `v2_code` bytes.
4. `sha256(v2_code) == self.codehash` passes → contract re-deploys v2 and re-executes `migrate()` with no governance VAA involved.
5. Step 3–4 can be repeated indefinitely by any unprivileged caller. [1](#0-0) [5](#0-4)
