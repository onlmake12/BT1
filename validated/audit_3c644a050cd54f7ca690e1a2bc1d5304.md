### Title
Missing Rate Limiter on Filter Protocol Enables Unbounded DB Read Amplification via Repeated `GetBlockFilters` — (`sync/src/filter/get_block_filters_process.rs`)

### Summary

The `BlockFilter` protocol handler has no rate limiter. An unprivileged remote peer can send repeated `GetBlockFilters(start_number=K)` messages at maximum rate, each triggering up to 1000 `get_block_hash` + 1000 `get_block_filter` DB reads and serialization of up to 1.8 MB of response data, with no throttling, caching, or deduplication.

### Finding Description

`GetBlockFiltersProcess::execute` performs a loop of up to `BATCH_SIZE = 1000` iterations, calling `active_chain.get_block_hash(block_number)` and `active_chain.get_block_filter(&block_hash)` on each iteration, then serializes and sends a response of up to 1.8 MB. [1](#0-0) [2](#0-1) 

The `BlockFilter` struct contains only `shared: Arc<SyncShared>` — no rate limiter field exists: [3](#0-2) 

`BlockFilter::try_process` dispatches directly to `GetBlockFiltersProcess::execute` with no rate check: [4](#0-3) 

Compare this to the `Relayer` protocol, which explicitly carries a `rate_limiter: RateLimiter<(PeerIndex, u32)>` and checks it at the top of `try_process` before dispatching any message handler: [5](#0-4) [6](#0-5) 

The `Relayer` rate limiter is initialized at 30 requests/second per `(PeerIndex, message_type)` key. No equivalent protection exists anywhere in the Filter protocol stack — not in `BlockFilter::received`, `BlockFilter::process`, or `BlockFilter::try_process`. [7](#0-6) 

The only network-level protection is a 2 MB `max_frame_length` for the Filter protocol, which limits the size of *incoming* messages. A `GetBlockFilters` message is a fixed 8-byte struct (`start_number: Uint64`), so this limit provides no protection against request flooding. [8](#0-7) 

### Impact Explanation

Each `GetBlockFilters(start_number=0)` message sent by an attacker causes the victim node to:
- Execute up to 1000 `get_block_hash` DB reads
- Execute up to 1000 `get_block_filter` DB reads
- Serialize and transmit up to ~1.8 MB of response data

The request itself is 8 bytes. The amplification ratio (request bytes → DB operations + response bytes) is extreme. A single peer sending at the rate the TCP stack allows can saturate the node's DB I/O and serialization CPU. The `BlockFilter` handler is `async` and single-threaded per protocol session, so flooding it also blocks legitimate filter requests from other peers.

### Likelihood Explanation

The attack requires only a valid P2P connection to the Filter protocol (protocol ID 121, `/ckb/filter`). No authentication, no PoW, no stake. The message is trivially constructed. The absence of rate limiting is a clear oversight given that the `Relayer` and `HolePunching` protocols both implement per-peer rate limiting with the `governor` crate. [9](#0-8) 

### Recommendation

Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`) and check it at the top of `BlockFilter::try_process` before dispatching to any process handler. A quota of 30 requests/second per `(peer, message_type)` pair (matching the Relayer's policy) would be a reasonable starting point.

### Proof of Concept

1. Connect to a CKB node as an unprivileged peer on the Filter protocol.
2. In a tight loop, send `GetBlockFilters { start_number: 0 }` (8 bytes each).
3. Instrument `active_chain.get_block_filter` call count on the victim node.
4. Observe that the call count grows linearly and unboundedly with the number of messages sent, with no throttling applied.
