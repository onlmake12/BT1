### Title
State UTxO Permanent Freeze via Extra-Token Output Poisoning — (`lazer/contracts/cardano/lib/state.ak`)

### Summary

`state.spend` enforces a strict single-token invariant on the **input** UTxO (line 141) but uses the weaker `has_nft_strict` on the **output** UTxO (line 155). Because `has_nft_strict` only verifies the specific policy's asset map and does not assert the absence of other policies, any unprivileged actor who submits a valid governance VAA can include extra tokens in the output state UTxO. The resulting UTxO permanently fails the input check on every subsequent spend, freezing the Wormhole or Pyth state forever.

---

### Finding Description

**Root cause — asymmetric strictness between input and output checks**

`state.spend` resolves the current state UTxO and immediately asserts:

```aiken
// state.ak line 141
expect [(id, asset, 1)] = assets.without_lovelace(value) |> assets.flatten()
```

This pattern-match on the flattened non-ADA assets requires **exactly one triple**. Any extra token from any policy causes an irrecoverable `expect` failure. [1](#0-0) 

When the same function validates the **output** that will become the next state UTxO, it uses:

```aiken
// state.ak line 155
expect assets.has_nft_strict(value, id, config.state_nft)
```

`assets.has_nft_strict` (Aiken stdlib `cardano/assets`) only inspects the entry for `id` in the value map. It checks that `{id: {state_nft: 1}}` is the sole entry **under that policy**, but it makes no assertion about other policies being absent. A value of `{id: {state_nft: 1}, attacker_policy: {junk: 1}}` passes `has_nft_strict` without issue. [2](#0-1) 

The comment on line 154 explicitly states the intent ("output with the state NFT should contain nothing more than ADA") but the code does not enforce it. [3](#0-2) 

The same gap exists in `mint_state`: the comment on line 89 says "nothing more than ADA" but the find-predicate is again only `has_nft_strict`. [4](#0-3) 

---

**Permissionless entry point**

Both `wormhole_state.update.spend` and `pyth_state.update.spend` have **no caller access control**. Any party who possesses a valid, guardian-signed VAA may submit the spending transaction:

```aiken
// wormhole_state.ak lines 29-41
validator update {
  spend(old: Option<Guardians>, upgrade: PreparedVAA, utxo: OutputReference, self: Transaction) {
    let old, _ <- state.spend(config, old, utxo, self)
    state.new(old.home, governance.parse_and_verify_guardian_set_upgrade(upgrade, old.data), ...)
  }
``` [5](#0-4) 

Wormhole guardian-set upgrade VAAs and Pyth governance VAAs are published on-chain / in public relayers. Any observer can obtain them without privileged access. [6](#0-5) 

---

**Attack sequence**

1. Attacker monitors public Wormhole/Pyth relayers for a valid, unsubmitted governance VAA.
2. Attacker constructs a Cardano transaction that:
   - Spends the current state UTxO (which has only `{id: {state_nft: 1}}` + ADA, so line 141 passes).
   - Provides the valid VAA as the redeemer.
   - Mints one unit of an attacker-controlled policy (`attacker_policy: {junk: 1}`) in the same transaction.
   - Sends the output state UTxO with value `{id: {state_nft: 1}, attacker_policy: {junk: 1}}` + ADA.
3. `has_nft_strict(value, id, config.state_nft)` at line 155 passes — it only checks the `id` policy entry.
4. The transaction is accepted; the new state UTxO carries the extra token.
5. Every future call to `state.spend` hits line 141 with `flatten()` returning two triples instead of one → irrecoverable `expect` failure → state UTxO is permanently unspendable.

---

### Impact Explanation

Permanent freezing of the Wormhole guardian-set UTxO and/or the Pyth state UTxO on Cardano. No further guardian-set upgrades, Pyth governance actions, or trusted-signer updates can ever be applied. The protocol is bricked on Cardano with no recovery path (the state NFT cannot be burned either, because burning also goes through `state.spend` → line 141 would fail first).

---

### Likelihood Explanation

- The attacker requires no privileged key, no governance majority, and no leaked secret.
- Valid governance VAAs are publicly observable from Wormhole relayers.
- The attacker only needs to be the first to submit a given VAA — or to front-run a legitimate submission in the Cardano mempool (straightforward since Cardano has no fee-priority ordering that would favor the legitimate submitter).
- Constructing a Cardano transaction with a self-minted token is trivial.

---

### Recommendation

Replace the output `has_nft_strict` check with the same strict flatten-and-match pattern used for the input:

```aiken
// Replace line 155 in state.ak:
// expect assets.has_nft_strict(value, id, config.state_nft)
expect [(out_id, out_asset, 1)] = assets.without_lovelace(value) |> assets.flatten()
expect out_id == id
expect out_asset == config.state_nft
```

Apply the same fix to `mint_state` (lines 89–94) to close the identical gap at initialization time. [1](#0-0) [2](#0-1) 

---

### Proof of Concept

Fuzz invariant (Aiken property test sketch):

```aiken
// For any value V satisfying has_nft_strict(V, id, state_nft),
// assert flatten(without_lovelace(V)) == [(id, state_nft, 1)]
//
// Counterexample:
let extra = assets.from_asset(#"deadbeef", "junk", 1)
let v = assets.merge(assets.from_asset(id, state_nft, 1), extra)
// has_nft_strict(v, id, state_nft) == True  ← passes
// assets.without_lovelace(v) |> assets.flatten()
//   == [(#"deadbeef", "junk", 1), (id, state_nft, 1)]  ← two elements, not one
// expect [(_, _, 1)] = ...  ← FAILS in state.spend line 141
```

A concrete on-chain exploit transaction would:
1. Spend the live state UTxO with a valid VAA redeemer.
2. Mint `1` of `attacker_policy."x"` in the same tx.
3. Pay `{id: {state_nft: 1}, attacker_policy: {"x": 1}} + minADA` to the state script address with the correct updated datum.

The validator accepts it (line 155 passes); the state UTxO is now permanently frozen.
