Audit Report

## Title
Unbounded `pending_delivered` HashMap Growth via Distinct `from` PeerIds in `ConnectionRequest` ‚Äî (`network/src/protocols/hole_punching/mod.rs`)

## Summary

An unprivileged remote peer can exhaust a CKB node's memory by sending `ConnectionRequest` messages addressed to the victim (`to == self_peer_id`) with a distinct, attacker-generated `from` PeerId per message. The per-`from` deduplication guard (`HOLE_PUNCHING_INTERVAL`) is keyed on `from`, so each fresh `from` bypasses it and unconditionally inserts a new entry into `pending_delivered`. The map has no size cap and is only cleaned up every 5 minutes, enabling sustained, linear memory growth that can OOM a production node.

## Finding Description

**Outer rate limiter** (`received()`, `mod.rs` L95‚Äì107): keyed on `(session_id, msg.item_id())`, capping a single connection at 30 `ConnectionRequest` messages per second. This does not bound the number of distinct `from` PeerIds that can be inserted. [1](#0-0) 

**`forward_rate_limiter`** (`execute()`, `connection_request.rs` L132‚Äì143): keyed on `(from, to, msg_item_id)`. With a fresh `from` PeerId per message, every message gets its own rate-limiter bucket ‚Äî this check provides zero protection against the attack. [2](#0-1) 

**Routing branch** (`execute()`, L145‚Äì146): when `self_peer_id == &content.to`, `respond_delivered()` is called directly with the attacker-controlled `from` and `listen_addrs`. [3](#0-2) 

**Deduplication guard** (`respond_delivered()`, L161‚Äì167): only fires when the *same* `from` PeerId is seen again within `HOLE_PUNCHING_INTERVAL`. A fresh `from` always passes. [4](#0-3) 

**Address filter** (L196‚Äì215): filters to TCP/IPv4/IPv6 only. The attacker trivially satisfies this with `/ip4/1.2.3.4/tcp/1234`. If non-empty, execution falls through to the insert. [5](#0-4) 

**Unconditional insert** (L234‚Äì237): after a successful `send_message_to` back to the attacker's session, the entry is inserted with no size check. [6](#0-5) 

**No size cap on `pending_delivered`** (`mod.rs` L44): plain `HashMap` with no maximum entry count. [7](#0-6) 

**Cleanup only every 5 minutes** (`notify()`, L169‚Äì175): `CHECK_INTERVAL` is 5 minutes and `TIMEOUT` is also 5 minutes, so entries accumulate for the full window before any eviction. [8](#0-7) [9](#0-8) 

**Secondary unbounded structure ‚Äî `forward_rate_limiter`**: `retain_recent()` is called only on `disconnected`, never periodically. Each distinct `(from, to, msg_item_id)` triple permanently occupies a slot in the `HashMapStateStore` until the session closes. [10](#0-9) 

## Impact Explanation

**Impact: High ‚Äî "Vulnerabilities which could easily crash a CKB node."**

Memory math (conservative, single connection):
- Rate: 30 insertions/sec (outer rate limiter)
- Window: 300 s (5-minute cleanup interval)
- Entries per connection: 9,000
- Per entry: PeerId key (~39 B) + `Vec<Multiaddr>` (up to 24 √ó ~60 B) + `u64` ‚âà 1,500 B
- Memory per connection: ~13.5 MB

With default max inbound connections (`max_peers - max_outbound_peers = 125 - 8 = 117`):
- Total entries: 117 √ó 9,000 = ~1,053,000
- Total memory: ~1.58 GB ‚Äî sufficient to OOM a typical production node

The attack is sustained: memory grows continuously every 5-minute window as long as the attacker maintains connections and rotates `from` PeerIds.

## Likelihood Explanation

- No authentication or proof-of-work required ‚Äî any peer that can open a TCP connection can execute this.
- `HolePunching` is enabled by default in `support_protocols` in `resource/ckb.toml`.
- Generating fresh valid `PeerId` values is trivial (generate a secp256k1 keypair, derive the PeerId).
- Even a single connection at 30/sec √ó 300 s = 9,000 entries √ó 1,500 B = ~13.5 MB per 5-minute cycle, growing without bound as long as the connection is held.
- The attack is locally reproducible with a simple loop.

## Recommendation

1. **Cap `pending_delivered` size**: Enforce a hard limit (e.g., 1,024 entries). Reject new insertions when the cap is reached, or evict the oldest entry (LRU).
2. **Cap `inflight_requests` size** similarly.
3. **Periodic `forward_rate_limiter` cleanup**: Call `retain_recent()` inside `notify()`, not only on `disconnected`, to prevent unbounded growth of the rate-limiter's internal `HashMapStateStore`.
4. **Per-session insertion budget**: Track how many `pending_delivered` entries originated from each session and evict them on disconnect.

## Proof of Concept

```rust
// Pseudocode ‚Äî run against a victim node with HolePunching enabled
for i in 0..N {
    let from_peer_id = generate_fresh_secp256k1_keypair().peer_id();
    let listen_addrs = vec!["/ip4/1.2.3.4/tcp/1234"; 24]; // valid TCP/IPv4
    let msg = ConnectionRequest {
        from: from_peer_id,
        to:   victim_peer_id,   // self_peer_id of target node
        max_hops: 6,
        route: [],
        listen_addrs,
    };
    send_to_victim(msg); // throttled to 30/sec by outer rate limiter
}
// Assert: victim's pending_delivered.len() grows linearly with N
// Assert: victim RSS grows by ~1,500 * N bytes
// After 300 s: ~9,000 entries, ~13.5 MB per attacker connection
// With 117 connections: ~1.58 GB ‚Üí OOM
```

Each iteration bypasses the `HOLE_PUNCHING_INTERVAL` guard (new `from` key), bypasses the `forward_rate_limiter` (new `(from, to)` pair), and is only throttled by the 30 req/sec outer limiter ‚Äî giving sustained, measurable, linear memory growth.
