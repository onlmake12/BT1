### Title
Unbounded `while` Loop in `executeCallback` Can Cause Out-of-Gas, Permanently Locking Requester Funds — (`Echo.sol`)

---

### Summary

The `executeCallback` function in `Echo.sol` contains an unbounded `while` loop that advances `_state.firstUnfulfilledSeq` by scanning forward through all fulfilled (inactive) sequence numbers. When a large number of requests have been fulfilled out of order, the loop must perform a storage read per sequence number, consuming gas proportional to the gap between `firstUnfulfilledSeq` and the next active request. If this gap is large enough, the loop exhausts the block gas limit, causing `executeCallback` to revert and permanently locking the requester's funds in the contract.

---

### Finding Description

In `Echo.sol`, after clearing a fulfilled request, `executeCallback` runs the following loop:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration of this loop calls `findRequest`, which performs at least one `SLOAD` against the fixed-size `Request[32]` array and potentially a second `SLOAD` against the `requestsOverflow` mapping: [2](#0-1) 

The `_state.firstUnfulfilledSeq` field is a `uint64` in storage, so each increment is also a `SSTORE`. After the first iteration warms the storage slot, subsequent iterations cost approximately 100–300 gas each (warm `SLOAD` + warm `SSTORE`). At ~300 gas/iteration and a 30M gas block limit, roughly **100,000 consecutive fulfilled-but-not-yet-scanned sequence numbers** are sufficient to exhaust the block gas limit.

The contract's own TODO comment at line 155–156 acknowledges the downstream consequence:

> "If executeCallback can revert, then funds can be permanently locked in the contract." [3](#0-2) 

Because `executeCallback` is the **only** mechanism to clear a request and release funds, a revert here leaves the requester's ETH permanently trapped.

The `EchoState` confirms there is no cap on `currentSequenceNumber` — it is a plain `uint64` that increments with every `requestPriceUpdatesWithCallback` call: [4](#0-3) 

---

### Impact Explanation

- **Permanent fund lock:** If `executeCallback` reverts due to OOG, the request is never cleared. The requester's ETH (paid as `req.fee`) cannot be recovered because there is no refund or cancellation path.
- **Provider DoS:** The assigned provider cannot earn their fee for the stuck request, and no other provider can step in to resolve it either.
- **Cascading effect:** Because `firstUnfulfilledSeq` is never advanced past the stuck request, every subsequent `executeCallback` call that would need to advance it also faces the same growing gas cost, compounding the problem over time.

---

### Likelihood Explanation

Two realistic paths exist:

1. **Organic accumulation:** A single request with sequence number `S` is never fulfilled (provider offline, callback always reverts, etc.). Meanwhile, thousands of subsequent requests are created and fulfilled normally. When `S` is eventually fulfilled, the while loop must scan through all fulfilled requests from `S` to the current frontier — a gap that grows unboundedly over the contract's lifetime.

2. **Deliberate attack:** An unprivileged attacker calls `requestPriceUpdatesWithCallback` many times (paying the fee each time), then has the provider fulfill all requests except the first. When the first request is finally fulfilled, the loop scans the entire gap. The `requestPriceUpdatesWithCallback` function is open to any caller: [5](#0-4) 

After the exclusivity period, `executeCallback` is also open to any caller, meaning the attacker can control the order of fulfillment: [6](#0-5) 

The gas benchmark test `testMultipleRequestsOutOfOrderFulfillment` in `EchoGasBenchmark.t.sol` explicitly documents that out-of-order fulfillment causes the last `executeCallback` to be the most expensive, confirming the developers are aware of the scaling problem but have not bounded it: [7](#0-6) 

---

### Recommendation

1. **Replace the linear scan with a doubly-linked list** of active requests (as the TODO comment already suggests). This makes advancing `firstUnfulfilledSeq` O(1) instead of O(gap).
2. **Alternatively, cap the number of iterations** in the while loop per call (e.g., advance at most `MAX_SCAN` steps per `executeCallback`), and allow a separate permissionless function to advance `firstUnfulfilledSeq` in batches.
3. **Add a refund/cancellation path** so that if `executeCallback` cannot be executed (e.g., due to gas constraints), the requester can reclaim their funds after a timeout.

---

### Proof of Concept

1. Attacker calls `requestPriceUpdatesWithCallback` N+1 times, obtaining sequence numbers `[1, 2, 3, ..., N+1]`.
2. The provider (or attacker after the exclusivity period) calls `executeCallback` for sequence numbers `[2, 3, ..., N+1]` — all except sequence number `1`.
3. After each of these calls, the while loop terminates immediately because `firstUnfulfilledSeq == 1` and request `1` is still active.
4. When `executeCallback` is finally called for sequence number `1`, the while loop must iterate from `firstUnfulfilledSeq = 1` through all N fulfilled sequence numbers before finding the next active request (or reaching `currentSequenceNumber`).
5. With N ≈ 100,000, the loop consumes ~30M gas, exceeding the block gas limit and causing the transaction to revert.
6. Request `1`'s funds are now permanently locked: `executeCallback` always reverts for it, and there is no other exit path. [8](#0-7)
