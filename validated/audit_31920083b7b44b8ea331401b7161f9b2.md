Based on my investigation of the actual code, this is a real vulnerability.

**Key findings:**

**1. The fee check is one-sided (line 274):** [1](#0-0) 

Only `base_update_fee <= coin::value(&fee)` is enforced — no upper bound. The entire `fee` coin is then passed to `deposit_fee_coins`.

**2. `deposit_fee_coins` absorbs the full coin value with no change returned:** [2](#0-1) 

The function takes `fee_coins: Coin<SUI>` by value, joins it entirely into the stored balance. No splitting, no change.

**3. No withdrawal function exists anywhere in the Pyth Sui contracts:**


`get_balance` exists (line 97) to read the stored amount, but there is no corresponding `withdraw_fee_coins` or any function that reduces the balance. The stored `Coin<SUI>` is permanently locked in the `PriceInfoObject` dynamic object field.

**4. `deposit_fee_coins` is `public`, not `public(friend)`:** [3](#0-2) 

Anyone can call it directly to deposit SUI into any `PriceInfoObject`, compounding the lock-in risk.

### Title
Excess fee permanently locked in `PriceInfoObject` due to missing coin split and no withdrawal path — (`target_chains/sui/contracts/sources/pyth.move`, `price_info.move`)

### Summary
`update_single_price_feed` accepts any `Coin<SUI>` with value ≥ `base_update_fee` and deposits the **entire** coin into `PriceInfoObject`. No change is returned and no withdrawal function exists, permanently locking any excess above the required fee.

### Finding Description
In `update_single_price_feed` (pyth.move:274–277), the fee guard only checks `base_update_fee <= coin::value(&fee)`. The full coin is then passed to `price_info::deposit_fee_coins`, which joins it entirely into a dynamic object field on `PriceInfoObject` via `coin::join`. There is no `split` call, no change returned to the caller, and no withdrawal function anywhere in the Pyth Sui contract suite. The `get_balance` accessor confirms the balance accumulates but provides no path to reduce it.

### Impact Explanation
Any SUI sent in excess of `base_update_fee` is permanently frozen inside the `PriceInfoObject`. With `base_update_fee` potentially being a small amount (e.g., 1 MIST), a user sending a standard wallet coin of 1 SUI loses `1 SUI - base_update_fee` with no recovery path. The funds are not stolen by an attacker but are irrecoverably locked in the object.

### Likelihood Explanation
Moderate. Sui wallets and integrators commonly pass whole coin objects rather than pre-splitting to exact amounts. Any caller who does not manually split their coin to exactly `base_update_fee` before calling `update_single_price_feed` will lose the excess. The public visibility of `deposit_fee_coins` also allows direct griefing deposits.

### Recommendation
Before depositing, split the coin to exactly `base_update_fee` and return the remainder to the caller:
```move
let exact_fee = coin::split(&mut fee, state::get_base_update_fee(pyth_state), ctx);
price_info::deposit_fee_coins(price_info_object, exact_fee);
transfer::public_transfer(fee, tx_context::sender(ctx));
```
Alternatively, enforce `coin::value(&fee) == base_update_fee` exactly.

### Proof of Concept
1. Obtain a `Coin<SUI>` with value = `base_update_fee * 100`.
2. Call `update_single_price_feed(pyth_state, price_updates, price_info_object, fee, clock)`.
3. Assert `price_info::get_balance(price_info_object) == base_update_fee * 100`.
4. Confirm no function in the module can reduce this balance — the excess `base_update_fee * 99` is permanently locked.
