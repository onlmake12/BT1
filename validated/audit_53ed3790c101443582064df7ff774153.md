### Title
Arbitrary `providerToCredit` in `executeCallback` Allows Fee Theft After Exclusivity Period — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary
`Echo.executeCallback` accepts a caller-supplied `providerToCredit` address and credits the entire request fee to it. During the exclusivity window the parameter is correctly pinned to `req.provider`, but once that window expires the check is dropped entirely. Any unprivileged actor can therefore call `executeCallback` after the exclusivity period, name themselves (or any address they control) as `providerToCredit`, and redirect the fee that the original provider earned.

---

### Finding Description
`executeCallback` enforces provider identity only inside the exclusivity window:

```solidity
// Echo.sol lines 114-121
if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
    require(
        providerToCredit == req.provider,
        "Only assigned provider during exclusivity period"
    );
}
```

After that window closes, `providerToCredit` is completely unconstrained. The fee is then unconditionally credited to whatever address the caller supplied:

```solidity
// Echo.sol lines 161-162
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

`req.fee` was funded by the original requester at request time and was intended to compensate the assigned provider. Because `providerToCredit` is user-controlled post-exclusivity, the caller can redirect that value to any registered provider account they own.

The withdrawal path is fully permissionless for a registered provider:

```solidity
// Echo.sol lines 381-393  (registerProvider — no access control)
// Echo.sol lines 350-357  (setFeeManager — sets msg.sender's fee manager)
// Echo.sol lines 360-379  (withdrawAsFeeManager — withdraws to msg.sender)
```

---

### Impact Explanation
The original provider loses the fee they earned for a fulfilled request. The attacker gains that fee. Because `req.fee` is set at request time from `msg.value` and can be non-trivial (it includes the provider's base fee, per-feed fee, and per-gas fee), repeated exploitation across many pending requests results in direct, measurable ETH loss for legitimate providers.

---

### Likelihood Explanation
The attack requires no privileged access. `registerProvider` is permissionless. The attacker only needs to:
1. Register once as a provider.
2. Set themselves as their own fee manager (`setFeeManager`).
3. Monitor the mempool or on-chain state for requests whose exclusivity period has elapsed.
4. Front-run or race the legitimate provider's `executeCallback` call.

The exclusivity period is a single configurable integer (`exclusivityPeriodSeconds`). Any request that the legitimate provider is slow to fulfill — or any request submitted with a past `publishTime` — is immediately exploitable.

---

### Recommendation
After the exclusivity period, credit the fee to `msg.sender` (the actual executor) rather than to a caller-supplied address. This mirrors the correct pattern used during the exclusivity window and eliminates the ability to redirect fees:

```solidity
address recipient = (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds)
    ? req.provider
    : msg.sender;   // <-- use msg.sender, not providerToCredit
_state.providers[recipient].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
```

If the design intent is to allow a relay to credit a third-party provider, the parameter should at minimum be restricted to `req.provider` at all times, or a separate, explicitly authorized delegation mechanism should be used.

---

### Proof of Concept

1. **Setup** — Attacker calls `registerProvider(0, 0, 0)` to create a provider entry, then calls `setFeeManager(attackerAddress)` so they can later withdraw.

2. **Victim request** — A user calls `requestPriceUpdatesWithCallback(legitimateProvider, publishTime, priceIds, gasLimit)` paying a fee `F`. The contract stores `req.fee = F - pythFeeInWei` and `req.provider = legitimateProvider`.

3. **Wait** — Attacker waits until `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`.

4. **Exploit** — Attacker calls:
   ```solidity
   executeCallback(
       attackerAddress,   // providerToCredit — no longer restricted
       sequenceNumber,
       validUpdateData,
       priceIds
   );
   ```
   `_state.providers[attackerAddress].accruedFeesInWei` is incremented by `req.fee + msg.value - pythFee`. `legitimateProvider` receives nothing.

5. **Withdraw** — Attacker calls `withdrawAsFeeManager(attackerAddress, stolenAmount)`, transferring the stolen ETH to themselves. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

    function setFeeManager(address manager) external override {
        require(
            _state.providers[msg.sender].isRegistered,
            "Provider not registered"
        );
        address oldFeeManager = _state.providers[msg.sender].feeManager;
        _state.providers[msg.sender].feeManager = manager;
        emit FeeManagerUpdated(msg.sender, oldFeeManager, manager);
    }

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        require(
            msg.sender == _state.providers[provider].feeManager,
            "Only fee manager"
        );
        require(
            _state.providers[provider].accruedFeesInWei >= amount,
            "Insufficient balance"
        );

        _state.providers[provider].accruedFeesInWei -= amount;

        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "Failed to send fees");

        emit FeesWithdrawn(msg.sender, amount);
