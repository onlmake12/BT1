Audit Report

## Title
Admission Fee Check Uses Size-Only Weight While Pool Accounting Uses `get_transaction_weight(size, cycles)`, Allowing `min_fee_rate` Bypass for Cycle-Heavy Transactions - (File: `tx-pool/src/util.rs`, `tx-pool/src/component/entry.rs`)

## Summary

`check_tx_fee` gates pool admission using `min_fee_rate.fee(tx_size)` — a size-only formula — while every downstream accounting path (`fee_rate()`, `AncestorsScoreSortKey`, `EvictKey`, fee estimators) uses `get_transaction_weight(size, cycles) = max(size, cycles × 0.000_170_571_4)`. For cycle-heavy transactions the two formulas diverge by up to ~60×, allowing a transaction whose true weight-based fee rate is far below `min_fee_rate` to pass the admission gate. The discrepancy is explicitly acknowledged in a code comment but left unresolved.

## Finding Description

**Admission gate** — `check_tx_fee` in `tx-pool/src/util.rs` lines 42–52:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { return Err(Reject::LowFeeRate(...)); }
```

The gate enforces: `fee ≥ min_fee_rate × tx_size / 1000`. [1](#0-0) 

**Pool fee rate** — `TxEntry::fee_rate()` in `tx-pool/src/component/entry.rs` lines 114–118:

```rust
pub fn fee_rate(&self) -> FeeRate {
    let weight = get_transaction_weight(self.size, self.cycles);
    FeeRate::calculate(self.fee, weight)
}
``` [2](#0-1) 

`get_transaction_weight` returns `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`: [3](#0-2) 

`FeeRate::fee` and `FeeRate::calculate` are inverses only when the same weight is used in both directions: [4](#0-3) 

The same weight formula is used consistently in `AncestorsScoreSortKey`, `EvictKey`, and fee estimators — but not at the admission gate: [5](#0-4) 

**Root cause**: When `cycles × 0.000_170_571_4 > tx_size`, the true weight exceeds `tx_size`, so the size-based admission threshold is strictly weaker than the weight-based threshold used everywhere else. A transaction can satisfy `fee ≥ min_fee_rate × size / 1000` while having a true fee rate of `fee × 1000 / weight` far below `min_fee_rate`.

## Impact Explanation

An attacker can flood the mempool with transactions whose true weight-based fee rate is far below the configured `min_fee_rate`, bypassing the anti-spam policy. With default config (`min_fee_rate = 1000 shannons/KW`, `max_tx_pool_size = 180 MB`, `max_tx_verify_cycles = 70,000,000`):

| Parameter | Value |
|---|---|
| `tx_size` | 200 bytes |
| `cycles` | 70,000,000 |
| `weight` | max(200, 11,940) = **11,940** |
| Admission requires | 1000 × 200 / 1000 = **200 shannons** |
| True fee rate | 200 × 1000 / 11,940 ≈ **16.7 shannons/KW** |

A 200-shannon fee passes admission while the true fee rate is ~60× below `min_fee_rate`. The attacker can fill the 180 MB pool with such entries (each ~200 bytes → ~900,000 entries), displacing legitimate transactions. The eviction logic (`EvictKey`) uses the correct weight-based fee rate, so these entries have the lowest eviction priority among same-fee-rate entries, worsening displacement.

This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

The attack is reachable by any unprivileged user via the `send_transaction` JSON-RPC endpoint or P2P relay. The only prerequisite is deploying a cycle-heavy RISC-V script on-chain (a one-time cost). After deployment, each attack transaction costs only ~200 shannons. No special privileges, keys, or hash power are required. The bypass is maximally effective at the cycle limit (70M cycles), which is the easiest case to target and yields the largest weight-to-size ratio.

## Recommendation

Pass `cycles` into `check_tx_fee` and compute weight consistently with the rest of the pool:

```rust
pub(crate) fn check_tx_fee(
    tx_pool: &TxPool,
    snapshot: &Snapshot,
    rtx: &ResolvedTransaction,
    tx_size: usize,
    cycles: Cycle,          // add this parameter
) -> Result<Capacity, Reject> {
    let fee = /* ... */;
    let weight = get_transaction_weight(tx_size, cycles);
    let min_fee = tx_pool.config.min_fee_rate.fee(weight);
    if fee < min_fee {
        return Err(Reject::LowFeeRate(...));
    }
    Ok(fee)
}
```

This aligns the admission gate with `TxEntry::fee_rate()`, `AncestorsScoreSortKey`, `EvictKey`, and the fee estimators, eliminating the bypass.

## Proof of Concept

1. Deploy a CKB RISC-V script on-chain that loops until it consumes ~70,000,000 cycles. Keep the script cell small; reference it via `code_hash` to minimize transaction serialized size.
2. Construct a transaction using that script as the lock, with `tx_size ≈ 200 bytes` and `inputs_capacity − outputs_capacity = 200 shannons`.
3. Submit via `send_transaction` RPC to a node with default `min_fee_rate = 1000`.
4. Observe acceptance: size-based check `200 ≥ 1000 × 200 / 1000 = 200` passes exactly.
5. Call `get_pool_tx_detail_info` and confirm the entry's actual fee rate ≈ 16.7 shannons/KW — ~60× below `min_fee_rate`.
6. Repeat in a loop to fill the pool with sub-minimum-fee-rate entries and observe displacement of legitimate transactions.
