### Title
OOG-Exploitable try-catch in `executeCallback` Permanently Clears Requests Without Delivering Callbacks - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback` performs all critical state mutations — crediting the provider's fees and clearing the request — **before** the try-catch that invokes the consumer's callback. A bare `catch {}` block silently absorbs out-of-gas (OOG) errors from the callback. An attacker (or malicious provider) can call `executeCallback` with a crafted gas amount that is sufficient to complete the pre-callback state changes but causes the forwarded callback call to run out of gas. The transaction succeeds, the provider is paid, the request is permanently cleared, and the consumer's callback is never executed — resulting in permanent loss of funds for the consumer.

---

### Finding Description

In `Echo.executeCallback`, the following sequence occurs:

1. **Fee is credited to the provider** (line 161–162)
2. **Request is cleared** via `clearRequest(sequenceNumber)` (line 164)
3. **`firstUnfulfilledSeq` is advanced** in a `while` loop (lines 169–174)
4. **The callback is attempted** inside a try-catch (lines 176–201)

```solidity
// EFFECTS happen BEFORE the try-catch:
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);   // provider paid

clearRequest(sequenceNumber);                        // request permanently deleted

while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}

// THEN the callback:
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
{
    emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
} catch Error(string memory reason) {
    emit PriceUpdateCallbackFailed(..., reason);
} catch {
    // Out of gas or other low-level errors — SILENT ABSORPTION
    emit PriceUpdateCallbackFailed(..., "low-level error (possibly out of gas)");
}
```

Per EIP-150, the `CALL` opcode forwards at most `63/64` of the remaining gas to the callee. If the caller of `executeCallback` provides a gas amount that is:
- Large enough to complete all pre-callback state changes (fee credit, `clearRequest`, `firstUnfulfilledSeq` loop), **and**
- Small enough that `(63/64) * gasleft_at_try` < `req.callbackGasLimit`

then the callback will OOG. The bare `catch {}` absorbs this silently. The transaction succeeds. The request is already cleared and cannot be retried. The provider has already been credited.

There is **no on-chain enforcement** of the minimum gas requirement. The `IEcho.sol` NatSpec comment states "Requires 1.5x the callback gas limit" but this is documentation only — the contract never checks `gasleft()` before the try-catch. [1](#0-0) 

The developer's own TODO comment at line 155 acknowledges the related concern:

> "if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."

The actual vulnerability is the inverse: if the callback silently OOGs and the catch block does not revert, the consumer loses funds without receiving the callback. [2](#0-1) 

Compare this to the **Entropy contract**, which uses `ExcessivelySafeCall` and explicitly checks whether sufficient gas was available before the call, reverting with `InsufficientGas()` if not:

```solidity
} else {
    revert EntropyErrors.InsufficientGas();
}
``` [3](#0-2) 

Echo has no equivalent protection. [4](#0-3) 

---

### Impact Explanation

- **Permanent loss of consumer funds**: The consumer paid `req.fee` (stored at request time) for a callback that is never delivered. The request is cleared and cannot be retried.
- **Provider unjust enrichment**: The provider's `accruedFeesInWei` is incremented before the callback attempt. The provider collects fees without delivering the service.
- **Irreversible**: Once `clearRequest` executes, the sequence number slot is freed and the request data is gone. There is no recovery path. [5](#0-4) 

---

### Likelihood Explanation

- `executeCallback` is **permissionless** — anyone can call it after the exclusivity period expires.
- A malicious provider can call it during the exclusivity period with a crafted gas limit.
- The gas amount needed to trigger the exploit is predictable: enough to cover the pre-callback logic (~50k–100k gas for storage writes + the `while` loop), but insufficient to forward the full `callbackGasLimit` to the callback.
- The `callbackGasLimit` is set by the requester and stored in the request, so the attacker can read it on-chain and compute the exact gas to supply.
- No special privileges are required. [6](#0-5) 

---

### Recommendation

Add a gas sufficiency check before the try-catch, mirroring the Entropy contract's approach:

```solidity
// Before the try-catch, verify sufficient gas is available to forward callbackGasLimit
// Per EIP-150, at most 63/64 of gasleft() is forwarded. Use 31/32 for a safety margin.
require(
    (gasleft() * 31) / 32 >= req.callbackGasLimit,
    "Insufficient gas for callback"
);
```

Alternatively, restructure so that state mutations (fee credit and `clearRequest`) only occur **after** a successful callback, or implement a retry mechanism (similar to Entropy's `CALLBACK_FAILED` state) that preserves the request if the callback OOGs. [7](#0-6) 

---

### Proof of Concept

1. Alice deploys a consumer contract implementing `IEchoConsumer` with an `echoCallback` that uses ~500,000 gas. She calls `requestPriceUpdatesWithCallback` with `callbackGasLimit = 500_000`, paying the required fee.

2. Bob (a malicious provider or any caller after exclusivity) observes the request on-chain. He reads `req.callbackGasLimit = 500_000` and `req.fee`.

3. Bob calls `executeCallback` with a total gas of ~580,000:
   - ~80,000 gas is consumed by pre-callback logic (Pyth fee parsing, storage writes for fee credit, `clearRequest`, `firstUnfulfilledSeq` loop).
   - ~500,000 gas remains at the try-catch. Per EIP-150, at most `(63/64) * 500,000 ≈ 492,187` gas is forwarded — less than the 500,000 needed.
   - Alice's callback OOGs.

4. The bare `catch {}` emits `PriceUpdateCallbackFailed` and the transaction succeeds.

5. Result: Bob's provider address is credited `req.fee` wei. Alice's request is cleared. Alice's callback was never executed. Alice has permanently lost her funds. [8](#0-7) [9](#0-8)

    function executeCallback(
        address providerToCredit,
        uint64 sequenceNumber,
        bytes[] calldata updateData,
        bytes32[] calldata priceIds
    ) external payable override {
        Request storage req = findActiveRequest(sequenceNumber);

        // Check provider exclusivity using configurable period
        if (
            block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds
        ) {
            require(
                providerToCredit == req.provider,
                "Only assigned provider during exclusivity period"
            );
        }

        // Verify priceIds match
        require(
            priceIds.length == req.priceIdPrefixes.length,
            "Price IDs length mismatch"
        );
        for (uint8 i = 0; i < req.priceIdPrefixes.length; i++) {
            // Extract first 8 bytes of the provided price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }

            // Compare with stored prefix
            if (prefix != req.priceIdPrefixes[i]) {
                // Now we can directly use the bytes8 prefix in the error
                revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
            }
        }

        // TODO: should this use parsePriceFeedUpdatesUnique? also, do we need to add 1 to maxPublishTime?
        IPyth pyth = IPyth(_state.pyth);
        uint256 pythFee = pyth.getUpdateFee(updateData);
        PythStructs.PriceFeed[] memory priceFeeds = pyth.parsePriceFeedUpdates{
            value: pythFee
        }(
            updateData,
            priceIds,
            SafeCast.toUint64(req.publishTime),
            SafeCast.toUint64(req.publishTime)
        );

        // TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert.
        // If executeCallback can revert, then funds can be permanently locked in the contract.
        // TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback.
        // This should take funds from the expected provider and give to providerToCredit. The penalty should probably scale
        // with time in order to ensure that the callback eventually gets executed.
        // (There may be exploits with ^ though if the consumer contract is malicious ?)
        _state.providers[providerToCredit].accruedFeesInWei += SafeCast
            .toUint128((req.fee + msg.value) - pythFee);

        clearRequest(sequenceNumber);

        // TODO: I'm pretty sure this is going to use a lot of gas because it's doing a storage lookup for each sequence number.
        // a better solution would be a doubly-linked list of active requests.
        // After successful callback, update firstUnfulfilledSeq if needed
        while (
            _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
            !isActive(findRequest(_state.firstUnfulfilledSeq))
        ) {
            _state.firstUnfulfilledSeq++;
        }

        try
            IEchoConsumer(req.requester)._echoCallback{
                gas: req.callbackGasLimit
            }(sequenceNumber, priceFeeds)
        {
            // Callback succeeded
            emitPriceUpdate(sequenceNumber, priceIds, priceFeeds);
        } catch Error(string memory reason) {
            // Explicit revert/require
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                reason
            );
        } catch {
            // Out of gas or other low-level errors
            emit PriceUpdateCallbackFailed(
                sequenceNumber,
                providerToCredit,
                priceIds,
                req.requester,
                "low-level error (possibly out of gas)"
            );
        }
