### Title
Missing Transaction Confirmation Timeout in `send_and_confirm` Causes Indefinite Keeper Loop Hang - (File: apps/fortuna/src/eth_utils/utils.rs)

### Summary

Fortuna's `send_and_confirm` utility function awaits transaction confirmation with **no timeout**, unlike the main entropy fulfillment path (`submit_tx`) which correctly applies a 30-second `TX_CONFIRMATION_TIMEOUT_SECS` guard. If a transaction submitted via `send_and_confirm` gets stuck in the mempool (e.g., due to network congestion or gas price issues), the calling keeper loop hangs indefinitely, permanently blocking commitment advancement and fee management operations.

### Finding Description

In `apps/fortuna/src/eth_utils/utils.rs`, two distinct transaction submission paths exist:

**Path 1 — `submit_tx` (entropy fulfillment, has timeout):** [1](#0-0) [2](#0-1) 

`submit_tx` wraps `pending_tx` in `tokio::time::timeout(Duration::from_secs(30), ...)` and resets the nonce manager on timeout.

**Path 2 — `send_and_confirm` (keeper maintenance, NO timeout):** [3](#0-2) 

`send_and_confirm` calls `pending_tx.await` with no timeout guard whatsoever. If the transaction is stuck in the mempool, this `await` never resolves.

`send_and_confirm` is called in three keeper maintenance paths:

1. **Commitment advancement** — `advance_provider_commitment` in `update_commitments_if_necessary`: [4](#0-3) 

2. **Fee withdrawal** — `withdraw_as_fee_manager` in `withdraw_fees_if_necessary`: [5](#0-4) 

3. **Fee adjustment** — `set_provider_fee_as_fee_manager` in `adjust_fee_if_necessary`: [6](#0-5) 

### Impact Explanation

**Commitment update loop hang:** The `update_commitments_loop` runs every 30 seconds and calls `send_and_confirm` when outstanding requests exceed 95% of `max_num_hashes`. [7](#0-6) 

If the `advance_provider_commitment` transaction gets stuck, the loop hangs permanently. The provider's hash chain is never advanced, causing `numHashes` to grow unboundedly for future reveals, increasing gas costs until the provider's `endSequenceNumber` is reached and no new requests can be served.

**Fee management loop hang:** If `withdraw_as_fee_manager` or `set_provider_fee_as_fee_manager` gets stuck, the fee withdrawal loop hangs. The keeper wallet balance is never replenished. Once the keeper balance drops below `min_keeper_balance`, the keeper stops submitting `revealWithCallback` transactions, causing entropy callbacks to go unfulfilled for all users on that chain.

### Likelihood Explanation

The commitment update is triggered when an unprivileged user spams entropy requests past the 95% threshold of `max_num_hashes`. On high-latency or congested EVM chains (e.g., Arbitrum, Blast), a transaction submitted without a gas escalation policy (as `send_and_confirm` does) can remain stuck in the mempool indefinitely. This is a realistic scenario under normal network congestion, not requiring any privileged access.

### Recommendation

Apply the same `timeout(Duration::from_secs(TX_CONFIRMATION_TIMEOUT_SECS), ...)` guard used in `submit_tx` to `send_and_confirm`. Additionally, consider using `submit_tx_with_backoff` (which includes nonce reset and fee escalation) for all keeper maintenance transactions, not just entropy fulfillment:

```rust
pub async fn send_and_confirm<A: Middleware + NonceManaged + 'static>(
    contract_call: ContractCall<A, ()>,
) -> Result<()> {
    let call_name = contract_call.function.name.as_str();
    let pending_tx = contract_call
        .send()
        .await
        .map_err(|e| anyhow!("Error submitting transaction({}) {:?}", call_name, e))?;

    let tx_result = timeout(Duration::from_secs(TX_CONFIRMATION_TIMEOUT_SECS), pending_tx)
        .await
        .map_err(|_| anyhow!("Timeout waiting for transaction({})", call_name))?
        ...
}
```

### Proof of Concept

1. Deploy Fortuna keeper on a congested EVM chain (e.g., Arbitrum under load).
2. Spam entropy requests until `outstanding_requests > 0.95 * max_num_hashes`, triggering `update_commitments_if_necessary`.
3. Ensure network congestion causes the `advance_provider_commitment` transaction to remain pending in the mempool beyond any reasonable time.
4. Observe that `send_and_confirm` at line 69 of `commitment.rs` never returns — the `update_commitments_loop` is permanently blocked.
5. The provider's commitment is never advanced; gas costs for future reveals grow until the chain is exhausted.

/// Send a transaction and wait for the receipt to ensure that it was confirmed on chain.
pub async fn send_and_confirm<A: Middleware>(contract_call: ContractCall<A, ()>) -> Result<()> {
    let call_name = contract_call.function.name.as_str();
    let pending_tx = contract_call
        .send()
        .await
        .map_err(|e| anyhow!("Error submitting transaction({}) {:?}", call_name, e))?;

    let tx_result = pending_tx
        .await
        .map_err(|e| {
            anyhow!(
                "Error waiting for transaction({}) receipt: {:?}",
                call_name,
                e
            )
        })?
        .ok_or_else(|| {
            anyhow!(
                "Can't verify the transaction({}), probably dropped from mempool",
                call_name
            )
        })?;
