### Title
Integer Division Truncation in Soft Fork Activation Threshold Allows Lock-In Below Configured Ratio - (File: `spec/src/versionbits/mod.rs`)

### Summary

The `threshold_number` function in `spec/src/versionbits/mod.rs` computes the minimum number of signaling blocks required to lock in a soft fork using integer (floor) division: `length * numer / denom`. Because Rust integer division truncates toward zero, the computed threshold is always rounded **down**, meaning a miner coalition can achieve `LockedIn` state with a block-signal ratio strictly less than the configured activation threshold (e.g., 80% on mainnet).

### Finding Description

**Root cause — `threshold_number` (lines 475–479):**

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| ret.checked_div(threshold.denom()))
}
```

This computes `floor(length × numer / denom)`. The result is used directly as the minimum signal count:

```rust
// spec/src/versionbits/mod.rs lines 342–344
let threshold_number = threshold_number(total, self.threshold())?;
if count >= threshold_number {
    next_state = ThresholdState::LockedIn;
```

The mainnet activation threshold is `Ratio::new(8, 10)` (80%) and the testnet threshold is `Ratio::new(3, 4)` (75%):

```rust
// spec/src/consensus.rs lines 99–101
pub const LC_MAINNET_ACTIVATION_THRESHOLD: Ratio = Ratio::new(8, 10);
pub const TESTNET_ACTIVATION_THRESHOLD: Ratio = Ratio::new(3, 4);
```

**Exploit path:**

The `total` block count is the sum of block lengths across `period` epochs (lines 326–340). CKB epoch lengths are variable and not guaranteed to be multiples of the threshold denominator. Whenever `total` is not divisible by `denom`, the floor division produces a threshold that is one block lower than the true ceiling, allowing lock-in with a ratio strictly below the configured threshold.

### Impact Explanation

**Concrete example (mainnet, threshold = 8/10):**

| Variable | Value |
|---|---|
| `total` | 1801 blocks |
| True 80% (ceiling) | `⌈1801 × 8 / 10⌉ = 1441` |
| `threshold_number` (floor) | `⌊1801 × 8 / 10⌋ = 1440` |
| Effective ratio | `1440 / 1801 = 79.956%` |

A miner coalition holding 79.956% of hash power — below the 80% governance threshold — can lock in a soft fork. The deviation is bounded by `(denom − 1) / (total × denom)`, which is small per period but is a real, systematic under-enforcement of the configured threshold. Any soft fork governed by this mechanism (RFC-0043 / versionbits) is affected, including future protocol upgrades.

### Likelihood Explanation

The CKB epoch length is dynamically adjusted by the consensus algorithm and is almost never an exact multiple of the threshold denominator (10 for mainnet). Therefore, truncation occurs in virtually every signaling period. A miner coalition that is aware of this can deliberately target periods where `total mod denom ≠ 0` to exploit the one-block reduction. The attacker role (miner) is explicitly in scope per the bounty rules.

### Recommendation

Replace floor division with ceiling division in `threshold_number`:

```rust
fn threshold_number(length: u64, threshold: Ratio) -> Option<u64> {
    // Use ceiling division: (length * numer + denom - 1) / denom
    length
        .checked_mul(threshold.numer())
        .and_then(|ret| ret.checked_add(threshold.denom().saturating_sub(1)))
        .and_then(|ret| ret.checked_div(threshold.denom()))
}
```

This ensures that `count >= threshold_number` is only satisfied when the true ratio is at or above the configured threshold, matching the intent of the governance parameter.

### Proof of Concept

**Setup:** Mainnet deployment, `threshold = Ratio::new(8, 10)`, one signaling period where epoch lengths sum to `total = 1801`.

**Step 1 — Current behavior:**
```
threshold_number(1801, Ratio::new(8, 10))
= floor(1801 * 8 / 10)
= floor(14408 / 10)
= floor(1440.8)
= 1440
```

**Step 2 — Attacker action:** A miner coalition controlling 1440 out of 1801 blocks (79.956%) sets the version bit in their blocks.

**Step 3 — Check at line 343:**
```
count (1440) >= threshold_number (1440)  →  true  →  LockedIn
```

**Step 4 — Expected behavior with ceiling division:**
```
threshold_number = ceil(1801 * 8 / 10) = ceil(1440.8) = 1441
count (1440) >= threshold_number (1441)  →  false  →  NOT LockedIn
```
