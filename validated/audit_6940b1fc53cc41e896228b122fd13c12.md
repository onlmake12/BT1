### Title
Panic-Induced Denial-of-Service in `NotifyService` via Non-UTF-8 Network Alert Message - (File: `notify/src/lib.rs`)

### Summary

The `NotifyService` background task panics and permanently terminates when it processes a network alert whose `message` field is not valid UTF-8. The panic occurs inside `handle_notify_network_alert` via an unconditional `.expect(...)` call. Because the entire `NotifyService` event loop runs in a single spawned task, the panic kills the task, permanently disabling all event notifications (new blocks, new transactions, proposed transactions, rejected transactions, network alerts, and log entries) for the lifetime of the node. This is analogous to the reported "Denial of Withdrawal" pattern: a single malformed item in a processing queue causes the entire service to become permanently frozen.

### Finding Description

In `notify/src/lib.rs`, `handle_notify_network_alert` unconditionally calls `.expect("alert message should be utf8")` on the raw alert message bytes:

```rust
fn handle_notify_network_alert(&self, alert: Alert) {
    let message = alert
        .as_reader()
        .raw()
        .message()
        .as_utf8()
        .expect("alert message should be utf8")  // <-- panics on non-UTF-8
        .to_owned();
    ...
}
``` [1](#0-0) 

This function is called from the single-threaded `NotifyService` event loop:

```rust
handle.spawn(async move {
    loop {
        tokio::select! {
            ...
            Some(msg) = network_alert_receiver.recv() => {
                self.handle_notify_network_alert(msg)
            },
            ...
        }
    }
});
``` [2](#0-1) 

A panic inside a `tokio::spawn`ed task causes that task to be silently dropped. The entire `NotifyService` loop — which handles **all** event types (new blocks, new transactions, proposed transactions, rejected transactions, network alerts, and log entries) — runs in this single task. Once it panics, all notification channels become permanently dead for the rest of the node's lifetime.

The alert relayer (`alert_relayer.rs`) does validate UTF-8 before accepting a P2P alert:

```rust
if alert.raw().message().is_utf8() ... {
    alert.to_entity()
} else {
    nc.ban_peer(...);
    return;
}
``` [3](#0-2) 

However, the `notify_network_alert` RPC path does **not** go through the P2P relayer. An RPC caller can submit an alert directly via the RPC interface. The `NotifyController::notify_network_alert` method sends the raw `packed::Alert` directly to the channel without any UTF-8 pre-validation:

```rust
pub fn notify_network_alert(&self, alert: Alert) {
    let network_alert_notifier = self.network_alert_notifier.clone();
    self.handle.spawn(async move {
        if let Err(e) = network_alert_notifier.send(alert).await {
            error!("notify_network_alert channel is closed: {}", e);
        }
    });
}
``` [4](#0-3) 

Additionally, the `Notifier::add` path in `util/network-alert/src/notifier.rs` calls `notify_controller.notify_network_alert(alert.clone())` after the alert has been stored, and the stored alert could theoretically be replayed. The core issue is that the `expect` is the only guard inside the consumer, and any code path that bypasses the P2P UTF-8 check can trigger it.

### Impact Explanation

When the panic fires, the `NotifyService` tokio task terminates permanently. All downstream consumers of the notification channels (RPC subscribers, indexer, block assembler notifications, log subscribers) stop receiving events. The node continues to process blocks and transactions internally, but:

- RPC subscription endpoints (`subscribe` method) stop delivering events.
- The block assembler's notification path via `NotifyController` is severed.
- Any component relying on `new_block_notify_script` or `network_alert_notify_script` stops executing.

This constitutes a permanent, node-lifetime denial of the notification subsystem, triggered by a single crafted alert message.

### Likelihood Explanation

The RPC `send_alert` method is accessible to any RPC caller with the `Alert` permission. A node operator who has granted RPC access (e.g., a local CLI user or a trusted RPC caller) can submit an alert with a non-UTF-8 `message` field. The P2P path is protected, but the RPC path is not. The vulnerability requires only a single malformed RPC call to permanently disable the notification service for the node's current session.

### Recommendation

Replace the `.expect(...)` call with graceful error handling. If the alert message is not valid UTF-8, log an error and return early rather than panicking:

```rust
fn handle_notify_network_alert(&self, alert: Alert) {
    let message = match alert.as_reader().raw().message().as_utf8() {
        Ok(s) => s.to_owned(),
        Err(e) => {
            error!("network alert message is not valid UTF-8: {}", e);
            return;
        }
    };
    // ... rest of function
}
``` [5](#0-4) 

### Proof of Concept

1. Start a CKB node with RPC access enabled and `Alert` permission granted.
2. Craft a `packed::Alert` where the `message` field contains bytes that are not valid UTF-8 (e.g., `[0xFF, 0xFE]`).
3. Submit the alert via the `send_alert` RPC method.
4. The `NotifyService` task receives the alert via `network_alert_receiver`, calls `handle_notify_network_alert`, hits `.expect("alert message should be utf8")`, and panics.
5. The tokio task terminates. All subsequent calls to `subscribe` RPC endpoints return no events; `new_block_notify_script` stops executing; the notification subsystem is permanently dead until the node is restarted.
