Audit Report

## Title
VerifyQueue Admits Transactions Without Fee-Rate Gating and Has No Eviction, Enabling Low-Cost DoS That Delays or Permanently Excludes Time-Sensitive Transactions — (File: `tx-pool/src/component/verify_queue.rs`)

## Summary
`VerifyQueue::add_tx()` performs no fee-rate check at admission and has no eviction mechanism. An unprivileged attacker can flood the 256 MB queue with large, zero-fee (or structurally invalid) transactions, causing all subsequent submissions to receive `Reject::Full`. Transactions already in the queue are drained in FIFO order, so even a victim transaction that entered before the cap was hit may sit behind thousands of attacker transactions and miss a `since`-encoded time-lock validity window, permanently invalidating it.

## Finding Description
`VerifyQueue::add_tx()` (lines 198–237) performs only two checks before inserting a transaction: duplicate detection and the size-based `is_full()` guard. No fee-rate is computed or compared. The `VerifyEntry` struct (lines 34–53) stores only `added_time` as a sort key — there is no fee-rate field, making fee-rate-based ordering structurally impossible at this layer.

`VerifyQueue::peek()` (lines 180–194) dequeues in `added_time` (FIFO) order, with the sole priority exception being `is_proposal_tx`. All other transactions — including high-fee, time-sensitive ones — wait behind earlier arrivals.

Because `add_tx()` does not validate transaction semantics (inputs, outputs, fees), an attacker can submit structurally well-formed but semantically invalid transactions with zero fees. These occupy queue space until dequeued and rejected by the verify worker. The `min_fee_rate` check found in `pool.rs` and `util.rs` applies only after a transaction is dequeued from the verify queue and enters the verification pipeline — it provides no admission protection.

Once the 256 MB cap is reached, `is_full()` returns `true` and every subsequent `add_tx()` call returns `Err(Reject::Full(...))` regardless of fee rate. There is no eviction path in `VerifyQueue` to displace a low-fee entry in favor of a higher-fee arrival, in contrast to `PoolMap` which uses `EvictKey` (fee-rate-first) for eviction.

The `TransactionTemplate.required` field being hardcoded to `false` (line 988) eliminates any fallback mechanism at the block-template layer to guarantee inclusion of a transaction the operator considers critical.

## Impact Explanation
This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

CKB's `since` field encodes absolute or relative time-lock conditions (block height or median-time-past). A transaction whose `since` window expires before it clears the verify queue becomes permanently invalid — it cannot be resubmitted with the same semantics after the window closes. An attacker who saturates the verify queue forces either immediate `Reject::Full` for new submissions, or indefinite FIFO delay for transactions already queued. Either outcome can cause a time-sensitive transaction to miss its validity window with no recovery path.

## Likelihood Explanation
The 256 MB cap divided by CKB's per-transaction size limit (on the order of hundreds of kilobytes) means a few hundred to a few thousand transactions saturate the queue. The attacker does not need valid UTXOs — `add_tx()` performs no semantic validation. Submissions can be made via the public `send_transaction` RPC or P2P relay without any privileged access, cryptographic material, or hash power. The attack is amplified during organic congestion when the queue is already partially full. It is repeatable: once the attacker's transactions are drained (rejected for low fees or invalid inputs), the attacker can immediately refill the queue.

## Recommendation
1. **Fee-rate gating at admission**: Before inserting into `VerifyQueue`, compute the transaction's declared fee rate and reject transactions below `min_fee_rate` immediately, consistent with the check already applied during verification.
2. **Fee-rate-based eviction in `VerifyQueue`**: When the queue is full and a new transaction with a higher fee rate arrives, evict the lowest-fee-rate entry, mirroring the `EvictKey` eviction logic in `PoolMap`.
3. **Add fee-rate as a sort key to `VerifyEntry`**: Extend the `MultiIndexMap` with a fee-rate index so that `peek()` can prioritize higher-fee transactions rather than relying solely on `added_time`.
4. **Implement `TransactionTemplate.required`**: Provide an operator-configurable mechanism to mark specific transactions as required in block templates.

## Proof of Concept
1. Attacker connects to a CKB node via RPC or P2P.
2. Attacker submits a stream of large transactions (large witness blobs, near the per-transaction size limit) with zero fees. Each call to `VerifyQueue::add_tx()` succeeds — no fee-rate check is performed — until `total_tx_size` approaches 256 MB.
3. Victim submits a time-sensitive transaction with a `since` field expiring in N blocks. `VerifyQueue::add_tx()` returns `Err(Reject::Full(...))` — the transaction never enters the pipeline. **Or**, if the victim's transaction entered before the cap was hit, `peek()` returns attacker transactions first (earlier `added_time`); verify workers drain them (rejecting for invalid inputs or low fees), but by the time the victim's transaction is dequeued, the `since` window has expired and the transaction is permanently invalid.
4. To verify mechanically: write a unit test that calls `set_total_tx_size_for_test()` (the existing test helper at line 99) to set `total_tx_size` near the cap, then call `add_tx()` with a high-fee transaction and observe `Err(Reject::Full(...))` returned despite the high fee rate.

    pub fn add_tx(
        &mut self,
        tx: TransactionView,
        is_proposal_tx: bool,
        remote: Option<(Cycle, PeerIndex)>,
    ) -> Result<bool, Reject> {
        if self.contains_key(&tx.proposal_short_id()) {
            if is_proposal_tx {
                self.remove_tx(&tx.proposal_short_id());
            } else {
                return Ok(false);
            }
        }
        let tx_size = tx.data().serialized_size_in_block();
        let is_large_cycle = remote
            .map(|(cycles, _)| cycles > self.large_cycle_threshold)
            .unwrap_or(false);
        if self.is_full(tx_size) {
            return Err(Reject::Full(format!(
                "verify_queue total_tx_size exceeded, failed to add tx: {:#x}",
                tx.hash()
            )));
        }
        let total_tx_size = self.total_tx_size.checked_add(tx_size).ok_or_else(|| {
            Reject::Full(format!(
                "verify_queue total_tx_size overflowed, failed to add tx: {:#x}",
                tx.hash()
            ))
        })?;
        self.inner.insert(VerifyEntry {
            id: tx.proposal_short_id(),
            added_time: unix_time_as_millis(),
            inner: Entry { tx, remote },
            is_large_cycle,
            is_proposal_tx,
        });
        self.total_tx_size = total_tx_size;
        self.ready_rx.notify_one();
        Ok(true)
    }

/// First compare fee_rate, select the smallest fee_rate,
/// and then select the latest timestamp, for eviction,
/// the latest timestamp which also means that the fewer descendants may exist.
#[derive(Eq, PartialEq, Clone, Debug)]
pub struct EvictKey {
    pub fee_rate: FeeRate,
    pub timestamp: u64,
    pub descendants_count: usize,
}

impl PartialOrd for EvictKey {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering> {
        Some(self.cmp(other))
    }
}

impl Ord for EvictKey {
    fn cmp(&self, other: &Self) -> Ordering {
        if self.fee_rate == other.fee_rate {
            if self.descendants_count == other.descendants_count {
                self.timestamp.cmp(&other.timestamp)
            } else {
                self.descendants_count.cmp(&other.descendants_count)
            }
        } else {
            self.fee_rate.cmp(&other.fee_rate)
        }
    }
