### Title
Missing Cancel/Refund Mechanism Permanently Locks User Fees in Echo Contract - (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

The `Echo` contract collects fees from users in `requestPriceUpdatesWithCallback` but provides no mechanism for users to cancel a pending request and recover the provider-fee portion of their payment. If a provider never calls `executeCallback`, the user's funds are permanently locked in the contract with no recovery path.

---

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` splits `msg.value` into two parts:

1. **Pyth protocol fee** — immediately credited to `_state.accruedFeesInWei` (line 99).
2. **Provider fee** — stored in `req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei)` (line 84), held inside the request struct until `executeCallback` is called. [1](#0-0) 

The provider fee is only released to `_state.providers[providerToCredit].accruedFeesInWei` inside `executeCallback` (line 161–162), which can only be called by the provider (or anyone after the exclusivity period). [2](#0-1) 

The contract has **no cancel function, no timeout, and no expiry** for pending requests. A grep across all Echo source files confirms zero occurrences of any cancel, refund, or request-withdrawal logic.

The developers themselves acknowledge the risk in an inline TODO comment:

> *"TODO: if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract."* [3](#0-2) 

And separately:

> *"TODO: there also needs to be some penalty mechanism in case the expected provider doesn't execute the callback."* [4](#0-3) 

---

### Impact Explanation

Any user who calls `requestPriceUpdatesWithCallback` and pays the required fee loses the provider-fee portion (`req.fee`) permanently if:

- The assigned provider goes offline or is unresponsive.
- The provider deliberately withholds fulfillment (griefing).
- `executeCallback` reverts for any reason (e.g., the actual Pyth fee `pyth.getUpdateFee(updateData)` exceeds `req.fee + msg.value` due to a Pyth fee increase, causing an arithmetic underflow at line 162).

The Pyth protocol fee is already gone (accrued at request time). The provider fee — which can be the dominant portion of the total fee — is locked with no recovery path. This constitutes a direct, permanent loss of user funds.

---

### Likelihood Explanation

The likelihood is **medium**. The exclusivity period design means the assigned provider has a time window during which only they can fulfill. If the provider is offline during that window, no one else can step in. After the exclusivity period, any actor can call `executeCallback`, but they must supply valid Pyth `updateData` for the exact `publishTime` stored in the request — data that may no longer be available from Pyth's off-chain API if the timestamp is old. This creates a realistic scenario where requests become permanently unfulfillable, locking user funds.

---

### Recommendation

Add a `cancelRequest` function that:
1. Verifies `msg.sender == req.requester`.
2. Optionally enforces a minimum waiting period (e.g., after the exclusivity period has elapsed and no fulfillment occurred).
3. Returns `req.fee` to `req.requester` via a `call{value: req.fee}`.
4. Deducts `_state.pythFeeInWei` from `_state.accruedFeesInWei` (or accepts that the Pyth protocol fee is non-refundable, consistent with Entropy's documented behavior).
5. Calls `clearRequest(sequenceNumber)`.

---

### Proof of Concept

1. User calls `requestPriceUpdatesWithCallback{value: totalFee}(provider, publishTime, priceIds, gasLimit)`.
   - `_state.accruedFeesInWei += pythFeeInWei` — Pyth fee immediately taken.
   - `req.fee = totalFee - pythFeeInWei` — provider fee locked in request struct.
2. Provider never calls `executeCallback` (offline, griefing, or data unavailable).
3. User attempts to recover funds — no `cancelRequest`, no timeout, no expiry exists anywhere in the Echo contract.
4. `req.fee` remains locked in the contract indefinitely. The only way funds leave the contract is via `executeCallback` (provider) or `withdrawFees`/`withdrawAsFeeManager` (admin/fee manager) — neither of which returns funds to the user. [5](#0-4) [6](#0-5)

    function requestPriceUpdatesWithCallback(
        address provider,
        uint64 publishTime,
        bytes32[] calldata priceIds,
        uint32 callbackGasLimit
    ) external payable override returns (uint64 requestSequenceNumber) {
        require(
            _state.providers[provider].isRegistered,
            "Provider not registered"
        );

        // FIXME: this comment is wrong. (we're not using tx.gasprice)
        // NOTE: The 60-second future limit on publishTime prevents a DoS vector where
        //      attackers could submit many low-fee requests for far-future updates when gas prices
        //      are low, forcing executors to fulfill them later when gas prices might be much higher.
        //      Since tx.gasprice is used to calculate fees, allowing far-future requests would make
        //      the fee estimation unreliable.
        require(publishTime <= block.timestamp + 60, "Too far in future");
        if (priceIds.length > MAX_PRICE_IDS) {
            revert TooManyPriceIds(priceIds.length, MAX_PRICE_IDS);
        }
        requestSequenceNumber = _state.currentSequenceNumber++;

        uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
        if (msg.value < requiredFee) revert InsufficientFee();

        Request storage req = allocRequest(requestSequenceNumber);
        req.sequenceNumber = requestSequenceNumber;
        req.publishTime = publishTime;
        req.callbackGasLimit = callbackGasLimit;
        req.requester = msg.sender;
        req.provider = provider;
        req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);

        // Create array with the right size
        req.priceIdPrefixes = new bytes8[](priceIds.length);

        // Copy only the first 8 bytes of each price ID to storage
        for (uint8 i = 0; i < priceIds.length; i++) {
            // Extract first 8 bytes of the price ID
            bytes32 priceId = priceIds[i];
            bytes8 prefix;
            assembly {
                prefix := priceId
            }
            req.priceIdPrefixes[i] = prefix;
        }
        _state.accruedFeesInWei += _state.pythFeeInWei;

        emit PriceUpdateRequested(req, priceIds);
    }
