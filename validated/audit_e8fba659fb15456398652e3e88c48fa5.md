### Title
`FeeRate::fee()` Rounds Down When Computing Minimum Fee Threshold, Allowing Sub-Minimum Fee Transactions - (File: `util/types/src/core/fee_rate.rs`)

### Summary

`FeeRate::fee()` uses floor (truncating) integer division when computing the minimum fee required for a given transaction size. Because this result is used as the admission threshold in `check_tx_fee` and as the extra-fee increment in `calculate_min_replace_fee`, a transaction submitter can craft transactions whose true fee rate is strictly below the configured `min_fee_rate` or `min_rbf_rate` and still pass the check.

### Finding Description

`FeeRate::fee()` computes the minimum fee for a given weight as:

```rust
// util/types/src/core/fee_rate.rs, line 35
let fee = self.0.saturating_mul(weight) / KW;   // KW = 1000, floor division
``` [1](#0-0) 

This result is consumed in two places:

**1. Tx-pool admission (`check_tx_fee`)**

```rust
// tx-pool/src/util.rs, line 45-47
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee { ... }
``` [2](#0-1) 

**2. RBF minimum replacement fee (`calculate_min_replace_fee`)**

```rust
// tx-pool/src/pool.rs, line 103
let extra_rbf_fee = self.config.min_rbf_rate.fee(size as u64);
``` [3](#0-2) 

Because `fee()` truncates, the computed threshold is strictly less than the true mathematical minimum whenever `fee_rate * weight` is not an exact multiple of `KW = 1000`. A transaction whose actual fee rate is below the configured minimum can satisfy `fee >= min_fee` after truncation.

**Concrete example with default `min_rbf_rate = 1500`:**

| `size` (bytes) | True extra RBF fee | `fee()` result (floor) | Shortfall |
|---|---|---|---|
| 1 | 1.5 shannons → ceil = 2 | 1 | 1 shannon |
| 3 | 4.5 shannons → ceil = 5 | 4 | 1 shannon |
| 5 | 7.5 shannons → ceil = 8 | 7 | 1 shannon |

For every odd-sized transaction with the default `min_rbf_rate = 1500`, the required replacement fee is 1 shannon lower than it should be.

**Concrete example with a non-default `min_fee_rate = 1001`:**

- `tx_size = 999`, `min_fee_rate = 1001`
- `min_fee = 1001 * 999 / 1000 = 999` (floor; true value = 999.999)
- A transaction paying exactly 999 shannons passes, but its effective fee rate is `999 * 1000 / 999 ≈ 1000 < 1001` [4](#0-3) 

### Impact Explanation

Any unprivileged RPC caller submitting via `send_transaction` can bypass the `min_fee_rate` or `min_rbf_rate` enforcement by at most `KW − 1 = 999` shannons per transaction. For RBF, a replacement transaction can pay slightly less than the required minimum replacement fee, weakening the economic guarantee that replacements must strictly outbid the evicted transaction. The shortfall is small (sub-shannon to sub-CKB range) but violates the stated invariant: *"txs with lower fee rate than this will not be relayed or be mined."* [5](#0-4) 

### Likelihood Explanation

The default `min_fee_rate = 1000` is an exact multiple of `KW = 1000`, so for that specific value the rounding is always exact. However, the default `min_rbf_rate = 1500` is **not** a multiple of 1000, so the rounding error is triggered for every odd-sized transaction under the default configuration. Any operator using a non-round `min_fee_rate` (e.g., 1001, 1500, 2000 is fine but 1999 is not) is also affected. The entry path is the public `send_transaction` RPC, reachable by any unprivileged caller. [6](#0-5) 

### Recommendation

Replace the floor division in `FeeRate::fee()` with ceiling division (`div_ceil`) so that the computed minimum fee is always at least as large as the true mathematical minimum:

```rust
pub fn fee(self, weight: u64) -> Capacity {
    // Use ceiling division: (a + b - 1) / b
    let fee = self.0.saturating_mul(weight).div_ceil(KW);
    Capacity::shannons(fee)
}
```

This matches the correct rounding direction for a threshold: when computing "how much must the user pay at minimum," round up to protect the network, not down to favor the submitter — the same principle violated in the ERC4626 `previewWithdraw` analog. [1](#0-0) 

### Proof of Concept

With `min_rbf_rate = FeeRate(1500)` (the default) and a transaction of `size = 1` byte:

```
extra_rbf_fee = FeeRate(1500).fee(1)
             = 1500 * 1 / 1000          // floor
             = 1 shannon

True minimum  = ceil(1500 * 1 / 1000)
             = ceil(1.5)
             = 2 shannons
```

A replacement transaction paying `sum(replaced_fees) + 1` shannon passes `calculate_min_replace_fee`, even though the correct threshold is `sum(replaced_fees) + 2` shannons. The submitter bypasses the RBF economic guard by 1 shannon per replacement. [7](#0-6)
