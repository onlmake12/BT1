### Title
Unprotected `setup()` in `ReceiverSetup` Allows Griefing of `WormholeReceiver` Deployment - (File: `target_chains/ethereum/contracts/contracts/wormhole-receiver/ReceiverSetup.sol`)

### Summary

`ReceiverSetup.setup()` is a `public` function with no access control and no "already initialized" guard. The `WormholeReceiver` proxy deployment is a multi-step process that can be executed non-atomically. If the proxy is deployed with empty `initData`, an attacker can front-run the legitimate `setup()` call and inject malicious guardian addresses and/or a malicious implementation contract, permanently bricking the deployed proxy and forcing a full redeployment.

### Finding Description

The `WormholeReceiver` deployment follows a three-step process:

1. Deploy `ReceiverSetup`
2. Deploy `ReceiverImplementation`
3. Deploy `WormholeReceiver` proxy — optionally passing encoded `setup()` call data to the constructor [1](#0-0) 

The `ERC1967Proxy` constructor only calls the logic contract if `initData` is non-empty. If `initData` is empty (as demonstrated in the test utilities), the proxy is deployed with `ReceiverSetup` as its implementation but `setup()` is never called — leaving the proxy in a fully uninitialized, publicly callable state. [2](#0-1) 

The `setup()` function is `public` with no caller restriction, no `onlyOwner`, no `initializer` modifier, and no "already initialized" check. It accepts arbitrary `implementation`, `initialGuardians`, `chainId`, `governanceChainId`, and `governanceContract` values from any caller.

The non-atomic deployment pattern is explicitly demonstrated in the test utilities: [3](#0-2) 

Step 3 (proxy deploy) and step 4 (`setup()` call) are separate transactions. Between them, any unprivileged attacker can call `setup()` on the proxy with attacker-controlled parameters.

### Impact Explanation

If an attacker front-runs `setup()`:

- They supply attacker-controlled `initialGuardians` — the proxy will accept VAAs signed by the attacker's keys as valid.
- They supply a malicious `implementation` address — `_upgradeTo(implementation)` permanently sets the proxy's logic to the attacker's contract, giving full control over all proxy state and behavior.
- After the attacker's `setup()` executes, the proxy's implementation is changed to the attacker's contract. The legitimate deployer's subsequent `setup()` call will be routed to the attacker's implementation and will revert or behave arbitrarily.

`WormholeReceiver` is the VAA verification layer consumed by `PythUpgradable`. A compromised `WormholeReceiver` with attacker-controlled guardians allows forged VAAs carrying arbitrary price data to pass verification, enabling direct manipulation of all Pyth price feeds on that chain. [4](#0-3) 

### Likelihood Explanation

The current production deployment scripts (`batchDeployReceivers.ts`, `common.ts`, `zkSyncDeployWormhole.ts`) all encode `initData` and pass it atomically to the proxy constructor, which mitigates the attack for existing deployments. [5](#0-4) 

However:
- The non-atomic pattern (empty `initData` + separate `setup()` call) is present in the test utilities and could be replicated by a developer deploying to a new chain.
- The `setup()` function itself carries no protection whatsoever, so any future deployment that deviates from the atomic pattern is immediately exploitable.
- Likelihood is **Medium**: the attack window exists whenever a new `WormholeReceiver` is deployed non-atomically, which is a realistic operational scenario.

### Recommendation

Add an "already initialized" guard to `ReceiverSetup.setup()` that checks whether the proxy has already been configured (e.g., whether `chainId != 0` or whether the guardian set is non-empty), and revert if so. Alternatively, restrict `setup()` to be callable only from within the proxy constructor (e.g., by checking `msg.sender == address(this)` or using a one-time flag in storage). The production deployment scripts should also enforce that `initData` is always non-empty.

### Proof of Concept

1. Attacker monitors the mempool for a `WormholeReceiver` deployment transaction with empty `initData`.
2. Attacker observes the proxy address (deterministic from deployer nonce or CREATE2 salt).
3. Attacker submits a higher-gas transaction calling:
   ```solidity
   ReceiverSetup(proxyAddress).setup(
       attackerImplementation,   // attacker-controlled logic
       [attackerGuardian],       // attacker-controlled guardian
       chainId,
       governanceChainId,
       governanceContract
   );
   ```
4. Attacker's `setup()` executes first. `_upgradeTo(attackerImplementation)` sets the proxy's implementation to the attacker's contract.
5. The legitimate deployer's `setup()` call is now routed to `attackerImplementation`, which can revert or do anything.
6. The proxy is permanently bricked from the deployer's perspective. The attacker's guardian key can now sign VAAs that `WormholeReceiver` will accept as valid, enabling forged Pyth price updates. [6](#0-5) [1](#0-0)

```text
contract ReceiverSetup is ReceiverSetters, ERC1967Upgrade {
    function setup(
        address implementation,
        address[] memory initialGuardians,
        uint16 chainId,
        uint16 governanceChainId,
        bytes32 governanceContract
    ) public {
        require(initialGuardians.length > 0, "no guardians specified");

        ReceiverStructs.GuardianSet memory initialGuardianSet = ReceiverStructs
            .GuardianSet({keys: initialGuardians, expirationTime: 0});

        storeGuardianSet(initialGuardianSet, 0);
        // initial guardian set index is 0, which is the default value of the storage slot anyways

        setChainId(chainId);

        setGovernanceChainId(governanceChainId);
        setGovernanceContract(governanceContract);

        _upgradeTo(implementation);
    }
```
