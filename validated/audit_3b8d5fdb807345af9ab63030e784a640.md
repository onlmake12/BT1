### Title
Excess ETH Permanently Captured by Pyth Fee Pool Due to Coupled Fee-Accounting in `requestHelper` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `requestHelper` function in `Entropy.sol` tightly couples fee validation, fee splitting between Pyth and the provider, and request-state management into a single function. The Pyth fee accrual formula uses `msg.value - providerFee` rather than `_state.pythFeeInWei`, so any ETH sent above the required minimum is silently captured in Pyth's fee pool with no refund path. This is a direct consequence of the high-coupling design: fee accounting and request lifecycle share the same code path, making the invariant "Pyth earns exactly `pythFeeInWei` per request" impossible to enforce.

---

### Finding Description

In `Entropy.sol`, `requestHelper` (lines 233–239) performs fee validation and fee distribution in the same block:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
```

`requiredFee = providerFee + pythFeeInWei`. The contract accepts any `msg.value >= requiredFee` but distributes the **entire surplus** to `_state.accruedPythFeesInWei`:

| `msg.value` | Provider receives | Pyth receives | User refunded |
|---|---|---|---|
| `== requiredFee` | `providerFee` | `pythFeeInWei` | 0 |
| `> requiredFee` | `providerFee` | `pythFeeInWei + surplus` | **0** |

There is no refund mechanism anywhere in the call chain. The surplus is permanently locked in Pyth's accrued-fee balance, withdrawable only by the Pyth admin.

The coupling is the root cause: because fee accounting and request-state mutation are interleaved in one function, the correct invariant (`accruedPythFeesInWei` grows by exactly `pythFeeInWei` per request) is never enforced. A separated fee-accounting module would make this invariant explicit and testable.

---

### Impact Explanation

Any Entropy user who sends `msg.value > getFeeV2()` permanently loses the surplus ETH. Concrete scenarios:

1. **Fee decrease between query and submission**: User calls `getFeeV2()` off-chain, fee drops before the transaction lands; user's original `msg.value` now exceeds the new `requiredFee`, and the surplus is captured.
2. **SDK/front-end buffer**: SDKs or dApps that add a small ETH buffer to avoid reverts (a common pattern) cause every request to silently donate the buffer to Pyth.
3. **Programmatic callers**: Smart contracts that forward `msg.value` without exact-fee calculation (e.g., `entropy.requestV2{value: msg.value}()`) lose all excess ETH.

The funds are not recoverable by the user; they accumulate in `accruedPythFeesInWei` and are only withdrawable by the Pyth admin via `withdrawFees`.

---

### Likelihood Explanation

Medium. Fee changes are infrequent but possible (admin-controlled via `setFeeInWei`). SDK-level buffering is a common defensive pattern. Programmatic callers forwarding `msg.value` are realistic. Each individual loss is small, but the pattern is systematic across all callers who do not send the exact fee.

---

### Recommendation

1. **Refund excess ETH**: After crediting `pythFeeInWei` and `providerFee`, refund `msg.value - requiredFee` to `msg.sender`.
2. **Alternatively, enforce exact fee**: Change the check to `if (msg.value != requiredFee) revert` to make overpayment impossible (though this is less user-friendly).
3. **Separate fee accounting**: Extract fee splitting into a dedicated internal function with a clear invariant (`Pyth earns exactly pythFeeInWei`), decoupled from sequence-number and commitment logic. This makes the invariant auditable independently.

---

### Proof of Concept

```
1. Admin sets pythFeeInWei = 100 wei, provider feeInWei = 400 wei.
   → getFeeV2() = 500 wei.

2. User's front-end adds a 10% buffer: sends msg.value = 550 wei.

3. requestHelper executes:
   requiredFee = 500 wei
   550 >= 500 → no revert
   providerFee = 400 wei
   providerInfo.accruedFeesInWei += 400
   _state.accruedPythFeesInWei += (550 - 400) = 150   ← should be 100

4. User loses 50 wei (the buffer) permanently.
   Pyth admin can withdraw the extra 50 wei via withdrawFees().
   User has no recourse.
```

```text
    function getProviderFee(
        address providerAddr,
        uint32 gasLimit
    ) internal view returns (uint128 feeAmount) {
        EntropyStructsV2.ProviderInfo memory provider = _state.providers[
            providerAddr
        ];

        // Providers charge a minimum of their configured feeInWei for every request.
        // Requests using more than the defaultGasLimit get a proportionally scaled fee.
        // This approach may be somewhat simplistic, but it allows us to continue using the
        // existing feeInWei parameter for the callback failure flow instead of defining new
        // configuration values.
        uint32 roundedGasLimit = uint32(roundTo10kGas(gasLimit)) * TEN_THOUSAND;
        if (
            provider.defaultGasLimit > 0 &&
            roundedGasLimit > provider.defaultGasLimit
        ) {
            // This calculation rounds down the fee, which means that users can get some gas in the callback for free.
            // However, the value of the free gas is < 1 wei, which is insignificant.
            uint128 additionalFee = ((roundedGasLimit -
                provider.defaultGasLimit) * provider.feeInWei) /
                provider.defaultGasLimit;
            return provider.feeInWei + additionalFee;
        } else {
            return provider.feeInWei;
        }
    }
```
