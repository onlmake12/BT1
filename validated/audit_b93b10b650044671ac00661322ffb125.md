### Title
Tx-Pool Minimum Fee Rate Check Uses Raw `size` While Actual Fee Rate Uses `weight`, Allowing Below-Minimum-Fee-Rate Transactions Into the Pool — (File: `tx-pool/src/util.rs`)

### Summary

The tx-pool enforces a minimum fee rate admission check using the raw serialized transaction size (`tx_size`), but the actual fee rate used for sorting, eviction, and block assembly uses `weight = max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`. For transactions with high cycle counts, `weight >> size`, so a transaction can pass the admission check with a fee computed against `size` while its true effective fee rate (computed against `weight`) is far below the configured minimum. This is the direct CKB analog of the MochiVault debt-tracking inconsistency: one quantity is used to record the individual value, a different quantity is used for the aggregate/enforcement check, and the divergence grows with the ratio between the two.

---

### Impact Explanation

An attacker crafts a transaction with:
- `size = 1,000` bytes (small serialized body)
- `cycles = 3,500,000,000` (near the block cycle limit)
- `fee = min_fee_rate × 1,000 / 1,000 = min_fee_rate` shannons (just above the size-based threshold)

Admission check passes: `fee ≥ min_fee_rate × size`.

Actual fee rate after admission: `fee / weight = min_fee_rate × 1,000 / 597,000 ≈ 0.0017 × min_fee_rate` — roughly **600× below the configured minimum**.

Consequences:
1. **Spam protection is bypassed.** The minimum fee rate is the primary anti-spam mechanism for the tx-pool. Transactions that should be rejected as too cheap are admitted.
2. **Pool space is consumed deceptively.** `total_tx_size` grows by only 1,000 bytes per such transaction, so the pool's size limit is not triggered, yet each transaction occupies a full block's worth of cycles.
3. **Legitimate transactions can be crowded out.** If the pool fills (by `total_tx_size`), honest transactions paying the correct fee rate are rejected while the attacker's below-minimum-rate transactions remain.
4. **Eviction does not fully mitigate this.** Eviction by fee rate (using `weight`) will eventually remove these transactions, but only after they have already occupied pool slots and potentially caused legitimate transactions to be rejected.

---

### Likelihood Explanation

The attack requires only the ability to submit transactions via the public `send_transaction` RPC — no privileged access, no key material, no majority hashpower. Any unprivileged RPC caller on a node with a public endpoint can exploit this. The crafting of a high-cycle, small-size transaction is straightforward: use a script that loops to consume cycles while keeping the transaction body small. The inconsistency is structural and present in every version of the code that contains this `check_tx_fee` implementation.

---

### Recommendation

Replace the size-based minimum fee check in `check_tx_fee` with a weight-based check, consistent with how fee rate is computed everywhere else:

```rust
// Instead of:
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);

// Use:
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

This requires passing `cycles` into `check_tx_fee`. Similarly, `total_tx_size` in `PoolMap` should track the sum of `weight` values (or maintain a separate `total_tx_weight` counter) so that the pool size limit reflects actual block-space consumption rather than raw byte counts.

---

### Proof of Concept

1. Configure a CKB node with `min_fee_rate = 1000` shannons/KW.
2. Construct a transaction with:
   - A lock script that loops for ~3 billion cycles (e.g., a tight RISC-V loop in CKB-VM).
   - A small cell output (≈ 61 bytes minimum occupied capacity).
   - Serialized size ≈ 1,000 bytes.
   - Fee = 1,000 shannons (= `min_fee_rate × size / 1000`).
3. Submit via `send_transaction` RPC.
4. Observe: the transaction is accepted into the pool (`check_tx_fee` passes because `fee ≥ min_fee_rate × 1000 / 1000`).
5. Query `get_raw_tx_pool` verbose: the entry's `size` is 1,000 bytes and `cycles` is ~3 billion.
6. Compute actual fee rate: `1000 × 1000 / max(1000, 3e9 × 0.000170571) ≈ 1000 × 1000 / 511713 ≈ 1.95 shannons/KW` — far below the 1,000 shannons/KW minimum.
7. Repeat with many such transactions; `total_tx_size` grows slowly (1,000 bytes each) while the pool fills with transactions that each consume nearly a full block's cycle budget. [7](#0-6) [8](#0-7) [9](#0-8)
