### Title
Wrong Variable in `DownloadScheduler::decrease` Causes Overly Aggressive Peer Disconnection â€” (File: `sync/src/types/mod.rs`)

---

### Summary

`DownloadScheduler::decrease` in the sync module assigns `self.task_count.saturating_add(num)` to `self.timeout_count` instead of accumulating `self.timeout_count.saturating_add(num)`. Because `task_count` starts at `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32), the guard `if self.timeout_count > 2` is always true on the very first call, so `task_count` is decremented on **every** slow block delivery rather than only after accumulating 2+ slow responses. When `task_count` reaches 0, the peer is disconnected.

---

### Finding Description

`DownloadScheduler` tracks per-peer block download concurrency. The `decrease` function is intended to accumulate a `timeout_count` and only reduce `task_count` once that accumulator exceeds 2:

```rust
fn decrease(&mut self, num: usize) {
    self.timeout_count = self.task_count.saturating_add(num); // BUG: should be self.timeout_count
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [1](#0-0) 

The correct line should be:
```rust
self.timeout_count = self.timeout_count.saturating_add(num);
```

Because `task_count` initialises to `INIT_BLOCKS_IN_TRANSIT_PER_PEER` (32) and `num` is 1 or 2, `self.task_count + num` is always â‰« 2, so the branch fires unconditionally and `task_count` is decremented on every single slow delivery. The accumulator is never actually accumulated â€” it is overwritten with a value that always exceeds the threshold, then immediately reset to 0. [2](#0-1) 

`decrease` is called from `remove_by_block` for the `NormalToUpper` and `UpperToMax` time quantiles:

```rust
TimeQuantile::NormalToUpper => { if should_punish { set.decrease(1) } }
TimeQuantile::UpperToMax   => { if should_punish { set.decrease(2) } }
``` [3](#0-2) 

When `task_count` reaches 0, `prune` removes the peer from `download_schedulers` and adds it to `disconnect_list`:

```rust
if v.task_count == 0 {
    disconnect_list.insert(*k);
    false
}
``` [4](#0-3) 

---

### Impact Explanation

With the bug, a peer is penalised (one `task_count` decrement) for **every** block whose delivery falls in the slow time quantile. With the correct accumulator logic, a decrement would only occur once every **3** slow deliveries (accumulate to > 2, then reset). Starting from `task_count = 32`, the peer reaches `task_count = 0` â€” and is disconnected â€” after only 32 slow deliveries instead of the intended ~96. The `task_count` also governs how many concurrent block requests are issued to the peer (`can_fetch = task_count - inflight`), so even before disconnection the peer's download window is collapsed far faster than the protocol intends, stalling block synchronisation.

---

### Likelihood Explanation

Any unprivileged sync peer that delivers blocks in the `NormalToUpper` or `UpperToMax` latency band (> ~1250 ms, which is normal under congestion or geographic distance) will trigger this path. No special privileges, keys, or majority hash-power are required. A malicious peer can deliberately pace block responses just above the `normal_time` threshold to exhaust its own `task_count` and force a disconnect, or â€” more impactfully â€” a network-level adversary who can add latency to a victim's connections with legitimate peers can cause those legitimate peers to be disconnected 3Ã— faster than the protocol intends, degrading the victim's sync connectivity.

---

### Recommendation

Replace the wrong variable reference in `DownloadScheduler::decrease`:

```rust
fn decrease(&mut self, num: usize) {
    // was: self.timeout_count = self.task_count.saturating_add(num);
    self.timeout_count = self.timeout_count.saturating_add(num);
    if self.timeout_count > 2 {
        self.task_count = self.task_count.saturating_sub(1);
        self.timeout_count = 0;
    }
}
``` [1](#0-0)

### Proof of Concept

1. A sync peer connects to the victim node and advertises a chain of blocks.
2. The peer delivers each block with a response time of ~1300 ms (above `normal_time = 1250 ms`, below `low_time = 1500 ms`), placing it in the `NormalToUpper` quantile.
3. Each delivery calls `remove_by_block` â†’ `decrease(1)`.
4. Due to the bug, `timeout_count = task_count + 1` (e.g., 33) which is always > 2, so `task_count` is decremented on every delivery and `timeout_count` is reset to 0.
5. After 32 such deliveries, `task_count = 0`.
6. The next call to `prune` adds the peer to `disconnect_list` and the node disconnects it.
7. With the correct fix, the same 32 deliveries would only decrement `task_count` ~10 times (once per 3 slow responses), leaving `task_count â‰ˆ 22` â€” well above the disconnection threshold. [5](#0-4)
