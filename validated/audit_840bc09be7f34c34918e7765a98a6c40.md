### Title
Unbounded Orphan Pool Growth via PoW-Free Block Flooding — (`chain/src/utils/orphan_block_pool.rs`, `chain/src/chain_service.rs`)

### Summary

`non_contextual_verify` does not check Proof-of-Work. An unprivileged remote peer can relay an unlimited number of structurally valid, PoW-invalid blocks with unknown parent hashes. Each block bypasses the only pre-insertion gate, is written to the database, and is inserted into an `OrphanBlockPool` that has no capacity cap and no per-insertion eviction. The 60-second cleanup timer only removes epoch-expired entries (blocks older than ~24 hours), so freshly injected orphans accumulate without bound, exhausting both memory and disk.

---

### Finding Description

**Gate: `non_contextual_verify` omits PoW**

`ChainService::non_contextual_verify` calls exactly two verifiers: [1](#0-0) 

`BlockVerifier::verify` runs only structural checks — `BlockProposalsLimitVerifier`, `BlockBytesVerifier`, `CellbaseVerifier`, `DuplicateVerifier`, `MerkleRootVerifier`: [2](#0-1) 

None of these verify the Eaglesong nonce. `PowVerifier` exists and is self-contained (it needs only the header, not the parent): [3](#0-2) 

But it is only called inside the context-dependent `HeaderVerifier`, which is never invoked on the non-contextual path. A block with a completely invalid nonce passes `non_contextual_verify` without error.

**Insertion path with no eviction**

After passing the gate, the block is written to the database and handed to `OrphanBroker::process_lonely_block`. When the parent hash is unknown, the block is unconditionally inserted: [4](#0-3) 

`InnerPool::insert` performs no capacity check: [5](#0-4) 

`with_capacity` sets only the initial `HashMap` allocation, not a maximum: [6](#0-5) 

**Cleanup fires every 60 s but only removes epoch-expired entries** [7](#0-6) 

`clean_expired_blocks` removes a block only when `epoch_number + EXPIRED_EPOCH < tip_epoch`, where `EXPIRED_EPOCH = 6`: [8](#0-7) [9](#0-8) 

CKB epochs last ~4 hours, so `EXPIRED_EPOCH = 6` means ~24 hours. Blocks injected in the current epoch are never touched by the timer. The 60-second tick provides no protection against a sustained flood.

---

### Impact Explanation

- **Memory exhaustion**: each orphan entry is held in two `HashMap`s (`blocks`, `parents`) and one `HashSet` (`leaders`) inside the `RwLock<InnerPool>`. Thousands of entries accumulate in RAM with no eviction.
- **Disk exhaustion**: `insert_block` commits every block to RocksDB before orphan insertion; the attacker also fills the node's block store.
- **Lock contention**: every insert and every 60-second cleanup acquires the `RwLock` write lock, degrading throughput for legitimate block processing.
- **Network participation degraded**: a node under memory/disk pressure becomes unresponsive to peers, effectively removing it from the network.

---

### Likelihood Explanation

The attack requires only:
1. A P2P connection to the target node (standard sync peer).
2. Crafting blocks that satisfy `BlockVerifier`: correct cellbase `since` = block number, valid merkle root, size ≤ `max_block_bytes` (~597 KB on mainnet), no duplicate transactions. All of this is computationally trivial — no mining hardware needed.
3. Setting the `parent_hash` field to any unknown 32-byte value.

No PoW, no stake, no privileged access. The cost per injected block is a few microseconds of CPU to compute the merkle root.

---

### Recommendation

1. **Add a PoW check to `non_contextual_verify`**: call `PowVerifier` (which only needs the header, not the parent) before `insert_block`. This is the minimal fix and eliminates the attack entirely for mainnet.
2. **Add a hard capacity cap to `OrphanBlockPool`**: reject or evict (LRU or oldest-epoch-first) when `parents.len() >= MAX_ORPHAN_POOL_SIZE`. This is defense-in-depth even if PoW is checked.
3. **Per-peer orphan rate limiting**: track how many orphans each peer has contributed and disconnect/ban peers that exceed a threshold.

---

### Proof of Concept

```
1. Connect to a CKB mainnet/testnet node as a sync peer.
2. In a loop:
   a. Build a BlockView with:
      - block_number = any value > 0
      - parent_hash  = random 32 bytes (guaranteed unknown)
      - one valid cellbase tx (since = block_number, no outputs)
      - transactions_root = calc_transactions_root([cellbase])
      - nonce = 0  (invalid PoW — not checked)
   b. Send via the SendBlock P2P message.
3. After 60 seconds, query the node's orphan pool metric
   `ckb_chain_orphan_count` (set at orphan_broker.rs:127-131).
4. Assert: pool size grows linearly with send rate R,
   peak pool.len() ≈ R × 60, unbounded by any cap.
5. Observe: node RSS grows proportionally; RocksDB size grows
   at ~block_size × R × time; RPC latency increases.
``` [10](#0-9)

    pub(crate) fn process_lonely_block(&self, lonely_block: LonelyBlockHash) {
        let block_hash = lonely_block.block_number_and_hash.hash();
        let block_number = lonely_block.block_number_and_hash.number();
        let parent_hash = lonely_block.parent_hash();
        let parent_is_pending_verify = self.is_pending_verify.contains(&parent_hash);
        let parent_status = self.shared.get_block_status(&parent_hash);
        if parent_is_pending_verify || parent_status.contains(BlockStatus::BLOCK_STORED) {
            debug!(
                "parent {} has stored: {:?} or is_pending_verify: {}, processing descendant directly {}-{}",
                parent_hash, parent_status, parent_is_pending_verify, block_number, block_hash,
            );
            self.process_descendant(lonely_block);
        } else if parent_status.eq(&BlockStatus::BLOCK_INVALID) {
            self.process_invalid_block(lonely_block);
        } else {
            self.orphan_blocks_broker.insert(lonely_block);
        }

        self.search_orphan_leaders();

        if let Some(metrics) = ckb_metrics::handle() {
            metrics
                .ckb_chain_orphan_count
                .set(self.orphan_blocks_broker.len() as i64)

    fn insert(&mut self, lonely_block: LonelyBlockHash) {
        let hash = lonely_block.hash();
        let parent_hash = lonely_block.parent_hash();
        self.blocks
            .entry(parent_hash.clone())
            .or_default()
            .insert(hash.clone(), lonely_block);
        // Out-of-order insertion needs to be deduplicated
        self.leaders.remove(&hash);
        // It is a possible optimization to make the judgment in advance,
        // because the parent of the block must not be equal to its own hash,
        // so we can judge first, which may reduce one arc clone
        if !self.parents.contains_key(&parent_hash) {
            // Block referenced by `parent_hash` is not in the pool,
            // and it has at least one child, the new inserted block, so add it to leaders.
            self.leaders.insert(parent_hash.clone());
        }
        self.parents.insert(hash, parent_hash);
    }

    pub fn clean_expired_blocks(&mut self, tip_epoch: EpochNumber) -> Vec<LonelyBlockHash> {
        let mut result = vec![];

        for hash in self.leaders.clone().iter() {
            if self.need_clean(hash, tip_epoch) {
                // remove items in orphan pool and return hash to callee(clean header map)
                let descendants = self.remove_blocks_by_parent(hash);
                result.extend(descendants);
            }
        }
        result
    }

    /// get 1st block belongs to that parent and check if it's expired block
    fn need_clean(&self, parent_hash: &packed::Byte32, tip_epoch: EpochNumber) -> bool {
        self.blocks
            .get(parent_hash)
            .and_then(|map| {
                map.iter().next().map(|(_, lonely_block)| {
                    lonely_block.epoch_number() + EXPIRED_EPOCH < tip_epoch
                })
            })
            .unwrap_or_default()
    }
