### Title
Unbounded `constructProviderCommitment` Hash Loop Enables Provider DoS via Request Flooding - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The `constructProviderCommitment` function in `Entropy.sol` contains an unbounded `while` loop that iterates `numHashes` times, computing a `keccak256` hash per iteration. The `numHashes` value stored in each request is the distance between the request's sequence number and the provider's last advanced commitment sequence number. An unprivileged attacker can permissionlessly inflate this value by flooding a provider with paid requests, eventually making the provider's `reveal`/`revealWithCallback` transactions exceed the block gas limit and revert, permanently blocking fulfillment of all in-flight requests until the provider manually advances their commitment in smaller steps. Providers who have not set `maxNumHashes` (the default is `0`, meaning no limit) receive no on-chain protection, and the Fortuna keeper's commitment-advancement guard is also disabled when `maxNumHashes == 0`.

### Finding Description

**Root cause — `constructProviderCommitment`:**

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
``` [1](#0-0) 

This function is called unconditionally from `revealHelper`, which is invoked by both `reveal` and `revealWithCallback`:

```solidity
bytes32 providerCommitment = constructProviderCommitment(
    req.numHashes,
    providerContribution
);
``` [2](#0-1) 

**How `numHashes` is set — `requestHelper`:**

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
``` [3](#0-2) 

`numHashes` equals the number of outstanding (unrevealed) requests since the provider's last commitment advancement. When `maxNumHashes == 0` (the default), the guard is skipped entirely.

**Attacker entry path:**

`requestV2()` is a public, payable function callable by any address:

```solidity
function requestV2()
    external
    payable
    override
    returns (uint64 assignedSequenceNumber)
{
    assignedSequenceNumber = requestV2(getDefaultProvider(), random(), 0);
}
``` [4](#0-3) 

Each call increments `providerInfo.sequenceNumber` by 1. If the provider does not advance their commitment between requests, every subsequent request's `numHashes` is one larger than the previous.

**Off-chain keeper guard is disabled for default providers:**

The Fortuna keeper that calls `advanceProviderCommitment` when outstanding requests exceed a threshold explicitly returns early when `maxNumHashes == 0`:

```rust
if provider_info.max_num_hashes == 0 {
    return Ok(());
}
``` [5](#0-4) 

So providers who rely on the default configuration receive no automatic protection from either the contract or the keeper.

**Gas exhaustion threshold:**

Each `keccak256` on 32 bytes costs approximately 30 gas. Ethereum mainnet block gas limit is ~30 million gas. Therefore, when `numHashes` exceeds approximately **1,000,000**, a single `reveal` or `revealWithCallback` call will exceed the block gas limit and revert. The attacker needs to submit ~1 million paid requests to reach this threshold. `advanceProviderCommitment` also calls `constructProviderCommitment` with the same `numHashes` arithmetic, so a naive single-step recovery attempt also fails:

```solidity
uint32 numHashes = SafeCast.toUint32(
    advancedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
bytes32 providerCommitment = constructProviderCommitment(
    numHashes,
    providerContribution
);
``` [6](#0-5) 

The provider must advance in many small incremental steps to recover, each costing gas.

### Impact Explanation

- All in-flight randomness requests assigned to the targeted provider become unrevealable until the provider manually advances their commitment in small steps (each step costs gas).
- Users who paid fees for randomness requests receive no randomness and no automatic refund; their fees are effectively locked until the provider recovers.
- A sustained attacker (continuing to submit requests) can force the provider into a perpetual recovery loop, making the Entropy service unavailable for that provider.
- The default provider used by `requestV2()` with no arguments is the most impactful target, as it serves all users who do not explicitly choose a provider.

### Likelihood Explanation

- The attack is permissionless: any address can call `requestV2()`.
- The economic cost is real but feasible: ~1 million requests at a typical fee of ~$0.001–$0.01 each = $1,000–$10,000 to trigger the initial DoS threshold.
- Providers who have not set `maxNumHashes` (the default) are fully exposed; the Fortuna keeper provides no protection for them.
- The attack is repeatable: after the provider recovers, the attacker can immediately re-flood.

### Recommendation

1. **Set a non-zero default for `maxNumHashes`** in the contract, or enforce a protocol-level cap on `numHashes` regardless of provider configuration, so that requests with an excessive hash distance are rejected at request time rather than at reveal time.
2. **Update the Fortuna keeper** to advance commitments proactively even when `maxNumHashes == 0`, using a fixed absolute threshold (e.g., 10,000 outstanding requests).
3. **Document the risk** of leaving `maxNumHashes` at 0 prominently in provider onboarding materials.

### Proof of Concept

1. Identify the default provider address via `getDefaultProvider()`.
2. Call `requestV2()` (or `requestWithCallback()`) in a loop ~1,000,001 times, paying the required fee each time. Each call increments `providerInfo.sequenceNumber` by 1 while `currentCommitmentSequenceNumber` remains fixed (assuming the provider has not advanced their commitment).
3. Observe that `req.numHashes` for the last request equals ~1,000,001.
4. When the provider (or Fortuna keeper) attempts to call `revealWithCallback(provider, lastSequenceNumber, ...)`, the transaction reverts with out-of-gas because `constructProviderCommitment` must execute ~1,000,001 `keccak256` iterations (~30M gas).
5. Observe that all prior in-flight requests are also unrevealable because their `numHashes` values are only slightly smaller.
6. The provider must call `advanceProviderCommitment` with a small step (e.g., 1,000 sequence numbers at a time) hundreds of times to recover, spending significant gas.
