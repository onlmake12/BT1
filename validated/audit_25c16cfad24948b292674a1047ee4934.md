### Title
Excess ETH Permanently Captured as Protocol Fee Instead of Refunded to Caller — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

`Entropy.sol`'s internal `requestHelper` function accepts `msg.value >= requiredFee` but never refunds the excess. Any ETH above the required fee is silently credited to `_state.accruedPythFeesInWei` (the Pyth protocol treasury), permanently removing it from the caller's control. This is a direct analog to the TokensFarm excess-fee-not-returned class.

---

### Finding Description

In `requestHelper`, the fee check is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value` is distributed: `providerFee` goes to the provider, and **the remainder — including any overpayment — goes to `accruedPythFeesInWei`**. There is no branch that computes `msg.value - requiredFee` and returns it to `msg.sender`.

This path is reached by all public entry points:

- `request(provider, userCommitment, useBlockHash)` — documented: *"excess value is not refunded"*
- `requestWithCallback(provider, userContribution)` — documented: *"excess value is not refunded"*
- `requestV2(provider, userRandomNumber, gasLimit)` — documented: *"excess value is not refunded"* [2](#0-1) [3](#0-2) [4](#0-3) 

The documentation acknowledges the behavior but does not treat it as a design choice that protects users — it is simply unimplemented refund logic. The root cause is identical to the TokensFarm pattern: a `>=` fee gate with no excess-return path.

By contrast, `PythLazer.verifyUpdate` correctly implements the refund:

```solidity
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [5](#0-4) 

The Entropy contract lacks this equivalent guard entirely.

---

### Impact Explanation

Any caller of `request`, `requestWithCallback`, or `requestV2` who sends `msg.value > getFeeV2(provider, gasLimit)` permanently loses the excess ETH to the Pyth protocol treasury. The caller has no mechanism to recover it. This constitutes a **direct, irreversible loss of user funds** with no compensating benefit to the user.

Realistic overpayment scenarios:
- Fee changes between the block where `getFeeV2` was queried and the block where the transaction is mined.
- Integrating contracts that add a gas buffer to `msg.value` (a common pattern).
- Users who hardcode a round-number fee slightly above the required amount.

---

### Likelihood Explanation

The entry path is fully unprivileged — any EOA or contract can call `requestV2`. The fee is dynamic (provider-configurable), so the gap between queried fee and actual fee at execution time is a realistic, recurring condition. Integrating contracts that forward a buffer are common in production. Likelihood is **medium**: the condition occurs regularly in normal usage, not only under adversarial conditions.

---

### Recommendation

Add an excess-refund path in `requestHelper`, mirroring the pattern already used in `PythLazer.verifyUpdate`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();

// Refund excess to caller
uint256 excess = msg.value - requiredFee;
if (excess > 0) {
    (bool ok, ) = msg.sender.call{value: excess}("");
    require(ok, "Refund failed");
}

uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(requiredFee) - providerFee);
```

Use `call` rather than `transfer` to avoid the 2300-gas stipend issue with smart-contract callers. Apply checks-effects-interactions: update all state before the external `call`. Add a reentrancy guard (`nonReentrant`) to `requestHelper` or its callers if not already present.

---

### Proof of Concept

1. Provider registers with `feeInWei = 1000 wei`, Pyth fee = `500 wei`. `getFeeV2` returns `1500 wei`.
2. Caller queries `getFeeV2` off-chain, gets `1500 wei`.
3. Provider raises fee to `1200 wei` before the tx is mined. `getFeeV2` now returns `1700 wei`.
4. Caller's tx lands with `msg.value = 1500 wei`. `1500 < 1700` → reverts. Caller bumps to `2000 wei` to be safe.
5. Tx mines. `requiredFee = 1700 wei`. `msg.value = 2000 wei`. Excess = `300 wei`.
6. `accruedPythFeesInWei += 2000 - 1200 = 800 wei` (should be `500 wei`). The extra `300 wei` is captured by the protocol.
7. Caller receives no refund. The `300 wei` is unrecoverable. [6](#0-5)
