### Title
Truncated Price ID Validation in Echo Allows Callback Fulfillment with Substitute Price Feed — (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol` stores only the first 8 bytes of each requested price ID at request time, and `executeCallback` validates caller-supplied price IDs against only those 8-byte prefixes. Any price ID sharing the same leading 8 bytes passes the check, allowing an unprivileged caller to fulfill a callback with a different Pyth price feed whose ID collides on the prefix, delivering wrong price data to the consumer contract.

### Finding Description

When a user calls `requestPriceUpdatesWithCallback`, the contract truncates each 32-byte price ID to 8 bytes and stores only the prefix:

```solidity
bytes8 prefix;
assembly { prefix := priceId }
req.priceIdPrefixes[i] = prefix;
``` [1](#0-0) 

The full 32-byte price ID is never persisted. Later, `executeCallback` — which is callable by anyone with no access-control restriction — validates the caller-supplied `priceIds` array against only those stored 8-byte prefixes:

```solidity
bytes8 prefix;
assembly { prefix := priceId }
if (prefix != req.priceIdPrefixes[i]) {
    revert InvalidPriceIds(priceIds[i], req.priceIdPrefixes[i]);
}
``` [2](#0-1) 

After this check passes, the contract calls `pyth.parsePriceFeedUpdates` with the caller-supplied `priceIds` and `updateData`, and the resulting `PriceFeed[]` array is forwarded directly to the consumer's `_echoCallback`: [3](#0-2) 

The Pyth oracle validates `updateData` against the caller-supplied `priceIds`, not against the original requested IDs. Because the stored request contains no full-length price ID to cross-check against, the consumer receives price data for whichever feed the attacker chose, as long as its first 8 bytes match.

The `Request` struct confirms that only `bytes8[] priceIdPrefixes` is stored — the full IDs are discarded after the request is created: [4](#0-3) 

Compare this to `Entropy.sol`, which correctly includes the `provider` address in its storage key to prevent cross-entity collisions: [5](#0-4) 

Echo's `requestKey` omits any such discriminator and hashes only the sequence number: [6](#0-5) 

### Impact Explanation

A consumer contract that requested BTC/USD price data (price ID `0xE62DF6...`) could receive price data for a different asset whose ID shares the same leading 8 bytes. The consumer's `_echoCallback` would process the wrong price, potentially triggering incorrect liquidations, mispricings, or fund transfers. The fee paid by the original requester is consumed and the callback slot is cleared, making the substitution permanent and unrecoverable.

### Likelihood Explanation

Pyth currently publishes hundreds of price feeds. With 8-byte (64-bit) prefixes, a birthday-paradox collision among ~500 feeds has non-negligible probability (~500² / 2⁶⁵ ≈ 6.7 × 10⁻¹³ per pair), making natural collisions rare today. However:

1. Pyth is actively adding new feeds; the collision surface grows quadratically.
2. An attacker who discovers any existing prefix collision can exploit it immediately and repeatedly at zero marginal cost.
3. The `executeCallback` function has no caller restriction after the exclusivity window, so any transaction sender can trigger the substitution. [7](#0-6) 

### Recommendation

Store the full 32-byte price IDs in the `Request` struct instead of the truncated `bytes8` prefixes, and validate the full IDs in `executeCallback`. The gas savings from truncation do not justify the security trade-off. Alternatively, hash the full price ID array and store the hash for O(1) validation.

### Proof of Concept

1. Alice calls `requestPriceUpdatesWithCallback(provider, t, [BTC_ID], gasLimit)`. The contract stores `priceIdPrefixes[0] = bytes8(BTC_ID)` (first 8 bytes of BTC/USD price ID).
2. Attacker identifies `ALT_ID` — a Pyth price feed whose first 8 bytes equal those of `BTC_ID`.
3. After the exclusivity period, the attacker calls `executeCallback(attacker, seq, altUpdateData, [ALT_ID])` where `altUpdateData` is valid Pyth proof data for `ALT_ID` at timestamp `t`.
4. The prefix check passes (`bytes8(ALT_ID) == bytes8(BTC_ID)`). `parsePriceFeedUpdates` succeeds for `ALT_ID`. Alice's consumer receives the price of the alternate asset instead of BTC/USD.
5. The request slot is cleared; Alice cannot retry with the correct price ID.
