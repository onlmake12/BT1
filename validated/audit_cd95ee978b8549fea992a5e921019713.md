### Title
Unauthenticated `ConnectionSync` Forwarding Allows Victim Node to Be Used as Message Proxy — (`network/src/protocols/hole_punching/component/connection_sync.rs`)

### Summary
`ConnectionSyncProcess::execute` forwards a `ConnectionSync` message to an attacker-controlled `to` PeerId when the message arrives with an empty `route`, without verifying the message belongs to a legitimate in-flight hole-punching session. Any unprivileged peer can exploit this to make the victim relay `ConnectionSync` messages to arbitrary connected peers.

### Finding Description

In `ConnectionSyncProcess::execute`, when `content.route` is empty (`route.last()` returns `None`) and `content.to != self_peer_id`, the node unconditionally calls `self.forward_sync(&content.to)`: [1](#0-0) 

`forward_sync` then looks up `content.to` in the peer registry and, if found, sends the message directly to that peer: [2](#0-1) 

The only guards present are:
1. A session-level rate limiter of **30 req/sec** per `(session_id, item_id)`.
2. A forward rate limiter of **1 req/sec** per `(from, to, item_id)` tuple. [3](#0-2) [4](#0-3) 

The forward rate limiter is trivially bypassed by varying the attacker-controlled `from` field — each distinct `(from, to, item_id)` tuple gets its own independent 1/sec bucket. The session-level limiter (30/sec) is the only binding constraint.

Critically, there is **no check against `inflight_requests` or `pending_delivered`** before forwarding. These maps exist on the `HolePunching` struct: [5](#0-4) 

But `inflight_requests` is only written in `notify()` and never consulted in the `ConnectionSync` forwarding path. `pending_delivered` is only consulted in the branch where `self_peer_id == content.to`, not in the forwarding branch.

### Impact Explanation
An attacker connected to the victim node can send up to 30 `ConnectionSync` messages/sec (session rate limit), each with a different `from` PeerId and the same `to=target_peer_id`. The victim will relay all of them to the target peer. This enables:
- Using the victim as a message proxy to reach peers the attacker is not directly connected to.
- Flooding a target peer with spurious `ConnectionSync` messages (up to 30/sec per attacker session, multiplied by the number of attacker sessions).
- Network congestion on the victim-to-target link.

### Likelihood Explanation
The exploit requires only that the attacker knows a PeerId connected to the victim — information that is routinely discoverable via the peer discovery protocol. No special privileges, keys, or hashpower are needed. The attacker only needs a single P2P connection to the victim.

### Recommendation
Before forwarding a `ConnectionSync` in the `None`-route branch, verify that a corresponding entry exists in `inflight_requests` (keyed by `content.to` or `content.from`) or that the `(from, to)` pair matches a known pending hole-punching session. If no such entry exists, return `StatusCode::Ignore` instead of forwarding.

### Proof of Concept
1. Attacker connects to victim node via P2P.
2. Attacker discovers `target_peer_id` (a peer connected to victim) via peer discovery.
3. Attacker sends `ConnectionSync { from: random_peer_id_N, to: target_peer_id, route: [] }` repeatedly (varying `from` each time to bypass the forward rate limiter).
4. Victim's `execute()` hits the `None` branch at line 100, finds `self_peer_id != target_peer_id`, calls `forward_sync(&target_peer_id)`.
5. Victim sends `ConnectionSync` to `target_peer_id` for each message received, up to 30/sec.
6. Target receives spurious `ConnectionSync` messages; its own `execute()` returns `Ignore` (line 170-171) since `content.from` is not in `pending_delivered`, but the relay traffic is real. [1](#0-0)
