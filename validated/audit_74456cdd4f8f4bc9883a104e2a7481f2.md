### Title
Unbounded `pending_get_block_proposals` Accumulation Causes DoS in `prune_tx_proposal_request` Timer Loop - (File: sync/src/relayer/mod.rs)

### Summary
An unprivileged P2P peer can repeatedly send `GetBlockProposal` relay messages containing proposal IDs that do not exist in the local tx-pool. These IDs are cached without any total-size cap in the `pending_get_block_proposals` `DashMap`. On every timer tick, `prune_tx_proposal_request` drains and iterates over the entire accumulated map in a single unbounded pass, issuing one bulk `fetch_txs` call for all entries. With multiple peers or sustained flooding, this causes unbounded memory growth and repeated CPU/IO spikes proportional to the map size.

### Finding Description

`GetBlockProposalProcess::execute()` enforces a per-message limit on the number of proposal IDs: [1](#0-0) 

Proposal IDs that are not found in the tx-pool are then inserted into `pending_get_block_proposals` with no total-size guard: [2](#0-1) 

The insertion function itself has no cap: [3](#0-2) 

The backing field is a plain `DashMap` with no capacity limit: [4](#0-3) 

On every timer tick, `prune_tx_proposal_request` drains the entire map and issues a single `fetch_txs` call for all accumulated IDs, then iterates over all results: [5](#0-4) 

### Impact Explanation

An attacker controlling one or more peers can:

1. Send `GetBlockProposal` messages at the rate-limiter ceiling (30 msg/sec per peer) each carrying up to `max_block_proposals_limit × max_uncles_num` = 1 500 × 2 = 3 000 unique proposal IDs on mainnet.
2. Use proposal IDs that are absent from the victim's tx-pool so every ID is cached.
3. Across multiple peers and timer intervals, `pending_get_block_proposals` grows without bound.
4. Each timer tick forces the node to process the entire accumulated set in one synchronous pass: a large `fetch_txs` RPC to the tx-pool actor, followed by nested iteration over all (proposal-id, peer-set) pairs.

The result is progressive memory exhaustion and periodic CPU/IO spikes that degrade or halt normal block-relay and tx-pool operations — a resource-accounting DoS analogous to iterating an unbounded winners array in a single call.

### Likelihood Explanation

The per-message limit (`max_block_proposals_limit × max_uncles_num`) and the 30-req/sec rate limiter per peer are both enforced, but they bound only the per-message input, not the total accumulated state. A single peer can inject up to ~90 000 unique entries per second before the rate limiter fires. With even a handful of colluding peers the map grows rapidly. No existing code path evicts or caps `pending_get_block_proposals` between timer ticks.

### Recommendation

Add a maximum-size cap to `pending_get_block_proposals`. When the map reaches the cap, either drop new entries or disconnect the offending peer. Additionally, `prune_tx_proposal_request` should process entries in bounded batches rather than draining the entire map at once, mirroring the chunked approach already used for `MAX_RELAY_TXS_BYTES_PER_BATCH` in the same function.

### Proof of Concept

1. Connect to a CKB mainnet/testnet node as an unprivileged relay peer.
2. Repeatedly send `GetBlockProposal` relay messages, each containing 3 000 unique `ProposalShortId` values that are not present in the target node's tx-pool. The IDs can be derived from arbitrary 32-byte hashes.
3. Observe `pending_get_block_proposals` growing in memory (via metrics or heap profiling).
4. On each `TX_PROPOSAL_TOKEN` timer tick (token 0), observe `prune_tx_proposal_request` issuing a single `fetch_txs` call for all accumulated IDs and iterating over the full result set, causing measurable latency spikes in the tx-pool actor and the relay event loop. [6](#0-5) [7](#0-6)

    async fn prune_tx_proposal_request(&self, nc: &Arc<dyn CKBProtocolContext + Sync>) {
        let get_block_proposals = self.shared().state().drain_get_block_proposals();
        let tx_pool = self.shared.shared().tx_pool_controller();

        let fetch_txs = tx_pool
            .fetch_txs(
                get_block_proposals
                    .iter()
                    .map(|kv_pair| kv_pair.key().clone())
                    .collect(),
            )
            .await;
        if let Err(err) = fetch_txs {
            debug_target!(
                crate::LOG_TARGET_RELAY,
                "relayer prune_tx_proposal_request internal error: {:?}",
                err,
            );
            return;
        }

        let txs = fetch_txs.unwrap();

        let mut peer_txs = HashMap::new();
        for (id, peer_indices) in get_block_proposals.into_iter() {
            if let Some(tx) = txs.get(&id) {
                for peer_index in peer_indices {
                    let tx_set = peer_txs.entry(peer_index).or_insert_with(Vec::new);
                    tx_set.push(tx.clone());
                }
            }
        }

        let mut relay_bytes = 0;
        let mut relay_proposals = Vec::new();
        for (peer_index, txs) in peer_txs {
            for tx in txs {
                let data = tx.data();
                let tx_size = data.total_size();
                if relay_bytes + tx_size > MAX_RELAY_TXS_BYTES_PER_BATCH {
                    send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals))
                        .await;
                    relay_bytes = tx_size;
                } else {
                    relay_bytes += tx_size;
                }
                relay_proposals.push(data);
            }
            if !relay_proposals.is_empty() {
                send_block_proposals(nc, peer_index, std::mem::take(&mut relay_proposals)).await;
                relay_bytes = 0;
            }
        }
