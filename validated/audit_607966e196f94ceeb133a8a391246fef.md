### Title
Deduplication Bypass via `work_id=0` Sentinel in `update_block_template` — (`miner/src/client.rs`)

---

### Summary

The `update_block_template` function contains a logic error in its deduplication guard: the condition `id == 0` permanently bypasses deduplication for any block template carrying `work_id=0`. Because the notify-mode HTTP listener (`127.0.0.1:8888`) has no authentication, any local process can POST a crafted `BlockTemplate` with `work_id=0` and cause the miner to re-dispatch `Works::New` on every push, indefinitely.

---

### Finding Description

`update_block_template` uses `AtomicU64::fetch_update` with this closure:

```rust
let updated = |id| {
    if id != work_id || id == 0 {
        Some(work_id)
    } else {
        None
    }
};
``` [1](#0-0) 

`current_work_id` is initialized to `0`: [2](#0-1) 

When `fetch_update` returns `Ok(...)` (closure returned `Some`), `Works::New` is sent to workers: [3](#0-2) 

The `id == 0` branch was intended to force dispatch on the very first call (before any real work arrives). However, if an incoming template has `work_id=0`, the atomic is updated to `0` — leaving `current_work_id` at `0` permanently. Every subsequent call with `work_id=0` evaluates `0 != 0 || 0 == 0` = `true`, so `Works::New` is dispatched unconditionally on every push, bypassing deduplication entirely.

The notify-mode HTTP handler has **no authentication**:

```rust
async fn handle(
    client: Client,
    req: Request<hyper::body::Incoming>,
) -> Result<Response<Empty<Bytes>>, Error> {
    let body = BodyExt::collect(req).await?.aggregate();
    if let Ok(template) = serde_json::from_reader(body.reader()) {
        client.update_block_template(template);
    }
    Ok(Response::new(Empty::new()))
}
``` [4](#0-3) 

Any process that can connect to `127.0.0.1:8888` can POST arbitrary `BlockTemplate` JSON with no credential check.

---

### Impact Explanation

An attacker with local process access (same machine) can:

1. POST a crafted `BlockTemplate` JSON with `work_id: 0` to `http://127.0.0.1:8888`.
2. `update_block_template` dispatches `Works::New` with the attacker-controlled template.
3. `current_work_id` stays at `0`; every subsequent push with `work_id=0` re-dispatches.
4. Workers receive `WorkerMessage::NewWork` on every push and mine on the attacker-controlled template.

The partial mitigation in `notify_new_work` — the `legacy_work` LRU cache — only deduplicates by parent hash of **already-submitted** blocks: [5](#0-4) 

A fresh attacker-crafted template with a novel parent hash bypasses this check entirely. The miner wastes PoW cycles on stale or attacker-controlled block templates and may submit blocks that benefit the attacker.

---

### Likelihood Explanation

- The listener binds to `127.0.0.1`, so the attacker must be a local process on the same machine. This is realistic in shared cloud VMs, container environments, or any multi-user server.
- No authentication is required — a single `curl` command suffices.
- The `work_id=0` value is trivially crafted in JSON.
- The bug is deterministic and reproducible: unit-testing `update_block_template` with `current_work_id=0` and incoming `work_id=0` will confirm `Works::New` is sent on every invocation.

---

### Recommendation

Remove the `|| id == 0` branch from the deduplication closure. The initial-state case is already handled correctly by `id != work_id` (since the first real `work_id` from the node will differ from `0`). Alternatively, represent the uninitialized state with `Option<u64>` or a dedicated sentinel that cannot appear as a valid `work_id`. Additionally, add authentication (e.g., a shared secret or local-socket transport) to the notify endpoint.

**Corrected closure:**
```rust
let updated = |id| {
    if id != work_id {
        Some(work_id)
    } else {
        None
    }
};
```

### Proof of Concept

```rust
// Pseudocode unit test
let client = Client::new(...); // current_work_id starts at 0
let template_0 = BlockTemplate { work_id: 0.into(), .. };

client.update_block_template(template_0.clone());
// Works::New received — expected on first call

client.update_block_template(template_0.clone());
// Works::New received AGAIN — dedup bypassed, bug confirmed

client.update_block_template(template_0.clone());
// Works::New received AGAIN — perpetual re-dispatch
```

```bash

# Repeat in a loop; miner re-dispatches on every iteration
```
