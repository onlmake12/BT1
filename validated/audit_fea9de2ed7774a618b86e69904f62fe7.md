### Title
`Scheduler._processFeesAndPayKeeper()` Uses `tx.gasprice` Ignoring L1 Data Fee on L2 Chains, Causing Keeper Undercompensation and Price Feed Staleness — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler._processFeesAndPayKeeper()` reimburses keepers using only `tx.gasprice * gasUsed`, which on L2 chains (Optimism, Base, Arbitrum) captures only the L2 execution gas cost. The L1 data fee — the cost of posting calldata to Ethereum — is not included in `tx.gasprice` on OP Stack and Arbitrum chains and is therefore never reimbursed. This is the direct Pyth analog of the FeeOracleV1 bug: both assume a single gas price covers all transaction costs, ignoring a separate, significant data-availability fee market.

---

### Finding Description

In `Scheduler.sol`, `_processFeesAndPayKeeper()` computes the keeper reimbursement as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [1](#0-0) 

On OP Stack chains (Optimism, Base), `tx.gasprice` is the L2 execution gas price, typically 0.001–0.01 gwei. Every L2 transaction additionally incurs an **L1 data fee** charged by the `GasPriceOracle` precompile (`0x420...000F`) for posting calldata to Ethereum. This fee is:

- Paid by the keeper's EOA at the EVM level, deducted from their ETH balance
- **Not** reflected in `tx.gasprice`
- **Not** captured by `gasleft()` accounting
- Proportional to calldata size and L1 base fee, and can be 10–100× the L2 execution cost

The static `singleUpdateKeeperFeeInWei` is an admin-set constant that does not dynamically track L1 gas price fluctuations: [2](#0-1) 

The `GAS_OVERHEAD` constant of 30,000 is described as "a rough estimate of the tx overhead" and does not account for L1 data costs either: [3](#0-2)

### Impact Explanation

On every L2 chain where Scheduler is deployed, keepers calling `updatePriceFeeds()` pay an L1 data fee that is never reimbursed. As L1 base fees fluctuate (e.g., during Ethereum congestion), the uncompensated L1 cost can exceed the total keeper reimbursement. Rational keepers will stop submitting updates when unprofitable, causing subscribed price feeds to go stale. Protocols that depend on Scheduler-maintained feeds (e.g., lending protocols, perpetuals) will then operate on stale prices, potentially enabling liquidation manipulation or mispriced trades — a direct financial loss to end users.

---

### Likelihood Explanation

Pyth's Scheduler is designed for EVM chains and Pyth is already deployed on Optimism, Base, Arbitrum, and other L2s. The L1 data fee on OP Stack chains is a well-documented, always-present cost. Any keeper running on these chains will observe net losses on each `updatePriceFeeds()` call during periods of elevated L1 gas prices. No special attacker action is required — the undercompensation is structural and automatic.

---

### Recommendation

For OP Stack chains, query the `GasPriceOracle` precompile to obtain the L1 data fee for the current transaction and add it to the keeper reimbursement:

```solidity
// OP Stack: 0x420000000000000000000000000000000000000F
uint256 l1Fee = IGasPriceOracle(0x420...000F).getL1Fee(msg.data);
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice + l1Fee;
```

For Arbitrum, use `ArbGasInfo.getL1BaseFeeEstimate()`. Alternatively, make `singleUpdateKeeperFeeInWei` dynamically adjustable by keepers (similar to Fortuna's `adjust_fee_if_necessary` pattern) so it can absorb L1 data cost volatility.

---

### Proof of Concept

1. Deploy `SchedulerUpgradeable` on Optimism (or Base).
2. Create a subscription with 2 price feeds; fund it with the minimum balance.
3. As a keeper EOA, call `updatePriceFeeds(subscriptionId, updateData)`.
4. Observe: the keeper receives `gasCost = gasUsed * tx.gasprice` (L2 execution cost, e.g., ~0.00001 ETH).
5. Inspect the keeper's ETH balance delta: actual spend = L2 execution cost + L1 data fee (e.g., ~0.0003 ETH at 20 gwei L1 base fee for ~1 KB calldata).
6. The keeper's net loss per call = L1 data fee − `singleUpdateKeeperFeeInWei * numFeeds`.
7. After several updates, the keeper's balance is drained; they stop submitting. Price feeds go stale beyond `PAST_TIMESTAMP_MAX_VALIDITY_PERIOD` (1 hour). [6](#0-5)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
