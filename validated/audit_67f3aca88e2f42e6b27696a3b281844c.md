### Title
Tx-Pool `total_tx_size`/`total_tx_cycles` Accounting Corrupted When Ancestor-Eviction Occurs During Entry Insertion — (`tx-pool/src/component/pool_map.rs`)

---

### Summary

In `PoolMap::add_entry`, the pool's aggregate size and cycle counters (`total_tx_size`, `total_tx_cycles`) are pre-computed before a conditional eviction step (`check_and_record_ancestors`) and then unconditionally written back after it. When `check_and_record_ancestors` evicts existing entries — correctly subtracting their sizes/cycles from the running totals — those corrections are silently overwritten by the stale pre-eviction values. The result is that `total_tx_size` and `total_tx_cycles` are permanently inflated by the aggregate size/cycles of every evicted entry, causing the pool to believe it is fuller than it actually is.

---

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```rust
// Step 1 — snapshot new totals BEFORE any eviction
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

// Step 2 — may evict entries; each eviction calls update_stat_for_remove_tx,
//           which correctly subtracts from self.total_tx_size / self.total_tx_cycles
evicts = self.check_and_record_ancestors(&mut entry)?;

// ... insert, record edges, etc. ...

// Step 3 — unconditionally overwrite with the STALE pre-eviction snapshot
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
``` [1](#0-0) 

`updated_stat_for_add_tx` computes `self.total_tx_size + entry.size` at the moment it is called — before any eviction has occurred. [2](#0-1) 

`check_and_record_ancestors` can evict existing pool entries (via `remove_entry_and_descendants`) when the new transaction's ancestor count exceeds `max_ancestors_count` but some ancestors are "cell-ref parents" that can be displaced: [3](#0-2) 

Each eviction calls `remove_entry`, which calls `update_stat_for_remove_tx`, correctly reducing `self.total_tx_size` and `self.total_tx_cycles`: [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, Step 3 overwrites `self.total_tx_size` with the stale snapshot, discarding the correct post-eviction value. The inflation equals the total serialized size (and cycles) of all evicted entries.

**Concrete example:**
- Pool state: `total_tx_size = 1000`
- New entry size: `100` → `total_tx_size` snapshot = `1100`
- `check_and_record_ancestors` evicts entries totalling `200` bytes → `self.total_tx_size` correctly becomes `800`
- Step 3 writes `self.total_tx_size = 1100` (should be `900`)
- Permanent inflation: **200 bytes**

---

### Impact Explanation

`total_tx_size` is the sole guard used by `limit_size` to decide whether to evict transactions from the pool: [6](#0-5) 

An inflated `total_tx_size` causes `limit_size` to evict additional legitimate, fee-paying transactions that would otherwise remain in the pool. This degrades miner revenue and disrupts normal transaction propagation. The inflation persists until the pool is cleared or the node is restarted, because no subsequent operation recomputes the totals from scratch (the `recompute_total_stat` fallback is only triggered on underflow, not overflow). [7](#0-6) 

Additionally, `estimate_fee_rate` uses pool size data derived from these counters, so fee estimates become incorrect for the duration of the inflation.

---

### Likelihood Explanation

The eviction path inside `check_and_record_ancestors` is triggered when:
1. A new transaction's ancestor count exceeds `max_ancestors_count`, AND
2. Some ancestors are "cell-ref parents" — i.e., in-pool transactions that hold a `cell_dep` pointing to an output the new transaction consumes as an input.

An unprivileged tx-pool submitter can deliberately engineer this scenario:
1. Submit a batch of transactions `T1…Tn` each referencing a specific live cell `O` as a `cell_dep`.
2. Submit a new transaction `T_new` that spends `O` as an input and has a long ancestor chain.

When `T_new` is submitted, `T1…Tn` become `cell_ref_parents`. If the ancestor count exceeds the limit, the eviction branch fires, corrupting the counters. No special privilege, key material, or majority hash power is required — only the ability to submit transactions to the RPC (`send_transaction`).

---

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes, so that any evictions are already reflected:

```rust
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Compute totals here, after evictions have already updated self.total_tx_size/cycles
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, accumulate the evicted sizes/cycles inside `check_and_record_ancestors` and subtract them from the pre-computed snapshot before writing it back.

---

### Proof of Concept

1. Configure a CKB node with `max_ancestors_count = N` (default 25).
2. Submit `N` transactions `T1…TN`, each with a `cell_dep` pointing to live cell `O`.
3. Submit a transaction `T_new` that:
   - Spends cell `O` as an input (making `T1…TN` cell-ref parents of `T_new`).
   - Has a chain of `N` unrelated ancestors (so ancestor count = `N+1 > max_ancestors_count`).
4. `add_entry` for `T_new` enters the eviction branch, removes some `Ti` entries (each calling `update_stat_for_remove_tx`), then overwrites `self.total_tx_size` with the pre-eviction snapshot.
5. Query `get_pool_info` via RPC and observe `total_tx_size` is larger than the sum of actual pool entry sizes.
6. Submit additional high-fee transactions; observe that `limit_size` evicts them despite the pool having physical space, because the inflated counter exceeds `max_tx_pool_size`. [8](#0-7)

    fn update_stat_for_remove_tx(&mut self, tx_size: usize, cycles: Cycle) {
        match (
            self.total_tx_size.checked_sub(tx_size),
            self.total_tx_cycles.checked_sub(cycles),
        ) {
            (Some(total_tx_size), Some(total_tx_cycles)) => {
                self.total_tx_size = total_tx_size;
                self.total_tx_cycles = total_tx_cycles;
            }
            _ => {
                if let Some((total_tx_size, total_tx_cycles)) = self.recompute_total_stat() {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, recomputed size {} cycles {}",
                        tx_size, cycles, total_tx_size, total_tx_cycles
                    );
                    self.total_tx_size = total_tx_size;
                    self.total_tx_cycles = total_tx_cycles;
                } else {
                    error!(
                        "tx-pool total stats underflowed when removing size {} cycles {}, and recomputing overflowed",
                        tx_size, cycles
                    );
                }
            }
        }
    }
