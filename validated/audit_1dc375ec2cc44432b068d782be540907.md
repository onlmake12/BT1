### Title
Excess ETH Sent to Entropy `request()`/`requestV2()` Is Permanently Captured by Protocol, Never Refunded to Caller — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `request()`, `requestWithCallback()`, and `requestV2()` functions in `Entropy.sol` are `payable` and accept native ETH as fee payment. Any ETH sent above the exact required fee (`getFee(provider)`) is permanently captured into the protocol's `accruedPythFeesInWei` pool and is never refunded to the caller. This is the same vulnerability class as the reference report: a payable function silently absorbs native coin that the user did not intend to donate.

---

### Finding Description

In `requestHelper()`, the fee accounting logic is:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `accruedPythFeesInWei`. If a user sends `msg.value = requiredFee + X` (any overpayment), the surplus `X` is silently absorbed into the Pyth protocol fee pool — it is not returned to the caller and cannot be recovered by the user.

The code comments at the public entry points explicitly acknowledge this:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) [3](#0-2) 

This affects all three public entry points: `request()`, `requestWithCallback()`, and all overloads of `requestV2()`. [4](#0-3) 

---

### Impact Explanation

Any user who sends more ETH than the exact required fee permanently loses the surplus. The surplus is credited to `accruedPythFeesInWei`, which is controlled by the Pyth admin and can only be withdrawn by the admin via `withdrawFee()` — it is not returned to the original sender. From the user's perspective, the excess ETH is irrecoverably lost.

This is directly analogous to the reference report: a payable function silently retains native coin that the caller did not intend to donate to the protocol.

---

### Likelihood Explanation

Several realistic scenarios cause overpayment:

1. **Fee race condition**: A user calls `getFee(provider)` off-chain, then the provider increases their `feeInWei` before the transaction is mined. The user's `msg.value` was computed against the old fee but is now above the new required fee — the excess is captured.
2. **Defensive overpayment**: Users or integrating contracts commonly send slightly more than the quoted fee to avoid reverts from minor fee changes. Every such call permanently loses the surplus.
3. **Programmatic callers**: Smart contracts that call `requestWithCallback()` and round up `msg.value` for safety will silently lose the rounded-up amount on every call.

---

### Recommendation

Refund any excess ETH to `msg.sender` at the end of `requestHelper()`:

```solidity
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "refund failed");
}
```

This mirrors the correct pattern already used in `PythLazer.sol`:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [5](#0-4) 

---

### Proof of Concept

1. Provider registers with `feeInWei = 1000 wei`; Pyth fee is `100 wei`; `requiredFee = 1100 wei`.
2. User calls `requestV2{value: 2000}(provider, 0)` — intentionally or accidentally overpaying by `900 wei`.
3. Inside `requestHelper`:
   - `providerInfo.accruedFeesInWei += 1000`
   - `_state.accruedPythFeesInWei += (2000 - 1000) = 1000` ← 900 wei of user's surplus absorbed here
4. User receives a valid sequence number. Their `900 wei` surplus is gone — credited to Pyth's admin-controlled fee pool, not returned.
5. The user has no mechanism to recover the `900 wei`. [6](#0-5)
