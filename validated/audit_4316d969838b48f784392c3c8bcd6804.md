Audit Report

## Title
Missing Per-Peer Rate Limiter in `LightClientProtocol` Allows Unbounded I/O Exhaustion via `GetBlocksProof` — (`util/light-client-protocol-server/src/lib.rs`, `util/light-client-protocol-server/src/components/get_blocks_proof.rs`)

## Summary

`LightClientProtocol` contains no per-peer rate limiter. Any unprivileged P2P peer can send a continuous stream of valid `GetBlocksProof` messages, each requesting up to 1000 block hashes, triggering up to 3000 synchronous RocksDB reads plus an `mmr.gen_proof(1000 positions)` call per message with zero throttle or ban. This directly contrasts with `Relayer` and `HolePunching`, which both enforce a `governor::RateLimiter` keyed by `(PeerIndex, message_type)` before any processing.

## Finding Description

**Root cause — no rate limiter field in `LightClientProtocol`:**

The struct holds only a `Shared` field with no rate-limiting state: [1](#0-0) 

**No rate-limit check in `received` or `try_process`:**

`received` decodes the message and immediately calls `try_process`, which dispatches directly to `GetBlocksProofProcess::execute` with no quota check: [2](#0-1) [3](#0-2) 

**`GetBlocksProofProcess::execute` performs up to 3000 DB reads per message:**

After structural validation, the handler iterates over all found block hashes and calls `get_block_header`, `get_block_uncles`, and `get_block_extension` for each — up to 3 × 1000 = 3000 synchronous RocksDB reads: [4](#0-3) 

The limit constant is confirmed at 1000: [5](#0-4) 

**`reply_proof` then calls `mmr.gen_proof` with up to 1000 positions (O(N log N)):** [6](#0-5) 

**Contrast: `Relayer` enforces 30 req/sec per `(PeerIndex, message_type)` before any dispatch:** [7](#0-6) [8](#0-7) 

**Contrast: `HolePunching` checks its rate limiter in `received` before any processing:** [9](#0-8) 

**LightClient is enabled by default** in `default_support_all_protocols`: [10](#0-9) 

## Impact Explanation

Each `GetBlocksProof(1000 valid hashes)` message causes up to 3000 synchronous RocksDB reads and one O(N log N) MMR proof generation. A single attacker peer sending these in a tight loop can saturate the full node's disk I/O and CPU, degrading or blocking service for honest sync peers and miners. This matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (10001–15000 points). The attacker cost is negligible: connect as a P2P peer, obtain 1000 valid block hashes from the public chain, loop.

## Likelihood Explanation

The LightClient protocol is on by default in every standard node configuration. Any peer that can establish a TCP connection can open protocol `/ckb/lightclient` (ID 120) and send `GetBlocksProof` messages. No proof-of-work, stake, or privilege is required. The attack is trivially scriptable using any P2P library that supports the tentacle/secio stack, and is locally reproducible on a dev chain with ≥1000 blocks.

## Recommendation

Add a `governor::RateLimiter<(PeerIndex, u32)>` field to `LightClientProtocol`, mirroring the pattern in `Relayer`:

```rust
type RateLimiter<T> = governor::RateLimiter<
    T,
    governor::state::keyed::HashMapStateStore<T>,
    governor::clock::DefaultClock,
>;

pub struct LightClientProtocol {
    pub shared: Shared,
    rate_limiter: RateLimiter<(PeerIndex, u32)>,
}
```

Initialize it with `Quota::per_second(NonZeroU32::new(30).unwrap())` in `LightClientProtocol::new`, and check it at the top of `try_process` (or `received`) before dispatching to any handler — consistent with `Relayer`'s existing guard at `sync/src/relayer/mod.rs` L116–123.

## Proof of Concept

1. Start a CKB full node with default config (`LightClient` enabled via `default_support_all_protocols`).
2. Mine or sync a chain of ≥1000 blocks.
3. From a separate process, open a P2P connection on protocol 120 (`/ckb/lightclient`) using the tentacle library with secio handshake.
4. Collect 1000 distinct valid main-chain block hashes (available from any public RPC or chain scan).
5. In a tight loop, send `GetBlocksProof { last_hash: <tip_hash>, block_hashes: [1000 hashes] }`.
6. Monitor the full node's disk I/O (`iostat -x 1`) and CPU (`top`). Observe that each message triggers 3000 RocksDB reads + MMR proof generation with no throttle applied.
7. Confirm that honest sync peers experience degraded block propagation throughput during the attack.

The node processes each message fully with no rate check, ban, or backpressure applied to the sending peer, as confirmed by the absence of any quota check in `received` (L55–92) and `try_process` (L96–125) of `util/light-client-protocol-server/src/lib.rs`.

            warn!("process {} from {}; result is {}", item_name, peer, status);
        } else if !status.is_ok() {
            debug!("process {} from {}; result is {}", item_name, peer, status);
