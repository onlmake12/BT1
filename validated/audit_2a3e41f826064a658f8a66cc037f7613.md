### Title
Unbounded Per-Message RocksDB Read Amplification in `BlockFilter` Due to Missing Per-Peer Rate Limiter — (`sync/src/filter/get_block_filter_hashes_process.rs`)

### Summary
`BlockFilter` has no per-peer rate limiter. Any unprivileged remote peer can send `GetBlockFilterHashes` messages at maximum rate, each triggering up to 4 000 RocksDB reads. The `Relayer` protocol has an explicit `governor::RateLimiter` keyed by `(PeerIndex, item_id)` capped at 30 req/s; `BlockFilter` has no equivalent guard.

### Finding Description

**Missing rate limiter in `BlockFilter`.**
The struct holds only `shared: Arc<SyncShared>` — no `rate_limiter` field exists. [1](#0-0) 

The `received` handler parses the message and immediately calls `self.process()` with no rate check: [2](#0-1) 

`process` calls `try_process` which dispatches directly to `GetBlockFilterHashesProcess::execute`: [3](#0-2) 

**Contrast with `Relayer`**, which declares and checks a `RateLimiter<(PeerIndex, u32)>` at 30 req/s before any handler runs: [4](#0-3) [5](#0-4) 

`grep` confirms `rate_limiter` appears zero times in `sync/src/filter/`.

**Per-message DB read amplification.**
`BATCH_SIZE = 2000`. For `start_number = 0` the parent-hash branch is skipped entirely (returns `Byte32::zero()`), so the loop runs the full 2 000 iterations: [6](#0-5) [7](#0-6) 

Each iteration issues two RocksDB reads:
- `active_chain.get_block_hash(block_number)` → `snapshot.get(COLUMN_INDEX, …)` (RocksDB snapshot read) [8](#0-7) [9](#0-8) 
- `active_chain.get_block_filter_hash(&block_hash)` → `store.get(COLUMN_BLOCK_FILTER_HASH, …)` (direct RocksDB read) [10](#0-9) [11](#0-10) 

Total: **up to 4 000 RocksDB reads per message** (2 reads × 2 000 blocks). For `start_number > 0` add 2 more for the parent lookup, giving 4 002.

`GetBlockFilterCheckPointsProcess` has the same `BATCH_SIZE = 2000` and the same missing rate limiter, compounding the surface: [12](#0-11) 

### Impact Explanation
An attacker connects N peers and sends `GetBlockFilterHashes(start_number=0)` at the maximum rate the TCP/yamux layer allows. Each message forces up to 4 000 RocksDB reads. RocksDB's block cache absorbs repeated reads of the same key range after warm-up, but the cache itself becomes a contention point (LRU lock, cache-line thrashing) and the CPU cost of message parsing, loop execution, and response serialization is unbounded per peer. With N peers the aggregate I/O and CPU load scales linearly, starving the chain-controller thread that processes new blocks and the sync protocol that services other peers.

### Likelihood Explanation
The attack requires only a standard CKB P2P connection — no PoW, no stake, no privileged key. The `Filter` protocol is advertised in the node's supported-protocols list and any peer can negotiate it. The message is tiny (8 bytes for `start_number`). The asymmetry with `Relayer` (which has been rate-limited since the governor integration) shows this was an oversight rather than an intentional design choice.

### Recommendation
Add a `governor::RateLimiter<(PeerIndex, u32)>` to `BlockFilter` mirroring the `Relayer` implementation, and check it at the top of `try_process` before dispatching to any handler. A quota of 1–2 req/s per peer per message type is sufficient for legitimate light-client sync while eliminating the amplification vector. Apply the same fix to `GetBlockFilterCheckPointsProcess` and `GetBlockFiltersProcess`.

### Proof of Concept
1. Start a CKB mainnet node with block filters enabled (≥ 2 000 blocks of filter data built).
2. Connect N peers (e.g. N = 10) each running a loop:
   ```
   loop { send GetBlockFilterHashes { start_number: 0 } }
   ```
3. Monitor RocksDB read IOPS (`rocksdb.block.cache.miss`, `rocksdb.number.keys.read`) and block-import latency.
4. Assert: read IOPS grows proportionally to N; block-import latency degrades with no floor; no peer is disconnected or rate-limited by the node.

The fix can be verified by adding the same `rate_limiter.check_key(&(peer, message.item_id()))` guard used in `Relayer::try_process` and confirming the IOPS plateau at the configured quota regardless of N.
