Audit Report

## Title
`total_tx_size`/`total_tx_cycles` Inflated by Stale Snapshot Overwrite After Evictions in `add_entry` — (`tx-pool/src/component/pool_map.rs`)

## Summary

`PoolMap::add_entry` captures a pre-eviction snapshot of `total_tx_size`/`total_tx_cycles` via `updated_stat_for_add_tx` (which takes `&self` and returns new values without mutating), then calls `check_and_record_ancestors`, which may evict pool entries and correctly subtract their sizes in-place via `update_stat_for_remove_tx`. After `check_and_record_ancestors` returns, `add_entry` unconditionally overwrites `self.total_tx_size` and `self.total_tx_cycles` with the stale pre-eviction snapshot, permanently inflating both counters by the aggregate size/cycles of all evicted transactions. Repeated triggering inflates `total_tx_size` until `limit_size` begins rejecting all new transactions with `Reject::Full`.

## Finding Description

**Root cause confirmed in code:**

`updated_stat_for_add_tx` is declared as `fn updated_stat_for_add_tx(&self, ...) -> Result<(usize, Cycle), Reject>` — it takes an immutable `&self` reference and returns a computed tuple without modifying `self.total_tx_size` or `self.total_tx_cycles`. [1](#0-0) 

In `add_entry`, the returned tuple is stored in local variables `(total_tx_size, total_tx_cycles)` at lines 210–211. At this point `self.total_tx_size` is still the original value `S`. [2](#0-1) 

`check_and_record_ancestors` at line 213 may call `remove_entry_and_descendants` → `remove_entry` → `update_stat_for_remove_tx`, which **directly mutates** `self.total_tx_size` and `self.total_tx_cycles` in-place for each evicted entry, correctly reducing them to `S - evict_size`. [3](#0-2) [4](#0-3) 

The eviction path is reachable when `ancestors_count > max_ancestors_count` but `ancestors_count - cell_ref_parents.len() <= max_ancestors_count`, meaning the excess ancestors are all `cell_ref_parents` (transactions sharing a cell dep that the new tx spends as an input). These are evicted via `remove_entry_and_descendants`. [5](#0-4) 

After `check_and_record_ancestors` returns, lines 218–219 unconditionally overwrite `self.total_tx_size = total_tx_size` (the stale `S + entry.size`) and `self.total_tx_cycles = total_tx_cycles`, discarding the correct post-eviction value `S - evict_size` and replacing it with `S + entry.size`. The inflation per cycle is `evict_size + entry.size` relative to the true pool contents. [6](#0-5) 

The inflated counter is the sole gate in `limit_size`, which loops evicting entries while `self.pool_map.total_tx_size > self.config.max_tx_pool_size` and returns `Reject::Full` for the current entry if it is evicted. [7](#0-6) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

Once `total_tx_size` is inflated past `max_tx_pool_size` (default 180 MB), `limit_size` continuously evicts legitimate pending transactions and every subsequent `send_transaction` call returns `Reject::Full`. The pool appears full while actually being nearly empty. The inflation is permanent until node restart or a `clear()` call, since no normal operation corrects the overwritten counter. An attacker targeting multiple nodes simultaneously degrades the mempool network-wide, preventing legitimate transaction propagation.

## Likelihood Explanation

No special privileges are required. The `send_transaction` RPC is open to any peer or RPC caller. The attacker submits valid, fee-paying transactions: first ≥26 transactions each using the same live cell as `cell_dep` (valid, accepted normally), then one transaction spending that cell as an input, triggering the eviction branch. Each cycle permanently inflates `total_tx_size` by ~15,600 bytes (26 × ~600 bytes). After ~11,700 iterations the counter exceeds 180 MB. The attack is repeatable with low cost (only transaction fees) and compounds with each iteration.

## Recommendation

Move the stat update to **after** `check_and_record_ancestors` returns, so the final assignment adds `entry.size`/`entry.cycles` to the already-correctly-updated `self.total_tx_size`/`self.total_tx_cycles`:

```rust
// Use updated_stat_for_add_tx only for the overflow/capacity check, discard the snapshot:
self.updated_stat_for_add_tx(entry.size, entry.cycles)?;

evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));

// Add entry.size/cycles to the post-eviction totals:
self.total_tx_size = self.total_tx_size.checked_add(entry.size).expect("overflow");
self.total_tx_cycles = self.total_tx_cycles.checked_add(entry.cycles).expect("overflow");
```

This ensures eviction subtractions performed by `update_stat_for_remove_tx` are not overwritten. Note that `updated_stat_for_add_tx` must also be changed to take `&mut self` and apply the addition directly, or the overflow check must be replicated inline.

## Proof of Concept

**Setup**: Pool with `max_ancestors_count = 25`, `max_tx_pool_size = 180 MB`.

1. Submit 26 transactions `T1…T26`, each using cell `C` as a `cell_dep`. All accepted. `total_tx_size = 26 × ~600 = ~15,600 bytes`.
2. Submit `T_spend` spending cell `C` as an input. Inside `add_entry`:
   - `updated_stat_for_add_tx` returns snapshot `total_tx_size_snapshot = 15,600 + size(T_spend)`.
   - `check_and_record_ancestors` finds 26 `cell_ref_parents`, evicts them. `update_stat_for_remove_tx` called 26 times → `self.total_tx_size = 0`.
   - Line 218 writes `self.total_tx_size = 15,600 + size(T_spend)`.
   - **Actual pool**: only `T_spend`. **Reported size**: ~16,200 bytes (inflated by ~15,600).
3. Repeat steps 1–2 `N` times. After `N` iterations, `total_tx_size ≈ N × 15,600 bytes` while the pool is nearly empty.
4. After ~11,700 iterations, `total_tx_size > 180 MB`. `limit_size` evicts all legitimate transactions; all new `send_transaction` calls return `Reject::Full`.

A unit test can verify this by constructing a `PoolMap` with `max_ancestors_count = 25`, inserting 26 mock entries sharing a cell dep, then calling `add_entry` with a transaction spending that cell, and asserting that `total_tx_size` equals only `size(T_spend)` after the call.
