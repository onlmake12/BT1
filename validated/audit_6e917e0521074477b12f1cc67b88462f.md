Audit Report

## Title
Unbounded Memory Allocation via Duplicate Indexes and Missing Byte-Size Cap in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary

`GetBlockTransactionsProcess::execute` accepts up to 32767 peer-supplied transaction indexes with no deduplication and no byte-size cap on the assembled response. An unprivileged peer can send 32767 identical indexes pointing at the same transaction, forcing the node to clone it 32767 times into a single `Vec` before any send occurs. The rate limiter operates per-request rather than per-byte, providing no bound on per-request memory allocation, enabling sustained memory exhaustion and OOM node crash.

## Finding Description

**Defect 1 — Off-by-one in the count guard:**
The guard at `get_block_transactions_process.rs:37` uses strict `>`:
```rust
if get_block_transactions.indexes().len() > MAX_RELAY_TXS_NUM_PER_BATCH {
```
`MAX_RELAY_TXS_NUM_PER_BATCH = 32767` (`mod.rs:60`). A message with exactly 32767 entries evaluates `32767 > 32767 == false` and passes unchecked.

**Defect 2 — No deduplication of indexes:**
Lines 61–71 of `get_block_transactions_process.rs` iterate every peer-supplied index and call `.cloned()` unconditionally:
```rust
let transactions = self.message.indexes().iter()
    .filter_map(|i| block.transactions().get(Into::<u32>::into(i) as usize).cloned())
    .collect::<Vec<_>>();
```
Sending `indexes = [0u32; 32767]` produces 32767 clones of `block.transactions()[0]` (the coinbase) in a single `Vec`. By contrast, `GetTransactionsProcess` (`get_transactions_process.rs:54–61`) and `GetBlockProposalProcess` (`get_block_proposal_process.rs:47–51`) both deduplicate via `HashSet` and reject if counts differ.

**Defect 3 — No `MAX_RELAY_TXS_BYTES_PER_BATCH` cap on the response:**
Lines 80–97 of `get_block_transactions_process.rs` pack the entire `transactions` vec into one message with no byte-size check before serialization and send. Both analogous handlers (`get_transactions_process.rs:87–101`, `get_block_proposal_process.rs:79–95`) enforce the 1 MiB batch cap defined at `mod.rs:61`.

**Exploit flow:**
1. Attacker establishes a TCP connection to the relay protocol (no PoW, no key required).
2. Attacker obtains any stored block hash (publicly available via block explorer or header sync).
3. Attacker sends `GetBlockTransactions { block_hash: <stored>, indexes: [0u32; 32767], uncle_indexes: [] }`.
4. Count guard passes (`32767 > 32767 == false`).
5. `execute()` clones the coinbase transaction 32767 times into a `Vec` with no byte-size check.
6. The entire vec is serialized into one `BlockTransactions` message and sent.
7. The rate limiter (30 req/s per peer, `mod.rs:91`) does not bound per-request allocation; the attacker repeats at full rate.

## Impact Explanation

Peak allocation per request: `32767 × coinbase_tx_size`. With a typical mainnet coinbase of ~500 bytes: ~16 MB per request. At 30 req/s (rate limit), a single peer drives ~480 MB/s of allocation. With multiple peers (up to `MAX_RELAY_PEERS = 128`, `mod.rs:59`), the aggregate rate is sufficient to exhaust available memory and crash the node. In the worst case (large coinbase approaching `max_block_bytes`), a single request can allocate tens of gigabytes, causing an immediate OOM crash.

**Impact: High — Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation

The attack requires only a TCP connection to the relay protocol. No proof-of-work, no cryptographic key, and no privileged role is needed. The attacker only needs a valid stored block hash, which is publicly available. The malformed message passes all existing structural and count checks. The rate limiter provides partial mitigation but does not bound per-request memory. The attack is repeatable and can be issued simultaneously from multiple peers.

## Recommendation

1. Change the count guard to `>=` to enforce the limit inclusively:
   ```rust
   if get_block_transactions.indexes().len() >= MAX_RELAY_TXS_NUM_PER_BATCH {
   ```
2. Deduplicate indexes before lookup, and reject the message if duplicates are detected:
   ```rust
   let indexes: HashSet<u32> = self.message.indexes().iter().map(Into::into).collect();
   if indexes.len() != self.message.indexes().len() {
       return StatusCode::RequestDuplicate.with_context("Request duplicate index");
   }
   ```
3. Add a `MAX_RELAY_TXS_BYTES_PER_BATCH` byte-size accumulator and split or truncate the response, matching the batching pattern in `GetTransactionsProcess::execute` and `GetBlockProposalProcess::execute`.

## Proof of Concept

