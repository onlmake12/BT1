### Title
Network Alert Signature Lacks Chain/Network Binding — Cross-Network Replay via `Verifier::verify_signatures` — (File: `util/network-alert/src/verifier.rs`)

---

### Summary

`Verifier::verify_signatures` hashes the raw `RawAlert` bytes with no chain or network identifier included. The same four Foundation public keys are compiled into every CKB binary (mainnet, testnet, preview) via a single embedded TOML file. Any unprivileged P2P peer or local RPC caller can take a legitimately signed alert from one network and replay it verbatim on any other network, where it passes signature verification and is then broadcast to all peers on that network.

---

### Finding Description

**Root cause — hash construction with no network binding:**

`calc_alert_hash()` is defined as a plain `blake2b_256` over the raw `RawAlert` molecule bytes:

```
impl<'r> packed::RawAlertReader<'r> {
    pub fn calc_alert_hash(&self) -> packed::Byte32 {
        self.calc_hash()   // = blake2b_256(self.as_slice())
    }
}
``` [1](#0-0) 

`RawAlert` contains only: `id`, `cancel`, `min_version`, `max_version`, `priority`, `notice_until`, `message`. None of these fields encode a chain ID or network name.

**Verifier uses this unbound hash directly:**

```rust
pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
    let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
    ...
    verify_m_of_n(&message, self.config.signatures_threshold, &signatures, &self.pubkeys)
``` [2](#0-1) 

**Same four Foundation keys compiled into every binary:**

`NetworkAlertConfig::default()` embeds `alert_signature.toml` at compile time via `include_bytes!`:

```rust
impl Default for Config {
    fn default() -> Self {
        let alert_config = include_bytes!("./alert_signature.toml");
        toml::from_slice(&alert_config[..]).expect("alert system config")
    }
}
``` [3](#0-2) 

The embedded file contains the same four keys for all networks:

```toml

**Entry points for the attacker:**

- `send_alert` RPC (callable by any local RPC user): [5](#0-4) 
- P2P `AlertRelayer` protocol handler, which accepts and re-broadcasts alerts from any connected peer: [6](#0-5) 

**Exploit flow:**

1. Foundation signs and broadcasts a mainnet alert (e.g., "CKB v0.105.x has a critical bug, upgrade immediately").
2. The alert is publicly visible on the mainnet P2P network.
3. An unprivileged attacker copies the `Alert` struct (raw bytes + signatures) verbatim.
4. The attacker submits it to a testnet node via `send_alert` RPC, or injects it via a P2P connection to a testnet peer.
5. The testnet `Verifier::verify_signatures` computes `blake2b_256(raw_alert.as_slice())` — identical to the mainnet hash because `RawAlert` contains no network field — and the 2-of-4 Foundation signatures verify successfully.
6. The testnet `AlertRelayer` broadcasts the alert to all testnet peers.
7. All testnet nodes display the mainnet alert as if it were a testnet alert.

The reverse direction (testnet → mainnet) is equally valid.

---

### Impact Explanation

- **Cross-network alert injection:** Any alert legitimately signed for one CKB network is unconditionally accepted by every other CKB network. An unprivileged attacker who observes a mainnet alert can force every testnet node to display it and re-broadcast it.
- **User deception:** Testnet operators and users receive alerts that were not intended for their network. A mainnet "upgrade immediately" alert replayed on testnet causes unnecessary disruption; a testnet-only "network reset" alert replayed on mainnet could cause mainnet operators to take incorrect emergency actions.
- **Alert system integrity broken:** The purpose of the alert system is to deliver authoritative, network-specific emergency messages. Cross-network replay defeats this guarantee entirely, since the Foundation cannot issue a message that is scoped to one network.

---

### Likelihood Explanation

- The Foundation has issued real alerts in the past (e.g., alert `20230001` for v0.105.x bugs is present in the test suite). Once any alert exists on one network, the replay requires zero cryptographic capability — only the ability to call `send_alert` on a node of the target network or connect to it as a P2P peer.
- The `send_alert` RPC is enabled in production configurations (`support_protocols` includes `"Alert"` by default). [7](#0-6) 
- No special privileges are required. Any party running a CKB node on any network can execute this replay.

---

### Recommendation

Bind the signed message to the network by including the genesis block hash (which is unique per network) in the hashed payload before signing and verifying:

```rust
// In calc_alert_hash / verify_signatures:
let message = blake2b_256(
    [genesis_hash.as_slice(), raw_alert.as_slice()].concat()
);
```

Alternatively, add a `chain_id` or `genesis_hash` field to `RawAlert` itself so the binding is structural and cannot be omitted.

    pub fn verify_signatures(&self, alert: &packed::Alert) -> Result<(), AnyError> {
        trace!("Verifying alert {:?}", alert);
        let message = Message::from_slice(alert.calc_alert_hash().as_slice())?;
        let signatures: Vec<Signature> = alert
            .signatures()
            .into_iter()
            .filter_map(
                |sig_data| match Signature::from_slice(sig_data.as_reader().raw_data()) {
                    Ok(sig) => {
                        if sig.is_valid() {
                            Some(sig)
                        } else {
                            debug!("invalid signature: {:?}", sig);
                            None
                        }
                    }
                    Err(err) => {
                        debug!("signature error: {}", err);
                        None
                    }
                },
            )
            .collect();
        verify_m_of_n(
            &message,
            self.config.signatures_threshold,
            &signatures,
            &self.pubkeys,
        )
        .map_err(|err| err.kind())?;
        Ok(())
    }
