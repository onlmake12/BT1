### Title
Overpayment Not Refunded in `requestPriceUpdatesWithCallback` — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.requestPriceUpdatesWithCallback` accepts `msg.value >= requiredFee` but never returns the excess to the caller. Any overpayment above `requiredFee` is silently credited to the fulfilling provider's accrued balance, causing a permanent, unrecoverable loss of user funds.

---

### Finding Description

`requestPriceUpdatesWithCallback` computes a `requiredFee` and enforces only a minimum check:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);  // line 75
if (msg.value < requiredFee) revert InsufficientFee();              // line 76
```

It then stores the provider's portion of the fee as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);       // line 84
```

If `msg.value = requiredFee + X` (overpayment of X), then `req.fee = providerFee + X`. No refund is issued.

Later, in `executeCallback`, the provider is credited:

```solidity
_state.providers[providerToCredit].accruedFeesInWei +=
    SafeCast.toUint128((req.fee + msg.value) - pythFee);            // line 161-162
```

Here `req.fee` already embeds the overpayment X, so the provider receives `providerFee + X` instead of `providerFee`. The user's X is permanently transferred to the provider with no mechanism for recovery.

Contrast this with `PythLazer.verifyUpdate`, which explicitly refunds excess:

```solidity
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);     // lazer/contracts/evm/src/PythLazer.sol line 75-77
}
```

Echo has no equivalent guard. The Entropy contract also lacks a refund, but that behavior is explicitly documented in NatSpec ("excess value is *not* refunded to the caller") — Echo carries no such disclosure.

---

### Impact Explanation

Any user who sends `msg.value > getFee(provider, callbackGasLimit, priceIds)` permanently loses the excess to the provider. The loss is proportional to the overpayment and is irreversible: there is no user-facing withdrawal function in Echo, and the funds are credited directly to `providers[providerToCredit].accruedFeesInWei`, which only the provider's fee manager can withdraw. **Severity: Medium** — partial, permanent loss of user funds with no recovery path.

---

### Likelihood Explanation

Overpayment is a realistic, common occurrence:

- Frontend fee estimation may be stale (fees are governance-adjustable).
- Users or integrating contracts may add a buffer to avoid `InsufficientFee` reverts.
- Race conditions between fee reads and transaction inclusion can cause the sent value to exceed the actual fee.
- Any contract wrapping Echo that passes `msg.value` through without exact calculation will silently overpay.

The entry path requires no privilege: any address can call `requestPriceUpdatesWithCallback` with an arbitrary `msg.value`.

---

### Recommendation

Cap the stored fee at `requiredFee` and refund the remainder before storing the request:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();

// Refund excess
if (msg.value > requiredFee) {
    (bool ok, ) = payable(msg.sender).call{value: msg.value - requiredFee}("");
    require(ok, "refund failed");
}

// Use only the exact required fee going forward
req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
```

---

### Proof of Concept

1. Provider is registered with `baseFeeInWei = 0.001 ether`, `feePerFeedInWei = 0`, `feePerGasInWei = 0`. `pythFeeInWei = 0.0001 ether`. `requiredFee = 0.0011 ether`.
2. User calls `requestPriceUpdatesWithCallback{value: 0.1 ether}(provider, publishTime, priceIds, gasLimit)`.
3. Line 76: `0.1 ether >= 0.0011 ether` — passes.
4. Line 84: `req.fee = 0.1 ether - 0.0001 ether = 0.0999 ether` (should be `0.001 ether`).
5. Provider calls `executeCallback`; line 161-162 credits `0.0999 ether` to `providers[provider].accruedFeesInWei`.
6. Provider withdraws `0.0999 ether`; user has lost `0.0989 ether` (≈ 90× the intended fee).
