### Title
Rolled-Back `insert_failure_ext` Allows Repeated Full Re-Verification of Invalid Blocks â€” (`chain/src/verify.rs`)

### Summary

When a block fails contextual verification inside `reconcile_main_chain`, `insert_failure_ext` writes `verified=Some(false)` into the uncommitted `StoreTransaction`. Because `reconcile_main_chain` returns `Err` and the outer `db_txn` is never committed, that write is silently discarded. The block's `verified` field remains `None` in the persistent store. The only compensating guard â€” `BlockStatus::BLOCK_INVALID` in the in-memory `block_status_map` â€” is incomplete: it is not checked on the full-block relay path (`accept_block`), and it is entirely lost across node restarts.

### Finding Description

**Step 1 â€” `insert_failure_ext` writes to an uncommitted transaction**

Inside `reconcile_main_chain`, when `ContextualBlockVerifier::verify` returns `Err`: [1](#0-0) 

`insert_failure_ext` sets `ext.verified = Some(false)` and calls `txn.insert_block_ext`: [2](#0-1) 

After the loop, `reconcile_main_chain` returns `Err`: [3](#0-2) 

**Step 2 â€” The outer `db_txn` is never committed**

Back in `verify_block`, the call site uses `?`: [4](#0-3) 

`db_txn.commit()` at line 359 is only reached on the success path: [5](#0-4) 

Because the `?` propagates the error, `db_txn` is dropped without commit. The `verified=Some(false)` write is rolled back. The block's `BlockExt` in the store retains `verified: None`.

**Step 3 â€” The fast-reject guard in `verify_block` cannot fire**

The guard that would short-circuit re-verification reads from the committed store: [6](#0-5) 

Since `verified` is `None` (never committed), `ext.verified` is `None`, the `let Some(verified) = ext.verified` pattern fails, and the guard is skipped entirely. The block proceeds to full re-verification.

**Step 4 â€” The compensating in-memory guard is incomplete**

`consume_unverified_blocks` does set `BLOCK_INVALID` in the in-memory map after failure: [7](#0-6) 

The compact-block relay path checks this flag: [8](#0-7) 

However, the **full-block relay path** (`accept_block`) only checks `BLOCK_STORED`, not `BLOCK_INVALID`: [9](#0-8) 

An attacker re-submitting via the full-block relay path bypasses the in-memory guard entirely.

**Step 5 â€” The guard is also lost across restarts**

`BlockStatus::BLOCK_INVALID` lives only in the in-memory `block_status_map`: [10](#0-9) 

After a restart, the map is empty. `get_block_status` falls back to `get_block_ext(...).verified`, which is `None` (never committed). The block is treated as `BLOCK_STORED` (or `UNKNOWN` if `delete_unverified_block` also removed it), and full re-verification is triggered again.

### Impact Explanation

An attacker who mines a single valid-PoW block that passes non-contextual checks but fails contextual verification (e.g., wrong `RewardVerifier` output, invalid uncle reward, or maximum-cycle script that deliberately fails) can:

1. Submit the block â€” full contextual verification runs (expensive).
2. Re-submit via the full-block relay path (bypasses `BLOCK_INVALID` check) â€” full verification runs again.
3. After any node restart â€” full verification runs again with no memory of prior failure.

Each re-submission forces the node to execute the full `ContextualBlockVerifier` pipeline, including CKB-VM script execution up to `max_block_cycles`. This causes sustained CPU exhaustion and degrades block-processing throughput across the network.

### Likelihood Explanation

The attacker must produce a valid-PoW block, which is a real cost. However, the cost is paid once; the block can be re-submitted indefinitely. The full-block relay path gap and the restart gap are both reachable from standard P2P connections with no privileged access required.

### Recommendation

1. **Commit the failure ext in a separate, immediately-committed transaction** before returning the error from `reconcile_main_chain`, so `verified=Some(false)` is durably persisted regardless of the outer transaction's fate.
2. **Check `BLOCK_INVALID` in `accept_block`** (full-block relay path) in addition to the compact-block path.
3. **Persist `BlockStatus::BLOCK_INVALID` to the database** (or derive it solely from the committed `block_ext.verified`) so it survives node restarts.

### Proof of Concept

1. Construct a block with valid PoW and valid non-contextual structure, but with an incorrect cellbase reward (fails `RewardVerifier` inside `ContextualBlockVerifier`).
2. Submit the block via the P2P full-block relay path. Observe full verification runs and fails.
3. Query `get_block_ext(block_hash).verified` â€” assert it is `None`, not `Some(false)`.
4. Re-submit the same block via the full-block relay path. Observe full verification runs again (not fast-rejected).
5. Repeat step 4 in a loop; measure CPU usage on the target node.
6. Restart the node; re-submit the block; observe full verification runs again with no memory of prior failure.

        match self.block_status_map().get(block_hash) {
            Some(status_ref) => *status_ref.value(),
            None => {
                if self.header_map().contains_key(block_hash) {

                        .get_block_ext(block_hash)

                        None => BlockStatus::UNKNOWN,
                        Some(None) => BlockStatus::BLOCK_STORED,
                        Some(Some(true)) => BlockStatus::BLOCK_VALID,
                        Some(Some(false)) => BlockStatus::BLOCK_INVALID,

                }
