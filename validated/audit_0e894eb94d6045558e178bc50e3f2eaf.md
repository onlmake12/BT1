### Title
CU Griefing via Appended Invalid Signature in `post_update_atomic` Scales with `minimum_signatures` — (`target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

`post_update_atomic` iterates over **all** signatures present in the attacker-supplied VAA bytes, not just `config.minimum_signatures`. Each iteration unconditionally invokes `secp256k1_recover` (~25k CU) plus a keccak hash (~13k CU). An unprivileged attacker can craft a VAA with `minimum_signatures` valid guardian signatures followed by one extra entry whose `guardian_index` is valid but whose `rs` bytes are corrupted, causing the loop to consume `(minimum_signatures + 1) × ~38k` CU before the transaction fails — while paying only the Solana base fee (~5,000 lamports). Because `minimum_signatures` is a governance-controlled parameter, raising it inadvertently amplifies the CU waste per failed transaction.

---

### Finding Description

The minimum-signatures gate in `post_update_atomic` is a **floor check only**: [1](#0-0) 

It verifies that the VAA carries **at least** `config.minimum_signatures` signatures, but the subsequent loop processes every signature present: [2](#0-1) 

Inside the loop, `verify_guardian_signature` unconditionally calls `secp256k1_recover` before any key-match check: [3](#0-2) 

The comment on line 639 confirms the cost: ~25k CU for recovery + ~13k CU for the keccak hash = **~38k CU per signature**.

Attack construction:
1. Obtain any real Wormhole VAA carrying `minimum_signatures` valid guardian signatures (these are publicly broadcast).
2. Append one extra signature entry: pick any unused `guardian_index` that is strictly greater than the last valid index (satisfying the `InvalidGuardianOrder` check at line 168) and that exists in the guardian set (satisfying `InvalidGuardianIndex` at line 172–174). Use arbitrary or random `rs` bytes.
3. Submit to `post_update_atomic`.

The minimum-signatures gate passes (count = `minimum_signatures + 1 ≥ minimum_signatures`). The loop runs `minimum_signatures + 1` times. The first `minimum_signatures` calls to `secp256k1_recover` succeed and the keys match. The final call either:
- succeeds but the recovered key ≠ guardian key → `InvalidGuardianKeyRecovery`, or
- fails outright → `InvalidSignature`.

Either way, all `(minimum_signatures + 1) × ~38k` CU are consumed before the transaction reverts. [4](#0-3) 

---

### Impact Explanation

On Solana, **failed transactions still consume CU from the block's CU budget** (48M CU/block). An attacker paying ~5,000 lamports per transaction can consume up to `(minimum_signatures + 1) × 38k` CU per failed tx. With the default Wormhole quorum of 13 guardians and `minimum_signatures = 13`, that is ~532k CU per transaction, allowing ~90 such transactions to fill an entire block for roughly 450,000 lamports (~$0.05 at typical SOL prices). This is a cheap, repeatable block-stuffing / griefing vector that delays legitimate price update transactions.

Critically, `set_minimum_signatures` is a governance action: [5](#0-4) 

Raising `minimum_signatures` to improve security inadvertently increases the CU waste per failed transaction linearly, amplifying the griefing surface.

---

### Likelihood Explanation

- **No privileged access required**: real VAAs are publicly available from the Wormhole guardian network.
- **Trivial to construct**: appending one extra signature entry with a valid index and random `rs` bytes is a simple byte-level operation.
- **Repeatable at low cost**: each failed transaction costs only the base Solana fee.
- **No on-chain state needed**: the attacker does not need to hold any tokens or accounts beyond a funded wallet.

---

### Recommendation

Cap the loop to process at most `config.minimum_signatures` signatures (or the quorum count, whichever is applicable), rejecting VAAs that carry more signatures than needed:

```rust
// After the minimum_signatures gate, reject excess signatures
require!(
    vaa.signature_count() <= MAX_ACCEPTED_SIGNATURES,
    ReceiverError::TooManySignatures
);
```

Alternatively, break out of the loop once `minimum_signatures` valid signatures have been verified, rather than processing every signature in the VAA. This bounds CU consumption to a fixed maximum regardless of attacker-supplied input.

---

### Proof of Concept

```
parametric test: for minimum_signatures in [1, 5, 13, 19]:
  1. build VAA with `minimum_signatures` valid guardian sigs (from real Wormhole VAA)
  2. append sig entry: guardian_index = (last_valid_index + 1), rs = [0xAA; 64]
  3. call post_update_atomic
  4. assert tx fails with InvalidGuardianKeyRecovery or InvalidSignature
  5. measure CU consumed
  6. assert CU ≈ (minimum_signatures + 1) × 38_000  (linear relationship)
```

The loop at line 164 processes all `minimum_signatures + 1` entries; `secp256k1_recover` is called for each before any early exit, confirming the linear CU scaling. [2](#0-1)
