Audit Report

## Title
TOCTOU Race in `_process_tx` Causes Spurious Tx-Pool Eviction via Concurrent Submission Paths — (File: `tx-pool/src/process.rs`)

## Summary
`_process_tx` checks for txid collision under a read lock in `pre_check`, releases that lock, runs `verify_rtx` with no lock held, then calls `submit_entry` under a write lock that does not re-check for txid collision. Two concurrent invocations for the same transaction — one via RPC and one via P2P relay — can both pass `pre_check`, both complete `verify_rtx`, and both enter `submit_entry`. The second `submit_entry` call finds the tx already present (`add_entry` returns `(false, empty_evicts)`) but still calls `limit_size` unconditionally, spuriously evicting a legitimate third-party transaction from the pool.

## Finding Description

**Three-phase unlock gap in `_process_tx`:**

```
pre_check()          ← read lock acquired + released
    check_txid_collision()   ← only collision check in the codebase
    resolve_tx()
    check_tx_fee()
                     ← READ LOCK RELEASED
verify_rtx()         ← no lock held; long async CKB-VM execution
submit_entry()       ← write lock acquired
    check_rbf()      ← re-checked (comment explicitly notes concurrent safety)
    tip_hash check   ← re-checked
    _submit_entry()  ← txid collision NOT re-checked
    limit_size()     ← called unconditionally, even when succ == false
```

`check_txid_collision` is called only inside `pre_check`'s read-lock closure and nowhere inside `submit_entry`. [7](#0-6) 

After `_submit_entry` returns, `limit_size` is called at lines 150–152 with no guard on whether the tx was actually inserted (`succ`). [8](#0-7) 

`pool_map.add_entry` returns `Ok((false, HashSet::new()))` for a duplicate — no error, no evicts from `add_entry` itself, but `limit_size` is still invoked by the caller. [4](#0-3) 

**Concurrent entry paths:**
- **Path A (RPC):** `send_transaction` → `process_tx` → `_process_tx` (direct, no queue).
- **Path B (P2P):** `RelayTransaction` → `resumeble_process_tx` → verify queue worker → `_process_tx`.

Both are independent tokio tasks. During `verify_rtx`'s await point in Path A, the runtime schedules Path B's worker. Both pass `pre_check` (pool does not yet contain the tx). Both complete `verify_rtx`. Both call `submit_entry` under the write lock (serialized):

- **First `submit_entry`:** `add_entry` returns `(true, evicts)` → tx inserted → `limit_size` called → if pool is at capacity, one legitimate tx is evicted (expected behavior).
- **Second `submit_entry`:** `add_entry` returns `(false, {})` → tx NOT inserted → `limit_size` still called → if pool is at capacity, a **second** legitimate tx is evicted unnecessarily.

Both `_process_tx` calls return `Some((Ok(verified), snapshot))`, so `after_process` fires twice: `send_result_to_relayer` is called twice and `process_orphan_tx` is called twice. [9](#0-8) 

The code itself acknowledges this pattern for `check_rbf` with the comment *"check_rbf must be invoked in `write` lock to avoid concurrent issues"* — but the same protection was never applied to `check_txid_collision`. [10](#0-9) 

## Impact Explanation

**Primary — spurious tx eviction:** When the pool is at or near its size limit, the second `limit_size` call evicts a legitimate third-party transaction that would not have been evicted without the race. An attacker who controls both an RPC connection and a P2P peer can reliably trigger this by submitting the same valid transaction on both paths simultaneously. Repeated triggering sustains a low-rate eviction DoS against specific transactions.

**Secondary — double relay and double orphan processing:** `send_result_to_relayer` fires twice, and orphan dependents are processed and potentially relayed twice, wasting peer bandwidth and processing resources.

The impact is bounded — one extra eviction per race instance, not unbounded pool corruption. This fits the **Low (501–2000 points)** category as an important correctness/performance improvement, with the double-relay aspect touching on network overhead.

## Likelihood Explanation

The race window spans the entire `verify_rtx` execution — up to `max_block_cycles` of CKB-VM time, potentially seconds for complex scripts. Both the RPC (`send_transaction`) and P2P relay interfaces are default-open. No special keys or privileges are required. Any actor who can call the RPC and connect as a P2P peer can trigger this by submitting the same valid high-cycle transaction on both paths simultaneously.

## Recommendation

Re-check for txid collision inside `submit_entry` under the write lock, before calling `_submit_entry`, mirroring the existing pattern for `check_rbf`. Additionally, guard `limit_size` behind the `succ` flag:

```rust
// Inside submit_entry's write-lock closure, before _submit_entry:
check_txid_collision(tx_pool, entry.transaction())?;

// After _submit_entry — propagate succ:
let (succ, evicts) = _submit_entry_returning_succ(tx_pool, status, entry.clone(), &self.callbacks)?;
if succ {
    tx_pool.limit_size(&self.callbacks, Some(&entry.proposal_short_id()))
        .map_or(Ok(()), Err)?;
}
```

This is consistent with the comment already present for `check_rbf` at line 104. [10](#0-9) 

## Proof of Concept

1. Connect to a CKB node as both an RPC client and a P2P peer.
2. Construct a valid transaction `T` with a script that loops near `max_tx_verify_cycles`. Fill the tx pool to near its size limit with lower-fee-rate transactions.
3. Simultaneously:
   - Send `T` via `send_transaction` RPC → triggers `_process_tx` directly.
   - Relay `T` via P2P `RelayTransaction` → triggers verify queue worker → `_process_tx`.
4. Both calls pass `pre_check` (pool does not yet contain `T`).
5. Both run `verify_rtx` concurrently during the long script execution window.
6. Both call `submit_entry`: first inserts `T` and calls `limit_size` (evicts tx A); second finds `T` already present (`succ=false`) but still calls `limit_size` (evicts tx B).
7. Observe that two legitimate transactions were evicted instead of one, and that `after_process` fired twice (double relay observable in peer logs).
