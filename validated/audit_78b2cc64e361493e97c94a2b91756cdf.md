### Title
`getTwapUpdateFee()` Charges for Only One VAA Instead of Both, Halving the Required Fee for Multi-Feed TWAP Queries - (File: `target_chains/ethereum/contracts/contracts/pyth/Pyth.sol`)

### Summary

`getTwapUpdateFee()` counts price feeds only from `updateData[0]` (the start VAA) and ignores `updateData[1]` (the end VAA). Since `parseTwapPriceFeedUpdates()` processes both VAAs — each containing `N` price feeds — the actual work is `2*N` price feed updates, but the fee charged is only `N * singleUpdateFeeInWei + transactionFeeInWei`. Every TWAP call is under-billed by exactly `N * singleUpdateFeeInWei`.

### Finding Description

`parseTwapPriceFeedUpdates()` enforces that `updateData.length == 2` (start and end VAAs), then calls `getTwapUpdateFee(updateData)` to compute the required fee.

Inside `getTwapUpdateFee()`, the comment reads:

> "We only need to count the number of updates in the first VAA since both VAAs will have the same number of price feeds."

The implementation counts `N` from `updateData[0]` only and passes that to `getTotalFee(N)`:

```solidity
// Pyth.sol L123-L151
function getTwapUpdateFee(bytes[] calldata updateData)
    public view override returns (uint feeAmount)
{
    uint totalNumUpdates = 0;
    // counts only updateData[0]
    totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(updateData[0], offset);
    return getTotalFee(totalNumUpdates);   // N * singleUpdateFeeInWei + txFee
}
```

But `parseTwapPriceFeedUpdates()` calls `extractTwapPriceInfos()` on **both** VAAs:

```solidity
// Pyth.sol L517, L525-L527
extractTwapPriceInfos(updateData[0]);   // processes N price feeds
extractTwapPriceInfos(updateData[1]);   // processes N price feeds again
```

`getTotalFee` is:

```solidity
// Pyth.sol L634-L639
function getTotalFee(uint totalNumUpdates) private view returns (uint requiredFee) {
    return (totalNumUpdates * singleUpdateFeeInWei()) + transactionFeeInWei();
}
```

The flat `transactionFeeInWei()` does not compensate for the missing `N` per-feed charges from the second VAA.

### Impact Explanation

For every `parseTwapPriceFeedUpdates` call with `N` price feeds, the caller pays:

```
Charged:  N  * singleUpdateFeeInWei + transactionFeeInWei
Correct:  2N * singleUpdateFeeInWei + transactionFeeInWei
Missing:  N  * singleUpdateFeeInWei
```

Protocol revenue from TWAP queries is permanently halved on the per-feed component. Integrators (bots, aggregators) requesting many price feeds in a single TWAP call amplify the shortfall linearly with `N`. This is a direct, quantifiable revenue leak with no compensating mechanism.

### Likelihood Explanation

`parseTwapPriceFeedUpdates` is a public, payable, permissionless entry point. Any caller can invoke it with valid Wormhole-signed TWAP update data. The under-charging is automatic and unconditional — no special conditions are required. Any integrator using TWAP for multiple price feeds simultaneously triggers the bug on every call.

### Recommendation

Count price feeds from both VAAs. Since both VAAs are guaranteed to contain the same number of price feeds (enforced by the `startPriceIds.length != endPriceIds.length` check), multiply by 2:

```solidity
function getTwapUpdateFee(bytes[] calldata updateData)
    public view override returns (uint feeAmount)
{
    uint numUpdatesPerVaa = parseWormholeMerkleHeaderNumUpdates(updateData[0], offset);
    // Both VAAs are processed; charge for both
    return getTotalFee(numUpdatesPerVaa * 2);
}
```

### Proof of Concept

1. Caller constructs two valid Wormhole-Merkle TWAP VAAs, each containing `N = 10` price feeds.
2. Caller calls `getTwapUpdateFee(updateData)` → returns `10 * singleUpdateFeeInWei + transactionFeeInWei`.
3. Caller calls `parseTwapPriceFeedUpdates{value: fee}(updateData, priceIds)` with `msg.value` equal to the above fee.
4. The contract accepts the call, processes 20 price feed updates (10 from start VAA + 10 from end VAA), and succeeds.
5. The correct fee would have been `20 * singleUpdateFeeInWei + transactionFeeInWei`.
6. Under-charge per call: `10 * singleUpdateFeeInWei`.

```text
    function getTwapUpdateFee(
        bytes[] calldata updateData
    ) public view override returns (uint feeAmount) {
        uint totalNumUpdates = 0;
        // For TWAP updates, updateData is always length 2 (start and end points),
        // but each VAA can contain multiple price feeds. We only need to count
        // the number of updates in the first VAA since both VAAs will have the
        // same number of price feeds.
        if (
            updateData[0].length > 4 &&
            UnsafeCalldataBytesLib.toUint32(updateData[0], 0) ==
            ACCUMULATOR_MAGIC
        ) {
            (
                uint offset,
                UpdateType updateType
            ) = extractUpdateTypeFromAccumulatorHeader(updateData[0]);
            if (updateType != UpdateType.WormholeMerkle) {
                revert PythErrors.InvalidUpdateData();
            }
            totalNumUpdates += parseWormholeMerkleHeaderNumUpdates(
                updateData[0],
                offset
            );
        } else {
            revert PythErrors.InvalidUpdateData();
        }

        return getTotalFee(totalNumUpdates);
    }
```

```text
        bytes[] calldata updateData,

        external
        payable
        override
        returns (PythStructs.TwapPriceFeed[] memory twapPriceFeeds)

        // TWAP requires exactly 2 updates: one for the start point and one for the end point
        if (updateData.length != 2) {
            revert PythErrors.InvalidUpdateData();
        }

        uint requiredFee = getTwapUpdateFee(updateData);
        if (msg.value < requiredFee) revert PythErrors.InsufficientFee();
```
