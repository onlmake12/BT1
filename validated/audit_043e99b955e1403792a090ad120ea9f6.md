### Title
Unrestricted `providerToCredit` Parameter After Exclusivity Period Allows Fee Theft - (File: target_chains/ethereum/contracts/contracts/echo/Echo.sol)

### Summary
`Echo.sol`'s `executeCallback` function enforces that only the assigned provider can be credited during an exclusivity window, but imposes **no restriction on `providerToCredit` after that window expires**. Any unprivileged caller can supply their own address as `providerToCredit`, redirect the entire request fee to themselves, and withdraw it — stealing funds that belong to the legitimate provider.

### Finding Description
`executeCallback` accepts a caller-supplied `providerToCredit` address and credits it with the full request fee:

```solidity
function executeCallback(
    address providerToCredit,
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
    ...
    if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
        require(
            providerToCredit == req.provider,
            "Only assigned provider during exclusivity period"
        );
    }
    ...
    _state.providers[providerToCredit].accruedFeesInWei += SafeCast
        .toUint128((req.fee + msg.value) - pythFee);
``` [1](#0-0) [2](#0-1) 

Once `block.timestamp >= req.publishTime + exclusivityPeriodSeconds`, the `require` is never reached and `providerToCredit` is completely unconstrained. There is no subsequent check that `providerToCredit == req.provider`.

Any address can register as a provider for free:

```solidity
function registerProvider(...) external override {
    ProviderInfo storage provider = _state.providers[msg.sender];
    require(!provider.isRegistered, "Provider already registered");
    provider.isRegistered = true;
    ...
}
``` [3](#0-2) 

A registered provider can set themselves as their own fee manager and then call `withdrawAsFeeManager` to drain the credited balance:

```solidity
function withdrawAsFeeManager(address provider, uint128 amount) external override {
    require(msg.sender == _state.providers[provider].feeManager, "Only fee manager");
    ...
    _state.providers[provider].accruedFeesInWei -= amount;
    (bool sent, ) = msg.sender.call{value: amount}("");
``` [4](#0-3) 

### Impact Explanation
The fee stored in `req.fee` — paid by the user at request time and intended for the assigned provider — is permanently redirected to the attacker. The legitimate provider receives nothing for fulfilling the request. If the attacker also supplies the valid `updateData` (publicly available from Hermes), they collect the fee without having been the assigned provider. This is a direct theft of provider revenue from every request whose exclusivity period has elapsed.

### Likelihood Explanation
The exclusivity period is a configurable but finite window. Any request that is not fulfilled within that window (e.g., due to provider downtime, network congestion, or deliberate delay) becomes exploitable. The attacker needs only to: (1) register as a provider (permissionless, zero cost), (2) set themselves as fee manager, (3) wait for the exclusivity period to expire, (4) fetch valid Pyth price update data from the public Hermes API, and (5) call `executeCallback` with their own address. All steps are permissionless and require no privileged access.

### Recommendation
After the exclusivity period, restrict `providerToCredit` to the originally assigned provider (`req.provider`), or at minimum require that `providerToCredit` is a registered provider and add an explicit check:

```solidity
require(
    providerToCredit == req.provider,
    "providerToCredit must be the assigned provider"
);
```

If the intent is to allow any executor to fulfill after the exclusivity period, the fee should still only be credited to `req.provider`, not to the arbitrary caller-supplied address.

### Proof of Concept
1. Attacker calls `registerProvider(0, 0, 0)` — registers with zero fees, costs nothing.
2. Attacker calls `setFeeManager(attackerAddress)` — sets themselves as fee manager.
3. A user calls `requestPriceUpdatesWithCallback(legitimateProvider, publishTime, priceIds, gasLimit)` paying `req.fee` wei.
4. Attacker waits until `block.timestamp >= publishTime + exclusivityPeriodSeconds`.
5. Attacker fetches valid `updateData` for `priceIds` from the public Hermes API.
6. Attacker calls `executeCallback(attackerAddress, sequenceNumber, updateData, priceIds)` with enough `msg.value` to cover the Pyth parsing fee.
7. `_state.providers[attackerAddress].accruedFeesInWei` is incremented by `req.fee`.
8. Attacker calls `withdrawAsFeeManager(attackerAddress, amount)` and receives the stolen fee.

The legitimate provider (`legitimateProvider`) receives zero fees for the request. [5](#0-4) [6](#0-5)
