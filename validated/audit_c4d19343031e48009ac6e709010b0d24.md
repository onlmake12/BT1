### Title
Inbound Peer Can Inject Arbitrary TCP Port (Including Port 0) Into Local Peer Store via `GetNodes.listen_port` — (`network/src/protocols/discovery/state.rs`, `network/src/protocols/discovery/mod.rs`)

---

### Summary

`RemoteAddress::update_port` unconditionally replaces the TCP component of an inbound peer's address with any attacker-supplied `u16` value, including `0`. The resulting address is stored in the peer store without port validation, polluting it with unreachable entries and causing wasted feeler connection attempts.

---

### Finding Description

**Step 1 — Inbound session initialization.**
When an inbound peer connects, `SessionState::new` sets `remote_addr = RemoteAddress::Init(context.session.address.clone())`. [1](#0-0) 

**Step 2 — `update_port` has no port validation.**
`update_port` only guards on the `RemoteAddress::Init` variant; it performs no range or sanity check on the supplied `port` value before replacing the `Protocol::Tcp(_)` component and transitioning to `RemoteAddress::Listen`. [2](#0-1) 

**Step 3 — `received` handler calls `update_port` then immediately stores the result.**
On receiving `GetNodes { listen_port: Some(port) }`, the handler calls `update_port(port)` and, if the state is now `RemoteAddress::Listen`, calls `add_new_addr` with the modified address — no intermediate validation. [3](#0-2) 

**Step 4 — `add_new_addrs` only validates IP reachability, not port.**
`is_valid_addr` checks whether the IP is globally reachable; it does not inspect the port component at all. [4](#0-3) 

**Step 5 — `peer_store.add_addr` performs no port check.**
The only guard in `add_addr` is a ban-list lookup. Port `0` passes through and is stored. [5](#0-4) 

**Contrast with DNS seed path**, which explicitly rejects port `0`: [6](#0-5) 

---

### Impact Explanation

**Confirmed impact (local):**
- The peer store is polluted with addresses carrying port `0` (or any attacker-chosen port).
- `fetch_addrs_to_feeler` returns these addresses for feeler connection attempts, which fail immediately at the TCP level, wasting connection resources. [7](#0-6) 

**Corrected claim (network-wide propagation does NOT occur):**
The question asserts these addresses are propagated to other nodes via `Nodes` messages. This is **incorrect**. `fetch_random_addrs` — used by `get_random` to build `Nodes` responses — requires `peer_addr.connected(|t| t > addr_expired_ms)`, i.e., a prior successful connection. Addresses added via `add_addr` have `last_connected_at_ms = 0` and are therefore excluded. [8](#0-7) 

The actual impact is therefore **local peer store pollution and wasted feeler connection attempts**, bounded by `ADDR_COUNT_LIMIT` and the peer store's eviction logic.

---

### Likelihood Explanation

Any inbound peer can trigger this with a single crafted `GetNodes` message. No authentication, no PoW, no special role required. The `listen_port` field is a plain `Option<u16>` decoded directly from the wire. [9](#0-8) 

---

### Recommendation

In `update_port`, reject port values that are not valid listen ports before transitioning state:

```rust
pub(crate) fn update_port(&mut self, port: u16) {
    if port == 0 {
        return; // port 0 is not a valid listen port
    }
    if let RemoteAddress::Init(addr) = self {
        // ... existing replacement logic
    }
}
```

Alternatively, add the check in `add_new_addrs` alongside `is_valid_addr`, consistent with the existing DNS seed validation in `SeedRecord::check`. [10](#0-9) 

---

### Proof of Concept

```
1. Attacker opens inbound TCP connection to victim node.
2. Discovery protocol handshake completes; victim sets
   remote_addr = RemoteAddress::Init(<attacker_ip>:<ephemeral_port>).
3. Attacker sends GetNodes { listen_port: Some(0), ... }.
4. Victim calls update_port(0):
     RemoteAddress::Init(/ip4/<attacker_ip>/tcp/<eph>)
     → RemoteAddress::Listen(/ip4/<attacker_ip>/tcp/0)
5. Victim calls add_new_addr(session_id, (/ip4/<attacker_ip>/tcp/0, flags)).
6. is_valid_addr passes (IP is globally reachable).
7. peer_store.add_addr(/ip4/<attacker_ip>/tcp/0, flags) succeeds.
8. fetch_addrs_to_feeler returns the address; victim attempts
   TCP connect to <attacker_ip>:0, which fails immediately.
```

Unit test: call `update_port(0)` on a `RemoteAddress::Init` and assert the resulting address is rejected before `peer_store.add_addr`. [2](#0-1)

        if let RemoteAddress::Init(addr) = self {

    pub fn fetch_addrs_to_feeler<F>(&mut self, count: usize, filter: F) -> Vec<AddrInfo>
    where
        F: Fn(&AddrInfo) -> bool,
    {
        // Get info:
        // 1. Not already connected
        // 2. Not already tried in a minute
        // 3. Not connected within 3 days

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let addr_expired_ms = now_ms.saturating_sub(ADDR_TRY_TIMEOUT_MS);
        let peers = &self.connected_peers;

        let filter = |peer_addr: &AddrInfo| {
            filter(peer_addr)
                && extract_peer_id(&peer_addr.addr)
                    .map(|peer_id| !peers.contains_key(&peer_id))
                    .unwrap_or_default()
                && !peer_addr.tried_in_last_minute(now_ms)
                && !peer_addr.connected(|t| t > addr_expired_ms)
        };

        self.addr_manager.fetch_random(count, filter)
    }

    pub fn fetch_random_addrs(&mut self, count: usize, required_flags: Flags) -> Vec<AddrInfo> {
        // Get info:
        // 1. Connected within 7 days

        let now_ms = ckb_systemtime::unix_time_as_millis();
        let addr_expired_ms = now_ms.saturating_sub(ADDR_TIMEOUT_MS);

        let filter = |peer_addr: &AddrInfo| {

        };

        // get success connected addrs.
        self.addr_manager.fetch_random(count, filter)
    }
