### Title
DAO Withdrawal Header-Dep Index Truncation Mismatch Between Rust Host and C VM Causes Consensus Split - (File: `util/dao/src/lib.rs`)

---

### Summary

The Rust `DaoCalculator::transaction_maximum_withdraw` reads the full `u64` header-dep index from the witness `input_type` field, while the on-chain C VM (`dao.c`) reads only the **lowest byte** of that value. A transaction sender can craft a DAO withdrawal with an index ≥ 256 that the C VM accepts but the Rust node rejects, causing a consensus split when such a transaction is included in a block.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit-header index from the witness as a full `u64` and uses it directly to index into `header_deps()`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
// ...
.and_then(|header_dep_index| {
    rtx.transaction
        .header_deps()
        .get(header_dep_index as usize)          // full u64 used as index
        .and_then(|hash| header_deps.get(&hash))
        .ok_or(DaoError::InvalidOutPoint)
})?;
``` [1](#0-0) 

The on-chain C VM (`dao.c`) interprets the same 8-byte little-endian field by reading only its **lowest byte**. When a transaction carries `input_type = 257` (little-endian bytes `[0x01, 0x01, 0x00, …]`):

- **C VM** reads lowest byte → index **1** → resolves `header_deps[1]` = deposit block ✓
- **Rust host** reads full `u64` → index **257** → resolves `header_deps[257]` = a different block (e.g., the withdraw block) ✗

The subsequent block-number consistency check then fails in Rust:

```rust
if deposit_header.number() != deposited_block_number {
    return Err(DaoError::InvalidOutPoint);
}
``` [2](#0-1) 

This discrepancy is explicitly documented in the test `check_dao_withdraw_header_dep_index_exceeds_u8`:

```
// Position 1: correct deposit block (what C VM resolves via lowest byte).
// Position 257: withdraw block (wrong — Rust resolves this with full u64).
``` [3](#0-2) 

The test asserts `result.is_err()`, confirming the Rust node rejects a transaction the C VM accepts. [4](#0-3) 

---

### Impact Explanation

`transaction_maximum_withdraw` is called from `withdrawed_interests`, which feeds into `dao_field_with_current_epoch`: [5](#0-4) 

`dao_field_with_current_epoch` is invoked during **block validation** to verify the DAO accumulation-rate field embedded in every block header. If the Rust node fails to compute the DAO field for a block that contains a crafted-but-C-VM-valid DAO withdrawal, it will reject that block as invalid. Nodes that faithfully implement the C VM's lowest-byte semantics will accept the same block. The result is a **consensus split**: the network forks between nodes that follow the C VM and nodes running the Rust implementation. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only that a transaction sender:

1. Holds a live DAO deposit cell (no special privilege).
2. Constructs a DAO withdrawal transaction whose `WitnessArgs.input_type` encodes an index ≥ 256 (e.g., `257u64` little-endian), while placing the actual deposit block hash at `header_deps[lowest_byte]` and padding `header_deps` to make the full-u64 index also valid.
3. Gets the transaction included in a block by any miner.

This is fully within the capability of an unprivileged transaction sender. No key compromise, majority hashpower, or social engineering is required.

---

### Recommendation

**Short term:** Mask the index to its lowest byte before use, matching the C VM's behavior:

```rust
let header_dep_index =
    (LittleEndian::read_u64(&header_deps_index_data.unwrap()) & 0xFF) as usize;
```

**Long term:** Add a consensus-level validation rule that rejects any DAO withdrawal witness whose `input_type` encodes an index whose upper 7 bytes are non-zero, so both the C VM and the Rust host agree on the valid range. Extend unit-test branch coverage to assert that the Rust host and C VM produce identical accept/reject decisions for all boundary index values.

---

### Proof of Concept

1. Alice holds a DAO deposit cell confirmed in block #100. Cell data = `100u64` LE.
2. Alice builds a DAO withdrawal transaction:
   - `header_deps` = 258 entries; `header_deps[1]` = deposit block hash; `header_deps[257]` = withdraw block hash; remaining entries = dummy hashes.
   - `witnesses[0].input_type` = `257u64` LE (bytes `[0x01, 0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00]`).
3. The C VM reads lowest byte → index 1 → deposit block #100 → block-number check passes → **script succeeds**.
4. A miner includes the transaction in block B and publishes it.
5. The Rust node's `transaction_maximum_withdraw` reads full u64 → index 257 → withdraw block #200 → block-number check: `200 ≠ 100` → `DaoError::InvalidOutPoint` → **block B rejected**.
6. Nodes running the C VM accept block B; Rust nodes reject it → **chain splits**. [7](#0-6) [8](#0-7)

    pub fn dao_field_with_current_epoch(
        &self,
        rtxs: impl Iterator<Item = &'a ResolvedTransaction> + Clone,
        parent: &HeaderView,
        current_block_epoch: &EpochExt,
    ) -> Result<Byte32, DaoError> {
        // Freed occupied capacities from consumed inputs
        let freed_occupied_capacities =
            rtxs.clone().try_fold(Capacity::zero(), |capacities, rtx| {
                self.input_occupied_capacities(rtx)
                    .and_then(|c| capacities.safe_add(c))
            })?;
        let added_occupied_capacities = self.added_occupied_capacities(rtxs.clone())?;
        let withdrawed_interests = self.withdrawed_interests(rtxs)?;

        let (parent_ar, parent_c, parent_s, parent_u) = extract_dao_data(parent.dao());

        // g contains both primary issuance and secondary issuance,
        // g2 is the secondary issuance for the block, which consists of
        // issuance for the miner, NervosDAO and treasury.
        // When calculating issuance in NervosDAO, we use the real
        // issuance for each block(which will only be issued on chain
        // after the finalization delay), not the capacities generated
        // in the cellbase of current block.
        let current_block_number = parent.number() + 1;
        let current_g2 = current_block_epoch.secondary_block_issuance(
            current_block_number,
            self.consensus.secondary_epoch_reward(),
        )?;
        let current_g = current_block_epoch
            .block_reward(current_block_number)
            .and_then(|c| c.safe_add(current_g2))?;

        let miner_issuance128 = u128::from(current_g2.as_u64()) * u128::from(parent_u.as_u64())
            / u128::from(parent_c.as_u64());
        let miner_issuance =
            Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);
        let nervosdao_issuance = current_g2.safe_sub(miner_issuance)?;

        let current_c = parent_c.safe_add(current_g)?;
        let current_u = parent_u
            .safe_add(added_occupied_capacities)
            .and_then(|u| u.safe_sub(freed_occupied_capacities))?;
        let current_s = parent_s
            .safe_add(nervosdao_issuance)
            .and_then(|s| s.safe_sub(withdrawed_interests))?;

        let ar_increase128 =
            u128::from(parent_ar) * u128::from(current_g2.as_u64()) / u128::from(parent_c.as_u64());
        let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
        let current_ar = parent_ar
            .checked_add(ar_increase)
            .ok_or(DaoError::Overflow)?;

        Ok(pack_dao_data(current_ar, current_c, current_s, current_u))
    }

            .safe_sub(input_capacities)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
