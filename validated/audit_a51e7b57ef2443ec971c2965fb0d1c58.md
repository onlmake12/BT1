Audit Report

## Title
Non-IP Multiaddr Bypass in `process_listens` Allows Peer Store Pollution via Identify Protocol — (`network/src/protocols/identify/mod.rs`)

## Summary

The `process_listens` function in `IdentifyProtocol` uses `None => true` as the fallback for addresses where `multiaddr_to_socketaddr` returns `None`, unconditionally admitting non-IP multiaddrs (e.g., `/dns4/attacker.com/tcp/8115/p2p/QmXXX`) past the `global_ip_only` guard. Since `global_ip_only` is hardcoded to `true` in production and the setter is `#[cfg(test)]`-gated, any peer completing a valid identify handshake can inject up to 10 attacker-controlled DNS addresses into the victim's peer store, which are subsequently selected by the feeler path for outbound connection attempts.

## Finding Description

**Root cause — `process_listens` filter (`network/src/protocols/identify/mod.rs`, lines 139–145):**

```rust
let reachable_addrs = listens
    .into_iter()
    .filter(|addr| match multiaddr_to_socketaddr(addr) {
        Some(socket_addr) => !global_ip_only || is_reachable(socket_addr.ip()),
        None => true,   // ← non-IP addrs pass unconditionally
    })
    .collect::<Vec<_>>();
```

`multiaddr_to_socketaddr` returns `None` for `/dns4/`, `/dns6/`, and other non-IP protocol stacks. The `None => true` arm admits all of them regardless of `global_ip_only`.

**`global_ip_only` is always `true` in production:** The constructor hardcodes `global_ip_only: true` (lines 93–98), and the only setter is `#[cfg(test)]`-gated (lines 101–105), making the bypass unconditional in production.

**`add_remote_listen_addrs` writes directly to the peer store** (lines 488–494) with no further address-family validation.

**`add_addr` performs only a ban-list check** (lines 71–80 of `peer_store_impl.rs`). The ban list's `is_addr_banned` calls `multiaddr_to_socketaddr(...).unwrap_or_default()` (`ban_list.rs` lines 68–72), which returns `false` for DNS addresses — they are never banned.

**Injected addresses are initially connectable.** `AddrInfo::new` sets `last_connected_at_ms = 0` and `attempts_count = 0` (`types.rs` lines 65–76). `is_connectable` returns `true` until `attempts_count >= ADDR_MAX_RETRIES (3)`, so injected addresses are not immediately evicted by `check_purge`.

**Feeler path selects injected addresses.** `fetch_addrs_to_feeler` → `fetch_random` (`addr_manager.rs` lines 74–90) includes non-IP addresses that pass `is_connectable` and whose peer ID is not in `connected_peers`. An attacker-supplied address like `/dns4/attacker.com/tcp/8115/p2p/QmAttacker` satisfies all feeler filter conditions after the attacker disconnects.

**Asymmetry with the local-send path** (lines 217–225) confirms the `None => true` in `process_listens` is unintentional — the outbound filter already restricts the `None` case to Onion3 only:

```rust
.filter(|addr| {
    if let Some(socket_addr) = multiaddr_to_socketaddr(addr) {
        !self.global_ip_only || is_reachable(socket_addr.ip())
    } else {
        // allow /onion3 address
        addr.iter()
            .any(|protocol| matches!(protocol, Protocol::Onion3(_)))
    }
})
```

## Impact Explanation

**Medium (2001–10000 points): Suboptimal implementation of CKB state storage mechanism.**

The peer store (`ADDR_COUNT_LIMIT = 16384`) is a critical state structure driving all outbound connection decisions. Injected DNS addresses: (1) consume peer store slots until evicted after `ADDR_MAX_RETRIES = 3` failed attempts; (2) cause the victim node to attempt outbound TCP connections to attacker-controlled hostnames, wasting feeler slots and TCP handshake resources; (3) can be repeatedly re-injected per new connection from the attacker. The pollution is self-limiting and the store has large capacity, so the impact does not rise to High (no node crash, no network-wide congestion with few costs, no consensus deviation).

## Likelihood Explanation

Any peer that can complete a TCP handshake and pass the identify network-name check can trigger this. No special privileges, keys, or majority hashpower are required. The attacker only needs to send a valid `IdentifyMessage` with DNS multiaddrs in `listen_addrs`. The `received_identify` check only validates the network name and a non-zero flag (lines 541–561) — both trivially satisfied by a legitimate-looking peer. The attack is repeatable: each new connection from the attacker injects up to 10 addresses (`MAX_ADDRS = 10`, line 30).

## Recommendation

Replace the `None => true` fallback in `process_listens` with the same Onion3-only allowlist already used in the local-send path (lines 217–225):

```rust
.filter(|addr| match multiaddr_to_socketaddr(addr) {
    Some(socket_addr) => !global_ip_only || is_reachable(socket_addr.ip()),
    None => addr.iter().any(|p| matches!(p, Protocol::Onion3(_))),
})
```

This aligns the inbound filter with the outbound filter and eliminates the asymmetry.

        // do not remove addr tried in last minute

            return true;

        true
    }
