Audit Report

## Title
`check_tx_fee` Uses Raw Serialized Size Instead of Weight for `min_fee_rate` Enforcement, Allowing Cycle-Heavy Transactions to Bypass Fee-Rate Admission — (File: `tx-pool/src/util.rs`)

## Summary
`check_tx_fee` computes the minimum required fee using only the serialized byte size of a transaction, while `FeeRate` is defined as shannons per kilo-weight and weight is `max(tx_size, cycles × DEFAULT_BYTES_PER_CYCLES)`. For cycle-heavy transactions, weight can be orders of magnitude larger than `tx_size`, so the admission gate accepts transactions at a fraction of the intended minimum fee. The code comment at the site explicitly acknowledges this deviation but proceeds with it anyway as a "cheap check."

## Finding Description
`FeeRate` is documented and implemented as shannons per kilo-weight (`util/types/src/core/fee_rate.rs`, lines 3–7). The correct weight formula is `get_transaction_weight(tx_size, cycles)` = `max(tx_size as u64, (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64)` (`util/types/src/core/tx_pool.rs`, lines 298–303). In `check_tx_fee` (`tx-pool/src/util.rs`, line 45), the call is `tx_pool.config.min_fee_rate.fee(tx_size as u64)` — passing only `tx_size`, not weight. The comment at lines 42–44 acknowledges this is theoretically wrong but justifies it as a cheap pre-check. The ordering in `_process_tx` (`tx-pool/src/process.rs`, lines 715–751) confirms that `pre_check` (which calls `check_tx_fee`) runs before `verify_rtx`, so actual cycle counts are unavailable at admission time. After admission, `AncestorsScoreSortKey` and `EvictKey` both call `get_transaction_weight(entry.size, entry.cycles)` correctly (`tx-pool/src/component/entry.rs`, lines 221–240). This creates a split: the admission gate uses size-only fee rate, while all post-admission mechanisms use the correct weight-based fee rate. A transaction with small serialized size but high actual cycles passes the admission gate at a fraction of the intended minimum fee, enters the pool, forces the node to spend CPU on full script verification, and occupies pool space until eviction.

## Impact Explanation
An attacker with UTXOs locked by a high-cycle script can submit transactions with ~200-byte serialized size and ~70,000,000 cycles (the default `max_tx_verify_cycles`). With `min_fee_rate = 1,000` shannons/kilo-weight: correct weight ≈ 11,940; correct minimum fee = 11,940 shannons; actual check requires only 200 shannons — a ~60× reduction. Repeated submissions fill the pool with sub-threshold-fee-rate transactions and force the node to execute expensive script verification for each, causing sustained pool churn and CPU load. This matches the allowed High impact: **bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation
The attacker needs only normal user capabilities: UTXOs locked by any script that consumes high cycles (e.g., a tight computation loop). No admin keys, no privileged RPC access, no majority hashpower. The `send_transaction` RPC endpoint is the trigger. The attack is repeatable as long as the attacker controls UTXOs, and the cost per submission is bounded by the size-only fee (as low as ~200 shannons), not the weight-based fee (~11,940 shannons). The `max_tx_verify_cycles` limit (70M by default) caps the maximum amplification at ~60×, but even modest cycle counts (e.g., 10M cycles → weight ~1,706 vs. 200-byte size) produce significant discrepancy.

## Recommendation
Replace the raw `tx_size` argument in `check_tx_fee` with the transaction weight. Since cycles are not available at `pre_check` time, use declared cycles for relay transactions or a conservative upper bound (e.g., `max_tx_verify_cycles`) for local RPC submissions:

```rust
// In check_tx_fee, replace:
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);

// With:
let cycles = rtx.transaction.cycles().unwrap_or(max_tx_verify_cycles);
let weight = get_transaction_weight(tx_size, cycles);
let min_fee = tx_pool.config.min_fee_rate.fee(weight);
```

This ensures the admission gate is at least as strict as the eviction gate, eliminating the split behavior.

## Proof of Concept
1. Deploy a CKB script that performs a tight computation loop consuming ~70,000,000 cycles.
2. Lock a UTXO with that script.
3. Construct a transaction spending that UTXO with a single output; serialized size ≈ 200 bytes.
4. Set the transaction fee to 200 shannons.
5. Submit via `send_transaction` RPC to a node with `min_fee_rate = 1,000`.
6. Observe: the transaction is admitted (fee check passes: `1000 × 200 / 1000 = 200 ≤ 200`).
7. Verify via `get_raw_tx_pool` that the transaction is in the pool despite an effective weight-based fee rate of `1000 × 200 / 11940 ≈ 16 shannons/kilo-weight`, far below the configured `min_fee_rate = 1,000`.
8. Repeat with different UTXOs to continuously fill the pool with sub-threshold-fee-rate transactions and force repeated expensive script verification on the node.
