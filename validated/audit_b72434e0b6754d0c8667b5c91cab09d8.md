### Title
TOCTOU Race in `TxPoolService::process_tx` Allows Duplicate Expensive Script Verification via Concurrent RPC/Relay Submissions — (File: `tx-pool/src/process.rs`)

---

### Summary

`TxPoolService::process_tx` and `resumeble_process_tx` perform duplicate-existence checks under separate, immediately-released read locks, then proceed to execute expensive CKB-VM script verification (`verify_rtx`) with no lock held. Concurrent RPC callers or relay peers submitting the same transaction can all pass the duplicate check simultaneously and trigger N independent full script verifications, enabling CPU exhaustion. This is the direct CKB analog of the Solidity reentrancy pattern: **check state → external call (expensive computation) → update state**, without holding a mutex across the entire sequence.

---

### Finding Description

**Root cause — `process_tx` (lines 401–426):** [1](#0-0) 

The duplicate check at line 409 acquires and immediately releases two separate read locks:

```
verify_queue_contains  →  acquires verify_queue read lock, checks, RELEASES
orphan_contains        →  acquires orphan read lock, checks, RELEASES
```

After both locks are dropped, `_process_tx` is called: [2](#0-1) 

Inside `_process_tx`:
1. **`pre_check`** — acquires `tx_pool` read lock, calls `check_txid_collision`, then **releases** the lock.
2. **`verify_rtx`** — runs CKB-VM script execution with **no lock held** (expensive).
3. **`submit_entry`** — acquires `tx_pool` write lock and inserts. [3](#0-2) 

The gap between step 1 (read lock released) and step 3 (write lock acquired) is the vulnerable window. Two concurrent calls with the same transaction both pass `check_txid_collision` (since neither is in the pool yet), both proceed to `verify_rtx`, and both execute the full script.

The developers themselves acknowledge the need for write-lock protection for concurrent checks — the comment at line 104 reads: [4](#0-3) 

> `// check_rbf must be invoked in 'write' lock to avoid concurrent issues.`

But `check_txid_collision` is **not** re-validated under the write lock in `submit_entry`.

**Same pattern in `resumeble_process_tx` (lines 335–353):** [5](#0-4) 

`orphan_contains` and `verify_queue_contains` are checked under separate read locks, both released before `enqueue_verify_queue` acquires the write lock. For `is_proposal_tx = true` (the `notify_tx` path), `add_tx` under the write lock **removes and re-inserts** the existing entry: [6](#0-5) 

This resets the `added_time` of the entry, allowing an attacker to manipulate verify-queue ordering by racing concurrent `notify_tx` calls.

**The service spawns a new task per message**, guaranteeing concurrent execution: [7](#0-6) 

---

### Impact Explanation

- **CPU exhaustion**: An attacker submitting the same transaction N times concurrently via RPC causes N full CKB-VM script executions instead of 1. `verify_rtx` can consume up to `max_block_cycles` per call. N concurrent submissions = N × `max_block_cycles` CPU work.
- **Service degradation**: Saturating the script verifier delays processing of legitimate transactions and block assembly.
- **Verify-queue ordering manipulation** (via `notify_tx` TOCTOU): An attacker can repeatedly race concurrent `notify_tx` calls to reset a proposal tx's `added_time`, causing it to be perpetually re-prioritized and starving other entries.

**Impact: 3 — Service degradation / resource exhaustion reachable by unprivileged caller.**

---

### Likelihood Explanation

- Any unprivileged RPC caller (`send_transaction`) or relay peer (`submit_remote_tx`) can trigger this with no special privileges.
- Requires only sending the same transaction multiple times concurrently — trivial with any HTTP client or P2P peer.
- The service architecture (one task spawned per message) guarantees concurrent execution.

**Likelihood: 3 — Easily triggered by any external caller.**

---

### Recommendation

Re-validate txid collision **inside `submit_entry` under the write lock**, mirroring the existing pattern for `check_rbf`:

```rust
// Inside submit_entry, under with_tx_pool_write_lock:
check_txid_collision(tx_pool, entry.transaction())?;  // add this
let conflicts = if tx_pool.enable_rbf() { ... };
```

Alternatively, maintain an in-flight set (protected by a mutex) that is checked-and-set atomically before `verify_rtx` begins, and cleared after `submit_entry` completes, preventing duplicate concurrent verifications of the same txid.

---

### Proof of Concept

1. Prepare transaction `tx` with a script consuming near-`max_block_cycles` cycles.
2. Send N concurrent `send_transaction` RPC calls with the same `tx`.
3. All N tasks spawn concurrently (service loop, line 621).
4. All N calls reach `verify_queue_contains` → `false` (tx not yet in queue).
5. All N calls reach `orphan_contains` → `false` (tx not yet in orphan pool).
6. All N calls enter `_process_tx` → `pre_check` → `check_txid_collision` → `false` (tx not yet in pool).
7. All N calls enter `verify_rtx` with no lock held — N full CKB-VM executions run in parallel.
8. First `submit_entry` (write lock) succeeds; subsequent ones may fail at `add_pending`.
9. **Result**: N × `max_block_cycles` CPU consumed instead of 1 × `max_block_cycles`.

This directly mirrors the Solidity reentrancy pattern: `require(!executed)` → `pause.executeTransaction(...)` → `executed = true` maps to `check_txid_collision` (read lock, released) → `verify_rtx` (no lock) → `submit_entry` (write lock).

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
    }

        let verify_cache = self.fetch_tx_verify_cache(&tx).await;
        let max_cycles = declared_cycles.unwrap_or_else(|| self.consensus.max_block_cycles());
        let tip_header = snapshot.tip_header();

        let verified_ret = verify_rtx(
            Arc::clone(&snapshot),
            Arc::clone(&rtx),
            tx_env,
            &verify_cache,
            max_cycles,
            command_rx,

        if let Some(declared) = declared_cycles

                declared,
                verified.cycles,

                snapshot,
            ));

        let (ret, submit_snapshot) = self.submit_entry(tip_hash, entry, status).await;
