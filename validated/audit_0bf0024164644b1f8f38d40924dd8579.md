Audit Report

## Title
Spoofed `from` Field in Hole-Punching Messages Bypasses Forward Rate Limiter and Enables SSRF — (`File: network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

In CKB's hole-punching protocol, the `from` peer ID in `ConnectionRequest`, `ConnectionRequestDelivered`, and `ConnectionSync` messages is deserialized directly from the attacker-controlled message payload with no verification against the actual transport-layer session identity. This allows any connected peer to supply arbitrary `from` values, rendering the `forward_rate_limiter` completely ineffective and enabling polynomial message amplification across the relay network. The same spoofing allows an attacker to poison the `pending_delivered` table with attacker-chosen IP addresses, causing the victim node to initiate sustained outbound TCP connections to arbitrary hosts (SSRF).

## Finding Description

**Root cause — `from` parsed from payload, never verified against session:**

In all three message processors, `from` is deserialized from the message body without any comparison to the actual sending session:

- `connection_request.rs` L36–38: `PeerId::from_bytes(value.from().raw_data().to_vec())`
- `connection_request_delivered.rs` L38–40: same pattern
- `connection_sync.rs` L42–44: same pattern

The actual sending session is available as `self.peer` (`PeerIndex`) and is used for the per-connection `rate_limiter` keyed on `(session_id, msg.item_id())` in `mod.rs` L95–107. However, `content.from` is never compared against the peer ID of the session that delivered the message.

**Impact 1 — Forward rate-limiter bypass and amplification:**

The `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)` in all three processors (`connection_request.rs` L132–143, `connection_request_delivered.rs` L134–145, `connection_sync.rs` L85–96). The limiter is configured at 1 request/second per `(from, to, item_id)` tuple (`mod.rs` L256–257). Because `content.from` is fully attacker-controlled, an attacker can supply a fresh random `from` value in every message, producing a unique key each time. The per-connection `rate_limiter` caps the attacker at 30 messages/second from a single session (`mod.rs` L251–252), but each of those 30 messages bypasses the `forward_rate_limiter` at every relay node and is forwarded to `sqrt(N)` peers (gossip fanout, `connection_request.rs` L280–305), each of which also forwards to `sqrt(N)` more peers, up to `MAX_HOPS = 6` hops. The amplification factor is `sqrt(N)^6 = N^3`. For a network of 100 peers, a single attacker connection generates up to 30 × 10^6 forwarded messages per second.

The `route` field loop-prevention check (`connection_request.rs` L128–130) only prevents a single node from processing the same message instance twice; it does not prevent amplification across distinct messages with distinct `from` values.

**Impact 2 — `pending_delivered` poisoning / SSRF:**

When the victim node is the `to` target of a `ConnectionRequest`, `respond_delivered` is called (`connection_request.rs` L155–239). It filters `remote_listens` to TCP/IPv4/IPv6 addresses only (L196–215) but stores the attacker-supplied addresses under the attacker-controlled `from_peer_id`:

```rust
// connection_request.rs L234–237
self.protocol
    .pending_delivered
    .insert(from_peer_id, (remote_listens, now));
```

When a subsequent `ConnectionSync` arrives with the same spoofed `from`, `connection_sync.rs` L111–124 retrieves those stored addresses and spawns `try_nat_traversal` tasks for each one. The `try_nat_traversal` function (`component/mod.rs` L49–115) makes real outbound TCP connection attempts with a 30-second retry loop and up to 24 addresses per request (`ADDRS_COUNT_LIMIT`). This causes the victim node to make sustained outbound TCP connections to arbitrary attacker-specified IP:port pairs.

**Existing guards are insufficient:**

- The per-connection `rate_limiter` (30 req/sec) limits the entry rate but does not prevent amplification at relay nodes.
- The `forward_rate_limiter` is the only throttle on forwarding, and it is completely defeated by `from` rotation.
- The `HOLE_PUNCHING_INTERVAL` check in `respond_delivered` (L161–167) only deduplicates by `from_peer_id`, which is also attacker-controlled and can be rotated.
- The `inflight_requests` check in `ConnectionRequestDeliveredProcess` L160–176 requires `content.to` to be in `inflight_requests`, which limits the active SSRF path for `ConnectionRequestDelivered`, but the `ConnectionRequest` → `ConnectionSync` SSRF path has no such guard.

## Impact Explanation

The forward rate-limiter bypass enables a single connected attacker to generate polynomial message amplification across the CKB relay network. With `MAX_HOPS = 6` and gossip fanout of `sqrt(N)`, the amplification is `N^3` in network size. This directly matches the allowed High impact: **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs"** (10001–15000 points). The SSRF additionally causes victim nodes to exhaust TCP connection resources (30-second retry loops, up to 24 concurrent targets per message), compounding the congestion and potentially contributing to node instability.

## Likelihood Explanation

Any peer that can establish a single P2P connection to a CKB node can send `HolePunching` protocol messages. The hole-punching protocol is enabled by default. No special privilege, key, or majority hashpower is required. Crafting a valid molecule-encoded `ConnectionRequest` with a spoofed `from` field is trivial. The attack is repeatable and stateless from the attacker's perspective — each message with a fresh random `from` independently bypasses the rate limiter.

## Recommendation

1. **Verify `from` against the actual session identity**: In `ConnectionRequestProcess::execute()`, look up the peer ID for `self.peer` from the peer registry and assert it equals `content.from`. Reject messages where they do not match. Apply the same check in `ConnectionRequestDeliveredProcess` and `ConnectionSyncProcess`.
2. **Key the `forward_rate_limiter` on the actual sending `PeerIndex`** (`self.peer`) rather than on the payload-supplied `content.from`. This mirrors the existing per-connection `rate_limiter` design in `mod.rs` L45.
3. **Validate `listen_addrs` against the actual session remote address** before storing in `pending_delivered`, or restrict `pending_delivered` entries to only those initiated by the node itself (i.e., where `content.to` is the local peer ID and `content.from` is a peer the node has an active session with).

## Proof of Concept

```
Attacker (single connected peer) → Victim node:

Step 1 (SSRF setup): Send HolePunchingMessage::ConnectionRequest {
    from: <random_peer_id_A>,        // spoofed
    to:   <victim_peer_id>,          // victim's known peer ID
    listen_addrs: [/ip4/192.168.1.1/tcp/8080],
    max_hops: 6,
    route: [],
}
→ Victim: self_peer_id == content.to → calls respond_delivered()
→ pending_delivered[random_peer_id_A] = ([192.168.1.1:8080], now)

Step 2 (SSRF trigger): Send HolePunchingMessage::ConnectionSync {
    from: <random_peer_id_A>,        // same spoofed ID
    to:   <victim_peer_id>,
    route: [],
}
→ Victim: looks up pending_delivered[random_peer_id_A]
→ spawns try_nat_traversal() → 30-second outbound TCP retry loop to 192.168.1.1:8080

Step 3 (amplification): Repeat Step 1 with fresh random_peer_id_B, _C, _D, ...
→ Each message: forward_rate_limiter key is unique → no throttling
→ Each message forwarded to sqrt(N) peers, each of which forwards to sqrt(N) more, up to 6 hops
→ 30 msg/sec from attacker → O(N^3) forwarded messages/sec network-wide
```

pub(crate) async fn try_nat_traversal(
    bind_addr: Option<SocketAddr>,
    addr: Multiaddr,
) -> Result<(TcpStream, Multiaddr), std::io::Error> {
    let net_addr = match multiaddr_to_socketaddr(&addr) {
        Some(addr) => addr,
        None => {
            debug!("Failed to convert multiaddr to socketaddr");
            return Err(std::io::ErrorKind::InvalidInput.into());
        }
    };

    // Use a fixed interval but add a small amount of randomness
    let base_retry_interval = Duration::from_millis(200);

    // total time
    let timeout_duration = Duration::from_secs(30);
    let start_time = Instant::now();
    let mut retry_count = 0u32;
    while start_time.elapsed() < timeout_duration {
        retry_count += 1;

        // Add a small amount of random jitter (±25ms) to avoid conflicts
        // caused by continuous precise synchronization
        let jitter = Duration::from_millis(rand::random::<u64>() % 50);
        let actual_interval = if rand::random::<bool>() {
            base_retry_interval + jitter
        } else {
            base_retry_interval.saturating_sub(jitter)
        };

        let socket = create_socket(bind_addr, net_addr)?;

        match runtime::timeout(
            std::time::Duration::from_millis(200),
            socket.connect(net_addr),
        )
        .await
        {
            Ok(Ok(stream)) => {
                // try get the stored error in the underlying socket
                // if the socket is not connected, it will return an error
                if let Err(err) = check_connection(&stream) {
                    debug!("Failed to connect to NAT(base check): {}", err);
                }
                return Ok((stream, addr));
            }
            Err(err) => {
                debug!("Failed to connect to NAT(timeout): {}", err);
            }
            Ok(Err(err)) => {
                if err.kind() == std::io::ErrorKind::AddrNotAvailable {
                    return Err(err);
                }
                debug!(
                    "Failed to connect to NAT(other error): {}, {}",
                    err.kind(),
                    err
                );
            }
        }
        runtime::delay_for(actual_interval).await;
    }

    debug!("Failed to connect to NAT after {} retries", retry_count);
    Err(std::io::ErrorKind::TimedOut.into())
}
