Audit Report

## Title
Unbounded O(fork_depth) DB Read Loop in `locate_latest_common_block` Triggered by Stored Fork Headers — (`sync/src/types/mod.rs`)

## Summary
The inner loop of `locate_latest_common_block` walks detached fork chains via `parent_hash` links with no depth bound, performing two store lookups per iteration. An attacker who has pre-stored N fork-chain blocks can craft `GetHeaders` messages that each trigger O(N) RocksDB reads, blocking the sync thread pool via `block_in_place` with no rate limiting and no ban on success.

## Finding Description
In `sync/src/types/mod.rs`, `locate_latest_common_block` first scans the locator for the first hash present in the snapshot (canonical chain): [1](#0-0) 

When `index > 0` and `latest_common != Some(0)`, it fetches `locator[index-1]` from the **store** (not the snapshot — fork/detached blocks qualify): [2](#0-1) 

It then enters an unbounded loop walking `parent_hash` links, performing one `store().get_block_header()` and one `snapshot.get_block_number()` per iteration, with no depth cap: [3](#0-2) 

`MAX_LOCATOR_SIZE = 101` bounds only the number of locator entries, not the depth of the inner walk: [4](#0-3) 

`GetHeadersProcess::execute` runs synchronously via `tokio::task::block_in_place`, blocking the async runtime thread for the full duration of the loop: [5](#0-4) 

When `locate_latest_common_block` returns `Some(block_number)` (the attack succeeds), the handler returns `Status::ok()` — the peer is **not banned**: [6](#0-5) 

The peer is only banned (`GetHeadersMissCommonAncestors`) when the function returns `None`: [7](#0-6) 

Rate limiting exists only in the `Relayer` (via `governor::RateLimiter`), not in the `Synchronizer`'s `try_process`: [8](#0-7) 

Blocks received via `SendBlock` P2P messages go through `non_contextual_verify` (which includes PoW verification) before being stored, so storing fork blocks requires valid PoW: [9](#0-8) 

## Impact Explanation
Once an attacker has stored N fork blocks in the target node's persistent store, they can repeatedly send `GetHeaders` messages (each O(1) effort) that each trigger O(N) RocksDB reads while blocking the sync thread pool via `block_in_place`. With N = 10,000 fork blocks, each `GetHeaders` causes ~20,000 DB reads synchronously. This degrades node performance and causes sync stalls for all peers, matching the **High** impact: *Vulnerabilities or bad designs which could cause CKB network congestion with few costs* (ongoing cost per message is O(1) after one-time setup).

## Likelihood Explanation
**Setup cost**: Storing N fork blocks requires mining N blocks with valid PoW — expensive on mainnet, feasible on testnet or low-difficulty devnets. This is a one-time cost. Additionally, natural forks from legitimate network activity provide pre-existing stored fork chains without any mining by the attacker. **Ongoing cost**: Each exploit message is a single `GetHeaders` P2P packet. There is no rate limiting on `GetHeaders` in the `Synchronizer`, no ban triggered by a successful (but expensive) `locate_latest_common_block` call, and no connection teardown. The attacker-to-victim work ratio grows linearly with fork depth.

## Recommendation
Add a depth/iteration cap to the inner loop in `locate_latest_common_block` in `sync/src/types/mod.rs`. Once the loop has walked more than a small constant number of steps (e.g., 32 or 64) without finding a canonical block, break and return `latest_common`. The legitimate use case (refining the common ancestor between two locator entries) does not require walking an arbitrarily deep fork. Additionally, consider adding a per-peer rate limiter for `GetHeaders` messages in `Synchronizer::try_process`, consistent with the rate limiting already present in the `Relayer`.

## Proof of Concept
1. Start a CKB node on testnet or a low-difficulty devnet.
2. Mine N fork blocks branching from genesis (or any canonical block) with valid PoW at the configured difficulty.
3. Submit all N fork blocks to the target node via `SendBlock` P2P messages; confirm they are stored (`BlockStatus::BLOCK_STORED`).
4. Craft a `GetHeaders` locator where:
   - `locator[0]` = fork tip hash (in store, not in snapshot)
   - `locator[1]` = genesis hash (in snapshot, satisfies the genesis check at line 1867)
5. Send the message repeatedly from the same peer. Instrument `sync_shared.store().get_block_header` call count per `execute()` invocation and assert it equals N.
6. Observe that the node's sync thread is blocked for O(N) DB reads per message with no ban or rate limiting applied, and that sync with other peers stalls during the processing window.
