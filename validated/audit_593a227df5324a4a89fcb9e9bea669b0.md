### Title
Unauthenticated `remove_transaction` and `clear_tx_pool` RPC Methods Allow Any Caller to Arbitrarily Manipulate Tx-Pool State — (`rpc/src/module/pool.rs`)

---

### Summary

The CKB JSON-RPC `Pool` module exposes `remove_transaction`, `clear_tx_pool`, and `clear_tx_verify_queue` with **no authentication or caller-identity check of any kind**. Any process that can reach the RPC endpoint — including any local process or any remote client on nodes that expose the RPC publicly — can silently evict arbitrary transactions from the pool or wipe it entirely. The RPC configuration struct contains no authentication fields, and the documentation itself warns that public exposure is dangerous yet provides no built-in mitigation.

---

### Finding Description

**Root cause — no authentication on state-mutating Pool RPC methods**

`remove_transaction` removes a caller-specified transaction and all its descendants from the pool:

```rust
fn remove_transaction(&self, tx_hash: H256) -> Result<bool> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.remove_local_tx(tx_hash.into()).map_err(|e| { ... })
}
```

`clear_tx_pool` wipes every pending and proposed transaction:

```rust
fn clear_tx_pool(&self) -> Result<()> {
    let snapshot = Arc::clone(&self.shared.snapshot());
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_pool(snapshot).map_err(|err| ...)?;
    Ok(())
}
```

`clear_tx_verify_queue` drains the async verification queue:

```rust
fn clear_tx_verify_queue(&self) -> Result<()> {
    let tx_pool = self.shared.tx_pool_controller();
    tx_pool.clear_verify_queue().map_err(|err| ...)?;
    Ok(())
}
```

None of these three functions perform any caller check before executing. The RPC configuration struct has no authentication fields at all:

```rust
pub struct Config {
    pub listen_address: String,
    pub modules: Vec<Module>,
    pub max_request_body_size: usize,
    // ... no token, no password, no IP allowlist
}
```

The `Pool` module is enabled in the default production configuration:

```toml
modules = ["Net", "Pool", "Miner", "Chain", "Stats", "Subscription", "Experiment", "Terminal"]
```

The only protection is the default bind address (`127.0.0.1:8114`), but the documentation explicitly acknowledges that operators routinely change this and warns it is dangerous — without providing any built-in authentication to compensate.

**Analogy to Smilee `trackVaultFee` / `receiveFee`**

In Smilee, `trackVaultFee` and `receiveFee` accepted arbitrary caller-supplied parameters and mutated accounting state with no identity check. Here, `remove_transaction` accepts a caller-supplied `tx_hash` and mutates the tx-pool (the node's primary pending-transaction accounting state) with no identity check. `clear_tx_pool` requires no parameters at all and wipes the entire accounting state in one call.

---

### Impact Explanation

1. **Targeted tx-pool eviction**: An attacker who knows (or can enumerate) a pending transaction hash can call `remove_transaction` to silently drop it and every descendant. This breaks fee accounting for those transactions, disrupts the fee estimator's `tracked_txs` map (the `ConfirmationFraction` algorithm records the tx as dropped/failed, skewing future fee-rate estimates), and prevents the transactions from being mined until resubmitted.

2. **Complete pool wipe (DoS)**: `clear_tx_pool` removes every pending and proposed transaction in a single unauthenticated call. All in-flight user transactions are lost. The fee estimator's internal state is reset. Block assembly is disrupted until the pool is repopulated.

3. **Verification queue drain**: `clear_tx_verify_queue` silently discards all transactions currently awaiting script verification, causing them to be silently dropped without any error returned to the original submitters.

---

### Likelihood Explanation

- The `Pool` module is enabled by default in production.
- The RPC has no built-in authentication mechanism; the only protection is network-level binding.
- Many operators expose the RPC publicly (required for dApps, light clients, wallets, and block explorers). The documentation acknowledges this is common while only offering a warning.
- The attack requires a single HTTP POST with a valid JSON-RPC body — no cryptographic material, no on-chain cost, no prior state.
- `clear_tx_pool` requires zero knowledge of pool contents; `remove_transaction` requires only a tx hash, which is publicly observable from any connected peer or block explorer.

---

### Recommendation

1. Add an optional bearer-token or HMAC authentication field to `rpc::Config` and enforce it on all state-mutating Pool methods (`remove_transaction`, `clear_tx_pool`, `clear_tx_verify_queue`).
2. Alternatively, split the Pool module into a read-only subset (enabled by default) and a write/admin subset (disabled by default, requiring explicit opt-in and authentication).
3. At minimum, add a configurable IP allowlist enforced at the RPC handler level for all state-mutating methods.

---

### Proof of Concept

Any process with HTTP access to the RPC endpoint can execute:

```bash
# Wipe the entire tx-pool — no authentication required
curl -X POST http://<node-rpc>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"clear_tx_pool","params":[]}'

# Remove a specific transaction and all its descendants
curl -X POST http://<node-rpc>:8114 \
  -H 'Content-Type: application/json' \
  -d '{"id":1,"jsonrpc":"2.0","method":"remove_transaction","params":["0xabc..."]}'
```
