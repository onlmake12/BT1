### Title
Stale Pool-Size Counter After Ancestor Eviction Causes Spurious `PoolIsFull` Rejection — (`File: tx-pool/src/component/pool_map.rs`)

### Summary

In `PoolMap::add_entry`, the new total pool size is computed **before** ancestor evictions occur. When `check_and_record_ancestors` evicts cell-ref-parent transactions to satisfy the ancestor-count limit, those evictions correctly decrement `self.total_tx_size` via `update_stat_for_remove_tx`. However, the pre-eviction snapshot stored in the local variable `total_tx_size` is then unconditionally written back to `self.total_tx_size`, silently discarding all eviction subtractions. The field is left inflated by the combined size of every evicted entry. The subsequent `limit_size` call in `submit_entry` then compares this inflated counter against `max_tx_pool_size` and may evict the just-inserted transaction — or other valid transactions — even though the pool has sufficient real capacity.

### Finding Description

`PoolMap::add_entry` in `tx-pool/src/component/pool_map.rs` follows this sequence:

```
1. (total_tx_size, total_tx_cycles) = updated_stat_for_add_tx(entry.size, entry.cycles)
   // local snapshot = self.total_tx_size + entry.size  (BEFORE any evictions)

2. evicts = check_and_record_ancestors(&mut entry)
   // may call remove_entry_and_descendants → remove_entry → update_stat_for_remove_tx
   // which does: self.total_tx_size -= evicted_entry.size  (modifies the FIELD)

3. self.total_tx_size = total_tx_size   // OVERWRITES field with stale local snapshot
   self.total_tx_cycles = total_tx_cycles
``` [1](#0-0) 

`updated_stat_for_add_tx` takes `&self` (immutable borrow) and returns a plain `(usize, Cycle)` pair — it does not update the field, it only computes what the field *would* be if the entry were added right now. [2](#0-1) 

`check_and_record_ancestors` takes `&mut self` and, when the ancestor count exceeds `max_ancestors_count` but the excess is entirely due to cell-ref parents, enters an eviction loop that calls `remove_entry_and_descendants`, which calls `remove_entry`, which calls `update_stat_for_remove_tx` — directly mutating `self.total_tx_size`. [3](#0-2) [4](#0-3) [5](#0-4) 

After `check_and_record_ancestors` returns, the field `self.total_tx_size` correctly reflects `old_total − Σ(evicted_sizes)`. But line 218 then writes `total_tx_size` (the stale local = `old_total + entry.size`) back to the field, producing:

```
self.total_tx_size = old_total + entry.size          (written)
correct value      = old_total − Σ(evicted) + entry.size
inflation          = Σ(evicted_sizes)
```

After `add_entry` returns, `submit_entry` calls `limit_size` with the inflated counter: [6](#0-5) 

`limit_size` loops while `self.pool_map.total_tx_size > self.config.max_tx_pool_size`, evicting the lowest-fee-rate pending transactions. [7](#0-6) 

If the newly submitted transaction has the lowest fee rate, `limit_size` returns `Some(Reject::Full(...))`, which `submit_entry` propagates as an error — rejecting a transaction that the pool had real capacity to hold.

### Impact Explanation

- **Spurious `PoolIsFull` rejection**: A transaction submitted via `send_transaction` RPC (or relayed via P2P) that legitimately fits in the pool is rejected with `Reject::Full` / `PoolIsFull (-1106)` because the pool's size counter is inflated by the sizes of entries that were already evicted to make room.
- **Unnecessary eviction of valid pool entries**: Even if the submitter's transaction survives, `limit_size` may evict other valid, higher-priority transactions that should have remained in the pool, degrading miner revenue and user experience.
- **Persistent counter corruption**: The inflated `total_tx_size` persists until the next recomputation (which only happens on underflow in `update_stat_for_remove_tx`), so subsequent submissions are also affected.

### Likelihood Explanation

The eviction path in `check_and_record_ancestors` is triggered when:
1. A transaction has more than `max_ancestors_count` (default 1,000) ancestors in the pool, **and**
2. The excess ancestors are cell-ref parents (not direct spend-chain parents).

The existing integration test `TxPoolLimitAncestorCount` in `test/src/specs/tx_pool/limit.rs` demonstrates exactly this scenario with 2,000 cell-ref transactions, confirming the path is reachable and exercised in practice. [8](#0-7) 

An unprivileged transaction sender can craft this scenario by first flooding the pool with cell-dep transactions referencing a common output, then submitting a transaction that consumes that output. No special privileges, keys, or majority hash power are required.

### Recommendation

Compute the new totals **after** `check_and_record_ancestors` completes, so that any eviction subtractions are already reflected in `self.total_tx_size` before the new entry's size is added:

```rust
// Remove the pre-eviction snapshot entirely.
// Let check_and_record_ancestors mutate self.total_tx_size via update_stat_for_remove_tx.
evicts = self.check_and_record_ancestors(&mut entry)?;
self.record_entry_edges(&entry)?;
self.insert_entry(&entry, status);
self.record_entry_descendants(&entry);
self.track_entry_statics(None, Some(status));
// Now add the new entry's contribution on top of the already-eviction-adjusted field.
let (total_tx_size, total_tx_cycles) =
    self.updated_stat_for_add_tx(entry.size, entry.cycles)?;
self.total_tx_size = total_tx_size;
self.total_tx_cycles = total_tx_cycles;
```

Alternatively, recompute the totals from scratch after all mutations, similar to the `recompute_total_stat` fallback already present in `update_stat_for_remove_tx`. [9](#0-8) 

### Proof of Concept

**Setup**: configure a node with a small `max_tx_pool_size` (e.g., 2,000 bytes) and `max_ancestors_count = 25`.

1. Submit 26 transactions that all use the same cell as a `cell_dep` (not as an input). Each is accepted; they become cell-ref ancestors of any future transaction that consumes that cell.
2. Submit a transaction `T_new` that **spends** the cell. Its ancestor count = 26 (the 26 cell-dep txs) + 1 = 27 > 25, but `cell_ref_parents.len()` = 26, so `27 − 26 = 1 ≤ 25`. The eviction branch fires: one cell-dep tx (say, size S_evict) is removed via `remove_entry_and_descendants`, correctly decrementing `self.total_tx_size` by S_evict.
3. Back in `add_entry`, line 218 writes `total_tx_size = old_total + T_new.size` (the pre-eviction snapshot), inflating `self.total_tx_size` by S_evict.
4. `submit_entry` calls `limit_size`. If `old_total + T_new.size > max_tx_pool_size` (even though the real size `old_total − S_evict + T_new.size` is within limit), `limit_size` evicts `T_new` and returns `Reject::Full`.
5. The caller receives `PoolIsFull (-1106)` for a transaction that should have been accepted.

        let mut cell_ref_txs = vec![];

            let cur = cur.as_advanced_builder().cell_dep(cell_dep.clone()).build();
