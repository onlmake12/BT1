Audit Report

## Title
Unbounded Response Amplification via Duplicate Indexes in `GetBlockTransactionsProcess::execute` — (`sync/src/relayer/get_block_transactions_process.rs`)

## Summary

`GetBlockTransactionsProcess::execute` accepts up to 32,767 transaction indexes from a remote peer but performs no deduplication and applies no byte-size cap on the resulting `BlockTransactions` response. A single `GetBlockTransactions` message containing 32,767 identical valid indexes causes the victim node to allocate, serialize, and transmit a response containing 32,767 copies of the same transaction, orders of magnitude larger than the request.

## Finding Description

The only guard in `execute` is a count check against `MAX_RELAY_TXS_NUM_PER_BATCH` (32,767): [1](#0-0) 

After passing that check, the code iterates over every index — including duplicates — and clones the corresponding transaction for each occurrence: [2](#0-1) 

The resulting `Vec<Transaction>` is packed directly into a `BlockTransactions` message and sent with no size check: [3](#0-2) 

`MAX_RELAY_TXS_BYTES_PER_BATCH` (1 MiB) is defined in the same module but is never imported or used in `get_block_transactions_process.rs`. The file only imports `MAX_RELAY_TXS_NUM_PER_BATCH`: [4](#0-3) [5](#0-4) 

By contrast, `GetBlockProposalProcess` has both protections: it deduplicates via `HashSet` and rejects if `proposals.len() != message_len`: [6](#0-5) 

And it enforces a byte-size cap during response construction: [7](#0-6) 

Neither protection exists in `GetBlockTransactionsProcess`. The rate limiter in `Relayer::try_process` (30 req/s per peer/message-type) does not prevent amplification within a single message — one message with 32,767 duplicate indexes is processed in full before any rate limit applies: [8](#0-7) 

## Impact Explanation

A single malicious peer can cause the victim node to allocate and serialize a response `Vec` containing 32,767 copies of a large stored transaction entirely in the relay async task before any send occurs. For a transaction of size T bytes, the response is 32,767 × T bytes. Even a modest 10 KB transaction yields a ~320 MB allocation per request. At 30 requests/second (the rate limit), this is ~9.6 GB/s of memory pressure and outbound bandwidth from a single peer sending ~200-byte requests. This maps to **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node** (memory exhaustion) and **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs**.

## Likelihood Explanation

No authentication, proof-of-work, or privileged access is required. The attacker only needs to connect as a normal P2P peer, observe any block hash on-chain (trivially available), and know the index of any large transaction within it. The `GetBlockTransactions` message is a standard relay protocol message accepted from any connected peer. The attack is repeatable up to the rate limit (30/s) and can be launched from multiple peers simultaneously.

## Recommendation

Apply both protections present in `GetBlockProposalProcess` to `GetBlockTransactionsProcess::execute`:

1. **Deduplicate indexes**: Collect indexes into a `HashSet<u32>` before processing and return `StatusCode::RequestDuplicate` if `deduped.len() != original_len`.
2. **Enforce a byte-size cap**: Accumulate the serialized size of transactions to be included in the response and stop (or batch) once `MAX_RELAY_TXS_BYTES_PER_BATCH` is reached, mirroring the logic in `GetBlockProposalProcess::execute` (lines 79–92).

## Proof of Concept

```
1. Attacker connects to victim as a normal P2P peer.
2. Attacker identifies any block hash H on-chain containing a large transaction at index 0
   (e.g., a transaction of 50 KB).
3. Attacker sends:
     GetBlockTransactions {
       block_hash: H,
       indexes: [0, 0, 0, ..., 0],   // 32,767 entries, all identical
       uncle_indexes: []
     }
4. Victim's execute():
   a. Passes count check: 32767 <= 32767.
   b. Loads block from store.
   c. filter_map produces 32,767 clones of the 50 KB transaction.
   d. Allocates ~1.6 GB Vec in the relay thread.
   e. Serializes and transmits a ~1.6 GB BlockTransactions message.
5. Attacker repeats at up to 30 req/s (rate limit), or from multiple peers,
   to sustain memory pressure and bandwidth exhaustion.
```
