### Title
Unvalidated `providerToCredit` Parameter in `executeCallback` Permanently Locks Provider Fees ŌĆö (`target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo::executeCallback` accepts a caller-supplied `address providerToCredit` parameter that is only validated against the stored `req.provider` during the exclusivity window. Once the exclusivity period expires, any caller may pass an arbitrary ŌĆö including unregistered ŌĆö address as `providerToCredit`. The function credits the entire request fee to that address and then permanently deletes the request via `clearRequest`. Because unregistered addresses have no `feeManager` set, the credited funds can never be withdrawn, locking them in the contract forever.

### Finding Description

`Echo::executeCallback` has the following logic:

```solidity
function executeCallback(
    address providerToCredit,   // ŌåÉ fully caller-controlled
    uint64 sequenceNumber,
    bytes[] calldata updateData,
    bytes32[] calldata priceIds
) external payable override {
    Request storage req = findActiveRequest(sequenceNumber);

    if (block.timestamp < req.publishTime + _state.exclusivityPeriodSeconds) {
        require(providerToCredit == req.provider,
            "Only assigned provider during exclusivity period");
    }
    // ŌåÉ NO validation of providerToCredit after exclusivity period

    ...
    _state.providers[providerToCredit].accruedFeesInWei += SafeCast
        .toUint128((req.fee + msg.value) - pythFee);   // fees credited to arbitrary address

    clearRequest(sequenceNumber);   // ŌåÉ request permanently deleted, cannot retry
``` [1](#0-0) 

After the exclusivity period, `providerToCredit` is never checked against `req.provider` or against the registered-provider registry. The fee is credited to `_state.providers[providerToCredit]`, and `clearRequest` permanently removes the in-flight request. [2](#0-1) 

The only withdrawal path for provider fees is `withdrawAsFeeManager`, which requires `msg.sender == _state.providers[provider].feeManager`. For any unregistered address, `feeManager` is `address(0)`, so the condition `msg.sender == address(0)` is never satisfiable. The credited funds are permanently locked. [3](#0-2) 

The developers themselves noted the related risk in a TODO comment:

> "if this effect occurs here, we need to guarantee that executeCallback can never revert. If executeCallback can revert, then funds can be permanently locked in the contract." [4](#0-3) 

### Impact Explanation

An attacker (or even an accidental caller) who invokes `executeCallback` after the exclusivity period with an unregistered `providerToCredit` address causes:

1. The entire `req.fee` (the ETH paid by the requester, minus the Pyth protocol fee) to be credited to an address from which it can never be withdrawn.
2. The request to be permanently cleared ŌĆö no retry is possible.
3. The legitimate provider who was supposed to fulfill the request loses their fee entirely.

This is a direct loss of funds for the provider and a permanent lock of ETH in the contract.

### Likelihood Explanation

- `executeCallback` is `external` with no access control ŌĆö any EOA or contract can call it.
- The exclusivity period (`_state.exclusivityPeriodSeconds`, default 15 seconds per tests) is very short; after it elapses, the vulnerability window is open for the entire remaining lifetime of the request.
- A griefing attacker needs only to monitor the mempool for `PriceUpdateRequested` events, wait 15 seconds, and call `executeCallback` with `providerToCredit = address(1)` (or any unregistered address).
- No special privileges, keys, or capital are required.

### Recommendation

After the exclusivity period, validate that `providerToCredit` is a registered provider:

```solidity
require(
    _state.providers[providerToCredit].isRegistered,
    "providerToCredit must be a registered provider"
);
```

Alternatively, remove the `providerToCredit` parameter entirely and always credit `req.provider` (or `msg.sender` if they are registered), consistent with how the request was originally assigned.

### Proof of Concept

1. Alice (a consumer contract) calls `requestPriceUpdatesWithCallback(defaultProvider, ...)` paying `req.fee` in ETH. The request is stored with `req.provider = defaultProvider`.
2. 16 seconds pass (exclusivity period of 15 s expires).
3. Attacker calls `executeCallback(address(0xdead), sequenceNumber, updateData, priceIds)` where `address(0xdead)` is not a registered provider.
4. The contract executes:
   - `_state.providers[0xdead].accruedFeesInWei += req.fee` ŌĆö fees credited to unregistered address.
   - `clearRequest(sequenceNumber)` ŌĆö request permanently deleted.
5. `defaultProvider` calls `withdrawAsFeeManager(defaultProvider, amount)` ŌĆö succeeds only for their own previously-accrued fees; the fee from this request is gone.
6. Nobody can call `withdrawAsFeeManager(0xdead, ...)` because `_state.providers[0xdead].feeManager == address(0)`.
7. The ETH is permanently locked in the Echo contract.
