Audit Report

## Title
`RecentReject::put` Never Increments `self.total_keys_num`, Disabling the Shrink Guard — (`tx-pool/src/component/recent_reject.rs`)

## Summary
In `RecentReject::put`, the incremented key count is computed into a local variable but never written back to `self.total_keys_num`. Because `self.total_keys_num` is initialized from a RocksDB estimate (0 on a fresh DB) and never updated outside of `shrink()`, the guard condition `total_keys_num > self.count_limit` always evaluates `0 + 1 > count_limit`, which is false for any `count_limit > 1`. `shrink()` is therefore never called, and the underlying RocksDB TTL store grows without bound, enabling disk exhaustion via sustained transaction spam.

## Finding Description
In `RecentReject::put` at lines 62–69 of `tx-pool/src/component/recent_reject.rs`:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
    // self.total_keys_num is never assigned here
} else {
    self.shrink()?;
}
```

The local binding `total_keys_num` holds the incremented value, but `self.total_keys_num` is never updated. On a fresh DB, `self.total_keys_num` is initialized to 0 from the RocksDB estimate (lines 39–52). Every subsequent call to `put()` re-computes `0 + 1 = 1`, which never exceeds `count_limit` (assuming `count_limit > 1`), so `shrink()` is never triggered. The only place `self.total_keys_num` is ever updated is inside `shrink()` itself (lines 110–111), which is unreachable via the normal path. `self.total_keys_num` stays frozen at 0 forever, and the DB grows without bound.

The existing test at line 39 (`assert!(recent_reject.total_keys_num < 100)`) passes trivially because `total_keys_num` is always 0, not because `shrink()` functioned correctly.

`LowFeeRate` satisfies `should_recorded()` (it is not `Duplicated`) and does NOT satisfy `is_malformed_tx()`, so the attacker is never banned and can sustain the spam indefinitely.

## Impact Explanation
The `RecentReject` RocksDB TTL store accumulates entries indefinitely. Even with a finite TTL, an attacker can submit rejected transactions faster than TTL expiry reclaims space, since RocksDB compaction is not guaranteed to run promptly. Sustained spam causes disk exhaustion, crashing the node. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

## Likelihood Explanation
The attack is fully unprivileged. Any peer can connect via P2P and submit transactions with fee rates below the minimum threshold. `LowFeeRate` rejections are recorded in `RecentReject` but do not trigger a ban. The attacker can sustain the spam indefinitely without being disconnected, at a cost of only network bandwidth.

## Recommendation
Add the missing assignment in `RecentReject::put` so that `self.total_keys_num` is incremented on every call:

```rust
if let Some(total_keys_num) = self.total_keys_num.checked_add(1) {
    self.total_keys_num = total_keys_num;  // <-- missing line
    if total_keys_num > self.count_limit {
        self.shrink()?;
    }
} else {
    self.shrink()?;
}
```

## Proof of Concept
Add the following test to `tx-pool/src/component/tests/recent_reject.rs`:

```rust
#[test]
fn test_total_keys_num_never_increments() {
    let tmp_dir = tempfile::Builder::new().tempdir().unwrap();
    let count_limit = 10u64;
    let mut rr = RecentReject::build(tmp_dir.path(), 2, count_limit, -1).unwrap();

    assert_eq!(rr.total_keys_num, 0);

    for i in 0..10000u64 {
        let key = Byte32::new(blake2b_256(i.to_le_bytes()));
        rr.put(&key, Reject::LowFeeRate(0, 0, 0)).unwrap();
    }

    // BUG: total_keys_num is still 0; shrink() was never called
    assert_eq!(rr.total_keys_num, 0);
    // The DB actually contains ~10000 keys, far exceeding count_limit
}
```

This test will pass (demonstrating the bug) against the current code, and fail (demonstrating the fix) after the missing assignment is added.
