### Title
Single-Step Ownership Transfer in `PythLazer` Allows Irrecoverable Loss of Trusted-Signer and Upgrade Control — (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer` inherits from `OwnableUpgradeable` (single-step `transferOwnership`), while every other upgradeable Pyth EVM contract uses `Ownable2StepUpgradeable`. A single erroneous call to `transferOwnership` immediately and irrecoverably hands full control of trusted-signer management and contract upgrades to the wrong address.

---

### Finding Description

`PythLazer.sol` imports and inherits `OwnableUpgradeable`:

```solidity
import "@openzeppelin/contracts-upgradeable/access/OwnableUpgradeable.sol";
contract PythLazer is OwnableUpgradeable, UUPSUpgradeable {
``` [1](#0-0) 

`OwnableUpgradeable.transferOwnership` immediately writes the new owner into storage with no pending-acceptance step. The owner exclusively controls two critical capabilities:

1. **`updateTrustedSigner`** — adds, updates, or removes the ECDSA signers whose signatures are accepted as valid Lazer price-feed proofs. [2](#0-1) 

2. **`_authorizeUpgrade`** — gates all UUPS proxy upgrades. [3](#0-2) 

Every other upgradeable Pyth EVM contract uses `Ownable2StepUpgradeable`, which requires the new owner to call `acceptOwnership()` before the transfer takes effect:

| Contract | Ownership model |
|---|---|
| `EntropyUpgradable` | `Ownable2StepUpgradeable` |
| `ExecutorUpgradable` | `Ownable2StepUpgradeable` |
| `EchoUpgradeable` | `Ownable2StepUpgradeable` |
| `SchedulerUpgradeable` | `Ownable2StepUpgradeable` |
| **`PythLazer`** | **`OwnableUpgradeable` ← single-step** | [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

`EntropyGovernance` and `SchedulerGovernance` additionally implement a `proposeAdmin` / `acceptAdmin` two-step pattern for their secondary admin role, reinforcing that Pyth's design intent is two-step for all critical role transfers. [8](#0-7) 

---

### Impact Explanation

If the current owner calls `transferOwnership(wrongAddress)` — e.g., a mistyped address, a contract address that cannot call back, or an address whose key is lost — the transfer is final and irreversible. The new (wrong) owner immediately gains the ability to:

- Call `updateTrustedSigner` to inject a malicious signer, causing `verifyUpdate` to accept attacker-crafted price payloads as valid Lazer updates.
- Call `upgradeTo` / `upgradeToAndCall` to replace the proxy implementation with an arbitrary contract, draining any accumulated `verification_fee` ETH and corrupting all downstream consumers of Lazer price data.

There is no recovery path: the original owner is permanently locked out.

---

### Likelihood Explanation

Ownership of `PythLazer` is actively managed — the repository contains a dedicated script (`PythLazerChangeOwnership.s.sol`) that transfers ownership to a governance executor contract, and off-chain tooling (`deploy_evm_lazer_contracts.ts`) that calls `transferOwnership` programmatically. [9](#0-8) 

Each governance rotation or key migration is an opportunity for a one-character address typo or a copy-paste error. Because `OwnableUpgradeable` in OZ v5 only guards against `address(0)`, any other wrong-but-non-zero address is accepted silently and immediately.

---

### Recommendation

Replace `OwnableUpgradeable` with `Ownable2StepUpgradeable` in `PythLazer.sol`, matching the pattern used by every other upgradeable Pyth EVM contract:

```solidity
// Before
import "@openzeppelin/contracts-upgradeable/access/OwnableUpgradeable.sol";
contract PythLazer is OwnableUpgradeable, UUPSUpgradeable {

// After
import "@openzeppelin/contracts-upgradeable/access/Ownable2StepUpgradeable.sol";
contract PythLazer is Ownable2StepUpgradeable, UUPSUpgradeable {
```

No storage layout change is required; `Ownable2StepUpgradeable` extends `OwnableUpgradeable` and adds only a `pendingOwner` slot.

---

### Proof of Concept

1. Current owner (governance executor) calls `PythLazer.transferOwnership(typoAddress)` during a routine key rotation.
2. OZ v5 checks `typoAddress != address(0)` — passes — and immediately sets `_owner = typoAddress`.
3. Original owner calls `updateTrustedSigner(maliciousSigner, farFutureExpiry)` → reverts with `OwnableUnauthorizedAccount`.
4. `typoAddress` (attacker-controlled or inaccessible) calls `updateTrustedSigner(attackerSigner, farFutureExpiry)` → succeeds.
5. All subsequent calls to `verifyUpdate` with attacker-signed payloads pass `isValidSigner` and return attacker-controlled price data to every on-chain consumer. [10](#0-9) [11](#0-10)

    function proposeAdmin(address newAdmin) public virtual {
        require(newAdmin != address(0), "newAdmin is zero address");

        _authoriseAdminAction();

        _state.proposedAdmin = newAdmin;
        emit NewAdminProposed(_state.admin, newAdmin);
    }

    /**
     * @dev The proposed admin accepts the admin transfer.
     */
    function acceptAdmin() external {
        if (msg.sender != _state.proposedAdmin)
            revert EntropyErrors.Unauthorized();

        address oldAdmin = _state.admin;
        _state.admin = msg.sender;

        _state.proposedAdmin = address(0);
        emit NewAdminAccepted(oldAdmin, msg.sender);
    }
