Audit Report

## Title
Unbounded heap allocation in `WeightUnitsFlow::do_estimate` via attacker-controlled fee rate — (`util/fee-estimator/src/estimator/weight_units_flow.rs`)

## Summary
`do_estimate` derives `max_bucket_index` directly from the highest fee rate in the live tx pool snapshot with no upper bound, then allocates `vec![0u64; max_bucket_index + 1]` twice. An attacker submitting a transaction with a sufficiently high fee-to-weight ratio causes `max_bucket_index` to reach hundreds of millions, triggering a multi-gigabyte heap allocation that OOM-kills any node with `WeightUnitsFlow` configured that calls `estimate_fee_rate`.

## Finding Description

`FeeRate::calculate` multiplies fee by 1000 before dividing by weight using `saturating_mul`: [1](#0-0) 

For a transaction with `fee = 1.25e12` shannons and `weight = 200` (realistic minimum), this yields `fee_rate = 6.25e12`.

`max_bucket_index_by_fee_rate` has no cap — its final branch is an unbounded linear formula: [2](#0-1) 

For `fee_rate = 6.25e12`, this returns `(6.25e12 + 11_500_000) / 100_000 ≈ 62_500_115`.

`do_estimate` then allocates two vectors of that size with no cap on `max_bucket_index`: [3](#0-2) [4](#0-3) 

`max_fee_rate` is taken directly from the highest-fee-rate entry in the live pool snapshot with no sanitization: [5](#0-4) 

`accept_tx` stores incoming transactions with no fee rate admission check: [6](#0-5) 

The live pool snapshot is taken directly from `get_all_entry_info` with no sanitization before being passed to `estimate_fee_rate`: [7](#0-6) 

Exploit path:
1. Attacker submits a tx with `fee = 1.25e12` shannons and minimal weight (~200)
2. Tx propagates via P2P relay to all peers
3. Any node with `WeightUnitsFlow` configured calls `estimate_fee_rate` RPC
4. `do_estimate` computes `max_bucket_index ≈ 62_500_115`
5. `vec![0u64; 62_500_116]` allocates ~500 MB twice → OOM crash of the entire node process

With `u64::MAX` fee rate, `max_bucket_index ≈ 1.844e12`, yielding ~14.7 TB allocation — guaranteed OOM on any hardware.

## Impact Explanation

**High (10001–15000 points): Vulnerabilities which could easily crash a CKB node.**

The OOM condition kills the entire node process (not merely the RPC handler), constituting a full node crash. The crash is reproducible on every subsequent `estimate_fee_rate` call until the attacker's transaction is confirmed or evicted. The `estimate_fee_rate` RPC is part of the publicly documented `Experiment` module and is accessible to any caller who can reach the RPC port. [8](#0-7) 

## Likelihood Explanation

Any CKB holder can submit a high-fee transaction via P2P relay — no special privilege is required. The transaction propagates to all peers, so a single submission affects all nodes with `WeightUnitsFlow` active simultaneously. The economic cost is proportional to the desired impact (~12,500 CKB for ~1 GB allocation). The attack is repeatable: the attacker can submit new high-fee transactions continuously to maintain the crash condition. The only precondition is that the target node has `fee_estimator.algorithm = WeightUnitsFlow` in its config, which is a supported and documented configuration option. [9](#0-8) 

## Recommendation

Cap `max_bucket_index` to a safe constant before allocating in `do_estimate`:

```rust
const MAX_BUCKET_COUNT: usize = 10_000;

let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate)
    .min(MAX_BUCKET_COUNT - 1);
```

Alternatively, cap the fee rate at ingestion time in `accept_tx` and in `estimate_fee_rate`'s pool snapshot processing to a reasonable maximum (e.g., 10,000,000 shannons/KW), so that `max_bucket_index` is bounded before any allocation occurs.

## Proof of Concept

```rust
#[test]
fn test_oom_via_high_fee_rate() {
    use ckb_types::core::FeeRate;
    // fee_rate = 6.25e12 → max_bucket_index ≈ 62_500_115
    // vec![0u64; 62_500_116] ≈ 500 MB × 2 allocations → OOM
    let bucket_index =
        Algorithm::max_bucket_index_by_fee_rate(FeeRate::from_u64(6_250_000_000_000));
    assert!(bucket_index > 60_000_000); // ~62.5 million
    // Allocating vec![0u64; bucket_index + 1] would consume ~500 MB
    // In do_estimate this happens unconditionally twice before any bounds check
}
```

To trigger the full crash: configure a node with `WeightUnitsFlow`, submit a transaction with `fee = 1_250_000_000_000` shannons and minimal size/cycles, then call the `estimate_fee_rate` RPC. The node process will be killed by the OOM killer or panic on allocation failure.

    fn max_bucket_index_by_fee_rate(fee_rate: FeeRate) -> usize {
        let t = FEE_RATE_UNIT;
        let index = match fee_rate.as_u64() {
            x if x <= 10_000 => x / t,
            x if x <= 50_000 => (x + t * 10) / (2 * t),
            x if x <= 200_000 => (x + t * 100) / (5 * t),
            x if x <= 500_000 => (x + t * 400) / (10 * t),
            x if x <= 1_000_000 => (x + t * 1_300) / (20 * t),
            x if x <= 2_000_000 => (x + t * 4_750) / (50 * t),
            x => (x + t * 11_500) / (100 * t),
        };
        index as usize
    }
