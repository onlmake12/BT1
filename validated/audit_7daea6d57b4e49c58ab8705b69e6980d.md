### Title
`conflicts_cache` and `conflicts_outputs_cache` Not Cleared During Chain Reorganization in `_update_tx_pool_for_reorg` - (File: `tx-pool/src/process.rs`, `tx-pool/src/pool.rs`)

### Summary
The `TxPool` maintains two caches — `conflicts_cache` and `conflicts_outputs_cache` — to track RBF-replaced transactions. The `_update_tx_pool_for_reorg` function, which reconciles the tx-pool after a chain reorganization, updates the snapshot, removes committed transactions, re-adds detached transactions, and evicts expired entries, but **never touches these two caches**. This is the direct analog of the "missing `_update` override" pattern: a state-tracking structure is only updated in some code paths (RBF submission, `clear_pool`) but not in the reorg path.

### Finding Description

`TxPool` holds two RBF-conflict caches:

```rust
// tx-pool/src/pool.rs:48-50
pub(crate) conflicts_cache: lru::LruCache<ProposalShortId, TransactionView>,
pub(crate) conflicts_outputs_cache: lru::LruCache<OutPoint, ProposalShortId>,
``` [1](#0-0) 

When an RBF replacement succeeds, the displaced transaction is moved into `conflicts_cache` via `record_conflict`:

```rust
// tx-pool/src/pool.rs:164-175
pub(crate) fn record_conflict(&mut self, tx: TransactionView) {
    let short_id = tx.proposal_short_id();
    for inputs in tx.input_pts_iter() {
        self.conflicts_outputs_cache.put(inputs, short_id.clone());
    }
    self.conflicts_cache.put(short_id.clone(), tx);
``` [2](#0-1) 

During a reorg, `_update_tx_pool_for_reorg` updates the snapshot, removes committed txs, removes detached proposals, and re-adds detached transactions — but **never clears or reconciles `conflicts_cache` or `conflicts_outputs_cache`**:

```rust
// tx-pool/src/process.rs:1039-1113
fn _update_tx_pool_for_reorg(...) {
    tx_pool.snapshot = Arc::clone(&snapshot);
    tx_pool.remove_committed_txs(attached.iter(), callbacks, detached_headers);
    tx_pool.remove_by_detached_proposal(detached_proposal_id.iter());
    // ... mine mode gap/proposed promotion ...
    tx_pool.remove_expired(callbacks);
    let _ = tx_pool.limit_size(callbacks, None);
    // conflicts_cache and conflicts_outputs_cache: never touched
}
``` [3](#0-2) 

The only place these caches are reset is `TxPool::clear()`, which is only called during `clear_pool()` (hardfork scenario):

```rust
// tx-pool/src/pool.rs:516-522
pub(crate) fn clear(&mut self, snapshot: Arc<Snapshot>) {
    self.pool_map.clear();
    self.snapshot = snapshot;
    self.committed_txs_hash_cache = LruCache::new(COMMITTED_HASH_CACHE_SIZE);
    self.conflicts_cache = LruCache::new(CONFLICTES_CACHE_SIZE);
    self.conflicts_outputs_cache = lru::LruCache::new(CONFLICTES_INPUTS_CACHE_SIZE);
}
``` [4](#0-3) 

### Impact Explanation

**Scenario:**
1. User submits `tx_A` (spends cell X).
2. Attacker submits `tx_B` (RBF replacement for `tx_A`, same input, higher fee). `tx_A` is evicted from the pool and placed into `conflicts_cache`. `tx_B` enters the pool.
3. `tx_B` is committed in block N.
4. A reorg detaches block N. `tx_B` is re-added to the pool via `readd_detached_tx`.
5. `tx_A` remains in `conflicts_cache` — it is **never re-added to the pool**.

After the reorg, `tx_B` (the attacker's replacement) is live in the pool, while `tx_A` (the user's original transaction) is permanently lost from the pool. The user must manually resubmit `tx_A`, but it will be rejected because `tx_B` is still in the pool consuming the same input. The user cannot recover without performing another RBF.

Additionally, `get_tx_from_pool_or_store` consults `conflicts_cache` as a fallback:

```rust
// tx-pool/src/pool.rs:434-446
pub(crate) fn get_tx_from_pool_or_store(...) -> Option<TransactionView> {
    self.get_tx_from_pool(proposal_id)
        .cloned()
        .or_else(|| self.conflicts_cache.peek(proposal_id).cloned())
        ...
}
``` [5](#0-4) 

This means stale conflict entries can be returned to callers (block assembler, relay logic) as if they were valid pool transactions.

### Likelihood Explanation

RBF is an enabled feature (`min_rbf_rate > min_fee_rate`). Reorgs are a normal part of blockchain operation. Any reorg that detaches a block containing an RBF replacement transaction triggers this bug. An attacker can deliberately engineer this: submit an RBF replacement, get it mined, then mine a longer competing chain to trigger a reorg. No privileged access is required — only the ability to submit transactions and mine blocks (or collude with a miner).

### Recommendation

In `_update_tx_pool_for_reorg`, after re-adding detached transactions, iterate over `conflicts_cache` and remove any entry whose inputs are no longer consumed by a live pool transaction (i.e., the replacing transaction was detached). Re-add those transactions to the pool as pending entries. Alternatively, clear both caches entirely during reorg and rely on re-submission, consistent with how `clear()` handles the hardfork case.

### Proof of Concept

1. Enable RBF in node config (`min_rbf_rate > min_fee_rate`).
2. Submit `tx_A` spending `OutPoint(cell_X)`.
3. Submit `tx_B` spending `OutPoint(cell_X)` with higher fee (RBF). Observe `tx_A` moves to `conflicts_cache` via `record_conflict`.
4. Mine a block containing `tx_B`.
5. Trigger a reorg that detaches that block (e.g., via `truncate` RPC or by mining a longer fork).
6. Observe: `tx_B` is re-added to the pool (via `readd_detached_tx`). `tx_A` is **not** re-added — it remains in `conflicts_cache`.
7. Query `get_all_entry_info`: `tx_A` appears in the `conflicted` list even though `tx_B` is no longer committed.
8. Attempt to submit `tx_A` directly: rejected because `tx_B` is in the pool consuming the same input. [6](#0-5) [7](#0-6)

            available_inputs.remove(&input);

            self.callbacks.call_reject(tx_pool, &old, reject);
        }
        assert!(!may_recovered_txs.contains(entry.transaction()));

        &self,

        let mine_mode = self.block_assembler.is_some();
        let mut detached = LinkedHashSet::default();
        let mut attached = LinkedHashSet::default();

        let detached_headers: HashSet<Byte32> = detached_blocks
            .iter()
            .map(|blk| blk.header().hash())
            .collect();

        for blk in detached_blocks {
            detached.extend(blk.transactions().into_iter().skip(1))
        }

        for blk in attached_blocks {
            self.fee_estimator.commit_block(&blk);
            attached.extend(blk.transactions().into_iter().skip(1));
        }
        let retain: Vec<TransactionView> = detached.difference(&attached).cloned().collect();

        let fetched_cache = self.fetch_txs_verify_cache(retain.iter()).await;

        // If there are any transactions requires re-process, return them.
        //
        // At present, there is only one situation:
        // - If the hardfork was happened, then re-process all transactions.
        {
            // This closure is used to limit the lifetime of mutable tx_pool.
            let mut tx_pool = self.tx_pool.write().await;

            _update_tx_pool_for_reorg(
                &mut tx_pool,
                &attached,
                &detached_headers,
                detached_proposal_id,
                snapshot,
                &self.callbacks,
                mine_mode,
            );

            // notice: readd_detached_tx don't update cache
            self.readd_detached_tx(&mut tx_pool, retain, fetched_cache)
                .await;
        }

        self.remove_orphan_txs_by_attach(&attached).await;
        {

            queue.remove_txs(attached.iter().map(|tx| tx.proposal_short_id()));

```rust
fn _update_tx_pool_for_reorg(
    tx_pool: &mut TxPool,
    attached: &LinkedHashSet<TransactionView>,
    detached_headers: &HashSet<Byte32>,
    detached_proposal_id: HashSet<ProposalShortId>,
    snapshot: Arc<Snapshot>,
    callbacks: &Callbacks,
    mine_mode: bool,
) {
    tx_pool.snapshot = Arc::clone(&snapshot);

    // NOTE: `remove_by_detached_proposal` will try to re-put the given expired/detached proposals into
    // pending-pool if they can be found within txpool. As for a transaction
    // which is both expired and committed at the one time(commit at its end of commit-window),
    // we should treat it as a committed and not re-put into pending-pool. So we should ensure
    // that involves `remove_committed_txs` before `remove_expired`.
    tx_pool.remove_committed_txs(attached.iter(), callbacks, detached_headers);
    tx_pool.remove_by_detached_proposal(detached_proposal_id.iter());

    // mine mode:
    // pending ---> gap ----> proposed
    // try move gap to proposed
    if mine_mode {
        let mut proposals = Vec::new();
        let mut gaps = Vec::new();

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Gap) {
            let short_id = entry.inner.proposal_short_id();
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push((short_id, entry.inner.clone()));
            }
        }

        for entry in tx_pool.pool_map.entries.get_by_status(&Status::Pending) {
            let short_id = entry.inner.proposal_short_id();
            let elem = (short_id.clone(), entry.inner.clone());
            if snapshot.proposals().contains_proposed(&short_id) {
                proposals.push(elem);
            } else if snapshot.proposals().contains_gap(&short_id) {
                gaps.push(elem);
            }
        }

        for (id, entry) in proposals {
            debug!("begin to proposed: {:x}", id);
            if let Err(e) = tx_pool.proposed_rtx(&id) {
                debug!(
                    "Failed to add proposed tx {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e);
            } else {
                callbacks.call_proposed(&entry)
            }
        }

        for (id, entry) in gaps {
            debug!("begin to gap: {:x}", id);
            if let Err(e) = tx_pool.gap_rtx(&id) {
                debug!(
                    "Failed to add tx to gap {}, reason: {}",
                    entry.transaction().hash(),
                    e
                );
                callbacks.call_reject(tx_pool, &entry, e.clone());
            }
        }
    }

    // Remove expired transaction from pending
    tx_pool.remove_expired(callbacks);

    // Remove transactions from the pool until its size <= size_limit.
    let _ = tx_pool.limit_size(callbacks, None);
}
