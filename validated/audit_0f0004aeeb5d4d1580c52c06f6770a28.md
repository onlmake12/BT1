### Title
Excess `msg.value` Silently Transferred to Provider Instead of Refunded to Caller - (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.requestPriceUpdatesWithCallback()` accepts `msg.value >= requiredFee` but stores the entire `msg.value - pythFeeInWei` as the provider's fee (`req.fee`). Any amount the caller sends above the required fee is silently credited to the provider rather than refunded to the caller.

### Finding Description

In `Echo.sol`, `requestPriceUpdatesWithCallback` computes a required fee and checks:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
``` [1](#0-0) 

Immediately after, the provider fee stored in the request is set as:

```solidity
req.fee = SafeCast.toUint96(msg.value - _state.pythFeeInWei);
``` [2](#0-1) 

`requiredFee` is `pythFeeInWei + providerBaseFee + providerFeedFee + gasFee`. When `msg.value == requiredFee`, `req.fee` correctly equals the provider's portion. But when `msg.value > requiredFee` (e.g., the caller pads their payment to guard against fee fluctuations), the excess is entirely absorbed into `req.fee` and later paid out to the provider in `executeCallback`:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast
    .toUint128((req.fee + msg.value) - pythFee);
``` [3](#0-2) 

There is no refund path for the excess. The Entropy contract explicitly documents this same behavior ("Note that excess value is *not* refunded to the caller"), but Echo.sol has no such documentation and the fee formula is more complex (per-feed and per-gas components), making overpayment more likely in practice. [4](#0-3) 

### Impact Explanation

Any caller who sends `msg.value` above `requiredFee` permanently loses the excess to the provider. Because `getFee` depends on `callbackGasLimit`, `priceIds.length`, and provider-configured rates (all of which can change between fee estimation and transaction inclusion), callers who defensively overpay to avoid reverts lose funds with no recourse. A provider who raises their fee between a user's `getFee()` call and transaction inclusion causes the user to either revert (underpay) or lose the excess (overpay). There is no safe middle ground.

### Likelihood Explanation

Any unprivileged user calling `requestPriceUpdatesWithCallback` is exposed. The fee has multiple variable components (`feePerFeedInWei`, `feePerGasInWei`, `baseFeeInWei`) that a registered provider can update at any time. Users who query `getFee()` off-chain and then submit a transaction with a small buffer to avoid reverts will silently lose that buffer to the provider. This is a realistic and common pattern for payable protocol interactions. [5](#0-4) 

### Recommendation

After deducting `requiredFee` from `msg.value`, refund the remainder to the caller:

```solidity
uint96 requiredFee = getFee(provider, callbackGasLimit, priceIds);
if (msg.value < requiredFee) revert InsufficientFee();
uint256 excess = msg.value - requiredFee;
req.fee = SafeCast.toUint96(requiredFee - _state.pythFeeInWei);
_state.accruedFeesInWei += _state.pythFeeInWei;
if (excess > 0) {
    (bool ok, ) = payable(msg.sender).call{value: excess}("");
    require(ok, "refund failed");
}
```

This mirrors the pattern already used in `PythLazer.verifyUpdate`:

```solidity
require(msg.value >= verification_fee, "Insufficient fee provided");
if (msg.value > verification_fee) {
    payable(msg.sender).transfer(msg.value - verification_fee);
}
``` [6](#0-5) 

### Proof of Concept

1. Provider registers with `feePerFeedInWei = 1e15`, `baseFeeInWei = 1e15`, `feePerGasInWei = 1`.
2. User calls `getFee(provider, 100000, priceIds)` off-chain and gets `requiredFee = X`.
3. User submits `requestPriceUpdatesWithCallback{value: X + 1e14}(...)` to guard against minor fee changes.
4. Transaction executes. `req.fee = (X + 1e14) - pythFeeInWei`, which is `1e14` more than the provider's legitimate share.
5. Provider calls `executeCallback`; their `accruedFeesInWei` is credited the full `req.fee + msg.value - pythFee`, absorbing the user's `1e14` excess.
6. User has no mechanism to recover the `1e14` overpayment. [7](#0-6)

    function getFee(
        address provider,
        uint32 callbackGasLimit,
        bytes32[] calldata priceIds
    ) public view override returns (uint96 feeAmount) {
        uint96 baseFee = _state.pythFeeInWei; // Fixed fee to Pyth
        // Note: The provider needs to set its fees to include the fee charged by the Pyth contract.
        // Ideally, we would be able to automatically compute the pyth fees from the priceIds, but the
        // fee computation on IPyth assumes it has the full updated data.
        uint96 providerBaseFee = _state.providers[provider].baseFeeInWei;
        uint96 providerFeedFee = SafeCast.toUint96(
            priceIds.length * _state.providers[provider].feePerFeedInWei
        );
        uint96 providerFeeInWei = _state.providers[provider].feePerGasInWei; // Provider's per-gas rate
        uint256 gasFee = callbackGasLimit * providerFeeInWei; // Total provider fee based on gas
        feeAmount =
            baseFee +
            providerBaseFee +
            providerFeedFee +
            SafeCast.toUint96(gasFee); // Total fee user needs to pay
```
