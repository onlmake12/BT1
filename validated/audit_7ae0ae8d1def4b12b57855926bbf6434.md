Audit Report

## Title
`remove_expired` Uses Single-Entry Removal Instead of Descendant-Aware Removal, Leaving Child Transactions Permanently Stuck — (`tx-pool/src/pool.rs`)

## Summary
`TxPool::remove_expired` calls `pool_map.remove_entry` (single-entry removal) for each expired transaction, while every other bulk-eviction path (`limit_size`, `resolve_conflict`, `resolve_conflict_header_dep`, `remove_tx`, `remove_by_detached_proposal`) uses `pool_map.remove_entry_and_descendants`. When a parent transaction expires before its children (because it was submitted slightly earlier), the children remain in the pool with inputs referencing outputs that no longer exist on-chain or in the pool, making them permanently unmineable until their own individual expiry fires or `limit_size` non-deterministically evicts them. This allows an unprivileged attacker to cheaply saturate the pool with unresolvable entries, causing network congestion.

## Finding Description
**Root cause — `remove_expired` (pool.rs L271–287):**
```rust
self.pool_map.remove_entry(&entry.proposal_short_id());
```
`remove_entry` (pool_map.rs L235–250) removes only the targeted entry: it calls `remove_entry_edges` (releasing the parent's own inputs from `edges.inputs`) and `remove_entry_links` (disconnecting the parent from the `links` map). It does **not** touch any child entry.

**Contrast with `limit_size` (pool.rs L307):**
```rust
let removed = self.pool_map.remove_entry_and_descendants(&id);
```
`remove_entry_and_descendants` (pool_map.rs L252–265) first collects the target plus the full transitive descendant set via `calc_descendants`, strips all links for every member, then removes every entry in that set.

**State left behind after `remove_expired` removes parent P:**
1. P's entry is gone; P's own inputs are released from `edges.inputs`.
2. Every child C that spends an output of P still has its input edge recorded in `edges.inputs` as `P_output_outpoint → C_short_id`.
3. C's `links` entry no longer lists P as a parent (because `remove_entry_links` was called for P), but C's input edge is unresolvable: P's output is neither on-chain nor in the pool.
4. C retains its `Pending`/`Gap`/`Proposed` status, is counted in pool statistics, and is considered by the block assembler — which will fail to resolve C's input.

The comment at pool.rs L270 explicitly states the intended behavior: *"Expire all transaction (and their dependencies) in the pool"* — confirming this is an implementation defect, not a design choice.

**Trigger path:** `_update_tx_pool_for_reorg` → `tx_pool.remove_expired(callbacks)` fires on every new block.

**Precondition for the asymmetry:** Parent P submitted at `t0`; child C submitted at `t0 + δ` (δ > 0). Both share the same pool `expiry` duration. P expires at `t0 + expiry`; C expires at `t0 + δ + expiry`. During the window `[t0 + expiry, t0 + δ + expiry]`, P is removed but C is stuck.

## Impact Explanation
**High — bad design causing CKB network congestion with few costs.**

An attacker constructs a fan-out chain: one parent P with N outputs, and N children C1…CN each spending one distinct output of P. After P's expiry fires, all N children are stuck in the pool occupying N pool slots, inflating `pending_count`/`gap_count`/`proposed_count`, and blocking legitimate transactions from being accepted once `max_tx_pool_size` is reached. The attacker can repeat this cycle continuously (each cycle requires only one new UTXO for a fresh P), keeping the pool partially or fully saturated with unresolvable entries. `limit_size` will eventually evict them by fee-rate, but this is non-deterministic and itself rejects legitimate low-fee transactions in the process. This matches the allowed impact: **High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation
Triggerable by any unprivileged user via the `send_transaction` RPC. No special privileges, no victim interaction, and no external dependency required. The only precondition is that P and C have different submission timestamps, which is the normal case whenever a user submits a parent-child chain sequentially (even milliseconds apart suffice). `remove_expired` fires on every block, so the trigger is continuous and the attacker need not time anything precisely beyond waiting for P's expiry window.

## Recommendation
Replace `remove_entry` with `remove_entry_and_descendants` inside `remove_expired`, matching the behavior of every other bulk-removal path and fulfilling the stated comment intent:

```rust
pub(crate) fn remove_expired(&mut self, callbacks: &Callbacks) {
    let now_ms = ckb_systemtime::unix_time_as_millis();

    // Collect only root expired entries (those whose parents are not also expired)
    // to avoid double-removal when iterating descendants.
    let expired_ids: Vec<_> = self
        .pool_map
        .iter()
        .filter(|&entry| self.expiry + entry.inner.timestamp < now_ms)
        .filter(|&entry| {
            // Only process entries whose parents are not themselves expired
            // (i.e., are root entries of expired subgraphs)
            self.pool_map
                .get_parents(&entry.inner.proposal_short_id())
                .all(|parent_id| {
                    self.pool_map
                        .get_by_id(parent_id)
                        .map_or(true, |p| self.expiry + p.inner.timestamp >= now_ms)
                })
        })
        .map(|entry| entry.inner.proposal_short_id())
        .collect();

    for id in expired_ids {
        let removed = self.pool_map.remove_entry_and_descendants(&id);
        for entry in removed {
            let tx_hash = entry.transaction().hash();
            debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
            let reject = Reject::Expiry(entry.timestamp);
            callbacks.call_reject(self, &entry, reject);
        }
    }
}
```

The simplest correct fix is to call `remove_entry_and_descendants` per expired entry and deduplicate (since a child may already have been removed as a descendant of an earlier root). The root-filtering approach above avoids redundant work entirely.

## Proof of Concept
1. Submit parent tx `P` (with ≥2 outputs) via `send_transaction`. Record `t0 = now`.
2. Immediately submit children `C1`, `C2`, …, `CN`, each spending one output of `P`. Their timestamps are `t0 + ε` (ε > 0, even 1 ms suffices).
3. Wait until `t0 + expiry` elapses (P's expiry fires) but before `t0 + ε + expiry` (children not yet expired).
4. Mine a block. `_update_tx_pool_for_reorg` → `remove_expired` fires.
5. Assert: `P` is absent from the pool; `C1…CN` are still present with status `Pending` or `Proposed`.
6. Attempt block assembly: the assembler cannot resolve `C1…CN`'s inputs (P's outputs are gone from both chain and pool).
7. Query pool stats: `pending_count` is inflated by N.
8. Repeat steps 1–4 with fresh UTXOs. After K cycles, the pool holds up to K×N stuck entries, causing legitimate transactions to be rejected by `limit_size`.

A unit test can be written against `TxPool` directly: insert a parent-child pair with timestamps 1 ms apart, advance mock time past the parent's expiry but before the child's, call `remove_expired`, and assert the child is no longer present in the pool.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }
