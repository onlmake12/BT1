### Title
Unconditional `expect` Panic in `block_median_time` When Ancestor Headers Are Unavailable — (File: `traits/src/header_provider.rs`)

### Summary

`block_median_time` in `traits/src/header_provider.rs` unconditionally calls `.expect("parent header exist")` while walking back up to 37 ancestor headers. If any ancestor is absent from the data loader, the node panics. This is called on every incoming block's parent chain during header timestamp verification (`TimestampVerifier`) and during `since`-field transaction verification (`SinceVerifier`). An attacker who can cause the data loader to miss a single ancestor in that 37-block window can crash the node.

### Finding Description

`block_median_time` in `traits/src/header_provider.rs` iterates up to `median_block_count` (37 on mainnet) ancestors:

```rust
fn block_median_time(&self, block_hash: &Byte32, median_block_count: usize) -> u64 {
    let mut timestamps: Vec<u64> = Vec::with_capacity(median_block_count);
    let mut block_hash = block_hash.clone();
    for _ in 0..median_block_count {
        let header_fields = self
            .get_header_fields(&block_hash)
            .expect("parent header exist");   // <-- unconditional panic
        timestamps.push(header_fields.timestamp);
        block_hash = header_fields.parent_hash;
        if header_fields.number == 0 {
            break;
        }
    }
    timestamps.sort_unstable();
    timestamps[timestamps.len() >> 1]
}
``` [1](#0-0) 

The only early-exit guard is `if header_fields.number == 0 { break; }`, which handles the genesis case. There is no guard for the case where `get_header_fields` returns `None` for a non-genesis ancestor. The call becomes `.expect(…)` — a hard panic.

This function is invoked from two consensus-critical paths:

1. **Header timestamp verification** — `TimestampVerifier::verify()` calls `self.data_loader.block_median_time(&self.header.data().raw().parent_hash(), self.median_block_count)`. This runs for every block header processed from a relay or sync peer. [2](#0-1) 

2. **Transaction `since` verification** — `SinceVerifier` (in `verification/src/transaction_verifier.rs`) also calls `block_median_time` when evaluating time-locked inputs. [3](#0-2) 

The `HeaderFieldsProvider` trait is implemented by the sync-layer data loaders used in `sync/src/synchronizer/headers_process.rs` and `sync/src/relayer/compact_block_process.rs`. [4](#0-3) 

### Impact Explanation

If `get_header_fields` returns `None` for any of the up-to-37 ancestors, the node process panics (Rust `expect` unwind/abort). This is a **remote denial-of-service**: the node crashes and must be restarted. Because the panic occurs inside block/header verification, it can be triggered before any PoW or signature check gates the path.

**Impact: High** — node crash / full DoS.

### Likelihood Explanation

The data loader implementations used during sync (`headers_process.rs`) and relay (`compact_block_process.rs`) combine the persistent store with in-flight, not-yet-committed headers. During initial block download or a reorg, a window of 37 ancestors may straddle the boundary between what is committed to the store and what is only held in the in-memory header map. A peer that sends a valid-PoW header whose parent chain has a gap in the combined view (e.g., by exploiting the ordering of batch header delivery) can trigger the panic.

**Likelihood: Low** — requires careful timing or ordering of header delivery, but no privileged access.

### Recommendation

Replace the unconditional `.expect` with a graceful error return:

```rust
let header_fields = match self.get_header_fields(&block_hash) {
    Some(f) => f,
    None => break,   // or return an Err propagated to the caller
};
```

The caller (`TimestampVerifier::verify`) should propagate an `UnknownParentError` rather than panicking, consistent with how the rest of the verifier handles missing data.

### Proof of Concept

1. Attacker operates a sync peer.
2. Attacker sends a `SendHeaders` batch to a victim node that is mid-IBD, where the batch's first header has a parent whose ancestor at depth ≤ 37 is not yet in the victim's store and not included in the batch.
3. The victim's `headers_process` calls `HeaderVerifier::verify` → `TimestampVerifier::verify` → `block_median_time`.
4. On the iteration that reaches the missing ancestor, `get_header_fields` returns `None`.
5. `.expect("parent header exist")` fires → node process panics. [5](#0-4) [6](#0-5)
