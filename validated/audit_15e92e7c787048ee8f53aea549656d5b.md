### Title
Unchecked Arithmetic Subtraction in `ema_price_no_older_than` Causes Panic Revert Instead of Graceful Error - (File: `target_chains/fuel/contracts/pyth-contract/src/main.sw`)

### Summary
The Fuel Pyth contract contains two staleness-check functions: `price_no_older_than` and `ema_price_no_older_than`. The former explicitly guards against timestamp underflow, while the latter performs an unchecked subtraction. When a price feed's EMA publish time exceeds the current block time (e.g., due to clock skew between Pythnet and Fuel), `ema_price_no_older_than` panics with an arithmetic underflow revert instead of returning the expected `PythError::OutdatedPrice` error.

### Finding Description

`price_no_older_than` explicitly guards against underflow with a comment acknowledging the risk:

```sway
// Mimicking saturating subtraction to avoid underflow
let time_difference = if current_time > price.publish_time {
    current_time - price.publish_time
} else {
    0
};
require(time_difference <= time_period, PythError::OutdatedPrice);
``` [1](#0-0) 

In contrast, `ema_price_no_older_than` performs the subtraction directly without any guard:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    require(
        current_time - price.publish_time <= time_period,
        PythError::OutdatedPrice,
    );
    price
}
``` [2](#0-1) 

In FuelVM/Sway, unsigned integer underflow causes a VM panic revert, not a silent wrap. So `current_time - price.publish_time` when `price.publish_time > current_time` produces a hard panic rather than the expected `PythError::OutdatedPrice`.

The `publish_time` stored in the contract is TAI64-converted via:

```sway
publish_time += TAI64_DIFFERENCE;
``` [3](#0-2) 

where `TAI64_DIFFERENCE = 4611686018427387904`. The block timestamp from `std::block::timestamp()` is also TAI64. Any clock skew between Pythnet publishers and the Fuel chain can produce a stored `ema_price.publish_time` slightly ahead of `current_time`.

A second unchecked multiplication exists in `total_fee`:

```sway
pub fn total_fee(
    total_number_of_updates: u64,
    single_update_fee: StorageKey<u64>,
) -> u64 {
    total_number_of_updates * single_update_fee.read()
}
``` [4](#0-3) 

This `u64 * u64` multiplication has no overflow protection. If `single_update_fee` is set to a large value by governance and an attacker submits many update items, the multiplication overflows and `update_price_feeds` panics instead of reverting with `PythError::InsufficientFee`.

### Impact Explanation

For `ema_price_no_older_than`: any on-chain protocol or user calling `ema_price(price_feed_id)` on a feed whose stored EMA timestamp is marginally ahead of the current Fuel block time receives an opaque VM panic revert instead of `PythError::OutdatedPrice`. Callers that catch `OutdatedPrice` to fall back to a secondary price source will not catch the panic, breaking their logic silently. This is a liveness/correctness impact on all consumers of the EMA price API on Fuel.

For `total_fee`: if governance sets `single_update_fee` to a large value, an attacker submitting a crafted batch with enough update items causes `update_price_feeds` to panic, preventing any price feed updates until the fee is lowered.

### Likelihood Explanation

Clock skew between Pythnet and Fuel is a realistic operational condition. The developers already acknowledged this risk for `price_no_older_than` (the comment "Mimicking saturating subtraction to avoid underflow" is explicit) but omitted the same fix from `ema_price_no_older_than`. Any user can call `ema_price` on any price feed ID — no privilege is required. The entry path is fully unprivileged.

### Recommendation

Apply the same saturating-subtraction guard to `ema_price_no_older_than`:

```sway
fn ema_price_no_older_than(time_period: u64, price_feed_id: PriceFeedId) -> Price {
    let price = ema_price_unsafe(price_feed_id);
    let current_time = timestamp();
    // Mimicking saturating subtraction to avoid underflow
    let time_difference = if current_time > price.publish_time {
        current_time - price.publish_time
    } else {
        0
    };
    require(time_difference <= time_period, PythError::OutdatedPrice);
    price
}
```

For `total_fee`, use a checked multiplication and revert with a meaningful error on overflow:

```sway
pub fn total_fee(
    total_number_of_updates: u64,
    single_update_fee: StorageKey<u64>,
) -> u64 {
    total_number_of_updates.checked_mul(single_update_fee.read())
        .unwrap_or_revert_with(PythError::ArithmeticOverflow)
}
```

### Proof of Concept

1. A Pythnet publisher submits a price update with `publish_time = T` (Unix). After TAI64 conversion the stored `ema_price.publish_time = T + TAI64_DIFFERENCE`.
2. Due to minor clock skew, the Fuel block at the time of the query has `timestamp() = T + TAI64_DIFFERENCE - 1` (one second behind).
3. A user calls `ema_price(price_feed_id)` → `ema_price_no_older_than(valid_time_period, price_feed_id)`.
4. Inside the function: `current_time - price.publish_time = (T + TAI64_DIFFERENCE - 1) - (T + TAI64_DIFFERENCE)` → underflow → FuelVM arithmetic panic revert.
5. The caller receives a generic panic revert, not `PythError::OutdatedPrice`, breaking any error-handling logic that expects the typed error. [2](#0-1) [5](#0-4) [4](#0-3)
