Audit Report

## Title
Predictable Time-Based Ping Nonce Allows Connection Slot Exhaustion via Forged Pong Replies - (File: `network/src/protocols/ping.rs`)

## Summary
The CKB ping protocol derives its nonce from elapsed wall-clock seconds since node startup (`t.saturating_duration_since(start_time).as_secs() as u32`), making it fully deterministic and recoverable from the first ping message. The same nonce is broadcast to all connected peers simultaneously. A malicious peer can recover the node's `start_time` from the first ping, pre-compute all future nonces, and send forged pong replies without processing any actual protocol messages. This permanently bypasses the ping-timeout disconnection mechanism and, because forged pongs update `last_ping_protocol_message_received_at`, also protects the attacker from inbound peer eviction — enabling durable connection-slot exhaustion.

## Finding Description

**Root cause — predictable nonce:** [1](#0-0) 

The nonce is the integer count of seconds the node has been running. It carries zero entropy and is fully recoverable from a single observed ping.

**Same nonce assigned to every peer in one round:** [2](#0-1) 

`send_nonce` is computed once and stored in every peer's `PingStatus.nonce`, then broadcast via `filter_broadcast`. Any peer that receives one ping immediately knows the nonce that will be sent to every other peer.

**Pong validation accepts any message matching `(processing==true, nonce==stored_nonce)`:** [3](#0-2) 

There is no per-peer randomness, no replay protection, and no requirement that the pong arrive after the ping was actually received.

**Timeout eviction only fires when `processing==true`:** [4](#0-3) 

A forged pong sets `processing = false` before the timeout window expires, permanently preventing `CHECK_TIMEOUT_TOKEN` from disconnecting the session.

**Forged pong also updates `last_ping_protocol_message_received_at`:** [5](#0-4) 

This is the exact field used by `try_evict_inbound_peer` to protect "most recently active" peers from eviction: [6](#0-5) 

An attacker sending forged pongs every 120 s continuously refreshes this timestamp, making their connections appear recently active and shielded from inbound eviction.

**Sync eviction does not cover inbound peers for chain-sync timeout:** [7](#0-6) 

The chain-sync timeout eviction path is gated on `is_outbound`, so inbound attacker connections are unaffected by that mechanism.

**Exploit path:**
1. Attacker connects as an inbound peer and opens all sub-protocols (to pass `ProtocolTypeCheckerService`).
2. Node sends ping with nonce `N = elapsed_seconds_since_start`. Attacker receives it and recovers `start_time ≈ now − N`.
3. Attacker pre-computes every future nonce: `future_nonce = (future_time − start_time).as_secs() as u32`.
4. Every 120 s, attacker sends a pong with the pre-computed nonce — without reading any other protocol message.
5. Node accepts the pong, clears `processing`, and refreshes `last_ping_protocol_message_received_at`.
6. Attacker repeats indefinitely, holding the slot open while ignoring all sync and relay traffic.
7. With enough attacker-controlled IPs filling all inbound slots, legitimate peers are rejected at `accept_peer` and cannot connect.

## Impact Explanation

This is a **High** severity finding matching: *"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."*

The predictable nonce is a bad design that enables durable inbound connection-slot exhaustion. CKB nodes have a bounded `max_inbound` peer limit. An attacker holding all inbound slots open with forged-liveness connections prevents honest peers from joining the node's peer set. Because the forged pongs also refresh the eviction-protection timestamp, the attacker's connections survive the inbound eviction algorithm. The node's view of the network degrades, reducing its ability to receive blocks and transactions and facilitating eclipse conditions.

## Likelihood Explanation

- **No privilege required**: any TCP-reachable peer can connect and execute this.
- **Trivial to implement**: one ping message is sufficient to recover `start_time`; all future nonces follow from a single integer subtraction.
- **Persistent**: `start_time` is set once at node startup and never changes, so the attack works for the entire process lifetime.
- **Undetectable**: from the node's perspective each attacker session appears perfectly healthy with fresh liveness timestamps.
- **Scalable**: the attacker needs one TCP connection per slot; pong messages are tiny and sent only every 120 s.

## Recommendation

Replace the time-based nonce with a cryptographically random value generated **per-ping, per-peer**:

```rust
use rand::RngCore;

fn nonce() -> u32 {
    rand::thread_rng().next_u32()
}
```

Inside `ping_peers`, call `nonce()` individually for each peer and store the result in that peer's `PingStatus.nonce`. Remove the shared `send_nonce` and the `start_time` field from `PingHandler`. This ensures no peer can predict its own or any other peer's future nonce, and forging a valid pong requires guessing a 32-bit random value (1-in-4-billion per attempt).

## Proof of Concept

```python
import time

# Step 1: Connect to CKB P2P port and complete tentacle handshake
# Step 2: Receive first Ping message, extract nonce N
first_ping_nonce = receive_ping()           # e.g. N = 3600 (node up 1 hour)
node_start_epoch = time.time() - first_ping_nonce  # recover start_time

# Step 3: For every subsequent ping interval (default 120 s):
while True:
    time.sleep(120)
    predicted_nonce = int(time.time() - node_start_epoch)
    send_pong(predicted_nonce)
    # Never read or process any other protocol message.
    # Node accepts pong, clears processing=False, refreshes liveness timestamp.
    # CHECK_TIMEOUT_TOKEN never fires for this session.
```
