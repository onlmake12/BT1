Audit Report

## Title
`remove_expired` Removes Parent Transactions Without Evicting Dependent Descendants, Leaving Zombie Entries in the Pool - (File: tx-pool/src/pool.rs)

## Summary

`TxPool::remove_expired` calls `pool_map.remove_entry(...)` for each expired transaction instead of `remove_entry_and_descendants`. Child transactions that were submitted after the parent remain in the pool in a permanently uncommittable state after the parent expires, consuming pool capacity and holding their input outpoints locked in `edges.inputs`. This contradicts both the function's own comment and the behavior of every other eviction path in the codebase.

## Finding Description

`remove_expired` at `tx-pool/src/pool.rs:270-288` iterates over all entries whose `timestamp + expiry < now_ms` and removes each via `pool_map.remove_entry(&entry.proposal_short_id())` (line 284). `remove_entry` at `pool_map.rs:235-250` calls `remove_entry_links(id)`, which severs the parentâ†”child relationship in `links`, and `remove_entry_edges`, which releases the parent's own inputs from `edges.inputs`. It does **not** enumerate or remove child transactions.

After the parent is evicted:
- Child transactions remain in `entries` with `Status::Pending/Gap/Proposed`.
- Their inputs, which reference the now-gone parent's output outpoints, remain registered in `edges.inputs`.
- Their `links.parents` is now empty (severed by `remove_entry_links`), so `calc_ancestors` returns an empty set â€” the pool treats them as root transactions.
- They can never be committed: the parent's outputs are neither in the pool nor on-chain.
- They will not be re-evicted until their own individual timestamps cross the expiry threshold.

Every other eviction path uses `remove_entry_and_descendants`:
- `limit_size` â†’ `pool_map.remove_entry_and_descendants` (pool.rs:307)
- `remove_tx` â†’ `pool_map.remove_entry_and_descendants` (pool.rs:359)
- `remove_by_detached_proposal` â†’ `pool_map.remove_entry_and_descendants` (pool.rs:343)
- `resolve_conflict` â†’ `pool_map.remove_entry_and_descendants` (pool_map.rs:310, 321)
- `resolve_conflict_header_dep` â†’ `pool_map.remove_entry_and_descendants` (pool_map.rs:285)

`remove_expired` is the sole eviction path that uses the single-entry variant, making it inconsistent with the rest of the codebase and with its own comment ("Expire all transaction **and their dependencies** in the pool").

## Impact Explanation

**High â€” Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker can keep a significant fraction of the pool filled with permanently unspendable zombie transactions. During the zombie window:
- `pool_map.total_tx_size` and `total_tx_cycles` budget are partially consumed by unspendable entries, reducing capacity for legitimate transactions.
- Any new transaction attempting to spend the same outpoints as the zombie children is rejected as a double-spend (because `edges.inputs` still records those outpoints as claimed), even though the claiming transactions can never be mined.
- By repeating the pattern with many parent/child chains, an attacker can sustain pool congestion for up to a full `expiry_hours` period (default: 12 hours) per cycle.

## Likelihood Explanation

The attack requires only the ability to submit transactions to the mempool, which is available to any unprivileged peer or RPC caller. The cost is the transaction fees for the parent and child transactions; since the parent is intentionally low-fee, the cost is minimal. The 12-hour default `expiry_hours` window gives a long period for zombie children to occupy pool space. The attack is repeatable and can be automated.

## Recommendation

Replace `remove_entry` with `remove_entry_and_descendants` inside `remove_expired`, consistent with every other eviction path:

```rust
// tx-pool/src/pool.rs  remove_expired()
for entry in removed {
    let tx_hash = entry.transaction().hash();
    debug!("remove_expired {} timestamp({})", tx_hash, entry.timestamp);
-   self.pool_map.remove_entry(&entry.proposal_short_id());
+   let evicted = self.pool_map.remove_entry_and_descendants(&entry.proposal_short_id());
    let reject = Reject::Expiry(entry.timestamp);
-   callbacks.call_reject(self, &entry, reject);
+   for e in evicted {
+       callbacks.call_reject(self, &e, reject.clone());
+   }
}
```

This aligns the implementation with the existing comment and with the behavior of all other eviction paths.

## Proof of Concept
