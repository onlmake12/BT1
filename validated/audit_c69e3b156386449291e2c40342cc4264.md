### Title
No Rate Limiting on Timestamp-Based REST Endpoints Triggers Unbounded External Benchmarks API Calls - (File: `apps/hermes/server/src/api/rest/v2/timestamp_price_updates.rs`)

---

### Summary

The Hermes REST API exposes several publicly accessible endpoints that accept arbitrary historical timestamps and perform expensive operations — including outbound HTTP calls to the Benchmarks API with a 30-second timeout — without any per-IP or global rate limiting. An unprivileged attacker can flood these endpoints to exhaust connection pools, memory, and async task capacity, degrading or denying service to legitimate Hermes users.

---

### Finding Description

The Hermes Axum router registers all REST endpoints with only a metrics-tracking middleware and no rate-limiting layer: [1](#0-0) 

The timestamp-based endpoints — `/v2/updates/price/:publish_time` (`timestamp_price_updates`), `/api/get_price_feed`, `/api/get_vaa`, and `/api/get_vaa_ccip` — all call `Aggregates::get_price_feeds_with_update_data` with `RequestTime::FirstAfter(publish_time)`: [2](#0-1) 

When the requested timestamp is not present in the local cache, the aggregate implementation falls back to an outbound HTTP call to the Benchmarks API: [3](#0-2) 

That outbound call uses a **30-second timeout** and creates a new `reqwest::Client` per request: [4](#0-3) 

By contrast, the WebSocket endpoint does enforce per-IP byte-rate limiting: [5](#0-4) 

No equivalent protection exists for any REST endpoint.

---

### Impact Explanation

An attacker sending a high volume of requests to `/v2/updates/price/:publish_time` with arbitrary historical timestamps (all cache misses) causes Hermes to spawn a large number of concurrent outbound HTTP connections to the Benchmarks API, each held open for up to 30 seconds. This exhausts:

- The OS-level file descriptor limit (each `reqwest` call opens a TCP socket)
- Tokio async task memory
- The Benchmarks API's own connection capacity

The result is degraded or complete denial of service for legitimate Hermes users, preventing them from obtaining price update proofs needed to submit on-chain price updates to Pyth contracts.

---

### Likelihood Explanation

The endpoints are publicly accessible with no authentication (authentication becomes required only after July 31, 2026 per the docs). Any unprivileged actor can send HTTP GET requests. The attack requires no special knowledge beyond the public API documentation. The 30-second timeout per request means even a modest request rate (e.g., 100 req/s) can accumulate thousands of in-flight connections.

---

### Recommendation

Add a per-IP rate-limiting middleware layer to the Axum router, applied to all REST routes, analogous to the existing WebSocket `DefaultKeyedRateLimiter` in `ws.rs`. For the timestamp endpoints specifically, also consider capping the number of concurrent in-flight Benchmarks API calls (e.g., via a `tokio::sync::Semaphore`) and reusing a single `reqwest::Client` instance rather than constructing one per request.

---

### Proof of Concept

```bash
# Flood /v2/updates/price/:publish_time with cache-miss timestamps (no auth required before July 31 2026)
for i in $(seq 1 500); do
  curl -s "https://hermes.pyth.network/v2/updates/price/1000000000?ids[]=e62df6c8b4a85fe1a67db44dc12de5db330f7ac66b72dc658afedf0f4a415b43" &
done
wait
```

Each request with a timestamp far in the past will miss the local cache and trigger a 30-second outbound call to the Benchmarks API. 500 concurrent requests create 500 open TCP connections held for up to 30 seconds each, exhausting file descriptors and degrading service for all other users.

pub async fn timestamp_price_updates<S>(
    State(state): State<ApiState<S>>,
    Path(path_params): Path<TimestampPriceUpdatesPathParams>,
    QsQuery(query_params): QsQuery<TimestampPriceUpdatesQueryParams>,
) -> Result<Json<PriceUpdate>, RestError>
where
    S: Aggregates,
{
    let price_id_inputs: Vec<PriceIdentifier> =
        query_params.ids.into_iter().map(|id| id.into()).collect();
    let price_ids: Vec<PriceIdentifier> = validate_price_ids(
        &state,
        &price_id_inputs,
        query_params.ignore_invalid_price_ids,
    )
    .await?;

    let state = &*state.state;
    let price_feeds_with_update_data = Aggregates::get_price_feeds_with_update_data(
        state,
        &price_ids,
        RequestTime::FirstAfter(path_params.publish_time),
    )
    .await
    .map_err(|e| {
        tracing::debug!(
            "Error getting price feeds {:?} with update data: {:?}",
            price_ids,
            e
        );
        RestError::UpdateDataNotFound
    })?;

    async fn get_verified_price_feeds(
        &self,
        price_ids: &[PriceIdentifier],
        publish_time: UnixTimestamp,
    ) -> Result<PriceFeedsWithUpdateData> {
        let endpoint = self
            .into()
            .endpoint
            .as_ref()
            .ok_or_else(|| anyhow::anyhow!("Benchmarks endpoint is not set"))?
            .join(&format!("/v1/updates/price/{publish_time}"))
            .context("failed to construct price endpoint")?;

        let mut request = reqwest::Client::new()
            .get(endpoint)
            .timeout(BENCHMARKS_REQUEST_TIMEOUT)
            .query(&[("encoding", "hex")])
            .query(&[("parsed", "true")]);

        for price_id in price_ids {
            request = request.query(&[("ids", price_id)])
        }

        let response = request.send().await?;

        if response.status() != reqwest::StatusCode::OK {
            return Err(anyhow::anyhow!(format!(
                "Price update for price ids {:?} with publish time {} not found in benchmarks. Status code: {}, message: {}",
                price_ids, publish_time, response.status(), response.text().await?
            )));
        }

        let price_update: PriceUpdate = response.json().await?;
        price_update.try_into()
    }
