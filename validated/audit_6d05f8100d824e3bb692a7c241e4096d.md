### Title
`feeManager` Approval Not Cleared on Provider Re-Registration — (`File: target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

### Summary

The Entropy `register()` function rotates a provider's hash-chain commitment but does not reset the `feeManager` field. This is the direct analog of the reported pattern: an "approval" (the fee manager) indexed by a provider address persists across a re-registration event that a provider may use to recover from a security incident, leaving the old fee manager with full authority to drain accrued fees and manipulate the provider's fee rate.

### Finding Description

`Entropy.register()` is the mechanism by which a provider rotates their commitment — the functional equivalent of "transferring" their active identity to a new hash chain. The function explicitly resets commitment fields, fee, URI, and sequence numbers, but leaves `feeManager` untouched:

```solidity
// register() — lines 129–139
provider.feeInWei = feeInWei;
provider.originalCommitment = commitment;
provider.originalCommitmentSequenceNumber = provider.sequenceNumber;
provider.currentCommitment = commitment;
provider.currentCommitmentSequenceNumber = provider.sequenceNumber;
provider.commitmentMetadata = commitmentMetadata;
provider.endSequenceNumber = provider.sequenceNumber + chainLength;
provider.uri = uri;
provider.sequenceNumber += 1;
// provider.feeManager is never touched
``` [1](#0-0) 

The code comment acknowledges that `accruedFeesInWei` is intentionally preserved, but `feeManager` receives no such documentation and is silently carried over. [2](#0-1) 

The `feeManager` role grants two powerful capabilities:

1. `withdrawAsFeeManager(provider, amount)` — drains any amount of the provider's accrued fees directly to `msg.sender`.
2. `setProviderFeeAsFeeManager(provider, newFeeInWei)` — overrides the provider's fee rate. [3](#0-2) [4](#0-3) 

The `setFeeManager` function itself is correctly documented as a single-slot override, but there is no warning that `register()` does not clear it: [5](#0-4) 

### Impact Explanation

A provider who re-registers after a security incident (e.g., a compromised hot wallet that was previously set as fee manager) believes they have rotated their state. In reality, the compromised address retains full `feeManager` authority. The attacker can:

- Call `withdrawAsFeeManager` to drain all accrued fees to themselves.
- Call `setProviderFeeAsFeeManager` to set the provider's fee to 0, destroying the provider's revenue stream and potentially disrupting the service for users who depend on that provider.

This is a direct financial loss to the provider and a functional disruption to the Entropy service.

### Likelihood Explanation

The scenario is realistic: providers are expected to rotate commitments when their hash chain is exhausted or compromised (the contract explicitly supports this). The Fortuna keeper service calls `register` as part of its `setup_provider` flow. [6](#0-5) 

A provider who rotates their commitment after a hot-wallet compromise has no on-chain signal that their fee manager was not cleared. The omission is silent.

### Recommendation

In `register()`, explicitly reset `feeManager` to `address(0)` when the function is called for the first time (i.e., `provider.sequenceNumber == 0`) **or** add a dedicated `feeManager` parameter to `register()` so the caller must consciously set or clear it on every rotation. At minimum, add a NatSpec warning that `feeManager` is preserved across re-registrations so providers are aware they must call `setFeeManager(address(0))` before re-registering if they wish to revoke the old manager.

### Proof of Concept

1. Provider `P` calls `register(...)` and then `setFeeManager(hotWallet)`.
2. `hotWallet` is compromised by attacker `A`.
3. `P` calls `register(...)` again to rotate their hash chain, believing this resets their state.
4. `A` (controlling `hotWallet`) calls `withdrawAsFeeManager(P, providerInfo.accruedFeesInWei)` — succeeds, draining all fees.
5. `A` calls `setProviderFeeAsFeeManager(P, 0)` — succeeds, setting the provider's fee to zero.

The `register()` call at step 3 does not emit any event or revert related to the persisting `feeManager`, giving `P` no indication the old approval survived. [7](#0-6)

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            provider,
            msg.sender,
            amount,
            bytes("")
        );
    }

    function setFeeManager(address manager) external override {
        EntropyStructsV2.ProviderInfo storage provider = _state.providers[
            msg.sender
        ];
        if (provider.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        address oldFeeManager = provider.feeManager;
        provider.feeManager = manager;
        emit ProviderFeeManagerUpdated(msg.sender, oldFeeManager, manager);
        emit EntropyEventsV2.ProviderFeeManagerUpdated(
            msg.sender,
