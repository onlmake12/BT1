### Title
Unbounded Concurrent NAT Traversal Spawns Enable File Descriptor Exhaustion via Hole-Punching Protocol — (`network/src/protocols/hole_punching/component/mod.rs`, `connection_sync.rs`, `connection_request_delivered.rs`)

### Summary
An unprivileged connected peer can exhaust the node's file descriptor table by sending crafted `ConnectionRequest` and `ConnectionSync` messages that trigger an unbounded number of concurrent `runtime::spawn` tasks, each holding up to 24 simultaneous TCP sockets.

### Finding Description
`ConnectionSyncProcess::execute()` calls `runtime::spawn` with a `select_ok` over up to 24 `try_nat_traversal` futures whenever the node is the `to` target and `pending_delivered` contains the `from` peer ID. [1](#0-0) 

`pending_delivered` is populated by `ConnectionRequestProcess::respond_delivered()` with no cap on the total number of entries — only a per-`from`-peer cooldown of 2 minutes. [2](#0-1) 

`pending_delivered` entries are read with `.get()` (not `.remove()`), so each entry can trigger repeated spawns, limited only by `forward_rate_limiter` at 1 spawn/sec per `(from, to)` pair. [3](#0-2) 

`try_nat_traversal` creates one `TcpSocket` per loop iteration via `create_socket()`, consuming it immediately in `socket.connect()`. The socket is held open for the 200ms connect timeout, then dropped. Peak concurrent sockets per task = 1. [4](#0-3) 

The per-session `rate_limiter` (30 msg/sec) caps message ingestion but does not bound the number of live spawned tasks or open sockets. [5](#0-4) 

### Impact Explanation
With 30 `ConnectionSync` messages/sec and 30-second task lifetimes, after 30 seconds there are ~900 concurrent spawns × 24 tasks × ~0.5 connect duty cycle ≈ **10,800 concurrent sockets**, far exceeding a typical fd limit of 1024. `EMFILE` errors propagate silently out of hole-punching tasks (via `?` at line 80), but the exhausted fd table causes all other fd-requiring operations — new P2P connections, database file opens, log writes — to fail, effectively taking the node offline. [6](#0-5) 

### Likelihood Explanation
The attacker needs only a standard P2P connection (no special privileges). The victim's peer ID is publicly advertised. The attacker generates arbitrary `from` peer IDs and valid-looking TCP `listen_addrs`. All message validation checks are satisfiable. The attack completes in ~30 seconds from a single connected peer.

### Recommendation
1. Add a global semaphore or atomic counter capping the total number of concurrent hole-punching spawns (e.g., 10–20).
2. Remove `pending_delivered` entries on first spawn trigger (change `.get()` to `.remove()`) to prevent repeated spawning per `from` peer.
3. Enforce a per-node limit on `pending_delivered` map size.
4. Consider reducing `try_nat_traversal` timeout or adding a per-task socket semaphore.

### Proof of Concept
```
1. Attacker connects to victim as a standard P2P peer.
2. Attacker sends 30 ConnectionRequest/sec for 30s, each with a unique from_peer_id
   and to=victim_peer_id, listen_addrs=[unreachable_tcp_addr].
   → pending_delivered accumulates ~900 entries.
3. Attacker sends 30 ConnectionSync/sec with those from_peer_ids and to=victim_peer_id.
   → Each triggers runtime::spawn(select_ok([24 × try_nat_traversal(unreachable)])).
4. After ~3s: ~90 spawns × 24 tasks × 0.5 duty cycle ≈ 1,080 concurrent sockets → EMFILE.
5. Node fails to accept new P2P connections, open DB files, etc.
```
