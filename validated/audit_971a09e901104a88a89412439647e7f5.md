### Title
Silent Truncating Cast in `calculate_maximum_withdraw` Produces Incorrect DAO Withdrawal Capacity - (File: `util/dao/src/lib.rs`)

### Summary

`DaoCalculator::calculate_maximum_withdraw` computes the maximum withdrawable capacity for a NervosDAO cell using a u128 intermediate value, then casts it to u64 with a silent truncating `as u64` cast. Every other analogous u128→u64 conversion in the same codebase uses `u64::try_from(…).map_err(|_| DaoError::Overflow)?`. The truncating cast silently produces a drastically wrong (much smaller) capacity value when the intermediate result exceeds `u64::MAX`, causing downstream fee calculation and DAO-field accounting to use the wrong amount — an exact structural analog to the reported Zaros bug where the wrong intermediate value was passed to the swap execution path.

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits when `withdraw_counted_capacity > u64::MAX`, producing a value equal to `withdraw_counted_capacity % 2^64` — potentially orders of magnitude smaller than the correct result.

Every other structurally identical u128→u64 narrowing in the same file uses the safe, checked form:

```rust
// dao_field_with_current_epoch, line 244-245
let miner_issuance =
    Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?);

// dao_field_with_current_epoch, line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;

// secondary_block_reward, line 204
let reward = u64::try_from(reward128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) [3](#0-2) [4](#0-3) 

`calculate_maximum_withdraw` is the sole exception. The inconsistency is the root cause.

**Downstream callers that consume the wrong value:**

1. `transaction_maximum_withdraw` → `transaction_fee`: fee = `maximum_withdraw - outputs_capacity`. If `maximum_withdraw` is silently truncated to a tiny value while `outputs_capacity` is the legitimate withdrawal amount, `safe_sub` returns `Overflow`, causing the DAO withdrawal transaction to be rejected. [5](#0-4) 

2. `withdrawed_interests` → `dao_field_with_current_epoch`: the DAO accumulator field `S` written into every block header is computed using `maximum_withdraws - input_capacities`. A truncated `maximum_withdraws` causes `safe_sub` to fail or produces a wrong `S` value, breaking consensus. [6](#0-5) 

3. `DaoHeaderVerifier::verify` rejects any block whose header DAO field does not exactly match the recalculated value, so a wrong `S` causes block rejection. [7](#0-6) 

### Impact Explanation

**Incorrect amount used in a critical operation (exact analog to the reported bug):** just as the Zaros bug passed the wrong intermediate value to the swap executor, here the wrong intermediate value (`withdraw_counted_capacity as u64`) is passed to `Capacity::shannons` and then propagated to fee calculation and DAO-field accounting.

Concrete effects:
- A legitimate NervosDAO withdrawal transaction is rejected with an `Overflow` error from `safe_sub`, constituting a **denial-of-service** for the depositor.
- If the truncated value happens to be non-zero and smaller than `outputs_capacity`, the block containing the withdrawal is rejected by `DaoHeaderVerifier`, causing a **consensus failure** for any miner who includes it.
- The depositor permanently loses access to their accrued interest because the node will never accept the withdrawal.

### Likelihood Explanation

Overflow of `withdraw_counted_capacity` requires:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`counted_capacity` is at most the total CKB supply in shannons (~3.36 × 10¹⁸). For overflow, the accumulation rate ratio `withdrawing_ar / deposit_ar` must exceed ~5.5×. Given the current secondary issuance schedule this takes many decades on mainnet, making the likelihood **low** under normal conditions. However:

- The condition is reachable by any unprivileged transaction sender — no special role is required.
- The bug is a latent correctness defect that will eventually be triggered as the chain ages and `ar` grows.
- The inconsistency with every other u128→u64 conversion in the same file confirms this is an unintentional omission, not a deliberate design choice.

### Recommendation

Replace the truncating cast with the same checked conversion used everywhere else in the file:

```rust
// Before (unsafe):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (safe, consistent with the rest of the file):
let withdraw_counted_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_u64).safe_add(occupied_capacity)?
``` [8](#0-7) 

### Proof of Concept

The existing test `check_withdraw_calculation_overflows` in `util/dao/src/tests.rs` already demonstrates that the overflow path is tested — but it tests the `safe_add` overflow at the end, not the silent truncation in the `as u64` cast. A targeted test exposing the silent truncation:

```rust
#[test]
fn check_withdraw_silent_truncation() {
    // deposit_ar = 10_000_000_000_000_000 (initial)
    // withdrawing_ar = 55_000_000_000_000_000 (5.5× growth — far future)
    // counted_capacity = u64::MAX - occupied_capacity
    // withdraw_counted_capacity (u128) = (u64::MAX - occ) * 55/10 > u64::MAX
    // `as u64` silently wraps; `u64::try_from` would return Err(Overflow)
    let deposit_ar:     u64 = 10_000_000_000_000_000;
    let withdrawing_ar: u64 = 55_000_000_000_000_000;
    let counted_capacity: u64 = u64::MAX / 2;  // large but realistic

    let result_u128: u128 = u128::from(counted_capacity)
        * u128::from(withdrawing_ar)
        / u128::from(deposit_ar);

    // result_u128 = ~1.01e19 > u64::MAX (~1.84e19 for this example)
    // `as u64` truncates silently:
    let truncated = result_u128 as u64;
    let checked   = u64::try_from(result_u128);

    // truncated != correct value; checked is Err
    assert!(checked.is_err(), "should overflow");
    assert_ne!(truncated as u128, result_u128, "silent truncation occurred");
}
``` [9](#0-8)

        )

                "Error generating dao data for block {}: {:?}",

            e
        })?;

        if dao != self.header.dao() {
            return Err((BlockErrorKind::InvalidDAO).into());
        }
        Ok(())

fn check_withdraw_calculation_overflows() {

    let epoch = EpochNumberWithFraction::new(1, 100, 1000);

    let deposit_block = BlockBuilder::default()
        .header(deposit_header)
        .transaction(tx)
        .build();

    let epoch = EpochNumberWithFraction::new(1, 200, 1000);
    let withdrawing_header = HeaderBuilder::default()
        .number(200)
        .epoch(epoch)
        .dao(pack_dao_data(
            10_000_000_001_123_456,
            Default::default(),
            Default::default(),
            Default::default(),
        ))
        .build();
    let withdrawing_block = BlockBuilder::default().header(withdrawing_header).build();

    let tmp_dir = TempDir::new().unwrap();

    let store = ChainDB::new(db, Default::default());
    let txn = store.begin_transaction();
    txn.insert_block(&deposit_block).unwrap();
    txn.attach_block(&deposit_block).unwrap();
    txn.insert_block(&withdrawing_block).unwrap();
    txn.attach_block(&withdrawing_block).unwrap();

        &output,
