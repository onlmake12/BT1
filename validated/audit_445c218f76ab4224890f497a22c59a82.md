### Title
Unguarded `initialize` Allows Any Caller to Seize `governance_authority` — (`target_chains/solana/programs/pyth-solana-receiver/src/lib.rs`)

---

### Summary

The `initialize` instruction has no access-control constraint. Any unprivileged signer can call it first on a fresh deployment and supply an attacker-controlled `Config`, making themselves the sole `governance_authority` with full control over all subsequent governance operations.

---

### Finding Description

The `Initialize` accounts struct requires only that `payer` is a `Signer`: [1](#0-0) 

There is no constraint of the form `payer.key() == <expected_deployer>` or any check against the program's upgrade authority. The instruction body performs only a `minimum_signatures > 0` check before writing the caller-supplied `initial_config` verbatim into the config PDA: [2](#0-1) 

Anchor's `init` constraint prevents double-initialization (the PDA can only be created once), but it does **not** restrict *who* may be the first caller. Whoever calls `initialize` first wins unconditionally.

---

### Impact Explanation

An attacker who calls `initialize` before the legitimate deployer can supply:

- `governance_authority` → attacker pubkey
- `wormhole` → attacker-controlled program
- `valid_data_sources` → attacker-controlled emitter
- `minimum_signatures` → 1

All subsequent governance instructions (`set_fee`, `set_data_sources`, `set_wormhole_address`, `set_minimum_signatures`, `request_governance_authority_transfer`) are gated solely by: [3](#0-2) 

Because the attacker set `governance_authority` to their own key, every one of these checks passes for the attacker and fails for the legitimate operator. The attacker has complete, irrevocable ownership of the receiver program from genesis.

---

### Likelihood Explanation

The precondition — config PDA uninitialized — is satisfied on every new deployment and on any cluster (devnet, testnet, or a new mainnet deployment) where `initialize` has not yet been called. On mainnet the PDA is already initialized, so the existing deployment is safe. However, any future redeployment, migration to a new program ID, or new-cluster deployment is vulnerable unless the deployer atomically initializes in the same transaction as the deployment. The code itself contains no protection.

---

### Recommendation

Add a constraint that restricts `initialize` to the program's upgrade authority (or a hardcoded deployer pubkey):

```rust
#[derive(Accounts)]
#[instruction(initial_config: Config)]
pub struct Initialize<'info> {
    #[account(mut, constraint = payer.key() == upgrade_authority.key()
        @ ReceiverError::NotUpgradeAuthority)]
    pub payer: Signer<'info>,
    #[account(init, space = Config::LEN, payer = payer,
              seeds = [CONFIG_SEED.as_ref()], bump)]
    pub config: Account<'info, Config>,
    // Verify payer is the current upgrade authority of this program
    #[account(constraint = program_data.upgrade_authority_address == Some(payer.key())
        @ ReceiverError::NotUpgradeAuthority)]
    pub program_data: Account<'info, ProgramData>,
    pub system_program: Program<'info, System>,
}
```

Alternatively, include the `initialize` call in the same transaction as the `BpfLoaderUpgradeable::deploy` instruction so no other transaction can be inserted between deployment and initialization.

---

### Proof of Concept

```rust
// Attacker keypair — completely unprivileged
let attacker = Keypair::new();

let attacker_config = Config {
    governance_authority: attacker.pubkey(),
    target_governance_authority: None,
    wormhole: attacker_wormhole_program,
    valid_data_sources: vec![DataSource {
        chain: 1,
        emitter: attacker_emitter,
    }],
    single_update_fee_in_lamports: 0,
    minimum_signatures: 1,
};

// Build and send initialize ix signed only by the attacker
let ix = instruction::Initialize::populate(&attacker.pubkey(), attacker_config);
send_transaction(&[ix], &[&attacker]);

// Attacker now calls set_fee — succeeds because governance_authority == attacker
let set_fee_ix = instruction::SetFee::populate(attacker.pubkey(), 1_000_000);
send_transaction(&[set_fee_ix], &[&attacker]);

// Attacker calls set_data_sources — succeeds
let set_ds_ix = instruction::SetDataSources::populate(attacker.pubkey(), vec![attacker_source]);
send_transaction(&[set_ds_ix], &[&attacker]);
```

The `Initialize::populate` helper in `sdk.rs` accepts any `payer` pubkey with no restriction: [4](#0-3)
