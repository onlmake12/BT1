### Title
Fortuna Keeper's Synchronous Block-Processing Loop Can Be Stalled by Event Flooding, Delaying All Entropy Fulfillments — (`apps/fortuna/src/keeper/block.rs`)

---

### Summary

The `process_new_blocks` function in Fortuna's keeper is the main event-processing loop. It consumes block ranges from a channel and processes each one **synchronously** (`.await`), calling `get_request_with_callback_events` (an RPC log query) for every 100-block batch. An unprivileged attacker can spam `requestWithCallback` to flood a block range with `RequestedWithCallback` events, causing the RPC call to return an oversized response or time out. Because `process_single_block_batch` retries indefinitely on error, the entire `process_new_blocks` loop stalls, preventing the keeper from processing any subsequent blocks. Simultaneously, the `fulfilled_requests_cache` (`HashSet<u64>`) grows without bound as every processed sequence number is inserted and never evicted, creating a memory-exhaustion vector over time.

---

### Finding Description

**Root cause 1 — synchronous, blocking main loop:** [1](#0-0) 

`process_new_blocks` calls `process_block_range(...).await` before it reads the next item from the channel. Inside `process_block_range`, batches of `BLOCK_BATCH_SIZE = 100` blocks are processed one at a time: [2](#0-1) 

Each batch calls `process_single_block_batch`, which issues `get_request_with_callback_events` — a raw Ethereum log query — and **retries indefinitely** on any RPC error: [3](#0-2) 

The RPC call itself queries all `Requested2` logs in the batch window filtered by provider address: [4](#0-3) 

If an attacker floods a 100-block window with thousands of `requestWithCallback` calls, the RPC node must return all matching logs. Many public/private RPC endpoints impose response-size or timeout limits; a sufficiently large log set will cause the call to fail. `process_single_block_batch` then enters its infinite retry loop (5-second sleep between attempts), holding up the entire `process_new_blocks` task.

**Root cause 2 — unbounded `fulfilled_requests_cache`:** [5](#0-4) 

Every sequence number seen is inserted into the shared `HashSet<u64>` and is never removed. An attacker who continuously submits requests causes this set to grow without bound, eventually exhausting keeper memory.

**Attacker entry point:**

`requestWithCallback` (and `requestV2`) are permissionless, payable functions: [6](#0-5) 

Any address that pays the provider fee can create an arbitrary number of `RequestedWithCallback` events targeting any registered provider.

**Configuration context — `backlog_range` default:** [7](#0-6) [8](#0-7) 

On startup, `process_backlog` also scans the last 1 000 blocks synchronously, compounding the exposure window.

---

### Impact Explanation

- The Fortuna keeper stalls: no new `revealWithCallback` transactions are submitted for any pending request while the loop is blocked.
- Users who called `requestWithCallback` do not receive their random-number callbacks until the keeper recovers.
- If the attacker sustains the flood, the keeper never catches up; all Entropy callbacks for the targeted provider are permanently delayed.
- Over a long run, the unbounded `fulfilled_requests_cache` causes the keeper process to OOM-crash, requiring a manual restart.
- Because only the provider's keeper holds the secret needed to generate `providerContribution`, no other party can substitute for the stalled keeper.

---

### Likelihood Explanation

- The attack requires only paying the provider's per-request fee repeatedly. On low-fee chains (Arbitrum, Base, BNB Chain) this cost is negligible.
- No privileged access, leaked key, or governance majority is needed — `requestWithCallback` is fully permissionless.
- The attacker does not need to sustain the flood indefinitely; a single burst that fills one 100-block window with enough events to trigger an RPC timeout is sufficient to stall the loop until the RPC call eventually succeeds with a smaller retry window (which never happens automatically — the retry always re-queries the same range).

---

### Recommendation

1. **Decouple block-range processing from the receive loop.** Spawn each `process_block_range` call as an independent Tokio task instead of awaiting it inline, so the channel consumer is never blocked.
2. **Cap events per batch.** After fetching logs, if the count exceeds a configurable threshold, split the batch into smaller sub-ranges rather than processing an unbounded list.
3. **Bound and periodically evict `fulfilled_requests_cache`.** Replace the unbounded `HashSet` with a bounded LRU cache or a time-windowed set that evicts entries older than the maximum expected request lifetime.
4. **Add RPC response-size / timeout guards.** Configure a hard timeout on `get_request_with_callback_events` and treat a timeout as a signal to halve the block range rather than retry the same range indefinitely.

---

### Proof of Concept

```
1. Attacker identifies the provider address registered with the target Fortuna keeper.
2. Attacker deploys a simple contract that calls requestWithCallback() in a loop,
   paying the provider fee each time.
3. Attacker submits a transaction that creates N (e.g., 5 000) requestWithCallback
   events within a single 100-block window.
4. The keeper's process_single_block_batch calls get_request_with_callback_events
   for that window; the RPC node returns a response exceeding its size/timeout limit
   and returns an error.
5. process_single_block_batch enters its infinite retry loop (RETRY_INTERVAL = 5 s),
   re-querying the same oversized range every 5 seconds.
6. process_new_blocks is blocked; no subsequent block ranges are dequeued from the
   channel.
7. All pending Entropy requests for the provider accumulate unprocessed; users never
   receive their callbacks.
8. If the attacker repeats the flood in every new 100-block window, the keeper never
   recovers without operator intervention.
```

/// Process a range of blocks in batches. It calls the `process_single_block_batch` method for each batch.
#[tracing::instrument(skip_all, fields(
    range_from_block = block_range.from, range_to_block = block_range.to
))]
pub async fn process_block_range(block_range: BlockRange, process_params: ProcessParams) {
    let BlockRange {
        from: first_block,
        to: last_block,
    } = block_range;
    let mut current_block = first_block;
    while current_block <= last_block {
        let mut to_block = current_block + BLOCK_BATCH_SIZE;
        if to_block > last_block {
            to_block = last_block;
        }

        // TODO: this is handling all blocks sequentially we might want to handle them in parallel in future.
        process_single_block_batch(
            BlockRange {
                from: current_block,
                to: to_block,
            },
            process_params.clone(),
        )
        .in_current_span()
        .await;

        current_block = to_block + 1;
    }
}

pub async fn process_single_block_batch(block_range: BlockRange, process_params: ProcessParams) {
    let label = ChainIdLabel {
        chain_id: process_params.chain_state.id.clone(),
    };
    loop {
        let events_res = process_params
            .chain_state
            .contract
            .get_request_with_callback_events(
                block_range.from,
                block_range.to,
                process_params.chain_state.provider_address,
            )
            .await;

        // Only update metrics if we successfully retrieved events.
        if events_res.is_ok() {
            // Track the last time blocks were processed. If anything happens to the processing thread, the
            // timestamp will lag, which will trigger an alert.
            let server_timestamp = SystemTime::now()
                .duration_since(UNIX_EPOCH)
                .map(|duration| duration.as_secs() as i64)
                .unwrap_or(0);
            process_params
                .metrics
                .process_event_timestamp
                .get_or_create(&label)
                .set(server_timestamp);

            let current_block = process_params
                .metrics
                .process_event_block_number
                .get_or_create(&label)
                .get();
            if block_range.to > current_block as u64 {
                process_params
                    .metrics
                    .process_event_block_number
                    .get_or_create(&label)
                    .set(block_range.to as i64);
            }
        }

        match events_res {
            Ok(events) => {
                tracing::info!(num_of_events = &events.len(), "Processing",);
                for event in &events {
                    // the write lock guarantees we spawn only one task per sequence number
                    let newly_inserted = process_params
                        .fulfilled_requests_cache
                        .write()
                        .await
                        .insert(event.sequence_number);
                    if newly_inserted {
                        spawn(
                            process_event_with_backoff(event.clone(), process_params.clone())
                                .in_current_span(),
                        );
                    }
                }
                tracing::info!(num_of_events = &events.len(), "Processed",);
                break;
            }
            Err(e) => {
                tracing::error!(
                    "Error while getting events. Waiting for {} seconds before retry. error: {:?}",
                    RETRY_INTERVAL.as_secs(),
                    e
                );
                time::sleep(RETRY_INTERVAL).await;
            }
        }
    }
}

pub async fn process_new_blocks(
    process_params: ProcessParams,
    mut rx: mpsc::Receiver<BlockRange>,
    block_delays: Vec<u64>,
) {
    tracing::info!("Waiting for new block ranges to process");
    loop {
        if let Some(block_range) = rx.recv().await {
            // Process blocks immediately first
            process_block_range(block_range.clone(), process_params.clone())
                .in_current_span()
                .await;

            // Then process with each configured delay
            for delay in &block_delays {
                let adjusted_range = BlockRange {
                    from: block_range.from.saturating_sub(*delay),
                    to: block_range.to.saturating_sub(*delay),
                };
                process_block_range(adjusted_range, process_params.clone())
                    .in_current_span()
                    .await;
            }
        }
    }
}

    function requestWithCallback(
        address provider,
        bytes32 userContribution
    ) public payable override returns (uint64) {
        return
            requestV2(
                provider,
                userContribution,
                0 // Passing 0 will assign the request the provider's default gas limit
            );
    }

    function requestV2(
        address provider,
        bytes32 userContribution,
        uint32 gasLimit
    ) public payable override returns (uint64) {
        EntropyStructsV2.Request storage req = requestHelper(
            provider,
            constructUserCommitment(userContribution),
            // If useBlockHash is set to true, it allows a scenario in which the provider and miner can collude.
            // If we remove the blockHash from this, the provider would have no choice but to provide its committed
            // random number. Hence, useBlockHash is set to false.
            false,
            true,
            gasLimit
        );

        emit RequestedWithCallback(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            EntropyStructConverter.toV1Request(req)
        );
        emit EntropyEventsV2.Requested(
            provider,
            req.requester,
            req.sequenceNumber,
            userContribution,
            uint32(req.gasLimit10k) * TEN_THOUSAND,
            bytes("")
        );
        return req.sequenceNumber;
    }
