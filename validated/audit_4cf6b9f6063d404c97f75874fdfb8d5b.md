### Title
Excess `msg.value` Permanently Captured by Pyth Treasury in Entropy Fee Accounting — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

In `Entropy.sol`, the `requestHelper()` function permanently credits any `msg.value` above the required fee into `_state.accruedPythFeesInWei` (the Pyth protocol treasury) with no refund path for the caller. Any user who overpays — due to a provider fee decrease between fee estimation and transaction execution, or by hardcoding a fee — permanently loses the excess ETH to the Pyth treasury with no recovery mechanism.

---

### Finding Description

In `requestHelper()`, fee accounting is performed as follows:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

The entire `msg.value - providerFee` is credited to `accruedPythFeesInWei`. If `msg.value > requiredFee`, the excess `msg.value - requiredFee` is silently absorbed into the Pyth treasury rather than refunded to `msg.sender`. This is confirmed by the interface documentation:

> "Note that excess value is *not* refunded to the caller." [2](#0-1) 

The only withdrawal functions available are `withdraw()` (providers only), `withdrawAsFeeManager()` (fee managers only), and `withdrawFee()` in `EntropyGovernance.sol` (admin only). There is no mechanism for a user to recover excess fees they paid. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

Any user who calls `requestV2()`, `requestWithCallback()`, or `request()` with `msg.value` exceeding the current required fee permanently loses the excess ETH. The funds are irrecoverably credited to `accruedPythFeesInWei` and can only be withdrawn by the Pyth admin — not the original payer. This is a direct, permanent loss of user funds with no recovery path.

---

### Likelihood Explanation

This is reachable by any unprivileged user. Concrete scenarios:

1. **Fee decrease race**: A provider calls `setProviderFee()` to lower their fee between the block where a user calls `getFeeV2()` and the block where their `requestV2()` transaction is mined. The user's `msg.value` was computed against the old (higher) fee; the excess is captured.
2. **Hardcoded fee**: Integrators who hardcode a fee value (a common pattern) will overpay whenever the provider lowers their fee.
3. **Intentional overpayment**: Users who send extra ETH to ensure their transaction succeeds lose the excess permanently.

The test suite itself demonstrates overpayment is a normal occurrence: [5](#0-4) 

---

### Recommendation

Refund excess `msg.value` to `msg.sender` in `requestHelper()`:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (requiredFee - providerFee);

// Refund excess to caller
uint128 excess = SafeCast.toUint128(msg.value) - requiredFee;
if (excess > 0) {
    (bool sent, ) = msg.sender.call{value: excess}("");
    require(sent, "Refund failed");
}
```

This mirrors the pattern already used in `PythLazer.sol`: [6](#0-5) 

---

### Proof of Concept

1. Provider registers with `feeInWei = 1000 wei`.
2. User calls `getFeeV2(provider, 0)` off-chain, gets `requiredFee = 1000 + pythFee`.
3. Provider calls `setProviderFee(500)` in a preceding transaction in the same block.
4. User's `requestV2{value: 1000 + pythFee}(provider, 0)` is mined. Now `requiredFee = 500 + pythFee`.
5. `accruedPythFeesInWei` receives `(1000 + pythFee) - 500 = 500 + pythFee` instead of just `pythFee`.
6. The user overpaid by 500 wei with no refund. The 500 wei excess is permanently in the Pyth treasury. [1](#0-0)

    function withdrawFee(address targetAddress, uint128 amount) external {
        require(targetAddress != address(0), "targetAddress is zero address");
        _authoriseAdminAction();

        if (amount > _state.accruedPythFeesInWei)
            revert EntropyErrors.InsufficientFee();

        _state.accruedPythFeesInWei -= amount;

        (bool success, ) = targetAddress.call{value: amount}("");
        require(success, "Failed to withdraw fees");

        emit FeeWithdrawn(targetAddress, amount);
    }
