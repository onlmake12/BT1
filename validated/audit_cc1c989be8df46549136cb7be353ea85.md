### Title
Reward Rate Update Without Prior Epoch Checkpoint Causes Stakers to Lose Current-Epoch Rewards - (File: `governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

---

### Summary

The `update_y` instruction in the Pyth integrity pool program updates the reward rate parameter `y` in `pool_config` without first calling `advance` to checkpoint the current epoch's reward event. If `y` is decreased before `advance` is called for the current epoch, all stakers and delegators lose the rewards they accrued during that epoch at the higher rate.

---

### Finding Description

The integrity pool program uses two key instructions:

**`advance`** — called once per epoch to record the epoch's reward event into `pool_data`. It reads `y` from `pool_config` at call time and writes the epoch event (including `y`) to `pool_data.events`. This is the checkpoint that locks in the reward rate for the epoch.

**`update_y`** — called by `reward_program_authority` to change the reward rate `y` in `pool_config`. Its account list is:
- `reward_program_authority` (signer)
- `pool_config` (writable)
- `system_program`

Critically, `update_y` does **not** include `pool_data` as an account and therefore cannot call `advance` internally. It simply overwrites `y` in `pool_config`. [1](#0-0) 

The `advance` instruction reads `y` from `pool_config` (passed as read-only) and writes the epoch event to `pool_data` (passed as writable): [2](#0-1) 

The `advance_delegation_record` instruction then distributes rewards to individual delegators by replaying historical events stored in `pool_data.events`: [3](#0-2) 

**Vulnerable ordering:**
1. Epoch N begins. `y` is non-zero. Stakers delegate throughout the epoch.
2. `reward_program_authority` calls `update_y(0)` (or any lower value) before `advance` is called for epoch N.
3. `advance` is called for epoch N — it reads `y = 0` from `pool_config` and records `y = 0` in the epoch N event.
4. `advance_delegation_record` is called for stakers — it replays epoch N's event with `y = 0`, distributing zero rewards for the entire epoch.

All rewards stakers would have earned during epoch N at the old non-zero `y` are permanently lost.

The `PoolConfig` struct confirms `y` is a mutable field with no epoch-boundary guard: [4](#0-3) 

The SDK's `yieldRate` is read directly from `poolConfig.y`, confirming `y` is the live rate used at `advance` time: [5](#0-4) 

---

### Impact Explanation

**High.** Any staker or delegator who had active positions during epoch N loses all rewards for that epoch. The epoch event is written once and is immutable after `advance` — there is no mechanism to retroactively correct the recorded `y`. The `advance_delegation_record` function replays events as stored; if the event records `y = 0`, no rewards are ever claimable for that epoch regardless of how long the staker was delegated. [6](#0-5) 

---

### Likelihood Explanation

**Medium.** The `reward_program_authority` must call `update_y` before `advance` is called for the current epoch. This is a realistic operational scenario: the authority may intend to change the reward rate "for the next epoch" without realizing the change takes effect immediately for the current epoch if `advance` has not yet been called. The `advance` instruction is permissionless and can be called by anyone, but there is no guarantee it is called before `update_y` in any given epoch. [1](#0-0) 

---

### Recommendation

The `update_y` instruction should require that `pool_data` is passed as a writable account and should internally invoke the `advance` logic (or assert that `advance` has already been called for the current epoch) before overwriting `y` in `pool_config`. This ensures the current epoch's rewards are checkpointed at the old rate before the new rate takes effect.

---

### Proof of Concept

1. Epoch N starts. `pool_config.y = 1000` (non-zero reward rate). Alice delegates 1,000,000 PYTH tokens to a publisher pool.
2. At the end of epoch N (but before `advance` is called), `reward_program_authority` calls `update_y(0)`.
3. `pool_config.y` is now `0`.
4. Anyone calls `advance` for epoch N. The instruction reads `y = 0` from `pool_config` and writes `Event { epoch: N, y: 0, ... }` to `pool_data.events`.
5. Alice calls `advance_delegation_record`. The instruction replays epoch N's event with `y = 0`. Alice receives `0` rewards for epoch N despite having staked for the entire epoch at `y = 1000`.
6. Alice's rewards for epoch N are permanently unclaimable. [7](#0-6)
