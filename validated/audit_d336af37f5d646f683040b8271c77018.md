Audit Report

## Title
Tx-Pool Admission Fee-Rate Check Uses `tx_size` Instead of Transaction Weight, Allowing Sub-Minimum Effective Fee-Rate Transactions Into the Pool - (File: tx-pool/src/util.rs)

## Summary

`check_tx_fee` in `tx-pool/src/util.rs` enforces `min_fee_rate` using only the serialized byte size of the transaction, while all post-admission operations (mining priority, eviction, fee estimation) use `get_transaction_weight(size, cycles) = max(size, cycles * DEFAULT_BYTES_PER_CYCLES)`, which is always ≥ `tx_size`. A transaction with a small serialized size but very high cycle consumption passes the admission gate while its actual effective fee rate is far below the configured `min_fee_rate`. The code itself acknowledges this with a comment labeling it a "cheap check," but the gap is exploitable to fill the pool at a fraction of the intended cost.

## Finding Description

In `tx-pool/src/util.rs:42-52`, `check_tx_fee` computes:

```rust
// Theoretically we cannot use size as weight directly to calculate fee_rate,
// here min fee rate is used as a cheap check,
// so we will use size to calculate fee_rate directly
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

This runs during `pre_check` in `process.rs:715-717`, before `verify_rtx` is called and before actual cycles are known. After script execution, `TxEntry::new(rtx, verified.cycles, fee, tx_size)` is constructed at `process.rs:751` with the real cycles. All subsequent fee-rate computations use `get_transaction_weight` (`util/types/src/core/tx_pool.rs:298-303`):

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

This function is used by `TxEntry::fee_rate()` (`entry.rs:114-118`), `AncestorsScoreSortKey` (`entry.rs:221-231`), and `EvictKey` (`entry.rs:234-247`). Because `weight ≥ tx_size` always holds, the admission check computes a `min_fee` that is always ≤ the `min_fee` that would be required if the true weight were used. There is no post-verification fee-rate re-check before `submit_entry` is called at `process.rs:753`.

The exploit path is fully reachable via the public `send_transaction` RPC or P2P relay — no privileges required.

## Impact Explanation

An attacker crafting a transaction with a small serialized body but a script consuming near-maximum cycles (e.g., 300 bytes, 70,000,000 cycles) achieves an actual weight of `max(300, 70,000,000 × 0.000170571) ≈ 11,940` weight units, but pays only `min_fee_rate × 300` shannons — roughly a **40× discount** relative to the true weight-based minimum. For transactions approaching the per-transaction cycle cap, the discount grows proportionally.

Because the transactions propagate over P2P to all peers (each of which applies the same size-only admission check), an attacker can flood the mempool of every node in the network with cycle-heavy, effectively-below-minimum-fee-rate transactions at a fraction of the intended cost. This matches the allowed impact: **High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

## Likelihood Explanation

Any unprivileged user can submit transactions via `send_transaction` RPC or P2P relay. Writing a CKB-VM script that performs many computation iterations to inflate cycle consumption while keeping the serialized transaction body small is straightforward for any script author. The attack is repeatable, requires no special access, and the cost reduction scales with the cycle-to-size ratio of the crafted script.

## Recommendation

After `verify_rtx` returns `verified.cycles` in `_process_tx` (`process.rs:734`), perform a second fee-rate check using the actual weight before calling `submit_entry`:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool_config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Some((Err(Reject::LowFeeRate(...)), snapshot));
}
```

This requires reading `min_fee_rate` and `fee` (already available in scope) after the verification step, with no additional script execution cost.

## Proof of Concept

1. Craft a transaction: serialized size = 300 bytes, script consumes 70,000,000 cycles.
2. With `min_fee_rate = 1000 shannons/KW`, `check_tx_fee` computes `min_fee = 1000 × 300 / 1000 = 300 shannons`. Pay 301 shannons. Admission passes.
3. After script execution, `verified.cycles = 70,000,000`. Actual weight = `max(300, 70,000,000 × 0.000170571) ≈ 11,940`.
4. Actual effective fee rate = `301 × 1000 / 11,940 ≈ 25 shannons/KW` — ~40× below `min_fee_rate`.
5. Transaction is admitted, propagated to all peers via P2P, occupies pool space on every node, and is sorted to the bottom of the mining queue. Repeat to fill the pool network-wide at ~1/40th the intended cost.
