### Title
Silent Callback Failure via Dangling Storage Reference After `clearRequest` for Overflow Requests — (File: `target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

In `Echo.sol`'s `executeCallback` function, a `Request storage req` reference is read **after** `clearRequest` has deleted the underlying storage for requests stored in the `requestsOverflow` mapping. This zeroes `req.requester` and `req.callbackGasLimit`, causing the user callback to be silently invoked on `address(0)` with 0 gas. The provider is still credited the full fee, but the user's callback is never executed and their funds are permanently lost.

---

### Finding Description

`executeCallback` obtains a storage reference via `findActiveRequest`:

```solidity
Request storage req = findActiveRequest(sequenceNumber);
```

`findRequest` returns a reference to either `_state.requests[shortKey]` (main array) or `_state.requestsOverflow[key]` (overflow mapping):

```solidity
function findRequest(uint64 sequenceNumber) internal view returns (Request storage req) {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        return req;
    } else {
        req = _state.requestsOverflow[key];   // overflow path
    }
}
```

After crediting the provider, `clearRequest` is called:

```solidity
_state.providers[providerToCredit].accruedFeesInWei += SafeCast.toUint128((req.fee + msg.value) - pythFee);
clearRequest(sequenceNumber);
```

`clearRequest` for overflow requests does a full `delete`:

```solidity
function clearRequest(uint64 sequenceNumber) internal {
    (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);
    Request storage req = _state.requests[shortKey];
    if (req.sequenceNumber == sequenceNumber) {
        req.sequenceNumber = 0;          // main array: only seqnum zeroed
    } else {
        delete _state.requestsOverflow[key];  // overflow: ENTIRE struct zeroed
    }
}
```

`delete` on a mapping entry zeroes every field of the struct in storage. The `req` storage pointer still points to that slot, so immediately after `clearRequest`:

- `req.requester` → `address(0)`
- `req.callbackGasLimit` → `0`

The callback is then executed with these zeroed values:

```solidity
try
    IEchoConsumer(req.requester)._echoCallback{
        gas: req.callbackGasLimit
    }(sequenceNumber, priceFeeds)
```

This becomes a call to `IEchoConsumer(address(0))._echoCallback{gas: 0}(...)`, which fails. The `catch` block silently emits `PriceUpdateCallbackFailed`. The user's callback is never executed.

---

### Impact Explanation

- The provider's `accruedFeesInWei` is incremented correctly (before `clearRequest`), so the provider is fully paid.
- The user's callback is never executed — the service they paid for is not delivered.
- There is no refund mechanism; the request has already been cleared.
- The user permanently loses their fee with no recourse.

This is a direct loss of user funds triggered by normal protocol usage.

---

### Likelihood Explanation

Overflow requests occur whenever two sequence numbers share the same `shortKey` = `uint8(keccak256(seqNum)[0] & 0x1f)` — a 5-bit value with 32 possible slots. By the birthday paradox, a collision is expected after roughly 6–7 requests. In any active deployment with more than ~7 concurrent unfulfilled requests, overflow requests are routine. No special attacker action is required; the provider simply calls `executeCallback` normally.

---

### Recommendation

Cache `req.requester` and `req.callbackGasLimit
