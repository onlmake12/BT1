### Title
Integer Overflow in `GET_LAST_STATE_PROOF_LIMIT` Guard Bypasses DoS Protection — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

The guard that enforces `GET_LAST_STATE_PROOF_LIMIT = 1000` in `GetLastStateProofProcess::execute` contains an integer overflow. An unprivileged remote peer can send a `GetLastStateProof` message with `last_n_blocks` set to any value `> usize::MAX/2` (e.g., `2^63`). In a release build, the expression `(last_n_blocks as usize) * 2` wraps to `0`, making the guard trivially false. The server then collects and processes every block from `start_block_number` to the chain tip, performing O(chain_length) MMR root computations and allocating a `Vec` of up to `chain_length` elements.

---

### Finding Description

**Overflow site** — `get_last_state_proof.rs`, line 201:

```rust
let last_n_blocks: u64 = self.message.last_n_blocks().into();

if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT          // 1000
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
``` [1](#0-0) 

On a 64-bit host `usize` is 64 bits. Casting `2^63_u64` to `usize` is lossless; multiplying by `2` produces `2^64`, which wraps to `0` in Rust release mode (no overflow checks). With an empty `difficulties` list the full expression evaluates to `0 > 1000 = false`, so the guard never fires.

**Unbounded collection** — after the guard, lines 291–296:

```rust
if last_block_number - start_block_number <= last_n_blocks {
    let last_n_numbers = (start_block_number..last_block_number).collect::<Vec<_>>();
``` [2](#0-1) 

With `last_n_blocks = 2^63` the condition is true for any realistic chain (e.g., `10_000_000 ≤ 2^63`), so `last_n_numbers` collects every block number from `start_block_number` to the chain tip — potentially millions of entries.

**Expensive per-block work** — `complete_headers` (lines 124–180) is then called for every collected block number. For each block it calls `snapshot.chain_root_mmr(*number - 1).get_root()`, an O(log N) MMR computation: [3](#0-2) 

Total work is O(chain_length × log N) CPU and O(chain_length) memory — unbounded by any remaining guard.

---

### Impact Explanation

A single malicious light-client peer can cause the CKB node to allocate a `Vec` of millions of `BlockNumber` entries and perform millions of MMR root computations synchronously within a single protocol handler. This exhausts CPU and heap memory, hanging or crashing the node. No PoW, no privileged role, and no prior state is required — only a valid P2P connection and knowledge of the current chain tip hash (publicly observable).

---

### Likelihood Explanation

The exploit requires only:
1. A P2P connection to the light-client protocol port (open by default when the light-client server feature is enabled).
2. A single crafted `GetLastStateProof` message: `last_n_blocks = 2^63`, `difficulties = []`, `start_number = 0`, `last_hash = <current tip hash>`.
3. A release build (the default for production deployments).

The tip hash is trivially obtained from any synced peer or public explorer. No authentication or special capability is needed.

---

### Recommendation

Replace the unchecked multiplication with a saturating or checked variant, or validate `last_n_blocks` against the limit before the combined expression:

```rust
// Option A: reject before the combined check
if last_n_blocks as usize > constant::GET_LAST_STATE_PROOF_LIMIT / 2 {
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}

// Option B: use saturating arithmetic in the guard
if self.message.difficulties().len()
    + (last_n_blocks as usize).saturating_mul(2)
    > constant::GET_LAST_STATE_PROOF_LIMIT
{
    return StatusCode::MalformedProtocolMessage.with_context("too many samples");
}
```

Either change closes the overflow and restores the intended O(GET_LAST_STATE_PROOF_LIMIT × log N) bound.

### Proof of Concept

```
1. Run a CKB node in release mode with the light-client server enabled on a chain
   of ≥ 1 000 000 blocks.
2. Obtain the current tip block hash H (from any peer or RPC).
3. Craft a GetLastStateProof molecule message:
     last_hash        = H
     start_hash       = genesis hash
     start_number     = 0
     difficulty_boundary = 0xffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff
     difficulties     = []          (empty)
     last_n_blocks    = 0x8000000000000000   (2^63)
4. Send the message over the light-client P2P protocol.
5. Observe: the server enters complete_headers and iterates over all ~1 000 000
   block numbers; CPU pegs at 100% and heap grows proportionally until the node
   hangs or OOMs.
6. Assert: with the saturating_mul fix applied, the message is rejected immediately
   with "too many samples" and no block iteration occurs.
```
