### Title
Unchecked Return Value of `suspend_chunk_process` / `continue_chunk_process` Allows Concurrent Tx-Pool Mutation During Block Verification — (File: `chain/src/verify.rs`)

### Summary
In `chain/src/verify.rs`, the results of `tx_pool_controller.suspend_chunk_process()` and `tx_pool_controller.continue_chunk_process()` are silently discarded with `let _ =`. If either call fails (e.g., the internal channel is full or the service is temporarily unavailable), the tx-pool chunk processor continues running concurrently with block verification and chain-state mutation, producing an inconsistent tx-pool state. This is the direct CKB analog of the unchecked `delegatecall` result: a critical coordination call whose failure is never observed, allowing the system to proceed as if the call succeeded.

### Finding Description
In `ConsumeUnverifiedBlocks::start()`, before and after each call to `consume_unverified_blocks`, the code issues:

```rust
// chain/src/verify.rs  line 81
let _ = self.tx_pool_controller.suspend_chunk_process();

// ... full contextual block verification + potential reorg ...

// chain/src/verify.rs  line 101
let _ = self.tx_pool_controller.continue_chunk_process();
```

Both return values are explicitly thrown away. `suspend_chunk_process` sends a message over a bounded channel to the tx-pool service asking it to pause its chunk-verification loop. If that channel is full (e.g., because the tx-pool service is busy processing a large batch), the send returns `Err(Full)`, which is silently ignored. Block verification then proceeds — including `reconcile_main_chain`, `rollback`, `attach_block`, and snapshot replacement — while the tx-pool chunk processor is still running and resolving transactions against the chain state that is simultaneously being mutated.

The same problem applies to `continue_chunk_process`: if it fails, the tx-pool chunk processor may remain suspended indefinitely (or never be resumed), stalling transaction admission.

Contrast this with the guarded calls immediately below in the same function:

```rust
// chain/src/verify.rs  line 386-397
if tx_pool_controller.service_started() {
    if let Err(e) = tx_pool_controller.update_tx_pool_for_reorg(...) {
        error!("[verify block] notify update_tx_pool_for_reorg error {}", e);
    }
```

Those results are checked. The `suspend`/`continue` pair is not, creating an asymmetry where the most time-sensitive coordination calls are the only ones whose failures are invisible.

### Impact Explanation
When `suspend_chunk_process` fails silently and block verification proceeds:

1. The tx-pool chunk processor resolves and validates pending transactions against the live `ChainStore` / snapshot.
2. Simultaneously, `reconcile_main_chain` is writing new block data, rolling back detached blocks, and replacing the shared snapshot.
3. The chunk processor may read a partially-written or already-replaced snapshot, causing it to accept transactions whose inputs were spent by the new block, or reject transactions that are valid under the new tip.
4. After the reorg, `update_tx_pool_for_reorg` is called to reconcile the pool — but it operates on a pool that was already mutated by the concurrent chunk processor against stale state, so the reconciliation may be incomplete or incorrect.

The result is a tx-pool that silently holds invalid (double-spending) transactions or has evicted valid ones, without any error surfaced to the operator or the submitting peer. Miners building block templates from this pool may include invalid transactions, causing their blocks to be rejected by the network.

### Likelihood Explanation
The entry path is fully unprivileged: any peer can relay a valid block, which triggers `ConsumeUnverifiedBlocks::start` → `consume_unverified_blocks`. The channel-full condition for `suspend_chunk_process` is realistic whenever the tx-pool is under load (e.g., during a mempool spike or when a large script-heavy transaction is being chunk-verified). An attacker who can time a block relay to coincide with heavy tx-pool activity can reliably trigger the race.

### Recommendation
Check and handle the return values of both calls, consistent with how `update_tx_pool_for_reorg` is handled:

```rust
if let Err(e) = self.tx_pool_controller.suspend_chunk_process() {
    error!("Failed to suspend tx-pool chunk process before block verification: {}", e);
    // decide: abort verification, or proceed with a warning
}
// ... verification ...
if let Err(e) = self.tx_pool_controller.continue_chunk_process() {
    error!("Failed to resume tx-pool chunk process after block verification: {}", e);
}
```

Additionally, consider using a retry loop or a blocking send for `suspend_chunk_process` to guarantee the tx-pool is actually paused before verification begins.

### Proof of Concept

1. Start a CKB node with a non-empty mempool and an active tx-pool chunk processor verifying a script-heavy transaction.
2. Relay a valid block to the node at the moment the tx-pool's internal command channel is at capacity.
3. `suspend_chunk_process()` returns `Err(Full)`; the `let _ =` discards it.
4. `consume_unverified_blocks` → `verify_block` → `reconcile_main_chain` begins mutating the store and replacing the snapshot.
5. The tx-pool chunk processor concurrently resolves transactions against the partially-replaced snapshot.
6. After verification completes, `update_tx_pool_for_reorg` runs on a pool already mutated by the concurrent chunk processor.
7. Inspect the tx-pool: it contains transactions whose inputs were spent by the newly-attached block (double-spends in the pool), or valid transactions were evicted based on stale state.
