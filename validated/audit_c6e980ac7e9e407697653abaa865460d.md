Audit Report

## Title
Quadratic Complexity in `IndexerSync::append()` Due to Per-Input Linear Transaction Scan — (`File: util/indexer/src/indexer.rs`)

## Summary
`IndexerSync::append()` in `util/indexer/src/indexer.rs` performs an O(T) linear scan over the block's transaction slice for every input whose previous output is not yet in the persistent store. For a block with T chained transactions each having I inputs, this produces O(T² × I) total work. The O(1) `HashMap`-based pattern that eliminates this already exists in the same codebase at `util/types/src/core/cell.rs` but is not applied in the indexer.

## Finding Description
At lines 347–362 of `util/indexer/src/indexer.rs`, the `or_else` fallback inside the double loop over transactions and inputs calls `transactions.iter().enumerate().find(...)`, which is a full O(T) scan:

```rust
if let Some(stored_live_cell) = self.store.get(&key_vec)?.or_else(|| {
    transactions
        .iter()
        .enumerate()
        .find(|(_i, tx)| tx.hash() == out_point.tx_hash())  // O(T) per input
        ...
}) { ... }
```

This fallback is reached whenever `self.store.get(&key_vec)` returns `None`, which is the normal state for any output created by an earlier transaction in the same block (intra-block spending). The outer loop iterates over T transactions, the inner loop over I inputs each, and each inner iteration can trigger the O(T) scan, yielding O(T² × I) total complexity.

The `if tx_index > 0` check at line 339 only skips the coinbase; all other transactions enter the input loop. No guard prevents the linear fallback from being reached for intra-block outputs.

The `HashMap` import is already present at line 11 (`use std::collections::HashMap`), and the identical O(1) pattern is used in `BlockCellProvider::new()` at lines 483–488 of `util/types/src/core/cell.rs`:

```rust
let output_indices: HashMap<Byte32, usize> = block
    .transactions()
    .iter()
    .enumerate()
    .map(|(idx, tx)| (tx.hash(), idx))
    .collect();
```

The indexer's `append()` does not apply this pattern, leaving the linear fallback in place.

## Impact Explanation
A sustained stream of blocks containing long intra-block transaction chains causes the indexer's `append()` to consume quadratically more CPU per block, causing the indexer to fall behind the chain tip. This makes all indexed RPC queries (`get_cells`, `get_transactions`, etc.) return stale or unavailable data. This matches the allowed impact: **Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism**.

## Likelihood Explanation
Intra-block spending is a normal, explicitly supported CKB feature. Any unprivileged user can submit a chain of N transactions (tx₁ → tx₂ → … → txₙ, each spending the previous output) to the mempool at minimum fee rate. A miner naturally includes them in a single block. No privileged access, key material, or majority hashpower is required. The attack is repeatable every block and the cost is proportional to minimum transaction fees.

## Recommendation
Before the outer loop in `append()`, build a `HashMap<Byte32, usize>` mapping each transaction hash to its index (the `HashMap` import already exists at line 11):

```rust
let tx_index_map: HashMap<Byte32, usize> = transactions
    .iter()
    .enumerate()
    .map(|(i, tx)| (tx.hash(), i))
    .collect();
```

Replace the `or_else` linear scan (lines 348–362) with an O(1) map lookup:

```rust
.or_else(|| {
    tx_index_map.get(&out_point.tx_hash()).map(|&i| {
        let tx = &transactions[i];
        let idx: usize = out_point.index().into();
        Value::Cell(
            block_number,
            i as u32,
            &tx.outputs().get(idx).expect("index should match"),
            &tx.outputs_data().get(idx).expect("index should match"),
        )
        .into()
    })
})
```

This reduces overall complexity from O(T² × I) to O(T × I), mirroring the pattern already used in `BlockCellProvider::new()`.

## Proof of Concept
Construct a block with N chained transactions:
- tx₁ spends an external UTXO → output₀
- tx₂ spends tx₁.output₀ → output₁
- …
- txₙ spends txₙ₋₁.outputₙ₋₂ → outputₙ₋₁

Submit all to the mempool; they are committed in one block. When the indexer calls `append()`:
- For tx₂'s input: `store.get()` returns `None` → linear scan of T txs → 1 comparison
- For tx₃'s input: `store.get()` returns `None` → linear scan → 2 comparisons
- …
- For txₙ's input: linear scan → N−1 comparisons

Total: 1 + 2 + … + (N−1) = N(N−1)/2 comparisons. With N = 500, this is ~125,000 hash comparisons per block instead of 500. A unit test can verify this by constructing a `BlockView` with N chained transactions, calling `append()`, and measuring execution time against the O(N) baseline.
