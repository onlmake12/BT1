### Title
Fee Rate Estimate Manipulable via Spot-State Pool Snapshot — (File: `tx-pool/src/component/pool_map.rs`, `util/fee-estimator/src/estimator/weight_units_flow.rs`)

---

### Summary

Both the fallback fee estimator (`pool_map::estimate_fee_rate`) and the primary `WeightUnitsFlow` algorithm derive the recommended fee rate exclusively from the **current, live tx-pool snapshot** — a spot-price read. An unprivileged attacker who can submit transactions to the pool can inflate or deflate the returned estimate, causing honest users to systematically overpay or underpay transaction fees.

---

### Finding Description

**Fallback path — pure spot read**

`pool_map::estimate_fee_rate` iterates the pool's entries sorted by score in reverse order and returns the fee rate of the entry that fills the simulated block:

```rust
let iter = self.entries.iter_by_score().rev();
for entry in iter {
    current_block_bytes += entry.inner.size;
    current_block_cycles += entry.inner.cycles;
    if current_block_bytes >= max_block_bytes || current_block_cycles >= max_block_cycles {
        target_blocks -= 1;
        if target_blocks == 0 {
            return entry.inner.fee_rate();   // ← spot price
        }
        ...
    }
}
```

There is no time-averaging, no smoothing, and no protection against sudden pool composition changes. The entire estimate is a function of the pool's instantaneous state.

**Primary path — `WeightUnitsFlow` still anchored to spot state**

`weight_units_flow::Algorithm::estimate_fee_rate` receives `all_entry_info` (the live pool snapshot) and builds `current_weight_buckets` from it:

```rust
let sorted_current_txs = {
    let mut current_txs: Vec<_> = all_entry_info
        .pending.into_values()
        .chain(all_entry_info.proposed.into_values())
        .map(TxStatus::new_from_entry_info)
        .collect();
    current_txs.sort_unstable_by(|a, b| b.cmp(a));
    current_txs
};
self.do_estimate(target_blocks, &sorted_current_txs)
```

Inside `do_estimate`, the number of fee-rate buckets is determined by the **highest fee rate currently in the pool**:

```rust
let max_fee_rate = sorted_current_txs.first().map(|tx| tx.fee_rate)...;
let max_bucket_index = Self::max_bucket_index_by_fee_rate(max_fee_rate);
let mut buckets = vec![0u64; max_bucket_index + 1];
```

`current_weight_buckets[i]` is then filled from the live pool. The final decision formula is:

```
final_weight = current_weight + added_weight - removed_weight
```

`current_weight` is the spot value; `added_weight` uses historical flow speed. An attacker who controls `current_weight` therefore controls the dominant term of the estimate.

**Call chain to the RPC**

```
estimate_fee_rate RPC
  → TxPoolService::estimate_fee_rate (process.rs:945)
      → tx_pool.get_all_entry_info()          ← live snapshot
      → FeeEstimator::estimate_fee_rate(...)  ← WeightUnitsFlow
      → fallback: tx_pool.estimate_fee_rate() ← pool_map spot read
```

Any RPC caller can trigger this path. The pool snapshot is taken without any time-window guard.

**Attack vectors**

| Goal | Action | Effect |
|---|---|---|
| Inflate estimate | Submit many small txs with very high fee rates | `current_weight_buckets` fills high-fee buckets; estimate rises |
| Deflate estimate | Flood pool with minimum-fee-rate txs (1 000 shannons/KB) | Low-fee buckets dominate; estimate falls to floor |

Because the pool accepts up to 180 MB of transactions, an attacker can cheaply occupy a large fraction of the pool with minimum-fee-rate transactions, pushing the estimate toward the floor. Conversely, a single tiny transaction with an extreme fee rate shifts `max_bucket_index` and the bucket allocation.

---

### Impact Explanation

Users and wallets that call `estimate_fee_rate` to set their transaction fee are exposed to two outcomes:

1. **Overpayment**: Attacker floods pool with high-fee transactions → estimate is inflated → honest users pay more than the market rate, transferring value to miners.
2. **Underpayment / stuck transactions**: Attacker floods pool with minimum-fee-rate transactions → estimate is deflated → honest users submit transactions that are deprioritised or never confirmed within the expected window.

Both outcomes are financially harmful to users who rely on the RPC for fee guidance, which is the documented and intended use of `estimate_fee_rate`.

---

### Likelihood Explanation

- The attack requires no special privilege: any `send_transaction` RPC caller or P2P peer can submit transactions.
- The minimum cost is bounded by `min_fee_rate` (1 000 shannons/KB) × transaction size. With a 180 MB pool, filling it costs on the order of 180 000 shannons ≈ negligible CKB.
- The attacker can reclaim their capacity by spending their own outputs after the manipulation window, making the net cost close to zero.
- The `estimate_fee_rate` RPC is publicly documented and widely used by wallets and tooling, making the attack surface broad.

---

### Recommendation

1. **Time-average the pool snapshot**: Instead of reading the live pool at query time, maintain a rolling TWAP of pool composition (e.g., sampled every block) and use that for `current_weight_buckets`.
2. **Bound the influence of a single block's pool state**: Cap the weight any single snapshot can contribute to the estimate.
3. **Rate-limit or authenticate `estimate_fee_rate`**: Since the RPC is advisory, consider restricting it to authenticated callers or adding a cooldown to reduce the incentive for repeated manipulation.
4. **Decouple `max_bucket_index` from live pool**: Derive the bucket count from a historical maximum rather than the current pool's highest fee rate, preventing bucket-space inflation from a single outlier transaction.

---

### Proof of Concept

```
# Step 1: Attacker floods pool with minimum-fee-rate transactions
for i in range(N):
    send_transaction(rpc, build_tx(fee_rate=1000_shannons_per_KB))

# Step 2: Honest user queries fee estimate
fee = rpc.estimate_fee_rate({"estimate_mode": "no_priority"})
# fee is now artificially low (deflation attack)

# Step 3: Honest user submits transaction at the returned rate
send_transaction(rpc, build_tx(fee_rate=fee))
# Transaction is deprioritised; attacker drains pool, honest tx stuck

# --- OR ---

# Step 1b: Attacker submits a single tiny tx with extreme fee rate
send_transaction(rpc, build_tx(fee=large_amount, size=min_size))
# max_bucket_index in WeightUnitsFlow is now very large

# Step 2b: Honest user queries fee estimate
fee = rpc.estimate_fee_rate({"estimate_mode": "no_priority"})
# fee is now artificially high (inflation attack)

# Step 3b: Honest user overpays
send_transaction(rpc, build_tx(fee_rate=fee))
```

    pub(crate) async fn estimate_fee_rate(
        &self,
        estimate_mode: EstimateMode,
        enable_fallback: bool,
    ) -> Result<FeeRate, AnyError> {
        let all_entry_info = self.tx_pool.read().await.get_all_entry_info();
        match self
            .fee_estimator
            .estimate_fee_rate(estimate_mode, all_entry_info)
        {
            Ok(fee_rate) => Ok(fee_rate),
            Err(err) => {
                if enable_fallback {
                    let target_blocks =
                        FeeEstimator::target_blocks_for_estimate_mode(estimate_mode);
                    self.tx_pool
                        .read()
                        .await
                        .estimate_fee_rate(target_blocks)
                        .map_err(Into::into)
                } else {
                    Err(err.into())
                }
            }
        }
    }
