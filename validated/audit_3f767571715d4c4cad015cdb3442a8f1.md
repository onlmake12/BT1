### Title
Silent u128-to-u64 Truncation in NervosDAO Maximum Withdrawal Capacity Calculation - (File: `util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::calculate_maximum_withdraw` performs a u128 intermediate multiplication to compute the interest-adjusted withdrawal capacity, then casts the result back to u64 using a **silent truncating `as u64` cast** instead of the checked `u64::try_from()` pattern used everywhere else in the same file. If the u128 result exceeds `u64::MAX`, the value is silently wrapped/truncated, producing a drastically underestimated withdrawal capacity and causing permanent loss of deposited CKB interest for the withdrawing user.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast is a **bit-truncating** operation in Rust: if `withdraw_counted_capacity > u64::MAX`, the upper bits are silently discarded and the result wraps to a small value with no error returned.

This is directly inconsistent with every other analogous u128→u64 narrowing in the same file:

- `secondary_block_reward` uses `u64::try_from(reward128).map_err(|_| DaoError::Overflow)?`
- `dao_field_with_current_epoch` uses `u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?` [2](#0-1) [3](#0-2) 

The overflow condition is:

```
counted_capacity × withdrawing_ar > u64::MAX × deposit_ar
```

`ar` starts at `10^16` (as confirmed by `DAOVerifier::ar0`) and grows with every block's secondary issuance. Even a tiny relative increase in `ar` is sufficient to push the product over `u64::MAX` when `counted_capacity` is near the upper range of valid cell capacities. [4](#0-3) 

The existing overflow test `check_withdraw_calculation_overflows` does **not** cover this path — it only exercises the `safe_add` overflow that occurs after the cast, not the truncation itself: [5](#0-4) 

---

### Impact Explanation

`calculate_maximum_withdraw` is the authoritative source for two critical values:

1. **`calculate_dao_maximum_withdraw` RPC** — the value returned to users and wallets to determine how much CKB to claim in a withdrawal transaction.
2. **`transaction_fee` / `transaction_maximum_withdraw`** — used by the block verifier to validate that a DAO withdrawal transaction does not claim more than it is entitled to. [6](#0-5) [7](#0-6) 

If truncation occurs:
- The RPC returns a silently wrong (far smaller) maximum withdrawal amount.
- A user who constructs a withdrawal transaction based on this value will set their output capacity to the truncated amount.
- The verifier will accept the transaction (the output is ≤ the (also truncated) computed maximum), and the user permanently loses the difference between the true entitlement and the truncated value — **loss of deposited CKB interest**.

---

### Likelihood Explanation

The overflow requires `counted_capacity × withdrawing_ar > u64::MAX × deposit_ar`. Since `ar` grows monotonically with secondary issuance (~4% annually), and `counted_capacity` can be up to the full cell capacity minus occupied bytes, a cell holding a large fraction of the circulating supply deposited early in the chain's life and withdrawn after many years of `ar` accumulation can reach this threshold. The condition is not reachable today with realistic supply figures, but it is a latent, time-dependent vulnerability with no on-chain guard — the code path is reachable by any unprivileged user calling the `calculate_dao_maximum_withdraw` RPC or submitting a DAO withdrawal transaction.

---

### Recommendation

Replace the silent `as u64` cast with the same checked conversion pattern used elsewhere in the file:

```rust
// Before (silent truncation):
Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?

// After (consistent with the rest of the file):
let withdraw_counted_capacity_u64 =
    u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?;
Capacity::shannons(withdraw_counted_capacity_u64).safe_add(occupied_capacity)?
``` [1](#0-0) 

---

### Proof of Concept

The inconsistency is directly visible by comparing the three u128→u64 narrowing sites in the same file:

| Location | Method | Overflow handling |
|---|---|---|
| `calculate_maximum_withdraw` L156 | `as u64` | **Silent truncation** |
| `secondary_block_reward` L204 | `u64::try_from(…)?` | Returns `DaoError::Overflow` |
| `dao_field_with_current_epoch` L258 | `u64::try_from(…)?` | Returns `DaoError::Overflow` | [1](#0-0) [2](#0-1) [3](#0-2) 

A concrete numeric scenario demonstrating the truncation path:

- `deposit_ar = 10_000_000_000_000_000` (initial `ar0 = 10^16`)
- `withdrawing_ar = 10_000_000_001_000_000` (tiny growth)
- `counted_capacity = 18_446_744_073_000_000_000` (≈ `u64::MAX`, a single large cell)

```
withdraw_counted_capacity (u128)
  = 18_446_744_073_000_000_000 × 10_000_000_001_000_000
    / 10_000_000_000_000_000
  = 18_446_744_073_000_000_000 + 18_446_744_073 × 1_000_000 / 10^16 × 10^16
  ≈ 18_446_744_091_446_744_073   ← exceeds u64::MAX (18_446_744_073_709_551_615)

withdraw_counted_capacity as u64
  = 18_446_744_091_446_744_073 % 2^64
  = 17_737_192_458              ← ~17 shannons, not ~18.4 × 10^18
```

The user's withdrawal is computed as ~17 shannons instead of ~18.4 billion CKB — a near-total loss of funds.
