### Title
Off-by-one in `Scheduler::process_message_box` Spawn guard allows 17 concurrent VMs instead of 16 — (`script/src/scheduler.rs`)

### Summary

`MAX_VMS_COUNT` is defined as `16`, but the spawn guard uses strict `>` instead of `>=`, so the check passes when exactly 16 VMs already exist, allowing a 17th to be booted.

### Finding Description

At line 34, the constant is declared:

```rust
pub const MAX_VMS_COUNT: u64 = 16;
``` [1](#0-0) 

At line 532, the spawn guard is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
``` [2](#0-1) 

The condition is `> 16`, not `>= 16`. When the root VM plus 15 children are alive (`suspended + instantiated == 16`), the expression `16 > 16` evaluates to `false`, the guard is skipped, and `boot_vm` is called at line 539, producing a 17th concurrent VM. [3](#0-2) 

The root VM is itself stored in `instantiated`, so the count at the moment of the 16th spawn attempt is:

| VMs alive | `suspended + instantiated` | `> 16`? | Result |
|---|---|---|---|
| root + 15 children | 16 | false | spawn proceeds → 17 total |
| root + 16 children | 17 | true | correctly blocked |

### Impact Explanation

Each script execution in a CKB transaction runs inside a `Scheduler`. An attacker submits a transaction whose lock/type script (ScriptVersion::V2) spawns 16 children that each block on `Wait` for a non-existent pid, keeping all 17 VMs alive simultaneously. Each extra VM carries a full `Snapshot2` and machine state allocation. Multiplied across many concurrent transactions in the tx-pool or during block validation, this produces higher-than-expected memory pressure on validating nodes.

### Likelihood Explanation

The path is fully unprivileged: submit a transaction via RPC or P2P relay. No PoW, no key, no operator access required. The script is a standard V2 ELF binary. The condition is deterministic and locally reproducible.

### Recommendation

Change the guard from strict greater-than to greater-than-or-equal:

```rust
// Before (off-by-one):
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {

// After (correct):
if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
``` [4](#0-3) 

### Proof of Concept

1. Write a V2 RISC-V ELF that calls `spawn` in a loop 16 times; each child calls `wait(999)` (non-existent pid) so it stays alive in `WaitForRead`/`Runnable` state.
2. The root VM then calls `spawn` a 17th time.
3. With the current code, the 17th `spawn` returns `SUCCESS` and `Scheduler::states.len()` reaches 17.
4. With the fix (`>=`), the 17th `spawn` returns `MAX_VMS_SPAWNED`.
