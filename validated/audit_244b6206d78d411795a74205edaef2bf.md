### Title
Unbounded `pending_delivered` HashMap Growth via Unique-`from` `ConnectionRequest` Flood — (`network/src/protocols/hole_punching/mod.rs`)

---

### Summary

A single unprivileged connected peer can exhaust heap memory on a victim node by sending `ConnectionRequest` messages at 30 msg/sec where each message carries a unique `from` PeerId and the `to` field equals the victim's own `local_peer_id`. Every such message inserts a new entry into `pending_delivered` with no size cap. The only cleanup occurs in `notify()`, which fires every 5 minutes (`CHECK_INTERVAL = 300s`), allowing up to 9,000 entries per session per window.

---

### Finding Description

**Entry point** — `HolePunching::received()`: [1](#0-0) 

The `rate_limiter` is keyed by `(session_id, msg.item_id())`, capping throughput at **30 msg/sec per session**. This is the only global throttle on the path.

**`forward_rate_limiter` bypass** — `ConnectionRequestProcess::execute()`: [2](#0-1) 

This limiter is keyed by `(content.from, content.to, msg_item_id)`. With a unique `from` PeerId per message, every check passes — the limiter is completely bypassed.

**Deduplication bypass** — `respond_delivered()`: [3](#0-2) 

The dedup guard looks up `from_peer_id` in `pending_delivered`. With a unique `from` per message, no prior entry exists, so the guard never fires.

**Unbounded insert** — `respond_delivered()`: [4](#0-3) 

No size cap. Each unique `from_peer_id` inserts a `(Vec<Multiaddr>, u64)` tuple.

**Cleanup only in `notify()`** — fires every `CHECK_INTERVAL = 300s`: [5](#0-4) 

Between `notify()` invocations, `pending_delivered` grows without bound.

---

### Impact Explanation

- **Rate**: 30 msg/sec × 300s = **9,000 entries per session per 5-minute window**.
- **Memory**: Each entry stores up to `ADDRS_COUNT_LIMIT = 24` `Multiaddr` values (~50–100 bytes each) plus a `PeerId` key (~39 bytes). At minimum 1 address per entry: ~9,000 × ~100 bytes = ~900 KB; at maximum 24 addresses: ~9,000 × ~2.4 KB = ~21 MB per session per window.
- **Multiple sessions**: The victim node accepts multiple inbound connections. Each session independently contributes 9,000 entries, multiplying the effect.
- **HashMap degradation**: As `pending_delivered` grows, all `HashMap::get` and `HashMap::insert` operations on it slow down, affecting every subsequent `ConnectionRequest` processed in the same protocol handler task.
- **Scoped impact**: Heap exhaustion and protocol handler slowdown, matching the "network congestion with few costs" scope.

---

### Likelihood Explanation

The attacker needs only a single standard P2P connection — no special privileges, no PoW, no key material. Generating unique `PeerId` values (random 32-byte Ed25519 public keys) is trivially cheap. The `listen_addrs` constraint (lines 115–118, 196–218) requires at least one valid TCP/IP address, which the attacker trivially satisfies with their own IP. The `send_message_to` at line 226 must succeed (victim sends `ConnectionRequestDelivered` back to the attacker's session), which it does as long as the attacker stays connected. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Cap `pending_delivered`**: Enforce a maximum size (e.g., 1,000 entries). On overflow, either reject new inserts or evict the oldest entry (LRU).
2. **Reduce `CHECK_INTERVAL`** or add an inline eviction path inside `respond_delivered` that prunes expired entries before inserting.
3. **Key the `forward_rate_limiter` on `(session_id, to, msg_item_id)`** rather than `(from, to, msg_item_id)`, so the attacker cannot bypass it by rotating `from` values.
4. **Limit `pending_delivered` entries per session**: Track how many entries originated from each session and cap per-session contributions.

---

### Proof of Concept

```
1. Connect to victim node via standard P2P (single TCP session).
2. Obtain victim's local_peer_id (advertised in identify/discovery).
3. In a loop at 30 msg/sec for 300 seconds:
     from_i = PeerId::random()          // unique each iteration
     listen_addr = /ip4/<attacker_ip>/tcp/<port>/p2p/<from_i>
     send ConnectionRequest { from: from_i, to: victim_peer_id,
                               listen_addrs: [listen_addr], max_hops: 6, route: [] }
4. After 300s (before notify() fires):
     assert pending_delivered.len() == 9000
     assert no eviction has occurred
5. Observe heap growth of ~1–21 MB per session; repeat with N sessions for N× amplification.
```

The invariant `pending_delivered.len()` is bounded between `notify()` invocations is violated because neither the `forward_rate_limiter` nor the `respond_delivered` dedup guard is effective against unique `from` PeerIds, and no size cap exists on the map. [8](#0-7) [9](#0-8)
