### Title
Unvalidated `publisher_caps` Account in `advance` Instruction Allows Reward Drainage — (`governance/pyth_staking_sdk/src/idl/integrity-pool.json`)

### Summary

The `advance` instruction of the Oracle Integrity Staking (OIS) integrity pool program (`pyti8TM4zRVBjmarcgAPmTNNAXYKJv7WVHrkrm6woLN`) accepts a `publisher_caps` account with no IDL-level ownership, address, PDA, or program constraints. The `PublisherCaps` struct uses `bytemuck` (zero-copy) serialization, which means Anchor performs no discriminator check. An unprivileged caller can pass a crafted account with `is_verified = 1` and attacker-controlled cap data to manipulate reward calculations and drain `pool_reward_custody`.

### Finding Description

The `advance` instruction distributes PYTH rewards from `pool_reward_custody` to stakers based on publisher cap data read from the `publisher_caps` account. In the IDL, `publisher_caps` is declared with no constraints:

```json
{
  "name": "publisher_caps"
}
``` [1](#0-0) 

Every other security-sensitive account in the same instruction is constrained: `pool_data` uses `relations: ["pool_config"]`, `pool_config` is a PDA, and `pool_reward_custody` is a PDA. Only `publisher_caps` is unconstrained.

The `PublisherCaps` struct is declared with `bytemuck` serialization and a C-repr layout:

```json
{
  "name": "PublisherCaps",
  "repr": {"kind": "c"},
  "serialization": "bytemuck",
  "type": {
    "fields": [
      {"name": "write_authority", "type": "pubkey"},
      {"name": "is_verified", "type": "u8"},
      ...
      {"name": "publisher_caps_message_buffer", "type": {"array": ["u8", 40971]}}
    ]
  }
}
``` [2](#0-1) 

With `bytemuck`, Anchor skips the 8-byte discriminator check that normally enforces account type identity. The absence of an IDL-level `program` or `address` constraint on `publisher_caps` is consistent with the account being declared as `UncheckedAccount` or `AccountInfo` in the Rust handler — meaning no program ownership check is enforced by the framework.

The `publisher_caps` program (`pytcD8uUjPxSLMsNqoVnm9dXQw9tKJJf3CQnGwa8oL7`) sets `is_verified = 1` only after a valid Wormhole VAA is provided via `verify_publisher_caps`. However, if the integrity pool program does not check that the supplied `publisher_caps` account is owned by `pytcD8uUjPxSLMsNqoVnm9dXQw9tKJJf3CQnGwa8oL7`, an attacker can create an account owned by their own program, set `is_verified = 1` at the correct byte offset, and populate `publisher_caps_message_buffer` with inflated cap values. [3](#0-2) 

### Impact Explanation

The `advance` instruction reads publisher cap data from `publisher_caps` to compute per-publisher reward amounts and transfers tokens from `pool_reward_custody`. An attacker who controls the cap data can set arbitrarily large caps for a publisher they control, causing the program to transfer the entire balance of `pool_reward_custody` to their stake account. This is a direct theft of staked PYTH reward tokens.

**Note:** The governance parameter `y` (reward rate) is currently set to 0 per OP-PIP-103, meaning no rewards are actively accruing. However, `pool_reward_custody` may still hold tokens, and governance can raise `y` at any time. The structural vulnerability exists regardless of the current `y` value. [4](#0-3) 

### Likelihood Explanation

The `advance` instruction is permissionless — any `signer` can call it. The attacker only needs to:
1. Deploy a Solana program that creates an account with the correct `PublisherCaps` byte layout, `is_verified = 1`, and inflated cap data.
2. Call `advance` passing this crafted account as `publisher_caps`.

No privileged access, leaked keys, or governance majority is required. The attack is executable by any on-chain actor. [5](#0-4) 

### Recommendation

In the `advance` instruction handler, add an explicit check that the `publisher_caps` account is:
1. Owned by the `publisher_caps` program (`pytcD8uUjPxSLMsNqoVnm9dXQw9tKJJf3CQnGwa8oL7`).
2. Has `is_verified == 1`.

In Anchor, this can be enforced by declaring the account as `AccountLoader<'info, PublisherCaps>` with an explicit `#[account(owner = publisher_caps_program::ID)]` constraint, or by adding a manual `require_keys_eq!(publisher_caps.owner, PUBLISHER_CAPS_PROGRAM_ID)` check at the top of the handler. The IDL should reflect a `program` constraint on the `publisher_caps` account analogous to how `pool_reward_custody` specifies its program.

### Proof of Concept

```
1. Attacker deploys a Solana program `MaliciousCaps` that allocates an account
   with the PublisherCaps byte layout:
     - bytes 0..32:  any pubkey (write_authority)
     - byte 32:      0x01  (is_verified = 1)
     - bytes 33..37: padding
     - bytes 37..:   crafted publisher_caps_message_buffer with:
                       - attacker's publisher pubkey
                       - cap = u64::MAX

2. Attacker calls integrity_pool::advance with:
     signer              = attacker
     pool_data           = legitimate pool_data PDA
     publisher_caps      = MaliciousCaps account  ← no ownership check
     pool_config         = legitimate pool_config PDA
     pool_reward_custody = legitimate pool_reward_custody PDA

3. advance reads cap = u64::MAX for attacker's publisher,
   computes reward = y * min(stake, u64::MAX) ≈ full custody balance,
   and transfers pool_reward_custody balance to attacker's stake account.
``` [6](#0-5) [2](#0-1)
