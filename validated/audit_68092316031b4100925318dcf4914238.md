### Title
Insufficient Forwarded TON to Target Contract Causes Silent Failure and Permanent Fund Lock — (`File: target_chains/ton/contracts/contracts/Pyth.fc`)

---

### Summary

The `send_price_feeds_response` function in the Pyth TON contract forwards the remaining balance to a caller-specified `target_address` after reserving the `update_fee`. The fee validation in `calculate_and_validate_fees` only ensures `msg_value >= compute_fee + update_fee`, but never validates that the residual amount forwarded to `target_address` is sufficient for that contract to process the response. When the forwarded amount is too small, the target contract's execution fails, the bounceable message returns to Pyth, and Pyth silently discards it — permanently locking the excess TON inside the Pyth contract with no refund path.

---

### Finding Description

`calculate_and_validate_fees` enforces only two conditions:

```
throw_unless(ERROR_INSUFFICIENT_GAS, msg_value >= compute_fee);
int remaining_msg_value = msg_value - compute_fee;
throw_unless(ERROR_INSUFFICIENT_FEE, remaining_msg_value >= update_fee);
``` [1](#0-0) 

After this check passes, `send_price_feeds_response` does:

```
raw_reserve(update_fee, RESERVE_MODE::INCREASE_BY_ORIGINAL_BALANCE);  ;; reserves original_balance + update_fee
send_raw_message(...target_address..., SENDMODE::CARRY_ALL_BALANCE);   ;; sends msg_value - update_fee - compute_fee_consumed
``` [2](#0-1) 

The amount actually forwarded to `target_address` is:

```
forwarded ≈ msg_value − compute_fee_consumed − update_fee
```

If the user sends exactly `compute_fee + update_fee` (the validated minimum), the forwarded amount is approximately **0 nanotons**. Any smart contract at `target_address` (e.g., a DEX, lending protocol, or EVAA Master) requires a non-trivial amount of TON to pay for its own gas when processing the incoming response. With ~0 TON forwarded, the target's `recv_internal` runs out of gas and the message bounces.

The bounce is silently discarded in `Main.fc`:

```
if (flags & 1) { ;; ignore all bounced messages
    return ();
}
``` [3](#0-2) 

The bounced TON re-enters the Pyth contract's balance. Because there is no withdrawal or refund mechanism for these accumulated funds, they are permanently locked. The contract's own comment acknowledges this:

> "If the target contract rejects the message, the excess TON will remain in this contract and won't be automatically refunded to the original sender." [4](#0-3) 

---

### Impact Explanation

Any user who sends `OP_PARSE_PRICE_FEED_UPDATES` or `OP_PARSE_UNIQUE_PRICE_FEED_UPDATES` with a `target_address` pointing to a smart contract and provides only the minimum validated fee will:

1. Have their price-feed operation fail silently (no error is emitted to the original sender — `emit_error` is only called on parse failures, not on bounce).
2. Lose the `update_fee` permanently (intended behavior), but also lose any excess TON they sent beyond the minimum, which gets stuck in the Pyth contract.
3. Have any downstream protocol action (liquidation, swap, supply/withdraw) that depended on the Pyth response never execute, potentially leaving their assets in an intermediate state in the target protocol. [5](#0-4) 

---

### Likelihood Explanation

The `target_address` field is fully user-controlled and is a documented feature of the `OP_PARSE_PRICE_FEED_UPDATES` interface. The TON documentation and Pyth SDK guide users to send `calculateUpdatePriceFeedsFee(numUpdates) + updateFee` — exactly the minimum validated amount — without any guidance on the additional gas buffer required for the target contract. Any integrator following the documented minimum fee will trigger this condition. The Pyth proxy and on-chain getter patterns described in the developer documentation both rely on this forwarding path. [6](#0-5) 

---

### Recommendation

1. **Short term**: In `calculate_and_validate_fees`, add a third check requiring `msg_value >= compute_fee + update_fee + MIN_FORWARD_FEE`, where `MIN_FORWARD_FEE` is a configurable constant representing the minimum TON a target contract needs to process the response (analogous to `gas::jetton_wallet::est_transfer` in the referenced report).

2. **Short term**: If the residual forwarded amount is below `MIN_FORWARD_FEE`, emit an error response back to `sender_address` (via `emit_error`) instead of silently sending an underfunded message to `target_address`.

3. **Long term**: Add a bounce handler that, instead of silently discarding bounced messages, refunds the bounced TON to the original `sender_address` stored in the response payload.

---

### Proof of Concept

1. Alice calls `OP_PARSE_PRICE_FEED_UPDATES` on the Pyth TON contract, specifying her EVAA Master contract as `target_address`. She sends exactly `compute_fee + update_fee` nanotons (the minimum accepted by `calculate_and_validate_fees`).
2. Pyth validates the fee, processes the price feeds, calls `raw_reserve(update_fee, 4)`, and sends the response to EVAA Master with `SENDMODE::CARRY_ALL_BALANCE` — forwarding approximately 0 nanotons.
3. EVAA Master's `recv_internal` is invoked with ~0 TON. It cannot pay for its own gas and throws an exception.
4. TON sends a bounce message back to Pyth. Pyth's `recv_internal` detects `flags & 1` and returns immediately, discarding the bounce.
5. The ~0 TON (or any excess Alice sent) remains in Pyth's balance. Alice's liquidation/supply-withdraw operation never executes. No error is returned to Alice. The `update_fee` is consumed. [7](#0-6) [1](#0-0) [8](#0-7)

() calculate_and_validate_fees(int msg_value, int num_updates) impure {
    int update_fee = single_update_fee * num_updates;
    int compute_fee = get_compute_fee(
        WORKCHAIN,
        UPDATE_PRICE_FEEDS_BASE_GAS + (UPDATE_PRICE_FEEDS_PER_UPDATE_GAS * num_updates)
    );
    throw_unless(ERROR_INSUFFICIENT_GAS, msg_value >= compute_fee);

    ;; Check if the sender has sent enough TON to cover the update_fee
    throw_unless(ERROR_INSUFFICIENT_FEE, remaining_msg_value >= update_fee);
}

() send_price_feeds_response(tuple price_feeds, int msg_value, int op, slice sender_address, slice target_address, slice custom_payload) impure {
    ;; Build response cell with price feeds
    builder response = begin_cell()
        .store_uint(op, 32)  ;; Response op
        .store_uint(price_feeds.tlen(), 8);  ;; Number of price feeds

    ;; Create and store price feed cell chain
    cell price_feeds_cell = create_price_feed_cell_chain(price_feeds);
    cell custom_payload_cell = begin_cell().store_slice(custom_payload).end_cell();
    response = response.store_ref(price_feeds_cell).store_slice(sender_address).store_ref(custom_payload_cell);

    ;; Calculate update fee
    int update_fee = single_update_fee * price_feeds.tlen();

    ;; Reserve current_balance + fee
    raw_reserve(update_fee, RESERVE_MODE::INCREASE_BY_ORIGINAL_BALANCE);

    ;; SECURITY: Integrators MUST validate that messages are from this Pyth contract
    ;; in their receive function. Otherwise, attackers could:
    ;; 1. Send invalid price responses
    ;; 2. Impersonate users via sender_address and custom_payload fields
    ;; 3. Potentially drain the protocol
    ;;
    ;; Note: This message is bounceable. If the target contract rejects the message,
    ;; the excess TON will remain in this contract and won't be automatically refunded to the
    ;; original sender. Integrators should handle failed requests and refunds in their implementation.
    send_raw_message(begin_cell()
        .store_uint(0x18, 6)
        .store_slice(target_address)
        .store_coins(0)
        .store_uint(1, MSG_SERIALIZE_BITS)
        .store_ref(response.end_cell())
        .end_cell(),
        SENDMODE::CARRY_ALL_BALANCE);
}

        ;; Handle any unexpected errors
        emit_error(error_code, OP_PARSE_PRICE_FEED_UPDATES,
            sender_address, begin_cell().store_slice(custom_payload).end_cell());
