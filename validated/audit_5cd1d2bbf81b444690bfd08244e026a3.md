Audit Report

## Title
Unbounded `forward_rate_limiter` HashMap Growth via Attacker-Controlled PeerId Keys — (`network/src/protocols/hole_punching/mod.rs`)

## Summary
`HolePunching::forward_rate_limiter` uses a `governor::RateLimiter` backed by `HashMapStateStore<(PeerId, PeerId, u32)>`, which has no internal eviction. Its only cleanup call, `retain_recent()`, is placed exclusively in `disconnected()`. An unprivileged attacker maintaining a single TCP session can send `ConnectionRequest` messages at 30 msg/sec with unique attacker-controlled `(from, to)` PeerId pairs, inserting a new HashMap entry per message indefinitely, causing unbounded memory growth and eventual OOM crash of the node.

## Finding Description
`forward_rate_limiter` is declared as `RateLimiter<(PeerId, PeerId, u32)>` backed by `HashMapStateStore`, which performs no internal eviction. [1](#0-0) 

`retain_recent()` is called on both rate limiters only inside `disconnected()`, so cleanup only occurs when a peer disconnects. [2](#0-1) 

The `notify()` handler (fired every `CHECK_INTERVAL` = 5 minutes) cleans `pending_delivered` and `inflight_requests` but never calls `retain_recent()` on either rate limiter. [3](#0-2) 

The outer per-session rate limiter keys on `(session_id, msg.item_id())`, capping message rate per message type per session, but placing no bound on the cardinality of keys inserted into `forward_rate_limiter`. [4](#0-3) 

Each message that passes the outer check calls `forward_rate_limiter.check_key()` with an attacker-controlled `(content.from, content.to, msg_item_id)` triple, inserting a new entry for every unique pair. [5](#0-4) 

`content.from` and `content.to` are parsed directly from the message payload with no constraint that they correspond to actually-connected peers. [6](#0-5) 

The existing validation checks (`max_hops > MAX_HOPS`, `listen_addrs` length, `route.contains(self_peer_id)`) are all bypassable by the attacker by supplying valid values. None of them constrain the cardinality of `(from, to)` pairs. [7](#0-6) 

## Impact Explanation
At 30 msg/sec (the outer rate limiter cap) with unique `(from, to)` pairs, the HashMap grows at 30 entries/sec. A `PeerId` is a 39-byte multihash; each HashMap entry is approximately 100–120 bytes. Over a 24-hour session: 30 × 86,400 ≈ 2.6M entries ≈ ~300 MB from a single attacker session. Multiple concurrent sessions scale this linearly. Sustained over days, or with multiple sessions, this causes node OOM, crashing block/tx relay. This matches **High (10001–15000 points): Vulnerabilities which could easily crash a CKB node**.

## Likelihood Explanation
- HolePunching is enabled by default in production config.
- No authentication is required to open a HolePunching session.
- The attacker only needs to maintain a TCP connection and send well-formed `ConnectionRequest` messages with varying `from`/`to` byte fields.
- The outer rate limiter does not prevent this — it only caps message rate per `(session_id, item_id)`, not key cardinality in `forward_rate_limiter`.
- The 5-minute `notify()` timer is the natural cleanup point but `retain_recent()` is absent from it. [8](#0-7) 

## Recommendation
Add `retain_recent()` calls inside `notify()`:

```rust
async fn notify(&mut self, context: &mut ProtocolContext, _token: u64) {
    self.rate_limiter.retain_recent();
    self.forward_rate_limiter.retain_recent();
    // ... rest of existing logic
}
```

This ensures stale entries are evicted every 5 minutes regardless of whether any peer disconnects. [3](#0-2) 

## Proof of Concept
1. Connect to a CKB node with `HolePunching` enabled.
2. In a loop at 30 msg/sec, send `ConnectionRequest` messages where `from` and `to` are freshly generated random valid `PeerId` bytes (39-byte multihash format) each iteration, with a valid `listen_addrs` entry and `max_hops` ≤ 6.
3. Monitor the node's RSS memory. Assert it grows linearly with message count.
4. After 1 hour (30 × 3600 = 108,000 entries), observe ~13 MB growth from this map alone; after 24 hours, ~300 MB; no eviction occurs until the TCP session is closed.
5. Confirm: after calling `disconnect()`, `retain_recent()` fires and the map shrinks to near-zero. [2](#0-1)
