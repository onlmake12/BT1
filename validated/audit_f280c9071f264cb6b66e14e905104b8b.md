Looking at the code carefully across the full relay and orphan pool path.

### Title
Orphan Pool Monopolization via Missing Per-Peer Quota — (`tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` enforces only a global cap of `DEFAULT_MAX_ORPHAN_TRANSACTIONS = 100` entries with no per-peer sub-quota. A single relay peer can fill all 100 slots through the normal two-step relay announcement protocol, then maintain near-100% occupancy indefinitely by continuously injecting new orphans, causing every other peer's orphan transactions to be immediately evicted.

---

### Finding Description

**Root cause — no per-peer accounting in `add_orphan_tx`:**

`OrphanPool` stores a `peer: PeerIndex` field in each `Entry`, but never uses it for admission control. [1](#0-0) 

The `limit_size()` eviction loop picks victims by `self.entries.keys().next()` — HashMap iteration order, which is effectively random and peer-blind. [2](#0-1) 

**Attack path through the relay protocol:**

The relay handler in `TransactionsProcess::execute()` does require that the node previously requested a transaction before accepting it (the `unknown_tx_hashes` / `requesting_peer` filter): [3](#0-2) 

However, this is not a meaningful barrier. The attacker simply:

1. Sends `RelayTransactionHashes` with 100 tx hashes (well within `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER = 32767`).
2. The node responds with `GetRelayTransactions` for those hashes.
3. The attacker sends 100 orphan transactions whose parents it controls and will never relay.
4. All 100 slots are filled. [4](#0-3) 

To maintain monopoly, the attacker repeats steps 1–3 with one new hash per cycle. Each new insertion triggers `limit_size()`, which evicts a random entry. With 100 attacker entries and 1 victim entry, the victim is evicted with probability 100/101 ≈ 99%. The attacker's own entries are occasionally self-evicted, but re-filling costs only one additional relay round-trip.

The `add_orphan` call in `process.rs` is the bridge from the relay path into the orphan pool: [5](#0-4) 

The relay-level rate limiter (30 req/sec per `(peer, message_type)`) slows the refresh rate but does not prevent sustained monopoly. [6](#0-5) 

---

### Impact Explanation

Any peer whose orphan transaction is admitted while the attacker holds ≥99 slots will see it immediately evicted. The evicted tx hash is sent back to the relayer as `TxVerificationResult::Reject`, marking it as "unknown" in the bloom filter, so the node will not re-request it until the filter expires. Legitimate orphan chains (child-before-parent relay sequences) stall: the child is evicted before the parent arrives, breaking the `process_orphan_tx` resolution chain. This degrades orphan-dependent transaction propagation across any node the attacker connects to.

---

### Likelihood Explanation

The attack requires only a standard P2P relay connection and 100 relay messages to establish dominance, then ~1 message per eviction cycle to maintain it. No PoW, no keys, no privileged access. The `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` limit (32,767) is 327× the orphan pool size, so the attacker is never rate-limited at the announcement layer. The `TooManyUnknownTransactions` ban threshold is never reached. [7](#0-6) 

---

### Recommendation

Add a per-peer slot cap inside `OrphanPool`, e.g., `MAX_ORPHAN_TRANSACTIONS_PER_PEER = DEFAULT_MAX_ORPHAN_TRANSACTIONS / 3`. In `add_orphan_tx`, count existing entries for the incoming `peer` and reject (or evict the oldest entry from that peer) when the per-peer cap is exceeded. This mirrors Bitcoin Core's `DEFAULT_MAX_ORPHAN_TRANSACTIONS` per-peer eviction strategy.

---

### Proof of Concept

```
1. Attacker connects via RelayV3.
2. Attacker sends RelayTransactionHashes{tx_hashes: [h1..h100]}.
3. Node replies GetRelayTransactions{tx_hashes: [h1..h100]}.
4. Attacker sends RelayTransactions with 100 txs, each spending a
   non-existent parent output (attacker-controlled, never relayed).
   → All 100 enter orphan pool (peer = attacker).
5. Victim peer B sends RelayTransactionHashes{tx_hashes: [v1]}.
6. Node replies GetRelayTransactions{tx_hashes: [v1]}.
7. Victim sends RelayTransactions with orphan tx v1.
   → limit_size() evicts one random entry; with 100 attacker entries
     and 1 victim entry, P(victim evicted) = 100/101 ≈ 99%.
8. Attacker repeats step 2–4 with one new hash to refill any
   self-evicted slot.
Assert: orphan pool contains 0 entries from peer B after step 7
        in ~99% of trials.
```

    pub fn add_orphan_tx(
        &mut self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) -> Vec<Byte32> {
        if self.entries.contains_key(&tx.proposal_short_id()) {
            return vec![];
        }

        debug!("add_orphan_tx {}", tx.hash());
        self.entries.insert(
            tx.proposal_short_id(),
            Entry::new(tx.clone(), peer, declared_cycle),
        );

        for out_point in tx.input_pts_iter() {
            self.by_out_point
                .entry(out_point)
                .or_default()
                .insert(tx.proposal_short_id());
        }

        // DoS prevention: do not allow OrphanPool to grow unbounded
        self.limit_size()
    }

    pub(crate) async fn add_orphan(
        &self,
        tx: TransactionView,
        peer: PeerIndex,
        declared_cycle: Cycle,
    ) {
        let evicted_txs = self
            .orphan
            .write()
            .await
            .add_orphan_tx(tx, peer, declared_cycle);
        // for any evicted orphan tx, we should send reject to relayer
        // so that we mark it as `unknown` in filter
        for tx_hash in evicted_txs {
            self.send_result_to_relayer(TxVerificationResult::Reject { tx_hash });
        }
    }

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }

        // Check `unknown_tx_hashes`'s length after inserting the arrival `tx_hashes`
        if unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE
            || unknown_tx_hashes.len()
                >= self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER
        {
            warn!(
                "unknown_tx_hashes is too long, len: {}",
                unknown_tx_hashes.len()
            );

            let mut peer_unknown_counter = 0;
            for (_hash, priority) in unknown_tx_hashes.iter() {
                for peer in priority.peers.iter() {
                    if *peer == peer_index {
                        peer_unknown_counter += 1;
                    }
                }
            }
            if peer_unknown_counter >= MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER {
                return StatusCode::TooManyUnknownTransactions.into();
            }

            return Status::ignored();
        }
