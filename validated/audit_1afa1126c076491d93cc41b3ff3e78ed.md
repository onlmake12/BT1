### Title
Unbounded `last_n_numbers` via `difficulty_boundary=1` bypasses `GET_LAST_STATE_PROOF_LIMIT` in `GetLastStateProofProcess::execute` — (`util/light-client-protocol-server/src/components/get_last_state_proof.rs`)

---

### Summary

An unprivileged remote peer can craft a `GetLastStateProof` message that causes the server to iterate over every block in the chain inside `complete_headers`, performing one `get_ancestor` DB lookup and one `chain_root_mmr().get_root()` MMR computation per block. The existing guard only bounds the *input* parameters, not the actual computed work, so the invariant that server-side work is bounded by `GET_LAST_STATE_PROOF_LIMIT` is broken.

---

### Finding Description

The guard at the top of `execute` is:

```rust
if self.message.difficulties().len() + (last_n_blocks as usize) * 2
    > constant::GET_LAST_STATE_PROOF_LIMIT  // = 1000
``` [1](#0-0) [2](#0-1) 

With `difficulties=[]` and `last_n_blocks=1`, this evaluates to `0 + 2 = 2`, well under 1000. The check passes.

Later, `difficulty_boundary_block_number` is resolved by calling `get_first_block_total_difficulty_is_not_less_than(start_block_number=0, last_block_number, difficulty_boundary=1)`. Because the genesis block's `total_difficulty >= 1`, the function returns `Some((0, …))` immediately:

```rust
if start_total_difficulty >= *min_total_difficulty {
    return Some((start_block_number, start_total_difficulty));
}
``` [3](#0-2) 

So `difficulty_boundary_block_number = 0`. The adjustment at lines 313–316 only fires when `last_block_number - difficulty_boundary_block_number < last_n_blocks`, i.e., `last_block_number < 1`, which is false for any non-trivial chain: [4](#0-3) 

`last_n_numbers` is then built as:

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();
// = (0..N) → N entries for a chain of height N
``` [5](#0-4) 

Because `difficulty_boundary_block_number == 0`, the `sampled_numbers` branch falls into the `else` arm and returns `(Vec::new(), last_n_numbers)`: [6](#0-5) 

`block_numbers` therefore contains all N block numbers, and `complete_headers` iterates over every one of them: [7](#0-6) 

For each block, `complete_headers` performs:
- `snapshot.get_ancestor(last_hash, *number)` — a DB traversal
- `snapshot.chain_root_mmr(*number - 1).get_root()` — an MMR root computation [8](#0-7) 

---

### Impact Explanation

Work per request is O(chain_height), not O(`GET_LAST_STATE_PROOF_LIMIT`). On a chain with millions of blocks, a single crafted message causes the server to perform millions of DB reads and MMR root computations synchronously, exhausting CPU and I/O. Because the P2P handler processes this before returning, repeated requests from one or more peers can fully saturate the node.

---

### Likelihood Explanation

The attack requires no credentials, no PoW, and no special chain state beyond a valid tip hash (which is publicly broadcast). The attacker only needs to know the current tip hash and set `start_block_number=0`, `last_n_blocks=1`, `difficulty_boundary=U256::from(1)`. This is trivially constructable by any peer connected to the light-client protocol port.

---

### Recommendation

After computing `difficulty_boundary_block_number` and building `last_n_numbers`, add an explicit size check before proceeding:

```rust
let last_n_numbers =
    (difficulty_boundary_block_number..last_block_number).collect::<Vec<_>>();

if last_n_numbers.len() + sampled_numbers.len() > constant::GET_LAST_STATE_PROOF_LIMIT {
    return StatusCode::MalformedProtocolMessage.with_context("too many blocks in response");
}
```

The guard at lines 201–205 must be extended (or replaced) to bound the *computed* `last_n_numbers` size, not just the raw input fields.

---

### Proof of Concept

```
last_hash        = <any valid main-chain tip hash>
start_hash       = <genesis hash>
start_number     = 0
last_n_blocks    = 1
difficulty_boundary = U256::from(1)   // 0x01
difficulties     = []                 // empty vec
```

1. Guard: `0 + 1*2 = 2 ≤ 1000` → passes.
2. `get_first_block_total_difficulty_is_not_less_than(0, N, 1)` → returns `(0, genesis_difficulty)` immediately.
3. Adjustment: `N - 0 < 1` → false → no adjustment.
4. `last_n_numbers = (0..N)` → N entries.
5. `complete_headers` loops N times, calling `get_ancestor` + `chain_root_mmr().get_root()` per iteration.
6. Server CPU/IO saturated for the duration of the request; repeat to sustain exhaustion.
