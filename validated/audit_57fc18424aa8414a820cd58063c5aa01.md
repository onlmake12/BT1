Audit Report

## Title
Off-by-One in VM Count Guard Allows Spawning Beyond `MAX_VMS_COUNT` — (File: script/src/scheduler.rs)

## Summary
The `Scheduler` in `script/src/scheduler.rs` uses `>` instead of `>=` at line 532 when comparing the running VM count against `MAX_VMS_COUNT` (16). When exactly 16 VMs exist, the guard evaluates to `false` and `boot_vm` is called, producing a 17th VM. This violates the consensus-defined resource cap enforced by `MAX_VMS_COUNT`.

## Finding Description
`MAX_VMS_COUNT` is defined as `16` at line 34. In the `Message::Spawn` handler at line 532, the guard is:

```rust
if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
```

When `suspended.len() + instantiated.len() == 16`, the expression `16 > 16` is `false`. The guard body (which sets the return register to `MAX_VMS_SPAWNED` and `continue`s) is never entered. Execution falls through to `boot_vm` at line 539, which instantiates a 17th VM. No other guard exists between the check and the `boot_vm` call. The correct operator is `>=`: once the total reaches `MAX_VMS_COUNT`, no further spawn should be permitted.

## Impact Explanation
This is an incorrect implementation of CKB-VM behavior (High, 10001–15000 points). `MAX_VMS_COUNT` is a consensus-level resource cap. Bypassing it means the VM runtime does not enforce its own declared invariant. All nodes running this code accept the 17th VM uniformly (no consensus split), but memory, register state, and snapshot data for the extra VM are allocated beyond what the cap was designed to allow.

## Likelihood Explanation
Any unprivileged script author can trigger this by submitting a transaction containing a script that recursively calls `ckb_spawn` until 16 VMs exist, then issues one additional spawn. No special keys, privileges, or majority hashpower are required. The condition is deterministic and reproducible on every node.

## Recommendation
Change the comparison operator at line 532 from `>` to `>=`:

```diff
- if self.suspended.len() + self.instantiated.len() > MAX_VMS_COUNT as usize {
+ if self.suspended.len() + self.instantiated.len() >= MAX_VMS_COUNT as usize {
```

## Proof of Concept
Write a CKB script in C that reads a counter argument, recursively calls `ckb_spawn` on itself with `counter + 1` while the counter is less than 16, then issues one more `ckb_spawn` when the counter reaches 16. Assert the return value of that final spawn is `SUCCESS` (0) rather than `MAX_VMS_SPAWNED`. The existing test scaffolding in `script/src/verify/tests/ckb_latest/features_since_v2023.rs` (e.g., `check_spawn_index_out_of_bound`) provides the integration test pattern for loading such a binary and verifying the 17th spawn succeeds.
