### Title
Attacker-Controlled `tx.gasprice` in Permissionless `updatePriceFeeds` Enables Subscription Balance Drain â€” (`target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler::updatePriceFeeds` is a permissionless function that pays the calling keeper from the subscription's balance. The keeper fee is computed using the uncapped, attacker-controlled `tx.gasprice`. Any unprivileged address can call this function with an artificially inflated gas price when trigger conditions are met, extracting far more from the subscription balance than the legitimate gas cost and draining victim subscriptions.

---

### Finding Description

`updatePriceFeeds` is intentionally open to any caller (the permissionless keeper network): [1](#0-0) 

After verifying the update conditions, it calls `_processFeesAndPayKeeper`, which computes the keeper's reward as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) * numPriceIds;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
``` [2](#0-1) 

`tx.gasprice` is fully attacker-controlled and is never capped or validated. `GAS_OVERHEAD` is a fixed constant of **30,000 gas units**: [3](#0-2) 

The attacker's net profit per call is:

```
profit = GAS_OVERHEAD * tx.gasprice + keeperSpecificFee
       = 30_000 * tx.gasprice + (singleUpdateKeeperFeeInWei * numPriceIds)
```

because the attacker pays `actualGasUsed * tx.gasprice` but receives `(actualGasUsed + GAS_OVERHEAD) * tx.gasprice + keeperSpecificFee`. The `GAS_OVERHEAD` term is pure profit scaled by the attacker's chosen gas price.

The trigger condition check in `_validateShouldUpdatePrices` does revert if conditions are not met: [4](#0-3) 

However, this only prevents calls when conditions are unmet â€” it does not prevent a legitimate-condition call from using an inflated gas price. Once the heartbeat interval elapses or a price deviation threshold is crossed, the attacker can immediately front-run legitimate keepers with an extreme `tx.gasprice`.

---

### Impact Explanation

- **Direct fund loss**: A subscription funded with, say, 1 ETH can be drained in a single call. With `tx.gasprice = 100,000 gwei`, the attacker extracts `30,000 * 100,000 gwei = 3 ETH` worth of keeper fee (capped by the subscription balance). The attacker's actual gas cost is `actualGasUsed * 100,000 gwei`, but the `GAS_OVERHEAD` surplus is pure profit.
- **Subscription deactivation**: Once the balance falls below `minimumBalance`, the subscription cannot be updated and legitimate keepers cannot service it, breaking the push-oracle guarantee for downstream consumers.
- **Targeted griefing**: An attacker can selectively drain high-value subscriptions (e.g., those backing DeFi protocols) to cause stale price feeds. [5](#0-4) 

---

### Likelihood Explanation

- **High**: The function is permissionless by design. No registration, stake, or privileged role is required.
- The attacker only needs to wait for a natural trigger condition (heartbeat or deviation), which occurs regularly for any active subscription.
- The attack is a single transaction with no setup cost beyond gas.
- MEV bots can automate this: monitor the mempool for trigger conditions and front-run with inflated `tx.gasprice`.

---

### Recommendation

1. **Cap `tx.gasprice` in the fee calculation** to a governance-set `maxGasPrice` parameter:
   ```solidity
   uint256 effectiveGasPrice = Math.min(tx.gasprice, _state.maxGasPrice);
   uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * effectiveGasPrice;
   ```
2. **Alternatively**, use a fixed fee schedule (e.g., only `keeperSpecificFee`) and remove the gas-reimbursement component, or reimburse gas at a TWAP of recent base fees rather than the current `tx.gasprice`.
3. **Require keepers to be registered** with a stake, so malicious behavior can be slashed â€” though this conflicts with the permissionless design goal.

---

### Proof of Concept

1. Alice creates a subscription with `heartbeatSeconds = 60`, 2 price feeds, and a balance of 1 ETH.
2. 60 seconds elapse; the heartbeat trigger condition is now met.
3. Attacker calls `updatePriceFeeds(subscriptionId, validUpdateData)` with `tx.gasprice = 50_000 gwei`.
4. Inside `_processFeesAndPayKeeper`:
   - `actualGasUsed â‰ˆ 200_000` (typical for this function)
   - `gasCost = (200_000 + 30_000) * 50_000 gwei = 11.5 ETH` â€” exceeds balance, so capped by `InsufficientBalance` revert.
   - With `tx.gasprice = 3_000 gwei`: `gasCost = 230_000 * 3_000 gwei = 0.69 ETH`; attacker pays `200_000 * 3_000 gwei = 0.6 ETH`, nets `0.09 ETH` profit from `GAS_OVERHEAD` alone, plus `keeperSpecificFee`.
5. The subscription balance is drained in one or a few calls; the subscription falls below `minimumBalance` and becomes non-functional. [6](#0-5)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
```
