Audit Report

## Title
Unconditional `expect()` on `txs_sizes` in `FeeRateCollector::statistics` panics on pre-v0.106 `BlockExt` records — (`rpc/src/util/fee_rate.rs`)

## Summary
The `From<packed::BlockExtReader>` conversion for old-format (5-field) `BlockExt` records hard-codes `txs_sizes: None`. The `collect()` helper in `FeeRateCollector` only skips blocks where `get_block_ext_by_number` returns `None`; old-format blocks return `Some(BlockExt { txs_sizes: None, … })` and pass through to the closure, which calls `.expect("expect txs_size's length >= 1")` unconditionally, triggering a panic. Any unauthenticated caller can trigger this via `get_fee_rate_statistics` on a node upgraded in-place from before v0.106.

## Finding Description
**Deserialization path — old format yields `txs_sizes: None`**

`get_block_ext` in `store/src/store.rs` dispatches on `count_extra_fields()`. When `count_extra_fields() == 0` (pre-v0.106 record), it calls `reader.into()` which resolves to `From<packed::BlockExtReader>`. [1](#0-0) 

That `From` impl hard-codes both `cycles: None` and `txs_sizes: None`: [2](#0-1) 

**`collect()` does not filter `txs_sizes: None`**

The `filter_map` in `collect()` only drops entries where `get_block_ext_by_number` returns `None`. Old-format blocks return `Some(…)` and are forwarded to the fold closure: [3](#0-2) 

**Unconditional `expect` in the closure**

Inside `statistics()`, the closure destructures `block_ext` and immediately calls `.expect(…)` with no guard for the `None` case: [4](#0-3) 

For every pre-v0.106 block, `txs_sizes` is `None`, so this line panics.

**No `catch_unwind` in the RPC layer**

A search of `rpc/**/*.rs` finds zero uses of `catch_unwind` or any panic handler. The panic propagates unguarded through the synchronous RPC handler.

**Test suite gap**

Every entry in the existing test suite sets `txs_sizes: Some(…)`, so the `None` path is never exercised: [5](#0-4) 

## Impact Explanation
The concrete, proven impact is an **RPC API crash**: the panic aborts the tokio task handling the `get_fee_rate_statistics` request. This matches the allowed bounty impact: **Note (0–500 points) — Any local RPC API crash**. The claim of full node-process termination is speculative; tokio's default behavior isolates task panics and does not terminate the process, and no evidence in the submission proves otherwise. The severity is therefore Note, not High.

## Likelihood Explanation
Any node upgraded in-place from before v0.106 without a full re-sync retains old-format `COLUMN_BLOCK_EXT` records. The attacker requires no credentials, no peer relationship, and no PoW — only knowledge of the node's public JSON-RPC endpoint. A single unauthenticated HTTP POST is sufficient to trigger the panic. The condition is permanent until the old records are migrated or the node re-syncs from genesis.

## Recommendation
Replace the unconditional `expect` at `rpc/src/util/fee_rate.rs` line 93 with a graceful skip:

```rust
let Some(txs_sizes) = txs_sizes else {
    return fee_rates; // skip pre-v0.106 blocks that lack txs_sizes
};
```

Alternatively, filter in `collect()` by adding `.filter(|ext| ext.txs_sizes.is_some())` before the fold, or add a startup migration that back-fills `txs_sizes` for old-format blocks.

## Proof of Concept
1. Start a CKB node that has `COLUMN_BLOCK_EXT` entries with 5-field `BlockExt` (any node upgraded from before v0.106 without re-sync).
2. Send:
   ```json
   {"jsonrpc":"2.0","method":"get_fee_rate_statistics","params":[{"value":"0x65"}],"id":1}
   ```
3. `collect(101, …)` iterates the last 101 blocks, hits the first old-format block, deserializes it to `core::BlockExt { txs_sizes: None, … }`, and the closure fires `None.expect("expect txs_size's length >= 1")` → panic.

A minimal unit test reproduces this by inserting a `BlockExt { txs_sizes: None, … }` into `DummyFeeRateProvider` and calling `FeeRateCollector::new(&provider).statistics(None)` — the existing test suite never covers this case.
