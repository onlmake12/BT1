Audit Report

## Title
Global `unknown_tx_hashes` Soft Limit Bypassed by Post-Insertion Check in `add_ask_for_txs` - (File: sync/src/types/mod.rs)

## Summary

`SyncState::add_ask_for_txs` inserts all incoming tx hashes into the global `unknown_tx_hashes` map before evaluating the global soft limit (`MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000). Because the per-peer quota (32,767) exceeds half the global cap, just two peers with disjoint novel hashes are sufficient to permanently exceed the cap. With up to 128 relay peers, the map can grow to ~4.2M entries, and every subsequent call triggers an O(map_size) full scan while holding the mutex, blocking the relay timer that dispatches transaction requests.

## Finding Description

**Root cause:** In `sync/src/types/mod.rs`, `add_ask_for_txs` (line 1483) iterates and inserts all incoming hashes (lines 1486–1504) — up to `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` (32,767) per call — then checks the global limit (lines 1507–1509). The code comment on line 1506 explicitly acknowledges this ordering: *"Check `unknown_tx_hashes`'s length after inserting the arrival `tx_hashes`"*. When the check fires and returns `TooManyUnknownTransactions` or `Status::ignored()`, the entries are already committed to the map and are not rolled back.

**Constants mismatch** (`util/constant/src/sync.rs`, lines 68–72):
- `MAX_RELAY_TXS_NUM_PER_BATCH` = `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` = 32,767
- `MAX_UNKNOWN_TX_HASHES_SIZE` = 50,000
- The global cap is smaller than 2 × per-peer quota (65,534), so two peers suffice to exceed it.

**Exploit flow:**
1. Peer A sends one `RelayTransactionHashes` message with 32,767 unique novel hashes. All inserted (map: 32,767 < 50,000 → limit check not triggered). Returns `Status::ok()`.
2. Peer B sends 32,767 different novel hashes. All inserted (map: 65,534). Post-insertion check fires (65,534 ≥ 50,000). Peer B's counter = 32,767 ≥ `MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` → returns `TooManyUnknownTransactions`. **Entries remain in the map.**
3. Each additional peer adds another 32,767 entries. With N peers: map grows to N × 32,767.
4. With `MAX_RELAY_PEERS` = 128: map reaches ~4.2M entries.

**Why existing guards fail:**
- The per-peer rate limiter (lines 116–123 of `sync/src/relayer/mod.rs`) limits message frequency (30 req/s per peer per message type). Only one message per peer is needed; the rate limiter is never triggered.
- `TransactionHashesProcess::execute` (lines 38–47 of `sync/src/relayer/transaction_hashes_process.rs`) only filters hashes already in `tx_filter`. Novel fake hashes (which need not correspond to real transactions) pass through unconditionally.
- The second condition `self.peers.state.len() * MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER` scales with peer count and does not prevent growth before the first condition triggers.

**O(N) mutex scan:** Once the global threshold is exceeded, every subsequent call to `add_ask_for_txs` executes the full-map iteration at lines 1516–1523 while holding `unknown_tx_hashes.lock()`. This is O(map_size × avg_peers_per_entry). With millions of entries, this blocks `pop_ask_for_txs` (which also acquires `unknown_tx_hashes.lock()` at line 1454) and `mark_as_known_txs`, stalling transaction relay.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

With `MAX_RELAY_PEERS` = 128, the map can reach ~4.2M entries. Each entry holds a `Byte32` (32 bytes) plus a `Vec<PeerIndex>`, yielding hundreds of MB of heap growth. More critically, the O(map_size) scan while holding the mutex blocks `pop_ask_for_txs`, which dispatches `GetRelayTransactions` requests to peers on the relay timer. Sustained mutex contention stalls transaction propagation across the node, contributing to network-wide relay degradation. The attack requires no special privilege and minimal bandwidth (one P2P message per peer).

## Likelihood Explanation

**Medium-High.** Any peer that completes the P2P handshake can send `RelayTransactionHashes`. Only two peers with disjoint sets of fake (non-existent) tx hashes are needed to exceed the 50,000 global limit. The hashes need not correspond to real transactions — the only filter is `tx_filter` (known transactions), which novel hashes bypass trivially. No key, hashpower, or elevated privilege is required. The rate limiter does not prevent this since only one message per peer is needed.

## Recommendation

Check the global limit **before** inserting, not after. At the start of `add_ask_for_txs`, before the insertion loop, check whether `unknown_tx_hashes.len() >= MAX_UNKNOWN_TX_HASHES_SIZE`. If so, perform the per-peer count check first and reject if the peer is already at quota, or drop the request entirely.

Additionally, maintain a separate `HashMap<PeerIndex, usize>` tracking per-peer entry counts to replace the O(map_size) scan at lines 1516–1523 with an O(1) lookup, eliminating the mutex contention amplification.

## Proof of Concept

1. Start a CKB node and connect two test peers (Peer A, Peer B) via the relay protocol.
2. Peer A sends one `RelayTransactionHashes` message containing 32,767 unique `Byte32` values not present in `tx_filter`. Observe `unknown_tx_hashes.len()` = 32,767; no limit check triggered; returns `Status::ok()`.
3. Peer B sends one `RelayTransactionHashes` message containing 32,767 **different** unique `Byte32` values. Observe `unknown_tx_hashes.len()` = 65,534 after the call returns `TooManyUnknownTransactions` — entries are already inserted, global cap bypassed.
4. Add a third peer (Peer C) with another 32,767 disjoint hashes. Observe map grows to 98,301.
5. Instrument `pop_ask_for_txs` (called on `ASK_FOR_TXS_TOKEN`) to measure mutex wait time; observe it spikes proportionally to map size due to the O(N) scan in `add_ask_for_txs` holding the lock.
6. Confirm that relay dispatch of `GetRelayTransactions` is delayed, blocking transaction propagation.

    pub fn add_ask_for_txs(&self, peer_index: PeerIndex, tx_hashes: Vec<Byte32>) -> Status {
        let mut unknown_tx_hashes = self.unknown_tx_hashes.lock();

        for tx_hash in tx_hashes
            .into_iter()
            .take(MAX_UNKNOWN_TX_HASHES_SIZE_PER_PEER)
        {
            match unknown_tx_hashes.entry(tx_hash) {
                keyed_priority_queue::Entry::Occupied(entry) => {
                    let mut priority = entry.get_priority().clone();
                    priority.push_peer(peer_index);
                    entry.set_priority(priority);
                }
                keyed_priority_queue::Entry::Vacant(entry) => {
                    entry.set_priority(UnknownTxHashPriority {
                        request_time: Instant::now(),
                        peers: vec![peer_index],
                        requested: false,
                    })
                }
            }
        }
