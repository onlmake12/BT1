### Title
Unguarded `set_parameters` Allows Any Signer to Seize Stake Cap Authority - (File: `pythnet/stake_caps_parameters/programs/stake_caps_parameters/src/lib.rs`)

---

### Summary
The `set_parameters` instruction in the `stake_caps_parameters` program declares its `signer` account with no address constraint. When the parameters PDA is uninitialized (`current_authority == Pubkey::default()`), any transaction sender can call `set_parameters`, set themselves as `current_authority`, and write arbitrary `m` and `z` values — permanently seizing control of the stake cap parameters before the legitimate operator can.

---

### Finding Description
The `SetParameters` accounts struct declares:

```rust
#[account(mut)]
pub signer: Signer<'info>,
```

with no `address` constraint binding it to a hard-coded admin pubkey. The authorization logic inside the handler is:

```rust
require!(
    ctx.accounts.signer.key() == stored_parameters.current_authority
        || stored_parameters.current_authority == Pubkey::default(),
    ErrorCode::WrongAuthority
);
**stored_parameters = parameters;
```

The second branch (`current_authority == Pubkey::default()`) is the zero-initialized state of a freshly created account. Because the account uses `init_if_needed`:

```rust
#[account(
    init_if_needed,
    seeds = ["parameters".as_bytes()],
    bump,
    payer = signer,
    space = Parameters::LEN
)]
pub parameters: Account<'info, Parameters>,
```

the account starts with all fields zero-initialized, so `current_authority` is `Pubkey::default()` on first creation. Any signer who calls `set_parameters` before the legitimate operator can pass the `parameters` argument with their own pubkey as `current_authority` and arbitrary `m`/`z` values. The entire struct is then written unconditionally (`**stored_parameters = parameters`), making the attacker the permanent authority.

Notably, the constant `PARAMETERS_ADDRESS` is defined in the program but is **never enforced** as a constraint on the `parameters` account — it is purely informational.

---

### Impact Explanation
The attacker permanently controls the `stake_caps_parameters` PDA. They can set `m` and `z` to extreme values (e.g., zero or `u64::MAX`), corrupting the stake cap calculations used by the Pyth staking system. Downstream staking logic that reads these parameters to compute per-publisher stake caps would operate on attacker-controlled values, potentially allowing unlimited staking or zeroing all caps, breaking the economic security model of the staking program.

---

### Likelihood Explanation
Medium during deployment. Solana transactions are publicly visible before finalization. An attacker monitoring the network for the first `set_parameters` call can submit a competing transaction with a higher priority fee. The race window exists from program deployment until the legitimate operator's initialization transaction is confirmed. If the parameters account is ever closed and needs re-initialization, the same race applies again.

---

### Recommendation
Add an `address` constraint on the `signer` account binding it to a hard-coded admin pubkey (analogous to the fix described in the report — using Anchor's `address` constraint):

```rust
#[account(
    mut,
    address = ADMIN_PUBKEY @ ErrorCode::WrongAuthority
)]
pub signer: Signer<'info>,
```

Alternatively, remove the `|| stored_parameters.current_authority == Pubkey::default()` branch entirely and use a separate one-time `initialize` instruction that is constrained to a hard-coded deployer key, separating initialization from updates.

---

### Proof of Concept
1. Deploy or observe a fresh deployment of `stake_caps_parameters` where the `parameters` PDA does not yet exist.
2. Construct a `set_parameters` transaction with:
   - `signer` = attacker's keypair
   - `parameters.current_authority` = attacker's pubkey
   - `parameters.m` = `u64::MAX`
   - `parameters.z` = `0`
3. Submit with a higher priority fee than the legitimate operator's initialization transaction.
4. If confirmed first, the attacker is now the permanent `current_authority` and the stake cap parameters are set to attacker-chosen values. All subsequent legitimate calls to `set_parameters` will fail with `WrongAuthority`. [1](#0-0) [2](#0-1) [3](#0-2)
