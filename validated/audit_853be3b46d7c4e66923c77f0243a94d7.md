Audit Report

## Title
Missing Per-Peer Rate Limit on BlockFilter Protocol Enables DB Read Amplification DoS — (`sync/src/filter/mod.rs`)

## Summary
The `BlockFilter` P2P protocol handler contains no rate-limiting mechanism of any kind. Any unprivileged peer can flood all three `Get*` message types at wire speed, each triggering up to thousands of synchronous RocksDB reads per message. Unlike the `Relayer` protocol, which enforces a `RateLimiter<(PeerIndex, u32)>` at 30 req/s, `BlockFilter` has no such guard, allowing sustained read amplification that can render a node unresponsive.

## Finding Description
The `BlockFilter` struct holds only a `shared` field with no rate limiter: [1](#0-0) 

The `received` handler parses the message and immediately calls `self.process(...)` with no rate check: [2](#0-1) 

By contrast, `Relayer` defines a `RateLimiter<(PeerIndex, u32)>` field and enforces it in `try_process` before any dispatch: [3](#0-2) [4](#0-3) 

A `grep` for `rate_limiter` across all of `sync/src/` confirms it appears **only** in `sync/src/relayer/mod.rs`.

Each message type performs a tight DB-read loop with no time or rate bound:

- **`GetBlockFilterCheckPoints`**: `BATCH_SIZE=2000`, `CHECK_POINT_INTERVAL=2000`, each iteration calls `get_block_hash` + `get_block_filter_hash` → up to **4,000 DB reads** per message: [5](#0-4) [6](#0-5) 

- **`GetBlockFilterHashes`**: `BATCH_SIZE=2000`, same two-read pattern plus one extra for the parent → up to **4,001 DB reads** per message: [7](#0-6) [8](#0-7) 

- **`GetBlockFilters`**: `BATCH_SIZE=1000`, calls `get_block_hash` + `get_block_filter` → up to **2,000 DB reads** per message: [9](#0-8) [10](#0-9) 

No ban or disconnect is ever triggered for excessive requests — the only ban path in `process` is for malformed messages or handler-returned ban statuses, neither of which fires for valid but repeated `Get*` requests: [11](#0-10) 

## Impact Explanation
Sustained flooding from a single peer saturates the shared RocksDB read thread pool and block-cache bandwidth. This degrades or blocks all concurrent DB operations — chain sync, block validation, tx-pool lookups — causing effective node unresponsiveness. This matches the **High** impact class: *"Vulnerabilities which could easily crash a CKB node"* (operational unresponsiveness is equivalent for a production full node or validator). Multiple attacking peers amplify the effect linearly with no additional cost.

## Likelihood Explanation
The attack requires only a standard P2P connection — no authentication, no PoW, no stake. The Filter protocol is enabled by default on nodes serving light clients. The attacker freely controls `start_number`; setting it to `0` guarantees the full batch is always served. The attack is trivially repeatable and requires no special tooling beyond a basic P2P client.

## Recommendation
Add a per-peer, per-message-type rate limiter to `BlockFilter`, mirroring the pattern in `Relayer`:

- Add a `rate_limiter: RateLimiter<(PeerIndex, u32)>` field to the `BlockFilter` struct in `sync/src/filter/mod.rs`.
- In `try_process`, check `self.rate_limiter.check_key(&(peer, message.item_id()))` before dispatching, returning `StatusCode::TooManyRequests` on `Err`.
- Tune the quota conservatively (e.g., 1–5 req/s per peer per message type) given the high per-message DB cost relative to the Relayer's 30 req/s cap.
- Call `self.rate_limiter.retain_recent()` in the `disconnected` handler to bound memory growth.

## Proof of Concept
```
1. Connect to a CKB node with the Filter protocol enabled (default for light-client-serving nodes).
2. In a tight loop, send valid messages:
     GetBlockFilterCheckPoints { start_number: 0 }
     GetBlockFilterHashes      { start_number: 0 }
     GetBlockFilters           { start_number: 0 }
3. Repeat at network speed from a single TCP connection.
4. Monitor RocksDB read latency and the node's ability to process
   new blocks or relay transactions — both degrade rapidly.
5. Observe that no ban or disconnect is triggered, as no status
   code indicating abuse is ever returned by the filter handlers.
6. Scale to multiple peers for linear amplification.
```
