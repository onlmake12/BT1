## Verdict

### Title
Cycle-to-CPU-Cost Imbalance in `load_by_field(LockHash/TypeHash)` Enables Node DoS via Unbounded Host-Side Blake2b Computation — (`script/src/syscalls/load_cell.rs`)

### Summary
`load_by_field` for `CellField::LockHash` and `CellField::TypeHash` charges only `transferred_byte_cycles(32) = 8` cycles (for the 32-byte hash output), but unconditionally recomputes `blake2b_256` over the full serialized script on every invocation. The host-side hash computation is O(script_size) but is entirely outside the VM cycle accounting. An attacker can exploit this to cause ~900× more CPU work than the cycle budget implies.

### Finding Description
In `load_by_field`, `CellField::LockHash` calls `output.calc_lock_hash()` which calls `blake2b_256(self.as_slice())` over the entire serialized script (code_hash + hash_type + args). The result is always 32 bytes, so `transferred_byte_cycles(32) = 8` cycles are charged. There is no caching of the hash result in `CellMeta` or anywhere in the syscall path. A script can call this syscall in a tight loop, paying ~11 VM cycles per iteration while forcing the host to compute blake2b over up to ~500 KB per call.

### Impact Explanation
A single transaction submitted via the public RPC can cause ~640 seconds of host CPU time during tx-pool verification, while the VM cycle limit is satisfied normally. This blocks or severely degrades the node's transaction verification capacity. The same applies to `TypeHash`.

### Likelihood Explanation
- Reachable via unauthenticated transaction submission RPC.
- Economic cost is negligible (~0.005 CKB for a 500 KB cell).
- No PoW or privileged access required.
- The transaction passes all existing validation checks (size, cycles, capacity).

### Recommendation
1. **Cache the lock/type hash** in `CellMeta` (pre-compute once when the cell is loaded, store alongside `mem_cell_data_hash`).
2. **Alternatively**, charge cycles proportional to the script size when computing the hash: `transferred_byte_cycles(script.as_slice().len())` instead of `transferred_byte_cycles(32)`.

### Proof of Concept
1. Create and commit a transaction with an output cell whose lock script has ~500 KB of args (costs ~0.005 CKB).
2. Write a script that calls `ckb_load_cell_by_field(buf, &len, 0, 0, CKB_SOURCE_INPUT, CKB_CELL_FIELD_LOCK_HASH)` in a tight loop until cycles are exhausted.
3. Submit the transaction via `send_transaction` RPC.
4. Observe that node CPU is saturated for ~640 seconds while the VM cycle counter only reaches 70M.
5. Benchmark: wall-clock time grows linearly with lock script size; cycles charged remain constant at 8 per call.

    fn load_by_field<Mac: SupportMachine>(
        &self,
        machine: &mut Mac,
        cell: &CellMeta,
    ) -> Result<(u8, u64), VMError> {
        let field = CellField::parse_from_u64(machine.registers()[A5].to_u64())?;
        let output = &cell.cell_output;

        let result = match field {
            CellField::Capacity => {
                let capacity: Capacity = output.capacity().into();
                let mut buffer = vec![];
                buffer.write_u64::<LittleEndian>(capacity.as_u64())?;
                (SUCCESS, store_data(machine, &buffer)?)
            }
            CellField::DataHash => {
                if let Some(bytes) = self.tx_info().data_loader.load_cell_data_hash(cell) {
                    (SUCCESS, store_data(machine, &bytes.as_bytes())?)
                } else {
                    (ITEM_MISSING, 0)
                }
            }
            CellField::OccupiedCapacity => {
                let mut buffer = vec![];
                buffer.write_u64::<LittleEndian>(
                    cell.occupied_capacity()
                        .map_err(|_| {
                            VMError::Unexpected(format!(
                                "Unexpected occupied_capacity overflow {}",
                                cell.out_point,
                            ))
                        })?
                        .as_u64(),
                )?;
                (SUCCESS, store_data(machine, &buffer)?)
            }
            CellField::Lock => {
                let lock = output.lock();
                let data = lock.as_slice();
                (SUCCESS, store_data(machine, data)?)
            }
            CellField::LockHash => {
                let hash = output.calc_lock_hash();
                let bytes = hash.as_bytes();
                (SUCCESS, store_data(machine, &bytes)?)
            }
            CellField::Type => match output.type_().to_opt() {
                Some(type_) => {
                    let data = type_.as_slice();
                    (SUCCESS, store_data(machine, data)?)
                }
                None => (ITEM_MISSING, 0),
            },
            CellField::TypeHash => match output.type_().to_opt() {
                Some(type_) => {
                    let hash = type_.calc_script_hash();
                    let bytes = hash.as_bytes();
                    (SUCCESS, store_data(machine, &bytes)?)
                }
                None => (ITEM_MISSING, 0),
            },
        };
        Ok(result)
    }

//! CKB VM cost model.
//!
//! The cost model assign cycles to instructions.

/// How many bytes can transfer when VM costs one cycle.
// 0.25 cycles per byte
pub const BYTES_PER_CYCLE: u64 = 4;

/// Calculates how many cycles spent to load the specified number of bytes.
pub fn transferred_byte_cycles(bytes: u64) -> u64 {
    // Compiler will optimize the divisin here to shifts.
    bytes.div_ceil(BYTES_PER_CYCLE)
}
