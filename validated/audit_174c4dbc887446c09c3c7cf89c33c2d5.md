### Title
Silent Truncating Cast in `calculate_maximum_withdraw` Returns Wrong DAO Withdrawal Amount — (`File: util/dao/src/lib.rs`)

### Summary
In `DaoCalculator::calculate_maximum_withdraw`, the intermediate `u128` result `withdraw_counted_capacity` is cast to `u64` using the silent truncating `as u64` operator instead of the checked `u64::try_from(...).map_err(|_| DaoError::Overflow)?` pattern used consistently everywhere else in the same file. When the product overflows `u64::MAX`, the function silently returns a drastically wrong (too-small) capacity value instead of propagating an overflow error, corrupting DAO withdrawal accounting.

---

### Finding Description

In `util/dao/src/lib.rs`, `calculate_maximum_withdraw` computes the withdrawable capacity as:

```rust
let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
    * u128::from(withdrawing_ar)
    / u128::from(deposit_ar);
let withdraw_capacity =
    Capacity::shannons(withdraw_counted_capacity as u64)   // ← silent truncation
        .safe_add(occupied_capacity)?;
``` [1](#0-0) 

The `as u64` cast silently discards the upper 64 bits of `withdraw_counted_capacity` when it exceeds `u64::MAX`. The rest of the same file uses the safe pattern consistently:

```rust
// line 245
Capacity::shannons(u64::try_from(miner_issuance128).map_err(|_| DaoError::Overflow)?)
// line 258
let ar_increase = u64::try_from(ar_increase128).map_err(|_| DaoError::Overflow)?;
``` [2](#0-1) 

The overflow condition is:

```
counted_capacity × withdrawing_ar / deposit_ar  >  u64::MAX
```

`counted_capacity` is at most the total CKB supply (~3.36 × 10¹⁸ shannons). `deposit_ar` starts at `10_000_000_000_000_000` (10¹⁶) and grows monotonically. The ratio `withdrawing_ar / deposit_ar` must exceed ~5.36× for overflow to occur with a maximum-capacity cell, which corresponds to many decades of protocol operation at the current secondary issuance rate.

When the overflow occurs, `withdraw_counted_capacity as u64` wraps to a small value (e.g., if the true value is `u64::MAX + 1000`, the cast yields `999`). The subsequent `safe_add` then succeeds and returns a silently wrong, drastically under-counted capacity.

The existing test `check_withdraw_calculation_overflows` does **not** cover this case — it only exercises the path where the final `safe_add` overflows, not the path where `withdraw_counted_capacity` itself overflows `u64` before the cast. [3](#0-2) 

---

### Impact Explanation

`calculate_maximum_withdraw` is called from three paths:

1. **RPC `calculate_dao_maximum_withdraw`** (`rpc/src/module/experiment.rs` lines 259, 288): returns a silently wrong (too-small) withdrawal amount to the caller, misleading users and tooling about available DAO interest. [4](#0-3) 

2. **`transaction_fee`** (line 31): computes `maximum_withdraw - outputs_capacity`. A wrong `maximum_withdraw` produces a wrong fee, breaking fee-based admission logic. [5](#0-4) 

3. **`withdrawed_interests` → `dao_field_with_current_epoch`** (lines 316–332): the truncated value feeds into `current_s = parent_s + nervosdao_issuance - withdrawed_interests`. If the truncated `maximum_withdraws` is smaller than `input_capacities`, `safe_sub` fails and block production/validation aborts — a **DoS on DAO withdrawals** at the block-processing level. If it happens to succeed, the DAO field `current_s` is inflated, corrupting the on-chain interest accounting. [6](#0-5) 

---

### Likelihood Explanation

**Low.** The overflow requires `withdrawing_ar / deposit_ar > ~5.36` for a maximum-capacity cell. Given the initial `ar = 10^16` and the slow secondary issuance rate (~4% annually), this ratio would take on the order of 40+ years to reach. However, the bug is structurally identical to the inconsistency flagged in the reference report: a missing overflow check in one arithmetic path while all sibling paths in the same function use the correct checked conversion. It is a latent correctness defect that will eventually manifest.

---

### Recommendation

Replace the silent truncating cast with the same checked conversion used elsewhere in the file:

```diff
-    Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?
+    Capacity::shannons(
+        u64::try_from(withdraw_counted_capacity).map_err(|_| DaoError::Overflow)?
+    ).safe_add(occupied_capacity)?
``` [7](#0-6) 

---

### Proof of Concept

Construct a scenario where `withdraw_counted_capacity` overflows `u64`:

```
deposit_ar      = 10_000_000_000_000_000   (initial ar)
withdrawing_ar  = 53_600_000_000_000_000   (ar after ~40 years, ratio ≈ 5.36×)
counted_capacity = 3_360_000_000_000_000_000  (≈ total CKB supply in shannons)

withdraw_counted_capacity
  = 3_360_000_000_000_000_000 × 53_600_000_000_000_000
    / 10_000_000_000_000_000
  = 18_009_600_000_000_000_000   >  u64::MAX (18_446_744_073_709_551_615)
```

With the current code, `withdraw_counted_capacity as u64` silently truncates to `18_009_600_000_000_000_000 - 18_446_744_073_709_551_616 = ...` a small residual value. `safe_add(occupied_capacity)` succeeds, and `calculate_maximum_withdraw` returns `Ok(small_wrong_value)` instead of `Err(DaoError::Overflow)`.

The same truncation propagates through `withdrawed_interests` into `dao_field_with_current_epoch`, corrupting the `current_s` field stored in every subsequent block header, or causing `safe_sub` to fail and DoS block production for any block containing a DAO withdrawal. [8](#0-7)

    /// Calculate maximum withdraw capacity of a deposited dao output
    pub fn calculate_maximum_withdraw(
        &self,
        output: &CellOutput,
        output_data_capacity: Capacity,
        deposit_header_hash: &Byte32,
        withdrawing_header_hash: &Byte32,
    ) -> Result<Capacity, DaoError> {
        let deposit_header = self
            .data_loader
            .get_header(deposit_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        let withdrawing_header = self
            .data_loader
            .get_header(withdrawing_header_hash)
            .ok_or(DaoError::InvalidHeader)?;
        if deposit_header.number() >= withdrawing_header.number() {
            return Err(DaoError::InvalidOutPoint);
        }

        let (deposit_ar, _, _, _) = extract_dao_data(deposit_header.dao());
        let (withdrawing_ar, _, _, _) = extract_dao_data(withdrawing_header.dao());

        let occupied_capacity = output.occupied_capacity(output_data_capacity)?;
        let output_capacity: Capacity = output.capacity().into();
        let counted_capacity = output_capacity.safe_sub(occupied_capacity)?;
        let withdraw_counted_capacity = u128::from(counted_capacity.as_u64())
            * u128::from(withdrawing_ar)
            / u128::from(deposit_ar);
        let withdraw_capacity =
            Capacity::shannons(withdraw_counted_capacity as u64).safe_add(occupied_capacity)?;

        Ok(withdraw_capacity)
    }
