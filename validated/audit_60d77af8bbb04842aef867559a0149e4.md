### Title
`PythUtils.deriveCrossRate()` Missing Zero-Price Guard Causes Unhandled Panic Revert - (File: `target_chains/ethereum/sdk/solidity/PythUtils.sol`)

### Summary

`PythUtils.deriveCrossRate()` validates that `price2` is not negative but does not guard against `price2 == 0`. When `price2 == 0`, both execution branches pass the denominator `uint64(price2) == 0` directly into `Math.mulDiv()`, which triggers an unhandled panic revert (division-by-zero) instead of a meaningful protocol error. This is structurally identical to the Knox `_getPositivePlaceValues()` bug: a math library function silently accepts an edge-case input that passes all explicit guards, then panics inside an arithmetic primitive.

### Finding Description

`deriveCrossRate` guards against negative prices with:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
```

Zero passes this check. Execution then reaches one of two `Math.mulDiv` calls:

**Branch 1 (`deltaExponent > 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    10 ** uint64(deltaExponent),
    uint64(price2)          // denominator == 0 when price2 == 0
);
```

**Branch 2 (`deltaExponent <= 0`):**
```solidity
result = Math.mulDiv(
    uint64(price1),
    1,
    10 ** uint64(Math.abs(deltaExponent)) * uint64(price2)  // denominator == 0
);
```

`Math.mulDiv` explicitly panics on a zero denominator:

```solidity
if (high == 0) {
    return low / denominator;   // Solidity 0.8 reverts here
}
if (denominator <= high) {
    panic(ternary(denominator == 0, DIVISION_BY_ZERO, UNDER_OVERFLOW));
}
```

The result is an opaque EVM panic (`0x4e487b71` / code `0x12`) rather than a typed `PythErrors` revert, breaking any `try/catch` that catches named errors and making the failure invisible to callers that only check for `PythErrors.NegativeInputPrice`.

### Impact Explanation

Any on-chain protocol that calls `deriveCrossRate` with a Pyth price feed whose `.price` field is `0` (possible during feed initialization, circuit-breaker states, or stale-feed fallback paths) will receive an unrecoverable panic revert. Because the panic is not a typed error, integrators cannot distinguish it from other panics and cannot handle it gracefully. Critical operations such as liquidations, collateral valuations, or cross-rate-gated trades would be silently blocked for the duration that `price2 == 0`.

### Likelihood Explanation

Pyth price feeds can legitimately return `price == 0` in edge cases (e.g., a feed that has never been updated, a feed in a halted state, or a feed whose confidence interval spans zero). The official SDK documentation and example code do not warn callers to pre-check for zero prices before calling `deriveCrossRate`. Any integrator following the documented pattern is exposed.

### Recommendation

Add an explicit zero-price guard immediately after the negativity check:

```solidity
if (price1 < 0 || price2 < 0) {
    revert PythErrors.NegativeInputPrice();
}
if (price2 == 0) {
    revert PythErrors.ZeroInputPrice(); // new error, or reuse NegativeInputPrice
}
```

This mirrors the fix recommended in the Knox report: guard the edge-case value before it reaches the arithmetic primitive.

### Proof of Concept

```solidity
// price2 = 0 passes the negativity check (0 is not < 0)
// deltaExponent = int64(0 - (0 + 0)) = 0  → else branch
// Math.mulDiv(price1, 1, 10**0 * uint64(0))
//           = Math.mulDiv(price1, 1, 0)   → PANIC 0x12
PythUtils.deriveCrossRate(
    100,   // price1
    0,     // expo1
    0,     // price2  ← zero, passes guard, causes panic
    0,     // expo2
    0      // targetExponent
);
