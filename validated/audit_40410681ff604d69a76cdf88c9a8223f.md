### Title
Protocol-Triggered `ban_session` Fails to Disconnect Co-Located Sessions from the Same IP — (`File: network/src/network.rs`)

### Summary

`NetworkState::ban_session`, called by every protocol handler when a peer misbehaves, bans the offending IP in the peer store but only disconnects the single triggering session. The RPC-triggered `NetworkController::ban` path correctly calls `disconnect_peers_in_ip_range` to evict all currently-connected peers sharing the banned IP, but `ban_session` omits this step entirely. A malicious peer that pre-establishes multiple inbound connections from the same IP (using distinct peer IDs) can have one session banned while its remaining sessions stay alive and continue to interact with the node.

### Finding Description

**Two ban code paths with inconsistent behavior:**

**Path 1 — RPC-triggered ban** (`NetworkController::ban`, `network/src/network.rs:1422-1428`):
```
ban(address, ban_until, ban_reason)
  → disconnect_peers_in_ip_range(address, &ban_reason)   // evicts ALL sessions in IP range
  → peer_store.ban_network(address, ban_until, ban_reason) // adds to ban list
```

**Path 2 — Protocol-triggered ban** (`NetworkState::ban_session`, `network/src/network.rs:241-281`):
```
ban_session(session_id, duration, reason)
  → peer_store.ban_addr(&peer.connected_addr, ...)        // adds IP to ban list
  → disconnect_with_message(p2p_control, peer.session_id) // evicts ONLY the one session
  // ← disconnect_peers_in_ip_range is NEVER called
```

`ban_session` is the path invoked by every protocol handler on misbehavior:
- `IdentifyCallback::received_identify` — wrong network [1](#0-0) 
- `HolePunching::received` — malformed message [2](#0-1) 
- `ServiceError::ProtocolError` handler — protocol panic [3](#0-2) 
- `report_session` — score-based ban (same single-session disconnect) [4](#0-3) 

`accept_peer` correctly rejects **new** connections from a banned IP: [5](#0-4) 

But it has no per-IP session limit, so multiple inbound sessions from the same IP with distinct peer IDs are all accepted before any ban occurs. Once `ban_session` fires on one of them, the ban list blocks future reconnects but the already-accepted sibling sessions are never evicted.

The RPC path explicitly handles this with `disconnect_peers_in_ip_range`: [6](#0-5) [7](#0-6) 

`ban_session` has no equivalent call: [8](#0-7) 

### Impact Explanation

A malicious peer that pre-establishes N inbound connections from the same IP (different peer IDs, different source ports) can trigger a ban on one session and keep the remaining N-1 sessions alive indefinitely. Through those surviving sessions it can:

- Continue relaying invalid compact blocks or transactions, forcing the victim node to perform repeated script/PoW verification work.
- Maintain persistent sync-protocol state (e.g., in-flight block requests) that the node cannot attribute to a banned entity.
- Exhaust inbound connection slots, preventing legitimate peers from connecting, because the surviving sessions still occupy slots counted against `max_inbound`.

**Impact: Medium** — persistent ban bypass enabling continued resource consumption and protocol abuse from a nominally-banned IP.

### Likelihood Explanation

- No privileged access required; any unprivileged peer can open multiple inbound connections to a CKB node.
- The `PeerRegistry` enforces no per-IP connection limit; only total inbound count and peer-ID uniqueness are checked. [9](#0-8) 
- The attacker only needs to connect with N distinct peer IDs from the same IP before sending a single ban-triggering message on one of them.
- The inconsistency is structural and reproducible on every protocol-triggered ban.

**Likelihood: Medium.**

### Recommendation

In `NetworkState::ban_session`, after adding the IP to the ban list, iterate over the peer registry and disconnect all remaining sessions whose `connected_addr` resolves to the same IP, mirroring the logic already present in `disconnect_peers_in_ip_range`:

```rust
// After peer_store.lock().ban_addr(...)
// Disconnect all other sessions from the same IP
if let Some(ip) = multiaddr_to_socketaddr(&peer.connected_addr).map(|s| s.ip()) {
    let sibling_sessions: Vec<SessionId> = self.with_peer_registry(|reg| {
        reg.peers()
            .iter()
            .filter_map(|(sid, p)| {
                multiaddr_to_socketaddr(&p.connected_addr)
                    .filter(|s| s.ip() == ip)
                    .map(|_| *sid)
            })
            .collect()
    });
    for sid in sibling_sessions {
        let _ = disconnect_with_message(p2p_control, sid, "banned (sibling session)");
    }
}
```

The same fix should be applied to `report_session` when `report_result.is_banned()` is true. [10](#0-9) 

### Proof of Concept

1. Attacker controls a host at IP `A.B.C.D`.
2. Attacker generates N distinct peer IDs (e.g., N = 5).
3. Attacker opens N inbound TCP connections to the victim CKB node, each with a different peer ID. All are accepted because `accept_peer` only checks total inbound count and peer-ID uniqueness. [11](#0-10) 
4. On session 1, attacker sends an Identify message with a wrong network magic. `received_identify` calls `ban_session(session_1, BAN_ON_NOT_SAME_NET, ...)`. [1](#0-0) 
5. `ban_session` removes session 1 from the registry, adds `A.B.C.D/32` to the ban list, and disconnects session 1. Sessions 2–5 remain in the peer registry and are fully operational. [8](#0-7) 
6. Victim node's `accept_peer` now rejects any new connection from `A.B.C.D`, but sessions 2–5 are never checked against the ban list and continue to exchange sync/relay messages.
7. Attacker uses sessions 2–5 to relay malformed compact blocks or flood the tx-pool, consuming victim CPU/memory while the operator believes the IP is banned.

    /// disconnect peers with matched peer_ip or peer_ip_network, eg: 192.168.0.2 or 192.168.0.0/24
    fn disconnect_peers_in_ip_range(&self, address: IpNetwork, reason: &str) {
        self.network_state.with_peer_registry(|reg| {
            reg.peers().iter().for_each(|(peer_index, peer)| {
                if let Some(addr) = multiaddr_to_socketaddr(&peer.connected_addr)
                    && address.contains(addr.ip())
                {
                    let _ = disconnect_with_message(
                        &self.p2p_control,
                        *peer_index,
                        &format!("Ban peer {}, reason: {}", addr.ip(), reason),
                    );
                }
            })
        });
    }
