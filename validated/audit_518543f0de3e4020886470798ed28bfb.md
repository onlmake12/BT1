### Title
Unconditional `snapshot.get(OutPoint)` Before Filter Evaluation in `IndexerHandle::get_cells` Enables Disk I/O Amplification DoS — (`util/indexer/src/service.rs`)

---

### Summary

In `IndexerHandle::get_cells`, for every key matching the script prefix, a random-access RocksDB point lookup (`snapshot.get(Key::OutPoint(...))`) is performed **unconditionally before any filter check**. An unprivileged RPC caller can craft a request with a broad script prefix (matching many cells) and an impossible `output_data` filter, causing O(N) random I/O operations per request where N is the number of prefix-matching entries. The only bound is the per-request timeout (default 10 seconds). Repeated requests sustain high disk I/O, degrading the entire node.

---

### Finding Description

In `util/indexer/src/service.rs`, `IndexerHandle::get_cells` constructs a `TimeoutIterator` over a RocksDB snapshot and iterates all keys matching the script prefix: [1](#0-0) 

The critical ordering is:

1. **Line 251**: `take_while(|(key, _)| key.starts_with(&prefix))` — iterates all N prefix-matching entries.
2. **Lines 270–275**: `snapshot.get(Key::OutPoint(&out_point).into_vec())` — **unconditional random I/O lookup** for every entry.
3. **Lines 277–355**: Filter checks (`output_data`, `script_prefix`, capacity range, etc.) — applied **after** the I/O.

The `output_data` exact-match filter at lines 327–330 returns `None` if the data doesn't match, but the `snapshot.get()` has already been issued: [2](#0-1) 

The `TimeoutIterator` checks elapsed time only at the start of each `next()` call, not inside the `filter_map` closure where `snapshot.get()` executes: [3](#0-2) 

The default timeout is 10 seconds with no per-scan-entry cap: [4](#0-3) 

The `request_limit` only bounds the number of **results returned**, not the number of entries **scanned**: [5](#0-4) 

There is no per-IP or per-connection rate limiting on the indexer RPC layer. The `IndexerRpcImpl` passes calls directly to `IndexerHandle::get_cells` with no throttling: [6](#0-5) 

---

### Impact Explanation

The indexer store defaults to `data_dir/indexer/store`, co-located on the same physical disk as the main chain DB (`data_dir/db`). [7](#0-6) 

With N = 100K cells under a common prefix (realistic on mainnet for any widely-used lock script like secp256k1), an attacker can drive ~100K `snapshot.get()` random I/O calls per 10-second window. Concurrent or back-to-back requests sustain this indefinitely, saturating disk I/O bandwidth and starving the main chain RocksDB of I/O, degrading block processing and network relay.

---

### Likelihood Explanation

- The indexer RPC is publicly accessible with no authentication.
- Mainnet has millions of cells under the secp256k1 lock `code_hash` prefix — a natural broad prefix.
- The attack requires only a single crafted JSON-RPC call repeated in a loop.
- No special privileges, keys, or hashpower are required.

---

### Recommendation

Move the `snapshot.get(OutPoint)` call to **after** all filter checks that can be evaluated from the index key/value alone (e.g., `script_len_range`, `block_range` are encoded in the key). Only perform the point lookup when the key-level filters pass. Additionally, add a per-request scan-entry cap (separate from `request_limit` on results) and consider RPC-level rate limiting per connection.

---

### Proof of Concept

1. Index 100K cells all sharing the same lock script `code_hash` (broad prefix).
2. Call `get_cells` with:
   - `script`: that `code_hash` with empty `args` (prefix match hits all 100K cells)
   - `filter.output_data`: 32 bytes of random data no cell has
   - `filter.output_data_filter_mode`: `"exact"`
   - `limit`: 1
3. Instrument `snapshot.get()` calls — observe ~100K calls per request before timeout fires at 10 seconds.
4. Repeat in a loop; measure disk I/O utilization and main chain block processing latency.
