Audit Report

## Title
`remove_entry_and_descendants()` Pre-Removes Links Before `update_ancestors_index_key()` Runs, Leaving Ancestor Entries With Stale Descendant Weights and Eviction Keys - (File: tx-pool/src/component/pool_map.rs)

## Summary

`PoolMap::remove_entry_and_descendants()` strips all link entries for the entire subtree in a first pass, then calls `remove_entry()` for each node. Because `remove_entry()` relies on `self.links.calc_ancestors()` to find which remaining entries need their descendant weights decremented, and those links are already gone, any in-pool ancestor of the root entry is never updated. Its `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` fields remain inflated, and its `evict_key` is never refreshed, causing incorrect eviction ordering for the lifetime of that ancestor entry.

## Finding Description

`remove_entry_and_descendants` builds `removed_ids = [root] + calc_descendants(root)`, then calls `remove_entry_links` for every id in that set before calling `remove_entry` on any of them: [1](#0-0) 

`remove_entry_links(B)` removes B's own link entry entirely (`self.links.remove(id)`) and also removes B from A's children set: [2](#0-1) 

When `remove_entry(B)` is subsequently called, `update_ancestors_index_key` tries to resolve B's ancestors via `self.links.calc_ancestors(&B.proposal_short_id())`: [3](#0-2) 

`calc_ancestors` calls `calc_relative_ids`, which does `self.inner.get(short_id)` on B's ID. Since B's link entry was already removed, this returns `None`, and `unwrap_or_default()` yields an empty set: [4](#0-3) 

The ancestor loop is therefore empty. `sub_descendant_weight` is never called on A, so A's `descendants_count`, `descendants_size`, `descendants_cycles`, and `descendants_fee` remain at their pre-removal values, and `e.evict_key = e.inner.as_evict_key()` is never executed for A: [5](#0-4) 

The `EvictKey` is built directly from these stale descendant fields: [6](#0-5) 

`next_evict_entry` iterates the pool ordered by `evict_key` ascending (lowest fee-rate first), so A's inflated key causes it to rank as more valuable than it actually is and be evicted later than it should: [7](#0-6) 

The comment on line 256 ("so that we won't update_descendants_index_key in remove_entry") documents the intentional design for the descendants side, but it inadvertently also suppresses the ancestor-side update. The existing `test_remove_entry_and_descendants` test only asserts that B and C are absent and that A's descendants set is empty; it does not assert A's `descendants_*` fields or `evict_key`, so the stale state goes undetected: [8](#0-7) 

## Impact Explanation

After `remove_entry_and_descendants(B)` in a chain A → B → C, entry A retains inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count`. Its `evict_key.fee_rate` is computed as `max(descendants_feerate, own_feerate)` using these stale values, making A appear more valuable than it truly is. The pool's eviction logic (`limit_size` → `next_evict_entry`) will rank A as higher-value, causing genuinely low-fee transactions to persist in the pool longer than they should. The stale state is permanent for A until A itself is removed or the pool is cleared. This maps to: **Medium (2001–10000 points) — Suboptimal implementation of CKB state storage mechanism.**

## Likelihood Explanation

Any code path that calls `remove_entry_and_descendants` on a transaction that has an in-pool ancestor triggers this. Reachable paths include `resolve_conflict`, `resolve_conflict_header_dep`, `check_and_record_ancestors`, and `limit_size`. An unprivileged user can trigger this by submitting A (low fee) → B (high fee) → C (high fee), then submitting a conflicting transaction that causes B to be double-spent, triggering `remove_entry_and_descendants(B)`. No special privileges are required. The condition (ancestor still in pool when a descendant subtree is removed) is common in normal pool operation.

## Recommendation

Before removing any links in `remove_entry_and_descendants`, update the ancestors of the root entry while links are still intact. Specifically, call `update_ancestors_index_key(root_entry, EntryOp::Remove)` for the root (and only the root, since its accumulated `descendants_*` fields already represent the entire subtree weight) before the link-removal loop. Alternatively, accumulate the total size/cycles/fee of all entries in `removed_ids` and apply a single `sub_descendant_weight`-equivalent update to each ancestor of the root after computing them while links are intact.

## Proof of Concept

Add the following unit test to `tx-pool/src/component/tests/score_key.rs` following the pattern of `test_remove_entry_and_descendants`:

1. Add `tx1` (low fee, e.g. 100 shannons/100 size) to the pool as a proposed entry.
2. Add `tx2` (high fee, 200/200) spending `tx1`'s output.
3. Add `tx3` (high fee, 200/200) spending `tx2`'s output.
4. Assert `tx1.descendants_count == 3` and `tx1.descendants_fee == 500 shannons`.
5. Call `pool_map.remove_entry_and_descendants(&tx2_id)`.
6. Assert `tx1` is still in the pool.
7. Assert `tx1.descendants_count` — it will still be `3`, not `1`.
8. Assert `tx1.evict_key` reflects the inflated `descendants_feerate`.
9. Compare with calling `pool_map.remove_entry(&tx2_id)` directly (with `tx3` already removed): `tx1.descendants_count` will correctly be `1` and `tx1.evict_key` will reflect only its own fee rate.

    pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
        let mut removed_ids = vec![id.to_owned()];
        removed_ids.extend(self.calc_descendants(id));

        // update links state for remove, so that we won't update_descendants_index_key in remove_entry
        for id in &removed_ids {
            self.remove_entry_links(id);
        }

        removed_ids
            .iter()
            .filter_map(|id| self.remove_entry(id))
            .collect()
    }

    fn remove_entry_links(&mut self, id: &ProposalShortId) {
        if let Some(parents) = self.links.get_parents(id).cloned() {
            for parent in parents {
                self.links.remove_child(&parent, id);
            }
        }
        if let Some(children) = self.links.get_children(id).cloned() {
            for child in children {
                self.links.remove_parent(&child, id);
            }
        }
        self.links.remove(id);
    }

    fn update_ancestors_index_key(&mut self, child: &TxEntry, op: EntryOp) {
        let ancestors: HashSet<ProposalShortId> =
            self.links.calc_ancestors(&child.proposal_short_id());
        for anc_id in &ancestors {
            // update parent score
            self.entries.modify_by_id(anc_id, |e| {
                match op {
                    EntryOp::Remove => e.inner.sub_descendant_weight(child),
                    EntryOp::Add => e.inner.add_descendant_weight(child),
                };
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

fn test_remove_entry_and_descendants() {
