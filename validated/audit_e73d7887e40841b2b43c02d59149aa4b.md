### Title
Tx-Pool Uses Truncated 10-Byte `ProposalShortId` as Unique Entry Key, Enabling Targeted Transaction DoS via Hash Collision - (File: `tx-pool/src/util.rs`)

---

### Summary

The CKB transaction pool (`tx-pool`) uses `ProposalShortId` — a **10-byte (80-bit) truncation** of the transaction hash — as the sole unique key for pool entries. The admission gate `check_txid_collision` rejects any incoming transaction whose `ProposalShortId` matches one already in the pool, even if the full 32-byte transaction hashes differ. An unprivileged attacker who can find a valid transaction with the same `ProposalShortId` as a victim's pending transaction (a birthday attack requiring ~2^40 hash evaluations) can pre-occupy that short-ID slot and permanently block the victim's transaction from entering the pool.

---

### Finding Description

`ProposalShortId` is defined as the first 10 bytes of the transaction hash: [1](#0-0) 

The `PoolEntry` struct in `pool_map.rs` declares `ProposalShortId` as a `hashed_unique` index — the **only** unique key for pool membership: [2](#0-1) 

`add_entry` silently drops any transaction whose `ProposalShortId` is already occupied: [3](#0-2) 

Before script verification even runs, `pre_check` calls `check_txid_collision`, which rejects the incoming transaction with `Reject::Duplicated` if any pool entry shares its `ProposalShortId`: [4](#0-3) [5](#0-4) 

The comment on line 280 says *"Same txid means exactly the same transaction"*, but the check on line 282 uses `proposal_short_id()` (10 bytes), **not** the full `tx_hash` (32 bytes). Two structurally distinct transactions with different full hashes can share the same `ProposalShortId` if their hashes collide in the first 80 bits.

---

### Impact Explanation

An attacker who pre-occupies a `ProposalShortId` slot causes every subsequent submission of a transaction sharing that short-ID to be rejected with `PoolRejectedDuplicatedTransaction` (`-1107`). The victim's transaction is permanently blocked from the pool until the attacker's occupying transaction is mined or evicted. This is a targeted, per-user Denial of Service on transaction submission — directly analogous to the MesonSwap `encodedSwap` front-running DoS.

---

### Likelihood Explanation

`ProposalShortId` is 80 bits. A birthday collision requires approximately **2^40 hash evaluations** (~10^12 operations). On commodity hardware performing ~10^9 SHA2 hashes/second, this takes roughly **~17 minutes** on a single machine, or seconds with GPU/ASIC parallelism. The attacker needs only one valid transaction (spending their own cells) whose hash collides in the first 10 bytes with the victim's. No privileged access, leaked keys, or majority hashpower is required — only an RPC/P2P-reachable node and modest computation.

---

### Recommendation

- **Short term:** Change `check_txid_collision` and `PoolEntry`'s unique index to use the full 32-byte `tx_hash` (`Byte32`) instead of `ProposalShortId`. The `ProposalShortId` is a protocol-level concept for the two-phase proposal/commit window and should not double as the pool's uniqueness key.
- **Long term:** Audit all pool lookup paths (`remove_tx`, `set_entry`, `remove_entry`, `get_by_id`) to ensure they operate on the full transaction hash where identity — not proposal membership — is the concern.

---

### Proof of Concept

The existing test suite already demonstrates the structural precondition: [6](#0-5) 

`tx1` and `tx2` share the same `tx_hash` / `proposal_short_id` but differ only in witnesses. The test confirms `tx2` is rejected as `PoolRejectedDuplicatedTransaction`. The same rejection path fires for any two transactions sharing a `ProposalShortId`, regardless of whether they are "cousin" transactions or entirely unrelated transactions from different users that happen to collide in 80 bits.

**Attack steps:**
1. Observe victim's pending transaction `T_victim` with `ProposalShortId` `P` (visible via mempool or P2P relay).
2. Enumerate valid transactions spending attacker-controlled cells, varying output amounts or lock args, until one hashes to a `tx_hash` whose first 10 bytes equal `P` (~2^40 work).
3. Submit the colliding transaction `T_attacker` via `send_transaction` RPC or P2P relay.
4. `T_attacker` is accepted into the pool (valid scripts, unique short-ID at submission time).
5. Every subsequent submission of `T_victim` hits `check_txid_collision` at `tx-pool/src/util.rs:22` and is rejected with `Reject::Duplicated`, permanently blocking the victim.

pub(crate) fn check_txid_collision(tx_pool: &TxPool, tx: &TransactionView) -> Result<(), Reject> {
    let short_id = tx.proposal_short_id();
    if tx_pool.contains_proposal_id(&short_id) {
        return Err(Reject::Duplicated(tx.hash()));
    }
    Ok(())
}

    pub(crate) async fn pre_check(
        &self,
        tx: &TransactionView,
    ) -> (Result<PreCheckedTx, Reject>, Arc<Snapshot>) {
        // Acquire read lock for cheap check
        let tx_size = tx.data().serialized_size_in_block();

        let (ret, snapshot) = self
            .with_tx_pool_read_lock(|tx_pool, snapshot| {
                let tip_hash = snapshot.tip_hash();

                // Same txid means exactly the same transaction, including inputs, outputs, witnesses, etc.
                // It's also not possible for RBF, reject it directly
                check_txid_collision(tx_pool, tx)?;

                // Try normal path first, if double-spending check success we don't need RBF check
                // this make sure RBF won't introduce extra performance cost for hot path
                let res = resolve_tx(tx_pool, &snapshot, tx.clone(), false);
                match res {
                    Ok((rtx, status)) => {
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(Reject::Resolve(OutPointError::Dead(out))) => {
                        let (rtx, status) = resolve_tx(tx_pool, &snapshot, tx.clone(), true)?;
                        let fee = check_tx_fee(tx_pool, &snapshot, &rtx, tx_size)?;
                        let conflicts = tx_pool.pool_map.find_conflict_outpoint(tx);
                        if conflicts.is_none() {
                            // this mean one input's outpoint is dead, but there is no direct conflicted tx in tx_pool
                            // we should reject it directly and don't need to put it into conflicts pool
                            error!(
                                "{} is resolved as Dead, but there is no conflicted tx",
                                rtx.transaction.proposal_short_id()
                            );
                            return Err(Reject::Resolve(OutPointError::Dead(out)));
                        }
                        // we also return Ok here, so that the entry will be continue to be verified before submit
                        // we only want to put it into conflicts pool after the verification stage passed
                        // then we will double-check conflicts txs in `submit_entry`

                        Ok((tip_hash, rtx, status, fee, tx_size))
                    }
                    Err(err) => Err(err),
                }
            })
            .await;
        (ret, snapshot)
    }
