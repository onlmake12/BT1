### Title
Out-of-Bounds Vec Index Panic in `remove_unconfirmed_tx` When `tx_age > MAX_CONFIRM_BLOCKS` — (`util/fee-estimator/src/estimator/confirmation_fraction.rs`)

### Summary

`TxConfirmStat::remove_unconfirmed_tx` correctly guards the `block_unconfirmed_txs` access for `tx_age >= MAX_CONFIRM_BLOCKS`, but unconditionally indexes `confirm_blocks_to_failed_txs[tx_age - 1]` without the same guard. When a tracked tx is evicted after more than 1000 blocks in the pool, `tx_age - 1 >= 1000` exceeds the Vec's valid range (0..999), causing a Rust index-out-of-bounds panic that crashes the tx-pool service.

### Finding Description

`confirm_blocks_to_failed_txs` and `block_unconfirmed_txs` are both allocated with length `MAX_CONFIRM_BLOCKS` = 1000: [1](#0-0) 

In `remove_unconfirmed_tx`, the guard at line 208 protects only the `block_unconfirmed_txs` access: [2](#0-1) 

But the `count_failure` branch at line 214–216 is **outside** that guard and uses the same unbounded `tx_age`: [3](#0-2) 

When `tx_age = 1001`, `tx_age - 1 = 1000` is out of bounds for a Vec of length 1000. Rust's indexing panics unconditionally on OOB access.

### Impact Explanation

The panic propagates through the tx-pool reject callback registered in `shared_builder.rs`: [4](#0-3) 

`reject_tx` calls `drop_tx` with `count_failure = true`: [5](#0-4) 

A panic inside the tx-pool write-lock callback crashes the tx-pool service task, making the node non-functional and the `estimate_fee_rate` RPC permanently unavailable until restart.

### Likelihood Explanation

**Preconditions:**
1. Node configured with `algorithm = ConfirmationFraction` — a supported, documented production option.
2. A valid tx with fee above `min_fee_rate` enters the pool and is tracked at height H.
3. The tx is never confirmed and stays in the pool for >1000 blocks (~2.2 hours at 8 s/block). The default `expiry_hours = 12` means a tx can easily survive 1000+ blocks before time-based expiry triggers eviction.
4. Eviction fires `reject_tx` → `drop_tx_inner(..., true)` → `remove_unconfirmed_tx(..., count_failure=true)`.

An unprivileged attacker can submit a valid low-fee tx via the standard RPC (`send_transaction`) and simply wait. No hashpower, no privileged access, no social engineering required.

### Recommendation

Add a bounds check before the `count_failure` branch, mirroring the existing guard:

```rust
if count_failure && tx_age <= self.confirm_blocks_to_failed_txs.len() {
    self.confirm_blocks_to_failed_txs[tx_age - 1][bucket_index] += 1f64;
}
```

### Proof of Concept

```rust
#[test]
fn test_remove_unconfirmed_tx_oob_panic() {
    let buckets = vec![FeeRate::from_u64(1000), FeeRate::from_u64(2000)];
    let max_confirm_blocks = 1000;
    let decay = 0.993f64;
    let mut stat = TxConfirmStat::new(buckets, max_confirm_blocks, decay);

    // Simulate a tx tracked at height 0, evicted at height 1001 (tx_age = 1001 > 1000)
    let entry_height: u64 = 0;
    let tip_height: u64 = 1001;
    let bucket_index: usize = 0;

    // Manually set old_unconfirmed_txs to avoid underflow on the other branch
    stat.bucket_stats[bucket_index].old_unconfirmed_txs = 1;

    // This panics: confirm_blocks_to_failed_txs[1000] is out of bounds (len=1000)
    stat.remove_unconfirmed_tx(entry_height, tip_height, bucket_index, true);
}
```
