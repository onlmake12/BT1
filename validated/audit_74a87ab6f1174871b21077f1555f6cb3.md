### Title
`ScriptHashTypeVerifier` Only Checks Lock Script Hash Type, Skipping Type Script — (`File: verification/src/transaction_verifier.rs`)

### Summary
`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only against a transaction output's **lock script**, completely omitting the same check for the output's **type script**. A transaction sender can embed a type script with a non-permitted `ScriptHashType` (e.g., `Data3 = 6`, `Data4 = 8`, …) and bypass the consensus-level gate, reaching script execution with an unsupported VM version.

### Finding Description

`ScriptHashTypeVerifier::verify()` iterates over every output and validates only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  L796-L814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(
                    TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                );
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            })
            .into());
        }
    }
    Ok(())
}
``` [1](#0-0) 

There is no corresponding call to `output.type_().to_opt()` and no check of the type script's `hash_type` field against `ENABLED_SCRIPT_HASH_TYPE`. [2](#0-1) 

The permitted set is:

```rust
// util/constant/src/consensus.rs
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // ScriptHashType::Data
    1u8, // ScriptHashType::Type
    2u8, // ScriptHashType::Data1
    4u8, // ScriptHashType::Data2
};
``` [3](#0-2) 

`ScriptHashType` is defined with a `seq!` macro that generates `Data3 = 6`, `Data4 = 8`, … `Data127 = 254` — all structurally valid byte values (even, non-zero) that pass `ScriptHashType::verify_value()` and therefore also pass `check_data()`, but are **not** in `ENABLED_SCRIPT_HASH_TYPE`. [4](#0-3) 

The `check_data` path on `CellOutputReader` validates both lock and type scripts for structural validity, but it only calls `verify_value` (even-or-one), not the allowlist check: [5](#0-4) 

```rust
impl<'r> packed::CellOutputReader<'r> {
    fn check_data(&self) -> bool {
        self.lock().check_data() && self.type_().check_data()
    }
}
```

So a type script with `hash_type = 6` (`Data3`) is structurally valid, passes `check_data`, and passes `ScriptHashTypeVerifier` — the only place that enforces the allowlist — because `ScriptHashTypeVerifier` never inspects the type script field.

`ScriptHashTypeVerifier` is composed into `NonContextualTransactionVerifier`, which is the first verification stage applied to every submitted transaction: [6](#0-5) 

### Impact Explanation

A transaction sender (unprivileged RPC caller or P2P relay peer) can craft a transaction whose output carries a type script with `hash_type = 6` (or any other even value ≥ 6 not in the allowlist). This transaction passes `NonContextualTransactionVerifier` and is forwarded to `ContextualTransactionVerifier` / `TransactionScriptsVerifier`. The script verifier only knows three `ScriptVersion` variants (V0, V1, V2); encountering an unrecognised hash type at execution time produces undefined/unspecified behaviour rather than the clean `ScriptHashTypeNotPermitted` consensus rejection that the verifier was designed to emit. [7](#0-6) 

The asymmetry means:
- Lock script with `hash_type = 6` → rejected at `NonContextualTransactionVerifier` with `ScriptHashTypeNotPermitted`.
- Type script with `hash_type = 6` → passes `NonContextualTransactionVerifier`, reaches script execution.

This is a direct analog to the reported issue: one variant (lock script) is checked; the other valid variant (type script) is silently skipped.

### Likelihood Explanation

Any unprivileged user who can submit a transaction via RPC (`send_transaction`) or relay it over P2P can trigger this path. No special privilege, key, or majority hashpower is required. Constructing a transaction with a crafted type script `hash_type` byte is trivial with any CKB SDK.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also check the type script hash type for every output, mirroring the existing lock-script check:

```rust
// After the existing lock-script check, add:
if let Some(type_script) = output.type_().to_opt() {
    if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(type_script.hash_type()) {
        let val: u8 = hash_type.into();
        if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
            return Err(
                TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
            );
        }
    } else {
        return Err((TransactionError::InvalidScriptHashType {
            hash_type: type_script.hash_type(),
        })
        .into());
    }
}
```

This mirrors the pattern already used in `check_data`, which validates both `lock` and `type_` fields of a `CellOutput`. [5](#0-4) 

### Proof of Concept

1. Build a transaction output with a valid lock script (`hash_type = 0`, Data) and a type script with `hash_type = 6` (Data3, not in `ENABLED_SCRIPT_HASH_TYPE`).
2. Submit via `send_transaction` RPC or relay over P2P.
3. `NonContextualTransactionVerifier::verify()` calls `ScriptHashTypeVerifier::verify()`, which only inspects `output.lock().hash_type()` → returns `Ok(())`.
4. The transaction proceeds to `ContextualTransactionVerifier` / `TransactionScriptsVerifier` with a type script carrying an unsupported hash type, bypassing the intended consensus gate. [8](#0-7)

seq!(N in 3..=127 {
    /// Specifies how the script `code_hash` is used to match the script code and how to run the code.
    /// The hash type is split into the high 7 bits and the low 1 bit,
    /// when the low 1 bit is 1, it indicates the type,
    /// when the low 1 bit is 0, it indicates the data,
    /// and then it relies on the high 7 bits to indicate
    /// that the data actually corresponds to the version.
     #[derive(Default, Clone, Copy, PartialEq, Eq, Debug, Hash, FromRepr)]
     #[repr(u8)]
    pub enum ScriptHashType {
        /// Type "type" matches script code via cell type script hash.
        Type = 1,
        /// Type "data" matches script code via cell data hash, and run the script code in v0 CKB VM.
        #[default]
        Data = 0,
        /// Type "data1" matches script code via cell data hash, and run the script code in v1 CKB VM.
        Data1 = 2,
        /// Type "data2" matches script code via cell data hash, and run the script code in v2 CKB VM.
        Data2 = 4,
        #(
            #[doc = concat!("Type \"data", stringify!(N), "\" matches script code via cell data hash, and runs the script code in v", stringify!(N), " CKB VM.")]
            Data~N = N << 1,
        )*
    }
});

impl ScriptHashType {
    /// when the low 1 bit is 1, it indicates the type
    /// when the low 1 bit is 0, it indicates the data
    #[inline]
    pub fn verify_value(v: u8) -> bool {
        v.is_multiple_of(2) || v == 1
    }
}
