## Title
`ScriptHashTypeVerifier::verify` Omits Type-Script `hash_type` Validation, Allowing Not-Permitted Hash Types to Bypass Non-Contextual Checks — (File: verification/src/transaction_verifier.rs)

## Summary
`ScriptHashTypeVerifier::verify()` validates `hash_type` only on each output's lock script, never on the type script. A transaction output carrying `type_script.hash_type = 6` (`Data3`) passes the entire `NonContextualTransactionVerifier` pipeline unchallenged and is only rejected inside `select_version` during contextual verification — after cell resolution and script-group construction have already been performed. The asymmetry is confirmed by the existing test suite, which covers the lock-script case but has no counterpart for type scripts.

## Finding Description
`ScriptHashTypeVerifier::verify()` at lines 796–814 of `verification/src/transaction_verifier.rs` iterates over outputs and checks only `output.lock().hash_type()` against `ENABLED_SCRIPT_HASH_TYPE = {0, 1, 2, 4}` (`util/constant/src/consensus.rs` lines 7–11). The `output.type_().to_opt()` branch is never reached. [1](#0-0) 

`ScriptHashType::Data3 = 6` is generated by `seq!(N in 3..=127 { Data~N = N << 1 })` in `util/gen-types/src/core.rs` lines 9–32, making it a structurally valid enum variant. [2](#0-1) 

`ScriptHashType::verify_value(6)` returns `true` because `6.is_multiple_of(2)` (line 40), so the earlier `check_data` gate at `util/gen-types/src/extension/check_data.rs` lines 24–27 — which does inspect both lock and type scripts — passes it through, since `check_data` only tests structural validity, not membership in `ENABLED_SCRIPT_HASH_TYPE`. [3](#0-2) [4](#0-3) 

The downstream catch is `select_version` in `script/src/types.rs` lines 930–935, which returns `ScriptError::InvalidScriptHashType` for any `hash_type` not in `{Data, Data1, Data2, Type}`. However, this function is only reached after cell resolution and script-group construction have already been performed. [5](#0-4) 

The existing test at lines 100–122 of `verification/src/tests/transaction_verifier.rs` confirms the lock-script path correctly returns `ScriptHashTypeNotPermitted`, but no symmetric test exists for the type-script path. [6](#0-5) 

## Impact Explanation
Every CKB full node runs `NonContextualTransactionVerifier` as a cheap first-pass filter. By submitting transactions whose outputs carry a type script with `hash_type ∈ {6, 8, 10, …}`, an attacker forces each receiving node to pass the cheap gate, resolve all input/output cells (database reads), build script groups, and only then reject via `select_version`. The attacker pays only broadcast cost; each node pays contextual resolution cost before discarding the work. This is a resource-amplification vector matching the **High** impact: "Vulnerabilities or bad designs which could cause CKB network congestion with few costs."

## Likelihood Explanation
The entry path is fully open. Any peer can submit a transaction via the `send_transaction` RPC or relay it over the P2P network — no key, stake, or special role is required. Crafting such a transaction is trivial: set `type_script.hash_type = 6` on any output with a structurally valid lock script. The attack is repeatable with no victim mistake required.

## Recommendation
Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` for every output that carries one:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        check_hash_type(output.lock().hash_type())?;
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}

fn check_hash_type(raw: packed::Byte) -> Result<(), Error> {
    match TryInto::<ScriptHashType>::try_into(raw) {
        Ok(ht) => {
            let val: u8 = ht.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
            Ok(())
        }
        Err(_) => Err(TransactionError::InvalidScriptHashType { hash_type: raw }.into()),
    }
}
```

Also add a corresponding test alongside `test_not_enabled_hash_type_output_lock` and update the `NonContextualTransactionVerifier` doc comment to reflect that both lock and type script hash types are checked.
