### Title
`ScriptHashTypeVerifier` Omits Allowlist Check for Output Type Scripts — (`File: verification/src/transaction_verifier.rs`)

### Summary
`ScriptHashTypeVerifier::verify()` enforces the `ENABLED_SCRIPT_HASH_TYPE` allowlist only on output **lock** scripts. It never checks the `hash_type` of output **type** scripts. An unprivileged transaction sender can craft a transaction whose output cell carries a type script with a future or unapproved `hash_type` (e.g., `Data3 = 6`, `Data4 = 8`, …) and it will pass the cheap non-contextual gate, proceeding into the expensive contextual/script-execution stage before being rejected there.

### Finding Description

`ScriptHashTypeVerifier::verify()` iterates over every output and validates only `output.lock().hash_type()` against `ENABLED_SCRIPT_HASH_TYPE`:

```rust
// verification/src/transaction_verifier.rs  lines 796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) =
            TryInto::<ScriptHashType>::try_into(output.lock().hash_type())
        {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into());
            }
        } else {
            return Err((TransactionError::InvalidScriptHashType {
                hash_type: output.lock().hash_type(),
            }).into());
        }
        // ← output.type_().to_opt() is never inspected here
    }
    Ok(())
}
```

The allowlist itself is:

```rust
// util/constant/src/consensus.rs  lines 7-12
pub static ENABLED_SCRIPT_HASH_TYPE: Set<u8> = phf_set! {
    0u8, // Data
    1u8, // Type
    2u8, // Data1
    4u8, // Data2
};
```

Values `6` (`Data3`), `8` (`Data4`), … are not in the set. The verifier's own doc comment acknowledges the gap — it says only "Check whether output **lock** hash type within enabled range." [1](#0-0) [2](#0-1) [3](#0-2) 

The lower-level structural check (`check_data` in `util/gen-types/src/extension/check_data.rs`) does inspect both lock and type scripts, but it only verifies that the raw byte is a structurally valid `ScriptHashType` value (i.e., even or equals 1). It does **not** check membership in `ENABLED_SCRIPT_HASH_TYPE`. [4](#0-3) 

The contextual script verifier (`select_version` in `script/src/types.rs`) does eventually reject unapproved hash types, but only after the transaction has already been admitted through the non-contextual gate and queued for expensive execution. [5](#0-4) 

### Impact Explanation

Any transaction sender (no privilege required) can submit a transaction whose output type script carries `hash_type = Data3` (or any other future value). The transaction passes `NonContextualTransactionVerifier` — the cheap, early gate — and is enqueued for contextual verification. Contextual verification runs the CKB-VM script scheduler, which is significantly more expensive. The transaction is ultimately rejected by `select_version`, but only after consuming CPU and memory in the verification pipeline. Repeated submission of such transactions is a low-cost, high-impact resource-exhaustion path against the tx-pool worker. [6](#0-5) 

### Likelihood Explanation

The entry point is the public `send_transaction` RPC (and the P2P relay path). No key, no stake, and no special role is required. The attacker only needs to construct a syntactically valid transaction with a non-zero fee and an output whose type script has `hash_type = 6`. The structural `check_data` pass accepts it, `ScriptHashTypeVerifier` ignores it, and the transaction enters the expensive verification queue.

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the `hash_type` of each output's optional type script against `ENABLED_SCRIPT_HASH_TYPE`, mirroring the existing lock-script check:

```rust
for output in self.transaction.outputs() {
    // existing lock check …
    check_hash_type(output.lock().hash_type())?;

    // add type-script check
    if let Some(type_script) = output.type_().to_opt() {
        check_hash_type(type_script.hash_type())?;
    }
}
```

Update the doc comment on `NonContextualTransactionVerifier` to reflect that **both** lock and type script hash types are validated.

### Proof of Concept

The existing test suite already demonstrates the lock-script path is enforced:

```rust
// verification/src/tests/transaction_verifier.rs  lines 100-122
pub fn test_not_enabled_hash_type_output_lock() {
    let transaction = TransactionBuilder::default()
        .output(CellOutput::new_builder()
            .lock(Script::default().as_builder()
                .hash_type(ScriptHashType::Data3).build())
            .build())
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);
    assert_error_eq!(verifier.verify().unwrap_err(),
        TransactionError::ScriptHashTypeNotPermitted { hash_type: ScriptHashType::Data3.into() });
}
``` [7](#0-6) 

An analogous test for the type script would be:

```rust
pub fn test_not_enabled_hash_type_output_type() {
    let transaction = TransactionBuilder::default()
        .output(CellOutput::new_builder()
            .lock(Script::default())          // valid lock
            .type_(Some(Script::default().as_builder()
                .hash_type(ScriptHashType::Data3).build()))  // unapproved type
            .build())
        .build();
    let verifier = ScriptHashTypeVerifier::new(&transaction);
    // Currently returns Ok(()), should return ScriptHashTypeNotPermitted
    assert!(verifier.verify().is_err());
}
```

This test would **pass today** (i.e., `verify()` returns `Ok(())`) because the type script hash_type is never checked, confirming the missing allowlist enforcement. [8](#0-7) [9](#0-8)

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {

                }

                return Err((TransactionError::InvalidScriptHashType {
