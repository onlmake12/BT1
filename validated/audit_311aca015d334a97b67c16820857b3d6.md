### Title
`break` on Unindexed Input Silently Skips Subsequent `spend_cell` Calls, Leaving Indexed Outputs Marked Unspent ŌĆö (`util/rich-indexer/src/indexer/mod.rs`)

### Summary

In `insert_transaction`, when iterating over a transaction's inputs, a `break` is issued whenever `spend_cell` returns `false` (i.e., the referenced output is absent from the indexer). This exits the entire loop, so all inputs after the first unindexed one are never processed. Their corresponding indexed outputs are never marked `is_spent = 1`, causing `get_cells` to return already-spent cells as live UTXOs.

### Finding Description

`spend_cell` issues an SQL `UPDATE output SET is_spent = 1 WHERE ...` and returns `Ok(updated_rows > 0)`. [1](#0-0) 

It returns `Ok(false)` when the referenced output is not present in the indexer ŌĆö which is the normal, expected case when `cell_filter` is active or `init_tip_hash` skips early blocks.

Back in `insert_transaction`, the caller reacts to that `false` with `break`:

```rust
if !spend_cell(&out_point, tx).await? {
    break;   // ŌåÉ exits the entire inputs loop
}
``` [2](#0-1) 

`break` terminates the `for (input_index, input) in tx_view.inputs()...` loop entirely. Every input after the first unindexed one is silently skipped: `spend_cell` is never called for them, so their outputs remain with `is_spent = 0` in the database.

The correct fix is `continue` (skip this one input, keep processing the rest).

### Impact Explanation

Any node running with `cell_filter` or `init_tip_hash` configured ŌĆö both are explicitly documented production features ŌĆö is affected. [3](#0-2) 

After a block containing such a transaction is appended, `get_cells` queries the `output` table filtered by `is_spent = 0` and returns the stale cell as a live UTXO. Wallet software or dApps that rely on the indexer for UTXO selection will see phantom spendable cells, leading to failed transactions or incorrect balance displays.

### Likelihood Explanation

The trigger condition is straightforward: a valid on-chain transaction whose first input spends any cell absent from the indexer (pre-`init_tip_hash` output, or output filtered by `cell_filter`) and whose second (or later) input spends an indexed cell. No special privileges are required; any transaction submitter can craft this. The `cell_filter` and `init_tip_hash` features are common in production deployments to reduce index size.

### Recommendation

Replace `break` with `continue` at line 232:

```rust
if !spend_cell(&out_point, tx).await? {
    continue;  // cell not indexed; skip this input, process the rest
}
``` [4](#0-3) 

### Proof of Concept

1. Configure the node with `cell_filter` that excludes cells with no type script.
2. Index block 0 containing `tx00` (output with no type script ŌĆö filtered out) and `tx01` (output with a type script ŌĆö indexed).
3. Submit a block containing `tx10` with two inputs: `input[0]` = `OutPoint(tx00.hash, 0)` (not in indexer) and `input[1]` = `OutPoint(tx01.hash, 0)` (indexed).
4. Call `append(block)`.
5. Query `SELECT is_spent FROM output WHERE tx_id = (SELECT id FROM ckb_transaction WHERE tx_hash = tx01.hash) AND output_index = 0`.
6. Observe `is_spent = 0` ŌĆö the cell is still reported as live.
7. Call `get_cells` with `tx01`'s lock/type script; the spent cell is returned as a live UTXO.
