### Title
Deviation Calculation in `Scheduler._validateShouldUpdatePrices` Ignores Price Exponent, Causing Unit Mismatch When Feed Exponent Changes - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

`Scheduler.sol`'s deviation check compares raw `int64` price mantissa values without normalizing for the Pyth price exponent (`expo`). Pyth prices are expressed as `mantissa × 10^expo`. When a price feed's exponent changes between the stored previous update and a new incoming update, the raw mantissa comparison is performed across incompatible units, producing a wildly incorrect deviation value — either triggering spurious updates or suppressing legitimate ones.

---

### Finding Description

In `_validateShouldUpdatePrices`, the deviation between the current and previous price is computed as:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;   // raw mantissa, unit: 10^currentExpo
int64 previousPrice = previousFeed.price.price;     // raw mantissa, unit: 10^previousExpo

uint256 numerator   = SignedMath.abs(currentPrice - previousPrice);
uint256 denominator = SignedMath.abs(previousPrice);
uint256 deviationBps = Math.mulDiv(numerator, 10_000, denominator);
``` [1](#0-0) 

The `expo` field stored in `previousFeed.price.expo` and `priceFeeds[i].price.expo` is never read or compared. The formula is only correct when both mantissas share the same exponent. If `currentExpo ≠ previousExpo`, the subtraction `currentPrice - previousPrice` is dimensionally invalid — it subtracts values with different implicit scale factors.

The previous price feed (including its exponent) is stored in `_state.priceUpdates[subscriptionId][priceFeeds[i].id]` after each successful update. [2](#0-1) 

Pyth price feeds encode prices as `PythStructs.PriceFeed` with a `price.expo` field (typically `-8` for USD pairs, but not guaranteed to be fixed forever). When Pyth changes the exponent for a feed, the next relayed update carries a new exponent while the stored previous feed retains the old one. [3](#0-2) 

---

### Impact Explanation

**Scenario — exponent changes from `-8` to `-7` for a feed:**

| | Previous stored | New incoming |
|---|---|---|
| `price.price` | `100_000_000` | `10_000_000` |
| `price.expo` | `-8` | `-7` |
| Actual USD price | `$1.00` | `$1.00` (unchanged) |

Raw deviation computed by Scheduler:
```
numerator   = |10_000_000 - 100_000_000| = 90_000_000
denominator = 100_000_000
deviationBps = 90_000_000 * 10_000 / 100_000_000 = 9_000 bps (90%)
```

A subscription with `deviationThresholdBps = 100` (1%) would fire a spurious update even though the actual price did not move at all. Conversely, a real 5% price move accompanied by an exponent change in the opposite direction could suppress a legitimate update.

**Concrete impacts:**
1. **Spurious updates**: Subscription balance is drained paying for unnecessary price pushes; downstream protocols receive "updates" that carry no new information.
2. **Missed updates**: A genuine price deviation is masked by an exponent change in the opposite direction; downstream protocols read stale prices.

---

### Likelihood Explanation

Pyth has historically changed exponents for price feeds (e.g., during feed migrations or precision adjustments). The pusher (relayer) is an unprivileged role that simply forwards authentic, Wormhole-signed Pyth price data — it cannot forge the exponent, but it faithfully relays whatever exponent Pyth publishes. No privileged access is required to trigger this path; any relayer submitting a legitimate update for a feed whose exponent has changed will exercise the bug.

---

### Recommendation

Before computing the deviation, normalize both prices to a common exponent:

```solidity
int64 currentPrice  = priceFeeds[i].price.price;
int32 currentExpo   = priceFeeds[i].price.expo;
int64 previousPrice = previousFeed.price.price;
int32 previousExpo  = previousFeed.price.expo;

// Normalize previousPrice to currentExpo before comparison
if (currentExpo != previousExpo) {
    int32 expoDiff = currentExpo - previousExpo;
    // scale previousPrice up or down accordingly
    // (use safe arithmetic; revert on overflow)
}
```

Alternatively, reject updates where the exponent has changed and require a heartbeat-triggered update to re-baseline the stored price, or simply store the exponent alongside the mantissa and always normalize before subtraction.

---

### Proof of Concept

1. Create a Scheduler subscription with `updateOnDeviation = true`, `deviationThresholdBps = 50` (0.5%).
2. Submit a first price update for feed F with `price = 100_000_000`, `expo = -8` (actual price: $1.00). This is stored as `previousFeed`.
3. Pyth changes the exponent for feed F to `-7`. Submit a second update with `price = 10_000_000`, `expo = -7` (actual price: still $1.00 — zero real deviation).
4. The Scheduler computes `deviationBps = |10_000_000 - 100_000_000| * 10_000 / 100_000_000 = 9_000 bps`.
5. `9_000 >= 50` → update is accepted and subscription balance is charged, despite zero actual price movement. [4](#0-3)
