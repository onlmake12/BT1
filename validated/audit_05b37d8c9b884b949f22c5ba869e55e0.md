Audit Report

## Title
Ancestors' `descendants_*` fields not decremented after `remove_entry_and_descendants` clears links before `remove_entry` — (`tx-pool/src/component/pool_map.rs`)

## Summary

`remove_entry_and_descendants` clears all parent/child links for every transaction in the removed subtree before calling `remove_entry` on each. Inside `remove_entry`, `update_ancestors_index_key` resolves the ancestor set via `self.links.calc_ancestors(...)`, but since the links are already gone, it returns an empty set and `sub_descendant_weight` is never called on surviving ancestors. Every surviving ancestor of the removed subtree permanently carries inflated `descendants_fee`, `descendants_size`, `descendants_cycles`, and `descendants_count` values, corrupting the `EvictKey` used for pool eviction ordering.

## Finding Description

`remove_entry_and_descendants` operates in two phases:

**Phase 1 — clear all links:** [1](#0-0) 

`remove_entry_links` removes the target from its parents' children sets, removes it from its children's parents sets, and deletes its own entry from `links.inner`: [2](#0-1) 

**Phase 2 — call `remove_entry` for each removed id:** [3](#0-2) 

Inside `remove_entry`, `update_ancestors_index_key` is called with `EntryOp::Remove`. It resolves ancestors via `self.links.calc_ancestors(...)`: [4](#0-3) 

`calc_ancestors` calls `calc_relative_ids`, which looks up `self.inner.get(short_id)`. Since phase 1 already called `self.links.remove(id)` for every removed id, the lookup returns `None` and the ancestor set is empty: [5](#0-4) 

The `sub_descendant_weight` call is therefore never reached for any surviving ancestor: [6](#0-5) 

The developer comment at line 256 acknowledges the intent ("so that we won't update_descendants_index_key in remove_entry"), but the same link-clearing also silently suppresses the necessary ancestor-side decrement. By contrast, when a single tx is removed via `remove_entry` directly (links still intact), `calc_ancestors` returns the correct set and the decrement is applied correctly.

## Impact Explanation

The `descendants_*` fields feed directly into `EvictKey`: [7](#0-6) 

`next_evict_entry` iterates by `evict_key` in ascending order to select the transaction to drop when the pool is full: [8](#0-7) 

`limit_size` calls `next_evict_entry` and then `remove_entry_and_descendants` on the result: [9](#0-8) 

A surviving ancestor with inflated `descendants_fee` has an overstated `EvictKey.fee_rate`, making it appear more valuable than it truly is. It is systematically skipped during eviction, and legitimate higher-fee-rate transactions are evicted in its place. An attacker can exploit this to make low-fee transactions immune to eviction, filling the pool with low-fee transactions and preventing legitimate high-fee transactions from entering — matching the allowed impact: **High: Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

The `score` key (used for mining priority) is based only on `ancestors_*` fields and is unaffected. [10](#0-9) 

## Likelihood Explanation

`remove_entry_and_descendants` is called from multiple ordinary, unprivileged code paths: `resolve_conflict` (on every committed block), `limit_size` (after every pool insertion), `remove_by_detached_proposal` (during every reorg), and `remove_tx`. [11](#0-10) 

An unprivileged attacker can deliberately trigger the bug:
1. Submit low-fee Tx A (parent).
2. Submit high-fee Tx B (child of A) and Tx C (child of B).
3. Submit a conflicting replacement for Tx B. `resolve_conflict` calls `remove_entry_and_descendants(B)`, removing B and C.
4. Tx A's `descendants_*` fields remain inflated with B's and C's fee/size/cycles.
5. Tx A is now effectively immune to eviction despite its low individual fee rate.

This is repeatable with low cost and requires no special privileges.

## Recommendation

Before clearing links in `remove_entry_and_descendants`, snapshot the ancestors of the root transaction and the full weight of the subtree being removed, then apply the decrement to those ancestors after the removal:

```rust
pub(crate) fn remove_entry_and_descendants(&mut self, id: &ProposalShortId) -> Vec<TxEntry> {
    let mut removed_ids = vec![id.to_owned()];
    removed_ids.extend(self.calc_descendants(id));

    // Capture ancestors of the root BEFORE links are cleared
    let root_ancestors = self.links.calc_ancestors(id);

    for id in &removed_ids {
        self.remove_entry_links(id);
    }

    let removed: Vec<TxEntry> = removed_ids
        .iter()
        .filter_map(|id| self.remove_entry(id))
        .collect();

    // Decrement each surviving ancestor's descendants_* fields
    for removed_entry in &removed {
        for anc_id in &root_ancestors {
            self.entries.modify_by_id(anc_id, |e| {
                e.inner.sub_descendant_weight(removed_entry);
                e.evict_key = e.inner.as_evict_key();
            });
        }
    }

    removed
}
```

## Proof of Concept

Using the existing test infrastructure in `tx-pool/src/component/tests/score_key.rs` as a model:

1. Create `tx1` (low fee, no pool parents).
2. Create `tx2` spending `tx1`'s output (high fee).
3. Create `tx3` spending `tx2`'s output (high fee).
4. Add all three to the pool via `add_proposed`.
5. Assert `tx1.descendants_count == 3`, `tx1.descendants_fee == fee1 + fee2 + fee3`.
6. Call `pool.remove_entry_and_descendants(&tx2.proposal_short_id())`.
7. Assert `tx1.descendants_count == 1`, `tx1.descendants_fee == fee1`.

**Actual result (bug):** Step 7 fails — `tx1.descendants_count` remains 3 and `tx1.descendants_fee` remains `fee1 + fee2 + fee3`, because `update_ancestors_index_key` finds no ancestors after links are cleared. [12](#0-11) [4](#0-3)

    pub(crate) fn limit_size(
        &mut self,
        callbacks: &Callbacks,
        current_entry_id: Option<&ProposalShortId>,
    ) -> Option<Reject> {
        let mut ret = None;
        while self.pool_map.total_tx_size > self.config.max_tx_pool_size {
            let next_evict_entry = || {
                self.pool_map
                    .next_evict_entry(Status::Pending)
                    .or_else(|| self.pool_map.next_evict_entry(Status::Gap))
                    .or_else(|| self.pool_map.next_evict_entry(Status::Proposed))
            };

            if let Some(id) = next_evict_entry() {
                let removed = self.pool_map.remove_entry_and_descendants(&id);
                for entry in removed {
                    let tx_hash = entry.transaction().hash();
                    debug!(
                        "Removed by size limit {} timestamp({})",
                        tx_hash, entry.timestamp
                    );
                    let reject = Reject::Full(format!(
                        "the fee_rate for this transaction is: {}",
                        entry.fee_rate()
                    ));
                    if let Some(short_id) = current_entry_id
                        && entry.proposal_short_id() == *short_id
                    {
                        ret = Some(reject.clone());
                    }
                    callbacks.call_reject(self, &entry, reject);
                }
            }
        }
        self.pool_map.entries.shrink_to_fit();
        ret
    }
