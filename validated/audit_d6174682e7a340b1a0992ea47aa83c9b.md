### Title
Pyth Price Feed Used Without Staleness Check in `latestAnswer()` and `latestRoundData()` - (File: `target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

### Summary

`PythAggregatorV3`, Pyth's official Chainlink-compatible adapter contract distributed as part of `@pythnetwork/pyth-sdk-solidity`, exposes `latestAnswer()`, `latestRoundData()`, and `getRoundData()` — all of which call `pyth.getPriceUnsafe(priceId)` with no staleness check on the returned `publishTime`. Any protocol that integrates this adapter as a drop-in Chainlink replacement will silently consume prices from arbitrarily far in the past.

### Finding Description

`PythAggregatorV3` is Pyth's official SDK contract for migrating Chainlink-dependent protocols to Pyth price feeds. It implements the Chainlink `AggregatorV3Interface` and is explicitly recommended in Pyth's documentation as the migration path.

Every price-returning function in the contract calls `pyth.getPriceUnsafe(priceId)`:

```solidity
function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    return int256(price.price);
}

function latestRoundData() external view returns (...) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    roundId = uint80(price.publishTime);
    return (roundId, int256(price.price), price.publishTime, price.publishTime, roundId);
}
```

`getPriceUnsafe()` is explicitly documented in `IPyth.sol` as returning "the most recent price update in this contract without any recency checks" and that "the returned price update may be arbitrarily far in the past." No staleness check against `block.timestamp` is performed anywhere in `PythAggregatorV3`.

The `updatedAt` field returned by `latestRoundData()` is set to `price.publishTime`, which may be hours or days old. Downstream protocols that check `updatedAt` against `block.timestamp` will receive a stale timestamp but no revert from the adapter itself. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation

Any protocol that deploys `PythAggregatorV3` as a Chainlink-compatible price feed and calls `latestAnswer()` or `latestRoundData()` will receive stale prices without any revert. This is the exact vulnerability class described in the external report.

Concrete impact scenarios:
- **Lending protocols**: Stale collateral prices allow under-collateralized borrowing or block legitimate liquidations.
- **Derivatives/perps**: Stale mark prices allow traders to open/close positions at favorable historical prices.
- **AMMs**: Stale prices allow arbitrageurs to drain liquidity at incorrect rates.

The `PythAggregatorV3` contract is actively promoted in Pyth's official documentation as the migration path from Chainlink, making widespread deployment likely. [4](#0-3) 

### Likelihood Explanation

- `PythAggregatorV3` is part of the official `@pythnetwork/pyth-sdk-solidity` npm package (v4.3.1) and is the recommended migration path from Chainlink.
- Protocols migrating from Chainlink expect `latestRoundData()` to revert on stale data (as Chainlink does via heartbeat checks). `PythAggregatorV3` never reverts on staleness.
- Pyth is a pull oracle — if no keeper submits a price update, the cached price ages indefinitely. An attacker can deliberately withhold updates or simply wait for a network disruption.
- The `getPriceUnsafe` call is the root cause entirely within Pyth's own SDK code. [5](#0-4) [6](#0-5) 

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan` using a configurable staleness threshold, or add an explicit staleness check after each `getPriceUnsafe` call:

```solidity
uint256 public constant MAX_PRICE_AGE = 3600; // configurable

function latestAnswer() public view virtual returns (int256) {
    PythStructs.Price memory price = pyth.getPriceUnsafe(priceId);
    require(
        block.timestamp - price.publishTime <= MAX_PRICE_AGE,
        "Stale Pyth price"
    );
    return int256(price.price);
}
```

Alternatively, use `pyth.getPriceNoOlderThan(priceId, MAX_PRICE_AGE)` directly, which reverts with `PythErrors.StalePrice()` if the price is too old. [7](#0-6) 

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing to a live Pyth contract and a price feed ID.
2. Do not call `updateFeeds()` for an extended period (e.g., 1 hour).
3. Call `latestAnswer()` — it returns the price from 1 hour ago with no revert.
4. Call `latestRoundData()` — it returns `updatedAt = price.publishTime` (1 hour ago) with no revert.
5. Any downstream protocol that calls `latestRoundData()` and uses the returned `answer` without independently checking `updatedAt` against `block.timestamp` will use the stale price.
6. An attacker can exploit this in a lending protocol: if the true price of collateral has dropped 20% but the stale price is still high, the attacker borrows against the inflated stale collateral value. [8](#0-7) [9](#0-8)

        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();

        return price;
    }
