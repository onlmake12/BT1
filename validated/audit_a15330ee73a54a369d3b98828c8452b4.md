### Title
Unbounded `feeInWei` in Entropy Provider Registration Causes Arithmetic Overflow DOS in `getFeeV2` — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

Any address can permissionlessly register as an Entropy randomness provider and set `feeInWei` to `type(uint128).max`. Because `getFeeV2` adds the provider fee to the protocol fee (`_state.pythFeeInWei`) in a single unchecked `uint128 + uint128` expression under Solidity 0.8+ checked arithmetic, the addition overflows and reverts. This makes it impossible for any user to call `request`, `requestWithCallback`, or `requestV2` against that provider, permanently DOS-ing all randomness requests to it.

---

### Finding Description

**Root cause — no upper bound on `feeInWei`:**

`register()` stores the caller-supplied fee with no validation:

```solidity
// Entropy.sol line 129
provider.feeInWei = feeInWei;
``` [1](#0-0) 

`setProviderFee()` likewise applies no upper bound:

```solidity
// Entropy.sol line 819
provider.feeInWei = newFeeInWei;
``` [2](#0-1) 

**Overflow site — `getFeeV2`:**

```solidity
// Entropy.sol line 764
return getProviderFee(provider, gasLimit) + _state.pythFeeInWei;
``` [3](#0-2) 

Both operands are `uint128`. Under Solidity ≥ 0.8 checked arithmetic, if `provider.feeInWei == type(uint128).max` and `pythFeeInWei > 0` (as it is in every production deployment), the addition reverts unconditionally.

**DOS propagation — `requestHelper`:**

Every user-facing entry point (`request`, `requestWithCallback`, `requestV2`) calls `requestHelper`, which calls `getFeeV2` on the first line of fee accounting:

```solidity
// Entropy.sol line 234
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
``` [4](#0-3) 

Because `getFeeV2` reverts, every randomness request to the affected provider reverts before any state is written.

---

### Impact Explanation

Any user who has integrated with the malicious provider — or any user who calls the no-argument `requestV2()` shorthand that routes to `getDefaultProvider()` if the default provider is the attacker — will have every randomness request permanently revert. Existing in-flight requests (already paid) are unaffected, but no new requests can be submitted. The provider's `feeInWei` can be set to `type(uint128).max` atomically in a single transaction, giving users no window to react. There is no admin function to forcibly reset a provider's fee.

---

### Likelihood Explanation

Registration is fully permissionless — any EOA can call `register()` with `feeInWei = type(uint128).max` at zero cost beyond gas. The attacker does not need any privileged role, leaked key, or governance majority. The attack is a single transaction. A griefing attacker or a provider who wants to exit the market while harming competitors' users can execute this at any time.

---

### Recommendation

Add an upper-bound check in both `register` and `setProviderFee` to ensure that `feeInWei + pythFeeInWei` cannot overflow `uint128`:

```solidity
require(
    feeInWei <= type(uint128).max - _state.pythFeeInWei,
    "feeInWei too large"
);
```

Alternatively, perform the addition in `getFeeV2` using a wider type (e.g., `uint256`) and revert explicitly if the result exceeds `type(uint128).max`, so the error is surfaced at read time rather than silently breaking all request paths.

---

### Proof of Concept

```solidity
// Attacker registers with max fee
vm.prank(attacker);
entropy.register(
    type(uint128).max,   // feeInWei — no upper bound check
    commitment,
    metadata,
    100,
    uri
);

// getFeeV2 now overflows and reverts for any gasLimit
// because type(uint128).max + pythFeeInWei > type(uint128).max
vm.expectRevert(); // arithmetic overflow
entropy.getFeeV2(attacker, 0);

// Consequently, every user request to this provider reverts
vm.deal(user, 1 ether);
vm.prank(user);
vm.expectRevert(); // overflow inside requestHelper → getFeeV2
entropy.request{value: 1 ether}(
    attacker,
    entropy.constructUserCommitment(bytes32(uint256(42))),
    false
);
```
