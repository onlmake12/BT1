### Title
Fixed `GAS_OVERHEAD` in Keeper Fee Calculation Enables Profit Extraction via Batched `updatePriceFeeds` Calls - (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `Scheduler` contract's keeper fee mechanism uses a fixed `GAS_OVERHEAD = 30000` constant to compensate for transaction overhead costs that fall outside the measured gas window. When a keeper batches multiple `updatePriceFeeds` calls across different subscriptions in a single transaction (via a smart contract), the 21k base transaction cost is paid only once, but `GAS_OVERHEAD` is added to every individual call. This allows a keeper to systematically extract more ETH from subscription balances than the actual gas cost incurred.

---

### Finding Description

In `_processFeesAndPayKeeper()`, the keeper reimbursement is calculated as:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
``` [1](#0-0) 

The `GAS_OVERHEAD` constant is defined as `30000` and is explicitly documented as "a rough estimate of the tx overhead for a keeper to call `updatePriceFeeds`": [2](#0-1) 

This overhead is intended to cover the ~21k base transaction gas cost plus calldata and other pre-execution costs. However, when a keeper deploys a smart contract that calls `updatePriceFeeds` in a loop across N subscriptions within a single transaction:

1. **The 21k base transaction cost is paid only once**, but `GAS_OVERHEAD` (30k) is charged to each subscription's balance — yielding a profit of ~21k gas per extra call.
2. **Storage slots become "warm"** after the first call (EIP-2929): subsequent SLOADs to the Scheduler's state, the Pyth contract, and the keeper's own address cost 100 gas instead of 2100 gas, saving ~2000 gas per warm slot per extra call.
3. **The ETH transfer target (`msg.sender`) is warm** after the first call, saving ~2500 gas on subsequent transfers.

The keeper is fully permissionless — no registration is required: [3](#0-2) 

The payment is sent directly to `msg.sender` at the end of each call: [4](#0-3) 

---

### Impact Explanation

Each subscription's `balanceInWei` is debited by `totalKeeperFee` which includes the full `GAS_OVERHEAD` per call: [5](#0-4) 

For a keeper batching N subscriptions in one transaction, the over-charge per extra call is approximately:
- `(GAS_OVERHEAD - actual_per_call_overhead) * tx.gasprice`
- ≈ `(30,000 - ~9,000) * tx.gasprice` = ~21,000 gas profit per extra call

At 100 gwei and ETH = $3,000: ~$6.30 profit per extra subscription call. With 50 active subscriptions batched, this is ~$315 per transaction, extracted from subscription holders' deposited funds. Subscription balances are drained faster than expected, potentially causing legitimate updates to fail with `InsufficientBalance` prematurely.

---

### Likelihood Explanation

The keeper role is fully permissionless — anyone can call `updatePriceFeeds`. A malicious keeper only needs to deploy a simple batching contract that loops over multiple subscription IDs and calls `updatePriceFeeds` for each. The Scheduler contract itself provides `getActiveSubscriptions` to enumerate all valid targets. The attack requires no special privileges, no leaked keys, and no governance access. As the number of active subscriptions grows, the profitability scales linearly. [6](#0-5) 

---

### Recommendation

1. **Remove the fixed `GAS_OVERHEAD` and instead measure gas from the very start of the transaction** using a wrapper or by capturing gas at the entry point before any calldata decoding. This eliminates the fixed assumption that is exploitable when batching.
2. **Alternatively, add a `require(msg.sender == tx.origin)` check** in `updatePriceFeeds` to prevent smart contracts from acting as keepers and batching calls. This directly prevents the batching attack vector.
3. **Or track the last block in which a given `msg.sender` was paid** and cap the number of reimbursements per block per address to 1, preventing the overhead from being collected multiple times in one transaction.

---

### Proof of Concept

A malicious keeper deploys the following contract:

```solidity
contract MaliciousKeeper {
    IScheduler scheduler;
    
    constructor(address _scheduler) {
        scheduler = IScheduler(_scheduler);
    }
    
    function batchUpdate(
        uint256[] calldata subscriptionIds,
        bytes[][] calldata updateDatas
    ) external {
        // 21k base tx cost paid ONCE here
        for (uint i = 0; i < subscriptionIds.length; i++) {
            // Each call charges GAS_OVERHEAD=30k to the subscription,
            // but actual per-call overhead is ~9k (warm slots, no base tx cost)
            // Profit: ~21k gas * tx.gasprice per extra call
            scheduler.updatePriceFeeds(subscriptionIds[i], updateDatas[i]);
        }
        // Total received: N * (actual_gas + 30k) * gasprice
        // Total spent:    actual_gas_total + 21k (base tx, paid once)
        // Net profit:     (N-1) * ~21k * gasprice
    }
}
```

The keeper calls `getActiveSubscriptions` to enumerate all valid subscription IDs, constructs valid `updateData` for each (fetched from Hermes), and calls `batchUpdate`. Each subscription's balance is charged `GAS_OVERHEAD * tx.gasprice` for overhead that was never actually incurred, with the surplus flowing to the keeper.
