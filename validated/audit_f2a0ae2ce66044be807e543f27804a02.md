### Title
DAO Withdrawal `header_dep_index` Parsed as Full `u64` in Rust but Only Lowest Byte in C DAO Script — Consensus Split Causing False Rejection of Valid Withdrawals (`util/dao/src/lib.rs`)

---

### Summary

`DaoCalculator::transaction_maximum_withdraw` in `util/dao/src/lib.rs` reads the `header_dep_index` from the witness `input_type` field as a full 8-byte little-endian `u64`, while the on-chain C DAO script reads only the **lowest byte** of that same 8-byte field. For any `input_type` value greater than 255, the Rust node and the C script resolve to different entries in `header_deps`. This creates a consensus split: a valid DAO withdrawal transaction (accepted by the C script) is rejected by the Rust node, permanently blocking the user from withdrawing their locked DAO funds through this node.

---

### Finding Description

In `util/dao/src/lib.rs`, `transaction_maximum_withdraw` extracts the deposit header by reading the full 8-byte `input_type` witness field as a `u64`:

```rust
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))
```

and then uses it directly as a `usize` index into `header_deps`:

```rust
rtx.transaction
    .header_deps()
    .get(header_dep_index as usize)
```

The on-chain C DAO script (`ckb-system-scripts/c/dao.c`) reads only the **lowest byte** of the same 8-byte field to obtain the index. For `input_type` values ≤ 255 both agree. For `input_type` values > 255 they diverge: the C script uses `input_type & 0xFF` while Rust uses the full value.

The test `check_dao_withdraw_header_dep_index_exceeds_u8` in `util/dao/src/tests.rs` explicitly documents this split with the comment:

> "Position 1: correct deposit block (what C VM resolves via lowest byte). Position 257: withdraw block (wrong — Rust resolves this with full u64)."

The test constructs a transaction with `input_type = 257` (lowest byte = 1), places the deposit block at `header_deps[1]`, and the withdraw block at `header_deps[257]`. The C script resolves index 1 → deposit block (correct). Rust resolves index 257 → withdraw block (wrong block number), and the block-number cross-check at line 105 fires, causing `DaoError::InvalidOutPoint`.

The block-number guard only catches the specific case where the resolved header's block number differs from `deposited_block_number`. It does **not** prevent the general false-rejection scenario: any transaction where the lowest byte of `input_type` correctly points to the deposit header but the full `u64` value points to a non-existent or wrong entry will be rejected by the Rust node even though the C script would accept it.

The `DaoCalculator` is invoked in the live transaction-verification pipeline via `FeeCalculator::transaction_fee` in `verification/src/transaction_verifier.rs`, and also in the tx-pool admission check in `tx-pool/src/util.rs`. A false rejection at either point prevents the transaction from entering the mempool or being committed.

---

### Impact Explanation

A DAO depositor who constructs a withdrawal transaction with `input_type > 255` — where the lowest byte correctly identifies the deposit header — will have their transaction rejected by the Rust node with `DaoError::InvalidOutPoint`, even though the on-chain C DAO script would accept it. Because the node is the gatekeeper for mempool admission and block verification, the user cannot withdraw their locked CKB through this node. The locked CKB (principal + accrued interest) is inaccessible via this node until the user either uses a patched node or reconstructs the transaction with `input_type ≤ 255`.

---

### Likelihood Explanation

A DAO withdrawal transaction with more than 256 `header_deps` is required to trigger this path. While uncommon in typical single-cell withdrawals, it is reachable for users withdrawing many DAO cells in a single transaction (each cell may require its own deposit-block header dep), or for any script author who deliberately or inadvertently encodes `input_type` as a value > 255. The protocol places no upper bound on the number of `header_deps` in a transaction. The discrepancy is a latent consensus split that becomes exploitable whenever the index exceeds one byte.

---

### Recommendation

The Rust `DaoCalculator` should match the C DAO script's behavior by masking the parsed index to its lowest byte before use:

```rust
// Current (wrong):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()))

// Fixed (matches C script):
Ok(LittleEndian::read_u64(&header_deps_index_data.unwrap()) & 0xFF)
```

Alternatively, add an explicit validation step that rejects any `input_type` value whose upper 7 bytes are non-zero, making the constraint explicit and consistent with the C script's effective behavior.

---

### Proof of Concept

The discrepancy is directly demonstrated by the existing test `check_dao_withdraw_header_dep_index_exceeds_u8`:

1. Build a DAO withdrawal transaction with 258 `header_deps`.
2. Place the deposit block at `header_deps[1]` and the withdraw block at `header_deps[257]`.
3. Set `input_type = 257` (little-endian u64; lowest byte = 1).
4. Set cell data to `deposit_block.number()` (100).

**C script path**: reads lowest byte → index 1 → deposit block (number 100) → matches cell data → **accepts**.

**Rust path**: reads full u64 → index 257 → withdraw block (number 200) → does not match cell data (100) → `DaoError::InvalidOutPoint` → **rejects**.

The Rust node rejects a transaction the C script would accept, blocking the user's DAO withdrawal. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

#[test]
fn check_dao_withdraw_header_dep_index_exceeds_u8() {
    let deposit_number = 100u64;
    let withdraw_number = 200u64;

    let (_tmp_dir, store, deposit_block, withdraw_block) =
        setup_store_with_headers(deposit_number, withdraw_number);

    let consensus = Consensus::default();
    let dao_type_script = Script::new_builder()
        .code_hash(consensus.dao_type_hash())
        .hash_type(ScriptHashType::Type)
        .build();

    // Pad header_deps to 258 entries so index 257 is valid.
    // Position 1: correct deposit block (what C VM resolves via lowest byte).
    // Position 257: withdraw block (wrong — Rust resolves this with full u64).
    let dummy = h256!("0x1").into();
    let mut header_deps = vec![dummy; 258];
    header_deps[1] = deposit_block.hash();
    header_deps[257] = withdraw_block.hash();

    let cell_data = Bytes::from(deposit_number.to_le_bytes().to_vec());
    let input_cell = CellOutput::new_builder()
        .capacity(capacity_bytes!(1000000))
        .type_(Some(dao_type_script).pack())
        .build();
    let tx_info = TransactionInfo::new(
        withdraw_block.number(),
        withdraw_block.epoch(),
        withdraw_block.hash(),
        0,
    );
    let cell_meta = CellMetaBuilder::from_cell_output(input_cell, cell_data)
        .transaction_info(tx_info)
        .build();

    // input_type = 257, lowest byte = 1
    let witness = WitnessArgs::new_builder()
        .input_type(Some(Bytes::from(257u64.to_le_bytes().to_vec())))
        .build();
    let witness_bytes: Bytes = witness.as_bytes();

    let tx = TransactionBuilder::default()
        .set_header_deps(header_deps)
        .witness(witness_bytes)
        .build();

    let rtx = ResolvedTransaction {
        transaction: tx,
        resolved_cell_deps: vec![],
        resolved_inputs: vec![cell_meta],
        resolved_dep_groups: vec![],
    };

    let data_loader = store.borrow_as_data_loader();
    let calculator = DaoCalculator::new(&consensus, &data_loader);
    let result = calculator.transaction_fee(&rtx);

    // Rust resolves index 257 → withdraw block (number 200), but cell data
    // says deposited at block 100. Block number check catches the mismatch.
    assert!(result.is_err(), "expected Err, got {result:?}");
}
