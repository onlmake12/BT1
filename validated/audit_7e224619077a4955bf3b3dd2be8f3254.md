### Title
Two-Phase Commit Verifier Matches Committed Transactions by Truncated 10-Byte `ProposalShortId` Instead of Full Transaction Hash, Allowing Proposal Substitution - (File: `verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

CKB's `TwoPhaseCommitVerifier` enforces the two-step-transaction-confirmation rule by checking that every committed transaction's `ProposalShortId` appears in the proposal window. However, `ProposalShortId` is only the **first 10 bytes (80 bits)** of the transaction hash. The verifier never checks the full 32-byte transaction hash. A miner who can craft two transactions sharing the same `ProposalShortId` (an 80-bit birthday collision, ~2^40 work) can commit transaction T2 — which was never proposed — by relying on a prior proposal of T1 that shares T2's short ID. The reward calculator has the same flaw: it attributes proposer rewards by short ID, so the proposer of T1 receives the reward for T2's commitment.

---

### Finding Description

`ProposalShortId` is defined as the first 10 bytes of the transaction hash:

```rust
// util/gen-types/src/extension/shortcut.rs
pub fn from_tx_hash(h: &packed::Byte32) -> Self {
    let mut inner = [0u8; 10];
    inner.copy_from_slice(&h.as_slice()[..10]);  // only first 10 bytes
    inner.into()
}
``` [1](#0-0) 

`TwoPhaseCommitVerifier::verify()` collects the `ProposalShortId`s of all committed transactions and checks that each is present in the proposal set. It never compares full transaction hashes:

```rust
// verification/contextual/src/contextual_block_verifier.rs
let committed_ids: HashSet<_> = self
    .block
    .transactions()
    .iter()
    .skip(1)
    .map(TransactionView::proposal_short_id)   // ← only 10-byte short ID
    .collect();

if committed_ids.difference(&proposal_txs_ids).next().is_some() {
    return Err((CommitError::Invalid).into());
}
``` [2](#0-1) 

The reward calculator has the same structural flaw. It derives `ProposalShortId` from committed transaction hashes and matches them against the proposal set by short ID alone:

```rust
// util/reward-calculator/src/lib.rs
let committed_idx_proc = |hash: &Byte32| -> Vec<ProposalShortId> {
    store
        .get_block_txs_hashes(hash)
        .into_iter()
        .skip(1)
        .map(|tx_hash| ProposalShortId::from_tx_hash(&tx_hash))  // ← short ID only
        .collect()
};
``` [3](#0-2) 

The reward is then attributed to the proposer of whichever short ID matches, regardless of whether the committed transaction is the same one that was proposed: [4](#0-3) 

---

### Impact Explanation

An attacker who is a miner (or who colludes with one) and can compute an 80-bit birthday collision can:

1. **Bypass the two-phase commit consensus rule**: Commit transaction T2 — which was never broadcast as a proposal — by relying on a prior proposal of T1 that shares T2's `ProposalShortId`. `TwoPhaseCommitVerifier` accepts T2 because its short ID is in the proposal set.

2. **Steal or misdirect proposer rewards**: The reward calculator attributes the 40% proposer reward to the miner who proposed T1, even though T2 (a different transaction) was committed. This is a direct financial misattribution.

3. **Compact block reconstruction failure**: Peers who have T1 in their mempool (not T2) will fail to reconstruct the committed block, causing network disruption and forced full-block downloads.

---

### Likelihood Explanation

`ProposalShortId` is 80 bits. A birthday collision requires approximately 2^40 hash evaluations. Modern GPU hardware can compute billions of Blake2b hashes per second; 2^40 ≈ 10^12 operations is achievable in hours on a GPU cluster. The attacker controls transaction content (witnesses, output amounts, version fields) to freely vary the hash. The attacker must also be a miner (or collude with one) to commit T2 in the proposal window. This is a meaningful barrier, but miners are the natural adversary in this threat model. The existing test suite already demonstrates that `ProposalShortId` collisions between two different transactions are possible and detectable: [5](#0-4) 

---

### Recommendation

The `TwoPhaseCommitVerifier` should verify that each committed transaction's **full 32-byte hash** was explicitly proposed, not just its truncated 10-byte `ProposalShortId`. Since the proposal zone only stores short IDs (by design, to save block space), the fix requires either:

1. Storing full transaction hashes in the proposal zone (a consensus-breaking change), or
2. Extending `ProposalShortId` to use more bytes of the hash (e.g., 20 bytes / 160 bits) to make birthday collisions computationally infeasible (~2^80 work), or
3. Documenting the 80-bit security bound explicitly and accepting it as a known design trade-off with a formal security analysis.

The reward calculator in `util/reward-calculator/src/lib.rs` has the same short-ID-only matching and should be reviewed under the same lens.

---

### Proof of Concept

```
1. Attacker generates a large set of valid transactions by varying witness data.
   Each transaction has a unique 32-byte hash but the attacker searches for
   two transactions T1 and T2 where:
     T1.hash()[0..10] == T2.hash()[0..10]   (ProposalShortId collision)
     T1.hash() != T2.hash()                 (different full transactions)

2. Attacker (as miner) mines block B_p containing T1's ProposalShortId
   in the proposal zone. T1 is broadcast to the mempool.

3. Within the proposal window [B_p + w_close, B_p + w_far], attacker
   mines block B_c containing T2 in the commitment zone (not T1).

4. TwoPhaseCommitVerifier on B_c:
     committed_ids = { T2.proposal_short_id() }  // == T1.proposal_short_id()
     proposal_txs_ids contains T1.proposal_short_id()
     committed_ids.difference(&proposal_txs_ids) == empty  → Ok(())
   → B_c is accepted as valid despite T2 never being proposed.

5. RewardCalculator attributes 40% of T2's fee to the proposer of T1
   (the attacker's own earlier block), misattributing the reward.

6. Peers with T1 in their mempool cannot reconstruct B_c via compact
   block relay, causing forced full-block downloads.
```

The existing test at `sync/src/relayer/tests/compact_block_process.rs:560` (`test_collision`) already constructs a `fake_tx` with the same `ProposalShortId` as a real transaction by zeroing the last 4 bytes of the hash, confirming the collision mechanism is reachable in the codebase. [6](#0-5)
