### Title
Duplicate Script IDs in `rollback_block` Cause O(N) Redundant SQL Queries and Potential Indexer Panic — (`util/rich-indexer/src/indexer/remove.rs`)

---

### Summary

`rollback_block` deletes outputs first, then iterates over every output to check whether its script still exists elsewhere. Because there is no deduplication, N outputs sharing the same `lock_script_id` cause `script_exists_in_output` to be called N times with the same argument (always returning false after the first call), and the same ID is appended to `script_id_list_to_remove` N times. The resulting `DELETE … WHERE id IN (…)` carries N duplicate parameters. On SQLite deployments with the older default `SQLITE_MAX_VARIABLE_NUMBER = 999`, a block with ≥ 1 000 outputs sharing one lock script causes the DELETE to fail. Because the sync loop calls `indexer.rollback().expect(…)`, that failure panics the indexer process.

---

### Finding Description

**Step 1 — outputs are deleted before the script-existence check** [1](#0-0) 

Line 25 removes every output row for the block's transactions. After this point the `output` table contains no rows for any script that was exclusive to this block.

**Step 2 — the loop calls `script_exists_in_output` once per output, with no deduplication** [2](#0-1) 

For a block with N outputs all carrying `lock_script_id = S`, the loop executes N iterations. Each iteration calls `script_exists_in_output(S, tx)`, which fires two `SELECT EXISTS` queries against the now-empty `output` table and returns `false` every time. `S` is pushed into `script_id_list_to_remove` on every iteration, producing N duplicate entries.

**Step 3 — `script_exists_in_output` issues 2 SQL queries per call** [3](#0-2) 

Each invocation unconditionally executes a `SELECT EXISTS … WHERE lock_script_id = $1` query; if that returns false it also executes `SELECT EXISTS … WHERE type_script_id = $1`. For N = 500 outputs sharing one lock script this is up to 1 000 round-trips to the database for work that one query could do.

**Step 4 — `remove_batch_by_blobs` builds a single unbatched DELETE with all N parameters** [4](#0-3) 

There is no deduplication and no chunking. `sqlx_param_placeholders(1..ids.len())` generates one placeholder per entry, so `script_id_list_to_remove` with 1 000 duplicate entries produces `DELETE FROM script WHERE id IN ($1, …, $1000)`. SQLite's default `SQLITE_MAX_VARIABLE_NUMBER` is 999 (pre-3.32.0 builds); this query fails with "too many SQL variables."

**Step 5 — rollback failure panics the indexer** [5](#0-4) 

The sync loop calls `indexer.rollback().expect("rollback block should be OK")`. Any `Err` from the DB layer is an unconditional panic, terminating the indexer process.

---

### Impact Explanation

| Scenario | Effect |
|---|---|
| Any block with N outputs sharing a lock script | O(N) redundant SQL queries during rollback; measurable latency spike |
| SQLite < 3.32.0, N ≥ 1 000 outputs sharing a lock script | `remove_batch_by_blobs` returns `Err`; `.expect()` panics the indexer |
| SQLite ≥ 3.32.0 (limit 32 766) | No crash (CKB block-size cap limits outputs to ~9 000); performance degradation only |

The rich-indexer is a production component of the CKB node used to serve cell/transaction queries. A panic crashes the indexer service; it must be restarted manually and may require re-indexing.

---

### Likelihood Explanation

An attacker must mine one valid PoW block containing ≥ 1 000 outputs all using the same lock script, have it briefly accepted by the network, and then have it rolled back during a natural reorg. This does not require majority hashpower — a single valid block suffices. The block is structurally valid (CKB consensus does not restrict output script reuse), so no consensus guard prevents it. The entry point is standard P2P block relay / reorg state, which is explicitly in scope.

---

### Recommendation

1. **Deduplicate before the loop** — collect unique script IDs from `output_lock_type_list` into a `HashSet` before calling `script_exists_in_output`, so each distinct script is checked at most once.
2. **Deduplicate `script_id_list_to_remove`** — or use a `HashSet` directly and convert to `Vec` only for the final DELETE.
3. **Chunk the DELETE** — split `script_id_list_to_remove` into batches of ≤ 999 (or a configurable limit) inside `remove_batch_by_blobs` to be safe across all SQLite versions.

async fn remove_batch_by_blobs(
    table_name: &str,
    column_name: &str,
    ids: &[i64],
    tx: &mut Transaction<'_, Any>,
) -> Result<(), Error> {
    if ids.is_empty() {
        return Ok(());
    }

    // build query str
    let mut query_builder = SqlBuilder::delete_from(table_name);
    let sql = query_builder
        .and_where_in(column_name, &sqlx_param_placeholders(1..ids.len())?)
        .sql()
        .map_err(|err| Error::DB(err.to_string()))?;

    // bind
    let mut query: sqlx::query::Query<'_, Any, sqlx::any::AnyArguments<'_>> = sqlx::query(&sql);
    for hash in ids {
        query = query.bind(hash);
    }

    // execute
    query
        .execute(tx.as_mut())
        .await
        .map_err(|err| Error::DB(err.to_string()))?;

    Ok(())
}

async fn script_exists_in_output(
    script_id: i64,
    tx: &mut Transaction<'_, Any>,
) -> Result<bool, Error> {
    let row_lock = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE lock_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => {
            if r {
                return Ok(true);
            }
        }
        Err(_) => {
            // sqlite type is BIGINT
            if row_lock.get::<i64, _>(0) == 1 {
                return Ok(true);
            }
        }
    }

    let row_type = sqlx::query(
        r#"
        SELECT EXISTS (
            SELECT 1
            FROM output
            WHERE type_script_id = $1
        )
        "#,
    )
    .bind(script_id)
    .fetch_one(tx.as_mut())
    .await
    .map_err(|err| Error::DB(err.to_string()))?;

    // pg type is BOOLEAN
    match row_lock.try_get::<bool, _>(0) {
        Ok(r) => Ok(r),
        // sqlite type is BIGINT
        Err(_) => Ok(row_type.get::<i64, _>(0) == 1),
    }
}
