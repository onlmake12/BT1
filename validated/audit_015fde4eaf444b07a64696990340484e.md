### Title
Missing Zero-Address Check for `_topAuthority` in `initialize()` Permanently Bricks PythLazer Contract - (File: `lazer/contracts/evm/src/PythLazer.sol`)

---

### Summary

`PythLazer.initialize()` accepts `_topAuthority` and passes it directly to `__Ownable_init(_topAuthority)` without any zero-address guard. If `address(0)` is supplied, the contract owner is permanently set to the zero address, making all `onlyOwner` functions — including `updateTrustedSigner` and `_authorizeUpgrade` — permanently inaccessible. The contract becomes irreparably bricked: no trusted signers can ever be registered, all `verifyUpdate` calls revert, and the UUPS proxy cannot be upgraded to recover.

---

### Finding Description

`PythLazer.sol` is a UUPS-upgradeable contract that manages trusted Lazer signers and exposes `verifyUpdate()` for on-chain Lazer payload verification. [1](#0-0) 

The `initialize` function accepts `_topAuthority` and passes it directly to `__Ownable_init(_topAuthority)` with no prior zero-address check:

```solidity
function initialize(address _topAuthority) public initializer {
    __Ownable_init(_topAuthority);   // ← no require(_topAuthority != address(0))
    __UUPSUpgradeable_init();
    verification_fee = 1 wei;
}
```

In OpenZeppelin v5 upgradeable, `__Ownable_init(address initialOwner)` calls `_transferOwnership(initialOwner)` directly — the zero-address guard exists only in the non-upgradeable constructor, not in `_transferOwnership`. Supplying `address(0)` silently sets `_owner` to `address(0)`.

Every other Pyth upgradeable contract performs an explicit guard before calling `__Ownable_init`:

- `EntropyUpgradable.sol`: `require(owner != address(0), "owner is zero address");`
- `EchoUpgradeable.sol`: `require(owner != address(0), "owner is zero address");`
- `SchedulerUpgradeable.sol`: `require(owner != address(0), "owner is zero address");` [2](#0-1) [3](#0-2) [4](#0-3) 

`PythLazer.sol` is the sole exception.

The two `onlyOwner`-gated functions are:

```solidity
function updateTrustedSigner(address trustedSigner, uint256 expiresAt) external onlyOwner { ... }
function _authorizeUpgrade(address) internal override onlyOwner {}
``` [5](#0-4) 

With owner `address(0)`, both are permanently locked.

---

### Impact Explanation

- **`updateTrustedSigner` is permanently inaccessible**: No trusted signer can ever be registered. The `trustedSigners` array remains empty and `trustedSignerToExpiresAtMapping` is always zero for every address.
- **`verifyUpdate` always reverts**: `isValidSigner(signer)` returns `false` for every recovered signer because `trustedSignerToExpiresAtMapping[signer] == 0`, causing `revert("invalid signer")` on every call.
- **`_authorizeUpgrade` is permanently inaccessible**: The UUPS proxy cannot be pointed to a new implementation. There is no recovery path.
- **Contract must be redeployed**: The `initializer` modifier prevents re-initialization, so the only remedy is a full proxy redeployment.

The entire Lazer on-chain verification system is rendered non-functional.

---

### Likelihood Explanation

The deployer passes `_topAuthority` as a constructor argument during proxy initialization. A misconfiguration (e.g., passing a zero address due to a scripting error, wrong environment variable, or copy-paste mistake) is a realistic deployment-time error. The absence of a revert guard means the error is silent — the transaction succeeds, the contract appears initialized, and the breakage is only discovered when `updateTrustedSigner` or `verifyUpdate` is first called. All other Pyth upgradeable contracts guard against this exact mistake, confirming the team is aware of the risk pattern.

---

### Recommendation

Add an explicit zero-address check at the top of `initialize()`, consistent with all other Pyth upgradeable contracts:

```solidity
function initialize(address _topAuthority) public initializer {
    require(_topAuthority != address(0), "topAuthority is zero address");
    __Ownable_init(_topAuthority);
    __UUPSUpgradeable_init();
    verification_fee = 1 wei;
}
``` [1](#0-0) 

---

### Proof of Concept

1. Deploy the `PythLazer` proxy and call `initialize(address(0))`.
2. The transaction succeeds. `owner()` returns `address(0)`.
3. Call `updateTrustedSigner(validSigner, block.timestamp + 1 days)` — reverts with `OwnableUnauthorizedAccount(msg.sender)` because `msg.sender != address(0)`.
4. Call `verifyUpdate(validLazerPayload)` — reverts with `"invalid signer"` because `isValidSigner` returns `false` for every address (mapping is empty).
5. Attempt `upgradeTo(newImpl)` — reverts with `OwnableUnauthorizedAccount(msg.sender)`.
6. No recovery is possible without redeployment. [6](#0-5) [7](#0-6)
