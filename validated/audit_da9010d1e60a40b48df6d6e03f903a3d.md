### Title
Echo `executeCallback` Silently Fails User Callback When Provider Supplies Insufficient Gas ‚Äî (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

### Summary

`Echo.sol`'s `executeCallback` clears the request and credits the provider **before** attempting the user callback, and performs no on-chain check that the calling context holds enough gas to actually forward `callbackGasLimit` gas to the callback (accounting for the EVM's 63/64 forwarding rule and function overhead). A provider can supply just enough gas to execute the outer function but not enough to properly forward `callbackGasLimit` to the callback. The callback silently fails, the provider is paid in full, the request is permanently deleted, and the user has no recovery path.

---

### Finding Description

In `Echo.sol`, `executeCallback` performs the following sequence:

1. Validates the request and price IDs
2. Calls `pyth.parsePriceFeedUpdates{value: pythFee}(...)` ‚Äî consuming significant gas
3. **Credits the provider**: `_state.providers[providerToCredit].accruedFeesInWei += ...`
4. **Clears the request**: `clearRequest(sequenceNumber)`
5. Advances `firstUnfulfilledSeq` via a linear scan ‚Äî consuming additional gas
6. Attempts the callback: `IEchoConsumer(req.requester)._echoCallback{gas: req.callbackGasLimit}(...)`
7. Catches any failure silently and emits `PriceUpdateCallbackFailed` [1](#0-0) 

The critical flaw is that steps 3 and 4 are irreversible and occur **before** step 6. There is no check of the form:

```solidity
require(gasleft() >= uint256(req.callbackGasLimit) * 64 / 63 + OVERHEAD);
```

Due to the EVM's 63/64 rule (EIP-150), the gas forwarded to a sub-call is `min(specified, gasleft() - gasleft()/64)`. If `gasleft()` at the point of the `try` statement equals `callbackGasLimit`, the callback only receives `callbackGasLimit * 63/64` gas ‚Äî approximately 1.6% less than requested. For a 1 M gas limit this is ~15,873 gas short. More broadly, if a provider supplies just enough gas to survive the outer function (storage reads, `parsePriceFeedUpdates`, the `firstUnfulfilledSeq` scan) but leaves `gasleft() < callbackGasLimit * 64/63` at the call site, the callback is silently starved. [2](#0-1) 

The `IEcho` interface documents "Requires 1.5x the callback gas limit to account for cross-contract call overhead" but this is **documentation only** ‚Äî it is not enforced on-chain. [3](#0-2) 

Compare with Entropy's `revealWithCallback`, which explicitly checks `(startingGas * 31) / 32 > gasLimit` after the call and reverts with `InsufficientGas` if the calling context was under-gassed, and which also maintains a `CALLBACK_FAILED` recovery state: [4](#0-3) 

Echo has neither the pre-call gas guard nor a recovery state.

---

### Impact Explanation

A provider (or any caller after the exclusivity period) can:

1. Call `executeCallback` with just enough gas to execute the outer function but not enough to properly forward `callbackGasLimit` to the callback.
2. The callback receives less than `callbackGasLimit` gas, runs out of gas, and reverts.
3. The `catch` block emits `PriceUpdateCallbackFailed("low-level error (possibly out of gas)")`.
4. The provider has already been credited the full fee (including the gas component `callbackGasLimit * feePerGasInWei`).
5. The request has already been cleared ‚Äî there is no `CALLBACK_FAILED` recovery state in Echo.
6. The user's callback never ran and the funds paid for it are gone.

The provider profits by saving on transaction gas costs while collecting the full fee. The user suffers a silent loss of callback execution with no recourse.

---

### Likelihood Explanation

**High.** Any registered provider (or any address after the exclusivity window) can call `executeCallback`. The attack requires only crafting a transaction with a specific gas limit ‚Äî a trivial operation. The fee model charges users `callbackGasLimit * feePerGasInWei`, so providers have a direct financial incentive to supply less gas than paid for. The `parsePriceFeedUpdates` call and `firstUnfulfilledSeq` linear scan consume variable amounts of gas, making the exact "safe" gas amount hard for users to predict or verify. [5](#0-4) 

---

### Recommendation

1. **Add a pre-call gas sufficiency check** before the callback, accounting for the 63/64 rule and the overhead consumed by the outer function:

```solidity
// Ensure enough gas remains to forward callbackGasLimit to the callback
// (64/63 accounts for EVM 63/64 forwarding rule; CALLBACK_OVERHEAD accounts for function overhead)
require(
    gasleft() >= uint256(req.callbackGasLimit) * 64 / 63 + CALLBACK_OVERHEAD,
    "Insufficient gas for callback"
);
```

2. **Move provider crediting and request clearing to after the callback**, or implement a `CALLBACK_FAILED` recovery state (as Entropy does) so users can retry if the callback fails due to insufficient gas.

3. **Enforce the "1.5x" gas requirement on-chain**, not just in documentation.

---
