Audit Report

## Title
Missing Per-Peer Rate Limit on `GetBlockFilterCheckPoints` Enables DB Read Amplification — (`sync/src/filter/get_block_filter_check_points_process.rs`)

## Summary
The `BlockFilter` protocol handler contains no rate limiter, allowing any connected peer to repeatedly send `GetBlockFilterCheckPoints` messages. Each message unconditionally triggers up to 4,000 synchronous RocksDB reads (2,000 × `get_block_hash` + 2,000 × `get_block_filter_hash`) with no throttling. This allows a single attacker peer to saturate the node's storage I/O at negligible cost, degrading or crashing the node.

## Finding Description
In `sync/src/filter/get_block_filter_check_points_process.rs`, `GetBlockFilterCheckPointsProcess::execute` iterates up to `BATCH_SIZE = 2000` times, performing two DB reads per iteration (`get_block_hash` and `get_block_filter_hash`), for a maximum of 4,000 reads per message. The `BlockFilter` struct in `sync/src/filter/mod.rs` holds only `shared: Arc<SyncShared>` — no rate limiter field exists. The `try_process` dispatcher routes `GetBlockFilterCheckPoints` (and `GetBlockFilterHashes`, `GetBlockFilters`) directly to their handlers with no rate-limit check at any point in the call chain. This contrasts directly with `Relayer` in `sync/src/relayer/mod.rs`, which holds a `RateLimiter<(PeerIndex, u32)>` and gates every non-PoW message through `check_key(&(peer, message.item_id()))` before dispatch. The condition `latest >= start_number` is trivially satisfied with `start_number=0` once any filter data exists, so the full loop always executes.

## Impact Explanation
An attacker streaming `GetBlockFilterCheckPoints(start_number=0)` messages from a single peer connection causes the node to issue up to 4,000 synchronous RocksDB point-reads per message with no bound on message rate. Sustained flooding saturates the node's disk I/O, causing it to become unresponsive to sync, relay, and block-propagation messages — effectively crashing the node or causing severe network congestion at negligible attacker cost. This matches the allowed High impact: **"Vulnerabilities which could easily crash a CKB node"** and **"Vulnerabilities or bad designs which could cause CKB network congestion with few costs."**

## Likelihood Explanation
`SupportProtocols::Filter` is a standard negotiated protocol. Any peer that connects and negotiates it can immediately send `GetBlockFilterCheckPoints` without authentication, proof-of-work, or any prior state. The message is a fixed 8-byte struct (`start_number: Uint64`), so attacker bandwidth cost is essentially zero. The only precondition is that at least one block filter has been built on the target node, which is the normal operating state of any node with filters enabled. The attack is fully repeatable and requires only one TCP connection.

## Recommendation
Add a `RateLimiter<(PeerIndex, u32)>` field to `BlockFilter` (mirroring `Relayer`), initialize it in `BlockFilter::new`, and check it at the top of `try_process` before dispatching any inbound request message (`GetBlockFilterCheckPoints`, `GetBlockFilterHashes`, `GetBlockFilters`). The rate limit quota should be set conservatively given that each message triggers up to 4,000 DB reads. Additionally, consider capping the effective `BATCH_SIZE` based on the actual number of available checkpoints to avoid unnecessary iteration.

## Proof of Concept
1. Run a CKB node with block filter enabled; wait until filter data is built for at least 1 block (so `latest >= 0`).
2. Connect a peer that negotiates `SupportProtocols::Filter`.
3. In a tight loop, send `BlockFilterMessage { GetBlockFilterCheckPoints { start_number: 0 } }` (8-byte payload each).
4. Monitor RocksDB metrics or use `perf stat -e block:block_rq_issue` to observe ~2,000 `get_block_hash` + ~2,000 `get_block_filter_hash` DB reads triggered per received message.
5. Confirm no rate-limit rejection is ever returned (`Status::ignored()` is only returned when `latest < start_number`, never for rate limiting), and observe the node's I/O wait climbing proportionally to message rate until the node becomes unresponsive.
