### Title
Insufficient Price Staleness Validation in Chainlink Adapter — (`File: target_chains/ethereum/sdk/solidity/PythAggregatorV3.sol`)

---

### Summary

`PythAggregatorV3` is Pyth's official Chainlink `AggregatorV3Interface` adapter, explicitly recommended for protocols migrating from Chainlink. Every price-returning function in this contract calls `pyth.getPriceUnsafe()`, which makes **no staleness guarantees** and can return a price from arbitrarily far in the past. No staleness threshold is enforced anywhere in the adapter. Any protocol using this adapter as a drop-in Chainlink replacement is exposed to stale-price manipulation.

---

### Finding Description

`PythAggregatorV3` implements the Chainlink `AggregatorV3Interface` and is the officially documented migration path for Chainlink-dependent protocols. The developer hub explicitly instructs integrators to deploy this contract and point their existing Chainlink-consuming app at it.

All price-returning functions delegate to `pyth.getPriceUnsafe(priceId)`:

- `latestAnswer()` — returns raw price, no staleness check
- `latestRoundData()` — returns `(roundId, answer, startedAt, updatedAt, answeredInRound)` where `updatedAt = price.publishTime`, no staleness check
- `getRoundData()` — same pattern
- `decimals()` — same pattern [1](#0-0) [2](#0-1) 

`getPriceUnsafe` is explicitly documented to return a price from arbitrarily far in the past with no recency guarantee: [3](#0-2) 

The contrast with the safe API is clear — `getPriceNoOlderThan` enforces a staleness bound: [4](#0-3) 

The migration guide actively directs protocols to deploy this adapter as a replacement for their Chainlink feeds: [5](#0-4) 

---

### Impact Explanation

Any protocol that uses `PythAggregatorV3` as a Chainlink adapter and relies on `latestRoundData()` or `latestAnswer()` for pricing decisions (e.g., lending, liquidations, derivatives settlement) will receive a price that may be arbitrarily old. The `updatedAt` field returned by `latestRoundData()` is set to `price.publishTime`, which a downstream protocol may check — but the adapter itself never enforces that this timestamp is within an acceptable window. An attacker can exploit a known stale price to:

- Borrow against overvalued collateral (if the stale price is higher than current)
- Avoid liquidation when they should be liquidated
- Settle derivatives at a favorable historical price

This is a direct protocol-level financial loss vector for every integrator that follows the official Pyth migration guide.

---

### Likelihood Explanation

The `PythAggregatorV3` contract is the **officially recommended** migration path for Chainlink users, documented in the Pyth developer hub. Any protocol that followed this guide and did not independently add their own staleness check (which Chainlink-native protocols typically do not, because Chainlink's own heartbeat mechanism provides freshness) is vulnerable. The entry path requires only a normal transaction sender interacting with the downstream protocol — no privileged access is needed. The Pyth pull-oracle model means on-chain prices are only updated when someone calls `updatePriceFeeds`, so price staleness is a routine operational condition, not a rare edge case.

---

### Recommendation

Replace all `getPriceUnsafe` calls in `PythAggregatorV3` with `getPriceNoOlderThan(priceId, maxAge)`, where `maxAge` is a configurable state variable (analogous to Hifi's `priceStalenessThreshold`). Add a constructor parameter and a setter (owner-gated) for this threshold. This mirrors the fix applied in Hifi commit `4ddb307` and aligns with Pyth's own best-practices documentation. [4](#0-3) 

---

### Proof of Concept

1. Deploy `PythAggregatorV3` pointing at the live Pyth contract for ETH/USD.
2. Do **not** call `updateFeeds` — allow the on-chain price to go stale (e.g., 10 minutes old).
3. Call `latestRoundData()` on the adapter. Observe that it returns successfully with `updatedAt` = a 10-minute-old `publishTime` and `answer` = the 10-minute-old price. No revert occurs.
4. A downstream lending protocol consuming this adapter would accept this stale price as valid, allowing an attacker to borrow against collateral valued at the stale (potentially higher) price.

Compare: calling `pyth.getPriceNoOlderThan(priceId, 60)` directly on the Pyth contract for the same feed would revert with `StalePrice`, demonstrating that the protection exists in the Pyth core but is bypassed by the adapter's use of `getPriceUnsafe`. [2](#0-1) [4](#0-3)

        bytes32 id,
        uint age
    ) public view virtual override returns (PythStructs.Price memory price) {
        price = getPriceUnsafe(id);

        if (diff(block.timestamp, price.publishTime) > age)
            revert PythErrors.StalePrice();
