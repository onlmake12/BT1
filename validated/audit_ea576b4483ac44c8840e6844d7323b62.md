### Title
Minimum Fee Rate Check Uses Only Serialized Size, Not Actual Transaction Weight (Cycles), Allowing Fee Rate Bypass and Fee Estimator Poisoning — (`tx-pool/src/util.rs`)

---

### Summary

The tx-pool admission check in `check_tx_fee` enforces the minimum fee rate using only the transaction's serialized byte size. However, the actual transaction weight used for pool ordering and fee estimation is `max(tx_size, cycles * DEFAULT_BYTES_PER_CYCLES)`. An unprivileged transaction sender can craft a high-cycle, low-size transaction that passes the admission check while having an effective fee rate far below `min_fee_rate`. This poisons the fee estimator and bypasses the intended fee floor.

---

### Finding Description

In `tx-pool/src/util.rs`, `check_tx_fee` computes the minimum required fee using only `tx_size`:

```rust
let min_fee = tx_pool.config.min_fee_rate.fee(tx_size as u64);
if fee < min_fee {
    return Err(Reject::LowFeeRate(...));
}
```

The code itself acknowledges the discrepancy with a comment:

> "Theoretically we cannot use size as weight directly to calculate fee_rate, here min fee rate is used as a cheap check, so we will use size to calculate fee_rate directly"

However, the actual transaction weight used everywhere else — pool ordering, fee rate calculation, and fee estimation — is computed by `get_transaction_weight`:

```rust
pub fn get_transaction_weight(tx_size: usize, cycles: u64) -> u64 {
    std::cmp::max(
        tx_size as u64,
        (cycles as f64 * DEFAULT_BYTES_PER_CYCLES) as u64,
    )
}
```

where `DEFAULT_BYTES_PER_CYCLES = 0.000_170_571_4`. For a transaction with `max_tx_verify_cycles = 70_000_000` cycles, the cycle-equivalent weight is `~11,940` bytes. A transaction with a 200-byte serialized size and 70M cycles has:

- **Admission check weight**: 200 bytes → minimum fee = `1000 * 200 / 1000 = 200 shannons`
- **Actual weight**: `max(200, 11940) = 11,940` bytes
- **Effective fee rate**: `200 * 1000 / 11940 ≈ 16.7 shannons/KW` — ~60× below `min_fee_rate`

After `pre_check` passes, `_process_tx` runs full script verification, obtains `verified.cycles`, and creates `TxEntry::new(rtx, verified.cycles, fee, tx_size)`. There is no second fee rate check using the actual weight. The entry enters the pool with its true cycles but only the size-based gate was applied.

The fee estimators (`weight_units_flow::Algorithm::accept_tx`, `confirmation_fraction::Algorithm::accept_tx`, `pool_map::estimate_fee_rate`) all call `get_transaction_weight(size, cycles)` — the correct weight — so they correctly observe the artificially low effective fee rate of these admitted transactions.

---

### Impact Explanation

1. **Fee estimator poisoning**: An attacker floods the pool with many high-cycle, low-size transactions paying just above `min_fee_rate * tx_size`. Each such entry appears in the fee estimator with an effective fee rate far below `min_fee_rate`. The `estimate_fee_rate` RPC and internal fallback estimator return artificially depressed fee rate recommendations, causing legitimate users to underpay and have their transactions stuck or evicted.

2. **Pool admission bypass**: Transactions with effective fee rates far below the configured `min_fee_rate` enter the pool, violating the intended invariant that all pooled transactions meet the minimum fee rate threshold.

---

### Likelihood Explanation

Any unprivileged user can submit transactions via the `send_transaction` RPC or P2P relay. A script author can deploy a lock or type script that consumes close to `max_tx_verify_cycles` (default 70M) cycles while keeping the transaction's serialized size small. The attacker needs no special access — only the ability to submit valid transactions with a custom script. The cost is low: the attacker pays only `min_fee_rate * tx_size` shannons per transaction, which is far less than what a legitimately weighted transaction would require.

---

### Recommendation

After script verification completes and `verified.cycles` is known, perform a second fee rate check using the actual weight:

```rust
let actual_weight = get_transaction_weight(tx_size, verified.cycles);
let min_fee_by_weight = tx_pool.config.min_fee_rate.fee(actual_weight);
if fee < min_fee_by_weight {
    return Err(Reject::LowFeeRate(...));
}
```

This ensures the effective fee rate (fee / actual_weight) always meets `min_fee_rate`, closing the gap between the admission check and the pool's internal accounting.
