### Title
Race Condition in `setFeeManager` Allows Compromised Fee Manager to Front-Run Revocation and Drain Provider Fees - (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

The `setFeeManager` function in `Entropy.sol` allows a registered Entropy provider to delegate fee-withdrawal and fee-setting authority to an external address. If that fee manager address is compromised, the provider's only recourse is to call `setFeeManager` again to revoke the delegation. An attacker who controls the compromised fee manager can observe the pending revocation in the public mempool and front-run it with a `withdrawAsFeeManager` call, draining all of the provider's accrued fees before the revocation is confirmed.

---

### Finding Description

`setFeeManager` performs an immediate, atomic overwrite of the `feeManager` field with no time-lock, pending-state, or two-step revocation mechanism:

```solidity
function setFeeManager(address manager) external override {
    EntropyStructsV2.ProviderInfo storage provider = _state.providers[msg.sender];
    if (provider.sequenceNumber == 0) {
        revert EntropyErrors.NoSuchProvider();
    }
    address oldFeeManager = provider.feeManager;
    provider.feeManager = manager;          // immediate overwrite
    emit ProviderFeeManagerUpdated(msg.sender, oldFeeManager, manager, ...);
}
``` [1](#0-0) 

The fee manager has two powerful capabilities:

1. **`withdrawAsFeeManager`** ã transfers the provider's entire `accruedFeesInWei` balance to `msg.sender` (the fee manager): [2](#0-1) 

2. **`setProviderFeeAsFeeManager`** ã overwrites the provider's `feeInWei` with any arbitrary value: [3](#0-2) 

The `ProviderInfo` struct confirms `feeManager` is the sole authorization gate for both operations: [4](#0-3) 

The `setFeeManager` feature is actively used in production ã the Fortuna keeper daemon calls `sync_fee_manager` on startup to keep the on-chain fee manager in sync with config: [5](#0-4) 

---

### Impact Explanation

A compromised fee manager can, in the window between the provider's revocation transaction being broadcast and it being mined:

- Call `withdrawAsFeeManager(provider, accruedFeesInWei)` to drain the provider's entire ETH fee balance to the attacker's address.
- Call `setProviderFeeAsFeeManager(provider, 0)` to set the provider's fee to zero, causing all future requests to accrue zero revenue, or to an extreme value to price out users.

The financial loss is bounded only by the provider's `accruedFeesInWei` at the time of the attack. For a high-volume Entropy provider this can be substantial.

---

### Likelihood Explanation

- The `setFeeManager` delegation pattern is a documented, production-used feature (Fortuna's `sync_fee_manager`).
- EVM chains with public mempools make pending transactions visible to anyone running a node or watching a mempool service.
- Front-running is a well-understood, low-effort attack on EVM: the attacker simply resubmits `withdrawAsFeeManager` with a higher gas price than the provider's revocation transaction.
- No special privilege is required beyond controlling the compromised fee manager address.

---

### Recommendation

**Short term:** Document the race condition risk prominently in the `setFeeManager` NatSpec. Advise providers to withdraw all accrued fees themselves (via `withdraw`) *before* broadcasting a fee manager revocation, so there is nothing left to steal.

**Long term:** Implement a two-step revocation with a time-lock (e.g., a pending revocation that takes effect after N blocks), or require the provider to zero out `accruedFeesInWei` before a fee manager change is accepted. Alternatively, emit a `ProviderFeeManagerUpdated` event and enforce a mandatory delay before the new manager takes effect, giving providers time to react to unauthorized changes.

---

### Proof of Concept

```
1. Provider Alice registers and accumulates fees: accruedFeesInWei = 10 ETH.
2. Alice calls setFeeManager(BobContract).
3. BobContract is compromised by Attacker.
4. Alice broadcasts setFeeManager(address(0)) to revoke BobContract.
5. Attacker sees Alice's pending tx in the mempool.
6. Attacker submits withdrawAsFeeManager(Alice, 10 ETH) with higher gas price.
7. Attacker's tx is mined first ã 10 ETH transferred to Attacker.
8. Alice's revocation tx mines next ã feeManager set to address(0), but balance is already 0.
9. Alice loses all accrued fees with no recourse.
```

    function withdrawAsFeeManager(
        address provider,
        uint128 amount
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        // Use checks-effects-interactions pattern to prevent reentrancy attacks.
        require(
            providerInfo.accruedFeesInWei >= amount,
            "Insufficient balance"
        );
        providerInfo.accruedFeesInWei -= amount;

        // Interaction with an external contract or token transfer
        (bool sent, ) = msg.sender.call{value: amount}("");
        require(sent, "withdrawal to msg.sender failed");

        emit EntropyEvents.Withdrawal(provider, msg.sender, amount);
        emit EntropyEventsV2.Withdrawal(
            provider,
            msg.sender,
            amount,
            bytes("")
        );
    }

async fn sync_fee_manager(
    contract: &Arc<SignablePythContract>,
    provider_info: &EntropyStructsV2ProviderInfo,
    fee_manager: Address,
) -> Result<()> {
    if provider_info.fee_manager != fee_manager {
        tracing::info!("Updating provider fee manager to {:?}", fee_manager);
        if let Some(receipt) = contract.set_fee_manager(fee_manager).send().await?.await? {
            tracing::info!("Updated provider fee manager: {:?}", receipt);
        }
    }
    Ok(())
}
