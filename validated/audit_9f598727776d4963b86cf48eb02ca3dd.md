### Title
Entropy Provider Can Front-Run `requestV2` With `setProviderFee` To Cause Revert or Capture Excess User ETH — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

An Entropy provider can monitor the mempool for pending `requestV2`/`requestWithCallback` calls and front-run them by calling `setProviderFee` (or `setProviderFeeAsFeeManager`) to instantly raise their `feeInWei`. Because `setProviderFee` has no timelock or delay, the fee change takes effect in the same block. This causes the user's transaction to revert with `InsufficientFee` (griefing), or — if the user sent excess ETH — allows the provider to capture that excess at Pyth's expense.

---

### Finding Description

In `Entropy.sol`, `requestHelper` reads the provider's fee from storage at execution time:

```solidity
uint128 requiredFee = getFeeV2(provider, callbackGasLimit);
if (msg.value < requiredFee) revert EntropyErrors.InsufficientFee();
uint128 providerFee = getProviderFee(provider, callbackGasLimit);
providerInfo.accruedFeesInWei += providerFee;
_state.accruedPythFeesInWei += (SafeCast.toUint128(msg.value) - providerFee);
``` [1](#0-0) 

`setProviderFee` applies the new fee immediately with no delay:

```solidity
function setProviderFee(uint128 newFeeInWei) external override {
    ...
    provider.feeInWei = newFeeInWei;
    emit ProviderFeeUpdated(msg.sender, oldFeeInWei, newFeeInWei);
``` [2](#0-1) 

`setProviderFeeAsFeeManager` has the same instant-effect property: [3](#0-2) 

The IEntropyV2 documentation explicitly warns users that excess `msg.value` is **not refunded**:

> "Note that the fee can change over time. Callers of this method should explicitly compute `getFeeV2()` prior to each invocation… excess value is *not* refunded to the caller." [4](#0-3) 

This warning causes rational users to send a buffer above `getFeeV2()` to avoid reverts. That buffer is the attack surface.

The same pattern exists in `Echo.sol`'s `setProviderFee`, which also takes effect immediately: [5](#0-4) 

---

### Impact Explanation

**Attack vector A — Griefing (revert):**
The provider raises `feeInWei` by 1 wei before the user's `requestV2{value: X}` lands. The check `msg.value < requiredFee` now fails and the user's transaction reverts, wasting their gas. The provider then restores the fee. A malicious provider can selectively block any user from requesting randomness at negligible cost.

**Attack vector B — Fee extraction (excess ETH capture):**
A user who sends `msg.value = 2 * providerFee + pythFeeInWei` (a 2× buffer) to guard against fee increases is vulnerable. The provider front-runs with `setProviderFee(2 * providerFee)`. The user's transaction now passes the fee check (barely), but `providerFee` is computed as `2 * providerFee` at execution time. The provider accrues double their normal fee; Pyth receives only `pythFeeInWei` instead of `providerFee + pythFeeInWei`. The provider gains the user's entire buffer at Pyth's expense.

Concretely, from `requestHelper`:
- Provider accrues: `getProviderFee(provider, gasLimit)` — evaluated at execution time with the raised fee
- Pyth accrues: `msg.value − providerFee` — shrinks to just `pythFeeInWei` [6](#0-5) 

---

### Likelihood Explanation

- The provider controls their own key and can submit `setProviderFee` at any time.
- On chains with a public mempool (Ethereum mainnet, most L2s), pending `requestV2` transactions are visible before inclusion.
- The attack requires only one extra transaction (`setProviderFee`) and can be automated.
- The documentation's explicit "excess is not refunded" warning incentivises users to send buffers, making attack vector B realistic.
- No special privilege beyond being a registered provider is required; any registered provider can do this.

---

### Recommendation

1. **Timelock on fee increases**: Require that fee increases are announced at least N blocks/seconds in advance before taking effect. Fee decreases can remain instant (they only benefit users).
2. **User-specified max fee**: Allow callers to pass a `maxFeeInWei` parameter; revert if the provider's current fee exceeds it at execution time.
3. **Snapshot fee at request time**: Store the provider fee in the request struct (as Echo already does for `req.fee`) so the amount credited to the provider is bounded by what the user agreed to pay.

---

### Proof of Concept

```
// Attack vector B — excess ETH capture

// 1. Provider registers with feeInWei = 1000 wei
provider.register(1000, commitment, metadata, chainLength, uri);

// 2. User queries fee: getFeeV2(provider, 0) = 1000 + pythFee
//    User decides to send 2× buffer: msg.value = 2000 + pythFee

// 3. User submits (pending in mempool):
//    requestV2{value: 2000 + pythFee}(provider, userRandom, 0)

// 4. Provider sees the pending tx, front-runs:
provider.setProviderFee(2000);   // raises fee to match user's buffer

// 5. User's tx executes:
//    requiredFee = getFeeV2(provider, 0) = 2000 + pythFee
//    msg.value (2000 + pythFee) >= requiredFee → passes
//    providerFee = getProviderFee(provider, 0) = 2000
//    providerInfo.accruedFeesInWei += 2000   ← double the normal fee
//    accruedPythFeesInWei += (2000 + pythFee) - 2000 = pythFee only

// 6. Provider restores fee:
provider.setProviderFee(1000);

// Result: provider captured 1000 wei extra; Pyth lost 1000 wei of protocol revenue.
```

    function setProviderFeeAsFeeManager(
        address provider,
        uint128 newFeeInWei
    ) external override {
        EntropyStructsV2.ProviderInfo storage providerInfo = _state.providers[
            provider
        ];

        if (providerInfo.sequenceNumber == 0) {
            revert EntropyErrors.NoSuchProvider();
        }

        if (providerInfo.feeManager != msg.sender) {
            revert EntropyErrors.Unauthorized();
        }

        uint128 oldFeeInWei = providerInfo.feeInWei;
        providerInfo.feeInWei = newFeeInWei;

        emit ProviderFeeUpdated(provider, oldFeeInWei, newFeeInWei);
        emit EntropyEventsV2.ProviderFeeUpdated(
            provider,
            oldFeeInWei,
            newFeeInWei,
            bytes("")
        );
    }
