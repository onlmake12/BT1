### Title
Permissionless Keeper Can Inflate `updateData` to Extract Profit and Drain Subscription Balances via Gas Reimbursement — (File: `target_chains/ethereum/contracts/contracts/pulse/Scheduler.sol`)

---

### Summary

The `updatePriceFeeds` function in `Scheduler.sol` reimburses the caller (keeper) for all execution gas consumed during the call, including gas consumed by the Pyth contract processing the caller-supplied `updateData`. Because `updateData` is fully attacker-controlled, unbounded in size, and `updatePriceFeeds` is permissionless, a malicious keeper can pad `updateData` with extra price-feed messages to inflate execution gas, extract profit from the subscription balance, and drain subscriptions prematurely.

---

### Finding Description

`updatePriceFeeds` captures gas at entry:

```solidity
uint256 startGas = gasleft();   // line 279
```

It then passes the caller-supplied `updateData` directly to the Pyth contract:

```solidity
uint256 pythFee = pyth.getUpdateFee(updateData);   // fee ∝ totalNumUpdates in updateData
...
status.balanceInWei -= pythFee;                    // subscription pays Pyth fee
pyth.parsePriceFeedUpdatesWithConfig{value: pythFee}(
    updateData,          // ← fully attacker-controlled, no size bound
    params.priceIds,
    ...
);
```

Finally, the keeper is paid for **all** execution gas consumed since `startGas`:

```solidity
uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
uint256 totalKeeperFee = gasCost + keeperSpecificFee;
...
(bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
```

`GAS_OVERHEAD` is the fixed constant `30 000` gas.

The Pyth contract's `getUpdateFee` and `parsePriceFeedUpdatesWithConfig` iterate over **every** message embedded in `updateData` — the fee and the execution gas both scale linearly with `totalNumUpdates`. A subscription only needs its `params.priceIds` to be present, but the Pyth contract still processes every extra message to locate them.

**Attack mechanics:**

| Component | Per extra message |
|---|---|
| Calldata cost paid by attacker | ~16 gas/byte × ~200 bytes ≈ **3 200 gas** |
| Execution gas reimbursed from subscription | Merkle-proof verification + storage ops ≈ **10 000 – 50 000 gas** |
| Net profit to attacker | **~6 800 – 46 800 gas × tx.gasprice** |

With up to 255 messages per accumulator element and an unbounded `updateData` array, the attacker can include thousands of extra messages per call.

The subscription balance is drained through two channels simultaneously:
1. **Extra Pyth fees** — `singleUpdateFeeInWei × extraMessages`, paid to the Pyth contract.
2. **Extra gas reimbursement** — execution gas for processing extra messages, paid to the attacker.

---

### Impact Explanation

- A malicious keeper drains a victim subscription's ETH balance far faster than the legitimate update cadence would require.
- The attacker extracts a net profit on every call (execution gas reimbursement > calldata cost).
- Subscriptions run out of funds prematurely, causing price-feed updates to stop entirely — directly harming any protocol that depends on those feeds.
- The attack is repeatable on every heartbeat interval and scales with the number of active subscriptions.

---

### Likelihood Explanation

`updatePriceFeeds` is explicitly permissionless — the README states *"Anyone can run a Keeper node; no registration is required to call `updatePriceFeeds`."* No privileged role, leaked key, or governance majority is required. Any EOA can deploy a malicious keeper contract, call `updatePriceFeeds` with bloated `updateData` on every eligible subscription, and collect profit. The attack is straightforward to automate.

---

### Recommendation

1. **Validate `updateData` message count.** Before calling `parsePriceFeedUpdatesWithConfig`, assert that the total number of price-feed messages in `updateData` does not exceed `params.priceIds.length` (or a small multiple thereof). Revert if it does.
2. **Cap gas reimbursement.** Bound `gasCost` to a maximum derived from `params.priceIds.length` and a per-feed gas estimate, so a keeper cannot extract more than the expected cost of a legitimate update.
3. **Separate Pyth-fee accounting from keeper-gas accounting.** The Pyth fee already scales with `updateData` size; the keeper should not also be reimbursed for the execution gas of processing extra messages.

---

### Proof of Concept

1. Create a subscription with 2 price IDs and fund it with 1 ETH.
2. Construct `updateData` containing one legitimate accumulator update (with the 2 required feeds) **plus** 253 additional valid-but-irrelevant price-feed messages in the same accumulator element (up to the `uint8` cap of 255 per element), and repeat across multiple array elements.
3. Call `scheduler.updatePriceFeeds(subscriptionId, updateData)` from an attacker EOA.
4. Observe:
   - `pythFee` charged to the subscription = `singleUpdateFeeInWei × (2 + 253 × N)`.
   - `gasCost` reimbursed to the attacker = `(startGas − gasleft() + 30 000) × tx.gasprice`, inflated by the Pyth contract processing all extra messages.
   - Attacker's net ETH gain = gas reimbursement − calldata cost > 0.
5. Repeat on every heartbeat interval until the subscription balance is exhausted. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    function _processFeesAndPayKeeper(
        SchedulerStructs.SubscriptionStatus storage status,
        uint256 startGas,
        uint256 numPriceIds
    ) internal {
        // Calculate fee components
        uint256 gasCost = (startGas - gasleft() + GAS_OVERHEAD) * tx.gasprice;
        uint256 keeperSpecificFee = uint256(_state.singleUpdateKeeperFeeInWei) *
            numPriceIds;
        uint256 totalKeeperFee = gasCost + keeperSpecificFee;

        // Check balance
        if (status.balanceInWei < totalKeeperFee) {
            revert SchedulerErrors.InsufficientBalance();
        }

        status.balanceInWei -= totalKeeperFee;
        status.totalSpent += totalKeeperFee;

        // Pay keeper and update status
        (bool sent, ) = msg.sender.call{value: totalKeeperFee}("");
        if (!sent) {
            revert SchedulerErrors.KeeperPaymentFailed();
        }
    }
