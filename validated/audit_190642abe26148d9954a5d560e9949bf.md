Audit Report

## Title
Unbounded TCP/WebSocket Connection and Subscription Growth Causes Node Resource Exhaustion — (`rpc/src/server.rs`, `rpc/src/module/subscription.rs`)

## Summary

The TCP RPC server accepts connections in an unbounded loop with no connection cap, and `SubscriptionRpcImpl::subscribe()` creates a new `broadcast::Receiver` and persistent async task per call with no per-connection or global limit. An attacker with access to `tcp_listen_address` or `ws_listen_address` can open arbitrarily many connections and subscribe to all six topics, causing linear growth in tokio tasks, broadcast receivers, and file descriptors, ultimately crashing the node through memory or FD exhaustion.

## Finding Description

**Unbounded TCP accept loop** — `start_tcp_server` in `rpc/src/server.rs` (lines 176–194) loops unconditionally on `listener.accept()` and spawns a new `tokio::spawn` task per connection with no semaphore, counter, or backpressure:

```rust
while let Ok((stream, _)) = listener.accept().await {
    tokio::spawn(async move { ... });
}
```

**No subscription guard** — `SubscriptionRpcImpl::subscribe()` in `rpc/src/module/subscription.rs` (lines 214–239) unconditionally calls `tx.subscribe()`, allocating a `broadcast::Receiver` and returning a `BoxStream` that the `jsonrpc_utils` pub_sub layer drives as a persistent async task. There is no check on `tx.receiver_count()`, no per-connection cap, and no global limit.

**Lagged receivers stay alive** — The `RecvError::Lagged` arm (lines 230–232) logs an error and continues the loop. Slow or non-reading clients never cause their receiver or task to be dropped.

**No config field for limits** — `RpcConfig` (`util/app-config/src/configs/rpc.rs`, lines 26–61) has no `max_connections` or `max_subscriptions` field.

**Resource growth mechanics:**
- N connections × 6 topics = 6N live tokio tasks + 6N `broadcast::Receiver` structs.
- On every new block, `broadcast::Sender::send()` wakes all N receivers for `new_tip_header` and `new_tip_block` simultaneously — an O(N) burst of task wakeups in the tokio scheduler.
- Each TCP connection consumes one OS file descriptor.

## Impact Explanation

**High — Vulnerabilities which could easily crash a CKB node.**

Memory exhaustion from O(N) tokio task stacks and receiver structs, FD exhaustion from O(N) open TCP sockets, and tokio scheduler saturation from O(N) simultaneous wakeups per block all converge to crash or severely degrade the node. Block validation, peer relay, and tx-pool processing share the same tokio runtime and are directly starved when the scheduler is saturated.

## Likelihood Explanation

Requires `tcp_listen_address` or `ws_listen_address` to be configured (opt-in, not default). When enabled, no credentials, PoW, or special privileges are required. TCP connections are cheap; a single attacker machine can open tens of thousands of connections using standard socket APIs. The exploit is linear, deterministic, and locally reproducible.

## Recommendation

1. **Connection limit**: Add a `max_connections: Option<usize>` field to `RpcConfig` and enforce it in `start_tcp_server` via a `tokio::sync::Semaphore` before accepting each connection.
2. **Per-connection subscription limit**: In `subscribe()`, check `tx.receiver_count()` against a configurable maximum and return a JSON-RPC error if exceeded.
3. **Global subscription limit**: Maintain an `Arc<AtomicUsize>` counter across all topics and connections, checked atomically in `subscribe()`.
4. **Idle connection timeout**: Drop connections that do not send or read within a configurable window.

## Proof of Concept

```python
import socket, json

N = 10000
socks = []
for i in range(N):
    s = socket.socket()
    s.connect(("127.0.0.1", 18114))
    for topic in ["new_tip_block", "new_tip_header", "new_transaction",
                  "proposed_transaction", "rejected_transaction", "log"]:
        req = json.dumps({"id": i, "jsonrpc": "2.0", "method": "subscribe",
                          "params": [topic]}) + "\n"
        s.sendall(req.encode())
    socks.append(s)

input("Hold connections open — observe node RSS grow O(N), tokio task count reach ~6N, "
      "and block-relay latency increase with each new block.")
```

Observable: node RSS grows proportionally to N; `tokio` task count reaches ~6N; block-relay latency degrades measurably as the scheduler processes the O(N) wakeup burst on each new block; FD count approaches the OS limit.
