### Title
Unbounded `numHashes` in `constructProviderCommitment` Loop Causes Permanent Out-of-Gas DoS on Entropy Reveal When `maxNumHashes` Is Unset — (File: `target_chains/ethereum/contracts/contracts/entropy/Entropy.sol`)

---

### Summary

When an Entropy provider registers without setting `maxNumHashes` (the default is `0`, which disables the guard), there is no protocol-level cap on the `numHashes` value stored in each request. An unprivileged user can make many sequential requests without the provider advancing its commitment, causing new requests to accumulate a `numHashes` value large enough to exhaust the block gas limit inside `constructProviderCommitment`. Those requests become permanently unfulfillable.

---

### Finding Description

**Step 1 — `numHashes` is computed at request time with no hard cap.**

In `requestHelper`, `req.numHashes` is set to the gap between the current sequence number and the provider's last-advanced commitment sequence number:

```solidity
req.numHashes = SafeCast.toUint32(
    assignedSequenceNumber -
        providerInfo.currentCommitmentSequenceNumber
);
```

The only guard is:

```solidity
if (
    providerInfo.maxNumHashes != 0 &&
    req.numHashes > providerInfo.maxNumHashes
) {
    revert EntropyErrors.LastRevealedTooOld();
}
```

When `maxNumHashes == 0` (the default after `register()`), this check is entirely skipped. There is no protocol-enforced upper bound on `numHashes`.

**Step 2 — `constructProviderCommitment` loops `numHashes` times.**

At reveal time, `revealHelper` calls:

```solidity
bytes32 providerCommitment = constructProviderCommitment(
    req.numHashes,
    providerContribution
);
```

which executes:

```solidity
function constructProviderCommitment(
    uint64 numHashes,
    bytes32 revelation
) internal pure returns (bytes32 currentHash) {
    currentHash = revelation;
    while (numHashes > 0) {
        currentHash = keccak256(bytes.concat(currentHash));
        numHashes -= 1;
    }
}
```

Each `keccak256` costs approximately 30 gas. With a 30 M-gas block limit, `numHashes ≳ 1,000,000` causes an out-of-gas revert. The request is then permanently stuck: it cannot be revealed, and the user's randomness is never delivered.

**Step 3 — The Fortuna keeper explicitly skips commitment advancement when `maxNumHashes == 0`.**

```rust
if provider_info.max_num_hashes == 0 {
    return Ok(());
}
```

So the off-chain safety valve that would otherwise call `advanceProviderCommitment` is also disabled for providers that never set `maxNumHashes`.

---

### Impact Explanation

Any user who submits a request after the gap between `providerInfo.sequenceNumber` and `providerInfo.currentCommitmentSequenceNumber` exceeds ~1 M will receive a request whose `numHashes` is too large to verify within the block gas limit. `reveal` and `revealWithCallback` will both revert with out-of-gas for those requests. The affected users permanently lose their randomness and the fees they paid. If the default Pyth provider operates with `maxNumHashes == 0`, this affects all users of the default Entropy endpoint.

---

### Likelihood Explanation

An unprivileged attacker can call `requestV2` (or `requestWithCallback`) repeatedly, paying the provider fee each time, to drive up `providerInfo.sequenceNumber` while the provider's `currentCommitmentSequenceNumber` stays fixed. Once the gap exceeds ~1 M, all subsequent requests (including legitimate ones from other users) are permanently unrevealable. The cost to the attacker is `~1,000,000 × feeInWei`; on chains where the Pyth fee is small this is feasible. The Fortuna keeper provides no automatic mitigation when `maxNumHashes == 0`.

---

### Recommendation

1. **Enforce a protocol-level hard cap on `numHashes`** at request time, independent of `maxNumHashes`. For example, revert if `req.numHashes > MAX_NUM_HASHES` where `MAX_NUM_HASHES` is a constant sized to fit within the block gas limit.
2. **Require `maxNumHashes != 0`** for providers that opt into the callback failure-state flow, or document clearly that `maxNumHashes == 0` is unsafe for production providers.
3. **Update the Fortuna keeper** to advance the commitment even when `maxNumHashes == 0`, using a fixed internal threshold.

---

### Proof of Concept

```
1. Provider registers with maxNumHashes = 0 (default).
   provider.currentCommitmentSequenceNumber = 0
   provider.sequenceNumber = 1

2. Attacker calls requestV2() 1,100,000 times (paying feeInWei each time).
   provider.sequenceNumber = 1,100,001
   provider.currentCommitmentSequenceNumber = 0  (never advanced)

3. The 1,100,000th request is stored with:
   req.numHashes = SafeCast.toUint32(1,100,000 - 0) = 1,100,000

4. Provider (or anyone) calls revealWithCallback(...) for that request.
   constructProviderCommitment(1,100,000, ...) loops 1,100,000 × keccak256
   ≈ 1,100,000 × 30 gas = 33,000,000 gas > 30,000,000 gas block limit
   → transaction reverts with out-of-gas.

5. The request is permanently unfulfillable; the user never receives randomness.
