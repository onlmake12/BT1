Audit Report

## Title
Unauthenticated `from` Field in `ConnectionRequestProcess::execute()` Enables Forward Rate-Limiter Bypass and Amplified Network Flooding — (`network/src/protocols/hole_punching/component/connection_request.rs`)

## Summary

`ConnectionRequestProcess::execute()` parses `content.from` from the attacker-controlled message payload and never verifies it matches the authenticated peer ID of the sending session. Because `forward_rate_limiter` is keyed on `(content.from, content.to, msg_item_id)`, an attacker can rotate spoofed `from` values to obtain a fresh rate-limit bucket per message, bypassing the 1 req/s per-pair cap. Each bypassed message is then broadcast to `sqrt(N)` peers, producing amplified network flooding from a single P2P connection.

## Finding Description

In `execute()`, `content` is parsed from the attacker-supplied payload with no subsequent comparison to the actual session's peer ID: [1](#0-0) 

`self.peer` (the authenticated `PeerIndex`) is available in the struct but is never used to validate `content.from`. The forward rate limiter is then checked against the spoofed value: [2](#0-1) 

The `forward_rate_limiter` is configured at 1 req/s per `(PeerId, PeerId, u32)` key: [3](#0-2) 

By sending each message with a distinct spoofed `from`, the attacker gets a fresh bucket every time, making the forward rate limiter ineffective. The only real constraint is the session-level limiter at 30 req/s, keyed on `(session_id, msg.item_id())`: [4](#0-3) [5](#0-4) 

When `to` is not directly connected, `forward_message()` broadcasts to `sqrt(total_peers)` peers: [6](#0-5) 

`forward_request()` only appends the relay's own ID to `route` and decrements `max_hops`, leaving `from` unchanged: [7](#0-6) 

Each receiving relay applies its own `forward_rate_limiter` against the same spoofed `(from, to)` key — also a fresh bucket — and re-broadcasts to another `sqrt(N)` peers. The route-based loop prevention (`content.route.contains(self_peer_id)`) prevents a single node from processing the same message twice, bounding per-message amplification to at most N total nodes (not `sqrt(N)^6` as claimed). The actual amplification is 30 messages/second × up to N nodes per message = up to 30N total message deliveries per second from a single attacker connection. The report's `sqrt(N)^6 = N^3` figure overstates the amplification, but the attack remains significant. [8](#0-7) 

When the relay is the `to` target, `respond_delivered()` stores the spoofed `from` into `pending_delivered`, poisoning NAT traversal state: [9](#0-8) 

The same `forward_rate_limiter` bypass applies identically to `ConnectionRequestDelivered`: [10](#0-9) 

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

A single attacker connection sustains 30 spoofed `ConnectionRequest` messages/second (session cap). Each message is broadcast to `sqrt(N)` peers, each of which re-broadcasts to `sqrt(N)` more peers, up to 6 hops, with per-node loop prevention bounding total deliveries per message to at most N nodes. For N=100, one attacker connection generates up to 3,000 message deliveries/second; for N=1,000, up to 30,000/second. This constitutes meaningful amplified network flooding requiring only a standard P2P connection.

## Likelihood Explanation

Requires only a single standard P2P connection to any relay node — no keys, no hashpower, no majority stake. The `ConnectionRequest` message is a normal production message type. The missing check is a single equality assertion. The attack is trivially scriptable and repeatable indefinitely.

## Recommendation

After parsing `content`, resolve the actual peer ID for `self.peer` from the peer registry and assert it equals `content.from`. If they differ, return a ban-triggering status:

```rust
let actual_peer_id = self.protocol.network_state
    .with_peer_registry(|reg| reg.get_peer(self.peer).map(|p| p.peer_id.clone()));
if actual_peer_id.as_ref() != Some(&content.from) {
    return StatusCode::InvalidFromPeerId
        .with_context("from peer id does not match the actual sender");
}
```

Apply the same check to `ConnectionRequestDelivered` and `ConnectionSync`. This ensures `content.from` is always the cryptographically authenticated identity of the sending session, making the `forward_rate_limiter` key unforgeable.

## Proof of Concept

1. Attacker connects to relay node R, authenticated as peer ID `A`.
2. Attacker sends 30 `ConnectionRequest` messages/second, each with a distinct spoofed `from` value (`B1`, `B2`, ..., `B30`) and a non-connected `to` target `C`.
3. `execute()` parses `content.from = Bi`, never checks `Bi == A`.
4. `forward_rate_limiter.check_key(&(Bi, C, item_id))` passes for each `i` (fresh bucket).
5. `forward_message()` broadcasts each message to `sqrt(N)` peers.
6. Each receiving peer repeats steps 3–5 with its own fresh buckets, re-broadcasting to `sqrt(N)` more peers, up to 6 hops (bounded by route loop prevention to at most N total nodes per message).
7. Network message volume grows as up to `30 × N` deliveries/second from a single attacker connection.
8. To verify: instrument a test network with N nodes, send 30 spoofed-`from` `ConnectionRequest` messages/second from one node, and measure total message deliveries across the network vs. the expected 30/s cap.

pub(crate) fn forward_request(
    request: packed::ConnectionRequestReader<'_>,
    current_id: &PeerId,
) -> packed::ConnectionRequest {
    let max_hops: u8 = request.max_hops().into();
    let message = request.to_entity();
    let new_route = message
        .route()
        .as_builder()
        .push(current_id.as_bytes())
        .build();
    message
        .as_builder()
        .max_hops(max_hops.saturating_sub(1))
        .route(new_route)
        .build()
}
