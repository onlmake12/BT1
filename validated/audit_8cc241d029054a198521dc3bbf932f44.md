Audit Report

## Title
`required_flags_filter` COMPATIBILITY bypass allows capability-mismatched peers to consume outbound slots and open all protocols — (`network/src/peer_store/peer_store_impl.rs`)

## Summary

The `required_flags_filter` function contains a special-case that allows a peer advertising only `Flags::COMPATIBILITY` (bits=`0b1`) to pass the capability check when the node's `required_flags` equals `RELAY | DISCOVERY | SYNC`. This causes the node to open all non-feeler protocols (SYNC, RELAY, DISCOVERY) to a peer that never declared support for any of them, consuming outbound connection slots with mismatched peers. An unprivileged attacker with a reachable IP can trigger this with a single crafted identify message.

## Finding Description

The root cause is in `required_flags_filter` at `network/src/peer_store/peer_store_impl.rs` L407–413:

```rust
pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    if required == Flags::RELAY | Flags::DISCOVERY | Flags::SYNC {
        t.contains(required) || t.contains(Flags::COMPATIBILITY)
    } else {
        t.contains(required)
    }
}
```

When `required == 0b1110` (RELAY|DISCOVERY|SYNC) and `t == 0b1` (COMPATIBILITY only):
- `t.contains(required)` → `false` (0b1 & 0b1110 = 0)
- `t.contains(Flags::COMPATIBILITY)` → `true`
- Function returns `true`

This result is consumed directly in `received_identify` at `network/src/protocols/identify/mod.rs` L434–443:

```rust
} else if required_flags_filter(required_flags, flags) {
    // The remote end can support all local protocols.
    let _ = context
        .open_protocols(
            context.session.id,
            TargetProtocol::Filter(Box::new(move |id| {
                id != &SupportProtocols::Feeler.protocol_id()
            })),
        )
        .await;
```

All non-feeler protocols are opened to the peer. The `verify()` function at L541–561 provides no additional guard — it only rejects `flag == 0` and mismatched network names. A `flag = 1` message with the correct network name passes cleanly.

Additionally, `add_outbound_addr` at L421–423 stores the peer with its `COMPATIBILITY`-only flags in the peer store before the protocol-open decision, meaning these addresses (with misleading flags) can be propagated to other nodes via the discovery protocol.

## Impact Explanation

**High — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An attacker operating a set of reachable nodes sending `flag=1` identify messages can:
1. Occupy outbound connection slots on victim nodes (typically 8–10 slots) with peers that support no useful protocols.
2. Prevent victim nodes from connecting to legitimate full-node peers, degrading block and transaction propagation.
3. Inject `COMPATIBILITY`-only addresses into the peer store, which the discovery protocol can then broadcast to other nodes, amplifying the effect across the network.

At scale, this degrades network-wide block relay and transaction propagation with minimal attacker cost (no PoW, no keys, just reachable IPs and a crafted P2P message).

## Likelihood Explanation

Any unprivileged external actor can trigger this. Requirements:
- A reachable IP address (or multiple, for scale)
- Ability to complete the tentacle P2P handshake
- Sending a valid `Identify` message with `name = <correct network name>`, `flag = 1`, `client_version = <any>`

No cryptographic material, no stake, no prior relationship with the target node is needed. The attack is repeatable and automatable.

## Recommendation

Remove the `COMPATIBILITY` special-case from `required_flags_filter`. The correct check is:

```rust
pub(crate) fn required_flags_filter(required: Flags, t: Flags) -> bool {
    t.contains(required)
}
```

If backward compatibility with pre-flag nodes is genuinely required, it must be handled through explicit version negotiation at a higher layer, not by silently bypassing capability checks for all nodes that set bit 0.

## Proof of Concept

1. Run a modified CKB node (or custom tentacle client) that responds to the Identify protocol with `name = "ckb"` (or the target network name), `flag = 1`, `client_version = "0.0.1"`.
2. Connect to a target CKB full node over TCP.
3. Complete the tentacle handshake; the Identify protocol opens automatically.
4. Send the crafted `Identify` message.
5. Observe the target node responds by opening SYNC, RELAY, and DISCOVERY protocols on the session (`open_protocols` with `TargetProtocol::Filter` excluding only Feeler).
6. The outbound slot is now held by a `COMPATIBILITY`-only peer. Repeat from multiple IPs to exhaust the target's outbound peer budget. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    fn verify(&self, data: &[u8]) -> Option<(Flags, String)> {
        let reader = packed::IdentifyReader::from_slice(data).ok()?;

        let name = reader.name().as_utf8().ok()?.to_owned();
        if self.name != name {
            warn!(
                "IdentifyProtocol detects peer has different network identifiers, local network id: {}, remote network id: {}",
                self.name, name,
            );
            return None;
        }

        let flag: u64 = reader.flag().into();
        if flag == 0 {
            return None;
        }

        let raw_client_version = reader.client_version().as_utf8().ok()?.to_owned();

        Some((Flags::from_bits_truncate(flag), raw_client_version))
    }
