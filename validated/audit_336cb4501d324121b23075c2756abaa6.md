### Title
`ScriptHashTypeVerifier` Omits `hash_type` Validation for `type_` Scripts in Transaction Outputs — (File: `verification/src/transaction_verifier.rs`)

---

### Summary

`ScriptHashTypeVerifier::verify()` checks the `hash_type` of each output's **lock** script against `ENABLED_SCRIPT_HASH_TYPE`, but the identical check is entirely absent for each output's **type_** script. This is a direct code-duplication omission: the same guard is written once for `lock` but never written for `type_`. A transaction submitter can craft an output whose `type_` script carries an unsupported `hash_type` (e.g., `Data3 = 3`), bypass the non-contextual gate, and force the node into expensive contextual script verification — or, if the VM silently skips execution for unknown hash types, bypass type-script enforcement entirely.

---

### Finding Description

`ScriptHashTypeVerifier` is part of `NonContextualTransactionVerifier` and is meant to reject transactions that reference script hash types outside the consensus-permitted set before any expensive work is done. [1](#0-0) 

The permitted set is: [2](#0-1) 

`verify()` iterates over outputs and validates only `output.lock().hash_type()`:

```rust
// verification/src/transaction_verifier.rs  lines 796-814
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
            let val: u8 = hash_type.into();
            if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                return Err(...ScriptHashTypeNotPermitted...);
            }
        } else {
            return Err(...InvalidScriptHashType...);
        }
        // ← output.type_() is never checked
    }
    Ok(())
}
``` [3](#0-2) 

`output.type_()` is a `Script` option that is structurally identical to `output.lock()` and carries its own `hash_type` field. No analogous check exists for it anywhere in `ScriptHashTypeVerifier`.

`NonContextualTransactionVerifier` composes `ScriptHashTypeVerifier` and is invoked in two places:

1. **Tx-pool admission** — `non_contextual_verify` in `tx-pool/src/util.rs`: [4](#0-3) 

2. **Block-level non-contextual check** — `NonContextualBlockTxsVerifier` in `verification/src/block_verifier.rs`: [5](#0-4) 

Both paths inherit the omission.

---

### Impact Explanation

**Scenario A — Resource exhaustion / tx-pool pollution**: An unprivileged RPC caller submits a transaction whose output `type_` script has `hash_type = 3` (`Data3`, not in `ENABLED_SCRIPT_HASH_TYPE`). The transaction passes `ScriptHashTypeVerifier`, is admitted to the tx-pool, and is queued for full contextual verification (script execution, capacity checks, fee calculation). The attacker can repeat this at scale, forcing the node to spend CPU and memory on transactions that will ultimately be rejected at the contextual stage.

**Scenario B — Type-script bypass**: If the CKB-VM / `TransactionScriptsVerifier` treats an unknown `hash_type` as "no script" or silently skips execution rather than returning an error, a transaction output with a `type_` script carrying an unsupported `hash_type` would have its type-script constraints silently unenforced. Type scripts are the primary mechanism for UDT (user-defined token) issuance rules, NFT ownership, and other application-layer invariants. Bypassing them would allow unauthorized token minting or destruction.

---

### Likelihood Explanation

The entry path requires only an RPC call to `send_transaction` or a P2P relay submission — no privileged access, no key material, no majority hashpower. Crafting a transaction with an output whose `type_` script has `hash_type = 3` is trivial. The omission is present in the current production code path for both tx-pool admission and block verification.

---

### Recommendation

Extend `ScriptHashTypeVerifier::verify()` to apply the same `ENABLED_SCRIPT_HASH_TYPE` guard to `output.type_()` when it is present:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        check_hash_type(output.lock().hash_type())?;
        // missing type_ check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

Extract the repeated guard into a private helper (analogous to the report's `isWithinFundingPeriod` suggestion) to prevent future divergence between the two checks.

### Proof of Concept

The existing test at `verification/src/tests/transaction_verifier.rs` line 101 demonstrates that a `lock` script with `hash_type = Data3` is correctly rejected: [6](#0-5) 

An analogous test for `type_` scripts does not exist and would currently **pass** (i.e., the verifier would return `Ok(())` for a transaction whose output `type_` script has `hash_type = Data3`), confirming the omission. The attacker-controlled entry path is `send_transaction` RPC → `non_contextual_verify` → `NonContextualTransactionVerifier::verify()` → `ScriptHashTypeVerifier::verify()`, which returns `Ok` without inspecting `output.type_().hash_type()`. // Verify that the ScriptHashType of transaction outputs
// is within the range permitted by the current consensus rules.
pub struct ScriptHashTypeVerifier<'a> {
    transaction: &'a TransactionView,
}

impl<'a> ScriptHashTypeVerifier<'a> {
    pub fn new(transaction: &'a TransactionView) -> Self {
        Self { transaction }
    }

    pub fn verify(&self) -> Result<(), Error> {
        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {
                    return Err(
                        TransactionError::ScriptHashTypeNotPermitted { hash_type: val }.into(),
                    );
                }
            } else {
                return Err((TransactionError::InvalidScriptHashType {
                    hash_type: output.lock().hash_type(),
                })
                .into());
            }
        }

        Ok(())
    }

pub(crate) fn non_contextual_verify(
    consensus: &Consensus,
    tx: &TransactionView,
) -> Result<(), Reject> {
    NonContextualTransactionVerifier::new(tx, consensus)
        .verify()
        .map_err(Reject::Verification)?;

    // The ckb consensus does not limit the size of a single transaction,
    // but if the size of the transaction is close to the limit of the block,
    // it may cause the transaction to fail to be packed
    let tx_size = tx.data().serialized_size_in_block() as u64;
    if tx_size > TRANSACTION_SIZE_LIMIT {
        return Err(Reject::ExceededTransactionSizeLimit(
            tx_size,
            TRANSACTION_SIZE_LIMIT,
        ));
    }
    // cellbase is only valid in a block, not as a loose transaction
    if tx.is_cellbase() {
        return Err(Reject::Malformed(
            "cellbase like".to_owned(),
            Default::default(),
        ));
    }

    Ok(())
}
