### Title
Orphan Pool `by_out_point` Index Grows Unbounded Relative to `entries` Cap, Enabling Remote Memory Exhaustion — (`tx-pool/src/component/orphan.rs`)

---

### Summary

The `OrphanPool` enforces a 100-transaction cap on `entries` but places **no independent bound** on `by_out_point`. A remote peer can relay 100 orphan transactions each carrying the maximum number of inputs permitted by `TRANSACTION_SIZE_LIMIT` (512 KB), causing `by_out_point` to accumulate ~1,000,000 entries (~145 MB) while `entries` stays at exactly 100. The stated DoS-prevention invariant is violated.

---

### Finding Description

`OrphanPool` maintains two data structures: [1](#0-0) 

```
entries:      HashMap<ProposalShortId, Entry>          ← capped at 100
by_out_point: HashMap<OutPoint, HashSet<ProposalShortId>> ← NOT independently capped
```

In `add_orphan_tx`, every input out-point of the incoming transaction is inserted into `by_out_point` **before** `limit_size()` is called: [2](#0-1) 

`limit_size()` only enforces the 100-entry cap on `entries`: [3](#0-2) 

When an entry is evicted, `remove_orphan_tx` correctly cleans up its `by_out_point` slots. The problem is not a leak — it is that the **steady-state size** of `by_out_point` is `entries.len() × inputs_per_tx`, not `entries.len()`. With 100 entries each carrying N inputs, `by_out_point` holds 100 × N keys.

**Input capacity per transaction:**

`TRANSACTION_SIZE_LIMIT` is 512,000 bytes: [4](#0-3) 

A molecule-encoded `CellInput` occupies ~48 bytes (44 bytes of data + 4-byte offset). A minimal transaction with N inputs and one output uses roughly 300 bytes of fixed overhead, leaving room for approximately **(512,000 − 300) / 48 ≈ 10,660 inputs**.

**Memory at steady state (100 orphan txs × 10,660 inputs each):**

| Structure | Entries | Per-entry cost | Total |
|---|---|---|---|
| `entries` (tx data) | 100 | 512 KB | ~51 MB |
| `by_out_point` | 1,066,000 | ~136 B (36 B key + HashSet overhead) | ~145 MB |
| **Combined** | | | **~196 MB** |

The comment on line 157 reads *"DoS prevention: do not allow OrphanPool to grow unbounded"*, but the bound it enforces covers only `entries`, not the index. [5](#0-4) 

---

### Impact Explanation

A single attacker-controlled peer can pin ~196 MB of heap memory on any reachable CKB node by continuously cycling 100 maximum-size orphan transactions. Because `limit_size()` evicts old entries and immediately accepts new ones, the attacker can sustain this pressure indefinitely at negligible cost (no PoW, no stake). On nodes with constrained RAM this can trigger OOM kills or severe GC pressure, degrading block propagation and transaction relay — i.e., network congestion.

---

### Likelihood Explanation

The attack path is fully reachable from the P2P relay layer:

1. Remote peer sends a transaction via the relay protocol.
2. `submit_remote_tx` → `non_contextual_verify` passes (tx ≤ 512 KB, not cellbase). [6](#0-5) 

3. Resolution fails with `OutPointError::Unknown` (all inputs reference non-existent cells).
4. `after_process` calls `add_orphan`. [7](#0-6) 

5. `add_orphan_tx` inserts ~10,660 entries into `by_out_point` per transaction.

No privileged access, no PoW, no Sybil attack is required. A single peer connection suffices.

---

### Recommendation

Bound `by_out_point` independently, or account for input count when enforcing the orphan pool limit. Concrete options:

1. **Cap total `by_out_point` entries**: reject (or evict) an orphan if adding it would push `by_out_point.len()` above `DEFAULT_MAX_ORPHAN_TRANSACTIONS × MAX_INPUTS_PER_ORPHAN`.
2. **Limit inputs per orphan**: reject orphan transactions whose input count exceeds a configurable threshold (e.g., 100 inputs).
3. **Count by weight**: replace the flat 100-tx cap with a byte-budget cap that accounts for both the transaction payload and its index footprint.

---

### Proof of Concept

```rust
// Pseudocode — directly exercises add_orphan_tx
let mut pool = OrphanPool::new();
for i in 0..101 {
    // Build a tx with ~10_000 distinct unknown inputs, total size ≤ 512 KB
    let tx = build_max_input_orphan_tx(i);
    pool.add_orphan_tx(tx, peer, declared_cycle);
}
assert_eq!(pool.entries.len(), 100);          // capped as advertised
assert!(pool.by_out_point.len() > 900_000);   // ~1_000_000 — far beyond the cap
```

`entries` stays at 100; `by_out_point` grows to ~1,000,000 entries, consuming ~145 MB of heap beyond the ~51 MB already used by the transaction payloads themselves.
