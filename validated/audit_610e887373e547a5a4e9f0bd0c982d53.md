### Title
`BLOCK_INVALID` Status Overwritten to `HEADER_VALID` via Lenient State Check in `HeaderAcceptor::accept` — (`sync/src/synchronizer/headers_process.rs`)

---

### Summary

`HeaderAcceptor::accept` in the sync protocol's header processing path does not guard against the `BLOCK_INVALID` terminal state. A block that has already been fully verified and rejected (contextually invalid) can have its `BLOCK_INVALID` status silently overwritten back to `HEADER_VALID` when any peer re-sends the same header via a `Headers` message. This allows an attacker to repeatedly re-submit a known-invalid block for full contextual re-verification, bypassing the peer-punishment mechanism and causing sustained CPU/memory/disk resource exhaustion.

---

### Finding Description

**Root cause — missing `BLOCK_INVALID` guard in `HeaderAcceptor::accept`:** [1](#0-0) 

The function opens with an acknowledged `FIXME`:

```rust
// FIXME If status == BLOCK_INVALID then return early. But which error
// type should we return?
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::HEADER_VALID) {
    // ... update peer state and return early
    return result;
}
``` [2](#0-1) 

The early-return guard checks only for `HEADER_VALID`. Because `BLOCK_INVALID` is a disjoint bit (`1 << 12 = 4096`) with no overlap with `HEADER_VALID` (`1`), a header in the `BLOCK_INVALID` state does **not** trigger the early return. [3](#0-2) 

The function then runs three checks — `prev_block_check`, `non_contextual_check`, `version_check` — and if all pass, calls `insert_valid_header`, which overwrites the entry in `block_status_map`: [4](#0-3) 

This is an unconditional `DashMap::insert`, so `BLOCK_INVALID` is silently replaced with `HEADER_VALID`.

**Why the three checks do not catch contextually-invalid blocks:**

A block can be marked `BLOCK_INVALID` by `ConsumeUnverifiedBlockProcessor::consume_unverified_blocks` after full CKB-VM contextual verification fails: [5](#0-4) 

This contextual failure is not re-checked by `prev_block_check`, `non_contextual_check`, or `version_check` in `HeaderAcceptor::accept`. Those three checks only cover: invalid parent, non-contextual header format errors, and version mismatch. A block whose header is structurally valid but whose transactions fail script execution will pass all three and have its status revived.

**Re-submission path after revival:**

Once the status is `HEADER_VALID`, `asynchronous_process_remote_block` accepts the block for re-processing: [6](#0-5) 

And `new_block_received` in `SyncShared` accepts it into the pipeline: [7](#0-6) 

The block then traverses the full async pipeline again: `ChainService::asynchronous_process_block` → `insert_block` (DB write) → `OrphanBroker` → `PreloadUnverified` → `ConsumeUnverifiedBlocks` (full CKB-VM re-execution) → fails again → `BLOCK_INVALID` again. [8](#0-7) 

---

### Impact Explanation

An attacker controlling one or more P2P peers can:

1. Craft a block whose header is non-contextually valid but whose transactions fail CKB-VM script execution (e.g., a script that consumes maximum cycles before aborting).
2. Submit the block once; it is stored, fully verified, fails, and is marked `BLOCK_INVALID`.
3. Reconnect (or use a second peer) and send a `Headers` message containing the same header.
4. `HeaderAcceptor::accept` revives the status to `HEADER_VALID`.
5. Send `SendBlock` with the same block body.
6. The node re-executes the full CKB-VM verification pipeline.
7. Repeat indefinitely.

Each cycle costs: one DB write (`insert_block`), one full CKB-VM script execution up to `max_tx_verify_cycles`, and associated memory allocation. The peer-punishment callback fires only after contextual verification completes asynchronously, and the attacker can rotate peers or reconnect before the ban takes effect. This constitutes a sustained, low-cost DoS against any CKB node reachable on the P2P network.

**Severity: Medium** — No consensus integrity or asset theft is possible, but the resource exhaustion is real and repeatable by any unprivileged peer.

---

### Likelihood Explanation

- Reachable by any unprivileged P2P peer via the standard `Headers` sync message.
- No special keys, majority hashpower, or privileged access required.
- The `FIXME` comment at line 301 of `headers_process.rs` confirms the developers are aware of the gap but have not resolved it.
- The attack requires only crafting a block with a valid header and invalid scripts, which is trivially achievable by any transaction sender who can also operate a P2P node.

---

### Recommendation

In `HeaderAcceptor::accept`, add an explicit early-return guard for `BLOCK_INVALID` before the existing `HEADER_VALID` check:

```rust
let status = self.active_chain.get_block_status(&self.header.hash());
if status.contains(BlockStatus::BLOCK_INVALID) {
    // Terminal state — do not allow revival.
    state.invalid(Some(ValidationError::InvalidParent)); // or a dedicated variant
    return result;
}
if status.contains(BlockStatus::HEADER_VALID) {
    // Already accepted; update peer state only.
    ...
    return result;
}
```

This mirrors the pattern already used in `CompactBlockProcess::contextual_check` and `asynchronous_process_remote_block`, which both correctly short-circuit on `BLOCK_INVALID`. [9](#0-8) 

---

### Proof of Concept

1. Attacker operates a P2P peer connected to the victim node.
2. Attacker constructs block `B` at height `tip+1` with a valid header (correct PoW, correct parent, correct epoch) but with a transaction whose lock script always fails (e.g., exits with error code 1 after consuming max cycles).
3. Attacker sends `SendBlock(B)` via the Sync protocol. The victim node:
   - Passes `non_contextual_verify` in `ChainService`.
   - Stores `B` in DB.
   - Sends `B` to `ConsumeUnverifiedBlocks`.
   - CKB-VM executes the script, fails.
   - Sets `block_status_map[B.hash] = BLOCK_INVALID`.
4. Attacker sends `Headers([B.header])` via the Sync protocol. `HeaderAcceptor::accept` runs:
   - `BLOCK_INVALID` does not contain `HEADER_VALID` → no early return.
   - `prev_block_check` passes (parent is valid).
   - `non_contextual_check` passes (header format is valid).
   - `version_check` passes.
   - `insert_valid_header` sets `block_status_map[B.hash] = HEADER_VALID`.
5. Attacker sends `SendBlock(B)` again. `new_block_received` sees `HEADER_VALID`, accepts it.
6. Full CKB-VM re-execution occurs. Fails again. `BLOCK_INVALID` set again.
7. Attacker repeats from step 4 indefinitely, causing continuous CPU/disk exhaustion on the victim node.

    pub fn accept(&self) -> ValidationResult {
        let mut result = ValidationResult::default();
        let sync_shared = self.active_chain.sync_shared();
        let state = self.active_chain.state();
        let shared = sync_shared.shared();

        // FIXME If status == BLOCK_INVALID then return early. But which error
        // type should we return?
        let status = self.active_chain.get_block_status(&self.header.hash());
        if status.contains(BlockStatus::HEADER_VALID) {
            let header_index = sync_shared
                .get_header_index_view(
                    &self.header.hash(),
                    status.contains(BlockStatus::BLOCK_STORED),
                )
                .unwrap_or_else(|| {
                    panic!(
                        "header {}-{} with HEADER_VALID should exist",
                        self.header.number(),
                        self.header.hash()
                    )
                })
                .as_header_index();
            state
                .peers()
                .may_set_best_known_header(self.peer, header_index);
            return result;
        }

        if self.prev_block_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-parent header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        if let Some(is_invalid) = self.non_contextual_check(&mut result).err() {
            debug!(
                "HeadersProcess rejected non-contextual header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            if is_invalid {
                shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            }
            return result;
        }

        if self.version_check(&mut result).is_err() {
            debug!(
                "HeadersProcess rejected invalid-version header: {} {}",
                self.header.number(),
                self.header.hash(),
            );
            shared.insert_block_status(self.header.hash(), BlockStatus::BLOCK_INVALID);
            return result;
        }

        sync_shared.insert_valid_header(self.peer, self.header);
        result
    }

    }

            status.insert(BlockStatus::BLOCK_RECEIVED);
            return true;
        }
        false
    }

    fn asynchronous_process_block(&self, lonely_block: LonelyBlock) {
        let block_number = lonely_block.block().number();
        let block_hash = lonely_block.block().hash();
        // Skip verifying a genesis block if its hash is equal to our genesis hash,
        // otherwise, return error and ban peer.
        if block_number < 1 {
            if self.shared.genesis_hash() != block_hash {
                warn!(
                    "receive 0 number block: 0-{}, expect genesis hash: {}",
                    block_hash,
                    self.shared.genesis_hash()

                self.shared
                    .insert_block_status(lonely_block.block().hash(), BlockStatus::BLOCK_INVALID);
                let error = InternalErrorKind::System
                    .other("Invalid genesis block received")
                    .into();
                lonely_block.execute_callback(Err(error));
            } else {
                warn!("receive 0 number block: 0-{}", block_hash);
                lonely_block.execute_callback(Ok(false));
            }
            return;
        }

        if lonely_block.switch().is_none()
            || matches!(lonely_block.switch(), Some(switch) if !switch.disable_non_contextual())

                return;

            return;
        }

        self.orphan_broker.process_lonely_block(lonely_block.into());
