### Title
Governance VAA Sequence Gap Allows Unprivileged Actor to Permanently Block Critical Governance Actions - (File: target_chains/ethereum/contracts/contracts/executor/Executor.sol)

### Summary

The `Executor.sol` contract's `verifyGovernanceVM()` function accepts any governance VAA whose sequence number is strictly greater than `lastExecutedSequence`, permitting arbitrary sequence gaps. Because `execute()` is a public function and Wormhole VAAs are publicly available once guardian-signed, any unprivileged transaction sender can front-run the governance relayer by submitting a higher-sequence VAA first, permanently rendering any lower-sequence VAA unexecutable.

### Finding Description

In `verifyGovernanceVM()`, the ordering check is:

```solidity
if (vm.sequence <= lastExecutedSequence)
    revert ExecutorErrors.MessageOutOfOrder();
lastExecutedSequence = vm.sequence;
``` [1](#0-0) 

The condition `<=` accepts any sequence strictly greater than the last executed one, meaning sequence gaps are silently allowed. If sequence N+1 is accepted first, `lastExecutedSequence` advances to N+1, and any subsequent attempt to execute sequence N will permanently revert with `MessageOutOfOrder` because `N <= N+1`.

The entry point is fully public:

```solidity
function execute(
    bytes memory encodedVm
) public payable returns (bytes memory response) {
``` [2](#0-1) 

Wormhole VAAs are publicly retrievable from the guardian API once signed. An attacker needs no privileged access — only the ability to call `execute()` with a valid, guardian-signed VAA that has a sequence number higher than the current `lastExecutedSequence`.

The same pattern exists in `PythGovernance.sol`:

```solidity
if (vm.sequence <= lastExecutedGovernanceSequence())
    revert PythErrors.OldGovernanceMessage();
setLastExecutedGovernanceSequence(vm.sequence);
``` [3](#0-2) 

And in the Aptos governance module:

```move
assert!(sequence > state::get_last_executed_governance_sequence(), error::invalid_governance_sequence_number());
state::set_last_executed_governance_sequence(sequence);
``` [4](#0-3) 

The Solana `remote_executor` enforces the same non-strict ordering:

```rust
constraint = posted_vaa.sequence > claim_record.sequence @ExecutorError::NonIncreasingSequence
``` [5](#0-4) 

The existing test `testOutOfOrder` confirms the contract intentionally accepts sequence 4 after sequence 3 was already executed (skipping nothing), but does not test the adversarial case where a higher-sequence VAA is submitted before a lower-sequence one that was intended to execute: [6](#0-5) 

### Impact Explanation

A governance action encoded in VAA sequence N — such as a critical security patch, a contract upgrade, or a data-source update — can be permanently blocked by an unprivileged actor who submits VAA sequence N+1 first. Once `lastExecutedSequence` advances past N, VAA N can never be executed. The governance team would be forced to re-issue the action under a new, higher sequence number, introducing delay and operational risk. In an emergency scenario (e.g., patching an active exploit), this delay could be catastrophic.

### Likelihood Explanation

Governance VAAs are publicly available on the Wormhole guardian REST API (`/v1/signed_vaa/...`) immediately after guardian quorum is reached. The governance relayer (`sync_governance_vaas.ts`) fetches and submits them sequentially: [7](#0-6) 

During any period where two or more governance VAAs are signed but not yet submitted (e.g., a governance emergency producing multiple rapid actions), an attacker monitoring the guardian API can submit the higher-sequence VAA first. This is a realistic front-running scenario requiring no special privileges.

### Recommendation

Replace the gap-permitting `<=` check with a strict consecutive-sequence requirement:

```solidity
if (vm.sequence != lastExecutedSequence + 1)
    revert ExecutorErrors.MessageOutOfOrder();
```

If intentional skipping of sequences is required (e.g., for VAAs targeting other chains), implement an explicit allowlist of skippable sequences controlled by governance, or use a per-chain sequence namespace so that cross-chain VAAs do not share a single monotonic counter.

### Proof of Concept

1. Governance multisig creates VAA sequence N (e.g., `UpgradeContract` security patch) and VAA sequence N+1 (e.g., `SetFee` routine update). Both are signed by Wormhole guardians and appear on the guardian API.
2. Attacker calls `executor.execute(encodedVaaN1)` (sequence N+1) before the governance relayer.
3. `verifyGovernanceVM` passes: `N+1 > lastExecutedSequence`. `lastExecutedSequence` is set to N+1.
4. Governance relayer calls `executor.execute(encodedVaaN)` (sequence N).
5. `verifyGovernanceVM` reverts: `N <= N+1` → `MessageOutOfOrder`.
6. The security patch (sequence N) is permanently unexecutable on this chain. The governance team must re-issue it as a new VAA with sequence > N+1, losing critical response time.
