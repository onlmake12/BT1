### Title
Unbounded `while` loop in `executeCallback()` can cause gas exhaustion, permanently locking requester funds — (`File: target_chains/ethereum/contracts/contracts/echo/Echo.sol`)

---

### Summary

`Echo.executeCallback()` contains an unbounded `while` loop that scans forward through all sequence numbers from `_state.firstUnfulfilledSeq` to `_state.currentSequenceNumber` after each fulfilled request. If a large number of requests have been created and fulfilled out-of-order, this loop can exceed the block gas limit, causing `executeCallback` to revert and permanently locking the requester's paid fee in the contract. The code itself acknowledges the problem with an explicit TODO comment.

---

### Finding Description

After clearing a fulfilled request, `executeCallback` runs the following loop to advance the `firstUnfulfilledSeq` pointer:

```solidity
// TODO: I'm pretty sure this is going to use a lot of gas because it's doing a
// storage lookup for each sequence number.
// a better solution would be a doubly-linked list of active requests.
while (
    _state.firstUnfulfilledSeq < _state.currentSequenceNumber &&
    !isActive(findRequest(_state.firstUnfulfilledSeq))
) {
    _state.firstUnfulfilledSeq++;
}
``` [1](#0-0) 

Each iteration calls `findRequest`, which performs at least one `SLOAD` (cold: 2,100 gas; warm: 100 gas after the first 32 unique slots). The loop runs until it finds an active request or exhausts all sequence numbers up to `currentSequenceNumber`. [2](#0-1) 

`currentSequenceNumber` is a `uint64` that increments with every call to `requestPriceUpdatesWithCallback`, and there is no cap on how large the gap `currentSequenceNumber - firstUnfulfilledSeq` can grow. [3](#0-2) 

The request storage uses a 32-slot fixed array (`NUM_REQUESTS = 32`) plus an unbounded overflow mapping. Requests evicted to the overflow mapping each occupy a unique cold storage slot, costing 2,100 gas per `findRequest` call for those entries. [4](#0-3) 

The `clearRequest` call that precedes the loop clears the request from storage **before** the loop runs. If the loop causes an out-of-gas revert, the entire transaction reverts (including `clearRequest`), so the request remains active but the provider cannot successfully call `executeCallback` again — every subsequent attempt will also hit the same OOG condition. [5](#0-4) 

---

### Impact Explanation

- `executeCallback` reverts with OOG for the oldest unfulfilled request once the gap is large enough.
- The request remains permanently active but can never be fulfilled — every attempt reverts.
- The requester's fee (`req.fee`, stored as `uint96`) is permanently locked in the contract with no recovery path.
- The consumer contract never receives its price update callback, breaking any downstream logic that depends on it. [6](#0-5) 

---

### Likelihood Explanation

- `requestPriceUpdatesWithCallback` is callable by any unprivileged user who pays the required fee.
- `executeCallback` is callable by any provider (or anyone after the exclusivity period expires).
- The attack requires creating many requests and fulfilling all but the oldest. Each request costs fees, but with a low `pythFeeInWei` or provider fee, the cost per request can be small.
- With a 30M gas block limit and ~2,100 gas per cold SLOAD iteration, approximately 14,000 fulfilled-but-scanned sequence numbers suffice to cause OOG.
- This can also occur **naturally** in high-volume usage without any malicious intent: if the oldest request is simply the last to be fulfilled in a busy period, the while loop scans all intervening fulfilled requests. [7](#0-6) 

---

### Recommendation

1. **Remove the while loop from `executeCallback`** and instead use a lazy/on-demand approach: do not advance `firstUnfulfilledSeq` inside `executeCallback` at all, or advance it by at most a small bounded number of steps (e.g., cap at 32 iterations matching `NUM_REQUESTS`).
2. **Use a doubly-linked list** of active requests as suggested in the existing TODO comment, so removal is O(1).
3. **Separate the `firstUnfulfilledSeq` advancement** into a standalone permissionless function with a caller-supplied step limit, so the cost is paid explicitly and does not block `executeCallback`.

---

### Proof of Concept

1. Attacker calls `requestPriceUpdatesWithCallback` N times, creating requests with sequence numbers 1 through N (paying fees for each).
2. Attacker (or any provider) calls `executeCallback` for requests 2 through N. Each of these succeeds: since `firstUnfulfilledSeq = 1` and request 1 is still active, the while loop immediately terminates (the condition `!isActive(findRequest(1))` is false).
3. Now: `firstUnfulfilledSeq = 1`, `currentSequenceNumber = N+1`, and requests 2..N are all cleared.
4. When request 1 is finally fulfilled, `clearRequest(1)` runs, then the while loop starts at seq 1 and must scan through all N-1 cleared sequence numbers (each requiring a storage read via `findRequest`) before reaching `currentSequenceNumber`.
5. For N ≈ 14,000 (at ~2,100 gas/cold SLOAD, 30M gas limit), the transaction reverts with OOG.
6. Request 1 can never be fulfilled. Its fee is permanently locked in `_state.providers[provider].accruedFeesInWei` — except the fee credit line also reverts, so the fee stays in the contract with no withdrawal path for the requester. [8](#0-7)

        uint64 sequenceNumber
    ) internal view returns (Request storage req) {
        (bytes32 key, uint8 shortKey) = requestKey(sequenceNumber);

        req = _state.requests[shortKey];
        if (req.sequenceNumber == sequenceNumber) {
            return req;
        } else {
            req = _state.requestsOverflow[key];
