Audit Report

## Title
`ScriptHashTypeVerifier` Omits Type Script Hash Type Validation Against `ENABLED_SCRIPT_HASH_TYPE` — (File: `verification/src/transaction_verifier.rs`)

## Summary

`ScriptHashTypeVerifier::verify()` validates each output's **lock** script `hash_type` against `ENABLED_SCRIPT_HASH_TYPE` but never reads `output.type_().hash_type()`. A transaction carrying a future/unactivated `hash_type` (e.g., `0x06` = `Data3`) in its type script passes every non-contextual gate and enters the contextual-verification queue, where it is rejected only after consuming CPU and memory. Because non-contextual checks do not verify input existence, an attacker can craft such transactions at essentially zero cost and flood the verification pipeline.

## Finding Description

**`ENABLED_SCRIPT_HASH_TYPE`** is defined as `{0, 1, 2, 4}`: [1](#0-0) 

**`ScriptHashTypeVerifier::verify()`** iterates outputs and checks only `output.lock().hash_type()`. The type script branch is entirely absent: [2](#0-1) 

The struct's own doc-comment acknowledges the scope: *"Check whether output **lock** hash type within enabled range."* [3](#0-2) 

**Why `check_data()` does not close the gap:** `CellOutputReader::check_data()` does call `self.type_().check_data()`: [4](#0-3) 

But `check_data()` delegates to `ScriptHashType::verify_value(v)`, which accepts any even byte or `1`: [5](#0-4) 

`Data3 = 6` (even), `Data4 = 8` (even), etc. all satisfy `verify_value`. This is a structural/encoding check, not a consensus-activation check. It cannot substitute for the missing `ENABLED_SCRIPT_HASH_TYPE` gate.

**Contextual rejection:** When the transaction reaches contextual verification, `select_version()` hits the catch-all arm for unrecognized hash types and returns an error: [6](#0-5) 

**Exploit path:**
1. Attacker calls `send_transaction` RPC with a transaction whose output `type_` script has `hash_type = 0x06`.
2. `check_data()` passes: `verify_value(6)` → `6.is_multiple_of(2)` → `true`.
3. `NonContextualTransactionVerifier::verify()` runs `ScriptHashTypeVerifier::verify()`, which only inspects the lock script — type script `hash_type` is never tested. Non-contextual check passes. [7](#0-6) 
4. Transaction enters the contextual-verification queue.
5. `select_version()` on the type script hits the catch-all → `Err(InvalidScriptHashType)` → rejected after consuming contextual-verification resources.

Because `NonContextualTransactionVerifier` does not verify input existence (that is a contextual check), the attacker can use fabricated inputs, making each malformed transaction free to construct.

**Note on `CellbaseVerifier`:** The claim that `CellbaseVerifier` has the same gap is incorrect. Cellbase outputs are explicitly required to have no type script at all; any cellbase output with a type script is rejected before the hash-type loop runs: [8](#0-7) 

## Impact Explanation

**High (10001–15000 points) — Vulnerabilities or bad designs which could cause CKB network congestion with few costs.**

An unprivileged attacker can submit an unbounded stream of transactions with fabricated inputs and `type_.hash_type = 6`. Each transaction passes the cheap non-contextual gate and enters the expensive contextual-verification pipeline before being rejected. The asymmetry between the near-zero cost to craft such transactions and the CPU/memory consumed per contextual verification attempt enables targeted resource exhaustion of the tx-pool verification pipeline.

## Likelihood Explanation

The entry point is the public `send_transaction` RPC, reachable by any caller with no keys, stake, or mining power. Constructing a transaction with a specific `hash_type` byte requires only standard transaction serialization. No Sybil capability or privileged access is needed. The attack is trivially repeatable at scale.

## Recommendation

Extend `ScriptHashTypeVerifier::verify()` to also validate the type script's `hash_type` when present, mirroring the existing lock-script check:

```rust
pub fn verify(&self) -> Result<(), Error> {
    for output in self.transaction.outputs() {
        // existing lock check
        check_hash_type(output.lock().hash_type())?;

        // ADD: type script check
        if let Some(type_script) = output.type_().to_opt() {
            check_hash_type(type_script.hash_type())?;
        }
    }
    Ok(())
}
```

Update the `NonContextualTransactionVerifier` doc-comment to reflect that both lock and type script hash types are validated.

## Proof of Concept

```
Transaction {
  inputs:  [ CellInput { previous_output: <any point, existence not checked non-contextually> } ]
  outputs: [
    CellOutput {
      lock: Script { hash_type: 0x01 (Type) },          // passes ScriptHashTypeVerifier
      type: Some(Script { hash_type: 0x06 (Data3) })    // NEVER checked by ScriptHashTypeVerifier
    }
  ]
}
```

Step-by-step:
1. `check_data()` → `verify_value(6)` → `6.is_multiple_of(2)` → `true` ✓
2. `ScriptHashTypeVerifier::verify()` → checks `lock.hash_type() = 1` ∈ `ENABLED_SCRIPT_HASH_TYPE` ✓; `type_.hash_type()` never read ✓
3. `NonContextualTransactionVerifier::verify()` returns `Ok(())`.
4. Transaction enters contextual-verification queue.
5. `select_version(type_script)` → catch-all arm → `Err(InvalidScriptHashType("Data3 not activated"))` → rejected.

Repeated at scale with distinct fabricated inputs, this constitutes a targeted resource-exhaustion attack against the tx-pool verification pipeline with no meaningful per-transaction cost to the attacker.

        for output in self.transaction.outputs() {
            if let Ok(hash_type) = TryInto::<ScriptHashType>::try_into(output.lock().hash_type()) {
                let val: u8 = hash_type.into();
                if !ENABLED_SCRIPT_HASH_TYPE.contains(&val) {

                })
