### Title
Post-Hoc Cycle-Limit Check in Parallel Block Transaction Verification Enables Miner-Triggered Verification Work Amplification — (`verification/contextual/src/contextual_block_verifier.rs`)

---

### Summary

`TxsVerifier::verify` in `verification/contextual/src/contextual_block_verifier.rs` verifies every committed transaction in a block in parallel, granting each transaction the **full** `max_block_cycles` budget as its individual limit. The block-level cycle sum is only checked **after** all transactions have been fully verified. A miner can craft a block containing N transactions each consuming just under `max_block_cycles` cycles; every transaction passes its individual check, but the post-hoc sum check rejects the block. All full nodes that receive the block are forced to perform O(N × `max_block_cycles`) verification work before the rejection fires — a direct analog to the circuit-breaker-fires-too-late pattern in the reference report.

---

### Finding Description

In `TxsVerifier::verify` (lines 387–473), all resolved transactions are iterated with `rayon`'s `par_iter()`. For every transaction that is not already in the verification cache, `ContextualTransactionVerifier::verify` is called with `self.context.consensus.max_block_cycles()` as the per-transaction cycle cap:

```rust
// verification/contextual/src/contextual_block_verifier.rs  lines 403-456
let ret = resolved
    .par_iter()
    .enumerate()
    .map(|(index, tx)| {
        // ...
        ContextualTransactionVerifier::new(...)
            .verify(
                self.context.consensus.max_block_cycles(),  // ← full block budget per tx
                skip_script_verify,
            )
        // ...
    })
    .skip(1)
    .collect::<Result<Vec<(Byte32, Completed)>, Error>>()?;
```

Only after the entire parallel collection completes is the aggregate sum compared to the limit:

```rust
// lines 458-472
let sum: Cycle = ret.iter().map(|(_, cache_entry)| cache_entry.cycles).sum();
// ...
if sum > self.context.consensus.max_block_cycles() {
    Err(BlockErrorKind::ExceededMaximumCycles.into())
} else {
    Ok((sum, cache_entires))
}
```

This is structurally identical to the reference vulnerability: the "circuit breaker" (`ExceededMaximumCycles`) fires only after the expensive work is already done, not during it.

The `max_block_cycles` consensus parameter is the total cycle budget for an entire block. Passing it as the per-transaction limit means a single transaction may legally consume the entire block budget. Because verification is parallel, there is no shared running counter that could abort sibling verifications early. A block with N such transactions causes N × `max_block_cycles` worth of script execution work on every verifying node before the sum check rejects it. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

Every full node that receives and attempts to verify the malicious block must execute all N transactions to completion before discovering the block is invalid. With `max_block_cycles` = 3,500,000,000 on mainnet and a block that can hold on the order of thousands of transactions within `max_block_bytes`, the amplification factor is:

```
verification_work = N × max_block_cycles
                  ≈ 3,000 × 3,500,000,000
                  = ~10.5 trillion cycles
```

compared to the normal worst-case of 3,500,000,000 cycles for a valid block. Because `par_iter()` uses all available CPU cores, this saturates the verifying node's CPU for the duration of the verification. Repeated across many blocks (the miner pays PoW cost once per block), this constitutes a sustained CPU-exhaustion DoS against all full nodes on the network.

The block is ultimately rejected, but the damage — wasted CPU on every peer — is already done, matching the "taker gain is maker loss" pattern: the miner pays one PoW solve; every full node pays N × `max_block_cycles` verification cost. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

The attacker role is a **miner** (explicitly in scope). No privileged access, leaked keys, or majority hashpower is required — a single solved block suffices. Constructing transactions that each consume close to `max_block_cycles` is straightforward: a script author can write a CKB-VM script that loops for a known number of cycles. The miner includes N such transactions in one block, each individually valid (cycles < `max_block_cycles`), but collectively exceeding the limit. The economic cost is one block's PoW solve; the benefit is forcing every full node to perform N × `max_block_cycles` verification work. [5](#0-4) 

---

### Recommendation

Pass a **remaining** cycle budget to each transaction verifier rather than the full `max_block_cycles`. Because `par_iter()` makes a shared running counter impractical, the simplest safe fix is to switch to sequential iteration for the cycle-budget tracking, or to impose a per-transaction cycle cap equal to `max_block_cycles` divided by the number of transactions in the block (a conservative upper bound). Alternatively, use an `AtomicU64` shared counter that each parallel worker checks and decrements; if the counter would go negative, the worker returns `ExceededMaximumCycles` immediately without executing the script. This mirrors the recommendation in the reference report: cap the individual operation at the remaining headroom, not at the full limit. [6](#0-5) 

---

### Proof of Concept

1. **Craft a high-cycle script**: Write a CKB-VM RISC-V script that executes exactly `max_block_cycles - 1` cycles (e.g., a tight loop with a known iteration count). Compile it and deploy it as a cell.

2. **Build N transactions**: Each transaction has one input cell locked by the high-cycle script. Each transaction individually consumes `max_block_cycles - 1` cycles — below the per-tx limit passed at line 433, so `ContextualTransactionVerifier::verify` returns `Ok`.

3. **Assemble the block**: Include all N transactions in one block (bounded by `max_block_bytes`). The block assembler's `package_txs` / `TxSelector::txs_to_commit` will include them because each transaction's individual cycle count is within the limit used during selection.

4. **Submit the block**: Broadcast the block to the network via the relay protocol.

5. **Observation**: Every full node calls `TxsVerifier::verify`. `par_iter()` launches N parallel workers, each running the high-cycle script to completion. After all workers finish, the sum check at line 468 fires: `sum = N × (max_block_cycles - 1) > max_block_cycles` → `Err(ExceededMaximumCycles)`. The block is rejected, but N × `max_block_cycles` cycles of CPU work have already been consumed on every verifying node. [7](#0-6) [8](#0-7) [9](#0-8)
